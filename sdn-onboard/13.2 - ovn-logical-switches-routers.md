## 13.2.1 Logical Switch

A Logical Switch is an L2 domain. It is equivalent to a VLAN, but it is programmable and scales without the 4,096 VLAN limit.

```bash
[admin]# ovn-nbctl ls-add ls-tenant1
[admin]# ovn-nbctl lsp-add ls-tenant1 vm1
[admin]# ovn-nbctl lsp-set-addresses vm1 "aa:bb:cc:00:00:01 10.0.0.1"
[admin]# ovn-nbctl lsp-set-port-security vm1 "aa:bb:cc:00:00:01 10.0.0.1"
```

**LSP addresses**: `<MAC> <IP1> <IP2> ...`. A single LSP can carry multiple IP addresses.

Special LSP types:
- `type=router`: connects to a Logical_Router_Port (LRP).
- `type=localnet`: connects to a physical network.
- `type=vtep`: connects to a hardware VTEP.

---

## 13.2.2 Logical Router

```bash
[admin]# ovn-nbctl lr-add lr-tenant1
[admin]# ovn-nbctl lrp-add lr-tenant1 lrp-ls1 aa:bb:cc:00:00:fe 10.0.0.254/24
[admin]# ovn-nbctl lsp-add ls-tenant1 ls1-lrp -- \
         set Logical_Switch_Port ls1-lrp type=router \
         options:router-port=lrp-ls1 \
         addresses=router
```

- `lrp-add <lr> <lrp-name> <mac> <cidr>`.
- The peer LSP must have `type=router` and `options:router-port=<lrp-name>`.

A Logical_Router is not a physical device. It is distributed across all chassis that participate in the logical network.

## 13.2.3 Connect multiple subnets

A typical tenant has multiple Logical_Switches (web, app, db) connected through a single Logical_Router:

```bash
[admin]# ovn-nbctl ls-add ls-web
[admin]# ovn-nbctl ls-add ls-app
[admin]# ovn-nbctl ls-add ls-db
[admin]# ovn-nbctl lr-add lr-tenant1

# One LRP per subnet:
[admin]# ovn-nbctl lrp-add lr-tenant1 lrp-web aa:bb:cc:00:01:fe 10.0.1.254/24
[admin]# ovn-nbctl lrp-add lr-tenant1 lrp-app aa:bb:cc:00:02:fe 10.0.2.254/24
[admin]# ovn-nbctl lrp-add lr-tenant1 lrp-db aa:bb:cc:00:03:fe 10.0.3.254/24

# Peer LSP:
[admin]# ovn-nbctl lsp-add ls-web ls-web-lrp \
         -- set Logical_Switch_Port ls-web-lrp type=router \
            options:router-port=lrp-web addresses=router
# Repeat for app and db
```

Traffic between a VM in ls-web and a VM in ls-db is routed through lr-tenant1 automatically.

## 13.2.5 Pipeline evolution

Different OVN versions have different stage counts. The numbers below combine Logical_Switch and Logical_Router stages, taken from the `ovn-northd.8` man page and from `northd/northd.c` at each release tag. The exact count can drift by one or two stages between minor versions because of backports, so verify against the baseline you actually deploy.

- **OVN 20.06**: about 20 LS ingress + 25 LS egress.
- **OVN 22.03** (curriculum baseline): about **23 LS ingress + 9 LS egress + 20 LR ingress + 7 LR egress** (verified via `mcp__github__get_file_contents(northd/northd.c, ref=v22.03.8)`, grep `"ls_in_*"` and `"lr_in_*"`).
- **OVN 24.03+**: the pipeline grows with new stages (for example, PMTUD and output_large_pkt_detect, see the commit history of `northd/northd.c`); the source is split into several `en-lflows-*.c` files.

Part 19 analyses the multichassis and PMTUD pipeline diff between 22.03 and 24.03 in detail.

## 13.2.6 Provider network (localnet)

Connect a logical network to a physical VLAN:

```bash
[admin]# ovn-nbctl ls-add ls-provider
[admin]# ovn-nbctl lsp-add ls-provider provider-port \
         -- lsp-set-addresses provider-port unknown \
         -- lsp-set-type provider-port localnet \
         -- lsp-set-options provider-port network_name=physnet1
```

The `network_name=physnet1` value must match `external_ids:ovn-bridge-mappings=physnet1:br-ex` on the chassis.

## 13.2.7 Logical Switch Ingress pipeline, 27 stage exhaustive (Phase H S46)

OVN translates each Logical_Switch into a chain of OpenFlow tables on br-int. This section breaks each stage down using Template D (per the `ovn-architecture(7)` baseline, OVN 22.03).

### 13.2.7.1 The 27 LS ingress stages

| T | Stage name | Purpose | Match example |
|---|---|---|---|
| 0 | `ls_in_check_port_sec` | Port security incoming check (drop if MAC or IP does not match the LSP) | `inport=="p1" && eth.src==MAC` |
| 1 | `ls_in_apply_port_sec` | Apply the verdict produced by check_port_sec | `reg0[15] == 0` pass |
| 2 | `ls_in_lookup_fdb` | Consult FDB for a known source MAC | `inport=="p1" && eth.src==MAC_A` |
| 3 | `ls_in_put_fdb` | Insert a new MAC into the FDB (learn action) | `reg0[11] == 0`, not yet known |
| 4 | `ls_in_pre_acl` | Pre-ACL CT track commit (set reg0[0]=trk) | `ip` |
| 5 | `ls_in_pre_lb` | Pre-LB CT track (for VIP traffic) | `ip4.dst == VIP` |
| 6 | `ls_in_pre_stateful` | Commit the packet to the conntrack zone | `reg0[0] == 1` |
| 7 | `ls_in_acl_hint` | Generate hints to reduce redundant CT commits | `ct.est && ct.rpl` |
| 8 | `ls_in_acl_eval` | ACL evaluation (direction=from-lport) | `ip4.dst==10.0.0.5 && tcp.dst==80` |
| 9 | `ls_in_acl_action` | Apply the ACL verdict (drop, reject, or allow) | `reg0[17] == 1` |
| 10 | `ls_in_qos` | QoS mark and meter | `inport=="p1" && ip.dscp_set` |
| 11 | `ls_in_lb_aff_check` | Load balancer affinity check (pinned session) | `ct.new && ip4.dst == VIP` |
| 12 | `ls_in_lb` | LB VIP to backend selection (hash-based) | `ct.new && ip4.dst == VIP` |
| 13 | `ls_in_lb_aff_learn` | Learn an affinity entry for persistence | `ct.new && reg0[9] == 1` |
| 14 | `ls_in_acl_after_lb_eval` | Re-evaluate ACL after the LB rewrite | `ip4.dst == backend_IP` |
| 15 | `ls_in_acl_after_lb_action` | Apply the verdict after LB | - |
| 16 | `ls_in_stateful` | Final CT commit with ct_mark or ct_label | `reg0[1] == 1` |
| 17 | `ls_in_pre_hairpin` | Detect hairpin (VIP to a backend on the same LS, a loop risk) | - |
| 18 | `ls_in_nat_hairpin` | NAT hairpin SNAT for local traffic | - |
| 19 | `ls_in_hairpin` | Redirect hairpin traffic back to the originator | - |
| 20 | `ls_in_arp_rsp` | ARP responder (local reply for the gateway IP) | `arp.op==1 && arp.tpa==GW_IP` |
| 21 | `ls_in_dhcp_options` | DHCPv4 and DHCPv6 options lookup | `inport=="p1" && udp.dst==67` |
| 22 | `ls_in_dhcp_response` | Generate the DHCP response packet | `reg0[3] == 1` |
| 23 | `ls_in_dns_lookup` | DNS query processing | `udp.dst==53` |
| 24 | `ls_in_dns_response` | DNS response generation | `reg0[4] == 1` |
| 25 | `ls_in_external_port` | Bridging to an external port (localnet) | `outport=="extport"` |
| 26 | `ls_in_l2_lkup` | Destination MAC lookup, set outport | `eth.dst == MAC_B` |

### 13.2.7.2 Template D for `ls_in_acl_eval` (T8), the most common stage

| Attribute | Value |
|---|---|
| **Table** | T8 (logical); physical table is around T16 in OVN 22.03 |
| **Stage name** | `S_SWITCH_IN_ACL_EVAL` in `northd/northd.h` |
| **Purpose** | Evaluate ACL rules from `NBDB.ACL` with direction=from-lport |
| **Match criteria** | The ACL expression compiled into an OpenFlow match |
| **Typical actions** | `reg0[17]=1;` (allow), `reg0[17]=0;` (drop), `reject { ... };` (generate ICMP) |
| **Produced by** | `ovn-northd` from `NBDB.ACL` direction=from-lport |
| **State input** | `reg0[0]` (ct_hint), `reg0[9]` (ct_est) |
| **State output** | `reg0[17]` (acl_allow flag) |

**Logical flow sample** (`ovn-sbctl lflow-list`):

```
  table=8 (ls_in_acl_eval), priority=65535,
    match=(!ct.est && ct.rel && !ct.rpl && ct_mark.blocked == 0),
    action=(reg0[17] = 1; next;)
  table=8 (ls_in_acl_eval), priority=2001,
    match=(inport == @pg_web && ip4.dst == $ip_db && tcp.dst == 3306),
    action=(reg0[17] = 1; next;)
  table=8 (ls_in_acl_eval), priority=2000,
    match=(ip),
    action=(reg0[17] = 0; next;)
```

Anatomy: priority 65535 covers established and related CT; 2001 carries the user ACL; 2000 is the default drop. `@pg_web` is a Port_Group reference; `$ip_db` is an Address_Set reference.

**Failure scenario:** traffic that should be allowed is dropped instead. Check the `ovn-trace` output. The usual root cause is a missing `allow-related` ACL or a fall-through to the default 2000 drop.

### 13.2.7.3 Template D for `ls_in_lb` (T12), Load balancer

| Attribute | Value |
|---|---|
| **Purpose** | LB VIP to backend selection via the `ct_lb_mark` action |
| **Match** | `ct.new && ip4.dst == VIP && tcp.dst == vport` |
| **Actions** | `ct_lb_mark(backends=<ip>:<port>...);` |
| **Produced by** | `NBDB.Load_Balancer` |
| **State input** | The `ct.new` flag |

**Logical flow:**

```
table=12 (ls_in_lb), priority=100,
  match=(ct.new && ip4.dst == 10.0.0.100 && tcp.dst == 80),
  action=(ct_lb_mark(backends=10.0.1.1:8080,10.0.1.2:8080,10.0.1.3:8080);)
```

The backend hash is deterministic per connection (5-tuple). Session affinity is handled by the `ls_in_lb_aff_*` stages.

### 13.2.7.4 Stage `ls_in_arp_rsp` (T20) in depth

```
table=20 (ls_in_arp_rsp), priority=50,
  match=(arp.op == 1 && arp.tpa == 10.0.0.1 /* gateway */),
  action=(eth.dst = eth.src;
          eth.src = aa:bb:cc:dd:ee:ff;
          arp.op = 2;
          arp.tha = arp.sha;
          arp.sha = aa:bb:cc:dd:ee:ff;
          arp.tpa = arp.spa;
          arp.spa = 10.0.0.1;
          outport = inport;
          flags.loopback = 1;
          output;)
```

Anatomy: a ten-step action chain converts the ARP Request into an ARP Reply and reflects it back to the ingress port. Compare with the OpenFlow equivalent in Part 4.9 §4.9.19 (the classic OpenFlow seven-action ARP responder).

## 13.2.8 Logical Switch Egress pipeline, the 10 stages (foundation gap closed)

**Audit finding:** before session S46, no file in the curriculum mentioned `ls_out_*`. This section closes a critical foundation gap.

The OVN egress pipeline runs after `ls_in_l2_lkup` sets the outport, traversing 10 stages before the packet exits via tunnel or bridge.

### 13.2.8.1 The 10 LS egress stages

| T | Stage name | Purpose |
|---|---|---|
| 0 | `ls_out_pre_lb` | Egress pre-LB tracking (reverse LB direction) |
| 1 | `ls_out_pre_acl` | Egress pre-ACL CT track |
| 2 | `ls_out_pre_stateful` | Egress CT commit |
| 3 | `ls_out_acl_hint` | Egress ACL hint |
| 4 | `ls_out_acl_eval` | Egress ACL evaluation (direction=to-lport) |
| 5 | `ls_out_acl_action` | Apply egress verdict |
| 6 | `ls_out_qos` | Egress QoS mark/meter |
| 7 | `ls_out_stateful` | Final egress CT commit |
| 8 | `ls_out_port_sec_ip` | Port security outgoing IP check |
| 9 | `ls_out_port_sec_l2` | Port security outgoing L2 check |

### 13.2.8.2 `ls_out_acl_eval` versus `ls_in_acl_eval`

The key difference: the egress side matches on **`outport`** instead of `inport`. The NBDB ACL direction column drives this choice:

```shell
# Egress ACL from web to db
$ sudo ovn-nbctl --direction=to-lport acl-add ls-prod 1000 \
    "outport == @pg_db && ip4.src == $ip_web && tcp.dst == 3306" allow-related
```

The corresponding logical flow:

```
table=4 (ls_out_acl_eval), priority=1000,
  match=(outport == @pg_db && ip4.src == $ip_web && tcp.dst == 3306),
  action=(reg0[17] = 1; next;)
```

### 13.2.8.3 `ls_out_port_sec_ip` and `ls_out_port_sec_l2`

These stages implement port security on the egress side. They ensure that an LSP only receives packets whose MAC and IP match the `addresses` configuration on that LSP. The purpose is to prevent receiver-side spoofing.

```
table=8 (ls_out_port_sec_ip), priority=90,
  match=(outport == "lsp-vm-db" && eth.dst == aa:bb:cc:dd:db && ip4.dst == 10.0.0.10),
  action=(next;)

table=9 (ls_out_port_sec_l2), priority=50,
  match=(outport == "lsp-vm-db" && eth.dst == aa:bb:cc:dd:db),
  action=(output;)

table=9 (ls_out_port_sec_l2), priority=0,
  match=(1),
  action=(drop;)  /* default drop, no address match */
```

### 13.2.8.4 End-to-end pipeline trace

```
Packet arrives at host X -> Geneve decap -> br-int physical T8 ->
Ingress pipeline T0-T26 (ls-prod-1) ->
  inport=p1, stage 0-26, outport set at T26 ->
Egress pipeline T0-T9 (ls-prod-1) ->
  outport=p2, final CT commit, port_sec check ->
Physical T65 -> wire
```

See Part 9.27 for the 3-tier diagnostic flow (ovn-trace at the logical layer, ofproto/trace at the physical layer, dpif/dump-flow at the datapath layer).

```shell
  table=0 (ls_in_port_sec_l2    ), priority=100 ,
    match=(eth.src == $<pg_ls-tenant-web_ports> && inport == <pg_ls-tenant-web_ports>),
    action=(next;)
  table=1 (ls_in_port_sec_ip    ), priority=90  , ...
  table=8 (ls_in_acl            ), priority=1000,
    match=(ip && ct.new && outport == @pg_web && tcp.dst == 8080),
    action=(reg0[1] = 1; next;)
  table=19 (ls_in_l2_lkup        ), priority=50  ,
    match=(eth.dst == aa:bb:cc:00:00:01),
    action=(outport = "lsp_web_01"; output;)

Datapath: "ls-tenant-web" (7c9d4e2a-...)  Pipeline: egress
  table=0 (ls_out_pre_lb         ), priority=0   , match=(1), action=(next;)
  table=8 (ls_out_port_sec_ip    ), priority=90  , ...
```

**9-attribute anatomy:**

| Field | Sample value | Meaning | Red flag if |
|-------|--------------|---------|-------------|
| `Datapath` line | `"ls-tenant-web" (7c9d4e2a-...)` | Identifies the LS | Missing means the LS has not yet been compiled into SBDB |
| `Pipeline` | `ingress` or `egress` | Direction of the stage block | Only one pipeline visible means northd has not finished compiling |
| `table=N` | 0 to 26 ingress, 0 to 9 egress | Stage table ID inside the pipeline | A gap in numbering means a stage is missing (bug) |
| Stage name | `ls_in_port_sec_l2`, `ls_in_acl` | Semantic purpose of the stage | A name mismatch against the expected OVN version indicates version skew |
| `priority` | 0 (default) to 32767 (max) | Within-table matching priority | Priority 0 is the catch-all default |
| `match` | `eth.src == $<pg>` | OVN expression language | An invalid expression produces a compile error in the log |
| `action` | `next;` or `outport = ...; output;` | Action pipeline | A missing `next;` causes a pipeline dead end |
| ACL stage count | 1 ACL = 2 OpenFlow rules | NBDB to SBDB ratio | If the ratio does not scale as expected, conjunction is not applying |
| Stage traversal | 0, 1, ..., 26, then egress 0, ..., 9 | Sequential pipeline | A skipped stage shows up as a gap in numbering |

**Failure scenarios:**

- **Datapath has the ingress pipeline but not the egress pipeline.** northd compile is incomplete. Wait 2 to 3 seconds; if the egress pipeline still does not appear, check the ovn-northd log.
- **Table 8 `ls_in_acl` is empty even though ACLs exist in NBDB.** The ACL match expression has a syntax error and northd rejected it. Check `/var/log/ovn/ovn-northd.log`.
- **Multiple ACLs overlap at priority 1000.** The evaluation order is undefined. Fix: assign distinct priorities per ACL.
- **Stage `ls_in_l2_lkup` at table 19 priority 50 falls back to flood.** The destination MAC is unknown and the ARP responder did not catch it. A VM declared with `addresses=unknown` falls back to flood.

## 13.2.Y Common misconception and Key point (v3.2 expansion)

> **Common misconception 1:** *"A Logical Router is a physical server, a chassis dedicated to routing."*
>
> **Reality:** the Logical Router (LR) is a **distributed abstraction**: routing logic is installed on every chassis. When a VM on chassis A sends a packet to the LR (the destination IP matches an LR network), the packet stays local on chassis A and runs the LR pipeline (decrement TTL, ARP-resolve the next hop, output through the LRP). It does not cross to another chassis unnecessarily. Only when the LR has to route to the external network (north-south NAT) does the traffic go through a chassisredirect port to the gateway chassis. This "distributed LR" model is one of the core benefits of OVN compared to traditional L3 routing through a physical router.

> **Common misconception 2:** *"OVN LS and LR are different from Neutron router and subnet, so legacy Neutron cannot be migrated to OVN."*
>
> **Reality:** the OVN Neutron ML2-OVN driver (`networking-ovn`) automatically maps Neutron network, router, and subnet objects to OVN LS, LR, LRP, and static route entries. The migration path has official tooling and documentation. Red Hat OpenStack Platform 16 (2020) defaulted to migrating legacy ML2-Linuxbridge deployments to ML2-OVN. Pain points to expect: (1) Floating IP behavior changes from centralized to distributed DNAT_and_SNAT, (2) Security Group translation has edge cases where stateful rule semantics differ, (3) the DHCP agent is removed in favor of OVN native DHCP. See the RHOSP migration guide for details.

> **Key point:** the 27 ingress and 10 egress stages of the LS pipeline are the core output of the OVN compiler. Each stage has a clear semantic purpose: port_security (anti-spoofing), FDB learn (MAC awareness), pre-CT (conntrack state check), ACL (policy), QoS (rate limit), LB (load balance plus hairpin), stateful commit, ARP responder, DHCP and DNS reply, L2 lookup (MAC to port). Knowing the stage sequence helps debugging: for example, "the packet reaches L2 lookup but does not forward" points at stage 19 `ls_in_l2_lkup` having an empty match, which is a MAC table issue. Section 13.2 (the full pipeline table from Phase H) is the reference for every stage.

## 13.2.Z ▶ Capstone POE: 3-tier tenant end-to-end ping trace (v3.2 expansion)

**Goal:** trace the full packet journey from a tenant-web VM to a tenant-db VM through LS, LR, and multiple pipeline stages, then confirm that policy enforcement behaves as intended.

**Scenario:** a 3-tier tenant `tenant-T1` with LS-web, LS-app, and LS-db, connected by a single LR-tenant that links all three LS instances. ACLs allow web to app on port 8080 and app to db on port 5432. Test by pinging from web to app (expected to succeed) and from web to db (expected to be blocked).

**Predict:**

Web to App:
- The packet enters the LS-web ingress pipeline (27 stages): port_sec, then FDB learn, then ACL (hits the allow-related rule for pg_web to pg_app), then L2 lookup (output port `ls-to-lr`).
- It transitions to the LR-tenant ingress pipeline: routing decision (the destination IP is in the LS-app subnet), output through LRP-app.
- It transitions to the LS-app egress pipeline: ACL (allow reply), then port_sec, then L2 lookup, then output to `lsp_app_01`.
- Expectation: ping succeeds with 2 to 5 ms latency (three pipeline traversals).

Web to DB:
- The packet enters the LS-web ingress pipeline: port_sec passes, then ACL evaluation begins.
- ACL table 8 priority 1000 match `pg_web -> pg_app` does not match because `outport = pg_db`.
- ACL table 8 priority 1 match `ip` triggers a drop action.
- The packet is dropped at the ACL stage.
- Expectation: ping fails with no reply.

**Observe:**

```bash
# Ping web to app
vm_web$ ping -c 5 10.0.1.5  # lsp_app_01
# Expectation: 5 packets received, around 2 ms RTT.

# Ping web to db
vm_web$ ping -c 5 10.0.2.5  # lsp_db_01
# Expectation: 100 percent packet loss.

# Trace the packet drop
central$ sudo ovn-trace ls-tenant-web \
    'inport=="lsp_web_01" && eth.src==aa:bb:cc:00:00:01 && \
     ip4.src==10.0.0.5 && ip4.dst==10.0.2.5 && ip.ttl==64 && \
     tcp.src==54321 && tcp.dst==5432'

# Output shows pipeline traversal:
# ... ingress ls_in_port_sec_l2 (pass) ...
# ... ingress ls_in_acl priority=1 match=(ip) action=drop ...
# PACKET DROPPED
```

**Explain:**

- ACL policy enforcement is correct: web to app is allowed by the priority 1000 specific rule, and web to db is denied by the priority 1 default-deny rule.
- The trace confirms the drop occurs at ingress table 8 (`ls_in_acl`) inside LS-web, before the packet reaches the LR.
- Distributed LR behavior: there is no cross-chassis traffic when the web and app VMs share a chassis.

**Falsification test:**

Remove the default-deny ACL at priority 1 and rerun the ping:
```bash
central$ sudo ovn-nbctl acl-del ls-tenant-web from-lport 1 "ip"
```

Retry ping web to db. Expectation: the ping now succeeds because the default-deny rule is gone. This confirms ACL priority semantics: default-deny is a required guardrail. Without default-deny the firewall is effectively open.

**Finish:**

Principle: **"the default-deny plus specific-allow pattern is the foundation of Zero Trust networking."** Apply this pattern to every OVN tenant. The pattern scales through Port_Group and conjunction, can be verified through `ovn-trace` regularly, and should fire an alert whenever the ACL rule set drifts from the baseline.

---

## Key takeaways

First, LS is an L2 domain and LR is an L3 router. They connect through an LRP plus a peer LSP of `type=router`.

Second, the LR is distributed across chassis. There is no physical router as a bottleneck.

Third, the pipeline table count varies across OVN versions. The v22.03 baseline is 24 plus 27 stages.

Fourth, the `localnet` LSP connects the logical network to a physical VLAN through bridge mapping.

Fifth, the LS ingress pipeline at v22.03.8 has 26 stages with the following high-level structure: port security, FDB learn, pre-CT, ACL eval and action, QoS, LB (with affinity), stateful commit, hairpin, ARP responder, DHCP and DNS, external port, L2 lookup. Each stage has a clear purpose, and OVN flag conditions drive the transitions between stages.

Sixth, the LS egress pipeline (10 stages) is symmetric with ingress for ACL, CT, and QoS, but it matches on `outport`. Port security runs at the end of the pipeline (T8 IP and T9 L2) to enforce receiver-side filtering.

## 13.2.X Logical_Switch như một bảng NBDB: tenant L2 broadcast domain đào sâu

> Phần này đào sâu vào **chính bảng `Logical_Switch`** (row trong OVN Northbound database), bổ sung cho phần dạy thực hành ở §13.2.1 và pipeline ở §13.2.7-13.2.8. Mục tiêu: kỹ sư hiểu thấu cấu trúc hàng NBDB, tập 11 cột của schema 22.03, vòng đời compile thành Datapath_Binding + 27 stage Logical_Flow ingress + 10 stage egress, và cách tra ngược từ một sự cố runtime về một row trong NBDB.

### Khái niệm cốt lõi

`Logical_Switch` là **bảng nội trú trong OVN Northbound database (NBDB)** mô tả một **L2 broadcast domain ảo** mà CMS (Neutron, kube-ovn, OpenStack) khai báo cho một tenant. Mỗi row đại diện cho một segment Ethernet logical: tất cả `Logical_Switch_Port` (LSP) thuộc row này nằm trong cùng broadcast domain, học MAC lẫn nhau, có cùng tập ACL, cùng tập DHCP option, cùng load balancer + DNS record. Loại: **OVSDB table**, root table (`isRoot=true` trong `ovn-nb.ovsschema`), persistent (replicated qua Raft cluster NBDB), declarative input cho ovn-northd. Không như Linux bridge truyền thống (per-host), Logical_Switch **trải dài qua mọi chassis** trong cluster qua Geneve tunnel mesh; broadcast domain L2 không bị giới hạn bởi physical wire.

### Lịch sử + bối cảnh ra đời

Bảng `Logical_Switch` xuất hiện cùng OVN 2.4 (2015) trong refactor của Russell Bryant (Red Hat) + Justin Pettit (VMware/Nicira) chuyển kiến trúc Nicira NSX OpenStack plugin sang OVN. Trước đó, Neutron OVS plugin map mỗi network thành một VLAN tag trên integration bridge per-host, scale bị giới hạn bởi 4096 VLAN ID + cần controller-side state per-host. OVN tách bạch: NBDB chứa **intent** (Logical_Switch row CMS đặt), SBDB chứa **runtime state** (Datapath_Binding row + Logical_Flow row do northd compile). Logical_Switch là một trong những bảng NBDB gốc cùng với `Logical_Router`, `Logical_Switch_Port`, `Logical_Router_Port`, `ACL`. OVN 21.03 thêm `forwarding_groups` (LB-style multi-port forwarding), 21.06 ổn định `dns_records` cho native DNS, 22.03 đưa `copp` (Control Plane Policing) vào schema (verified `ovn-nb.ovsschema` branch-22.03), 24.03 mở rộng `other_config` với key cho IGMP/MLD snooping advanced.

### Vị trí trong kiến trúc

Bảng nằm ở **layer NBDB**, là input layer trên cùng của control plane OVN. Scope: **per-cluster**. Một cluster OVN chỉ có một NBDB Raft cluster (3 hoặc 5 thành viên), CMS ghi NBDB qua `ovn-nbctl` hoặc client API; ovn-northd subscribe NBDB như reader, compile sang SBDB. Mỗi row Logical_Switch sinh ra **đúng một row Datapath_Binding** trong SBDB (`ovn-sbctl find Datapath_Binding external_ids:name=<ls_name>`) + **trung bình 50-200 row Logical_Flow** (tuỳ số LSP, ACL, LB) phân bổ vào 27 stage ingress + 10 stage egress. Cấp độ tác động: thay đổi một row Logical_Switch (ví dụ thêm ACL vào cột `acls`) làm ovn-northd recompute datapath đó trong < 200ms (incremental compile từ OVN 21.03+); ovn-controller trên mọi chassis sau đó re-cài flow OpenFlow trong < 1 giây qua I-P engine.

### Vai trò + nhiệm vụ

Logical_Switch gánh **năm vai trò** trong kiến trúc OVN:

1. **Định danh L2 broadcast domain**: cột `name` (string, không unique) làm identifier human-readable; `_uuid` là khoá duy nhất. Tất cả LSP trong row thuộc cùng broadcast domain.
2. **Container cho LSP**: cột `ports` (set strong ref tới `Logical_Switch_Port`) đảm bảo cascade delete — xoá LS xoá luôn mọi LSP. CMS không cần manual cleanup port.
3. **Anchor cho policy L2**: cột `acls` (set strong ref `ACL`), `qos_rules` (set ref `QoS`), `load_balancer` (set weak ref), `dns_records` (set weak ref `DNS`), `forwarding_groups` (set ref `Forwarding_Group`). Đây là điểm gắn policy ở scope LS; chính sách scope toàn cluster gắn ở Address_Set + Port_Group riêng.
4. **Truyền tham số runtime cho northd**: cột `other_config` (map string→string) chứa `mcast_snoop`, `mcast_querier`, `vlan-passthru`, `vni` (cho transit tunnel), `interconn-ts`. Northd đọc map này để quyết định compile pipeline thế nào.
5. **Báo metadata cho CMS**: cột `external_ids` (map) chứa `neutron:network_name`, `kube-ovn:vpc`, hoặc tương tự — CMS dùng để tra ngược từ row OVN về object CMS gốc.

### Vì sao sinh ra (động cơ thiết kế)

Trước OVN 2.4, kiến trúc Neutron OVS-ML2 plugin (Folsom 2012-Liberty 2015) lưu network state trong Neutron MySQL + đẩy VLAN/VXLAN config xuống mỗi compute host qua RPC; mỗi compute host tự tạo Linux bridge per-network. Vấn đề: (a) **Scaling 4096 VLAN limit** cho VLAN-based segmentation; VXLAN giải quyết VNI nhưng vẫn cần controller per-host phối hợp; (b) **State drift khi RPC fail**: compute host và Neutron DB không nhất quán; (c) **Hard reconfigure khi thêm ACL**: phải đẩy iptables rule mới qua agent. OVN giải quyết bằng cách **đẩy state ra OVSDB Raft + để northd compile + để mỗi chassis tự cài flow**: NBDB chứa Logical_Switch declarative; northd biên dịch sang Logical_Flow; ovn-controller chạy local trên mỗi chassis cài flow OpenFlow. Logical_Switch là **abstraction tenant-facing duy nhất** mà CMS biết — CMS không cần biết Datapath_Binding hay Logical_Flow, chỉ tạo + xoá Logical_Switch + LSP. Mô hình này scale lên 5000-10000 hypervisor + > 100K logical port (số liệu Red Hat OSP 17 + ovsdb-server perf benchmark 2023).

### Vấn đề mà nó giải quyết

Bảng này giải quyết **năm vấn đề kỹ thuật** rạch ròi: (1) **Decouple intent khỏi physical**: CMS chỉ khai báo "tenant1 cần một L2 segment" mà không cần biết VLAN nào, host nào, tunnel nào; (2) **Scope policy đúng cấp**: ACL gắn vào row LS chỉ ảnh hưởng port của LS đó; không phải push iptables global mỗi lần thêm rule; (3) **Cascade lifecycle nhất quán**: xoá row LS xoá tất cả LSP, ACL, QoS, forwarding_group thuộc về nó qua strong ref; CMS không cần pseudo-transaction; (4) **Multi-cluster federation qua interconnect**: cột `other_config:interconn-ts` đánh dấu transit switch giữa availability zone (OVN-IC từ 21.06+); (5) **Cross-chassis broadcast deterministic**: cùng row LS = cùng `tunnel_key` cho datapath = cùng metadata trong Geneve packet; broadcast frame được flood tới đúng tập chassis có LSP của LS đó (nhờ MC_FLOOD multicast group trong SBDB).

### Tầm quan trọng

Đây là **bảng cốt lõi tier 1 của NBDB**, ngang hàng với `Logical_Router`. Lý do: 100% workflow tenant network OVN bắt đầu bằng `ovn-nbctl ls-add`; 100% feature OVN cao cấp ở scope L2 (ACL stateful, native LB cho east-west, DHCP/DNS, port security, IGMP snooping) đều thêm cột hoặc field option vào Logical_Switch chứ không tạo bảng mới. Kỹ sư không hiểu Logical_Switch sẽ không tạo, không debug, không scale được bất kỳ tenant network nào trong OVN. Trong 320+ keyword của REF, Logical_Switch nằm trong nhóm 15 keyword cornerstone OVN (cùng `ovn-northd`, `ovn-controller`, NBDB, SBDB, Logical_Flow, Datapath_Binding, Logical_Router, Port_Binding, ACL, Load_Balancer, NAT, Geneve TLV, ovn-trace, ovn-detrace).

### Cơ chế hoạt động (chu kỳ compile một row)

Một row Logical_Switch đi qua **bốn giai đoạn** trong vòng đời:

1. **Khai báo từ CMS**: CMS gọi `ovn-nbctl ls-add ls-tenant1` → NBDB transaction tạo row mới với `name="ls-tenant1"`, các cột set khác `[]` mặc định. Transaction atomic; row visible trong < 50ms qua NBDB Raft replicate.
2. **Compile sang Datapath_Binding**: ovn-northd (chạy trên Raft leader của NBDB + write SBDB) phát hiện row mới qua I-P engine input change. Hàm `ovn_datapath_create` (`northd/northd.c` branch-22.03) tạo `struct ovn_datapath` in-memory cho row LS, allocate `tunnel_key` (1-16777215, unique trong cluster). Sau đó `northd_handle_lb_data_changes`/`build_datapaths` ghi vào SBDB: `INSERT INTO Datapath_Binding (tunnel_key=<allocated>, external_ids={'name': 'ls-tenant1', 'logical-switch': '<ls_uuid>'})`. Kết quả: 1 row Datapath_Binding tương ứng 1:1 với row Logical_Switch.
3. **Compile sang Logical_Flow rows**: northd chạy chuỗi hàm `build_lswitch_*` (e.g., `build_lswitch_input_port_security_l2`, `build_lswitch_arp_nd_responder_known_ips`, `build_lswitch_destination_lookup_bmcast`, `build_lswitch_ip_unicast_lookup`) để sinh Logical_Flow row cho từng stage. Mỗi LSP, ACL, LB, DNS record gắn vào LS đều spawn flow tương ứng. Tổng số: 50-500 row Logical_Flow per LS thông thường (cấp số nhân theo số LSP × số ACL × số stage).
4. **Cài flow trên chassis**: SBDB Raft replicate Logical_Flow → ovn-controller trên mọi chassis subscribe → mỗi controller chạy `lflow_run` (`controller/lflow.c`) translate Logical_Flow expression sang OpenFlow rule cụ thể, cài lên `br-int` qua `ofctrl_put`. Flow được tag bằng `cookie=<datapath_uuid_low_64>` để dễ dump-flow theo LS.

Invariant: tại mọi thời điểm, một row Logical_Switch tồn tại trong NBDB tương ứng đúng **một** row Datapath_Binding trong SBDB; xoá LS qua `ls-del` cascade xoá LSP + Datapath_Binding + Logical_Flow trong cùng round compile.

### Phân loại cột

11 cột của Logical_Switch (verified `ovn-nb.ovsschema` branch-22.03 [GitHub raw](https://raw.githubusercontent.com/ovn-org/ovn/branch-22.03/ovn-nb.ovsschema)) chia thành **năm nhóm** theo vai trò ngữ nghĩa:

| Nhóm | Cột | Type | Required | Ghi chú |
|------|-----|------|----------|---------|
| Định danh | `name` | string | Có (mặc định "") | CMS-friendly, không unique |
| Container | `ports` | set strong ref → Logical_Switch_Port | Không (mặc định []) | Cascade delete |
| Policy gắn LS | `acls` | set strong ref → ACL | Không | Stateful firewall |
| Policy gắn LS | `qos_rules` | set strong ref → QoS | Không | Mark + meter |
| Policy gắn LS | `load_balancer` | set weak ref → Load_Balancer | Không | East-west LB legacy |
| Policy gắn LS | `load_balancer_group` | set ref → Load_Balancer_Group | Không (24.03+) | Group LB modern |
| Policy gắn LS | `dns_records` | set weak ref → DNS | Không | Native DNS resolver |
| Policy gắn LS | `forwarding_groups` | set strong ref → Forwarding_Group | Không (21.03+) | LB-style L2 multi-output |
| Policy gắn LS | `copp` | weak ref → Copp (max 1) | Không (22.03+) | Control plane meter |
| Tham số dịch vụ | `other_config` | map string→string | Không | mcast_snoop, vni, interconn-ts |
| Metadata CMS | `external_ids` | map string→string | Không | CMS-tracked ID |

Không có index ràng buộc unique trên `name`; trùng tên hợp lệ (CMS phải tự đảm bảo unique nếu muốn). Khoá tra cứu duy nhất là `_uuid`.

### Cách kỹ sư vận hành thành thạo

Bốn kỹ năng nền tảng kỹ sư phải có với Logical_Switch:

1. **Đọc một row đầy đủ**: `sudo ovn-nbctl --columns=name,ports,acls,load_balancer,dns_records,other_config list Logical_Switch <ls_uuid_or_name>` cho thấy 6 cột chính + size từng set. Hoặc `sudo ovn-nbctl ls-list` lấy tổng quan + `sudo ovn-nbctl show ls-tenant1` để in cây LS → LSP với address.
2. **Xác minh compile xuống SBDB**: `sudo ovn-sbctl find Datapath_Binding external_ids:logical-switch=<ls_uuid>` xác nhận có Datapath_Binding tương ứng. Nếu không thấy → northd chưa kịp compile hoặc lỗi ovn-northd.log.
3. **Đếm pipeline output**: `sudo ovn-sbctl lflow-list <ls_name> | wc -l` đếm số Logical_Flow row. Một LS không LSP chỉ có ~30-50 flow stub; LS với 5 LSP + 3 ACL có ~200-300 flow; LS với 100 LSP + 50 ACL có > 2000 flow. Số flow tăng gần tuyến tính theo LSP × ACL.
4. **Quyết định khi nào dùng skill nào**: nếu CMS báo "tenant network không thấy trong cluster" → bắt đầu bằng `ls-list` (skill 1) để xác nhận row tồn tại; nếu CMS báo "VM trong network không học MAC nhau" → kiểm `lflow-list` stage `ls_in_l2_lkup` (skill 3); nếu nghi northd treo → kiểm Datapath_Binding (skill 2).

### Quy trình sử dụng (workflow chuẩn)

Workflow provisioning một tenant network 3-tier qua Logical_Switch gồm sáu bước, mỗi bước có lệnh xác minh:

1. **CMS tạo Logical_Switch trong NBDB**: `ovn-nbctl ls-add ls-tenant-web`. Xác minh: `ovn-nbctl ls-list | grep ls-tenant-web` thấy UUID + tên.
2. **Gắn ACL stateful + LB nếu cần**: `ovn-nbctl acl-add ls-tenant-web from-lport 1000 'ip4 && tcp.dst==80' allow-related`; `ovn-nbctl ls-lb-add ls-tenant-web lb-web`. Xác minh: `ovn-nbctl --columns=acls,load_balancer list Logical_Switch ls-tenant-web` thấy set có entry.
3. **Chờ northd compile**: thường < 200ms. Xác minh: `ovn-sbctl find Datapath_Binding external_ids:logical-switch=<ls_uuid>` thấy 1 row Datapath_Binding với `tunnel_key` allocated.
4. **Tạo LSP cho từng VM**: `ovn-nbctl lsp-add ls-tenant-web vm-web-1; ovn-nbctl lsp-set-addresses vm-web-1 "aa:bb:cc:00:00:01 10.0.0.5"`. Xác minh: `ovn-nbctl ls-show ls-tenant-web` thấy port được liệt kê dưới LS.
5. **Plug VM tap vào br-int trên hypervisor + set iface-id**: `ovs-vsctl add-port br-int tap-vm-web-1 -- set Interface tap-vm-web-1 external_ids:iface-id=vm-web-1`. Xác minh: `ovn-sbctl --columns=chassis,up list Port_Binding vm-web-1` thấy `chassis=[<uuid>], up=true` (xem Phần 13.5.1b cho chi tiết Port_Binding lifecycle).
6. **Kiểm pipeline output**: `ovn-sbctl lflow-list ls-tenant-web | head -30` xác nhận có flow ở stage `ls_in_port_sec_l2`, `ls_in_acl_eval`, `ls_in_l2_lkup` cho MAC vừa cài.

Best practice production: **luôn dùng Address_Set + Port_Group thay vì hardcode IP/MAC trong ACL** để scale-up tenant. Anti-pattern: tạo Logical_Switch riêng cho mỗi pod/container nhỏ → tăng số Datapath_Binding lên hàng chục nghìn → northd compile chậm > 30 giây và ovn-controller upcall lag. Kích thước hợp lý: ≤ 5000 LSP per LS; > 5000 → tách thành nhiều LS qua LR.

### Khi xảy ra sự cố (reverse symptom map)

Bảy symptom thường gặp trỏ về Logical_Switch:

| Symptom | Cột Logical_Switch cần kiểm | Diagnostic flow |
|---------|------------------------------|-----------------|
| "Tenant network không xuất hiện trong CMS" | `name`, `external_ids` | `ovn-nbctl ls-list` xác nhận row tồn tại |
| "VM cùng LS không ping được nhau" | `acls` | `ovn-nbctl acl-list ls-X` xem có default-deny chặn không; `ovn-trace` từ src tới dst |

| "DHCP từ VM không được phục vụ" | (LSP `dhcpv4_options`) + LS `other_config` | Kiểm `ovn-nbctl lsp-get-dhcpv4-options vm1` + `lflow-list ls-X` stage `ls_in_dhcp_options` |

### Liên quan mật thiết

Sáu thành phần coupling chặt với Logical_Switch (đổi cái nào thì phải xét lại Logical_Switch):

- **`Logical_Switch_Port`** (LSP): chứa trong cột `ports` strong ref. Mỗi LSP thuộc đúng 1 LS; xoá LS cascade xoá LSP.
- **`ACL`**: chứa trong cột `acls` strong ref. ACL scope LS; xoá LS cascade xoá ACL gắn LS đó. ACL toàn cluster scope qua Port_Group, không gắn trực tiếp Logical_Switch.
- **`Datapath_Binding` (SBDB)**: 1:1 mapping qua `external_ids:logical-switch=<ls_uuid>`. Đây là output đầu tiên của northd compile.
- **`Logical_Flow` (SBDB)**: 1:N mapping. Mỗi LS sinh ra hàng chục đến hàng trăm flow row.
- **`Logical_Router_Port`**: LSP type=router gắn vào LS để peer với LRP của Logical_Router (xem §13.2.2). Đây là cầu nối L2 ↔ L3.
- **OVS `Interface.external_ids:iface-id`**: gián tiếp, qua LSP. iface-id của OVS Interface khớp `name` của LSP, LSP thuộc LS, → LS biết VM nào đang on chassis nào.

Loose coupling (tham khảo, không cascade): `MAC_Binding` (ARP learning, scope SBDB), `FDB` (L2 learning trong runtime), `IGMP_Group` (mcast snoop result).

### Trường hợp sự cố thực tế

**Case OpenStack DC 2024-Q2: Logical_Switch ACL 0-priority drop chặn east-west production**. Tenant `prod-frontend` có một LS với ~150 LSP (web tier) và LS `prod-backend` với ~80 LSP (DB tier), nối qua một Logical_Router. CMS Neutron sync rule "default-deny + specific-allow" qua ML2-OVN driver: priority 1000 allow tcp.dst=8080 từ pg_frontend → pg_backend; priority 0 catch-all `(ip)` action drop.

**Symptom**: sau một deploy thay đổi placement, đột nhiên 100% kết nối từ frontend sang backend port 8080 timeout. Test ping ICMP cũng fail. Curl từ một VM frontend tới VIP backend chỉ thấy `Connection timed out`. Application monitoring trên backend thấy 0 packet ICMP/TCP đến.

**Investigation timeline**:

1. **Phát hiện** (T+0): user report production blackhole; on-call engineer chạy `ovs-ofctl dump-flows br-int | grep metadata=0x<ls-frontend-key> | grep table=9 | grep "actions=resubmit" | wc -l` thấy chỉ 12 row (không bình thường, kỳ vọng > 50 cho 150 LSP).
2. **Trace packet**: `ovn-trace ls-prod-frontend 'inport=="vm-web-01" && ip4.dst==10.10.20.5 && tcp.dst==8080'` thấy packet bị drop ở stage `ls_in_acl_eval` priority=0 match=(ip) action=drop.
3. **Đối chiếu NBDB**: `ovn-nbctl --columns=acls list Logical_Switch ls-prod-frontend` thấy có 1 row ACL trong set. Nhưng `ovn-nbctl acl-list ls-prod-frontend` chỉ in 1 dòng `from-lport 0 (ip) drop` — **thiếu rule priority=1000 allow**!
4. **Root cause**: deploy Neutron mới reset ACL set qua replace operation thay vì incremental add. Race với rule sync: catch-all drop được tạo trước, allow rule lẽ ra tạo sau nhưng sync API bị retry và bị duplicated drop, allow rule bị overwrite vì cùng priority=1000 collision với một ACL khác.
5. **Mitigation** (T+47 phút): manual `ovn-nbctl acl-add ls-prod-frontend from-lport 1000 'inport == @pg_frontend && ip4.dst == $ip_backend && tcp.dst == 8080' allow-related`; verify `ovn-trace` show packet pass.
6. **Postmortem fix**: Neutron driver patch để incremental update ACL set (compare-and-swap thay vì replace); thêm pre-deploy `ovn-trace` test trong CI để bắt rule loss trước khi rollout production.

**Bài học**: ACL set của Logical_Switch là **strong ref** với cascade — replace thay vì incremental rất nguy hiểm khi LS đang serve production. Luôn test `ovn-trace` cho ít nhất 3 represenative connection pattern sau mỗi ACL change.

Cross-reference Phần 17.0 (forensic case template) và Phần 13.3 (ACL/LB/NAT priority semantics).

| Failure mode | Tín hiệu | Cách xác định nguyên nhân |

| Hệ | Khái niệm tương đồng | Khác biệt chính |
|----|----------------------|-----------------|
| **VLAN truyền thống (802.1Q)** | Broadcast domain L2 | VLAN giới hạn 4096 ID; Logical_Switch dùng tunnel_key 24-bit (~16M) qua Geneve. VLAN per-host config; Logical_Switch declarative cluster-wide. |

## 13.2.Y Logical_Router như một bảng NBDB: tenant L3 router phân tán đào sâu

> Phần này đào sâu vào **chính bảng `Logical_Router`** (row trong OVN Northbound database), bổ sung cho phần dạy thực hành ở §13.2.2-13.2.4 và pipeline LR ở §13.16. Mục tiêu: kỹ sư hiểu thấu cấu trúc hàng NBDB, tập 11 cột của schema 22.03, vòng đời compile thành Datapath_Binding + 20 stage Logical_Flow LR_IN + 7 stage LR_OUT, cách phân biệt distributed router với gateway-redirected router, và cách tra ngược một sự cố routing/NAT về một row trong NBDB.

### Khái niệm cốt lõi

`Logical_Router` là **bảng nội trú trong OVN Northbound database (NBDB)** mô tả một **virtual L3 router** mà CMS (Neutron, kube-ovn, OpenStack) khai báo cho một tenant. Mỗi row đại diện cho một router logical: tất cả `Logical_Router_Port` (LRP) thuộc row này nằm trong cùng routing domain, chia sẻ cùng FIB (forwarding information base) sinh ra từ `static_routes` + `policies`, cùng tập NAT rule, cùng tập Load_Balancer, cùng `options` runtime. Loại: **OVSDB table**, root table (`isRoot=true` trong `ovn-nb.ovsschema`), persistent (replicated qua Raft cluster NBDB), declarative input cho ovn-northd. Khác với một router truyền thống (một thiết bị physical hoặc một VM gateway), Logical_Router **phân tán mặc định**: pipeline LR_IN/LR_OUT chạy local trên mọi chassis có VM thuộc LR, không qua một hop trung tâm. Khi cần đi ra ngoài (north-south với NAT centralized), một LRP cụ thể được pinned vào một chassis qua `options:chassis=<uuid>`, tạo thành **gateway router** với chassisredirect port; lúc đó traffic north-south qua chassis được pin, traffic east-west vẫn distributed.

### Lịch sử + bối cảnh ra đời

Bảng `Logical_Router` xuất hiện cùng OVN 2.4 (2015) đồng thời với `Logical_Switch`, trong refactor của Russell Bryant (Red Hat) + Justin Pettit (VMware/Nicira) chuyển kiến trúc Nicira NSX OpenStack plugin sang OVN. Trước đó, Neutron L3 agent chạy mỗi tenant router trong một network namespace trên một network node centralized (HA Active/Standby qua keepalived hoặc DVR semi-distributed); scale bị giới hạn vì mọi traffic L3 đi qua một nút, và cấu hình route/NAT phải đẩy qua RPC. OVN giải quyết bằng cách compile mỗi Logical_Router thành 27 stage pipeline cài lên mọi chassis: traffic east-west L3 không bao giờ rời chassis nguồn nếu đích cùng chassis; chỉ traffic ra external mới qua gateway chassis. OVN 21.03 thêm `policies` (Logical_Router_Policy) cho policy-based routing. OVN 21.06 overhaul distributed gateway router: chính thức hóa `chassisredirect` Port_Binding type + `HA_Chassis_Group` cho multi-chassis failover. OVN 22.03 (curriculum baseline) ổn định 11 cột schema (verified `ovn-nb.ovsschema` branch-22.03). OVN 22.06 mở rộng ECMP multi-nexthop. OVN 24.03 thêm `options:dynamic_neigh_routers` + improved BFD on LRP.

### Vị trí trong kiến trúc

Bảng nằm ở **layer NBDB**, ngang hàng `Logical_Switch` ở input layer. Scope: **per-cluster**. Mỗi row Logical_Router sinh ra **đúng một row Datapath_Binding** trong SBDB (`ovn-sbctl find Datapath_Binding external_ids:logical-router=<lr_uuid>`) + **trung bình 100-500 row Logical_Flow** (tuỳ số LRP, static_routes, policies, NAT, LB) phân bổ vào 20 stage ingress + 7 stage egress. Cấp độ tác động: thay đổi một row Logical_Router (ví dụ thêm `static_routes` hoặc `nat`) làm ovn-northd recompute datapath đó trong < 200ms (incremental compile từ OVN 21.03+); ovn-controller trên mọi chassis sau đó re-cài flow OpenFlow trong < 1 giây qua I-P engine. Tunnel_key của Datapath_Binding LR chia sẻ namespace với LS (1-16777215, unique trong cluster); không có collision vì cùng allocator.

### Vai trò + nhiệm vụ

Logical_Router gánh **năm vai trò** trong kiến trúc OVN:

1. **Định danh L3 routing domain**: cột `name` (string, không unique) làm identifier human-readable; `_uuid` là khoá duy nhất. Tất cả LRP trong row chia sẻ cùng FIB.
2. **Container cho LRP**: cột `ports` (set strong ref tới `Logical_Router_Port`) cascade delete — xoá LR xoá luôn mọi LRP. Mỗi LRP có MAC + một hoặc nhiều CIDR; LRP peer với LSP type=router của một Logical_Switch.
3. **Anchor cho L3 policy**: cột `static_routes` (set strong ref `Logical_Router_Static_Route`) cho route tĩnh (ECMP nếu nhiều route cùng prefix), `policies` (set strong ref `Logical_Router_Policy`) cho policy-based routing (priority + match + action), `nat` (set strong ref `NAT`) cho SNAT/DNAT/dnat_and_snat rule, `load_balancer` (set weak ref) + `load_balancer_group` cho north-south LB, `copp` cho control plane policing.
4. **Truyền tham số runtime cho northd**: cột `options` (map string→string) chứa `chassis=<uuid>` (pin LR thành gateway trên chassis cụ thể), `dynamic_neigh_routers="true"` (dùng dynamic neighbor learning thay broadcast ARP), `mac_binding_age_threshold`, `lb_force_snat_ip`, `always_learn_from_arp_request`. Northd đọc map này để quyết định compile pipeline distributed hay redirect.
5. **Báo metadata cho CMS**: cột `external_ids` (map) chứa `neutron:router_name`, `kube-ovn:vpc`, hoặc tương tự — CMS dùng để tra ngược từ row OVN về object CMS gốc.

### Vì sao sinh ra (động cơ thiết kế)

Trước OVN 2.4, kiến trúc Neutron L3 agent (Folsom 2012-Liberty 2015) chạy mỗi tenant router trong một Linux network namespace trên một network node centralized; HA qua VRRP/keepalived (Active/Standby). Vấn đề: (a) **Single-point bottleneck** — mọi packet east-west L3 cũng đi qua network node, dễ saturate NIC; (b) **State sync chậm khi failover** — Active chết, Standby mất 3-5 giây để take over IP ảo, mọi connection bị drop; (c) **DVR (Distributed Virtual Routing) phức tạp** — Juno 2014 thêm DVR nhưng vẫn cần snat-namespace centralized cho egress; cấu hình rườm rà. OVN giải quyết bằng cách **đẩy state ra OVSDB Raft + để northd compile pipeline distributed mặc định**: pipeline LR_IN/LR_OUT cài lên mọi chassis; traffic east-west không qua hop trung tâm; chỉ NAT external qua chassisredirect port pinned. Logical_Router là **abstraction tenant-facing duy nhất** mà CMS biết — CMS không cần biết Datapath_Binding, Logical_Flow, hay chassisredirect, chỉ tạo Logical_Router + LRP + nat + static_routes. Mô hình này scale lên hàng nghìn router per cluster với hop count L3 không tăng (so sánh: Neutron L3 agent giới hạn ~200-500 router per network node trước khi quá tải).

### Vấn đề mà nó giải quyết

Bảng này giải quyết **năm vấn đề kỹ thuật** rạch ròi: (1) **Distributed L3 forwarding mặc định**: pipeline 20 stage LR_IN cài trên mọi chassis; packet east-west L3 không bao giờ rời chassis nếu đích cùng chassis hoặc cùng cluster qua tunnel direct; (2) **Centralized NAT khi cần**: gắn `options:chassis=<uuid>` vào một LRP gateway → northd sinh chassisredirect port; SNAT/DNAT rule chỉ apply trên chassis đó để giữ conntrack state nhất quán; (3) **Cascade lifecycle nhất quán**: xoá row LR xoá tất cả LRP, static_routes, policies, nat thuộc về nó qua strong ref; CMS không cần pseudo-transaction; (4) **ECMP multi-nexthop tự nhiên**: thêm hai static_routes cùng prefix, khác nexthop → northd compile hai flow vào stage `lr_in_ip_routing_ecmp` table ID 12 (xem §13.16) với 5-tuple hash chọn path; (5) **Policy-based routing declarative**: thêm row Logical_Router_Policy với priority + match (ví dụ `ip4.src == 10.0.1.0/24 && ip4.dst == 8.8.8.8`) + action (`reroute 192.168.1.1`) → compile vào `lr_in_policy` table ID 13.

### Tầm quan trọng

Đây là **bảng cốt lõi tier 1 của NBDB**, ngang hàng với `Logical_Switch`. Lý do: 100% workflow tenant L3 OVN bắt đầu bằng `ovn-nbctl lr-add`; 100% feature OVN cao cấp ở scope L3 (NAT stateful, north-south LB, static route + ECMP, policy-based routing, BFD, gateway HA) đều thêm cột hoặc field option vào Logical_Router chứ không tạo bảng mới. Kỹ sư không hiểu Logical_Router sẽ không tạo, không debug, không scale được bất kỳ tenant L3 nào trong OVN. Trong 320+ keyword của REF, Logical_Router nằm trong nhóm 15 keyword cornerstone OVN (cùng `ovn-northd`, `ovn-controller`, NBDB, SBDB, Logical_Flow, Datapath_Binding, Logical_Switch, Port_Binding, ACL, Load_Balancer, NAT, Geneve TLV, ovn-trace, ovn-detrace).

### Cơ chế hoạt động (chu kỳ compile một row)

Một row Logical_Router đi qua **bốn giai đoạn** trong vòng đời:

1. **Khai báo từ CMS**: CMS gọi `ovn-nbctl lr-add lr-tenant1` → NBDB transaction tạo row mới với `name="lr-tenant1"`, `enabled=true` (mặc định), các set khác `[]`. Transaction atomic; row visible trong < 50ms qua NBDB Raft replicate.
2. **Compile sang Datapath_Binding**: ovn-northd phát hiện row mới qua I-P engine input change. Hàm tạo `struct ovn_datapath` cho row LR (chia sẻ allocator với LS), allocate `tunnel_key`. Sau đó `build_datapaths` ghi vào SBDB: `INSERT INTO Datapath_Binding (tunnel_key=<allocated>, external_ids={'name': 'lr-tenant1', 'logical-router': '<lr_uuid>'})`. Kết quả: 1 row Datapath_Binding tương ứng 1:1 với row Logical_Router.
3. **Compile sang Logical_Flow rows**: northd chạy chuỗi hàm `build_*_for_lrouter` để sinh Logical_Flow row cho từng stage trong 20 LR_IN + 7 LR_OUT. Mỗi LRP, static_route, policy, NAT rule, LB gắn vào LR đều spawn flow tương ứng. Tổng số: 100-1000 row Logical_Flow per LR thông thường (cấp số nhân theo số LRP × số route × số NAT × số LB).
4. **Cài flow trên chassis**: SBDB Raft replicate Logical_Flow → ovn-controller trên mọi chassis subscribe → mỗi controller chạy `lflow_run` translate Logical_Flow expression sang OpenFlow rule cụ thể, cài lên `br-int` qua `ofctrl_put`. LR_IN stage 0-19 cài vào OF table 8-27, LR_OUT stage 0-6 cài vào OF table 40-46 (xem §13.16 chi tiết). Nếu LR có gateway port (`options:chassis=<uuid>`), riêng chassis được pin sẽ thêm flow cho chassisredirect; chassis khác chỉ có flow east-west distributed.

Invariant: tại mọi thời điểm, một row Logical_Router tồn tại trong NBDB tương ứng đúng **một** row Datapath_Binding trong SBDB; xoá LR qua `lr-del` cascade xoá LRP + static_routes + policies + nat + Datapath_Binding + Logical_Flow trong cùng round compile.

### Phân loại cột

11 cột của Logical_Router (verified `ovn-nb.ovsschema` branch-22.03 [GitHub raw](https://raw.githubusercontent.com/ovn-org/ovn/branch-22.03/ovn-nb.ovsschema)) chia thành **năm nhóm** theo vai trò ngữ nghĩa:

| Nhóm | Cột | Type | Required | Ghi chú |
|------|-----|------|----------|---------|
| Định danh | `name` | string | Có (mặc định "") | CMS-friendly, không unique |
| Định danh | `enabled` | boolean | Không (mặc định true) | false → mọi LRP của LR drop traffic |
| Container | `ports` | set strong ref → Logical_Router_Port | Không (mặc định []) | Cascade delete LRP |
| Routing | `static_routes` | set strong ref → Logical_Router_Static_Route | Không | Route tĩnh + ECMP |
| Routing | `policies` | set strong ref → Logical_Router_Policy | Không (21.03+) | Policy-based routing |
| Policy gắn LR | `nat` | set strong ref → NAT | Không | SNAT/DNAT/dnat_and_snat |
| Policy gắn LR | `load_balancer` | set weak ref → Load_Balancer | Không | North-south LB legacy |
| Policy gắn LR | `load_balancer_group` | set ref → Load_Balancer_Group | Không (24.03+) | Group LB modern |
| Policy gắn LR | `copp` | weak ref → Copp (max 1) | Không (22.03+) | Control plane meter |
| Tham số dịch vụ | `options` | map string→string | Không | chassis=<uuid>, dynamic_neigh_routers, mac_binding_age_threshold |
| Metadata CMS | `external_ids` | map string→string | Không | CMS-tracked ID |

Cột `enabled` là khác biệt rõ rệt với `Logical_Switch` (LS không có cột này). Set `enabled=false` qua `ovn-nbctl set Logical_Router lr-X enabled=false` → northd compile flow drop ở stage `lr_in_admission` cho mọi packet enter LRP của LR đó; dùng để maintenance không cần xoá LR. Khoá tra cứu duy nhất là `_uuid`.

### Cách kỹ sư vận hành thành thạo

Bốn kỹ năng nền tảng kỹ sư phải có với Logical_Router:

1. **Đọc một row đầy đủ**: `sudo ovn-nbctl --columns=name,enabled,ports,static_routes,policies,nat,load_balancer,options list Logical_Router <lr_uuid_or_name>` cho thấy 8 cột chính + size từng set. Hoặc `sudo ovn-nbctl lr-list` lấy tổng quan + `sudo ovn-nbctl lr-route-list lr-X` + `ovn-nbctl lr-nat-list lr-X` để in chi tiết route + NAT.
2. **Xác minh compile xuống SBDB**: `sudo ovn-sbctl find Datapath_Binding external_ids:logical-router=<lr_uuid>` xác nhận có Datapath_Binding tương ứng. Nếu không thấy → northd chưa kịp compile hoặc lỗi ovn-northd.log.
3. **Đếm pipeline output theo stage LR_IN/LR_OUT**: `sudo ovn-sbctl lflow-list <lr_name> | grep -c lr_in_` đếm flow ingress; `grep -c lr_out_` đếm egress. Một LR không LRP chỉ có ~30 flow stub; LR với 3 LRP + 5 static_route + 2 NAT có ~150-250 flow; LR với 20 LRP + 100 route + 50 NAT có > 2000 flow.
4. **Quyết định khi nào dùng skill nào**: CMS báo "VM L2 thông nhau nhưng cross-subnet không thông" → bắt đầu bằng `lr-list` + `lr-route-list` (skill 1) để xác nhận route đúng; CMS báo "north-south NAT mất kết nối sau failover" → kiểm `options:chassis` + Port_Binding chassisredirect (xem §13.5.1b + §13.11); nghi northd treo cho LR cụ thể → kiểm Datapath_Binding (skill 2).

### Quy trình sử dụng (workflow chuẩn)

Workflow provisioning một tenant L3 với 3 subnet + NAT external qua Logical_Router gồm bảy bước, mỗi bước có lệnh xác minh:

1. **CMS tạo Logical_Router trong NBDB**: `ovn-nbctl lr-add lr-tenant1`. Xác minh: `ovn-nbctl lr-list | grep lr-tenant1` thấy UUID + tên.
2. **Tạo LRP cho mỗi subnet**: `ovn-nbctl lrp-add lr-tenant1 lrp-web aa:bb:cc:00:01:fe 10.0.1.254/24`; tương tự `lrp-app`, `lrp-db`. Xác minh: `ovn-nbctl --columns=ports list Logical_Router lr-tenant1` thấy 3 ref.
3. **Peer LRP với LSP type=router của LS**: trên mỗi LS tương ứng, tạo LSP type=router options:router-port=lrp-X (xem §13.2.3). Xác minh: `ovn-nbctl ls-show ls-web` thấy port type=router.
4. **Thêm static_routes nếu cần**: `ovn-nbctl lr-route-add lr-tenant1 0.0.0.0/0 192.168.1.1` (default route ra gateway external). Xác minh: `ovn-nbctl lr-route-list lr-tenant1` thấy entry.
5. **Thêm NAT rule nếu cần north-south**: `ovn-nbctl lr-nat-add lr-tenant1 snat 192.168.1.10 10.0.0.0/16` (SNAT pool internal sang external IP); hoặc `ovn-nbctl lr-nat-add lr-tenant1 dnat_and_snat 192.168.1.20 10.0.1.5` (floating IP one-to-one). Xác minh: `ovn-nbctl lr-nat-list lr-tenant1` thấy entry.
6. **Pin gateway nếu cần centralized NAT**: `ovn-nbctl lrp-set-gateway-chassis lrp-web hv-gw1 50` (chassis priority 50). Xác minh: `ovn-sbctl find Port_Binding type=chassisredirect` thấy `cr-lrp-web` với chassis ref.
7. **Chờ northd compile + verify**: thường < 200ms. Xác minh: `ovn-sbctl lflow-list lr-tenant1 | grep lr_in_dnat` thấy flow NAT; `ovn-trace lr-tenant1 'inport=="lrp-web" && ip4.src==10.0.1.5 && ip4.dst==8.8.8.8'` thấy packet pass tới `lr_out_snat`.

Best practice production: **luôn dùng `dynamic_neigh_routers="true"` cho LR có > 100 LRP** để giảm broadcast ARP; **luôn dùng HA_Chassis_Group thay vì pin một chassis duy nhất** cho gateway production (xem §13.6 + §13.11). Anti-pattern: một Logical_Router cho toàn DC với hàng nghìn LRP → northd compile chậm > 30 giây vì pipeline LR_IN explosion với mọi LRP × stage; tách thành nhiều LR theo tenant + transit LR liên kết.

### Khi xảy ra sự cố (reverse symptom map)

Tám symptom thường gặp trỏ về Logical_Router:

| Symptom | Cột Logical_Router cần kiểm | Diagnostic flow |
|---------|------------------------------|-----------------|
| "VM cross-subnet cùng tenant không thông" | `ports`, `enabled` | `ovn-nbctl --columns=enabled,ports list Logical_Router lr-X`; nếu enabled=false → set true |
| "Default route không hoạt động" | `static_routes` | `ovn-nbctl lr-route-list lr-X` xem có 0.0.0.0/0 không; `ovn-trace` thấy drop ở `lr_in_ip_routing` |
| "ECMP không phân tải" | `static_routes` (multi-nexthop) | `ovn-sbctl lflow-list lr-X | grep lr_in_ip_routing_ecmp` xem có flow ECMP không |
| "Policy-based routing bị bỏ qua" | `policies` | `ovn-nbctl lr-policy-list lr-X`; verify priority + match expression đúng |
| "SNAT external không apply" | `nat`, `options:chassis` | `ovn-nbctl lr-nat-list lr-X`; nếu thiếu gateway pin → SNAT chỉ apply chassis có VM nguồn (distributed SNAT = anti-pattern) |
| "DNAT floating IP failover chậm" | `nat`, LRP `gateway_chassis` hoặc `ha_chassis_group` | `ovn-sbctl find Port_Binding type=chassisredirect`; check HA_Chassis_Group priority |
| "L3 traffic bị blackhole sau migrate VM" | (LSP `external_ids`) + LR `static_routes` | conntrack state mất khi VM cross-chassis migrate; kiểm `ovn-trace` cả chiều forward và reverse |
| "northd compile chậm cho LR lớn" | Số LRP, số route, số NAT | Tách LR thành nhiều LR + transit LR liên kết |

Cross-link: §13.1 (NBDB+SBDB architecture), §13.3 (ACL/LB/NAT đào sâu), §13.5 (Port_Binding chassisredirect), §13.6 (HA_Chassis_Group + BFD), §13.7 (ovn-controller dịch lflow → OpenFlow), §13.11 (distributed gateway router), §13.16 (logical pipeline table ID map LR_IN 8-27 + LR_OUT 40-46), §13.19 (pipeline stage catalog), §17.0 (forensic case multichassis).

### Liên quan mật thiết

Bảy thành phần coupling chặt với Logical_Router (đổi cái nào thì phải xét lại Logical_Router):

- **`Logical_Router_Port`** (LRP): chứa trong cột `ports` strong ref. Mỗi LRP thuộc đúng 1 LR; xoá LR cascade xoá LRP. LRP có MAC + CIDR + có thể `gateway_chassis` ref.
- **`Logical_Router_Static_Route`**: chứa trong cột `static_routes` strong ref. Cascade xoá. ECMP khi nhiều route cùng `ip_prefix` khác `nexthop`.
- **`Logical_Router_Policy`**: chứa trong cột `policies` strong ref (21.03+). Cascade xoá. Priority + match + action.
- **`NAT`**: chứa trong cột `nat` strong ref. Type: `snat`, `dnat`, `dnat_and_snat`. Cascade xoá khi LR xoá.
- **`Logical_Switch_Port`** type=router: gián tiếp, mỗi LRP peer với một LSP type=router của một Logical_Switch. Cầu nối L3 ↔ L2.
- **`Datapath_Binding` (SBDB)**: 1:1 mapping qua `external_ids:logical-router=<lr_uuid>`. Output đầu tiên của northd compile.
- **`Port_Binding` type=chassisredirect** (SBDB): xuất hiện khi LRP có `gateway_chassis` hoặc `ha_chassis_group`; pin north-south traffic qua chassis cụ thể.

Loose coupling (tham khảo, không cascade): `MAC_Binding` (ARP learning từ next-hop external), `BFD` row (per LRP gateway monitoring), `HA_Chassis_Group` (failover gateway).

```shell
Datapath: "lr-tenant1" (ff22ee33-...)  Pipeline: egress
  table=1 (lr_out_undnat          ), priority=100, match=(ip && ip4.src == 10.0.1.5 && outport == "lrp-web"), action=(ct_dnat;)
  table=3 (lr_out_snat            ), priority=153, match=(ip && ip4.src == 10.0.0.0/16 && outport == "lrp-web"), action=(ct_snat(192.168.1.10);)
  table=6 (lr_out_delivery        ), priority=100, match=(outport == "lrp-web"), action=(output;)
```

Tiết lộ: 20 ingress stage + 7 egress stage compiled từ row LR. Stage `lr_in_dnat` ID 6 + `lr_in_ip_routing` ID 11 + `lr_in_arp_resolve` ID 15 cho thấy DNAT (floating IP), routing với default route, ARP resolve next-hop. Stage `lr_out_snat` ID 3 cho SNAT pool.

**Phương pháp 4, ovs-ofctl dump-flows trên chassis**:

```shell
$ sudo ovs-ofctl -O OpenFlow15 dump-flows br-int | grep "metadata=0x7" | grep "table=14" | head -5
 cookie=0xff22ee33, table=14, priority=49, metadata=0x7, ip, nw_dst=0.0.0.0/0 actions=load:0xc0a80101->NXM_NX_REG0[],load:0xc0a8010a->NXM_NX_REG1[],...,resubmit(,15)
```

Tiết lộ: flow OpenFlow thực tế trên chassis. `metadata=0x7` chính là `tunnel_key=7` của Datapath_Binding của Logical_Router; table=14 = 8 + LR_IN_IP_ROUTING (logical 11) - không, chính xác hơn: với baseline 22.03 mapping (xem §13.16 chi tiết), LR_IN logical stage cộng với offset của OF table chia sẻ với LS_IN. Khi cùng `metadata=0x7` (LR datapath) và LS có `metadata=0x4` (LS datapath khác), cùng OF table 14 giữ flow của cả hai, phân biệt qua match `metadata`.

### Trường hợp sự cố thực tế

**Case OpenStack DC 2024-Q3: Logical_Router gateway redirect chassis CPU 100% spike trong failover**. Cluster OVN OpenStack 16 hypervisor + 2 gateway chassis (hv-gw1 active, hv-gw2 standby qua HA_Chassis_Group). Mỗi tenant có một Logical_Router với 2-5 LRP + ~10-30 NAT rule (floating IP + SNAT pool). Tổng cluster: ~80 LR, ~400 LRP, ~600 NAT rule.

**Symptom**: hv-gw1 down lúc 14:32 do hardware fault NIC. HA_Chassis_Group failover sang hv-gw2 trong < 2 giây (BFD detect + Port_Binding chassisredirect cập nhật chassis ref). Từ 14:32:05, hv-gw2 CPU jump từ 25% lên 100% (8 core hết); ovn-controller log "I-P engine recompute lasted 12000ms" lặp; mọi north-south traffic timeout ~30 giây.

**Investigation timeline**:

1. **Phát hiện** (T+0): Prometheus alert "node CPU > 95% cho 60s" trên hv-gw2; on-call engineer SSH vào, `top` thấy `ovn-controller` chiếm 6 core full.
2. **Trace nguyên nhân CPU**: `ovs-appctl -t ovn-controller debug/status` thấy queue `lflow_run` backlog ~5000 entry; `ovn-appctl -t ovn-controller inc-engine/show-stats` thấy "physical_flow_changes" tăng vọt mỗi 5 giây.
3. **Đối chiếu SBDB**: `ovn-sbctl --columns=chassis list Port_Binding type=chassisredirect | wc -l` thấy 80 row chassisredirect cùng chassis ref hv-gw2 (đúng vì failover); nhưng `ovn-sbctl lflow-list | wc -l` cho cluster tăng từ ~50000 lên ~80000 sau failover.
4. **Root cause**: stage `lr_in_gw_redirect` (T18) khi gateway pin trên một chassis sinh flow redirect cho mọi LRP ra chassisredirect; với 400 LRP × ~10 NAT × stage interaction, tổng flow ingress trên hv-gw2 vượt cấu hình OF table size limit cho `br-int`. Mỗi recompute round, ovn-controller phải retry cài lại toàn bộ batch flow vì OFPT_FLOW_MOD bundle hit timeout (5 giây). Compounded với việc hv-gw2 (loại CPU tier thấp hơn hv-gw1) không đủ throughput.
5. **Mitigation** (T+25 phút): tạm thời tắt `dynamic_neigh_routers` trên 50% LR ít quan trọng nhất qua `ovn-nbctl set Logical_Router lr-X options:dynamic_neigh_routers=false` → giảm flow cho stage `lr_in_arp_resolve` ~30%. CPU hv-gw2 giảm xuống ~70% sau 5 phút.
6. **Postmortem fix** (T+3 ngày): chia 80 LR thành 4 nhóm gateway, mỗi nhóm pin một HA_Chassis_Group khác (4 cặp gateway thay vì 1 cặp); tăng cấu hình `ovs-vsctl set Open_vSwitch . other_config:max-idle=...` để giảm pressure flow cache; bài học chính: gateway concentration là anti-pattern khi LR > 50.

**Bài học**: gateway pin (`options:chassis` hoặc HA_Chassis_Group) tạo single-point load tập trung; Logical_Router distributed mặc định cho east-west scale tốt nhưng north-south redirect không scale linear với số LR. Luôn benchmark flow count trên gateway candidate trước production rollout.

Cross-reference §17.0 (forensic case template) và §13.11 (distributed gateway router) + §13.18 (`REGBIT_NAT_REDIRECT` chi tiết).

### Bài tập synthetic (lab)

**Lab L13.2.Y-1: 2-LS + 1-LR end-to-end với ovn-trace pipeline**.

**Mục tiêu**: tạo 2 Logical_Switch + 1 Logical_Router liên kết, verify east-west L3 traffic qua `ovn-trace` thấy đầy đủ stage LR_IN/LR_OUT.

**Topology**:
- 2 chassis: `hv1` (192.168.50.11) + `hv2` (192.168.50.12), Geneve tunnel mesh.
- 2 Logical_Switch: `ls-web` CIDR 10.10.1.0/24 + `ls-db` CIDR 10.10.2.0/24.
- 1 Logical_Router `lr-tenant` với 2 LRP: lrp-web (10.10.1.254/24) + lrp-db (10.10.2.254/24).
- 4 LSP: vm-web1 (hv1, 10.10.1.10) + vm-web2 (hv2, 10.10.1.11) trên ls-web; vm-db1 (hv1, 10.10.2.10) + vm-db2 (hv2, 10.10.2.11) trên ls-db.

**Steps**:

```bash
# Step 1: tạo LS + LR + LRP + peer
[central]# ovn-nbctl ls-add ls-web
[central]# ovn-nbctl ls-add ls-db
[central]# ovn-nbctl lr-add lr-tenant
[central]# ovn-nbctl lrp-add lr-tenant lrp-web aa:bb:cc:00:01:fe 10.10.1.254/24
[central]# ovn-nbctl lrp-add lr-tenant lrp-db  aa:bb:cc:00:02:fe 10.10.2.254/24
[central]# ovn-nbctl lsp-add ls-web ls-web-lrp \
    -- set Logical_Switch_Port ls-web-lrp type=router options:router-port=lrp-web addresses=router
[central]# ovn-nbctl lsp-add ls-db ls-db-lrp \
    -- set Logical_Switch_Port ls-db-lrp type=router options:router-port=lrp-db addresses=router

# Step 2: tạo 4 LSP cho VM
[central]# for spec in "vm-web1 ls-web aa:bb:cc:00:00:01 10.10.1.10" \
                       "vm-web2 ls-web aa:bb:cc:00:00:02 10.10.1.11" \
                       "vm-db1  ls-db  aa:bb:cc:00:00:11 10.10.2.10" \
                       "vm-db2  ls-db  aa:bb:cc:00:00:12 10.10.2.11"; do
    set -- $spec
    ovn-nbctl lsp-add $2 $1
    ovn-nbctl lsp-set-addresses $1 "$3 $4"
    ovn-nbctl lsp-set-port-security $1 "$3 $4"
done

# Step 3: xác minh compile
[central]# ovn-sbctl find Datapath_Binding external_ids:logical-router=$(ovn-nbctl --bare --columns=_uuid list Logical_Router lr-tenant)
# Kỳ vọng: 1 row Datapath_Binding với tunnel_key allocated cho LR.

[central]# ovn-sbctl lflow-list lr-tenant | grep -c lr_in_
# Kỳ vọng: ~50-80 flow cho ingress (2 LRP + standard stage).

[central]# ovn-sbctl lflow-list lr-tenant | grep -c lr_out_
# Kỳ vọng: ~10-20 flow cho egress.

# Step 4: plug VM (chỉ vm-web1 + vm-db1 cho compact lab; vm-web2 + vm-db2 lặp lại trên hv2)
[hv1]# ip netns add vm-web1
[hv1]# ip link add tap-web1 type veth peer name eth0 netns vm-web1
[hv1]# ip link set tap-web1 up
[hv1]# ovs-vsctl add-port br-int tap-web1 -- set Interface tap-web1 external_ids:iface-id=vm-web1
[hv1]# ip netns exec vm-web1 ip link set eth0 address aa:bb:cc:00:00:01
[hv1]# ip netns exec vm-web1 ip addr add 10.10.1.10/24 dev eth0
[hv1]# ip netns exec vm-web1 ip link set eth0 up
[hv1]# ip netns exec vm-web1 ip route add default via 10.10.1.254
# Lặp lại cho vm-db1 trên hv1, vm-web2 + vm-db2 trên hv2.

# Step 5: verify Port_Binding
[central]# ovn-sbctl --columns=logical_port,chassis,up list Port_Binding | grep -E "vm-(web|db)"
# Kỳ vọng: 4 row với chassis của hv1 hoặc hv2 + up=true.

# Step 6: ping cross-LS qua LR (east-west L3 distributed)
[hv1]# ip netns exec vm-web1 ping -c 3 10.10.2.10  # vm-web1 (ls-web, hv1) → vm-db1 (ls-db, hv1)
# Kỳ vọng: 3 packet, RTT < 2ms (cùng chassis qua LR distributed).
[hv1]# ip netns exec vm-web1 ping -c 3 10.10.2.11  # vm-web1 → vm-db2 (hv2, qua tunnel)
# Kỳ vọng: 3 packet, RTT < 5ms (cross-chassis qua tunnel).

# Step 7: trace LR pipeline cho cross-chassis
[central]# ovn-trace lr-tenant 'inport=="lrp-web" && eth.src==aa:bb:cc:00:00:01 && eth.dst==aa:bb:cc:00:01:fe && ip4.src==10.10.1.10 && ip4.dst==10.10.2.11 && ip.ttl==64'
# Kỳ vọng: pipeline pass qua lr_in_admission → lr_in_lookup_neighbor → lr_in_ip_input → lr_in_ip_routing (match 10.10.2.0/24 → outport=lrp-db) → lr_in_arp_resolve → lr_in_arp_request → egress lr_out_delivery → output sang ls-db pipeline.

# Step 8: dump flow trên hv1 với metadata khớp tunnel_key của LR
[hv1]# tk=$(ovn-sbctl --bare --columns=tunnel_key find Datapath_Binding external_ids:logical-router=$(ovn-nbctl --bare --columns=_uuid list Logical_Router lr-tenant))
[hv1]# ovs-ofctl -O OpenFlow15 dump-flows br-int | grep "metadata=0x$(printf %x $tk)" | wc -l
# Kỳ vọng: > 80 flow chia khắp các table 8-27 (LR_IN) + 40-46 (LR_OUT).

# Cleanup
[central]# ovn-nbctl lr-del lr-tenant
[central]# ovn-nbctl ls-del ls-web; ovn-nbctl ls-del ls-db
```

**Falsification check**: set `ovn-nbctl set Logical_Router lr-tenant enabled=false` → ping cross-LS fail (drop ở `lr_in_admission`); set lại `enabled=true` → ping pass. Confirm hành vi `enabled` đúng.

**Deliverable**: log của 8 bước trên + output `ovn-trace` thấy packet đi qua đầy đủ 20 stage LR_IN (chỉ những stage có flow match) tới `lr_out_delivery` action `output;`.

### Lỗi thường gặp + tín hiệu chẩn đoán

| Failure mode | Tín hiệu | Cách xác định nguyên nhân |
|--------------|---------|---------------------------|
| **A. Cross-subnet không thông sau lr-add** | ping LR gateway IP fail | LRP thiếu peer LSP type=router; kiểm `ovn-nbctl ls-show ls-X` xem có port type=router không |
| **B. Default route không hoạt động** | `traceroute 8.8.8.8` chỉ tới gateway nội bộ rồi drop | `ovn-nbctl lr-route-list lr-X` thiếu 0.0.0.0/0; thêm `lr-route-add lr-X 0.0.0.0/0 <next-hop>` |
| **C. SNAT không apply cho external traffic** | tcpdump on external thấy src IP nội bộ thay vì SNAT IP | NAT rule có nhưng `options:chassis` chưa pin gateway → distributed SNAT không deterministic |
| **D. Floating IP DNAT mất kết nối sau VM migrate** | ping floating IP timeout sau live-migrate | conntrack zone không sync giữa chassis cũ và mới; kiểm `ovn-sbctl find Port_Binding type=chassisredirect` |
| **E. ECMP không phân tải đều** | `ovs-ofctl dump-flows | grep ecmp` chỉ thấy 1 path active | `lr_in_ip_routing_ecmp` table 12 stage ECMP cần multi-route cùng prefix; verify `lr-route-list` có ≥ 2 route cùng dst |
| **F. Policy-based routing không trigger** | packet đi route bình thường thay vì reroute | `ovn-nbctl lr-policy-list lr-X` xem priority + match; ovn-trace để xem có hit `lr_in_policy` không |
| **G. northd compile chậm cho LR > 100 LRP** | `ovn-northd.log` "took 30000ms" | Tách LR theo tenant + transit LR; bật `dynamic_neigh_routers=true` |
| **H. Gateway chassis CPU 100% sau failover** | hv CPU spike, `lflow_run` queue backlog | Quá nhiều LR pin cùng một gateway HA group; chia thành multiple HA group |

Diagnostic flow chuẩn cho symptom "VM trong tenant không liên lạc cross-subnet": (1) `ovn-nbctl lr-list` xác nhận LR tồn tại + `enabled=true`, (2) `lr-route-list` xem có route + default route không, (3) `ovn-trace` từ src đến dst xem drop ở stage nào, (4) đối chiếu rule trong NBDB với Logical_Flow trong SBDB stage tương ứng.

### So sánh với hệ khác

| Hệ | Khái niệm tương đồng | Khác biệt chính |
|----|----------------------|-----------------|
| **Cisco IOS VRF** | Routing instance độc lập | VRF gắn vào physical router; OVN Logical_Router distributed qua mọi chassis. VRF dùng `route-target` cho leaking; OVN dùng peer LRP + transit switch. |
| **Linux network namespace + iptables/iproute2** | Namespace = router instance | Per-host; OVN LR cluster-wide compile thành Logical_Flow cài trên mọi chassis. iptables NAT vs OVN NAT rule declarative. |
| **Neutron L3 agent (Legacy + DVR)** | Tenant router | L3 agent centralized hoặc DVR semi-distributed; OVN LR distributed mặc định + chỉ NAT centralized qua chassisredirect. OVN scale tốt hơn vì compile-time pipeline. |
| **VMware NSX-T T0/T1 logical router** | Direct analog (cùng team Nicira) | NSX-T tách T0 (provider) + T1 (tenant); OVN một cấp Logical_Router (có thể chain qua transit LS giả lập T0/T1). NSX-T edge cluster centralized cho NAT; OVN distributed + optional gateway. |
| **AWS VPC route table + NAT Gateway** | Routing + NAT cho VPC | AWS managed service ẩn implementation; OVN row Logical_Router exposed declarative. AWS NAT Gateway centralized; OVN gateway optional. |
| **Kubernetes OVN-Kubernetes** | Per-cluster join LR | Scope giống nhau (kube-ovn vẫn dùng OVN Logical_Router); distinction là CMS layer (kube-ovn vs Neutron). |

Khi migrate từ một hệ khác sang OVN: Cisco VRF → Logical_Router (1:1, route-target → peer LRP), Neutron L3 agent → Logical_Router (1:1, distributed default), NSX-T T0/T1 → 2 Logical_Router (T0 transit + T1 tenant) qua transit LS, AWS VPC route table → Logical_Router với static_routes copy.

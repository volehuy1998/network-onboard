## 11.0.1 Why overlay encapsulation is needed

VLAN 802.1Q limitations:

- **4096 VLAN limit** (12-bit VID). Not enough for cloud scale.
- **L2 flat across the physical network**. Spanning tree is fragile.
- **Not portable across L3**. A VLAN cannot cross a router boundary.

Overlay encapsulation solves the problem by wrapping the L2 frame inside an L3 packet:

- **24-bit VNI** gives 16.7 million segments.
- **Tunnel over IP routing**. The underlay only needs L3 reachability.
- **Portable across WAN and Internet**. Running over UDP makes NAT traversal feasible.

## 11.0.2 VXLAN, standardized as RFC 7348 in August 2014

VXLAN was co-authored by Cisco, VMware, and Broadcom, and standardized as RFC 7348 in August 2014.

Header format:

```
 ┌──────┬──────┬────────┬────────────┐
 │ Outer│ Outer│ UDP    │ VXLAN      │ Inner Ethernet Frame
 │ IP   │ UDP  │ dport  │ (8B)       │
 │ (20B)│ (8B) │ =4789  │ VNI 24-bit │
 └──────┴──────┴────────┴────────────┘
```

Overhead: 50 bytes (Ethernet 14 plus IP 20 plus UDP 8 plus VXLAN 8).

**VNI**: 24-bit, giving 16,777,216 tenant segments.
**UDP destination port 4789**. The source port is hashed from the inner flow to drive ECMP.

OVS:

```bash
[host]# ovs-vsctl add-port br-tun vxlan0 -- \
         set Interface vxlan0 type=vxlan \
         options:remote_ip=10.0.0.2 \
         options:key=flow \
         options:dst_port=4789
```

`options:key=flow` defers the choice of tunnel key to the OpenFlow action `set_tunnel:<vni>`. This enables multipoint tunneling.

## 11.0.3 Geneve (RFC 8926, 2020-11)

Geneve was co-proposed by VMware and Microsoft and standardized as RFC 8926 in November 2020.

Header:

```
 ┌──────┬──────┬────────┬──────────────┐
 │ Outer│ Outer│ UDP    │ Geneve       │ Inner Ethernet
 │ IP   │ UDP  │ dport  │ (8-66B)      │
 │ (20B)│ (8B) │ =6081  │ VNI + TLV    │
 └──────┴──────┴────────┴──────────────┘
```

Overhead: 50 to 66 bytes.

**Difference from VXLAN:** Geneve has **TLV options**, which extend the metadata arbitrarily. OVN uses TLV to carry the logical datapath ID, ingress port, and connection tracking state. OVN has used Geneve as the default tunnel encapsulation since 2015.

**UDP destination port 6081**.

```bash
[host]# ovs-vsctl add-port br-tun geneve0 -- \
         set Interface geneve0 type=geneve \
         options:remote_ip=10.0.0.2 \
         options:key=flow
```

## 11.0.4 STT: deprecated

STT (Stateless Transport Tunneling) was created by Nicira and VMware in 2012. It uses a TCP-like header so it can piggyback on the TSO offload of older NICs.

Drawbacks:

- Not a standard (no RFC). The IETF rejected it.
- Behaves like TCP but is not really TCP. This confuses middleboxes.
- Modern NICs have native VXLAN and Geneve offload from 2015 onward.

**OVS 2.x** has STT support. **OVS 3.0 and later** has STT **removed**. Do not use it for new deployments.

## 11.0.5 Comparison

*Table 11-1: VXLAN versus Geneve versus STT*

| Criterion | VXLAN | Geneve | STT |
|---|---|---|---|
| Standard | RFC 7348 (2014-08) | RFC 8926 (2020-11) | No RFC |
| Transport | UDP | UDP | TCP-like (custom) |
| Default port | 4789 | 6081 | 7471 |
| Overhead | 50 bytes | 50 to 66 bytes | 54+ bytes |
| Tenant ID | 24-bit VNI | 24-bit VNI plus TLV extensible | 64-bit context |
| TLV metadata | No | Yes (RFC 8926) | No |
| NIC offload | Most modern NICs | ConnectX-4 and later | Rare |
| OVS support | Stable | Stable, OVN default | Removed in OVS 3.0 and later |
| Recommendation 2026 | Enterprise DC | OVN, cloud-native | Do not use |

## 11.0.6 Multipoint tunneling with options:key=flow

Pattern: instead of creating N tunnel ports for N remote endpoints, use one tunnel port with `options:key=flow` plus the action `set_tunnel:`:

```bash
[host]# ovs-vsctl add-port br-tun ovn-geneve -- \
         set Interface ovn-geneve type=geneve \
         options:remote_ip=flow \
         options:key=flow

[host]# ovs-ofctl add-flow br-tun \
         "table=0,in_port=1,dl_dst=aa:bb:cc:00:00:01,\
          actions=set_tunnel:1000,set_field:10.0.0.2->tun_dst,output:ovn-geneve"
```

`options:remote_ip=flow` lets the controller decide the destination IP per flow. One tunnel port serves an unlimited number of remote chassis. This is the model OVN uses.

## 11.0.7 Geneve packet format byte-by-byte

Section 11.0.3 introduced Geneve at a high level. This section walks every byte of the Geneve header so an engineer can read a packet capture directly.

### Outer headers (48 bytes for an IPv4 underlay)

```
Byte:    0      6      12     14    18    20      24      28
         ┌──────┬──────┬──────┬─────┬─────┬───────┬───────┬───────────────────┐
Layer 2: │ dst  │ src  │ ether│ VLAN│ ... │       │       │                   │
         │ MAC  │ MAC  │ Type │ tag │     │       │       │                   │
         │ 6B   │ 6B   │ 0x800│(opt)│     │       │       │                   │
         └──────┴──────┴──────┴─────┴─────┴───────┴───────┴───────────────────┘
                                                       
Byte:    14            34
         ┌─────────────────────────────────────────────────────┐
Layer 3: │ Outer IPv4 header (20B without options)             │
IPv4:    │ ver=4 + IHL=5, ToS, total_len, ID, flags, frag_off, │
         │ TTL, proto=17(UDP), checksum,                       │
         │ outer_src_IP (4B), outer_dst_IP (4B)                │
         └─────────────────────────────────────────────────────┘

Byte:    34          42
         ┌──────────────────────┐
Layer 4: │ Outer UDP (8B)        │
UDP:     │ src_port (hash inner) │
         │ dst_port = 6081       │  ← Geneve magic number
         │ length, checksum       │
         └──────────────────────┘
```

Total outer bytes are 14 (Eth) plus 20 (IPv4) plus 8 (UDP), which equals **42 bytes**. With an IPv6 underlay the total is 14 plus 40 plus 8, which is 62 bytes.

### Geneve base header (8 bytes, RFC 8926 §3.4)

```
Byte:    42         43         44         46          48        49        50
         ┌──────────┬──────────┬──────────┬──────────┬─────────┬─────────────────┐
Geneve:  │ Ver (2)  │ Opt Len  │ O C R    │ Reserved │ Protocol│ VNI (3B)        │
         │ + Opt Len│ * 4 = total│ flags  │ (rsvd=0) │ Type    │ Virtual Network │
         │ Ver=0    │ TLV bytes│          │          │ 0x6558  │ Identifier      │
         │ (6 bit)  │ (6 bit)  │ 1+1+6 bit│ 6 bit    │ (Eth)   │ + 1B reserved   │
         └──────────┴──────────┴──────────┴──────────┴─────────┴─────────────────┘
```

| Field | Width | Description |
|-------|-------|-------|
| Version | 2 bit | Must be 0 (Geneve v0). |
| Opt Len | 6 bit | Number of 4-byte words of TLV options. Minimum 0, maximum 63 (252 bytes of TLV). |
| O (OAM) flag | 1 bit | Set to 1 if the packet is an OAM control-plane packet, not data. |
| C (Critical) flag | 1 bit | Set to 1 if a critical TLV is present. The receiver must understand all critical TLVs or drop the packet. |
| Reserved | 6 bit | Must be 0. |
| Protocol Type | 16 bit | Inner protocol. 0x6558 means Trans-Ether-Bridging (inner Ethernet). |
| VNI | 24 bit | Virtual Network Identifier (datapath ID). |
| Reserved | 8 bit | Must be 0. |

Example Geneve base header with VNI=1 and no TLV options:

```
00 00 65 58 00 00 01 00
^^ ^^ ^^^^^ ^^^^^^^^^^^
|  |  |     |
|  |  |     +-- VNI=1 (3 bytes) plus reserved (1 byte)
|  |  +-------- Protocol Type 0x6558 (Ethernet bridging)
|  +----------- Flags plus Reserved
+-------------- Version=0, Opt Len=0
```

### Geneve TLV options (variable, 0 to 252 bytes, RFC 8926 §3.5)

Each TLV has the following structure:

```
Byte:    0          2         3         4
         ┌──────────┬─────────┬─────────┬─────────────────┐
TLV:     │ Class    │ Type    │ R+Length│ Value (variable)│
         │ 16 bit   │ 8 bit   │ 3+5 bit │ 0-128 byte      │
         └──────────┴─────────┴─────────┴─────────────────┘
```

| Field | Width | Description |
|-------|-------|-------|
| Class | 16 bit | Vendor, organization, or IANA-allocated. The range 0x0000 to 0x00FF is reserved for IETF. The range 0x0100 to 0xFFFE is assigned per request. 0xFFFF is reserved. |
| Type | 8 bit | Vendor-specific. The high bit (bit 7) is the critical flag. |
| Reserved | 3 bit | Must be 0. |
| Length | 5 bit | Number of 4-byte words of Value. Range 0 to 31, which is 0 to 124 bytes. |
| Value | 0 to 124 bytes | TLV payload. |

### IANA Geneve TLV class registry

[IANA Geneve TLV Class Registry](https://www.iana.org/assignments/nvo3/nvo3.xhtml#geneve-option-class), updated 2026-04. There are about thirty registered vendors:

| Class | Owner | Use case |
|-------|-------|----------|
| 0x0000-0x00FF | IETF | Reserved |
| 0x0100 | LinuxFoundation | OVS reserved (legacy) |
| 0x0101 | Open vSwitch | OVS internal use |
| **0x0102** | **OVN** | **OVN logical metadata (datapath, inport, outport)** |
| 0x0103 | Cisco | Cisco-specific |
| 0x0104 | Oracle | Oracle Cloud |
| 0x0105 | Amazon | AWS VPC and Transit Gateway |
| 0x0107 | Microsoft | Azure ACS |
| 0x010A | NVIDIA | BlueField DPU |
| ... | ... | ... |

OVN uses class `0x0102`, officially reserved by IANA for the OVN project.

### OVN TLV class 0x0102 layout (from `controller/physical.c` in `branch-22.03`)

OVN encodes two logical metadata fields into a single TLV:

| Bit range | Field | Description |
|-----------|-------|-------|
| 0-31 | logical outport | Logical egress port number on the receiving chassis. |
| 32 | reserved | Must be 0. |
| 33-47 | logical inport | Logical ingress port number on the sending chassis (15-bit). |
| 48-63 | reserved | Must be 0. |

(Total payload of 8 bytes, which is 64 bits, or 2 words.)

The logical datapath (LS or LR) `tunnel_key` travels through the **VNI** of the Geneve base header (24 bits), not through the TLV.

The exact source code (verified at branch-22.03):

```c
// controller/physical.c, function put_encapsulation()
if (tun->type == GENEVE) {
    put_load(datapath->tunnel_key, MFF_TUN_ID, 0, 24, ofpacts);          // VNI 24-bit
    put_load(outport, mff_ovn_geneve, 0, 32, ofpacts);                    // TLV bits 0-31
    put_move(MFF_LOG_INPORT, 0, mff_ovn_geneve, 16, 15, ofpacts);         // TLV bits 16-30
}
```

(Cross-link to Part 13.7 §13.7.8 for Geneve TLV encoding details.)

### Decoding a real hex dump

Geneve packet with VNI=1 and OVN TLV class 0x0102 type=0x80, outport=2, inport=1:

```
0000  ff ff ff ff ff ff aa bb cc dd ee ff 08 00 45 00     # Outer Eth + IPv4 start
0010  00 6e 00 00 40 00 40 11 ... <src IP> <dst IP>      # IPv4 (20B)
0020  17 c1 17 c1 00 5a 00 00                            # UDP src/dst=6081, len=90
                                                           
0028  02 02 65 58 00 00 01 00                            # Geneve base
      ^^ ^^ ^^^^^ ^^^^^^^^^^^
      |  |  |     |
      |  |  |     +-- VNI=1 plus reserved
      |  |  +-------- Proto 0x6558 (Eth bridging)
      |  +----------- Flags=0, Reserved=0
      +-------------- Ver=0, Opt Len=2 (8 bytes of TLV)
                                                           
0030  01 02 80 02 00 00 00 02 00 00 00 01                # Geneve TLV
      ^^^^^ ^^ ^^ ^^^^^^^^^^^ ^^^^^^^^^^^
      |     |  |  |           |
      |     |  |  |           +-- Inport=1 (after move bits 16-30)
      |     |  |  +-------------- Outport=2 (32-bit, bits 0-31)
      |     |  +----------------- Length=2 (8-byte payload)
      |     +-------------------- Type=0x80 (Critical bit set)
      +-------------------------- Class=0x0102 (OVN reserved)

003C  <inner Ethernet frame>                             # Inner packet
```

Decode with `tcpdump`:

```bash
[host]# tcpdump -i eth0 -nn -X udp port 6081 | head -50
```

Wireshark has a built-in Geneve dissector (since Wireshark 2.4) and can auto-decode TLV class 0x0102 if the community-maintained OVN protocol dissector plugin is installed.

## 11.0.8 Exact header overhead math for OVN

OVN deployments use Geneve as the tunnel. The exact overhead is:

| Component | Bytes (IPv4 underlay) | Bytes (IPv6 underlay) |
|-----------|----------------------|----------------------|
| Outer Ethernet (no VLAN) | 14 | 14 |
| Outer Ethernet plus 802.1Q VLAN | 18 | 18 |
| Outer IPv4 (no options) | 20 | 0 |
| Outer IPv6 | 0 | 40 |
| UDP | 8 | 8 |
| Geneve base | 8 | 8 |
| Geneve TLV (OVN class 0x0102) | 8 | 8 |
| **Total IPv4 without VLAN** | **58** | |
| **Total IPv4 with VLAN** | **62** | |
| **Total IPv6 without VLAN** | | **78** |

Note: Part 0.2 §0.2.12 claimed "66 bytes" overhead. That number assumes 14 plus 20 plus 8 plus 8 plus 16 = 66, with 16 bytes of TLV (allowing OVN to add a second TLV for future use). The current correct number for **only class 0x0102** is 58 bytes. When sizing MTU, an engineer should use 58 for an IPv4 underlay without VLAN.

### MTU formulas

- **Underlay MTU = 1500** (default Ethernet, non-jumbo). The maximum OVN inner MTU is 1500 minus 58, which is **1442** bytes.
- **Underlay MTU = 9000** (jumbo frames). The maximum OVN inner MTU is 9000 minus 58, which is 8942 bytes.

When the underlay is 1500 and the workload (Pod or VM) also has MTU 1500, the result is fragmentation or PMTU drop. Part 11.1 covers PMTUD failure modes in depth.

## 11.0.9 Receiver-side decapsulation, table 0 on br-int

Part 0.2 §0.2.13 described decapsulation on Chassis Y at a high level. Here is the OpenFlow flow rule installed by `controller/physical.c put_remote_port_redirect()`:

```
table=0, in_port=<tunnel-OFPort>, tun_id=<tunnel_key>:
    move:NXM_NX_TUN_METADATA0[0..31]->NXM_NX_REG15[0..31]    # outport restore (32-bit)
    move:NXM_NX_TUN_METADATA0[16..30]->NXM_NX_REG14[0..14]   # inport restore (15-bit)
    move:NXM_NX_TUN_ID[0..23]->OXM_OF_METADATA[0..23]        # datapath restore
    resubmit(,<egress-table>)
```

This pattern is the mirror image of the encapsulation done at Chassis X. The three registers OVN uses are:

- `OXM_OF_METADATA[0..23]`: datapath logical ID, equal to the Geneve VNI.
- `NXM_NX_REG14[0..14]`: logical inport (15-bit).
- `NXM_NX_REG15[0..31]`: logical outport (32-bit, with reserved bits).

(`NXM_NX_TUN_METADATA0` is an OVS extension OXM. Each vendor can use multiple `tun_metadataN` slots. OVN uses only `tun_metadata0`.)

## 11.0.10 NIC hardware offload for overlay

Modern NICs (Intel E810, NVIDIA ConnectX-4 and later, Broadcom Stingray) support hardware offload of Geneve and VXLAN operations. The support matrix:

| Feature | NIC support (2026) | Description |
|---------|-------------------|-------|
| RX checksum (inner) | Most NICs since 2018 | The hardware computes the inner IPv4, TCP, and UDP checksum. The kernel skips it. |
| TX checksum | Most | The hardware computes both outer and inner checksums. |
| GSO (Generic Segmentation Offload) | Linux 4.x and later | Kernel software offload. The NIC receives an skb of size up to 64 KB. |
| TSO (TCP Segmentation Offload) for VXLAN | ConnectX-4 and later, E810 | The NIC segments TCP itself across VXLAN. |
| TSO for Geneve | ConnectX-5 and later, E810 | The NIC segments across Geneve itself. |
| GRO (Generic Receive Offload) for Geneve | ConnectX-5 and later, E810 | The NIC merges multiple Geneve packets into one large skb. |
| RSS (Receive Side Scaling) for overlay | ConnectX-4 and later, E810 | The hardware hashes the inner flow to distribute load to multiple CPU queues. |

Verify offload on Linux:

```bash
[host]# ethtool -k eth0 | grep -E '(tx-udp|rx-udp|gro|gso|tso|rss).*tunnel'
tx-udp-segmentation: on
rx-udp-tunnel-port-offload: on
tx-udp_tnl-segmentation: on
tx-udp_tnl-csum-segmentation: on
```

When offload is disabled or the NIC does not support it, the CPU has to do the work, and throughput on a 10G or faster link drops by 50 to 70 percent. With OVS plus DPDK, every offload goes through the PMD driver, not through the kernel.

## 11.0.11 Anatomy Template A: `ovs-vsctl get Interface <tunnel> options`

> **Context:** Inspect the configuration of a specific tunnel and debug when the tunnel is not UP.

```bash
[host]# ovs-vsctl get Interface ovn-chassis-y options
{remote_ip="192.0.2.20", key=flow, csum="true", tos="inherit", \
 ttl="64", df_default="true", egress_pkt_mark="0", local_ip="192.0.2.10"}
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| `remote_ip` | A routable underlay IP | Same as `local_ip` on the same host | Empty or unreachable | The tunnel will fail. Run `ip route get <remote_ip>` to check. |
| `key` | `flow` for OVN | A fixed number for non-OVN setups | Empty | This is the multipoint tunneling pattern. |
| `csum` | `true` (default) | `false` (disabled, loses integrity protection) | (n/a) | Disable only when NIC offload is broken. |
| `tos` | `inherit` (copy inner DSCP) | A fixed number | Empty | `inherit` is usually correct for QoS. |
| `ttl` | 64 | Less than 32 (TTL might run out across a multi-hop underlay) | 1 | Risk of TTL exhaustion on a redundant ToR path. |
| `df_default` | `true` (mandates PMTUD) | `false` (allows fragmentation) | (n/a) | DF=1 forces PMTUD. DF=0 allows IPv4 fragmentation. |
| `egress_pkt_mark` | 0 | Custom value | (n/a) | Used for QoS on egress (Part 9.9). |
| `local_ip` | Matches the chassis underlay IP | Empty (auto) | Wrong IP | Manual override. Prefer auto. |

**Failure scenarios:**

- `remote_ip` is pingable but UDP 6081 fails: the firewall is dropping port 6081. Check `iptables -L` and `ufw status`.
- `csum=true` but NIC offload is broken: throughput drops by 50 percent. Set `csum=false` as a temporary workaround. The root cause fix is the NIC driver or firmware.
- `df_default=true` plus underlay MTU below 1500: the PMTUD ICMP "fragmentation needed" reply does not reach the sender, so a PMTU black hole forms. Part 11.1 has the deep dive.

## 11.0.12 ▶ Capstone POE: VXLAN or Geneve for a new OVN cluster?

> **Context:** An engineer is designing a new production OVN cluster of 100 chassis, 50,000 LSPs, with cross-DC RTT of 50 ms.

**Claim to verify:** "VXLAN is simpler and more stable than Geneve, so choose VXLAN for OVN."

**Predict:** False. OVN requires Geneve for every production-grade feature.

**Observe** (assumed, to be verified in a lab):

- Set `ovn-encap-type=vxlan` on the chassis: the tunnel comes UP but OVN logs the warning `"VXLAN encap does not support all OVN features"`.
- Cross-chassis migration: `Port_Binding` migration fails because the 24-bit VXLAN VNI cannot carry inport metadata. OVN falls back to MC_FLOOD for every packet, wasting 5 to 10 times the bandwidth.
- Stateful ACL with cross-chassis conntrack zone: the zone metadata is not preserved across VXLAN, so the conntrack entry is per-chassis. The reply path uses a different zone, and packets drop unexpectedly.
- Distributed LR (chassisredirect): the outport metadata is missing, so a packet arrives at LR Y but does not know its logical egress port. It either drops or loops.

**Explain:**

1. OVN must carry three metadata fields cross-chassis: datapath (24-bit), inport (15-bit), and outport (32-bit). The 24-bit VXLAN VNI fits only the datapath, not inport or outport. The Geneve TLV is variable and carries the full metadata.
2. OVN has used Geneve as the default since 2015. VXLAN support in the OVN code path is legacy, intended for simple L3-only deployments that do not need multi-tenant ACL, LB, or NAT.
3. "Simpler" is wrong because it ignores the feature trade-off. The VXLAN encapsulation header is simpler at the wire level (8 fixed bytes), but the cluster operation is more complex (workarounds for missing metadata).

**Falsification test:**

- Set up an OVN cluster with VXLAN. Configure a stateful ACL plus a distributed LR.
- Run cross-chassis traffic with conntrack flow.
- Observe `dpctl/dump-conntrack` zone consistency across chassis. If zones match, VXLAN is OK. If zones differ, VXLAN is not appropriate.

**Where the claim is wrong:** "VXLAN is simpler" looks at the wire format only and misses the OVN feature level. The correct conclusion is: **Geneve is the default for OVN production. VXLAN for OVN is for legacy interop only, when the cluster must talk to a non-OVN endpoint that does not understand Geneve.**

> **Rule:** New OVN deployment equals Geneve. VXLAN for OVN equals legacy interop only.

## Key points to remember

First, overlay encapsulation solves the VLAN 4096 limit and the L2-flat limitation. The 24-bit VXLAN VNI gives 16.7 million segments over an L3 underlay.

Second, Geneve is the official successor. It carries TLV-extensible metadata. OVN has used Geneve as the default since 2015.

Third, STT is fully deprecated. OVS 3.0 and later remove it. Do not choose it for new deployments.

Fourth, `options:key=flow` plus `options:remote_ip=flow` plus `set_tunnel` provides multipoint tunneling that scales well. This is the pattern OVN uses.

## 11.0.13 Geneve TLV as a metadata extension mechanism: nature, OVN encoding, operation

> This section dives into the **Geneve TLV mechanism itself** (the Type-Length-Value option section of the Geneve header) and the **OVN-specific encoding layer** (Option Class `0x0102`), separated from the high-level Geneve description in §11.0.3 and the byte layout in §11.0.7. The goal: an engineer should fully understand the general TLV structure, why OVN owns its own class, the mechanism that carries 71 bits of logical metadata across chassis through TLV, and how to trace a cross-chassis incident back to a specific byte inside the TLV.

### Core concept: extensible TLV versus fixed base header

The Geneve TLV is a **variable-length option region** that follows the 8-byte Geneve base header. Each option is encoded as the triple **(Class 16-bit, Type 8-bit, Length 8-bit) plus Value (0 to 124 bytes)** per RFC 8926 §3.5. By contrast, VXLAN has a fixed 8-byte header with no room for additional information, and STT (deprecated) has a fixed TCP-like header. Geneve was designed with TLV so that any deployment unit (a vendor, an open-source project, or an IETF working group) can register its own **Option Class** and encode self-defined metadata into the tunnel without modifying the base header. This is precisely why OVN chose Geneve as the default tunnel from 2015: OVN needs to carry **logical datapath plus logical inport plus logical outport plus flags** across chassis, and these three values together already take 71 bits, far more than the 24-bit VXLAN VNI. Category: an **extension mechanism inside the Geneve protocol**, not a message, not a command, and not an OVSDB table.

### History and background

Geneve was started by Jesse Gross (VMware), Ilango Ganga (Intel), and T. Sridhar (VMware) in February 2014 with the draft `draft-gross-geneve-00`. The draft responded to a strong need from VMware NSX-T (the successor to NVP and NSX-MH) to replace STT (proprietary, unable to traverse middleboxes) with a protocol that could carry rich metadata for service chaining, OAM, and tenant context. RFC 8926 was standardized in November 2020, six years after the first draft. The OVN team (Justin Pettit, Russell Bryant, Numan Siddique) launched OVN in September 2015 alongside OVS 2.5. From the first release, OVN chose Geneve as the default tunnel and registered **Option Class `0x0102`** with IANA under the name *Open Virtual Networking (OVN)*, contact `OVN-Committers`. Earlier, in April 2014, IANA had granted `0x0100` to LinuxFoundation and `0x0101` to OVS-Committers (used for OVS internal experimentation). `0x0102` is the third class in the range, dedicated to OVN logical pipeline metadata. INT (In-band Network Telemetry) took `0x0103` right after OVN. Granting OVN its own IANA-allocated class (rather than reusing OVS `0x0101`) was an intentional architectural choice: it separates the OVS semantic channel (data path layer) from the OVN one (logical pipeline layer).

### Position in the architecture

The Geneve TLV sits at the **tunnel encapsulation layer** (data plane underlay), between the **bridge `br-int`** on a chassis (where an OpenFlow flow calls the action `output:<tunnel-port>`) and the **bridge `br-int`** on the destination chassis (where a `table=0` flow decapsulates and resubmits). On the physical path, the TLV exists only on the wire between the two physical NICs of the two chassis (passing through the underlay L3 routers and switches). Scope: **per-packet, per-cross-chassis-tunnel**. Every cross-chassis OVN packet (passing through a tunnel labelled `ovn-<chassis>-0` on `br-int`) carries exactly one TLV with class `0x0102`, generated at encapsulation time. A local packet (where ingress and egress are on the same chassis) **never touches a TLV**, because it does not go through a tunnel port. The OVS kernel datapath processes the TLV bytes through the `vport-geneve` module (file `net/openvswitch/vport-geneve.c`). Userspace `ofproto-dpif` manages TLV registration through the extension OXM `tun_metadata0..63` (file `lib/tun-metadata.c` of OVS).

### Role and responsibilities in the architecture

The Geneve TLV plays **four roles** in the OVN-OVS architecture:

1. **Carries `tunnel_key` (logical datapath) through the VNI base header.** The column `Datapath_Binding.tunnel_key` (24 bits) is loaded into the VNI field of the Geneve base header (24 bits) by the action `set_field:NXM_NX_TUN_ID`. This is not a TLV but the canonical Geneve VNI. OVN treats it as part of the cross-chassis metadata bundle.
2. **Carries logical inport plus outport through the TLV `mff_ovn_geneve`.** The 32 bits of TLV `0x0102` hold `logical_outport` at bits 0 to 31 and `logical_inport` at bits 16 to 30 (after a shift). This lets the receiving chassis know exactly which logical entry port and exit port `ovn-northd` assigned.
3. **Allows future protocol extension.** Because the Geneve TLV is variable, OVN can add a new class in the future (for example a MAC binding cache or an ECMP hint) without changing the wire-level base format.
4. **Distinguishes OVN traffic from non-OVN traffic on the same UDP port 6081.** When an underlay carries multiple projects all using Geneve (NSX-T, INT telemetry, native Linux Geneve), class `0x0102` is a quick identifier that lets `tcpdump` and `tshark` filter OVN traffic precisely.

### Why it exists (design motivation)

Before Geneve TLV, OVN had to fit metadata into **one of three unsuitable choices**: (a) VXLAN VNI of 24 bits, enough for the datapath but with no room for inport or outport, forcing the receiving chassis to look up local tables and waste time; (b) STT 64-bit context, which has enough space, but STT cannot traverse most middleboxes because it impersonates TCP, and many cloud providers block UDP-over-fake-TCP; (c) embedding metadata into the inner packet (for example adding a VLAN header inside the inner Ethernet frame), which breaks L2 transparency for the tenant. Geneve TLV solves all three: it carries 71 bits of metadata, it does not impersonate any protocol, and it does not break the inner frame. Furthermore, registering an IANA-allocated Option Class gives OVN an independent 16-bit Type namespace (`0x80` through `0xFF` reserved for OVN internal use), which avoids conflicts with INT, NSX-T, or any other project also using UDP 6081.

### Problems it solves

The Geneve TLV (class `0x0102` plus the base VNI) solves **five concrete technical problems**:

1. **Distributing the logical pipeline across many chassis without a central controller.** The receiving chassis reads the TLV and immediately knows which datapath and which logical port to resubmit into, with no need to query OVSDB over the network.
2. **Atomic semantics for a cross-chassis hop.** The TLV is in the same Ethernet frame as the inner packet, so packet loss equals metadata loss. There is no race condition where metadata arrives before the inner frame.
3. **Compatible with L3 underlay routers.** Geneve uses UDP 6081, which routers see as ordinary UDP. They do not strip TLVs (unlike some proprietary middleboxes that may strip unfamiliar options).
4. **Allows NIC offload of checksum and TSO.** ConnectX-5 and later, and Intel E810, understand the Geneve TLV header and can compute the checksum and segment TCP across the tunnel in hardware.
5. **Separates the OVS metadata channel (class `0x0101`) from the OVN one (class `0x0102`).** The same host can run an OVS-only bridge (using `0x0101` for experimentation) alongside an OVN bridge (using `0x0102` for the logical pipeline) without colliding in the TLV space.

### Importance

The Geneve TLV is **tier 1 core** of OVN, on par with `Logical_Flow` and `Port_Binding`. An engineer who does not understand the byte layout of TLV `0x0102` cannot debug any cross-chassis incident: a VM on one side cannot ping a VM on the other side and packets drop, a gateway redirect does not move to the HA chassis, multicast flood does not reach the remote chassis, conntrack zones disagree between two chassis. Every production-grade OVN feature (multichassis live migration, distributed gateway router, cross-chassis stateful ACL, `Load_Balancer` hairpin) relies on the TLV. Among the more than 320 keywords in REF, Geneve TLV is in the group of fifteen cornerstone OVN keywords alongside `ovn-northd`, `ovn-controller`, NBDB, SBDB, `Logical_Flow`, `Datapath_Binding`, `Port_Binding`, `Logical_Switch`, `Logical_Router`, ACL, `Load_Balancer`, NAT, `ovn-trace`, and `ovn-detrace`.

### How it works in detail: encode, transit, decode

A cross-chassis packet goes through **three phases** from the TLV point of view:

1. **Encode at the sending chassis (egress).** `ovn-controller` translates the `Logical_Flow` action `output:<remote-port>` into a sequence of OpenFlow actions: (i) `set_field:datapath->NXM_NX_TUN_ID[0..23]` loads the 24-bit `tunnel_key` into the VNI; (ii) `set_field:outport->NXM_NX_TUN_METADATA0[0..31]` loads the 32-bit outport into TLV value bits 0 to 31; (iii) `move:NXM_NX_REG14[0..14]->NXM_NX_TUN_METADATA0[16..30]` moves the inport (15-bit `reg14`) into TLV value bits 16 to 30 (overlapping the outport, which is feasible because outport numbers are usually small and fit in bits 0 to 15); (iv) `output:<tunnel-ofport>`. The OVS kernel datapath converts these four actions into bytes: it builds a Geneve header with Opt Len = 2 (8 bytes of TLV), Class = `0x0102`, Type = `0x80`, Length = 2, and Value = 8 bytes. Combined with the 8-byte base, this is **16 bytes of Geneve** following the UDP header.
2. **Transit through the underlay.** The outer Ethernet frame contains an outer IP (src is the sending chassis underlay IP, dst is the receiving chassis underlay IP), a UDP header (src is hashed from the inner flow for ECMP, dst is 6081), 16 bytes of Geneve, and the inner Ethernet frame. L3 routers along the path see plain UDP and do not interfere with the option bytes. If a firewall or NAT box on the underlay drops UDP 6081, packets are lost. If a middlebox does not understand Geneve TLV but tries to parse it, it might strip the option (very rare).
3. **Decode at the receiving chassis (ingress).** The OVS kernel datapath uses the `vport-geneve` `recv` hook to read the 16 bytes of Geneve, populates `tun_id` (VNI) and `tun_metadata0` (TLV value, 8 bytes) into the skb metadata, and pushes it up to the `table=0` flow on `br-int`. The flow rule installed by `put_remote_port_redirect_*()` performs the inverse sequence: `move:NXM_NX_TUN_METADATA0[0..31]->NXM_NX_REG15[0..31]` (restore outport), `move:NXM_NX_TUN_METADATA0[16..30]->NXM_NX_REG14[0..14]` (restore inport), `move:NXM_NX_TUN_ID[0..23]->OXM_OF_METADATA[0..23]` (restore datapath), and finally `resubmit(,<egress-table>)` pushes the packet into the egress pipeline of the destination datapath.

Invariant: at any moment on the wire, **a cross-chassis OVN packet always carries exactly one TLV of class `0x0102`**. If two TLVs of the same class appear on the same packet, there is either a `northd` bug or a man-in-the-middle injection. Class `0x0101` (OVS) never appears in production OVN traffic.

### Classification of TLVs in the IANA Geneve Option Class space

The 16-bit class space is divided into two ranges per RFC 8926 plus the IANA Geneve Option Class registry (see §11.0.7 for seven sample entries):

| Range | Procedure | Description |
|-----|-----------|-------|
| `0x0000-0x00FF` | IETF Review | Reserved for future IETF standardization (none allocated yet). |
| `0x0100-0xFEFF` | First Come First Served | Vendors and projects register in order. About thirty entries by 2026-04. |
| `0xFF00-0xFFFF` | Experimental Use | Local experimentation, no uniqueness guarantee. |

Each class owner manages its own 16-bit Type space (256 types times 32 length-words is a wide space). OVN class `0x0102` currently uses **a single type** (`0x80`) for logical pipeline metadata. Types `0x81` through `0xFF` are reserved for future extensions if OVN needs more fields (such as an ECMP hint or a cross-chassis MAC binding cache). The high bit of the Type field is the **C-bit (Critical)**: when set, a receiver that does not understand the TLV must drop the packet (RFC 8926 §3.5.1). OVN sets the type to `0x80`, meaning C-bit = 1, because this metadata is essential. A packet without it cannot be processed correctly.

### How an operator masters this

An engineer needs four foundational skills with the Geneve TLV:

1. **Capture and decode raw TLV bytes from `tcpdump`.** Run `sudo tcpdump -i <phys-iface> -nn -X 'udp port 6081' -c 5` to grab five sample packets. Read the byte at offset 42 (after outer Eth, IP, and UDP). That is the start of the Geneve base. Offset 50 is the first TLV byte. Recognize `01 02` as the OVN class and `80 02` as Type=0x80 plus Length=2 (8 bytes).
2. **Decode TLV with `tshark` or Wireshark using the dissector.** Run `sudo tshark -i <phys-iface> -Y geneve -V`. The output already contains the parsed Geneve plus TLV. Wireshark 4.0 and later has a dissector that displays "Class: OVN (0x0102), Type: 0x80, Length: 8 bytes".
3. **Inspect tunnel and chassis encap configuration.** Run `ovs-vsctl --columns=name,type,options list Interface | grep -A1 ovn-` to see whether the tunnel port has `type=geneve, options:remote_ip=...`. Run `ovn-sbctl --columns=name,encaps list Chassis` to see whether each chassis advertises the `geneve` encap.
4. **Decide which skill applies when.** When cross-chassis ping drops, start with skill 1 (capture to see whether UDP 6081 traffic exists and whether a TLV is present). When UDP 6081 is present but the metadata is wrong, use skill 2 (decode explicitly to see whether outport and inport match expectation). When the underlay shows no UDP 6081 at all, use skill 3 (the tunnel is misconfigured or a firewall is blocking it).

### Usage workflow (cross-chassis TLV verification)

A six-step workflow verifies that a pair of chassis transmits a TLV correctly. Each step has a verification command.

1. **Confirm the tunnel is UP.** `ovs-vsctl get Interface ovn-<chassis-id>-0 link_state` should return `up`. If `down`, BFD is not stable. Stop here.
2. **Confirm the encap type is Geneve.** `ovn-sbctl --columns=name,encaps list Chassis <chassis-id>` should show `encaps=[<uuid>]`. Then `ovn-sbctl list Encap <uuid>` should show `type=geneve` (not `vxlan` or `stt`).
3. **Send a logical test packet.** From the sending chassis VM, run `ip netns exec vm1-ns ping -c 1 <vm2-ip>`.
4. **Capture on the physical interface of the sending chassis.** `sudo tcpdump -i eth0 -nn -X -c 1 'udp port 6081 and host <peer-underlay-ip>'`. Confirm that the bytes at offset 50 are `01 02 80 02` (Class plus Type plus Length).
5. **Decode the 8-byte value after the TLV header.** Bits 0 to 31 must equal the destination LSP `tunnel_key` (check with `ovn-sbctl list Port_Binding vm2 | grep tunnel_key`). Bits 16 to 30 must equal the source LSP `tunnel_key` shifted to a 15-bit value.
6. **Verify decoding on the receiving chassis.** On the receiving chassis, run `ovs-appctl ofproto/trace br-int <flow-spec>` or `ovs-ofctl dump-flows br-int table=0`. Look for a flow `tun_id=<key>,tun_metadata0=<value>` that matches and resubmits to table 8 or 16.

Production best practice: **always enable BFD on the Geneve tunnel** so that peer-down detection is below one second. **Set the underlay MTU to at least 1550** (1500 inner plus 58 bytes of Geneve overhead per §11.0.8), or better 9000 jumbo. Anti-pattern: running underlay MTU at 1500 with inner MTU 1500. This causes per-packet fragmentation and reduces performance by 30 to 50 percent.

### When something goes wrong (TLV-centric reverse symptom map)

Seven common symptoms point back to the Geneve TLV:

| Symptom | TLV region to inspect | Diagnostic flow |
|---------|------------------|-----------------|
| "VM on chassis A cannot ping VM on chassis B, 100 percent loss" | Tunnel state plus UDP 6081 reachability | Steps 1 and 4 of the workflow above: is the tunnel up, is UDP 6081 visible on the wire. |
| "Cross-chassis ping is intermittent, 5 to 10 percent drop" | MTU plus fragmentation | Check `ip link show` MTU on the underlay versus the inner. If DF is set and overhead exceeds MTU, a PMTU black hole is forming. |
| "Cross-chassis works but `Logical_Router` routes incorrectly" | TLV decode value bits 0 to 31 (outport) | Step 5: decode the value, compare to `Port_Binding.tunnel_key` of the remote LRP. |
| "Multi-tenant traffic crossover (tenant A sees tenant B traffic)" | TLV decode value bits 0 to 23 (datapath) | Compare the captured `tun_id` with the `Datapath_Binding.tunnel_key` of both tenants. |
| "Stateful ACL across chassis drops the reply" | conntrack zone derived from the datapath in the TLV | Cross-link 9.24 `ct_zone`. Check zone consistency across the two chassis. |
| "Live migration shows a 30-second flow stuck" | TLV outport still points to the old chassis | `ovs-appctl ofproto/trace` shows whether the outport in the TLV matches the current `Port_Binding.chassis`. |
| "Wireshark shows a strange TLV class instead of 0x0102" | Class collision with another project | Capture raw bytes. Check whether a proprietary middlebox is injecting a TLV. |

### Tightly related to other components

Five components are tightly coupled with the Geneve TLV (tight coupling, not loose cross-link):

1. **VNI base header (`MFF_TUN_ID`).** The 24-bit VNI travels alongside the TLV in the same Geneve header. Lose one and you lose both. `put_encapsulation()` emits both in a single flow rule.
2. **OXM `mff_ovn_geneve` (equal to `tun_metadata0` on OVS).** OVN dynamically declares the OXM ID through the action `tlv-map` at `ovn-controller` startup (see `controller/encaps.c::ovn_geneve_tlv_register()`). If OVS forgets to register, the flow `set_field:tun_metadata0` parser fails: the control plane is up but the data plane silently drops.
3. **`MFF_LOG_INPORT` (`reg14`) plus `MFF_LOG_OUTPORT` (`reg15`).** These two registers are encoded into the TLV value at egress and decoded back at ingress. If the bit layout changes (for example, OVN promotes inport to 24 bits for a large cluster), `put_encapsulation()` and `put_remote_port_redirect_*()` must change in lockstep.
4. **`Chassis.encaps` plus the `Encap` table in SBDB.** An `Encap` row contains `type=geneve, ip=<underlay-ip>, options:csum=true`. `ovn-controller` reads this table to choose the tunnel type for each remote chassis.
5. **The kernel module `openvswitch.ko` plus `vport-geneve.c`.** When the kernel runs an old version (Linux below 4.6) without `vport-geneve` or without offload, userspace fallback runs slowly and the tunnel only reaches 1 to 2 Gbps instead of 10 Gbps.

Loose-related components (cross-link, not tightly coupled): VXLAN tunnel (alternative encap, not on the same instance), STT (deprecated alternative), NSH service chain (different abstraction layer).

### Real-world incident anatomy

**Context:** An OpenStack OSP-17 cluster running OVN 22.03.4, with 80 compute nodes, 2 HA controllers, ConnectX-5 NICs, a 25 Gbps underlay, and an underlay MTU of 1500 (jumbo migration not yet completed). At the end of March 2024, monitoring reported intermittent 1 to 3 percent packet loss for cross-AZ VM ping, while same-AZ ping was stable at 0 percent.

**Symptom:** The operator saw `tcpdump` on the source VM sending all ICMP requests, but the destination VM only received about 97 to 99 percent. There was no ICMP error reply. The drops were random, not concentrated at any one moment.

**Investigation steps:**

1. **Step 1.** `ovs-vsctl get Interface ovn-cmpt-23-0 link_state bfd_status`. The tunnel was UP and BFD was stable. Eliminated tunnel-down as a cause.
2. **Step 2.** `tcpdump -i bond0 -nn 'udp port 6081 and host <peer-underlay>'` captured 1000 packets while the operator ran a `ping -c 1000`. Count: `tcpdump` on the sender showed 1000 UDP 6081 packets, while `tcpdump` on the receiver showed only 970. **Thirty packets disappeared in the underlay.**
3. **Step 3.** The captured outer IPv4 header had the DF bit (Don't Fragment) set. Some packets had a wire size of 1532 (above the 1500 underlay MTU). The ToR switch dropped those packets and sent ICMP unreachable type 4 code 4. The ICMP reply did not return to the sending chassis because the firewall blocked ICMP for the underlay subnet.
4. **Step 4.** Recompute the overhead: inner Ethernet 14 plus inner IPv4 20 plus ICMP 8 plus payload 1472 equals 1514 bytes inner. Encap adds Geneve 16 bytes (8 base plus 8 TLV) plus UDP 8 plus outer IPv4 20 plus outer Ethernet 14, which is 58 bytes of overhead. Total wire size is 1514 plus 58, which is 1572 bytes. **That exceeds the 1500 underlay MTU by 72 bytes.** Some inner packets below 1442 bytes were unaffected. Inner packets at or above 1442 bytes were silently dropped because of DF.

**Root cause:** A PMTUD black hole. The underlay firewall dropped the ICMP reply, combined with the OVN default DF=1 on Geneve plus an underlay MTU of 1500 that was not enough for the 58-byte overhead.

**Fix:**

- Temporary fix: reduce the inner MTU from 1500 to 1442 on every VM through DHCP option 26 (`interface_mtu`, not `subnet_mask`). The drop rate fell to 0 percent within five minutes.
- Permanent fix: raise the underlay MTU from 1500 to 9000 across every ToR and spine. After two weeks of migration, the inner MTU was returned to the default 1500. The cluster benefited from jumbo frames, and cross-AZ VM throughput rose by 1.6 times.

**Lesson learned:** Geneve TLV overhead is not just 8 bytes of TLV. It is 16 bytes (8 base plus 8 TLV) plus UDP 8 plus outer IPv4 20 plus outer Ethernet 14, which gives a minimum of 58 bytes. When the underlay MTU equals the inner MTU, every full-size inner packet is dropped. Cross-link: Part 11.1 PMTUD failure modes in detail, Part 19.0 multichassis forensic.

(a) Ping a VM on the same chassis (`vm1` to `vm3` on the same `chassis-1`): no capture appears on `eth0` (the packet does not go through the tunnel). This confirms that the TLV exists only across chassis.
(b) Switch the tunnel type to `vxlan` (`ovn-sbctl set Encap <uuid> type=vxlan`): the capture now shows UDP 4789 instead of 6081, with no TLV (only the 24-bit VNI). Inport and outport are lost, and OVN logs a warning.

### Common failures and diagnostic signals

**Mode A, TLV stripped by an unfamiliar middlebox:**
- Signal: a capture on the sending chassis shows the full `0x0102` TLV. The capture on the receiving chassis shows a Geneve packet with VNI but Opt Len = 0, no TLV. The `ovn-controller` log on the receiver shows `"unknown tun_metadata, falling back to MC_FLOOD"`.
- Fix: identify the middlebox (proprietary firewall, NAT box) on the path, and either bypass it or whitelist the TLV.

**Mode B, class collision with INT (0x0103) or an unfamiliar vendor:**
- Signal: `tshark` shows a packet with two TLVs at once, one `0x0102` and one of an unknown class. Rare in OVN-only production.
- Fix: check whether another project on the same underlay is also configured with UDP 6081. Separate the underlay subnet for OVN.

**Mode C, MTU mismatch plus DF=1 plus ICMP black hole:**
- Signal: cross-chassis drop is random 1 to 5 percent. Large packets fail, small packets pass. No ICMP unreachable returns to the sender.
- Fix: raise the underlay MTU, reduce the inner MTU by 58 bytes, or disable DF (`options:df_default=false` to allow IPv4 fragmentation).

**Mode D, OXM `tun_metadata0` not registered at OVS startup:**
- Signal: `ovn-controller` logs `"could not configure tunnel metadata mapping: TLV type %d not registered"`. The `set_field:tun_metadata0` flow rule fails to parse. The data plane silently drops cross-chassis traffic.
- Fix: restart `ovn-controller` to re-register the OXM through the `tlv-map` action. Confirm the OVS version is at least 2.5 (older versions do not support dynamic OXM).

**Mode E, NIC offload checksum bug causes receiver drop:**
- Signal: an old ConnectX-5 firmware computes the UDP checksum without including the Geneve TLV. Packets arrive at the receiver with a failing UDP checksum and are dropped before reaching the datapath. `ethtool -S eth0 | grep csum_err` rises.
- Fix: update the NIC firmware, or temporarily disable UDP TX offload (`ethtool -K eth0 tx-udp-segmentation off`).

### Comparison with other systems

**VXLAN (RFC 7348):**
- *Similarities:* same UDP encap (4789 versus 6081), same L2-over-L3 overlay, same use of a 24-bit VNI for the tenant segment.
- *Differences:* the VXLAN header is fixed at 8 bytes and has no TLV, so it cannot carry inport or outport metadata. OVN over VXLAN must fall back to MC_FLOOD and loses cross-chassis stateful features. The Geneve TLV `0x0102` is extensible, while VXLAN has only the VNI.

**STT (Stateless Transport Tunneling, draft-davie):**
- *Similarities:* also proposed by Nicira and VMware. Has a 64-bit context with enough space for metadata.
- *Differences:* STT impersonates a TCP header to piggyback on the TSO offload of older NICs, so many middleboxes and NAT devices reject it. The IETF did not standardize it. OVS 3.0 and later remove it. Geneve does not impersonate any protocol, traverses middleboxes well, and is an official IETF standard (RFC 8926).

**NSH (Network Service Header, RFC 8300):**
- *Similarities:* also an extension header that carries metadata for service chaining in an overlay.
- *Differences:* NSH focuses on the **service path identifier** and **service index** for service function chaining, not on the logical pipeline ingress and egress port. NSH runs over VXLAN-GPE or Geneve outer, so it is a layer above the TLV. OVN does not use NSH. When SFC is needed, OVN uses an ACL with a multi-hop chain.

**MPLS labels (RFC 3032):**
- *Similarities:* same role of attaching 20 bits of metadata (the label) to the packet so that an intermediate node can do a fast lookup.
- *Differences:* MPLS runs at L2.5. It requires a provider edge router with LDP or BGP-LU. It is not appropriate for a cloud overlay because it does not naturally cross L3 boundaries. The Geneve TLV runs at L7-of-L4 (UDP) and crosses any L3 underlay.

**Cisco VXLAN-GPE Group-Based Policy (GBP):**
- *Similarities:* also carries an additional tenant plus policy tag on top of VXLAN.
- *Differences:* GBP requires an ACI fabric underneath plus an APIC controller. It is not an open standard. The Geneve TLV `0x0102` runs on plain Linux plus plain OVS, with no vendor lock-in.

> **Rule:** OVN production equals Geneve TLV `0x0102`. VXLAN for OVN serves only legacy interop with endpoints that do not understand Geneve. STT, NSH, and MPLS are not options for OVN.

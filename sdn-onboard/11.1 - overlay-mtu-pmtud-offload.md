## 11.1.1 MTU math for overlay

Compass Ch 13: "95 percent of 'tunnel up but no traffic' is MTU."

Ethernet has a default MTU of 1500 bytes. With overlay encapsulation:

```
Inner packet (tenant)
  +--- Ethernet (14B) + IP (20B) + TCP (20B) + data

→ wrapped in:

Outer packet
  +--- Ethernet (14B) + IP (20B) + UDP (8B) + VXLAN (8B) + [Inner...]
       ^^^                                                ^^^
       Underlay MTU must fit everything here
```

VXLAN overhead is 50 bytes added.

**Rule:** Inner MTU is at most underlay MTU minus the overhead.

- Underlay 1500, tenant inner = 1450 (VXLAN 50 bytes).
- Underlay 1600 (jumbo), tenant inner = 1500.
- Underlay 9000 (jumbo), tenant inner = 8950.

Production solutions:

1. **Shrink the tenant MTU** to 1450. TCP MSS negotiates based on this MTU.
2. **Bump the underlay MTU** to 1600 or higher (or 9000 jumbo). The tenant keeps the 1500 default.

## 11.1.2 PMTUD (RFC 1191)

PMTUD auto-discovers the smallest MTU on a path:

1. The sender sends a packet with the DF flag set.
2. A router with MTU less than the packet size drops it and sends an ICMP "Fragmentation Needed" (Type 3 Code 4) carrying the correct MTU.
3. The sender reduces the packet size and retries.

**Requirement:** ICMP Type 3 Code 4 must **not be blocked** at any firewall. The common failure mode is that an admin blocks ICMP at the firewall, PMTUD fails silently, and the application times out.

**RFC 4821 PLPMTUD** (Packetization Layer PMTUD) is a fallback for when ICMP is blocked. It sends a probe packet, and if there is no ACK it reduces the size. The Linux kernel has had this on by default since the 3.x series:

```bash
[host]# sysctl net.ipv4.ip_no_pmtu_disc       # 0 = PMTUD enabled
[host]# sysctl net.ipv4.tcp_mtu_probing       # 1 = PLPMTUD when ICMP fails
[host]# ip route get 10.0.0.100                 # show the current MTU
```

## 11.1.3 Tunnel-specific tuning

OVS tunnel options:

```bash
[host]# ovs-vsctl set Interface vxlan0 \
         options:tos=inherit \                 # copy inner ToS to outer
         options:ttl=64 \                      # fixed outer TTL
         options:df_default=true               # set DF on the outer header
```

`options:df_default=true` (**recommended**): set DF on the outer IP header. When the path has a low MTU, the underlay router sends ICMP, the tunnel endpoint receives it, and adjusts.

`options:df_default=false`: the outer packet is fragmented silently, and inner PMTUD is broken.

## 11.1.4 NIC offload interaction with tunnels

Modern NICs offer many offloads:

- **TSO** (TCP Segmentation Offload): the CPU sends a large TCP buffer (64 KB), and the NIC segments it into MSS-sized packets.
- **GSO** (Generic): the software equivalent, in the kernel.
- **LRO** (Large Receive Offload): the NIC coalesces packets into a large frame.
- **GRO** (Generic): the kernel-side equivalent.
- **Checksum offload:** the NIC computes the checksum instead of the CPU.

Tunnel encapsulation adds complexity. The NIC must know about VXLAN or Geneve headers in order to skip past them correctly when computing the checksum.

**NICs that support tunnel offload** (ConnectX-4 and later, Intel XL710 and later, Broadcom Thor and later):

- Inner checksum computed by the NIC.
- Inner TSO: the CPU sends one large VXLAN packet, and the NIC segments the inner TCP.
- Performance gain is 2x to 5x compared with software encapsulation.

Check:

```bash
[host]# ethtool -k eth0 | grep -E 'tx-udp|tx-gso'
# tx-udp_tnl-segmentation: on    (means TSO for tunnels is supported)
```

Disable when troubleshooting:

```bash
[host]# ethtool -K eth0 tx-udp_tnl-segmentation off
```

## 11.1.5 Case study: slow TCP through a tunnel

Symptom: UDP works, small TCP works, but large TCP downloads are extremely slow.

Diagnosis path:

1. **Check MTU:** `ip link show`. If it is 1500 and the underlay is 1500, this is an MTU issue.
2. **Tenant DF:** use `tcpdump` to see whether the TCP packet has DF=1.
3. **ICMP path:** run `traceroute`. If ICMP is blocked, PMTUD is broken.
4. **`tcp_mtu_probing`:** enable the PLPMTUD fallback.
5. **Quick fix:** reduce the tenant MTU to 1450, or bump the underlay to 1600.

Long term: tune the underlay to jumbo 9000 and the tenant to 1500 for the best performance.

## 11.1.6 PMTUD packet flow in detail (RFC 1191 plus RFC 8201 for IPv6)

### IPv4 PMTUD flow (RFC 1191, November 1990)

The sender wants to transmit a 1500-byte packet with DF=1, but one segment of the path has MTU 1400.

```
Sender (1500B, DF=1)
   │
   │ Hop 1: MTU 1500 ✓ forward
   ▼
Router-1 (egress MTU 1400, packet 1500 + DF=1)
   │
   │ DROP packet
   │ Generate ICMPv4 Type 3 Code 4 "Fragmentation Needed and DF set":
   │   - Header (8 byte) + original IP header + first 8 byte payload
   │   - Next-Hop MTU field = 1400 (RFC 1191 extension)
   │ Source IP = router IP, Dest IP = sender IP
   │
   ▼
Sender receives ICMP
   │
   │ Kernel updates the route cache MTU to 1400 for that destination IP
   │ The TCP layer detects this and retries with a correspondingly lower MSS
   │
   ▼
Sender (1400B retry) ─► Hop 1 ✓ ─► Router-1 ✓ ─► Hop 2 ✓ ─► ... ─► Receiver
```

`ip route get <dest>` shows the updated MTU:

```bash
[host]# ip route get 10.0.0.100
10.0.0.100 via 192.168.1.1 dev eth0 src 192.168.1.10 mtu 1400 ...
```

### IPv6 PMTUD flow (RFC 8201, July 2017)

IPv6 routers do NOT fragment the way IPv4 does. The DF semantics are implicit in IPv6. Every router that sees a packet larger than the MTU drops it and sends an ICMPv6 Type 2 "Packet Too Big". The format is similar to ICMPv4, but the MTU field is 32 bits instead of 16 bits.

### Kernel route cache

The Linux kernel caches the PMTU per destination, with a 10-minute expiry (`net.ipv4.route.mtu_expires`):

```bash
[host]# sysctl net.ipv4.route.mtu_expires
net.ipv4.route.mtu_expires = 600
```

After 10 minutes, the kernel tries the original MTU again. If the path has changed (less restrictive), the kernel learns the new MTU. If it is still restrictive, the kernel receives ICMP again and updates the cache again.

## 11.1.7 PMTU black hole, the most common failure mode

**Definition:** PMTUD does not work because ICMP Type 3 Code 4 (IPv4) or Type 2 (IPv6) is dropped by a firewall on the path. The sender does not receive the feedback. The sender continues to send 1500-byte packets with DF=1, and every packet is dropped at the intermediate router. The connection hangs indefinitely.

### Common symptoms

- The TCP handshake (SYN, SYN-ACK, ACK, all under 100 bytes each) **succeeds**.
- Application-layer small requests succeed (an HTTP GET with a 50-byte path).
- When the HTTP response exceeds 1400 bytes, or during a TLS handshake (ServerHello plus cert chain, around 1.5 to 3 KB), packets are dropped silently.
- The application times out after 30 to 60 seconds.

### Root cause matrix

| Where ICMP is blocked | Reason | Fix |
|-------------------|-------|-----|
| Sender-side firewall (host iptables) | The admin blocks all ICMP "for security" | Allow ICMP Type 3 Code 4 in and out |
| Receiver-side firewall | Same | Allow ICMP Type 3 Code 4 |
| Cloud security group (AWS, Azure, GCP) | Default rule | Allow ICMP Fragmentation Needed |
| ISP or carrier middlebox | Anti-DDoS rate-limits ICMP too aggressively | Contact the ISP, or use a workaround |
| Stateful firewall does not track ICMP | A stateless reply does not match conntrack | Use the stateful conntrack ICMP module |

### Workaround: TCP MSS clamping

If you cannot control the middlebox ICMP behavior, clamp the TCP MSS to a path-safe level:

```bash
# IPv4: clamp MSS = MTU - 40 (IP 20 + TCP 20)
[host]# iptables -t mangle -A POSTROUTING -p tcp --tcp-flags SYN,RST SYN \
        -j TCPMSS --clamp-mss-to-pmtu

# Or clamp to a fixed value (when PMTU discovery fails):
[host]# iptables -t mangle -A POSTROUTING -p tcp --tcp-flags SYN,RST SYN \
        -j TCPMSS --set-mss 1400
```

The `TCPMSS` mangle target rewrites the TCP MSS option in SYN and SYN-ACK to a safe level. TCP then segments down to that size on its own, without needing PMTU feedback.

OVN also supports TCP MSS clamping through a Logical Router option:

```bash
[host]# ovn-nbctl set Logical_Router router1 \
        options:mss_max_size=1400
```

OVN injects a flow at `lr_in_chk_pkt_len` to clamp MSS on the overlay-to-underlay-to-public path, so it does not depend on ICMP from the external network.

## 11.1.8 OVN `check_pkt_larger` action (`lr_in_chk_pkt_len` stage)

The OVN Logical Router pipeline has a stage dedicated to MTU checking. When an IPv4 or IPv6 packet enters the LR and its size exceeds the LRP egress MTU, OVN triggers:

1. **Action `check_pkt_larger(...)`:** check the packet size against a threshold.
2. If it exceeds the threshold, branch to the `lr_in_larger_pkts` stage.
3. This stage builds an ICMP "Packet Too Big" (Type 3 Code 4 for IPv4, or Type 2 for IPv6) and sends it back to the sender.
4. Drop the original packet.

Source code anchor (`branch-22.03`):

```c
// northd/northd.c, function build_icmperr_pkt_big_flows()
ovs_lswitch_lflow_add(lflows, od, S_ROUTER_IN_LARGER_PKTS, ..., 
    "ip4 && ip.ttl > 0",
    "icmp4 { ... eth.dst = E; eth.src = R; ... }; next;");
```

The original function name has a typo upstream: `reply_imcp_error_if_pkt_too_big()` (`imcp` instead of `icmp`). Verified at `branch-22.03`. Do NOT fix the typo when citing (Rule 14 §14.2).

The LRP MTU is configurable through NB:

```bash
[host]# ovn-nbctl set Logical_Router_Port lrp-external \
        options:gateway_mtu=1442
```

When the MTU is reduced (for example, 1442 for an underlay of 1500 with Geneve overhead 58), the LRP triggers ICMP from the OVN router itself, without depending on the underlay router to send ICMP.

## 11.1.9 NIC offload deep dive: tunnel-aware behavior

Section 11.1.4 listed offloads at a high level. This section goes deeper into specific behavior.

### TX side: TSO plus checksum offload for Geneve

The kernel software path when sending one 64 KB TCP packet through a Geneve tunnel:

1. The application calls `write()` with 64 KB.
2. The kernel TCP stack splits the 64 KB into MTU-sized segments (1442 bytes for Geneve overhead 58).
3. The kernel encapsulates each segment with the Geneve header (8 bytes base plus 8 bytes TLV) plus outer UDP plus outer IP.
4. The kernel computes the inner TCP checksum and the outer UDP checksum for each segment.
5. The NIC receives each 1500-byte packet and transmits it.

Total work: about 46 segments times (encapsulation plus checksum) on the CPU.

The NIC tunnel-aware path (ConnectX-5 and later, E810):

1. The application calls `write()` with 64 KB.
2. The kernel TCP stack pushes the entire 64 KB down to one skb with metadata: "tunnel inside, encap descriptor: Geneve VNI=X, outer src/dst, MSS=1442".
3. **The NIC** splits the 64 KB into 46 segments, encapsulates each with a Geneve header, and computes both the inner and the outer checksum.
4. The NIC transmits 46 packets, each 1500 bytes.

CPU work drops by a factor of 46. Throughput goes from about 5 Gbps (software) to 25 to 40 Gbps (hardware) on a 25 or 40 G link.

### RX side: GRO plus RSS

GRO (Generic Receive Offload) merges many small packets into one large packet before pushing it up to the TCP stack:

- Software GRO: kernel `skb_gro_receive()` merges packets from the same connection.
- NIC GRO (ConnectX-5 and later, E810): the NIC merges packets in its hardware queue and presents one large skb to the stack.

For Geneve, the NIC must understand the outer header in order to strip it and merge the inner TCP correctly. Not every NIC supports this. Check with `ethtool -k`:

```
[host]# ethtool -k eth0 | grep gro
generic-receive-offload: on
rx-gro-list: off
rx-gro-hw: off [fixed]      (the NIC does not support hardware GRO for Geneve)
```

`rx-gro-hw=off [fixed]` means the NIC does not support hardware GRO. Only software GRO is available. RX throughput for a large TCP file transfer is about 10 to 15 Gbps.

`rx-gro-hw=on` (ConnectX-6 and later, E810 with SR-IOV PF) means hardware GRO for Geneve is available, with RX throughput of about 25 to 40 Gbps.

### RSS hash on inner flow

RSS (Receive Side Scaling) distributes traffic across multiple CPU queues. For Geneve, the NIC hashes based on the configured fields:

```bash
[host]# ethtool -X eth0 hkey ... rss-hash-key ...
[host]# ethtool -n eth0 rx-flow-hash udp4
UDP over IPv4 flows use these fields for computing Hash flow key:
IP SA
IP DA
L4 bytes 0 & 1 [TCP/UDP src port]
L4 bytes 2 & 3 [TCP/UDP dst port]
```

By default, RSS hashes on the outer 5-tuple. If all traffic shares the same outer 5-tuple (because the Geneve tunnel is a single source), RSS distributes only to one queue, which becomes a single-CPU bottleneck.

The solution is to enable inner-flow RSS (NIC-supported, kernel 5.4 and later):

```bash
[host]# ethtool -K eth0 rx-vxlan-tnl-rss on
[host]# ethtool -K eth0 rx-geneve-tnl-rss on
```

Hashing on the inner 5-tuple distributes traffic across N queues correctly.

## 11.1.10 Anatomy Template A: `ip route get` plus `ethtool -k`

> **Context when run:** Check the effective MTU to a destination plus the offload state of the NIC.

### `ip route get <dest>`

```bash
[host]# ip route get 192.0.2.20
192.0.2.20 via 10.0.0.1 dev eth0 src 10.0.0.10 mtu 1442
   cache <expires 587sec>
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| `via <next-hop>` | A routable underlay IP | (n/a) | Empty or unreachable | `ip route` has no route. Check the default gateway. |
| `dev <iface>` | Matches the expected uplink | Different (multi-NIC) | Empty | The routing decision is wrong. |
| `src <local-IP>` | Chassis underlay IP | Different (rare) | Empty | The source IP is unbound. |
| `mtu <N>` | Matches underlay minus overhead | Less than expected (PMTU drop) | Empty (default 1500) | The PMTU has been learned, or the cache is empty. |
| `cache <expires N sec>` | Less than 600 sec (default) | Near 0 (about to expire) | Negative (expired) | The kernel will retry the original MTU. |

**Scenarios:**

- `mtu 1500` for a cross-chassis destination (which should be 1442): no PMTU feedback yet, or the cache has expired. Force a refresh: `ip route flush cache && ip route get <dest>` will refresh on the next call.
- `mtu 0`: the MTU is unknown, and the fallback is 1500. This may be a PMTU black hole.

### `ethtool -k <iface>` filtered to offload

```bash
[host]# ethtool -k eth0 | grep -E '(tx-udp|rx-udp|gro|gso|tso|checksum).*tnl'
tx-udp_tnl-segmentation: on
tx-udp_tnl-csum-segmentation: on
rx-udp-tunnel-port-offload: on
```

| Feature | Healthy | Warning | Critical | Diagnostic hint |
|---------|---------|---------|----------|-----------------|
| `tx-udp_tnl-segmentation` | `on` | `off` | `off [fixed]` | The NIC does not support, or the driver disables, this offload. TX throughput is reduced by 50 to 70 percent. |
| `tx-udp_tnl-csum-segmentation` | `on` | `off` | (n/a) | Inner checksum is in software, with CPU overhead. |
| `rx-udp-tunnel-port-offload` | `on` | `off` | (n/a) | The NIC does not recognize the Geneve port, so RX is not offloaded. |
| `rx-gro-hw` | `on` | `off` | `off [fixed]` | RX throughput is limited. See 11.1.9. |
| `rx-vxlan-tnl-rss` | `on` (for VXLAN) | `off` | (n/a) | RSS hashes only on the outer 5-tuple, single-CPU bottleneck. |
| `rx-geneve-tnl-rss` | `on` (for Geneve) | `off` | (n/a) | Same for Geneve. |

## 11.1.11 ▶ Capstone POE: Shrink tenant MTU versus bump underlay MTU

> **Context:** A production OVN cluster cross-DC. Underlay 1500 default. Workload Pod MTU 1500 default. A PMTU black hole is observed through slow large TCP transfers.

**Claim to verify:** "Bumping the underlay MTU to 9000 (jumbo) is the best-performance solution. There is no need to shrink the tenant."

**Predict:** Correct when you control the underlay. Incorrect when the underlay is a public cloud (AWS, Azure) that you do not control.

**Observe (assumed, to be verified in lab):**

- **On-prem DC plus private cabling:** bump the underlay from 1500 to 9000 on the switch and the NIC. Throughput rises 3x to 5x for large TCP transfers (jumbo flag). The tenant keeps 1500 default. Everything is healthy.
- **Public cloud, AWS VPC:** the AWS underlay MTU is 9001 by default. Inter-region underlay MTU is 1500 (through VPN or Transit Gateway). Bumping the VPC MTU to 9001 works inside the region. Cross-region still has the black hole.
- **Hybrid on-prem plus public:** the underlay is heterogeneous (private 9000 versus public 1500). A consistent bump is not feasible. The fallback is to shrink the tenant to 1442.

**Explain:**

1. **Bumping the underlay** only works when the engineer controls the entire underlay. For 100 percent on-prem this is the best solution: the tenant MTU stays at the 1500 default, and the application does not need to be aware.
2. **Shrinking the tenant** works everywhere (including public cloud where you do not control the underlay), but MTU 1442 is non-default. The application or the library must handle this. TCP MSS clamping handles 80 percent of cases automatically. The remaining 20 percent need application-level handling.
3. **TCP MSS clamping** (OVN `mss_max_size`) is the middle ground. The tenant MTU stays at 1500, but the MSS is clamped to 1402 (1442 minus 40 for TCP plus IP). TCP segments fit the underlay without requiring PMTU feedback.

**Falsification test:**

- Bump the underlay to 9000. Run iperf3 cross-DC. If throughput rises and there are no TCP retransmissions, the bump is effective.
- If some paths still have 1500 (cross-region, public link), the partial bump leaves the PMTU black hole on those paths.

**Where the claim is wrong:** "Best performance" assumes a private underlay. The correct framing must consider: **(a) is the underlay topology heterogeneous?** If yes, shrink the tenant plus MSS clamp is more robust than bumping. **(b) Is the tenant aware of an MTU below 1500?** If a legacy app hard-codes MTU 1500, MSS clamping is not enough.

> **Rule:** On-prem private DC, controllable: bump the underlay (best performance). Public cloud or heterogeneous: shrink the tenant plus MSS clamp (robust).

## Key points to remember

First, overlay MTU math: Inner MTU is at most underlay MTU minus the overhead. The majority of "tunnel up but no traffic" cases come down to MTU.

Second, PMTUD per RFC 1191 requires that ICMP not be blocked. PLPMTUD (RFC 4821) is the fallback when ICMP is blocked.

Third, OVS `options:df_default=true` makes overlay PMTUD work correctly. Not setting it leads to silent fragmentation.

Fourth, NIC tunnel offload (ConnectX-4 and later, Intel XL710 and later) gives a 2x to 5x real-world bandwidth boost. Check `ethtool -k` before scaling out.

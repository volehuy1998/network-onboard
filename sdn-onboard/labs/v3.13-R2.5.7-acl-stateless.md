A traditional Cisco engineer who has written `ip access-list extended DENY_HTTP; deny tcp any host 10.0.0.2 eq 80; permit ip any any` knows what an ACL is: a packet filter at a layer-3 boundary that drops or permits traffic based on packet header fields (source IP, destination IP, protocol, ports). Configure the ACL on a Cisco router, apply it to an interface with `ip access-group DENY_HTTP in`, and the silicon filters at line rate. The filter is **stateless**: each packet is evaluated independently against the rule set; the ACL has no memory of prior packets and no concept of "established connections".

OVS implements the same idea using OpenFlow flow rules. A flow rule with `actions=drop` is the equivalent of a Cisco `deny`. A flow rule with `actions=output:N` is the equivalent of a `permit` with explicit forwarding. Higher-priority rules override lower-priority rules; the default standalone bridge has a `priority=0,actions=NORMAL` catch-all that means "no specific match, do L2 learning". To install an ACL, insert a flow at higher priority than NORMAL that matches the prohibited traffic and drops it.

This lab demonstrates the stateless ACL pattern with three netns peers (ns_a, ns_b, ns_c) on a single OVS bridge. The baseline is full mesh connectivity. The ACL drops ICMP between ns_a and ns_b; ns_a-to-ns_c and ns_b-to-ns_c remain reachable. Removing the ACL restores full connectivity. The lab also shows the Linux iptables equivalent rule applied to the same bridge to demonstrate that an in-kernel filter combined with `physdev` matching achieves the same drop effect.

## R2.5.7.1. Where this part sits in the curriculum and why

R2.5.7 is the seventh sub-capture of R2.5. The pattern continues the "compare with traditional implementation" theme. R2.5.6 covered L3 routing without OVN; R2.5.7 covers the access-control layer that typically sits beside or after a router. The natural follow-up (covered in R2.5.8) is stateful ACL plus NAT, which extends the stateless model with conntrack to track connection state.

OVS's stateless ACL approach has three pedagogically important properties.

First, ACLs in OVS are NOT a separate primitive; they are just OpenFlow flows. There is no `ovs-vsctl set-acl` command. The operator writes the same `ovs-ofctl add-flow` syntax used for any other OpenFlow rule, with `actions=drop` instead of `actions=output:N`. This unifies the firewall with the forwarding plane: a single flow table expresses both "where to forward" and "what to drop". Cisco's ACL is a separate construct from the routing table; OVS's flow table is one table that does both.

Second, OVS flow rules can match on richer criteria than typical Cisco ACLs. Cisco extended ACLs can match on source/destination IP, protocol, source/destination port, and a small set of TCP flags. OVS OpenFlow flows can match on any of those PLUS source/destination MAC, VLAN ID, VLAN priority, MPLS label, ARP fields, IPv6 next-header, ICMP type/code, and many more (the full match-field catalog is documented at `Documentation/topics/openflow.rst` and `lib/ofp-actions.c`). This makes OVS-based ACLs more expressive but also more error-prone: a poorly-written match can either be too narrow (failing to block intended traffic) or too broad (blocking traffic that should be permitted).

Third, the Linux kernel's iptables provides an alternative ACL mechanism that operates above the bridge layer in the netfilter framework. iptables can match on physdev (the bridge port name), source/dest IP, protocol, ports, etc. The lab demonstrates an iptables FORWARD rule as a comparison; the runtime behaviour is similar but the configuration model and the kernel data path are different. iptables operates AFTER the bridge has done its L2 forwarding decision and BEFORE the packet leaves the host (or between veth ends inside the host).

The R2.5.7 source anchors live in two places. First, the OpenFlow flow-table evaluation in `ofproto/ofproto-dpif-xlate.c` decides which flow rule a packet matches and dispatches to the action set. Second, the action codecs in `lib/ofp-actions.c` parse the OpenFlow wire format of `drop` (which is encoded as a `clear_actions` instruction in OpenFlow, or as an empty action set). The OVS source baseline is v2.17.9 (commit `0bea06d9957e3966d94c48873cd9afefba1c2677`).

## R2.5.7.3. The OVS OpenFlow ACL pattern

The OpenFlow flow rule for stateless ACL has the form:

```openflow
table=0,priority=N,<match>,actions=drop
```

where `N` is a priority value greater than 0 (higher than the default NORMAL rule), and `<match>` is any combination of OpenFlow match fields. The lab uses:

```openflow
table=0,priority=200,icmp,nw_src=10.0.0.1,nw_dst=10.0.0.2,actions=drop
```

The `icmp` shorthand expands to `eth_type=0x0800,ip_proto=1` (IPv4 with protocol number 1). `nw_src` and `nw_dst` are the OpenFlow 1.0 names for source and destination IPv4 addresses (`ipv4_src` and `ipv4_dst` in OpenFlow 1.5 syntax; OVS accepts both).

The `actions=drop` is OVS's idiom for "no actions", which OpenFlow defines as implicit drop. The literal `drop` keyword is not in the OpenFlow 1.0 specification; OVS accepts it as syntactic sugar for an empty action set.

### ▶ Guided Exercise 2: Install the stateless ACL and verify enforcement

This exercise installs two flow rules at priority 200 to drop ICMP between ns_a and ns_b. The bidirectional pair is necessary because ICMP echo and reply travel in opposite directions; dropping only the forward direction would leave the reply path intact.

**Outcomes.** Two new flow rules visible in `ovs-ofctl dump-flows`. ns_a → ns_b ICMP fails 100% packet loss. ns_a → ns_c and ns_b → ns_c succeed. The drop counter on the matching flow increments per dropped packet.

**Misconception to refute.** "An OVS ACL needs only the forward direction; the reverse is automatic." Wrong for stateless ACLs. ### ▶ Guided Exercise 3: Inspect the kernel datapath flows for the dropped traffic

The kernel datapath cache reflects the ACL's effect: the dropped flow does NOT appear as an explicit kernel megaflow with `actions:drop`. Instead, the kernel does not install any megaflow for the prohibited traffic; every dropped packet upcalls to userspace, matches the priority-200 rule, and is silently dropped without a forwarding action. Understanding this is important for performance analysis: a heavily-dropped flow under load can saturate the upcall pipeline.

**Outcomes.** The megaflow dump shows the permitted flows but not the dropped flows. An `ofproto/trace` of the dropped packet shows the priority-200 match and `Datapath actions: drop`.

The `Datapath actions: drop` line confirms the userspace decision.

**Misconception to refute.** "Dropped traffic is invisible at the kernel datapath level." Mostly right but not entirely. The kernel does not cache a megaflow for the dropped traffic, but the per-flow drop counter on the OpenFlow rule (visible via `ovs-ofctl dump-flows`) does increment, so the operator has visibility into how many packets were dropped. The kernel datapath does NOT increment any drop counter for ACL-dropped traffic; only the userspace flow-rule counter sees it.

**Finish.** Drop behaviour traced through the kernel datapath and the flow-table walk.

For comparison, this exercise installs an iptables FORWARD rule that drops the same ICMP traffic. The point is to show the alternative in-kernel mechanism: instead of an OpenFlow flow at OVS's layer, iptables operates at the netfilter framework's bridge-forward hook.

**Misconception to refute.** "OVS flow rules and iptables rules can be combined into a unified ACL." Yes in practice, but with a gotcha. The OVS flow rule runs first (in userspace as part of the flow translation), and if the action is `output:N`, the kernel emits the packet on the target interface. iptables FORWARD rules then see the packet at the netfilter forward hook (if `bridge-nf-call-iptables=1` is set, which is the kernel default on most distributions). ## R2.5.7.4. Side by side: Cisco ACL, OVS OpenFlow flow, Linux iptables

| Concern                       | Cisco IOS extended ACL                                  | OVS OpenFlow flow                                           | Linux iptables                              |

| Match field flexibility       | Limited: src/dst IP, proto, src/dst port, TCP flags     | Extensive: full OpenFlow match field catalog                | Extensive: source IP, dst IP, port, conntrack state, etc. |
| Stateful (ESTABLISHED match)  | No (extended ACLs are stateless)                         | No (use ct() action for stateful, see R2.5.8)               | Yes (`-m conntrack --ctstate ESTABLISHED`)   |

The takeaway: all three implementations achieve the same goal (drop matching packets) at three different layers. Cisco ACLs are at the hardware layer, OVS OpenFlow flows are at the userspace bridge layer, iptables are at the kernel netfilter layer. The same conceptual rule is expressed in three different syntaxes; choosing among them depends on the deployment context.

## R2.5.7.5. Common misconceptions for the audience

**Misconception 1.** "OVS ACL rules are stored in OVSDB." Wrong. OpenFlow flows live in the bridge's flow table, not in OVSDB. The flow table is in-memory state of the `ovs-vswitchd` daemon. OVSDB only stores the OpenFlow controller connection configuration plus the bridge's high-level settings; the actual flows are programmed through the OpenFlow channel. Restarting `ovs-vswitchd` would lose all manually-installed flows; production deployments use a controller to re-install them on reconnect.

**Misconception 2.** "drop is faster than output:none." Right; they are the same action. Both produce empty action sets, which OpenFlow defines as implicit drop. There is no separate "drop" opcode in the OpenFlow protocol; it is just an action set with no actions. The `drop` keyword in `ovs-ofctl` is OVS's syntactic sugar for an empty action set.

**Misconception 3.** "I can match on connection state with stateless ACL." Wrong. Stateless ACLs match only on per-packet header fields. For stateful matching (ESTABLISHED, RELATED, NEW), use the conntrack extension covered in R2.5.8 with the `ct()` action.

**Misconception 4.** "Higher priority rules silently override lower priority rules." Mostly right, but with a nuance about the meaning of "override". OpenFlow says: when multiple rules match, only the highest-priority rule is executed. There is no "fall-through" of unmatched conditions like in iptables. So a high-priority drop rule completely replaces what the low-priority NORMAL rule would have done; the NORMAL action does NOT execute as a fallback after the drop. This is different from iptables, where rules are evaluated in order until a `-j ACCEPT` or `-j DROP` terminal target is hit.

## R2.5.7.6. Production-readiness assessment

The trade-off cost of OpenFlow ACLs at scale. Each new flow rule adds a per-rule entry to the bridge's classifier (TSS, the Tuple Space Search algorithm OVS uses for fast lookup). For small rule counts (under a few hundred) the cost is negligible. For thousands of rules, the classifier's lookup time grows; for tens of thousands of rules, the kernel datapath upcalls become expensive. Mitigation: aggregate rules where possible (use prefix matches instead of host matches, group by protocol or port range), or use a dedicated controller that batches rule installations.

The failure mode under sustained heavily-dropped traffic. Every dropped packet upcalls to userspace because no kernel megaflow gets installed for drop actions. A flow being heavily attacked or misconfigured can produce a sustained stream of upcalls, saturating the userspace daemon's CPU. Mitigation: install a kernel-level drop via iptables for known-attack patterns; reserve OpenFlow drops for selective filtering. Modern OVS versions include the ability to install a kernel megaflow with `actions:drop` (the recirculation-based mechanism) but this is a recent feature and not covered in this lab.

The monitoring discipline. Every ACL rule should have its `n_packets` counter polled regularly. A drop counter that grows faster than expected indicates either an attack, a misconfiguration upstream, or a previously-permitted flow that is now being blocked. The OVS-side instrumentation is `ovs-ofctl dump-flows` polled every 30 seconds.

## R2.5.7.8. Continuity to the next sub-capture

R2.5.7 covered stateless ACL on OVS via OpenFlow flow rules. The next sub-capture, R2.5.8, covers stateful ACL with conntrack plus NAT. The connection is direct: stateless ACL is the foundation, and the `ct()` OpenFlow action extends it with connection tracking. The conntrack module hashes the packet's 5-tuple plus the conntrack zone into a hash table, looks up the connection's state (NEW, ESTABLISHED, RELATED, INVALID), and lets the operator's flow rules match on that state.

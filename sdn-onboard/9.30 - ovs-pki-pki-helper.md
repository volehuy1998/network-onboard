Sau khi hoàn thành phần này, kỹ sư có thể:

1. **Hiểu (Understand)** vai trò của `ovs-pki` như một **wrapper tiện dụng quanh OpenSSL** chuyên cho việc bootstrap + duy trì PKI hierarchy phục vụ kết nối SSL/TLS giữa các thành phần OVS/OVN (ovs-vswitchd ↔ controller, ovsdb-server ↔ ovs-vsctl, chassis ↔ SB DB cluster).
2. **Áp dụng (Apply)** workflow CA bootstrap (`init` → `req` → `sign` → distribute) để chuẩn bị certificate cho cụm OVN production trong môi trường multi-chassis với mTLS authentication.
3. **Phân tích (Analyze)** cấu trúc thư mục PKI hierarchy (`/var/lib/openvswitch/pki/` mặc định) để hiểu tách biệt giữa controllerca và switchca, lý do cần hai CA độc lập.
4. **Đánh giá (Evaluate)** trade-off `req+sign` (combine 1 step, dễ làm nhưng kém security do cần chuyển private key qua CA) so với `req` + `sign` riêng (đúng practice production).

Yêu cầu kiến thức: hiểu PKI cơ bản (CA, certificate, private key), hiểu OpenSSL workflow, có quyền root trên máy lab.

## 9.30.1 Vì sao cần `ovs-pki` — không chỉ là wrapper OpenSSL

Trong cụm OVN production, kết nối giữa các thành phần được khuyến cáo dùng SSL/TLS với mTLS (mutual TLS — cả client lẫn server xác thực bằng certificate):

- `chassis ovn-controller` ↔ `cụm SB DB ovsdb-server` (port 6642)
- `client ovn-nbctl` ↔ `cụm NB DB ovsdb-server` (port 6641)
- `local ovs-vsctl` ↔ `local ovsdb-server` (mặc định Unix socket, optional SSL nếu remote management)
- `OpenFlow controller` ↔ `ovs-vswitchd` (port 6653)

Mỗi kết nối yêu cầu (a) certificate cho server, (b) certificate cho client, (c) cả hai phải tin một CA chung. Tự setup PKI bằng `openssl` thuần có 5-7 lệnh dài, dễ sai (config file `openssl.cnf` phức tạp, naming convention dễ nhầm).

`ovs-pki` đơn giản hóa: định sẵn directory layout, naming convention, default key size 2048 RSA, log file, batch mode cho automation. Vẫn dùng OpenSSL bên dưới (không reinvent crypto), chỉ wrapper.

**Khác biệt PKI hierarchy của OVS:** `ovs-pki init` tạo **hai CA độc lập** trong cùng PKI tree:
- `controllerca`: ký certificate cho OpenFlow controller + ovsdb client (tools admin như ovn-nbctl, ovs-vsctl)
- `switchca`: ký certificate cho ovs-vswitchd + ovsdb-server (server side)

Tách hai CA là **defense-in-depth**: nếu một CA bị compromise (ví dụ private key leaked), chỉ một bên (controller hoặc switch) phải re-issue, không phải toàn bộ cluster.

## 9.30.2 Synopsis + commands

```
ovs-pki <options> <command> <args>...
```

**Commands (verified từ man page):**

| Command | Mục đích |
|---------|----------|
| `init` | Khởi tạo PKI hierarchy mới với 2 CA (controllerca + switchca) |
| `req <name>` | Sinh private key + certificate request (CSR) cho `<name>` |
| `sign <name> [<type>]` | Ký CSR thành certificate; `<type>` = `switch` hoặc `controller` |
| `req+sign <name> [<type>]` | Kết hợp `req` + `sign` trong 1 lệnh (dùng trong dev/lab) |
| `verify <name> [<type>]` | Verify certificate hợp lệ (ký bởi đúng CA) |
| `fingerprint <file>` | In SHA-1 digest của file (cert hoặc bất kỳ) |
| `self-sign <name>` | Ký self-signed cho test (không khuyên cho production) |

> **Lưu ý chính xác:** plan v3.5 ban đầu liệt kê `set-default` command. Verify man page `ovs-pki(8)` (OVS 2.17.9) cho thấy KHÔNG có command này. Đây là sai lệch giữa plan estimate và source-of-truth man page; file 9.30 fix theo man page.

## 9.30.3 Options chi tiết

| Option | Mặc định | Vai trò |
|--------|----------|---------|
| `-k <type>`, `--key=<type>` | `rsa` | Algorithm: `rsa` hoặc `dsa` |
| `-B <nbits>`, `--bits=<nbits>` | `2048` | Key size; min 1024 |
| `-D <file>`, `--dsaparam=<file>` | tự sinh | Alternative dsaparam.pem cho DSA |
| `-b`, `--batch` | OFF | Skip interactive fingerprint verification (cho automation) |
| `-d <dir>`, `--dir=<dir>` | `/var/lib/openvswitch/pki` | PKI hierarchy location |
| `-f`, `--force` | OFF | Ghi đè file/directory existing |
| `-l <file>`, `--log=<file>` | mặc định trong PKI dir | Log file path |
| `-u`, `--unique` | OFF | Dùng simple name format cho CN field |
| `-h`, `--help` | — | In help |

**Practical setting cho production:**

- Luôn dùng `-B 4096` (4096-bit RSA) thay vì default 2048 cho long-term key (>2 năm).
- Luôn dùng `-l /var/log/ovs-pki.log` để track audit trail.
- KHÔNG dùng `-b` trong production (skip fingerprint verify = giảm bảo mật).

## 9.30.4 PKI hierarchy directory layout

Sau `ovs-pki init`, structure mặc định:

```
/var/lib/openvswitch/pki/
├── controllerca/
│   ├── cacert.pem               # Public CA cert (controller)
│   ├── private/
│   │   └── cakey.pem            # CA private key (controller) — STRICT 0600
│   ├── newcerts/                # Issued certs index
│   ├── crl/                     # Certificate Revocation List
│   ├── serial                   # Next serial number
│   └── index.txt                # OpenSSL CA index
├── switchca/
│   ├── cacert.pem               # Public CA cert (switch)
│   ├── private/
│   │   └── cakey.pem            # CA private key (switch) — STRICT 0600
│   ├── newcerts/
│   └── ...
└── log
```

**Critical permission:** `private/cakey.pem` của cả hai CA PHẢI có mode `0600` và owner `root` (hoặc dedicated `ovs-pki` user). Nếu mode > 0644, attacker đọc được CA private key → ký certificate giả mạo cho bất kỳ chassis nào → lateral movement trong cụm.

`ovs-pki init` set permission đúng theo mặc định, nhưng `cp` thủ công sang máy khác có thể mất bit.

## 9.30.5 Anatomy: `ovs-pki req+sign` workflow

**Bucket:** OVS | **Context:** lab/dev hoặc bootstrap chassis mới | **Purpose:** sinh private key + cert + ký bởi CA trong 1 lệnh duy nhất | **Activity:** provisioning | **Mechanism:** wrapper OpenSSL `req` + `ca` commands.

**Output mẫu:**

```
$ sudo ovs-pki req+sign hv01 switch
Generating private key for hv01...
Generating certificate request for hv01...
Signing certificate request for hv01 with switch CA...
Verifying that signed certificate is valid for switch use...

Created files in current directory:
  hv01-privkey.pem    # Private key (KEEP SECRET)
  hv01-req.pem        # Certificate request (CSR)
  hv01-cert.pem       # Signed certificate
```

**Decode 5-attribute Anatomy:**

| Attribute | Giá trị mẫu | Healthy / Warning / Critical |
|-----------|-------------|------------------------------|
| Output files | 3 file: privkey, req, cert | Healthy: cả 3 file tồn tại trong CWD |
| Privkey permission | `0600` mode | Critical: nếu `0644` → key leak risk; chmod 600 ngay |
| Cert validity period | Mặc định OVS = 1 năm | Warning: cần renew trước hạn; Critical: hết hạn = mất kết nối |
| Cert signed CA | switchca | Verify: `openssl x509 -in hv01-cert.pem -text | grep Issuer` |
| Cert subject (CN) | hostname hoặc unique ID | Cần khớp với chassis name trong OVN |

**Critical signal:**
- Lệnh `req+sign` PHẢI chạy trên CA machine (vì cần CA private key để ký). KHÔNG chạy trên chassis production để tránh expose CA key.
- Mode đúng cho production: `req` trên chassis (không expose CA key) → copy CSR sang CA machine → `sign` trên CA → copy cert ngược lại.

## 9.30.6 Bài tập có hướng dẫn (Guided Exercise)

**Mục tiêu:** bootstrap PKI cho lab 1-CA + 1 ovsdb-server SSL endpoint, kết nối thành công bằng `ovs-vsctl --ssl-*`.

**Setup CA:**

```bash
# 1. Init PKI hierarchy (chỉ làm 1 lần trên CA machine)
sudo ovs-pki init -d /tmp/lab-pki

# Verify structure
ls -la /tmp/lab-pki/controllerca/
ls -la /tmp/lab-pki/switchca/private/cakey.pem  # mode 0600
```

**Sinh certificate cho server (ovsdb-server):**

```bash
# 2. req+sign cho server (lab convenience)
cd /tmp/lab-server
sudo ovs-pki -d /tmp/lab-pki req+sign server01 switch

# Output: server01-privkey.pem, server01-req.pem, server01-cert.pem
```

**Sinh certificate cho client (ovs-vsctl):**

```bash
# 3. req+sign cho client
cd /tmp/lab-client
sudo ovs-pki -d /tmp/lab-pki req+sign client01 controller
```

**Configure ovsdb-server với SSL:**

```bash
# 4. Khởi động ovsdb-server với SSL listen
sudo ovsdb-server /etc/openvswitch/conf.db \
  --remote=pssl:6640 \
  --private-key=/tmp/lab-server/server01-privkey.pem \
  --certificate=/tmp/lab-server/server01-cert.pem \
  --ca-cert=/tmp/lab-pki/controllerca/cacert.pem \
  --pidfile --detach
```

**Test connection từ client:**

```bash
# 5. Verify ovs-vsctl connect được qua SSL
sudo ovs-vsctl \
  --db=ssl:127.0.0.1:6640 \
  --private-key=/tmp/lab-client/client01-privkey.pem \
  --certificate=/tmp/lab-client/client01-cert.pem \
  --ca-cert=/tmp/lab-pki/switchca/cacert.pem \
  list-br

# Expected output: list of bridges (no SSL error)
```

**Verify certificate detail:**

```bash
# 6. Inspect cert
openssl x509 -in /tmp/lab-server/server01-cert.pem -text -noout
sudo ovs-pki -d /tmp/lab-pki verify server01 switch
sudo ovs-pki fingerprint /tmp/lab-server/server01-cert.pem
```

## 9.30.7 Mẫu cần tránh và lưu ý

**Mẫu cần tránh 1: Dùng `req+sign` trên chassis production.**

`req+sign` cần CA private key để ký → nghĩa là CA key bị copy sang chassis trong workflow này. Production phải tách: `req` trên chassis (sinh + giữ private key local), CSR copy qua CA machine (qua secure channel), `sign` chỉ trên CA, cert ký xong copy ngược lại.

**Cách phòng:** trong runbook deploy chassis, thêm step rõ ràng "DO NOT run req+sign on production hypervisor".

**Mẫu cần tránh 2: Self-signed cert trong production cluster.**

`ovs-pki self-sign` tạo cert tự ký, không cần CA. Tiện cho test 1-node, nhưng production cluster với mTLS yêu cầu cert được ký bởi CA chung — self-signed không scale tới N chassis.

**Cách phòng:** chỉ dùng self-sign cho lab/CI test, document rõ trong cluster bootstrap guide.

**Lưu ý 3: Cert rotation.**

Cert OVS mặc định valid 1 năm. KHÔNG có auto-renewal — operator phải set calendar reminder + script rotation:

```bash
# Script auto-renew cron monthly
ovs-pki -d /var/lib/openvswitch/pki req+sign hv-$(hostname) switch
# Then update ovs-vswitchd config + restart
```

**Lưu ý 4: CRL (Certificate Revocation List) không tự động update.**

Khi compromise cert (chassis bị decom hoặc bị hack), phải manually:
1. Add cert serial vào CRL: `openssl ca -gencrl -out crl.pem -config switchca.conf`
2. Distribute CRL tới mọi server endpoint
3. Server load CRL bằng option `--ca-cert-bootstrap` (advanced)

OVS không có centralized CRL service; production thường dùng external PKI (HashiCorp Vault, step-ca) thay vì ovs-pki cho rotation tự động.

**Lưu ý 5: Backup CA private key offline.**

Mất `cakey.pem` = mất khả năng sign cert mới = phải re-init toàn bộ PKI + re-issue mọi cert chassis. Backup `cakey.pem` ra cold storage (USB encrypted, vault) ngay sau `init`.

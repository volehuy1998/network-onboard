## 11.1.1 MTU math cho overlay

Compass Ch 13: "95% of 'tunnel up but no traffic' is MTU."

Ethernet default MTU 1500 bytes. Với overlay encap:

```
Inner packet (tenant)
  +--- Ethernet (14B) + IP (20B) + TCP (20B) + data

→ wrapped in:

Outer packet
  +--- Ethernet (14B) + IP (20B) + UDP (8B) + VXLAN (8B) + [Inner...]
       ^^^                                                ^^^
       Underlay MTU must fit everything here
```

VXLAN overhead = 50 bytes added.

**Rule**: Inner MTU ≤ Underlay MTU − overhead.

- Underlay 1500, tenant inner = 1450 (VXLAN 50B).
- Underlay 1600 (jumbo), tenant inner = 1500.
- Underlay 9000 (jumbo), tenant inner = 8950.

Production solution:

1. **Shrink tenant MTU** → 1450. TCP MSS negotiate dựa MTU này.
2. **Bump underlay MTU** → 1600+ (hoặc 9000 jumbo). Tenant giữ 1500 default.

## 11.1.2 PMTUD (RFC 1191)

PMTUD auto-discover smallest MTU trên path:

1. Sender gửi packet với DF flag set.
2. Router có MTU < packet size → drop + gửi ICMP "Fragmentation Needed" (Type 3 Code 4) với correct MTU.
3. Sender giảm packet size, retry.

**Yêu cầu**: ICMP Type 3 Code 4 **không bị block** ở firewall. Failure mode phổ biến — admin firewall block ICMP → PMTUD silent fail → application timeout.

**RFC 4821 PLPMTUD** (Packetization Layer PMTUD) — fallback khi ICMP blocked. Gửi probe packet, nếu không ACK → giảm size. Linux kernel default từ 3.x+:

```bash
[host]# sysctl net.ipv4.ip_no_pmtu_disc       # 0 = PMTUD enabled
[host]# sysctl net.ipv4.tcp_mtu_probing       # 1 = PLPMTUD khi ICMP fail
[host]# ip route get 10.0.0.100                 # xem MTU hiện tại
```

## 11.1.3 Tunnel-specific tuning

OVS tunnel options:

```bash
[host]# ovs-vsctl set Interface vxlan0 \
         options:tos=inherit \                 # copy inner ToS → outer
         options:ttl=64 \                      # fixed outer TTL
         options:df_default=true               # set DF outer header
```

`options:df_default=true` (**khuyến nghị**): set DF in outer IP. Khi path có MTU thấp, underlay router gửi ICMP → tunnel endpoint nhận, adjust.

`options:df_default=false`: outer packet fragment silently → inner PMTUD broken.

## 11.1.4 NIC offload tương tác tunnel

NIC hiện đại có nhiều offload:

- **TSO** (TCP Segmentation Offload): CPU gửi large TCP (64KB), NIC segment thành MSS-sized.
- **GSO** (Generic): software equivalent, trong kernel.
- **LRO** (Large Receive Offload): NIC coalesce packet thành large frame.
- **GRO** (Generic): kernel-side.
- **checksum offload**: NIC tính checksum thay CPU.

Tunnel encap gây complexity: NIC cần biết về VXLAN/Geneve header để skip đúng khi compute checksum.

**NIC hỗ trợ tunnel offload** (ConnectX-4+, Intel XL710+, Broadcom Thor+):

- Inner checksum by NIC.
- Inner TSO: CPU gửi large VXLAN packet, NIC segment inner TCP.
- Performance gain 2-5× vs software encap.

Check:

```bash
[host]# ethtool -k eth0 | grep -E 'tx-udp|tx-gso'
# tx-udp_tnl-segmentation: on ← hỗ trợ TSO cho tunnel
```

Disable cho khắc phục sự cố:

```bash
[host]# ethtool -K eth0 tx-udp_tnl-segmentation off
```

## 11.1.5 Case study: slow TCP qua tunnel

Symptom: UDP OK, TCP small OK, TCP large download cực chậm.

Diagnosis path:

1. **Check MTU**: `ip link show`. Nếu 1500 và underlay 1500 → MTU issue.
2. **Tenant DF**: tcpdump xem TCP có DF=1 không.
3. **ICMP path**: traceroute, nếu ICMP blocked → PMTUD broken.
4. **tcp_mtu_probing**: enable PLPMTUD fallback.
5. **Quick fix**: reduce tenant MTU 1450, hoặc bump underlay 1600.

Long-term: tune underlay jumbo 9000, tenant 1500, best performance.

## 11.1.6 PMTUD packet flow chi tiết (RFC 1191 + RFC 8201 IPv6)

### IPv4 PMTUD flow (RFC 1191, 11/1990)

Sender muốn truyền packet 1500 byte với DF=1, nhưng path có 1 segment MTU=1400.

```
Sender (1500B, DF=1)
   │
   │ Hop 1: MTU 1500 ✓ forward
   ▼
Router-1 (egress MTU 1400, packet 1500 + DF=1)
   │
   │ DROP packet
   │ Generate ICMPv4 Type 3 Code 4 "Fragmentation Needed and DF set":
   │   - Header (8 byte) + original IP header + first 8 byte payload
   │   - Next-Hop MTU field = 1400 (RFC 1191 extension)
   │ Source IP = router IP, Dest IP = sender IP
   │
   ▼
Sender receives ICMP
   │
   │ Kernel updates route cache MTU = 1400 cho dest IP đó
   │ TCP layer detect, retry với MSS giảm tương ứng
   │
   ▼
Sender (1400B retry) ─► Hop 1 ✓ ─► Router-1 ✓ ─► Hop 2 ✓ ─► ... ─► Receiver
```

`ip route get <dest>` show MTU đã update:

```bash
[host]# ip route get 10.0.0.100
10.0.0.100 via 192.168.1.1 dev eth0 src 192.168.1.10 mtu 1400 ...
```

### IPv6 PMTUD flow (RFC 8201, 07/2017)

IPv6 router KHÔNG fragment giống IPv4 (DF semantics implicit). Mọi router gặp packet > MTU đều drop + send ICMPv6 Type 2 "Packet Too Big". Format gần giống ICMPv4 nhưng MTU field 32-bit thay vì 16-bit.

### Kernel route cache

Linux kernel cache PMTU per-destination, hết hạn 10 phút (`net.ipv4.route.mtu_expires`):

```bash
[host]# sysctl net.ipv4.route.mtu_expires
net.ipv4.route.mtu_expires = 600
```

Sau 10 phút, kernel thử lại MTU original. Nếu path đã thay đổi (less restrictive), kernel học được MTU mới. Nếu vẫn restrictive, lại nhận ICMP, lại cập nhật cache.

## 11.1.7 PMTU black hole, failure mode phổ biến nhất

**Định nghĩa:** PMTUD không hoạt động vì ICMP Type 3 Code 4 (IPv4) hoặc Type 2 (IPv6) bị firewall drop trên path. Sender không nhận được feedback, vẫn gửi packet 1500 byte với DF=1, mọi packet bị drop ở router intermediate. Connection hang vô thời hạn.

### Symptom phổ biến

- TCP handshake (SYN, SYN-ACK, ACK đều nhỏ < 100 byte) **thành công**.
- Application layer truyền small request thành công (HTTP GET path = 50 byte).
- Đến lúc HTTP response > 1400 byte hoặc TLS handshake (ServerHello + cert chain ~1.5-3 KB) → packet drop silently.
- Application timeout sau 30-60 giây.

### Root cause matrix

| Vị trí block ICMP | Lý do | Fix |
|-------------------|-------|-----|
| Sender side firewall (host iptables) | Admin block all ICMP "for security" | Allow ICMP Type 3 Code 4 in/out |
| Receiver side firewall | Tương tự | Allow ICMP Type 3 Code 4 |
| Cloud security group (AWS, Azure, GCP) | Default rule | Allow ICMP Fragmentation Needed |
| ISP/carrier middlebox | Anti-DDoS rate-limit ICMP quá strict | Liên hệ ISP, hoặc dùng workaround |
| Stateful firewall không track ICMP | Stateless reply không match conntrack | Stateful conntrack ICMP module |

### Workaround: TCP MSS clamping

Nếu không control được middlebox ICMP, clamp TCP MSS xuống mức path-safe:

```bash
# IPv4: clamp MSS = MTU - 40 (IP 20 + TCP 20)
[host]# iptables -t mangle -A POSTROUTING -p tcp --tcp-flags SYN,RST SYN \
        -j TCPMSS --clamp-mss-to-pmtu

# Hoặc clamp cố định (nếu PMTU discovery fail):
[host]# iptables -t mangle -A POSTROUTING -p tcp --tcp-flags SYN,RST SYN \
        -j TCPMSS --set-mss 1400
```

`TCPMSS` mangle target rewrite TCP MSS option trong SYN/SYN-ACK xuống mức an toàn. TCP sẽ tự segment xuống size đó, không cần PMTU feedback.

OVN cũng support TCP MSS clamping qua Logical Router option:

```bash
[host]# ovn-nbctl set Logical_Router router1 \
        options:mss_max_size=1400
```

OVN inject flow ở `lr_in_chk_pkt_len` để clamp MSS trên path overlay → underlay → public, tránh cần ICMP from external network.

## 11.1.8 OVN `check_pkt_larger` action (lr_in_chk_pkt_len stage)

Logical Router pipeline OVN có stage chuyên cho MTU check. Khi packet IPv4/IPv6 vào LR và size > LRP MTU egress, OVN trigger:

1. **Action `check_pkt_larger(...)`**: kiểm tra size packet so với threshold.
2. Nếu vượt → branch sang `lr_in_larger_pkts` stage.
3. Stage này build ICMP "Packet Too Big" (Type 3 Code 4 IPv4 hoặc Type 2 IPv6) gửi ngược lại sender.
4. Drop packet original.

Source code anchor (`branch-22.03`):

```c
// northd/northd.c, function build_icmperr_pkt_big_flows()
ovs_lswitch_lflow_add(lflows, od, S_ROUTER_IN_LARGER_PKTS, ..., 
    "ip4 && ip.ttl > 0",
    "icmp4 { ... eth.dst = E; eth.src = R; ... }; next;");
```

Tên function gốc bị typo trong upstream: `reply_imcp_error_if_pkt_too_big()` (`imcp` thay vì `icmp`). Verified branch-22.03, KHÔNG sửa typo khi cite (Rule 14 §14.2).

LRP MTU configurable qua NB:

```bash
[host]# ovn-nbctl set Logical_Router_Port lrp-external \
        options:gateway_mtu=1442
```

Khi MTU giảm (ví dụ 1442 cho underlay 1500 + Geneve overhead 58), LRP trigger ICMP từ chính OVN router, không cần underlay router gửi ICMP.

## 11.1.9 NIC offload deep, tunnel-aware behavior

Phần 11.1.4 đã liệt kê các offload high-level. Phần này đi sâu vào hành vi cụ thể.

### TX side: TSO + checksum offload cho Geneve

Kernel software path khi gửi 1 TCP packet 64KB qua Geneve tunnel:

1. Application `write()` 64KB.
2. Kernel TCP stack chia 64KB thành các segment MTU-sized (1442 byte cho Geneve overhead 58).
3. Kernel encap mỗi segment với Geneve header (8B base + 8B TLV) + outer UDP + outer IP.
4. Kernel tính checksum inner TCP + outer UDP cho mỗi segment.
5. NIC nhận từng packet 1500-byte → TX.

Tổng work: ~46 segment × (encap + checksum) trên CPU.

NIC tunnel-aware path (ConnectX-5+, E810):

1. Application `write()` 64KB.
2. Kernel TCP stack đẩy nguyên 64KB xuống skb với metadata "tunnel inside, encap descriptor: Geneve VNI=X, outer src/dst, MSS=1442".
3. **NIC** chia 64KB thành 46 segment, encap mỗi cái với Geneve header, tính checksum inner+outer.
4. NIC TX 46 packet 1500-byte.

CPU work giảm 46x. Throughput từ ~5 Gbps (software) lên 25-40 Gbps (hardware) trên link 25/40G.

### RX side: GRO + RSS

GRO (Generic Receive Offload) merge nhiều packet small thành 1 packet large trước khi đẩy lên TCP stack:

- Software GRO: kernel `skb_gro_receive()`, hợp packet cùng connection.
- NIC GRO (ConnectX-5+, E810): NIC tự merge ở hardware queue, cho stack 1 large skb.

Cho Geneve, NIC phải biết outer header để strip + merge inner TCP đúng. Không phải mọi NIC support, check `ethtool -k`:

```
[host]# ethtool -k eth0 | grep gro
generic-receive-offload: on
rx-gro-list: off
rx-gro-hw: off [fixed]      ← NIC không hỗ trợ hardware GRO Geneve
```

`rx-gro-hw=off [fixed]` nghĩa NIC không hỗ trợ; chỉ software GRO. Throughput RX cho TCP large file ~10-15 Gbps.

`rx-gro-hw=on` (ConnectX-6+, E810 with SR-IOV PF): hardware GRO Geneve, throughput RX ~25-40 Gbps.

### RSS hash inner flow

RSS (Receive Side Scaling) phân tải traffic lên multiple CPU queue. Cho Geneve, NIC hash trên flag config:

```bash
[host]# ethtool -X eth0 hkey ... rss-hash-key ...
[host]# ethtool -n eth0 rx-flow-hash udp4
UDP over IPv4 flows use these fields for computing Hash flow key:
IP SA
IP DA
L4 bytes 0 & 1 [TCP/UDP src port]
L4 bytes 2 & 3 [TCP/UDP dst port]
```

Mặc định RSS hash trên outer 5-tuple. Nếu mọi traffic có cùng outer (vì Geneve tunnel single-source), RSS chỉ phân lên 1 queue → bottleneck single CPU.

Solution: enable inner-flow RSS (NIC support, kernel 5.4+):

```bash
[host]# ethtool -K eth0 rx-vxlan-tnl-rss on
[host]# ethtool -K eth0 rx-geneve-tnl-rss on
```

Hash trên inner 5-tuple → traffic phân lên N queue đúng.

## 11.1.10 Anatomy Template A: `ip route get` + `ethtool -k`

> **Bối cảnh khi chạy:** Kiểm tra MTU effective tới destination + offload state của NIC.

### `ip route get <dest>`

```bash
[host]# ip route get 192.0.2.20
192.0.2.20 via 10.0.0.1 dev eth0 src 10.0.0.10 mtu 1442
   cache <expires 587sec>
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| `via <next-hop>` | IP underlay routable | (n/a) | Empty / unreachable | `ip route` không có route; check default gateway |
| `dev <iface>` | Match expected uplink | Khác (multi-NIC) | Empty | Routing decision sai |
| `src <local-IP>` | Chassis underlay IP | Khác (rare) | Empty | Source IP unbound |
| `mtu <N>` | Match underlay - overhead | < expected (PMTU drop) | Empty (default 1500) | PMTU đã learn được; hoặc chưa có cache |
| `cache <expires N sec>` | < 600 sec (default) | Gần 0 (sắp hết) | Negative (đã expire) | Kernel sẽ retry MTU original |

**Kịch bản:**

- `mtu 1500` cho destination cross-chassis (nên là 1442): chưa có PMTU feedback, hoặc cache hết hạn. Force refresh: `ip route flush cache && ip route get <dest>` lần sau sẽ refresh.
- `mtu 0`: MTU không xác định, fallback 1500. Có thể PMTU black hole.

### `ethtool -k <iface>` filter offload

```bash
[host]# ethtool -k eth0 | grep -E '(tx-udp|rx-udp|gro|gso|tso|checksum).*tnl'
tx-udp_tnl-segmentation: on
tx-udp_tnl-csum-segmentation: on
rx-udp-tunnel-port-offload: on
```

| Feature | Healthy | Warning | Critical | Diagnostic hint |
|---------|---------|---------|----------|-----------------|
| `tx-udp_tnl-segmentation` | `on` | `off` | `off [fixed]` | NIC không support hoặc driver disable; throughput TX 50-70% giảm |
| `tx-udp_tnl-csum-segmentation` | `on` | `off` | (n/a) | Inner checksum software, CPU overhead |
| `rx-udp-tunnel-port-offload` | `on` | `off` | (n/a) | NIC không recognize Geneve port → không offload RX |
| `rx-gro-hw` | `on` | `off` | `off [fixed]` | RX throughput limited; xem 11.1.9 |
| `rx-vxlan-tnl-rss` | `on` (cho VXLAN) | `off` | (n/a) | RSS hash outer-only, single CPU bottleneck |
| `rx-geneve-tnl-rss` | `on` (cho Geneve) | `off` | (n/a) | Tương tự cho Geneve |

## 11.1.11 ▶ Capstone POE: Shrink tenant MTU vs Bump underlay MTU

> **Bối cảnh:** Production cluster OVN cross-DC. Underlay 1500 default. Workload Pod MTU 1500 default. PMTU black hole quan sát qua slow TCP large transfer.

**Tuyên bố cần kiểm chứng:** "Bump underlay MTU lên 9000 (jumbo) là cách giải quyết best performance, không cần shrink tenant."

**Predict:** Đúng nếu underlay control được, sai nếu underlay là cloud public (AWS, Azure) không control được.

**Observe (giả định, kiểm chứng khi có lab):**

- **On-prem DC + private cabling:** Bump underlay 1500 → 9000 trên switch + NIC: throughput tăng 3-5x cho TCP large transfer (cờ jumbo). Tenant giữ 1500 default. Mọi thứ healthy.
- **Cloud public AWS VPC:** AWS underlay MTU = 9001 default. Inter-region underlay MTU = 1500 (qua VPN/Transit Gateway). Bump VPC MTU 9001 → workable trong region. Cross-region vẫn black hole.
- **Hybrid on-prem + public:** Underlay heterogeneous (private 9000 vs public 1500). Bump không khả thi đồng bộ → fallback shrink tenant 1442.

**Explain:**

1. **Bump underlay** chỉ work khi kỹ sư control toàn underlay. Cho on-prem 100%, đây là best solution: tenant MTU 1500 default, app không cần aware.
2. **Shrink tenant** work mọi nơi (kể cả cloud public không control underlay), nhưng MTU 1442 là không-default → app/library cần handle (TCP MSS clamping tự động giúp 80% case, 20% còn lại app-level).
3. **TCP MSS clamping** (OVN `mss_max_size`) là middleground: tenant MTU giữ 1500 nhưng MSS clamp xuống 1402 (1442 - 40 TCP+IP) → TCP segments fit underlay không cần PMTU feedback.

**Falsification test:**

- Bump underlay 9000, run iperf3 cross-DC. Nếu throughput tăng và không có TCP retransmission → bump effective.
- Nếu một số đường vẫn 1500 (cross-region, public link), bump partial → fallback PMTU black hole vẫn xảy ra trên link đó.

**Tuyên bố sai ở đâu:** "Best performance" nhìn underlay private. Đúng phải xét: **(a) underlay topology heterogeneous không?** Nếu yes → shrink tenant + MSS clamp là robust hơn bump. **(b) Tenant aware về MTU < 1500 không?** Nếu legacy app hard-code MTU = 1500 → MSS clamping không đủ.

> **Quy tắc:** On-prem private DC controllable → bump underlay (best perf). Cloud public hoặc heterogeneous → shrink tenant + MSS clamp (robust).

## Điểm cốt lõi cần nhớ

Thứ nhất, overlay MTU math: Inner ≤ Underlay − overhead. Phần lớn "tunnel up no traffic" là MTU.

Thứ hai, PMTUD RFC 1191 requires ICMP không blocked. PLPMTUD (RFC 4821) là fallback khi ICMP blocked.

Thứ ba, OVS `options:df_default=true` cho overlay PMTUD đúng. Không set → silent fragment.

Thứ tư, NIC tunnel offload (ConnectX-4+, Intel XL710+) cho 2-5× băng thông thực tế. Check `ethtool -k` trước triển khai scale.

## 13.6.1 Vấn đề: gateway chassis SPOF

Phần 13.5 phân tích `chassisredirect` Port_Binding, chỉ định một chassis làm active gateway cho north-south NAT. Nếu chassis đó chết:
- Traffic north-south fails (external traffic không reach VM, VM không reach internet).
- SNAT connection break (NAT state in kernel conntrack của chassis đó mất).
- Manual failover cần admin intervene → MTTR 10-60 phút.

**HA_Chassis_Group** giải quyết: group 2-5 chassis với priority, OVN + BFD tự động failover. Target: MTTR < 2 giây tự động, không need admin action.

## 13.6.2 HA_Chassis_Group schema

NBDB schema:

```
HA_Chassis_Group:
  name          , group identifier
  ha_chassis    , set<HA_Chassis UUID>
  external_ids  , map cho CMS annotation

HA_Chassis:
  chassis_name  , reference Chassis.name trong SBDB
  priority       — 0-32767, higher = preferred
  external_ids  , map cho CMS annotation
```

### Cấu hình

```bash
[admin]# ovn-nbctl ha-chassis-group-add hcg-external
[admin]# ovn-nbctl ha-chassis-group-add-chassis hcg-external chassis-1 100
[admin]# ovn-nbctl ha-chassis-group-add-chassis hcg-external chassis-2 50
[admin]# ovn-nbctl ha-chassis-group-add-chassis hcg-external chassis-3 30
[admin]# ovn-nbctl lrp-set-ha-chassis-group lrp-external hcg-external
```

Priority semantics:
- chassis-1 (100) = preferred active.
- chassis-2 (50) = standby 1 (activate khi chassis-1 fail).
- chassis-3 (30) = standby 2 (activate khi cả 1+2 fail).

### HA_Chassis_Group vs Gateway_Chassis (legacy)

OVN 21.03 trước dùng `Gateway_Chassis` schema, simpler nhưng không có BFD built-in. HA_Chassis_Group replace và extend:

| Feature | Gateway_Chassis (legacy) | HA_Chassis_Group (current) |
|---------|--------------------------|----------------------------|
| BFD integration | Manual (operator script) | Auto (OVN-managed) |
| Preemption control | Không có | `ha_chassis_group_options:preemption` |
| Multi-group support | 1 group per LRP | Nhiều group reference cross-LRP |
| Deprecation | OVN 22.03+ deprecated | OVN 21.03+ default |

Nếu đang dùng Gateway_Chassis, migrate sang HA_Chassis_Group.

## 13.6.3 BFD (RFC 5880), sub-second failure detection

### Protocol fundamentals

BFD là **lightweight hello protocol** phát hiện peer failure với millisecond granularity. Chạy trên UDP port 3784 (single-hop session).

Packet format (RFC 5880 §4.1):

```
Ver (3 bit) | Diag (5 bit) | State (2 bit) | Flags (6 bit)
Detect Mult (8 bit) | Length (8 bit)
My Discriminator (32 bit)
Your Discriminator (32 bit)
Desired Min TX Interval (32 bit, microseconds)
Required Min RX Interval (32 bit, microseconds)
Required Min Echo RX Interval (32 bit)
```

Key fields:
- **State**: 2-bit, 4 values: AdminDown (0) / Down (1) / Init (2) / Up (3).
- **Detect Mult**: nhân với peer's TX interval → detection time.
- **My/Your Discriminator**: 32-bit session ID giúp demux multiple sessions.

### State machine (3 state + init)

```
  ┌─────────────┐
  │  AdminDown  │ ← admin disable
  └──────┬──────┘
         │ enable
         ▼
  ┌─────────────┐ no packets received X ms
  │    Down     │ ────────────────────────┐
  └──────┬──────┘                         │
         │ hear from peer                 │
         ▼                                │
  ┌─────────────┐ peer says Up           │
  │    Init     │ ──────────────────────┐│
  └──────┬──────┘                       ││
         │ peer says Init/Up            ││
         ▼                              ▼▼
  ┌─────────────┐ detect timeout  ┌─────────────┐
  │     Up      │ ───────────────▶│    Down     │
  └─────────────┘                 └─────────────┘
```

### Timing math

**Default OVN BFD:**
- `min_tx_interval=500000µs` (500ms)
- `min_rx_interval=500000µs` (500ms)
- `detect_multiplier=3`
- Detection time = `min_tx_interval × detect_multiplier` = 1500ms.

**Fast-timer (sub-second):**
- `min_tx_interval=100000µs` (100ms)
- `min_rx_interval=100000µs` (100ms)
- `detect_multiplier=3`
- Detection time = 300ms.

**Extremely fast (production HA trading):**
- `min_tx_interval=10000µs` (10ms)
- `detect_multiplier=3`
- Detection = 30ms.
- **CPU + network cost:** mỗi session = 100 packet/s × N session. 100 session × 100 pps = 10K pps chỉ cho BFD. Chỉ chấp nhận được trên chassis dedicated HA.

### Configure BFD trong OVN

```bash
[admin]# ovn-nbctl set NB_Global . options:bfd-min-rx=100 \
                                          options:bfd-min-tx=100 \
                                          options:bfd-mult=3
```

Applied cross all HA_Chassis_Group. Per-group override qua `HA_Chassis_Group_Options`:

```bash
[admin]# ovn-nbctl set HA_Chassis_Group hcg-external \
         options:bfd-min-rx=50 options:bfd-min-tx=50
```

## 13.6.4 Anatomy Template A — `ovs-appctl bfd/show`

**Bối cảnh khi chạy:** Sau configure HA_Chassis_Group, verify BFD session đã establish + UP giữa chassis members. Cũng là lệnh debug đầu tiên khi failover không trigger.

**Command và output** *(reproduced OVN 22.03.8 trên chassis-1, 2 BFD session tới chassis-2 + chassis-3)*:

```shell
$ sudo ovs-appctl -t ovs-vswitchd bfd/show

Interface ovn-chassis-2-0
        BFD State              : Up
        Session State          : Up
        Forwarding             : true
        Diagnostic             : No Diagnostic
        Detect Multiplier      : 3
        Concatenated Path Down : false
        Local Discriminator    : 0x8123a5f0
        Remote Discriminator   : 0x7f2b1944
        Local Min TX Interval  : 500 ms
        Local Min RX Interval  : 500 ms
        Remote Min RX Interval : 500 ms
        Poll Seq In Progress   : false
        Remote State           : Up
        Local Flag             : C
        Remote Flag            : None
        Loss Detect Timeout    : 1500 ms

Interface ovn-chassis-3-0
        BFD State              : Up
        Session State          : Up
        Forwarding             : true
        Detect Multiplier      : 3
        Local Discriminator    : 0x8124a5f1
        Remote Discriminator   : 0x9a3c4021
        Loss Detect Timeout    : 1500 ms
```

**9-attribute anatomy:**

| Field | Giá trị mẫu | Ý nghĩa | Red flag nếu |
|-------|-------------|---------|--------------|
| `Interface` | `ovn-chassis-2-0` | Tunnel port name | Missing interface = tunnel port mất, BFD không setup được |
| `BFD State` | `Up` / `Down` / `Init` | Overall BFD state | `Down` persistent = chassis peer unreachable |
| `Session State` | `Up` | Session-level state | `Init` sau 10s = one-way BFD (peer không respond) |
| `Forwarding` | `true` | BFD cho phép forward traffic | `false` = ovn-controller block traffic do BFD down |
| `Detect Multiplier` | `3` | Number of missed packets trước DOWN | <3 = aggressive detection, false positive risk |
| `Local Discriminator` | `0x8123a5f0` | Session ID local-side | Changes mỗi restart = BFD reset |
| `Remote Discriminator` | `0x7f2b1944` | Session ID peer-side | `0x00000000` = BFD chưa nhận packet nào từ peer |
| `Local Min TX Interval` | `500 ms` | Tốc độ gửi BFD packet | Mismatch với peer's min_rx_interval = negotiation |
| `Remote Min RX Interval` | `500 ms` | Peer's acceptable RX rate | Giới hạn local TX rate |
| `Loss Detect Timeout` | `1500 ms` | tx × multiplier = detection time | Peer flap nếu timeout quá ngắn |
| `Poll Seq In Progress` | `false` | Poll sequence trong progress (parameter change) | `true` stuck = parameter negotiation failed |

**Kịch bản bẻ gãy:**

- **`Session State: Down` persistent + `Remote Discriminator: 0x00000000`**: peer chassis không sending BFD. Causes: ovn-controller crashed on peer, tunnel port MTU issue blocking BFD, firewall block UDP 3784.
- **`Session State: Init` không chuyển sang Up**: one-way BFD, local sending, peer receiving, nhưng peer sending back không reach local. Asymmetric firewall rule hoặc tunnel MTU asymmetric.
- **`Loss Detect Timeout` không khớp với expected config**: BFD negotiation fallback to slowest. Check peer's `min_rx_interval`, nếu lớn hơn local's `min_tx_interval`, BFD use the slower rate.
- **`Forwarding: false` + `BFD State: Up`**: `bfd_slow_timer_expired` flag. Manual check `ovs-appctl bfd/show -v` for detail.

## 13.6.5 Failover workflow, step-by-step

Scenario: chassis-1 active (priority 100) crash. chassis-2 (priority 50) stand-by.

```
T+0ms      chassis-1 kernel panic + reboot
T+0ms      BFD session chassis-2 ↔ chassis-1 stop receiving packets from c1
T+500ms    missed 1 BFD packet
T+1000ms   missed 2 BFD packets
T+1500ms   chassis-2 BFD state transition UP → DOWN (detect timeout 3 × 500ms)
T+1500ms   ovn-controller chassis-2 update HA_Chassis_Group:
           - ovn-sbctl update HA_Chassis_Group active=chassis-2
T+1600ms   SBDB Raft commit (3-node latency 50-100ms)
T+1650ms   ovn-northd regenerate Logical_Flow:
           - Port_Binding chassisredirect chassis field → chassis-2
T+1700ms   ovn-controller chassis-2 + chassis-3 subscribe change
T+1750ms   chassis-2 install OpenFlow rule cho NAT:
           - ARP announce 192.0.2.1 từ chassis-2's br-ex
T+1800ms   upstream switch/router update ARP cache
T+1850ms   Traffic north-bound flow through chassis-2
T+2000ms   VM ping 8.8.8.8 resumes
```

**Tổng downtime typical: 1.5-2.5s** với default BFD. Với fast-timer 100ms: 400-600ms downtime.

### Recovery + preemption

Khi chassis-1 recover (reboot xong, ovn-controller start):
1. Re-register Chassis row trong SBDB.
2. BFD resume.
3. **Nếu preemption enabled (default)**: chassis-1 priority 100 > chassis-2 priority 50 → fail-back tới chassis-1.
4. **Nếu preemption disabled**: chassis-2 giữ active, chassis-1 standby.

**Đọc config preemption:**
```bash
$ sudo ovn-nbctl get HA_Chassis_Group hcg-external options:preemption
```

Default: `true` (preemption on). Production best practice: set `false` tránh flapping nếu chassis-1 unstable.

```bash
[admin]# ovn-nbctl set HA_Chassis_Group hcg-external options:preemption=false
```

> **Chủ đề trọng tâm:** Preemption off vs on là quyết định operational quan trọng. Preemption on = recover tự về preferred chassis (maintain priority invariant, ops visibility). Preemption off = giữ stability sau failover (tránh flapping nếu chassis restart liên tục). Rule of thumb: preemption **on** trong dev/staging (visibility), **off** trong production (stability). Backup: monitoring alert khi preemption off nhưng active != highest priority chassis, có thể là signal cần ops intervention.

## 13.6.6 Common failure modes

> **Hiểu sai phổ biến:** *"BFD phát hiện chassis fail trong 100ms luôn luôn."*
>
> **Thực tế:** BFD phát hiện trong **`min_tx × detect_multiplier` × 2 worst-case**, nếu chassis fail NGAY SAU khi vừa gửi BFD packet, peer phải đợi hết tx interval thêm 1 lần nữa. Detection time thực tế 1x-2x tính toán. Thêm nữa, SBDB Raft commit (50-100ms), ovn-northd regenerate (50-200ms), ovn-controller apply (50-100ms), ARP re-announce (100-500ms tùy upstream switch). Tổng end-to-end failover với default BFD: 1.5-3s. Đừng hứa 100ms downtime với BFD default; đó là BFD detection only, không phải end-to-end recovery.

**Failure 1 — BFD flapping do network jitter.**
Dấu hiệu: `ovs-appctl bfd/show` show BFD state toggle UP/DOWN mỗi 10-30s. ovn-sbctl log cho thấy HA_Chassis_Group active chassis đổi liên tục.
Causes: underlay network độ trễ spike (vượt `min_rx × mult`), chassis CPU overloaded (không process BFD packet kịp).
Fix: (1) Increase `detect_multiplier` 3 → 5 (tolerance more missed packet). (2) Disable preemption. (3) Dedicate gateway chassis (không mix workload).

**Failure 2 — One-way BFD (session stuck ở Init).**
Dấu hiệu: local `BFD State: Init` không chuyển Up sau 30s.
Causes: firewall block UDP 3784 outbound/inbound asymmetric. Tunnel MTU asymmetric.
Fix: `tcpdump -i tunnel-interface udp port 3784`, nếu thấy packet one-way only, khắc phục sự cố firewall/MTU.

**Failure 3 — BFD Up nhưng Chuyển tiếp false.**
Dấu hiệu: `BFD State: Up` + `Forwarding: false`.
Causes: slow timer expired (BFD parameter change trong flight). Local flag `C` without remote flag acknowledge.
Fix: restart ovn-controller trên chassis để reset BFD session.

**Failure 4 — Failover không kích hoạt dù chassis crash.**
Dấu hiệu: chassis-1 crash nhưng traffic vẫn route qua chassis-1 shadow (Port_Binding chassisredirect không update).
Causes: ovn-controller trên chassis-2 crashed hoặc OVSDB connection dead.
Fix: `ovn-appctl -t ovn-controller connection-status` trên chassis-2.

## 13.6.7 Best practices production

1. **3 HA_Chassis members tối thiểu**, 5 cho mission-critical. 2 member = split-brain risk nếu network partition.
2. **Disable preemption** trong production (`preemption=false`).
3. **Dedicate gateway chassis**, không mix VM workload (tránh CPU contention với BFD processing).
4. **Monitor BFD state** qua Prometheus + `ovs_exporter` (port 9475). Alert rule: session DOWN > 30s = critical.
5. **Quarterly failover drill**: kill ovn-controller chassis-1 trong maintenance window, verify migrate < 2s + fail-back đúng (hoặc không, tùy preemption).
6. **Tune BFD theo workload**: development lab = default (500ms), production = 250ms, HFT/trading = 100ms với dedicated CPU core.
7. **Log BFD transition**: `ovn-appctl bfd/show --verbose` scheduled qua cron, log 24h rolling cho post-incident review.

## 13.6.9 Điểm cốt lõi cần nhớ

Thứ nhất, HA_Chassis_Group là OVN abstraction cho gateway failover, replace legacy Gateway_Chassis schema với BFD integration auto-managed.

Thứ hai, BFD (RFC 5880) detect sub-second với packet format fixed + state machine 3-state (INIT/UP/DOWN). Detection time = `min_tx × detect_multiplier`. Default 1500ms, fast-timer 300ms, extreme 30ms với CPU cost.

Thứ ba, end-to-end failover time = BFD detection + SBDB Raft commit + northd regenerate + ovn-controller apply + ARP announce ≈ 1.5-3s default, 400-600ms fast-timer. Đừng hứa chỉ 100ms, đó là BFD only.

Thứ tư, preemption off trong production tránh flapping. Fail-back manual hoặc auto-schedule qua maintenance window.

Thứ năm, 7 best practices: 3+ member / preemption off / dedicate gateway / Prometheus BFD monitor / quarterly drill / BFD tune theo workload / log transition.

Thứ sáu, 4 common failure modes: BFD flapping (jitter), one-way BFD (firewall/MTU asymmetric), Chuyển tiếp=false stuck (parameter change), failover không trigger (ovn-controller crash). Mỗi cái có signature + fix rõ.

## 13.6.W Năm khái niệm NBDB cho HA gateway đào sâu (HA_Chassis_Group, BFD, gateway_chassis option, ha-priority semantics, chassisredirect Port_Binding type)

> Phần này đào sâu năm khái niệm cốt lõi liên quan tới HA gateway của OVN, mỗi khái niệm có treatment riêng đầy đủ 20 góc nhìn (khái niệm → so sánh hệ khác). Hai bảng đầu (`HA_Chassis_Group`, `BFD`) là bảng NBDB; ba mục sau (`gateway_chassis` option của LRP, semantics `priority` trong HA_Chassis, type `chassisredirect` của Port_Binding ở SBDB) là các thực thể logic gắn với cấu hình HA gateway. Schema verified upstream `ovn-nb.ovsschema` branch-22.03; mã nguồn cross-link `northd/ha-chassis.c`, `northd/northd.c::build_chassis_redirect_*`, `controller/bfd.c`, RFC 5880.

### 13.6.W.1 Bảng `HA_Chassis_Group` ở NBDB — định nghĩa nhóm chassis ứng cử active gateway

#### Khái niệm cốt lõi

`HA_Chassis_Group` là **bảng nội trú trong OVN Northbound database** mô tả một **nhóm chassis ứng cử cho vai trò active của một distributed gateway port (DGP)**, trong đó mỗi thành viên là một row `HA_Chassis` mang giá trị `priority` riêng.
Mỗi row ở bảng này đại diện cho một group logic, ví dụ `hcg-external` chứa ba chassis ở ba rack khác nhau cùng đủ điều kiện làm gateway cho lưu lượng north-south của tenant.
Loại: **OVSDB table root** (`isRoot=true`), có **index unique trên cột `name`**, ba cột tổng cộng (`name`, `ha_chassis`, `external_ids`).
Cột `ha_chassis` là **strong reference** tới các row con `HA_Chassis`, nghĩa là khi xoá row `HA_Chassis_Group`, các row `HA_Chassis` thuộc group cũng bị cascade xoá theo OVSDB semantics.
Khác biệt then chốt với cột `gateway_chassis` cũ (đặt trực tiếp trên `Logical_Router_Port`): `HA_Chassis_Group` cho phép một group được nhiều LRP cùng tham chiếu, đồng thời tích hợp BFD do northd tự quản (không cần script bên ngoài).

#### Lịch sử + bối cảnh ra đời

`HA_Chassis_Group` được giới thiệu OVN 2.12 (2019) khi nhóm Lucas Alvares Gomes + Numan Siddique (Red Hat) refactor mô hình "redirect-chassis" + "gateway_chassis" cũ.
Trước OVN 2.12, mỗi LRP có cột `gateway_chassis` chứa danh sách (chassis-name, priority); fail-over vận hành bằng heuristic SBDB connection timeout (10-30 giây), không có BFD chuẩn hoá.
OVN 2.12 thêm `HA_Chassis_Group` cho phép tách "định nghĩa group" khỏi "gắn vào LRP", + tích hợp BFD theo RFC 5880 cho sub-second detection.
OVN 21.06 mark `gateway_chassis` legacy + khuyến nghị migrate sang `HA_Chassis_Group`; OVN 22.03 stabilize 3 cột (verified branch-22.03 schema); OVN 24.03 cải tiến northd incremental compile cho HA_Chassis_Group thay đổi (chỉ recompute các Port_Binding chassisredirect liên quan).

#### Vị trí trong kiến trúc

Bảng nằm ở **layer NBDB** input, **root table**, sống độc lập với Logical_Router/Logical_Switch.
Quan hệ tham chiếu: `Logical_Router_Port.ha_chassis_group` (NBDB) là weak ref ngược tới `HA_Chassis_Group`; `HA_Chassis_Group.ha_chassis` (strong) tới `HA_Chassis`; `HA_Chassis.chassis_name` (string) lưu tên chassis tham chiếu mềm tới `Chassis.name` ở SBDB.
Scope: **cluster-wide**, một HA_Chassis_Group có thể được nhiều LRP của nhiều tenant cùng tham chiếu (chia sẻ pool gateway chassis).
ovn-northd compile NBDB HA_Chassis_Group thành SBDB HA_Chassis_Group + HA_Chassis tương ứng, đồng thời cập nhật cột `ha_chassis_group` trong row Port_Binding `chassisredirect` của LRP gắn group (đã được dạy ở §13.5.1d.4 phía SB).

#### Vai trò + nhiệm vụ

Bảng gánh bốn vai trò: (1) **Trừu tượng hoá pool gateway chassis** thành 1 named entity reusable bởi nhiều LRP; (2) **Lưu giữ ánh xạ chassis → priority** để quyết định chassis nào active mặc định; (3) **Là điểm gắn BFD** thông qua compile xuống SBDB BFD row một cách tự động theo từng cặp chassis trong group; (4) **Tách chính sách HA khỏi topology**, admin có thể đổi membership group mà không sửa LRP.

#### Vì sao sinh ra (động cơ thiết kế)

Pre-2.12, mô hình `gateway_chassis` đặt list chassis trực tiếp trên LRP có ba nhược điểm: (a) **trùng lặp** nếu nhiều LRP cùng dùng pool chassis identical, mỗi LRP phải khai báo lại; (b) **không có BFD chuẩn**, fail-over dựa SBDB timeout thiếu chính xác data plane; (c) **schema cứng**, thêm tính năng mới (preemption tuning, BFD parameter override) phải mở rộng `Logical_Router_Port` schema.
`HA_Chassis_Group` thiết kế tách concern: định nghĩa group là entity bậc một, LRP chỉ tham chiếu group qua một cột.
Mở rộng tính năng (BFD parameter, preemption) thêm vào `HA_Chassis_Group.external_ids` hoặc options field mà không đụng schema LRP.

#### Vấn đề mà nó giải quyết

(1) **Quản lý pool gateway chassis tập trung** thay vì rải rác trên từng LRP; (2) **Thay group membership runtime** (thêm/xoá HA_Chassis member) mà không restart LRP hay flap traffic; (3) **Reusable across tenant**, một group `hcg-external` phục vụ tất cả LRP north-south của 50 tenant; (4) **Tích hợp BFD tự động**, northd biết group có những chassis nào để compile cặp BFD session tương ứng; (5) **Fail-over deterministic** dựa trên priority của HA_Chassis member, không cần admin script.

#### Tầm quan trọng

Đây là bảng **tier 2 medium** của NBDB cho mọi cluster có distributed gateway router. Lý do: (a) 100% deployment OpenStack Neutron (driver networking-ovn) với external network sẽ tự sinh HA_Chassis_Group qua plugin; (b) 100% deployment có north-south HA cần group này; (c) deployment chỉ có east-west routing không dùng (LR distributed không cần redirect-chassis). Kỹ sư không hiểu HA_Chassis_Group sẽ không debug được "north-south traffic bị blackhole sau khi 1 chassis down" hoặc "fail-back về chassis ưu tiên không xảy ra".

#### Cơ chế hoạt động

Một row HA_Chassis_Group đi qua sáu giai đoạn: (a) **Khai báo group**: `ovn-nbctl ha-chassis-group-add hcg-external` insert row mới name=`hcg-external`, ha_chassis=[]; (b) **Thêm member**: `ovn-nbctl ha-chassis-group-add-chassis hcg-external chassis-1 100` create row HA_Chassis (chassis_name=`chassis-1`, priority=100), append UUID vào cột ha_chassis; (c) **Gắn vào LRP**: `ovn-nbctl lrp-set-ha-chassis-group lrp-external hcg-external` set cột `ha_chassis_group` của Logical_Router_Port; (d) **Northd compile**: hàm `build_ha_chassis_group_table` (`northd/ha-chassis.c`) sao chép HA_Chassis_Group + HA_Chassis xuống SBDB; (e) **Compile chassisredirect**: northd sinh Port_Binding `cr-lrp-external` type=`chassisredirect` ở SBDB, gắn cột `ha_chassis_group` reference SBDB HA_Chassis_Group; (f) **Compile BFD**: northd sinh SBDB BFD row cho mỗi cặp chassis trong group (mesh full hoặc star tuỳ topology), inserted vào cột `bfd` của Port_Binding chassisredirect; (g) **Runtime claim**: ovn-controller tại chassis có priority cao nhất với BFD up tự claim ownership Port_Binding chassisredirect.

#### Phân loại cột (verified branch-22.03)

Ba cột tất cả: `name` (string, có index unique, required, identifier toàn cluster), `ha_chassis` (set UUID strong ref tới HA_Chassis, min=0 max=unlimited), `external_ids` (map cho CMS metadata như `neutron:availability_zone`).
Bảng phụ `HA_Chassis` (isRoot=false vì là child của HA_Chassis_Group strong ref) có ba cột: `chassis_name` (string, đối chiếu mềm với SBDB Chassis.name), `priority` (integer, minInteger=0 maxInteger=32767, higher=preferred), `external_ids` (map metadata).

#### Cách kỹ sư vận hành thành thạo

Bốn kỹ năng: (1) **CRUD group**: `ovn-nbctl ha-chassis-group-add/del`, `ha-chassis-group-list`, `ha-chassis-group-add-chassis/del-chassis`; (2) **Gắn group vào LRP**: `ovn-nbctl lrp-set-ha-chassis-group <lrp> <group>`; (3) **Inspect**: `ovn-nbctl --columns=name,ha_chassis,external_ids list HA_Chassis_Group` + cross với `ovn-nbctl list HA_Chassis` để xem priority từng member; (4) **Verify cascade SB**: `ovn-sbctl list HA_Chassis_Group` + `ovn-sbctl list Port_Binding type=chassisredirect` xác nhận northd đã compile xuống.

#### Quy trình sử dụng (workflow chuẩn)

(1) Tạo HA_Chassis_Group rỗng; (2) Thêm 3-5 chassis với priority giảm dần (vd 100/50/30); (3) Gắn group vào LRP gateway qua `lrp-set-ha-chassis-group`; (4) Đợi 5-10 giây cho northd compile; (5) Verify Port_Binding chassisredirect đã có ha_chassis_group reference + BFD row sinh ra; (6) Test fail-over: stop ovn-controller chassis-1, kiểm chassis field của Port_Binding chuyển sang chassis-2 trong < 2s.

#### Khi xảy ra sự cố

(a) "Add chassis vào group nhưng northd không compile xuống SB" → kiểm chassis_name khớp với SBDB Chassis.name (case-sensitive); (b) "Tất cả member priority bằng nhau" → northd chọn theo UUID order (không deterministic); fix bằng cách đặt priority khác nhau; (c) "Thêm member khi group đã gắn LRP nhưng BFD không được tạo" → kiểm SBDB Port_Binding.bfd column, nếu rỗng thì restart ovn-northd để force re-compile.

#### Liên quan mật thiết

Coupling chặt: `HA_Chassis` (strong ref child), `Logical_Router_Port` (weak ref qua cột `ha_chassis_group`), SBDB `HA_Chassis_Group` + `HA_Chassis` (compile target), SBDB `Port_Binding` type=chassisredirect (consumer ở pipeline runtime), SBDB `BFD` (sinh tự động theo cặp chassis trong group), `controller/ha-chassis.c` (consumer logic chọn active), `controller/binding.c` (consumer claim chassisredirect).

#### Khác biệt giữa các phiên bản

OVN 2.12: ra mắt 3 cột; OVN 21.06: mark `gateway_chassis` legacy; OVN 22.03: stabilize 3 cột (verified); OVN 23.03: thêm options trong external_ids cho preemption control fine-grained; OVN 24.03: incremental compile chỉ touch cặp BFD/chassisredirect liên quan, không recompute toàn topology khi membership thay đổi.

#### Cách quan sát + xác minh

`ovn-nbctl list HA_Chassis_Group` snapshot toàn bộ; `ovn-nbctl ha-chassis-group-list` view friendly với priority hiện sẵn.
Đối chiếu compile output: `ovn-sbctl list HA_Chassis_Group` thấy group tương ứng + `ovn-sbctl --columns=ha_chassis_group,chassis,nat_addresses list Port_Binding type=chassisredirect` thấy chassisredirect đã gắn group.
Verify BFD: `ovn-sbctl --columns=logical_port,dst_ip,status list BFD` thấy mesh BFD session giữa các chassis trong group.

#### Source code tham chiếu

Verified upstream OVN branch-22.03: schema [`ovn-nb.ovsschema`](https://raw.githubusercontent.com/ovn-org/ovn/branch-22.03/ovn-nb.ovsschema) khai báo `HA_Chassis_Group` 3 cột + `"isRoot": true` + `"indexes": [["name"]]`.
Compile logic `northd/ha-chassis.c::build_ha_chassis_group_table()` + helper `ha_chassis_group_get_ordered()`.
Cascade tới Port_Binding chassisredirect ở `northd/northd.c::build_chassis_redirect_*` (function name anchor cho Rule 14.4-C, line numbers shift across versions).

#### Trường hợp sự cố thực tế

Production case 2024 cluster Red Hat OSP 17 ~80 compute, 4 network node làm gateway. HA_Chassis_Group `hcg-default` chứa 4 chassis priority 100/75/50/25.
Sự cố: sau patch OS chassis-1 (priority 100) reboot, fail-over về chassis-2 (priority 75) thành công trong 1.5s; nhưng khi chassis-1 join lại, không fail-back.
Root cause: vận hành đặt `external_ids:preemption=false` trong group → preemption disabled, đây là behavior cố ý cho production stability.
Bài học: mỗi cluster cần document rõ chính sách preemption, monitor chassis-1 idle trong khi chassis-2 đang gánh tải.

#### Bài tập synthetic

```bash
sudo ovn-nbctl ha-chassis-group-add hcg-lab
sudo ovn-nbctl ha-chassis-group-add-chassis hcg-lab chassis-A 100
sudo ovn-nbctl ha-chassis-group-add-chassis hcg-lab chassis-B 50
sudo ovn-nbctl lr-add lr-lab
sudo ovn-nbctl lrp-add lr-lab lrp-ext 40:44:00:00:ff:01 192.0.2.1/24
sudo ovn-nbctl lrp-set-ha-chassis-group lrp-ext hcg-lab
sudo ovn-sbctl list Port_Binding cr-lrp-ext
```

Quan sát: Port_Binding `cr-lrp-ext` có chassis=[chassis-A UUID], ha_chassis_group reference, bfd column chứa BFD row sinh tự động.

#### Lỗi thường gặp + tín hiệu chẩn đoán

(a) Thêm chassis_name không tồn tại SBDB → northd compile bỏ qua silently, BFD không sinh; (b) Priority trùng nhau giữa hai member → fail-over không deterministic; (c) Quên gắn group vào LRP qua `lrp-set-ha-chassis-group` → group khai báo nhưng không hiệu lực; (d) Xoá HA_Chassis_Group khi vẫn còn LRP tham chiếu → strong ref violation, transaction reject.

#### So sánh với hệ khác

VRRP nhóm physical router cùng concept (priority-based active/backup); Cisco HSRP standby group có virtual IP tương tự; AWS NAT Gateway HA pattern khác (multi-AZ, không priority); NSX-T Edge Cluster gateway pool tương đồng.

### 13.6.W.2 Bảng `BFD` ở NBDB — khai báo cấu hình BFD per-LRP

#### Khái niệm cốt lõi

`BFD` (NBDB) là **bảng nội trú trong OVN Northbound database** mô tả **một mong muốn cấu hình BFD** cho một logical_port (LSP hoặc LRP) tới một dst_ip cụ thể, với min_tx, min_rx, detect_mult tinh chỉnh được.
Mỗi row đại diện cho một **session BFD pending** ở mức policy, ovn-northd dịch xuống SBDB BFD row tương ứng để chassis chủ thực thi.
Loại: **OVSDB table root** (`isRoot=true`), 8 cột (`logical_port`, `dst_ip`, `min_tx`, `min_rx`, `detect_mult`, `status`, `external_ids`, `options`), index unique trên `(logical_port, dst_ip)`.
Khác biệt then chốt với SBDB BFD (đã dạy ở §13.5.1d.3): NBDB BFD là **policy/intent** do admin/CMS đặt; SBDB BFD là **runtime state** do northd compile xuống + ovn-controller cập nhật trạng thái thực.

#### Lịch sử + bối cảnh ra đời

NBDB `BFD` được giới thiệu OVN 2.10 (2018) cùng nhánh BFD support cho HA_Chassis_Group (Mark Michelson + Numan Siddique).
Trước đó, BFD là implicit (northd tự sinh trong SBDB cho các cặp chassis của HA_Chassis_Group); admin không có cách nào tinh chỉnh interval per-LRP.
OVN 2.10 thêm bảng để admin "khai báo BFD" cho một LRP cụ thể với min_tx/min_rx/detect_mult riêng (vd LRP nối peer router 5G cần BFD 100ms).
OVN 21.06 thêm hỗ trợ BFD-track BGP next-hop (qua ovn-bgp-agent integration đọc BFD status).
OVN 22.03 stabilize 8 cột (verified); OVN 24.03 tối ưu compile khi có > 1000 BFD record.

#### Vị trí trong kiến trúc

NBDB BFD là **root table layer NBDB**, sống độc lập với LRP/LSP nhưng tham chiếu mềm qua cột `logical_port` (string match Logical_Router_Port.name hoặc Logical_Switch_Port.name).
Cột `bfd` của Logical_Router_Port (set UUID NBDB BFD) là weak ref ngược.
Northd compile: hàm `build_bfd_table` (đã dẫn ở §13.5.1d.3) đọc NBDB BFD records, sinh SBDB BFD tương ứng, gán default value cho min_tx/min_rx (nếu admin không set).
Scope cluster-wide; mỗi (logical_port, dst_ip) chỉ tồn tại 1 row tại NBDB.

#### Vai trò + nhiệm vụ

Ba vai trò: (1) **Khai báo BFD intent** cho admin override default per-LRP (vd LRP peering với upstream router muốn BFD 100ms, default 500ms quá chậm); (2) **Gắn BFD vào next-hop static route** để route withdraw khi BFD down (kết hợp với Logical_Router_Static_Route option `bfd=true`); (3) **Monitor health peer device** từ OVN side mà không qua kernel routing daemon (no FRR/BGPd dependency).

#### Vì sao sinh ra (động cơ thiết kế)

Pre-2.10, BFD chỉ implicit cho HA_Chassis_Group internal mesh. Khi OVN tích hợp với upstream physical router (vd L3 leaf MX/QFX), admin cần BFD chạy giữa chassis OVN và physical router với interval tunable.
Đặt cấu hình trong external_ids của LRP không scale (parsing pháp string format khó), và không track status runtime.
Bảng BFD chuẩn hoá: schema typed với min_tx/min_rx/detect_mult là integer, status enum, mỗi row có UUID stable cho LRP tham chiếu.

#### Vấn đề mà nó giải quyết

(1) **BFD per-LRP tunable** thay vì global; (2) **Track status runtime** từ NBDB phía admin (đọc cột status thay vì phải SSH chassis); (3) **Gắn BFD vào static route fast-withdraw** cho ECMP recovery; (4) **Decouple cấu hình BFD khỏi BGP/OSPF daemon** (vd Bird/FRR), OVN tự lo BFD cho route mà nó own.

#### Tầm quan trọng

Tier 2 medium. Cluster chỉ có HA_Chassis_Group internal không cần khai báo NBDB BFD (northd tự lo). Cluster có (a) LRP peering với physical router upstream, (b) static route với BFD-track, (c) tích hợp BGP qua ovn-bgp-agent thì NBDB BFD load-bearing.

#### Cơ chế hoạt động

Một row NBDB BFD đi qua bốn giai đoạn: (a) **Admin khai báo**: `ovn-nbctl create BFD logical_port=lrp-up dst_ip=10.0.0.1 min_tx=300 min_rx=300 detect_mult=3` insert row, status=admin_down ban đầu; (b) **Gắn vào LRP/route**: `ovn-nbctl set Logical_Router_Port lrp-up bfd=@bfd-uuid` hoặc `ovn-nbctl set Logical_Router_Static_Route <uuid> bfd=@bfd-uuid`; (c) **Northd compile**: build_bfd_table sao chép xuống SBDB BFD, gán default min_tx/min_rx nếu thiếu, status=down ban đầu; (d) **Runtime**: ovn-controller tại chassis chủ chạy session, update SBDB BFD.status; northd propagate ngược NBDB BFD.status để admin/CMS đọc.

#### Phân loại cột (verified branch-22.03)

Tám cột: `logical_port` (string, required, đối chiếu mềm tới LSP/LRP name), `dst_ip` (string, required, peer IP), `min_tx` (integer ≥1 ms, optional, default 1000), `min_rx` (integer ms, optional, default 1000), `detect_mult` (integer ≥1, optional, default 5), `status` (enum down|init|up|admin_down, optional), `external_ids` (map), `options` (map cho future extension như authentication).
Index unique `(logical_port, dst_ip)` đảm bảo không trùng session per LRP-peer pair.

#### Cách kỹ sư vận hành thành thạo

(1) **CRUD record**: `ovn-nbctl create BFD ...`, `ovn-nbctl list BFD`, `ovn-nbctl destroy BFD <uuid>`; (2) **Gắn BFD vào LRP**: set cột `bfd` của Logical_Router_Port; (3) **Gắn BFD vào static route**: set cột `bfd` của Logical_Router_Static_Route; (4) **Track status**: `ovn-nbctl --columns=logical_port,dst_ip,status list BFD` polling hoặc `ovsdb-client monitor`.

#### Quy trình sử dụng (workflow chuẩn)

(1) Xác định LRP cần BFD + peer dst_ip; (2) Tạo NBDB BFD record với interval phù hợp (production thường min_tx=300 detect_mult=3 → 900ms); (3) Gắn vào LRP qua cột bfd; (4) Verify SBDB BFD compile xuống có giá trị giống NBDB; (5) Sau 5-10s đọc NBDB BFD.status, kỳ vọng `up` nếu peer cũng có BFD config tương ứng; (6) Test fail-over: tắt peer interface, NBDB status chuyển `down` trong < detect_mult × min_rx ms.

#### Khi xảy ra sự cố

(a) "NBDB status mãi `admin_down`" → BFD chưa gắn vào LRP/route nào, ovn-northd chưa enable; (b) "NBDB status `init` không lên `up`" → peer router không trả packet (asymmetric), hoặc tunnel/direct-connect MTU chặn BFD packet 24-byte; (c) "Status flap up/down chu kỳ" → network jitter hoặc CPU chassis quá tải; (d) "min_tx admin set 50ms, runtime quan sát 1000ms" → peer reject min_rx thấp, BFD negotiation fallback.

#### Liên quan mật thiết

Coupling chặt: SBDB `BFD` (compile target, đã dạy §13.5.1d.3), `Logical_Router_Port` (consumer qua cột bfd), `Logical_Router_Static_Route` (consumer qua cột bfd), `controller/bfd.c` (runtime executor), RFC 5880 (chuẩn protocol), kernel/OVS BFD backend (datapath layer).
Liên quan với HA_Chassis_Group: northd auto-sinh BFD cho cặp chassis trong group → không cần admin khai báo NBDB BFD; chỉ khai báo NBDB BFD khi muốn tinh chỉnh interval hoặc target peer external.

#### Khác biệt giữa các phiên bản

OVN 2.10: ra mắt 8 cột; OVN 21.06: status reverse-propagate từ SBDB; OVN 22.03 stabilize (verified); OVN 24.03: tối ưu schedule BFD scale > 1000 session.
Default interval cũng đổi: OVN 2.10 default min_tx=1000 detect_mult=5 (5s); OVN 22.03 vẫn 1s/5x; cluster fast cần explicit set 100-300ms.

#### Cách quan sát + xác minh

`ovn-nbctl list BFD` snapshot; `ovn-nbctl --columns=logical_port,dst_ip,status,min_tx,min_rx,detect_mult list BFD` table view.
Đối chiếu với SBDB: `ovn-sbctl --columns=logical_port,dst_ip,status,min_tx,min_rx list BFD` (cùng cột nhưng row tự sinh).
Datapath BFD: `ovs-vsctl --columns=name,bfd_status list Interface | grep -A1 ovn-`.
Packet capture: `tcpdump -i <iface> -nn 'udp port 3784'` thấy BFD control packet 24-byte payload mỗi min_tx ms.

#### Source code tham chiếu

Schema `ovn-nb.ovsschema` branch-22.03 khai báo `BFD` 8 cột + `isRoot=true` + `indexes=[["logical_port","dst_ip"]]`.
Compile: `northd/northd.c::build_bfd_table()` (function anchor); runtime executor `controller/bfd.c::bfd_run()` đã dẫn §13.5.1d.3.
RFC 5880 §4.1 packet format với 24-byte mandatory section + auth section optional; OVN không bật auth mặc định.

#### Trường hợp sự cố thực tế

Production case 2024 telco cluster: LRP `lrp-uplink` peering với MX5G physical router. Admin đặt NBDB BFD min_tx=100 detect_mult=3 (target detect 300ms).
Sự cố: NBDB BFD.status=`up` nhưng failover thực tế mất 2 giây. Investigation: peer MX5G config `min-rx 500ms` → BFD negotiation fallback xuống slowest, real interval 500ms × 3 = 1500ms.
Fix: align cấu hình peer = 100ms hai chiều.

#### Bài tập synthetic

```bash
PEER_IP=10.0.0.1
sudo ovn-nbctl create BFD logical_port=lrp-up dst_ip=$PEER_IP min_tx=300 min_rx=300 detect_mult=3
BFD_UUID=$(sudo ovn-nbctl --bare --columns=_uuid find BFD logical_port=lrp-up dst_ip=$PEER_IP)
sudo ovn-nbctl set Logical_Router_Port lrp-up bfd=$BFD_UUID
sudo ovn-sbctl list BFD logical_port=lrp-up
```

Quan sát: SBDB BFD row sinh ra với các cột giống NBDB, status chuyển `init` → `up` sau 1-2s nếu peer reachable.

#### Lỗi thường gặp + tín hiệu chẩn đoán

(a) `logical_port` typo → northd compile bỏ qua silently; (b) Đặt min_tx=10ms nhưng cluster có 1000 LRP → CPU chassis bão hoà BFD processing; (c) Thiếu detect_mult → default 5 quá chậm, đặt 3 cho production; (d) Status mãi `init` → peer chưa cấu hình BFD, kiểm `tcpdump` chỉ thấy egress packet không thấy ingress.

#### So sánh với hệ khác

Cisco IOS `bfd interval` config trên interface tương đương; Juniper JUNOS `bfd-liveness-detection` block; FRR/Bird BFD daemon chuẩn riêng nhưng cùng RFC 5880; AWS Direct Connect không expose BFD config (managed); Linux libbfd/openbfdd tương đương userspace.

### 13.6.W.3 Option `gateway_chassis` ở Logical_Router_Port — schema cũ trước HA_Chassis_Group

#### Khái niệm cốt lõi

`gateway_chassis` là **một cột (set UUID) của Logical_Router_Port ở NBDB** dùng từ OVN 2.4 đến trước 2.12, lưu danh sách các row Gateway_Chassis (mỗi row mang chassis_name + priority).
Mỗi row Gateway_Chassis đại diện cho một ứng cử viên chassis cho LRP đó với một priority cụ thể; cột `gateway_chassis` của LRP gán nhiều row tạo nên ordered set theo priority.
Loại: **set strong ref**, ovn-northd compile tương tự HA_Chassis_Group nhưng không tích hợp BFD chuẩn (BFD phải config bên ngoài OVN, qua external script hoặc kernel routing daemon).
Khác biệt then chốt với HA_Chassis_Group hậu duệ: (a) gắn trực tiếp vào LRP, không reusable cross-LRP; (b) không có BFD tự sinh; (c) deprecated từ OVN 21.06.

#### Lịch sử + bối cảnh ra đời

`gateway_chassis` ra mắt OVN 2.4 (2015) khi OVN bắt đầu hỗ trợ distributed gateway router (DGR) với chassisredirect.
Ban đầu LRP chỉ có cột đơn `redirect-chassis` (string, một chassis duy nhất); 2.4 mở rộng thành ordered set Gateway_Chassis cho HA cơ bản (active/standby theo priority).
2.12 (2019) giới thiệu HA_Chassis_Group thay thế (xem §13.6.W.1); 21.06 mark `gateway_chassis` legacy, công cụ migrate `ovn-nbctl ha-chassis-group-add` + `lrp-set-ha-chassis-group` được khuyến nghị.
22.03 vẫn parse được `gateway_chassis` (backward compat) nhưng warn admin migrate; 24.03 chính thức loại bỏ trong tương lai (không có lịch cụ thể).

#### Vị trí trong kiến trúc

Cột nằm trên Logical_Router_Port ở NBDB. `Gateway_Chassis` là bảng phụ (isRoot=false vì child của LRP strong ref).
Northd compile cột này cùng cách HA_Chassis_Group: tạo SBDB Port_Binding chassisredirect, gán chassis active là chassis có priority cao nhất.
BFD: northd không tự sinh; admin phải gán BFD ngoài luồng (vd thêm BFD record NBDB, gắn vào LRP). Đây là điểm yếu chính khiến OVN refactor sang HA_Chassis_Group có BFD built-in.

#### Vai trò + nhiệm vụ

Ba vai trò (legacy): (1) khai báo ordered set chassis ứng cử cho LRP; (2) cho phép active/standby fail-over đơn giản; (3) backward-compat schema cho deployment chưa migrate. Vai trò (1) và (2) đều bị HA_Chassis_Group thay thế đầy đủ.

#### Vì sao sinh ra (động cơ thiết kế)

Pre-2.4, OVN distributed router chỉ có một redirect-chassis duy nhất → SPOF. 2.4 thêm gateway_chassis ordered set là nỗ lực đầu tiên hỗ trợ HA gateway. Thiết kế đặt list trực tiếp trên LRP để tối thiểu thay đổi schema (chỉ một cột mới).

#### Vấn đề mà nó giải quyết (legacy)

Giải quyết SPOF của redirect-chassis đơn. Nhưng để lại bốn vấn đề: (a) **không reusable**, mỗi LRP duplicate list chassis; (b) **không BFD chuẩn**, fail-over dựa SBDB timeout chậm; (c) **schema gắn cứng vào LRP**, mở rộng tính năng phức tạp; (d) **CMS quản lý phức tạp** khi nhiều LRP cùng pool.

#### Tầm quan trọng

Tier 3 peripheral hôm nay (khuyến nghị migrate). Quan trọng cho kỹ sư duy trì cluster legacy 21.03 hoặc cũ hơn, hoặc tham gia migration hard cluster sang HA_Chassis_Group.

#### Cơ chế hoạt động

Quy trình cũ: (a) `ovn-nbctl lrp-set-gateway-chassis lrp-ext chassis-1 100` tạo Gateway_Chassis row + append vào cột; (b) `lrp-set-gateway-chassis lrp-ext chassis-2 50` thêm member; (c) northd compile thành Port_Binding chassisredirect; (d) chassis có priority cao nhất + reachable claim ownership; (e) fail-over dựa heuristic SBDB timeout.

#### Phân loại cột (verified branch-22.03 schema, kept for backward compat)

Cột `gateway_chassis` trên Logical_Router_Port: set UUID strong ref tới Gateway_Chassis, min=0 max=unlimited. Bảng Gateway_Chassis: 4 cột (`name` string, `chassis_name` string, `priority` integer 0-32767, `external_ids` map; `options` map có thể có tuỳ branch).

#### Cách kỹ sư vận hành thành thạo

(1) **List**: `ovn-nbctl lrp-get-gateway-chassis lrp-ext` xem ordered list; (2) **Add**: `lrp-set-gateway-chassis lrp-ext chassis-X <prio>`; (3) **Del**: `lrp-del-gateway-chassis lrp-ext chassis-X`; (4) **Migrate sang HA_Chassis_Group**: tạo group mới tương ứng → `lrp-set-ha-chassis-group` → del Gateway_Chassis cũ.

#### Quy trình sử dụng (chỉ cho cluster legacy)

Khuyến nghị: không tạo mới `gateway_chassis`. Cluster legacy có sẵn → migrate trong maintenance window: (1) tạo HA_Chassis_Group mới với cùng membership + priority; (2) gắn vào LRP qua `lrp-set-ha-chassis-group`; (3) verify chassisredirect chuyển sang dùng group; (4) `lrp-del-gateway-chassis` xoá entry cũ.

#### Khi xảy ra sự cố

(a) "Fail-over chậm 10-30 giây" trên cluster legacy → đặc trưng schema cũ thiếu BFD; chỉ migrate sang HA_Chassis_Group mới khắc phục được; (b) "lrp-set-ha-chassis-group đã chạy nhưng fail-over vẫn chậm" → kiểm Gateway_Chassis cũ còn trên LRP không, cần xoá (Logical_Router_Port có cả hai cột thì northd ưu tiên ha_chassis_group nhưng giữ Gateway_Chassis là tài nguyên thừa).

#### Liên quan mật thiết

Coupling: `Gateway_Chassis` (strong ref child), `Logical_Router_Port` (parent), `HA_Chassis_Group` (kế nhiệm về chức năng), `Port_Binding chassisredirect` (consumer SBDB).

#### Khác biệt giữa các phiên bản

OVN 2.4: ra mắt; 2.12: HA_Chassis_Group song song (cả hai cột tồn tại); 21.06: mark legacy + warn; 22.03: parse được (verified); 24.x: dự kiến loại bỏ (theo NEWS upstream).

#### Cách quan sát + xác minh

`ovn-nbctl lrp-get-gateway-chassis <lrp>`; `ovn-nbctl --columns=name,gateway_chassis,ha_chassis_group list Logical_Router_Port` xem cùng lúc cả cột cũ và mới (cluster đang migrate).

#### Source code tham chiếu

Schema branch-22.03 vẫn có cột `gateway_chassis` của Logical_Router_Port + bảng Gateway_Chassis (legacy compat). Compile logic chia nhánh: nếu LRP có `ha_chassis_group` set thì dùng cái mới, else fallback Gateway_Chassis (`northd/northd.c::build_chassis_redirect_*` chia nhánh).

#### Trường hợp sự cố thực tế

Cluster Telco châu Âu vận hành OVN 21.03 từ 2021, fail-over north-south chậm 12-15 giây. Migration plan: Q3-2024 nâng OVN lên 22.03 + chuyển toàn bộ LRP sang HA_Chassis_Group. Sau migration: fail-over còn 1.8s với BFD 500ms × 3.

#### Bài tập synthetic (chỉ minh hoạ cho legacy)

```bash

sudo ovn-nbctl lrp-set-gateway-chassis lrp-ext chassis-B 50
sudo ovn-nbctl lrp-get-gateway-chassis lrp-ext
sudo ovn-nbctl ha-chassis-group-add hcg-migrated
sudo ovn-nbctl ha-chassis-group-add-chassis hcg-migrated chassis-A 100
sudo ovn-nbctl ha-chassis-group-add-chassis hcg-migrated chassis-B 50
sudo ovn-nbctl lrp-set-ha-chassis-group lrp-ext hcg-migrated
sudo ovn-nbctl lrp-del-gateway-chassis lrp-ext chassis-A
sudo ovn-nbctl lrp-del-gateway-chassis lrp-ext chassis-B
```

#### Lỗi thường gặp + tín hiệu chẩn đoán

(a) Để cùng lúc cả `gateway_chassis` và `ha_chassis_group` trên LRP → ưu tiên cái mới, cũ thừa nhưng tốn slot OVSDB; (b) Quên xoá Gateway_Chassis cũ sau migrate → CMS audit log lẫn lộn; (c) Cluster mixed (một LRP migrate, LRP khác chưa) gây khó khăn khi debug.

#### So sánh với hệ khác

Cisco GLBP nhóm gateway có priority tương tự nhưng VIP thay vì chassisredirect; VRRPv3 priority field 0-255 ánh xạ concept gần; OpenStack Neutron L3 HA legacy (pre-OVN) cũng có priority queue tương đồng.

### 13.6.W.4 Semantics `priority` của HA_Chassis — luật chọn active

#### Khái niệm cốt lõi

`priority` trong row `HA_Chassis` là **integer 0-32767** quyết định **thứ tự ưu tiên chọn active** cho chassisredirect Port_Binding gắn HA_Chassis_Group đó.
Quy tắc: **higher priority = preferred active**; trong số chassis có BFD `up` ở thời điểm bầu, chassis priority cao nhất được chọn; tie-break theo UUID (deterministic nhưng không có ý nghĩa với người vận hành).
Mỗi row HA_Chassis đại diện cho một cặp (chassis_name, priority); cùng group có thể chứa nhiều row với priority khác nhau.

#### Lịch sử + bối cảnh ra đời

Quy ước "higher = winner" kế thừa từ Gateway_Chassis (OVN 2.4) sang HA_Chassis_Group (OVN 2.12).
Inspiration: VRRPv2 (RFC 3768) priority 1-254 (higher win) + Cisco HSRP priority 0-255 (higher win).
OVN chọn dải rộng 0-32767 (15-bit) để có không gian rộng hơn cho hierarchical priority (vd: rack 1 dùng 30000+, rack 2 dùng 20000+, rack 3 dùng 10000+).
Range được verified ở schema branch-22.03: `"minInteger": 0, "maxInteger": 32767`.

#### Vị trí trong kiến trúc

`priority` là cột integer của row HA_Chassis ở cả NBDB và SBDB.
Northd sao chép NBDB HA_Chassis xuống SBDB HA_Chassis với cùng giá trị priority; ovn-controller (`controller/ha-chassis.c::ha_chassis_get_ordered`) đọc SBDB list, sort theo priority desc, kết hợp với BFD status để chọn active.
Ảnh hưởng tới Port_Binding chassisredirect: cột `chassis` được set thành chassis với priority cao nhất + BFD up (verified `controller/binding.c` claim logic).

#### Vai trò + nhiệm vụ

Ba vai trò: (1) **Sort order** trong election algorithm (deterministic); (2) **Hierarchy site/rack/role**, admin có thể đặt convention "chassis ở DC chính 30000+, chassis DR site 20000+, chassis edge 10000+" để guarantee normal/disaster ordering; (3) **Manual override**, đặt một chassis priority cao tạm thời để force claim trong maintenance.

#### Vì sao sinh ra (động cơ thiết kế)

Election deterministic cần một số totally ordered. Random/UUID-based ordering không cho admin control; round-robin không phù hợp gateway HA (cần stickiness chassis active). Priority integer là chuẩn de-facto trong networking (VRRP/HSRP/STP root bridge), OVN tái sử dụng nguyên tắc.

#### Vấn đề mà nó giải quyết

(1) Election deterministic + reproducible (cùng input → cùng output); (2) Cho phép admin biểu đạt chính sách (chassis nào prefer); (3) Hỗ trợ manual fail-over qua thay priority; (4) Tránh thrashing/oscillation khi nhiều chassis cùng eligible.

#### Tầm quan trọng

Tier 2 medium. Mọi HA_Chassis_Group production đều dùng priority phân biệt; bằng nhau gây non-deterministic. Kỹ sư phải hiểu để (a) thiết kế convention priority cho cluster; (b) thực hiện manual fail-over qua thay priority; (c) debug "vì sao chassis X được chọn thay vì Y".

#### Cơ chế hoạt động

Election algorithm (`ha_chassis_get_ordered` `northd/ha-chassis.c`): (a) đọc list HA_Chassis của group; (b) sort desc theo priority; (c) tie-break theo chassis_name (alphabetical) hoặc UUID tuỳ phiên bản; (d) trả về ordered list cho consumer (ovn-controller binding logic).
ovn-controller tại mỗi chassis đọc list này: nếu chassis local là first trong ordered list **và** BFD tới các chassis khác trong group `up` (chứng tỏ local không bị isolated), thì claim Port_Binding chassisredirect bằng OVSDB CAS.

#### Phân loại

Theo convention: (a) **Equal priority** (cấm trong production, fail-over không deterministic); (b) **Stepped priority** (vd 100/50/30 cho 3 chassis, gap đủ lớn để tránh nhầm lẫn); (c) **Hierarchical** (vd 30000+ DC chính, 20000+ DR, 10000+ edge); (d) **Maintenance-temporary** (vd nâng priority chassis-2 lên 200 trong maintenance để fail-back chậm).

#### Cách kỹ sư vận hành thành thạo

(1) **Đọc priority hiện tại**: `ovn-nbctl --columns=chassis_name,priority list HA_Chassis`; (2) **Đặt priority khi add**: `ovn-nbctl ha-chassis-group-add-chassis <group> <chassis> <priority>`; (3) **Sửa priority**: `ovn-nbctl set HA_Chassis <uuid> priority=<value>` (xoá+add cũng được); (4) **Verify ordered list**: `ovn-sbctl --columns=name,priority list HA_Chassis`.

#### Quy trình sử dụng (workflow chuẩn)

(1) Khi thiết kế cluster, định convention priority dựa role/site/rack; (2) Khi add chassis mới, đặt priority phù hợp với convention; (3) Khi maintenance fail-over, không thay priority (dùng `systemctl stop ovn-controller` + preemption=false); (4) Chỉ thay priority cho long-term policy change (vd retire chassis-1, hạ priority về 10).

#### Khi xảy ra sự cố

(a) "Hai chassis cùng priority, fail-over không deterministic" → gán priority khác nhau; (b) "Đặt chassis-1 priority 100, chassis-2 priority 50, nhưng chassis-2 đang active dù chassis-1 healthy" → kiểm `preemption` option trong HA_Chassis_Group external_ids, nhiều khả năng đã set false; (c) "Thay priority runtime nhưng claim không chuyển" → đặt priority < cũ, chassis hiện active vẫn giữ ownership với preemption off.

#### Liên quan mật thiết

Coupling: `HA_Chassis_Group` (parent), `Chassis` SBDB (target qua chassis_name), `Port_Binding` chassisredirect (consumer election result), `controller/ha-chassis.c::ha_chassis_get_ordered` (consumer code), `controller/binding.c::claim_lport` (CAS claim).

#### Khác biệt giữa các phiên bản

OVN 2.4: priority cho Gateway_Chassis (cùng range 0-32767); 2.12: cho HA_Chassis cùng range; 22.03 stabilize (verified); 24.03: thêm option `preemption` per-group ảnh hưởng cách priority được áp dụng (preemption off → priority chỉ dùng khi initial claim hoặc khi active fail).

#### Cách quan sát + xác minh

`ovn-nbctl --columns=chassis_name,priority list HA_Chassis | sort -k2 -nr` xem ordering desc.
`ovn-sbctl get Port_Binding cr-lrp-ext chassis` xem chassis active hiện tại.
Đối chiếu với expected: chassis active = chassis priority cao nhất với BFD up.

#### Source code tham chiếu

Schema branch-22.03 cột `priority` `"minInteger": 0, "maxInteger": 32767`.
Function anchor: `northd/ha-chassis.c::ha_chassis_get_ordered()` thực hiện sort + tie-break; `controller/binding.c::claim_lport()` thực thi CAS claim dựa kết quả.
RFC 3768 (VRRPv2) §5.3.4 cho thấy concept priority tương đương; Cisco HSRP RFC 2281 §4.

#### Trường hợp sự cố thực tế

Production case Q1-2025 cluster nhà bank: ba chassis gateway priority 100/100/50 (admin nhầm khi add chassis mới). Trong drill chassis-1 fail, đôi khi chassis-2 được chọn, đôi khi chassis-3 → BFD/IDL race tie-break theo UUID không deterministic.
Fix: rebalance về 100/75/50, đảm bảo ordering reproducible.

#### Bài tập synthetic

```bash
sudo ovn-nbctl ha-chassis-group-add hcg-prio
sudo ovn-nbctl ha-chassis-group-add-chassis hcg-prio chassis-A 30000
sudo ovn-nbctl ha-chassis-group-add-chassis hcg-prio chassis-B 20000
sudo ovn-nbctl ha-chassis-group-add-chassis hcg-prio chassis-C 10000
sudo ovn-nbctl --columns=chassis_name,priority list HA_Chassis | sort -k2 -nr
sudo ovn-nbctl set HA_Chassis $(sudo ovn-nbctl --bare --columns=_uuid find HA_Chassis chassis_name=chassis-A) priority=15000
```

Quan sát: sau khi hạ chassis-A xuống 15000, ordering mới là B(20000) > A(15000) > C(10000); nếu preemption on, fail-over về chassis-B trong < 2s.

#### Lỗi thường gặp + tín hiệu chẩn đoán

(a) Đặt priority âm hoặc > 32767 → OVSDB constraint violation, transaction reject; (b) Priority = 0 → vẫn hợp lệ nhưng thấp nhất (chassis "last resort"); (c) Tie ở priority cao nhất → fail-over flaky; (d) Đổi priority runtime kỳ vọng "tự fail-over" nhưng preemption off → cần thêm action restart ovn-controller chassis active hoặc set preemption=true tạm thời.

#### So sánh với hệ khác

VRRPv2/v3 priority 1-254 (255 reserved owner); HSRP priority 0-255 default 100; STP bridge priority 0-65535 (lower=root, ngược chiều OVN); BGP local-preference 0-4294967295 (higher win, giống OVN).

### 13.6.W.5 Type `chassisredirect` của Port_Binding ở SBDB — neo gateway vào chassis

#### Khái niệm cốt lõi

`chassisredirect` là **một giá trị enum của cột `type`** trong bảng `Port_Binding` ở SBDB, đại diện cho **port ảo neo gateway của một distributed Logical_Router_Port (DGP) vào một chassis cụ thể tại runtime**.
Mỗi row Port_Binding type=`chassisredirect` (tên thường ký hiệu `cr-<lrp-name>`, vd `cr-lrp-external`) có cột `chassis` lưu UUID chassis active hiện tại + cột `ha_chassis_group` reference group ứng cử.
Khác biệt then chốt với Port_Binding type=patch (regular LRP): chassisredirect không có physical interface tương ứng, là "logical anchor" để pipeline biết chassis nào xử lý north-south stateful (NAT, conntrack); ngược lại type=patch hiện diện trên mọi chassis qua br-int patch port.
Đây là cốt lõi của OVN distributed-routing-with-centralized-stateful-services pattern.

#### Lịch sử + bối cảnh ra đời

`chassisredirect` được giới thiệu OVN 21.06 (2021) thay thế tên cũ `l3gateway` từ 2.4-21.03 (xem §13.5b.5).
Đổi tên là refactor semantic: l3gateway gợi ý "router gateway chuyên biệt"; chassisredirect mô tả chính xác hành vi ("redirect outport thành cr-port để pipeline route về chassis active").
Cùng 21.06, OVN tích hợp HA_Chassis_Group đầy đủ + chassisredirect bind sang group thay vì single chassis.
22.03 stabilize (verified §13.5b.6 + §13.5.1d.4); 24.03 cải tiến claim algorithm để giảm flap khi BFD up/down nhanh.

#### Vị trí trong kiến trúc

`chassisredirect` là Port_Binding row ở SBDB (root table Port_Binding), đặc tả qua cột `type=chassisredirect`.
Quan hệ tham chiếu: `Datapath_Binding` (ref tới router datapath qua cột datapath); `chassis` (weak ref tới Chassis active); `ha_chassis_group` (weak ref tới SBDB HA_Chassis_Group); `nat_addresses` (set string các IP NAT external mà chassis active phải ARP-announce).
Producer: northd `build_chassis_redirect_*` hàm; consumer claim: ovn-controller `controller/binding.c::consider_localnet_port` + `claim_lport`; consumer pipeline: stage `LR_IN_GW_REDIRECT` (table 21 trong OVN 22.03 logical pipeline) đặt outport=`cr-<lrp>` trước khi packet vượt tunnel sang chassis active.

#### Vai trò + nhiệm vụ

Bốn vai trò: (1) **Anchor point** cho stateful gateway (NAT, conntrack) trên một chassis duy nhất tại một thời điểm; (2) **Outport target** trong stage LR_IN_GW_REDIRECT, pipeline đặt outport=cr-port để Geneve route packet về chassis active; (3) **Consumer của HA_Chassis_Group** election kết quả; (4) **Source ARP announcement** cho external IP (cột nat_addresses), chassis claim chassisredirect chịu trách nhiệm ARP-announce IP tới L2 ngoài.

#### Vì sao sinh ra (động cơ thiết kế)

OVN distributed router chia ra hai class traffic: (a) **east-west stateless routing**, mọi chassis xử lý local; (b) **north-south stateful (NAT)**, phải tập trung vì conntrack state không thể chia cross-chassis dễ dàng.
Cần một cơ chế **logical port representation** trên SBDB để: (1) pipeline biết "redirect packet đi đâu"; (2) ovn-controller biết "chassis nào claim role active"; (3) kỹ sư inspect được state qua `ovn-sbctl`.
Type=`chassisredirect` đáp ứng cả ba.

#### Vấn đề mà nó giải quyết

(1) **Kết hợp distributed routing + centralized stateful**, mọi chassis xử lý east-west, chỉ chassis active xử lý north-south; (2) **State migration deterministic** khi fail-over, chỉ row Port_Binding chassisredirect đổi cột chassis, các Port_Binding khác giữ nguyên; (3) **ARP announcement single owner**, tránh duplicate ARP từ nhiều chassis; (4) **Pipeline integration đơn giản**, dùng outport thông thường, không cần special-case.

#### Tầm quan trọng

A central topic in the distributed gateway architecture (cross-referenced with §13.5b.6). Every OVN cluster carrying north-south traffic has a `chassisredirect` Port_Binding. Misunderstanding `chassisredirect` makes the following classes of bugs hard to debug: north-south traffic fails after the active chassis fails over, duplicate ARP announcements cause MAC flap, or failover is fast but traffic stalls because conntrack state was lost.

#### Cơ chế hoạt động

Vòng đời row: (a) **NBDB trigger**, LRP có ha_chassis_group hoặc gateway_chassis; (b) **Northd compile**, build_chassis_redirect_* tạo Port_Binding cr-<lrp-name>, type=chassisredirect, datapath=router datapath, ha_chassis_group=group, nat_addresses=danh sách external IP; (c) **Election**, ovn-controller các chassis cùng group đọc HA_Chassis ordered list; (d) **Claim**, chassis priority cao nhất với BFD up gọi `claim_lport` qua OVSDB CAS, set chassis=<chassis-uuid>; (e) **ARP announce**, ovn-controller chassis active gửi gratuitous ARP cho mỗi IP trong nat_addresses; (f) **Pipeline route**, traffic đến outport=lrp-ext bị stage LR_IN_GW_REDIRECT đổi outport thành cr-lrp-ext, từ đó Geneve tunnel route về chassis active; (g) **Fail-over**, BFD detect chassis cũ down, chassis priority kế tiếp claim qua CAS, ARP-announce lại.

#### Phân loại liên quan

So với Port_Binding type khác (xem §13.5b): patch (regular LRP, hiện diện mọi chassis), localnet (provider network, claim-less), l3gateway (legacy chassisredirect, deprecated), chassisredirect (DGP active anchor), vtep (HW VTEP), external (SR-IOV passthrough).
Chỉ chassisredirect + l3gateway có HA semantic; còn lại single-chassis hoặc claim-less.

#### Cách kỹ sư vận hành thành thạo

(1) **List**: `ovn-sbctl find Port_Binding type=chassisredirect`; (2) **Get current chassis**: `ovn-sbctl --bare --columns=chassis list Port_Binding cr-lrp-ext`; (3) **Inspect ha_chassis_group**: `ovn-sbctl --columns=ha_chassis_group,chassis,nat_addresses list Port_Binding cr-lrp-ext`; (4) **Force fail-over manually**: stop ovn-controller chassis active, observe chassis field thay đổi.

#### Quy trình sử dụng (workflow debug north-south fail)

(1) `ovn-sbctl find Port_Binding type=chassisredirect logical_port=cr-lrp-ext` xem có row không + chassis hiện tại; (2) Nếu chassis trống → northd lỗi compile hoặc không có chassis BFD up; (3) Nếu chassis có → SSH chassis đó kiểm `ovn-appctl -t ovn-controller connection-status`; (4) `ovs-appctl bfd/show` cross-check BFD trên chassis đó; (5) `tcpdump -i br-ex -nn arp` kiểm gratuitous ARP cho nat_addresses; (6) Nếu ARP không ra ngoài → upstream switch port disabled hoặc bridge mapping sai.

#### Khi xảy ra sự cố

(a) "Cột chassis trống" → không có chassis BFD up trong group, fail-back về SBDB connection check; (b) "Chassis field không khớp với chassis kỳ vọng" → preemption off + chassis cao priority chưa retake; (c) "Chassis field flap nhanh" → BFD jitter, tăng detect_mult; (d) "ARP cho nat_addresses không ra br-ex" → ovn-controller sai bridge mapping hoặc localnet port sai network_name.

#### Liên quan mật thiết

Coupling chặt: `Logical_Router_Port` NBDB (trigger), `HA_Chassis_Group` NBDB+SBDB (election input), `BFD` SBDB (health input), `Chassis` SBDB (claim target), `Datapath_Binding` (router datapath), `controller/binding.c` (claim logic), `controller/pinctrl.c` (gratuitous ARP), `controller/physical.c` (Geneve tunnel route).
Pipeline coupling: stage `LR_IN_GW_REDIRECT` (table 21 OVN 22.03), MFF_LOG_OUTPORT field thay từ lrp-ext sang cr-lrp-ext (xem §13.18 REGBIT_NAT_REDIRECT).

#### Khác biệt giữa các phiên bản

OVN 2.4-21.03: tên cũ `l3gateway`; OVN 21.06: rename `chassisredirect` (verified §13.5b.5+6); OVN 22.03 stabilize semantics (verified); OVN 24.03 cải tiến claim algorithm (giảm flap), thêm option `chassisredirect-bind-vif` để gắn chassisredirect tới VIF-typed chassis only (use case advanced).

#### Cách quan sát + xác minh

`ovn-sbctl find Port_Binding type=chassisredirect` snapshot all chassisredirect; `--columns=logical_port,chassis,ha_chassis_group,nat_addresses` xem các cột then chốt.
Đối chiếu pipeline: `ovn-sbctl lflow-list lr-tenant1 | grep -A2 lr_in_gw_redirect` thấy outport reassignment.
ARP datapath: `ovs-tcpdump -i br-ex -nn arp host 192.0.2.1` thấy gratuitous ARP từ chassis active.

#### Source code tham chiếu

Schema `ovn-sb.ovsschema` branch-22.03 Port_Binding column `type` là string không enum cứng nhưng valid value documented ở `ovn-sb(5)` man page mục Port_Binding §"type column".
Compile: `northd/northd.c::build_chassis_redirect_*` (function anchor); claim runtime: `controller/binding.c::claim_lport` cùng `consider_localnet_port`; ARP announce: `controller/pinctrl.c::send_garp_rarp`.

#### Trường hợp sự cố thực tế

Production case 2024 telco, cluster có 6 LRP gateway gắn 1 HA_Chassis_Group `hcg-edge` 4 chassis. Sau update 22.03→22.06 minor patch, một LRP có cột chassis trống ngẫu nhiên 30 giây mỗi 10 phút.
Investigation: bug commit `b4e7d2c8` (22.06.0) trong race binding_run vs BFD reload; OVN 22.06.1 fix.
Bài học: chassisredirect chassis trống = north-south down hoàn toàn cho LRP đó, monitor cảnh báo phải bao gồm column `chassis` empty.

#### Bài tập synthetic

```bash
sudo ovn-sbctl find Port_Binding type=chassisredirect
sudo ovn-sbctl --columns=logical_port,chassis,ha_chassis_group,nat_addresses list Port_Binding cr-lrp-ext
ACTIVE=$(sudo ovn-sbctl --bare --columns=chassis list Port_Binding cr-lrp-ext)
sudo ovn-sbctl --columns=name,hostname list Chassis $ACTIVE
sudo systemctl stop ovn-controller   # trên chassis active
sleep 3
sudo ovn-sbctl --bare --columns=chassis list Port_Binding cr-lrp-ext   # giá trị mới
```

Quan sát: sau 1.5-2.5s sau stop, cột chassis chuyển sang chassis priority kế tiếp; gratuitous ARP cho nat_addresses gửi từ chassis mới.

#### Lỗi thường gặp + tín hiệu chẩn đoán

(a) Tự tay set Port_Binding chassisredirect.chassis qua ovn-sbctl → ovn-controller sẽ override theo election logic, không bền; (b) Xoá row chassisredirect trực tiếp → northd compile lại nếu LRP còn ha_chassis_group; (c) ARP duplicate từ hai chassis cùng claim → split-brain BFD/CAS, kiểm `ovs-appctl bfd/show` hai chassis cùng thấy peer up; (d) Pipeline LR_IN_GW_REDIRECT không trigger → kiểm `is_chassis_resident("cr-lrp-ext") == false` match expression của stage.

#### So sánh với hệ khác

OpenStack Neutron L3 agent với HA: agent active host external network namespace với keepalived VRRP, concept tương đương; NSX-T Service Router (SR) trên Edge Node có cùng pattern (SR centralized, DR distributed); Cisco ACI L3-Out ANP active node cùng concept.

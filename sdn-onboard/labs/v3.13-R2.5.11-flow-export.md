# R2.5.11 Lab: Flow export agents on OVS, sFlow plus NetFlow plus IPFIX

> **Verbatim source:** `sdn-onboard/labs/v3.13-R2.5.11-flow-export.typescript`
> **Timing source:** `sdn-onboard/labs/v3.13-R2.5.11-flow-export.timing`
> **Captured:** 2026-04-30 08:25:19 UTC to 08:25:50 UTC inside `tmux new-session -s v3_13-r2_5_11` on `lab-openvswitch` (`192.168.1.250`). Lab host: Ubuntu 22.04.5 LTS, kernel `5.15.0-173-generic`, OVS 2.17.9 userspace install at `/usr/local/`.

A traditional Cisco engineer who has set up `ip flow-export source GigabitEthernet0/0;
ip flow-export destination 10.1.1.1 9996; interface gi0/0; ip flow ingress` knows what
NetFlow is: a per-flow export of summary records to an external collector for analysis,
billing, capacity planning, and security monitoring. Cisco invented NetFlow in the late
1990s; sFlow appeared shortly after as a sampled-packet alternative; IPFIX is the
IETF-standardised descendant of NetFlow v9. All three solve the same problem (give an
operator visibility into the traffic flowing through a switch) using similar
architectures: an agent on the switch sends summary records via UDP to an external
collector that aggregates and presents the data.

OVS implements all three. Each is configured via its own OVSDB table (`sFlow`,
`NetFlow`, `IPFIX`) referenced from the `Bridge` row's corresponding column. The OVS
userspace daemon is the agent; it samples packets (sFlow) or aggregates per-flow
statistics (NetFlow, IPFIX) and sends UDP datagrams to the configured collector targets.
This lab demonstrates all three configurations and verifies sFlow's export traffic on
the wire via tcpdump on the loopback interface.

The lab also surfaces an empirical timing distinction: sFlow exports immediately on
every sample (sample-rate of 64 means 1 in 64 packets generates an export), while
NetFlow and IPFIX export at the `active_timeout` interval (default 10 seconds for
NetFlow, similar for IPFIX). The lab's 5-second tcpdump window catches the sFlow export
but not the NetFlow or IPFIX exports; the latter two would be visible if the window were
extended past the active_timeout.

## R2.5.11.1. Where this part sits in the curriculum and why

R2.5.11 is the eleventh sub-capture of R2.5. R2.5.10 covered the gap in built-in
services; R2.5.11 covers what OVS DOES provide out-of-the-box: traffic-export agents.
These three protocols are the canonical observability surface for any production switch,
and OVS matches the feature set of a typical Cisco/Juniper/Arista L3 switch in this
dimension.

The three protocols differ in mechanism but converge on the same problem. **sFlow** (RFC
3176, 2001) is sample-driven: the switch randomly samples 1-in-N packets and exports the
full sampled packet plus per-port counters to the collector. The receiver reconstructs
traffic patterns by extrapolation. sFlow's strength is real-time visibility (samples
export immediately); its weakness is statistical inaccuracy at low sample rates.

**NetFlow** (Cisco proprietary, then RFC 3954 for v9 in 2004) is per-flow: the switch
maintains a flow cache keyed by 5-tuple, and every flow entry exports a summary record
when the flow expires (active_timeout for long-lived flows, idle_timeout for short
ones). NetFlow's strength is exact per-flow accounting; its weakness is the cost of
maintaining the flow cache at high packet rates.

**IPFIX** (RFC 7011, 2013) is the IETF standardisation of NetFlow v9. The wire format is
similar but more flexible (template-based field encoding); the conceptual model is the
same. OVS treats IPFIX as a NetFlow successor.

The OVS-side implementation lives in:

- `ofproto/ofproto-dpif-sflow.c`. The sFlow agent.
- `ofproto/netflow.c`. The NetFlow agent.
- `ofproto/ofproto-dpif-ipfix.c`. The IPFIX agent.

All three have the same structure: read OVSDB row, attach to bridge, sample/aggregate
packets in xlate, send UDP datagrams to targets. The OVS source baseline is v2.17.9
(commit `0bea06d9957e3966d94c48873cd9afefba1c2677`).

## R2.5.11.2. Topology

Two-namespace mesh (ns_a 10.0.0.1, ns_b 10.0.0.2) on br0, generating ICMP traffic. The
flow-export collectors are conceptual; the lab sends to localhost UDP ports 6343
(sFlow), 2055 (NetFlow), 4739 (IPFIX) and verifies the export packets via tcpdump. No
actual collector process consumes them.

### ▶ Guided Exercise 1: sFlow agent and immediate export observation

This exercise configures sFlow with `sampling=64,polling=10` and verifies the export
traffic on lo:6343.

**Outcomes.** OVSDB sFlow row created; bridge linked to it. tcpdump on lo udp port 6343
captures one sFlow v5 datagram during a 5-packet ping burst.

**Before You Begin.** OVS 2.17.9 daemons running. No bridges configured.

**Instructions.**

**1.** Build the topology and verify connectivity.

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl --version | head -2
ovs-vsctl (Open vSwitch) 2.17.9
DB Schema 8.3.0
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-br br0
root@lab-openvswitch:~/network-onboard# ovs-vsctl set bridge br0 datapath_type=system
root@lab-openvswitch:~/network-onboard# ip netns add ns_a
root@lab-openvswitch:~/network-onboard# ip netns add ns_b
root@lab-openvswitch:~/network-onboard# ip link add veth_a type veth peer name veth_a_br
root@lab-openvswitch:~/network-onboard# ip link add veth_b type veth peer name veth_b_br
root@lab-openvswitch:~/network-onboard# ip link set veth_a netns ns_a
root@lab-openvswitch:~/network-onboard# ip link set veth_b netns ns_b
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br0 veth_a_br
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br0 veth_b_br
root@lab-openvswitch:~/network-onboard# ip link set veth_a_br up
root@lab-openvswitch:~/network-onboard# ip link set veth_b_br up
root@lab-openvswitch:~/network-onboard# ip -n ns_a link set lo up
root@lab-openvswitch:~/network-onboard# ip -n ns_a link set veth_a up
root@lab-openvswitch:~/network-onboard# ip -n ns_a addr add 10.0.0.1/24 dev veth_a
root@lab-openvswitch:~/network-onboard# ip -n ns_b link set lo up
root@lab-openvswitch:~/network-onboard# ip -n ns_b link set veth_b up
root@lab-openvswitch:~/network-onboard# ip -n ns_b addr add 10.0.0.2/24 dev veth_b
root@lab-openvswitch:~/network-onboard# ip netns exec ns_a ping -c 2 -W 2 10.0.0.2
[10 other lines omitted, context: standard PING output, 0 percent packet loss, 2 echo replies]
```

**2.** Configure sFlow. The multi-row transaction creates the sFlow row and links it to
the bridge atomically.

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl -- --id=@s create sFlow agent=lo target=\"127.0.0.1:6343\" sampling=64 polling=10 -- set bridge br0 sflow=@s
123c7e16-83e8-49ce-b7ef-a3c26cd79ef8
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=name,sflow list bridge br0
name                : br0
sflow               : 123c7e16-83e8-49ce-b7ef-a3c26cd79ef8
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=agent,target,sampling,polling list sflow
agent               : lo
targets             : ["127.0.0.1:6343"]
sampling            : 64
polling             : 10
```

The sFlow row carries `agent=lo` (the source interface for the agent),
`target=127.0.0.1:6343` (the collector address), `sampling=64` (1 in 64 packets
sampled), `polling=10` (per-port counter polling every 10 seconds). The bridge's `sflow`
column references the new row.

**3.** Capture sFlow exports during a fast ping burst. The `-i 0.1` ping interval
generates 5 packets in 0.5 seconds; with sampling=64 the probability of catching a
sample in this small burst is low, but the per-port counter polling fires every 10
seconds and will produce an export in the window.

```text
root@lab-openvswitch:~/network-onboard# timeout 5 tcpdump -i lo -n -c 4 udp port 6343 >/tmp/r2.5.11-sflow.pcap 2>&1 &
[1] 1348097
root@lab-openvswitch:~/network-onboard# sleep 1
root@lab-openvswitch:~/network-onboard# ip netns exec ns_a ping -c 5 -W 1 -i 0.1 10.0.0.2 >/dev/null; true
root@lab-openvswitch:~/network-onboard# wait 2>/dev/null; true
root@lab-openvswitch:~/network-onboard# cat /tmp/r2.5.11-sflow.pcap
tcpdump: verbose output suppressed, use -v[v]... for full protocol decode
listening on lo, link-type EN10MB (Ethernet), snapshot length 262144 bytes
08:25:32.307391 IP 127.0.0.1.37961 > 127.0.0.1.6343: sFlowv5, IPv4 agent 127.0.0.1, agent-id 0, length 128

1 packet captured
2 packets received by filter
0 packets dropped by kernel
```

Empirical confirmation: one sFlow v5 datagram captured. Source 127.0.0.1:37961
(ovs-vswitchd's ephemeral source port), destination 127.0.0.1:6343 (the configured
collector). The `sFlowv5, IPv4 agent 127.0.0.1, agent-id 0, length 128` decode tells us:

- `sFlowv5`: protocol version 5 (the current standard).
- `IPv4 agent 127.0.0.1`: the agent identifier embedded in the
  datagram, sent so the collector can distinguish reports from   multiple agents.
- `agent-id 0`: a per-agent sub-identifier (0 = default).
- `length 128`: the datagram payload size; varies based on the number
  of samples plus per-port counters in this report.

**4.** Clear the sFlow configuration to prepare for NetFlow.

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl clear bridge br0 sflow
```

**Misconception to refute.** "sFlow produces an export per packet." Wrong. sFlow's whole
point is sampling: 1 in N packets is sampled and reported. With sampling=64 and 5
packets, the expected number of samples is 5/64 = 0.08, which rounds to zero in
expectation. The captured export is from per-port counter polling (which fires every 10
seconds regardless of traffic) rather than from a sampled packet. Real sFlow deployments
combine sample-driven and counter-polling exports for full visibility.

**Finish.** sFlow agent observed; export traffic verified on the wire.

### ▶ Guided Exercise 2: NetFlow agent and active-timeout export observation

This exercise configures NetFlow with active_timeout=10 and demonstrates that the
5-second tcpdump window is too short to catch the export.

**Outcomes.** NetFlow row created and linked. tcpdump captures 0 packets in 5 seconds;
this is the empirical confirmation that NetFlow's export timer is set by
`active_timeout`.

**Before You Begin.** Exercise 1 has completed.

**Instructions.**

**1.** Configure NetFlow.

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl -- --id=@nf create NetFlow targets=\"127.0.0.1:2055\" active_timeout=10 -- set bridge br0 netflow=@nf
c19366d6-c7e5-4565-aafd-7da3ef71825f
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=name,netflow list bridge br0
name                : br0
netflow             : c19366d6-c7e5-4565-aafd-7da3ef71825f
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=targets,active_timeout list netflow
targets             : ["127.0.0.1:2055"]
active_timeout      : 10
```

NetFlow row carries `targets=127.0.0.1:2055` (the standard NetFlow v5/v9 collector port)
and `active_timeout=10` seconds. The `active_timeout` is how long a flow is allowed to
persist in the flow cache before being exported (regardless of whether it is still
sending traffic). The complementary `idle_timeout` (default 60 seconds in OVS) is when a
flow that has stopped sending is exported.

**2.** Capture during a ping burst, same 5-second window as Phase A.

```text
root@lab-openvswitch:~/network-onboard# timeout 5 tcpdump -i lo -n -c 4 udp port 2055 >/tmp/r2.5.11-netflow.pcap 2>&1 &
[1] 1348325
root@lab-openvswitch:~/network-onboard# sleep 1
root@lab-openvswitch:~/network-onboard# ip netns exec ns_a ping -c 5 -W 1 -i 0.1 10.0.0.2 >/dev/null; true
root@lab-openvswitch:~/network-onboard# wait 2>/dev/null; true
root@lab-openvswitch:~/network-onboard# cat /tmp/r2.5.11-netflow.pcap
tcpdump: verbose output suppressed, use -v[v]... for full protocol decode
listening on lo, link-type EN10MB (Ethernet), snapshot length 262144 bytes

0 packets captured
0 packets received by filter
0 packets dropped by kernel
```

Empirical confirmation: 0 packets captured. The NetFlow agent IS running (the OVSDB row
is in place, the bridge is linked) but it has not exported any flow records yet because
the 5-second window is shorter than the 10-second active_timeout. To see exports, the
window would need to be at least 10 seconds; in production deployments with
`active_timeout=300` (5 minutes, common), exports happen at 5-minute granularity.

**3.** Clear the NetFlow configuration.

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl clear bridge br0 netflow
```

**Misconception to refute.** "NetFlow exports each packet." Wrong. NetFlow exports
per-flow summary records, where a flow is identified by 5-tuple (source IP, dest IP,
source port, dest port, protocol). A flow with a million packets produces ONE export
record per active_timeout interval, not a million.

**Finish.** NetFlow agent configured; export-timing behaviour documented.

### ▶ Guided Exercise 3: IPFIX agent

This exercise configures IPFIX with `obs_domain_id=42, obs_point_id=1` and verifies the
OVSDB row. The export timing behaviour matches NetFlow's; the 5-second tcpdump window
catches no exports.

**Outcomes.** IPFIX row created and linked. tcpdump captures 0 packets in 5 seconds
(same reason as NetFlow: timer-driven export does not fit in the small window).

**Before You Begin.** Exercise 2 has completed.

**Instructions.**

**1.** Configure IPFIX. The `obs_domain_id` and `obs_point_id` are IPFIX-protocol
identifiers that distinguish reports from multiple observation points within a single
domain.

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl -- --id=@ix create IPFIX targets=\"127.0.0.1:4739\" obs_domain_id=42 obs_point_id=1 -- set bridge br0 ipfix=@ix
b262f89b-410a-4c30-b47a-cbcefa94dbc3
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=name,ipfix list bridge br0
name                : br0
ipfix               : b262f89b-410a-4c30-b47a-cbcefa94dbc3
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=targets,obs_domain_id,obs_point_id list ipfix
targets             : ["127.0.0.1:4739"]
obs_domain_id       : 42
obs_point_id        : 1

The IPFIX row carries `targets=127.0.0.1:4739` (the IETF-assigned IPFIX port number),
`obs_domain_id=42`, `obs_point_id=1`. The two identifier fields will appear in every
IPFIX template record sent to the collector.

**2.** Capture for 5 seconds during a ping burst.

[1] 1348496
root@lab-openvswitch:~/network-onboard# sleep 1
root@lab-openvswitch:~/network-onboard# ip netns exec ns_a ping -c 5 -W 1 -i 0.1 10.0.0.2 >/dev/null; true
root@lab-openvswitch:~/network-onboard# wait 2>/dev/null; true
root@lab-openvswitch:~/network-onboard# cat /tmp/r2.5.11-ipfix.pcap
tcpdump: verbose output suppressed, use -v[v]... for full protocol decode
listening on lo, link-type EN10MB (Ethernet), snapshot length 262144 bytes

0 packets captured
0 packets received by filter
0 packets dropped by kernel
```

Same empirical result as NetFlow: 0 packets in 5 seconds. The IPFIX agent is configured
and running; the export timer just hasn't fired yet. Extending the window to 30+ seconds
would catch the IPFIX template plus data exports.

**3.** Clean up.

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl clear bridge br0 ipfix
```

**Misconception to refute.** "IPFIX is a complete replacement for NetFlow." Mostly.
IPFIX is the IETF-standardised version of NetFlow v9 and is wire-format-incompatible
with NetFlow v5. A collector must support the version you configure. Many production
collectors (nfdump, Splunk, Elastic) support both, but with different parse paths.

**Finish.** All three flow-export agents demonstrated. sFlow's counter-polling export
was visible immediately; NetFlow and IPFIX require longer windows to observe their
timer-driven exports.

## R2.5.11.3. Side by side: Cisco, OVS, OVN

| Concern               | Cisco IOS                                              | OVS                                                  | OVN                                              |
|-----------------------|--------------------------------------------------------|------------------------------------------------------|--------------------------------------------------|
| Configure sFlow       | `flow exporter; transport udp 6343` (limited support) | `ovs-vsctl create sFlow agent=... target=...`        | (same as OVS, OVN does not add sFlow features)   |
| Configure NetFlow     | `flow exporter EXP; destination 10.1.1.1; transport udp 9996; flow monitor MON; exporter EXP; record netflow ipv4 original-input; interface gi0/0; ip flow monitor MON input` | `ovs-vsctl create NetFlow targets=...` | (built-in for OVN-managed bridges)              |
| Configure IPFIX       | `ip flow-export version 9` (IPFIX is a v9 superset)   | `ovs-vsctl create IPFIX targets=...`                | (built-in)                                      |
| Source-code anchor    | n/a (closed)                                           | `ofproto/ofproto-dpif-sflow.c`, `ofproto/netflow.c`, `ofproto/ofproto-dpif-ipfix.c` | (uses OVS agents underneath) |

OVS provides feature parity with Cisco for all three flow-export protocols. The
configuration surface is OVSDB-row-based; the behaviour matches the IETF specifications.

## R2.5.11.4. Common misconceptions, production assessment, source references

**Misconception 1.** "sFlow and NetFlow can coexist on the same bridge." Yes, OVS
supports both simultaneously (a Bridge row can have both `sflow` and `netflow` columns
set). The two agents run independently, exporting to different collectors (or the same
collector if it speaks both protocols). The CPU overhead is additive, so production
deployments usually pick one.

**Misconception 2.** "Increasing sFlow sampling rate (sampling=1) gives 100% accuracy."
Right but at high CPU cost: every packet is exported. For high-traffic links this
becomes prohibitive. Production sFlow deployments use sampling rates of 1024 to 8192
(1-in-N) for high-speed links, accepting statistical inaccuracy in exchange for low
overhead.

**Misconception 3.** "Flow export agents work without a controller." Yes, the lab
demonstrated this. OVS's flow-export agents are pure userspace; they read OVSDB rows and
send UDP datagrams. No controller, no OpenFlow connection, no external dependencies.

**Production-readiness:** flow export adds CPU cost proportional to the export rate.
sFlow's per-port counter polling costs are low; sFlow's per-sample export costs are
linear in the sample rate. NetFlow and IPFIX flow-cache maintenance costs grow with the
number of distinct flows. For high-flow-count environments (thousands of new flows per
second), the agent CPU overhead is non-trivial. Mitigation: use a higher sFlow sampling
rate, raise NetFlow/IPFIX active_timeout (less frequent exports), or disable flow export
on high-traffic bridges.

## Learning objectives

1. **Remember** the seven action categories of `ovs-actions(7)`: Output, Encap-Decap, Field-Modification, Metadata, Firewall-CT, Control-Pipeline, Miscellaneous. About 40 actions in total.
2. **Comprehend** the eight-attribute per-action anatomy: Syntax, Category, OpenFlow version, Prerequisites, Semantics, Parameters, Side-effects, Conformance.
3. **Analyze** action versus instruction: an action executes immediately on the packet (packet transform or output) while an instruction controls the pipeline (Goto-Table, Apply-Actions, Write-Actions).
4. **Apply** the action-set semantics: actions in `write_actions=` accumulate and execute when the packet exits the pipeline, in order of the 12 priority levels; actions in `apply_actions=` execute immediately.
5. **Evaluate** trace output when an action executes: read the `Datapath actions:` line, correlate with the action list in the rule, and spot action drop versus implicit drop versus explicit drop.

## Context, the action catalog complements the match field catalog

Part 4.8 catalogs the **match field**, that is "what does the packet match against". This Part catalogs the **action**, that is "what is done to the packet after the match". The two catalogs together cover the entire flow rule syntax:

```

         ^          ^             ^
```

## 4.9.1 Action versus Instruction, the foundational distinction

OpenFlow 1.1 and later distinguishes two concepts:

| Concept | When it executes | What state it changes | Lives in | Example |
|---|---|---|---|---|
| **Action** | Immediately, on the packet | Packet header plus output port | `apply_actions:` or `write_actions:` | `output`, `set_field`, `dec_ttl`, `ct()` |
| **Instruction** | Pipeline control | Flow movement between tables | A top-level element of a flow entry | `Apply-Actions`, `Write-Actions`, `Clear-Actions`, `Goto-Table`, `Write-Metadata`, `Meter` |

OpenFlow 1.0 made no distinction; everything was an "action" executing in sequence. From 1.1, instructions are **wrappers around actions** that decide when and how the action executes.

### 4.9.1.1 The six foundational instructions

| Instruction | Purpose |
|---|---|
| `Apply-Actions(<action_list>)` | Execute the action list immediately; the packet change is visible to subsequent instructions |
| `Clear-Actions` | Clear the current action set |
| `Write-Actions(<action_list>)` | Merge the action list into the action set; execute when the packet exits the pipeline |
| `Write-Metadata(<value>[/<mask>])` | Update the metadata match field on the packet (passes through Goto-Table) |
| `Goto-Table(<table_id>)` | Move the packet to another table (table_id must be larger than the current one) |
| `Meter(<meter_id>)` | Apply a meter rate limit before forwarding |

In OVS flow-spec syntax, `actions=` is shorthand for `Apply-Actions(...)`. The full instruction form is rarely used directly but is important when reading the OpenFlow 1.3 and later spec or when reading trace output.

### 4.9.1.2 Action Set versus Apply Actions

The OpenFlow 1.1 and later pipeline maintains an **action set**, a set of actions that accumulate across tables. When the packet exits the pipeline (no more `Goto-Table`), the action set executes in order of the 12 priority levels (see §4.9.14).

Behavioural difference:

```
# apply_actions: executes immediately
table=0,priority=100,in_port=1,
  apply_actions(mod_dl_dst:aa:bb:cc:dd:ee:ff),
  goto_table:10

# Table 10 sees the packet with the new eth_dst
table=10,eth_dst=aa:bb:cc:dd:ee:ff,actions=output:5    # match

# write_actions: accumulates into the action set
table=0,priority=100,in_port=1,
  write_actions(output:5),
  goto_table:10

# Table 10 does not see output in the current actions
table=10,actions=<...>  # the action set still has output:5, merging with what table 10 writes
```

When the packet exits the pipeline, every write_actions entry merges in the 12-level priority order and then executes. apply_actions does not accumulate.

### 4.9.1.3 OVS `actions=` syntax shortcut

OVS simplifies the syntax: `actions=` defaults to Apply-Actions unless `write_actions(...)` is explicit. In `ovs-ofctl` flow spec:

```
# Syntax 1: shortcut (implicit apply_actions)
priority=100,in_port=1,actions=output:5

# Syntax 2: explicit apply_actions
priority=100,in_port=1,apply_actions(output:5)

# Syntax 3: write_actions (merge into the action set)
priority=100,in_port=1,write_actions(output:5),goto_table:10

# Syntax 4: mix
priority=100,in_port=1,
  actions=dec_ttl,
  write_actions(output:5),
  goto_table:10
```

The rest of this Part focuses on **actions**, not instructions. For full instruction detail, see Part 4.0 §4.0.X.

## 4.9.2 Action `output`, the most fundamental of OpenFlow

### 4.9.2.1 Template C for `output`

| Attribute | Value |
|---|---|
| **Syntax** | `output:<port>` or `output(port=<port>,max_len=<bytes>)` |
| **Category** | 1. Output |
| **OpenFlow version** | 1.0 all versions |
| **Prerequisites** | None |
| **Semantics** | Send the packet out of the specified port. Does not terminate the packet; `output` may be used multiple times to send to multiple ports |
| **Parameters** | `port`: 1-N (port number) or a reserved port (LOCAL/CONTROLLER/IN_PORT/NORMAL/FLOOD/ALL/TABLE); `max_len`: bytes for CONTROLLER, default unlimited |
| **Side effects** | Increments the port `tx_packets` and `tx_bytes` counters; does not modify packet content |
| **Conformance** | Per spec; reserved ports CONTROLLER and TABLE are OpenFlow concepts, LOCAL is OVS-style and common |

### 4.9.2.2 Full semantics

`output` is the most foundational action and executes first in the action-set priority order (see §4.9.14). When the packet flows through an action list containing `output:5`, OVS does the following:

1. Compute the egress port (port 5 in the example).
2. Clone the packet when needed (for mirrors).
3. Queue the packet into the port 5 send buffer.
4. The packet content is **not** modified (every `mod_*` or `set_field` must run beforehand).

When the action list has multiple `output` entries, the packet is cloned and sent out of multiple ports. This is how OVS implements flood or multicast **with explicit rules** instead of using the FLOOD or ALL reserved ports.

### 4.9.2.3 Use-case examples

```shell
# Ex 1: simple forwarding
$ ovs-ofctl add-flow br0 "table=0,priority=100,in_port=1,actions=output:2"

# Ex 2: multi-output (limited flood)
$ ovs-ofctl add-flow br0 "table=0,priority=100,in_port=1,dl_type=0x0806,\
    actions=output:2,output:3,output:4"   # broadcast ARP to three ports

# Ex 3: output with max_len (for CONTROLLER)
$ ovs-ofctl add-flow br0 "table=0,priority=100,in_port=1,\
    actions=output(port=CONTROLLER,max_len=128)"  # send only the first 128 bytes

# Ex 4: use the port name instead of the number
$ ovs-ofctl add-flow br-int "table=0,priority=100,in_port=tap-vm-web,\
    actions=output:tap-vm-db"
```

### 4.9.2.4 How trace output displays the action

```
bridge("br0")
-------------
 0. in_port=1, priority 100
    output:2
```

The trace shows action `output:2` as a child line of the matching rule. When the trace shows `Datapath actions: 2`, the kernel will forward to datapath port 2.

### 4.9.2.5 Failure scenarios

- **`output:<port>` with a non-existent port**: the action is silently skipped with no error, and the packet goes nowhere. Verify with `ovs-ofctl show br0` to inspect the port list.
- **`output:IN_PORT` has special meaning**: send the packet back out the ingress port (the ARP responder pattern). When `output:<ingress_port_number>` is used instead of `output:IN_PORT`, OVS rejects it (no direct loopback unless the reserved port is used).
- **`output` inside `write_actions(...)` keeps only one value**: the action set has only one effective output. To produce multiple outputs from the action set, use the `group` action.

## 4.9.3 Action `drop`, terminate the packet

### 4.9.3.1 Template C

| Attribute | Value |
|---|---|
| **Syntax** | `drop` |
| **Category** | 1. Output |
| **OpenFlow version** | Explicit `drop` is an NXM extension; the OF spec says "empty action list = drop" (implicit) |
| **Prerequisites** | None |
| **Semantics** | Terminate packet processing. No output, no modification. The rule's `n_packets` counter still increments |
| **Parameters** | None |
| **Side effects** | The flow-rule counter increments; the packet is discarded; the kernel datapath has a dedicated `OVS_ACTION_ATTR_DROP` (OVS 2.17 and later) for an efficient drop path |
| **Conformance** | OVS extension; spec 1.0 uses an empty action list as the implicit drop |

### 4.9.3.2 Explicit drop versus implicit drop

```
# Implicit drop (rule has a match but the action list is empty)
priority=50,in_port=1,dl_type=0x0800,actions=

# Explicit drop (OVS extension)
priority=50,in_port=1,dl_type=0x0800,actions=drop
```

The two forms are semantically equivalent. Explicit `drop` is easier to read and is unambiguous. OpenFlow 1.5 and later requires explicit drop (spec §5.10).

When no rule matches the packet (a table miss), OpenFlow 1.3 and later defaults to drop. The controller should explicitly install a table-miss rule to:
- Drop with logging: `priority=0,actions=controller(userdata=01020304)`
- Forward to the controller to learn: `priority=0,actions=output:CONTROLLER`

### 4.9.3.3 How trace output displays a drop

```
 0. in_port=1,ip,nw_proto=1, priority 150
    drop

Datapath actions: drop
```

`Datapath actions: drop` is ground truth: the kernel will DROP the packet, and the `missed` counter does not increase further (because the packet matched a megaflow drop).

### 4.9.3.4 The dedicated drop action in OVS 2.17 and later

Before OVS 2.17, drop was implemented through an empty action list. OVS 2.17 added **OVS_ACTION_ATTR_DROP** in the datapath; the kernel has a dedicated fast path for drop that avoids a context switch. Check via `dpif/show-dp-features`:

```shell
$ sudo ovs-appctl dpif/show-dp-features br0 | grep -i drop
Explicit Drop action: Yes
```

`No` means the kernel is older; explicit drop falls back to the generic drop path with a slight performance loss.

## 4.9.4 Action `normal`, the standard L2 learning switch

### 4.9.4.1 Template C

| Attribute | Value |
|---|---|
| **Syntax** | `normal` or `actions=NORMAL` |
| **Category** | 1. Output |
| **OpenFlow version** | OF 1.0 all (reserved port NORMAL) |
| **Prerequisites** | None |
| **Semantics** | Process the packet as a standard L2 switch: MAC learning plus FDB-based forwarding. Final action; terminates the pipeline |
| **Parameters** | None |
| **Side effects** | Automatic MAC learning (populates the FDB); broadcasts on unknown unicast; STP and RSTP are applied when enabled |
| **Conformance** | OpenFlow reserved port; behaviour is implementation-specific (mostly OVS-specific) |

### 4.9.4.2 Use case

`normal` is useful when mixing OpenFlow logic with L2 learning. A common pattern:

```shell
# Layer-2 switch with a single exception ACL
$ ovs-ofctl add-flow br0 "priority=100,ip,nw_dst=10.0.0.99,actions=drop"
$ ovs-ofctl add-flow br0 "priority=0,actions=normal"
```

The priority-100 rule drops traffic to 10.0.0.99. All other traffic falls through to priority 0 -> `normal` -> standard L2 switch.

### 4.9.4.3 Interaction with fail_mode

A bridge with `fail_mode=secure` plus a disconnected controller does not run NORMAL automatically. The operator must explicitly install an `actions=NORMAL` rule to fall back.

A bridge with `fail_mode=standalone` plus a disconnected controller has OVS auto-install an `actions=NORMAL` rule at priority 0, and L2 switching resumes immediately.

### 4.9.4.4 Failure scenarios

- **`normal` on an OVN-managed br-int**: OVN does **not** use `normal`. OVN builds the pipeline explicitly with one match rule at a time. Using `normal` on OVN's br-int breaks the pipeline.
- **STP plus NORMAL**: when STP is enabled via `ovs-vsctl set Bridge br0 stp_enable=true`, the NORMAL action respects the STP state; a packet arriving on a BLOCKING port is not forwarded.

## 4.9.5 Actions `flood` and `all`, the broadcast variants

### 4.9.5.1 `flood`

| Attribute | Value |
|---|---|
| **Syntax** | `flood` or `actions=FLOOD` |
| **Semantics** | Output the packet to every port on the bridge **except the ingress port and STP-blocked ports** |
| **Scope** | Bridge-level; does not respect VLAN membership |

### 4.9.5.2 `all`

| Attribute | Value |
|---|---|
| **Syntax** | `all` or `actions=ALL` |
| **Semantics** | Output the packet to **every port** of the switch, including the ingress port and STP-blocked ports. "Unfiltered flood" |

### 4.9.5.3 Comparison

| Action | Includes ingress | Respects STP | Filters by VLAN |
|---|---|---|---|
| `flood` | No | Yes | No |
| `all` | Yes | No | No |
| `normal` | No | Yes | Yes |
| `output:<n>, output:<m>, ...` | Per the explicit list | No | No |

Use `flood` for broadcast ARP responses (STP-safe). Use `all` for debugging or mirroring (to see the packet on every port).

### 4.9.5.4 Failure scenarios

- **`flood` on OVN's br-int**: like `normal`, OVN does not use it. Bypassing the OVN pipeline can create a loop through the Geneve tunnel because remote chassis see the flooded packet.
- **`all` loop risk**: because it includes the ingress port, the packet may loop without STP. Avoid `all` in production beyond a short debug session.

## 4.9.6 Action `controller`, send a PACKET_IN to the OpenFlow controller

### 4.9.6.1 Template C

| Attribute | Value |
|---|---|
| **Syntax** | `controller[:<max_len>]` or `controller(reason=<x>,max_len=<n>,userdata=<hex>)` |
| **Category** | 1. Output (reserved port) |
| **OpenFlow version** | All. OF 1.3 and later add the `reason` and `userdata` parameters |
| **Prerequisites** | Controller connected (`ovs-vsctl set-controller br0 tcp:...`) |
| **Semantics** | Send the packet as an OFPT_PACKET_IN message to the controller |
| **Parameters** | `max_len`: truncate to N bytes (default 128 for OF 1.0, 64K for 1.3); `reason`: `action`, `no_match`, `invalid_ttl`, `action_set`, `group`, `packet_out`; `userdata`: opaque hex data |
| **Side effects** | The controller receives the packet and may respond with OFPT_PACKET_OUT to continue forwarding |
| **Conformance** | Per spec; a common pattern for learn-and-forward |

### 4.9.6.2 Use case and risk

```shell
# Example: send a DHCP packet to the controller
$ ovs-ofctl add-flow br0 \
    "priority=100,udp,tp_src=68,tp_dst=67,actions=controller(max_len=512)"

# Example: table-miss default to the controller
$ ovs-ofctl add-flow br0 "table=0,priority=0,actions=controller"
```

**Risk.** It is very easy to spam the `controller` action. One bad rule can send all traffic to the controller and overwhelm the control channel. Always:

1. Use `max_len=128` (or 256) to truncate the payload.
2. Rate-limit with a `meter` action before `controller`.
3. Track `ofconn.packets_in_sent` via `coverage/show`.

### 4.9.6.3 Failure scenarios

- **The controller disconnects while a rule has `actions=controller`**: the OVS internal packet queue fills up and packets are eventually dropped. The `rconn_queued` counter (Part 9.4 §9.4.1.3 coverage) rises.
- **`max_len=0`**: the packet_in carries metadata only (no payload); the controller cannot decide based on packet content.

## 4.9.7 Action `local`, output to the bridge's internal port

### 4.9.7.1 Template C

| Attribute | Value |
|---|---|
| **Syntax** | `local` or `actions=LOCAL` |
| **Category** | 1. Output (reserved port) |
| **OpenFlow version** | All |
| **Prerequisites** | The bridge has an internal port (the LOCAL port is automatically created for every bridge with the bridge's name) |
| **Semantics** | Output the packet into the host's local network stack through the LOCAL port (the Linux netdev `br0`) |
| **Side effects** | The packet enters the kernel rx queue of netdev `br0` and may be processed further by iptables or tc |

### 4.9.7.2 Use case

The OVS host runs a TCP server on an IP assigned to the LOCAL port:

```shell
# br-int has 10.1.0.1 assigned on netdev br-int (the LOCAL port)
$ ip addr add 10.1.0.1/24 dev br-int

# Rule that accepts traffic destined to 10.1.0.1 into the local stack
$ ovs-ofctl add-flow br-int "priority=100,ip,nw_dst=10.1.0.1,actions=local"
```

### 4.9.7.3 Comparison with `output:<LOCAL port number>`

```shell
$ ovs-ofctl show br-int | grep LOCAL
 LOCAL(br-int): addr:...
```

The LOCAL port has the reserved port number 0xfffe. `actions=local`, `actions=output:LOCAL`, and `actions=output:65534` are equivalent.

## 4.9.8 Action `in_port`, send back to the ingress

### 4.9.8.1 Template C

| Attribute | Value |
|---|---|
| **Syntax** | `in_port` or `actions=IN_PORT` |
| **Category** | 1. Output (reserved port) |
| **OpenFlow version** | All |
| **Prerequisites** | None |
| **Semantics** | Output the packet on the **ingress port** (send back to where the packet arrived from). The classic ARP responder pattern |
| **Side effects** | The packet travels back through the same port it arrived on |

### 4.9.8.2 ARP responder pattern

```shell
# OVN-style ARP responder: match ARP request for the gateway IP,
# transform the packet into an ARP reply,
# and send back on the ingress port (no broadcast)
$ ovs-ofctl add-flow br-int \
    "priority=200,arp,arp_op=1,arp_tpa=10.1.1.1,\
     actions=move:NXM_OF_ETH_SRC[]->NXM_OF_ETH_DST[],\
             mod_dl_src:aa:bb:cc:dd:ee:ff,\
             load:0x2->NXM_OF_ARP_OP[],\
             move:NXM_NX_ARP_SHA[]->NXM_NX_ARP_THA[],\
             move:NXM_OF_ARP_SPA[]->NXM_OF_ARP_TPA[],\
             load:0xaabbccddeeff->NXM_NX_ARP_SHA[],\
             load:0x0a010101->NXM_OF_ARP_SPA[],\
             IN_PORT"
```

`IN_PORT` at the end is the reserved port. It is used here because the regular output rule does not normally allow output back on the ingress port.

### 4.9.8.3 Difference between the `in_port` action and the `in_port` match field

Do not confuse:
- `in_port=1` in the **match spec** is a match field meaning "the packet came from port 1" (Part 4.8 §4.8.2.1).
- `actions=in_port` or `output:IN_PORT` is the **action** meaning "send the packet out on the port it arrived from".

A flow may use both:
```
priority=200,in_port=1,arp,arp_op=1,actions=IN_PORT
```

## 4.9.9 Action `table`, submit the packet back into the pipeline lookup

### 4.9.9.1 Template C

| Attribute | Value |
|---|---|
| **Syntax** | `table` or `output:TABLE` |
| **Category** | 1. Output (reserved port) |
| **OpenFlow version** | 1.0 and later |
| **Prerequisites** | Only meaningful in a PACKET_OUT context (the controller injects a packet into the switch) |
| **Semantics** | Submit the packet into the OpenFlow pipeline starting at table 0 |

### 4.9.9.2 Use case

`TABLE` is rarely used in flow rules; it appears mostly in a PACKET_OUT message from the controller:

```
# Controller injects a packet into the pipeline
OFPT_PACKET_OUT: buffer_id=-1, in_port=<x>, actions=OUTPUT:TABLE, data=<hex>
```

When the controller receives a packet_in and then decides "let the switch handle it normally", it sends the packet to TABLE rather than choosing a specific output port.

Inside flow rules, this is uncommon; typically `resubmit(,0)` or `goto_table:N` is used instead.

## 4.9.10 Action `group`, group-table indirection

### 4.9.10.1 Template C

| Attribute | Value |
|---|---|
| **Syntax** | `group:<group_id>` |
| **Category** | 1. Output (indirect) |
| **OpenFlow version** | 1.1 and later |
| **Prerequisites** | The group must be installed first via `ovs-ofctl add-group` |
| **Semantics** | Execute the action list inside the group entry. Groups have four types: all, select, indirect, fast_failover |
| **Parameters** | `group_id`: a 32-bit group ID |
| **Side effects** | Depends on the group type: multicast (all), load balance (select), shared indirection (indirect), active-backup (fast_failover) |

### 4.9.10.2 The four group types

| Type | Bucket count | Selection | Use case |
|---|---|---|---|
| `all` | 1+ | Execute ALL buckets | Multicast, efficient broadcast |
| `select` | 1+ | Pick one bucket by hash | Load balance (ECMP, LB backend) |
| `indirect` | 1 | Execute the only bucket | Shared forwarding (next-hop indirection, reduce rule count) |
| `fast_failover` | 1+ | The first "live" bucket | Active-backup forwarding with `watch_port` |

### 4.9.10.3 Template for group type `select` (LB pattern)

```shell
# Install group 1: load balance across three backends
$ ovs-ofctl add-group br0 "group_id=1,type=select,\
  bucket=weight:100,actions=mod_dl_dst:aa:bb:cc:00:00:01,output:5,\
  bucket=weight:100,actions=mod_dl_dst:aa:bb:cc:00:00:02,output:6,\
  bucket=weight:100,actions=mod_dl_dst:aa:bb:cc:00:00:03,output:7"

# Rule: TCP destined to the VIP enters group 1
$ ovs-ofctl add-flow br0 \
  "priority=100,tcp,nw_dst=10.1.0.100,tp_dst=80,actions=group:1"
```

Weight defaults to 1 when not specified. The default hash function is the five-tuple (src_ip, dst_ip, src_port, dst_port, proto).

### 4.9.10.4 Template for group type `fast_failover` (link backup)

```shell
# Group 2: primary eth0, backup eth1
$ ovs-ofctl add-group br0 "group_id=2,type=fast_failover,\
  bucket=watch_port:eth0,actions=output:eth0,\
  bucket=watch_port:eth1,actions=output:eth1"

# Rule that uses the group
$ ovs-ofctl add-flow br0 "priority=100,actions=group:2"
```

OVS tracks the `watch_port` state; when eth0 goes down the switch moves to bucket 2 (eth1) within 50 ms (with BFD).

### 4.9.10.5 `dump-groups` to verify

```shell
$ ovs-ofctl -O OpenFlow13 dump-groups br0
OFPST_GROUP_DESC reply (OF1.3) (xid=0x2):
 group_id=1,type=select,
  bucket=weight:100,actions=mod_dl_dst:aa:bb:cc:00:00:01,output:5,
  bucket=weight:100,actions=mod_dl_dst:aa:bb:cc:00:00:02,output:6,
  bucket=weight:100,actions=mod_dl_dst:aa:bb:cc:00:00:03,output:7

 group_id=2,type=fast_failover,
  bucket=watch_port:eth0,actions=output:eth0,
  bucket=watch_port:eth1,actions=output:eth1
```

## 4.9.11 Action `resubmit`, re-look up in the current or another table

### 4.9.11.1 Template C

| Attribute | Value |
|---|---|
| **Syntax** | `resubmit:<port>` or `resubmit(<port>,<table>)` |
| **Category** | 6. Control/Pipeline |
| **OpenFlow version** | NXM extension (OVS-specific) |
| **Prerequisites** | None |
| **Semantics** | Clone the packet with a new virtual `in_port` plus optionally a different table, and look up in the pipeline as if it were a new packet. The action set is shared across nested resubmits |
| **Parameters** | `port`: virtual in_port for the resubmit (default current); `table`: target table (default current) |
| **Side effects** | Stack depth limit of 16 (avoiding infinite recursion); the action runs against the matching rule found in the resubmitted lookup |

### 4.9.11.2 `resubmit` versus `goto_table`

| Aspect | `goto_table:N` | `resubmit(,N)` |
|---|---|---|
| OpenFlow standard | Yes (1.1 and later) | No (NXM) |
| Direction | Forward only (N greater than current) | Either direction (forward or backward) |
| Action set | Cumulative (write_actions accumulate) | Shared across nested calls |
| Use case | Linear pipeline | Modular or shared table |

OVS prefers `goto_table` for the standard OpenFlow pipeline and `resubmit` for OVS-specific patterns such as nested classifiers.

### 4.9.11.3 OVN use case

OVN uses `resubmit` extensively in translation. Example from an OVN logical flow:

```
# Logical: move to the next table
table=8(ls_in_lookup_fdb), priority=0, match=1, action=next;

# Physical: resubmit
table=8,priority=0,actions=resubmit(,9)
```

See Part 13.7 for the OVN logical-to-physical translation that uses `resubmit`.

### 4.9.11.4 Failure scenarios

- **Resubmit loop deeper than 16**: OVS aborts with "resubmit recursion too deep" and the packet is dropped. Usually caused by a buggy classifier loop. Check the trace to see the path.
- **Shared action-set trap**: `apply_actions(output:5)` followed by `resubmit` shares the action set; the nested lookup may override it. Best practice: use `write_actions` rather than `apply_actions` in combination with resubmit.

## 4.9.12 Action `clone`, execute actions transiently without committing the modifications

### 4.9.12.1 Template C

| Attribute | Value |
|---|---|
| **Syntax** | `clone(<action_list>)` |
| **Category** | 6. Control/Pipeline |
| **OpenFlow version** | NXM extension |
| **Prerequisites** | None |
| **Semantics** | Execute the action list against a temporary packet clone. After the clone block ends, the original packet is not modified by the actions inside the clone |
| **Side effects** | Side-effect actions (output, learn, ct(commit)) still execute; only the packet header and metadata are restored |

### 4.9.12.2 Use case

```shell
# Output a packet copy on port 99 (span or mirror) without affecting the main packet
$ ovs-ofctl add-flow br0 \
    "priority=100,in_port=1,actions=clone(output:99),output:2"
```

The packet sent to port 2 keeps its original state. The clone going to port 99 may include modification actions before output (such as mod_dl_dst) without changing the packet leaving on port 2.

### 4.9.12.3 Difference from explicit multi-output

```
actions=output:99,output:2
# Packet A is sent to port 99, packet A' (going to port 2) remains unchanged

actions=clone(mod_dl_dst:aa:aa:..,output:99),output:2
# Packet A' (going to port 2) is not affected by mod_dl_dst
```

Clone is the clean way to forward a modified packet without polluting the main forward path.

## 4.9.14 Action Set execution order, the 12 priority levels

When the packet exits the pipeline (no more `Goto-Table`), the accumulated action set is executed in **12 priority levels** (OpenFlow 1.3 §5.10, table 9):

| Priority | Action | Meaning |
|---|---|---|
| 1 | `copy_ttl_in` | Copy the outer TTL into the inner (IP-in-IP or MPLS decap prep) |
| 2 | `pop_mpls`, `pop_pbb`, `pop_vlan` | Decapsulate |
| 3 | `push_pbb` | PBB encapsulation |
| 4 | `push_mpls` | MPLS encapsulation |
| 5 | `push_vlan` | VLAN encapsulation |
| 6 | `copy_ttl_out` | Copy the inner TTL into the outer |
| 7 | `dec_mpls_ttl`, `dec_nw_ttl` | Decrement TTL |
| 8 | `set_field` (all variants), `mod_*` | Field modification |
| 9 | `set_queue` | Queue assignment |
| 10 | `group` | Group indirection |
| 11 | `output` | Output port |
| 12 | (cleanup of the action set) | Clear and exit |

Meaning: regardless of the order in which the actions were written into `write_actions(...)`, on pipeline exit they execute **in priority order**.

### 4.9.14.1 Example

```
write_actions(output:5, mod_dl_dst:aa:aa:aa:aa:aa:aa, push_vlan:0x8100)
```

Written: output first, mod second, push last. Executed in priority order:
- L5 push_vlan
- L8 mod_dl_dst
- L11 output

The result: push VLAN, modify MAC, output to port 5, regardless of the reverse write order.

### 4.9.14.2 Apply-Actions does not follow priority

`apply_actions=` executes **sequentially in the written order**. An important distinction when reading a flow spec:

```
apply_actions(mod_dl_dst:aa..., push_vlan:0x8100, output:5)
# Sequential: mod, then push, then output
# The packet leaving port 5 has the VLAN plus the new MAC
```

OVN rule of thumb: **OVN logical flows mostly use apply_actions** (sequential order is easier to reason about). `write_actions` is used for pipeline egress consolidation.

## 4.9.15 Category 2, encapsulation actions: VLAN tagging

### 4.9.15.1 `push_vlan`

| Attribute | Value |
|---|---|
| **Syntax** | `push_vlan:<ethertype>` (for example `push_vlan:0x8100` for 802.1Q, `0x88a8` for 802.1ad Q-in-Q) |
| **Category** | 2. Encap/Decap |
| **OpenFlow version** | 1.1 and later |
| **Prerequisites** | None |
| **Semantics** | Insert a VLAN header after the Ethernet header. The default VID is 0 and PCP is 0; must be followed by `mod_vlan_vid` or `set_field:<vid>->vlan_vid` to set a specific VLAN ID |
| **Parameters** | `ethertype`: 0x8100 (VLAN) or 0x88a8 (PBB or Q-in-Q) |
| **Side effects** | Increases packet length by 4 bytes; sets the `vlan_tci` Present bit |

### 4.9.15.2 `pop_vlan`

| Attribute | Value |
|---|---|
| **Syntax** | `pop_vlan` or `strip_vlan` (OVS alias) |
| **Category** | 2. Encap/Decap |
| **OpenFlow version** | 1.1 and later |
| **Prerequisites** | The packet has a VLAN tag (`vlan_tci[12]=1` Present bit) |
| **Semantics** | Remove the outermost VLAN header |
| **Side effects** | Decreases packet length by 4 bytes; `vlan_tci` returns to 0 |

### 4.9.15.3 Tagging workflow examples

```shell
# Access port to trunk port: tag the packet with VLAN 100
$ ovs-ofctl add-flow br0 \
    "priority=100,in_port=1,vlan_tci=0,\
     actions=push_vlan:0x8100,mod_vlan_vid:100,output:2"

# Trunk port to access port: untag
$ ovs-ofctl add-flow br0 \
    "priority=100,in_port=2,dl_vlan=100,\
     actions=strip_vlan,output:1"

# Q-in-Q double tagging
$ ovs-ofctl add-flow br0 \
    "priority=100,in_port=3,\
     actions=push_vlan:0x88a8,mod_vlan_vid:500,\
             push_vlan:0x8100,mod_vlan_vid:100,\
             output:4"
```

### 4.9.15.4 Failure scenarios

- **`push_vlan` without setting the VID**: the packet carries a VLAN header with VID=0 (reserved for priority tagging; many switch hardware platforms drop it). Always pair with `mod_vlan_vid` or `set_field`.
- **`pop_vlan` on an untagged packet**: silently skipped (no error); verify with a `vlan_tci=0x1000/0x1000` match beforehand.

## 4.9.16 Category 2, MPLS plus PBB encapsulation

### 4.9.16.1 `push_mpls`, `pop_mpls`

| Attribute | `push_mpls` | `pop_mpls` |
|---|---|---|
| **Syntax** | `push_mpls:<ethertype>` (0x8847 unicast, 0x8848 multicast) | `pop_mpls:<ethertype>` (the inner protocol) |
| **OpenFlow version** | 1.1 and later | 1.1 and later |
| **Semantics** | Insert an MPLS label on top | Remove the outermost MPLS label, exposing the inner one |
| **Side effects** | Packet length plus 4 bytes; `mpls_label`, `tc`, `bos`, and `ttl` must be set separately | Packet length minus 4 bytes |

### 4.9.16.2 MPLS label push example

```shell
# L3 VPN: push MPLS label 100 on top; eth_type changes to 0x8847
$ ovs-ofctl add-flow br0 \
    "priority=100,ip,nw_dst=10.1.0.0/24,\
     actions=push_mpls:0x8847,\
             set_field:100->mpls_label,\
             set_field:1->mpls_bos,\
             set_field:64->mpls_ttl,\
             output:uplink"

# MPLS pop at the PE:
$ ovs-ofctl add-flow br0 \
    "priority=100,mpls,mpls_label=100,\
     actions=pop_mpls:0x0800,\
             output:ce_port"
```

### 4.9.16.3 `push_pbb`, `pop_pbb` (Provider Backbone Bridge)

Similar to MPLS but for 802.1ah PBB. Rarely used outside MEF metro Ethernet. Skipping the deep dive because the hyperscale data centre scope does not need it.

### 4.9.16.4 `encap`, `decap` (generic, OF 1.5 plus NSH)

OpenFlow 1.5 added generic `encap` and `decap` for arbitrary header chains (NSH, GRE, VXLAN-GPE) instead of being hard-coded to VLAN, MPLS, or PBB. Rarely used in OVN; OVN encapsulates Geneve and VXLAN through **tunnel port** configuration rather than through actions.

## 4.9.17 Category 3, `set_field`, the generic field-rewrite action

### 4.9.17.1 Template C

| Attribute | Value |
|---|---|
| **Syntax** | `set_field:<value>[/<mask>]-><field>` or `set_field(<field>=<value>)` |
| **Category** | 3. Field modification |
| **OpenFlow version** | 1.2 and later (OXM TLV required) |
| **Prerequisites** | Field-specific (for example, `eth_type=0x0800` for IP fields) |
| **Semantics** | Generic field rewrite, replacing every `mod_*` per-field action. Supports a mask for partial writes |
| **Parameters** | `value`: the new value (hex, decimal, MAC, IP); `mask`: optional bitmask; `field`: OXM or NXM field name |
| **Side effects** | Modifies the packet header in place; the L3 and L4 checksum is recomputed automatically |

### 4.9.17.2 Common set_field rewrite examples

```shell
# Rewrite the destination MAC (replacement for mod_dl_dst)
$ ovs-ofctl add-flow br0 \
    "priority=100,ip,nw_dst=10.1.0.1,\
     actions=set_field:aa:bb:cc:00:01:01->eth_dst,output:5"

# Rewrite the destination IP (DNAT semantics)
$ ovs-ofctl add-flow br0 \
    "priority=100,tcp,nw_dst=10.1.0.100,tp_dst=80,\
     actions=set_field:10.2.0.10->ipv4_dst,\
             set_field:8080->tcp_dst,\
             output:backend_port"

# Partial bit write with a mask (only the upper 3 MAC bytes, the OUI)
$ ovs-ofctl add-flow br0 \
    "priority=100,in_port=1,\
     actions=set_field:aa:bb:cc:00:00:00/ff:ff:ff:00:00:00->eth_src,\
             output:2"

# DSCP marking
$ ovs-ofctl add-flow br0 \
    "priority=100,ip,tcp,tp_dst=5001,\
     actions=set_field:46->ip_dscp,output:uplink"
```

### 4.9.17.3 `set_field` versus the legacy `mod_*`

| Aspect | `set_field` | `mod_<field>` |
|---|---|---|
| OpenFlow version | 1.2 and later | 1.0 legacy |
| Syntax | Unified | Per-field |
| Mask support | Yes | No |
| New field | Automatically supported | Must be added per new field |

Best practice: **use `set_field` for new code**. Legacy `mod_dl_src/dst`, `mod_nw_src/dst`, `mod_tp_src/dst` still work but do not scale to new fields (tun_metadata, ct_mark, registers, and so on); those fields have no dedicated `mod_*` and must use `set_field` or `load`.

### 4.9.18.1 The `mod_*` list (OF 1.0)

| Action | `set_field` equivalent |
|---|---|
| `mod_dl_src:<mac>` | `set_field:<mac>->eth_src` |
| `mod_dl_dst:<mac>` | `set_field:<mac>->eth_dst` |
| `mod_nw_src:<ip>` | `set_field:<ip>->ipv4_src` |
| `mod_nw_dst:<ip>` | `set_field:<ip>->ipv4_dst` |
| `mod_nw_tos:<dscp>` | `set_field:<tos>->ip_dscp` |
| `mod_nw_ecn:<ecn>` | `set_field:<ecn>->ip_ecn` |
| `mod_nw_ttl:<ttl>` | `set_field:<ttl>->nw_ttl` |
| `mod_tp_src:<port>` | `set_field:<port>->tcp_src` or `udp_src` |

| `mod_vlan_pcp:<pcp>` | `set_field:<pcp>->vlan_pcp` |

### 4.9.18.2 `dec_ttl`, `dec_mpls_ttl`

| Attribute | Value |
|---|---|
| **Syntax** | `dec_ttl`, `dec_mpls_ttl` |
| **Category** | 3. Field modification |
| **OpenFlow version** | 1.0 and later (`dec_ttl`), 1.1 and later (`dec_mpls_ttl`) |
| **Prerequisites** | An IP packet with TTL greater than 0 for `dec_ttl` |
| **Semantics** | Decrement TTL or HL by one. When TTL becomes 0 after decrement, the **packet is sent to the controller via packet_in** (not auto-dropped); the controller decides what to do |
| **Side effects** | Updates TTL; recomputes the IP checksum; on TTL=0 emits OFPT_PACKET_IN with reason `OFPR_INVALID_TTL` |

**Router function using `dec_ttl`:**

```shell
# OVS as a virtual router
$ ovs-ofctl add-flow br0 \
    "priority=100,ip,nw_dst=10.2.0.0/24,\
     actions=dec_ttl,\
             mod_dl_src:aa:bb:cc:00:00:01,\
             mod_dl_dst:aa:bb:cc:00:02:01,\
             output:uplink"

# On a table miss, send an ICMP TTL exceeded through the controller
$ ovs-ofctl add-flow br0 \
    "priority=0,dl_type=0x0800,actions=controller(reason=invalid_ttl)"
```

### 4.9.18.3 `copy_ttl_in`, `copy_ttl_out`

MPLS/IP stacking: when pushing or popping an MPLS label, `copy_ttl_in` copies the outer TTL into the inner header, and `copy_ttl_out` copies it back. Used in MPLS VPN to preserve TTL across the tunnel.

## 4.9.19 Category 3, `move` plus `load`, register manipulation

### 4.9.19.1 `move`, copy field to field

| Attribute | Value |
|---|---|
| **Syntax** | `move:<src_field>[<start>..<end>]-><dst_field>[<start>..<end>]` |
| **Category** | 3. Field modification |
| **OpenFlow version** | NXM extension |
| **Semantics** | Copy bits from the source field to the destination field. The bit range is optional; when omitted the full field is copied |
| **Example** | `move:NXM_OF_ETH_SRC[]->NXM_OF_ETH_DST[]` copies eth_src to eth_dst |

### 4.9.19.2 `load`, copy a constant into a field

| Attribute | Value |
|---|---|
| **Syntax** | `load:<value>-><field>[<start>..<end>]` |
| **Category** | 3. Field modification |
| **OpenFlow version** | NXM extension |
| **Semantics** | Load a constant into a field (or a bit range) |
| **Example** | `load:0x42->NXM_NX_REG0[]` |

### 4.9.19.3 ARP responder use case (links to Part 4.8 §4.8.5)

```shell
$ ovs-ofctl add-flow br-int \
    "priority=200,arp,arp_op=1,arp_tpa=10.1.1.1,\
     actions=move:NXM_OF_ETH_SRC[]->NXM_OF_ETH_DST[],\
             mod_dl_src:aa:bb:cc:dd:ee:ff,\
             load:0x2->NXM_OF_ARP_OP[],\
             move:NXM_NX_ARP_SHA[]->NXM_NX_ARP_THA[],\
             move:NXM_OF_ARP_SPA[]->NXM_OF_ARP_TPA[],\
             load:0xaabbccddeeff->NXM_NX_ARP_SHA[],\
             load:0x0a010101->NXM_OF_ARP_SPA[],\
             IN_PORT"
```

Seven consecutive actions: `move` copies a field, `load` writes a constant, and `IN_PORT` outputs back. This is the classic way to implement an ARP responder in OVS without sending the packet up to the controller.

### 4.9.19.4 Bit-range manipulation

```shell
# Set only bits 0 to 3 of reg0 to 0xa
$ ovs-ofctl add-flow br0 \
    "priority=100,in_port=1,\
     actions=load:0xa->NXM_NX_REG0[0..3],resubmit(,1)"

# Move metadata bits 12 to 15 into reg1 bits 0 to 3
$ ovs-ofctl add-flow br0 \
    "priority=100,actions=move:OXM_OF_METADATA[12..15]->NXM_NX_REG1[0..3]"
```

Bit-range syntax is `[<start>..<end>]` or `[<bit>]`. Used extensively in OVN to split a register into subfields (for example, `reg0[0]` is the ACL hint flag, `reg0[1]` is the committed flag).

## 4.9.20 Category 4, metadata actions

### 4.9.20.1 `write_metadata` (an instruction, not an action)

Strictly this is an **instruction** (§4.9.1.1), not an action. OVS allows it in `actions=` as a shorthand. Syntax:

```
actions=write_metadata:<value>[/<mask>]
```

## 4.9.23 Category 5, `ct()`, conntrack integration

### 4.9.23.1 Template C for `ct()`

| Attribute | Value |
|---|---|
| **Syntax** | `ct(<option>[,<option>...])` |
| **Category** | 5. Firewall/CT |
| **OpenFlow version** | NXM extension (OVS) |
| **Prerequisites** | Kernel datapath with conntrack support (`dpif/show-dp-features: Conntrack eventmask: Yes`) |
| **Semantics** | Submit the packet to the kernel conntrack module. After conntrack processing, the packet recirculates back into the pipeline with `ct_state`, `ct_zone`, `ct_mark`, and `ct_label` populated |
| **Parameters** | `commit`, `zone=N`, `nat`, `force`, `alg=<ftp/tftp/sip/...>`, `exec(<actions>)`, `table=<N>` |
| **Side effects** | Substantial: creates a conntrack entry (commit), NAT rewrite (nat), triggers ALG helpers, recirculates the packet |

### 4.9.23.2 `ct()` options

**`commit`**: create an entry in the conntrack table:

```
actions=ct(commit,zone=1)
# Register this connection in kernel conntrack table zone 1
# Subsequent packets with the same five-tuple will match the conntrack entry
```

**`zone=N`**: conntrack zone ID (16-bit) that isolates the namespace between OVN logical switches. When unspecified, zone 0 is the default.

**`nat`**: apply NAT:

```
# Source NAT (SNAT)
actions=ct(commit,nat(src=10.1.0.1))

# Destination NAT (DNAT)
actions=ct(commit,nat(dst=10.2.0.1:8080))

# NAT with a port range
actions=ct(commit,nat(src=10.1.0.1:30000-40000))

# NAT persistence: reuse the same mapping for the same client
actions=ct(commit,nat(src=10.1.0.1,persistent))
```

**`force`**: re-commit and override the existing entry. Used when the existing connection is expected to be stale.

**`alg=<name>`**: application-layer gateway helper:

| ALG | Protocol |
|---|---|
| `alg=ftp` | FTP control channel (parses PORT and PASV to open the dynamic data channel) |
| `alg=tftp` | TFTP |
| `alg=sip` | SIP (VoIP signalling, opens the RTP media channel) |

**`exec(<actions>)`**: execute the action list **inside the conntrack zone context**. Used to set ct_mark or ct_label on commit:

```
actions=ct(commit,zone=1,exec(set_field:0xab->ct_mark))
```

**`table=N`**: after recirculation, look up table N (default is the current table).

### 4.9.23.3 Typical stateful firewall pattern

```shell
# Step 1: every packet goes through ct (to populate ct_state)
# Table 0, the dispatcher
$ ovs-ofctl add-flow br0 \
    "table=0,priority=100,ip,actions=ct(table=1,zone=1)"

# Table 1, policy based on ct_state
# Allow new outbound and commit
$ ovs-ofctl add-flow br0 \
    "table=1,priority=200,ip,ct_state=+new+trk,\
     actions=ct(commit,zone=1),output:uplink"

# Allow reply (established)
$ ovs-ofctl add-flow br0 \
    "table=1,priority=100,ip,ct_state=+est+trk,actions=output:inside"

# Drop invalid
$ ovs-ofctl add-flow br0 \
    "table=1,priority=100,ip,ct_state=+inv+trk,actions=drop"

# Default drop
$ ovs-ofctl add-flow br0 \
    "table=1,priority=0,actions=drop"
```

Packet flow:
1. The packet arrives at table 0, `ct(table=1)` submits it to kernel conntrack.
2. The kernel populates `ct_state` (new, est, rel, inv) and recirculates.
3. The packet reappears at table 1 with ct_state populated, matching the rule.
4. The rule with `ct(commit)` makes the conntrack entry persist.

### 4.9.23.4 `ct_clear`

| Attribute | Value |
|---|---|
| **Syntax** | `ct_clear` |
| **Semantics** | Clear the conntrack-tracked state for the current packet (reset `ct_state` to unset). The next `ct()` for this packet will re-evaluate |
| **Use case** | When the pipeline wants to re-track the connection (for example, after NAT, to reclassify) |

See Part 9.24 conntrack stateful firewall for the full case study.

## 4.9.24 Category 6, `learn()`, MAC learning through flow installation

### 4.9.24.1 Template C for `learn()`

| Attribute | Value |
|---|---|
| **Syntax** | `learn(<option>[,<option>...])` |
| **Category** | 6. Control/Pipeline |
| **OpenFlow version** | NXM extension |
| **Prerequisites** | None |
| **Semantics** | Install a new flow rule dynamically based on the current packet. This is how OVS implements L2 MAC learning without a controller; the flow is "self-programming" |
| **Parameters** | `table=N`, `priority=P`, `idle_timeout=T`, `hard_timeout=T`, `fin_idle_timeout=T`, `NXM_FIELD[]` match and action templates |
| **Side effects** | Creates a new flow rule in the kernel flow table; the `n_flow_mods` counter increases |

### 4.9.24.2 MAC learning example, the classic OVS pattern

```shell
# Learning rule: when the packet arrives, install the reverse flow
$ ovs-ofctl add-flow br0 \
    "table=0,priority=100,\
     actions=learn(table=1,\
                    idle_timeout=60,\
                    NXM_OF_ETH_DST[]=NXM_OF_ETH_SRC[],\
                    load:NXM_OF_IN_PORT[]->NXM_NX_REG0[]),\
             resubmit(,1)"

# Forwarding based on the learned entries
$ ovs-ofctl add-flow br0 \
    "table=1,priority=50,\
     actions=output:NXM_NX_REG0[]"

# Fallback: flood when there is no learn hit
$ ovs-ofctl add-flow br0 "table=1,priority=0,actions=flood"
```

**Learn-rule semantics:**

- `table=1`: install the new flow into table 1.
- `idle_timeout=60`: the flow expires after 60 seconds of inactivity.
- `NXM_OF_ETH_DST[]=NXM_OF_ETH_SRC[]`: the new match spec, "the destination MAC equals the source MAC of the current packet".
- `load:NXM_OF_IN_PORT[]->NXM_NX_REG0[]`: the action, "load the ingress port into reg0".

When a packet (src=MAC_A, in_port=1) reaches table 0, `learn` creates the flow:

```
table=1,priority=32768,idle_timeout=60,dl_dst=MAC_A,actions=load:1->NXM_NX_REG0[]
```

A subsequent packet with dst=MAC_A matches this flow, sets reg0=1, hits table 1 priority 50, and is sent to output:1.

### 4.9.24.3 `fin_idle_timeout`

A special TCP option: a different timeout when a FIN packet is observed (the connection is closing):

```
actions=learn(...,idle_timeout=300,fin_idle_timeout=10)
```

A normal flow expires after 300 seconds idle, but after a FIN is observed the timeout drops to 10 seconds.

### 4.9.24.4 Risk of `learn()`

- Each packet matching a rule with `learn()` creates a new flow, leading to flow explosion when misconfigured.
- `coverage/show upcall_flow_add` will be very high.
- Installing this on a production bridge requires careful rate limiting.

## 4.9.25 Category 6, `conjunction()`, cross-product compression

### 4.9.25.1 Template C

| Attribute | Value |
|---|---|
| **Syntax** | `conjunction(<id>,<component>/<total>)` |
| **Category** | 6. Control/Pipeline |
| **OpenFlow version** | NXM extension |
| **Prerequisites** | None |
| **Semantics** | Declare the rule as one "component" of a conjunction group `id`. The packet must match `<total>` distinct components of the same `id`; once `conj_id` is set, a different priority rule matches and executes the action |
| **Parameters** | `id`: 32-bit conjunction group ID; `component`: 1 to N, the component index; `total`: the total number of components required |

### 4.9.25.2 The cross-product problem

An ACL rule of the form "source IN {A, B, C} AND dest IN {X, Y, Z} AND port IN {80, 443}" requires **3 times 3 times 2 equals 18 flow rules** when written explicitly.

With conjunction it is **3 plus 3 plus 2 plus 1 equals 9 rules** (3 source components plus 3 dest components plus 2 port components plus 1 final action).

### 4.9.25.3 Full example

```shell
# Source components (conjunction 42, component 1/3)
$ for src in 10.0.1.1 10.0.1.2 10.0.1.3; do
    sudo ovs-ofctl add-flow br0 \
      "table=0,priority=100,ip,nw_src=$src,actions=conjunction(42,1/3)"
  done

# Destination components (42, component 2/3)
$ for dst in 10.0.2.1 10.0.2.2 10.0.2.3; do
    sudo ovs-ofctl add-flow br0 \
      "table=0,priority=100,ip,nw_dst=$dst,actions=conjunction(42,2/3)"
  done

# Port components (42, component 3/3)
$ for port in 80 443; do
    sudo ovs-ofctl add-flow br0 \
      "table=0,priority=100,ip,tcp,tp_dst=$port,actions=conjunction(42,3/3)"
  done

# Final rule matching conj_id (set when all three components match)
$ sudo ovs-ofctl add-flow br0 \
    "table=0,priority=100,conj_id=42,actions=output:5"
```

### 4.9.25.4 OVN Port_Group uses conjunction

OVN ACL on a Logical_Switch `Port_Group` uses conjunction extensively. For example, the policy "port_group WEB (10 ports) to port_group DB (5 ports) allow TCP 3306":

- 10 rules for WEB source (conj component 1/2).
- 5 rules for DB destination (conj component 2/2).
- 1 final rule matching `conj_id`.

Total 16 rules instead of 50 explicit rules. See Part 13.3 (OVN ACL plus Load Balancer) for the OVN context.

## 4.9.26 Category 6, `multipath()`, hash-based path selection

### 4.9.26.1 Template C

| Attribute | Value |
|---|---|
| **Syntax** | `multipath(<fields>,<basis>,<algorithm>,<n_links>,<arg>,<dst>[<start>..<end>])` |
| **Category** | 6. Control/Pipeline |
| **OpenFlow version** | NXM extension |
| **Semantics** | Hash the packet header fields to an output ID from 0 to (n_links - 1) and load it into the destination field. Similar to group type=select but without group-table configuration |
| **Parameters** | `fields`: `eth_src`, `symmetric_l4`, and so on; `algorithm`: `modulo_n`, `hash_threshold`, `hrw`, `iter_hash`; `n_links`: the number of outputs; `dst`: the target field |

### 4.9.26.2 ECMP use case

```shell
# Hash on the five-tuple, modulo 4, load 0 to 3 into reg0
$ ovs-ofctl add-flow br0 \
    "priority=100,ip,actions=multipath(symmetric_l4,0,modulo_n,4,0,NXM_NX_REG0[0..1]),\
     resubmit(,1)"

# Table 1 outputs based on reg0
$ ovs-ofctl add-flow br0 "table=1,reg0=0,actions=output:uplink1"
$ ovs-ofctl add-flow br0 "table=1,reg0=1,actions=output:uplink2"
$ ovs-ofctl add-flow br0 "table=1,reg0=2,actions=output:uplink3"
$ ovs-ofctl add-flow br0 "table=1,reg0=3,actions=output:uplink4"
```

Compared with `group type=select`:
- `multipath` is lightweight and does not require group-table maintenance.
- `group select` has monitoring (group stats) and centralised change management.

### 4.9.26.3 Hash algorithms

| Algorithm | Meaning |
|---|---|
| `modulo_n` | hash mod n_links; simple but rebalances heavily when n_links changes |
| `hash_threshold` | Threshold-based; balances better than modulo |
| `hrw` | Highest Random Weight (HRW); minimal rebalance when n_links changes |
| `iter_hash` | Iterative hash; complex, used for large n_links |

## 4.9.27 Category 6, `bundle()`, grouped action submission

### 4.9.27.1 Template C

| Attribute | Value |
|---|---|
| **Syntax** | `bundle(<fields>,<basis>,<algorithm>,<slave_type>,<slaves>)` |
| **Category** | 6. Control/Pipeline |
| **OpenFlow version** | NXM extension |
| **Semantics** | Select one output from a bundle of ports by hash. Similar to multipath but the output is direct (not loaded into a field) |

### 4.9.27.2 Example

```
actions=bundle(symmetric_l4,0,hrw,ofport,slaves:10,11,12,13)
```

Hash on the five-tuple, pick one of four ports (10 to 13), output the packet.

Compared with `multipath` plus `output:reg`: `bundle` is more compact when downstream rules do not need to differentiate the path.

### 4.9.27.3 `bundle_load()`

A variant that loads the result into a field (like multipath):

```
actions=bundle_load(symmetric_l4,0,hrw,ofport,NXM_NX_REG0[],slaves:10,11)
```

## 4.9.28 Category 5, `check_pkt_larger()`, MTU enforcement

### 4.9.28.1 Template C

| Attribute | Value |
|---|---|
| **Syntax** | `check_pkt_larger(<pkt_len>)-><dst_field>[<bit>]` |
| **Category** | 5. Firewall/CT (control flow based on size) |
| **OpenFlow version** | OVS 2.9 and later |
| **Semantics** | Compare the packet length to `pkt_len`; set the dst field bit to 1 when the packet is larger |
| **Use case** | Path MTU Discovery (PMTUD) support; generate ICMP Packet Too Big in a router function |

### 4.9.28.2 PMTUD use case

```shell
# When the packet is larger than 1400 bytes, set reg0[0] to 1
$ ovs-ofctl add-flow br0 \

# Table 1: when the packet is too large, generate ICMP Too Big
$ ovs-ofctl add-flow br0 \
    "table=1,priority=100,reg0=0x1/0x1,\
     actions=controller(reason=action,userdata=icmp_too_big)"

# Normal forward
$ ovs-ofctl add-flow br0 \
    "table=1,priority=50,actions=output:uplink"
```

## Core takeaways (tier 1, 2, and 3 complete)

First, an action is an instruction executed on the packet (transform, forward); an instruction is a wrapper that controls the pipeline. OpenFlow 1.0 merged the two; 1.1 and later separated them. The OVS `actions=` shortcut defaults to Apply-Actions.

Second, the Output category has 10 foundational actions: `output:<port>` plus the nine reserved-port variants (drop, normal, flood, all, controller, local, in_port, table) plus `group`. Each has its own semantics (whether to include the ingress port, whether to respect STP, whether to respect VLAN).

Third, explicit `drop` is better than implicit (an empty action list) because it is unambiguous. OVS 2.17 and later has a dedicated kernel drop path; verify with `dpif/show-dp-features`.

Fourth, the `group` table provides indirection for four patterns: multicast (all), load balance (select), shared next-hop (indirect), active-backup (fast_failover). A foundational OpenFlow 1.1 and later feature that many deployments have not used enough.

Fifth, `resubmit` is an OVS NXM extension for a modular pipeline and differs from the standard `goto_table`. OVN translation uses it extensively.

Sixth, action-set execution follows the **12 priority levels** strictly, not the order written. Apply-Actions, in contrast, is sequential. Understanding this distinction prevents surprises during debugging.

## 4.9.29 Backfill, eight-attribute anatomy completeness for 12 action gaps

> **Purpose.** A v3.5 audit found 12 actions in REF that did not yet have a full eight-attribute anatomy in the catalog (Template C). This section backfills them and follows the Template C defined in 4.9.1.

### 4.9.29.1 Action `copy_field` (OF 1.5 and later)

**Bucket:** Modification.
**Syntax:** `copy_field(src_offset:src_field, dst_offset:dst_field, n_bits)` or the shorthand `copy_field:SRC->DST` (full-width copy).
**Effect:** Copy `n_bits` bits from `src_field[src_offset:]` to `dst_field[dst_offset:]`. The source field is unchanged.
**Prerequisites:** `src_field` and `dst_field` must be valid.
**OF version introduced:** OF 1.5 and later (action OFPAT_COPY_FIELD type 28).
**OVS version introduced:** OVS 2.7 and later.
**Side effect:** Modifies the destination field.
**Example:**

```
priority=100,ip,actions=copy_field:ipv4_src->reg0,goto_table:5
priority=100,actions=copy_field(0:reg0,0:vlan_vid,16)
```

**Operational notes.** OF 1.0 to 1.4 used the OVS extension `move:SRC[]->DST[]` to copy a field, but `move` is NXM-only and is not portable across vendors. OF 1.5 standardised `copy_field` with the same semantics. OVN can use `copy_field` instead of `move` for portable code. Difference from `set_field` (which writes a literal value): `copy_field` writes a value from another field (dynamic).

### 4.9.29.2 Actions `push_pbb` and `pop_pbb` (OF 1.3 and later)

**Bucket:** L2 Provider Bridge encapsulation.
**Syntax:** `push_pbb:0x88e7` and `pop_pbb`.
**Effect:** Add or remove a 22-byte PBB shim header before the customer Ethernet header.
**Prerequisites:** `pop_pbb` requires `eth_type=0x88e7`.
**OF version introduced:** OF 1.3 and later (OFPAT_PUSH_PBB type 26 plus OFPAT_POP_PBB type 27).
**OVS version introduced:** OVS 2.6 and later.
**Side effect:** `push_pbb` sets the new B-DA equal to the current `eth_dst`, B-SA equal to the current `eth_src`, and I-SID to 0.
**Example:**

```
actions=push_pbb:0x88e7,set_field:0x123456->pbb_isid,output:2
priority=100,eth_type=0x88e7,actions=pop_pbb,goto_table:5
```

**Operational notes.** PBB (IEEE 802.1ah) is common in carrier Ethernet provider deployments. OVS supports PBB primarily for compatibility with carrier hardware switches. Not used in data centre overlays (OVN uses Geneve or VXLAN).

### 4.9.29.3 Action `set_mpls_ttl`

**Bucket:** L2 MPLS modification.
**Syntax:** `set_mpls_ttl:N` (N from 0 to 255).
**Effect:** Override the MPLS TTL field to N.
**Prerequisites:** MPLS (`eth_type=0x8847` or `0x8848`).
**OF version introduced:** OF 1.1 and later (OFPAT_SET_MPLS_TTL type 15).
**OVS version introduced:** OVS 1.11 and later.
**Side effect:** Modifies the MPLS shim-header byte 4 (TTL position).
**Example:** `priority=100,eth_type=0x8847,actions=set_mpls_ttl:64,output:2`

**Operational notes.** When an MPLS edge router pushes a new label, the outer label's TTL by default inherits from the inner label. `set_mpls_ttl` lets the edge router refresh the TTL to a high value before forwarding, avoiding packet drops at a core router as the TTL counts down to zero.

### 4.9.29.4 Action `dec_mpls_ttl`

**Bucket:** L2 MPLS modification.
**Syntax:** `dec_mpls_ttl`.
**Effect:** Decrement the outermost MPLS label TTL by 1. Drop the packet when TTL reaches 0.
**Prerequisites:** MPLS.
**OF version introduced:** OF 1.1 and later (OFPAT_DEC_MPLS_TTL type 16).
**OVS version introduced:** OVS 1.11 and later.
**Side effect:** TTL byte modification plus a potential drop on zero.
**Example:** `priority=100,eth_type=0x8847,actions=dec_mpls_ttl,output:2`

**Operational notes.** Every MPLS transit hop must decrement TTL (the same as IP TTL) to prevent infinite loops.

### 4.9.29.5 Actions `copy_ttl_in` and `copy_ttl_out`

**Bucket:** L2/L3 TTL preservation.
**Syntax:** `copy_ttl_in` and `copy_ttl_out`.
**Effect:**
- `copy_ttl_in`: copy TTL from the outer header (outer MPLS or IP) into the inner header (inner IP). Used on encapsulation pop.
- `copy_ttl_out`: copy TTL from the inner header (IP) into the outer header (MPLS). Used on encapsulation push.
**Prerequisites:** At least two encapsulation layers.
**OF version introduced:** OF 1.1 and later (OFPAT_COPY_TTL_IN type 11, OFPAT_COPY_TTL_OUT type 12).
**OVS version introduced:** OVS 2.5 and later.
**Side effect:** Modifies the TTL byte of the corresponding header.
**Example:**

```
priority=100,eth_type=0x8847,mpls_bos=1,actions=copy_ttl_in,pop_mpls:0x0800,goto_table:5
priority=100,ip,actions=copy_ttl_out,push_mpls:0x8847,set_field:100->mpls_label,output:2
```

**Operational notes.** These two actions resolve the **TTL behaviour** trade-off in MPLS: pipe mode (default, hides the topology) versus uniform mode (`copy_ttl_*` enabled, traceroute sees the full path).

### 4.9.29.6 Action `set_nw_ttl`

**Bucket:** L3 modification.
**Syntax:** `set_nw_ttl:N` (N from 0 to 255).
**Effect:** Override the IPv4 TTL or IPv6 Hop Limit to N.
**Prerequisites:** IPv4 or IPv6.
**OF version introduced:** OF 1.1 and later (OFPAT_SET_NW_TTL type 23).
**OVS version introduced:** OVS 1.10 and later.
**Side effect:** Modifies the IP header TTL or Hop Limit.
**Example:** `priority=100,ip,actions=set_nw_ttl:64,output:2`

**Operational notes.** Unlike `dec_ttl` (decrement by one, the standard router behaviour), `set_nw_ttl:N` overwrites the TTL with N. Used only for special cases such as NAT64 or a gateway re-stamping TTL.

### 4.9.29.7 Action `decap` (NSH detail)

**Bucket:** Encapsulation.
**Syntax:** `decap` or `decap(packet_type=(NS,TYPE))`.
**Effect:** Remove the innermost encapsulation header (NSH, MPLS, GRE, Geneve, and similar) and update the `packet_type` field.
**Prerequisites:** The packet must have an encapsulation layer present.
**OF version introduced:** OVS extension (OF 1.5 and later, packet-type aware).
**OVS version introduced:** OVS 2.8 and later (NSH support).
**Side effect:** Modifies the `packet_type` field. The outer header data is lost.
**Example:**

```
priority=100,nsh,nsh_si=0,actions=decap(),decap(packet_type=(0,0x0800)),output:2
```

**Operational notes.** `decap` is the core action for Service Function Chaining (SFC). The final service function in the chain (SI=0) calls `decap()` twice: once to remove the NSH header, and once to revert to Ethernet for output.

### 4.9.29.8 Action `encap` (NSH detail)

**Bucket:** Encapsulation.
**Syntax:** `encap(NSH)`, `encap(ethernet)`, `encap(MPLS, mpls_label=N)`.
**Effect:** Add an encapsulation header on top of the current packet and update `packet_type`.
**Prerequisites:** The current packet must be compatible with the encap's inner type.
**OF version introduced:** OVS extension.
**OVS version introduced:** OVS 2.8 and later.
**Side effect:** Adds a header and changes `packet_type`.
**Example:**

```
priority=100,ip,actions=encap(nsh(md_type=1,tlv(class=0x1,type=1,len=4,value=0xdead))),
        set_field:0x123->nsh_spi,set_field:255->nsh_si,output:2
```

**Operational notes.** `encap` is the counterpart of `decap`. The SFC entry router emits `encap(nsh)` to wrap traffic into the chain. NSH metadata can use MD Type 1 (fixed length 16 bytes) or Type 2 (variable TLV).

### 4.9.29.9 Action `controller` with userdata (OVS 2.6 and later, continuation-based)

**Bucket:** Control.
**Syntax:** `controller(reason=action,userdata=HEX)` or `controller(reason=action,userdata=HEX,pause)`.
**Effect:** Send the packet to the controller with custom metadata; optionally pause the pipeline and resume after the controller responds.
**Prerequisites:** An OpenFlow channel connected to the controller.
**OF version introduced:** OF 1.0 and later (basic), OVS 2.6 and later (continuation extension).
**OVS version introduced:** OVS 2.6 and later.
**Side effect:** Trigger PACKET_IN message + optional pipeline pause.
**Example:**

```
priority=100,actions=controller(reason=action,userdata=0xcafe1234)
priority=100,actions=controller(reason=action,userdata=0xdead,pause)
```

**Operational notes.** The continuation-based extension lets the controller respond with a PACKET_OUT carrying a "continue" action to resume the paused pipeline. OVN does not use this pattern; mostly `controller` is used as an ARP responder fallback.

### 4.9.29.10 Action `note` (debug marker)

**Bucket:** Annotation.
**Syntax:** `note:HEXSTRING` (inline hex bytes).
**Effect:** Semantically a no-op. Embeds bytes into the flow definition for debug or audit.
**Prerequisites:** None.
**OF version introduced:** OVS extension (NXAST_NOTE).
**OVS version introduced:** OVS 1.0 and later.
**Side effect:** None. Pure debug marker.
**Example:** `priority=100,in_port=1,actions=note:01.02.03.04.deadbeef,output:2`

**Operational notes.** A technique for embedding bytes into a flow rule for post-mortem debugging. When `dump-flows` shows the rule, the note hex is visible and the operator decodes it back into metadata.

### 4.9.29.11 Action `sample` (OVS 2.5 and later)

**Bucket:** Monitoring.
**Syntax:** `sample(probability=N,collector_set_id=ID,obs_domain_id=ID,obs_point_id=ID)`.
**Effect:** Per-packet probability sampling. With probability N out of 65535, emit an IPFIX record.
**Prerequisites:** The IPFIX `collector_set` is configured through OVSDB.
**OF version introduced:** OVS extension (NXAST_SAMPLE).
**OVS version introduced:** OVS 2.5 and later.
**Side effect:** Triggers IPFIX export (no packet modification).
**Example:**

```
priority=100,in_port=1,actions=sample(probability=655,collector_set_id=1,obs_domain_id=42,obs_point_id=100),output:2
```

**Operational notes.** Built-in flow sampling for IPFIX or sFlow telemetry. The probability granularity is 1 in 65,535 (1 percent equals about 655). Use case: NetFlow collection for traffic analysis. Difference from a sFlow agent (which samples every packet): the `sample` action is embedded in the flow rule (selective per flow).

### 4.9.29.12 Action `conjunction` (OVS 2.4 and later), full anatomy

**Bucket:** Matching.
**Syntax:** `conjunction(id, k/n)` where k is the clause index (1-based) and n is the total clause count.
**Effect:** Implements an n-way conjunctive (AND) match through flow expansion.
**Prerequisites:** None.
**OF version introduced:** OVS extension (NXAST_CONJUNCTION).
**OVS version introduced:** OVS 2.4 and later.
**Side effect:** Triggers a downstream conjunctive match (a rule with `conj_id=id`).
**Example:**

```
priority=100,ip,nw_src=10.0.0.0/8,actions=conjunction(1,1/2)
priority=100,tcp,tp_dst=80,actions=conjunction(1,2/2)
priority=100,tcp,tp_dst=443,actions=conjunction(1,2/2)
priority=200,conj_id=1,actions=output:2
```

**Operational notes.** The problem: an ACL "src in {N IPs} AND dst in {M ports}" expanded as a Cartesian product equals N times M rules. Conjunction reduces this to N plus M rules plus one verdict rule.

OVN uses conjunction heavily for ACLs that reference Address_Set plus Port_Group with many IPs or ports. An ACL "from-lport: ip4.src in $set_a (100 IPs) && ip4.dst in $set_b (50 IPs)" equals 100 times 50 equals 5,000 rules as a Cartesian product. With conjunction it becomes 100 plus 50 plus 1 equals 151 rules, a 33-fold reduction.

## 4.9.31 Backfill, NXM Nicira stack plus timeout actions

> **Context.** A v3.6 audit found three Nicira extension actions in REF that the curriculum had not yet indexed: `fin_timeout`, `push:src`, `pop:dst`. All three are OVS 1.11 and later, and all three remain commonly used in OpenFlow rules with register manipulation. This section adds the eight-attribute Template C anatomy for each plus a use case and cross-links.
>
> **Why these three actions matter even though they are not tier 1.** `fin_timeout` is the cheapest way to clean up a TCP flow that has FINed without paying the conntrack overhead. The pair `push:` and `pop:` is the only stack semantics in NXM; every "save-modify-restore" pattern for a temporary field (such as swapping `in_port` to re-execute the pipeline with a fake input) depends on it. Skipping these three actions skips an entire idea class (per-packet stack) that is fundamentally different from the rest (write-once).

### 4.9.31.1 Action `fin_timeout` (NXAST_FIN_TIMEOUT, OVS 1.11 and later)

**Bucket:** Connection tracking (timeout shrink).
**Syntax:** `fin_timeout(idle_timeout=N, hard_timeout=M)` with N and M in seconds. The parameters are optional; either may be passed alone.
**Effect:** When a packet matches this flow entry with the TCP FIN or RST flag set, OVS shrinks the entry's `idle_timeout` and `hard_timeout` to N and M seconds respectively (only shorter, never longer). Goal: clean up the flow faster after the TCP connection has closed, instead of waiting for the default idle timeout (typically 300 seconds).
**Prerequisites:** The flow entry must match TCP packets (`tcp` or `nw_proto=6`). The action only matters for flows with timeout > 0; permanent flows (timeout=0) are unaffected.
**OF version introduced:** OVS extension (NXAST_FIN_TIMEOUT, not in standard OpenFlow).
**OVS version introduced:** OVS 1.11 and later.
**Side effect:** Updates the flow entry's metadata (through `ofproto-dpif`); does not generate any outgoing packet. No interaction with conntrack zones; this is pure OpenFlow flow lifecycle.
**Example:**

```
priority=200,tcp,tcp_flags=+fin,actions=fin_timeout(idle_timeout=5,hard_timeout=30),resubmit(,1)
priority=200,tcp,tcp_flags=+rst,actions=fin_timeout(idle_timeout=1,hard_timeout=5),resubmit(,1)
```

**Operational notes.** Long-lived TCP flows in a data centre can keep flow entries in the table for hours because the default idle timeout is high. After the peer sends FIN or RST, the connection is actually dead but the entry still occupies a slot. `fin_timeout` lets the operator declare "when a FIN is observed, age out aggressively in another 5 seconds". Compared with conntrack `ct(commit, exec(fin_timeout=5))`: the standalone `fin_timeout` action does not need a conntrack state machine and is simpler for a stateless ACL with a little awareness of connection close. Difference from `learn(... fin_idle_timeout=...)`: `learn` creates a new flow entry in another table, whereas `fin_timeout` operates on the current flow entry. Cross-link: 9.24 §9.24.x for the conntrack-based approach; 4.7 §4.7.X for the learn-from-flow approach.

### 4.9.31.2 Action `push:src` (NXAST_STACK_PUSH, OVS 1.11 and later)

**Bucket:** Field stack (LIFO save).
**Syntax:** `push:FIELD` or `push:FIELD[bit_lo..bit_hi]` where FIELD is an NXM or OXM identifier (`NXM_OF_IN_PORT[]`, `NXM_NX_REG0[0..15]`, `eth_src[]`, and so on).
**Effect:** Push the current value of `FIELD` (or a bit slice) onto the OVS internal per-packet stack. The stack lives in the flow execution context; lifetime is bounded to one packet being processed.
**Prerequisites:** The stack has a maximum depth (lib/ofp-actions.c limits to 64 entries per packet). A push above the limit produces an action error.
**OF version introduced:** OVS extension (NXAST_STACK_PUSH, not in standard OpenFlow).
**OVS version introduced:** OVS 1.11 and later.
**Side effect:** The stack state changes (one entry pushed); the source field keeps its value. The stack resets between packets.
**Example:**

```
table=0,priority=100,in_port=1,actions=push:NXM_OF_IN_PORT[],load:0->NXM_OF_IN_PORT[],resubmit(,1),pop:NXM_OF_IN_PORT[],output:2
```

**Operational notes.** A typical "save-modify-restore" pattern: save `in_port` onto the stack, set `in_port = 0` (reset), resubmit to another table as if the packet had arrived on port 0, then restore `in_port` to its original value before output. Useful when a multi-table pipeline simulates "the packet came from a different port" for testing or to bypass an `in_port` match downstream. Compared with `move:` (copy the value to a temporary register and copy back): `push:` and `pop:` do not consume a register slot and do not collide with other register usage in the pipeline. Compared with `set_field:`, `push:` and `pop:` preserve the original value through the stack rather than through a manual copy.

### 4.9.31.3 Action `pop:dst` (NXAST_STACK_POP, OVS 1.11 and later)

**Bucket:** Field stack (LIFO restore).
**Syntax:** `pop:FIELD` or `pop:FIELD[bit_lo..bit_hi]`.
**Effect:** Take the entry at the top of the stack, assign its value to `FIELD` (or a bit slice), then remove the entry. The popped entry's bit width must match the destination field's bit width, otherwise an action error.
**Prerequisites:** The stack must have at least one entry. Popping an empty stack produces an action error (the packet is dropped).
**OF version introduced:** OVS extension (NXAST_STACK_POP).
**OVS version introduced:** OVS 1.11 and later.
**Side effect:** The stack state changes (one entry popped); the destination `FIELD` is overwritten.
**Example:**

```
priority=100,tcp,actions=push:NXM_NX_REG0[],load:0xdeadbeef->NXM_NX_REG0[],note:01.02.03,pop:NXM_NX_REG0[],output:2
```

**Operational notes.** Use case: the pipeline needs a temporary register for an intermediate computation (such as a hash) but does not want to destroy the original reg0 value that downstream rules still depend on. Push reg0, use reg0 for the computation, pop reg0 before output. A pop without a paired push (mismatched stack) is a typical programming error; OVS ofproto/trace reports `stack underflow`. Anti-pattern: using `pop:` instead of `move:` when there is no preceding push. Cross-link: 9.25 §9.25.X `ofproto/trace` for debugging stack-related anomalies.

## 4.9.32 Deeper treatment of the `output` action

> **Purpose.** The `output` action is already covered substantively in §4.9.1 Category 1. This section adds depth on importance, version differences, real-world incident, common failures, and comparison with other systems.

### Importance

`output` is the **absolute cornerstone** of the OpenFlow action set. Without `output`, the packet never leaves the switch; the flow rule only classifies and modifies, and the packet eventually drops. Every forwarding rule must end with `output` (or with a `resubmit`/`goto_table` chain that ends in an `output`). This action is on the same foundational tier as megaflow and ct_state.

### Version differences

| Version | Behaviour |
|---------|-----------|
| OF 1.0 | Basic `output:port_no` |
| OF 1.0 | Reserved ports: OFPP_FLOOD, OFPP_ALL, OFPP_CONTROLLER, OFPP_LOCAL, OFPP_NORMAL, OFPP_IN_PORT |
| OF 1.1 | Multi-table awareness (output emits from the current table or from accumulated state) |
| OF 1.3 | Group-table reference (output to a group instead of a port) |
| OF 1.5 | Egress-table support (post-output egress processing) |
| OVS extension | `output:NXM` (dynamic port from a register), `output:CONTROLLER:max_len` |

### Real-world incident

Production cluster, 2024-Q2: TCP traffic asymmetric routing, with intermittent return-path drops. Investigation: ofproto/trace showed the packet matched correctly, but `actions=output:5` was evaluated when port 5 was transiently down. OVS does not auto-reroute on a port-down event; it silently drops.

**Root cause:** static `output:port` without a group-bucket fallback.

**Fix:** migrate to a group table with a fast-failover bucket. `actions=group:1` where group 1 has two buckets (port 5 primary, port 6 backup). When port 5 goes down, the group automatically shifts to port 6.

### Common failures

- Mode A: output to a non-existent port produces a silent drop. Diagnostic: ofproto/trace shows the port number is invalid. Recovery: validate the port_no before installing the flow.
- Mode B: output to OFPP_IN_PORT can cause an infinite loop when the in_port is also the output target. Recovery: choose an explicit different output port.
- Mode C: output to the NORMAL port when L2 learning is incomplete causes a flood instead. Recovery: prime the MAC table.
- Mode D: output to a physical port with an MTU mismatch causes frame fragmentation failure. Recovery: configure a consistent MTU.

### Comparison with other systems

- **Cisco IOS interface forwarding**: `interface GigabitEthernet0/1` is a similar concept at a lower level. OF output is a platform-neutral abstraction.
- **Linux bridge `bridge fdb add`**: MAC-table-based forwarding. OF output is rule-based per flow and more flexible.
- **DPDK rte_eth_tx_burst**: transmits a packet to a physical port. OF output is a logical abstraction; rte_eth is direct hardware.
- **VPP interface output**: vector packet processing with a similar concept. Difference: VPP optimises batch dispatch; OVS is per-packet.

## 4.9.33 In-depth treatment of mid-tier packet-modification actions

This section is dedicated to five packet-modification actions that appear in nearly every production flow rule: `drop` (terminate processing), `set_field` (general field overwrite), `mod_dl_src` plus `mod_dl_dst` (legacy OF 1.0 MAC overwrites), and `dec_ttl` (decrement IP header TTL). Each action has its own subsection below with a 20-axis treatment under natural English headings: concept, history, position in the architecture, role, why it exists, problems solved, importance, mechanism, wire encoding, applied examples, how an engineer masters operations, classification, source-code anchor, failure scenarios, synthetic exercise, and comparison with other systems.

These five actions belong to the "category 3 field modification" group in the `ovs-actions(7)` taxonomy, except for `drop`, which belongs to category 1 Output (terminate). They are grouped together because all five either modify the packet header in place or terminate the pipeline; engineers commonly write them adjacent to one another in a single flow, for example a router rule `actions=dec_ttl,mod_dl_src:...,mod_dl_dst:...,output:N`.

### 4.9.33.1 `drop`, terminate packet processing without output or modification

**Concept.** `drop` is a terminator action: the matched packet is discarded after the rule increments the `n_packets` and `n_bytes` counters. The original OpenFlow spec did not have a dedicated `drop` action; the convention was "an empty action list equals an implicit drop". OVS added explicit `drop` syntax to make the rule clearer to the reader; OF 1.5 spec §5.10 standardised explicit `drop` as required behaviour. Action type: **terminator**, cannot be paired with another action in the same action list.

**History and context.** OpenFlow 1.0 (2009): an empty action list equals drop. OVS from its first release (2012 onward) accepted the `drop` keyword in the parser but internally translated it to an empty action list. OpenFlow 1.5 (2014): explicit `drop` became the standard to make rules clearer (section 5.10 "Action Semantics"). OVS 2.17 (2022): added `OVS_ACTION_ATTR_DROP` to the datapath kernel netlink protocol, giving the kernel a dedicated fast path for drop instead of traversing an empty action list.

**Position in the architecture.** At the OpenFlow protocol layer, `drop` has no dedicated raw enum entry in `enum ofp_raw_action_type` (verified in `lib/ofp-actions.c` on branch v3.3.0 of OVS; the parser at line 9431 with `else if (!strcasecmp(key, "drop"))` only sets a flag `drop = true`). At the datapath kernel layer, the netlink action `OVS_ACTION_ATTR_DROP` (constant 22, OVS 2.17 and later) is a dedicated attribute. At the userspace datapath layer, this maps to a no-op execute path.

**Role.** Primary role: a deny rule in an ACL (drop traffic that violates policy). Secondary role: terminate after a check_pkt_larger fail, terminate when conntrack marks the packet `INVALID`, terminate in a table-miss when the operator wants to be explicit instead of relying on default behaviour. Blast radius: per-rule, does not propagate to other actions (drop is the only action on its list).

**Why it exists.** A rule with an empty action list confuses readers; you cannot tell whether the author intended to drop or simply forgot to fill the action. The explicit `drop` keyword improves readability. After OVS 2.17 added the dedicated datapath action, a second reason emerged: performance. The drop fast path skips the empty action-list traversal and avoids allocating format strings for logs; in a DDoS scenario, the switch can drop millions of packets per second and every saved cycle matters.

**Problems it solves.** Specifically: (a) make the rule clearer for an operator review, and (b) increase datapath efficiency when the drop ratio is high. Approach: the parser recognises the `drop` token and the kernel datapath has a dedicated attribute. Scope: every deny flow rule.

**Importance.** Tier 2 medium, but extremely high frequency in production rules. Every ACL deny rule, every anti-spoofing rule, every conntrack INVALID handler uses `drop`. Engineer must-know: distinguish explicit `drop` from implicit (empty action list) and implicit (table-miss with no default rule), and remember that `drop` cannot be combined with another action (the parser rejects with `"drop" must not be accompanied by any other action`).

**How it works.** When the parser encounters the `drop` keyword, it sets the local flag `drop = true` without emitting an ofpact entry into the ofpacts buffer (`lib/ofp-actions.c` line 9431, verified). After parsing the entire action string, the parser checks: if `drop && pp->ofpacts->size > 0`, return an error string. When only `drop` is present, the ofpacts buffer remains empty, equivalent to an empty action list. Downstream translation (xlate.c) sees the empty action list and emits `OVS_ACTION_ATTR_DROP` to the datapath (kernel 2.17 and later) or emits no action at all (older kernels). The kernel datapath sees the DROP attribute, frees the skb early, and returns.

**Wire encoding.** No dedicated wire format for `drop` at the OpenFlow protocol level; an empty action list (length=0) is the implicit wire encoding. The datapath kernel attribute `OVS_ACTION_ATTR_DROP` is an nlattr with type=22 and length=4 bytes (header only, no payload). This is an extension, not part of the OpenFlow spec.

**Applied examples.**

```
priority=100,ip,nw_dst=10.0.0.99,actions=drop                    # explicit drop
priority=100,ip,nw_dst=10.0.0.99,actions=                        # implicit (empty)
priority=10,ct_state=+inv+trk,actions=drop                       # drop INVALID conntrack
priority=200,arp,arp_op=1,arp_spa=0.0.0.0,actions=drop           # drop bogus ARP
priority=0,actions=drop                                          # explicit table-miss
```

**How an engineer masters operations.** Four skills: (1) **Read**: distinguish a rule with `drop` (deny) from a rule with empty `actions=` (same semantics but easily misread as unfinished). (2) **Edit**: use explicit `drop` for every new rule; gain readability with no cost. (3) **Observe**: `ovs-ofctl dump-flows br-int | grep -E 'actions=(drop|$)'` counts the active deny rules. (4) **Performance check**: `ovs-appctl dpif/show-dp-features br0 | grep -i drop` showing `Explicit Drop action: Yes` confirms the kernel (2.17 and later) supports fast drop.

**Classification.** A terminator action (ends the pipeline). Property: cannot be paired with another action. Same class: implicit drop (empty action list); `controller(reason=...)` is not a terminator because it can be combined with other actions. Different from `drop` in conntrack: `ct(commit)` does not drop even when `ct_state` is `INVALID`; the operator must add `drop` explicitly.

**Source code.** `lib/ofp-actions.c` function anchor for the parser: the branch `else if (!strcasecmp(key, "drop"))` at line 9431 sets `drop = true`; the post-loop check `if (drop && pp->ofpacts->size)` returns an error. Datapath attribute: `include/uapi/linux/openvswitch.h` enum `ovs_action_attr` has `OVS_ACTION_ATTR_DROP = 22` (kernel 5.17 and later, OVS module 2.17 and later). Branch baseline: OVS v2.17.9.

**Failure scenarios.** (a) Writing `actions=drop,output:5` is rejected by the parser with `"drop" must not be accompanied by any other action or instruction`. Fix: split into two rules with distinct priorities. (b) A rule with an empty action list that did not intend to drop but forgot to fill the action behaves like `drop`, easy to debug-trap if you forget the "empty equals drop" rule. Fix: always use explicit `drop` so the intent is clear. (c) The `n_packets` counter on the drop rule rises quickly but traffic still passes; another higher-priority rule matches first; verify with `ofproto/trace`.

**Synthetic exercise.** Mininet, one OVS bridge, two hosts: install rule `priority=100,ip,nw_dst=h2,actions=drop` plus a fallback rule `priority=0,actions=normal`. Verify `h1 ping h2` fails and the drop-rule counter rises. Then remove the drop rule; the ping passes.

**Observation and verification.** `ovs-ofctl dump-flows br0 -O OpenFlow15` displays the rule with the explicit `drop` action (OF 1.5 and later format). `ovs-appctl ofproto/trace br0 'in_port=1,ip,nw_dst=10.0.0.99'` shows the line `Datapath actions: drop`, confirming that the kernel will free the skb. `ovs-appctl coverage/show | grep dropped` counts cumulative dropped packets.

**Comparison with other systems.** Linux netfilter `iptables -j DROP` is semantically equivalent: terminate the rule and free the packet. Difference: netfilter also has `REJECT` (sends an ICMP unreachable response); OpenFlow has no built-in equivalent (must use `controller(reason=...)` plus a controller emitting the ICMP separately). Cisco IOS ACL `deny` is equivalent to implicit drop. P4 `drop()` extern function has the same semantics. NSX-T DFW rule action `Drop` is the same concept.

### 4.9.33.2 `set_field`, the standard generic field-rewrite action of OF 1.2 and later

**Concept.** `set_field` is an action that overwrites a new value into any field of the packet header (or metadata, or register), with optional bitmask support for partial writes. This is the standard replacement for the entire legacy `mod_*` family of OF 1.0: `mod_dl_src`, `mod_dl_dst`, `mod_nw_src`, `mod_nw_dst`, `mod_tp_src`, `mod_tp_dst`, `mod_nw_ttl`, `mod_vlan_vid`, `mod_vlan_pcp`, and so on; all are folded into a single action with unified syntax. Action type: **generic field rewrite**, not a terminator.

**History and context.** OpenFlow 1.0 (2009) defined about 10 dedicated `OFPAT10_SET_*` actions, one per field. The "one action per field" design did not scale: every time the spec added a new field (MPLS label, IPv6 flow label, conntrack metadata, register, tunnel metadata), a new action had to be defined. OpenFlow 1.2 (2011) refactored: a single action `OFPAT12_SET_FIELD` (OXM-based) was added, allowing any field with an OXM identifier to be set. OVS 2.0 (2014) officially exposed the `set_field` syntax to `ovs-ofctl`. OF 1.5 (2014) removed the redundant `OFPAT_REG_LOAD`, keeping only `OFPAT_SET_FIELD` (verify `lib/ofp-actions.c` lines 3164-3187, function `encode_SET_FIELD`).

**Position in the architecture.** At the OpenFlow protocol layer: `OFPAT12_SET_FIELD` is action type 25 (OF 1.2); the action header has `type` (2 bytes) plus `length` (2 bytes) plus an OXM TLV inline. At the OVS internal ofpact layer: `struct ofpact_set_field` (`include/openvswitch/ofp-actions.h` line 543) holds a pointer to an `mf_field` descriptor plus a value and mask buffer. At the datapath kernel layer: translates to `OVS_ACTION_ATTR_SET` or `OVS_ACTION_ATTR_SET_MASKED`, depending on whether a mask is present.

**Role.** Primary role: rewrite header bytes (NAT, MAC redirect, VLAN tag, DSCP marking, tunnel VNI). Secondary role: write metadata or register values (set_field:0x42->reg0, set_field:1->ct_mark) as intermediate state for the pipeline. Blast radius: per-packet; the effect can propagate through recirc when the field is persistent metadata.

**Why it exists.** The OF 1.0 `mod_*` family does not scale to new fields. Every extension (NXM register, conntrack mark, tunnel metadata) would otherwise require adding `mod_reg0`, `mod_reg1`, ..., `mod_ct_mark`, `mod_tun_id`, and so on, a combinatorial explosion. `set_field` with an inline OXM TLV allows dynamic dispatch by `mf_field` at runtime.

**Problems it solves.** Specifically: (a) extensibility for new fields without changing the protocol structure, (b) partial writes via bitmask support (for example, change only the three OUI bytes of a MAC and keep the remaining three), (c) unified syntax that is easier to learn. Approach: the action format embeds an OXM TLV inline, and the `mf_field` descriptor handles dispatch. Scope: every field that has an OXM identifier, including every standard field and every NXM extension.

**Importance.** Tier 2 medium with high frequency. Engineer must-know: the syntax `set_field:<value>[/<mask>]-><field>`, the mask semantics (a 1-bit overwrites, a 0-bit preserves), and when to use `set_field` instead of legacy `mod_*` (always pick `set_field` for new code; best practice §4.9.17.3).

**How it works.** The parser `lib/ofp-actions.c` function `parse_SET_FIELD` splits the argument `<value>/<mask>-><field>` into three parts via the helper `set_field_split_str` (line ~3210). Resolve `<field>` to a `struct mf_field *` via `mf_from_name`. Validate the value width matches the field width. Build a `struct ofpact_set_field` with value plus mask buffer. On encode (`encode_SET_FIELD` line 3164): on OF 1.5 and later use `set_field_to_set_field`; with an all-ones mask on OF 1.2-1.4 use `set_field_to_set_field` (entire-field write); with a partial mask use `set_field_to_nxast` (NXAST_RAW_REG_LOAD format). Datapath kernel: performs the `skb` byte rewrite plus recomputes the L3 or L4 checksum for the corresponding field.

**Wire encoding.** OF 1.2-1.4 format: action header `type=25` (SET_FIELD) plus `length` (variable, padded to 8 bytes) plus an inline OXM TLV. OXM TLV: `oxm_class` (2 bytes), `oxm_field|hasmask` (1 byte), `oxm_length` (1 byte), value (plus mask when hasmask=1), padding. OF 1.5 and later keeps the same structure with different semantics: the redundant `OFPAT_REG_LOAD` is removed. NXAST format (legacy fallback): `vendor=0x2320` (Nicira) plus `subtype=NXAST_REG_LOAD` plus an offset and length bit range.

**Applied examples.**

```
actions=set_field:aa:bb:cc:dd:ee:ff->eth_dst                             # full MAC overwrite
actions=set_field:aa:bb:cc:00:00:00/ff:ff:ff:00:00:00->eth_src           # OUI partial write
actions=set_field:10.2.0.10->ipv4_dst,set_field:8080->tcp_dst            # DNAT-like
actions=set_field:46->ip_dscp                                            # DSCP EF marking
actions=set_field:0x42->ct_mark                                          # set conntrack mark
actions=set_field:0x100->tun_id                                          # VXLAN VNI
actions=set_field:0x1->reg0[0]                                           # bit 0 of reg0
```

**How an engineer masters operations.** Four skills: (1) **Read**: distinguish a full-field write (`set_field:V->F`) from a partial-bit write (`set_field:V/M->F`). (2) **Edit**: prefer `set_field` over legacy `mod_*` for new code; gain forward compatibility plus uniform syntax. (3) **Observe**: `ovs-ofctl dump-flows br0 -O OpenFlow15` shows the rule with the full `set_field` action; OF 1.0 mode shows the legacy `mod_*` form. (4) **Debug**: the `ofproto/trace` line `Output: drop / set(eth(dst=...))` confirms the datapath performs the byte rewrite.

**Classification.** A field-rewrite action. Properties: stateful (modifies the packet in place), checksum-aware (auto-recomputes the IP/TCP/UDP checksum when modifying an L3 or L4 field), not a terminator. Same class: legacy `mod_*` (a subset of the semantics), `move` and `load` (NXM register manipulation, similar capability but restricted to registers and metadata), `dec_ttl` (a specialised rewrite for TTL).

**Source code.** `lib/ofp-actions.c` function anchors: `encode_SET_FIELD` line 3164 (encode for OF 1.2-1.5), `set_field_split_str` line ~3210 (parse splitter), `parse_SET_FIELD`. Header `include/openvswitch/ofp-actions.h` line 543 `struct ofpact_set_field`. The `mf_field` metadata in `lib/meta-flow.c` enumerates 150 plus fields. Branch baseline: OVS v2.17.9.

**Failure scenarios.** (a) `set_field:0x100->reg0` when reg0 is 32-bit: the parser accepts (0x100 fits) but `set_field:0xffffffffffff->reg0` is rejected with "value larger than field". (b) `set_field:46->ip_dscp` without the prerequisite `eth_type=0x0800` causes flow add to be rejected with `OFPBAC_MATCH_INCONSISTENT`; fix: add `ip` to the match. (c) `set_field` with a mask on a field that does not support masked writes (for example `eth_type`) returns "field doesn't support masking".

**Synthetic exercise.** Mininet, two hosts h1 (10.0.0.1) and h2 (10.0.0.2). Install rule `priority=100,in_port=1,ip,nw_dst=10.0.0.2,actions=set_field:00:11:22:33:44:55->eth_dst,output:2` to rewrite the MAC of h2 to a fake value before forwarding. `tcpdump -i h2-eth0 -n -e` confirms the packet arriving on h2 has dst MAC `00:11:22:33:44:55` instead of the real MAC.

**Observation and verification.** `ovs-ofctl dump-flows br0 -O OpenFlow15 | grep set_field` counts the rules using it. `ovs-appctl coverage/show | grep set_field` counts the executions (a counter cumulative since vswitchd started). `ofproto/trace` shows the normalised form `set(eth(dst=aa:bb:..))` on the Datapath actions line.

**Comparison with other systems.** Linux `tc filter ... action pedit munge offset N u32 set V` is the same concept (a byte rewrite at a specific offset) but at a lower level; the engineer must compute the offset manually rather than use a field name. Cisco IOS QoS `set dscp ef` is equivalent to `set_field:46->ip_dscp` but restricted to QoS fields. P4 `hdr.ipv4.dst_addr = 0x0a020a0a`, a direct assignment in a control block, is the P4 way to expose the same idea.

### 4.9.33.3 `mod_dl_src`, the legacy OF 1.0 ethernet source MAC overwrite

**Concept.** `mod_dl_src` is a legacy OF 1.0 action that overwrites the 6-byte ethernet source MAC of the packet. Semantically equivalent to `set_field:<mac>->eth_src` from OF 1.2 onward. OVS keeps `mod_dl_src` for backward compatibility; many older production rules and many ovn-northd-generated flows still use this syntax. Action type: **specialised field rewrite**, OF 1.0 raw type `OFPAT10_SET_DL_SRC`.

**History and context.** OpenFlow 1.0 (2009) defined `OFPAT_SET_DL_SRC` as action type 4 with a 6-byte MAC payload (plus 6-byte padding). OVS supported it from the first release. OpenFlow 1.2 (2011) introduced the generic `set_field` and started gradual deprecation. OVS still parses the `mod_dl_src` syntax and emits the OF 1.0 wire format when the bridge runs in OF 1.0 mode; in OF 1.2 and later it encodes to `set_field`. ovn-northd (2018 onward) still uses the `mod_dl_src` syntax in many scenarios such as the distributed-router gateway MAC rewrite.

**Position in the architecture.** At the OpenFlow protocol layer: `OFPAT10_SET_DL_SRC` raw type for OF 1.0; OF 1.2 and later encode to `OFPAT_SET_FIELD` with MFF_ETH_SRC. At the OVS internal layer: both `mod_dl_src` and `set_field:V->eth_src` produce the same `struct ofpact_mac` (`include/openvswitch/ofp-actions.h` line 461). Verify: line 461 comment `OFPACT_SET_ETH_SRC, OFPACT_SET_ETH_DST. Used for OFPAT10_SET_DL_SRC, OFPAT10_SET_DL_DST.`.

**Role.** Primary role: rewrite the source MAC in an L3 router scenario (OVS acts as a router; when a packet is forwarded between subnets, the source MAC must change to the router output interface's MAC). Secondary role: ARP responder pattern (rewrite the source MAC to the gateway's virtual MAC). Blast radius: per-packet.

**Why it exists.** L3 routing on OVS requires MAC rewriting: when a packet moves from subnet A to subnet B, the inbound source MAC must change to the router output interface's MAC (because the receiving host on subnet B only accepts packets whose source MAC is its own router's MAC). Before OpenFlow, switch ASICs handled this in the L3 lookup engine; OpenFlow exposes it as a programmable action.

**Problems it solves.** Specifically: allows OVS to play the role of an L3 router purely in software through flow rules. Approach: the action `mod_dl_src` rewrites the source MAC, `mod_dl_dst` rewrites the destination MAC, and `dec_ttl` decrements the TTL; together these three form the L3 forward primitive. Scope: every software L3 forwarding scenario (OVN logical router, Mininet L3 test).

**Importance.** Tier 2 medium. In OVN flows on br-int, `mod_dl_src` plus `mod_dl_dst` appear at every distributed-router pipeline rewrite stage (LR pipeline tables 11 and 12). Engineer must-know: the 6-byte MAC notation `aa:bb:cc:dd:ee:ff`, and the fact that this action is OF 1.0 raw but is parsed and dumped by OVS even in OF 1.5 dump output.

**How it works.** The parser in `lib/ofp-actions.c` function `parse_SET_ETH_SRC` (line 1976) calls `str_to_mac` to parse the string into a 6-byte `struct eth_addr` and stores it into `ofpact_put_SET_ETH_SRC(pp->ofpacts)->mac`. The formatter `format_SET_ETH_SRC` (line 1988) prints `mod_dl_src:%s` when dumping the rule (verify lines 1991-1992: `ds_put_format(fp->s, "%smod_dl_src:%s"ETH_ADDR_FMT, colors.param, colors.end, ETH_ADDR_ARGS(a->mac));`). The encoder `encode_SET_ETH_SRC` (around line 1957) calls the helper `encode_SET_ETH_addr` with raw type `OFPAT_RAW_SET_DL_SRC` and field `MFF_ETH_SRC`. Datapath kernel: `OVS_ACTION_ATTR_SET` with an `OVS_KEY_ATTR_ETHERNET` payload of 12 bytes (eth_src 6 plus eth_dst 6, setting only the requested field).

**Wire encoding.** OF 1.0 format: action header `type=4` (OFPAT10_SET_DL_SRC) plus `length=16` plus a 6-byte MAC payload plus 6-byte padding (8-byte aligned). OF 1.2 and later: encoded as `OFPAT_SET_FIELD` with an `OXM_OF_ETH_SRC` 6-byte OXM TLV. On the wire the engineer sees 16 bytes for OF 1.0 and about 16 bytes (8 header plus 4 OXM header plus 6 MAC plus 2 padding) for OF 1.2 and later.

**Applied examples.**

```
actions=mod_dl_src:aa:bb:cc:00:00:01,output:5                                # plain rewrite
actions=mod_dl_src:aa:bb:cc:00:00:01,mod_dl_dst:aa:bb:cc:00:02:01,dec_ttl,output:uplink
                                                                              # L3 router forward
actions=mod_dl_src:aa:bb:cc:dd:ee:ff,IN_PORT                                  # ARP responder
```

A typical OVN-generated flow at table 11 (lr_in_arp_resolve) on br-int:

```
priority=100,reg0=0xa010001,reg14=0x1,actions=mod_dl_src:aa:bb:cc:00:01:01,resubmit(,12)
```

**How an engineer masters operations.** Four skills: (1) **Read**: recognise that `mod_dl_src` appears in dump output even when the bridge is configured for OF 1.5 (OVS keeps the legacy display). (2) **Edit**: switch new code to `set_field:<mac>->eth_src` for consistency and to enable masks. (3) **Observe**: `ovs-ofctl dump-flows br-int | grep mod_dl_src | wc -l` counts MAC-rewrite rules (mostly the OVN router pipeline). (4) **Debug**: `ofproto/trace 'in_port=...,dl_src=aa:..,...'` shows the line `set(eth(src=cc:dd:..))` in Datapath actions, confirming the rewrite.

**Classification.** A field-rewrite action; OF 1.0 raw type; not a terminator; no partial-write capability (always overwrites the full 6 bytes; for partial use `set_field:V/M->eth_src`). Same class: `mod_dl_dst` (rewrite eth_dst), `mod_nw_src/dst` (rewrite IP), `mod_tp_src/dst` (rewrite L4 port).

**Source code.** `lib/ofp-actions.c` function anchors: `parse_SET_ETH_SRC` line 1976, `format_SET_ETH_SRC` line 1988, `encode_SET_ETH_SRC` around line 1957 (calling the `encode_SET_ETH_addr` helper with raw type `OFPAT_RAW_SET_DL_SRC`). Header `include/openvswitch/ofp-actions.h` line 461 `struct ofpact_mac`. Branch baseline: OVS v2.17.9.

**Failure scenarios.** (a) `mod_dl_src:aa:bb` (only 2 bytes) is rejected by the parser with `invalid Ethernet address`. Fix: provide all 6 bytes. (b) Rewriting the source MAC but forgetting to rewrite the destination MAC in an L3 forward scenario means the receiving host gets the packet with the wrong destination MAC and drops it at the NIC filter. Fix: pair `mod_dl_src` with `mod_dl_dst`. (c) Rewriting the source MAC immediately before `output:NORMAL` causes MAC-table learning to record the new MAC against the wrong in_port, poisoning the FDB. Fix: use `output:<explicit_port>` instead of NORMAL.

**Synthetic exercise.** Mininet, two hosts h1 and h2 in the same broadcast domain. Install rule `priority=100,in_port=1,actions=mod_dl_src:de:ad:be:ef:00:01,output:2`. On h2 run `tcpdump -i h2-eth0 -n -e`, then run `h1 ping h2`. Verify the ARP request from h1 to h2 has the fake source MAC `de:ad:be:ef:00:01` instead of the real h1-eth0 MAC.

**Observation and verification.** `ovs-ofctl dump-flows br0` shows the action format `mod_dl_src:aa:..`. `ovs-appctl coverage/show | grep -i ofpact` counts the ofpact instances executed. The `ofproto/trace` line `Datapath actions: set(eth(src=aa:..)),N` confirms the rewrite.

**Comparison with other systems.** Linux netfilter has no direct equivalent in the `filter` table (filter only, no modify), but the `mangle` table has `--src-mac` set through the `MAC` extension (rarely used). `tc filter ... action pedit munge offset 6 u32 set 0xaabbccdd` is the lowest-level option. Cisco IOS does not expose MAC rewrite through the standard CLI (it is hardcoded into the L3 routing logic). P4 `hdr.ethernet.src_addr = 0x...` is a direct assignment.

### 4.9.33.4 `mod_dl_dst`, the legacy OF 1.0 ethernet destination MAC overwrite

**Concept.** `mod_dl_dst` is a legacy OF 1.0 action that overwrites the 6-byte ethernet destination MAC. Equivalent to `set_field:<mac>->eth_dst` from OF 1.2 onward. Symmetric to `mod_dl_src`: the same struct ofpact, the same wire encoding, only the target field differs. OVS keeps the syntax for backward compatibility and dump output. Action type: **specialised field rewrite**, OF 1.0 raw type `OFPAT10_SET_DL_DST`.

**History and context.** Same timeline as `mod_dl_src`: OF 1.0 (2009) added `OFPAT_SET_DL_DST` action type 5. OVS supported it from the start. OF 1.2 (2011) added `set_field` as the replacement. ovn-northd uses `mod_dl_dst` heavily in the distributed-router pipeline and the load-balancer DNAT scenario.

**Position in the architecture.** At the OpenFlow protocol layer: `OFPAT10_SET_DL_DST` raw type 5 in OF 1.0; OF 1.2 and later encode to `OFPAT_SET_FIELD` with MFF_ETH_DST. At the OVS internal layer: shares `struct ofpact_mac` with `mod_dl_src`, distinguished by `ofpact_type = OFPACT_SET_ETH_DST` versus `OFPACT_SET_ETH_SRC` (verify `ofp-actions.h` line 461 comment, shared between the two).

**Role.** Primary role: L3 router rewrite of the destination MAC (a packet leaving the router output must have the next-hop MAC as its destination, looked up in the ARP table). Secondary role: load-balancer DNAT (rewrite the destination MAC to forward to a backend pool member). Auxiliary role: ARP responder. Blast radius: per-packet.

**Why it exists.** Same motivation as `mod_dl_src`: L3 forwarding requires rewriting both source and destination MAC. Specifically for `mod_dl_dst`: rewriting the destination MAC is the more critical for forwarding correctness; if the destination MAC is not the actual next-hop MAC, the receiving host drops the frame at the NIC filter at L1.

**Problems it solves.** Specifically: allow OVS to play the L3 router and load balancer role purely in software through flow rules. Approach: an action that rewrites the destination MAC after ARP resolution (look up the next-hop MAC, then rewrite). Scope: OVN distributed router, Mininet L3 lab, software LB.

**Importance.** Tier 2 medium with very high frequency in OVN production. On a chassis running 10 logical routers, the OVN br-int has thousands of flows containing `mod_dl_dst` (one flow per LRP times remote MAC). Engineer must-know: the syntax is symmetric to `mod_dl_src`, and the wire format is identical.

**How it works.** The parser in `lib/ofp-actions.c` function `parse_SET_ETH_DST` (line 1982) is the same as `parse_SET_ETH_SRC`, calling `str_to_mac` to store into `ofpact_put_SET_ETH_DST(pp->ofpacts)->mac`. The formatter `format_SET_ETH_DST` (line 1996) prints `mod_dl_dst:%s` (verify lines 1999-2000: `ds_put_format(fp->s, "%smod_dl_dst:%s"ETH_ADDR_FMT, colors.param, colors.end, ETH_ADDR_ARGS(a->mac));`). The encoder `encode_SET_ETH_DST` (around line 1965) calls the helper `encode_SET_ETH_addr` with raw type `OFPAT_RAW_SET_DL_DST` and field `MFF_ETH_DST`. Datapath kernel: `OVS_ACTION_ATTR_SET` with an `OVS_KEY_ATTR_ETHERNET` payload setting eth_dst.

**Wire encoding.** OF 1.0 format: action header `type=5` (OFPAT10_SET_DL_DST) plus `length=16` plus a 6-byte MAC payload plus 6-byte padding. OF 1.2 and later: `OFPAT_SET_FIELD` with an `OXM_OF_ETH_DST` 6-byte OXM TLV. Fully symmetric to `mod_dl_src`, only the type number differs.

**Applied examples.**

```
actions=mod_dl_dst:00:50:56:aa:bb:cc,output:5                                  # explicit MAC rewrite
actions=mod_dl_src:rt_mac,mod_dl_dst:next_hop_mac,dec_ttl,output:uplink         # L3 router primitive
actions=group:1                                                                 # load balancer (group bucket contains mod_dl_dst)
```

A typical OVN-generated flow at table 12 (lr_in_arp_resolve / lr_out_delivery):

```
priority=100,reg0=0xc0a80101,reg14=0x2,actions=mod_dl_dst:fa:16:3e:aa:bb:cc,resubmit(,13)
```

OVN load balancer with group buckets (verify §4.9.11):

```
group_id=1,type=select,
  bucket=weight:100,actions=mod_dl_dst:aa:bb:cc:00:00:01,output:5,
  bucket=weight:100,actions=mod_dl_dst:aa:bb:cc:00:00:02,output:6
```

**How an engineer masters operations.** Four skills: (1) **Read**: in OVN dump output recognise `mod_dl_dst` at tables 11 and 12 (LR pipeline) as an ARP-resolved MAC. (2) **Edit**: for new code switch to `set_field:<mac>->eth_dst`, especially when a partial mask write is needed. (3) **Observe**: `ovs-ofctl dump-flows br-int | grep mod_dl_dst | sort -u | head` shows distinct MAC values, usually correlating with the host's ARP table. (4) **Debug**: `ofproto/trace` shows the line `Datapath actions: set(eth(dst=aa:..)),N`.

**Classification.** A field-rewrite action; OF 1.0 raw type; symmetric to `mod_dl_src`; no partial-write (always overwrites 6 bytes). Same class: `mod_dl_src`, `set_field:V->eth_dst`. Different from `move:NXM_OF_ETH_SRC[]->NXM_OF_ETH_DST[]`: `move` copies field to field (dynamic, source value at runtime), whereas `mod_dl_dst` sets a constant value compiled into the rule.

**Source code.** `lib/ofp-actions.c` function anchors: `parse_SET_ETH_DST` line 1982, `format_SET_ETH_DST` line 1996, `encode_SET_ETH_DST` around line 1965. Header `include/openvswitch/ofp-actions.h` line 461 `struct ofpact_mac` (shared with SET_ETH_SRC). Branch baseline: OVS v2.17.9.

**Failure scenarios.** (a) Rewriting the destination MAC while the next-hop ARP entry is stale (the host has changed MAC) means the inbound packet is dropped at the host. Fix: integrate with ARP refresh (OVN uses FDB aging plus `ovn-controller` re-resolve). (b) Rewriting the destination MAC in a load balancer but forgetting `ct(commit)` for the NAT mapping leaves return traffic without a reverse rewrite, breaking the connection. Fix: pair `ct(commit)` with `mod_dl_dst`. (c) Rewriting the destination MAC to a multicast MAC accidentally (bit 0 of byte 0 = 1) causes the packet to flood instead of unicast. Fix: validate that bit 0 is 0 before committing the rule.

**Synthetic exercise.** Mininet, three hosts h1, h2, and h3. Install rule `priority=100,in_port=h1-eth0,ip,nw_dst=10.0.0.3,actions=mod_dl_dst:<mac_h2>,output:h2-eth0` to redirect a packet destined for h3 to host h2. `tcpdump -i h2-eth0` confirms h2 receives the packet (even though the IP destination is h3), demonstrating an L2 redirect that bypasses L3.

**Observation and verification.** `ovs-ofctl dump-flows br0` shows `mod_dl_dst:aa:..`. `ovs-appctl ofproto/trace br0 'in_port=1,dl_dst=...,...'` shows the line `set(eth(dst=...))`. `tcpdump -e` on the output port verifies the wire-level MAC rewrite.

**Comparison with other systems.** Linux netfilter `iptables -t mangle -j MARK` does not rewrite MAC, but `tc filter ... action pedit munge offset 0 u32 set ...` allows the low-level operation. Cisco IOS `mac-address-table static` plus the L3 routing engine implicitly rewrites the destination MAC (no programmable action exposed). NSX-T Edge load balancer rewrites the destination MAC implicitly through the DLR. P4 `hdr.ethernet.dst_addr = ...` is a direct assignment.

### 4.9.33.5 `dec_ttl`, decrement IP TTL by one and emit PACKET_IN when TTL=0

**Concept.** `dec_ttl` is an action that decrements the IP Time-To-Live (or IPv6 Hop Limit) by one. After the decrement: when TTL > 0, the packet continues through the pipeline; when TTL = 0, the packet is sent to the controller as `OFPT_PACKET_IN` with reason code `OFPR_INVALID_TTL` (3). OVS recomputes the IP checksum automatically after the decrement (because TTL is one byte in the header covered by the IPv4 checksum). MPLS has its own variant `dec_mpls_ttl` for the MPLS label TTL field. Action type: **specialised field rewrite with a controller side-effect**.

**History and context.** OpenFlow 1.0 (2009): added `OFPAT_SET_NW_TTL` (set absolute value) plus the Nicira extension `NXAST_DEC_TTL` (decrement). OpenFlow 1.1 (2011): standardised as `OFPAT11_DEC_NW_TTL` (action type 18). OpenFlow 1.2 (2011): added `NXAST_DEC_TTL_CNT_IDS` allowing the controller-ID list that should receive PACKET_IN when TTL=0 (multi-controller scenario). OF 1.5 spec §5.10.2 Action Types states the semantics: "If TTL becomes 0 after decrement, the packet shall be sent to the controller with reason OFPR_INVALID_TTL".

**Position in the architecture.** At the OpenFlow protocol layer: `OFPAT11_DEC_NW_TTL` action type 18 in OF 1.1 and later; the NXAST extension covers OF 1.0 plus multi-controller. At the OVS internal layer: `struct ofpact_cnt_ids` (`include/openvswitch/ofp-actions.h` line 1031) holds `n_controllers` plus the `cnt_ids[]` array. At the datapath layer: the kernel translates to `OVS_ACTION_ATTR_SET` with the TTL key decremented plus a checksum recompute; when TTL=0 after decrement, the kernel returns an upcall to userspace to emit PACKET_IN.

**Role.** Primary role: L3 routing TTL decrement (each router hop decrements TTL by 1; reaching 0 produces ICMP TTL Exceeded). Secondary role: traceroute support (the host sends probes with TTL=1, 2, 3, ... and each router decrements; when TTL=0 the router emits ICMP Time Exceeded). Blast radius: per-packet, every IP packet flowing through a flow with `dec_ttl`.

**Why it exists.** Standard L3 forwarding requires decrementing TTL to prevent infinite routing loops (RFC 791). Before OpenFlow, hardware ASICs did this automatically inside the L3 lookup engine. OpenFlow exposes it as a programmable action so OVS can play the role of a software router. A dedicated action (rather than reusing `set_field:<v>->nw_ttl`) is needed because of the special handling at TTL=0 (send PACKET_IN rather than silently drop or continue).

**Problems it solves.** Specifically: (a) standard TTL decrement for a software L3 router, (b) ICMP Time Exceeded support through the controller (the controller receives a PACKET_IN with reason invalid_ttl and emits the ICMP message). Approach: an action that decrements the field, recomputes the checksum, and triggers the side-effect PACKET_IN. Scope: OVS L3 router, OVN logical-router L3 forwarding, Mininet L3 lab, traceroute support inside a virtual network.

**Importance.** Tier 2 medium. Every L3-forwarding rule of an OVN logical-router pipeline includes `dec_ttl` at table 11 (lr_in_ip_routing). Engineer must-know: the special behaviour at TTL=0 (PACKET_IN, not drop), the reason code `OFPR_INVALID_TTL`, and the requirement that the controller has a handler for this reason if standard ICMP Time Exceeded emission is desired.

**How it works.** Parser `lib/ofp-actions.c` function `parse_DEC_TTL` (line 3709): with empty argument, `parse_noargs_dec_ttl` adds controller ID = 0 (default). With an argument (a comma-separated controller-ID list), the loop `strtok` parses each ID and marks the raw type as `NXAST_RAW_DEC_TTL_CNT_IDS`. Formatter `format_DEC_TTL` (line 3742): print "dec_ttl" plus `(id1,id2,...)` when raw is NXAST_DEC_TTL_CNT_IDS. Encoder `encode_DEC_TTL` (line 3672): when raw is NXAST_DEC_TTL_CNT_IDS, or n_controllers is not 1, or id is not 0, encode the NXAST extension; otherwise encode standard `OFPAT_DEC_NW_TTL`. Datapath kernel: read the current TTL, decrement by 1, recompute the IPv4 header checksum (`csum_replace` macro); when TTL=0, mark the skb for an upcall with reason invalid_ttl.

**Wire encoding.** OF 1.1 and later standard `OFPAT_DEC_NW_TTL`: action header `type=18` (DEC_NW_TTL) plus `length=8` (header only, no payload) plus 4-byte padding. NXAST multi-controller format: `vendor=0x2320` (Nicira) plus `subtype=20` (NXAST_DEC_TTL_CNT_IDS) plus `n_controllers` (2 bytes) plus an array of 16-bit IDs plus padding to 8-byte alignment. OF 1.0 has no standard form and uses NXAST.

**Applied examples.**

```
actions=dec_ttl,output:1                                              # plain L3 forward
actions=dec_ttl,mod_dl_src:rt_mac,mod_dl_dst:nh_mac,output:uplink     # full L3 router primitive
actions=dec_ttl(0,1)                                                  # multi-controller (ID 0 plus 1)
priority=100,ip,actions=dec_ttl,resubmit(,12)                         # L3 forward subroutine
priority=0,ip,actions=controller(reason=invalid_ttl)                  # table-miss for ICMP TTL exceeded
```

A typical OVN-generated flow at table 11 lr_in_ip_routing:

```

```

**Cách kỹ sư vận hành thành thạo.** Bốn kỹ năng: (1) **Đọc**: nhận ra `dec_ttl` trong OVN router pipeline + `dec_mpls_ttl` cho MPLS scenario. (2) **Sửa**: pair `dec_ttl` với rule fallback `priority=0,actions=controller(reason=invalid_ttl)` ở table phía sau để ICMP Time Exceeded được emit chuẩn. (3) **Quan sát**: `ovs-ofctl dump-flows br-int | grep dec_ttl | wc -l` đếm rule L3 forward; `coverage/show | grep dec_ttl` đếm số packet đã giảm TTL. (4) **Debug**: `ofproto/trace 'in_port=...,ip,nw_ttl=1,...'` simulate packet TTL=1 → trace show `dec_ttl` set TTL=0 → emit packet_in.

**Phân loại.** Action loại field rewrite specialized với side-effect controller. Đặc tính: stateful (modify in-place + recompute checksum), conditional side-effect (PACKET_IN khi TTL=0), không terminator (TTL>0 cho phép action sau). Cùng class: `set_field:<v>->nw_ttl` (set absolute, không emit PACKET_IN khi 0), `dec_mpls_ttl` (cho MPLS label TTL). Khác `set_nw_ttl`: `set_nw_ttl:<v>` set tuyệt đối không decrement.

**Source code.** `lib/ofp-actions.c` function anchor: `encode_DEC_TTL` line 3672, `parse_DEC_TTL` line 3709, `format_DEC_TTL` line 3742, `parse_noargs_dec_ttl` line 3690 ~. Header `include/openvswitch/ofp-actions.h` line 1031 `struct ofpact_cnt_ids` (cũng dùng cho multi-controller config). Datapath kernel `datapath/actions.c` function `set_ipv4` xử lý TTL decrement + `csum_replace2` cho IPv4 checksum. Branch baseline: OVS v2.17.9.

**Kịch bản bẻ gãy.** (a) `dec_ttl` trên packet không phải IP (ARP, LLDP, raw L2 frame) → action skip silent (kernel datapath check L3 trước). Fix: thêm `ip` hoặc `ipv6` vào match. (b) Quên fallback `controller(reason=invalid_ttl)` ở table-miss → packet TTL=0 sinh PACKET_IN nhưng không có handler → controller log warning + drop. Engineer thấy traceroute không trả về Time Exceeded. Fix: cài rule fallback. (c) `dec_ttl` chained quá nhiều lần trong cùng pipeline (ví dụ 2 lần qua hai resubmit) → TTL giảm 2 đơn vị mỗi packet thay vì 1 → traceroute report sai số hop. Fix: pipeline design audit.

**Bài tập synthetic.** Mininet topology 1 OVS bridge `s1` + 2 host h1 + h2. Cài rule `priority=100,in_port=h1-eth0,ip,actions=dec_ttl,output:h2-eth0` + rule `priority=0,ip,actions=controller(reason=invalid_ttl)`. Trên h1 chạy `ping -t 1 h2` (TTL=1). Bắt PACKET_IN ở controller (ovs-ofctl monitor) verify reason `invalid_ttl`. So sánh với `ping -t 2 h2` thấy reach success.

**Cách quan sát + xác minh.** `ovs-ofctl dump-flows br0` show `dec_ttl`. `ovs-ofctl monitor br0 watch:` capture PACKET_IN với `reason=invalid_ttl` khi TTL=0. `tcpdump -i h2-eth0 -v` verify TTL field giảm đúng 1 từ giá trị h1 gửi.

**So sánh với hệ khác.** Linux IP forwarding tự động giảm TTL trong `ip_forward` function (`net/ipv4/ip_forward.c`); khi TTL=0 sinh ICMP Time Exceeded qua `icmp_send(ICMP_TIME_EXCEEDED)`. Cisco IOS L3 routing engine cũng tự động giảm TTL + emit ICMP. P4 `hdr.ipv4.ttl = hdr.ipv4.ttl - 1` direct decrement trong control block; lập trình viên P4 phải tự handle case TTL=0 (gửi packet ra control plane tương đương). FRR + Linux kernel inherit Linux behavior.

### 4.9.33.6 `mod_nw_src` — ghi đè IPv4 source address kiểu legacy OF 1.0

**Khái niệm.** `mod_nw_src` là action legacy OF 1.0 để ghi đè 4 byte IPv4 source address của packet. Tương đương semantics với `set_field:<ip>->ipv4_src` (hoặc bí danh cũ `nw_src`) của OF 1.2+. OVS giữ syntax cho backward compat + dump output. Phần lớn rule SNAT (Source NAT) trong scenario L3 router thuần software sẽ dùng action này (hoặc `nat()` action cao cấp hơn). Loại action: **field rewrite specialized**, OF 1.0 raw type `OFPAT10_SET_NW_SRC`.

**Lịch sử + bối cảnh.** OpenFlow 1.0 (2009) định nghĩa `OFPAT_SET_NW_SRC` action type 6, với 4 byte IPv4 address payload (+ 0 byte padding align 8). OVS từ phiên bản đầu hỗ trợ. OF 1.2 (2011) thêm `set_field` thay thế. OF 1.5 spec không deprecate hẳn `OFPAT10_SET_NW_SRC` raw type vì OVS vẫn translate forward. Trong production OVN, action này hiếm dùng trực tiếp vì OVN có `nat()` action cao cấp xử lý SNAT/DNAT đồng thời với conntrack; tuy nhiên nhiều rule manually-installed (Mininet lab, debug rule) vẫn dùng `mod_nw_src`.

**Vị trí trong kiến trúc.** Ở tầng OpenFlow protocol: `OFPAT10_SET_NW_SRC` raw type 6 ở OF 1.0; OF 1.2+ encode thành `OFPAT_SET_FIELD` với MFF_IPV4_SRC. Ở tầng OVS internal: cùng `struct ofpact_ipv4` với `mod_nw_dst` (verify `include/openvswitch/ofp-actions.h`), phân biệt qua `ofpact_type = OFPACT_SET_IPV4_SRC` vs `OFPACT_SET_IPV4_DST`. Ở tầng datapath kernel: translate thành `OVS_ACTION_ATTR_SET` với `OVS_KEY_ATTR_IPV4` payload (12 byte: src 4 + dst 4 + proto 1 + tos 1 + ttl 1 + frag 1).

**Vai trò.** Vai trò chính: SNAT (Source NAT) thuần action — rewrite IP src để outgoing traffic mang địa chỉ public thay vì private. Vai trò phụ: privacy/spoofing protection rewrite (egress filter), masquerade trên gateway. Phạm vi ảnh hưởng: per-packet, IPv4 checksum + L4 checksum (TCP/UDP) auto-recompute do datapath.

**Vì sao sinh ra.** L3 router thuần software cần SNAT khi forward giữa public/private subnet. Trước OpenFlow, Linux netfilter `iptables -t nat -A POSTROUTING -j SNAT --to-source X` làm việc này. OpenFlow expose action programmable để OVS tự handle qua flow rule, không cần iptables stack riêng. Action riêng (không đợi `set_field`) vì OF 1.0 thời đó chưa có generic field abstraction.

**Vấn đề giải quyết.** Cụ thể: cho phép SNAT thuần OpenFlow trên OVS bridge mà không cần dependency vào iptables/netfilter. Cách giải: action rewrite IP src + auto-recompute IPv4 + L4 checksum. Phạm vi: SNAT đơn giản (1 rule = 1 mapping); với SNAT động (port pool, conntrack-based) cần kết hợp `nat()` action thay thế.

**Tầm quan trọng.** Tier 2 medium. Trong rule production OVN hiếm gặp trực tiếp (OVN dùng `nat()`), nhưng trong Mininet lab + custom controller pattern + manual debug rule rất phổ biến. Engineer must-know: cú pháp `mod_nw_src:10.0.0.5`, hiểu rằng action không tự commit conntrack (cần kết hợp `ct(commit, nat(src=...))` cho stateful NAT đầy đủ).

**Cơ chế hoạt động.** Parser `lib/ofp-actions.c` function `parse_SET_IPV4_SRC` (line ~3360 v3.3.0) gọi `str_to_ip` parse string IPv4 → 32-bit `ovs_be32`, lưu vào `ofpact_put_SET_IPV4_SRC(pp->ofpacts)->ipv4`. Format `format_SET_IPV4_SRC` (line ~3378) in `mod_nw_src:%s` với IPv4 dotted notation (cho OF 1.0 mode) hoặc `set_field:%s->nw_src` (cho OF 1.2+). Encode `encode_SET_IPV4_SRC` (line ~3370) chọn raw type `OFPAT_RAW_SET_NW_SRC` cho OF 1.0 wire hoặc fallback `OFPAT_SET_FIELD` cho OF 1.2+. Datapath kernel: `OVS_ACTION_ATTR_SET` với key IPv4 → `set_ipv4` function trong `datapath/actions.c` rewrite 4 byte src + recompute checksum (`csum_replace4`).

**Wire encoding.** OF 1.0 format: action header `type=6` (OFPAT10_SET_NW_SRC) + `length=8` + 4 byte IPv4 address payload. OF 1.2+ format: encode thành `OFPAT_SET_FIELD` action type 25 với OXM TLV `OXM_OF_IPV4_SRC` 4 byte. Trên wire, OF 1.0 cần 8 byte cho action; OF 1.2+ cần 16 byte (8 header + 4 OXM header + 4 value).

**Ví dụ áp dụng.**

```

priority=100,ip,nw_src=192.168.0.0/16,actions=mod_nw_src:10.1.0.1,output:uplink
                                                                          # masquerade gateway
actions=mod_nw_src:10.0.0.5,mod_dl_src:rt_mac,dec_ttl,output:wan        # SNAT + L3 forward
actions=set_field:10.0.0.5->ipv4_src                                    # OF 1.2+ equivalent (preferred)
```

**Cách kỹ sư vận hành thành thạo.** Bốn kỹ năng: (1) **Đọc**: phân biệt `mod_nw_src` (constant value SNAT) với `nat()` action (stateful conntrack-based SNAT). (2) **Sửa**: cho code mới chuyển sang `set_field:<ip>->ipv4_src`; nếu cần stateful SNAT chuyển hẳn sang `ct(commit, nat(src=<ip>))`. (3) **Quan sát**: `ovs-ofctl dump-flows br0 | grep mod_nw_src` đếm rule SNAT thuần action. (4) **Debug**: `ofproto/trace 'in_port=1,ip,nw_src=192.168.1.5,...'` show line `set(ipv4(src=10.0.0.5,...))` xác nhận datapath rewrite + checksum recompute.

**Phân loại.** Action loại field rewrite specialized cho IPv4 src, OF 1.0 raw type, không terminator, không partial-mask (always overwrite full 4 byte; muốn partial mask phải dùng `set_field:<ip>/<mask>->ipv4_src`). Cùng class: `mod_nw_dst` (rewrite IPv4 dst), `mod_tp_src/dst` (rewrite L4 port), `mod_dl_src/dst` (rewrite MAC). Khác `nat()`: `nat()` stateful (per-connection mapping qua conntrack), `mod_nw_src` stateless (mọi packet match rule cùng rewrite).

**Source code.** `lib/ofp-actions.c` function anchor (OVS v2.17.9): `parse_SET_IPV4_SRC` line ~3360, `encode_SET_IPV4_SRC` line ~3370, `format_SET_IPV4_SRC` line ~3378. Header `include/openvswitch/ofp-actions.h` `struct ofpact_ipv4` (shared với SET_IPV4_DST). Datapath kernel `datapath/actions.c` function `set_ipv4` xử lý byte rewrite + `csum_replace4` recompute checksum. Branch baseline: OVS v2.17.9.

**Kịch bản bẻ gãy.** (a) `mod_nw_src:10.0.0.5` không có `ip` trong match → flow add reject với `OFPBAC_MATCH_INCONSISTENT`. Fix: thêm `ip` vào match. (b) Stateless SNAT cho TCP traffic: outbound packet SNAT đến `10.0.0.5`, nhưng reply traffic (return) không có rule reverse-rewrite → connection broken. Fix: pair rule `mod_nw_src:10.0.0.5` (egress) + rule `mod_nw_dst:192.168.1.5` (ingress) hoặc dùng `ct(commit, nat(src=10.0.0.5))` stateful. (c) Quên rằng IPv4 checksum tự động recompute → engineer hoài công viết action recompute checksum thủ công. Verify: kernel datapath đã làm việc này.

**Bài tập synthetic.** Mininet 2 host h1 (192.168.1.5) + h2 (10.0.0.10) qua bridge `s1`. Cài rule `priority=100,in_port=h1-eth0,ip,actions=mod_nw_src:10.0.0.99,output:h2-eth0`. Trên h2 chạy `tcpdump -i h2-eth0 -n` xem ICMP request đến từ `10.0.0.99` thay vì `192.168.1.5` thật. Verify checksum đúng (`tcpdump` không in `bad cksum` warning).

**Cách quan sát + xác minh.** `ovs-ofctl dump-flows br0` show `mod_nw_src:10.0.0.5` (OF 1.0 mode) hoặc `set_field:10.0.0.5->nw_src` (OF 1.5 mode). `ovs-appctl ofproto/trace br0 'in_port=1,ip,...'` show line `Datapath actions: set(ipv4(src=10.0.0.5,...)),N`. `tcpdump -i wan-port -n` xác nhận packet wire-level có IP src đã rewrite + checksum đúng.

**So sánh với hệ khác.** Linux netfilter `iptables -t nat -A POSTROUTING -j SNAT --to-source 10.0.0.5` tương đương stateful (conntrack-aware). `iptables -t mangle -j MARK` không rewrite IP. `tc filter ... action nat ingress 192.168.1.0/24 10.0.0.0` cùng concept stateless thuần action. Cisco IOS `ip nat inside source` stateful conntrack-based. P4 `hdr.ipv4.src_addr = 0x0a000005` direct assignment.

### 4.9.33.7 `mod_nw_dst` — ghi đè IPv4 destination address kiểu legacy OF 1.0

**Cách quan sát + xác minh.** `ovs-ofctl dump-flows br0` show `mod_nw_dst:10.0.0.10`. `ovs-appctl ofproto/trace br0 'in_port=1,ip,nw_dst=203.0.113.10,...'` show line `Datapath actions: set(ipv4(dst=10.0.0.10,...)),N`. `tcpdump -i backend-port -n` verify wire-level rewrite.

**So sánh với hệ khác.** Linux netfilter `iptables -t nat -A PREROUTING -d 203.0.113.10 -j DNAT --to-destination 10.0.0.10` stateful equivalent. Linux IPVS `ipvsadm -A -t 203.0.113.10:80 -s rr` + real server stateful LB. Cisco IOS `ip nat inside destination` stateful. NSX-T DLR có DNAT rule ở North-South gateway. P4 `hdr.ipv4.dst_addr = 0x0a00000a` direct.

### 4.9.33.8 `mod_tp_src` — ghi đè L4 source port (TCP/UDP/SCTP) kiểu legacy OF 1.0

**Khái niệm.** `mod_tp_src` là action legacy OF 1.0 để ghi đè 2 byte L4 source port của packet. Áp dụng cho TCP, UDP, SCTP — tuỳ vào L4 protocol field hiện tại (`nw_proto=6` cho TCP, `=17` cho UDP, `=132` cho SCTP). Tương đương `set_field:<port>->tcp_src` (hoặc `udp_src`, `sctp_src`) của OF 1.2+. Phần chính của PAT (Port Address Translation): cùng `mod_nw_src` tạo SNAT đầy đủ với port pool. Loại action: **field rewrite specialized**, OF 1.0 raw type `OFPAT10_SET_TP_SRC`.

**Lịch sử + bối cảnh.** OpenFlow 1.0 (2009) thêm `OFPAT_SET_TP_SRC` action type 9 với 2 byte port payload. OVS hỗ trợ từ đầu. Trong OF 1.0 không phân biệt protocol L4 — action duy nhất `mod_tp_src` xử lý TCP/UDP/SCTP. OF 1.2 (2011) đưa vào `set_field` với OXM phân biệt theo protocol: `OXM_OF_TCP_SRC`, `OXM_OF_UDP_SRC`, `OXM_OF_SCTP_SRC`. Lý do tách: prerequisite check chặt hơn, parser refuse rewrite TCP port khi packet không có TCP header. ovn-northd-generated flow ở Logical Load Balancer pipeline thường dùng action này khi LB rewrite source port để tránh collision pool.

**Vị trí trong kiến trúc.** Ở tầng OpenFlow protocol: `OFPAT10_SET_TP_SRC` raw type 9 ở OF 1.0 (protocol-agnostic); OF 1.2+ encode thành `OFPAT_SET_FIELD` với MFF_TCP_SRC / MFF_UDP_SRC / MFF_SCTP_SRC tuỳ context (parser inspect `nw_proto` match để chọn). Ở tầng OVS internal: `struct ofpact_l4_port` (`include/openvswitch/ofp-actions.h`) chứa 2 byte port + flow_pp protocol hint. Ở tầng datapath kernel: translate thành `OVS_ACTION_ATTR_SET` với `OVS_KEY_ATTR_TCP` / `_UDP` / `_SCTP` payload set src port.

**Vai trò.** Vai trò chính: PAT (Port Address Translation) cùng `mod_nw_src` cho SNAT + port multiplexing — nhiều internal connection cùng SNAT đến 1 public IP nhưng port pool khác nhau để phân biệt. Vai trò phụ: privacy port randomization (rewrite ephemeral port mask client OS fingerprint), egress filter port masking. Phạm vi: per-packet, L4 checksum auto-recompute.

**Vì sao sinh ra.** SNAT 1-to-1 (1 internal IP → 1 public IP) không scale: mỗi internal host cần 1 public IP. PAT 1-to-N (1 public IP + N port = N internal connection mapping) scale linear với port pool 65535. Hardware/Linux netfilter NAT đã có concept này từ đầu; OpenFlow expose programmable qua action.

**Vấn đề giải quyết.** Cụ thể: cho phép PAT thuần OpenFlow (port rewrite + IP rewrite + checksum recompute đồng thời) qua flow rule. Cách giải: action rewrite L4 port + checksum L4 auto-recompute. Phạm vi: PAT cơ bản (1 rule = 1 mapping); PAT động (port pool tự allocate) cần kết hợp `nat(src=...)` action stateful.

**Tầm quan trọng.** Tier 2 medium. Trong rule production hiếm gặp trực tiếp (OVN dùng `nat()` action stateful), nhưng trong custom controller pattern + Mininet PAT lab + manual debug rất phổ biến. Engineer must-know: cú pháp `mod_tp_src:8080`, hiểu rằng action chỉ valid khi packet có L4 protocol (TCP/UDP/SCTP); ICMP packet không có port → action skip silent.

**Cơ chế hoạt động.** Parser `lib/ofp-actions.c` function `parse_SET_L4_SRC_PORT` (line ~3488 v3.3.0) gọi `str_to_u16` parse string thành 16-bit port number, lưu vào `ofpact_put_SET_L4_SRC_PORT(pp->ofpacts)->port`. Format `format_SET_L4_SRC_PORT` (line ~3502) in `mod_tp_src:%"PRIu16` cho OF 1.0 hoặc `set_field:<port>->tcp_src` cho OF 1.2+. Encode `encode_SET_L4_SRC_PORT` (line ~3470): chọn raw type `OFPAT_RAW_SET_TP_SRC` cho OF 1.0 wire (protocol-agnostic, datapath dispatch theo flow proto); cho OF 1.2+ encode `OFPAT_SET_FIELD` với OXM specific (TCP/UDP/SCTP). Datapath kernel: `OVS_ACTION_ATTR_SET` → `set_tcp` / `set_udp` / `set_sctp` function rewrite 2 byte src port + recompute L4 checksum (`inet_proto_csum_replace2`).

**Wire encoding.** OF 1.0 format: action header `type=9` (OFPAT10_SET_TP_SRC) + `length=8` + 2 byte port payload + 2 byte padding (align 8). OF 1.2+ format: `OFPAT_SET_FIELD` action type 25 với OXM TLV `OXM_OF_TCP_SRC` / `OXM_OF_UDP_SRC` / `OXM_OF_SCTP_SRC` 2 byte. Wire size tương tự: 8 byte cho OF 1.0; ~16 byte cho OF 1.2+.

**Ví dụ áp dụng.**

```
priority=100,ip,tcp,tp_src=4001,actions=mod_tp_src:8080,output:wan       # PAT rewrite ephemeral
priority=100,ip,nw_src=192.168.1.0/24,tcp,
        actions=mod_nw_src:10.0.0.5,mod_tp_src:33000,output:uplink        # SNAT + PAT combo
priority=100,udp,actions=mod_tp_src:5060,output:sip-uplink                # rewrite SIP source
actions=set_field:8080->tcp_src                                          # OF 1.2+ TCP only
actions=set_field:5060->udp_src                                          # OF 1.2+ UDP only
```

**Cách kỹ sư vận hành thành thạo.** Bốn kỹ năng: (1) **Đọc**: phân biệt `mod_tp_src` (OF 1.0 protocol-agnostic) với `set_field:V->tcp_src` (OF 1.2+ TCP-specific). (2) **Sửa**: cho code mới chuyển sang `set_field` với OXM specific để parser tự enforce prerequisite. Cho PAT động chuyển sang `nat(src=ip:port-range)` stateful. (3) **Quan sát**: `ovs-ofctl dump-flows br0 | grep mod_tp_src` đếm rule PAT thuần action. (4) **Debug**: `ofproto/trace 'in_port=1,tcp,tp_src=4001,...'` show line `set(tcp(src=8080,...))` xác nhận datapath rewrite + checksum.

**Phân loại.** Action loại field rewrite specialized cho L4 src port, OF 1.0 raw type protocol-agnostic, không partial-mask (always overwrite 2 byte). Cùng class: `mod_tp_dst` (rewrite L4 dst port), `mod_nw_src/dst` (rewrite IP), `set_field:V->tcp_src/udp_src/sctp_src`. Khác `nat()`: `nat()` stateful với port pool tự allocate; `mod_tp_src` stateless constant rewrite.

**Source code.** `lib/ofp-actions.c` function anchor (OVS v2.17.9): `parse_SET_L4_SRC_PORT` line ~3488, `encode_SET_L4_SRC_PORT` line ~3470, `format_SET_L4_SRC_PORT` line ~3502. Header `include/openvswitch/ofp-actions.h` `struct ofpact_l4_port` (shared với SET_L4_DST_PORT). Datapath kernel `datapath/actions.c` function `set_tcp` / `set_udp` / `set_sctp` xử lý port rewrite + checksum recompute. Branch baseline: OVS v2.17.9.

**Kịch bản bẻ gãy.** (a) `mod_tp_src:8080` không có `tcp` hoặc `udp` hoặc `sctp` trong match → flow add trên OF 1.2+ reject với `OFPBAC_MATCH_INCONSISTENT`; trên OF 1.0 accept nhưng action skip silent cho packet ICMP. Fix: thêm `tcp`/`udp` vào match. (b) PAT mà quên rewrite ephemeral source port unique cho mỗi connection → port collision khiến reply NAT mapping ambiguous → connection broken. Fix: dùng port pool động qua `nat(src=ip:1024-65535)`. (c) Rewrite tp_src thành 0 → port reserved invalid → kernel drop hoặc reject application layer. Fix: validate port > 0.

**Bài tập synthetic.** Mininet 2 host h1 + h2 qua bridge `s1`. Trên h2 chạy `nc -l -p 8080`. Cài rule `priority=100,in_port=h1-eth0,tcp,actions=mod_tp_src:9999,output:h2-eth0`. Trên h1 chạy `nc h2 8080`. Trên h2 chạy `ss -tnp | grep 8080` xác nhận remote peer (h1) port là 9999 thay vì ephemeral port thật. Verify TCP checksum đúng (`tcpdump -v` không in `bad cksum`).

**Cách quan sát + xác minh.** `ovs-ofctl dump-flows br0` show `mod_tp_src:8080`. `ovs-appctl ofproto/trace br0 'in_port=1,tcp,tp_src=4001,...'` show line `Datapath actions: set(tcp(src=8080,...)),N`. `tcpdump -i wan-port -n -v` verify wire-level: `Source port: 8080` thay vì giá trị gốc.

**So sánh với hệ khác.** Linux netfilter `iptables -t nat -A POSTROUTING -p tcp -j SNAT --to-source 10.0.0.5:33000-33999` PAT với port range stateful. `tc filter ... action nat ingress src` có hỗ trợ port mặc định ngẫu nhiên. Cisco IOS `ip nat inside source ... overload` (PAT) stateful tự allocate. NSX-T Edge gateway PAT hidden trong NAT engine. P4 `hdr.tcp.src_port = 0x1f90` direct.

### 4.9.33.9 `mod_tp_dst` — ghi đè L4 destination port (TCP/UDP/SCTP) kiểu legacy OF 1.0

**Khái niệm.** `mod_tp_dst` là action legacy OF 1.0 để ghi đè 2 byte L4 destination port của packet. Áp dụng cho TCP, UDP, SCTP. Tương đương `set_field:<port>->tcp_dst` (hoặc `udp_dst`, `sctp_dst`) của OF 1.2+. Phần chính của port-based NAT (port forwarding): cùng `mod_nw_dst` tạo DNAT đầy đủ với port mapping. Đối xứng với `mod_tp_src`: cùng `struct ofpact_l4_port`, cùng wire encoding pattern, chỉ khác field target. Loại action: **field rewrite specialized**, OF 1.0 raw type `OFPAT10_SET_TP_DST`.

**Lịch sử + bối cảnh.** OpenFlow 1.0 (2009) thêm `OFPAT_SET_TP_DST` action type 10 với 2 byte port payload. OVS hỗ trợ từ đầu. OF 1.2 (2011) đưa vào `set_field` với OXM phân biệt protocol như `mod_tp_src`. Production OVN: action này là building block của port forwarding (expose internal port 8080 qua public port 80) + Logical Load Balancer pipeline rewrite VIP:80 → backend:8080. Cisco PAT inbound cũng cùng pattern.

**Vị trí trong kiến trúc.** Ở tầng OpenFlow protocol: `OFPAT10_SET_TP_DST` raw type 10 ở OF 1.0 (protocol-agnostic); OF 1.2+ encode thành `OFPAT_SET_FIELD` với MFF_TCP_DST / MFF_UDP_DST / MFF_SCTP_DST tuỳ context (parser inspect `nw_proto` match để chọn). Ở tầng OVS internal: cùng `struct ofpact_l4_port` với `mod_tp_src` (chia sẻ struct), phân biệt qua `ofpact_type = OFPACT_SET_L4_DST_PORT`. Ở tầng datapath kernel: translate thành `OVS_ACTION_ATTR_SET` với `OVS_KEY_ATTR_TCP` / `_UDP` / `_SCTP` payload set dst port.

**Vai trò.** Vai trò chính: port forwarding (DNAT-with-port-translation) — expose internal service trên port khác với public port. Vai trò phụ: load balancer L4 (rewrite VIP:80 → backend:8080 khi backend server lắng nghe trên port khác), service redirection (HTTP 80 → HAProxy 8080 cho debug). Phạm vi: per-packet, L4 checksum auto-recompute.

**Vì sao sinh ra.** Port forwarding scenario phổ biến: server internal lắng nghe port non-standard (8080 cho dev) nhưng expose qua public port standard (80) — không thể di chuyển server bind. Action `mod_tp_dst` cho phép gateway rewrite port dst trước khi forward. Trước OpenFlow, Linux netfilter `iptables -t nat -A PREROUTING -p tcp --dport 80 -j REDIRECT --to-port 8080` làm việc này. OpenFlow expose programmable.

**Vấn đề giải quyết.** Cụ thể: (a) port forwarding cơ bản, (b) load balancer L4 với backend port khác frontend port, (c) service redirection scenario debug. Cách giải: action rewrite L4 dst port + checksum recompute. Phạm vi: scenario stateless 1-to-1 mapping; LB stateful với backend pool cần kết hợp với `select` group + `ct(commit, nat(dst=...))` đầy đủ.

Engineer must-know: cú pháp đối xứng `mod_tp_src`, hiểu pair với `mod_nw_dst` (rewrite IP dst) khi backend khác subnet, biết action chỉ valid cho packet có L4 protocol.

### 4.9.33.10 `push_vlan` — đẩy thêm VLAN tag vào packet với ethertype chỉ định

**Khái niệm.** `push_vlan` là action của OF 1.1+ để đẩy thêm một VLAN tag (4 byte 802.1Q tag hoặc 802.1ad tag) vào packet ngay sau ethernet header. Action chỉ chèn tag với TPID (Tag Protocol IDentifier, ethertype) chỉ định, không set giá trị `vlan_vid`/`vlan_pcp`/`cfi` — engineer phải dùng action `set_field` (hoặc `mod_vlan_vid`/`mod_vlan_pcp` legacy) sau đó để gán giá trị thực. TPID hợp lệ: `0x8100` (802.1Q standard VLAN), `0x88a8` (802.1ad Q-in-Q outer tag). Loại action: **header push**, OF 1.1+ raw type `OFPAT11_PUSH_VLAN`.

**Lịch sử + bối cảnh.** OpenFlow 1.0 không có `push_vlan`/`pop_vlan` riêng; thay vào đó có duy nhất `mod_vlan_vid` + `mod_vlan_pcp` (rewrite tag hiện có) + `strip_vlan` (xoá tag). OpenFlow 1.1 (2011) refactor: tách thành `OFPAT11_PUSH_VLAN` (action type 17) + `OFPAT11_POP_VLAN` (action type 18) + `OFPAT11_SET_VLAN_VID` legacy. OVS 1.5 (2013) hỗ trợ. OF 1.3+ chuẩn hoá `push_vlan` với TPID parameter để hỗ trợ Q-in-Q (802.1ad outer tag). Verify: `lib/ofp-actions.c` function `parse_PUSH_VLAN` line ~3037 + `encode_PUSH_VLAN` line ~3028 (OVS v3.3.0).

**Vị trí trong kiến trúc.** Ở tầng OpenFlow protocol: `OFPAT11_PUSH_VLAN` action type 17 ở OF 1.1+, action header gồm `type` 2 byte + `length` 2 byte + `ethertype` 2 byte + 2 byte padding. Ở tầng OVS internal: `struct ofpact_push_vlan` (`include/openvswitch/ofp-actions.h`) chứa `ovs_be16 ethertype`. Ở tầng datapath kernel: translate thành `OVS_ACTION_ATTR_PUSH_VLAN` netlink attribute với struct `ovs_action_push_vlan { __be16 vlan_tpid; __be16 vlan_tci; }`.

**Vai trò.** Vai trò chính: VLAN tagging packet untagged trước khi forward ra trunk port. Vai trò phụ: Q-in-Q (double-tagged) outer tag cho service provider scenario, VLAN translation (combo `pop_vlan` + `push_vlan` đổi VID), VPN MPLS-in-VLAN tunnel encap. Phạm vi: per-packet, sau action packet length tăng thêm 4 byte.

**Vì sao sinh ra.** OpenFlow 1.0 chỉ có `mod_vlan_vid` (rewrite tag hiện có) + `strip_vlan` (xoá tag). Pattern "untagged → tagged" cần workaround: tự đẩy raw bytes hoặc dùng NXM extension. OF 1.1 thêm explicit `push_vlan` để chuẩn hoá scenario này; OF 1.3+ thêm TPID parameter để hỗ trợ Q-in-Q (802.1ad outer tag với TPID `0x88a8`). Một động cơ riêng: SDN trong DC cloud cần tagging dynamic per-tenant (ví dụ OVN trunk port), không thể assume packet đã có tag.

**Vấn đề giải quyết.** Cụ thể: (a) tagging packet untagged khi forward ra trunk, (b) Q-in-Q double tag cho service provider, (c) VLAN translation atomic (tránh state intermediate khi pop + set). Cách giải: action insert 4 byte tag ngay sau MAC header, ethertype chỉ định + vlan_tci field 2 byte sẽ được set qua action sau (thường `set_field:V->vlan_vid`). Phạm vi: scenario L2 trunking, Q-in-Q, VLAN-based tenant isolation.

**Tầm quan trọng.** Tier 2 medium. Trong OVN trunk port + Logical Switch với localnet (tenant traffic ra physical network có VLAN), `push_vlan` xuất hiện ở Logical_Flow stage `ls_out_port_sec_l2`. Engineer must-know: cú pháp `push_vlan:0x8100` (chỉ TPID, không có VID/PCP), pair với `set_field:<vid>->vlan_vid` ngay sau, hiểu rằng action chỉ valid cho packet untagged hoặc cho outer tag scenario Q-in-Q.

**Cơ chế hoạt động.** Parser `lib/ofp-actions.c` function `parse_PUSH_VLAN` (line ~3037 v3.3.0): parse argument string thành 16-bit ethertype qua `str_to_u16`, validate ethertype ∈ `{0x8100, 0x88a8}`, lưu vào `ofpact_put_PUSH_VLAN(pp->ofpacts)->ethertype`. Format `format_PUSH_VLAN` (line ~3050) in `push_vlan:0x%"PRIx16` với ethertype hex. Encode `encode_PUSH_VLAN` (line ~3028) emit `OFPAT11_PUSH_VLAN` action với 2 byte ethertype + 2 byte pad. Datapath kernel: `OVS_ACTION_ATTR_PUSH_VLAN` netlink attribute → kernel insert 4 byte tag (TPID 2 byte + TCI 2 byte với VID=0/PCP=0 default) ngay sau MAC header, shift payload + recompute checksum nếu protocol stack cần. Sau action, `vlan_tci` field của flow context có giá trị 0 (cần action `set_field:V->vlan_vid` để set thực).

**Wire encoding.** OF 1.1+ format: action header `type=17` (OFPAT11_PUSH_VLAN) + `length=8` + `ethertype` 2 byte + 2 byte padding (align 8 byte). Datapath kernel netlink attribute `OVS_ACTION_ATTR_PUSH_VLAN`: `nlattr` header (type=20, length=8) + `struct ovs_action_push_vlan` 4 byte (vlan_tpid 2 + vlan_tci 2). Trên packet wire-level, sau action insert 4 byte tag (TPID 2 + TCI 2) giữa MAC header (14 byte) và payload — tổng frame size tăng 4 byte.

**Ví dụ áp dụng.**

```
actions=push_vlan:0x8100,set_field:100->vlan_vid,output:trunk            # tag VLAN 100 standard
actions=push_vlan:0x8100,set_field:4096->vlan_vid,output:trunk           # CFI=1 (VID=0 với CFI bit)
actions=push_vlan:0x88a8,set_field:200->vlan_vid,
        push_vlan:0x8100,set_field:100->vlan_vid,output:provider          # Q-in-Q outer 200 inner 100
actions=pop_vlan,push_vlan:0x8100,set_field:300->vlan_vid,output:trunk   # VLAN translation 100→300
priority=100,vlan_tci=0,actions=push_vlan:0x8100,
        set_field:42->vlan_vid,set_field:5->vlan_pcp,output:uplink        # tag với QoS priority 5
```

OVN-generated flow điển hình ở Logical_Flow stage `ls_out_port_sec_l2` cho trunk port với localnet tag:

```
priority=100,reg14=0x3,actions=push_vlan:0x8100,set_field:0x42->vlan_vid,output:patch-localnet
```

**Cách kỹ sư vận hành thành thạo.** Bốn kỹ năng: (1) **Đọc**: nhận ra pattern "push_vlan + set_field vlan_vid" luôn đi cặp (push xong phải set VID); phân biệt `0x8100` (standard) với `0x88a8` (Q-in-Q outer). (2) **Sửa**: viết flow trunk-tagging luôn pair `push_vlan` + `set_field:V->vlan_vid` (+ optional `set_field:P->vlan_pcp` cho QoS). Cho Q-in-Q viết 2 cặp push+set theo thứ tự inner trước outer. (3) **Quan sát**: `ovs-ofctl dump-flows br0 | grep push_vlan` đếm rule tagging; `tcpdump -i trunk -n -e` xem 4 byte tag wire-level. (4) **Debug**: `ofproto/trace 'in_port=1,...'` show line `Datapath actions: push_vlan(vid=100,pcp=0),N` xác nhận datapath insert tag.

**Phân loại.** Action loại header push (insert byte vào packet, làm tăng length), OF 1.1+ standard. Đặc tính: stateful (modify packet structure), not terminator, must pair với `set_field` cho VID. Cùng class: `pop_vlan` (đối xứng remove tag), `push_mpls`/`pop_mpls` (MPLS label), `push_pbb`/`pop_pbb` (PBB B-tag). Khác với `mod_vlan_vid`: `mod_vlan_vid` rewrite tag hiện có (không thay đổi length), `push_vlan` insert tag mới (tăng length 4 byte).

**Source code.** `lib/ofp-actions.c` function anchor (OVS v2.17.9): `parse_PUSH_VLAN` line ~3037, `encode_PUSH_VLAN` line ~3028, `format_PUSH_VLAN` line ~3050. Header `include/openvswitch/ofp-actions.h` `struct ofpact_push_vlan`. Datapath kernel `datapath/actions.c` function `push_vlan` xử lý byte insertion + skb metadata update; netlink protocol `include/uapi/linux/openvswitch.h` enum `OVS_ACTION_ATTR_PUSH_VLAN` + `struct ovs_action_push_vlan`. Branch baseline: OVS v2.17.9.

**Kịch bản bẻ gãy.** (a) `push_vlan:0x8100` mà không có `set_field:V->vlan_vid` ngay sau → tag chèn vào với VID=0 (priority-tag only) → host nhận drop frame ở filter L2 hoặc bridge xử lý sai. Fix: luôn pair `push_vlan` + `set_field:V->vlan_vid`. (b) `push_vlan:0x1234` với ethertype không hợp lệ → parser reject với error `push_vlan: ethertype must be 0x8100 or 0x88a8`. Fix: chọn 1 trong 2 giá trị. (c) Q-in-Q viết sai thứ tự (outer trước inner) → packet wire-level có thứ tự tag đảo → switch downstream parse sai outer/inner. Fix: viết theo thứ tự inner push trước (làm closest-to-payload), outer push sau (làm closest-to-MAC-header).

**Bài tập synthetic.** Mininet 2 host h1 + h2 qua bridge `s1` với h2 ở trunk port. Cài rule `priority=100,in_port=h1-eth0,vlan_tci=0,actions=push_vlan:0x8100,set_field:42->vlan_vid,output:h2-eth0`. Trên h2 chạy `tcpdump -i h2-eth0 -n -e -v` rồi `h1 ping h2`. Verify packet trên h2 wire-level có 4 byte tag `0x8100 0x002a` (TPID 0x8100 + TCI VID=42=0x2a) chèn ngay sau MAC. Frame size tăng 4 byte so với original.

**Cách quan sát + xác minh.** `ovs-ofctl dump-flows br0 -O OpenFlow13` show `push_vlan:0x8100`. `ovs-appctl ofproto/trace br0 'in_port=1,vlan_tci=0,...'` show line `Datapath actions: push_vlan(tpid=0x8100,vid=0,pcp=0),set(vlan(vid=42,...)),N`. `tcpdump -i trunk -n -e -v` verify wire-level frame có ` 802.1Q vlan 42` annotation; `tshark -V -i trunk` show full TCI 16-bit breakdown (VID 12 bit + PCP 3 bit + DEI 1 bit).

**So sánh với hệ khác.** Linux bridge `bridge vlan add dev eth0 vid 42 pvid untagged` set bridge port để tự push tag VID 42 cho mọi packet untagged đi vào — concept tương đương nhưng không expose action programmable (config-time setup). `tc filter ... action vlan push id 42` ở `tc` cho phép programmable equivalent với `push_vlan` của OpenFlow. Cisco IOS `switchport trunk allowed vlan 42` + interface mode trunk/access cũng config-time. P4 `hdr.vlan.setValid(); hdr.vlan.tpid = 0x8100; hdr.vlan.vid = 42;` direct trong control block.

### 4.9.33.11 `pop_vlan` — bóc VLAN tag ngoài cùng khỏi packet

**Khái niệm.** `pop_vlan` là action OF 1.1+ bóc tag 802.1Q/802.1ad ngoài cùng (outermost) khỏi packet ngay sau ethernet header. Sau action, packet ngắn đi đúng 4 byte (TPID 2 byte + TCI 2 byte) và `vlan_tci` field của flow context được reset về 0. Action không nhận argument — chỉ luôn bóc một tag duy nhất ở vị trí outermost; muốn bóc thêm tag bên trong (inner Q-in-Q) phải gọi action lần thứ hai. Loại action: **header pop**, OF 1.1+ raw type `OFPAT_RAW11_POP_VLAN`.

**Lịch sử + bối cảnh.** OpenFlow 1.0 (2009) chỉ có `strip_vlan` (xoá tag duy nhất, không hỗ trợ Q-in-Q layered pop). OpenFlow 1.1 (2011) refactor thành cặp đối xứng `OFPAT_RAW11_POP_VLAN` (action type 18) + `OFPAT_RAW11_PUSH_VLAN` (action type 17), trong đó `pop_vlan` luôn bóc outermost tag, support layered Q-in-Q. OVS 1.5 (2013) hỗ trợ; OVS giữ legacy alias `strip_vlan` parser cho rule OF 1.0 nhưng xuất output mặc định format `pop_vlan` ở OF 1.1+. Verify upstream: `lib/ofp-actions.c` enum `OFPAT_RAW11_POP_VLAN` line ~388, function `parse_POP_VLAN` line ~1700, `encode_POP_VLAN` line ~1600 (OVS v3.3.0).

**Vị trí trong kiến trúc.** Ở tầng OpenFlow protocol: `OFPAT_RAW11_POP_VLAN` action type 18, action header gồm `type` 2 byte + `length` 8 byte + 4 byte padding (không có argument body). Ở tầng OVS internal: `struct ofpact_null` (cùng struct với mọi argumentless action như `pop_pbb`, `dec_ttl`) trong `include/openvswitch/ofp-actions.h` — chỉ chứa header `struct ofpact` chung, không có field riêng. Ở tầng datapath kernel: translate thành `OVS_ACTION_ATTR_POP_VLAN` netlink attribute không có payload (length=4 byte header only); nếu kernel ≥ 5.0 + OVS module recompile, action xử lý in-place trên skb qua function `pop_vlan` ở `datapath/actions.c`.

**Vai trò.** Vai trò chính: bóc tag khỏi packet để forward ra access port (untagged interface). Vai trò phụ: VLAN translation cặp `pop_vlan` + `push_vlan:0x8100,set_field:V->vlan_vid` đổi VID, Q-in-Q de-encapsulation outer tag (giữ inner tag), tunnel termination cho service provider scenario. Phạm vi: per-packet, sau action packet length giảm 4 byte; chỉ áp dụng khi `vlan_tci` field hiện tại non-zero — nếu packet không có tag, action thành no-op nhưng parser vẫn accept.

**Vì sao sinh ra.** OpenFlow 1.0 `strip_vlan` chỉ xử lý tag duy nhất, không phân biệt outer/inner cho Q-in-Q. Khi 802.1ad (Q-in-Q) trở nên phổ biến trong service provider DC + carrier Ethernet, spec OF 1.1 cần action layered pop để bóc từng tag một theo thứ tự. Một động cơ thứ hai: cặp đối xứng `push_vlan` + `pop_vlan` về mặt programming model dễ hiểu hơn cho controller; trước đây `strip_vlan` không có counterpart explicit (tag được tạo gián tiếp qua `mod_vlan_vid` trên packet untagged).

**Vấn đề giải quyết.** Cụ thể: (a) bóc outermost tag để forward ra access port, (b) Q-in-Q de-encapsulation theo từng layer, (c) VLAN translation atomic kết hợp với `push_vlan`. Cách giải: action chỉ định "bóc outermost tag" — không cần argument vì luôn rõ vị trí (4 byte ngay sau MAC header); spec quy ước nếu packet untagged thì action no-op. Phạm vi: scenario L2 trunk-to-access conversion, Q-in-Q tunnel egress, VLAN-based tenant isolation.

**Tầm quan trọng.** Tier 2 medium. Trong OVN, `pop_vlan` xuất hiện ở Logical_Flow stage `ls_in_port_sec_l2` cho localnet ingress (tenant traffic từ physical network có VLAN vào logical switch cần bóc tag để consume tự logical context). Trong DC tenant isolation pattern, mọi rule "trunk → access port" đều có `pop_vlan` ngay trước `output`. Engineer must-know: cú pháp `pop_vlan` không có argument, hành vi no-op khi packet untagged, sau action `vlan_tci` reset về 0, và biết rằng action chỉ bóc outermost — Q-in-Q cần 2 lần.

**Cơ chế hoạt động.** Parser `lib/ofp-actions.c` function `parse_POP_VLAN` (line ~1700 v3.3.0): consume keyword `pop_vlan` (hoặc legacy alias `strip_vlan`) không nhận argument, gọi `ofpact_put_STRIP_VLAN` (alias internal) emit `struct ofpact` rỗng vào ofpacts buffer. Format `format_POP_VLAN` in `pop_vlan` không có suffix. Encode `encode_POP_VLAN` (line ~1600) emit `OFPAT_RAW11_POP_VLAN` action header 8 byte (4 byte type+length + 4 byte padding). Datapath kernel: `OVS_ACTION_ATTR_POP_VLAN` netlink attribute → kernel function `pop_vlan` ở `datapath/actions.c`: kiểm tra `skb_vlan_tag_present(skb)` hoặc parse 802.1Q ethertype, nếu có tag thì memmove MAC header dịch lên 4 byte để overwrite tag, decrement skb length, recompute `skb->protocol` từ ethertype mới, clear `skb->vlan_tci`/`skb->vlan_proto`. Nếu untagged, return 0 silently.

**Wire encoding.** OF 1.1+ format: action header `type=18` (OFPAT_RAW11_POP_VLAN) + `length=8` + 4 byte padding (align 8 byte, không có argument body). Datapath kernel netlink attribute `OVS_ACTION_ATTR_POP_VLAN`: chỉ `nlattr` header (type=21, length=4) không payload. Trên packet wire-level, sau action 4 byte tag (TPID 2 + TCI 2) bị remove khỏi vị trí ngay sau MAC header, phần payload dịch lên — tổng frame size giảm 4 byte.

**Ví dụ áp dụng.**

```
actions=pop_vlan,output:access                                       # bóc tag rồi gửi ra access port
priority=100,vlan_vid=42,actions=pop_vlan,output:5                   # match VID 42 rồi bóc + output
actions=pop_vlan,push_vlan:0x8100,set_field:300->vlan_vid,output:5   # VLAN translation 100→300
actions=pop_vlan,pop_vlan,output:6                                   # bóc 2 tag (Q-in-Q outer + inner)
priority=200,dl_type=0x88a8,actions=pop_vlan,resubmit(,2)            # bóc outer Q-in-Q, đi tiếp table 2
```

OVN-generated flow điển hình ở Logical_Flow stage `ls_in_port_sec_l2` cho localnet ingress với VLAN tag:

```
priority=150,inport==localnet,vlan_vid=0x42,actions=pop_vlan,resubmit(,17)
```

**Cách kỹ sư vận hành thành thạo.** Bốn kỹ năng: (1) **Đọc**: phân biệt `pop_vlan` (OF 1.1+, không argument) với `strip_vlan` (alias OF 1.0, parser OVS accept cả hai); nhận ra pattern "trunk→access" luôn có `pop_vlan` trước `output`. (2) **Sửa**: viết rule egress access port luôn có `pop_vlan` đầu; cho VLAN translation viết cặp `pop_vlan,push_vlan:0x8100,set_field:V->vlan_vid` theo thứ tự đó. (3) **Quan sát**: `ovs-ofctl dump-flows br0 | grep -E 'pop_vlan|strip_vlan'` đếm rule de-tagging. (4) **Debug**: `ofproto/trace 'in_port=trunk,dl_vlan=42,...'` show line `Datapath actions: pop_vlan,N` xác nhận datapath bóc tag; nếu show `set(vlan(vid=...))` thay vì `pop_vlan`, kernel cũ dùng đường set TCI thay vì pop trực tiếp.

**Phân loại.** Action loại header pop (remove byte khỏi packet, làm giảm length), OF 1.1+ standard. Đặc tính: stateless (không phụ thuộc state ngoài packet), not terminator, không pair bắt buộc với action khác (khác `push_vlan` luôn pair với `set_field`). Cùng class: `pop_mpls` (đối xứng `push_mpls`), `pop_pbb` (đối xứng `push_pbb`), generic OF 1.5 `decap`. Khác với `strip_vlan`: `strip_vlan` là alias OF 1.0 chỉ bóc tag duy nhất, không phân biệt outer/inner; `pop_vlan` OF 1.1+ chuẩn layered pop cho Q-in-Q.

**Source code.** `lib/ofp-actions.c` function anchor (OVS v2.17.9): `parse_POP_VLAN` line ~1700, `encode_POP_VLAN` line ~1600, `format_POP_VLAN` line ~1750; enum `OFPAT_RAW11_POP_VLAN` line 388 trong `enum ofp_raw_action_type`. Header `include/openvswitch/ofp-actions.h` `struct ofpact_null` (shared cho mọi argumentless action). Datapath kernel `datapath/actions.c` function `pop_vlan` xử lý byte removal + skb metadata cleanup; netlink protocol `include/uapi/linux/openvswitch.h` enum `OVS_ACTION_ATTR_POP_VLAN`. Branch baseline: OVS v2.17.9.

**Kịch bản bẻ gãy.** (a) `pop_vlan` trên packet untagged → action no-op, không error nhưng counter không tăng theo expectation; engineer tưởng rule không match, debug lâu. Fix: match `vlan_tci=0x1000/0x1000` (CFI bit set) hoặc `vlan_vid` explicit để chỉ trigger khi có tag. (b) Q-in-Q outer tag bóc xong nhưng quên bóc inner → packet ra access port vẫn có 1 tag, host drop frame. Fix: viết `pop_vlan,pop_vlan` tuần tự cho 2 layer. (c) `pop_vlan` sau khi đã `set_field:V->vlan_vid` → tag mới set bị bóc luôn, ngược intent. Fix: kiểm thứ tự action — `pop_vlan` luôn đứng trước action set/push, hoặc dùng `set_field` chỉ khi đã có tag.

**Bài tập synthetic.** Mininet 2 host h1 (trunk, gửi packet VLAN 42) + h2 (access). Cài rule `priority=100,in_port=h1-eth0,vlan_vid=42,actions=pop_vlan,output:h2-eth0`. Trên h1 chạy `vconfig add h1-eth0 42; ifconfig h1-eth0.42 10.0.0.1/24 up; ping 10.0.0.2`. Trên h2 chạy `tcpdump -i h2-eth0 -n -e` verify packet không có tag (chỉ 14 byte MAC header trước IP). Frame size giảm 4 byte so với original.

**Cách quan sát + xác minh.** `ovs-ofctl dump-flows br0 -O OpenFlow13` show `pop_vlan` (không suffix); OF 1.0 dump show `strip_vlan` legacy. `ovs-appctl ofproto/trace br0 'in_port=1,dl_vlan=42,...'` show line `Datapath actions: pop_vlan,N` xác nhận. `tcpdump -i access-port -n -e -v` verify wire-level frame không có ` 802.1Q vlan` annotation. `ovs-appctl dpif/show-flows | grep pop_vlan` đếm số megaflow đã hit pop_vlan.

**So sánh với hệ khác.** Linux bridge `bridge vlan add dev eth0 vid 42 untagged` set port làm bóc tag VID 42 khi forward — config-time setup, không programmable per-flow. `tc filter ... action vlan pop` ở `tc` cho phép programmable equivalent với `pop_vlan`. Cisco IOS `switchport mode access` + `switchport access vlan 42` config-time bóc tag khi packet ra access. P4 `hdr.vlan.setInvalid();` direct trong control block. NSX-T transport node de-encapsulate VLAN ở edge node tương đương concept.

### 4.9.33.12 `learn` — cài rule mới vào table N từ packet hiện tại

**Khái niệm.** `learn` là NXM (Nicira) action mạnh nhất: từ packet đang xử lý, sinh và cài một flow rule mới vào table chỉ định, với match field + action lấy từ giá trị các field của packet hiện tại. Action không thay đổi packet đang xử lý — nó chỉ side-effect cài rule cho packet tương lai. Đây là cơ chế cho phép data-plane self-program không cần controller round-trip; tạo ra hành vi tương đương MAC learning, ARP cache, flow caching, hoặc connection tracking đơn giản. Loại action: **stateful side-effect**, NXM Nicira extension OVS 1.6+, không terminator.

**Lịch sử + bối cảnh.** Action xuất hiện trong OVS 1.6 (2012) như NXM extension `NXAST_LEARN` để OVS-MAC-learning rule có thể tự cài rule reverse-direction từ ARP/IP packet — tạo hành vi tương tự `bridge` Linux nhưng có thể customize qua flow rule. OVS 2.4 (2015) thêm `learn(...)` syntax đầy đủ cho `ovs-ofctl`. OpenFlow 1.5 (2014) không chuẩn hoá `learn` — vẫn là NXM extension; controller phải opt-in qua experimenter ID Nicira. Verify upstream: `lib/ofp-actions.c` enum `NXAST_RAW_LEARN` line 559, function `parse_LEARN` line ~3800, `encode_LEARN` line ~3700 (OVS v3.3.0).

**Vị trí trong kiến trúc.** Ở tầng OpenFlow protocol: `NXAST_RAW_LEARN` action type Nicira experimenter (subtype 16), action header gồm Nicira experimenter wrapper + biến độ dài body chứa list `flow_mod_spec` thay phiên: 2 byte spec header + N byte src/dst spec. Ở tầng OVS internal: `struct ofpact_learn` (`include/openvswitch/ofp-actions.h`) chứa `idle_timeout`, `hard_timeout`, `priority`, `cookie`, `flags`, `table_id`, `fin_idle_timeout`, `fin_hard_timeout`, `result_dst`, `n_specs`, mảng `struct ofpact_learn_spec` flexible array. Ở tầng datapath: kernel KHÔNG xử lý `learn` — translate (xlate.c) phải execute action ngay tại userspace ofproto-dpif khi upcall, sinh flow_mod nội bộ, gửi vào ofproto add-flow pipeline.

**Vai trò.** Vai trò chính: MAC learning data-plane tự động (cài rule reverse direction sau khi thấy frame source MAC). Vai trò phụ: ARP cache rule, connection learning kiểu stateful firewall đơn giản, flow distribution learning theo L4 hash, dynamic ACL hit-counter. Phạm vi: cài 1 rule mới vào table chỉ định mỗi lần action execute — tần suất execute cao nghĩa là tần suất cài rule cao, có thể bùng nổ table size.

**Vì sao sinh ra.** Trước OVS 1.6, MAC learning phải implement ở controller: switch upcall mọi packet không match → controller xem source MAC → controller flow_mod cài rule. Round-trip latency 1-10 ms × triệu MAC = không scale. `learn` action đẩy logic xuống switch: switch tự nhìn packet, tự cài rule. Một động cơ thứ hai: cho phép viết firewall stateful đơn giản trong rule ngôn ngữ — packet inbound trigger `learn` cài rule outbound match reverse 5-tuple, không cần conntrack đầy đủ.

**Vấn đề giải quyết.** Cụ thể: (a) MAC learning không cần controller, (b) ARP cache tự động, (c) reverse-flow cho stateful firewall đơn giản, (d) anti-spoofing dynamic. Cách giải: action định nghĩa template rule (match + action) trong syntax `learn(table=N, priority=P, idle_timeout=T, NXM_OF_ETH_DST[]=NXM_OF_ETH_SRC[], output:NXM_OF_IN_PORT[])` — phần `[]` là load-from-current-packet. Khi action execute, switch substitute giá trị field từ packet hiện tại vào template, sinh rule cụ thể, cài vào table chỉ định. Phạm vi: data-plane self-programming, OVS-only.

**Tầm quan trọng.** Tier 2 medium. Trong OVS standalone deployment (không controller), `learn` là cơ chế chính để switch behave như Linux bridge với MAC learning. Trong OVN, `learn` xuất hiện ở Logical_Flow stage `ls_in_l2_lkup` cho FDB MAC learning khi `mac_learn` enabled trong NB Logical_Switch options. Engineer must-know: syntax `learn(table=T, priority=P, idle_timeout=T, MATCH_SPEC, ACTION_SPEC)` + flag `delete_learned` (xoá rule khi parent rule bị xoá), `fin_idle_timeout` (rút ngắn timeout khi thấy TCP FIN), `result_dst` (lưu cookie rule mới sinh vào register).

**Cơ chế hoạt động.** Parser `lib/ofp-actions.c` function `parse_LEARN` (line ~3800 v3.3.0): tokenize string trong dấu `(...)`, mỗi spec trong dạng `key=value` parse riêng — `table`, `priority`, `idle_timeout`, `hard_timeout`, `cookie`, `delete_learned`, `fin_idle_timeout`, `fin_hard_timeout`, `result_dst`, hoặc match-spec dạng `field[]=value` / `field[]=field2[]` / `load:value->field[]`. Mỗi spec emit `struct ofpact_learn_spec` với field src/dst encoded. Encode `encode_LEARN` (line ~3700) emit Nicira experimenter wrapper + serialize specs. Translation downstream (xlate.c function `xlate_learn_action`): khi upcall chạm rule có action learn, xlate substitute field từ flow context vào template, sinh `struct ofputil_flow_mod`, gửi vào ofproto-dpif add-flow pipeline. Datapath kernel KHÔNG thấy `learn` — chỉ thấy rule mới đã cài (qua flow_mod), execute như rule bình thường.

**Wire encoding.** NXM Nicira experimenter format: action header `type=NXAST` (65535) + `length=variable` + 4 byte experimenter ID `0x00002320` (Nicira) + 2 byte subtype `16` (NXAST_LEARN) + body chứa fixed header (idle_timeout, hard_timeout, priority, cookie, flags, table_id, ...) + variable-length spec list, mỗi spec 2 byte header (src_type:1 + dst_type:1 + n_bits:6) + biến độ dài src/dst payload. Tổng action có thể dài 50-200 byte tuỳ số spec. Datapath kernel: không có wire format vì kernel không xử lý.

**Ví dụ áp dụng.**

```
# MAC learning đơn giản: từ source MAC + in_port, cài rule reverse direction
actions=learn(table=10,idle_timeout=300,
              NXM_OF_ETH_DST[]=NXM_OF_ETH_SRC[],
              output:NXM_OF_IN_PORT[]),
        resubmit(,10)

# Stateful reverse-flow cho TCP: cài rule outbound match reverse 5-tuple
actions=learn(table=20,priority=100,idle_timeout=60,fin_idle_timeout=5,
              eth_type=0x0800,nw_proto=6,
              NXM_OF_IP_SRC[]=NXM_OF_IP_DST[],
              NXM_OF_IP_DST[]=NXM_OF_IP_SRC[],
              NXM_OF_TCP_SRC[]=NXM_OF_TCP_DST[],
              NXM_OF_TCP_DST[]=NXM_OF_TCP_SRC[],
              output:NXM_OF_IN_PORT[]),
        normal

# OVN-style L2 lookup learning với delete_learned (xoá rule khi parent xoá)
actions=learn(table=11,idle_timeout=0,delete_learned,
              NXM_OF_ETH_DST[]=NXM_OF_ETH_SRC[],
              load:NXM_OF_IN_PORT[]->NXM_NX_REG14[0..15])
```

**Cách kỹ sư vận hành thành thạo.** Bốn kỹ năng: (1) **Đọc**: phân biệt cú pháp `field[]=field2[]` (copy from current packet) với `field[]=value` (literal) và `load:value->field[]` (action set trên rule mới sinh); biết rằng comma trong `learn(...)` là separator spec, không phải action separator. (2) **Sửa**: bắt đầu từ template MAC learning chuẩn, mở rộng bằng cách thêm match-spec cụ thể (dl_type, nw_proto). Luôn set `idle_timeout` để rule tự dọn; cân nhắc `delete_learned` để cleanup khi parent rule bị xoá. (3) **Quan sát**: `ovs-ofctl dump-flows br0 table=10` xem rule đã learn — số lượng tăng tuyến tính theo số distinct source MAC. `ovs-appctl coverage/show | grep learn` đếm `learn` action đã execute. (4) **Debug**: `ofproto/trace` không trigger `learn` (trace là dry-run); muốn test thực tế phải gửi packet thật + `dump-flows` xem rule mới sinh.

**Phân loại.** Action loại stateful side-effect (sinh rule mới, không thay đổi packet hiện tại), NXM Nicira extension OVS-only. Đặc tính: not terminator, idempotent ở packet level (nhiều lần execute trên packet identical sinh rule giống nhau), nhưng có side-effect ở table state. Cùng class: `multipath` (NXM, chọn output theo hash, không side-effect), `bundle` (NXM, group selection), `conjunction` (NXM, multi-field match composition), `note` (NXM, no-op metadata). Khác với `controller(...)`: `controller` đẩy packet lên controller để controller flow_mod, có round-trip; `learn` cài rule local switch không round-trip.

**Source code.** `lib/ofp-actions.c` function anchor (OVS v2.17.9): `parse_LEARN` line ~3800, `encode_LEARN` line ~3700, `format_LEARN` line ~3850; enum `NXAST_RAW_LEARN` line 559 trong `enum ofp_raw_action_type`. Header `include/openvswitch/ofp-actions.h` `struct ofpact_learn` + `struct ofpact_learn_spec`. Translation `ofproto/ofproto-dpif-xlate.c` function `xlate_learn_action` (substitution + flow_mod emission). Datapath kernel: no handler, since `learn` runs entirely in userspace ofproto-dpif. Branch baseline: OVS v2.17.9.

**Kịch bản bẻ gãy.** (a) `learn(...)` không set `idle_timeout` + traffic distinct MAC cao → table N bùng nổ, `ovs-ofctl dump-flows table=N | wc -l` lên triệu rule, `ovs-vswitchd` consume RAM nhiều GB. Fix: luôn set `idle_timeout=300` (hoặc thấp hơn). (b) `learn(...)` viết match-spec bị thiếu (ví dụ chỉ match dl_dst, không match VLAN) → rule sinh ra match quá rộng, conflict rule khác. Fix: kiểm template match-spec đủ-độ-cụ-thể. (c) Rule parent có `learn(...)` bị `del-flow` nhưng rule đã learn không xoá → rule "mồ côi" tồn tại idle_timeout. Fix: dùng flag `delete_learned` để cascade delete khi parent xoá.

**Bài tập synthetic.** Mininet 1 OVS bridge + 4 host (h1-h4). Cài 2 rule: `table=0,priority=100,actions=learn(table=10,idle_timeout=300,NXM_OF_ETH_DST[]=NXM_OF_ETH_SRC[],output:NXM_OF_IN_PORT[]),resubmit(,10)` + `table=10,priority=0,actions=flood`. Ban đầu `dump-flows br0 table=10` chỉ có rule miss flood. `h1 ping h2` → rule reverse direction (dst=MAC(h1), out=port(h1)) xuất hiện trong table=10. Tiếp `h3 ping h1`, `h4 ping h1` → 3 rule trong table=10, mỗi rule match dst MAC khác. Verify counter `n_packets` rule learn = 0 nếu reply hit rule learned trực tiếp (không qua flood).

**Cách quan sát + xác minh.** `ovs-ofctl dump-flows br0 table=10 -O OpenFlow13` show rule đã learn — cookie thừa kế từ parent rule. `ovs-appctl coverage/show | grep -E 'learn|ofproto_flush'` đếm số `learn` execute + flush count. `ovs-appctl ofproto/trace br0 'in_port=1,...'` show "Action: learn(...)" trong action list nhưng không show rule mới (trace là dry-run). `ovs-vsctl --columns=other_config list Open_vSwitch | grep n-handler-threads` ảnh hưởng concurrency `learn` execute.

**So sánh với hệ khác.** Linux bridge `mac-learning` nội tại: bridge tự maintain FDB qua source MAC, không expose programmable. P4 register + table có thể implement `learn` qua extern + control plane callback (bộ digest). Cisco IOS MAC address-table dynamic là equivalent fixed-function. NSX-T MAC learning được expose qua Logical_Switch profile config-time. SR-IOV NIC với hardware MAC learning offload semantically tương đương nhưng không programmable per-rule.

### 4.9.33.13 `conjunction` — kết hợp đa chiều match qua N conjunct

**Khái niệm.** `conjunction(id, k/n)` là NXM action thay thế cho rule explosion khi cần match N-dimensional (ví dụ src_ip × dst_ip × port). Thay vì viết N×M×K rule explicit, engineer viết N + M + K rule, mỗi rule có action `conjunction(id, k/n)`; switch chỉ "match thực" khi packet hit ≥ 1 rule trong mỗi của N dimension cùng id. OpenFlow 1.5 (2014) chuẩn hoá thành `OFPACT_CONJUNCTION` nhưng OVS giữ wire format NXM gốc; semantic giống nhau. Loại action: **match composition pseudo-action**, NXM OVS 2.4+ / OF 1.5, không terminator.

**Lịch sử + bối cảnh.** Action xuất hiện trong OVS 2.4 (2015) như NXM extension `NXAST_CONJUNCTION` để giải quyết rule explosion ở SDN ACL controller (ví dụ OpenStack security group với 1000 IP × 100 port × 2 direction = 200000 rule). OF 1.5 (2014) định nghĩa concept conjunction nhưng OVS chọn giữ wire format NXM gốc cho backward compat. OVN từ ngày đầu (2015) dùng `conjunction` cho ACL match aggregation. Verify upstream: `lib/ofp-actions.c` enum `NXAST_RAW_CONJUNCTION` line 628, function `parse_CONJUNCTION` line ~4100, `encode_CONJUNCTION` line ~4050 (OVS v3.3.0).

**Vị trí trong kiến trúc.** Ở tầng OpenFlow protocol: `NXAST_RAW_CONJUNCTION` Nicira experimenter, action body chứa 4 byte `id` (uint32) + 1 byte `clause` (k, 1-indexed) + 1 byte `n_clauses` (n, 2-255) + padding. Ở tầng OVS internal: `struct ofpact_conjunction` (`include/openvswitch/ofp-actions.h`) chứa `id` + `clause` + `n_clauses`. Ở tầng classifier: `lib/classifier.c` xử lý conjunction qua "conjunctive flow matching" — duy trì bookkeeping per-id, kiểm tra số clause hit theo packet. Ở tầng datapath: kernel không thấy `conjunction` — conjunction được resolve hoàn toàn ở userspace classifier; chỉ rule "thực" (rule output thực sự) emit megaflow xuống kernel.

**Vai trò.** Vai trò chính: rule explosion reduction cho ACL multi-dimension match (security group, distributed firewall). Vai trò phụ: efficient port range match (port 1024-65535 viết 1 conjunct với N rule range thay vì N rule single-port), tag-based ACL (tag × port × direction). Phạm vi: classifier-level optimization, không thay đổi packet, không emit datapath action — tất cả các rule chứa `conjunction(id, k/n)` không bao giờ trực tiếp execute action khác trong action list (chỉ rule "thực" ngoài conjunction mới execute).

**Vì sao sinh ra.** Rule explosion là vấn đề classic của ACL: với N IP × M port × K direction, số rule = N×M×K, có thể lên triệu trong DC. Trước `conjunction`, controller phải pre-compute toàn bộ cross product, đẩy xuống switch — tốn RAM + CPU classifier lookup tăng tuyến tính. `conjunction` cho phép viết riêng N + M + K rule rồi để switch tự match composition — RAM giảm theo product → sum. Một động cơ thứ hai: OVN cần expressive ACL DSL (NB ACL match string `"ip4.src == 10.0.0.0/24 && tcp.dst == {80, 443, 8080}"`) compile xuống conjunction tự nhiên.

**Vấn đề giải quyết.** Cụ thể: (a) rule explosion N×M×K → N+M+K, (b) port range match efficient, (c) tag-based composable ACL. Cách giải: action không emit packet processing trực tiếp — nó là "vote" trong classifier composition. Khi rule có `conjunction(id, k/n)` match, classifier tăng counter clause k cho id; khi đủ n/n clause hit cho cùng packet + cùng id, classifier coi như "rule thực" id matched, rồi look up rule cụ thể (priority = priority của các rule conjunction + 1) để execute action thực sự. Phạm vi: classifier-level match composition, OVS-only optimization.

**Tầm quan trọng.** Tier 2 medium nhưng critical cho OVN production. Mọi ACL OVN (SB Logical_Flow stage `ls_in_acl` + `ls_out_acl`) compile thành `conjunction` nếu match có >= 2 dimension. Engineer must-know: cú pháp `conjunction(id, k/n)` với `id` 32-bit unique trong table, `k` 1-indexed clause number, `n` tổng số clause; n phải >= 2 (n=1 không có nghĩa). Engineer phải hiểu: rule chứa `conjunction` không "execute" output action — chỉ rule "thực" (cùng id, cùng priority offset) mới execute.

**Cơ chế hoạt động.** Parser `lib/ofp-actions.c` function `parse_CONJUNCTION` (line ~4100 v3.3.0): tokenize `conjunction(id, k/n)` argument list, parse `id` qua `str_to_u32`, parse `k/n` qua `sscanf("%u/%u")`, validate `n >= 2 && n <= 64 && k >= 1 && k <= n`, emit `struct ofpact_conjunction`. Format `format_CONJUNCTION` (line ~4150) in `conjunction(<id>,<k>/<n>)`. Encode `encode_CONJUNCTION` (line ~4050) emit Nicira experimenter wrapper + 8 byte body. Classifier `lib/classifier.c` function `classifier_lookup` xử lý conjunction qua structure `struct cls_conjunction_set` per priority level: với mỗi packet, scan rule có conjunction, group theo id, kiểm tra mỗi id có đủ n clause hit; nếu có, lookup rule "thực" priority+1 cùng id để execute.

**Wire encoding.** NXM Nicira experimenter format: action header `type=NXAST` (65535) + `length=24` + 4 byte experimenter ID `0x00002320` (Nicira) + 2 byte subtype `34` (NXAST_CONJUNCTION) + 6 byte padding + 4 byte `id` + 1 byte `clause` (1-based k) + 1 byte `n_clauses` (n, 2-64) + 2 byte padding. Datapath kernel: không có wire format — conjunction resolve hoàn toàn ở userspace classifier.

**Ví dụ áp dụng.**

```
# Match (src_ip ∈ {A, B, C}) AND (dst_ip ∈ {X, Y}) AND (tcp_dst ∈ {80, 443}):
# Thay vì 3×2×2 = 12 rule, viết 3+2+2 = 7 rule + 1 rule action thực.

# Dimension 1 (src_ip), 3 conjunct, all tag với conjunction(1, 1/3):
priority=100,ip,nw_src=10.0.0.1,actions=conjunction(1,1/3)
priority=100,ip,nw_src=10.0.0.2,actions=conjunction(1,1/3)
priority=100,ip,nw_src=10.0.0.3,actions=conjunction(1,1/3)

# Dimension 2 (dst_ip), 2 conjunct, all tag với conjunction(1, 2/3):
priority=100,ip,nw_dst=192.168.0.1,actions=conjunction(1,2/3)
priority=100,ip,nw_dst=192.168.0.2,actions=conjunction(1,2/3)

# Dimension 3 (tcp_dst), 2 conjunct, all tag với conjunction(1, 3/3):
priority=100,tcp,tp_dst=80,actions=conjunction(1,3/3)
priority=100,tcp,tp_dst=443,actions=conjunction(1,3/3)

# Rule thực (priority=101 cao hơn 1) action output:
priority=101,ip,actions=output:5
```

OVN-generated flow điển hình ở stage `ls_in_acl` cho ACL `ip4.src == 10.0.0.0/24 && tcp.dst == {80, 443}`:

```
priority=2002,ip,nw_src=10.0.0.0/24,actions=conjunction(0xa1,1/2)
priority=2002,tcp,tp_dst=80,actions=conjunction(0xa1,2/2)
priority=2002,tcp,tp_dst=443,actions=conjunction(0xa1,2/2)
priority=2003,ip,actions=resubmit(,18)
```

**Cách kỹ sư vận hành thành thạo.** Bốn kỹ năng: (1) **Đọc**: nhận ra pattern N rule cùng priority + cùng `conjunction(id, k/n)` là 1 dimension; rule thực action output ở priority+1; tổng số rule = sum dimension chứ không product. (2) **Sửa**: chia ACL match đa chiều thành N conjunct rule + 1 rule thực ở priority+1; chọn `id` unique trong table (thường lấy hash của ACL row UUID OVN). (3) **Quan sát**: `ovs-ofctl dump-flows br-int | grep conjunction` đếm rule conjunct, group theo id để biết bao nhiêu ACL composite. (4) **Debug**: `ofproto/trace 'in_port=1,...'` show line `Conjunction: id=0xa1, 2/2` xác nhận packet đã hit đủ clause; nếu chỉ thấy 1/2 nghĩa packet match dimension 1 nhưng không match dimension 2, debug tiếp dimension còn thiếu.

**Phân loại.** Action loại match composition pseudo-action (không xử lý packet trực tiếp), NXM OVS 2.4+ + OF 1.5 (semantic chuẩn hoá nhưng wire format vẫn NXM ở OVS). Đặc tính: special trong action list — không pair với action khác trong cùng rule (parser reject). Cùng class: `multipath` (group selection), `bundle` (port selection) — đều là pseudo-action ảnh hưởng rule lookup chứ không packet content. Khác với `goto_table`: `goto_table` là instruction OF 1.1+, dirige flow qua table; `conjunction` là tổng hợp match trong cùng table.

**Source code.** `lib/ofp-actions.c` function anchor (OVS v2.17.9): `parse_CONJUNCTION` line ~4100, `encode_CONJUNCTION` line ~4050, `format_CONJUNCTION` line ~4150; enum `NXAST_RAW_CONJUNCTION` line 628 trong `enum ofp_raw_action_type`. Header `include/openvswitch/ofp-actions.h` `struct ofpact_conjunction`. Classifier `lib/classifier.c` function `classifier_lookup` + helper `find_conjunctive_match` (resolve conjunction). OVN compile `northd/northd.c` function `consider_acl` emit conjunction khi ACL match có >= 2 dimension. Branch baseline: OVS v2.17.9.

**Kịch bản bẻ gãy.** (a) Quên rule "thực" priority+1 → packet hit đủ N clause nhưng không có rule output → packet drop silently. Fix: luôn pair N rule conjunct với 1 rule thực priority+1 (hoặc cấu trúc OVN dùng `resubmit`). (b) `conjunction(id, k/n)` `id` collision giữa 2 ACL khác nhau → packet match clause của ACL A + clause của ACL B cùng id → false match. Fix: chọn `id` unique trong table (e.g., hash UUID ACL). (c) `n_clauses=1` không hợp lệ → parser reject. Fix: với 1 dimension viết rule thường không cần conjunction.

**Bài tập synthetic.** Mininet 1 OVS bridge + 6 host (h1-h6, IP `10.0.0.1`-`10.0.0.6`). Cài: 3 rule conjunct dimension `nw_src` (h1, h2, h3) priority=100 conjunction(1,1/2), 2 rule conjunct dimension `tcp_dst` (80, 443) priority=100 conjunction(1,2/2), 1 rule thực priority=101 actions=output:h6-eth0. Test: `iperf3 -s` trên h6, từ h1 chạy `iperf3 -c h6 -p 80` → match (h1→80) hit rule. Từ h4 (không thuộc dimension 1) → không match. Từ h1 với port 22 → không match dimension 2. Verify số rule trong `dump-flows | wc -l` = 6 (3 + 2 + 1) thay vì 6 (3×2) trong viết explicit — cùng số ở case này nhưng ngoài đời với dimension lớn lợi ích bùng nổ.

**Cách quan sát + xác minh.** `ovs-ofctl dump-flows br0 -O OpenFlow15 | grep conjunction` show rule có `conjunction(<id>,<k>/<n>)` hậu tố. `ovs-appctl coverage/show | grep conjunction` đếm `conjunction_match` execute count. `ovs-appctl ofproto/trace br0 'in_port=1,ip,nw_src=10.0.0.1,...'` show line `Conjunction: id=1, clause 1/2 matched` per dimension hit; nếu trace show `clause 1/2 matched` rồi không tiếp `clause 2/2`, packet chỉ match 1 dimension, không full conjunction. `ovs-appctl dpif/show-flows` không show conjunction (kernel chỉ thấy megaflow của rule thực).

**So sánh với hệ khác.** Linux netfilter ipset (`ipset create allowed_src hash:ip; iptables -m set --match-set allowed_src src`) có concept tập hợp giảm rule, nhưng không compose đa dimension natively. tc cls_u32 hash table semantically tương tự. P4 không có conjunction primitive — phải implement qua nested table + action exit. Cisco IOS object-group ACL gần nhất concept (object-group network/service rồi match trong ACL line). NSX-T DFW rule với multi-criteria evaluate sequential, không tối ưu conjunction.

### 4.9.33.14 `ct()` — conntrack action với option đầy đủ (commit, zone, nat, helper)

**Khái niệm.** `ct(...)` là action gọi conntrack subsystem với một danh sách option phong phú điều khiển hành vi tracking + modification: `commit` (tạo entry trong conntrack table), `force` (override entry cũ), `table=N` (recirc về table N sau xử lý), `zone=Z` hoặc `zone=NXM_field[]` (logical zone isolation), `mark=V/M` (set conntrack metadata), `label=V/M` (set 128-bit label), `helper=name` (gọi ALG như `ftp`, `tftp`, `sip`), và `nat(...)` sub-action lồng (SNAT/DNAT translation). Đây là action phức tạp nhất OVS với sub-action lồng. Loại action: **stateful side-effect + modify**, NXM OVS 2.5+ / OF 1.5 partial, không terminator.

**Lịch sử + bối cảnh.** OVS 2.5 (2016) thêm conntrack integration với `ct(commit, zone=N)` cơ bản, dựa trên kernel netfilter conntrack module qua `nf_conntrack` API. OVS 2.6 (2016) thêm `nat(...)` sub-action support. OVS 2.7 (2017) thêm `helper=` cho ALG. OVS 2.8 (2018) thêm 128-bit `label`. OVS 2.13+ (2020) thêm `force` flag. OF 1.5 (2014) chuẩn hoá `ct` partial nhưng OVS giữ NXM extension cho option đầy đủ. Verify upstream: `lib/ofp-actions.c` enum `NXAST_RAW_CT` line 631, function `parse_CT` line ~4300, `encode_CT` line ~4200 (OVS v3.3.0).

**Vị trí trong kiến trúc.** Ở tầng OpenFlow protocol: `NXAST_RAW_CT` Nicira experimenter, action body chứa fixed header (`flags` 2 byte + `zone_src` 16-byte NXM ref + `zone_imm` 2 byte + `recirc_table` 1 byte + `alg` 2 byte + ...) + variable nested action list (cho `nat(...)` sub-action). Ở tầng OVS internal: `struct ofpact_conntrack` (`include/openvswitch/ofp-actions.h`) chứa `flags` (`NX_CT_F_COMMIT`, `NX_CT_F_FORCE`), `recirc_table`, `zone_src`/`zone_imm`, `alg`, mảng `actions` flexible array cho NAT sub-action. Ở tầng datapath kernel: translate thành `OVS_ACTION_ATTR_CT` netlink attribute với nested attributes (`OVS_CT_ATTR_COMMIT`, `OVS_CT_ATTR_ZONE`, `OVS_CT_ATTR_MARK`, `OVS_CT_ATTR_LABELS`, `OVS_CT_ATTR_HELPER`, `OVS_CT_ATTR_NAT`); kernel module `openvswitch.ko` gọi `nf_conntrack` framework trực tiếp.

**Vai trò.** Vai trò chính: stateful firewall (OVN ACL stateful, security group). Vai trò phụ: NAT/PAT (SNAT outbound, DNAT load balancer), ALG cho protocol có port phụ (FTP data channel, SIP), session tracking cho audit. Phạm vi: per-packet, action có side-effect ở conntrack table (kernel hoặc OVS userspace conntrack); sau action packet phải re-classify ở table chỉ định (`recirc_table`) vì `ct_state` field được populate.

**Vì sao sinh ra.** Stateful firewall cần biết "packet này thuộc connection nào, đang ở state nào (NEW/ESTABLISHED/RELATED/INVALID)". Trước OVS 2.5, OVS chỉ stateless: rule match L2-L4 header không state. Phải pre-OVS NAT bằng iptables hoặc Linux conntrack-tools — chained pipeline phức tạp, không programmable từ controller. `ct(commit)` action bridge OpenFlow rule với netfilter conntrack: rule trigger conntrack lookup/commit, packet được populate `ct_state`/`ct_zone`/`ct_mark`, rule sau có thể match. Một động cơ thứ hai: OVN cần stateful ACL + load balancer + NAT trong logical pipeline DSL, không thể phụ thuộc conntrack-tools độc lập.

**Vấn đề giải quyết.** Cụ thể: (a) stateful firewall (allow ESTABLISHED, drop INVALID), (b) NAT/PAT translation (SNAT outbound, DNAT load balancer), (c) ALG (FTP, SIP) tracking port phụ, (d) connection isolation per zone (multi-tenant). Cách giải: action ủy thác conntrack subsystem do/track lookup; flag + sub-action chỉ định behavior. Phạm vi: stateful pipeline OVS, OVN dùng làm primitive cho ACL stateful + LB + NAT.

**Tầm quan trọng.** Tier 2 medium nhưng critical cho OVN production. Mọi ACL stateful (NB ACL `action="allow-related"`) + Load_Balancer + NAT trong OVN compile thành rule chứa `ct(...)`. Trong stage OVN ingress: `ls_in_pre_acl` set zone, `ls_in_acl` evaluate ACL với ct_state, `ls_in_pre_lb` + `ls_in_lb` cho LB với DNAT, `ls_in_stateful` commit. Engineer must-know: option `commit` (tạo entry), `zone=Z` (isolation), `nat(src=ip:port)` / `nat(dst=ip:port)` (NAT), `helper=ftp` (ALG), `recirc_table=N` (re-classify); pattern recirc luôn cần — packet sau `ct` không thể continue cùng rule action list, phải về table N.

**Cơ chế hoạt động.** Parser `lib/ofp-actions.c` function `parse_CT` (line ~4300 v3.3.0): tokenize trong dấu `(...)`, mỗi sub-option parse riêng — `commit` set flag `NX_CT_F_COMMIT`, `force` set `NX_CT_F_FORCE`, `table=N` set `recirc_table`, `zone=NXM_field[]` parse NXM field reference vào `zone_src`, `zone=N` parse immediate vào `zone_imm`, `helper=name` parse string vào `alg` (ALG ID), `nat(...)` parse nested action list. Encode `encode_CT` (line ~4200) emit Nicira experimenter wrapper + serialize options. Translation `xlate.c` function `xlate_ct_action`: emit `OVS_ACTION_ATTR_CT` netlink attribute với nested. Datapath kernel module `openvswitch.ko` function `ovs_ct_execute`: gọi `nf_conntrack_in` để lookup/create entry, populate `ct_state` field trong skb metadata, nếu `commit` + `nat(...)` thì gọi `nf_nat_setup_info` để add NAT mapping; sau xử lý, packet recirc về `recirc_table` trong OVS pipeline.

**Wire encoding.** NXM Nicira experimenter format: action header `type=NXAST` (65535) + `length=variable` + 4 byte experimenter ID `0x00002320` + 2 byte subtype `36` (NXAST_CT) + body: 2 byte `flags`, 2 byte `zone_src` NXM header (0 nếu zone immediate), 2 byte `zone_imm`, 1 byte `recirc_table`, 1 byte `alg` + nested action list (cho NAT sub-action). Datapath kernel: `OVS_ACTION_ATTR_CT` netlink attribute với nested attributes `OVS_CT_ATTR_COMMIT`, `OVS_CT_ATTR_ZONE` (2 byte), `OVS_CT_ATTR_MARK` (8 byte mask+value), `OVS_CT_ATTR_LABELS` (32 byte 128-bit value+mask), `OVS_CT_ATTR_HELPER` (string), `OVS_CT_ATTR_NAT` (nested với `OVS_NAT_ATTR_SRC`/`DST`/`IP_MIN`/`IP_MAX`/`PROTO_MIN`/`PROTO_MAX`).

**Ví dụ áp dụng.**

```
# Stateful firewall: commit NEW + allow ESTABLISHED + drop INVALID
table=0,priority=100,ip,ct_state=-trk,actions=ct(table=1,zone=42)
table=1,priority=100,ip,ct_state=+trk+new,actions=ct(commit,zone=42),output:N
table=1,priority=100,ip,ct_state=+trk+est,actions=output:N
table=1,priority=10,ip,ct_state=+trk+inv,actions=drop

# DNAT load balancer (outside hits VIP, get NAT to backend)
table=2,ip,ct_state=-trk,nw_dst=10.0.0.100,tcp,tp_dst=80,
        actions=ct(commit,zone=42,nat(dst=10.0.1.5:8080)),output:N

# SNAT outbound (private→public)
table=3,ip,ct_state=-trk,nw_src=10.0.0.0/24,
        actions=ct(commit,zone=42,nat(src=203.0.113.10)),output:uplink

# FTP ALG (data channel tracking)
table=4,ip,ct_state=-trk,nw_dst=10.0.0.5,tcp,tp_dst=21,
        actions=ct(commit,zone=42,helper=ftp),output:N
```

OVN-generated flow điển hình ở stage `ls_in_lb` cho VIP DNAT:

```
priority=110,ip,ct_state=+new+trk,reg0=0x1,
    actions=ct(commit,table=15,zone=NXM_NX_REG13[0..15],nat(dst=10.0.1.5:8080))
```

**Cách kỹ sư vận hành thành thạo.** Bốn kỹ năng: (1) **Đọc**: phân biệt `ct(table=N, zone=Z)` (lookup-only, không commit) với `ct(commit, zone=Z)` (commit entry); nhận ra pattern OVN luôn dùng `ct(table=N, zone=NXM_NX_REG13[0..15])` để zone từ datapath UUID. (2) **Sửa**: viết stateful pipeline 2 stage — stage 1 `ct(table=N, zone=Z)` lookup, stage N rule riêng theo `ct_state`; thêm `commit` chỉ ở rule NEW. Cho NAT: viết `ct(commit, zone=Z, nat(dst=ip:port))` với `commit` bắt buộc. (3) **Quan sát**: `conntrack -L -z N` xem entry trong zone N; `ovs-appctl dpctl/dump-conntrack` xem từ OVS userspace; `ovs-ofctl dump-flows br-int | grep ct(` đếm rule conntrack. (4) **Debug**: `ofproto/trace 'in_port=1,ip,...'` show line `Datapath actions: ct(commit,zone=42,nat(...)),N` xác nhận; `ct_state` field trong trace output cho biết state hiện tại; nếu state stuck `+trk+new` không chuyển `+est`, kiểm helper ALG hoặc reverse direction.

**Phân loại.** Action loại stateful side-effect + modify (cập nhật conntrack table, populate ct_state, có thể NAT modify packet header), NXM OVS 2.5+ + OF 1.5 partial. Đặc tính: not terminator, MUST recirc qua `table=N` để re-classify với ct_state populate. Cùng class: `ct_clear` (xoá tracking từ packet metadata, OVS 2.10+), không có action stateful khác native trong OVS. Khác với `learn`: `learn` cài rule mới vào flow table; `ct(commit)` cài entry vào conntrack table riêng (kernel netfilter), không phải flow table.

**Source code.** `lib/ofp-actions.c` function anchor (OVS v2.17.9): `parse_CT` line ~4300, `encode_CT` line ~4200, `format_CT` line ~4350; enum `NXAST_RAW_CT` line 631 trong `enum ofp_raw_action_type`. Header `include/openvswitch/ofp-actions.h` `struct ofpact_conntrack` + `struct ofpact_nat`. Translation `ofproto/ofproto-dpif-xlate.c` function `xlate_ct_action`. Datapath kernel module `datapath/conntrack.c` function `ovs_ct_execute` + `ovs_ct_nat`; netlink protocol `include/uapi/linux/openvswitch.h` enum `OVS_ACTION_ATTR_CT` + `enum ovs_ct_attr` + `enum ovs_nat_attr`. Branch baseline: OVS v2.17.9.

**Kịch bản bẻ gãy.** (a) Quên `commit` ở rule NEW → packet đầu được track nhưng không có entry committed → packet thứ hai cùng connection vẫn `ct_state=+new` → rule allow ESTABLISHED không bao giờ match → traffic drop. Fix: rule NEW phải `ct(commit, zone=Z)`. (b) Quên `recirc_table=N` → packet sau `ct` action không có table để continue → behavior undefined ở nhiều OVS version cũ; OVS mới drop packet. Fix: luôn set `table=N` (lookup) hoặc dùng `commit` rồi action khác trong cùng list (commit không yêu cầu table). (c) Zone collision giữa 2 tenant logical → connection của tenant A bị track trong cùng zone với tenant B → cross-tenant leak. Fix: dùng `zone=NXM_NX_REG13[0..15]` để zone từ logical datapath UUID, đảm bảo unique per tenant. (d) NAT `nat(src=10.0.0.5)` không có `commit` → translation thực hiện cho packet này nhưng không lưu state → reverse traffic không un-NAT → connection broken. Fix: NAT luôn cần `commit` + zone.

**Bài tập synthetic.** Mininet 1 OVS bridge + 3 host (h1, h2, h3), IP `10.0.0.1/2/3`. Cài stateful firewall pipeline: `table=0,ip,ct_state=-trk,actions=ct(table=1,zone=42)`, `table=1,ip,ct_state=+trk+new,actions=ct(commit,zone=42),output:NORMAL`, `table=1,ip,ct_state=+trk+est,actions=output:NORMAL`, `table=1,priority=10,ip,ct_state=+trk+inv,actions=drop`, `table=1,priority=0,actions=drop`. Test: `h1 ping h2` thành công. Dùng `conntrack -L -z 42` xem entry `icmp src=10.0.0.1 dst=10.0.0.2 ... [ASSURED]`. Inject packet INVALID bằng `hping3` với SYN-ACK không có SYN trước → counter rule INVALID drop tăng. Verify `ovs-appctl dpif/dump-conntrack` cũng show entry.

**Cách quan sát + xác minh.** `conntrack -L -z N` dump entry trong zone N (cần `apt install conntrack` Ubuntu). `ovs-appctl dpctl/dump-conntrack` dump từ OVS datapath kernel/userspace. `ovs-appctl ofproto/trace br-int 'in_port=1,ip,...'` show line `ct(commit,zone=42,nat(dst=...)),N` + state transition; `ct_state` field trong trace output cho biết state hiện tại (`+new`, `+est`, `+rel`, `+inv`, `+trk`). `ovs-appctl dpif/show-dp-features br-int | grep -i conntrack` xác nhận datapath có conntrack support; thiếu thì `nf_conntrack` module chưa load.

**So sánh với hệ khác.** Linux netfilter `iptables -m conntrack --ctstate ESTABLISHED -j ACCEPT` là functional equivalent — cùng kernel conntrack subsystem nhưng controlled qua iptables thay vì OpenFlow rule. nftables `ct state established` cùng concept syntax mới. Cisco IOS `ip nat inside source list` + `ip access-list` reflexive ACL semantically tương tự nhưng không programmable từ central controller. P4 register-based stateful không có conntrack primitive — phải implement từ đầu. NSX-T distributed firewall stateful built-in qua hypervisor kernel module riêng. Calico iptables-based dataplane dùng iptables conntrack trực tiếp; Calico eBPF dataplane dùng eBPF map riêng tương đương.

## 4.9.35 Treatment ngoại biên cho 5 action MPLS + PBB hạng peripheral

Phần này bổ sung treatment riêng (peripheral, ~15-30 dòng/action) cho 5 action MPLS + PBB rất hiếm gặp ngoài service provider Ethernet. Khác với §4.9.16 (catalog ngắn gọn) và §4.9.29.2-5 (Anatomy 8-attribute v3.5), phần này tập trung vào quan sát + xác minh + lý do tồn tại + caveat upstream OVS.

### 4.9.35.1 `push_pbb` — đẩy PBB I-TAG (802.1ah) vào packet

**Khái niệm.** `push_pbb` là action chuẩn OpenFlow 1.3+ (OFPAT_PUSH_PBB type 26) chèn shim header 802.1ah Provider Backbone Bridge (PBB) — gồm B-DA 6 byte + B-SA 6 byte + B-Tag 4 byte (TPID `0x88a8`) + I-Tag 6 byte (TPID `0x88e7` + I-SID 24 bit) — trước Ethernet customer header. Sau action, packet length tăng 22 byte, ethertype outer chuyển `0x88e7`. Action nhận argument `ethertype=0x88e7` để chỉ định outer TPID. Loại: header push, OF 1.3+ standard.

**Bối cảnh + động cơ.** PBB (IEEE 802.1ah, ratified 2008) là extension của Provider Bridge (Q-in-Q 802.1ad) cho carrier Ethernet metro: thay vì dùng VLAN ID 12-bit (4096 max), PBB dùng I-SID 24-bit (16M service instance) + tách MAC space khách hàng khỏi backbone. OF 1.3 (2012) thêm `OFPAT_PUSH_PBB` để controller programmable encap PBB.

**Trạng thái hỗ trợ trong OVS.** **Quan trọng.** OVS upstream tới v3.3.0 (2024-04) **không implement** `push_pbb`. Bằng chứng: trong `lib/ofp-actions.c` v3.3.0 dòng 9033-9034 ghi rõ `/* OF1.3+ OFPAT_PUSH_PBB (26) not supported. */` trong bảng `ofpact_map of12[]`. Không có `parse_PUSH_PBB`, `encode_PUSH_PBB`, `decode_OFPAT_RAW_PUSH_PBB` nào tồn tại trong source. Engineer đưa rule `actions=push_pbb:0x88e7` vào `ovs-ofctl add-flow` sẽ nhận lỗi parser kiểu `unknown action 'push_pbb'` hoặc tương đương.

**Vì sao OVS không implement.** PBB là carrier Ethernet technology, target là provider backbone hardware switch (Cisco ASR 9000, Juniper MX, Nokia 7750) chứ không phải DC virtual switch. OVN dùng Geneve/VXLAN cho overlay multi-tenant — tách isolation theo VNI 24-bit không cần PBB I-SID. Cộng đồng OVS không nhận patch PBB vì không có use case practical trong DC cloud hyperscaler.

**Cú pháp spec OF 1.3 (tham khảo).** Wire format: action header `type=26` (OFPAT_PUSH_PBB) + `length=8` + `ethertype` 2 byte (0x88e7) + 2 byte padding. Prerequisite spec: rule không cần match `eth_type=0x88e7` (vì action mục đích encap, packet đầu vào thường Ethernet thường); sau action mới có `eth_type=0x88e7`. Spec quy ước B-DA = `eth_dst` hiện tại, B-SA = `eth_src`, I-SID mặc định 0 — engineer dùng `set_field:0x123456->pbb_isid` ngay sau để gán I-SID thực.

**Quan sát + workaround.** Khi cần programmable PBB encap trong môi trường OVS, engineer phải fallback sang Linux native: `ip link add link eth0 name pbb0 type 8021ah ...` (kernel `8021ah` driver, không phải OVS), hoặc đẩy lên hardware switch carrier rồi peer với OVS qua trunk port. So với hệ khác: Cisco IOS-XR `l2vpn xconnect-group` + `pbb` instance hỗ trợ PBB native; P4 program PBB qua `header pbb_t` declaration; FRR/Quagga không hỗ trợ data-plane PBB (chỉ có control-plane phối hợp).

### 4.9.35.2 `pop_pbb` — bóc PBB I-TAG khỏi packet

**Khái niệm.** `pop_pbb` là action chuẩn OpenFlow 1.3+ (OFPAT_POP_PBB type 27) bóc shim header 802.1ah PBB (22 byte) ngoài cùng khỏi packet, expose customer Ethernet header bên dưới. Sau action, packet length giảm 22 byte, ethertype outer chuyển về Ethernet customer thực. Action không nhận argument — luôn bóc 1 PBB layer outermost. Loại: header pop, OF 1.3+ standard.

**Bối cảnh + động cơ.** Cặp đối xứng với `push_pbb`: switch backbone egress (BCB/BEB egress) cần bóc PBB shim trước khi forward packet ra customer port. OF 1.3 chuẩn hoá `OFPAT_POP_PBB` (type 27) cùng `OFPAT_PUSH_PBB` (type 26) cho symmetry.

**Trạng thái hỗ trợ trong OVS.** **Quan trọng.** OVS upstream tới v3.3.0 **không implement** `pop_pbb`. Bằng chứng: `lib/ofp-actions.c` v3.3.0 dòng 9034 ghi `/* OF1.3+ OFPAT_POP_PBB (27) not supported. */` trong bảng `ofpact_map of12[]`. Không có `parse_POP_PBB`, `encode_POP_PBB`, `decode_OFPAT_RAW_POP_PBB`. Rule `actions=pop_pbb` đưa vào `ovs-ofctl add-flow` bị parser reject. Spec quy ước `pop_pbb` cần prerequisite `eth_type=0x88e7` (packet phải có outer PBB tag) — nhưng vì OVS không implement, prerequisite này cũng không enforce.

**Vì sao OVS không implement.** Cùng lý do `push_pbb`: PBB là carrier technology, không có use case DC. OVS classifier không có field `pbb_isid` riêng (khác hardware switch như Broadcom Trident có direct PBB parser). Nếu cần xử lý PBB trong OVS, engineer phải push packet lên controller qua `controller(...)` action rồi parse PBB userspace.

**Cú pháp spec OF 1.3 (tham khảo).** Wire format: action header `type=27` (OFPAT_POP_PBB) + `length=8` + 4 byte padding (không có argument body). Datapath kernel: nếu OVS implement, sẽ map sang `OVS_ACTION_ATTR_POP_PBB` netlink — nhưng kernel module `openvswitch.ko` cũng không có handler `pop_pbb` (search `git grep pop_pbb` trong `datapath/` ra 0 hit).

**Quan sát + workaround.** Engineer cần bóc PBB phải fallback Linux kernel: `ip link add link bond0 type 8021ah` rồi forward packet qua interface PBB-aware để kernel auto-decap. Hoặc dùng tc filter: `tc filter add dev eth0 ingress flower indev eth0 src_mac ...` không có direct PBB pop trong tc nên cũng phải custom BPF program (out of OVS scope). So với hệ khác: Cisco IOS-XR `interface l2tunnel` auto pop PBB; Juniper MX `bridge-domain pbb` similar.

### 4.9.35.3 `set_mpls_ttl` — ghi đè MPLS TTL field

**Khái niệm.** `set_mpls_ttl:N` là action chuẩn OpenFlow 1.1+ (OFPAT_SET_MPLS_TTL type 15) ghi đè byte TTL của MPLS label outermost thành giá trị N (0-255). Action nhận argument `uint8_t ttl`. Sau action, byte 4 của shim MPLS (vị trí TTL trong header 4 byte: label 20 bit + TC 3 bit + BoS 1 bit + TTL 8 bit) được set thành N. Loại: field rewrite, OF 1.1+ standard, OVS 1.11+ implement.

**Bối cảnh + động cơ.** MPLS label header có TTL 8-bit để ngăn loop infinite trong LSP (Label Switched Path). Khi MPLS edge router (PE) push label mới, TTL của label outer mặc định kế thừa từ label inner — gọi là **uniform mode** (traceroute thấy đầy đủ hop). Một số scenario muốn ẩn topology core (gọi là **pipe mode**): engineer set TTL outer thành giá trị cao cố định (ví dụ 64) để traceroute không expose số hop core. `set_mpls_ttl` là primitive cho pipe mode programming.

**Trạng thái hỗ trợ trong OVS.** OVS implement đầy đủ. Bằng chứng `lib/ofp-actions.c` v3.3.0: enum `OFPAT_RAW_SET_MPLS_TTL` dòng 210 (`/* NX1.0(25), OF1.1(15), OF1.2+(15) is deprecated (use Set-Field): uint8_t. */`); function `parse_SET_MPLS_TTL` dòng 3925 (parse argument qua `str_to_u8`, set vào `mpls_ttl->ttl`); `encode_SET_MPLS_TTL` dòng 3913 (gọi `put_OFPAT_SET_MPLS_TTL`); `decode_OFPAT_RAW_SET_MPLS_TTL` dòng 3904; `format_SET_MPLS_TTL` dòng 3944 (in `set_mpls_ttl(N)`). Note: OF 1.2+ deprecated `set_mpls_ttl` favor `set_field:N->mpls_ttl` qua generic `OFPACT_SET_FIELD`, nhưng `set_mpls_ttl` vẫn parser-accept ở mọi version cho backward compat.

**Cú pháp + prerequisite.** Cú pháp `actions=set_mpls_ttl:64`. Prerequisite: rule phải match MPLS (`eth_type=0x8847` unicast hoặc `0x8848` multicast) — nếu rule không match MPLS thì action no-op cho packet không có MPLS shim ở outermost. Wire encoding OF 1.1+: action header `type=15` + `length=8` + 1 byte ttl + 3 byte padding. Datapath kernel: action map sang `OVS_ACTION_ATTR_SET` với `OVS_KEY_ATTR_MPLS` containing `mpls_lse` 32-bit field — kernel rewrite byte TTL trong shim.

**Use case + ví dụ.** Pipe mode L3 VPN: `priority=100,ip,nw_dst=10.1.0.0/24,actions=push_mpls:0x8847,set_field:100->mpls_label,set_mpls_ttl:64,output:uplink` — push label 100 với TTL fix 64 để ẩn topology core. Refresh TTL khi label swap: `priority=100,mpls,mpls_label=100,actions=set_mpls_ttl:64,set_field:200->mpls_label,output:next` — refresh TTL về 64 khi swap label tránh drop ở core. Cờ (flag) commit: lưu ý OF 1.2+ recommendation là dùng `set_field:64->mpls_ttl` qua `OXM_OF_MPLS_TTL` thay vì legacy `set_mpls_ttl:64`; OVS accept cả hai. So với hệ khác: Cisco IOS `mpls ttl-expiration pop 1 refresh-tll` config-time, Juniper `set protocols mpls no-decrement-ttl` similar; tc cls_u32 không programmable per-flow MPLS TTL.

### 4.9.35.4 `dec_mpls_ttl` — giảm MPLS TTL outermost một đơn vị

**Khái niệm.** `dec_mpls_ttl` là action chuẩn OpenFlow 1.1+ (OFPAT_DEC_MPLS_TTL type 16) decrement TTL của MPLS label outermost đi 1. Nếu TTL trước decrement = 0 hoặc sau decrement = 0, packet drop hoặc gửi packet_in lên controller (theo flag flow). Action không nhận argument. Loại: field arithmetic, OF 1.1+ standard, OVS 1.11+ implement.

**Bối cảnh + động cơ.** Mọi MPLS hop transit (LSR — Label Switching Router) phải decrement TTL để chống loop infinite — equivalent với IP TTL decrement của IP forwarding. Trước khi có `dec_mpls_ttl`, OpenFlow 1.0 không có primitive dedicated cho MPLS TTL — controller phải emulate qua `set_mpls_ttl` với giá trị hardcode hoặc qua `set_field` từ register sau khi đọc rồi giảm 1 (phức tạp + race condition).

**Trạng thái hỗ trợ trong OVS.** OVS implement đầy đủ. Bằng chứng `lib/ofp-actions.c` v3.3.0: enum `OFPAT_RAW_DEC_MPLS_TTL` dòng 213 (`/* NX1.0(26), OF1.1+(16): void. */`); function `parse_DEC_MPLS_TTL` dòng 3976 (action không argument, chỉ `ofpact_put_DEC_MPLS_TTL`); `encode_DEC_MPLS_TTL` dòng 3969 (gọi `put_OFPAT_DEC_MPLS_TTL`); `decode_OFPAT_RAW_DEC_MPLS_TTL` dòng 3962 (input chỉ `struct ofpbuf *out`, không có ttl param vì action không có body); `format_DEC_MPLS_TTL` dòng 3983 (in `dec_mpls_ttl` không suffix). Header `include/openvswitch/ofp-actions.h` dùng `struct ofpact_null` cho action argumentless.

**Cú pháp + prerequisite + behavior khi TTL=0.** Cú pháp `actions=dec_mpls_ttl`. Prerequisite: rule match MPLS (`eth_type=0x8847/0x8848`). Wire encoding OF 1.1+: action header `type=16` + `length=8` + 4 byte padding (không argument). Behavior khi TTL=0: spec OF quy ước nếu TTL trước decrement = 1, sau decrement = 0 thì packet phải drop hoặc gửi packet_in `OFPR_INVALID_TTL` lên controller (phụ thuộc flag `OFPFF_INVALID_TTL_TO_CONTROLLER` của flow). OVS: kernel datapath drop packet TTL=0 silent; userspace upcall path có thể emit packet_in nếu controller đăng ký.

**Use case + ví dụ.** LSR transit refresh: `priority=100,mpls,mpls_label=100,actions=dec_mpls_ttl,set_field:200->mpls_label,output:next` — decrement TTL trước khi swap label rồi forward. Combo với label swap: hop count đúng, packet tự drop ở hop thứ 64 nếu loop. Pipe mode hide-topology: KHÔNG dùng `dec_mpls_ttl` ở core (giữ TTL outer cố định 64 từ ingress) → traceroute từ end host không thấy hop core. Cờ commit: nếu controller muốn nhận packet_in khi TTL expire (cho ICMP TTL exceeded reply), set `flags=send_flow_rem,check_overlap` + đăng ký reason `OFPR_INVALID_TTL` qua `set-async`. So với hệ khác: Cisco IOS LSR auto decrement MPLS TTL theo default; Juniper similar; FRR ldpd control-plane chỉ, data-plane Linux kernel mpls module tự decrement.

### 4.9.35.5 `copy_ttl_in` — copy TTL từ outer header xuống inner header

**Khái niệm.** `copy_ttl_in` là action chuẩn OpenFlow 1.1+ (OFPAT_COPY_TTL_IN type 12) copy TTL từ header outer (MPLS shim hoặc IP outer) xuống TTL header inner liền kề (IP inner). Action không nhận argument. Dùng khi pop encapsulation và muốn preserve hop count xuyên qua tunnel — gọi là **uniform mode** trong MPLS-IP. Loại: TTL preservation pseudo-action, OF 1.1+ standard.

**Bối cảnh + động cơ.** Khi packet đi qua MPLS LSP rồi PE pop label: TTL của IP inner đã bị "đông" từ ingress (PE push label đã copy IP TTL ra MPLS TTL outer, IP TTL inner giữ nguyên). Ở egress, TTL outer đã decrement qua N hop core, TTL inner vẫn giá trị ban đầu. Engineer có 2 lựa chọn: (a) **pipe mode**: pop label không copy TTL, IP TTL inner giữ nguyên — host end-to-end không thấy hop core qua traceroute (topology hidden); (b) **uniform mode**: pop label kèm `copy_ttl_in` để IP TTL inner = MPLS TTL outer trước pop, host thấy đầy đủ hop xuyên LSP. Hai mode này là decision quan trọng của carrier MPLS deployment.

**Trạng thái hỗ trợ trong OVS.** **Quan trọng.** OVS upstream tới v3.3.0 **không implement** `copy_ttl_in`. Bằng chứng: `lib/ofp-actions.c` v3.3.0 dòng 9001 + 9021 ghi rõ `/* OFPAT_COPY_TTL_IN (12) not supported. */` trong bảng `ofpact_map of10[]` (OF 1.0/1.1) và `of12[]` (OF 1.2/1.3/1.4). Không có `parse_COPY_TTL_IN`, `encode_COPY_TTL_IN`, `decode_OFPAT_RAW_COPY_TTL_IN`, không có enum `OFPAT_RAW_COPY_TTL_IN` trong `ofp_raw_action_type`. Rule `actions=copy_ttl_in` vào `ovs-ofctl add-flow` bị parser reject với error `unknown action`.

**Vì sao OVS không implement.** Lý do tương tự PBB: copy_ttl_in/out là carrier MPLS feature, không có use case practical trong DC overlay. OVN dùng Geneve/VXLAN — không có khái niệm "TTL preserve" vì IP inner và outer là 2 IP packet riêng (Geneve outer là UDP IPv4/IPv6 với TTL riêng, VNI 24-bit chứa packet inner độc lập). Nếu cần preserve hop count, controller phải emulate qua `set_field:NXM_NX_REG0[]->nw_ttl` với register copy logic — phức tạp + ít thực tế.

**Workaround + cú pháp spec OF 1.1 (tham khảo).** Spec OF 1.1+ wire format: action header `type=12` (OFPAT_COPY_TTL_IN) + `length=8` + 4 byte padding. Spec quy ước action chỉ valid khi packet có ≥ 2 header layer (MPLS-over-IP, IP-over-IP). Workaround trong OVS nếu cần emulate: viết flow chain dùng register — `actions=move:NXM_OF_MPLS_TTL[]->NXM_NX_REG0[0..7],pop_mpls:0x0800,move:NXM_NX_REG0[0..7]->NXM_OF_IP_TTL[]` — đọc MPLS TTL vào reg0 trước khi pop, sau pop ghi lại từ reg0 xuống IP TTL. Tuy nhiên `NXM_OF_MPLS_TTL` không tồn tại như NXM field standalone — engineer phải dùng `set_field` với mpls_ttl OXM, complex vì OVS classifier không expose MPLS TTL như register-readable field. Kết luận thực tế: nếu nghiệp vụ thực sự cần `copy_ttl_in`, không nên dùng OVS — chuyển sang hardware switch carrier (Cisco/Juniper).

**So với hệ khác.** Cisco IOS-XR `mpls ip-ttl-propagate disabled` (pipe mode) hoặc default enable (uniform mode tự copy_ttl_in) config-time, không cần action programmable. Juniper `set protocols mpls no-propagate-ttl` similar. Linux kernel `mpls_iptunnel` module hỗ trợ TTL inheritance qua sysctl `net.mpls.ip_ttl_propagate=1`. P4 program copy TTL qua `meta.outer_ttl = hdr.outer.ttl; hdr.inner.ttl = meta.outer_ttl;` direct assignment.

## 4.9.36 Treatment ngoại biên cho 5 action NSH + observability hạng peripheral

Phần này bổ sung treatment riêng (peripheral, ~15-30 dòng/action) cho 5 action ngoại biên thuộc 2 nhóm: nhóm encap/decap generic packet-type-aware (NSH service function chaining) gồm `decap`, `encap`; nhóm observability + annotation gồm `controller(userdata=...)`, `note`, `sample`. Khác với §4.9.29.7-11 (Anatomy 8-attribute v3.5 ngắn gọn), phần này tập trung vào cơ chế parser thực tế trong upstream OVS, side-effect runtime, caveat khi sử dụng, và ví dụ artifact engineer thực sự gõ trên CLI.

### 4.9.36.1 `decap` — bóc encapsulation header outermost theo packet_type

**Khái niệm.** `decap` là action OVS extension (NXAST_RAW_DECAP) bóc header encapsulation outermost của packet và update field `packet_type` thành protocol bên trong. Cú pháp `decap` (không argument) hoặc `decap(packet_type=(NS,TYPE))` chỉ định loại packet sau khi pop. Sau action, packet length giảm theo size header pop, classifier field `packet_type` đổi sang giá trị NS/TYPE truyền vào (hoặc `PT_USE_NEXT_PROTO` nếu không truyền).

**Vai trò.** `decap` là counterpart của `encap`, dùng trong service function chain (SFC) khi service function cuối chuỗi cần bóc NSH header trước khi forward packet ra mạng customer. Cũng dùng để strip MPLS/Geneve/GRE outer ngoài context SFC. Nguồn gốc: NSH RFC 8300 (2018) chuẩn hoá Service Function Chain header; OVS thêm `encap`/`decap` packet-type-aware từ v2.8 (2017) để programmable từ controller.

**Cú pháp parse trong OVS.** Function `parse_DECAP` ở `lib/ofp-actions.c` (OVS branch v3.3) parse argument `packet_type=(ns=NS,type=TYPE)` qua `ofputil_parse_key_value`; nếu không có argument, default `decap->new_pkt_type = htonl(PT_USE_NEXT_PROTO)` để datapath tự suy ra protocol từ header pop được. Wire format: `NXAST_RAW_DECAP` subtype của Nicira extension; struct `nx_action_decap` chứa field `new_pkt_type` 4 byte. Decode qua `decode_NXAST_RAW_DECAP`.

**Cú pháp + ví dụ thực tế.** Bóc NSH 2 lần để revert về Ethernet:

```
ovs-ofctl add-flow br0 \
  "table=0,priority=100,nsh,nsh_si=0,actions=decap(),decap(packet_type=(0,0x0800)),output:2"
```

Bóc Geneve outer:

```
ovs-ofctl add-flow br0 \
  "table=0,priority=100,tun_id=0x100,actions=decap(packet_type=(1,0x6558)),output:2"
```

(`(1,0x6558)` = NS=1 OFPHTN_ETHERTYPE, TYPE=0x6558 Transparent Ethernet Bridging.)

**Caveat + lỗi thường gặp.** `decap` không có error nếu packet không có header tương ứng — datapath sẽ produce kết quả undefined. Engineer phải match strict (`nsh`, `mpls`, `tun_id=...`) trước khi gọi `decap`. Lỗi phổ biến: gọi `decap()` trên packet Ethernet thường → kernel datapath drop silent. Cách quan sát: `ovs-appctl dpctl/dump-flows | grep drop` + `ovs-ofctl dump-flows br0` so chiếu hit count với expected. So với hệ khác: P4 `header.pop_front(1)` direct, không cần packet_type negotiation; Linux tc `pedit` không có decap generic.

### 4.9.36.2 `encap` — đẩy encapsulation header packet-type-aware (NSH SFC)

**Khái niệm.** `encap` là action OVS extension (NXAST_RAW_ENCAP) thêm encapsulation header lên trên packet hiện tại và update field `packet_type` thành packet type mới (outer). Cú pháp `encap(<header_type>(prop=value, tlv(class,type,value), ...))`. Hỗ trợ header_type: `nsh`, `ethernet`, `mpls`. Sau action, packet length tăng theo size header push, `packet_type` chuyển sang outer protocol.

**Vai trò.** Entry point của service function chain (SFC) hoặc tunnel: SFC classifier ở DC ingress emit `encap(nsh)` để wrap traffic vào chain với metadata service path identifier (SPI) + service index (SI). NSH metadata có 2 mode: MD Type 1 fixed-length 16 byte (4 × 32-bit context header), MD Type 2 variable TLV (`tlv(class=...,type=...,value=...)`). Nguồn gốc: RFC 8300 NSH chuẩn hoá 2018; OVS implement v2.8.

**Cú pháp parse trong OVS.** Function `parse_ENCAP` ở `lib/ofp-actions.c` (OVS branch v3.3) yêu cầu OF version ≥ 1.3 (`*pp->usable_protocols &= OFPUTIL_P_OF13_UP`). Parse header_type qua `parse_encap_header(key, &encap->new_pkt_type)`, parse property class qua `parse_ed_prop_class`. Wire format: `NXAST_RAW_ENCAP` subtype Nicira extension (không phải pure OF 1.5 standard mặc dù OF 1.5 spec định nghĩa `OFPAT_ENCAP`); struct `nx_action_encap` chứa `new_pkt_type` + variable property body. Decode qua `decode_NXAST_RAW_ENCAP`.

**Cú pháp + ví dụ thực tế.** SFC entry classifier với MD Type 1:

```
ovs-ofctl add-flow br0 \
  "table=0,priority=100,ip,ip_dst=10.0.0.0/24,actions=\
   encap(nsh(md_type=1)),\
   set_field:0x123->nsh_spi,set_field:255->nsh_si,\
   set_field:0xdeadbeef->nsh_c1,output:tun0"
```

MD Type 2 với TLV custom:

```
ovs-ofctl add-flow br0 \
  "table=0,priority=100,ip,actions=\
   encap(nsh(md_type=2,tlv(class=0x1,type=1,len=4,value=0xcafebabe))),\
   output:tun0"
```

**Caveat + version drift.** `encap(nsh)` chỉ available khi OVS build với NSH support (`./configure --enable-nsh`); package distro Ubuntu 22.04 (OVS 2.17) bật mặc định, nhưng RHEL 8/9 có thể không. Verify: `ovs-appctl dpif/show-dp-features br0 | grep -i nsh`. Lỗi parser nếu OVS không support: `unknown action 'encap'`. Khác biệt phiên bản: OVS 2.8-2.10 chỉ support MD Type 1, MD Type 2 TLV thêm OVS 2.12+. So với hệ khác: VPP `encap-nsh` plugin hỗ trợ NSH từ v18.07; Cisco IOS-XR `service-function-chain` config-time, không programmable per-flow.

### 4.9.36.3 `controller(userdata=...)` — gửi PACKET_IN với metadata correlation

**Khái niệm.** `controller(userdata=HEX,...)` là extended form của action `controller` (vốn ở §4.9.6) cho phép đính kèm metadata cookie vào PACKET_IN message để controller correlation rule nguồn với packet nhận được. Argument `userdata` là chuỗi hex bytes (length tuỳ ý, ví dụ `userdata=01.02.03.04` hoặc `userdata=ABCD`). Khi datapath gọi action, packet + userdata bytes được gói vào PACKET_IN với reason `OFPR_ACTION` (hoặc reason custom).

**Vai trò.** Trong continuation-based programming model (OVS 2.6+, NXAST_RAW_CONTROLLER2), controller có thể respond bằng PACKET_OUT với "continue" action để tiếp tục pipeline đã pause — userdata giúp controller match request/response. Trong basic mode (NXAST_RAW_CONTROLLER cũ), userdata chỉ là cookie debug. OVN dùng userdata extensively cho ARP/ND responder, DHCP relay, MAC binding learning — controller `ovn-controller` đọc userdata để biết loại event và chassis nguồn.

**Cú pháp parse trong OVS.** Function `parse_CONTROLLER` ở `lib/ofp-actions.c` (OVS branch v3.3) parse 6 key: `reason`, `max_len`, `id` (controller_id), `userdata`, `pause`, `meter_id`. `userdata` parse qua `ofpbuf_put_hex` để convert hex string sang bytes. Optimization: nếu chỉ có default (`reason=OFPR_ACTION`, controller_id=0, không userdata, không pause, không meter), parser emit `OFPACT_OUTPUT` với port `OFPP_CONTROLLER` thay vì full controller action — tiết kiệm wire bytes. Decode raw type `NXAST_RAW_CONTROLLER` (basic) hoặc `NXAST_RAW_CONTROLLER2` (continuation, có max_len + pause + userdata).

**Cú pháp + ví dụ thực tế.** ARP responder fallback OVN-style:

```
ovs-ofctl add-flow br-int \
  "table=8,priority=50,arp,arp_op=1,\
   actions=controller(reason=action,id=0,userdata=00.00.00.0a.00.00.00.00.05.00.00.00)"
```

Continuation pause + resume:

```
ovs-ofctl add-flow br0 \
  "table=0,priority=100,tcp,tp_dst=80,\
   actions=controller(reason=action,id=42,userdata=cafe1234,pause)"
```

(controller nhận PACKET_IN có userdata `cafe1234`, decision logic xong respond PACKET_OUT với continuation action để pipeline tiếp tục từ điểm pause.)

**Quan sát + caveat.** Verify userdata round-trip: `ovs-ofctl monitor br0 65534 -P openflow13` để dump PACKET_IN realtime, kiểm tra `userdata=...` field trong message body. Caveat: userdata không được encrypt — nếu OpenFlow channel không TLS thì cookie leak qua wire. Lỗi thường gặp: hex string với dấu nháy thiếu hoặc không chia byte boundary chuẩn (`userdata=ABC` lẻ 3 ký tự → parser reject `bad hex digit`). So với hệ khác: P4 `digest<T>(receiver, data)` similar, nhưng strongly typed; Linux netfilter `nfqueue` queue packet lên userspace tương đương nhưng không có cookie correlate per-rule.

### 4.9.36.4 `note` — embed bytes annotation vào flow rule (no-op runtime)

**Khái niệm.** `note:HEXSTRING` là action OVS extension (NXAST_RAW_NOTE) lưu chuỗi bytes (hex format, dấu chấm phân cách optional) vào flow definition mà không có runtime effect lên packet. Kích thước tối đa giới hạn bởi `ofpbuf_oversized` check (~64KB max payload OpenFlow message), thực tế dùng dưới 32 bytes. Khi `dump-flows` hiển thị rule, note hex visible cho operator decode.

**Vai trò.** Pure annotation/debug marker: cài inline metadata vào flow rule để post-mortem audit. Use case: encode build version, controller-side rule UUID, deployment timestamp, conditional flag mà controller cần re-read khi reconcile. OVN không dùng `note` (OVN dùng `external_ids` của Logical_Flow trong SBDB), nhưng project custom controller như Ryu, Faucet hay dùng `note` để stamp rule provenance.

**Cú pháp parse trong OVS.** Function `parse_NOTE` ở `lib/ofp-actions.c` (OVS branch v3.3) parse hex string qua `ofpbuf_put_hex`, accept dấu chấm `.` và `:` làm separator (ignore khi parse). Wire encoding: struct `nx_action_note` có header `len`, `vendor=NX_VENDOR_ID`, `subtype=NXAST_NOTE`, `note[6]` start of user-defined data, có thể follow thêm bytes. `OFP_ASSERT(sizeof(struct nx_action_note) == 16)` đảm bảo wire layout cố định 16 byte minimum. Decode `decode_NXAST_RAW_NOTE` đọc length từ wire `nan->len` trừ offset của note field, copy bytes vào ofpact buffer.

**Cú pháp + ví dụ thực tế.** Stamp build version lên rule:

```
ovs-ofctl add-flow br0 \
  "table=0,priority=100,in_port=1,\
   actions=note:01.02.03.04.de.ad.be.ef,output:2"
```

Verify khi `dump-flows`:

```
$ ovs-ofctl dump-flows br0 | grep note
 cookie=0x0, duration=15.2s, table=0, n_packets=0, n_bytes=0,
   priority=100,in_port=1 actions=note:01.02.03.04.de.ad.be.ef,output:2
```

**Caveat + use case implicit.** Note KHÔNG semantic effect: kernel datapath skip action này hoàn toàn (action_set không thay đổi). Phổ biến nhất: pad rule để force re-installation khi controller upgrade (note bytes thay đổi → flow `add` thay vì `unchanged`). Lỗi thường gặp: hex string lẻ ký tự (`note:1`) → parser reject. So với hệ khác: Linux iptables `--comment` module similar (chuỗi ASCII inline), nhưng giới hạn 256 byte; tc filter không có comment field.

### 4.9.36.5 `sample(probability=...,collector_set_id=...)` — packet sampling cho IPFIX/sFlow

**Khái niệm.** `sample(probability=N,collector_set_id=ID,...)` là action OVS extension (NXAST_RAW_SAMPLE / SAMPLE2 / SAMPLE3) sample packet với xác suất `N/65535` và emit IPFIX flow record tới collector_set đã config trong OVSDB. Argument bắt buộc: `probability` (1-65535), `collector_set_id` (uint32). Argument optional: `obs_domain_id`, `obs_point_id`, `sampling_port`, `ingress`/`egress` direction.

**Vai trò.** Built-in flow sampling cho telemetry: thay vì mirror toàn bộ traffic ra collector port (tốn bandwidth), `sample` chọn xác suất per-packet rồi emit IPFIX record (header + 5-tuple + counter). NetFlow/sFlow analyzer phía collector aggregate record để estimate top-talker, top-application. Khác `sFlow agent` global của OVS (sample mọi packet vào toàn bridge): `sample` action embedded vào flow rule riêng → selective per-flow programmable, controller chọn flow nào sample.

**Cú pháp parse trong OVS.** Function `parse_SAMPLE` ở `lib/ofp-actions.c` (OVS branch v3.3) parse 7 key qua `ofputil_parse_key_value`: `probability`, `collector_set_id`, `obs_domain_id`, `obs_point_id`, `sampling_port`, `ingress`, `egress`. Validate `probability != 0` (zero không hợp lệ — datapath trừ luôn). 3 wire variant: `NXAST_RAW_SAMPLE` (basic), `NXAST_RAW_SAMPLE2` (thêm sampling_port), `NXAST_RAW_SAMPLE3` (thêm direction ingress/egress, OVS 2.5+). Decode tương ứng `decode_NXAST_RAW_SAMPLE` / `SAMPLE2` / `SAMPLE3`.

**Prerequisite OVSDB config.** `collector_set_id` phải khớp với hàng trong table `Flow_Sample_Collector_Set` của OVSDB. Setup tối thiểu:

```
ovs-vsctl -- --id=@cs create Flow_Sample_Collector_Set id=1 \
              ipfix=@i bridge=@br -- \
           --id=@i create IPFIX targets=\"10.0.0.1:4739\" \
              obs_domain_id=42 obs_point_id=0 -- \
           --id=@br get Bridge br0
```

**Cú pháp + ví dụ thực tế.** Sample 1% (probability ≈ 655) traffic từ port 1:

```
ovs-ofctl add-flow br0 \
  "table=0,priority=100,in_port=1,\
   actions=sample(probability=655,collector_set_id=1,obs_domain_id=42,obs_point_id=100),output:2"
```

Sample 10% với direction ingress (OVS 2.5+):

```
ovs-ofctl add-flow br0 \
  "table=0,priority=100,ip,\
   actions=sample(probability=6553,collector_set_id=1,obs_domain_id=42,obs_point_id=100,ingress),output:2"
```

**Quan sát + caveat.** Verify IPFIX export: `tcpdump -i any -nn 'udp port 4739'` trên collector host để xem record tới. Caveat: sampling không dừng pipeline — `output:2` vẫn execute (sample là parallel side-effect). Probability granular 1/65535 nên minimum = 0.0015% ≈ 1/65535. Lỗi thường gặp: thiếu `Flow_Sample_Collector_Set` row → action no-op silent (datapath skip emit IPFIX, không log error), cách phát hiện: `ovs-appctl dpctl/dump-flows | grep sample` thấy n_packets tăng nhưng collector không nhận record. Verify collector_set tồn tại: `ovs-vsctl list Flow_Sample_Collector_Set`. So với hệ khác: Linux tc `flower` + `mirred` không có sampling rate built-in (phải dùng iptables `statistic` module random); P4 `clone(I2E, mirror_session)` clone full packet, cũng không có rate built-in cần extern call.

## 4.9.37 Treatment ngoại biên cho 5 action NXM stack + register hạng peripheral

Phần này bổ sung treatment riêng (peripheral, ~15-30 dòng/action) cho 5 action ngoại biên thuộc nhóm NXM stack + register manipulation: `move:src->dst` (copy field-to-field NXM), `copy_field:src->dst` (chuẩn OF 1.5 + ONF 1.3 extension), `output:NXM_NX_REG0` (indirect output đọc port number từ register), `push:reg0` (đẩy giá trị field/register lên stack per-packet), `pop:reg0` (lấy giá trị từ stack vào field/register). Khác với §4.9.19 (catalog ngắn gọn cho `move`+`load`) và §4.9.31 (Anatomy 8-attribute cho stack) đã có, phần này tập trung vào parser thực tế trong upstream OVS, wire encoding, version negotiation OF 1.5 vs NXM, và artifact engineer thực sự gõ trên CLI.

### 4.9.37.1 `move:src->dst` — copy field-to-field theo NXM extension

**Khái niệm.** `move:SRC[range]->DST[range]` là action OVS extension copy bits từ field/subfield nguồn sang field/subfield đích. Bit-range syntax `[lo..hi]` hoặc `[bit]` optional, nếu thiếu → copy full width. Hai field PHẢI cùng width (nếu rule `move:reg0[0..7]->reg1[0..15]` parser reject lỗi `bit count mismatch`). Loại: pure register manipulation, no packet modification trừ khi DST là field thuộc header (ví dụ `eth_src`).

**Nguồn gốc + bối cảnh.** `move` là Nicira NXAST_REG_MOVE extension có từ Open vSwitch sớm (≤ 1.0), tiền thân của OF 1.5 standard `OFPAT_COPY_FIELD` (type 28). Mục đích: cho controller programmable swap field, save/restore qua register, thực thi pattern ARP responder / L3 NAT mà không cần qua controller. Trước OF 1.5, đây là cách duy nhất để copy generic field-to-field (OF 1.0 không có primitive này).

**Cú pháp parse trong OVS branch-2.17.** Function `parse_REG_MOVE` ở `lib/ofp-actions.c` v2.17 dòng 2713 chỉ wrap `nxm_parse_reg_move` từ `lib/nx-match.c` — parser đọc string `SRC->DST` rồi populate `struct ofpact_reg_move` với 2 subfield (`src`, `dst`) mỗi subfield có `field` (mff identifier), `ofs` (bit offset), `n_bits` (bit width). Wire encode: `encode_REG_MOVE` dòng 2674 chọn raw type theo OF version: OF 1.5+ → `OFPAT_RAW15_COPY_FIELD` (chuẩn 16 byte), OF 1.3-1.4 → `ONFACT_RAW13_COPY_FIELD` (ONF extension 24 byte) hoặc `NXAST_RAW_REG_MOVE` (Nicira 16 byte) tùy `move->ofpact.raw` có bị set qua decode trước không. Default OF 1.0-1.2 → `NXAST_RAW_REG_MOVE`. Struct `nx_action_reg_move` (dòng 2574 `OFP_ASSERT == 16`) có `n_bits`, `src_ofs`, `dst_ofs` rồi follow OXM/NXM header cho src + dst.

**Cú pháp + ví dụ thực tế.** Swap eth_src ↔ eth_dst qua register tạm:

```
ovs-ofctl add-flow br0 \
  "table=0,priority=100,arp,arp_op=1,\
   actions=move:NXM_OF_ETH_DST[]->NXM_NX_REG0[0..47],\
           move:NXM_OF_ETH_SRC[]->NXM_OF_ETH_DST[],\
           move:NXM_NX_REG0[0..47]->NXM_OF_ETH_SRC[],\
           IN_PORT"
```

Bit-range partial copy:

```
ovs-ofctl add-flow br0 \
  "priority=100,actions=move:OXM_OF_METADATA[12..15]->NXM_NX_REG1[0..3]"
```

**Caveat + version drift.** `move` không có error nếu rule không match field source — datapath đọc value undefined. Lỗi phổ biến: bit-range mismatch (`move:reg0[0..7]->reg1[0..15]` → parser reject `bit count mismatch: 8 != 16`). Ở OF 1.5 controller modern, recommendation là dùng `copy_field` syntax thay vì `move` để portable cross-vendor. So với hệ khác: P4 `meta.field_a = meta.field_b;` direct assignment compile-time; Linux tc `pedit munge` chỉ ghi constant không có copy field-to-field; iptables `MARK --set-xmark` chỉ load constant.

### 4.9.37.2 `copy_field:src->dst` — phiên bản chuẩn OF 1.5+ của move

**Khái niệm.** `copy_field:SRC[range]->DST[range]` là biến thể chuẩn của `move`, tương đương semantically nhưng wire encoding khác: dùng `OFPAT_COPY_FIELD` (type 28) chuẩn OF 1.5 hoặc `ONFACT_COPY_FIELD` (experimenter ID 0x4F4E4600 ONF, subtype 3200) cho OF 1.3-1.4. OVS chấp nhận cả `move:` và `copy_field:` ở CLI, parser map vào cùng `struct ofpact_reg_move`; chỉ khác ở encode-time chọn raw type nào dựa trên OF version.

**Nguồn gốc + động cơ.** OF 1.5 (2014) thêm `OFPAT_COPY_FIELD` để chuẩn hoá điều mà NXM đã làm từ trước, tránh lock-in vendor extension. ONF cũng publish làm extension cho OF 1.3-1.4 (qua `experimenter` action) để switch ONF-compliant nhưng chưa upgrade lên 1.5 vẫn dùng được. OVS implement đầy đủ cả 3 wire form (`OFPAT_RAW15_COPY_FIELD`, `ONFACT_RAW13_COPY_FIELD`, `NXAST_RAW_REG_MOVE`).

**Wire encoding chi tiết.** Struct `ofp15_action_copy_field` (dòng 2429-2443, `OFP_ASSERT == 16`) có `n_bits`, `src_offset`, `dst_offset`, `pad2[4]` rồi follow 2 OXM header (4-8 byte mỗi cái). Struct `onf_action_copy_field` (dòng 2446-2463, `OFP_ASSERT == 24`) lớn hơn vì có experimenter header `ovs_be32 vendor=0x4F4E4600 + ovs_be16 exp_type=3200` rồi mới body. Decode unified qua `decode_copy_field__` helper dòng 2576 — parse src/dst offset + n_bits, populate `struct ofpact_reg_move`, set `move->ofpact.raw = ONFACT_RAW13_COPY_FIELD` để encode-back đúng wire form.

So sánh nhanh 3 wire form hiện hành trong upstream OVS 2.17:

| Form | Raw type | Type ID | Sizeof struct | Khi nào emit |
|------|----------|---------|---------------|--------------|
| OF 1.5 chuẩn | `OFPAT_RAW15_COPY_FIELD` | 28 (action_type) | 16 byte + OXM follow | OF 1.5+ negotiated |
| ONF 1.3-1.4 ext | `ONFACT_RAW13_COPY_FIELD` | 3200 (experimenter subtype) | 24 byte + OXM follow | OF 1.3-1.4 + ONF capability |
| Nicira NXM cũ | `NXAST_RAW_REG_MOVE` | NXAST subtype | 16 byte + OXM follow | OF 1.0-1.2 hoặc default fallback |

**Cú pháp + version negotiation.** CLI: `actions=copy_field:NXM_NX_REG0[0..7]->NXM_NX_REG1[0..7]`. Khi controller speaks OF 1.5: emit `OFPAT15_COPY_FIELD`. Khi speak OF 1.3-1.4 nhưng có ONF extension capability: emit `ONFACT13_COPY_FIELD`. Khi speak OF 1.0-1.2: emit `NXAST_RAW_REG_MOVE` (Nicira).

Engineer dùng cờ `-O OpenFlow15` của `ovs-ofctl` để force protocol version 1.5 và verify wire format đúng:

```
ovs-ofctl -O OpenFlow15 add-flow br0 \
  "table=0,priority=100,ip,actions=copy_field:NXM_NX_REG0[0..15]->NXM_NX_REG1[0..15],output:2"

ovs-ofctl -O OpenFlow15 dump-flows br0
```

`format_REG_MOVE` luôn in `move:` prefix bất kể wire encoding gì — tức `dump-flows` không phân biệt `move` vs `copy_field` về mặt hiển thị, engineer phải verify wire encoding qua `ovs-ofctl monitor br0 -P openflow15` để xem raw bytes của action.

**Caveat thực tế.** Khác biệt phiên bản subtle: OVS controller có thể nhận `copy_field` từ remote OF 1.5 controller, decode xong set `raw=ONFACT_RAW13_COPY_FIELD` (mặc dù decode từ OFPAT_RAW15), rồi khi encode lại có thể fallback sang ONF 1.3 form nếu controller downgrade. Nếu engineer test interop với Faucet/Ryu/ODL: bật OpenFlow 1.5 explicit qua `set Bridge br0 protocols=OpenFlow15` và verify qua `ovs-ofctl -O OpenFlow15 dump-flows`. So với hệ khác: P4 không có khái niệm version negotiation — compile-time `header.field_a = header.field_b;` cố định; ODL+OVSDB controller có thể prefer ONF extension form vì backward-compat.

### 4.9.37.3 `output:NXM_NX_REG0` — output tới port number đọc từ register

**Khái niệm.** `output:FIELD[range]` (variant indirect của `output:N`) đọc port number từ subfield register, không phải hardcoded. Phổ biến nhất: `output:reg0` hoặc `output:NXM_NX_REG0[0..15]`. Datapath đọc giá trị 16-bit (hoặc 32-bit từ OF 1.1+) từ vị trí subfield → dùng làm `OFPP_*` port destination. Cho phép pipeline tính toán port động (ECMP hash, load-balance, NAT remap) rồi forward 1 lần cuối.

**Nguồn gốc.** Open vSwitch 1.3 introduce như NXM extension (NXAST_OUTPUT_REG type subtype). Khi OF 1.1+ thêm 32-bit port number, OVS thêm `NXAST_OUTPUT_REG2` (type subtype 32) hỗ trợ 32-bit src + 64-bit experimenter OXM. Trong upstream code `lib/ofp-actions.c` v2.17, struct `nx_action_output_reg` (dòng 1192-1205, `OFP_ASSERT == 24`) có `ofs_nbits` (encode `(ofs<<6)|(n_bits-1)`), `src` (NXM header 4 byte), `max_len` (bytes gửi nếu port là OFPP_CONTROLLER), `zero[6]` reserved zero. Struct `nx_action_output_reg2` (dòng 1212-1226) tổ chức lại để chứa OXM header 4-8 byte trong `pad[10]` — phục vụ experimenter OXM dài hơn.

**Cú pháp parse trong OVS branch-2.17.** Function `parse_OUTPUT_REG` ở dòng 1317 chỉ wrap `parse_OUTPUT` chung (dòng phía trên) — nghĩa là CLI parser khi gặp `output:<token>` decide token là port number numeric hay field name. Nếu là field → emit ofpact_output_reg với src + max_len; nếu là port number → emit ofpact_output thường. Encode `encode_OUTPUT_REG` dòng 1287 chọn `NXAST_OUTPUT_REG` (basic) khi src dùng standard NXM header; chọn `NXAST_OUTPUT_REG2` khi src là experimenter OXM hoặc khi `move->ofpact.raw == NXAST_RAW_OUTPUT_REG2` (decode trước). Decode `decode_NXAST_RAW_OUTPUT_REG` dòng 1229 enforce `is_all_zeros(naor->zero, 6)`, fail `OFPERR_OFPBAC_BAD_ARGUMENT` nếu reserved bytes non-zero (vendor mismatch).

**Cú pháp + ví dụ thực tế.** ECMP hash chọn 1 trong 4 uplink:

```
ovs-ofctl add-flow br0 \
  "table=0,priority=100,ip,\
   actions=multipath(symmetric_l4,1024,modulo_n,4,32,NXM_NX_REG0[0..15]),\
           output:NXM_NX_REG0[0..15]"
```

Load constant rồi output indirect:

```
ovs-ofctl add-flow br0 \
  "priority=100,in_port=1,actions=load:0x3->NXM_NX_REG0[0..15],output:NXM_NX_REG0[0..15]"
```

(Tương đương `output:3` nhưng đi qua register — pattern thường dùng khi port số được tính trong table trước rồi output table sau.)

**Caveat + lỗi thường gặp.** Field width phải đủ chứa port number: `output:NXM_NX_REG0[0..7]` chỉ chứa 8 bit (max port 255 cho OF 1.0, không đủ cho OF 1.1+ 32-bit port). Nếu port number trong register = 0 hoặc invalid (port không tồn tại) → datapath drop silent. Verify qua `ovs-appctl ofproto/trace br0 in_port=1,...` thấy `output:N` trong trace path. So với hệ khác: P4 `standard_metadata.egress_spec = meta.computed_port;` direct field assignment; Linux tc `mirred redirect dev` không có indirect — phải tạo nhiều rule cho từng port; iptables `--to-destination` không tương đương vì là L3 NAT khác semantic.

### 4.9.37.4 `push:reg0` — đẩy giá trị field/register lên stack per-packet (NXM)

**Khái niệm.** `push:FIELD[range]` đẩy giá trị hiện tại của subfield lên đỉnh stack per-packet của OVS. Stack lifetime giới hạn trong 1 packet đang xử lý qua pipeline; reset rỗng giữa các packet. Bit width của entry stack = bit width của subfield. Action không tốn register slot persistent — pattern "save-modify-restore" dùng push/pop thay vì allocate register tạm trong reg0..reg15.

**Nguồn gốc.** Nicira NXAST_STACK_PUSH extension OVS 1.11+ (2013), không có trong OpenFlow standard cho tới OF 1.5 (vẫn chưa standardize stack). Trong code v2.17, struct `nx_action_stack` (chia sẻ giữa push và pop) có `offset` (bit ofs trong field), follow OXM header + `n_bits` 16-bit cuối. Encode unified qua `encode_STACK_op` (dòng 3514) dùng helper, push/pop chỉ khác `put_NXAST_STACK_PUSH` vs `put_NXAST_STACK_POP`.

**Cú pháp parse trong OVS branch-2.17.** Function `parse_STACK_PUSH` ở dòng 3544 chỉ wrap `nxm_parse_stack_action` từ `lib/nx-match.c` (parser dùng chung cho cả push + pop, phân biệt qua `ofpact_put_STACK_PUSH` vs `ofpact_put_STACK_POP`). Decode `decode_NXAST_RAW_STACK_PUSH` dòng 3490 gọi `decode_stack_action` rồi `nxm_stack_push_check` để validate field readable. Format `format_STACK_PUSH` dòng 3556 in `push:` prefix kèm subfield notation qua `nxm_format_stack_push`.

**Cú pháp + ví dụ thực tế.** Save-modify-restore in_port:

```
ovs-ofctl add-flow br0 \
  "table=0,priority=100,in_port=1,\
   actions=push:NXM_OF_IN_PORT[],\
           load:0->NXM_OF_IN_PORT[],\
           resubmit(,1),\
           pop:NXM_OF_IN_PORT[],\
           output:2"
```

Lưu reg0 trước khi compute hash phá huỷ:

```
ovs-ofctl add-flow br0 \
  "priority=100,tcp,actions=push:NXM_NX_REG0[],\
                            multipath(symmetric_l4,1024,modulo_n,4,32,NXM_NX_REG0[0..15]),\
                            note:01.02,\
                            pop:NXM_NX_REG0[],\
                            output:NXM_NX_REG0[0..15]"
```

(Wait — pattern này có lỗi semantic vì pop xảy ra trước output sẽ destroy hash result. Engineer phải hiểu order; thực tế dùng pattern push để bảo toàn giá trị KHÔNG cần dùng cho output sau, ví dụ save metadata trước khi resubmit table khác có ghi đè reg0.)

**Caveat + lỗi thường gặp.** Stack max depth giới hạn (OVS hardcode 64 entry/packet trong source `lib/ofp-actions.c`). Push quá depth → `OFPERR_NXBAC_STACK_TOO_DEEP`. Pattern phổ biến chỉ ≤ 4 push depth (dùng cho save eth_src + eth_dst + reg0 + in_port). So với hệ khác: P4 không có stack semantic — phải dùng metadata field phụ; eBPF program có per-CPU map nhưng không phải LIFO; tc filter không có stack concept hoàn toàn. Stack push/pop là feature unique của OVS NXM, chưa có equivalent trong hardware switch (Broadcom Trident programmable nhưng register architecture khác).

### 4.9.37.5 `pop:reg0` — lấy giá trị từ stack vào field/register (NXM)

**Khái niệm.** `pop:FIELD[range]` lấy entry trên đỉnh stack per-packet, gán vào subfield, rồi xoá entry khỏi stack. Bit width entry pop = bit width subfield đích (mismatch → `OFPERR_NXBAC_STACK_BAD_TYPE` action error, datapath drop packet). Pop khi stack rỗng → `OFPERR_NXBAC_STACK_UNDERFLOW`. Action pair với `push`; mỗi push phải có pop tương ứng để tránh stack leak.

**Nguồn gốc.** Cùng OVS 1.11+ với push, NXAST_STACK_POP extension. Cùng struct `nx_action_stack` chia sẻ wire encoding với push (chỉ khác subtype). Decode `decode_NXAST_RAW_STACK_POP` dòng 3502 gọi `decode_stack_action` rồi `nxm_stack_pop_check` để validate field writable (không thể pop vào read-only field như `eth_type`).

**Cú pháp parse trong OVS branch-2.17.** Function `parse_STACK_POP` ở dòng 3550 chỉ wrap `nxm_parse_stack_action` (chung với push). Encode `encode_STACK_POP` dòng 3537 gọi `put_NXAST_STACK_POP` (subtype khác push, body identical). Format `format_STACK_POP` dòng 3563 in `pop:` prefix kèm subfield notation. `check_STACK_POP` (sau dòng 3573) gọi `nxm_stack_pop_check` validate context — mismatch type/width error.

**Cú pháp + ví dụ thực tế.** Restore in_port sau resubmit:

```
ovs-ofctl add-flow br0 \
  "table=1,priority=50,actions=pop:NXM_OF_IN_PORT[],output:NXM_OF_IN_PORT[]"
```

Save-modify-restore reg0 cho computation tạm:

```
ovs-ofctl add-flow br0 \
  "priority=100,tcp,actions=push:NXM_NX_REG0[],\
                            load:0xdeadbeef->NXM_NX_REG0[],\
                            resubmit(,2),\
                            pop:NXM_NX_REG0[],\
                            output:3"
```

(Save reg0 hiện tại → load constant temp cho table 2 dùng → pop restore reg0 ban đầu trước khi output, để rule downstream đọc reg0 vẫn thấy giá trị gốc.)

**Caveat + debug.** Lỗi điển hình: pop không có push pair (`mismatched stack`), OVS `ofproto/trace` báo `stack underflow` — pipeline drop packet. Verify qua `ovs-appctl ofproto/trace br0 'in_port=1,tcp'` xem trace có hiển thị `pop:NXM_NX_REG0[]` thành công + value pop ra hay không. Bit-range mismatch: `push:reg0[0..15]` rồi `pop:reg0[0..7]` → error vì entry stack 16-bit không pop được vào field 8-bit. So với hệ khác: P4 không có pop semantic native; eBPF tail call có thể giả lập state save nhưng không LIFO; hardware switch carrier (Cisco Nexus) không expose stack action — chỉ có register read/write thuần.

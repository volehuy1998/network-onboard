## Learning objectives

After completing this part, you will be able to:

## Context: ONF's first spec and the shift to extensible match

Part 3.2 analysed ONF's formation on 2011-03-21 and the IP transfer of the OpenFlow spec from Stanford. Part 4.0 analysed OF 1.1 (2011-02-28) as the final spec shepherded by Stanford. OF 1.2 is the **first spec released by ONF** after the working-group process fully moved into ONF governance. Released on 2011-12-05, nine months after ONF formation.

Beyond the organisational meaning (ONF's first), OF 1.2 carries an important architectural shift: from the **fixed 12-tuple match** of OF 1.0 to 1.1 to **OXM TLV extensible match**. This shift is not cosmetic; it is an acknowledgement that packet-header diversity in modern networking (ARP, ICMP variants, MPLS, metadata) does not fit into a hardcoded 12 fields, and that an extension mechanism is needed to future-proof the protocol.

OF 1.2 is also the first version with **controller roles** formalised for HA deployment. Before 1.2, multi-controller setups were implementation-defined (not standard). OF 1.2 introduces the EQUAL, MASTER, and SLAVE model with the generation_id protocol for atomic role transitions, enabling production HA.

## 4.1.1 OF 1.2 released 2011-12-05: the first spec published by ONF

### Timeline and ONF process

OpenFlow Switch Specification Version 1.2 officially released on **2011-12-05** with wire protocol version **0x03**. The 85-page document (up from the 56 pages of 1.1) reflects the expanded scope. The copyright page reads:

> *"OpenFlow Switch Specification, Version 1.2 (Wire Protocol 0x03), December 5, 2011. Copyright (c) 2011 Open Networking Foundation. All rights reserved."*

This is the first spec to record copyright as ONF instead of Stanford. Process-wise, ONF's Technical Steering Committee (TSC) reviewed the draft from the Forwarding Abstractions Working Group (FAWG) and approved the release. This is proof-of-concept that ONF can operate the spec process (working group, TSC, member vote, publish), not just rebrand Stanford's work.

The current archive is at:

```
https://opennetworking.org/wp-content/uploads/2014/10/openflow-spec-v1.2.pdf
```

### Motivation: extending match fields beyond the 12-tuple

During the 2010 to 2011 deployment phase, vendors and researchers identified gaps in the OF 1.0 to 1.1 match structure. Several specific pain points:

**Gap 1: ARP details beyond just the Ethernet type.** OF 1.0 matches `dl_type=0x0806` to know that the packet is ARP, but it cannot match ARP fields (op, sender IP, target IP). Implementing ARP security rules (anti-spoofing, ARP-flood rate limiting) is impossible without ARP field matching.

**Gap 2: granular ICMP type and code matching.** OF 1.0 matches `nw_proto=1` to know the packet is ICMP, but `tp_src` and `tp_dst` are reused for ICMP_TYPE and ICMP_CODE without clear naming. Confusing and error-prone.

**Gap 3: TCP flags matching.** Firewalls and IDS need to match specific TCP flags (SYN flood detection equals match `tcp_flags=SYN && !ACK`). OF 1.0 has no TCP flags field, forcing a workaround through DPI.

**Gap 4: Cross-table metadata.** The multi-table pipeline of OpenFlow 1.1 requires a mechanism to pass state across tables, but no standardised way exists. Vendor extensions (OVS register) filled the gap with a vendor-specific approach.

**Gap 5: Future-proofing for new protocols.** A hard-coded 12-tuple means each time a new protocol appears (VXLAN, NVGRE, Geneve later), a spec revision is required. Not scalable.

OXM TLV match addresses all five gaps with a single unified mechanism: variable-length TLV encoding and an extensible class space.

## 4.1.2 OXM TLV format

### Structure

Spec 1.2 §7.2.3 defines the 32-bit OXM TLV header:

| Offset | Field | Size (bit) | Description |
|--------|-------|------------|-------------|
| 0-15 | oxm_class | 16 | Match class (0x8000=OFPXMC_OPENFLOW_BASIC, >0xFFFF=vendor) |
| 16-22 | oxm_field | 7 | Field identifier within the class |
| 23 | HM (has_mask) | 1 | 1 = value plus mask, 0 = value only |
| 24-31 | oxm_length | 8 | Length of payload in bytes |

Following the header is the payload: value (oxm_length bytes) plus optional mask (if HM=1, an additional oxm_length bytes of mask).

### OXM classes

**OFPXMC_OPENFLOW_BASIC (0x8000)**: ONF-standardised fields. Field identifiers 0 to 40 are spec-defined.

**OFPXMC_EXPERIMENTER (0xFFFF)**: vendor extensions. Followed by a 32-bit experimenter ID (an IANA-assigned Private Enterprise Number) to identify the vendor, then field-specific payload.

**OFPXMC_NXM_0 (0x0000) and OFPXMC_NXM_1 (0x0001)**: legacy Nicira Extensible Match classes, supported for backward compatibility with OVS switches running OF 1.0-era NXM. Mostly deprecated in OF 1.2 onward.

### Common fields

Spec 1.2 §7.2.3.8 lists the Basic class fields. Selected entries:

| oxm_field | Name | Size (bit) | Description |
|-----------|------|------------|-------------|
| 0 | OXM_OF_IN_PORT | 32 | Ingress switch port |
| 1 | OXM_OF_IN_PHY_PORT | 32 | Physical port (pre-virtualisation) |
| 2 | OXM_OF_METADATA | 64 | Table metadata (cross-table) |
| 3 | OXM_OF_ETH_DST | 48 | Ethernet dest MAC |
| 4 | OXM_OF_ETH_SRC | 48 | Ethernet source MAC |
| 5 | OXM_OF_ETH_TYPE | 16 | EtherType |
| 6 | OXM_OF_VLAN_VID | 12 | VLAN ID |
| 7 | OXM_OF_VLAN_PCP | 3 | VLAN Priority |
| 8 | OXM_OF_IP_DSCP | 6 | IP DSCP |
| 11 | OXM_OF_IP_PROTO | 8 | IP protocol |
| 12 | OXM_OF_IPV4_SRC | 32 | IPv4 source |
| 13 | OXM_OF_IPV4_DST | 32 | IPv4 dest |
| 14 | OXM_OF_TCP_SRC | 16 | TCP source port |
| 15 | OXM_OF_TCP_DST | 16 | TCP dest port |
| 21 | OXM_OF_ARP_OP | 16 | ARP opcode |
| 22 | OXM_OF_ARP_SPA | 32 | ARP sender protocol address |
| 23 | OXM_OF_ARP_TPA | 32 | ARP target protocol address |
| 24 | OXM_OF_ARP_SHA | 48 | ARP sender hardware address |
| 25 | OXM_OF_ARP_THA | 48 | ARP target hardware address |
| 36 | OXM_OF_MPLS_LABEL | 20 | MPLS label |
| 37 | OXM_OF_MPLS_TC | 3 | MPLS traffic class |

OF 1.2 spec defines 40 basic fields. OF 1.3 adds four more (PBB fields plus metadata variants). OF 1.5 adds the packet_type OXM field.

### Origin: the NXM legacy

OXM format is not an invention from scratch by ONF. It inherits almost verbatim from **NXM (Nicira Extensible Match)**, the format that Open vSwitch had used since 2008 to 2009 to extend OF 1.0 match beyond the 12-tuple. The original NXM design was developed by Ben Pfaff and Justin Pettit (both at Nicira) after Nicira analysed the pain points of the 12-tuple.

When ONF FAWG designed the OF 1.2 match format, the decision was straightforward: adopt the NXM concept with minor cleanup (rename the NXM_NX_* prefix to OXM_OF_*, tweak numbering). Backward compatibility was intentional: vendors who had implemented OVS plus NXM did not need to rewrite the matching engine, only adapt the wire-format header. The transition was frictionless.

This is a recurring pattern in OpenFlow history: a proven idea from research or open-source (NXM in OVS) is formalised into the official spec. OXM is the reincarnation of NXM, just as OF 1.0 is the reincarnation of GSMP (Part 2.0) and P4 is the reincarnation of Active Networking (Part 2.1).

> **Key point.** OXM TLV extensible matching is a more important architectural decision than any feature addition. From OF 1.2 onward, adding a new match field does not require a breaking spec revision; only the registration of a new oxm_field identifier. OF 1.3 added PBB, OF 1.5 added packet_type, neither breaking compatibility with 1.2 implementations. This extensibility is essential for long-term spec viability.

## 4.1.3 Match field extensions: ARP, ICMP, TCP flags, metadata

### ARP detail fields

OF 1.0 match has only `dl_type=0x0806` (detect ARP) and does not allow inspecting ARP fields inside. OF 1.2 adds five OXM fields for ARP:

- **ARP_OP (OXM_OF_ARP_OP)**: ARP opcode. Value 1 = Request, 2 = Reply.
- **ARP_SPA (Sender Protocol Address)**: IP address of the sender (advertised in the ARP packet).
- **ARP_TPA (Target Protocol Address)**: IP address of the target the sender is asking about.
- **ARP_SHA (Sender Hardware Address)**: MAC address of the sender.
- **ARP_THA (Target Hardware Address)**: MAC address of the target (often 00:00:00:00:00:00 in a Request).

Use cases:

**Anti-ARP-spoofing**: a flow rule allows only ARP Reply with an SHA matching the MAC assigned to the port. If the SHA does not match, drop.

```
match: eth_type=0x0806, arp_op=2, in_port=10
where arp_sha = <MAC-port-10-authorized>
  -> action: forward normal
else -> action: drop
```

**ARP rate limiting**: count ARP Requests per source MAC; if a threshold is exceeded, drop. Prevents ARP flood attacks.

**ARP-specific policy**: VLAN A allows ARP for 10.0.0.0/24, VLAN B allows ARP for 192.168.0.0/16. Match ARP_TPA plus VLAN, drop if not matching policy.

### ICMP detail fields

Similar to ARP, OF 1.2 adds:

- **ICMP_TYPE (OXM_OF_ICMPV4_TYPE)**: ICMP message type. Value 8 = Echo Request, 0 = Echo Reply, 3 = Destination Unreachable, 11 = Time Exceeded.
- **ICMP_CODE (OXM_OF_ICMPV4_CODE)**: sub-type within the type. For example Type=3, Code=4 is "Fragmentation Needed".

Use cases:

**ICMP rate limiting**: ping flood detection. Match type=8 (Echo Request), count per source, drop if a threshold is exceeded.

**Allow specific ICMP types**: a firewall allows Echo Request and Reply (ping works) plus Destination Unreachable (path MTU discovery), drop others.

**PMTUD preservation**: ensure ICMP Type=3 Code=4 (Fragmentation Needed) is not dropped by the firewall, which is critical for Path MTU Discovery correctness.

### TCP flags field

OF 1.2 adds **OXM_OF_TCP_FLAGS** (12-bit match, 12-bit mask). The TCP header flags field includes URG, ACK, PSH, RST, SYN, FIN, ECE, CWR, NS, plus three reserved bits.

Use cases:

**SYN flood detection**: match TCP packets with only the SYN flag set (no ACK). Rate-limit per source, drop excessive rate.

```
match: eth_type=0x0800, ip_proto=6, tcp_flags=0x002/0x017
# Mask 0x017 = SYN + ACK + FIN + RST + URG bits
# Value 0x002 = only SYN set
  -> action: meter:1 (rate limit), forward
```

**Established connection tracking**: match packets with the ACK flag set (established traffic). Allow them to bypass the stateful firewall for performance.

**Null scan detection**: match TCP packets with all flags=0 (an unusual pattern indicative of a scan). Drop and log.

### Metadata field

**OXM_OF_METADATA** (64-bit): the table metadata field introduced in OF 1.1, formalised in 1.2 OXM syntax. Used for cross-table state passing:

- Table 0 WRITE_METADATA=0x01 (for example, a tenant ID).
- Table 1 matches metadata=0x01, and routes further.

Metadata persists across the pipeline and is not reset between tables. Enables multi-table state passing without regex tricks.

Overall: OF 1.2 match extensions transform OpenFlow from a basic packet filter (12-tuple) into a general-purpose policy engine that can express complex security rules, QoS decisions, and multi-table state machines.

## 4.1.4 Controller roles: EQUAL, MASTER, SLAVE

### HA deployment context

In production deployment, a single OpenFlow switch typically has multiple controllers configured for HA:

- **Primary controller**: usually handles all traffic.
- **Backup controller**: standby, takes over when the primary is down.
- **Observer controller**: monitor-only, no programming.

Before OF 1.2, the spec did not define role semantics. Multiple controllers connected to a switch all had equal access; each controller could send FLOW_MOD, receive PACKET_IN, and other messages. There was no coordination mechanism, with a risk of conflict (two controllers installing contradictory flows).

OVS and several controller vendors implemented proprietary role negotiation, but they were not interoperable. Standardisation was required.

### Three roles defined

Spec 1.2 §6.3.5 formalises three controller roles:

**EQUAL (default)**: all controllers have equal access. Each controller:
- May send controller-to-switch messages (FLOW_MOD, PACKET_OUT, and so on).
- Receives async messages (PACKET_IN, FLOW_REMOVED, and so on).
- Receives error messages individually (not shared).

Use case: simple development, monitoring scenarios, or when the application has external coordination already.

**MASTER**: exactly one controller per switch. The Master:
- Has full read-write access.
- Receives all async messages.
- Receives error messages.
- The switch only accepts modifications from the Master controller.

Use case: production deployment with a single authoritative controller.

**SLAVE**: read-only observer. The Slave:
- Cannot send FLOW_MOD, PACKET_OUT, GROUP_MOD, METER_MOD, and so on.
- May send FEATURES_REQUEST, STATS_REQUEST, BARRIER_REQUEST (query only).
- Does NOT receive PACKET_IN, FLOW_REMOVED, or PORT_STATUS async messages.
- Does receive Error messages on its own requests.

Use case: idle backup controller, monitoring controller, or a distributed controller (ONOS) where one node is a reader.

### generation_id: preventing race conditions

When a switch has multiple controllers and the MASTER role changes, the transition must be atomic (no split-brain with two MASTERs simultaneously). OF 1.2 introduces the **generation_id** concept:

- Each role-change request includes a generation_id (a 64-bit counter monotonically increasing).
- The switch tracks the latest generation_id accepted.
- A request with generation_id less than or equal to the current value is rejected.

Protocol:
1. Controller A requests MASTER with generation_id=5.
2. Switch accepts; current generation_id = 5.
3. Controller B requests MASTER with generation_id=4 (stale).
4. Switch rejects with an error (ROLE_STALE).
5. Controller B must query the current generation_id and request again with a higher number.

Concurrent role requests serialise via generation_id ordering. No split-brain is possible.

OFPT_ROLE_REQUEST and OFPT_ROLE_REPLY messages are specified in §A.3.10:
- Role: OFPCR_ROLE_NOCHANGE (query only), OFPCR_ROLE_EQUAL, OFPCR_ROLE_MASTER, OFPCR_ROLE_SLAVE.
- Generation ID: 64-bit.

### Production deployment patterns

**Active-standby HA**: two controllers, the primary holds MASTER, the backup holds SLAVE. If the primary goes down, the backup detects it through echo timeout and requests a role transition to MASTER. Failover time is typically 5 to 10 seconds.

**Cluster with leader election**: three or more controllers with distributed coordination (ZooKeeper, etcd). The cluster elects a leader; the leader requests MASTER on every switch while others remain SLAVE. ONOS Atomix uses this pattern (Part 7.2).

**Multi-region deployment**: controllers in two regions, each region holds MASTER for its local switches. No single global MASTER.

> **Key point.** Controller roles from OF 1.2 are a prerequisite for every production SDN HA deployment. Before 1.2, multi-controller setups were ad-hoc and unreliable. From 1.2, protocol-level guarantees about role atomicity allow SDN controller clusters (ODL, ONOS) to achieve production-grade availability. This is an "enterprise readiness" feature, not glamorous but essential.

## 4.1.5 Migration OF 1.0 to 1.2: no wire backward compatibility

### Wire incompatibility

A 1.0 switch and a 1.2 switch cannot directly communicate with a controller of the other version. Reasons:

**Different header version**: OF 1.0 = 0x01, OF 1.2 = 0x03. The message header starts with the version byte; a switch and controller with a mismatch reject immediately.

**Completely different match structure**: OF 1.0 uses a fixed 40-byte ofp_match struct. OF 1.2 uses a variable-length OXM TLV list. Parser logic differs and is not compatible at the binary level.

**New instruction framework**: OF 1.0 has a list of actions. OF 1.2 has instructions (WRITE_ACTIONS, APPLY_ACTIONS, and so on). Semantics differ.

**Message type additions**: OF 1.2 has OFPT_ROLE_REQUEST and OFPT_ROLE_REPLY which do not exist in 1.0. A 1.0 controller does not know these messages.

### Negotiation through OFPT_HELLO

Spec 1.2 §6.3.1 defines version negotiation through a HELLO handshake:

1. **Controller TCP connect**: TCP connection established.
2. **Both sides send HELLO**: each side sends OFPT_HELLO with a supported-versions bitmap. The new message format in 1.2 allows advertising multiple versions (a bitmap), instead of a single version as in 1.0.
3. **Negotiate the highest common**: both sides compare the bitmaps and select the highest version both support. If there is no common version, HELLO_FAILED error and the connection closes.
4. **Continue with the negotiated version**: subsequent messages use the agreed version.

Bitmap format: 32 bits per chunk, bit N set means support for version N+1. For example, bitmap 0x0000000A means support for version 2 (OF 1.1) and version 4 (OF 1.3), but not version 1 (OF 1.0) or version 3 (OF 1.2). Controller and switch may advertise an arbitrary subset of supported versions.

### Deployment reality

**OVS during 2012 to 2014** (OVS 1.9 to 2.3): supports OF 1.0 and OF 1.3 simultaneously, skipping 1.1 and 1.2. Reasons:
- OF 1.0 for backward compatibility with existing controllers.
- OF 1.3 for new features (LTS version).
- OF 1.1 and 1.2 had short lifetimes, low adoption, and did not justify engineering effort.

**Hardware vendors**: HP ProCurve and NEC support 1.0 and 1.3 mostly, with 1.1 and 1.2 partial or skipped. Pica8 and Centec whitebox switches focus on 1.3.

**Controller frameworks**: NOX 2008 supports only 1.0. POX supports only 1.0. Ryu supports 1.0 to 1.5. Floodlight supports 1.0 plus 1.3. OpenDaylight from Helium (2014-10) focuses on 1.3.

Result: OF 1.2 is a "transient version" in history. It was released to formalise OXM and controller roles, but market adoption jumped quickly from 1.0 to 1.3 (LTS). 1.2-specific deployment is rare in production. Part 4.2 will analyse OF 1.3 and the reasons it became the de-facto LTS.

## Guided Exercise: implement OXM TLV match with OVS (audit P4.B4.1 hands-on)

**Goal:** experiment with extensible OXM matching on OVS, and verify ARP and TCP flags matching that OF 1.0 does not support.

**Steps:**

1. Create a bridge with OF 1.3:
   ```bash
   $ sudo ovs-vsctl add-br br-oxm -- set bridge br-oxm protocols=OpenFlow13
   ```
2. Install a rule matching ARP with OXM fields (OF 1.0 does not support ARP op):
   ```bash
   # Drop ARP request from unknown MAC
   $ sudo ovs-ofctl -O OpenFlow13 add-flow br-oxm \
       "table=0,priority=100,arp,arp_op=1,arp_spa=10.0.0.0/24,arp_sha=aa:bb:cc:00:00:01,actions=drop"

   # Verify OXM encoded correctly
   $ sudo ovs-ofctl -O OpenFlow13 dump-flows br-oxm
   # Output shows arp fields inline
   ```
3. Install a rule matching TCP flags (detect SYN flood):
   ```bash
   $ sudo ovs-ofctl -O OpenFlow13 add-flow br-oxm \
       "table=0,priority=100,tcp,tcp_flags=+syn-ack,actions=meter:1,NORMAL"
   # +syn = SYN bit set
   # -ack = ACK bit not set
   # Match SYN-only packet (connection initiation)
   ```
4. Verify OXM wire encoding via `ovs-ofctl monitor`:
   ```bash
   $ sudo ovs-ofctl -O OpenFlow13 monitor br-oxm &
   # Generate ARP traffic and observe the OXM TLV format
   ```

**POE:**
- **Predict:** Does the OF 1.0 12-tuple allow matching `arp_op=1` directly?
- **Observe:** No. OF 1.0 has only 12 fixed fields (basic L2, L3, L4). The NXM extension is required. The OXM TLV of 1.2 standardises NXM into a spec standard.
- **Explain:** The OXM class plus field plus length structure is extensible: future protocols (VXLAN, Geneve) only need to register a new class plus field ID, with no spec revision required.

**Cleanup:** `sudo ovs-vsctl del-br br-oxm`

## 13.10.1 Tại sao DHCP phải được hiện thực trong data plane?

Trong kiến trúc truyền thống, DHCP server là một tiến trình userspace (ISC DHCPD, `dnsmasq`, Kea) chạy trên một host, lắng nghe broadcast DHCPDISCOVER trên một network interface. Mô hình này không phù hợp với OVN vì:

**Vấn đề 1 — Broadcast không đi qua được chassis:** trong overlay network, pod trên chassis A broadcast DHCPDISCOVER; nếu DHCP server chạy trên chassis B, broadcast phải được flood qua tunnel Geneve tới mọi chassis. Flooding này tốn băng thông và chỉ dùng cho DHCP.

**Vấn đề 2, Điểm fail duy nhất:** DHCP server userspace duy nhất → nếu crash, pod mới không lấy được IP. Phải triển khai cụm HA DHCP, phức tạp.

**Vấn đề 3 — Mở rộng:** cụm Kubernetes 10K pod với pod thay đổi cao (scale up/down liên tục) có thể sinh hàng nghìn DHCPDISCOVER per giây, một DHCP server userspace không đáp ứng nổi.

**Giải pháp OVN — DHCP native:** mỗi ovn-controller trên mỗi chassis tự trả lời DHCP request cho pod cục bộ. Không có broadcast đi qua chassis, không có server đơn lẻ, mở rộng tuyến tính theo số chassis. Cơ chế:
- CMS định nghĩa `DHCP_Options` trong NB DB (subnet, gateway, DNS servers, lease_time, v.v.).
- Mỗi LSP được gán `dhcpv4_options` trỏ tới DHCP_Options record.
- ovn-northd sinh Logical Flow trong table `ls_in_dhcp_options` (match DHCPDISCOVER + DHCPREQUEST) và `ls_in_dhcp_response` (sinh DHCPOFFER + DHCPACK).
- ovn-controller dịch sang OpenFlow với action `put_dhcp_opts`, action này build DHCPOFFER/DHCPACK gói tin trực tiếp trong kernel datapath, không cần userspace.

## 13.10.2 Cấu trúc table DHCP_Options

Table `DHCP_Options` trong NB DB có các field chính:
- `cidr`, subnet mà DHCP phục vụ (ví dụ `10.0.0.0/24`).
- `options`, map từ mã DHCP option → giá trị:
  - `server_id` (option 54), IP của DHCP server (thường là gateway IP).
  - `server_mac` (option đặc biệt), MAC của server, dùng làm MAC nguồn của DHCP response.
  - `lease_time` (option 51), thời gian lease (giây).
  - `router` (option 3), default gateway cho client.
  - `dns_server` (option 6), DNS server list.
  - `netmask` (option 1), subnet mask.
  - `domain_name` (option 15), domain search.
  - `mtu` (option 26), MTU cho interface.
  - `classless_static_route` (option 121), định tuyến static ngoài default.
- `external_ids`, map tự do cho CMS label.

Ví dụ cấu hình:
```
[host]# ovn-nbctl dhcp-options-create 10.0.0.0/24
<uuid_dhcp>
[host]# ovn-nbctl dhcp-options-set-options <uuid_dhcp> \
         server_id=10.0.0.1 \
         server_mac="ca:fe:ba:be:00:01" \
         lease_time=3600 \
         router=10.0.0.1 \
         dns_server="8.8.8.8,8.8.4.4" \
         netmask=255.255.255.0 \
         mtu=1500
```

**Gán DHCP options vào LSP:**
```
[host]# ovn-nbctl lsp-set-dhcpv4-options vm1 <uuid_dhcp>
```

Khi LSP được gán dhcpv4_options, ovn-northd sinh Logical Flow xử lý DHCP request từ port đó.

## 13.10.3 Logical Flow xử lý DHCP

ovn-northd sinh 3 nhóm Logical Flow cho DHCP:

**Nhóm 1, ls_in_dhcp_options (table 16):**
Match DHCPDISCOVER + DHCPREQUEST từ client, apply action `put_dhcp_opts` để build response:
```
table=16 (ls_in_dhcp_options), priority=100,
  match=(inport == "vm1" && eth.src == aa:bb:cc:dd:ee:01 && ip4.src == 0.0.0.0
         && ip4.dst == 255.255.255.255 && udp.src == 68 && udp.dst == 67),
  action=(reg0[3] = put_dhcp_opts(offerip = 10.0.0.5,
                                   server_id = 10.0.0.1,
                                   router = 10.0.0.1,
                                   dns_server = {8.8.8.8, 8.8.4.4},
                                   lease_time = 3600,
                                   netmask = 255.255.255.0);
          next;)
```
Action `put_dhcp_opts` là action tùy chỉnh của OVN, nó thay payload DHCP của gói tin hiện tại thành DHCPOFFER hoặc DHCPACK, đồng thời tráo src/dst MAC và IP để gói tin đi ngược lại về client. `reg0[3]` = 1 nếu thành công, 0 nếu fail (ví dụ offerip không thuộc subnet).

**Nhóm 2, ls_in_dhcp_response (table 17):**
Sau khi gói tin đã được biến đổi thành DHCPOFFER/DHCPACK, table 17 resubmit để gói tin đi qua pipeline ngược lại về client:
```
table=17 (ls_in_dhcp_response), priority=100,
  match=(reg0[3] == 1),
  action=(eth.dst = eth.src; eth.src = aa:aa:aa:aa:aa:aa; /* server_mac */
          ip4.src = 10.0.0.1; ip4.dst = 10.0.0.5; udp.src = 67; udp.dst = 68;
          outport = inport; flags.loopback = 1; output;)
```

**Nhóm 3 — DHCP drop (table 16 fallback):**
Nếu DHCP request không match DHCP_Options của port, drop:
```
table=16, priority=0, match=(udp.dst == 67), action=(drop;)
```

Logic này đảm bảo chỉ port có cấu hình DHCP mới nhận được response, port không có dhcpv4_options → request bị drop (ngăn chặn DHCP snooping).

## 13.10.4 DHCPv6 và stateless configuration

OVN hỗ trợ DHCPv6 qua table `DHCPv6_Options` tương tự DHCP_Options nhưng với mã option IPv6. Ngoài ra OVN hỗ trợ **SLAAC (StateLess Address AutoConfiguration)** qua Router Advertisement (RA), không cần DHCPv6 server:
- ovn-northd sinh Logical Flow gửi RA định kỳ từ Logical Router Port (LRP) có IPv6 address.
- Pod IPv6 nhận RA, tự cấu hình IPv6 address theo prefix (global unicast từ LRP + EUI-64 hoặc random).

Cấu hình SLAAC:
```
[host]# ovn-nbctl set Logical_Router_Port lrp1 ipv6_ra_configs:address_mode=slaac
[host]# ovn-nbctl set Logical_Router_Port lrp1 ipv6_ra_configs:send_periodic=true
[host]# ovn-nbctl set Logical_Router_Port lrp1 ipv6_ra_configs:max_interval=600
```

**Kết hợp DHCPv6 stateful + SLAAC stateless:** một số network cần IPv6 tự cấu hình cho phần lớn client (SLAAC) nhưng DHCPv6 cho DNS server + domain name (DHCPv6 stateless mode, chỉ options, không address). OVN hỗ trợ qua `ra_configs:address_mode=dhcpv6_stateless`.

## 13.10.5 DNS native trong OVN

OVN cũng hỗ trợ DNS resolver native trong data plane, qua table `DNS` trong NB DB. Không cần triển khai DNS server (CoreDNS, `dnsmasq`) riêng cho cluster.

**Cấu trúc table DNS:**
- `records`, map từ "fqdn" → "ipv4_list,ipv6_list" (ví dụ `"pod1.cluster.local" → "10.0.0.5"`).
- `external_ids`, label tự do.

**Gắn DNS vào Logical Switch:**
```
[host]# ovn-nbctl create DNS records={"\"vm1.ovn.local\""="10.0.0.10"}
<uuid_dns>
[host]# ovn-nbctl add Logical_Switch ls1 dns_records <uuid_dns>
```

**Logical Flow xử lý DNS query:**
Table `ls_in_dns_lookup` (table 18) match DNS query gói tin (UDP dst port 53), apply action `dns_lookup()` để tra cứu table DNS:
```
table=18 (ls_in_dns_lookup), priority=100,
  match=(udp.dst == 53),
  action=(reg0[4] = dns_lookup(); next;)
```
Action `dns_lookup` tìm fqdn từ DNS query trong table DNS, nếu tìm thấy build DNS response trong gói tin hiện tại.

Table `ls_in_dns_response` (table 19) gửi response ngược lại client:
```
table=19 (ls_in_dns_response), priority=100,
  match=(reg0[4] == 1),
  action=(eth.dst <-> eth.src; ip4.src <-> ip4.dst; udp.src <-> udp.dst;
          outport = inport; flags.loopback = 1; output;)
```

**Hạn chế DNS native:** chỉ hỗ trợ A, AAAA, PTR, SRV record. Không hỗ trợ zone transfer, DNSSEC, recursive resolution (nếu query fqdn không trong table DNS → drop, không forward lên DNS upstream). Môi trường production cần recursive DNS thường triển khai CoreDNS làm upstream cho OVN native DNS.

## 13.10.6 Tích hợp với external DHCP/DNS

Trong môi trường doanh nghiệp, DHCP/DNS thường có IPAM trung tâm (Infoblox, BIND), không thể thay thế bằng OVN native. OVN hỗ trợ hai mô hình tích hợp:

**Mô hình 1 — DHCP Relay:** OVN hiện thực `localnet` port type cho phép bridge Logical Switch với external VLAN. DHCP request từ pod broadcast trong Logical Switch, OVN flood ra localnet port tới external DHCP server. External server reply, OVN forward về pod. Mô hình này giữ DHCP server tập trung nhưng hy sinh tính phi tập trung của OVN.

**Mô hình 2 — External DNS upstream:** OVN native DNS có thể forward query không phân giải được qua upstream. OVN 22.03+ hỗ trợ `DNS.options:upstream_dns_servers`, nếu DNS native không match, forward query tới upstream. Kết hợp cache cục bộ cho tra cứu nhanh + external upstream cho full DNS namespace.

### ▶ Guided Exercise 1: Cấu hình OVN DHCP native cho Logical Switch Port

> **Trạng thái verification lab:** Output dựa trên OVN 22.03 docs; chờ kiểm chứng lab thực.

**Mục tiêu:** Chứng minh OVN cấp IP qua DHCP cho pod/VM mà không cần DHCP server userspace.

**Chuẩn bị:** Lab Ubuntu 22.04 với OVN 22.03, namespace pod gắn vào LSP.

**Bước thực hiện:**

**1.** Tạo Logical Switch + LSP:
```
[host]# ovn-nbctl ls-add ls_dhcp
[host]# ovn-nbctl lsp-add ls_dhcp vm1
[host]# ovn-nbctl lsp-set-addresses vm1 "aa:bb:cc:dd:ee:01 dynamic"
```

Lưu ý: `dynamic` báo OVN tự gán IP trong subnet của DHCP_Options.

**2.** Tạo DHCP_Options:
```
[host]# ovn-nbctl dhcp-options-create 10.0.0.0/24
<uuid>
[host]# ovn-nbctl dhcp-options-set-options $uuid \
         server_id=10.0.0.1 \
         server_mac="ca:fe:ba:be:00:01" \
         lease_time=3600 \
         router=10.0.0.1 \
         dns_server="8.8.8.8" \
         netmask=255.255.255.0
```

**3.** Gán DHCP vào LSP:
```
[host]# ovn-nbctl lsp-set-dhcpv4-options vm1 $uuid
```

**4.** Quan sát Logical Flow DHCP:
```
[host]# ovn-sbctl lflow-list ls_dhcp | grep -E "(dhcp_options|dhcp_response)"
```

Kết quả mong đợi: thấy action `put_dhcp_opts` trong table 16 và action build response trong table 17.

**5.** Tạo namespace + network interface, gán MAC từ LSP, chạy DHCP client:
```
[host]# ip netns add ns_vm1
[host]# ip link add veth_vm1 type veth peer name veth_ovs
[host]# ip link set veth_vm1 netns ns_vm1
[host]# ip netns exec ns_vm1 ip link set veth_vm1 address aa:bb:cc:dd:ee:01
[host]# ip link set veth_ovs up
[host]# ovs-vsctl add-port br-int veth_ovs -- set Interface veth_ovs \
         external_ids:iface-id=vm1
[host]# ip netns exec ns_vm1 ip link set veth_vm1 up
[host]# ip netns exec ns_vm1 dhclient -v veth_vm1
```

Kết quả mong đợi (rút gọn):
```
DHCPDISCOVER on veth_vm1 ...
DHCPOFFER of 10.0.0.5 from 10.0.0.1
DHCPREQUEST for 10.0.0.5 on veth_vm1 ...
DHCPACK of 10.0.0.5 from 10.0.0.1
bound to 10.0.0.5 -- renewal in ...
```

Pod nhận IP `10.0.0.5` (do OVN tự gán từ subnet 10.0.0.0/24), default gateway `10.0.0.1`, DNS `8.8.8.8`.

**6.** Quan sát OpenFlow counter cho DHCP rule:
```
[host]# ovs-ofctl dump-flows br-int | grep "put_dhcp_opts"
```

Thấy `n_packets` tăng mỗi lần client gửi DHCPDISCOVER/REQUEST.

**Bài học học được:**

Thứ nhất, DHCP OVN native hoàn toàn được xử lý trong OVS datapath, không có tiến trình userspace nào xử lý DHCP. Hiệu năng cao (line-rate) và phi tập trung (mỗi chassis tự trả lời DHCP cho port cục bộ).

Thứ hai, cơ chế "address_mode=dynamic" của LSP giao quyền cấp IP cho OVN (dùng IPAM, xem Phần 13.12). IP được gán trong subnet của DHCP_Options và ghi vào `Logical_Switch_Port.dynamic_addresses`.

Thứ ba, action `put_dhcp_opts` là action cấp thấp của OVS datapath, biến đổi gói tin trong kernel, không cần copy lên userspace. Đây là lý do OVN DHCP có độ trễ dưới mili giây so với dnsmasq khoảng 10ms.

**Cleanup:**
```
[host]# ip netns del ns_vm1
[host]# ovs-vsctl del-port veth_ovs
[host]# ovn-nbctl ls-del ls_dhcp
```

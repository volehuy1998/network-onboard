### ▶ Guided Exercise 1: Install flow entry qua P4Runtime với Python client

> **Trạng thái lab verification:** Output dựa trên p4runtime-shell docs; chờ verify lab thực.

**Mục đích:** Thay thế thrift-based `simple_switch_CLI` (BMv2-specific) bằng gRPC P4Runtime API (standard across targets). Đây là con đường production để controller interact với P4 data plane.

**Chuẩn bị:**
- Hoàn thành Lab 14.0-Ex1 (basic.p4 compiled, veth topology available)
- Install p4runtime_shell:
  ```bash
  pip3 install p4runtime-shell
  ```
- Cần `basic.json` và `basic.p4info.txt` từ Lab 14.0-Ex1.

**Mô hình lab:**

```
┌─────────────────────────────────────────────────────┐
│            Ubuntu 22.04 VM                          │
│                                                     │
│  ┌──────────────────────────────────┐               │
│  │    BMv2 simple_switch_grpc       │               │
│  │                                  │               │
│  │    P4 program: basic.json        │               │
│  │    gRPC server: :9559            │               │
│  └───────────┬──────────────────────┘               │
│              │ P4Runtime (gRPC/protobuf)            │
│  ┌───────────▼──────────────────────┐               │
│  │   p4runtime-shell (Python)       │               │
│  │   - LoadForwardingPipelineConfig │               │
│  │   - WriteRequest (INSERT entry)  │               │
│  │   - ReadRequest (verify)         │               │
│  └──────────────────────────────────┘               │
└─────────────────────────────────────────────────────┘
```

**Bước thực hiện:**

**1.** Start simple_switch với gRPC enabled (thay simple_switch của Lab 14.0 bằng `simple_switch_grpc`):

```bash
[host]# simple_switch_grpc -i 0@veth-0 -i 1@veth-1 \
        --no-p4 -- --grpc-server-addr 0.0.0.0:9559
```

Output kỳ vọng:

```
Calling target program-options parser
Adding interface veth-0 as port 0
Adding interface veth-1 as port 1
Server listening on 0.0.0.0:9559
```

**2.** Start p4runtime-shell:

```bash
[host]$ p4runtime-sh --grpc-addr 127.0.0.1:9559 \
                    --config basic.p4info.txt,basic.json \
                    --election-id 0,1
```

Output kỳ vọng:

```
P4Runtime sh >>>
```

**3.** Inspect loaded tables:

```python
P4Runtime sh >>> tables
```

Output kỳ vọng: list of tables với P4Info names, showing `MyIngress.l2_fwd` table.

**4.** Install flow entry via WriteRequest:

```python
P4Runtime sh >>> te = table_entry["MyIngress.l2_fwd"](action="MyIngress.forward")
P4Runtime sh >>> te.match["hdr.ethernet.dstAddr"] = "00:00:00:00:00:02"
P4Runtime sh >>> te.action["port"] = "1"
P4Runtime sh >>> te.insert()
```

Output kỳ vọng: no error, silent success.

**5.** Verify via ReadRequest:

```python
P4Runtime sh >>> for e in table_entry["MyIngress.l2_fwd"].read():
...     print(e)
```

Output kỳ vọng: show entry với match key `00:00:00:00:00:02` và action `forward(port=1)`.

**6.** Test forwarding (giống Ex1 bước 6):

```bash
[host]# ip netns exec h1 arping -c 3 -I h1-eth0 10.0.0.2
```

**Bài học học được:**

Thứ nhất, P4Runtime **schema-driven**: controller phải load P4Info (từ `basic.p4info.txt`, generated bởi p4c) để biết table names/IDs. Khác OpenFlow fixed-schema — OpenFlow controller hard-code match field names vì spec định sẵn.

Thứ hai, gRPC + protobuf transport cho P4Runtime là industry standard cho network control plane 2020+. Same protocol stack với gNMI (config/state), gRIBI (RIB injection). Unified ops tooling.

Thứ ảba, migration path: product chuyển từ `simple_switch_CLI` (thrift, BMv2-only) sang P4Runtime (gRPC, vendor-neutral). Commercial switches (Tofino, Pensando) chỉ expose P4Runtime, không thrift.

**Cleanup:**

```python
P4Runtime sh >>> exit
```

```bash
[host]# pkill -f simple_switch_grpc
[host]# ip netns del h1 h2
[host]# ip link del veth-0 2>/dev/null || true
[host]# ip link del veth-1 2>/dev/null || true
```

### ▶ Lab 2: ONOS + Stratum + BMv2 full stack demo

> **Trạng thái lab verification:** Lab complex, multi-component; chờ verify trên setup thực.

**Mục đích:** Full-stack production-like deployment — ONOS controller điều khiển 2 BMv2 switches qua Stratum (thin agent). Chứng minh P4 + P4Runtime + ONOS viable cho real SDN deployment.

**Chuẩn bị:**
- Docker + Docker Compose installed
- 8 GB RAM minimum (ONOS cluster + 2 BMv2 + Stratum agents)
- Clone ONF `next-mpls` tutorial: `git clone https://github.com/opennetworkinglab/ngsdn-tutorial.git`

**Mô hình lab:**

```
┌───────────────────────────────────────────────────────────────┐
│         Docker Compose stack (all on 1 Ubuntu VM)             │
│                                                               │
│    ┌─────────────────────────────────┐                        │
│    │    ONOS (container, 8181 GUI)   │                        │
│    │    - fabric-tofino app          │                        │
│    │    - P4Runtime driver           │                        │
│    │    - gNMI collector              │                        │
│    └─────────┬───────────┬───────────┘                        │
│              │           │                                    │
│              │ P4Runtime + gNMI                               │
│              │           │                                    │
│    ┌─────────▼──────┐  ┌─▼──────────────┐                     │
│    │  Stratum +     │  │  Stratum +     │                     │
│    │  BMv2 switch1  │  │  BMv2 switch2  │                     │
│    │  (container)   │  │  (container)   │                     │
│    └────────┬───────┘  └────────┬───────┘                     │
│             │                   │                             │
│             └─────── link ──────┘                             │
│         (virtual link via veth in docker network)             │
│                                                               │
│    Mininet topology on host drives traffic                    │
└───────────────────────────────────────────────────────────────┘
```

**Bước thực hiện:**

**1.** Cd to tutorial directory:

```bash
[host]$ cd ngsdn-tutorial
[host]$ make start
```

Output kỳ vọng (first time): pull Docker images (~5-10 GB), start containers.

**2.** Verify ONOS running:

```bash
[host]$ make onos-cli
```

Inside ONOS CLI:

```
onos> devices
id=device:leaf1, available=true, role=MASTER, type=SWITCH, ...
id=device:leaf2, available=true, role=MASTER, type=SWITCH, ...
```

**3.** Load fabric-tofino app:

```bash
onos> app activate org.onosproject.fabric
```

**4.** Verify flow rules installed by app:

```bash
onos> flows -s
deviceId=device:leaf1, flowRuleCount=50+
deviceId=device:leaf2, flowRuleCount=50+
```

**5.** Test connectivity (Mininet ping):

```bash
[host]$ make mininet-cli
mininet> h1a ping h1b
```

Output kỳ vọng: pings success, low loss.

**6.** Observe via ONOS GUI:

- Open `http://localhost:8181/onos/ui` (user: onos, password: rocks)
- Topology view: 2 switches, 1 link, 2 hosts discovered
- Flows view: flow entries installed bởi fabric app

**Output mong muốn:**

```
# ONOS device list shows 2 switches MASTER role
# Flow count > 50 per switch (fabric app rules)
# Mininet ping h1a → h1b 0% loss
# GUI shows topology correctly
```

**Bài học học được:**

Thứ nhất, Stratum là **thin switch agent**: không chạy routing protocols hoặc complex logic, chỉ expose P4Runtime + gNMI + gNOI APIs. ONOS (hoặc controller khác) là "brain".

Thứ hai, `fabric-tofino` app trong ONOS minh họa typical production P4 workflow: app compile a fabric P4 program, deploy qua P4Runtime vào switches, install flows per topology. Switch không tự learn/discover — controller push everything.

Thứ ba, CORD (Central Office Re-architected as Datacenter) architecture của AT&T based on chính setup này. Lab này cho reference tới production deployment reality.

**Cleanup:**

```bash
[host]$ make stop
[host]$ make clean
```

## R2.5.1.1 - What a Cisco engineer expects from MAC learning

Every Cisco or Juniper-trained engineer has watched the MAC address table populate in real
time using `show mac address-table dynamic`. On a real Cisco Catalyst switch, the table
lives in the ASIC's CAM (Content Addressable Memory) silicon. When a frame arrives on port
Gi0/1 with source MAC `aa:bb:cc:dd:ee:ff`, the ASIC programs a new entry in nanoseconds,
with no CPU involvement at all. A subsequent frame destined for `aa:bb:cc:dd:ee:ff` is
looked up in the same CAM table and forwarded in hardware, also with no CPU involvement.
The MAC table, the lookup, the aging sweep, and the eviction are all operations that the
silicon handles autonomously without calling out to any software layer.

```ascii
Cisco hardware switch:
  Frame IN   -->  ASIC reads src MAC  -->  CAM write (~1 ns, hardware)
  Frame IN   -->  ASIC reads dst MAC  -->  CAM lookup (~1 ns, hardware) -->  Port forward
  No CPU. No software path. No process. Pure silicon.
```

The aging timer on a Cisco switch (default 300 seconds, configurable with `mac address-
table aging-time`) is also enforced by the ASIC hardware timer. The CPU does not run any
sweep loop; the silicon handles eviction autonomously. From the Cisco engineer's
standpoint, the MAC table is a silicon data structure that the management plane can
observe but does not operate.

OVS has no ASIC. It is a software switch whose "forwarding ASIC" is a C program called
`ovs-vswitchd`. The MAC table is a C heap structure (`struct mac_learning`) allocated
inside that program's address space on the host CPU. The aging sweep is a function inside
that program's main event loop, scheduled by the same polling mechanism that handles OVSDB
watch events, OpenFlow controller messages, and netlink notifications. Every part of this
system that a Cisco switch handles in silicon, OVS handles in software, running on the
same general-purpose CPU cores that the host operating system uses for everything else.

The Linux bridge presents an intermediate case. It is also a software switch, but its MAC
table lives in kernel memory rather than in a userspace process. From the Cisco engineer's
mental model, the Linux bridge is closer to a "software ASIC running in the OS kernel"
while OVS is more like a "software ASIC running as a userspace service". The wire-level
forwarding outcome for both is identical. The structural differences are in where the
table lives, who can read it, what happens when that component crashes, and what latency
the first packet experiences on each new traffic class.

Understanding this three-way comparison (Cisco ASIC, Linux kernel bridge, OVS userspace
bridge) is the conceptual foundation for everything that follows in this lab. The
exercises use real measurements to turn this architectural description into verifiable
evidence.

## R2.5.1.2 - The OVS MAC table: a userspace heap structure

On a Cisco switch the MAC table is the CAM array inside the ASIC package. On an OVS bridge
the MAC table is `struct mac_learning`, a C heap structure inside `ovs-vswitchd`'s virtual
address space. There is no equivalent structure in the Linux kernel for an OVS bridge. The
kernel knows about OVS bridges as netdev interfaces, but it keeps no forwarding database
for them.

The type is defined in `lib/mac-learning.h`. Each learned entry is a `struct mac_entry`
containing the source MAC address, the bridge port it arrived on, and an expiry timestamp
computed from `time_now() + ml->idle_time`. The hash map is `ml->table`, keyed on (MAC,
VLAN) pairs. The hash map is a standard OVS hmap, a power-of-two open-address table with
linked collision chains.

The entry lifecycle runs through three functions in `lib/mac-learning.c`. When a new
source MAC arrives through the upcall path, `mac_learning_insert` at line 393 (tag
`v2.17.9`) allocates a new `struct mac_entry` and adds it to the hash map. When an
existing MAC is seen again on any port, `mac_learning_insert` finds the existing entry and
resets its expiry timestamp, effectively refreshing the age without creating a duplicate.
When the main loop runs `mac_learning_run` at line 638, the function walks the entire hash
map and removes every entry whose expiry timestamp has passed. Crucially,
`mac_learning_run` is not a background thread. It is a function called from the vswitchd
main poll loop on each iteration. The sweep happens when the poll loop fires, not at a
precise hardware-timer cadence.

The minimum aging time is enforced by `normalize_idle_time` at line 194, which clamps any
value below 15 seconds up to 15, and any value above 3600 seconds down to 3600. This means
that even if an operator sets `other_config:mac-aging-time=5`, the effective floor inside
`ovs-vswitchd` is 15 seconds. The clamping is silent: `ovs-vsctl get bridge br0
other_config` will show the value you set, but the actual idle time used internally is the
clamped value. An operator unaware of `normalize_idle_time` will be confused when a
5-second configured value does not produce 5-second evictions.

The function that drives MAC learning is `xlate_normal` in `ofproto/ofproto-dpif-xlate.c`
at line 3083. `xlate_normal` is the OVS software equivalent of the ASIC's per-packet
processing logic. It is invoked for every packet that causes an upcall from the kernel
datapath to `ovs-vswitchd`, which happens whenever the kernel datapath has no matching
megaflow rule for that traffic class. Inside `xlate_normal`, the call chain for learning
is:

```ascii
OVS upcall arrival (packet with no kernel megaflow match):
  |
  v
xlate_normal(ctx)                          ofproto/ofproto-dpif-xlate.c:3083
  |
  +---> update_learning_table(ctx, ...)    ofproto/ofproto-dpif-xlate.c:2731
  |       |
  |       +---> mac_learning_insert(ml, src_mac, vlan, in_port)
  |                                        lib/mac-learning.c:393
  |               stores (src_MAC, VLAN) -> in_port in struct mac_learning
  |               sets entry expiry = time_now() + ml->idle_time
  |
  +---> mac_learning_lookup(ml, dst_mac, vlan)
  |                                        lib/mac-learning.c:597
  |     returns the port for dst_mac, or NULL if not known
  |
  +---> build output action:
          if lookup hit:  unicast to the known port
          if lookup miss: flood to all ports except in_port
          install as megaflow rule in kernel datapath
```

After `xlate_normal` builds the forwarding decision and installs it as a megaflow rule in
the kernel datapath, subsequent packets in the same traffic class (same source MAC,
destination MAC, and ethernet type) are handled entirely by the kernel datapath without
any further call to `xlate_normal` or `mac_learning_lookup`. The megaflow rule encodes the
forwarding decision directly as a kernel action. The kernel executes that action at line
rate without any userspace involvement until the megaflow rule expires. This means the MAC
table is consulted exactly once per new traffic class; after that, it is bypassed entirely
until the megaflow expires and the next packet of that class causes a fresh upcall.

This two-layer structure has a non-obvious consequence: even after an FDB entry ages out
from `struct mac_learning`, existing megaflow rules in the kernel continue to forward
traffic correctly for a while, because they have the forwarding decision already embedded.
An FDB eviction does not immediately invalidate megaflow rules. The two tables operate on
different timescales and through different eviction mechanisms.

> **Key Topic:** The OVS MAC table lives inside `ovs-vswitchd`'s heap. The kernel datapath has no MAC table. It has megaflow rules that embed the forwarding decision already resolved by `xlate_normal`. This two-layer structure (FDB in userspace, megaflow in kernel) is fundamentally different from both the Cisco hardware (single CAM in ASIC) and the Linux bridge (single FDB in kernel).

## R2.5.1.3 - The Linux bridge MAC table: a kernel hash table

The Linux bridge keeps its forwarding database entirely inside the kernel. The entry type
is `struct net_bridge_fdb_entry` in `net/bridge/br_fdb.c`. Every frame processed by the
Linux bridge module causes a call to `br_fdb_update`, which writes or refreshes the entry
for the source MAC in a per-bridge kernel hash table. The lookup function `br_fdb_find`
then determines whether to unicast or flood for the destination MAC. Both operations
happen synchronously within the same kernel interrupt or softirq context that is
processing the incoming frame. There is no userspace component in this path at all.

```ascii
Linux bridge (entirely kernel-space):
  Frame arrives at bridge netdev
  |
  v
br_handle_frame()               net/bridge/br_input.c
  |
  +-> br_fdb_update()           net/bridge/br_fdb.c
  |     looks up or inserts (src_MAC, port) in kernel hash table
  |     no upcall, no userspace, no IPC overhead
  |
  +-> __br_forward() or         looks up dst_MAC in same kernel hash table;
      br_flood()                unicast or flood entirely within the kernel
```

The consequence of this architecture is that the Linux bridge incurs no interprocess
communication for any forwarding decision, including the first packet on a new flow. Every
packet, including the ARP request that precedes the first ICMP exchange, is forwarded with
no process context switch. The kernel bridge code runs directly in the kernel's networking
stack, in the same execution context as the code that passes the frame up through the
netdev layer. This is why the Linux bridge has consistently lower first-packet latency
than OVS.

The user-visible interface to the kernel FDB is `bridge fdb show`, which queries the
kernel via netlink (specifically, `RTM_GETNEIGH` requests) and formats the result. The
`ageing_time` parameter, set via `ip link set <bridge> type bridge ageing_time
<centiseconds>`, configures the entry lifetime. Note the unit: centiseconds, not seconds.
A value of 500 means 5 seconds, not 500 seconds. This unit difference from OVS (which uses
seconds) is a frequent source of operator error when configuring both systems on the same
host. The aging sweep runs inside the kernel's `br_fdb_cleanup` work callback, which fires
periodically based on a garbage collection timer that is distinct from the `ageing_time`
value itself. The gc_timer period is visible in `ip -d link show <bridge>`.

Three structural consequences follow from placing the FDB entirely in the kernel. First,
`bridge fdb show` and `ovs-appctl fdb/show` are completely non-interchangeable: they query
different data structures in completely different memory spaces, one in kernel memory and
one in a userspace daemon's heap. Second, the Linux bridge FDB is not visible to OVS and
vice versa, even when both switches operate on the same physical host with the same
interfaces. Third, the Linux bridge aging is driven by the kernel's timer infrastructure,
which fires at a more predictable interval than OVS's userspace main-loop sweep, because
the kernel timer is a hardware-driven interrupt source rather than a software poll loop.

There is also a category distinction in the Linux bridge FDB that has no direct OVS
equivalent. The Linux bridge stores three types of entries: `permanent` multicast entries
for IPv4 and IPv6 multicast group addresses (these are never aged), `permanent` static
unicast entries for the bridge's own port MAC addresses (also never aged), and dynamic
learned unicast entries for the remote stations the bridge has seen (subject to
`ageing_time`). Only the dynamic learned entries are affected by `ageing_time`. The OVS
FDB stores only learned unicast entries; there are no permanent or multicast equivalents
in `struct mac_learning`. When comparing the two FDBs side by side, the OVS FDB appears
terse and the Linux bridge FDB appears verbose, but they contain the same logical
information in the "learned unicast" category. The additional entries in the Linux bridge
FDB represent different categories that OVS simply does not represent in its FDB at all.

> **Misconception:** "OVS and the Linux bridge both use the kernel FDB, they just have different tools to query it."
>
> **Reality:** OVS bridges have no kernel FDB at all. The Linux bridge subsystem (`net/bridge/`) maintains its own per-bridge hash table in kernel memory. OVS bridges use a completely separate `struct mac_learning` inside `ovs-vswitchd`'s heap. Running `bridge fdb show br br0` against an OVS bridge returns empty output silently, not because the FDB is empty, but because `br0` is not registered in the Linux bridge subsystem. The two systems do not share any data structure.

## R2.5.1.4 - Topology and experimental design

This lab runs two parallel topologies on the same host and the same two network namespaces
(`ns_red` and `ns_blue`). Section A uses an OVS bridge (`br0`) with its own veth pair
(`veth_red_br` and `veth_blue_br`). Section B uses a Linux bridge (`lbr0`) with a separate
veth pair (`veth_l1_br` and `veth_l2_br`). The IP subnets are different: 10.0.0.0/24 for
the OVS section and 10.0.1.0/24 for the Linux bridge section. Separate subnets prevent IP
address conflicts and ensure that the ARP tables inside the namespaces do not confuse
entries learned from one section's traffic with entries from the other.

```ascii
Section A (OVS):

  ns_red (10.0.0.1)              ns_blue (10.0.0.2)
  MAC 56:a6:87:c1:15:9c          MAC e2:10:43:fa:d2:bd
  veth_red                        veth_blue
     |  (namespace boundary)          |
  veth_red_br (OF port 1, DP port 2)  veth_blue_br (OF port 2, DP port 3)
     |                                    |
     +------------- br0 (OVS) -----------+
                    br0 internal port (OFPP_LOCAL / OF port 65534 / DP port 1)
                    FDB: struct mac_learning in ovs-vswitchd heap

Section B (Linux bridge):

  ns_red (10.0.1.1)              ns_blue (10.0.1.2)
  MAC d2:7f:2f:01:d3:b6          MAC d6:6a:ff:88:00:d7
  veth_l1                         veth_l2
     |  (namespace boundary)          |
  veth_l1_br                      veth_l2_br
     |                                    |
     +------------- lbr0 (Linux) --------+
                    lbr0 bridge (kernel net/bridge/)
                    FDB: struct net_bridge_fdb_entry in kernel hash table
```

The experiment sequence within each section follows the same pattern: build topology, read
the FDB before sending any traffic (to establish the baseline and catch NDP-seeded
entries), send a ping, read the FDB after the ping (to confirm learning and entry
refresh), configure a short aging timer, wait, and read the FDB again (to observe
eviction). Section C re-triggers learning on both bridges with a fresh ping and reads both
FDBs back-to-back, making the format contrast and the category structure visible in a
single operation.

The MAC addresses shown in the diagram above are the actual values assigned by the kernel
during the lab capture. They appear verbatim in the exercise outputs that follow. Using
the actual lab MACs (rather than anonymous placeholders like `<ns_red_mac>`) is a
deliberate choice: it makes the outputs readable as a historical record and allows the
reader to verify that each FDB entry corresponds to a specific interface.

### ▶ Guided Exercise 1: Establish carry-forward state and build the OVS topology

no extra veth pairs. This state must be confirmed before building any topology, because
the lab transcripts in this series are a faithful historical record and the starting
conditions affect every subsequent observation.

The `ovs-vsctl show` output confirms a clean OVSDB: only the `Open_vSwitch` root row

Now build the OVS topology. The sequence follows the pattern established in R2: create
bridge, create namespaces, create veth pairs, move namespace-side ends into namespaces,
add bridge-side ends as OVS ports, bring up all interfaces, and assign IPs. All sixteen
commands run without any output on success, which is the expected Unix convention for
commands that complete without error.

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-br br0
root@lab-openvswitch:~/network-onboard# ip netns add ns_red
root@lab-openvswitch:~/network-onboard# ip netns add ns_blue
root@lab-openvswitch:~/network-onboard# ip link add veth_red type veth peer name veth_red_br
root@lab-openvswitch:~/network-onboard# ip link add veth_blue type veth peer name veth_blue_br
root@lab-openvswitch:~/network-onboard# ip link set veth_red netns ns_red
root@lab-openvswitch:~/network-onboard# ip link set veth_blue netns ns_blue

root@lab-openvswitch:~/network-onboard# ip netns exec ns_blue ip addr add 10.0.0.2/24 dev veth_blue
```

Several important things happen during this build sequence that are not visible in the
command output. The `ovs-vsctl add-br br0` command creates the `br0` bridge in OVSDB and
instructs `ovs-vswitchd` to create a corresponding kernel datapath. The kernel module
refcount rises from 0 to 1 because `br0` now owns `ovs-system`. The `ovs-vsctl add-port`
commands create Port and Interface rows in OVSDB for the two veth bridge ends, and `ovs-
vswitchd` instructs the kernel datapath to add those interfaces as datapath ports.
Bringing the bridge-side veth ends up (`ip link set veth_red_br up`) has a side effect:
the Linux kernel automatically sends IPv6 Router Solicitation and Neighbor Advertisement
packets on every newly activated interface. Those NDP frames travel into the `br0` kernel
datapath, miss the (empty) megaflow cache, and generate upcalls to `ovs-vswitchd`. Those
upcalls enter `xlate_normal`, which calls `update_learning_table`, which calls
`mac_learning_insert`. The source MACs of those NDP frames are stored in `struct
mac_learning` before any operator-initiated ping runs. This is why the FDB is not empty
when we read it in the next exercise.

Confirm the topology and inspect the datapath port assignments:

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl show
15b3fd6f-e30f-4ae3-b8ca-4531189afee5
    Bridge br0
        Port veth_blue_br
            Interface veth_blue_br
        Port br0
            Interface br0
                type: internal
        Port veth_red_br
            Interface veth_red_br
    ovs_version: "2.17.9"
root@lab-openvswitch:~/network-onboard# ovs-appctl dpif/show
system@ovs-system: hit:7 missed:6
  br0:
    br0 65534/1: (internal)
    veth_blue_br 2/3: (system)
    veth_red_br 1/2: (system)
```

The `ovs-vsctl show` confirms all three ports: `veth_red_br`, `veth_blue_br`, and the
`br0` internal port (created automatically when the bridge is created). The `dpif/show`
output encodes the OpenFlow port number and the datapath port number for each interface in
the format `OF_port/DP_port`. Reading it: `br0` is OpenFlow port 65534 (the reserved
`OFPP_LOCAL` constant) at datapath port 1; `veth_blue_br` is OpenFlow port 2 at datapath
port 3; `veth_red_br` is OpenFlow port 1 at datapath port 2.

The `hit:7 missed:6` counters in `dpif/show` are significant. They show that 13 packets
have already been processed by the kernel datapath since the bridge was created. Seven hit
the megaflow cache; six missed and caused upcalls to `ovs-vswitchd`. Those 6 missed
packets are the NDP frames that seeded the FDB. This counter evidence confirms that MAC
learning happened through upcalls before any explicit operator traffic.

The port number mapping is important to keep in mind for the rest of this lab. The FDB's
`port` column uses OpenFlow port numbers. The megaflow dump uses datapath port numbers.
They are different number spaces. A port assignment like `veth_red_br 1/2` means OpenFlow
port 1, datapath port 2 for that interface. The number after the slash is the datapath
port number used in `dump-flows` output.

### ▶ Guided Exercise 2: Inspect MACs and read the pre-ping FDB

Before sending any explicit traffic, record the MAC addresses that the namespace-side veth
ends were assigned by the kernel, and then read the OVS FDB to confirm which MACs were
already learned from NDP traffic. The MAC addresses are assigned pseudo-randomly by the
kernel when the veth pair is created. Recording them here lets us match them against the
FDB entries.

```text
root@lab-openvswitch:~/network-onboard# ip netns exec ns_red ip link show veth_red
180: veth_red@if179: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 qdisc noqueue state UP mode DEFAULT group default qlen 1000
    link/ether 56:a6:87:c1:15:9c brd ff:ff:ff:ff:ff:ff link-netnsid 0
root@lab-openvswitch:~/network-onboard# ip netns exec ns_blue ip link show veth_blue
182: veth_blue@if181: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 qdisc noqueue state UP mode DEFAULT group default qlen 1000
    link/ether e2:10:43:fa:d2:bd brd ff:ff:ff:ff:ff:ff link-netnsid 0
```

The `link/ether` fields are the data we need: `ns_red` side is `56:a6:87:c1:15:9c` and
`ns_blue` side is `e2:10:43:fa:d2:bd`. The interface index numbers (180, 182) confirm
these are the namespace-side ends: a veth pair creates two interfaces, and the `@if<N>`
notation shows each end's peer interface index. The `veth_red` at index 180 is the peer of
the interface at index 179, which is `veth_red_br` in the root namespace.

Now read the OVS FDB:

```text
root@lab-openvswitch:~/network-onboard# ovs-appctl fdb/show br0
 port  VLAN  MAC                Age
    1     0  56:a6:87:c1:15:9c    3
    2     0  e2:10:43:fa:d2:bd    0
```

Both MACs appear already, at age 3 seconds and 0 seconds respectively. Cross-referencing
with `dpif/show`: port 1 in the FDB corresponds to `veth_red_br` (OpenFlow port 1), and
port 2 corresponds to `veth_blue_br` (OpenFlow port 2). The FDB correctly maps each MAC to
the OVS port through which that MAC's NDP frames arrived. The `Age` column shows elapsed
seconds since the entry was last refreshed by `mac_learning_insert`, so age 3 means this
entry was learned 3 seconds ago and age 0 means this entry was learned in the current
second.

This finding is the first important difference from a Cisco switch. On a Cisco switch, the
CAM is empty at startup and stays empty until application-layer unicast frames arrive on
the port. On OVS, the IPv6 NDP frames that the Linux kernel sends automatically when any
network interface is brought up are enough to trigger `xlate_normal` upcalls, which run
`mac_learning_insert` for the source MAC of each NDP frame. The FDB is therefore already
populated before any operator-initiated traffic. This is not a bug; it is a consequence of
how OVS processes all traffic classes through the upcall path, regardless of whether the
traffic is ARP, NDP, or application data.

The implication for production environments is that OVS bridges begin forwarding correctly
for known MACs almost immediately after the topology is built, without requiring an
explicit "warm-up" ping. The CAM-is-empty mental model that Cisco engineers carry from
hardware does not apply to OVS.

> **Key Topic:** OVS's FDB is seeded by any traffic that causes an upcall, including protocol traffic (IPv6 NDP, ARP) that the Linux kernel sends automatically at interface bring-up time. On a Cisco hardware switch, the CAM is populated only by data-plane unicast frames that traverse the data plane.

### ▶ Guided Exercise 3: Send ICMP traffic and read the FDB after learning

Now send three ICMP echo requests from `ns_red` to `ns_blue`. Before running this command,
reason through what should happen. Both source MACs are already in the FDB from the NDP-
seeded entries, so `xlate_normal` already knows the output port for both directions.
However, the kernel ARP cache inside `ns_red` does not yet have an entry for `10.0.0.2`.
The first ICMP packet will therefore trigger an ARP exchange before the ICMP packet itself
can be sent. The ARP request is a broadcast frame (destination `ff:ff:ff:ff:ff:ff`). OVS
will flood it to all ports. The ARP reply from `ns_blue` will be a unicast frame addressed
to `ns_red`'s MAC; since `56:a6:87:c1:15:9c` is already in the FDB, OVS can forward it
directly without flooding. After the ARP exchange completes, the ICMP packets themselves
should use the megaflow rules installed during ARP processing, producing a drop in RTT
from the first packet to subsequent packets.

```text
root@lab-openvswitch:~/network-onboard# ip netns exec ns_red ping -c 3 -W 2 10.0.0.2
PING 10.0.0.2 (10.0.0.2) 56(84) bytes of data.
64 bytes from 10.0.0.2: icmp_seq=1 ttl=64 time=0.771 ms
64 bytes from 10.0.0.2: icmp_seq=2 ttl=64 time=0.080 ms
64 bytes from 10.0.0.2: icmp_seq=3 ttl=64 time=0.043 ms

--- 10.0.0.2 ping statistics ---
3 packets transmitted, 3 received, 0% packet loss, time 2055ms
rtt min/avg/max/mdev = 0.043/0.298/0.771/0.334 ms
root@lab-openvswitch:~/network-onboard# ovs-appctl fdb/show br0
 port  VLAN  MAC                Age
    1     0  56:a6:87:c1:15:9c    1
    2     0  e2:10:43:fa:d2:bd    1
```

The RTT pattern matches the prediction precisely. The first packet at 0.771 ms includes
the ARP round trip (one broadcast upcall plus one unicast reply forwarded by megaflow) and
then the first ICMP packet's upcall to `xlate_normal`. Once those two upcalls installed
their respective megaflow rules, packets 2 and 3 (0.080 ms and 0.043 ms) hit the kernel
megaflow cache directly without any userspace involvement. The improvement from 0.771 ms
on the first packet to 0.043 ms on the third represents the measurable overhead of the
upcall path: two process context switches (kernel to userspace for the upcall, userspace
to kernel for the action installation) plus the processing time inside `xlate_normal`.

Both FDB entries are now at age 1 second, confirming that the ARP exchange refreshed them:
`mac_learning_insert` resets the expiry timestamp when an existing entry's source MAC is
seen again, whether on the same port or on a different one. The FDB age column is not a
one-shot countdown; it shows elapsed seconds since the most recent refresh, and it resets
to 0 each time the source MAC appears in new traffic.

> **Misconception:** "The RTT improvement from packet 1 to packet 2 means OVS learned the MACs during the ping."
>
> **Reality:** OVS already knew both MACs from the NDP-seeded entries before the ping started. The RTT improvement from packet 1 to packet 2 reflects the cost of installing new megaflow rules for the ICMP and ARP traffic classes, not the cost of FDB learning. FDB learning from NDP happened silently during topology build. Megaflow installation happens on the first packet of each new traffic class (as defined by the megaflow key: source MAC, destination MAC, ethernet type, and optionally IPv4 protocol field).

### ▶ Guided Exercise 4: Read the kernel megaflow table after learning

The datapath megaflow table encodes the result of every `xlate_normal` invocation as a
forwarding rule that the kernel datapath can execute without any further userspace
involvement. Reading `dump-flows` immediately after the ping reveals the complete set of
rules that were installed during the ARP exchange and the three ICMP packets. These rules
are the concrete representation of the forwarding decisions that `mac_learning_lookup`
produced inside `xlate_normal`.

There are seven rules. A critical point before reading them: the port numbers in `dump-

port 2 is `veth_red_br`, DP port 3 is `veth_blue_br`, DP port 1 is the `br0` internal
port.

The second rule captures the unicast IPv4 ICMP forward path: `in_port(2),
src=56:a6:87:c1:15:9c, dst=e2:10:43:fa:d2:bd, eth_type=0x0800 (IPv4), actions:3`. This

request arrived. It says: when a frame comes in on DP port 2 (`veth_red_br`), with the
known source MAC of `ns_red` and destination MAC of `ns_blue`, carrying an IPv4 payload,
forward it to DP port 3 (`veth_blue_br`). The `packets:2` counter shows that packets 2 and
3 used this rule without any upcall.

The seventh rule is the symmetric ICMP return path: `in_port(3), src=e2:10:43:fa:d2:bd,
dst=56:a6:87:c1:15:9c, eth_type=0x0800, actions:2`. This handled the ICMP echo replies
from `ns_blue` back to `ns_red` through DP port 2.

The third rule (`actions:1,3`) is the broadcast ARP flood rule. When `ns_red` sent an ARP
request with destination `ff:ff:ff:ff:ff:ff`, `xlate_normal` saw a broadcast destination
and therefore flooded: `actions:1,3` means send to DP ports 1 and 3 simultaneously. DP
port 1 is the `br0` internal port (consumed by OVS itself for control-plane frames) and DP
port 3 is `veth_blue_br`. The `packets:0, used:never` counter shows this flood rule was
installed speculatively but never matched again after the initial ARP was resolved.

The megaflow table demonstrates the two-layer forwarding model in concrete terms. Each row
represents one class of traffic for which `xlate_normal` made a forwarding decision. The
kernel holds that decision as an executable rule and applies it to every subsequent packet
in the class without any call to `xlate_normal` or `mac_learning_lookup`.

### ▶ Guided Exercise 5: OVS MAC aging (POE)

This exercise uses a separate capture: `sdn-onboard/labs/v3.13-R2.5.1-aging-cycles.typescript`
(2026-04-30 06:36:16 UTC to 06:36:54 UTC). The topology is rebuilt fresh with one critical
difference from the main transcript: IPv6 NDP is disabled on both namespace veth interfaces
before the ping. This isolation step is essential for the validity of the experiment.

Without disabling IPv6, the Linux kernel automatically sends IPv6 Neighbor Discovery Protocol
(NDP) packets on every active interface at regular intervals (approximately every 30 seconds
for Router Solicitation, more frequently for Neighbor Advertisement). Each NDP frame that
traverses the OVS bridge causes an upcall to `xlate_normal`, which calls `mac_learning_insert`
and resets the expiry timestamp on the corresponding FDB entry. From the aging sweep's
perspective, every NDP frame is a legitimate refresh event, indistinguishable from
application-layer traffic. The result is that the FDB entry for each namespace-side MAC
address is periodically refreshed by background protocol traffic, preventing aging from
demonstrating the intended 15-second eviction cleanly.

Disabling IPv6 with `sysctl -w net.ipv6.conf.<dev>.disable_ipv6=1` removes this background
refresh entirely. After that syscall, the kernel sends no further NDP frames on those
interfaces, and any FDB entry can age out at the exact moment `mac_learning_run` next sweeps
it past the 15-second threshold. This makes the experiment controllable and reproducible.

The property to prove is: OVS enforces a minimum MAC aging time of 15 seconds regardless of
the configured value, because `normalize_idle_time` in `lib/mac-learning.c` at line 194 clamps
any value below 15 up to 15. The eviction sweep is not perfectly synchronous with the timer:
entries expire on the next `mac_learning_run` main-loop iteration after the threshold passes,
which may add a few seconds of latency beyond the nominal 15 seconds.

**Predict (counter-hypothesis):** If `normalize_idle_time` clamped at a lower value (for
example, 5 seconds), then entries set with `mac-aging-time=15` would expire before 15 seconds
of idle time. Alternatively, if `mac_learning_run` were perfectly synchronous with a hardware
timer, entries would disappear at exactly the 15-second mark between two consecutive 3-second
polls. Both sub-predictions are wrong. The correct prediction is: entries survive at least 15
seconds and disappear within a few seconds after that window, consistent with a software sweep
running on the next main-loop iteration after the 15-second threshold is crossed.

The OVSDB key that configures aging is `other_config:mac-aging-time`. It is read by
`vswitchd/bridge.c` at line 2179 (`int idle_time = smap_get_int(oc, "mac-aging-time", 0)`) and
then passed to `mac_learning_set_idle_time`. That function calls `normalize_idle_time` before
storing the value in `ml->idle_time`. The `normalize_idle_time` clamping is invisible to the
operator: `ovs-vsctl get bridge br0 other_config` still shows the configured value, not the
clamped internal value.

The experiment polls `fdb/show` every 3 seconds for 36 seconds after setting the aging timer.
Each poll reading is labelled `cycle N (Ns)` where N is the number of seconds elapsed since
the aging-time was configured.


**Observe:** Entries are present through cycle 6 (18 seconds elapsed since the aging-time
was set). At cycle 7 (21 seconds elapsed), the FDB is empty and stays empty through cycle 8.

The age values in cycle 6 are 14 and 13, meaning the entries were last refreshed 14 and 13
seconds before cycle 6 ran. Cycle 6 ran approximately 18 seconds after the aging timer was
set, and the entries were seeded at age 1. So the actual elapsed idle time since the ping is
approximately 18 + 1 = 19 seconds, but the age column shows 14 and 13. This discrepancy
reflects the quantization of the Age counter: `fdb/show` reports elapsed time in whole
seconds, and the vswitchd main-loop timer resolution is coarser than a wall-clock second.
The critical observation is that both entries survive through cycle 6 (age 14 and 13, both
below the 15-second threshold) and are absent at cycle 7.

**Explain:** The counter-hypothesis is rejected in the direction of "too early": entries
survived beyond 15 seconds of elapsed wall-clock time but were evicted between 18 and 21
seconds. Two mechanisms explain the window.

The first mechanism is the 15-second floor from `normalize_idle_time`. Entries cannot expire
before their age reaches 15 seconds, regardless of what value is configured. In this
experiment, the entries' age values at cycle 6 (14 and 13) are still below 15, so
`mac_learning_run` cannot yet evict them even if it fires at exactly that moment. The entries
must remain for at least one more sweep cycle.

The second mechanism is the sweep timing of `mac_learning_run`. The function runs inside the
vswitchd main poll loop, not on a dedicated hardware timer. After an entry's expiry timestamp
passes (at age 15), the sweep removes it on the next main-loop iteration, which may fire
seconds later. In this experiment, the entries crossed the 15-second threshold during the
3-second window between cycle 6 (18s, entries at age 14 and 13) and cycle 7 (21s, FDB
empty). The sweep ran somewhere in that window and found entries with ages that had just
exceeded 15. This is consistent with the source code: `mac_learning_run` at line 638 walks
the FDB and calls `mac_entry_delete` on any entry whose `expires` timestamp is in the past.

The combined result is that with `mac-aging-time=15` and no background traffic, entries
survive 15 to 21 seconds of idle time, with the upper bound determined by how frequently the
vswitchd main loop runs its sweep. This is the correct characterisation of OVS MAC aging:
it is a lower-bounded, best-effort sweep, not a real-time hardware timer.

> **Misconception:** "Setting `mac-aging-time=5` on an OVS bridge will evict entries after
> 5 seconds."
>
> **Reality:** Two separate mechanisms prevent this. First, `normalize_idle_time` in
> `lib/mac-learning.c` clamps any value below 15 up to 15, silently. The value 5 becomes 15
> inside the daemon without any log message or warning. Second, even at 15 seconds,
> `mac_learning_run` is a main-loop function rather than a hardware timer. Its cadence
> depends on the vswitchd poll interval and system load. Eviction happens on the next sweep
> after the entry's expiry timestamp passes, which may be several seconds after the threshold
> is crossed. The practical eviction window is 15 to approximately 20 seconds for a lightly
> loaded host, as measured in this experiment.

### ▶ Guided Exercise 6: Build the Linux bridge topology

The Linux bridge experiment reuses the same two namespaces (`ns_red` and `ns_blue`) that
are still running from section A. The OVS bridge (`br0`) is left in place and will remain
running during the Linux bridge experiments. A new veth pair goes into each namespace for
the Linux bridge side, on the separate 10.0.1.0/24 subnet to avoid IP address conflicts.

The Linux bridge is created as a kernel netdev of type `bridge`, not through OVS. The `ip
link add name lbr0 type bridge` command creates the bridge device; `ip link set veth_l1_br
master lbr0` makes a veth bridge end a member of that bridge. This sequence is entirely
within the Linux kernel, with no interaction with OVS or OVSDB.

```text
root@lab-openvswitch:~/network-onboard# ip link add name lbr0 type bridge
root@lab-openvswitch:~/network-onboard# ip link set lbr0 up
root@lab-openvswitch:~/network-onboard# ip link add name veth_l1 type veth peer name veth_l1_br
root@lab-openvswitch:~/network-onboard# ip link add name veth_l2 type veth peer name veth_l2_br
root@lab-openvswitch:~/network-onboard# ip link set veth_l1 netns ns_red
root@lab-openvswitch:~/network-onboard# ip link set veth_l2 netns ns_blue
root@lab-openvswitch:~/network-onboard# ip link set veth_l1_br master lbr0
root@lab-openvswitch:~/network-onboard# ip link set veth_l1_br up
root@lab-openvswitch:~/network-onboard# ip link set veth_l2_br master lbr0
root@lab-openvswitch:~/network-onboard# ip link set veth_l2_br up
root@lab-openvswitch:~/network-onboard# ip netns exec ns_red ip link set veth_l1 up
root@lab-openvswitch:~/network-onboard# ip netns exec ns_red ip addr add 10.0.1.1/24 dev veth_l1
root@lab-openvswitch:~/network-onboard# ip netns exec ns_blue ip link set veth_l2 up
root@lab-openvswitch:~/network-onboard# ip netns exec ns_blue ip addr add 10.0.1.2/24 dev veth_l2
```

The `ip link set veth_l1_br master lbr0` command registers `veth_l1_br` as a member of the
Linux bridge `lbr0`. From this point, all frames arriving on `veth_l1_br` are passed to
the kernel bridge module rather than routed through the host's IP stack. As soon as a port
becomes a bridge member, the bridge module installs the bridge-side veth end's own MAC
address as a `permanent` entry in the kernel FDB. It also immediately installs the
multicast reservation entries. All of this happens inside the kernel before any data plane
frame arrives. The result is that the pre-ping FDB for the Linux bridge will already
contain entries from two of the three categories described in R2.5.1.3.

### ▶ Guided Exercise 7: Inspect the pre-ping Linux bridge FDB and understand its three entry categories

The first command checks which ports `lbr0` sees and their STP state. Then the FDB is read
to show all entries before any ping traffic.

```text
root@lab-openvswitch:~/network-onboard# bridge link show
```

Both ports are in `forwarding` state immediately. On a real Cisco switch running 802.1D
STP, a newly activated port would start in Blocking, advance to Listening (15 seconds),
then Learning (15 seconds), and finally reach Forwarding after a total of 30 seconds. The
Linux bridge has STP disabled by default (`stp_state 0`, visible in the `ip -d link show`
output later). Without STP, there is no convergence delay; ports go directly to forwarding
as soon as the link is up. This makes the Linux bridge unsuitable for use in a network
where loops are possible without an external loop-prevention mechanism.

The output contains 19 lines. Contrast this with the OVS FDB, which showed exactly 2
lines. Both FDBs represent the same two communicating hosts, but the Linux bridge FDB is
nine times larger. This verbosity arises from the three-category structure of the Linux
bridge FDB.

The first category is the `permanent` multicast reservation entries. These are the entries
whose MAC addresses begin with `33:33:` (IPv6 multicast) or `01:00:5e:` (IPv4 multicast).
The Linux bridge must be able to handle multicast frames without flooding them to all
ports uncontrollably, so it maintains a per-bridge reservation table for multicast group
addresses. The `33:33:00:00:00:01` entry on `lbr0 self permanent` is the all-nodes IPv6
multicast address. The `33:33:ff:29:92:ac` entry is the solicited-node multicast address
derived from the `lbr0` bridge's own MAC (`ca:2d:00:29:92:ac`). All multicast entries are
`permanent`: they are kernel-managed protocol reservations, never subject to
`ageing_time`, never learned from data-plane frames. On a Cisco switch, equivalent
multicast handling is managed by the ASIC's multicast replication engine and static
multicast table entries.

The second category is the `permanent` static unicast entries. These are entries with a
specific unicast MAC address and the `permanent` keyword, associated with the bridge's own
ports. The `ca:2d:00:29:92:ac dev lbr0 ... permanent` entries record the MAC address of
the `lbr0` bridge interface itself. The `a2:8c:e2:6f:26:0c dev veth_l1_br ... permanent`
entries record the MAC address of the `veth_l1_br` interface (the bridge-side end of the
veth pair; the namespace side has a different MAC). The `f2:ed:e9:36:32:79 dev veth_l2_br
... permanent` entries record the MAC of `veth_l2_br`. The bridge installs these as
permanent entries so it can forward frames destined for its own port MAC addresses without
a lookup miss. They are never aged because they represent fixed hardware identities of the
bridge's own ports that cannot change while the bridge is running.

The third category is the dynamic learned unicast entries: `d2:7f:2f:01:d3:b6 dev
veth_l1_br master lbr0` and `d6:6a:ff:88:00:d7 dev veth_l2_br master lbr0`. These two
lines lack the `permanent` keyword. They are the namespace-side veth MACs: `ns_red`'s
`veth_l1` at `d2:7f:2f:01:d3:b6`, and `ns_blue`'s `veth_l2` at `d6:6a:ff:88:00:d7`. The
bridge learned them from the IPv6 NDP frames that the Linux kernel sent automatically when
those interfaces came up, exactly as OVS did. These are the only entries that
`ageing_time` will eventually remove. The two entries in the OVS FDB correspond exactly to
this third category; OVS simply does not store or display entries from the first two
categories.

```text
64 bytes from 10.0.1.2: icmp_seq=3 ttl=64 time=0.078 ms

--- 10.0.1.2 ping statistics ---
3 packets transmitted, 3 received, 0% packet loss, time 2033ms
rtt min/avg/max/mdev = 0.039/0.055/0.078/0.016 ms
```

All three RTTs (0.049 ms, 0.039 ms, 0.078 ms) are well below the OVS first-packet RTT
(0.771 ms) and the OVS second-packet RTT (0.080 ms). Even the slowest Linux bridge packet
(0.078 ms) is faster than the OVS second-packet RTT (0.080 ms). The maximum Linux bridge
RTT is lower than the OVS third-packet RTT (0.043 ms) only slightly, and that measurement
reflects measurement noise rather than a structural difference.

The key observation is the absence of a first-packet penalty on the Linux bridge. Where
OVS showed 0.771 ms on the first packet (due to the ARP plus ICMP upcall latency) before
dropping to 0.080 ms, the Linux bridge shows consistent sub-0.1ms latency on all three
packets including the first. This is because the Linux bridge has no upcall path. The ARP
exchange and every ICMP packet are processed entirely inside the kernel's network stack,
in the same execution context as the TCP/IP code that handles normal packet routing. There
is no process context switch from the kernel to a userspace daemon and back.

The consistency of the Linux bridge RTTs (0.049, 0.039, 0.078 ms) compared to the OVS RTTs
(0.771, 0.080, 0.043 ms) reflects a fundamental architectural difference. The Linux bridge
is a straightforward kernel module that processes every packet with the same per-packet
code path, leading to consistent latency. OVS has a two-regime behaviour: new traffic
classes pay the upcall cost, while established traffic classes benefit from the megaflow
cache. For workloads with many distinct traffic classes (many different source-destination
MAC pairs), OVS pays the upcall cost repeatedly. For workloads with stable traffic classes
(the same set of MAC pairs sending sustained traffic), OVS amortizes the upcall cost over
many packets.

The FDB after the ping contains the same 19 entries as before the ping; the dynamic
entries' timestamps were refreshed by the ping traffic. Since the format is identical to
what was already shown in exercise 7, it is not repeated here.

### ▶ Guided Exercise 9: Linux bridge MAC aging (POE)

bridge uses centiseconds, and the kernel silently accepts the value without a unit
conversion warning.

machine even when features like STP and VLAN filtering are currently disabled, which is
different from OVS where STP and VLAN features are controlled through OVSDB or OpenFlow
rather than kernel bridge parameters.

Linux bridge FDB lines (multicast reservations and permanent unicast) have no OVS
equivalent because OVS does not represent those categories in `struct mac_learning`.

### ▶ Guided Exercise 11: Confirm OVS bridge opacity to the Linux bridge tools

The final check confirms the architecture claim from section R2.5.1.3: OVS bridges are
invisible to the Linux bridge subsystem. The Linux bridge tool (`bridge`) queries the
kernel's netlink interface for bridge FDB information, and the kernel only returns
information for devices registered in the Linux bridge subsystem. An OVS bridge's internal
port is a kernel netdev, but it is not registered in the Linux bridge subsystem. The
`bridge fdb show br br0` command therefore receives an empty response from the kernel.

```text
root@lab-openvswitch:~/network-onboard# bridge fdb show br br0 2>&1 || true
```

The command returns no output and no error code other than 0. The `2>&1 || true` ensures
that any error from the `bridge` tool would be visible if it occurred; its absence
confirms the empty-but-silent behaviour. The kernel processed the `RTM_GETNEIGH` netlink
request for the device named `br0`, found no bridge FDB records for that device in the
Linux bridge subsystem, and returned an empty result set. This is the definitive empirical
confirmation that OVS's `br0` bridge is opaque to the Linux bridge tools: the kernel FDB
has no entries for `br0` because OVS does not write to the kernel bridge FDB at all.

## R2.5.1.8 - Summary of findings

The six exercises in section A and B, together with the side-by-side comparison in section
C, produce a consistent picture of how MAC learning works at the structural level on each
platform. The results map directly to the architectural claims made in sections R2.5.1.2
and R2.5.1.3.

*Table R2.5.1-1: OVS versus Linux bridge, L2 learning*

| Property | OVS `br0` | Linux bridge `lbr0` |
|---|---|---|
| FDB location | `ovs-vswitchd` heap, userspace | Kernel (`struct net_bridge_fdb_entry`) |
| Learning trigger | Upcall to `xlate_normal`; then `update_learning_table`; then `mac_learning_insert` | In-kernel `br_fdb_update` on every received frame |
| Read tool | `ovs-appctl fdb/show <bridge>` | `bridge fdb show br <bridge>` |
| Age visibility | Yes (seconds since last refresh) | No age column; kernel tracks internally |
| FDB entry categories | One (dynamic learned unicast only) | Three (permanent multicast, permanent static, dynamic learned) |
| Pre-ping FDB state | Two entries (from NDP upcalls) | Same two dynamic entries plus 17 permanent entries |
| First-packet RTT | 0.771 ms (ARP + ICMP upcall latency) | 0.049 ms (fully in-kernel, no upcall) |
| Subsequent RTT | 0.043 ms (kernel megaflow cache hit) | 0.039 ms (every packet in-kernel) |
| Aging config | `ovs-vsctl set bridge <br> other_config:mac-aging-time=<s>` (15-second floor enforced by `normalize_idle_time`) | `ip link set <br> type bridge ageing_time <centiseconds>` (no enforced floor) |
| Aging mechanism | `mac_learning_run` in vswitchd main loop (best-effort, not synchronous) | Kernel GC timer `br_fdb_cleanup` (fires every `gc_timer` seconds) |
| Visibility to each other | `bridge fdb show br br0` returns empty; OVS FDB not in kernel bridge subsystem | `ovs-appctl fdb/show lbr0` would error; Linux FDB not in vswitchd memory |

The wire-level forwarding outcome is identical: both platforms deliver frames to the
correct destination MAC once they have seen the source MAC. The differences are
structural: where the table lives, who can read it, how precise the aging is, and what
latency the first packet of a new traffic class experiences. For a Cisco-trained engineer
choosing between OVS and the Linux bridge for a specific use case, the first-packet
latency difference (0.771 ms vs 0.049 ms) is the most operationally significant result
from this lab. In environments where many distinct traffic classes arrive in short bursts
(for example, an OpenStack compute node with hundreds of VMs sending initial ARP and NDP
at boot time), OVS generates many upcalls during the burst and the first-packet penalty
accumulates across all new traffic classes simultaneously. The Linux bridge processes all
frames inline and never generates upcalls, so it has no equivalent burst cost.


The four teardown commands clean up in the correct order. `ovs-vsctl del-br br0` removes
the bridge and all its OVSDB rows atomically: the Bridge row, the three Port rows, and the
three Interface rows are all deleted in a single OVSDB transaction, and `ovs-vswitchd`
removes the kernel datapath `ovs-system` and drops its reference to the `openvswitch`
kernel module. The module refcount returns to 0. `ip link del lbr0` removes the Linux
bridge device; all veth pairs that were members of the bridge lose their `master lbr0`
association, and since the veth bridge ends no longer have an explicit owner, they are
removed along with the bridge. The `ip netns del ns_red` and `ip netns del ns_blue`
commands remove the two namespaces and all interfaces inside them (including `veth_red`,
`veth_blue`, `veth_l1`, and `veth_l2`).

the typescript capture window of `06:03:32 UTC` to `06:05:44 UTC`.

The host is back to the clean carry-forward state for R2.5.2: `ovs-vswitchd` and `ovsdb-
server` running, kernel module at refcount 0, no bridges, no namespaces, no extra veth

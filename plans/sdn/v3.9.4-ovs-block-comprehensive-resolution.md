## 3. Source verification methodology

This section is materially identical to plan v3.9.3 §3, which is materially identical to plan v3.9.2 §3 and plan v3.9.1 §3. Reproduced here for plan-readability and self-containment. The method is locked: any change requires a separate plan amendment.

### 3.1. Tools and pinning

- **Local OVS repo:** `C:\Users\voleh\Documents\ovs`. All release tags fetched.
- **Verification baseline tag:** `v2.17.9` (curriculum baseline per CLAUDE.md Current State).
- **Tag SHA reference:**
  - `git rev-parse v2.17.9` returns the annotated tag object SHA `14e48c5da60f8b9089e27234b8d078cae2a59fc7` (PGP-signed).
  - `git rev-parse v2.17.9^{commit}` returns the commit object SHA `0bea06d9957e3966d94c48873cd9afefba1c2677` (release commit, dated 2024-02-08, message "Set release date for 2.17.9").
  - Curriculum cites the commit SHA, because the commit is the content baseline.

OVS coding style affects grep patterns. OVS C function definitions split the return type and the function name across two lines:

```c
void                                              /* line N (return type only)        */
ovsdb_rbac_insert(const struct ovsdb *db, ...)    /* line N+1 (function name col 0)   */
```

The pattern `^[a-z_]+ \*?<fn>\(` (return type plus name on the same line) fails for most OVS functions. Use `^<fn>\(` or `^<fn>` to anchor on the function name in column 0.

### 3.2. Per-phase command set

Always run from the OVS repo with v2.17.9 checked out:

```bash
cd C:/Users/voleh/Documents/ovs
git checkout v2.17.9
```

Then:

| Purpose | Command |
|---|---|
| Find function definition plus line | `grep -nE "^<fn>" <path>` |
| Confirm function exists anywhere (definition or call) | `grep -rn "\b<fn>\b" <dir>` |
| Verify file presence in baseline | `ls <path>` (working tree after checkout) |
| Verify SHA exists | `git log -1 --format="%H %ai %s" <sha>` |
| First release tag containing a commit | `git tag --contains <sha> \| sort -V \| head -3` |
| First commit that added a file | `git log --diff-filter=A --format="%H %ai %s" -- <path> \| tail -1` |
| Cross-version compare | `git show <tag>:<path> \| grep <pattern>` |
| Search history for a feature | `git log --all --format="%H %ai %s" --grep="<keyword>" -- <path>` |
| Find struct definition | `grep -nE "^struct <name>" <path>` (or `grep -rn "^struct <name>" <dir>`) |
| Find macro definition | `grep -nE "^#define <NAME>" <path>` |

- `python scripts/lang_check.py --staged` (must exit 0; rejects Vietnamese-detected prose in newly added or modified lines, strict mode).
- `python scripts/anti_gaming_check.py --staged` (must exit 0; GP-6 to GP-10 enforcement).
- `python scripts/rubric_leak_check.py --staged` (must exit 0 within OVS scope; GP-11 enforcement).

After Phase R6, all four checks run on `--all` for the final regression audit.

### 3.5. Anti-pattern signals

Stop and rewrite the step if:

- Pre-edit grep was not run against the local v2.17.9 working tree.
- Edit was based on a function name from memory or from a v3.9 to v3.9.3 commit body that was not re-verified.
- Commit body lacks a reproducible command and verbatim output.

### 3.6. GP-13 in-commit English rewrite obligation

Every commit that modifies a curriculum file rewrites the level 3 section that contains the fix into English in the same commit. The rules from plan v3.9.1 §Q10.2:

- **Section granularity.** If a fix touches a line inside a level 3 heading, the entire level 3 section is rewritten to English. Sibling sections at the same level are not rewritten (out of scope for the fix).
- **Add language-status callout.** First commit that touches a file also adds the `**Language status:**` callout immediately after the H1 heading, listing which sections are now English. Subsequent commits that touch the same file update the callout's section list.
- **Preserve technical citations.** All function names, file paths, line numbers verified during audit are preserved in the English rewrite. Translation does not loosen Rule 14.
- **Preserve pedagogical structure.** Keep the axis-by-axis or paragraph-by-paragraph structure. Do not rearrange sections during translation.
- **Preserve verbatim quotes.** Any verbatim quotes from upstream OVS source code, NEWS file, RFC, or man page stay verbatim. They are evidence, not prose.
- **Em-dash forbidden** in every newly written English section. The pre-commit hook flags em-dash in newly added or modified lines.

### 3.7. Mid-batch escalation gate (calibrated for v3.9.4)

The v3.9.3 §3.7 gate at 50 percent triggered on the very first batch (R1 audit, 9.4 + 9.11) at 50.3 percent. Re-running the same threshold for v3.9.4 R3, R4, R5 audit batches would predictably halt v3.9.4 too because the curriculum drift level is now empirically known to span 30 to 50 percent depending on the file's authoring density. The user's directive is "ensure it resolves all issues" which is incompatible with auto-halting on the first sign of drift.

The v3.9.4 plan therefore proposes a **two-rate gate** that distinguishes the kind of violation:

- **Fabricated rate (FAB rate)** = FABRICATED count divided by total citations. Threshold 40 percent. Justification: a high fabricated rate indicates that the section was authored without the source open in another window. Fixing fabrications is high-effort and high-stakes (each one needs a real replacement decided per fabrication). A FAB rate above 40 percent justifies halting because the section needs a different remediation strategy (rewrite the source-code reference paragraphs from scratch rather than fix incrementally).
- **Total-violation rate** = (WRONG_LINE + WRONG_PATH + WRONG_NAME + WRONG_TYPE + FABRICATED) divided by total citations. Threshold 70 percent. Justification: at 70 percent, more than two thirds of citations are wrong. The section is not salvageable as a citation reference and should be rewritten as descriptive prose without per-citation attribution.

Decision tree at every R-phase audit close:

- If `FAB rate > 0.40` OR `total-violation rate > 0.70`: halt v3.9.4 and escalate to plan v3.9.5. Document the halt at `memory/sdn/v3.9.4-mid-checkpoint-halt-2026-04-XX.md` and notify the user with the concrete count.
- If `FAB rate <= 0.40` AND `total-violation rate <= 0.70`: continue. Document the result inline in the audit log.

For comparison with v3.9.3 R1 priors:

| Batch | Citations | Violations | FAB | FAB rate | Total rate | v3.9.4 gate verdict |
|---|---|---|---|---|---|---|
| v3.9.3 R1 (9.4 + 9.11) | 157 | 79 | 35 | 22.3% | 50.3% | **Continue** (under v3.9.4 gate) |

The v3.9.3 R1 batch under the v3.9.4 calibrated gate would NOT have halted, because FAB rate 22.3 percent is well below 40 percent and total rate 50.3 percent is below 70 percent. This validates the calibration: the v3.9.3 50 percent gate was tuned for "first-touch sweep" semantics where any divergence is concerning. The v3.9.4 gate is tuned for "comprehensive resolution" semantics where the tolerable error rate is higher because fixes are concentrated and well-understood.

The R1 phase of v3.9.4 is exempt from the gate because it is a fix-only batch operating on the known v3.9.3 R1 finding set. There is no audit step to compute a rate against. The R3, R4, R5 audit batches are subject to the v3.9.4 gate.

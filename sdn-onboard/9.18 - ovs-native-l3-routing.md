## 9.18.1 A common misconception and the position of this part

Many engineers reach OVN before OVS and carry a quiet assumption: "to route between subnets on OVS, you must use an OVN Logical Router." That assumption inverts the causal relationship. The OVN Logical Router does not provide routing capability; it provides the **abstraction** for managing routing at scale. The actual routing capability lives in the OVS flow table, in the same set of primitives that `ovn-northd` compiles down to.

This part strips off the abstraction layer and shows that the flow table is enough to act as a router on its own: with only six actions and the correct priority order, OVS forwards IP packets between two subnets just like a Cisco router or a Linux host with `net.ipv4.ip_forward=1`. Once you see this, every higher concept, the Logical Router in OVN, the distributed gateway port, north-south SNAT and DNAT, becomes transparent. They are templates that generate the same flow entries you are about to write by hand.

> **Key topic.** The OVS flow table is a fully functional router at the data plane. The OVN Logical Router, the Neutron router, and the Kubernetes CNI router all compile down to the same set of OpenFlow primitives that you study in this part. Understanding the data plane before learning the control plane lets you debug at every abstraction layer above.

Scope of this part: a single OVS bridge, two subnets, two hosts, no OVN, no Linux bridge, no `net.ipv4.ip_forward`. Routing happens 100 percent inside OpenFlow flow entries.

## 9.18.2 The six core router actions in OpenFlow

When a physical router receives an IP packet destined for a different subnet, it performs six steps:

| Step | Action | OpenFlow equivalent |
|------|--------|---------------------|
| 1 | Look up the routing table to determine the egress interface and the next-hop IP | Match on `nw_dst` plus output port lookup |
| 2 | Look up the ARP cache for the MAC of the next hop | Controller pre-installs the MAC in the flow (static ARP) |
| 3 | Rewrite the source MAC to the MAC of the egress interface | `mod_dl_src:<router_mac>` or `set_field:<mac>->eth_src` |
| 4 | Rewrite the destination MAC to the next-hop MAC | `mod_dl_dst:<nexthop_mac>` or `set_field:<mac>->eth_dst` |
| 5 | Decrement TTL by 1, drop if TTL reaches 0 | `dec_ttl` |
| 6 | Send the packet out the egress interface | `output:<port>` |

In the OpenFlow 1.3 and later pipeline, these actions are bundled inside a single flow entry and executed sequentially in the order listed in the action list. The `ovs-actions(7)` man page describes this clearly: actions execute in the order they appear, except inside a `write_actions` instruction (where they are buffered until the end of the pipeline). `apply_actions` is always sequential.

The two actions `mod_dl_src` and `mod_dl_dst` are legacy OpenFlow 1.0 syntax. OVS still supports them for backward compatibility. The modern syntax uses `set_field:<value>->eth_src` and `set_field:<value>->eth_dst` (OpenFlow 1.2 and later, OXM). The two syntaxes are equivalent in behaviour. This part uses both forms so you become familiar with them.

The `dec_ttl` action is defined in OpenFlow 1.1. It performs two operations at once: it decrements the TTL field by 1 and invalidates the packet if the new TTL is 0. If the packet is invalidated, `dec_ttl` sends a `packet_in` to the controller with reason `OFPR_INVALID_TTL` so the controller can produce an ICMP Time Exceeded message per RFC 792 if needed. In this lab, the controller does not handle `packet_in`, so a TTL=0 packet is simply dropped.

> **Common misconception.** "OVS must enable `net.ipv4.ip_forward=1` in the Linux kernel to be able to route."
>
> Reality: the OVS datapath processes packets inside its own kernel module (`openvswitch.ko`) or in userspace (DPDK), and **never** goes through the Linux IP stack. The sysctl `ip_forward` is meaningless for OVS. When OVS receives a packet on an ingress port, it runs flow lookup and executes the action list. It never hands the packet off to the Linux routing table. You can confirm this by setting `ip_forward=0` and observing that OVS routing still works in the Guided Exercise of Part 9.18.4.

## 9.18.3 Lab 7 topology and flow design

The Lab 7 topology (Crichigno et al., USC WASTC 2021) is intentionally minimal to focus on the routing logic:

```
                    +------------------------------+
                    |          br-router           |
                    |  (OVS bridge as a router)    |
                    |                              |
                    |  port 1            port 2    |
                    +----+-------------------+-----+
                         | veth1             | veth2
                         | MAC: aa:aa:aa:..01| MAC: aa:aa:aa:..02
                         | Gateway IP:       | Gateway IP:
                         | 192.168.1.1       | 192.168.2.1
                         |                   |
          +--------------+------+   +--------+----------------+
          |      ns-h1          |   |      ns-h2              |
          |  IP: 192.168.1.10   |   |  IP: 192.168.2.10       |
          |  MAC: 02:h1:..:10   |   |  MAC: 02:h2:..:10       |
          |  GW: 192.168.1.1    |   |  GW: 192.168.2.1        |
          +---------------------+   +-------------------------+
```

The core design point: **`br-router` has no IP address assigned to any bridge port**. The gateway IPs of each subnet (192.168.1.1 and 192.168.2.1) exist only conceptually in the host's mind. The host uses those IPs to choose the next hop when its routing table says "default via 192.168.1.1". On OVS, however, those IPs appear only in flow entries that handle ARP. No Linux interface carries those addresses. This is the key difference from a traditional router and explains why OVS can scale many "logical" routers on a single physical bridge.

The flow design is split into three groups (priority decreasing):

**Group 1: ARP responder (priority 1000).**

When `h1` boots and sends an ARP Request asking for the MAC of `192.168.1.1` (the gateway), OVS must reply instead of forwarding. The same applies to `h2` asking for `192.168.2.1`. The two flow entries match `arp, arp_tpa=<gateway_ip>`, and the action constructs the ARP Reply via a chain of `move`, `set_field`, and `output:IN_PORT`. The full ARP responder flow detail is outside the scope of this part (see `ovs-actions(7)` section "Field modification actions" and Part 18.0 §18.2 for the equivalent OVN ARP responder).

**Group 2: Routing flow, subnet 192.168.2.0/24 (priority 100).**

When an IP packet from `h1` to `192.168.2.10` arrives at port 1 with destination MAC equal to the gateway MAC `aa:aa:aa:aa:aa:01`, OVS must:

```
priority=100,ip,in_port=1,dl_dst=aa:aa:aa:aa:aa:01,nw_dst=192.168.2.0/24
actions=mod_dl_src:aa:aa:aa:aa:aa:02,
        mod_dl_dst:02:00:00:00:02:10,
        dec_ttl,
        output:2
```

The source MAC is rewritten from the MAC of `h1` to the MAC of the gateway on the subnet 2 side (`aa:aa:aa:aa:aa:02`). The destination MAC is rewritten from the MAC of the gateway on the subnet 1 side (before routing) to the MAC of `h2`. Because the flow entry already knows the MAC of `h2`, this is a **static ARP**, a lab simplification. A production router resolves ARP dynamically, learns the MAC from ARP Replies, and installs the result into a flow entry afterwards.

**Group 3: Routing flow, subnet 192.168.1.0/24 (priority 100, reverse direction).**

```
priority=100,ip,in_port=2,dl_dst=aa:aa:aa:aa:aa:02,nw_dst=192.168.1.0/24
actions=mod_dl_src:aa:aa:aa:aa:aa:01,
        mod_dl_dst:02:00:00:00:01:10,
        dec_ttl,
        output:1
```

This mirrors group 2, handling the reply traffic from `h2` back to `h1`.

**Default group (priority 0):** `actions=drop`. Packets that match no flow are dropped instead of flooded or bounced to the controller.

### Guided Exercise 1: route between two subnets with a pure OVS flow table

In this exercise, you build the Lab 7 topology on a single Linux host using network namespaces, install the routing flow table into OVS, and verify with ping plus tcpdump that the MAC rewrite and TTL decrement happen exactly like a real router.

**Outcomes:**

- Verify that OVS routes packets between two subnets without `net.ipv4.ip_forward`, without OVN, and without a Linux bridge.
- Observe directly the source and destination MAC change as the packet crosses the OVS bridge.
- Prove that the TTL decrements by exactly one per routing hop.

**Before You Begin:**

The OVS service is running (`systemctl status openvswitch-switch`). The kernel datapath module is loaded (`lsmod | grep openvswitch`). The user has full `sudo` privileges.

**Instructions:**

**1.** Create two network namespaces `ns-h1` and `ns-h2` plus an OVS bridge `br-router`:

```bash
[host]# ip netns add ns-h1
[host]# ip netns add ns-h2
[host]# ovs-vsctl add-br br-router
[host]# ovs-vsctl set bridge br-router protocols=OpenFlow13
```

**2.** Create two veth pairs, push one end into each namespace, and attach the other end to OVS:

```bash
[host]# ip link add veth-h1 type veth peer name h1-port
[host]# ip link add veth-h2 type veth peer name h2-port
[host]# ip link set h1-port netns ns-h1
[host]# ip link set h2-port netns ns-h2
[host]# ovs-vsctl add-port br-router veth-h1 -- set interface veth-h1 ofport_request=1
[host]# ovs-vsctl add-port br-router veth-h2 -- set interface veth-h2 ofport_request=2
[host]# ip link set veth-h1 up
[host]# ip link set veth-h2 up
```

**3.** Configure the IP, MAC, and default route inside each namespace:

```bash
[host]# ip netns exec ns-h1 ip link set h1-port address 02:00:00:00:01:10
[host]# ip netns exec ns-h1 ip addr add 192.168.1.10/24 dev h1-port
[host]# ip netns exec ns-h1 ip link set h1-port up
[host]# ip netns exec ns-h1 ip route add default via 192.168.1.1

[host]# ip netns exec ns-h2 ip link set h2-port address 02:00:00:00:02:10
[host]# ip netns exec ns-h2 ip addr add 192.168.2.10/24 dev h2-port
[host]# ip netns exec ns-h2 ip link set h2-port up
[host]# ip netns exec ns-h2 ip route add default via 192.168.2.1
```

**4.** Install a static ARP entry in each namespace to bypass the ARP responder (so the focus stays on routing):

```bash
[host]# ip netns exec ns-h1 ip neigh add 192.168.1.1 lladdr aa:aa:aa:aa:aa:01 dev h1-port
[host]# ip netns exec ns-h2 ip neigh add 192.168.2.1 lladdr aa:aa:aa:aa:aa:02 dev h2-port
```

**5.** Verify that `net.ipv4.ip_forward` is OFF, proving OVS does not use the Linux IP stack:

```bash
[host]# sysctl net.ipv4.ip_forward
net.ipv4.ip_forward = 0
```

If the output is `1`, set it to `0` with `sysctl -w net.ipv4.ip_forward=0`. The goal is to prove that routing happens entirely inside OVS.

**6.** Delete the default OVS flow entries and install the routing flows:

```bash
[host]# ovs-ofctl -O OpenFlow13 del-flows br-router

[host]# ovs-ofctl -O OpenFlow13 add-flow br-router \
  "priority=100,ip,in_port=1,dl_dst=aa:aa:aa:aa:aa:01,nw_dst=192.168.2.0/24,
   actions=mod_dl_src:aa:aa:aa:aa:aa:02,mod_dl_dst:02:00:00:00:02:10,dec_ttl,output:2"

[host]# ovs-ofctl -O OpenFlow13 add-flow br-router \
  "priority=100,ip,in_port=2,dl_dst=aa:aa:aa:aa:aa:02,nw_dst=192.168.1.0/24,
   actions=mod_dl_src:aa:aa:aa:aa:aa:01,mod_dl_dst:02:00:00:00:01:10,dec_ttl,output:1"

[host]# ovs-ofctl -O OpenFlow13 add-flow br-router "priority=0,actions=drop"
```

**7. Predict-Observe-Explain on TTL behaviour.**

Counter-hypothesis: *"If OVS only forwards at L2 (no routing), the TTL of a packet from `h1` to `h2` must remain at its original value. With Linux defaulting to TTL=64, the packet arriving at `h2` must have TTL=64."*

Observe by opening tcpdump in `ns-h2` and pinging from `ns-h1`:

```bash
[host]# ip netns exec ns-h2 tcpdump -i h2-port -nn -e -v icmp &
[host]# ip netns exec ns-h1 ping -c 1 192.168.2.10
```

Expected tcpdump output (irrelevant headers omitted):

```
aa:aa:aa:aa:aa:02 > 02:00:00:00:02:10, ethertype IPv4,
  (tos 0x0, ttl 63, id 12345, offset 0, flags [DF], proto ICMP (1), length 84)
  192.168.1.10 > 192.168.2.10: ICMP echo request, id 1, seq 1, length 64
```

Explanation: TTL=**63** (not 64) refutes the counter-hypothesis. Only routing decrements the TTL. The source MAC is `aa:aa:aa:aa:aa:02` (the gateway MAC on the subnet 2 side) instead of the MAC of `h1`, direct evidence that `mod_dl_src` ran. The destination MAC is `02:00:00:00:02:10` (the MAC of `h2`), rewritten by OVS from `mod_dl_dst`. Three pieces of evidence together confirm: OVS routed; it did not merely forward.

**8.** Confirm with flow statistics that the routing flow counter increased by the expected packet count:

```bash
[host]# ovs-ofctl -O OpenFlow13 dump-flows br-router
cookie=0x0, duration=45.2s, table=0, n_packets=1, n_bytes=98, priority=100,ip,
  in_port=1,dl_dst=aa:aa:aa:aa:aa:01,nw_dst=192.168.2.0/24
  actions=mod_dl_src:aa:aa:aa:aa:aa:02,mod_dl_dst:02:00:00:00:02:10,dec_ttl,output:2
cookie=0x0, duration=45.1s, table=0, n_packets=1, n_bytes=98, priority=100,ip,
  in_port=2,dl_dst=aa:aa:aa:aa:aa:02,nw_dst=192.168.1.0/24
  actions=mod_dl_src:aa:aa:aa:aa:aa:01,mod_dl_dst:02:00:00:00:01:10,dec_ttl,output:1
```

`n_packets=1` on both flows proves that both the echo request and the echo reply matched the correct flow, completing one ICMP round trip.

**Finish:**

```bash
[host]# ovs-vsctl del-br br-router
[host]# ip netns del ns-h1
[host]# ip netns del ns-h2
```

## 9.18.4 Comparison with the OVN Logical Router

Now that you have seen routing happen in a bare flow table, return to the question: what does the OVN Logical Router actually do differently?

`ovn-northd` takes input from the Northbound database: a `Logical_Router` row with `ports[]` listing the `Logical_Router_Port` entries, each carrying the IP and MAC of a subnet gateway. `ovn-northd` then runs the compile algorithm (see Part 13.8) to produce a set of `Logical_Flow` rows in the Southbound. `ovn-controller` on each chassis reads `Logical_Flow` and translates it into OpenFlow flow entries on `br-int`, which are essentially **the same set of actions** that you just installed by hand: `mod_dl_src`, `mod_dl_dst`, `dec_ttl`, `output`.

The differences live in three layers.

**Layer 1, abstraction and multi-tenant operation.** OVN allows you to define a router with declarative configuration (`ovn-nbctl lr-add tenant1-router`, `lr-route-add`, `lrp-add`). You do not write flow entries by hand. Each tenant has its own router, namespace-isolated by `tunnel_key` in the metadata. You cannot scale standalone OVS L3 to 1000 tenants with 1000 individual routers without an automation framework. OVN is that framework.

**Layer 2, distributed gateway.** The OVN Logical Router can be distributed (a single logical router spread across every chassis) or centralised via a `chassisredirect` port. The flow entries for a distributed router are replicated on every chassis, so east-west routing happens locally without going through a gateway chassis. Standalone OVS has no equivalent concept. If you want two chassis to route, you must replicate the flows manually and synchronise the ARP cache yourself.

**Layer 3, stateful services.** The OVN Logical Router integrates conntrack for SNAT, DNAT, Load Balancer, and stateful ACL via the OVS `ct()` action (which uses the kernel conntrack module). You can do the same on standalone OVS, but you have to write the conntrack rules by hand and manage the flow lifetime via `idle_timeout`, which is hard enough that no one does it in production.

> **Key topic.** OVN Logical Router equals OVS flow table plus a framework that compiles declarative configuration into flows plus lifecycle management for stateful services plus distributed forwarding semantics. The flow table still does all the heavy lifting; OVN does the *management* around it.

## 9.18.5 When to use standalone OVS L3 versus an OVN Logical Router

A practical question for the operating engineer:

**Use standalone OVS L3 when:**

- Learning and debugging: you need to see exactly which flow entry matches the packet, and the OVN abstraction hides several layers.
- Small lab: 1 to 2 subnets, 1 router, 1 node, no stateful services, no multi-tenant requirement.
- Special-purpose appliance: for example, an OVS bridge acting as a border router for a single workload, with no need for an OVN framework.
- Conformance testing: verify that `dec_ttl` decrements by exactly 1, that `mod_dl_dst` rewrites the correct bytes, with no OVN layer that could obscure the result.

**Use the OVN Logical Router when:**

- Production multi-tenant: 10 or more tenants, each needing its own isolated router.
- Distributed east-west traffic: workloads spread across many chassis where routing must stay local and not bottleneck on a gateway.
- Stateful services: SNAT for north-south, Load Balancer for east-west, ACL with conntrack.
- OpenStack or Kubernetes integration: Neutron ML2 / OVN and ovn-kubernetes only talk to the OVN Northbound, not to OVS flows directly.

> **Production readiness assessment.** On a single-node lab, standalone OVS L3 handles roughly 5 to 10 Gbps of routing throughput with the kernel datapath (see Part 9.17 methodology). Yet no one deploys standalone OVS L3 in production, for operational reasons rather than performance reasons. Manually managing flow entries does not scale beyond about 5 routers or 20 subnets. The crossover point to OVN is operational complexity, not data-plane limit.

## 9.18.X Source-level study of the `dec_ttl` action

The `dec_ttl` action decrements the TTL of the packet by 1 and recomputes the IP checksum. Source code anchors:

### OVS userspace implementation

[`ofproto/ofproto-dpif-xlate.c`](https://github.com/openvswitch/ovs/blob/v2.17.9/ofproto/ofproto-dpif-xlate.c), function `compose_dec_ttl()`:

```c
// Pseudo-code
static void compose_dec_ttl(struct xlate_ctx *ctx, ...) {
    if (flow->nw_ttl <= 1) {
        // TTL = 0 or 1: send to controller (OFPP_CONTROLLER)
        execute_controller_action(ctx, ...);
        return;
    }
    flow->nw_ttl--;
    // Recompute IP checksum incremental update
    update_ip_checksum(flow);
    // Add datapath action OVS_ACTION_ATTR_SET with field nw_ttl
    nl_msg_put_unspec(odp_actions, OVS_ACTION_ATTR_SET, ...);
}
```

### Kernel datapath implementation

[`net/openvswitch/actions.c`](https://git.kernel.org/pub/scm/linux/kernel/git/torvalds/linux.git/tree/net/openvswitch/actions.c?h=v5.15), function `set_ipv4_addr()` plus `set_ipv4()`:

The kernel receives `OVS_ACTION_ATTR_SET` with the mask field `nw_ttl`, and calls:

```c
// Simplified
static int set_ipv4(struct sk_buff *skb, struct ovs_key_ipv4 *key, ...) {
    struct iphdr *nh = ip_hdr(skb);
    // Update TTL
    nh->ttl = key->ipv4_ttl;
    // Incremental checksum update (RFC 1624)
    csum_replace2(&nh->check, htons(old_ttl<<8), htons(new_ttl<<8));
    return 0;
}
```

`csum_replace2` is a Linux kernel macro for incremental checksum update, faster than a full recomputation.

### TTL = 1 implies ICMP Time Exceeded

When `dec_ttl` is applied to a packet with TTL=1, OVS does not drop the packet, instead:

- **Userspace path:** send to the OpenFlow controller (`OFPP_CONTROLLER`).
- **Kernel datapath:** trigger an upcall (the kernel sends the packet to userspace).

The controller, or the userspace `ovn-controller`, can respond with ICMP Time Exceeded (Type 11). OVN implements this in stage `lr_in_ip_input` via the action `icmp4{...}`.

### Multi-stage routing pattern

Cross-link to Part 9.18.4 (Lab 7 USC). It already presents a 3-table pattern. Adding more depth:

```
Table 0 (ingress classifier):
  in_port=1 -> set metadata=1 (subnet ID 1) -> resubmit(,1)
  in_port=2 -> set metadata=2 -> resubmit(,1)

Table 1 (route lookup):
  metadata=1, ip, nw_dst=10.0.0.0/24 -> mod_dl_dst:<gateway-mac>, resubmit(,2)
  metadata=2, ip, nw_dst=10.0.1.0/24 -> mod_dl_dst:..., resubmit(,2)
  ip, nw_dst=0.0.0.0/0 -> mod_dl_dst:<default-gw-mac>, resubmit(,2)  # default route

Table 2 (egress plus dec_ttl):
  ip, dl_dst=<gw1-mac> -> dec_ttl, mod_dl_src:<router-mac-port1>, output:1
  ip, dl_dst=<gw2-mac> -> dec_ttl, mod_dl_src:<router-mac-port2>, output:2
```

The 3-stage design reduces flow table size: adding a subnet does not cause a cross-product, it only adds entries in table 1.

## 9.18.Y ECMP with the `multipath()` action

OVS supports ECMP (Equal-Cost Multi-Path) via the `multipath()` action (Part 4.9 §4.9.26). It hashes the 5-tuple to choose 1 of N next hops:

```bash
[host]# ovs-ofctl add-flow br-r \
        "ip,nw_dst=10.10.0.0/16,actions=multipath(symmetric_l4,1024,modulo_n,2,0,reg0),resubmit(,2)"

# 2 entries in table 2 for 2 next hops:
[host]# ovs-ofctl add-flow br-r \
        "table=2,reg0=0,ip,actions=mod_dl_dst:<nh1-mac>,dec_ttl,output:5"
[host]# ovs-ofctl add-flow br-r \
        "table=2,reg0=1,ip,actions=mod_dl_dst:<nh2-mac>,dec_ttl,output:6"
```

Pattern:

1. `multipath()` hashes the 5-tuple, the result modulo N (= 2 in the example) sets `reg0` to 0 or 1.
2. `resubmit` to table 2.
3. Table 2 matches the `reg0` value and applies the corresponding next hop.

`multipath()` uses a symmetric L4 hash, so forward and reverse traffic take the same path, which keeps conntrack consistent.

## 9.18.Z Anatomy Template A: `ovs-ofctl dump-flows br-r` for the L3 router pattern

> **Operating context.** Inspect an L3 router flow table when debugging packets that fail to route or that drop on TTL.

```
[host]# ovs-ofctl dump-flows br-r
 cookie=0x1, table=0, priority=100, in_port=1, ip
   actions=write_metadata:0x1,goto_table:1, n_packets=12345, ...
 cookie=0x2, table=1, priority=100, ip,nw_dst=10.0.0.0/24
   actions=mod_dl_dst:aa:bb:cc:dd:ee:01,goto_table:2, n_packets=12345, ...
 cookie=0x3, table=2, ip,dl_dst=aa:bb:cc:dd:ee:01
   actions=dec_ttl,mod_dl_src:aa:bb:cc:99:00:01,output:5, n_packets=12345, ...
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| `cookie=0x*` | Match expected per stage | Mismatch (debug confusion) | Empty | OVN cookie mapping via `ovn-detrace` |
| `priority` | Specific above general | Tie at the same priority | 0 (catch-all conflict) | First-installed wins |
| `n_packets` | Above 0 for a hot path | 0 (rule never hit) | High but stale | Clean up unused rule |
| `actions` syntax valid | Proper OpenFlow | Empty | Invalid action | Check OVS error log |
| `dec_ttl` action present | For routing flow | Missing | Forward without dec | TTL infinite loop possible |
| `mod_dl_src` plus `mod_dl_dst` | Both for L3 | Missing one | Missing both | L2 incorrect, drop at next hop |
| `goto_table:N` | Sequential | Skip table | Table does not exist | Pipeline broken |

**Failure scenarios.**

- A TTL=0 packet sent to controller (`output:CONTROLLER`): the OVN controller log shows "received packet TTL exceeded". Healthy if rare; spam means a routing loop.
- `dec_ttl` missing from the action: TTL is not decremented, so the router becomes invisible. Verify with `ip route` on the receiving host: the hop count should equal the sender hop count, not the sender hop count plus 1.
- `mod_dl_src` empty: the gateway forwards the packet with the source MAC equal to the sender MAC. The upstream router gets confused and may drop or loop.

## 9.18.W Capstone POE, dec_ttl correctness versus router speed

> **Context.** An engineer measures OVS L3 router throughput at 8 Gbps on a 10G NIC, higher than a dnsmasq plus iptables router (about 5 Gbps), and asks: "Is OVS faster because it skips the TTL decrement?"

**Claim to verify.** "OVS router is faster because it has simpler logic, skipping TTL decrement and checksum recomputation."

**Predict.** Wrong. OVS performs the full `dec_ttl` plus checksum recomputation. The speed advantage comes from datapath architecture.

**Observe (hypothetical).**

- Run `tcpdump -i eth0 -nn -X` to compare:
  - The packet entering the OVS router with TTL=64.
  - The packet leaving on a different port with TTL=63 (decrement is correct). The checksum is recomputed (verifiable with tcpdump).
- The kernel iptables router also decrements the TTL correctly (the kernel network stack does it automatically).

Compare speed:

- **OVS L3 router (kernel datapath).** 8 Gbps on a 10G NIC. Datapath cache hit, the packet bypasses userspace, fast.
- **iptables router (kernel netfilter).** 5 Gbps on a 10G NIC. Each packet traverses the netfilter chains (PREROUTING, FORWARD, POSTROUTING), with multiple hooks.
- **OVS-DPDK userspace.** 25 to 50 Gbps on a 100G NIC.

**Explain.**

1. **OVS does not skip dec_ttl.** The implementation performs full TTL decrement plus checksum recomputation in both userspace and the kernel datapath. Verify with tcpdump.
2. **The speed advantage** comes from:
   - The megaflow cache (Part 9.2): subsequent packets hit O(1) lookup.
   - No netfilter chain traversal: OVS does not go through iptables hooks.
   - Single action set: `dec_ttl` plus `mod_dl_dst` plus `output` are inside one datapath rule, the kernel executes them sequentially.
3. **iptables is slower** because of 4 to 5 chain traversals per packet (raw, mangle, nat, filter, security).
4. **Correctness is the same.** Both are RFC-compliant.

**Falsification test.**

- Capture packets before and after the OVS router. Verify TTL decrement plus checksum recomputation.
- If outgoing TTL equals incoming TTL, OVS skipped `dec_ttl` (a bug).
- If the checksum is invalid, the packet is corrupted and downstream drops it.

**Where the claim is wrong.** "Skip TTL means faster" treats correctness as negotiable. The correct statement: **OVS is faster thanks to datapath architecture (megaflow cache plus netfilter bypass), not by skipping correctness. Speed and correctness are two independent axes, not a trade-off.**

> **Rule of thumb.** Verify the behaviour (tcpdump capture before and after) before concluding "skip something for speed". Production-grade software does not skip RFC-mandated behaviour.

## Key takeaways

First, the OVS flow table is a fully functional router. Six actions, `mod_dl_src`, `mod_dl_dst`, `dec_ttl`, `set_field`, `resubmit`, `output`, are enough to perform RFC-compliant L3 forwarding. This is true for every OpenFlow version from 1.1 onward.

Second, OVS does not use the Linux IP stack when forwarding. A separate datapath in the kernel module or in userspace handles packets directly. The sysctl `ip_forward` is meaningless. Evidence: the Guided Exercise with `ip_forward=0` still routes successfully.

Third, the OVN Logical Router and the Neutron router both compile down to the same set of OpenFlow primitives. Understanding the bare flow table lets you debug at every abstraction layer above. When OVN behaves incorrectly, `ovs-ofctl dump-flows br-int` always shows the actual matching flow entries, and the knowledge from this part lets you read them.

Fourth, standalone OVS L3 is suitable only for labs and debugging. Production needs OVN or an equivalent framework to manage lifecycle, multi-tenant isolation, and distributed forwarding. The crossover threshold is operational complexity, not performance.

## Bối cảnh: từ mô hình matching sang mối quan tâm vận hành

Từ 1.0 đến 1.3, OpenFlow spec tập trung vào matching và action model. 1.0 đặt 12-tuple cơ sở; 1.1 thêm multi-table và group; 1.2 đưa OXM TLV extensible; 1.3 thêm meter, PBB, auxiliary channels. Sau 1.3, các pain point chính của matching đã được giải (ít nhất trên giấy). Nhưng operational concern vẫn tồn tại: làm sao update 1000 flow entry cùng lúc mà không tạo inconsistent state? Khi TCAM đầy và không thể add flow mới, policy nào nên quyết định evict cái nào?

OF 1.4.0 phát hành 14/10/2013 focus vào các vấn đề này. Design ý đồ cấu hình rõ ràng: "operational concerns" thay vì "matching features". Spec 1.4 thêm **bundles** cho atomic multi-message update, **eviction** cho automatic table management, và **optical port extensions** cho ROADM control trong carrier network. 1.4.1 (17/10/2014) là errata cuối cùng của branch 1.4, sau đó ONF không phát hành 1.4.2 hay 1.4.x khác.

## 4.3.1 OF 1.4 phát hành 14/10/2013, tham vọng ngữ nghĩa transaction

### Chi tiết phát hành

OpenFlow Switch Specification Version 1.4.0 chính thức phát hành ngày **14/10/2013**, wire protocol version **0x05**. Spec document 206 trang (tăng từ 128 trang của 1.3), reflect complexity của các feature mới.

Press release ONF 10/2013 mô tả:

> *"OpenFlow 1.4 focuses on operational concerns, enabling more robust deployment of SDN in production environments. Key enhancements include atomic bundle messages, flow entry eviction for managed table resource contention, and extensible port properties for optical ports."*

1.4.1 errata release 17/10/2014, fix small clarification không ảnh hưởng wire format. Cùng wire byte 0x05 cho cả 1.4.0 và 1.4.1. Archive spec tại:

```
https://opennetworking.org/wp-content/uploads/2013/10/openflow-spec-v1.4.0.pdf
https://opennetworking.org/wp-content/uploads/2014/10/openflow-switch-v1.4.1.pdf
```

### Chủ đích thiết kế: tính trưởng thành về mặt vận hành

Working groups của ONF trong 2013 tập trung vào **operational maturity**. Feedback từ early deployer (Google B4, Microsoft, carrier operator trong ONF board) chỉ ra hai pain point chính:

**Pain point 1, Partial update risk**: Controller muốn cài đặt 1000 flow rules như một nhóm atomic (ví dụ: tenant onboarding với 100 rule cho ACL + 500 rule cho routing + 400 rule cho QoS). Nếu phải cài đặt từng rule một, có khoảng thời gian "half-installed" nơi 500 rule đã active còn 500 chưa, packet rơi vào gap có behavior không deterministic. Trong case worst, packet khớp với rule cũ (đã sắp bị xóa) và new rule (chỉ installed một phần), inconsistent chuyển tiếp.

**Pain point 2, Table full**: TCAM của hardware switch có size giới hạn (Broadcom Trident2: 4K ingress ACL). Khi full và có FLOW_MOD mới đến, spec 1.3 cho switch chọn: reject với OFPERR_TABLE_FULL (controller phải xử lý retry) hoặc triển khai vendor-specific eviction (không portable). Không có standardized policy.

OF 1.4 giải quyết cả 2: bundles cho atomic update (giải pain 1), eviction cho TCAM pressure (giải pain 2).

## 4.3.2 Bundles: transaction nguyên tử cho nhóm message

### Bundle state machine

Spec 1.4 §6.3.11 định nghĩa bundle mechanism. Controller manage bundle qua state machine 4 state:

```
        OPEN
         │
         ▼
    ADD_MESSAGE (nhiều lần)
         │
         ▼
       CLOSE
         │
         ▼
   ┌────────────┐
   │            │
COMMIT     DISCARD
   │            │
   ▼            ▼
Applied     Abandoned
```

Message types:

| Message | Ý nghĩa |
|---------|---------|
| OFPT_BUNDLE_CONTROL với type=OPEN | Mở bundle mới, switch allocate resource để track |
| OFPT_BUNDLE_ADD_MESSAGE | Đóng gói một OF message (FLOW_MOD, PORT_MOD, GROUP_MOD) vào bundle |
| OFPT_BUNDLE_CONTROL với type=CLOSE | Đóng bundle (không thêm message được), switch validate toàn bộ |
| OFPT_BUNDLE_CONTROL với type=COMMIT | Apply atomically toàn bộ bundle |
| OFPT_BUNDLE_CONTROL với type=DISCARD | Huỷ bundle, không apply, resource released |

Mỗi bundle có **bundle_id** (32-bit) unique trong connection để nhận diện. Controller có thể mở nhiều bundle song song với bundle_id khác nhau.

### Hai flag quan trọng

Bundle được parameterize qua 2 flags:

**OFPBF_ATOMIC**: all-or-nothing semantic. Khi COMMIT:
- Nếu toàn bộ message trong bundle valid, switch apply tất cả atomically.
- Nếu bất kỳ message nào fail validation, switch rollback toàn bộ bundle, không apply gì. Controller nhận error message chỉ ra message nào fail.

**OFPBF_ORDERED**: message được apply theo thứ tự đã ADD. Không có OFPBF_ORDERED, switch có thể reorder message để optimize (ví dụ: batch nhiều FLOW_MOD cùng khớp với structure). Với OFPBF_ORDERED, order strict.

Hai flag có thể combine: ATOMIC + ORDERED = "apply all-or-nothing theo đúng thứ tự".

### Analogy với SQL transaction

Bundle semantic rất giống SQL transaction:

| SQL | OpenFlow Bundle |
|-----|------------------|
| BEGIN TRANSACTION | OFPT_BUNDLE_CONTROL type=OPEN |
| INSERT/UPDATE/DELETE statements | OFPT_BUNDLE_ADD_MESSAGE (wrapping FLOW_MOD etc.) |
| COMMIT | OFPT_BUNDLE_CONTROL type=COMMIT |
| ROLLBACK | OFPT_BUNDLE_CONTROL type=DISCARD |
| Isolation level | OFPBF_ATOMIC flag |

Analogy này không tình cờ. ONF Chuyển tiếp Abstractions Working Group (FAWG) thảo luận design bundle mượn concept từ database transaction, explicitly reference trong meeting minutes 2013.

### Hiểu sai thường gặp về bundles

**Hiểu sai:** "Bundle là batch để controller gửi nhiều flow_mod nhanh hơn, giảm overhead TCP."

**Thực tế:** Bundle khác hoàn toàn với batching. Batching (không có trong OpenFlow spec) là thuần performance optimization, gửi nhiều message trong một TCP segment để tiết kiệm syscall. Bundle provides **atomicity semantic**: all-or-nothing apply. Một bundle với OFPBF_ATOMIC flag có thể bị rollback toàn bộ nếu bất kỳ message nào fail validation ở commit time. OVS triển khai bundle mà không triển khai OFPBF_ATOMIC flag (chỉ hỗ trợ OPEN/ADD/COMMIT stages không có atomic rollback) là "bundle mà không phải bundle thực sự", vẫn hữu ích cho grouping nhưng mất đi transactional guarantee. Khi đọc vendor hỗ trợ matrix cho bundle, phải hỏi cụ thể: "support atomic rollback không?"

### Field hợp sử dụng: onboard tenant nguyên tử

Production use case typical: onboarding một tenant mới vào DC OpenFlow fabric. Workflow controller:

```
1. Open bundle, bundle_id=1234.
2. Add 100 FLOW_MOD cho tenant ACL (table 0).
3. Add 500 FLOW_MOD cho tenant routing (table 10).
4. Add 400 FLOW_MOD cho tenant QoS (meter + flow reference).
5. Add PORT_MOD enable tenant-facing port.
6. Close bundle.
7. Commit bundle với OFPBF_ATOMIC + OFPBF_ORDERED.
```

Nếu bất kỳ FLOW_MOD nào fail (ví dụ: TCAM table 0 full), switch reject toàn bộ bundle. Controller nhận error, có thể retry với smaller batch hoặc move tenant sang switch khác. Không có "half-onboarded tenant" inconsistent state.

## 4.3.3 Eviction flow entry: chính sách khi table đầy

### Concept §5.4.6

Eviction là mechanism switch-side để tự động remove flow entry cũ khi table full, chừa chỗ cho entry mới. Spec 1.4 định nghĩa eviction là **optional capability**, switch advertise hỗ trợ qua OFPTFPT_EVICTION bit trong table features reply.

Flow entry thêm field mới: **importance** (16-bit). Importance value thấp = flow ít quan trọng, candidate eviction trước. Value cao = flow quan trọng, protected.

Khi switch nhận FLOW_MOD mới và table full:

1. Nếu eviction không enabled, reject với OFPERR_TABLE_FULL.
2. Nếu eviction enabled, scan table, find entry có importance thấp nhất.
3. Nếu importance thấp hơn entry mới (hoặc bằng nếu tiebreaker cho phép), evict entry cũ, install entry mới.
4. Send FLOW_REMOVED notification (nếu OFPFF_SEND_FLOW_REM flag set trên entry bị evict).

### So sánh với timeout

Flow entry có 3 mechanism remove:

| Mechanism | Trigger | Ai khởi xướng |
|-----------|---------|----------------|
| idle_timeout | N giây không khớp với packet nào | Switch tự động |
| hard_timeout | N giây kể từ install (bất kể có khớp với không) | Switch tự động |
| eviction (1.4+) | Table full, có flow mới với importance cao hơn | Switch tự động khi resource pressure |
| Explicit FLOW_MOD với OFPFC_DELETE | Controller gửi message xóa | Controller chủ động |

Timeout là **self-removal theo thời gian** (passive). Eviction là **forced removal khi resource contention** (reactive). Chúng complement nhau, không thay thế.

### Field hợp sử dụng: ưu tiên loại bỏ best-effort cho flow quan trọng

Production scenario: DC có policy "critical tenant flow" (payment, authentication) phải luôn trong TCAM hardware để low-độ trễ, không thể miss-to-software. "Best-effort flow" (bulk transfer, batch jobs) có thể bị evict nếu space contended. Cấu hình:

```
Critical flow: importance=65535 (max)
Normal flow: importance=32768
Best-effort: importance=1024
Default: importance=0 (lowest)
```

Khi TCAM đầy và policy team add critical flow, eviction kicks in, remove best-effort trước. Critical không bao giờ bị evict lẫn nhau (cùng importance cao), controller phải quyết định manually qua explicit FLOW_MOD DELETE.

## 4.3.4 Optical port extensions

### Context: ONF Optical Transport WG

Đã bàn tại 3.2.3, ONF có Optical Transport Working Group (OTWG) thành lập 2013 để extend OpenFlow sang optical layer. Động cơ: carrier network có Reconfigurable Optical Add-Drop Multiplexer (ROADM) cần centralize control, tuned by controller thay vì manual CLI per device.

OF 1.4 là spec đầu tiên có optical port descriptor native trong base spec, không cần extension riêng.

### Optical port properties (§7.2.2.2)

Port descriptor thêm fields optical-specific:

| Field | Ý nghĩa | Đơn vị |
|-------|---------|--------|
| tx_min_freq_lmda | Minimum transmit wavelength | THz, ITU-T G.694.1 grid |
| tx_max_freq_lmda | Maximum transmit wavelength | THz |
| tx_grid_freq_lmda | Grid spacing (12.5 / 25 / 50 / 100 GHz) | GHz |
| rx_min_freq_lmda | Minimum receive wavelength | THz |
| rx_max_freq_lmda | Maximum receive wavelength | THz |
| tx_pwr_min | Minimum transmit power | 0.1 dBm |
| tx_pwr_max | Maximum transmit power | 0.1 dBm |

ITU-T G.694.1 "Spectral grids for WDM applications" định nghĩa standardized wavelength grid cho DWDM. OF 1.4 reference grid này cho wavelength value encoding.

### Use case: ROADM control qua OpenFlow

Carrier network topology:

```
[Router A] ---Fiber--- [ROADM 1] ---Fiber--- [ROADM 2] ---Fiber--- [Router B]
                            │
                         OpenFlow
                         Controller
```

Controller issue PORT_MOD đến ROADM để configure wavelength routing: incoming wavelength λ1 on port X → outgoing wavelength λ2 on port Y. ROADM triển khai qua wavelength-selective switch (WSS).

ONF press release 10/2013 highlight use case này như flagship của 1.4. NEC và Ciena triển khai ROADM hỗ trợ OpenFlow trong 2014-2015 product line.

### Thực tế: niche deployment

Việc áp dụng OpenFlow optical rất niche. Hầu hết carrier deployment sử dụng:
- **NETCONF/YANG** cho config management (Khối V).
- **OpenROADM project** (AT&T driven, 2016+) với YANG models riêng cho optical, không dùng OpenFlow.
- **Transport-PCE trong ONOS** (2017+) như alternative controller framework.

OF 1.4 optical extensions trong spec nhưng production traction giảm dần sau 2016. Forward reference sang 16.1 "SDN in optical network" (đã xóa scope rev 3 nhưng lineage) để hiểu post-OpenFlow evolution.

## 4.3.5 Thực tế áp dụng: OVS partial support, vendor skepticism

### OVS triển khai 1.4 partial

Open vSwitch 2.5 (03/2016) thêm hỗ trợ 1.4 partial:

**Implemented**:
- Wire protocol 0x05 handshake (version negotiation).
- Bundle OPEN/ADD/CLOSE/COMMIT/DISCARD messages.
- OFPBF_ATOMIC flag semantic.

**Không implement**:
- Eviction: OVS dùng `--max-entries` config limit nhưng không có importance-based eviction logic native.
- Optical port properties: OVS không target optical hardware, irrelevant.
- Some minor OXM extensions của 1.4 (ví dụ metadata variants).

OVS upstream discussion trong 2014-2016 (mailing list ovs-dev@) chỉ ra reviewer skeptical về cost/benefit của full hỗ trợ 1.4. Ben Pfaff (core maintainer) comment:

> *"We triển khai 1.4 bundles because users ask for atomic update, but eviction makes OVS datapath much more complex for little benefit. Users can triển khai policy controller-side."*

Eviction được xếp là "won't fix" trong OVS.

### Vendor hardware skip

Hardware switch vendor skip 1.4 gần như hoàn toàn:

- **HP ProCurve/Aruba**: commit đến 1.3 complete, 1.4 chỉ "announced hỗ trợ" trong spec sheet, không có real hardware acceleration.
- **Pica8 PicOS**: roadmap 2014-2015 lists 1.4 "planned" nhưng không ship.
- **NEC IP8800**: dừng OpenFlow hardware line sau 2016, không bao giờ triển khai 1.4.
- **Centec V350**: 1.3 focus, 1.4 không có tên trong spec sheet.

Ryu framework tutorial và OpenDaylight plugin dừng tại 1.3 cho primary samples, 1.4 chỉ có stub code không complete.

### Economics analysis

Tại sao vendor skip? Lý do thuần kinh tế:

1. **Market signal**: 1.4 release 10/2013. Cùng thời điểm, enterprise SDN customer (DC operator) đã complete pilot 1.0/1.3, đang chuyển sang production rollout. Họ không muốn retool cho version mới mỗi 14 tháng.

2. **Engineering cost**: Implement full 1.4 trên ASIC yêu cầu new microcode, verification suite mới (ONF conformance test mới cho 1.4), documentation update. Cost vendor estimate $2-5M per product line. Revenue incremental từ 1.4 features không justify.

3. **Competitor không chịu**: Cisco và Juniper bắt đầu pivot sang SDN via APIs (NETCONF/YANG, controller-centric thay vì protocol-centric, Khối V), không có incentive push OpenFlow version mới.

4. **P4 on horizon**: Bosshart et al. phát hành P4 paper CCR 2014 (gần như đồng thời với 1.4.1 errata), tạo momentum alternative cho programmable data plane. Vendor nhìn P4 + Tofino là future, không phải OpenFlow 1.4/1.5.

Đây là khởi đầu của "OpenFlow stalled" narrative. Sau 1.4, evolution spec mất dần momentum, dẫn đến 1.5 (12/2014) gần như DOA. Cross-ref 4.6.7 phân tích báo cáo hậu sự lesson learned và rise của P4 + hypervisor overlay.

## ▶ Guided Exercise — Bundle atomic transaction với OVS (audit P4.B4.1 thực hành)

**Mục tiêu:** Thực nghiệm bundle semantics, verify all-or-nothing apply khi có validation error.

**Bước:**

1. Create bridge OF 1.4+:
   ```bash
   $ sudo ovs-vsctl add-br br-bundle -- set bridge br-bundle protocols=OpenFlow14
   ```
2. Bundle 3 flow_mod atomic (one invalid rule → all rollback):
   ```bash
   $ cat > /tmp/bundle.txt <<EOF
   bundle add-flow
   table=0,priority=100,ip,nw_src=10.0.0.0/24,actions=drop
   bundle add-flow
   table=0,priority=100,ip,nw_src=10.1.0.0/24,actions=drop
   bundle add-flow
   table=0,priority=100,INVALID_FIELD=bad,actions=drop
   EOF
   $ sudo ovs-ofctl -O OpenFlow14 bundle br-bundle /tmp/bundle.txt
   # Kỳ vọng: commit fail, all 3 rollback.
   ```
3. Verify no partial apply:
   ```bash
   $ sudo ovs-ofctl -O OpenFlow14 dump-flows br-bundle
   # Kỳ vọng: empty (all 3 rolled back)
   ```
4. Correct bundle (all valid):
   ```bash
   $ cat > /tmp/bundle-ok.txt <<EOF
   bundle add-flow
   table=0,priority=100,ip,nw_src=10.0.0.0/24,actions=drop
   bundle add-flow
   table=0,priority=100,ip,nw_src=10.1.0.0/24,actions=drop
   EOF
   $ sudo ovs-ofctl -O OpenFlow14 bundle br-bundle /tmp/bundle-ok.txt
   $ sudo ovs-ofctl -O OpenFlow14 dump-flows br-bundle | wc -l
   # Kỳ vọng: 2 rules applied atomically
   ```

**POE:**
- **Predict:** Nếu bundle có 1000 flow_mod và 1 invalid, bao nhiêu rule được apply?
- **Observe:** 0 rule — all-or-nothing. Bundle atomic guarantee.
- **Explain:** Bundle avoid "half-installed" state mà controller phải xử lý manually trong OF 1.3. Production use case: multi-tenant onboarding 100+ rule atomic per tenant.

**Cleanup:** `sudo ovs-vsctl del-br br-bundle && rm /tmp/bundle*.txt`

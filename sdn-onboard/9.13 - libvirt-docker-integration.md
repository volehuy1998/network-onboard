## 9.13.1 Libvirt with OVS

Compass Ch S pattern: in the VM domain XML, use `<virtualport type='openvswitch'/>`:

```xml
<interface type='bridge'>
  <mac address='52:54:00:aa:bb:01'/>
  <source bridge='br-int'/>
  <virtualport type='openvswitch'/>
  <model type='virtio'/>
</interface>
```

Libvirt automatically:

1. Creates a tap interface (for example `vnet0`).
2. Calls `ovs-vsctl add-port br-int vnet0`.
3. Calls `ovs-vsctl set Interface vnet0 external_ids:attached-mac=... external_ids:iface-id=<uuid>`.

`iface-id` is how OVN (and OpenStack Neutron) discover the port that belongs to a specific VM.

VIF plug troubleshooting:

```bash
[host]# ovs-vsctl list Interface | grep -A2 vnet
[host]# virsh domiflist <vm-name>
```

## 9.13.2 Docker with OVS

Compass Ch T defines two patterns:

**Pattern 1, the `ovs-docker` helper** (simple):

```bash
[host]# ovs-docker add-port br0 eth0 <container-id> \
         --ipaddress=10.0.0.5/24 --gateway=10.0.0.1
[host]# ovs-docker del-port br0 eth0 <container-id>
[host]# ovs-docker set-vlan br0 eth0 <container-id> 100
```

The script creates a veth pair, moves it into the container namespace, and plugs it into OVS.

**Pattern 2, manual veth** (production):

```bash
[host]# ip link add veth0 type veth peer name veth0-br
[host]# ip link set veth0-br up
[host]# ovs-vsctl add-port br0 veth0-br -- \
         set Interface veth0-br external_ids:container_id=<container-id>
[host]# PID=$(docker inspect -f '{{.State.Pid}}' <container-id>)
[host]# ip link set veth0 netns $PID
[host]# nsenter -t $PID -n ip addr add 10.0.0.5/24 dev veth0
[host]# nsenter -t $PID -n ip link set veth0 up
```

Manual mode gives fine-grained control (QoS, Mirror, per-veth OpenFlow rules).

## 9.13.3 Security baseline (compass Ch U)

**File permissions:**

```bash
[host]# chown openvswitch:openvswitch /etc/openvswitch
[host]# chmod 0750 /etc/openvswitch
[host]# chmod 0755 /var/run/openvswitch
[host]# chmod 0400 /etc/openvswitch/key.pem
```

**TLS is mandatory.** Every Controller and Manager target uses `ssl:` (Part 9.10). Never use `ptcp:` in production.

**IPsec for tunnels** (compass Ch 12):

```bash
[host]# ovs-vsctl add-port br0 vxlan-ipsec -- \
         set Interface vxlan-ipsec type=vxlan \
         options:remote_ip=10.0.0.2 options:key=flow \
         options:psk=mysecret
[host]# systemctl enable --now openvswitch-ipsec
```

Verify with `ipsec statusall` (strongSwan or Libreswan).

**SELinux and AppArmor.** The distribution ships a default profile. Do not disable it.

- **Windows:** Hyper-V extensible-switch forwarding extension. The same `ovs-vsctl` and `ovs-ofctl` commands apply.
- **OpenBSD and NetBSD:** userspace datapath only, no DPDK or AF_XDP.

Recommended scope: production Linux. Windows, OpenBSD, and NetBSD remain for special use cases.

## 9.13.6 libvirt-OVS protocol contract deep dive

### `external_ids:iface-id` semantics

When libvirt plugs a VM into an OVS bridge with `<virtualport type='openvswitch'/>`, libvirt automatically sets `external_ids:iface-id` on the OVS Interface row. This is the **contract** between the orchestrator and OVN or Neutron for discovering the port that belongs to a specific VM.

Source code anchor (libvirt 8.0+, Ubuntu 22.04):

```c
// src/util/virnetdevopenvswitch.c, function virNetDevOpenvswitchAddPort()
// Pseudo-flow:
//   1. virCommandNew("ovs-vsctl")
//   2. virCommandAddArgList("add-port", brname, ifname, "--", ...)
//   3. virCommandAddArgList("set", "Interface", ifname,
//                            "external-ids:attached-mac=...",
//                            "external-ids:iface-id=...",
//                            "external-ids:vm-uuid=...")
```

**Format of `iface-id`:**

- **OVN-Kubernetes:** `<namespace>_<pod-name>` (for example `default_my-app-pod-abc123`).
- **OpenStack Neutron:** the Neutron port UUID (for example `12345678-1234-...`).
- **Libvirt direct (no orchestrator):** typically the MAC address or the VM UUID.

The OVN `ovn-controller` subscribes to the SBDB `Port_Binding` table on `name = iface-id`. When it sees a match, it claims the port and installs flows on `br-int`.

### Libvirt event hooks (qemu hooks)

Libvirt supports qemu hooks at `/etc/libvirt/hooks/qemu` for lifecycle events:

```bash
#!/bin/bash
# /etc/libvirt/hooks/qemu
DOMAIN=$1
EVENT=$2

case "$EVENT" in
    prepare|start)
        # VM about to start, ensure OVS bridge exists
        ovs-vsctl --may-exist add-br br-int
        ;;
    stopped|release)
        # VM stopped, clean up orphan OVS port
        for port in $(ovs-vsctl list-ports br-int | grep "vnet"); do
            if ! virsh domiflist "$DOMAIN" 2>/dev/null | grep -q "$port"; then
                ovs-vsctl --if-exists del-port br-int "$port"
            fi
        done
        ;;
esac
```

The hook ensures idempotency: a VM start retry does not create a duplicate port, and a VM stop cleans up the orphan veth.

### Hot-plug semantics

The VM is running. Attach a new NIC with `virsh attach-device <vm> nic.xml`:

```xml
<interface type='bridge'>
  <source bridge='br-int'/>
  <virtualport type='openvswitch'/>
  <model type='virtio'/>
</interface>
```

Libvirt then:

1. Sends `device_add virtio-net-pci` to `qemu-monitor`.
2. Creates a tap interface (`vnet1`).
3. Runs `ovs-vsctl add-port br-int vnet1 ... external_ids:iface-id=...`.
4. The VM kernel detects the new NIC and registers the driver.

The whole flow takes less than 1 second on a healthy cluster. Failure mode: if the OVS bridge does not exist, libvirt warns and the NIC stays detached. Fix: create the bridge first, then retry the attach.

## 9.13.7 Docker-OVS container netns lifecycle

### Container start workflow

Docker does not understand OVS. The manual workflow is:

```
docker run --network=none -> container netns created, lo only
                          v
        operator or script creates veth pair
                          v
        ovs-vsctl add-port br-int <veth-host>
                          v
        ip link set <veth-container> netns <pid>
                          v
        nsenter -t <pid> -n ip addr add ...
```

Problem: Docker does not know about the veth, so a container restart leaves the veth orphaned on the OVS bridge.

### Container restart pattern

Three restart policies:

- `--restart=no` (default): the container exits, Docker does not restart it, an orphan veth remains on OVS.
- `--restart=on-failure`: Docker restarts the container, but creates a new netns (the PID is different), so the old veth is orphaned.
- `--restart=always`: behaves like `on-failure`.

A healthy production setup needs a cleanup mechanism:

```bash
# Cron daily orphan veth cleanup
[host]# for port in $(ovs-vsctl list-ports br-int | grep -E '^c[0-9a-f]{12}'); do
[host]>     container_id=$(echo "$port" | sed 's/^c//')
[host]>     if ! docker ps -q --no-trunc | grep -q "$container_id"; then
[host]>         echo "Orphan port: $port (container $container_id)"
[host]>         ovs-vsctl --if-exists del-port br-int "$port"
[host]>     fi
[host]> done
```

### Veth orphan failure mode

Production symptom:

```bash
[host]# ovs-vsctl list-ports br-int | wc -l
342

[host]# docker ps -q | wc -l
85

# 342 - 85 = 257 orphan ports
```

Cause: a container restart loop over a few weeks, one orphan per restart. OVS does not auto-clean.

Fix: a cron job and a monitoring metric. Production OVN-Kubernetes has a CNI plugin that auto-cleans (the `ovn-cni` ADD and DEL hooks). Standalone Docker requires a custom script.

### Atomic transaction batch

When 100 containers restart at the same time, a race condition appears as 100 processes run `ovs-vsctl add-port` in parallel:

```bash
# BAD: race condition
for c in c1 c2 ... c100; do
    ovs-vsctl add-port br-int veth-$c &
done
wait
```

OVSDB transactions can conflict, retry, and slow down.

The healthy pattern is a single batched transaction:

```bash
# GOOD: 1 transaction
cmd="ovs-vsctl"
for c in c1 c2 ... c100; do
    cmd="$cmd -- add-port br-int veth-$c"
done
$cmd
```

100 add-port operations commit atomically in a single transaction in less than 100 ms total, instead of more than 10 seconds with the race.

## 9.13.8 Production security baseline expanded

### File permission and ownership matrix

| Path | Owner:Group | Mode | Reason |
|------|-------------|------|--------|
| `/etc/openvswitch` | openvswitch:openvswitch | 0750 | Configuration directory, only daemon and admin can read |
| `/etc/openvswitch/conf.db` | openvswitch:openvswitch | 0640 | OVSDB file, contains state |
| `/etc/openvswitch/cacert.pem` | openvswitch:openvswitch | 0644 | Public certificate, world-readable |
| `/etc/openvswitch/cert.pem` | openvswitch:openvswitch | 0644 | Public certificate |
| `/etc/openvswitch/privkey.pem` | openvswitch:openvswitch | 0400 | Private key, daemon-only read |
| `/var/run/openvswitch` | openvswitch:openvswitch | 0755 | Socket directory |
| `/var/run/openvswitch/db.sock` | openvswitch:openvswitch | 0660 | OVSDB socket, group write |
| `/var/log/openvswitch` | openvswitch:openvswitch | 0750 | Log directory |

Verify:

```bash
[host]# stat -c '%U:%G %a %n' /etc/openvswitch/privkey.pem
openvswitch:openvswitch 400 /etc/openvswitch/privkey.pem
```

### TLS mandatory for controller and manager

Cross-link Part 9.10 (TLS PKI tier 2). Quick check:

```bash
[host]# ovs-vsctl list Manager | grep target
target              : "ssl:192.0.2.1:6640"     <- good

[host]# ovs-vsctl list Manager | grep target
target              : "tcp:192.0.2.1:6640"     <- BAD, plaintext
```

Any `ptcp:` (passive TCP) or `tcp:` (active TCP) in production is a security violation.

### SELinux module

The OVS package ships a SELinux policy:

```bash
[host]# rpm -ql openvswitch-selinux | grep '\.pp$'    # RHEL
/usr/share/selinux/packages/openvswitch.pp

[host]# semodule -l | grep openvswitch
openvswitch    1.0
```

The module defines context for:

- `openvswitch_t` for the daemon.
- `openvswitch_var_run_t` for `/var/run/openvswitch`.
- `openvswitch_etc_t` for `/etc/openvswitch`.
- Allows the daemon to `connectto` socket TCP ports 6640, 6641, 6642, and 6653.

Do NOT disable SELinux in production. In particular, avoid `setenforce 0` "temporarily".

### AppArmor profile (Ubuntu)

Ubuntu 22.04 ships `/etc/apparmor.d/usr.sbin.ovs-vswitchd`:

```
[host]# aa-status | grep ovs
   /usr/sbin/ovs-vswitchd
   /usr/sbin/ovsdb-server
```

The profile is in `enforce` mode by default. Verify:

```bash
[host]# aa-status | head -20
apparmor module is loaded.
Profiles loaded: ...
   In enforce mode: 30
   In complain mode: 0
```

## 9.13.9 Anatomy Template A: `ovs-vsctl list Interface external_ids`

> **Context:** Inspect the `external_ids` contract on an Interface row. Use this when debugging cases where OVN or another orchestrator does not claim a port, or where a port is stuck.

```bash
[host]# ovs-vsctl list Interface vnet0
_uuid               : a1b2c3d4-...
external_ids        : {attached-mac="52:54:00:aa:bb:01", \
                       iface-id="default_my-pod-abc123", \
                       iface-id-ver="789def", \
                       iface-status="active", \
                       neutron:port_id="12345678-...", \
                       vm-uuid="9f8e7d6c-..."}
ifindex             : 23
mac_in_use          : "52:54:00:aa:bb:01"
name                : "vnet0"
ofport              : 15
ofport_request      : []
options             : {}
statistics          : {rx_packets=12345, tx_packets=23456, ...}
status              : {driver_name=tun}
type                : ""
```

| `external_ids` key | Healthy | Warning | Critical | Diagnostic hint |
|--------------------|---------|---------|----------|-----------------|
| `attached-mac` | Matches `mac_in_use` | Mismatch | Empty | Libvirt set the value incorrectly |
| `iface-id` | Matches SBDB `Port_Binding` name | Empty (port not yet claimed) | Mismatch | Orchestrator-OVN sync issue |
| `iface-id-ver` | Increments when port is recreated | Stale | Empty | Version conflict; OVN does not pick up the update |
| `iface-status` | `active` for a live port | `inactive` (host disconnected) | Empty | Hot-plug failed |
| `neutron:port_id` | Neutron port UUID | (n/a non-Neutron) | Mismatch with `iface-id` | OpenStack inconsistency |
| `vm-uuid` | Matches the libvirt VM UUID | (n/a non-libvirt) | Empty for a libvirt VM | Libvirt did not set it |
| `container_id` | Matches the Docker ID | (n/a non-Docker) | Stale (container exited) | Orphan port |
| `mac_in_use` | Matches `attached-mac` | Mismatch | Empty | NIC driver issue |
| `ofport` | A number greater than 0 | -1 | Empty | OpenFlow port not assigned |

**Failure scenarios:**

- `iface-id` is empty on a VM port: libvirt did not set it, or the XML is missing `<virtualport type='openvswitch'/>`. Check with `virsh dumpxml <vm> | grep virtualport`.
- `iface-id` is set but `ovn-controller` does not claim it: the SBDB `Port_Binding` `name` field does not match. Check with `ovn-sbctl list Port_Binding | grep <iface-id>`.
- `container_id` is stale (the Docker container has been removed): orphan port. Use the cleanup script in §9.13.7.
- `iface-id-ver` differs between libvirt and OVN: the orchestrator updated the version but `ovn-controller` has not picked it up. Force an `ovn-controller` restart, or reload `ovsdb-server`.

## 9.13.10 Capstone POE: `ovs-docker` helper versus manual veth, which is production-grade?

> **Context:** A production cluster with Docker, more than 100 containers, restart frequency 5 to 10 per hour. The engineer chose the `ovs-docker` helper because the syntax is shorter and simpler than manual veth.

**Claim to verify:** "The `ovs-docker` helper is simpler than manual veth, so it is more stable for production."

**Predict:** Wrong. `ovs-docker` is a bash script wrapper. It is not transactional, not atomic, and races when multiple containers restart at the same time.

**Observe (assumed):**

- 100 containers with `restart-policy=on-failure`, simulated 10 percent failure rate per minute (10 containers fail and restart per minute).
- Use `ovs-docker add-port` per container restart: bash invokes 5 to 10 sequential `ovs-vsctl` calls, 200 to 500 ms per add-port.
- Race condition: 2 containers restart at the same time, 2 `ovs-docker` invocations both try `add-port` with the same port name (if the user configured it that way), 1 fails and 1 succeeds, leaving an orphan state.
- After 1 hour: 30 or more orphan ports, 5 to 10 containers without a network (silent fail).

Manual veth with atomic batch:

- A script batches 10 container restarts into a single OVSDB transaction.
- One transaction is atomic across all 10 add-port and `set external_ids` operations.
- Total time is less than 100 ms for 10 containers.
- No race condition (OVSDB serializes transactions internally).
- After 1 hour: 0 orphans, 100 percent of containers have a network.

**Explain:**

1. **`ovs-docker` is a bash convenience script**, suitable for development or test (1 to 2 containers). The source code (`utilities/ovs-docker.sh`) is about 200 lines of bash, with no transaction support, no retry, no atomicity.
2. **Production needs atomic OVSDB transactions:** use `ovs-vsctl --no-wait <op1> -- <op2> -- <op3>` to batch multiple operations in a single transaction.
3. **Race conditions in bash scripts:** parallel invocations lead to non-deterministic state. Testing with 100 containers may not reveal it because timing is luck-based. The bug surfaces at 1000 or more.
4. **The manual veth pattern** (see §9.13.2 Pattern 2) allows fine-grained control: per-container QoS, mirror, and OpenFlow policy.

**Falsification test:**

- Set up 100 containers, restart loop for 1 hour with `ovs-docker`. Measure the orphan count.
- Set up 100 containers, restart loop with manual plus batch. Measure the orphan count.
- If `ovs-docker` orphan count is 0, that is a counter-example. In practice, `ovs-docker` always produces orphans at scale.

**Where the claim is wrong:** "Simpler, so more stable" is wrong. Simpler syntax does not equal atomic and transactional behaviour. Production requires transactional integrity, not syntactic sugar.

> **Rule:** `ovs-docker` is for development or test with 1 to 2 containers only. Production means manual veth with an OVSDB atomic batch transaction.

> **Note (per directive ban):** the OVN-Kubernetes CNI plugin (Part 15.x) handles this container lifecycle pattern automatically through CNI ADD and DEL hooks. We do not deep-dive here in Part 9.13. Cross-link to Part 15.x for the deployment context.

### Guided Exercise 14: Plug a container into OVS through manual veth

**Outcomes:** Create a container, plug a veth into OVS, send traffic through OVS.

**Before You Begin:**

```bash
[host]# ovs-vsctl add-br br-docker
[host]# docker run -dit --name c1 --network none ubuntu:22.04 sleep 3600
```

**Instructions:**

**1.** Create a veth pair:

```bash
[host]# ip link add c1-veth type veth peer name c1-br
[host]# ip link set c1-br up
```

**2.** Plug into OVS:

```bash
[host]# ovs-vsctl add-port br-docker c1-br -- \
         set Interface c1-br external_ids:container_id=c1
```

**3.** Move `c1-veth` into the container namespace:

```bash
[host]# PID=$(docker inspect -f '{{.State.Pid}}' c1)
[host]# ip link set c1-veth netns $PID
[host]# nsenter -t $PID -n ip addr add 10.99.0.1/24 dev c1-veth
[host]# nsenter -t $PID -n ip link set c1-veth up
```

**4.** Verify: `docker exec c1 ip addr show c1-veth`.

**5.** Cleanup: `docker rm -f c1 && ovs-vsctl del-br br-docker`.

## Key points to remember

First, libvirt with OVS through `<virtualport type='openvswitch'/>` is the standard pattern. Libvirt auto-plugs the VM with `external_ids` metadata.

Second, Docker with OVS has two patterns: `ovs-docker` (development) and manual veth (production).

Third, the security baseline is non-negotiable: strict file permissions, TLS, IPsec, SELinux and AppArmor enabled.

Fourth, `ovs-sandbox` is an important development tool. It allows testing without a kernel module or a physical NIC.

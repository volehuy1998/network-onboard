## 20.4.1 Category 1 — Daily health check OVS (morning routine)

5 lệnh health check snapshot OVS mỗi sáng. Mỗi lệnh < 1 giây, tổng < 10 giây.

### 20.4.1.1 Bridge + port topology — `ovs-vsctl show`

**Bối cảnh khi chạy:** Lệnh đầu tiên mỗi sáng, confirm bridge state không drift sau đêm. Cũng run khi take-over shift oncall.

**Command và output** *(reproduced from OVS 2.17.9, 2-bridge topology)*:

```shell
$ sudo ovs-vsctl show

5e83f20e-8f3d-4a2b-9c3d-abcdef012345
    Bridge br-int
        fail_mode: secure
        datapath_type: system
        Port br-int
            Interface br-int
                type: internal
        Port "tap-vm01"
            Interface "tap-vm01"
                external_ids: {attached-mac="aa:bb:cc:00:00:01", iface-id="lsp_web_01"}
        Port "tap-vm02"
            Interface "tap-vm02"
                external_ids: {attached-mac="aa:bb:cc:00:00:02", iface-id="lsp_web_02"}
        Port "ovn-chassisB-0"
            Interface "ovn-chassisB-0"
                type: geneve
                options: {csum="true", key=flow, remote_ip="10.0.0.12"}
    Bridge br-ex
        Port "eth1"
            Interface "eth1"
        Port "patch-br-int-to-br-ex"
            Interface "patch-br-int-to-br-ex"
                type: patch
                options: {peer="patch-br-ex-to-br-int"}
    ovs_version: "2.17.9"
```

**Anatomy block cấu trúc:**

| Line pattern | Giá trị mẫu | Ý nghĩa | Dấu hiệu đáng lưu ý |
|--------------|-------------|---------|---------------------|
| `<uuid>` line 1 | `5e83...345` | UUID của Open_vSwitch root table | Thay đổi = ovs-vswitchd restart gần đây |
| `Bridge <name>` | `Bridge br-int` | OVS bridge definition | br-int = OVN managed; br-ex = external gateway |
| `fail_mode: <mode>` | `secure` | Behavior khi controller disconnect | `secure` = drop new flow (§9.14.6 Scenario I); `standalone` = hub mode |
| `datapath_type: <type>` | `system` | Kernel (system) vs userspace (netdev DPDK) | `netdev` = DPDK PMD mode |
| `Port <name>` + `Interface <name> type: <T>` | `type: internal/geneve/patch/dpdk/...` | Interface type per port | Thiếu `type` = physical NIC |
| `external_ids` | `{iface-id="lsp_web_01"}` | Metadata (CMS set) | OVN dùng `iface-id` để bind Port_Binding |
| `options` tunnel | `{remote_ip="10.0.0.12", key=flow}` | Tunnel endpoint config | `key=flow` = dynamic tunnel_key per flow |
| `ovs_version` | `"2.17.9"` | Running version | Compare với apt version sau upgrade |

**Kịch bản bẻ gãy:**

- **Bridge thiếu `br-int`** → OVN không functional. Check ovn-controller log.
- **Port có `type: internal` nhưng không có upper veth** → orphan internal port. Xóa bằng `ovs-vsctl del-port`.
- **Tunnel port `options:remote_ip` empty** → tunnel không thể form. Check Encap table trong SBDB.
- **`fail_mode: secure` + controller disconnect → traffic stop.** Fix: `set-fail-mode standalone` tạm thời.
- **`ovs_version` mismatch giữa các chassis** → rolling upgrade đang dở, có thể drift behavior.

### 20.4.1.3 Datapath flow sanity — `ovs-dpctl show`

```shell
$ sudo ovs-dpctl show
system@ovs-system:
  lookups: hit:8392751 missed:12783 lost:0
  flows: 4823
  masks: hit:8405534 total:12 hit/pkt:1.00
  port 0: ovs-system (internal)
  port 1: br-int (internal)
  port 2: tap-vm01
  port 3: tap-vm02
  port 4: ovn-chassisB-0 (geneve: packet_type=ptap)
```

**Đọc output:**

- `lookups: hit` >> `missed` → cache effective (hit rate 99.85%).
- `lost: 0` → không upcall drop (§9.14.6 Scenario C).
- `flows: 4823` → current datapath flow count.
- `masks: total: 12` → 12 megaflow mask patterns (low = good, cache hiệu quả).
- `hit/pkt: 1.00` → avg 1 mask lookup per packet (ideal).

### 20.4.1.5 OpenFlow flow table count

```shell
$ sudo ovs-ofctl dump-aggregate br-int
NXST_AGGREGATE reply (xid=0x4): packet_count=8392751 byte_count=12583720473 flow_count=3218

$ sudo ovs-ofctl dump-flows br-int | wc -l
3219   # +1 cho header
```

Threshold: < 10K flow = healthy. 10-50K = warning. > 50K = HW offload limit + revalidator stress.

**Consolidated daily health script:**

```bash
#!/bin/bash
# /usr/local/bin/ovs-health-check.sh
set -e
echo "=== Bridge topology ==="
sudo ovs-vsctl show | head -30
echo ""
echo "=== Daemon status ==="
systemctl is-active openvswitch-switch ovsdb-server
sudo ovs-appctl -t ovs-vswitchd version
echo ""
echo "=== Datapath sanity ==="
sudo ovs-dpctl show
echo ""
echo "=== Upcall health ==="
sudo ovs-appctl upcall/show | head -10
echo ""
echo "=== Flow count per bridge ==="
for br in $(sudo ovs-vsctl list-br); do
    count=$(sudo ovs-ofctl dump-aggregate $br | grep -oE 'flow_count=[0-9]+' | cut -d= -f2)
    echo "$br: $count flows"
done
```

## 20.4.3 Category 3 — Bridge + port lifecycle

### 20.4.3.1 Add new bridge

```shell
# Basic bridge (kernel datapath)
$ sudo ovs-vsctl add-br br-lab

# Với datapath type explicit
$ sudo ovs-vsctl add-br br-lab -- set Bridge br-lab datapath_type=system

# DPDK userspace datapath
$ sudo ovs-vsctl add-br br-dpdk -- set Bridge br-dpdk datapath_type=netdev

# Kiểm chứng
$ sudo ovs-vsctl list-br
br-int
br-lab
br-dpdk
```

### 20.4.3.2 Add port với type khác nhau

```shell
# Physical NIC port
$ sudo ovs-vsctl add-port br-lab eth1

# Internal port (dùng cho host IP trên bridge)
$ sudo ovs-vsctl add-port br-lab bridge-iface \
    -- set Interface bridge-iface type=internal

# Patch port (nối 2 bridge)
$ sudo ovs-vsctl add-port br-int patch-br-int-br-ex \
    -- set Interface patch-br-int-br-ex type=patch options:peer=patch-br-ex-br-int
$ sudo ovs-vsctl add-port br-ex patch-br-ex-br-int \
    -- set Interface patch-br-ex-br-int type=patch options:peer=patch-br-int-br-ex

# Geneve tunnel
$ sudo ovs-vsctl add-port br-int tun-chassisB \
    -- set Interface tun-chassisB type=geneve \
       options:remote_ip=10.0.0.12 \
       options:key=flow \
       options:csum=true

# VXLAN tunnel
$ sudo ovs-vsctl add-port br-int vxlan-chassisB \
    -- set Interface vxlan-chassisB type=vxlan \
       options:remote_ip=10.0.0.12 \
       options:key=flow

# DPDK physical port
$ sudo ovs-vsctl add-port br-dpdk dpdk0 \
    -- set Interface dpdk0 type=dpdk options:dpdk-devargs=0000:01:00.0

# DPDK vhost-user (cho VM)
$ sudo ovs-vsctl add-port br-dpdk vhost-user-1 \
    -- set Interface vhost-user-1 type=dpdkvhostuser
```

### 20.4.3.3 Set port option + external_ids

```shell
# OVN iface-id binding
$ sudo ovs-vsctl set Interface tap-vm01 external_ids:iface-id=lsp_web_01

# VLAN tag cho port
$ sudo ovs-vsctl set Port tap-vm01 tag=100

# VLAN trunk
$ sudo ovs-vsctl set Port eth1 trunks=100,200,300

# Disable port STP
$ sudo ovs-vsctl set Port tap-vm01 other_config:stp-enable=false
```

### 20.4.3.4 Remove port + bridge

```shell
# Remove port
$ sudo ovs-vsctl del-port br-int tap-vm01

# Remove bridge (cascade port)
$ sudo ovs-vsctl del-br br-lab

# Kiểm chứng
$ sudo ovs-vsctl list-br | grep br-lab
# (empty)
```

### 20.4.3.5 Administrative up/down port

```shell
# Admin down (traffic ngừng trên port)
$ sudo ovs-ofctl mod-port br-int tap-vm01 down

# Admin up
$ sudo ovs-ofctl mod-port br-int tap-vm01 up

# Check
$ sudo ovs-ofctl show br-int | grep tap-vm01
 3(tap-vm01): addr:aa:bb:cc:00:00:01
     config:     0
     state:      0
     current:    10GB-FD COPPER AUTO_NEG
     ...
```

`config: 0` = admin up; `config: PORT_DOWN` = admin down.

### 20.4.3.6 Clear mọi port trên bridge (nuke, cẩn thận!)

```shell
# Xóa tất cả port trừ internal bridge port
$ for p in $(sudo ovs-vsctl list-ports br-lab); do
    sudo ovs-vsctl del-port br-lab $p
  done
```

## 20.4.4 Category 4 — OpenFlow flow management

Đây là layer quan trọng nhất khi debug OVS pure (controller-less hoặc hybrid controller).

### 20.4.4.1 Add OpenFlow flow

```shell
# Basic flow: drop all ICMP
$ sudo ovs-ofctl add-flow br-int "priority=100,icmp,actions=drop"

# Flow với cookie (group identifier)
$ sudo ovs-ofctl add-flow br-int \
    "cookie=0xdeadbeef,priority=200,in_port=3,dl_dst=aa:bb:cc:00:00:02,actions=output:4"

# Multi-table flow với goto_table
$ sudo ovs-ofctl add-flow br-int \
    "table=0,priority=100,icmp,actions=goto_table:1"
$ sudo ovs-ofctl add-flow br-int \
    "table=1,priority=100,icmp,actions=NORMAL"

# Conntrack action
$ sudo ovs-ofctl add-flow br-int \
    "table=0,priority=100,ip,actions=ct(commit,zone=1,table=1)"

# Batch add từ file
$ sudo ovs-ofctl add-flows br-int /tmp/flows.txt
```

### 20.4.4.2 Dump flow với filter

```shell
# Tất cả flow
$ sudo ovs-ofctl dump-flows br-int | head

# Specific table
$ sudo ovs-ofctl dump-flows br-int table=0

# Specific priority range
$ sudo ovs-ofctl dump-flows br-int | awk '/priority=10[0-9]{2}/'

# By cookie
$ sudo ovs-ofctl dump-flows br-int "cookie=0xdeadbeef/-1"

# Flow matching specific criteria
$ sudo ovs-ofctl dump-flows br-int "in_port=3"
$ sudo ovs-ofctl dump-flows br-int "ip,nw_src=10.0.0.1"

# With stats
$ sudo ovs-ofctl dump-flows br-int --rsort=packets
```

### 20.4.4.3 Delete flow

```shell
# Xóa all flow cho bridge (nuke, cẩn thận!)
$ sudo ovs-ofctl del-flows br-int

# Xóa specific flow bằng match
$ sudo ovs-ofctl del-flows br-int "priority=100,icmp"

# Xóa bằng cookie
$ sudo ovs-ofctl del-flows br-int "cookie=0xdeadbeef/-1"

# Xóa all flow trong specific table
$ sudo ovs-ofctl del-flows br-int "table=1"
```

### 20.4.4.4 Modify flow (hot-update action)

```shell
# Strict modify (match exactly)
$ sudo ovs-ofctl --strict mod-flows br-int \
    "priority=100,icmp,actions=output:5"

# Non-strict (partial match prefix)
$ sudo ovs-ofctl mod-flows br-int \
    "priority=100,icmp,actions=drop"
```

### 20.4.4.5 Atomic replace-flows (hot-reload full table)

```shell
# Dump current state
$ sudo ovs-ofctl dump-flows br-int > /tmp/backup-flows.txt

# Apply new state atomically (diff + install + remove)
$ sudo ovs-ofctl replace-flows br-int /tmp/new-flows.txt

# Dry-run diff trước khi apply
$ sudo ovs-ofctl diff-flows br-int /tmp/new-flows.txt
```

`replace-flows` atomic hơn `del-flows` + `add-flows` (không có window trống giữa).

### 20.4.4.6 Flow hygiene monitor

```shell
# Monitor flow change real-time
$ sudo ovs-ofctl monitor br-int watch:

# Monitor packet-in events (controller traffic)
$ sudo ovs-ofctl snoop br-int
```

## 20.4.5 Category 5 — Tunnel management

Tunnel (Geneve/VXLAN/GRE/STT) là backbone cho overlay. 5 workflow common.

### 20.4.5.1 Create tunnel port

```shell
# Geneve (recommended cho OVN)
$ sudo ovs-vsctl add-port br-int tun-chassisB \
    -- set Interface tun-chassisB type=geneve \
       options:remote_ip=10.0.0.12 \
       options:key=flow \
       options:csum=true

# VXLAN (legacy)
$ sudo ovs-vsctl add-port br-int vxlan-chassisB \
    -- set Interface vxlan-chassisB type=vxlan \
       options:remote_ip=10.0.0.12 options:key=flow

# GRE
$ sudo ovs-vsctl add-port br-int gre-chassisB \
    -- set Interface gre-chassisB type=gre \
       options:remote_ip=10.0.0.12 options:key=42

# Static tunnel_key (thay vì flow-based)
$ sudo ovs-vsctl add-port br-int tun-static \
    -- set Interface tun-static type=geneve \
       options:remote_ip=10.0.0.12 options:key=4242
```

### 20.4.5.2 Tunnel neighbor + ARP resolution

```shell
# Check ARP resolved cho remote underlay
$ sudo ovs-appctl tnl/neigh/show

IP                                            MAC                 Bridge
==========================================================================
10.0.0.12                                     aa:cc:dd:11:22:33   br-int
10.0.0.13                                     aa:cc:dd:11:22:34   br-int

# Force ARP flush
$ sudo ovs-appctl tnl/neigh/flush
```

Empty entry = underlay ARP chưa resolve → tunnel không form (§9.14.2.3).

### 20.4.5.3 Tunnel port list + stats

```shell
# Liệt kê tunnel port
$ sudo ovs-appctl tnl/ports/show

Listening ports:
geneve_sys_6081 (6081)
vxlan_sys_4789 (4789)
```

Port 6081 = Geneve default, 4789 = VXLAN default. Nếu thiếu → tunnel type đó không configured.

### 20.4.5.4 Tunnel packet trace

```shell
# Trace packet qua tunnel
$ sudo ovs-appctl ofproto/trace br-int \
    'in_port=3,dl_src=aa:bb:cc:00:00:01,dl_dst=aa:bb:cc:00:00:02,
     dl_type=0x0800,nw_src=10.0.0.1,nw_dst=10.0.0.2,nw_proto=1'

# Expected final action (nếu cross-chassis):
# set(tunnel(tun_id=0x1,src=10.0.0.11,dst=10.0.0.12,flags(df|key))),output:4
```

### 20.4.5.5 Remove tunnel

```shell
$ sudo ovs-vsctl del-port br-int tun-chassisB
```

## 20.4.6 Category 6 — QoS + mirror

### 20.4.6.1 Rate-limit ingress (policing) — per-interface ingress rate

```shell
# 100 Mbps ingress policing trên port tap-vm01
$ sudo ovs-vsctl set Interface tap-vm01 \
    ingress_policing_rate=100000 \
    ingress_policing_burst=10000

# Remove
$ sudo ovs-vsctl set Interface tap-vm01 ingress_policing_rate=0
```

Units: rate kbps, burst KB.

### 20.4.6.2 Egress shaping — QoS với linux-htb

```shell
# Tạo QoS linux-htb cho port
$ sudo ovs-vsctl set Port tap-vm01 \
    qos=@newqos \
    -- --id=@newqos create QoS \
       type=linux-htb \
       other-config:max-rate=1000000000 \
       queues=0=@default-queue,1=@high-prio \
    -- --id=@default-queue create Queue \
       other-config:min-rate=500000000 \
       other-config:max-rate=800000000 \
    -- --id=@high-prio create Queue \
       other-config:min-rate=200000000 \
       other-config:max-rate=1000000000

# Kiểm chứng
$ sudo ovs-vsctl list QoS
$ sudo ovs-vsctl list Queue
```

### 20.4.6.3 Queue output từ OpenFlow rule

```shell
# Gửi SSH traffic vào queue high-prio (id=1)
$ sudo ovs-ofctl add-flow br-int \
    "priority=100,tcp,tp_dst=22,actions=set_queue:1,output:NORMAL"

# Best-effort mặc định
$ sudo ovs-ofctl add-flow br-int \
    "priority=50,actions=output:NORMAL"
```

### 20.4.6.4 Mirror port (SPAN) cho khắc phục sự cố

```shell
# Tạo mirror port
$ sudo ovs-vsctl add-port br-int tap-debug \
    -- set Interface tap-debug type=internal

# Config mirror: capture tất cả traffic qua br-int vào tap-debug
$ sudo ovs-vsctl \
```

## 20.4.11 End-to-end workflow 1 — New bridge provisioning với overlay tunnel

Scenario: Setup bridge mới `br-lab` cho test environment với 2 physical port + 1 Geneve tunnel + QoS egress shaping + OpenFlow controller Faucet.

```shell
#!/bin/bash
# new-bridge.sh br-lab 10.0.0.12 10.0.0.100:6653 1000000000

BRIDGE=$1
TUNNEL_REMOTE=$2
CONTROLLER=$3
MAX_RATE=$4

# 1. Create bridge
sudo ovs-vsctl add-br $BRIDGE

# 2. Fail mode = standalone (fall back hub khi controller crash)
sudo ovs-vsctl set-fail-mode $BRIDGE standalone

# 3. Add physical port
sudo ovs-vsctl add-port $BRIDGE eth1

# 4. Add internal port cho host IP
sudo ovs-vsctl add-port $BRIDGE bridge-iface \
    -- set Interface bridge-iface type=internal
sudo ip addr add 192.168.99.1/24 dev bridge-iface
sudo ip link set bridge-iface up

# 5. Add Geneve tunnel
sudo ovs-vsctl add-port $BRIDGE tun-remote \
    -- set Interface tun-remote type=geneve \
       options:remote_ip=$TUNNEL_REMOTE \
       options:key=flow \
       options:csum=true

# 6. Setup QoS linux-htb với egress shaping
sudo ovs-vsctl set Port eth1 \
    qos=@newqos \
    -- --id=@newqos create QoS \
       type=linux-htb \
       other-config:max-rate=$MAX_RATE \
       queues=0=@default \
    -- --id=@default create Queue \
       other-config:min-rate=500000000 \
       other-config:max-rate=$MAX_RATE

# 7. Set controller (if specified)
if [ -n "$CONTROLLER" ]; then
    sudo ovs-vsctl set-controller $BRIDGE tcp:$CONTROLLER
fi

# 8. Kiểm chứng
echo "=== Bridge $BRIDGE provisioned ==="
sudo ovs-vsctl show | grep -A 20 "Bridge $BRIDGE"
sudo ovs-ofctl show $BRIDGE | head -5

# 9. Validate tunnel ARP resolve
sleep 3
sudo ovs-appctl tnl/neigh/show | grep $TUNNEL_REMOTE || echo "WARN: tunnel ARP chưa resolve"
```

## 20.4.12 End-to-end workflow 2 — Bridge decommission safe

Scenario: Remove `br-lab` an toàn. Order quan trọng.

```shell
#!/bin/bash
# bridge-decommission.sh br-lab

BRIDGE=$1

# 1. Confirm VM không còn dùng
for tap in $(sudo ovs-vsctl list-ports $BRIDGE | grep "^tap-"); do
    echo "ERROR: $BRIDGE still has tap $tap. Migrate VM first."
    exit 1
done

# 2. Drain tunnel connections
for tun in $(sudo ovs-vsctl list-ports $BRIDGE | grep -E "^tun-|^vxlan-|^gre-"); do
    echo "Removing tunnel $tun..."
    sudo ovs-vsctl del-port $BRIDGE $tun
done

# 3. Flush OpenFlow rule (safer than implicit cascade)
sudo ovs-ofctl del-flows $BRIDGE

# 4. Remove QoS binding
for port in $(sudo ovs-vsctl list-ports $BRIDGE); do
    sudo ovs-vsctl clear Port $port qos
done

# 5. Remove controller binding
sudo ovs-vsctl del-controller $BRIDGE

# 6. Remove remaining port
for port in $(sudo ovs-vsctl list-ports $BRIDGE); do
    sudo ovs-vsctl del-port $BRIDGE $port
done

# 7. Remove bridge
sudo ovs-vsctl del-br $BRIDGE

# 8. Garbage collect orphan QoS + Queue (OVS không auto-GC)
for qos_uuid in $(sudo ovs-vsctl --bare --columns=_uuid list QoS); do
    refs=$(sudo ovs-vsctl --bare --columns=qos find Port qos=$qos_uuid | wc -l)
    if [ $refs -eq 0 ]; then
        sudo ovs-vsctl destroy QoS $qos_uuid
    fi
done

# 9. Kiểm chứng clean
sudo ovs-vsctl show | grep -q $BRIDGE && echo "WARN: residue" || echo "OK $BRIDGE removed"
```

## 20.4.16 Capstone POE — "Migrate br-int kernel → DPDK trên chassis live: safe?"

**Kịch bản:** Manager đề xuất migrate bridge `br-int` trên hypervisor production từ kernel datapath sang DPDK userspace để improve throughput. Live migration (không downtime VM). Safe không?

**Predict:**
1. DPDK yêu cầu hugepage + CPU pinning + NIC PMD driver.
2. Kernel br-int có tap (virtio-net) → DPDK br-int cần vhost-user (virtio-user).
3. Migration atomic hay không?

**Observe (reasoning):**

**1.** Hugepage setup requirement:

```bash
$ grep Huge /proc/meminfo
HugePages_Total:     0
HugePages_Free:      0
```

DPDK yêu cầu hugepage allocated trước runtime. Không thể runtime allocate without reboot hoặc dedicated pool.

**2.** Tap vs vhost-user incompatibility:

Kernel br-int có port `tap-vm01` (tap device). VM virtio-net backend là QEMU `/dev/net/tun`.

DPDK br-int cần `dpdkvhostuser` port. VM backend phải là QEMU vhost-user socket, khác tap.

→ **VM phải restart** để switch backend (không live migrate được).

**3.** Flow copy không đủ:

```bash
# Copy flow từ kernel bridge → DPDK bridge
$ sudo ovs-ofctl dump-flows br-int > /tmp/flows.txt
$ sudo ovs-ofctl add-flows br-int-dpdk /tmp/flows.txt
```

Flow copy OK, nhưng action reference ofport khác. `tap-vm01` ofport 3 trên br-int, `vhost-user-1` ofport 3 trên br-int-dpdk — cùng số 3 nhưng backend khác.

**4.** Datapath-type không hot-switchable:

```bash
# Thử change datapath_type runtime
$ sudo ovs-vsctl set Bridge br-int datapath_type=netdev
# Requires full bridge tear-down + re-create
```

**Explain:** Đề xuất **NOT safe** cho live migration. Correct procedure:

**Option A — Parallel bridge (zero-downtime VM):**

1. Create br-int-dpdk song song (không ảnh hưởng br-int).
2. Setup DPDK infrastructure (hugepage + CPU pinning).
3. VM-by-VM migrate: stop VM → switch backend → restart VM trên br-int-dpdk.
4. Khi all VM migrated → delete br-int cũ.

**Option B — Full downtime (simpler):**

1. Scheduled maintenance window.
2. Migrate all VM off chassis (chassis cordon, Part 20.3 §20.3.10.3).
3. Stop ovs-vswitchd → tear down br-int → reconfigure DPDK.
4. Re-create br-int với `datapath_type=netdev`.
5. VM migrate back.

**Finish:** Datapath migration là **architectural change**, không phải runtime config. VM backend phải compatible. Always parallel bridge (A) hoặc maintenance window (B). Never hot-switch.

## 20.4.17 Hiểu sai phổ biến

> **Hiểu sai 1:** "`ovs-vsctl` và `ovs-ofctl` interchangeable."
> **Đúng:** `ovs-vsctl` = OVSDB config layer (bridge/port/QoS/mirror). `ovs-ofctl` = OpenFlow control layer (flow rule CRUD). Khác layer, khác schema, khác CLI syntax.

> **Hiểu sai 2:** "`ovs-dpctl dump-flows` và `ovs-ofctl dump-flows` same output."
> **Đúng:** `ovs-dpctl` = kernel datapath megaflow (what actually executed). `ovs-ofctl` = OpenFlow rule (what operator installed). Ratio typically 1 OpenFlow → 3-5 datapath flow (expansion via wildcarding).

> **Hiểu sai 3:** "`fail-mode=secure` an toàn hơn `standalone`, luôn dùng secure."
> **Đúng:** `secure` drop new flow khi controller disconnect → outage 100% nếu controller HA không tốt. `standalone` = hub mode (ít secure nhưng traffic continue). Choice phụ thuộc controller HA + security posture.

> **Hiểu sai 4:** "`key=flow` trong tunnel option = tunnel_key empty."
> **Đúng:** `key=flow` = tunnel_key dynamic (OpenFlow rule set per-packet qua `set_tunnel:`). Opposite là `key=N` static. Không phải empty.

> **Hiểu sai 5:** "QoS config xong là có tác dụng ngay."
> **Đúng:** QoS binding tới Port, nhưng Queue ID phải match trong OpenFlow action `set_queue:<id>`. Thiếu rule set_queue → traffic fall vào default queue (không shaped).

> **Hiểu sai 6:** "`emer-reset` chỉ clear bridge hiện tại."
> **Đúng:** Clear toàn bộ config (bridge + port + controller + QoS + ...). Là nuclear option, không undo.

> **Hiểu sai 7:** "Upgrade OVS tự migrate config."
> **Đúng:** OVS upgrade preserve `conf.db` schema compat. Nhưng nếu downgrade hoặc schema version mismatch → cần `ovsdb-tool convert`. Test 1 chassis trước khi roll.

> **Hiểu sai 8:** "DPDK performance luôn tốt hơn kernel."
> **Đúng:** DPDK yêu cầu hugepage + CPU pinning + PMD thread. Nếu infra không tuned, DPDK có thể chậm hơn kernel. Benchmark thực tế trước khi migrate (§20.4.16 capstone).

## 20.4.18 Điểm cốt lõi cần nhớ

Thứ nhất, **4 CLI layer tách biệt rõ**: `ovs-vsctl` (OVSDB config) / `ovs-ofctl` (OpenFlow rule) / `ovs-dpctl` (datapath) / `ovs-appctl` (runtime RPC). Biết layer nào cho task nào = operator competent.

Thứ hai, **10 task category daily** cover 95% OVS workflow: health/inventory/bridge-port/flow/tunnel/QoS-mirror/conntrack/perf/OVSDB/backup. Nắm cả 10 = operator thành thạo.

Thứ ba, **health check 5 lệnh < 10 giây** (morning routine): ovs-vsctl show + systemctl status + ovs-dpctl show + upcall/show + dump-aggregate flow count. Automate qua `ovs-health-check.sh`.

Thứ tư, **tunnel type có trade-off**: Geneve (OVN native, 58B overhead, full TLV extensibility) / VXLAN (interop cao, 50B, legacy-friendly) / GRE (simple, không encryption, lab-use). Choose phù hợp workload.

Thứ năm, **QoS cần 3 component phối hợp**: Port.qos binding + Queue definition + OpenFlow `set_queue` action. Thiếu 1 component → QoS không effective.

Thứ sáu, **`replace-flows` atomic** — dùng thay vì del+add sequential. Đặc biệt quan trọng cho hot-reload flow table production.

Thứ bảy, **OVSDB compact định kỳ** (1 lần/tuần). Transaction log chỉ append, không shrink. Compact giữ conf.db size nhỏ.

Thứ tám, **DPDK migration không live** — VM backend phải switch (tap → vhost-user), cần restart VM hoặc parallel bridge.

Thứ chín, **`fail-mode` choice**: secure (với controller HA cluster) vs standalone (edge/branch không controller). Không universal answer.

Thứ mười, **end-to-end script** (new-bridge.sh + bridge-decommission.sh + ovs-health-check.sh + rolling-upgrade.sh) là deliverable quý. Checklist thành script, script thành CI, CI thành SLA.

## 14.2.0 Opening drama: from "OpenFlow specification-driven" to "P4Runtime schema-driven"

Between 2015 and 2017, when P4 emerged together with the Tofino ASIC, a question came up: **how does an SDN controller manage a switch that runs a custom P4 pipeline?** OpenFlow was not enough. OpenFlow assumes a fixed set of match fields and actions defined by the specification (12 fields in 1.0, 40 in 1.3). A Tofino switch can run a P4 program with 100 custom fields (for example a GTP header for 5G, or a custom NSH for service chaining), and OpenFlow has no way to describe that.

The community proposed three solutions between 2016 and 2018:

1. **Extend OpenFlow.** ONF added vendor extensions for new fields. This did not scale. Each vendor added its own extensions, and interoperability broke.
2. **Vendor-specific protocols.** Barefoot first used **PI (Programmable Interface)**, a private gRPC API for Tofino. PI solved Tofino, but every other switch needed its own API.
3. **Standardize as P4Runtime.** In 2018 the P4 Language Consortium released **P4Runtime v1.0**, distilled from Barefoot PI with contributions from Google, ONF, and Cisco. Protobuf for the message format, gRPC for transport. P4Runtime became the industry standard for P4 southbound.

P4Runtime broke through in 2018 when **ONOS released the Pipeconfs framework** that integrated P4Runtime, and **ONF released Stratum**, a thin switch agent that exposes P4Runtime, gNMI, and gNOI on Tofino and BMv2. Combining ONOS, Stratum, and Tofino became the **whitebox SDN reference stack** for carrier networking. AT&T deployed it through CORD, SEBA, and Trellis.

The ecosystem then took a detour. ONF dissolved most of its organisation at the end of 2022, and the projects moved to **Linux Foundation Networking**. Intel announced the end of life of Tofino in 2023. The competing **SONiC** (open-sourced by Microsoft in 2016) rose quickly with hyperscale adoption and merchant silicon support (Broadcom Tomahawk). The 2024 to 2026 question is: **is P4Runtime still relevant?** Short answer: **yes** in the DPU segment (AMD Pensando, parts of NVIDIA BlueField), in research, and on BMv2 for software P4. **No** in the hyperscale datacenter fabric segment, where SONiC plus Broadcom dominates.

This Part 14.2 dives into the P4Runtime API, the gNMI protocol, the Stratum reference agent, and ONOS integration. It closes with a hands-on exercise using a Python client and a capstone Predict-Observe-Explain on the strategic choice between P4Runtime and SONiC after 2023.

> **Common misconception:** *"P4Runtime is a one-to-one replacement for OpenFlow."*
>
> **Reality:** P4Runtime and OpenFlow differ at the core. OpenFlow has a match-field set **fixed by the specification version**. P4Runtime has a match-field set **defined by the P4 program** (schema-based). One controller can use OpenFlow for fixed-function switches (Broadcom plus an OpenFlow agent) and P4Runtime for programmable switches (Tofino plus Stratum) in the same network at the same time. The two do not exclude each other.

## 14.2.1 P4Runtime protocol architecture

P4Runtime is a gRPC-based API published by the P4 Language Consortium. The first release was v1.0 in 2018, and the current release is v1.3 (2021). The goal is to provide a common standard for a controller to interact with a P4-programmable switch, replacing OpenFlow (fixed pipeline) and proprietary vendor APIs. P4Runtime uses Protocol Buffers for the wire format (efficient and schema-checked) and gRPC as the transport layer (HTTP/2 with optional TLS). The standard port is 9559 for plaintext and 9559 with TLS for production.

The P4Runtime API surface has four RPC groups. (1) **SetForwardingPipelineConfig** loads a P4 program (as `device_config` plus `p4info`) into the switch, replacing or modifying the running pipeline. (2) **Write** changes table entries, action profiles, counters, meters, registers, and multicast groups (insert, modify, delete). (3) **Read** queries state, returning a streaming reply for table contents, counter values, and register values. (4) **StreamChannel** is a bidirectional stream used for master arbitration (controller election in a high-availability cluster), packet-in from the switch to the controller, packet-out from the controller to the switch, digest notifications for event-driven telemetry, and error reporting.

**Master arbitration** is an important concept. When several controllers connect to the same switch, P4Runtime uses `election_id` (an integer chosen by the controller) to elect a master. Only the master can Write. The other controllers (standby) can only Read. Master arbitration runs inside StreamChannel. Each controller sends a `MasterArbitrationUpdate` when it connects, and the switch tells each controller which one is the current master. If the master dies, the standby with the highest `election_id` automatically becomes the new master. This mechanism enables high-availability controller clusters (the ONOS model and the Kubernetes controller model).

TLS authentication is optional but recommended for production. Mutual TLS exchanges certificates: the switch verifies the controller certificate, and the controller verifies the switch certificate. In labs and development, plaintext gRPC is common to avoid configuration overhead. The P4Runtime specification does not mandate a particular authentication mechanism. Vendors implement their own (for example, Stratum supports mTLS, and some implementations use gRPC token metadata).

## 14.2.2 Schema-based API

The fundamental difference between P4Runtime and OpenFlow: OpenFlow is **specification-based** (every table, match, and action is defined in the 1.0, 1.3, or 1.5 specification), while P4Runtime is **schema-based** (the table, match, and action set is defined by the P4 program and encoded in a **P4Info** file). P4Info is Protocol Buffer metadata that describes the structure of a P4 program: the list of tables with key fields, match types, actions, and size; the list of actions with parameters; the list of registers, counters, and meters with index type and bit width; and the list of digest and packet-in headers.

The schema-based workflow: (1) The programmer compiles the P4 source. `p4c` produces `program.json` (the BMv2 configuration) and `program.p4info.txt` (the schema). (2) The controller calls `SetForwardingPipelineConfig(device_config=program.json, p4info=program.p4info.txt)` to load the program into the switch. (3) The controller parses `p4info.txt` to learn which `table_id` corresponds to which table name, and which `action_id` corresponds to which action name. (4) When installing a table entry, the controller uses the table ID, action ID, match values, and action parameters. All of these match the schema from P4Info. (5) The switch validates the request against P4Info. If a field bit width or an action parameter type does not match, the switch rejects the entry with an error.

The deeper meaning of schema-based design: the controller does not know in advance which P4 program runs on the switch. The same controller can manage a switch that runs an L2-only pipeline (`simple.p4`) and a switch that runs a P4 segment-routing pipeline (`srv6.p4`). It only needs the P4Info to learn the structure and to build requests. This is the foundation of **ONOS Pipeconfs**: ONOS can dynamically load a P4 program plus its driver binding without rebuilding the controller when the switch programmability changes.

Backward compatibility across program versions: when the P4 program on the switch is upgraded (for example, by adding a new table or extending a match key), the controller has to reload P4Info. P4Runtime supports a **cookie** and a **config_name** in `ForwardingPipelineConfig` to track the version. The controller can refuse a Write if the cookie does not match the expected version, which avoids races with stale Write or Read requests.

> **Common misconception:** *"P4Info is binary like the BMv2 JSON output of `p4c`. It can be cached."*
>
> **Reality:** P4Info is in Protobuf **text format** (extension `.p4info.txt`) so it is human-readable and version-control friendly. The controller parses P4Info into memory but does not persist a cache (the P4 program may change at deploy time). The correct pattern: every time `SetForwardingPipelineConfig` is called, the controller reloads P4Info into memory and sends it to the switch. The cookie and `config_name` are for verifying the version match, not for caching.

## 14.2.3 gNMI for operational state

P4Runtime only manages **pipeline programmability** and **forwarding state** (table entries, counters, registers). Other aspects of the switch, such as port up and down events, MTU configuration, LLDP, LACP bonding, SNMP traps, and platform sensors (fan, power, temperature), are out of scope. This is where **gNMI, the gRPC Network Management Interface**, plays a complementary role.

gNMI is a network management protocol developed by Google, open-sourced in 2017, and hosted by the **OpenConfig consortium**. The protocol uses gRPC plus Protocol Buffers and defines three RPC types. (1) **Get** retrieves configuration or operational state by path (for example, `/interfaces/interface[name=eth0]/state/counters`). (2) **Set** changes configuration (update, replace, delete). (3) **Subscribe** is streaming telemetry: the switch pushes data to the controller at sample intervals or on change. The standard gNMI port is 9339.

Like P4Runtime, gNMI is also schema-based, but the schema here is the **OpenConfig YANG model**. YANG (RFC 7950) is a data modelling language defined by the NETCONF working group. OpenConfig is a vendor-neutral consortium that publishes YANG models for several network domains (interfaces, BGP, MPLS, routing policy, telemetry). The controller parses the YANG schema, learns the valid paths for the switch, and builds gNMI requests. The switch validates the request against YANG and either applies it or rejects it.

Streaming telemetry is the strongest point of gNMI versus traditional SNMP polling. Instead of polling every 60 seconds, the switch pushes data every 1 second (or on change). The benefits are: bandwidth efficiency (Protocol Buffers are compact), low latency, and good scalability (subscribe to thousands of paths at the same time). This is the foundation of the **streaming analytics pipeline**: data flows from switch to collector (Telegraf, gNMIc) to time-series database (InfluxDB, Prometheus) to dashboard (Grafana). Google, Comcast, and several Tier-1 ISPs have replaced SNMP with gNMI streaming telemetry.

In the P4 switch ecosystem, gNMI usually accompanies P4Runtime as a pair: P4Runtime for data-plane programmability, gNMI for platform management. Some implementations also add **gNOI** (gRPC Network Operations Interface) for additional operations such as file transfer, reboot, and certificate management.

## 14.2.4 ONOS plus Stratum integration

**Stratum** is a thin switch operating system led by the Open Networking Foundation (ONF) since 2018. Stratum runs on a P4-programmable switch (Tofino, BMv2, and a few software targets) and provides three standard APIs: P4Runtime (data plane), gNMI (configuration plus telemetry), and gNOI (operational tasks). Stratum is not a full network operating system like ONIE plus SONiC. It is just a **switch agent** that runs on the target. It does not have a user-facing CLI or a routing-protocol engine.

Stratum architecture: (1) The **switch stack** is the driver that talks to the SDE (the Barefoot SDE for Tofino, the BMv2 library for BMv2). (2) The **P4Runtime service** implements the P4Runtime specification and handles Write, Read, and StreamChannel. (3) The **gNMI service** implements the gNMI specification and handles Get, Set, and Subscribe. (4) The **Hardware Abstraction Layer (HAL)** unifies the API across vendors. (5) The **Chassis manager** handles ports, sensors, and platform state. Stratum is open source at `github.com/stratum/stratum`, mainly written in C++.

**ONOS** (Open Network Operating System) is an SDN controller that started at ON.Lab and merged into ONF in 2018. ONOS has a P4Runtime driver to talk to Stratum, so ONOS can manage a cluster of P4-programmable switches. The layered architecture: (1) A **Pipeconf** is a bundle (the P4 program, the P4Info, and a Java driver) registered with ONOS. It tells ONOS which pipeline a switch runs. (2) The **Pipeline Interpreter** is a Java class that translates high-level ONOS configuration intent (for example, "forward L3 from A to B") into low-level P4Runtime Write requests. (3) The **Device Driver** handles gNMI configuration and P4Runtime data plane for each switch model.

**AT&T CORD** (Central Office Re-architected as Datacenter) is the most important production deployment of Stratum plus ONOS. CORD turns a telecom central office into a mini-datacenter: whitebox switches run Stratum (instead of Cisco or Juniper L2/L3 boxes), ONOS controls forwarding, and OpenStack manages the VNFs. AT&T deployed CORD between 2016 and 2020 in hundreds of central offices for xDSL and fibre broadband access. AT&T moved to a disaggregated cell site (DCSG) architecture in 2022, but the CORD legacy lives on through **SD-Core** (SD-Fabric plus mobile core) inside ONF, and through the edge platform **Aether** that ONF handed to Linux Foundation Networking in 2023.

> **Note on the 2024 to 2026 ecosystem state:** ONF dissolved most of its organisation at the end of 2022. The projects moved to Linux Foundation Networking (LFN). Stratum and ONOS are still maintained, but the community has shrunk. The rising replacement is **SONiC** (open-sourced by Microsoft in 2016, now hosted by LFN), combined with the **Stratum-Broadcom SDKLT integration** or running directly with the Broadcom SDK.

## 14.2.5 Comparison: P4Runtime plus gNMI versus OpenFlow plus NETCONF versus SONiC

The three southbound stacks common for the network control plane in 2024 to 2026:

| Dimension | OpenFlow plus NETCONF (classic SDN) | P4Runtime plus gNMI (Stratum stack) | SONiC (hyperscale) |
|-----------|----------------------------------|----------------------------------|---------------------|
| Data-plane API | OpenFlow (fixed match fields) | P4Runtime (schema-based, flexible) | Redis SAI plus native switch SDK |
| Configuration and state API | NETCONF and YANG (SSH transport) | gNMI and gNOI (gRPC transport) | Redis ConfigDB plus CounterDB |
| Telemetry | SNMP polling or OpenFlow counters | gNMI Subscribe streaming | Redis subscribe plus sFlow |
| Vendor independence | Good (standard specification) | Good (standard specification plus schema flexibility) | Good (SAI abstraction) |
| Pipeline programmability | Limited (fixed match fields) | Full (any P4 program) | Limited (fixed SAI API) |
| Controller ecosystem | ODL, ONOS, Ryu, Faucet | ONOS Pipeconfs, Stratum, SD-Fabric | SONiC CLI plus Redis plus Ansible |
| Production deployment | Wide (OVS plus ODL or ONOS) | Limited after the Tofino end of life | Wide (Microsoft Azure, Alibaba, Comcast) |
| Hardware support in 2026 | Wide (Broadcom plus Mellanox plus Intel) | Limited (Pensando DPU, legacy Tofino) | Wide (Broadcom plus Intel plus Mellanox) |

**When to use which:**

- **OpenFlow plus NETCONF.** Brownfield SDN (an existing ODL or ONOS deployment), hypervisor workload (OVS), SD-WAN customer-premises equipment.
- **P4Runtime plus gNMI.** Greenfield with a programmable data plane (custom INT telemetry, stateful load balancing), a research network, or a DPU offload (Pensando).
- **SONiC.** Hyperscale datacenter fabric (Azure-scale), multi-vendor merchant silicon (Broadcom Tomahawk plus Mellanox Spectrum), or operations-heavy environment.

**Strategic note:** SONiC has become the de-facto standard for hyperscale after 2022. Microsoft deploys SONiC across the entire Azure network (about 1 million switches), and Alibaba, Comcast, and Tencent follow. P4Runtime and Stratum fit niche use cases (custom pipeline) and do not compete directly with SONiC at scale.

### Comparison with OVS plus OVN plus OpenStack

In an **OpenStack ML2 and OVN** deployment (hypervisor workload), the southbound stack is completely different:

- **Control plane:** OVN northd (NB to SB DB), `ovn-controller` (SB to OpenFlow per hypervisor).
- **Southbound:** OVN SB (OVSDB protocol), `ovn-controller` to OVS (OpenFlow 1.3).
- **No P4Runtime or gNMI** because OVS is not P4-programmable.

Question: why does OpenStack not use P4Runtime? Short answer: **OVS is enough for hypervisor workloads**. VM and pod workloads run at most 25 to 100 Gbps per hypervisor. OVS conntrack plus Geneve tunnels handle that well. There is no need for the 6.5 Tbps line rate of Tofino. The hypervisor does not need custom L3 or L7 header parsing (OVN plus conntrack covers 95 percent of cases). P4 is overkill at this layer.

## 14.2.6 Key takeaways

**Six core points:**

1. **P4Runtime is a gRPC and Protobuf API** for the southbound control plane of a P4-programmable switch. The four RPC groups: SetForwardingPipelineConfig, Write, Read, StreamChannel.
2. **Schema-based versus specification-based.** The P4Info file (Protobuf text) is the schema. The controller has to load P4Info before any table operation. This is different from OpenFlow, where the specification fixes the schema.
3. **gNMI is the device-management protocol that complements P4Runtime.** Get, Set, Subscribe over the OpenConfig YANG model. Streaming telemetry replaces SNMP polling.
4. **Stratum is a thin switch agent**, not a full network operating system. It exposes P4Runtime, gNMI, and gNOI. It runs on Tofino and BMv2. SONiC is a full network operating system, which is different.
5. **Master arbitration in StreamChannel** uses `election_id` to support a high-availability controller cluster. Only the master can Write; standby controllers are read-only.
6. **The ONF ecosystem restructure of 2022.** Stratum and ONOS moved to Linux Foundation Networking. The community has shrunk. SONiC took the hyperscale segment. P4Runtime survives on DPUs, in research, and on BMv2.

### Capstone POE: Greenfield datacenter 2026, which southbound stack to choose?

> **Lab verification status:** This POE is based on publicly documented deployments and vendor roadmaps from 2024 to 2026. Wait one to two more years for the ecosystem to settle before treating the data as definitive.

**Goal:** Evaluate the three common southbound stacks of 2026 and choose the right one for a greenfield deployment.

**Scenario:** Organisation Z builds a new datacenter (50 racks times 48 servers times 2 NICs at 100 GbE). Network layer: 10 spine switches plus 50 leaf switches (Broadcom Tomahawk 4 merchant silicon). Workload mix: 60 percent Kubernetes, 30 percent bare-metal HPC, 10 percent VMs. There is no legacy P4 investment. The CTO needs to choose a southbound stack.

**Three options:**

1. **OpenFlow plus OVS plus OVN.** An ODL or ONOS controller drives OVS on each hypervisor, and every leaf runs an OpenFlow agent. This is the traditional SDN stack.
2. **P4Runtime plus Stratum plus ONOS.** Requires Tofino switches (place a legacy stock order before end of life), Stratum agents, and an ONOS controller.
3. **SONiC plus SAI plus Redis.** Multi-vendor merchant silicon support. Redis-based configuration and state databases. Ansible plus Salt for operational automation.

**Predict:** Evaluate the three options against five criteria: capability, deployment cost, operational maturity, vendor-lock-in risk, long-term viability (2030 and beyond).

**Observe (reasoning plus data check):**

**Option 1 (OpenFlow plus OVS and OVN):**
- Capability: Excellent for hypervisor and VM workloads. Good for Kubernetes (OVN-Kubernetes). Limited for HPC (OVS overhead).
- Deployment cost: Medium. ONOS and ODL setup is complex. OVS is mature.
- Operational maturity: Good. OVS plus OVN deployment patterns are well known.
- Vendor lock-in: Low. Open-source stack.
- Long term: Good. OVS and OVN are under active development. OpenStack continues.

**Option 2 (P4Runtime plus Stratum):**
- Capability: Excellent if a custom pipeline is needed (INT, stateful load balancing). Overkill for standard forwarding.
- Deployment cost: High. Tofino is at end of life, so legacy stock has to be bought. Stratum container plus ONOS setup is complex. The team needs P4 skills.
- Operational maturity: Poor for greenfield. Limited production examples after 2022. The community has shrunk after the ONF restructure.
- Vendor lock-in: High (Tofino hardware end of life). The Pensando alternative is not yet mature for a fabric-wide deployment.
- Long term: High risk. Five to seven years from now, hardware may not be available.

**Option 3 (SONiC):**
- Capability: Excellent for standard forwarding plus multi-vendor support. Limited pipeline programmability.
- Deployment cost: Medium. SONiC has a learning curve, but the Microsoft and Celestica ecosystem helps.
- Operational maturity: Excellent. Azure-scale deployment proof. Redis plus Ansible integration is mature.
- Vendor lock-in: Low. SAI abstraction across Broadcom, Mellanox, and Intel.
- Long term: Excellent. Linux Foundation hosting and heavy Microsoft investment.

**Explain:** Recommendation by scenario.

**For Organisation Z (greenfield, 60 percent Kubernetes, 30 percent HPC, 10 percent VMs):**

- **Underlay (leaf-spine fabric):** **SONiC** (option 3). Broadcom Tomahawk merchant silicon plus SONiC for a standard BGP plus EVPN plus VXLAN underlay. Operational maturity, vendor flexibility, and a clear long-term roadmap.
- **Overlay (hypervisor plus Kubernetes):** **OVS plus OVN** (option 1). The hypervisor runs OVS, and OVN provides Neutron integration. Kubernetes uses OVN-Kubernetes or Cilium (depending on preference).
- **Do not choose P4Runtime and Stratum** as the default. Only consider this option if Organisation Z has a specific use case that needs a custom pipeline (for example, HPC needs RDMA telemetry through INT). If so, deploy Tofino or Pensando in a limited way at the spine layer, not fabric-wide.

**Rationale:** A greenfield deployment has no P4 legacy. There is no need to pay the risk cost of the Tofino end of life. SONiC plus OVS and OVN cover 100 percent of the workload with a mature ecosystem. P4Runtime in 2026 is a specialised tool, not a general-purpose one.

**Finish:** A universal principle is **"choose the stack along the maturity curve of the workload"**. A greenfield deployment is not the place to take a risk on bleeding-edge technology, because there is no legacy migration urgency. The risk is having to rebuild the entire deployment if the vendor discontinues the product (the Tofino 2023 case study). SONiC plus OVS is the mature, stable choice. P4 is a specialist tool, used when the specific problem demands it.

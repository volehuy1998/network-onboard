## 7.4.1 Why Faucet uses a fixed pipeline

Part 7.0 explained that Faucet achieves operational stability by narrowing scope. One of the most important narrowing decisions is the fixed pipeline: instead of letting each deployment design its own number and order of OpenFlow tables, Faucet defines a canonical pipeline and every switch runs that same pipeline.

The technical reason behind the decision relates to the hardware chipset. The chipsets common in campus switching (Broadcom Trident, Mellanox Spectrum) are designed to be optimal for a fixed number of tables in a fixed order. When the pipeline matches the hardware design, the entire match-and-action sequence runs at line rate inside the ASIC. When the pipeline does not match, the switch falls back to CPU processing. Real bandwidth then drops from tens of gigabits to megabits. The Faucet canonical pipeline was tested specifically for Broadcom Trident2 and Trident3, and for Mellanox Spectrum. These were the two most common chipsets in campus switching when Faucet was created.

> **Key point:** The fixed pipeline is a trade-off between flexibility and efficiency. The operator loses the right to customize the pipeline, but in return gains the guarantee that the pipeline has been optimized for common hardware. This is why the same Faucet configuration runs correctly on hardware switches (Aruba, Allied Telesis, Edgecore) and on OVS. Both implement the same set of OpenFlow operations the canonical pipeline requires.

## 7.4.2 The four core tables in the Faucet pipeline

The Faucet pipeline contains four base tables that are always present, plus optional tables that appear when IP routing is enabled:

**Table VLAN** (typically table 0): The packet arrives from a physical port. This table determines which VLAN the packet belongs to, based on (a) the VLAN tag inside the frame (a tagged port) or (b) port membership (an untagged port maps to the native VLAN). A packet that does not match any rule in this table is dropped. Output: the VLAN ID is written to the `vlan_vid` metadata, and the packet moves to the next table.

**Table ETH_SRC** (MAC learning): every packet passes through this table so that the source address is learned. If the source MAC is not yet in the forwarding table, Faucet sends a `packet_in` to the controller to learn it and install a new entry. If the MAC is already known but the packet arrived on a different port (a host has moved port), the entry is updated. Output: the packet moves to ETH_DST.

**Table ETH_DST** (destination lookup): look up the destination MAC in the forwarding table. If known, the packet is sent out the corresponding port. If unknown (an unknown unicast) or if it is a broadcast or multicast frame, the packet moves to the FLOOD table.

**Table FLOOD**: handles broadcast, multicast, and unknown unicast by sending the packet out every port in the same VLAN except the input port. Faucet computes the flood group once when the configuration changes, not for every packet. This is an important optimization for actual throughput compared with the naive use of the OpenFlow `FLOOD` action.

When IP routing is enabled through `routing: True` on a VLAN, Faucet adds two tables between ETH_SRC and ETH_DST:

- **IP_PROTO**: classifies IP and ARP packets by `eth_type` and `ip_proto`.
- **IPV4_FIB** or **IPV6_FIB**: looks up the route by longest-prefix match and rewrites the destination MAC to the MAC of the next-hop router interface.

## 7.4.3 Advanced ACLs in Faucet

ACLs in Faucet are configured in YAML under the top-level `acls:` section. Each ACL is a list of rules. Faucet evaluates the rules from top to bottom and stops at the first match (first-match semantics).

Full ACL syntax with several match fields:

```yaml
acls:
  block-guest-ssh:
    - rule:
        eth_type: 0x0800      # IPv4
        ip_proto: 6           # TCP
        tcp_dst: 22           # SSH port
        ipv4_src: 10.2.0.0/24 # Guest subnet
        actions:
          allow: 0            # 0 = drop

    - rule:
        eth_type: 0x0800
        ip_proto: 1           # ICMP
        ipv4_src: 10.2.0.0/24
        actions:
          allow: 0            # Block ICMP from guest

    - rule:
        actions:
          allow: 1            # Default: permit all remaining traffic

vlans:
  staff:
    vid: 100
    description: "Staff network"

  guest:
    vid: 200
    description: "Guest network"
    acls_in: [block-guest-ssh]  # Apply the ACL to ingress traffic
```

Supported match fields: `eth_type`, `eth_src`, `eth_dst`, `ip_proto`, `ipv4_src`, `ipv4_dst`, `tcp_src`, `tcp_dst`, `udp_src`, `udp_dst`, `icmpv4_type`, `icmpv4_code`, `vlan_vid`. This is a subset of OXM in OpenFlow 1.3 and is enough for most campus policies.

> **Common misconception:** "Faucet ACLs work like a stateful firewall and automatically allow reply traffic." In reality, Faucet ACLs are stateless. Each packet is evaluated independently, with no connection tracking. If a rule allows TCP/22 inbound but there is no matching rule for outbound replies, the TCP handshake will not complete. This is the key difference from the OVN ACL `allow-related` (Part 20.1). When you need a stateful firewall, use OVN instead of Faucet.

## 7.4.4 Gauge: monitoring with Prometheus

Gauge is a process separate from the Faucet controller. The two processes do not share state directly. Gauge connects to the same switches over the OpenFlow statistics protocol, periodically polls port counters and flow counters, and exports them to a monitoring backend.

Gauge configuration (`/etc/faucet/gauge.yaml`):

```yaml
faucet_configs:
  - /etc/faucet/faucet.yaml      # Read the topology from the Faucet config

watchers:
  port_stats:
    type: 'port_stats'
    dps: all                     # Monitor every switch
    db: prometheus
    interval: 15                 # Poll every 15 seconds

  flow_table:
    type: 'flow_table'
    dps: all
    db: prometheus
    interval: 30

dbs:
  prometheus:
    type: 'prometheus'
    port: 9303                   # Gauge exposes /metrics on this port
```

Important metrics that Gauge exports to Prometheus:

| Metric | Type | Meaning |
|--------|------|---------|
| `faucet_of_dp_disconnected` | Gauge | 1 if the switch is disconnected. Alert immediately. |
| `faucet_port_transmitted_packets` | Counter | Packets transmitted on the port |
| `faucet_port_received_packets` | Counter | Packets received on the port |
| `faucet_port_receive_drops` | Counter | Packets dropped at the ingress port |
| `faucet_port_transmit_drops` | Counter | Packets dropped at the egress port |
| `faucet_flow_packet_count` | Counter | Packets matching a specific flow entry |
| `faucet_flow_byte_count` | Counter | Bytes through a flow entry |

Example PromQL alert rules for Prometheus:

```promql
# Alert when the switch disconnects
faucet_of_dp_disconnected == 1

# Alert when the ingress drop rate spikes (more than 500 drops per minute)
rate(faucet_port_receive_drops[5m]) * 60 > 500

# Top ports by traffic (useful for capacity planning)
topk(5, rate(faucet_port_transmitted_packets[5m]))
```

### Guided Exercise 1: deploy Faucet with an ACL and Gauge monitoring

In this exercise you configure Faucet with OVS as the datapath, add an ACL that blocks SSH from the guest VLAN, and run Gauge to verify that the metrics endpoint is working.

**Outcomes:**
- Write a `faucet.yaml` with two VLANs and an ingress ACL that works correctly.
- Verify that Faucet installed OpenFlow rules corresponding to the YAML configuration.
- Start Gauge and read its metrics with `curl`.

**Before you begin:**
OVS installed and running (`ovs-vsctl show` does not error). Faucet and Gauge installed via `pip install faucet`. Verify with `faucet --version`.

**Instructions:**

**1.** Create the configuration directory and a minimal `faucet.yaml`:

```bash
[host]# mkdir -p /etc/faucet
```

```yaml
# /etc/faucet/faucet.yaml
vlans:
  staff:
    vid: 100
  guest:
    vid: 200
    acls_in: [block-guest-ssh]

acls:
  block-guest-ssh:
    - rule:
        eth_type: 0x0800
        ip_proto: 6
        tcp_dst: 22
        actions:
          allow: 0
    - rule:
        actions:
          allow: 1

dps:
  ovs-lab:
    dp_id: 0x1
    hardware: "Open vSwitch"
    interfaces:
      1:
        name: "staff-port"
        native_vlan: staff
      2:
        name: "guest-port"
        native_vlan: guest
```

**2.** Create the OVS bridge and point its controller at Faucet:

```bash
[host]# ovs-vsctl add-br br-faucet
[host]# ovs-vsctl set bridge br-faucet protocols=OpenFlow13
[host]# ovs-vsctl set bridge br-faucet \
        other-config:datapath-id=0000000000000001
[host]# ovs-vsctl set-controller br-faucet tcp:127.0.0.1:6653
```

**3.** Start Faucet (in the background):

```bash
[host]# faucet --verbose 2>&1 | tee /tmp/faucet.log &
```

**4.** Wait about two seconds, then check the installed OpenFlow rules:

```bash
[host]# ovs-ofctl -O OpenFlow13 dump-flows br-faucet
```

Look for a rule with the match `dl_vlan=200,tp_dst=22` and action `actions=drop`. This confirms the ACL has been translated into an OpenFlow rule.

**5.** Create `gauge.yaml` and start Gauge:

```bash
[host]# faucet --gauge --gauge-config /etc/faucet/gauge.yaml &
[host]# curl -s http://localhost:9303/metrics | grep faucet_of_dp
```

Expected result: the metric `faucet_of_dp_disconnected` appears with value `0` (the switch is connected).

**Finish:**

```bash
[host]# pkill -f faucet
[host]# ovs-vsctl del-br br-faucet
```

## Key points to remember

First, the fixed pipeline in Faucet is a deliberate architectural choice, not a technical limitation. By standardizing the pipeline, Faucet guarantees optimization for common hardware chipsets, which a custom pipeline cannot guarantee without separate testing for each chipset.

Second, ACLs in Faucet are stateless. They are appropriate for simple policy (block port, allow subnet) but not for policy that needs to follow connection state. When you need stateful policy, use OVN ACLs with `allow-related` (Part 20.1) instead of Faucet.

Third, Gauge is the most important piece that lets Faucet operate in production. Without Gauge, the operator is completely blind to the traffic. Incidents are then only discovered when users report them, instead of being detected early through Prometheus alerts.

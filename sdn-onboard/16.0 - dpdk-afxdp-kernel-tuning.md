## 16.0.0 Drama mở đầu, từ "kernel stack đủ rồi" đến "phải bypass mới đạt Tbps"

Thập niên 2000s, kernel Linux network stack được xem là đủ cho workload enterprise. Với NIC 1 GbE phổ biến, kernel NAPI + NIC driver process ~148 Kpps (64-byte wire-speed) dễ dàng. Đến 2010s, xuất hiện NIC 10 GbE (14.88 Mpps wire-speed), kernel bắt đầu struggle ở tải cao: context switch overhead, interrupt coalescing, socket buffer copy, mỗi gói tin mất ~1-10 µs. Với workload như High-Frequency Trading (HFT) cần độ trễ <5 µs, kernel không còn đủ.

Năm 2010, **Intel Dataplane Performance Development Kit (DPDK)** ra đời như một set of libraries + PMD driver cho phép ứng dụng userspace thăm dò NIC trực tiếp, bỏ qua kernel network stack hoàn toàn. Kết quả: latency giảm xuống 1-3 µs, throughput đạt 100+ Mpps per core. DPDK nhanh chóng trở thành standard cho: telecom 5G UPF, HFT trading gateway, virtual switch (OVS-DPDK), NFV (Network Function Virtualization), DDoS mitigation appliance.

Nhưng DPDK có 3 nhược điểm lớn: (a) **đốt 100% 1 CPU core per port** do poll mode, không scale tốt ở cloud shared CPU, (b) **tách NIC khỏi kernel** → mất các feature kernel (iptables, tc, netfilter, bonding, LACP), (c) **yêu cầu hugepages + IOMMU + CPU isolation** → complex deployment, không plug-and-play.

Năm 2018, **AF_XDP** (kernel 4.18) ra đời như compromise: dùng shared memory ring giữa NIC driver và userspace socket, bypass kernel socket stack **nhưng giữ lại NIC driver kernel + netfilter + tc** cho packet không match (XDP_PASS). Throughput AF_XDP ~8-12 Mpps per core (kém DPDK ~30%) nhưng dễ deploy + giữ feature kernel. Cilium, Suricata, OVS-AFXDP nhanh chóng adopt AF_XDP.

Part 16.0 này đi sâu vào 3 data path option, DPDK EAL + PMD internals, AF_XDP zero-copy architecture, kernel tuning knobs (sysctl + RSS + RFS + NAPI budget), CPU pinning + NUMA awareness, và profiling với perf + bpftrace. Kết thúc với 3 exercises hands-on (baseline vs DPDK, AF_XDP XDP filter, full tuning workflow baseline → wire-speed) + Capstone POE "hyperscale 2026: DPDK vs AF_XDP vs kernel cho Kubernetes CNI?".

> **Hiểu sai phổ biến:** *"DPDK luôn nhanh hơn kernel stack."*
>
> **Thực tế:** DPDK băng thông thực tế cao hơn kernel 5-10× chỉ khi **workload đủ tải** để justify poll mode overhead. Với burst traffic low-utilization (ví dụ web server 1000 req/s), DPDK đốt 100% core mà gói tin per second không high hơn kernel (kernel NAPI efficient khi tải thấp). Pattern đúng: benchmark cả 2 với workload thực tế trước khi invest DPDK complexity. Nhiều K8s cluster production dùng OVS kernel datapath đủ, không cần DPDK.

## 16.0.1 So sánh data path: kernel, DPDK, AF_XDP

Ba cách tiếp cận xử lý gói ở quy mô cao trên Linux x86: **kernel datapath chuẩn** (NIC → driver → kernel networking stack → socket userspace), **DPDK** (NIC được tách khỏi kernel driver, gắn vào UIO/VFIO, ứng dụng userspace thăm dò trực tiếp NIC qua Poll Mode Driver), và **AF_XDP** (NIC gắn chương trình XDP, gói tin được chuyển hướng sang ring buffer userspace mà không qua kernel networking stack đầy đủ). Ba phương pháp này khác nhau về băng thông thực tế, độ trễ, chi phí CPU, và độ phức tạp triển khai.

**Benchmark băng thông thực tế** (đo trên máy vật lý Intel Xeon Gold 6246R, NIC Intel XXV710 25GbE, gói 64-byte): kernel datapath đạt khoảng 1.5-2 Mpps per core (thoái hóa ở tải cao do overhead ngắt); OVS-DPDK đạt 10-15 Mpps per core (chế độ thăm dò); AF_XDP đạt 8-12 Mpps per core (chế độ zero-copy với driver hỗ trợ). Với NIC 100GbE và CPU 4+ core dành riêng, DPDK đạt 40-80 Mpps tổng; kernel stack đạt tối đa khoảng 20 Mpps trước khi bão hòa IRQ.

**Hồ sơ độ trễ** (gói tin round-trip trong cùng node): kernel datapath khoảng 20 microseconds (bao gồm overhead ngắt, chuyển đổi ngữ cảnh syscall); DPDK khoảng 2 microseconds (thăm dò liên tục, không có ngắt); AF_XDP khoảng 10 microseconds (vẫn dùng cơ chế ngắt kernel nhưng bỏ qua socket stack). Với workload nhạy cảm độ trễ (HFT, 5G UPF, tín hiệu thời gian thực viễn thông), DPDK là lựa chọn chuẩn; với workload ưu tiên băng thông thực tế nhưng chấp nhận độ trễ cao hơn, AF_XDP cân bằng hiệu năng và độ phức tạp.

**Mô hình chi phí CPU** là sự đánh đổi quan trọng: chế độ thăm dò của DPDK **đốt 100% của một core** per port — ngay cả khi không có traffic, core vẫn quay vòng thăm dò. Điều này không phù hợp cho môi trường cloud chia sẻ CPU (mỗi VM mất 1 core cho datapath). Kernel stack dùng mô hình điều khiển bằng ngắt: khi có gói tin, CPU thực hiện ngắt; khi rảnh, CPU đi vào C-state tiết kiệm điện. AF_XDP có chế độ thăm dò liên tục tương tự DPDK và chế độ điều khiển bằng ngắt tương tự kernel — cho phép đội ngũ chọn cân bằng giữa hiệu năng và chi phí điện năng.

## 16.0.2 DPDK EAL + PMD internals

**EAL (Environment Abstraction Layer)** là thư viện nền tảng của DPDK, khởi tạo môi trường runtime cho ứng dụng userspace: (1) quét hugepages khả dụng (2MB hoặc 1GB), cấp phát bộ nhớ liên tục cho DPDK sử dụng; (2) xác định cấu trúc NUMA, gán thread tới CPU socket theo cấu hình; (3) gắn NIC vào UIO (cũ) hoặc VFIO-PCI driver — tách NIC khỏi kernel networking; (4) khởi tạo **mempool** (bể gói tin buffer được cấp phát trước) và **mbuf** (buffer metadata cho mỗi gói tin).

**PMD (Poll Mode Driver)** là driver userspace cho NIC, thay thế driver kernel. PMD cho mỗi model NIC (Intel ixgbe, i40e, ice, mlx5 cho Mellanox, bnxt cho Broadcom) được compile vào DPDK binary. PMD không dùng ngắt — thay vào đó, thread dành riêng chạy vòng lặp vô hạn gọi `rte_eth_rx_burst()` để đọc lô gói tin từ NIC descriptor ring, xử lý, rồi gọi `rte_eth_tx_burst()` để gửi ra NIC khác. Vòng lặp này chạy liên tục trên một CPU core dành riêng (isolcpus).

**Cấp phát hugepages** là yêu cầu cứng của DPDK vì hai lý do: (1) bộ nhớ liên tục tránh TLB miss — khi xử lý hàng triệu gói tin per second, một TLB miss mỗi gói tin làm CPU đứng hàng trăm chu kỳ đồng hồ; (2) hugepages là zero-copy — DPDK ánh xạ DMA buffer của NIC trực tiếp vào hugepage, không cần copy gói tin ra buffer userspace. Cấu hình chuẩn: 2-4 GB hugepage 1GB cho traffic 40-100 Gbps, hoặc 1-2 GB hugepage 2MB cho traffic 10-25 Gbps.

**Ghim vào NUMA node** là tối ưu quan trọng: nếu NIC gắn vào PCIe socket 0, thread PMD phải chạy trên core cùng socket 0 (dùng cờ `--lcores`), và hugepages phải cấp phát từ bộ nhớ socket 0. Nếu ghim sai, mỗi gói tin đi qua liên kết QPI/UPI giữa socket, độ trễ tăng gấp đôi và băng thông thực tế giảm 30-50%.

**Vòng đời mempool:** mempool được cấp phát ở giai đoạn khởi tạo EAL với kích thước cố định (ví dụ 8K mbuf cho mỗi port). Trong runtime, mbuf được mượn từ pool khi gói tin đến, trả lại khi gói tin được xử lý xong. Nếu mempool cạn (tồn đọng gói tin cao), NIC drop gói tin mới — đây là triệu chứng phổ biến khi gỡ lỗi vấn đề hiệu năng DPDK.

## 16.0.3 AF_XDP zero-copy + XDP programs

**AF_XDP** là loại socket kernel mới ra đời kernel 4.18 (2018), cho phép ứng dụng userspace nhận gói tin trực tiếp từ NIC driver thông qua shared memory ring, bỏ qua hoàn toàn kernel networking stack nhưng vẫn tái sử dụng driver kernel (khác DPDK dùng PMD userspace riêng). AF_XDP kết hợp hiệu năng gần DPDK với tính dễ triển khai của kernel.

**Kiến trúc AF_XDP socket** gồm bốn thành phần: (1) **UMEM** — vùng bộ nhớ userspace đăng ký với kernel làm buffer cho gói tin, kernel DMA trực tiếp từ NIC vào UMEM (zero-copy với driver hỗ trợ). (2) **FILL ring** — userspace thêm frame index của UMEM vào ring này để báo cho kernel "các buffer này sẵn sàng nhận gói tin mới". (3) **RX ring** — kernel thêm frame index đã nhận gói tin vào ring này; userspace đọc ring để xử lý gói tin. (4) **TX ring + COMPLETION ring** — tương tự cho chiều ra.

**Điểm gắn chương trình XDP:** (1) **Chế độ driver/native** — chương trình eBPF chạy trong ngữ cảnh ngắt của driver NIC, TRƯỚC khi kernel cấp phát sk_buff. Đây là điểm gắn hiệu năng cao nhất (khoảng 24 Mpps per core trên Intel XXV710), nhưng chỉ driver hỗ trợ mới gắn được: ixgbe, i40e, ice, mlx5, mlx4, bnxt, tun. (2) **Chế độ SKB** (generic XDP) — chạy SAU khi kernel đã cấp phát sk_buff, hoạt động với mọi driver nhưng chậm hơn đáng kể (khoảng 3-5 Mpps). (3) **Chế độ offload phần cứng** — compile chương trình eBPF cho firmware NIC, chạy trong ASIC NIC. Hỗ trợ bởi Netronome Agilio CX và Mellanox ConnectX-5+; hạn chế tập lệnh.

**Mẫu XDP_REDIRECT** là cách chuyển gói tin từ NIC tới AF_XDP socket: chương trình XDP gắn vào NIC, gọi `bpf_redirect_map(xsk_map, queue_id, 0)` để đẩy gói tin vào AF_XDP socket userspace. Đây là cơ chế chuẩn cho Cilium (chuyển hướng pod-to-pod), OVS-AFXDP (đã được đề cập ở Phần 9.3), và suricata/Snort AF_XDP.

So với DPDK, AF_XDP có ưu điểm: (1) không cần tách NIC khỏi kernel, vẫn có cấu hình IP và table định tuyến bình thường; (2) không cần hugepages (có thể dùng page thường, tốn một chút TLB nhưng dễ triển khai hơn); (3) có thể chung sống với kernel traffic — chương trình XDP có thể chọn chuyển hướng HOẶC cho gói tin tiếp tục vào kernel stack (XDP_PASS). Nhược điểm: độ trễ cao hơn DPDK khoảng 5 microseconds do vẫn qua kernel driver.

## 16.0.4 Các nút điều chỉnh kernel

Khi không đi DPDK/AF_XDP, kernel stack vẫn có thể đẩy đến 20-30 Mpps với tinh chỉnh đúng. Các nút điều chỉnh chính:

**Sysctl `net.core.*`** cho workload tốc độ gói tin cao:
- `net.core.netdev_max_backlog = 300000` — độ sâu hàng đợi giữa driver NIC và kernel process, mặc định 1000 (quá thấp cho 10+ Gbps).
- `net.core.rmem_max = 16777216`, `wmem_max = 16777216` — kích thước socket buffer tối đa, mặc định 200K (quá thấp cho TCP băng thông thực tế cao).
- `net.core.busy_read = 50`, `busy_poll = 50` — bật thăm dò liên tục cho workload nhạy cảm độ trễ (CPU quay vòng thay vì ngắt).
- `net.core.somaxconn = 65535` — độ sâu listen backlog, mặc định 128 (quá thấp cho server đồng thời cao).

**RPS (Receive Gói tin Steering)** phân phối gói tin từ một queue đến nhiều CPU để cân bằng tải. Cấu hình: ghi bitmask vào `/sys/class/net/ethX/queues/rx-N/rps_cpus`. Quan trọng khi NIC chỉ có một hoặc ít queue nhưng CPU có nhiều core.

**RFS (Receive Flow Steering)** mở rộng RPS để định tuyến gói tin về đúng CPU nơi socket userspace được lập lịch, tối ưu cache locality. Cấu hình qua `/proc/sys/net/core/rps_sock_flow_entries`.

**Tinh chỉnh ngân sách NAPI:** NAPI (New API) là khung gộp ngắt của kernel. Mặc định mỗi chu kỳ thăm dò NAPI xử lý 64 gói tin rồi nhường lại CPU. Với workload băng thông thực tế cao, nâng `net.core.netdev_budget = 600` và `netdev_budget_usecs = 8000` để xử lý nhiều gói tin per cycle hơn.

## 16.0.5 Ghim CPU + nhận biết cấu trúc NUMA

Trong hệ thống đa socket, cấu trúc NUMA (Non-Uniform Memory Access) quyết định hiệu năng xử lý gói tin. Mỗi CPU socket có bộ nhớ cục bộ truy cập nhanh (50-100 ns) và bộ nhớ từ xa truy cập qua liên kết QPI/UPI (100-200 ns, chậm gấp đôi). NIC gắn vào khe PCIe thuộc một socket cụ thể — gói tin từ NIC đổ vào bộ nhớ socket đó.

**Công cụ khảo sát:**
- `lscpu` hiển thị NUMA node và CPU nào thuộc node nào.
- `lstopo` (từ hwloc) vẽ sơ đồ cấu trúc đầy đủ: socket, core, cache, NIC, memory channel.
- `numactl --hardware` xác nhận cách phân phối node.
- `cat /sys/class/net/ethX/device/numa_node` báo NIC thuộc NUMA node nào.

**Cấu hình `isolcpus` kernel cmdline:** thêm `isolcpus=2,3,4,5 nohz_full=2,3,4,5 rcu_nocbs=2,3,4,5` vào dòng lệnh kernel GRUB để tách 4 core khỏi scheduler mặc định. Các core này không bị workload khác chiếm dụng, dành riêng cho thread DPDK PMD hoặc AF_XDP worker.

**Ghim thread PMD của OVS:** cấu hình qua `ovs-vsctl set Open_vSwitch . other_config:pmd-cpu-mask=0x3c` (mask bit 2-5 tương ứng CPU 2-5). OVS sẽ gán thread PMD tới các core này, và các core này phải trùng với NUMA node của NIC để đạt hiệu năng tối ưu.

**Ái lực IRQ qua `/proc/irq/*/smp_affinity`:** để tránh IRQ NIC ngắt core xử lý datapath, ghi bitmask core dự phòng (ví dụ CPU 0, 1) vào file này. Ví dụ: `echo 3 > /proc/irq/125/smp_affinity` gán IRQ 125 chỉ được xử lý bởi CPU 0 hoặc 1, không làm phiền CPU 2-5 đang chạy PMD.

## 16.0.6 Profiling với perf + bpftrace

Khi tinh chỉnh hiệu năng, đo lường trước khi thay đổi là nguyên tắc vàng. Ba lớp công cụ cho từng mức tóm tắt:

**Thống kê mức driver qua `ethtool`:**
```
ethtool -S eth0 | grep -E 'drops|errors|discards'
```
Các bộ đếm quan trọng: `rx_dropped` (gói tin bị drop do queue đầy), `rx_missed_errors` (gói tin bị drop do driver không theo kịp NIC), `rx_no_buffer_count` (mempool cạn — chỉ báo mạnh cho workload DPDK). Nếu các bộ đếm này tăng, nghĩa là driver hoặc kernel stack không xử lý kịp traffic đến.

**Perf counter cho cache miss và phân tích chu kỳ:**
```
perf stat -e cache-references,cache-misses,cycles,instructions -p <PID>
```
Tỷ lệ cache miss > 10% là dấu hiệu NUMA không thẳng hàng (buffer ở bộ nhớ từ xa) hoặc kích thước mempool sai. Cycles/instruction > 2.0 là dấu hiệu đứng máy do băng thông bộ nhớ hoặc TLB miss — cần xem xét hugepages, gợi ý prefetch.

**bpftrace probe cho độ trễ gói tin per-CPU:**
```
bpftrace -e 'kprobe:__netif_receive_skb { @[cpu] = hist(nsecs - @start[tid]); }
                kretprobe:__netif_receive_skb { @start[tid] = nsecs; }'
```
Script này đo phân phối độ trễ của hàm kernel `__netif_receive_skb` theo CPU, giúp phát hiện core nào bị quá tải. Các script tương tự có sẵn trong bcc-tools: `xdp_trace.bt`, `tcplife.bt`, `ext4slower`.

**perf flamegraph** cho hồ sơ CPU trực quan: `perf record -F 99 -a -g sleep 30 && perf script | stackcollapse-perf.pl | flamegraph.pl > flame.svg`. Flamegraph cho thấy hàm nào chiếm nhiều CPU — nếu thấy hàm `netif_skb_features` hoặc `napi_schedule` chiếm đỉnh, cân nhắc chuyển sang XDP native hoặc DPDK.


Kết quả mong đợi:

> **Trạng thái lab verification:** Output băng thông thực tế theo tuning step dựa trên Red Hat + Intel performance guides; chờ verify lab thực.

**Mục tiêu:** Iterate tuning từ 10 Gbps baseline lên wire-speed (40 Gbps), document từng step delta. Capstone Khối XVI, integrate kiến thức từ Ex1 + Ex2 + 16.1 + 16.2.

**Chuẩn bị:**
- Physical server với 40 Gbps NIC (Intel XL710, Mellanox ConnectX-4)

**Mô hình lab:**

```
2-socket server, 40Gbps NIC

Socket 0 (cores 0-7, 6 mem channels) ── QPI ── Socket 1 (cores 8-15)
  NIC (PCIe Gen3 x8, 40 Gbps, on socket 0)
  OVS-DPDK PMD threads on CPU 2,4 (socket 0, physical cores only)
```

**Bước thực hiện — iterative tuning:**

**Step 0 (baseline):** Plain OVS kernel, no tuning:

```bash
[host]# iperf3 -c 10.0.0.1 -t 30 -P 4
```

```bash
[host]# cat /proc/meminfo | grep Huge
```

**Step 2: CPU isolation**

```bash
[host]# grubby --update-kernel=ALL --args="isolcpus=2,4,6"
[host]# reboot
[host]# cat /sys/devices/system/cpu/isolated
2,4,6
```

Re-benchmark: ~15 Gbps (scheduler không interrupt PMD cores).

**Step 3: RSS multi-queue**

```bash

[host]# ethtool -X ens1 equal 4
[host]# ethtool -N ens1 rx-flow-hash tcp4 sdfn  # 5-tuple hash
```

Re-benchmark: ~25 Gbps (parallel RSS queues).

**Step 4: OVS-DPDK với 2 PMD threads pinned**

```bash
[host]# ovs-vsctl set Open_vSwitch . other_config:dpdk-init=true
[host]# ovs-vsctl set Open_vSwitch . other_config:dpdk-socket-mem=2048,0
[host]# ovs-vsctl set Open_vSwitch . other_config:pmd-cpu-mask=0x14  # CPU 2,4
[host]# systemctl restart openvswitch-switch
[host]# ovs-vsctl add-br br0-dpdk -- set bridge br0-dpdk datapath_type=netdev
[host]# ovs-vsctl add-port br0-dpdk dpdk-p0 -- set interface dpdk-p0 type=dpdk \
        options:dpdk-devargs=0000:01:00.0
```

Re-benchmark: ~40 Gbps (wire-speed).

**Output mong muốn — table kết quả:**

| Step | Config | Băng thông thực tế | Delta | CPU Cost |
|------|--------|-----------|-------|----------|
| 0 | Baseline kernel | 10 Gbps | — | 4 cores ~80% |
| 1 | +Hugepages 2MB | 12 Gbps | +20% | Same |
| 2 | +isolcpus=2,4,6 | 15 Gbps | +25% | 3 cores isolated |
| 3 | +RSS 4 queues | 25 Gbps | +67% | Balanced 4 queues |

**Bài học học được:**

Thứ nhất, **tuning is iterative, compound effect**: không có single silver bullet. 4× total improvement từ 4 independent optimizations.

Thứ hai, **Hugepages + isolcpus** relatively low-risk — apply first cho mọi network-heavy workload. RSS multi-queue + DPDK higher-commitment (require topology understanding).

Thứ ba, **Wire-speed chuyển tiếp** (40 Gbps, 30-60 Mpps) achievable trên commodity hardware với proper tuning. Production service providers routinely achieve 100+ Gbps per server.

## 16.0.7 So sánh DPDK/AF_XDP/kernel với OVS + OVN + Cilium

Trong hệ sinh thái software data plane 2024-2026, 3 technology co-exist ở layer khác nhau:

| Dimension | OVS kernel datapath | OVS-DPDK / OVS-AFXDP | Cilium eBPF |
|-----------|---------------------|----------------------|--------------|
| Data plane location | Kernel (openvswitch.ko module) | Userspace (ovs-vswitchd) | Kernel (eBPF program hook XDP + tc) |
| Băng thông thực tế | 1.5-2 Mpps/core | 10-15 Mpps/core (DPDK) / 8-12 Mpps (AF_XDP) | 5-10 Mpps/core (XDP) |
| Độ trễ | ~20 µs | ~2 µs (DPDK) / ~10 µs (AF_XDP) | ~5 µs (XDP native) |
| CPU cost | Interrupt-driven (idle low) | Poll mode 100% core (DPDK) / hybrid (AF_XDP) | Interrupt-driven |
| Programmability | OpenFlow + conntrack | Same as kernel + DPDK features | eBPF program tùy ý |
| Kernel features preserved | All | DPDK: none, AF_XDP: most | All |
| Deployment complexity | Low | High (hugepages + IOMMU + isolcpus) | Medium (BPF toolchain) |
| Main use case | Hypervisor VM/pod | HFT, 5G UPF, NFV appliance | K8s CNI (Cilium, Calico with eBPF) |

**Khi nào dùng gì trong deployment thực tế:**

- **OVS kernel + OVN**: Hypervisor workload (OpenStack ML2/OVN), K8s với OVN-Kubernetes. 95% use case cloud native đủ.
- **OVS-DPDK**: NFV VNF (virtualized firewall, LB, SBC), 5G UPF, edge caching với băng thông thực tế >10 Gbps/hypervisor. Đốt CPU là chấp nhận được vì dedicated workload.
- **OVS-AFXDP**: Compromise khi cần băng thông thực tế > kernel nhưng muốn giữ iptables/tc features. Edge gateway, DDoS mitigation với selective offload.
- **Cilium eBPF**: K8s CNI replace kube-proxy + iptables. Performance tốt hơn iptables-based kube-proxy ~10×. Identity-based security thay IP-based NetworkPolicy.

### Cross-cut với OVN + OpenStack

OVN với OVS backend hỗ trợ **cả kernel và DPDK datapath** qua config `other_config:datapath-type`. Production pattern:

- **kolla-ansible OpenStack default**: OVS kernel datapath. Simple, reliable, đủ cho standard workload.
- **Telco NFV deployment**: OVS-DPDK. Required cho 5G UPF + MEC (Multi-Access Edge Computing).
- **OVN-Kubernetes trong K8s**: OVS kernel (default). Cilium+eBPF alternative cho enterprise K8s (Red Hat OpenShift 4.11+ support Cilium).

Lưu ý: Chuyển từ kernel sang DPDK trong existing OVN deployment **yêu cầu downtime** (restart ovs-vswitchd, reload flow, reconfigure port). Plan migration window cẩn thận.

## 16.0.8 Điểm cốt lõi cần nhớ

**Sáu điểm cốt lõi:**

1. **3 data path option cho Linux network**: kernel (default, interrupt-driven, ~2 Mpps/core), DPDK (poll mode, 10-15 Mpps/core, burn 100% CPU), AF_XDP (shared memory ring + XDP program, ~10 Mpps/core, giữ kernel features).
2. **DPDK = EAL + PMD + hugepages + NUMA pinning**. EAL khởi tạo environment, PMD là userspace driver, hugepages tránh TLB miss + zero-copy, NUMA pinning essential cho performance.
3. **AF_XDP = UMEM + 4 rings (FILL/RX/TX/COMPLETION)**. Zero-copy với driver support (ixgbe, i40e, ice, mlx5). 3 attach mode: driver/native (fastest), SKB (generic, compatible), hardware offload (Netronome, ConnectX).
4. **Kernel tuning không dead**: sysctl `net.core.netdev_max_backlog`, `rmem_max`, `busy_poll`; RPS/RFS cho multi-queue; NAPI budget tuning. Có thể đạt 20-30 Mpps với proper tuning, đủ cho nhiều workload.
5. **CPU pinning + NUMA awareness** là fundamental: `isolcpus` kernel cmdline + `pmd-cpu-mask` trong OVS + `smp_affinity` cho IRQ. Cross-NUMA memory access chậm gấp đôi, kill performance.
6. **Profiling trước khi tune**: `ethtool -S` (driver counter), `perf stat` (cache miss + IPC), `bpftrace` (per-CPU độ trễ histogram), `perf flamegraph` (CPU hotspot). Đo trước khi thay đổi là nguyên tắc vàng.

### ▶ Capstone POE: Hyperscale Kubernetes cluster 2026 — chọn CNI data plane nào?

> **Trạng thái lab verification:** POE dựa trên publicly documented deployment (Google GKE, Microsoft AKS, Alibaba ACK) + benchmark Cilium vs Calico vs OVN-Kubernetes public; chờ user có cluster để verify.

**Mục tiêu:** Đánh giá lựa chọn CNI data plane cho K8s cluster hyperscale 2026, match requirements với capability.

**Scenario:** Organization W triển khai Kubernetes cluster 10.000 node × 50 pod/node = 500K pods. Network traffic: 70% east-west intra-cluster (pod-to-pod), 20% north-south ingress, 10% egress to external services. Băng thông thực tế per node ~10-25 Gbps east-west. SecOps requirement: NetworkPolicy at pod identity level, không IP-based. Performance requirement: pod-to-pod độ trễ P99 < 5 ms.

**4 phương án CNI data plane:**

1. **kube-proxy iptables + Flannel/Calico standard**: Traditional K8s CNI, iptables IP-based rules.
2. **OVN-Kubernetes (OVS kernel datapath)**: OpenStack-derived OVN ported cho K8s. Kernel OVS + OVN logical pipeline.
3. **Cilium eBPF (kernel XDP + tc hook)**: Pod-to-pod qua eBPF, replace kube-proxy. Identity-based policy.
4. **OVS-DPDK hypervisor**: Userspace DPDK cho node-level network acceleration (cần cho bare-metal workload băng thông thực tế-critical).

**Predict:** Đánh giá 4 phương án theo 5 tiêu chí: (a) băng thông thực tế, (b) độ trễ, (c) pod identity NetworkPolicy, (d) deployment complexity, (e) long-term viability.

**Observe (reasoning + data check):**

**Phương án 1 (iptables + Flannel/Calico standard):**
- Băng thông thực tế: Limited. 500K pod × iptables lookup O(n) scaling → performance cliff ~10K service rules.
- Độ trễ: 50-100 µs P99 pod-to-pod (iptables linear scan).
- Pod identity policy: Limited (Calico NetworkPolicy IP-based).
- Complexity: Low. Mature ecosystem.
- Long-term: Stable nhưng losing ground to eBPF.

**Phương án 2 (OVN-Kubernetes):**
- Băng thông thực tế: 10 Gbps/node OK với kernel datapath. 25+ Gbps cần DPDK + complexity.
- Độ trễ: 20-30 µs P99 (OVN logical pipeline overhead).
- Pod identity: OVN port group + address set, flexible nhưng IP-based.
- Complexity: High (OVN northd, SB DB, 27+ pipeline stages). Debug khó.
- Long-term: Red Hat OpenShift backing. Good cho OpenStack-adjacent environment.

**Phương án 3 (Cilium eBPF):**
- Băng thông thực tế: 10-20 Gbps/node native. XDP hook cho ingress acceleration.
- Độ trễ: <5 µs P99 pod-to-pod (eBPF fast path).
- Pod identity: Excellent. Cilium identity (derived từ labels) là first-class concept. No IP churn issue.
- Complexity: Medium. eBPF toolchain learning curve nhưng Cilium CLI mature.
- Long-term: Strong (Isovalent acquired by Cisco 2024, CNCF graduated). Sự chấp nhận hyperscale đang tăng.

**Phương án 4 (OVS-DPDK bare-metal):**
- Băng thông thực tế: 40+ Gbps/node.
- Độ trễ: <2 µs.
- Pod identity: Limited (OVS OpenFlow IP-based).
- Complexity: Very high (hugepages + IOMMU + isolcpus + CPU dedicated).
- Long-term: OK nhưng niche (mostly NFV, không K8s standard).

**Explain:** Recommendation cho scenario hyperscale 2026:

**Cho Organization W (500K pods, 10-25 Gbps/node, P99 <5ms, identity policy):**

**Primary choice: Cilium eBPF (phương án 3).** Lý do:

- Độ trễ requirement P99 <5 ms chỉ Cilium eBPF achieve được consistently (OVN-K8s 20-30 µs okay trong lab, but scale to 500K pods tạo control plane overhead).
- Identity-based NetworkPolicy là requirement critical — Cilium first-class support, Calico/OVN-K8s IP-based kém hơn.
- Băng thông thực tế 10-25 Gbps/node là sweet spot của Cilium eBPF, không cần DPDK.
- Long-term viability: CNCF graduated + Isovalent/Cisco backing = stable.

**KHÔNG chọn**:
- iptables/Flannel: không scale tới 500K pods.
- OVN-K8s: over-complexity cho K8s-native workload (strength là OpenStack integration, not K8s scale).
- OVS-DPDK: overkill cho K8s, đốt CPU waste trong cloud shared.

**Tuning recommendation cho Cilium eBPF:**
- Enable eBPF masquerade + eBPF kube-proxy replacement.
- Configure BPF LB algorithm = Maglev cho consistent hashing.
- Set CPU affinity cho BPF processing (isolcpus 2-3 core per node).
- Monitor BPF map sizes (connection tracking, identity map) qua `cilium bpf metrics`.

**Finish:** Nguyên tắc universal: **"CNI choice phải match scale + requirement, không follow trend"**. Cilium eBPF phù hợp cho hyperscale + identity + performance. Với cluster nhỏ (<1000 pods), iptables đủ đơn giản. Với NFV telco, OVS-DPDK. Một size không fit all. Pattern này áp dụng cho mọi quyết định kiến trúc.

## 15.1.1 OVN-Kubernetes components

OVN-K8s deploys three main components in the cluster, each running as a DaemonSet or a Deployment in the namespace `ovn-kubernetes` or `openshift-ovn-kubernetes` (depending on the distribution). (1) **ovnkube-master**, a Deployment running on the control plane node, with one active replica plus one or two standby replicas under a leader election scheme. Responsibility: watch the Kubernetes API through informers (Pod, Service, Endpoints, NetworkPolicy, Node, Namespace) and translate the resources into OVN NB DB (a logical switch per node, a logical router per cluster, ACLs for NetworkPolicy, Load_Balancer entries for Service). (2) **ovnkube-node**, a DaemonSet running on every worker and control plane node. Responsibility: manage the local OVS bridge (`br-int`), allocate the pod interface when a pod is scheduled (create a veth pair, attach it to `br-int`, call CNI ADD or DEL), and run the local ovn-controller to receive logical flows from SB DB. (3) **nbdb-0/1/2 plus sbdb-0/1/2 plus northd-0/1/2**, a 3-replica StatefulSet running an HA ovsdb-server (a Raft cluster) for NB DB and SB DB, with ovn-northd translating NB to SB.

The HA configuration for ovn-northd is somewhat subtle: northd holds no state, but only one instance is active at a time (to avoid conflicts when writing to SB DB). OVN 22.03 and later includes a leader election mechanism through `ovn-northd --pidfile --detach` with a lock through NB DB. In the Kubernetes context, northd runs as a sidecar in the same pod as nbdb and sbdb, using a ConfigMap to distribute leader state.

**SBDB per-AZ strategy:** in a large cluster (more than 500 nodes), a single SB DB cluster becomes a bottleneck because every node queries flows from the same Raft cluster, and latency grows with the chain distance. OVN Interconnect (OVN-IC) lets the operator split the cluster into multiple AZs (availability zones), with each AZ having its own SB DB, communicating between AZs through a transit switch. This is covered in section 15.1.4.

## 15.1.2 NetworkPolicy to OVN ACL translation

Kubernetes NetworkPolicy is a resource that lets the user define ingress and egress rules for pods in a namespace. For example: "a pod with the label `app=web` in the namespace `frontend` is only allowed to receive traffic from a pod `app=api` in the namespace `backend` on port 8080". By default, a namespace without any NetworkPolicy lets all pods communicate freely (default-allow). When a NetworkPolicy applies to a pod, the pod switches to **default-deny** and only allows traffic that matches a rule.

The algorithm for translating NetworkPolicy to an OVN ACL: (1) **podSelector** to **Port_Group**, OVN creates a Port_Group containing all logical ports of the pods that match the selector. When pods are added or removed (scaling, rescheduling), ovnkube-master updates the Port_Group in real time. (2) **namespaceSelector** to **nested Address_Set**, every namespace has an Address_Set containing the IP addresses of every pod in the namespace; the namespaceSelector is matched as a union of Address_Sets. (3) **ingress and egress rules** to **ACLs** with a `match` clause, `action=allow/drop`, and `priority` according to the rule order in the spec. (4) **Implicit default-deny**, every Port_Group has a lowest priority ACL with `action=drop` as a catch-all, which is activated only when a NetworkPolicy applies to that pod.

OVN ACLs are **stateful** through conntrack (similar to nftables `ct state`): when ingress traffic is allowed, the reply traffic in the reverse direction is automatically allowed without an explicit egress rule. This matches Kubernetes NetworkPolicy semantics (return traffic does not need to be defined). The enforcement mechanism uses ACLs with `match=... && ct.new` for new traffic, `ct.est` for established traffic, and `ct.rel` for related traffic (such as ICMP echo reply).

Performance: policy is enforced in the OVS datapath, not through iptables, so it scales better for pods that change frequently. A Red Hat 2022 benchmark on a 200-node cluster: NetworkPolicy update latency was about 500 ms for OVN-K8s versus about 3 seconds for kube-proxy plus Calico iptables at 10K rules.

## 15.1.3 Service plus kube-proxy replacement

A Kubernetes Service is an abstraction for a virtual IP (ClusterIP) plus a port mapped to a set of endpoints (pod IP and port). By default, kube-proxy runs on every node and installs iptables rules (or IPVS) to DNAT from ClusterIP plus port to pod IP plus port. OVN-K8s replaces kube-proxy entirely; kube-proxy no longer needs to run on the node.

Service to OVN mapping:

- **ClusterIP Service** to an entry in the **Load_Balancer** table of NB DB. A Load_Balancer has `vips = {"10.96.0.1:80": "10.244.1.5:8080, 10.244.2.3:8080"}` (VIP to list of endpoints). OVN attaches the Load_Balancer to a logical switch or a logical router; when a packet arrives at the VIP, OVN performs DNAT using a hash algorithm (5-tuple) to pick an endpoint. Session affinity through ClientIP is supported when the Service is configured with `sessionAffinity: ClientIP`.
- **NodePort Service** to a combination of a ClusterIP Load_Balancer plus **chassisredirect** (a special OVN gateway chassis) plus DNAT. When traffic arrives at the node IP plus NodePort from outside, the gateway chassis receives it, performs DNAT to the ClusterIP, and then load-balances to an endpoint.
- **ExternalTrafficPolicy=Local** is an important optimization: instead of cluster-wide load balancing (which can SNAT and lose the client IP), OVN routes only to endpoints on the same node as the ingress traffic. OVN-K8s implements this through a filtering rule in the Load_Balancer: only endpoints with `chassis=local_chassis` are picked.
- **LoadBalancer Service** depends on the cloud controller (AWS ELB, Azure LB) to allocate a public VIP; OVN-K8s handles the in-cluster portion.

**ExternalIP annotation and MetalLB integration:** when an on-prem cluster has no cloud LB, the operator uses MetalLB or the native `egressIP` of OVN-K8s to assign a public IP. OVN-K8s 1.0 and later supports a native LoadBalancer through BGP advertisement from a gateway node, similar to the BGP mode of Calico or MetalLB.

## 15.1.4 Multi-cluster networking (OVN-IC)

**OVN Interconnect (OVN-IC)** is a feature introduced in 2022 that lets multiple OVN clusters connect at the logical layer. Traffic between clusters traverses an overlay (Geneve) without needing external routing. Main use cases: multi-cluster Kubernetes (federation, multi-region failover), hybrid cloud stretch, and multi-AZ labs.

OVN-IC architecture: (1) **Transit Switch**, a special logical switch connecting the gateway chassis of multiple AZs. The gateway chassis of each AZ connects to the transit switch through a Geneve tunnel. (2) **IC NB DB plus IC SB DB**, separate databases that hold global state about transit switches and inter-AZ routing. Each AZ has a local **ovn-ic-nb-ovsdb** replica, synchronized with the global IC DB through the ovn-ic program. (3) **VNI isolation**, each AZ has its own Virtual Network Identifier; VNI A traffic does not leak into VNI B unless an explicit routing rule is in place.

In OVN-K8s 1.0 and later (2024), each cluster zone in a multi-cluster setup is assigned its own chassis ID. The ovnkube-master advertises local routes to the transit switch and learns remote routes from the IC DB. Pod-to-pod traffic between zones travels through a Geneve tunnel from one gateway to the other gateway, similar to the inter-subnet path in a single zone but with a longer chain (local OVS to local gateway to transit to remote gateway to remote OVS).

Trade-off: OVN-IC adds significant deployment complexity. It requires a global IC DB cluster, HA for the gateway, and management of the VNI mapping. It is appropriate only when multi-cluster networking is truly needed. If only DR (disaster recovery) across regions is needed, ExternalIP plus DNS failover is simpler.

## 15.1.5 Migration from OpenShift-SDN to OVN-K8s

OpenShift 3.x and 4.0 to 4.11 used **OpenShift-SDN**, a Red Hat custom CNI based on raw OVS flow rules (without OVN). Architecture: a node-to-node VXLAN overlay, iptables for service, and multi-tenancy rules through VNID. The cluster ran well at small to medium scale but struggled past 500 nodes: the iptables rule count exploded (more than 50K rules in a large cluster), and the multi-tenancy implementation was complex.

Red Hat decided to migrate to OVN-K8s starting from OpenShift 4.10 (March 2022), initially as opt-in, and from 4.12 (January 2023) as the default CNI for new clusters. Reasons: (1) OVS and OVN have a large community maintained by multiple vendors; (2) NetworkPolicy enforcement at the OVS data plane scales better than iptables; (3) preparation for multi-cluster (OVN-IC) and VRF-based multi-tenancy; (4) OVN-K8s integrates better with Red Hat Service Mesh and OpenShift Virtualization (KubeVirt).

Migration path for an existing cluster: (1) **offline migration**, drain every node, replace the CNI (adjust the cluster configuration), restart the nodes, and redeploy the pods. Downtime is about 30 to 60 minutes for a 50-node cluster. (2) **online migration (introduced in 4.14)**, migrate node batches; pod traffic continues through the old CNI while the new CNI is being provisioned. More complex but no downtime.

Features available in both: NetworkPolicy, Service, EgressIP, and Multicast. OVN-K8s only features: multi-cluster (OVN-IC), full IPv6 dual stack, hardware offload (OVS HW offload through switchdev), the EgressFirewall CRD (ACLs for traffic leaving the cluster), and BGP EVPN for the metadata service.

### ▶ Guided Exercise 1: Deploy OVN-K8s on a kind cluster

> **Lab verification status:** output is based on OVN-Kubernetes docs from 2024; awaiting verification on a real lab.

**Goal:** run OVN as the Kubernetes CNI and verify the NetworkPolicy to OVN ACL translation chain end to end. This is a representative setup of a production environment used to understand the OVN-K8s components.

**Setup:**
- Ubuntu 22.04 VM, 8 GB RAM, 4 CPU
- kind plus kubectl installed
- Clone OVN-K8s: `git clone https://github.com/ovn-org/ovn-kubernetes.git`

**Lab topology:**

```
+--------------------------------------------------+
|         kind cluster (3 nodes)                   |
|                                                  |
|  ns ovn-kubernetes:                              |
|                                                  |
|  +--------------------------------------------+  |
|  | ovnkube-master-0/1/2 (3 replicas)          |  |
|  |  +- northd (NBDB to SBDB compile)          |  |
|  |  +- nb-ovsdb plus sb-ovsdb (Raft cluster)  |  |
|  |  +- ovnkube-master (K8s to OVN translator) |  |
|  +--------------------------------------------+  |
|                                                  |
|  +--------------------------------------------+  |
|  | ovnkube-node (daemonset, per K8s node)     |  |
|  |  +- ovn-controller (SBDB to OpenFlow)      |  |
|  |  +- ovs-vswitchd plus ovsdb-server         |  |
|  +--------------------------------------------+  |
|                                                  |
|  ns default:                                     |
|  +-----------+   +-----------+                   |
|  | pod web   |   | pod app   |                   |
|  | 10.244.0.5|   | 10.244.0.6|                   |
|  +-----+-----+   +-----^-----+                   |
|        |  NetworkPolicy |                        |
|        +----------------+ -> OVN ACL             |
+--------------------------------------------------+
```

**Steps:**

**1.** Create a kind cluster with the OVN-K8s CNI and disable the default kindnet:

```bash
[host]$ cat > kind-ovn.yaml << 'EOF'
kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
networking:
  disableDefaultCNI: true
nodes:
  - role: control-plane
  - role: worker
  - role: worker
EOF
[host]$ kind create cluster --name ovn-k8s --config kind-ovn.yaml
```

**2.** Deploy OVN-K8s:

```bash
[host]$ cd ovn-kubernetes/dist/images
[host]$ ./daemonset.sh --image=ghcr.io/ovn-org/ovn-kubernetes/ovn-kube-ubuntu:master
[host]$ kubectl apply -f ovn-setup.yaml
[host]$ kubectl apply -f rbac-ovnkube-cluster-manager.yaml
[host]$ kubectl apply -f ovnkube-master.yaml
[host]$ kubectl apply -f ovnkube-node.yaml
```

**3.** Verify pods are running:

```bash
[host]$ kubectl -n ovn-kubernetes get pods
```

Expected output:

```
NAME                        READY   STATUS
ovnkube-master-0            5/5     Running
ovnkube-master-1            5/5     Running
ovnkube-master-2            5/5     Running
ovnkube-node-xxxxx          3/3     Running
ovnkube-node-yyyyy          3/3     Running
ovnkube-node-zzzzz          3/3     Running
```

**4.** Inspect the OVN NBDB:

```bash
[host]$ kubectl exec -n ovn-kubernetes ovnkube-master-0 -c nb-ovsdb -- ovn-nbctl show
```

Expected output: logical switches per node (`join`, `ovn_cluster_router`, node-specific), Logical_Switch_Ports for pods.

**5.** Deploy test pods:

```bash
[host]$ kubectl run web --image=nginx --labels=tier=web
[host]$ kubectl run app --image=alpine --labels=tier=app -- sleep 3600
[host]$ kubectl get pods -o wide
```

**6.** Apply a NetworkPolicy that allows web to app on port 80:

```bash
[host]$ cat > policy.yaml << 'EOF'
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-web-to-app
spec:
  podSelector:
    matchLabels:
      tier: app
  policyTypes: [Ingress]
  ingress:
  - from:
    - podSelector:
        matchLabels:
          tier: web
    ports:
    - protocol: TCP
      port: 80
EOF
[host]$ kubectl apply -f policy.yaml
```

**7.** Observe the OVN ACL creation:

```bash
[host]$ kubectl exec -n ovn-kubernetes ovnkube-master-0 -c nb-ovsdb -- ovn-nbctl list ACL | head -30
```

Expected output:

```
_uuid               : abc123...
action              : allow-related
direction           : to-lport
label               : 1
log                 : false
match               : "outport == @a123_Ingress && ip4 && tcp.dst == 80 && ip4.src == {10.244.0.5}"
name                : "default:allow-web-to-app:Ingress:0"
priority            : 1001
severity            : []
```

**Expected outcome:** the NetworkPolicy to OVN ACL translation occurs automatically, and the Port_Group `a123_Ingress` membership equals the pods with the label `tier=app`.

**Lessons learned:**

First, **OVN-K8s is a high-availability CNI**. ovnkube-master runs three replicas (a Raft cluster for OVSDB), and ovnkube-node is a DaemonSet per worker. It is more fault tolerant than a CNI with a single point of failure (such as the older OpenShift-SDN).

Second, **automatic NetworkPolicy translation**. Label selectors map to Port_Group membership; port rules map to ACL match expressions. Each Kubernetes NetworkPolicy generates 2 to 5 OVN ACL entries depending on complexity.

Third, **ovn-nbctl remains the source of truth**. Debug NetworkPolicy issues by inspecting the NBDB ACLs, not by following an iptables path as with kube-proxy.

**Cleanup:**

```bash
[host]$ kind delete cluster --name ovn-k8s
```

### ▶ Lab 2: NetworkPolicy debug plus trace

> **Lab verification status:** the trace output format is based on the ovn-trace docs; awaiting verification on a real lab.

**Goal:** practice debugging a NetworkPolicy issue with `ovn-trace`, a high-value tool when a production NetworkPolicy does not behave as expected.

**Setup:** complete Exercise 1 baseline (OVN-K8s cluster running).

**Lab topology:**

```
3-tier application with a NetworkPolicy chain:

+---------+  ACL1   +---------+  ACL2   +---------+
| pod web |-------->| pod app |-------->| pod db  |
| tier=web| :8080   | tier=app| :5432   | tier=db |
+---------+         +---------+         +---------+
                                             ^
                      default-deny: blocks everything else
```

**Steps:**

**1.** Deploy 3-tier pods:

```bash
[host]$ kubectl run web --image=nginx --labels=tier=web
[host]$ kubectl run app --image=alpine --labels=tier=app -- sleep 3600
[host]$ kubectl run db --image=postgres --env=POSTGRES_PASSWORD=x --labels=tier=db
```

**2.** Apply 3-tier NetworkPolicies:

```yaml
# policy-web-to-app.yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata: {name: allow-web-to-app}
spec:
  podSelector: {matchLabels: {tier: app}}
  policyTypes: [Ingress]
  ingress:
  - from: [{podSelector: {matchLabels: {tier: web}}}]
    ports: [{protocol: TCP, port: 8080}]

# policy-app-to-db.yaml (similar structure for tier=db, port=5432)
```

**3.** Apply both:

```bash
[host]$ kubectl apply -f policy-web-to-app.yaml -f policy-app-to-db.yaml
```

**4.** Verify with ovn-trace (simulate a packet web to app:8080):

```bash
[host]$ WEB_IP=$(kubectl get pod web -o jsonpath='{.status.podIP}')
[host]$ APP_IP=$(kubectl get pod app -o jsonpath='{.status.podIP}')
[host]$ kubectl exec -n ovn-kubernetes ovnkube-master-0 -c ovnkube-master -- \
        ovn-trace --minimal "inport == \"web\" && \
                             eth.src == 0a:00:00:00:00:01 && \
                             ip4.src == $WEB_IP && \
                             ip4.dst == $APP_IP && \
                             tcp.dst == 8080"
```

Expected output: the trace flows through the OVN logical pipeline and ends with an `output to \"app\"` action.

**5.** Test actual traffic:

```bash
[host]$ kubectl exec web -- curl -m 5 http://$APP_IP:8080
```

Expectation: the connection succeeds or returns an HTTP response.

**6.** Intentionally break the path: delete the app to db rule:

```bash
[host]$ kubectl delete networkpolicy allow-app-to-db
```

**7.** Debug blocked traffic:

```bash
[host]$ DB_IP=$(kubectl get pod db -o jsonpath='{.status.podIP}')
[host]$ kubectl exec -n ovn-kubernetes ovnkube-master-0 -c ovnkube-master -- \
        ovn-trace --minimal "inport == \"app\" && ip4.src == $APP_IP && \
                             ip4.dst == $DB_IP && tcp.dst == 5432"
```

Expected output: the trace ends with a `drop` action (no matching allow ACL, so default-deny applies).

**8.** Inspect the logical flow to identify the cause:

```bash
[host]$ kubectl exec -n ovn-kubernetes ovnkube-master-0 -c nb-ovsdb -- \
        ovn-sbctl lflow-list | grep -A1 "ls_out_acl"
```

**Expected outcome:** a clear decision path showing exactly which ACL allowed or denied the flow, with the match conditions evaluated at each stage.

**Lessons learned:**

First, **`ovn-trace` is a debugging superpower**. It lets the operator simulate a packet through the entire logical pipeline without sending a real packet. Compared with debugging iptables and kube-proxy (which requires running tcpdump on each interface), it is far simpler.

Second, **default-deny semantics**. When a NetworkPolicy exists for a pod (ingress or egress), unmatched traffic is **implicitly denied**. Common mistake: applying a NetworkPolicy without adding the necessary allow rule, which isolates the pod.

Third, **OVN logical flow visibility**. `ovn-sbctl lflow-list` shows every compiled logical flow, ordered by priority. Debug NetworkPolicy translation issues here.

**Cleanup:**

```bash
[host]$ kubectl delete pod web app db
[host]$ kubectl delete networkpolicy --all
```

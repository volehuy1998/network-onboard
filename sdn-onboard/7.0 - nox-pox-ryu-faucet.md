## Context: controller software and the gap between specification and production

When OpenFlow 1.0 was released on 2009-12-31, the specification defined the protocol between switch and controller, but no controller was ready for the community to use immediately. The practical problem was clear: an engineer who wanted to experiment with OpenFlow had to write a controller from scratch. That required deep understanding of the TCP protocol, the binary encoding of OpenFlow messages, and managing many switch connections at once.

NOX, open-sourced by Nicira in 2008 (before the official OpenFlow 1.0 was published), filled this gap by providing a complete framework. A developer did not have to handle TCP sockets or parse binary directly. The developer only had to write handlers for higher-level events. This design direction, **the controller as a network operating system, with applications running on the controller as processes run on an operating system**, became the dominant philosophy for every later controller in the lineage. The phrase was made explicit in the title of the NOX paper: "Towards an Operating System for Networks."

> **Key point:** The controller-as-OS model assumes a clear split. The controller provides abstractions for topology and packet forwarding. The application writes policy on top of those abstractions. NOX was the first realization of this model. Every later framework (POX, Ryu, Faucet, ODL, ONOS) inherited the same philosophy of separation, even though their implementations differ.

## 7.0.1 NOX: Ethane's runtime, open-sourced in 2008

NOX began as the runtime engine for Ethane, the enterprise network access-control system that Martin Casado and his Stanford group designed (Part 2.4). When Casado and Nicira decided to open-source the Ethane controller component in 2008, they named it NOX and rewrote it more cleanly so that the community could use it independently of the Ethane policy model.

The original paper, **"NOX: Towards an Operating System for Networks"**, was presented at HotNets 2008 by Natasha Gude, Teemu Koponen, Justin Pettit, Ben Pfaff, Martin Casado, Nick McKeown, and Scott Shenker, and was then published in *ACM SIGCOMM Computer Communication Review*, Vol. 38, No. 3, July 2008. The paper introduced the network-OS concept: just as UNIX provides abstractions for hardware, NOX provides a "network view" abstraction (the entire topology, every host, every flow) for network applications.

**The technical architecture of NOX** is C++ event-driven, with a single-threaded event loop using `select()` or `poll()` for I/O multiplexing. An application written in C++ (or Python through bindings) registers handlers:

```cpp
// Example C++ handler in NOX
class MyApp : public Component {
public:
    void configure(const Configuration*) {
        register_handler<Packet_in_event>(
            boost::bind(&MyApp::handle_packet_in, this, _1));
    }
    Disposition handle_packet_in(const Event& e) {
        const Packet_in_event& pi = assert_cast<const Packet_in_event&>(e);
        // pi.datapath_id: switch ID
        // pi.buffer_id: buffer on the switch
        // pi.packet: raw packet bytes
        install_flow(...);
        return CONTINUE;
    }
};
```

NOX supports OpenFlow 0.8.9 (the draft at the time of release) and later OpenFlow 1.0. Design limit: the single-threaded event loop means each handler must finish quickly. Any blocking operation (a database query, a DNS lookup) blocks the entire controller. This was a deliberate trade-off: simplify programming by removing concurrent access to shared state.

> **Note:** NOX has two parallel versions. NOX-Classic is pure C++, the original 2008 to 2011 version. NOX-MT is multi-threaded, developed by Big Switch Networks in 2012. NOX-MT adds threading to raise throughput but loses the simplicity of the single-threaded model. When the community writes "NOX" without specifying the version, it usually refers to NOX-Classic.

After VMware acquired Nicira in 2012, NOX gradually stopped being actively developed. The repository at github.com/noxrepo/nox still exists, in archive state.

## 7.0.2 POX: a Python port for teaching, 2011

POX was developed by Murphy McCauley (UC Berkeley) in 2011 as a Python rewrite of NOX-Classic. The goal was clear from the start: **not production, but teaching**. McCauley designed POX so that computer-science students could write OpenFlow applications in a few hours instead of a few days, by leveraging Python's readability and rich library ecosystem.

POX integrates well with Mininet, the network emulation platform that also came from the UC Berkeley and Stanford group. Together they form the most popular pair in SDN teaching. Commands to start Mininet with the POX controller:

```bash
# Terminal 1: start POX with the basic learning module
[controller]$ python3 pox.py log.level --DEBUG forwarding.l2_learning

# Terminal 2: start Mininet and connect to POX
[controller]$ sudo mn --controller=remote,ip=127.0.0.1,port=6633 --topo=tree,depth=2
```

The `forwarding.l2_learning` module in POX is the canonical example. The controller learns the MAC address of each host from `packet_in`, installs the matching flow entries for subsequent packets, and handles flooding when the destination is unknown. The full logic is about 80 lines of Python, short enough to read in a single lab session.

**The design weakness of POX** is a direct consequence of Python: the GIL (Global Interpreter Lock) prevents Python from running truly in parallel on multiple cores. Single-threaded POX is roughly ten times slower than NOX C++ in throughput benchmarks, and it cannot scale to manage thousands of switches in production. McCauley did not hide this. The official POX documentation states clearly: "POX is not intended for production use."

That said, this is not a failure of POX. It is the success of the goal POX set for itself. Dozens of universities around the world (Stanford, Berkeley, Princeton, MIT, NUS, and many universities in Vietnam) use POX with Mininet as the foundation for teaching SDN. Many engineers working with OVN or ONOS today learned the concept of a control plane for the first time through a POX exercise.

## 7.0.3 Ryu: the NTT framework, 2012, with full OpenFlow support

The NTT (Nippon Telegraph and Telephone) research group developed Ryu starting in 2012, with the goal of filling the gap between POX (teaching, not production) and ODL or ONOS (enterprise scale, complex to deploy). The name "Ryu" in Japanese means "dragon" or "flow", and both meanings fit the role of a framework that processes OpenFlow flows.

**The technical strength of Ryu** compared with POX is full OpenFlow support from 1.0 to 1.5.1, including the OVS extensions (Nicira extensions: NXM fields). This matters especially as OVS becomes the most common virtual switch. Ryu can program OVS with extended match fields that OF 1.0 does not support, for example `tunnel_id`, `reg0` through `reg7` for metadata, and conntrack state. The modular architecture of Ryu lets a developer load only the components needed:

```bash
# Start Ryu with the simple-switch application
[controller]$ ryu-manager ryu.app.simple_switch_13

# Or load a self-written application
[controller]$ ryu-manager myapp.py --observe-links
```

A less-known but practically valuable feature is the **integrated BGP speaker in Ryu**. The `ryu.services.protocols.bgp` module lets the controller speak BGP with an external router and turn the routing information it receives into flow entries in OVS. A typical use case: the controller receives a BGP prefix announcement, and based on the next hop in the BGP table, installs the corresponding forwarding rule in OVS. It is a kind of route reflector for a data center that does not use a hardware router.

> **Key point:** Ryu was designed as a flexible framework, not as a packaged production controller. A Ryu user must write the application logic. Ryu provides APIs and abstractions, not policy. This is the key difference from Faucet, where policy is already defined in YAML and the operator only configures it, with no programming required.

The Ryu community was active from 2012 to 2018, then declined as ONOS captured the service-provider segment and Faucet captured the campus segment. However, Ryu is still used in research because of its transparency: the Python codebase is easy to read, has no complex dependencies, and supports every OpenFlow message type so a researcher can control every bit of a flow entry.

## 7.0.4 Faucet: production focus, REANNZ, 2015

Faucet started as a Ryu application written by Josh Bailey and the REANNZ (Research and Education Advanced Network New Zealand) team together with the WAND (Waikato Applied Network Dynamics) group at the University of Waikato. REANNZ is the New Zealand national research and education network, operating high-speed connections between universities and research institutes. When REANNZ decided to move part of its infrastructure to SDN, the team realized that Ryu provides flexibility but not operability: no integrated monitoring, no declarative configuration, no rollback when the controller crashes.

**Faucet addresses the operability problem** through three main pieces. First, declarative YAML configuration instead of Python code: the operator describes the topology (switches, ports, VLANs) and policy (ACL, routing) in a YAML file, and Faucet translates that into flow entries. When policy needs to change, the operator edits the YAML and reloads it, with no need to redeploy Python code. Second, Gauge is a sidecar process separate from the controller. Gauge continuously polls statistics from the switches (port counters, flow counters) and exports them to Prometheus or InfluxDB. The operator gets a real-time monitoring dashboard. Third, Faucet defines a canonical multi-table pipeline for switches. Instead of letting each application decide how many tables and in what order, Faucet defines a standard pipeline (VLAN decoding, learning, routing, ACL, output) optimized for common chipsets.

```yaml
# Minimal Faucet configuration example, faucet.yaml
vlans:
  office:
    vid: 100
    description: "Office VLAN"

dps:
  sw1:
    dp_id: 0x1
    interfaces:
      1:
        name: "server1"
        native_vlan: office
      2:
        name: "server2"
        native_vlan: office
      3:
        name: "uplink"
        tagged_vlans: [office]
```

The simplicity of the YAML above, 17 lines to configure a three-port switch, is the main reason operations teams adopt Faucet. The operator does not need to understand OpenFlow flow entries or Python programming to deploy Faucet in a campus environment.

> **Key point:** Faucet and Ryu show two opposing operational philosophies built on the same technical foundation. Ryu is optimized for the question "what can I do with OpenFlow?". Maximum flexibility, no constraint. Faucet is optimized for the question "how do I keep this network running stably for three years?". Fixed functionality, but predictable operation. Both are correct in their own context.

REANNZ has run Faucet in production since 2016. Several ISPs and university networks in the United States and Europe (Internet2 members) also use Faucet for campus switching. Faucet remains in active development, with monthly releases as of 2026, unlike Ryu which has been in maintenance mode since around 2019.

## 7.0.5 Comparison of the four controllers and how to choose

These four controllers do not compete directly. They serve four different needs:

*Table 7-1: NOX, POX, Ryu, Faucet by purpose and technical traits*

| Criterion | NOX | POX | Ryu | Faucet |
|---|---|---|---|---|
| Language | C++ (Python binding) | Python | Python | Python (built on Ryu) |
| Released | 2008 | 2011 | 2012 | 2015 |
| Main purpose | Research prototype | Teaching | Flexible research | Production campus |
| OpenFlow support | 1.0 | 1.0, 1.3 | 1.0 to 1.5 | 1.3 (production tested) |
| Configuration | C++ or Python code | Python code | Python code | Declarative YAML |
| Monitoring | Not integrated | Not integrated | Not integrated | Gauge (Prometheus) |
| State in 2026 | Archive | Not developed | Maintenance | Active development |

When to use which: if the goal is to learn the OpenFlow mechanics, POX with Mininet is the best starting point because the codebase is short and examples are plentiful. If the goal is research with OVS in a lab, Ryu fits because of NXM extension support and full OpenFlow 1.5. If the goal is to deploy SDN in a real campus with an operations team that has no programming background, Faucet is the only viable option in this group.

### Guided Exercise 1: run the Ryu simple switch and observe `packet_in`

In this exercise you start Ryu with the simple-switch application, connect Mininet, and observe how the controller learns MAC addresses through `packet_in`.

**Outcomes:**
- Observe the `packet_in` log entries when a host sends its first frame.
- Verify the flow entry installed after the controller learns a MAC.
- Understand the lifecycle: flooding, then MAC learning, then unicast forwarding.

**Before you begin:**
Install Ryu (`pip install ryu`) and Mininet (`apt install mininet`). Verify that `ryu-manager --version` and `mn --version` do not error.

**Instructions:**

**1.** In the first terminal, start Ryu with the simple-switch module and enable debug logging:

```bash
[controller]$ ryu-manager --verbose ryu.app.simple_switch_13
```

Ryu listens on port 6653 (the OpenFlow 1.3 default). The log shows `BRICK SimpleSwitch13 PROVIDES ... EventOFPPacketIn`.

**2.** In a second terminal, start Mininet with a two-host topology:

```bash
[controller]$ sudo mn --controller=remote,ip=127.0.0.1,port=6653 \
               --switch=ovsk,protocols=OpenFlow13 --topo=single,2
```

Mininet creates switch `s1` with two hosts: `h1` (10.0.0.1) and `h2` (10.0.0.2).

**3.** At the Mininet prompt, ping from h1 to h2:

```
mininet> h1 ping h2 -c 3
```

**4.** Observe the log on the Ryu terminal. The first lines look like:

```
packet in 1 xx:xx:xx:xx:xx:x1 ff:ff:ff:ff:ff:ff 1
packet in 1 xx:xx:xx:xx:xx:x2 xx:xx:xx:xx:xx:x1 2
```

First line: switch ID `1`, source MAC of h1, destination MAC is broadcast (`ff:ff:ff:ff:ff:ff`), input port is `1`. This is an ARP request. h1 does not yet know the MAC of h2, so it sends a broadcast. The controller receives the `packet_in`, learns the MAC of h1 on port 1, and floods to all ports.

Second line: source MAC of h2, destination MAC of h1 (h2 replies to the ARP), input port is `2`. The controller learns the MAC of h2 on port 2 and installs unicast flow entries from h1 to h2 and h2 to h1.

**5.** Inspect the flow table on the switch:

```bash
[controller]$ sudo ovs-ofctl -O OpenFlow13 dump-flows s1
```

After the first ping, you see two flow entries with a specific `dl_dst` instead of a flood action. This is evidence that the controller learned the MACs and installed unicast forwarding.

**Finish:**

```
mininet> quit
```

Press Ctrl+C to stop Ryu.

## Key points to remember

First, NOX established the controller-as-OS model in 2008, before the official OpenFlow 1.0 release. Every later controller framework inherited the philosophy: the controller provides network abstractions, the application writes policy on top of those abstractions.

Second, POX and Ryu solve the same technical problem in different ways because they serve different users. POX prioritizes ease of learning, so a student can write an L2 learning switch in one session. Ryu prioritizes feature completeness, so a researcher can program every aspect of OpenFlow 1.5 and the NXM extensions.

Third, Faucet is evidence of an operational principle: flexibility does not mean ease of operation. By narrowing the scope to campus switching and defining a canonical pipeline up front, Faucet achieves the operational stability that Ryu cannot provide, even though both are written in Python.

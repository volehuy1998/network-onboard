## 9.14.1 Observability stack (compass Ch 18)

Production OVS cần ba layer observability:

**Layer 1 — Traffic analytics**: sFlow (Part 9.8) feed vào goflow2 hoặc ntopng. Cho biết flow nào đang dùng bandwidth, pattern bất thường, top talkers.

**Layer 2 — Metrics**: Prometheus scrape `ovs_exporter` cho metric về:
- Interface statistics (rx/tx packets, bytes, errors, drops).
- Flow count, megaflow count per bridge.
- PMD utilization (DPDK).
- OVSDB commit rate, connection count.

Alert rule điển hình:
- `ovs_interface_rx_dropped` rate > 1% → interface driver issue.
- `ovs_datapath_lost` > 0 → upcall rate limit hit.
- `ovs_bridge_flow_count` > 100,000 → approaching limit.

**Layer 3 — Coverage delta**: cron chạy `ovs-appctl coverage/show` mỗi phút, ghi delta vào log aggregator (Loki, Elasticsearch). Detect anomaly như `cache-misses` rate spike, `bridge_flush` tăng đột biến.

## 9.14.2 Decision tree: 5 loại sự cố

Compass Ch 20 cô đọng:

### 9.14.2.1 Packet loss

```
1. ovs-dpctl show                    → check counter
2. Check "lost" counter               → > 0 = upcall rate limit hit
3. ovs-appctl upcall/show            → xem flow-install rate
4. Action: tăng upcall/set-flow-limit hoặc n-handler-threads
```

Nếu `lost = 0` nhưng vẫn loss → vấn đề kernel hoặc physical NIC.

### 9.14.2.2 Unexpected drops

```
1. ovs-appctl ofproto/trace br0 "<flow spec>"
2. Đọc output: xác định table nào dropping
3. Fix: sửa rule priority hoặc action
```

Debug bằng `ofproto/trace` không cần reproduce traffic thật — scenario quan trọng nhất.

### 9.14.2.3 Tunnel down

```
1. ovs-appctl tnl/neigh/show          → check ARP resolve underlay
2. Underlay connectivity: ping underlay IP
3. MTU mismatch: ip link show, shrink inner MTU hoặc bump underlay
```

95% "tunnel up nhưng no traffic" là MTU issue (VXLAN +50B, Geneve +50-66B).

### 9.14.2.4 Bond flap

```
1. ovs-appctl bond/show bond0
2. ovs-appctl lacp/show bond0
3. Check physical switch LACP config; enable lacp-fallback-ab
```

### 9.14.2.5 OVSDB unavailable

```
1. systemctl status ovsdb-server
2. Clustered: cluster/status → quorum còn không?
3. Restore: restart failed node hoặc restore quorum
```

Cluster 3-node chịu 1 failure, 5-node chịu 2. Mất quorum → cluster read-only.

## 9.14.3 Packet capture strategies

Khi decision tree không đủ:

```bash
[host]# tcpdump -i eth0 -nn -w /tmp/phys.pcap &          # physical
[host]# ovs-vsctl add-port br0 tap-debug -- \
         set Interface tap-debug type=internal
[host]# ovs-vsctl -- --id=@m create Mirror name=dbg \
         select-all=true output-port=@p \
         -- --id=@p get Port tap-debug \
         -- set Bridge br0 mirrors=@m
[host]# tcpdump -i tap-debug -nn -w /tmp/ovs.pcap &      # OVS internal
```

So sánh hai pcap — diff giữa physical wire và OVS internal cho biết packet bị OVS drop hoặc modify.

## 9.14.4 Capstone end-to-end incident

**Scenario**: VM trên host-A không ping được VM trên host-B qua VXLAN, cả hai cùng subnet.

**Step 1** — Physical layer:

```bash
[host-A]# ping <host-B-underlay-ip>
[host-A]# tcpdump -i eth0 -nn udp port 4789
```

Underlay fail → escalate network team.

**Step 2** — OVS config:

```bash
[host-A]# ovs-vsctl show | grep -A5 vxlan
[host-A]# ovs-appctl tnl/neigh/show
```

**Step 3** — OpenFlow pipeline:

```bash
[host-A]# ovs-appctl ofproto/trace br-int "in_port=vm1,..."
```

**Step 4** — Kernel datapath:

```bash
[host-A]# ovs-dpctl dump-flows -m | grep vxlan
[host-A]# ovs-dpctl show | grep lookups
```

**Step 5** — Egress wire:

```bash
[host-A]# tcpdump -i eth0 -nn 'udp port 4789' -w egress.pcap
```

**Step 6** — MTU check:

```bash
[host-A]# ip link show eth0
[host-A]# ip link show vm1-veth
```

Inner MTU ≤ underlay MTU − 50. Nếu 1500 = 1500 → large packet fragment/drop.

## 9.14.5 Post-incident process

Postmortem:

1. **Timeline**: chronological với timestamp.
2. **Root cause**: layer nào.
3. **Evidence**: coverage delta, log, pcap.
4. **Fix applied**: commit/config change.
5. **Prevention**: monitoring alert mới, playbook update.

Postmortem feed knowledge base → giảm MTTR cho incident tương tự.

## Điểm cốt lõi cần nhớ

Thứ nhất, observability stack ba layer (traffic analytics + metrics + coverage delta) là foundation. Không có observability thì debugging chỉ là guess-work.

Thứ hai, 5 loại incident phổ biến có decision tree cụ thể — thuộc thứ tự check giúp MTTR giảm từ giờ xuống phút.

Thứ ba, `ofproto/trace` là công cụ vàng — debug 80% incident không cần reproduce traffic thật.

Thứ tư, postmortem biến incident thành learning material. Team không có postmortem chuyên nghiệp lặp lại incident nhiều lần.

## 9.14.X Missing tool `ovs-bugtool` — diagnostic bundle automation (Phase H S50)

**Audit finding:** trước S50, curriculum 0 mention `ovs-bugtool`. Section này fix gap.

`ovs-bugtool` là utility tự động thu thập diagnostic bundle đầy đủ từ OVS environment. Output là tar.gz chứa 30-50 file: config dumps, flow dumps, log, system info. Dùng khi support vendor cần thông tin đầy đủ hoặc khi escalate issue lên upstream mailing list.

### 9.14.X.1 Basic usage

```shell
# Collect full bundle
$ sudo ovs-bugtool
Running /usr/share/openvswitch/scripts/ovs-save ...
...
/var/log/openvswitch/ovs-bugtool-2026-04-24-11-23-45.tar.gz collected

# Decompress xem nội dung
$ tar tzf /var/log/openvswitch/ovs-bugtool-*.tar.gz | head -20
ovs-bugtool-2026-04-24-11-23-45/commands/
ovs-bugtool-2026-04-24-11-23-45/commands/ovs-vsctl-show
ovs-bugtool-2026-04-24-11-23-45/commands/ovs-ofctl-show
ovs-bugtool-2026-04-24-11-23-45/commands/ovs-ofctl-dump-flows
ovs-bugtool-2026-04-24-11-23-45/commands/ovs-appctl-coverage-show
ovs-bugtool-2026-04-24-11-23-45/commands/ovs-appctl-memory-show
ovs-bugtool-2026-04-24-11-23-45/commands/ovs-dpctl-show
ovs-bugtool-2026-04-24-11-23-45/commands/ip-a
ovs-bugtool-2026-04-24-11-23-45/commands/ip-r
ovs-bugtool-2026-04-24-11-23-45/commands/iptables-save
ovs-bugtool-2026-04-24-11-23-45/commands/lsmod
ovs-bugtool-2026-04-24-11-23-45/commands/uname-a
ovs-bugtool-2026-04-24-11-23-45/log/ovs-vswitchd.log
ovs-bugtool-2026-04-24-11-23-45/log/ovsdb-server.log
ovs-bugtool-2026-04-24-11-23-45/system/meminfo
ovs-bugtool-2026-04-24-11-23-45/system/cpuinfo
...
```

### 9.14.X.2 Anatomy bundle

| Directory | Content |
|---|---|
| `commands/` | Output của 30+ diagnostic command (ovs-vsctl show, dump-flows, coverage, etc.) |
| `log/` | OVS daemon log files (ovs-vswitchd.log, ovsdb-server.log) |
| `system/` | `/proc` snapshots (meminfo, cpuinfo, interrupts, mounts) |
| `network/` | `ip a`, `ip r`, iptables, brctl |

### 9.14.X.3 Selective collection

```shell
# Only collect for specific bridge
$ sudo ovs-bugtool --include-bridge=br-int

# Exclude potentially sensitive flow dump
$ sudo ovs-bugtool --exclude-flow-dump

# Specific output path
$ sudo ovs-bugtool --output=/tmp/ovs-debug.tar.gz
```

### 9.14.X.4 Usage trong incident workflow

```shell
# Step 1: incident xảy ra → collect bundle NGAY (state evidence, không chờ)
$ sudo ovs-bugtool

# Step 2: send bundle + issue description
$ curl -F 'file=@ovs-bugtool-*.tar.gz' https://support.company.com/incident/XXX

# Step 3: parse bundle local cho triage
$ tar xzf ovs-bugtool-*.tar.gz
$ less ovs-bugtool-*/commands/ovs-appctl-coverage-show
$ less ovs-bugtool-*/log/ovs-vswitchd.log
```

### 9.14.X.5 `ovs-pcap` — parse OVS saved packet trace

Companion tool: `ovs-pcap` decode pcap file dùng trong OVS (packet-in log, debug dump).

```shell
# Convert OVS internal packet log → standard pcap
$ sudo ovs-pcap /var/log/openvswitch/packet-in.log > /tmp/packetin.pcap

# Read với tcpdump / wireshark như bình thường
$ tcpdump -r /tmp/packetin.pcap -nne
```

Ít khi dùng trực tiếp — thường là backfill cho forensic reconstruct traffic pattern từ log cũ.

### 9.14.X.6 `ovs-testcontroller` — dummy OpenFlow controller cho testing

Khi test bridge OpenFlow mà không có controller thật (ONOS/ODL):

```shell
# Listen trên port 6633
$ sudo ovs-testcontroller ptcp:6633

# Config bridge trỏ controller test
$ sudo ovs-vsctl set-controller br-test tcp:127.0.0.1:6633

# ovs-testcontroller install basic learn-switch flow — bridge work như L2 switch standard
```

Không dùng cho production — chỉ cho lab + tutorial.

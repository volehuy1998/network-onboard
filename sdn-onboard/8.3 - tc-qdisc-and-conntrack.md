## 8.3.1 tc qdisc taxonomy

Queuing discipline (qdisc) có hai loại:

- **Classful**: có class hierarchy (HTB, HFSC). Filter classify packet vào class.
- **Classless**: không hierarchy (fq_codel, pfifo_fast, red, sfq).

Root qdisc attach vào interface:

```bash
[host]# tc qdisc add dev eth0 root handle 1: htb default 10
[host]# tc class add dev eth0 parent 1: classid 1:1 htb rate 100mbit
[host]# tc class add dev eth0 parent 1:1 classid 1:10 htb rate 50mbit
[host]# tc filter add dev eth0 parent 1: protocol ip u32 \
         match ip dst 10.0.0.1 flowid 1:10
```

Xem qdisc hiện tại:

```bash
[host]# tc qdisc show dev eth0
[host]# tc -s qdisc show dev eth0      # với statistics
```

## 8.3.2 HTB, HFSC, fq_codel

**HTB** (Hierarchical Token Bucket): token bucket với borrow mechanism giữa sibling classes. Class con mượn bandwidth từ class cha khi cha rảnh.

**HFSC** (Hierarchical Fair Service Curve): thời gian thực + link-share + upper-limit curve. Phức tạp hơn HTB nhưng guarantee độ trễ tốt hơn cho thời gian thực traffic.

**fq_codel** (Fair Queue + CoDel AQM): default qdisc từ kernel 5.0+. Giải quyết bufferbloat. Minimal config:

```bash
[host]# tc qdisc replace dev eth0 root fq_codel
```

**pfifo_fast**: classic simple FIFO với 3 priority band. Deprecated — fq_codel thay thế.

CoDel algorithm: track sojourn time của packet trong queue, drop packet nếu sojourn > 5ms persistent. Keep queue ngắn → độ trễ thấp.

## 8.3.3 conntrack: zones, marks, states

Connection tracking kernel module (`nf_conntrack`) track state TCP/UDP/ICMP connection. Foundation cho stateful firewall và NAT.

**State** của connection:

- **NEW**: packet đầu tiên, chưa có entry trong conntrack table.
- **ESTABLISHED**: connection đã hoàn tất handshake (TCP 3-way) hoặc đã thấy 2-way traffic (UDP).
- **RELATED**: ICMP error related đến existing connection, hoặc FTP data connection child của FTP control.
- **INVALID**: packet không khớp với state nào.

Xem conntrack table:

```bash
[host]# cat /proc/net/nf_conntrack | head -5
[host]# conntrack -L                         # cần conntrack-tools package
```

**Zone**: isolate overlapping address space. Nhiều tenant có overlap IP range (10.0.0.0/8) share cùng conntrack table với zone ID riêng:

```bash
[host]# iptables -t raw -A PREROUTING -i ten1 -j CT --zone 1
[host]# iptables -t raw -A PREROUTING -i ten2 -j CT --zone 2
```

**connmark**: mark trên packet propagate vào conntrack entry, phục vụ policy routing + QoS.

## 8.3.4 iptables-nft vs nftables native

Ubuntu 20.04+ ship `iptables-nft` — iptables command với nft kernel backend. Backward compat: rule iptables cũ vẫn hoạt động, kernel path là nftables mới.

Native nftables syntax ngắn gọn hơn:

```bash
# iptables:
iptables -A INPUT -p tcp --dport 22 -j ACCEPT

# nftables equivalent:
nft add rule ip filter input tcp dport 22 accept
```

Migration: `iptables-save | iptables-restore-translate | nft -f -`.

RHEL 9+ default nftables native. Ubuntu 24.04+ cũng chuyển sang nftables native.

## 8.3.5 OVS ct() action tích hợp conntrack

OVS 2.5+ (02/2016) add `ct()` action — OpenFlow rule đẩy packet qua kernel netfilter conntrack module, reuse code path của iptables/nftables. Không reimplement.

Compass Ch 9 canonical 5-flow stateful firewall (verbatim):

```
ovs-ofctl add-flow br0 "table=0,priority=50,ct_state=-trk,tcp,in_port=veth_l0,actions=ct(table=0)"
ovs-ofctl add-flow br0 "table=0,priority=50,ct_state=+trk+new,tcp,in_port=veth_l0,actions=ct(commit),veth_r0"
ovs-ofctl add-flow br0 "table=0,priority=50,ct_state=-trk,tcp,in_port=veth_r0,actions=ct(table=0)"
ovs-ofctl add-flow br0 "table=0,priority=50,ct_state=+trk+est,tcp,in_port=veth_r0,actions=veth_l0"
ovs-ofctl add-flow br0 "table=0,priority=50,ct_state=+trk+est,tcp,in_port=veth_l0,actions=veth_r0"
```

Flag `commit` cài đặt conntrack entry. Flag `zone=N` cho multi-tenant isolation.

`ct_state` match: `+trk` (tracked), `+new`, `+est`, `+rel`, `+inv`, `+snat`, `+dnat` — combine với `+` và `-`.

## 8.3.6 tc qdisc + filter classifier internals

### Kernel queueing path

Khi packet đi qua interface `eth0`, kernel call chain:

```
__dev_queue_xmit()                      [net/core/dev.c]
  └─> qdisc_xmit_skb_skb()
        └─> sch->enqueue()              (ví dụ HTB: htb_enqueue)
              └─> Filter classify       (tc filter rule)
                    └─> Class queue
        └─> qdisc_run()
              └─> sch->dequeue()        (ví dụ HTB: htb_dequeue)
                    └─> ndo_start_xmit()  (driver TX)
```

`sch_*` là module per-qdisc-type:

- `net/sched/sch_htb.c` (HTB)
- `net/sched/sch_hfsc.c` (HFSC)
- `net/sched/sch_fq_codel.c` (fq_codel)
- `net/sched/sch_pfifo_fast.c` (default cũ)

Mỗi module implement `enqueue()` (push packet vào queue) + `dequeue()` (pop packet khi driver ready TX) + `peek()` + `init()` + `destroy()`.

### HTB token bucket math

```c
// net/sched/sch_htb.c, function htb_dequeue()
static struct sk_buff *htb_dequeue(struct Qdisc *sch)
{
    struct htb_class *cl = htb_lookup_leaf(sch);
    if (cl->tokens >= L2T(cl, skb_size)) {
        // Có đủ token, dequeue + trừ token
        cl->tokens -= L2T(cl, skb_size);
        return skb;
    }
    // Không đủ token, mượn từ parent class
    if (cl->cmode == HTB_MAY_BORROW) {
        return htb_dequeue_borrow(cl);
    }
    // Chưa đủ token, throttle
    return NULL;
}
```

`L2T()` (length-to-time) = packet bytes × kernel internal ticks-per-byte. Token bucket fill rate = configured `rate` parameter. Burst capacity = `burst` parameter.

### Filter classifier types

```bash
[host]# tc filter add dev eth0 parent 1: protocol ip u32 \
        match ip dst 10.0.0.1/32 flowid 1:10
```

5 classifier types:

- `u32`: bit-pattern match arbitrary offset trong packet header.
- `flow`: hash-based classification (per-flow load balance).
- `cgroup`: classify theo cgroup v1 net_cls (deprecated cgroup v2).
- `bpf`: eBPF classifier program (modern programmable classifier).
- `matchall`: catch-all without filter logic.

eBPF classifier (TC offload) là path modern cho complex policy:

```bash
[host]# tc filter add dev eth0 parent 1: protocol ip bpf da \
        obj /path/to/classifier.o sec classifier flowid 1:10
```

eBPF program là path mà OVS hardware offload dùng (Phần 9.5 TC offload).

## 8.3.7 nf_conntrack source code tier 2

### State machine implementation

```c
// net/netfilter/nf_conntrack_proto_tcp.c
const struct nf_conntrack_l4proto nf_conntrack_l4proto_tcp = {
    .l4proto = IPPROTO_TCP,
    .pkt_to_tuple = tcp_pkt_to_tuple,
    .invert_tuple = tcp_invert_tuple,
    .new = tcp_new,
    .packet = tcp_packet,
    .timeout = ...,
};
```

Mỗi protocol (TCP, UDP, ICMP) có `nf_conntrack_l4proto` struct với callback. Khi packet đến netfilter hook (PREROUTING):

1. `nf_conntrack_in()` extract tuple (5-tuple, hoặc adjusted cho ICMP).
2. Lookup hash table `nf_conntrack_htable_size` (default 65536 buckets).
3. Hit → existing conntrack entry → run `packet()` callback to update state.
4. Miss → run `new()` callback to create entry → INSERT vào hash.

### TCP state transitions

```
SYN_SENT → SYN_RECV (after SYN+ACK) → ESTABLISHED (after ACK)
ESTABLISHED → FIN_WAIT (after FIN) → CLOSE_WAIT → LAST_ACK → CLOSE
                                               │
                                               └─> TIME_WAIT (active close)
```

Mỗi state có timeout riêng (in `/proc/sys/net/netfilter/nf_conntrack_tcp_timeout_*`):

```bash
[host]# sysctl net.netfilter.nf_conntrack_tcp_timeout_established
432000     # 5 days default
[host]# sysctl net.netfilter.nf_conntrack_tcp_timeout_time_wait
120        # 2 minutes
[host]# sysctl net.netfilter.nf_conntrack_tcp_timeout_close
10         # 10 seconds (after FIN+ACK exchange)
```

### Hash table

`nf_conntrack_htable_size` default 65536 buckets. Mỗi bucket là `hlist_nulls_head`. Lookup O(1) với good hash. Hash function: jhash trên 5-tuple.

Sysctl quan trọng:

```bash
[host]# sysctl net.netfilter.nf_conntrack_max
262144     # max entries in table (default 65536, scaled với memory)
[host]# sysctl net.netfilter.nf_conntrack_buckets
65536
[host]# cat /proc/sys/net/netfilter/nf_conntrack_count
3421       # hiện tại entry count
```

Khi `nf_conntrack_count` đạt `nf_conntrack_max`: kernel drop new connections, log "nf_conntrack: table full, dropping packet". Pre-emptive: monitor + tăng `nf_conntrack_max`.

### Zone implementation

```c
// net/netfilter/nf_conntrack_zones_core.c
struct nf_conntrack_zone {
    u16 id;
    u8 flag;        // NF_CT_FLAG_MARK, NF_CT_DIRECTION_*
    u8 dir;
};
```

Zone embedded trong conntrack tuple key: `(zone, src_ip, dst_ip, src_port, dst_port, proto)`. Khác zone → khác bucket → no collision dù cùng 5-tuple.

OVN dùng zone heavily: mỗi tenant có `ct_zone` riêng để isolate conntrack state. Zone allocate trong `controller/ct-zone.c` của OVN, gán per-LSP qua `lsp-set-options ct_zone=...` (rare, usually auto).

| `requeues` | 0 | < 10/s | > 100/s | Driver TX backpressure |
| `new_flow_count` (fq_codel) | Match application connection rate | Spike | Sustained spike | DDoS hoặc flow churn |
| `ecn_mark` | 0 | > 0 | Sustained | Congestion, ECN-aware app sẽ slow down |

### `conntrack -S` (system-wide stats)

```
[host]# conntrack -S

cpu=1   ...
[total]
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| `found` | Match traffic volume | Drop nếu nf_conntrack_max | (n/a) | Lookup count |
| `invalid` | < 0.1% | 0.1-1% | > 1% | Packet không khớp state; possible scan/spoof |
| `insert_failed` | 0 | > 0 | Sustained | Hash collision; tăng `buckets` |
| `early_drop` | 0 | > 0 | Sustained | Table near full + stochastic eviction |
| `drop` | 0 | > 0 | Sustained | nf_conntrack_max reached |
| `search_restart` | < 0.01% found | 0.01-1% | > 1% | Hash bucket churn; consider larger table |
| `chaintoolong` | 0 | 1-2 chains | Many chains > 8 | Hash collision concentrated |

```bash
[host]# cat /proc/sys/net/netfilter/nf_conntrack_count
[host]# cat /proc/sys/net/netfilter/nf_conntrack_max
```

Healthy ratio `count / max` < 70%. Warning > 80%. Critical > 95% (drop imminent).

## 8.3.9 ▶ Capstone POE: nf_conntrack_max bao nhiêu là đủ cho 100K concurrent connection?

> **Bối cảnh:** OVN cluster với multi-tenant ACL stateful, expected 100K concurrent TCP connection per chassis.

**Tuyên bố cần kiểm chứng:** "Set `nf_conntrack_max=100000` là đủ cho 100K concurrent connection."

**Predict:** Sai. Cần buffer cho TIME_WAIT + multi-zone overhead.

**Observe (giả định):**

- Set `nf_conntrack_max=100000`, generate workload 100K concurrent TCP. Conntrack count đạt ~120K-150K.
- Lý do: mỗi TCP connection có vòng đời:
  - SYN_SENT/SYN_RECV: ~ms
  - ESTABLISHED: connection lifetime (vài giây tới giờ)
  - TIME_WAIT: 120s sau close
  - Plus các ICMP error related (RELATED) + UDP (NEW + state)
- Multi-tenant với zone: mỗi tenant 1 zone, 5K connection per tenant × 20 tenant = 100K. Nhưng zone collision tăng search overhead.
- Result: `count/max` đạt 90%+ → kernel start `early_drop`. Khi peak burst → `drop` thuần → connection failure.

**Explain:**

1. **TIME_WAIT factor:** TCP close sequence để TIME_WAIT 120s default. App churn cao → TIME_WAIT entries > current connection. Multiplier 1.5-2x.
2. **RELATED + ICMP error:** error packet trigger conntrack entry. Mỗi connection trong production có ~0.05-0.1 RELATED entries.
3. **Per-zone overhead:** kernel hash bucket size shared across zones. 20 zones × 5K connection = 100K real entries nhưng search depth đáng kể.
4. **Recommendation:** `nf_conntrack_max = 2-3x expected concurrent`. Cho 100K target → set 250K-300K.

**Falsification test:**

- Set `nf_conntrack_max=300000`, run iperf3 với 100K parallel connection.
- Monitor `count`, `drop`, `early_drop`. Nếu `drop > 0` → tăng max.
- Đo `search_restart` rate. Nếu > 1% → tăng `nf_conntrack_buckets` (default = max/4).

**Tuyên bố sai ở đâu:** "100K connection cần 100K max" sai về buffer. Đúng phải: **`nf_conntrack_max = 2-3x expected concurrent` for TIME_WAIT + RELATED + multi-zone overhead. Plus monitor `early_drop` + `search_restart` để tune thêm.**

> **Quy tắc:** Conntrack max = 2-3x peak concurrent. Buckets = max/4. Monitor count/max ratio, alert khi > 80%.

## Điểm cốt lõi cần nhớ

Thứ nhất, fq_codel là default qdisc từ kernel 5.0+, giải quyết bufferbloat. HTB cho bandwidth limit explicit, HFSC cho thời gian thực guarantee.

Thứ hai, conntrack state machine (NEW/ESTABLISHED/RELATED/INVALID) là foundation cho stateful firewall. Zone cho multi-tenant isolation.

Thứ ba, OVS `ct()` action reuse kernel conntrack — không reimplement. Sysctl `net.netfilter.nf_conntrack_max` áp dụng cả iptables và OVS.

Thứ tư, Ubuntu 20.04+ và RHEL 9+ chuyển sang nftables backend. Migrate khi có cơ hội sang native nft syntax.

## 11.2.1 Flood-and-learn: the problem with native VXLAN

VXLAN (RFC 7348) does not define a control plane. The VTEP must:

1. Flood unknown unicast and broadcast through a multicast group.
2. Learn the MAC-to-VTEP binding from the source.
3. Age out the entry after a timeout.

Problems:

- **Multicast underlay requirement:** many DCs do not deploy multicast.
- **Flood overhead:** the broadcast domain is large, traffic is replicated.
- **Does not scale:** every VTEP holds a MAC table covering every MAC in the fabric.

---

## 11.2.2 BGP EVPN (RFC 7432, February 2015)

This extends BGP to advertise MAC and IP bindings between VTEPs. It is a deterministic push-through-BGP approach that replaces flood-and-learn.

Architecture:

- Each VTEP runs a BGP speaker.
- A VTEP learns the local MAC, then advertises it through a BGP EVPN route.
- Remote VTEPs install the route and do not need to flood.

A BGP EVPN route uses **SAFI 70 (EVPN)**. The NLRI contains the Ethernet Segment Identifier, the Ethernet Tag, the MAC, the IP, and the VTEP IP.

---

## 11.2.3 The 5 EVPN route types

*Table 11-2: EVPN route types (RFC 7432)*

| Type | Name | Purpose |
|---|---|---|
| 1 | Ethernet AutoDiscovery | Per-ES auto-discovery plus per-EVI split horizon |
| 2 | MAC/IP Advertisement | Advertise a MAC (and optionally an IP) to remote VTEPs |
| 3 | Inclusive Multicast | Advertise a VTEP so it joins the BUM replication group |
| 4 | Ethernet Segment | Multi-homing ES membership |
| 5 | IP Prefix Advertisement | Pure L3 (RFC 9136) |

**Type 2** is the most important. Most deployments use only Type 2 plus Type 3.

---

## 11.2.4 VXLAN plus EVPN workflow

**Bring-up:**

1. VTEP-A learns MAC `aa:bb:cc:00:00:01` on a local port in VLAN 10.
2. VTEP-A sends a BGP UPDATE Type 2: `{RD: VTEP-A-IP:10, MAC: aa:bb:..., IP: 10.0.0.1, VNI: 1000, VTEP: VTEP-A-IP}`.
3. The route reflector reflects the UPDATE to every other VTEP.
4. VTEP-B installs: "MAC aa:bb:... is reachable through VXLAN to VTEP-A-IP, VNI 1000".
5. A packet from host B to host A is encapsulated as VXLAN VNI 1000, with remote VTEP-A-IP.

The traffic flow is pure unicast, with no flooding.

## 11.2.6 OVN versus BGP EVPN

OVN (Block XIII) **does not use BGP EVPN** in its main control plane (NBDB to northd to SBDB). OVN has its own control plane that synchronizes MAC and IP bindings through `Port_Binding` rows. However, modern OVN (22.06 and later) added BGP EVPN integration for two specific use cases (see В§11.2.10).

- **OVN core:** cloud platforms (OpenStack, Kubernetes) get tight integration with the orchestrator. Control plane = NBDB intent plus northd compile plus SBDB output.
- **BGP EVPN plus VXLAN:** campus and enterprise fabrics get multi-vendor support. Control plane = BGP route exchange.
- **OVN plus BGP EVPN integration (22.06 and later):** a hybrid scenario (see В§11.2.10).

## 11.2.7 EVPN Type 2 NLRI byte by byte (RFC 7432 В§7.2)

Type 2 (MAC/IP Advertisement) is the most important route type. It advertises the MAC plus IP binding to remote VTEPs.

### NLRI structure

```
+---------------------------------------+
| RD (8 byte)                           |  вҶҗ Route Distinguisher, defined per VRF
+---------------------------------------+
| ESI (10 byte)                         |  вҶҗ Ethernet Segment Identifier (multi-homing)
+---------------------------------------+
| Ethernet Tag ID (4 byte)              |  вҶҗ VLAN ID, or 0 if single-tag
+---------------------------------------+
| MAC Address Length (1 byte) = 48      |
+---------------------------------------+
| MAC Address (6 byte)                  |  вҶҗ Host MAC
+---------------------------------------+
| IP Address Length (1 byte) = 0/32/128 |  вҶҗ 0 if no IP, 32 IPv4, 128 IPv6
+---------------------------------------+
| IP Address (0/4/16 byte)              |  вҶҗ IPv4/IPv6 (optional)
+---------------------------------------+
| MPLS Label1 (3 byte)                  |  вҶҗ VNI for VXLAN-EVPN (RFC 8365)
+---------------------------------------+
| MPLS Label2 (3 byte, optional)        |  вҶҗ L3 VNI (for symmetric IRB)
+---------------------------------------+
```

### Route Distinguisher (RD)

The RD separates multi-tenant routes: each VRF or EVI has a unique RD. Format: `<ASN>:<unique-id>` or `<router-id>:<unique-id>`. It is typically auto-derived as `<VTEP-IP>:<VNI>`.

### Ethernet Segment Identifier (ESI)

ESI = 0 for a single-homed host (the default). For a multi-homed host (active-active LAG to two leaves), the ESI is a 10-byte unique value configured identically on both leaves.

### VNI in MPLS Label1

RFC 8365 redefines the MPLS Label field as a 24-bit VNI for VXLAN. Encoding: 3 bytes = VNI (24 bits). When a VTEP receives the route, it looks up the VNI in the VRF table and knows which VNI to use for VXLAN encapsulation.

### Wire format example

VTEP-A (192.0.2.10) advertises MAC `aa:bb:cc:00:00:01` with IP `10.0.0.1` for VNI 1000:

```
RD: 192.0.2.10:1000          (8 byte: 00 14 c0 00 02 0a 03 e8)
ESI: 0x0...0                  (10 byte zero)
Ethernet Tag: 0               (4 byte: 00 00 00 00)
MAC Length: 48                (1 byte: 30)
MAC: aa:bb:cc:00:00:01        (6 byte)
IP Length: 32                 (1 byte: 20)
IP: 10.0.0.1                  (4 byte: 0a 00 00 01)
MPLS Label1: VNI 1000         (3 byte: 00 03 e8 hex)
```

Total NLRI = 33 bytes. Plus BGP Path Attributes (next-hop = VTEP IP, route-target communities), the total UPDATE message is about 80 to 120 bytes per route.

## 11.2.8 Type 3 plus Type 4 plus Type 5 deep

### Type 3, Inclusive Multicast Ethernet Tag

Each VTEP advertises its presence in one EVI (Ethernet VPN Instance). When BUM (Broadcast, Unknown unicast, Multicast) traffic appears, the source VTEP looks up every Type 3 route that matches the EVI, builds a replication list, and replicates the packet as unicast to every destination VTEP.

NLRI:

```
RD (8B) + Ethernet Tag (4B) + IP Address Length (1B=32) + Originator Router IP (4B)
```

Plus a PMSI Tunnel Attribute that defines the replication mode:

- **Ingress Replication (IR):** the source VTEP replicates as unicast (the default for VXLAN-EVPN).
- **PIM-SM or PIM-SSM:** uses an underlay multicast group (requires the underlay to support PIM).

Most deployments use IR because the underlay is simpler.

### Type 4, Ethernet Segment Route

Multi-home one host to two leaves through LACP. The ESI is shared. Type 4 advertises membership in the ES.

Use case: leaf-2 announces "I also belong to ES X". Multi-leaf forwarding rules apply, with split-horizon (do not forward back to the source ES) plus DF (Designated Forwarder) election so that only one leaf forwards BUM.

This is complex. Typically only large enterprises use it. SMB and cloud DCs are usually single-homed (Type 1 plus 2 plus 3 are sufficient).

### Type 5, IP Prefix Advertisement (RFC 9136, October 2021)

Advertise an IP prefix (route) through EVPN, instead of a host route in Type 2. Use case: external prefix injection into the EVPN fabric.

```
RD (8B) + ESI (10B) + Ethernet Tag (4B) + IP Prefix Length (1B) + IP Prefix (variable) + GW IP (variable) + MPLS Label (3B)
```

Symmetric IRB (Integrated Routing and Bridging) uses Type 5 for L3 forwarding between subnets in the fabric.

## 11.2.9 Symmetric versus Asymmetric IRB

IRB is the mechanism for routing between two subnets in the same fabric (cross-VLAN routing). Two main modes:

### Asymmetric IRB

- The source leaf routes plus bridges.
- The destination leaf only bridges.
- Each leaf must know the routes of every VRF, so memory consumption is O(VRF x prefix).

Workflow:

1. Host A in subnet-1 sends to leaf-1 (the gateway of subnet-1).
2. Leaf-1 routes: it looks up destination 10.2.0.5 (subnet-2), the next-hop is host MAC C, then encapsulates VXLAN with VNI subnet-2.
3. Leaf-3 receives VXLAN VNI subnet-2, bridges to host C.

Problem: leaf-1 must install every subnet-2 route in the VRF.

### Symmetric IRB (preferred)

- The source leaf routes locally in the VRF and encapsulates VXLAN with an **L3 VNI** (transit VNI).
- Transit through the underlay uses the L3 VNI (Type 5 route).
- The destination leaf decapsulates, looks up the L3 VNI, routes within the destination VRF, then bridges as the final step.

Workflow (with two VNIs):

1. Host A in subnet-1 (VNI 1000) sends to leaf-1.
2. Leaf-1: routes locally in the "tenant" VRF, then encapsulates VXLAN with the **L3 VNI** = 5000.
3. Leaf-3: decapsulates VXLAN VNI 5000, looks up the "tenant" VRF, finds subnet-2, encapsulates intra-leaf with VNI 2000, then bridges to host C.

Memory: each leaf only needs host routes for its local subnets plus the L3 VNI route. This scales better.

In 2026, most deployments use Symmetric IRB. Type 2 carries MPLS Label2 = L3 VNI for symmetric mode.

## 11.2.10 OVN plus BGP EVPN integration (22.06 and later)

OVN core does not need BGP EVPN inside the cluster. However, two use cases call for integration:

### Use case 1: External BGP gateway

The OVN cluster needs to advertise prefixes to an external network (campus, WAN, Internet). The OVN gateway router chassis runs FRR plus BGP and advertises the prefix:

```
NB DB:
  Logical_Router (router1)
    options:dynamic-routing=true
    
ovn-northd compiles into:
  - Static route in NBDB вҶ’ Logical_Flow in SBDB
  - LRP advertises the prefix through BGP
```

OVN 22.06 and later add BGP redistribution from NBDB to FRR on the gateway chassis. The internal control plane is still OVN; only the external announcement uses BGP. This is not BGP EVPN, only regular BGP unicast with an external peer.

### Use case 2: OVN-BGP-Agent (hybrid with an external EVPN fabric)

[OVN BGP Agent](https://opendev.org/x/ovn-bgp-agent) is a Red Hat-led project (2022 and later) that lets an OVN cluster integrate with a BGP EVPN fabric:

- A VM in the OVN cluster can be advertised as a Type 2 EVPN route into the EVPN fabric.
- A VM can receive traffic from an external VTEP (not an OVN chassis) through VXLAN-EVPN.
- Use case: hybrid cloud, where an OVN tenant needs to connect to a legacy on-prem EVPN fabric.

This differs from OVN core: the agent reads OVN SBDB Port_Binding rows and generates BGP EVPN Type 2 advertisements. The OVN chassis effectively becomes a VTEP that speaks EVPN.

> **Note:** OVN-BGP-Agent is a community project, not yet upstreamed into OVN core. Production deployments must evaluate maturity. Recommendation: do not use it for mission-critical deployments without lab validation.

> **Context when run:** A chassis running FRR with BGP EVPN (use case 1 or 2). Check the routes received and advertised.

```
[chassis-A]# vtysh -c "show bgp l2vpn evpn route"

   Network          Next Hop            Metric LocPrf Weight Path
Route Distinguisher: 192.0.2.20:1000
*>i[2]:[0]:[0]:[48]:[aa:bb:cc:00:00:02]:[32]:[10.0.0.2]
                    192.0.2.20             0    100      0 i
                    Extended Community: ET:8 RT:65000:1000
*>i[3]:[0]:[32]:[192.0.2.20]
                    192.0.2.20             0    100      0 i
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| Route prefix `[2]:...` | Type 2 routes for known MAC/IP | Type 2 missing | (n/a) | The remote VTEP has not advertised. Check the BGP session. |
| Route prefix `[3]:...` | Type 3 for every remote VTEP | Type 3 missing for a VTEP | (n/a) | A VTEP is not joined to the EVI. Check the configuration. |
| `Next Hop` IP | Matches the remote VTEP IP | Self-IP (loop) | Empty | BGP next-hop misconfiguration. |
| `LocPrf` (Local Pref) | 100 default | Less than 100 (lower priority) | (n/a) | Path manipulation. Check the inbound policy. |
| Path | `i` (iBGP) or `e` (eBGP) | (n/a) | Empty path attribute | BGP attribute corruption. |
| `RT:<ASN>:<VNI>` (Route Target) | Matches the expected RT per VNI | Mismatch | Missing | Tenant isolation is broken. |
| `ET:8` (Encap Type 8) | 8 = VXLAN | 0 = MPLS | 11 = NVGRE | VTEP encap type mismatch. |

**Failure scenarios:**

- The Type 2 route count is lower than the expected MAC count: BGP UPDATE is filtered inbound. Check `show bgp l2vpn evpn route detail` for the specific MAC.
- Route distinguisher conflict (two VTEPs share the same RD): tenant data leak. Ensure that the RD is unique per VTEP.
- Encap Type mismatch (VTEP advertises VXLAN but receives MPLS-EVPN): the configured mode is wrong (RFC 8365 versus native MPLS RFC 7432).

## 11.2.12 в–¶ Capstone POE: BGP EVPN for an OVN cluster, yes or no?

> **Context:** A production OVN cluster with 50 chassis and 100 multi-tenant VRFs. An architect proposes adding BGP EVPN as the control plane "to scale better".

**Claim to verify:** "Adding BGP EVPN to the OVN cluster will scale better than OVN's flood-and-learn."

**Predict:** Wrong. OVN does not use flood-and-learn. OVN has its own control plane (NBDB plus northd plus SBDB).

**Observe (assumed, to be verified in lab):**

- Set up a healthy OVN cluster, with only NBDB plus SBDB as the control plane: 50 chassis, 10K LSP, 100 VRFs. Convergence is below 5 seconds for an NB change. The `ovn-controller` memory per chassis is about 200 MB.
- Add FRR plus OVN-BGP-Agent: this adds about 100 MB of memory per chassis for FRR plus the BGP EVPN cache. Latency for an external prefix advertisement is 30 to 60 seconds (BGP keepalive interval).
- Convergence for a Port_Binding migration cross-chassis is equivalent (both use OVSDB inside the cluster). BGP EVPN adds no value inside the cluster.

**Explain:**

1. **OVN does not use flood-and-learn.** The MAC and IP bindings are pushed by the CMS through NBDB `Logical_Switch_Port.addresses`, `ovn-northd` compiles into SBDB Port_Binding, and `ovn-controller` installs flows on `br-int`. There is no "learn from data plane" the way a traditional VTEP works.
2. **BGP EVPN solves the flood-and-learn limitation,** which does not apply to OVN. Adding BGP EVPN inside the cluster duplicates the control plane, creating two sources of truth, which is conflict-prone and increases complexity.
3. **BGP EVPN has value for OVN in two specific use cases** (Part 11.2.10): external advertisement (FRR plus regular BGP, not EVPN) and hybrid integration with an external EVPN fabric (OVN-BGP-Agent). Neither replaces the OVN core control plane.

**Falsification test:**

- Measure scale and latency for two setups: OVN-only versus OVN plus BGP EVPN inside the cluster.
- If BGP EVPN improves a metric by more than 10 percent, it has value. Otherwise it does not.
- In practice, no metric shows that intra-cluster BGP EVPN improves anything for OVN.

**Where the claim is wrong:** "Scales better than flood-and-learn" assumes OVN uses flood-and-learn. The correct framing is: **OVN has no intra-cluster flood-and-learn. BGP EVPN adds overhead with no benefit inside the cluster. Use BGP EVPN only when external integration is required (Use case 1 or 2 in В§11.2.10).**

> **Rule:** OVN core = NBDB plus SBDB plus northd. BGP EVPN is not needed. BGP EVPN for OVN is only for external or hybrid integration.

## Key points to remember

First, BGP EVPN solves the flood-and-learn limitation of native VXLAN. It is a deterministic control plane that replaces the multicast dependency.

Second, there are 5 route types. Type 2 plus Type 3 are the most common. Type 5 is for pure L3 VPN.

Third, modern leaf-spine DCs use BGP EVPN as the standard. It scales to 1000+ leaves.

Fourth, OVN does not use BGP EVPN. It has its own SBDB. The choice depends on orchestrator integration versus multi-vendor support.

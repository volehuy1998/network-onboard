A traditional Cisco engineer who has set up a SPAN session knows the
problem: an analyst, an IDS, or a packet recorder needs to see the
traffic flowing through a switch port without affecting that traffic. On
Cisco IOS this is `monitor session 1 source interface gi0/1 both;
monitor session 1 destination interface gi0/24`, two CLI lines that
cause the switch's ASIC to copy every frame entering or leaving gi0/1
to the additional egress port gi0/24. The mirrored frames are not
forwarded to the original destination through gi0/24; they are
exclusively forwarded to the destination port for the analyst to
consume. The analyst sees a perfect copy of the production traffic and
can analyse it without being inline.

This lab proves the same idea on two implementations. First, OVS using
the OVSDB `Mirror` table with `select_src_port` plus `output_port`
columns. Second, the Linux kernel's traffic-control framework using the
`mirred` action attached to an ingress qdisc. The pedagogical goal is
threefold: (a) understand the OVSDB mirror configuration model, which
is one of the cleanest in OVS because of its multi-row transactional
flavour; (b) understand the in-kernel `tc mirred` mechanism, which is
the Linux-side toolkit that does the same thing without a userspace
daemon; (c) recognise the wire-byte equivalence that proves both
implementations follow the same conceptual operation.

## R2.5.5.1. Where this part sits in the curriculum and why

R2.5.5 is the fifth sub-capture of R2.5. Like the previous four, the
structure is "do it on OVS, do it on Linux, observe the result is
byte-equivalent". The previous four covered MAC learning (R2.5.1),
VLAN access and trunk (R2.5.2), spanning tree (R2.5.3), and link
aggregation (R2.5.4). This one covers port mirroring, the third
read-only observability primitive in the suite (the first two being the
FDB and the spanning-tree state).

The OVS Mirror configuration model is one of the more interesting
OVSDB entries because it uses cross-row references that demand a
multi-row transactional `ovs-vsctl` invocation rather than a single
`set` call. The reason is structural: a `Mirror` row references one or
more `Port` rows (`select_src_port`, `select_dst_port`, `output_port`)
and is itself referenced from a `Bridge` row (`Bridge.mirrors`). The
two reference levels mean the operator must give names to row
identifiers within a single transaction so all the references can be
resolved together. The `--id=@vsrc get Port veth_src_br` syntax used
in this lab is `ovs-vsctl`'s way of expressing the same thing as a SQL
common-table expression: name a row, refer to it later in the same
transaction.

The Linux side uses an entirely different mechanism. The `tc` traffic-
control subsystem is a packet-pipeline framework in the kernel that
allows arbitrary actions to be applied to packets as they ingress or
egress an interface. The `mirred` action is one of those actions; it
copies the packet to another interface. The configuration is per-port
(attached to a specific interface's ingress or egress qdisc) and uses
filters to decide which packets to mirror. Compared to the OVS Mirror
table, `tc mirred` is more flexible (any tc filter can match on any
packet attribute) and more low-level (the operator writes the filter
match rules by hand).

The R2.5.5 source anchors live in three places. First, the per-mirror
state machine is in `ofproto/ofproto-dpif-mirror.c` (537 lines at OVS
v2.17.9, commit `0bea06d9957e3966d94c48873cd9afefba1c2677`). Second,
the OVSDB-to-runtime configuration plane is in
`vswitchd/bridge.c:bridge_configure_mirrors` and `mirror_configure`.
Third, the translation engine that injects the mirror copy as an
extra `output` action is in `ofproto/ofproto-dpif-xlate.c:output_normal`
plus the per-port mirror walk inside the NORMAL action. The Linux
counterparts are in `net/sched/act_mirred.c` (the action) and
`net/sched/cls_u32.c` (the filter classifier the lab uses).

## R2.5.5.2. Topology

The OVS topology is one bridge with three netns peers. ns_src is the
traffic source, ns_dst is the traffic destination, ns_mon is the
packet-capture point. Each namespace has its own veth pair into the
bridge.

```ascii
   ns_src                         ns_dst                       ns_mon
   10.0.0.1/24                    10.0.0.2/24                  (no IP, capture-only)
       |                              |                                 |
   [veth_src]                    [veth_dst]                       [veth_mon]
       |                              |                                 |
   ----+------------------------------+---------------------------------+----
       |                              |                                 |
   veth_src_br                  veth_dst_br                       veth_mon_br
   (port 1, OF)                 (port 2, OF)                      (port 3, OF)
       |                              |                                 |
       +------------------+-----------+----------------+----------------+
                          |                            |
                          |     OVS bridge br0,        |
                          |     Mirror configured to   |
                          |     copy veth_src_br traffic
                          |     to veth_mon_br         |
                          +----------------------------+
```

The Linux topology is structurally identical (three veth pairs, three
netns peers, one bridge `lbr0`) but the bridge is the in-kernel Linux
bridge and the mirroring mechanism is `tc mirred` rather than OVS's
Mirror table.

## R2.5.5.3. The OVS Mirror configuration model

The OVSDB `Mirror` table has the following relevant columns:

| Column            | Type             | Meaning                                                            |
|-------------------|------------------|--------------------------------------------------------------------|
| `name`            | string           | Mirror identifier within the bridge. Required.                     |
| `select_src_port` | set of Port refs | Ports whose ingress traffic is mirrored. Optional.                 |
| `select_dst_port` | set of Port refs | Ports whose egress traffic is mirrored. Optional.                  |
| `select_all`      | boolean          | If true, every port is mirrored (overrides src/dst selectors).     |
| `select_vlan`     | set of integers  | Only mirror frames in these VLAN IDs. Optional, default any.       |
| `output_port`     | Port ref         | Where to send the mirrored frames. Optional, mutually exclusive with `output_vlan`. |
| `output_vlan`     | integer          | Tag mirrored frames with this VLAN ID instead of an output port.   |
| `statistics`      | map (read-only)  | tx_bytes and tx_packets counters.                                  |

Multiple Mirror rows can be configured per bridge; the bridge's
`Bridge.mirrors` column is a set of references to Mirror rows. The
runtime applies each mirror independently: a frame ingressing on a
`select_src_port` is mirrored once per matching Mirror row, so a single
frame may produce multiple copies if multiple mirrors apply.

The runtime introspection commands include `ovs-appctl dpctl/dump-flows`
(the megaflow dump shows the extra output action injected by the mirror)
and the OVSDB read of `Bridge.mirrors` plus `Mirror.statistics`.

### ▶ Guided Exercise 2: Configure a Mirror with select_src_port

This exercise creates a Mirror row that copies all ingress traffic on
`veth_src_br` to `veth_mon_br`. The configuration uses `ovs-vsctl`'s
multi-row transaction syntax with named row identifiers.

**Outcomes.** A `Mirror` row exists with `name=mirror1`,
`select_src_port=[veth_src_br's UUID]`, `output_port=veth_mon_br's
UUID`. The bridge's `mirrors` column references the new Mirror.
A subsequent ICMP ping from ns_src to ns_dst is observed on ns_mon as
ICMP echo request frames.

**Before You Begin.** Exercise 1 has completed.

**Instructions.**

**1.** Configure the Mirror via a single multi-row transaction. The
syntax `--id=@<name>` declares a temporary identifier within the
transaction; `get Port veth_src_br` looks up an existing Port row;
`create Mirror name=mirror1 select_src_port=@vsrc output_port=@vmon`
creates a new Mirror row referencing the previously-named ports;
`set Bridge br0 mirrors=@m` adds the new Mirror to the bridge.

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl -- --id=@vsrc get Port veth_src_br -- --id=@vmon get Port veth_mon_br -- --id=@m create Mirror name=mirror1 select_src_port=@vsrc output_port=@vmon -- set Bridge br0 mirrors=@m
708e1dff-c293-40c6-b016-104f6038e885
```

The single output line is the UUID of the newly-created Mirror row,
returned by the `create` portion of the transaction. The order of the
sub-commands matters: every `--id` lookup must come BEFORE the row is
referenced in a subsequent sub-command. An attempt to put `create
Mirror ... select_src_port=@vsrc` before `--id=@vsrc get Port ...`
produces the error `row id "@vsrc" specified on "get" command was used
before it was defined`. This was an empirical correction during the
lab capture.

**2.** Read back the Mirror row to confirm.

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=name,select_src_port,select_dst_port,output_port,output_vlan list mirror mirror1
name                : mirror1
select_src_port     : [32868a80-3727-47d2-963f-7646039ee5a5]
select_dst_port     : []
output_port         : 973179b3-2e3c-47c3-bb32-61df6de8a9cf
output_vlan         : []
```

The fields are exactly as configured. `select_src_port` is a set
containing one UUID (the one for `veth_src_br`'s Port row).
`select_dst_port` is empty (we did not configure egress mirroring).
`output_port` is a single UUID (the one for `veth_mon_br`'s Port row;
the column is a single reference, not a set, because a Mirror has at
most one output port). `output_vlan` is empty.

**3.** Read back the bridge's mirrors set.

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=name,mirrors list bridge br0
name                : br0
mirrors             : [708e1dff-c293-40c6-b016-104f6038e885]
```

The bridge's `mirrors` column carries one UUID, matching the Mirror's
own UUID returned in step 1. The bridge is now linked to the Mirror.

**4.** Flush the kernel datapath flow cache. This is empirically
necessary: pre-mirror traffic between ns_src and ns_dst left megaflow
entries in the kernel cache that do NOT include the mirror's extra
output action. Without flushing, the new ping uses the cached flow and
the mirror is bypassed. The flush forces the kernel to upcall on the
next packet, giving userspace a chance to install a new megaflow with
the mirror's output action.

```text
root@lab-openvswitch:~/network-onboard# ovs-appctl list-commands | grep -i mirror | head -5
root@lab-openvswitch:~/network-onboard# ovs-appctl dpctl/del-flows
```

The `list-commands | grep -i mirror` returns no output (no commands
matching "mirror"), which is informational; OVS's appctl introspection
of mirrors goes through the OVSDB `Mirror.statistics` column, not a
dedicated appctl command. The `dpctl/del-flows` invalidates all kernel
megaflows, forcing a fresh upcall on the next packet.

**5.** Capture on ns_mon while sending fresh ICMP. Predict: this time
ns_mon should see the ICMP traffic, because the mirror is configured
to copy ingress on veth_src_br to egress on veth_mon_br.

```text

```

The empirical confirmation: ns_mon now sees `2 packets captured`. The
captured frames are exactly the two ICMP echo REQUESTS from
10.0.0.1 to 10.0.0.2 (the source MAC `26:39:fe:7a:ef:b3` is
ns_src's veth_src; the destination MAC `4e:fd:6e:41:bf:1b` is
ns_dst's veth_dst). The reply traffic from 10.0.0.2 to 10.0.0.1 is
NOT in the capture because the mirror was configured to mirror only
INGRESS on veth_src_br (i.e. frames sent FROM ns_src). The reply
frames ingress on veth_dst_br, not veth_src_br, so they are not
selected by the mirror.

This is the canonical empirical confirmation of selective mirroring:
the mirror copies precisely the configured selection (ingress on
veth_src_br) and nothing else.

**Misconception to refute.** "If a frame goes from ns_src to ns_dst,
the mirror copies BOTH directions." Wrong. The mirror is asymmetric
by configuration: `select_src_port=veth_src_br` selects only frames
ingressing on veth_src_br, which is half of the bidirectional flow.
To capture both directions, the operator must configure both
`select_src_port` and `select_dst_port`, or use `select_all=true`,
or configure two Mirror rows.

**Finish.** Mirror works as configured: ingress on veth_src_br is
copied to ns_mon.

ns_mon now sees ALL traffic on the bridge, in both directions, including

**Outcomes.** With `select_all=true`, ns_mon's tcpdump captures both

## R2.5.5.4. The Linux tc mirred mechanism

The Linux kernel's traffic control framework (`tc`) is a pipeline that
applies arbitrary actions to packets at network interfaces. The pipeline
is constructed from three building blocks: **qdiscs** (queueing
disciplines, attached to an interface; root qdiscs handle egress and
the special `ingress` qdisc handles ingress), **classes** (subdivisions
of a qdisc; not used in this lab), and **filters** (rules that match
packets and apply actions). The `mirred` action ("mirror or redirect")
is one of the action types and copies (or redirects) the packet to
another interface.

To set up port mirroring, the operator: (1) attaches an `ingress`
qdisc to the source-side interface, (2) attaches a filter to that
qdisc with a match rule (typically "match all" via the `u32` classifier
with the always-matching pattern `match u32 0 0`), (3) gives the filter
a `mirred egress mirror dev <dst>` action that copies the matched packet
to the destination interface's egress.

The terminology here can confuse the engineer who maps `tc` mental
model onto OVS or Cisco. "Egress mirror" in `tc` does not mean "mirror
the egress traffic"; it means "the mirror action sends the copy out the
egress of the destination interface". So `mirred egress mirror dev
veth_lmon_br` means "send a copy of the matched packet out the egress
direction of veth_lmon_br", which from the lab's perspective causes the
packet to appear on `veth_lmon`'s side of the veth pair (where ns_lmon
captures it). The copy is not bound by the bridge's normal forwarding
rules; it is a direct interface-to-interface copy.

The filter classifier `u32` is the universal 32-bit-pattern classifier;
`u32 match u32 0 0` is the universal "match every packet" pattern. The
classifier offers far more sophisticated patterns (match on IP source,
TCP destination port, etc.), but for full mirroring the simplest pattern
suffices.

Read the output carefully. The `tc qdisc show` reports two qdiscs on
`veth_lsrc_br`: the default `noqueue` root qdisc (which handles egress;
this is the kernel's default for veth and other simple interface types)
and the new `ingress ffff:` qdisc that we added. The `ffff:fff1` is a
synthetic parent identifier the kernel assigns to ingress qdiscs.

The `tc filter show` output is more complex. The `protocol all pref
49152 u32 chain 0` line is the filter's general info: `protocol all`
matches every protocol, `pref 49152` is the priority (higher = lower
preference; this is the default `u32` priority), `u32` is the
classifier type, `chain 0` is the filter chain. The next two lines
describe the filter's hash-table internals (`u32` uses a per-prefix
hash table for fast matching; `ht divisor 1` is a one-bucket trivial
hash). The `fh 800::800 order 2048 ... terminal flowid ??? not_in_hw`
line is the filter handle; `terminal` means this filter does not
fallthrough to the next; `not_in_hw` means the filter is implemented in
software (no hardware offload).

The `match 00000000/00000000 at 0` line is the match rule: at offset 0
in the packet, match value 0 (i.e. anything) with mask 0 (i.e. no bits
matter). This is the "always match" pattern.

The `action order 1: mirred (Egress Mirror to device veth_lmon_br)
pipe` line is the action: mirred to veth_lmon_br with mode "Egress
Mirror". The `pipe` keyword means "after this action, pass the packet
through to the next action or default disposition" (the alternative is
`stolen` which would drop the original after the mirror).

## R2.5.5.5. Side by side: Cisco SPAN, OVS Mirror, Linux tc mirred

| Concern                            | Cisco IOS                                              | OVS                                                       | Linux tc mirred                              |
|------------------------------------|--------------------------------------------------------|-----------------------------------------------------------|----------------------------------------------|
| Configure source-port mirror       | `monitor session 1 source interface gi0/1 both`        | `ovs-vsctl create Mirror name=m1 select_src_port=...` | `tc filter add dev <ifc> parent ffff: u32 ... action mirred egress mirror dev <dst>` |
| Configure destination port         | `monitor session 1 destination interface gi0/24`       | `ovs-vsctl create Mirror ... output_port=...`             | (the `dev <dst>` argument to the mirred action) |
| Show mirror config                 | `show monitor`                                          | `ovs-vsctl --columns=... list mirror m1`                  | `tc filter show dev <ifc> parent ffff:`      |
| Show mirror statistics             | `show monitor session 1`                                | `ovs-vsctl --columns=statistics list mirror m1`           | `tc -s filter show dev <ifc> parent ffff:`   |
| Tag mirrored frames with VLAN      | `monitor session 1 destination interface gi0/24 encapsulation ...` | `ovs-vsctl set mirror m1 output_vlan=N`           | (write a u32 chain that adds a `vlan push` action before mirred) |
| Filter to a single VLAN            | `monitor session 1 source vlan 10`                     | `ovs-vsctl set mirror m1 select_vlan=10`                  | (write a u32 filter that matches dot1q VID) |
| Match arbitrary packet attribute   | n/a (limited)                                           | n/a (use OpenFlow flows instead)                          | (full u32 / flower matching available)       |
| State storage location              | ASIC TCAM plus IOS process memory                     | OVSDB plus `ovs-vswitchd` userspace heap (`struct mirror`)| Kernel `tc` qdisc/filter table              |
| Implementation language             | C plus closed-source vendor SDK                        | OVS userspace C: `ofproto/ofproto-dpif-mirror.c`         | Kernel C: `net/sched/act_mirred.c`           |
| Source-code anchor (mirror logic)   | n/a (closed)                                           | `ofproto/ofproto-dpif-mirror.c:mirror_set` plus `output_normal` mirror walk in `ofproto-dpif-xlate.c` | `net/sched/act_mirred.c:tcf_mirred_act`     |

The key takeaway: all three implementations achieve the same conceptual
operation (copy traffic to a destination interface for analysis). The
configuration model on OVS is closest to Cisco SPAN (per-port selectors,
explicit Mirror entity); the Linux tc mirred model is more fine-grained
but more verbose. The wire bytes of the mirrored frames are byte-equivalent
across all three: the destination port sees a copy of the original frame
exactly as it appeared on the source port.

## R2.5.5.6. Common misconceptions for the audience

**Misconception 1.** "OVS mirroring is invisible to the OpenFlow flow
table." Mostly right, but missing nuance. The Mirror configuration is
NOT expressed as an OpenFlow flow; it lives in the OVSDB `Mirror`
table. However, the runtime injects an extra `output` action into the
NORMAL action's flow translation, so the kernel datapath megaflow does
include the mirror's output port. An operator who wants to see how the
mirror manifests in the kernel can run `ovs-appctl dpctl/dump-flows`
and look for the extra output port in the action list.

**Misconception 2.** "Setting `select_all=true` is the same as configuring
both `select_src_port` and `select_dst_port` to the union of all ports."
Mostly right at the user-visible level. The runtime treats them
equivalently. The OVSDB representation differs: `select_all=true` is
one boolean; the equivalent per-port form would be a set of every Port
UUID on the bridge in both selector columns. The boolean is more compact
and self-updating (when ports are added or removed, the boolean still
applies; the per-port set would have to be manually updated).

**Misconception 3.** "Mirrored frames count toward the source port's
per-port traffic statistics." Wrong. The mirror's `tx_bytes` and
`tx_packets` counters in `Mirror.statistics` track the mirrored copies;
the source port's `Interface.statistics:tx_packets` only includes
genuine outgoing frames, not mirror copies. The destination port's
`Interface.statistics:tx_packets` does include the mirrored copies,
which can be confusing (the destination port appears to be transmitting
without any user-visible traffic source on the bridge).

**Misconception 4.** "If the mirror's destination port is also a
regular bridge port carrying production traffic, the mirror copies will
mix with the production traffic and the analyst will see both."
Correct. This is exactly the trap E2.5 §E.9 calls out (mirror loop with
mirror destination on the same bridge). The mitigation is to dedicate
the destination port to the analyst and do NOT use it for production
traffic. In this lab, ns_mon's veth_mon has no IP and is not used for
anything else; it is purely a capture port.

**Misconception 5.** "tc mirred replaces the original frame at the
destination." Wrong. The `mirror` direction of mirred copies the frame;
the original continues on its forwarding path. The alternative is
`mirred ... redirect dev <dst>`, which moves the frame entirely to the
destination, dropping it from the original path. This lab uses
`mirror`, which is the non-destructive mode equivalent to OVS Mirror.

## R2.5.5.7. Production-readiness assessment

The trade-off cost of port mirroring under sustained load. Each mirrored
frame is duplicated, doubling the per-frame processing cost on the
matched path. For a port mirroring 100 Mbit/s of source traffic to a
local analyst, the bridge processes 200 Mbit/s of total traffic. For a
port mirroring 10 Gbit/s, the bridge processes 20 Gbit/s. On a
software switch like OVS or the Linux bridge, this can saturate CPU
quickly. Mitigation: use `select_src_port` plus `select_vlan` to
narrow the scope; mirror only the production traffic that needs
analysis, not all of it. For very high traffic rates, hardware-offloaded
mirroring (sFlow, NetFlow) is more efficient because it samples instead
of copying every frame.

The failure mode under destination-port saturation. If the analyst's
capture rate exceeds the destination port's line rate, frames are
dropped at the destination. Both OVS and Linux silently drop in this
case; there is no explicit log message. Mitigation: use `tcpdump -B
<buf-mb>` to ensure the libpcap buffer is large enough for bursty
traffic, monitor `Mirror.statistics:tx_bytes` for the rate of mirrored
traffic, and reduce the mirror selection if the rate exceeds the
capture infrastructure's capacity.

The monitoring discipline. Every Mirror should be considered a
production-impacting configuration change, even though it is a
read-only observation tool. The reason is that mirrored traffic
consumes capacity on the destination port and on the bridge's CPU; a
mirror configured for diagnostic purposes and forgotten can degrade
production performance after a routine port saturation. A monitoring
agent should track `Bridge.mirrors` size (how many active mirrors)
and `Mirror.statistics:tx_packets` rate for any active mirror; alerting
on mirrors that have been configured for more than 24 hours catches
the "diagnostic mirror left on" failure mode.

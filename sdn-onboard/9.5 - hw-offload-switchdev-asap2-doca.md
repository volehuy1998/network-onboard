## Mục tiêu bài học

Sau khi hoàn thành phần này, bạn có thể:

1. **Mô tả (Understand)** mô hình ba-tầng DPIF của OVS: Kernel DPIF — DPDK DPIF — DOCA DPIF (NVIDIA giới thiệu 2023) và lý do NVIDIA khuyến nghị OVS-DOCA cho hiệu năng tối đa.
2. **Phân biệt (Analyze)** switchdev mode vs legacy SR-IOV mode — khi nào VF representor netdev được tạo và tại sao đó là điều kiện tiên quyết cho hardware offload.
3. **Liên hệ (Analyze)** ASAP² eSwitch offload ↔ OVS control plane — tại sao control plane OVS không bị sửa đổi khi datapath được offload xuống eSwitch trong NIC subsystem.
4. **Đánh giá (Evaluate)** trade-off giữa hai chế độ steering (SMFS vs DMFS) và hai chế độ vPort match (Metadata vs Legacy) dưới góc nhìn production tuning.
5. **Áp dụng (Apply)** cấu hình OVS-DOCA trên NIC ConnectX-6 Lx/Dx hoặc BlueField-2 trở lên — nếu có lab hardware tương thích.

## Dàn bài

## 9.5.1 Tại sao hardware offload: CPU bottleneck ở 25G/100G+ và sự ra đời của smart NIC/DPU

*Sẽ phát triển: Software OVS bottleneck ở tốc độ cao (25G/100G); CPU cycles per packet; PPS limitation của x86 even with DPDK; sự ra đời của smart NIC (ConnectX-5+) và DPU (BlueField); nhu cầu offload ở cloud-scale và NFV.*

## 9.5.2 Linux switchdev framework: VF representor, devlink, eSwitch abstraction

*Sẽ phát triển: Documentation/networking/switchdev.rst — kernel framework cho HW offload; VF representor netdev (pf0vf0_r) vs VF netdev thật (passed to VM); devlink API (`devlink dev eswitch set pci/0000:3b:00.0 mode switchdev`); sysfs alternative `/sys/class/net/<pf>/compat/devlink/mode`; kernel upstream versions; tc flower offload.*

## 9.5.3 NVIDIA ASAP² và eSwitch: kiến trúc offload + unmodified OVS control plane

*Sẽ phát triển: ASAP² (Accelerated Switching and Packet Processing) technology; eSwitch inside NIC subsystem; data plane offload while control plane unchanged; relationship between OVS userspace (ovs-vswitchd) and hardware eSwitch; mlx5_core driver role; mandatory prerequisite: `hw-offload=true`.*

## 9.5.4 Ba DPIF của OVS: Kernel — DPDK — DOCA (so sánh kiến trúc)

*Sẽ phát triển: DPIF = Datapath Interface; three implementations — Kernel DPIF (default, uses openvswitch.ko), DPDK DPIF (userspace, uses PMD+hugepages), DOCA DPIF (new 2023, uses DOCA Flow library); NVIDIA's explicit recommendation: "DPDK và Kernel DPIFs được duy trì chủ yếu cho backward compatibility, không được cập nhật feature mới"; bảng so sánh feature matrix.*

## 9.5.5 OVS-DOCA internals: DOCA Flow library, pre-allocated offload tables, pipe resize

*Sẽ phát triển: DOCA Flow library — NVIDIA's networking API; pre-allocated offload structures (entries and containers) khi start; `other_config:doca-init=true`; `other_config:doca-congestion-threshold` default 80%; DOCA pipe resize mechanism; counter `doca_pipe_resize` và `doca_pipe_resize_over_10_ms` cho troubleshooting; limitations như không support CT-CT-NAT.*

## 9.5.6 Offload feature coverage: VXLAN encap/decap, CT offload, CT+NAT, VF-LAG, meters (RFC 2697 srTCM), sFlow, mirror

*Sẽ phát triển: VXLAN encap/decap offload (br-phy + br-ovs pattern); CT offload — tracking connections trong hardware; CT+NAT offload (SNAT, DNAT) — exception: ICMP không support; VF-LAG — SR-IOV bonding offload (active-backup, XOR, LACP); OpenFlow meters implementation theo RFC 2697 srTCM; sFlow offload (chỉ internal bridge khi dùng VXLAN); mirror configuration `doca-mirror-max=2048`.*

## 9.5.7 Steering modes (SMFS vs DMFS) và vPort match modes (Metadata vs Legacy) — production tuning

*Sẽ phát triển: SMFS (Software-Managed Flow Steering) default, tối ưu insertion rate; DMFS (Device-Managed Flow Steering) firmware-based, tối ưu throughput khi rule ít; cấu hình qua sysfs hoặc devlink; PHẢI set trước khi chuyển switchdev; vPort match — Metadata (default, support SR-IOV live migration + dual-port RoCE) vs Legacy (higher performance, không support live migration); quyết định nào cho production scenario nào.*

## 9.5.8 vDPA: Software vDPA vs Hardware vDPA — cầu nối virtio portability + SR-IOV performance

*Sẽ phát triển: vDPA (Virtual Data Path Acceleration) — kết hợp SR-IOV performance với virtio portability; `netdev type=dpdkvdpa`; Software vDPA — translation on host; Hardware vDPA — direct hardware support (mặc định nếu NIC support); kiểm tra mode qua `ovs-ofctl dump-ports` tìm `hw-mode flag`; use case: live migration khả thi, VM image generic không bị tied to vendor driver.*

## 9.5.9 BlueField DPU architecture: Arm cores, pf0hpf, embedded OVS, BFB installation

*Sẽ phát triển: BlueField-2/3 DPU — Arm cores + NIC subsystem trên cùng board; pf0hpf interface — host PF representor trên DPU side; default bridges ovsbr1/ovsbr2 created on first boot; BFB (BlueField Bundle) installation; `/sbin/mlnx_bf_configure` auto-enable HW offload; use case: isolation giữa tenant workload (host) và infrastructure services (DPU Arm cores).*

## 9.5.10 Megaflow scaling trong OVS-DOCA: flow-limit default 200k, ct-size default 250k, max 2M connections

*Sẽ phát triển: Default megaflow capacity 200k; `other_config:flow-limit` tuning; `other_config:hw-offload-ct-size` default 250k, max 2M (warning nếu >1M cho best performance); `other_config:max-revalidator` default 250ms; mối quan hệ với max-idle; sysctl `net.netfilter.nf_conntrack_max` đồng thời phải điều chỉnh; devlink `ct_max_offloaded_conns`.*

### ▶ Guided Exercise 1: Xác minh switchdev mode và VF representor bằng devlink + ip link show

*Sẽ phát triển: Unbind VFs → devlink eswitch set mode switchdev → observe VF representor created → verify via `ip -d link show` xem `switchid` và `portname pf0vf0`; POE structure (predict — eSwitch legacy không tạo representor; observe — sau switchdev mode tạo; explain — switchdev framework exposes representor cho OVS/tc flower).*

### ▶ Guided Exercise 2: Đọc offload coverage counters để phát hiện bottleneck

*Sẽ phát triển: `ovs-appctl coverage/show` → đọc `doca_async_queue_full`, `doca_async_queue_blocked`, `doca_pipe_resize`, `doca_pipe_resize_over_10_ms`; giải thích ý nghĩa từng counter (queue_full = slowed hardware, blocked = bug, resize_over_10ms = insertion drop cause); POE structure để chứng minh insertion rate giảm khi pipe resize diễn ra.*

### ▶ Lab: So sánh throughput OVS-Kernel vs OVS-DPDK vs OVS-DOCA cùng topology (điều kiện NIC ConnectX/BlueField)

*Sẽ phát triển: Lab chỉ cung cấp kịch bản và gợi ý — user tự build topology 3-bridge, switch giữa 3 DPIF, đo throughput bằng iperf3 hoặc TRex, verify offload rate qua `ovs-appctl dpctl/dump-flows type=offloaded`; Evaluation: kỳ vọng OVS-DOCA > OVS-DPDK > OVS-Kernel ở 25G+ tốc độ; nếu không có NIC hỗ trợ, chạy chỉ Kernel vs DPDK và note limitation.*

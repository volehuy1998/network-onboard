## What R2.8 establishes, end to end

### The five claims R2.8 proves on the lab host

R2.8 uses one continuous lab session to prove five claims about how OVS handles L2 MAC-port bindings. Each claim is paired with an explicit observation in the verbatim transcript.

Claim 1, the byte-for-byte binding picture. For each of two namespaces and their bridge-side veth peers, the kernel netdev's `link/ether`, the OVSDB Interface row's `mac_in_use`, and the FDB entry the bridge learns from a real ping all agree byte for byte. The lab uses deterministic MACs (`02:00:00:00:0a:0a` for ns-a's veth, `02:00:00:00:0a:0b` for the bridge-side peer, `02:00:00:00:0b:0a` for ns-b's veth, `02:00:00:00:0b:0b` for ns-b's bridge-side peer) so the reader can read each value at each observation surface and confirm equality.

Claim 2, the negative claim about bridge-side MACs. The bridge's L2 learning code learns the *source MAC of every frame*, not the MAC of the netdev the frame arrived on. Since the veth driver pass-through preserves the source MAC, the bridge sees `02:00:00:00:0a:0a` (namespace-side) as the source on every frame from `ns-a`, never `02:00:00:00:0a:0b` (bridge-side). The lab proves this by running a real ping and grepping the FDB for the bridge-side MACs; the grep returns no match.

Claim 3, the trace's forward-or-flood discrimination. `ofproto/trace` against a learned destination produces a single-port output (`Datapath actions: 3`) and the narrator emits `forwarding to learned port`. The same trace against an unlearned destination produces a flood (`Datapath actions: 1,3`) and the narrator emits `no learned MAC for destination, flooding`. The two traces side by side are the cleanest possible demonstration that the FDB drives the forwarding decision.

Claim 4, the FDB-versus-megaflow-cache independence. `ovs-appctl fdb/flush br0` empties the daemon-side FDB but does not touch the kernel megaflow cache. The lab counts IPv4 megaflows before the flush (2), runs the flush, counts again (2 still). The kernel cache survives.

Claim 5, the real-frame MAC-move learning. A Python AF_PACKET injector sends one Ethernet frame from inside `ns-b` on `veth-b` with source MAC `02:00:00:00:0a:0a` (ns-a's MAC, deliberately spoofed). The bridge sees this frame as ingress on OF port 2 with a source MAC that was previously learned on OF port 1, and moves the FDB binding to port 2. A subsequent ping from `ns-a` (real frames on port 1) moves the binding back to port 1. The MAC-move learning is unconditional and immediate.

### Why this matters

The five claims together close the L2-learning chapter of the OVS bridge primitive. R2.0 introduced the architecture, R2.1 toggled fail-mode, R2.2 traced a frame end to end, R2.3 distinguished port types, R2.4 proved the patch-port crossing, R2.6 split the datapath, R2.7 nailed down the OVSDB UUID semantics, and R2.8 ties everything to the empirical L2-learning behaviour the rest of the curriculum will rely on. A reader who has worked through R2 in sequence has a complete operational mental model of an OVS bridge, with every claim backed by a verbatim observation on the lab host. The Cisco-trained engineer who started R2.0 unable to draw the four-piece architecture should now be able to predict the FDB state after any sequence of frames, predict the megaflow cache invalidation behaviour for any operator action, and design a debugging procedure for any L2-forwarding incident.

## Build the topology with deterministic MACs

### From zero to a connected two-namespace L2 bridge

```text
root@lab-openvswitch:~/network-onboard# date -u
Thu Apr 30 05:23:00 AM UTC 2026
root@lab-openvswitch:~/network-onboard# ovs-vsctl --may-exist add-br br0
root@lab-openvswitch:~/network-onboard# ip netns add ns-a
root@lab-openvswitch:~/network-onboard# ip netns add ns-b
root@lab-openvswitch:~/network-onboard# ip link add veth-a type veth peer name br-a
root@lab-openvswitch:~/network-onboard# ip link add veth-b type veth peer name br-b
root@lab-openvswitch:~/network-onboard# ip link set dev veth-a address 02:00:00:00:0a:0a
root@lab-openvswitch:~/network-onboard# ip link set dev br-a address 02:00:00:00:0a:0b
root@lab-openvswitch:~/network-onboard# ip link set dev veth-b address 02:00:00:00:0b:0a
root@lab-openvswitch:~/network-onboard# ip link set dev br-b address 02:00:00:00:0b:0b
root@lab-openvswitch:~/network-onboard# ip link set veth-a netns ns-a
root@lab-openvswitch:~/network-onboard# ip link set veth-b netns ns-b
root@lab-openvswitch:~/network-onboard# ip netns exec ns-a ip addr add 10.50.0.10/24 dev veth-a
root@lab-openvswitch:~/network-onboard# ip netns exec ns-b ip addr add 10.50.0.20/24 dev veth-b
root@lab-openvswitch:~/network-onboard# ip netns exec ns-a ip link set lo up
root@lab-openvswitch:~/network-onboard# ip netns exec ns-b ip link set lo up
root@lab-openvswitch:~/network-onboard# ip netns exec ns-a ip link set veth-a up
root@lab-openvswitch:~/network-onboard# ip netns exec ns-b ip link set veth-b up
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br0 br-a
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br0 br-b
root@lab-openvswitch:~/network-onboard# ip link set br-a up
root@lab-openvswitch:~/network-onboard# ip link set br-b up
root@lab-openvswitch:~/network-onboard# ip link set br0 up
```

The opening 23 commands build the topology. Twelve of them are the standard two-namespace pattern from R2.2 (create namespaces, create veth pairs, move into namespaces, IP-and-up). Four are deterministic MAC assignments: `02:00:00:00:0a:0a` for `ns-a`'s veth (set before the move into `ns-a`, while the netdev is still in the root namespace), `02:00:00:00:0a:0b` for the bridge-side `br-a`, `02:00:00:00:0b:0a` for `ns-b`'s veth, `02:00:00:00:0b:0b` for `br-b`. The naming convention encodes the namespace letter in the fifth byte and the role (peer-namespace `0a` or peer-bridge `0b`) in the sixth byte; a reader scanning `fdb/show` output later can immediately identify which side of which veth pair a MAC belongs to.

The order of operations matters. The MAC must be set *before* the veth end is moved into a namespace; once moved, the netdev is no longer in the root namespace and `ip link set dev veth-a address ...` from the root shell would fail. The lab's orchestrator follows this order strictly. The reader who scripts this pattern should adopt the same order.

The seven remaining commands are the bridge-side wiring: two `ovs-vsctl add-port` to claim the bridge ends for OVS, three `ip link set ... up` to bring up the bridge ends and the bridge LOCAL. After these 23 commands, the topology is complete: `ns-a` (10.50.0.10) connected via `veth-a`-`br-a` to `br0`, and `ns-b` (10.50.0.20) connected via `veth-b`-`br-b` to `br0`.

## Claim 1: byte-for-byte binding agreement

### Namespace-side `link/ether`

```text
root@lab-openvswitch:~/network-onboard# ip netns exec ns-a ip link show veth-a
161: veth-a@if160: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 qdisc noqueue state UP mode DEFAULT group default qlen 1000
    link/ether 02:00:00:00:0a:0a brd ff:ff:ff:ff:ff:ff link-netnsid 0
root@lab-openvswitch:~/network-onboard# ip netns exec ns-b ip link show veth-b
163: veth-b@if162: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 qdisc noqueue state UP mode DEFAULT group default qlen 1000
    link/ether 02:00:00:00:0b:0a brd ff:ff:ff:ff:ff:ff link-netnsid 0
```

Inside each namespace, `ip link show <veth>` reports the kernel's view. `ns-a`'s `veth-a` has `link/ether 02:00:00:00:0a:0a` and ifindex 161 with peer at ifindex 160. `ns-b`'s `veth-b` has `link/ether 02:00:00:00:0b:0a` and ifindex 163 with peer at ifindex 162. Both values match exactly what the operator set before the namespace move. The `link-netnsid 0` field is the kernel's per-namespace identifier for the peer's namespace; value 0 is conventionally the parent (root) namespace, which is correct because both peers live in the host's root namespace.

### Bridge-side `link/ether` and the master flag

```text
root@lab-openvswitch:~/network-onboard# ip link show br-a
160: br-a@if161: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 qdisc noqueue master ovs-system state UP mode DEFAULT group default qlen 1000
    link/ether 02:00:00:00:0a:0b brd ff:ff:ff:ff:ff:ff link-netns ns-a
root@lab-openvswitch:~/network-onboard# ip link show br-b
162: br-b@if163: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 qdisc noqueue master ovs-system state UP mode DEFAULT group default qlen 1000
    link/ether 02:00:00:00:0b:0b brd ff:ff:ff:ff:ff:ff link-netns ns-b
```

In the root namespace, the bridge-side ends report `link/ether 02:00:00:00:0a:0b` (`br-a`) and `02:00:00:00:0b:0b` (`br-b`), at ifindexes 160 and 162. Each one's `@if<N>` matches the namespace-side ifindex. The `master ovs-system` flag indicates the kernel datapath has claimed these netdevs (R2.3 introduced this flag). The `link-netns ns-a` and `link-netns ns-b` fields are the kernel's record of which namespace each peer lives in.

### OVSDB-side `mac_in_use`

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=name,ofport,mac_in_use list Interface br-a br-b br0
name                : br-a
ofport              : 1
mac_in_use          : "02:00:00:00:0a:0b"

name                : br-b
ofport              : 2
mac_in_use          : "02:00:00:00:0b:0b"

name                : br0
ofport              : 65534
mac_in_use          : "b6:06:b1:ca:d3:44"
```

The OVSDB Interface rows report `mac_in_use : "02:00:00:00:0a:0b"` for `br-a` and `mac_in_use : "02:00:00:00:0b:0b"` for `br-b`. These match the kernel's `link/ether` reports byte for byte. The agreement is the proof that the OVSDB column tracks the kernel-side reality through the daemon's IDL. The bridge LOCAL `br0` has its own kernel-assigned MAC `b6:06:b1:ca:d3:44` because the operator did not pin the LOCAL's MAC; the kernel allocated a random one at netdev-creation time.

### Datapath-side `dpif/show` port mapping

```text
root@lab-openvswitch:~/network-onboard# ovs-appctl dpif/show
system@ovs-system: hit:9 missed:9
  br0:
    br-a 1/2: (system)
    br-b 2/3: (system)
    br0 65534/1: (internal)
```

`br-a` is at OF 1 / DP 2, `br-b` is at OF 2 / DP 3, `br0` LOCAL is at OF 65534 / DP 1. The OF numbers match the OVSDB `ofport` values. The DP numbers are the kernel datapath's vport-creation order: 1 was the LOCAL, 2 was the first add-port (`br-a`), 3 was the second (`br-b`).

### FDB before any operator-driven traffic

```text
root@lab-openvswitch:~/network-onboard# ovs-appctl fdb/show br0
 port  VLAN  MAC                Age
    1     0  02:00:00:00:0a:0a    2
LOCAL     0  b6:06:b1:ca:d3:44    1
    2     0  02:00:00:00:0b:0a    1
```

Three FDB entries. The bridge has already learned the namespace-side MACs (`02:00:00:00:0a:0a` against OF port 1, `02:00:00:00:0b:0a` against OF port 2) plus its own LOCAL. The MACs were learned from the IPv6 link-local autoconfiguration multicast traffic that the kernel's IPv6 stack emitted as soon as each veth end came up. The reader who has read R2.2 will recognise this; the IPv6 chatter does L2 learning before the operator sends the first frame.

The FDB entries against ports 1 and 2 are the *namespace-side* MACs, not the bridge-side MACs. The bridge learns the source MAC of every frame; the source of a frame coming out of `ns-a` through `veth-a` and into the bridge-side `br-a` is `02:00:00:00:0a:0a` (the namespace-side MAC). The bridge-side MAC `02:00:00:00:0a:0b` is the peer end's address; the bridge does not see frames *with that source MAC* arriving on its side because the veth driver pass-through preserves the source MAC of the original sender. This is the property R2.8 will prove rigorously below.

## Run a real ping and read the post-traffic FDB

### The three-packet ping with mid-traffic command queueing

```text
root@lab-openvswitch:~/network-onboard# ip netns exec ns-a ping -c 3 -W 2 10.50.0.20
PING 10.50.0.20 (10.50.0.20) 56(84) bytes of data.

64 bytes from 10.50.0.20: icmp_seq=2 ttl=64 time=0.049 ms
ovs-appctl fdb/show br0

rtt min/avg/max/mdev = 0.049/0.655/1.865/0.855 ms
```

A three-packet ping. The first round trip took 1.87 ms (the upcall-driven cold cache path R2.2 explained); the subsequent two were 0.049 ms and 0.052 ms (warm cache). The ratio of cold-to-warm is more than 35x.

The reader will notice an extra line in the middle of the ping output: `ovs-appctl fdb/show br0`. This is the *next operator command* the orchestrator typed during the ping's wall-clock window. The tmux feed-keys raced with the ping's per-packet inter-arrival timer; the shell echoed the typed command, but bash had not yet read it as input because the ping was still running in the foreground. After the ping completed, bash processed the queued command. The result is verbatim what the operator saw on the screen, captured by `script` exactly as it was rendered. Per Rule 18 absolute integrity, the lab transcript preserves the race; the curriculum's job is to explain it, not to hide it.

### The FDB after the ping

## Claim 2: the bridge does not learn bridge-side veth MACs

```text
root@lab-openvswitch:~/network-onboard# ovs-appctl fdb/show br0 | grep -E '02:00:00:00:0a:0b|02:00:00:00:0b:0b' || echo '(neither bridge-side MAC is in FDB; the bridge does not learn bridge-side veth MACs because they never appear as a frame source)'
(neither bridge-side MAC is in FDB; the bridge does not learn bridge-side veth MACs because they never appear as a frame source)
```

The grep on the FDB returns no match for either bridge-side MAC. The `||` branch runs and prints the verdict. Claim 2 is empirically supported.

The mechanism, restated. The veth driver forwards a frame from one peer to the other without rewriting the source MAC. A frame originating in `ns-a` with source `02:00:00:00:0a:0a` arrives at `br-a` with the same source MAC. The bridge's L2 learning code (`xlate_normal()` in `ofproto/ofproto-dpif-xlate.c`, calling `mac_learning_insert()` in `lib/mac-learning.c`) inserts the source MAC against the input OF port. The bridge has no occasion to see `02:00:00:00:0a:0b` as a source MAC because the kernel does not synthesise frames from `br-a` with that MAC as source; if it did (for example, an ARP from the host's IP stack out of `br-a`), the bridge would learn it, but no such frame is generated in this lab.

The pedagogical implication. The reader who is debugging a "MAC not in FDB" question on a veth-connected port should ask which side of the veth the MAC belongs to. Bridge-side MACs are normally not in the FDB. Namespace-side MACs are. R5.5 will revisit this in the inter-host context, where the source MAC of a frame arriving over a tunnel is the *remote* VM's MAC, learned against the local tunnel port.

```text
 0. priority 0
    NORMAL

Final flow: unchanged
Megaflow: recirc_id=0,eth,ip,in_port=1,dl_src=02:00:00:00:0a:0a,dl_dst=02:00:00:00:0b:0a,nw_frag=no
```

A simulated ICMP echo from `ns-a` to `ns-b`. The destination `02:00:00:00:0b:0a` is a learned MAC (the FDB has it on OF port 2). The trace narrator reports `forwarding to learned port` instead of `flooding`. The single Datapath action is `3` (DP port 3, `br-b`); no flood, just direct output.


Same source MAC, same input port, different destination MAC `02:00:00:00:de:ad`. The destination was never learned. The trace narrator reports `no learned MAC for destination, flooding` and produces actions `1,3` (LOCAL plus the other namespace's bridge-side end), skipping the input port `2` per the IEEE 802.1D rule.

The two traces side by side are Claim 3 demonstrated. Same input, same source MAC, same EtherType; the only difference is the destination MAC. For a learned destination, single-port output. For an unlearned destination, flood-minus-input-port.

## Claim 4: fdb/flush does not invalidate the kernel megaflow cache

### Count megaflows before flush

```text
root@lab-openvswitch:~/network-onboard# ovs-appctl dpctl/dump-flows | grep -c 'eth_type(0x0800)'
2
```

Two IPv4 megaflows in the kernel cache. These are the forward and reverse ICMP entries from the ping (one for `ns-a`-to-`ns-b`, one for `ns-b`-to-`ns-a`).

### Flush the FDB

```text
root@lab-openvswitch:~/network-onboard# ovs-appctl fdb/flush br0
table successfully flushed
root@lab-openvswitch:~/network-onboard# ovs-appctl fdb/show br0
 port  VLAN  MAC                Age
```

`fdb/flush br0` empties the daemon-side L2 learning table. The next `fdb/show` returns zero entries. The conjecture is that the kernel megaflow cache is unaffected; let us verify.

### Count megaflows after flush

```text
root@lab-openvswitch:~/network-onboard# ovs-appctl dpctl/dump-flows | grep -c 'eth_type(0x0800)'
2
```

Still two IPv4 megaflows. The `fdb/flush` did not invalidate them. Claim 4 is empirically supported: the FDB and the kernel megaflow cache are independent caches with independent lifetimes.

The implementation. `ovs-appctl fdb/flush` calls `mac_learning_flush()` in `lib/mac-learning.c`; this clears the in-memory `struct mac_learning *ml` for the bridge. The kernel datapath has no `mac_learning` state of its own; the megaflows installed there are concrete actions like `output:DP_port_3` that do not consult any FDB. The two caches operate independently.

The operational consequence. A frame in flight after `fdb/flush` may still hit a cached megaflow and forward correctly; the megaflow cache can continue to serve traffic for ages even if the FDB is empty. The reader who is troubleshooting a "stale forwarding" complaint should clear *both* caches: `ovs-appctl fdb/flush <bridge>` plus `ovs-appctl dpctl/del-flows`.

## Claim 5: a real frame causes a MAC-move

### Re-populate the FDB by waiting for ageing-driven re-learning

```text
root@lab-openvswitch:~/network-onboard# ovs-appctl fdb/show br0
 port  VLAN  MAC                Age
    1     0  02:00:00:00:0a:0a    0
    2     0  02:00:00:00:0b:0a    0
```

Between the previous `fdb/show` (empty after flush) and this one, the kernel's IPv6 chatter and the running FDB-learning side effect of the cached megaflows refilled the table. The two namespace MACs are back, against their correct ports. This is the baseline state for the move proof.

### Inject a frame from veth-b with ns-a's source MAC

```text
root@lab-openvswitch:~/network-onboard# ip netns exec ns-b python3 /tmp/inject.py veth-b 02:00:00:00:0a:0a ff:ff:ff:ff:ff:ff
sent ARP from veth-b src=02:00:00:00:0a:0a dst=ff:ff:ff:ff:ff:ff
root@lab-openvswitch:~/network-onboard# sleep 0.5
```

The `inject.py` script (a 10-line AF_PACKET raw-socket sender, available at `/tmp/inject.py` on the lab host) sends a single Ethernet frame from inside `ns-b` on `veth-b`, with source MAC `02:00:00:00:0a:0a` (ns-a's real MAC, deliberately spoofed) and destination broadcast. The frame's EtherType is 0x0806 (ARP); the payload is a minimal ARP request. The frame transits the veth into `br-b` on the host side, and from there into the bridge as ingress on OF port 2. Since the source MAC matches a MAC the bridge has already learned on a different port (port 1), the L2 learning code's MAC-move logic should trigger.

### Observe the FDB after the move

```text
root@lab-openvswitch:~/network-onboard# ovs-appctl fdb/show br0
 port  VLAN  MAC                Age
    2     0  02:00:00:00:0b:0a    2
    2     0  02:00:00:00:0a:0a    1
```

The FDB now shows `02:00:00:00:0a:0a` against **port 2**, not port 1. The bridge moved the binding because it observed the same MAC arriving on a different port. Claim 5 is empirically supported.

The implementation. `xlate_normal()` calls `update_learning_table_xc()` (visible in `ofproto/ofproto-dpif-xlate.c`), which itself calls `update_learning_table__()` in `lib/mac-learning.c` line 520. The function looks up the source MAC; if the existing entry has a different port, it deletes and re-inserts, effectively moving the binding. The IEEE 802.1D specification mandates this behaviour: a learning bridge must follow the most recent observation when the source moves.

The pedagogical observation. The MAC-move learning is what happens during a VM live-migration in production: the VM's MAC moves from one host to another, frames begin arriving from the new host's tunnel-side veth, and the bridge updates the FDB without operator intervention. R5.5 (inter-host VM-to-VM) will exercise this in the production-flavour multi-host topology; here the property is proven in the simpler single-host case.

## The misdirected reply causes a ping failure

```text
root@lab-openvswitch:~/network-onboard# ip netns exec ns-a ping -c 1 -W 2 10.50.0.20
PING 10.50.0.20 (10.50.0.20) 56(84) bytes of data.
ovs-appctl fdb/show br0

--- 10.50.0.20 ping statistics ---
1 packets transmitted, 0 received, 100% packet loss, time 0ms
```

`ns-a` sends one ICMP echo request to `10.50.0.20`. The frame reaches the bridge as ingress on OF port 1 (correct, source MAC `02:00:00:00:0a:0a` arrives on port 1). The bridge's `xlate_normal()` learning *moves the binding back to port 1* on this frame's ingress (because the MAC is now seen on port 1, having just been on port 2). Then the bridge looks up the destination MAC `02:00:00:00:0b:0a` (ns-b's MAC), finds it on port 2, outputs to port 2.

The ping reports 100% packet loss. The reader can investigate why; the most likely explanation is that the kernel megaflow cache has stale entries from the inject step (the inject's broadcast flood produced a megaflow with `actions:1,2,4` that overlaps the request's match key), and the request's actual path takes time to settle through revalidation. The 100% loss is captured verbatim and is itself a pedagogically useful demonstration: a recently-spoofed MAC topology produces real packet loss until the cache settles. R4 will measure the cache-coherence latency directly.

A second extra `ovs-appctl fdb/show br0` line appears in the middle of the ping output, the same tmux race condition this lab encountered earlier. The bash queued the next command while the ping was running.

### Observe the FDB after the failed ping

```text
root@lab-openvswitch:~/network-onboard# ovs-appctl fdb/show br0
 port  VLAN  MAC                Age
    2     0  02:00:00:00:0b:0a    1
    1     0  02:00:00:00:0a:0a    1
LOCAL     0  b6:06:b1:ca:d3:44    1
```

The FDB now shows `02:00:00:00:0a:0a` on **port 1** again. The MAC moved back. The bridge's L2 learning is unconditional and immediate: a single real frame from `ns-a` (the failed ping's request) was sufficient to re-learn the original binding. Claim 5's converse, the move-back property, is also empirically supported.

The pedagogical implication, important for the reader to internalise. In a production environment, a misconfigured VM that spoofs another VM's MAC on a different port can hijack the binding for as long as the spoof continues. The mitigation is OpenFlow rules with explicit per-port MAC binding (R3 territory) or OVN's port-security feature; the bridge's NORMAL-mode learning is by-design vulnerable to this kind of attack. The reader who knows this property can design defenses; the reader who does not can be surprised.

## Misconception block

> **Misconception / Reality.**
>
> Misconception: "the bridge learns the MAC of the bridge-side veth end (`br-a`'s MAC) when a frame enters."
>
> Reality: the bridge learns the *source MAC of the frame*, not the MAC of the netdev the frame entered through. Claim 2 above is the empirical proof; the FDB grep returns no match for either bridge-side MAC after a real ping. The bridge-side MAC `02:00:00:00:0a:0b` is the netdev's hardware address, used only for frames the kernel synthesises *originating* on the bridge-side end (rare in practice). The reader who confuses the two will be unable to debug L2-learning issues correctly.

> **Misconception / Reality (second).**
>
> Misconception: "`fdb/flush` is the same as `dpctl/del-flows`; both clear the bridge's forwarding state."
>
> Reality: they touch different caches. `fdb/flush` clears the daemon's L2-learning table. `dpctl/del-flows` clears the kernel's megaflow cache. Either can be cleared without the other; Claim 4 above proves the independence. A frame in flight after `fdb/flush` may still hit a cached megaflow and forward correctly without re-learning. The two caches are populated independently and are invalidated independently.

> **Misconception / Reality (third).**
>
> Misconception: "the bridge will not believe a sudden change of port for an existing MAC; some kind of validation prevents MAC-spoofing."
>
> Reality: the bridge believes every observation immediately. Claim 5 above is the empirical proof; a single injected frame with a spoofed source MAC moved the FDB binding to a different port, and a single legitimate frame moved it back. The bridge's NORMAL-mode L2 learning has no built-in validation. Anti-spoofing must be configured separately, either via explicit OpenFlow rules with `dl_src=` matches (R3) or via port-security features that an OVN-style higher-level controller installs.

## A few practical notes

A small note on the `link-netnsid 0` value the kernel reported on the namespace-side ip link show outputs. The `link-netnsid` is a per-namespace integer the kernel assigns when a netdev is moved into a namespace; it is the *peer-relative* id, used by ip-route's namespace-aware routing extensions. The value 0 means "the peer is in the namespace whose id is 0 from this namespace's perspective" (which on Linux is conventionally the parent namespace, the host's root). The reader who is debugging cross-namespace routing can use `ip netns identify <pid>` and the netnsid to map between perspectives; R5.5 will return to this in the multi-host context.

A small note on the experiment-design choice. The deterministic MACs `02:00:00:00:0a:0a` and the variants are chosen with the locally-administered bit set (the `02` prefix). Locally-administered MACs are guaranteed never to collide with any vendor's globally-assigned MAC, which means the lab's MACs cannot accidentally match a real hardware MAC on the host's NIC and confuse subsequent diagnostic. The reader who designs MAC-rich experiments should adopt the same convention; the IEEE 802 standard reserves OUI prefixes ending in `:02:`, `:06:`, `:0a:`, `:0e:` (the second-to-last bit of the first byte set) for locally-administered use.

A reflection on the `Age` column the reader has now seen many times. The Age is in seconds since the last frame from this MAC was observed; ageing is implemented in `lib/mac-learning.c` by a background thread that scans entries against the configured ageing time and evicts those older. The Age column in `fdb/show` is a snapshot value from the moment the daemon rendered the report; reading the same FDB twice in close succession will show the Age values incremented by the elapsed wall time. The reader who watches Age values change across multiple reads has a useful diagnostic for "are frames still flowing through this port" without needing tcpdump.

A note on `Bridge.other_config:mac-aging-time` and `Bridge.other_config:mac-table-size`. The default ageing timer is 300 seconds; the default FDB capacity is 2,048 entries per bridge. Operators who run high-churn environments lower the ageing timer; operators who run static topologies raise it. The capacity knob is rarely touched in production but matters for sizing. R10 will exercise both knobs in the context of high-flow-count workloads.

A practical note on the `--columns` flag and the script-friendly `--bare` mode introduced in R2.7. R2.8 used `--columns=name,ofport,mac_in_use` to pin the three values for each Interface row in one read. The `--columns` flag is essential for stable scripting because `ovs-vsctl list Interface` without it prints every column (more than 30), and the column order is not guaranteed across OVS versions. A script that pins specific columns is robust against schema additions; a script that parses the full `list` output is brittle. R6 onward will rely on `--columns` heavily for stable diagnostic output.

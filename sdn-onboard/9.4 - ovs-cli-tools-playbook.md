## Bối cảnh: Tại sao cần một playbook có hệ thống

OVS là hệ thống phân tầng với ba thành phần lớn (ovs-vswitchd user-space daemon, ovsdb-server database daemon, kernel datapath module hoặc userspace DPDK/AF_XDP datapath) và bốn lớp cấu hình (OVSDB config → OpenFlow pipeline → megaflow cache → wire). Khi sự cố *"packet không đi qua"* xảy ra, cách tiếp cận ad-hoc (gõ lệnh ngẫu nhiên xem có gì không) tốn giờ, gọi ra nhiều tab terminal, và thường không tìm ra root cause.

Playbook có hệ thống giải quyết vấn đề bằng cách đặt ra trình tự kiểm tra cố định. Mỗi bước kiểm tra loại trừ một vùng có thể gây lỗi. Compass Appendix C đúc kết triết lý thành sáu câu hỏi:

1. Packet có đến NIC không? Đo qua `tcpdump` trên physical NIC.
2. Packet có vào OVS không? Đo qua counter `lookups:missed` tăng khi có traffic, bằng `ovs-dpctl show`.
3. Flow nào khớp với packet? Dùng `ovs-appctl ofproto/trace` mô phỏng pipeline.
4. Datapath action đúng không? Đọc dòng `Datapath actions:` trong theo dõi output.
5. Datapath có agree không? Dùng `ovs-dpctl dump-flows` đọc kernel/netdev thực.
6. Wire có agree không? `tcpdump` trên egress NIC.

Nếu sáu bước đều agree mà packet vẫn drop, root cause nằm **ngoài OVS**: upstream switch, firewall, MTU mismatch, ARP unresolved, BGP route withdrawn. Không tiếp tục debug OVS — xoay sang network path.

Phần này trình bày năm công cụ CLI với depth đủ để thực thi playbook, sau đó tích hợp thành §9.4.6 6-layer troubleshooting workflow. Cấu trúc mỗi section công cụ: grammar tổng thể → subcommand catalog → Anatomy block cho output quan trọng → 2-4 scenario operational → kịch bản bẻ gãy.

## 9.4.1 `ovs-vsctl`, transactional configuration language

### 9.4.1.1 Grammar, atomicity, và tại sao mọi thay đổi config đi qua ovs-vsctl

`ovs-vsctl` là công cụ quản lý OVSDB config. Mọi thay đổi *cấu hình* trên bridge, port, interface, QoS, mirror, netflow, controller, OpenFlow version — đều đi qua đây. Compass Ch A nêu bản chất cốt lõi mà nhiều kỹ sư mới bỏ qua:

> *"`ovs-vsctl` không phải shell bao quanh các lệnh đơn lẻ. Nó là transactional compiler: một invocation duy nhất trở thành một OVSDB transaction, và `--` là statement separator bên trong transaction đó."*

Nói cách khác, `ovs-vsctl -- A -- B -- C` biên dịch thành một transaction đơn nguyên với ba thao tác, commit atomic. Nếu thao tác B fail, A không được apply. Đây là bảo vệ quan trọng khỏi half-configured state mà thiếu nó, production system có thể rơi vào state: bridge đã tạo nhưng port chưa gắn → traffic rớt trong lúc config đang rollout.

Grammar tổng thể (man `ovs-vsctl(8)` §SYNOPSIS):

```
ovs-vsctl [global-options] command [-- command]...

global-options:
  --db=socket-path      (default: /var/run/openvswitch/db.sock)
  --timeout=seconds     (default: infinite)
  --no-wait
  --dry-run
  --oneline             (compact output)
  --pretty              (pretty-print JSON-like values)

command:
  <verb> <table> [<record>] [<column>[:<key>][=<value>]]...

verbs for scalar/composite column:
  list    find   get   set   add   remove   clear

verbs for row:
  create   destroy

helpers:
  --id=@name            (bind record to symbolic name in transaction)
  --may-exist           (succeed if row already matches)
  --if-exists           (succeed if row missing)
```

Reference cross-table trong cùng transaction dùng `--id=@name`. Ví dụ kinh điển (tạo bridge + add port trong một transaction):

```shell
$ ovs-vsctl add-br br0 -- add-port br0 eth0
```

Nếu `add-port` fail (eth0 đã thuộc bridge khác), `br0` không được tạo. Không có Windows state trung gian nửa-tạo.

### 9.4.1.2 Catalog subcommand nhóm bridge/port/interface

| Nhóm | Command | Syntax | Mục đích | Atomic? |
|---|---|---|---|---|
| Bridge lifecycle | `add-br` | `ovs-vsctl add-br BRIDGE [PARENT VLAN] [-- set Bridge BRIDGE datapath_type=...]` | Tạo bridge mới với datapath_type optional (`system` / `netdev`) | Yes |
| | `del-br` | `ovs-vsctl [--if-exists] del-br BRIDGE` | Xóa bridge và mọi port thuộc nó | Yes |
| | `br-exists` | `ovs-vsctl br-exists BRIDGE` | Exit 0 nếu tồn tại, 2 nếu không; script-friendly | - |
| | `list-br` | `ovs-vsctl list-br` | Liệt kê toàn bộ bridge (bao gồm fake bridge VLAN) | - |
| | `br-to-vlan` | `ovs-vsctl br-to-vlan BRIDGE` | Return VLAN ID nếu là fake bridge, 0 nếu real | - |
| | `br-to-parent` | `ovs-vsctl br-to-parent BRIDGE` | Return tên parent nếu fake, chính nó nếu real | - |
| Port management | `add-port` | `ovs-vsctl add-port BRIDGE PORT [COLUMN=VALUE]...` | Gắn port vào bridge; trunk/access/bond đều qua column | Yes |
| | `del-port` | `ovs-vsctl [--if-exists] [--with-iface] del-port [BRIDGE] PORT` | Tháo port khỏi bridge | Yes |
| | `list-ports` | `ovs-vsctl list-ports BRIDGE` | Liệt kê port (trừ local port cùng tên bridge) | - |
| | `port-to-br` | `ovs-vsctl port-to-br PORT` | Return bridge chứa port | - |
| Interface | `list-ifaces` | `ovs-vsctl list-ifaces BRIDGE` | Liệt kê interface (trừ local) | - |
| | `iface-to-br` | `ovs-vsctl iface-to-br IFACE` | Return bridge chứa interface | - |
| Controller | `set-controller` | `ovs-vsctl set-controller BRIDGE TARGET [TARGET]...` | Gắn bridge vào OpenFlow controller (ssl:host:port / tcp:host:port / unix:path) | Yes |
| | `del-controller` | `ovs-vsctl del-controller BRIDGE` | Tháo controller; bridge fall back về `fail_mode` | Yes |
| | `get-controller` | `ovs-vsctl get-controller BRIDGE` | Liệt kê controller đang gắn | - |
| | `set-fail-mode` | `ovs-vsctl set-fail-mode BRIDGE MODE` | `standalone` (tự forward) hoặc `secure` (drop mọi packet không có rule) | Yes |
| Database core | `list` | `ovs-vsctl list TABLE [RECORD]...` | In ra mọi column của record, format table | - |
| | `find` | `ovs-vsctl find TABLE [COLUMN[:KEY]=VALUE]...` | Filter rows theo điều kiện column | - |
| | `get` | `ovs-vsctl get TABLE RECORD [COLUMN[:KEY]]...` | Lấy giá trị cụ thể một column | - |
| | `set` | `ovs-vsctl set TABLE RECORD COLUMN[:KEY]=VALUE...` | Gán giá trị (atomic với other set trong cùng transaction) | Yes |
| | `add` | `ovs-vsctl add TABLE RECORD COLUMN [KEY=]VALUE...` | Append vào set/map column | Yes |
| | `remove` | `ovs-vsctl remove TABLE RECORD COLUMN VALUE|KEY[=VALUE]...` | Xóa entry khỏi set/map | Yes |
| | `clear` | `ovs-vsctl clear TABLE RECORD COLUMN...` | Reset column về empty set/map | Yes |
| | `create` | `ovs-vsctl create TABLE [COLUMN[:KEY]=VALUE]...` | Tạo row mới; return UUID | Yes |
| | `destroy` | `ovs-vsctl destroy TABLE RECORD...` | Xóa row | Yes |

Total: 30+ subcommand. Đa phần là convenience wrapper cho database core (`list`/`set`/`get`), nhưng bridge-level shortcut (`add-br`/`add-port`) phổ biến hơn vì rõ nghĩa.

**Anatomy từng dòng:**

| Dòng output | Ý nghĩa | Dấu hiệu bất thường |
|---|---|---|
| `c5e7a4c2-…` (dòng đầu) | UUID của record Open_vSwitch, unique per host; không bao giờ trùng | Mất UUID → `ovsdb-server` chưa load schema, check `journalctl -u ovsdb-server` |
| `Bridge br-int` | Một bridge entry, mỗi bridge render một block | Không có `br-int` sau OVN setup → `ovn-controller` chưa khởi động |
| `fail_mode: secure` | Khi controller disconnect: `secure` (drop mọi packet không khớp với flow) hoặc `standalone` (learn-normal). `secure` là chuẩn OVN | `standalone` trên br-int OVN → lộ lỗ hổng khi OVN controller down, traffic flood |
| `datapath_type: system` | Kernel datapath (mặc định). `netdev` là userspace DPDK/AF_XDP | `netdev` nếu không config DPDK → OVS fallback AF_XDP, performance thấp |
| `Port <name>` | Port entry thuộc bridge hiện tại | - |
| `Interface <name>` | Interface entry thuộc port hiện tại (thường 1:1; bond có N interface trên 1 port) | - |
| `type: internal` | OVS-managed TAP; có `internal` port linux netdev | - |
| `type: geneve` | Overlay tunnel; port được OVN dựng tự động | Thiếu option `key=flow` → OVN không inject tun_id per packet, overlay hỏng |
| `options: {csum="true", key=flow, remote_ip="10.200.0.12"}` | Tunnel options: csum (UDP checksum), key=flow (tun_id từ flow action), remote_ip endpoint xa | `key=<integer>` thay vì `key=flow` → OVN không dùng được (OVN cần per-flow tun_id) |
| `tag: 100` | Access VLAN tag trên port; packet vào port này có `vlan_tci` set 100 | Tag sai hoặc thiếu → traffic nhảy broadcast domain sai |
| `ovs_version: "2.17.9"` | Phiên bản OVS user-space | < 2.17 có thể không hỗ trợ một số OpenFlow 1.5 feature; < 2.15 không có SMC cache |

**Kịch bản bẻ gãy:**

- **Nếu dòng `ovs_version` không hiển thị** → `ovsdb-server` không chạy hoặc không response trong timeout; chạy `sudo systemctl status ovsdb-server` và `sudo ovs-vsctl --timeout=3 show`.
- **Nếu bridge có `datapath_type: netdev` mà host không cấu hình DPDK** → OVS fallback kernel datapath qua AF_XDP, throughput có thể giảm 30-40%; fix bằng `ovs-vsctl set Bridge br-int datapath_type=system`.
- **Nếu port `ovn-*` vắng mặt sau khi OVN đã provision LSP** → `ovn-controller` chưa pick up binding; kiểm tra `ovn-sbctl show` xem chassis đã register chưa và `ovn-sbctl list Port_Binding` xem binding có chassis gán chưa.

### 9.4.1.4 Scenario operational, tạo bridge với internal port có IP

**Bối cảnh:** dựng host development OVS, cần bridge với VLAN access port làm gateway nội bộ. Đây là mẫu phổ biến trong kolla-ansible khi config br-ex.

```shell
# Step 1: tạo bridge
$ sudo ovs-vsctl add-br br-demo

# Step 2: verify
$ sudo ovs-vsctl list-br
br-demo
br-ex
br-int

# Step 3: add VLAN access port tag=10 với internal interface
$ sudo ovs-vsctl add-port br-demo vlan10 tag=10 -- set Interface vlan10 type=internal

# Step 4: gán IP linux-netdev cho internal port
$ sudo ip link set vlan10 up
$ sudo ip addr add 192.168.10.1/24 dev vlan10

# Step 5: verify cả ba thao tác cùng apply atomic
$ sudo ovs-vsctl list Port vlan10
_uuid               : 8f3a2b4c-...
name                : vlan10
tag                 : 10
interfaces          : [1a2b3c4d-...]
...

$ sudo ovs-vsctl list Interface vlan10
_uuid               : 1a2b3c4d-...
name                : vlan10
type                : internal
admin_state         : up
link_state          : up
mac_in_use          : "3e:6a:...:..."
mtu                 : 1500
...

# Step 6 (teardown): xóa bridge, mọi port tự detach
$ sudo ovs-vsctl del-br br-demo
```

**Điểm mấu chốt:**

- `tag=10` trong `add-port` set column `tag` của table `Port`. Đây là access VLAN, khác với `trunks=10,20,30` (trunk port).
- `type=internal` của Interface tạo linux-netdev ảo. Không có `internal` thì interface phải là tên device thật (`eth0`, `enp3s0`).
- `add-port` và `set Interface` commit atomic vì cùng `ovs-vsctl` invocation.

### 9.4.1.5 Scenario, cấu hình QoS 2 queue với transaction phức tạp

**Bối cảnh:** lift từ man `ovs-vsctl(8)` §`CONFIGURATION COOKBOOK, Quality of Service`. Tạo QoS record với hai queue để chia băng thông 100 Mbps và 500 Mbps cho eth0 và eth1.

```shell
$ sudo ovs-vsctl \
    -- set Port eth0 qos=@newqos \
    -- set Port eth1 qos=@newqos \
    -- --id=@newqos create QoS type=linux-htb \
         other-config:max-rate=1000000000 \
         queues=0=@q0,1=@q1 \
    -- --id=@q0 create Queue other-config:min-rate=100000000 \
         other-config:max-rate=100000000 \
    -- --id=@q1 create Queue other-config:min-rate=500000000
```

**Anatomy transaction:**

| Dòng | Thao tác | Kết quả OVSDB |
|---|---|---|
| `set Port eth0 qos=@newqos` | Gán QoS (chưa tồn tại, bind symbolic) | Cột `qos` của row Port `eth0` point đến UUID resolved sau |
| `set Port eth1 qos=@newqos` | Gán cùng QoS cho eth1 | Hai port share cùng QoS record |
| `create QoS type=linux-htb` | Tạo QoS record kiểu HTB với max-rate 1 Gbps tổng | `--id=@newqos` bind UUID mới vào symbol `@newqos` |
| `queues=0=@q0,1=@q1` | Cột `queues` (map) gán queue-id 0 → Queue record `@q0`, 1 → `@q1` | Reference cross-table |
| `create Queue` với `min-rate=100M max-rate=100M` | Queue 0 capped 100 Mbps | - |
| `create Queue` với `min-rate=500M` | Queue 1 guarantee 500 Mbps min, không cap | - |

Sáu thao tác commit atomic. Nếu ví dụ `create QoS` fail (syntax lỗi), hai `set Port` không được apply, eth0/eth1 không bị gán QoS broken.

**Kịch bản bẻ gãy:**

- **Queue không cap max-rate** (chỉ có min-rate) → HTB share available bandwidth; có thể vượt xa mức dự định khi port khác idle. Thiết kế có chủ đích nhưng phải document.
- **`max-rate=<min-rate>` của Queue** → không đạt được target max-rate vì min-rate là sàn cứng. Coerced theo HTB algorithm.
- **Bỏ quên clear khi xóa QoS:** phải `ovs-vsctl clear Port eth0 qos` + `destroy QoS <uuid>`. Nếu chỉ `destroy`, reference từ Port tồn tại → OVSDB raise "dangling reference" error.

**Upstream:** man `ovs-vsctl(8)` §`CONFIGURATION COOKBOOK, Quality of Service`.

### 9.4.1.6 Scenario, query `find` + `get` cho script automation

**Bối cảnh:** script giám sát cần pull state định kỳ. `show` không parse-friendly; `list` dài; `find` + `get` là format-machine.

```shell
# Tìm mọi Port có VLAN tag 100
$ sudo ovs-vsctl find Port tag=100
_uuid               : 7c9d4e2a-1f8b-4c3d-a7e6-5b2a9d1e8f3c
name                : "tap-vm-db"
tag                 : 100
interfaces          : [6e1a3b8c-2d4f-4a5e-b8c7-9d3e1a2b7f5c]
...

# Tìm Interface có RX error > 0
$ sudo ovs-vsctl --columns=name,statistics find Interface 'statistics:rx_errors>0'
name                : eth1
statistics          : {..., rx_errors=42, ...}

# Get 1 giá trị scalar (script-friendly)
$ sudo ovs-vsctl get Interface eno1 statistics:rx_bytes
184756234890

# Get giá trị từ nested map
$ sudo ovs-vsctl get Interface ovn-huyvl-0 options:remote_ip
"10.200.0.12"

# Liệt kê bridge + fail_mode + datapath_type dưới dạng table
$ sudo ovs-vsctl --columns=name,fail_mode,datapath_type list Bridge
_uuid               : 3a8b9c4d-...
name                : br-int
fail_mode           : secure
datapath_type       : system

_uuid               : 5e7f8a9b-...
name                : br-ex
fail_mode           : standalone
datapath_type       : system
```

**Anatomy query operator:**

| Operator | Ý nghĩa | Ví dụ |
|---|---|---|
| `COLUMN=VALUE` | Exact match | `tag=100` |
| `COLUMN!=VALUE` | Not equal | `fail_mode!=secure` |
| `COLUMN<VALUE` | Less than (numeric) | `statistics:rx_bytes<1000` |
| `COLUMN>VALUE` | Greater than | `statistics:rx_errors>0` |
| `COLUMN{=}{A,B}` | Set membership | `protocols{=}{OpenFlow13,OpenFlow15}` |
| `COLUMN:KEY=VALUE` | Map entry | `options:remote_ip=10.0.0.1` |

Script production ví dụ — poll RX error rate per interface:

```bash
#!/bin/bash
for iface in $(sudo ovs-vsctl --bare --columns=name list Interface); do
  rx_err=$(sudo ovs-vsctl get Interface "$iface" statistics:rx_errors 2>/dev/null || echo 0)
  rx_bytes=$(sudo ovs-vsctl get Interface "$iface" statistics:rx_bytes 2>/dev/null || echo 0)
  if [ "$rx_err" -gt 0 ]; then
    echo "INTERFACE $iface: rx_errors=$rx_err rx_bytes=$rx_bytes"
  fi
done
```

## 9.4.2 `ovs-ofctl`, OpenFlow flow management

### 9.4.2.1 Grammar, version negotiation, và flow syntax

`ovs-ofctl` nói OpenFlow với bridge: dump flow, thêm/sửa/xóa flow entry, theo dõi event stream. Đây là công cụ **write-heavy** duy nhất cho pipeline programming — `ovs-vsctl` không sửa được flow, chỉ config bridge.

Grammar tổng thể (man `ovs-ofctl(8)` §SYNOPSIS):

```
ovs-ofctl [OPTIONS] COMMAND [SWITCH] [ARGS...]

key OPTIONS:
  -O, --protocols=VERSION[,VERSION...]
       OpenFlow10 | OpenFlow11 | OpenFlow12 | OpenFlow13 | OpenFlow14 | OpenFlow15
  --strict           (match exactly, no wildcards)
  --bundle           (atomic group using OF 1.4 bundle transactions)
  --sort[=FIELD]     (sort dump-flows output)
  --rsort[=FIELD]
  --color[=never|always|auto]
  --names | --no-names (port names vs numbers)

COMMANDS (switch management):
  show SWITCH
  dump-tables SWITCH
  dump-ports SWITCH [NETDEV]
  dump-flows SWITCH [FLOWSPEC]
  dump-aggregate SWITCH [FLOWSPEC]
  add-flow SWITCH FLOW
  add-flows SWITCH FILE
  mod-flows [--strict] SWITCH FLOW
  del-flows [--strict] SWITCH [FLOW]
  replace-flows [--readd] SWITCH FILE
  diff-flows SOURCE1 SOURCE2
  add-group SWITCH GROUP    /  dump-groups SWITCH [GROUP]
  add-meter SWITCH METER    /  dump-meters SWITCH [METER]
  monitor SWITCH [MISS-LEN] [invalid_ttl] [watch:SPEC...]
  snoop SWITCH
```

**Quan trọng:** LUÔN pass `-O OpenFlow13` (hoặc 1.4, 1.5). Default OVS `ovs-ofctl` là OpenFlow 1.0 — version cổ, không hỗ trợ multi-table, group, meter, TLV match, v.v. Không pass → tham chiếu pipeline thất bại trên setup hiện đại (OVN yêu cầu OpenFlow 1.4+).

### 9.4.2.2 Flow syntax, các keyword control

Flow spec là chuỗi `FIELD=VALUE` phân tách bởi comma hoặc whitespace. Khi có space cần quote. Các keyword control quan trọng:

| Keyword | Ý nghĩa | Default | Khi thiếu |
|---|---|---|---|
| `table=<id>` | Giới hạn operation tại table ID (0-254) | 0 với `add-flow`; wildcard với dump/del | Tất cả table |
| `priority=<val>` | Priority 0-65535; cao hơn khớp với trước | 32768 | 32768 (không nằm ở top) |
| `cookie=<val>[/<mask>]` | Opaque 64-bit ID, dùng để group/tag flow | 0 | - |
| `idle_timeout=<sec>` | Flow expire sau N giây không packet hit | 0 (không timeout) | - |
| `hard_timeout=<sec>` | Flow expire sau N giây tuyệt đối | 0 | - |
| `actions=<spec>` | Danh sách action (comma-separated) | Required | Flow drop nếu thiếu |
| `duration=...`, `n_packets=...`, `n_bytes=...` | Ignored trong add/mod (chỉ dump show) | - | - |

Match field xem `ovs-fields(7)`. Ví dụ flow spec phức tạp:

```
table=10,priority=1000,
  cookie=0xdead1000/0xffff0000,
  idle_timeout=60,hard_timeout=600,
  dl_type=0x0800,nw_proto=6,tp_dst=80,
  actions=mod_dl_dst:aa:bb:cc:dd:ee:ff,
          mod_nw_src:10.1.0.100,
          goto_table:20
```

Dòng mạch có: table 10, priority 1000, cookie cao masked 0xffff, idle 60s hard 600s, match TCP destination 80, ba action (rewrite MAC dest, rewrite source IP, goto table 20).

**Anatomy từng dòng:**

| Thành phần | Ý nghĩa | Dấu hiệu bất thường |
|---|---|---|
| `OFPT_FEATURES_REPLY (OF1.5)` | Version negotiation thành công với OF 1.5 | Nếu hiển thị `(OF1.0)` → bạn quên `-O OpenFlow15`, dùng wrong version |
| `dpid:0000000000000001` | Datapath ID 64-bit, unique per switch | Trùng dpid giữa hai bridge → controller mistakenly thấy một switch |
| `n_tables:254` | Số OpenFlow table hỗ trợ. OVS default 254 | < 254 → bridge bị restrict, check `ovs-vsctl get Bridge br-int flow_tables` |
| `n_buffers:0` | OVS không buffer packet_in nữa (OF 1.4+) | - |
| `capabilities` | List cờ switch support (FLOW_STATS, TABLE_STATS, PORT_STATS, GROUP_STATS, QUEUE_STATS, ARP_MATCH_IP) | Thiếu `GROUP_STATS` → Group multicast không accurate counter |
| `actions: (none)` | OF 1.1+ không hard-code action; trả `(none)` | - |
| `1(tap-vm-web)` | Port number `1`, name `tap-vm-web` | Port number disappear → VM unplugged hoặc ovs-vswitchd restart assign lại |
| `addr:3e:6a:...` | MAC của port | MAC 00:... → internal port chưa gán MAC, check Interface `mac` column |
| `config: PORT_DOWN` | Admin down (set bởi `ovs-ofctl mod-port PORT down`) | PORT_DOWN mà traffic expected → user config sai |
| `state: LINK_DOWN` | Physical link layer down | LINK_DOWN trên Geneve `ovn-*` port → bình thường (Geneve không có physical link) |
| `current: 10GB-FD COPPER` | Link speed + duplex + media | SFP module error hiển thị `current:` trống |
| `speed:` | Bandwidth report trong Mbps | Geneve port luôn 0 Mbps (virtual) |
| `LOCAL(br-int)` | Port number `LOCAL` reserved cho bridge internal | Mất LOCAL port → bridge corruption |
| `OFPT_GET_CONFIG_REPLY: frags=normal miss_send_len=0` | Bridge config: fragments reassemble normal; không gửi packet_in header-only lên controller | `miss_send_len=N` → bridge sẽ gửi N byte packet_in, có thể spam controller |

**Kịch bản bẻ gãy:**

- **`state: LINK_DOWN` kéo dài trên port eth0** → vấn đề vật lý (cable, SFP, driver). Dùng `ethtool eth0` kiểm chứng autoneg và media type.
- **Port number không stable giữa các lần `show`** → `ovs-vswitchd` vừa restart; luôn dùng port name (`tap-vm-web`) trong flow spec thay vì port number (`1`) để tránh broken flow sau restart.
- **`(OF1.0)` thay vì `(OF1.5)`** → thiếu flag `-O`. Fix: `ovs-ofctl -O OpenFlow15 show br-int`.

| Field | Ý nghĩa | Dấu hiệu bất thường |
|---|---|---|
| `cookie=<64-bit hex>` | Tag do controller set khi add flow. OVN dùng cookie để tag logical flow UUID (high 32 bit) + datapath (low 32 bit) | `cookie=0x0` là default; OVN-generated flow phải có cookie khác 0 |

| `table=<N>` | Table number 0-254. OF 1.1+ có multi-table | - |
| `n_packets=<N>` | Counter packet khớp với tới flow này | `n_packets=0` + `duration > 600s` với flow expected → flow không bao giờ match, check khớp với spec |
| `n_bytes=<N>` | Counter byte match | Ratio `n_bytes / n_packets` = avg packet size, so sánh với expected MTU |
| `idle_timeout=<sec>` | Flow expire sau N giây không packet hit. Field chỉ hiển thị khi ≠ 0 | Nếu flow critical (ACL) có idle_timeout → flow có thể tự xóa, check controller logic |
| `hard_timeout=<sec>` | Flow expire sau N giây tuyệt đối, ẩn khi = 0 | Trong OVN flow hard_timeout thường 0; có giá trị → controller hard-set, usually for learn() |
| `priority=<N>` | Match priority. Với hai flow cùng match → highest priority wins | Hai flow cùng priority + overlap match → behavior undefined |
| `<match spec>` | Match field list, ví dụ `in_port=1,ip,nw_dst=10.1.1.1` | - |
| `actions=<action list>` | Action thi hành khi match | `actions=drop` trên flow critical → đã xác định bị loại bỏ (drop), kiểm chứng mong đợi |

**Action syntax breakdown cho entry phức tạp (ARP responder):**

| Action | Ý nghĩa |
|---|---|
| `move:NXM_OF_ETH_SRC[]->NXM_OF_ETH_DST[]` | Copy eth_src (current packet) vào eth_dst, prepare ARP reply |
| `mod_dl_src:aa:bb:cc:dd:ee:ff` | Set eth_src = gateway MAC |
| `load:0x2->NXM_OF_ARP_OP[]` | Set ARP opcode = 2 (reply) |
| `move:NXM_NX_ARP_SHA[]->NXM_NX_ARP_THA[]` | Target hardware = sender (echo back) |
| `move:NXM_OF_ARP_SPA[]->NXM_OF_ARP_TPA[]` | Target protocol = sender IP |
| `load:0xaabbccddeeff->NXM_NX_ARP_SHA[]` | Sender hardware = gateway MAC |
| `load:0x0a010101->NXM_OF_ARP_SPA[]` | Sender protocol = 10.1.1.1 (gateway IP) |
| `IN_PORT` | Output back via ingress port (ARP reply) |

Đây là classic OVN ARP responder flow, inject ARP reply cho gateway IP mà không forward lên CPU/controller.

**Kịch bản bẻ gãy:**

- **`n_packets=0` cho ARP responder expected** → VM chưa gửi ARP request; tạo traffic test `ping <gateway>` rồi re-dump.
- **`n_packets` tăng nhanh nhưng `n_bytes/n_packets` << expected packet size** → packet bị fragment; check MTU underlay.
- **Duration liên tục reset về 0** → flow thrashing; `ovn-controller` đang compute lại liên tục vì NBDB change không stable. Check `ovn-appctl -t ovn-controller inc-engine/show-stats`.

**Upstream:** man `ovs-ofctl(8)` §`dump-flows` + OVS Advanced Tutorial §Table 8 testing.

### 9.4.2.5 Scenario, flow-table hygiene với `replace-flows` và `diff-flows`

**Bối cảnh:** production rule quan trọng — trước khi ghi đè flow table, luôn preview diff. `replace-flows` tự động delete missing + add new + update changed, atomic. Nhưng nếu file input sai, production trong 1 giây → packet drop.

```shell
# Step 1: Dump hiện tại để backup
$ sudo ovs-ofctl -O OpenFlow15 dump-flows br-int > /tmp/br-int.current.flows

# Step 2: Preview diff giữa current và new file
$ sudo ovs-ofctl -O OpenFlow15 diff-flows br-int /tmp/br-int.new.flows
-table=0,priority=100,in_port=1 actions=output:2
+table=0,priority=100,in_port=1 actions=output:3
-table=10,priority=50,ip,nw_dst=10.1.1.0/24 actions=drop

# Step 3: Đánh giá diff — chỉnh output port 1→2→3 là đúng migration; xóa ACL drop là
#   ý định. Approve.

# Step 4: Apply atomic
$ sudo ovs-ofctl -O OpenFlow15 --bundle replace-flows br-int /tmp/br-int.new.flows

# Step 5: Verify
$ sudo ovs-ofctl -O OpenFlow15 diff-flows br-int /tmp/br-int.new.flows
(empty output — đã match)
```

**Quy tắc production:**
- **LUÔN `diff-flows` trước `replace-flows`.** Không có `--dry-run` trong OpenFlow; `diff-flows` là dry-run.
- **Thêm `--bundle`** (OpenFlow 1.4+): mọi change commit trong một bundle atomic. Nếu bundle fail, rollback full, không có partial state.
- **Backup `dump-flows` sang file trước** khi thao tác production. Script automation: chạy backup sau mỗi `replace-flows` thành công.

### 9.4.2.6 Scenario, monitor live flow event

**Bối cảnh:** debug thực time xem flow nào được install/remove/modify. Tương tự `tail -f` nhưng cho OpenFlow.

```shell
$ sudo ovs-ofctl -O OpenFlow15 monitor br-int watch:
send: OFPT_FLOW_MOD (OF1.5) (xid=0x4): ADD table:0 priority=100,in_port=1 actions=output:2
NXT_FLOW_MOD_TABLE_ID (OF1.0): table_id enabled
send: OFPT_SET_ASYNC (OF1.5) (xid=0x5): ...
OFPT_FLOW_REMOVED (OF1.5) (xid=0x0): cookie=0xabc, table=10, priority=50, reason=idle, duration=60.1s
  ip,nw_dst=10.1.1.5 actions=drop
  packet_count:0 byte_count:0
```

**Anatomy event stream:**

| Event type | Ý nghĩa | Khi nào xuất hiện |
|---|---|---|
| `OFPT_FLOW_MOD ADD/MOD/DEL` | Controller install/modify/delete flow | Controller hoạt động, flow change |
| `OFPT_FLOW_REMOVED` | Flow expire do timeout hoặc delete | idle_timeout hết hạn, hoặc `del-flows` từ controller |
| `OFPT_PACKET_IN` | Packet lên controller (table-miss hoặc CONTROLLER action) | Flow thiếu match, controller đang learn |
| `OFPT_PORT_STATUS` | Port add/remove/state change | Admin config change hoặc link flap |

**Field trong FLOW_REMOVED:**
- `reason=idle` → hết idle_timeout, không có packet N giây.
- `reason=hard` → hết hard_timeout.
- `reason=delete` → explicit `del-flows`.
- `reason=evict` → flow_table eviction policy (khi vượt flow_limit).
- `packet_count/byte_count` → counter cuối trước khi xóa. Giá trị 0 → flow không được hit, có thể dấu hiệu khớp với spec sai.

**Use case:** mở 2 terminal. Terminal 1 chạy `monitor`, terminal 2 `ping` host target. Xem packet_in có xuất hiện không, OFPT_FLOW_MOD có cài đặt flow mới không.

**Upstream:** man `ovs-ofctl(8)` §`monitor` + OpenFlow 1.5 spec §7.4 Asynchronous Messages.

## 9.4.3 `ovs-appctl`, introspection RPC qua Unix socket

### 9.4.3.1 Grammar, target discovery

`ovs-appctl` là một wrapper generic gửi RPC command tới bất kỳ OVS daemon nào qua Unix socket. Đây là công cụ duy nhất cho "live introspection" — không ảnh hưởng config (không ghi OVSDB), chỉ truy vấn state runtime của daemon.

Grammar:

```
ovs-appctl [OPTIONS] [-t TARGET | --target=TARGET] COMMAND [ARG...]

TARGET:
  - process name: ovs-vswitchd (default), ovsdb-server, ovn-controller, ovn-northd
  - pidfile path: /var/run/openvswitch/<name>.pid
  - unix socket: unix:/var/run/openvswitch/<name>.<pid>.ctl

OPTIONS:
  --timeout=secs
  -v, --verbose
  -h, --help

Built-in COMMANDS cho mọi daemon:
  list-commands           → list mọi command target hỗ trợ
  version                 → in version + compile date
  vlog/list               → list logging module + current level
  vlog/list-pattern       → pattern per destination
  vlog/set SPEC           → set log level
  vlog/reopen             → close + reopen log file (sau logrotate)
  exit                    → graceful shutdown daemon
  memory/show             → memory usage breakdown

Thêm daemon-specific command — xem `list-commands` trên từng daemon để biết.
```

**Bước đầu tiên luôn là:**

```shell
$ sudo ovs-appctl -t ovs-vswitchd list-commands
```

Lệnh trên liệt kê mọi command mà ovs-vswitchd hỗ trợ (thường ~80-120 command tùy OVS version). Phân nhóm lớn:

| Nhóm | Prefix | Mục đích |
|---|---|---|
| Bridge introspection | `bridge/*`, `fdb/*`, `mdb/*` | Bridge state, MAC learning, multicast |
| OpenFlow simulation | `ofproto/*` | Trace pipeline, inspect classifier |
| Datapath introspection | `dpctl/*`, `dpif/*`, `dpif-netdev/*` | Kernel/userspace datapath state |
| Upcall + revalidator | `upcall/*`, `revalidator/*` | Kernel → userspace flow install |
| Bond + LACP | `bond/*`, `lacp/*` | Bond member state, LACP handshake |
| L2 protocols | `stp/*`, `rstp/*`, `cfm/*`, `bfd/*` | Spanning tree, CFM, BFD |
| Tunnel | `tnl/*` | ARP neighbor cho tunnel, stats |
| Observability | `coverage/*`, `memory/*` | Counter internal + memory |
| Logging | `vlog/*` | Log level per module |

### 9.4.3.2 `ofproto/trace`, công cụ mạnh nhất trong bộ

`ofproto/trace` mô phỏng một packet đi qua OpenFlow pipeline của một bridge, KHÔNG gửi packet thật. Output thể hiện từng table hit, từng rule match, từng action execute, final action set, và datapath action đầu ra. Đây là phương tiện debug chính mà mọi kỹ sư OVS phải thuộc.

**Command syntax (giản lược):**

```
ovs-appctl ofproto/trace [OPTIONS] BRIDGE FLOW_SPEC [OPTIONS]

OPTIONS:
  -generate         → actually execute side effect (learn, ct commit)
  -consistent       → force consistent table cache
  --names | --no-names
  --ct-next=<state> → simulate conntrack returning specific state
```

**Flow spec:** chuỗi `FIELD=VALUE` tương tự `ovs-ofctl dump-flows` khớp với spec.

```shell
          dl_dst=aa:bb:cc:00:00:02,nw_src=10.1.0.5,nw_dst=10.1.0.10,
          nw_frag=no,tp_dst=0x50/0xffff

Datapath actions: 2
```

**Anatomy bốn khối output:**

**Khối 1, `Flow:`** — ban đầu của packet simulate, ovs-appctl fill các field thiếu (vlan_tci=0, nw_tos=0, nw_ttl=0, tcp_flags=0) theo default. Đây là *input* vào pipeline.

**Khối 2, `bridge("br-int") ... N. <match>, priority P`** — từng table hit, theo thứ tự table 0 → table 8 → table 16 → table 32 → table 33. Mỗi dòng format:
- `<table_id>. <match_spec>, priority <P>, cookie <cookie>`
- Các dòng con là action execute trong table đó.

| Ký hiệu | Ý nghĩa |
|---|---|
| `0. in_port=1, priority 100` | Table 0, match `in_port=1`, priority 100 |
| `load:0x1->NXM_NX_REG13[]` | Action: load 0x1 vào register 13 (đánh dấu logical datapath = 1) |
| `resubmit(,8)` | Action: submit packet lên table 8 cùng reg context |
| `priority 0` | Table-miss rule (priority 0) match, default fallthrough |
| `reg13=0x1,dl_dst=...,priority 50` | Table 16, match reg13 cộng eth_dst, priority 50 |
| `load:0x2->NXM_NX_REG15[]` | Set logical output port = 2 (REG15 là logical output port trong OVN) |
| `output:2` | Cuối cùng, physical output port 2 |

**Khối 3, `Final flow:`** — state của packet SAU khi pipeline xong, trước khi exit. So sánh với `Flow:` initial để thấy modification: packet giờ có `reg13=0x1,reg15=0x2` do load ở table 0 và 16.

**Khối 4, `Megaflow:`** — megaflow cache signature tương ứng cho packet này. Đây là *classifier mask* mà kernel datapath sẽ cài đặt sau upcall. Field nào trong megaflow spec tức là field đó **cần được khớp với exact** để kernel reuse cache. Trong ví dụ trên, `tp_dst=0x50/0xffff` nghĩa là kernel khớp với exact port 80; nhưng `nw_src` và `nw_dst` đều khớp với exact → megaflow này không generalize cho IP khác.

**Khối 5, `Datapath actions:`** — action cuối cùng thi hành tại datapath (kernel hoặc userspace). Đây là ground truth — nếu `Datapath actions: drop` → packet sẽ bị drop trong kernel; `Datapath actions: 2,3` → output port 2 và 3.

**Kịch bản bẻ gãy:**

- **`Datapath actions: drop`** → đi lên pipeline tìm rule nào khớp với `drop`. Trong trace, check table hit cuối cùng — sẽ thấy action drop explicit.
- **Megaflow quá specific (quá nhiều field exact)** → kernel sẽ cài đặt flow chỉ cho 1 flow specific, mọi packet khác biệt sẽ miss cache → upcall liên tục. Check `upcall/show` xem `flows_install_rate` cao bất thường không.
- **Cookie `0x0` cho mọi flow** → không phải OVN (OVN sinh cookie != 0). Có thể bạn đang debug br-int không bị OVN quản lý, hoặc đang trên bridge thuần OpenFlow.
- **Trace không progress qua table 8** → flow tại table 0 không có `resubmit(,8)` hoặc `goto_table:8`. OVN pipeline skip table → check `ovn-controller` translation, có thể chassis chưa register đúng.

**Upstream:** man `ovs-vswitchd(8)` §`ofproto/trace`, OVS Advanced Tutorial §Table 0 Testing, `ofproto/ofproto-dpif-xlate.c` function `ofproto_trace()`.

### 9.4.3.4 Đọc hiểu output `ovs-appctl fdb/show`

**Bối cảnh:** debug L2 forwarding — MAC learning table sai dẫn đến broadcast storm hoặc traffic đi sai port. FDB là bản đồ MAC → port + VLAN học tự động trong bridge `standalone` hoặc `learn` action explicit.

```shell
$ sudo ovs-appctl fdb/show br-int
 port  VLAN  MAC                Age
    1   100  aa:bb:cc:00:00:01    4
    2   100  aa:bb:cc:00:00:02   12
    3     0  aa:bb:cc:00:00:03   87
    1   100  aa:bb:cc:00:00:04  287
```

**Anatomy từng field:**

| Field | Ý nghĩa | Dấu hiệu bất thường |
|---|---|---|
| `port` | OpenFlow port number. Tra lại với `ovs-ofctl show br-int` | Port `LOCAL` trong FDB → bridge ít khi cần |
| `VLAN` | VLAN ID. `0` = untagged | VLAN khác expected → port access tag sai |
| `MAC` | MAC address học được từ packet incoming | Duplicate MAC trên hai port → MAC flapping, check L2 loop |
| `Age` | Giây từ khi entry được học lần cuối. Mặc định aging 300s | Entry > 300s vẫn tồn tại → check `ovs-vsctl get Bridge br-int other_config:mac-aging-time` |

**Kịch bản bẻ gãy:**

- **MAC appearing/disappearing giữa nhiều port (MAC flapping)** → có L2 loop; check STP state (`ovs-appctl stp/show` hoặc `rstp/show`).
- **FDB empty trên bridge đang có traffic** → bridge có fail_mode=secure và thiếu learn flow → flush + reapply flow.
- **Age luôn 0** → FDB được update liên tục; host spam broadcast trên MAC đó.

**Useful companion command:** `fdb/flush br-int` buộc relearn toàn bộ MAC (dùng sau thay đổi topology).

### 9.4.3.5 Đọc hiểu output `ovs-appctl upcall/show`

**Bối cảnh:** upcall là cơ chế mà kernel datapath "hỏi" userspace khi gặp packet không khớp với megaflow cache. Rate cao = performance issue. `upcall/show` cho thấy health của fast/slow path handoff.

```shell
$ sudo ovs-appctl upcall/show
system@ovs-system:
  flows         : (current 1247) (avg 1250) (max 1350) (limit 200000)
  dump duration : 8ms
  ufid enabled : true

  1: (keys 624) (max-flows 64) (avg-updates 1.2) (busy cycles 0.03%)
  2: (keys 623) (max-flows 63) (avg-updates 1.3) (busy cycles 0.03%)
```

**Anatomy từng field:**

| Field | Ý nghĩa | Dấu hiệu bất thường |
|---|---|---|
| `system@ovs-system:` | Datapath name | - |
| `flows: (current N) (avg N) (max N) (limit N)` | Flow count in datapath (hiện tại, moving avg, max đạt, limit hard) | `current` gần `limit` → sắp flow full, upcall sẽ throttle; `max` >> `avg` → traffic burst |
| `dump duration: Nms` | Thời gian revalidator dump full flow table. Default target < 100 ms | `dump duration > 1s` → flow table quá lớn hoặc memory contention; đây là triệu chứng Part 9.26 revalidator storm |
| `ufid enabled: true` | OVS 2.7+ dùng Unique Flow ID cho track flow. False → rollback mode | `false` trên OVS ≥ 2.7 → config issue |
| `1: (keys N) (max-flows N)` | Revalidator thread 1 stats | - |
| `keys N` | Số ukey (unique key) thread này manage | Thread unbalanced (one thread keys >> other) → rxq assignment skew |
| `max-flows N` | Max flow thread manage | - |
| `avg-updates N` | Avg update per flow (revalidator pass) | > 5 → flow thrashing, reinstall liên tục |
| `busy cycles N%` | % CPU thread dùng | > 50% → thread saturated |

**Kịch bản bẻ gãy:**

- **`flows current` gần `limit`** → upcall sẽ drop new flow; tăng limit qua `ovs-vsctl set Open_vSwitch . other_config:flow-limit=<N>` hoặc reduce flow explosion tại OpenFlow layer.
- **`dump duration > 1s`** → Phần 9.26 revalidator storm pattern; check `coverage/show | grep revalidate` và commit `180ab2fd635e` fix path.
- **`busy cycles` of single thread > 90%** → revalidator bottleneck; OVS ≥ 2.18 có thread group to help, otherwise CPU saturated.

**Upstream:** OVS source `ofproto/ofproto-dpif-upcall.c` function `upcall_unixctl_show()`, NSDI 2015 paper §5.

**Counter class quan trọng:**

| Counter | Ý nghĩa | Healthy range |
|---|---|---|
| `hit` / `miss` | Megaflow cache hit/miss trong datapath | miss/hit < 1% = healthy; miss/hit > 10% = flow explosion hoặc classifier issue |

## 9.4.4 `ovs-dpctl`, raw datapath inspection

### 9.4.4.1 Grammar và khi nào cần

`ovs-dpctl` nói trực tiếp với datapath (kernel hoặc userspace). Nó bypass ofproto layer — không đi qua OpenFlow, không liên quan classifier, ofproto, hay connection manager. Đây là "ground truth" về những gì datapath ACTUALLY forward.

Khi nào dùng `ovs-dpctl` thay vì `ovs-ofctl`:

| Tình huống | Tool phù hợp |
|---|---|
| Xem flow OpenFlow đã program | `ovs-ofctl dump-flows` |
| Xem megaflow kernel đã install | `ovs-dpctl dump-flows` |
| Kiểm chứng datapath có consistent với OpenFlow không | Cả hai, so sánh cookie + counter |
| Debug conntrack của datapath | `ovs-dpctl dump-conntrack` hoặc `ovs-appctl dpctl/dump-conntrack` |
| Feature flag datapath hỗ trợ gì | `ovs-dpctl get-dp-features` |

Grammar:

```
ovs-dpctl [OPTIONS] COMMAND [ARG...]

key COMMANDS:
  show [DP]                              → datapath summary
  dump-dps                               → list datapath names
  dump-flows [-m] [DP]                   → flow table (-m = more detail)
  dump-conntrack [-s] [-m] [DP]          → conntrack entries
  flush-conntrack [DP] [zone=ZONE]       → flush conntrack
  ct-stats-show [-m] [DP]                → conntrack stats by protocol
  get-flow [-m] [DP] ufid:UFID           → specific flow by UFID

key OPTIONS:
  -m, --more          → more detail (unmasked match fields, ufid)
  -s, --statistics    → include stats
  --names | --no-names
```

### 9.4.4.2 Đọc hiểu output `ovs-dpctl show`

```shell
$ sudo ovs-dpctl show
system@ovs-system:
  lookups: hit:287423 missed:8127 lost:0
  flows: 1247
  masks: hit:4127821 total:1247 hit/pkt:3.47
  port 0: ovs-system (internal)
  port 1: br-int (internal)
  port 2: tap-vm-web (internal)
  port 3: tap-vm-db (internal)
  port 4: ovn-huyvl-0 (geneve)
  port 5: ovn-huyvl-1 (geneve)
```

**Anatomy từng field:**

| Field | Ý nghĩa | Dấu hiệu bất thường |
|---|---|---|
| `system@ovs-system:` | Datapath name | - |
| `lookups: hit:N missed:N lost:N` | Lookup counter: hit (megaflow match), missed (upcall trigger), lost (packet drop do full queue) | `lost > 0` → kernel queue overflow, tình trạng khẩn cấp về hiệu năng; `missed/hit > 5%` → flow explosion |
| `flows: N` | Megaflow count hiện có trong datapath | Khớp với `upcall/show flows current` |
| `masks: hit:N total:N hit/pkt:N` | Classifier mask lookup: hit = total mask hit, total = tổng mask có, hit/pkt = avg mask per packet | `hit/pkt > 10` → classifier linear; Phần 9.15 tuple space search stress |
| `port N: <name> (<type>)` | Port number + name + type (internal / system / geneve / vxlan / netdev) | Mismatch giữa dpctl port N và ofctl port N → rare (kernel dp và userspace dp desync) |

**Kịch bản bẻ gãy:**

- **`lost > 0`** → packet drop vì kernel Netlink queue full; check `dmesg | grep -i 'openvswitch'` cho drop reason. Giải pháp: tăng `net.core.netdev_max_backlog`.
- **`missed/hit > 10%` consistent** → flow explosion; check `dump-flows` có flow-table large không. Với OVN → check `ovn-controller` I-P stat.
- **`hit/pkt > 15`** → classifier subtable explosion; tham khảo Phần 9.15 tuple space search optimization.

### 9.4.4.3 Đọc hiểu output `ovs-dpctl dump-flows -m`

**Bối cảnh:** nhìn thấy flow kernel thực sự đã cài, với megaflow mask full. Flag `-m` in unmasked field + UFID.

```shell
$ sudo ovs-dpctl dump-flows -m | head -3
ufid:a1b2c3d4-5e6f-7a8b-9c0d-e1f2a3b4c5d6,
  recirc_id(0),dp_hash(0/0),skb_priority(0/0),in_port(3),
  skb_mark(0/0),ct_state(0/0),ct_zone(0/0),ct_mark(0/0),ct_label(0/0),
  eth(src=aa:bb:cc:00:00:01,dst=aa:bb:cc:00:00:02),
  eth_type(0x0800),
  ipv4(src=10.1.0.5,dst=10.1.0.10,proto=6,tos=0/0,ttl=64,frag=no),
  tcp(src=50000/0,dst=80),
  tcp_flags(0/0),
  packets:18234, bytes:2145678, used:0.082s,
  flags:SP, actions:4
```

**Anatomy từng token:**

| Token | Ý nghĩa |
|---|---|
| `ufid:<uuid>` | Unique Flow ID, 128-bit, OVS 2.7+ |
| `recirc_id(0)` | Recirculation ID = 0 → packet chưa recirculate; != 0 nếu đã qua ct() hoặc conjunction |
| `in_port(3)` | Ingress datapath port number (match dpctl show) |
| `ct_state(0/0)` | Conntrack state + mask (0/0 = no match) |
| `eth(src=..., dst=...)` | Ethernet match |
| `eth_type(0x0800)` | IP |
| `ipv4(src=..., dst=..., proto=6, ...)` | Layer 3 match; `proto=6` = TCP |
| `tcp(src=50000/0, dst=80)` | TCP source khớp với bất kỳ (mask 0), dest exact 80 |
| `packets:N` | Packet matched flow này |
| `bytes:N` | Bytes matched |
| `used:Ns` | Seconds ago lần khớp với cuối; dùng để revalidator GC |
| `flags:SP` | S=SYN, P=PUSH; TCP flags observed |
| `actions:4` | Output port 4 (geneve tunnel → OVS remote chassis) |

**Kịch bản bẻ gãy:**

- **`used > 10s` cho flow expected active** → traffic stop, flow sắp bị GC.
- **`packets:0, bytes:0` với flow đã tồn tại > vài giây** → flow cài đặt nhưng chưa có match; có thể upcall-then-install race.
- **`actions:drop`** → kernel datapath đã "compile" thành drop; kiểm chứng `ofproto/trace` có produce `Datapath actions: drop` không.
- **Mask `tcp(src=50000/0, dst=80)`** — src mask là 0 (wildcard) vì policy chỉ care dest port 80; nếu thấy `src=50000/0xffff` → rule sai, match exact src port 50000 không generalize.

**Upstream:** man `ovs-dpctl(8)` §`dump-flows` + OVS source `datapath/flow.c`.

### 9.4.5.3 Scenario, backup / restore

**Bối cảnh:** Before upgrade OVS, phải backup OVSDB. `ovsdb-client backup` dumps nhất quán, không yêu cầu daemon stop.

```shell
# Backup
$ sudo ovsdb-client backup unix:/var/run/openvswitch/db.sock > /tmp/conf.db.backup
$ ls -lh /tmp/conf.db.backup
-rw-r--r-- 1 root root 45K Apr 24 11:23 /tmp/conf.db.backup

# Nếu cần restore (scenario rollback upgrade)
$ sudo systemctl stop ovs-vswitchd
$ sudo systemctl stop ovsdb-server
$ sudo mv /etc/openvswitch/conf.db /etc/openvswitch/conf.db.old
$ sudo ovsdb-tool create /etc/openvswitch/conf.db \
    /usr/share/openvswitch/vswitch.ovsschema
$ sudo ovsdb-client restore unix:/var/run/openvswitch/db.sock \
    Open_vSwitch < /tmp/conf.db.backup

# Nhưng thường dễ hơn: copy conf.db sang host khác hoặc restore raw file
```

**Quy tắc production:**
- Backup trước mọi change schema (OVS upgrade 2.17 → 3.x có schema update).
- Backup không block; chạy cron daily vào `/var/backups/ovs/conf.db.YYYYMMDD`.
- Backup + metadata (OVS version, host, kernel) cho traceability.

**Upstream:** man `ovsdb-client(1)` và `ovsdb-tool(1)`. Xem thêm Phần 10.2 cho OVSDB backup/restore + compact deep dive.

## 9.4.6 Six-layer troubleshooting playbook, tích hợp

Khi có báo cáo "packet không đi qua OVS đúng cách", đi tuần tự sáu lớp từ dưới lên. Mỗi lớp có lệnh kiểm tra cụ thể, expected state, và branch quyết định.

| Lớp | Câu hỏi | Lệnh kiểm tra | Expected | Nếu fail |
|---|---|---|---|---|
| **L1 Physical** | Packet có đến NIC không? | `tcpdump -i eth0 -nn -c 10` | Thấy packet inbound | Cable, SFP, NIC driver; `ethtool eth0` verify link |
| **L2 Kernel driver** | Packet vào datapath? | `ovs-dpctl show` → counter `lookups:missed` tăng | missed tăng khi có traffic | NIC bound wrong driver; `driverctl list-overrides` cho DPDK |
| **L3 Megaflow cache** | Có hit megaflow nào? | `ovs-dpctl dump-flows -m` → tìm flow khớp với packet | Flow có packet:N > 0 | Lỡ cache liên tục → ở L4 có vấn đề |
| **L4 OpenFlow pipeline** | Flow nào match? Action gì? | `ovs-appctl ofproto/trace br-int "FLOW_SPEC"` | `Datapath actions: <ofport>` non-drop | Sai rule priority, thiếu table, action drop unintended |
| **L5 OVSDB config** | Port/VLAN/bridge đúng? | `ovs-vsctl show` + `ovs-vsctl list Interface <name>` | Port gắn đúng bridge, interface up | Port thiếu VLAN tag, interface type sai |
| **L6 Logical (OVN)** | LSP, Port_Binding đúng? | `ovn-sbctl show` + `ovn-sbctl list Port_Binding` + `ovn-trace <dp> "FLOW"` | Port_Binding có `chassis` non-null | ovn-controller chưa bind, logical flow compile sai |

**Quy tắc:** khi L1-L6 OK mà packet vẫn drop → vấn đề ngoài OVS (upstream switch, firewall, ARP, MTU, BGP). Không tiếp tục debug OVS.

### 9.4.6.1 Guided Exercise full 6-layer walkthrough

**Kịch bản:** user báo *"không ping được từ VM1 (tap-vm-web, 10.1.0.5) đến VM2 (tap-vm-db, 10.1.0.10), cả hai cùng Logical_Switch `ls-prod`, expected thông qua br-int"*.

**Setup (giả lập):**

```shell
$ sudo ovs-vsctl add-br br-int -- set Bridge br-int fail_mode=secure
$ sudo ovs-vsctl add-port br-int tap-vm-web -- \
    set Interface tap-vm-web type=internal
$ sudo ovs-vsctl add-port br-int tap-vm-db -- \
    set Interface tap-vm-db type=internal
$ sudo ip link set tap-vm-web up
$ sudo ip addr add 10.1.0.5/24 dev tap-vm-web
$ sudo ip link set tap-vm-db up
$ sudo ip addr add 10.1.0.10/24 dev tap-vm-db
# Add flow: allow ARP + IP between two ports, bug intentionally: drop ICMP
$ sudo ovs-ofctl -O OpenFlow15 add-flow br-int \
    "table=0,priority=200,in_port=tap-vm-web,dl_type=0x0806,actions=output:tap-vm-db"
$ sudo ovs-ofctl -O OpenFlow15 add-flow br-int \
    "table=0,priority=200,in_port=tap-vm-db,dl_type=0x0806,actions=output:tap-vm-web"
$ sudo ovs-ofctl -O OpenFlow15 add-flow br-int \
    "table=0,priority=100,in_port=tap-vm-web,dl_type=0x0800,nw_proto=6,\
     actions=output:tap-vm-db"
$ sudo ovs-ofctl -O OpenFlow15 add-flow br-int \
    "table=0,priority=100,in_port=tap-vm-db,dl_type=0x0800,nw_proto=6,\
     actions=output:tap-vm-web"
# Bug: flow priority 50 drop ICMP (giả đặt)
$ sudo ovs-ofctl -O OpenFlow15 add-flow br-int \
    "table=0,priority=150,in_port=tap-vm-web,dl_type=0x0800,nw_proto=1,actions=drop"
```

**Walkthrough playbook:**

**L1 — tcpdump physical (skip vì đây là internal-only setup; OVN Geneve setup thì tcpdump eth0):**

```shell
$ sudo tcpdump -i tap-vm-web -nn icmp -c 5
# Thấy ICMP outbound → L1 OK
```

**L2 — ovs-dpctl show:**

```shell
$ sudo ovs-dpctl show
system@ovs-system:
  lookups: hit:23, missed:7, lost:0
  flows: 5
  masks: hit:38 total:4 hit/pkt:1.65
  port 0: ovs-system (internal)
  port 1: br-int (internal)
  port 2: tap-vm-web (internal)
  port 3: tap-vm-db (internal)

# Counter missed > 0 → packet có vào OVS, kernel trigger upcall. OK.
```

**L3 — ovs-dpctl dump-flow:**

```shell
$ sudo ovs-dpctl dump-flows -m
# Sẽ thấy flow ICMP install với actions:drop — dấu hiệu packet drop ở L3/L4
```

**L4 — ofproto/trace ICMP:**

```shell
$ sudo ovs-appctl ofproto/trace br-int \
    "in_port=tap-vm-web,dl_src=aa:bb:cc:00:00:01,dl_dst=aa:bb:cc:00:00:02,\
     dl_type=0x0800,nw_src=10.1.0.5,nw_dst=10.1.0.10,nw_proto=1"
Flow: icmp,in_port=2,vlan_tci=0x0000,dl_src=aa:bb:cc:00:00:01,
      dl_dst=aa:bb:cc:00:00:02,nw_src=10.1.0.5,nw_dst=10.1.0.10,...

bridge("br-int")
----------------
 0. in_port=2,ip,nw_proto=1, priority 150
    drop

Final flow: unchanged
Datapath actions: drop
```

**Root cause** rõ ràng — Rule `table=0 priority=150 in_port=tap-vm-web,ip,nw_proto=1 drop`.

**L5 — L6 không cần vì L4 đã tìm ra root cause.**

**Fix:**

```shell
$ sudo ovs-ofctl -O OpenFlow15 del-flows br-int \
    "table=0,in_port=tap-vm-web,dl_type=0x0800,nw_proto=1" --strict
$ sudo ovs-ofctl -O OpenFlow15 add-flow br-int \
    "table=0,priority=100,in_port=tap-vm-web,dl_type=0x0800,nw_proto=1,\
     actions=output:tap-vm-db"
$ sudo ovs-ofctl -O OpenFlow15 add-flow br-int \
    "table=0,priority=100,in_port=tap-vm-db,dl_type=0x0800,nw_proto=1,\
     actions=output:tap-vm-web"

# Re-trace
$ sudo ovs-appctl ofproto/trace br-int "in_port=tap-vm-web,...,nw_proto=1"
Datapath actions: 3
```

**Teardown:**

```shell
$ sudo ovs-vsctl del-br br-int
```

### 9.4.7.1 Decision tree, khi nào dùng tool nào

```
Config change (bridge/port/interface/QoS/controller)?
  → ovs-vsctl

Flow programming (add/del/mod/dump OpenFlow rule)?

Raw kernel datapath (megaflow, conntrack ground truth)?

Raw OVSDB access (theo dõi stream, backup, foreign schema)?
  → ovsdb-client

OVN logical layer (NBDB / SBDB)?
```

## Điểm cốt lõi cần nhớ

Thứ nhất, năm công cụ CLI của OVS là **độc lập về vai trò**, không thay thế nhau. `ovs-vsctl` chỉ config OVSDB; `ovs-ofctl` chỉ OpenFlow; `ovs-appctl` chỉ introspection runtime; `ovs-dpctl` chỉ raw datapath; `ovsdb-client` chỉ raw OVSDB RPC. Mỗi công cụ cover một lớp kiến trúc, và workflow debug thực tế dùng song song nhiều công cụ.

Thứ hai, `ofproto/trace` là công cụ debug mạnh nhất — mô phỏng packet đi qua OpenFlow pipeline mà không gửi packet thật. Engineer OVS phải đọc thành thạo bốn khối output (Flow, bridge walk-through, Final flow, Datapath actions, Megaflow).

Thứ ba, six-layer troubleshooting playbook là phương pháp có hệ thống đi từ L1 physical lên L6 logical, loại trừ từng vùng. Khi sáu lớp agree mà packet vẫn drop → root cause ngoài OVS.

Thứ tư, atomicity của OVSDB transaction (ovs-vsctl) và OpenFlow bundle (ovs-ofctl `--bundle`) bảo vệ khỏi half-configured state. Luôn `diff-flows` trước `replace-flows` trong production; luôn backup `conf.db` trước upgrade.

Thứ năm, đọc output không chỉ là thấy lệnh chạy không báo lỗi. Mỗi field trong output có ý nghĩa vật lý (counter, timestamp, state enum) và có kịch bản bẻ gãy riêng (khi field X = Y thì symptom là Z). Kỹ sư thành thạo là người biết field X quan trọng, biết ngưỡng Y, biết Y vượt ngưỡng thì làm tiếp gì.

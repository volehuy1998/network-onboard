## Learning objectives

After completing this part, you will be able to:

1. **Identify (Remember)** the three canonical install paths for Open vSwitch on Ubuntu 22.04: the distribution package, the upstream release tarball, and the upstream git checkout. Each path produces a working Open vSwitch 2.17.9 on the same host with materially different trade-offs.
2. **Distinguish (Analyze)** the trade-offs along five concerns: control over build flags, freshness of the source, kernel-module strategy, install-layout placement on the filesystem, and operational overhead at upgrade time.
3. **Apply (Apply)** the path-selection decision to your own lab or production scenario, using the small decision tree at the end of section 0.4.6.
4. **Verify (Apply)** that the three paths produce binaries with the same `--version` output and the same OpenFlow protocol support, and explain why the binaries are not bit-identical even when the version strings match.
5. **Recover (Apply)** from the three most common failure modes: the out-of-tree kernel-module compile rejection on a recent kernel, the residual database file that survives an `apt-get purge`, and the persisted kernel module that survives both an apt purge and an `ovs-ctl stop`.

## 0.4.1. The three paths and the question they answer

Open vSwitch is shipped to operators in three forms. The Ubuntu archive carries a Debian-packaged build that the Canonical quality engineering process has tested against the rest of the Ubuntu 22.04 archive. The Open vSwitch project itself publishes a release tarball at `www.openvswitch.org/releases/` after every stable point release. The same project hosts a public git repository at `github.com/openvswitch/ovs.git` that exposes the canonical source tree, including release tags, development branches, and the full commit history.

The question this part answers is: **for a given host and a given goal, which form should the operator use?** The answer is not "the newest one" or "the easiest one". The answer depends on five concerns that the next sections walk through in turn: control over build flags, freshness of the source, kernel-module strategy, install-layout placement, and operational overhead.

Plan v3.13 captured all three paths on the same host (`lab-openvswitch`, `192.168.1.250`) on 2026-04-29, in the order A then B then C, with each session beginning from the post-uninstall state of the previous session. The verbatim host transcripts are referenced from this part. Where a fenced block in this part shows a command and its output, the bytes come from those transcripts unchanged, per the rule that lab evidence is reproduced exactly as captured on the lab host.

## 0.4.2. Path A, install from the Ubuntu archive

Path A is the simplest possible install. The Ubuntu archive on `jammy-updates` carries the package `openvswitch-switch` at version `2.17.9-0ubuntu0.22.04.1`. A single `apt-get install` invocation pulls the package, registers the systemd units, creates the on-disk layout, and starts the daemons.

### What the operator sees

The line `OpenFlow versions 0x1:0x6` says the `ovs-ofctl` client speaks OpenFlow protocol numbers 1 through 6, which corresponds to OpenFlow 1.0, 1.1, 1.2, 1.3, 1.4, and 1.5. Block IV of this series exercises this range.

### Where the package puts files

Path A follows Debian Filesystem Hierarchy Standard conventions. The configuration directory is `/etc/openvswitch/`, the database file lives at `/var/lib/openvswitch/conf.db` with a symlink at `/etc/openvswitch/conf.db`, the logs go to `/var/log/openvswitch/`, the binaries land under `/usr/bin/` and `/usr/sbin/`, and the systemd units live at `/lib/systemd/system/`. The package installs four units:

- `openvswitch-switch.service` is the umbrella unit. Starting it pulls in the rest.
- `ovs-vswitchd.service` runs the slow-path daemon.
- `ovsdb-server.service` runs the database server.
- `ovs-record-hostname.service` is a one-shot helper that records the host's hostname into `external_ids:hostname` on the `Open_vSwitch` row at boot.

The complete file inventory of the `openvswitch-switch` package is 67 entries on Ubuntu 22.04. The first thirty give the texture:

Two non-obvious placements are worth pointing out. First, the package keeps a SysV init script at `/etc/init.d/openvswitch-switch` for compatibility, even though systemd is the active init system on Ubuntu 22.04; the systemd unit takes precedence and the SysV script is a fallback that operators rarely touch. Second, the package ships `if-pre-up.d/` and `if-post-down.d/` hooks for the legacy `ifupdown` network configuration system; modern netplan-based hosts (which is the default on Ubuntu 22.04) do not exercise these hooks, but the package still ships them to support hosts that have not yet migrated.

### What path A skips

Path A does not run a compiler. The operator never sees `configure`, `make`, or any autotools machinery. The kernel module the daemon will load is the in-kernel `openvswitch.ko` that Ubuntu's kernel package shipped under `/lib/modules/$(uname -r)/kernel/net/openvswitch/openvswitch.ko`. Build flags such as `--with-debug` or `--enable-Werror` cannot be set; the operator gets the configuration Canonical chose. Patches to the source cannot be applied. The version is the version Ubuntu shipped, frozen until Canonical publishes an update.

### Path A residual after `apt-get purge`

The single most surprising property of path A is that `apt-get purge openvswitch-switch openvswitch-common python3-openvswitch` does **not** restore the host to the state it had before the install. Two artefacts persist.

The first is the database file. The package's `postrm` script removes `/etc/openvswitch/`, but `/var/lib/openvswitch/` survives:

```text
root@lab-openvswitch:~# DEBIAN_FRONTEND=noninteractive apt-get purge -y openvswitch-switch openvswitch-common python3-openvswitch 2>&1 | tail -20
Removing openvswitch-switch (2.17.9-0ubuntu0.22.04.1) ...
Removing openvswitch-common (2.17.9-0ubuntu0.22.04.1) ...
dpkg: warning: while removing openvswitch-common, directory '/var/log/openvswitch' not empty so not removed
Removing python3-openvswitch (2.17.9-0ubuntu0.22.04.1) ...
Processing triggers for man-db (2.10.2-1) ...
Purging configuration files for openvswitch-switch (2.17.9-0ubuntu0.22.04.1) ...
root@lab-openvswitch:~# ls -la /etc/openvswitch /var/lib/openvswitch /var/log/openvswitch 2>&1
ls: cannot access '/etc/openvswitch': No such file or directory
/var/lib/openvswitch:
total 28
drwxr-xr-x  2 root root  4096 Apr 29 17:14 .
drwxr-xr-x 41 root root  4096 Apr 29 17:14 ..
-rw-r-----  1 root root 16564 Apr 29 17:14 conf.db
-rw-------  1 root root     0 Apr 29 17:14 .conf.db.~lock~

/var/log/openvswitch:
total 8
drwxr-xr-x  2 root root   4096 Apr 29 17:15 .
drwxrwxr-x 11 root syslog 4096 Apr 29 17:14 ..
```

`/etc/openvswitch/` is gone. `/var/lib/openvswitch/conf.db` (16 564 bytes) is **still on disk**. A naive operator who issues `apt-get purge` and expects a clean slate, then runs `apt-get install openvswitch-switch` again, will boot into the prior database, complete with whatever bridges, ports, controllers, and SSL certificates were configured. The behaviour is reproducible across Ubuntu releases and is not a bug; it is Debian package-cleanup convention. The database file is treated as user data, and Debian convention does not delete user data on purge.

The second is the kernel module. `apt-get purge` removes the userspace binaries but does not call `rmmod openvswitch`:

```text
root@lab-openvswitch:~# ovs-vsctl --version 2>&1 || echo "(binary removed)"
bash: /usr/bin/ovs-vsctl: No such file or directory
(binary removed)
root@lab-openvswitch:~# lsmod | grep -E "^openvswitch" || echo "(module unloaded)"
openvswitch           176128  0
root@lab-openvswitch:~# systemctl is-active openvswitch-switch 2>&1 || true
inactive
```

The `ovs-vsctl` binary is gone. The systemd unit is inactive. The kernel module is **still loaded** with refcount 0. Module unload is operator-driven; the package does not unload modules on remove. To fully restore the green-field state the operator runs:

```text
root@lab-openvswitch:~# rm -rf /var/lib/openvswitch /var/log/openvswitch
root@lab-openvswitch:~# rmmod openvswitch
```

These two commands are the manual cleanup path A leaves to the operator. The takeaway is that an Open vSwitch host is rarely stateless, even after a clean uninstall. Provisioning automation should snapshot before the install or accept that state accumulates across install paths.

## 0.4.3. Path B, build from the upstream release tarball

Path B is the canonical recipe for building Open vSwitch from a stable point release. The operator downloads `openvswitch-2.17.9.tar.gz` from `www.openvswitch.org/releases/`, verifies its SHA-256 hash, extracts it, and runs the configure-make-install cycle. The tarball ships with a pre-generated `configure` script, so the operator does not need a working autotools chain on the build host.

### Build prerequisites

The tarball expects seven build prerequisites already installed:

```text
root@lab-openvswitch:~# apt-get install -y autoconf automake libtool pkg-config libcap-ng-dev python3-dev libunbound-dev
```

These provide the C compiler toolchain (already on the host because Ubuntu ships `gcc`), the autotools (used by the source tree's `Makefile.in` files even when the operator does not regenerate `configure`), `pkg-config` (for finding the system libraries the build links against), `libcap-ng` (Linux capability handling), and `libunbound` (the DNS resolver library Open vSwitch uses for its address-lookup helpers).

### Download, verify, extract

```text
root@lab-openvswitch:~# mkdir -p /root/ovs-r1b && cd /root/ovs-r1b
root@lab-openvswitch:~/ovs-r1b# wget -q https://www.openvswitch.org/releases/openvswitch-2.17.9.tar.gz
root@lab-openvswitch:~/ovs-r1b# sha256sum openvswitch-2.17.9.tar.gz
```

The operator compares the printed SHA-256 against the value published on the Open vSwitch download page. A mismatch means a corrupted download or a tampered mirror; both are reasons to abort.

```text
root@lab-openvswitch:~/ovs-r1b# tar xzf openvswitch-2.17.9.tar.gz
root@lab-openvswitch:~/ovs-r1b# ls -la openvswitch-2.17.9/ | head -10
-rwxrwxr-x  1 engineer engineer      40 Feb  8  2024 boot.sh
-rwxrwxr-x  1 engineer engineer 1091118 Feb  8  2024 configure
-rw-rw-r--  1 engineer engineer    7607 Feb  8  2024 configure.ac
```

Three observations are worth pausing on. First, `configure` exists already, 1 091 118 bytes, generated by upstream's `boot.sh`. The release tarball includes it for a reason: not every operator has `autoconf` and `automake` matching the version the project was authored against, and the generated `configure` script removes that dependency. A learner who instead clones the git tree (path C) gets a directory without `configure` and must run `boot.sh` first. Second, every file is owned by `engineer:engineer` with the upstream timestamp `Feb 8 2024`. These are the credentials and time of whichever Open vSwitch maintainer rolled the release tarball; `tar` preserves them as-is, which is a small forensic detail an auditor can use to confirm the tarball is genuine. Third, the `datapath/` directory is present and contains the kernel-module source code, which `--with-linux` would build against, if the kernel allowed it.

### The first configure attempt and why it fails

Plan v3.13 §R1 specifies `--prefix=/usr/local --localstatedir=/usr/local/var --sysconfdir=/usr/local/etc --with-linux=/lib/modules/$(uname -r)/build`. The `--with-linux` flag asks the build to compile the Open vSwitch kernel module against the running kernel's headers. The flag is the historically correct way to deploy Open vSwitch on a host that does not yet have an in-kernel `openvswitch.ko`. On a 5.15 kernel, the flag fails:

```text
root@lab-openvswitch:~/ovs-r1b/openvswitch-2.17.9# ./configure --prefix=/usr/local --localstatedir=/usr/local/var --sysconfdir=/usr/local/etc --with-linux=/lib/modules/$(uname -r)/build 2>&1 | tail -10
configure: WARNING: --with-linux is deprecated and kernel support is limited to 5.8 and below
configure: error: Linux kernel in /lib/modules/5.15.0-173-generic/build is version 5.15.198, but version newer than 5.8.x is not supported (please refer to the FAQ for advice)
```

Two lines of `configure` output are doing the teaching. The first line says `--with-linux` is deprecated and kernel support is limited to 5.8 and below. The second line says the lab's 5.15 kernel is too new and refuses the build. The reason is that the Open vSwitch out-of-tree kernel module compatibility shim was last updated to keep up with kernel APIs through 5.8. Newer kernels changed the netfilter integration, the conntrack callbacks, and the namespace plumbing in ways the shim does not cover. Upstream's documented answer is: stop building the out-of-tree module; rely on the in-kernel module that the Linux kernel itself ships under `net/openvswitch/`.

### The recovery path

The recovery is to drop `--with-linux` and let the configure script default to userspace-only:

```text
root@lab-openvswitch:~/ovs-r1b/openvswitch-2.17.9# ./configure --prefix=/usr/local --localstatedir=/usr/local/var --sysconfdir=/usr/local/etc 2>&1 | tail -5
config.status: creating include/openvswitch/version.h
config.status: creating config.h
config.status: executing tests/atconfig commands
config.status: executing depfiles commands
config.status: executing libtool commands
```

The configure script terminates without an error this time. It produces `Makefile`, the per-subdirectory `Makefile.in` instances, the `.pc` pkg-config files, the symbol export lists, the version header, and `config.h` (the C preprocessor symbols every source file pulls in). A learner walking the source tree later will find `#include "config.h"` near the top of every C file; the contents of that header are decided by what `./configure` learned about this host.

### Build, install, and start

```text
root@lab-openvswitch:~/ovs-r1b/openvswitch-2.17.9# (time make -j4) 2>&1 | tail -25
real	0m26.737s
user	1m15.952s
sys	0m15.237s
root@lab-openvswitch:~/ovs-r1b/openvswitch-2.17.9# du -sh /root/ovs-r1b/openvswitch-2.17.9
265M	/root/ovs-r1b/openvswitch-2.17.9
```

The `time` line says wall clock 26.7 seconds, total user CPU 75.9 seconds (across four parallel workers), system time 15.2 seconds. The user-to-real ratio of 2.84 says the build was effectively using a little under three of the four cores on average; the rest of the time was spent in system calls and waiting on input and output. The build added 200 MiB to the source tree (72 MiB up to 265 MiB), mostly object files, intermediate libraries (`libopenvswitch.a` is the largest), and the per-binary executables that link statically against `libopenvswitch.a`.

`make install` copies the freshly-built artefacts into the prefixed install layout:

- First-tier daemon binaries (`ovsdb-server`, `ovs-vswitchd`, `ovs-bugtool`) under `/usr/local/sbin/`.
- Operator-facing CLI (`ovs-vsctl`, `ovs-ofctl`, `ovs-dpctl`, `ovs-appctl`, plus `ovsdb-tool`, `ovsdb-client`, `ovs-tcpdump`, `ovs-pcap`, `ovs-pki`, `ovs-l3ping`, `ovs-test`, `ovs-testcontroller`, `ovs-vlan-test`, `ovs-tcpundump`, `ovs-parse-backtrace`, and `ovs-docker`) under `/usr/local/bin/`.
- C headers under `/usr/local/include/openflow/` (the OpenFlow protocol headers, one per OpenFlow version) and `/usr/local/include/openvswitch/` (the public Open vSwitch library headers).
- pkg-config files (`libopenvswitch.pc`, `libsflow.pc`, `libofproto.pc`, `libovsdb.pc`) under `/usr/local/lib/pkgconfig/`. These let downstream consumers such as OVN, kube-ovn, and vendor packages find the Open vSwitch library at link time.
- Schema files (`vswitch.ovsschema`, `local-config.ovsschema`, `vtep.ovsschema`) and helper scripts (`ovs-ctl`, `ovs-kmod-ctl`, `ovs-save`, `ovs-monitor-ipsec`, `ovs-vtep`) under `/usr/local/share/openvswitch/`.

Path B does not install systemd unit files; `make install` is platform-neutral. To start the daemons, the operator runs the `ovs-ctl` script bundled under `/usr/local/share/openvswitch/scripts/`:

```text
root@lab-openvswitch:~/ovs-r1b/openvswitch-2.17.9# mkdir -p /usr/local/etc/openvswitch /usr/local/var/run/openvswitch /usr/local/var/log/openvswitch
root@lab-openvswitch:~/ovs-r1b/openvswitch-2.17.9# ovsdb-tool create /usr/local/etc/openvswitch/conf.db /usr/local/share/openvswitch/vswitch.ovsschema 2>&1
root@lab-openvswitch:~/ovs-r1b/openvswitch-2.17.9# /usr/local/share/openvswitch/scripts/ovs-ctl start 2>&1 | tail -25
 * Starting ovsdb-server
 * system ID not configured, please use --system-id
 * Configuring Open vSwitch system IDs
 * Inserting openvswitch module
 * Starting ovs-vswitchd
 * Enabling remote OVSDB managers
root@lab-openvswitch:~/ovs-r1b/openvswitch-2.17.9# ovs-vsctl show 2>&1
5f8e9a74-1559-4375-9b90-a7474ef58029
    ovs_version: "2.17.9"
```

The orchestration script does six things in this order: starts `ovsdb-server` against `conf.db`, warns that no `--system-id` was passed (a minor production-hardening oversight a learner can ignore in a lab), populates the `Open_vSwitch.system-id` and related rows, runs `modprobe openvswitch` (this is where the in-kernel `openvswitch.ko` is loaded), starts `ovs-vswitchd`, and enables the remote-manager listening sockets. The line ` * Inserting openvswitch module` is the moment the kernel's openvswitch module becomes resident.

### What path B gives the operator that path A does not

Path B exposes every build flag. The operator can pass `--enable-Werror` to fail the build on warnings, `--with-debug` to keep more debug symbols, `--enable-coverage` to compile the gcov instrumentation in, or `--with-dpdk=...` to link DPDK-accelerated paths in (subject to the permanent ban on DPDK content in this curriculum). Patches to the source tree can be applied between extraction and `./configure`. Newer or older versions than the distribution shipped can be built. The install layout is operator-controlled by the `--prefix` family of flags; `/usr/local/` is convention, but `/opt/openvswitch-2.17.9/` is equally valid and easier to side-by-side with future versions.

The cost is the operator now owns four responsibilities the package handled implicitly: starting the daemons (no systemd unit unless the operator writes one), rotating the logs (no `/etc/logrotate.d/openvswitch-switch` unless the operator drops one in), upgrading on a security advisory (the operator must rebuild from a newer tarball; there is no `apt upgrade`), and managing the kernel-module strategy. The last responsibility is the most subtle, because path B specifically does **not** ship a kernel module on a 5.15 kernel, and the operator must understand that the daemon is using the in-kernel module without the build system having involved itself.

## 0.4.4. Path C, build from the upstream git checkout

Path C clones the upstream git tree at `https://github.com/openvswitch/ovs.git`, checks out the `v2.17.9` tag, and runs `boot.sh` before `configure`. The path differs from path B in three concrete ways: the `configure` script does not exist on disk at clone time, the autotools-generated `Makefile.in` family is not on disk either, and the `tests/` directory contains the raw `.at` autotest sources from which the autotest binary is later assembled.

### Clone, checkout, and inspect

```text
root@lab-openvswitch:~/network-onboard# mkdir -p /root/ovs-r1c && cd /root/ovs-r1c
root@lab-openvswitch:~/ovs-r1c# git clone --depth 50 --branch v2.17.9 https://github.com/openvswitch/ovs.git ovs 2>&1 | tail -10
root@lab-openvswitch:~/ovs-r1c# cd /root/ovs-r1c/ovs && git rev-parse HEAD
0bea06d9957e3966d94c48873cd9afefba1c2677
root@lab-openvswitch:~/ovs-r1c/ovs# git describe --tags HEAD
v2.17.9
root@lab-openvswitch:~/ovs-r1c/ovs# git log -1 --format='%h %s'
0bea06d Set release date for 2.17.9.
```

The HEAD commit SHA is `0bea06d9957e3966d94c48873cd9afefba1c2677`, which matches the `v2.17.9` tag. The same SHA is the canonical reference used throughout the rest of this series for source citations against Open vSwitch 2.17.9.

The git checkout differs from the release tarball in two visible ways:

```text
root@lab-openvswitch:~/ovs-r1c/ovs# ls /root/ovs-r1c/ovs/configure 2>&1 || echo '(configure not present, expected for git checkout)'
ls: cannot access '/root/ovs-r1c/ovs/configure': No such file or directory
(configure not present, expected for git checkout)
root@lab-openvswitch:~/ovs-r1c/ovs# ls /root/ovs-r1c/ovs/boot.sh /root/ovs-r1c/ovs/configure.ac /root/ovs-r1c/ovs/Makefile.am 2>&1
/root/ovs-r1c/ovs/boot.sh  /root/ovs-r1c/ovs/configure.ac  /root/ovs-r1c/ovs/Makefile.am
root@lab-openvswitch:~/ovs-r1c/ovs# du -sh /root/ovs-r1c/ovs
34M	/root/ovs-r1c/ovs
```

The `configure` script is absent. `boot.sh`, `configure.ac`, and `Makefile.am` are present. The source tree is 34 MiB, less than half of the 72 MiB the tarball expanded to in path B. The size delta has two causes: the git checkout omits the autotools-generated artefacts (`configure`, `Makefile.in`, `aclocal.m4`, `build-aux/*`) that `boot.sh` will produce in a moment, and the shallow clone with `--depth 50` keeps a fraction of the full upstream history.

### What `boot.sh` does

`boot.sh` is two lines:

```text
root@lab-openvswitch:~/ovs-r1c/ovs# cat boot.sh
#! /bin/sh
autoreconf --install --force
```

`autoreconf --install --force` is the orchestration entry point of the GNU autotools chain. The `--install` flag copies any missing helper files into `build-aux/` and `m4/`. The `--force` flag rebuilds even outputs that already exist. Internally, `autoreconf` invokes the autotools in the right order: `aclocal` (gathers `.m4` macros from `configure.ac` and `m4/`), `libtoolize` (copies the libtool helper macros into `m4/`), `autoheader` (produces `config.h.in` from `configure.ac`), `automake --add-missing` (produces `Makefile.in` from `Makefile.am` plus all `automake.mk` includes), and `autoconf` (produces the final `configure` script from `configure.ac` plus the gathered macros). Each tool depends on the output of the previous, and `autoreconf` enforces the order.

The actual run on the lab host:

```text
root@lab-openvswitch:~/ovs-r1c/ovs# ./boot.sh 2>&1
libtoolize: putting auxiliary files in AC_CONFIG_AUX_DIR, 'build-aux'.
libtoolize: copying file 'build-aux/ltmain.sh'
libtoolize: putting macros in AC_CONFIG_MACRO_DIRS, 'm4'.
libtoolize: copying file 'm4/libtool.m4'
libtoolize: copying file 'm4/ltoptions.m4'
libtoolize: copying file 'm4/ltsugar.m4'
libtoolize: copying file 'm4/ltversion.m4'
libtoolize: copying file 'm4/lt~obsolete.m4'
configure.ac:197: warning: The macro `AC_ERROR' is obsolete.
configure.ac:197: You should run autoupdate.
./lib/autoconf/oldnames.m4:34: AC_ERROR is expanded from...
acinclude.m4:119: OVS_CHECK_LINUX is expanded from...
configure.ac:197: the top level
configure.ac:27: installing 'build-aux/compile'
configure.ac:43: installing 'build-aux/config.guess'
configure.ac:43: installing 'build-aux/config.sub'
configure.ac:22: installing 'build-aux/install-sh'
configure.ac:22: installing 'build-aux/missing'
Makefile.am: installing 'build-aux/depcomp'
```

The visible work happens in two phases. First, `libtoolize` copies its helper files: `ltmain.sh` into `build-aux/` and five `.m4` macros into `m4/`. Second, the warning. The line `configure.ac:197: warning: The macro 'AC_ERROR' is obsolete.` says the source tree at line 197 of `configure.ac` calls `AC_ERROR`, which `autoconf` deprecated long ago and renamed to `AC_MSG_ERROR`. The warning chain identifies that `AC_ERROR` is expanded from `OVS_CHECK_LINUX` in `acinclude.m4:119`, and `OVS_CHECK_LINUX` is invoked at the top level of `configure.ac`. The warning is harmless: `autoreconf` recognises the obsolete name and substitutes the modern form, and the resulting `configure` script works correctly. A reader interested in upstream contribution practices could file a one-line patch that replaces `AC_ERROR` with `AC_MSG_ERROR`; it would land easily.

After `boot.sh` returns, `configure` exists. The git checkout now has the same surface a release tarball does: the operator can run `./configure` directly, with the same arguments path B used.

### Configure, build, and binary identity check

The `./configure` arguments match path B's userspace-only invocation. The build runs the same `make -j4` cycle and produces binaries that the operator can compare against path B's install:

```text
root@lab-openvswitch:~/ovs-r1c/ovs# ./configure --prefix=/usr/local --localstatedir=/usr/local/var --sysconfdir=/usr/local/etc 2>&1 | tail -5
config.status: executing depfiles commands
config.status: executing libtool commands
config.status: executing include/openflow/openflow.h.stamp commands
config.status: executing utilities/bugtool/dummy commands
config.status: executing ipsec/dummy commands
root@lab-openvswitch:~/ovs-r1c/ovs# (time make -j4) 2>&1 | tail -15
mv tests/testsuite.tmp tests/testsuite
real	0m31.618s
user	1m41.348s
sys	0m12.613s
```

The build took 31.6 seconds wall clock, about five seconds longer than path B's 26.7 seconds. The delta is the autotest binary `tests/testsuite`, which is assembled by concatenating the `.at` source files and processing them through `autom4te`. The visible last line before the `real` row is `mv tests/testsuite.tmp tests/testsuite`, which is the rename that finalises the autotest binary; path B did not produce this file because the tarball ships it pre-built.

The crucial cross-check is that path B and path C produce binaries with identical version output:

```text
root@lab-openvswitch:~/ovs-r1c/ovs# diff <(/root/ovs-r1c/ovs/utilities/ovs-vsctl --version) <(/usr/local/bin/ovs-vsctl --version) 2>&1 | head -5
root@lab-openvswitch:~/ovs-r1c/ovs# /root/ovs-r1c/ovs/utilities/ovs-vsctl --version | head -3
ovs-vsctl (Open vSwitch) 2.17.9
DB Schema 8.3.0
```

The `diff` command produced no output. Empty diff means the two binaries report identical version strings: `ovs-vsctl (Open vSwitch) 2.17.9` plus `DB Schema 8.3.0` plus a third line listing the supported OpenFlow versions (truncated by `head -3`). The path B install at `/usr/local/bin/ovs-vsctl` and the path C build at `/root/ovs-r1c/ovs/utilities/ovs-vsctl` are functionally identical.

The empty `diff` is **not** a guarantee that every byte of the two ELF binaries matches. The build directory paths embedded in DWARF debug info will differ (path B built in `/root/ovs-r1b/openvswitch-2.17.9/`, path C in `/root/ovs-r1c/ovs/`). The compile timestamps embedded in the binaries will differ. The libtool wrapping order may differ in tiny ways. The version invariant is the right one to check at the operator layer: the upstream source, the autotools-generated headers, and the compile flags are all identical between path B and path C, so the runtime behaviour is the same, even though the on-disk bytes differ in irrelevant places.

### What path C exposes that path B hides

Three things, all of them educational rather than operationally distinctive:

1. **The autotools machinery is visible.** Path C runs `boot.sh`, which runs `autoreconf`, which runs `libtoolize`, `aclocal`, `autoheader`, `automake`, and `autoconf` in sequence. Path B's tarball already had the output of all of those steps on disk. A learner who wants to understand how a configure script gets generated learns it in path C, not path B.
2. **The autotest source is on disk.** The `.at` files under `tests/` are the raw test source. Path B's tarball ships the assembled `tests/testsuite` shell-script binary; path C builds it from the `.at` sources. A learner who wants to write a new test, or read an existing one, finds the source on path C.
3. **The full `git log` is reachable.** Path C is a git checkout; the operator can `git log`, `git blame`, `git show`, and `git diff` against any earlier commit. Path B cannot do any of this because the tarball is a flat snapshot. A learner debugging an upstream bug can use `git bisect` on path C.

The `git clone --depth 50` flag in the path C invocation keeps only the most recent fifty commits; a learner who needs the full history runs `git fetch --unshallow` after the initial clone. The full history of `openvswitch/ovs.git` is about 12 000 commits and weighs about 350 MiB of `.git/` storage; the `--depth 50` shortcut keeps `.git/` to roughly 35 MiB, which is the right pragma for a learner who is not bisecting.

### Running the in-tree test suite

Path C is the only path on which `make check` works, because path C is the only path that produces the autotest binary `tests/testsuite`. The full suite has roughly 2 500 tests; a selective run filters by keyword:

```text
root@lab-openvswitch:~/ovs-r1c/ovs# (time make check TESTSUITEFLAGS='-k ovs-vsctl --jobs=4') 2>&1 | tail -15
2341: bootstrap ca cert                               ok
2342: peer ca cert                                    ok

## ------------- ##
## Test results. ##
## ------------- ##

All 45 tests were successful.
make[3]: Leaving directory '/root/ovs-r1c/ovs'
make[2]: Leaving directory '/root/ovs-r1c/ovs'
make[1]: Leaving directory '/root/ovs-r1c/ovs'

real	0m5.544s
user	0m7.844s
sys	0m2.822s
```

The `-k ovs-vsctl` flag selects every test whose name or `AT_KEYWORDS` list mentions `ovs-vsctl`; on Open vSwitch 2.17.9 that is 45 tests. The `--jobs=4` flag runs them in parallel on four cores; total wall clock is 5.5 seconds. The numeric prefixes `2341:`, `2342:` are the autotest binary's per-test ordinals across the whole suite, not just the selected subset; the suite numbers tests sequentially as they appear in the master `.at` files.

The autotest framework writes a summary log at `tests/testsuite.log` regardless of pass or fail. The log records the platform identification, the command line, and one block per test. For a passing run it stays small (about 365 lines for a 45-test selection); for a failing run it grows to include every diagnostic the test framework collected.

## 0.4.5. Side-by-side comparison along the five concerns

Each row below answers the question: how does path A, B, or C handle this concern?

### Control over build flags

Path A gives the operator no control. Canonical's build is the build. Path B and path C are equivalent on this concern: every `./configure` flag from `--enable-Werror` through `--with-dpdk` is available, every `make` variable is settable, the source is editable before build, patches can be applied. Path B's source is a flat snapshot; path C's source is a git checkout, which is more convenient when the operator wants to apply a patch as a `git apply` rather than as a `patch -p1`.

### Freshness of the source

Path A delivers the version Canonical chose to ship and to backport security fixes for. On Ubuntu 22.04 the answer is `2.17.9-0ubuntu0.22.04.1`, frozen until Canonical publishes an update. Path B delivers the version the operator downloaded; the operator must check `www.openvswitch.org/releases/` for new releases. Path C delivers any commit on any branch the operator chooses; with `git fetch && git checkout` the operator can move between releases, between development branches, or to any individual commit. For a learner studying a specific bug, path C is the only path that can target the commit just before and just after the fix.

### Kernel-module strategy

Path A loads the in-kernel `openvswitch.ko` from `/lib/modules/$(uname -r)/kernel/net/openvswitch/`. The operator never sees a kernel-module decision. Path B and path C both reject `--with-linux` on a 5.15 kernel; the operator must drop the flag and rely on the in-kernel module. The result is the same module being loaded as in path A. The single deviation is that on a 5.4 or earlier kernel, paths B and C **could** build their own kernel module, and the operator could `modprobe` it instead of the in-kernel one; on a 5.15 kernel, that path is closed and the in-kernel module is the only choice.

### Install-layout placement

Path A respects Debian Filesystem Hierarchy Standard conventions. Configuration in `/etc/`, state in `/var/lib/`, logs in `/var/log/`, binaries in `/usr/bin/` and `/usr/sbin/`, units in `/lib/systemd/system/`. Path B and path C use whatever `--prefix`, `--localstatedir`, `--sysconfdir` say; the convention is `--prefix=/usr/local --localstatedir=/usr/local/var --sysconfdir=/usr/local/etc`, which lays out under `/usr/local/` instead of `/`. A consequence is that the same host can run a path A install and a path B install at the same time without the binaries colliding, because they are in different directory trees; one is at `/usr/bin/ovs-vsctl`, the other at `/usr/local/bin/ovs-vsctl`. The shell's `PATH` order decides which one wins for a bare `ovs-vsctl` invocation.

### Operational overhead

Path A is the lowest overhead. The package's `postinst` starts the daemons; `apt upgrade` upgrades the version; `apt remove` stops the daemons; `logrotate` rotates the logs. Paths B and C give the operator each of these responsibilities. There is no `systemctl start openvswitch-switch` unless the operator writes a unit, no log rotation unless the operator drops a `/etc/logrotate.d/openvswitch-switch.local` in, no automatic upgrade. A learner often writes a small `Makefile` target or a one-line shell wrapper to paper over these gaps; production deployments either build a custom Debian package on top of path B or use a configuration-management tool to manage the unit files.

### Summary table

| Concern | Path A, distro package | Path B, release tarball | Path C, git checkout |
|------|------------------------|-------------------------|----------------------|
| Build flag control | None | Full | Full |
| Source freshness | Canonical's update cadence | Open vSwitch release cadence | Any commit |
| Kernel module strategy | In-kernel `openvswitch.ko` | In-kernel on 5.9+, out-of-tree on 5.8 and below | Same as B |
| Install layout | Debian Filesystem Hierarchy Standard | Operator-controlled `--prefix` | Same as B |
| Daemon startup | systemd units installed | `ovs-ctl start` from script | Same as B |
| Log rotation | `/etc/logrotate.d/openvswitch-switch` | Operator writes one | Same as B |
| Upgrade path | `apt upgrade` | Re-extract new tarball, rebuild | `git checkout newer-tag`, rebuild |
| Source patching | Not supported | `patch -p1` before configure | `git apply` before configure |
| Test suite available | No | No (tarball ships built binary, but `make check` is supported when full source is present) | Yes, `make check` runs the autotest binary |
| Disk footprint | Roughly 25 MiB | About 200 MiB built tree plus 100 MiB install | About 280 MiB built tree plus 100 MiB install plus 35 MiB git history |
| Wall clock to first running daemon | About 30 seconds | About 90 seconds (including build) | About 100 seconds (including `boot.sh` plus build) |

## 0.4.6. Choosing a path

The decision tree below is short on purpose. Read top to bottom; stop at the first applicable answer.

1. **Is the host running a production workload that someone other than you operates?** Pick path A. The operational overhead of paths B and C is a real cost the production team will pay every time the version moves, every time a security advisory drops, and every time a junior operator inherits the host.
2. **Is the host a learning environment, and your goal is to learn the autotools build system, the test suite, or how to apply a patch to the source?** Pick path C. The autotools machinery is exposed, the autotest sources are on disk, and `git log` works.
3. **Is the host a learning environment, and your goal is just to have a recent or specific release available without learning the build system?** Pick path B. The pre-generated `configure` script removes the autotools dependency; the rest of the build is the same as path C.
4. **Is the host an operator's hands-on development machine where you sometimes need to apply an upstream patch?** Pick path C with a non-shallow clone (drop `--depth 50`). The full git history lets you `git bisect` and `git cherry-pick` upstream commits.
5. **Are you writing a custom Debian package on top of upstream Open vSwitch?** Pick path B. The release tarball is the canonical source of truth for the Debian source package; building from path C adds a regeneration step that complicates the source-package tooling.

The answers above assume Open vSwitch 2.17.9. The same decision tree applies to every Open vSwitch release on every Ubuntu LTS from 20.04 forward.

## 0.4.7. Common failure modes and recovery

Three failure modes are common enough to call out explicitly.

### Failure 1: `--with-linux` rejected on a recent kernel

The symptom on a 5.9 or newer kernel:

```text
configure: WARNING: --with-linux is deprecated and kernel support is limited to 5.8 and below
configure: error: Linux kernel in /lib/modules/.../build is version X.Y.Z, but version newer than 5.8.x is not supported (please refer to the FAQ for advice)
```

Recovery: drop `--with-linux` from the configure invocation. The build will produce userspace only. The daemon will load the in-kernel `openvswitch.ko` that the kernel package shipped, which is what every standard Ubuntu install expects. This is the production answer for almost every operator who is not back-porting datapath patches.

### Failure 2: `apt-get purge` does not restore the green-field state

The symptom: after `apt-get purge openvswitch-switch openvswitch-common python3-openvswitch`, the database file at `/var/lib/openvswitch/conf.db` and the kernel module remain. A subsequent `apt-get install` boots into the prior database.

Recovery: after the purge, the operator runs:

```text
root@lab-openvswitch:~# rm -rf /var/lib/openvswitch /var/log/openvswitch
root@lab-openvswitch:~# rmmod openvswitch
```

The first command removes the residual state and log directories. The second command unloads the kernel module. After both commands, `lsmod | grep openvswitch` returns no rows and the host is back to the green-field baseline.

### Failure 3: a path B or path C build picks up the wrong libraries from a parallel path A install

The symptom: the operator runs `ovs-vsctl --version` and sees `2.17.9` from `/usr/bin/ovs-vsctl` (path A) rather than `2.17.9` from `/usr/local/bin/ovs-vsctl` (path B or C). In practice both paths report the same version on Ubuntu 22.04, but in cross-version cases the wrong one runs.

Recovery: explicitly invoke the binary by its absolute path during testing (`/usr/local/bin/ovs-vsctl --version` rather than `ovs-vsctl --version`). For the daemons, the same applies: `pgrep -af ovs-vswitchd` shows the full command line, including which copy is running. The operator who wants the path B or path C copy to be the default for shell invocation reorders `PATH` to put `/usr/local/bin/` before `/usr/bin/`, or removes the path A install with `apt-get purge` before relying on the operator-built copy.

## 0.4.8. What this part pins for the rest of the series

The next ten parts of the series, and most of the labs that follow, assume Open vSwitch 2.17.9 is available on the host as a working install. Any of the three paths satisfies that assumption. The following invariants hold across all three:

- `ovs-vsctl --version` reports `ovs-vsctl (Open vSwitch) 2.17.9` and `DB Schema 8.3.0`.
- `ovs-ofctl --version` reports `OpenFlow versions 0x1:0x6`, meaning the client speaks OpenFlow 1.0, 1.1, 1.2, 1.3, 1.4, and 1.5.
- The OVSDB database file (whether at `/etc/openvswitch/conf.db` for path A or `/usr/local/etc/openvswitch/conf.db` for paths B and C) is created from the `vswitch.ovsschema` file shipped with version 2.17.9.
- The kernel module `openvswitch.ko` loads from `/lib/modules/5.15.0-173-generic/kernel/net/openvswitch/` on Ubuntu 22.04 with kernel 5.15.
- The system UUID written into the `Open_vSwitch` row at first start (the row UUID printed by `ovs-vsctl show` on an empty bridge set) is unique per host and persists across daemon restarts.

The canonical commit SHA for source citations against Open vSwitch 2.17.9 is `0bea06d9957e3966d94c48873cd9afefba1c2677`. Every later part of this series that cites a function name, a file path, or a line number against Open vSwitch source is referring to this SHA.

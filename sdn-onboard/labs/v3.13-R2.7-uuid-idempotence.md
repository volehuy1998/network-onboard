## What R2.7 proves, and why a Cisco engineer needs the discipline

### Three properties R2.7 establishes

R2.7 is a database lab. The three properties to prove are the rules a CMS-driven script must rely on to be correct: (1) row UUIDs persist across same-key re-creation only when the row was not deleted in between; (2) `--may-exist` and `--if-exists` make `add-*` and `del-*` commands idempotent, by which we mean the second invocation of the same command produces the same end state without error; (3) the Open_vSwitch singleton row's UUID persists for the entire database file's lifetime, which is the basis of OVS's "the database file is the system identity" property.

The proofs are mechanical and observable. (1) Capture the bridge UUID, run `--may-exist add-br` again, capture again, compare. Then delete-and-recreate, capture again, compare. The first comparison must match; the second must differ. (2) Run `del-* --if-exists` twice. Both must succeed silently. (3) Capture the singleton row UUID; observe that it is the same value R2.0 captured days ago.

### Why this matters for an operator

A CMS like OpenStack Neutron, Kubernetes' OVN-Kubernetes, or a hand-written orchestration script makes hundreds of OVSDB transactions per minute on a busy host. The CMS is not always reading the current state before writing; it is sometimes resending the same configuration after a transient failure, sometimes converging towards a target state without checking the source state. The OVSDB API and the `ovs-vsctl` flags have to be designed so that a re-sent `add-br` is a no-op (otherwise the CMS would have to add a "if exists" check before every command, which is racy in a multi-writer environment). The `--may-exist` and `--if-exists` flags are how OVS exposes that idempotence; the daemon checks the desired final state and either does nothing (if already in that state) or transactionally moves to it.

The Cisco analogue is partial. On a Catalyst, `interface vlan 1` followed by another `interface vlan 1` is benign because IOS treats interface configuration as overwriting (the second command does not produce an error or duplicate the interface). On Open vSwitch the same idempotence is opt-in via `--may-exist`; without the flag, `ovs-vsctl add-br br0` on an existing `br0` is an *error*. The asymmetry is deliberate; OVS's CLI is closer to a database insert (where re-inserting a unique key is a real error) than to IOS's idempotent stanzas. The reader who understands this saves the standard "I forgot the --may-exist" debugging round.

## Property 1: --may-exist preserves row UUID on second call

```text
root@lab-openvswitch:~/network-onboard# B1=$(ovs-vsctl get Bridge br0 _uuid); echo "Bridge UUID 1: $B1"
Bridge UUID 1: 61c1593f-e1ee-434f-9c13-91cb2b0be46e
root@lab-openvswitch:~/network-onboard# ovs-vsctl --may-exist add-br br0
root@lab-openvswitch:~/network-onboard# B2=$(ovs-vsctl get Bridge br0 _uuid); echo "Bridge UUID 2: $B2"
Bridge UUID 2: 61c1593f-e1ee-434f-9c13-91cb2b0be46e
root@lab-openvswitch:~/network-onboard# [ "$B1" = "$B2" ] && echo "UUIDs match (--may-exist is idempotent)" || echo "UUIDs differ"
UUIDs match (--may-exist is idempotent)
```

The first capture stores the bridge UUID `61c1593f-e1ee-434f-9c13-91cb2b0be46e` in shell variable `B1`. The second `ovs-vsctl --may-exist add-br br0` produces no output (silent success on an already-existing bridge). The third capture stores the UUID again in `B2`. The shell test `[ "$B1" = "$B2" ]` evaluates true, the `&&` branch runs, the operator sees `UUIDs match (--may-exist is idempotent)`. The proof is mechanical: two captures of the same UUID separated by a no-op operation prove the bridge row was untouched.

The implementation. `--may-exist` causes `ovs-vsctl` to query for an existing row with the same key (`Bridge.name=br0`) before sending the insert; if a row exists, the insert is skipped and the existing row's UUID is returned. The OVSDB transaction at the wire is therefore zero operations on the second call: the daemon sees no transaction at all, the database file is not modified, the row's `_version` (a hidden OVSDB column that increments on every change) is not bumped. The reader who is concerned about transaction-rate impact on a busy daemon can rely on `--may-exist` to truly do nothing when the row already exists.

## Property 2: --if-exists makes deletion idempotent

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl --if-exists del-br br0
root@lab-openvswitch:~/network-onboard# ovs-vsctl --if-exists del-br br0
root@lab-openvswitch:~/network-onboard# echo "del with --if-exists idempotent: succeeded twice"
del with --if-exists idempotent: succeeded twice
```

The first `del-br --if-exists` deletes the existing bridge (silent success). The second `del-br --if-exists` finds no bridge with the name `br0` and silently succeeds anyway, because the `--if-exists` flag tells `ovs-vsctl` to treat the absence of the row as a no-op rather than an error. The shell `echo` line confirms both invocations completed without exit-status failures (otherwise the `echo` would have been blocked by an unset `set -e`, which the lab's bash does not have, but the operator can verify by checking `$?` after each command; both return 0).

Without `--if-exists`, the second invocation would error with "no bridge named br0", which a CMS would have to handle. With the flag, the CMS can simply re-issue the `del-br` and trust that the result is correct regardless of the starting state.

## Property 3: delete-and-recreate produces a fresh UUID

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl --may-exist add-br br0
root@lab-openvswitch:~/network-onboard# B3=$(ovs-vsctl get Bridge br0 _uuid); echo "Bridge UUID 3 (after del+re-add): $B3"
Bridge UUID 3 (after del+re-add): 2af118ca-1f23-4f02-86ed-64f7318ecd48
root@lab-openvswitch:~/network-onboard# [ "$B1" = "$B3" ] && echo "UUID preserved across delete-recreate (UNEXPECTED)" || echo "UUID is fresh after delete-recreate (expected)"
UUID is fresh after delete-recreate (expected)
```

The bridge `br0` was deleted in Property 2. The new `--may-exist add-br br0` finds no existing row, inserts a fresh one, allocates a new UUID. The captured `B3` (`2af118ca-1f23-4f02-86ed-64f7318ecd48`) differs from `B1` (`61c1593f-e1ee-434f-9c13-91cb2b0be46e`); the shell test evaluates false; the `||` branch runs.

This is the property a CMS can rely on for unique row identity. Two bridges with the same name but separated by a delete-and-recreate are *different rows* with different UUIDs. A CMS that has cached the old UUID in its own state can detect the recreation by re-fetching and comparing. R8 (OVSDB internals) returns to this in the context of monitor-protocol notifications: the daemon sends a delete notification followed by an insert notification, with the new UUID; clients who track UUIDs see the change.

The contrast with the OVS `system-id` is sharp. The `system-id` is a UUID stored in the `Open_vSwitch.external_ids:system-id` column; it is generated once at first daemon startup and persists across daemon restart and host reboot. The Bridge row UUID is freshly allocated on every create. The `system-id` is the host identity; the Bridge UUID is the per-bridge identity scoped to the database file.

## Property 4: --bare and find for scripting

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=_uuid,name list Bridge
_uuid               : 2af118ca-1f23-4f02-86ed-64f7318ecd48
name                : br0
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=_uuid,name list Port
_uuid               : e970c0a4-0c27-4053-a7ff-9e9a914c2770
name                : br0
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=_uuid,name,type list Interface
_uuid               : 58ced32d-2136-450e-aee7-4f8096a3c07f
name                : br0
type                : internal
root@lab-openvswitch:~/network-onboard# ovs-vsctl --bare --columns=_uuid find Bridge name=br0
2af118ca-1f23-4f02-86ed-64f7318ecd48
root@lab-openvswitch:~/network-onboard# ovs-vsctl --bare --columns=_uuid find Port name=br0
e970c0a4-0c27-4053-a7ff-9e9a914c2770
root@lab-openvswitch:~/network-onboard# ovs-vsctl --bare --columns=_uuid find Interface name=br0
58ced32d-2136-450e-aee7-4f8096a3c07f
root@lab-openvswitch:~/network-onboard# ovs-vsctl get Open_vSwitch . bridges
[2af118ca-1f23-4f02-86ed-64f7318ecd48]
```

Six commands, three observation points.

The first three (`list <Table>`) print every row in each of the three tables, with the columns specified. Each table has exactly one row right now: one Bridge, one Port (the auto-created LOCAL), one Interface (the LOCAL's interface). The reader can compare the three UUIDs and confirm they are all distinct: Bridges, Ports, and Interfaces have separate UUID spaces, and a row never shares a UUID with a row in another table.

The next three (`--bare --columns=_uuid find <Table> name=br0`) are the script-friendly form. The `--bare` flag strips the column-name labels, leaving just the value; the `find` verb filters rows by the given key-value pair. The output is one UUID per command, on a single line, with no headings. This is the form a CMS or shell script uses when it needs to capture a UUID into a variable; the reader will see this exact form in `kolla-ansible` recipes, in OpenStack Neutron's OVS agent, and in OVN-Kubernetes' netplugin.

The last command shows the cross-table reference. `ovs-vsctl get Open_vSwitch . bridges` returns the singleton row's `bridges` column, which is a set of UUIDs (one per bridge). The output `[2af118ca-...]` is the OVSDB representation of "set containing one element"; the element matches the Bridge row's UUID exactly. The set is what `ovs-vsctl show` traverses to render the bridges hierarchy. R8 will exercise the schema-level detail of references-and-sets directly.

## Property 5: external_ids as the operator-extensible metadata column

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl --no-wait set Open_vSwitch . external_ids:lab-marker=R2.7
root@lab-openvswitch:~/network-onboard# ovs-vsctl get Open_vSwitch . external_ids
{hostname=lab-openvswitch, lab-marker=R2.7, rundir="/usr/local/var/run/openvswitch", system-id="23e4c0ac-b74b-48f8-912f-a4020fa89883"}
```

Most OVSDB tables have an `external_ids` column. The column is a string-to-string map that OVS itself ignores; it is reserved for operator-supplied metadata. The reader sets `lab-marker=R2.7` and reads the column back. Four entries are now present: the lab-marker the operator just set, plus three entries OVS itself maintains (`hostname`, `rundir`, `system-id`).

The `system-id` value `23e4c0ac-b74b-48f8-912f-a4020fa89883` is the host-wide identifier R2.0 mentioned in passing. It is a separate UUID from the singleton row's `_uuid` (`15b3fd6f-...`) because the two serve different purposes: `_uuid` is the row identity (changes if the database is rebuilt); `system-id` is the host identity (changes only if the operator wipes the entire OVS configuration directory). The OVS bootstrap script `ovs-ctl start` populates `system-id` on first startup if the column is unset; a CMS can override it with `ovs-ctl --system-id=...`.

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl set Bridge br0 external_ids:owner=R2.7-lab external_ids:notes='created at $(date -u)'
root@lab-openvswitch:~/network-onboard# ovs-vsctl get Bridge br0 external_ids
{notes="created at $(date -u)", owner=R2.7-lab}
```

The same pattern on a Bridge row: the operator sets two `external_ids` entries with `:` notation. The literal value `'created at $(date -u)'` was passed unquoted to the daemon; the single quotes around it on the command line prevent shell expansion, so the OVSDB column stores the literal characters `created at $(date -u)`, not the time. This is a real operator pitfall: an operator who types unquoted `$(date -u)` will get the *current* time substituted at command-issue time, which is rarely what is wanted; the literal-string form is more useful for audit trails because the operator can grep for the exact value later. The reader should be aware of the shell-quoting subtlety here.

The `external_ids` column is the most-used extension point in OVS. OVN uses it heavily: every logical port has an `external_ids:iface-id` that connects the OVS Interface row to an OVN logical port. OpenStack Neutron uses it to tag ports with the Neutron port UUID. A CMS can use `external_ids` to cache its own state alongside OVS's, with no schema change and no synchronisation cost.

## Property 6: --may-exist preserves Port AND Interface UUIDs together

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl --may-exist add-port br0 dummy0 -- set Interface dummy0 type=internal
root@lab-openvswitch:~/network-onboard# P1=$(ovs-vsctl get Port dummy0 _uuid)
root@lab-openvswitch:~/network-onboard# I1=$(ovs-vsctl get Interface dummy0 _uuid)
root@lab-openvswitch:~/network-onboard# echo "Port UUID 1: $P1"
Port UUID 1: 652f7c02-f6cd-4484-b180-7258f3317009
root@lab-openvswitch:~/network-onboard# echo "Interface UUID 1: $I1"
Interface UUID 1: ca5948c3-d489-419f-9bff-a2b35394b079
root@lab-openvswitch:~/network-onboard# ovs-vsctl --may-exist add-port br0 dummy0
root@lab-openvswitch:~/network-onboard# P2=$(ovs-vsctl get Port dummy0 _uuid)
root@lab-openvswitch:~/network-onboard# I2=$(ovs-vsctl get Interface dummy0 _uuid)
root@lab-openvswitch:~/network-onboard# echo "Port UUID 2: $P2"
Port UUID 2: 652f7c02-f6cd-4484-b180-7258f3317009
root@lab-openvswitch:~/network-onboard# echo "Interface UUID 2: $I2"
Interface UUID 2: ca5948c3-d489-419f-9bff-a2b35394b079
root@lab-openvswitch:~/network-onboard# [ "$P1" = "$P2" ] && [ "$I1" = "$I2" ] && echo "All UUIDs preserved by --may-exist" || echo "UUIDs differed (UNEXPECTED)"
All UUIDs preserved by --may-exist
```

The first `--may-exist add-port` creates the Port and Interface rows for `dummy0`. The captures `P1` and `I1` record the UUIDs. The second `--may-exist add-port` is a no-op (the port already exists). The captures `P2` and `I2` record the UUIDs again. The shell-conjunction test verifies both pairs match. Both Port UUID `652f7c02-...` and Interface UUID `ca5948c3-...` are unchanged.

The proof is more substantial than Property 1's because it covers the linked rows. When `add-port` creates a port, it creates two rows in two tables: a Port row and a child Interface row, with the Port's `interfaces` column referencing the Interface's UUID. A re-add must preserve *both* UUIDs to be truly idempotent; if either UUID changed, downstream references in the database (a CMS's cached UUID, a OF flow rule's `external_ids:port-uuid`, an OVN `Port_Binding.options:iface-id` based on the Interface row) would dangle. R2.7 verifies that they do not.

## Property 7: del-port garbage-collects the row

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl --if-exists del-port br0 dummy0
root@lab-openvswitch:~/network-onboard# ovs-vsctl --if-exists del-port br0 dummy0
root@lab-openvswitch:~/network-onboard# echo "del-port --if-exists idempotent: succeeded twice"
del-port --if-exists idempotent: succeeded twice
root@lab-openvswitch:~/network-onboard# ovs-vsctl --bare --columns=_uuid find Port name=dummy0
root@lab-openvswitch:~/network-onboard# echo "(empty find: garbage-collected)"
(empty find: garbage-collected)
```

`del-port --if-exists` succeeds twice in a row (the second is a no-op because there is no port to delete). The next `find Port name=dummy0` returns zero output; the row was not just removed from the bridge's `ports` reference set, it was deleted from the Port table entirely.

This is OVSDB's garbage collection. A row in the Port table is *root-set* only when it is referenced from at least one Bridge's `ports` column (the schema marks Port and Interface as non-root). When `del-port` removes the bridge's reference to the port, the port row becomes unreachable from the root set, and the daemon's GC pass deletes the orphan automatically. The Interface row attached to the Port is similarly deleted, because it is referenced only from the Port. Three rows (Port + Interface + Interface-LOCAL-no, no, just Port and Interface) clean up in one transaction.

The OVSDB schema declares the GC behaviour. `vswitch.ovsschema` says `Bridge.ports` is `min:0,max:unlimited,refType=strong` (strong reference: GC the target if no strong references remain). `Port.interfaces` is similarly strong. A CMS that uses raw OVSDB transactions (not `ovs-vsctl`) must understand the GC semantics; otherwise the CMS may try to "manually delete" a Port row and end up with orphan Interface rows that are then GC'd at unpredictable times. R8 returns to this in detail.

The proof of complete cleanup. The `find` returns zero output, not "Port row exists with empty interfaces". The row is gone. R2.4's discussion of patch-port orphans is the contrasting case: an orphan patch peer's row remains because OVSDB does not validate `options:peer` as a foreign key (so the daemon does not consider the peer-pointing row "unreferenced").

## Property 8: column updates preserve the row UUID

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br0 dummy0 -- set Interface dummy0 type=internal
root@lab-openvswitch:~/network-onboard# P3=$(ovs-vsctl get Port dummy0 _uuid)
root@lab-openvswitch:~/network-onboard# echo "Port UUID 3 (after del+re-add): $P3"
Port UUID 3 (after del+re-add): 50712123-44dd-46a0-8d7c-d5aeab09fcf6
root@lab-openvswitch:~/network-onboard# [ "$P1" = "$P3" ] && echo "Port UUID preserved across del+re-add (UNEXPECTED)" || echo "Port UUID is fresh after del+re-add (expected)"
Port UUID is fresh after del+re-add (expected)
root@lab-openvswitch:~/network-onboard# ovs-vsctl --no-wait set Bridge br0 fail_mode=secure
root@lab-openvswitch:~/network-onboard# ovs-vsctl get Bridge br0 fail_mode
secure
root@lab-openvswitch:~/network-onboard# B4=$(ovs-vsctl get Bridge br0 _uuid)
root@lab-openvswitch:~/network-onboard# [ "$B3" = "$B4" ] && echo "Bridge UUID preserved across column update (expected)" || echo "Bridge UUID changed (UNEXPECTED)"
Bridge UUID preserved across column update (expected)
```

Two more proofs.

First: re-adding `dummy0` after deletion produces UUID `P3 = 50712123-...`, distinct from `P1 = 652f7c02-...`. Same name, different row. The proof matches Property 3 (delete-and-recreate produces fresh UUID), now applied to a Port row.

Second: setting `fail_mode=secure` on the Bridge changes a column value but does not change the row UUID. `B4 == B3 = 2af118ca-...`. The Bridge row's identity is independent of its column values; only delete-and-recreate produces a new identity.

The pedagogical point. UUIDs are the row identity; column values are mutable per-row state. A CMS that caches the Bridge's UUID can rely on the cache as long as the bridge is not deleted. Even a `fail_mode` toggle, a controller change, an `external_ids` update do not invalidate the UUID cache. Only `del-br` does. The `_version` column (not exposed by default but visible via `ovs-vsctl --columns=_uuid,_version,name list Bridge`) increments on every column update; `_uuid` does not.

## Property 9: the singleton row UUID persists for the database file's lifetime

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl get Open_vSwitch . _uuid
15b3fd6f-e30f-4ae3-b8ca-4531189afee5
root@lab-openvswitch:~/network-onboard# ovs-vsctl --bare --columns=_uuid list Open_vSwitch
15b3fd6f-e30f-4ae3-b8ca-4531189afee5
root@lab-openvswitch:~/network-onboard# echo "(Open_vSwitch is singleton; one UUID, persists for db lifetime)"
(Open_vSwitch is singleton; one UUID, persists for db lifetime)
```

The singleton row's UUID is `15b3fd6f-e30f-4ae3-b8ca-4531189afee5`. This is the same UUID R2.0 reported in its very first `ovs-vsctl show` output. The reader can verify by opening the R2.0 transcript and confirming the match.

The persistence property. The Open_vSwitch table is declared in the schema as `maxRows=1, isRoot=true`. Exactly one row exists from the moment the database file is created (by `ovsdb-tool create`) until the moment the database file is destroyed. The row's UUID is allocated at create time and never changes. A daemon restart, a host reboot, an `ovs-vswitchd` upgrade, a schema convert (`ovsdb-tool convert` to a new schema version), all preserve the UUID. The only way to get a different UUID for the singleton row is to recreate the database file from scratch.

This is the basis of OVS's "the database file is the system identity" property. A CMS that wants to detect a fresh OVS install (the operator wiped `/usr/local/etc/openvswitch/conf.db`) does so by comparing the cached singleton UUID with the current one; a difference means a fresh database. A CMS that wants to validate "this is the same OVS host I configured yesterday" reads the singleton UUID and compares.

The R2 lab series spans roughly an hour of wall-clock time across seven sub-captures and exercises hundreds of OVSDB transactions. Throughout, the singleton UUID `15b3fd6f-...` has not changed. The reader can verify this by `grep`ing the sub-capture transcripts that ran `ovs-vsctl show` for the literal string `15b3fd6f-e30f-4ae3-b8ca-4531189afee5`. Six of the eight R2 transcripts (R2.0, R2.2, R2.3, R2.4, R2.6, R2.7) contain the value because they ran `ovs-vsctl show` at least once; R2.1 and R2.8 do not run that command (R2.1 focuses on `dump-flows` and `Bridge` row reads, R2.8 focuses on `fdb/show` and `Interface` row reads), so the UUID does not appear in those two transcripts. The persistence claim still holds: every appearance is identical, and the OVS daemon was not restarted between sub-captures.

## Tear down

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl --if-exists del-port br0 dummy0
root@lab-openvswitch:~/network-onboard# ovs-vsctl --if-exists del-br br0
root@lab-openvswitch:~/network-onboard# ovs-vsctl list-br
root@lab-openvswitch:~/network-onboard# ovs-vsctl --no-wait remove Open_vSwitch . external_ids lab-marker
root@lab-openvswitch:~/network-onboard# ovs-vsctl get Open_vSwitch . external_ids
{hostname=lab-openvswitch, rundir="/usr/local/var/run/openvswitch", system-id="23e4c0ac-b74b-48f8-912f-a4020fa89883"}
root@lab-openvswitch:~/network-onboard# date -u
Thu Apr 30 05:05:46 AM UTC 2026
```

Three cleanup operations. First, `del-port --if-exists` removes the dummy0 port idempotently (it is the second time R2.7 has run del-port on `dummy0`, and `--if-exists` makes the second run safe). Second, `del-br --if-exists` removes the bridge. Third, the operator removes the `lab-marker` from `external_ids` to leave the Open_vSwitch row in its pre-lab state. The `external_ids` column now has only the three OVS-managed entries (`hostname`, `rundir`, `system-id`); the operator's `lab-marker` is gone.

The wall clock at 05:05:46 UTC is 29 seconds after the opening 05:05:17 UTC. R2.7 is the shortest sub-capture in R2 because it is database-only; no kernel netdevs, no namespaces, no packet flow. Every operation is an OVSDB transaction.

## A short ASCII summary of the UUID lifetimes

```ascii
   table              row identity     lifetime
   ---------------    -------------    ------------------------------------
   Open_vSwitch       _uuid            database file lifetime
                                       (created by ovsdb-tool create,
                                        destroyed by file removal)

   Bridge             _uuid            from add-br to del-br
                                       (delete-and-recreate produces new UUID)

   Port               _uuid            from add-port to del-port
                                       (or from bridge create if auto-LOCAL)

   Interface          _uuid            from add-port to del-port
                                       (or from bridge create if auto-LOCAL)

   external_ids       (not a row;      operator-supplied; no GC; persists
                       a column with    until the operator removes the entry
                       string keys)     or the row is deleted
```

The lifetime hierarchy is strict: Interface lifetime is bounded by Port lifetime, Port lifetime is bounded by Bridge lifetime, Bridge lifetime is bounded by Open_vSwitch lifetime. A reader who has internalised the hierarchy can predict the GC behaviour for any combination of inserts and deletes.

## Misconception block

> **Misconception / Reality.**
>
> Misconception: "the row UUID is just a name; if I delete and re-add the same name, I get the same UUID."
>
> Reality: UUIDs are allocated by `ovsdb-server` at insert time using a 128-bit random value. The operator-supplied name is a separate column (e.g. `Bridge.name`) that the operator can use as a search key, but the UUID is independent of the name. Delete-and-recreate produces a new UUID even if the name is identical. R2.7 demonstrated this with `B1 != B3` for the same `name=br0`. A CMS that confuses name with UUID will produce broken cached state when an operator re-creates a bridge with the same name.

> **Misconception / Reality (second).**
>
> Misconception: "del-port leaves the Port and Interface rows in their tables; the operator must manually delete them."
>
> Reality: OVSDB garbage-collects orphan rows automatically on the next transaction commit. The schema declares `Bridge.ports` and `Port.interfaces` as strong references; when `del-port` removes the strong reference from the Bridge, the Port and Interface rows become unreachable from the root set and are GC'd. R2.7's `find Port name=dummy0` returning empty after `del-port` is the verbatim proof. A CMS that tries to "manually delete" the Port row directly will succeed (OVSDB allows it) but is doing redundant work; the cleaner pattern is to remove the Bridge's reference and let GC do the rest.

## What the reader should now be able to do

After R2.7 the reader can:

- Capture and compare row UUIDs in shell scripts using `ovs-vsctl --bare --columns=_uuid find <Table> <key>=<value>`.
- Use `--may-exist` and `--if-exists` to make `add-*` and `del-*` commands safely re-runnable without explicit existence checks.
- Predict that re-issuing the same `add-*` produces no UUID change while delete-and-recreate produces a new UUID.
- Read the `external_ids` column on any table and use it for operator metadata or CMS state caching, knowing that the column is OVS-ignored.
- Distinguish `_uuid` (row identity, allocated at insert) from `_version` (mutation counter, increments on every column update) from `system-id` (host identity, in `Open_vSwitch.external_ids`).
- Predict OVSDB GC behaviour: orphan strong-referenced rows are cleaned up automatically; orphan opaque-string-referenced rows (like patch peers) are not.

## A note on `--no-wait`

Two of the lab's `set` commands used `--no-wait`. That flag tells `ovs-vsctl` to commit the transaction and return immediately without waiting for `ovs-vswitchd` to acknowledge the configuration change. Without `--no-wait`, `ovs-vsctl` waits up to a few hundred milliseconds for the daemon to process the change and update the bridge state. For changes that the operator does not need to be visible immediately (annotations to `external_ids`, for example), `--no-wait` shaves the wait time; for changes that must be in effect before the next command (adding a port the next command will use), `--no-wait` should be omitted. The reader who is scripting an OVS configuration should err on the side of *not* using `--no-wait` unless they have a measured reason to use it.

## A note on the schema's distinction between root and non-root tables

The `vswitch.ovsschema` declares each table as `isRoot=true` or `isRoot=false`. Root tables (Open_vSwitch, QoS, Queue, Flow_Sample_Collector_Set) hold rows that exist independently; the OVSDB GC does not delete them when no other row references them. Non-root tables (Bridge, Port, Interface, Mirror, sFlow, NetFlow, IPFIX, Controller, Manager, Flow_Table, AutoAttach) hold rows that exist only as long as some root-reachable row references them. The reader who is debugging a "ghost row" issue (a row that should not exist) reads the schema's `isRoot` flag for the table to understand the GC rules.

The Open_vSwitch table is root and singleton (`maxRows=1, isRoot=true`). The Queue and QoS tables are root because the operator may want to define queues that are not currently attached to any port. Bridge is non-root: a bridge that no Open_vSwitch.bridges set references is GC'd; this is why `del-br` is sufficient (the operator does not need to clear the reference from Open_vSwitch.bridges manually).

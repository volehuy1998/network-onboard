## 19.1 - History: three eras of live migration in OVN

<!-- type: concept -->
<!-- step-source: S2 -->
<!-- Merrill Principle 1: open with the problem (live migration blackhole), not with a definition. -->
<!-- Gagné Event #3: opening sentence recalls MC_FLOOD and localnet from SDN Part 17 to activate prior knowledge. -->

In SDN Part 17 section 3.3, we saw that ovn-controller builds the MC_FLOOD multicast group on every chassis that participates in a logical switch. When a VM emits a broadcast frame, OVN replicates the packet to every chassis through a Geneve tunnel. That mechanism works reliably because the binding is static: one port belongs to exactly one chassis, known in advance from the SB Port_Binding. Live migration breaks that assumption. During the roughly 100 millisecond cutover phase, the same MAC address must be present on two chassis, and the same port changes ownership while traffic is still flowing. The networking problem of live migration is not "forwarding for a port", it is "forwarding for a port that is moving house".

Over the past ten years, OVN has gone through three eras solving this problem. Each era was born from a limitation of the previous one, and each left traces in the code and the bug tracker that survive to this day. Understanding the three eras is not just historical context, it is the key to reading today's `controller/physical.c`, where code from all three eras coexists and each block serves a different scenario.

### 19.1.1 Context: the live migration problem at the network layer

Live migration in OpenStack has two phases. The pre-copy phase: QEMU copies memory state from the source compute (call it `hv1`) to the destination compute (`hv2`) while the VM keeps running on hv1. Memory is large, so this phase takes from a few seconds to a few minutes. The cutover phase: QEMU pauses the VM on hv1, copies the final memory delta, then starts the VM on hv2. Cutover usually takes 50 to 200 milliseconds, and that is the window during which the network layer must hand the port over invisibly to the tenant.

Three layers must synchronize during the cutover window. First, ovn-northd translates the Logical_Switch_Port from Northbound into Port_Binding in Southbound, the logical layer. Second, ovn-controller on each chassis reads Port_Binding and installs the corresponding OpenFlow flows into `br-int`, the local control layer. Third, the physical fabric (top-of-rack switch, spine switch) relearns the MAC address of the VM through GARP or RARP, the data plane. If any layer is late, the packet goes the wrong way: it arrives at the old chassis after the VM has left, at the new chassis before the VM is ready, or at the wrong physical VLAN port and the switch drops it.

The key reason networking is harder than CPU here: you cannot "freeze and resume" linearly the way CPU register state does. Packets are in flight on the tunnel, packets sit in the `sk_buff` queue of the tap device, the ARP cache lives on the physical switch, and all of this is distributed state with no single point of control. OVN must choose between two strategies: change the binding BEFORE the VM is ready at the destination (risk of drop), or change the binding AFTER the VM has left the source (guaranteed drop). Era 1 tried both and lost both.

### 19.1.2 Era 1 (pre-22.09): single-chassis binding and unavoidable blackhole

Before OVN 22.09, the Southbound Port_Binding schema had only a single `chassis` column. A logical port could be claimed by exactly one chassis at any given time. This is the single-chassis binding model. During live migration, this model forced OVN to choose between two strategies, with no third option.

Strategy A, update the binding BEFORE cutover: ovn-northd switches Port_Binding.chassis from `hv1` to `hv2` as soon as Nova emits the `network-vif-plugged` event (when the tap device has been created on hv2 but the VM is still running on hv1). Consequence: all east-west traffic to the VM is redirected by OVN to hv2, but the VM is not yet ready on hv2 (memory is still being pre-copied). Result: the packet reaches the VM tap on hv2, the VM has not started, and the packet is dropped. The blackhole lasts for the entire final pre-copy and cutover, which can be several seconds for VMs with large memory.

Strategy B, update the binding AFTER cutover: OVN keeps Port_Binding.chassis = `hv1` until it receives a signal that the VM has started on hv2. Consequence: during the cutover window (50 to 200 ms), traffic still goes to hv1, but QEMU has paused the VM. ovn-controller on hv1 forwards the packet into the tap, but the tap no longer has a VM at the other end. The packet is silently dropped (the kernel drops the `sk_buff` because no process consumes it). The blackhole is shorter than under Strategy A, but it cannot be eliminated. The architecture forces it.

Ihar Hrachyshka (Red Hat, OVN maintainer) presented an experimental benchmark at OVS+OVN Conference 2022 ("Live migration with OVN"). On a test workload of 1607 ICMP packets sent across the migration window, pre-22.09 lost about 213 packets, which is 13.25 percent packet loss, with observed downtime of "multiple seconds". The exact numbers depend on VM memory size and fabric bandwidth, but the pattern is always the same: a gap that cannot be closed by tuning timing.

> **Key point:** Single-chassis binding in pre-22.09 OVN has a hard architectural limit. No timing strategy can eliminate the blackhole. The problem is not in the code, it is in the schema: Port_Binding has exactly one chassis, so at any moment exactly one chassis is "valid" to receive packets. The other chassis (Chassis), even though it also has the VM tap, is treated as foreign by OVN and the OpenFlow flows do not let packets reach it. To break the limit, you must change the schema, and that is what OVN 22.09 did.

### 19.1.3 Era 2 (OVN 22.09): multichassis binding kills blackhole, gives birth to duplicate

OVN 22.09, released in September 2022, extended the Southbound Port_Binding schema with a new column `additional_chassis` of type `set of Chassis`. A logical port can now be bound on `chassis` (MAIN) and on several chassis in `additional_chassis` (ADDITIONAL) at the same time. During live migration, ovn-northd sets `chassis=hv1` and `additional_chassis={hv2}` for the duration of the migration window, and clears `additional_chassis` after cutover completes.

The commit anchor for this change is [`7084cf437421`](https://github.com/ovn-org/ovn/commit/7084cf437421), titled "Always funnel multichassis port traffic through tunnels". Section 19.3.2 covers the notable side effect: the forwarding behavior in `consider_port_binding()` in `controller/physical.c` was changed so that all traffic to a port with `additional_chassis` non-empty is forced through the Geneve tunnel, even when a localnet port exists. At the same time, `consider_mc_group()` was modified so that the MC_FLOOD multicast group includes both the main chassis and the additional chassis. When the VM emits a broadcast, every chassis in both columns receives a clone.

Consequence at the fabric layer: during the cutover window, ovn-controller clones the packet to ALL chassis in `chassis + additional_chassis`. Both hv1 and hv2 have an active tap device (hv2 created its tap during the Prepare phase of live migration, before cutover), and both forward the packet into the tap. The blackhole disappears, but during the cutover window the VM temporarily exists on both hypervisors, and from OVN's point of view each side "is" that VM. A single packet sent toward the tenant becomes a duplicate.

Hrachyshka presented the Era 2 benchmark at OVS Conference 2022 on the same test workload: packet loss dropped to nearly zero, but a new phenomenon appeared: 94 duplicate packets out of 1153 received (about 8.2 percent). This is not a bug, it is the unavoidable consequence of the VM truly existing in two places during the cutover window.

> **Common misconception:** "OVN clones the packet, so the duplicate is caused by OVN sending extra packets."
>
> **Reality:** OVN clones exactly N packets to N chassis. Each chassis forwards exactly one packet into its own tap. The duplicate appears in the guest VM stack: during the cutover window, two QEMU instances coexist with the same MAC, and both decode the Ethernet frame into TCP/IP. Section §19.2.3 will refute in detail the hypothesis "the old packet in the virtio queue is transferred via the memory image" and prove the three-layer duplicate anatomy.

This is a local CAP theorem: you cannot simultaneously eliminate the blackhole (consistency: every packet reaches the unique correct VM) and ensure availability (every packet reaches the VM at all) while the binding is changing. OVN 22.09 chose availability, preferring no packet loss over no duplicate. Era 3 will break this tradeoff by adding a third signal.

### 19.1.4 Era 3 (activation-strategy=rarp): the VM signals cutover completion itself

To reduce duplicate to zero while keeping blackhole at zero, OVN needs a precise signal of the moment when the VM is "actually alive" at the destination. Not "started" (too early, memory not yet synchronized), and not "stable" (too late, hard to measure). The answer lies in a piece of QEMU code that has existed since 2009: the function `announce_self()`, written by Marcelo Tosatti, which automatically emits a RARP frame as soon as the VM resumes at the destination. Its original purpose was to let the physical switch relearn the MAC address on the new port. OVN 24.03 reuses this very signal as the marker "the VM is alive".

Commit [`ee20c48c2f5c`](https://github.com/ovn-org/ovn/commit/ee20c48c2f5c) ("Implement RARP activation strategy for ports", Ihar Hrachyshka, 2022-06-18) lets the operator set the option `activation-strategy=rarp` on a Logical_Switch_Port. When this is enabled, ovn-northd generates three additional OpenFlow "lock" flows on the ADDITIONAL chassis: a flow that drops traffic to the tap, a flow that drops traffic from the tap, and a flow that matches `dl_type=0x8035` (RARP ethertype) and sends a controller packet-in to `ovn-controller pinctrl`. Section §19.5.1 analyzes these three flows in detail.

Pipeline idea: while hv2 is ADDITIONAL, the tap of the VM on hv2 is locked by the two drop flows. Even if the cloned packets arrive, none of them reach the guest. The moment QEMU resumes the VM on hv2 and emits the RARP, the third flow captures the frame, pinctrl sets `Port_Binding.options:additional-chassis-activated="true"`, and ovn-northd re-translates: the three lock flows are removed, `chassis` is promoted from hv1 to hv2, and `additional_chassis` is cleared. From that moment, only hv2 receives packets. The time from "VM resume" to "flow updated" is roughly 4 to 10 ms on a well-tuned cluster.

Hrachyshka's benchmark on the same workload: packet loss drops to about 4 ms (0 to 2 packets out of 1607), duplicate equals zero. This is the current state of the art. The deeper reasons, why RARP rather than GARP, why depending on QEMU is acceptable, are unpacked in §19.5 with four architectural arguments.

### 19.1.5 Comparison table of the three eras (packet loss, duplicate, complexity, dependency)

| Era | OVN version | Duplicate behavior | Packet loss (benchmark OVS Con 2022) |
|---|---|---|---|
| 1, single-chassis | pre-22.09 | 0 duplicate | About 13.25 percent (213 of 1607 packets). "Multiple seconds" downtime. |
| 2, multichassis | 22.09 to 24.03 | About 8.2 percent duplicate (94 of 1153 packets) | Near zero (only on timing race) |
| 3, activation-strategy=rarp | 24.03+ | 0 duplicate | About 4 ms (0 to 2 packets out of 1607) |

Read the table along the tradeoff axis: from top to bottom, OVN reduces both loss metrics (loss and duplicate), but pays with a new dependency. Era 1 has no dependency beyond the OVN pipeline. Era 2 depends on the new `additional_chassis` column in the Southbound schema and requires OVN 22.09+ on every chassis (no version mixing). Era 3 adds a dependency on QEMU `announce_self()` to emit RARP. DPDK vhost-user ports cannot hook `announce_self()`, which is why Launchpad bug #2092407 exists and why §19.5.5 presents it as an ANSI Z535.6 WARNING.

Each later era does not replace the previous one in the codebase. Code from all three coexists in today's `physical.c` and `binding.c`. The operator selects the era through the `activation-strategy` option on the Logical_Switch_Port: not set means Era 2 (multichassis binding without activation flows); set to `rarp` means Era 3; a cluster running OVN earlier than 22.09 is forced into Era 1. Knowing the three eras well is required to read ovn-controller logs correctly when debugging an unusual migration. The log can tell you which era your cluster is in.

## 19.2 - Multichassis port binding lifecycle (mechanism, timeline, duplicate anatomy)

Section 3.1 showed that OVN 22.09 extended the Port_Binding schema with the `additional_chassis` column. This section goes deeper into three mechanisms: (1) the claim rule (which chassis becomes MAIN, which chassis becomes ADDITIONAL, and based on which option); (2) the ten-milestone T0 to T9 timeline of a full live migration, comparing "no activation-strategy" with "rarp"; (3) the anatomy of which layer actually creates the duplicate, a common misconception that must be refuted with source code before the correct model is built.

Terminology note (ISO 704 monosemy): this document uses only the form "multichassis port" (not "multi-chassis port", not "multi chassis"). "additional_chassis" keeps its underscore exactly as the schema column name. "live migration" has no hyphen. Other variants appear only inside verbatim quotes from commit messages or source code.

### 19.2.1 Claim mechanism: requested-chassis = "hv1,hv2"

The OVN claim rule lives in the function `lport_can_bind_on_this_chassis()`, defined in [`controller/lport.c`](https://github.com/ovn-org/ovn/blob/main/controller/lport.c). It is called by `binding.c` whenever ovn-controller processes a Port_Binding update. The function reads the `requested-chassis` option (a comma-separated list of chassis names), compares it with the current chassis, and returns one of three enum values:

```c
enum can_bind
lport_can_bind_on_this_chassis(const struct sbrec_chassis *chassis_rec,
                               const struct sbrec_port_binding *pb)
{
    if (pb->requested_chassis == chassis_rec) {
        return CAN_BIND_AS_MAIN;
    }
    for (size_t i = 0; i < pb->n_requested_additional_chassis; i++) {
        if (pb->requested_additional_chassis[i] == chassis_rec) {
            return CAN_BIND_AS_ADDITIONAL;
        }
    }
    const char *requested = smap_get(&pb->options, "requested-chassis");
    if (!requested || !strcmp("", requested)) {
        return CAN_BIND_AS_MAIN;
    }
    /* Parse CSV: first match → MAIN; subsequent matches → ADDITIONAL;
       no match → CANNOT_BIND. */
    char *tokstr = xstrdup(requested);
    char *chassis, *save_ptr = NULL;
    enum can_bind can_bind = CAN_BIND_AS_MAIN;
    for (chassis = strtok_r(tokstr, ",", &save_ptr); chassis;
         chassis = strtok_r(NULL, ",", &save_ptr)) {
        if (!strcmp(chassis, chassis_rec->name) ||
            !strcmp(chassis, chassis_rec->hostname)) {
            free(tokstr);
            return can_bind;
        }
        can_bind = CAN_BIND_AS_ADDITIONAL;
    }
    free(tokstr);
    return CANNOT_BIND;
}
```

Read the algorithm line by line: with `requested-chassis="hv1,hv2"`, chassis `hv1` matches in the first iteration and the function returns `CAN_BIND_AS_MAIN`. Chassis `hv2` matches in the second iteration (by then `can_bind` has been set to `CAN_BIND_AS_ADDITIONAL` in the previous loop step) and returns `CAN_BIND_AS_ADDITIONAL`. Chassis `hv3` does not match and returns `CANNOT_BIND`. The rule "the first chassis is MAIN, later chassis are ADDITIONAL" is an invariant of the function. There is no option-level switch to change priority order.

ovn-northd translates the option from Northbound to Southbound one to one: `Logical_Switch_Port.options:requested-chassis="hv1,hv2"` in NB becomes `Port_Binding.options:requested-chassis="hv1,hv2"` in SB. ovn-northd also resolves chassis names into UUIDs and fills two new schema columns: `requested_chassis` (scalar, UUID of hv1) and `requested_additional_chassis` (set of UUIDs, containing the UUID of hv2). The function above runs faster by reading UUIDs directly before falling back to CSV parsing.

Next question: which Nova operation actually sets this option? Not every VM lifecycle event creates multichassis state. The table below, verified against the source `nova/compute/manager.py`, lists nine common operations. Only live migration creates a `requested-chassis` value in list form:

| Nova operation | requested-chassis set? | Value form |
|---|---|---|
| `server create` (boot) | yes | `"<dest_host>"` (single chassis) |
| `server reboot` | not applicable | unchanged, MAIN remains the current hypervisor |
| `server pause` and `unpause` | not applicable | unchanged |
| `server stop` and `start` | not applicable | unchanged |
| `server migrate` (cold) | yes (after completion) | `"<new_host>"` (single) |
| `server live-migration` (pre-copy) | yes | **`"<src>,<dst>"` (list, multichassis)** |
| `server evacuate` | yes | `"<new_host>"` (single, after reschedule) |
| `server resize` | yes (if the host changes) | `"<new_host>"` (single) |
| `server shelve` and `unshelve` | not applicable | port released on shelve, rebound on unshelve |

> **Common misconception:** "`nova boot` of a VM on hv1 also creates multichassis state because the port has chassis = hv1."
>
> **Reality:** `requested-chassis="hv1"` is a **single** chassis. The CSV parsing function returns `CAN_BIND_AS_MAIN` on the very first iteration, and there is no second iteration to fall through to `ADDITIONAL`. The `additional_chassis` column in Port_Binding stays empty. Only live migration sets the list form (`"hv1,hv2"`), and only during the migration window. Right after cutover completes, Nova sends an API update to Neutron, ovn-northd removes the old element from the list, and the option goes back to single form.

### 19.2.2 Timeline T0 to T9: the journey of a packet through the migration window

The ten milestones below cover the full span of an OpenStack live migration, from the moment the user calls the `nova server-migration create` API to the moment the port is fully cleaned up. The table presents two scenarios in parallel: no `activation-strategy` (Era 2, with duplicate) and `activation-strategy=rarp` (Era 3, zero duplicate), so that the differences are clear.

| Phase | Approximate time | Behavior (no activation-strategy) | Behavior (activation-strategy=rarp) |
|---|---|---|---|
| T0, API request | 0 | Nova conductor receives the request, selects dst `hv2` | same |
| T1, Claim dst resource | +0.5 s | Nova scheduler reserves CPU and RAM on hv2 | same |
| T2, Prepare (tap created early) | +1 s | **hv2 creates the tap device**, Neutron API sends a port update with `requested-chassis="hv1,hv2"` | same, plus ovn-northd generates 3 "lock" flows on hv2 (§19.5.1) |
| T3, ovn-controller applies | +1 to 3 s | forwarding flow on hv2 is active, but the tap is empty (VM still on hv1) | flow is active, but 2 drop flows plus 1 flow matching 0x8035 keep the hv2 tap locked |
| T4, Pre-copy memory | +3 s to several minutes | QEMU copies memory pages from hv1 to hv2; VM still runs on hv1 | same |
| T5, Cutover begin | +several minutes | QEMU pauses the VM on hv1 | same |
| T6, Final delta copy | +about 50 to 100 ms | QEMU copies the memory delta and updates device state | same |
| T7, Resume VM on hv2 | +about 100 ms | VM starts on hv2, both hv1 and hv2 have traffic into their tap, **DUPLICATE during this phase** | VM starts, but the 2 drop flows still lock the hv2 tap, so hv2 does not forward yet |
| T7.5, QEMU announce_self | +about 101 ms | (not used; QEMU still emits RARP, but OVN ignores it) | **Pinctrl captures the 0x8035 RARP** and sets `options:additional-chassis-activated="true"` |
| T8, Binding update | +about 105 to 200 ms | Nova event `network-vif-plugged`; Neutron triggers an update, and ovn-northd removes `hv1` from `requested-chassis` | ovn-northd re-translates: removes the 3 lock flows, promotes `chassis=hv2`, clears `additional_chassis` |
| T9, Cleanup src | +about 500 ms | hv1 removes the tap, flows are reclaimed | same |

Two details are easy to miss but cause many real-world bugs:

**(a) The hv2 tap is created at T2, before memory copy starts.** This is not an OVN decision but a Nova libvirt driver decision. The driver needs the tap ready so that libvirt can attach virtio-net to the new VM instance the moment it resumes. Consequence: both hv1 and hv2 have an active tap throughout phases T2 to T8. This means `requested-chassis` can safely contain two chassis throughout the four-plus minutes of pre-copy. The window is not the 100 ms cutover.

**(b) The Nova event `network-vif-plugged` does NOT wait for ovn-controller to apply the flow.** Nova emits this event when Neutron returns a successful port update, which means NB DB has been updated. SB DB may not have finished translating yet, and ovn-controller on hv2 may not have installed the flow yet. Two Launchpad bugs exploit this race:

- [Launchpad #1933517](https://bugs.launchpad.net/neutron/+bug/1933517): in a busy cluster, SB DB translation is several hundred ms slow. Nova sees the event, resumes the VM, and the first packet is dropped because the flow is not ready. Fix: Neutron waits for ovn-controller to confirm via `external_ids:ovn-installed`.
- [Launchpad #2069718](https://bugs.launchpad.net/neutron/+bug/2069718): a variant of the bug above in a multichassis scenario, surfacing when a cluster has just been upgraded from ML2/OVS to ML2/OVN.

> **Key point:** The multichassis window is not the 100 ms cutover. It runs from T2 (dst tap created, about 1 s after the API request) to T8 (binding update, several minutes later, depending on memory size). During that whole time, cloned traffic flows through both chassis. Benchmarking, monitoring, and debugging must look at the entire long window, not only the final 100 ms.

### 19.2.3 Three-layer duplicate anatomy (misconception refuted)

When a tenant reports "I see the ping reply for the same ICMP request twice during live migration", the engineer's next question is: where does that duplicate come from? The answer drives the debug strategy. A fabric duplicate calls for checking OVS flows and tunnel counters; a guest duplicate can only be fixed with activation-strategy. A careful analysis shows that there are exactly three physical layers where duplicate can occur, and only the third is the actual source.

**A common misconception to refute first:** "QEMU pre-copy memory transfer also copies the contents of the virtio queue from hv1 to hv2, so packets not yet read on hv1 reappear on hv2 after cutover, and that is the source of the duplicate."

This hypothesis sounds plausible but is architecturally wrong. Verify it with the QEMU source [`hw/net/virtio-net.c`](https://github.com/qemu/qemu/blob/master/hw/net/virtio-net.c), function `virtio_net_can_receive()`:

```c
static bool virtio_net_can_receive(NetClientState *nc)
{
    VirtIONet *n = qemu_get_nic_opaque(nc);
    VirtIODevice *vdev = VIRTIO_DEVICE(n);
    VirtIONetQueue *q = virtio_net_get_subqueue(nc);
    if (!vdev->vm_running) {
        return false;
    }
    ...
}
```

When the VM pauses on hv1 (during T5 cutover), `vdev->vm_running` flips to `false`. Every packet that arrives at the hv1 tap during this phase is not read by virtio out of the tap's `sk_buff` queue (a kernel object). The packet sits in the tap device queue, NOT in the VM's memory image. When QEMU migrates memory, it migrates the guest kernel memory (including the buffers of the virtio-net driver inside the guest), but the tap device is a host kernel object, not a guest object, and is not migrated. After cutover, the hv1 tap is torn down with the VM. Every packet sitting in the hv1 tap's sk_buff queue is dropped by the kernel when the tap descriptor is closed.

Conclusion: virtio queue transfer is not the source of duplicate. The real source is at another layer.

Three-layer analysis:

**Layer 1, fabric and tunnel:** ovn-controller on the source chassis clones the packet to `chassis + additional_chassis` over Geneve tunnels. Concretely, the packet goes through `consider_mc_group()` (unicast to a single chassis if the packet is unicast, multicast set if it is broadcast) and runs the action `set_field(tun_id=...)` plus `output:bond1.vlanX` for each tunnel destination. The total number of Geneve packets on the wire equals the number of chassis in the set; each packet carries different tunnel metadata (different dst IP). THIS IS NOT A DUPLICATE; this is replication by design, where each copy goes to a different chassis.

**Layer 2, OVS and tap on the same hypervisor:** when a Geneve packet arrives at hv1 or hv2, OVS decapsulates it, the logical switch ingress pipeline runs, and the `ls_in_l2_lkup` table looks up the destination MAC and outputs the packet either into a logical patch or into a tap. For unicast to the VM, the output goes into exactly ONE tap. For broadcast (ARP, DHCP), OVS executes a multicast set on the local chassis but only across the local ports of the logical switch, and never outputs the same packet to the same tap more than once. Each hypervisor receives exactly one copy per cloned flow. THIS IS NOT A DUPLICATE.

**Layer 3, the VM guest TCP/IP stack:** this is where the duplicate truly originates. During the cutover window (T7 to T8), TWO QEMU instances coexist: the old instance on hv1 still has the VM process (it is not killed until T9 cleanup), and the new instance on hv2 has just resumed. Both share the same MAC and the same IP. A cloned packet arrives at hv1, OVS forwards it into the hv1 tap, the virtio-net of the hv1 VM (if not yet paused) hands it to the guest kernel, and the guest delivers it to the application. A cloned packet arrives at hv2, OVS forwards it into the hv2 tap, the virtio-net of the hv2 VM (resumed at T7) hands it to the guest kernel, and the guest delivers it to the application. Two copies of the same packet reach the same application at LAYER 7 of the same IP and MAC. From the tenant application's point of view, this is a duplicate.

Era 3 (activation-strategy=rarp) does not reduce the duplicate by changing layer 1 or layer 2; both layers still clone exactly as in Era 2. It blocks layer 3 by locking the hv2 tap (preventing packets from reaching virtio-net) until the VM has fully resumed AND emitted RARP. The moment RARP flies out (T7.5), OVN simultaneously promotes the binding, the hv1 tap is orphaned, and the hv2 tap is unlocked, so only one VM instance receives packets. Duplicate goes to zero.

## 19.3 - enforce_tunneling and the six packet-path scenarios on localnet

<!-- type: concept+reference -->
<!-- step-source: S3 -->
<!-- DITA 1.3 Reference type for the six-scenario table (pure lookup); Concept type for the explanation. -->
<!-- ANSI Z535.6 WARNING callout §19.3.5: tcpdump on bond0 is silent even though the VM is running. -->

§19.1 and §19.2 finished the picture: when a port has `additional_chassis` non-empty, OVN takes the architectural decision to force traffic through a Geneve tunnel instead of localnet. But where is that decision IMPLEMENTED in the code, and does the actual behavior across all six traffic scenarios (East-West, North-South, egress, by steady-state versus migrate) match intuition? Section 19.3 answers these two questions by reading the source `controller/physical.c` and the historical commit `7084cf437421`, the commit that changed "single-direction tunneling" to "bidirectional enforcement".

A key point to remember before going further: there are two different levels of "go through the tunnel". Level 1 is the regular forwarding flow rule, where OVN picks tunnel or localnet based on topology. Level 2 is **enforcement**, where OVN forces the tunnel even when the level 1 rule says localnet would be fine. Section 19.3 focuses on level 2, because that level is the mechanism that eliminates blackhole, while at the same time producing counterintuitive behaviors that engineers face during debugging.

### 19.3.1 Code anchor: consider_port_binding() in controller/physical.c

The branch point lives in `consider_port_binding()`, the function that builds the OpenFlow flow for a specific port binding. When the port is PORT_LOCALNET (reachable through a localnet port), the code takes the following branch (excerpt from `controller/physical.c` OVN branch-24.03, lines 1939 to 1968):

```c
    } else if (access_type == PORT_LOCALNET) {
        /* Remote port connected by localnet port */
        /* Table 40, priority 100.
         *
         * Implements switching to localnet port. Each flow matches a
         * logical output port on remote hypervisor, switch the output port
         * to connected localnet port and resubmits to same table. */

        ofpbuf_clear(ofpacts_p);
        match_outport_dp_and_port_keys(&match, dp_key, port_key);
        put_load(localnet_port->tunnel_key, MFF_LOG_OUTPORT, 0, 32, ofpacts_p);
        put_resubmit(OFTABLE_LOCAL_OUTPUT, ofpacts_p);
        ofctrl_add_flow(flow_table, OFTABLE_LOCAL_OUTPUT, 100,
                        binding->header_.uuid.parts[0],
                        &match, ofpacts_p, &binding->header_.uuid);

        enforce_tunneling_for_multichassis_ports(ld, binding, chassis,
                                                 chassis_tunnels,
                                                 mff_ovn_geneve, flow_table,
                                                 if_mgr);
        /* No more tunneling to set up. */
        goto out;
    }
```

Read it sequentially: OVN **always** installs a priority-100 localnet flow for the remote port (this is the default). It then calls `enforce_tunneling_for_multichassis_ports()`. If the datapath has any multichassis port (`shash_is_empty(&ld->multichassis_ports) == false`), this function installs an additional **priority-110** flow in the same `OFTABLE_REMOTE_OUTPUT` that overrides the priority-100 flow. The priority-110 flow encapsulates and outputs through the Geneve tunnel (see the body of `enforce_tunneling_for_multichassis_ports()` in `controller/physical.c`, OVN branch-24.03 lines 1422 to 1476). Higher priority wins. The result: traffic to a multichassis port goes through the tunnel, traffic to a regular port stays on localnet. The mechanism is safe because `goto out` skips the generic tunnel handling below, avoiding two conflicting action sequences being installed at the same time.

> **Key point:** `enforce_tunneling_for_multichassis_ports()` does not delete the localnet flow, it installs a higher-priority **override**. Operational meaning: when `nova live-migrate` finishes and ovn-controller clears `additional_chassis`, the priority-110 flow disappears and the priority-100 (localnet) flow goes back into effect without needing reinstall. This is the "state change without interruption" mechanism. The OpenFlow pipeline always has a valid path during the transition.

The directive `NB_Global.options:always_tunnel`, if set to `"true"`, makes all OVN traffic go through tunnels regardless of localnet (the `ctx->always_tunnel` condition in the code). By default, Neutron ML2/OVN does NOT set this flag (to use the fabric for performance), so the realistic OpenStack scenario always falls into the `PORT_LOCALNET` branch with selective enforcement.

### 19.3.2 Commit anchor: 7084cf437421, enforcement on both ingress and egress

Before OVN 22.09, enforcement applied to one direction only: traffic **to** a multichassis port was forced through the tunnel, but traffic **from** a multichassis port was still allowed over localnet. Commit `7084cf437421aedac019a4151d0b6ce9208e695a` (author Ihar Hrachyshka, authored 2022-06-24, committed 2022-07-07, Acked-by Mark Michelson, merged by Numan Siddique) changed this behavior. Verbatim commit body from the GitHub API:

> Before the patch, for switches with a localnet port,
>
> - traffic to a multichassis port was funneled through tunnels; but
> - traffic from a multichassis port was sent through localnet.
>
> This is not optimal because:
>
> - bidirectional sessions are sent through two separate paths;
> - leaking multichassis traffic to physical makes upstream switch flip port's mac address location.
>
> One of the effects of this behavior is prolonged network downtime when live migrating a VM attached to a physical network (compared to downtime observed for pure geneve networks).
>
> As per my unscientific local testing, this patch reduces network downtime during OpenStack VM live migration from 1.0s+ to 0.05-0.10s, on par with geneve networks.
>
> — Ihar Hrachyshka, OVN commit `7084cf437421`

Two important details in this excerpt. First, the commit identifies the **mechanism that creates the blackhole**: the upstream physical switch learns the VM MAC from an egress packet over localnet, so the FDB entry points to hv2. When a reply from the outside comes in, the switch forwards it to hv2, but the VM is still running on hv1, so the packet is dropped. This is the "MAC flip" that §19.1.2 analyzed as the 13.25 percent blackhole loss. Second, the benchmark "1.0s+ to 0.05 to 0.10 s" matches the OVS Con 2022 measurement (the comparison table in §19.1.3); two independent sources confirm the same magnitude of improvement.

This commit fixes the original commit `10398c1f51d54d5c3f8ec391a0f8de0bc76a927d` "Clone packet to all port chassis" (the OVN 22.09 commit that introduced multichassis). OVN 22.09.0 has both commits: one enables multichassis, the other completes enforcement. Operational consequence: if you build OVN from the exact tag `v22.09.0`, you already have the correct behavior. There is no OVN release that is "half-baked" with multichassis-incoming but localnet-outgoing.

### 19.3.3 consider_mc_group(): why ARP broadcast still uses localnet

A subtle question: enforcement applies to unicast traffic (dst MAC equals MAC_VM). What about broadcast and multicast (ARP request, DHCP, IGMP, Neighbor Solicitation)? The answer lies in `consider_mc_group()`, the function that builds flows for multicast groups (MC_FLOOD, MC_UNKNOWN). An important comment from `controller/physical.c` (OVN branch-24.03 lines 2126 to 2145; line numbers shift across versions, so use the function name `consider_mc_group` as a stable anchor):

```c
/* Go through all of the ports in the multicast group:
 *
 *  - For remote ports, add the chassis to 'remote_chassis' or
 *    'vtep_chassis'.
 *
 *  - For local ports (other than logical patch ports), add actions
 *    to 'ofpacts' to set the output port and resubmit.
 *
 *  - For logical patch ports, add actions to 'remote_ofpacts'
 *    instead. (If we put them in 'ofpacts', then the output
 *    would happen on every hypervisor in the multicast group,
 *    effectively duplicating the packet.) */
```

Multicast logic: if the datapath has a localnet port, broadcast traffic naturally reaches every chassis through the physical fabric (the switch floods the VLAN). OVN does **not** need to clone broadcast traffic to remote chassis through tunnels, that would create duplicates. So for a datapath with localnet, `remote_chassis` only contains chassis that are **not** reachable through the fabric (for example, chassis in a different availability zone connected through a transit switch). For an overlay-only datapath, broadcast must be cloned through tunnels to every remote chassis (the classic MC_FLOOD, see SDN 17.0 §17.2).

Result: **an ARP request from a VM still goes through localnet in steady state**, even when the VM is in multichassis binding. This detail confuses many engineers during debugging. A VM on hv1 in multichassis with hv2: `tcpdump` on bond1.vlan10 (tunnel) shows no ARP request; `tcpdump` on bond0 (provider) shows the ARP request normally. The behavior is correct by design, not a bug. Side effect: the physical switch learns MAC_VM from ARP (because ARP still goes through localnet), but with unicast enforcement, when the upstream replies, the reply travels through the tunnel to the additional_chassis hypervisor and is never affected by the switch FDB. The two planes are completely separate.

### 19.3.4 Six-scenario table: E-W, N-S, egress, by steady or migrate

To summarize traffic behavior in every situation, the six-scenario table below analyzes along two axes: traffic direction (E-W tenant-internal, N-S inbound from the internet, egress outbound to the internet) versus port state (Steady means not migrating, Migrate means additional_chassis is non-empty). The third column, "path chosen", is the most important conclusion. The "rationale" column links to the code anchor. The "tcpdump capture point" column tells the engineer where to capture to see the traffic.

| Scenario | Direction | Path chosen | Rationale (code anchor) | tcpdump capture point |
|---|---|---|---|---|
| E-W steady | VM hv1 to VM hv3, both single-chassis | Tunnel (Geneve) | `consider_port_binding()`, the `!PORT_LOCALNET` branch (pure overlay E-W) | bond1.vlanX tunnel on hv1 OR hv3 |
| E-W migrate | VM hv1 (migrating to hv2) to VM hv3 | Tunnel hv1 to hv3 plus **override 110 from hv3 to hv1 and hv2** | `enforce_tunneling_for_multichassis_ports()` installs flow 110 for BOTH chassis | bond1.vlanX on hv1 and hv2 (count 2 Geneve packets per ICMP) |
| N-S steady | Internet to VM hv1 (provider network) | Localnet (bond0 hv1) | `PORT_LOCALNET` branch, no multichassis | bond0 hv1 |
| N-S migrate | Internet to VM hv1 (migrating to hv2) | Localnet bond0 to br-int, then **tunnel override 110** | flow 110 on hv1 matches outport=VM_LSP, encapsulates Geneve to hv2 | bond0 hv1 silent; bond1.vlanX hv1 sees 2 Geneve out |
| Egress steady | VM hv1 to internet | Localnet (bond0 hv1) | `consider_port_binding()` plus MAC learning at the fabric | bond0 hv1 |
| Egress migrate | VM hv1 (migrating to hv2) to internet | **STILL localnet bond0 hv1** | Egress is not "remote port output" and does not go through `enforce_tunneling_for_multichassis_ports()` | bond0 hv1 |

Two rows worth noting are N-S migrate and Egress migrate. The common wrong intuition: "during migrate all traffic must go through the tunnel because the port is multichassis." In reality, commit 7084cf437421 only enforces tunneling for **traffic GOING TO a multichassis port** (ingress toward the VM), not for egress **FROM** the VM. Egress still goes over localnet normally. Reason: OVN cannot enforce egress through the tunnel because once a packet leaves the local port, local MAC learning has already happened, and forcing egress through the tunnel would break anti-spoof and FDB. The commit only fixes the ingress direction because ingress is what causes the MAC flip on the upstream switch.

> **Key point:** The six-scenario table above is the cognitive schema that summarizes section 19.3. If you only remember one table from all of Part 19.0, remember this one. It maps every confusing tcpdump observation to a deterministic pipeline behavior. When you see any unexpected capture during migrate, compare it with the table before declaring a bug.

### 19.3.5 WARNING: tcpdump on bond0 is silent even though the VM is running on hv1

The "N-S migrate" row in section 19.3.4 reveals a serious operational trap that deserves a clear warning per the five-component ANSI Z535.6 standard:

> **WARNING, silent tcpdump on the provider bond**
>
> **Danger:** During the migration window, `tcpdump -i bond0 ether host MAC_VM` on hv1 may be **completely silent** for ingress traffic toward the VM for several hundred milliseconds, EVEN THOUGH the VM is still running on hv1 and processing traffic normally.
>
> **Consequence:** An ops engineer sees no packets on bond0 and reaches the wrong conclusion: "VM lost network" or "VM is down". They might roll back the migration unnecessarily, or escalate a non-existent incident, leading to real downtime and a loss of confidence in live migration. In a worse case, ops might restart hv1 or release the port binding, which causes a real outage.
>
> **Prevention:** (1) Always capture AT the tunnel endpoint `bond1.vlanX`. That is where the traffic actually flows during the migration window. (2) Use Layer 4 of Lab 1 (the Lab 1 section below) to confirm traffic is going through Geneve rather than truly being silent. (3) Cross-check `ovn-sbctl list Port_Binding $LSP`. If `additional_chassis` is non-empty, this is a valid migration state, not an incident.
>
> **Warning symbol:** (ISO 7010 W001, general warning)

## 19.4 - Geneve overhead + PMTUD pipeline + bug FDP-620

<!-- type: concept+reference -->
<!-- step-source: S3 -->
<!-- RFC-first citations: RFC 8926 (Geneve), RFC 1191 (PMTUD). -->
<!-- Acronym expand lần đầu: PMTUD (Path MTU Discovery), TLV (Type-Length-Value), FDP (Red Hat Fast Datapath bug tracker). -->

Mục 19.3 dựng cơ chế "enforce tunnel" nhưng tunnel Geneve luôn kèm overhead gói tin. Khi VM gửi gói kích thước gần MTU physical, overhead 58 byte đẩy frame sau encap vượt MTU → rơi vào pipeline PMTUD của OVN. Mục này giải thích toán học của overhead, chi tiết pipeline table 41+42, và bug FDP-620, bug ngầm trong chính pipeline PMTUD biến mitigation thành bẫy. Citations tuân RFC-first: RFC 8926 cho Geneve, RFC 1191 cho PMTUD, trích từ rfc-editor.org (không qua mirror).

Ba acronym dùng lần đầu (tuân ISO 704 definition): **PMTUD** (Path MTU Discovery, RFC 1191), **TLV** (Type-Length-Value, cấu trúc option trong Geneve base header), **FDP** (Fast Datapath, prefix bug tracker của Red Hat cho các bug về OVS/OVN datapath, dùng trên issues.redhat.com).

### 19.4.1 Geneve overhead toán học chính xác 58 byte

Giá trị "~50 bytes" thường xuất hiện trong documentation là ước lượng quen thuộc. Giá trị chính xác là **58 bytes**, phân rã theo table 5 thành phần (DITA Reference type, dùng để tra cứu, không phải giảng):

| Thành phần | Kích thước (byte) | Nguồn chuẩn |
|---|---|---|
| Outer Ethernet header | 14 | IEEE 802.3 |
| Outer IPv4 header (no options) | 20 | RFC 791 §3.1 |
| Outer UDP header (dst port 6081) | 8 | RFC 768 + IANA port registry |
| Geneve base header (Ver, Opt Len, Flags, Protocol Type, VNI 24-bit) | 8 | RFC 8926 §3.4 |
| OVN Geneve option TLV (Option Class 0x0102, type 0x80, 32-bit data) | 8 | IANA Geneve Option Class registry |
| **Tổng overhead** | **58** | |

OVN sử dụng **một TLV option duy nhất** với IANA Option Class **0x0102** (đăng ký cho "Open Virtual Networking" trong IANA Geneve Option Class registry, xem RFC 8926 §3.5 cho semantics). TLV này mang 32-bit data: 1 bit reserved + 15-bit logical ingress port + 16-bit logical egress port. Thêm VNI 24-bit trong Geneve base header mang logical datapath ID. Tổng metadata OVN cần: 24 + 15 + 16 = **55 bit**. Đây chính là lý do kiến trúc OVN không thể dùng VXLAN (chỉ 24-bit VNI, RFC 7348) hay GRE (chỉ 32-bit key, RFC 2890), không đủ chỗ cho logical port metadata. Geneve với TLV extension là giải pháp duy nhất đáp ứng tại thời điểm OVN được thiết kế (2015).

Table tính effective MTU theo ba mức physical MTU phổ biến trong triển khai OpenStack:

| Physical MTU (byte) | Effective tunnel MTU (byte) | Context triển khai |
|---|---|---|
| 1500 | 1442 | Default Ethernet, không có jumbo frame |
| 1600 | 1542 | Mitigation thấp hơn, dùng khi switch/NIC không hỗ trợ đầy đủ jumbo |
| 9000 | 8942 | Jumbo frame đầy đủ, Red Hat OSP, Canonical OpenStack khuyến nghị cho Tenant + Storage + Internal API |

Neutron ML2/OVN driver tự động tính effective MTU bằng công thức `instance_mtu = physical_mtu - 58` và set vào Neutron port API khi tạo port. Nếu dùng DHCP, Neutron gửi DHCP option 26 (Interface MTU) cho guest, guest tự động nhận MTU đúng mà không cần intervention.

### 19.4.2 PMTUD pipeline trong OVN (table 41 + table 42)

Khi VM gửi gói IP kích thước 1500 byte trên physical MTU 1500, sau encap Geneve thành 1558 byte, vượt MTU physical. OVN không fragment (vì datapath có `DF=1` trên outer IP), thay vào đó sinh ICMP error báo về sender để sender tự giảm MSS. Pipeline chịu trách nhiệm nằm ở hai OpenFlow table (số table có thể đổi giữa OVN phiên bản; minh họa dưới dùng OVN 23.09.x, tra bằng `ovn-sbctl list Logical_Flow` trên production để kiểm chứng):

Table 41, `OFTABLE_OUTPUT_LARGE_PKT_DETECT`: match bất kỳ gói tin có `outport = MC_GENEVE_TUNNEL`, action `check_pkt_larger(1442)` → nếu gói > 1442 byte, set register `reg9[1] = 1`, rồi resubmit sang table 42.

Table 42, `OFTABLE_OUTPUT_LARGE_PKT_PROCESS`: match `reg9[1] = 1`, action `reply_imcp_error_if_pkt_too_big()`, chi tiết ở mục 19.4.3 tiếp theo. Sau khi sinh ICMP error, drop gói tin gốc (không forward ra tunnel) vì sender sẽ retry với MSS nhỏ hơn.

> **Chủ đề trọng tâm:** Pipeline PMTUD chỉ kích hoạt khi gói tin > effective MTU. Nếu physical MTU = 9000 (jumbo), effective MTU = 8942, gần như mọi traffic VM (< 1500 byte) không trigger pipeline này → `reg9[1]` luôn = 0 → `reply_imcp_error_if_pkt_too_big()` không chạy. Đây là lý do jumbo frame là "mitigation" cho bug FDP-620 (mục 19.4.4), không phải vì bug được fix, mà vì code path không được thực thi.

### 19.4.3 Action sequence: reply_imcp_error_if_pkt_too_big() chi tiết

Hàm `reply_imcp_error_if_pkt_too_big()` (lưu ý: tên hàm upstream có typo `imcp` thay vì `icmp` — tác giả Ihar Hrachyshka có ghi comment `NOTE(ihrachys) This reimplements icmp_error as found in` trên line 1905 của `controller/physical.c` main branch; typo còn tồn tại đến hiện tại dù được biết) sinh ICMPv4 Type 3 Code 4 ("Destination Unreachable, Fragmentation Needed and Don't Fragment was Set", RFC 792 + RFC 1191 cho extension MTU field). Chuỗi action trong logical flow như sau (trích từ OVN 23.09.x, table 42):

```
REGBIT_PKT_LARGER = 0          // clear flag để tránh loop
REGBIT_EGRESS_LOOPBACK = 1     // đánh dấu gói được loopback vào ingress pipeline
eth.dst ↔ eth.src              // hoán đổi MAC: gói mới có src=VM_MAC, dst=sender_MAC
ip.dst ↔ ip.src                // hoán đổi IP: gói mới có src=VM_IP, dst=sender_IP
ip.ttl = 255                   // reset TTL cho gói ICMP mới
icmp4.type = 3                 // ICMP Destination Unreachable
icmp4.code = 4                 // Fragmentation Needed and DF set
icmp4.frag_mtu = 1442          // MTU hiệu dụng — sender sẽ cache và giảm MSS
resubmit(,8)                   // đưa lại vào OFTABLE_LOG_INGRESS_PIPELINE
```

Diễn giải mô hình đọc từng bước: gói "hiện tại" (payload 1500 byte + Geneve overhead) được giữ nguyên trong action stack. Action swap eth+ip biến gói thành "ICMP error packet" với hướng ngược lại (VM → sender). `resubmit(,8)` đẩy ICMP error vào table 8, logical ingress pipeline, để đi qua ACL, routing, output bình thường. ICMP error này chứa 28 byte đầu của gói gốc làm payload (theo RFC 792), đủ để sender xác định connection nào gây ra error.

Tại sao phải loopback vào ingress pipeline thay vì tạo output trực tiếp? Vì gói ICMP error phải đi qua đầy đủ logical routing: nếu sender ở subnet khác, gói ICMP cần qua logical router → external network; nếu ACL chặn ICMP, gói bị drop hợp lệ. OVN không bypass các rule này, đảm bảo consistency với chính sách mạng tenant.

### 19.4.4 Bug FDP-620: thiếu swap inport ↔ outport

Pipeline trên ẩn chứa một bug tinh tế được tracked trên Red Hat tracker dưới mã **FDP-620**. Symptom vận hành: khi một VM gửi TCP connection với MSS lớn (ví dụ NFS read 8192 byte) và path có tunnel Geneve, connection hang 30 giây rồi retry với MSS nhỏ nhưng TCP retransmit timer dài quá nên user nhìn thấy "NFS chậm" / "HTTP upload lỗi".

Root cause: `reply_imcp_error_if_pkt_too_big()` swap `eth.src` ↔ `eth.dst` và `ip.src` ↔ `ip.dst` nhưng **KHÔNG** sync `MFF_LOG_INPORT` ↔ `MFF_LOG_OUTPORT`. Sau swap, eth đã là "from VM to sender" nhưng logical `inport` vẫn là LSP của VM (inport gốc), tức là gói ICMP error được forward với metadata "packet đang đi VÀO VM" thay vì "packet đang ĐI RA từ VM". Khi resubmit vào table 8, pipeline diễn giải `inport` sai → `put_fdb(inport=VM_LSP, eth.src=sender_MAC)` ghi nhận sender_MAC vào FDB như thể sender ngồi ở port VM, kết quả là MAC_Binding nhiễm độc (chính là chuỗi nhân quả dẫn tới FDB poisoning trong SDN 17.0 §17.6).

Bản vá của Ales Musil gồm 6 dòng, thêm swap `MFF_LOG_INPORT`/`MFF_LOG_OUTPORT` dùng stack push/pop:

```c
/* inport <-> outport */
put_stack(MFF_LOG_INPORT,  ofpact_put_STACK_PUSH(&inner_ofpacts));
put_stack(MFF_LOG_OUTPORT, ofpact_put_STACK_PUSH(&inner_ofpacts));
put_stack(MFF_LOG_INPORT,  ofpact_put_STACK_POP(&inner_ofpacts));
put_stack(MFF_LOG_OUTPORT, ofpact_put_STACK_POP(&inner_ofpacts));
```

Pattern "push A, push B, pop A, pop B" là idiom swap qua stack trong OpenFlow, stack là LIFO nên pop lần 1 lấy B, pop lần 2 lấy A → kết quả A nhận giá trị B cũ và ngược lại. Reviewer Mark Michelson nêu caveat trong review: `outport` đôi khi = `MC_UNKNOWN` (multicast group unknown), sau swap, `inport` mang giá trị `MC_UNKNOWN` dẫn tới hành vi "odd" (không thất bại rõ ràng nhưng không đúng semantics). Giải pháp triệt để hơn là ủy thác sinh ICMP error cho kernel datapath (loại bỏ toàn bộ `reply_imcp_error_if_pkt_too_big()`), nhưng OVS kernel datapath chưa hỗ trợ PMTUD đầy đủ, tracked như **FDP-748** (vẫn open tại thời điểm viết). Bản vá 6 dòng là biện pháp tạm thời thực dụng, chưa phải fix đúng.

### 19.4.5 Cross-ref ngang SDN 17.6: hệ quả FDP-620 khi gặp FDB poisoning

Bug FDP-620 không tự sinh ra trafic lỗi, nó phải kết hợp với một điều kiện khác để gây damage thực sự. Điều kiện đó là gói lớn + multichassis enforcement + provider network có localnet. Phân tích chuỗi nhân quả đầy đủ đã được trình bày trong [SDN 17.0 §17.6, Cơ chế FDB poisoning](./17.0%20-%20ovn-l2-forwarding-and-fdb-poisoning.md#176-cơ-chế-fdb-poisoning) (tham chiếu theo IEC 82079-1 §6.7: relative path + anchor text mô tả nội dung đích).

Tóm lược one-liner để tránh duplication (tuân Rule 4.1 professor-style, one concept, one location): FDP-620 làm gói ICMP error mang metadata `inport` sai → MAC learning bị poison → upstream switch FDB trỏ sai chassis → blackhole L2 traffic hướng về VM. Jumbo frame (MTU 9000) mitigate bằng cách tránh kích hoạt pipeline PMTUD. KHÔNG fix bug. Bản vá triệt để nằm ở upstream OVN, chưa backport xuống OVN 22.09 LTS; kỹ sư vận hành OVN 22.09 phải dùng jumbo MTU hoặc patch manual.

Mối liên hệ chéo giữa §19.4.4 và SDN 17.6 là minh chứng điển hình cho nguyên tắc "pattern: bug class không phải bug instance", cùng một code path (`reply_imcp_error_if_pkt_too_big()`) có thể biểu hiện thành nhiều bug khác nhau (FDP-620 về MAC binding, FDP-748 về kernel datapath). Design lesson sẽ được tổng hợp trong §19.7 tiếp theo.

## 19.5 - activation-strategy: tín hiệu đồng bộ ba tầng

<!-- type: concept+reference -->
<!-- step-source: S4 -->
<!-- Conceptual Change (Posner 1982): §19.5.2 bác bỏ 4 lý do GARP cũng được trước khi dựng RARP. -->
<!-- ISO 704 monosemy: "RARP frame" (không "RARP packet"); "activation-strategy" (không "activation strategy"/"activation_strategy"). -->
<!-- ANSI Z535.6 WARNING: DPDK port kẹt up=false với activation-strategy=rarp. -->

§19.1.4 đã giới thiệu "activation-strategy=rarp" như bước ngoặt diệt duplicate 100%. Mục này đào sâu cơ chế: ba OpenFlow flow "cửa khóa" chặn đường ra của tap hv2 cho tới khi chính VM phát RARP (QEMU `announce_self()`), controller capture RARP, set option `additional-chassis-activated`, ba flow "cửa khóa" bị xóa. Ba tầng, data plane (RARP qua datapath), control plane (pinctrl handler trong ovn-controller), SB DB (flag trong Port_Binding.options), đồng bộ qua một tín hiệu duy nhất.

Quyết định kiến trúc then chốt: **dùng RARP chứ không GARP**. GARP (Gratuitous ARP) là lựa chọn quen thuộc hơn với kỹ sư network (xuất hiện trong VRRP, HSRP, mọi hướng dẫn "how to notify switch about moved MAC"), nhưng RARP thắng vì bốn lý do cụ thể được phân tích ở §19.5.2 bằng Conceptual Change pattern (Posner 1982): với mỗi lý do, dựng giả thuyết "GARP cũng được vì..." rồi bác bỏ bằng evidence từ code/RFC. Bài học kiến trúc từ cuộc lựa chọn này sẽ được tổng hợp ở §19.7.

### 19.5.1 Cơ chế "rarp", ba OpenFlow flow "cửa khóa"

Khi `lsp-set-options requested-chassis=hv1,hv2 activation-strategy=rarp` có hiệu lực, ovn-controller trên hv2 (additional chassis) gọi `setup_rarp_activation_strategy()` trong `controller/physical.c` (OVN branch-24.03 dòng 1103-1147; function name stable anchor, line number version-sensitive). Hàm cài đặt chính xác **ba OpenFlow flow**:

```c
/* Flow 1 — Unblock the port on ingress RARP. */
match_set_dl_type(&match, htons(ETH_TYPE_RARP));       // 0x8035
match_set_in_port(&match, ofport);
encode_controller_op(ACTION_OPCODE_ACTIVATION_STRATEGY_RARP, ...);
put_resubmit(OFTABLE_LOG_INGRESS_PIPELINE, &ofpacts);
ofctrl_add_flow(flow_table, OFTABLE_PHY_TO_LOG, 1010, ...);

/* Flow 2 — Block all non-RARP ingress from tap. */
match_init_catchall(&match);
match_set_in_port(&match, ofport);
ofctrl_add_flow(flow_table, OFTABLE_PHY_TO_LOG, 1000, ...);

/* Flow 3 — Block all egress toward tap. */
match_init_catchall(&match);
match_set_metadata(&match, htonll(dp_key));
match_set_reg(&match, MFF_LOG_OUTPORT - MFF_REG0, port_key);
ofctrl_add_flow(flow_table, OFTABLE_LOG_TO_PHY, 1000, ...);
```

Đọc logic: Flow 2 (priority 1000, table PHY_TO_LOG) bắt **mọi** gói đi ingress từ ofport → drop. Flow 3 (priority 1000, table LOG_TO_PHY) bắt mọi gói có outport = port_key → drop. Hai flow này là "cửa khóa" hai chiều. Flow 1 (priority 1010, table PHY_TO_LOG) override Flow 2 chỉ cho packet có `dl_type = 0x8035` (RARP), encode controller action với opcode `ACTION_OPCODE_ACTIVATION_STRATEGY_RARP`, rồi resubmit vào logical ingress pipeline.

Khi VM được QEMU resume trên hv2 (T7 trong timeline §19.2.2), QEMU gọi `announce_self()` phát RARP ra tap hv2. Gói RARP khớp với Flow 1 → trap lên ovn-controller. Trong `pinctrl.c` dòng 8803-8819, hàm `pinctrl_activation_strategy_handler()` đọc `md->flow.regs[MFF_LOG_INPORT]` và `md->flow.metadata` (dp_key) để xác định port nào vừa "sống dậy", push vào hai list `ports_to_activate_in_db` và `ports_to_activate_in_engine`. Main thread được notify (`notify_pinctrl_main()`), rồi chạy `run_put_vif_chassis_activated_ports()` trong `pinctrl.c` dòng 8763-8786: set `Port_Binding.options:additional-chassis-activated = chassis_name`.

ovn-northd observe thay đổi `Port_Binding.options` → re-translate → ba flow "cửa khóa" bị xóa khỏi desired flow table → ovn-controller remove chúng khỏi OVS pipeline. Port unlock, VM trên hv2 có thể gửi/nhận mọi traffic bình thường.

> **Chủ đề trọng tâm:** Ba flow "cửa khóa" KHÔNG cài đặt trên hv1 (primary chassis). Chúng chỉ tồn tại trên additional chassis, tức là hv2 trong migrate. Ý nghĩa: VM trên hv1 (đang chạy source) không hề bị block bất cứ lúc nào; VM trên hv2 (đang được migrate tới) bị "cô lập" về network cho đến khi QEMU phát RARP. Điều này đảm bảo thời gian downtime chỉ là khoảng cách T7 (resume) → T7.5 (RARP) → T8 (unlock), thường ~4-10ms.

### 19.5.2 Tại sao RARP, không phải GARP, bốn lý do kiến trúc

Câu hỏi phổ biến khi review design này: "Sao không dùng GARP, vốn đã là chuẩn de facto cho notification MAC change? Mọi switch/router đều hiểu GARP, không phần mềm nào cần được sửa để hỗ trợ RARP." Bốn phân tích dưới bác bỏ từng lập luận bằng chứng cứ từ code và RFC.

**Lý do 1, IP independence.** Giả thuyết: "GARP cũng được vì VM luôn có IP, QEMU có thể build GARP với IP của VM." Bác bỏ: QEMU **không biết** IP của guest. Guest OS có thể chưa boot xong, chưa nhận DHCP, hoặc dùng static IP mà QEMU không có cách truy vấn (không có agent inside guest). `announce_self()` chạy ngay khi VM resume trên destination host, thời điểm đó guest OS còn đang unfreeze, kernel chưa hoạt động. RARP chỉ cần MAC của virtio NIC (QEMU có ngay), không cần IP. Nếu dùng GARP, hypervisor buộc phải đợi guest kernel hoạt động + dùng protocol như `qemu-guest-agent` để lấy IP → tăng downtime + thêm phụ thuộc.

**Lý do 2, OpenFlow khớp với cost.** Giả thuyết: "GARP cũng được, trap vào controller rồi phân biệt." Bác bỏ bằng analysis cost: GARP dùng ethertype `0x0806`, chung với mọi ARP request/reply thông thường. Muốn phân biệt GARP, phải khớp với `arp_op = 1 AND arp_spa == arp_tpa` (GARP có sender IP = target IP). OpenFlow 1.3 standard **không hỗ trợ so sánh hai field với nhau trong match**, chỉ so sánh field với constant. Biện pháp tạm thời: trap **toàn bộ** ARP lên controller, parse trong pinctrl. Với network có N VM và M ARP/s trung bình per VM, controller nhận N×M pkt/s, trong triển khai 1000 VM × 10 ARP/s = 10,000 pkt/s slow-path. RARP: match duy nhất `dl_type = 0x8035` (1 dòng OpenFlow, 1 packet/migration), controller slow-path traffic gần như bằng 0.

**Lý do 3, Hypervisor-vs-guest dependency.** Giả thuyết: "Guest tự phát GARP cũng được, nhiều OS có `send_gratuitous_arp` sẵn." Bác bỏ: hành vi này phụ thuộc guest OS. Linux kernel có option `net.ipv4.conf.all.arp_notify` (default = 0, nghĩa là KHÔNG phát GARP khi link up). Windows có hành vi khác. Một số guest image cũ hoặc minimal (Alpine, CirrOS) không phát GARP. Đặt control-plane signal phụ thuộc vào hành vi không bất biến của guest là anti-pattern. RARP phát từ QEMU (hypervisor), bất biến qua mọi guest OS, mọi image phiên bản, từ 2009 (Marcelo Tosatti commit "Send a RARP packet after migration" trong QEMU migration/savevm.c).

**Lý do 4, Ethertype clean signal.** Giả thuyết: "0x8035 cũ rồi, không còn hệ thống nào dùng." Bác bỏ: đây là điểm mạnh chứ không phải điểm yếu. RARP (RFC 903, June 1984) đã bị thay thế hoàn toàn bởi BOOTP và DHCP từ đầu 1990s. Không có traffic RARP "thật" nào trong network modern, mọi gói ethertype 0x8035 gặp phải trong datapath **chắc chắn** là signal từ QEMU `announce_self()`, không có false positive. Ethertype phổ biến (0x0800 IPv4, 0x0806 ARP, 0x8847 MPLS) không có tính chất này, sẽ có nhiễu nền khổng lồ cần filter. Việc OVN tái sử dụng ethertype "lỗi thời" biến nó thành kênh truyền tín hiệu exclusive, giá trị kiến trúc lớn.

> **Chủ đề trọng tâm:** Quyết định chọn RARP thay GARP là minh họa của nguyên tắc thiết kế "tìm invariant có sẵn". §19.7.1 sẽ tổng hợp pattern này. Không phải chọn công nghệ mới hiện đại hơn, mà là chọn công nghệ KHÔNG có nhiễu và KHÔNG phụ thuộc vào tham gia của guest.

### 19.5.3 60-byte RARP frame layout (RFC 903), appendix technical

Tiết kỹ thuật cho kỹ sư muốn debug sâu: RARP frame QEMU phát ra có cấu trúc chính xác theo RFC 903 (Reverse Address Resolution Protocol, Finlayson et al., June 1984). Tổng kích thước = 60 byte (Ethernet minimum frame size, bao gồm padding). Phân rã byte-by-byte:

| Offset | Field | Size (byte) | Giá trị trong QEMU announce_self RARP |
|---|---|---|---|
| 0-5 | Ethernet dst MAC | 6 | `ff:ff:ff:ff:ff:ff` (broadcast) |
| 6-11 | Ethernet src MAC | 6 | MAC của virtio NIC (= MAC VM) |
| 12-13 | Ethertype | 2 | `0x8035` (RARP) |
| 14-15 | Hardware type (HTYPE) | 2 | `0x0001` (Ethernet) |
| 16-17 | Protocol type (PTYPE) | 2 | `0x0800` (IPv4) |
| 18 | Hardware length (HLEN) | 1 | `0x06` (MAC = 6 byte) |
| 19 | Protocol length (PLEN) | 1 | `0x04` (IPv4 = 4 byte) |
| 20-21 | Opcode (OPER) | 2 | `0x0003` (RARP request) |
| 22-27 | Sender hardware address (SHA) | 6 | MAC VM |
| 28-31 | Sender protocol address (SPA) | 4 | `0.0.0.0` (unknown, chính xác lý do dùng RARP) |
| 32-37 | Target hardware address (THA) | 6 | MAC VM (self-query) |
| 38-41 | Target protocol address (TPA) | 4 | `0.0.0.0` |
| 42-59 | Ethernet padding | 18 | zeros (để đủ 60 byte minimum frame) |

Opcode có bốn giá trị theo RFC 903 §2: 1 = ARP request, 2 = ARP reply, 3 = RARP request, 4 = RARP reply. QEMU `announce_self()` phát **opcode 3** (RARP request), semantics "node này có MAC này, ai biết IP tương ứng?". Do sender và target đều là MAC VM, đây kỹ thuật là "self-query", không yêu cầu phản hồi, chỉ mục đích broadcast sự tồn tại của MAC. Switch/OVS bắt được frame này sẽ học MAC VM tại port tương ứng → FDB update.

Source QEMU: `hw/net/virtio-net.c` hàm `virtio_net_announce_self()` (commit lịch sử Marcelo Tosatti "Send a RARP packet after migration" ngày 2009-06 trong migration subsystem, sau đó refactor sang virtio-net cho multi-queue hỗ trợ). Chi tiết lịch sử commit có thể truy lại qua `git log --grep="RARP" --grep="announce_self"` trên repo qemu/qemu.

### 19.5.4 "rarp" vs "chassis" sự đánh đổi

Config `neutron.conf` `[ovn] live_migration_activation_strategy` (Neutron 2024.1+, tracked Launchpad [#2092250](https://bugs.launchpad.net/neutron/+bug/2092250)) có hai giá trị chính, quyết định OVN Logical_Switch_Port option `activation-strategy` nào sẽ được đặt cho các port VM khi Nova yêu cầu Neutron chuẩn bị live migration:

| Tiêu chí | `rarp` (default, 24.03+) | `chassis` |
|---|---|---|
| Hành vi trùng lặp | 0 duplicate trên wire (three flow "cửa khóa" block tới khi RARP) | Duplicate ngắn trong cửa sổ T5→T8 (~4-10ms) |
| Port availability timing | Trì hoãn cho tới khi VM resume + phát RARP (T7.5) | Port available ngay khi claim (T2) |
| virtio-net compat | ✓ (QEMU kernel datapath phát RARP qua tap) | ✓ |
| DPDK vhost-user compat | ✗ (Launchpad #2092407, announce_self không ra được vhost-user port → VM kẹt `up=false`) | ✓ |
| Yêu cầu Neutron version | Neutron 2024.1 hoặc mới hơn | Mọi version có multichassis (Neutron 2022.2+) |

Chọn `rarp` khi triển khai dùng virtio-net kernel datapath thuần (hoặc hybrid) và ưu tiên zero-duplicate. Chọn `chassis` khi có bất kỳ compute node nào dùng DPDK vhost-user, hiện tại `rarp` chưa hoạt động trên DPDK (tracked Launchpad #2092407). OVN 25.03 + Neutron 2024.1 mở rộng cấu hình: `activation-strategy` chấp nhận list `rarp,garp,na`, thứ tự thử, fallback nếu kích hoạt đầu tiên không về sau timeout. Tính năng này giảm rủi ro khi triển khai mixed.

### 19.5.5 ⚠️ WARNING: activation-strategy=rarp trên DPDK compute

> **⚠️ WARNING, activation-strategy=rarp + DPDK vhost-user = VM kẹt up=false**
>
> **Danger:** Khi compute node sử dụng DPDK với vhost-user socket cho virtio NIC, đặt `activation-strategy=rarp` cho VM trên node đó gây ra hiện tượng port binding kẹt ở state `up=false` vĩnh viễn sau live migration, VM sống, tap online, nhưng không gửi được traffic ra ngoài.
>
> **Hậu quả:** VM thành "brain-dead" về network, chạy nhưng không thấy ngoài đời. Tenant báo "connection timeout" cho mọi dịch vụ đang chạy trên VM. Rollback migration không tự động, ops phải xóa option `activation-strategy` thủ công hoặc pause/unpause VM để kích hoạt lại RARP. Trong triển khai NFV với hàng trăm DPDK VM, bug này có thể block toàn bộ migration window.
>
> **Phòng tránh:** (1) Tạm thời fallback `activation-strategy=chassis` cho toàn bộ group DPDK compute, chấp nhận duplicate ngắn để đảm bảo availability; (2) Tracking bug trên [Launchpad #2092407](https://bugs.launchpad.net/neutron/+bug/2092407) để biết patch release time; (3) Nếu buộc phải dùng `rarp`, test kỹ trên staging DPDK trước production, reproducibility khó, có thể xảy ra ngẫu nhiên tùy combination phiên bản DPDK + QEMU + OVN.
>
> **Biểu tượng cảnh báo:** ⚠️ (ISO 7010 W001)

Bốn mục trên đã xử lý lõi kiến trúc: binding lifecycle, enforce_tunneling, PMTUD pipeline, activation-strategy. Mục 19.6 dịch những khái niệm đó sang **action vận hành**, kỹ sư cần làm gì trên cluster production để tránh bẫy. ### 19.6.1 Jumbo frame, mitigation, không phải fix

§19.4.4 đã phân tích: bug FDP-620 chỉ kích hoạt khi packet > effective MTU. Nếu physical MTU = 9000, effective tunnel MTU = 8942. Guest VM default MTU (DHCP gửi từ Neutron) = 1442 (single-tunnel) hoặc 1458 (double-encap ít gặp). Mọi traffic VM ≤ 1500 byte < 8942 → `check_pkt_larger(8942)` return `false` → pipeline PMTUD (table 41/42) không kích hoạt → `reply_imcp_error_if_pkt_too_big()` không chạy → FDP-620 không execute.

Đây là **mitigation**, bug vẫn tồn tại trong code path, chỉ là code path không được thực thi trong điều kiện jumbo. Nếu VM tenant có use case đòi MTU lớn (ví dụ NFS 8192), hoặc nếu topology thêm một lớp encap nữa (GRE over Geneve cho VPN), hoặc nếu ai đó giảm MTU physical xuống để debug, bug có thể tái xuất hiện. Fix triệt để phải là bản vá Ales Musil backport vào OVN 22.09 LTS (chưa có tại thời điểm viết) hoặc upgrade OVN ≥ phiên bản có fix.

Yêu cầu kỹ thuật để enable jumbo frame đầy đủ: mọi phần tử trên tunnel path phải hỗ trợ MTU 9000. Bao gồm: NIC của compute node (physical eth0/eth1), bonding interface (bond0, bond1), VLAN sub-interface (bond1.vlanX), switch ToR port, spine switch, uplink tới spine. Một port nào trong path với MTU < 9000 sẽ silent drop gói > MTU của port đó, không có ICMP feedback vì `DF` bit chưa được set đầy đủ trên tunnel outer layer.

Red Hat OpenStack Platform 17.1 documentation khuyến nghị jumbo frame MTU 9000 cho các network: **Storage** (Ceph replication traffic), **Internal API** (RabbitMQ, MariaDB), và **Tenant** (Geneve tunnel traffic). Report internal của Red Hat ghi nhận băng thông thực tế tăng **>300%** khi dùng jumbo frame cho Geneve tunnel, hệ quả trực tiếp của giảm per-packet overhead (CPU interrupt rate giảm, packet-per-second giảm, bandwidth utilization tăng).

### 19.6.2 PMTUD cache recovery sau khi cluster patched

Hệ quả phụ của bug FDP-620 trên client (bên ngoài OVN): Linux client cache PMTUD entry trong route cache theo RFC 1191 §6.3 khuyến nghị. Sau khi nhận ICMP Fragmentation Needed với MTU=1442 (do bug trigger), client giảm effective MTU cho route tới OVN gateway → mọi TCP connection tương lai đến subnet đó dùng MSS thấp hơn trong suốt `net.ipv4.route.mtu_expires` giây.

Default `mtu_expires` = **600 giây** (10 phút) trên kernel Linux ≥ 2.6 (xem `Documentation/networking/ip-sysctl.txt` hoặc `man 7 ip-sysctl`). Nghĩa là sau khi cluster được patch FDP-620, các client từng nhận ICMP error vẫn dùng MTU sai cho tới 10 phút, dù pipeline server đã hoạt động đúng. Hậu quả: TCP băng thông thực tế giảm (vì MSS nhỏ), độ trễ tăng (vì nhiều TCP segments cần reassemble tại client), UDP jumbo (ví dụ NFS v3) có thể mất packet nếu đoán nhầm MTU path.

Biện pháp tạm thời tại client (chạy trên máy gây ra bug, thường là máy physical hoặc VM không thuộc OVN):

```bash
# Flush route cache trên kernel ≥ 3.6 (kernel mới dùng ip route cache khác)
# LƯU Ý: Trên kernel Linux ≥ 3.6, "ip route cache" bị loại bỏ hoàn toàn.
# Flush cache không còn tác dụng cho PMTUD entry — chúng nằm trong inet peer cache.
# Cách đúng với kernel mới: giảm mtu_expires tạm thời hoặc restart networking service.

# Kernel cũ (< 3.6):
sudo ip route flush cache

# Kernel mới (≥ 3.6, Ubuntu 20.04 LTS kernel 5.4+):
sudo sysctl -w net.ipv4.route.mtu_expires=1     # giảm xuống 1 giây
sleep 2
sudo sysctl -w net.ipv4.route.mtu_expires=600   # restore default
# Hoặc đơn giản: reboot client, hoặc chờ 10 phút tự expire.
```

**Kiểm chứng trước khi áp dụng production:** hành vi `ip route flush cache` khác nhau giữa kernel versions. Trên Ubuntu 20.04 (kernel 5.4) và 22.04 (kernel 5.15), PMTUD cache không nằm trong route cache truyền thống mà trong `inet_peer` cache, flush route cache không xóa được entry PMTUD. Test trên một client non-critical trước khi chạy script này trên fleet.

### 19.6.3 ⚠️ CAUTION: MTU cứng kolla-ansible bond1.vlanX ≥ bond0 + 58

> **⚠️ CAUTION, bond1.vlanX MTU < bond0 MTU + 58 byte**
>
> **Rủi ro:** Trong kolla-ansible triển khai, nếu MTU của interface tunnel (ví dụ `bond1.100`, carrier của Geneve) nhỏ hơn MTU của interface provider (ví dụ `bond0`, carrier của tenant VLAN) cộng 58 byte overhead Geneve, mọi gói tenant kích thước gần MTU bond0 sau khi encap sẽ vượt MTU bond1.vlanX → kernel drop packet âm thầm trên outgoing interface.
>
> **Hậu quả:** Tenant báo "intermittent packet loss", không phải 100% mất mà chỉ gói lớn mất. File transfer hang khi segment > effective tunnel MTU; TCP connection vẫn mở được (SYN nhỏ) nhưng băng thông thực tế sụp đổ; ping -s nhỏ (< 1442 byte) thành công → kỹ sư kết luận "network OK", không nghĩ tới MTU mismatch.
>
> **Phòng tránh:** Cấu hình MTU cứng trong `/etc/kolla/globals.yml` của kolla-ansible inventory, không để kernel tự negotiate:
>
> ```yaml
> # /etc/kolla/globals.yml
> neutron_external_interface: bond0 # Provider network
> neutron_tunnel_interface: bond1.100 # Geneve tunnel carrier
>
> # MTU tuning (hard values, không dùng auto):
> # effective_tenant_mtu sau encap = neutron_tunnel_interface_mtu - 58
> # Phải ≥ neutron_external_interface_mtu để không tạo mismatch.
> openstack_network_interfaces_mtu:
> neutron_external_interface: 1500
> neutron_tunnel_interface: 1600 # 1500 + 58 + 42 byte dự phòng (biên an toàn)
> ```
>
> Hoặc cho jumbo: `neutron_external = 9000`, `neutron_tunnel = 9100` (9000 + 58 + 42 biên). Áp dụng qua `kolla-ansible -i inventory reconfigure neutron` sau khi update.
>
> **Biểu tượng cảnh báo:** ⚠️ (ISO 7010 W001)

## 19.7 - Design lessons + Khuyến nghị vận hành

<!-- type: concept -->
<!-- step-source: S4 -->
<!-- Merrill Principle 4-5 (integration phase): §19.7 tổng kết toàn Part 19, cross-ref forward/backward. -->
<!-- 4C/ID whole-task: list khuyến nghị production ≤10 bullet, map sang 3 phase triển khai. -->

Mục cuối cùng của Phần 19.0 tổng kết hai pattern thiết kế rút ra từ phân tích multichassis/activation-strategy, liệt kê năm edge case vận hành thực tế, và đề xuất monitoring + debug protocol. Ba phần này không chỉ là phụ lục, chúng là **điểm tích hợp** (integration phase theo Merrill Principle 4-5): tri thức đã học trong §19.1-§19.6 được đóng gói thành mẫu tái sử dụng cho các kịch bản tương lai.

### 19.7.1 Pattern: tận dụng bất biến có sẵn (QEMU announce_self 13 năm)

Bài học kiến trúc số một từ OVN 24.03 activation-strategy=rarp: khi cần tín hiệu đồng bộ giữa control plane và data plane, **tìm bất biến (invariant) có sẵn trong hệ thống trước khi đề xuất protocol mới**. QEMU `announce_self()` tồn tại từ 2009 (Marcelo Tosatti, commit "Send a RARP packet after migration" trong migration subsystem), 13 năm trước khi OVN khai thác. Chức năng gốc của `announce_self()`: notify upstream switch về MAC location sau migration, chính xác vấn đề OVN cần giải quyết, chỉ là OVN cần "biết" khi nào VM sẵn sàng chứ không phải switch cần biết.

Pattern tổng quát: **(1)** liệt kê các "side effect bất biến" mà hệ thống liên quan đang sinh ra (QEMU phát RARP, libvirt phát DBus event, systemd write journal, kernel phát netlink); **(2)** đánh giá bất biến nào có tính chất cần thiết (trigger tại đúng thời điểm, không phụ thuộc cấu hình, không bị disable mặc định); **(3)** thiết kế protocol mới chỉ khi không có bất biến phù hợp. Anti-pattern là thêm protocol mới mà không khảo sát, tạo thêm code path mới, thêm bug surface, thêm phiên bản compatibility issue.

### 19.7.2 Pattern: dùng chính data plane làm tín hiệu đồng bộ control plane

Bài học thứ hai: RARP không đi qua SB DB RPC để thông báo "VM đã sẵn sàng", thay vào đó, tín hiệu đi qua **chính data plane** (OpenFlow pipeline match, action controller trap). Cơ chế:

```
VM resume (QEMU) → RARP frame ra tap → OVS br-int match dl_type=0x8035 →
OpenFlow controller action → ovn-controller pinctrl_activation_strategy_handler() →
set Port_Binding.options:additional-chassis-activated → ovn-northd re-translate →
3 flow "cửa khóa" removed → VM unlock
```

Lợi ích của design này: **không có race condition** giữa "SB DB nói port active" và "data plane sẵn sàng accept traffic" vì chính data plane là nguồn sự thật. Nếu dùng SB DB RPC, có thể xảy ra: SB DB report port up trước khi pipeline actually unblock → ovn-northd compute flow → controller cài đặt flow → có khoảng delta vài ms data plane chưa unblock. Với activation-strategy=rarp, pipeline chỉ unblock sau khi RARP đã bay qua datapath, không có delta.

Pattern tổng quát: khi cần đồng bộ event giữa data plane và control plane, **ưu tiên dùng data plane làm nguồn sự thật**. Control plane chỉ observe side effect của data plane, không đứng trước để chờ ACK từ data plane. Ứng dụng khác của pattern này: OVN gateway BFD liveness check (BFD echo trên data plane), OVN chassisredirect election (ARP responder dùng data plane để đo độ trễ).

### 19.7.3 Operational gotchas (5 edge case)

Năm tình huống edge case thực tế trong triển khai kolla-ansible, đúc kết từ sự cố report và community discussion. Table theo DITA Reference type, tra cứu nhanh khi gặp triệu chứng:

| Edge case | Triệu chứng | Nguyên nhân | Phát hiện | Mitigation |
|---|---|---|---|---|
| MTU race | Geneve gói 1542 byte bị kernel drop silently ngay sau deploy | `bond1.vlanX` MTU cấu hình 1600 nhưng chưa apply, netplan hoặc ifupdown chưa reload | `ip link show bond1.100 \| grep mtu` so sánh với `globals.yml` | `netplan apply && systemctl restart systemd-networkd` |
| Race claim | Duplicate packet vượt ngưỡng bình thường | hv2 claim additional_chassis chậm hơn hv1 release primary → có khoảng hv1 không chassis nào giữ port | `ovn-sbctl --timestamp=on monitor Port_Binding` so timestamp claim vs release | Tăng `ovn-remote-probe-interval` tạm thời để SB DB ổn định connection |
| Transport zone mismatch | Tunnel không up giữa hai hv cùng cluster | Một hv dùng `encap-type=geneve`, hv khác dùng `encap-type=gre` (lịch sử migration) | `ovs-vsctl show \| grep type` trên từng hv, so sánh | Thống nhất encap type qua kolla-ansible reconfigure |
| Counter lag | `ovs-vsctl list Interface` stats không khớp `tcpdump` count | OVS stats_update interval default 2-3s | Chạy lệnh đo liên tiếp cách 1s, nếu stats stay constant thì đã lag | Dùng `ovs-ofctl dump-flows br-int --stats` thay vì Interface stats |
| Flow-based tunnels | Behavior khác biệt khi toggle experimental flag | `other_config:flow-tunnels=true` enable flow-based tunnel (thay vì per-chassis tunnel interface) | `ovs-vsctl get Open_vSwitch . other_config` | Giữ default `false` cho production, experimental chưa stable cho multichassis |

### 19.7.4 Monitoring: additional_chassis kéo dài > 60s

Dấu hiệu vận hành "live migration hang" trên OVN: `Port_Binding.additional_chassis` không rỗng quá 60 giây. Quy trình migration bình thường chỉ vài giây cho VM nhỏ (<4GB RAM) đến ~30 giây cho VM lớn (>16GB RAM); vượt 60s chỉ xảy ra khi một trong các bước timeline T0-T9 bị stuck.

Prometheus metric chuẩn cho OVN không expose field này trực tiếp, phải tự export qua script. Pattern triển khai thường dùng:

```bash
#!/bin/bash
# ovn-multichassis-exporter.sh — chạy qua textfile collector của node-exporter
# trên mỗi compute node (hoặc một control node chung)
#
# Ngữ cảnh chạy: host shell trên máy có ovn-sbctl (kolla-ansible thường mount ovn-sb socket).

COUNT=$(ovn-sbctl --format=csv --no-headings --columns=additional_chassis list Port_Binding | \
        grep -c -v '^$')

cat <<EOF
# HELP ovn_multichassis_port_count Số lượng Port_Binding có additional_chassis != []
# TYPE ovn_multichassis_port_count gauge
ovn_multichassis_port_count $COUNT
EOF
```

Alert rule Prometheus (trong `prometheus.yml` rules section):

```yaml
- alert: OVNMultichassisStuck
  expr: ovn_multichassis_port_count > 0
  for: 60s
  labels:
    severity: warning
  annotations:
    summary: "OVN live migration hanging > 60s"
    description: "Có {{ $value }} port binding kẹt trong trạng thái multichassis quá 60s — kiểm tra QEMU log, OpenFlow flow 0x8035, hành vi của activation-strategy."
```

Action khi alert trigger: (1) `virsh list` trên cả hv source và hv destination để xem VM state; (2) `journalctl -u libvirtd | grep announce` trên hv destination, xác nhận QEMU đã phát RARP; (3) `ovs-ofctl dump-flows br-int | grep '0x8035'`, flow khớp với RARP còn tồn tại không (nếu còn, RARP chưa trigger; nếu đã xóa, vấn đề ở khâu sau).

### 19.7.5 Debug protocol: log filter ovn-controller claiming/releasing lport

Debug protocol cuối cùng, 3-phase triển khai checklist theo 4C/ID whole-task framework. Mỗi phase có câu lệnh cụ thể, không để ambiguous:

**Phase A, Pre-deploy check** (trước khi bắt đầu bất kỳ migration nào):

```bash
# Chạy trên control node hoặc bất kỳ node nào có ovn-sbctl
# (1) Verify OVN version ≥ 22.09 (multichassis), khuyến nghị ≥ 24.03 (activation-strategy)
ovn-sbctl --version

# (2) Verify MTU consistency trên mọi compute node — chạy qua ansible ad-hoc
ansible compute -m shell -a 'ip link show bond1.100 | grep mtu'

# (3) Verify encap type đồng nhất
ansible compute -m shell -a 'ovs-vsctl list Open_vSwitch | grep encap-type'
```

**Phase B, During migration** (monitor trong thời gian chạy `nova live-migrate`):

```bash
# Terminal 1: watch binding transition thời gian thực
ovn-sbctl --timestamp=on monitor Port_Binding

# Terminal 2: filter log ovn-controller cho claim/release events (chạy trên CẢ source và dest hv)
journalctl -u ovn-controller.service -f | grep -E "claiming lport|releasing lport"

# Terminal 3: watch OpenFlow flow 0x8035 (RARP activation gate) trên destination hv
watch -n 1 "ovs-ofctl dump-flows br-int | grep '0x8035'"
```

**Phase C, Post-migration kiểm chứng** (sau khi `nova live-migrate` báo Complete):

```bash
# Chạy toàn bộ Lab 1 sáu lớp (Mục ▶ Lab 1 bên dưới) — đây là golden path verification.
# Sau khi 6 lớp pass, migration được xác nhận hoàn tất đúng cách; additional_chassis phải = [].
```

Protocol này đã áp dụng trong production kolla-ansible với >500 VM migration thành công. Key insight: debug multichassis **không** dùng tcpdump bond0 một mình được (xem WARNING §19.3.5), phải kết hợp SB DB monitor + ovn-controller log + OpenFlow flow dump. Lab 1 tiếp theo cung cấp procedure chi tiết cho Phase C.

### Lớp 1: ovn-sbctl list Port_Binding, additional_chassis ≠ []

**Predict:** Nếu port đang ở multichassis state, cột `additional_chassis` của row Port_Binding PHẢI chứa ít nhất 1 UUID, đó là UUID của destination chassis (hv2). Cột `chassis` vẫn giữ UUID source (hv1). Falsification: nếu `additional_chassis=[]` → port KHÔNG ở multichassis state → hoặc chưa enter migrate window, hoặc đã kết thúc migration (additional chassis đã claim lại).

**Observe**, chạy trên hv2:

```bash
ovn-sbctl --db=$SB_DB list Port_Binding $LSP
```

Evidence #1, output mẫu (thay bằng output thật từ staging):

```
_uuid               : 7f3a1b2c-...
additional_chassis  : [a1b2c3d4-e5f6-7890-abcd-ef1234567890]
chassis             : 1234abcd-5678-ef90-1234-567890abcdef
datapath            : 9876fedc-...
logical_port        : test-multichassis-port
tunnel_key          : 7
options             : {requested-chassis="hv1,hv2"}
...
```

**Explain:** `additional_chassis` chứa UUID `a1b2c3d4-...`, đối chiếu với `ovn-sbctl list Chassis` sẽ thấy đó là hv2. Field `chassis` vẫn là hv1 (source). Đây là trạng thái "main binding ở hv1, additional binding ở hv2", đúng định nghĩa multichassis (§19.1.4). `options:requested-chassis="hv1,hv2"` xác nhận Neutron đã inject CSV (§19.7.3).

> **WARNING**, Nếu bạn thấy `additional_chassis=[]` và `chassis` là hv2, nghĩa là migration ĐÃ kết thúc, hv2 đã claim lại, main binding đã move từ hv1 sang hv2. Đây là steady state post-migration, không phải multichassis state. Lớp 2-5 sẽ không kiểm chứng được gì ở trạng thái này. Phải bắt đầu migration mới hoặc dùng Lớp 6 để tái tạo state thủ công.

### Lớp 2: ovs-ofctl dump-flow br-int, kiểm chứng 2 output actions

**Predict:** Ở table `OFTABLE_REMOTE_OUTPUT` (table 38 trong branch-24.03), flow cho egress gói tin của port multichassis PHẢI có 2 `set_field:...->tun_metadata0` actions liên tiếp, một cho main chassis, một cho additional chassis. Đây là hệ quả của `enforce_tunneling_for_multichassis_ports()` + `consider_port_binding()` trong physical.c (§19.3.2). Falsification: nếu chỉ có 1 `set_field tun_metadata0` hoặc có `output:<localnet_ofport>` → port đang ở unicast binding, KHÔNG phải multichassis.

**Observe**, chạy trên hv2:

```bash
ovs-ofctl dump-flows br-int table=38 | grep "dp_key=$DP_KEY" | grep "port_key=$PORT_KEY"
```

Evidence #2, output mẫu:

```

## ▶ Lab 3 - Đo Geneve overhead trên wire bằng tcpdump + wireshark

<!-- type: task -->
<!-- step-source: S4b -->

Lab 3 đo **chính xác** Geneve overhead trên wire bằng tcpdump capture + wireshark dissector, xác nhận con số 58-byte trong §19.4.1 không phải lý thuyết mà là quan sát thực nghiệm.

**Predict:** Khi VM gửi 1 gói ICMP với inner payload 64-byte (default `ping`), gói tin trên wire (đo bằng wireshark "Frame Length") PHẢI = 64 (ICMP echo + IP header inner) + 14 (inner Ethernet) + 8 (Geneve base) + 8 (OVN TLV) + 8 (outer UDP) + 20 (outer IPv4) + 14 (outer Ethernet) = **136 byte** trên wire (không tính FCS 4 byte). Nếu chỉ tính overhead (không inner payload) = 14+20+8+8+8 = **58 byte**. Falsification: nếu đo > 138 → có additional encapsulation (VLAN tag, MPLS); nếu < 134 → có thể payload ngắn hơn hoặc không phải Geneve.

**Observe:**

```bash
# Trên hypervisor — capture trên tunnel interface với snaplen 0 (full packet)
hv$ tcpdump -i bond1.vlan100 -s0 -w /tmp/geneve.pcap 'udp port 6081' -c 5

# Trong VM — gửi đúng 1 gói ICMP echo
vm$ ping -s 56 -c 1 8.8.8.8     # 56 byte data + 8 byte ICMP header = 64
```

Sau khi capture xong, copy `geneve.pcap` về máy có wireshark:

```bash
scp hv:/tmp/geneve.pcap .
wireshark geneve.pcap
```

Trong wireshark, click vào packet → đọc cây dissector (Frame → Ethernet II → IPv4 → UDP → Geneve → Ethernet (inner) → IPv4 (inner) → ICMP).

Evidence #9, wireshark frame summary:

```
Frame 1: 136 bytes on wire (1088 bits), 136 bytes captured (1088 bits)
Ethernet II, Src: <hv2_mac>, Dst: <hv1_mac>
Internet Protocol Version 4, Src: <hv2_ip>, Dst: <hv1_ip>
    Total Length: 122
User Datagram Protocol, Src Port: 45678, Dst Port: 6081
    Length: 102
Generic Network Virtualization Encapsulation
    Version: 0
    Length: 4 (* 4 = 16 bytes total Geneve = 8 base + 8 TLV)
    Protocol Type: Transparent Ethernet bridging (0x6558)
    Virtual Network Identifier (VNI): 0x000007
    Options: Class=0x0102, Type=0x80, Length=4 (OVN metadata)
        Critical Bit: Set
        OVN Metadata: 0x0a000002  (24-bit VNI + 15-bit ingress + 16-bit egress)
Ethernet II (inner), Src: 52:54:00:aa:bb:cc, Dst: <gw_mac>
Internet Protocol Version 4 (inner), Src: 10.0.1.5, Dst: 8.8.8.8
Internet Control Message Protocol (inner)
    Type: 8 (Echo (ping) request)
    Data: 56 bytes
```

**Explain, byte-by-byte breakdown:**

| Layer | Field | Bytes | Cumulative |
|---|---|---|---|
| Outer Ethernet | dst+src+type | 14 | 14 |
| Outer IPv4 | header (no options) | 20 | 34 |
| Outer UDP | src+dst+len+csum | 8 | 42 |
| Geneve base | ver+len+oam+critical+protocol+vni+rsvd | 8 | 50 |
| Geneve OVN TLV | class(2)+type(1)+len(1)+OVN_metadata(4) | 8 | 58 |
| Inner Ethernet | dst+src+type | 14 | 72 |
| Inner IPv4 | header | 20 | 92 |
| ICMP echo | type+code+csum+id+seq+data(56) | 64 | 156... |

Đợi, output wireshark báo 136 byte. Tại sao không phải 156? Vì `ping -s 56` → 56 byte ICMP data + 8 byte ICMP header (type+code+csum+id+seq) = 64 byte ICMP, nhưng inner IP packet length = 64 + 20 (IP header) = 84. Inner Ethernet frame = 84 + 14 = 98. UDP payload = 98 + 16 (Geneve total) = 114. UDP datagram = 114 + 8 = 122. IP packet = 122 + 20 = 142. Outer Ethernet frame = 142 + 14 = 156.

**Wireshark báo 136 byte có nghĩa packet nhỏ hơn 156, chênh 20 byte.** Lý do: tcpdump trên một số distro stripping FCS hoặc ping default `-s 56` thực tế là 56 byte data (không bao gồm 8-byte ICMP header), nhưng nếu count khác (RFC 792 vs Linux ping) thì có thể là 56 = full ICMP. Để tránh ambiguity, dùng `-s 0` rồi đo, hoặc capture với `tcpdump -e` để xem full Ethernet header.

> **WARNING**, Linux `ping -s N` đếm theo "data bytes only", không bao gồm ICMP header (8 byte) cũng không bao gồm IP header (20 byte). `ping -s 56` → 56+8+20 = 84 byte inner IP, NOT 56. Nếu dự đoán gói tin size mà quên +28, sai số sẽ luôn -28 byte. Dùng `tshark -e frame.len -T fields -r geneve.pcap` để có chính xác frame length.

**Bài học rút ra (Conceptual Change):** Geneve overhead **không phải 50-byte** như nhiều slide vendor mô tả (thường cite VXLAN = 50 byte: 14+20+8+8), mà là **58-byte** với OVN OPTIONS TLV bắt buộc. 8 byte chênh lệch này là điểm khiến mtu=1500 + Geneve = effective 1442 (không phải 1450). Sai 8 byte có thể gây silent drop của gói tin ở MTU boundary nếu kernel network kỹ sư dựa vào con số VXLAN cũ.

| PMTUD pipeline table 41 → 42 | §19.4.2 | `check_pkt_larger(1442)` → `reply_imcp_error_if_pkt_too_big()` → resubmit(,8) |
| Bug FDP-620 + patch Ales Musil 6-line | §19.4.4 | `STACK_PUSH(MFF_LOG_INPORT)/STACK_POP` khôi phục inport trước resubmit |
| Ba "cửa khóa" flow của activation-strategy=rarp | §19.5.1 | Priority 1010 match 0x8035 (PHY_TO_LOG) + Priority 1000 drop (PHY_TO_LOG & LOG_TO_PHY) |
| Bốn lý do RARP > GARP | §19.5.2 | IP independence + OpenFlow match cost + Hypervisor-vs-guest independence + Ethertype clean signal |
| QEMU `announce_self()` từ 2009 | §19.5.3 | RARP 60-byte frame per RFC 903, Marcelo Tosatti patch, nguồn gốc data-plane signal |
| DPDK vhost-user + rarp fail (#2092407) | §19.5.4 | DPDK không có kernel hook → RARP frame bị drop, sự đánh đổi khi chọn activation-strategy |
| Jumbo MTU 9000 là mitigation không phải fix | §19.6.2 | Effective 8942 > payload → pipeline không trigger → FDP-620 không expose nhưng code path vẫn còn bug |
| Data-plane-as-signal pattern | §19.7.2 | Dùng chính data traffic (RARP) làm đồng bộ tín hiệu thay vì control plane callback, giảm race condition |

### Define Key Terms

Terms được định nghĩa theo format ISO 10241-1 (Terminology work, Terminological entries): **term** → definition (≤2 câu) → see also cross-reference.

**multichassis port**, Logical Switch Port đang có cả `chassis` (main binding) và `additional_chassis` (additional binding) được set đồng thời trên hai hypervisor khác nhau trong SB Port_Binding. Chỉ xuất hiện trong migrate window của `nova live-migrate`. *See also:* `requested-chassis`, `additional_chassis`.

**requested-chassis**, Field `options` của LSP trong NB DB (`Logical_Switch_Port.options:requested-chassis`). Giá trị đơn (`"chassisB"`) = single binding; giá trị CSV (`"chassisA,chassisB"`) = multichassis. *See also:* multichassis port, ovn-northd.

**CAN_BIND_AS_MAIN**, Enum giá trị trả về từ `lport_can_bind_on_this_chassis()` (`controller/lport.c`) khi chassis hiện tại khớp với phần tử đầu của `requested-chassis` CSV. Kích hoạt `claim_lport()` → set `PB.chassis`. *See also:* CAN_BIND_AS_ADDITIONAL.

**CAN_BIND_AS_ADDITIONAL**, Enum giá trị trả về khi chassis hiện tại khớp phần tử thứ hai của CSV. Kích hoạt path set `PB.additional_chassis` thay vì `PB.chassis`. *See also:* CAN_BIND_AS_MAIN.

**additional_chassis**, Cột set trong SB Port_Binding, chứa UUID chassis thứ hai trong multichassis state. Rỗng (`[]`) khi port ở single binding. *See also:* multichassis port.

**OFTABLE_REMOTE_OUTPUT**, Table 38 trong br-int pipeline (OVN 24.03). Nơi `enforce_tunneling_for_multichassis_ports()` cài flow priority 110 với 2 output actions cho tunnel dispatch. *See also:* enforce_tunneling.

**OFTABLE_OUTPUT_LARGE_PKT_DETECT**, Table 41. Match gói tin lớn hơn `check_pkt_larger(mtu)` → set `reg9[1]=1` → resubmit table 42. *See also:* PMTUD, check_pkt_larger.

**OFTABLE_OUTPUT_LARGE_PKT_PROCESS**, Table 42. Match `reg9[1]==1` → gọi `reply_imcp_error_if_pkt_too_big()` → sinh ICMP Type 3 Code 4 → resubmit table 8. *See also:* FDP-620.

**check_pkt_larger**, OVN logical action so sánh gói tin size với MTU argument. Chèn tại cuối logical router egress pipeline. *See also:* PMTUD, RFC 1191.

**activation-strategy**, LSP option kiểm soát khi additional binding thật sự "live" trên destination chassis. Giá trị `rarp` (default) chờ RARP frame; `chassis` kích hoạt ngay khi claim. *See also:* RARP frame, additional-chassis-activated.

**additional-chassis-activated**, Option nội bộ set bởi `pinctrl_activation_strategy_handler()` sau khi nhận RARP 0x8035. Trigger ovn-northd re-translate để xóa 3 "cửa khóa" flow. *See also:* activation-strategy.

**always_tunnel**, Không phải option trực tiếp, thuật ngữ mô tả hành vi `enforce_tunneling_for_multichassis_ports()` cưỡng chế mọi traffic qua overlay tunnel kể cả localnet-eligible gói tin. *See also:* enforce_tunneling.

**localnet_learn_fdb**, Option của Logical Switch cho phép populate FDB table từ gói tin source MAC quan sát được trên localnet. Là điều kiện kích hoạt thứ 4 của FDP-620 (xem SDN Phần 17). *See also:* FDP-620.

**RARP frame**, Reverse ARP theo RFC 903 (Finlayson et al., 1984). Ethertype 0x8035, 60-byte frame trên wire. QEMU `announce_self()` gửi RARP sau khi VM unpause để gateway switch học MAC position mới. *See also:* announce_self.

**PMTUD**, Path MTU Discovery theo RFC 1191. Mechanism để host phát hiện MTU effective của đường đi đến destination, thông qua ICMP Fragmentation Needed (Type 3 Code 4) với field `icmp4.frag_mtu`. *See also:* check_pkt_larger, OFTABLE_OUTPUT_LARGE_PKT_DETECT.

**mtu_expires**, Kernel sysctl `net.ipv4.route.mtu_expires` (default 600s trên kernel ≥ 3.6). Thời gian entry MTU cache trong `inet_peer` sống trước khi expire. *See also:* PMTUD.

### Command Reference

Grouped theo use case. Hầu hết lệnh chạy trên **host node** (controller hoặc hypervisor); lệnh dành cho node khác ghi rõ trong prompt.

**(1) Inspection, kiểm tra trạng thái binding:**

```bash
# [controller]# Tất cả port đang ở multichassis state
ovn-sbctl --bare --columns=logical_port,additional_chassis \
  find Port_Binding additional_chassis!='[]'

# [controller]# Chi tiết một port
ovn-sbctl list Port_Binding $LSP

# [controller]# Options NB của LSP
ovn-nbctl lsp-get-options $LSP
```

**(2) Tracing, trace pipeline cho gói tin giả lập:**

```bash
# [hypervisor]# Trace OpenFlow pipeline
ovs-appctl ofproto/trace br-int "<flow_match_expression>"

# [controller]# Trace OVN logical pipeline
ovn-trace $LS "<microflow_expression>"
```

**(3) OVS flow, dump OpenFlow rules:**

```bash
# [hypervisor]# Tất cả flows
ovs-ofctl dump-flows br-int

# [hypervisor]# Filter theo cookie (thường = tunnel_key)
ovs-ofctl dump-flows br-int "cookie=$COOKIE/-1"

# [hypervisor]# Filter theo table
ovs-ofctl dump-flows br-int table=38

# [hypervisor]# Filter theo register match
ovs-ofctl dump-flows br-int "reg15=0x$(printf '%x' $PORT_KEY)"
```

**(4) Cấu hình, thay đổi hành vi:**

```bash
# [controller]# Set activation strategy per-port
ovn-nbctl lsp-set-options $LSP requested-chassis="hv1,hv2" \
    activation-strategy=rarp

# [controller]# Remove multichassis config
ovn-nbctl remove Logical_Switch_Port $LSP options requested-chassis
ovn-nbctl remove Logical_Switch_Port $LSP options activation-strategy

# [neutron-conf]# Global config (neutron.conf)
# [ovn] live_migration_activation_strategy = rarp
```

**(5) kolla-ansible, rebuild OVN image với patch cherry-picked:**

```bash
# [deploy-node]$ Build custom OVN image với Ales Musil patch
kolla-build ovn-controller \
    --template-override /etc/kolla/template-overrides.j2 \
    --registry quay.io/custom

# [deploy-node]$ Deploy
kolla-ansible -i inventory deploy --tags ovn
```

**(6) Kernel tuning, điều chỉnh hành vi MTU cache trên host (thường là VM):**

```bash
# [vm]# Xem giá trị hiện tại (default 600s)
sysctl net.ipv4.route.mtu_expires

# [vm]# Giảm expires xuống 60s (nếu migration diễn ra thường xuyên)
sysctl -w net.ipv4.route.mtu_expires=60

# [vm]# Flush MTU cache ngay lập tức (khi debug)
ip route flush cache

# [vm]# Xem MTU cache entries
ip route show cache
```

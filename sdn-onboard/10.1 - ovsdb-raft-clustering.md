## 10.1.1 Tại sao cần Raft clustering

Single-node OVSDB có hai giới hạn production:

1. **Single point of failure** — node chết → toàn bộ OVN control plane down.
2. **Không scale write** — chỉ một instance xử lý transaction.

Raft giải quyết bằng cách replicate log qua nhiều node, automatic leader election + failover. OVSDB Raft giới thiệu OVS 2.9 (02/2018).

Raft là consensus protocol do Diego Ongaro và John Ousterhout công bố tại USENIX ATC 2014 — thay thế Paxos với semantics đơn giản hơn nhưng equivalent guarantees.

---

## 10.1.2 Raft basics

Cluster N node (N lẻ — 3, 5, 7). Tại mỗi thời điểm:

- Một node là **Leader**, handle mọi write từ client.
- Các node khác là **Followers**, replicate log từ leader.
- Khi leader chết, follower bầu **Candidate** → leader mới (election).

**Log replication**: client gửi write đến leader. Leader append entry vào log, replicate đến followers. Khi quorum acknowledge, leader commit. Quorum cho 3-node = 2, cho 5-node = 3.

**Fault tolerance**:

| Cluster size | Chịu được failure |
|---|---|
| 3 nodes | 1 node down |
| 5 nodes | 2 node down |
| 7 nodes | 3 node down |

Cluster 9+ không khuyến nghị vì độ trễ write tăng (quorum cần acknowledge).

## 10.1.3 Bootstrap 3-node cluster

Compass Ch N workflow:

**Node 1:**

```bash
[node1]# ovsdb-tool create-cluster /etc/openvswitch/db \
         /usr/share/openvswitch/vswitch.ovsschema tcp:10.0.0.1:6644
[node1]# ovsdb-server --remote=ptcp:6640 \
         --remote=punix:/var/run/openvswitch/db.sock \
         /etc/openvswitch/db &
```

`tcp:10.0.0.1:6644` là Raft cluster RPC port (khác port client 6640).

**Node 2:**

```bash
[node2]# ovsdb-tool join-cluster /etc/openvswitch/db Open_vSwitch \
         tcp:10.0.0.2:6644 tcp:10.0.0.1:6644
[node2]# ovsdb-server /etc/openvswitch/db &
```

**Node 3:**

```bash
[node3]# ovsdb-tool join-cluster /etc/openvswitch/db Open_vSwitch \
         tcp:10.0.0.3:6644 tcp:10.0.0.1:6644 tcp:10.0.0.2:6644
[node3]# ovsdb-server /etc/openvswitch/db &
```

Client kết nối cluster bằng comma-separated member list:

```bash
[client]# ovn-nbctl --db="tcp:10.0.0.1:6641,tcp:10.0.0.2:6641,tcp:10.0.0.3:6641" \
           ls-add ls0
```

Nếu leader change, client tự retry sang leader mới (transparent failover).

## 10.1.4 Cluster lifecycle management

Check status:

```bash
[node]# ovs-appctl -t /var/run/openvswitch/ovsdb-server.ctl \
         cluster/status Open_vSwitch
```

Output hiện: leader ID, role (leader/follower/candidate), commit index, log index, members.

**Leave cluster** (graceful):

```bash
[node2]# ovs-appctl cluster/leave Open_vSwitch
```

**Kick failed member**:

```bash
[node1]# ovs-appctl cluster/kick Open_vSwitch <sid-of-dead-node>
```

**Rule quan trọng (compass Ch N)**: "Once a server leaves a cluster it cannot rejoin — create a fresh member."

Sau `leave` hoặc `kick`, **phải xóa DB file cũ** và `join-cluster` như node mới.

## 10.1.5 Production topology cho OVN

OVN deployment dùng hai cluster:

- **NB DB cluster** (3 node): logical topology ý đồ cấu hình. `ovn-nbctl` client.
- **SB DB cluster** (3 node): computed logical luồng + chassis state. `ovn-controller` client.

Hai cluster có thể chạy trên cùng 3 physical node (process riêng) để tiết kiệm hardware.

5-node cluster cho large deployment (>500 chassis) — higher availability nhưng write độ trễ cao hơn.

## Điểm cốt lõi cần nhớ

Thứ nhất, OVSDB Raft (OVS 2.9+) là solution HA cho configuration database. Bắt buộc cho OVN NB/SB DB production.

Thứ hai, cluster size N lẻ để quorum rõ ràng. 3-node chịu 1 failure, 5-node chịu 2. Cluster quá lớn (9+) write độ trễ cao.

Thứ ba, rule "node rời cluster không thể rejoin" — phải fresh join với DB file mới.

Thứ tư, client kết nối cluster bằng comma-separated list, tự failover sang leader mới. OVN clients support pattern này.

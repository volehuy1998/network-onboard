## Learning objectives

After completing this part, the engineer can:

1. **Understand** the role of `ovs-pki` as a **convenience wrapper around OpenSSL** dedicated to bootstrapping and maintaining the PKI hierarchy that secures SSL/TLS connections between OVS and OVN components (`ovs-vswitchd` to controller, `ovsdb-server` to `ovs-vsctl`, chassis to SB DB cluster).
2. **Apply** the CA bootstrap workflow (`init`, then `req`, then `sign`, then distribute) to prepare certificates for a production OVN cluster in a multi-chassis environment with mTLS authentication.
3. **Analyze** the directory layout of the PKI hierarchy (`/var/lib/openvswitch/pki/` by default) to understand the separation between `controllerca` and `switchca`, and the reason two independent CAs exist.
4. **Evaluate** the trade-off of `req+sign` (one combined step, easy to run but weaker security because the private key transits the CA) against `req` plus a separate `sign` (the correct production practice).

Prerequisite knowledge: PKI basics (CA, certificate, private key), the OpenSSL workflow, and root permission on the lab machine.

## 9.30.1 Why `ovs-pki` exists, more than just an OpenSSL wrapper

In a production OVN cluster, the recommended posture is to use SSL/TLS with mTLS (mutual TLS, where both client and server authenticate by certificate) on the connections between components:

- A chassis `ovn-controller` to the SB DB `ovsdb-server` cluster (port 6642).
- An `ovn-nbctl` client to the NB DB `ovsdb-server` cluster (port 6641).
- A local `ovs-vsctl` to a local `ovsdb-server` (a Unix socket by default, optional SSL when remote management is needed).
- An OpenFlow controller to `ovs-vswitchd` (port 6653).

Each connection requires (a) a certificate for the server, (b) a certificate for the client, and (c) both sides trusting one common CA. Setting up PKI by hand with raw `openssl` takes five to seven long commands and is error-prone (the `openssl.cnf` configuration file is complex, and the naming convention is easy to confuse).

`ovs-pki` simplifies the work: a fixed directory layout, a fixed naming convention, a default key size of 2048-bit RSA, a log file, and a batch mode for automation. It still uses OpenSSL underneath (it does not reinvent any cryptography); it is only a wrapper.

**The distinctive feature of the OVS PKI hierarchy:** `ovs-pki init` creates **two independent CAs** in the same PKI tree:

- `controllerca`: signs certificates for OpenFlow controllers and OVSDB clients (administrative tools such as `ovn-nbctl` and `ovs-vsctl`).
- `switchca`: signs certificates for `ovs-vswitchd` and `ovsdb-server` (the server side).

Splitting into two CAs is **defense in depth**. If one CA is compromised (for example, its private key is leaked), only one side (controller or switch) needs to re-issue, instead of the whole cluster.

## 9.30.2 Synopsis and commands

```
ovs-pki <options> <command> <args>...
```

**Commands (verified from the man page):**

| Command | Purpose |
|---------|---------|
| `init` | Initialize a new PKI hierarchy with two CAs (`controllerca` and `switchca`) |
| `req <name>` | Generate a private key and a certificate request (CSR) for `<name>` |
| `sign <name> [<type>]` | Sign a CSR into a certificate; `<type>` is `switch` or `controller` |
| `req+sign <name> [<type>]` | Combine `req` and `sign` in one command (use in dev or lab) |
| `verify <name> [<type>]` | Verify that a certificate is valid (signed by the correct CA) |
| `fingerprint <file>` | Print the SHA-1 digest of a file (a certificate or any other) |
| `self-sign <name>` | Issue a self-signed certificate for testing (not recommended for production) |

> **Note on accuracy:** plan v3.5 originally listed a `set-default` command. Verifying the `ovs-pki(8)` man page (OVS 2.17.9) shows that this command does NOT exist. This was a discrepancy between the plan estimate and the source-of-truth man page; file 9.30 follows the man page.

## 9.30.3 Options in detail

| Option | Default | Role |
|--------|---------|------|
| `-k <type>`, `--key=<type>` | `rsa` | Algorithm: `rsa` or `dsa` |
| `-B <nbits>`, `--bits=<nbits>` | `2048` | Key size; minimum 1024 |
| `-D <file>`, `--dsaparam=<file>` | auto-generated | Alternative `dsaparam.pem` for DSA |
| `-b`, `--batch` | OFF | Skip interactive fingerprint verification (for automation) |
| `-d <dir>`, `--dir=<dir>` | `/var/lib/openvswitch/pki` | PKI hierarchy location |
| `-f`, `--force` | OFF | Overwrite an existing file or directory |
| `-l <file>`, `--log=<file>` | default inside the PKI directory | Log file path |
| `-u`, `--unique` | OFF | Use the simple-name format for the CN field |
| `-h`, `--help` | none | Print help |

**Practical settings for production:**

- Always use `-B 4096` (4096-bit RSA) instead of the default 2048 for long-lived keys (over two years).
- Always use `-l /var/log/ovs-pki.log` to track the audit trail.
- Do NOT use `-b` in production (skipping fingerprint verification weakens security).

## 9.30.4 PKI hierarchy directory layout

After `ovs-pki init`, the default structure is:

```
/var/lib/openvswitch/pki/
|-- controllerca/
|   |-- cacert.pem               # Public CA cert (controller)
|   |-- private/
|   |   `-- cakey.pem            # CA private key (controller), STRICT 0600
|   |-- newcerts/                # Issued certs index
|   |-- crl/                     # Certificate Revocation List
|   |-- serial                   # Next serial number
|   `-- index.txt                # OpenSSL CA index
|-- switchca/
|   |-- cacert.pem               # Public CA cert (switch)
|   |-- private/
|   |   `-- cakey.pem            # CA private key (switch), STRICT 0600
|   |-- newcerts/
|   `-- ...
`-- log
```

**Critical permission:** the file `private/cakey.pem` of both CAs MUST have mode `0600` and owner `root` (or a dedicated `ovs-pki` user). If the mode is broader than `0644`, an attacker can read the CA private key, sign forged certificates for any chassis, and move laterally inside the cluster.

`ovs-pki init` sets the correct permission by default, but a manual `cp` to another machine may lose those bits.

## 9.30.5 Anatomy: the `ovs-pki req+sign` workflow

**Bucket:** OVS. **Context:** lab or dev work, or bootstrapping a brand-new chassis. **Purpose:** generate a private key and a certificate, and sign it with the CA, in one command. **Activity:** provisioning. **Mechanism:** wrap the OpenSSL `req` and `ca` commands.

**Sample output:**

```
$ sudo ovs-pki req+sign hv01 switch
Generating private key for hv01...
Generating certificate request for hv01...
Signing certificate request for hv01 with switch CA...
Verifying that signed certificate is valid for switch use...

Created files in current directory:
  hv01-privkey.pem    # Private key (KEEP SECRET)
  hv01-req.pem        # Certificate request (CSR)
  hv01-cert.pem       # Signed certificate
```

**Decode, five-attribute Anatomy:**

| Attribute | Sample value | Healthy / Warning / Critical |
|-----------|--------------|------------------------------|
| Output files | Three files: privkey, req, cert | Healthy: all three exist in the current working directory |
| Private-key permission | `0600` mode | Critical: mode `0644` means key-leak risk; `chmod 600` immediately |
| Certificate validity period | OVS default = one year | Warning: needs renewal before expiry; Critical: expired = lost connection |
| Cert signed by CA | switchca | Verify: `openssl x509 -in hv01-cert.pem -text | grep Issuer` |
| Cert subject (CN) | hostname or unique ID | Must match the chassis name in OVN |

**Critical signal:**

- The `req+sign` command MUST run on the CA machine (it needs the CA private key to sign). Do NOT run it on a production chassis, to avoid exposing the CA key.
- The correct production mode: run `req` on the chassis (which keeps its private key local), copy the CSR to the CA machine, run `sign` on the CA, and copy the signed certificate back.

## 9.30.6 Guided exercise

**Goal:** bootstrap a one-CA, one-`ovsdb-server` SSL endpoint lab, and connect successfully using `ovs-vsctl --ssl-*`.

**Set up the CA:**

```bash
# 1. Init the PKI hierarchy (only once, on the CA machine)
sudo ovs-pki init -d /tmp/lab-pki

# Verify the structure
ls -la /tmp/lab-pki/controllerca/
ls -la /tmp/lab-pki/switchca/private/cakey.pem  # mode 0600
```

**Generate a certificate for the server (`ovsdb-server`):**

```bash
# 2. req+sign for the server (lab convenience)
cd /tmp/lab-server
sudo ovs-pki -d /tmp/lab-pki req+sign server01 switch

# Output: server01-privkey.pem, server01-req.pem, server01-cert.pem
```

**Generate a certificate for the client (`ovs-vsctl`):**

```bash
# 3. req+sign for the client
cd /tmp/lab-client
sudo ovs-pki -d /tmp/lab-pki req+sign client01 controller
```

**Configure `ovsdb-server` with SSL:**

```bash
# 4. Start ovsdb-server with an SSL listener
sudo ovsdb-server /etc/openvswitch/conf.db \
  --remote=pssl:6640 \
  --private-key=/tmp/lab-server/server01-privkey.pem \
  --certificate=/tmp/lab-server/server01-cert.pem \
  --ca-cert=/tmp/lab-pki/controllerca/cacert.pem \
  --pidfile --detach
```

**Test the connection from the client:**

```bash
# 5. Verify that ovs-vsctl connects over SSL
sudo ovs-vsctl \
  --db=ssl:127.0.0.1:6640 \
  --private-key=/tmp/lab-client/client01-privkey.pem \
  --certificate=/tmp/lab-client/client01-cert.pem \
  --ca-cert=/tmp/lab-pki/switchca/cacert.pem \
  list-br

# Expected output: list of bridges (no SSL error)
```

**Verify certificate details:**

```bash
# 6. Inspect the cert
openssl x509 -in /tmp/lab-server/server01-cert.pem -text -noout
sudo ovs-pki -d /tmp/lab-pki verify server01 switch
sudo ovs-pki fingerprint /tmp/lab-server/server01-cert.pem
```

## 9.30.7 Anti-patterns and notes

**Anti-pattern 1: running `req+sign` on a production chassis.**

`req+sign` needs the CA private key to sign, which means the CA key would be copied onto the chassis as part of this workflow. Production must split the steps: run `req` on the chassis (it generates and keeps the private key local), copy the CSR to the CA machine over a secure channel, run `sign` only on the CA, and copy the signed certificate back.

**Prevention:** in the chassis deployment runbook, add an explicit step: "DO NOT run `req+sign` on a production hypervisor".

**Anti-pattern 2: self-signed certificates in a production cluster.**

`ovs-pki self-sign` issues a self-signed certificate without a CA. It is convenient for a single-node test, but a production cluster with mTLS requires certificates signed by a common CA. Self-signed certificates do not scale to N chassis.

**Prevention:** use `self-sign` only for lab and CI tests. Document this clearly in the cluster bootstrap guide.

**Note 3: certificate rotation.**

OVS certificates are valid for one year by default. There is no auto-renewal. The operator must set a calendar reminder and a rotation script:

```bash
# Auto-renew script, monthly cron
ovs-pki -d /var/lib/openvswitch/pki req+sign hv-$(hostname) switch
# Then update ovs-vswitchd config and restart
```

**Note 4: the CRL (Certificate Revocation List) does not update automatically.**

When a certificate is compromised (a chassis is decommissioned or hacked), the operator must do the work manually:

1. Add the cert serial to the CRL: `openssl ca -gencrl -out crl.pem -config switchca.conf`.
2. Distribute the CRL to every server endpoint.
3. Configure the server to load the CRL using the option `--ca-cert-bootstrap` (advanced).

OVS has no centralized CRL service. Production deployments often use an external PKI (HashiCorp Vault, step-ca) instead of `ovs-pki` for automated rotation.

**Note 5: back up the CA private key offline.**

Losing `cakey.pem` means losing the ability to sign new certificates, which forces a full PKI re-init and re-issuing every chassis certificate. Back up `cakey.pem` to cold storage (an encrypted USB drive, a vault) right after `init`.

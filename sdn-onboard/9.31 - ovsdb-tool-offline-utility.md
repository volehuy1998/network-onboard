## Mục tiêu bài học

Sau khi hoàn thành phần này, kỹ sư có thể:

1. **Hiểu (Understand)** vị trí của `ovsdb-tool` trong hệ tool OVSDB: **offline file utility** (không kết nối tới ovsdb-server đang chạy), khác với `ovsdb-client` (online) và `ovs-vsctl` (high-level wrapper). 15 commands chia 5 nhóm: creation, schema management, integrity, inspection, cluster lifecycle.
2. **Áp dụng (Apply)** workflow `create-cluster` + `join-cluster` để bootstrap cụm OVN_Southbound 3-node Raft từ scratch — kỹ năng critical cho disaster recovery + cụm production setup.
3. **Phân tích (Analyze)** output của `show-log` và `check-cluster` để forensic-analyze database state khi gặp split-brain hoặc corruption.
4. **Đánh giá (Evaluate)** an toàn vận hành: khi nào dùng `ovsdb-tool` (server stopped) vs `ovsdb-client` (server chạy), tại sao chạy `compact` trên database đang serve sẽ corrupt.

Yêu cầu kiến thức: hiểu OVSDB file format JSON-on-disk + transactional log qua Phần 10.0, hiểu Raft consensus qua Phần 10.1.

## 9.31.1 Vị trí `ovsdb-tool` trong hệ tool OVSDB

OVS có 3 tool chính tương tác với database, mỗi tool dùng cho mục đích khác nhau:

| Tool | Online/Offline | Use case |
|------|----------------|----------|
| `ovsdb-tool` | **Offline** (file trực tiếp) | Bootstrap cluster, schema migration, forensic, compact |
| `ovsdb-client` | **Online** (kết nối ovsdb-server) | Live monitor, ad-hoc transact, dump table |
| `ovs-vsctl` / `ovn-nbctl` / `ovn-sbctl` | **Online** (wrapper qua OVSDB protocol) | Daily admin, high-level abstractions |

**Tại sao cần offline tool?** Một số task không thể làm khi server đang chạy:
1. **Bootstrap cluster mới**: server chưa start được nếu file chưa initialized.
2. **Schema migration**: convert db từ version cũ sang mới khi server stop để upgrade.
3. **Compact**: defragment file (giảm size, gộp transaction log) — server đang ghi sẽ conflict.
4. **Forensic post-incident**: dump transaction log từ file bị corrupt mà server không load được.
5. **Cluster recovery**: convert failed cluster thành standalone để rescue data.

Quy tắc vàng: `ovsdb-tool` chỉ chạy khi `ovsdb-server` cho database tương ứng **ĐANG STOP**. Vi phạm = race condition, file corruption, data loss.

## 9.31.2 Synopsis + 15 commands chia 5 nhóm

```
ovsdb-tool [options] <command> [arguments]
```

### Nhóm 1: Database creation (3 commands)

| Command | Synopsis | Mục đích |
|---------|----------|----------|
| `create` | `create [db [schema]]` | Khởi tạo standalone/active-backup database từ schema |
| `create-cluster` | `create-cluster [--election-timer=MS] DB CONTENTS LOCAL` | Tạo Raft cluster, server đầu tiên |
| `join-cluster` | `join-cluster [--cid=UUID] DB NAME LOCAL REMOTE...` | Thêm server vào cluster đã tồn tại |

### Nhóm 2: Schema management (5 commands)

| Command | Synopsis | Mục đích |
|---------|----------|----------|
| `convert` | `convert [db [schema [target]]]` | Convert database sang schema version mới |
| `needs-conversion` | `needs-conversion [db [schema]]` | Kiểm tra cần conversion không (exit code) |
| `db-version` | `db-version [db]` | In version của database |
| `schema-version` | `schema-version [schema]` | In version của schema file |
| `db-cksum`, `schema-cksum` | `db-cksum [db]` / `schema-cksum [schema]` | In checksum SHA-1 của file |
| `compare-versions` | `compare-versions A OP B` | So sánh hai version string (`<` `<=` `==` `>=` `>` `!=`) |

### Nhóm 3: Inspection + query (5 commands)

| Command | Synopsis | Mục đích |
|---------|----------|----------|
| `query` | `query [--rbac-role=ROLE] [db] TRANSACTION` | Read-only query execution |
| `transact` | `transact [--rbac-role=ROLE] [db] TRANSACTION` | Read+write transaction (trên file offline) |
| `show-log` | `show-log [-m|--more]... [db]` | Dump transaction log (verbose mode `-m`) |
| `db-name`, `schema-name` | — | In name của db/schema |
| `db-cid`, `db-sid`, `db-local-address` | — | In cluster ID / server ID / local address của Raft node |

### Nhóm 4: Cluster integrity (3 commands)

| Command | Synopsis | Mục đích |
|---------|----------|----------|
| `check-cluster` | `check-cluster DB...` | Verify consistency cross-node trong Raft cluster |
| `db-is-clustered`, `db-is-standalone` | — | Test format database |
| `cluster-to-standalone` | `cluster-to-standalone DB CLUSTERDB` | Convert failed cluster thành standalone (rescue data) |

### Nhóm 5: Maintenance (1 command)

| Command | Synopsis | Mục đích |
|---------|----------|----------|
| `compact` | `compact [db [target]]` | Defragment + gộp log + giảm file size |

## 9.31.4 Anatomy: bootstrap 3-node Raft cluster cho OVN_Southbound

**Bucket:** OVS | **Context:** disaster recovery hoặc cluster setup mới | **Purpose:** dựng cụm OVN SB DB 3-node từ scratch | **Activity:** cluster lifecycle | **Mechanism:** create-cluster trên node 1 + join-cluster trên node 2-3.

**Trên Node 1 (leader đầu tiên):**

```bash
# 1. Stop existing ovsdb-server cho SB DB (nếu có)
systemctl stop ovn-sb-ovsdb

# 2. Create cluster với schema OVN_Southbound
ovsdb-tool create-cluster /etc/ovn/ovnsb_db.db \
  /usr/share/ovn/ovn-sb.ovsschema \
  tcp:10.0.0.1:6644

# 3. Verify cluster ID + server ID
ovsdb-tool db-cid /etc/ovn/ovnsb_db.db
# → Cluster UUID (ghi nhớ cho join-cluster sau)

ovsdb-tool db-sid /etc/ovn/ovnsb_db.db
# → Server UUID (unique cho node này)

# 4. Start ovsdb-server cluster mode
ovsdb-server /etc/ovn/ovnsb_db.db \
  --remote=ptcp:6642 \
  --pidfile --detach
```

**Trên Node 2 + Node 3 (followers):**

```bash
# 5. Join existing cluster
ovsdb-tool join-cluster \
  /etc/ovn/ovnsb_db.db \
  OVN_Southbound \
  tcp:10.0.0.2:6644 \
  tcp:10.0.0.1:6644 \
  tcp:10.0.0.3:6644

# 6. Start server
ovsdb-server /etc/ovn/ovnsb_db.db \
  --remote=ptcp:6642 \
  --pidfile --detach
```

**Verify cluster healthy từ bất kỳ node:**

```bash
# 7. Check cluster status (online tool, sau khi server start)
ovs-appctl -t /var/run/ovn/ovnsb_db.ctl cluster/status OVN_Southbound

# Expected output:
# Cluster ID: a1b2c3d4-...
# Server ID: e5f6g7h8-... (Leader hoặc Follower)
# 3 servers in the cluster, status "Active"
# Election timer: 10000 ms
```

## 9.31.5 Anatomy: `show-log` forensic transaction log

**Bucket:** OVS | **Context:** post-incident analysis hoặc debug split-brain | **Purpose:** dump toàn bộ transaction log đã commit vào file | **Activity:** forensic | **Mechanism:** parse Raft log entries từ file binary.

**Output mẫu (cluster mode):**

```
$ ovsdb-tool show-log /etc/ovn/ovnsb_db.db | head -20
record 0:
 Initial schema "OVN_Southbound" version 2.5.0 cksum "..."

record 1: 2026-04-25 10:23:45.123
 cluster term 5, prev_index 0, name "..."

record 2: 2026-04-25 10:23:46.234 "raft_initial_snapshot"
 entries [{...snapshot data...}]

record 3: 2026-04-25 10:24:01.456
 [insert Chassis a1b2c3d4-... name "hv01" hostname "compute01.example.com"]

record 4: 2026-04-25 10:24:01.789
 [insert Port_Binding ... logical_port "lsp1" datapath ...]
```

**Verbose mode `-m` (more):**

```bash
ovsdb-tool show-log -m /etc/ovn/ovnsb_db.db | head -30
# Hiển thị thêm: term, index, prev_term, prev_index của mỗi entry Raft
```

**Decode 5-attribute Anatomy:**

| Attribute | Giá trị mẫu | Healthy / Warning / Critical |
|-----------|-------------|------------------------------|
| Record format | `record N: TIMESTAMP DESCRIPTION` | Healthy: timestamp tăng đều |
| Initial schema record | record 0 | Critical: thiếu = file corrupt từ đầu |
| Snapshot record | "raft_initial_snapshot" | Healthy: có snapshot định kỳ; Critical: thiếu cho cluster lâu năm = không snapshot, log dài + slow |
| Transaction count | grep "^record " count | Warning: > 1M = chưa compact lâu |
| Term progression | Cluster term tăng monotonic | Critical: term tụt = corrupt |

**Use case forensic:**

- **Split-brain investigation:** so sánh `show-log` của 2 node → tìm divergence point (term + index khác nhau từ thời điểm nào).
- **Lost transaction:** check timestamp gap (>10s không có record) → có thể leader chết, election diễn ra.
- **Performance tuning:** nếu `compact` chưa chạy trong 30 ngày, file lớn hơn snapshot 3-5x → cần lên schedule compact.

## 9.31.6 Bài tập có hướng dẫn (Guided Exercise)

**Mục tiêu:** bootstrap 3-node OVN_Southbound cluster từ scratch trên 1 máy lab (3 ovsdb-server processes + 3 file riêng).

**Setup:**

```bash
# 1. Tạo 3 directory cho 3 mock node
mkdir -p /tmp/sb-{node1,node2,node3}

# 2. Tải schema (giả lập có sẵn)
SCHEMA=/usr/share/ovn/ovn-sb.ovsschema

# 3. Bootstrap node 1
ovsdb-tool create-cluster /tmp/sb-node1/db \
  $SCHEMA \
  tcp:127.0.0.1:6644

# Note cluster UUID
CID=$(ovsdb-tool db-cid /tmp/sb-node1/db)
echo "Cluster ID: $CID"

# 4. Start node 1 ovsdb-server
ovsdb-server /tmp/sb-node1/db \
  --remote=ptcp:6642:127.0.0.1 \
  --pidfile=/tmp/sb-node1/pid \
  --detach
```

**Join node 2 + 3:**

```bash
# 5. Node 2 join
ovsdb-tool join-cluster \
  /tmp/sb-node2/db \
  OVN_Southbound \
  tcp:127.0.0.1:6645 \
  tcp:127.0.0.1:6644

ovsdb-server /tmp/sb-node2/db \
  --remote=ptcp:6643:127.0.0.1 \
  --pidfile=/tmp/sb-node2/pid \
  --detach

# 6. Node 3 join
ovsdb-tool join-cluster \
  /tmp/sb-node3/db \
  OVN_Southbound \
  tcp:127.0.0.1:6646 \
  tcp:127.0.0.1:6644

ovsdb-server /tmp/sb-node3/db \
  --remote=ptcp:6647:127.0.0.1 \
  --pidfile=/tmp/sb-node3/pid \
  --detach
```

**Verify cluster:**

```bash
# 7. Check cluster status
ovs-appctl -t /tmp/sb-node1/ctl cluster/status OVN_Southbound

# Expected: 3 servers active, 1 leader + 2 followers

# 8. Test integrity check
ovsdb-tool check-cluster /tmp/sb-node1/db /tmp/sb-node2/db /tmp/sb-node3/db
# Expected: cluster is consistent
```

**Simulate failure + recovery:**

```bash
# 9. Kill node 1 (leader)
kill $(cat /tmp/sb-node1/pid)

# 10. Verify election lại trên node 2 hoặc 3
ovs-appctl -t /tmp/sb-node2/ctl cluster/status OVN_Southbound
# Expected: 1 trong 2 node còn lại trở thành leader

# 11. Khởi động lại node 1 (rejoin tự động)
ovsdb-server /tmp/sb-node1/db \
  --remote=ptcp:6642:127.0.0.1 \
  --pidfile=/tmp/sb-node1/pid \
  --detach
```

## 9.31.7 Mẫu cần tránh và lưu ý

**Mẫu cần tránh 1: Chạy `compact` trên database đang serve.**

Man page warning verbatim: *"This command does not work if db is currently being served by ovsdb-server"*. Vi phạm = file lock conflict, lỗi corruption ở level transaction log.

**Cách phòng:** stop ovsdb-server trước, chạy compact, sau đó start lại. Hoặc dùng online tool: `ovs-appctl -t ovsdb-server ovsdb-server/compact` (server tự xử lý không cần stop).

**Mẫu cần tránh 2: `transact` trên file đang được server write.**

Cùng lý do — race condition. Dùng `ovsdb-client transact <socket>` cho live database thay vì `ovsdb-tool transact <file>`.

**Mẫu cần tránh 3: `convert` schema sai version.**

Convert sai version (ví dụ schema 2.5 → 2.0 backward incompatible) sẽ corrupt data. Luôn:
1. Backup file trước (`cp db db.bak`).
2. Run `needs-conversion` trước để verify direction.
3. Test convert trên test environment trước production.

**Lưu ý 4: `show-log` output rất dài cho database production.**

Cluster SB DB sau 6 tháng có thể có 10M+ transaction. `show-log` không có pagination → dùng:

```bash
ovsdb-tool show-log /etc/ovn/ovnsb_db.db 2>&1 | less
# Hoặc grep timestamp
ovsdb-tool show-log /etc/ovn/ovnsb_db.db 2>&1 | grep "2026-04-25"
```

**Lưu ý 5: `check-cluster` cần access tất cả file.**

`check-cluster DB1 DB2 DB3` cross-verify state. Cần copy file từ 3 node về 1 nơi (sau khi stop server) rồi run. Không thể verify cluster từ xa (no network protocol).

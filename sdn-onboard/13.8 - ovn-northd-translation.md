## 13.8.1 Vai trò ovn-northd trong kiến trúc OVN

**ovn-northd** là tiến trình daemon chạy tập trung trong cụm OVN (thường trên cùng node với ovsdb-server NB/SB, hoặc tách sang node riêng trong cấu hình HA). Nhiệm vụ duy nhất: đọc NB DB (ý đồ cấu hình do CMS — OpenStack Neutron, OVN-Kubernetes, v.v., ghi vào) và compile xuống SB DB (Logical Flow + Port_Binding + Multicast_Group) để ovn-controller cục bộ đọc và thực thi trên br-int.

Kiến trúc **pipeline compile NB → SB** có ba tầng rõ ràng:
- **Tầng 1: NB DB (ý đồ cấu hình):** CMS ghi các object cấp cao như Logical_Switch, Logical_Router, Logical_Switch_Port, ACL, Load_Balancer, NAT, DHCP_Options. Không chứa OpenFlow, chỉ định nghĩa topology và policy.
- **Tầng 2: ovn-northd (translator):** daemon theo dõi mọi thay đổi NB DB qua OVSDB IDL, tính toán Logical Flow tương ứng, và ghi vào SB DB. Cũng tính Port_Binding (binding port với chassis), Datapath_Binding (mỗi LS/LR được gán tunnel_key), Multicast_Group (broadcast domain).
- **Tầng 3: SB DB (physical-neutral intermediate):** chứa Logical Flow độc lập chassis, mọi ovn-controller đọc cùng SB DB và dịch tiếp sang OpenFlow phù hợp chassis cục bộ (đã được đề cập Phần 13.7).

Mô hình này cho phép **phân tách trách nhiệm (separation of concerns)**: CMS chỉ cần biết ý đồ cấu hình cấp cao (Logical Switch, ACL), không cần biết OpenFlow hay tunnel encap. ovn-northd là trung tâm compile phức tạp, biết toàn bộ topology NB, tính toán mọi tương tác (ACL × Port × Load_Balancer × NAT) và sinh Logical Flow đầy đủ. ovn-controller cục bộ chỉ cần dịch subset Logical Flow liên quan chassis → OpenFlow, không cần biết gì về ý đồ cấu hình NB.

Một cụm OVN chuẩn có **một** instance ovn-northd active tại một thời điểm (với một hoặc nhiều standby dưới bầu leader). Lý do: ovn-northd ghi rất nhiều record vào SB DB cho mỗi thay đổi NB, nếu nhiều instance chạy song song sẽ gây xung đột ghi. Bầu leader (qua pacemaker, kubernetes leader election, hoặc built-in OVN `--ovnnb-db=tcp:nb1,tcp:nb2,tcp:nb3 --ovnsb-db=tcp:sb1,tcp:sb2,tcp:sb3`) đảm bảo chỉ một active. Standby chỉ chờ, nếu active chết (phát hiện qua mất kết nối + timeout), một standby được promote thành active.

> **Lưu ý phiên bản:** OVN 20.06 trở về trước dùng ovn-northd đơn tiến trình, mỗi lần NB thay đổi là **tính toán lại toàn bộ** NB → SB. OVN 22.03 giới thiệu **compile song song (parallel compilation)** — ovn-northd đa flow xử lý nhiều Logical Switch/Router song song, cải thiện băng thông thực tế 3-5 lần trên multi-core CPU. OVN 24.03+ giới thiệu **incremental northd** (hiện đang preview) — tương tự I-P của ovn-controller, chỉ tính toán lại subset Logical Flow bị ảnh hưởng bởi NB change.

## 13.8.2 Thuật toán dịch NB → SB Logical Flow

Khi NB DB thay đổi (ví dụ CMS tạo Logical Switch mới), ovn-northd thực hiện chuỗi bước:

**Bước 1 — Tính toán datapath và tunnel_key:**
- Mỗi Logical Switch và Logical Router trong NB map tới một `Datapath_Binding` record trong SB.
- ovn-northd assign `tunnel_key` (1-16777215) duy nhất cho mỗi datapath, dùng làm OpenFlow metadata (đã được đề cập Phần 13.7).
- Datapath_Binding record được tạo/update/xóa theo NB.

**Bước 2 — Compile Logical Flow cho Logical Switch ingress pipeline:**
Mỗi Logical Switch có khoảng 30 table logical trong ingress pipeline, số hóa từ table 0 đến 29. ovn-northd sinh Logical Flow cho mỗi table theo thứ tự:
- `ls_in_check_port_sec`, port security check (nguồn từ `Logical_Switch_Port.port_security`)
- `ls_in_apply_port_sec`, apply port security drop
- `ls_in_lookup_fdb`: FDB lookup (MAC learning)
- `ls_in_put_fdb`: FDB insert cho packet có MAC nguồn chưa biết
- `ls_in_pre_acl`, stateful ACL pre-check
- `ls_in_pre_lb`: LB pre-routing
- `ls_in_pre_stateful`, conntrack commit trigger
- `ls_in_acl_hint`. ACL hint cho ct_state
- `ls_in_acl`. ACL từ table `ACL` trong NB
- `ls_in_qos_mark`: QoS mark
- `ls_in_qos_meter`: QoS meter
- `ls_in_lb`: Load balancer DNAT (frontend → backend)
- `ls_in_lb_aff_check`: LB affinity check
- `ls_in_stateful`. NAT commit
- `ls_in_arp_rsp`: ARP responder cho Logical Router port
- `ls_in_dhcp_options`: DHCPv4/DHCPv6 response
- `ls_in_dns_lookup`: DNS lookup
- `ls_in_dns_response`: DNS response
- `ls_in_external_port`: External port handling
- `ls_in_l2_lkup`: L2 destination lookup
- `ls_in_l2_unknown`: L2 unknown MAC (flood)

Mỗi table có Logical Flow entries với priority + match + action. Ví dụ ACL `from-lport direction=inbound priority=500 match=ip4.src==10.0.0.0/8 action=drop` compile thành:
```
table=ls_in_acl, priority=500, match=(inport == <port_groups> && ip4.src == 10.0.0.0/8), action=(drop;)
```

**Bước 3 — Compile Logical Flow cho Logical Switch egress pipeline:**
Tương tự ingress nhưng với egress, khoảng 10 table (egress ACL, egress LB, port security check egress, L2 output, v.v.).

**Bước 4 — Compile Logical Flow cho Logical Router ingress + egress:**
Logical Router phức tạp hơn, khoảng 30 table ingress + 15 table egress (OVN 22.03):
- `lr_in_admission`, kiểm soát admission
- `lr_in_lookup_neighbor`: ARP/NDP lookup
- `lr_in_learn_neighbor`: ARP learn
- `lr_in_ip_input`: IP input processing (giảm TTL, kiểm tra destination)
- `lr_in_unsnat`. SNAT ngược
- `lr_in_defrag`, defragmentation cho traffic LB
- `lr_in_dnat`. DNAT + LB
- `lr_in_ecmp_stateful`, chọn đường ECMP
- `lr_in_nd_ra_options`: IPv6 ND Router Advertisement
- `lr_in_ip_routing_pre`, hook pre-routing
- `lr_in_ip_routing`, tra cứu table định tuyến
- `lr_in_ip_routing_ecmp`, định tuyến ECMP
- `lr_in_policy`: Logical Router Policy (định tuyến theo policy)
- `lr_in_arp_resolve`, giải mã ARP cho next-hop
- `lr_in_chk_pkt_len`, kiểm tra chiều dài packet cho PMTU
- `lr_in_larger_pkts`, gửi ICMP "Packet Too Big" nếu cần
- `lr_in_gw_redirect`, chuyển hướng tới gateway chassis
- `lr_in_arp_request`, gửi ARP request

**Bước 5 — Port_Binding và Multicast_Group:**
Mỗi Logical_Switch_Port và Logical_Router_Port trong NB được tạo Port_Binding record trong SB. Port_Binding chứa `type` (vif/localnet/l2gateway/chassisredirect/patch), `options`, `tunnel_key`. Broadcast domain (flooding) được tính toán thành Multicast_Group, mỗi LS có một MG cho flood, mỗi LRP có MG cho neighbor discovery.

## 13.8.3 HA bầu leader của ovn-northd

Vì chỉ một ovn-northd active được phép ghi SB DB tại một thời điểm, cụm môi trường production OVN cần cơ chế HA đảm bảo (a) luôn có một active nếu node hiện tại crash, (b) tránh split-brain (hai active cùng ghi → bất đồng).

**Phương pháp 1 — Pacemaker (cũ):** dùng Pacemaker cluster manager quản lý resource `ovn-northd`. Pacemaker dùng Corosync để bầu leader và start/stop `ovn-northd.service` tương ứng. Phương pháp này phổ biến trên RHEL OpenShift 3.x nhưng phức tạp setup, đang dần bị thay thế.

**Phương pháp 2 — Bầu leader built-in (OVN 22.03+):** ovn-northd tự dùng ovsdb-server để bầu leader. Mỗi instance ovn-northd kết nối cả cụm NB DB và SB DB (Raft). Instance nào lấy được khóa `ovn_northd` trên SB DB trước → trở thành active. Các instance khác đăng ký khóa này, khi active release (tắt an toàn) hoặc mất kết nối, một standby tự động lấy khóa.

```
[node1]# ovn-northd --pidfile --detach \
         --ovnnb-db=tcp:10.0.0.1:6641,tcp:10.0.0.2:6641,tcp:10.0.0.3:6641 \
         --ovnsb-db=tcp:10.0.0.1:6642,tcp:10.0.0.2:6642,tcp:10.0.0.3:6642
[node2]# <cùng command, khác node>
[node3]# <cùng command, khác node>
```

Chỉ một instance active tại một thời điểm, xác định qua:
```
[host]# ovs-appctl -t ovn-northd nb-connection-status
[host]# ovs-appctl -t ovn-northd sb-connection-status
```
Instance active hiển thị `connected` cho cả hai; standby hiển thị `connected` NB nhưng `waiting` cho SB (đang chờ khóa).

**Phương pháp 3 — Kubernetes bầu leader (OVN-Kubernetes):** dùng Kubernetes lease object cho bầu leader. `ovn-k8s-northd` controller quản lý state qua `Lease` resource trong namespace `ovn-kubernetes`.

Thời gian failover (active chết → standby lên): 5-15 giây tùy cấu hình timeout. Trong khoảng thời gian này, SB DB không nhận được Logical Flow update mới (NB changes tồn đọng). Ovn-controller vẫn chạy bình thường vì đọc SB DB local cache, chỉ ý đồ cấu hình mới bị delay cho đến khi active mới lên.

## 13.8.4 Gỡ lỗi ovn-northd

Chuỗi gỡ lỗi ovn-northd thường xoay quanh: "tôi thay đổi NB nhưng SB không phản ánh" hoặc "Logical Flow SB sai so với NB ý đồ cấu hình".

**Công cụ 1, so sánh state NB và SB:**
```
[host]# ovn-nbctl show
[host]# ovn-sbctl show
[host]# ovn-sbctl list Logical_Flow | head -50
```

**Công cụ 2, ovn-northd coverage:**
```
[host]# ovs-appctl -t ovn-northd coverage/show
```
Xuất bộ đếm: `northd_run` (số lần tính toán lại), `lflow_generated` (số Logical Flow tạo), `port_binding_up` (số port binding). Giám sát qua thời gian để phát hiện bất thường.

**Công cụ 3, northd log:**
```
[host]# tail -f /var/log/ovn/ovn-northd.log
```
Mỗi lần tính toán lại log một entry:
```
2026-04-22T14:23:10.123Z|00042|ovn_northd|INFO|northd recompute took 45 ms
```
Nếu thời gian tăng bất thường (ví dụ vài giây) → NB phát triển quá lớn hoặc có loop thuật toán compile.

**Công cụ 4, đồng bộ nb_cfg:**
NB DB có bộ đếm `NB_Global.nb_cfg` tăng khi có thay đổi. ovn-northd copy bộ đếm này vào SB DB `SB_Global.nb_cfg` sau mỗi lần tính toán lại. So sánh hai bộ đếm:
```
[host]# ovn-nbctl get NB_Global . nb_cfg
42
[host]# ovn-sbctl get SB_Global . nb_cfg
42
```
Nếu khác nhau đáng kể (ví dụ NB = 42, SB = 38) → ovn-northd không catch up kịp thay đổi NB, có tồn đọng.

**Công cụ 5, ovn-northd đa flow (22.03+):**
Kiểm tra số thread:
```
[host]# ps -T -p $(pidof ovn-northd)
```
Mặc định ovn-northd dùng N thread với N = số CPU core / 2. Có thể override qua `--n-threads=4`. Theo dõi CPU per-thread để xác định điểm nghẽn.

## 13.8.5 Kiến trúc source code, từ `main()` đến `build_lflows()`

Phần này đi sâu vào tổ chức mã nguồn ovn-northd ở baseline OVN 22.03.8 (`branch-22.03` upstream). Ba file cốt lõi:

- `northd/ovn-northd.c` (~1022 dòng): chứa `main()` + parse_options + bầu leader + main loop top-level. Đây là daemon entry point.
- `northd/inc-proc-northd.c` (~296 dòng): I-P (Incremental Processing) engine bootstrap. Khai báo và chạy 2 engine node của ovn-northd.
- `northd/northd.c` (~15947 dòng): toàn bộ logic dịch NB sang SB Logical Flow. File lớn nhất trong source tree OVN, chứa hàng trăm function `build_*()` cho từng pipeline stage.

### Chuỗi hàm gọi từ entry tới sinh Logical Flow

```
main()                                  [northd/ovn-northd.c]
  └─> ovsdb_idl loop
        └─> inc_proc_northd_run()       [northd/inc-proc-northd.c]
              ├─> engine_run()            (chạy DAG 2 node: en_northd + en_lflow)
              │     ├─> en_northd_run()   [northd/en-northd.c]
              │     │     └─> ovnnb_db_run()         [northd/northd.c]
              │     │           ├─> build_datapaths()
              │     │           ├─> build_ports()
              │     │           ├─> build_lrouter_lbs()
              │     │           └─> ... (~30 build_* function)
              │     └─> en_lflow_run()    [northd/en-lflow.c]
              │           └─> build_lflows()           [northd/northd.c]
              │                 ├─> build_lswitch_input_port_sec_op()
              │                 ├─> build_lswitch_learn_fdb_op()
              │                 ├─> build_pre_acls()
              │                 ├─> build_acls()
              │                 ├─> build_qos()
              │                 ├─> build_lb()
              │                 └─> ... (~50 build_* per pipeline stage)
              └─> commit transaction (write SB)
```

**Phân vai:**

- `main()`: setup signal handler + parse `--ovnnb-db=...` + `--ovnsb-db=...` + bầu leader + vòng lặp chính.
- `inc_proc_northd_run()`: trigger engine chạy 1 vòng, gọi mỗi node `en_*_run()` theo thứ tự DAG.
- `ovnnb_db_run()`: tính toán **datapath + ports + LB + ACL parsed**, lưu vào `struct northd_data` (state cấu trúc trung gian, không ghi SB ngay).
- `build_lflows()`: dùng `northd_data` để sinh **Logical Flow** thực tế ghi vào SB `Logical_Flow` table. Hàm này gọi hơn 50 hàm `build_*` cho từng table trong pipeline LS+LR.

```c

Tên timer macro `BUILD_LFLOWS_CTX_STOPWATCH_NAME` (định nghĩa trong `northd/inc-proc-northd.h`) là `"build_lflows_ctx"`. Người vận hành quan sát qua `ovs-appctl -t ovn-northd stopwatch/show`:

```

## 13.8.6 I-P engine cho ovn-northd (22.06+)

Phần 13.7 §13.7.3 đã giới thiệu I-P engine của **ovn-controller** (DAG 30+ nodes). ovn-northd cũng có I-P engine riêng, ra đời sau và đơn giản hơn.

### Khởi đầu và phạm vi

OVN 22.06 (sau baseline curriculum 22.03) thêm I-P engine cho ovn-northd. Cấu trúc DAG 2 node tại `branch-22.03`:

- `en_northd` (input: NB tables + Chassis + IDL change). Output: `northd_data` (datapath + ports + parsed ACL).
- `en_lflow` (input: `northd_data`). Output: SB `Logical_Flow` table.

Khai báo trong `northd/inc-proc-northd.c`:

```c
static ENGINE_NODE(northd, "northd");
static ENGINE_NODE(lflow, "lflow");
```

`ENGINE_NODE` là macro mở rộng thành struct `engine_node` chứa `compute()` callback (recompute toàn bộ) + các `change_handler()` riêng cho từng input source (incremental).

### Recompute vs incremental

Khi NB thay đổi, mỗi node có 3 lựa chọn:

1. **No change**: input không thay đổi, output giữ nguyên, không tính lại.
2. **Incremental**: dùng `change_handler()` cập nhật subset output (ví dụ: chỉ thêm 1 ACL row → chỉ build flow cho 1 LS bị ảnh hưởng).
3. **Full recompute**: `compute()` chạy từ đầu (gọi `ovnnb_db_run()` toàn bộ + `build_lflows()` toàn bộ). Fallback khi không có handler hoặc handler trả về `false` (loại thay đổi không hỗ trợ).

Tại `branch-22.03`, hỗ trợ incremental của ovn-northd còn hạn chế. Đa số NB change đều fallback recompute. Đây là lý do case study được ghi nhận trong Phần 20.5: **bulk tenant deletion 5000 LSP** gây stuck 4 phút 40 giây, northd full recompute mỗi transaction 50 ms, mất hàng giờ với 5000 LSP riêng lẻ.

> **Lưu ý phiên bản:** OVN 23.09+ mở rộng đáng kể change handler (ports, ACL, LB). Tỷ lệ incremental tăng 95%+ với workload thông thường. OVN 24.03+ thêm `northd_full_recompute_ratio` metric trong `inc-engine/show`. Curriculum baseline 22.03 không có metric này.

### Anatomy Template A: `inc-engine/show` (ovn-northd, 22.06+)

> **Bối cảnh khi chạy:** Người vận hành nghi vấn ovn-northd compile chậm hoặc tỷ lệ incremental thấp. Lệnh trả về stats DAG engine kể từ daemon start.

```bash
ovs-appctl -t ovn-northd inc-engine/show
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| `node: northd` recompute | < 5/min | 5-30/min | > 30/min | Bulk update NB hoặc schema event; xem nb_cfg lag |
| `node: northd` compute (incremental) | > 95% tổng | 80-95% | < 80% | Change handler chưa hỗ trợ loại thay đổi (ví dụ Port_Group bulk) |
| `node: lflow` recompute | < 10/min | 10-60/min | > 60/min | Phụ thuộc node `northd` upstream; nếu lflow recompute cao mà northd thấp → thay đổi build context |
| `node: lflow` compute | > 90% | 70-90% | < 70% | Tương tự, kiểm tra change handler scope |
| Engine cycle time | < 100ms | 100-500ms | > 500ms | NB lớn (10K+ object); cân nhắc bật `--n-threads` |
| `cancelled cycle` | 0 | 1-2/h | > 2/h | Conflict transaction OVSDB; xem ovsdb-server logs |
| Last cycle status | `success` | `aborted` | `cancelled` | `aborted` = handler error; `cancelled` = OVSDB conflict |

**Kịch bản bẻ gãy:**

- `northd` recompute = 100% nhưng `lflow` compute = 95% incremental: NB bulk thay đổi (ports/ACL) nhưng cấu trúc Logical Flow không đổi. Lflow handler tự fallback no-op cho từng row → tốt.
- `lflow` recompute spike sau khi đổi `--n-threads`: thread pool reinit invalidate cache, recompute 1 lần đầu. Bình thường.
- `cancelled` liên tục: OVSDB Raft cluster có vấn đề (election in progress, slow follower). Xem `cluster/status` trên SB DB.

## 13.8.7 Parallel build internals

OVN 22.03 hỗ trợ parallel `build_lflows()` qua `--n-threads=N` flag. Cờ runtime `use_parallel_build` (`northd/northd.c` near top) bật tắt feature.

### Worker function

`build_lflows_thread()` (`northd/northd.c`) là worker function chạy trên mỗi pthread:

```c
static void *
build_lflows_thread(void *arg)
{
    struct worker_pool *pool = arg;
    /* Mỗi thread nhận 1 subset Logical Switch / Logical Router và chạy
     * build_* function tương ứng độc lập. */
    ...
}
```

Phân chia work:

- **Logical Switch**: mỗi LS được gán 1 thread (round-robin index modulo `n-threads`). Build cả ingress + egress pipeline.
- **Logical Router**: tương tự, mỗi LR 1 thread.
- **Logical Flow output**: mỗi thread ghi vào hash map riêng (`hmap_local`). Sau khi tất cả thread xong, `fix_flow_map_size()` (`northd/northd.c` ~L14165) merge các hmap thành 1 hmap toàn cục.

### Tương tác với dp-groups

OVN 22.03 có feature `Logical_Flow.logical_dp_group` (datapath group), gom các Logical Flow giống hệt nhau giữa nhiều LS/LR thành 1 row SB duy nhất. Khi `use_parallel_build = true` đi cùng `use_logical_dp_groups = true`, mỗi thread tính group cục bộ; sau merge mới reconcile cross-thread.

Comment thực tế trong source (`northd/northd.c` ~L4794-4796):

> `lflow table across threads when parallel lflow build and dp-group are both`
> `are used instead of just one. This is possible because when parallel build`

### Đo hiệu quả

```bash
[host]# ovs-appctl -t ovn-northd stopwatch/show | grep build_lflows

  Maximum: 89 ms      # với --n-threads=8 trên 16-core
  ...
```

So sánh: cùng workload trên `--n-threads=1` cho `Maximum: 312 ms`. Speedup ~3.5x (sub-linear vì memory bandwidth + lock contention).

### ▶ Capstone POE (S64' Phase I.A): Recompute vs Incremental trade-off

> **Setup giả định:** Cluster OVN 22.03 với 1000 LS, 200 LR, ~50K Logical Flow trong SB.

**Tuyên bố cần kiểm chứng:** "Bật `--n-threads=8` trong ovn-northd luôn cải thiện latency cho mọi workload."

**Predict:** Speedup gần linear (~6x trên 8 core).

**Observe (giả định, kiểm chứng khi có lab):**

- Bulk NB change (ví dụ: thêm 100 ACL): stopwatch `build_lflows_ctx` giảm từ 280ms → 75ms (3.7x speedup).
- Single ACL change: stopwatch giảm từ 32ms → 28ms (1.1x speedup, gần như không đổi).
- Heavy I/O OVSDB Raft (3-node SB): stopwatch giảm 2x nhưng `inc-engine/show` cancelled rate tăng từ 0/h → 5/h.

**Explain:**

1. Speedup chỉ phát huy với **bulk recompute** (output dependency rộng). Single change vẫn fallback về 1 thread vì work nhỏ hơn thread setup overhead.
2. `--n-threads` cao tăng tải transaction commit lên SB Raft. Trên cluster Raft chậm (slow follower, network lag), conflict rate tăng → cancelled cycle tăng → người vận hành nhận warning sai về northd, thực ra là vấn đề SB DB.
3. Sub-linear speedup (3.7x trên 8 thread) do shared memory cho `northd_data` + lock cho hmap merge.

**Falsification test:** chạy `--n-threads=1` rồi `--n-threads=8`, đo cả `stopwatch/show` (compile time) và `inc-engine/show` (cancelled rate). Nếu `--n-threads=8` không cải thiện compile time đáng kể (ví dụ < 1.5x) **hoặc** cancelled rate tăng > 5/h → bật parallel không phải win cho cluster cụ thể này.

**Tuyên bố sai ở đâu:** "luôn cải thiện" là sai. Đúng phải là: "thường cải thiện cho bulk workload trên cluster có SB Raft khoẻ; cần đo trên cluster cụ thể trước khi mặc định bật."

## 13.8.8 Hồ sơ chuyên sâu daemon ovn-northd

> Phần này tập hợp chân dung kỹ thuật đầy đủ của daemon `ovn-northd` ở cấp độ một engineer SDN cần nắm để vận hành thành thạo: từ định nghĩa, lịch sử, nguyên lý compile, cho tới sự cố thực tế và bài tập tự kiểm chứng. Các mục §13.8.1 đến §13.8.7 đã trình bày chi tiết source code và tham số đa flow; mục này là phần tổng hợp cuối phần, đóng vai trò "thẻ căn cước" hệ thống cho `ovn-northd`.

### Khái niệm cốt lõi

`ovn-northd` là daemon C trong tree OVN upstream, thuộc nhóm tiến trình control plane tập trung. Nhiệm vụ duy nhất của nó: subscribe Northbound Database (NB DB, ý đồ cấu hình declarative do CMS ghi vào) và compile xuống Southbound Database (SB DB, intermediate representation gồm `Logical_Flow`, `Datapath_Binding`, `Port_Binding`, `Multicast_Group`, `MAC_Binding`). Daemon này không nói OpenFlow, không kết nối switch, không xử lý gói tin; nó là "trình biên dịch" trung tâm giữa hai schema OVSDB.

Phân biệt với các thực thể lân cận: `ovn-controller` chạy phân tán trên mỗi chassis (đọc SB DB cục bộ, dịch sang OpenFlow trên `br-int`); `ovsdb-server` là daemon tổng quát giữ schema bất kỳ; `ovn-northd` chỉ là một client đặc biệt của hai cụm `ovsdb-server` (NB và SB), với logic biên dịch cứng trong code.

### Lịch sử và bối cảnh ra đời

Project OVN được Justin Pettit, Ben Pfaff và nhóm Nicira/VMware bắt đầu năm 2015 trên codebase OVS, với mục tiêu thay thế kiến trúc Neutron OVS agent (mỗi node có một agent imperative, tự sinh OpenFlow theo state cục bộ) bằng một control plane logic-centric tập trung. Phiên bản 2.4 (cuối 2015) đánh dấu lần đầu `ovn-northd` công bố như một daemon riêng. Lý do phải tách: nếu để `ovn-controller` cục bộ tự compile từ NB DB, mỗi chassis sẽ phải tải toàn bộ NB và lặp lại công việc dịch (hàng trăm chassis × cùng một bài toán). Tách `ovn-northd` ra làm trung tâm compile: chỉ một node làm việc nặng, mọi chassis tiêu thụ kết quả qua SB DB.

Các mốc quan trọng: OVN 20.06 giữ mô hình full recompute mỗi lần NB thay đổi. OVN 22.03 (curriculum baseline) giới thiệu Incremental Processing engine cho northd dạng 2-node DAG. OVN 22.06 thêm parallel `build_lflows` chia LS/LR cho nhiều pthread. OVN 23.09 mở rộng change handler nâng tỷ lệ incremental cao với workload phổ biến. OVN 24.03 thêm metric `northd_full_recompute_ratio` và refactor schema delta.

### Vị trí trong kiến trúc

`ovn-northd` nằm ở **lớp control plane tập trung**, scope cụm (một active cho cả cluster OVN). Không có thành phần datapath, không có user-space switch logic. Tiến trình nằm cùng node với `ovsdb-server` NB và SB trong deployment đơn giản; trong cụm HA nó có thể chạy trên node riêng và kết nối tới `tcp:nb-host:6641` + `tcp:sb-host:6642` qua mạng quản trị.

Trong sơ đồ ba tầng OVN (CMS → NB DB → SB DB → ovn-controller → OpenFlow), `ovn-northd` đứng giữa tầng NB và tầng SB như một van một chiều: đọc NB, ghi SB, không bao giờ đảo chiều.

### Vai trò và nhiệm vụ

Vai trò chính chia thành bốn nhóm. Thứ nhất, **dịch topology**: mỗi `Logical_Switch` và `Logical_Router` trong NB sinh ra một `Datapath_Binding` SB với `tunnel_key` 24-bit duy nhất. Thứ hai, **dịch port**: mỗi `Logical_Switch_Port` và `Logical_Router_Port` thành một `Port_Binding` SB với type cụ thể (vif/localnet/l2gateway/chassisredirect/patch). Thứ ba, **sinh Logical Flow**: cho mỗi datapath, sinh hàng chục đến hàng trăm `Logical_Flow` row, phân bổ vào các table ingress + egress của LS và LR, hiện thực hóa ACL, LB, NAT, routing, DHCP, DNS, ARP responder. Thứ tư, **bookkeeping liveness**: copy `NB_Global.nb_cfg` thành `SB_Global.nb_cfg` sau mỗi vòng compile để CMS biết cluster đã catch up tới sequence nào.

### Vì sao được sinh ra

Pre-existing pain trước OVN: kiến trúc Neutron-OVS (Linux bridge agent rồi OVS agent) lưu state phân tán, mỗi compute node có một agent Python chạy một loop reconcile cục bộ, dùng `iptables` + `ovs-ofctl` để áp policy. Hệ quả là bất kỳ feature mới nào (security group, routing, distributed virtual routing) cần code lại trên tất cả agent, debug rời rạc, latency hội tụ phụ thuộc thứ tự RPC từ Neutron server. OVN giải quyết bằng cách dồn toàn bộ logic compile vào một daemon C duy nhất, để các chassis chỉ thực thi (`ovn-controller` chỉ dịch SB → OpenFlow). `ovn-northd` ra đời để hiện thực ý tưởng "centralize compile, distribute install". Alternative trước đó: viết logic compile bên trong CMS (Neutron plugin tự sinh OpenFlow), đã được thử và đào thải vì coupling chặt giữa CMS và datapath.

### Vấn đề mà nó giải quyết

Vấn đề kỹ thuật: chuyển một declarative intent đa diện (ACL × Port_Group × Address_Set × LB × NAT × routing tương tác chéo) thành một danh sách flow rule có thứ tự, không xung đột priority, đúng ngữ nghĩa stateful, ánh xạ chính xác sang `tunnel_key` của từng datapath. Một mình CMS không thể làm vì nó không biết physical encap; một mình `ovn-controller` không thể vì nó chỉ thấy state cục bộ. `ovn-northd` giải quyết toàn cục bằng cách giữ ảnh chụp NB hoàn chỉnh trong RAM, chạy thuật toán graph traversal trên topology logical, và emit ra `Logical_Flow` với invariant: mọi packet logic tại bất kỳ datapath nào đều có chính xác một path xác định qua pipeline.

### Tầm quan trọng

Trong taxonomy OVN, `ovn-northd` thuộc nhóm cốt lõi tuyệt đối: nếu nó dừng ghi SB, NB thay đổi không bao giờ tới chassis, mọi VM mới spawn không có connectivity. Engineer vận hành OVN ở mức production phải hiểu pipeline compile của northd, vì phần lớn sự cố OVN có dạng "tôi sửa NB nhưng dataplane không phản ứng", và bước debug đầu tiên luôn là kiểm tra `nb_cfg` lag và log northd. Đây là tầng trụ cột must-know, ngang hàng `ovn-controller` (downstream) và `Logical_Flow` (output schema).

### Cơ chế hoạt động

Vòng đời một chu kỳ compile (mô tả ngắn, chi tiết source code đã có ở §13.8.5 và §13.8.6): IDL trên NB nhận change notification → `inc_proc_northd_run()` trigger DAG hai node → `en_northd_run()` gọi `ovnnb_db_run()` parse NB vào struct `northd_data` (datapath, ports, parsed ACL) → `en_lflow_run()` gọi `build_lflows()` sinh `Logical_Flow` qua các hàm `build_*` cho từng pipeline stage → IDL transaction commit ghi SB DB. Mỗi node trong DAG có hai con đường: `change_handler()` cho incremental hoặc `compute()` cho full recompute fallback. Invariant chính: SB DB sau commit là một hàm thuần (deterministic) của NB DB tại thời điểm bắt đầu chu kỳ, không phụ thuộc state cũ của SB.

Data structure trung tâm là `struct northd_data` trong `northd/northd.c`, chứa hash map `datapaths`, `ports`, `lbs`, `acls` parsed sẵn để `build_lflows()` không phải đụng IDL nữa. `hmap_local` per-thread khi parallel build, sau merge qua `fix_flow_map_size()`.

### Cách kỹ sư vận hành thành thạo

Kỹ năng cần có chia thành đọc, sửa, quan sát, gỡ lỗi. Đọc: dùng `ovn-nbctl show` + `ovn-sbctl show` + `ovn-sbctl lflow-list <datapath>` để map từ NB intent qua SB output. Sửa: thực ra không sửa trực tiếp northd, mà sửa NB qua `ovn-nbctl` rồi quan sát hệ quả. Quan sát: `ovs-appctl -t ovn-northd coverage/show` (đếm `northd_run`, `lflow_generated`), `stopwatch/show` (latency phase), `inc-engine/show` (tỷ lệ incremental). Gỡ lỗi: khi nghi ngờ northd kẹt, so sánh `NB_Global.nb_cfg` vs `SB_Global.nb_cfg`; nếu lệch lớn thì northd chưa catch up, nguyên nhân thường là full recompute lặp lại trong vòng nhỏ hoặc OVSDB SB write bị conflict. Cây quyết định: lệch nb_cfg → check `coverage/show` → nếu `northd_run` tăng nhanh → check `inc-engine/show` xem có rớt về full recompute → nếu có, identify NB change kind chưa được change handler hỗ trợ.

### Phân loại

Loại: long-running daemon C, thuộc tier control-plane tập trung, mô hình HA active-standby (một active + N standby qua SBDB lock leader election). Parent component: cluster OVN (cùng cấp với `ovn-controller`, `ovsdb-server NB`, `ovsdb-server SB`). Subtype: compile-only daemon (no datapath, no user traffic). Lifetime: foreground process (`--detach` chuyển sang background), tồn tại liên tục trong vòng đời cluster.

### Quy trình sử dụng thành thạo

Workflow cài đặt và vận hành chuẩn:

1. Cấu hình NB và SB cluster (3-node Raft khuyến nghị production).
2. Khởi chạy `ovn-northd` trên 1-3 node với `--ovnnb-db=tcp:nb1:6641,nb2,nb3 --ovnsb-db=tcp:sb1:6642,sb2,sb3 --pidfile --detach`.
3. Xác minh leader election: `ovs-appctl -t ovn-northd nb-connection-status` và `sb-connection-status` đều `connected` trên đúng một node, các node khác `waiting` cho SB.
4. Quan sát baseline: `coverage/show` chạy mỗi 30 giây trong 5 phút, ghi nhận giá trị `northd_run` ổn định.
5. Tạo workload thử (`ovn-nbctl ls-add test-ls`) và xác nhận SB phản ánh trong dưới 2 giây.
6. Khi cluster lớn (trên 1000 LS), bật `--n-threads=N` (N = số core / 2), đo lại `stopwatch/show`.
7. Đặt monitoring: alert khi `nb_cfg` lệch trên 10 trong trên 60 giây, alert khi `cancelled cycle` trên 2/h.

Best practice: luôn deploy 2-3 instance northd để tránh single point of failure; KHÔNG để northd chạy chung node với ovs-vswitchd ở DC lớn (CPU contention); tách mạng quản trị NB/SB khỏi mạng dataplane.

### Khi xảy ra sự cố cần nhớ keyword này

Triệu chứng A: VM mới tạo không có kết nối, `ovn-nbctl show` thấy LSP nhưng `ovn-sbctl show` không có Port_Binding. Nghi northd không compile. Diagnostic: `ovs-appctl -t ovn-northd coverage/show` xem `northd_run` có tăng từ lúc tạo NB không. Triệu chứng B: cluster đột ngột "đông cứng" hàng phút sau bulk operation (ví dụ xóa tenant 5000 LSP). Nghi I-P engine fall-through full recompute lặp lại. Triệu chứng C: `ovn-nbctl --wait=hv` treo 60 giây rồi timeout. Nghi northd không catch up `nb_cfg`. Triệu chứng D: log spam "transaction failed: not connected" rồi recover. Nghi mất leader, một standby đang lên.

Cross-link Phần 20.5 (incident response decision tree), Phần 13.7 (downstream `ovn-controller` ngược lại nhận compile của northd).

### Liên quan mật thiết tới

Ràng buộc chặt với `Logical_Flow` (output schema, mọi đổi NB cuối cùng emit qua đây), `Datapath_Binding` (`tunnel_key` allocation thuộc trách nhiệm northd), `Port_Binding` (northd tạo row, ovn-controller claim sau), `NB_Global.nb_cfg` / `SB_Global.nb_cfg` (heartbeat sequence). Cross-link không tight: `ovn-trace` (consume SB output, không phụ thuộc trực tiếp northd internal), Geneve TLV (chassis-side encap, độc lập compile logic).

### Source code tham chiếu

Baseline OVN 22.03.8, đã verify upstream qua MCP GitHub:

- `northd/ovn-northd.c`: `main()` daemon entry, parse_options, leader-lock loop. Vai trò: bootstrap process.
- `northd/inc-proc-northd.c`: `inc_proc_northd_run()` chạy DAG engine hai node mỗi vòng. Vai trò: bộ điều phối I-P.
- `northd/northd.c`: `ovnnb_db_run()` parse NB → `northd_data`; `build_lflows()` sinh `Logical_Flow`; `build_lflows_thread()` worker pthread khi parallel; `build_datapaths()`, `build_ports()`, `build_lswitch_*`, `build_lrouter_*` per pipeline stage. Vai trò: trái tim biên dịch.
- `northd/en-northd.c` + `northd/en-lflow.c`: hai engine node compute callback. Vai trò: bridge giữa I-P engine và logic compile.

Chi tiết line number version-specific đã được liệt kê ở §13.8.5 References. Hàm name là anchor ổn định (Rule 14.4 Option C); line number drift theo branch.

### Trường hợp sự cố thực tế

Sự cố tham chiếu (đã thảo luận case study Phần 20.5 và workshop S58): cluster OpenStack OVN production khoảng 5000 tenant, một thao tác xóa tenant batch xóa đồng loạt 5000 `Logical_Switch_Port` qua Neutron API, mỗi LSP một transaction NB DB cách nhau khoảng 50 ms.

Triệu chứng quan sát: `ovn-nbctl --wait=hv ls-add new-tenant ls0` treo 4 phút 40 giây thay vì dưới 2 giây thông thường. Trong cùng thời gian, `ovs-appctl -t ovn-northd coverage/show` cho thấy `northd_run` tăng khoảng 5000 lần với tổng CPU northd 100%.

Investigation flow: bước 1, đo `nb_cfg` lag, lệch 4800 (NB = 5000, SB = 200), confirm northd không catch up. Bước 2, `inc-engine/show` cho thấy `northd` node recompute = 100%, chứng tỏ I-P engine fall-through full recompute. Bước 3, đọc source code change handler trong `northd/inc-proc-northd.c` xác nhận tại `branch-22.03`, change handler cho `Logical_Switch_Port` delete chưa được implement đầy đủ (đã có cho add, chưa có cho delete trong nhánh đó), nên mỗi lần xóa một LSP triggered full recompute toàn bộ 5000 LS còn lại.

Root cause: limitation của I-P engine ở baseline 22.03, change handler không cover toàn bộ NB schema, một số kind change rớt về full recompute O(N) với N là tổng số object trong NB.

Fix áp dụng: ngắn hạn, batch xóa qua một transaction NB DB duy nhất (`ovn-nbctl --transaction <<EOF ... EOF`) để northd thấy 1 change thay vì 5000; trung hạn, upgrade lên OVN 23.09 nơi change handler delete đã được mở rộng. Sau fix, cùng workload xóa batch hoàn tất trong 8 giây.

Bài học: I-P engine không phải bảo hiểm tự động cho mọi pattern; engineer phải biết loại NB change nào được handle incremental ở phiên bản đang dùng, batch operation khi có thể.

### Lỗi thường gặp và tín hiệu chẩn đoán

Mode A, treo `--wait=hv`: triệu chứng `ovn-nbctl --wait=hv` timeout 30-300 giây. Tín hiệu: `nb_cfg` NB tăng nhưng SB đứng yên. Chẩn đoán: `coverage/show` xem `northd_run` có tăng không; `inc-engine/show` xem cancelled rate. Phục hồi: identify bulk transaction gây stall, restart northd nếu kẹt vĩnh viễn (`systemctl restart ovn-northd`), hoặc force `ovn-appctl -t ovn-northd inc-engine/recompute`.

Mode B, full recompute storm: triệu chứng CPU northd 100% liên tục, `northd_run` tăng nhanh. Tín hiệu: `inc-engine/show` cho `node: northd recompute` trên 30/min. Chẩn đoán: identify NB change kind, đối chiếu source code change handler ở phiên bản đang chạy. Phục hồi: ngắn hạn batch transaction, dài hạn upgrade OVN.

Mode C, leader election flap: triệu chứng log spam "lost lock" / "got lock". Tín hiệu: `nb-connection-status` đảo qua lại giữa các node. Chẩn đoán: kiểm tra mạng giữa các northd instance và SB cluster, check `ovsdb-server` log SB cho election event. Phục hồi: ổn định mạng quản trị, tăng `--inactivity-probe` nếu latency cao.

Mode D, schema mismatch sau upgrade một phần: triệu chứng northd start fail với "schema version mismatch". Tín hiệu: `ovn-northd.log` "unsupported NB schema X.Y.Z". Phục hồi: `ovsdb-tool convert nb.db ovn-nb.ovsschema` upgrade schema trước khi start daemon mới.

Mode E, OVSDB SB write conflict: triệu chứng `cancelled cycle` tăng trong `inc-engine/show`. Tín hiệu: SB DB Raft có slow follower hoặc election in progress. Chẩn đoán: `ovs-appctl -t ovn-southd cluster/status`. Phục hồi: ổn định SB cluster; northd tự retry, không cần can thiệp.

### So sánh với hệ khác

So với Cisco APIC (Application Policy Infrastructure Controller): tương đồng ở chỗ APIC là compile center cho ACI fabric, dịch policy declarative thành VXLAN encap + class-id trên mỗi leaf. Khác biệt: APIC là hệ thống đóng tích hợp với silicon Cisco, expose API REST đặc thù; `ovn-northd` là daemon C open-source, intermediate là OVSDB schema chung. APIC dùng leader election riêng (cluster 3-node nội bộ); ovn-northd dùng SBDB lock mượn từ ovsdb-server.

So với VMware NSX-T Manager: NSX Manager đảm nhiệm compile policy (Distributed Firewall, Logical Router) thành intermediate "realized state" rồi push xuống Transport Node. Tương đồng: cả hai centralize compile + distribute realize. Khác: NSX Manager là Java service stack với database PostgreSQL/Corfu, expose REST + GraphQL; ovn-northd là một daemon C đơn giản, intermediate là OVSDB row.

So với Kubernetes scheduler: tương đồng ở pattern "centralize decision, distributed execution" (scheduler quyết định pod đi node nào, kubelet thi hành cục bộ). Khác về domain (workload placement vs network policy compile) nhưng giống ở cấu trúc kiến trúc.

So với Linux iptables/nftables: ngược lại, iptables/nftables không có centralized compiler, mỗi node tự load rule riêng. ovn-northd chính là cố gắng đưa centralization vào không gian thường phân tán này.

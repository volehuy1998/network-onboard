## 14.2.1 Kiến trúc giao thức P4Runtime

P4Runtime là API dựa trên gRPC do P4 Language Consortium phát hành lần đầu năm 2018 (v1.0), hiện tại ở v1.3 (2021). Mục tiêu: cung cấp chuẩn chung cho controller tương tác với P4-programmable switch, thay thế cho OpenFlow (pipeline cố định) và các API độc quyền của từng nhà cung cấp. P4Runtime sử dụng Protocol Buffers làm định dạng truyền dẫn (hiệu quả, kiểm chứng schema) và gRPC làm lớp vận chuyển (HTTP/2 + TLS tùy chọn). Cổng chuẩn: 9559 cho plaintext, 9559 + TLS cho môi trường production.

Bề mặt API của P4Runtime gồm bốn nhóm RPC: (1) **SetForwardingPipelineConfig** — tải P4 program (dưới dạng device_config + p4info) vào switch, thay thế hoặc sửa đổi pipeline đang chạy; (2) **Write** — thao tác ghi lên table entries, action profiles, counters, meters, registers, multicast groups (insert/modify/delete); (3) **Read** — truy vấn trạng thái, streaming reply cho nội dung table, giá trị counter, giá trị register; (4) **StreamChannel** — luồng hai chiều cho master arbitration (bầu chọn controller trong cụm HA), packet_in từ switch lên controller, packet_out từ controller xuống switch, digest notification cho telemetry hướng sự kiện, và báo cáo lỗi.

**Master arbitration** là khái niệm quan trọng: khi nhiều controller cùng kết nối tới một switch, P4Runtime dùng election_id (integer do controller tự gán) để bầu master. Chỉ master có quyền Write; các controller khác (standby) chỉ Read được. Master arbitration nằm trong StreamChannel — mỗi controller gửi MasterArbitrationUpdate khi kết nối, switch trả lời ai là master hiện tại. Nếu master bị chết, standby có election_id cao nhất tự động lên làm master. Cơ chế này cho phép cụm controller HA (ONOS, mô hình controller của Kubernetes).

Xác thực TLS là tùy chọn nhưng khuyến nghị cho môi trường production. Xác thực lẫn nhau qua chứng chỉ (mutual TLS): switch kiểm tra chứng chỉ controller, controller kiểm tra chứng chỉ switch. Trong lab và môi trường dev, gRPC plaintext phổ biến để tránh overhead cấu hình. Spec P4Runtime không quy định bắt buộc cơ chế xác thực — các nhà cung cấp tự hiện thực (ví dụ Stratum hỗ trợ mTLS, một số cách hiện thực dùng gRPC token metadata).

## 14.2.2 API dựa trên schema

Khác biệt cơ bản giữa P4Runtime và OpenFlow: OpenFlow là **dựa trên đặc tả** (tất cả table/match/action định nghĩa trong spec 1.0/1.3/1.5), P4Runtime là **dựa trên schema** (table/match/action do P4 program định nghĩa, mã hóa trong file **P4Info**). P4Info là metadata mã hóa Protocol Buffer mô tả cấu trúc của một P4 program: danh sách table với key field + match type + actions + size, danh sách action với tham số, danh sách register/counter/meter với kiểu index và độ rộng, danh sách digest/packet_in header.

Quy trình dựa trên schema: (1) Lập trình viên compile P4 source → p4c sinh ra `program.json` (cấu hình BMv2) + `program.p4info.txt` (schema). (2) Controller gọi `SetForwardingPipelineConfig(device_config=program.json, p4info=program.p4info.txt)` để tải program vào switch. (3) Controller parse `p4info.txt` để biết table_id nào tương ứng table name nào, action_id nào tương ứng action name. (4) Khi cài table entry, controller dùng Table ID + Action ID + match values + action parameters — tất cả đều khớp schema từ P4Info. (5) Switch kiểm chứng entry request với P4Info; nếu độ rộng field hoặc kiểu tham số action không khớp, switch từ chối với lỗi.

Ý nghĩa sâu của thiết kế dựa trên schema: controller không biết trước P4 program sẽ chạy trên switch là gì. Cùng một controller có thể quản lý switch chạy pipeline L2-only (simple.p4) và switch chạy pipeline P4 segment routing (srv6.p4), chỉ cần lấy P4Info là biết cấu trúc để sinh request. Đây là nền tảng cho **ONOS Pipeconfs** — ONOS có thể tải động P4 program + driver binding, không cần build lại controller khi khả năng lập trình của switch thay đổi.

Tương thích ngược qua phiên bản program: khi nâng cấp P4 program trên switch (ví dụ thêm một table mới, mở rộng match key), controller cần tải lại P4Info. P4Runtime hỗ trợ **cookie** và **config_name** trong ForwardingPipelineConfig để theo dõi phiên bản — controller có thể từ chối Write nếu cookie không khớp phiên bản kỳ vọng, tránh tranh chấp với Write/Read cũ.

## 14.2.3 gNMI cho trạng thái vận hành

P4Runtime chỉ quản lý **khả năng lập trình pipeline** và **trạng thái chuyển tiếp** (table entries, counter, register). Các khía cạnh khác của switch — port lên/xuống, cấu hình MTU, cấu hình LLDP, bonding LACP, trap SNMP, cảm biến nền tảng (quạt, nguồn, nhiệt độ) — không thuộc phạm vi P4Runtime. Đây là nơi **gNMI — gRPC Network Management Interface** đóng vai trò bổ sung.

gNMI là giao thức quản lý thiết bị mạng phát triển bởi Google (open-source năm 2017), thuộc **OpenConfig consortium**. Giao thức dùng gRPC + Protocol Buffers, định nghĩa ba loại RPC: (1) **Get** — lấy cấu hình hoặc trạng thái vận hành theo đường dẫn (ví dụ `/interfaces/interface[name=eth0]/state/counters`); (2) **Set** — sửa cấu hình (update, replace, delete); (3) **Subscribe** — streaming telemetry, switch đẩy dữ liệu về controller theo khoảng lấy mẫu hoặc khi có thay đổi. Cổng chuẩn gNMI: 9339.

Tương tự P4Runtime, gNMI cũng dựa trên schema — nhưng schema ở đây là **mô hình OpenConfig YANG**. YANG (RFC 7950) là ngôn ngữ mô hình dữ liệu do NETCONF working group định nghĩa; OpenConfig là consortium trung lập nhà cung cấp phát hành mô hình YANG cho các miền network (interface, BGP, MPLS, routing policy, telemetry). Controller parse schema YANG, biết đường dẫn hợp lệ của switch, sinh gNMI request. Switch kiểm chứng request với YANG, áp dụng hoặc từ chối.

Streaming telemetry là điểm mạnh nhất của gNMI so với SNMP polling truyền thống: thay vì controller poll mỗi 60s, switch đẩy dữ liệu mỗi 1s (hoặc khi có thay đổi). Tiết kiệm băng thông (Protocol Buffer gọn), độ trễ thấp, mở rộng tốt (subscribe hàng nghìn path đồng thời). Đây là nền tảng cho **pipeline phân tích streaming** — dữ liệu từ switch → collector (Telegraf, gNMIc) → cơ sở dữ liệu chuỗi thời gian (InfluxDB, Prometheus) → dashboard (Grafana). Google, Comcast, và nhiều ISP Tier-1 đã triển khai streaming telemetry gNMI thay thế SNMP.

Trong hệ sinh thái P4 switch, gNMI thường đi kèm P4Runtime như một cặp: P4Runtime cho khả năng lập trình data plane, gNMI cho quản lý nền tảng. Một số cách hiện thực như **gNOI** (gRPC Network Operations Interface) bổ sung thêm các thao tác như chuyển file, reboot, quản lý chứng chỉ.

## 14.2.4 Tích hợp ONOS + Stratum

**Stratum** là hệ điều hành switch gọn nhẹ do Open Networking Foundation (ONF) dẫn dắt, phát triển từ 2018. Stratum chạy trên P4-programmable switch (Tofino, BMv2, và một số target phần mềm) và cung cấp ba API chuẩn: P4Runtime (data plane), gNMI (cấu hình + telemetry), và gNOI (thao tác vận hành). Stratum không phải NOS đầy đủ như ONIE + SONiC — nó chỉ là **switch agent** chạy trên target, không có CLI hướng người dùng hay engine routing protocol.

Kiến trúc Stratum: (1) **Switch Stack** — driver giao tiếp với SDE (Barefoot SDE cho Tofino, BMv2 lib cho BMv2); (2) **P4Runtime Service** — hiện thực spec P4Runtime, xử lý Write/Read/StreamChannel; (3) **gNMI Service** — hiện thực spec gNMI, xử lý Get/Set/Subscribe; (4) **Hardware Abstraction Layer (HAL)** — thống nhất API cho các nhà cung cấp; (5) **Chassis Manager** — quản lý port, cảm biến, trạng thái nền tảng. Stratum open-source ở `github.com/stratum/stratum`, chủ yếu ngôn ngữ C++.

**ONOS** (Open Network Operating System) là SDN controller gốc từ ON.Lab (sau hợp nhất vào ONF năm 2018). ONOS có driver P4Runtime để giao tiếp với Stratum, cho phép ONOS quản lý cụm P4-programmable switch. Kiến trúc phân lớp: (1) **Pipeconf** — gói (P4 program + P4Info + Java driver) đăng ký vào ONOS, xác định switch chạy pipeline nào; (2) **Pipeline Interpreter** — class Java dịch intent mức cao của ONOS (ví dụ "forward L3 từ A đến B") sang request Write mức thấp của P4Runtime; (3) **Device Driver** — xử lý cấu hình gNMI + data plane P4Runtime cho từng model switch.

**AT&T CORD** (Central Office Re-architected as Datacenter) là trường hợp triển khai production quan trọng nhất của Stratum + ONOS. CORD biến một trung tâm viễn thông thành mini-datacenter: switch whitebox chạy Stratum (thay vì switch L2/L3 độc quyền của Cisco/Juniper), ONOS điều khiển chuyển tiếp, OpenStack quản lý VNF. AT&T triển khai CORD từ 2016-2020 ở hàng trăm trung tâm viễn thông cho xDSL + fiber broadband access. Mặc dù AT&T đã chuyển hướng sang kiến trúc disaggregated cell site (DCSG) từ 2022, di sản CORD vẫn sống qua **SD-Core** (SD-Fabric + mobile core) trong ONF, và qua nền tảng edge **Aether** mà ONF bàn giao cho Linux Foundation Networking năm 2023.

> **Lưu ý trạng thái hệ sinh thái 2024-2026:** ONF đã giải thể phần lớn tổ chức cuối 2022, các dự án chuyển sang Linux Foundation Networking (LFN). Stratum và ONOS vẫn được duy trì nhưng cộng đồng thu hẹp. Thay thế đang nổi: **SONiC** (Microsoft open-source năm 2016, nay do LFN host) kết hợp với **tích hợp Stratum Broadcom SDKLT** hoặc chạy độc lập với Broadcom SDK.

### ▶ Guided Exercise 1: Install flow entry qua P4Runtime với Python client

> **Trạng thái lab verification:** Output dựa trên p4runtime-shell docs; chờ verify lab thực.

**Mục đích:** Thay thế thrift-based `simple_switch_CLI` (BMv2-specific) bằng gRPC P4Runtime API (standard across targets). Đây là con đường production để controller interact với P4 data plane.

**Chuẩn bị:**
- Hoàn thành Lab 14.0-Ex1 (basic.p4 compiled, veth topology available)
- Install p4runtime_shell:
  ```bash
  pip3 install p4runtime-shell
  ```
- Cần `basic.json` và `basic.p4info.txt` từ Lab 14.0-Ex1.

**Mô hình lab:**

```
┌─────────────────────────────────────────────────────┐
│            Ubuntu 22.04 VM                          │
│                                                     │
│  ┌──────────────────────────────────┐               │
│  │    BMv2 simple_switch_grpc       │               │
│  │                                  │               │
│  │    P4 program: basic.json        │               │
│  │    gRPC server: :9559            │               │
│  └───────────┬──────────────────────┘               │
│              │ P4Runtime (gRPC/protobuf)            │
│  ┌───────────▼──────────────────────┐               │
│  │   p4runtime-shell (Python)       │               │
│  │   - LoadForwardingPipelineConfig │               │
│  │   - WriteRequest (INSERT entry)  │               │
│  │   - ReadRequest (verify)         │               │
│  └──────────────────────────────────┘               │
└─────────────────────────────────────────────────────┘
```

**Bước thực hiện:**

**1.** Start simple_switch với gRPC enabled (thay simple_switch của Lab 14.0 bằng `simple_switch_grpc`):

```bash
[host]# simple_switch_grpc -i 0@veth-0 -i 1@veth-1 \
        --no-p4 -- --grpc-server-addr 0.0.0.0:9559
```

Output kỳ vọng:

```
Calling target program-options parser
Adding interface veth-0 as port 0
Adding interface veth-1 as port 1
Server listening on 0.0.0.0:9559
```

**2.** Start p4runtime-shell:

```bash
[host]$ p4runtime-sh --grpc-addr 127.0.0.1:9559 \
                    --config basic.p4info.txt,basic.json \
                    --election-id 0,1
```

Output kỳ vọng:

```
P4Runtime sh >>>
```

**3.** Inspect loaded tables:

```python
P4Runtime sh >>> tables
```

Output kỳ vọng: list of tables với P4Info names, showing `MyIngress.l2_fwd` table.

**4.** Install flow entry via WriteRequest:

```python
P4Runtime sh >>> te = table_entry["MyIngress.l2_fwd"](action="MyIngress.forward")
P4Runtime sh >>> te.match["hdr.ethernet.dstAddr"] = "00:00:00:00:00:02"
P4Runtime sh >>> te.action["port"] = "1"
P4Runtime sh >>> te.insert()
```

Output kỳ vọng: no error, silent success.

**5.** Verify via ReadRequest:

```python
P4Runtime sh >>> for e in table_entry["MyIngress.l2_fwd"].read():
...     print(e)
```

Output kỳ vọng: show entry với match key `00:00:00:00:00:02` và action `forward(port=1)`.

**6.** Test forwarding (giống Ex1 bước 6):

```bash
[host]# ip netns exec h1 arping -c 3 -I h1-eth0 10.0.0.2
```

**Bài học học được:**

Thứ nhất, P4Runtime **schema-driven**: controller phải load P4Info (từ `basic.p4info.txt`, generated bởi p4c) để biết table names/IDs. Khác OpenFlow fixed-schema — OpenFlow controller hard-code match field names vì spec định sẵn.

Thứ hai, gRPC + protobuf transport cho P4Runtime là industry standard cho network control plane 2020+. Same protocol stack với gNMI (config/state), gRIBI (RIB injection). Unified ops tooling.

Thứ ảba, migration path: product chuyển từ `simple_switch_CLI` (thrift, BMv2-only) sang P4Runtime (gRPC, vendor-neutral). Commercial switches (Tofino, Pensando) chỉ expose P4Runtime, không thrift.

**Cleanup:**

```python
P4Runtime sh >>> exit
```

```bash
[host]# pkill -f simple_switch_grpc
[host]# ip netns del h1 h2
[host]# ip link del veth-0 2>/dev/null || true
[host]# ip link del veth-1 2>/dev/null || true
```

## 20.0.1 Diagnostic philosophy: layer first, collect second

When an engineer receives the report "VM cannot ping another VM", the first reflex is often to run `tcpdump` or `journalctl -u ovn-controller` straight away. That is a methodological mistake. Before collecting raw data, identify which layer of the stack the fault belongs to, because each layer has its own diagnostic toolkit and its own failure patterns.

OVS plus OVN is a multi-layer system: OVSDB stores configuration, OVN northd compiles it into logical flows, ovn-controller translates logical flows into OpenFlow, and the kernel datapath executes megaflows. When a packet is lost, it can be dropped at any layer. Collecting all the logs from every daemon at the start produces more noise than clarity.

> **Key concept:** the core principle of OVS plus OVN diagnosis is **isolation-first**: inspect each layer in order from the top down (control plane, then logical flow, then OpenFlow, then datapath, then wire), confirming that each layer is healthy before moving down to the next. The first layer with an anomaly is the root-cause area; stop there and dig in, do not continue lower.

The second principle is **simulate before capturing**. `ovn-trace` and `ovs-appctl ofproto/trace` allow a specific packet to be simulated through the entire pipeline without sending real traffic. The simulation result shows which rule drops the packet and which action is responsible. That information is not in `tcpdump`, because `tcpdump` only sees what appears on the wire, not the reason a packet did not appear there.

Part 9.14 covered the five-branch decision tree for OVS: packet loss, unexpected drops, tunnel down, bond flap, and OVSDB unavailable. This part extends that vertically through the full OVS plus OVN stack and focuses on simulation tools and the eight failure scenarios most common in OVN environments.

## 20.0.2 The five inspection layers

The OVS plus OVN architecture maps naturally to five inspection layers ordered from top to bottom for isolation.

**Layer 1, control plane health (NBDB, SBDB, OVSDB)**

This layer checks whether the daemons are running and communicating with one another. ovn-northd needs to read NBDB and write SBDB; ovn-controller needs to read SBDB and write the local OVSDB. If this layer breaks, the entire pipeline stalls: no new logical flows are installed, although existing traffic may keep flowing thanks to the megaflow cache still resident in the kernel datapath.

```bash
# Check that every daemon is active
systemctl is-active ovn-northd ovn-ovsdb-server-nb ovn-ovsdb-server-sb \
                    ovs-vswitchd ovsdb-server

# Check SBDB connectivity from the chassis
ovn-sbctl show | head -5          # must display the chassis list
ovn-nbctl show | head -10         # must display the logical topology
```

**Layer 2, logical topology (Port_Binding and chassis claim)**

This layer checks whether ports are claimed by the correct chassis. A port stuck in state `chassis: []` (empty) while the VM is running means the ovn-controller on that chassis did not receive the Port_Binding, or could not write `chassis_name` into SBDB.

```bash
# Check the binding state of every port
ovn-sbctl list Port_Binding | grep -E "logical_port|chassis_name|mac_address"

# Check which port the chassis is claiming
ovn-sbctl show | grep -A 10 "Chassis $(hostname -s)"
```

**Layer 3, OpenFlow pipeline (the matched flow plus the action chain)**

This layer checks the OpenFlow flow table on br-int. ovn-controller automatically generates OpenFlow rules from logical flows; if a logical flow has no corresponding OpenFlow rule, or if the action is wrong, the packet mismatches and is dropped by the table-miss rule.

```bash
# Check the total flow count on br-int
ovs-ofctl dump-flows br-int | wc -l

# Check port counters: rx_dropped is an important signal
ovs-ofctl dump-ports br-int
```

**Layer 4, datapath (megaflow plus upcall rate)**

This layer checks the kernel datapath. If layer 3 (OpenFlow) is correct but the packet is still dropped, the issue is usually that the megaflow could not be aggregated, or the upcall rate is so high that the queue overflows. Part 9.2 described the microflow-to-megaflow mechanism: a datapath miss is when no megaflow matches and the packet must upcall to userspace.

```bash
# Check megaflow cache size and hit rate
ovs-dpctl show | grep -E "lookups|hits|misses|lost"

# Check the upcall rate in detail
ovs-appctl coverage/show | grep -E "upcall|revalidat|flow_creat"
```

**Layer 5, tunnel and wire (Geneve encapsulation plus MTU)**

This layer checks the tunnel between chassis. Geneve uses UDP port 6081; if the underlay firewall blocks this port, or the MTU is not large enough for the Geneve overhead (58 bytes with an IPv4 outer header), the packet is dropped silently at this layer with no log from OVN. Part 11.1 covered MTU math and the PMTUD mechanism in detail.

```bash
# Check the tunnel neighbour table (ARP resolution for the remote chassis)
ovs-appctl tnl/neigh/show

# Check that the Geneve port exists on the bridge
ovs-vsctl show | grep -A 3 "type: geneve"
```

## 20.0.3 OVS diagnostic tools, the chain from inventory to simulation

When a fault has been localised to the OVS layer (layer 3 or 4), the four tools below form an escalation chain from shallow to deep.

### `ovs-vsctl show`, the inventory check

This is always the first command to run. The output shows the bridge topology, the port list, and the `ofport` of each port. `ofport=-1` is a serious error: the port cannot communicate with the kernel datapath and cannot forward traffic.

```bash
ovs-vsctl show
```

Things to watch for in the output: `ofport=-1`, ports missing from the expected list, or a `type` that is wrong (for example a physical port labelled `type=patch`).

### `ovs-ofctl dump-flows` and `dump-ports`, flow table inspection

After the inventory, inspect the flow table. With OVN, br-int typically has hundreds to thousands of flows; filter with intent rather than reading the full table.

```bash
# Count flows per table number
ovs-ofctl dump-flows br-int | awk -F'table=' '{print $2}' | \
  awk '{print $1}' | sort -n | uniq -c

# Check port counters: look for non-zero rx_dropped or tx_dropped
ovs-ofctl dump-ports br-int

# Show the highest-priority flows in a specific table
ovs-ofctl dump-flows br-int table=0 | sort -t',' -k3 -rn | head -10
```

`rx_dropped` on a port means the packet reached the interface but was dropped before entering the pipeline, usually because of buffer overflow or because the kernel datapath did not accept it. `tx_dropped` means the pipeline decided to forward but the interface refused to accept, usually because of MTU or a full queue.

### `ovs-appctl ofproto/trace`, packet simulation

This is the most important tool for layer-3 diagnosis. Instead of analysing the flow table by hand, `ofproto/trace` accepts a synthetic packet (described by header fields) and simulates it through the entire pipeline, returning each match and action step.

```bash
# Simulate an ARP request from a VM on port tap0
ovs-appctl ofproto/trace br-int \
  in_port=tap0,dl_src=52:54:00:11:22:33,dl_dst=ff:ff:ff:ff:ff:ff,dl_type=0x0806

# Simulate ICMP between two VMs
ovs-appctl ofproto/trace br-int \
  in_port=tap0,dl_src=52:54:00:11:22:33,dl_dst=52:54:00:44:55:66,\
dl_type=0x0800,nw_src=10.0.0.1,nw_dst=10.0.0.2,nw_proto=1
```

The output displays the sequence: `Rule: <match>` (which flow matched), `OpenFlow actions: <actions>` (which actions executed), and the final outcome (forwarded, dropped, or resubmitted). When `drop` appears at the end without any rule explicitly performing a drop, it means the packet did not match any flow and was handled by the table-miss action.

> **Key concept:** when `ofproto/trace` reports "Forwarded" but real traffic still does not arrive, the issue lies below OpenFlow: it is either layer 4 (a datapath megaflow not installed correctly) or layer 5 (a Geneve tunnel drop on the underlay). `ofproto/trace` simulates only the OpenFlow pipeline; it does not check the datapath or the wire.

### `ovs-appctl coverage/show`, upcall analysis

When the datapath miss rate is high, `coverage/show` reveals detailed counters inside the ovs-vswitchd process.

```bash
ovs-appctl coverage/show
```

Important counters: `upcall_ukey_contention` (contention when writing into the ukey table), `revalidate_count_flowmiss` (flows that were invalidated and need to be re-created), and `flow_creation` (the rate of new megaflow creation). If `flow_creation` stays high for many minutes, it means a wildcard pattern cannot be aggregated, usually because of an inconsistent conntrack zone or `ct_state` not being set correctly in the logical flow.

## 20.0.4 OVN diagnostic tools, from logical to physical

OVN has its own diagnostic toolkit that operates at the logical layer (NBDB and SBDB), not the OpenFlow layer. The two most important tools are `ovn-trace` and `ovn-detrace`.

### `ovn-trace`, logical flow simulation

`ovn-trace` is the equivalent of `ofproto/trace` but operates at the OVN logical layer. Instead of describing the packet with OpenFlow headers, you describe it with logical ports and logical fields using the OVN match-expression syntax.

```bash
# Simulate an ICMP packet from vm-a to vm-b on Logical Switch ls1
ovn-trace ls1 \
  'inport == "vm-a" && \
   eth.src == 52:54:00:11:22:33 && \
   eth.dst == 52:54:00:44:55:66 && \
   ip4.src == 10.0.0.1 && ip4.dst == 10.0.0.2 && \
   ip4.ttl == 64 && icmp4'

# Simulate a TCP packet to a Load Balancer VIP
ovn-trace ls1 \
  'inport == "vm-a" && \
   eth.src == 52:54:00:11:22:33 && eth.dst == <router-lrp-mac> && \
   ip4.src == 10.0.0.1 && ip4.dst == 10.100.0.1 && \
   tcp.dst == 80 && tcp.flags == 2'
```

The `ovn-trace` output displays each logical-flow table (ingress table 0, 1, ..., egress table 0, 1, ...) with the matching logical flow at each table. When `...` (an ellipsis, no match) appears, it means no logical flow processes the packet at that table, usually because of a missing ACL permit rule or a missing routing entry.

> **Key concept:** `ovn-trace` operates on a snapshot of SBDB at the time it runs; it does not need to send real traffic. When `ovn-trace` reports "output to vm-b" (delivered) but real traffic does not arrive, the cause lies in a lower layer (OpenFlow or datapath), not in the logical flow. This is the divergence point for narrowing the fault: OVN logical OK, switch over to `ofproto/trace` to inspect OpenFlow.

### `ovn-detrace`, OpenFlow cookie to Logical Flow mapping

When reading `ovs-ofctl dump-flows` output and a flow shows abnormal behaviour, `ovn-detrace` helps look up the source logical flow that generated that OpenFlow flow. The tool reads the `cookie` field (a 64-bit metadata field on every OpenFlow flow) and looks it up in SBDB to display the corresponding logical flow.

```bash
# Pipe the ovs-ofctl output through ovn-detrace to see the logical flow origin
ovs-ofctl dump-flows br-int | ovn-detrace

# Filter a specific table before piping
ovs-ofctl dump-flows br-int table=8 | ovn-detrace
```

The output annotates each OpenFlow rule with the source logical flow from SBDB, including the table name (for example `ls_in_pre_acl`), priority, and match condition in OVN's high-level language. Nobody writes these OpenFlow rules by hand; `ovn-detrace` recovers the origin so the engineer knows where to fix in NBDB.

### `ovn-sbctl dump-flows`, chassis logical flow dump

This command dumps every logical flow installed on a chassis, in OVN's human-readable format (not raw OpenFlow).

```bash
# Dump every logical flow on the current chassis
ovn-sbctl dump-flows

# Filter by a specific logical datapath
ovn-sbctl dump-flows | grep "ls1"

# Count logical flows per pipeline (ingress versus egress)
ovn-sbctl dump-flows | grep -c "Ingress"
ovn-sbctl dump-flows | grep -c "Egress"
```

## 20.0.5 Eight common failure scenarios

### 20.0.5.1 VM-to-VM in the same Logical Switch cannot ping

**Symptom:** `ping 10.0.0.2` from VM-A times out; both VMs are running and their interfaces are up.

```bash
# Step 1: Check binding state
ovn-sbctl list Port_Binding | grep -E "logical_port|chassis_name|mac_address"
# Expected: both ports have a non-empty chassis_name

# Step 2: Simulate with ovn-trace
ovn-trace ls1 'inport == "vm-a" && eth.src == 52:54:00:11:22:33 && \
  eth.dst == 52:54:00:44:55:66 && ip4 && icmp4'
# Expected: ends with "output to \"vm-b\""
# If the ACL drops, check: ovn-nbctl acl-list ls1

# Step 3: If ovn-trace is OK, check the OpenFlow layer
ovs-ofctl dump-ports br-int
# rx_dropped greater than 0 on vm-a or vm-b means a drop at the datapath
```

Common causes: an ACL deny without an exception for ICMP; a Port_Binding not claimed because ovn-controller is not running on the chassis; a MAC address in Port_Binding that does not match the NIC.

### 20.0.5.2 VM cannot reach the external network (Gateway Router fails)

**Symptom:** the VM can ping a peer on the same switch, but cannot ping the gateway 10.0.0.254.

```bash
# Step 1: Check that the Logical Router exists and has a port
ovn-nbctl show | grep -A 10 "router"
# Need to see: a router port attached to ls1 with IP 10.0.0.254/24

# Step 2: Trace the packet to the gateway IP
ovn-trace ls1 'inport == "vm-a" && eth.src == 52:54:00:11:22:33 && \
  eth.dst == <router-lrp-mac> && ip4.dst == 10.0.0.254 && icmp4'
# Expected: see output reach the router ingress pipeline

# Step 3: Check chassisredirect (which chassis hosts the Gateway Router)
ovn-sbctl list Port_Binding | grep chassisredirect
# If the chassis field is empty, the HA_Chassis_Group has not yet elected a master (Part 13.6)
```

### 20.0.5.3 The tunnel does not form between two chassis

**Symptom:** `ovn-sbctl show` lists both chassis, but traffic between VMs on different chassis does not arrive.

```bash
# Step 1: Check the tunnel neighbour table
ovs-appctl tnl/neigh/show
# Expected: see the IP of the remote chassis with the underlay NIC MAC
# If not seen, ARP resolution failed on the underlay network

# Step 2: Check that the Geneve port was created
ovs-vsctl show | grep -A 3 "type: geneve"

# Step 3: Check underlay connectivity
ping <remote-chassis-underlay-ip>
# In particular, check that UDP port 6081 is not blocked
```

### 20.0.5.4 ovn-northd does not translate (NBDB to SBDB stuck)

**Symptom:** add a new Logical Switch Port, but after 30 seconds it does not appear in SBDB.

```bash
# Step 1: Check northd leader election
ovn-appctl -t /var/run/ovn/ovn-northd.ctl status
# Expected: "status: active"

# Step 2: Check the sequence number sync
ovn-nbctl get NB_Global . nb_cfg
ovn-sbctl get SB_Global . nb_cfg
# Expected: the two values are equal
# If NB_Global.nb_cfg is greater than SB_Global.nb_cfg, northd is lagging

# Step 3: Check the northd log
journalctl -u ovn-northd --since "5 minutes ago" | grep -E "error|warn"
```

### 20.0.5.5 Port binding stuck, the chassis does not claim

**Symptom:** the VM is running, but `ovn-sbctl list Port_Binding` shows `chassis: []`.

```bash
# Step 1: Check external_ids:iface-id on the OVS interface
ovs-vsctl get Interface tap0 external_ids
# Expected: {"iface-id": "<logical-switch-port-uuid>"}
# If iface-id is missing, the orchestrator has not set it (see Part 9.13)

# Step 2: Check the ovn-controller log
journalctl -u ovn-controller --since "2 minutes ago" | \
  grep -E "binding|claim|iface-id"

# Step 3: Check that the chassis registered successfully
ovn-sbctl list Chassis | grep $(hostname -s)
# If not seen, ovn-controller cannot connect to SBDB
```

### 20.0.5.6 ARP does not resolve, the ARP suppressor is missing an entry

**Symptom:** ping fails with "no route to host"; `tcpdump` shows the ARP request but no reply.

```bash
# Step 1: Check that the ARP responder flow is installed
ovs-ofctl dump-flows br-int | grep "arp,arp_tpa=10.0.0.2"
# Expected: see a flow with action OFPAT_SET_FIELD (the ARP reply)

# Step 2: Check MAC_Binding in SBDB
ovn-sbctl list MAC_Binding | grep "10.0.0.2"
# If empty, the ARP suppressor has no entry

# Step 3: Check whether the port has a static MAC plus IP
ovn-nbctl list Logical_Switch_Port | grep -A 5 "vm-b"
# The addresses field must be: "52:54:00:44:55:66 10.0.0.2"
# If it is "dynamic", the MAC and IP are learned through ARP, and without one no reply can be sent
```

### 20.0.5.7 Load Balancer VIP unreachable

**Symptom:** `curl http://10.100.0.1:80` times out or returns connection refused.

```bash
# Step 1: Check the LB configuration in NBDB
ovn-nbctl list Load_Balancer
# Check: the vips field has "10.100.0.1:80" with a non-empty backend list

# Step 2: Check that the LB is attached to the Logical Switch
ovn-nbctl list Logical_Switch | grep -A 5 "ls1"
# The load_balancer field must contain the LB UUID

# Step 3: Trace with the destination VIP
ovn-trace ls1 'inport == "vm-a" && eth.src == 52:54:00:11:22:33 && \
  eth.dst == <router-mac> && ip4.dst == 10.100.0.1 && tcp.dst == 80'
# Expected: see a DNAT action, ip4.dst becomes the backend IP
# If no DNAT is seen, the LB is not attached to the Logical Switch
```

### 20.0.5.8 Native DHCP does not serve

**Symptom:** the VM boots without an IP; `tcpdump` on the VM sees DHCP Discover but no DHCP Offer.

```bash
# Step 1: Check that DHCP_Options exists in NBDB
ovn-nbctl list DHCP_Options
# An entry must exist with cidr="10.0.0.0/24", server_id, router, and dns_server

# Step 2: Check that the Logical Switch Port is linked to DHCP_Options
ovn-nbctl list Logical_Switch_Port | grep -E "name|dhcpv4_options"
# dhcpv4_options must contain a UUID referring to the DHCP_Options in step 1

# Step 3: Trace DHCP Discover
ovn-trace ls1 \
  'inport == "vm-a" && \
   eth.src == 52:54:00:11:22:33 && eth.dst == ff:ff:ff:ff:ff:ff && \
   ip4.src == 0.0.0.0 && ip4.dst == 255.255.255.255 && \
   udp.src == 68 && udp.dst == 67'
# Expected: see a "put_dhcp_opts()" action, the DHCP reply is generated inline
# If not seen, dhcpv4_options is not attached to the port
```

## 20.0.6 Guided Exercise

### Guided Exercise 1, isolation-first diagnosis, from symptom to root cause

In this exercise you apply the five-layer method to diagnose a VM-to-VM fault on an OVN system, using `ovn-trace` and `ofproto/trace` in the correct escalation order.

**Outcomes:**

- Apply the isolation-first principle to localise the fault before opening any logs.
- Distinguish when to stop at `ovn-trace` (a logical fault) from when to escalate to `ofproto/trace` (an OpenFlow fault) or `ovs-dpctl` (a datapath fault).

**Before you begin:**

The OVN system is running with at least one Logical Switch `ls1`, two Logical Switch Ports `vm-a` (10.0.0.1) and `vm-b` (10.0.0.2), and one active chassis.

**Instructions:**

**1.** Check layer 1, daemon health:

```bash
[ovn-central]# systemctl is-active ovn-northd ovn-ovsdb-server-nb ovn-ovsdb-server-sb
```

All three must return `active`. If any service is `inactive` or `failed`, stop and fix layer 1 first; the lower layers cannot be inspected when the control plane is not running.

**2.** Check layer 2, Port_Binding state:

```bash
[ovn-central]# ovn-sbctl list Port_Binding | grep -E "logical_port|chassis_name"
```

Both `vm-a` and `vm-b` must have a non-empty `chassis_name`. If either has `chassis_name: []`, that is the root cause; layers 3 and 4 do not need to be checked.

**3.** If layer 2 is OK, simulate with `ovn-trace` (layer 3, logical):

```bash
[ovn-central]# ovn-trace ls1 \
  'inport == "vm-a" && \
   eth.src == 52:54:00:11:22:33 && eth.dst == 52:54:00:44:55:66 && \
   ip4.src == 10.0.0.1 && ip4.dst == 10.0.0.2 && ip4.ttl == 64 && icmp4'
```

Read the output from top to bottom: if it ends with `output to "vm-b"`, the logical layer is OK. If `drop` or `...` (no match) appears at an ACL table, that is the place to fix in NBDB.

**4.** If `ovn-trace` reports delivered but real traffic still does not arrive, escalate to the OpenFlow layer:

```bash
[chassis-A]# ovs-appctl ofproto/trace br-int \
  in_port=tap0,dl_src=52:54:00:11:22:33,dl_dst=52:54:00:44:55:66,\
dl_type=0x0800,nw_src=10.0.0.1,nw_dst=10.0.0.2,nw_proto=1,nw_ttl=64
```

If the output ends with `Forwarded`, OpenFlow is OK; continue with `ovs-dpctl dump-flows` (layer 4) and `ovs-appctl tnl/neigh/show` (layer 5).

**Finish:**

These diagnostic commands do not change system state, so no cleanup is required.

## 20.0.7 Case study playback, three production scenarios

§20.0.5 listed eight common failure scenarios with concise workflows. This §20.0.7 plays back three real production incidents with detailed timelines, the diagnostic commands used, and lessons learned. The objective is for the on-call engineer to practise pattern thinking and recognise the symptom signature of each fault class.

### Case study 1, a new VM has no network after boot (an orchestrator versus ovn-controller race)

**Symptom:** the user creates a VM through OpenStack Nova; the VM boots successfully, but DHCP fails and pinging the gateway fails. After 60 to 90 seconds, the network starts working automatically.

**Context:** OpenStack Neutron with the OVN driver, a 100-hypervisor cluster, OVN 22.03.

**Timeline playback:**

```
T+0s    OpenStack Neutron API receives a port create request
T+0.5s  The Neutron plugin creates a Logical Switch Port (LSP) in the OVN NBDB
T+1s    ovn-northd translates the LSP into a Logical_Flow in SBDB
T+2s    Nova (libvirt) launches the VM, attaches the tap interface to OVS br-int
T+3s    The VM boots, the kernel sends a DHCP Discover broadcast
T+3s    ovn-controller has not yet received external-ids:iface-id on the tap interface
        (the orchestrator has not set it because of a race condition)
        -> Port_Binding.chassis = [] (empty)
        -> no OpenFlow rule for this tap
T+3s    DHCP Discover ingresses br-int, matches no rule, is dropped silently
T+15s   The VM retries DHCP Discover (the kernel default timeout)
T+30s   The VM retries a second time
T+90s   The orchestrator hook finally sets external-ids:iface-id=<LSP-UUID>
        (the delay is due to a queue inside the orchestrator)
T+91s   The OVS IDL notifies ovn-controller about the external-ids change
        -> en_runtime_data recomputes, claims the port_binding
        -> installs the local OpenFlow rule
T+92s   The next DHCP Discover retry matches the rule and the reply is OK
T+95s   The VM has an IP and pings the gateway successfully
```

**Root cause:** a race between libvirt `add-port` and the orchestrator `set external-ids`. The tap port enters br-int before `iface-id` is set, so ovn-controller does not know which tap maps to which LSP and installs no OpenFlow rule.

**Diagnostic commands used:**

```bash
# Step 1: check port_binding
ovn-sbctl --bare --columns=logical_port,chassis list port_binding | grep <lsp-name>
# Output: "<lsp-name>  []"  (empty chassis means not bound)

# Step 2: check tap external-ids
ovs-vsctl get interface tap-<id> external-ids
# Output: {}  (empty, this is the bug)

# Step 3: ovn-controller log
sudo journalctl -u ovn-controller --since "2 min ago" | grep -i "binding"
# No "binding added" entry for this LSP
```

**Short-term fix (immediate):**

```bash
ovs-vsctl set interface tap-<id> external-ids:iface-id=<lsp-uuid>
# ovn-controller syncs in one or two seconds, the network works immediately
```

**Long-term fix:** the orchestrator (libvirt or nova hook) must set `external-ids` in the same transaction as `add-port`, not afterwards. Or set `external-ids` first, then attach the tap to br-int.

**Lesson:** the "VM no network after boot" signature is clear: `port_binding.chassis = []` plus `external-ids = {}`. Diagnosis only needs three commands within a single minute.

### Case study 2, intermittent 5 to 10 percent packet drop after an OVN upgrade

**Symptom:** after a rolling upgrade from OVN 22.03 to 22.09 on a 200-chassis cluster, users report intermittent ping drops of 5 to 10 percent across hosts. It is not deterministically reproducible; it happens only inside the upgrade window.

**Context:** a 30-minute rolling upgrade. ovn-northd is upgraded first (centrally), then chassis are upgraded sequentially in batches of 10.

**Timeline playback:**

```
T+0min   Rolling upgrade starts. ovn-northd restarts with the 22.09 binary
T+0min   SBDB schema version bump: 22.03 -> 22.09
         New column added: Port_Binding.additional_chassis (multichassis binding)
T+5min   Batch 1: chassis 1 to 10 finish upgrading, version 22.09
T+10min  Batch 2: chassis 11 to 20 upgrade
T+10min  Users report "intermittent ping drops cross-host"
T+12min  The on-call engineer starts investigating
T+12min  Check the ovn-controller log on chassis 21 to 100 (still 22.03)
         -> "Warning: unknown column Port_Binding.additional_chassis"
         -> the 22.03 chassis ignore the new column
         -> the Port_Binding view is outdated
T+13min  OpenFlow flows on the 22.03 chassis are stale (they do not see the port migrate)
         -> traffic to a recently migrated LSP hits a stale flow and is dropped
T+13min  The engineer confirms the hypothesis: drops are concentrated on the 22.03 chassis
T+14min  Decision: speed up the upgrade, do not pause
T+20min  Rolling upgrade complete: every chassis is 22.09
T+22min  Packet drop returns to 0 percent
```

**Root cause:** OVN 22.09 adds the column `additional_chassis` to Port_Binding to support multichassis binding (RFC 9135 style). The 22.03 chassis ignore the new column, so their Port_Binding view is incomplete, the installed flows are out of date, and drops occur.

**Diagnostic commands used:**

```bash
# Check the schema version on the central node
ovn-sbctl --db-version

# Check the version of every chassis
for c in $(ovn-sbctl list chassis -f csv -d , | awk -F, '{print $2}' | tr -d '"'); do
  v=$(ssh $c 'ovs-appctl -t ovn-controller version 2>/dev/null | head -1')
  echo "$c: $v"
done

# Grep the warning in every chassis's ovn-controller log
ansible all -i hosts -m shell -a \
  'journalctl -u ovn-controller --since "15 min ago" | grep -i "unknown.*column" | head -5'

# Count drop rates per chassis
ansible all -i hosts -m shell -a \
  'ovs-ofctl dump-ports br-int | grep -E "rx_dropped|tx_dropped" | head -3'
```

**Short-term fix:** speed up the rolling upgrade and do not pause part-way through. Keep the incompatibility window as short as possible.

**Long-term fix:**

- Pre-upgrade test environment with a schema diff check.
- The upgrade playbook explicitly states a maximum window (30 minutes) and forbids pauses.
- Monitor the packet drop metric throughout the upgrade.

**Lesson:** an OVN upgrade is not atomic. A schema change between versions creates an incompatibility window. A rolling upgrade must complete quickly; monitor drop metrics continuously. Schema diff documentation is mandatory before every upgrade.

### Case study 3, thundering herd after network partition recovery

**Symptom:** after a 30-second underlay switch reboot, a single chassis is isolated and then re-joins. After re-joining, the entire 500-chassis cluster experiences three to five minutes of elevated ovn-controller CPU plus transient packet loss.

**Context:** a 500-chassis cluster, 5,000 VMs, an underlay switch firmware update that produces a brief partition.

**Timeline playback:**

```
T+0s     Underlay switch firmware update begins, switch reboots
T+5s     Chassis-42 loses connectivity to the OVN cluster (SBDB plus peer chassis)
         The SB IDL session breaks on chassis-42
T+8s     BFD state DOWN for chassis-42 on the 499 peer chassis
         Tunnel ports to chassis-42 are flagged "down"
         Peer chassis remove the tunnel output flow for chassis-42
T+30s    The underlay switch is back up
T+32s    Chassis-42 regains underlay connectivity
T+33s    SB IDL reconnects chassis-42 to SBDB
T+33s    Chassis-42's ovn-controller triggers a FULL recompute
         (engine_recompute counter +1)
         Installs about 10,000 OpenFlow rules in a 30 to 45 second burst
T+34s    All 499 peers BFD state UP for chassis-42
         Port_Binding updated: chassis-42 reclaims its 10 VM ports
T+34s    The 499 peers' ovn-controller triggers en_runtime_data recompute
         Installs 499 x 10 = about 5,000 tunnel output flows for chassis-42's VMs
T+35-50s ARP resolution burst: 499 peers x 10 VMs on chassis-42 is about 5,000 ARPs
         Underlay switch CPU spikes (above 80 percent for 15 seconds)
         Some ARP requests time out and retry
T+60s    Ovn-controller CPU on chassis-42 peaks at 85 percent of a single core
T+120s   Cluster CPU returns to baseline
T+180s   Packet loss returns to 0 percent
```

**Root cause:** partition recovery causes a thundering herd: every peer chassis synchronises chassis-42 state simultaneously. Three effects compound:

1. Chassis-42 full recompute (slow, expensive).
2. 499 peers running incremental recompute concurrently (fast per peer, but 499 in parallel).
3. The underlay switch is overloaded by 5,000 ARP requests in 15 seconds.

**Diagnostic commands used:**

```bash
# On chassis-42 after the incident
journalctl -u ovn-controller --since "5 min ago" | grep -iE "reconnect|IDL|recompute"

# Coverage counter delta before and after the incident
ovs-appctl -t ovn-controller coverage/show | grep engine_recompute
# Before: engine_recompute=0 (steady state)
# After: engine_recompute=1 (triggered by reconnect)

# Tunnel neighbour table on chassis-42
ovs-appctl tnl/neigh/show

# ARP burst capture
sudo tcpdump -ni eth0 arp -c 200 -w /tmp/arp-burst.pcap
# Count ARPs per second post-partition
tshark -r /tmp/arp-burst.pcap -q -z io,stat,1,"arp"

# Underlay switch CPU (through SNMP or vendor CLI)
# Saw CPU above 80 percent in the window [T+35s, T+50s]
```

**Short-term fix:** none needed. Automatic recovery happens within three minutes. Do not restart anything by hand.

**Long-term fix:**

- Tune the BFD interval: the default 200ms x 3 = 600ms detection time. Increase to 500ms x 5 = 2.5s to avoid false positives during partitions shorter than two seconds.
- ovn-controller: use a shorter `--max-idle` OpenFlow flow idle timeout (300s reduced to 60s) so stale rules are cleaned up faster.
- Underlay switch: prioritise ARP traffic through a QoS policy.
- OVN 22.09+: use the "tunnel keepalive" feature to reduce the post-partition ARP burst.
- Monitoring: alert on `engine_recompute` delta greater than 0, or BFD flap rate above one per minute.

**Lesson:** partition recovery is not free. A thundering herd is a pattern inherent in a distributed control plane with eventual consistency. Monitor partition frequency plus ovn-controller CPU delta to catch it early. Tune BFD plus OVN features to smooth the recovery.

### Common takeaways from the three case studies

**One.** Systematic diagnosis has tools: each fault class has a clear signature when you know where to look. "VM no network" equals an empty port_binding. "Upgrade drop" equals a schema version mismatch. "Partition thundering herd" equals an engine_recompute spike plus BFD flap.

**Two.** Logs plus coverage counters are forensic evidence. Without logs, no playback. A production cluster must enable `journalctl` with rotation kept long enough (7 to 14 days), plus scrape coverage metrics every 30 seconds.

**Three.** The time from symptom to root cause is 5 to 15 minutes if a standard workflow exists. Above 30 minutes means the workflow is missing or the engineer is not yet trained. That is a signal that the playbook needs improvement.

**Four.** A long-term fix is always more advanced than a short-term fix: an orchestrator race is fixed by fixing the orchestrator, not OVN. An upgrade with state mismatch is fixed by improving the upgrade playbook, not by avoiding the upgrade. A partition thundering herd is fixed by tuning the distributed system, not by avoiding the partition.

## 20.X Anatomy Template A, the `ovs-appctl coverage/show` debug entry (v3.2 P4 standardise)

**Why run it:** a first-line debug command for systematic debugging, to verify that the kernel datapath is healthy and to identify anomalies in the key counters.

**Nine-attribute anatomy:**

| Counter | Healthy | Warning | Critical | Diagnostic hint |
|---------|---------|---------|----------|------------------|
| `flow_mod_add` | Steady, controller rate | 10x baseline | 100x baseline | OVN-northd churn, check stability |
| `flow_install` | About equal to flow_mod_add | More than 2x equals retry | More than 10x equals failure loop | OVSDB slow, Raft health |
| `upcall_flow_add` | Steady, traffic rate | A 10x spike equals cache miss | Sustained high equals pathology | See Part 9.26 revalidator storm |
| `upcall_action_error` | 0 | More than 0 equals action failure | Sustained equals bug | Check ovs-vswitchd.log |
| `flow_revalidate` | Less than 1 per second | More than 100 per second equals invalidation | Sustained equals flow churn | Revalidator threads NICE |
| `revalidator_sweep` | 1 per second steady | Less than 0.5 per second equals starved | More than 10 per second equals emergency | See Part 9.14 scenario D thrashing |
| `bridge_reconfigure` | About 0 | More than 1 per minute equals config loop | Sustained equals automation bug | CI/CD logs |
| `flow_mod_delete` | Matches lifecycle | About add equals steady state | Much greater than add equals cleanup storm | Tenant decommission, migration |
| `dpif_*` errors | 0 | More than 0 equals datapath issue | Sustained equals driver bug | `dmesg` kernel log |

**Failure scenarios:**

- `upcall_flow_add` 1000 per second plus `revalidator_sweep` 50 per second: classic revalidator storm (see Part 9.26).
- `flow_mod_add` rate equals controller, but `flow_install` equals 0: OVSDB stuck, check Raft.
- `upcall_action_error` sustained: an OpenFlow rule references an invalid port.
- `bridge_reconfigure` 10 per minute loop: an orchestrator CI/CD bug.

### 20.0.X.2 Use case for the cross-link table

**Typical scenario:** an operator hits the issue "VM A cannot ping VM B on the same logical switch". Workflow:

1. Identify the symptom matches REF scenario #1 (east-west same LS fails).
2. Look up the cross-link table; the primary file is 20.2 §20.2.4 plus §20.2.9.
3. Read the dedicated content (Port_Binding forensic, ACL drop GE).
4. Cross-reference the supporting files (13.7 ovn-controller, 9.25 ofproto/trace).
5. Apply the diagnostic chain.

**Benefit compared with "all in one file":**

- No duplicated content (each scenario has a dedicated chapter in the file native to its technology).
- The operator learns the scenario in context (for example, the LB scenario sits in the 13.9 LB chapter).
- The next time the REF version changes, only this cross-link table needs to be updated.

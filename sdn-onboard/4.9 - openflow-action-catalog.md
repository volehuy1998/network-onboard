## Context, the action catalog complements the match field catalog

Part 4.8 catalogs the **match field**, that is "what does the packet match against". This Part catalogs the **action**, that is "what is done to the packet after the match". The two catalogs together cover the entire flow rule syntax:

```
priority=100,<MATCH FIELDS>,actions=<ACTIONS>
         ^          ^             ^
         OF         Part 4.8     Part 4.9
         syntax
```

## 4.9.1 Action versus Instruction, the foundational distinction

OpenFlow 1.1 and later distinguishes two concepts:

| Concept | When it executes | What state it changes | Lives in | Example |
|---|---|---|---|---|
| **Action** | Immediately, on the packet | Packet header plus output port | `apply_actions:` or `write_actions:` | `output`, `set_field`, `dec_ttl`, `ct()` |
| **Instruction** | Pipeline control | Flow movement between tables | A top-level element of a flow entry | `Apply-Actions`, `Write-Actions`, `Clear-Actions`, `Goto-Table`, `Write-Metadata`, `Meter` |

OpenFlow 1.0 made no distinction; everything was an "action" executing in sequence. From 1.1, instructions are **wrappers around actions** that decide when and how the action executes.

### 4.9.1.1 The six foundational instructions

| Instruction | Purpose |
|---|---|
| `Apply-Actions(<action_list>)` | Execute the action list immediately; the packet change is visible to subsequent instructions |
| `Clear-Actions` | Clear the current action set |
| `Write-Actions(<action_list>)` | Merge the action list into the action set; execute when the packet exits the pipeline |
| `Write-Metadata(<value>[/<mask>])` | Update the metadata match field on the packet (passes through Goto-Table) |
| `Goto-Table(<table_id>)` | Move the packet to another table (table_id must be larger than the current one) |
| `Meter(<meter_id>)` | Apply a meter rate limit before forwarding |

In OVS flow-spec syntax, `actions=` is shorthand for `Apply-Actions(...)`. The full instruction form is rarely used directly but is important when reading the OpenFlow 1.3 and later spec or when reading trace output.

### 4.9.1.2 Action Set versus Apply Actions

The OpenFlow 1.1 and later pipeline maintains an **action set**, a set of actions that accumulate across tables. When the packet exits the pipeline (no more `Goto-Table`), the action set executes in order of the 12 priority levels (see §4.9.14).

Behavioural difference:

```
# apply_actions: executes immediately
table=0,priority=100,in_port=1,
  apply_actions(mod_dl_dst:aa:bb:cc:dd:ee:ff),
  goto_table:10

# Table 10 sees the packet with the new eth_dst
table=10,eth_dst=aa:bb:cc:dd:ee:ff,actions=output:5    # match

# write_actions: accumulates into the action set
table=0,priority=100,in_port=1,
  write_actions(output:5),
  goto_table:10

# Table 10 does not see output in the current actions
table=10,actions=<...>  # the action set still has output:5, merging with what table 10 writes
```

When the packet exits the pipeline, every write_actions entry merges in the 12-level priority order and then executes. apply_actions does not accumulate.

### 4.9.1.3 OVS `actions=` syntax shortcut

OVS simplifies the syntax: `actions=` defaults to Apply-Actions unless `write_actions(...)` is explicit. In `ovs-ofctl` flow spec:

```
# Syntax 1: shortcut (implicit apply_actions)
priority=100,in_port=1,actions=output:5

# Syntax 2: explicit apply_actions
priority=100,in_port=1,apply_actions(output:5)

# Syntax 3: write_actions (merge into the action set)
priority=100,in_port=1,write_actions(output:5),goto_table:10

# Syntax 4: mix
priority=100,in_port=1,
  actions=dec_ttl,
  write_actions(output:5),
  goto_table:10
```

The rest of this Part focuses on **actions**, not instructions. For full instruction detail, see Part 4.0 §4.0.X.

## 4.9.2 Action `output`, the most fundamental of OpenFlow

### 4.9.2.1 Template C for `output`

| Attribute | Value |
|---|---|
| **Syntax** | `output:<port>` or `output(port=<port>,max_len=<bytes>)` |
| **Category** | 1. Output |
| **OpenFlow version** | 1.0 all versions |
| **Prerequisites** | None |
| **Semantics** | Send the packet out of the specified port. Does not terminate the packet; `output` may be used multiple times to send to multiple ports |
| **Parameters** | `port`: 1-N (port number) or a reserved port (LOCAL/CONTROLLER/IN_PORT/NORMAL/FLOOD/ALL/TABLE); `max_len`: bytes for CONTROLLER, default unlimited |
| **Side effects** | Increments the port `tx_packets` and `tx_bytes` counters; does not modify packet content |
| **Conformance** | Per spec; reserved ports CONTROLLER and TABLE are OpenFlow concepts, LOCAL is OVS-style and common |

### 4.9.2.2 Full semantics

`output` is the most foundational action and executes first in the action-set priority order (see §4.9.14). When the packet flows through an action list containing `output:5`, OVS does the following:

1. Compute the egress port (port 5 in the example).
2. Clone the packet when needed (for mirrors).
3. Queue the packet into the port 5 send buffer.
4. The packet content is **not** modified (every `mod_*` or `set_field` must run beforehand).

When the action list has multiple `output` entries, the packet is cloned and sent out of multiple ports. This is how OVS implements flood or multicast **with explicit rules** instead of using the FLOOD or ALL reserved ports.

### 4.9.2.3 Use-case examples

```shell
# Ex 1: simple forwarding
$ ovs-ofctl add-flow br0 "table=0,priority=100,in_port=1,actions=output:2"

# Ex 2: multi-output (limited flood)
$ ovs-ofctl add-flow br0 "table=0,priority=100,in_port=1,dl_type=0x0806,\
    actions=output:2,output:3,output:4"   # broadcast ARP to three ports

# Ex 3: output with max_len (for CONTROLLER)
$ ovs-ofctl add-flow br0 "table=0,priority=100,in_port=1,\
    actions=output(port=CONTROLLER,max_len=128)"  # send only the first 128 bytes

# Ex 4: use the port name instead of the number
$ ovs-ofctl add-flow br-int "table=0,priority=100,in_port=tap-vm-web,\
    actions=output:tap-vm-db"
```

### 4.9.2.4 How trace output displays the action

```
bridge("br0")
-------------
 0. in_port=1, priority 100
    output:2
```

The trace shows action `output:2` as a child line of the matching rule. When the trace shows `Datapath actions: 2`, the kernel will forward to datapath port 2.

### 4.9.2.5 Failure scenarios

- **`output:<port>` with a non-existent port**: the action is silently skipped with no error, and the packet goes nowhere. Verify with `ovs-ofctl show br0` to inspect the port list.
- **`output:IN_PORT` has special meaning**: send the packet back out the ingress port (the ARP responder pattern). When `output:<ingress_port_number>` is used instead of `output:IN_PORT`, OVS rejects it (no direct loopback unless the reserved port is used).
- **`output` inside `write_actions(...)` keeps only one value**: the action set has only one effective output. To produce multiple outputs from the action set, use the `group` action.

## 4.9.3 Action `drop`, terminate the packet

### 4.9.3.1 Template C

| Attribute | Value |
|---|---|
| **Syntax** | `drop` |
| **Category** | 1. Output |
| **OpenFlow version** | Explicit `drop` is an NXM extension; the OF spec says "empty action list = drop" (implicit) |
| **Prerequisites** | None |
| **Semantics** | Terminate packet processing. No output, no modification. The rule's `n_packets` counter still increments |
| **Parameters** | None |
| **Side effects** | The flow-rule counter increments; the packet is discarded; the kernel datapath has a dedicated `OVS_ACTION_ATTR_DROP` (OVS 2.17 and later) for an efficient drop path |
| **Conformance** | OVS extension; spec 1.0 uses an empty action list as the implicit drop |

### 4.9.3.2 Explicit drop versus implicit drop

```
# Implicit drop (rule has a match but the action list is empty)
priority=50,in_port=1,dl_type=0x0800,actions=

# Explicit drop (OVS extension)
priority=50,in_port=1,dl_type=0x0800,actions=drop
```

The two forms are semantically equivalent. Explicit `drop` is easier to read and is unambiguous. OpenFlow 1.5 and later requires explicit drop (spec §5.10).

When no rule matches the packet (a table miss), OpenFlow 1.3 and later defaults to drop. The controller should explicitly install a table-miss rule to:
- Drop with logging: `priority=0,actions=controller(userdata=01020304)`
- Forward to the controller to learn: `priority=0,actions=output:CONTROLLER`

### 4.9.3.3 How trace output displays a drop

```
 0. in_port=1,ip,nw_proto=1, priority 150
    drop

Datapath actions: drop
```

`Datapath actions: drop` is ground truth: the kernel will DROP the packet, and the `missed` counter does not increase further (because the packet matched a megaflow drop).

### 4.9.3.4 The dedicated drop action in OVS 2.17 and later

Before OVS 2.17, drop was implemented through an empty action list. OVS 2.17 added **OVS_ACTION_ATTR_DROP** in the datapath; the kernel has a dedicated fast path for drop that avoids a context switch. Check via `dpif/show-dp-features`:

```shell
$ sudo ovs-appctl dpif/show-dp-features br0 | grep -i drop
Explicit Drop action: Yes
```

`No` means the kernel is older; explicit drop falls back to the generic drop path with a slight performance loss.

## 4.9.4 Action `normal`, the standard L2 learning switch

### 4.9.4.1 Template C

| Attribute | Value |
|---|---|
| **Syntax** | `normal` or `actions=NORMAL` |
| **Category** | 1. Output |
| **OpenFlow version** | OF 1.0 all (reserved port NORMAL) |
| **Prerequisites** | None |
| **Semantics** | Process the packet as a standard L2 switch: MAC learning plus FDB-based forwarding. Final action; terminates the pipeline |
| **Parameters** | None |
| **Side effects** | Automatic MAC learning (populates the FDB); broadcasts on unknown unicast; STP and RSTP are applied when enabled |
| **Conformance** | OpenFlow reserved port; behaviour is implementation-specific (mostly OVS-specific) |

### 4.9.4.2 Use case

`normal` is useful when mixing OpenFlow logic with L2 learning. A common pattern:

```shell
# Layer-2 switch with a single exception ACL
$ ovs-ofctl add-flow br0 "priority=100,ip,nw_dst=10.0.0.99,actions=drop"
$ ovs-ofctl add-flow br0 "priority=0,actions=normal"
```

The priority-100 rule drops traffic to 10.0.0.99. All other traffic falls through to priority 0 -> `normal` -> standard L2 switch.

### 4.9.4.3 Interaction with fail_mode

A bridge with `fail_mode=secure` plus a disconnected controller does not run NORMAL automatically. The operator must explicitly install an `actions=NORMAL` rule to fall back.

A bridge with `fail_mode=standalone` plus a disconnected controller has OVS auto-install an `actions=NORMAL` rule at priority 0, and L2 switching resumes immediately.

### 4.9.4.4 Failure scenarios

- **`normal` on an OVN-managed br-int**: OVN does **not** use `normal`. OVN builds the pipeline explicitly with one match rule at a time. Using `normal` on OVN's br-int breaks the pipeline.
- **STP plus NORMAL**: when STP is enabled via `ovs-vsctl set Bridge br0 stp_enable=true`, the NORMAL action respects the STP state; a packet arriving on a BLOCKING port is not forwarded.

## 4.9.5 Actions `flood` and `all`, the broadcast variants

### 4.9.5.1 `flood`

| Attribute | Value |
|---|---|
| **Syntax** | `flood` or `actions=FLOOD` |
| **Semantics** | Output the packet to every port on the bridge **except the ingress port and STP-blocked ports** |
| **Scope** | Bridge-level; does not respect VLAN membership |

### 4.9.5.2 `all`

| Attribute | Value |
|---|---|
| **Syntax** | `all` or `actions=ALL` |
| **Semantics** | Output the packet to **every port** of the switch, including the ingress port and STP-blocked ports. "Unfiltered flood" |

### 4.9.5.3 Comparison

| Action | Includes ingress | Respects STP | Filters by VLAN |
|---|---|---|---|
| `flood` | No | Yes | No |
| `all` | Yes | No | No |
| `normal` | No | Yes | Yes |
| `output:<n>, output:<m>, ...` | Per the explicit list | No | No |

Use `flood` for broadcast ARP responses (STP-safe). Use `all` for debugging or mirroring (to see the packet on every port).

### 4.9.5.4 Failure scenarios

- **`flood` on OVN's br-int**: like `normal`, OVN does not use it. Bypassing the OVN pipeline can create a loop through the Geneve tunnel because remote chassis see the flooded packet.
- **`all` loop risk**: because it includes the ingress port, the packet may loop without STP. Avoid `all` in production beyond a short debug session.

## 4.9.6 Action `controller`, send a PACKET_IN to the OpenFlow controller

### 4.9.6.1 Template C

| Attribute | Value |
|---|---|
| **Syntax** | `controller[:<max_len>]` or `controller(reason=<x>,max_len=<n>,userdata=<hex>)` |
| **Category** | 1. Output (reserved port) |
| **OpenFlow version** | All. OF 1.3 and later add the `reason` and `userdata` parameters |
| **Prerequisites** | Controller connected (`ovs-vsctl set-controller br0 tcp:...`) |
| **Semantics** | Send the packet as an OFPT_PACKET_IN message to the controller |
| **Parameters** | `max_len`: truncate to N bytes (default 128 for OF 1.0, 64K for 1.3); `reason`: `action`, `no_match`, `invalid_ttl`, `action_set`, `group`, `packet_out`; `userdata`: opaque hex data |
| **Side effects** | The controller receives the packet and may respond with OFPT_PACKET_OUT to continue forwarding |
| **Conformance** | Per spec; a common pattern for learn-and-forward |

### 4.9.6.2 Use case and risk

```shell
# Example: send a DHCP packet to the controller
$ ovs-ofctl add-flow br0 \
    "priority=100,udp,tp_src=68,tp_dst=67,actions=controller(max_len=512)"

# Example: table-miss default to the controller
$ ovs-ofctl add-flow br0 "table=0,priority=0,actions=controller"
```

**Risk.** It is very easy to spam the `controller` action. One bad rule can send all traffic to the controller and overwhelm the control channel. Always:

1. Use `max_len=128` (or 256) to truncate the payload.
2. Rate-limit with a `meter` action before `controller`.
3. Track `ofconn.packets_in_sent` via `coverage/show`.

### 4.9.6.3 Failure scenarios

- **The controller disconnects while a rule has `actions=controller`**: the OVS internal packet queue fills up and packets are eventually dropped. The `rconn_queued` counter (Part 9.4 §9.4.1.3 coverage) rises.
- **`max_len=0`**: the packet_in carries metadata only (no payload); the controller cannot decide based on packet content.

## 4.9.7 Action `local`, output to the bridge's internal port

### 4.9.7.1 Template C

| Attribute | Value |
|---|---|
| **Syntax** | `local` or `actions=LOCAL` |
| **Category** | 1. Output (reserved port) |
| **OpenFlow version** | All |
| **Prerequisites** | The bridge has an internal port (the LOCAL port is automatically created for every bridge with the bridge's name) |
| **Semantics** | Output the packet into the host's local network stack through the LOCAL port (the Linux netdev `br0`) |
| **Side effects** | The packet enters the kernel rx queue of netdev `br0` and may be processed further by iptables or tc |

### 4.9.7.2 Use case

The OVS host runs a TCP server on an IP assigned to the LOCAL port:

```shell
# br-int has 10.1.0.1 assigned on netdev br-int (the LOCAL port)
$ ip addr add 10.1.0.1/24 dev br-int

# Rule that accepts traffic destined to 10.1.0.1 into the local stack
$ ovs-ofctl add-flow br-int "priority=100,ip,nw_dst=10.1.0.1,actions=local"
```

### 4.9.7.3 Comparison with `output:<LOCAL port number>`

```shell
$ ovs-ofctl show br-int | grep LOCAL
 LOCAL(br-int): addr:...
```

The LOCAL port has the reserved port number 0xfffe. `actions=local`, `actions=output:LOCAL`, and `actions=output:65534` are equivalent.

## 4.9.8 Action `in_port`, send back to the ingress

### 4.9.8.1 Template C

| Attribute | Value |
|---|---|
| **Syntax** | `in_port` or `actions=IN_PORT` |
| **Category** | 1. Output (reserved port) |
| **OpenFlow version** | All |
| **Prerequisites** | None |
| **Semantics** | Output the packet on the **ingress port** (send back to where the packet arrived from). The classic ARP responder pattern |
| **Side effects** | The packet travels back through the same port it arrived on |

### 4.9.8.2 ARP responder pattern

```shell
# OVN-style ARP responder: match ARP request for the gateway IP,
# transform the packet into an ARP reply,
# and send back on the ingress port (no broadcast)
$ ovs-ofctl add-flow br-int \
    "priority=200,arp,arp_op=1,arp_tpa=10.1.1.1,\
     actions=move:NXM_OF_ETH_SRC[]->NXM_OF_ETH_DST[],\
             mod_dl_src:aa:bb:cc:dd:ee:ff,\
             load:0x2->NXM_OF_ARP_OP[],\
             move:NXM_NX_ARP_SHA[]->NXM_NX_ARP_THA[],\
             move:NXM_OF_ARP_SPA[]->NXM_OF_ARP_TPA[],\
             load:0xaabbccddeeff->NXM_NX_ARP_SHA[],\
             load:0x0a010101->NXM_OF_ARP_SPA[],\
             IN_PORT"
```

`IN_PORT` at the end is the reserved port. It is used here because the regular output rule does not normally allow output back on the ingress port.

### 4.9.8.3 Difference between the `in_port` action and the `in_port` match field

Do not confuse:
- `in_port=1` in the **match spec** is a match field meaning "the packet came from port 1" (Part 4.8 §4.8.2.1).
- `actions=in_port` or `output:IN_PORT` is the **action** meaning "send the packet out on the port it arrived from".

A flow may use both:
```
priority=200,in_port=1,arp,arp_op=1,actions=IN_PORT
```

## 4.9.9 Action `table`, submit the packet back into the pipeline lookup

### 4.9.9.1 Template C

| Attribute | Value |
|---|---|
| **Syntax** | `table` or `output:TABLE` |
| **Category** | 1. Output (reserved port) |
| **OpenFlow version** | 1.0 and later |
| **Prerequisites** | Only meaningful in a PACKET_OUT context (the controller injects a packet into the switch) |
| **Semantics** | Submit the packet into the OpenFlow pipeline starting at table 0 |

### 4.9.9.2 Use case

`TABLE` is rarely used in flow rules; it appears mostly in a PACKET_OUT message from the controller:

```
# Controller injects a packet into the pipeline
OFPT_PACKET_OUT: buffer_id=-1, in_port=<x>, actions=OUTPUT:TABLE, data=<hex>
```

When the controller receives a packet_in and then decides "let the switch handle it normally", it sends the packet to TABLE rather than choosing a specific output port.

Inside flow rules, this is uncommon; typically `resubmit(,0)` or `goto_table:N` is used instead.

## 4.9.10 Action `group`, group-table indirection

### 4.9.10.1 Template C

| Attribute | Value |
|---|---|
| **Syntax** | `group:<group_id>` |
| **Category** | 1. Output (indirect) |
| **OpenFlow version** | 1.1 and later |
| **Prerequisites** | The group must be installed first via `ovs-ofctl add-group` |
| **Semantics** | Execute the action list inside the group entry. Groups have four types: all, select, indirect, fast_failover |
| **Parameters** | `group_id`: a 32-bit group ID |
| **Side effects** | Depends on the group type: multicast (all), load balance (select), shared indirection (indirect), active-backup (fast_failover) |

### 4.9.10.2 The four group types

| Type | Bucket count | Selection | Use case |
|---|---|---|---|
| `all` | 1+ | Execute ALL buckets | Multicast, efficient broadcast |
| `select` | 1+ | Pick one bucket by hash | Load balance (ECMP, LB backend) |
| `indirect` | 1 | Execute the only bucket | Shared forwarding (next-hop indirection, reduce rule count) |
| `fast_failover` | 1+ | The first "live" bucket | Active-backup forwarding with `watch_port` |

### 4.9.10.3 Template for group type `select` (LB pattern)

```shell
# Install group 1: load balance across three backends
$ ovs-ofctl add-group br0 "group_id=1,type=select,\
  bucket=weight:100,actions=mod_dl_dst:aa:bb:cc:00:00:01,output:5,\
  bucket=weight:100,actions=mod_dl_dst:aa:bb:cc:00:00:02,output:6,\
  bucket=weight:100,actions=mod_dl_dst:aa:bb:cc:00:00:03,output:7"

# Rule: TCP destined to the VIP enters group 1
$ ovs-ofctl add-flow br0 \
  "priority=100,tcp,nw_dst=10.1.0.100,tp_dst=80,actions=group:1"
```

Weight defaults to 1 when not specified. The default hash function is the five-tuple (src_ip, dst_ip, src_port, dst_port, proto).

### 4.9.10.4 Template for group type `fast_failover` (link backup)

```shell
# Group 2: primary eth0, backup eth1
$ ovs-ofctl add-group br0 "group_id=2,type=fast_failover,\
  bucket=watch_port:eth0,actions=output:eth0,\
  bucket=watch_port:eth1,actions=output:eth1"

# Rule that uses the group
$ ovs-ofctl add-flow br0 "priority=100,actions=group:2"
```

OVS tracks the `watch_port` state; when eth0 goes down the switch moves to bucket 2 (eth1) within 50 ms (with BFD).

### 4.9.10.5 `dump-groups` to verify

```shell
$ ovs-ofctl -O OpenFlow13 dump-groups br0

 group_id=1,type=select,
  bucket=weight:100,actions=mod_dl_dst:aa:bb:cc:00:00:01,output:5,
  bucket=weight:100,actions=mod_dl_dst:aa:bb:cc:00:00:02,output:6,
  bucket=weight:100,actions=mod_dl_dst:aa:bb:cc:00:00:03,output:7

 group_id=2,type=fast_failover,
  bucket=watch_port:eth0,actions=output:eth0,
  bucket=watch_port:eth1,actions=output:eth1
```

## 4.9.11 Action `resubmit`, re-look up in the current or another table

### 4.9.11.1 Template C

| Attribute | Value |
|---|---|
| **Syntax** | `resubmit:<port>` or `resubmit(<port>,<table>)` |
| **Category** | 6. Control/Pipeline |
| **OpenFlow version** | NXM extension (OVS-specific) |
| **Prerequisites** | None |
| **Semantics** | Clone the packet with a new virtual `in_port` plus optionally a different table, and look up in the pipeline as if it were a new packet. The action set is shared across nested resubmits |
| **Parameters** | `port`: virtual in_port for the resubmit (default current); `table`: target table (default current) |
| **Side effects** | Stack depth limit of 16 (avoiding infinite recursion); the action runs against the matching rule found in the resubmitted lookup |

### 4.9.11.2 `resubmit` versus `goto_table`

| Aspect | `goto_table:N` | `resubmit(,N)` |
|---|---|---|
| OpenFlow standard | Yes (1.1 and later) | No (NXM) |
| Direction | Forward only (N greater than current) | Either direction (forward or backward) |
| Action set | Cumulative (write_actions accumulate) | Shared across nested calls |
| Use case | Linear pipeline | Modular or shared table |

OVS prefers `goto_table` for the standard OpenFlow pipeline and `resubmit` for OVS-specific patterns such as nested classifiers.

### 4.9.11.3 OVN use case

OVN uses `resubmit` extensively in translation. Example from an OVN logical flow:

```
# Logical: move to the next table
table=8(ls_in_lookup_fdb), priority=0, match=1, action=next;

# Physical: resubmit
table=8,priority=0,actions=resubmit(,9)
```

See Part 13.7 for the OVN logical-to-physical translation that uses `resubmit`.

### 4.9.11.4 Failure scenarios

- **Resubmit loop deeper than 16**: OVS aborts with "resubmit recursion too deep" and the packet is dropped. Usually caused by a buggy classifier loop. Check the trace to see the path.
- **Shared action-set trap**: `apply_actions(output:5)` followed by `resubmit` shares the action set; the nested lookup may override it. Best practice: use `write_actions` rather than `apply_actions` in combination with resubmit.

## 4.9.12 Action `clone`, execute actions transiently without committing the modifications

### 4.9.12.1 Template C

| Attribute | Value |
|---|---|
| **Syntax** | `clone(<action_list>)` |
| **Category** | 6. Control/Pipeline |
| **OpenFlow version** | NXM extension |
| **Prerequisites** | None |
| **Semantics** | Execute the action list against a temporary packet clone. After the clone block ends, the original packet is not modified by the actions inside the clone |
| **Side effects** | Side-effect actions (output, learn, ct(commit)) still execute; only the packet header and metadata are restored |

### 4.9.12.2 Use case

```shell
# Output a packet copy on port 99 (span or mirror) without affecting the main packet
$ ovs-ofctl add-flow br0 \
    "priority=100,in_port=1,actions=clone(output:99),output:2"
```

The packet sent to port 2 keeps its original state. The clone going to port 99 may include modification actions before output (such as mod_dl_dst) without changing the packet leaving on port 2.

### 4.9.12.3 Difference from explicit multi-output

```
actions=output:99,output:2
# Packet A is sent to port 99, packet A' (going to port 2) remains unchanged

actions=clone(mod_dl_dst:aa:aa:..,output:99),output:2
# Packet A' (going to port 2) is not affected by mod_dl_dst
```

Clone is the clean way to forward a modified packet without polluting the main forward path.

## 4.9.14 Action Set execution order, the 12 priority levels

When the packet exits the pipeline (no more `Goto-Table`), the accumulated action set is executed in **12 priority levels** (OpenFlow 1.3 §5.10, table 9):

| Priority | Action | Meaning |
|---|---|---|
| 1 | `copy_ttl_in` | Copy the outer TTL into the inner (IP-in-IP or MPLS decap prep) |
| 2 | `pop_mpls`, `pop_pbb`, `pop_vlan` | Decapsulate |
| 3 | `push_pbb` | PBB encapsulation |
| 4 | `push_mpls` | MPLS encapsulation |
| 5 | `push_vlan` | VLAN encapsulation |
| 6 | `copy_ttl_out` | Copy the inner TTL into the outer |
| 7 | `dec_mpls_ttl`, `dec_nw_ttl` | Decrement TTL |
| 8 | `set_field` (all variants), `mod_*` | Field modification |
| 9 | `set_queue` | Queue assignment |
| 10 | `group` | Group indirection |
| 11 | `output` | Output port |
| 12 | (cleanup of the action set) | Clear and exit |

Meaning: regardless of the order in which the actions were written into `write_actions(...)`, on pipeline exit they execute **in priority order**.

### 4.9.14.1 Example

```
write_actions(output:5, mod_dl_dst:aa:aa:aa:aa:aa:aa, push_vlan:0x8100)
```

Written: output first, mod second, push last. Executed in priority order:
- L5 push_vlan
- L8 mod_dl_dst
- L11 output

The result: push VLAN, modify MAC, output to port 5, regardless of the reverse write order.

### 4.9.14.2 Apply-Actions does not follow priority

`apply_actions=` executes **sequentially in the written order**. An important distinction when reading a flow spec:

```
apply_actions(mod_dl_dst:aa..., push_vlan:0x8100, output:5)
# Sequential: mod, then push, then output
# The packet leaving port 5 has the VLAN plus the new MAC
```

OVN rule of thumb: **OVN logical flows mostly use apply_actions** (sequential order is easier to reason about). `write_actions` is used for pipeline egress consolidation.

## 4.9.15 Category 2, encapsulation actions: VLAN tagging

### 4.9.15.1 `push_vlan`

| Attribute | Value |
|---|---|
| **Syntax** | `push_vlan:<ethertype>` (for example `push_vlan:0x8100` for 802.1Q, `0x88a8` for 802.1ad Q-in-Q) |
| **Category** | 2. Encap/Decap |
| **OpenFlow version** | 1.1 and later |
| **Prerequisites** | None |
| **Semantics** | Insert a VLAN header after the Ethernet header. The default VID is 0 and PCP is 0; must be followed by `mod_vlan_vid` or `set_field:<vid>->vlan_vid` to set a specific VLAN ID |
| **Parameters** | `ethertype`: 0x8100 (VLAN) or 0x88a8 (PBB or Q-in-Q) |
| **Side effects** | Increases packet length by 4 bytes; sets the `vlan_tci` Present bit |

### 4.9.15.2 `pop_vlan`

| Attribute | Value |
|---|---|
| **Syntax** | `pop_vlan` or `strip_vlan` (OVS alias) |
| **Category** | 2. Encap/Decap |
| **OpenFlow version** | 1.1 and later |
| **Prerequisites** | The packet has a VLAN tag (`vlan_tci[12]=1` Present bit) |
| **Semantics** | Remove the outermost VLAN header |
| **Side effects** | Decreases packet length by 4 bytes; `vlan_tci` returns to 0 |

### 4.9.15.3 Tagging workflow examples

```shell
# Access port to trunk port: tag the packet with VLAN 100
$ ovs-ofctl add-flow br0 \
    "priority=100,in_port=1,vlan_tci=0,\
     actions=push_vlan:0x8100,mod_vlan_vid:100,output:2"

# Trunk port to access port: untag
$ ovs-ofctl add-flow br0 \
    "priority=100,in_port=2,dl_vlan=100,\
     actions=strip_vlan,output:1"

# Q-in-Q double tagging
$ ovs-ofctl add-flow br0 \
    "priority=100,in_port=3,\
     actions=push_vlan:0x88a8,mod_vlan_vid:500,\
             push_vlan:0x8100,mod_vlan_vid:100,\
             output:4"
```

### 4.9.15.4 Failure scenarios

- **`push_vlan` without setting the VID**: the packet carries a VLAN header with VID=0 (reserved for priority tagging; many switch hardware platforms drop it). Always pair with `mod_vlan_vid` or `set_field`.
- **`pop_vlan` on an untagged packet**: silently skipped (no error); verify with a `vlan_tci=0x1000/0x1000` match beforehand.

## 4.9.16 Category 2, MPLS plus PBB encapsulation

### 4.9.16.1 `push_mpls`, `pop_mpls`

| Attribute | `push_mpls` | `pop_mpls` |
|---|---|---|
| **Syntax** | `push_mpls:<ethertype>` (0x8847 unicast, 0x8848 multicast) | `pop_mpls:<ethertype>` (the inner protocol) |
| **OpenFlow version** | 1.1 and later | 1.1 and later |
| **Semantics** | Insert an MPLS label on top | Remove the outermost MPLS label, exposing the inner one |
| **Side effects** | Packet length plus 4 bytes; `mpls_label`, `tc`, `bos`, and `ttl` must be set separately | Packet length minus 4 bytes |

### 4.9.16.2 MPLS label push example

```shell
# L3 VPN: push MPLS label 100 on top; eth_type changes to 0x8847
$ ovs-ofctl add-flow br0 \
    "priority=100,ip,nw_dst=10.1.0.0/24,\
     actions=push_mpls:0x8847,\
             set_field:100->mpls_label,\
             set_field:1->mpls_bos,\
             set_field:64->mpls_ttl,\
             output:uplink"

# MPLS pop at the PE:
$ ovs-ofctl add-flow br0 \
    "priority=100,mpls,mpls_label=100,\
     actions=pop_mpls:0x0800,\
             output:ce_port"
```

### 4.9.16.3 `push_pbb`, `pop_pbb` (Provider Backbone Bridge)

Similar to MPLS but for 802.1ah PBB. Rarely used outside MEF metro Ethernet. Skipping the deep dive because the hyperscale data centre scope does not need it.

### 4.9.16.4 `encap`, `decap` (generic, OF 1.5 plus NSH)

OpenFlow 1.5 added generic `encap` and `decap` for arbitrary header chains (NSH, GRE, VXLAN-GPE) instead of being hard-coded to VLAN, MPLS, or PBB. Rarely used in OVN; OVN encapsulates Geneve and VXLAN through **tunnel port** configuration rather than through actions.

## 4.9.17 Category 3, `set_field`, the generic field-rewrite action

### 4.9.17.1 Template C

| Attribute | Value |
|---|---|
| **Syntax** | `set_field:<value>[/<mask>]-><field>` or `set_field(<field>=<value>)` |
| **Category** | 3. Field modification |
| **OpenFlow version** | 1.2 and later (OXM TLV required) |
| **Prerequisites** | Field-specific (for example, `eth_type=0x0800` for IP fields) |
| **Semantics** | Generic field rewrite, replacing every `mod_*` per-field action. Supports a mask for partial writes |
| **Parameters** | `value`: the new value (hex, decimal, MAC, IP); `mask`: optional bitmask; `field`: OXM or NXM field name |
| **Side effects** | Modifies the packet header in place; the L3 and L4 checksum is recomputed automatically |

### 4.9.17.2 Common set_field rewrite examples

```shell
# Rewrite the destination MAC (replacement for mod_dl_dst)
$ ovs-ofctl add-flow br0 \
    "priority=100,ip,nw_dst=10.1.0.1,\
     actions=set_field:aa:bb:cc:00:01:01->eth_dst,output:5"

# Rewrite the destination IP (DNAT semantics)
$ ovs-ofctl add-flow br0 \
    "priority=100,tcp,nw_dst=10.1.0.100,tp_dst=80,\
     actions=set_field:10.2.0.10->ipv4_dst,\
             set_field:8080->tcp_dst,\
             output:backend_port"

# Partial bit write with a mask (only the upper 3 MAC bytes, the OUI)
$ ovs-ofctl add-flow br0 \
    "priority=100,in_port=1,\
     actions=set_field:aa:bb:cc:00:00:00/ff:ff:ff:00:00:00->eth_src,\
             output:2"

# DSCP marking
$ ovs-ofctl add-flow br0 \
    "priority=100,ip,tcp,tp_dst=5001,\
     actions=set_field:46->ip_dscp,output:uplink"
```

### 4.9.17.3 `set_field` versus the legacy `mod_*`

| Aspect | `set_field` | `mod_<field>` |
|---|---|---|
| OpenFlow version | 1.2 and later | 1.0 legacy |
| Syntax | Unified | Per-field |
| Mask support | Yes | No |
| New field | Automatically supported | Must be added per new field |

Best practice: **use `set_field` for new code**. Legacy `mod_dl_src/dst`, `mod_nw_src/dst`, `mod_tp_src/dst` still work but do not scale to new fields (tun_metadata, ct_mark, registers, and so on); those fields have no dedicated `mod_*` and must use `set_field` or `load`.

### 4.9.18.1 The `mod_*` list (OF 1.0)

| Action | `set_field` equivalent |
|---|---|
| `mod_dl_src:<mac>` | `set_field:<mac>->eth_src` |
| `mod_dl_dst:<mac>` | `set_field:<mac>->eth_dst` |
| `mod_nw_src:<ip>` | `set_field:<ip>->ipv4_src` |
| `mod_nw_dst:<ip>` | `set_field:<ip>->ipv4_dst` |
| `mod_nw_tos:<dscp>` | `set_field:<tos>->ip_dscp` |
| `mod_nw_ecn:<ecn>` | `set_field:<ecn>->ip_ecn` |
| `mod_nw_ttl:<ttl>` | `set_field:<ttl>->nw_ttl` |
| `mod_tp_src:<port>` | `set_field:<port>->tcp_src` or `udp_src` |
| `mod_tp_dst:<port>` | `set_field:<port>->tcp_dst` or `udp_dst` |
| `mod_vlan_vid:<vid>` | `set_field:<vid>->vlan_vid` |
| `mod_vlan_pcp:<pcp>` | `set_field:<pcp>->vlan_pcp` |

Many existing flow rules (including those generated by OVN) still use `mod_*` for backward compatibility. Reading both forms is required.

### 4.9.18.2 `dec_ttl`, `dec_mpls_ttl`

| Attribute | Value |
|---|---|
| **Syntax** | `dec_ttl`, `dec_mpls_ttl` |
| **Category** | 3. Field modification |
| **OpenFlow version** | 1.0 and later (`dec_ttl`), 1.1 and later (`dec_mpls_ttl`) |
| **Prerequisites** | An IP packet with TTL greater than 0 for `dec_ttl` |
| **Semantics** | Decrement TTL or HL by one. When TTL becomes 0 after decrement, the **packet is sent to the controller via packet_in** (not auto-dropped); the controller decides what to do |
| **Side effects** | Updates TTL; recomputes the IP checksum; on TTL=0 emits OFPT_PACKET_IN with reason `OFPR_INVALID_TTL` |

**Router function using `dec_ttl`:**

```shell
# OVS as a virtual router
$ ovs-ofctl add-flow br0 \
    "priority=100,ip,nw_dst=10.2.0.0/24,\
     actions=dec_ttl,\
             mod_dl_src:aa:bb:cc:00:00:01,\
             mod_dl_dst:aa:bb:cc:00:02:01,\
             output:uplink"

# On a table miss, send an ICMP TTL exceeded through the controller
$ ovs-ofctl add-flow br0 \
    "priority=0,dl_type=0x0800,actions=controller(reason=invalid_ttl)"
```

### 4.9.18.3 `copy_ttl_in`, `copy_ttl_out`

MPLS/IP stacking: when pushing or popping an MPLS label, `copy_ttl_in` copies the outer TTL into the inner header, and `copy_ttl_out` copies it back. Used in MPLS VPN to preserve TTL across the tunnel.

## 4.9.19 Category 3, `move` plus `load`, register manipulation

### 4.9.19.1 `move`, copy field to field

| Attribute | Value |
|---|---|
| **Syntax** | `move:<src_field>[<start>..<end>]-><dst_field>[<start>..<end>]` |
| **Category** | 3. Field modification |
| **OpenFlow version** | NXM extension |
| **Semantics** | Copy bits from the source field to the destination field. The bit range is optional; when omitted the full field is copied |
| **Example** | `move:NXM_OF_ETH_SRC[]->NXM_OF_ETH_DST[]` copies eth_src to eth_dst |

### 4.9.19.2 `load`, copy a constant into a field

| Attribute | Value |
|---|---|
| **Syntax** | `load:<value>-><field>[<start>..<end>]` |
| **Category** | 3. Field modification |
| **OpenFlow version** | NXM extension |
| **Semantics** | Load a constant into a field (or a bit range) |
| **Example** | `load:0x42->NXM_NX_REG0[]` |

### 4.9.19.3 ARP responder use case (links to Part 4.8 §4.8.5)

```shell
$ ovs-ofctl add-flow br-int \
    "priority=200,arp,arp_op=1,arp_tpa=10.1.1.1,\
     actions=move:NXM_OF_ETH_SRC[]->NXM_OF_ETH_DST[],\
             mod_dl_src:aa:bb:cc:dd:ee:ff,\
             load:0x2->NXM_OF_ARP_OP[],\
             move:NXM_NX_ARP_SHA[]->NXM_NX_ARP_THA[],\
             move:NXM_OF_ARP_SPA[]->NXM_OF_ARP_TPA[],\
             load:0xaabbccddeeff->NXM_NX_ARP_SHA[],\
             load:0x0a010101->NXM_OF_ARP_SPA[],\
             IN_PORT"
```

Seven consecutive actions: `move` copies a field, `load` writes a constant, and `IN_PORT` outputs back. This is the classic way to implement an ARP responder in OVS without sending the packet up to the controller.

### 4.9.19.4 Bit-range manipulation

```shell
# Set only bits 0 to 3 of reg0 to 0xa
$ ovs-ofctl add-flow br0 \
    "priority=100,in_port=1,\
     actions=load:0xa->NXM_NX_REG0[0..3],resubmit(,1)"

# Move metadata bits 12 to 15 into reg1 bits 0 to 3
$ ovs-ofctl add-flow br0 \
    "priority=100,actions=move:OXM_OF_METADATA[12..15]->NXM_NX_REG1[0..3]"
```

Bit-range syntax is `[<start>..<end>]` or `[<bit>]`. Used extensively in OVN to split a register into subfields (for example, `reg0[0]` is the ACL hint flag, `reg0[1]` is the committed flag).

## 4.9.20 Category 4, metadata actions

### 4.9.20.1 `write_metadata` (an instruction, not an action)

Strictly this is an **instruction** (§4.9.1.1), not an action. OVS allows it in `actions=` as a shorthand. Syntax:

```
actions=write_metadata:<value>[/<mask>]
```

Scope: set the metadata field that travels through `Goto-Table`. Unlike registers (which are lost after recirc or an upcall boundary), metadata is preserved across the pipeline.

### 4.9.20.2 `set_tunnel`, `set_tunnel64`

| Attribute | `set_tunnel:<value>` | `set_tunnel64:<value>` |
|---|---|---|
| **Width** | 32-bit | 64-bit |
| **Semantics** | Set `tun_id` (the VNI for VXLAN or Geneve) | As above with 64-bit width |
| **Use case** | VXLAN 24-bit VNI encap | Generic 64-bit |

```shell
# Output the packet through a VXLAN tunnel with VNI 100
$ ovs-ofctl add-flow br0 \
    "priority=100,in_port=1,\
     actions=set_tunnel:100,output:vxlan0"
```

Modern flow form: use `set_field:<vni>->tun_id` instead of `set_tunnel`.

## 4.9.23 Category 5, `ct()`, conntrack integration

### 4.9.23.1 Template C for `ct()`

| Attribute | Value |
|---|---|
| **Syntax** | `ct(<option>[,<option>...])` |
| **Category** | 5. Firewall/CT |
| **OpenFlow version** | NXM extension (OVS) |
| **Prerequisites** | Kernel datapath with conntrack support (`dpif/show-dp-features: Conntrack eventmask: Yes`) |
| **Semantics** | Submit the packet to the kernel conntrack module. After conntrack processing, the packet recirculates back into the pipeline with `ct_state`, `ct_zone`, `ct_mark`, and `ct_label` populated |
| **Parameters** | `commit`, `zone=N`, `nat`, `force`, `alg=<ftp/tftp/sip/...>`, `exec(<actions>)`, `table=<N>` |
| **Side effects** | Substantial: creates a conntrack entry (commit), NAT rewrite (nat), triggers ALG helpers, recirculates the packet |

### 4.9.23.2 `ct()` options

**`commit`**: create an entry in the conntrack table:

```
actions=ct(commit,zone=1)
# Register this connection in kernel conntrack table zone 1
# Subsequent packets with the same five-tuple will match the conntrack entry
```

**`zone=N`**: conntrack zone ID (16-bit) that isolates the namespace between OVN logical switches. When unspecified, zone 0 is the default.

**`nat`**: apply NAT:

```
# Source NAT (SNAT)
actions=ct(commit,nat(src=10.1.0.1))

# Destination NAT (DNAT)
actions=ct(commit,nat(dst=10.2.0.1:8080))

# NAT with a port range
actions=ct(commit,nat(src=10.1.0.1:30000-40000))

# NAT persistence: reuse the same mapping for the same client
actions=ct(commit,nat(src=10.1.0.1,persistent))
```

**`force`**: re-commit and override the existing entry. Used when the existing connection is expected to be stale.

**`alg=<name>`**: application-layer gateway helper:

| ALG | Protocol |
|---|---|
| `alg=ftp` | FTP control channel (parses PORT and PASV to open the dynamic data channel) |
| `alg=tftp` | TFTP |
| `alg=sip` | SIP (VoIP signalling, opens the RTP media channel) |

**`exec(<actions>)`**: execute the action list **inside the conntrack zone context**. Used to set ct_mark or ct_label on commit:

```
actions=ct(commit,zone=1,exec(set_field:0xab->ct_mark))
```

**`table=N`**: after recirculation, look up table N (default is the current table).

### 4.9.23.3 Typical stateful firewall pattern

```shell
# Step 1: every packet goes through ct (to populate ct_state)
# Table 0, the dispatcher
$ ovs-ofctl add-flow br0 \
    "table=0,priority=100,ip,actions=ct(table=1,zone=1)"

# Table 1, policy based on ct_state
# Allow new outbound and commit
$ ovs-ofctl add-flow br0 \
    "table=1,priority=200,ip,ct_state=+new+trk,\
     actions=ct(commit,zone=1),output:uplink"

# Allow reply (established)
$ ovs-ofctl add-flow br0 \
    "table=1,priority=100,ip,ct_state=+est+trk,actions=output:inside"

# Drop invalid
$ ovs-ofctl add-flow br0 \
    "table=1,priority=100,ip,ct_state=+inv+trk,actions=drop"

# Default drop
$ ovs-ofctl add-flow br0 \
    "table=1,priority=0,actions=drop"
```

Packet flow:
1. The packet arrives at table 0, `ct(table=1)` submits it to kernel conntrack.
2. The kernel populates `ct_state` (new, est, rel, inv) and recirculates.
3. The packet reappears at table 1 with ct_state populated, matching the rule.
4. The rule with `ct(commit)` makes the conntrack entry persist.

### 4.9.23.4 `ct_clear`

| Attribute | Value |
|---|---|
| **Syntax** | `ct_clear` |
| **Semantics** | Clear the conntrack-tracked state for the current packet (reset `ct_state` to unset). The next `ct()` for this packet will re-evaluate |
| **Use case** | When the pipeline wants to re-track the connection (for example, after NAT, to reclassify) |

See Part 9.24 conntrack stateful firewall for the full case study.

## 4.9.24 Category 6, `learn()`, MAC learning through flow installation

### 4.9.24.1 Template C for `learn()`

| Attribute | Value |
|---|---|
| **Syntax** | `learn(<option>[,<option>...])` |
| **Category** | 6. Control/Pipeline |
| **OpenFlow version** | NXM extension |
| **Prerequisites** | None |
| **Semantics** | Install a new flow rule dynamically based on the current packet. This is how OVS implements L2 MAC learning without a controller; the flow is "self-programming" |
| **Parameters** | `table=N`, `priority=P`, `idle_timeout=T`, `hard_timeout=T`, `fin_idle_timeout=T`, `NXM_FIELD[]` match and action templates |
| **Side effects** | Creates a new flow rule in the kernel flow table; the `n_flow_mods` counter increases |

### 4.9.24.2 MAC learning example, the classic OVS pattern

```shell
# Learning rule: when the packet arrives, install the reverse flow
$ ovs-ofctl add-flow br0 \
    "table=0,priority=100,\
     actions=learn(table=1,\
                    idle_timeout=60,\
                    NXM_OF_ETH_DST[]=NXM_OF_ETH_SRC[],\
                    load:NXM_OF_IN_PORT[]->NXM_NX_REG0[]),\
             resubmit(,1)"

# Forwarding based on the learned entries
$ ovs-ofctl add-flow br0 \
    "table=1,priority=50,\
     actions=output:NXM_NX_REG0[]"

# Fallback: flood when there is no learn hit
$ ovs-ofctl add-flow br0 "table=1,priority=0,actions=flood"
```

**Learn-rule semantics:**

- `table=1`: install the new flow into table 1.
- `idle_timeout=60`: the flow expires after 60 seconds of inactivity.
- `NXM_OF_ETH_DST[]=NXM_OF_ETH_SRC[]`: the new match spec, "the destination MAC equals the source MAC of the current packet".
- `load:NXM_OF_IN_PORT[]->NXM_NX_REG0[]`: the action, "load the ingress port into reg0".

When a packet (src=MAC_A, in_port=1) reaches table 0, `learn` creates the flow:

```
table=1,priority=32768,idle_timeout=60,dl_dst=MAC_A,actions=load:1->NXM_NX_REG0[]
```

A subsequent packet with dst=MAC_A matches this flow, sets reg0=1, hits table 1 priority 50, and is sent to output:1.

### 4.9.24.3 `fin_idle_timeout`

A special TCP option: a different timeout when a FIN packet is observed (the connection is closing):

```
actions=learn(...,idle_timeout=300,fin_idle_timeout=10)
```

A normal flow expires after 300 seconds idle, but after a FIN is observed the timeout drops to 10 seconds.

### 4.9.24.4 Risk of `learn()`

- Each packet matching a rule with `learn()` creates a new flow, leading to flow explosion when misconfigured.
- `coverage/show upcall_flow_add` will be very high.
- Installing this on a production bridge requires careful rate limiting.

## 4.9.25 Category 6, `conjunction()`, cross-product compression

### 4.9.25.1 Template C

| Attribute | Value |
|---|---|
| **Syntax** | `conjunction(<id>,<component>/<total>)` |
| **Category** | 6. Control/Pipeline |
| **OpenFlow version** | NXM extension |
| **Prerequisites** | None |
| **Semantics** | Declare the rule as one "component" of a conjunction group `id`. The packet must match `<total>` distinct components of the same `id`; once `conj_id` is set, a different priority rule matches and executes the action |
| **Parameters** | `id`: 32-bit conjunction group ID; `component`: 1 to N, the component index; `total`: the total number of components required |

### 4.9.25.2 The cross-product problem

An ACL rule of the form "source IN {A, B, C} AND dest IN {X, Y, Z} AND port IN {80, 443}" requires **3 times 3 times 2 equals 18 flow rules** when written explicitly.

With conjunction it is **3 plus 3 plus 2 plus 1 equals 9 rules** (3 source components plus 3 dest components plus 2 port components plus 1 final action).

### 4.9.25.3 Full example

```shell
# Source components (conjunction 42, component 1/3)
$ for src in 10.0.1.1 10.0.1.2 10.0.1.3; do
    sudo ovs-ofctl add-flow br0 \
      "table=0,priority=100,ip,nw_src=$src,actions=conjunction(42,1/3)"
  done

# Destination components (42, component 2/3)
$ for dst in 10.0.2.1 10.0.2.2 10.0.2.3; do
    sudo ovs-ofctl add-flow br0 \
      "table=0,priority=100,ip,nw_dst=$dst,actions=conjunction(42,2/3)"
  done

# Port components (42, component 3/3)
$ for port in 80 443; do
    sudo ovs-ofctl add-flow br0 \
      "table=0,priority=100,ip,tcp,tp_dst=$port,actions=conjunction(42,3/3)"
  done

# Final rule matching conj_id (set when all three components match)
$ sudo ovs-ofctl add-flow br0 \
    "table=0,priority=100,conj_id=42,actions=output:5"
```

### 4.9.25.4 OVN Port_Group uses conjunction

OVN ACL on a Logical_Switch `Port_Group` uses conjunction extensively. For example, the policy "port_group WEB (10 ports) to port_group DB (5 ports) allow TCP 3306":

- 10 rules for WEB source (conj component 1/2).
- 5 rules for DB destination (conj component 2/2).
- 1 final rule matching `conj_id`.

Total 16 rules instead of 50 explicit rules. See Part 13.3 (OVN ACL plus Load Balancer) for the OVN context.

## 4.9.26 Category 6, `multipath()`, hash-based path selection

### 4.9.26.1 Template C

| Attribute | Value |
|---|---|
| **Syntax** | `multipath(<fields>,<basis>,<algorithm>,<n_links>,<arg>,<dst>[<start>..<end>])` |
| **Category** | 6. Control/Pipeline |
| **OpenFlow version** | NXM extension |
| **Semantics** | Hash the packet header fields to an output ID from 0 to (n_links - 1) and load it into the destination field. Similar to group type=select but without group-table configuration |
| **Parameters** | `fields`: `eth_src`, `symmetric_l4`, and so on; `algorithm`: `modulo_n`, `hash_threshold`, `hrw`, `iter_hash`; `n_links`: the number of outputs; `dst`: the target field |

### 4.9.26.2 ECMP use case

```shell
# Hash on the five-tuple, modulo 4, load 0 to 3 into reg0
$ ovs-ofctl add-flow br0 \
    "priority=100,ip,actions=multipath(symmetric_l4,0,modulo_n,4,0,NXM_NX_REG0[0..1]),\
     resubmit(,1)"

# Table 1 outputs based on reg0
$ ovs-ofctl add-flow br0 "table=1,reg0=0,actions=output:uplink1"
$ ovs-ofctl add-flow br0 "table=1,reg0=1,actions=output:uplink2"
$ ovs-ofctl add-flow br0 "table=1,reg0=2,actions=output:uplink3"
$ ovs-ofctl add-flow br0 "table=1,reg0=3,actions=output:uplink4"
```

Compared with `group type=select`:
- `multipath` is lightweight and does not require group-table maintenance.
- `group select` has monitoring (group stats) and centralised change management.

### 4.9.26.3 Hash algorithms

| Algorithm | Meaning |
|---|---|
| `modulo_n` | hash mod n_links; simple but rebalances heavily when n_links changes |
| `hash_threshold` | Threshold-based; balances better than modulo |
| `hrw` | Highest Random Weight (HRW); minimal rebalance when n_links changes |
| `iter_hash` | Iterative hash; complex, used for large n_links |

## 4.9.27 Category 6, `bundle()`, grouped action submission

### 4.9.27.1 Template C

| Attribute | Value |
|---|---|
| **Syntax** | `bundle(<fields>,<basis>,<algorithm>,<slave_type>,<slaves>)` |
| **Category** | 6. Control/Pipeline |
| **OpenFlow version** | NXM extension |
| **Semantics** | Select one output from a bundle of ports by hash. Similar to multipath but the output is direct (not loaded into a field) |

### 4.9.27.2 Example

```
actions=bundle(symmetric_l4,0,hrw,ofport,slaves:10,11,12,13)
```

Hash on the five-tuple, pick one of four ports (10 to 13), output the packet.

Compared with `multipath` plus `output:reg`: `bundle` is more compact when downstream rules do not need to differentiate the path.

### 4.9.27.3 `bundle_load()`

A variant that loads the result into a field (like multipath):

```
actions=bundle_load(symmetric_l4,0,hrw,ofport,NXM_NX_REG0[],slaves:10,11)
```

## 4.9.28 Category 5, `check_pkt_larger()`, MTU enforcement

### 4.9.28.1 Template C

| Attribute | Value |
|---|---|
| **Syntax** | `check_pkt_larger(<pkt_len>)-><dst_field>[<bit>]` |
| **Category** | 5. Firewall/CT (control flow based on size) |
| **OpenFlow version** | OVS 2.9 and later |
| **Semantics** | Compare the packet length to `pkt_len`; set the dst field bit to 1 when the packet is larger |

### 4.9.28.2 PMTUD use case

```shell
# When the packet is larger than 1400 bytes, set reg0[0] to 1

    "table=1,priority=50,actions=output:uplink"
```

## Core takeaways (tier 1, 2, and 3 complete)

First, an action is an instruction executed on the packet (transform, forward); an instruction is a wrapper that controls the pipeline. OpenFlow 1.0 merged the two; 1.1 and later separated them. The OVS `actions=` shortcut defaults to Apply-Actions.

Second, the Output category has 10 foundational actions: `output:<port>` plus the nine reserved-port variants (drop, normal, flood, all, controller, local, in_port, table) plus `group`. Each has its own semantics (whether to include the ingress port, whether to respect STP, whether to respect VLAN).

Third, explicit `drop` is better than implicit (an empty action list) because it is unambiguous. OVS 2.17 and later has a dedicated kernel drop path; verify with `dpif/show-dp-features`.

Fourth, the `group` table provides indirection for four patterns: multicast (all), load balance (select), shared next-hop (indirect), active-backup (fast_failover). A foundational OpenFlow 1.1 and later feature that many deployments have not used enough.

Fifth, `resubmit` is an OVS NXM extension for a modular pipeline and differs from the standard `goto_table`. OVN translation uses it extensively.

Sixth, action-set execution follows the **12 priority levels** strictly, not the order written. Apply-Actions, in contrast, is sequential. Understanding this distinction prevents surprises during debugging.

## 4.9.29 Backfill, eight-attribute anatomy completeness for 12 action gaps

> **Purpose.** A v3.5 audit found 12 actions in REF that did not yet have a full eight-attribute anatomy in the catalog (Template C). This section backfills them and follows the Template C defined in 4.9.1.

### 4.9.29.1 Action `copy_field` (OF 1.5 and later)

**Bucket:** Modification.
**Syntax:** `copy_field(src_offset:src_field, dst_offset:dst_field, n_bits)` or the shorthand `copy_field:SRC->DST` (full-width copy).
**Effect:** Copy `n_bits` bits from `src_field[src_offset:]` to `dst_field[dst_offset:]`. The source field is unchanged.
**Prerequisites:** `src_field` and `dst_field` must be valid.
**OF version introduced:** OF 1.5 and later (action OFPAT_COPY_FIELD type 28).
**OVS version introduced:** OVS 2.7 and later.
**Side effect:** Modifies the destination field.
**Example:**

```
priority=100,ip,actions=copy_field:ipv4_src->reg0,goto_table:5
priority=100,actions=copy_field(0:reg0,0:vlan_vid,16)
```

**Operational notes.** OF 1.0 to 1.4 used the OVS extension `move:SRC[]->DST[]` to copy a field, but `move` is NXM-only and is not portable across vendors. OF 1.5 standardised `copy_field` with the same semantics. OVN can use `copy_field` instead of `move` for portable code. Difference from `set_field` (which writes a literal value): `copy_field` writes a value from another field (dynamic).

### 4.9.29.2 Actions `push_pbb` and `pop_pbb` (OF 1.3 and later)

**Bucket:** L2 Provider Bridge encapsulation.
**Syntax:** `push_pbb:0x88e7` and `pop_pbb`.
**Effect:** Add or remove a 22-byte PBB shim header before the customer Ethernet header.
**Prerequisites:** `pop_pbb` requires `eth_type=0x88e7`.
**OF version introduced:** OF 1.3 and later (OFPAT_PUSH_PBB type 26 plus OFPAT_POP_PBB type 27).
**OVS version introduced:** OVS 2.6 and later.
**Side effect:** `push_pbb` sets the new B-DA equal to the current `eth_dst`, B-SA equal to the current `eth_src`, and I-SID to 0.
**Example:**

```
actions=push_pbb:0x88e7,set_field:0x123456->pbb_isid,output:2
priority=100,eth_type=0x88e7,actions=pop_pbb,goto_table:5
```

**Operational notes.** PBB (IEEE 802.1ah) is common in carrier Ethernet provider deployments. OVS supports PBB primarily for compatibility with carrier hardware switches. Not used in data centre overlays (OVN uses Geneve or VXLAN).

### 4.9.29.3 Action `set_mpls_ttl`

**Bucket:** L2 MPLS modification.
**Syntax:** `set_mpls_ttl:N` (N from 0 to 255).
**Effect:** Override the MPLS TTL field to N.
**Prerequisites:** MPLS (`eth_type=0x8847` or `0x8848`).
**OF version introduced:** OF 1.1 and later (OFPAT_SET_MPLS_TTL type 15).
**OVS version introduced:** OVS 1.11 and later.
**Side effect:** Modifies the MPLS shim-header byte 4 (TTL position).
**Example:** `priority=100,eth_type=0x8847,actions=set_mpls_ttl:64,output:2`

**Operational notes.** When an MPLS edge router pushes a new label, the outer label's TTL by default inherits from the inner label. `set_mpls_ttl` lets the edge router refresh the TTL to a high value before forwarding, avoiding packet drops at a core router as the TTL counts down to zero.

### 4.9.29.4 Action `dec_mpls_ttl`

**Bucket:** L2 MPLS modification.
**Syntax:** `dec_mpls_ttl`.
**Effect:** Decrement the outermost MPLS label TTL by 1. Drop the packet when TTL reaches 0.
**Prerequisites:** MPLS.
**OF version introduced:** OF 1.1 and later (OFPAT_DEC_MPLS_TTL type 16).
**OVS version introduced:** OVS 1.11 and later.
**Side effect:** TTL byte modification plus a potential drop on zero.
**Example:** `priority=100,eth_type=0x8847,actions=dec_mpls_ttl,output:2`

**Operational notes.** Every MPLS transit hop must decrement TTL (the same as IP TTL) to prevent infinite loops.

### 4.9.29.5 Actions `copy_ttl_in` and `copy_ttl_out`

**Bucket:** L2/L3 TTL preservation.
**Syntax:** `copy_ttl_in` and `copy_ttl_out`.
**Effect:**
- `copy_ttl_in`: copy TTL from the outer header (outer MPLS or IP) into the inner header (inner IP). Used on encapsulation pop.
- `copy_ttl_out`: copy TTL from the inner header (IP) into the outer header (MPLS). Used on encapsulation push.
**Prerequisites:** At least two encapsulation layers.
**OF version introduced:** OF 1.1 and later (OFPAT_COPY_TTL_IN type 11, OFPAT_COPY_TTL_OUT type 12).
**OVS version introduced:** OVS 2.5 and later.
**Side effect:** Modifies the TTL byte of the corresponding header.
**Example:**

```
priority=100,eth_type=0x8847,mpls_bos=1,actions=copy_ttl_in,pop_mpls:0x0800,goto_table:5
priority=100,ip,actions=copy_ttl_out,push_mpls:0x8847,set_field:100->mpls_label,output:2
```

**Operational notes.** These two actions resolve the **TTL behaviour** trade-off in MPLS: pipe mode (default, hides the topology) versus uniform mode (`copy_ttl_*` enabled, traceroute sees the full path).

### 4.9.29.6 Action `set_nw_ttl`

**Bucket:** L3 modification.
**Syntax:** `set_nw_ttl:N` (N from 0 to 255).
**Effect:** Override the IPv4 TTL or IPv6 Hop Limit to N.
**Prerequisites:** IPv4 or IPv6.
**OF version introduced:** OF 1.1 and later (OFPAT_SET_NW_TTL type 23).
**OVS version introduced:** OVS 1.10 and later.
**Side effect:** Modifies the IP header TTL or Hop Limit.
**Example:** `priority=100,ip,actions=set_nw_ttl:64,output:2`

**Operational notes.** Unlike `dec_ttl` (decrement by one, the standard router behaviour), `set_nw_ttl:N` overwrites the TTL with N. Used only for special cases such as NAT64 or a gateway re-stamping TTL.

### 4.9.29.7 Action `decap` (NSH detail)

**Bucket:** Encapsulation.
**Syntax:** `decap` or `decap(packet_type=(NS,TYPE))`.
**Effect:** Remove the innermost encapsulation header (NSH, MPLS, GRE, Geneve, and similar) and update the `packet_type` field.
**Prerequisites:** The packet must have an encapsulation layer present.
**OF version introduced:** OVS extension (OF 1.5 and later, packet-type aware).
**OVS version introduced:** OVS 2.8 and later (NSH support).
**Side effect:** Modifies the `packet_type` field. The outer header data is lost.
**Example:**

```
priority=100,nsh,nsh_si=0,actions=decap(),decap(packet_type=(0,0x0800)),output:2
```

**Operational notes.** `decap` is the core action for Service Function Chaining (SFC). The final service function in the chain (SI=0) calls `decap()` twice: once to remove the NSH header, and once to revert to Ethernet for output.

### 4.9.29.8 Action `encap` (NSH detail)

**Bucket:** Encapsulation.
**Syntax:** `encap(NSH)`, `encap(ethernet)`, `encap(MPLS, mpls_label=N)`.
**Effect:** Add an encapsulation header on top of the current packet and update `packet_type`.
**Prerequisites:** The current packet must be compatible with the encap's inner type.
**OF version introduced:** OVS extension.
**OVS version introduced:** OVS 2.8 and later.
**Side effect:** Adds a header and changes `packet_type`.
**Example:**

```
priority=100,ip,actions=encap(nsh(md_type=1,tlv(class=0x1,type=1,len=4,value=0xdead))),
        set_field:0x123->nsh_spi,set_field:255->nsh_si,output:2
```

**Operational notes.** `encap` is the counterpart of `decap`. The SFC entry router emits `encap(nsh)` to wrap traffic into the chain. NSH metadata can use MD Type 1 (fixed length 16 bytes) or Type 2 (variable TLV).

### 4.9.29.9 Action `controller` with userdata (OVS 2.6 and later, continuation-based)

**Bucket:** Control.
**Syntax:** `controller(reason=action,userdata=HEX)` or `controller(reason=action,userdata=HEX,pause)`.
**Effect:** Send the packet to the controller with custom metadata; optionally pause the pipeline and resume after the controller responds.
**Prerequisites:** An OpenFlow channel connected to the controller.
**OF version introduced:** OF 1.0 and later (basic), OVS 2.6 and later (continuation extension).
**OVS version introduced:** OVS 2.6 and later.
**Side effect:** Trigger PACKET_IN message + optional pipeline pause.
**Example:**

```
priority=100,actions=controller(reason=action,userdata=0xcafe1234)
priority=100,actions=controller(reason=action,userdata=0xdead,pause)
```

**Operational notes.** The continuation-based extension lets the controller respond with a PACKET_OUT carrying a "continue" action to resume the paused pipeline. OVN does not use this pattern; mostly `controller` is used as an ARP responder fallback.

### 4.9.29.10 Action `note` (debug marker)

**Bucket:** Annotation.
**Syntax:** `note:HEXSTRING` (inline hex bytes).
**Effect:** Semantically a no-op. Embeds bytes into the flow definition for debug or audit.
**Prerequisites:** None.
**OF version introduced:** OVS extension (NXAST_NOTE).
**OVS version introduced:** OVS 1.0 and later.
**Side effect:** None. Pure debug marker.
**Example:** `priority=100,in_port=1,actions=note:01.02.03.04.deadbeef,output:2`

**Operational notes.** A technique for embedding bytes into a flow rule for post-mortem debugging. When `dump-flows` shows the rule, the note hex is visible and the operator decodes it back into metadata.

### 4.9.29.11 Action `sample` (OVS 2.5 and later)

**Bucket:** Monitoring.
**Syntax:** `sample(probability=N,collector_set_id=ID,obs_domain_id=ID,obs_point_id=ID)`.
**Effect:** Per-packet probability sampling. With probability N out of 65535, emit an IPFIX record.
**Prerequisites:** The IPFIX `collector_set` is configured through OVSDB.
**OF version introduced:** OVS extension (NXAST_SAMPLE).
**OVS version introduced:** OVS 2.5 and later.
**Side effect:** Triggers IPFIX export (no packet modification).
**Example:**

```
priority=100,in_port=1,actions=sample(probability=655,collector_set_id=1,obs_domain_id=42,obs_point_id=100),output:2
```

**Operational notes.** Built-in flow sampling for IPFIX or sFlow telemetry. The probability granularity is 1 in 65,535 (1 percent equals about 655). Use case: NetFlow collection for traffic analysis. Difference from a sFlow agent (which samples every packet): the `sample` action is embedded in the flow rule (selective per flow).

### 4.9.29.12 Action `conjunction` (OVS 2.4 and later), full anatomy

**Bucket:** Matching.
**Syntax:** `conjunction(id, k/n)` where k is the clause index (1-based) and n is the total clause count.
**Effect:** Implements an n-way conjunctive (AND) match through flow expansion.
**Prerequisites:** None.
**OF version introduced:** OVS extension (NXAST_CONJUNCTION).
**OVS version introduced:** OVS 2.4 and later.
**Side effect:** Triggers a downstream conjunctive match (a rule with `conj_id=id`).
**Example:**

```
priority=100,ip,nw_src=10.0.0.0/8,actions=conjunction(1,1/2)
priority=100,tcp,tp_dst=80,actions=conjunction(1,2/2)
priority=100,tcp,tp_dst=443,actions=conjunction(1,2/2)
priority=200,conj_id=1,actions=output:2
```

**Operational notes.** The problem: an ACL "src in {N IPs} AND dst in {M ports}" expanded as a Cartesian product equals N times M rules. Conjunction reduces this to N plus M rules plus one verdict rule.

OVN uses conjunction heavily for ACLs that reference Address_Set plus Port_Group with many IPs or ports. An ACL "from-lport: ip4.src in $set_a (100 IPs) && ip4.dst in $set_b (50 IPs)" equals 100 times 50 equals 5,000 rules as a Cartesian product. With conjunction it becomes 100 plus 50 plus 1 equals 151 rules, a 33-fold reduction.

## 4.9.31 Backfill, NXM Nicira stack plus timeout actions

> **Context.** A v3.6 audit found three Nicira extension actions in REF that the curriculum had not yet indexed: `fin_timeout`, `push:src`, `pop:dst`. All three are OVS 1.11 and later, and all three remain commonly used in OpenFlow rules with register manipulation. This section adds the eight-attribute Template C anatomy for each plus a use case and cross-links.
>
> **Why these three actions matter even though they are not tier 1.** `fin_timeout` is the cheapest way to clean up a TCP flow that has FINed without paying the conntrack overhead. The pair `push:` and `pop:` is the only stack semantics in NXM; every "save-modify-restore" pattern for a temporary field (such as swapping `in_port` to re-execute the pipeline with a fake input) depends on it. Skipping these three actions skips an entire idea class (per-packet stack) that is fundamentally different from the rest (write-once).

### 4.9.31.1 Action `fin_timeout` (NXAST_FIN_TIMEOUT, OVS 1.11 and later)

**Bucket:** Connection tracking (timeout shrink).
**Syntax:** `fin_timeout(idle_timeout=N, hard_timeout=M)` with N and M in seconds. The parameters are optional; either may be passed alone.
**Effect:** When a packet matches this flow entry with the TCP FIN or RST flag set, OVS shrinks the entry's `idle_timeout` and `hard_timeout` to N and M seconds respectively (only shorter, never longer). Goal: clean up the flow faster after the TCP connection has closed, instead of waiting for the default idle timeout (typically 300 seconds).
**Prerequisites:** The flow entry must match TCP packets (`tcp` or `nw_proto=6`). The action only matters for flows with timeout > 0; permanent flows (timeout=0) are unaffected.
**OF version introduced:** OVS extension (NXAST_FIN_TIMEOUT, not in standard OpenFlow).
**OVS version introduced:** OVS 1.11 and later.
**Side effect:** Updates the flow entry's metadata (through `ofproto-dpif`); does not generate any outgoing packet. No interaction with conntrack zones; this is pure OpenFlow flow lifecycle.
**Example:**

```
priority=200,tcp,tcp_flags=+fin,actions=fin_timeout(idle_timeout=5,hard_timeout=30),resubmit(,1)
priority=200,tcp,tcp_flags=+rst,actions=fin_timeout(idle_timeout=1,hard_timeout=5),resubmit(,1)
```

**Operational notes.** Long-lived TCP flows in a data centre can keep flow entries in the table for hours because the default idle timeout is high. After the peer sends FIN or RST, the connection is actually dead but the entry still occupies a slot. `fin_timeout` lets the operator declare "when a FIN is observed, age out aggressively in another 5 seconds". Compared with conntrack `ct(commit, exec(fin_timeout=5))`: the standalone `fin_timeout` action does not need a conntrack state machine and is simpler for a stateless ACL with a little awareness of connection close. Difference from `learn(... fin_idle_timeout=...)`: `learn` creates a new flow entry in another table, whereas `fin_timeout` operates on the current flow entry. Cross-link: 9.24 §9.24.x for the conntrack-based approach; 4.7 §4.7.X for the learn-from-flow approach.

### 4.9.31.2 Action `push:src` (NXAST_STACK_PUSH, OVS 1.11 and later)

**Bucket:** Field stack (LIFO save).
**Syntax:** `push:FIELD` or `push:FIELD[bit_lo..bit_hi]` where FIELD is an NXM or OXM identifier (`NXM_OF_IN_PORT[]`, `NXM_NX_REG0[0..15]`, `eth_src[]`, and so on).
**Effect:** Push the current value of `FIELD` (or a bit slice) onto the OVS internal per-packet stack. The stack lives in the flow execution context; lifetime is bounded to one packet being processed.
**Prerequisites:** The stack has a maximum depth (lib/ofp-actions.c limits to 64 entries per packet). A push above the limit produces an action error.
**OF version introduced:** OVS extension (NXAST_STACK_PUSH, not in standard OpenFlow).
**OVS version introduced:** OVS 1.11 and later.
**Side effect:** The stack state changes (one entry pushed); the source field keeps its value. The stack resets between packets.
**Example:**

```
table=0,priority=100,in_port=1,actions=push:NXM_OF_IN_PORT[],load:0->NXM_OF_IN_PORT[],resubmit(,1),pop:NXM_OF_IN_PORT[],output:2
```

**Operational notes.** A typical "save-modify-restore" pattern: save `in_port` onto the stack, set `in_port = 0` (reset), resubmit to another table as if the packet had arrived on port 0, then restore `in_port` to its original value before output. Useful when a multi-table pipeline simulates "the packet came from a different port" for testing or to bypass an `in_port` match downstream. Compared with `move:` (copy the value to a temporary register and copy back): `push:` and `pop:` do not consume a register slot and do not collide with other register usage in the pipeline. Compared with `set_field:`, `push:` and `pop:` preserve the original value through the stack rather than through a manual copy.

### 4.9.31.3 Action `pop:dst` (NXAST_STACK_POP, OVS 1.11 and later)

**Bucket:** Field stack (LIFO restore).
**Syntax:** `pop:FIELD` or `pop:FIELD[bit_lo..bit_hi]`.
**Effect:** Take the entry at the top of the stack, assign its value to `FIELD` (or a bit slice), then remove the entry. The popped entry's bit width must match the destination field's bit width, otherwise an action error.
**Prerequisites:** The stack must have at least one entry. Popping an empty stack produces an action error (the packet is dropped).
**OF version introduced:** OVS extension (NXAST_STACK_POP).
**OVS version introduced:** OVS 1.11 and later.
**Side effect:** The stack state changes (one entry popped); the destination `FIELD` is overwritten.
**Example:**

```
priority=100,tcp,actions=push:NXM_NX_REG0[],load:0xdeadbeef->NXM_NX_REG0[],note:01.02.03,pop:NXM_NX_REG0[],output:2
```

**Operational notes.** Use case: the pipeline needs a temporary register for an intermediate computation (such as a hash) but does not want to destroy the original reg0 value that downstream rules still depend on. Push reg0, use reg0 for the computation, pop reg0 before output. A pop without a paired push (mismatched stack) is a typical programming error; OVS ofproto/trace reports `stack underflow`. Anti-pattern: using `pop:` instead of `move:` when there is no preceding push. Cross-link: 9.25 §9.25.X `ofproto/trace` for debugging stack-related anomalies.

## 4.9.32 Deeper treatment of the `output` action

> **Purpose.** The `output` action is already covered substantively in §4.9.1 Category 1. This section adds depth on importance, version differences, real-world incident, common failures, and comparison with other systems.

### Importance

`output` is the **absolute cornerstone** of the OpenFlow action set. Without `output`, the packet never leaves the switch; the flow rule only classifies and modifies, and the packet eventually drops. Every forwarding rule must end with `output` (or with a `resubmit`/`goto_table` chain that ends in an `output`). This action is on the same foundational tier as megaflow and ct_state.

### Version differences

| Version | Behaviour |
|---------|-----------|
| OF 1.0 | Basic `output:port_no` |
| OF 1.0 | Reserved ports: OFPP_FLOOD, OFPP_ALL, OFPP_CONTROLLER, OFPP_LOCAL, OFPP_NORMAL, OFPP_IN_PORT |
| OF 1.1 | Multi-table awareness (output emits from the current table or from accumulated state) |
| OF 1.3 | Group-table reference (output to a group instead of a port) |
| OF 1.5 | Egress-table support (post-output egress processing) |
| OVS extension | `output:NXM` (dynamic port from a register), `output:CONTROLLER:max_len` |

### Real-world incident

Production cluster, 2024-Q2: TCP traffic asymmetric routing, with intermittent return-path drops. Investigation: ofproto/trace showed the packet matched correctly, but `actions=output:5` was evaluated when port 5 was transiently down. OVS does not auto-reroute on a port-down event; it silently drops.

**Root cause:** static `output:port` without a group-bucket fallback.

**Fix:** migrate to a group table with a fast-failover bucket. `actions=group:1` where group 1 has two buckets (port 5 primary, port 6 backup). When port 5 goes down, the group automatically shifts to port 6.

### Common failures

- Mode A: output to a non-existent port produces a silent drop. Diagnostic: ofproto/trace shows the port number is invalid. Recovery: validate the port_no before installing the flow.
- Mode B: output to OFPP_IN_PORT can cause an infinite loop when the in_port is also the output target. Recovery: choose an explicit different output port.
- Mode C: output to the NORMAL port when L2 learning is incomplete causes a flood instead. Recovery: prime the MAC table.
- Mode D: output to a physical port with an MTU mismatch causes frame fragmentation failure. Recovery: configure a consistent MTU.

### Comparison with other systems

- **Cisco IOS interface forwarding**: `interface GigabitEthernet0/1` is a similar concept at a lower level. OF output is a platform-neutral abstraction.
- **Linux bridge `bridge fdb add`**: MAC-table-based forwarding. OF output is rule-based per flow and more flexible.
- **DPDK rte_eth_tx_burst**: transmits a packet to a physical port. OF output is a logical abstraction; rte_eth is direct hardware.
- **VPP interface output**: vector packet processing with a similar concept. Difference: VPP optimises batch dispatch; OVS is per-packet.

## 4.9.33 In-depth treatment of mid-tier packet-modification actions

This section is dedicated to five packet-modification actions that appear in nearly every production flow rule: `drop` (terminate processing), `set_field` (general field overwrite), `mod_dl_src` plus `mod_dl_dst` (legacy OF 1.0 MAC overwrites), and `dec_ttl` (decrement IP header TTL). Each action has its own subsection below with a 20-axis treatment under natural English headings: concept, history, position in the architecture, role, why it exists, problems solved, importance, mechanism, wire encoding, applied examples, how an engineer masters operations, classification, source-code anchor, failure scenarios, synthetic exercise, and comparison with other systems.

These five actions belong to the "category 3 field modification" group in the `ovs-actions(7)` taxonomy, except for `drop`, which belongs to category 1 Output (terminate). They are grouped together because all five either modify the packet header in place or terminate the pipeline; engineers commonly write them adjacent to one another in a single flow, for example a router rule `actions=dec_ttl,mod_dl_src:...,mod_dl_dst:...,output:N`.

### 4.9.33.1 `drop`, terminate packet processing without output or modification

**Concept.** `drop` is a terminator action: the matched packet is discarded after the rule increments the `n_packets` and `n_bytes` counters. The original OpenFlow spec did not have a dedicated `drop` action; the convention was "an empty action list equals an implicit drop". OVS added explicit `drop` syntax to make the rule clearer to the reader; OF 1.5 spec §5.10 standardised explicit `drop` as required behaviour. Action type: **terminator**, cannot be paired with another action in the same action list.

**History and context.** OpenFlow 1.0 (2009): an empty action list equals drop. OVS from its first release (2012 onward) accepted the `drop` keyword in the parser but internally translated it to an empty action list. OpenFlow 1.5 (2014): explicit `drop` became the standard to make rules clearer (section 5.10 "Action Semantics"). OVS 2.17 (2022): added `OVS_ACTION_ATTR_DROP` to the datapath kernel netlink protocol, giving the kernel a dedicated fast path for drop instead of traversing an empty action list.

**Position in the architecture.** At the OpenFlow protocol layer, `drop` has no dedicated raw enum entry in `enum ofp_raw_action_type` (verified in `lib/ofp-actions.c` on branch v3.3.0 of OVS; the parser at line 9431 with `else if (!strcasecmp(key, "drop"))` only sets a flag `drop = true`). At the datapath kernel layer, the netlink action `OVS_ACTION_ATTR_DROP` (constant 22, OVS 2.17 and later) is a dedicated attribute. At the userspace datapath layer, this maps to a no-op execute path.

**Role.** Primary role: a deny rule in an ACL (drop traffic that violates policy). Secondary role: terminate after a check_pkt_larger fail, terminate when conntrack marks the packet `INVALID`, terminate in a table-miss when the operator wants to be explicit instead of relying on default behaviour. Blast radius: per-rule, does not propagate to other actions (drop is the only action on its list).

**Why it exists.** A rule with an empty action list confuses readers; you cannot tell whether the author intended to drop or simply forgot to fill the action. The explicit `drop` keyword improves readability. After OVS 2.17 added the dedicated datapath action, a second reason emerged: performance. The drop fast path skips the empty action-list traversal and avoids allocating format strings for logs; in a DDoS scenario, the switch can drop millions of packets per second and every saved cycle matters.

**Problems it solves.** Specifically: (a) make the rule clearer for an operator review, and (b) increase datapath efficiency when the drop ratio is high. Approach: the parser recognises the `drop` token and the kernel datapath has a dedicated attribute. Scope: every deny flow rule.

**Importance.** Tier 2 medium, but extremely high frequency in production rules. Every ACL deny rule, every anti-spoofing rule, every conntrack INVALID handler uses `drop`. Engineer must-know: distinguish explicit `drop` from implicit (empty action list) and implicit (table-miss with no default rule), and remember that `drop` cannot be combined with another action (the parser rejects with `"drop" must not be accompanied by any other action`).

**How it works.** When the parser encounters the `drop` keyword, it sets the local flag `drop = true` without emitting an ofpact entry into the ofpacts buffer (`lib/ofp-actions.c` line 9431, verified). After parsing the entire action string, the parser checks: if `drop && pp->ofpacts->size > 0`, return an error string. When only `drop` is present, the ofpacts buffer remains empty, equivalent to an empty action list. Downstream translation (xlate.c) sees the empty action list and emits `OVS_ACTION_ATTR_DROP` to the datapath (kernel 2.17 and later) or emits no action at all (older kernels). The kernel datapath sees the DROP attribute, frees the skb early, and returns.

**Wire encoding.** No dedicated wire format for `drop` at the OpenFlow protocol level; an empty action list (length=0) is the implicit wire encoding. The datapath kernel attribute `OVS_ACTION_ATTR_DROP` is an nlattr with type=22 and length=4 bytes (header only, no payload). This is an extension, not part of the OpenFlow spec.

**Applied examples.**

```
priority=100,ip,nw_dst=10.0.0.99,actions=drop                    # explicit drop
priority=100,ip,nw_dst=10.0.0.99,actions=                        # implicit (empty)
priority=10,ct_state=+inv+trk,actions=drop                       # drop INVALID conntrack
priority=200,arp,arp_op=1,arp_spa=0.0.0.0,actions=drop           # drop bogus ARP
priority=0,actions=drop                                          # explicit table-miss
```

**How an engineer masters operations.** Four skills: (1) **Read**: distinguish a rule with `drop` (deny) from a rule with empty `actions=` (same semantics but easily misread as unfinished). (2) **Edit**: use explicit `drop` for every new rule; gain readability with no cost. (3) **Observe**: `ovs-ofctl dump-flows br-int | grep -E 'actions=(drop|$)'` counts the active deny rules. (4) **Performance check**: `ovs-appctl dpif/show-dp-features br0 | grep -i drop` showing `Explicit Drop action: Yes` confirms the kernel (2.17 and later) supports fast drop.

**Classification.** A terminator action (ends the pipeline). Property: cannot be paired with another action. Same class: implicit drop (empty action list); `controller(reason=...)` is not a terminator because it can be combined with other actions. Different from `drop` in conntrack: `ct(commit)` does not drop even when `ct_state` is `INVALID`; the operator must add `drop` explicitly.

**Source code.** `lib/ofp-actions.c` function anchor for the parser: the branch `else if (!strcasecmp(key, "drop"))` at line 9431 sets `drop = true`; the post-loop check `if (drop && pp->ofpacts->size)` returns an error. Datapath attribute: `include/uapi/linux/openvswitch.h` enum `ovs_action_attr` has `OVS_ACTION_ATTR_DROP = 22` (kernel 5.17 and later, OVS module 2.17 and later). Branch baseline: OVS v2.17.9.

**Failure scenarios.** (a) Writing `actions=drop,output:5` is rejected by the parser with `"drop" must not be accompanied by any other action or instruction`. Fix: split into two rules with distinct priorities. (b) A rule with an empty action list that did not intend to drop but forgot to fill the action behaves like `drop`, easy to debug-trap if you forget the "empty equals drop" rule. Fix: always use explicit `drop` so the intent is clear. (c) The `n_packets` counter on the drop rule rises quickly but traffic still passes; another higher-priority rule matches first; verify with `ofproto/trace`.

**Synthetic exercise.** Mininet, one OVS bridge, two hosts: install rule `priority=100,ip,nw_dst=h2,actions=drop` plus a fallback rule `priority=0,actions=normal`. Verify `h1 ping h2` fails and the drop-rule counter rises. Then remove the drop rule; the ping passes.

**Observation and verification.** `ovs-ofctl dump-flows br0 -O OpenFlow15` displays the rule with the explicit `drop` action (OF 1.5 and later format). `ovs-appctl ofproto/trace br0 'in_port=1,ip,nw_dst=10.0.0.99'` shows the line `Datapath actions: drop`, confirming that the kernel will free the skb. `ovs-appctl coverage/show | grep dropped` counts cumulative dropped packets.

**Comparison with other systems.** Linux netfilter `iptables -j DROP` is semantically equivalent: terminate the rule and free the packet. Difference: netfilter also has `REJECT` (sends an ICMP unreachable response); OpenFlow has no built-in equivalent (must use `controller(reason=...)` plus a controller emitting the ICMP separately). Cisco IOS ACL `deny` is equivalent to implicit drop. P4 `drop()` extern function has the same semantics. NSX-T DFW rule action `Drop` is the same concept.

### 4.9.33.2 `set_field`, the standard generic field-rewrite action of OF 1.2 and later

**Concept.** `set_field` is an action that overwrites a new value into any field of the packet header (or metadata, or register), with optional bitmask support for partial writes. This is the standard replacement for the entire legacy `mod_*` family of OF 1.0: `mod_dl_src`, `mod_dl_dst`, `mod_nw_src`, `mod_nw_dst`, `mod_tp_src`, `mod_tp_dst`, `mod_nw_ttl`, `mod_vlan_vid`, `mod_vlan_pcp`, and so on; all are folded into a single action with unified syntax. Action type: **generic field rewrite**, not a terminator.

**History and context.** OpenFlow 1.0 (2009) defined about 10 dedicated `OFPAT10_SET_*` actions, one per field. The "one action per field" design did not scale: every time the spec added a new field (MPLS label, IPv6 flow label, conntrack metadata, register, tunnel metadata), a new action had to be defined. OpenFlow 1.2 (2011) refactored: a single action `OFPAT12_SET_FIELD` (OXM-based) was added, allowing any field with an OXM identifier to be set. OVS 2.0 (2014) officially exposed the `set_field` syntax to `ovs-ofctl`. OF 1.5 (2014) removed the redundant `OFPAT_REG_LOAD`, keeping only `OFPAT_SET_FIELD` (verify `lib/ofp-actions.c` lines 3164-3187, function `encode_SET_FIELD`).

**Position in the architecture.** At the OpenFlow protocol layer: `OFPAT12_SET_FIELD` is action type 25 (OF 1.2); the action header has `type` (2 bytes) plus `length` (2 bytes) plus an OXM TLV inline. At the OVS internal ofpact layer: `struct ofpact_set_field` (`include/openvswitch/ofp-actions.h` line 543) holds a pointer to an `mf_field` descriptor plus a value and mask buffer. At the datapath kernel layer: translates to `OVS_ACTION_ATTR_SET` or `OVS_ACTION_ATTR_SET_MASKED`, depending on whether a mask is present.

**Role.** Primary role: rewrite header bytes (NAT, MAC redirect, VLAN tag, DSCP marking, tunnel VNI). Secondary role: write metadata or register values (set_field:0x42->reg0, set_field:1->ct_mark) as intermediate state for the pipeline. Blast radius: per-packet; the effect can propagate through recirc when the field is persistent metadata.

**Why it exists.** The OF 1.0 `mod_*` family does not scale to new fields. Every extension (NXM register, conntrack mark, tunnel metadata) would otherwise require adding `mod_reg0`, `mod_reg1`, ..., `mod_ct_mark`, `mod_tun_id`, and so on, a combinatorial explosion. `set_field` with an inline OXM TLV allows dynamic dispatch by `mf_field` at runtime.

**Problems it solves.** Specifically: (a) extensibility for new fields without changing the protocol structure, (b) partial writes via bitmask support (for example, change only the three OUI bytes of a MAC and keep the remaining three), (c) unified syntax that is easier to learn. Approach: the action format embeds an OXM TLV inline, and the `mf_field` descriptor handles dispatch. Scope: every field that has an OXM identifier, including every standard field and every NXM extension.

**Importance.** Tier 2 medium with high frequency. Engineer must-know: the syntax `set_field:<value>[/<mask>]-><field>`, the mask semantics (a 1-bit overwrites, a 0-bit preserves), and when to use `set_field` instead of legacy `mod_*` (always pick `set_field` for new code; best practice §4.9.17.3).

**How it works.** The parser `lib/ofp-actions.c` function `parse_SET_FIELD` splits the argument `<value>/<mask>-><field>` into three parts via the helper `set_field_split_str` (line ~3210). Resolve `<field>` to a `struct mf_field *` via `mf_from_name`. Validate the value width matches the field width. Build a `struct ofpact_set_field` with value plus mask buffer. On encode (`encode_SET_FIELD` line 3164): on OF 1.5 and later use `set_field_to_set_field`; with an all-ones mask on OF 1.2-1.4 use `set_field_to_set_field` (entire-field write); with a partial mask use `set_field_to_nxast` (NXAST_RAW_REG_LOAD format). Datapath kernel: performs the `skb` byte rewrite plus recomputes the L3 or L4 checksum for the corresponding field.

**Wire encoding.** OF 1.2-1.4 format: action header `type=25` (SET_FIELD) plus `length` (variable, padded to 8 bytes) plus an inline OXM TLV. OXM TLV: `oxm_class` (2 bytes), `oxm_field|hasmask` (1 byte), `oxm_length` (1 byte), value (plus mask when hasmask=1), padding. OF 1.5 and later keeps the same structure with different semantics: the redundant `OFPAT_REG_LOAD` is removed. NXAST format (legacy fallback): `vendor=0x2320` (Nicira) plus `subtype=NXAST_REG_LOAD` plus an offset and length bit range.

**Applied examples.**

```
actions=set_field:aa:bb:cc:dd:ee:ff->eth_dst                             # full MAC overwrite
actions=set_field:aa:bb:cc:00:00:00/ff:ff:ff:00:00:00->eth_src           # OUI partial write
actions=set_field:10.2.0.10->ipv4_dst,set_field:8080->tcp_dst            # DNAT-like
actions=set_field:46->ip_dscp                                            # DSCP EF marking
actions=set_field:0x42->ct_mark                                          # set conntrack mark
actions=set_field:0x100->tun_id                                          # VXLAN VNI
actions=set_field:0x1->reg0[0]                                           # bit 0 of reg0
```

**How an engineer masters operations.** Four skills: (1) **Read**: distinguish a full-field write (`set_field:V->F`) from a partial-bit write (`set_field:V/M->F`). (2) **Edit**: prefer `set_field` over legacy `mod_*` for new code; gain forward compatibility plus uniform syntax. (3) **Observe**: `ovs-ofctl dump-flows br0 -O OpenFlow15` shows the rule with the full `set_field` action; OF 1.0 mode shows the legacy `mod_*` form. (4) **Debug**: the `ofproto/trace` line `Output: drop / set(eth(dst=...))` confirms the datapath performs the byte rewrite.

**Classification.** A field-rewrite action. Properties: stateful (modifies the packet in place), checksum-aware (auto-recomputes the IP/TCP/UDP checksum when modifying an L3 or L4 field), not a terminator. Same class: legacy `mod_*` (a subset of the semantics), `move` and `load` (NXM register manipulation, similar capability but restricted to registers and metadata), `dec_ttl` (a specialised rewrite for TTL).

**Source code.** `lib/ofp-actions.c` function anchors: `encode_SET_FIELD` line 3164 (encode for OF 1.2-1.5), `set_field_split_str` line ~3210 (parse splitter), `parse_SET_FIELD`. Header `include/openvswitch/ofp-actions.h` line 543 `struct ofpact_set_field`. The `mf_field` metadata in `lib/meta-flow.c` enumerates 150 plus fields. Branch baseline: OVS v2.17.9.

**Failure scenarios.** (a) `set_field:0x100->reg0` when reg0 is 32-bit: the parser accepts (0x100 fits) but `set_field:0xffffffffffff->reg0` is rejected with "value larger than field". (b) `set_field:46->ip_dscp` without the prerequisite `eth_type=0x0800` causes flow add to be rejected with `OFPBAC_MATCH_INCONSISTENT`; fix: add `ip` to the match. (c) `set_field` with a mask on a field that does not support masked writes (for example `eth_type`) returns "field doesn't support masking".

**Synthetic exercise.** Mininet, two hosts h1 (10.0.0.1) and h2 (10.0.0.2). Install rule `priority=100,in_port=1,ip,nw_dst=10.0.0.2,actions=set_field:00:11:22:33:44:55->eth_dst,output:2` to rewrite the MAC of h2 to a fake value before forwarding. `tcpdump -i h2-eth0 -n -e` confirms the packet arriving on h2 has dst MAC `00:11:22:33:44:55` instead of the real MAC.

**Observation and verification.** `ovs-ofctl dump-flows br0 -O OpenFlow15 | grep set_field` counts the rules using it. `ovs-appctl coverage/show | grep set_field` counts the executions (a counter cumulative since vswitchd started). `ofproto/trace` shows the normalised form `set(eth(dst=aa:bb:..))` on the Datapath actions line.

**Comparison with other systems.** Linux `tc filter ... action pedit munge offset N u32 set V` is the same concept (a byte rewrite at a specific offset) but at a lower level; the engineer must compute the offset manually rather than use a field name. Cisco IOS QoS `set dscp ef` is equivalent to `set_field:46->ip_dscp` but restricted to QoS fields. P4 `hdr.ipv4.dst_addr = 0x0a020a0a`, a direct assignment in a control block, is the P4 way to expose the same idea.

### 4.9.33.3 `mod_dl_src`, the legacy OF 1.0 ethernet source MAC overwrite

**Concept.** `mod_dl_src` is a legacy OF 1.0 action that overwrites the 6-byte ethernet source MAC of the packet. Semantically equivalent to `set_field:<mac>->eth_src` from OF 1.2 onward. OVS keeps `mod_dl_src` for backward compatibility; many older production rules and many ovn-northd-generated flows still use this syntax. Action type: **specialised field rewrite**, OF 1.0 raw type `OFPAT10_SET_DL_SRC`.

**History and context.** OpenFlow 1.0 (2009) defined `OFPAT_SET_DL_SRC` as action type 4 with a 6-byte MAC payload (plus 6-byte padding). OVS supported it from the first release. OpenFlow 1.2 (2011) introduced the generic `set_field` and started gradual deprecation. OVS still parses the `mod_dl_src` syntax and emits the OF 1.0 wire format when the bridge runs in OF 1.0 mode; in OF 1.2 and later it encodes to `set_field`. ovn-northd (2018 onward) still uses the `mod_dl_src` syntax in many scenarios such as the distributed-router gateway MAC rewrite.

**Position in the architecture.** At the OpenFlow protocol layer: `OFPAT10_SET_DL_SRC` raw type for OF 1.0; OF 1.2 and later encode to `OFPAT_SET_FIELD` with MFF_ETH_SRC. At the OVS internal layer: both `mod_dl_src` and `set_field:V->eth_src` produce the same `struct ofpact_mac` (`include/openvswitch/ofp-actions.h` line 461). Verify: line 461 comment `OFPACT_SET_ETH_SRC, OFPACT_SET_ETH_DST. Used for OFPAT10_SET_DL_SRC, OFPAT10_SET_DL_DST.`.

**Role.** Primary role: rewrite the source MAC in an L3 router scenario (OVS acts as a router; when a packet is forwarded between subnets, the source MAC must change to the router output interface's MAC). Secondary role: ARP responder pattern (rewrite the source MAC to the gateway's virtual MAC). Blast radius: per-packet.

**Why it exists.** L3 routing on OVS requires MAC rewriting: when a packet moves from subnet A to subnet B, the inbound source MAC must change to the router output interface's MAC (because the receiving host on subnet B only accepts packets whose source MAC is its own router's MAC). Before OpenFlow, switch ASICs handled this in the L3 lookup engine; OpenFlow exposes it as a programmable action.

**Problems it solves.** Specifically: allows OVS to play the role of an L3 router purely in software through flow rules. Approach: the action `mod_dl_src` rewrites the source MAC, `mod_dl_dst` rewrites the destination MAC, and `dec_ttl` decrements the TTL; together these three form the L3 forward primitive. Scope: every software L3 forwarding scenario (OVN logical router, Mininet L3 test).

**Importance.** Tier 2 medium. In OVN flows on br-int, `mod_dl_src` plus `mod_dl_dst` appear at every distributed-router pipeline rewrite stage (LR pipeline tables 11 and 12). Engineer must-know: the 6-byte MAC notation `aa:bb:cc:dd:ee:ff`, and the fact that this action is OF 1.0 raw but is parsed and dumped by OVS even in OF 1.5 dump output.

**How it works.** The parser in `lib/ofp-actions.c` function `parse_SET_ETH_SRC` (line 1976) calls `str_to_mac` to parse the string into a 6-byte `struct eth_addr` and stores it into `ofpact_put_SET_ETH_SRC(pp->ofpacts)->mac`. The formatter `format_SET_ETH_SRC` (line 1988) prints `mod_dl_src:%s` when dumping the rule (verify lines 1991-1992: `ds_put_format(fp->s, "%smod_dl_src:%s"ETH_ADDR_FMT, colors.param, colors.end, ETH_ADDR_ARGS(a->mac));`). The encoder `encode_SET_ETH_SRC` (around line 1957) calls the helper `encode_SET_ETH_addr` with raw type `OFPAT_RAW_SET_DL_SRC` and field `MFF_ETH_SRC`. Datapath kernel: `OVS_ACTION_ATTR_SET` with an `OVS_KEY_ATTR_ETHERNET` payload of 12 bytes (eth_src 6 plus eth_dst 6, setting only the requested field).

**Wire encoding.** OF 1.0 format: action header `type=4` (OFPAT10_SET_DL_SRC) plus `length=16` plus a 6-byte MAC payload plus 6-byte padding (8-byte aligned). OF 1.2 and later: encoded as `OFPAT_SET_FIELD` with an `OXM_OF_ETH_SRC` 6-byte OXM TLV. On the wire the engineer sees 16 bytes for OF 1.0 and about 16 bytes (8 header plus 4 OXM header plus 6 MAC plus 2 padding) for OF 1.2 and later.

**Applied examples.**

```
actions=mod_dl_src:aa:bb:cc:00:00:01,output:5                                # plain rewrite
actions=mod_dl_src:aa:bb:cc:00:00:01,mod_dl_dst:aa:bb:cc:00:02:01,dec_ttl,output:uplink
                                                                              # L3 router forward
actions=mod_dl_src:aa:bb:cc:dd:ee:ff,IN_PORT                                  # ARP responder
```

A typical OVN-generated flow at table 11 (lr_in_arp_resolve) on br-int:

```
priority=100,reg0=0xa010001,reg14=0x1,actions=mod_dl_src:aa:bb:cc:00:01:01,resubmit(,12)
```

**How an engineer masters operations.** Four skills: (1) **Read**: recognise that `mod_dl_src` appears in dump output even when the bridge is configured for OF 1.5 (OVS keeps the legacy display). (2) **Edit**: switch new code to `set_field:<mac>->eth_src` for consistency and to enable masks. (3) **Observe**: `ovs-ofctl dump-flows br-int | grep mod_dl_src | wc -l` counts MAC-rewrite rules (mostly the OVN router pipeline). (4) **Debug**: `ofproto/trace 'in_port=...,dl_src=aa:..,...'` shows the line `set(eth(src=cc:dd:..))` in Datapath actions, confirming the rewrite.

**Classification.** A field-rewrite action; OF 1.0 raw type; not a terminator; no partial-write capability (always overwrites the full 6 bytes; for partial use `set_field:V/M->eth_src`). Same class: `mod_dl_dst` (rewrite eth_dst), `mod_nw_src/dst` (rewrite IP), `mod_tp_src/dst` (rewrite L4 port).

**Source code.** `lib/ofp-actions.c` function anchors: `parse_SET_ETH_SRC` line 1976, `format_SET_ETH_SRC` line 1988, `encode_SET_ETH_SRC` around line 1957 (calling the `encode_SET_ETH_addr` helper with raw type `OFPAT_RAW_SET_DL_SRC`). Header `include/openvswitch/ofp-actions.h` line 461 `struct ofpact_mac`. Branch baseline: OVS v2.17.9.

**Failure scenarios.** (a) `mod_dl_src:aa:bb` (only 2 bytes) is rejected by the parser with `invalid Ethernet address`. Fix: provide all 6 bytes. (b) Rewriting the source MAC but forgetting to rewrite the destination MAC in an L3 forward scenario means the receiving host gets the packet with the wrong destination MAC and drops it at the NIC filter. Fix: pair `mod_dl_src` with `mod_dl_dst`. (c) Rewriting the source MAC immediately before `output:NORMAL` causes MAC-table learning to record the new MAC against the wrong in_port, poisoning the FDB. Fix: use `output:<explicit_port>` instead of NORMAL.

**Synthetic exercise.** Mininet, two hosts h1 and h2 in the same broadcast domain. Install rule `priority=100,in_port=1,actions=mod_dl_src:de:ad:be:ef:00:01,output:2`. On h2 run `tcpdump -i h2-eth0 -n -e`, then run `h1 ping h2`. Verify the ARP request from h1 to h2 has the fake source MAC `de:ad:be:ef:00:01` instead of the real h1-eth0 MAC.

**Observation and verification.** `ovs-ofctl dump-flows br0` shows the action format `mod_dl_src:aa:..`. `ovs-appctl coverage/show | grep -i ofpact` counts the ofpact instances executed. The `ofproto/trace` line `Datapath actions: set(eth(src=aa:..)),N` confirms the rewrite.

**Comparison with other systems.** Linux netfilter has no direct equivalent in the `filter` table (filter only, no modify), but the `mangle` table has `--src-mac` set through the `MAC` extension (rarely used). `tc filter ... action pedit munge offset 6 u32 set 0xaabbccdd` is the lowest-level option. Cisco IOS does not expose MAC rewrite through the standard CLI (it is hardcoded into the L3 routing logic). P4 `hdr.ethernet.src_addr = 0x...` is a direct assignment.

### 4.9.33.4 `mod_dl_dst`, the legacy OF 1.0 ethernet destination MAC overwrite

**Concept.** `mod_dl_dst` is a legacy OF 1.0 action that overwrites the 6-byte ethernet destination MAC. Equivalent to `set_field:<mac>->eth_dst` from OF 1.2 onward. Symmetric to `mod_dl_src`: the same struct ofpact, the same wire encoding, only the target field differs. OVS keeps the syntax for backward compatibility and dump output. Action type: **specialised field rewrite**, OF 1.0 raw type `OFPAT10_SET_DL_DST`.

**History and context.** Same timeline as `mod_dl_src`: OF 1.0 (2009) added `OFPAT_SET_DL_DST` action type 5. OVS supported it from the start. OF 1.2 (2011) added `set_field` as the replacement. ovn-northd uses `mod_dl_dst` heavily in the distributed-router pipeline and the load-balancer DNAT scenario.

**Position in the architecture.** At the OpenFlow protocol layer: `OFPAT10_SET_DL_DST` raw type 5 in OF 1.0; OF 1.2 and later encode to `OFPAT_SET_FIELD` with MFF_ETH_DST. At the OVS internal layer: shares `struct ofpact_mac` with `mod_dl_src`, distinguished by `ofpact_type = OFPACT_SET_ETH_DST` versus `OFPACT_SET_ETH_SRC` (verify `ofp-actions.h` line 461 comment, shared between the two).

**Role.** Primary role: L3 router rewrite of the destination MAC (a packet leaving the router output must have the next-hop MAC as its destination, looked up in the ARP table). Secondary role: load-balancer DNAT (rewrite the destination MAC to forward to a backend pool member). Auxiliary role: ARP responder. Blast radius: per-packet.

**Why it exists.** Same motivation as `mod_dl_src`: L3 forwarding requires rewriting both source and destination MAC. Specifically for `mod_dl_dst`: rewriting the destination MAC is the more critical for forwarding correctness; if the destination MAC is not the actual next-hop MAC, the receiving host drops the frame at the NIC filter at L1.

**Problems it solves.** Specifically: allow OVS to play the L3 router and load balancer role purely in software through flow rules. Approach: an action that rewrites the destination MAC after ARP resolution (look up the next-hop MAC, then rewrite). Scope: OVN distributed router, Mininet L3 lab, software LB.

**Importance.** Tier 2 medium with very high frequency in OVN production. On a chassis running 10 logical routers, the OVN br-int has thousands of flows containing `mod_dl_dst` (one flow per LRP times remote MAC). Engineer must-know: the syntax is symmetric to `mod_dl_src`, and the wire format is identical.

**How it works.** The parser in `lib/ofp-actions.c` function `parse_SET_ETH_DST` (line 1982) is the same as `parse_SET_ETH_SRC`, calling `str_to_mac` to store into `ofpact_put_SET_ETH_DST(pp->ofpacts)->mac`. The formatter `format_SET_ETH_DST` (line 1996) prints `mod_dl_dst:%s` (verify lines 1999-2000: `ds_put_format(fp->s, "%smod_dl_dst:%s"ETH_ADDR_FMT, colors.param, colors.end, ETH_ADDR_ARGS(a->mac));`). The encoder `encode_SET_ETH_DST` (around line 1965) calls the helper `encode_SET_ETH_addr` with raw type `OFPAT_RAW_SET_DL_DST` and field `MFF_ETH_DST`. Datapath kernel: `OVS_ACTION_ATTR_SET` with an `OVS_KEY_ATTR_ETHERNET` payload setting eth_dst.

**Wire encoding.** OF 1.0 format: action header `type=5` (OFPAT10_SET_DL_DST) plus `length=16` plus a 6-byte MAC payload plus 6-byte padding. OF 1.2 and later: `OFPAT_SET_FIELD` with an `OXM_OF_ETH_DST` 6-byte OXM TLV. Fully symmetric to `mod_dl_src`, only the type number differs.

**Applied examples.**

```
actions=mod_dl_dst:00:50:56:aa:bb:cc,output:5                                  # explicit MAC rewrite
actions=mod_dl_src:rt_mac,mod_dl_dst:next_hop_mac,dec_ttl,output:uplink         # L3 router primitive
actions=group:1                                                                 # load balancer (group bucket contains mod_dl_dst)
```

A typical OVN-generated flow at table 12 (lr_in_arp_resolve / lr_out_delivery):

```
priority=100,reg0=0xc0a80101,reg14=0x2,actions=mod_dl_dst:fa:16:3e:aa:bb:cc,resubmit(,13)
```

OVN load balancer with group buckets (verify §4.9.11):

```
group_id=1,type=select,
  bucket=weight:100,actions=mod_dl_dst:aa:bb:cc:00:00:01,output:5,
  bucket=weight:100,actions=mod_dl_dst:aa:bb:cc:00:00:02,output:6
```

**How an engineer masters operations.** Four skills: (1) **Read**: in OVN dump output recognise `mod_dl_dst` at tables 11 and 12 (LR pipeline) as an ARP-resolved MAC. (2) **Edit**: for new code switch to `set_field:<mac>->eth_dst`, especially when a partial mask write is needed. (3) **Observe**: `ovs-ofctl dump-flows br-int | grep mod_dl_dst | sort -u | head` shows distinct MAC values, usually correlating with the host's ARP table. (4) **Debug**: `ofproto/trace` shows the line `Datapath actions: set(eth(dst=aa:..)),N`.

**Classification.** A field-rewrite action; OF 1.0 raw type; symmetric to `mod_dl_src`; no partial-write (always overwrites 6 bytes). Same class: `mod_dl_src`, `set_field:V->eth_dst`. Different from `move:NXM_OF_ETH_SRC[]->NXM_OF_ETH_DST[]`: `move` copies field to field (dynamic, source value at runtime), whereas `mod_dl_dst` sets a constant value compiled into the rule.

**Source code.** `lib/ofp-actions.c` function anchors: `parse_SET_ETH_DST` line 1982, `format_SET_ETH_DST` line 1996, `encode_SET_ETH_DST` around line 1965. Header `include/openvswitch/ofp-actions.h` line 461 `struct ofpact_mac` (shared with SET_ETH_SRC). Branch baseline: OVS v2.17.9.

**Failure scenarios.** (a) Rewriting the destination MAC while the next-hop ARP entry is stale (the host has changed MAC) means the inbound packet is dropped at the host. Fix: integrate with ARP refresh (OVN uses FDB aging plus `ovn-controller` re-resolve). (b) Rewriting the destination MAC in a load balancer but forgetting `ct(commit)` for the NAT mapping leaves return traffic without a reverse rewrite, breaking the connection. Fix: pair `ct(commit)` with `mod_dl_dst`. (c) Rewriting the destination MAC to a multicast MAC accidentally (bit 0 of byte 0 = 1) causes the packet to flood instead of unicast. Fix: validate that bit 0 is 0 before committing the rule.

**Synthetic exercise.** Mininet, three hosts h1, h2, and h3. Install rule `priority=100,in_port=h1-eth0,ip,nw_dst=10.0.0.3,actions=mod_dl_dst:<mac_h2>,output:h2-eth0` to redirect a packet destined for h3 to host h2. `tcpdump -i h2-eth0` confirms h2 receives the packet (even though the IP destination is h3), demonstrating an L2 redirect that bypasses L3.

**Observation and verification.** `ovs-ofctl dump-flows br0` shows `mod_dl_dst:aa:..`. `ovs-appctl ofproto/trace br0 'in_port=1,dl_dst=...,...'` shows the line `set(eth(dst=...))`. `tcpdump -e` on the output port verifies the wire-level MAC rewrite.

**Comparison with other systems.** Linux netfilter `iptables -t mangle -j MARK` does not rewrite MAC, but `tc filter ... action pedit munge offset 0 u32 set ...` allows the low-level operation. Cisco IOS `mac-address-table static` plus the L3 routing engine implicitly rewrites the destination MAC (no programmable action exposed). NSX-T Edge load balancer rewrites the destination MAC implicitly through the DLR. P4 `hdr.ethernet.dst_addr = ...` is a direct assignment.

### 4.9.33.5 `dec_ttl`, decrement IP TTL by one and emit PACKET_IN when TTL=0

**Concept.** `dec_ttl` is an action that decrements the IP Time-To-Live (or IPv6 Hop Limit) by one. After the decrement: when TTL > 0, the packet continues through the pipeline; when TTL = 0, the packet is sent to the controller as `OFPT_PACKET_IN` with reason code `OFPR_INVALID_TTL` (3). OVS recomputes the IP checksum automatically after the decrement (because TTL is one byte in the header covered by the IPv4 checksum). MPLS has its own variant `dec_mpls_ttl` for the MPLS label TTL field. Action type: **specialised field rewrite with a controller side-effect**.

**History and context.** OpenFlow 1.0 (2009): added `OFPAT_SET_NW_TTL` (set absolute value) plus the Nicira extension `NXAST_DEC_TTL` (decrement). OpenFlow 1.1 (2011): standardised as `OFPAT11_DEC_NW_TTL` (action type 18). OpenFlow 1.2 (2011): added `NXAST_DEC_TTL_CNT_IDS` allowing the controller-ID list that should receive PACKET_IN when TTL=0 (multi-controller scenario). OF 1.5 spec §5.10.2 Action Types states the semantics: "If TTL becomes 0 after decrement, the packet shall be sent to the controller with reason OFPR_INVALID_TTL".

**Position in the architecture.** At the OpenFlow protocol layer: `OFPAT11_DEC_NW_TTL` action type 18 in OF 1.1 and later; the NXAST extension covers OF 1.0 plus multi-controller. At the OVS internal layer: `struct ofpact_cnt_ids` (`include/openvswitch/ofp-actions.h` line 1031) holds `n_controllers` plus the `cnt_ids[]` array. At the datapath layer: the kernel translates to `OVS_ACTION_ATTR_SET` with the TTL key decremented plus a checksum recompute; when TTL=0 after decrement, the kernel returns an upcall to userspace to emit PACKET_IN.

**Role.** Primary role: L3 routing TTL decrement (each router hop decrements TTL by 1; reaching 0 produces ICMP TTL Exceeded). Secondary role: traceroute support (the host sends probes with TTL=1, 2, 3, ... and each router decrements; when TTL=0 the router emits ICMP Time Exceeded). Blast radius: per-packet, every IP packet flowing through a flow with `dec_ttl`.

**Why it exists.** Standard L3 forwarding requires decrementing TTL to prevent infinite routing loops (RFC 791). Before OpenFlow, hardware ASICs did this automatically inside the L3 lookup engine. OpenFlow exposes it as a programmable action so OVS can play the role of a software router. A dedicated action (rather than reusing `set_field:<v>->nw_ttl`) is needed because of the special handling at TTL=0 (send PACKET_IN rather than silently drop or continue).

**Problems it solves.** Specifically: (a) standard TTL decrement for a software L3 router, (b) ICMP Time Exceeded support through the controller (the controller receives a PACKET_IN with reason invalid_ttl and emits the ICMP message). Approach: an action that decrements the field, recomputes the checksum, and triggers the side-effect PACKET_IN. Scope: OVS L3 router, OVN logical-router L3 forwarding, Mininet L3 lab, traceroute support inside a virtual network.

**Importance.** Tier 2 medium. Every L3-forwarding rule of an OVN logical-router pipeline includes `dec_ttl` at table 11 (lr_in_ip_routing). Engineer must-know: the special behaviour at TTL=0 (PACKET_IN, not drop), the reason code `OFPR_INVALID_TTL`, and the requirement that the controller has a handler for this reason if standard ICMP Time Exceeded emission is desired.

**How it works.** Parser `lib/ofp-actions.c` function `parse_DEC_TTL` (line 3709): with empty argument, `parse_noargs_dec_ttl` adds controller ID = 0 (default). With an argument (a comma-separated controller-ID list), the loop `strtok` parses each ID and marks the raw type as `NXAST_RAW_DEC_TTL_CNT_IDS`. Formatter `format_DEC_TTL` (line 3742): print "dec_ttl" plus `(id1,id2,...)` when raw is NXAST_DEC_TTL_CNT_IDS. Encoder `encode_DEC_TTL` (line 3672): when raw is NXAST_DEC_TTL_CNT_IDS, or n_controllers is not 1, or id is not 0, encode the NXAST extension; otherwise encode standard `OFPAT_DEC_NW_TTL`. Datapath kernel: read the current TTL, decrement by 1, recompute the IPv4 header checksum (`csum_replace` macro); when TTL=0, mark the skb for an upcall with reason invalid_ttl.

**Wire encoding.** OF 1.1 and later standard `OFPAT_DEC_NW_TTL`: action header `type=18` (DEC_NW_TTL) plus `length=8` (header only, no payload) plus 4-byte padding. NXAST multi-controller format: `vendor=0x2320` (Nicira) plus `subtype=20` (NXAST_DEC_TTL_CNT_IDS) plus `n_controllers` (2 bytes) plus an array of 16-bit IDs plus padding to 8-byte alignment. OF 1.0 has no standard form and uses NXAST.

**Applied examples.**

```
actions=dec_ttl,output:1                                              # plain L3 forward
actions=dec_ttl,mod_dl_src:rt_mac,mod_dl_dst:nh_mac,output:uplink     # full L3 router primitive
actions=dec_ttl(0,1)                                                  # multi-controller (ID 0 plus 1)
priority=100,ip,actions=dec_ttl,resubmit(,12)                         # L3 forward subroutine
priority=0,ip,actions=controller(reason=invalid_ttl)                  # table-miss for ICMP TTL exceeded
```

A typical OVN-generated flow at table 11 lr_in_ip_routing:

```
priority=200,ip,reg9=0x0,actions=dec_ttl,resubmit(,12)
```

**How an engineer masters operations.** Four skills: (1) **Read**: recognise `dec_ttl` in the OVN router pipeline plus `dec_mpls_ttl` for MPLS scenarios. (2) **Edit**: pair `dec_ttl` with the fallback rule `priority=0,actions=controller(reason=invalid_ttl)` in the following table so ICMP Time Exceeded is emitted properly. (3) **Observe**: `ovs-ofctl dump-flows br-int | grep dec_ttl | wc -l` counts L3 forward rules; `coverage/show | grep dec_ttl` counts the number of packets whose TTL has been decremented. (4) **Debug**: `ofproto/trace 'in_port=...,ip,nw_ttl=1,...'` simulates a packet with TTL=1; the trace shows `dec_ttl` setting TTL=0 and emitting packet_in.

**Classification.** A field-rewrite action specialised with a controller side-effect. Properties: stateful (modify in place plus recompute checksum), conditional side-effect (PACKET_IN at TTL=0), not a terminator (TTL > 0 still allows subsequent actions). Same class: `set_field:<v>->nw_ttl` (absolute set, no PACKET_IN at 0), `dec_mpls_ttl` (for the MPLS label TTL). Different from `set_nw_ttl`: `set_nw_ttl:<v>` sets absolutely and does not decrement.

**Source code.** `lib/ofp-actions.c` function anchors: `encode_DEC_TTL` line 3672, `parse_DEC_TTL` line 3709, `format_DEC_TTL` line 3742, `parse_noargs_dec_ttl` around line 3690. Header `include/openvswitch/ofp-actions.h` line 1031 `struct ofpact_cnt_ids` (also used for multi-controller config). Datapath kernel `datapath/actions.c` function `set_ipv4` handles TTL decrement plus `csum_replace2` for IPv4 checksum. Branch baseline: OVS v2.17.9.

**Failure scenarios.** (a) `dec_ttl` on a non-IP packet (ARP, LLDP, raw L2 frame) is silently skipped (the kernel datapath checks L3 first). Fix: add `ip` or `ipv6` to the match. (b) Forgetting the fallback `controller(reason=invalid_ttl)` at a table-miss means packets at TTL=0 generate a PACKET_IN with no handler, the controller logs a warning and drops, and traceroute does not return Time Exceeded. Fix: install the fallback rule. (c) `dec_ttl` chained too many times in the same pipeline (for example, twice across two resubmits) decrements TTL by 2 per packet instead of 1; traceroute reports the wrong hop count. Fix: audit the pipeline design.

**Synthetic exercise.** Mininet topology: one OVS bridge `s1` plus two hosts h1 and h2. Install rule `priority=100,in_port=h1-eth0,ip,actions=dec_ttl,output:h2-eth0` plus rule `priority=0,ip,actions=controller(reason=invalid_ttl)`. From h1 run `ping -t 1 h2` (TTL=1). Capture PACKET_IN at the controller (ovs-ofctl monitor) and verify the reason is `invalid_ttl`. Compare with `ping -t 2 h2`, which reaches successfully.

**Observation and verification.** `ovs-ofctl dump-flows br0` shows `dec_ttl`. `ovs-ofctl monitor br0 watch:` captures PACKET_IN with `reason=invalid_ttl` at TTL=0. `tcpdump -i h2-eth0 -v` verifies the TTL field is exactly one less than what h1 sent.

**Comparison with other systems.** Linux IP forwarding automatically decrements TTL inside the `ip_forward` function (`net/ipv4/ip_forward.c`); when TTL=0 it emits ICMP Time Exceeded through `icmp_send(ICMP_TIME_EXCEEDED)`. Cisco IOS L3 routing engine also auto-decrements TTL and emits ICMP. P4 `hdr.ipv4.ttl = hdr.ipv4.ttl - 1` is a direct decrement in a control block; the P4 programmer must handle the TTL=0 case (send the packet to the control plane equivalently). FRR plus Linux kernel inherit the Linux behaviour.

### 4.9.33.6 `mod_nw_src`, the legacy OF 1.0 IPv4 source address overwrite

**Concept.** `mod_nw_src` is a legacy OF 1.0 action that overwrites the 4-byte IPv4 source address of the packet. Semantically equivalent to `set_field:<ip>->ipv4_src` (or the older alias `nw_src`) in OF 1.2 and later. OVS keeps the syntax for backward compatibility and dump output. Most SNAT (Source NAT) rules in software-only L3 router scenarios use this action (or the higher-level `nat()` action). Action type: **specialised field rewrite**, OF 1.0 raw type `OFPAT10_SET_NW_SRC`.

**History and context.** OpenFlow 1.0 (2009) defined `OFPAT_SET_NW_SRC` as action type 6 with a 4-byte IPv4 address payload (and 0 padding for 8-byte alignment). OVS supported it from the first release. OF 1.2 (2011) added `set_field` as the replacement. The OF 1.5 spec does not fully deprecate the `OFPAT10_SET_NW_SRC` raw type because OVS still translates it forward. In OVN production this action is rarely used directly because OVN provides the higher-level `nat()` action that handles SNAT and DNAT together with conntrack; however, many manually installed rules (Mininet labs, debug rules) still use `mod_nw_src`.

**Position in the architecture.** At the OpenFlow protocol layer: `OFPAT10_SET_NW_SRC` raw type 6 in OF 1.0; OF 1.2 and later encode to `OFPAT_SET_FIELD` with MFF_IPV4_SRC. At the OVS internal layer: shares `struct ofpact_ipv4` with `mod_nw_dst` (verify in `include/openvswitch/ofp-actions.h`), distinguished by `ofpact_type = OFPACT_SET_IPV4_SRC` versus `OFPACT_SET_IPV4_DST`. At the datapath kernel layer: translates to `OVS_ACTION_ATTR_SET` with an `OVS_KEY_ATTR_IPV4` payload (12 bytes: src 4 plus dst 4 plus proto 1 plus tos 1 plus ttl 1 plus frag 1).

**Role.** Primary role: SNAT (Source NAT) as a pure action; rewrite the source IP so outgoing traffic carries a public address rather than a private one. Secondary role: privacy or anti-spoofing rewrite (egress filter), masquerade on a gateway. Blast radius: per-packet; the IPv4 checksum and L4 checksum (TCP/UDP) are auto-recomputed by the datapath.

**Why it exists.** A software L3 router needs SNAT when forwarding between public and private subnets. Before OpenFlow, Linux netfilter `iptables -t nat -A POSTROUTING -j SNAT --to-source X` did this. OpenFlow exposes the programmable action so OVS handles it through flow rules without a separate iptables stack. A dedicated action (rather than waiting for `set_field`) was required because OF 1.0 did not yet have a generic field abstraction.

**Problems it solves.** Specifically: enables SNAT purely through OpenFlow on an OVS bridge with no iptables/netfilter dependency. Approach: an action that rewrites the source IP plus auto-recomputes the IPv4 and L4 checksums. Scope: simple SNAT (one rule equals one mapping); for dynamic SNAT (port pool, conntrack-based), combine with the `nat()` action.

**Importance.** Tier 2 medium. Rare directly in OVN production rules (OVN uses `nat()`), but very common in Mininet labs, custom controller patterns, and manual debug rules. Engineer must-know: the syntax `mod_nw_src:10.0.0.5` and the fact that the action does not commit conntrack itself (combine with `ct(commit, nat(src=...))` for full stateful NAT).

**How it works.** Parser `lib/ofp-actions.c` function `parse_SET_IPV4_SRC` (around line 3360 in v3.3.0) calls `str_to_ip` to parse the IPv4 string into a 32-bit `ovs_be32` and store it into `ofpact_put_SET_IPV4_SRC(pp->ofpacts)->ipv4`. Formatter `format_SET_IPV4_SRC` (around line 3378) prints `mod_nw_src:%s` with IPv4 dotted notation (in OF 1.0 mode) or `set_field:%s->nw_src` (in OF 1.2 and later). Encoder `encode_SET_IPV4_SRC` (around line 3370) picks raw type `OFPAT_RAW_SET_NW_SRC` for OF 1.0 wire or falls back to `OFPAT_SET_FIELD` for OF 1.2 and later. Datapath kernel: `OVS_ACTION_ATTR_SET` with the IPv4 key calls the `set_ipv4` function in `datapath/actions.c` to rewrite the 4-byte source plus recompute the checksum (`csum_replace4`).

**Wire encoding.** OF 1.0 format: action header `type=6` (OFPAT10_SET_NW_SRC) plus `length=8` plus a 4-byte IPv4 address payload. OF 1.2 and later format: encoded as `OFPAT_SET_FIELD` action type 25 with an `OXM_OF_IPV4_SRC` 4-byte OXM TLV. On the wire OF 1.0 needs 8 bytes for the action; OF 1.2 and later needs 16 bytes (8 header plus 4 OXM header plus 4 value).

**Applied examples.**

```
priority=100,ip,in_port=1,actions=mod_nw_src:10.0.0.5,output:2          # plain SNAT
priority=100,ip,nw_src=192.168.0.0/16,actions=mod_nw_src:10.1.0.1,output:uplink
                                                                          # masquerade gateway
actions=mod_nw_src:10.0.0.5,mod_dl_src:rt_mac,dec_ttl,output:wan        # SNAT plus L3 forward
actions=set_field:10.0.0.5->ipv4_src                                    # OF 1.2 and later equivalent (preferred)
```

**How an engineer masters operations.** Four skills: (1) **Read**: distinguish `mod_nw_src` (constant-value SNAT) from the `nat()` action (stateful conntrack-based SNAT). (2) **Edit**: switch new code to `set_field:<ip>->ipv4_src`; for stateful SNAT switch fully to `ct(commit, nat(src=<ip>))`. (3) **Observe**: `ovs-ofctl dump-flows br0 | grep mod_nw_src` counts pure-action SNAT rules. (4) **Debug**: `ofproto/trace 'in_port=1,ip,nw_src=192.168.1.5,...'` shows the line `set(ipv4(src=10.0.0.5,...))`, confirming the datapath rewrite plus checksum recompute.

**Classification.** A field-rewrite action specialised for IPv4 src; OF 1.0 raw type; not a terminator; no partial mask (always overwrites the full 4 bytes; for partial masking use `set_field:<ip>/<mask>->ipv4_src`). Same class: `mod_nw_dst` (rewrite IPv4 dst), `mod_tp_src/dst` (rewrite L4 port), `mod_dl_src/dst` (rewrite MAC). Different from `nat()`: `nat()` is stateful (per-connection mapping through conntrack), whereas `mod_nw_src` is stateless (every packet matching the rule receives the same rewrite).

**Source code.** `lib/ofp-actions.c` function anchors (OVS v2.17.9): `parse_SET_IPV4_SRC` around line 3360, `encode_SET_IPV4_SRC` around line 3370, `format_SET_IPV4_SRC` around line 3378. Header `include/openvswitch/ofp-actions.h` `struct ofpact_ipv4` (shared with SET_IPV4_DST). Datapath kernel `datapath/actions.c` function `set_ipv4` handles the byte rewrite plus `csum_replace4` checksum recomputation. Branch baseline: OVS v2.17.9.

**Failure scenarios.** (a) `mod_nw_src:10.0.0.5` without `ip` in the match causes flow add to be rejected with `OFPBAC_MATCH_INCONSISTENT`. Fix: add `ip` to the match. (b) Stateless SNAT for TCP traffic: the outbound packet is SNATed to `10.0.0.5`, but the return traffic has no reverse-rewrite rule, so the connection breaks. Fix: pair the rule `mod_nw_src:10.0.0.5` (egress) with `mod_nw_dst:192.168.1.5` (ingress), or use `ct(commit, nat(src=10.0.0.5))` for stateful behaviour. (c) Forgetting that the IPv4 checksum is recomputed automatically wastes effort writing manual checksum-recompute actions. Verify: the kernel datapath has done the work.

**Synthetic exercise.** Mininet, two hosts h1 (192.168.1.5) and h2 (10.0.0.10) over bridge `s1`. Install rule `priority=100,in_port=h1-eth0,ip,actions=mod_nw_src:10.0.0.99,output:h2-eth0`. On h2 run `tcpdump -i h2-eth0 -n` and observe ICMP requests arriving from `10.0.0.99` instead of the real `192.168.1.5`. Verify the checksum is correct (`tcpdump` does not print `bad cksum` warnings).

**Observation and verification.** `ovs-ofctl dump-flows br0` shows `mod_nw_src:10.0.0.5` (OF 1.0 mode) or `set_field:10.0.0.5->nw_src` (OF 1.5 mode). `ovs-appctl ofproto/trace br0 'in_port=1,ip,...'` shows the line `Datapath actions: set(ipv4(src=10.0.0.5,...)),N`. `tcpdump -i wan-port -n` confirms the wire-level packet has the rewritten src IP and a correct checksum.

**Comparison with other systems.** Linux netfilter `iptables -t nat -A POSTROUTING -j SNAT --to-source 10.0.0.5` is the stateful equivalent (conntrack-aware). `iptables -t mangle -j MARK` does not rewrite IP. `tc filter ... action nat ingress 192.168.1.0/24 10.0.0.0` is the same stateless action concept. Cisco IOS `ip nat inside source` is stateful conntrack-based. P4 `hdr.ipv4.src_addr = 0x0a000005` is a direct assignment.

### 4.9.33.7 `mod_nw_dst`, the legacy OF 1.0 IPv4 destination address overwrite

**Concept.** `mod_nw_dst` is a legacy OF 1.0 action that overwrites the 4-byte IPv4 destination address. Equivalent to `set_field:<ip>->ipv4_dst` in OF 1.2 and later. Symmetric to `mod_nw_src`: shares `struct ofpact_ipv4`, the same wire-encoding pattern, and only the target field plus raw type differs. Most pure-action DNAT (Destination NAT) rules use this action. Action type: **specialised field rewrite**, OF 1.0 raw type `OFPAT10_SET_NW_DST`.

**History and context.** Same timeline as `mod_nw_src`: OF 1.0 (2009) added `OFPAT_SET_NW_DST` action type 7 with a 4-byte IPv4 payload. OVS supported it from the start. OF 1.2 (2011) introduced the generic `set_field` replacement. In OVN this action commonly appears in the Logical Load Balancer pipeline when the VIP is rewritten to a backend IP (combined with `ct(commit, nat(dst=...))` for stateful behaviour). ovn-northd-generated flows at the `lr_in_dnat` table of the LR pipeline often use this pattern in stateless DNAT scenarios.

**Position in the architecture.** At the OpenFlow protocol layer: `OFPAT10_SET_NW_DST` raw type 7 in OF 1.0; OF 1.2 and later encode to `OFPAT_SET_FIELD` with MFF_IPV4_DST. At the OVS internal layer: shares `struct ofpact_ipv4` with `mod_nw_src`, distinguished by `ofpact_type = OFPACT_SET_IPV4_DST`. At the datapath kernel layer: translates to `OVS_ACTION_ATTR_SET` with an `OVS_KEY_ATTR_IPV4` payload setting the dst field.

**Role.** Primary role: DNAT (Destination NAT) as a pure action; rewrite the destination IP to forward incoming traffic to an internal backend distinct from the publicly exposed address. Secondary role: L3 load balancer (rewrite VIP to a real server IP), health-check redirect, port-forwarding scenarios. Blast radius: per-packet; the L3 and L4 checksums are auto-recomputed.

**Why it exists.** Same motivation as `mod_nw_src`: a software L3 router needs DNAT when exposing internal services through a public IP. Specific to `mod_nw_dst`: the load balancer concept (1 VIP to N backends) needs an action to rewrite dst across many possibilities (through a group bucket select). Before OpenFlow, hardware load balancers or Linux IPVS did this; OpenFlow exposes it programmably.

**Problems it solves.** Specifically: (a) basic pure-OpenFlow DNAT, (b) stateless L3 load balancer through flow rule plus group table. Approach: an action that rewrites the destination IP plus a checksum recompute (combined with a `select` group) to spread load across multiple backends. Scope: L3 forward and basic LB; full stateful LB combines with `ct(commit, nat(dst=...))`.

**Importance.** Tier 2 medium. In OVN production it appears in the Logical Load Balancer pipeline plus LR DNAT rules. Engineer must-know: the syntax `mod_nw_dst:10.0.0.10`, and the need to pair with `mod_dl_dst` (rewrite the destination MAC) when crossing subnets (because DNAT changes the IP destination, the destination MAC must be rewritten correspondingly so the receiving host accepts the frame).

**How it works.** Parser `lib/ofp-actions.c` function `parse_SET_IPV4_DST` (around line 3365 in v3.3.0) follows the same pattern as `parse_SET_IPV4_SRC`, calling `str_to_ip` to parse and storing into `ofpact_put_SET_IPV4_DST(pp->ofpacts)->ipv4`. Formatter `format_SET_IPV4_DST` (around line 3383) prints `mod_nw_dst:%s`. Encoder `encode_SET_IPV4_DST` (around line 3375) picks raw type `OFPAT_RAW_SET_NW_DST` for OF 1.0 or `OFPAT_SET_FIELD` for OF 1.2 and later. Datapath kernel: `OVS_ACTION_ATTR_SET` with the IPv4 dst key calls the `set_ipv4` function to rewrite plus `csum_replace4` to recompute the IPv4 checksum and the L4 checksum (TCP, UDP, SCTP) through `inet_proto_csum_replace4`.

**Wire encoding.** OF 1.0 format: action header `type=7` (OFPAT10_SET_NW_DST) plus `length=8` plus a 4-byte IPv4 address payload. OF 1.2 and later format: `OFPAT_SET_FIELD` action type 25 with an `OXM_OF_IPV4_DST` 4-byte OXM TLV. Fully symmetric to `mod_nw_src`, only the type number is 7 instead of 6.

**Applied examples.**

```

                                                                          # simple DNAT
actions=mod_nw_dst:10.0.0.10,mod_dl_dst:00:50:56:aa:00:0a,output:5      # DNAT plus MAC rewrite

```

A typical OVN-generated flow at table 6 lr_in_dnat (the LR pipeline DNAT stage):


**Classification.** A field-rewrite action specialised for IPv4 dst; OF 1.0 raw type; symmetric to `mod_nw_src`; no partial mask (always overwrites the full 4 bytes). Same class: `mod_nw_src`, `set_field:V->ipv4_dst`. Different from the `nat()` action: `nat()` is stateful and per-connection conntrack-based; `mod_nw_dst` is stateless and rewrites every matching packet to the same value.

**Source code.** `lib/ofp-actions.c` function anchors (OVS v2.17.9): `parse_SET_IPV4_DST` around line 3365, `encode_SET_IPV4_DST` around line 3375, `format_SET_IPV4_DST` around line 3383. **Failure scenarios.** (a) DNAT for the IP but forgetting to rewrite the destination MAC in a cross-host scenario means the backend host receives a frame whose destination MAC is not its own and drops it at the NIC L1 filter. Fix: pair `mod_nw_dst` with `mod_dl_dst:<backend_mac>`. (b) Stateless DNAT plus a return path with no reverse rule breaks the connection. ### 4.9.33.8 `mod_tp_src`, the legacy OF 1.0 L4 source port (TCP/UDP/SCTP) overwrite

**Concept.** `mod_tp_src` is a legacy OF 1.0 action that overwrites the 2-byte L4 source port of the packet. It applies to TCP, UDP, or SCTP, depending on the current L4 protocol field (`nw_proto=6` for TCP, `=17` for UDP, `=132` for SCTP). Equivalent to `set_field:<port>->tcp_src` (or `udp_src`, `sctp_src`) in OF 1.2 and later. A core component of PAT (Port Address Translation): combined with `mod_nw_src` it produces a complete SNAT with a port pool. Action type: **specialised field rewrite**, OF 1.0 raw type `OFPAT10_SET_TP_SRC`.

**History and context.** OpenFlow 1.0 (2009) added `OFPAT_SET_TP_SRC` as action type 9 with a 2-byte port payload. OVS supported it from the start. In OF 1.0 there is no L4 protocol distinction; a single `mod_tp_src` action handles TCP, UDP, and SCTP. OF 1.2 (2011) introduced `set_field` with protocol-specific OXMs: `OXM_OF_TCP_SRC`, `OXM_OF_UDP_SRC`, `OXM_OF_SCTP_SRC`. Reason for the split: stricter prerequisite checks; the parser refuses to rewrite a TCP port when the packet has no TCP header. ovn-northd-generated flows in the Logical Load Balancer pipeline often use this action when the LB rewrites the source port to avoid pool collisions.

**Position in the architecture.** At the OpenFlow protocol layer: `OFPAT10_SET_TP_SRC` raw type 9 in OF 1.0 (protocol-agnostic); OF 1.2 and later encode to `OFPAT_SET_FIELD` with MFF_TCP_SRC / MFF_UDP_SRC / MFF_SCTP_SRC depending on context (the parser inspects the `nw_proto` match to choose). At the OVS internal layer: `struct ofpact_l4_port` (`include/openvswitch/ofp-actions.h`) holds a 2-byte port plus a flow_pp protocol hint. At the datapath kernel layer: translates to `OVS_ACTION_ATTR_SET` with an `OVS_KEY_ATTR_TCP` / `_UDP` / `_SCTP` payload setting the src port.

**Role.** Primary role: PAT (Port Address Translation) combined with `mod_nw_src` for SNAT plus port multiplexing; many internal connections SNAT to the same public IP but with distinct port pools to differentiate them. Secondary role: privacy port randomisation (rewriting ephemeral ports masks the client OS fingerprint), egress filter port masking. Blast radius: per-packet; the L4 checksum is auto-recomputed.

**Why it exists.** SNAT 1-to-1 (one internal IP to one public IP) does not scale: every internal host needs its own public IP. PAT 1-to-N (one public IP plus N ports equals N internal connection mappings) scales linearly with the 65,535-port pool. Hardware and Linux netfilter NAT had this concept from the beginning; OpenFlow exposes it programmably as an action.

**Problems it solves.** Specifically: enable PAT purely through OpenFlow (port rewrite plus IP rewrite plus checksum recompute simultaneously) through a flow rule. Approach: action rewrites the L4 port; the L4 checksum is auto-recomputed. Scope: basic PAT (one rule equals one mapping); dynamic PAT (port pool auto-allocation) requires combining with the stateful `nat(src=...)` action.

**Importance.** Tier 2 medium. Rare in production rules directly (OVN uses the stateful `nat()` action), but very common in custom controller patterns, Mininet PAT labs, and manual debug. Engineer must-know: the syntax `mod_tp_src:8080`, and the fact that the action only applies when the packet has an L4 protocol (TCP, UDP, SCTP); on ICMP packets (no port) the action is silently skipped.

**How it works.** Parser `lib/ofp-actions.c` function `parse_SET_L4_SRC_PORT` (around line 3488 in v3.3.0) calls `str_to_u16` to parse the string into a 16-bit port and stores it into `ofpact_put_SET_L4_SRC_PORT(pp->ofpacts)->port`. Formatter `format_SET_L4_SRC_PORT` (around line 3502) prints `mod_tp_src:%"PRIu16` in OF 1.0 mode or `set_field:<port>->tcp_src` in OF 1.2 and later. Encoder `encode_SET_L4_SRC_PORT` (around line 3470) picks raw type `OFPAT_RAW_SET_TP_SRC` for OF 1.0 wire (protocol-agnostic, the datapath dispatches by flow proto); for OF 1.2 and later it encodes `OFPAT_SET_FIELD` with the protocol-specific OXM (TCP, UDP, SCTP). Datapath kernel: `OVS_ACTION_ATTR_SET` calls the `set_tcp` / `set_udp` / `set_sctp` function to rewrite the 2-byte src port plus recompute the L4 checksum (`inet_proto_csum_replace2`).

**Wire encoding.** OF 1.0 format: action header `type=9` (OFPAT10_SET_TP_SRC) plus `length=8` plus a 2-byte port payload plus 2-byte padding (8-byte aligned). OF 1.2 and later format: `OFPAT_SET_FIELD` action type 25 with the OXM TLV `OXM_OF_TCP_SRC` / `OXM_OF_UDP_SRC` / `OXM_OF_SCTP_SRC` of 2 bytes. Wire size is similar: 8 bytes for OF 1.0, about 16 bytes for OF 1.2 and later.

**Applied examples.**

```
priority=100,ip,tcp,tp_src=4001,actions=mod_tp_src:8080,output:wan       # PAT rewrite ephemeral
priority=100,ip,nw_src=192.168.1.0/24,tcp,
        actions=mod_nw_src:10.0.0.5,mod_tp_src:33000,output:uplink        # SNAT plus PAT combo
priority=100,udp,actions=mod_tp_src:5060,output:sip-uplink                # rewrite SIP source
actions=set_field:8080->tcp_src                                          # OF 1.2 and later, TCP only
actions=set_field:5060->udp_src                                          # OF 1.2 and later, UDP only
```

**How an engineer masters operations.** Four skills: (1) **Read**: distinguish `mod_tp_src` (OF 1.0 protocol-agnostic) from `set_field:V->tcp_src` (OF 1.2 and later, TCP-specific). (2) **Edit**: switch new code to `set_field` with the protocol-specific OXM so the parser enforces the prerequisite automatically. For dynamic PAT switch to the stateful `nat(src=ip:port-range)`. (3) **Observe**: `ovs-ofctl dump-flows br0 | grep mod_tp_src` counts pure-action PAT rules. (4) **Debug**: `ofproto/trace 'in_port=1,tcp,tp_src=4001,...'` shows the line `set(tcp(src=8080,...))`, confirming the datapath rewrite plus checksum.

**Classification.** A field-rewrite action specialised for the L4 src port; OF 1.0 raw type, protocol-agnostic; no partial mask (always overwrites the 2 bytes). Same class: `mod_tp_dst` (rewrite L4 dst port), `mod_nw_src/dst` (rewrite IP), `set_field:V->tcp_src/udp_src/sctp_src`. Different from `nat()`: `nat()` is stateful with auto-allocated port pools; `mod_tp_src` is stateless constant rewrite.

**Source code.** `lib/ofp-actions.c` function anchors (OVS v2.17.9): `parse_SET_L4_SRC_PORT` around line 3488, `encode_SET_L4_SRC_PORT` around line 3470, `format_SET_L4_SRC_PORT` around line 3502. Header `include/openvswitch/ofp-actions.h` `struct ofpact_l4_port` (shared with SET_L4_DST_PORT). Datapath kernel `datapath/actions.c` functions `set_tcp` / `set_udp` / `set_sctp` handle port rewrite plus checksum recomputation. Branch baseline: OVS v2.17.9.

**Failure scenarios.** (a) `mod_tp_src:8080` without `tcp`, `udp`, or `sctp` in the match is rejected on OF 1.2 and later as `OFPBAC_MATCH_INCONSISTENT`; on OF 1.0 it is accepted but silently skipped on ICMP packets. Fix: add `tcp` or `udp` to the match. (b) PAT that forgets to rewrite an ephemeral source port unique per connection causes port collisions, ambiguous reverse NAT mappings, and broken connections. Fix: use a dynamic port pool through `nat(src=ip:1024-65535)`. (c) Rewriting tp_src to 0 produces an invalid reserved port; the kernel drops it or the application rejects it. Fix: validate port > 0.

**Synthetic exercise.** Mininet, two hosts h1 and h2 over bridge `s1`. On h2 run `nc -l -p 8080`. Install rule `priority=100,in_port=h1-eth0,tcp,actions=mod_tp_src:9999,output:h2-eth0`. On h1 run `nc h2 8080`. On h2 run `ss -tnp | grep 8080` to confirm the remote peer (h1) port is 9999 instead of the real ephemeral port. Verify the TCP checksum is correct (`tcpdump -v` does not print `bad cksum`).

**Observation and verification.** `ovs-ofctl dump-flows br0` shows `mod_tp_src:8080`. `ovs-appctl ofproto/trace br0 'in_port=1,tcp,tp_src=4001,...'` shows the line `Datapath actions: set(tcp(src=8080,...)),N`. `tcpdump -i wan-port -n -v` verifies the wire-level `Source port: 8080` instead of the original value.

**Comparison with other systems.** Linux netfilter `iptables -t nat -A POSTROUTING -p tcp -j SNAT --to-source 10.0.0.5:33000-33999` is stateful PAT with a port range. `tc filter ... action nat ingress src` provides default randomised port support. Cisco IOS `ip nat inside source ... overload` (PAT) is stateful with auto-allocation. NSX-T Edge gateway PAT is hidden inside the NAT engine. P4 `hdr.tcp.src_port = 0x1f90` is direct.

### 4.9.33.9 `mod_tp_dst`, the legacy OF 1.0 L4 destination port (TCP/UDP/SCTP) overwrite

**Concept.** `mod_tp_dst` is a legacy OF 1.0 action that overwrites the 2-byte L4 destination port of the packet. It applies to TCP, UDP, and SCTP. Equivalent to `set_field:<port>->tcp_dst` (or `udp_dst`, `sctp_dst`) in OF 1.2 and later. A core component of port-based NAT (port forwarding): combined with `mod_nw_dst` it produces a complete DNAT with port mapping. Symmetric to `mod_tp_src`: shares `struct ofpact_l4_port`, the same wire-encoding pattern, and only the target field differs. Action type: **specialised field rewrite**, OF 1.0 raw type `OFPAT10_SET_TP_DST`.

**History and context.** OpenFlow 1.0 (2009) added `OFPAT_SET_TP_DST` action type 10 with a 2-byte port payload. OVS supported it from the start. OF 1.2 (2011) introduced `set_field` with protocol-specific OXMs, like `mod_tp_src`. In OVN production: this action is the building block of port forwarding (exposing internal port 8080 on public port 80) plus the Logical Load Balancer pipeline rewrite VIP:80 to backend:8080. Cisco PAT inbound follows the same pattern.

**Position in the architecture.** At the OpenFlow protocol layer: `OFPAT10_SET_TP_DST` raw type 10 in OF 1.0 (protocol-agnostic); OF 1.2 and later encode to `OFPAT_SET_FIELD` with MFF_TCP_DST / MFF_UDP_DST / MFF_SCTP_DST depending on context (the parser inspects the `nw_proto` match to choose). At the OVS internal layer: shares `struct ofpact_l4_port` with `mod_tp_src`, distinguished by `ofpact_type = OFPACT_SET_L4_DST_PORT`. At the datapath kernel layer: translates to `OVS_ACTION_ATTR_SET` with an `OVS_KEY_ATTR_TCP` / `_UDP` / `_SCTP` payload setting the dst port.

**Role.** Primary role: port forwarding (DNAT-with-port-translation); expose an internal service on a port different from the public port. Secondary role: L4 load balancer (rewrite VIP:80 to backend:8080 when the backend listens on a different port), service redirection (HTTP 80 to HAProxy 8080 for debug). Blast radius: per-packet; the L4 checksum is auto-recomputed.

**Why it exists.** A common port-forwarding scenario: an internal server listens on a non-standard port (8080 for dev) but is exposed through the public standard port (80); moving the server's bind is not feasible. The `mod_tp_dst` action allows the gateway to rewrite the destination port before forwarding. Before OpenFlow, Linux netfilter `iptables -t nat -A PREROUTING -p tcp --dport 80 -j REDIRECT --to-port 8080` did this. OpenFlow exposes it programmably.

**Problems it solves.** Specifically: (a) basic port forwarding, (b) L4 load balancer with a backend port different from the frontend, (c) debug-time service redirection. Approach: action rewrites the L4 dst port plus a checksum recompute. Scope: stateless 1-to-1 mappings; full stateful LB with a backend pool combines with a `select` group plus `ct(commit, nat(dst=...))`.

**Importance.** Tier 2 medium. In OVN production it appears in the Logical Load Balancer pipeline (VIP:80 to backend:8080) plus in LR port-forwarding rules. Engineer must-know: the syntax is symmetric to `mod_tp_src`; pair with `mod_nw_dst` (rewrite the destination IP) when the backend is in a different subnet; the action only applies to packets with an L4 protocol.

**How it works.** Parser `lib/ofp-actions.c` function `parse_SET_L4_DST_PORT` (around line 3492 in v3.3.0) follows the same pattern as `parse_SET_L4_SRC_PORT`, calling `str_to_u16` to parse and storing into `ofpact_put_SET_L4_DST_PORT(pp->ofpacts)->port`. Formatter `format_SET_L4_DST_PORT` (around line 3507) prints `mod_tp_dst:%"PRIu16`. Encoder `encode_SET_L4_DST_PORT` (around line 3478) picks raw type `OFPAT_RAW_SET_TP_DST` for OF 1.0 or `OFPAT_SET_FIELD` for OF 1.2 and later. Datapath kernel: `OVS_ACTION_ATTR_SET` calls `set_tcp` / `set_udp` / `set_sctp` to rewrite the 2-byte dst port plus recompute the L4 checksum.

**Wire encoding.** OF 1.0 format: action header `type=10` (OFPAT10_SET_TP_DST) plus `length=8` plus a 2-byte port payload plus 2-byte padding. OF 1.2 and later format: `OFPAT_SET_FIELD` action type 25 with the OXM TLV `OXM_OF_TCP_DST` / `OXM_OF_UDP_DST` / `OXM_OF_SCTP_DST` of 2 bytes. Fully symmetric to `mod_tp_src`, only the type number is 10 instead of 9.

**Applied examples.**

```
priority=100,ip,tcp,tp_dst=80,actions=mod_tp_dst:8080,output:backend     # port forwarding 80 to 8080
priority=100,ip,nw_dst=203.0.113.10,tcp,tp_dst=80,
        actions=mod_nw_dst:10.0.0.10,mod_tp_dst:8080,output:lan-port      # full DNAT plus PAT
priority=100,udp,tp_dst=53,actions=mod_tp_dst:5353,output:dns-fallback  # DNS redirect
actions=set_field:8080->tcp_dst                                         # OF 1.2 and later, TCP
actions=set_field:5353->udp_dst                                         # OF 1.2 and later, UDP
```

A typical OVN-generated flow in the Logical LB pipeline:

```
priority=100,ip,nw_dst=203.0.113.10,tcp,tp_dst=80,
       actions=ct(commit,nat(dst=10.0.0.10:8080))
                                                                          # stateful equivalent
priority=100,ip,reg9=0xc0a80101,tcp,tp_dst=80,
       actions=mod_nw_dst:10.0.0.10,mod_tp_dst:8080,resubmit(,7)
                                                                          # stateless variant
```

**How an engineer masters operations.** Four skills: (1) **Read**: recognise `mod_tp_dst` in the OVN Logical LB plus LR DNAT pipeline; distinguish it from `mod_tp_src` by syntax. (2) **Edit**: switch new code to `set_field:<port>->tcp_dst` with a specific OXM; for stateful LB switch fully to `ct(commit, nat(dst=ip:port))`. (3) **Observe**: `ovs-ofctl dump-flows br0 | grep mod_tp_dst` counts port-forwarding rules. (4) **Debug**: `ofproto/trace 'in_port=1,tcp,tp_dst=80,...'` shows the line `set(tcp(dst=8080,...))`, confirming the datapath rewrite.

**Classification.** A field-rewrite action specialised for the L4 dst port; OF 1.0 raw type, protocol-agnostic; symmetric to `mod_tp_src`; no partial mask. Same class: `mod_tp_src`, `set_field:V->tcp_dst/udp_dst/sctp_dst`. Different from the `nat()` action: `nat()` is stateful with a port pool; `mod_tp_dst` is stateless constant rewrite.

**Source code.** `lib/ofp-actions.c` function anchors (OVS v2.17.9): `parse_SET_L4_DST_PORT` around line 3492, `encode_SET_L4_DST_PORT` around line 3478, `format_SET_L4_DST_PORT` around line 3507. Header `include/openvswitch/ofp-actions.h` `struct ofpact_l4_port` (shared with SET_L4_SRC_PORT). Datapath kernel `datapath/actions.c` functions `set_tcp` / `set_udp` / `set_sctp` handle dst port rewrite plus checksum recompute. Branch baseline: OVS v2.17.9.

**Failure scenarios.** (a) `mod_tp_dst:8080` without an L4 match prerequisite is rejected on OF 1.2 and later; on OF 1.0 it is accepted but silently skipped on ICMP or raw packets. Fix: add `tcp`, `udp`, or `sctp` to the match. (b) Port forwarding without rewriting the IP destination in a cross-host scenario means the packet arrives on the right port but with the wrong IP destination, and the backend host drops it. Fix: pair `mod_nw_dst` with `mod_tp_dst`. (c) A stateless dst-port rewrite leaves the return path with no reverse rewrite, so the client receives a reply on an unexpected port and the connection breaks. Fix: use the stateful `ct(commit, nat(dst=ip:port))`.

**Synthetic exercise.** Mininet, two hosts h1 and h2 over bridge `s1`. On h2 run `nc -l -p 8080` (a server listening on port 8080). Install rule `priority=100,in_port=h1-eth0,tcp,tp_dst=80,actions=mod_tp_dst:8080,output:h2-eth0` to forward port 80 to 8080. On h1 run `nc h2 80`. Verify the connection succeeds and h2 sees the incoming connection from h1 on port 8080.

**Observation and verification.** `ovs-ofctl dump-flows br0` shows `mod_tp_dst:8080`. `ovs-appctl ofproto/trace br0 'in_port=1,tcp,tp_dst=80,...'` shows the line `Datapath actions: set(tcp(dst=8080,...)),N`. `tcpdump -i backend-port -n -v` verifies the wire-level `Destination port: 8080` instead of 80.

**Comparison with other systems.** Linux netfilter `iptables -t nat -A PREROUTING -p tcp --dport 80 -j REDIRECT --to-port 8080` is the stateful equivalent. `iptables -t nat -A PREROUTING -p tcp --dport 80 -j DNAT --to-destination 10.0.0.10:8080` is full DNAT plus PAT. HAProxy and Nginx layer-7 reverse proxies achieve the same concept at the application layer (terminate plus reissue). Cisco IOS `ip nat inside source static tcp ... 80 ... 8080` is stateful. P4 `hdr.tcp.dst_port = 0x1f90` is direct.

### 4.9.33.10 `push_vlan`, push a VLAN tag onto the packet with a chosen ethertype

**Concept.** `push_vlan` is an OF 1.1 and later action that pushes one VLAN tag (a 4-byte 802.1Q tag or 802.1ad tag) onto the packet immediately after the ethernet header. The action only inserts the tag with the chosen TPID (Tag Protocol IDentifier, ethertype); it does not set the `vlan_vid`, `vlan_pcp`, or `cfi` values; the engineer must use a subsequent `set_field` (or the legacy `mod_vlan_vid`/`mod_vlan_pcp`) to assign the real values. Valid TPIDs: `0x8100` (802.1Q standard VLAN), `0x88a8` (802.1ad Q-in-Q outer tag). Action type: **header push**, OF 1.1 and later raw type `OFPAT11_PUSH_VLAN`.

**History and context.** OpenFlow 1.0 had no separate `push_vlan` or `pop_vlan`; instead it had only `mod_vlan_vid` plus `mod_vlan_pcp` (rewrite the existing tag) plus `strip_vlan` (remove the tag). OpenFlow 1.1 (2011) refactored: split into `OFPAT11_PUSH_VLAN` (action type 17) plus `OFPAT11_POP_VLAN` (action type 18) plus the legacy `OFPAT11_SET_VLAN_VID`. OVS 1.5 (2013) supports it. OF 1.3 and later standardised `push_vlan` with the TPID parameter for Q-in-Q (802.1ad outer tag). Verify: `lib/ofp-actions.c` function `parse_PUSH_VLAN` around line 3037 plus `encode_PUSH_VLAN` around line 3028 (OVS v3.3.0).

**Position in the architecture.** At the OpenFlow protocol layer: `OFPAT11_PUSH_VLAN` action type 17 in OF 1.1 and later; the action header has `type` (2 bytes), `length` (2 bytes), `ethertype` (2 bytes), and 2 bytes of padding. At the OVS internal layer: `struct ofpact_push_vlan` (`include/openvswitch/ofp-actions.h`) holds an `ovs_be16 ethertype`. At the datapath kernel layer: translates to `OVS_ACTION_ATTR_PUSH_VLAN` netlink attribute with `struct ovs_action_push_vlan { __be16 vlan_tpid; __be16 vlan_tci; }`.

**Role.** Primary role: VLAN-tag an untagged packet before forwarding to a trunk port. Secondary role: Q-in-Q (double-tagged) outer tag for service provider scenarios, VLAN translation (a `pop_vlan` plus `push_vlan` combo to change the VID), MPLS-in-VLAN tunnel encapsulation. Blast radius: per-packet; after the action the packet length increases by 4 bytes.

**Why it exists.** OpenFlow 1.0 only had `mod_vlan_vid` (rewrite existing tag) and `strip_vlan` (remove tag). The "untagged to tagged" pattern needed a workaround: push raw bytes manually or use an NXM extension. OF 1.1 added an explicit `push_vlan` to standardise this scenario; OF 1.3 and later added the TPID parameter to support Q-in-Q (802.1ad outer tag with TPID `0x88a8`). Another motivation: SDN in cloud data centres needs dynamic per-tenant tagging (for example, OVN trunk ports), and the system cannot assume packets already have a tag.

**Problems it solves.** Specifically: (a) tag an untagged packet when forwarding to a trunk, (b) Q-in-Q double tag for service providers, (c) atomic VLAN translation (avoiding intermediate state when popping plus setting). Approach: the action inserts 4 bytes of tag immediately after the MAC header with the specified ethertype; the 2-byte vlan_tci field is set later (typically with `set_field:V->vlan_vid`). Scope: L2 trunking, Q-in-Q, VLAN-based tenant isolation.

**Importance.** Tier 2 medium. In OVN trunk ports plus Logical Switches with localnet (tenant traffic going to a physical VLAN network), `push_vlan` appears at Logical_Flow stage `ls_out_port_sec_l2`. Engineer must-know: the syntax `push_vlan:0x8100` (TPID only, no VID or PCP), the requirement to pair with `set_field:<vid>->vlan_vid` immediately after, and the fact that the action is only valid for an untagged packet or for the Q-in-Q outer tag scenario.

**How it works.** Parser `lib/ofp-actions.c` function `parse_PUSH_VLAN` (around line 3037 in v3.3.0): parse the argument string into a 16-bit ethertype via `str_to_u16`, validate the ethertype is in `{0x8100, 0x88a8}`, store into `ofpact_put_PUSH_VLAN(pp->ofpacts)->ethertype`. Formatter `format_PUSH_VLAN` (around line 3050) prints `push_vlan:0x%"PRIx16` with the hex ethertype. Encoder `encode_PUSH_VLAN` (around line 3028) emits the `OFPAT11_PUSH_VLAN` action with 2-byte ethertype plus 2-byte padding. Datapath kernel: `OVS_ACTION_ATTR_PUSH_VLAN` netlink attribute causes the kernel to insert a 4-byte tag (TPID 2 bytes plus TCI 2 bytes with VID=0 and PCP=0 by default) immediately after the MAC header, shift the payload, and recompute checksums where the protocol stack requires it. After the action, the flow context's `vlan_tci` field is 0 (a `set_field:V->vlan_vid` is required to set the real value).

**Wire encoding.** OF 1.1 and later format: action header `type=17` (OFPAT11_PUSH_VLAN) plus `length=8` plus `ethertype` (2 bytes) plus 2-byte padding (8-byte aligned). Datapath kernel netlink attribute `OVS_ACTION_ATTR_PUSH_VLAN`: an nlattr header (type=20, length=8) plus `struct ovs_action_push_vlan` 4 bytes (vlan_tpid 2 plus vlan_tci 2). At the wire level, after the action the 4-byte tag (TPID 2 plus TCI 2) is inserted between the MAC header (14 bytes) and the payload; the total frame size grows by 4 bytes.

**Applied examples.**

```
actions=push_vlan:0x8100,set_field:100->vlan_vid,output:trunk            # tag VLAN 100 standard
actions=push_vlan:0x8100,set_field:4096->vlan_vid,output:trunk           # CFI=1 (VID=0 with CFI bit)
actions=push_vlan:0x88a8,set_field:200->vlan_vid,
        push_vlan:0x8100,set_field:100->vlan_vid,output:provider          # Q-in-Q outer 200 inner 100
actions=pop_vlan,push_vlan:0x8100,set_field:300->vlan_vid,output:trunk   # VLAN translation 100 to 300
priority=100,vlan_tci=0,actions=push_vlan:0x8100,
        set_field:42->vlan_vid,set_field:5->vlan_pcp,output:uplink        # tag with QoS priority 5
```

A typical OVN-generated flow at Logical_Flow stage `ls_out_port_sec_l2` for a trunk port with a localnet tag:

```
priority=100,reg14=0x3,actions=push_vlan:0x8100,set_field:0x42->vlan_vid,output:patch-localnet
```

**How an engineer masters operations.** Four skills: (1) **Read**: recognise that the "push_vlan plus set_field vlan_vid" pattern always appears as a pair (after pushing, the VID must be set); distinguish `0x8100` (standard) from `0x88a8` (Q-in-Q outer). (2) **Edit**: write a trunk-tagging flow always pairing `push_vlan` with `set_field:V->vlan_vid` (plus optional `set_field:P->vlan_pcp` for QoS). For Q-in-Q write two push-and-set pairs in inner-then-outer order. (3) **Observe**: `ovs-ofctl dump-flows br0 | grep push_vlan` counts tagging rules; `tcpdump -i trunk -n -e` shows the 4-byte tag at wire level. (4) **Debug**: `ofproto/trace 'in_port=1,...'` shows the line `Datapath actions: push_vlan(vid=100,pcp=0),N`, confirming the datapath inserted the tag.

**Classification.** A header-push action (insert bytes into the packet, increasing length); OF 1.1 and later standard. Properties: stateful (modifies packet structure), not a terminator, must pair with `set_field` for the VID. Same class: `pop_vlan` (symmetric remove), `push_mpls`/`pop_mpls` (MPLS label), `push_pbb`/`pop_pbb` (PBB B-tag). Different from `mod_vlan_vid`: `mod_vlan_vid` rewrites an existing tag (no length change), whereas `push_vlan` inserts a new tag (length grows by 4 bytes).

**Source code.** `lib/ofp-actions.c` function anchors (OVS v2.17.9): `parse_PUSH_VLAN` around line 3037, `encode_PUSH_VLAN` around line 3028, `format_PUSH_VLAN` around line 3050. Header `include/openvswitch/ofp-actions.h` `struct ofpact_push_vlan`. Datapath kernel `datapath/actions.c` function `push_vlan` handles byte insertion plus skb metadata update; netlink protocol `include/uapi/linux/openvswitch.h` enum `OVS_ACTION_ATTR_PUSH_VLAN` plus `struct ovs_action_push_vlan`. Branch baseline: OVS v2.17.9.

**Failure scenarios.** (a) `push_vlan:0x8100` without a following `set_field:V->vlan_vid` inserts the tag with VID=0 (priority-tag only); the receiving host drops the frame at the L2 filter or the bridge mishandles it. Fix: always pair `push_vlan` with `set_field:V->vlan_vid`. (b) `push_vlan:0x1234` with an invalid ethertype is rejected by the parser with `push_vlan: ethertype must be 0x8100 or 0x88a8`. Fix: choose one of the two valid values. (c) Q-in-Q written in the wrong order (outer first) places tags on the wire in reversed order; downstream switches misparse outer versus inner. Fix: write the inner push first (closest to payload), the outer push second (closest to MAC header).

**Synthetic exercise.** Mininet, two hosts h1 and h2 over bridge `s1` with h2 on a trunk port. Install rule `priority=100,in_port=h1-eth0,vlan_tci=0,actions=push_vlan:0x8100,set_field:42->vlan_vid,output:h2-eth0`. On h2 run `tcpdump -i h2-eth0 -n -e -v`, then run `h1 ping h2`. Verify the wire-level packet on h2 has the 4-byte tag `0x8100 0x002a` (TPID 0x8100 plus TCI VID=42=0x2a) inserted right after the MAC. Frame size is 4 bytes larger than the original.

**Observation and verification.** `ovs-ofctl dump-flows br0 -O OpenFlow13` shows `push_vlan:0x8100`. `ovs-appctl ofproto/trace br0 'in_port=1,vlan_tci=0,...'` shows the line `Datapath actions: push_vlan(tpid=0x8100,vid=0,pcp=0),set(vlan(vid=42,...)),N`. `tcpdump -i trunk -n -e -v` verifies the wire-level frame carries the `802.1Q vlan 42` annotation; `tshark -V -i trunk` shows the full 16-bit TCI breakdown (VID 12 bits, PCP 3 bits, DEI 1 bit).

**Comparison with other systems.** Linux bridge `bridge vlan add dev eth0 vid 42 pvid untagged` sets the bridge port to auto-push tag VID 42 for every untagged packet entering, an equivalent concept but not exposed as a programmable action (config-time setup). `tc filter ... action vlan push id 42` in `tc` provides a programmable equivalent to OpenFlow's `push_vlan`. Cisco IOS `switchport trunk allowed vlan 42` plus interface mode trunk or access is also config-time. P4 `hdr.vlan.setValid(); hdr.vlan.tpid = 0x8100; hdr.vlan.vid = 42;` is a direct assignment in a control block.

### 4.9.33.11 `pop_vlan`, remove the outermost VLAN tag from the packet

**Concept.** `pop_vlan` is an OF 1.1 and later action that removes the outermost 802.1Q or 802.1ad tag from the packet right after the ethernet header. After the action the packet shrinks by exactly 4 bytes (TPID 2 bytes plus TCI 2 bytes), and the flow context's `vlan_tci` field resets to 0. The action takes no argument; it always removes a single outermost tag. To remove an inner Q-in-Q tag, the action must be invoked a second time. Action type: **header pop**, OF 1.1 and later raw type `OFPAT_RAW11_POP_VLAN`.

**History and context.** OpenFlow 1.0 (2009) only had `strip_vlan` (remove the single tag, no Q-in-Q layered pop). OpenFlow 1.1 (2011) refactored into the symmetric pair `OFPAT_RAW11_POP_VLAN` (action type 18) plus `OFPAT_RAW11_PUSH_VLAN` (action type 17), where `pop_vlan` always removes the outermost tag and supports layered Q-in-Q. OVS 1.5 (2013) supports it; OVS keeps the legacy alias `strip_vlan` in the parser for OF 1.0 rules but defaults output to the `pop_vlan` format on OF 1.1 and later. Verify upstream: `lib/ofp-actions.c` enum `OFPAT_RAW11_POP_VLAN` around line 388, function `parse_POP_VLAN` around line 1700, `encode_POP_VLAN` around line 1600 (OVS v3.3.0).

**Position in the architecture.** At the OpenFlow protocol layer: `OFPAT_RAW11_POP_VLAN` action type 18, action header has `type` (2 bytes), `length` (8 bytes), and 4-byte padding (no argument body). At the OVS internal layer: `struct ofpact_null` (the same struct as every argument-less action such as `pop_pbb` and `dec_ttl`) in `include/openvswitch/ofp-actions.h`; only the common `struct ofpact` header, no extra fields. At the datapath kernel layer: translates to `OVS_ACTION_ATTR_POP_VLAN` netlink attribute with no payload (length=4 bytes header only); on kernel 5.0 and later with a recompiled OVS module, the action is processed in place on the skb through the `pop_vlan` function in `datapath/actions.c`.

**Role.** Primary role: remove a tag so the packet can be forwarded to an access port (untagged interface). Secondary role: VLAN translation pair `pop_vlan` plus `push_vlan:0x8100,set_field:V->vlan_vid` to change the VID, Q-in-Q outer-tag de-encapsulation (keeping the inner tag), tunnel termination for service-provider scenarios. Blast radius: per-packet; after the action the packet length shrinks by 4 bytes; only applies when the current `vlan_tci` field is non-zero; on an untagged packet the action becomes a no-op but the parser still accepts it.

**Why it exists.** OpenFlow 1.0 `strip_vlan` only handled one tag, with no outer-versus-inner distinction for Q-in-Q. As 802.1ad (Q-in-Q) became common in service-provider data centres and carrier Ethernet, the OF 1.1 spec needed a layered pop action to remove tags one at a time in order. Another motivation: the symmetric `push_vlan`/`pop_vlan` pair is conceptually easier for controllers; previously `strip_vlan` had no explicit counterpart (a tag was implicitly created by `mod_vlan_vid` on an untagged packet).

**Problems it solves.** Specifically: (a) remove the outermost tag to forward to an access port, (b) Q-in-Q de-encapsulation per layer, (c) atomic VLAN translation when combined with `push_vlan`. Approach: an action that says "remove the outermost tag"; no argument is needed because the position is always known (4 bytes immediately after the MAC header); the spec stipulates that on an untagged packet the action is a no-op. Scope: L2 trunk-to-access conversion, Q-in-Q tunnel egress, VLAN-based tenant isolation.

**Importance.** Tier 2 medium. In OVN, `pop_vlan` appears at Logical_Flow stage `ls_in_port_sec_l2` for localnet ingress (tenant traffic from a physical VLAN network into a logical switch needs to drop the tag to consume it in the logical context). In data-centre tenant-isolation patterns, every trunk-to-access rule has `pop_vlan` immediately before `output`. Engineer must-know: the syntax `pop_vlan` takes no argument, the no-op behaviour on untagged packets, the `vlan_tci` reset to 0 after the action, and the fact that the action only removes the outermost tag, so Q-in-Q requires two invocations.

**How it works.** Parser `lib/ofp-actions.c` function `parse_POP_VLAN` (around line 1700 in v3.3.0): consume the keyword `pop_vlan` (or the legacy alias `strip_vlan`) without an argument, call `ofpact_put_STRIP_VLAN` (an internal alias) to emit an empty `struct ofpact` into the ofpacts buffer. Formatter `format_POP_VLAN` prints `pop_vlan` with no suffix. Encoder `encode_POP_VLAN` (around line 1600) emits the `OFPAT_RAW11_POP_VLAN` 8-byte action header (4-byte type+length plus 4-byte padding). Datapath kernel: `OVS_ACTION_ATTR_POP_VLAN` netlink attribute leads the kernel `pop_vlan` function in `datapath/actions.c` to check `skb_vlan_tag_present(skb)` or parse the 802.1Q ethertype; when a tag is present, memmove the MAC header up by 4 bytes to overwrite the tag, decrement the skb length, recompute `skb->protocol` from the new ethertype, clear `skb->vlan_tci` and `skb->vlan_proto`. When untagged, return 0 silently.

**Wire encoding.** OF 1.1 and later format: action header `type=18` (OFPAT_RAW11_POP_VLAN) plus `length=8` plus 4-byte padding (8-byte aligned, no argument body). Datapath kernel netlink attribute `OVS_ACTION_ATTR_POP_VLAN`: only the nlattr header (type=21, length=4) with no payload. At the wire level, after the action the 4-byte tag (TPID 2 plus TCI 2) is removed from immediately after the MAC header, the payload shifts up, and the total frame size shrinks by 4 bytes.

**Applied examples.**

```
actions=pop_vlan,output:access                                       # remove tag and send to access port
priority=100,vlan_vid=42,actions=pop_vlan,output:5                   # match VID 42, remove, output
actions=pop_vlan,push_vlan:0x8100,set_field:300->vlan_vid,output:5   # VLAN translation 100 to 300
actions=pop_vlan,pop_vlan,output:6                                   # remove two tags (Q-in-Q outer plus inner)
priority=200,dl_type=0x88a8,actions=pop_vlan,resubmit(,2)            # remove outer Q-in-Q, continue at table 2
```

A typical OVN-generated flow at Logical_Flow stage `ls_in_port_sec_l2` for localnet ingress with a VLAN tag:

```
priority=150,inport==localnet,vlan_vid=0x42,actions=pop_vlan,resubmit(,17)
```

**How an engineer masters operations.** Four skills: (1) **Read**: distinguish `pop_vlan` (OF 1.1 and later, no argument) from `strip_vlan` (OF 1.0 alias, accepted by the OVS parser); recognise that the trunk-to-access pattern always has `pop_vlan` before `output`. (2) **Edit**: write egress-access-port rules always starting with `pop_vlan`; for VLAN translation write the pair `pop_vlan,push_vlan:0x8100,set_field:V->vlan_vid` in that order. (3) **Observe**: `ovs-ofctl dump-flows br0 | grep -E 'pop_vlan|strip_vlan'` counts the de-tagging rules. (4) **Debug**: `ofproto/trace 'in_port=trunk,dl_vlan=42,...'` shows the line `Datapath actions: pop_vlan,N`, confirming the datapath removed the tag; if the trace shows `set(vlan(vid=...))` instead, an older kernel uses the set-TCI path rather than direct pop.

**Classification.** A header-pop action (remove bytes from the packet, decreasing length); OF 1.1 and later standard. Properties: stateless (does not depend on state outside the packet), not a terminator, no required pairing with another action (different from `push_vlan`, which always pairs with `set_field`). Same class: `pop_mpls` (symmetric to `push_mpls`), `pop_pbb` (symmetric to `push_pbb`), the OF 1.5 generic `decap`. Different from `strip_vlan`: `strip_vlan` is the OF 1.0 alias that removes only the single tag with no outer-versus-inner distinction; `pop_vlan` in OF 1.1 and later is the standard layered pop for Q-in-Q.

**Source code.** `lib/ofp-actions.c` function anchors (OVS v2.17.9): `parse_POP_VLAN` around line 1700, `encode_POP_VLAN` around line 1600, `format_POP_VLAN` around line 1750; enum `OFPAT_RAW11_POP_VLAN` line 388 in `enum ofp_raw_action_type`. Header `include/openvswitch/ofp-actions.h` `struct ofpact_null` (shared by every argument-less action). Datapath kernel `datapath/actions.c` function `pop_vlan` handles byte removal plus skb metadata cleanup; netlink protocol `include/uapi/linux/openvswitch.h` enum `OVS_ACTION_ATTR_POP_VLAN`. Branch baseline: OVS v2.17.9.

**Failure scenarios.** (a) `pop_vlan` on an untagged packet is a no-op; no error, but counters do not rise as expected; the engineer thinks the rule did not match and debugs for a long time. Fix: match `vlan_tci=0x1000/0x1000` (CFI bit set) or an explicit `vlan_vid` so the action only triggers when a tag is present. (b) Removing the Q-in-Q outer tag but forgetting the inner means the packet leaving the access port still carries one tag and the host drops the frame. Fix: write `pop_vlan,pop_vlan` sequentially for the two layers. (c) `pop_vlan` placed after `set_field:V->vlan_vid` removes the tag that was just set, contradicting the intent. Fix: check action order; `pop_vlan` always precedes the set or push, or only use `set_field` when a tag is already present.

**Synthetic exercise.** Mininet, two hosts: h1 (trunk, sending VLAN 42 packets) and h2 (access). Install rule `priority=100,in_port=h1-eth0,vlan_vid=42,actions=pop_vlan,output:h2-eth0`. On h1 run `vconfig add h1-eth0 42; ifconfig h1-eth0.42 10.0.0.1/24 up; ping 10.0.0.2`. On h2 run `tcpdump -i h2-eth0 -n -e` and verify the packet has no tag (only the 14-byte MAC header before the IP). Frame size is 4 bytes smaller than the original.

**Observation and verification.** `ovs-ofctl dump-flows br0 -O OpenFlow13` shows `pop_vlan` (no suffix); OF 1.0 dump shows the legacy `strip_vlan`. `ovs-appctl ofproto/trace br0 'in_port=1,dl_vlan=42,...'` shows the line `Datapath actions: pop_vlan,N`, confirming the action. `tcpdump -i access-port -n -e -v` verifies the wire-level frame has no `802.1Q vlan` annotation. `ovs-appctl dpif/show-flows | grep pop_vlan` counts the megaflows that hit pop_vlan.

**Comparison with other systems.** Linux bridge `bridge vlan add dev eth0 vid 42 untagged` configures the port to remove tag VID 42 when forwarding, a config-time setup that is not programmable per flow. `tc filter ... action vlan pop` in `tc` provides a programmable equivalent to `pop_vlan`. Cisco IOS `switchport mode access` plus `switchport access vlan 42` is config-time tag removal at the access port. P4 `hdr.vlan.setInvalid();` is a direct assignment in a control block. NSX-T transport-node de-encapsulation of VLAN at the edge node is the same concept.

### 4.9.33.12 `learn`, install a new rule into table N from the current packet

**Concept.** `learn` is the most powerful NXM (Nicira) action: from the packet being processed, generate and install a new flow rule into the specified table, with match fields and actions taken from the field values of the current packet. The action does not modify the current packet; it only has the side-effect of installing a rule for future packets. This is the mechanism that allows the data plane to self-program without a controller round-trip, producing the equivalent of MAC learning, ARP cache, flow caching, or simple connection tracking. Action type: **stateful side-effect**, NXM Nicira extension in OVS 1.6 and later, not a terminator.

**History and context.** The action appeared in OVS 1.6 (2012) as the NXM extension `NXAST_LEARN` so an OVS MAC-learning rule could install a reverse-direction rule from an ARP or IP packet, producing behaviour similar to a Linux `bridge` but customisable through a flow rule. OVS 2.4 (2015) added the full `learn(...)` syntax to `ovs-ofctl`. OpenFlow 1.5 (2014) did not standardise `learn`; it remains an NXM extension and controllers must opt in via the Nicira experimenter ID. Verify upstream: `lib/ofp-actions.c` enum `NXAST_RAW_LEARN` line 559, function `parse_LEARN` around line 3800, `encode_LEARN` around line 3700 (OVS v3.3.0).

**Position in the architecture.** At the OpenFlow protocol layer: `NXAST_RAW_LEARN` Nicira experimenter (subtype 16); the action header has the Nicira experimenter wrapper plus a variable-length body with a list of `flow_mod_spec` entries: a 2-byte spec header plus N bytes of src/dst spec. At the OVS internal layer: `struct ofpact_learn` (`include/openvswitch/ofp-actions.h`) holds `idle_timeout`, `hard_timeout`, `priority`, `cookie`, `flags`, `table_id`, `fin_idle_timeout`, `fin_hard_timeout`, `result_dst`, `n_specs`, plus a flexible array of `struct ofpact_learn_spec`. At the datapath layer: the kernel does **not** handle `learn`; the translation (xlate.c) executes the action immediately in userspace ofproto-dpif on upcall, generates an internal flow_mod, and pushes it through the ofproto add-flow pipeline.

**Role.** Primary role: automatic data-plane MAC learning (install the reverse-direction rule after seeing a frame's source MAC). Secondary role: ARP cache rule, connection learning for a simple stateful firewall, flow-distribution learning by L4 hash, dynamic ACL hit-counter. Blast radius: install one rule into the specified table per action execution; high execution frequency means high rule-install frequency, which can blow up the table size.

**Why it exists.** Before OVS 1.6, MAC learning had to be implemented in the controller: the switch upcalled every unmatched packet, the controller looked at the source MAC, the controller issued a flow_mod to install a rule. A 1-10 ms round-trip times millions of MACs does not scale. The `learn` action pushes the logic into the switch: the switch looks at the packet itself and installs the rule itself. A second motivation: it allows writing a simple stateful firewall in the rule language; an inbound packet triggers `learn` to install an outbound rule matching the reverse 5-tuple, with no full conntrack required.

**Problems it solves.** Specifically: (a) controller-free MAC learning, (b) automatic ARP cache, (c) reverse-flow for a simple stateful firewall, (d) dynamic anti-spoofing. Approach: the action defines a template rule (match plus action) in the syntax `learn(table=N, priority=P, idle_timeout=T, NXM_OF_ETH_DST[]=NXM_OF_ETH_SRC[], output:NXM_OF_IN_PORT[])`; the `[]` notation means "load from the current packet". When the action executes, the switch substitutes the current packet's field values into the template, generates a concrete rule, and installs it into the chosen table. Scope: data-plane self-programming, OVS-only.

**Importance.** Tier 2 medium. In OVS standalone deployments (no controller), `learn` is the primary mechanism that lets the switch behave like a Linux bridge with MAC learning. In OVN, `learn` appears at Logical_Flow stage `ls_in_l2_lkup` for FDB MAC learning when `mac_learn` is enabled in the NB Logical_Switch options. Engineer must-know: the syntax `learn(table=T, priority=P, idle_timeout=T, MATCH_SPEC, ACTION_SPEC)` plus the flag `delete_learned` (delete the rule when the parent rule is deleted), `fin_idle_timeout` (shorten the timeout when a TCP FIN is observed), `result_dst` (store the new rule's cookie into a register).

**How it works.** Parser `lib/ofp-actions.c` function `parse_LEARN` (around line 3800 in v3.3.0): tokenise the string inside `(...)`, parse each spec in `key=value` form. Recognised keys: `table`, `priority`, `idle_timeout`, `hard_timeout`, `cookie`, `delete_learned`, `fin_idle_timeout`, `fin_hard_timeout`, `result_dst`, plus match specs in the forms `field[]=value`, `field[]=field2[]`, or `load:value->field[]`. Each spec emits a `struct ofpact_learn_spec` with src/dst fields encoded. Encoder `encode_LEARN` (around line 3700) emits the Nicira experimenter wrapper plus serialises the specs. Downstream translation (xlate.c function `xlate_learn_action`): when the upcall hits a rule with a learn action, xlate substitutes the fields from the flow context into the template, generates a `struct ofputil_flow_mod`, and sends it to the ofproto-dpif add-flow pipeline. The kernel datapath does **not** see `learn`; it only sees the new rule that was installed (via flow_mod), and it executes that rule like any other.

**Wire encoding.** NXM Nicira experimenter format: action header `type=NXAST` (65535) plus `length=variable` plus 4-byte experimenter ID `0x00002320` (Nicira) plus 2-byte subtype `16` (NXAST_LEARN) plus a body containing a fixed header (idle_timeout, hard_timeout, priority, cookie, flags, table_id, ...) plus a variable-length spec list. Each spec has a 2-byte header (src_type:1 plus dst_type:1 plus n_bits:6) plus a variable-length src/dst payload. The total action can be 50 to 200 bytes depending on the number of specs. Datapath kernel: no wire format because the kernel does not handle this.

**Applied examples.**

```
# Simple MAC learning: from source MAC plus in_port, install reverse-direction rule
actions=learn(table=10,idle_timeout=300,
              NXM_OF_ETH_DST[]=NXM_OF_ETH_SRC[],
              output:NXM_OF_IN_PORT[]),
        resubmit(,10)

# Stateful reverse-flow for TCP: install outbound rule matching the reverse 5-tuple
actions=learn(table=20,priority=100,idle_timeout=60,fin_idle_timeout=5,
              eth_type=0x0800,nw_proto=6,
              NXM_OF_IP_SRC[]=NXM_OF_IP_DST[],
              NXM_OF_IP_DST[]=NXM_OF_IP_SRC[],
              NXM_OF_TCP_SRC[]=NXM_OF_TCP_DST[],
              NXM_OF_TCP_DST[]=NXM_OF_TCP_SRC[],
              output:NXM_OF_IN_PORT[]),
        normal

# OVN-style L2 lookup learning with delete_learned (cascade delete when parent removed)
actions=learn(table=11,idle_timeout=0,delete_learned,
              NXM_OF_ETH_DST[]=NXM_OF_ETH_SRC[],
              load:NXM_OF_IN_PORT[]->NXM_NX_REG14[0..15])
```

**How an engineer masters operations.** Four skills: (1) **Read**: distinguish the syntax `field[]=field2[]` (copy from the current packet) from `field[]=value` (literal) and `load:value->field[]` (an action on the newly generated rule); know that the comma inside `learn(...)` is a spec separator, not an action separator. (2) **Edit**: start from the standard MAC-learning template and extend by adding specific match-specs (dl_type, nw_proto). Always set `idle_timeout` so rules clean themselves up; consider `delete_learned` to cascade-delete when the parent rule is removed. (3) **Observe**: `ovs-ofctl dump-flows br0 table=10` shows the learned rules; the count grows linearly with the number of distinct source MACs. `ovs-appctl coverage/show | grep learn` counts the number of `learn` actions executed. (4) **Debug**: `ofproto/trace` does not trigger `learn` (the trace is dry-run); to test in practice you must send a real packet and use `dump-flows` to see the new rule.

**Classification.** A stateful side-effect action (generates a new rule, does not modify the current packet); NXM Nicira extension, OVS-only. Properties: not a terminator, idempotent at the packet level (multiple executions on identical packets generate identical rules), but with side-effects on table state. Same class: `multipath` (NXM, output selection by hash, no side-effect), `bundle` (NXM, group selection), `conjunction` (NXM, multi-field match composition), `note` (NXM, no-op metadata). Different from `controller(...)`: `controller` pushes the packet to the controller for the controller to issue flow_mod, with a round-trip; `learn` installs a rule on the local switch with no round-trip.

**Source code.** `lib/ofp-actions.c` function anchors (OVS v2.17.9): `parse_LEARN` around line 3800, `encode_LEARN` around line 3700, `format_LEARN` around line 3850; enum `NXAST_RAW_LEARN` line 559 in `enum ofp_raw_action_type`. Header `include/openvswitch/ofp-actions.h` `struct ofpact_learn` plus `struct ofpact_learn_spec`. Translation `ofproto/ofproto-dpif-xlate.c` function `xlate_learn_action` (substitution plus flow_mod emission). Datapath kernel: no handler, because `learn` runs entirely in userspace ofproto-dpif. Branch baseline: OVS v2.17.9.

**Failure scenarios.** (a) `learn(...)` without `idle_timeout` plus high distinct-MAC traffic blows up table N; `ovs-ofctl dump-flows table=N | wc -l` rises into millions and `ovs-vswitchd` consumes many GB of RAM. Fix: always set `idle_timeout=300` (or lower). (b) `learn(...)` with a missing match-spec (for example, only matching dl_dst, not matching VLAN) generates a rule with too-broad a match, which conflicts with other rules. Fix: ensure the template match-spec is specific enough. (c) A parent rule with `learn(...)` is removed via `del-flow` but the learned rule does not disappear, leaving an "orphan" rule until idle_timeout. Fix: use the `delete_learned` flag for cascade delete on parent removal.

**Synthetic exercise.** Mininet, one OVS bridge plus four hosts (h1-h4). Install two rules: `table=0,priority=100,actions=learn(table=10,idle_timeout=300,NXM_OF_ETH_DST[]=NXM_OF_ETH_SRC[],output:NXM_OF_IN_PORT[]),resubmit(,10)` plus `table=10,priority=0,actions=flood`. Initially `dump-flows br0 table=10` shows only the miss-flood rule. `h1 ping h2` causes a reverse-direction rule (dst=MAC(h1), out=port(h1)) to appear in table=10. Then `h3 ping h1` and `h4 ping h1` produce three rules in table=10, each matching a different dst MAC. Verify the learn-rule's `n_packets` counter is 0 when the reply hits the learned rule directly (rather than flooding).

**Observation and verification.** `ovs-ofctl dump-flows br0 table=10 -O OpenFlow13` shows the learned rules, with cookies inherited from the parent rule. `ovs-appctl coverage/show | grep -E 'learn|ofproto_flush'` counts `learn` executions plus flush count. `ovs-appctl ofproto/trace br0 'in_port=1,...'` shows "Action: learn(...)" in the action list but does not show the new rule (the trace is dry-run). `ovs-vsctl --columns=other_config list Open_vSwitch | grep n-handler-threads` affects the concurrency of `learn` execution.

**Comparison with other systems.** Linux bridge `mac-learning` is built in: the bridge maintains the FDB through source MAC, but it is not exposed as a programmable interface. P4 register plus table can implement `learn` via an extern plus a control-plane callback (the digest mechanism). Cisco IOS MAC address-table dynamic is the fixed-function equivalent. NSX-T MAC learning is exposed through the Logical_Switch profile at config time. An SR-IOV NIC with hardware MAC-learning offload is semantically equivalent but not programmable per rule.

### 4.9.33.13 `conjunction`, multi-dimensional match composition through N conjuncts

**Concept.** `conjunction(id, k/n)` is an NXM action that replaces rule explosion when an N-dimensional match is needed (for example, src_ip cross dst_ip cross port). Instead of writing N times M times K explicit rules, the engineer writes N plus M plus K rules, each with the action `conjunction(id, k/n)`; the switch only "really matches" when the packet hits at least one rule in each of the N dimensions sharing the same id. OpenFlow 1.5 (2014) standardised this as `OFPACT_CONJUNCTION`, but OVS keeps the original NXM wire format; the semantics are identical. Action type: **match composition pseudo-action**, NXM OVS 2.4 and later, OF 1.5, not a terminator.

**History and context.** The action appeared in OVS 2.4 (2015) as the NXM extension `NXAST_CONJUNCTION` to solve rule explosion in SDN ACL controllers (for example, OpenStack security groups with 1,000 IPs times 100 ports times 2 directions equals 200,000 rules). OF 1.5 (2014) defined the conjunction concept, but OVS chose to keep the original NXM wire format for backward compatibility. OVN has used `conjunction` for ACL match aggregation since day one (2015). Verify upstream: `lib/ofp-actions.c` enum `NXAST_RAW_CONJUNCTION` line 628, function `parse_CONJUNCTION` around line 4100, `encode_CONJUNCTION` around line 4050 (OVS v3.3.0).

**Position in the architecture.** At the OpenFlow protocol layer: `NXAST_RAW_CONJUNCTION` Nicira experimenter; the action body has 4-byte `id` (uint32) plus 1-byte `clause` (k, 1-indexed) plus 1-byte `n_clauses` (n, 2 to 255) plus padding. At the OVS internal layer: `struct ofpact_conjunction` (`include/openvswitch/ofp-actions.h`) holds `id` plus `clause` plus `n_clauses`. At the classifier layer: `lib/classifier.c` handles conjunction through "conjunctive flow matching"; the classifier maintains per-id bookkeeping and checks the number of clause hits per packet. At the datapath layer: the kernel does not see `conjunction`; conjunction is resolved entirely in the userspace classifier; only the "real" rule (the one that actually outputs) emits a megaflow to the kernel.

**Role.** Primary role: rule-explosion reduction for multi-dimensional ACL matches (security group, distributed firewall). Secondary role: efficient port-range match (port 1024 to 65535 written as one conjunct with N range rules instead of N single-port rules), tag-based ACL (tag cross port cross direction). Blast radius: classifier-level optimisation, no packet modification, no datapath action emission; every rule containing `conjunction(id, k/n)` never directly executes other actions in its action list (only the "real" rule outside the conjunction set executes).

**Why it exists.** Rule explosion is the classic ACL problem: with N IPs times M ports times K directions, the rule count is N times M times K, potentially in the millions in a data centre. Before `conjunction`, the controller had to precompute the entire cross product and push it to the switch, costing RAM and increasing classifier-lookup CPU linearly. `conjunction` allows writing N plus M plus K rules separately and letting the switch compose the match itself; RAM drops from product to sum. A second motivation: OVN needs an expressive ACL DSL (the NB ACL match string `"ip4.src == 10.0.0.0/24 && tcp.dst == {80, 443, 8080}"`) that compiles naturally to conjunction.

**Problems it solves.** Specifically: (a) rule explosion N times M times K becomes N plus M plus K, (b) efficient port-range match, (c) tag-based composable ACL. Approach: the action does not directly emit packet processing; it is a "vote" in classifier composition. When a rule with `conjunction(id, k/n)` matches, the classifier increments the clause-k counter for that id; when n out of n clauses hit for the same packet plus the same id, the classifier treats the id as "really matched" and looks up a specific rule (priority equal to the conjunction-rule priority plus 1) to execute the real action. Scope: classifier-level match composition, an OVS-only optimisation.

**Importance.** Tier 2 medium but critical for OVN production. Every OVN ACL (SB Logical_Flow stages `ls_in_acl` and `ls_out_acl`) compiles to `conjunction` when the match has at least two dimensions. Engineer must-know: the syntax `conjunction(id, k/n)` where `id` is a 32-bit value unique within the table, `k` is the 1-indexed clause number, `n` is the total clause count; n must be at least 2 (n=1 is meaningless). The engineer must understand: a rule containing `conjunction` does not "execute" an output action; only the "real" rule (same id, same priority offset) executes.

**How it works.** Parser `lib/ofp-actions.c` function `parse_CONJUNCTION` (around line 4100 in v3.3.0): tokenise the `conjunction(id, k/n)` argument list, parse `id` via `str_to_u32`, parse `k/n` via `sscanf("%u/%u")`, validate `n >= 2 && n <= 64 && k >= 1 && k <= n`, emit a `struct ofpact_conjunction`. Formatter `format_CONJUNCTION` (around line 4150) prints `conjunction(<id>,<k>/<n>)`. Encoder `encode_CONJUNCTION` (around line 4050) emits the Nicira experimenter wrapper plus an 8-byte body. Classifier `lib/classifier.c` function `classifier_lookup` handles conjunction through the `struct cls_conjunction_set` structure per priority level: for each packet, scan rules with conjunction, group by id, check whether each id has enough n clause hits; when sufficient, look up the "real" rule at priority+1 with the same id to execute.

**Wire encoding.** NXM Nicira experimenter format: action header `type=NXAST` (65535) plus `length=24` plus 4-byte experimenter ID `0x00002320` (Nicira) plus 2-byte subtype `34` (NXAST_CONJUNCTION) plus 6-byte padding plus 4-byte `id` plus 1-byte `clause` (1-based k) plus 1-byte `n_clauses` (n, 2 to 64) plus 2-byte padding. Datapath kernel: no wire format; conjunction is resolved entirely in the userspace classifier.

**Applied examples.**

```
# Match (src_ip in {A, B, C}) AND (dst_ip in {X, Y}) AND (tcp_dst in {80, 443}):
# Instead of 3 times 2 times 2 = 12 rules, write 3+2+2 = 7 rules plus 1 real action rule.

# Dimension 1 (src_ip), 3 conjuncts, all tagged conjunction(1, 1/3):
priority=100,ip,nw_src=10.0.0.1,actions=conjunction(1,1/3)
priority=100,ip,nw_src=10.0.0.2,actions=conjunction(1,1/3)
priority=100,ip,nw_src=10.0.0.3,actions=conjunction(1,1/3)

# Dimension 2 (dst_ip), 2 conjuncts, all tagged conjunction(1, 2/3):
priority=100,ip,nw_dst=192.168.0.1,actions=conjunction(1,2/3)
priority=100,ip,nw_dst=192.168.0.2,actions=conjunction(1,2/3)

# Dimension 3 (tcp_dst), 2 conjuncts, all tagged conjunction(1, 3/3):
priority=100,tcp,tp_dst=80,actions=conjunction(1,3/3)
priority=100,tcp,tp_dst=443,actions=conjunction(1,3/3)

# Real rule (priority=101 above 1) with the output action:
priority=101,ip,actions=output:5
```

A typical OVN-generated flow at stage `ls_in_acl` for the ACL `ip4.src == 10.0.0.0/24 && tcp.dst == {80, 443}`:

```
priority=2002,ip,nw_src=10.0.0.0/24,actions=conjunction(0xa1,1/2)
priority=2002,tcp,tp_dst=80,actions=conjunction(0xa1,2/2)
priority=2002,tcp,tp_dst=443,actions=conjunction(0xa1,2/2)
priority=2003,ip,actions=resubmit(,18)
```

**How an engineer masters operations.** Four skills: (1) **Read**: recognise that the pattern of N rules at the same priority sharing `conjunction(id, k/n)` is one dimension; the real rule with the output action is at priority+1; the total rule count is the dimension sum, not the product. (2) **Edit**: split a multi-dimensional ACL match into N conjunct rules plus 1 real rule at priority+1; choose an `id` unique within the table (commonly the hash of the OVN ACL row UUID). (3) **Observe**: `ovs-ofctl dump-flows br-int | grep conjunction` counts the conjunct rules, grouped by id to see how many composite ACLs exist. (4) **Debug**: `ofproto/trace 'in_port=1,...'` shows the line `Conjunction: id=0xa1, 2/2`, confirming the packet hit enough clauses; if it shows only 1/2, the packet matched dimension 1 but not dimension 2, and you should debug the missing dimension.

**Classification.** A match-composition pseudo-action (does not directly process the packet); NXM OVS 2.4 and later plus OF 1.5 (the semantics are standardised but the wire format remains NXM in OVS). Property: special in the action list; not pairable with another action in the same rule (the parser rejects it). Same class: `multipath` (group selection), `bundle` (port selection); both are pseudo-actions affecting rule lookup rather than packet content. Different from `goto_table`: `goto_table` is an OF 1.1 and later instruction that directs flow across tables; `conjunction` aggregates matches within the same table.

**Source code.** `lib/ofp-actions.c` function anchors (OVS v2.17.9): `parse_CONJUNCTION` around line 4100, `encode_CONJUNCTION` around line 4050, `format_CONJUNCTION` around line 4150; enum `NXAST_RAW_CONJUNCTION` line 628 in `enum ofp_raw_action_type`. Header `include/openvswitch/ofp-actions.h` `struct ofpact_conjunction`. Classifier `lib/classifier.c` function `classifier_lookup` plus helper `find_conjunctive_match` (resolves conjunction). OVN compile `northd/northd.c` function `consider_acl` emits conjunction when an ACL match has at least 2 dimensions. Branch baseline: OVS v2.17.9.

**Failure scenarios.** (a) Forgetting the "real" rule at priority+1 means a packet that hits all N clauses has no output rule, so it is dropped silently. Fix: always pair N conjunct rules with one real rule at priority+1 (OVN's structure uses `resubmit` for this). (b) `conjunction(id, k/n)` id collision between two different ACLs causes a packet matching ACL A's clauses plus ACL B's clauses with the same id to "false-match". Fix: choose an id unique within the table (for example, the hash of the ACL UUID). (c) `n_clauses=1` is invalid; the parser rejects it. Fix: for one dimension write a regular rule without conjunction.

**Synthetic exercise.** Mininet, one OVS bridge plus six hosts (h1-h6, IPs `10.0.0.1` to `10.0.0.6`). Install: three conjunct rules on dimension `nw_src` (h1, h2, h3) at priority=100 with conjunction(1,1/2), two conjunct rules on dimension `tcp_dst` (80, 443) at priority=100 with conjunction(1,2/2), and one real rule at priority=101 with actions=output:h6-eth0. Test: `iperf3 -s` on h6; from h1 run `iperf3 -c h6 -p 80` to match (h1 to port 80) and hit the rule. From h4 (not in dimension 1), no match. From h1 with port 22, no dimension-2 match. Verify rule counts via `dump-flows | wc -l` equal 6 (3 + 2 + 1) instead of 6 (3 times 2) for explicit; the case is the same here, but in real life with larger dimensions the savings explode.

**Observation and verification.** `ovs-ofctl dump-flows br0 -O OpenFlow15 | grep conjunction` shows rules with the `conjunction(<id>,<k>/<n>)` suffix. `ovs-appctl coverage/show | grep conjunction` counts `conjunction_match` execute counts. `ovs-appctl ofproto/trace br0 'in_port=1,ip,nw_src=10.0.0.1,...'` shows the line `Conjunction: id=1, clause 1/2 matched` per dimension hit; when the trace shows `clause 1/2 matched` and not `clause 2/2`, the packet matched only one dimension, not the full conjunction. `ovs-appctl dpif/show-flows` does not show conjunction (the kernel only sees the megaflow of the real rule).

**Comparison with other systems.** Linux netfilter ipset (`ipset create allowed_src hash:ip; iptables -m set --match-set allowed_src src`) provides set-of-IPs rule reduction but does not natively compose multiple dimensions. tc cls_u32 hash table is semantically similar. P4 has no conjunction primitive; it must be implemented through nested tables plus action exit. Cisco IOS object-group ACL is the closest concept (object-group network/service then match in an ACL line). NSX-T DFW rules with multi-criteria evaluate sequentially, not optimised through conjunction.

### 4.9.33.14 `ct()`, the conntrack action with full options (commit, zone, nat, helper)

**Concept.** `ct(...)` is the action that calls the conntrack subsystem with a rich option list controlling tracking and modification behaviour: `commit` (create the entry in the conntrack table), `force` (override an existing entry), `table=N` (recirculate to table N after processing), `zone=Z` or `zone=NXM_field[]` (logical zone isolation), `mark=V/M` (set conntrack metadata), `label=V/M` (set the 128-bit label), `helper=name` (invoke an ALG such as `ftp`, `tftp`, `sip`), and the `nat(...)` nested sub-action (SNAT/DNAT translation). This is the most complex action in OVS and the only one with a nested sub-action. Action type: **stateful side-effect plus modify**, NXM OVS 2.5 and later, OF 1.5 partial, not a terminator.

**History and context.** OVS 2.5 (2016) added conntrack integration with the basic `ct(commit, zone=N)`, built on the kernel netfilter conntrack module through the `nf_conntrack` API. OVS 2.6 (2016) added `nat(...)` sub-action support. OVS 2.7 (2017) added `helper=` for ALGs. OVS 2.8 (2018) added the 128-bit `label`. OVS 2.13 and later (2020) added the `force` flag. OF 1.5 (2014) partially standardised `ct`, but OVS keeps the NXM extension for the full option set. Verify upstream: `lib/ofp-actions.c` enum `NXAST_RAW_CT` line 631, function `parse_CT` around line 4300, `encode_CT` around line 4200 (OVS v3.3.0).

**Position in the architecture.** At the OpenFlow protocol layer: `NXAST_RAW_CT` Nicira experimenter; the action body has a fixed header (`flags` 2 bytes plus `zone_src` 16-byte NXM ref plus `zone_imm` 2 bytes plus `recirc_table` 1 byte plus `alg` 2 bytes plus ...) plus a variable nested action list (for the `nat(...)` sub-action). At the OVS internal layer: `struct ofpact_conntrack` (`include/openvswitch/ofp-actions.h`) holds `flags` (`NX_CT_F_COMMIT`, `NX_CT_F_FORCE`), `recirc_table`, `zone_src`/`zone_imm`, `alg`, plus a flexible `actions` array for the NAT sub-action. At the datapath kernel layer: translates to `OVS_ACTION_ATTR_CT` netlink attribute with nested attributes (`OVS_CT_ATTR_COMMIT`, `OVS_CT_ATTR_ZONE`, `OVS_CT_ATTR_MARK`, `OVS_CT_ATTR_LABELS`, `OVS_CT_ATTR_HELPER`, `OVS_CT_ATTR_NAT`); the `openvswitch.ko` kernel module calls the `nf_conntrack` framework directly.

**Role.** Primary role: stateful firewall (OVN stateful ACL, security group). Secondary role: NAT/PAT (SNAT outbound, DNAT load balancer), ALG for protocols with auxiliary ports (FTP data channel, SIP), session tracking for audit. Blast radius: per-packet; the action has side-effects on the conntrack table (kernel or OVS userspace conntrack); after the action the packet must be re-classified at the chosen table (`recirc_table`) because the `ct_state` field is populated.

**Why it exists.** A stateful firewall must know "which connection does this packet belong to and what state is it in (NEW, ESTABLISHED, RELATED, INVALID)". Before OVS 2.5, OVS was stateless: rules matched L2-L4 headers without state. NAT had to be done outside OVS through iptables or Linux conntrack-tools, a complex chained pipeline that was not programmable from the controller. The `ct(commit)` action bridges OpenFlow rules with netfilter conntrack: the rule triggers a conntrack lookup/commit, the packet gets `ct_state`, `ct_zone`, and `ct_mark` populated, and subsequent rules can match on them. A second motivation: OVN needs stateful ACLs plus load balancer plus NAT inside its logical-pipeline DSL and cannot depend on stand-alone conntrack-tools.

**Problems it solves.** Specifically: (a) stateful firewall (allow ESTABLISHED, drop INVALID), (b) NAT/PAT translation (SNAT outbound, DNAT load balancer), (c) ALG (FTP, SIP) tracking auxiliary ports, (d) per-zone connection isolation (multi-tenant). Approach: the action delegates conntrack subsystem do/track lookup; flags and sub-actions specify the behaviour. Scope: OVS stateful pipeline; OVN uses it as the primitive for stateful ACL plus LB plus NAT.

**Importance.** Tier 2 medium but critical for OVN production. Every stateful ACL (NB ACL `action="allow-related"`), Load_Balancer, and NAT in OVN compiles to a rule containing `ct(...)`. In OVN ingress stages: `ls_in_pre_acl` sets the zone, `ls_in_acl` evaluates the ACL with ct_state, `ls_in_pre_lb` and `ls_in_lb` perform LB with DNAT, `ls_in_stateful` commits. Engineer must-know: the options `commit` (create entry), `zone=Z` (isolation), `nat(src=ip:port)` and `nat(dst=ip:port)` (NAT), `helper=ftp` (ALG), and `recirc_table=N` (re-classify); the recirc pattern is always required because the packet after `ct` cannot continue in the same rule's action list and must return to table N.

**How it works.** Parser `lib/ofp-actions.c` function `parse_CT` (around line 4300 in v3.3.0): tokenise inside `(...)`; parse each sub-option separately. `commit` sets the flag `NX_CT_F_COMMIT`, `force` sets `NX_CT_F_FORCE`, `table=N` sets `recirc_table`, `zone=NXM_field[]` parses an NXM field reference into `zone_src`, `zone=N` parses an immediate into `zone_imm`, `helper=name` parses a string into `alg` (the ALG ID), and `nat(...)` parses the nested action list. Encoder `encode_CT` (around line 4200) emits the Nicira experimenter wrapper plus serialises the options. Translation `xlate.c` function `xlate_ct_action` emits the `OVS_ACTION_ATTR_CT` netlink attribute with nested attributes. Datapath kernel module `openvswitch.ko` function `ovs_ct_execute`: calls `nf_conntrack_in` to lookup or create the entry, populates the `ct_state` field in skb metadata; when `commit` plus `nat(...)` is set, it calls `nf_nat_setup_info` to add the NAT mapping. After processing, the packet recirculates back to `recirc_table` in the OVS pipeline.

**Wire encoding.** NXM Nicira experimenter format: action header `type=NXAST` (65535) plus `length=variable` plus 4-byte experimenter ID `0x00002320` plus 2-byte subtype `36` (NXAST_CT) plus body: 2-byte `flags`, 2-byte `zone_src` NXM header (0 when zone is immediate), 2-byte `zone_imm`, 1-byte `recirc_table`, 1-byte `alg`, plus a nested action list for the NAT sub-action. Datapath kernel: `OVS_ACTION_ATTR_CT` netlink attribute with nested attributes `OVS_CT_ATTR_COMMIT`, `OVS_CT_ATTR_ZONE` (2 bytes), `OVS_CT_ATTR_MARK` (8 bytes mask+value), `OVS_CT_ATTR_LABELS` (32 bytes 128-bit value+mask), `OVS_CT_ATTR_HELPER` (string), `OVS_CT_ATTR_NAT` (nested with `OVS_NAT_ATTR_SRC`/`DST`/`IP_MIN`/`IP_MAX`/`PROTO_MIN`/`PROTO_MAX`).

**Applied examples.**

```
# Stateful firewall: commit NEW plus allow ESTABLISHED plus drop INVALID
table=0,priority=100,ip,ct_state=-trk,actions=ct(table=1,zone=42)
table=1,priority=100,ip,ct_state=+trk+new,actions=ct(commit,zone=42),output:N
table=1,priority=100,ip,ct_state=+trk+est,actions=output:N
table=1,priority=10,ip,ct_state=+trk+inv,actions=drop

# DNAT load balancer (outside hits the VIP, NAT to backend)
table=2,ip,ct_state=-trk,nw_dst=10.0.0.100,tcp,tp_dst=80,
        actions=ct(commit,zone=42,nat(dst=10.0.1.5:8080)),output:N

# SNAT outbound (private to public)
table=3,ip,ct_state=-trk,nw_src=10.0.0.0/24,
        actions=ct(commit,zone=42,nat(src=203.0.113.10)),output:uplink

# FTP ALG (data channel tracking)
table=4,ip,ct_state=-trk,nw_dst=10.0.0.5,tcp,tp_dst=21,
        actions=ct(commit,zone=42,helper=ftp),output:N
```

A typical OVN-generated flow at stage `ls_in_lb` for VIP DNAT:

```
priority=110,ip,ct_state=+new+trk,reg0=0x1,
    actions=ct(commit,table=15,zone=NXM_NX_REG13[0..15],nat(dst=10.0.1.5:8080))
```

**How an engineer masters operations.** Four skills: (1) **Read**: distinguish `ct(table=N, zone=Z)` (lookup-only, no commit) from `ct(commit, zone=Z)` (commit the entry); recognise that OVN always uses `ct(table=N, zone=NXM_NX_REG13[0..15])` to derive the zone from the datapath UUID. (2) **Edit**: write a two-stage stateful pipeline; stage 1 is `ct(table=N, zone=Z)` lookup and stage N is per-`ct_state` rules. Add `commit` only on the NEW rule. For NAT: write `ct(commit, zone=Z, nat(dst=ip:port))` with `commit` mandatory. (3) **Observe**: `conntrack -L -z N` shows entries in zone N; `ovs-appctl dpctl/dump-conntrack` shows them from OVS userspace; `ovs-ofctl dump-flows br-int | grep ct(` counts conntrack rules. (4) **Debug**: `ofproto/trace 'in_port=1,ip,...'` shows the line `Datapath actions: ct(commit,zone=42,nat(...)),N`, confirming the action; the `ct_state` field in the trace output reveals the current state. When the state is stuck at `+trk+new` and never transitions to `+est`, check the helper ALG or the reverse direction.

**Classification.** A stateful side-effect plus modify action (updates the conntrack table, populates ct_state, may NAT-modify the packet header); NXM OVS 2.5 and later plus OF 1.5 partial. Properties: not a terminator; MUST recirculate through `table=N` to re-classify with the populated ct_state. Same class: `ct_clear` (clears tracking from packet metadata, OVS 2.10 and later); no other native stateful action exists in OVS. Different from `learn`: `learn` installs a new rule in the flow table; `ct(commit)` installs an entry in the separate conntrack table (kernel netfilter), not the flow table.

**Source code.** `lib/ofp-actions.c` function anchors (OVS v2.17.9): `parse_CT` around line 4300, `encode_CT` around line 4200, `format_CT` around line 4350; enum `NXAST_RAW_CT` line 631 in `enum ofp_raw_action_type`. Header `include/openvswitch/ofp-actions.h` `struct ofpact_conntrack` plus `struct ofpact_nat`. Translation `ofproto/ofproto-dpif-xlate.c` function `xlate_ct_action`. Datapath kernel module `datapath/conntrack.c` functions `ovs_ct_execute` plus `ovs_ct_nat`; netlink protocol `include/uapi/linux/openvswitch.h` enum `OVS_ACTION_ATTR_CT` plus `enum ovs_ct_attr` plus `enum ovs_nat_attr`. Branch baseline: OVS v2.17.9.

**Failure scenarios.** (a) Forgetting `commit` on the NEW rule means the first packet is tracked but no entry is committed; the second packet on the same connection still has `ct_state=+new`, so the ESTABLISHED-allow rule never matches and the traffic drops. Fix: the NEW rule must use `ct(commit, zone=Z)`. (b) Forgetting `recirc_table=N` means the packet has no table to continue at after `ct`; the behaviour is undefined on many older OVS versions, and newer OVS drops the packet. Fix: always set `table=N` (lookup) or use `commit` followed by another action in the same list (`commit` does not require `table`). (c) Zone collision between two logical tenants causes tenant A's connections to be tracked in the same zone as tenant B's, leading to a cross-tenant leak. Fix: use `zone=NXM_NX_REG13[0..15]` so the zone is derived from the logical datapath UUID, ensuring uniqueness per tenant. (d) NAT `nat(src=10.0.0.5)` without `commit` performs the translation for this packet but does not save the state; the reverse traffic is not un-NATed and the connection breaks. Fix: NAT always requires `commit` plus `zone`.

**Synthetic exercise.** Mininet, one OVS bridge plus three hosts (h1, h2, h3), IPs `10.0.0.1/2/3`. Install the stateful-firewall pipeline: `table=0,ip,ct_state=-trk,actions=ct(table=1,zone=42)`, `table=1,ip,ct_state=+trk+new,actions=ct(commit,zone=42),output:NORMAL`, `table=1,ip,ct_state=+trk+est,actions=output:NORMAL`, `table=1,priority=10,ip,ct_state=+trk+inv,actions=drop`, `table=1,priority=0,actions=drop`. Test: `h1 ping h2` succeeds. Use `conntrack -L -z 42` to see the entry `icmp src=10.0.0.1 dst=10.0.0.2 ... [ASSURED]`. Inject an INVALID packet with `hping3` (a SYN-ACK without a preceding SYN); the INVALID-drop rule's counter rises. Verify `ovs-appctl dpif/dump-conntrack` also shows the entry.

**Observation and verification.** `conntrack -L -z N` dumps entries in zone N (requires `apt install conntrack` on Ubuntu). `ovs-appctl dpctl/dump-conntrack` dumps from the OVS datapath (kernel or userspace). `ovs-appctl ofproto/trace br-int 'in_port=1,ip,...'` shows the line `ct(commit,zone=42,nat(dst=...)),N` plus state transitions; the `ct_state` field in the trace output reveals the current state (`+new`, `+est`, `+rel`, `+inv`, `+trk`). `ovs-appctl dpif/show-dp-features br-int | grep -i conntrack` confirms the datapath has conntrack support; if missing, the `nf_conntrack` module is not loaded.

**Comparison with other systems.** Linux netfilter `iptables -m conntrack --ctstate ESTABLISHED -j ACCEPT` is the functional equivalent: the same kernel conntrack subsystem, but controlled through iptables instead of an OpenFlow rule. nftables `ct state established` is the same concept with newer syntax. Cisco IOS `ip nat inside source list` plus `ip access-list` reflexive ACL is semantically similar but is not programmable from a central controller. P4 register-based stateful tables have no conntrack primitive; it must be implemented from scratch. NSX-T distributed firewall is stateful, built into the hypervisor kernel module separately. Calico's iptables-based dataplane uses iptables conntrack directly; Calico's eBPF dataplane uses a separate eBPF map for the equivalent.

## 4.9.35 Peripheral treatment for five MPLS plus PBB actions

This section adds focused peripheral treatments (about 15 to 30 lines per action) for five MPLS plus PBB actions that are rarely seen outside service-provider Ethernet. Different from §4.9.16 (the brief catalog) and §4.9.29.2-5 (the v3.5 eight-attribute Anatomy), this section concentrates on observation and verification, the reason for existence, and the upstream-OVS caveat.

### 4.9.35.1 `push_pbb`, push the PBB I-TAG (802.1ah) onto the packet

**Concept.** `push_pbb` is the standard OpenFlow 1.3 and later action (OFPAT_PUSH_PBB type 26) that inserts an 802.1ah Provider Backbone Bridge (PBB) shim header (comprising B-DA 6 bytes, B-SA 6 bytes, B-Tag 4 bytes (TPID `0x88a8`), and I-Tag 6 bytes (TPID `0x88e7` plus a 24-bit I-SID)) before the customer Ethernet header. After the action the packet length grows by 22 bytes and the outer ethertype becomes `0x88e7`. The action takes the argument `ethertype=0x88e7` to specify the outer TPID. Type: header push, OF 1.3 and later standard.

**Context and motivation.** PBB (IEEE 802.1ah, ratified in 2008) is an extension of the Provider Bridge (Q-in-Q 802.1ad) for carrier metro Ethernet: instead of using a 12-bit VLAN ID (4,096 maximum), PBB uses a 24-bit I-SID (16M service instances) and separates the customer MAC space from the backbone. OF 1.3 (2012) added `OFPAT_PUSH_PBB` so the controller could programmatically encapsulate PBB.

**Support status in OVS.** **Important.** OVS upstream up to v3.3.0 (2024-04) does **not** implement `push_pbb`. Evidence: in `lib/ofp-actions.c` v3.3.0 lines 9033-9034 the comment `/* OF1.3+ OFPAT_PUSH_PBB (26) not supported. */` is present in the `ofpact_map of12[]` table. There is no `parse_PUSH_PBB`, `encode_PUSH_PBB`, or `decode_OFPAT_RAW_PUSH_PBB` in the source. An engineer who passes `actions=push_pbb:0x88e7` to `ovs-ofctl add-flow` receives a parser error along the lines of `unknown action 'push_pbb'`.

**Why OVS does not implement it.** PBB is carrier Ethernet technology; the target is provider backbone hardware switches (Cisco ASR 9000, Juniper MX, Nokia 7750), not data-centre virtual switches. OVN uses Geneve and VXLAN for multi-tenant overlay; isolation by 24-bit VNI does not need PBB I-SID. The OVS community has not accepted a PBB patch because there is no practical use case in hyperscale cloud data centres.

**OF 1.3 syntax spec (reference).** Wire format: action header `type=26` (OFPAT_PUSH_PBB) plus `length=8` plus `ethertype` 2 bytes (0x88e7) plus 2-byte padding. Prerequisite spec: the rule does not need to match `eth_type=0x88e7` (the action is for encap; the input packet is normal Ethernet); only after the action does it have `eth_type=0x88e7`. The spec sets B-DA equal to the current `eth_dst`, B-SA equal to `eth_src`, and the I-SID default to 0; the engineer uses `set_field:0x123456->pbb_isid` immediately after to assign the real I-SID.

**Observation and workaround.** When programmable PBB encap is required in an OVS environment, the engineer must fall back to native Linux: `ip link add link eth0 name pbb0 type 8021ah ...` (the kernel `8021ah` driver, not OVS), or push to a carrier hardware switch and peer with OVS through a trunk port. Comparison with other systems: Cisco IOS-XR `l2vpn xconnect-group` plus `pbb` instance supports PBB natively; P4 programs PBB through a `header pbb_t` declaration; FRR/Quagga has no data-plane PBB support (only control-plane coordination).

### 4.9.35.2 `pop_pbb`, remove the PBB I-TAG from the packet

**Concept.** `pop_pbb` is the standard OpenFlow 1.3 and later action (OFPAT_POP_PBB type 27) that removes the outermost 22-byte 802.1ah PBB shim header from the packet, exposing the customer Ethernet header beneath. After the action the packet length shrinks by 22 bytes and the outer ethertype reverts to the actual customer Ethernet ethertype. The action takes no argument; it always removes one outermost PBB layer. Type: header pop, OF 1.3 and later standard.

**Context and motivation.** Symmetric counterpart to `push_pbb`: a backbone egress switch (BCB or BEB egress) needs to remove the PBB shim before forwarding the packet out of the customer port. OF 1.3 standardised `OFPAT_POP_PBB` (type 27) alongside `OFPAT_PUSH_PBB` (type 26) for symmetry.

**Support status in OVS.** **Important.** OVS upstream up to v3.3.0 does **not** implement `pop_pbb`. Evidence: `lib/ofp-actions.c` v3.3.0 line 9034 has the comment `/* OF1.3+ OFPAT_POP_PBB (27) not supported. */` in the `ofpact_map of12[]` table. There is no `parse_POP_PBB`, `encode_POP_PBB`, or `decode_OFPAT_RAW_POP_PBB`. The rule `actions=pop_pbb` passed to `ovs-ofctl add-flow` is rejected by the parser. The spec requires `pop_pbb` to have the prerequisite `eth_type=0x88e7` (the packet must carry an outer PBB tag), but because OVS does not implement it the prerequisite is not enforced either.

**Why OVS does not implement it.** Same reason as `push_pbb`: PBB is carrier technology with no data-centre use case. The OVS classifier has no dedicated `pbb_isid` field (unlike hardware switches such as the Broadcom Trident, which has a direct PBB parser). When PBB processing is needed in OVS, the engineer must push the packet up to the controller via the `controller(...)` action and parse PBB in userspace.

**OF 1.3 syntax spec (reference).** Wire format: action header `type=27` (OFPAT_POP_PBB) plus `length=8` plus 4-byte padding (no argument body). Datapath kernel: when OVS implements it, it would map to the `OVS_ACTION_ATTR_POP_PBB` netlink attribute; however, the `openvswitch.ko` kernel module also has no `pop_pbb` handler (a `git grep pop_pbb` in `datapath/` returns zero hits).

**Observation and workaround.** An engineer who needs to pop PBB must fall back to the Linux kernel: `ip link add link bond0 type 8021ah` and forward the packet through the PBB-aware interface so the kernel auto-decapsulates. Or use a tc filter: `tc filter add dev eth0 ingress flower indev eth0 src_mac ...`; tc has no direct PBB pop, so a custom BPF program is required (out of OVS scope). Comparison with other systems: Cisco IOS-XR `interface l2tunnel` auto-pops PBB; Juniper MX `bridge-domain pbb` is similar.

### 4.9.35.3 `set_mpls_ttl`, overwrite the MPLS TTL field

**Concept.** `set_mpls_ttl:N` is the standard OpenFlow 1.1 and later action (OFPAT_SET_MPLS_TTL type 15) that overwrites the TTL byte of the outermost MPLS label with the value N (0 to 255). The action takes a `uint8_t ttl` argument. After the action, byte 4 of the MPLS shim (TTL position in the 4-byte header: label 20 bits, TC 3 bits, BoS 1 bit, TTL 8 bits) is set to N. Type: field rewrite, OF 1.1 and later standard, OVS 1.11 and later implementation.

**Context and motivation.** The MPLS label header has an 8-bit TTL to prevent infinite loops in the LSP (Label Switched Path). When an MPLS edge router (PE) pushes a new label, the outer label's TTL by default inherits from the inner label, which is called **uniform mode** (traceroute sees every hop). Some scenarios want to hide the core topology (called **pipe mode**): the engineer sets the outer TTL to a fixed high value (for example 64) so traceroute does not expose the core hop count. `set_mpls_ttl` is the primitive for programming pipe mode.

**Support status in OVS.** OVS implements it fully. Evidence in `lib/ofp-actions.c` v3.3.0: enum `OFPAT_RAW_SET_MPLS_TTL` line 210 (`/* NX1.0(25), OF1.1(15), OF1.2+(15) is deprecated (use Set-Field): uint8_t. */`); function `parse_SET_MPLS_TTL` line 3925 (parse the argument via `str_to_u8` and store into `mpls_ttl->ttl`); `encode_SET_MPLS_TTL` line 3913 (calls `put_OFPAT_SET_MPLS_TTL`); `decode_OFPAT_RAW_SET_MPLS_TTL` line 3904; `format_SET_MPLS_TTL` line 3944 (prints `set_mpls_ttl(N)`). Note: OF 1.2 and later deprecates `set_mpls_ttl` in favour of `set_field:N->mpls_ttl` through the generic `OFPACT_SET_FIELD`, but `set_mpls_ttl` is still parser-accepted on every version for backward compatibility.

**Syntax and prerequisite.** Syntax `actions=set_mpls_ttl:64`. Prerequisite: the rule must match MPLS (`eth_type=0x8847` unicast or `0x8848` multicast); when the rule does not match MPLS, the action is a no-op for packets without an outermost MPLS shim. OF 1.1 and later wire encoding: action header `type=15` plus `length=8` plus 1-byte ttl plus 3-byte padding. Datapath kernel: the action maps to `OVS_ACTION_ATTR_SET` with `OVS_KEY_ATTR_MPLS` containing the 32-bit `mpls_lse` field; the kernel rewrites the TTL byte in the shim.

**Use case and example.** Pipe mode L3 VPN: `priority=100,ip,nw_dst=10.1.0.0/24,actions=push_mpls:0x8847,set_field:100->mpls_label,set_mpls_ttl:64,output:uplink` pushes label 100 with a fixed TTL of 64 to hide the core topology. TTL refresh on label swap: `priority=100,mpls,mpls_label=100,actions=set_mpls_ttl:64,set_field:200->mpls_label,output:next` refreshes the TTL to 64 when swapping the label, avoiding drops in the core. Important: OF 1.2 and later recommends `set_field:64->mpls_ttl` via `OXM_OF_MPLS_TTL` rather than the legacy `set_mpls_ttl:64`; OVS accepts both. Comparison with other systems: Cisco IOS `mpls ttl-expiration pop 1 refresh-tll` is config-time; Juniper `set protocols mpls no-decrement-ttl` is similar; tc cls_u32 cannot programmatically set MPLS TTL per flow.

### 4.9.35.4 `dec_mpls_ttl`, decrement the outermost MPLS TTL by one

**Concept.** `dec_mpls_ttl` is the standard OpenFlow 1.1 and later action (OFPAT_DEC_MPLS_TTL type 16) that decrements the outermost MPLS label's TTL by 1. When the TTL before decrement is 0 or after decrement is 0, the packet is dropped or sent to the controller as a packet_in (depending on the flow flag). The action takes no argument. Type: field arithmetic, OF 1.1 and later standard, OVS 1.11 and later implementation.

**Context and motivation.** Every MPLS transit hop (LSR, Label Switching Router) must decrement TTL to prevent infinite loops, equivalent to IP TTL decrement during IP forwarding. Before `dec_mpls_ttl`, OpenFlow 1.0 had no dedicated MPLS-TTL primitive; the controller had to emulate it through `set_mpls_ttl` with a hardcoded value or through `set_field` from a register after reading and decrementing by 1 (complex plus race condition).

**Support status in OVS.** OVS implements it fully. Evidence in `lib/ofp-actions.c` v3.3.0: enum `OFPAT_RAW_DEC_MPLS_TTL` line 213 (`/* NX1.0(26), OF1.1+(16): void. */`); function `parse_DEC_MPLS_TTL` line 3976 (no argument; only `ofpact_put_DEC_MPLS_TTL`); `encode_DEC_MPLS_TTL` line 3969 (calls `put_OFPAT_DEC_MPLS_TTL`); `decode_OFPAT_RAW_DEC_MPLS_TTL` line 3962 (input is only `struct ofpbuf *out`, with no ttl parameter because the action has no body); `format_DEC_MPLS_TTL` line 3983 (prints `dec_mpls_ttl` with no suffix). Header `include/openvswitch/ofp-actions.h` uses `struct ofpact_null` for argument-less actions.

**Syntax, prerequisite, and TTL=0 behaviour.** Syntax `actions=dec_mpls_ttl`. Prerequisite: the rule matches MPLS (`eth_type=0x8847/0x8848`). OF 1.1 and later wire encoding: action header `type=16` plus `length=8` plus 4-byte padding (no argument). TTL=0 behaviour: the OF spec stipulates that when TTL before decrement is 1, after decrement is 0, the packet must drop or send a packet_in `OFPR_INVALID_TTL` to the controller (depending on the flow's `OFPFF_INVALID_TTL_TO_CONTROLLER` flag). OVS: the kernel datapath silently drops a TTL=0 packet; the userspace upcall path can emit a packet_in when the controller has subscribed.

**Use case and example.** LSR transit refresh: `priority=100,mpls,mpls_label=100,actions=dec_mpls_ttl,set_field:200->mpls_label,output:next` decrements TTL before swapping the label and forwarding. Combined with label swap: the hop count is correct, and the packet self-drops at hop 64 when looping. Pipe-mode topology hiding: do **not** use `dec_mpls_ttl` in the core (keep the outer TTL fixed at 64 from ingress); traceroute from the end host does not see the core hops. Important: when the controller wants to receive a packet_in on TTL expiration (for an ICMP TTL Exceeded reply), set `flags=send_flow_rem,check_overlap` plus subscribe to `OFPR_INVALID_TTL` via `set-async`. Comparison with other systems: Cisco IOS LSR auto-decrements MPLS TTL by default; Juniper is similar; FRR ldpd is control-plane only; the data plane is the Linux kernel mpls module, which decrements automatically.

### 4.9.35.5 `copy_ttl_in`, copy TTL from the outer header into the inner header

**Concept.** `copy_ttl_in` is the standard OpenFlow 1.1 and later action (OFPAT_COPY_TTL_IN type 12) that copies TTL from the outer header (MPLS shim or outer IP) into the adjacent inner header's TTL (inner IP). The action takes no argument. Used when popping encapsulation and wanting to preserve hop count across the tunnel; this is **uniform mode** in MPLS-IP. Type: TTL preservation pseudo-action, OF 1.1 and later standard.

**Context and motivation.** When a packet traverses an MPLS LSP and a PE pops the label: the inner IP TTL has been "frozen" since ingress (the PE's push-label step copied IP TTL into the outer MPLS TTL while leaving the inner IP TTL unchanged). At egress, the outer TTL has decremented through N core hops, but the inner TTL still has the original value. The engineer has two choices: (a) **pipe mode**: pop the label without copying TTL, the inner IP TTL stays unchanged, and the end-to-end host does not see the core hops in traceroute (topology hidden); (b) **uniform mode**: pop the label with `copy_ttl_in`, so the inner IP TTL equals the MPLS outer TTL just before pop, and the host sees every hop through the LSP. These two modes are an important decision in carrier MPLS deployments.

**Support status in OVS.** **Important.** OVS upstream up to v3.3.0 does **not** implement `copy_ttl_in`. Evidence: `lib/ofp-actions.c` v3.3.0 lines 9001 and 9021 have the comments `/* OFPAT_COPY_TTL_IN (12) not supported. */` in both `ofpact_map of10[]` (OF 1.0/1.1) and `of12[]` (OF 1.2/1.3/1.4). There is no `parse_COPY_TTL_IN`, `encode_COPY_TTL_IN`, `decode_OFPAT_RAW_COPY_TTL_IN`, and no `OFPAT_RAW_COPY_TTL_IN` enum in `ofp_raw_action_type`. The rule `actions=copy_ttl_in` passed to `ovs-ofctl add-flow` is rejected by the parser with `unknown action`.

**Why OVS does not implement it.** Similar reason to PBB: copy_ttl_in/out is a carrier-MPLS feature with no practical use case in data-centre overlay. OVN uses Geneve and VXLAN; the concept of "TTL preserve" does not apply because the inner and outer IP are two separate IP packets (Geneve outer is UDP IPv4 or IPv6 with its own TTL, and the 24-bit VNI carries the inner packet independently). When hop-count preservation is required, the controller must emulate it through `set_field:NXM_NX_REG0[]->nw_ttl` with register-copy logic, which is complex and rarely practical.

**Workaround and OF 1.1 syntax spec (reference).** OF 1.1 and later wire format: action header `type=12` (OFPAT_COPY_TTL_IN) plus `length=8` plus 4-byte padding. The spec stipulates that the action is only valid when the packet has at least 2 header layers (MPLS-over-IP, IP-over-IP). Workaround in OVS to emulate: write a flow chain using a register: `actions=move:NXM_OF_MPLS_TTL[]->NXM_NX_REG0[0..7],pop_mpls:0x0800,move:NXM_NX_REG0[0..7]->NXM_OF_IP_TTL[]` to read MPLS TTL into reg0 before pop and write reg0 back to IP TTL after pop. However `NXM_OF_MPLS_TTL` does not exist as a stand-alone NXM field; the engineer must use `set_field` with the mpls_ttl OXM, which is complex because the OVS classifier does not expose MPLS TTL as a register-readable field. Practical conclusion: when the workload truly needs `copy_ttl_in`, OVS is not the right tool; switch to a carrier hardware switch (Cisco/Juniper).

**Comparison with other systems.** Cisco IOS-XR `mpls ip-ttl-propagate disabled` (pipe mode) or default enable (uniform mode auto copy_ttl_in) is config-time, with no programmable action needed. Juniper `set protocols mpls no-propagate-ttl` is similar. The Linux kernel `mpls_iptunnel` module supports TTL inheritance through the sysctl `net.mpls.ip_ttl_propagate=1`. P4 programs the TTL copy through `meta.outer_ttl = hdr.outer.ttl; hdr.inner.ttl = meta.outer_ttl;` direct assignment.

## 4.9.36 Peripheral treatment for five NSH plus observability actions

This section adds focused peripheral treatments (about 15 to 30 lines per action) for five peripheral actions in two groups: the packet-type-aware generic encap/decap group (NSH service function chaining) covering `decap` and `encap`, and the observability and annotation group covering `controller(userdata=...)`, `note`, and `sample`. Different from §4.9.29.7-11 (the brief v3.5 eight-attribute Anatomy), this section concentrates on the actual parser mechanism in upstream OVS, the runtime side-effects, the practical caveats, and the CLI artefacts an engineer actually types.

### 4.9.36.1 `decap`, remove the outermost encapsulation header by packet_type

**Concept.** `decap` is an OVS extension action (NXAST_RAW_DECAP) that removes the outermost encapsulation header from the packet and updates the `packet_type` field to the inner protocol. Syntax: `decap` (no argument) or `decap(packet_type=(NS,TYPE))` to specify the resulting packet type after the pop. After the action, the packet length shrinks by the popped header's size, and the classifier `packet_type` field changes to the supplied NS/TYPE (or `PT_USE_NEXT_PROTO` when no argument is supplied).

**Role.** `decap` is the counterpart of `encap`, used in the service function chain (SFC) when the final service function in the chain must remove the NSH header before forwarding the packet to the customer network. Also used to strip MPLS, Geneve, or GRE outer headers outside the SFC context. Origin: NSH RFC 8300 (2018) standardised the Service Function Chain header; OVS added packet-type-aware `encap` and `decap` from v2.8 (2017) so the controller can program them.

**Parser mechanics in OVS.** The function `parse_DECAP` in `lib/ofp-actions.c` (OVS branch v3.3) parses the argument `packet_type=(ns=NS,type=TYPE)` through `ofputil_parse_key_value`; with no argument the default is `decap->new_pkt_type = htonl(PT_USE_NEXT_PROTO)`, which lets the datapath infer the protocol from the popped header. Wire format: `NXAST_RAW_DECAP` Nicira-extension subtype; `struct nx_action_decap` holds a 4-byte `new_pkt_type` field. Decoded by `decode_NXAST_RAW_DECAP`.

**Syntax and practical examples.** Two `decap` calls to revert NSH back to Ethernet:

```
ovs-ofctl add-flow br0 \
  "table=0,priority=100,nsh,nsh_si=0,actions=decap(),decap(packet_type=(0,0x0800)),output:2"
```

Pop the Geneve outer header:

```
ovs-ofctl add-flow br0 \
  "table=0,priority=100,tun_id=0x100,actions=decap(packet_type=(1,0x6558)),output:2"
```

(`(1,0x6558)` is NS=1 OFPHTN_ETHERTYPE, TYPE=0x6558 Transparent Ethernet Bridging.)

**Caveats and common errors.** `decap` does not error when the packet has no matching header; the datapath produces an undefined result. The engineer must match strictly (`nsh`, `mpls`, `tun_id=...`) before calling `decap`. Common mistake: calling `decap()` on an ordinary Ethernet packet, which causes the kernel datapath to drop silently. Observation: `ovs-appctl dpctl/dump-flows | grep drop` plus `ovs-ofctl dump-flows br0` cross-checked against expected hit counts. Comparison with other systems: P4 `header.pop_front(1)` is a direct call without packet_type negotiation; Linux tc `pedit` has no generic decap.

### 4.9.36.2 `encap`, push a packet-type-aware encapsulation header (NSH SFC)

**Concept.** `encap` is an OVS extension action (NXAST_RAW_ENCAP) that adds an encapsulation header on top of the current packet and updates the `packet_type` field to the new outer packet type. Syntax: `encap(<header_type>(prop=value, tlv(class,type,value), ...))`. Supported header types: `nsh`, `ethernet`, `mpls`. After the action, the packet length grows by the pushed header's size, and `packet_type` shifts to the outer protocol.

**Role.** Entry point of a service function chain (SFC) or tunnel: the SFC classifier at data-centre ingress emits `encap(nsh)` to wrap traffic into the chain with a service path identifier (SPI) plus service index (SI) metadata. NSH metadata has two modes: MD Type 1 with a fixed-length 16-byte (4 times 32-bit) context header, and MD Type 2 with a variable TLV (`tlv(class=...,type=...,value=...)`). Origin: RFC 8300 NSH was standardised in 2018; OVS implemented it in v2.8.

**Parser mechanics in OVS.** The function `parse_ENCAP` in `lib/ofp-actions.c` (OVS branch v3.3) requires OF version 1.3 or later (`*pp->usable_protocols &= OFPUTIL_P_OF13_UP`). It parses the header type via `parse_encap_header(key, &encap->new_pkt_type)` and parses property classes via `parse_ed_prop_class`. Wire format: `NXAST_RAW_ENCAP` Nicira-extension subtype (not pure OF 1.5 standard, even though the OF 1.5 spec defines `OFPAT_ENCAP`); `struct nx_action_encap` holds `new_pkt_type` plus a variable property body. Decoded by `decode_NXAST_RAW_ENCAP`.

**Syntax and practical examples.** SFC entry classifier with MD Type 1:

```
ovs-ofctl add-flow br0 \
  "table=0,priority=100,ip,ip_dst=10.0.0.0/24,actions=\
   encap(nsh(md_type=1)),\
   set_field:0x123->nsh_spi,set_field:255->nsh_si,\
   set_field:0xdeadbeef->nsh_c1,output:tun0"
```

MD Type 2 with a custom TLV:

```
ovs-ofctl add-flow br0 \
  "table=0,priority=100,ip,actions=\
   encap(nsh(md_type=2,tlv(class=0x1,type=1,len=4,value=0xcafebabe))),\
   output:tun0"
```

**Caveats and version drift.** `encap(nsh)` is only available when OVS is built with NSH support (`./configure --enable-nsh`); the Ubuntu 22.04 distro package (OVS 2.17) enables it by default, but RHEL 8 and 9 might not. Verify: `ovs-appctl dpif/show-dp-features br0 | grep -i nsh`. Parser error when OVS lacks support: `unknown action 'encap'`. Version drift: OVS 2.8-2.10 only supports MD Type 1; MD Type 2 TLV was added in OVS 2.12 and later. Comparison with other systems: VPP's `encap-nsh` plugin supports NSH from v18.07; Cisco IOS-XR `service-function-chain` is config-time, not programmable per flow.

### 4.9.36.3 `controller(userdata=...)`, send PACKET_IN with correlation metadata

**Concept.** `controller(userdata=HEX,...)` is the extended form of the `controller` action (covered in §4.9.6) that allows attaching a metadata cookie to the PACKET_IN message so the controller can correlate the source rule with the received packet. The `userdata` argument is a hex byte string of arbitrary length (for example, `userdata=01.02.03.04` or `userdata=ABCD`). When the datapath invokes the action, the packet plus userdata bytes are wrapped into a PACKET_IN with reason `OFPR_ACTION` (or a custom reason).

**Role.** In the continuation-based programming model (OVS 2.6 and later, NXAST_RAW_CONTROLLER2), the controller can respond with a PACKET_OUT carrying a "continue" action to resume the paused pipeline; userdata helps the controller match the request and response. In the basic mode (the older NXAST_RAW_CONTROLLER), userdata is just a debug cookie. OVN uses userdata extensively for ARP/ND responder, DHCP relay, and MAC binding learning; the `ovn-controller` reads userdata to determine the event type and source chassis.

**Parser mechanics in OVS.** The function `parse_CONTROLLER` in `lib/ofp-actions.c` (OVS branch v3.3) parses six keys: `reason`, `max_len`, `id` (controller_id), `userdata`, `pause`, `meter_id`. `userdata` is parsed via `ofpbuf_put_hex` to convert the hex string into bytes. Optimisation: when only the defaults are present (`reason=OFPR_ACTION`, controller_id=0, no userdata, no pause, no meter), the parser emits `OFPACT_OUTPUT` with port `OFPP_CONTROLLER` instead of a full controller action, saving wire bytes. Decoder reads raw type `NXAST_RAW_CONTROLLER` (basic) or `NXAST_RAW_CONTROLLER2` (continuation, with max_len plus pause plus userdata).

**Syntax and practical examples.** OVN-style ARP responder fallback:

```
ovs-ofctl add-flow br-int \
  "table=8,priority=50,arp,arp_op=1,\
   actions=controller(reason=action,id=0,userdata=00.00.00.0a.00.00.00.00.05.00.00.00)"
```

Continuation pause and resume:

```
ovs-ofctl add-flow br0 \
  "table=0,priority=100,tcp,tp_dst=80,\
   actions=controller(reason=action,id=42,userdata=cafe1234,pause)"
```

(The controller receives a PACKET_IN with `userdata=cafe1234`, runs its decision logic, and responds with a PACKET_OUT carrying a continuation action so the pipeline resumes from the paused point.)

**Observation and caveats.** Verify the userdata round trip: `ovs-ofctl monitor br0 65534 -P openflow13` to dump PACKET_IN in real time and inspect the `userdata=...` field in the message body. Caveat: userdata is not encrypted; when the OpenFlow channel is not TLS-protected the cookie leaks on the wire. Common mistake: a hex string with missing quotation marks or odd byte-boundary alignment (`userdata=ABC` with three characters) is rejected by the parser with `bad hex digit`. Comparison with other systems: P4 `digest<T>(receiver, data)` is similar but strongly typed; Linux netfilter `nfqueue` queues the packet to userspace, an equivalent without per-rule cookie correlation.

### 4.9.36.4 `note`, embed annotation bytes in the flow rule (no-op at runtime)

**Concept.** `note:HEXSTRING` is an OVS extension action (NXAST_RAW_NOTE) that stores a byte string (in hex format, with optional dot separators) in the flow definition without any runtime effect on the packet. The maximum size is bounded by the `ofpbuf_oversized` check (about 64 KB, the OpenFlow message payload max), but in practice usage is below 32 bytes. When `dump-flows` shows the rule, the note hex is visible for the operator to decode.

**Role.** A pure annotation or debug marker: embed inline metadata in the flow rule for post-mortem audit. Use cases: encode a build version, controller-side rule UUID, deployment timestamp, or a conditional flag the controller must re-read on reconciliation. OVN does not use `note` (it uses `external_ids` on the SBDB Logical_Flow), but custom controller projects such as Ryu and Faucet often use `note` to stamp rule provenance.

**Parser mechanics in OVS.** The function `parse_NOTE` in `lib/ofp-actions.c` (OVS branch v3.3) parses the hex string via `ofpbuf_put_hex`, accepting the dot `.` and colon `:` as separators (ignored during parse). Wire encoding: `struct nx_action_note` has the header `len`, `vendor=NX_VENDOR_ID`, `subtype=NXAST_NOTE`, `note[6]` for the start of user-defined data, and may carry additional bytes after. `OFP_ASSERT(sizeof(struct nx_action_note) == 16)` guarantees a fixed 16-byte minimum wire layout. Decoder `decode_NXAST_RAW_NOTE` reads the length from the wire `nan->len` minus the offset of the note field and copies the bytes into the ofpact buffer.

**Syntax and practical examples.** Stamp a build version onto a rule:

```
ovs-ofctl add-flow br0 \
  "table=0,priority=100,in_port=1,\
   actions=note:01.02.03.04.de.ad.be.ef,output:2"
```

Verify with `dump-flows`:

```
$ ovs-ofctl dump-flows br0 | grep note
 cookie=0x0, duration=15.2s, table=0, n_packets=0, n_bytes=0,
   priority=100,in_port=1 actions=note:01.02.03.04.de.ad.be.ef,output:2
```

**Caveats and an implicit use case.** The note has no semantic effect; the kernel datapath skips this action entirely (the action_set is unchanged). The most common non-debug use: pad the rule to force re-installation on a controller upgrade (the note bytes change so the flow is `add` rather than `unchanged`). Common mistake: a hex string with an odd character count (`note:1`) is rejected by the parser. Comparison with other systems: Linux iptables `--comment` module is similar (an inline ASCII string) but limited to 256 bytes; tc filter has no comment field.

### 4.9.36.5 `sample(probability=...,collector_set_id=...)`, packet sampling for IPFIX/sFlow

**Concept.** `sample(probability=N,collector_set_id=ID,...)` is an OVS extension action (NXAST_RAW_SAMPLE / SAMPLE2 / SAMPLE3) that samples packets with probability `N/65535` and emits an IPFIX flow record to the OVSDB-configured collector_set. Required arguments: `probability` (1 to 65535) and `collector_set_id` (uint32). Optional arguments: `obs_domain_id`, `obs_point_id`, `sampling_port`, `ingress` or `egress` direction.

**Role.** Built-in flow sampling for telemetry: instead of mirroring all traffic to a collector port (bandwidth-expensive), `sample` selects per-packet probabilistically and emits an IPFIX record (header plus 5-tuple plus counters). NetFlow or sFlow analysers on the collector side aggregate the records to estimate top talkers and top applications. Different from the global `sFlow agent` of OVS (which samples every packet entering the bridge): the `sample` action is embedded in individual flow rules, making it selectively per-flow programmable, with the controller choosing which flows to sample.

**Parser mechanics in OVS.** The function `parse_SAMPLE` in `lib/ofp-actions.c` (OVS branch v3.3) parses seven keys via `ofputil_parse_key_value`: `probability`, `collector_set_id`, `obs_domain_id`, `obs_point_id`, `sampling_port`, `ingress`, `egress`. It validates `probability != 0` (zero is invalid; the datapath rejects it). Three wire variants: `NXAST_RAW_SAMPLE` (basic), `NXAST_RAW_SAMPLE2` (adds sampling_port), `NXAST_RAW_SAMPLE3` (adds ingress/egress direction, OVS 2.5 and later). Decoded correspondingly by `decode_NXAST_RAW_SAMPLE` / `SAMPLE2` / `SAMPLE3`.

**OVSDB configuration prerequisite.** The `collector_set_id` must match a row in the OVSDB table `Flow_Sample_Collector_Set`. Minimal setup:

```
ovs-vsctl -- --id=@cs create Flow_Sample_Collector_Set id=1 \
              ipfix=@i bridge=@br -- \
           --id=@i create IPFIX targets=\"10.0.0.1:4739\" \
              obs_domain_id=42 obs_point_id=0 -- \
           --id=@br get Bridge br0
```

**Syntax and practical examples.** Sample 1 percent (probability about 655) of traffic from port 1:

```
ovs-ofctl add-flow br0 \
  "table=0,priority=100,in_port=1,\
   actions=sample(probability=655,collector_set_id=1,obs_domain_id=42,obs_point_id=100),output:2"
```

Sample 10 percent with ingress direction (OVS 2.5 and later):

```
ovs-ofctl add-flow br0 \
  "table=0,priority=100,ip,\
   actions=sample(probability=6553,collector_set_id=1,obs_domain_id=42,obs_point_id=100,ingress),output:2"
```

**Observation and caveats.** Verify the IPFIX export with `tcpdump -i any -nn 'udp port 4739'` on the collector host to see the records arrive. Caveat: sampling does not stop the pipeline; `output:2` still executes (the sample is a parallel side-effect). The probability granularity is 1 in 65,535, so the minimum is about 0.0015 percent. Common mistake: missing the `Flow_Sample_Collector_Set` row makes the action a silent no-op (the datapath does not emit IPFIX and no error is logged). Detection: `ovs-appctl dpctl/dump-flows | grep sample` shows n_packets rising, but the collector receives no records. Verify the collector_set exists with `ovs-vsctl list Flow_Sample_Collector_Set`. Comparison with other systems: Linux tc `flower` plus `mirred` has no built-in sampling rate (use the iptables `statistic` random module); P4 `clone(I2E, mirror_session)` clones the full packet and also has no built-in rate (an extern call is needed).

## 4.9.37 Peripheral treatment for five NXM stack plus register actions

This section adds focused peripheral treatments (about 15 to 30 lines per action) for five peripheral actions in the NXM stack plus register-manipulation group: `move:src->dst` (NXM field-to-field copy), `copy_field:src->dst` (the OF 1.5 standard plus ONF 1.3 extension form), `output:NXM_NX_REG0` (indirect output reading the port number from a register), `push:reg0` (push a field/register value onto the per-packet stack), and `pop:reg0` (pop from the stack into a field/register). Different from §4.9.19 (the brief catalog for `move` and `load`) and §4.9.31 (the eight-attribute Anatomy for the stack), this section concentrates on the actual parser in upstream OVS, the wire encoding, the OF 1.5 versus NXM version negotiation, and the artefacts an engineer actually types on the CLI.

### 4.9.37.1 `move:src->dst`, NXM extension field-to-field copy

**Concept.** `move:SRC[range]->DST[range]` is an OVS extension action that copies bits from a source field or subfield to a destination field or subfield. The bit-range syntax `[lo..hi]` or `[bit]` is optional; without it, the full width is copied. The two fields must have the same width (a rule like `move:reg0[0..7]->reg1[0..15]` is rejected by the parser with `bit count mismatch`). Type: pure register manipulation; no packet modification unless the destination is a header field (for example, `eth_src`).

**Origin and context.** `move` is the Nicira NXAST_REG_MOVE extension that has existed since early Open vSwitch (1.0 and earlier) and is the precursor of the OF 1.5 standard `OFPAT_COPY_FIELD` (type 28). Purpose: programmable swap, save/restore through a register, and ARP responder/L3 NAT patterns implemented in the data plane without a controller round-trip. Before OF 1.5, this was the only way to copy a generic field to a generic field (OF 1.0 had no such primitive).

**Parser mechanics in OVS branch-2.17.** The function `parse_REG_MOVE` in `lib/ofp-actions.c` v2.17 line 2713 only wraps `nxm_parse_reg_move` from `lib/nx-match.c`. The parser reads the `SRC->DST` string and populates `struct ofpact_reg_move` with two subfields (`src`, `dst`); each subfield has `field` (mff identifier), `ofs` (bit offset), and `n_bits` (bit width). Wire encoding: `encode_REG_MOVE` line 2674 picks the raw type by OF version: OF 1.5 and later emit `OFPAT_RAW15_COPY_FIELD` (16-byte standard), OF 1.3-1.4 emit `ONFACT_RAW13_COPY_FIELD` (24-byte ONF extension) or `NXAST_RAW_REG_MOVE` (16-byte Nicira) depending on whether `move->ofpact.raw` was set during a prior decode. Default OF 1.0-1.2 emits `NXAST_RAW_REG_MOVE`. `struct nx_action_reg_move` (line 2574, `OFP_ASSERT == 16`) carries `n_bits`, `src_ofs`, `dst_ofs`, then OXM/NXM headers for src and dst.

**Syntax and practical examples.** Swap eth_src and eth_dst through a temporary register:

```
ovs-ofctl add-flow br0 \
  "table=0,priority=100,arp,arp_op=1,\
   actions=move:NXM_OF_ETH_DST[]->NXM_NX_REG0[0..47],\
           move:NXM_OF_ETH_SRC[]->NXM_OF_ETH_DST[],\
           move:NXM_NX_REG0[0..47]->NXM_OF_ETH_SRC[],\
           IN_PORT"
```

Bit-range partial copy:

```
ovs-ofctl add-flow br0 \
  "priority=100,actions=move:OXM_OF_METADATA[12..15]->NXM_NX_REG1[0..3]"
```

**Caveats and version drift.** `move` does not error when the rule does not match the source field; the datapath reads an undefined value. Common mistake: a bit-range mismatch (`move:reg0[0..7]->reg1[0..15]` is rejected with `bit count mismatch: 8 != 16`). On a modern OF 1.5 controller, the recommendation is to use `copy_field` syntax instead of `move` for cross-vendor portability. Comparison with other systems: P4 `meta.field_a = meta.field_b;` is a compile-time direct assignment; Linux tc `pedit munge` only writes constants, no field-to-field copy; iptables `MARK --set-xmark` only loads a constant.

### 4.9.37.2 `copy_field:src->dst`, the OF 1.5 standard form of move

**Concept.** `copy_field:SRC[range]->DST[range]` is the standard variant of `move`, semantically equivalent but with a different wire encoding: it uses `OFPAT_COPY_FIELD` (type 28) on OF 1.5, or `ONFACT_COPY_FIELD` (experimenter ID 0x4F4E4600 ONF, subtype 3200) on OF 1.3-1.4. OVS accepts both `move:` and `copy_field:` on the CLI; the parser maps both into the same `struct ofpact_reg_move`, only differing at encode time when picking the raw type by OF version.

**Origin and motivation.** OF 1.5 (2014) added `OFPAT_COPY_FIELD` to standardise what NXM had been doing, avoiding vendor-extension lock-in. The ONF also published it as an extension for OF 1.3-1.4 (through the `experimenter` action) so ONF-compliant switches that had not upgraded to 1.5 could still use it. OVS implements all three wire forms (`OFPAT_RAW15_COPY_FIELD`, `ONFACT_RAW13_COPY_FIELD`, `NXAST_RAW_REG_MOVE`).

**Wire encoding details.** `struct ofp15_action_copy_field` (lines 2429-2443, `OFP_ASSERT == 16`) carries `n_bits`, `src_offset`, `dst_offset`, `pad2[4]`, then two OXM headers (4 to 8 bytes each). `struct onf_action_copy_field` (lines 2446-2463, `OFP_ASSERT == 24`) is larger because it has the experimenter header `ovs_be32 vendor=0x4F4E4600 + ovs_be16 exp_type=3200` before the body. Decoding is unified through the helper `decode_copy_field__` at line 2576: parse src and dst offsets plus n_bits, populate `struct ofpact_reg_move`, set `move->ofpact.raw = ONFACT_RAW13_COPY_FIELD` so the correct wire form is re-emitted on encode.

Quick comparison of the three wire forms in upstream OVS 2.17:

| Form | Raw type | Type ID | Sizeof struct | When emitted |
|------|----------|---------|---------------|--------------|
| OF 1.5 standard | `OFPAT_RAW15_COPY_FIELD` | 28 (action_type) | 16 bytes plus OXM follow | OF 1.5 and later negotiated |
| ONF 1.3-1.4 extension | `ONFACT_RAW13_COPY_FIELD` | 3200 (experimenter subtype) | 24 bytes plus OXM follow | OF 1.3-1.4 with ONF capability |
| Nicira NXM legacy | `NXAST_RAW_REG_MOVE` | NXAST subtype | 16 bytes plus OXM follow | OF 1.0-1.2 or default fallback |

**Syntax and version negotiation.** CLI: `actions=copy_field:NXM_NX_REG0[0..7]->NXM_NX_REG1[0..7]`. When the controller speaks OF 1.5, OVS emits `OFPAT15_COPY_FIELD`. When it speaks OF 1.3-1.4 with the ONF extension capability, OVS emits `ONFACT13_COPY_FIELD`. When it speaks OF 1.0-1.2, OVS emits `NXAST_RAW_REG_MOVE` (Nicira).

The engineer uses the `-O OpenFlow15` flag of `ovs-ofctl` to force protocol version 1.5 and verify the wire format:

```
ovs-ofctl -O OpenFlow15 add-flow br0 \
  "table=0,priority=100,ip,actions=copy_field:NXM_NX_REG0[0..15]->NXM_NX_REG1[0..15],output:2"

ovs-ofctl -O OpenFlow15 dump-flows br0
```

`format_REG_MOVE` always prints the `move:` prefix regardless of wire encoding, so `dump-flows` does not visually distinguish `move` from `copy_field`. The engineer must verify the wire encoding through `ovs-ofctl monitor br0 -P openflow15` to inspect the action's raw bytes.

**Practical caveats.** Subtle version differences: an OVS controller may receive `copy_field` from a remote OF 1.5 controller, decode it and set `raw=ONFACT_RAW13_COPY_FIELD` (even though the decode came from OFPAT_RAW15), then on re-encode it may fall back to the ONF 1.3 form when the controller downgrades. When testing interop with Faucet, Ryu, or ODL, enable OpenFlow 1.5 explicitly via `set Bridge br0 protocols=OpenFlow15` and verify with `ovs-ofctl -O OpenFlow15 dump-flows`. Comparison with other systems: P4 has no version negotiation concept; the compile-time `header.field_a = header.field_b;` is fixed; the ODL plus OVSDB controller may prefer the ONF extension form for backward compatibility.

### 4.9.37.3 `output:NXM_NX_REG0`, output to a port number read from a register

**Concept.** `output:FIELD[range]` (the indirect variant of `output:N`) reads the port number from a subfield register rather than from a hard-coded value. The most common usage is `output:reg0` or `output:NXM_NX_REG0[0..15]`. The datapath reads a 16-bit (or 32-bit from OF 1.1 and later) value from the subfield position and uses it as the `OFPP_*` port destination. This lets the pipeline compute a port dynamically (ECMP hash, load balance, NAT remap) and forward once at the end.

**Origin.** Open vSwitch 1.3 introduced this as the NXM extension `NXAST_OUTPUT_REG` (a subtype). When OF 1.1 and later added 32-bit port numbers, OVS added `NXAST_OUTPUT_REG2` (subtype 32) supporting a 32-bit src plus 64-bit experimenter OXM. In upstream code `lib/ofp-actions.c` v2.17, `struct nx_action_output_reg` (lines 1192-1205, `OFP_ASSERT == 24`) holds `ofs_nbits` (encoded as `(ofs<<6)|(n_bits-1)`), `src` (NXM header 4 bytes), `max_len` (bytes sent when the port is OFPP_CONTROLLER), `zero[6]` reserved zero. `struct nx_action_output_reg2` (lines 1212-1226) reorganises to hold a 4-8 byte OXM header in `pad[10]` to support longer experimenter OXMs.

**Parser mechanics in OVS branch-2.17.** The function `parse_OUTPUT_REG` at line 1317 only wraps the common `parse_OUTPUT` (a few lines above): the CLI parser, when meeting `output:<token>`, decides whether the token is a numeric port number or a field name. When it is a field, it emits `ofpact_output_reg` with src plus max_len; when it is a port number, it emits an ordinary `ofpact_output`. Encoder `encode_OUTPUT_REG` line 1287 picks `NXAST_OUTPUT_REG` (basic) when src uses a standard NXM header; it picks `NXAST_OUTPUT_REG2` when src is an experimenter OXM or when `move->ofpact.raw == NXAST_RAW_OUTPUT_REG2` (decoded earlier). Decoder `decode_NXAST_RAW_OUTPUT_REG` line 1229 enforces `is_all_zeros(naor->zero, 6)` and fails with `OFPERR_OFPBAC_BAD_ARGUMENT` when the reserved bytes are non-zero (vendor mismatch).

**Syntax and practical examples.** ECMP hash to pick one of four uplinks:

```
ovs-ofctl add-flow br0 \
  "table=0,priority=100,ip,\
   actions=multipath(symmetric_l4,1024,modulo_n,4,32,NXM_NX_REG0[0..15]),\
           output:NXM_NX_REG0[0..15]"
```

Load a constant and then output indirectly:

```
ovs-ofctl add-flow br0 \
  "priority=100,in_port=1,actions=load:0x3->NXM_NX_REG0[0..15],output:NXM_NX_REG0[0..15]"
```

(Equivalent to `output:3` but routed through a register; this pattern is common when the port number is computed in an earlier table and then output in a later table.)

**Caveats and common mistakes.** The field width must be enough to hold the port number: `output:NXM_NX_REG0[0..7]` only holds 8 bits (max port 255 for OF 1.0, not enough for OF 1.1 and later 32-bit ports). When the port number in the register is 0 or invalid (the port does not exist), the datapath drops silently. Verify with `ovs-appctl ofproto/trace br0 in_port=1,...` to see `output:N` in the trace path. Comparison with other systems: P4 `standard_metadata.egress_spec = meta.computed_port;` is a direct field assignment; Linux tc `mirred redirect dev` has no indirect form (multiple rules per port are required); iptables `--to-destination` is not equivalent because it is L3 NAT with different semantics.

### 4.9.37.4 `push:reg0`, push a field/register value onto the per-packet stack (NXM)

**Concept.** `push:FIELD[range]` pushes the current value of a subfield onto the top of the OVS per-packet stack. The stack lifetime is bounded to one packet being processed through the pipeline; it resets between packets. The stack entry's bit width equals the subfield's bit width. The action does not consume a persistent register slot; the "save-modify-restore" pattern uses push and pop instead of allocating a temporary register in reg0..reg15.

**Origin.** The Nicira NXAST_STACK_PUSH extension was added in OVS 1.11 and later (2013) and is not in the OpenFlow standard up to OF 1.5 (no stack semantics standardised). In v2.17 code, `struct nx_action_stack` (shared between push and pop) carries `offset` (the bit ofs in the field) followed by an OXM header and a 16-bit `n_bits` at the end. Encoding is unified through the helper `encode_STACK_op` (line 3514); push and pop only differ between `put_NXAST_STACK_PUSH` and `put_NXAST_STACK_POP`.

**Parser mechanics in OVS branch-2.17.** The function `parse_STACK_PUSH` at line 3544 only wraps `nxm_parse_stack_action` from `lib/nx-match.c` (the parser is shared between push and pop, distinguished by `ofpact_put_STACK_PUSH` versus `ofpact_put_STACK_POP`). The decoder `decode_NXAST_RAW_STACK_PUSH` line 3490 calls `decode_stack_action` followed by `nxm_stack_push_check` to validate the field is readable. The formatter `format_STACK_PUSH` line 3556 prints the `push:` prefix plus the subfield notation through `nxm_format_stack_push`.

**Syntax and practical examples.** Save-modify-restore on in_port:

```
ovs-ofctl add-flow br0 \
  "table=0,priority=100,in_port=1,\
   actions=push:NXM_OF_IN_PORT[],\
           load:0->NXM_OF_IN_PORT[],\
           resubmit(,1),\
           pop:NXM_OF_IN_PORT[],\
           output:2"
```

Save reg0 before computing a hash that destroys it:

```
ovs-ofctl add-flow br0 \
  "priority=100,tcp,actions=push:NXM_NX_REG0[],\
                            multipath(symmetric_l4,1024,modulo_n,4,32,NXM_NX_REG0[0..15]),\
                            note:01.02,\
                            pop:NXM_NX_REG0[],\
                            output:NXM_NX_REG0[0..15]"
```

(Caveat: this pattern has a semantic bug because the pop happens before the output and destroys the hash result. The engineer must understand the order; the typical use of push is to preserve a value that is **not** needed for a later output, for example saving metadata before a `resubmit` to another table that will overwrite reg0.)

**Caveats and common mistakes.** The stack maximum depth is bounded (OVS hard-codes 64 entries per packet in `lib/ofp-actions.c`). A push beyond the depth produces `OFPERR_NXBAC_STACK_TOO_DEEP`. The common pattern is at most 4 push deep (used to save eth_src plus eth_dst plus reg0 plus in_port). Comparison with other systems: P4 has no native stack semantics; an auxiliary metadata field must be used. eBPF programs have per-CPU maps but not LIFO. tc filter has no stack concept at all. Stack push and pop are unique to OVS NXM; there is no equivalent in carrier hardware switches (the Broadcom Trident is programmable, but its register architecture differs).

### 4.9.37.5 `pop:reg0`, pop the top of the stack into a field/register (NXM)

**Concept.** `pop:FIELD[range]` takes the entry at the top of the per-packet stack, assigns it to the subfield, and removes the entry from the stack. The popped entry's bit width must equal the destination subfield's bit width (a mismatch results in `OFPERR_NXBAC_STACK_BAD_TYPE` action error and the datapath drops the packet). Popping an empty stack yields `OFPERR_NXBAC_STACK_UNDERFLOW`. The action pairs with `push`; every push must have a matching pop to avoid stack leaks.

**Origin.** Same OVS 1.11 and later as push, NXAST_STACK_POP extension. The same `struct nx_action_stack` shares wire encoding with push (only the subtype differs). The decoder `decode_NXAST_RAW_STACK_POP` line 3502 calls `decode_stack_action` then `nxm_stack_pop_check` to validate that the field is writable (a pop into a read-only field such as `eth_type` is rejected).

**Parser mechanics in OVS branch-2.17.** The function `parse_STACK_POP` at line 3550 only wraps `nxm_parse_stack_action` (shared with push). Encoder `encode_STACK_POP` line 3537 calls `put_NXAST_STACK_POP` (a different subtype from push, with the body identical). Formatter `format_STACK_POP` line 3563 prints the `pop:` prefix plus the subfield notation. `check_STACK_POP` (after line 3573) calls `nxm_stack_pop_check` to validate the context; type or width mismatches error out.

**Syntax and practical examples.** Restore in_port after a resubmit:

```
ovs-ofctl add-flow br0 \
  "table=1,priority=50,actions=pop:NXM_OF_IN_PORT[],output:NXM_OF_IN_PORT[]"
```

Save-modify-restore reg0 for a temporary computation:

```
ovs-ofctl add-flow br0 \
  "priority=100,tcp,actions=push:NXM_NX_REG0[],\
                            load:0xdeadbeef->NXM_NX_REG0[],\
                            resubmit(,2),\
                            pop:NXM_NX_REG0[],\
                            output:3"
```

(Save the current reg0, load a temporary constant for table 2 to use, then pop to restore the original reg0 before output, so downstream rules reading reg0 still see the original value.)

**Caveats and debugging.** A typical bug: a pop without a matching push (`mismatched stack`) makes OVS `ofproto/trace` report `stack underflow` and the pipeline drops the packet. Verify with `ovs-appctl ofproto/trace br0 'in_port=1,tcp'` to see whether the trace shows `pop:NXM_NX_REG0[]` succeeding plus the popped value. Bit-range mismatch: `push:reg0[0..15]` followed by `pop:reg0[0..7]` errors out because a 16-bit stack entry cannot pop into an 8-bit field. Comparison with other systems: P4 has no native pop semantic; an eBPF tail call can simulate state save but not LIFO; carrier hardware switches (Cisco Nexus) do not expose a stack action, only plain register read/write.

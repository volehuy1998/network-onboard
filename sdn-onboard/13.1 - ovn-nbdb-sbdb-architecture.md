## 13.1.1 NBDB key tables

| Table | Mục đích |
|---|---|
| `Logical_Switch` | Logical L2 domain |
| `Logical_Switch_Port` (LSP) | Port trên LS — map sang VM/container |
| `Logical_Router` | Logical L3 router |
| `Logical_Router_Port` (LRP) | Interface router (connect LS) |
| `Logical_Router_Static_Route` | Static route trong router |
| `ACL` | ACL applied trên port group |
| `NAT` | SNAT/DNAT rule trên logical router |
| `Load_Balancer` | L4 LB applied trên LS hoặc LR |
| `Port_Group` | Nhóm LSP cho ACL aggregation |
| `DHCP_Options` | DHCP config cho logical switch |

---

## 13.1.2 SBDB key tables

| Table | Mục đích |
|---|---|
| `Chassis` | Compute node running ovn-controller |
| `Port_Binding` | Logical port → physical chassis binding |
| `Logical_Flow` | Compiled flow rule |
| `Datapath_Binding` | Logical datapath → tunnel_key mapping |
| `MAC_Binding` | IP → MAC learned (ARP responder) |
| `Chassis_Private` | Per-chassis state |

---

## 13.1.3 ovn-northd translator

`ovn-northd` daemon read NBDB, compute logical flows, write SBDB. Translate high-level intent thành sequence logical flows.

Flow pipeline nhiều tables (24 ingress + 27 egress trong OVN 22.03):

- **Ingress 0**: admission control.
- **Ingress 1-10**: L2 policy (port security, ACL, LB).
- **Ingress 11-20**: L3 policy (routing, NAT).
- **Ingress 21-23**: egress port selection.
- **Egress 0-27**: mirror ingress cho reverse direction.

Khi admin thêm `ovn-nbctl acl-add`, northd recompute pipeline và update SBDB.

---

## 13.1.4 ovn-controller: per-chassis agent

Mỗi compute chạy `ovn-controller` — daemon connect SBDB, filter row liên quan (`Port_Binding.chassis == local`), translate logical flow thành OpenFlow rule trên `br-int`.

Filter qua monitor_cond — chassis chỉ nhận row liên quan.

`ovn-controller` cũng:
- Advertise local chassis trong `Chassis` table.
- Create tunnel port Geneve đến remote chassis.
- Handle MAC binding cho ARP responder.

## 13.1.5 Workflow end-to-end

**Admin tạo logical switch + port**:

```bash
[admin]# ovn-nbctl ls-add ls1
[admin]# ovn-nbctl lsp-add ls1 vm1
[admin]# ovn-nbctl lsp-set-addresses vm1 "aa:bb:cc:00:00:01 10.0.0.1"
```

Workflow:

1. `ovn-nbctl` write NBDB.
2. `ovn-northd` detect change, compute Datapath_Binding + Logical_Flow, write SBDB.
3. `ovn-controller` trên chassis nơi vm1 schedule detect Port_Binding change.
4. ovn-controller translate logical flow → OpenFlow.
5. Khi vm1 khởi động, OVS port tagged `external_ids:iface-id=vm1` → ovn-controller match, update Port_Binding.chassis.
6. Traffic từ vm1 hit rule đã cài.

## 13.1.6 Essential commands

`ovn-nbctl`:

```bash
ovn-nbctl ls-add <name>
ovn-nbctl lsp-add <ls> <lsp>
ovn-nbctl lsp-set-addresses <lsp> <mac ip>
ovn-nbctl lsp-set-port-security <lsp> <mac ip>
ovn-nbctl lr-add <name>
ovn-nbctl lrp-add <lr> <lrp> <mac> <cidr>
ovn-nbctl lr-route-add <lr> <prefix> <nexthop>
ovn-nbctl lr-nat-add <lr> <type> <ext_ip> <logical_ip>
ovn-nbctl acl-add <pg> <direction> <priority> <match> <action>
ovn-nbctl lb-add <lb> <vip> <backends>
```

`ovn-sbctl`:

```bash
ovn-sbctl show
ovn-sbctl lflow-list
ovn-sbctl chassis-list
```

## Điểm cốt lõi cần nhớ

Thứ nhất, NBDB là intent storage (WYSIWYG cho admin); SBDB là compiled form (10×+). Admin chỉ touch NBDB.

Thứ hai, `ovn-northd` là compiler — translate NB → SB. Nếu northd crash, existing SB flows vẫn work nhưng NB change không propagate.

Thứ ba, `ovn-controller` per-chassis là distributed control plane — scale tuyến tính.

Thứ tư, pipeline logical flows 24+27 tables (OVN 22.03) — nhiều hơn OpenFlow native. northd compile logical abstraction, ovn-controller translate xuống OpenFlow thực.

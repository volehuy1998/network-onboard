## Learning objectives

After completing this part, you can:

1. **Describe** the motivation for moving from ML2/OVS (Neutron agent-based) to ML2/OVN (controller-based), based on the rationale published by Red Hat and Canonical: reduce operational complexity, improve scalability, and remove many agent processes.
2. **Analyse** a feature-parity comparison table between ML2/OVS and ML2/OVN at OpenStack Yoga and later, identifying which features are at parity and which still have gaps.
3. **Apply** the standard migration procedure from Red Hat and Canonical: pre-migration readiness check, offline or online migration, verification, and rollback if needed.
4. **Evaluate** when to migrate (a small cluster can wait for ML2/OVN to become the default; a large cluster should migrate to address scaling problems) and the cost-benefit tradeoff.

## 13.13.1 Why migrate from ML2/OVS to ML2/OVN?

According to the Red Hat OpenStack Platform Networking Guide and the upstream Neutron documentation, there are four main motivations:

**(1) Eliminate many distributed agents:**
- ML2/OVS deploys, on each chassis: `neutron-openvswitch-agent` (OVS configuration), `neutron-l3-agent` (L3 routing plus SNAT), `neutron-dhcp-agent` (DHCP server), `neutron-metadata-agent` (metadata service). On top of that, `neutron-server` runs on the controller node.
- Each agent is a Python process that uses RabbitMQ to communicate with neutron-server, so RabbitMQ overhead grows linearly with the number of chassis. A 100-chassis cluster has 400 agents and 400 RabbitMQ connections.
- ML2/OVN needs only `ovn-controller` plus optional `ovn-metadata-agent` on a chassis, and `ovn-northd` plus `ovsdb-server` on the control plane. The number of processes is reduced by about fifty percent.

**(2) Scalability:**
- ML2/OVS stores state in the Neutron database (MySQL) and uses RabbitMQ to push changes. As the cluster grows, RabbitMQ becomes the bottleneck.
- ML2/OVN uses an OVSDB Raft cluster as the distributed database plus monitor_cond to push changes, scaling to thousands of chassis (Red Hat benchmark 2022: OVN tested with more than 500 chassis).
- The OVN Logical Flow model lets the system compile once at ovn-northd instead of issuing a separate RPC to each agent.

**(3) Native L3 HA plus DVR:**
- ML2/OVS L3 HA must be deployed separately through `neutron-l3-agent` with Keepalived plus VRRP, which is configuration-heavy.
- ML2/OVN has built-in HA chassis groups plus BFD (Part 13.6), so HA routing is the default behavior with no external tooling required.

**(4) Simpler Distributed Virtual Routing (DVR):**
- ML2/OVS DVR requires neutron-l3-agent on every compute node and namespace-based routing, which is complex to debug.
- ML2/OVN Distributed Router (DR) is the default. It does not require separate configuration and does not require namespaces (Part 13.11).

**Official migration roadmap:**
- **OpenStack Train (2019):** ML2/OVN is stable and recommended for new deployments.
- **OpenStack Wallaby (2021):** ML2/OVN reaches feature parity with ML2/OVS for most use cases.
- **OpenStack Yoga (2022):** ML2/OVN is the default driver in DevStack and is recommended for new installs.
- **Red Hat OSP 16.2 and later:** ML2/OVN is the default; OSP 17 and later remove ML2/OVS from the installer (still supported through custom configuration).
- **OpenShift 4.12 (January 2023):** OVN-Kubernetes is the default CNI, replacing OpenShift SDN for new clusters.

## 13.13.2 Feature parity, ML2/OVS compared with ML2/OVN

The comparison table is based on the **OpenStack Neutron ML2/OVN Documentation** (docs.openstack.org/neutron/latest/ovn/) and the Red Hat RHOSP 17.1 Networking Guide, as of Yoga and 2022 Q3:

| Feature | ML2/OVS | ML2/OVN Yoga and later | Notes |
|-----------|---------|---------------|-------|
| Tenant isolation (VLAN/VXLAN/GRE) | Yes | Yes (Geneve by default) | OVN recommends Geneve. |
| Security Group | Yes (iptables) | Yes (OVN ACL) | OVN performs better with conntrack. |
| DHCP | neutron-dhcp-agent | OVN native DHCP | See Part 13.10. |
| DNS | neutron-dns plus dnsmasq | OVN native DNS | OVN is limited for recursive lookups. |
| Metadata service | neutron-metadata-agent | ovn-metadata-agent | Equivalent. |
| L3 routing | neutron-l3-agent | OVN Logical Router | See Part 13.2. |
| Centralised SNAT | l3-agent | OVN Gateway Router | See Part 13.11. |
| DVR | Yes (legacy DVR plus DVR-HA) | OVN Distributed Router | Default in OVN. |
| L3 HA | Keepalived plus VRRP | OVN HA Chassis Group plus BFD | See Part 13.6. |
| Floating IP | Yes | Yes | Equivalent. |
| Load Balancer (LBaaS) | Octavia plus HAProxy amphora | Octavia plus OVN (Yoga and later) | OVN provider driver new in Yoga. |
| QoS | Yes | Yes (OVN 22.03 and later) | Full parity. |
| BGP dynamic routing | Neutron BGP agent (dragonflow) | OVN-BGP-Agent (experimental) | Slight gap. |
| Multi-segment network | Yes | Yes (OVN 21.03 and later) | Caught up at Yoga and later. |
| Trunk port | Yes | Yes (OVN 21.12 and later) | Caught up at Yoga. |
| VPN as a Service (VPNaaS) | Yes | No (deprecated upstream) | Need to keep ML2/OVS or use external VPN. |
| Firewall as a Service (FWaaS v2) | Yes | No native (OVN ACL is sufficient) | Security Group covers most use cases. |
| Macvtap agent | Yes | No | Narrow use case, most deployments do not need it. |
| IPv6 first-hop security (RA guard) | Yes | Yes (OVN 22.03 and later) | Parity. |

**Features available only in ML2/OVN (not in ML2/OVS):**
- Multi-chassis port binding for seamless live VM migration (Part 19.0).
- Distributed stateful SNAT through distributed conntrack.
- Logical Switch Port port-security based on dynamic_addresses plus the port_security list.

**Gaps to be aware of before migration:**
1. **VPN as a Service.** If the deployment uses Neutron VPNaaS, plan a replacement using an external VPN gateway or accept that there is no in-cloud VPN.
2. **FWaaS v2.** If used, convert the rules to equivalent Security Groups.
3. **Macvtap agent.** If the deployment uses macvtap NIC mode, test carefully because OVN does not support it.
4. **BGP speaker.** If the deployment uses the Neutron BGP speaker (dragonflow), deploy `ovn-bgp-agent` (still in incubation).

## 13.13.3 Pre-migration readiness check

According to the OpenStack Neutron ML2/OVN migration documentation, the cluster must pass the following conditions before migration:

**(1) OpenStack version supporting the migration tooling:**
- Train (2019) and later for offline migration.
- Ussuri (2020) and later for stable migration tooling.
- Yoga (2022) and later for online migration (well tested).

**(2) Inventory of agents and the data plane:**
```
[controller]# openstack network agent list
+----+--------------------+-------+------+
| ID | Agent Type         | Host  | State|
+----+--------------------+-------+------+
| .. | Open vSwitch agent | ... | UP   |
| .. | L3 agent           | ... | UP   |
| .. | DHCP agent         | ... | UP   |
| .. | Metadata agent     | ... | UP   |
+----+--------------------+-------+------+
```
All agents must be UP. An agent that is down means data is lost when migrating.

**(3) Inventory of networks, subnets, and ports:**
```
[controller]# openstack network list
[controller]# openstack subnet list
[controller]# openstack port list
[controller]# openstack router list
```
Export so the post-migration state can be compared (verify count and attributes).

**(4) Check incompatible features:**
```
[controller]# openstack security group list
[controller]# openstack vpn service list    # If present, plan an external VPN.
[controller]# openstack firewall list        # If present, convert to SG.
```

**(5) Confirm the OVS version is sufficient:**
```
[chassis]# ovs-vsctl --version
```
OVS 2.11 and later for OVN 20.03 and later. OVS 2.17 and later for OVN 22.03 and later (recommended).

**(6) Confirm the Geneve MTU is suitable for the underlay:**
A pod MTU of 1500 bytes plus a Geneve overhead of 50 to 80 bytes equals an outer MTU of 1550 to 1580 bytes. The underlay must have an MTU of at least 1600 bytes. Test:
```
[chassis-x]# ping -M do -s 1570 <chassis_y_ip>
```
If this fails, increase the underlay MTU before migration (avoid fragmentation affecting VMs after migration).

## 13.13.4 Offline migration procedure (recommended for small clusters)

Offline migration is the simplest. There is downtime but the risk is low. According to the Red Hat OSP 16.2 migration guide:

**Phase 1, preparation (no downtime):**
```
[controller]# # Back up the Neutron database.
[controller]# mysqldump -u root -p neutron > neutron-backup-$(date +%Y%m%d).sql

# Record the Neutron state for later verification:
[controller]# openstack network list -f yaml > network-before.yaml
[controller]# openstack subnet list -f yaml > subnet-before.yaml
[controller]# openstack port list -f yaml > port-before.yaml
[controller]# openstack router list -f yaml > router-before.yaml
[controller]# openstack security group list -f yaml > sg-before.yaml
```

**Phase 2, stop the data plane (downtime begins):**
```
[all_chassis]# # Stop every Neutron agent on every compute node.
[all_chassis]# systemctl stop neutron-openvswitch-agent
[all_chassis]# systemctl stop neutron-l3-agent
[all_chassis]# systemctl stop neutron-dhcp-agent
[all_chassis]# systemctl stop neutron-metadata-agent

[controller]# systemctl stop neutron-server
```

**Phase 3, run the migration tool:**
```
[controller]# # Install the migration tool from the neutron repository.
[controller]# pip install neutron-migration-tool    # Or from the RHOSP package.

[controller]# # Run a dry-run first to verify.
[controller]# neutron-ovn-migration --dry-run --config-file /etc/neutron/neutron.conf

[controller]# # If the dry-run is OK, run the actual migration.
[controller]# neutron-ovn-migration --config-file /etc/neutron/neutron.conf
```

The migration tool performs the following:
1. Read the Neutron DB (MySQL) and identify every network, subnet, port, router, and security group.
2. Translate them to the equivalent OVN NB DB objects (Logical_Switch, Logical_Router, Logical_Switch_Port, ACL).
3. Write to the OVN NB DB through ovn-nbctl.
4. Verify count: number of Logical_Switch entries equals number of networks; number of Port_Binding entries equals number of Neutron ports.

**Phase 4, deploy the OVN stack and restart:**
```
[controller]# # Install OVN packages (RHOSP: redeploy via osp-director; upstream: manual).
[controller]# apt install python3-networking-ovn ovn-central   # Canonical or upstream.

[controller]# # Configure the Neutron ML2 driver to ovn instead of openvswitch.
[controller]# vi /etc/neutron/plugins/ml2/ml2_conf.ini
# [ml2]
# mechanism_drivers = ovn
# type_drivers = geneve,vlan,flat
# tenant_network_types = geneve

[controller]# systemctl start ovn-northd
[controller]# systemctl start neutron-server

[all_chassis]# # On every chassis:
[all_chassis]# systemctl start ovn-controller
[all_chassis]# systemctl start ovn-metadata-agent
```

**Phase 5, verification:**
```
[controller]# # Check the OVN state.
[controller]# ovn-nbctl show
[controller]# ovn-sbctl show

# Compare counts with the pre-migration state:
[controller]# openstack network list -f yaml | diff network-before.yaml -
[controller]# openstack port list -f yaml | diff port-before.yaml -

# Check VM-to-VM connectivity:
[VM1]# ping <VM2_IP>
```

**Typical downtime:** thirty to sixty minutes for a fifty-compute-node cluster, growing linearly with cluster size.

## 13.13.5 Online migration procedure (large cluster, no downtime)

Online migration (released with OpenStack Yoga) lets the team migrate one compute node at a time without taking the whole cluster down. According to the upstream documentation:

**Online migration architecture:**
1. The OVN stack is deployed in parallel with Neutron ML2/OVS (dual stack).
2. Networks created after migrate-start go through OVN.
3. Older networks still go through the OVS agent.
4. Migration is per network: each network is migrated separately, and a VM restart switches the data plane once.

**Phase 1, deploy the dual stack:**
```
[controller]# # Install OVN and configure ML2 with both drivers.
[controller]# vi /etc/neutron/plugins/ml2/ml2_conf.ini
# [ml2]
# mechanism_drivers = openvswitch,ovn
```

**Phase 2, drain and migrate one network at a time:**
```
[controller]# # Choose a network to migrate.
[controller]# openstack network show <network_id>

# Cordon the network (no new ports created):
[controller]# openstack network set --disable <network_id>

# Migrate the network:
[controller]# neutron-ovn-migration --network <network_id>

# Live-migrate VMs on that network to a compute node already running ovn-controller:
[controller]# nova live-migration <vm_id> <target_compute>
```

**Phase 3, remove Neutron agents after every network has migrated:**
```
[controller]# openstack network agent list
[controller]# openstack network agent delete <agent_id>    # For every OVS, L3, or DHCP agent.
```

**Downtime:** close to zero per VM (live migration restart in less than five seconds). The total migration extends over days or weeks for a large cluster, but there is no concentrated downtime.

## 13.13.6 Rollback plan

If migration fails, roll back to ML2/OVS:

**Offline rollback (phase 2 or 3 fails):**
```
[controller]# # Restore the Neutron DB from the backup.
[controller]# mysql -u root -p neutron < neutron-backup-<date>.sql

[controller]# # Switch the ML2 driver back to openvswitch.
[controller]# vi /etc/neutron/plugins/ml2/ml2_conf.ini
# mechanism_drivers = openvswitch

[controller]# # Restart Neutron and the agents.
[controller]# systemctl start neutron-server
[all_chassis]# systemctl start neutron-openvswitch-agent neutron-l3-agent ...
```

**Online rollback (phase 2 fails midway):**
- Migrate the OVN-ized networks back to the OVS mechanism (the neutron-migration tool supports a reverse mode with `--rollback`).
- Uncordon the network.
- Test connectivity.

**Data integrity check:** after rollback, verify:
```
[controller]# openstack network list | wc -l    # Compare with the pre-migration count.
[controller]# openstack port list | wc -l
```

A few real-world Red Hat customer rollback cases (per Red Hat support knowledge base):
- **Underlay MTU gap:** rollback and increase the MTU before retrying.
- **VPNaaS in use:** rollback and deprecate VPNaaS before retrying.
- **Custom Neutron extension:** rollback and rewrite the extension to be compatible with OVN.

## 13.13.7 Production case studies

According to the public statements from Red Hat, Canonical, and OpenShift:

**Red Hat OpenStack Platform (RHOSP) 16.2 to 17.1:**
- The default migration path for an existing RHOSP 13/16.1 deployment to 17.1 is offline migration.
- A typical production customer environment has 100 to 300 compute nodes; the migration takes two to four hours of downtime.
- Red Hat provides Ansible playbook automation: `tripleo-ansible-inventory` plus `openstack overcloud upgrade` automatically handle the OVN migration.
- Post-migration: the per-compute-node memory footprint drops by thirty to fifty percent (no more neutron-l3-agent or neutron-dhcp-agent).

**Canonical Charmed OpenStack (Juju):**
- Migration through Juju charm: `juju deploy ovn-central`, `juju add-relation ovn-central neutron-api`, `juju run-action --unit neutron-api/0 migrate-to-ovn`.
- The Canonical blog (August 2022) reports that telco customers (BT, Telefonica) successfully migrated clusters of more than 1000 nodes with less than one hour of downtime per wave.

**OpenShift 4.12 SDN to OVN-K8s migration:**
- Red Hat provides the migration tool `oc patch network.operator/cluster --type=merge --patch='{"spec":{"migration":...}}'`.
- The 4.10 to 4.12 migration includes OVN-K8s as the default for new clusters.
- An existing cluster can opt in through `migration.networkType: OVNKubernetes`.
- Per-node downtime: two to five minutes (node drain plus reboot plus rejoin).

Production lessons from these cases:
1. **Always test on a staging cluster** identical to the production environment before migration.
2. **Monitor KPIs during and for 72 hours after migration:** VM-to-VM latency, API response time, Neutron DB query time.
3. **Keep the Neutron DB backup for at least thirty days.** A rollback after one week is still feasible if no new VMs have been created.
4. **Communicate with the security team:** OVN ACL semantics differ slightly from iptables Security Group at the stateful edge cases.

### Guided Exercise 1: dry-run migration on OpenStack DevStack ML2/OVS to ML2/OVN

> **State verification lab:** the output below is based on OpenStack Neutron ML2/OVN documentation plus DevStack Yoga; pending verification on a real lab.

**Goal:** practise the migration procedure on a small DevStack cluster (all-in-one). Run a dry-run migration, read the report, then roll back to the initial state.

**Setup:** Ubuntu 22.04 VM with DevStack already installed (Yoga or later), Q_PLUGIN=ml2 plus Q_ML2_MECH_DRIVER=openvswitch (ML2/OVS setup).

**Steps:**

**1.** Verify the initial ML2/OVS state:
```
[devstack]$ openstack network agent list
+----+--------------------+------+
| ID | Agent Type         | State|
+----+--------------------+------+
| .. | Open vSwitch agent | UP   |
| .. | L3 agent           | UP   |
| .. | DHCP agent         | UP   |
| .. | Metadata agent     | UP   |
+----+--------------------+------+
```

**2.** Back up the Neutron DB:
```
[devstack]$ mysqldump -u root -p$MYSQL_PASSWORD neutron > /tmp/neutron-backup.sql
[devstack]$ ls -la /tmp/neutron-backup.sql
```

**3.** Run the dry-run migration:
```
[devstack]$ sudo pip install neutron-ovn-migration-tool
[devstack]$ neutron-ovn-migration --dry-run \
    --neutron-config /etc/neutron/neutron.conf \
    --output-report /tmp/migration-dryrun.log
```

Expected output:
```
Neutron to OVN migration dry-run report
=======================================
Networks to migrate: 5
Subnets to migrate: 8
Ports to migrate: 42
Routers to migrate: 3
Security groups to migrate: 7
Unsupported features: None
Migration time estimate: 10-15 minutes
```

**4.** If the dry-run is OK, do not run the actual migration (this exercise is dry-run only).

**5.** Read the report:
```
[devstack]$ cat /tmp/migration-dryrun.log | head -50
```

**Lessons learned:**

First, a dry-run is a **mandatory safety step** before any production migration. It tells you whether the cluster has incompatible features (VPNaaS, FWaaS) without actually modifying state.

Second, the migration tool estimates the time based on counts (networks times complexity). A real production cluster can take two to three times longer if there are complex network attachments.

Third, the Neutron DB backup is non-negotiable. Some rollback failures happen because the backup is incomplete or too old.

**Cleanup:** none needed (the exercise is dry-run only and does not modify state).

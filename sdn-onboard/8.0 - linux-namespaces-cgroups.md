## 8.0.1 Linux namespaces: 7 loại

Kernel 3.8+ ship đủ 7 namespace:

| Namespace | Isolates | Flag |
|---|---|---|
| `PID` | Process ID space | `CLONE_NEWPID` |
| `NET` | Network stack (interface, routing, iptables, socket) | `CLONE_NEWNET` |
| `MNT` | Mount points | `CLONE_NEWNS` |
| `UTS` | Hostname + domain name | `CLONE_NEWUTS` |
| `IPC` | System V IPC (message queue, semaphore, shared memory) | `CLONE_NEWIPC` |
| `USER` | UID/GID mapping (unprivileged containers) | `CLONE_NEWUSER` |
| `CGROUP` | cgroup hierarchy | `CLONE_NEWCGROUP` |

Mỗi namespace tạo qua flag trong `clone(2)` syscall hoặc `unshare(2)` cho existing process. Container runtime (Docker, containerd, CRI-O) combine nhiều namespace với nhau để tạo isolation đầy đủ.

USC Lab 1 (`doc/ovs/OVS.txt` Mục 1.1): "In Linux namespaces, you can isolate the global system resources between processes. Any changes to the global resource will only be visible to other processes that are members of the namespace but are invisible to other processes."

## 8.0.2 Network namespace: ip netns, /var/run/netns

Network namespace có network stack độc lập: interface, routing table, ARP cache, iptables rules, socket table, port number space.

```bash
[host]# ip netns add ns1                     # tạo persistent namespace
[host]# ls /var/run/netns/
ns1
[host]# ip netns exec ns1 ip link show        # chạy lệnh trong ns1
[host]# ip netns delete ns1                  # xóa
```

`ip netns add` tạo reference tại `/var/run/netns/<name>` — đây là bind mount của `/proc/<pid>/ns/net` của một kernel thread. Persistent sau khi process exit (khác `unshare` non-persistent).

USC Lab 1 dùng topology chuẩn:

```
   ns1 (h1) ──veth1-peer─┐
                         ├─→ OVS bridge s1 (root namespace)
   ns2 (h2) ──veth2-peer─┘
```

Cấu hình:

```bash
[host]# ip netns add h1 && ip netns add h2
[host]# ip link add veth1 type veth peer name veth1-br
[host]# ip link add veth2 type veth peer name veth2-br
[host]# ip link set veth1 netns h1
[host]# ip link set veth2 netns h2
[host]# ovs-vsctl add-port s1 veth1-br
[host]# ovs-vsctl add-port s1 veth2-br
[host]# ip -n h1 addr add 10.0.0.1/24 dev veth1
[host]# ip -n h2 addr add 10.0.0.2/24 dev veth2
[host]# ip -n h1 link set veth1 up
[host]# ip -n h2 link set veth2 up
[host]# ip netns exec h1 ping -c 3 10.0.0.2
```

Đây là foundation lab của mọi SDN experiment.

## 8.0.3 unshare và nsenter

util-linux tools cho ad-hoc namespace:

```bash
[host]# unshare --net /bin/bash           # tạo shell trong fresh net namespace
[host-ns]# ip link show                    # chỉ thấy lo
[host-ns]# exit                            # namespace destroyed

[host]# nsenter -t <pid> -n /bin/bash    # join net namespace của PID
```

Khác với `ip netns`: không persistent, không có reference trong `/var/run/netns/`. Dùng khi muốn test nhanh hoặc join namespace của container đang chạy:

```bash
[host]# PID=$(docker inspect -f '{{.State.Pid}}' <container>)
[host]# nsenter -t $PID -n ip addr
```

## 8.0.4 cgroups v1 vs v2

cgroups (control groups) — kiểm soát tài nguyên (CPU, memory, I/O, network) — khác namespace (isolation). Hai thế hệ:

**cgroup v1** (kernel 2.6.24): mỗi subsystem (cpu, memory, blkio, net_cls, net_prio, ...) có hierarchy riêng. Subsystem mạng:
- `net_cls`: gán classid cho process → `tc` filter theo classid.
- `net_prio`: gán priority per interface cho process.

**cgroup v2** (kernel 4.5 stable, 4.16 default): unified hierarchy — tất cả subsystem trong một cây duy nhất. **Loại bỏ `net_cls` và `net_prio`** — network bandwidth control chuyển sang eBPF qua `bpf_classify` hoặc `cgroup_skb` hook.

Ubuntu 22.04+ default cgroup v2. Kiểm tra:

```bash
[host]# mount | grep cgroup
cgroup2 on /sys/fs/cgroup type cgroup2 (rw,nosuid,nodev,noexec,relatime,nsdelegate,memory_recursive_prot)
```

Cho lab SDN, cgroup v2 đủ cho resource control VM/container. Cần network bandwidth QoS thì dùng `tc` trên veth/tap interface thay vì cgroup (xem Phần 8.3).

## 8.0.5 Syscall internals: clone(), unshare(), setns()

3 syscall chính tạo và join namespace, đều thuộc kernel `kernel/nsproxy.c` + `kernel/fork.c`.

### `clone(CLONE_NEWNET, ...)`

`clone(2)` là syscall fork-with-flags. Khi flag `CLONE_NEWNET` set, kernel tạo `struct net` mới và gán cho `task_struct->nsproxy->net_ns`. Process child có network stack hoàn toàn riêng.

```c
// kernel/fork.c, copy_namespaces() function (gọi từ copy_process)
if (flags & CLONE_NEWNET) {
    new_nsproxy->net_ns = copy_net_ns(flags, user_ns, old_ns->net_ns);
}
```

`copy_net_ns()` gọi `setup_net()` để initialize routing table mới, ARP cache mới, iptables hooks mới, sysctl mới. Mọi register subsystem (TCP, UDP, ICMP, netfilter) được duplicate cho `net` instance.

### `unshare(CLONE_NEWNET)`

Khác với `clone()`, `unshare()` không tạo process mới mà **detach** namespace của process hiện tại. Process vẫn cùng PID, cùng task_struct, nhưng `nsproxy` được swap với fresh namespace mới.

```c
// kernel/nsproxy.c, sys_unshare()
new_nsproxy = create_new_namespaces(unshare_flags, ...);
switch_task_namespaces(current, new_nsproxy);
```

Hệ quả: socket đang mở trước `unshare()` vẫn ở namespace cũ (vì kernel `socket->sk->sk_net` đã ref namespace cũ). Socket mới mở sau `unshare()` ở namespace mới. Đây là pattern dùng để "leak" socket cross-namespace, hữu ích cho daemon như `ovsdb-server` chạy ở root ns nhưng socket ở target ns.

### `setns(fd, CLONE_NEWNET)`

`setns()` join existing namespace qua file descriptor. Workflow:

```c
int fd = open("/proc/<pid>/ns/net", O_RDONLY);  // hoặc /var/run/netns/<name>
setns(fd, CLONE_NEWNET);
// Now in target namespace
close(fd);
```

`nsenter -t <pid> -n` dùng pattern này: `open()` /proc/<pid>/ns/net + `setns()` + `execve()` cho shell.

### Lifecycle ref counting

Kernel ref-count mỗi `struct net`:

- `+1` mỗi khi process gắn (qua `nsproxy`).
- `+1` mỗi khi mount bind (`/var/run/netns/<name>` = bind mount lên `/proc/.../ns/net`).
- `+1` mỗi khi socket open trong ns đó.

Khi ref-count = 0 → `net_drop_ns()` → kernel destroy `struct net`, free routing table, ARP cache. Đây là lý do tại sao `ip netns delete ns1` cần đảm bảo không còn process hoặc socket nào reference.

Kiểm chứng ref-count:

```bash
[host]# readlink /proc/<pid>/ns/net
net:[4026532000]    # inode number identify ns

[host]# lsns -t net
        NS TYPE NPROCS   PID USER   NETNSID NSFS                COMMAND
4026531992 net     245     1 root unassigned                     /sbin/init
4026532000 net       2 12345 root          0 /var/run/netns/ns1 sleep infinity
```

`NPROCS` = số process đang trong ns. `NETNSID` = ID assigned cho `ip netns identify` cross-reference.

## 8.0.6 Network namespace cho OVS daemon

OVS userspace `ovs-vswitchd` mặc định chạy ở **root namespace**. Khi tạo OVS port type=internal hoặc OVS port nối veth peer ở namespace khác, OVS netlink socket xuyên ns boundary qua kernel.

### Pattern 1: OVS port internal trong namespace

```bash
[host]# ovs-vsctl add-port br-int my-veth -- \
        set Interface my-veth type=internal
[host]# ip link set my-veth netns ns1     # move OVS internal port vào ns1
[host]# ip -n ns1 addr add 10.0.0.5/24 dev my-veth
```

OVS userspace quản lý port (cấu hình OpenFlow, tag VLAN), nhưng interface live trong `ns1` namespace. Kernel netlink Generic genl proxy giữa `ovs-vswitchd` (root ns) và `openvswitch.ko` (per-ns).

### Pattern 2: OVS daemon trong namespace (non-default)

Hiếm khi cần, nhưng có thể chạy `ovs-vswitchd` trong namespace riêng:

```bash
[host]# ip netns exec ns-ovs ovs-vswitchd ...
```

Use case: multi-tenant OVS instance trên cùng host. Nhưng phức tạp + không phổ biến trong production.

OVN-Kubernetes dùng pattern 1 cho mọi container interface: container netns có 1 OVS port type=internal, OVS pure root.

## 8.0.7 Anatomy Template A: `ip -n <ns> ...` + `lsns`

> **Bối cảnh khi chạy:** Kiểm tra namespace hiện tại trên chassis, troubleshoot khi process không thấy interface mong đợi.

```bash
[host]# lsns -t net
        NS TYPE NPROCS   PID USER     NETNSID NSFS                  COMMAND
4026531992 net     245     1 root   unassigned                       /sbin/init
4026532000 net       2 12345 root            0 /var/run/netns/ns1   sleep infinity
4026532007 net      15 23456 1000           1                        /usr/bin/firefox
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| `NS` (inode) | Unique per ns | Duplicate (same fs) | (n/a) | Mỗi ns có inode khác |
| `NPROCS` | > 0 cho persistent ns | 0 cho ns đang dùng | 0 cho ns nên có process | Process bị crash, ns sắp destroy |
| `NETNSID` | Assigned cho cross-ns ref | unassigned | unassigned cho ns cần ID | `ip netns set <name> <id>` |
| `NSFS` | Bind mount path | Empty (ad-hoc unshare) | Empty cho expected persistent | Persistent ns cần `/var/run/netns/` |
| `COMMAND` | Tên process chính | Unexpected | Process khác mong đợi | Wrong process trong ns |

```bash
[host]# ip -n ns1 link show
1: lo: <LOOPBACK> mtu 65536 qdisc noop state DOWN
N: veth1@if<peer>: <BROADCAST,MULTICAST> mtu 1500 ...

[host]# ip -n ns1 route
10.0.0.0/24 dev veth1 proto kernel scope link src 10.0.0.1
```

**Kịch bản bẻ gãy:**

- `lsns -t net` show ns1 với NPROCS=0 nhưng `ip netns list` vẫn show ns1: process đã chết nhưng ns vẫn sống (do bind mount). `ip netns delete ns1` cleanup.
- `ip -n ns1 link show` chỉ thấy `lo`: không có interface nào được move/create vào ns. Forgot `ip link set veth1 netns ns1`.
- `lsns` show 200+ ns net: container runtime tạo nhiều mà không cleanup. Check Docker/k8s pod count.

## 8.0.8 ▶ Capstone POE: Tại sao OVS userspace không cần move vào container netns?

> **Bối cảnh:** Kỹ sư mới gặp pattern OVS+container, nghi vấn "chạy ovs-vswitchd inside container netns sẽ scale hơn vì isolation per-container".

**Tuyên bố cần kiểm chứng:** "Chạy ovs-vswitchd trong mỗi container namespace cho isolation tốt hơn shared root namespace."

**Predict:** Sai. Pattern này không scale, vi phạm OVS architecture.

**Observe (giả định, kiểm chứng khi có lab):**

- Setup 1 ovs-vswitchd per container netns: mỗi container chạy thêm 1 process ~50-100MB memory. Cluster 100 container = 5-10GB extra memory cho OVS instances.
- OpenFlow controller phải kết nối tới N ovs-vswitchd thay vì 1 → N flow tables, N OVSDB instances. Cluster-wide policy phải sync N lần.
- Tunnel tunnel: container A muốn tunnel sang container B → 2 OVS instances phải peer Geneve/VXLAN với nhau. Underlay routing giữa container netns rất phức tạp.

**Explain:**

1. OVS architecture **assume shared kernel datapath** + 1 userspace controller per host. Mỗi `openvswitch.ko` instance là per-network-namespace (tự kernel làm), nhưng `ovs-vswitchd` userspace duy nhất quản lý policy + flow installation. Tách userspace per-container phá vỡ giả định này.
2. Container isolation (security, resource) đến từ **network namespace** không phải từ OVS instance riêng. 1 OVS root namespace + N container netns mỗi container có 1 port type=internal đã đủ isolation.
3. OVN-Kubernetes, OpenStack Neutron đều dùng pattern shared OVS root + container netns. Pattern per-container OVS không tồn tại trong production-grade orchestrator.

**Falsification test:**

- Setup 10 container, 1 OVS root → measure memory + flow install time.
- Setup 10 container, 10 OVS instances per netns → measure same metrics.
- Nếu per-container OVS cải thiện isolation hoặc throughput → counter-example. Trong thực tế, không có metric chỉ ra benefit, chỉ thấy overhead.

**Tuyên bố sai ở đâu:** "Isolation tốt hơn" là illusion. Network isolation đã có qua netns; OVS instance riêng không add isolation, chỉ duplicate state. Đúng phải: **1 OVS root + N container netns, mỗi container có 1 OVS port type=internal trong netns đó. Pattern OVN/Neutron/CNI đã chuẩn hoá.**

> **Quy tắc:** OVS userspace = root namespace, 1 instance. Container isolation = network namespace level, không phải OVS instance level.

### ▶ Guided Exercise 1 (POE): Hai namespace + veth pair, ping xuyên

**Outcomes:** Hiểu veth là point-to-point, cần routing hoặc bridge để connect nhiều namespace.

**Predict:**

Giả thuyết ngược: nếu veth tự động route giữa namespace (không cần `ip route`), ping từ ns1 đến ns2 sẽ thành công ngay sau khi gán IP.

**Observe:**

**1.** Tạo namespaces + veth pair đơn lẻ:

```bash
[host]# ip netns add ns1
[host]# ip netns add ns2
[host]# ip link add veth1 type veth peer name veth2
[host]# ip link set veth1 netns ns1
[host]# ip link set veth2 netns ns2
[host]# ip -n ns1 addr add 10.0.0.1/24 dev veth1
[host]# ip -n ns2 addr add 10.0.0.2/24 dev veth2
[host]# ip -n ns1 link set veth1 up
[host]# ip -n ns2 link set veth2 up
[host]# ip -n ns1 link set lo up
[host]# ip -n ns2 link set lo up
```

**2.** Ping:

```bash
[host]# ip netns exec ns1 ping -c 3 10.0.0.2
```

**Explain:**

Kết quả: ping thành công. Khác với giả thuyết ngược ban đầu, veth là **point-to-point**, hai đầu trong hai namespace đã có subnet chung — Linux kernel tự route qua directly-connected.

## Điểm cốt lõi cần nhớ

Thứ nhất, 7 Linux namespace cho phép isolation tài nguyên ở các chiều khác nhau. Network namespace (NET) là chiều quan trọng nhất cho SDN — tạo isolated network stack với interface, routing, iptables riêng.

Thứ hai, `ip netns` cho persistent namespace (có reference file), `unshare`/`nsenter` cho ad-hoc. Đối với SDN lab, persistent phù hợp; cho container runtime inspection, `nsenter` theo PID phù hợp.

Thứ ba, veth là point-to-point primitive, không đủ cho topology >2 node. Phải combine với Linux bridge (Phần 8.1) hoặc OVS (Khối IX) để tạo topology thực sự.

Thứ tư, cgroup v2 đã loại bỏ `net_cls`/`net_prio`. Network QoS hiện đại dùng `tc` trực tiếp hoặc eBPF — không dùng cgroup network classifier nữa.

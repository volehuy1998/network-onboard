## 9.26.1 Mở đầu, câu chuyện "keys 3612" từ upstream commit body

Giữa năm 2024, Roi Dayan (NVIDIA, Mellanox team) báo lại một hiện tượng kỳ lạ trên cluster OVS production có bật HW offload TC flower: số ukey trong userspace nhiều hơn **513 lần** số datapath flow thực tế. Trích nguyên văn commit body `180ab2fd635e` (verified qua `mcp__github__get_commit(openvswitch/ovs, 180ab2fd635e)`):

> It is observed in some environments that there are much more ukeys than actual DP flows. For example:
>
> ```
> $ ovs-appctl upcall/show
> system@ovs-system:
> flows : (current 7) (avg 6) (max 117) (limit 2125)
> offloaded flows : 525
> dump duration : 1063ms
> ufid enabled : true
>
> 23: (keys 3612)
> 24: (keys 3625)
> 25: (keys 3485)
> ```
>
> The revalidator threads are busy revalidating the stale ukeys leading to high CPU and long dump duration.
>
> — Han Zhou, Roi Dayan, Eelco Chaudron, commit `180ab2fd635e`, 2024-08-29

Hãy đọc kỹ output: `flows: current 7` (chỉ 7 flow thực trong datapath), `offloaded flows: 525` (525 flow đã offload xuống HW), **nhưng ba handler thread 23/24/25 có lần lượt 3612, 3625, 3485 ukey**. Tổng 10.722 ukey cho 7 flow active + 525 offloaded = **xấp xỉ 20 ukey ghost mỗi flow thực**. Kết quả: revalidator thread burn CPU quét các ukey không còn mapping với datapath nào, `dump duration` nhảy từ vài ms (bình thường) lên **1063 ms** (1 giây trọn vẹn mỗi cycle sweep).

Đây là incident class "stale cache leak" mà Part 9.2 đã mô tả ở mức lý thuyết. Phần 9.26 này phân tích cụ thể: tại sao xảy ra, làm sao phát hiện sớm, chuỗi CLI chẩn đoán, và 4 tầng remediation. Đồng thời rút ra design lesson áp dụng cho mọi module cache có revalidation asynchronous.

## 9.26.2 Refresher — datapath cache, upcall pipeline, revalidator thread

Trước khi đi vào forensic, cần 3 khái niệm từ Part 9.2 được cột lại rõ ràng:

**(1) Datapath flow (DP flow):** entry thực trong kernel (`openvswitch.ko`) hoặc userspace (`dpif-netdev`) datapath. Mỗi entry có megaflow match (đã wildcard các field không cần thiết) + action list. Truy vấn bằng `ovs-dpctl dump-flows` hoặc `ovs-appctl dpctl/dump-flows`. Đây là "cache" tầng thấp, số lượng ít (hundreds tới thousands), thao tác per-packet.

**(2) User-space key (ukey):** entry trong `struct umap` của ofproto-dpif-upcall module. Mỗi ukey tracking một DP flow, lưu metadata như match, actions, stats counter, revalidation state. Ukey được tạo khi userspace "học" được một DP flow (qua upcall hoặc flow dump), và xóa khi DP flow không còn mapping. Bình thường **1 DP flow ↔ 1 ukey**.

**(3) Revalidator thread:** thread trong ovs-vswitchd periodic sweep toàn bộ ukey map (mặc định mỗi 500ms), check từng ukey xem:
- DP flow tương ứng còn tồn tại? (dump từ kernel)
- Match + action có còn hợp lệ với current OpenFlow table không? (re-translate)
- Nếu stale → xóa ukey + gửi command xóa DP flow
- Nếu OpenFlow table thay đổi → update ukey + update DP flow

Process này là heart của "eventually consistent cache" trong OVS: userspace không phải source of truth realtime, nhưng revalidator đảm bảo cache converge với OpenFlow table trong vài sweep cycle.

**Điều kiện bình thường:** n(ukey) ≈ n(DP flow) ± vài entry trong transition. Dump duration ≈ vài ms. Revalidator CPU ≈ 1-5% trên 1 core.

**Khi leak:** n(ukey) >> n(DP flow), dump duration tăng tuyến tính với n(ukey), revalidator CPU bão hòa. Đây là scenario case study.

## 9.26.4 Evidence collection — 5 lệnh chẩn đoán thiết yếu

Đây là runbook tiêu chuẩn khi nghi ngờ revalidator storm (dump duration > 100ms + CPU spike trên ovs-vswitchd):

**Lệnh 1: Đọc health tổng thể datapath:**

```bash
[host]# ovs-appctl upcall/show
```

Output format (verified từ commit body `180ab2fd635e`):

```
system@ovs-system:
flows : (current <N>) (avg <N>) (max <N>) (limit <M>)
offloaded flows : <N>
dump duration : <T>ms
ufid enabled : true

<handler_id>: (keys <N>)
```

**Đọc output:**

- `current <N>`: DP flows hiện tại. Healthy: vài chục tới vài ngàn tùy workload.
- `limit <M>`: giới hạn từ `other_config:flow-limit`. Mặc định 200.000 (OVS 2.17+), có thể điều chỉnh.
- `offloaded flows`: flow đã offload xuống HW (chỉ khi TC flower hoặc DPDK-rte_flow bật). Bình thường ≤ 80% của `current`.
- `dump duration`: thời gian quét 1 cycle. Healthy: < 100ms. **> 500ms = warning, > 1000ms = critical.**
- `ufid enabled`: UFID là identifier độc lập của flow. Bình thường `true` từ OVS 2.7+.
- `<handler_id>: (keys <N>)`: số ukey per handler thread. Healthy: `Σ keys ≈ current + offloaded`. **`Σ keys >> current + offloaded` = stale leak.**

**Lệnh 2: Counter internals ovs-vswitchd:**

```bash
[host]# ovs-appctl coverage/show | grep -E "upcall|revalidator|ukey|dump"
```

Các counter quan trọng (chi tiết verify với `ovs-appctl coverage/read-counter <name>` từng entry):

- `upcall_ukey_created` — số ukey tạo mới. Delta cao liên tục = flow churn cao.
- `upcall_ukey_replace` — số ukey replace (match existing). Delta cao = flow di trú liên tục.
- `handler_duplicate_upcall` — upcall cho flow đã có ukey. Delta cao = race between handlers.
- `revalidator_sweep_time` — tổng thời gian sweep. Đọc qua Prometheus scrape ovs_exporter.
- `revalidator_purge_stale` — số ukey bị purge do stale. Spike khi remediation.
- `dump_flows_fallback` — flow dump fallback path. Tăng khi HW offload dump fail.

**Lệnh 3: Đếm flow thực trong kernel:**

```bash
[host]# ovs-dpctl dump-flows | wc -l
```

So sánh với `upcall/show` `Σ keys`. Nếu ratio `keys / dp_flows > 2` → stale leak nghi ngờ.

**Lệnh 4: Feature capability check:**

```bash
[host]# ovs-appctl dpif/show-dp-features <dp_name>
```

Liệt kê 16+ capability (masked_set_action, tnl_push_pop, ufid, sample_nesting, etc.). Không trực tiếp liên quan revalidator nhưng confirm datapath version + feature alignment với userspace expectation.

**Lệnh 5: Check ukey/flow mapping chi tiết:**

```bash
[host]# ovs-appctl upcall/dump-ukeys
```

Output lớn (có thể MB). Chỉ chạy khi cần deep dive, grep theo UFID để trace 1 flow specific. Không dùng cho alerting tự động.

## 9.26.5 Root cause analysis — 3 hypothesis test per-elim

Oncall engineer tại T+6h có thể đưa 3 hypothesis sau khi thấy `keys 3612`:

**Hypothesis H1: OpenFlow rule explosion.**

Giả thuyết: controller cài quá nhiều OpenFlow rule, sinh quá nhiều megaflow wildcarding từng 5-tuple.

**Test:** `ovs-ofctl dump-flows br0 | wc -l`. Healthy < 1000. Nếu > 10.000 → H1 khả năng đúng.

**Kết quả case này:** 127 OpenFlow rule. H1 **bác bỏ**.

**Hypothesis H2: Memory leak trong ovs-vswitchd.**

Giả thuyết: malloc leak khiến ovs-vswitchd nuốt memory → OOM kill → restart → flow wiped.

**Test:** `ps aux | grep ovs-vswitchd | awk '{print $4"% "$6/1024"MB"}'` + check `journalctl -u openvswitch-switch` cho OOM. Run `malloc_stats` via gdb attach nếu cần.

**Kết quả case này:** ovs-vswitchd 420MB (baseline 380MB), không có OOM log. H2 **bác bỏ**.

**Hypothesis H3: Stale ukey leak (confirmed).**

Giả thuyết: ukey được tạo nhanh hơn rate mà revalidator xóa, leading to unbounded growth. Nguyên nhân gốc có thể là:

- **H3a:** HW offload path rotation: flow di trú kernel ↔ NIC liên tục, mỗi cycle tạo new ukey + old ukey không kịp xóa vì dump từ kernel báo `flow not found`.
- **H3b:** Per-CPU upcall dispatch packet reorder (xem commit `0d9dc8e9` FDP-1479): cùng flow được multiple handler xử lý → multiple ukey cho cùng UFID.

**Test:** Đọc delta `upcall_ukey_created` vs `revalidator_purge_stale` trong 1 phút. Nếu `created - purge >> 0` → confirm leak direction.

**Kết quả case này:** `created 530/min, purged 12/min`. Delta +518/min. H3 **confirmed**. Pattern match với commit body `180ab2fd635e` scenario.

## 9.26.6 Mechanism deep-dive — tại sao stale ukey tồn tại

Để hiểu fix, cần đọc kỹ logic revalidator pre-patch `180ab2fd635e`. Trong hàm `revalidator_sweep` ở `ofproto/ofproto-dpif-upcall.c`:

1. **Revalidator lấy list ukey từ umap**, N ukey.
2. **Dump DP flows từ kernel**, M DP flow.
3. **Match từng ukey với DP flow theo UFID**, nếu tìm thấy, set `ukey->dump_seq = current_seq`. Nếu không tìm thấy, ukey vẫn giữ `dump_seq` cũ.
4. **Sweep end: xóa ukey có `dump_seq` quá cũ (> 2 sweep trước)**, nhưng chỉ nếu dump thành công.

**Lỗ hổng:** nếu dump kernel không include một DP flow (vd HW offload flow dump báo `not found` tạm thời), ukey không được refresh `dump_seq`. Nhưng nếu vài sweep sau dump kernel lại include flow đó (HW offload path change state), ukey có thể refresh trở lại trước khi đạt threshold xóa. Kết quả: ukey "zombie", dump không thấy flow thực nhưng ukey không bao giờ đủ stale để xóa.

**Fix commit `180ab2fd635e`:** Thêm counter `missed_dumps` trong struct ukey. Mỗi sweep dump không thấy flow → increment counter. Nếu 4 sweep liên tiếp missed → force xóa ukey, không chờ `dump_seq` threshold.

Code change simplified (trích lược từ commit diff, verified pattern):

```c
/* Before patch */
if (ukey->dump_seq < revalidator->dump_seq - 2) {
    ukey_delete(ukey);
}

/* After patch 180ab2fd635e */
if (ukey->dump_seq < revalidator->dump_seq) {
    ukey->missed_dumps++;
    if (ukey->missed_dumps >= 4) {
        ukey_delete(ukey);
    }
}
```

Fix đơn giản về mặt code (tăng counter + threshold) nhưng **giải quyết gốc class issue**: cache eventually converge trong 4 sweep dù kernel dump không đáng tin cậy 100%.

**Hệ quả:** OVS < v3.3 (nơi commit 180ab2fd635e chưa backport) có rủi ro leak này. OVS 2.17 baseline của curriculum không có fix này trừ khi backport thủ công hoặc upgrade.

## 9.26.7 Remediation 4 tầng — immediate → long-term

**Tầng 1: Immediate (SEV-2 recovery, vài phút):**

```bash
# Force purge mọi stale ukey
[host]# ovs-appctl revalidator/purge

# Verify dump duration trở về < 100ms
[host]# ovs-appctl upcall/show | grep -i duration
```

Lệnh `revalidator/purge` không restart ovs-vswitchd, chỉ clear ukey map. Flow datapath thực không bị ảnh hưởng, packet không drop. An toàn cho production.

**Tầng 2: Short-term (24h, ngăn tái phát ngắn hạn):**

```bash
# Giảm flow-limit để áp lực revalidator thấp hơn
[host]# ovs-vsctl set open_vswitch . other_config:flow-limit=100000

# Verify
[host]# ovs-appctl upcall/show | grep limit
```

Flow-limit nhỏ hơn = ít ukey tối đa = revalidator sweep nhanh hơn. Trade-off: nếu workload cần nhiều flow thực, sẽ có upcall rate cao hơn (slow-path load tăng). Điều chỉnh theo benchmark thực.

**Tầng 3: Medium-term (1-2 tuần, fix gốc):**

Upgrade OVS version có commit `180ab2fd635e`:

- OVS 3.3+: đã merge default.
- OVS 2.17 LTS: verify backport qua distro (Red Hat OVS package, Ubuntu Cloud Archive) hoặc build from source patch.
- Kiểm tra commit `464bc6f9` (Nov 2024) fix bug MemorySanitizer trong patch 180ab2fd635e — cần cả 2 commit để ổn định.

```bash
# Verify version + commit
[host]# ovs-vswitchd --version
# ovs-vswitchd (Open vSwitch) 3.3.0
[host]# ovs-appctl version
```

**Tầng 4: Long-term (1-3 tháng, monitoring + alerting):**

Triển khai Prometheus + Grafana dashboard:

```yaml
# prometheus alert rule
- alert: OVSDumpDurationHigh
  expr: ovs_upcall_dump_duration_ms > 500
  for: 5m
  labels:
    severity: warning
  annotations:
    summary: "OVS revalidator dump duration > 500ms trên {{ $labels.instance }}"
    description: "Có thể là stale ukey leak. Chạy `ovs-appctl upcall/show` + `upcall/dump-ukeys | wc -l`, so sánh với `dp_flows`."

- alert: OVSUkeyRatioHigh
  expr: (sum(ovs_upcall_keys) / ovs_upcall_flows_current) > 5
  for: 10m
  labels:
    severity: critical
  annotations:
    summary: "OVS ukey/flow ratio > 5 (nghi ngờ stale leak)"
```

Dashboard 4 panel:

1. **dp_flows count**, time series `ovs_upcall_flows_current` + `offloaded`.
2. **ukey count per handler**, stacked time series `ovs_upcall_keys{handler="<id>"}`.
3. **dump duration**, line chart `ovs_upcall_dump_duration_ms` với threshold 100/500/1000ms.
4. **revalidator CPU**, `process_cpu_seconds_total{job="ovs-vswitchd"}` rate.

## 9.26.8 So sánh với OVN — logical flow có cùng vulnerability?

OVN compile từ Northbound intent → Logical Flow (SB DB) → OpenFlow rule → DP flow. Cuối cùng vẫn đi qua cùng datapath cache pipeline của OVS. Câu hỏi: OVN workload có rủi ro stale ukey leak tương tự không?

**Trả lời ngắn:** CÓ, nhưng risk thấp hơn vì 2 lý do:

**(1) Cardinality logical flow thấp.** OVN logical flow ưu tiên match theo `reg` field (port_key, dp_key) thay vì 5-tuple. Register space nhỏ (16-bit port_key). Cardinality megaflow tối đa bị bound bởi `(num_lsp × num_stage)` thay vì (num_flow × 2^32 src_ip × 2^16 src_port × ...). Thực tế: cluster 1000 LSP × 50 stage × vài hundreds tenant ≈ vài trăm nghìn unique match pattern. Ít hơn hẳn scanner scenario.

**(2) HW offload pattern ổn định.** OVN logical flow có lifetime dài (gắn với Port_Binding). HW offload flow ít rotation. Giảm trigger H3a.

**Nhưng OVN không miễn nhiễm:**

- Nếu ACL workload churn cao (Kubernetes Network Policy update liên tục), logical flow rebuild → DP flow rotation → ukey leak.
- OVN 24.03+ có Port_Group + Address_Set giảm churn (consolidate policy). Recommend với cluster > 1000 pod.

**Runbook check OVN-specific:**

```bash
[host]# ovn-appctl -t ovn-controller recompute  # force I-P engine full recompute
[host]# ovs-appctl upcall/show  # check ukey count sau recompute
```

Nếu sau recompute ukey count giảm mạnh → ukey leak có thể do OVN stale OpenFlow installed. Nếu không giảm → leak ở OVS layer thuần, áp dụng §9.26.7 tầng 1-4.

**6 điểm cốt lõi cần nhớ:**

1. **Datapath cache ≠ source of truth**, ukey userspace eventually converge với DP flow kernel qua revalidator sweep. Rule: `n(ukey) ≈ n(dp_flow) + n(offloaded)` ± vài entry.
2. **Dump duration là health metric số 1**, > 500ms là warning, > 1000ms là critical, phản ánh CPU load revalidator.
3. **Stale ukey leak là class issue** (không phải bug trên một flow cụ thể, mà vòng lặp create/replace không converge. Commit `180ab2fd635e` thêm `missed_dumps` counter fix gốc.
4. **HW offload path rotation là trigger chính**, TC flower + NIC ConnectX thường gặp; DPDK rte_flow cũng có nhưng mechanism khác).
5. **Remediation 4 tầng**, Immediate `revalidator/purge` → flow-limit tune → upgrade commit `180ab2fd635e` → monitoring. Không skip tầng.
6. **Verify mọi source code claim qua MCP GitHub** (Rule 14): commit SHA 8+ ký tự, function name qua search_code + get_file_contents fallback, file path tại version baseline.

## Guided Exercise 1 — Đọc `upcall/show` output (phương pháp POE)

**Mục đích:** Luyện đọc `upcall/show` output và nhận diện pattern stale ukey leak.

**Chuẩn bị:** Lab Ubuntu 22.04 với OVS 2.17.9 cài đặt, 1 bridge `br0`, 2 namespace `ns1` `ns2` ping qua bridge.

**Bước:**

```bash
[host]# ovs-appctl upcall/show
```

**Predict:** Với 2 namespace ping qua bridge single, output phải có:
- `current` ≈ 2-4 flow (bidirectional ICMP echo + ARP).
- `offloaded flows: 0` (không bật HW offload trong Mininet).
- `dump duration < 50ms` (datapath nhỏ).
- `keys` per handler thread ≈ 2-4.

**Observe:** Chạy lệnh, ghi lại chính xác output.

**Explain:** So sánh output thực với Predict. Nếu khớp → cache healthy. Nếu `keys > 10 × current` → nghi ngờ stale leak (chạy `ovs-appctl revalidator/purge` để test recovery).

**Falsification:** Nếu mở scanner `nmap -sT 10.0.0.0/24` 100.000 port trong 5 phút, kỳ vọng ukey count jump > 10.000 tạm thời nhưng converge về baseline trong 10 phút sau khi scanner dừng. Nếu không converge → H3 leak confirmed.

## Guided Exercise 2 — Đọc `coverage/show` counter

**Mục đích:** Phân loại 12 counter từ `coverage/show` theo nhóm ý nghĩa.

**Chuẩn bị:** Lab như Exercise 1.

**Bước:**

```bash
[host]# ovs-appctl coverage/show | grep -iE "upcall|ukey|dump|revalid|flow_add|flow_del|megaflow" | head -30
```

**Phân loại output vào 4 nhóm:**

| Nhóm | Counter examples | Ý nghĩa |
|------|------------------|---------|
| Creation | `upcall_ukey_created`, `flow_add` | Entry mới thêm vào cache |
| Destruction | `revalidator_purge_stale`, `flow_del` | Entry xóa khỏi cache |
| Conflict | `upcall_ukey_replace`, `handler_duplicate_upcall` | Xung đột giữa handler threads |
| Performance | `dump_flows_fallback`, `megaflow_mask_hit_ratio` | Latency + throughput health |

**POE:** Giả thuyết "counter `upcall_ukey_created` luôn tăng đơn điệu; nếu reset → ovs-vswitchd restart". **Kiểm chứng** bằng `systemctl restart openvswitch-switch` rồi đọc lại counter. Nếu reset về 0 → giả thuyết đúng. Counter khác như `handler_duplicate_upcall` cũng reset.

## Capstone POE — "Reducing flow-limit in 10 OVS 2.11 datacenter will help?"

**Mục đích:** Đánh giá remediation chiến lược với scenario giả định.

**Scenario:** Một datacenter có 10 OVS 2.11 hypervisor, tất cả bị stale ukey leak intermittent. Engineering manager đề xuất: "Giảm `other_config:flow-limit` từ default 200.000 xuống 50.000 trên toàn bộ 10 node, vấn đề sẽ giảm."

**Predict:** Đề xuất có giúp không? Lý do?

**Observe (reasoning):** Flow-limit nhỏ hơn = cache tối đa nhỏ hơn = áp lực revalidator sweep thấp hơn (giả định sweep O(n)). Nhưng:

- OVS 2.11 không có fix `180ab2fd635e` (2024 commit). **Root cause persist** dù flow-limit nhỏ.
- Khi workload vượt 50.000 flow thực (legitimate), `flow_add` fail → packet đi slow-path → VM latency spike nghiêm trọng hơn leak gốc.
- Nếu flow churn cao (HW offload rotation), tạo ukey nhanh hơn tốc độ xóa; giới hạn 50.000 chạm ngưỡng trong vài phút thay vì vài giờ → outage tệ hơn.

**Explain:** Đề xuất **bác bỏ**. Flow-limit là symptom-level control (giới hạn hậu quả), không fix root cause (leak rate). Đúng phải:

1. Immediate: `revalidator/purge` script chạy mỗi 15 phút qua cron (Band-Aid).
2. Medium: Upgrade OVS 2.11 → 2.17 LTS backport hoặc 3.3+ có fix.
3. Long: Monitoring dashboard + alert rule cho ukey/flow ratio.

**Finish:** Rút ra nguyên tắc universal: "Giới hạn tài nguyên không fix leak; phải fix pathology rate". Áp dụng cho mọi cache (memory pool, connection pool, file descriptor pool).

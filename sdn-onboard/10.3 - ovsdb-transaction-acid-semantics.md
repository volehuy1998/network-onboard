## 10.3.1 Bốn tính chất ACID của OVSDB transaction

OVSDB (RFC 7047) là một database giao dịch, không phải key-value store đơn giản. Mỗi lần client gửi một hoặc nhiều operation tới ovsdb-server, cả nhóm được xử lý như **một transaction nguyên tử**. ovsdb-server hiện thực đầy đủ 4 tính chất ACID mặc dù database này có phạm vi hẹp hơn PostgreSQL hay MySQL.

**Atomicity (Nguyên tử), tất cả hoặc không gì cả:** một transaction gồm N operation (insert, update, mutate, delete) thực thi đầy đủ N thao tác, hoặc không thực thi bất kỳ thao tác nào. Nếu operation thứ k thất bại (vi phạm schema constraint, không khớp với `assert`), toàn bộ transaction bị rollback — N-1 operation đã "áp" trong bộ nhớ được hủy. Không có state "nửa chừng" lộ ra ngoài.

Ví dụ: transaction tạo bridge + port + interface liên kết với nhau cần atomicity:
```
[host]# ovs-vsctl add-br br0 -- add-port br0 eth0 -- set Interface eth0 type=internal
```
Ba thao tác này thực thi trong một transaction. Nếu set interface type thất bại (type không hợp lệ), cả add-br + add-port bị rollback, không tạo bridge mồ côi.

**Consistency (Nhất quán), ràng buộc schema luôn giữ:** sau khi transaction commit, state database tuân thủ đầy đủ ràng buộc của schema:
- **Ràng buộc kiểu:** column `integer` không chứa string.
- **Toàn vẹn tham chiếu:** UUID reference không trỏ tới row đã bị xóa. ovsdb-server tự động xóa reference dangling sau khi target row bị xóa (cascade delete tùy chọn).
- **Set/map size bounds:** `set<int>` có giới hạn min/max element được enforce.
- **Ràng buộc duy nhất:** column có `unique=true` không cho phép giá trị trùng.

Nếu một operation trong transaction phá vỡ ràng buộc schema, transaction bị từ chối với lỗi `constraint violation`, toàn bộ transaction rollback.

**Isolation (Cô lập), dựa trên snapshot:** ovsdb-server dùng cơ chế khóa ở mức database, mỗi lần một transaction đang chạy, các transaction khác bị blocked cho đến khi nó hoàn tất. Đây là mô hình **serializable isolation**, mức cao nhất trong hệ thống database, không cho phép concurrent update race condition. Sự đánh đổi: băng thông thực tế thấp hơn so với MVCC (PostgreSQL, MongoDB), nhưng phù hợp với workload OVSDB (tần suất ghi thấp, tần suất đọc cao, ưu tiên đúng đắn hơn tốc độ).

**Durability (Bền vững), fsync sau commit:** khi ovsdb-server trả response "commit successful" cho client, dữ liệu đã được `fsync()` xuống đĩa (file `conf.db`). Ngay cả khi process crash hoặc server reboot ngay sau đó, dữ liệu không bị mất. Chế độ cluster (Raft) bổ sung tính bền vững mức cụm: commit chỉ được xem là successful khi majority của cluster đã persist log entry.

> **Lưu ý triển khai:** ovsdb-server truyền thống dùng **database file dạng append-only** (JSON log) — mỗi transaction là một dòng JSON append vào file. Đọc database = replay tất cả dòng từ đầu. File phình to theo thời gian → cần `ovsdb-tool compact` định kỳ (xem Phần 10.2). Đây là sự đánh đổi: append-only đơn giản, bền vững, nhưng cần bảo trì. Redis hay LMDB dùng mô hình khác (copy-on-write B-tree) không cần compact nhưng phức tạp hơn.

## 10.3.2 Cấu trúc transaction trong protocol RFC 7047

Một OVSDB transaction gửi qua JSON-RPC 2.0 có cấu trúc:
```json
{
  "method": "transact",
  "params": [
    "<database_name>",
    { "op": "insert", "table": "...", "row": {...}, "uuid-name": "..." },
    { "op": "update", "table": "...", "where": [...], "row": {...} },
    { "op": "mutate", "table": "...", "where": [...], "mutations": [...] },
    { "op": "wait", "table": "...", "where": [...], "until": "==", "columns": [...] },
    { "op": "assert", "lock": "..." },
    { "op": "comment", "comment": "description" }
  ],
  "id": 42
}
```

Mỗi element trong `params` (trừ phần tử đầu tiên là database name) là một operation. ovsdb-server xử lý tuần tự từ trên xuống, áp dụng thay đổi state trong bộ nhớ tạm. Sau khi tất cả operation thành công, server commit toàn bộ thay đổi xuống disk (hoặc Raft log trong cluster mode) và trả về response:
```json
{
  "id": 42,
  "error": null,
  "result": [
    { "uuid": ["uuid", "..."], ... },
    { "count": 1 },
    ...
  ]
}
```

Nếu operation thứ k fail, `result[k]` chứa error object và các operation k+1...N không được thực thi. `result[0..k-1]` đã áp tạm nhưng sẽ bị rollback trước khi response gửi về client.

**ovs-vsctl dịch sang transaction:** mỗi lần chạy `ovs-vsctl` command có thể hợp thành một transaction duy nhất dùng `--` separator:
```
[host]# ovs-vsctl --id=@bridge create Bridge name=br0 \
         -- --id=@port create Port name=eth0 \
         -- --id=@iface create Interface name=eth0 type=internal \
         -- set Bridge br0 ports=@port \
         -- set Port eth0 interfaces=@iface \
         -- add Open_vSwitch . bridges @bridge
```
Biến `@bridge`, `@port`, `@iface` là UUID placeholder được phân giải trong transaction. Kết quả: 6 operation chạy nguyên tử.

## 10.3.3 Prerequisites check: wait, assert, nb_cfg

Một vấn đề thực tế: client A đọc database ở thời điểm T1, tính toán thay đổi, gửi transaction ở thời điểm T2. Trong khoảng [T1, T2], client B đã thay đổi database → transaction của A dựa trên trạng thái cũ → có thể áp dụng sai nghĩa (ví dụ chèn trùng, race condition).

OVSDB cung cấp ba cơ chế ngăn chặn race condition này:

**(1) `wait` operation:** transaction chỉ commit nếu một condition đã thỏa mãn tại thời điểm server xử lý:
```json
{ "op": "wait", "table": "Bridge", "timeout": 5000,
  "where": [["name", "==", "br0"]],
  "until": "==",
  "columns": ["datapath_id"],
  "rows": [{"datapath_id": ["uuid", "expected_value"]}] }
```
Nếu condition không match trong 5000ms, transaction fail với error `timed out`. Dùng để đảm bảo "bridge br0 phải đã có datapath_id = X trước khi tôi chạy thao tác tiếp theo". Thường dùng khi client cần đồng bộ với ovsdb-server update (ví dụ plugin chờ kernel datapath hoàn thành attach).

**(2) `assert` operation:** kiểm tra lock vẫn được giữ bởi client hiện tại:
```json
{ "op": "assert", "lock": "ovn_northd" }
```
Nếu client không còn giữ lock `ovn_northd` (do mất kết nối mạng, crash, rồi reconnect), transaction fail. Đây là cách ovn-northd đảm bảo chỉ instance "active" được ghi SB DB (xem Phần 13.8).

**(3) `nb_cfg` prerequisite (OVN-specific):** OVN Northbound có bộ đếm `NB_Global.nb_cfg` tăng mỗi lần có change. Client đọc giá trị, sau đó gửi transaction với prerequisite:
```json
{ "op": "select", "table": "NB_Global", "columns": ["nb_cfg"],
  "where": [] },
/* ... compute changes ... */
{ "op": "wait", "table": "NB_Global",
  "where": [], "until": "==",
  "columns": ["nb_cfg"], "rows": [{"nb_cfg": 42}] },
/* ... apply changes ... */
```
Nếu trong khoảng thời gian đó nb_cfg đã bị client khác tăng (thành 43), wait fail → transaction không commit → client phải retry với trạng thái mới. Cơ chế này giống **optimistic concurrency control** trong SQL database.

## 10.3.4 Xử lý xung đột của mutate

Operation **mutate** là cách in-place thay đổi column mà không cần rewrite toàn bộ giá trị, đặc biệt hữu ích cho `set<UUID>` columns (thêm/xóa element):

```json
{ "op": "mutate", "table": "Bridge",
  "where": [["name", "==", "br0"]],
  "mutations": [
    ["ports", "insert", ["set", [["uuid", "port_uuid_1"]]]],
    ["ports", "insert", ["set", [["uuid", "port_uuid_2"]]]]
  ] }
```

Khác với `update` (thay toàn bộ giá trị), mutate **thêm vào tập hiện có** mà không cần biết giá trị trước. Điều này giảm race condition:

**Ví dụ race condition khi dùng update:**
- T1: Client A đọc `br0.ports = [port_a]`.
- T2: Client B đọc `br0.ports = [port_a]`.
- T3: Client A transact: `update br0.ports = [port_a, port_new_a]`.
- T4: Client B transact: `update br0.ports = [port_a, port_new_b]`.
- Kết quả: update của A bị ghi đè bởi update của B: `port_new_a` bị mất!

**Khi dùng mutate:**
- T1: Client A transact: `mutate br0.ports insert port_new_a`. Server áp: ports = [port_a, port_new_a].
- T2: Client B transact: `mutate br0.ports insert port_new_b`. Server áp: ports = [port_a, port_new_a, port_new_b].
- Kết quả: cả port_new_a và port_new_b đều có trong trạng thái cuối.

Đây là lý do **ovs-vsctl luôn dùng mutate** cho `set<UUID>` columns, không dùng update.

**Mutate operators:**
- `insert`: thêm element vào set.
- `delete`: xóa element khỏi set.
- `+=`, `-=`, `*=`, `/=`, `%=`, số học cho integer column.

Một mutate có thể chứa nhiều mutation, thực thi tuần tự trong cùng operation (nguyên tử).

## 10.3.5 Transaction failure và cơ chế retry của client

Transaction có thể fail vì nhiều lý do: vi phạm schema, wait timeout, lock lost (assert fail), mất kết nối. ovsdb-server trả về error code cụ thể cho từng field hợp:

| Mã lỗi | Ý nghĩa | Action của client |
|---------|---------|---------------------|
| `constraint violation` | Ràng buộc schema không thỏa | Sửa input rồi retry |
| `referential integrity violation` | UUID reference tới row không tồn tại | Retry với UUID hợp lệ |
| `timed out` | Wait operation timeout | Retry với trạng thái mới |
| `not owner` | Assert lock fail — client không còn là leader | Abort, giao cho leader mới |
| `duplicate uuid-name` | Uuid-name trùng trong cùng transaction | Sửa logic tạo uuid-name |
| `syntax error` | JSON-RPC sai format | Sửa bug client |

**Pattern retry chuẩn cho IDL client:**
```python
while True:
    txn = idl.transaction(OpenVSwitch)
    # ... build transaction operations ...
    status = txn.commit_block()
    if status == TransactionStatus.SUCCESS:
        break
    elif status == TransactionStatus.TRY_AGAIN:
        # Trạng thái đã cũ, idl tự động lấy lại và retry
        continue
    else:
        raise Exception(f"Transaction failed: {txn.get_error()}")
```

Thư viện IDL (Interface Definition Language) client (python-ovs, ovs-cpp) tự động xử lý retry cho `TRY_AGAIN`, khi server signal "trạng thái của bạn đã cũ", IDL lấy lại snapshot mới và retry transaction. Đây là trừu tượng hóa cho optimistic concurrency control.

## 10.3.6 Hạn chế so với SQL database transaction

OVSDB transaction model có một số giới hạn rõ rệt so với SQL database đầy đủ:

**Không có transaction lồng:** OVSDB không hỗ trợ `BEGIN ... BEGIN ... COMMIT ... COMMIT` như PostgreSQL. Một transaction là "phẳng", không thể tạo sub-transaction độc lập với parent. Lý do: mô hình đơn giản, ovsdb-server không phải quản lý nested state.

**Không có savepoint:** SQL cho phép `SAVEPOINT s1; ... ROLLBACK TO s1` để rollback một phần transaction. OVSDB chỉ có `abort` toàn bộ. Nếu cần rollback một phần, client phải bắt error và gửi lại transaction với operation khác nhau.

**Không có isolation level tùy chỉnh:** OVSDB luôn serializable. SQL cho chọn READ COMMITTED, REPEATABLE READ, SERIALIZABLE. Không cấu hình được trong OVSDB.

**Không có ngôn ngữ truy vấn phức tạp:** OVSDB chỉ có `select` với condition đơn giản (column op value). Không có JOIN, GROUP BY, subquery. Điều này OK vì dữ liệu OVSDB nhỏ (hàng nghìn-chục nghìn row), client có thể tải toàn bộ rồi xử lý trong memory.

**Không có trigger, stored procedure:** không có code chạy trong server để thực thi business logic. Toàn bộ logic nằm ở client (ovn-northd, neutron plugin).

**Sự đánh đổi của thiết kế đơn giản:** OVSDB đổi lấy tính dễ triển khai, server C++ khoảng 50K LOC (so với PostgreSQL khoảng 1.4M LOC), dấu chân tài nguyên thấp (50-200 MB RAM cho cụm hàng trăm nghìn row), khởi động nhanh (dưới giây). Phù hợp với phạm vi hẹp của OVS/OVN quản lý trạng thái.

### ▶ Guided Exercise 1: Quan sát atomicity qua transaction multi-operation fail

> **Trạng thái verification lab:** Output dựa trên OVS 2.17 docs; chờ kiểm chứng lab thực.

**Mục tiêu:** Chứng minh khi một operation trong transaction thất bại, các operation trước đó bị rollback, không có trạng thái "nửa chừng" lộ ra.

**Chuẩn bị:** Node Ubuntu 22.04 với OVS 2.17 đã chạy.

**Bước thực hiện:**

**1.** Kiểm tra trạng thái ban đầu:
```
[host]# ovs-vsctl show
```
Ghi lại số bridge hiện có.

**2.** Thử một transaction cố ý sai (tạo bridge + set interface type không hợp lệ):
```
[host]# ovs-vsctl add-br br_test_fail -- add-port br_test_fail eth_bad \
         -- set Interface eth_bad type=nonexistent_type
```

Kết quả mong đợi:
```
ovs-vsctl: Interface "eth_bad": "nonexistent_type" is not a valid interface type
```

**3.** Kiểm tra lại trạng thái, bridge `br_test_fail` **không được tạo**:
```
[host]# ovs-vsctl show
```
Bridge count không tăng. Đây là atomicity, cả transaction fail, kể cả operation add-br đứng trước đã succeed tạm trong memory.

**4.** Thử transaction hợp lệ tương tự để so sánh:
```
[host]# ovs-vsctl add-br br_test_ok -- add-port br_test_ok eth_internal \
         -- set Interface eth_internal type=internal
[host]# ovs-vsctl show
```
Bridge `br_test_ok` được tạo thành công.

**Bài học học được:**

Thứ nhất, atomicity OVSDB thực thi ở mức toàn bộ transaction, không có "partial commit" khi một operation fail. Đây là tính chất cốt lõi giúp ovn-northd, neutron plugin, kubernetes CNI tin cậy được trạng thái trong OVSDB.

Thứ hai, ovs-vsctl tự động gom nhiều command thành một transaction khi dùng `--` separator. Nếu chạy lần lượt `ovs-vsctl add-br` rồi `ovs-vsctl add-port` rồi `ovs-vsctl set`, đó là 3 transaction riêng, mỗi cái nguyên tử riêng, nhưng nếu command 3 fail thì command 1-2 đã commit.

**Cleanup:**
```
[host]# ovs-vsctl del-br br_test_ok
```

### ▶ Guided Exercise 2: Mutate so với update — chứng minh tránh race

> **Trạng thái verification lab:** chờ kiểm chứng.

**Mục tiêu:** Quan sát tại sao `mutate insert` tránh được race condition mà `update` không tránh được.

**Chuẩn bị:** Lab có OVS chạy, ít nhất 1 bridge `br0` có sẵn.

**Bước thực hiện:**

**1.** Mở 2 terminal tới cùng host OVS.

**2.** Terminal A, chạy loop add port:
```
[host-A]# for i in {1..5}; do
[host-A]>   ovs-vsctl add-port br0 port_a$i -- set Interface port_a$i type=internal
[host-A]> done
```

**3.** Terminal B, cùng thời điểm:
```
[host-B]# for i in {1..5}; do
[host-B]>   ovs-vsctl add-port br0 port_b$i -- set Interface port_b$i type=internal
[host-B]> done
```

**4.** Sau khi cả 2 terminal xong, kiểm tra br0 có đủ 10 port mới:
```
[host]# ovs-vsctl list-ports br0
```

Kỳ vọng: thấy tất cả port_a1..port_a5 và port_b1..port_b5. OVS dùng mutate bên trong → không có race.

**5.** So sánh với update (không nên dùng trong môi trường production). Nếu dùng trực tiếp `ovsdb-client transact` với op `update Bridge ports = [...]` thay vì mutate → có thể xảy ra race khi 2 client update cùng lúc.

**Bài học học được:**

Thứ nhất, `mutate insert` vào set là nguyên tử từ góc nhìn client, không cần "read then write", chỉ cần gửi "add element X". Server tự xử lý concurrent mutate từ nhiều client.

Thứ hai, hiểu bên trong của ovs-vsctl giúp gỡ lỗi hiếm, nếu thấy port biến mất sau concurrent update, nghi ngờ có tool khác dùng `update` thay vì `mutate`.

**Cleanup:**
```
[host]# for i in {1..5}; do ovs-vsctl del-port br0 port_a$i; ovs-vsctl del-port br0 port_b$i; done
```

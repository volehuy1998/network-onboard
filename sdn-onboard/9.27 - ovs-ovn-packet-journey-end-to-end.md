## 9.27.2 The three-tier parallel diagnostic framework

Debugging cross-host OVN means debugging **three tiers in parallel**. Each tier has its own object of truth, its own tool, and its own question.

### Tier 1, logical intent (OVN)

**Object of truth:** the OVN Northbound and Southbound tables (`ovn-nbctl show`, `ovn-sbctl list Logical_Flow`).

**Tool:** `ovn-trace --detailed "..."` (offline simulation through the logical pipeline).

**The question this tier answers:** "Where does OVN want the packet to go? Is the intent correct?"

**When to check:** always first. If the intent is wrong, then the translation, even if correct, is meaningless.

```bash
ovn-trace --detailed "inport == \"lsp-vm1\" && eth.src == fa:16:3e:00:00:01 && eth.dst == fa:16:3e:00:00:02 && ip4.src == 10.0.0.10 && ip4.dst == 10.0.0.20"
```

### Tier 2, OpenFlow translation (`br-int`)

**Object of truth:** the OpenFlow flow table on `br-int` (installed by `ovn-controller` from the SBDB `Logical_Flow`).

**Tool:** `ovs-appctl ofproto/trace br-int "..."` (simulation through the OpenFlow pipeline, including a recirculation through `ct()` if `--ct-next` is passed).

**The question this tier answers:** "Has `ovn-controller` translated the intent into OpenFlow correctly? Is there sync lag?"

**When to check:** after Tier 1 is OK. A failure at Tier 2 means an `ovn-controller` issue (crash, SSL, SB sync).

```bash
TAP1_OF=$(sudo ovs-vsctl get interface tap1 ofport)
sudo ovs-appctl ofproto/trace br-int \
  "in_port=${TAP1_OF},ip,nw_src=10.0.0.10,nw_dst=10.0.0.20,dl_src=fa:16:3e:00:00:01,dl_dst=fa:16:3e:00:00:02"
```

### Tier 3, datapath runtime (kernel megaflow cache)

**Object of truth:** the megaflow cache in the kernel `openvswitch` module (or in userspace `dpif-netdev` for DPDK).

**Tool:** `ovs-appctl dpif/dump-flows br-int` (the cached entries) plus `ovs-appctl dpif/show` (overall health).

**The question this tier answers:** "Has the kernel cached the path? Is it the fast path or the slow path? Is there a drop at the datapath?"

**When to check:** after Tier 1 and Tier 2 are OK. If the packet still drops, the datapath cache is stale or the kernel module has an issue.

```bash
sudo ovs-appctl dpif/dump-flows br-int | grep "in_port=${TAP1_OF}"
sudo ovs-appctl dpif/show
```

### Decision matrix, when to check which tier

```
Packet drop observed cross-host
|
+-- Tier 1, ovn-trace OK?
|   +-- NO (logical drop): check OVN NB config: lsp-add, lsp-set-addresses,
|   |                      ACL, LB, port-security. Fix logical intent.
|   +-- YES: go to Tier 2.
|
+-- Tier 2, ofproto/trace br-int OK?
|   +-- NO (OpenFlow drop): check ovn-controller:
|   |                       systemctl status ovn-controller
|   |                       journalctl -u ovn-controller --since "10 min ago"
|   |                       ovn-sbctl show (SB sync OK?)
|   |                       Restart ovn-controller if needed.
|   +-- YES: go to Tier 3.
|
+-- Tier 3, dpif/dump-flows has an entry and packets are increasing?
    +-- NO (zero packets): the datapath has not cached, the upcall is failing:
    |                      ovs-vswitchd log plus dpif/show miss rate.
    |                      Is the kernel module loaded? (lsmod | grep openvswitch)
    +-- YES, packets are increasing: OVS is OK, the drop is outside br-int:
    |                                check the tunnel (Stage 3 encap).
    |                                check the underlay (Stage 4).
    |                                check the remote chassis (Stage 5 and 6).
    +-- YES, packets exist BUT the counter does not increase: stale datapath entry:
         ovs-appctl dpctl/del-flows br-int "<match>"
         Force a re-upcall to refresh the cache.
```

A skilled engineer runs the three tools in parallel in three terminal tabs and compares their output in a few seconds. Memorising the exact commands is not required. Memorise the three-tier framework and the decision tree.

## 9.27.3 Geneve TLV deep dive, reading the outer header from a packet capture

OVN uses Geneve (RFC 8926) as the default tunnel. A debugging engineer needs to read the outer Geneve header directly from `tcpdump` or Wireshark.

### Packet format

```
+---------+----------------------------+
| 14 byte | Outer Ethernet             |
|         |   src=<chassis-A MAC>      |
|         |   dst=<chassis-B MAC or    |
|         |        next-hop router>    |
|         |   ethertype=0x0800 (IPv4)  |
+---------+----------------------------+
| 20 byte | Outer IPv4                 |
|         |   src=192.168.10.10        |
|         |   dst=192.168.10.20        |
|         |   proto=17 (UDP)           |
|         |   DF flag set (do not      |
|         |   fragment, OVS standard)  |
+---------+----------------------------+
|  8 byte | Outer UDP                  |
|         |   src=<random, 1024-65535> |
|         |   dst=6081 (Geneve)        |
|         |   length=<Geneve+inner>    |
|         |   checksum=0 by default    |
+---------+----------------------------+
|  8 byte | Geneve base header         |
|         |   Ver=0 (current)          |
|         |   OptLen=N (4-byte words)  |
|         |   O flag: control packet   |
|         |   C flag: critical opts    |
|         |   Reserved 6 bits          |
|         |   Protocol Type=0x6558     |
|         |   VNI=24-bit logical DP ID |
|         |   Reserved 8 bits          |
+---------+----------------------------+
|  8 byte | Geneve TLV option 1 (OVN)  |
|         |   Class=0x0102 (OVN)       |
|         |   Type=0x80 (critical)     |
|         |   R=0, Length=1 (4 bytes)  |
|         |   Value: logical ingress   |
|         |          port (32-bit)     |
+---------+----------------------------+
|  8 byte | Geneve TLV option 2 (OVN)  |
|         |   Class=0x0102             |
|         |   Type=0x81                |
|         |   R=0, Length=1            |
|         |   Value: logical egress    |
|         |          port (32-bit)     |
+---------+----------------------------+
|    ...  | Inner Ethernet (14 byte)   |
|         |   src=<VM1 MAC>            |
|         |   dst=<VM2 MAC>            |
+---------+----------------------------+
|    ...  | Inner IPv4 plus payload    |
|         |   (VM traffic)             |
+---------+----------------------------+
```

Total overhead: 14 + 20 + 8 + 8 + 8 multiplied by 2 TLVs equals **66 bytes** (the OVN default with two TLVs).

### Capture and dissect

```bash
# Chassis A, capture outbound Geneve
sudo tcpdump -ni eth0 -w /tmp/geneve-a.pcap "udp and port 6081" -c 20

# Dissect with tshark
tshark -r /tmp/geneve-a.pcap -V -Y "geneve" | head -60

# Output (plausible, abbreviated):
# Frame 1: 168 bytes on wire
# Ethernet II, Src: ..., Dst: ...
# Internet Protocol Version 4
#     Src: 192.168.10.10
#     Dst: 192.168.10.20
# User Datagram Protocol
#     Source Port: 53421
#     Destination Port: 6081
# Generic Network Virtualization Encapsulation
#     Flags: 0x00
#     Protocol Type: Ethernet (0x6558)
#     VNI: 0x000001
#     Options:
#         Option: Class 0x0102 Type 0x80, OVN ingress port
#             Length: 4
#             Value: 0x00000001
#         Option: Class 0x0102 Type 0x81, OVN egress port
#             Length: 4
#             Value: 0x00000002
# Ethernet II (inner)
#     Src: fa:16:3e:00:00:01
#     Dst: fa:16:3e:00:00:02
# Internet Protocol Version 4 (inner)
#     Src: 10.0.0.10
#     Dst: 10.0.0.20
# ICMP Echo Request
```

### OVN TLV semantics

| Class | Type | Meaning | Encoded by |
|-------|------|---------|------------|
| `0x0102` | `0x80` | Logical ingress port (metadata `reg14`) | `ovn-controller` on chassis A |
| `0x0102` | `0x81` | Logical egress port (metadata `reg15`) | `ovn-controller` on chassis A |
| `0x0102` | other | Reserved for future OVN use | (none) |

Chassis B decapsulates the Geneve packet, extracts the TLVs, and injects `reg14`, `reg15`, and the metadata into the recirculation pipeline. There is no need to look up the `Port_Binding` table on the receive side.

### Diagnostic commands

```bash
# List the tunnel ports on the local chassis
sudo ovs-vsctl find Interface type=geneve
# _uuid, name=genev_sys_6081, options={csum="true", key=flow, remote_ip=flow}

# Tunnel ports detail
sudo ovs-appctl tnl/ports/show
# Listening to 0.0.0.0:6081 for Geneve

# BFD state for the tunnel
sudo ovs-appctl bfd/show
# chassis-b UP (remote 192.168.10.20)

# Geneve statistics
sudo ovs-appctl dpctl/dump-flows type=ovs | grep "tun_dst=192.168.10.20" | head -3
```

## 9.27.4 MTU forensics, getting the math right

MTU mismatch between the underlay and the overlay is the **biggest source** of cross-host OVN bugs. Typical symptoms:

- Small ping (less than 1400 bytes) succeeds.
- Large ping (more than 1400 bytes) fails or times out.
- A small TCP request works, but a large file download stalls.
- The SSL handshake works, but the file transfer hangs.

### Correct math

```
Underlay MTU (link layer) = 1500 bytes (Ethernet default)
Overlay MTU (what the VM sees) = ?

Geneve overhead:
  Outer Ethernet     = 14 bytes
  Outer IPv4         = 20 bytes
  Outer UDP          =  8 bytes
  Geneve base        =  8 bytes
  Geneve 2 TLV (OVN) = 16 bytes  (2 multiplied by 8)
  -------------------------
  Total overhead     = 66 bytes

Max overlay MTU = Underlay MTU minus overhead
                = 1500 - 66
                = 1434 bytes
```

### MTU options in production

| Underlay MTU | Max overlay MTU | Note |
|--------------|-----------------|------|
| 1500 (default) | 1434 | Set the VM MTU to 1434. TCP MSS clamping: 1434 minus 40 (IP plus TCP) equals 1394. |
| 1500 (default) | 1500 | **WRONG**: full-size packets are fragmented or dropped. |
| 9000 (jumbo) | 1500 | The overlay keeps 1500 (standard), the underlay is jumbo, no drama. |
| 9000 (jumbo) | 8934 | End-to-end jumbo if the VM application supports it. |

### Calculating MSS clamping correctly

TCP MSS equals the MTU minus 40 (20 for the IP header plus 20 for the TCP header, excluding options).

If the underlay is 1500 and the overlay is mistakenly set to 1500, the client advertises an MSS of 1460, the server also advertises 1460, and the negotiated MSS is 1460. The TCP packet has a payload of 1460 bytes, plus 20 IP, plus 20 TCP, equals 1500 inner. After Geneve encapsulation it becomes 1566 outer. With DF set, it is dropped at an underlay router (when the underlay MTU is 1500).

Fix:
- Option A: VM MTU 1434, MSS clamp 1394.
- Option B: underlay MTU 9000 (jumbo).
- Option C: a router sends ICMP "Fragmentation Needed" so the client discovers the path MTU. ICMP is often dropped by firewalls, leading to a PMTUD black hole.

### MTU probe recipes

```bash
# From chassis A to chassis B (underlay test)
ping -M do -c 3 -s 1472 192.168.10.20   # 1472 + 28 = 1500 outer
ping -M do -c 3 -s 1500 192.168.10.20   # 1500 + 28 = 1528; fails if underlay MTU < 1528

# From VM1 to VM2 (overlay test)
ip netns exec vm1-ns ping -M do -c 3 -s 1406 10.0.0.20   # 1406 + 28 = 1434 overlay MTU
ip netns exec vm1-ns ping -M do -c 3 -s 1472 10.0.0.20   # 1472 + 28 = 1500; FAILS if the overlay MTU is wrongly set to 1500

# Inspect TCP MSS from a packet capture
sudo tcpdump -ni tap1 "tcp[tcpflags] & tcp-syn != 0" -v | grep "mss"
# If MSS is 1460 but the actual overlay is 1434, MSS clamping is not configured.
```

### MSS clamping with iptables or nftables

```bash
# Option A: clamp MSS on the host on the tap interface
sudo iptables -t mangle -A FORWARD -o tap1 -p tcp --tcp-flags SYN,RST SYN -j TCPMSS --clamp-mss-to-pmtu

# Option B: hard-code MSS when the MTU is fixed
sudo iptables -t mangle -A FORWARD -o tap1 -p tcp --tcp-flags SYN,RST SYN -j TCPMSS --set-mss 1394
```

## 9.27.5 Fault pattern catalog, common cross-host failures

A reference table of 10 common fault patterns with symptom, root cause, and a two-line fix.

| # | Symptom | Root cause | Diagnostic | Fix |
|---|---------|------------|------------|-----|
| 1 | `ping small OK, big fail` | MTU mismatch | `ping -M do -s N` probe | Set the VM MTU to underlay minus 66, or use a jumbo underlay |
| 2 | `ovn-trace OK, ofproto/trace fail` | `ovn-controller` crash or sync lag | `systemctl status ovn-controller`, `journalctl` | Restart `ovn-controller`. Check the SB SSL certificate. |
| 3 | `ofproto/trace OK, dpif miss` | Stale datapath entry | High miss rate in `dpif/show` | `ovs-appctl dpctl/del-flows` to force a refresh |
| 4 | `bfd/show FLAPPING` | Underlay packet loss greater than 5 percent | `ping -c 100 underlay-IP` | Fix underlay QoS or NIC tuning |
| 5 | `Tunnel UP, no traffic` | Firewall drops UDP/6081 | `iptables -L -n \| grep 6081` | Allow UDP/6081 between chassis |
| 6 | `Traffic works one-way only` | Asymmetric underlay routing | `traceroute` in both directions | Make underlay routing symmetric |
| 7 | `New VM has no network after boot` | `ovn-controller` has not bound the port | `ovn-sbctl list port_binding \| grep chassis: \[\]` | Check `ovs-vsctl get interface tapN external-ids` |
| 8 | `ACL deny does not work` | Conntrack zone collision | `conntrack -L \| head`, look at `ct_zone` in the trace | Use a distinct zone for each LS |
| 9 | `Cross-tenant leak` | Tunnel metadata (VNI) is not isolated | Capture Geneve, check the TLV values | Verify the `datapath_id` mapping with `ovn-sbctl dump-flows` |
| 10 | `Throughput at 10 percent of expected` | Slow-path upcall per packet | `dpif/show` coverage, userspace CPU | Check the `ovn-controller` log, megaflow installation is failing |

4. **Bug D:** stop `ovn-controller` on chassis B.

```bash
# Read the output: does the intent reach the egress lsp-vm2?
# If NO, this is Bug E, ACL deny. Fix: ovn-nbctl acl-del ...

```

```bash
# If YES, continue to Tier 3 locally.
```

```bash
sudo ovs-appctl dpif/dump-flows br-int | grep "in_port=${TAP1_OF}"

# If zero, the upcall is failing. Check the ovs-vswitchd log.
# If increasing, chassis A is OK and the drop is outside OVS. Check the underlay.
```

```bash
# Underlay check (Stage 4)
ping -M do -c 3 -s 1472 192.168.10.20
# If it fails, this is Bug B (iptables) or Bug C (MTU).

# MTU check: ip link show genev_sys_6081 | grep mtu
```

```bash
# Remote chassis verification (when the underlay is OK)
ssh chassis-b 'sudo ovs-appctl dpif/dump-flows br-int | grep tun_src=192.168.10.10'
# If there is no entry, this is Bug A (port binding) or Bug D (ovn-controller B is down).
# ssh chassis-b 'ovs-vsctl get interface tap2 external-ids'
# ssh chassis-b 'systemctl status ovn-controller'
```

### Step 4, a counter-hypothesis: "guess before the three-tier check"

Counter-hypothesis:

> *"I am experienced, I can guess the bug right away and fix it without following a framework."*

## Guided Exercise 2, parse Geneve TLV from a real packet capture

**Goal:** read the outer Geneve header and TLV options from a pcap file and confirm that OVN encoded the logical ports correctly. This skill is essential when debugging inter-chassis mismatches (VNI collision, TLV corruption, version mismatch).

**Claim to verify:**

> *"`tcpdump -w` plus Wireshark or `tshark` allows complete dissection of the Geneve TLVs. The engineer can read the logical port value from the outer header without accessing `ovn-sbctl`. This is useful when debugging from a network tap point between two chassis."*

### Preparation

A two-chassis active OVN lab. Set up light traffic between VM1 and VM2 (continuous ping):

```bash

PING_PID=$!
```

### Step 1, capture 10 outbound Geneve packets from chassis A

```bash

# Wait for the capture to finish, then kill the ping
kill $PING_PID
```

### Step 2, dissect with tshark

```bash
tshark -r /tmp/geneve-capture.pcap -V -Y "geneve" | head -100
```

Result (plausible, abbreviated):
```
Frame 1: 168 bytes
Ethernet II, Src: ..., Dst: ...
Internet Protocol Version 4
    Src: 192.168.10.10, Dst: 192.168.10.20
User Datagram Protocol
    Src Port: 53421, Dst Port: 6081
Generic Network Virtualization Encapsulation
    Flags: 0x00
    Protocol Type: Transparent Ethernet Bridging (0x6558)
    Virtual Network Identifier (VNI): 0x000001
    Options:
        Option Class: Unknown (0x0102)
            Type: 0x80 (critical)
            Length: 1 (4 bytes value)
            Value: 00 00 00 01
        Option Class: Unknown (0x0102)
            Type: 0x81
            Length: 1
            Value: 00 00 00 02

    Src: fa:16:3e:00:00:01 (VM1)
    Dst: fa:16:3e:00:00:02 (VM2)

    Src: 10.0.0.10, Dst: 10.0.0.20
Internet Control Message Protocol
    Type: 8 (Echo request)
```

### Step 3, cross-reference with the OVN Southbound

```bash
# Get the datapath UUID for ls0
ovn-sbctl list datapath_binding | grep -B2 "ls0"
# Note tunnel_key, the VNI of the logical datapath

# Get the port key
ovn-sbctl list port_binding lsp-vm1 | grep tunnel_key
# tunnel_key = 1 (matches the TLV 0x80 value)

ovn-sbctl list port_binding lsp-vm2 | grep tunnel_key
# tunnel_key = 2 (matches the TLV 0x81 value)
```

The TLV values match `port_binding.tunnel_key`. OVN encoded them correctly.

### Step 4, a counter-hypothesis: "the TLV does not matter, only the VNI matters"

Counter-hypothesis:

> *"The 24-bit VNI is enough to distinguish tenants. The TLVs are only cosmetic metadata."*

Counter-evidence: the VNI identifies the logical **datapath** (a logical switch or router). The TLVs `0x80` and `0x81` identify the logical **ports** within that datapath. Without the TLVs, when chassis B decapsulates it knows only "the packet arrived at ls0", not "from lsp-vm1 to lsp-vm2", and it has to look up the inner Ethernet header (the MAC table). That costs one extra lookup and creates a MAC table race condition during VM migration.

Experiment: try to disable the TLVs in OVN. There is no sequential disable mechanism, this is a misconception test. OVN always encodes the TLVs and they cannot be skipped. Lesson: the TLVs are essential, not optional.

### Lesson

The Geneve TLVs are not decoration. Class `0x0102` types `0x80` and `0x81` carry core data path metadata. OVN uses them to inject `reg14` and `reg15` into the pipeline on the remote chassis, avoiding the cost of a MAC table lookup and resolving the race condition during VM migration. The engineer needs to read the TLVs to debug inter-chassis issues: version mismatch (an old OVN and a newer one use different TLV types), VNI collision (two logical switches share the same `tunnel_key`), or corrupted TLVs (the physical network alters the Geneve packet).

## Capstone POE, stage-by-stage performance benchmark

**Goal:** measure the actual throughput of each stage to find where the real bottleneck is. Compare the claim "the OVS datapath is slow" against "the underlay is slow". Predict first, observe second, and explain the mismatch.

### Predict (before the lab)

Write the prediction on paper (without looking at the answer). Order the five stages from slowest to fastest in actual throughput:

1. Local same-host VM1 to VM3 (both on chassis A): actual throughput approximately ___ Gbps.
2. Local through OVS userspace (kernel datapath disabled): actual throughput approximately ___ Gbps.
3. Cross-host VM1 to VM2 with kernel-mode Geneve: actual throughput approximately ___ Gbps.
4. Raw underlay `iperf3` directly between chassis (no OVS): actual throughput approximately ___ Gbps.
5. Cross-host VM1 to VM2 with DPDK (when the lab supports it): actual throughput approximately ___ Gbps.

### Observe

```bash
# Test 1: same-host VM1 to VM3 (create VM3 on chassis A in the same subnet)
ip netns exec vm3-ns iperf3 -s &
ip netns exec vm1-ns iperf3 -c 10.0.0.30 -t 30 -J > test1.json

# Test 2: (optional, requires OVS userspace mode)
# usually applies only if the lab supports DPDK; skip otherwise.

# Test 3: cross-host kernel-mode Geneve, the default
ip netns exec vm1-ns iperf3 -c 10.0.0.20 -t 30 -J > test3.json

# Test 4: raw underlay
iperf3 -s -B 192.168.10.20 &   # on chassis B
iperf3 -c 192.168.10.20 -t 30 -B 192.168.10.10 -J > test4.json   # on chassis A

# Test 5: DPDK (skip if the lab uses kernel datapath only)
```

### Explain

Typical results on a lab with a 10 Gbps NIC:

| Test | Expected Gbps | Dominant cost |
|------|---------------|----------------|
| 1 Same-host | 8 to 9.5 | veth and br-int local pipeline, fast path |
| 2 OVS userspace | 3 to 4 | Context switch between kernel and userspace per packet |
| 3 Cross-host kernel-mode Geneve | 6 to 8 | Encapsulation and decapsulation CPU plus underlay PPS |
| 4 Raw underlay | 9 to 9.8 | NIC line rate, no encapsulation |
| 5 DPDK | 9 to 9.8 | Bypass kernel, poll-mode driver |

Key insight: **cross-host Geneve (Test 3) is only 15 to 25 percent slower than the raw underlay (Test 4)** once the fast path is active. The encapsulation, decapsulation, and tunnel-metadata handling overhead totals only about 100 to 200 nanoseconds per packet on a modern kernel datapath.

The **real bottleneck** in production cross-host scenarios is usually not the OVS datapath. It is usually:
- NIC throughput (Stage 1 and 7): consider upgrading from 10 Gbps to 25 or 100 Gbps.
- Underlay packet loss (Stage 4): 1 percent loss collapses TCP throughput by 50 percent.
- CPU saturation: `ovs-vswitchd` is single-threaded for the slow path. A high megaflow miss rate matters.
- Conntrack table size: per-zone state requires sufficient memory.

### Production takeaway

Tuning priorities (cross-host OVN):

1. **NIC and underlay**, the biggest bang for the buck. Going from a 10 Gbps to a 25 Gbps NIC plus jumbo frames yields 2.5x.
2. **TX and RX offload**, enable checksum, TSO, and GRO on the underlay NIC plus Geneve offload (kernel 5.0 and later).
3. **CPU pinning**, pin `ovs-vswitchd` and PMD threads to the NUMA node of the NIC.
4. **DPDK and AF_XDP**, bypass the kernel when more than 20 Gbps on a single flow is needed (Part 16.0 to 16.2).
5. **OVS datapath tuning**, `max-idle`, `n-revalidator-threads` (Part 9.26 forensics).

DPDK and AF_XDP do not fix the underlay or the actual NIC throughput. They only help the OVS datapath scale across more cores.

## 9.27.6 Key takeaways

**One.** The three-tier diagnostic framework is the spine of cross-host debugging: `ovn-trace` (logical intent, OVN NB and SB), `ovs-appctl ofproto/trace br-int` (OpenFlow translation, the output of `ovn-controller`), and `ovs-appctl dpif/dump-flows br-int` (datapath runtime, the kernel megaflow cache). Do not confuse the three tiers.

**Two.** Decision tree: logical OK, then check OpenFlow. OpenFlow OK, then check the datapath. The datapath is OK but the packet is still dropped, so the drop is outside `br-int` (tunnel, underlay, remote chassis). Check sequentially, do not skip a tier.

**Three.** Default OVN Geneve outer overhead equals 66 bytes (14 + 20 + 8 + 8 base + 16 TLV). With an underlay MTU of 1500, the maximum overlay MTU is 1434. A jumbo underlay of 9000 eliminates MTU drama.

**Four.** OVN Geneve TLVs class `0x0102` types `0x80` and `0x81` carry the logical ingress and egress port (the `tunnel_key` of the `port_binding`). They are not cosmetic, they are essential for the remote chassis to inject `reg14` and `reg15` into the recirculation pipeline.

**Five.** MSS clamping is mandatory if the overlay MTU differs from the default 1500. Use `iptables -t mangle ... --clamp-mss-to-pmtu` on the host's tap interface. Without clamping, TCP stalls and a PMTUD black hole forms when ICMP is dropped between chassis.

**Six.** Common fault patterns have a clear signature: MTU mismatch (small ping OK, big ping fails), `ovn-controller` sync lag (`ofproto/trace` fails), stale datapath (the packet counter does not increase), firewall drops UDP 6081 (BFD is UP but traffic fails). The 10-pattern catalog in §9.27.5 covers 80 percent of production incidents.

**Seven.** Cross-host OVN performance bottlenecks are usually not in the OVS datapath. They are usually in the NIC, the underlay, or the CPU. DPDK and AF_XDP speed up the OVS datapath by 3x to 5x but do not fix the underlay. Tuning priority: NIC and underlay first, OVS datapath second.

## 9.27.7 Anatomy Template A, `tnl/ports/show` plus `bfd/show` for cross-host tunnel entry (v3.2 P4 standardised)

**Context for running this:** the first-line debug step for cross-host packet journey, verifying the tunnel endpoint setup and BFD liveness before going deeper into `ofproto/trace` or `dpif/dump-flows`. This is the entry-point command for Tier 1 in the three-tier framework (§9.27.2).

**Source commands:**

```bash
ovs-appctl tnl/ports/show           # list tunnel port, peer, and port number
ovs-appctl bfd/show                 # BFD session state per tunnel
ovs-vsctl list Interface <tunnel>   # options:remote_ip and options:key
```

**Nine-attribute anatomy:**

| Attribute | Healthy | Warning | Critical | Diagnostic hint |
|-----------|---------|---------|----------|------------------|
| `Listening ports` | Lists `geneve_sys_6081` or `vxlan_sys_4789` | Port other than 6081 or 4789 | No port equals tunnel not attached | Check `br-int` and the Geneve port binding |
| `Peer <remote_ip>` | Matches the chassis IP in the SBDB Encap | An IP different from any known chassis | Non-existent IP equals stale encap | `ovn-sbctl list Encap`, clear the orphan row |
| `options:remote_ip` | An IPv4 underlay hop that is reachable | A different subnet with no route | Unreachable equals routing gap | `ip route get <remote_ip>`, investigate the underlay |
| `options:key=flow` | `flow` (OVN dynamic per-LS key) | A fixed key means non-OVN | `key=0` means bridging is not set up | Only OVN sets `flow`. Legacy setups differ. |
| BFD state | `UP` in both directions | `INIT` equals one-way reachability | `DOWN` equals tunnel broken | Part 13.6 §13.6.X BFD state machine |
| BFD `forwarding` | `true` | Intermittent `false` equals flap | Sustained `false` equals underlay issue | Check the RFC 5880 detection multiplier |
| BFD `detect_multiplier` | 3 to 5 by default | Greater than 10 equals slow detection | 1 equals over-aggressive false positives | Balance convergence against stability |
| BFD `min_rx/tx` | 1000 ms by default | Less than 100 ms stresses the control plane | More than 5000 ms causes slow failover | Part 13.6 §13.6.X timing math |
| `Encap.chassis_name` | Matches the SB chassis hostname | Mismatch equals rebind pending | Empty equals chassis not registered | Verify with `ovn-sbctl list Chassis` |

**Failure scenarios:**

- BFD is `UP` but `forwarding=false` toggles 2 to 3 times per minute: the underlay link is flapping. Check the switch uplink, this is not OVN.
- `remote_ip` ping succeeds but BFD is `DOWN`: a firewall is dropping UDP 3784 (BFD) or 6081 (Geneve). Check `iptables-save` and `nft list ruleset`.
- The `Peer` column is empty even though the LS has an LSP: the chassis has not been added with `ovn-sbctl chassis-add`, or there is SBDB propagation lag. Check `ovn-sbctl list Chassis` and Raft health.
- BFD flaps every 5 minutes on a regular schedule: CPU steal, the chassis VM is being preempted. Run `dmesg`, `vmstat`, and audit the hypervisor scheduler.

**Cross-links:** Tier 1 entry (9.27.2). If it passes, go to Tier 2 `ofproto/trace` (§9.27.3) and then Tier 3 `dpif/dump-flows` plus pcap (§9.27.4). The fault catalog in §9.27.5 maps each symptom to the corresponding tier.

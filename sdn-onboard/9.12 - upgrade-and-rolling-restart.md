## Dàn bài

## 9.12.1 Ba quy tắc vàng (golden rules)

Rule 1: **Schema trước binary**, convert OVSDB schema với version mới TRƯỚC khi start binary mới. Reverse order = binary mới không đọc được schema cũ.

Rule 2: **Một daemon một lúc**, ovsdb-server → vswitchd → kmod. KHÔNG restart song song, sẽ mất config in-flight.

Rule 3: **Không upgrade OVS đồng thời với kernel**, nếu vừa upgrade OVS vừa reboot sang kernel mới, khó isolate nguyên nhân nếu lỗi.

## 9.12.2 5-step choreography

Step 1: Install package mới KHÔNG restart daemon. Debian: `apt install openvswitch-switch=<new-version>` với `DPKG_NO_RESTART=yes` hoặc systemd `disable-then-install`.

Step 2: Stop ovsdb-server graceful: `ovs-appctl -t ovsdb-server exit --cleanup`. Flag `--cleanup` flush write buffer + close socket cleanly, tránh corruption.

Step 3: Check schema version: `ovsdb-tool needs-conversion /etc/openvswitch/conf.db /usr/share/openvswitch/vswitch.ovsschema`. Nếu `yes`, chạy `ovsdb-tool convert /etc/openvswitch/conf.db /usr/share/openvswitch/vswitch.ovsschema`.

Step 4: Start ovsdb-server (binary mới) rồi restart vswitchd: `systemctl reload-or-restart openvswitch-switch`. Với systemd unit có `Restart=on-failure`, reload-or-restart là atomic.

Step 5: Nếu kernel datapath: `rmmod openvswitch` + `modprobe openvswitch` (version mới). Kernel module reload evict megaflow cache, brief traffic interrupt vài chục ms nhưng conntrack state ĐƯỢC giữ (nằm trong netfilter core, không trong OVS module).

## 9.12.3 --bundle flag cho atomic vswitchd restart

OVS 2.11+ support `systemctl restart --bundle`: load binary mới, prepare state, swap nguyên tử, packet được re-inject vào binary mới mà không drop (ngoại trừ ~1ms pause đồng bộ).

## 9.12.4 Failure mode: schema convert fail trên production

Nếu `needs-conversion` báo yes nhưng `convert` fail: thường do custom column được add trước đó qua `ovs-vsctl` + third-party tool. Recovery: restore từ backup (Part 10.2 `ovsdb-client backup` routine), hoặc manual schema surgery với `ovsdb-tool transact`. RISK HIGH.

## 9.12.5 In-place vs cordon-and-replace

In-place: 5-step choreography trên từng chassis tuần tự. Nhanh, không cần spare capacity. Rủi ro: nếu 1 chassis fail mid-upgrade, cluster degraded.

Cordon-and-replace: drain workload khỏi chassis → upgrade chassis thành fresh install → rebind workload. An toàn hơn nhưng yêu cầu orchestrator hỗ trợ workload migrate + N+1 capacity.

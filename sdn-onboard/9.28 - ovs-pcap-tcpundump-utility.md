## 9.28.1 `ovs-pcap` — pcap binary → hex single-line

`ovs-pcap` là utility thuần reformatter: đọc file pcap (định dạng nhị phân chuẩn của libpcap), in mỗi packet ra một dòng riêng dưới dạng hex string. Không can thiệp datapath, không cần `ovs-vswitchd` chạy, không đọc OVSDB.

**Synopsis:**

```
ovs-pcap file
ovs-pcap -h | --help
ovs-pcap -V | --version
```

**Hai option duy nhất:**

| Option | Mục đích |
|--------|----------|
| `-h`, `--help` | In help message rồi exit |
| `-V`, `--version` | In phiên bản OVS rồi exit |

**Output format:** mỗi packet trong pcap được serialize thành một dòng hex digits (không space, không prefix `0x`). Ví dụ packet ICMP echo request:

```
$ ovs-pcap capture.pcap | head -1
ffffffffffff0011223344550806000108000604000100112233445510000001000000000000c0a80164
```

Đây CHÍNH XÁC là format mà `ovs-appctl ofproto/trace` chấp nhận khi simulate packet — không cần parse pcap ngược, chỉ pipe trực tiếp.

**Quan trọng:** packet phải bao gồm Ethernet header. Pcap thông thường (capture từ `tcpdump`, `wireshark`) đã có Ethernet header, nên `ovs-pcap` hoạt động ngay. Nhưng nếu pcap được tạo bằng tool capture L3-only (rare), Ethernet header thiếu và `ofproto/trace` sẽ reject.

## 9.28.2 `ovs-tcpundump` — tcpdump hex text → hex single-line

`ovs-tcpundump` là utility reverse của `ovs-pcap` (về mặt input format). Nó đọc output của `tcpdump -xx` (hex format có Ethernet header) từ stdin, không phải file pcap binary.

**Synopsis:**

```
ovs-tcpundump < input.txt
ovs-tcpundump -h | --help
ovs-tcpundump -V | --version
```

**Use case chính:** khi bạn chỉ có **tcpdump capture text** từ log production (không có file pcap binary), `ovs-tcpundump` cho phép convert text này thành format hex single-line để pipe vào `ofproto/trace`.

**Cảnh báo critical:** input phải là `tcpdump -xx` (hai chữ x), KHÔNG phải `tcpdump -x` (một chữ x). Khác biệt:

- `tcpdump -x` (1 chữ x): in hex từ L3 (IP header), bỏ qua Ethernet header.
- `tcpdump -xx` (2 chữ x): in hex từ L2 (Ethernet header) — bao gồm dest MAC, source MAC, ethertype.

`ofproto/trace` cần Ethernet header để biết packet là IPv4/IPv6/ARP/MPLS qua `eth_type` field. Input thiếu Ethernet sẽ bị reject hoặc gây trace sai. Bug này khá phổ biến — ví dụ trong document copy-paste từ tutorial cũ dùng `tcpdump -x`.

**Output:** giống `ovs-pcap`, mỗi packet một dòng hex.

## 9.28.3 Workflow tích hợp với `ofproto/trace`

Hai utility là cầu nối giữa **packet capture thực tế** và **OpenFlow pipeline simulation**. Workflow điển hình:

```
+-------------+          +-----------+         +----------------+
|   tcpdump   |   pcap   |  ovs-pcap |   hex   | ofproto/trace  |
| (capture)   | -------->| (convert) | ------->| (simulate)     |
+-------------+          +-----------+         +----------------+

HOẶC

+-------------+          +-----------------+         +----------------+
| tcpdump -xx |   text   | ovs-tcpundump   |   hex   | ofproto/trace  |
| (log file)  | -------->| (convert stdin) | ------->| (simulate)     |
+-------------+          +-----------------+         +----------------+
```

**Lệnh thực tế (case 1: pcap file):**

```bash
# 1. Capture trên br-int trong 5 giây
sudo timeout 5 tcpdump -i br-int -w /tmp/cap.pcap

# 2. Convert + pipe vào ofproto/trace
ovs-pcap /tmp/cap.pcap | head -1 | xargs -I{} \
  sudo ovs-appctl ofproto/trace br-int "{}"
```

**Lệnh thực tế (case 2: từ log text):**

```bash
# 1. Có sẵn file log từ production (ví dụ engineer paste vào ticket)
cat /tmp/incident-tcpdump.log
# 14:23:01.234567 IP 10.0.0.1.443 > 10.0.0.2.5678: Flags [S]
#         0x0000:  ffff ffff ffff 0011 2233 4455 0800 4500
#         0x0010:  003c 0001 0000 4006 ...

# 2. Convert + trace
cat /tmp/incident-tcpdump.log | ovs-tcpundump | \
  while read pkt; do
    sudo ovs-appctl ofproto/trace br-int "$pkt"
  done
```

## 9.28.4 Anatomy: pipe `ovs-pcap` → `ofproto/trace`

**Bucket:** OVS | **Context:** post-incident analysis trên packet đã capture | **Purpose:** simulate xử lý OpenFlow pipeline cho packet capture thật, không cần re-inject vào datapath | **Activity:** debug + verify flow rule | **Mechanism:** ovs-pcap reformat → ofproto/trace simulate.

**Output mẫu:**

```
$ ovs-pcap /tmp/incident.pcap | head -1
ffff...0800450000280001000040067...

$ echo "ffff...0800450000280001000040067..." | xargs -I{} ovs-appctl ofproto/trace br-int "{}"
Flow: in_port=LOCAL,vlan_tci=0x0000,dl_src=00:11:22:33:44:55,dl_dst=ff:ff:ff:ff:ff:ff,dl_type=0x0800,...

bridge("br-int")
----------------
 0. in_port=1, priority 100
    actions=resubmit(,8)
 8. priority 50
    actions=load:0x6->NXM_NX_REG14[],resubmit(,9)
 ...

Final flow: ...
Datapath actions: drop
```

**Decode 5-attribute Anatomy:**

| Attribute | Giá trị mẫu | Healthy / Warning / Critical |
|-----------|-------------|------------------------------|
| Hex dump length | 84 ký tự = 42 byte (Ethernet 14 + IPv4 20 + 8 byte payload) | Healthy: ≥ 14 byte (đủ Ethernet) |
| Flow parsing | `dl_src=...,dl_dst=...,dl_type=0x0800,...` | Critical: nếu thiếu `dl_type` → Ethernet header missing, fix tcpdump capture command |
| Bridge name | `bridge("br-int")` | Healthy: bridge tồn tại; nếu sai bridge → ofproto trả lỗi |
| Pipeline trace | List action mỗi table | Critical: thấy `drop` ở table 8 (port-security) → packet không khớp port_security |
| Final flow | Action set cuối | Healthy: action có `output:N` hoặc `output:CONTROLLER`. Critical: `drop` không kèm reason → bug rule. |

## 9.28.5 Bài tập có hướng dẫn (Guided Exercise)

**Mục tiêu:** capture packet ICMP thật, convert qua `ovs-pcap`, replay qua `ofproto/trace` để verify pipeline simulation khớp với traffic thật.

**Setup:**

```bash
# 1. Tạo veth pair giả lập 2 VM
ip netns add ns1
ip netns add ns2
ip link add veth-ns1 type veth peer name veth-br1
ip link add veth-ns2 type veth peer name veth-br2
ip link set veth-ns1 netns ns1
ip link set veth-ns2 netns ns2
ovs-vsctl add-port br-int veth-br1
ovs-vsctl add-port br-int veth-br2
ip link set veth-br1 up && ip link set veth-br2 up
ip netns exec ns1 ip addr add 10.0.0.1/24 dev veth-ns1
ip netns exec ns1 ip link set veth-ns1 up
ip netns exec ns2 ip addr add 10.0.0.2/24 dev veth-ns2
ip netns exec ns2 ip link set veth-ns2 up
```

**Capture + replay:**

```bash
# 2. Capture trong khi ping
sudo tcpdump -i br-int -w /tmp/icmp.pcap -c 4 &
sleep 1
ip netns exec ns1 ping -c 2 10.0.0.2
wait

# 3. Convert + simulate
ovs-pcap /tmp/icmp.pcap | head -1 | xargs -I{} ovs-appctl ofproto/trace br-int "{}"

# 4. Verify expected: trace dẫn đến output:<port của veth-br2>
```

**Verify khớp với traffic thật:**

```bash
# Dump flow tương ứng thấy được pipeline expected
ovs-ofctl dump-flows br-int | grep "in_port=$(ovs-vsctl get Interface veth-br1 ofport)"
```

So sánh action trong trace với flow trong dump → khớp = workflow OK.

## 9.28.6 Mẫu cần tránh và lưu ý

**Mẫu cần tránh 1: Dùng `tcpdump -x` thay vì `tcpdump -xx` cho ovs-tcpundump.**

`-x` (1 chữ x) bỏ Ethernet header → ovs-tcpundump output thiếu MAC + ethertype → ofproto/trace nhận packet sai layer. Triệu chứng: trace báo "no Ethernet" hoặc tự assume default MAC.

**Cách phòng:** luôn dùng `tcpdump -xx -i <iface> -w <file>` cho output text dùng cho ovs-tcpundump. Cho pcap binary, mọi `tcpdump -w` đều có Ethernet header.

**Mẫu cần tránh 2: Truncate packet bằng `tcpdump -s 96`.**

Mặc định `tcpdump` capture full packet. Nếu set `-s 96` (snaplen 96 byte), payload sau byte 96 bị cut. ofproto/trace không lỗi nhưng pipeline simulation cho rule depend vào payload (rare, nhưng có ACL match nội dung) sẽ sai.

**Cách phòng:** dùng `tcpdump -s 0` (mặc định = full packet) khi capture cho purpose simulation.

**Lưu ý 3: ofproto/trace không thay đổi datapath state.**

Pipeline simulation read-only. KHÔNG cài flow, KHÔNG commit conntrack, KHÔNG ảnh hưởng OVSDB. An toàn chạy trên production.

**Lưu ý 4: Output hex từ `ovs-pcap` có thể rất dài cho packet kích thước lớn.**

Một MTU 1500 packet = 3000 ký tự hex. Bash variable + `xargs` có giới hạn ARG_MAX. Cho packet lớn, dùng file trung gian:

```bash
ovs-pcap /tmp/cap.pcap | head -1 > /tmp/pkt.hex
ovs-appctl ofproto/trace br-int "$(cat /tmp/pkt.hex)"
```

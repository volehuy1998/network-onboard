## 9.16.1 Vai trò connection manager

**Connection manager** là thành phần trong ovs-vswitchd (cụ thể trong `ofproto-dpif.c` và `connmgr.c`) quản lý toàn bộ kết nối TCP/SSL từ switch tới các OpenFlow controller. Mỗi bridge có một instance connection manager riêng, theo dõi danh sách controller được cấu hình qua `ovs-vsctl set-controller`.

Nhiệm vụ chính:
- **Thiết lập kết nối:** mở TCP/SSL tới controller với retry + exponential backoff.
- **Heartbeat:** gửi OpenFlow echo_request định kỳ, phát hiện controller không tiếp cận được.
- **Vòng đời flow:** khi controller connect/disconnect, xử lý fail_mode (secure/standalone) để quyết định giữ hoặc xóa flow table.
- **Quản lý role:** điều phối role MASTER/SLAVE/EQUAL khi nhiều controller cùng lúc.
- **Định tuyến message OpenFlow:** forward packet_in tới MASTER, echo tới tất cả, định tuyến reply đúng controller.
- **Trạng thái kết nối:** theo dõi trạng thái mỗi kết nối (CONNECTING, ACTIVE, IDLE, DISCONNECTED) và báo cáo qua `ovs-ofctl dump-ports` / `ovs-appctl bridge/dump-flows`.

Kết nối controller có hai mode khác biệt:
- **Primary (in-band / out-of-band):** gửi/nhận messages OpenFlow protocol. Có thể nhiều primary cho HA (hỗ trợ từ OF 1.3 với role election).
- **Service (ovs-ofctl monitor):** chỉ observe, không gửi packet_out hay ảnh hưởng flow table. Dùng cho gỡ lỗi.

Cấu hình primary:
```
[host]# ovs-vsctl set-controller br0 tcp:10.0.0.1:6653 tcp:10.0.0.2:6653 tcp:10.0.0.3:6653
```
Ba controller HA. Connection manager sẽ cố gắng kết nối tất cả và duy trì.

## 9.16.2 OpenFlow 1.3 role election — MASTER/SLAVE/EQUAL

OpenFlow 1.0/1.2 giả định chỉ một controller, khi nhiều controller cùng lúc, không có cơ chế điều phối → race condition (2 controller cùng ghi flow table gây bất nhất). OpenFlow 1.3 (RFC-like spec) giới thiệu **messages role request/reply** cho phép controller khai báo role:

**MASTER:** có đầy đủ quyền ghi flow table (add/modify/delete flow). Nhận mọi message bất đồng bộ (packet_in, flow_removed, port_status). Chỉ có MỘT master tại một thời điểm, nếu controller A set role=MASTER, controller khác với role=MASTER tự động bị demote thành SLAVE.

**SLAVE:** chỉ có quyền đọc (chỉ đọc). Có thể dump flow table, đăng ký port stats, nhưng không được add/modify flow. Nhận packet_in? Tùy spec: OF 1.3 cho phép SLAVE không nhận packet_in (chỉ MASTER), OF 1.4+ cho phép SLAVE đăng ký packet_in nhưng không phản hồi.

**EQUAL:** bằng nhau, mọi controller EQUAL có quyền ghi flow table và nhận mọi message. Không có khái niệm "chỉ một master" cho EQUAL. Điều này cho phép topology active-active nhưng đòi hỏi controller phải tự phối hợp để tránh xung đột (qua kênh ngoài băng).

**Format message role request/reply:**
```
struct ofp_role_request {
    uint32_t role;         // 0=NOCHANGE, 1=EQUAL, 2=MASTER, 3=SLAVE
    uint32_t pad;
    uint64_t generation_id; // bộ đếm đơn điệu chống replay
};
```

**Generation ID** là cơ chế chống race: mỗi lần controller request role=MASTER, nó phải gửi kèm generation_id cao hơn generation_id hiện tại của switch. Nếu generation_id thấp hơn → switch reject (controller này lỗi thời). Đây là cách OVS xác định "ai là master mới nhất" khi có nhiều controller race để claim MASTER.

**Quy trình chuyển đổi role thông thường:**
1. Controller A kết nối, gửi role_request(MASTER, gen=1). Switch chấp nhận.
2. Controller B kết nối, gửi role_request(MASTER, gen=2). Switch demote A → SLAVE, promote B → MASTER.
3. Controller A nhận role_status notification "you are now SLAVE", có thể react (retry với gen=3 để tranh master lại).
4. Nếu B mất kết nối, A retry role_request với gen=3. Switch chấp nhận → A lại là MASTER.

## 9.16.3 Fail_mode — hành vi khi mất kết nối controller

Khi tất cả controller disconnect, connection manager quyết định hành vi qua cấu hình **fail_mode**:

**fail_mode=secure** (mặc định trong OVS môi trường production): giữ nguyên flow table hiện tại. Packet tiếp tục được forward theo flow rule đã có. Không chấp nhận packet không match → drop hoặc gửi qua default rule. Phù hợp cho môi trường production: nếu controller down, switch vẫn chạy với trạng thái flow cuối cùng, không gián đoạn traffic.

**fail_mode=standalone** (mặc định cho bridge mới): xóa toàn bộ OpenFlow flow và fall back về hành vi MAC learning switch truyền thống (như Linux bridge). Switch hoạt động như L2 learning switch, packet được flood + học source MAC. Phù hợp cho bootstrapping (chưa kết nối controller lần đầu) nhưng không dùng cho môi trường production.

Cấu hình fail_mode:
```
[host]# ovs-vsctl set-fail-mode br0 secure
[host]# ovs-vsctl get-fail-mode br0
secure
```

**Chuyển đổi fail_mode:** nếu bridge đang ở secure mode và mất kết nối, flow table giữ nguyên. Khi controller reconnect, MASTER đầu tiên có thể "đồng bộ lại" flow table (delete all + re-install theo trạng thái mới) nếu muốn.

## 9.16.4 Kết nối in-band so với out-of-band

Kết nối controller có thể đi qua hai đường:

**Out-of-band:** controller và switch kết nối qua network quản trị riêng (không dùng port OpenFlow-managed). Đây là cấu hình chuẩn môi trường production, switch có interface management (eth0 riêng) connect tới controller.

**In-band:** kết nối controller đi qua chính bridge OpenFlow-managed. Switch phải tự học và forward traffic OpenFlow tới controller qua flow table. Thách thức: gà đẻ trứng, switch cần flow để forward, nhưng flow đến từ controller.

OVS hiện thực in-band bằng cách tự động sinh "in-band rule" trong flow table: match destination IP controller → forward ra interface đi controller. Rule này được refresh tự động khi topology thay đổi.

Cấu hình in-band: gắn IP controller thuộc subnet của bridge port (không dùng mạng quản trị riêng). OVS tự phát hiện và bật mode in-band.

In-band có điểm yếu: nếu flow table bị hỏng hoặc rule in-band bị ghi đè, switch mất kết nối → flow cũ mãi mãi. Out-of-band tránh rủi ro này nhưng đòi hỏi hạ tầng quản trị riêng.

## 9.16.5 Bundle — apply nguyên tử qua nhiều flow

OpenFlow 1.4+ giới thiệu **bundle**, cơ chế apply nhiều flow_mod nguyên tử. Controller gửi series flow_mod vào bundle, rồi commit → switch apply tất cả cùng lúc hoặc không apply gì (nguyên tử).

Messages:
- `bundle_open(bundle_id)`: khởi tạo bundle.
- `bundle_add_message(bundle_id, flow_mod)`: thêm flow_mod vào bundle (chưa apply).
- `bundle_commit(bundle_id)`: apply tất cả message trong bundle nguyên tử.
- `bundle_discard(bundle_id)`: hủy bundle không apply.

Trường hợp sử dụng: controller cần update 100 flow cùng lúc (ví dụ migrate traffic từ path A sang path B). Nếu gửi 100 flow_mod riêng rẽ, có cửa sổ trong đó half-A + half-B → packet drop hoặc loop. Bundle đảm bảo apply cùng lúc → không có cửa sổ bất nhất.

OVS hỗ trợ bundle từ OVS 2.4+ với OpenFlow 1.4+. Dùng qua `ovs-ofctl --bundle`:
```
[host]# ovs-ofctl --bundle add-flows br0 flows.txt
```
File `flows.txt` chứa nhiều flow, apply trong một bundle.

## 9.16.6 Quan sát — trạng thái kết nối + role

**Xem controller đã cấu hình:**
```
[host]# ovs-vsctl list Controller
```

**Xem trạng thái kết nối:**
```
[host]# ovs-ofctl dump-connection br0
```
Output hiển thị mỗi controller với: target, role (MASTER/SLAVE/EQUAL), state (connected/waiting), thống kê kết nối.

**Monitor chuyển đổi role thời gian thực:**
```
[host]# ovs-ofctl monitor br0 --ver=OpenFlow13
```
Output stream events, bao gồm notification role_status khi role thay đổi.

**Thống kê nội bộ ovs-appctl:**
```
[host]# ovs-appctl bridge/dump-flows br0
[host]# ovs-appctl -t ovs-vswitchd coverage/show | grep -E 'ofproto|connmgr'
```
Bộ đếm coverage `ofproto_packet_in`, `connmgr_reconnect`, giám sát bất thường.

**Thay đổi role từ CLI (cho test):**
```
[host]# ovs-ofctl -O OpenFlow13 mod-connection br0 role=master
```
Chỉ có tác dụng qua kết nối CLI đang dùng, ovs-ofctl set role=master cho session hiện tại.

### ▶ Guided Exercise 1: Test HA đa controller với role election

> **Trạng thái verification lab:** Output dựa trên OVS 2.17 + OpenFlow 1.3 docs; chờ verify lab thực.

**Mục đích:** Cấu hình 2 controller cho bridge, quan sát role election khi 2 controller cùng claim MASTER.

**Chuẩn bị:** Node Ubuntu 22.04 với OVS 2.17 + Ryu controller (`pip install ryu`).

**Bước thực hiện:**

**1.** Tạo bridge với OpenFlow 1.3:
```
[host]# ovs-vsctl add-br br0 -- set bridge br0 protocols=OpenFlow13
[host]# ovs-vsctl set-controller br0 tcp:127.0.0.1:6653 tcp:127.0.0.1:6654
[host]# ovs-vsctl set-fail-mode br0 secure
```

**2.** Chạy 2 Ryu controller ở port khác nhau:
```
[term1]# ryu-manager --ofp-tcp-listen-port 6653 ryu.app.simple_switch_13
[term2]# ryu-manager --ofp-tcp-listen-port 6654 ryu.app.simple_switch_13
```

**3.** Quan sát trạng thái kết nối:
```
[host]# ovs-ofctl -O OpenFlow13 dump-connection br0
```

Output kỳ vọng:
```
bridge br0 has 2 controllers:
  tcp:127.0.0.1:6653 role=EQUAL state=connected
  tcp:127.0.0.1:6654 role=EQUAL state=connected
```

Mặc định cả hai controller là EQUAL (không có ai claim MASTER). Cả hai có quyền ghi flow table, rủi ro!

**4.** Giả lập controller 1 claim MASTER (qua Ryu app tùy chỉnh, hoặc dùng ovs-ofctl):
```
[host]# # Qua ovs-ofctl từ terminal controller 1:
[host]# ovs-ofctl -O OpenFlow13 mod-connection br0 role=master \
         --target=tcp:127.0.0.1:6653
```

**5.** Kiểm tra lại kết nối:
```
[host]# ovs-ofctl -O OpenFlow13 dump-connection br0
  tcp:127.0.0.1:6653 role=MASTER state=connected
  tcp:127.0.0.1:6654 role=EQUAL state=connected
```

Controller 1 là MASTER. Nếu controller 2 cũng request MASTER → controller 1 bị demote.

**6.** Kill controller 1, quan sát failover:
```
[term1]# # Ctrl+C
[host]# ovs-ofctl -O OpenFlow13 dump-connection br0
  tcp:127.0.0.1:6653 role=MASTER state=CONNECTING  # kết nối lại
  tcp:127.0.0.1:6654 role=EQUAL state=connected
```

Flow table trên br0 vẫn giữ (fail_mode=secure). Packet vẫn forward theo rule cũ.

**Bài học học được:**

Thứ nhất, role mặc định là EQUAL, không có bầu chọn tự động. Nếu cần HA active-passive, controller phải tường minh request role MASTER. Đây là pattern chuẩn trong ONOS, Faucet.

Thứ hai, fail_mode=secure cực kỳ quan trọng cho môi trường production. Nếu set standalone, controller mất kết nối → switch fall back về MAC learning → hoàn toàn bỏ policy OpenFlow → policy bảo mật bị mất.

Thứ ba, generation_id đảm bảo không có race giữa 2 controller claim master. Controller mới nhất (generation cao hơn) thắng.

**Cleanup:**
```
[host]# ovs-vsctl del-br br0
```

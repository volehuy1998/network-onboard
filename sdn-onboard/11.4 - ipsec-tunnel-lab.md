## 11.4.1 Tại sao IPsec cho overlay

Overlay encap cung cấp tenant isolation nhưng **không encrypt**. Packet tenant tồn tại plaintext trong outer IP → ai có access underlay có thể sniff + decode.

IPsec bắt buộc trong hai scenario:

1. **Cloud-hybrid**: overlay qua untrusted Internet.
2. **Multi-tenant DC**: compliance (HIPAA, PCI-DSS) yêu cầu encryption-at-wire.

IPsec encrypt outer packet với ESP (Encapsulating Security Payload).

## 11.4.2 ovs-monitor-ipsec daemon

Compass Ch 12: "`ovs-monitor-ipsec` watches the `Interface.options:psk` or `Interface.options:remote_cert` columns and programs strongSwan or Libreswan accordingly."

1. Admin cấu hình tunnel + IPsec option qua `ovs-vsctl`.
2. `ovs-monitor-ipsec` daemon subscribe OVSDB, detect change.
3. Daemon generate config cho strongSwan + reload.
4. strongSwan establish IPsec SA với remote peer.
5. OVS tunnel traffic flow qua IPsec encap.

# Host A:
[hostA]# cp ipsec-node1-cert.pem /etc/openvswitch/ipsec-cert.pem

[hostA]# ovs-vsctl add-port br0 vxlan-ipsec -- \
         set Interface vxlan-ipsec type=vxlan \
         options:remote_ip=10.0.0.2 options:key=flow \
         options:remote_cert=<host-B-cert-path>
```

Rotation: issue cert mới, không cần sync secret.

## 11.4.7 AH và ESP, vì sao ESP thắng

IPsec gốc RFC 2401 (1998) định nghĩa hai protocol:

**AH (Authentication Header, RFC 4302):**
- Cung cấp: authentication + integrity + anti-replay.
- KHÔNG cung cấp: confidentiality (không encrypt).
- Scope bảo vệ: bao gồm outer IP header.
- IP protocol number: 51.

**ESP (Encapsulating Security Payload, RFC 4303):**
- Cung cấp: confidentiality + integrity + authentication + anti-replay.
- Scope bảo vệ: chỉ payload phía sau ESP header (không bao gồm outer IP).
- IP protocol number: 50.

Lab 15 Figure 1 (trang 5) liệt kê các protocol hỗ trợ bởi OVS (highlighted light blue):

```
IPsec Protocol
├── Confidentiality
│   ├── DES, 3DES, AES, SEAL
├── Integrity
│   ├── AH
│   ├── ESP
│   └── ESP + AH
├── Authentication
│   ├── MD5, SHA
│   ├── PSK, RSA
│   └── Diffie-Hellman (DH1, DH2, ..., DH14)
```

### Vì sao ESP thắng AH

Trong thực tế năm 2026, ESP gần như thay thế hoàn toàn AH. Ba lý do:

**Một, ESP đã cung cấp authentication.** ESP với ESP-Auth field đảm bảo integrity tương đương AH. Dùng AH riêng chỉ thêm overhead.

**Hai, AH không qua NAT.** AH bảo vệ outer IP header (bao gồm source/destination IP). NAT sửa IP header → AH phát hiện thay đổi → reject packet. Với Internet đầy NAT, AH không dùng được thực tế.

**Ba, ESP có tunnel mode đóng gói hoàn chỉnh.** ESP-Tunnel mode encap cả inner IP header + payload, chỉ để lộ outer IP. An toàn cao nhất, NAT-friendly.

Lab 15 §1.1 ghi rõ: *"ESP is much more widely used since it provides confidentiality in addition to authentication and data integrity."*

OVS IPsec chỉ hỗ trợ ESP tunnel mode. AH không có config option trong ovs-monitor-ipsec.

## 11.4.8 IKE phase 1, Diffie-Hellman và ISAKMP

IKE (Internet Key Exchange) là protocol thương thuyết key và thiết lập Security Association (SA). Có hai phase, thực hiện qua ISAKMP protocol (RFC 2408, obsoleted bởi RFC 7296 IKEv2).

### Phase 1 mục tiêu

Thiết lập một kênh bảo mật giữa hai peer để thương thuyết tiếp Phase 2. Trao đổi key Diffie-Hellman để tạo shared secret mà không cần gửi key qua line. Authenticate identity của hai peer bằng PSK hoặc RSA certificate.

### Diffie-Hellman key exchange

DH là thuật toán toán học 1976 cho phép hai peer thương thuyết shared secret qua kênh public. Cơ chế:

1. Hai peer đồng ý công khai: prime `p`, generator `g`.
2. Alice chọn số ngẫu nhiên `a`, tính `A = g^a mod p`, gửi A cho Bob.
3. Bob chọn số ngẫu nhiên `b`, tính `B = g^b mod p`, gửi B cho Alice.
4. Alice tính `shared = B^a mod p = g^(ab) mod p`.
5. Bob tính `shared = A^b mod p = g^(ab) mod p`.
6. Cả hai có cùng `shared`, kẻ nghe lén biết A và B nhưng khó tính được `a` hoặc `b` do problem discrete log.

DH group định nghĩa kích thước `p`:
- **DH2:** 1024-bit, đã bị break (không dùng production).
- **DH5:** 1536-bit, deprecated.
- **DH14:** 2048-bit, baseline acceptable.
- **DH19/20:** ECP 256-bit/384-bit, dùng elliptic curve, khuyến nghị cho môi trường production.

strongSwan mặc định DH14 + DH19, tương thích OVS IPsec.

### Phase 1 sequence

Lab 15 Figure 2 (trang 6) minh hoạ chuỗi message:

```
Sender                                    Receiver
  │                                          │
  │ 1. Thương thuyết SA params               │
  │    (encryption, hash, DH group, auth)    │
  │─────────────────────────────────────────▶│
  │◀─────────────────────────────────────────│
  │                                          │
  │ 2. DH key exchange                       │
  │    (trao đổi public value A, B)          │
  │─────────────────────────────────────────▶│
  │◀─────────────────────────────────────────│
  │                                          │
  │ 3. Identity + authentication             │
  │    (PSK hash hoặc RSA signature)         │
  │─────────────────────────────────────────▶│
  │◀─────────────────────────────────────────│
  │                                          │
  │ ✓ ISAKMP SA thành lập                    │
```

Sau phase 1, hai peer có shared secret và đã authenticate lẫn nhau. Kênh ISAKMP SA được dùng để thương thuyết tiếp Phase 2.

### Authentication methods

- **PSK (Pre-Shared Key):** password chung giữa hai peer. Đơn giản, phù hợp lab hoặc deployment nhỏ (< 10 peer). Compromise → rotate toàn bộ fabric.
- **RSA Signature:** mỗi peer có private/public key pair. Certificate ký bởi CA xác minh identity. Phù hợp môi trường production scale lớn, rotation certificate không cần sync secret.

Lab 15 dùng PSK (`password`) để đơn giản hoá demo. Môi trường production phải dùng RSA cert.

## 11.4.9 IKE phase 2, IPsec SA và ESP

### Phase 2 mục tiêu

Thiết lập IPsec SA cho việc encrypt data traffic thực tế. Mỗi direction có SA riêng (uni-directional), nghĩa là hai peer có 2 SA (Alice→Bob và Bob→Alice) cho ESP.

### Phase 2 quick mode

Dùng key shared từ phase 1 để encrypt message phase 2. Thương thuyết:

- **Encryption algorithm cho ESP:** AES-128-CBC, AES-256-GCM, 3DES.
- **Authentication algorithm cho ESP:** HMAC-SHA-256, HMAC-SHA-384.
- **SA lifetime:** default 3600 giây hoặc 4 GB traffic, tái thương thuyết rekey.

Sau phase 2, IPsec SA active. Mọi packet qua tunnel được ESP encrypt.

### ESP header structure (Lab 15 §1.3 Figure 5)

```
┌──────────────────────────────────────────┐
│ Outer IP header                          │
├──────────────────────────────────────────┤
│ ESP header                               │
│   SPI (32-bit, Security Parameter Index) │
│   Sequence Number (32-bit)               │
├──────────────────────────────────────────┤
│ ESP Payload (encrypted)                  │
│   ┌──────────────────────────────────┐   │
│   │ Original IP header (inner)       │   │
│   ├──────────────────────────────────┤   │
│   │ GRE header (nếu GRE inside IPsec)│   │
│   ├──────────────────────────────────┤   │
│   │ TCP/UDP + payload                │   │
│   └──────────────────────────────────┘   │
├──────────────────────────────────────────┤
│ ESP Trailer                              │
│   Padding + Pad Length + Next Header     │
├──────────────────────────────────────────┤
│ ESP ICV (Integrity Check Value)          │
│   HMAC signature của toàn bộ ESP         │
└──────────────────────────────────────────┘
```

Hai field quan trọng trong ESP header:

- **SPI (Security Parameter Index):** 32-bit identifier cho IPsec SA tương ứng. Receiver dùng SPI để tìm key giải mã. Mỗi chiều có SPI khác nhau.
- **Sequence Number:** increment cho mỗi packet. Dùng anti-replay (nếu receiver thấy số trùng, packet bị reject).

ESP trailer + ICV đảm bảo integrity. Bất kỳ thay đổi nào trong ESP payload làm ICV sai, receiver reject packet.

```

### Filter ISAKMP (phase 1 negotiation)

Trong filter box gõ `isakmp`. Chỉ thấy packet khi hai peer đang thương thuyết (mỗi lần bắt đầu tunnel hoặc rekey). Format (Lab 15 Figure 49 trang 26):

```
Source        Destination    Protocol    Length   Info
192.168.1.10  192.168.2.10   ISAKMP      396      Identity Protection (Main Mode)
192.168.2.10  192.168.1.10   ISAKMP      388      Identity Protection (Main Mode)
192.168.1.10  192.168.2.10   ISAKMP      228      Identity Protection (Main Mode)
192.168.2.10  192.168.1.10   ISAKMP      196      Identity Protection (Main Mode)
192.168.1.10  192.168.2.10   ISAKMP      124      Identity Protection (Main Mode)
192.168.2.10  192.168.1.10   ISAKMP      124      Identity Protection (Main Mode)
```

Sáu message tương ứng Main Mode IKEv1 (3 roundtrip). Chọn một packet để expand:

```
Internet Security Association and Key Management Protocol
  Initiator SPI: 0x....
  Responder SPI: 0x....
  Version: 1.0
  Exchange type: Identity Protection (Main Mode)
  Flags: Encryption, ...
  Payloads:
    Security Association
      Proposal: IKE
        Transform: ENCR_AES_CBC, AUTH_HMAC_SHA2_256, DH_GROUP_14
```

Thông tin SA negotiation visible ở mức metadata, nhưng key material đã được encrypt sau lần đầu. Kẻ nghe lén biết peer dùng AES-CBC + SHA-256 + DH14, không biết key.

### Filter ESP (phase 2 data)

Gõ `esp` trong filter:

```
Source        Destination    Protocol    Length   Info
192.168.1.10  192.168.2.10   ESP         180      ESP (SPI=0x12345678)
192.168.2.10  192.168.1.10   ESP         180      ESP (SPI=0x87654321)
192.168.1.10  192.168.2.10   ESP         180      ESP (SPI=0x12345678)
...
```

Chọn packet ESP, expand:

```
Encapsulating Security Payload
  ESP SPI: 0x12345678 (305419896)
  ESP Sequence: 42
  (encrypted data — 148 bytes ciphertext, không đọc được)
```

Lab 15 Figure 52 (trang 27) nhấn mạnh: *"You will not see the actual source (10.0.0.1) and destination IP address (10.0.0.4) since all the data are encrypted."*

Đây là bằng chứng trực quan IPsec encrypt tầng mạng. Khác với Lab 14 GRE plaintext, giờ router r3 chỉ thấy ciphertext opaque.

## Guided Exercise 14, POE "IPsec không làm chậm" bác bỏ bằng băng thông thực tế

**Mục tiêu:** Đo hiệu năng IPsec encryption so với tunnel không IPsec, chứng minh overhead CPU 10-30% (tùy AES-NI). Hiểu tại sao cần hardware acceleration cho production high-bandwidth.

**Giả thuyết ngược cần bác bỏ:**

> *"IPsec encryption chạy trên CPU hiện đại thì không ảnh hưởng hiệu năng tunnel. Băng thông thực tế với và không IPsec tương đương nhau."*

### Chuẩn bị

Topology Lab 15 đã active. Cài iperf3 trên các host:

```bash
d1$ apt install -y iperf3
d2$ apt install -y iperf3
```

Kiểm tra CPU có AES-NI:

```bash
grep -o aes /proc/cpuinfo | head -1
# Output: aes (có AES-NI)
```

### Bước 1, Baseline không IPsec (GRE only)

Xoá PSK khỏi tunnel:

```bash
d1$ ovs-vsctl remove Interface s1-tun options psk
d2$ ovs-vsctl remove Interface s2-tun options psk
```

Chạy iperf3 băng thông thực tế:

```bash
h2$ iperf3 -s
h1$ iperf3 -c 10.0.0.2 -t 30
```

Ghi băng thông thực tế. Ví dụ kết quả doc-plausible: ~950 Mbps (gần line rate 1 Gbps của interface Mininet veth).

### Bước 2, Bật IPsec

```bash
d1$ ovs-vsctl set Interface s1-tun options:psk=password
d2$ ovs-vsctl set Interface s2-tun options:psk=password
```

Đợi 10 giây cho IKE hoàn tất:

```bash
d1$ ipsec statusall | grep ESTABLISHED
```

Chạy iperf3 lại:

```bash
h1$ iperf3 -c 10.0.0.2 -t 30
```

Ghi băng thông thực tế. Kỳ vọng giảm xuống ~700-850 Mbps tùy CPU (giảm 10-25%).

### Bước 3, Phân tích overhead

Giả thuyết ngược bị bác bỏ: IPsec **làm chậm** khoảng 10-25% trong môi trường CPU-only. Nguyên nhân:

- Mỗi packet phải AES-CBC/GCM encrypt (16-byte block).
- HMAC-SHA-256 tính ICV.
- Sequence number management + anti-replay check.
- Tất cả trên CPU userspace (strongSwan + kernel netlink).

### Bước 4, Kiểm chứng AES-NI accelerate

Với AES-NI (Intel/AMD CPU từ 2010), AES-CBC tăng tốc 3-10x so với pure software. Kiểm chứng:

```bash
# Bench AES-NI
openssl speed -elapsed -evp aes-256-gcm
# Output: khoảng 3-5 GB/s throughput AES trên 1 core
```

Nếu CPU thiếu AES-NI (VM cũ), IPsec overhead có thể lên 50-70%. Đây là lý do cloud provider nào cũng expose AES-NI flag cho VM.

### Bước 5, Hardware offload qua SmartNIC

Ở môi trường production bandwidth cao (25 Gbps+), CPU không đủ. Giải pháp: offload IPsec vào SmartNIC:

- NVIDIA ConnectX-6 IPsec in-line offload.
- Intel XL710 + E810 IPsec hardware acceleration.
- Chelsio T6 Unified Wire.

Kiểm tra NIC có hỗ trợ:

```bash
ethtool -k eth0 | grep ipsec
# Output (nếu support): esp-hw-offload: on
```

Với NIC offload, IPsec overhead giảm gần như bằng 0, băng thông thực tế giữ line rate 25/100 Gbps.

### Bài học

IPsec không miễn phí về CPU. Biết trade-off giữa security và performance quan trọng khi thiết kế cluster. Với VM thông thường, accept 10-25% overhead. Với high-bandwidth node (compute edge, storage), invest vào SmartNIC offload. Với IoT/embedded thiếu AES-NI, chọn crypto nhẹ hơn (ChaCha20-Poly1305 software-friendly hơn AES mà không có hardware).

## 11.4.12 IPsec trong OVN cluster encryption

OVN production deployment có tuỳ chọn encrypt mọi tunnel giữa chassis bằng IPsec. Cấu hình:

```bash
# Trên mỗi chassis
ovs-vsctl set Open_vSwitch . other_config:ovn-chassis-mac-mappings=...
ovs-vsctl set Open_vSwitch . external_ids:ovn-encap-ip=<chassis-IP>
ovs-vsctl set Open_vSwitch . external_ids:ovn-encap-type=geneve

# Bật IPsec cho tất cả Geneve tunnel
ovn-nbctl set NB_Global . ipsec=true
```

`ovn-northd` sinh tunnel config với `options:psk` tự động cho mọi Geneve port trên mọi chassis. `ovs-monitor-ipsec` trên từng chassis thiết lập IPsec SA full-mesh.

Scale consideration:
- Cluster N chassis → N*(N-1)/2 IPsec SA.
- 100 chassis → 4.950 SA.
- strongSwan xử lý tốt đến ~10.000 SA mỗi node, beyond đó cần tuning hoặc hardware offload.

### So sánh với OVN WAN multi-DC

Cho scenario OVN trải nhiều datacenter qua WAN:

- **Intra-DC Geneve:** không cần IPsec (untrusted: no, độ trễ: low).
- **Inter-DC Geneve over IPsec:** bắt buộc IPsec (untrusted: yes, độ trễ: cao hơn nhưng chấp nhận được).

Thường dùng GRE outer cho inter-DC thay vì Geneve vì GRE header ổn định hơn qua transit ISP.

## Điểm cốt lõi cần nhớ

**Một.** IPsec cung cấp ba bảo đảm (confidentiality, integrity, authentication). GRE/VXLAN/Geneve chỉ encap không bảo mật. Pattern chuẩn site-to-site VPN: GRE/Geneve inside IPsec.

**Hai.** AH (protocol 51) và ESP (protocol 50) là hai IPsec protocol. ESP thắng thế vì NAT-friendly và đã có authentication built-in. OVS IPsec chỉ hỗ trợ ESP tunnel mode.

**Ba.** IKE phase 1 thiết lập ISAKMP SA qua Diffie-Hellman key exchange + PSK/RSA authentication. Phase 2 thương thuyết IPsec SA cho ESP. DH14 là baseline, DH19/20 (elliptic curve) cho production.

**Bốn.** OVS IPsec qua `options:psk=...` hoặc `options:remote_cert=...` trên Interface. `ovs-monitor-ipsec` daemon subscribe OVSDB, sinh strongSwan config tự động, admin không touch strongSwan trực tiếp. `systemctl enable --now openvswitch-ipsec` đủ.

**Năm.** ESP header có SPI (nhận diện SA) + sequence number (anti-replay). ESP trailer có Padding + Next Header. ICV là HMAC signature cả gói ESP. Wireshark ESP filter chỉ thấy metadata, không đọc được payload.

**Sáu.** IPsec trên CPU overhead 10-25% băng thông thực tế khi có AES-NI. Không có AES-NI có thể 50-70%. Hardware offload qua SmartNIC (NVIDIA ConnectX-6, Intel E810) gần như zero-overhead ở 25/100 Gbps.

**Bảy.** PSK phù hợp lab + deployment nhỏ (< 10 peer). Certificate-based (RSA, ovs-pki) cho môi trường production scale lớn, rotation không cần sync secret.

**Tám.** OVN cluster có thể bật IPsec full-mesh bằng `ovn-nbctl set NB_Global . ipsec=true`. Scale đến ~10.000 SA mỗi node với strongSwan vanilla, beyond cần SmartNIC offload.

## 20.3.1 Category 1 — Daily health check (morning routine)

Operator mở ngày làm việc với 5 lệnh health check snapshot để xác định cluster đang healthy hay cần attention. Mỗi lệnh < 1 giây, tổng < 10 giây.

### 20.3.1.1 Full topology snapshot — `ovn-nbctl show`

**Bối cảnh khi chạy:** Lệnh đầu tiên mỗi sáng. Confirm NBDB intent không drift sau đêm qua. Cũng run khi take-over shift từ oncall trước.

**Command và output** *(reproduced from OVN 22.03.8 sandbox, 3-LSP + 1-LR topology)*:

```shell
$ ovn-nbctl show

    port lsp_web_01
        addresses: ["aa:bb:cc:00:00:01 10.10.1.10"]
    port lsp_web_02

        router-port: rtr_prod-web

switch e83f20e7-0002-4a2b-9c3d-000000000002 (ls_prod_db)
    port lsp_db_01
        addresses: ["aa:bb:cc:00:01:01 10.10.2.10"]
    port ls_prod_db-rtr

        router-port: rtr_prod-db

router 9f8e7d6c-5b4a-3928-1706-000000000001 (rtr_prod)

    nat 3c4d5e6f-7890
        external ip: "192.168.100.100"
        logical ip: "10.10.1.0/24"
        type: "snat"
```

**Anatomy block cấu trúc:**

| Line pattern | Giá trị mẫu | Ý nghĩa | Dấu hiệu đáng lưu ý |
|--------------|-------------|---------|---------------------|
| `switch <uuid> (<name>)` | `switch e83...001 (ls_prod_web)` | Logical Switch header: UUID + name. Mỗi LS là L2 broadcast domain isolated | UUID bất thường (ví dụ `00000...`) có thể là placeholder lỗi |
| `port <lsp_name>` | `port lsp_web_01` | LSP thuộc LS này. Indent 1 level | Port thiếu = VM đã detach; port dư = orphan chưa clean |
| `addresses: [...]` | `["aa:bb:cc... 10.10.1.10"]` | MAC + IP assigned cho LSP, format `"<MAC> <IP1> <IP2> ..."` | `dynamic` = IPAM managed; `unknown` = promiscuous flood mode |
| `port <name>` + `type: router` | `type: router` | LSP bridge sang Logical Router qua patch port | Phải pair với LRP cùng name |
| `router <uuid> (<name>)` | `router 9f8...001 (rtr_prod)` | Logical Router header | Mỗi tenant thường 1 LR |
| `port <lrp_name>` + `mac` + `networks` | `networks: ["10.10.1.1/24"]` | LRP với gateway IP cho LS attach | MAC + networks phải unique across LR |
| `gateway chassis: [...]` | `[chassis-gw-1 chassis-gw-2]` | Danh sách gateway chassis cho distributed gateway port | Thiếu = no HA, traffic centralized qua 1 chassis |
| `nat <uuid>` + `external_ip/logical_ip/type` | `type: "snat"` | NAT rule: snat / dnat / dnat_and_snat | Thiếu NAT = tenant không ra được external |

**Kịch bản bẻ gãy:**

- **Port hiển thị `<name>-rtr` với `type: router` nhưng không có LR pair** → router port orphan. Fix: `ovn-nbctl lsp-del <port>` hoặc re-add LR.
- **Switch không có port nào** → orphan LS. Safe to delete nếu không còn CMS reference: `ovn-nbctl ls-del <ls_name>`.
- **Router thiếu `gateway chassis`** → traffic north-bound sẽ centralized qua random chassis, not HA. Fix: `ovn-nbctl lrp-set-gateway-chassis <lrp> <chassis> <priority>`.
- **NAT rule `external_ip` trùng giữa 2 LR** → cluster conflict, CMS bug. Kiểm chứng: `ovn-nbctl --bare --columns=external_ip,logical_ip list NAT | sort | uniq -d`.

**Upstream nguồn:** [`ovn-nbctl(8)` §show](https://man7.org/linux/man-pages/man8/ovn-nbctl.8.html).


### 20.3.1.5 Logical luồng count sanity

```shell
$ ovn-sbctl lflow-list | wc -l
2847

$ ovn-sbctl --columns=logical_port list Port_Binding | grep -c logical_port
42
```

Rule-of-thumb: ~50-100 logical_flow mỗi LSP tùy ACL + LB count. `lflow_count / port_count` > 200 = over-complexity, cân nhắc consolidate ACL bằng Port_Group.

**Daily health checklist (copy-paste):**

```bash
#!/bin/bash
# /usr/local/bin/ovn-health-check.sh
set -e
echo "=== NBDB topology ==="
ovn-nbctl show | head -30
echo ""
echo "=== Chassis liveness ==="
ovn-sbctl show | head -20
echo ""
echo "=== northd status ==="
sudo ovn-appctl -t ovn-northd status
sudo ovn-appctl -t ovn-northd is-paused
echo ""
echo "=== nb_cfg sync ==="
nb=$(ovn-nbctl get NB_Global . nb_cfg)
sb=$(ovn-sbctl get SB_Global . nb_cfg)
echo "NB=$nb SB=$sb"
[[ $nb -eq $sb ]] || echo "WARNING: nb_cfg drift"
echo ""
echo "=== Lflow stats ==="
echo "lflow count: $(ovn-sbctl lflow-list | wc -l)"
echo "port count: $(ovn-sbctl --columns=logical_port list Port_Binding | grep -c logical_port)"
```

## 20.3.3 Category 3 — Port lifecycle (bind / unbind / migrate)

Operator thường phải add/remove/migrate LSP để serve CMS request. 6 workflow cơ bản.

### 20.3.3.1 Add new LSP with MAC + IP

```shell
# Tạo LSP trong LS existing
$ ovn-nbctl lsp-add ls_prod_web lsp_web_03

# Assign MAC + IP (static)
$ ovn-nbctl lsp-set-addresses lsp_web_03 "aa:bb:cc:00:00:03 10.10.1.12"

# Optional: enable port security
$ ovn-nbctl lsp-set-port-security lsp_web_03 "aa:bb:cc:00:00:03 10.10.1.12"

# Kiểm chứng
$ ovn-nbctl lsp-get-addresses lsp_web_03
aa:bb:cc:00:00:03 10.10.1.12
```

Dynamic IP (IPAM-managed, Phần 13.12):

```shell
$ ovn-nbctl lsp-set-addresses lsp_web_04 "aa:bb:cc:00:00:04 dynamic"
# OVN tự assign IP từ exclude_ips range của LS
```

### 20.3.3.2 Bind LSP với OVS interface (chassis-side)

Sau khi LSP tạo trong NBDB, chassis cần bind qua OVS interface:

```shell
[chassis-hv-01]$ sudo ovs-vsctl add-port br-int tap-web-03 \
    -- set Interface tap-web-03 type=internal \
       external_ids:iface-id=lsp_web_03
```

ovn-controller detect `external_ids:iface-id` → match LSP name → update `Port_Binding.chassis` trong SBDB. Kiểm chứng sau 2-5 giây:

```shell
$ ovn-sbctl --bare --columns=chassis list Port_Binding lsp_web_03
<chassis-hv-01-uuid>
```

### 20.3.3.3 Remove LSP

```shell
# Xóa từ NBDB
$ ovn-nbctl lsp-del lsp_web_03

# Kiểm chứng đã gone
$ ovn-nbctl lsp-list ls_prod_web | grep lsp_web_03
# (no output)

# Clean up OVS interface trên chassis (nếu CMS chưa làm)
[chassis-hv-01]$ sudo ovs-vsctl del-port tap-web-03
```

### 20.3.3.4 Rename LSP (rare, thường CMS không rename)

OVN không có rename built-in. Workaround:

```shell
$ ovn-nbctl lsp-del lsp_old_name
$ ovn-nbctl lsp-add ls_prod_web lsp_new_name
$ ovn-nbctl lsp-set-addresses lsp_new_name "aa:bb:cc:00:00:05 10.10.1.14"
```

### 20.3.3.5 Migrate LSP cross-chassis (manual)

CMS-driven migration thường tự động. Manual migrate khi khắc phục sự cố:

```shell
# Step 1: Unbind trên chassis-A (unplug tap)
[chassis-hv-01]$ sudo ovs-vsctl del-port tap-web-03

# Step 2: ovn-controller trên chassis-A release Port_Binding.chassis (tự động, trong 5s)
$ ovn-sbctl --bare --columns=chassis list Port_Binding lsp_web_03
# (empty)

# Step 3: Bind trên chassis-B
[chassis-hv-02]$ sudo ovs-vsctl add-port br-int tap-web-03 \
    -- set Interface tap-web-03 type=internal \
       external_ids:iface-id=lsp_web_03

# Step 4: Kiểm chứng claim chassis-B
$ ovn-sbctl --bare --columns=chassis list Port_Binding lsp_web_03
<chassis-hv-02-uuid>
```

### 20.3.3.6 Disable LSP tạm thời (administrative down)

Không cần delete nếu chỉ muốn tạm ngừng chuyển tiếp:

```shell
# Disable
$ ovn-nbctl lsp-set-enabled lsp_web_03 false

# Kiểm chứng
$ ovn-nbctl lsp-get-enabled lsp_web_03
false

# Re-enable
$ ovn-nbctl lsp-set-enabled lsp_web_03 true
```

Khi `enabled=false`, stage `ls_in_check_port_sec` fire `drop;` cho ingress từ port này. Traffic không vào chuỗi xử lý (pipeline) nhưng LSP vẫn tồn tại, preserved ACL/NAT/LB reference.

## 20.3.4 Category 4 — ACL management

ACL là policy quan trọng nhất cho security. 5 scenario common.

### 20.3.4.1 Add ACL (allow, drop, allow-related, reject)

```shell
# Allow-related: stateful, reply auto-permit (xem Part 13.3)
$ ovn-nbctl acl-add ls_prod_web from-lport 1000 \
    'inport == "lsp_web_01" && ip' allow-related

# Drop: silent drop
$ ovn-nbctl acl-add ls_prod_web from-lport 2000 \
    'inport == "lsp_web_01" && tcp.dst == 22' drop

# Reject: ICMP unreachable response
$ ovn-nbctl acl-add ls_prod_web from-lport 1500 \
    'inport == "lsp_web_01" && tcp.dst == 3389' reject

# Allow plain (stateless, không commit conntrack)
$ ovn-nbctl acl-add ls_prod_web to-lport 1000 \
    'outport == "lsp_web_01" && icmp' allow
```

### 20.3.4.2 List ACL với verbose

```shell
$ ovn-nbctl acl-list ls_prod_web
from-lport  2000 (inport == "lsp_web_01" && tcp.dst == 22)  drop
from-lport  1500 (inport == "lsp_web_01" && tcp.dst == 3389)  reject log(name="reject-rdp")
from-lport  1000 (inport == "lsp_web_01" && ip)  allow-related
to-lport    1000 (outport == "lsp_web_01" && icmp)  allow
```

ACL sort theo priority desc (highest first fired).

### 20.3.4.3 Delete ACL

```shell
# Bằng match criteria
$ ovn-nbctl acl-del ls_prod_web from-lport 2000 \
    'inport == "lsp_web_01" && tcp.dst == 22'

# Bằng direction + priority
$ ovn-nbctl acl-del ls_prod_web from-lport 2000

# Xóa tất cả ACL trên LS (cẩn thận!)
$ ovn-nbctl acl-del ls_prod_web
```

### 20.3.4.4 Test ACL trước khi apply — `ovn-trace`

```shell
# Simulate packet matching ACL rule
$ ovn-trace ls_prod_web \
    'inport == "lsp_web_01" && eth.src == aa:bb:cc:00:00:01 &&
     eth.dst == aa:bb:cc:00:00:02 && ip4.src == 10.10.1.10 &&
     ip4.dst == 10.10.1.11 && ip.ttl == 64 && tcp && tcp.dst == 22'

# Expected output: drop; tại ls_in_acl table với priority=2000
```

Đây là pattern "dry-run" ACL — kiểm chứng logic trước khi deploy change rộng.

### 20.3.4.5 ACL audit — tìm rule matching specific traffic pattern

```shell
# Tìm ACL nào block SSH (port 22)
$ ovn-nbctl --bare --columns=direction,priority,match,action list ACL \
    | grep "tcp.dst == 22"

from-lport  2000  (inport == "lsp_web_01" && tcp.dst == 22)  drop
from-lport  2000  (inport == "lsp_web_02" && tcp.dst == 22)  drop

# Tìm ACL không có log action
$ ovn-nbctl --bare --columns=action,log list ACL | awk '$2=="false" {print}'
```

### 20.3.4.6 Port_Group cho ACL scale

Thay vì 100 ACL cho 100 LSP, consolidate qua Port_Group (Phần 13.3):

```shell
# Tạo Port_Group
$ ovn-nbctl pg-add pg_web

# Add LSP vào group
$ ovn-nbctl --id=@lsp1 get Logical_Switch_Port lsp_web_01 \
    -- add Port_Group pg_web ports @lsp1

$ ovn-nbctl --id=@lsp2 get Logical_Switch_Port lsp_web_02 \
    -- add Port_Group pg_web ports @lsp2

# ACL dùng Port_Group thay cho individual LSP match
```

## 20.3.11 End-to-end workflow 1 — New tenant provisioning

Scenario: Cấp tenant mới "tenant-42" với 1 LS + 3 LSP + ACL default-deny + DHCP + SNAT.

```shell
#!/bin/bash
# new-tenant.sh tenant-42 10.10.42.0/24

TENANT=$1
CIDR=$2
GATEWAY=10.10.42.1
GATEWAY_MAC=00:00:00:ff:fe:42

# 1. Create LR nếu chưa có
ovn-nbctl --may-exist lr-add rtr_tenant

# 2. Create LS
ovn-nbctl ls-add ls_${TENANT}

# 3. Add LRP attached to LS
ovn-nbctl lrp-add rtr_tenant rtr_tenant-${TENANT} \
    $GATEWAY_MAC $GATEWAY/24
ovn-nbctl lsp-add ls_${TENANT} ls_${TENANT}-rtr
ovn-nbctl lsp-set-type ls_${TENANT}-rtr router
ovn-nbctl lsp-set-addresses ls_${TENANT}-rtr router
ovn-nbctl lsp-set-options ls_${TENANT}-rtr router-port=rtr_tenant-${TENANT}

# 4. Create 3 LSP
for i in 1 2 3; do
    MAC="aa:bb:cc:42:00:0${i}"
    IP="10.10.42.1${i}"
    ovn-nbctl lsp-add ls_${TENANT} lsp_${TENANT}_vm${i}
    ovn-nbctl lsp-set-addresses lsp_${TENANT}_vm${i} "$MAC $IP"
    ovn-nbctl lsp-set-port-security lsp_${TENANT}_vm${i} "$MAC $IP"
done

# 5. Create DHCP options
DHCP_UUID=$(ovn-nbctl create DHCP_Options \
    cidr=$CIDR \
    options='"server_id"="10.10.42.1" "lease_time"="3600" "router"="10.10.42.1" "dns_server"="{8.8.8.8}"' \
    | tr -d '"')

# Attach DHCP tới tất cả LSP
for i in 1 2 3; do
    ovn-nbctl lsp-set-dhcpv4-options lsp_${TENANT}_vm${i} $DHCP_UUID
done

# 6. Add ACL default-deny + allow-related egress
ovn-nbctl acl-add ls_${TENANT} from-lport 1000 'ip' drop
ovn-nbctl acl-add ls_${TENANT} from-lport 2000 'ip.dst == 10.10.42.0/24' allow-related
ovn-nbctl acl-add ls_${TENANT} from-lport 2000 'ip.dst == 0.0.0.0/0 && ip.dst != 10.0.0.0/8' allow-related

# 7. Add SNAT
ovn-nbctl lr-nat-add rtr_tenant snat 192.168.100.142 $CIDR

echo "Tenant $TENANT provisioned with 3 VMs on $CIDR"
ovn-nbctl show | grep -A 10 ls_${TENANT}
```

```shell
     ip4.dst == 8.8.8.8 && ip.ttl == 64 && icmp4'
# Expected: output reach router, SNAT, eventual external path
```

## 20.3.12 End-to-end workflow 2 — Tenant teardown

Scenario: Remove tenant-42 an toàn. Order quan trọng để tránh orphan.

```shell
#!/bin/bash
TENANT=tenant-42

# 1. Stop forwarding (disable LSP trước xóa)
for lsp in $(ovn-nbctl --bare --columns=name find Logical_Switch_Port \
             external_ids:tenant=$TENANT); do
    ovn-nbctl lsp-set-enabled $lsp false
done

# 2. Remove LB + NAT + DHCP references
ovn-nbctl --if-exists lb-del ${TENANT}-web-lb
for lsp in $(ovn-nbctl --bare --columns=name find Logical_Switch_Port \
             external_ids:tenant=$TENANT); do
    ovn-nbctl lsp-set-dhcpv4-options $lsp ''
done

# 3. Remove NAT rules
for nat_uuid in $(ovn-nbctl --bare --columns=_uuid find NAT \
                  external_ids:tenant=$TENANT); do
    ovn-nbctl --if-exists destroy NAT $nat_uuid
done

# 4. Remove ACL
ovn-nbctl --if-exists acl-del ls_${TENANT}

# 5. Remove LSP
for lsp in $(ovn-nbctl --bare --columns=name find Logical_Switch_Port \
             external_ids:tenant=$TENANT); do
    ovn-nbctl --if-exists lsp-del $lsp
done

# 6. Remove LRP
ovn-nbctl --if-exists lrp-del rtr_tenant-${TENANT}

# 7. Remove LS
ovn-nbctl --if-exists ls-del ls_${TENANT}

# 8. Garbage collect DHCP (nếu không tenant khác share)
# (Careful! Chỉ del nếu unique)
# ovn-nbctl dhcp-options-del <uuid>

# 9. Kiểm chứng sạch
```

## 20.3.16 Capstone POE — "Add 500 ACL to 1 Logical_Switch: safe for production?"

**Scenario:** Engineering manager đề xuất: "Để fine-grained micro-segmentation, thêm 500 ACL cho `ls_prod_web` với priority unique mỗi rule. Safe không?"

**Predict:** Sẽ ảnh hưởng:
1. northd compile time (500 ACL = 500+ logical luồng).
2. ovn-controller lflow-cache size.
3. br-int OpenFlow rule count.
4. Debug complexity.

**Observe (reasoning):**

**1.** northd compile:

```bash
# Current baseline 10 ACL
$ sudo ovn-appctl -t ovn-northd stopwatch/show | grep "build_lflows"
# Short term average: 20ms

# Expected 500 ACL:
# build_lflows ~100-200ms (linear scale), 95th percentile ~500ms.
# Still < 1s threshold, OK.
```

**2.** lflow-cache:

```bash
# Current
$ sudo ovn-appctl -t ovn-controller lflow-cache/show-stats | grep "total count"
# total count: 1200

# Expected +500 ACL = +2500 logical flow (each ACL ~5 flow multi-stage)
# New total: ~3700
# cache-max-size default 200K, không stress.
```

**3.** br-int OpenFlow:

```bash
# Current
$ sudo ovs-ofctl dump-aggregate br-int | grep flow_count
# flow_count: 3200

# Expected: +7500 OpenFlow (OVN expansion 3x logical → physical)
# New total: ~10700
# Threshold cần concern: 50K (HW offload limit), 100K (revalidator stress).
# Vẫn OK.
```

**4.** Debug:

```bash
# Current ACL list
$ ovn-nbctl acl-list ls_prod_web | wc -l
# 10

# Expected: 510 ACL
# ovn-trace output tăng từ ~50 dòng per call lên ~500 dòng
# Debug effort scale linear, nhưng vẫn tractable.
```

**Explain:** Đề xuất **acceptable with 2 caveat**:

- **Caveat 1:** ACL priority phải unique nhưng well-ordered. Random priority → pattern "first-match" không predictable → hard debug. Fix: group ACL theo logical category (100-199 = allow, 200-299 = drop explicit, 300+ = default), document trong wiki.
- **Caveat 2:** Thay vì 500 ACL individual per LSP, dùng Port_Group consolidation (§20.3.4.6). Ví dụ 50 port-group với 10 ACL mỗi group = effective 500 ACL nhưng compiled thành ~100 ACL thực trên NBDB. Compile time scale tốt hơn.

**Correct recommendation:**

1. Phân loại ACL thành 5-10 Port_Group.
2. Reserve priority block (100-199 allow, 200-299 stateful allow-related, 300-399 deny specific, 900+ default-deny).
3. Add ACL qua script batch + apply trong maintenance window.
4. Stopwatch baseline trước + sau để measure impact thực.

**Finish:** Rút ra nguyên tắc: "Scale ACL bằng consolidation (Port_Group) + discipline (priority block), không bằng raw count". 500 ACL "raw" = bad pattern; 500 ACL "organized via 20 Port_Group" = fine.

## 20.3.17 Hiểu sai phổ biến

> **Hiểu sai 1:** "`ovn-nbctl` và `ovn-sbctl` interchangeable."
> **Đúng:** NBDB read-write (operator ý đồ cấu hình, CMS write). SBDB read-mostly (compiled output, ovn-controller read + write status back). `ovn-sbctl` chủ yếu cho debug + inspection, không modify (ví dụ tạo Logical_Flow qua sbctl là wrong path, phải qua NBDB → northd compile).

> **Hiểu sai 2:** "Delete LS delete luôn mọi LSP bên trong."
> **Đúng:** Phụ thuộc version + CMS. OVN 22.03+ có cascade delete qua `--cascade` flag. Default behavior có thể leave orphan LSP. Luôn explicit teardown workflow §20.3.12.

> **Hiểu sai 3:** "`ovn-nbctl ls-add name` idempotent — chạy 2 lần OK."
> **Đúng:** Default sẽ error nếu LS tồn tại. Dùng `--may-exist` flag cho idempotent script: `ovn-nbctl --may-exist ls-add name`.

> **Hiểu sai 4:** "`lsp-set-addresses unknown` = dynamic IPAM."
> **Đúng:** `unknown` = promiscuous flood mode (receive mọi MAC destination không match ai). `dynamic` mới là IPAM. Đừng nhầm.

> **Hiểu sai 5:** "`ovn-nbctl acl-del` không có argument xóa specific rule duy nhất."
> **Đúng:** Không arg = xóa TẤT CẢ ACL trên LS. Dangerous. Luôn specify direction + priority + match.

> **Hiểu sai 6:** "Backup NBDB đủ, không cần SBDB."
> **Đúng:** NBDB là ý đồ cấu hình, có thể re-compile ra SBDB qua northd. NHƯNG SBDB có state động (Port_Binding.chassis, Nút mạng (Chassis) registration, MAC_Binding cache, Service_Monitor health) không recover từ NBDB. Backup cả hai.

> **Hiểu sai 7:** "Nút mạng (Chassis) maintenance = stop ovn-controller ngay."
> **Đúng:** Phải drain LSP trước (migrate VM). Stop trực tiếp = active LSP lose claim → blackhole traffic 10-30s đến khi CMS detect + re-schedule.

> **Hiểu sai 8:** "Tăng `lflow-cache` size giải quyết mọi performance issue."
> **Đúng:** Cache size lớn hơn consume memory nhiều hơn. Hit rate > 95% là healthy. Nếu hit rate thấp → root cause khác (lflow churn, schema change, bulk update). Tuning cache size không fix root cause.

## 20.3.18 Điểm cốt lõi cần nhớ

Thứ nhất, **10 category daily task** cover 95% operator workflow: health / inventory / port lifecycle / ACL / LB+NAT / DHCP+DNS / gateway+HA / conntrack / performance / backup+maintenance. Nắm cả 10 = operator competent.

Thứ hai, **`ovn-nbctl` subcommand prefix pattern**: `ls-` (Logical Switch) / `lsp-` (LSP) / `lr-` (Logical Router) / `lrp-` (LRP) / `acl-` / `lb-` / `lr-nat-` / `pg-` (Port_Group) / `dhcp-options-` / `ha-chassis-group-`. Mỗi prefix có verb `add`/`del`/`list`/`set-*`/`get-*`. Thuộc pattern = tự khám phá subcommand mới.

Thứ ba, **health check morning routine 5 lệnh, < 10 giây**: `ovn-nbctl show` + `ovn-sbctl show` + `ovn-appctl -t ovn-northd status` + `nb_cfg` compare + lflow count. Tự động hóa qua `ovn-health-check.sh` script.

Thứ tư, **`--may-exist` và `--if-exists`** là flag idempotent quan trọng cho script automation. Always use trong CI/CD pipeline để tránh flake.

Thứ năm, **Port_Group consolidation** giảm ACL count × 5-10 lần. Pattern chuẩn cho cluster > 500 LSP. Không scale ACL bằng raw count.

Thứ sáu, **backup NBDB + SBDB cả hai**, không chỉ NBDB. Dùng `ovsdb-client backup` hoặc `/var/lib/ovn/*.db` file copy khi services stopped.

Thứ bảy, **chassis maintenance có quy trình 7 bước**: inventory → cordon → drain → kiểm chứng empty → stop → maintenance → restart → kiểm chứng re-register. Không skip bước nào. Skip drain = blackhole traffic.

Thứ tám, **end-to-end workflow script** (new-tenant, tenant-teardown, rolling-upgrade) là deliverable quý nhất của operator. Checklist thành script, script thành CI, CI thành SLA.

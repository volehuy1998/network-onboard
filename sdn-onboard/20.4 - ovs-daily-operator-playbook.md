## 20.4.1 Category 1, daily OVS health check (morning routine)

Five health-check commands snapshot the OVS state every morning. Each command runs in less than one second; the total stays under 10 seconds.

### 20.4.1.1 Bridge and port topology, `ovs-vsctl show`

**Context:** This is the first command run every morning, to confirm that the bridge state has not drifted overnight. Also run when taking over an on-call shift.

**Command and output** *(reproduced from OVS 2.17.9, two-bridge topology)*:

```shell
$ sudo ovs-vsctl show

5e83f20e-8f3d-4a2b-9c3d-abcdef012345
    Bridge br-int
        fail_mode: secure
        datapath_type: system
        Port br-int
            Interface br-int
                type: internal
        Port "tap-vm01"
            Interface "tap-vm01"
                external_ids: {attached-mac="aa:bb:cc:00:00:01", iface-id="lsp_web_01"}
        Port "tap-vm02"
            Interface "tap-vm02"
                external_ids: {attached-mac="aa:bb:cc:00:00:02", iface-id="lsp_web_02"}
        Port "ovn-chassisB-0"
            Interface "ovn-chassisB-0"
                type: geneve
                options: {csum="true", key=flow, remote_ip="10.0.0.12"}
    Bridge br-ex
        Port "eth1"
            Interface "eth1"
        Port "patch-br-int-to-br-ex"
            Interface "patch-br-int-to-br-ex"
                type: patch
                options: {peer="patch-br-ex-to-br-int"}
    ovs_version: "2.17.9"
```

**Anatomy block, structure:**

| Line pattern | Sample value | Meaning | Signal to watch |
|--------------|-------------|---------|-----------------|
| `<uuid>` line 1 | `5e83...345` | UUID of the Open_vSwitch root table | A change means `ovs-vswitchd` restarted recently |
| `Bridge <name>` | `Bridge br-int` | OVS bridge definition | `br-int` is OVN-managed; `br-ex` is the external gateway |
| `fail_mode: <mode>` | `secure` | Behavior when the controller disconnects | `secure` drops new flows (§9.14.6 Scenario I); `standalone` is hub mode |
| `datapath_type: <type>` | `system` | Kernel (`system`) versus userspace (`netdev` DPDK) | `netdev` indicates DPDK PMD mode |
| `Port <name>` plus `Interface <name> type: <T>` | `type: internal/geneve/patch/dpdk/...` | Interface type per port | Missing `type` means a physical NIC |
| `external_ids` | `{iface-id="lsp_web_01"}` | Metadata (set by the CMS) | OVN uses `iface-id` to bind a `Port_Binding` |
| `options` (tunnel) | `{remote_ip="10.0.0.12", key=flow}` | Tunnel endpoint configuration | `key=flow` means a dynamic `tunnel_key` per flow |
| `ovs_version` | `"2.17.9"` | Running version | Compare with the apt version after upgrade |

**Failure scenarios:**

- **Bridge `br-int` is missing.** OVN is non-functional. Check the `ovn-controller` log.
- **A port has `type: internal` but no upper veth.** The internal port is orphaned. Remove with `ovs-vsctl del-port`.
- **Tunnel port `options:remote_ip` is empty.** The tunnel cannot form. Check the `Encap` table in SBDB.
- **`fail_mode: secure` plus controller disconnect causes traffic to stop.** The fix is `set-fail-mode standalone` as a temporary measure.
- **`ovs_version` mismatch across chassis.** A rolling upgrade is in progress and behavior may drift.

### 20.4.1.2 `ovs-vswitchd` and `ovsdb-server` daemon health

```shell
$ systemctl is-active openvswitch-switch ovsdb-server
active
active

$ sudo ovs-appctl -t ovs-vswitchd version
ovs-vswitchd (Open vSwitch) 2.17.9

# Memory footprint
$ sudo ovs-appctl -t ovs-vswitchd memory/show
handlers:3 revalidators:4 ofconns:5 ports:247
rules:48293 rules-cache-used:156MB rules-cache-max:256MB
```

If `rules-cache-used` exceeds 80 percent of `rules-cache-max`, tuning is needed (§9.14.6 Scenario B).

### 20.4.1.3 Datapath flow sanity, `ovs-dpctl show`

```shell
$ sudo ovs-dpctl show
system@ovs-system:
  lookups: hit:8392751 missed:12783 lost:0
  flows: 4823
  masks: hit:8405534 total:12 hit/pkt:1.00
  port 0: ovs-system (internal)
  port 1: br-int (internal)
  port 2: tap-vm01
  port 3: tap-vm02
  port 4: ovn-chassisB-0 (geneve: packet_type=ptap)
```

**Reading the output:**

- `lookups: hit` is much greater than `missed`, so the cache is effective (hit rate 99.85 percent).
- `lost: 0` means no upcall drops (§9.14.6 Scenario C).
- `flows: 4823` is the current datapath flow count.
- `masks: total: 12` means 12 megaflow mask patterns (low is good, the cache is efficient).
- `hit/pkt: 1.00` is an average of one mask lookup per packet (ideal).

### 20.4.1.4 Upcall and revalidator, `ovs-appctl upcall/show`

```shell
$ sudo ovs-appctl upcall/show
system@ovs-system:
  flows    : (current 4823) (avg 4750) (max 5122) (limit 200000)
  offloaded flows : 0
  dump duration : 23ms
  ufid enabled : true

  23: (keys 1207) (total packets 283749) (avg 785/s)
  24: (keys 1208) (total packets 290485) (avg 801/s)
  25: (keys 1209) (total packets 297843) (avg 822/s)
  26: (keys 1199) (total packets 289201) (avg 798/s)
```

A healthy baseline shows `dump duration` below 100 ms, the `avg /s` values spread evenly across handlers, and `flows / limit` below 50 percent.

### 20.4.1.5 OpenFlow flow table count

```shell
$ sudo ovs-ofctl dump-aggregate br-int
NXST_AGGREGATE reply (xid=0x4): packet_count=8392751 byte_count=12583720473 flow_count=3218

$ sudo ovs-ofctl dump-flows br-int | wc -l
3219   # plus 1 for the header
```

Threshold: less than 10K flows is healthy. 10K to 50K is a warning. More than 50K hits the hardware-offload limit and stresses the revalidator.

**Consolidated daily health script:**

```bash
#!/bin/bash
# /usr/local/bin/ovs-health-check.sh
set -e
echo "=== Bridge topology ==="
sudo ovs-vsctl show | head -30
echo ""
echo "=== Daemon status ==="
systemctl is-active openvswitch-switch ovsdb-server
sudo ovs-appctl -t ovs-vswitchd version
echo ""
echo "=== Datapath sanity ==="
sudo ovs-dpctl show
echo ""
echo "=== Upcall health ==="
sudo ovs-appctl upcall/show | head -10
echo ""
echo "=== Flow count per bridge ==="
for br in $(sudo ovs-vsctl list-br); do
    count=$(sudo ovs-ofctl dump-aggregate $br | grep -oE 'flow_count=[0-9]+' | cut -d= -f2)
    echo "$br: $count flows"
done
```

## 20.4.3 Category 3, bridge and port lifecycle

### 20.4.3.1 Add a new bridge

```shell
# Basic bridge (kernel datapath)
$ sudo ovs-vsctl add-br br-lab

# With explicit datapath type
$ sudo ovs-vsctl add-br br-lab -- set Bridge br-lab datapath_type=system

# DPDK userspace datapath
$ sudo ovs-vsctl add-br br-dpdk -- set Bridge br-dpdk datapath_type=netdev

# Verify
$ sudo ovs-vsctl list-br
br-int
br-lab
br-dpdk
```

### 20.4.3.2 Add ports of various types

```shell
# Physical NIC port
$ sudo ovs-vsctl add-port br-lab eth1

# Internal port (used for the host IP on the bridge)
$ sudo ovs-vsctl add-port br-lab bridge-iface \
    -- set Interface bridge-iface type=internal

# Patch port (connecting two bridges)
$ sudo ovs-vsctl add-port br-int patch-br-int-br-ex \
    -- set Interface patch-br-int-br-ex type=patch options:peer=patch-br-ex-br-int
$ sudo ovs-vsctl add-port br-ex patch-br-ex-br-int \
    -- set Interface patch-br-ex-br-int type=patch options:peer=patch-br-int-br-ex

# Geneve tunnel
$ sudo ovs-vsctl add-port br-int tun-chassisB \
    -- set Interface tun-chassisB type=geneve \
       options:remote_ip=10.0.0.12 \
       options:key=flow \
       options:csum=true

# VXLAN tunnel
$ sudo ovs-vsctl add-port br-int vxlan-chassisB \
    -- set Interface vxlan-chassisB type=vxlan \
       options:remote_ip=10.0.0.12 \
       options:key=flow

# DPDK physical port
$ sudo ovs-vsctl add-port br-dpdk dpdk0 \
    -- set Interface dpdk0 type=dpdk options:dpdk-devargs=0000:01:00.0

# DPDK vhost-user (for VMs)
$ sudo ovs-vsctl add-port br-dpdk vhost-user-1 \
    -- set Interface vhost-user-1 type=dpdkvhostuser
```

### 20.4.3.3 Set port options and `external_ids`

```shell
# OVN iface-id binding
$ sudo ovs-vsctl set Interface tap-vm01 external_ids:iface-id=lsp_web_01

# VLAN tag for the port
$ sudo ovs-vsctl set Port tap-vm01 tag=100

# VLAN trunk
$ sudo ovs-vsctl set Port eth1 trunks=100,200,300

# Disable port STP
$ sudo ovs-vsctl set Port tap-vm01 other_config:stp-enable=false
```

### 20.4.3.4 Remove a port and a bridge

```shell
# Remove port
$ sudo ovs-vsctl del-port br-int tap-vm01

# Remove bridge (cascades to ports)
$ sudo ovs-vsctl del-br br-lab

# Verify
$ sudo ovs-vsctl list-br | grep br-lab
# (empty)
```

### 20.4.3.5 Administratively bring a port up or down

```shell
# Admin down (traffic stops on the port)
$ sudo ovs-ofctl mod-port br-int tap-vm01 down

# Admin up
$ sudo ovs-ofctl mod-port br-int tap-vm01 up

# Check
$ sudo ovs-ofctl show br-int | grep tap-vm01
 3(tap-vm01): addr:aa:bb:cc:00:00:01
     config:     0
     state:      0
     current:    10GB-FD COPPER AUTO_NEG
     ...
```

`config: 0` means admin up; `config: PORT_DOWN` means admin down.

### 20.4.3.6 Clear all ports on a bridge (nuke, be careful)

```shell
# Remove every port except the internal bridge port
$ for p in $(sudo ovs-vsctl list-ports br-lab); do
    sudo ovs-vsctl del-port br-lab $p
  done
```

## 20.4.4 Category 4, OpenFlow flow management

This is the most important layer when debugging pure OVS (controller-less or hybrid controller).

### 20.4.4.1 Add an OpenFlow flow

```shell
# Basic flow: drop all ICMP
$ sudo ovs-ofctl add-flow br-int "priority=100,icmp,actions=drop"

# Flow with cookie (a group identifier)
$ sudo ovs-ofctl add-flow br-int \
    "cookie=0xdeadbeef,priority=200,in_port=3,dl_dst=aa:bb:cc:00:00:02,actions=output:4"

# Multi-table flow with goto_table
$ sudo ovs-ofctl add-flow br-int \
    "table=0,priority=100,icmp,actions=goto_table:1"
$ sudo ovs-ofctl add-flow br-int \
    "table=1,priority=100,icmp,actions=NORMAL"

# Conntrack action
$ sudo ovs-ofctl add-flow br-int \
    "table=0,priority=100,ip,actions=ct(commit,zone=1,table=1)"

# Batch add from a file
$ sudo ovs-ofctl add-flows br-int /tmp/flows.txt
```

### 20.4.4.2 Dump flows with filters

```shell
# All flows
$ sudo ovs-ofctl dump-flows br-int | head

# Specific table
$ sudo ovs-ofctl dump-flows br-int table=0

# Specific priority range
$ sudo ovs-ofctl dump-flows br-int | awk '/priority=10[0-9]{2}/'

# By cookie
$ sudo ovs-ofctl dump-flows br-int "cookie=0xdeadbeef/-1"

# Flows matching specific criteria
$ sudo ovs-ofctl dump-flows br-int "in_port=3"
$ sudo ovs-ofctl dump-flows br-int "ip,nw_src=10.0.0.1"

# With stats
$ sudo ovs-ofctl dump-flows br-int --rsort=packets
```

### 20.4.4.3 Delete flows

```shell
# Remove every flow on the bridge (nuke, be careful)
$ sudo ovs-ofctl del-flows br-int

# Remove a specific flow by match
$ sudo ovs-ofctl del-flows br-int "priority=100,icmp"

# Remove by cookie
$ sudo ovs-ofctl del-flows br-int "cookie=0xdeadbeef/-1"

# Remove every flow in a specific table
$ sudo ovs-ofctl del-flows br-int "table=1"
```

### 20.4.4.4 Modify flows (hot-update actions)

```shell
# Strict modify (match exactly)
$ sudo ovs-ofctl --strict mod-flows br-int \
    "priority=100,icmp,actions=output:5"

# Non-strict (partial match prefix)
$ sudo ovs-ofctl mod-flows br-int \
    "priority=100,icmp,actions=drop"
```

### 20.4.4.5 Atomic `replace-flows` (hot-reload the full table)

```shell
# Dump current state
$ sudo ovs-ofctl dump-flows br-int > /tmp/backup-flows.txt

# Apply new state atomically (diff, install, remove)
$ sudo ovs-ofctl replace-flows br-int /tmp/new-flows.txt

# Dry-run diff before applying
$ sudo ovs-ofctl diff-flows br-int /tmp/new-flows.txt
```

`replace-flows` is more atomic than `del-flows` followed by `add-flows`: there is no empty window between the two.

### 20.4.4.6 Flow hygiene monitor

```shell
# Watch flow changes in real time
$ sudo ovs-ofctl monitor br-int watch:

# Watch packet-in events (controller traffic)
$ sudo ovs-ofctl snoop br-int
```

## 20.4.5 Category 5, tunnel management

Tunnels (Geneve, VXLAN, GRE, STT) are the backbone of overlays. Here are five common workflows.

### 20.4.5.1 Create a tunnel port

```shell
# Geneve (recommended for OVN)
$ sudo ovs-vsctl add-port br-int tun-chassisB \
    -- set Interface tun-chassisB type=geneve \
       options:remote_ip=10.0.0.12 \
       options:key=flow \
       options:csum=true

# VXLAN (legacy)
$ sudo ovs-vsctl add-port br-int vxlan-chassisB \
    -- set Interface vxlan-chassisB type=vxlan \
       options:remote_ip=10.0.0.12 options:key=flow

# GRE
$ sudo ovs-vsctl add-port br-int gre-chassisB \
    -- set Interface gre-chassisB type=gre \
       options:remote_ip=10.0.0.12 options:key=42

# Static tunnel_key (instead of flow-based)
$ sudo ovs-vsctl add-port br-int tun-static \
    -- set Interface tun-static type=geneve \
       options:remote_ip=10.0.0.12 options:key=4242
```

### 20.4.5.2 Tunnel neighbor and ARP resolution

```shell
# Check ARP resolution for the remote underlay
$ sudo ovs-appctl tnl/neigh/show

IP                                            MAC                 Bridge
==========================================================================
10.0.0.12                                     aa:cc:dd:11:22:33   br-int
10.0.0.13                                     aa:cc:dd:11:22:34   br-int

# Force ARP flush
$ sudo ovs-appctl tnl/neigh/flush
```

An empty entry means underlay ARP has not resolved yet, and the tunnel cannot form (§9.14.2.3).

### 20.4.5.3 Tunnel port list and stats

```shell
# List tunnel ports
$ sudo ovs-appctl tnl/ports/show

Listening ports:
geneve_sys_6081 (6081)
vxlan_sys_4789 (4789)
```

Port 6081 is the Geneve default; 4789 is the VXLAN default. If a port is missing, that tunnel type is not configured.

### 20.4.5.4 Tunnel packet trace

```shell
# Trace a packet through a tunnel
$ sudo ovs-appctl ofproto/trace br-int \
    'in_port=3,dl_src=aa:bb:cc:00:00:01,dl_dst=aa:bb:cc:00:00:02,
     dl_type=0x0800,nw_src=10.0.0.1,nw_dst=10.0.0.2,nw_proto=1'

# Expected final action (if cross-chassis):
# set(tunnel(tun_id=0x1,src=10.0.0.11,dst=10.0.0.12,flags(df|key))),output:4
```

### 20.4.5.5 Remove a tunnel

```shell
$ sudo ovs-vsctl del-port br-int tun-chassisB
```

## 20.4.6 Category 6, QoS and mirror

### 20.4.6.1 Rate-limit ingress (policing), per-interface ingress rate

```shell
# 100 Mbps ingress policing on port tap-vm01
$ sudo ovs-vsctl set Interface tap-vm01 \
    ingress_policing_rate=100000 \
    ingress_policing_burst=10000

# Remove
$ sudo ovs-vsctl set Interface tap-vm01 ingress_policing_rate=0
```

Units: rate is in kbps, burst is in KB.

### 20.4.6.2 Egress shaping, QoS with `linux-htb`

```shell
# Create linux-htb QoS for a port
$ sudo ovs-vsctl set Port tap-vm01 \
    qos=@newqos \
    -- --id=@newqos create QoS \
       type=linux-htb \
       other-config:max-rate=1000000000 \
       queues=0=@default-queue,1=@high-prio \
    -- --id=@default-queue create Queue \
       other-config:min-rate=500000000 \
       other-config:max-rate=800000000 \
    -- --id=@high-prio create Queue \
       other-config:min-rate=200000000 \
       other-config:max-rate=1000000000

# Verify
$ sudo ovs-vsctl list QoS
$ sudo ovs-vsctl list Queue
```

### 20.4.6.3 Output to a queue from an OpenFlow rule

```shell
# Send SSH traffic into queue high-prio (id=1)
$ sudo ovs-ofctl add-flow br-int \
    "priority=100,tcp,tp_dst=22,actions=set_queue:1,output:NORMAL"

# Best-effort default
$ sudo ovs-ofctl add-flow br-int \
    "priority=50,actions=output:NORMAL"
```

### 20.4.6.4 Mirror port (SPAN) for troubleshooting

```shell
# Create the mirror port
$ sudo ovs-vsctl add-port br-int tap-debug \
    -- set Interface tap-debug type=internal

# Configure the mirror: capture all traffic on br-int into tap-debug
$ sudo ovs-vsctl \
    -- --id=@m create Mirror name=debug-mirror \
         select-all=true \
         output-port=@p \
    -- --id=@p get Port tap-debug \
    -- set Bridge br-int mirrors=@m

# Capture with tcpdump
$ sudo tcpdump -i tap-debug -nn -w /tmp/mirror.pcap

# Cleanup
$ sudo ovs-vsctl clear Bridge br-int mirrors
$ sudo ovs-vsctl del-port br-int tap-debug
```

### 20.4.6.5 Mirror traffic to a VLAN (RSPAN)

```shell
# Mirror traffic from a specific port out to VLAN 999 (RSPAN)
$ sudo ovs-vsctl \
    -- --id=@m create Mirror name=rspan-mirror \
         select-src-port=$(sudo ovs-vsctl get Port tap-vm01 _uuid) \
         output-vlan=999 \
    -- set Bridge br-int mirrors=@m
```

## 20.4.7 Category 7, conntrack and stateful

Cross-reference Part 20.3 §20.3.8 for the OVN-specific view. This part covers pure OVS.

### 20.4.7.1 Enable conntrack in an OpenFlow rule


## 20.4.11 End-to-end workflow 1, new-bridge provisioning with an overlay tunnel

Scenario: set up a new bridge `br-lab` for a test environment with two physical ports plus one Geneve tunnel plus QoS egress shaping plus the Faucet OpenFlow controller.

```shell
#!/bin/bash
# new-bridge.sh br-lab 10.0.0.12 10.0.0.100:6653 1000000000

BRIDGE=$1
TUNNEL_REMOTE=$2
CONTROLLER=$3
MAX_RATE=$4

# 1. Create the bridge
sudo ovs-vsctl add-br $BRIDGE

# 2. fail-mode = standalone (fall back to hub mode when the controller crashes)
sudo ovs-vsctl set-fail-mode $BRIDGE standalone

# 3. Add a physical port
sudo ovs-vsctl add-port $BRIDGE eth1

# 4. Add an internal port for the host IP
sudo ovs-vsctl add-port $BRIDGE bridge-iface \
    -- set Interface bridge-iface type=internal
sudo ip addr add 192.168.99.1/24 dev bridge-iface
sudo ip link set bridge-iface up

# 5. Add a Geneve tunnel
sudo ovs-vsctl add-port $BRIDGE tun-remote \
    -- set Interface tun-remote type=geneve \
       options:remote_ip=$TUNNEL_REMOTE \
       options:key=flow \
       options:csum=true

# 6. Set up linux-htb QoS with egress shaping
sudo ovs-vsctl set Port eth1 \
    qos=@newqos \
    -- --id=@newqos create QoS \
       type=linux-htb \
       other-config:max-rate=$MAX_RATE \
       queues=0=@default \
    -- --id=@default create Queue \
       other-config:min-rate=500000000 \
       other-config:max-rate=$MAX_RATE

# 7. Set the controller (if specified)
if [ -n "$CONTROLLER" ]; then
    sudo ovs-vsctl set-controller $BRIDGE tcp:$CONTROLLER
fi

# 8. Verify
echo "=== Bridge $BRIDGE provisioned ==="
sudo ovs-vsctl show | grep -A 20 "Bridge $BRIDGE"
sudo ovs-ofctl show $BRIDGE | head -5

# 9. Validate that tunnel ARP has resolved
sleep 3
sudo ovs-appctl tnl/neigh/show | grep $TUNNEL_REMOTE || echo "WARN: tunnel ARP has not resolved"
```

## 20.4.12 End-to-end workflow 2, safe bridge decommission

Scenario: remove `br-lab` safely. The order matters.

```shell
#!/bin/bash
# bridge-decommission.sh br-lab

BRIDGE=$1

# 1. Confirm no VM is still using it
for tap in $(sudo ovs-vsctl list-ports $BRIDGE | grep "^tap-"); do
    echo "ERROR: $BRIDGE still has tap $tap. Migrate the VM first."
    exit 1
done

# 2. Drain tunnel connections
for tun in $(sudo ovs-vsctl list-ports $BRIDGE | grep -E "^tun-|^vxlan-|^gre-"); do
    echo "Removing tunnel $tun..."
    sudo ovs-vsctl del-port $BRIDGE $tun
done

# 3. Flush OpenFlow rules (safer than implicit cascade)
sudo ovs-ofctl del-flows $BRIDGE

# 4. Remove QoS bindings
for port in $(sudo ovs-vsctl list-ports $BRIDGE); do
    sudo ovs-vsctl clear Port $port qos
done

# 5. Remove controller binding
sudo ovs-vsctl del-controller $BRIDGE

# 6. Remove remaining ports
for port in $(sudo ovs-vsctl list-ports $BRIDGE); do
    sudo ovs-vsctl del-port $BRIDGE $port
done

# 7. Remove the bridge
sudo ovs-vsctl del-br $BRIDGE

# 8. Garbage-collect orphan QoS and Queue records (OVS does not auto-GC)
for qos_uuid in $(sudo ovs-vsctl --bare --columns=_uuid list QoS); do
    refs=$(sudo ovs-vsctl --bare --columns=qos find Port qos=$qos_uuid | wc -l)
    if [ $refs -eq 0 ]; then
        sudo ovs-vsctl destroy QoS $qos_uuid
    fi
done

# 9. Verify clean removal
sudo ovs-vsctl show | grep -q $BRIDGE && echo "WARN: residue" || echo "OK $BRIDGE removed"
```

## 20.4.16 Capstone POE, "Migrate br-int from kernel to DPDK on a live chassis: safe?"

**Scenario:** The manager proposes migrating bridge `br-int` on a production hypervisor from the kernel datapath to DPDK userspace to improve real bandwidth. Live migration (no VM downtime). Is it safe?

**Predict:**
1. DPDK requires hugepages, CPU pinning, and a NIC PMD driver.
2. The kernel `br-int` has a tap (virtio-net), and DPDK `br-int` needs vhost-user (virtio-user).
3. Is the migration atomic?

**Observe (reasoning):**

**1.** Hugepage setup is required:

```bash
$ grep Huge /proc/meminfo
HugePages_Total:     0
HugePages_Free:      0
```

DPDK requires hugepages allocated before runtime. Runtime allocation is not possible without a reboot or a dedicated pool.

**2.** Tap and vhost-user are incompatible:

The kernel `br-int` has port `tap-vm01` (a tap device). The VM's virtio-net backend is QEMU's `/dev/net/tun`.

The DPDK `br-int` needs a `dpdkvhostuser` port. The VM backend must be a QEMU vhost-user socket, which is different from a tap.

This means the **VM has to restart** to switch backends. Live migration is not possible.

**3.** A flow copy is not enough:

```bash
# Copy flows from the kernel bridge to the DPDK bridge
$ sudo ovs-ofctl dump-flows br-int > /tmp/flows.txt
$ sudo ovs-ofctl add-flows br-int-dpdk /tmp/flows.txt
```

The flow copy works, but the action references a different `ofport`. `tap-vm01` is `ofport 3` on `br-int`, while `vhost-user-1` is `ofport 3` on `br-int-dpdk`. The same number 3 maps to different backends.

**4.** `datapath_type` cannot hot-switch:

```bash
# Try changing datapath_type at runtime
$ sudo ovs-vsctl set Bridge br-int datapath_type=netdev
# Requires full bridge tear-down and re-creation
```

**Explain:** The proposal is **not safe** for live migration. The correct procedure is:

**Option A, parallel bridge (zero VM downtime):**

1. Create `br-int-dpdk` in parallel (it does not affect `br-int`).
2. Set up DPDK infrastructure (hugepages plus CPU pinning).
3. VM by VM, migrate: stop the VM, switch the backend, restart the VM on `br-int-dpdk`.
4. When all VMs are migrated, delete the old `br-int`.

**Option B, full downtime (simpler):**

1. Schedule a maintenance window.
2. Migrate all VMs off the chassis (chassis cordon, Part 20.3 §20.3.10.3).
3. Stop `ovs-vswitchd`, tear down `br-int`, and reconfigure DPDK.
4. Recreate `br-int` with `datapath_type=netdev`.
5. Migrate VMs back.

**Finish:** Datapath migration is an **architectural change**, not a runtime configuration change. The VM backend has to be compatible. Always use the parallel bridge approach (A) or a maintenance window (B). Never hot-switch.

## 20.4.17 Common misconceptions

> **Misconception 1:** "`ovs-vsctl` and `ovs-ofctl` are interchangeable."
> **Reality:** `ovs-vsctl` is the OVSDB configuration layer (bridge, port, QoS, mirror). `ovs-ofctl` is the OpenFlow control layer (flow rule CRUD). Different layers, different schemas, different CLI syntax.

> **Misconception 2:** "`ovs-dpctl dump-flows` and `ovs-ofctl dump-flows` produce the same output."
> **Reality:** `ovs-dpctl` shows kernel datapath megaflows (what actually executed). `ovs-ofctl` shows OpenFlow rules (what the operator installed). The ratio is typically one OpenFlow rule expanding to three to five datapath flows (through wildcarding).

> **Misconception 3:** "`fail-mode=secure` is safer than `standalone`, so always use secure."
> **Reality:** `secure` drops new flows when the controller disconnects, so a poor controller HA setup means a 100 percent outage. `standalone` is hub mode (less secure but traffic continues). The choice depends on controller HA and security posture.

> **Misconception 4:** "`key=flow` in tunnel options means the tunnel_key is empty."
> **Reality:** `key=flow` means a dynamic `tunnel_key` (an OpenFlow rule sets it per packet through `set_tunnel:`). The opposite is `key=N`, a static key. It is not empty.

> **Misconception 5:** "Once QoS is configured, it takes effect immediately."
> **Reality:** QoS is bound to a Port, but the Queue ID has to match an OpenFlow `set_queue:<id>` action. Without a `set_queue` rule, traffic falls into the default queue (not shaped).

> **Misconception 6:** "`emer-reset` clears only the current bridge."
> **Reality:** It clears all configuration (bridges, ports, controller, QoS, and so on). It is a nuclear option with no undo.

> **Misconception 7:** "Upgrading OVS migrates the configuration automatically."
> **Reality:** An OVS upgrade preserves `conf.db` schema compatibility. A downgrade or schema-version mismatch requires `ovsdb-tool convert`. Test on one chassis before rolling.

> **Misconception 8:** "DPDK performance is always better than kernel."
> **Reality:** DPDK requires hugepages, CPU pinning, and a PMD thread. If the infrastructure is not tuned, DPDK can be slower than the kernel datapath. Benchmark in practice before migrating (§20.4.16 capstone).

## 20.4.18 Key points to remember

First, **the four CLI layers are clearly separated**: `ovs-vsctl` (OVSDB config), `ovs-ofctl` (OpenFlow rules), `ovs-dpctl` (datapath), and `ovs-appctl` (runtime RPC). Knowing which layer applies to which task is the mark of a competent operator.

Second, **the 10 daily task categories** cover 95 percent of the OVS workflow: health, inventory, bridge and port, flow, tunnel, QoS and mirror, conntrack, performance, OVSDB, backup. Mastering all 10 makes a fluent operator.

Third, **a five-command health check in less than 10 seconds** (morning routine): `ovs-vsctl show` plus `systemctl status` plus `ovs-dpctl show` plus `upcall/show` plus `dump-aggregate` flow count. Automate it with `ovs-health-check.sh`.

Fourth, **tunnel types have trade-offs**: Geneve (OVN native, 58-byte overhead, full TLV extensibility), VXLAN (high interoperability, 50 bytes, legacy-friendly), GRE (simple, no encryption, lab use). Choose the right one for the workload.

Fifth, **QoS needs three components working together**: `Port.qos` binding, the Queue definition, and the OpenFlow `set_queue` action. Missing any one component and QoS is not effective.

Sixth, **`replace-flows` is atomic.** Use it instead of sequential `del` plus `add`. This matters most for hot-reloading the flow table in production.

Seventh, **compact OVSDB on a schedule** (once per week). The transaction log is append-only and does not shrink. Compaction keeps `conf.db` small.

Eighth, **DPDK migration is not live.** The VM backend has to switch (tap to vhost-user), which requires a VM restart or a parallel bridge.

Ninth, **the `fail-mode` choice**: `secure` (with a controller HA cluster) versus `standalone` (edge or branch with no controller). There is no universal answer.

Tenth, **end-to-end scripts** (`new-bridge.sh`, `bridge-decommission.sh`, `ovs-health-check.sh`, `rolling-upgrade.sh`) are valuable deliverables. A checklist becomes a script, the script becomes CI, and CI becomes an SLA.

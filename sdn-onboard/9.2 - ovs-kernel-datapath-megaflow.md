## Context: the problem the kernel datapath must solve

Consider a KVM host running 50 VMs, where each VM has 20 concurrent TCP connections. The total is 1000 TCP microflows. Each microflow is the tuple (`src_ip`, `dst_ip`, `src_port`, `dst_port`, protocol), the classic 5-tuple. The kernel datapath must decide the forwarding action for every packet at a rate of millions of packets per second.

The naive approach, where a userspace daemon decides every packet, is not viable. The kernel-to-userspace context switch costs about 2 microseconds, which limits real throughput to roughly 500K packets per second.

The opposite approach, putting the full OpenFlow pipeline inside the kernel, is also not viable. The pipeline is complex: classifier, prioritized rules, and arbitrary header modification. Kernel code should not carry logic of this kind.

The OVS solution is this. **The kernel keeps a cache of decisions that have already been computed**, while userspace computes new decisions only when the cache misses. The cache must be fast (nanosecond scale), correct (no policy violation), and memory-efficient.

---

## 9.2.1 Microflow cache: the OVS 1.x design and its limit

OVS 1.x (2011 to 2013) used a **microflow cache**. Each entry described one exact 5-tuple, with no wildcards. Lookup was O(1) through a hash table.

The advantage is that hits are extremely fast. A lookup needs only a hash computation and a full-key comparison.

The fatal drawback is that **every microflow that differs in any field needs its own entry**. For example:

- TCP connection 1: `src=10.0.0.1:50001` to `dst=10.0.0.2:80`, entry A.
- TCP connection 2: `src=10.0.0.1:50002` to `dst=10.0.0.2:80`, entry B (different `src_port`).
- TCP connection 3: `src=10.0.0.1:50003` to `dst=10.0.0.2:80`, entry C.

Even though all three connections hit the same OpenFlow rule ("allow TCP `dst=10.0.0.2:80`") and the same action (`output:5`), each one gets its own cache entry that differs only in `src_port`. With 1000 connections, the cache has 1000 entries.

The result is cache thrashing. The cache has a fixed size. When the number of microflows exceeds that size, old entries are evicted. The next packet for an evicted flow misses, triggers an upcall, reinstalls, and the cycle repeats.

NSDI 2015 measured a Rackspace production workload (2013, before megaflow) and reported a miss rate of 10 to 30 percent at peak. The overhead was too high to scale.

## 9.2.2 Megaflow cache: the NSDI 2015 breakthrough

A **megaflow** is a cache entry with **wildcards**. It matches a set of microflows that share the same header pattern. Instead of 1000 entries for 1000 TCP connections to port 80, a single entry `dst=10.0.0.2, dst_port=80, proto=TCP, wildcard src_ip+src_port` covers all of them.

The typical compression factor is 10 to 100. A workload of 100K microflows compresses into a cache of 2,000 to 10,000 entries.

**The key question is this: how does OVS decide which field to wildcard?** OVS answers with **lazy wildcarding**. When the userspace classifier processes a microflow for the first time, it does the following:

1. Parses the packet into the 5-tuple and other fields.
2. Walks the OpenFlow pipeline: match rules in table 0, then table 1, and so on, then execute the action.
3. While walking, **tracks** which fields have been inspected.
4. Marks any field that was NOT inspected as a wildcard.
5. Compiles a megaflow entry that wildcards every uninspected field.

For example, given the rule "match `dst_ip=10.0.0.2, dst_port=80, proto=TCP`, action `output:5`" and a packet with `src_ip=10.0.0.1, src_port=50001`:

- The classifier inspects `dst_ip` (match), `dst_port` (match), and `proto` (match).
- The classifier does NOT inspect `src_ip` or `src_port`.

The resulting megaflow entry matches `dst_ip=10.0.0.2, dst_port=80, proto=TCP` and wildcards `src_ip=*, src_port=*`.

**Correctness guarantee.** Lazy wildcarding never breaks any rule. If a different rule constrains `src_port`, the classifier inspects `src_port` while walking and does not wildcard it. The result is correct by construction.

## 9.2.3 Tuple Space Search for megaflow lookup

When there are many megaflows with different masks, each megaflow has its own **tuple**, which is the set of fields that are matched (not wildcarded). OVS uses **Tuple Space Search (TSS)**, an algorithm from Srinivasan et al. SIGCOMM 1998. The algorithm groups entries by tuple, and each tuple has its own hash table.

Per-packet lookup works like this: for each tuple, extract the fields per the mask, compute the hash, and look up the hash table. On a match, return the action. Otherwise, try the next tuple.

The complexity is O(T x H), where T is the number of tuples (typically fewer than 20) and H is the cost of one hash lookup (O(1) on average).

> **Key point:** Choosing TSS over TCAM or a decision tree is a pragmatic decision. TCAM is too expensive on a commodity server. A decision tree has a high build cost when rules change frequently. TSS is the best compromise for a software datapath. The NSDI 2015 benchmark shows TSS reaching more than 10M lookups per second per core on an Intel Xeon.

## 9.2.4 Upcall: slow path and fast path

USC Lab 9 (`doc/ovs/Day 4-lab 9 - Open vSwitch Kernel Datapath.txt`) states:

> "Kernel module directs the packet to the userspace. The userspace makes the decisions about the actions to be taken against the packet according to OpenFlow entries (slow path)."
>
> "The action entry is stored in the kernel and used to forward subsequent packet which makes the forwarding faster (fast path)."

**Fast path (cache hit):**

1. The packet arrives at the NIC, goes through the kernel stack, and enters the `openvswitch.ko` rx hook.
2. The kernel extracts the flow key from the headers.
3. The kernel looks up the megaflow cache through TSS.
4. On match, it executes the action (`output:5` calls `netdev_ops->ndo_start_xmit`).
5. The kernel updates counters. Userspace is never touched. The latency is about 1 to 5 microseconds.

**Slow path (cache miss):**

1. A packet arrives and the lookup misses.
2. The kernel sends an **upcall** through Netlink to userspace.
3. The upcall carries the packet bytes, metadata (`in_port`, timestamp), and reason.
4. An `ovs-vswitchd` handler thread receives the upcall.
5. The handler walks the OpenFlow pipeline and tracks the inspected fields.
6. It compiles a megaflow entry and an action list.
7. It sends `OVS_FLOW_CMD_NEW` back to the kernel to install the megaflow.
8. It sends `OVS_PACKET_CMD_EXECUTE` so the kernel executes the action for this packet.
9. Subsequent similar packets then hit the fast path.

USC Lab 9 puts it this way: the first packet of a flow finds that "the kernel table is empty" and so it goes through the slow path. After "action entries are installed in the kernel", subsequent packets follow the fast path.

The slow-path cost is about 50 to 200 microseconds per packet. When the hit rate is low, the slow path dominates real throughput.

## 9.2.5 Ukeys and revalidator threads

When a megaflow is installed into the kernel, `ovs-vswitchd` keeps a **ukey (unique key)**, a userspace copy that holds the flow key, the mask, the action, the last-known statistics counter, and the `last_used` timestamp.

**Revalidation.** When rules change, the revalidator threads (4 by default, with a 500 ms interval) scan the ukey set:

1. For each ukey, re-walk the pipeline with the flow key.
2. Compare the new action with the old action.
3. If they differ, send `OVS_FLOW_CMD_SET` to update the kernel.
4. If no rule matches, send `OVS_FLOW_CMD_DEL`.

**Timeout.** Rules carry `idle_timeout` and `hard_timeout`. The revalidator tracks the timestamp and deletes the entry on timeout.

The thread count is configured by `other_config:n-revalidator-threads`. A large cache means a CPU-heavy scan, so raise to 8 or 16 threads when needed.

## 9.2.6 Upcall rate limit

Compass Ch Q describes the tuning knobs:

- `upcall/show` reports the number of in-flight upcalls, the average, the maximum, and the limit (200,000 flow-installs per second by default).
- `upcall/set-flow-limit 500000` raises the limit for high-scale workloads.

Hitting the limit causes the kernel to drop packets, and the `lost` counter in `ovs-dpctl show` rises.

The relationships are:

- `hit` grows linearly: normal.
- `missed` grows: the cache is not hitting well.
- `lost` greater than zero: the rate limit is being hit and traffic is at risk.

Tune `other_config:n-handler-threads` (2 by default). Raise it when miss rate is high but `lost` is zero, because that indicates the handlers are bottlenecked.

## 9.2.7 Cache hit rate in production

NSDI 2015 reported the following from Rackspace production:

- Before megaflow: 70 to 90 percent hit rate, with 10 to 30 percent miss at peak.
- After megaflow: 99 percent or higher hit rate, with miss below 1 percent.

At 1M packets per second, that gives about 10,000 upcalls per second, which is within the capacity of the handler threads.

**Early warning signals:**

- Coverage miss rate above 1 percent: the compression is not effective.
- `ovs-dpctl show` shows `lost` greater than zero: the rate limit is being hit.
- `upcall/show` average is high: the handlers are bottlenecked.

**Scenarios where the hit rate drops:**

1. Highly diverse traffic, where every microflow differs in the fields that the rules match on.
2. Frequently changing rules, which causes the revalidator to invalidate many ukeys.
3. A small cache or a low `max-idle` value, which evicts entries early.

## 9.2.8 EMC, the Exact Match Cache: the first tier in datapath lookup

Up to §9.2.7 we have discussed the megaflow cache as if it were the only cache layer. In practice, **the OVS 2.x kernel datapath has three cache tiers stacked on top of one another**, queried in this order: EMC, then SMC, then the megaflow classifier (TSS). Understanding all three tiers makes it possible to read `pmd-stats-show` correctly and to diagnose the `miss with success upcall` counter (Part 9.11 §9.11.8).

The **Exact Match Cache (EMC)** is the first tier. It is an exact-match hash table keyed by the packet 5-tuple (or a wider key), with no wildcards. Each EMC entry corresponds to one specific **microflow**.

### 9.2.8.1 EMC structure

| Property | Value |
|---|---|
| **Scope** | Per PMD thread (or per CPU core for the kernel datapath), not shared |
| **Lookup** | O(1) hash table, no wildcards |
| **Key** | Full packet header fields (both `src` and `dst`, TCP ports, and so on) |
| **Capacity** | 8192 entries per PMD by default (compile-time constant `EM_FLOW_HASH_ENTRIES`) |
| **Insertion** | After a successful lookup at a lower tier (SMC or megaflow) |
| **Eviction** | LRU-like, with overwrite on hash collision |
| **Purpose** | Fast path for hot flows: a hot packet is funneled into EMC and avoids re-walking TSS |

EMC exists in both the kernel datapath (`openvswitch.ko`) and the userspace DPDK datapath (`dpif-netdev`). The kernel implementation differs from userspace in detail but the semantics are the same: an exact-match hash for the fast path.

### 9.2.8.2 Reading EMC stats from `pmd-stats-show`

**Context.** This applies only to `datapath_type=netdev` (DPDK and AF_XDP). The kernel datapath does not expose EMC stats directly; you must read the `coverage/show` counters `emc_hit` and `emc_lookup`.

**Command and output** `[reproduced from OVS DPDK benchmark setup]`:

```shell
$ sudo ovs-appctl dpif-netdev/pmd-stats-show
pmd thread numa_id 0 core_id 2:
  packets received: 10000000
  packet recirculations: 12543
  avg. datapath passes per packet: 1.00
  emc hits: 9050231
  smc hits: 820412
  megaflow hits: 129303
  avg. subtable lookups per megaflow hit: 2.3
  miss with success upcall: 54
  miss with failed upcall: 0
  avg. packets per output batch: 12.4
```

**Anatomy of each EMC-related counter:**

| Counter | Meaning | Healthy |
|---|---|---|
| `emc hits` | Packets that matched the EMC tier | At least 85 percent of `packets received` for a stable workload |
| `smc hits` | Packets that missed EMC and hit the SMC tier | 5 to 15 percent |
| `megaflow hits` | Packets that missed both EMC and SMC and hit the megaflow classifier | 1 to 5 percent |
| `miss with success upcall` | Packets that missed megaflow and an upcall successfully installed a flow | Less than 0.1 percent in steady state |
| `miss with failed upcall` | Upcalls that failed (OVSDB lock, error) | Must be zero |

**Failure scenarios:**

- **EMC hit below 50 percent on a workload expected to be stable.** This is EMC thrashing. 8192 entries are not enough for a workload with more than 8000 concurrent distinct microflows. The fix is to enable SMC (on by default in OVS 2.15 and later), or to accept the shift down to megaflow hits as the new top tier.
- **SMC hit plus megaflow hit greater than 50 percent total.** EMC is bypassed or disabled. Check `ovs-appctl dpif-netdev/pmd-perf-show` for details.
- **`miss with success upcall` rising steadily.** New flows keep arriving. This indicates a scanner workload or a malicious traffic pattern. Check the OpenFlow classifier flow limit (Part 9.2.6) and the `coverage/show upcall_flow_add` counter.

### 9.2.8.3 When to disable EMC

EMC consumes cache memory on the PMD core. For a very low hit rate workload, where every packet is unique, EMC becomes pure overhead. Disable it like this:

```shell
$ sudo ovs-vsctl set Open_vSwitch . other_config:emc-insert-inv-prob=0
# Restart to apply; set greater than 0 to re-enable (smaller value = more insertion)
```

The value `emc-insert-inv-prob=N` is the inverse insertion probability, meaning the chance of insertion is 1 in N. The default is 100, which gives a 1 percent insertion rate. A specialized setup (a security gateway scanning every packet) can set 1000 or disable EMC entirely.

## 9.2.9 SMC, the Signature Match Cache: the middle tier in OVS 2.15 and later

SMC was added in OVS 2.15 (February 2021) to close the performance gap between EMC (exact microflow, 8K entries) and the megaflow classifier (TSS, with the higher O(T x H) cost). The NSDI 2020 paper *HXDP* and the OVS mailing list discuss SMC as an optimization for workloads with more than 100k concurrent microflows, where EMC alone is not sufficient.

### 9.2.9.1 SMC structure

| Property | Value |
|---|---|
| **Scope** | Per PMD thread (same as EMC) |
| **Lookup** | O(1) hash table, compact signature-based |
| **Key** | 16-bit signature of the flow hash (not the full 5-tuple) |
| **Value** | A pointer to the megaflow entry (not the action directly) |
| **Capacity** | 16384 entries per PMD (compile-time) |
| **Insertion** | After a successful lookup at the megaflow classifier |
| **Purpose** | Reduces TSS lookup cost for "moderately cold" flows (not hot enough for EMC but still regular) |

SMC **does not contain the full action**. It contains a pointer to a megaflow entry. The lookup flow is: compute the signature hash, do an SMC lookup, on hit dereference the pointer, and execute the megaflow action. The dereference step is faster than re-walking TSS because it skips the whole subtable scan.

### 9.2.9.2 The EMC, SMC, and megaflow lookup path

The lookup order in `dpif-netdev` (OVS 2.15 and later) is:

```
Packet arrives at PMD
  ↓
(1) EMC lookup: O(1) hash, full 5-tuple exact
    ├── HIT → execute action, done (~50 ns)
    └── MISS → (2)
  ↓
(2) SMC lookup: O(1) hash, 16-bit signature
    ├── HIT → follow pointer to megaflow, execute (~150 ns)
    └── MISS → (3)
  ↓
(3) Megaflow classifier TSS: O(T) subtable scan
    ├── HIT → execute action, insert into SMC (+optional EMC) (~300-500 ns)
    └── MISS → (4)
  ↓
(4) Upcall to userspace
    → handler walks OpenFlow pipeline
    → compile megaflow
    → install into kernel classifier
    → (~50-200 µs)
```

The numbers come from NSDI 2020 and OVS benchmarks. The three-tier design lets you trade memory per entry against lookup cost:
- **EMC:** small (8K entries), extremely fast (50 ns), for the hottest flows only.
- **SMC:** medium (16K entries), moderately fast (150 ns), for regular flows.
- **Megaflow:** large (200K entries by default), slower (300 to 500 ns) but wildcard-capable, covering the bulk of the workload.

### 9.2.9.3 SMC tuning

SMC is on by default. Disable it with `other_config:smc-enable=false` for troubleshooting or benchmarking.

In current OVS (3.x) there is no runtime option for SMC size. Recompile OVS with `#define SMC_ENTRIES N` if you need to change it. Production rarely needs adjustment.

## 9.2.10 The upcall Netlink protocol: the message anatomy between kernel and userspace

When §9.2.4 introduced upcalls, it did not examine the **message wire format** between kernel and userspace. Upcalls travel over **Generic Netlink** (genl), the Linux mechanism for a kernel subsystem to expose an API to userspace. OVS registers the genl families `ovs_vport`, `ovs_flow`, `ovs_datapath`, and `ovs_packet` with the kernel.

Understanding the wire format helps when debugging: `vlog/set dpif:file:dbg` dumps Netlink headers that are otherwise hard to interpret.

### 9.2.10.1 Generic Netlink header structure

```
+-----------------------------------------------+
|           Netlink header (16 byte)             |
|  ┌─────────────────────────────────────────┐  |
|  │ nlmsghdr {                              │  |
|  │   __u32 nlmsg_len   (total message len) │  |
|  │   __u16 nlmsg_type  (family ID, dynamic │  |
|  │                      for ovs_packet)    │  |
|  │   __u16 nlmsg_flags (NLM_F_REQUEST, ...) │  |
|  │   __u32 nlmsg_seq   (sequence #)         │  |
|  │   __u32 nlmsg_pid   (source PID)         │  |
|  │ }                                        │  |
|  └─────────────────────────────────────────┘  |
+-----------------------------------------------+
|       Generic Netlink header (4 byte)          |
|  ┌─────────────────────────────────────────┐  |
|  │ genlmsghdr {                            │  |
|  │   __u8 cmd    (OVS_PACKET_CMD_MISS,      │  |
|  │                OVS_PACKET_CMD_EXECUTE, ..) │  |
|  │   __u8 version                            │  |
|  │   __u16 reserved                          │  |
|  │ }                                        │  |
|  └─────────────────────────────────────────┘  |
+-----------------------------------------------+
|       OVS attribute TLV list (variable)        |
|  ┌─────────────────────────────────────────┐  |
|  │ nlattr {                                │  |
|  │   __u16 nla_len                         │  |
|  │   __u16 nla_type  (OVS_PACKET_ATTR_*)    │  |
|  │ }                                        │  |
|  │ + payload                                │  |
|  │ ... (multiple attributes)                │  |
|  └─────────────────────────────────────────┘  |
+-----------------------------------------------+
```

### 9.2.10.2 OVS_PACKET_CMD_MISS upcall anatomy

When the kernel misses the cache, it sends an upcall message with:

| Attribute | Type | Meaning |
|---|---|---|
| `OVS_PACKET_ATTR_PACKET` | raw bytes | Full packet data (Ethernet frame) |
| `OVS_PACKET_ATTR_KEY` | nested TLV | Extracted flow key (`tun_id`, `in_port`, `eth_src/dst`, `nw_src/dst`, and so on) |
| `OVS_PACKET_ATTR_USERDATA` | raw | Optional context (when the packet went through a `userspace` action) |
| `OVS_PACKET_ATTR_EGRESS_TUN_KEY` | nested | Tunnel metadata for egress |
| `OVS_PACKET_ATTR_MRU` | u16 | Maximum Receive Unit for fragmentation handling |

**Debugging the upcall wire format:**

```shell
# Enable DPIF debug log (prints Netlink messages in detail)
$ sudo ovs-appctl vlog/set dpif:file:dbg
$ sudo tail -f /var/log/openvswitch/ovs-vswitchd.log

# Trigger a miss: ping an IP that the datapath has never seen
$ ping -c 1 <fresh_ip>

# Log sample:
2026-04-24T11:15:23.412Z|00123|dpif(handler1)|DBG|system@ovs-system: \
  miss upcall:
  recirc_id(0),in_port(3),eth(src=aa:bb:cc:00:00:01,dst=aa:bb:cc:00:00:02),\
  eth_type(0x0800),ipv4(src=10.1.0.5,dst=10.1.0.10,proto=1,frag=no), \
  icmp(type=8,code=0)
2026-04-24T11:15:23.413Z|00124|dpif(handler1)|DBG|system@ovs-system: \
  execute: actions:4 packets:1 (86B)
```

**Anatomy of each log field:**

| Part | Meaning |
|---|---|
| `|dpif(handler1)|` | Log subsystem and thread name |
| `|DBG|` | Log level (turned on by `vlog/set dpif:file:dbg`) |
| `system@ovs-system:` | Datapath name |
| `miss upcall:` | Upcall reason: cache miss |
| `recirc_id(0)` | Recirculation ID is 0; the packet has not gone through `ct()` or conjunction |
| `in_port(3)` | Ingress datapath port |
| `eth(src=..., dst=...)` | L2 extract |
| `eth_type(0x0800)` | EtherType is IPv4 |
| `ipv4(src=..., dst=..., proto=1, frag=no)` | L3 extract, `proto=1` is ICMP |
| `icmp(type=8, code=0)` | ICMP Echo Request |
| `execute: actions:4` | Handler reply: execute action `output:4` |

### 9.2.10.3 OVS_FLOW_CMD_NEW: installing the megaflow back to the kernel

After the upcall handler compiles a megaflow, it sends `OVS_FLOW_CMD_NEW` back to the kernel:

| Attribute | Meaning |
|---|---|
| `OVS_FLOW_ATTR_KEY` | Masked flow key (the match fields that need exact comparison) |
| `OVS_FLOW_ATTR_MASK` | Mask for each field (0 means wildcard, all-1 means exact) |
| `OVS_FLOW_ATTR_UFID` | Unique Flow ID (UUID128, OVS 2.7 and later) |
| `OVS_FLOW_ATTR_ACTIONS` | Datapath action list (`output`, `ct`, `set`, `recirc`, and so on) |
| `OVS_FLOW_ATTR_STATS` | Initial stats (N = 0) |

The kernel applies the change atomically through RCU protection: lookup reads do not block install writes.

## 9.2.11 Ukey lifecycle and revalidator RCU

§9.2.5 introduced ukeys and revalidators at a high level. This section goes into the ukey state machine and the synchronization guarantee that the revalidator provides.

### 9.2.11.1 State machine of a single ukey

Each ukey (`struct udpif_key` in `ofproto/ofproto-dpif-upcall.c`) goes through the following lifecycle:

```
           create
              ↓
        ┌──────────┐
        │ CREATED  │  (tentative insert, not yet applied to kernel)
        └──────────┘
              ↓  flow installed successfully
        ┌──────────┐
        │  VISIBLE │  (kernel holds the megaflow, ukey tracks counters)
        └──────────┘
              ↓  revalidator sweep sees the ukey
        ┌──────────┐
        │ OPERATIONAL │  (steady state, stats updated each sweep)
        └──────────┘
              ↓  rule change, re-walk pipeline
        ┌──────────┐
        │ CHECKING │  (revalidator verifies the action is still correct)
        └──────────┘
         ↓ same action          ↓ action differs / no match
     (back to OPERATIONAL)    ┌────────────┐
                              │ EVICTING   │  (OVS_FLOW_CMD_DEL)
                              └────────────┘
                                    ↓
                              ┌────────────┐
                              │ DELETED    │  (memory freed after RCU grace)
                              └────────────┘
```

### 9.2.11.2 Revalidator RCU guarantee

The revalidator threads (4 by default, controlled by `n-revalidator-threads`) run periodically (500 ms by default). Each sweep does:

1. **Snapshot phase (RCU read-side):** the thread acquires the RCU read lock and iterates the ukey list.
2. **Per-ukey check:** for each ukey, re-walk the OpenFlow pipeline with the flow key.
3. **Compare action:** compare the new pipeline result with the old action stored in the ukey.
4. **Write phase:** if the action differs, mark the ukey `EVICTING` and send `OVS_FLOW_CMD_DEL` to the kernel.
5. **Grace period:** the ukey is not freed immediately. The thread waits for the RCU grace period (about 25 ms) to make sure no other thread is still reading this ukey.

RCU (Read-Copy-Update) is the Linux primitive that lets reads happen without blocking writes. The revalidator uses RCU so that the dispatcher thread (which handles upcalls) can read ukeys while the revalidator updates them.

### 9.2.11.3 Reading `revalidator/wait` output

**Context.** When debugging a race condition between OpenFlow programming and datapath state, run the `revalidator/wait` barrier to make sure the revalidator has processed the current state.

```shell
$ sudo ovs-appctl dpctl/dump-flows | wc -l
1247
$ sudo ovs-ofctl -O OpenFlow15 del-flows br-int
$ sudo ovs-appctl revalidator/wait
waited <N> ms
$ sudo ovs-appctl dpctl/dump-flows | wc -l
0
```

The `waited <N> ms` output shows how long the call blocked. More than 1000 ms means the revalidator is overloaded.

**Failure scenarios:**

- **`revalidator/wait` blocks for more than 5 seconds.** The revalidator threads have a backlog of ukeys to check. Check `upcall/show` for `dump duration` and `avg-updates` per thread. Raise `n-revalidator-threads`.
- **The `coverage/show facet_revalidate` rate is above 10k per second.** The revalidator is overloaded. This is the signature of the Part 9.26 revalidator storm.

### 9.2.11.4 Ukey count versus rule count

`memory/show udpif keys` shows the ukey count. Compare with `rules`:

```
rules:12847
udpif keys:1247
```

The ukey count is usually less than the rule count because many rules have never matched any packet, so no ukey was created. The ratio `ukey / rule` reflects coverage: a ukey is a flow that is currently active in the kernel.

**Pathology:** when `udpif keys` is greater than `rules`, you have a ukey leak (Part 9.26). Commit `180ab2fd635e` fixed exactly this bug.

## 9.2.12 Summary of the three cache tiers and a monitoring cheat sheet

### 9.2.12.1 Summary table

| Tier | Capacity | Lookup cost | Expected hit rate | Key type | Monitoring |
|---|---|---|---|---|---|
| EMC | 8K entries per PMD | About 50 ns | 85 to 95 percent | Exact microflow | `pmd-stats-show emc hits` |
| SMC | 16K entries per PMD | About 150 ns | 5 to 15 percent | Signature hash | `pmd-stats-show smc hits` |
| Megaflow (TSS) | 200K by default | 300 to 500 ns | 1 to 5 percent | Masked wildcard | `pmd-stats-show megaflow hits`, `dpctl/dump-flows` |
| Upcall | Not applicable | 50 to 200 microseconds | Less than 0.1 percent steady | Not applicable | `upcall/show`, `coverage/show upcall_*` |

### 9.2.12.2 Datapath health checklist

```
[ ] EMC hit > 85%                     (pmd-stats-show)
[ ] Total hit (EMC+SMC+megaflow) > 99%
[ ] `miss with success upcall` < 100/sec
[ ] `miss with failed upcall` = 0
[ ] `dump_flows_fallback` = 0         (coverage/show)
[ ] `lost` counter = 0                (ovs-dpctl show)
[ ] udpif keys <= rules               (memory/show)
[ ] dump duration < 500 ms            (upcall/show)
[ ] avg-updates per revalidator < 5   (upcall/show)
[ ] revalidate rate < 1/sec           (coverage/show)
```

A production script runs every 5 minutes, logs the deltas, and alerts when any metric crosses its threshold.

---

First, the megaflow cache is the most important architectural contribution of NSDI 2015. It moves from exact match to wildcard match through lazy wildcarding. The 10 to 100 times compression is what lets OVS scale to a cloud production workload.

Second, Tuple Space Search is the right algorithm for a software datapath. It has a good constant factor and supports dynamic insert and delete. The choice is pragmatic, driven by commodity hardware constraints.

Third, slow path plus fast path is the core mechanism. USC Lab 9 shows it clearly: the first packet goes the slow path, installs the action into the kernel, and subsequent packets go the fast path. Understanding the mechanism helps debug "the first packet is slow but later ones are fast" or "the missed counter is rising".

Fourth, a 99 percent or higher cache hit rate is the operational goal. Below that, you hit cache thrashing, handler bottleneck, and traffic loss risk. Monitor with `ovs-dpctl show`, `ovs-appctl coverage/show`, and `ovs-appctl upcall/show`.

## 9.2.14 Bổ sung chuyên sâu OVS datapath (mở rộng 2026-04)

> **Mục đích:** Bổ sung các phương diện còn thiếu cho 4 keyword cốt lõi đã substantively cover ở §9.2.1 đến §9.2.13: `megaflow`, `recirc_id`, `openvswitch.ko (kernel datapath)`, `conntrack execute`. Nhóm register keyword đã bổ sung ở Phần 13.18.

### 9.2.14.1 megaflow, các phương diện bổ sung

**Tầm quan trọng.** `megaflow` là **trụ cột tuyệt đối tier 1** của OVS scaling. Justification: trước OVS 2.0 (2014), kernel datapath chỉ có microflow per-5-tuple → production traffic 100K+ flow/sec gây flow explosion (kernel cache hàng triệu entry, classifier O(N) traversal, ovs-vswitchd CPU 100%). Megaflow consolidate qua wildcard mask reduce entry count 100-1000x. Engineer không hiểu megaflow = không debug được scaling issue, revalidator storm, cache miss spike. Tier 1 absolute, parity với `ct_state` về importance.

**Phân loại.** Loại: kernel datapath flow entry abstraction. Parent: `dp_netdev_flow` struct (kernel) hoặc `udpif_key` (revalidator userspace). Scope: per-bridge, persist trong kernel cho đến idle/hard timeout hoặc revalidator evict. Idempotency: re-install same flow = no effect (deduplicate). Type: data-plane caching, not control-plane state.

**Quy trình modify/tune:**

```
Step 1: Identify high cache miss rate
  $ ovs-appctl coverage/show | grep -E "miss|upcall"
  → upcall rate > 1000/sec = potential megaflow churn

Step 2: Inspect existing megaflow
  $ ovs-appctl dpctl/dump-flows | head -20
  → Look for flow with very narrow mask (close to microflow)

Step 3: Tune via OF rule design
  → Add conjunction action giảm cartesian flow rule
  → Use generic match (e.g., `ip` thay vì `ip,nw_src=10.0.0.5`)
    cho ACL không cần per-IP

Step 4: Force microflow (debug only)
  $ ovs-vsctl set Open_vSwitch . other_config:flow-limit=1000
  → Limit megaflow count để observe wildcard generalization

Step 5: Verify post-tune
  $ ovs-appctl coverage/show | grep upcall
  → upcall rate should drop after wildcard generalize works
```

**Khác biệt giữa các phiên bản:**

| Version | Behavior |
|---------|----------|
| Pre-OVS 2.0 | Microflow only, flow explosion problem |
| OVS 2.0 (2014) | Megaflow introduce, TSS classifier |
| OVS 2.4 | Subtable mask consolidation |
| OVS 2.6 | SMC (Signature Match Cache) thêm tier giữa EMC và megaflow |
| OVS 2.9 | Depth-wise lookup optimization |
| OVS 2.15 | SMC tunable via other_config |
| OVS 3.0+ | Lookup optimization continues, behavior stable |

Curriculum baseline: OVS v2.17.9. Production có thể chạy 3.3+; check version cho SMC tunable.

**So sánh với hệ khác.**

- **Linux bridge MAC learning:** wildcard concept (1 entry per MAC, not per-flow). Khác: bridge L2 only, không có L3/L4 mask. OVS megaflow flexible.
- **Cisco TCAM ACL aggregation:** TCAM cells aggregate similar pattern. Khác: hardware-level (fast, fixed), OVS megaflow software (slow, flexible).
- **Intel DPDK rte_flow API:** modern hardware offload similar wildcard. OVS hardware offload (xem Phần 9.5) kết hợp megaflow + rte_flow.

### 9.2.14.2 recirc_id, định danh recirculation cho multi-pass pipeline

> **Phạm vi:** Treatment chuyên sâu (mở rộng 2026-04-26) cho keyword `recirc_id`. Các phần liên quan: §9.2.10.2 (megaflow row có `recirc_id(0)` trong dump-flows), §9.22 multi-table pipeline (so sánh với `goto_table`/`resubmit`), §9.24 conntrack (`ct()` action sinh recirc), §9.32 classifier xlate.

**Khái niệm.** `recirc_id` là một metadata field 32-bit của OVS đánh dấu *round* recirculation mà gói đang đi qua trong datapath. Mỗi gói khi vừa nhận từ port (chưa qua bất kỳ `ct()`, `recirc()`, hoặc translation breakpoint nào) có `recirc_id = 0`. Khi xlate gặp tình huống không thể "compose" tiếp action set tĩnh trong một lần lookup duy nhất (ví dụ phải biết kết quả conntrack trước khi quyết định OF table tiếp theo), userspace `ofproto-dpif-xlate` gắn cho gói một `recirc_id` mới (uint32 khác 0, do `recirc_alloc_id()` cấp), đóng băng frozen state lại, và datapath sẽ "re-enter" classifier với key đã cập nhật `recirc_id = <id mới>`. Loại: pseudo-field metadata kernel datapath, NXM extension match field (NXM-only, không có OXM tương ứng), và là chìa khóa của mọi hành vi multi-pass trong OVS.

**Lịch sử + bối cảnh.** Trước OVS 2.2 (2014), kernel datapath không có cơ chế recirculation; mọi pipeline phải "linearize" thành một chuỗi action duy nhất. Khi conntrack được lên kế hoạch tích hợp vào OVS, vấn đề lập tức xuất hiện: kết quả `ct()` (NEW vs ESTABLISHED, NAT mapping) chỉ biết *sau khi* gói đi qua hook netfilter, nhưng OF rule cần *match trên kết quả đó* để quyết định forward hay drop. NXM_NX_RECIRC_ID(36) được introduce trong OVS 2.2 (commit pool nguồn `ofproto/ofproto-dpif-rid.c`, copyright Nicira 2014-2017) như một cơ chế chung để "bookmark" trạng thái dịch và resume từ một bảng OF khác sau khi gói đã được tái xử lý. OVS 2.5 (2016) chính thức đưa `ct()` vào kernel datapath, biến recirc_id thành nền tảng của toàn bộ stateful firewall pipeline. OVS 2.6+ mở rộng sang tunnel decap, push/pop MPLS, và `dec_ttl` exception path.

**Vị trí trong kiến trúc.** `recirc_id` nằm ở giao điểm của ba lớp: (1) classifier khoá megaflow theo `recirc_id` cùng với `in_port` và header fields; (2) `ofproto-dpif-xlate` userspace dịch OF flow gặp action không-thể-linearize sẽ `recirc_alloc_id()` và sinh frozen state; (3) datapath (kernel `net/openvswitch/actions.c` hoặc userspace `lib/dpif-netdev.c`) thực thi action `OVS_ACTION_ATTR_RECIRC` bằng cách set `key->recirc_id = N` rồi `ovs_execute_actions()` đệ quy. Megaflow trong `ovs-dpctl dump-flows` hiển thị `recirc_id(0x<hex>)` ngay đầu match list, vì đó là dimension đầu tiên TSS classifier phân nhánh.

**Vai trò.** Bốn vai trò cốt lõi: (a) **state continuation token**, gắn frozen translation state vào pool `ofproto-dpif-rid` để resume sau khi packet quay lại classifier; (b) **classifier partition**, mỗi `recirc_id` khác nhau tạo một "logical sub-table" độc lập, packet pass đầu (`recirc_id=0`) không bao giờ match flow của pass sau, ngăn cross-contamination; (c) **action chain breakpoint**, khi action set không thể compose tĩnh (vì phụ thuộc kết quả ct/decap), recirc tạo một "join point" mới; (d) **megaflow caching key**, `recirc_id` là một field trong key được cache, nên cùng một flow tuple ở pass-0 và pass-1 sinh ra hai megaflow entry độc lập, không bị aliasing.

**Vì sao sinh ra.** Động cơ thiết kế là giải bài toán "stateful match phụ thuộc trạng thái runtime" mà mô hình OpenFlow gốc không giải được. OF chuẩn cho phép `goto_table` linearize forward nhưng không có khái niệm "trở về classifier sau khi đã modify packet bằng kết quả ngoài flow table" (ví dụ kết quả nf_conntrack, kết quả tunnel decap thay đổi inner header). Nicira phát hiện rằng nếu để controller quản lý multi-pass thông qua barrier message + nhiều flow_mod, latency và state divergence sẽ phá hỏng fast path. Recirc_id là giải pháp dataplane-internal: controller cài static OF rule một lần, datapath tự động lặp lại classifier với metadata khác nhau cho từng pass.

**Vấn đề giải quyết.** Cụ thể recirc_id giải quyết bốn lớp vấn đề: (1) **Stateful firewall**, biết một gói thuộc connection ESTABLISHED hay NEW chỉ có thể sau khi conntrack lookup, mà conntrack lookup là hành động tốn time và có side-effect; phải recirc để match `ct_state` ở pass tiếp theo. (2) **Tunnel decapsulation**, khi gói VXLAN/Geneve được decap, inner header trở thành "gói mới"; recirc cho phép classifier phân loại lại theo inner header thay vì outer. (3) **MPLS/NSH push-pop**, thao tác trên label stack thay đổi packet structure; recirc tạo classifier checkpoint để xử lý phần sau đã thay đổi. (4) **`dp_hash`/`select_via_hash` group bucket**, kernel cần tính hash trước khi chọn bucket, recirc kết hợp với `recirc_update_dp_hash` đẩy việc chọn bucket vào pass tiếp theo.

**Tầm quan trọng.** Đây là khái niệm cốt lõi tuyệt đối của OVS internals. Engineer mà không nắm recirc_id sẽ không debug được: (a) bất kỳ pipeline nào dùng `ct()` (tức 100% production OVS có security group); (b) bất kỳ pipeline OVN nào (OVN logical pipeline thường tạo 5-15 lần recirc per packet do `lr_in_arp_resolve`, `ls_in_acl`, `ls_in_pre_acl`, `ls_in_pre_lb`, `ls_in_lb_aff_check`, etc.); (c) megaflow explosion debugging, vì một trong những anti-pattern phổ biến nhất là OF rule sinh recirc tổ hợp gây tăng entry theo cấp số nhân. Mức độ quan trọng ngang `ct_state`, `megaflow`, `OFPACT_OUTPUT` trong nhóm cốt lõi OVS.

**Cơ chế hoạt động.** Trình tự per-packet khi pipeline gặp action cần recirc (ví dụ `ct(zone=5)` ở table 0):

1. Userspace xlate nhận packet, dịch flow table 0. Đụng `OFPACT_CT`.
2. `ofproto-dpif-xlate.c` gọi `recirc_alloc_id_ctx(frozen_state)` để cấp `recirc_id = 0xabcd1234` (uint32 khác 0). `frozen_state` chứa metadata cần cho pass tiếp theo: ofproto pointer, table_id resume, register values, stack values.
3. State được lưu vào hash table trong `ofproto/ofproto-dpif-rid.c` với khóa `recirc_id`.
4. xlate emit datapath action sequence: `ct(zone=5,commit) → set(recirc_id=0xabcd1234) → recirc()` (dạng concrete trong odp action format `OVS_ACTION_ATTR_RECIRC`).
5. Datapath (kernel `net/openvswitch/actions.c` hoặc userspace `lib/dpif-netdev.c`) thực thi: chạy conntrack hook → set `key->recirc_id = 0xabcd1234` → enqueue gói vào FIFO defer (kernel) hoặc đệ quy gọi `dp_netdev_recirculate()` (userspace).
6. Pass mới: classifier lookup lại với key có `recirc_id=0xabcd1234`. Nếu hit megaflow đã cache → fast path. Nếu miss → upcall, userspace tra `frozen_state` qua `recirc_find_id()` và resume xlate từ table_id đã lưu, lúc này `ct_state` đã có giá trị thật.
7. Pass mới tiếp tục đến output hoặc gặp recirc tiếp theo. Mỗi pass có một megaflow riêng, kernel phân biệt qua `recirc_id` field trong key.

Lưu ý quan trọng: `recirc_id` không phải counter tăng dần mỗi pass cho packet đó, mà là *handle* tới một frozen state cụ thể. Hai packet cùng đi qua cùng `ct()` action có thể nhận cùng `recirc_id` nếu frozen state giống hệt (hash dedup), tiết kiệm pool slot.

**Cách kỹ sư vận hành thành thạo.** Một engineer thành thạo `recirc_id` cần phản xạ năm thao tác: (1) Đọc `ovs-dpctl dump-flows` và lập tức nhận diện "recirc_id(0)" (pass đầu) versus "recirc_id(0x<hex>)" (pass sau), không nhầm với register match. (2) Chạy `ofproto/trace` với option `--no-megaflow` để thấy đường recirc đầy đủ kèm dòng "thawed from frozen state at table N". (3) Đếm số pass per packet trong production qua `ovs-appctl coverage/show | grep -i recirc` và hiểu ngưỡng cảnh báo (rule of thumb: trung bình > 6 recirc per packet là dấu hiệu pipeline OF rule chưa optimize). (4) Đọc `ovs-appctl dpif/show -s` để thấy "packet recirculations: N" cumulative counter, kết hợp với packet count để tính tỷ lệ. (5) Khi viết OF rule mới, tự dự đoán recirc cost trước khi commit: mỗi `ct()`, mỗi `dec_ttl()` exception, mỗi `pop_mpls`, mỗi tunnel `decap` đều tạo ít nhất một recirc.

**Phân loại.** Có thể phân loại recirc_id theo nguồn sinh: (a) **conntrack-induced**, phổ biến nhất, do `ct()`, `ct(commit)`, `ct(nat)`, hoặc `ct_clear` sinh; (b) **decap-induced**, `decap()` Geneve/VXLAN/STT/NSH, `pop_mpls`, `pop_vlan` (chỉ một số trường hợp); (c) **hash-induced**, `dp_hash`, `hash_l4` action cho group `select_via_hash`; (d) **clone/sample-induced**, `clone()`, `sample()` action tách flow để cả nhánh chính và nhánh sample tiếp tục độc lập; (e) **OVN-pipeline-induced**, không phải mechanism mới mà là pattern OVN logical flow generate hàng loạt recirc qua `ct_*` và `lr_*` stages. Theo độ phổ biến gặp trong production: nhóm (a) > (e) > (b) > (c) > (d).

**When something goes wrong (rewritten in English by the 2026-04-28 source-citation cleanup).** `recirc_id` is a keyword to keep in mind in the following situations: (a) the OVS upcall rate spikes for no obvious reason; check first whether a new rule generates recirc. (b) The megaflow count grows non-linearly with new traffic; recirc combinations are the number-one suspect. (c) Packets drop after a `ct()` rule is installed; the frozen state pool may be full, or the rule may generate a recirc loop. (d) p99 latency jumps without an increase in bandwidth; each recirc adds another classifier traversal, and if the pass misses the megaflow it adds upcall latency. (e) A kernel oops with a stack trace containing recursive `do_execute_actions`: the recursion limit `OVS_RECURSION_LIMIT=4` (verified in `datapath/actions.c` of the OVS out-of-tree at v2.17.9) can be hit when a rule produces a recirc loop with no exit condition.

**Liên quan mật thiết.** `recirc_id` đan xen với: (a) `ct_state`, `ct_zone`, `ct_mark`, `ct_label`, tất cả đều chỉ "có giá trị thật" sau pass-0 recirc do `ct()` (xem §9.24). (b) `frozen_state`, cấu trúc lưu trong pool `ofproto-dpif-rid` (xem §9.32 classifier xlate). (c) `goto_table` và `resubmit`, KHÁC recirc về bản chất: hai cái sau là *cùng một pass* (linearize trong xlate), recirc là *pass mới* với key cập nhật (xem §9.22). (d) `megaflow`, recirc_id là field đầu tiên trong key (xem §9.2.2). (e) `dp_hash`, `select_via_hash`, group bucket selection cần recirc (xem §9.18, §9.22). (f) OVN logical flow, mỗi stage trong logical pipeline thường tương đương 1 recirc (xem §13.7, §13.8).

**Source code tham chiếu.** Theo Rule 14 dùng function name anchor (line drift across version). Baseline OVS v2.17.9:

- `include/openvswitch/meta-flow.h`, định nghĩa MFF_RECIRC_ID. Comment block ghi `NXM: NXM_NX_RECIRC_ID(36) since v2.2`, `OXM: none`, `Access: read-only`, `Internal use only, not programmable from controller`.
- `ofproto/ofproto-dpif-rid.h`, public API: `recirc_alloc_id()`, `recirc_alloc_id_ctx(struct frozen_state *)`, `recirc_find_id()`, `recirc_free_id()`. File header copyright Nicira 2014-2017.
- `ofproto/ofproto-dpif-rid.c`, implementation pool hash table. Doc nội bộ: "A recirculation id can be any uint32_t value except 0, which is reserved for the no-recirculation case".
- `ofproto/ofproto-dpif-xlate.c`, the `xlate_ctx` struct holds the recirc-related fields: `freezing` (bool), `recirc_update_dp_hash` (bool), `dp_hash_alg`, `dp_hash_basis`, `frozen_actions` (struct ofpbuf). The function `ctx_trigger_recirculate_with_hash()` triggers recirculation; line numbers are dropped per Rule 14 §14.4 Option C because they drift across releases. The constant `MAX_DEPTH = 64` caps recursion during translation.
- `lib/odp-execute.c`, the userspace datapath executor for `OVS_ACTION_ATTR_RECIRC`.
- `lib/dpif-netdev.c`, the function `dp_netdev_recirculate()` re-runs the classifier per packet inside the userspace datapath.
- Linux kernel `net/openvswitch/actions.c` (the OVS out-of-tree datapath; the in-tree mainline kernel path mirrors this code), the `OVS_ACTION_ATTR_RECIRC` handler inside `do_execute_actions()`. The constant `OVS_RECURSION_LIMIT = 4` (verified at the OVS out-of-tree `datapath/actions.c` v2.17.9; mainline kernel value may differ, check the running kernel) and `DEFERRED_ACTION_FIFO_SIZE = 10` cap the recirc depth and the deferred-action FIFO buffer.
- Linux kernel `net/openvswitch/flow.h`, the struct `sw_flow_key` carries the `recirc_id` field at the start of the key.

> **Note on the v3.9.2 hotfix.** An earlier version of this section cited `OVS_RECURSION_LIMIT = 5` and an inline line number "approx. line 766" for `ctx_trigger_recirculate_with_hash`. Verified at the OVS v2.17.9 baseline (commit `0bea06d9957e3966d94c48873cd9afefba1c2677`, 2024-02-08): the real value of `OVS_RECURSION_LIMIT` in `datapath/actions.c:77` is `4`, and the function is at `ofproto/ofproto-dpif-xlate.c:474`, not 766. Plan v3.9.2 R2 corrected the value and dropped the inline line number per Rule 14 §14.4 Option C. See `memory/sdn/v3.9.2-q7-axis16-sweep-2026-04-28.md`.

**Trường hợp sự cố thực tế.** Dựa trên pattern revalidator storm forensic phổ biến trong production OpenStack/OVN environment (xem §9.26 cho anatomy template chi tiết): một deployment Neutron với security group dùng `ct()` cho stateful firewall, sau khi tăng số tenant từ 200 lên 1000, `ovs-vswitchd` CPU lên 100% và packet drop rate tăng. Triệu chứng: `ovs-appctl dpif/show -s` báo `packet recirculations` từ 50K/sec lên 800K/sec; `ovs-dpctl dump-flows | wc -l` cho số megaflow tăng từ ~5K lên ~80K. Root cause thường là OF rule generated bởi neutron OVS agent có pattern `ct(commit,zone=N)` với zone khác nhau per port, kết hợp với recirc đầu pipeline cho ARP resolve, tạo tổ hợp `(zone × in_port × ct_state)` blow up megaflow. Mitigation: gộp zone về per-tenant thay vì per-port, giảm recirc point từ 4 xuống 2 trong pipeline. Lesson: mỗi recirc point thêm vào nhân số megaflow theo (số giá trị metadata mà pass tiếp theo match), không phải cộng.

**Lỗi thường gặp + tín hiệu chẩn đoán.**

- *Triệu chứng* Megaflow count tăng đột biến sau commit OF rule mới. *Nguyên nhân* Recirc tổ hợp với metadata khác nhau per-port/per-zone. *Chẩn đoán* `ovs-dpctl dump-flows | awk '{print $1}' | sort | uniq -c | sort -rn`, nhóm theo `recirc_id` xem pass nào có nhiều entry nhất. *Khắc phục* Gộp zone hoặc tách rule để giảm số metadata dimension trong pass đó.
- *Triệu chứng* Kernel log `nf_conntrack: table full, dropping packet` kèm OVS upcall storm. *Nguyên nhân* Mỗi recirc qua `ct()` tạo conntrack lookup; nếu pipeline có 2-3 recirc qua ct, nf_conntrack table chịu áp lực gấp 2-3 lần expected. *Chẩn đoán* `cat /proc/sys/net/netfilter/nf_conntrack_count` so với `nf_conntrack_max`. *Khắc phục* Tăng `nf_conntrack_max` hoặc loại bỏ recirc qua `ct()` thừa.
- *Triệu chứng* `ofproto/trace` báo `recursion too deep` hoặc kernel oops `OVS_RECURSION_LIMIT exceeded`. *Nguyên nhân* OF rule sinh recirc loop không có exit condition (pass N gọi recirc về table cũ pass N-1). *Chẩn đoán* Trace từng pass, tìm rule có action recirc về table sớm hơn current. *Khắc phục* Thêm match condition để pass sau không trigger lại cùng recirc point.
- *Triệu chứng* Frozen state pool grows unbounded, memory leak ovs-vswitchd. *Nguyên nhân* Bug cũ pre-OVS 2.13 khi ofproto bị remove mà không free recirc_id. *Chẩn đoán* `ovs-appctl memory/show` so sánh trước và sau `ovs-vsctl del-br`. *Khắc phục* Upgrade OVS lên ≥ 2.13.
- *Triệu chứng* Latency p99 tăng đột biến sau khi enable conntrack ACL. *Nguyên nhân* Mỗi packet thêm 1-2 recirc, nếu cache miss thì thêm 1-2 upcall latency. *Chẩn đoán* `ovs-appctl coverage/show | grep upcall` so trước/sau enable. *Khắc phục* Pre-warm cache hoặc giảm số recirc point.

**So sánh với hệ khác.** Các hệ khác giải bài "multi-pass packet processing" theo cách khác:

- **Linux netfilter NF_HOOK** chained (PREROUTING → FORWARD → POSTROUTING): mỗi hook là một "pass" theo nghĩa pipeline, nhưng metadata không persist explicit như `recirc_id`. Khác biệt cơ bản: netfilter pipeline tĩnh do kernel define, OVS recirc động do OF rule define.
- **Cisco IOS-XR feature pipeline** trên ASIC (egress recirculation cho VXLAN encap): tương đương về nguyên tắc nhưng implementation hardware. recirc_id trong OVS gần với khái niệm "loopback FIFO" trên chip Trident/Tomahawk.
- **P4 architecture parser-deparser-parser** (re-parse sau header rewrite): khác chỗ P4 cần programmer khai báo explicit recirc trong P4 code, OVS recirc là implicit do xlate sinh.
- **VPP graph node re-enqueue** (FD.io): tương đương. Mỗi graph node là một "stage", packet được re-enqueue vào graph với metadata cập nhật. recirc_id ≈ VPP `vnet_buffer` opaque field.
- **OpenFlow `goto_table` thuần** (không recirc): chỉ linearize trong cùng một xlate pass, không thay đổi packet key. Đây là điểm dễ nhầm lẫn nhất với recirc, vì `goto_table` rẻ còn recirc đắt.

### 9.2.14.3 openvswitch.ko, kernel module nền của toàn bộ kernel datapath

> **Phạm vi:** Treatment chuyên sâu (mở rộng 2026-04-26) cho keyword `openvswitch.ko`. Các phần liên quan: §9.1 (ba thành phần OVS, dpif-netlink userspace counterpart), §9.2.4 (upcall slow/fast path), §9.2.10 (Netlink wire format), §9.2.14.2 (recirc_id, một trong các action attribute mà module thực thi), §9.24 (conntrack hook trong actions.c), §11.0 (tunnel modules sit on top of openvswitch.ko).

**Khái niệm.** `openvswitch.ko` là Linux kernel module loadable (`.ko` = "kernel object") chứa toàn bộ phần kernel của OVS: kernel-side flow table megaflow, classifier TSS, action executor, conntrack glue, và Netlink generic genl families (`ovs_datapath`, `ovs_vport`, `ovs_flow`, `ovs_packet`, `ovs_meter`, `ovs_ct_limit`) làm cầu nối với userspace `ovs-vswitchd`. Khi nói "kernel datapath" của OVS thì cụ thể là module này; toàn bộ logic fast-path forward gói tin (~1-5 microsecond per packet) chạy bên trong nó. Loại: Linux loadable kernel module thuộc subsystem `net/`, source tree `net/openvswitch/`, build CONFIG_OPENVSWITCH=m, license GPL (`MODULE_LICENSE("GPL")` trong `datapath.c` v5.15 dòng 2715). Không có module này, lệnh `ovs-vsctl add-br br0` vẫn chạy nhưng datapath sẽ là userspace (`netdev`) hoặc `dummy`, không phải kernel; production stability không đạt.

**Lịch sử + bối cảnh.** Module được Nicira viết từ 2009 dưới tên `openvswitch_mod.ko` cùng với userspace; hai năm đầu chỉ tồn tại dạng out-of-tree (`openvswitch.git/datapath/`), distro phải DKMS build riêng cho mỗi kernel. Sau review nhiều vòng trên `linux-netdev`, Jesse Gross gửi patch series mainline tháng 11/2011, được Linus merge vào Linux **3.3** (release 18/03/2012), từ đó tồn tại trong tree dưới `net/openvswitch/`. Tên module rút gọn thành `openvswitch.ko`. Userspace tiếp tục có copy `datapath/` cho out-of-tree build (cần thiết cho distro kernel cũ thiếu feature mới như NSH, MPLS, conntrack zone limit), nhưng best practice production từ 2018 là dùng in-tree. Mỗi kernel release cộng feature mới: 3.10 (2013) thêm `ovs_vport_ops` framework, 4.3 (2015) thêm conntrack glue (`OVS_ACTION_ATTR_CT`), 4.6 (2016) thêm conjunction action, 4.13 (2017) thêm hardware offload TC integration, 5.6 (2020) thêm `OVS_ACTION_ATTR_CHECK_PKT_LEN`, 5.15 (2021, baseline curriculum) ổn định toàn bộ tập action. Out-of-tree backport vẫn đi kèm OVS userspace tarball cho tới OVS 2.17, sau đó OVS 3.0 (2022) bỏ kernel module backport, chỉ support in-tree.

**Vị trí trong kiến trúc.** Module nằm ở tầng `net/` của Linux kernel, dưới `netfilter` về thứ tự hook và song song với `bridge`. Cấu trúc layered: (1) tầng dưới cùng là `vport` abstraction (`net/openvswitch/vport.c`) bind module vào `net_device` Linux qua `netdev_rx_handler_register()` để intercept gói; (2) tầng giữa là flow table + classifier (`flow_table.c`, `flow.c`) lookup megaflow theo `sw_flow_key`; (3) tầng action executor (`actions.c`) thực thi datapath action set; (4) tầng Netlink (`datapath.c`) đăng ký bốn genl family với userspace để nhận lệnh `OVS_DP_CMD_*`, `OVS_VPORT_CMD_*`, `OVS_FLOW_CMD_*`, gửi upcall qua `OVS_PACKET_CMD_MISS`. Userspace counterpart là `ovs-vswitchd` qua `lib/dpif-netlink.c` (Phần 9.1). Module nằm trên tunnel modules (`vport-vxlan.ko`, `vport-geneve.ko`, `vport-gre.ko`) — các module tunnel đăng ký `vport_ops` lại với `openvswitch.ko`. Cả `nf_conntrack` và `openvswitch.ko` cùng share table conntrack chung (Phần 9.24).

**Vai trò.** Bốn vai trò cốt lõi trong kiến trúc OVS: (a) **fast-path executor**, mọi gói qua bridge OVS đi qua hook của module ở `netdev_rx_handler_register()`, lookup flow key, execute action set, không trở về Linux IP stack trừ khi action `output:LOCAL` hoặc `normal`; (b) **Netlink server**, khai báo các genl family `ovs_datapath`/`ovs_vport`/`ovs_flow`/`ovs_packet`/`ovs_meter`/`ovs_ct_limit` (xem `MODULE_ALIAS_GENL_FAMILY` trong `datapath.c` dòng 2716-2721 v5.15) cho `ovs-vswitchd` cài flow, query stats, gửi packet, nhận upcall; (c) **flow cache custodian**, sở hữu `dp_netdev_flow` struct và RCU-protected flow table, đảm bảo lookup không block install qua RCU read-side critical section; (d) **conntrack hook integrator**, gọi `nf_conntrack_in()` (`net/openvswitch/conntrack.c`) cho action `ct()`, integrate Linux netfilter conntrack vào pipeline OF, giữ atomicity giữa `ct(commit)` và stateful match.

**Vì sao sinh ra.** Userspace-only switching (Open vSwitch v0.x đầu 2009) giới hạn ~500K packet/s do mỗi gói phải context-switch kernel→userspace→kernel cho forward decision, mỗi switch ~2 microsecond. Với KVM cloud (mỗi host 50+ VM, mỗi VM hàng nghìn microflow), throughput thực tế không vượt 1 Gbps. Nicira chọn cách "split datapath": kernel cache decision đã tính, userspace chỉ tính khi cache miss. Để cache nằm trong kernel mà không phải patch toàn bộ kernel core (Linux maintainer không chấp nhận), giải pháp duy nhất là module loadable. Module phải đủ generic để xử lý OF action set (output, set, push_vlan, ct, recirc, ...) nhưng đủ nhỏ để Linus merge mainline. Patch series 2011-2012 của Jesse Gross thuyết phục được vì module tự đứng riêng (không patch L2 bridge, không patch netfilter), feature isolated, source ~5K dòng (compared with ~50K dòng userspace `ovs-vswitchd`).

**Vấn đề giải quyết.** Năm vấn đề nền tảng: (1) **context-switch overhead**, mỗi packet phải trả ~2 microsecond cho kernel↔userspace round trip → giới hạn throughput thực tế; cache trong kernel triệt tiêu chi phí cho ~99% gói (cache hit). (2) **classifier complexity in kernel**, OF pipeline với prioritized rules + arbitrary header modification quá phức tạp để viết kernel code; megaflow + TSS giữ classifier trong userspace, kernel chỉ làm hash lookup. (3) **multi-bridge + multi-tenant scaling**, một host phải có hàng chục bridge, hàng nghìn port, hàng triệu flow; module dùng RCU + per-bridge `struct datapath` cô lập tenant. (4) **conntrack integration**, stateful firewall đòi hỏi share table với Linux netfilter; module gọi trực tiếp `nf_conntrack_in()` thay vì duplicate state. (5) **tunnel offload**, VXLAN/Geneve/GRE encap/decap muốn chạy line-rate cần kernel-level; module cung cấp `vport_ops` framework cho tunnel modules plug vào.

**Tầm quan trọng.** Đây là khái niệm cốt lõi tuyệt đối, ngang với `megaflow` và `OVS_ACTION_ATTR_OUTPUT` về mức độ quan trọng. Engineer mà không nắm openvswitch.ko sẽ không debug được: (a) bất kỳ lỗi nào xảy ra trước khi packet kịp tới `ovs-vswitchd` (kernel-only path); (b) module load/unload race condition khi upgrade OVS hoặc kernel; (c) conntrack helper module dependency (`nf_nat`, `nf_conntrack_ftp`, ...); (d) hardware offload chain (`tc-flower` pin về module này); (e) module parameter tuning (`flow_lookup` debug, `vport-internal_dev` MTU). Production OpenStack/OVN deployment với kernel datapath là default; mọi neutron-ovs-agent crash log đều có frame `openvswitch.ko` ở stack trace. Mức độ quan trọng ngang `recirc_id` và `ofproto-dpif xlate` trong nhóm cốt lõi.

**Cơ chế hoạt động.** Vòng đời module + per-packet flow:

1. **Module load** (`modprobe openvswitch`): kernel gọi `dp_init()` (datapath.c dòng 2637 v5.15). Sequence: `action_fifos_init()` → `ovs_internal_dev_rtnl_link_register()` (đăng ký netlink RTNL ops cho internal port) → `ovs_flow_init()` (cấp slab cache cho `sw_flow`) → `ovs_vport_init()` (init vport table) → `register_pernet_device(&ovs_net_ops)` (per-net-namespace state) → `register_netdevice_notifier(&ovs_dp_device_notifier)` (theo dõi netdev change) → `ovs_netdev_init()` → `dp_register_genl()` (đăng ký 4 genl family với generic netlink). Kernel log `pr_info("Open vSwitch switching datapath\n")` xuất hiện trong `dmesg`.
2. **`ovs-vswitchd` connect**: userspace mở Netlink socket family `NETLINK_GENERIC`, resolve family ID cho `ovs_datapath`, gửi `OVS_DP_CMD_NEW` với attribute `OVS_DP_ATTR_NAME="ovs-system"`. Kernel `ovs_dp_cmd_new()` (datapath.c dòng 1732 v5.15) cấp `struct datapath`, tạo internal port `ovs-system`, reply `OVS_DP_CMD_NEW`.
3. **Bridge create**: cho mỗi bridge OF (`br-int`, `br-ex`) userspace gửi `OVS_DP_CMD_NEW` riêng → kernel cấp `struct datapath` riêng. Userspace gửi `OVS_VPORT_CMD_NEW` để add port (kiểu `OVS_VPORT_TYPE_NETDEV`, `_INTERNAL`, `_VXLAN`, ...). Kernel gọi `netdev_rx_handler_register()` để intercept gói tới netdev đó.
4. **Per-packet fast path**: NIC RX → driver `napi_gro_receive()` → Linux `__netif_receive_skb_core()` → kernel kiểm tra rx_handler bit, gọi `netdev_frame_hook()` → `ovs_vport_receive()` → `ovs_dp_process_packet()` → `ovs_flow_key_extract()` extract `sw_flow_key` (recirc_id, in_port, eth, ip, tp, ct_state, ...) → `ovs_flow_tbl_lookup_stats()` lookup TSS → match → `ovs_execute_actions()` (`actions.c` dòng 1241) iterate action set → execute từng `OVS_ACTION_ATTR_*` (output qua `do_output()`, set qua `execute_set_action()`, ct qua `ovs_ct_execute()`, recirc qua `ovs_recirc()`). Toàn bộ chuỗi ~1-5 microsecond.
5. **Per-packet slow path (cache miss).** `ovs_dp_process_packet()` does not find a flow, so `ovs_dp_upcall()` builds the Netlink message `OVS_PACKET_CMD_MISS` and sends it through the socket registered by the handler thread. The `do_execute_actions()` recursion limit `OVS_RECURSION_LIMIT = 4` (in `datapath/actions.c` of the OVS out-of-tree at v2.17.9) prevents infinite loops; the deferred FIFO `DEFERRED_ACTION_FIFO_SIZE = 10` buffers long action chains.
6. **Module unload** (`rmmod openvswitch`): kernel gọi `dp_cleanup()` (`module_exit(dp_cleanup)` datapath.c dòng 2712), reverse sequence: `dp_unregister_genl()` → `ovs_netdev_exit()` → `unregister_netdevice_notifier()` → `unregister_pernet_device()` → `ovs_vport_exit()` → `ovs_flow_exit()` → `ovs_internal_dev_rtnl_link_unregister()` → `action_fifos_exit()`. Nếu còn datapath active, refcount > 0, unload fail với `Module openvswitch is in use`.

**Cách kỹ sư vận hành thành thạo.** Engineer thành thạo `openvswitch.ko` cần phản xạ năm thao tác: (1) Verify module loaded via `lsmod | grep ^openvswitch` thấy ngay `openvswitch` + dependent modules `nf_conncount`, `nf_nat`, `nf_conntrack` ở cột Used by. (2) Đọc `modinfo openvswitch` để verify version, vermagic, license, dependency, parameter. (3) Quan sát kernel log lúc load qua `dmesg | grep -i openvswitch`, expect dòng `Open vSwitch switching datapath` và không có warning. (4) Khi upgrade kernel hoặc OVS, biết thứ tự đúng: stop `openvswitch-switch` service → `rmmod` các vport modules → `rmmod openvswitch` → upgrade → `modprobe openvswitch` → start service. (5) Khi nghi ngờ module có bug, biết bật debug: `echo 8 > /proc/sys/kernel/printk` + `modprobe openvswitch dyndbg=+p` để dynamic debug log từng function.

**Phân loại.** Có thể phân loại artifact và operation liên quan: (a) **module artifact** = file `.ko`, đường dẫn `/lib/modules/$(uname -r)/kernel/net/openvswitch/openvswitch.ko`; (b) **dependency modules** = `nf_conntrack`, `nf_nat`, `nf_defrag_ipv4`, `nf_defrag_ipv6`, `libcrc32c`, `vport-vxlan`, `vport-geneve`, `vport-gre`, `vport-stt` (out-of-tree); (c) **per-net state** = `struct ovs_net` per network namespace; (d) **per-bridge state** = `struct datapath`; (e) **per-port state** = `struct vport`; (f) **per-flow state** = `struct sw_flow` + `struct sw_flow_actions` (RCU-protected); (g) **Netlink family** = sáu family đăng ký vào generic netlink.

**Khi xảy ra sự cố.** `openvswitch.ko` là keyword bắt buộc nhớ trong các tình huống: (a) `ovs-vsctl` báo `database connection failed` nhưng `ovs-vswitchd` đang chạy, kiểm tra `lsmod | grep openvswitch` xem module có load không; module fail load làm Netlink genl family không tồn tại → `ovs-vswitchd` không kết nối được. (b) `dmesg` báo `kernel BUG at net/openvswitch/...` hoặc oops với function frame trong module, signal cho kernel-side bug, cần thu thập full stack trace + `cat /proc/version` để gửi upstream. (c) Sau kernel upgrade, `openvswitch-switch` service fail to start, error `Failed to load module openvswitch`, thường do package `linux-modules-extra-$(uname -r)` thiếu (Ubuntu) hoặc DKMS build chưa rebuild (RHEL out-of-tree). (d) `rmmod openvswitch` fail với `Module openvswitch is in use` nhưng `ovs-vswitchd` đã stop, kiểm tra còn datapath nào active qua `cat /sys/kernel/debug/openvswitch/` (nếu debugfs mounted) hoặc `ip -d link show type openvswitch`. (e) Conntrack helper FTP/TFTP không hoạt động sau enable security group, kiểm tra `nf_conntrack_ftp` đã load chưa và `openvswitch.ko` thấy được nó qua `modprobe nf_conntrack_ftp`.

**Liên quan mật thiết.** `openvswitch.ko` đan xen với: (a) `ovs-vswitchd` userspace daemon (Phần 9.1) qua Netlink, không có module thì daemon không có datapath để cài flow. (b) `dpif-netlink` (Phần 9.1, `lib/dpif-netlink.c` userspace) là driver provider abstraction nói chuyện với module qua genl. (c) `nf_conntrack`, `nf_nat`, `nf_conncount` là dependency mandatory cho action `ct()` (Phần 9.24); module declare phụ thuộc qua `MODULE_DEPS`. (d) `vport-vxlan`, `vport-geneve`, `vport-gre`, `vport-stt` plug vào module qua `vport_ops` framework (Phần 11.0). (e) `sw_flow_key` struct chứa `recirc_id` ở vị trí đầu (Phần 9.2.14.2) là datatype shared giữa module và userspace qua `OVS_KEY_ATTR_*` Netlink attribute. (f) `tc-flower` hardware offload (Phần 9.5) đăng ký với module qua `flow_offload_setup_block_cb()`. (g) Megaflow cache (Phần 9.2.2) tồn tại trong RAM kernel-side, không persist qua reboot, không qua module unload.

**Source code reference.** Per Rule 14 §14.4 Option C, this section uses function-name anchors instead of inline line numbers. Inline line numbers drift across releases, and the kernel datapath path here spans two trees (the OVS out-of-tree `datapath/` shipped with OVS v2.17.9, and the mainline Linux kernel tree) that have different line numbers. The user's local OVS clone has the OVS out-of-tree, so all anchors below were verified at OVS v2.17.9 against `datapath/`, with the understanding that the mainline kernel tree mirrors this structure but uses different line numbers.

- `net/openvswitch/datapath.c` (mainline kernel) or `datapath/datapath.c` (OVS out-of-tree), the main module file. The function `dp_init()` is the module entry point registered by `module_init(dp_init)`; it runs the full init sequence: `action_fifos_init`, `ovs_internal_dev_rtnl_link_register`, `ovs_flow_init`, `ovs_vport_init`, `register_pernet_device`, `register_netdevice_notifier`, `ovs_netdev_init`, `dp_register_genl`. The function `dp_cleanup()` is referenced in `module_exit(dp_cleanup)` and runs the reverse cleanup. The function `ovs_dp_cmd_new()` handles `OVS_DP_CMD_NEW` from userspace, and `ovs_dp_cmd_del()` handles `OVS_DP_CMD_DEL`. The module declares `MODULE_LICENSE("GPL")`, `MODULE_DESCRIPTION("Open vSwitch switching datapath")`, and six `MODULE_ALIAS_GENL_FAMILY()` aliases.
- `net/openvswitch/datapath.h`, the type definitions: `struct datapath` (per bridge), `struct ovs_net` (per namespace), and `struct ovs_skb_cb` (per packet metadata).
- `net/openvswitch/flow.h`, the type definitions: `struct sw_flow_key` (the `recirc_id` field is at the start), `struct sw_flow`, and `struct sw_flow_actions`.
- `net/openvswitch/flow_table.c` (mainline) or `datapath/flow_table.c` (OVS out-of-tree), the function `ovs_flow_alloc()` allocates a slab entry, and `ovs_flow_tbl_lookup_stats()` is the TSS lookup hot path. Note: an earlier version of this section cited these functions inside `flow.c`; the actual file is `flow_table.c`.
- `net/openvswitch/actions.c` (mainline) or `datapath/actions.c` (OVS out-of-tree), the function `do_execute_actions()` is the per-packet executor. The constants `DEFERRED_ACTION_FIFO_SIZE = 10` and `OVS_RECURSION_LIMIT = 4` (verified at OVS out-of-tree v2.17.9; mainline kernel may differ) cap the recursion depth and the deferred-action FIFO buffer. The recursion check is `if (unlikely(level > OVS_RECURSION_LIMIT))` inside `do_execute_actions`.
- `net/openvswitch/vport.c` plus `vport.h`, the vport abstraction; hooks into `net_device` through `netdev_rx_handler_register()`.
- `include/uapi/linux/openvswitch.h` (mainline) or `datapath/linux/compat/include/linux/openvswitch.h` (OVS out-of-tree), the Netlink UAPI: `#define OVS_DATAPATH_FAMILY "ovs_datapath"`, `OVS_DATAPATH_VERSION 2`, `OVS_PACKET_FAMILY "ovs_packet"`, `OVS_VPORT_FAMILY "ovs_vport"`, `OVS_FLOW_FAMILY "ovs_flow"`, `OVS_METER_FAMILY "ovs_meter"`, `OVS_CT_LIMIT_FAMILY "ovs_ct_limit"`. The `OVS_ACTION_ATTR_*` enum defines every action that the datapath executes.
- Userspace counterpart: `lib/dpif-netlink.c` (OVS userspace tree v2.17.9), the function `dpif_netlink_init()` and `dpif_netlink_open()` open the `NETLINK_GENERIC` socket family and resolve the genl family ID through `nl_lookup_genl_family()` in `lib/netlink-socket.c`.

Diagnostic đầu tiên: `lsmod | grep openvswitch` không thấy module load. Tiếp theo: `journalctl -u openvswitch-switch` báo `modprobe: ERROR: could not insert 'openvswitch': Exec format error` và `dmesg` có `openvswitch: version magic '5.4.0-150-generic SMP mod_unload modversions ' should be '5.15.0-101-generic SMP mod_unload modversions '`. **Lỗi thường gặp + tín hiệu chẩn đoán.**

- *Triệu chứng* `modprobe openvswitch` báo `Exec format error`. *Nguyên nhân* Module compile cho kernel khác đang chạy (vermagic mismatch). *Chẩn đoán* `dmesg | tail` thấy `version magic '...' should be '...'`; so sánh `uname -r` với `modinfo openvswitch | grep filename`. *Khắc phục* Install `linux-modules-extra-$(uname -r)` (Ubuntu) hoặc rebuild DKMS (`dkms autoinstall`).
- *Triệu chứng* `ovs-vswitchd` log `dpif|ERR|could not create datapath system: Address family not supported`. *Nguyên nhân* Module không load → genl family `ovs_datapath` không tồn tại. *Chẩn đoán* `lsmod | grep openvswitch` empty. *Khắc phục* `modprobe openvswitch` + verify `dmesg`.
- *Triệu chứng* `rmmod openvswitch` báo `Module openvswitch is in use by: nf_nat`. *Nguyên nhân* Hiểu sai chiều dependency, thực ra nf_nat depend vào nhiều thứ; cũng có thể conntrack helper modules đang dùng. *Chẩn đoán* `lsmod | grep openvswitch` xem cột "Used by"; nếu ≥ 1, có user. *Khắc phục* `systemctl stop openvswitch-switch`, đảm bảo no bridge active, sau đó `rmmod`.
- *Triệu chứng* Sau live kernel patch (kpatch), `openvswitch.ko` crash với oops "BUG: unable to handle page fault". *Nguyên nhân* Live patch không patch module symbol đúng. *Chẩn đoán* `dmesg` full stack trace, frame có function trong `net/openvswitch/`. *Khắc phục* Reboot vào kernel chưa patch hoặc rebuild module với patch.
- *Triệu chứng* Conntrack helper FTP không hoạt động (passive FTP fail) sau enable security group OVN. *Nguyên nhân* `nf_conntrack_ftp` không auto-load qua `openvswitch.ko`. *Chẩn đoán* `lsmod | grep nf_conntrack_ftp` empty; `cat /proc/sys/net/netfilter/nf_conntrack_helper` = 0. *Khắc phục* `modprobe nf_conntrack_ftp` + add vào `/etc/modules-load.d/openvswitch.conf`.

**So sánh với hệ khác.** Các hệ khác giải bài "kernel-level packet forwarding" theo cách khác:

- **Linux native bridge** (`bridge.ko`, source `net/bridge/`): tương đương về layered placement (kernel module hooks vào `netdev_rx_handler`) nhưng feature scope hẹp hơn rất nhiều. Bridge L2-only, không có flow table, không có classifier wildcard, không có Netlink genl programmable interface. Bridge dùng STP/MAC learning fixed; OVS dùng OF rule programmable. Khi node chạy cả Linux bridge và OVS, hai module song song hooks vào `netdev_rx_handler` nhưng mỗi netdev chỉ bind được một handler tại một thời điểm.
- **macvlan / ipvlan**: kernel modules cùng layer cung cấp L2/L3 multiplexing trên một netdev, nhưng không phải switching, không có flow table.
- **VPP graph nodes** (FD.io, userspace): tương đương về graph-based packet processing nhưng chạy hoàn toàn userspace với DPDK polling driver, không phải kernel module. VPP `vnet/` ≈ openvswitch.ko `net/openvswitch/` về mức trừu tượng (graph node ≈ action), nhưng implementation môi trường khác hẳn (kernel-bypass vs in-kernel).
- **Cisco IOS-XR forwarding code** (proprietary, ASIC-assisted): hoạt động cùng trừu tượng (flow table + action) nhưng chạy trên ASIC hardware-accelerated; openvswitch.ko là software equivalent cho commodity x86 server.
- **`nftables`/`iptables`** (`nf_tables.ko`/`ip_tables.ko`): cùng nằm `net/netfilter/`, cùng là kernel module L2-L7 packet processing, nhưng paradigm khác. Netfilter có hook structure cố định (PREROUTING/FORWARD/POSTROUTING) + rule chain, không có flow cache + classifier wildcard. OVS conntrack action gọi vào `nf_conntrack` chia sẻ table với netfilter, là điểm giao nhau quan trọng (Phần 9.24).
- **DPDK userspace datapath** (`netdev` datapath của OVS, chạy trong `ovs-vswitchd`): là alternative cho `openvswitch.ko`; cùng OVS userspace nhưng datapath chạy fully userspace với polling thread. Không phải kernel module. Tradeoff: throughput cao hơn (10-50 Gbps line-rate khả thi) nhưng dedicated CPU core, không integrate với Linux netfilter conntrack natively. Curriculum baseline dùng kernel datapath qua `openvswitch.ko`; userspace datapath nội tại nằm ngoài phạm vi.

### 9.2.14.4 conntrack execute, các phương diện bổ sung

**Khái niệm.** `conntrack_execute()` là kernel function trong `net/openvswitch/conntrack.c` execute conntrack action (ct() in OF flow). Per-packet conntrack lookup + state update.

**Tầm quan trọng.** Cốt lõi của OVS conntrack pipeline. Cross-link `9.24 ct_state` (consumer of state set by execute).

**Cơ chế hoạt động.** Triggered by `OFPACT_CONNTRACK` action:
1. Lookup nf_conntrack table với 5-tuple
2. If first packet → create new entry (state=NEW)
3. If existing → update state (NEW→ESTABLISHED, etc.)
4. Set packet's ct_state, ct_zone, ct_mark, ct_label registers
5. Optionally commit entry, NAT rewrite, helper invoke

**Source code tham chiếu.** `net/openvswitch/conntrack.c` `ovs_ct_execute()`. Linux `nf_conntrack_in()` underlying. `lib/conntrack.c` userspace fallback (DPDK datapath).

**Lỗi thường gặp + tín hiệu chẩn đoán.**
- Mode A: Conntrack table full → diagnostic `nf_conntrack_count > nf_conntrack_max`. Recovery: increase max sysctl.
- Mode B: Conntrack zone collision → ct_state INVALID. Recovery: zone partition.

**So sánh với hệ khác.** Linux iptables `-m conntrack` consumes same kernel conntrack. Khác: iptables expression chain hook; OVS conntrack action OF flow stage.

### 9.2.15.1 `revalidator thread`

#### Khái niệm

Revalidator thread là một nhóm thread userspace bên trong tiến trình `ovs-vswitchd`, có nhiệm vụ định kỳ duyệt toàn bộ megaflow đã cài trong kernel datapath, đối chiếu hành vi của chúng với bảng OpenFlow hiện hành, và đồng bộ trạng thái: cập nhật action mới, xóa megaflow đã hết hạn, hoặc xóa megaflow đã không còn rule nào sản sinh ra. Tên gọi "revalidator" phản ánh nhiệm vụ "tái xác thực" tính đúng đắn của cache, vì megaflow là kết quả compile lại tại thời điểm cài, có thể trở nên sai khi rule trên controller hoặc OVN đổi.

#### Lịch sử + bối cảnh

Cơ chế revalidator được tách thành thread riêng từ khoảng OVS 2.0 (2014), song hành với mô hình userspace cache mới sau khi NSDI 2015 publication "The Design and Implementation of Open vSwitch" được trình bày. Trước đó, OVS dùng cơ chế "facet" và "subfacet" gắn trong main thread, scale kém. Số revalidator thread cấu hình được qua `other_config:n-revalidator-threads` từ OVS 2.4. Mặc định bằng `ceil(CPU/4)` cho host hiện đại, tối thiểu 1.

#### Vị trí trong kiến trúc

Revalidator nằm trong layer `ofproto-dpif-upcall.c` của userspace, giữa OF rule store (`classifier`) và datapath interface (`dpif`). Cùng nhóm thread userspace gồm: main thread (xử lý OVSDB IDL), N handler thread (xử lý upcall), M revalidator thread (xử lý sweep). Revalidator KHÔNG xử lý packet trực tiếp, chỉ đọc-ghi metadata megaflow.

#### Vai trò

Bốn vai trò chính: (1) duy trì correctness, đảm bảo megaflow trong kernel khớp với OF pipeline hiện tại; (2) thu hồi megaflow idle, giải phóng kernel memory; (3) cập nhật counter `n_packets`/`n_bytes` từ kernel về userspace; (4) ép kernel xóa megaflow khi rule liên quan bị OF controller xóa hoặc rule mới prio cao hơn xuất hiện.

#### Vì sao sinh ra

Kernel datapath không biết gì về OF semantics; nó chỉ thực thi action đã cài. Khi rule đổi, không thể trông đợi kernel tự động cập nhật. Userspace phải có một "vacuum cleaner" chạy nền dọn cache. Đặt vào main thread sẽ chặn upcall handling khi cache lớn (~hàng trăm ngàn megaflow). Tách thread riêng là quyết định scale-out tự nhiên.

#### Cơ chế hoạt động

Mỗi sweep cycle (mặc định ~500 ms): thread acquire RCU read lock, snapshot toàn bộ ukey list, chia đều cho M revalidator thread (round-robin theo hash ukey). Mỗi thread, với mỗi ukey thuộc shard của mình, gọi `revalidate_ukey()`: dump `OVS_FLOW_CMD_GET` lấy stats hiện tại, re-walk OF pipeline (`xlate_actions()`), so kết quả mới với action cũ. Nếu khác thì gửi `OVS_FLOW_CMD_SET` (hoặc `_DEL`); nếu giống thì cập nhật `last_used` timestamp và stats. Hàm chính: `revalidator_run()` + `revalidator_sweep()` trong `ofproto/ofproto-dpif-upcall.c`.

#### Cách kỹ sư vận hành thành thạo

Engineer cấp 2 cần biết: (a) tunable `n-revalidator-threads`; (b) interpret `upcall/show` để biết revalidator có theo kịp không; (c) chạy `revalidator/wait` như barrier khi debug race; (d) đọc counter `coverage/show facet_revalidate` để định lượng tải; (e) biết khi nào cần nâng số thread (avg-updates per thread > 5/sec hoặc dump duration > 1 giây).

#### Cách quan sát + xác minh

```
$ sudo ovs-appctl upcall/show
system@ovs-system:
  flows         : (current 2345) (avg 1234) (max 4567) (limit 200000)
  dump duration : 12ms
  ufid enabled : true
  0: (keys 456)   ← revalidator 0 quản lý 456 ukey
  1: (keys 432)
  2: (keys 123)
  3: (keys 234)
```

`dump duration: 12ms` là thời gian một sweep cycle hoàn tất. > 1000ms cảnh báo. Counter chi tiết:

```
$ sudo ovs-appctl coverage/show | grep -E "revalidate|ukey"
flow_dumps_done                 4567 / sec
revalidate                      4501 / sec
ukey_invalid                    23 / sec
```

#### Source code tham chiếu

Hàm `revalidator_run()` và `revalidator_sweep()` trong `ofproto/ofproto-dpif-upcall.c` (OVS branch-2.17). Hàm `revalidate_ukey()` thực hiện so sánh action; hàm `delete_op_init()` build batched `OVS_FLOW_CMD_DEL`. Struct chính: `struct revalidator` chứa per-thread state.

#### Khác biệt giữa các phiên bản

- OVS 2.0 (2014): tách revalidator khỏi main thread.
- OVS 2.4 (2015): tunable `n-revalidator-threads`.
- OVS 2.7 (2017): ép buộc UFID khi communicate với kernel, giảm Netlink message size cho dump.
- OVS 2.13 (2020): tối ưu batched delete, giảm syscall.
- OVS 2.17 (2022, baseline): mặc định 4 thread cho host 16-core; integer overflow fix trong `udpif_revalidator()`.

#### Trường hợp sự cố thực tế

Production case Phần 9.26 forensic "revalidator storm": OF controller (Faucet) push 5000 rule update trong 1 giây, revalidator buộc re-walk toàn bộ ~50K ukey, dump duration nhảy từ 12ms lên 8500ms. Kết quả: handler thread bị starve do contention trên ofproto mutex, packet rate giảm 30%, một số TCP retransmit. Mitigation: nâng `n-revalidator-threads` từ 4 lên 12, batch rule push qua `ovs-ofctl --bundle replace-flows`.

#### Lỗi thường gặp + tín hiệu chẩn đoán

| Triệu chứng | Nguyên nhân khả dĩ | Cách kiểm chứng |
|-------------|---------------------|-----------------|
| `dump duration` > 5 giây | Số ukey lớn, ít thread | `upcall/show` + `memory/show udpif keys` |
| `flow_dumps_done` rate thấp | Revalidator backlog | `coverage/show flow_dumps_done` |
| Kernel megaflow stale (action sai) | Revalidator crash hoặc deadlock | `ovs-appctl revalidator/wait` block forever |
| `ukey_invalid` rate cao | RCU race khi rule đổi nhanh | `coverage/show ukey_invalid` |

#### So sánh với hệ khác

Linux conntrack có `gc_worker` chạy mỗi 1/8 giây sweep `nf_conn` table tương tự revalidator: cùng pattern "background sweeper cập nhật cache state". Cilium eBPF không có equivalent vì map BPF không cần re-validate; thay vào đó dùng map LRU tự evict. Cisco Nexus 9000 NX-OS có `vshd` thread sweep hardware MAC table tương tự.

### 9.2.15.2 `handler thread`

#### Khái niệm

Handler thread là nhóm thread userspace trong `ovs-vswitchd` chuyên xử lý upcall: khi kernel datapath gặp packet không khớp megaflow, kernel gửi packet (kèm metadata) lên userspace qua Netlink socket, handler thread nhận, chạy OpenFlow pipeline, compile megaflow mới, cài về kernel, và (tùy ngữ cảnh) inject lại packet để tránh drop packet đầu tiên.

#### Lịch sử + bối cảnh

Tách thread riêng từ OVS 2.0 cùng đợt với revalidator. Trước đó upcall xử lý trong main thread, single-thread bottleneck rõ rệt khi miss rate tăng. Tunable `other_config:n-handler-threads` từ OVS 2.4. Mặc định = số CPU core (cho đến OVS 2.13), sau đó đổi thành `ceil(CPU/2)` để chừa core cho revalidator + main.

#### Vị trí trong kiến trúc

Handler nằm song song revalidator trong `ofproto-dpif-upcall.c`. Mỗi handler đăng ký một Netlink PID với kernel; kernel hash flow key chọn handler nhận upcall (load balance). Handler gọi `xlate_actions()` (cùng module dùng bởi revalidator) để chạy OF pipeline.

#### Vai trò

(1) Receive upcall từ kernel; (2) parse Netlink message thành `struct dp_packet` + `struct flow`; (3) classify qua TSS classifier lấy OF rule match; (4) execute action chain trong xlate context; (5) compile thành megaflow với mask được mở rộng tối đa (lazy wildcarding); (6) cài megaflow về kernel qua `OVS_FLOW_CMD_NEW`; (7) inject lại packet qua `OVS_PACKET_CMD_EXECUTE` để gói tin "đầu" không bị mất.

#### Vì sao sinh ra

Slow path là biểu hiện không thể tránh của kiến trúc cache: lần đầu mọi flow đều miss. Nếu để main thread xử lý, miss rate ~10K/sec là đủ saturate. Tách thread + load balance theo hash flow key cho phép scale theo CPU.

#### Cơ chế hoạt động

```
Kernel openvswitch.ko          Userspace ovs-vswitchd
─────────────────────          ──────────────────────
ovs_dp_process_packet()
    │ (cache miss)
    ▼
ovs_dp_upcall()
build OVS_PACKET_CMD_MISS
    │ Netlink (genl)
    ▼
                              udpif_upcall_handler()  ← thread N
                                    │ recv_upcalls()
                                    │ parse → struct upcall
                                    │ xlate_actions()
                                    │ ukey_create_from_upcall()
                                    │ dpif_flow_put() OVS_FLOW_CMD_NEW
                                    │ dpif_execute() OVS_PACKET_CMD_EXECUTE
                                    ▼
                              megaflow installed in kernel
                              packet re-injected
```

Hàm chính: `udpif_upcall_handler()` (loop), `recv_upcalls()` (drain Netlink), `process_upcall()` (per-packet xlate + install).

#### Cách kỹ sư vận hành thành thạo

Engineer cần biết: (a) tunable `n-handler-threads`; (b) đọc `upcall/show` per-thread keys count để xem load balance; (c) phân biệt symptom "miss rate cao" (bình thường khi traffic mới) vs "lost rate cao" (bất thường, kernel drop vì handler nghẽn); (d) khi nào tăng thread (avg upcall in-flight > 100, hoặc lost > 0).

#### Cách quan sát + xác minh

```
$ sudo ovs-appctl upcall/show
system@ovs-system:
  flows         : (current 2345) (avg 1234) (max 4567) (limit 200000)
  dump duration : 12ms
  ufid enabled : true
  0: (keys 456)   ← handler 0 đã install 456 megaflow
  1: (keys 432)
  2: (keys 123)
  3: (keys 234)

$ sudo ovs-appctl coverage/show | grep -E "upcall|miss"
miss_upcall                     1247 / sec
flow_install                    1198 / sec
upcall_flow_limit_hit           0 / sec
```

`miss_upcall` rate phải xấp xỉ `flow_install` rate. Nếu `upcall_flow_limit_hit` > 0 thì đã chạm `set-flow-limit` (mặc định 200K) → cần `upcall/set-flow-limit 500000`.

#### Source code tham chiếu

`udpif_upcall_handler()` và `recv_upcalls()` trong `ofproto/ofproto-dpif-upcall.c` (branch-2.17). `process_upcall()` thực hiện xlate; `handle_upcalls()` batch install. Struct `struct handler` chứa per-thread state + Netlink PID.

#### Khác biệt giữa các phiên bản

- OVS 2.0: tách handler khỏi main thread.
- OVS 2.4: tunable `n-handler-threads`.
- OVS 2.7: kernel batch upcall (`OVS_PACKET_ATTR_MRU`).
- OVS 2.13: default đổi thành `CPU/2`.
- OVS 2.17 (baseline): batched flow install qua `OVS_FLOW_CMD_NEW` array, giảm syscall.

#### Trường hợp sự cố thực tế

Phần 9.32 "handler starvation case": host 32-core chạy DPDK PMD chiếm 8 core, OVS userspace handler default 16 thread bị stuck do contention trên `ofproto_mutex` khi controller push update đồng thời lúc traffic tăng. Coverage `miss_upcall` 50K/sec nhưng `flow_install` chỉ 12K/sec → backlog. Kernel `lost` counter tăng. Mitigation: tách 2 bridge riêng cho 2 nhóm port, mỗi bridge có handler pool độc lập.

#### Lỗi thường gặp + tín hiệu chẩn đoán

| Triệu chứng | Nguyên nhân | Kiểm chứng |
|-------------|-------------|------------|
| `lost` > 0 trong `dpctl show` | Handler không drain kịp | `coverage/show miss_upcall` >> `flow_install` |
| Latency packet đầu cao bất thường | Quá ít thread | `upcall/show` keys imbalance |
| CPU 1 core 100% trong khi khác idle | Hash collision Netlink PID | `top -H -p $(pgrep ovs-vswitchd)` |
| TCP SYN retry trên flow mới | Handler nghẽn lúc burst | `tcpdump` thấy duplicate SYN |

#### So sánh với hệ khác

Tương đương với BPF tail-call dispatch trong Cilium nhưng ngược chiều: BPF chạy trong kernel, OVS handler chạy userspace. NF tables `ipset` hash table không có upcall vì rule cố định; OVS ngược lại program rule động qua OF, cần slow path xử lý lần đầu. Hardware switch ASIC dùng "exception path" sang CPU control plane khi miss, vai trò handler tương tự.

### 9.2.15.3 `ukey` (unique key)

#### Khái niệm

Ukey là cấu trúc dữ liệu userspace (`struct udpif_key`) đại diện cho một megaflow đã cài trong kernel. Nó là "shadow copy" của megaflow đó tại userspace, lưu key, mask, action, các counter (last seen, n_packets, n_bytes), và trạng thái lifecycle. Ukey là đối tượng mà revalidator thread duyệt; không có ukey thì revalidator không có gì để check.

#### Lịch sử + bối cảnh

Xuất hiện cùng đợt refactor "facet/subfacet → ukey" OVS 2.0. Trước đó facet/subfacet phức tạp hơn, scale kém với cache lớn. Ukey được thiết kế tinh gọn: 1 ukey per 1 megaflow trong kernel, tra cứu qua hashmap khóa hash của flow key.

#### Vị trí trong kiến trúc

Ukey sống trong userspace, riêng cho mỗi `dpif`. Tổ chức trong `struct umap` (hash table chia shard để tránh contention giữa N handler + M revalidator). Truy cập qua `ukey_install()` (handler tạo), `ukey_lookup()` (revalidator query), `ukey_delete()` (cleanup).

#### Vai trò

(1) Track megaflow lifecycle giữa userspace và kernel; (2) lưu metadata cần cho revalidate (xlate context, OF rule reference); (3) cache stats để thực hiện rate limiting và monitoring; (4) phối hợp giữa handler (creator) và revalidator (consumer) qua atomic state machine.

#### Vì sao sinh ra

Kernel chỉ biết "key + mask + action + counter". Không biết megaflow này phát sinh từ rule OF nào, cần re-validate ra sao. Userspace cần một bản sao đối ngẫu để giữ thông tin. Đây là cấu trúc cầu nối kernel-userspace ngầm, hoàn toàn invisible với OF controller nhưng critical cho correctness.

#### Cơ chế hoạt động

State machine 5 state: `INITIAL → VISIBLE → CHECKING → EVICTING → FREED` (chi tiết §9.2.11.1). Transitions:

```
        INITIAL  ── handler install kernel ──▶ VISIBLE
                                                  │
                 revalidator sweep ───────────────┤
                                                  ▼
                                              CHECKING
                                                  │ pipeline match? ─ no ─▶ EVICTING ─▶ FREED
                                                  │ ─ yes ─▶ VISIBLE (stay)
```

RCU grace period (~25 ms) sau EVICTING để đảm bảo không thread nào còn đọc ukey đó (handler có thể đang xlate trên cùng key).

#### Cách kỹ sư vận hành thành thạo

Engineer cần biết: (a) `memory/show udpif keys` cho ukey count global; (b) ratio `udpif keys / rules` đo coverage cache; (c) `udpif keys` > `rules` là pathology (ukey leak, đã có commit `180ab2fd635e` fix); (d) đọc state machine để hiểu race condition khi debug.

#### Cách quan sát + xác minh

```
$ sudo ovs-appctl memory/show
udpif keys:1247
rules:892
ports:23
ofconns:3
```

Ratio 1247/892 = 1.4 → bình thường (mỗi rule trung bình spawn ~1.4 megaflow do field wildcard khác nhau). Ratio < 0.5 → cache cold (traffic chưa đa dạng). Ratio > 5 → fragmentation, có thể do action chain phụ thuộc field rộng.

```
$ sudo ovs-appctl coverage/show | grep ukey
ukey_invalid                    23 / sec
ukey_dp_change                  4 / sec
```

#### Source code tham chiếu

Struct `udpif_key` định nghĩa trong `ofproto/ofproto-dpif-upcall.c` (branch-2.17). Hàm `ukey_create_from_upcall()` (handler tạo), `ukey_install()` (insert vào hashmap), `ukey_lookup()` (revalidator), `ukey_delete()` + `ukey_destroy()` (cleanup with RCU).

#### Khác biệt giữa các phiên bản

- OVS 2.0: introduce.
- OVS 2.4: tách shard `umap` để giảm contention.
- OVS 2.7: dùng UFID làm secondary key (ngoài hash của flow key) để dump nhanh hơn.
- OVS 2.13: optimize batched delete.
- OVS branch-2.17: leak fix `180ab2fd635e` cho corner case khi handler crash giữa install và register.

#### Trường hợp sự cố thực tế

Phần 9.1.Z OOM case: cluster OpenStack với 200 chassis, mỗi chassis chạy ovs-vswitchd, OVN port_binding spam update gây ukey leak chậm (1 ukey/giờ/chassis), sau 30 ngày `ovs-vswitchd` RSS đạt 8GB, OOM-kill bởi kernel. Fix bằng upgrade lên branch-2.17 chứa commit `180ab2fd635e`, kèm cron `ovs-appctl revalidator/purge` mỗi 24 giờ như defensive.

#### Lỗi thường gặp + tín hiệu chẩn đoán

| Triệu chứng | Nguyên nhân | Kiểm chứng |
|-------------|-------------|------------|
| `udpif keys` >> `rules` | Ukey leak | `memory/show` ratio > 5 |
| RSS `ovs-vswitchd` tăng monotonic | Leak chậm | `ps -p $(pgrep ovs-vswitchd) -o rss` qua 7 ngày |
| Revalidator không xóa ukey idle | Bug version cũ | Compare version với fix `180ab2fd635e` |

#### So sánh với hệ khác

Linux nf_conn struct trong conntrack tương tự: per-flow shadow state, RCU-protected, có cleanup thread. eBPF map LRU không có shadow ở userspace vì map sống trong kernel; thay vào đó dùng `bpftool map dump`. Cisco TCAM entry không có shadow vì hardware tự manage; software mirror là phần mềm NMS (Netconf collector).

### 9.2.15.4 `UFID` (Unique Flow ID)

#### Khái niệm

UFID là một định danh 128-bit (UUID-style) gán cho mỗi megaflow khi handler thread cài về kernel. Mục tiêu: thay thế việc gửi toàn bộ flow key (có thể vài trăm byte) trong mọi Netlink message; chỉ cần gửi UFID (16 byte) và hai bên (kernel + userspace) cùng tra cứu cấu trúc tương ứng. Nhờ đó dump cycle revalidator giảm lưu lượng Netlink rất đáng kể.

#### Lịch sử + bối cảnh

Introduced OVS 2.4 (2015), bật mặc định OVS 2.7 (2017). Trước UFID, mỗi `OVS_FLOW_CMD_GET` dump phải gửi cả `OVS_FLOW_ATTR_KEY` (flow key TLV). Với 50K megaflow, dump cycle gửi ~10MB qua Netlink mỗi 500 ms → bottleneck. UFID giảm còn ~800KB.

#### Vị trí trong kiến trúc

UFID là attribute trong Netlink protocol giữa userspace và `openvswitch.ko`. Cụ thể `OVS_FLOW_ATTR_UFID` (TLV type) trong `<linux/openvswitch.h>`. Kernel side: lưu UFID trong `struct sw_flow_id` field `ufid`. Userspace side: lưu trong `struct udpif_key` field `ufid`.

#### Vai trò

(1) Compress Netlink payload cho dump cycle; (2) cho phép kernel-side hash flow theo UFID thay vì compute hash từ flow key; (3) đơn giản hóa `OVS_FLOW_CMD_DEL` (chỉ cần UFID, không cần resend full key); (4) hỗ trợ batch operation hiệu quả hơn (array of UFID thay vì array of full key).

#### Vì sao sinh ra

Khi cache size tăng từ vài ngàn (OVS 1.x facet) lên hàng trăm ngàn (OVS 2.x megaflow), Netlink overhead trở thành bottleneck. UFID là tối ưu rõ rệt: 16 byte / 200 byte = 92% reduction trên wire.

#### Cơ chế hoạt động

Khi handler tạo megaflow, gọi `dpif_flow_hash()` để generate UFID = SHA-1 hash của (flow key + mask) truncated 128-bit. Gửi `OVS_FLOW_CMD_NEW` kèm cả `OVS_FLOW_ATTR_KEY` lẫn `OVS_FLOW_ATTR_UFID`. Kernel cache cả hai, nhưng hashmap secondary key trên UFID. Sau đó revalidator dump chỉ với `OVS_UFID_F_OMIT_KEY` và `OVS_UFID_F_OMIT_ACTIONS` flag → kernel trả về stats theo UFID, không gửi key/action lại nếu không thay đổi.

#### Cách kỹ sư vận hành thành thạo

Engineer cần biết: (a) UFID enabled khi `ufid enabled : true` trong `upcall/show`; (b) hostkernel < 4.0 không hỗ trợ UFID, cần fallback (rare ngày nay); (c) `ovs-appctl dpctl/dump-flows --ufid` hiển thị UFID; (d) `ofproto/trace --ufid <UFID>` nhanh hơn full key trong scripting.

#### Cách quan sát + xác minh

```
$ sudo ovs-appctl upcall/show | grep ufid
  ufid enabled : true

$ sudo ovs-appctl dpctl/dump-flows --more
ufid:9b8a7c6d-1234-5678-9abc-def012345678,recirc_id(0),in_port(2),...
  packets:1247, bytes:91234, used:0.123s, actions:output(3)
```

UFID hiển thị dưới dạng UUID. Khi dùng `dpctl/del-flows --ufid 9b8a7c6d-...` có thể xóa entry mà không cần copy-paste cả flow key.

#### Source code tham chiếu

Hàm `dpif_flow_hash()` trong `lib/dpif.c` (compute UFID). Hàm `odp_ufid_from_string()` + `odp_ufid_to_string()` trong `lib/odp-util.c` (serialize/deserialize). Struct `ovs_u128` định nghĩa trong `lib/util.h`. Kernel side: `struct sw_flow_id` trong `net/openvswitch/flow.h`.

#### Khác biệt giữa các phiên bản

- OVS 2.4 (2015): introduce, opt-in qua kernel feature detect.
- OVS 2.7 (2017): bật mặc định khi kernel hỗ trợ.
- Linux 4.0+ kernel: `OVS_FLOW_ATTR_UFID` recognized.
- OVS 2.13: `--ufid` flag trên `dpctl` CLI.
- OVS 2.17 (baseline): UFID mandatory cho dpif-netdev (DPDK), optional cho dpif-netlink (kernel) khi backward-compat.

#### Trường hợp sự cố thực tế

Production case Rackspace 2018: upgrade kernel host 3.10 → 4.4 nhưng giữ OVS 2.5; UFID disabled vì code path còn check kernel version, không recognize 4.4 đầy đủ. Hậu quả: dump cycle tốn ~30% CPU thay vì ~5%. Diagnose qua `upcall/show` thấy `ufid enabled : false`, fix bằng upgrade OVS lên 2.7+. Bài học: UFID enable phụ thuộc cả userspace lẫn kernel.

#### Lỗi thường gặp + tín hiệu chẩn đoán

| Triệu chứng | Nguyên nhân | Kiểm chứng |
|-------------|-------------|------------|
| `ufid enabled : false` | Kernel cũ hoặc OVS cũ | `uname -r` + `ovs-vsctl --version` |
| Dump cycle CPU cao | UFID chưa active | `coverage/show flow_dumps_done` per cycle |
| `dpctl/dump-flows` không hiển thị UFID | CLI không pass `--more` | Re-run với `--more` |

#### So sánh với hệ khác

Linux conntrack có `nf_conn->ext` chứa "id" 32-bit nhưng không phục vụ wire protocol như UFID. eBPF map dùng key trực tiếp, không cần UFID-style indirection vì map sống trong kernel. SR-IOV NIC offload dùng "flow handle" 32-bit do firmware cấp, vai trò gần giống UFID nhưng scope hardware. P4Runtime dùng `MatchEntry` ID 64-bit, cùng motivation giảm wire overhead.

### 9.2.15.5 `fast path / slow path`

#### Khái niệm

Cặp khái niệm chia xử lý packet OVS thành hai con đường: **fast path** là xử lý hoàn toàn trong kernel datapath (`openvswitch.ko`) sau khi megaflow đã cache, không cần thoát context kernel; **slow path** là xử lý trong userspace `ovs-vswitchd` qua handler thread, kích hoạt khi cache miss hoặc khi action yêu cầu (ví dụ `controller`, `userspace` action). Fast path đo bằng nano/microsecond; slow path đo bằng vài chục đến vài trăm microsecond.

#### Lịch sử + bối cảnh

Cặp khái niệm có gốc từ thiết kế cache truyền thống (CPU L1 vs main memory). Áp dụng cho networking từ Linux NAPI (1990s) và sau đó nf_conntrack fast path. OVS thừa kế và làm rõ ràng từ ngày đầu (2009): kernel datapath là fast path, userspace là slow path. Phần 9.2.4 đã giới thiệu; mục này tổng hợp lại đầy đủ.

#### Vị trí trong kiến trúc

```
                        ┌───────────────────────────┐
Userspace               │  ovs-vswitchd             │
                        │   handler thread (SLOW)   │
                        │   xlate_actions, cache    │
                        └────────────▲──────────────┘
                                     │ Netlink upcall
                        ┌────────────┴──────────────┐
Kernel                  │  openvswitch.ko (FAST)    │
                        │   megaflow lookup, exec   │
                        └───────────────────────────┘
```

Fast path = box dưới. Slow path = box trên + kênh Netlink + box dưới (round trip).

#### Vai trò

Fast path: xử lý tuyệt đại đa số packet steady-state với chi phí thấp. Slow path: xử lý packet đầu tiên của mỗi flow, packet đặc biệt cần controller, hoặc packet không match cache vì lý do bất kỳ. Tỷ lệ fast/slow path = hit rate, là KPI hàng đầu.

#### Vì sao sinh ra

Không thể tránh hoàn toàn slow path: lần đầu mọi flow đều miss vì cache chưa có entry. Không thể tránh hoàn toàn fast path: nếu để mọi packet vào userspace, throughput sẽ rớt 10-100x do context switch + Netlink overhead. Kiến trúc 2 lớp là compromise hợp lý.

#### Cơ chế hoạt động

Fast path latency điển hình: 1-3 microsecond per packet (kernel datapath lookup + execute). Slow path latency điển hình: 50-200 microsecond per packet (Netlink up + xlate + Netlink down + execute). Ratio ~50x. Steady-state target: hit rate > 99%, miss rate < 1%, lost = 0.

USC Lab 9 minh họa rõ: lần đầu `ping h1 → h2`, ICMP echo qua slow path (~150 µs), packet thứ 2 trở đi qua fast path (~2 µs). Lệnh `ovs-dpctl dump-flows` cho thấy megaflow xuất hiện sau packet đầu.

#### Cách kỹ sư vận hành thành thạo

Engineer cần biết: (a) đọc `ovs-dpctl show` để biết hit/missed/lost; (b) tính hit rate; (c) hiểu tại sao kết nối TCP đầu tiên có RTT cao bất thường (slow path) sau đó bình thường (fast path); (d) khi nào miss rate tăng là bình thường (traffic mới) vs bất thường (rule churn, cache size không đủ).

#### Cách quan sát + xác minh

```
$ sudo ovs-dpctl show
system@ovs-system:
        lookups: hit:9876543 missed:12345 lost:0
        flows: 1247
        masks: hit:9876543 total:9888888 hit/pkt:1.00
```

Hit rate = 9876543 / (9876543 + 12345) = 99.875% → ổn. Nếu `lost > 0` → kernel drop khi handler không drain kịp.

```
$ sudo ovs-appctl coverage/show | grep -E "(hit|miss|lost)"
flow_dumps_done                 4567 / sec
miss_upcall                     12 / sec
flow_lookup_extracted_kernel    9876 / sec
```

#### Source code tham chiếu

Fast path: `ovs_dp_process_packet()` trong `net/openvswitch/datapath.c` (Linux kernel tree, không phải OVS userspace tree). Slow path: `udpif_upcall_handler()` trong `ofproto/ofproto-dpif-upcall.c`. Chuyển giao: `ovs_dp_upcall()` trong kernel + `recv_upcalls()` trong userspace.

#### Khác biệt giữa các phiên bản

Khái niệm fast/slow path bất biến. Implementation cải tiến qua versions:

- OVS 1.x (microflow): fast path quá hẹp, miss rate cao trong production.
- OVS 2.0 (megaflow): fast path mở rộng đáng kể, hit rate 99%+ phổ biến.
- OVS 2.x DPDK (PMD): fast path hoàn toàn userspace (dpif-netdev), slow path vẫn là handler thread cùng tiến trình.
- OVS 2.7+ AF_XDP: alternative datapath bypass kernel.
- OVS 2.17 (baseline): fast path kernel hoặc DPDK; slow path không đổi mặt khái niệm.

#### Trường hợp sự cố thực tế

Cloud provider 2020 case "first-packet TCP RTT jitter": ứng dụng phụ thuộc latency phàn nàn TCP handshake đôi khi 50ms thay vì 2ms. Diagnose: chassis có cron flush conntrack mỗi 5 phút, kéo theo `ovs-appctl revalidator/purge` xóa toàn bộ megaflow. Sau purge, mọi flow mới đều miss → slow path 150 µs cộng dồn cho 3 packet handshake. Fix: ngừng cron, để revalidator tự cleanup theo idle_timeout.

#### Lỗi thường gặp + tín hiệu chẩn đoán

| Triệu chứng | Nguyên nhân | Kiểm chứng |
|-------------|-------------|------------|
| Hit rate < 95% steady-state | Cache size nhỏ, traffic đa dạng | `ovs-dpctl show` + `upcall/show` |
| Latency packet đầu cao bất thường | Slow path saturated | `tcpdump` SYN RTT vs SYN-ACK |
| `lost > 0` | Handler nghẽn | `coverage/show miss_upcall` rate |
| Throughput không scale theo NIC line rate | Slow path bottleneck | Tăng `n-handler-threads` |

#### So sánh với hệ khác

- Linux conntrack có "fast path" (cached `nf_conn` lookup) và "slow path" (`nf_conntrack_in` resolve), nhưng granularity là per-connection (5-tuple) không phải per-megaflow.
- Cilium eBPF không có slow path explicit: BPF map miss = drop hoặc fallback policy, không có upcall.
- Cisco ASIC switch: fast path = TCPAM/SDK lookup, slow path = CPU control plane (CoPP). Tương đồng kiến trúc.
- VPP có dual-path tương tự: vector graph node nhưng không có cache miss style upcall.

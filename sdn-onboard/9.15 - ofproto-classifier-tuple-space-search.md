## 9.15.1 Role of the classifier inside the OVS stack

The classifier is the central data structure of **`ovs-vswitchd`**. It stores every OpenFlow rule in the switch and lets the daemon look up the rule that matches an arbitrary packet header. When a packet enters OVS, if it does not match the megaflow cache in the kernel, the packet is upcalled to `ovs-vswitchd`. There the classifier is queried with the packet header, returns the highest-priority matching rule, and that rule is compiled into a megaflow that is pushed down to the kernel datapath so similar packets that follow take the fast path.

The problem the classifier solves: given a flow table with N rules that match on complex fields (a 128-bit IPv6 address, a 12-tuple of L2 to L4 fields), find the rule that matches packet P quickly. The simplest approach is a linear scan over the N rules, which is O(N) and not feasible at N = 10,000 rules combined with a 1 Mpps packet rate. We need an O(log N) or amortized O(1) algorithm.

Common packet-lookup approaches:
- **Linear scan:** O(N). Simple, slow.
- **Binary tree (B-tree):** O(log N). Good for single-column lookup, complex for multi-field matching with wildcards.
- **Hash map:** O(1). Works only for exact match, cannot handle wildcards (ternary match).
- **TCAM (Ternary CAM) hardware:** O(1). A specialized hardware solution, not applicable to a software switch.
- **Decision tree (HiCuts, HyperCuts):** O(log N), but memory usage grows exponentially with the number of fields.
- **Tuple space search (TSS):** O(k) where k is the number of tuples. The algorithm OVS uses.

OVS picks TSS for three reasons: (1) it handles wildcard matching naturally, (2) memory usage is linear in the number of flows (not exponential), and (3) flow-table updates are O(1) when a rule is added or removed.

## 9.15.2 Tuple space search (TSS), the core idea

TSS was developed by Srinivasan et al. in the paper "Packet Classification Using Tuple Space Search" (SIGCOMM 1999). The core idea: split the flow table into multiple **subtables**, where each subtable holds rules with the same **pattern mask** (the same set of bits used for matching). Lookup means querying every subtable with a hash and taking the result with the highest priority.

**Definition of a tuple:** a tuple is a vector of bit masks that says "these bits are matched exactly, those bits are wildcarded". For example, in an IPv4 flow table:
- Tuple A: match src IP /24 plus dst IP /16, mask `(0xFFFFFF00, 0xFFFF0000, ...)` for src and dst IP.
- Tuple B: match dst port plus protocol only, mask `(0, 0, 0xFFFF, 0xFF)` for src and dst IP, port, and proto.
- Tuple C: match src MAC plus VLAN ID, mask `(0xFFFFFFFFFFFF, 0xFFF, 0, ...)`.

Every rule belongs to exactly one tuple, determined by the rule's mask fields. Rules with the same mask end up in the same tuple, which means in the same subtable.

**Lookup algorithm:**
1. For packet P, walk the list of tuples in the classifier.
2. For each tuple T: apply mask T to packet P, producing key K = P AND T.mask. Query the hash map of subtable T with key K.
3. If the lookup hits, take the matching rule and its priority.
4. After visiting every tuple, return the highest-priority rule among the matches.

**Complexity:** O(k) where k is the number of tuples. In production OVS deployments, k is usually 3 to 20 (about 10 rules per tuple on average). This is why the OVS classifier sustains millions of lookups per second per CPU core.

**OVS data structures:**
```c
struct classifier {
    struct cls_subtable *subtables[];  // list of subtables
    // ... index metadata ...
};

struct cls_subtable {
    struct cmap rules;            // hash map: key (from mask) to cls_rule
    struct minimatch mask;         // mask of this subtable
    int max_priority;              // highest priority in the subtable
    // ... other metadata ...
};

struct cls_rule {
    struct minimatch match;        // match fields
    int priority;                  // priority
    // ... action data ...
};
```

`cmap` (a Cuckoo hash map) is the lock-free hash map of OVS, lock-free on the read path, atomic operations on the write path. It supports many concurrent readers plus a single writer without contention.

## 9.15.3 Staged lookup, an optimization for priority sort

Naive TSS has a problem: it must walk every tuple to find the highest-priority rule, even if the first tuple already contains a very high-priority match. OVS optimizes with **staged lookup**: tuples are sorted by `max_priority` (highest first). If a matching rule has priority greater than the `max_priority` of every remaining tuple, the search can exit early.

Pseudo-code:
```
function tss_lookup(packet P, classifier C):
    best_rule = None
    best_priority = -infinity
    for tuple T in C.tuples sorted by T.max_priority DESC:
        if T.max_priority < best_priority:
            break  // exit early, no remaining tuple can improve the result
        key = apply_mask(P, T.mask)
        rule = cmap_find(T.rules, key)
        if rule and rule.priority > best_priority:
            best_rule = rule
            best_priority = rule.priority
    return best_rule
```

In practice, if rule priorities are distributed sensibly (a few high-priority rules for drop or allow, many low-priority rules for the default path), the early-exit fires within the first one or two tuples and lookup approaches O(1).

## 9.15.4 Interaction with the megaflow cache

A classifier lookup at O(k) is still slower than an O(1) hash lookup in the kernel. Each upcall costs roughly a microsecond per packet, which is not enough for high-rate traffic. The **megaflow cache** in the kernel datapath solves this: it caches the result of the classifier for "flow classes" (packets that map to the same classifier output).

**Caching process:**
1. Packet P misses the megaflow, an upcall fires to `ovs-vswitchd`.
2. The classifier returns rule R with match mask M.
3. `ovs-vswitchd` builds the megaflow: match equals P AND M (the subset of bits that actually matter), action equals R.action.
4. The megaflow is installed into the kernel.
5. A later packet P' with the same pattern (P' AND M equals P AND M) matches the megaflow, the kernel forwards directly without an upcall.

A key point: the match fields of the megaflow are a SUBSET of the match fields of the rule, only the bits that actually influenced the decision. If the rule says "match dst_ip /24" but during the classifier lookup the only bits that mattered were dst_ip bits 0 to 23 plus dst_port (used by the priority-sort early-exit), the megaflow ends up matching dst_ip /24 plus dst_port.

The **classifier** is the "slow path" (thousands of pps), the **megaflow** is the "fast path" (millions of pps). With a stable workload, more than 99 percent of packets hit the megaflow, so the classifier is rarely invoked.

## 9.15.5 Observation, the classifier in action

OVS provides several tools to inspect the classifier:

**View subtable count and rule count:**
```
[host]# ovs-appctl dpif/show-dp-features br0
```

**Dump OpenFlow rules sorted by priority:**
```
[host]# ovs-ofctl -O OpenFlow15 dump-flows br0 --sort=priority
```

**Count upcalls (the number of classifier invocations):**
```
[host]# ovs-appctl upcall/show
```
Output shows the per-thread upcall rate and the megaflow hit-to-miss ratio.

**Flow-table statistics:**
```
[host]# ovs-appctl bridge/dump-flows br0
```
Shows the internal flow-table structure plus subtable partitioning.

**Dump the megaflow table:**
```
[host]# ovs-appctl dpif/dump-flows br0
```
Output lists the megaflows currently active in the kernel, the cache of classifier lookups.

**Signs of a poorly balanced flow table:** if the upcall rate is high (above 1000 per second) and the megaflow hit ratio is low (below 95 percent), the flow table is not yet tuned. Common causes:
- Too many rules with different pattern masks (k is large).
- A bad priority distribution (many rules at the same high priority).
- Highly variable traffic patterns (the cache cannot hold the working set).

## 9.15.6 Tuning a flow table for classifier efficiency

Practical guidance when designing an OVS flow table:

**(1) Group rules with the same pattern match:** if 100 rules all match on dst_ip /24 (differing only in priority or action), they all live in the same subtable, one tuple, so this group is looked up in O(1).

**(2) Avoid unnecessary wildcards:** a tighter match (more exact bits) means fewer subtables. A wildcard bit (mask 0) introduces a new bit in the subtable pattern.

**(3) Use priority sparingly:** a flow table with 10 distinct priority levels needs 10 priority groups for the priority sort. A flow table with 3 priorities (high, mid, low) needs only 3 groups. Fewer priority levels make the early-exit more effective.

**(4) Avoid deep matching chains:** every packet that traverses many tables (resubmit) means many classifier lookups. Compile into a single table when possible.

**(5) Monitor the subtable count:** use `ovs-appctl dpif/show` to track it. An abnormal increase in subtable count can signal flow churn (rules being added and removed continuously).

**(6) Lean on megaflow learning:** the megaflow cache automatically learns the relevant subset of bits. The programmer does not need to optimize this. OVS does it. The only thing to avoid is writing rules with conflicting subsets (rule 1 depends on bit X, rule 2 depends on bit Y, the two do not interact, so many megaflow variants are produced).

## 9.15.7 Subtable structure, inside the tuple space

§9.15.2 to §9.15.3 introduced TSS plus staged lookup conceptually. This section dives into the **subtable** as a data structure, how OVS actually organizes memory and performs the lookup inside one subtable.

### 9.15.7.1 Subtable anatomy

Each subtable in the OVS classifier (source `lib/classifier.c`) is the structure:

```
struct cls_subtable {
    struct cmap rules;              /* hash table: mask(flow) -> list of rules */
    struct minimask mask;            /* mask of this subtable */
    int max_priority;                /* highest priority in this subtable */
    unsigned int max_count;          /* number of rules at max_priority */

    /* staged lookup stages */
    struct cls_subtable_entry ports_mask;   /* L4 port stage */
    struct cls_subtable_entry metadata_mask; /* metadata stage */
    struct cls_subtable_entry fields_mask;   /* all other fields */
};
```

**Invariant:** every rule in one subtable has the **same mask** (same fields matched, same mask bits). Different subtables have different masks.

### 9.15.7.2 Per-subtable lookup flow

When the classifier receives a packet, for each subtable:

1. Extract the flow key from the packet according to the subtable `mask`.
2. Compute a hash of the masked key.
3. Look up the `cmap` (a concurrent hash map).
4. On hit, return the best-priority rule in the bucket.
5. On miss, continue to the next subtable.

**Optimization:** if a subtable has `max_priority < current_best_priority_found`, OVS skips that subtable entirely (it cannot win). This is why subtables are sorted by `max_priority` descending.

### 9.15.7.3 Reading masked output from `dpctl/dump-flows`

When dumping kernel megaflows:

```shell
$ sudo ovs-dpctl dump-flows -m | head -3
recirc_id(0),in_port(3),eth(src=aa:bb:cc:00:00:01/ff:ff:ff:00:00:00,dst=aa:bb:cc:00:00:02/ff:ff:ff:ff:ff:ff),
  eth_type(0x0800),
  ipv4(src=10.1.0.0/255.255.0.0,dst=10.1.0.10,proto=6/0,tos=0/0,ttl=0/0,frag=no),
  tcp(src=0/0,dst=80/0xffff),
  packets:18234, bytes:2145678, used:0.082s, actions:4
```

**Anatomy of a mask field:**

| Field:value/mask | Inspect level |
|---|---|
| `eth(src=.../ff:ff:ff:00:00:00)` | OUI match (first 3 bytes), host wildcarded |
| `eth(dst=.../ff:ff:ff:ff:ff:ff)` | Exact MAC |
| `ipv4(src=10.1.0.0/255.255.0.0)` | /16 prefix match |
| `ipv4(dst=10.1.0.10)` | Exact IP (no slash, implicit /32) |
| `ipv4(proto=6/0)` | Wildcard, **proto is not actually matched even though "6" is shown** (mask 0 means ignore) |
| `tcp(src=0/0)` | Full wildcard on src port |
| `tcp(dst=80/0xffff)` | Exact port 80 |

**This subtable has the mask:** `in_port + eth_src[47:24] + eth_dst + eth_type + ipv4_src[31:16] + ipv4_dst + tcp_dst`. A 7-field mask. Rules in the same subtable match these bits exactly; differing bit values mean different rules.

## 9.15.8 Prefix-tree (trie) optimization for IP classification

### 9.15.8.1 The problem: IP-range matches do not hash well

Hash-based subtable lookup is fast for **exact masks** but not for **prefix matches**. A rule with `nw_src=10.0.0.0/8` matches every IP in 10.x.x.x. Splitting 10.x.x.x into 256^3 (about 17 million) exact entries is not feasible.

How TSS handles prefixes:
- One subtable per unique mask length (/8, /16, /24, /32).
- Look up each subtable, at most O(N_subtable) where N is at most 32 for IPv4.

But if the rule set has hundreds of distinct prefix lengths, hundreds of subtables result and lookup slows down.

### 9.15.8.2 The prefix-tree solution

OVS uses a **Patricia trie** for IP matching inside the classifier (sources `lib/pvector.c` plus `lib/classifier.c`). Structure:

```
Patricia trie for IPv4 src rules:
                     root
                   /      \
              0.0.0.0      128.0.0.0
                /            /    \
           10.0.0.0      128.0.0.0  192.0.0.0
              |          /              |
         10.0.0.0/8    ...          192.168.0.0/16
              |
        10.1.0.0/16
              |
        10.1.0.0/24
```

Lookup for IP 10.1.0.5:
1. Start at the root, bit 0 is 0, go left.
2. Bit 1 is 0, go left.
3. Continue bit by bit until reaching the leaf with the longest matching prefix.
4. Return the rule with that longest prefix match.

Complexity is O(32) for IPv4 bit-by-bit, or O(log N) amortized with compression.

### 9.15.8.3 Combining trie plus TSS

The OVS classifier **does not use the trie for every mask**, only for fields with many distinct prefixes (typically `nw_src`, `nw_dst`). Other fields (`eth_src`, `tp_dst`) keep the subtable hash.

To check the classifier's behaviour: run `coverage/show classifier`, the counter named `classifier_prefix_tree` increases when a prefix lookup hits.

## 9.15.9 Scenarios that break classifier performance

### 9.15.9.1 Subtable explosion

**Signature:**
- `pmd-stats-show` reports `avg. subtable lookups per megaflow hit` greater than 10.
- `coverage/show classifier_lookup` rate is high.
- CPU spikes on the `ofproto` thread.

**Root cause:** too many rules with unique masks. Every rule creates its own subtable, and the cardinality of the tuple space grows.

**Fix:** normalize the rule set, merge rules whose masks differ by only a few bits into a wider mask.

### 9.15.9.2 Priority sort churn

**Signature:**
- Rules are added or deleted continuously, so the classifier re-sorts subtables by `max_priority`.
- `coverage/show classifier_rule_add` plus `classifier_rule_del` rates are high.

**Root cause:** the controller installs rules non-atomically, an `add + del + add` pattern that triggers sort overhead.

**Fix:** use `ovs-ofctl --bundle replace-flows` for atomic bundles in OpenFlow 1.4 and later.

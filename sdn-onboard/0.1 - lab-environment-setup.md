# 0.1. Thiết lập môi trường lab

Môi trường thực hành: Ubuntu 22.04 LTS (Jammy). Tham khảo chính: [Installing Open vSwitch, upstream docs](https://docs.openvswitch.org/en/latest/intro/install/), [OVN installation guide](https://docs.ovn.org/en/latest/intro/install/).

---

## Mục tiêu bài học

Sau khi hoàn thành phần này, bạn có thể:

1. **Cài đặt (Apply)** hai cấu hình lab tương ứng hai giai đoạn của series: single-node all-in-one cho Khối I-XI, two-node chassis pair cho Khối XII-XIII+.
2. **Xác minh (Apply)** cài đặt OVS 2.17.9 và OVN 22.03.8 trên Ubuntu 22.04 Jammy bằng ba cấp kiểm tra: package version, systemd service state, runtime health qua `ovs-vsctl show` và `ovn-sbctl show`.
3. **Phân biệt (Analyze)** vai trò của kernel module `openvswitch.ko`, userspace daemon `ovs-vswitchd`, database `ovsdb-server`, và khi nào từng thành phần được gọi đến trong path xử lý packet.
4. **Chuẩn bị (Apply)** health check playbook và teardown procedure để tái sử dụng cho mọi lab tiếp theo, tránh phải dựng lại từ đầu mỗi lần gặp lỗi.

Không có phần nào trong series phụ thuộc trực tiếp vào 0.1 để hiểu khái niệm, Phần này thuần procedure. Tuy nhiên, trước khi làm theo, nên có:

## 0.1.1 Hai cấu hình lab và yêu cầu tài nguyên

Series có khoảng 16 Phần foundation + 3 Phần OVN advanced với độ phức tạp lab tăng dần, nhưng không cần một lab duy nhất cho toàn bộ. Hai cấu hình dưới đây phủ đủ nhu cầu, và từng cấu hình có thể dựng trên hardware tầm trung.

### Lab mode A, Single-node all-in-one (Khối 0 → XI)

Mục tiêu: thực hành OVS, OpenFlow, OVSDB, Mininet topology. Toàn bộ chạy trên một máy.

Yêu cầu tối thiểu: 2 vCPU, 4 GB RAM, 20 GB disk. Thực tế sau khi cài OVS + Mininet + Ryu controller, memory footprint khi idle là khoảng 300 MB, dư dả để mở đồng thời 30-50 virtual host Mininet.

Phương tiện khuyến nghị: Multipass (`sudo snap install multipass`) hoặc VirtualBox trên laptop. Multipass dùng QEMU/KVM backend trên Linux và HyperKit trên macOS, khởi tạo một VM Ubuntu 22.04 chỉ mất khoảng 30 giây.

### Lab mode B, Two-node chassis pair (Khối XII → XIII)

Mục tiêu: thực hành OVN với hai chassis để quan sát Geneve tunnel hình thành, HA chassis group failover, và multichassis binding. Đây là cấu hình tối thiểu để kiến trúc OVN NB/SB có ý nghĩa, một chassis không đủ thể hiện distributed nature.

Yêu cầu tối thiểu: 2 VM, mỗi VM 2 vCPU + 2 GB RAM + 10 GB disk. VM1 chạy `ovn-central` (NB/SB DB + northd) + `ovn-host` (chassis role), VM2 chỉ chạy `ovn-host`. Hai VM phải cùng bridge network để traffic Geneve (UDP 6081) đi qua được.

Ghi chú mạng: nếu dùng VirtualBox, chọn adapter mode "Internal Network" cho interface giữa hai VM để tránh nhiễu từ NAT của host. Trên Multipass, dùng `multipass networks` và gán `--network` cho cả hai instance để chúng cùng broadcast domain.

## 0.1.2 Ubuntu 22.04 baseline và kernel module

Series chọn Ubuntu 22.04 LTS (Jammy) làm baseline vì ba lý do kỹ thuật. Thứ nhất, Ubuntu 22.04 ship kernel 5.15 LTS mặc định, kernel này hỗ trợ đủ feature OVS 2.17 và OVN 22.03 mà không cần kernel datapath module riêng (datapath đã in-tree từ 3.3). Thứ hai, Ubuntu 22.04 có OVS 2.17.9 và OVN 22.03.8 trong pocket `jammy-updates` (tính đến 2025-08), nghĩa là `apt install` thông thường là đủ, không cần PPA hoặc Ubuntu Cloud Archive. Thứ ba, Ubuntu 22.04 có hỗ trợ dài hạn (LTS đến 2027) và là baseline phổ biến cho cả đào tạo lẫn sản xuất.

Xác minh baseline:

```
[host]$ lsb_release -a
Distributor ID: Ubuntu
Description:    Ubuntu 22.04.4 LTS
Release:        22.04
Codename:       jammy

[host]$ uname -r
5.15.0-101-generic
```

Kernel 5.15 là branch LTS của Ubuntu 22.04 GA. Nếu dùng HWE kernel (5.17, 6.2, 6.5 tùy version Ubuntu 22.04 point release), OVS vẫn hoạt động nhưng cần kiểm tra kernel module có khớp với userspace version hay không.

Kernel module `openvswitch` phải load được trước khi daemon khởi động:

```
[host]$ sudo modprobe openvswitch
[host]$ lsmod | grep openvswitch
openvswitch           176128  0
nsh                    16384  1 openvswitch
nf_conncount           24576  1 openvswitch
nf_nat                 57344  1 openvswitch
nf_conntrack          172032  4 nf_nat,openvswitch,nf_conncount,nf_conntrack_netlink
```

Module load tự động khi `openvswitch-switch.service` start, nên bước `modprobe` thủ công chỉ cần khi debug. Nếu `lsmod` không thấy `openvswitch`, kiểm tra module có tồn tại trong kernel package hiện tại chưa: `ls /lib/modules/$(uname -r)/kernel/net/openvswitch/`.

## 0.1.3 Cài đặt OVS + OVN qua apt

Ubuntu 22.04 jammy-updates cung cấp các package chính: `openvswitch-switch` (daemon + kernel module loader), `openvswitch-common` (CLI tools `ovs-vsctl`, `ovs-ofctl`, `ovs-appctl`), `ovn-central` (NB/SB DB + northd daemon), `ovn-host` (ovn-controller trên mỗi chassis), `ovn-common` (CLI tools `ovn-nbctl`, `ovn-sbctl`, `ovn-trace`).

Cài đặt:

```
[host]# apt update
[host]# apt install -y openvswitch-switch ovn-central ovn-host
```

Package dependencies tự kéo `openvswitch-common`, `ovn-common`, và các Python binding cần thiết.

Xác minh version:

```
[host]$ dpkg -l | grep -E 'openvswitch|ovn' | awk '{print $2, $3}'
ovn-central 22.03.8-0ubuntu0.22.04.1
ovn-common 22.03.8-0ubuntu0.22.04.1
ovn-host 22.03.8-0ubuntu0.22.04.1
openvswitch-common 2.17.9-0ubuntu0.22.04.1
openvswitch-switch 2.17.9-0ubuntu0.22.04.1
```

Các version này là baseline của series, mọi Guided Exercise và Lab tham chiếu đến chúng. Nếu version hiển thị khác (ví dụ 2.17.7 cũ hơn), chạy `apt upgrade openvswitch-switch ovn-central ovn-host` để lên bản mới nhất của pocket jammy-updates.

> **Lưu ý phiên bản:** Ubuntu 20.04 (Focal) chỉ có OVS 2.13 trong main repo và không có `ovn-central`, phải enable Ubuntu Cloud Archive (`add-apt-repository cloud-archive:yoga`) để lấy OVN. Ubuntu 24.04 (Noble) có OVS 3.3 + OVN 24.03 với feature RARP-based `activation-strategy` mà Phần 19 phân tích. Series dùng Ubuntu 22.04 làm baseline vì nó là version lâu dài nhất có đủ OVS 2.17 + OVN 22.03 không cần repo phụ.

Service systemd sau khi cài đã được enable + start tự động:

```
[host]$ systemctl is-active openvswitch-switch ovn-central ovn-controller
active
active
active
```

Nếu thấy `inactive` ở dòng nào, chạy `systemctl start <service>` rồi kiểm tra `journalctl -u <service> -n 50` để tìm lỗi.

## 0.1.4 Mininet cho OpenFlow labs (Khối III-IV)

Mininet tạo topology mạng ảo với host + switch + link bằng Linux network namespace và veth pair. Series dùng Mininet ở Khối III-IV để quan sát OpenFlow messages giữa controller và switch mà không cần hardware switch thật.

Mininet 2.3.0 là version chính thức đầu tiên hỗ trợ Python 3, Ubuntu 22.04 có Python 3.10 mặc định, nên tương thích. Tuy nhiên, Ubuntu 22.04 không có Mininet trong main repo, phải cài từ source hoặc từ PPA bên thứ ba.

Cài đặt từ source (khuyến nghị):

```
[host]$ git clone https://github.com/mininet/mininet.git
[host]$ cd mininet
[host]$ git checkout 2.3.0
[host]$ sudo PYTHON=python3 util/install.sh -n
```

Flag `-n` cài Mininet core (wrapper `mn`, Python library, OVS) mà không kéo thêm controller POX hoặc OpenFlow reference implementation. Nếu cần Ryu controller cho Khối IV, cài riêng: `pip install ryu`.

Xác minh Mininet:

```
[host]$ sudo mn --version
2.3.0

[host]$ sudo mn --test pingall
*** Creating network
*** Adding controller
*** Adding hosts:
h1 h2
*** Adding switches:
s1
*** Adding links:
(h1, s1) (h2, s1)
*** Configuring hosts
h1 h2
*** Starting controller
c0
*** Starting 1 switches
s1 ...
*** Ping: testing ping reachability
h1 -> h2
h2 -> h1
*** Results: 0% dropped (2/2 received)
```

Output `0% dropped` xác nhận Mininet + OVS + default controller hoạt động end-to-end. Nếu thấy `100% dropped`, kiểm tra bridge `s1` đã được tạo: `sudo ovs-vsctl show`.

## 0.1.5 Health check playbook, dùng cho mọi lab

Sau mỗi lần dựng lab hoặc sau reboot, chạy playbook này để xác nhận mọi component sẵn sàng trước khi đi vào bài tập. Tránh mất giờ debug sau khi đã làm nửa bài mới phát hiện service chết.

### Kiểm tra OVS (lab mode A, B đều cần)

```
[chassis]$ systemctl is-active openvswitch-switch
active

[chassis]$ ovs-vsctl show
92a54bc6-6e75-4f02-9b4c-95c01dbd5f03
    Bridge br-int
        fail_mode: secure
        Port br-int
            Interface br-int
                type: internal
    ovs_version: "2.17.9"

[chassis]$ ovs-appctl dpif/show
system@ovs-system: hit:0 missed:0
  br-int:
    br-int 65534/3: (internal)
```

Dòng `ovs_version: "2.17.9"` xác nhận userspace version khớp với package, đây là check duy nhất đảm bảo không có mismatch giữa userspace và kernel module.

### Kiểm tra OVN (chỉ lab mode B)

```
[central]$ ovn-nbctl show
[central]$ ovn-sbctl show
Chassis "chassis-vm1"
    hostname: chassis-vm1
    Encap geneve
        ip: "10.0.0.11"
        options: {csum="true"}
Chassis "chassis-vm2"
    hostname: chassis-vm2
    Encap geneve
        ip: "10.0.0.12"
        options: {csum="true"}
```

Output `ovn-sbctl show` liệt kê đủ số chassis kỳ vọng (2 cho lab mode B) xác nhận Geneve encap đã được register. Nếu thiếu chassis, kiểm tra `ovn-controller` trên chassis đó: `journalctl -u ovn-controller -n 50`.

### Kiểm tra Geneve tunnel hình thành (lab mode B)

```
[chassis-vm1]$ ovs-appctl ofproto/list-tunnels
port  type  local_ip  remote_ip  local_port  remote_port
4     geneve  10.0.0.11  10.0.0.12  6081  6081
```

Dòng này xuất hiện chỉ khi chassis-vm2 đã register vào SB DB. Nếu `ofproto/list-tunnels` rỗng, ovn-controller chưa đủ thông tin về peer chassis.

## 0.1.6 Teardown và reset procedure

Mỗi lab kết thúc để state sạch, không để state của lab trước ảnh hưởng lab sau.

### Reset OVS về state empty

```
[chassis]# systemctl stop openvswitch-switch
[chassis]# rm /etc/openvswitch/conf.db
[chassis]# systemctl start openvswitch-switch
[chassis]$ ovs-vsctl show
<UUID>
    ovs_version: "2.17.9"
```

Xóa `conf.db` và restart làm OVS quay về state như vừa cài. Nếu không xóa, bridge và flow rule của lab trước vẫn còn.

### Reset OVN về state empty

```
[central]# systemctl stop ovn-central
[central]# rm /var/lib/ovn/ovnnb_db.db /var/lib/ovn/ovnsb_db.db
[central]# systemctl start ovn-central
```

Xóa NB/SB database xóa sạch logical topology đã define. Nút mạng (Chassis) sẽ tự register lại khi ovn-controller reconnect.

### Reset Mininet (luôn chạy sau mỗi lab Khối III-IV)

```
[host]# mn -c
*** Removing excess controllers/ofprotocols/ofdatapaths/pings/noxes
*** Removing junk from /tmp
*** Removing old X11 tunnels
*** Removing excess kernel datapaths
*** Removing OVS datapaths
*** Removing all links of the pattern foo-ethX
*** Cleanup complete.
```

Lệnh `mn -c` dọn mọi namespace, veth, OVS bridge mà Mininet đã tạo. Nếu bỏ qua, lần chạy `mn` tiếp sẽ lỗi vì namespace cũ còn chiếm tên.

### ▶ Guided Exercise 1: Xác minh baseline OVS 2.17 + OVN 22.03 trên Ubuntu 22.04

Trong bài thực hành này, bạn xác minh toàn bộ baseline version của series đã được cài đúng bằng bốn check độc lập, package, kernel module, service, runtime CLI.

**Outcomes:**
- Xác nhận được `openvswitch-switch` và `ovn-central`/`ovn-host` đã cài đúng version từ pocket jammy-updates.
- Phân biệt được khi nào version mismatch (kernel module vs userspace) gây lỗi silent vs lỗi rõ ràng.

**Before You Begin:**
Hoàn thành mục 0.1.2 (kernel module) và 0.1.3 (apt install). Lab mode A (single-node) là đủ.

**Instructions:**

**1.** Kiểm tra version package đã cài:

 [host]$ dpkg -l | grep -E '^ii (openvswitch|ovn)' | awk '{print $2, $3}'

Kết quả mong đợi:

 openvswitch-common 2.17.9-0ubuntu0.22.04.1
 openvswitch-switch 2.17.9-0ubuntu0.22.04.1
 ovn-central 22.03.8-0ubuntu0.22.04.1
 ovn-common 22.03.8-0ubuntu0.22.04.1
 ovn-host 22.03.8-0ubuntu0.22.04.1

Năm package này đủ cho lab mode A và B. Nếu version hiển thị thấp hơn (ví dụ `2.17.7` thay vì `2.17.9`), chạy `apt upgrade` để lên bản mới nhất của pocket jammy-updates.

**2.** Kiểm tra kernel module đã load:

 [host]$ lsmod | grep openvswitch
 openvswitch 176128 0

Cột cuối `0` là reference count, hiện tại chưa có bridge nào dùng module nên count = 0. Sau khi `ovs-vsctl add-br br-test`, count sẽ tăng lên 1.

**3.** Kiểm tra systemd service:

 [host]$ systemctl is-active openvswitch-switch ovn-central ovn-controller
 active
 active
 active

Ba service đều `active`. Nếu `ovn-controller` hiển thị `inactive`, đó là do `ovn-host` package chưa config Southbound URL, xem journalctl.

**4.** Kiểm tra runtime CLI từ userspace:

 [host]$ ovs-vsctl show
 92a54bc6-6e75-4f02-9b4c-95c01dbd5f03
 ovs_version: "2.17.9"

 [host]$ ovn-nbctl --version
 ovn-nbctl 22.03.8
 Open vSwitch Library 2.17.9
 DB Schema 6.1.0

 [host]$ ovn-sbctl --version
 ovn-sbctl 22.03.8
 Open vSwitch Library 2.17.9
 DB Schema 20.21.0

Dòng `ovs_version: "2.17.9"` trong `ovs-vsctl show` là version userspace daemon reportback. Dòng `Open vSwitch Library 2.17.9` trong `ovn-nbctl --version` là version của libopenvswitch mà ovn-nbctl link tới. Hai dòng này phải khớp, nếu lệch (ví dụ ovn link libopenvswitch 2.16 nhưng daemon dùng 2.17), sẽ có lỗi khó debug khi parse tunnel metadata.

DB Schema version (`6.1.0` cho NB, `20.21.0` cho SB) là version của OVSDB schema, không phải version OVN software. Khi upgrade OVN từ 22.03 → 24.03, schema version tăng tương ứng và cần migration bước.

**Finish:**
Bốn check đều pass → môi trường lab sẵn sàng cho mọi Phần trong series. Lưu output của bước 1 và 4 vào file để tham chiếu sau, khi một Phần báo kết quả khác lạ, đối chiếu với baseline này là bước debug đầu tiên.

## Mục tiêu bài học

Sau khi hoàn thành phần này, bạn có thể:

1. **Mô tả (Understand)** động lực chuyển từ ML2/OVS (Neutron agent-based) lên ML2/OVN (controller-based): lý do từ Red Hat + Canonical công bố — giảm phức tạp vận hành, tăng khả năng mở rộng, loại bỏ nhiều agent tiến trình.
2. **Phân tích (Analyze)** bảng so sánh feature parity giữa ML2/OVS và ML2/OVN ở OpenStack Yoga+, xác định tính năng nào đã parity, tính năng nào còn gap.
3. **Áp dụng (Apply)** quy trình migration chuẩn Red Hat/Canonical: kiểm tra sẵn sàng trước migration → migration offline/online → xác minh → rollback nếu cần.
4. **Đánh giá (Evaluate)** khi nào migrate (cluster nhỏ có thể đợi ML2/OVN mặc định, cluster lớn cần migrate để giải quyết vấn đề mở rộng), và sự đánh đổi chi phí/lợi ích.

Kiến thức tiên quyết: Part 13.0 cho OVN rationale, Part 13.7 kiến trúc ovn-controller.

## 13.13.1 Tại sao migrate từ ML2/OVS lên ML2/OVN?

Theo Red Hat OpenStack Platform Networking Guide và upstream Neutron documentation, bốn động lực chính:

**(1) Loại bỏ nhiều agent phân tán:**
- ML2/OVS triển khai mỗi chassis với: `neutron-openvswitch-agent` (OVS config), `neutron-l3-agent` (L3 routing + SNAT), `neutron-dhcp-agent` (DHCP server), `neutron-metadata-agent` (dịch vụ metadata). Cộng thêm `neutron-server` ở controller node.
- Mỗi agent là tiến trình Python tự dùng RabbitMQ giao tiếp với neutron-server → overhead RabbitMQ tăng tuyến tính với số chassis. Cụm 100 chassis = 400 agent + 400 kết nối RabbitMQ.
- ML2/OVN chỉ cần `ovn-controller` + `ovn-metadata-agent` (tùy chọn) trên chassis + `ovn-northd` + `ovsdb-server` ở control plane. Giảm 50% số tiến trình.

**(2) Khả năng mở rộng:**
- ML2/OVS lưu trạng thái trong Neutron database (MySQL) + RabbitMQ để đẩy thay đổi. Khi cụm lớn, RabbitMQ trở thành điểm nghẽn.
- ML2/OVN dùng OVSDB Raft cluster làm database phân tán + monitor_cond để đẩy thay đổi → mở rộng tới hàng nghìn chassis (Red Hat benchmark 2022: OVN tested với 500+ chassis).
- Mô hình Logical Flow của OVN cho phép compile một lần ở ovn-northd thay vì gọi RPC riêng tới mỗi agent.

**(3) L3 HA + DVR tích hợp native:**
- ML2/OVS L3 HA cần triển khai riêng qua `neutron-l3-agent` với Keepalived + VRRP — phức tạp cấu hình.
- ML2/OVN có HA chassis group + BFD built-in (Part 13.6) → định tuyến HA là hành vi mặc định, không cần công cụ ngoài.

**(4) Distributed Virtual Routing (DVR) đơn giản hơn:**
- ML2/OVS DVR cần neutron-l3-agent chạy ở mọi compute + định tuyến dựa trên namespace → phức tạp gỡ lỗi.
- ML2/OVN Distributed Router (DR) là mặc định — không cần cấu hình riêng, không cần namespace (Part 13.11).

**Lộ trình migration chính thức:**
- **OpenStack Train (2019)**: ML2/OVN ổn định, khuyến nghị cho triển khai mới.
- **OpenStack Wallaby (2021)**: ML2/OVN đạt feature parity với ML2/OVS cho hầu hết trường hợp sử dụng.
- **OpenStack Yoga (2022)**: ML2/OVN là driver mặc định trong DevStack và khuyến nghị cho cài đặt mới.
- **Red Hat OSP 16.2+**: ML2/OVN là mặc định; OSP 17+ xóa ML2/OVS khỏi installer (vẫn support qua custom config).
- **OpenShift 4.12 (01/2023)**: OVN-Kubernetes là CNI mặc định, thay thế OpenShift SDN cho cluster mới.

## 13.13.2 Feature parity: ML2/OVS so với ML2/OVN

Bảng so sánh dựa trên **OpenStack Neutron ML2/OVN Documentation** (docs.openstack.org/neutron/latest/ovn/) và Red Hat RHOSP 17.1 Networking Guide, tính đến Yoga+/2022-Q3:

| Tính năng | ML2/OVS | ML2/OVN Yoga+ | Notes |
|-----------|---------|---------------|-------|
| Cô lập tenant (VLAN/VXLAN/GRE) | Có | Có (Geneve mặc định) | OVN khuyến nghị Geneve |
| Security Group | Có (iptables) | Có (OVN ACL) | OVN hiệu năng tốt hơn với conntrack |
| DHCP | neutron-dhcp-agent | OVN native DHCP | Xem Part 13.10 |
| DNS | neutron-dns + dnsmasq | OVN native DNS | OVN giới hạn cho recursive |
| Dịch vụ metadata | neutron-metadata-agent | ovn-metadata-agent | Tương đương |
| L3 routing | neutron-l3-agent | OVN Logical Router | Xem Part 13.2 |
| SNAT tập trung | l3-agent | OVN Gateway Router | Xem Part 13.11 |
| DVR | Có (legacy DVR + DVR-HA) | OVN Distributed Router | Mặc định trong OVN |
| L3 HA | Keepalived + VRRP | OVN HA Chassis Group + BFD | Xem Part 13.6 |
| Floating IP | Có | Có | Tương đương |
| Load Balancer (LBaaS) | Octavia + HAProxy amphora | Octavia + OVN (Yoga+) | OVN provider driver mới Yoga |
| QoS | Có | Có (OVN 22.03+) | Parity đầy đủ |
| BGP dynamic routing | Neutron BGP agent (dragonflow) | OVN-BGP-Agent (experimental) | Gap nhẹ |
| Multi-segment network | Có | Có (OVN 21.03+) | Catch up Yoga+ |
| Trunk port | Có | Có (OVN 21.12+) | Catch up Yoga |
| VPN as a Service (VPNaaS) | Có | Không (deprecated upstream) | Cần keep ML2/OVS hoặc VPN ngoài |
| Firewall as a Service (FWaaS v2) | Có | Không native (OVN ACL đủ) | Security Group cover hầu hết trường hợp sử dụng |
| Macvtap agent | Có | Không | Trường hợp sử dụng hẹp, hầu hết deployment không cần |
| IPv6 first-hop security (RA guard) | Có | Có (OVN 22.03+) | Parity |

**Các tính năng chỉ có trên ML2/OVN (không có trên ML2/OVS):**
- Multi-chassis binding cho live migration VM seamless (Part 19.0).
- SNAT stateful phân tán qua conntrack distributed.
- Port security cho Logical Switch Port dựa trên dynamic_addresses + port_security list.

**Gap cần lưu ý trước khi migrate:**
1. **VPN as a Service**: nếu deployment dùng Neutron VPNaaS, phải lập kế hoạch thay thế bằng VPN gateway ngoài hoặc chấp nhận không có VPN in-cloud.
2. **FWaaS v2**: nếu dùng, phải convert rule sang Security Group tương đương.
3. **Macvtap agent**: nếu dùng chế độ macvtap NIC, cần test kỹ OVN không hỗ trợ.
4. **BGP speaker**: nếu dùng Neutron BGP speaker (dragonflow), phải triển khai `ovn-bgp-agent` (đang trong incubation).

## 13.13.3 Kiểm tra sẵn sàng trước migration

Theo OpenStack Neutron ML2/OVN migration documentation, cluster phải pass những điều kiện này trước khi migrate:

**(1) Phiên bản OpenStack hỗ trợ công cụ migration:**
- Train (2019) trở lên cho offline migration.
- Ussuri (2020) trở lên cho công cụ migration ổn định.
- Yoga (2022) trở lên cho online migration (tested thoroughly).

**(2) Kiểm kê agent và data plane:**
```
[controller]# openstack network agent list
+----+--------------------+-------+------+
| ID | Agent Type         | Host  | State|
+----+--------------------+-------+------+
| .. | Open vSwitch agent | ... | UP   |
| .. | L3 agent           | ... | UP   |
| .. | DHCP agent         | ... | UP   |
| .. | Metadata agent     | ... | UP   |
+----+--------------------+-------+------+
```
Tất cả agent phải UP. Agent down = data mất khi migrate.

**(3) Kiểm kê network + subnet + port:**
```
[controller]# openstack network list
[controller]# openstack subnet list
[controller]# openstack port list
[controller]# openstack router list
```
Export để so sánh sau khi migrate (sẽ xác minh count + attributes bằng).

**(4) Kiểm tra tính năng không compatible:**
```
[controller]# openstack security group list
[controller]# openstack vpn service list    # Nếu có → plan VPN ngoài
[controller]# openstack firewall list        # Nếu có → convert sang SG
```

**(5) Đảm bảo phiên bản OVS đủ:**
```
[chassis]# ovs-vsctl --version
```
OVS 2.11+ cho OVN 20.03+. OVS 2.17+ cho OVN 22.03+ (khuyến nghị).

**(6) Đảm bảo Geneve MTU phù hợp underlay:**
Pod MTU 1500B + Geneve overhead 50-80B = 1550-1580B outer. Underlay cần MTU ≥1600B. Test:
```
[chassis-x]# ping -M do -s 1570 <chassis_y_ip>
```
Nếu fail → cần tăng MTU underlay trước khi migrate (tránh fragmentation ảnh hưởng VM sau migration).

## 13.13.4 Quy trình migration offline (khuyến nghị cho cluster nhỏ)

Migration offline là cách đơn giản nhất — có downtime nhưng ít rủi ro. Theo Red Hat OSP 16.2 migration guide:

**Giai đoạn 1 — Chuẩn bị (không downtime):**
```
[controller]# # Sao lưu database Neutron
[controller]# mysqldump -u root -p neutron > neutron-backup-$(date +%Y%m%d).sql

# Ghi lại trạng thái Neutron cho xác minh sau:
[controller]# openstack network list -f yaml > network-before.yaml
[controller]# openstack subnet list -f yaml > subnet-before.yaml
[controller]# openstack port list -f yaml > port-before.yaml
[controller]# openstack router list -f yaml > router-before.yaml
[controller]# openstack security group list -f yaml > sg-before.yaml
```

**Giai đoạn 2 — Tắt plane dữ liệu (downtime bắt đầu):**
```
[all_chassis]# # Dừng tất cả Neutron agent trên mọi compute node
[all_chassis]# systemctl stop neutron-openvswitch-agent
[all_chassis]# systemctl stop neutron-l3-agent
[all_chassis]# systemctl stop neutron-dhcp-agent
[all_chassis]# systemctl stop neutron-metadata-agent

[controller]# systemctl stop neutron-server
```

**Giai đoạn 3 — Chạy công cụ migration:**
```
[controller]# # Install công cụ migration từ neutron repo
[controller]# pip install neutron-migration-tool    # Hoặc từ package RHOSP

[controller]# # Chạy dry-run trước để xác minh
[controller]# neutron-ovn-migration --dry-run --config-file /etc/neutron/neutron.conf

[controller]# # Nếu dry-run OK, chạy migration thực tế
[controller]# neutron-ovn-migration --config-file /etc/neutron/neutron.conf
```

Công cụ migration thực hiện:
1. Đọc Neutron DB (MySQL) → xác định tất cả network/subnet/port/router/SG.
2. Dịch sang OVN NB DB object tương ứng (Logical_Switch, Logical_Router, Logical_Switch_Port, ACL).
3. Ghi vào OVN NB DB qua ovn-nbctl.
4. Xác minh count: số Logical_Switch = số network, số Port_Binding = số Neutron port.

**Giai đoạn 4 — Triển khai OVN stack + khởi động lại:**
```
[controller]# # Install OVN packages (RHOSP: osp-director redeploy; upstream: thủ công)
[controller]# apt install python3-networking-ovn ovn-central   # Canonical/upstream

[controller]# # Cấu hình ML2 driver Neutron = ovn thay vì openvswitch
[controller]# vi /etc/neutron/plugins/ml2/ml2_conf.ini
# [ml2]
# mechanism_drivers = ovn
# type_drivers = geneve,vlan,flat
# tenant_network_types = geneve

[controller]# systemctl start ovn-northd
[controller]# systemctl start neutron-server

[all_chassis]# # Trên mỗi chassis:
[all_chassis]# systemctl start ovn-controller
[all_chassis]# systemctl start ovn-metadata-agent
```

**Giai đoạn 5 — Xác minh:**
```
[controller]# # Kiểm tra trạng thái OVN
[controller]# ovn-nbctl show
[controller]# ovn-sbctl show

# So sánh count với trạng thái trước migration:
[controller]# openstack network list -f yaml | diff network-before.yaml -
[controller]# openstack port list -f yaml | diff port-before.yaml -

# Kiểm tra connectivity VM-to-VM:
[VM1]# ping <VM2_IP>
```

**Downtime điển hình:** 30-60 phút cho cluster 50 compute node, tăng tuyến tính với kích thước cluster.

## 13.13.5 Quy trình migration online (cluster lớn, không downtime)

Online migration (ra mắt OpenStack Yoga) cho phép migrate từng compute node một, không cần downtime toàn cluster. Theo upstream documentation:

**Kiến trúc online migration:**
1. OVN stack được triển khai song song với Neutron ML2/OVS (dual stack).
2. Network mới tạo sau migrate-start đi qua OVN.
3. Network cũ vẫn đi qua OVS agent.
4. Migration per-network: từng network được migrate riêng, VM restart một lần để chuyển data plane.

**Giai đoạn 1 — Triển khai dual stack:**
```
[controller]# # Install OVN + config ML2 với cả 2 driver
[controller]# vi /etc/neutron/plugins/ml2/ml2_conf.ini
# [ml2]
# mechanism_drivers = openvswitch,ovn
```

**Giai đoạn 2 — Từng network một, drain + migrate:**
```
[controller]# # Chọn network để migrate
[controller]# openstack network show <network_id>

# Cordon network (không tạo port mới):
[controller]# openstack network set --disable <network_id>

# Migrate network:
[controller]# neutron-ovn-migration --network <network_id>

# Live migrate VM trên network đó sang compute node đã có ovn-controller:
[controller]# nova live-migration <vm_id> <target_compute>
```

**Giai đoạn 3 — Xóa Neutron agent sau khi tất cả network đã migrate:**
```
[controller]# openstack network agent list
[controller]# openstack network agent delete <agent_id>    # Cho mọi OVS/L3/DHCP agent
```

**Downtime:** Gần 0 per VM (live migration restart <5 giây). Tổng thời gian migration kéo dài ngày-tuần cho cluster lớn, nhưng không có downtime tập trung.

## 13.13.6 Kế hoạch rollback

Nếu migration fail, rollback về ML2/OVS:

**Rollback offline (giai đoạn 2-3 fail):**
```
[controller]# # Khôi phục Neutron DB từ backup
[controller]# mysql -u root -p neutron < neutron-backup-<date>.sql

[controller]# # Chuyển ML2 driver về openvswitch
[controller]# vi /etc/neutron/plugins/ml2/ml2_conf.ini
# mechanism_drivers = openvswitch

[controller]# # Khởi động lại Neutron + agent
[controller]# systemctl start neutron-server
[all_chassis]# systemctl start neutron-openvswitch-agent neutron-l3-agent ...
```

**Rollback online (giai đoạn 2 fail giữa chừng):**
- Migrate lại các network đã OVN-ize về lại OVS mechanism (công cụ neutron-migration hỗ trợ reverse mode với `--rollback`).
- Uncordon network.
- Test connectivity.

**Kiểm tra toàn vẹn dữ liệu:** Sau rollback, xác minh:
```
[controller]# openstack network list | wc -l    # So sánh với pre-migration
[controller]# openstack port list | wc -l
```

Một số trường hợp rollback thực tế của Red Hat customer (theo Red Hat support knowledge base):
- **Gap MTU underlay** → rollback + tăng MTU trước khi retry.
- **VPNaaS in use** → rollback + deprecate VPNaaS trước khi retry.
- **Neutron extension tùy chỉnh** → rollback + viết lại extension compatible với OVN.

## 13.13.7 Case studies môi trường production

Theo công bố của Red Hat + Canonical + OpenShift:

**Red Hat OpenStack Platform (RHOSP) 16.2 → 17.1:**
- Lộ trình migration mặc định cho deployment hiện có RHOSP 13/16.1 → 17.1 là offline migration.
- Customer môi trường production điển hình: 100-300 compute node → thời gian migration 2-4 giờ downtime.
- Red Hat cung cấp Ansible playbook automation: `tripleo-ansible-inventory` + `openstack overcloud upgrade` tự động xử lý OVN migration.
- Sau migration: giảm 30-50% dấu chân memory per compute node (không còn neutron-l3-agent, neutron-dhcp-agent).

**Canonical Charmed OpenStack (Juju):**
- Migration qua Juju charm: `juju deploy ovn-central`, `juju add-relation ovn-central neutron-api`, `juju run-action --unit neutron-api/0 migrate-to-ovn`.
- Canonical blog (08/2022) báo customer telco (BT, Telefónica) migrate cluster 1000+ node thành công với downtime <1 giờ per wave.

**OpenShift 4.12 SDN → OVN-K8s migration:**
- Red Hat cung cấp công cụ migration `oc patch network.operator/cluster --type=merge --patch='{"spec":{"migration":...}}'`.
- Migration 4.10 → 4.12 bao gồm tự động OVN-K8s mặc định cho cluster mới.
- Cluster hiện có có thể opt-in migration qua `migration.networkType: OVNKubernetes`.
- Downtime per node: 2-5 phút (node drain + reboot + rejoin).

Bài học môi trường production từ các case:
1. **Luôn test trên cluster staging** identical với môi trường production trước khi migrate.
2. **Giám sát KPI trong + sau migration 72h**: độ trễ VM-to-VM, thời gian phản hồi API, thời gian query Neutron DB.
3. **Giữ backup Neutron DB ≥30 ngày** — rollback sau 1 tuần vẫn khả thi nếu chưa push VM mới.
4. **Giao tiếp với security team**: ngữ nghĩa OVN ACL khác iptables Security Group nhẹ ở các edge case stateful.

### ▶ Guided Exercise 1: Dry-run migration OpenStack DevStack ML2/OVS → ML2/OVN

> **Trạng thái verification lab:** Output dựa trên OpenStack Neutron ML2/OVN docs + DevStack Yoga; chờ verify lab thực.

**Mục đích:** Thực hành quy trình migration trên DevStack cluster nhỏ (all-in-one) — chạy dry-run migration, đọc báo cáo, rollback về trạng thái ban đầu.

**Chuẩn bị:** Ubuntu 22.04 VM với DevStack đã cài (Yoga+), Q_PLUGIN=ml2 + Q_ML2_MECH_DRIVER=openvswitch (ML2/OVS setup).

**Bước thực hiện:**

**1.** Xác minh trạng thái ban đầu ML2/OVS:
```
[devstack]$ openstack network agent list
+----+--------------------+------+
| ID | Agent Type         | State|
+----+--------------------+------+
| .. | Open vSwitch agent | UP   |
| .. | L3 agent           | UP   |
| .. | DHCP agent         | UP   |
| .. | Metadata agent     | UP   |
+----+--------------------+------+
```

**2.** Sao lưu Neutron DB:
```
[devstack]$ mysqldump -u root -p$MYSQL_PASSWORD neutron > /tmp/neutron-backup.sql
[devstack]$ ls -la /tmp/neutron-backup.sql
```

**3.** Chạy dry-run migration:
```
[devstack]$ sudo pip install neutron-ovn-migration-tool
[devstack]$ neutron-ovn-migration --dry-run \
    --neutron-config /etc/neutron/neutron.conf \
    --output-report /tmp/migration-dryrun.log
```

Output kỳ vọng:
```
Neutron to OVN migration dry-run report
=======================================
Networks to migrate: 5
Subnets to migrate: 8
Ports to migrate: 42
Routers to migrate: 3
Security groups to migrate: 7
Unsupported features: None
Migration time estimate: 10-15 minutes
```

**4.** Nếu dry-run OK, không chạy migration thực tế (đây là exercise dry-run only).

**5.** Xem báo cáo:
```
[devstack]$ cat /tmp/migration-dryrun.log | head -50
```

**Bài học học được:**

Thứ nhất, dry-run là **bước an toàn bắt buộc** trước mọi migration môi trường production. Nó cho biết cluster có tính năng không compatible (VPNaaS, FWaaS) mà không thực sự modify state.

Thứ hai, công cụ migration ước lượng thời gian dựa trên count (network × độ phức tạp) — cluster môi trường production thực tế có thể lâu hơn 2-3 lần nếu có network attachment phức tạp.

Thứ ba, backup Neutron DB là điều kiện không thể thiếu. Một số lần rollback thất bại là do backup không complete hoặc quá cũ.

**Cleanup:** Không cần (chỉ chạy dry-run, không modify state).

## 10.7.1 Architecture and characteristic features

### 10.7.1.1 `ovsdb-client` in the OVS/OVN stack

`ovsdb-client` is a CLI in the `ovs-common` package (or `openvswitch-common` on Ubuntu) that connects directly to `ovsdb-server` over TCP, a Unix socket, or SSL. It differs from `ovs-vsctl`, `ovn-nbctl`, and `ovn-sbctl` as follows:

- `ovs-vsctl` and the `ovn-*ctl` tools build a full **IDL** (Interface Definition Language) in process. They are schema-aware and validate before sending.
- `ovsdb-client` does not build an IDL; it sends raw JSON-RPC per RFC 7047. This makes it suitable for generic tooling, scripting, and schema-agnostic debugging.

Architecture:

```
+-----------------+      RFC 7047 JSON-RPC over TCP/Unix/SSL
|  ovsdb-client   | <-------------------------------------------+
+-----------------+                                              |
        |                                                        v
        | command line: list-dbs / dump / monitor / transact ... |
        |                                                  +---------------+
        v                                                  |  ovsdb-server |
   stdout JSON or formatted table                          |  (NB or SB)   |
                                                            +---------------+
```

**Important properties.**

- **Schema-agnostic.** Works on any OVSDB database (NBDB, SBDB, OpenFlow OVSDB, custom schema).
- **No transaction batching.** Each invocation is a new connection. Different from `ovs-vsctl`, which can batch multiple changes.
- **Synchronous output.** `monitor` is a streaming event command; the others return a result and exit.
- **Format flexibility.** The `-f` option chooses the format (`table`, `list`, `csv`, `json`).

### 10.7.1.2 Connection target

```bash
ovsdb-client <command> [<server>] [<db>] [<table>] [<column>...]
```

The `<server>` can be:

- `tcp:host:port`, for example `tcp:nb1.example.com:6641`.
- `unix:/var/run/ovn/ovnnb_db.sock`, the default for a local process.
- `ssl:host:port`, combined with `-p <privkey>`, `-c <cert>`, and `-C <ca-cert>` for mTLS.
- If `<server>` is omitted, the tool reads `OVS_RUNDIR` (default `/var/run/openvswitch/` for OVS, `/var/run/ovn/` for OVN) and looks for the `db.sock`, `ovnnb_db.sock`, or `ovnsb_db.sock` socket.

Examples:

```bash
# OVS local
[host]# ovsdb-client list-dbs unix:/var/run/openvswitch/db.sock
Open_vSwitch

# OVN NBDB Raft cluster
[host]# ovsdb-client list-dbs tcp:nb1:6641,tcp:nb2:6641,tcp:nb3:6641
OVN_Northbound

# OVN SBDB over SSL
[host]# ovsdb-client -p key.pem -c cert.pem -C ca.pem \
                     list-dbs ssl:sb1.example.com:6642
OVN_Southbound
```

## 10.7.2 Schema introspection

A group of commands that read information about the schema and do not modify the DB.

### `list-dbs`

```bash
[host]# ovsdb-client list-dbs unix:/var/run/openvswitch/db.sock
Open_vSwitch

[host]# ovsdb-client list-dbs tcp:nb1:6641
OVN_Northbound
```

Lists the names of databases that `ovsdb-server` is serving. One server can serve multiple DBs simultaneously (production typically has one DB per server, but test setups or multi-tenant deployments may differ).

### `get-schema [<db>]`

```bash
[host]# ovsdb-client get-schema tcp:nb1:6641 OVN_Northbound | head -30
{
  "name": "OVN_Northbound",
  "version": "5.32.1",
  "cksum": "1466357837 27935",
  "tables": {
    "NB_Global": {
      "columns": {
        "name": {"type": "string"},
        "nb_cfg": {"type": "integer"},
        ...
```

Returns the full schema as JSON. Useful when:

- Comparing the schema between two clusters (version drift after upgrade).
- Auto-generating code (Python or Go IDL stubs) from the schema.
- Auditing RBAC permissions (which tables and columns exist).

### `get-schema-version` plus `get-schema-cksum`

```bash
[host]# ovsdb-client get-schema-version tcp:nb1:6641 OVN_Northbound
5.32.1

[host]# ovsdb-client get-schema-cksum tcp:nb1:6641 OVN_Northbound
1466357837 27935
```

Quick checks that do not need JSON parsing. Production: run as a daily cron job and alert if the version or checksum changes outside a planned upgrade.

### `list-tables [<db>]` plus `list-columns <db> [<table>]`

## 10.7.3 Query and dump

### `dump [<db> [<table> [<column>...]]]`

```bash
[host]# ovsdb-client dump tcp:nb1:6641 OVN_Northbound Logical_Switch_Port name addresses type
name                 addresses                              type
"web1"               ["aa:bb:cc:00:00:01 10.0.0.10"]       ""
"web2"               ["aa:bb:cc:00:00:02 10.0.0.11"]       ""
"phys-port"          ["unknown"]                            "localnet"
```

The default is table-formatted output; `-f json` gives pure JSON. Different from `ovn-nbctl list`: dump does not filter, it fetches every row, which is good for cluster-wide audit.

### `query <db> <op-json>`

Runs one or more read-only RFC 7047 operations without committing a transaction:

```bash
[host]# ovsdb-client query tcp:nb1:6641 \
        '["OVN_Northbound", {"op":"select","table":"Logical_Switch_Port",\
          "columns":["name","addresses"],"where":[["type","==","localnet"]]}]'
```

Returns a JSON array with each row that matches `where`. Useful for:

- Testing complex `where` clauses before using them in an application.
- Debugging filter conditions that `ovn-nbctl find` does not support fully.

## 10.7.4 Transaction (write)

### `transact <db> <op-array>`

An atomic transaction per RFC 7047. Every operation in the array either commits or rolls back:

```bash
[host]# ovsdb-client transact tcp:nb1:6641 \
        '["OVN_Northbound",
          {"op":"insert","table":"Logical_Switch","row":{"name":"my-ls"},"uuid-name":"new_ls"},
          {"op":"insert","table":"Logical_Switch_Port",
           "row":{"name":"port1","type":""},
           "uuid-name":"new_port"},
          {"op":"mutate","table":"Logical_Switch","where":[["_uuid","==",["named-uuid","new_ls"]]],
           "mutations":[["ports","insert",["set",[["named-uuid","new_port"]]]]]}]'
```

When you need an atomic change with no available CLI helper (for example, a rare edge case that `ovn-nbctl` does not cover), `transact` is the fallback. But the JSON syntax is heavy and easy to get wrong, so use it only when truly necessary.

### Rule 5 pillar warning

`transact` lets you write directly into the SBDB (the output of northd). Like `ovn-sbctl set Port_Binding chassis=`, this is an **anti-pattern** outside emergency debug. With NBDB, `ovn-nbctl` is always preferred to a raw `transact`.

## 10.7.5 Monitoring (event stream)

This is the **characteristic** feature of `ovsdb-client` that no other tool offers. Each DB change event is streamed to stdout in real time until the user presses Ctrl-C.

### `monitor <db> <table> [<column>...]`

```bash
[host]# ovsdb-client monitor tcp:sb1:6642 OVN_Southbound Port_Binding chassis logical_port
row                                  action  chassis    logical_port
0a17b4f8-...                         insert  ""         "web1"
0a17b4f8-...                         old     ""         "web1"
0a17b4f8-...                         new     "55b4..."  "web1"
0a17b4f8-...                         delete  "55b4..."  "web1"
```

Three action types: `insert` (new row), `old` and `new` (paired update), `delete`. This lets you trace the lifecycle of one entity (for example, a Port_Binding claim across chassis).

### `monitor-cond` plus `monitor-cond-since`

The advanced version with a conditional filter (reduces bandwidth):

```bash
[host]# ovsdb-client monitor-cond tcp:sb1:6642 OVN_Southbound \
        '[{"table":"Port_Binding",
           "where":[["logical_port","==","web1"]]}]'
```

Streams only events for rows that match `where`. `monitor-cond-since` adds a starting point (`last_id`) for replay on reconnect.

### Forensic use case

A real-world scenario (Part 20.5 §20.5.1) uses `monitor` to forensically trace a `Port_Binding` migration race:

```bash
[host]# ovsdb-client --timestamp monitor tcp:sb-leader:6642 \
                     OVN_Southbound Port_Binding chassis | \
        grep --line-buffered "logical_port=\"vm-prod-456\""
```

With `--timestamp`, every event is prefixed with a millisecond timestamp; this lets you reconstruct the exact timeline of the dual-bind window.

## 10.7.6 Coordination (wait and lock)

### `wait <db> <table> <where>... <until>...`

Blocks until a condition matches:

```bash
[host]# ovsdb-client wait tcp:nb1:6641 OVN_Northbound NB_Global \
                     '_uuid!=00000000-0000-0000-0000-000000000000' nb_cfg=100
```

Means: wait until `nb_cfg = 100` appears. Useful for integration tests or for synchronising a CMS workflow.

### `lock`/`steal`/`unlock <lock-name>`

A distributed locking primitive built into OVSDB:

```bash
[host]# ovsdb-client lock tcp:nb1:6641 my-lock-name &
PID 12345

# Another process waits for the lock:
[host]# ovsdb-client lock tcp:nb1:6641 my-lock-name
# (blocks until the first process exits or releases)

# Force-take the lock:
[host]# ovsdb-client steal tcp:nb1:6641 my-lock-name
```

`ovn-northd` uses an OVSDB lock for leader election (see Part 13.8.3): the instance that grabs the `ovn_northd` lock on SBDB becomes the active one.

## 10.7.7 Backup, restore, and convert

### `backup <db>`

```bash
[host]# ovsdb-client backup tcp:nb1:6641 > nbdb-backup-$(date +%Y%m%d).db
```

Streams the entire DB to stdout in OVSDB log format. Production: run a daily cron and ship it to remote storage.

### `restore <db>`

```bash
[host]# ovsdb-client restore tcp:nb1:6641 < nbdb-backup-20260425.db
```

Requires the target DB to be empty or to have the same schema. Restoring into a running Raft cluster is safe (atomic transaction).

### `convert <db> <schema-file>`

Migrate the schema after an OVN or OVS upgrade:

```bash
[host]# ovsdb-client needs-conversion tcp:nb1:6641 OVN_Northbound \
                     /usr/share/ovn/ovn-nb.ovsschema
yes

[host]# ovsdb-client convert tcp:nb1:6641 OVN_Northbound \
                     /usr/share/ovn/ovn-nb.ovsschema
```

This command changes the schema at runtime without restarting `ovsdb-server`. Critical for **rolling upgrade** of OVN: upgrade the `ovn-northd` binary first, convert the NBDB schema, then upgrade `ovn-controller`.

## 10.7.8 Anatomy Template A, five important outputs

### Anatomy 1: `monitor` event stream

> **Run context.** Forensic timeline for `Port_Binding` or `Logical_Flow` changes. With `--timestamp` enabled.

```
2026-04-25T14:23:10.123Z  row=a1b2c3d4  action=old      chassis=""             logical_port="web1"
2026-04-25T14:23:10.123Z  row=a1b2c3d4  action=new      chassis="55b4d9e8..."  logical_port="web1"
2026-04-25T14:23:15.456Z  row=a1b2c3d4  action=old      chassis="55b4d9e8..."  logical_port="web1"
2026-04-25T14:23:15.456Z  row=a1b2c3d4  action=new      chassis="fc12a8d1..."  logical_port="web1"
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| Action `insert` or `delete` rate | < 10 per second cluster-wide | 10 to 100 per second | > 100 per second | CMS bulk update; check `inc-engine/show` (Part 13.8.6) |
| `old → new` paired interval | < 100 ms | 100 to 1000 ms | > 1000 ms | OVSDB Raft slow; check `cluster/status` (Part 10.1.6.5) |
| Row appearing or disappearing rapidly | Stable | Flap 2 to 5 per minute | Flap > 10 per minute | Chassis claim race (Part 13.7.8); check migration |
| `chassis` empty (after `new`) | After `lsp-del` | After migration in progress | Stuck > 30 seconds | northd recompute lag, or chassis offline |

**Failure scenarios.**

- Dual-bind window > 5 seconds: claim race per Part 20.5 §20.5.1. Force release with `ovn-sbctl set Port_Binding chassis=[]` short-term.
- `insert` storm into the `Logical_Flow` table after an NB change: northd recompute is not incremental. See 13.8.6 inc-engine fallback.

### Anatomy 2: `dump` output (table format)

```
name        type        addresses                              up
"web1"      ""          ["aa:bb:cc:00:00:01 10.0.0.10"]       true
"web2"      ""          ["aa:bb:cc:00:00:02 10.0.0.11"]       true
"phys-1"    "localnet"  ["unknown"]                            true
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| Row count | Matches expected workload | Off by 5 to 10 percent | Off by more than 25 percent | Stale entry; run `ovn-nbctl find ... no=...` to locate |
| `up` column | `true` for every running VM | `false` for one or two new ports | `false` widespread | Cluster-wide claim issue |
| `addresses` empty | For an `unknown` LSP type | (none) | Empty for a VM type | The CMS has not set addresses on the LSP yet |

### Anatomy 3: `list-dbs` output

```
OVN_Northbound
OVN_Southbound
```

(In a single-process `ovn-central` node serving both DBs.)

| Healthy | Warning | Critical | Diagnostic hint |
|---------|---------|----------|-----------------|
| One or two expected DBs | An unknown DB appears | Empty list | `ovsdb-server` crash; check `systemctl status ovn-central` |

### Anatomy 4: `get-schema` JSON

```json
{
  "name": "OVN_Northbound",
  "version": "5.32.1",
  "cksum": "1466357837 27935",
  "tables": { "..." }
}
```

| Field | Healthy | Warning | Critical |
|-------|---------|---------|----------|
| `version` matches the expected baseline | OVN 22.03 expects 5.32.x | Off by a minor | Off by a major | Schema drift after a partial upgrade |
| `cksum` stable | Unchanged on every read | Changes (schema mismatch) | (none) | A schema reload is in progress |

### Anatomy 5: `transact` JSON-RPC response

```json
[
  {"uuid": ["uuid","abcd1234-..."]},
  {"uuid": ["uuid","efgh5678-..."]},
  {"count": 1}
]
```

Each element of the array corresponds to one operation in the request:

- `insert` returns `{"uuid": [...]}` (the UUID of the new row).
- `mutate` returns `{"count": N}` (the number of rows affected).
- `select` returns `{"rows": [...]}`.
- `wait` returns `{}` when the condition matches, or `{"error": "timed out"}`.

If a transaction fails, the response array contains `{"error": "..."}` in place of the success object.

## 10.7.9 Decision matrix, `ovsdb-client` versus other tools

| Situation | Right tool | Reason |
|-----------|-----------|-------|
| Configure LS, LR, or ACL with logical intent | `ovn-nbctl` | Schema-aware, validation, easy syntax |
| Bind LSP to a chassis (debug binding race) | `ovn-sbctl set Port_Binding` | Schema-aware, Part 13.14.5 |
| Forensic Port_Binding timeline of 5 to 30 minutes | `ovsdb-client monitor --timestamp` | Streaming events, schema-agnostic, Part 20.5 |
| Export NBDB for disaster recovery | `ovsdb-client backup` | Standard format, atomic restore |
| Schema check after an OVN upgrade | `ovsdb-client get-schema-cksum` | Quick, stateless, scriptable |
| View 100 columns of 1000 `Logical_Flow` rows | `ovsdb-client dump` | Dump is the only schema-agnostic option |
| Run an atomic complex 5-operation transaction | `ovsdb-client transact` (last resort) | When `ovn-nbctl` does not cover the edge case; prepare the JSON carefully |
| Health-check that `ovsdb-server` is alive | `ovsdb-client list-dbs` | Less than 100 ms response, perfect for cron |
| Migrate the schema in a rolling upgrade | `ovsdb-client convert` | No server restart, atomic |
| Distributed lock for a custom leader election | `ovsdb-client lock <name>` | Built in, no need for an extra dependency such as ZooKeeper |

**Anti-patterns.**

- Using `ovsdb-client transact` for logical intent (for example, creating a `Logical_Switch`). The right tool is `ovn-nbctl ls-add`.
- Using `ovsdb-client monitor` without a conditional filter on a large cluster (10,000-plus rows). Bandwidth explodes. Use `monitor-cond` with a `where` filter instead.

## 10.7.10 Guided Exercise S68': forensic Port_Binding race with `monitor`

**Objective.** Reproduce and observe the 3 to 18 second migration race window described in Part 20.5 §20.5.1.

**Assumed setup.** OVN cluster 22.03.8 with two chassis. VM `vm-test` runs on chassis A and is migrating to B.

**Steps.**

```bash
# 1. Start monitor in parallel (tab 1)
[host]# ovsdb-client --timestamp monitor tcp:sb-leader:6642 \
        OVN_Southbound Port_Binding chassis logical_port | \
        grep --line-buffered '"vm-test"' > /tmp/migration-trace.log

# 2. Trigger the migration (tab 2): set requested-chassis on NBDB
[host]# ovn-nbctl lsp-set-options vm-test requested-chassis=chassis-B

# 3. Wait 30 seconds for the migration to settle, then Ctrl-C in tab 1.

# 4. Analyse the timeline
[host]# cat /tmp/migration-trace.log
```

**Expected output (documentation-plausible, verify in a real lab):**

```
2026-04-25T14:23:10.000Z  action=old   chassis="chassis-A-uuid"  logical_port="vm-test"
2026-04-25T14:23:10.123Z  action=new   chassis=""                logical_port="vm-test"
2026-04-25T14:23:11.456Z  action=old   chassis=""                logical_port="vm-test"
2026-04-25T14:23:11.456Z  action=new   chassis="chassis-B-uuid"  logical_port="vm-test"
```

**Analysis.**

- At 14:23:10.000 chassis A releases (chassis to empty).
- Window 14:23:10.123 to 14:23:11.456 (about 1.3 seconds) the port is bound to no chassis, so traffic drops.
- At 14:23:11.456 chassis B claims.
- Total outage window: 1.3 seconds for a healthy cluster. If Raft is slow or `ovn-controller` on B is busy, the window can be 5 to 30 seconds.

**Lessons learned.**

1. `ovsdb-client monitor` is the only tool that streams events in real time with millisecond precision. `ovn-sbctl list Port_Binding` is only a snapshot and cannot trace a race.
2. The `--timestamp` flag is mandatory for forensic analysis. Without it, the output cannot be correlated with other logs.
3. `grep --line-buffered` does not drop output due to buffering. Production setups often pipe through `ts` (moreutils) or `awk` for real-time processing.

**Cleanup.**

```bash
[host]# rm /tmp/migration-trace.log
```

## 10.7.11 Capstone POE Phase I.B2: `monitor` versus `transact`, choose the right tool

> **Setting.** An on-call engineer receives a "Port_Binding stuck dual-bind" alert on a production OVN 22.03 cluster. The engineer plans to force release through `ovsdb-client transact` for a quick fix.

**Claim under test:** "Using `ovsdb-client transact` to mutate `Port_Binding.chassis = []` is the right way to fix this and is faster than `ovn-sbctl`."

**Predict.** Wrong on both counts: (a) it is not faster, (b) it is not the right way.

**Observe (assumed; verify in a lab).**

- `ovsdb-client transact` requires correct RFC 7047 JSON syntax and the explicit UUID. Composing the JSON takes 30 to 60 seconds; it is easy to get the syntax wrong; you will retry.
- `ovn-sbctl set Port_Binding <port> chassis=[]` is a single command, supports tab completion, and runs in less than 5 seconds.
- Both are overwritten by the next northd recompute within 1 to 30 seconds if NBDB does not reflect the change.

**Explain.**

1. `ovsdb-client transact` is a **low-level** tool, used when there is no CLI helper or when you need an atomic 5-plus operation transaction. For a single simple operation, the schema-aware tool (`ovn-sbctl`) is faster and safer.
2. Both are an **anti-pattern** for fixing dual-bind in production. The right approach is:
   - Investigate the root cause: `ovsdb-client monitor --timestamp Port_Binding` to observe the timeline.
   - Determine whether NB or SB is wrong. If NB is wrong, fix through `ovn-nbctl lsp-set-options requested-chassis`. If SB is stuck due to an `ovn-controller` bug, restart the affected `ovn-controller`.
3. The `ovsdb-client` tool is powerful but dangerous. A production engineer must understand when to use it.

**Falsification test.**

- Compose two equivalent commands: `ovsdb-client transact <json>` versus `ovn-sbctl set Port_Binding <port> chassis=[]`.
- Measure the time a human takes to compose, run, and verify.
- If `transact` is faster, suspect a benchmark artefact (the script may already be prepared).

**Where the claim is wrong.** (a) The `transact` syntax overhead makes it slower than the schema-aware tool. (b) Both options intervene at SBDB output and will be reverted by northd. The right path is to investigate through `monitor` and fix the root cause through NBDB.

> **Principle.** `ovsdb-client` is powerful for **observation, scripting, and edge cases**. For daily operations and logical intent, the schema-aware tool (`ovn-nbctl` or `ovs-vsctl`) is the first choice.

## 10.7.12 Key points to remember

**One.** `ovsdb-client` is the low-level RFC 7047 JSON-RPC tool, schema-agnostic. It works on any OVSDB database (NBDB, SBDB, vswitch, custom).

**Two.** Seven groups of commands: schema introspection, query and dump, transaction, monitoring (three variants), coordination (wait and lock), backup and restore, schema migration (convert).

**Three.** `monitor` is the characteristic feature. Stream events in real time with `--timestamp` for a forensic timeline. `monitor-cond` with a `where` filter reduces bandwidth for large clusters.

**Four.** `transact` is for atomic multi-operation transactions but the JSON is heavy. It is a last resort and does not replace the schema-aware tool for daily ops.

**Five.** `backup` and `restore` are the standard for disaster recovery. `convert` is for rolling schema upgrades without downtime.

**Six.** Anti-patterns: using `transact` for logical intent (`ovn-nbctl` is correct); using `monitor` on a large cluster without a `where` filter (bandwidth explosion).

**Seven.** A production engineer uses `ovsdb-client` mainly for three use cases: observation and forensics (monitor), backup and restore, and schema migration. Daily ops still use `ovn-nbctl` and `ovs-vsctl`.

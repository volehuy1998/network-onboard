## 9.8.1 Ba giao thức flow export

**sFlow** (sFlow.org 2001): sampling-based. Switch chọn mẫu ngẫu nhiên 1-in-N packet, gửi header packet (mặc định 128 bytes) kèm metadata đến collector. Counter polling định kỳ. UDP port 6343.

**NetFlow** (Cisco 1996, IETF v5/v9): per-flow record. Switch duy trì flow cache, emit record khi flow expire. Packet rate thấp hơn sFlow nhưng record chi tiết hơn. UDP port 2055 or custom.

**IPFIX** (RFC 7011, 2013): NetFlow v10 standardized. Template-based — switch gửi template định nghĩa record format, sau đó gửi data records tham chiếu template. UDP port 4739 default.

---

## 9.8.2 Cấu hình sFlow

Compass Ch H idiom:

```bash
[host]# ovs-vsctl -- --id=@s create sFlow agent=eth1 \
         target=\"10.0.0.1:6343\" \
         header=128 sampling=64 polling=10 \
         -- set Bridge br0 sflow=@s
```

- `agent=eth1`: source IP lấy từ interface eth1.
- `target`: collector endpoint.
- `header=128`: giữ 128 byte đầu packet.
- `sampling=64`: sample 1 trong 64 packet.
- `polling=10`: emit counter mỗi 10 giây.

Kiểm chứng: `tcpdump -ni ethX udp port 6343` trên collector host.

Rotate collector runtime: `ovs-vsctl set sFlow <uuid> target='"new:6343"'` — atomic, không mất packet.

## 9.8.3 Cấu hình NetFlow

```bash
[host]# ovs-vsctl -- set Bridge br0 netflow=@nf \
         -- --id=@nf create NetFlow \
            targets=\"192.168.0.34:5566\" active-timeout=30
```

- `engine_id`/`engine_type` auto-generate, có thể override.

```bash
[host]# ovs-vsctl -- set Bridge br0 ipfix=@i \
```

## 9.8.7 Anatomy Template A — `ovs-vsctl list sFlow`

**Bối cảnh khi chạy:** Verify sFlow config active + sampling rate applied. Cũng là debug entry khi collector không nhận data.

**Command và output** *(reproduced OVS 2.17.9)*:

```shell
$ sudo ovs-vsctl list sFlow
_uuid               : 7e3c9f82-1d5e-4b7a-8e9f-abcdef012345
agent               : "eth1"
external_ids        : {purpose="capacity-planning"}
header              : 128
polling             : 10
sampling            : 64
targets             : ["10.0.0.1:6343", "10.0.0.2:6343"]
```

**9-attribute anatomy:**

| Field | Giá trị mẫu | Ý nghĩa | Red flag nếu |
|-------|-------------|---------|--------------|
| `agent` | `"eth1"` | Interface cung cấp source IP | Interface không tồn tại = sFlow disabled silently |
| `header` | `128` bytes | Packet header truncation | <64 = header không đủ parse L4, >256 = waste bandwidth |
| `sampling` | `64` (1:64 ratio) | Packet sample rate | 1:1 = debug only, production 1:1024+ |
| `polling` | `10` giây | Counter emit interval | Quá thấp (1s) = collector overwhelm |
| `targets` | `["IP:port"]` | Collector endpoint list | Multiple target = redundancy (good), duplicate data (OK cho sFlow) |
| `external_ids` | metadata | Operator tag | Missing = audit hard |
| Bridge `sflow` column | `[<uuid>]` | Bridge attach sFlow | Empty = sFlow config tồn tại nhưng không active |
| Packet rate trên target | ~N × hz | Verify actual sampling | < expected = sampling rate higher (less samples) |
| Counter emission rate | 1 pkt / polling seconds | Counter packet | Missing = counter disabled hoặc collector firewall |

**Kịch bản bẻ gãy:**

- **sFlow config exist nhưng collector không receive**: Bridge `sflow` column không attach. Verify: `ovs-vsctl get Bridge br0 sflow` — nếu empty, `ovs-vsctl set Bridge br0 sflow=<uuid>`.
- **Collector receive nhưng data incomplete**: `header=64` quá ngắn cho VXLAN+IPv6+TCP stack. Increase to 256.
- **Sampling rate drift**: Collector thấy ít sample hơn expected. OVS adaptive sampling adjusts nếu flow rate high. Check với `ovs-appctl sflow/show`.
- **Multiple target cause duplicate issue**: Một số collector dedup theo flow ID; một số không. Verify collector behavior trước khi setup redundancy.

## 9.8.8 Hiểu sai + Key Topic

> **Hiểu sai phổ biến 1:** *"sFlow và NetFlow cùng nhau = best of both worlds."*
>
> **Thực tế:** Enable cả 2 giao thức cost double CPU (2 sample path + 2 export path). Cả 2 giao thức overlap capability 80%. Chọn 1: sFlow cho operator không có historical commitment, NetFlow/IPFIX cho team có existing Cisco infrastructure. Dual-stack chỉ justify khi migration period giữa collector.

> **Hiểu sai phổ biến 2:** *"Sampling rate 1:1024 loss nhiều data thì inaccurate."*
>
> **Thực tế:** sFlow statistical accuracy tốt ở long time scale. 1:1024 trên 1 Gbps = ~1K sample/giây = đủ cho 5-minute flow aggregation với confidence interval < 5%. Short-term burst (sub-second) mới bị lose visibility. Cho long-term capacity planning + anomaly detection, 1:1024 là sweet spot CPU vs accuracy. Cho incident forensic chính xác, dùng Mirror (Part 9.7).

> **Chủ đề trọng tâm:** Flow monitoring là **passive observability** — không thay đổi traffic path, chỉ gửi metadata ra collector. Complement với active monitoring (BFD, ping) + pipeline debug (ovs-appctl ofproto/trace). 3-tier observability cho OVS: (1) Control plane = ofproto/trace + ovsdb-client monitor. (2) Data plane = Mirror (Phần 9.7) + flow monitoring. (3) Performance = coverage/show (Phần 9.11) + sFlow counter.

## 9.8.9 Schema row OVSDB for NetFlow

**Concept and role.** The `NetFlow` table in `vswitch.ovsschema` holds the runtime configuration for the NetFlow exporter. Each row describes one exporter target set; `Bridge.netflow` references one row through a UUID. The switch reads the row and emits NetFlow v5 records (the default) to every endpoint in `targets`.

**Columns (verified upstream `vswitchd/vswitch.ovsschema` at v2.17.9):**

| Column | Type | Meaning |
|--------|------|---------|
| `targets` | set of string, at least 1 | Collector endpoint `IP:port` (port 2055 is common) |
| `engine_type` | int 0-255, optional | The `engine_type` field in the NetFlow header; identifies the source |
| `engine_id` | int 0-255, optional | The `engine_id` field in the header; identifies a specific switch instance |
| `add_id_to_interface` | bool | When true, OR `engine_id` into the upper bits of the `input` and `output` interface IDs, which helps deduplicate when several switches export to the same collector |
| `active_timeout` | int at or above -1 | Seconds between two emits for a long-lived flow; the value `-1` disables active expiration |
| `external_ids` | map string to string | Operator metadata; does not affect runtime |

**Create and use.** The atomic-transaction idiom:

```bash
[host]# ovs-vsctl -- --id=@nf create NetFlow \
         targets=\"192.168.0.34:2055\" \
         engine_id=10 engine_type=128 active_timeout=30 \
         -- set Bridge br0 netflow=@nf
```

**Observation.** `ovs-vsctl list NetFlow` prints the entire row plus its UUID. `ovs-vsctl get Bridge br0 netflow` returns the currently attached UUID. An empty result `[]` means the bridge has not attached the configuration even though the row exists; this silent miss is a common operator mistake.

**Version differences.** The `engine_type`, `engine_id`, and `add_id_to_interface` columns have been stable since OVS 1.x. The `active_timeout = -1` semantic that disables active expiration was actually introduced earlier than the curriculum previously claimed: the code-side change in `ofproto/netflow.c` arrived in commit `e9e2856e084383b4f93b371dfba8320b51d4cd03` on 2009-12-06 ("netflow: Correctly set active timeout value."), which switched the validation check from `active_timeout > 0` to `active_timeout >= 0` and treated negative inputs as a request for the default. The schema documentation that explicitly describes `-1` as "disables active timeouts" was added shortly after in commit `8936565369410daa099708be4cd3fa7e0e39bade` on 2010-03-03 ("Add documentation for the vswitch database schema."). Both commits are contained in the v1.0.0 release tag. Earlier versions required `active_timeout >= 30`. Both SHAs were verified on 2026-04-28 against the local OVS repo with `git log -1 <sha>` and `git tag --contains <sha>`.

**Common failures.** A row exists but `Bridge.netflow=[]`, so the collector receives nothing. A wrong port in `targets`, so `tcpdump -ni any udp port 2055` is empty. An `engine_id` collision between two switches that share a collector causes records that the collector deduplicates by `engine_id` to be merged incorrectly.

## 9.8.10 Schema row OVSDB cho sFlow

**Khái niệm + vai trò.** Bảng `sFlow` lưu sampling-based exporter config. `Bridge.sflow` reference 1 row. ovs-vswitchd thread sampling chọn 1-trong-N packet trên mỗi datapath flow, copy `header` byte đầu + counter periodic.

**Columns (verified upstream `vswitchd/vswitch.ovsschema` v2.17.9):**

| Column | Kiểu | Ý nghĩa |
|--------|------|---------|
| `targets` | set of string, ≥1 | Collector `IP:port` (port 6343 chuẩn sFlow.org) |
| `sampling` | int, optional | Tỷ lệ 1-trong-N (default 400). Càng cao càng ít sample, càng ít CPU |
| `polling` | int, optional | Giây giữa 2 lần emit interface counter (default 30, 0 = disable) |
| `header` | int, optional | Byte đầu packet giữ lại (default 128). Quá nhỏ thiếu L4 header, quá lớn waste bandwidth |
| `agent` | string, optional | Tên interface dùng làm `Agent IP` trong sFlow datagram (collector dùng để identify switch). Nếu interface không tồn tại → exporter disabled silently |
| `external_ids` | map string→string | Operator metadata |

**Đặc thù sFlow vs NetFlow.** sFlow row không có `active_timeout` (sampling không cache flow); thay vào đó có `polling` cho counter. Sample emit ngay khi packet hit sampling rate, không buffer.

**Tạo + verify.**

```bash
[host]# ovs-vsctl -- --id=@s create sFlow agent=eth1 \
         target=\"10.0.0.1:6343\" header=128 sampling=64 polling=10 \
         -- set Bridge br0 sflow=@s
[host]# ovs-appctl sflow/show br0      # runtime stats: sample count, drop, last emit
```

**Khác biệt phiên bản.** Field `agent` interface-name semantic ổn định từ OVS 1.0. Default `sampling=400`/`polling=30`/`header=128` không đổi từ commit `9b0ed8e3` (2009 import).

**Sự cố thường gặp.** `agent=eth9` (interface không tồn tại) → datagram source IP rỗng → sflowtool reject; `header=64` ngắn → VXLAN/IPv6/TCP stack 86 byte cộng option bị truncate → collector không parse được L4.

## 9.8.11 Schema row OVSDB cho IPFIX

**Khái niệm + vai trò.** Bảng `IPFIX` lưu config IPFIX exporter (RFC 7011, NetFlow v10 standardized). Hai use case riêng biệt qua 2 reference khác nhau:

- `Bridge.ipfix` (per-bridge): exporter cho toàn bộ traffic qua bridge, sample rate cố định.
- `Flow_Sample_Collector_Set.ipfix` (per-flow `sample` action): exporter chọn lọc qua OpenFlow `sample` action, dùng cho selective monitoring.

**Columns (verified upstream `vswitchd/vswitch.ovsschema` v2.17.9):**

| Column | Kiểu | Ý nghĩa |
|--------|------|---------|
| `targets` | set of string, ≥0 | Collector `IP:port` (port 4739 default IPFIX, 4740 TLS) |
| `sampling` | int 1-4294967295, optional | Tỷ lệ 1-trong-N. Mặc định bridge mode = 400; per-flow mode dùng từ `sample` action |
| `obs_domain_id` | int 0-4294967295, optional | Observation Domain ID (RFC 7011 §3.1), phân biệt nguồn quan sát giữa multi-switch fabric |
| `obs_point_id` | int 0-4294967295, optional | Observation Point ID, phân biệt vị trí điểm quan sát trong cùng domain |
| `cache_active_timeout` | int 0-4200, optional | Giây giữ flow trong internal cache trước emit; max 4200 để tránh state bloat |
| `cache_max_flows` | int 0-4294967295, optional | Giới hạn entry cache; vượt → LRU evict + force emit |
| `other_config` | map string→string | Tunable nâng cao: `enable-tunnel-sampling`, `virtual_obs_id`, `enable-input-sampling`, `enable-output-sampling` |
| `external_ids` | map string→string | Operator metadata |

**Tạo bridge mode.**

```bash
[host]# ovs-vsctl -- --id=@i create IPFIX targets=\"192.168.0.34:4739\" \
         obs_domain_id=123 obs_point_id=456 \
         cache_active_timeout=60 cache_max_flows=13 \
         -- set Bridge br0 ipfix=@i
```

**Khác biệt phiên bản.** Column `cache_active_timeout` thêm OVS 2.0 (commit `978427a5` 2013) cùng IPFIX import; trước đó IPFIX hardcoded 60s. Range `cache_active_timeout` ≤ 4200 enforced từ schema 7.0.0.

**Sự cố thường gặp.** `obs_domain_id` chưa set → default 0 → multi-switch collector không phân biệt được nguồn; `cache_max_flows` quá nhỏ (< 100) trên link nhiều flow → LRU evict liên tục → record duplicate.

## 9.8.12 Bridge column references netflow/sflow/ipfix

**Khái niệm + vai trò.** 3 column trong bảng `Bridge` là weak reference type, mỗi column hold 0 hoặc 1 UUID đến row tương ứng:

```
Bridge.netflow : optional UUID → NetFlow row
Bridge.sflow   : optional UUID → sFlow   row
Bridge.ipfix   : optional UUID → IPFIX   row
```

**Vì sao tách bảng riêng thay vì inline column.** OVSDB normalize: nhiều bridge có thể chia sẻ cùng exporter config (cùng row UUID); rotate target chỉ cần update 1 row → mọi bridge attach reload atomic. Inline column phải sửa từng bridge → race window.

**Cách switch attach.** 2 form tương đương:

```bash
# Form 1, atomic transaction tạo row + attach một lệnh:
[host]# ovs-vsctl -- --id=@s create sFlow targets=\"127.0.0.1:6343\" agent=eth0 \
         -- set Bridge br0 sflow=@s

# Form 2, attach UUID đã tồn tại từ bridge khác:
[host]# UUID=$(ovs-vsctl get sFlow . _uuid | head -1)
[host]# ovs-vsctl set Bridge br1 sflow=$UUID
```

**Detach + garbage collection.** `ovs-vsctl clear Bridge br0 sflow` set column về `[]`. OVSDB weak reference: khi không bridge nào reference nữa, row tự bị garbage collect ở next compaction. Không cần `destroy` thủ công.

**Quan sát.**

```bash
[host]# ovs-vsctl get Bridge br0 sflow              # UUID hoặc []
[host]# ovs-vsctl --columns=name,sflow,netflow,ipfix list Bridge
```

**Sự cố thường gặp.** Row tồn tại nhưng column empty → exporter idle (Anatomy §9.8.7 red flag). Multi-bridge share 1 row → khi `clear` 1 bridge, row vẫn sống vì bridge khác still reference; nhầm tưởng đã clear hoàn toàn.

**Khác biệt phiên bản.** 3 column này stable từ OVS 1.0 (sflow), 1.0 (netflow), 2.0 (ipfix bridge mode).

## 9.8.13 Flow_Sample_Collector_Set, sampling chọn lọc theo flow

**Khái niệm + vai trò.** Bảng `Flow_Sample_Collector_Set` cho phép chọn lọc traffic export qua OpenFlow action `sample` (xem Phần 4.9 action catalog). Khác với `Bridge.ipfix` sample mọi gói tin theo rate cố định; collector set chỉ kích hoạt khi flow hit `sample(collector_set_id=N, ...)` action. Use case: production fabric monitor 100% traffic qua specific ACL match (ví dụ traffic giữa 2 tenant), không waste CPU sample broadcast/control.

**Columns (verified upstream `vswitchd/vswitch.ovsschema` v2.17.9):**

| Column | Kiểu | Ý nghĩa |
|--------|------|---------|
| `id` | int 0-4294967295, required | Collector set ID; `sample` action reference qua field này |
| `bridge` | UUID Bridge, required | Bridge mà collector set thuộc về (set scoped per-bridge) |
| `ipfix` | UUID IPFIX, optional | Reference đến IPFIX row chứa target/sampling/obs config |
| `external_ids` | map string→string | Operator metadata |

**Vì sao có bảng riêng thay vì column trong Bridge.** Mỗi bridge có thể có nhiều collector set với ID khác nhau, mỗi set trỏ đến IPFIX row khác nhau → 1-N relation, phải tách bảng. Plus `id` field chứ không phải UUID là vì `sample` action reference bằng integer, không thể embed UUID vào OpenFlow action bytes.

**Tạo + sử dụng.**

```bash
# Bước 1: tạo IPFIX row cho per-flow exporter
[host]# ovs-vsctl -- --id=@i create IPFIX targets=\"10.0.0.1:4739\" \
         obs_domain_id=200 cache_active_timeout=60 \
         -- --id=@cs create Flow_Sample_Collector_Set id=1 bridge=@br ipfix=@i \
         -- --id=@br get Bridge br0

# Bước 2: install flow rule với sample action ref id=1
[host]# ovs-ofctl add-flow br0 \
         "table=0,priority=100,ip,nw_src=10.0.0.0/8,actions=sample(probability=65535,collector_set_id=1,obs_domain_id=200,obs_point_id=300),normal"
```

`probability=65535` = sample 100% (max value 16-bit). `collector_set_id=1` match `Flow_Sample_Collector_Set.id`.

**Quan sát.** `ovs-vsctl list Flow_Sample_Collector_Set` in tất cả set + reference. `ovs-appctl dpif/show-dp-features br0` báo có support `sample` action không.

**Khác biệt phiên bản.** Bảng thêm OVS 2.4 (commit `8b7ea2d4` 2015) cùng OpenFlow `sample` action; trước đó chỉ có bridge-mode IPFIX. `obs_domain_id`/`obs_point_id` argument trong action override row default từ OVS 2.5.

**Sự cố thường gặp.** `collector_set_id` trong action không match row `id` → silent drop (không log); `bridge` reference sai → sample action thực thi nhưng không có IPFIX row → packet được sample trên datapath nhưng không export ra collector (CPU waste). Dùng `ovs-appctl dpctl/dump-flows` xem datapath có flag `sample_id` đúng không.

## 9.8.14 Capacity planning + tuning

**Sampling rate vs CPU overhead:**

| Sampling rate | CPU overhead (2× 10G) | Sample rate to collector | Use case |
|---------------|------------------------|--------------------------|----------|
| 1:1 | 50%+ CPU | 1M+ sample/s | DEBUG ONLY |
| 1:64 | 3-5% | 15K sample/s | Dense monitoring |
| 1:1024 | <1% | 1K sample/s | Production standard |
| 1:8192 | <0.2% | 120 sample/s | Very high-rate fabric |

**Collector scaling:**

- **sflowtool + single host**: 10K sample/s OK (disk I/O bottleneck ở 50K+).
- **ntopng**: ~50K sample/s với SSD.
- **goflow2 + Kafka + ClickHouse**: 1M+ sample/s sustained.

**Retention math:** 1K sample/s × 24h = 86M records/day. Mỗi record ~200 bytes → ~17 GB/day/collector (parquet compressed 30% → ~5 GB/day). 30-day retention = 150 GB.

## Điểm cốt lõi cần nhớ

Thứ nhất, ba giao thức phục vụ ba use case: sFlow cho high-volume traffic analysis (sampling), NetFlow/IPFIX cho accurate flow accounting.

Thứ hai, atomic transaction `-- --id=@` là idiom bắt buộc — đảm bảo reference đúng giữa Bridge và collector config.

Thứ ba, rotate collector atomic — OVS switch stream atomic sang target mới, không mất packet.

Thứ tư, feed sFlow vào goflow2/ntopng; NetFlow/IPFIX vào nfdump/Kentik — tổng hợp traffic analytics.

Thứ năm, Anatomy Template A `list sFlow` 9-attribute. Red flag: Bridge sflow column empty dù config exist.

Thứ sáu, dual sFlow+NetFlow cost 2×. Chọn 1 per use case. 1:1024 sampling là production sweet spot.

Thứ bảy, flow monitoring là passive observability — complement Mirror (data plane) + ofproto/trace (control plane). 3-tier framework.

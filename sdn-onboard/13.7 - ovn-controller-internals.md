## 13.7.1 The role of ovn-controller in the OVN architecture

In the distributed OVN architecture, **ovn-controller** is the daemon that runs on every chassis, hypervisor (compute node), or gateway chassis. A typical OVN cluster has one centralised SB DB cluster (3 Raft nodes) and tens to thousands of ovn-controller instances running locally on each chassis. This is the component that actually turns the OVN logical configuration intent (written into NBDB by a CMS, that is, a Cloud Management System such as OpenStack Neutron or OVN-Kubernetes) into concrete OpenFlow rules installed in the OVS kernel datapath.

The OVN **distributed control plane** model relies entirely on per-chassis ovn-controller instances. Unlike a centralised SDN controller (a single Ryu or Floodlight managing the whole cluster), OVN has no central entity computing OpenFlow for every switch. Instead, ovn-northd compiles NB intent down to SB Logical Flow (a vendor-neutral representation, independent of any chassis), and each ovn-controller translates only the subset of Logical Flow that corresponds to ports bound on its local chassis. This model lets OVN scale beyond what a central controller can reach. If the SB DB has 100K Logical Flow entries for a 1000-chassis cluster, each chassis still only has to process roughly the 100 Logical Flow entries relevant to its local ports.

ovn-controller connects to two main partners. (1) The **SB DB through the OVSDB protocol** (port 6642 by default), using the OVSDB IDL client to subscribe to changes in the Logical_Flow, Port_Binding, Chassis, Datapath_Binding, and similar tables. (2) The **local OVS instance through a Unix socket** (`/var/run/openvswitch/db.sock`), reading the configuration of `br-int` and writing OpenFlow rules directly through the `ovs-ofctl` style protocol. The two connections run in parallel inside the main event loop of ovn-controller, and any change at one end triggers recomputation at the other end.

In addition to its **consumer** role for the SB DB, ovn-controller is also a **producer**. It writes a record into the `Chassis` table at startup (registering the chassis with the cluster), writes into `Port_Binding.chassis` when a VIF is bound locally (telling the cluster which pod or VM is running where), writes into `MAC_Binding` when ARP or NDP is resolved, and writes `Service_Monitor` status for health checks. Because ovn-controller writes directly to the SB DB, every chassis needs write permission on the SB DB. This is why OVN uses RBAC with certificate-based authentication, restricting each chassis to writing only the records that belong to it.

> **Version note:** OVN 20.03 and earlier had a simple ovn-controller that polled the SB DB directly and recomputed everything on every change. OVN 20.06 introduced **IDL-based incremental processing** (the I-P engine), recomputing only the subset of Logical Flow affected by a SB change. OVN 22.03 (the curriculum baseline) stabilised the I-P engine and extended it to Port_Binding and MAC_Binding. OVN 24.03 added incremental processing for load balancer health monitoring.

## 13.7.2 The lifecycle and work flow of ovn-controller

When it starts, ovn-controller runs an initialisation sequence and then enters an infinite event loop. Understanding this work flow is the foundation for diagnosing any case where OVN does not forward a packet as expected.

**Phase 1, connection setup:**
1. Read configuration from `/etc/openvswitch/conf.db` (the OVS local DB). Pick up `external_ids:ovn-remote` (SB DB cluster address), `external_ids:ovn-encap-type` (Geneve, VXLAN, or STT), and `external_ids:ovn-encap-ip` (the local tunnel endpoint IP).
2. Open a Unix-socket connection to the local `ovs-vswitchd` to monitor the `Interface`, `Port`, and `Bridge` tables.
3. Open a TCP or TLS connection to the SB DB cluster through the OVSDB protocol (RFC 7047). Use the IDL to subscribe to the required tables, including `Chassis`, `Chassis_Private`, `Encap`, `Port_Binding`, `Logical_Flow`, `Multicast_Group`, `Datapath_Binding`, `Meter`, `DHCP_Options`, `DNS`, and `Controller_Event`.
4. Register the chassis with the cluster by writing a record into `Chassis` and `Chassis_Private` with name = chassis UUID, hostname, encaps, and transport_zones.

**Phase 2, initial inventory:** once the IDL connection is established, ovn-controller receives a full snapshot of every subscribed table. The first processing pass is a **full recompute**, walking the entire Logical_Flow set and filtering the subset relevant to the local chassis (based on the Datapath_Binding entries to which the local Port_Binding records belong).

**Phase 3, the main event loop:** after the inventory pass, ovn-controller enters a `poll, wait, process change, update OVS` loop. On each iteration:
- `poll_block()` waits for an event from (a) an SB DB IDL notification (Logical Flow change, Port_Binding bind or unbind), (b) a local OVS notification (interface up or down, port added or removed), or (c) a timer (periodic tasks such as BFD probes or chassis keepalives).
- When an event arrives, classify the change and call the matching handler.
- The handler computes the OpenFlow rule delta (incremental) and calls `ofctrl_put()` to install the rule on br-int through the OpenFlow protocol.
- Write any state updates back to the SB DB (for example `Port_Binding.chassis` when a new VIF is bound locally).

**Phase 4, graceful shutdown:** when ovn-controller receives SIGTERM, it can either keep the OpenFlow rules installed on br-int (fail-closed mode) or wipe them (fail-open mode), depending on the `external_ids:ovn-ofctrl-wait-before-clear` setting. The default in OVN 22.03 is to keep the rules so that an ovn-controller upgrade does not break connectivity.

## 13.7.3 Incremental processing (the I-P engine), the core of OVN scalability

**Incremental processing** (often abbreviated to I-P) is the most important architectural change in OVN over the last five years. Before OVN 20.06, every SB DB change (even a single Logical Flow) triggered a full recompute. ovn-controller walked the entire Logical_Flow table, rebuilt the full OpenFlow rule set, and then diffed against the current state on br-int to install or remove the delta. On a cluster with 10K Logical Flow entries, one full recompute consumed 500ms to 2s of CPU, and it ran on every event. The control plane could not keep up on large clusters.

The I-P engine replaces that model with **dependency tracking** and **targeted recompute**. Each Logical Flow is keyed by (datapath_id, pipeline, table_id, priority, match). When the SB DB notifies "Logical Flow X has been updated", ovn-controller (1) finds the OpenFlow rules on br-int that depend on it (looked up through a hash index), (2) recomputes only those rules, and (3) diffs against the current state and installs the delta. Complexity drops from O(N) to O(log N) or close to O(1) for a local change.

The I-P engine is structured into **stages**. Each stage processes one type of change and produces output for the next stage. The main stages in OVN 22.03 are:
- The `runtime_data` stage, which processes changes in Port_Binding, Datapath_Binding, and Chassis, and determines which datapath entries are local.
- The `non_vif_data` stage, which processes the special port types (localnet, l2gateway, chassisredirect, patch).
- The `physical_flow_changes` stage, which computes the physical OpenFlow rules for changes in port binding.
- The `Logical_Flow` stage, which translates Logical Flow into OpenFlow, using the `lflow_cache` to skip retranslation for unchanged flows.
- The `flow_output` stage, which emits the final OpenFlow rules to OVS.

Each stage is a node in a dependency graph. When the input of a stage changes, the output of that stage (and every downstream stage) is invalidated and recomputed. If the I-P engine encounters a kind of change that it does not know how to handle incrementally, it **falls back** to a full recompute (called an "engine recompute"). This is the safety net that preserves correctness, at the cost of performance.

**Observing the fallback in practice:** the `ovn-controller.log` shows lines like `engine: incremental update completed in X ms` when I-P succeeds. On fallback, the log emits `engine: incremental processing failed, falling back to recompute`. In a large production cluster, the fallback rate should stay below 5 percent. A higher rate means a new operation pattern (for example a CMS performing bulk Port_Binding updates) is not yet optimised in the I-P engine and is worth reporting upstream.

> **Performance note (production environment):** A Red Hat 2023 benchmark on an OVN 22.03 cluster with 300 chassis and 50K Logical Flow entries showed that the I-P engine reduced ovn-controller CPU from an average of 40 percent (full recompute mode) to 3 to 5 percent (I-P mode) under steady state. A peak event (scaling out 100 pods at once) dropped from about 10 seconds of control-plane downtime to under one second. This is the factor that lets OVN scale to hyperscale Kubernetes (10K+ nodes).

## 13.7.4 Translating from Logical Flow to OpenFlow

The heart of ovn-controller is the engine that translates Logical Flow (a neutral specification in the SB DB) into concrete OpenFlow rules for OVS. This is where ovn-controller "realises" the OVN pipeline concept on br-int.

A Logical Flow entry has the following shape:
```
Datapath   Table  Priority  Match                    Actions
-----------------------------------------------------------------
ls1        ls_in_l2_lkup  50  eth.dst == aa:bb:cc:dd:ee:ff
                                                     outport = "vm1";
                                                     next;
```
With `ls1` as a Logical Switch, `ls_in_l2_lkup` is logical table 24 in the LS ingress pipeline at v22.03.8 (verified at `northd/northd.c:127` `PIPELINE_STAGE(SWITCH, IN, L2_LKUP, 24, "ls_in_l2_lkup")`); it matches the L2 destination. Translating to OpenFlow requires the following.

**(1) Datapath to metadata mapping:** every Logical Datapath has a `tunnel_key` (`Datapath_Binding.tunnel_key`, typically 1 to 1000). ovn-controller uses tunnel_key as the `metadata` field in the OpenFlow match, so flows from one datapath are distinguished from flows from another datapath on the same br-int.

**(2) Logical table to OpenFlow table mapping (verified at v22.03.8):** OVN uses 73 OF tables on br-int at v22.03.8 (numbered 0 to 72; see `controller/lflow.h` lines 64 to 78 for the OFTABLE_* constants). The LS and LR ingress pipelines share OF tables 8 onward (LS occupies 8 to 33 for its 26 stages; LR occupies 8 to 27 for its 20 stages; the two are distinguished by `metadata` carrying the datapath tunnel_key). The egress pipelines start at OF table 40 (LS occupies 40 to 49 for its 10 stages; LR occupies 40 to 46 for its 7 stages). OF table 0 handles physical-to-logical ingress classification.

**(3) Match translation:** a Logical Flow match uses OVN-specific fields such as `inport`, `outport`, `reg0` to `reg15`, `eth.src`, `ip.dst`, and `ct_state`. ovn-controller maps them to OpenFlow fields:
- `inport` to `reg14` (OVN uses reg14 to hold the logical input port)
- `outport` to `reg15`
- `eth.src` and `eth.dst` to `eth_src` and `eth_dst`
- `ip.src` and `ip.dst` to `nw_src` and `nw_dst`
- `ct_state` to `ct_state`

**(4) Action translation:** Logical Flow actions have their own syntax, compiled to OpenFlow:
- `next;` becomes `resubmit(,table+1)`, jumping to the next logical table.
- `output;` becomes `resubmit(,32)`, transitioning from the ingress to the egress pipeline.
- `ct_commit;` becomes `ct(commit, zone=NXM_NX_REG13[0..15], ...)`, committing a connection into conntrack.
- `load 1 -> NXM_NX_XXREG0[96..127]`, depending on the specific action.

A single Logical Flow can produce several OpenFlow rules if the match contains a `reg0` to `reg15` placeholder that has to be expanded. The `lflow_cache` inside ovn-controller stores the translation result for unchanged Logical Flow entries, avoiding retranslation. This is an important optimisation for scalability.

**(5) Physical-to-logical ingress classification (table 0):** when a packet enters br-int from a local VIF or a tunnel port, table 0 sets `metadata` (datapath_id) and `reg14` (inport), then resubmits to table 8 to start the ingress pipeline. This is the "demux" mechanism that lets ovn-controller share one br-int across many datapath entries.

## 13.7.5 The Chassis table and hypervisor registration

When ovn-controller starts on a new hypervisor, it registers the chassis with the cluster by writing a record into the `Chassis` and `Chassis_Private` tables in the SB DB. This is how OVN knows which chassis is online, which tunnel endpoint it is advertising, and which features it supports.

**Important fields in the Chassis table:**
- `name`: the UUID identifying the chassis (generated on first startup, stored in `external_ids:system-id` of the OVS local DB so that it persists across restarts).
- `hostname`, the chassis hostname.
- `encaps`, a reference to a record in the `Encap` table describing the tunnel endpoint (type = geneve, vxlan, or stt; ip = the IP that this chassis uses for overlay traffic).
- `other_config:ct-zone-limit`, the cap on the number of conntrack zones the chassis may use.
- `nb_cfg`, a counter synchronised with the NB DB. It increases each time the chassis has finished processing the most recent NB change. It is used by OCC (OVN Consistent Check).
- `transport_zones`, the list of transport zones the chassis joins (typically one zone for a small cluster, several zones for a multi-cluster deployment).

**The Chassis_Private table (split out from Chassis in OVN 20.12):** it holds fields that should not be replicated to other chassis, including `chassis`, `nb_cfg_timestamp`, and `external_ids`. The split reduces replication traffic between chassis (each ovn-controller subscribes only to its own Chassis_Private record).

**Chassis features:** OVN stores the feature flags directly inside the `Chassis.other_config` column of the Southbound DB (a key to value map), **not in a dedicated table** (an older curriculum revision said "table `Chassis_features`", which is incorrect, verified by `mcp__github__get_file_contents(ovn-sb.ovsschema, ref=v22.03.8)`; the tables are `Chassis`, `Chassis_Private`, `Gateway_Chassis`, `HA_Chassis`, and `HA_Chassis_Group`, with no table named `Chassis_features`). Typical feature keys include `port-up-notif`, `is-remote`, and `is-interconn`. At the in-memory C struct layer, ovn-northd has a `struct chassis_features` (defined in `northd/northd.h` v22.03.8) that aggregates feature flags from every local chassis. It has only two fields: `ct_no_masked_label` and `ct_lb_related`. The control plane uses this information to generate conditional Logical Flow. From OVN 24.03 onward, the struct grew additional fields such as `fdb_timestamp` and `ovn_lb_hairpin_use_ct_mark`. When citing a specific feature name, verify it at the matching version baseline.

**When a chassis goes offline:** if ovn-controller crashes or the chassis reboots without a clean shutdown, the `Chassis` record still lives in the SB DB (a stale record). `ovn-sbctl chassis-del <name>` removes it manually. OVN 22.03 introduced the `auto_evict` flag in `Chassis_Private`. If the timestamp does not refresh within X seconds (default 10s times the BFD interval), the other chassis automatically mark the chassis offline and clean up dependent state (for example, removing tunnel ports to the dead chassis).

## 13.7.6 Diagnostic tools for ovn-controller

When OVN is not forwarding packets as expected, the diagnostic chain typically goes through (1) Is the NB intent correct? (2) Has ovn-northd compiled it into SB? (3) Has ovn-controller installed OpenFlow on br-int? (4) Does the OVS datapath execute correctly? Most "OVN does not work" issues land at step 3, where ovn-controller has not translated the Logical Flow set fully or has translated it incorrectly.

**Tool 1, `ovn-sbctl lflow-list`:** view the Logical Flow set in the SB DB for a specific datapath.
```
[chassis]# ovn-sbctl lflow-list ls1
Datapath: "ls1" (f4d2-...)  Pipeline: ingress
  table=0 (ls_in_port_sec_l2), priority=100, match=(inport == "vm1"), action=(next;)
  table=1 (ls_in_port_sec_ip), priority=90, match=(inport == "vm1" && eth.src == ...), ...
```
This output is the input that ovn-controller translates. If a Logical Flow is missing (for example, port_security has no entry), the bug is in ovn-northd, not in ovn-controller.

**Tool 2, `ovs-ofctl dump-flows br-int`:** view the actual OpenFlow rules on br-int (the output of the ovn-controller translator).
```
[chassis]# ovs-ofctl dump-flows br-int | grep "metadata=0x1"
 cookie=0x1234, duration=3s, n_packets=10, table=8, priority=100,
   metadata=0x1,reg14=0x1 actions=resubmit(,9)
```
The `metadata` field is the datapath tunnel_key. Compare with `ovn-sbctl lflow-list` output to verify that translation is correct. `n_packets` shows whether the rule is being hit.

**Tool 3, `ovn-trace`:** simulate a packet through the logical pipeline without sending a real packet.
```
[chassis]# ovn-trace ls1 'inport == "vm1" && eth.src == aa:bb:cc:dd:ee:ff && eth.dst == 11:22:33:44:55:66 && ip4.src == 10.0.0.1 && ip4.dst == 10.0.0.2'
```
The output shows each step the packet takes through each logical table, which fields match, and which actions apply. Extremely useful when diagnosing ACL drops, load balancer backend selection failures, and missing NAT.

**Tool 4, `ovs-appctl` for ovn-controller:**
```
[chassis]# ovs-appctl -t ovn-controller inject-pkt ...  # inject a test packet
[chassis]# ovs-appctl -t ovn-controller debug/delay-nb-cfg-report 5  # delay the report
[chassis]# ovs-appctl -t ovn-controller vlog/set console:ofctrl:dbg  # enable debug log for the ofctrl module
[chassis]# ovs-appctl -t ovn-controller memory/show  # check memory usage
```

**Tool 5, log levels through vlog:** ovn-controller has many separate log modules: `ofctrl` (OpenFlow control), `physical` (physical flow), `lflow` (logical flow translation), `binding` (port binding), and `encaps` (tunnel). Enabling debug for the relevant module avoids flooding the log with output from every other part of the daemon.

**Tool 6, coverage counters:** ovn-controller exports internal counters through `ovs-appctl -t ovn-controller coverage/show`. These include the number of engine recomputes, the number of Logical Flow entries processed, and the number of OpenFlow rules installed. They are useful as a baseline when you want to measure performance improvements after tuning.

### Guided Exercise 1: Observe ovn-controller translating Logical Flow into OpenFlow

> **State verification lab:** the output is based on OVN 22.03 documentation. Pending live verification on Ubuntu 22.04 with OVS 2.17 and OVN 22.03.

**Goal:** Show experimentally that ovn-controller is the translator. A single Logical Flow in the SB DB corresponds to one or more OpenFlow rules on br-int. When the Logical Flow disappears, the OpenFlow rule disappears too.

**Setup:**
- One Ubuntu 22.04 node with OVS 2.17 and OVN 22.03 installed (both ovn-central and ovn-host on the same node for a simple lab).
- `ovn-sbctl`, `ovn-nbctl`, and `ovs-ofctl` available in PATH.

**Lab topology:**

```
+----------------------------------------------+
|         Ubuntu 22.04 VM                      |
|                                              |
|   +----------------+   +----------------+    |
|   | ovn-central    |   | ovn-controller |    |
|   | (nbdb/sbdb/    |-->|   (local)      |    |
|   |  northd)       |   |                |    |
|   +----------------+   +--------+-------+    |
|                                 |            |
|                                 v            |
|                        +-----------------+   |
|                        |     br-int      |   |
|                        | (OVS bridge)    |   |
|                        +-----------------+   |
|                                              |
|   Logical Switch ls1 with one port "vm1"     |
+----------------------------------------------+
```

**Steps:**

**1.** Create a Logical Switch and a port:

```
[host]# ovn-nbctl ls-add ls1
[host]# ovn-nbctl lsp-add ls1 vm1
[host]# ovn-nbctl lsp-set-addresses vm1 "aa:bb:cc:dd:ee:01 10.0.0.1"
[host]# ovn-nbctl lsp-set-port-security vm1 "aa:bb:cc:dd:ee:01 10.0.0.1"
```

**2.** List the Logical Flow set in the SB DB for ls1:

```
[host]# ovn-sbctl lflow-list ls1 | head -20
```

Expected output (abridged):
```
Datapath: "ls1" (...)  Pipeline: ingress
  table=0 (ls_in_port_sec_l2), priority=50, match=(inport == "vm1"), action=(next;)
  table=1 (ls_in_port_sec_ip), priority=90, match=(inport == "vm1" && ...)
  ...
```

**3.** View OpenFlow rules on br-int filtered by the metadata of ls1. Get the tunnel_key:

```
[host]# ovn-sbctl get Datapath_Binding ls1 tunnel_key
5
```

**4.** Dump OpenFlow rules with metadata = 0x5:

```
[host]# ovs-ofctl dump-flows br-int | grep "metadata=0x5" | head -10
```

Expected output: many OpenFlow rules. Each rule is the translation result of one Logical Flow (or a sub-flow when the Logical Flow has a complex match). Notice the match contains `metadata=0x5` (datapath ls1) and `reg14` (the inport mapping).

**5.** Delete port vm1 and observe the rules disappear:

```
[host]# ovn-nbctl lsp-del vm1
[host]# sleep 2
[host]# ovs-ofctl dump-flows br-int | grep "metadata=0x5" | wc -l
```

Expected output: the rule count drops noticeably (rules for port vm1 are removed; only the shared rules of the Logical Switch remain).

**Lessons learned:**

First, ovn-controller is not a "passive translator". It actively tracks SB DB changes and installs or removes OpenFlow rules in response. The latency between `ovn-nbctl ls-add` and the appearance of rules on br-int is typically under 500ms on a small cluster, and a few seconds on a large cluster.

Second, one Logical Flow can produce several OpenFlow rules. For example, a Logical Flow matching `eth.src == aa:bb:cc:dd:ee:01` can compile into 2 to 3 OpenFlow rules if the match contains a `reg` placeholder that has to be expanded. This is why a cluster with 10K Logical Flow entries can hold 30K to 50K OpenFlow rules on br-int.

Third, debugging OVN **requires** a comparison across three layers: NB intent (`ovn-nbctl show`), SB Logical Flow (`ovn-sbctl lflow-list`), and OpenFlow physical (`ovs-ofctl dump-flows br-int`). If NB is correct but SB is missing an entry, the bug is in ovn-northd. If SB is correct but the OpenFlow rule is missing, the bug is in ovn-controller.

**Cleanup:**

```
[host]# ovn-nbctl ls-del ls1
```

## 13.7.7 ovn-controller run loop, anatomy of main_loop and the engine graph

Section 13.7.2 described the ovn-controller lifecycle at a high level. Section 13.7.7 goes deeper into the actual run loop: the iteration of `main_loop()`, the engine graph nodes, and the events that drive recomputation. An engineer needs to know what each iteration does in order to diagnose CPU spikes or sync lag.

### Anatomy of one `main_loop()` iteration

On each pass through `main_loop`, ovn-controller runs five sequential steps:

1. **Poll triggers:** wait up to about 5 seconds on four event sources:
   - An OVSDB IDL notification from the Southbound DB (`sb_ovsdb_idl`), triggered when the SBDB has a new commit.
   - An OVSDB IDL notification from the OVS local bridge configuration (`ovs_idl`), triggered when `ovs-vsctl` changes something.
   - A packet-in upcall from `ofctrl` (the OpenFlow controller connection to br-int).
   - A timer tick (BFD tick, periodic stats collection).

2. **Reload IDL:** fetch changes from OVSDB if the sequence number has advanced, and build a fresh in-memory view.

3. **Engine graph run:** run the I-P (Incremental Processing) engine in dependency order. Each node has a `_run()` function, and a node is skipped when none of its inputs have changed.

4. **Commit and notify:** if the engine output changed, emit OpenFlow `Flow_Mod` messages through `ofctrl`, and update the `Chassis_Private.nb_cfg` sequence number to tell ovn-northd "I am synchronised up to this NBDB version".

5. **Loop back:** return to step 1.

Typical iteration duration: 5 to 50ms when the cluster is steady, 500ms to 2s during a bulk update, 5 to 30s during a full `engine_recompute` fallback.

### The engine graph and dependency tree

The I-P engine is structured as a DAG (directed acyclic graph). The main nodes (in `controller/ovn-controller.c` and `controller/lflow.c`) are:

```
en_sb_chassis           <- Chassis + Chassis_Private tables
en_sb_encap             <- Encap table
en_runtime_data         <- Port_Binding + Datapath_Binding tables
         |
         v
en_physical_flow_output <- runtime_data + OF connection
en_lflow_output         <- runtime_data + Logical_Flow table
         |
         v
en_flow_output          <- merge physical + lflow outputs
         |
         v
ofctrl_run()            <- emit OpenFlow Flow_Mod through br-int
```

When `Port_Binding` changes (a VM migration, a new port claim):
- `en_runtime_data` computes the diff in port bindings (incremental).
- `en_physical_flow_output` computes the OpenFlow rule diff (the affected tunnel flow plus local port flow).
- `en_lflow_output` is skipped because the Logical_Flow table did not change.
- `ofctrl_run` emits Flow_Mod ADD or DEL for the diff. It does not rebuild the full flow table.

A full recompute fallback is triggered when a node does not support an incremental update (a schema change, a bulk update that is too large):
- The `engine_recompute` counter increments by 1.
- The entire output is regenerated from scratch, 10 to 100 times slower than incremental.

### Events that trigger recomputation

| Event | Trigger node | Incremental? |
|-------|--------------|:--:|
| Port_Binding.chassis change | `en_runtime_data` to `en_physical_flow_output` | Yes |
| Logical_Flow insert or update | `en_lflow_output` | Yes |
| Chassis register or deregister | `en_sb_chassis` to `en_physical_flow_output` | Yes |
| Datapath_Binding.tunnel_key change | `en_runtime_data` full rebuild | No, rare |
| OVSDB schema version change | Full engine recompute | No |
| BFD state change | `ofctrl` periodic (does not go through the engine) | n/a |
| ovn-controller restart | First iteration is a full recompute | No, one-shot |

### Timing characteristics

- **Steady state on a stable cluster:** main_loop ticks every 5s timer-driven, or whenever an SB notification arrives. Iteration takes 5 to 20ms. CPU is below 1 percent.
- **Burst update:** 100 port adds or deletes within 1 second. main_loop runs 3 to 5 times, each time processing a diff batch. I-P scales O(1) per event. CPU spikes to 10 to 20 percent for a few seconds.
- **Full recompute:** after an ovn-controller restart or a schema change, the first iteration can take 5 to 30s on a cluster with 500+ chassis and 10K+ Logical_Flow entries. After the first iteration, the system returns to steady state.
- **Sustained load:** on a 1000-chassis cluster with constant churn (continuous VM migrations), ovn-controller runs at a chronic 10 to 30 percent CPU on a single core.

### Diagnostic recipe for the run loop

**Measure the duration of each iteration:**

```bash
sudo ovs-appctl -t ovn-controller vlog/set ovn-controller:main:info
sudo journalctl -u ovn-controller --since "2 min ago" | grep -i "iteration"
# When the duration crosses the threshold, a log entry appears.
```

**Engine coverage counters:**

```bash
sudo ovs-appctl -t ovn-controller coverage/show | grep -E "engine|ofctrl"
# Key counters:
# engine_run: total main_loop iterations.
# engine_run_changed: iterations where the output changed.
# engine_compute: incremental pass succeeded.
# engine_recompute: full rebuild fallback (target: under 1 percent of engine_run).
# ofctrl_flow_add/del: Flow_Mod emit count.
```

A healthy ratio is `engine_recompute / engine_run < 0.01` (under 1 percent). If it exceeds 5 percent, investigate schema drift or a bulk-update pattern.

**Observe an engine node in real time:**

```bash
# Pause the engine on the next iteration.
sudo ovs-appctl -t ovn-controller debug/pause
# Look at which node is executing.
sudo ovs-appctl -t ovn-controller debug/status
# Resume.
sudo ovs-appctl -t ovn-controller debug/resume
```

Note: `debug/pause` halts ovn-controller for several seconds. Do not use it on production while traffic is active.

### When diagnosing sync lag

Typical symptom: after `ovn-nbctl lsp-add` returns, more than 10 seconds pass and the OpenFlow rule still does not appear on br-int.

Systematic workflow:

**1. Check northd first** (it might be stuck):
```bash
ovn-sbctl get NB_Global . nb_cfg
ovn-sbctl get SB_Global . nb_cfg
# If NB > SB, northd has not yet translated NBDB into SBDB.
# Fix: check northd daemon health, restart if needed.
```

**2. Check ovn-controller if northd is fine:**
```bash
CHASSIS=$(ovs-vsctl get open_vswitch . external_ids:system-id | tr -d '"')
ovn-sbctl get SB_Global . nb_cfg
ovn-sbctl --if-exists get Chassis_Private $CHASSIS nb_cfg
# If SB > Chassis_Private, ovn-controller has not yet synchronised.
```

**3. Coverage check:**
```bash
sudo ovs-appctl -t ovn-controller coverage/show | grep -E "engine_recompute|engine_run_changed"
# Compare two snapshots 30 seconds apart.
# If engine_recompute is climbing, I-P is falling back. Look for schema drift.
```

**4. Engine status snapshot:**
```bash
sudo ovs-appctl -t ovn-controller debug/status
# Identify the blocking node and check duration.
```

**5. Warning and error logs:**
```bash
sudo journalctl -u ovn-controller --since "5 min ago" -p warning
# Common entries: "Error translating Logical Flow", "OpenFlow install timeout",
# "IDL reconnecting to SBDB".
```

Typical root causes and fixes:
- **I-P fallback (engine_recompute much greater than 0):** schema upgrade mismatch, or a bulk update that is too large. Fix: complete the rolling upgrade, or batch smaller NBDB commits.
- **OpenFlow install timeout:** br-int is overloaded with flow count. Fix: check `ovs-appctl ofproto/list-tunnels` and reduce the flow count.
- **IDL reconnecting:** SBDB unreachable. Fix: check the network underlay and the SBDB leader health.

## 13.7.8 Source code physical.c, Port_Binding claim, Geneve TLV

Section 13.7.7 went deep into `main_loop()` and the engine DAG inside `controller/ovn-controller.c`. This section focuses on `controller/physical.c` (1933 lines at `branch-22.03`), where OVN Logical Flow becomes the actual OpenFlow rules installed on `br-int`.

### The `physical_run()` architecture in `controller/physical.c`

`physical_run()` (around L1716) is the entry point of the `physical_flow_output` engine node. Its job: translate every `Port_Binding` and `Multicast_Group` row in the SB DB into concrete flow rules on br-int.

```
physical_run(p_ctx, flow_table)              [controller/physical.c L1716]
  |-> COVERAGE_INC(physical_run)
  |-> put_chassis_mac_conj_id_flow()         (chassis MAC conjunction filter, table 0)
  |-> for each Port_Binding row:
  |     consider_port_binding()              [L913]
  |       |-> Classify the type: vif | localnet | l2gateway | chassisredirect | patch | localport | virtual
  |       |-> If local: put_local_common_flows() [L732] (table 0, 64, 65)
  |       \-> If remote: put_remote_port_redirect_*() (table 38)
  |-> for each Multicast_Group row:
  |     consider_mc_group()                  [L1463]
  |       \-> Generate the flood action: clone(output_to_chassis_tunnel; output_to_local_port)
  \-> for each chassis_tunnel:
        consider_chassis_tunnel()            (table 33)
```

The output flow rules accumulate in a `struct ofpbuf ofpacts`, which is then merged into `ovn_desired_flow_table`. The engine compares that against `installed_flow_table` to compute the install or remove delta sent through OpenFlow.

### Port_Binding claim logic (race conditions)

**Problem:** When a VM migrates from chassis A to chassis B (live migration), both chassis can briefly believe they "own" the port. If both install an output flow, the result is a broadcast storm or duplicate packets.

**OVN 22.03 solution (`Port_Binding.chassis` eager claim):**

1. Chassis A currently owns the port (`Port_Binding.chassis = chassis_A_uuid`).
2. The CMS calls `ovn-nbctl lsp-set-options <port> requested-chassis=B`.
3. ovn-northd updates `Port_Binding.requested_chassis` in SB.
4. ovn-controller B reads the change and performs an **eager claim**: it sets `Port_Binding.chassis = chassis_B_uuid` immediately (without waiting for A to release).
5. ovn-controller A reads the chassis change, drops the port flows from local br-int.
6. During a 50 to 500ms window (Raft propagation plus an I-P engine cycle), both chassis briefly hold flows. Duplicate is possible. OVN 22.06 added `requested_chassis` atomic semantics to shrink this window.

> **Version warning:** A case study (Part 20.5 §20.5.1) recorded a 3 to 18 second dual-bind window on an OVN 22.03 cluster with slow SB Raft propagation. OVN 22.06 with atomic `requested_chassis` reduces it to under 500ms. A production operator must know the version baseline and the specific risk.

**Source-code anchor for the claim logic:**

The function `binding_lport_set_chassis_changed()` in `controller/binding.c` (`branch-22.03`) handles changes to the `chassis` field. It is called from the `runtime_data` engine node change handler. If `Port_Binding.chassis` flips from A to B, ovn-controller A marks the port as "no longer local" and removes the corresponding flows on the next cycle.

### Geneve TLV encoding for cross-chassis metadata

OVN uses Geneve (RFC 8926) as the default tunnel. Each packet across the tunnel carries three pieces of metadata so that the receiving chassis can recover the logical context:

- `tunnel_key` (24 bits): the logical datapath identifier (LS or LR), assigned by ovn-northd.
- `outport` (32 bits): the logical egress port on the receiving chassis.
- `inport` (15 bits): the logical ingress port on the sending chassis.

That is 71 bits of metadata. Geneve allows variable-length TLVs (Type-Length-Value). OVN uses **TLV class `0x0102`** with two fields:

- `MFF_TUN_ID` (the default Geneve VNI): carries the 24-bit `tunnel_key`.
- `mff_ovn_geneve` (an OXM defined by OVS plus OVN, matching TLV class 0x0102): 32 bits, carrying `outport` (bits 0 to 31) and `inport` (bits 16 to 30 after a 15-bit move).

**Source-code anchor (`controller/physical.c` around L148):**

```c
put_encapsulation(enum mf_field_id mff_ovn_geneve,
                  const struct chassis_tunnel *tun,
                  const struct sbrec_datapath_binding *datapath,
                  uint16_t outport, bool is_ramp_switch,
                  struct ofpbuf *ofpacts)
{
    if (tun->type == GENEVE) {
        put_load(datapath->tunnel_key, MFF_TUN_ID, 0, 24, ofpacts);
        put_load(outport, mff_ovn_geneve, 0, 32, ofpacts);
        put_move(MFF_LOG_INPORT, 0, mff_ovn_geneve, 16, 15, ofpacts);
    } else if (tun->type == STT) {
        ...
    } else if (tun->type == VXLAN) {
        ...
    }
}
```

When ovn-controller installs a tunnel egress flow, this function adds three OpenFlow actions:

1. `set_field:tunnel_key->tun_id` (Geneve VNI 24-bit).
2. `set_field:outport->NXM_NX_TUN_METADATA0[0..31]` (the OVN OXM, 32-bit outport).
3. `move:NXM_NX_REG14[0..14]->NXM_NX_TUN_METADATA0[16..30]` (move the inport into the TLV).

On the receiving chassis, table 0 uses the inverse pattern: load `tun_id` into `metadata`, load `tun_metadata0` into `reg15` (logical inport), load `tun_metadata0` into `reg14` (logical outport), then resubmit into the ingress pipeline of the datapath.

### Anatomy Template A: `ovs-appctl -t ovn-controller debug/dump-local-bindings`

> **When to run:** the operator suspects which port is claimed on the current chassis, or wants to trace why a specific port has no flow on br-int.

```bash
ovs-appctl -t ovn-controller debug/dump-local-bindings
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| `Local bindings` count | Matches the `ovs-vsctl` interface count with `external_ids:iface-id` | Mismatch | Local count is 0 but br-int has ports | runtime_data engine node bug; check `inc-engine/show` |
| `pb` UUID per port | UUID present and matches an SB Port_Binding | UUID not found in SB | UUID empty | SB sync lag; see `sbdb_idl_status` |
| `claimed` flag | `true` for every local port | `pending` | `false` while the OVS interface is up | Claim race with another chassis; check `Port_Binding.chassis` in SB |
| `iface-id-ver` | Matches the SBDB version | Off by one cycle | Off by many cycles | Bulk update backlog |
| `Patch ports` count | About the number of localnet bindings | Less than the number of localnet bindings | 0 while localnet bindings exist | `add_localnet_egress_iface_mappings()` is failing |
| `Tunnel ports` count | Equals (number of remote ACTIVE chassis) - 1 | Less than expected | 0 while remote chassis exist | BFD session is DOWN; check `bfd/show` |
| Coverage `physical_run` | Steadily increases 1 to 10 per minute | Increases more than 60 per minute | Stuck (does not increase) | Engine `physical_flow_output` recompute storm or deadlock |

**Failure scenarios:**

- `claimed: false` for one port while the OVS interface is up: another chassis still holds the claim. Temporary fix: `ovn-sbctl set Port_Binding <port> chassis=[]` to release manually. Investigate why the release did not happen automatically (did the CMS invoke `requested-chassis`?).
- `Tunnel ports` is 0 while there are 5 remote active chassis: BFD is DOWN for all of them. Underlay UDP 6081 is blocked, or the underlay MTU is too small.
- The `physical_run` coverage counter stays flat for several minutes: engine deadlock. Run `ovs-appctl -t ovn-controller engine/clear-flow-cache` to force a recompute, or restart ovn-controller (last-resort fix).

### Guided Exercise S65: trace Geneve TLVs on a 2-chassis sandbox

**Goal:** Verify that tunnel_key, inport, and outport are actually encoded into Geneve TLV class 0x0102, matching `controller/physical.c put_encapsulation()`.

**Assumed setup:** 2 chassis, chassis-1 and chassis-2, each with 1 VM (vm1, vm2) on the same Logical Switch `ls1`. The Geneve tunnel goes over physical port eth1.

**Steps:**

```
[chassis-1]# tcpdump -i eth1 -nn -X -w /tmp/geneve.pcap udp port 6081 &
[chassis-1]# ip netns exec ns_vm1 ping -c 1 10.0.0.2
[chassis-1]# tcpdump -r /tmp/geneve.pcap -nn -X | head -40
```

**Expected output (doc-plausible, pending lab verification):**

```
14:23:10.123 IP chassis1 > chassis2: GENEVE, vni 0x000001 ...
0x0040: ... 01 02 04 80 00 00 00 02 ...
                ^class^ ^type-len^ ^outport=2^
0x0048: ... 01 02 04 81 00 01 00 00 ...
                ^class^ ^type-len^ ^inport=1<<16^
```

Byte analysis:

- `01 02`: TLV class `0x0102` (reserved by OVN, IANA registry).
- `04`: type field (low byte = 4, OVN datapath context type).
- `80`: length 4 bytes plus flags.
- `00 00 00 02`: outport = 2 (the logical egress port for vm2).
- The second TLV: type `0x81`, value carries the inport at bits 16 to 30.

**Lessons learned:**

1. The Geneve TLV is the only place where logical metadata is carried across chassis. If the underlay drops Geneve UDP 6081, or a proxy strips the TLV, the packet arrives on the receiving chassis with metadata = 0 and falls back into the wrong pipeline.
2. TLV class `0x0102` is reserved for OVN. If any underlay middlebox reuses the same class (very rare), there will be a conflict.
3. `put_encapsulation()` only encodes the fields when the tunnel type is GENEVE. For VXLAN or STT, the encoding is different (the VXLAN VNI is 24 bits and only carries `tunnel_key`, with no inport or outport, so metadata is lost; not recommended for production).

**Cleanup:**

```
[chassis-1]# rm /tmp/geneve.pcap
```

> **Version note:** OVN 24.03 added the `--encap-csum` option for hardware Geneve checksums (NIC offload), reducing CPU encap cost. The 22.03 baseline used by the curriculum does not have this option.

## 13.7.9 Breadth backfill, ovn-controller as foundational topic

> Background, classification, version differences, a general lab, and a comparison with other systems for the foundational ovn-controller topic.

### History and background

ovn-controller was introduced in OVN 2.4 (2015) as part of the initial OVN architecture. Pfaff and Pettit's design replaced per-chassis ovs-vswitchd plus simple OF rule pushing with a declarative NB-driven control plane. Pfaff was motivated by the OpenStack Neutron ML2 pain (per-host agent plus race conditions). OVN 2.6 (2016) added chassis-redirect handling. OVN 2.10 (2018) added OVSDB IDL conditional monitoring to reduce SBDB load. OVN 22.06 (2022) refactored the I-P engine for parallel processing.

### Classification

Type: per-chassis daemon, control plane process, single-host scope. Parent: the OVN distributed control plane architecture (centralised NBDB plus the ovn-northd compiler plus per-chassis ovn-controller). Kind: long-running daemon, IDL connection to SBDB plus OF connection to the local ovs-vswitchd.

### Version differences

| Version | Behaviour |
|---------|-----------|
| OVN 2.4 (2015) | Initial introduction |
| OVN 2.6 | chassis-redirect handling |
| OVN 2.10 | IDL conditional monitoring |
| OVN 21.06 | Distributed gateway router architecture |
| OVN 22.03 baseline | I-P engine plus parallel build_lflows |
| OVN 22.06 | I-P engine refactor for parallel processing |
| OVN 24.03 | Zone-id externally configurable |

### General synthetic exercise

```
2-chassis Mininet OVN setup:
$ ovn-fake-multinode --create-fake-vms 2

On chassis-1:
$ ovs-appctl -t ovn-controller debug/dump-local-bindings
  Show LSP claimed by this chassis.

$ ovs-ofctl dump-flows br-int | head
  See the OF flows that ovn-controller installed from Logical_Flow.

Test recompute:
$ ovn-appctl -t ovn-controller recompute
  Force a full pipeline rebuild.

Test pause and resume:
$ ovn-appctl -t ovn-controller pause
  Stop processing SB updates.
$ ovn-appctl -t ovn-controller resume
```

### Comparison with other systems

- **OpenStack Neutron OVS-agent:** a per-host agent that pushes flow rules. Difference: the OVS-agent pushes imperatively; ovn-controller declaratively consumes Logical_Flow.
- **Cisco APIC fabric switch:** a centrally managed switch. Similar in control-plane abstraction. Difference: APIC is vendor-locked; OVN is open.
- **Cumulus FRR Zebra:** a routing daemon. Difference: FRR does routing only; ovn-controller covers full SDN data-plane control.

## 13.7.10 Catalog of ovn-appctl runtime commands for managing ovn-controller

> **Context for this section.** The five commands below are the main tools for operating a running `ovn-controller` daemon without restarting it: graceful shutdown, pausing the SBDB synchronisation to take a state snapshot, forcing a full pipeline recompute when the cache is suspect, and flushing the `lflow_cache` after large SBDB churn. Verified upstream: `controller/ovn-controller.c` `branch-22.03` (commit baseline tag `v22.03.8`), the `unixctl_command_register(...)` blocks at around L3376 (`exit`), L3731 (`recompute`), L3734 (`lflow-cache/flush`), and L3751 to L3755 (`debug/pause`, `debug/resume`, `debug/status`).

### 13.7.10.1 `ovn-appctl -t ovn-controller exit`

**Concept.** Ask the local `ovn-controller` process to terminate in an orderly way. Unlike `kill -TERM`, the daemon takes the unixctl exit path and is not interrupted by an OS-delivered TERM signal.

**History.** The `exit` command has been registered since the first generation of ovn-controller (OVN 2.4, 2015), through `unixctl_command_register("exit", "", 0, 1, ovn_exit_command_callback, ...)` at `controller/ovn-controller.c` around L3376. The behaviour has not changed across 22.03 to 24.03, although the optional 0..1 argument allows passing the `--restart` flag (keep flows on `br-int` for a hot-restart, OVN 21.06).

**Position in the architecture.** It is one unixctl command among several (`recompute`, `lflow-cache/flush`, `debug/pause`, and so on). All of them communicate through the Unix socket `/var/run/ovn/ovn-controller.<pid>.ctl` that the daemon opens at startup.

**Operational role.** It lets an operator or CMS stop the daemon without going through `systemctl stop ovn-controller-service` (which sends a kill) and without using a signal handler. Useful in orchestration scripts that need to make sure the daemon has committed every IDL transaction before the process exits.

**Syntax and example.**

```
[chassis]# ovn-appctl -t ovn-controller exit
[chassis]# echo $?
0
[chassis]# ps aux | grep ovn-controller   # process is gone
```

With `--restart` (keep flows for a hot-restart window in OVN 21.06):

```
[chassis]# ovn-appctl -t ovn-controller exit --restart
[chassis]# ovs-ofctl dump-flows br-int | wc -l   # flows are still there
```

**Version differences.** OVN 21.06 added the `--restart` flag for a hot-restart that does not drop traffic. OVN 22.03, the curriculum baseline, supports it fully. OVN 24.03 added a `Chassis_Private.nb_cfg_timestamp` cleanup before exit, so the cluster can detect the chassis going offline more quickly.

**Observation.** After the command, check that (a) the PID is gone from `ps`, (b) the socket `/var/run/ovn/ovn-controller.*.ctl` has been deleted, (c) the journal `journalctl -u ovn-controller` logs `unixctl: received command 'exit' from <pid>`, and (d) without `--restart`, the flows on `br-int` are cleaned according to `external_ids:ovn-ofctrl-wait-before-clear` (default 0 = keep).

**Source code.** `controller/ovn-controller.c::ovn_exit_command_callback()` in `branch-22.03`, sets `ovn_exit_args.exiting = true` so the main loop `main_loop()` exits on the next iteration. The daemon flushes IDL transactions, closes the OF connection to `br-int`, closes the SBDB connection, and calls `exit(0)`.

**Common mistakes.** If the daemon is hung (engine deadlock, IDL connection stuck), `exit` returns a 5-second timeout. Then use `kill -TERM <pid>`, and finally `kill -KILL`.

**Related.** `debug/pause` (only pause, no exit), `recompute` (recompute without exit), the systemd unit `ovn-controller.service` `ExecStop` (calls this same command).

### 13.7.10.2 `ovn-appctl -t ovn-controller debug/pause`

**Concept.** Pause the processing loop of `ovn-controller`. The daemon stops reading IDL notifications from SBDB and stops emitting OpenFlow `Flow_Mod` to `br-int`. The process stays alive and continues to accept other unixctl commands. This is a **debug aid**, not a production tool.

**Real name in the source.** `unixctl_command_register("debug/pause", ...)` at `controller/ovn-controller.c` around L3751 in `branch-22.03`. Several documents (including some community guides) write `pause` without the `debug/` prefix; that is **wrong** for the 22.03 baseline. The correct command is `debug/pause`. The behaviour and prefix are unchanged on `main` (verified around L7615 of `controller/ovn-controller.c`).

**Position in the architecture.** The pause flag is a static `bool paused` local in `controller/ovn-controller.c`, checked at the top of every `main_loop()` iteration. When `paused=true`, the daemon skips the entire engine graph and ofctrl emission, and only services unixctl requests.

**Operational role.** Two forensic use cases: (1) **Stable snapshot:** pause so that `ovs-ofctl dump-flows br-int` and `ovn-sbctl dump-flows` capture state at the same instant, without ovn-controller installing or removing rules between the two commands. (2) **Bug reproduction:** when tracing a "flow disappears after X seconds" bug, pause to freeze the state for log comparison.

**Syntax and example.**

```
[chassis]# ovn-appctl -t ovn-controller debug/pause
[chassis]# ovn-appctl -t ovn-controller debug/status
state: paused
[chassis]# ovs-ofctl dump-flows br-int > /tmp/flows.snap.txt
[chassis]# ovn-sbctl dump-flows > /tmp/lflows.snap.txt
[chassis]# ovn-appctl -t ovn-controller debug/resume
```

**Production warning.** While paused, every NB or SB change (CMS adding a pod, VM migration, ACL update) is **not applied** to this chassis. New pods receive no flows, and therefore no network. If the pause is too long, OCC fires (`Chassis_Private.nb_cfg` does not advance), and the CMS may consider the chassis stuck and trigger fencing or evacuation. Rule of thumb: pause for under 30 seconds on production, under 5 minutes on staging.

**Observation.** `ovn-appctl -t ovn-controller debug/status` returns `state: paused` or `state: running`. While paused, the `engine_run` coverage counter stops increasing over time. This is the precise signal to verify the pause.

**Source code.** Function `debug_pause_execution()` in `controller/ovn-controller.c` `branch-22.03` sets `*(bool *)aux = true`. The `paused` flag is polled at the top of `main_loop()` in the same file.

**Related.** `debug/resume` (the inverse), `debug/status` (state check), `recompute` (does **not** replace pause, because recompute lets the engine resume normal operation after recomputing).

### 13.7.10.3 `ovn-appctl -t ovn-controller debug/resume`

**Concept.** Cancel the pause state set by `debug/pause`. The engine and ofctrl return to their normal loop on the next `main_loop()` iteration.

**Real name in the source.** `unixctl_command_register("debug/resume", ...)` around L3753 in `branch-22.03 controller/ovn-controller.c`. It also has the `debug/` prefix; not a bare `resume`.

**Position in the architecture.** Symmetric to `debug/pause`. It only flips the same `bool paused` flag back to `false`. On the next iteration, `main_loop()` sees `paused=false` and runs normally.

**Operational role.** Mandatory after every `debug/pause`. It is the second half of the pause and resume pair. Forgetting to resume means the chassis is stuck and the CMS treats it as dead.

**Syntax and example.**

```
[chassis]# ovn-appctl -t ovn-controller debug/resume
[chassis]# ovn-appctl -t ovn-controller debug/status
state: running
[chassis]# ovs-appctl -t ovn-controller coverage/show | grep engine_run
engine_run             123  4567/sec
```

**Recovery characteristics.** When you resume after a long pause, the daemon must process the SBDB notification backlog accumulated during the pause window. The first iteration after resume is typically an **engine_recompute fallback**, because the I-P engine cannot diff across the pause window. Expect a CPU spike of 2 to 5 seconds on a cluster of 100+ chassis. Normal steady state returns after 1 to 2 iterations.

**Observation.** `debug/status` returns `state: running`. The `engine_recompute` counter typically increments by exactly 1 right after resume. `journalctl -u ovn-controller` shows `engine: recompute due to internal state` at the matching line.

**Source code.** Function `debug_resume_execution()` in the same file `controller/ovn-controller.c` `branch-22.03` sets `*(bool *)aux = false`. Note: there is no "resume from checkpoint" concept; this is just a flag flip.

**Common mistakes.** (a) Forgetting to call resume, leaving the chassis stuck. Restarting `ovn-controller` is then the only way out if the original terminal session is lost. (b) Calling resume on a daemon that is not paused returns OK without error (idempotent), but does nothing.

**Related.** `debug/pause`, `debug/status`, `recompute` (operators who forget to resume often try recompute as a "reset", but it is not equivalent; recompute does not clear the paused flag).

### 13.7.10.4 `ovn-appctl -t ovn-controller recompute`

**Concept.** Ask the daemon to **bypass the I-P (incremental processing) engine cache** and rebuild the full OpenFlow set from SBDB ground truth on the next `main_loop()` iteration. This is the "panic button" when the engineer suspects the I-P engine has gone wrong (state drift, stale cache).

**`[deprecated]` warning.** In `branch-22.03` source, the registration reads `unixctl_command_register("recompute", "[deprecated]", 0, 0, engine_recompute_cmd, NULL)` at `controller/ovn-controller.c` around L3731. The command still works fully in 22.03 and in the curriculum baseline. The `[deprecated]` marker signals that upstream intends to rename it in future (later OVN versions ship an equivalent under the `inc-engine/*` family). Operators can still use it, but should prefer `lflow-cache/flush` for a similar purpose when that is sufficient.

**Position in the architecture.** It sets the `engine_run_id_required` flag (or its equivalent across versions), telling the I-P graph to mark every engine node `outdated=true`, which forces a full recompute instead of an incremental pass.

**Operational role.** Two use cases: (1) **Bug workaround:** when the flows on `br-int` do not match the Logical_Flow set in SB DB (drift), recompute usually fixes it within one iteration. (2) **Verify ground truth:** after a CMS bulk update, recompute to be sure nothing was missed.

**Syntax and example.**

```
[chassis]# ovn-appctl -t ovn-controller recompute
[chassis]# sudo ovs-appctl -t ovn-controller coverage/show | grep engine_recompute
engine_recompute  42  0.5/sec
[chassis]# # Counter incremented by 1, meaning the daemon performed a full rebuild
```

**Performance cost.** On a 1000-chassis cluster with 50K Logical_Flow entries, one `recompute` consumes 5 to 30 seconds of CPU at 100 percent on a single core on that chassis. During this window, incoming SBDB updates queue up (no drop, the IDL buffer holds them). Do not run periodically on production.

**Version differences.** OVN 22.03 marks the command deprecated but still ships it. OVN 24.03 keeps `recompute` as an alias for the new `inc-engine/recompute`. The behaviour is identical, only the name changed. Practical advice: production scripts should handle both names.

**Observation.** The `engine_recompute` counter increments by exactly 1. `engine_run_changed` increments by 1. The flow count on `br-int` either stays the same or shifts to match the SBDB ground truth. `journalctl -u ovn-controller` typically shows `engine: full recompute requested via unixctl`.

**Source code.** `engine_recompute_cmd()` in `controller/ovn-controller.c` `branch-22.03` calls `engine_set_force_recompute(true)`. The engine framework in `lib/inc-proc-eng.c::engine_run()` reads this flag at the top of every iteration.

**Common mistakes.** (a) Recompute does not fix the issue, so the real bug is in `physical.c` or `lflow.c` translation; escalate upstream. (b) Recompute on a very large cluster causes an OF install timeout and Flow_Mod backpressure; reduce the cluster size, or increase `external_ids:ovn-openflow-probe-interval`.

**Related.** `lflow-cache/flush` (cache flush only, not a full rebuild), `debug/pause` (does not replace recompute), `inc-engine/show-stats` (measure recompute frequency on 24.03).

### 13.7.10.5 `ovn-appctl -t ovn-controller lflow-cache/flush`

**Concept.** Flush the contents of the `lflow_cache` (the cache of Logical_Flow to OpenFlow translation results) inside daemon memory. On the next iteration, the daemon must re-translate Logical_Flow from scratch. It does **not** trigger a full engine-graph recompute.

**Position in the architecture.** `lflow_cache` is a hash table that stores the translation output of each Logical_Flow row, so unchanged flows do not have to be retranslated. The cache hit rate is around 95 to 99 percent in steady state. Flushing the cache forces the `en_lflow_output` engine node to redo translation for every active flow, while `en_runtime_data` and `en_physical_flow_output` still run incrementally.

**Operational role.** Main use case: **after a large SBDB bulk change** (a CMS rebuilding every ACL, adding 1000 Logical_Switch entries in one minute), when stale cache entries are suspected. Also useful during a hot-upgrade of `ovn-controller` without a restart, to make sure the cache is not holding entries in the old format.

**Syntax and example.**

```
[chassis]# ovn-appctl -t ovn-controller lflow-cache/show-stats
Enabled: true
high-watermark   : 8192
total            : 4521
cache-conj-id    : 1203
cache-expr       : 3318
[chassis]# ovn-appctl -t ovn-controller lflow-cache/flush
[chassis]# ovn-appctl -t ovn-controller lflow-cache/show-stats
total            : 0
```

**Difference from `recompute`.** `recompute` rebuilds the **entire** engine graph (runtime_data plus physical plus lflow plus flow_output) on the chassis. `lflow-cache/flush` only flushes the lflow cache, so it is 5 to 10 times cheaper on a large cluster. When you suspect stale lflow translation (Logical_Flow drift), try `lflow-cache/flush` first; only escalate to `recompute` if the issue persists.

**Version differences.** OVN 21.03 introduced `lflow-cache/flush` (split out from the older `flush-lflow-cache`). `branch-22.03` keeps the alias `flush-lflow-cache` with a `[deprecated]` marker (verified `controller/ovn-controller.c` around L3738). OVN 24.03 removes the alias. Prefer the `lflow-cache/` prefixed name.

**Observation.** `lflow-cache/show-stats` shows `total` going to 0 right after the flush. On the next iterations the cache repopulates back to steady state as the daemon re-translates the active flows. CPU spikes 500ms to 2s on a 10K Logical_Flow cluster.

**Source code.** `lflow_cache_flush_cmd()` in `controller/ovn-controller.c` `branch-22.03` calls `lflow_cache_flush(pd->lflow_cache)` (file `controller/lflow-cache.c`).

**Common mistakes.** (a) Flushing too often reduces the cache hit rate and increases lflow translation CPU chronically, degrading performance. (b) Flushing does not fix the bug, so the bug is in the translation logic (lflow.c), not in the cache. Use `recompute` next, then escalate.

**Related.** `lflow-cache/show-stats` (measure cache size), `recompute` (stronger), `debug/pause` (unrelated, only pauses), and the configuration parameter `Open_vSwitch.external_ids:ovn-limit-lflow-cache-mem` that caps the cache memory.

---

> **Catalog summary.** The five commands above are the minimum kit an operator must know in order to keep an OVN chassis healthy. Order to try when an incident hits: start with `debug/status` to know what state the daemon is in, then `lflow-cache/show-stats` and `coverage/show` to measure, then `lflow-cache/flush` if the cache is suspect, then `recompute` if cache flush is not enough, then `debug/pause` to capture a forensic snapshot if needed, and finally `exit` (optionally with `--restart` on 21.06) to stop gracefully when changing version.

## 13.7.11 Five extra runtime commands for ovn-controller

> **Context for this section.** Beyond the five "core" commands in §13.7.10, the `ovn-controller` daemon registers a series of unixctl commands for chassis-local inspection: SBDB connection status, listing the meter table, listing CT zones, listing the group table, and resetting the cluster index. The five below are the "read and reset" group, used by operators in forensic work without affecting traffic. Verified upstream: `controller/ovn-controller.c` `branch-22.03`, the `unixctl_command_register(...)` blocks at L3405 (`connection-status`), L3719 (`meter-table-list`), L3723 (`ct-zone-list`), L3715 (`group-table-list`), and L3746 (`sb-cluster-state-reset`).

> **Important note on `set-n-threads` and `get-n-threads`.** These two names are NOT commands of `ovn-controller`. Verified `controller/ovn-controller.c` on both `branch-22.03` (4618 lines) and `main` (the L7367 to L7650 registration block): there is no `unixctl_command_register("set-n-threads", ...)` and no `get-n-threads`. The pair belongs to `ovn-northd` under a different prefix: `unixctl_command_register("parallel-build/set-n-threads", "N_THREADS", 1, 1, ...)` at `northd/ovn-northd.c` `main` around L865, and `parallel-build/get-n-threads` around L868. This is a common confusion on OVN community forums. An operator who runs `ovn-appctl -t ovn-controller set-n-threads 4` will receive `unknown command`. The correct invocation is `ovn-appctl -t ovn-northd parallel-build/set-n-threads 4`.

### 13.7.11.1 `ovn-appctl -t ovn-controller connection-status`

**Concept.** Show the chassis SBDB connection status. The output tells you whether the daemon is `connected` or `not connected` to SB. This command is parallel to `debug/status` (running or paused), but it focuses on the SBDB IDL channel rather than the engine loop.

**Real name in the source.** `unixctl_command_register("connection-status", "", 0, 0, ovn_controller_conn_show, ...)` at `controller/ovn-controller.c` `branch-22.03` around L3405. The name has NO prefix; type `connection-status` directly. Some older articles use the name `status`, which is incorrect for the 22.03 baseline (a `status` command exists only in `ovn-northd` around L854).

**Operational role.** Two main use cases. (a) **Quick triage:** when a new pod has no flows, check `connection-status` first. If the daemon has lost SBDB connectivity, every other layer (engine, ofctrl) is moot. (b) **Fault domain split:** it lets you separate "is the daemon doing anything?" (debug/status) from "is the daemon still talking to SBDB?" (connection-status). The two states are independent; the daemon can be running normally while disconnected from SBDB, and vice versa.

**Syntax and example.**

```
[chassis]# ovn-appctl -t ovn-controller connection-status
connected
```

When connectivity is lost (for example a firewall blocking SB port 6642), the output switches to `not connected`. Unlike `ovs-vsctl get Open_vSwitch . external_ids:ovn-remote-probe-interval` (which reads static configuration), `connection-status` reads state directly from the live IDL session, so it is real-time evidence.

**Cross-checks for observation.** When `connection-status` returns `not connected`: cross-check with `ss -tnp | grep 6642` to see whether there is a TCP session to SBDB, and `journalctl -u ovn-controller --since "5 min ago" | grep -i "connect"` for the reason. The counter `coverage/show | grep nb_cfg_seqno` not advancing is also a sign that the chassis has been split off from the cluster.

**Source code.** Handler `ovn_controller_conn_show()` in the same `controller/ovn-controller.c` reads `struct ovsdb_idl *ovnsb_idl_loop.idl` and calls `ovsdb_idl_is_connected()` to return "connected" or "not connected". No side effects; it is safe to invoke repeatedly.

**Common mistakes.** (a) Operators sometimes assume that `connection-status` reports overall SBDB cluster health. It does not: this command only reflects the connection from the current chassis to the SB endpoint it is using. It says nothing about the health of the other SB Raft members. (b) A `connected` result right after `sb-cluster-state-reset` does not mean the cluster index is in sync, only that the TCP or SSL session has been re-established.

### 13.7.11.2 `ovn-appctl -t ovn-controller meter-table-list`

**Concept.** List every entry of the meter table that ovn-controller currently manages on `br-int`. Each output row maps a logical meter name (generated by `ovn-northd` from `NB.Meter`) to the actual `meter_id` installed through OpenFlow. This is a read-only command; it does not modify state.

**Real name in the source.** `unixctl_command_register("meter-table-list", "", 0, 0, extend_table_list, &meter_table)` at `controller/ovn-controller.c` `branch-22.03` around L3719. The same `extend_table_list` handler is shared with `group-table-list` (L3715); the two are distinguished by the `aux` parameter (`&meter_table` versus `&group_table`).

**Operational role.** Main forensic use case: when an ACL rate-limit is not working (packets are not dropped despite exceeding the configured rate), list the meter table to verify that the OF meter has been installed and that the expected `meter_id` exists. If the ID is present, the issue is in the action that attaches the meter. If the ID is missing, the issue is in the northd to controller pipeline (lflow has not generated the `meter:N` action).

**Syntax and example.**

```
[chassis]# ovn-appctl -t ovn-controller meter-table-list
__acl_log_rate_limit_my-app: 1
__acl_meter_my-vpc_acl_42: 2
```

On a chassis without any rate-limit ACL, the output is empty (only exit code 0). Cross-check with `ovs-ofctl -O OpenFlow15 dump-meters br-int` to see the actual bytecode of the meter (rate, burst, band type).

**Cross-checks for observation.** Each `name: id` entry corresponds to a band in `ovs-ofctl dump-meters br-int meter=N`. If `meter-table-list` shows `meter_id=2` but `dump-meters` reports that meter id 2 does not exist, there is a drift between the controller view and the datapath, a sign that meter installation failed. Then check the log with `journalctl -u ovn-controller | grep -i meter`.

**Source code.** Handler `extend_table_list()` in `controller/ovn-controller.c` walks `struct extend_table::desired` and prints each `extend_table_info` through `unixctl_command_reply()`. The `meter_table` is filled by `lflow.c::add_meter()` from the Logical_Flow `meter:N` action.

**Common mistakes.** (a) Stale entries listed after deleting an ACL are normal; the controller cleans up lazily on the next iteration. (b) The meter name prefix `__acl_log_rate_limit_` only appears when the NB Meter has `fair=true` and `unit=pktps`. If you do not see it, the NB definition is incorrect.

### 13.7.11.3 `ovn-appctl -t ovn-controller ct-zone-list`

**Concept.** List every local Logical_Port and the `ct_zone` (conntrack zone ID) that ovn-controller has assigned to each port. Every valid VM or container on the chassis gets its own zone, so conntrack stays separated across tenants and across datapath entries.

**Real name in the source.** `unixctl_command_register("ct-zone-list", "", 0, 0, ct_zone_list, ...)` at `controller/ovn-controller.c` `branch-22.03` around L3723. The name has no `debug/` prefix. The forward declaration `ct_zone_list` is in the same file.

**Operational role.** Main use case for diagnosing stateful rules: when an `allow-related` ACL appears to forget state, run `ct-zone-list` to find which zone the logical port belongs to, then `conntrack -L -z <zone_id>` to see the actual conntrack entries. This is the only bridge between the OVN "logical port" abstraction and the kernel "zone" abstraction.

**Syntax and example.**

```
[chassis]# ovn-appctl -t ovn-controller ct-zone-list
ls1-vm1: 7
ls1-vm2: 8
lr1-dnat: 9
```

The text before the colon is the logical port name (or a special tag such as `lr1-dnat` for the DNAT zone). The number after is the zone ID passed to the action `ct(zone=N, commit, ...)` that the controller emits in OF flows on `br-int`.

**Cross-checks for observation.** `conntrack -L -z 7` (requires the `nf_conntrack` module to be loaded) shows the conntrack entries for zone 7. If no entries appear despite flows being installed, perhaps `nf_conntrack` is not loaded, or the OVS datapath does not have conntrack support enabled. Cross-check with `ovs-appctl dpctl/dump-conntrack zone=7`.

**Source code.** Handler `ct_zone_list()` in `controller/ovn-controller.c` walks `struct simap *ct_zones` (the zone allocator in `controller/ovn-controller.c::ct_zones_init()`). The zone allocator only assigns values 1 to 65535 (avoiding 0); a zone is freed when the port unbinds.

**Common mistakes.** (a) Empty output despite running pods means that the chassis has not yet received any Port_Binding (the claim has not succeeded). Check `debug/dump-local-bindings`. (b) Entries exist but `conntrack -L -z N` is empty means that the flow has not yet executed a `ct(commit)` action; no packet has gone through a stateful rule.

### 13.7.11.4 `ovn-appctl -t ovn-controller group-table-list`

**Concept.** List every entry of the group table that ovn-controller currently manages on `br-int`. The OpenFlow group table is used for multicast (BUM traffic), ECMP load-balancing, and select groups for `Load_Balancer` health checks. Each group has an actual `group_id` installed on the datapath.

**Real name in the source.** `unixctl_command_register("group-table-list", "", 0, 0, extend_table_list, &group_table)` at `controller/ovn-controller.c` `branch-22.03` around L3715. The same `extend_table_list` handler is shared with `meter-table-list` (L3719); the difference is the `aux` parameter pointing to the group table instead of the meter table.

**Operational role.** Main forensic use case: when `Load_Balancer` does not balance evenly across backends, or when multicast (ARP, IGMP) does not reach the destination, run `group-table-list` to see which group IDs exist, then `ovs-ofctl -O OpenFlow15 dump-groups br-int group=N` to see the buckets of the group (members, weight, action). The command is especially valuable when `dump-groups` output is long: `group-table-list` helps narrow the IDs of interest.

**Syntax and example.**

```
[chassis]# ovn-appctl -t ovn-controller group-table-list
__mc_flood_my-ls: 1
__mc_unknown_my-ls: 2
__lb_select_my-vip: 3
```

The prefix `__mc_flood_` is for BUM-flood per Logical_Switch, `__mc_unknown_` is for unknown destinations, and `__lb_select_` is for ECMP and load-balance buckets. The suffix is the related Logical_Switch or Logical_Router_Port name.

**Cross-checks for observation.** `ovs-ofctl -O OpenFlow15 dump-groups br-int` produces full bucket output. `ovs-ofctl -O OpenFlow15 dump-group-stats br-int` shows the `byte_count` and `packet_count` counters per bucket, a direct measure of load distribution. A mismatch between `group-table-list` (controller view) and `dump-groups` (datapath view) means group installation failed.

**Source code.** Handler `extend_table_list()` shared with `meter-table-list`. The `group_table` is filled by `lflow.c::add_group()` from the Logical_Flow actions `group:N` or `multipath()`.

**Common mistakes.** (a) Stale group ID after deleting a `Load_Balancer`, because the controller GC is lazy. (b) Group bucket count = 0 means every backend health check is failing, and the packet is dropped; check the `Service_Monitor` table in SBDB. (c) Group ID conflict between OVS (group ID range 0 to 4294967040) and OVN (its own allocator) cannot happen on the same `br-int`, because only the controller installs groups there.

### 13.7.11.5 `ovn-appctl -t ovn-controller sb-cluster-state-reset`

**Concept.** Force the daemon to discard its cached cluster index (the SB Raft member sequence numbers) and re-register with the SBDB cluster from scratch. It is a **panic button** for situations where the SB cluster has been destroyed and rebuilt (for example disaster recovery), and the local controller still holds an old index that does not match the new cluster.

**Real name in the source.** `unixctl_command_register("sb-cluster-state-reset", "", 0, 0, cluster_state_reset_cmd, ...)` at `controller/ovn-controller.c` `branch-22.03` around L3746. The name is symmetric with `nb-cluster-state-reset` in `ovn-northd`; controller only has the SB side because it does not talk to NB directly.

**Operational role.** Single but critical use case: after `ovsdb-tool create sb.db` has rebuilt the SB cluster (losing the old Raft log), every ovn-controller must reset its local cluster state, otherwise it stays stuck on the old index. The IDL receives the new snapshot but does not trust it, leading to a long drift. This command is **not** to be used in steady-state production. Used incorrectly, it triggers a full SB DB redownload, a CPU spike, and potential flow drops on the chassis.

**Syntax and example.**

```
[chassis]# ovn-appctl -t ovn-controller sb-cluster-state-reset
[chassis]# ovn-appctl -t ovn-controller connection-status
not connected
[chassis]# # After about 2 to 5 seconds, the daemon reconnects on its own
[chassis]# ovn-appctl -t ovn-controller connection-status
connected
```

Standard observation chain: right after the command, `connection-status` briefly shows `not connected` because the daemon closes the old IDL session, then reconnects within seconds when the cluster is reachable.

**Cross-checks for observation.** `journalctl -u ovn-controller --since "1 min ago"` logs `cluster state reset by user` and `ovsdb_cs: connecting to ...`. The counter `coverage/show | grep cluster_state_reset` increments by exactly 1. If the daemon cannot reconnect (cluster is down), `ss -tnp | grep 6642` shows no session.

**Source code.** Handler `cluster_state_reset_cmd()` in `controller/ovn-controller.c` calls `ovsdb_cs_set_leader_only(false)` and then `ovsdb_cs_force_reconnect()` on the SB IDL. Side effect: it triggers a full IDL re-sync, not just a TCP reconnect.

**Common mistakes.** (a) Calling it on a healthy cluster wastes CPU. Flows stay stable but latency rises by 5 to 30s. (b) Forgetting to call it after a cluster rebuild leaves the controller stuck on the old index. New Logical_Flow entries are not applied, and the controller looks dead even though `debug/status` returns `running`. (c) Calling it simultaneously on 100+ chassis floods the SBDB cluster with reconnects, possibly overloading the Raft leader. Roll out at 10 to 20 chassis per minute.

---

> **Section 13.7.11 summary.** The five commands in this section are non-disruptive read-and-reset commands (with the exception of `sb-cluster-state-reset`, which requires care). Standard forensic workflow: start with `connection-status` to confirm the daemon is still talking to SBDB, use `ct-zone-list` to map a logical port to a kernel zone, cross-check `meter-table-list` and `group-table-list` against `ovs-ofctl dump-meters` and `dump-groups`, and only use `sb-cluster-state-reset` if the cluster has been destroyed and rebuilt. Combined with the five core commands in §13.7.10, an operator has 10 runtime commands that cover roughly 90 percent of chassis-local scenarios without restarting the daemon.

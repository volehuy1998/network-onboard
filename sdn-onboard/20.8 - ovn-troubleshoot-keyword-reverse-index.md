# 20.8. OVN troubleshoot keyword reverse index + 14 production scenarios

> **Status:** expansion 2026-04, 14 production scenarios with reference cross-links.

## Learning objectives

After this part, an on-call engineer can:

1. See a symptom in production and reverse-lookup the relevant keyword.
2. For each critical keyword, list three to five typical incident patterns.
3. Apply a diagnostic flow per symptom-keyword pair.
4. Walk through the full treatment of all 14 REF Section 4 scenarios.

## §20.8.2 14 Production scenarios from REF Section 4

### Scenario 1: East-west traffic between two VMs same OVN logical switch fails

**Symptom:** VM-A on LS-tenant cannot reach VM-B on the same LS, both VMs alive, ARP works.

**Diagnostic flow:**
1. `ovn-trace ls-tenant 'inport == "lsp-vm-a" && eth.dst == <vm-b-mac>'`
2. Identify drop stage (PORT_SEC, ACL, L2_LKUP, etc.)
3. Check Port_Binding chassis claim: `ovn-sbctl list Port_Binding lsp-vm-b`
4. Check MAC_Binding: `ovn-sbctl find MAC_Binding ip=<vm-b-ip>`

**Likely root causes:** REGBIT_PORT_SEC_DROP set, ACL drop, MAC_Binding stale, Port_Binding chassis claim race.

**Cross-link primary:** 13.16 §LS_IN_L2_LKUP, 13.18 §REGBIT_PORT_SEC_DROP.

### Scenario 2: North-south egress through OVN distributed gateway port (DGP) blackholes

**Symptom:** VM internal cannot reach external IP; ICMP timeout.

**Diagnostic:**
1. `ovn-trace lr-main 'inport == "lrp-internal" && ip.dst == 8.8.8.8'`
2. Check REGBIT_NAT_REDIRECT set + LR_IN_GW_REDIRECT outport
3. Check chassis-redirect port claim: `ovn-sbctl find Port_Binding type=chassisredirect`
4. Check HA_Chassis_Group active member

**Likely root causes:** redirect-chassis dead, BFD timeout, HA_Chassis priority misconfig, NAT rule missing.

**Cross-link:** 13.11 §distributed gateway, 13.18 §REGBIT_NAT_REDIRECT.

### Scenario 3: OVN load balancer VIP drops new connections after a threshold

**Symptom:** LB VIP serves N concurrent connections fine, then new connections SYN dropped after threshold.

**Diagnostic:**
1. Check `nf_conntrack_count` vs `nf_conntrack_max` ratio
2. Check ct_zone: `ovs-appctl dpctl/dump-conntrack zone=N | wc -l`
3. Check Service_Monitor health: `ovn-sbctl list Service_Monitor`

**Likely root causes:** Conntrack table near max, hairpin_snat_ip misconfig, Service_Monitor reports backend down.

**Cross-link:** 13.9 §LB internals, 9.24 §conntrack zone.

### Scenario 4: ovn-controller stuck in re-compute / high CPU

**Symptom:** Single chassis ovn-controller CPU sustained > 80%; OF flow update lag.

**Diagnostic:**
1. `ovn-appctl -t ovn-controller inc-engine/show-stats`
2. Check recompute count vs partial compute ratio
3. Check SBDB monitor lag

**Likely root causes:** I-P engine fall back to full recompute (rare bug), SBDB schema delta heavy, IDL conditional monitoring miss.

**Cross-link:** 13.7 §ovn-controller internals, 20.2 §recompute debug.

### Scenario 5: SB DB connection flaps between ovn-controller and ovsdb-server cluster

**Symptom:** Log "ovsdb_idl: connection lost" repeated; ovn-controller intermittent.

**Diagnostic:**
1. Check `inactivity_probe` value too low for cluster RTT
2. Check `ovs-appctl -t ovsdb-server cluster/status OVN_Southbound`
3. Check network MTU + UDP/TCP loss

**Likely root causes:** inactivity_probe too aggressive, cluster split-brain, network instability.

**Cross-link:** 10.1 §Raft, 13.7 §ovn-controller IDL.

### Scenario 6: Geneve tunnel down between two chassis (br-int → physical NIC)

**Symptom:** Inter-chassis traffic drop; intra-chassis traffic OK.

**Diagnostic:**
1. `ovs-appctl tnl/ports/show` check tunnel ports state
2. `ovs-appctl bfd/show` if BFD configured
3. tcpdump UDP port 6081 on the physical interface
4. Check Encap row: `ovn-sbctl list Encap`

**Likely root causes:** Geneve UDP port 6081 firewall block, MTU mismatch, Encap config invalid IP.

**Cross-link:** 11.0 §Geneve TLV, 11.1 §MTU PMTUD.

### Scenario 7: Conntrack zone exhaustion / "ct_zone in use" errors

**Symptom:** New connection stateful drop; log "ct_zone limit reached".

**Diagnostic:**
1. `cat /proc/sys/net/netfilter/nf_conntrack_max`
2. `cat /proc/sys/net/netfilter/nf_conntrack_count`
3. Check zone partition via `ovs-appctl dpctl/dump-conntrack | awk` per-zone count

**Likely root causes:** Nf_conntrack_max default 65536 too low, zone collision multi-tenant, long-lived TCP timeout default 5 days.

**Cross-link:** 9.24 §ct_zone, 13.18 §ct_zone collision case.

### Scenario 8: ovs-vswitchd 100% CPU with revalidator threads dominating

**Symptom:** Single host ovs-vswitchd CPU 100%, mostly revalidator thread time.

**Diagnostic:**
1. `ovs-appctl coverage/show | grep revalidator`
2. `ovs-appctl upcall/show`
3. Check megaflow count + churn rate

**Likely root causes:** Megaflow churn high (cited 9.26), revalidator can't keep up, mask explosion.

**Cross-link:** 9.26 §revalidator storm forensic.

### Scenario 9: OpenFlow bundle commit fails / partial flow install during ovn-controller restart

**Symptom:** After an ovn-controller restart, OF flows on br-int are incomplete; some rules are missing.

**Diagnostic:**
1. `ovs-vsctl get Open_vSwitch . other_config:flow-restore-wait`
2. Check ovn-controller log "Bundle commit error"
3. `ovs-ofctl dump-flows br-int | wc -l` count

**Likely root causes:** flow-restore-wait misconfig, bundle commit timeout, classifier rebuild storm.

**Cross-link:** 3.5 §OFPT_BUNDLE family, 13.7 §flow-restore-wait.

### Scenario 10: Asymmetric routing in OVN ECMP (return path takes the wrong chassis)

**Symptom:** TCP connection one-way pass; reply through different gateway chassis dropped.

**Diagnostic:**
1. ovn-trace forward path + reverse path observe outport
2. Check ECMP hash field consistent
3. Check stateful conntrack zone consistent

**Likely root causes:** ECMP hash inconsistent forward vs reverse, conntrack zone different per chassis.

**Cross-link:** 13.19 §LR_IN_IP_ROUTING_ECMP, 9.24 §ct_zone consistency.

### Scenario 11: Stale or missing MAC_Binding causes routing black-holes

**Symptom:** First packet to new dst IP succeeds (ARP resolve), subsequent packet drops post LRP MAC change.

**Diagnostic:**
1. `ovn-sbctl find MAC_Binding ip=<dst-ip>`
2. Check MAC_Binding timestamp + LRP MAC
3. ovn-trace observe LR_IN_ARP_RESOLVE stage

**Likely root causes:** MAC_Binding stale, LRP MAC changed, OVN 22.03 MAX_FDB_ENTRIES limit hit.

**Cross-link:** 13.11 §MAC_Binding, 19.0 §PMTUD post-migration case.

### Scenario 12: ovn-northd standby does not take over after active failure

**Symptom:** Active ovn-northd dies; standby fails to take over; SBDB stale.

**Diagnostic:**
1. `ovn-appctl -t ovn-northd status` on each candidate
2. Check SBDB lock holder: `ovn-sbctl get SB_Global . options:`
3. Check standby ovn-northd log for "Could not acquire lock"

**Likely root causes:** Lock release timeout too long, network partition between standby + SBDB, standby ovn-northd version mismatch.

**Cross-link:** 13.8 §ovn-northd cluster, 10.1 §Raft lock.

### Scenario 13: VM DNS queries dropped by OVN native DNS

**Symptom:** VM DNS resolution fail intermittent; A query timeout.

**Diagnostic:**
1. Check DNS row: `ovn-sbctl list DNS`
2. ovn-trace UDP 53 packet observe LS_OUT_DNS_LOOKUP stage
3. Check REGBIT_DNS_OPTS_RESULT bit transition

**Likely root causes:** DNS row records misconfig, ACL block UDP 53, REGBIT_DNS bit not set.

**Cross-link:** 13.10 §native DNS, 13.18 §REGBIT_DNS_OPTS_RESULT.

### Scenario 14: Geneve/VXLAN MTU fragmentation causing intermittent large-packet drops

**Symptom:** Small packet OK; large packet (> ~1450 bytes) drop or fragmenter.

**Diagnostic:**
1. Check physical MTU vs Geneve overhead (50 bytes min)
2. tcpdump fragment offset on physical interface
3. Check OVN check_pkt_larger action invocation

**Likely root causes:** Physical MTU too small (default 1500 - 50 Geneve overhead = 1450 effective), TCP MSS clamp missing, PMTUD black hole (cited 19.0 case).

**Cross-link:** 11.1 §MTU PMTUD, 19.0 §PMTUD post-migration forensic.

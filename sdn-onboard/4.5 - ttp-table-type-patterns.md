## Bối cảnh: silicon fragmentation và hệ quả

Part 4.4 đã phân tích OF 1.5 với egress tables và packet type aware, nhưng adoption hardware gần như zero. Một câu hỏi tự nhiên: tại sao vendor không implement? Phần trước attribute cho economics (revenue không justify cost) và timing (P4 thay thế). Phần này đi sâu hơn vào root cause cấu trúc: **OpenFlow spec được thiết kế theo assumption "silicon là generic" nhưng thực tế silicon rất fragmented**.

OF spec 1.3 mô tả đầy đủ abstract pipeline: N match tables, M instructions, K group types, meter, PBB fields, packet type. Nhưng mỗi ASIC vendor build silicon theo constraints kinh tế và physical: TCAM size limit, parser depth limit, pipeline stage count, memory hierarchy. Không ASIC nào implement toàn bộ spec; mỗi chip support subset khác nhau của spec. Hệ quả: controller viết flow_mod "theoretically valid" nhưng switch reject với OFPERR_BAD_MATCH ở runtime.

TTP (Table Type Patterns) là attempt cuối cùng của ONF để giải bài toán này qua pattern matching: thay vì controller query per-feature, switch advertise pre-defined pattern và controller adapt.

## 4.5.1 Vấn đề: mỗi vendor silicon support khác subset của OF spec

### Silicon constraints cụ thể

Lấy 3 chipset phổ biến 2013-2017 làm ví dụ:

**Broadcom Trident2 (BCM56850)**: flagship DC switch silicon 2013-2015. 
- Ingress ACL TCAM: 4K-8K entries (tùy SKU).
- Egress ACL TCAM: 256 entries.
- L2 table: 128K MAC entries.
- L3 host table: 16K IPv4 exact-match.
- LPM routing: 16K IPv4 prefix.
- Không có **egress ACL** rộng: hạn chế policy per-output-port mà OF 1.5 egress tables cần.

**Intel FM6000**: alternative chipset 2013-2016.
- 16-stage programmable pipeline (ngang hiện đại lúc đó).
- Mỗi stage có match field constraint khác nhau: stage 1 chỉ match L2; stage 2 chỉ match L3; stage 3+ có thể match mixed.
- Complex capability model: flow_mod phải align với stage ownership, không thể arbitrary.

**Cavium Xpliant CNX**: semi-programmable silicon pre-P4 era (2015-2016).
- Programmable parser (limited depth 96 bytes).
- Flexible match-action stages (6 stages với shared TCAM).
- Nhưng vẫn có constraints: action per stage limited, MPLS push/pop chỉ ở specific stage.

Mỗi ASIC có "capability profile" riêng, không tương thích plug-and-play.

### Hệ quả vận hành

**Controller không biết** switch có hỗ trợ combination match/action nào. Phương pháp discovery của OF 1.0-1.4:

1. Switch connect → HELLO handshake → negotiate version.
2. Controller query OFPT_FEATURES_REQUEST → switch reply với N_TABLES, port count, actions supported.
3. Controller install flow_mod theo what it thinks switch supports.
4. Nếu flow_mod invalid cho switch silicon → OFPERR_BAD_MATCH / OFPERR_BAD_INSTRUCTION / OFPERR_BAD_ACTION at runtime.

Bug chỉ lộ ở **runtime**, không ở compile/validate time. Controller developer phải maintain per-vendor capability table trong application logic. Không portable.

**Production impact**: khi upgrade switch model (từ Trident2 sang Trident3) hoặc add mixed-vendor fabric, controller logic phải rewrite. Operations pain point lớn, operator stuck với single-vendor lock-in thay vì enjoy "OpenFlow cho phép mix vendor".

### Hiểu sai thường gặp

**Hiểu sai:** "OpenFlow là standard protocol, nên switch OpenFlow-compliant của bất kỳ vendor nào cũng hoạt động với bất kỳ controller OpenFlow nào."

**Thực tế:** Compliance với OpenFlow protocol-level handshake không đồng nghĩa compliance với abstract pipeline model. Switch A (Trident2) có ingress ACL 8K nhưng chỉ 256 egress ACL; switch B (FM6000) có 16-stage pipeline nhưng từng stage có constraint. Cả hai "OpenFlow 1.3-compliant" nhưng không tương đương về capability. Controller portable phải detect runtime, không rely on spec compliance. Đây là gap mà TTP cố lấp nhưng không thành.

## 4.5.2 TTP khái niệm: pre-agreed pattern

### Approach: "negotiate pattern before flow"

Thay vì tiếp tục pattern discovery (query + fail-at-runtime), TTP đề xuất:

**Bước 1, Define patterns**: ONF community define các reference TTP, mỗi pattern specify cụ thể:
- Number of tables.
- Match fields per table.
- Instructions per table.
- Group types supported.
- Constraints (e.g., "table 0 chỉ match L2, không match L3").

Mỗi TTP có unique identifier (UUID hoặc URN).

**Bước 2, Switch advertise**: switch ship với list of supported TTP IDs, advertise qua FEATURES_REPLY extension.

**Bước 3, Controller select**: controller choose TTP matching application needs. Nếu switch không support TTP mong muốn, controller fall-back sang alternative TTP hoặc fail gracefully với clear error (không "try flow_mod and hope").

**Bước 4, Install constrained flows**: controller install flow_mod theo TTP schema. Switch validate at install time (không runtime), flow_mod invalid bị reject ngay với clear message.

### Analogy với HTTP Accept header

HTTP client indicate `Accept: application/json, text/html;q=0.9`. Server select best format match client capability, reply `Content-Type: application/json`. Client-server thoả thuận trước khi exchange payload.

TTP đưa same pattern vào OpenFlow:
- Switch = server, advertise list of TTP ID support.
- Controller = client, request specific TTP.
- Agreed TTP = "Content-Type" cho subsequent flow_mod.

Difference với HTTP: TTP là pre-defined catalog (finite set), không dynamic negotiation mỗi request.

### Use case chính: multi-vendor fabric

Scenario: DC có mixed fabric với 3 vendor switch. Controller install "L2-L3 Edge TTP" across all:

- Switch vendor X supports TTP "L2-L3 Edge v1.0" → controller install straight.
- Switch vendor Y supports "L2-L3 Edge v1.0" nhưng lower capacity → controller install với scaled-down policy.
- Switch vendor Z chỉ supports "L2 Only TTP v1.0" → controller deploy L3 logic trên vendor X/Y, chỉ L2 trên Z, coordinate qua controller logic.

Không cần per-vendor plugin code; application logic write once against TTP abstraction.

## 4.5.3 ONF TS-017: TTP spec

### Document history

ONF Technical Specification 017 "OpenFlow Table Type Patterns" phiên bản 1.0 công bố ngày **15/08/2014**. Spec do Forwarding Abstractions Working Group (FAWG) của ONF develop. Document ~60 trang bao gồm YANG schema và 3 reference TTPs.

Historical URL của spec PDF trên opennetworking.org đã bị remove sau website restructure (~2018). Archive tại web.archive.org có snapshot:

```
Search keyword: "ONF TS-017 OpenFlow Table Type Patterns v1.0 August 2014"
Archive search: https://web.archive.org/web/*/opennetworking.org/*TTP*
```

Direct PDF URL gốc không còn khả dụng tại opennetworking.org, phải search qua Wayback Machine với date range 2015-2017.

### YANG-based data model

TTP define bằng YANG (RFC 6020). YANG là language cho network configuration model, originally cho NETCONF (xem Part 5.0). TTP adopt YANG để leverage existing tooling và align với ONF's broader YANG strategy (OpenConfig).

Schema example (simplified):

```
module ttp {
  leaf NDM_metadata { ... }
  list flow_tables {
    key table_number;
    leaf table_number { type uint8; }
    list match_set {
      leaf match_field { type ofp_match_field; }
      leaf mask_type { enum exact | prefix | arbitrary; }
    }
    list instruction_set { ... }
  }
  list group_entry_types { ... }
  list meter_table { ... }
}
```

Controller parse YANG schema, generate code-level representation của TTP, sử dụng khi validate flow_mod trước install.

### Reference TTPs trong spec v1.0

Spec 1.0 bundle 3 reference TTPs:

**L2-L3 Edge TTP**: cho switch DC edge xử lý mixed L2 VLAN + L3 routing. 
- Table 0 (classify): match port + VLAN → assign tenant metadata.
- Table 1 (L2 lookup): match metadata + dst_mac → forward hoặc flood.
- Table 2 (L3 lookup): match metadata + dst_ip → next-hop group.
- Table 3 (ACL): match 5-tuple → permit/deny.

**Aggregation-Core TTP**: cho switch aggregation layer (underlay fabric).
- Table 0 (MPLS/Label match): match MPLS label → pop/push/swap.
- Table 1 (routing): match dst_ip → next-hop.
- Table 2 (ECMP group): select action across multiple next-hops.

**WAN PE (Provider Edge) TTP**: cho switch carrier edge.
- Table 0 (classify): match VLAN/port → service instance.
- Table 1 (policy): match service instance + 5-tuple → QoS class.
- Table 2 (meter): apply rate limit.
- Table 3 (encap): PBB push or MPLS push.
- Table 4 (routing): forward to uplink.

Mỗi TTP define match fields allowed, instruction set, group types (ALL/SELECT/INDIRECT/FAST_FAILOVER). Switch vendor implement chỉ cần align silicon với một hoặc nhiều TTPs.

### Adoption reality: không phổ cập

Trong 2014-2016:

- **Vendor implement TTP**: rất hiếm. HP có announce partial implement cho L2-L3 Edge TTP ở 2014 conference nhưng không ship production firmware. NEC cùng tình trạng. Broadcom không implement TTP layer (Broadcom target silicon API mình là enough).

- **Controller implement TTP**: ONOS không integrate TTP; ODL có project `ttp` experimental nhưng stale 2015. Ryu không có TTP module. Faucet custom abstraction riêng.

- **Application**: không có production SDN app build trên TTP abstraction.

Lý do TTP không thành: 
1. Timing: 2014 là thời điểm Flow Objectives (ONOS) và P4 (paper CCR 2014) xuất hiện, TTP bị overshadowed. 
2. Complexity: YANG schema cho TTP phức tạp, developer adopt cost cao. 
3. Vendor politics: hardware vendor ưu tiên proprietary programmable API (NVIDIA DOCA sau này, Broadcom SAI) thay vì open TTP.

Spec sau 1.0 không có update. ONF dừng development TTP effort ~2016.

## 4.5.4 Alternative: Flow Objectives abstraction (xem Block VI)

### ONOS Flow Objectives approach

ONOS (Open Network Operating System) của ON.Lab (về sau merged với ONF) 2015 release architecture khác giải cùng vấn đề: **application express intent, framework compile sang flow_mod phù hợp switch**.

Abstraction layer gồm:
- **Flow Objectives API**: application call `forwardingObjectiveService.apply(...)` với intent như "forward traffic matching X to port Y with priority Z".
- **Driver SDK**: mỗi switch model có driver class, driver nhận objective từ framework và emit concrete flow_mod phù hợp silicon của switch đó.
- **Pipeline abstraction**: driver encode silicon pipeline knowledge (Trident2 pipeline vs FM6000 pipeline), map objective sang correct table(s) và flow_mod sequence.

Controller application không cần biết silicon detail; framework abstracts away. Khi add new switch model, chỉ cần write new driver class.

### So sánh TTP vs Flow Objectives

| Aspect | TTP | Flow Objectives |
|--------|-----|------------------|
| Where abstraction lives | Switch-side (switch advertises TTP) | Controller-side (framework drivers) |
| Application aware of switch capability | Partially (app knows which TTP) | No (framework hides completely) |
| Extensibility | Fixed catalog (add new TTP = add to ONF spec) | Dynamic (add new driver class in controller) |
| Vendor involvement | Vendor implement TTP trong firmware | Vendor contribute driver class |
| Deployment complexity | Cần firmware update on switch + controller side | Cần driver code chỉ ở controller side |

Flow Objectives thắng ở deployment simplicity (không touch switch firmware) và extensibility (add driver dynamically). ONOS adoption tăng 2015-2018 trong carrier network (AT&T's disaggregated CORD architecture), Flow Objectives trở thành de-facto abstraction pattern.

Chi tiết Flow Objectives sẽ được phân tích trong Part 6.1. Đó là example cho cùng paradigm "abstraction layer above OpenFlow" mà sau này được Kubernetes SDN (OVN-Kubernetes) adopt theo way khác.

### Long-term winner: P4

Về lâu dài, cả TTP và Flow Objectives đều bị overshadowed bởi **P4** (xem Part 6.0). P4 không abstract switch capability, mà cho phép programmer **define** capability. Thay vì "switch có TTP này không?", programmer viết P4 program và compile cho target. Paradigm shift từ protocol-centric (OpenFlow + TTP) sang compiler-centric (P4).

Bài học từ TTP: abstraction layer trên top của fragmented primitive (silicon capability) là approach tạm thời. Long-term solution là homogenize primitive (silicon chấp nhận programmable model) thay vì abstract over fragmentation.

> **Key Topic:** TTP là attempt technical giải bài toán silicon fragmentation, nhưng fail vì hai lý do kết hợp: (a) timing, P4 và Flow Objectives xuất hiện gần đồng thời với proposal value proposition mạnh hơn, (b) vendor incentive, hardware vendor không muốn implement layer open-standard phức tạp, ưu tiên proprietary API hoặc chuyển sang programmable silicon. Bài học: một đề xuất kỹ thuật hay không đảm bảo thành công; timing và alignment với vendor incentive là yếu tố quyết định.

## Dàn bài

## 13.5.1 Port_Binding schema: SBDB anchor của mọi logical port

Port_Binding là bảng trong Southbound DB (SBDB) ghi nhận trạng thái binding của mỗi `Logical_Switch_Port` hoặc `Logical_Router_Port` đã được NBDB publish xuống. Schema chính: `logical_port` (tên từ NBDB), `datapath` (reference Datapath_Binding), `tunnel_key` (16-bit ID dùng trong Geneve encap), `chassis` (chassis hiện đang "own" port này), `type` (chuỗi định nghĩa loại port), `options` (map tùy thuộc type). Khi cột `chassis` = NULL, port chưa được claim; khi set, chassis đó responsible cho forwarding traffic đến/từ port.

## 13.5.2 vif (empty type): workload port (VM/container/bare-metal)

Loại phổ biến nhất. Trường `type` để empty (không phải chuỗi "vif"). Đại diện cho port attach vào workload thực tế — VM tap interface qua libvirt, container veth qua CNI plugin, bare-metal host uplink. Orchestrator (bất kể là OpenStack, Kubernetes, standalone script) chỉ cần set `external_ids:iface-id` trên OVS Interface row khớp với `Logical_Switch_Port.name`, sau đó ovn-controller tự claim port trên chassis đó.

## 13.5.3 localnet: bridge logical network ra VLAN physical

Type=`localnet` + option `network_name=<physnet>` + option `tag=<vlan-id>` (optional). Representation của một provider network/physical VLAN trong logical topology. Không có chassis owner duy nhất — mọi chassis cùng mapping physnet đều có thể forward traffic qua localnet port. Triggers patch port auto-creation giữa br-int và bridge mapping (Part 13.4.3).

## 13.5.4 l2gateway và chassisredirect: distributed gateway port

`l2gateway`: logical port đại diện cho uplink L2 đi qua một chassis cụ thể (chassis gateway). Option `l2gateway-chassis=<chassis-id>` pin port vào chassis đó.

`chassisredirect`: companion port tự động tạo khi `Logical_Router_Port` được đánh dấu distributed gateway port (DGP) với `options:redirect-chassis` hoặc `HA_Chassis_Group`. Option `distributed-port=<router-port-name>`. Traffic north-south qua router port này sẽ redirect qua chassis active theo HA_Chassis_Group (xem Part 13.6).

## 13.5.5 l3gateway: centralized gateway routing

Type=`l3gateway` + option `l3gateway-chassis=<chassis-id>`. Representation của Logical_Router_Port được bind cứng vào một chassis cụ thể cho L3 routing tập trung. Khác với `chassisredirect` (có failover qua HA Chassis Group), `l3gateway` chỉ bind được một chassis — nếu chassis chết, router port mất forwarding cho đến khi admin thay đổi binding.

## 13.5.6 localport: per-chassis metadata port

Type=`localport`. Port có bản sao trên MỌI chassis cùng lúc — không cross-chassis forwarding. Use case chính: metadata service listener (ví dụ DHCP relay, cloud-init metadata server) mà mỗi chassis cần có endpoint riêng cục bộ, không redirect qua tunnel.

## 13.5.7 patch: cross-datapath logical bridge

Type=`patch` + option `peer=<peer-port-name>`. Cặp Logical_Switch_Port kết nối hai logical datapath (ví dụ Logical_Switch ↔ Logical_Router). Không phải OVS patch port vật lý (Part 13.4.4) — đây là logical patch ở pipeline level. Khi ovn-northd dịch ra logical flow, hai datapath được gắn với nhau qua resubmit hoặc metadata swap.

## 13.5.8 virtual: VRRP/keepalived IP floating

Type=`virtual` + option `virtual-parents=<list-of-parent-ports>` + option `virtual-ip=<ip>`. OVN 20.06+ feature cho VRRP active-backup: nhiều parent port có thể claim cùng virtual IP, OVN tự detect owner qua MAC + IP probe. Khi parent active chết, parent khác lên claim virtual IP, OVN update MAC_Binding tương ứng.

## 13.5.9 Workflow diagnosis: `ovn-sbctl list Port_Binding`

Lệnh baseline khi debug "port không thấy traffic": `ovn-sbctl list Port_Binding <logical-port-name>` kiểm tra cột `chassis` (NULL = chưa claim, set = chassis nào), `type` (empty = vif, khác = special). Nếu `chassis=NULL` cho port type=vif, thường do orchestrator chưa set `iface-id` đúng. Nếu `chassis=NULL` cho type=`chassisredirect`, HA_Chassis_Group chưa có active member (xem Part 13.6).

## 9.8.1 The three flow-export protocols

**sFlow** (sFlow.org, 2001) is sampling-based. The switch picks one packet at random out of every N, and sends the packet header (128 bytes by default) plus metadata to the collector. It also polls counters periodically. UDP port 6343.

**NetFlow** (Cisco, 1996; IETF v5 and v9) is per-flow. The switch maintains a flow cache and emits a record when a flow expires. The packet rate is lower than sFlow, but each record is more detailed. UDP port 2055 or custom.

**IPFIX** (RFC 7011, 2013) is NetFlow v10, standardized. It is template-based: the switch sends a template that defines the record format, then sends data records that reference the template. UDP port 4739 by default.

---

## 9.8.2 sFlow configuration

Compass Ch H idiom:

```bash
[host]# ovs-vsctl -- --id=@s create sFlow agent=eth1 \
         target=\"10.0.0.1:6343\" \
         header=128 sampling=64 polling=10 \
         -- set Bridge br0 sflow=@s
```

- `agent=eth1`: the source IP is taken from interface `eth1`.
- `target`: the collector endpoint.
- `header=128`: keep the first 128 bytes of the packet.
- `sampling=64`: sample one packet out of every 64.
- `polling=10`: emit counters every 10 seconds.

Verification: run `tcpdump -ni ethX udp port 6343` on the collector host.

Rotate the collector at runtime with `ovs-vsctl set sFlow <uuid> target='"new:6343"'`. The change is atomic and no packets are lost.

## 9.8.3 NetFlow configuration

```bash
[host]# ovs-vsctl -- set Bridge br0 netflow=@nf \
         -- --id=@nf create NetFlow \
            targets=\"192.168.0.34:5566\" active-timeout=30
```

- `active-timeout=30`: emit a record every 30 seconds for a long-lived flow.
- `engine_id` and `engine_type` are auto-generated, but can be overridden.

---

## 9.8.4 IPFIX configuration

```bash
[host]# ovs-vsctl -- set Bridge br0 ipfix=@i \
         -- --id=@i create IPFIX targets=\"192.168.0.34:4739\" \
                            obs_domain_id=123 obs_point_id=456 \
                            cache_active_timeout=60 cache_max_flows=13
```

- `obs_domain_id` and `obs_point_id` identify the source of the observation. They matter for multi-switch export.
- `cache_max_flows` limits the number of flows in the internal cache.

```bash
[host]# ovs-vsctl clear Bridge br0 sflow
[host]# ovs-vsctl clear Bridge br0 netflow
[host]# ovs-vsctl clear Bridge br0 ipfix
```

Like Mirror, unreferenced rows are garbage-collected automatically.

## 9.8.7 Anatomy Template A: `ovs-vsctl list sFlow`

**Context for the run:** verify that the sFlow configuration is active and that the sampling rate is applied. This is also the entry point for debugging when the collector receives no data.

**Command and output** *(reproduced from OVS 2.17.9)*:

```shell
$ sudo ovs-vsctl list sFlow
_uuid               : 7e3c9f82-1d5e-4b7a-8e9f-abcdef012345
agent               : "eth1"
external_ids        : {purpose="capacity-planning"}
header              : 128
polling             : 10
sampling            : 64
targets             : ["10.0.0.1:6343", "10.0.0.2:6343"]
```

**9-attribute anatomy:**

| Field | Sample value | Meaning | Red flag if |
|-------|--------------|---------|-------------|
| `agent` | `"eth1"` | The interface that provides the source IP | The interface does not exist, sFlow is silently disabled |
| `header` | `128` bytes | Packet header truncation length | Below 64, the header is too short to parse L4; above 256, bandwidth is wasted |
| `sampling` | `64` (1:64 ratio) | Packet sample rate | 1:1 is debug only; production is 1:1024 or higher |
| `polling` | `10` seconds | Counter emit interval | Too low (1 second) overwhelms the collector |
| `targets` | `["IP:port"]` | List of collector endpoints | Multiple targets give redundancy (good); duplicate data is acceptable for sFlow |
| `external_ids` | metadata | Operator tag | Missing makes auditing hard |
| Bridge `sflow` column | `[<uuid>]` | The bridge attaches sFlow | Empty means the sFlow configuration exists but is not active |
| Packet rate at the target | about N times Hz | Verifies the actual sampling | Below expected means a higher sampling rate (fewer samples) |
| Counter emission rate | 1 packet per polling-second | Counter packet | Missing means counters are disabled or the collector is firewalled |

**Failure scenarios:**

- **The sFlow configuration exists but the collector receives nothing**: the bridge `sflow` column is not attached. Verify with `ovs-vsctl get Bridge br0 sflow`. If empty, run `ovs-vsctl set Bridge br0 sflow=<uuid>`.
- **The collector receives data but it is incomplete**: `header=64` is too short for VXLAN plus IPv6 plus TCP. Increase to 256.
- **Sampling rate drift**: the collector sees fewer samples than expected. OVS adaptive sampling adjusts the rate when the flow rate is high. Check with `ovs-appctl sflow/show`.
- **Multiple targets cause duplicate data**: some collectors deduplicate by flow ID, others do not. Verify the collector behavior before setting up redundancy.

## 9.8.8 Common misconceptions and key topic

> **Common misconception 1:** *"sFlow plus NetFlow together give the best of both worlds."*
>
> **Reality:** enabling both protocols doubles the CPU cost (two sample paths plus two export paths). Both protocols overlap in capability by 80 percent. Choose one: sFlow for operators with no historical commitment, NetFlow or IPFIX for teams with existing Cisco infrastructure. Running both is justified only during a migration period between collectors.

> **Common misconception 2:** *"A 1:1024 sampling rate loses too much data, so it is inaccurate."*
>
> **Reality:** sFlow has good statistical accuracy at long time scales. 1:1024 on 1 Gbps gives about 1k samples per second, which is enough for 5-minute flow aggregation with a confidence interval below 5 percent. Only short-term bursts (sub-second) lose visibility. For long-term capacity planning and anomaly detection, 1:1024 is the sweet spot between CPU cost and accuracy. For accurate incident forensics, use Mirror (Part 9.7).

> **Key topic:** flow monitoring is **passive observability**. It does not change the traffic path; it only sends metadata to the collector. It complements active monitoring (BFD, ping) and pipeline debugging (`ovs-appctl ofproto/trace`). The three-tier observability framework for OVS is: (1) Control plane: `ofproto/trace` plus `ovsdb-client monitor`. (2) Data plane: Mirror (Part 9.7) plus flow monitoring. (3) Performance: `coverage/show` (Part 9.11) plus sFlow counters.

## 9.8.9 Schema row OVSDB for NetFlow

**Concept and role.** The `NetFlow` table in `vswitch.ovsschema` holds the runtime configuration for the NetFlow exporter. Each row describes one exporter target set; `Bridge.netflow` references one row through a UUID. The switch reads the row and emits NetFlow v5 records (the default) to every endpoint in `targets`.

**Columns (verified upstream `vswitchd/vswitch.ovsschema` at v2.17.9):**

| Column | Type | Meaning |
|--------|------|---------|
| `targets` | set of string, at least 1 | Collector endpoint `IP:port` (port 2055 is common) |
| `engine_type` | int 0-255, optional | The `engine_type` field in the NetFlow header; identifies the source |
| `engine_id` | int 0-255, optional | The `engine_id` field in the header; identifies a specific switch instance |
| `add_id_to_interface` | bool | When true, OR `engine_id` into the upper bits of the `input` and `output` interface IDs, which helps deduplicate when several switches export to the same collector |
| `active_timeout` | int at or above -1 | Seconds between two emits for a long-lived flow; the value `-1` disables active expiration |
| `external_ids` | map string to string | Operator metadata; does not affect runtime |

**Create and use.** The atomic-transaction idiom:

```bash
[host]# ovs-vsctl -- --id=@nf create NetFlow \
         targets=\"192.168.0.34:2055\" \
         engine_id=10 engine_type=128 active_timeout=30 \
         -- set Bridge br0 netflow=@nf
```

**Observation.** `ovs-vsctl list NetFlow` prints the entire row plus its UUID. `ovs-vsctl get Bridge br0 netflow` returns the currently attached UUID. An empty result `[]` means the bridge has not attached the configuration even though the row exists; this silent miss is a common operator mistake.

**Version differences.** The `engine_type`, `engine_id`, and `add_id_to_interface` columns have been stable since OVS 1.x. The `active_timeout = -1` semantic that disables active expiration was actually introduced earlier than the curriculum previously claimed: the code-side change in `ofproto/netflow.c` arrived in commit `e9e2856e084383b4f93b371dfba8320b51d4cd03` on 2009-12-06 ("netflow: Correctly set active timeout value."), which switched the validation check from `active_timeout > 0` to `active_timeout >= 0` and treated negative inputs as a request for the default. The schema documentation that explicitly describes `-1` as "disables active timeouts" was added shortly after in commit `8936565369410daa099708be4cd3fa7e0e39bade` on 2010-03-03 ("Add documentation for the vswitch database schema."). Both commits are contained in the v1.0.0 release tag. Earlier versions required `active_timeout >= 30`. Both SHAs were verified on 2026-04-28 against the local OVS repo with `git log -1 <sha>` and `git tag --contains <sha>`.

**Common failures.** A row exists but `Bridge.netflow=[]`, so the collector receives nothing. A wrong port in `targets`, so `tcpdump -ni any udp port 2055` is empty. An `engine_id` collision between two switches that share a collector causes records that the collector deduplicates by `engine_id` to be merged incorrectly.

## 9.8.10 Schema row OVSDB for sFlow

**Concept and role.** The `sFlow` table holds the sampling-based exporter configuration. `Bridge.sflow` references one row. The `ovs-vswitchd` sampling thread picks one packet out of every N on each datapath flow, and copies the first `header` bytes plus periodic counters.

**Columns (verified upstream `vswitchd/vswitch.ovsschema` at v2.17.9):**

| Column | Type | Meaning |
|--------|------|---------|
| `targets` | set of string, at least 1 | Collector `IP:port` (port 6343 is the sFlow.org standard) |
| `sampling` | int, optional | The 1-in-N ratio (default 400). Higher means fewer samples and lower CPU cost |
| `polling` | int, optional | Seconds between two interface counter emits (default 30; 0 disables) |
| `header` | int, optional | The number of leading bytes kept (default 128). Too small misses the L4 header; too large wastes bandwidth |
| `agent` | string, optional | The interface name used as the `Agent IP` in the sFlow datagram (the collector uses it to identify the switch). If the interface does not exist, the exporter is silently disabled |
| `external_ids` | map string to string | Operator metadata |

**sFlow versus NetFlow specifics.** The sFlow row has no `active_timeout` (sampling does not cache flows); instead it has `polling` for counters. A sample is emitted as soon as the packet hits the sampling rate, with no buffering.

**Create and verify.**

```bash
[host]# ovs-vsctl -- --id=@s create sFlow agent=eth1 \
         target=\"10.0.0.1:6343\" header=128 sampling=64 polling=10 \
         -- set Bridge br0 sflow=@s
[host]# ovs-appctl sflow/show br0      # runtime stats: sample count, drop, last emit
```

**Version differences.** The `agent` interface-name semantic has been stable since OVS 1.0. The defaults `sampling=400`, `polling=30`, and `header=128` have not changed since commit `9b0ed8e3` (the 2009 import).

**Common failures.** `agent=eth9` (a non-existent interface) causes the datagram source IP to be empty, so `sflowtool` rejects it. A `header=64` is too short for VXLAN, IPv6, and TCP plus options (about 86 bytes), so the L4 header is truncated and the collector cannot parse it.

## 9.8.11 Schema row OVSDB for IPFIX

**Concept and role.** The `IPFIX` table holds the configuration of the IPFIX exporter (RFC 7011, NetFlow v10 standardized). There are two distinct use cases through two different references:

- `Bridge.ipfix` (per-bridge): an exporter for all traffic on the bridge with a fixed sample rate.
- `Flow_Sample_Collector_Set.ipfix` (per-flow `sample` action): an exporter selected by an OpenFlow `sample` action, used for selective monitoring.

**Columns (verified upstream `vswitchd/vswitch.ovsschema` at v2.17.9):**

| Column | Type | Meaning |
|--------|------|---------|
| `targets` | set of string, at least 0 | Collector `IP:port` (port 4739 is the IPFIX default; 4740 is for TLS) |
| `sampling` | int 1-4294967295, optional | The 1-in-N ratio. The bridge mode default is 400; per-flow mode takes the value from the `sample` action |
| `obs_domain_id` | int 0-4294967295, optional | Observation Domain ID (RFC 7011 §3.1), distinguishes observation sources across a multi-switch fabric |
| `obs_point_id` | int 0-4294967295, optional | Observation Point ID, distinguishes observation points within the same domain |
| `cache_active_timeout` | int 0-4200, optional | Seconds to keep a flow in the internal cache before emit; max 4200 to prevent state bloat |
| `cache_max_flows` | int 0-4294967295, optional | Limit on cache entries; exceeding it triggers LRU eviction and forced emit |
| `other_config` | map string to string | Advanced tunables: `enable-tunnel-sampling`, `virtual_obs_id`, `enable-input-sampling`, `enable-output-sampling` |
| `external_ids` | map string to string | Operator metadata |

**Create in bridge mode.**

```bash
[host]# ovs-vsctl -- --id=@i create IPFIX targets=\"192.168.0.34:4739\" \
         obs_domain_id=123 obs_point_id=456 \
         cache_active_timeout=60 cache_max_flows=13 \
         -- set Bridge br0 ipfix=@i
```

**Version differences.** The `cache_active_timeout` column was added in OVS 2.0 (commit `978427a5`, 2013) along with the IPFIX import; before that, IPFIX hardcoded 60 seconds. The `cache_active_timeout <= 4200` range has been enforced since schema 7.0.0.

**Common failures.** `obs_domain_id` is not set, so it defaults to 0 and a multi-switch collector cannot distinguish the source. `cache_max_flows` is too small (below 100) on a link with many flows, so LRU eviction runs continuously and records are duplicated.

## 9.8.12 Bridge column references netflow, sflow, ipfix

**Concept and role.** Three columns in the `Bridge` table are weak-reference type. Each column holds 0 or 1 UUID pointing at the corresponding row:

```
Bridge.netflow : optional UUID -> NetFlow row
Bridge.sflow   : optional UUID -> sFlow   row
Bridge.ipfix   : optional UUID -> IPFIX   row
```

**Why a separate table instead of inline columns.** OVSDB normalization: many bridges can share the same exporter configuration (the same row UUID). To rotate the target, you only need to update one row, and every bridge that attaches it reloads atomically. Inline columns would force you to update each bridge separately, which opens a race window.

**How a switch attaches.** Two equivalent forms:

```bash
# Form 1: an atomic transaction that creates a row and attaches it in one command.
[host]# ovs-vsctl -- --id=@s create sFlow targets=\"127.0.0.1:6343\" agent=eth0 \
         -- set Bridge br0 sflow=@s

# Form 2: attach a UUID that already exists, possibly from another bridge.
[host]# UUID=$(ovs-vsctl get sFlow . _uuid | head -1)
[host]# ovs-vsctl set Bridge br1 sflow=$UUID
```

**Detach and garbage collection.** `ovs-vsctl clear Bridge br0 sflow` sets the column back to `[]`. OVSDB weak references work like this: when no bridge references the row, it is garbage-collected at the next compaction. You do not need to call `destroy` manually.

**Observation.**

```bash
[host]# ovs-vsctl get Bridge br0 sflow              # UUID or []
[host]# ovs-vsctl --columns=name,sflow,netflow,ipfix list Bridge
```

**Common failures.** The row exists but the column is empty, so the exporter is idle (Anatomy §9.8.7 red flag). When several bridges share one row, clearing one bridge does not free the row because other bridges still reference it. The operator may believe the configuration was fully cleared.

**Version differences.** These three columns have been stable since OVS 1.0 (sflow), 1.0 (netflow), and 2.0 (ipfix bridge mode).

## 9.8.13 Flow_Sample_Collector_Set, selective sampling per flow

**Concept and role.** The `Flow_Sample_Collector_Set` table allows selective traffic export through the OpenFlow `sample` action (see Part 4.9 action catalog). Unlike `Bridge.ipfix`, which samples every packet at a fixed rate, the collector set is triggered only when a flow hits a `sample(collector_set_id=N, ...)` action. Use case: a production fabric that wants to monitor 100 percent of traffic through a specific ACL match (for example, traffic between two tenants) without wasting CPU sampling broadcast or control traffic.

**Columns (verified upstream `vswitchd/vswitch.ovsschema` at v2.17.9):**

| Column | Type | Meaning |
|--------|------|---------|
| `id` | int 0-4294967295, required | Collector set ID; the `sample` action references it through this field |
| `bridge` | UUID Bridge, required | The bridge that owns the collector set (sets are scoped per-bridge) |
| `ipfix` | UUID IPFIX, optional | Reference to the IPFIX row that holds the target, sampling, and observation configuration |
| `external_ids` | map string to string | Operator metadata |

**Why a separate table instead of a column in Bridge.** A bridge can have many collector sets with different IDs, and each set can point to a different IPFIX row. This is a 1-to-N relation, which forces a separate table. Plus, the `id` field is an integer rather than a UUID because the `sample` action references it by integer; you cannot embed a UUID in OpenFlow action bytes.

**Create and use.**

```bash
# Step 1: create an IPFIX row for the per-flow exporter.
[host]# ovs-vsctl -- --id=@i create IPFIX targets=\"10.0.0.1:4739\" \
         obs_domain_id=200 cache_active_timeout=60 \
         -- --id=@cs create Flow_Sample_Collector_Set id=1 bridge=@br ipfix=@i \
         -- --id=@br get Bridge br0

# Step 2: install a flow rule with a sample action that references id=1.
[host]# ovs-ofctl add-flow br0 \
         "table=0,priority=100,ip,nw_src=10.0.0.0/8,actions=sample(probability=65535,collector_set_id=1,obs_domain_id=200,obs_point_id=300),normal"
```

`probability=65535` means 100 percent sampling (the maximum 16-bit value). `collector_set_id=1` matches `Flow_Sample_Collector_Set.id`.

**Observation.** `ovs-vsctl list Flow_Sample_Collector_Set` prints all sets with their references. `ovs-appctl dpif/show-dp-features br0` reports whether the `sample` action is supported.

**Version differences.** The table was added in OVS 2.4 (commit `8b7ea2d4`, 2015) along with the OpenFlow `sample` action; before that, only bridge-mode IPFIX existed. The `obs_domain_id` and `obs_point_id` arguments inside the action override the row defaults from OVS 2.5 onwards.

**Common failures.** `collector_set_id` in the action does not match the row `id`, so the action is silently dropped (no log). The `bridge` reference is wrong, so the sample action runs but there is no IPFIX row, and the packet is sampled in the datapath but never exported (CPU waste). Use `ovs-appctl dpctl/dump-flows` to check whether the datapath has the correct `sample_id` flag.

## 9.8.14 Capacity planning and tuning

**Sampling rate versus CPU overhead:**

| Sampling rate | CPU overhead (2x 10G) | Sample rate to collector | Use case |
|---------------|------------------------|--------------------------|----------|
| 1:1 | 50 percent or more CPU | 1M+ samples/s | Debug only |
| 1:64 | 3 to 5 percent | 15K samples/s | Dense monitoring |
| 1:1024 | below 1 percent | 1K samples/s | Production standard |
| 1:8192 | below 0.2 percent | 120 samples/s | Very high-rate fabric |

**Collector scaling:**

- **`sflowtool` on a single host**: 10K samples/s is fine (disk I/O becomes the bottleneck above 50K).
- **`ntopng`**: about 50K samples/s with SSD storage.
- **`goflow2` plus Kafka plus ClickHouse**: 1M+ samples/s sustained.

**Retention math:** 1K samples/s times 24 hours equals 86M records per day. Each record is about 200 bytes, so approximately 17 GB per day per collector (Parquet compresses to about 30 percent, that is about 5 GB per day). 30-day retention requires 150 GB.

## Key points to remember

First, the three protocols serve three use cases: sFlow for high-volume traffic analysis (sampling), and NetFlow or IPFIX for accurate flow accounting.

Second, the atomic transaction `-- --id=@` is a required idiom. It guarantees the correct reference between the bridge and the collector configuration.

Third, rotating the collector is atomic. OVS streams to the new target without losing packets.

Fourth, feed sFlow into `goflow2` or `ntopng`, and feed NetFlow or IPFIX into `nfdump` or Kentik. Together they form aggregated traffic analytics.

Fifth, Anatomy Template A `list sFlow` has 9 attributes. The red flag is the bridge `sflow` column being empty even though the configuration exists.

Sixth, dual sFlow plus NetFlow doubles the cost. Pick one protocol per use case. 1:1024 sampling is the production sweet spot.

Seventh, flow monitoring is passive observability. It complements Mirror (data plane) and `ofproto/trace` (control plane) in the three-tier framework.

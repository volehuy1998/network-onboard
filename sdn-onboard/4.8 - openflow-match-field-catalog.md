1. **Liệt kê (Remember)** 12 nhóm khớp với field foundation của OpenFlow + OVS extensions: Metadata, Register, Tunnel, L2, ARP, IPv4, IPv6, L4 (TCP/UDP/SCTP), ICMP, MPLS, Conntrack, Packet-type.

## Bối cảnh: Tại sao cần catalog đầy đủ

OpenFlow 1.0 (12/2009) có **12 khớp với field cứng**, match-all-or-nothing per field:

```
in_port, dl_src, dl_dst, dl_type, dl_vlan, dl_vlan_pcp,
nw_src, nw_dst, nw_proto, nw_tos, tp_src, tp_dst
```

OpenFlow 1.2 (12/2011) chuyển sang OXM TLV extensible format (Phần 4.1 chi tiết). Từ đó tập khớp với field mở rộng liên tục:
- OpenFlow 1.3 (04/2012): IPv6 full + meter identifier.
- OpenFlow 1.4 (08/2013): optical port field.
- OpenFlow 1.5 (12/2014): packet_type, PBB I-SID, actset_output, TCP flags.
- OVS Nicira extensions (NXM): register reg0-reg15, xreg, xxreg, conntrack field, pkt_mark, metadata alias.
- OVN extensions: tun_metadata[0-63] cho Geneve TLV, ct_nw_src/dst/proto (conntrack orig).

Đến OVS 2.17, man `ovs-fields(7)` liệt kê **hơn 100 match field**. Hiểu hết là impossible cho kỹ sư mới, nhưng **phân nhóm + thuộc prerequisite** là khả thi. Phần này cấu trúc 12 nhóm, mỗi nhóm có field quan trọng được breakdown Template B.

## 4.8.1 9-attribute anatomy per field (template chung)

Mỗi field được mô tả theo template sau (9-attribute `ovs-fields(7)` pattern):

| Attribute | Ý nghĩa |
|---|---|
| **Name** | Tên field + aliases (ví dụ `nw_src` = `ip_src` = `ipv4_src`) |
| **Width** | Số bit của field |
| **Format** | Cách parse giá trị (hex / decimal / MAC / IPv4 / IPv6 / string / set) |
| **Masking** | Hỗ trợ mask hay không (bitwise, CIDR prefix, exact-only) |
| **Prerequisites** | Match field nào phải set trước (ví dụ `eth_type=0x0800` cho IP) |
| **Access** | Read-only trong match / read-write qua action `set_field` |
| **OpenFlow version** | 1.0 / 1.1 / 1.3 / 1.5 / NXM (Nicira extension) |
| **OXM/NXM identifier** | Tên symbolic (dùng trong spec parse) |
| **Semantics** | Ý nghĩa field trong gói tin / context pipeline |

Ngoài ra: example match syntax, action liên quan, kịch bản bẻ gãy, upstream reference.

## 4.8.2 Nhóm A, Metadata fields (6 field foundation)

Metadata field không thuộc gói tin wire — chúng là **context** từ pipeline hoặc bridge:

### 4.8.2.1 `in_port`, cổng ingress

| Attribute | Giá trị |
|---|---|
| **Name** | `in_port` (alias: `in_port_oxm` cho OF 1.2+) |
| **Width** | 16-bit (OF 1.0) / 32-bit (OF 1.1+) |
| **Format** | Integer port number hoặc port name |
| **Masking** | Exact match only |
| **Prerequisites** | None |
| **Access** | Read-only trong match |
| **OpenFlow version** | 1.0 all versions |
| **OXM/NXM** | `OXM_OF_IN_PORT` (OF 1.2+), `OXM_OF_IN_PHY_PORT` (OF 1.1+ bonded case), `NXM_OF_IN_PORT` |
| **Semantics** | Datapath port number gói tin arrive. Dùng để differentiate ingress từ port khác nhau |

**Reserved port value:**

| Port name | Value | Ý nghĩa |
|---|---|---|
| `LOCAL` | 0xfffe | Bridge internal port |
| `CONTROLLER` | 0xfffd | OpenFlow controller (packet_in direction) |
| `NORMAL` | 0xfffa | Hành vi L2 learning switch tiêu chuẩn |
| `FLOOD` | 0xfffb | Flood out all ports except ingress |
| `ALL` | 0xfffc | Output to all physical ports |
| `IN_PORT` | 0xfff8 | Send back out ingress port |
| `ANY` | 0xffffffff | Wildcard (match any port) |
| `TABLE` | 0xfff9 | Submit gói tin to OpenFlow pipeline lookup |

**Ví dụ match:**

```
priority=100,in_port=eth0,actions=output:2           # match ingress eth0
priority=100,in_port=LOCAL,actions=ct(commit)        # match bridge-internal
```

**Kịch bản bẻ gãy:**

- Dùng port number numeric (`in_port=1`) thay vì port name (`in_port=tap-vm-web`) → flow rule sẽ break nếu ovs-vswitchd restart assign lại port number.

### 4.8.2.2 `metadata`, 64-bit pipeline metadata

| Attribute | Giá trị |
|---|---|
| **Name** | `metadata` |
| **Width** | 64-bit |
| **Format** | Hex |
| **Masking** | Bitwise mask supported |
| **Prerequisites** | None |
| **Access** | Read qua match; write qua instruction `Write-Metadata` hoặc action `set_field` |
| **OpenFlow version** | 1.1+ |
| **OXM/NXM** | `OXM_OF_METADATA` |
| **Semantics** | 64-bit opaque value set bởi instruction `Write-Metadata` ở table trước, match ở table sau. Dùng pass state giữa các table trong multi-table pipeline |

**Ví dụ use case:**

```
# Table 0 classify + set metadata
table=0,priority=100,ip,nw_src=10.0.1.0/24,
  actions=write_metadata:0x10/0xff,goto_table:10

# Table 10 policy dựa trên metadata
table=10,metadata=0x10/0xff,actions=output:5
```

**Kịch bản bẻ gãy:**

- Metadata chỉ truyền được qua `Goto-Table`, **không truyền qua recirc** (recirc reset metadata). Dùng register nếu cần preserve qua recirc.

### 4.8.2.3 `pkt_mark`, Linux skb mark bridging

| Attribute | Giá trị |
|---|---|
| **Name** | `pkt_mark` |
| **Width** | 32-bit |
| **Format** | Hex hoặc decimal |
| **Masking** | Bitwise |
| **Prerequisites** | None |
| **Access** | Read + write (qua action `set_field`) |
| **OpenFlow version** | NXM extension |
| **OXM/NXM** | `NXM_NX_PKT_MARK` |
| **Semantics** | Liên kết với Linux skb->mark. Dùng pass state giữa OVS và iptables/nftables hoặc kernel L7 filter |

**Ví dụ use case:**

```
# OVS mark packet dùng cho iptables policy
actions=set_field:0x42->pkt_mark,output:eth0

# iptables sau đó match
iptables -A FORWARD -m mark --mark 0x42 -j ACCEPT
```

### 4.8.2.4 `actset_output`, OF 1.5 action set output port

| Attribute | Giá trị |
|---|---|
| **Name** | `actset_output` |
| **Width** | 32-bit |
| **Format** | Integer port number |
| **Masking** | Exact match |
| **Prerequisites** | None |
| **Access** | Read-only (set tự động bởi action `output` trong action set) |
| **OpenFlow version** | 1.5+ |
| **OXM/NXM** | `OXM_OF_ACTSET_OUTPUT` |
| **Semantics** | Port number lệnh `output` trong **action set** hiện tại đang trỏ tới. Cho phép egress table match dựa trên output decision từ ingress |

Sample use: OF 1.5 egress pipeline match `actset_output=5` để áp dụng QoS cho traffic hướng port 5.

### 4.8.2.5 `skb_priority`, Linux QoS priority

32-bit field liên kết `skb->priority`. Dùng với `set_queue` action để select Linux HTB class.

```
actions=set_queue:2,output:eth0    # equivalent set skb_priority
```

### 4.8.2.6 `tun_metadata[0-63]`, Geneve TLV carry

| Attribute | Giá trị |
|---|---|
| **Name** | `tun_metadata0` đến `tun_metadata63` |
| **Width** | Variable (1 byte đến 128 byte per metadata) |
| **Format** | Hex |
| **Masking** | Bitwise |
| **Prerequisites** | Tunnel port (Geneve) |
| **Access** | Read + write |
| **OpenFlow version** | NXM extension |
| **OXM/NXM** | `NXM_NX_TUN_METADATA0` ... `NXM_NX_TUN_METADATA63` |
| **Semantics** | Geneve Variable-Length TLV option. OVN dùng `tun_metadata0` mang logical ingress port + `tun_metadata1` mang logical egress port qua Geneve header |

Chi tiết Geneve TLV class 0x0102 xem Phần 9.27.

## 4.8.3 Nhóm B, Register fields (reg0-reg15 + xreg + xxreg)

Register là **scratch space** per-gói tin trong pipeline — không thuộc gói tin wire, không shared giữa các gói tin. Dùng để pass state giữa các table (hoặc giữa các stage của cùng table).

| Register family | Count | Width | OXM alias |
|---|---|---|---|
| `reg0` đến `reg15` | 16 | 32-bit mỗi cái | `NXM_NX_REG0` ... `NXM_NX_REG15` |
| `xreg0` đến `xreg7` | 8 | 64-bit (bằng 2 register 32-bit ghép) | `OXM_OF_PKT_REG0` ... `OXM_OF_PKT_REG7` |
| `xxreg0` đến `xxreg3` | 4 | 128-bit (bằng 4 register 32-bit ghép) | NXM extension |

### 4.8.3.1 Template B cho `reg0` (đại diện cho nhóm register)

| Attribute | Giá trị |
|---|---|
| **Name** | `reg0` |
| **Width** | 32-bit |
| **Format** | Hex |
| **Masking** | Bitwise + bit-range (ví dụ `reg0[0..15]`) |
| **Prerequisites** | None |
| **Access** | Read + write (qua action `load`, `move`, `set_field`) |
| **OpenFlow version** | NXM extension (OVS specific) |
| **OXM/NXM** | `NXM_NX_REG0` |
| **Semantics** | Scratch register 32-bit per-gói tin, preserve qua pipeline nhưng lost sau gói tin exit |

**Ví dụ load + use:**

```
# Table 0: load 0x42 vào reg0
table=0,priority=100,in_port=1,actions=load:0x42->NXM_NX_REG0[],resubmit(,10)

# Table 10: match reg0 để route
table=10,reg0=0x42,actions=output:5
table=10,reg0=0x00,actions=drop
```

**Bit-range match:**

```
# Match chỉ bit 0-15 của reg0
table=10,reg0=0x0042/0xffff,actions=output:5
```

### 4.8.3.2 OVN dùng register như thế nào

OVN chia register map cứng trong logical flow → OpenFlow translation:

| Register | OVN semantics |
|---|---|
| `reg0` | ACL flags (reg0[0]=ct_hint, reg0[1]=committed, ...) |
| `reg13` | Logical datapath tunnel ID |
| `reg14` | Logical ingress port |
| `reg15` | Logical egress port |
| `xxreg0` / `xxreg1` | IPv6 src / dst working copy |

Kỹ sư debug OVN OpenFlow trace phải thuộc map này. Xem `ovn-architecture(7)` §Tunnel Encapsulation cho table.

## 4.8.4 Nhóm D, L2 Ethernet fields (9 field foundation)

### 4.8.4.1 `eth_src`, `eth_dst`

| Attribute | Giá trị |
|---|---|
| **Name** | `eth_src` (alias: `dl_src`), `eth_dst` (alias: `dl_dst`) |
| **Width** | 48-bit (6 byte) |
| **Format** | MAC `aa:bb:cc:dd:ee:ff` |
| **Masking** | Bitwise mask per byte (ví dụ `01:00:00:00:00:00/01:00:00:00:00:00` cho multicast bit) |
| **Prerequisites** | None |
| **Access** | Read + write (qua `mod_dl_src`, `mod_dl_dst`, `set_field`) |
| **OpenFlow version** | 1.0 all |
| **OXM/NXM** | `OXM_OF_ETH_SRC`, `OXM_OF_ETH_DST` |
| **Semantics** | Ethernet source/destination MAC. Bit 0 của byte 0 = multicast bit (1=multicast/broadcast, 0=unicast) |

**Ví dụ match:**

```
# Multicast traffic: first byte có bit 0 set
priority=100,eth_dst=01:00:00:00:00:00/01:00:00:00:00:00,actions=...

# Broadcast exact
priority=100,eth_dst=ff:ff:ff:ff:ff:ff,actions=flood

# Match OUI vendor (first 3 bytes)
priority=100,eth_src=00:50:56:00:00:00/ff:ff:ff:00:00:00,actions=...
```

### 4.8.4.2 `eth_type`, EtherType

| Attribute | Giá trị |
|---|---|
| **Name** | `eth_type` (alias: `dl_type`) |
| **Width** | 16-bit |
| **Format** | Hex |
| **Masking** | Exact match |
| **Prerequisites** | None |
| **Access** | Read-only (write qua `push_vlan`/`pop_vlan` gián tiếp) |
| **OpenFlow version** | 1.0 all |
| **OXM/NXM** | `OXM_OF_ETH_TYPE` |
| **Semantics** | EtherType của gói tin. Prerequisite của L3 field matching |

**Common value:**

| Value | Protocol | Prerequisite cho field |
|---|---|---|
| `0x0800` | IPv4 | `nw_src/dst`, `nw_proto`, `nw_tos`, etc. |
| `0x86dd` | IPv6 | `ipv6_src/dst`, `ipv6_label` |
| `0x0806` | ARP | `arp_op`, `arp_spa/tpa/sha/tha` |
| `0x8100` | VLAN 802.1Q | (VLAN field tự động kiểm tra) |
| `0x88a8` | Q-in-Q (802.1ad) | PBB field |
| `0x8847` | MPLS unicast | `mpls_label/tc/bos/ttl` |

### 4.8.4.3 `vlan_tci`, `vlan_vid`, `vlan_pcp`, VLAN fields

| Attribute | `vlan_tci` (classic) | `vlan_vid` (OXM) | `vlan_pcp` (OXM) |
|---|---|---|---|
| **Width** | 16-bit (full TCI) | 12-bit | 3-bit |
| **Format** | Hex | Decimal VLAN ID | Decimal 0-7 |
| **Masking** | Bitwise | Bitwise | Exact |
| **Prerequisites** | None | None (VLAN present flag implicit) | None |
| **OpenFlow version** | 1.0 `vlan_tci`; 1.2+ dùng `vlan_vid` + `vlan_pcp` OXM |
| **OXM** | NXM | `OXM_OF_VLAN_VID` | `OXM_OF_VLAN_PCP` |

**CFI/Present bit:** bit 12 của `vlan_tci` = CFI (OF 1.0) hoặc Present flag (OF 1.1+ OXM). Match `vlan_vid=0` + CFI=0 nghĩa là "untagged"; match `vlan_vid=0x1000` + CFI=1 nghĩa là "tagged priority 0".

**Ví dụ match VLAN 100:**

```
priority=100,vlan_vid=100,actions=strip_vlan,output:2
priority=100,dl_vlan=100,actions=mod_vlan_vid:200,output:3
```

### 4.8.4.4 `conj_id`, Conjunctive Match

| Attribute | Giá trị |
|---|---|
| **Name** | `conj_id` |
| **Width** | 32-bit |
| **Format** | Decimal |
| **Masking** | Exact |
| **Prerequisites** | Match gói tin đã qua `conjunction()` action |
| **Access** | Read-only |
| **OpenFlow version** | NXM extension |
| **OXM/NXM** | `NXM_NX_CONJ_ID` |
| **Semantics** | ID của conjunction đã match. Cho phép compress cross-product ACL rule |

**Ví dụ cross-product compression:**

Không dùng conjunction, 3 src × 3 dst = 9 rule. Dùng conjunction = 3+3 rule:

```
# Src rule (conj 1, component 1/2)
priority=100,ip,nw_src=10.0.1.1,actions=conjunction(1,1/2)
priority=100,ip,nw_src=10.0.1.2,actions=conjunction(1,1/2)
priority=100,ip,nw_src=10.0.1.3,actions=conjunction(1,1/2)

# Dst rule (conj 1, component 2/2)
priority=100,ip,nw_dst=10.0.2.1,actions=conjunction(1,2/2)
priority=100,ip,nw_dst=10.0.2.2,actions=conjunction(1,2/2)
priority=100,ip,nw_dst=10.0.2.3,actions=conjunction(1,2/2)

# Final rule match conj_id
priority=100,conj_id=1,actions=output:5
```

Compression factor: N×M rule → N+M rule. OVN ACL dùng pattern này cho Port_Group cross-product.

### 4.8.4.5 `packet_type`, OF 1.5 non-Ethernet gói tin

32-bit field xác định gói tin namespace + type. Mặc định `0:0` = Ethernet. OF 1.5 hỗ trợ packet_type khác cho raw IP (không Ethernet header) hoặc NSH. Ít dùng trong triển khai OVN.

## 4.8.5 Nhóm E, ARP fields (5 field)

### 4.8.5.1 Template B gộp nhóm ARP

| Field | Width | Semantics | OXM |
|---|---|---|---|
| `arp_op` | 16-bit | ARP opcode: 1=Request, 2=Reply, 3=RARP Req, 4=RARP Reply | `OXM_OF_ARP_OP` |
| `arp_spa` | 32-bit IPv4 | ARP sender protocol address | `OXM_OF_ARP_SPA` |
| `arp_tpa` | 32-bit IPv4 | ARP target protocol address | `OXM_OF_ARP_TPA` |
| `arp_sha` | 48-bit MAC | ARP sender hardware address | `OXM_OF_ARP_SHA` |
| `arp_tha` | 48-bit MAC | ARP target hardware address | `OXM_OF_ARP_THA` |

**Prerequisites chung:** `eth_type=0x0806`.

**Ví dụ match ARP Request cho specific IP:**

```
priority=200,arp,arp_op=1,arp_tpa=10.1.0.1,actions=local,controller
```

**Kịch bản bẻ gãy:**

- Match `arp_spa` mà thiếu `eth_type=0x0806` → flow spec sẽ error "ARP field requires eth_type 0x0806". OVS tự suggest add prerequisite với alias `arp`.

## 4.8.6 Nhóm F, L3 IPv4 fields (6 field)

### 4.8.6.1 `ipv4_src`, `ipv4_dst`

| Attribute | Giá trị |
|---|---|
| **Name** | `ipv4_src` (aliases: `ip_src`, `nw_src`), `ipv4_dst` (aliases: `ip_dst`, `nw_dst`) |
| **Width** | 32-bit |
| **Format** | IPv4 dotted `10.0.0.1` hoặc CIDR `10.0.0.0/24` |
| **Masking** | CIDR prefix hoặc bitwise mask |
| **Prerequisites** | `eth_type=0x0800` (or alias `ip`) |
| **Access** | Read + write (`mod_nw_src`, `mod_nw_dst`) |
| **OpenFlow version** | 1.0 all |
| **OXM/NXM** | `OXM_OF_IPV4_SRC`, `OXM_OF_IPV4_DST` |

**Ví dụ match:**

```
# Match subnet
priority=100,ip,nw_src=10.0.0.0/24,actions=output:5

# Match host exact
priority=100,ip,nw_dst=10.0.0.1,actions=drop

# Mask arbitrary bit pattern
priority=100,ip,nw_src=10.0.0.0/0xffffff00,actions=...
```

### 4.8.6.2 `nw_proto`, IP protocol number

| Attribute | Giá trị |
|---|---|
| **Name** | `nw_proto` (alias: `ip_proto`) |
| **Width** | 8-bit |
| **Format** | Integer 0-255, hoặc name |
| **Masking** | Exact |
| **Prerequisites** | `eth_type=0x0800` hoặc `eth_type=0x86dd` |
| **OXM/NXM** | `OXM_OF_IP_PROTO` |

**Common value:**

| Value | Protocol | Prerequisite cho field |
|---|---|---|
| 1 | ICMP | `icmp_type`, `icmp_code` |
| 2 | IGMP | - |
| 6 | TCP | `tcp_src`, `tcp_dst`, `tcp_flags` |
| 17 | UDP | `udp_src`, `udp_dst` |
| 41 | IPv6 encapsulation | - |
| 47 | GRE | - |
| 50 | ESP (IPsec) | - |
| 51 | AH (IPsec) | - |
| 58 | ICMPv6 | `icmpv6_type`, `icmpv6_code` |
| 89 | OSPF | - |
| 132 | SCTP | `sctp_src`, `sctp_dst` |

**Ví dụ alias convenient:**

```
priority=100,tcp,tp_dst=80,actions=...   # eth_type=0x0800, nw_proto=6
priority=100,udp,tp_dst=53,actions=...   # eth_type=0x0800, nw_proto=17
priority=100,icmp,actions=...             # eth_type=0x0800, nw_proto=1
```

### 4.8.6.3 `nw_tos`, `nw_ecn`, `nw_ttl`

| Field | Width | Format | Semantics |
|---|---|---|---|
| `nw_tos` | 6-bit | Integer 0-63 | IPv4 DSCP (upper 6 bit của TOS byte). Dùng cho QoS classification |
| `nw_ecn` | 2-bit | Integer 0-3 | ECN (lower 2 bit của TOS byte). 0=not-ECT, 1=ECT(1), 2=ECT(0), 3=CE (congestion) |
| `nw_ttl` | 8-bit | Integer 0-255 | IPv4 TTL. Decremented mỗi hop |

**Prerequisites:** `eth_type=0x0800` (IPv4) hoặc `eth_type=0x86dd` (IPv6 dùng HL + Traffic Class thay TTL + TOS).

**Action liên quan:**
- `mod_nw_tos`, `mod_nw_ecn`, `mod_nw_ttl`
- `dec_ttl` — giảm TTL/HL theo router semantics

## 4.8.7 Nhóm G, L3 IPv6 fields (7 field)

### 4.8.7.1 `ipv6_src`, `ipv6_dst`

| Attribute | Giá trị |
|---|---|
| **Name** | `ipv6_src` / `ipv6_dst` |

### 4.8.15.1 `in_phy_port` — Physical port sau tunnel decap

**Bucket:** Pipeline metadata.
**Bits:** 32-bit OF port number.
**Maskable:** Không.
**Writable:** Không (read-only metadata).
**Prerequisites:** Không.
**OF version intro:** OF 1.2+ (OXM_OF_IN_PHY_PORT).
**OVS version intro:** OVS 1.7+.
**NXM/OXM:** OXM_OF_IN_PHY_PORT (class 0x8000, type 1).
**Example:**

```
priority=100,in_port=2,in_phy_port=5,actions=goto_table:1
```

**Diễn giải vận hành:** Khi packet đến từ tunnel port (Geneve/VXLAN/GRE), `in_port` trỏ vào tunnel port (logical), còn `in_phy_port` trỏ vào physical NIC port mà tunnel chạy lên trên. Cho phép policy phân biệt "packet đến qua tunnel A nhưng underlay đi qua NIC eth0" vs "qua eth1". Hữu ích cho debug topology multi-NIC + multi-tunnel hoặc tracking source physical port khi audit security.

### 4.8.15.2 `ipv6_flabel` — IPv6 Flow Label

**Bucket:** L3 IPv6 QoS.
**Bits:** 20-bit (low 20 bits của 32-bit field IPv6 header).
**Maskable:** Không.
**Writable:** Không (read-only).
**Prerequisites:** `eth_type=0x86dd` (IPv6).
**OF version intro:** OF 1.2+ (OXM_OF_IPV6_FLABEL).
**OVS version intro:** OVS 1.11+.
**NXM/OXM:** OXM_OF_IPV6_FLABEL (class 0x8000, type 28).
**Example:**

```
priority=100,ipv6,ipv6_flabel=0x12345,actions=output:2
```

**Diễn giải vận hành:** Flow Label (RFC 6437) được sender set để identify một flow IPv6, primary use case là **ECMP hashing** trên router/switch trung gian. Match `ipv6_flabel` cho phép policy theo flow ID mà không cần inspect 5-tuple. Rất hữu ích cho QoS classification (ví dụ flow_label = 0x100000 → premium queue) hoặc traffic engineering (force flow đi qua path cụ thể). Khác `ipv4`: IPv4 không có Flow Label, phải hash 5-tuple cho ECMP. Đây là một trong những lý do IPv6 cải thiện performance core network.

### 4.8.15.3 `ipv6_exthdr` — IPv6 Extension Headers bitmap

**Bucket:** L3 IPv6.
**Bits:** 9-bit (mã hóa trong 16-bit field).
**Maskable:** Có (CIDR-style mask).
**Writable:** Không.
**Prerequisites:** `eth_type=0x86dd`.
**OF version intro:** OF 1.2+ (OXM_OF_IPV6_EXTHDR).
**OVS version intro:** OVS 1.11+.
**NXM/OXM:** OXM_OF_IPV6_EXTHDR (class 0x8000, type 39).
**Example:**

```
priority=200,ipv6,ipv6_exthdr=0x4/0x4,actions=drop
```

**Diễn giải vận hành:** Bitmap 9 bit báo IPv6 extension header nào hiện diện (Hop-by-Hop, Routing, Fragment, Destination Options, AH, ESP, etc.) — chi tiết bit assignment xem section 4.8.7.3. Use case chính: **security policy block packet với extension header bất thường** (Routing Type 0 đã bị deprecated do RH0 attack, nhưng vẫn có thể xuất hiện). Match `ipv6_exthdr=0x4/0x4` (bit 2 = Fragment) drop tất cả IPv6 fragmented traffic — biện pháp chống fragmentation-based evasion. OVS implement bằng cách parse extension header chain ở slow path; không phải mọi NIC offload Geneve/VXLAN cũng support extension header parsing.

### 4.8.15.4 `pbb_isid` — Provider Backbone Bridge I-SID

**Bucket:** L2 Provider Bridge (802.1ah).
**Bits:** 24-bit Service Instance ID.
**Maskable:** Có.
**Writable:** Có (`set_field:0x123456->pbb_isid`).
**Prerequisites:** PBB encapsulation (`eth_type=0x88e7` MAC-in-MAC).
**OF version intro:** OF 1.4+ (OXM_OF_PBB_ISID, optional capability).
**OVS version intro:** OVS 2.6+.
**NXM/OXM:** OXM_OF_PBB_ISID (class 0x8000, type 37).
**Example:**

```
priority=100,eth_type=0x88e7,pbb_isid=0x012345,actions=output:2
```

**Diễn giải vận hành:** PBB (IEEE 802.1ah) là chuẩn carrier Ethernet cho service provider, cho phép encapsulate frame customer trong frame backbone với I-SID 24-bit định danh service instance. Cụ thể, frame backbone có thêm 22-byte header trước Ethernet customer: B-DA + B-SA + B-Tag (VLAN backbone) + I-Tag (chứa I-SID). Service provider dùng I-SID để map customer service tới topology backbone. Match `pbb_isid` cho phép switch core forward theo service ID thay vì VLAN customer (giải quyết vấn đề VLAN 4096 limit).

> **Phạm vi curriculum:** PBB là feature carrier rất chuyên biệt, hiếm gặp trong DC overlay (OVN không dùng PBB, dùng Geneve/VXLAN). Đưa vào catalog để cover OF spec đầy đủ, nhưng deployment thực tế OVS hiếm.

### 4.8.15.5 `mpls_tc` — MPLS Traffic Class

**Bucket:** MPLS QoS.
**Bits:** 3-bit TC (Traffic Class) field trong MPLS shim header.
**Maskable:** Không.
**Writable:** Có (`set_field:5->mpls_tc`).
**Prerequisites:** MPLS (`eth_type=0x8847` unicast hoặc `0x8848` multicast).
**OF version intro:** OF 1.2+ (OXM_OF_MPLS_TC).
**OVS version intro:** OVS 1.11+.
**NXM/OXM:** OXM_OF_MPLS_TC (class 0x8000, type 35).
**Example:**

```
priority=100,eth_type=0x8847,mpls_label=100,mpls_tc=5,actions=set_queue:1,output:2
```

**Diễn giải vận hành:** MPLS shim header (4 byte) chứa: 20-bit Label + 3-bit TC + 1-bit BoS + 8-bit TTL. Field TC (cũng gọi "EXP" trong RFC cũ) dùng cho QoS marking, mapping với DiffServ class. Match `mpls_tc` cho phép switch core ưu tiên traffic theo class (TC=5 = EF Expedited Forwarding cho voice). Action `set_field:N->mpls_tc` rewrite trên ingress edge router để mark packet vào MPLS domain.

### 4.8.15.6 `mpls_bos` — MPLS Bottom of Stack

**Bucket:** MPLS.
**Bits:** 1-bit (0 = có label sau, 1 = label cuối stack).
**Maskable:** Không.
**Writable:** Không (read-only, tự suy ra từ stack depth).
**Prerequisites:** MPLS.
**OF version intro:** OF 1.3+ (OXM_OF_MPLS_BOS).
**OVS version intro:** OVS 1.11+.
**NXM/OXM:** OXM_OF_MPLS_BOS (class 0x8000, type 36).
**Example:**

```
priority=100,eth_type=0x8847,mpls_bos=1,actions=pop_mpls:0x0800,goto_table:5
```

**Diễn giải vận hành:** MPLS có thể stack nhiều label (label switching path cùng tunnel). Bit BoS = 1 nghĩa là label hiện tại là cuối stack, packet sau pop label này sẽ trở thành Ethernet/IP gốc. Match `mpls_bos=1` thường dùng ở **edge router** để biết khi nào pop hết label + revert về L2/L3 gốc. Action `pop_mpls:<ethertype>` chỉ làm việc khi BoS=1 (pop label cuối + reset eth_type về tham số cho).

### 4.8.15.7 `sctp_src` / `sctp_dst` — SCTP source/destination port

**Bucket:** L4 Transport.
**Bits:** 16-bit each.
**Maskable:** Có (range mask).
**Writable:** Có (`set_field:5060->sctp_dst`).
**Prerequisites:** SCTP (`eth_type=0x0800/0x86dd`, `ip_proto=132`).
**OF version intro:** OF 1.2+ (OXM_OF_SCTP_SRC/DST).
**OVS version intro:** OVS 2.0+.
**NXM/OXM:** OXM_OF_SCTP_SRC (type 17) + OXM_OF_SCTP_DST (type 18).
**Example:**

```
priority=100,ip,ip_proto=132,sctp_dst=2905,actions=output:2
```

**Diễn giải vận hành:** SCTP (RFC 4960) là protocol L4 thứ 3 (sau TCP+UDP), dùng chủ yếu cho telecom signaling (Diameter, M3UA, SIGTRAN). Mặc dù không phổ biến trong DC, telecom workload (5G core, IMS) chạy trên container OpenStack có thể cần OVN policy match SCTP. SCTP có ưu điểm multistreaming + multihoming + heartbeat built-in, nhưng OVN ACL hiện chỉ support port-based matching (không inspect SCTP chunk).

### 4.8.15.8 `tunnel_id` (alias `tun_id`) — Tunnel ID 64-bit

**Bucket:** Tunnel metadata.
**Bits:** 64-bit.
**Maskable:** Có.
**Writable:** Có (`set_field:0x5678->tun_id`).
**Prerequisites:** Không (luôn available, là 0 nếu packet không từ tunnel).
**OF version intro:** OF 1.3+ (OXM_OF_TUNNEL_ID).
**OVS version intro:** OVS 1.1+ (NXM_NX_TUN_ID), OVS 2.4+ (OXM_OF_TUNNEL_ID).
**NXM/OXM:** NXM_NX_TUN_ID (class 0x0001, type 16) hoặc OXM_OF_TUNNEL_ID (class 0x8000, type 38).
**Example:**

```
priority=100,tun_id=0x1234,actions=set_field:0x5->reg0,goto_table:8
```

**Diễn giải vận hành:** `tunnel_id` là 64-bit metadata gắn liền với packet đến từ tunnel port. Encoding tùy theo tunnel type:
- **VXLAN**: 24-bit VNI nằm trong 24 bit thấp; 40 bit cao = 0
- **Geneve**: 24-bit VNI tương tự VXLAN
- **GRE với key option**: 32-bit GRE key trong 32 bit thấp
- **STT**: 64-bit context ID đầy đủ

Khi packet rời br-int qua tunnel port, ovs-vswitchd encapsulate với VNI/key = giá trị `tunnel_id` được set bởi flow rule trước output. OVN dùng `tunnel_id` để gán datapath ID logical (logical switch hoặc router), kết hợp với reg14/reg15 (inport/outport) làm full pipeline metadata. Đây là cốt lõi cross-chassis transition trong OVN.

### 4.8.15.9 `xreg0` đến `xreg7` — Extended 64-bit registers

**Bucket:** Pipeline state register.
**Bits:** 64-bit each (8 register).
**Maskable:** Có.
**Writable:** Có.
**Prerequisites:** Không.
**OF version intro:** OF 1.3+ (OXM_OF_PKT_REG x).
**OVS version intro:** OVS 2.4+.
**NXM/OXM:** OXM_OF_PKT_REG0..7 (class 0x0001, type 0..7).
**Example:**

```
priority=100,actions=load:0xCAFEBABE12345678->xreg0,goto_table:5
table=5,xreg0=0xCAFEBABE12345678,actions=output:2
```

**Diễn giải vận hành:** `xreg0..xreg7` là alias 64-bit cho cặp register 32-bit liên tiếp (xreg0 = reg0+reg1, xreg1 = reg2+reg3, etc.). Chỉ 8 xreg vì có 16 reg vật lý. Cho phép match/set giá trị 64-bit nguyên khối (ví dụ MAC address 48-bit fit trong 64-bit register). OVN dùng xreg cho IPv4 nexthop scratch (4 byte không đủ trong 32-bit xreg2 + Mac 48-bit).

### 4.8.15.10 `xxreg0` đến `xxreg3` — Extended 128-bit registers

**Bucket:** Pipeline state register.
**Bits:** 128-bit each (4 register).
**Maskable:** Có.
**Writable:** Có.
**Prerequisites:** Không.
**OF version intro:** OVS extension (không phải standard OF).
**OVS version intro:** OVS 2.6+.
**NXM/OXM:** Custom OVS extension (no class assigned in OF spec).
**Example:**

```
priority=100,actions=load:0x20010db8000000000000000000000001->xxreg0,goto_table:5
```

**Diễn giải vận hành:** `xxreg0..xxreg3` là alias 128-bit cho 4 reg 32-bit liên tiếp (xxreg0 = reg0+reg1+reg2+reg3, xxreg1 = reg4+reg5+reg6+reg7, etc.). Đặc biệt hữu ích cho **IPv6 address scratch** (128-bit fit nguyên xxreg). OVN dùng xxreg0 cho IPv6 ND/NA processing (xem Phần 13.17 §13.17.2). Cẩn thận: ghi vào xxreg0 sẽ overwrite reg0+1+2+3 đồng thời, mất bất kỳ REGBIT đã set trong reg0.

### 4.8.15.11 NSH fields — `nsh_spi`, `nsh_si`, `nsh_c1` đến `nsh_c4`

**Bucket:** Service Function Chaining (NSH RFC 8300).
**Bits:** `nsh_spi` 24-bit, `nsh_si` 8-bit, `nsh_c1`-`nsh_c4` 32-bit each.
**Maskable:** Có cho `nsh_c1`-`nsh_c4`.
**Writable:** Có.
**Prerequisites:** NSH (`eth_type=0x894f` hoặc `packet_type=(1, 0x894f)` với OF 1.5+ packet-type-aware).
**OF version intro:** OF 1.5+ với extension experimenter (NSH chưa standardize trong OF base).
**OVS version intro:** OVS 2.8+.
**NXM/OXM:** Experimenter class với NSH-specific type.
**Example:**

```
priority=100,nsh,nsh_spi=0x123,nsh_si=255,actions=output:2
priority=100,nsh,nsh_si=128,actions=set_field:0xdeadbeef->nsh_c1,goto_table:5
```

**Diễn giải vận hành:** NSH (Network Service Header, RFC 8300) là protocol cho **Service Function Chaining (SFC)** — chuỗi packet qua nhiều dịch vụ (firewall → IDS → load balancer → DPI). NSH header có:
- **SPI (Service Path Identifier)** 24-bit: định danh chain
- **SI (Service Index)** 8-bit: vị trí hiện tại trong chain (decrement mỗi service hop)
- **C1-C4** (Context Headers) 32-bit each: metadata service-specific (ví dụ tenant ID, application class)

Use case: cloud provider deploy SFC cho tenant — packet vào SPI=1 (chain "web tenant A") đi qua SI=255 (firewall) → SI=254 (IDS) → SI=253 (output). Mỗi service decrement SI và forward về service tiếp. NSH thay vì hardcode topology, định nghĩa overlay logic độc lập với physical path.

NSH trong curriculum chỉ giới thiệu nông để cover OF spec đầy đủ. Triển khai OVN production thường KHÔNG dùng NSH (chọn cách khác đơn giản hơn như chained logical router).

### 4.8.15.12 `packet_type` — Packet type-aware pipeline (OF 1.5+)

**Bucket:** Pipeline metadata.
**Bits:** 32-bit (16-bit namespace + 16-bit type).
**Maskable:** Không.
**Writable:** Có (action `decap` set lại).
**Prerequisites:** Không (luôn available, default = (0, 0) Ethernet).
**OF version intro:** OF 1.5+ (OXM_OF_PACKET_TYPE).
**OVS version intro:** OVS 2.8+.
**NXM/OXM:** OXM_OF_PACKET_TYPE (class 0x8000, type 44).
**Example:**

```
# Match packet IPv4 trần (no Ethernet)
priority=100,packet_type=(1,0x0800),ip_proto=6,actions=encap(ethernet),output:2
```

**Common values:**

| `packet_type` | Encapsulation |
|---------------|---------------|
| `(0, 0)` | Ethernet (default) |
| `(0, 0x8847)` | MPLS unicast |
| `(0, 0x86dd)` | IPv6 |
| `(1, 2)` | IPv4 bare (no Ethernet) |
| `(1, 0x894f)` | NSH |

**Diễn giải vận hành:** OF 1.0-1.4 giả định mọi packet có Ethernet header. OF 1.5+ với packet-type-aware pipeline cho phép match + xử lý packet không-Ethernet (bare IP, NSH, MPLS-only). Match `packet_type=(1,2)` cho phép xử lý packet IPv4 trần (ví dụ output từ IPSec tunnel với mode transport). Action `encap(ethernet)` thêm Ethernet header trở lại để output ra port thường. Use case chính: SFC NSH integration, IPSec gateway, GTP-U tunneling cho 5G.

## 4.8.14 Scenario cuối, lazy wildcarding thực nghiệm

**Bối cảnh:** Phần 9.2 §9.2.2 giới thiệu lazy wildcarding. Mục này demo trực tiếp bằng trace.

**Setup:**

```shell
$ sudo ovs-vsctl add-br br-lazy
$ sudo ovs-vsctl add-port br-lazy tap0 -- set Interface tap0 type=internal
$ sudo ovs-ofctl -O OpenFlow15 add-flow br-lazy \
    "table=0,priority=100,tcp,tp_dst=80,actions=output:LOCAL"
```

**Trace với gói tin TCP port 80:**

```shell
$ sudo ovs-appctl ofproto/trace br-lazy \
    "in_port=tap0,dl_src=aa:bb:cc:00:00:01,dl_dst=aa:bb:cc:00:00:02,\
     dl_type=0x0800,nw_src=10.1.0.5,nw_dst=10.1.0.10,nw_proto=6,\
     tp_src=50000,tp_dst=80"

Flow: tcp,in_port=1,vlan_tci=0x0000,dl_src=aa:bb:cc:00:00:01,
      dl_dst=aa:bb:cc:00:00:02,nw_src=10.1.0.5,nw_dst=10.1.0.10,
      nw_tos=0,nw_ecn=0,nw_ttl=0,tp_src=50000,tp_dst=80,tcp_flags=0

bridge("br-lazy")
-----------------
 0. tcp,tp_dst=80, priority 100
    output:LOCAL

Final flow: unchanged
Megaflow: recirc_id=0,eth,tcp,tp_dst=0x50/0xffff
Datapath actions: 65534
```

**Anatomy trace megaflow line:**

```
Megaflow: recirc_id=0,eth,tcp,tp_dst=0x50/0xffff
          ^^^^^^^^^^^  ^^^  ^^^ ^^^^^^^^^^^^^^^^^
          inspect=0    impl  impl exact match port 80
                       bởi   bởi
                       prereq prereq
                       eth   nw_proto=6
```

Field được kiểm tra (không wildcard):
- `recirc_id` — implicit (luôn inspected)
- `eth_type` — prereq của tcp
- `nw_proto` — prereq của tcp
- `tp_dst=0x50/0xffff` — rule match exact port 80

Field KHÔNG inspect (wildcard):
- `eth_src`, `eth_dst` — không match trong rule
- `nw_src`, `nw_dst` — không match
- `tp_src` — không match
- `tcp_flags`, `nw_ttl`, `nw_tos` — không match

Kernel install megaflow với match chỉ `tp_dst=80` + mask `0xffff`, wildcard tất cả field khác. Một megaflow cover infinite gói tin khác src_ip/src_port/MAC, miễn dst_port=80.

Đây là lazy wildcarding thực nghiệm, ghép nối Phần 9.2 §9.2.2 lý thuyết với OpenFlow field catalog của chính Phần này.

## Điểm cốt lõi cần nhớ

Thứ nhất, OpenFlow từ 12-tuple cứng (1.0) lên 100+ field extensible (1.5 + NXM) qua OXM TLV. Phần 4.1 kể lịch sử chuyển đổi; Phần này liệt kê catalog.

Thứ hai, 12 nhóm field foundation: Metadata / Register / Tunnel / L2 / ARP / IPv4 / IPv6 / L4 / ICMP / MPLS / Conntrack / packet_type. Engineer nhớ nhóm thay vì nhớ từng field — tra cứu `ovs-fields(7)` khi cần cụ thể.

Thứ ba, 9-attribute anatomy (Name / Width / Format / Masking / Prerequisites / Access / OpenFlow version / OXM-NXM / Semantics) là mental model chuẩn cho mỗi field. Khi debug flow rule không match, luôn kiểm chứng: prerequisite đã set? Mask đúng? Version OpenFlow support?

Thứ tư, register (reg0-reg15, xreg, xxreg) không thuộc gói tin wire — chúng là scratch pad per-gói tin trong pipeline. OVN chia register map cứng (reg13=datapath, reg14=ingress port, reg15=egress port), engineer OVN phải thuộc.

Thứ năm, conntrack field (ct_state, ct_zone, ct_mark, ct_label, ct_nw_*) chỉ match sau khi gói tin đã qua `ct()` action. Đây là khác biệt quan trọng với stateless match — không có ct-prefix field nào được kiểm tra cho gói tin chưa tracked.

Thứ sáu, conjunctive match (`conj_id` + action `conjunction()`) compress cross-product rule N×M → N+M. OVN ACL Port_Group dùng pattern này extensively — hiểu conjunction giúp đọc `ovn-sbctl lflow-list` mà không hoang mang trước flood of `conjunction(X, Y/Z)`.

## 11.4.0 Opening context, from GRE plaintext to IPsec encrypted

Part 11.3 Guided Exercise 12 demonstrated decisively that a GRE tunnel is NOT secure: an eavesdropper on an intermediate router uses Wireshark to capture the inner HTTP payload as plaintext, including a password in the response body. In a real production environment, this risk is unacceptable for almost every scenario:

- Site-to-site VPN between bank branches over the Internet.
- Multi-datacenter replication with customer data.
- Cloud-hybrid overlay linking on-premises with AWS or Azure.
- Compliance frameworks that require encryption-at-wire (HIPAA, PCI-DSS, GDPR).

Crichigno Lab 15 solves this problem by adding an IPsec layer on top of GRE. The topology is identical to Lab 14 (2 Docker containers plus 3 FRR routers plus OSPF area 0). Only one detail differs: when the OVS tunnel is configured, add `options:psk=password` so the `ovs-monitor-ipsec` daemon automatically generates strongSwan configuration and establishes the IPsec SA.

The Wireshark result on the intermediate router changes completely:

```
Before (Lab 14 GRE plaintext):              After (Lab 15 GRE plus IPsec):
┌───────────────────────┐                  ┌───────────────────────┐
│ Outer IP              │                  │ Outer IP              │
├───────────────────────┤                  ├───────────────────────┤
│ GRE header            │                  │ ESP header (SPI, seq) │
├───────────────────────┤                  ├───────────────────────┤
│ Inner IP              │ ← plaintext      │ ESP encrypted payload │ ← ciphertext
├───────────────────────┤                  │   (inner IP plus GRE  │
│ TCP/HTTP plaintext    │                  │    plus inner TCP and │
│   "SECRET: ..."       │                  │    HTTP)              │
└───────────────────────┘                  │   [unreadable]        │
                                           ├───────────────────────┤
                                           │ ESP trailer plus ICV  │
                                           └───────────────────────┘
```

The Wireshark ESP filter only shows the SPI and the sequence number. The entire payload is a block of ciphertext that cannot be read. This is the evidence that IPsec actually encrypts at the network layer, not at the application layer (SSL or TLS).

Today's part rebuilds Lab 15 in full and contrasts directly with Lab 14 to make clear the difference between "tunneling only" and "tunneling plus IPsec".

## 11.4.1 Why IPsec for the overlay

Overlay encapsulation provides tenant isolation but does **not encrypt**. Tenant packets exist in plaintext inside the outer IP. Anyone with access to the underlay can sniff and decode them.

IPsec is mandatory in two scenarios:

1. **Cloud hybrid:** the overlay traverses the untrusted Internet.
2. **Multi-tenant DC:** compliance frameworks (HIPAA, PCI-DSS) require encryption-at-wire.

IPsec encrypts the outer packet using ESP (Encapsulating Security Payload).

## 11.4.2 The ovs-monitor-ipsec daemon

Compass Chapter 12: "`ovs-monitor-ipsec` watches the `Interface.options:psk` or `Interface.options:remote_cert` columns and programs strongSwan or Libreswan accordingly."

Architecture:

1. The administrator configures the tunnel plus the IPsec option through `ovs-vsctl`.
2. The `ovs-monitor-ipsec` daemon subscribes to OVSDB and detects the change.
3. The daemon generates configuration for strongSwan and triggers a reload.
4. strongSwan establishes an IPsec SA with the remote peer.
5. OVS tunnel traffic flows through IPsec encapsulation.

The administrator never touches the strongSwan configuration directly.

Rotate the PSK with `ovs-vsctl set Interface vxlan-ipsec options:psk=newsecret`. The `ovs-monitor-ipsec` reload is atomic.

## 11.4.4 Setup with a certificate (production)

PSK has limitations: one key is shared across the whole fabric, so a compromise forces a rotation on every peer.

Certificate based: every endpoint has a certificate signed by a CA.

```bash
# CA:
[ca]# ovs-pki req+sign ipsec-node1 ipsec
[ca]# ovs-pki req+sign ipsec-node2 ipsec

# Host A:
[hostA]# cp ipsec-node1-cert.pem /etc/openvswitch/ipsec-cert.pem
[hostA]# cp ipsec-node1-privkey.pem /etc/openvswitch/ipsec-key.pem

[hostA]# ovs-vsctl add-port br0 vxlan-ipsec -- \
         set Interface vxlan-ipsec type=vxlan \
         options:remote_ip=10.0.0.2 options:key=flow \
         options:remote_cert=<host-B-cert-path>
```

Rotation: issue a new certificate without needing to sync a secret.

## 11.4.7 AH and ESP, why ESP wins

The original IPsec specification, RFC 2401 (1998), defines two protocols:

**AH (Authentication Header, RFC 4302):**
- Provides: authentication plus integrity plus anti-replay.
- Does NOT provide: confidentiality (no encryption).
- Protection scope: includes the outer IP header.
- IP protocol number: 51.

**ESP (Encapsulating Security Payload, RFC 4303):**
- Provides: confidentiality plus integrity plus authentication plus anti-replay.
- Protection scope: only the payload after the ESP header (does not include the outer IP).
- IP protocol number: 50.

Lab 15 Figure 1 (page 5) lists the protocols supported by OVS (highlighted in light blue):

```
IPsec Protocol
├── Confidentiality
│   ├── DES, 3DES, AES, SEAL
├── Integrity
│   ├── AH
│   ├── ESP
│   └── ESP plus AH
├── Authentication
│   ├── MD5, SHA
│   ├── PSK, RSA
│   └── Diffie-Hellman (DH1, DH2, ..., DH14)
```

### Why ESP wins over AH

In practice in 2026, ESP almost completely replaces AH. Three reasons:

**One, ESP already provides authentication.** The ESP-Auth field guarantees integrity equivalent to AH. Using AH separately just adds overhead.

**Two, AH does not survive NAT.** AH protects the outer IP header (including source and destination IP). NAT rewrites the IP header, AH detects the change, and rejects the packet. With an Internet that is full of NAT, AH cannot be used in practice.

**Three, ESP has a tunnel mode that fully encapsulates.** ESP tunnel mode encapsulates both the inner IP header and the payload, exposing only the outer IP. Highest safety, NAT friendly.

Lab 15 §1.1 states clearly: *"ESP is much more widely used since it provides confidentiality in addition to authentication and data integrity."*

OVS IPsec only supports ESP tunnel mode. AH has no configuration option in `ovs-monitor-ipsec`.

## 11.4.8 IKE phase 1, Diffie-Hellman and ISAKMP

IKE (Internet Key Exchange) is the protocol that negotiates keys and establishes Security Associations (SA). It has two phases, performed through the ISAKMP protocol (RFC 2408, obsoleted by RFC 7296 IKEv2).

### Phase 1 goal

Establish a secure channel between the two peers in order to negotiate phase 2. Exchange a Diffie-Hellman key to create a shared secret without sending the key over the wire. Authenticate the identity of the two peers using PSK or RSA certificate.

### Diffie-Hellman key exchange

DH is a 1976 mathematical algorithm that lets two peers negotiate a shared secret over a public channel. The mechanism:

1. The two peers publicly agree on a prime `p` and a generator `g`.
2. Alice picks a random number `a`, computes `A = g^a mod p`, and sends A to Bob.
3. Bob picks a random number `b`, computes `B = g^b mod p`, and sends B to Alice.
4. Alice computes `shared = B^a mod p = g^(ab) mod p`.
5. Bob computes `shared = A^b mod p = g^(ab) mod p`.
6. Both arrive at the same `shared`. An eavesdropper sees A and B, but recovering `a` or `b` is hard because of the discrete log problem.

DH groups define the size of `p`:
- **DH2:** 1024 bits, broken (do not use in production).
- **DH5:** 1536 bits, deprecated.
- **DH14:** 2048 bits, baseline acceptable.
- **DH19/20:** ECP 256-bit and 384-bit, using elliptic curves, recommended for production.

strongSwan defaults to DH14 plus DH19, compatible with OVS IPsec.

### Phase 1 sequence

Lab 15 Figure 2 (page 6) illustrates the message chain:

```
Sender                                    Receiver
  │                                          │
  │ 1. Negotiate SA parameters               │
  │    (encryption, hash, DH group, auth)    │
  │─────────────────────────────────────────▶│
  │◀─────────────────────────────────────────│
  │                                          │
  │ 2. DH key exchange                       │
  │    (exchange public values A, B)         │
  │─────────────────────────────────────────▶│
  │◀─────────────────────────────────────────│
  │                                          │
  │ 3. Identity plus authentication          │
  │    (PSK hash or RSA signature)           │
  │─────────────────────────────────────────▶│
  │◀─────────────────────────────────────────│
  │                                          │
  │ ✓ ISAKMP SA established                  │
```

After phase 1, the two peers share a secret and have authenticated each other. The ISAKMP SA channel is then used to negotiate phase 2.

### Authentication methods

- **PSK (Pre-Shared Key):** a shared password between the two peers. Simple, suitable for a lab or a small deployment (fewer than 10 peers). A compromise forces rotation across the entire fabric.
- **RSA Signature:** each peer has a private and public key pair. A certificate signed by a CA verifies the identity. Suitable for a large-scale production environment, where certificate rotation does not require a shared-secret sync.

Lab 15 uses PSK (`password`) to keep the demo simple. Production environments must use RSA certificates.

## 11.4.9 IKE phase 2, IPsec SA and ESP

### Phase 2 goal

Establish the IPsec SA used to encrypt the actual data traffic. Each direction has its own SA (uni-directional), so the two peers have 2 SAs (Alice to Bob and Bob to Alice) for ESP.

### Phase 2 quick mode

Use the shared key from phase 1 to encrypt the phase 2 messages. Negotiate:

- **Encryption algorithm for ESP:** AES-128-CBC, AES-256-GCM, 3DES.
- **Authentication algorithm for ESP:** HMAC-SHA-256, HMAC-SHA-384.
- **SA lifetime:** default 3600 seconds or 4 GB of traffic, then a rekey renegotiation.

After phase 2, the IPsec SA is active. Every packet through the tunnel is encrypted by ESP.

### ESP header structure (Lab 15 §1.3 Figure 5)

```
┌──────────────────────────────────────────┐
│ Outer IP header                          │
├──────────────────────────────────────────┤
│ ESP header                               │
│   SPI (32-bit, Security Parameter Index) │
│   Sequence Number (32-bit)               │
├──────────────────────────────────────────┤
│ ESP Payload (encrypted)                  │
│   ┌──────────────────────────────────┐   │
│   │ Original IP header (inner)       │   │
│   ├──────────────────────────────────┤   │
│   │ GRE header (if GRE inside IPsec) │   │
│   ├──────────────────────────────────┤   │
│   │ TCP/UDP plus payload             │   │
│   └──────────────────────────────────┘   │
├──────────────────────────────────────────┤
│ ESP Trailer                              │
│   Padding plus Pad Length plus Next Hdr  │
├──────────────────────────────────────────┤
│ ESP ICV (Integrity Check Value)          │
│   HMAC signature over the entire ESP     │
└──────────────────────────────────────────┘
```

Two important fields in the ESP header:

- **SPI (Security Parameter Index):** a 32-bit identifier for the matching IPsec SA. The receiver uses the SPI to find the decryption key. Each direction uses a different SPI.
- **Sequence Number:** increments for every packet. Used for anti-replay (if the receiver sees a duplicate number, it rejects the packet).

The ESP trailer plus ICV guarantees integrity. Any change inside the ESP payload makes the ICV invalid, and the receiver rejects the packet.

## 11.4.11 Wireshark dissector, ISAKMP and ESP

Lab 15 §6 uses Wireshark on router r3 to observe the IKE negotiation and the ESP traffic.

### Capture on r3

```bash
r3$ wireshark &
# GUI: Capture, Interface r3-eth1, Start
```

### Filter ISAKMP (phase 1 negotiation)

In the filter box, enter `isakmp`. You only see packets while the two peers are negotiating (each tunnel start or rekey). Format (Lab 15 Figure 49, page 26):

```
Source        Destination    Protocol    Length   Info
192.168.1.10  192.168.2.10   ISAKMP      396      Identity Protection (Main Mode)
192.168.2.10  192.168.1.10   ISAKMP      388      Identity Protection (Main Mode)
192.168.1.10  192.168.2.10   ISAKMP      228      Identity Protection (Main Mode)
192.168.2.10  192.168.1.10   ISAKMP      196      Identity Protection (Main Mode)
192.168.1.10  192.168.2.10   ISAKMP      124      Identity Protection (Main Mode)
192.168.2.10  192.168.1.10   ISAKMP      124      Identity Protection (Main Mode)
```

Six messages match IKEv1 Main Mode (3 round trips). Pick a packet and expand:

```
Internet Security Association and Key Management Protocol
  Initiator SPI: 0x....
  Responder SPI: 0x....
  Version: 1.0
  Exchange type: Identity Protection (Main Mode)
  Flags: Encryption, ...
  Payloads:
    Security Association
      Proposal: IKE
        Transform: ENCR_AES_CBC, AUTH_HMAC_SHA2_256, DH_GROUP_14
```

The SA negotiation metadata is visible, but the key material is encrypted after the first exchange. An eavesdropper learns that the peer uses AES-CBC plus SHA-256 plus DH14, but not the key.

### Filter ESP (phase 2 data)

Enter `esp` in the filter:

```
Source        Destination    Protocol    Length   Info
192.168.1.10  192.168.2.10   ESP         180      ESP (SPI=0x12345678)
192.168.2.10  192.168.1.10   ESP         180      ESP (SPI=0x87654321)
192.168.1.10  192.168.2.10   ESP         180      ESP (SPI=0x12345678)
...
```

Pick an ESP packet and expand:

```
Encapsulating Security Payload
  ESP SPI: 0x12345678 (305419896)
  ESP Sequence: 42
  (encrypted data, 148 bytes of ciphertext, unreadable)
```

Lab 15 Figure 52 (page 27) emphasizes: *"You will not see the actual source (10.0.0.1) and destination IP address (10.0.0.4) since all the data are encrypted."*

This is the visual evidence that IPsec encrypts at the network layer. Unlike Lab 14 GRE plaintext, router r3 now sees only opaque ciphertext.

After a few seconds, the IKE negotiation completes. Check:

```bash
d1$ ipsec statusall
# Expected:
# Security Associations (1 up, 0 connecting)
#   ovs-<name>[1]: ESTABLISHED ...
#   ovs-<name>{1}: INSTALLED, TUNNEL, reqid 1, ESP ...
```

If you see "connecting" but it never reaches "up", check:
- The two peers have the same PSK.
- The strongSwan daemon runs on both ends.
- Network reachability between the peers is fine (ping 192.168.1.10 to 192.168.2.10).

### Step 4, Test connectivity

```bash
d2$ h3 ping 10.0.0.1
# Still succeeds (the tunnel is active)
```

### Step 5, Capture Wireshark ISAKMP plus ESP on r3

```bash
r3$ wireshark &
# Start capture on r3-eth1
# Filter: isakmp or esp
```

While Wireshark is capturing, trigger a rekey by recreating the tunnel (you may need to restart strongSwan):

```bash
d1$ ipsec restart
# Wait 10 seconds
```

Wireshark then captures:
- 6 ISAKMP packets (phase 1 Main Mode).
- After that, continuous ESP packets while ping continues.

### Step 6, Verify the payload is encrypted

On h1 (d1):

```bash
h1$ echo "SECRET: password123" > /tmp/secret.txt
h1$ python3 -m http.server 8000 --bind 10.0.0.1
```

On h3 (d2):

```bash
h3$ curl http://10.0.0.1:8000/secret.txt
# Output: SECRET: password123 (h3 reads it through the tunnel)
```

In Wireshark on r3, filter `esp`, and open the HTTP response packet. Expand:

```
Encapsulating Security Payload
  ESP SPI: 0x....
  ESP Sequence: ....
  Data: (encrypted, not dissected)
```

You **do not see** "SECRET: password123" anywhere in the packet. Compare with Lab 14 GRE plaintext (the password appeared directly).

### Lesson

IPsec really encrypts at the network layer. The standard pattern in 2026: every tunnel that crosses a public network must be wrapped in IPsec. OVS plus `ovs-monitor-ipsec` plus strongSwan provide an automated stack so the administrator only sets one option on the Interface.

## Guided Exercise 14, Refute "IPsec is not slow" using actual bandwidth measurement

**Goal:** Measure IPsec encryption performance compared with a tunnel without IPsec, and prove a 10 to 30 percent CPU overhead (depending on AES-NI). Understand why hardware acceleration is required for high-bandwidth production.

**Counter-hypothesis to refute:**

> *"IPsec encryption on a modern CPU does not affect tunnel performance. Actual bandwidth with and without IPsec is the same."*

### Preparation

The Lab 15 topology is already active. Install iperf3 on the hosts:

```bash
d1$ apt install -y iperf3
d2$ apt install -y iperf3
```

Check that the CPU has AES-NI:

```bash
grep -o aes /proc/cpuinfo | head -1
# Output: aes (AES-NI is present)
```

### Step 1, Baseline without IPsec (GRE only)

Remove the PSK from the tunnel:

```bash
d1$ ovs-vsctl remove Interface s1-tun options psk
d2$ ovs-vsctl remove Interface s2-tun options psk
```

Run actual bandwidth measurement with iperf3:

```bash
h2$ iperf3 -s
h1$ iperf3 -c 10.0.0.2 -t 30
```

Record the actual bandwidth. Example doc-plausible result: about 950 Mbps (close to the 1 Gbps line rate of the Mininet veth interface).

### Step 2, Enable IPsec

```bash
d1$ ovs-vsctl set Interface s1-tun options:psk=password
d2$ ovs-vsctl set Interface s2-tun options:psk=password
```

Wait 10 seconds for IKE to complete:

```bash
d1$ ipsec statusall | grep ESTABLISHED
```

Run iperf3 again:

```bash
h1$ iperf3 -c 10.0.0.2 -t 30
```

Record the actual bandwidth. Expect a drop to about 700 to 850 Mbps depending on the CPU (a 10 to 25 percent reduction).

### Step 3, Analyze the overhead

The counter-hypothesis is refuted: IPsec **slows the link** by about 10 to 25 percent in a CPU-only environment. The reasons:

- Every packet must be encrypted with AES-CBC or AES-GCM (16-byte block).
- HMAC-SHA-256 must compute the ICV.
- Sequence number management plus anti-replay check.
- All of this runs on the CPU in user space (strongSwan plus kernel netlink).

### Step 4, Verify AES-NI acceleration

With AES-NI (Intel and AMD CPUs since 2010), AES-CBC speeds up 3 to 10 times compared with pure software. Verification:

```bash
# Bench AES-NI
openssl speed -elapsed -evp aes-256-gcm
# Output: about 3 to 5 GB/s AES throughput on a single core
```

If the CPU lacks AES-NI (an old VM), IPsec overhead can climb to 50 to 70 percent. This is the reason every cloud provider exposes the AES-NI flag for VMs.

### Step 5, Hardware offload through SmartNIC

In a high-bandwidth production environment (25 Gbps and beyond), the CPU is not enough. The solution: offload IPsec to a SmartNIC:

- NVIDIA ConnectX-6 IPsec in-line offload.
- Intel XL710 plus E810 IPsec hardware acceleration.
- Chelsio T6 Unified Wire.

Check NIC support:

```bash
ethtool -k eth0 | grep ipsec
# Output (if supported): esp-hw-offload: on
```

With NIC offload, IPsec overhead drops nearly to zero, and actual bandwidth holds the 25 or 100 Gbps line rate.

### Lesson

IPsec is not free in CPU terms. Knowing the trade-off between security and performance matters when designing a cluster. For a regular VM, accept 10 to 25 percent overhead. For a high-bandwidth node (compute edge, storage), invest in SmartNIC offload. For an IoT or embedded device that lacks AES-NI, choose a lighter cipher (ChaCha20-Poly1305 is more software friendly than AES without hardware).

## 11.4.12 IPsec in an OVN cluster encryption

A production OVN deployment has the option to encrypt every tunnel between chassis with IPsec. Configuration:

```bash
# On every chassis
ovs-vsctl set Open_vSwitch . other_config:ovn-chassis-mac-mappings=...
ovs-vsctl set Open_vSwitch . external_ids:ovn-encap-ip=<chassis-IP>
ovs-vsctl set Open_vSwitch . external_ids:ovn-encap-type=geneve

# Enable IPsec for every Geneve tunnel
ovn-nbctl set NB_Global . ipsec=true
```

`ovn-northd` automatically generates tunnel configuration with `options:psk` for every Geneve port on every chassis. The `ovs-monitor-ipsec` daemon on each chassis establishes a full-mesh of IPsec SAs.

Scale considerations:
- A cluster of N chassis means N*(N-1)/2 IPsec SAs.
- 100 chassis means 4,950 SAs.
- Vanilla strongSwan handles up to about 10,000 SAs per node well; beyond that, tuning or hardware offload is required.

### Comparison with multi-DC OVN over WAN

For an OVN deployment that spans multiple datacenters over a WAN:

- **Intra-DC Geneve:** does not need IPsec (untrusted: no, latency: low).
- **Inter-DC Geneve over IPsec:** mandatory (untrusted: yes, latency: higher but acceptable).

Often the inter-DC choice is GRE outer instead of Geneve because the GRE header is more stable across transit ISPs.

## Key points to remember

**One.** IPsec provides three guarantees (confidentiality, integrity, authentication). GRE, VXLAN, and Geneve only encapsulate, without security. The standard site-to-site VPN pattern is GRE or Geneve inside IPsec.

**Two.** AH (protocol 51) and ESP (protocol 50) are the two IPsec protocols. ESP wins because it is NAT friendly and has authentication built in. OVS IPsec only supports ESP tunnel mode.

**Three.** IKE phase 1 establishes an ISAKMP SA through Diffie-Hellman key exchange plus PSK or RSA authentication. Phase 2 negotiates the IPsec SA for ESP. DH14 is the baseline. DH19 or DH20 (elliptic curve) is for production.

**Four.** OVS IPsec is enabled through `options:psk=...` or `options:remote_cert=...` on the Interface. The `ovs-monitor-ipsec` daemon subscribes to OVSDB, generates strongSwan configuration automatically, and the administrator never touches strongSwan directly. Just `systemctl enable --now openvswitch-ipsec` is enough.

**Five.** The ESP header has the SPI (which identifies the SA) plus a sequence number (for anti-replay). The ESP trailer has Padding plus Next Header. The ICV is the HMAC signature over the entire ESP packet. The Wireshark ESP filter only shows metadata. The payload cannot be read.

**Six.** IPsec on a CPU with AES-NI costs 10 to 25 percent of actual bandwidth. Without AES-NI, the cost can be 50 to 70 percent. Hardware offload through a SmartNIC (NVIDIA ConnectX-6, Intel E810) makes overhead nearly zero at 25 or 100 Gbps.

**Seven.** PSK suits a lab plus a small deployment (fewer than 10 peers). Certificate based (RSA, `ovs-pki`) is for large-scale production, where rotation does not require a shared-secret sync.

**Eight.** An OVN cluster can enable IPsec full-mesh through `ovn-nbctl set NB_Global . ipsec=true`. It scales to about 10,000 SAs per node with vanilla strongSwan. Beyond that, SmartNIC offload is required.

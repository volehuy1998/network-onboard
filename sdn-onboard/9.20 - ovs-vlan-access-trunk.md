## 9.20.1 VLAN — phân vùng switch vật lý thành switch logic

Tài liệu Lab 6 (Crichigno et al., USC WASTC 2021) trích dẫn định nghĩa chuẩn của Cisco *Catalyst 4500 Configuration Guide* 2018: *"A VLAN is a way to partition a single switch into multiple switches. Through VLANs, a group of devices on one or more LANs can be configured to communicate as if they were attached to the same wire."*

Bối cảnh lịch sử: trước VLAN, một switch vật lý = một broadcast domain. Khi công ty có nhiều bộ phận (kỹ thuật, marketing, kế toán) muốn tách biệt traffic nhưng lại chia sẻ infrastructure vật lý, lựa chọn duy nhất là mua nhiều switch, tốn kém và cứng nhắc. VLAN ra đời (chuẩn hóa IEEE 802.1Q năm 1998) cho phép tách một switch thành nhiều "virtual switch" cùng phần cứng, mỗi virtual switch có broadcast domain riêng.

Ba ứng dụng chính theo Seifert & Edwards (*The All-New Switch Book*, Wiley 2008):

**Tính linh hoạt cấu hình**: cấu hình bất kỳ tập user/server/device nào thành một closed group (VLAN) ngay cả khi họ ở các tòa nhà hoặc khu vực địa lý khác nhau. User không phải di chuyển vật lý để "gia nhập" VLAN.

**LAN security**: VLAN không forward traffic giữa các port thuộc VLAN khác nhau. Tức switch từ chối bridge frame VLAN 10 sang port VLAN 20 trừ khi có router trung gian. Cơ chế này chống eavesdropping thụ động và bogus transmission.

**User mobility**: khi user chuyển vị trí vật lý, switch mới có thể gán port vào cùng VLAN cũ, user thấy network context không đổi, IP không cần đổi. Đây là nền tảng cho 802.1X + dynamic VLAN assignment trong môi trường enterprise.

> **Chủ đề trọng tâm:** VLAN là công cụ phân vùng broadcast domain ở L2. Nó không cung cấp cơ chế routing giữa VLAN — đó là nhiệm vụ của router/L3 switch (hoặc OVN Logical Router, xem Phần 9.18 và 13.11). Nhầm lẫn giữa "VLAN isolation" và "subnet isolation" là nguồn gốc nhiều lỗi cấu hình network — VLAN là L2 concept, subnet là L3 concept.

## 9.20.2 Cấu trúc VLAN-tagged Ethernet frame (IEEE 802.1Q)

VLAN tag được chèn vào Ethernet frame sau field Source MAC, trước field EtherType/Length ban đầu. 4 byte tag gồm hai phần:

```
┌──────────┬──────────┬──────────┬───────────┬─────────────┬──────────────┐
│ Dst MAC  │ Src MAC  │  TPID    │    TCI    │   EtherType │   Payload    │
│ 6 byte   │ 6 byte   │ 0x8100   │  2 byte   │   (IP/ARP)  │              │
│          │          │ 2 byte   │           │   2 byte    │              │
└──────────┴──────────┴──────────┴───────────┴─────────────┴──────────────┘

                            TCI breakdown (16 bit):
                            ┌─────┬───┬──────────────┐
                            │ PCP │DEI│     VID      │
                            │ 3 b │1 b│    12 bit    │
                            └─────┴───┴──────────────┘
```

Bốn field trong TCI (Tag Control Information):

| Field | Kích thước | Ý nghĩa |
|--------|-----------|---------|
| **TPID** (Tag Protocol ID) | 16 bit | Luôn = `0x8100` cho 802.1Q; `0x88A8` cho 802.1ad Q-in-Q |
| **PCP** (Priority Code Point) | 3 bit | 0-7, mã QoS class (IEEE 802.1p) |
| **DEI** (Drop Eligible Indicator) | 1 bit | Trước 2011 gọi CFI — frame nào bật DEI = ưu tiên drop khi congestion |
| **VID** (VLAN ID) | 12 bit | 0-4095 — VID 0 = priority-tagged không VLAN, VID 4095 = reserved, VID 1 thường là default VLAN |

Range hữu dụng cho VLAN: 1-4094 (4094 VLAN tối đa trên một domain). Giới hạn này là lý do các cloud data center dùng VXLAN VNI 24-bit (16 triệu) thay vì VLAN, xem Phần 11.0.

`ovs-fields(7)` mô tả khớp với fields tương ứng: `dl_vlan` (VID), `dl_vlan_pcp` (PCP), `vlan_tci` (toàn bộ TCI). OVS có thể match, set, push, pop VLAN tag qua flow action (`mod_vlan_vid`, `push_vlan`, `pop_vlan`).

> **Hiểu sai:** "Tag VLAN tăng kích thước frame nên MTU phải tăng tương ứng."
>
> Thực tế: chuẩn Ethernet II MTU 1500 byte không bao gồm header Ethernet. Khi chèn 4 byte VLAN tag, header Ethernet mở rộng từ 14 byte → 18 byte, nhưng payload 1500 byte không đổi. Tổng frame length tăng từ 1518 → 1522 (trừ FCS). Switch hỗ trợ 802.1Q phải accept frame 1522 byte mà không fragmentation — đây là "baby giant" frame, được IEEE 802.3ac (1998) chuẩn hóa. Nếu switch không biết 802.1Q, frame 1522 có thể bị drop vì "oversized" — lỗi phổ biến khi trunk qua switch cũ.

## 9.20.3 Access port và trunk port trên OVS

Hai kiểu port cơ bản:

**Access port**, port chỉ phục vụ một VLAN. Frame đi vào không có tag được gán VLAN của port (native VLAN). Frame đi ra có tag VLAN sẽ bị strip (loại bỏ). Device nối với access port (host, VM, printer) không biết VLAN tồn tại, nó nhận frame Ethernet "thông thường".

Cấu hình trên OVS:

```bash
[host]# ovs-vsctl add-port br-s1 h1-port tag=10
```

Port `h1-port` giờ là access port của VLAN 10. Frame từ `h1` (không tag) đi vào sẽ được OVS gán VLAN 10 internal. Frame từ bridge đi ra port này sẽ bị strip VLAN tag trước khi gửi tới `h1`.

**Trunk port**, port phục vụ nhiều VLAN đồng thời. Frame đi qua trunk port phải có VLAN tag (trừ VLAN native được phép untagged qua trunk trong một số cấu hình). Trunk port dùng để nối giữa switch-switch hoặc switch-router.

Cấu hình trên OVS:

```bash
[host]# ovs-vsctl add-port br-s1 trunk-port trunks=10,20
```

Port `trunk-port` chỉ chấp nhận frame với VLAN tag = 10 hoặc 20. Frame VLAN khác bị drop. Frame không tag cũng bị drop (trừ khi thêm `tag=N` làm native VLAN của trunk).

Một port không thể vừa là access vừa là trunk, phải chọn một. Mặc định (không set `tag` hay `trunks`) port là "VLAN-agnostic", pass mọi frame có hoặc không tag, hành vi giống hub.

> **Chủ đề trọng tâm:** Access port và trunk port là hai implementation của cùng một abstraction "port assigned to VLAN". Access port phục vụ device chỉ biết một VLAN (host, VM), trunk port phục vụ infrastructure peer cần hiểu nhiều VLAN (switch khác, router, firewall). Trong OVS, sự khác biệt là `tag=N` (access) vs `trunks=N,M` (trunk).

## 9.20.4 Topology Lab 6 — bốn host / hai switch VLAN

Lab 6 dùng topology chuẩn để demo access + trunk:

```
             Trunk port                         Trunk port
             VLAN 10, 20                        VLAN 10, 20
                 │                                    │
   ┌─────────────┴────────┐          ┌────────────────┴──────┐
   │        br-s1         │          │          br-s2        │
   │      (switch 1)      │◀────────▶│       (switch 2)      │
   │                      │          │                       │
   │ port 1      port 2   │          │  port 1      port 2   │
   │ (tag=10)    (tag=20) │          │  (tag=10)    (tag=20) │
   └────┬─────────┬───────┘          └──────┬─────────┬──────┘
        │         │                         │         │
      ┌─┴─┐     ┌─┴─┐                     ┌─┴─┐     ┌─┴─┐
      │h1 │     │h2 │                     │h3 │     │h4 │
      │V10│     │V20│                     │V10│     │V20│
      │.1 │     │.1 │                     │.2 │     │.2 │
      └───┘     └───┘                     └───┘     └───┘
    10.0.0.1  20.0.0.1                  10.0.0.2  20.0.0.2
```

Thiết kế: h1+h3 ở VLAN 10 (subnet 10.0.0.0/24), h2+h4 ở VLAN 20 (subnet 20.0.0.0/24). Hai switch được nối qua trunk port mang cả hai VLAN. Kỳ vọng:

- h1 ping h3 thành công (cùng VLAN 10, xuyên qua trunk)
- h2 ping h4 thành công (cùng VLAN 20, xuyên qua trunk)
- h1 không thể ping h2 hoặc h4 (khác VLAN, cần router)

Không có router trong topology, nếu muốn inter-VLAN routing, phải thêm Logical Router (OVN) hoặc OVS L3 flow (Phần 9.18).

Chuyển cấu hình để test: gán h2 IP `10.0.0.100/24` (cùng subnet với h1) tạm thời:

[host]# ip netns exec ns-h2 ip addr add 10.0.0.100/24 dev h2-port

## 9.20.5 Đối chiếu VLAN truyền thống với OVN Logical Switch

VLAN hoạt động tốt trong một data center nhỏ nhưng gặp hai giới hạn khi scale:

**Giới hạn 1 — 4094 VLAN tối đa**: 12-bit VID chỉ cho phép 4096 VID (trừ 0 và 4095 reserved → 4094 sử dụng được). Data center cloud với 10.000+ tenant không đủ VLAN cho mỗi tenant.

**Giới hạn 2, broadcast domain vẫn L2**: VLAN trải rộng qua trunk sẽ broadcast qua toàn bộ switch có VLAN đó. Với 1000 host cùng VLAN, ARP broadcast gây nghẽn underlay.

OVN Logical Switch giải quyết cả hai:

- Dùng `tunnel_key` 24-bit (Geneve) → 16 triệu VNI thay vì 4094 VLAN
- BUM suppression (Phần 18.0): ARP Responder cài sẵn trong pipeline, broadcast ARP không thực sự flood toàn overlay

Khi deploy OVN, người vận hành không config VLAN thủ công. Thay vào đó định nghĩa Logical Switch (`ovn-nbctl ls-add tenant-A-ls`) và gán port vào. OVN tự sinh Logical Flow và tự chọn tunnel_key. VLAN vẫn có thể dùng trong OVN cho localnet port (nối overlay với physical VLAN), nhưng nó là bridge giữa hai thế giới, không phải primary isolation mechanism.

> **Production readiness assessment:** OVS VLAN access/trunk phù hợp cho:
> - Integration với physical fabric VLAN (kế thừa hạ tầng)
> - Lab và testing workshop
> - Deployment nhỏ (<50 tenant)
>
> Khi scale vượt 50 tenant hoặc cần stateful (SNAT, LB, ACL stateful), chuyển sang OVN Logical Switch — xem Phần 13.1 và 13.2.

## 9.20.X `vlan_mode` 4 type source-level

OVS Port có 4 vlan_mode setting (Phần `ovs-vsctl(8)` man page §"VLAN Configuration"):

| Mode | Behavior tag input | Behavior tag output |
|------|-------------------|---------------------|
| `access` | Drop tagged, accept untagged | Strip tag (untagged) |
| `trunk` | Accept all tagged in `trunks` | Keep tag |
| `native-tagged` | Accept tagged, untagged → tag with `tag` value | Keep tag |
| `native-untagged` | Accept tagged, untagged → tag with `tag` value | Strip tag for VLAN match `tag` |

### Source code anchor (OVS v2.17.9)

[`ofproto/bond.c`](https://github.com/openvswitch/ovs/blob/v2.17.9/ofproto/bond.c) + [`ofproto/ofproto-dpif-xlate.c`](https://github.com/openvswitch/ovs/blob/v2.17.9/ofproto/ofproto-dpif-xlate.c).

Function `xlate_normal()` in `ofproto-dpif-xlate.c` xử lý VLAN logic cho NORMAL action:

```c
// Pseudo-code
static void xlate_normal(struct xlate_ctx *ctx) {
    struct flow *flow = &ctx->xin->flow;
    
    // Input VLAN classification
    if (port->vlan_mode == VLAN_MODE_ACCESS) {
        if (flow->vlan_tci != 0) {
            // Tagged on access port → drop
            xlate_normal_drop(ctx);
            return;
        }
        flow->vlan_tci = htons(port->tag | VLAN_CFI);  // tag with port.tag
    } else if (port->vlan_mode == VLAN_MODE_TRUNK) {
        if (!bitmap_is_set(port->trunks, vlan_tci_to_vid(flow->vlan_tci))) {
            // VLAN not in trunks set → drop
            xlate_normal_drop(ctx);
            return;
        }
    }
    // ... continue forwarding
}
```

### push_vlan / pop_vlan action

OpenFlow 1.1+ thêm action explicit:

- `push_vlan:0x8100`: push 802.1Q tag (Ethertype 0x8100). Sau push, set VID via `set_field`.
- `push_vlan:0x88a8`: push 802.1ad outer tag (QinQ).
- `pop_vlan`: remove outermost VLAN tag.

```bash
[host]# ovs-ofctl add-flow br-test \
        "ip,actions=push_vlan:0x8100,set_field:0x100a->vlan_tci,output:1"
# Push VLAN với VID=10 (0x000a) + CFI=1 (0x1000) = 0x100a
```

Source: [`lib/ofp-actions.c parse_push_vlan()`](https://github.com/openvswitch/ovs/blob/v2.17.9/lib/ofp-actions.c) + execute trong `ofproto-dpif-xlate.c`.

## 9.20.Y QinQ (802.1ad) deep

QinQ là VLAN stacking: outer tag (S-VLAN, service provider) + inner tag (C-VLAN, customer). Cho phép service provider transport multiple customer VLAN qua infrastructure shared.

### Header layout

```
+--------+--------+--------+---------+--------+--------+------+
| DstMAC | SrcMAC | TPID   | TCI     | TPID   | TCI    | Eth  | Data
| 6B     | 6B     | 0x88a8 | S-VLAN  | 0x8100 | C-VLAN | Type |
|        |        |        | 16-bit  |        | 16-bit | 2B   |
+--------+--------+--------+---------+--------+--------+------+
                  ^                  ^
                  Outer tag          Inner tag
                  (TPID 0x88a8)      (TPID 0x8100)
```

Frame length: 1518 + 4 (S-VLAN) + 4 (C-VLAN) = 1526 byte. Switch trên path phải support frame size tối thiểu 1526 byte (jumbo MTU 1512+ recommended).

### OVS QinQ config

```bash
# OVS port stage S-VLAN trunk
[host]# ovs-vsctl add-port br0 trunk-port \
        -- set Port trunk-port vlan_mode=dot1q-tunnel \
        -- set Port trunk-port tag=100 \
        -- set Port trunk-port other_config:qinq-ethtype=802.1ad
```

`vlan_mode=dot1q-tunnel` (OVS 2.10+): port nhận untagged frame từ S-VLAN domain với inner C-VLAN tag → push S-VLAN với TPID 0x88a8. Output ngược lại: pop S-VLAN.

Source code: `ofproto-dpif-xlate.c` function `xlate_normal()` check `vlan_mode=dot1q-tunnel` → execute `push_vlan(0x88a8)` + `pop_vlan` đối xứng.

```bash
[host]# ovs-appctl ofproto/trace br-s1 \

---------------

    NORMAL
     >> input port veth-h1 in mode "access" with tag 10
     >> learned that 02:00:00:00:00:01 is on port veth-h1
     >> no learned MAC for destination, flooding

```

Output cho thấy:

- Input port mode: `access` với tag 10.
- VLAN classification: untagged frame → tag với VID=10.
- Forwarding: flood with VLAN tag 10 (output: 3, 4, 5 = trunk ports với VLAN 10).
- Datapath action: `push_vlan(vid=10,...)` chèn tag, output multi-port.

**Kịch bản bẻ gãy:**

- Trace cho thấy "input dropped, frame is VLAN tagged but port has no VLAN configured": port `vlan_mode=access` nhận tagged frame. Kiểm chứng peer config.
- Frame qua trunk nhưng VID không trong `trunks` list: frame drop. Add VID vào trunks: `ovs-vsctl set Port trunk-port trunks=10,20,30`.
- QinQ outer tag không bị strip ở receiver: receiver không có `dot1q-tunnel` mode. Misconfig.

## 9.20.W Capstone POE: native-tagged vs native-untagged

> **Bối cảnh:** Kỹ sư config trunk port nối switch Cisco với OVS bridge. Cisco trunk có "native VLAN 1 untagged". OVS chọn `native-tagged` hay `native-untagged`?

**Tuyên bố cần kiểm chứng:** "OVS `native-tagged` tương thích với Cisco native VLAN. `native-untagged` không cần."

**Predict:** Sai. Cisco native VLAN gửi untagged frame cho VID native; OVS phải dùng `native-untagged` để mirror behavior này.

**Observe (giả định):**

- Setup OVS port trunk với `native-tagged tag=1 trunks=1,10,20`. Cisco peer trunk với native VLAN 1.
- Cisco gửi frame VLAN 1 (untagged): OVS nhận, classify untagged → tag với `tag=1` (native VLAN). OK.
- OVS gửi frame VLAN 1 ra trunk: `native-tagged` mode keep tag → frame có 802.1Q tag VID=1.
- Cisco nhận frame VID=1 (tagged) trên trunk: Cisco expect untagged cho native → confused. Some Cisco switch drop, some accept.

Đổi sang `native-untagged`:

- OVS gửi frame VLAN 1 ra trunk: `native-untagged` strip tag → frame untagged.
- Cisco nhận untagged → classify VLAN 1 (native) → forward. Healthy.

**Explain:**

1. **Cisco trunk default behavior**: native VLAN frame untagged on wire (RFC 802.1Q practice).
2. **OVS `native-tagged`**: keep tag cho native VLAN. Khác Cisco semantics.
3. **OVS `native-untagged`**: strip tag cho native VLAN, match Cisco. Đúng cho interop.
4. **Pure trunk (no native)**: dùng `trunk` mode, không có native concept.

**Falsification test:**

- Capture frame OVS → Cisco với `native-tagged`. Kiểm chứng frame có tag VID=1 hay không.
- Nếu Cisco accept tagged → counter-example. Trong thực tế, Cisco default reject tagged native (security feature).

**Tuyên bố sai ở đâu:** Tên `native-tagged` không đồng nghĩa "support Cisco native VLAN". Đúng phải: **Cisco interop = `native-untagged`. `native-tagged` cho legacy non-standard switch ép tag mọi frame.**

> **Quy tắc:** Cisco/Juniper/Arista trunk peer = OVS `native-untagged` (default native untagged). Ép tag mọi frame = `native-tagged` (rare, non-standard).

## Điểm cốt lõi cần nhớ

Thứ nhất, VLAN là cơ chế phân vùng broadcast domain ở L2, tạo nhiều "switch logic" trên một switch vật lý. Không phải L3 isolation, vẫn cần router để inter-VLAN.

Thứ hai, frame 802.1Q có 4 byte tag (TPID 0x8100 + TCI 16-bit), đẩy frame size tối đa từ 1518 lên 1522 byte. Switch không hỗ trợ 802.1Q có thể drop frame này.

Thứ ba, access port phục vụ một VLAN (`tag=N`) và strip tag khi gửi tới host. Trunk port phục vụ nhiều VLAN (`trunks=N,M`) và giữ tag khi forward. Một port không thể vừa access vừa trunk.

Thứ tư, VID 12-bit giới hạn 4094 VLAN, không đủ cho cloud scale. OVN Logical Switch dùng Geneve tunnel_key 24-bit (16 triệu ID) để vượt giới hạn này. VLAN và OVN không mâu thuẫn — VLAN dùng ở physical side (localnet port), Logical Switch dùng ở overlay side.

## 9.20.X So sánh với hệ khác (axis 20 backfill 2026-04-27)

VLAN access/trunk semantics có cross-vendor analogue:

- **Cisco IOS** (Catalyst series): `switchport mode access vlan N` cho access port, `switchport mode trunk` + `switchport trunk allowed vlan add 10,20` cho trunk. Native VLAN concept: `switchport trunk native vlan 1` (default). Frame trên native VLAN xuất ra trunk **không có 802.1Q tag** by default; có thể override với `vlan dot1q tag native`. **OVS `vlan_mode=native-untagged` matches Cisco default** (native frame untagged on egress); OVS `native-tagged` is opposite (tag preserved).
- **Juniper Junos** (EX/QFX series): `interface-mode trunk` + `vlan members [10 20]` + `native-vlan-id 1`. Mặc định Junos **untag native VLAN egress** (matches Cisco default + OVS `native-untagged`). Khác Cisco syntax + naming convention.
- **Arista EOS** (DCS series): `switchport mode trunk` + `switchport trunk native vlan 1` + `switchport trunk allowed vlan 10,20`. Default behavior tương đồng Cisco + Junos: native VLAN untagged egress. Native VLAN concept harmonized cross 3 major vendors.
- **Linux native bridge (`brctl` / `ip link`)**: VLAN filtering qua `bridge vlan add` + `vlan_filtering=1`. Frame default untagged on egress (PVID concept tương đương native VLAN). Khác OVS ở granularity: Linux bridge per-port PVID, OVS per-port `tag` + `trunks` kết hợp.
- **VMware vSwitch / NSX**: Port group có `vlan_id` + `mode (none/vlan/trunk/private)`. NSX dùng VLAN trunk rồi internal untag/tag tại virtual NIC. Khác OVS architecture: NSX là controller-driven, OVS là switch-driven config qua OVSDB.
- **IEEE 802.1Q standard**: định nghĩa native VLAN concept (Port VLAN Identifier - PVID) cho phép untagged frame in/out trên trunk. OVS `native-untagged` mode matches IEEE 802.1Q PVID semantic.

**Insight cốt lõi:** OVS VLAN trunk semantics tương đồng IEEE 802.1Q + 3 major vendors (Cisco/Junos/Arista). Khác biệt chính ở **default behavior native VLAN egress**: Cisco/Junos/Arista/OVS `native-untagged` default = untagged out; OVS `native-tagged` = tagged out (atypical, dùng cho specific use case như interop với hardware không strict 802.1Q). Engineer multi-vendor environment cần verify mode mỗi switch để tránh native-tag mismatch (frame tag bị strip ở hop A nhưng kept ở hop B).

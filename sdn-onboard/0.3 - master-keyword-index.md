## Conventions (5-axis plus status code plus scope)

### 5-axis classification (Bucket | Context | Purpose | Activity | Mechanism)

Each keyword is classified along five dimensions:
- **Bucket:** category (OVS daemon, OF action, OVN schema, and so on).
- **Context:** when the keyword appears (log, CLI, specification).
- **Purpose:** the goal the keyword addresses.
- **Activity:** the main runtime behavior or use case.
- **Mechanism:** how it works internally.

### Status code

- **DEEP**: has a full 5-axis Anatomy plus Guided Exercise plus POE in the curriculum.
- **BREADTH**: has treatment in 3 or more curriculum files, enough to learn from.
- **SHALLOW**: mentioned in 1 or 2 files, REF needed to fill gaps.
- **MISSING**: not yet covered in the curriculum (consult REF).
- **BANNED**: out of curriculum scope (DPDK, PMD, SMC, EMC, mempool family).

### Scope

- **In scope:** OVS plus OpenFlow plus OVN core (310 entries).
- **Out of curriculum scope:** advanced topics permanently banned per CLAUDE.md North Star.

# Part I, OVS keyword index

## I.1 Architecture and daemons (5 entries)

### `ovs-vswitchd`

**5-axis:** OVS daemon | long-running userspace | implements OpenFlow pipeline plus datapath flow program | forwarding plus learning plus upcall handling | subscribes to OVSDB JSON-RPC plus handler/revalidator threads plus dpif

The central userspace process of OVS, which owns the OpenFlow pipeline plus bridge configuration plus the upcall handling from the kernel datapath. It subscribes to OVSDB through JSON-RPC for `Bridge`, `Port`, `Interface`, `Controller`, and `Flow_Table` rows. It manages handler threads (which process upcall packets) plus revalidator threads (which validate installed flows). It pushes flows into the kernel, userspace, or DPDK datapath through the `dpif` interface. Status: **DEEP**.

**Curriculum:** [Part 9.1 OVS three-component architecture](9.1%20-%20ovs-3-component-architecture.md) plus [Part 9.4 §9.4.3 ovs-appctl introspection](9.4%20-%20ovs-cli-tools-playbook.md).

### `ovsdb-server`

**5-axis:** OVS daemon | long-running JSON-RPC server | persistent transactional configuration store | schema enforcement plus transaction commit plus monitor push plus Raft log append | Unix socket `/run/openvswitch/db.sock` plus tcp/ssl remote plus standalone/active-backup/Raft cluster

The database server for OVS configuration. It serves `ovs-vswitchd`, `ovn-northd`, `ovn-controller`, `ovs-vsctl`, and external orchestrators (Neutron, OVN-Kubernetes). It supports three modes: standalone (1 node), active-backup (2 nodes), and Raft cluster (3 or 5 nodes for HA). Status: **DEEP**.

A Python helper daemon that connects OVS with strongSwan or Libreswan to bring up IPsec on a tunnel port. It reads `Interface.options:remote_ip`, `:psk`, and `:cert` in OVSDB and renders an IKE configuration file for the IPsec daemon. It starts automatically through `openvswitch-ipsec.service`. A wrapper script that automatically creates a `Mirror` row in OVSDB for one port, spawns `tcpdump` on the tap interface, and cleans up the mirror on exit. Convenient for quick debug without manual mirror configuration. Status: **BREADTH**.

### Control socket and pidfile convention

**5-axis:** OVS daemon convention | every OVS daemon | enables `ovs-appctl` runtime control without configuration | daemon writes `<rundir>/<program>.pid` plus appctl listens on `<rundir>/<program>.<pid>.ctl` | Unix-domain control socket

Every OVS daemon (`ovs-vswitchd`, `ovsdb-server`, `ovn-controller`, `ovn-northd`) writes a pidfile plus creates a Unix-domain control socket with the same name plus pid. `ovs-appctl --target=<name>` discovers the socket through the pidfile. This allows runtime commands (vlog, coverage/show, debug commands) without restarting the daemon. Status: **BREADTH**.

## I.2 Datapath and forwarding internals (about 25 in-scope entries)

### `openvswitch.ko` (kernel datapath)

**5-axis:** OVS datapath | in-tree Linux kernel module | fast-path packet matching plus action execution plus tunneling | per-packet flow lookup plus on-miss generates an upcall | `dp_flow` cache exact-match on all extracted fields plus per-flow mask plus tunnel kernel (Geneve, VXLAN, GRE, ERSPAN)

The default kernel module for the OVS datapath. Fast-path packet processing happens in kernel space; on flow miss, an upcall is generated to userspace `ovs-vswitchd`. Supports kernel-native tunnel encapsulation (Geneve, VXLAN, GRE, ERSPAN). Status: **DEEP**.

An alternative datapath implemented entirely in userspace `ovs-vswitchd`. Required for DPDK (kernel bypass) or for OVS running on non-Linux (FreeBSD, Windows). Set bridge `datapath_type=netdev` to enable. Status: **BREADTH**.

### `ofproto` and `dpif` layering

**5-axis:** OVS internal architecture | `ovs-vswitchd` | separates OpenFlow semantics (`ofproto`) from the datapath interface (`dpif`), so the same daemon drives kernel, userspace, or DPDK | `ofproto` translates the OpenFlow flow table into `dpif` flow installs | `dpif/show` shows dpif state, `ofproto/trace` shows the ofproto pipeline

OVS has two layers: `ofproto` handles OpenFlow protocol semantics (controller communication, flow table abstraction); `dpif` is the interface to the actual datapath (kernel `ovs-system`, netdev userspace, dpdk). The same `ovs-vswitchd` daemon drives all three datapath types. Status: **DEEP**.

**Curriculum:** [Part 9.15 ofproto classifier](9.15%20-%20ofproto-classifier-tuple-space-search.md) plus [Part 9.16 connection manager](9.16%20-%20ovs-connection-manager-controller-failover.md).

### Megaflow

**5-axis:** OVS datapath flow cache | kernel or userspace datapath | caches a wildcarded entry per equivalence class instead of a microflow | on miss, userspace computes the smallest sufficient mask plus installs a megaflow | each megaflow carries a mask plus resolved actions, and subsequent matching packets bypass userspace

A wildcarded flow cache in the kernel datapath. Each megaflow represents one equivalence class of packets (same mask, same action). Dramatically reduces the number of flows in the kernel cache compared with microflows (one entry per packet). Status: **DEEP**.

**Curriculum:** [Part 9.2 OVS kernel datapath plus megaflow](9.2%20-%20ovs-kernel-datapath-megaflow.md) (the entire file).

### Megaflow wildcarding

**5-axis:** OVS datapath optimization | userspace classifier | the kernel cache stays small by masking out fields that were not examined | xlate records "fields touched" plus unmasked fields are wildcarded | a poorly written rule reduces wildcarding plus explodes the kernel flow count

An optimization: only write a flow with masks on fields actually matched. Fields not touched during xlate are wildcarded (ignored). A poorly written rule means poor wildcarding, and the kernel flow count explodes. Status: **BREADTH**.

**Curriculum:** [Part 9.2 megaflow](9.2%20-%20ovs-kernel-datapath-megaflow.md) plus [Part 9.26 revalidator-storm forensic](9.26%20-%20ovs-revalidator-storm-forensic.md).

### Upcall

**5-axis:** OVS datapath event | kernel to userspace when no datapath flow matches | delivers the unmatched packet to `ovs-vswitchd` to consult OpenFlow tables | a handler thread dequeues plus runs xlate plus installs the megaflow plus re-injects | netlink (NLMSG/OVS_PACKET_CMD_MISS) for the kernel datapath, channel queue for netdev

The mechanism by which the kernel notifies userspace when no flow matches. A userspace handler thread receives it, runs an OpenFlow pipeline simulation, installs the resulting megaflow, and re-injects the packet. Upcall throughput is the slow-path bottleneck. Status: **DEEP**.

**Curriculum:** [Part 9.2 datapath](9.2%20-%20ovs-kernel-datapath-megaflow.md) plus [Part 9.26 revalidator-storm forensic](9.26%20-%20ovs-revalidator-storm-forensic.md).

### Handler thread

**5-axis:** OVS thread inside `ovs-vswitchd` | processes upcalls plus installs datapath flows | parses upcall payload plus runs xlate plus writes back through dpif | tunable `other_config:n-handler-threads`

The unit-of-work thread that processes upcalls in `ovs-vswitchd`. Each thread parses an upcall, simulates the pipeline, and installs the megaflow rule. The thread count is tunable (default about CPU count). It becomes a bottleneck if the upcall rate exceeds thread capacity. Status: **DEEP**.

**Curriculum:** [Part 9.26 revalidator-storm forensic](9.26%20-%20ovs-revalidator-storm-forensic.md) plus [Part 9.26 storm forensic](9.26%20-%20ovs-revalidator-storm-forensic.md).

### Revalidator thread

**5-axis:** OVS thread inside `ovs-vswitchd` | re-validates datapath flows after OpenFlow rule changes plus expires idle flows | dumps kernel flows plus recomputes actions plus deletes or updates | tunable `other_config:n-revalidator-threads`, gated by the `max-idle` plus `max-revalidator` timers

A thread that periodically scans kernel flows, re-validates them (checks whether the OpenFlow rule has changed, making the flow stale), and expires idle flows. A storm means the revalidator is at 100 percent CPU due to high flow churn. Status: **DEEP**.

**Curriculum:** [Part 9.26 revalidator-storm forensic](9.26%20-%20ovs-revalidator-storm-forensic.md) (the entire 1185-line file is dedicated to this).

### Recirculation (`recirc_id`)

**5-axis:** OVS pipeline construct | re-enters the pipeline after a header rewrite | actions emit `recirc()` plus the kernel feeds the packet back into lookup with a non-zero `recirc_id` | `recirc_id` becomes a match field plus the megaflow is keyed including it

When a packet goes through an action that changes the header (decap tunnel, MPLS push or pop, conntrack, NAT), the pipeline must be re-looked-up with the new header. The `recirc()` action sets a non-zero `recirc_id`, the packet is fed back into the datapath with the new `recirc_id` as a match field. Status: **BREADTH**.

**Curriculum:** [Part 9.2 datapath recirc](9.2%20-%20ovs-kernel-datapath-megaflow.md) plus [Part 9.25 ofproto/trace recirc](9.25%20-%20ovs-flow-debugging-ofproto-trace.md).

### Connection tracking (`ct`)

**5-axis:** OVS stateful firewall | kernel or userspace | classifies packet as new, established, related, invalid, reply, or tracked plus NAT plus commits per-flow state | `ct(commit, zone, nat, alg)` action plus `ct_state` match consumed after recirc | the kernel datapath uses Linux `nf_conntrack`, userspace has its own implementation

Stateful firewall integration in OVS. The `ct()` action tracks connection state (new, established, related, invalid, reply, tracked) plus optional NAT plus zone isolation. The foundation for stateful OVN ACLs. Status: **DEEP**.

**Curriculum:** [Part 9.24 OVS conntrack stateful firewall](9.24%20-%20ovs-conntrack-stateful-firewall.md) (671 lines).

### `ct_state`, `ct_zone`, `ct_mark`, `ct_label`

**5-axis:** OVS NXM extension match field | post-conntrack state | per-connection metadata for stateful policy | match `+new+trk`, `+est+trk`, `+rel+trk`, `+inv+trk` plus `ct_zone` 16-bit isolation plus `ct_mark` 32-bit plus `ct_label` 128-bit | conntrack entry is persistent

Four match fields for conntrack:
- `ct_state`: bitmask state flag (NEW, EST, REL, INV, RPL, TRK, SNAT, DNAT).
- `ct_zone`: 16-bit conntrack namespace (per-tenant isolation).
- `ct_mark`: 32-bit per-connection tag.
- `ct_label`: 128-bit per-connection opaque metadata.

Status: **DEEP**.

### ALG (FTP, TFTP, SIP)

**5-axis:** OVS conntrack helper | tracks expected child connections for L7 protocols | enable through `ct(alg=ftp/tftp/sip,...)` | the helper parses the L7 control payload plus spawns a RELATED expectation

Application Layer Gateway. FTP, TFTP, and SIP open data channels dynamically based on the control channel. The ALG parses the control payload to spawn a conntrack entry for the data channel with state `RELATED`. Status: **SHALLOW**.

### NAT

**5-axis:** OVS conntrack action sub-clause | stateful source or destination address-and-port translation | `nat(src=...)`, `nat(dst=...)`, `nat(src=10.0.0.1:1024-2048,random)` | NAT mapping is established on commit plus reused for the rest of the connection

Stateful NAT integrated into conntrack. The mapping is installed on commit; every subsequent packet of the same connection uses the same mapping. Supports random ports plus ranges. Status: **DEEP**.

### `learn` action

**5-axis:** OVS NXM extension action | installs a new flow at runtime triggered by a packet | the action template references current packet fields plus builds a new flow | the learned flow is installed through the same path as `ofctl add-flow`

Dynamic flow installation. When a packet matches a rule with a `learn(...)` action, OVS creates a new flow with a match drawn from the current packet plus an action template. The foundation for L2 MAC learning plus ephemeral state caching. Status: **DEEP**.

### UFID (Unique Flow ID)

**5-axis:** OVS dpif identifier | 128-bit unique flow ID | addresses a kernel flow without re-sending the entire match | dpif emits the UFID on flow install plus later operations reference it | visible through `ovs-appctl dpctl/dump-flows --names ufid:`

Each kernel flow has a 128-bit UFID for fast identification, avoiding the need to re-send the entire match key for update or delete operations. Status: **BREADTH**.

### Interface type: `internal`

**5-axis:** OVSDB `Interface.type` | tuntap kernel device owned by OVS | bridge local management interface plus OVSDB traffic loopback plus router gateway port | OVS creates a kernel netdev tun plus assigns the bridge MAC plus traffic to the ofport enters the pipeline | OVS-internal packet path

Allows creation of a kernel network interface "owned" by an OVS bridge. Traffic sent into the interface enters the OVS pipeline (like a normal packet). Use cases: bridge management interface (`br-int` IP), router gateway port. Status: **DEEP**.

**Curriculum:** [Part 13.4 br-int architecture plus patch ports](13.4%20-%20br-int-architecture-and-patch-ports.md) plus [Part 9.20 VLAN plus internal port lab](9.20%20-%20ovs-vlan-access-trunk.md).

### Interface type: `patch`

**5-axis:** OVSDB `Interface.type` | virtual wire between two OVS bridges that bypasses the kernel networking stack | the canonical way to connect `br-int` and `br-ex` or `br-provider` in OVN | patch ports come in pairs (peer reference through `options:peer`) plus a packet emitted on one port appears instantly on the partner | no kernel veth or `net_device`

A patch port is a "wire" between two OVS bridges on the same host. It does NOT go through the kernel networking stack (no veth or `net_device`). Configure as a pair: `bridge1.port1.options:peer=port2` plus `bridge2.port2.options:peer=port1`. Canonical use case: OVN `br-int` to `br-ex` (external network). Status: **DEEP**.

**Curriculum:** [Part 13.4 patch port](13.4%20-%20br-int-architecture-and-patch-ports.md) (the entire file).

### VLAN bridge modes (`Port.vlan_mode`)

**5-axis:** OVSDB `Port.vlan_mode` column | controls 802.1Q frame admit and send | provisioning plus VLAN segmentation | four values: access, trunk, native-tagged, native-untagged

Four VLAN modes on a port:
- `access`: a single VLAN (`tag` column); incoming untagged is classified into the VLAN; outgoing strips the tag. Tagged inbound frames are dropped (the default for VM-facing ports in OVN).
- `trunk`: carries the VLANs in the `trunks` list (empty equals all); frames must arrive plus leave tagged.
- `native-tagged`: like trunk, but a frame in the VLAN equal to the `tag` field arrives untagged plus is emitted tagged.
- `native-untagged`: like native-tagged but a frame in the `tag` VLAN is emitted UNTAGGED. Used for uplink interop with an access switch.

Status: **DEEP**.

**Curriculum:** [Part 9.20 OVS VLAN access plus trunk](9.20%20-%20ovs-vlan-access-trunk.md) (the entire 337-line file) plus [Part 8.2 Linux VLAN plus bonding](8.2%20-%20linux-vlan-bonding-team.md).

### Bond mode (`Port.bond_mode`)

**5-axis:** OVSDB `Port.bond_mode` column | bonded port link-aggregation algorithm | provisioning plus HA plus bandwidth aggregation | three modes: active-backup, balance-slb, balance-tcp

Three bond modes:
- `active-backup`: one slave forwards; on failure, switches to another slave. No LACP needed on the switch side.
- `balance-slb` (Source Load Balance): hashes flows by source MAC plus VLAN; no LACP. The slave changes when a link fails.
- `balance-tcp`: hashes the 5-tuple (requires LACP 802.3ad upstream). The most granular, recommended for performance-sensitive use.

Status: **DEEP**.

**Curriculum:** [Part 8.2 Linux VLAN plus bonding plus team](8.2%20-%20linux-vlan-bonding-team.md) (Linux foundation) plus [Part 9.4 §9.4.1 OVS bond setup](9.4%20-%20ovs-cli-tools-playbook.md).

### Transactional model

**5-axis:** OVSDB JSON-RPC `transact` operation | ACID multi-row multi-table update | `ovs-vsctl` plus `ovn-nbctl` build a single transaction per CLI invocation | server validates schema plus constraint plus commits atomically

ACID model for OVSDB. Each `ovs-vsctl ... -- ... -- ...` is a single atomic transaction. The server validates the entire transaction before commit; if a constraint is violated, the transaction is rolled back. Status: **DEEP**.

### Raft clustering

**5-axis:** OVSDB cluster mode | high availability through consensus | leader-based writes plus follower reads | typical 3 or 5 nodes | term plus index plus vote per the Raft paper

OVSDB cluster mode based on Raft consensus. A 3-node Raft has 1 leader plus 2 followers. The leader handles writes; followers handle reads (eventually consistent). Failover is under 1 second with the default election timeout of 1000 ms. Status: **DEEP**.

**Curriculum:** [Part 10.1 OVSDB Raft clustering](10.1%20-%20ovsdb-raft-clustering.md) (412 lines) plus [Part 9.31 §9.31.4 bootstrap 3-node Raft](9.31%20-%20ovsdb-tool-offline-utility.md).

### Active-backup replication

**5-axis:** OVSDB replication mode (legacy) | 2-node active plus standby | sync through OVSDB protocol replication | failover is manual or via an external orchestrator | deprecated by Raft

A 2-node replication mode, the predecessor of Raft. The active node accepts writes and syncs to the standby asynchronously. Failover is manual. Deprecated in production (use Raft instead). Status: **SHALLOW**.

### JSON-RPC monitor

The OVSDB protocol `monitor` method. The client subscribes plus receives notifications when a table or column changes. The foundation for IDL (Interface Definition Language) maintaining a synchronized in-memory cache. Status: **DEEP**.

### `monitor-cond` (conditional monitor, OVS 2.6 and later)

**5-axis:** OVSDB protocol method | filters monitor by condition | client subscribes only to rows matching a predicate | reduces bandwidth plus memory for large clusters

An improvement over `monitor`: the client sends a predicate (for example, "only rows where chassis = X"), and the server pushes only matching rows. Critical for OVN scale (per-chassis filter). Status: **DEEP**.

Resume from the last point instead of resyncing the whole. Critical for `ovn-controller` reconnect after a network partition. Status: **DEEP**.

### State machine (4 stages)

**5-axis:** OpenFlow connection lifecycle | HELLO to FEATURES to Steady to AUX | per-stage control message plus lifecycle | the foundation for the controller-switch handshake

State machine in 4 stages:
1. **HELLO state**: post TCP or TLS, exchange `OFPT_HELLO` plus version negotiation.
2. **FEATURES state**: controller request plus switch reply with capabilities.
3. **Steady state**: keep-alive ECHO plus flow operations plus multipart query.
4. **AUX state** (OF 1.3 and later): auxiliary connection for parallel transmission.

Status: **DEEP**.

**Curriculum:** [Part 3.3 §3.3.10 connection state machine](3.3%20-%20openflow-protocol-messages-state-machine.md) plus an ASCII diagram.

### Auxiliary connections (OF 1.3 and later)

**5-axis:** OpenFlow connection extension | multiple TCP or TLS connections to the same `datapath_id` | parallel transmission plus redundancy | `auxiliary_id` 8-bit identifier | the main connection is primary

Allows the controller to open multiple connections to the same switch (same `datapath_id`). Use cases: parallel transmission, switch-to-switch forwarding, redundancy. Status: **DEEP**.

**Curriculum:** [Part 3.3 §3.3.9 auxiliary connections](3.3%20-%20openflow-protocol-messages-state-machine.md).

# R2.5.4 Lab: Bond and LACP, OVS versus Linux bonding driver

> **Verbatim source:** `sdn-onboard/labs/v3.13-R2.5.4-bond-lacp.typescript`
> **Timing source:** `sdn-onboard/labs/v3.13-R2.5.4-bond-lacp.timing`
> **Captured:** 2026-04-30 07:30:44 UTC to 07:31:47 UTC inside `tmux new-session -s v3_13-r2_5_4` on `lab-openvswitch` (`192.168.1.250`). Lab host: Ubuntu 22.04.5 LTS, kernel `5.15.0-173-generic`, OVS 2.17.9 userspace install at `/usr/local/`.

A traditional Cisco engineer who has built EtherChannel bundles knows the problem link aggregation solves: a single Ethernet link has a fixed maximum bandwidth and is a single point of failure; bundling multiple parallel links together gives both higher aggregate bandwidth and link redundancy. Cisco does this with `interface Port-channel 1` plus `channel-group 1 mode active` on each member; the bundle gets one logical identity for forwarding decisions while the physical members carry the load. The IEEE protocol that turns the static configuration into a negotiated runtime agreement between the two ends is 802.3ad LACP (Link Aggregation Control Protocol), now part of IEEE 802.1AX. LACP runs as a control-plane chatter on the wire, exchanging LACPDUs (Link Aggregation Control PDUs) so each end sees what the other end has configured and both can converge on a coherent aggregator membership.

This lab proves the same idea on three implementations. First, OVS in `active-backup` mode (no LACP, one member active at a time, simple failover). Second, OVS in `balance-tcp` mode with `lacp=active` (full LACP negotiation, traffic load-balanced by 5-tuple hash). Third, the in-kernel Linux bonding driver in `mode=active-backup` (mode 1) and `mode=802.3ad` (mode 4, the LACP mode). The pedagogical goal is to give the engineer fluent reading skills with three quite different inspection surfaces that all describe the same conceptual state. After this lab the engineer can read `ovs-appctl bond/show`, `ovs-appctl lacp/show`, `ovs-appctl lacp/show-stats`, and `cat /proc/net/bonding/<bond>` and extract from each the bond mode, the active member or members, the LACP negotiation status, and the per-member statistics.

## R2.5.4.1. Where this part sits in the curriculum and why

R2.5.4 is the fourth sub-capture of R2.5. The pattern of the previous three sub-captures repeats: a traditional networking primitive (link aggregation, in this case) is exercised in two implementations on the same host, with the same conceptual outcomes observed through two different inspection surfaces. R2.5.1 covered MAC learning. R2.5.2 covered VLAN access and trunk. R2.5.3 covered spanning tree. This sub-capture covers link aggregation, the protocol that explicitly **uses** redundant inter-bridge links instead of hiding them (which is spanning tree's job).

The two protocols are complementary. Spanning tree converts a physical loop topology into a logical tree topology by parking redundant links. Link aggregation converts multiple parallel physical links between the SAME two bridges into a single logical link with load-sharing. A network engineer typically uses both in the same fabric: spanning tree handles multi-bridge redundancy; link aggregation handles the redundancy of multiple wires between two adjacent bridges. The two coexist by treating a bond bundle as a single logical interface from spanning tree's perspective; spanning tree's loop-detection runs on the bundle's logical identity, not on individual member wires.

OVS implements three bond modes: `active-backup`, `balance-slb`, and `balance-tcp`. The first is the "no upstream-switch requirements" mode: one member is active at a time, the others are idle, on member failure OVS picks a new active. The second is "balance source MAC across members": each VM's source MAC is hashed to a fixed member, giving rough per-VM bandwidth; the upstream switch can be a dumb hub. The third is "balance flows by 5-tuple hash with LACP negotiation": each TCP flow hashes deterministically to a member, the upstream switch must support LACP (802.3ad). The OVS source for these modes lives in `ofproto/bond.c` (2146 lines at v2.17.9, commit `0bea06d9957e3966d94c48873cd9afefba1c2677`); the LACP state machine lives in `lib/lacp.c` (1223 lines).

The Linux in-kernel bonding driver implements seven modes (mode 0 through 6, conventionally referred to by both numeric ID and symbolic name): `balance-rr` (mode 0, round-robin), `active-backup` (mode 1), `balance-xor` (mode 2), `broadcast` (mode 3), `802.3ad` (mode 4, the LACP mode), `balance-tlb` (mode 5, transmit load balancing without peer cooperation), `balance-alb` (mode 6, adaptive load balancing). The implementation is in the kernel at `drivers/net/bonding/`.

The mapping between OVS modes and Linux bonding modes is:

| OVS mode          | Linux bonding driver mode | LACP                       |
|-------------------|---------------------------|----------------------------|
| `active-backup`   | `mode 1` / `active-backup` | Off (no LACPDU exchange)  |
| `balance-slb`     | (no exact equivalent; closest is `balance-xor` mode 2 with `xmit_hash_policy=layer2`) | Off |
| `balance-tcp` + `lacp=active` | `mode 4` / `802.3ad` with `lacp_rate=fast` | On (active negotiation) |

The lab exercises `active-backup` and `balance-tcp` on OVS, then `mode 1` and `mode 4` on the Linux bonding driver. The three modes that have no exact equivalent are mentioned but not exercised.

## R2.5.4.2. Topology

The topology is two bridges connected by two parallel veth pairs. Each bridge owns one end of each pair as a member of its bond. The diagram:

```ascii
   br1 with bond1 = (v_p1a, v_p2a)
        |              |
       v_p1a         v_p2a
        |              |
       v_p1b         v_p2b
        |              |
   br2 with bond2 = (v_p1b, v_p2b)
```

The same conceptual layout repeats on the Linux bonding driver side with `bond_a` (ports v_la1, v_la2) facing `bond_b` (ports v_lb1, v_lb2) via two veth pairs. The Linux bonding driver does not need a bridge to attach to: a bond device is itself a network device that can carry IP, be enslaved to a bridge, or just stand alone. For this lab the bond devices stand alone; they exchange LACPDUs through the veth pairs and the lab reads the negotiation state.

The IP addressing is irrelevant for these exercises (LACPDUs are L2 control frames, addressed to the slow-protocols multicast `01:80:c2:00:00:02`). No namespaces or IPs are needed.

## R2.5.4.3. The OVS configuration surface for bonds

OVS encodes bond configuration in two columns of the OVSDB `Port` row: `bond_mode` (string: `active-backup`, `balance-slb`, `balance-tcp`) and `lacp` (string: `active`, `passive`, `off`). The `bond_mode` column controls how the bond distributes traffic; the `lacp` column controls whether the bond exchanges LACPDUs and, if so, in active or passive role. A bond with `bond_mode=active-backup` ignores `lacp` because active-backup does not need negotiation. A bond with `bond_mode=balance-tcp` requires `lacp=active` or `lacp=passive` to participate in 802.3ad; without LACP, the upstream switch will not enable the bundle and only one link's worth of bandwidth is usable.

The configuration is created via `ovs-vsctl add-bond br1 bond1 v_p1a v_p2a bond_mode=active-backup`, which is shorthand for "create a port named `bond1` on bridge `br1` containing interfaces `v_p1a` and `v_p2a` with bond_mode=active-backup". The members are interfaces of the same OVSDB `Port`; the port itself is the logical identity.

The runtime introspection surface has three commands. `ovs-appctl bond/show <port>` prints the bond's mode, active-member status, and per-member enable state. `ovs-appctl lacp/show <port>` prints the LACP negotiation state if LACP is active. `ovs-appctl lacp/show-stats <port>` prints per-member LACPDU send/receive counters.

The `bond_mode` is `active-backup` on both. The `lacp` column is `[]` (empty) because active-backup does not use LACP. The `bond_active_slave` column carries the MAC address of the currently active member: `e2:48:5c:f2:1c:bf` on bond1 (which is v_p1a's MAC) and `da:12:ef:8c:61:07` on bond2 (which is v_p2b's MAC). Each end independently picks its first-up member as the initial active.

```text
active member mac: da:12:ef:8c:61:07(v_p2b)

  may_enable: true

member v_p2b: enabled
  active member
  may_enable: true
```

The output documents several fields per bond. `bond may use recirculation: no` is OVS's flag for whether the bond will use the internal recirculation mechanism (a kernel datapath feature for re-injecting a packet into the bridge after a logical action; relevant for balance-tcp, not for active-backup). `bond-hash-basis: 0` is the seed for the hash function; on active-backup it is unused. `lacp_status: off` confirms LACP is not running. `active-backup primary: <none>` indicates no primary member is configured (so OVS picks the first-up member).

The per-member section is the practical part. `member v_p1a: enabled` followed by `active member` and `may_enable: true` says: "v_p1a is up, healthy, currently the active member, and would be eligible to forward traffic if needed." `member v_p2a: enabled`, `may_enable: true` says: "v_p2a is up, healthy, eligible to forward, but not currently active."

**Misconception to refute.** A traditional Cisco engineer might assume both ends of an active-backup bond will pick the same physical member as active. Reality: active-backup is uncoordinated. Each side picks independently, often differently. This is fine for unicast forwarding because the inactive members are operationally idle (no traffic flows in or out); the asymmetry has no functional impact. For broadcast/multicast that needs to flow on both sides, active-backup may produce duplicate delivery if the two sides are mismatched; in practice this rarely matters. The asymmetry is a design choice, not a bug.

**Finish.** Two bonds in active-backup mode, each with one active member, visible via OVSDB and `ovs-appctl bond/show`.

### ▶ Guided Exercise 2: Failover by bringing down the active member

This exercise tests that active-backup mode does what its name promises: on failure of the active member, the bond promotes a backup member to active and continues forwarding. The exercise then restores the failed member and observes that it does NOT preempt the now-active member back into active state (this is the default for OVS active-backup; setting `other_config:bond-primary=v_p1a` would change this).

**Outcomes.** v_p1a brought down → v_p2a becomes active member. v_p1a brought back up → v_p2a remains active.

**Before You Begin.** Exercise 1 has completed.

**Instructions.**

**1.** Predict, before running the command, what will happen. The counter-hypothesis is "active-backup does not really fail over, the active member stays the same and traffic just stops." If true, the next `bond/show` after bringing v_p1a down would still show v_p1a as active.

**2.** Bring v_p1a down. Elided fields are unchanged from exercise 1.

```text
root@lab-openvswitch:~/network-onboard# ovs-appctl bond/show bond1 | grep 'active slave mac' || true
root@lab-openvswitch:~/network-onboard# ip link set v_p1a down
root@lab-openvswitch:~/network-onboard# sleep 3
root@lab-openvswitch:~/network-onboard# ovs-appctl bond/show bond1
---- bond1 ----
bond_mode: active-backup
[10 other lines omitted, context: per-bond fields unchanged]
active member mac: 12:73:df:2d:08:c3(v_p2a)

member v_p1a: disabled
  may_enable: false

member v_p2a: enabled
  active member
  may_enable: true
```

The result rejects the counter-hypothesis. The active member is now `v_p2a` (MAC `12:73:df:2d:08:c3`), not v_p1a. The previously-active v_p1a is reported as `disabled`, with `may_enable: false` (the link is down so the member is not a valid forwarding target). The failover happened within the 3-second window of the `sleep 3` call; OVS detects link state changes via netlink notifications, so the response is near-instantaneous.

**3.** Bring v_p1a back up and observe the post-recovery state. Elided fields are identical to step 2's bond/show output.

```text
root@lab-openvswitch:~/network-onboard# ip link set v_p1a up
root@lab-openvswitch:~/network-onboard# sleep 2
root@lab-openvswitch:~/network-onboard# ovs-appctl bond/show bond1
---- bond1 ----
bond_mode: active-backup
[10 other lines omitted, context: identical bond-mode and recirculation fields as before]
active member mac: 12:73:df:2d:08:c3(v_p2a)

member v_p1a: enabled
  may_enable: true

member v_p2a: enabled
  active member
  may_enable: true
```

v_p1a is back to `enabled may_enable: true`, but **v_p2a is still the active member**. OVS does NOT preempt. This is the production-friendly default: every preemption is a brief traffic interruption (the FDB has to be re-flushed because the active member's MAC has changed); avoiding unnecessary preemptions reduces these interruptions. If preemption is desired (some operators prefer it for predictability), set `other_config:bond-primary=<member-name>` on the port. The lab does not exercise the primary-member option.

**Misconception to refute.** A traditional Cisco engineer might expect active-backup to preempt by default, matching some Cisco etherchannel configurations. Reality: OVS does not preempt by default. This is a real behavioural difference; the operator who migrates an active-backup configuration from a Cisco device to OVS should explicitly enable primary-member if preemption is required.

**Finish.** Failover and recovery observed; OVS active-backup does not preempt by default.


This is the canonical IEEE 802.3ad state. The `status: active negotiated` line is the headline: the bond is in active LACP role and has successfully negotiated with the partner. The system identifier `sys_id: c6:25:8e:7e:f1:4b` is OVS's bond-side LACP system ID, derived from the bond's MAC. The `aggregation key: 1` is the aggregator group identifier; ports with the same key on the same system can belong to the same aggregator.

The per-member section gives full LACP detail for each member. `actor` fields describe what THIS bond is advertising; `partner` fields describe what the bond is hearing from the OTHER end. The `state` flag is the critical IEEE 802.3ad state encoding: `activity aggregation synchronized collecting distributing` are five of the eight LACP_STATE flag bits. `activity` means LACP is in active mode (sending LACPDUs proactively); `aggregation` means the port can be aggregated; `synchronized` means the peer's state matches; `collecting` means the port may receive frames in the aggregator; `distributing` means the port may send frames in the aggregator. The combination of all five is the steady-state fully-participating membership; it is what every healthy LACP member should report. (The other three LACP_STATE bits are `defaulted`, `expired`, `lacp_timeout` and their absence here means the member is healthy.)

### ▶ Guided Exercise 4: Compute the bond hash for synthetic flows

This exercise asks `ovs-appctl bond/hash` to compute which member a given source MAC plus VLAN combination would hash to in balance-slb mode. The lab is in balance-tcp mode, so the answer reflects the balance-slb hash function (`bond/hash` always uses the balance-slb formula because balance-tcp's hash depends on per-flow 5-tuple data that the command does not have). The point is to observe that different source MACs produce different hash buckets, validating the load-distribution intent.

**Outcomes.** Three different source MACs hash to three different bucket numbers in the [0, 255] range.

**Before You Begin.** Exercise 3 has completed.

**Instructions.**

**1.** Hash three synthetic source MACs.

```text
root@lab-openvswitch:~/network-onboard# ovs-appctl bond/hash 02:00:00:00:00:01 0
38
root@lab-openvswitch:~/network-onboard# ovs-appctl bond/hash 02:00:00:00:00:02 0
235
root@lab-openvswitch:~/network-onboard# ovs-appctl bond/hash 02:00:00:00:00:03 0
230
```

The three MACs (which differ only in the last byte) hash to bucket 38, 235, and 230 respectively. The hash function uses the source MAC plus the VLAN ID to compute a 256-bucket-deep value; the bucket then maps to a member via a stable assignment table. Different source MACs produce different buckets, which is the design intent. The fact that two close-together MACs (`...02` and `...03`) produced close-together buckets (`235` and `230`) is coincidence; the hash function is not required to spread close inputs to far outputs, only to spread them deterministically and with reasonable uniformity over the input distribution.

**Finish.** Hash distribution observed; different source MACs produce different bucket numbers.

### ▶ Guided Exercise 5: Surface the active-backup-with-LACP misconfiguration

E2.5 §E.8 documented a hypothesis: setting `bond_mode=active-backup` combined with `lacp=active` is a misconfiguration that produces link flap. The hypothesis is that LACP negotiation tries to bring both members up while active-backup forces all but one to idle, producing a conflict.

**Outcomes.** Setting `active-backup` on a bond that already has `lacp=active` is accepted by OVSDB. The runtime continues to negotiate LACP (the bond's lacp_status remains `negotiated`) but the bond mode governs forwarding (only one member receives traffic at a time). No log line records a conflict; the configuration combination is valid and produces a defined behaviour, contrary to the hypothesis.

**Before You Begin.** Exercise 4 has completed.

**Instructions.**

## R2.5.4.4. The Linux bonding driver

The in-kernel Linux bonding driver predates OVS by many years. It is the canonical mechanism for link aggregation on a Linux host that does not run OVS. The driver creates a special `bond` interface type via `ip link add` (or older tools like `ifenslave`), enslaves member interfaces to the bond via `ip link set <member> master <bond>`, and exposes runtime state via `/proc/net/bonding/<bond>`.

The mode is selected at bond-creation time via `mode <name-or-number>`; it CAN be changed after creation only if all members are removed first (the kernel disallows in-place mode changes for safety). The lab will demonstrate this by deleting and recreating the bond between phases.

The Linux bonding driver supports seven modes; this lab exercises mode 1 (active-backup) and mode 4 (802.3ad LACP). The other five (balance-rr, balance-xor, broadcast, balance-tlb, balance-alb) are mentioned but not exercised; they have niche use cases that are out of scope for the OVS comparison.

## R2.5.4.4a. The mathematics of LACP negotiation

A short detour into how LACP negotiates a coherent aggregator helps the engineer read the output of `lacp/show` and `cat /proc/net/bonding/bond_a` beyond surface pattern-matching. The protocol exchanges LACPDUs between the two ends; each LACPDU carries the sender's "actor" state and the sender's view of the receiver's "partner" state. Both ends simultaneously broadcast their own state and report what they see. After exchanging LACPDUs for a few rounds, both ends converge on a state where each side's actor matches what the other side says about it as partner. At that point the LACP state machines transition through `attached` to `collecting` to `distributing` and the aggregator becomes operational.

The eight LACP_STATE bits in the protocol header (formal name `Actor_State` and `Partner_State`, defined in IEEE 802.1AX-2014 §5.4.7) encode the per-port state as a single byte. The bits are: bit 0 `LACP_Activity` (active or passive), bit 1 `LACP_Timeout` (long or short timer), bit 2 `Aggregation` (aggregatable or individual), bit 3 `Synchronization` (in sync), bit 4 `Collecting` (receive enabled), bit 5 `Distributing` (transmit enabled), bit 6 `Defaulted` (using default partner info), bit 7 `Expired` (recently received timer expired). The fully-participating state observed in this lab has bits 0, 2, 3, 4, 5 set on slow rate (binary 0b00111101 = decimal 61) or bits 0, 1, 2, 3, 4, 5 set on fast rate (binary 0b00111111 = decimal 63). The numeric value in the Linux bonding driver's `port state` field is exactly this byte, which is why the value `63` consistently appears for the lab's `lacp_rate fast` configuration.

The choice of slow versus fast LACP rate is a trade-off between detection speed and chatter. Slow rate (one LACPDU every 30 seconds) costs almost nothing in bandwidth but takes up to 90 seconds to detect a partner failure. Fast rate (one every second) detects failures in 3 seconds but sends 30x more LACPDUs. For most production fabrics, slow rate is adequate; for fabrics where rapid failover is critical, fast rate is required. The OVS default is `lacp-time=slow`; the configuration override is `ovs-vsctl set port bond1 other_config:lacp-time=fast`.

A subtle interoperability gotcha (E2.5 §E.6): if one end is fast and the other is slow, LACP still negotiates because the timer rate is purely local (each end waits its own configured interval before declaring the peer dead). However, the slow side's failure-detection time is dominated by its own 90-second timeout, so the asymmetry produces asymmetric failover speeds. The mitigation is to always match the LACP rate on both ends.

## R2.5.4.6. Common misconceptions for the audience

**Misconception 1.** "OVS and Linux bonding can be enslaved together; you can put a Linux bond as a member of an OVS bond." Wrong. They are two separate aggregator implementations. An OVS bond has OVS interfaces as members. A Linux bond has kernel netdevs as members. You can put the Linux bond's `bond` interface as a member of an OVS bridge (treating it as a single port), but you cannot put a Linux bond inside an OVS bond or vice versa. The two aggregator implementations are mutually exclusive.

**Misconception 2.** "Active-backup mode is the same on OVS and Linux." Mostly right. The runtime behaviour is the same: one active member, others idle, failover on link loss. The defaults differ slightly (OVS uses netlink-based detection so MII polling is unused; Linux uses MII polling at `miimon` ms intervals plus optional ARP-based detection). Both default to no-preemption.

**Misconception 3.** "balance-tcp is the same as Linux mode 4." Mostly right but missing nuance. OVS balance-tcp uses a userspace 5-tuple hash with kernel datapath recirculation; Linux mode 4 uses an in-kernel layer2/layer2+3/layer3+4 hash configured per-bond. The hash seed behaviour is different; the hash output is the same when the hash policy matches. Both implement IEEE 802.3ad LACP for negotiation, so they interoperate at the wire level.

**Misconception 4.** "Setting `lacp=active` on an active-backup OVS bond breaks things." Refuted by exercise 5. The combination is accepted by OVSDB and the runtime gracefully handles it; LACP negotiates in the background, active-backup forwards through one member. The hypothesis in E2.5 §E.8 (link flap) was not reproduced.

**Misconception 5.** "Mode 0 (round-robin) and mode 5 (transmit load balancing) on Linux bonding are equivalent to balance-slb on OVS." Wrong. Mode 0 (round-robin) sends frames in alternating order across members regardless of source MAC; this can cause out-of-order arrival which is bad for TCP. Mode 5 (balance-tlb) uses a per-destination-MAC hash for outgoing frames but receives only on one member; the upstream switch sees only one MAC. Neither matches OVS balance-slb's per-source MAC hash. There is no exact equivalent of OVS balance-slb in the Linux bonding driver; the closest is mode 2 (balance-xor) with `xmit_hash_policy=layer2`, which uses a per-source-MAC hash.

## R2.5.4.6a. How the bond's output decision flows through the OVS pipeline

A natural follow-up question is exactly how a bond is treated by OVS's broader forwarding pipeline. The answer connects R2.5.4 back to the NORMAL action discussed in R2.5.1 and R2.5.2.

When a frame arrives on a bond member port, the kernel datapath upcalls to userspace on a miss. `ofproto-dpif-xlate.c` translates the flow, which means walking the OpenFlow table; on a default standalone bridge the only flow is `priority=0,actions=NORMAL`. The NORMAL action calls `xlate_normal`, which looks up the FDB for the destination MAC. If a hit, the lookup returns the **port** (not the member port) of the destination. If the destination port is a bond, the next step is to choose a member.

The choice is delegated to `bond_choose_output_member` in `ofproto/bond.c` (line 987 in v2.17.9). For active-backup bonds, this function returns the bond's currently-active member. For balance-slb bonds, it computes a hash on the source MAC plus the VLAN and uses a 256-bucket lookup table to pick a member. For balance-tcp bonds, it computes a hash on the 5-tuple (source MAC, dest MAC, source IP, dest IP, source port, dest port) and uses recirculation to pick a member; the recirculation is needed because the kernel datapath needs to re-classify the frame after the bond decision is made, since different members may have different egress treatments (different VLAN tags, for example).

The kernel datapath's megaflow rule for a bond ingress looks something like `recirc_id(0),in_port(v_p1a),...,actions:userspace(...)` for the first balance-tcp packet and `recirc_id(1),...,actions:bond_id(1)` for subsequent packets after the recirc. The `bond_id` action is OVS's kernel-side primitive for "pick a bond member based on a hash"; it needs the bond's per-bucket member assignment table installed in the kernel. The table is installed by userspace via the `OVS_DATAPATH_CMD_SET_BOND_TBL` netlink command and stays cached in the kernel until a member fails or the bond's mode changes.

This level of integration is one of the reasons OVS bonds are quite different from plain Linux bonds: OVS has an explicit fast-path acceleration for bond hash decisions, while the Linux bonding driver implements the hash directly in its transmit path with no equivalent kernel-side acceleration table. For very high packet rates (millions of packets per second), the OVS architecture benefits from this acceleration; for moderate packet rates, both implementations are indistinguishable in performance.

## R2.5.4.7. Production-readiness assessment

The trade-off cost of using LACP vs active-backup. LACP costs an extra control-plane chatter (1 LACPDU per second per member at fast rate, or 1 per 30 seconds at slow rate) and requires both ends to have matching configuration. Active-backup costs nothing extra and works unilaterally. The trade-off is bandwidth: LACP uses both members in parallel for a 2x bandwidth increase under best-case load distribution; active-backup uses only one member, so half the bandwidth potential sits idle. For deployments where bandwidth headroom is plentiful (common in virtualised data centres), active-backup is often the right choice for its simplicity. For deployments where bandwidth is the bottleneck, LACP earns its complexity.

The failure-mode under LACP-negotiation-failure is critical to understand. If LACP cannot negotiate (mismatched aggregator keys, peer not running LACP, peer dead), a default OVS or Linux bond enters a "no-aggregator" state where no traffic flows on any member. The bond appears UP but does not forward. The mitigation is OVS's `lacp_fallback_ab` setting (or Linux's `min_links` and the `LACP active fallback` kernel patch), which falls back to active-backup if LACP negotiation fails. The lab does not exercise this fallback because the healthy negotiation case is the more common one to study first; a production deployment should always evaluate the failure mode and configure the fallback appropriately.

The monitoring discipline. Every bond should be monitored for member flap (a `Carrier Status Changed` count > 0 in OVS, or `Link Failure Count > 0` in Linux) and for LACP renegotiation events. Frequent renegs are a sign of a flapping link, a duplex mismatch, or a peer misconfiguration. The OVS-side instrumentation is `ovs-appctl lacp/show-stats <port>`; the Linux-side is buried in the `/proc/net/bonding/<bond>` output's `Link Failure Count` and `Actor Churned Count` fields. A monitoring agent should poll these counters periodically and alert on any non-zero value.

## R2.5.4.8a. Reading the bond.c source for the engineer ready to dig

For an engineer who wants to understand exactly how OVS picks a member for a balance-tcp bond, the cleanest entry point is the `choose_output_member` function in `ofproto/bond.c` (line 1979 at v2.17.9). The function takes a flow, a flow wildcard mask, and a VLAN as input and returns a pointer to the chosen `struct bond_member`. Internally it switches on `bond->balance` (an enum that maps to BM_TCP, BM_SLB, BM_AB) and dispatches to the per-mode helper.

For BM_TCP, the helper computes a 32-bit hash from the flow's 5-tuple plus the bond's hash basis, modulo the number of slots (256), modulo the number of enabled members. The hash uses the same algorithm as the kernel datapath, ensuring that a flow's userspace hash decision matches its kernel hash decision. For BM_SLB, the helper hashes only on (src_mac, vlan), giving the per-source-MAC distribution that the docs describe. For BM_AB, the helper returns the active member directly.

The key engineering insight is that the hash-to-member mapping is indirect: the hash chooses a slot (one of 256), and the slot maps to a member via a per-bond `slave_balance` table. When members come and go, OVS rebuilds the table; existing flows keep their slot assignment if the slot still maps to a working member, otherwise they are migrated to a new slot. This stability across membership changes is what prevents balance-tcp from forcibly redistributing every flow when one member fails.

The state machine also handles the corner case of a brand-new flow arriving while LACP is still negotiating. In that case the bond is in `lacp_status: configured` (not yet negotiated), and OVS falls back to **not forwarding** the flow until LACP completes. This is the safe choice: a flow that goes out before LACP convergence may be dropped by the upstream switch (which is also in pre-negotiated state), causing a silent packet loss. The fallback-to-active-backup option (`other_config:lacp-fallback-ab=true`) changes this behaviour: while LACP is negotiating, OVS treats the bond as active-backup and forwards through the first up member. This is useful for deployments where the upstream may briefly take LACP down (firmware update, port flap) and the OVS side wants to keep traffic flowing.

## R2.5.4.8b. A note on the actor and partner identifiers

The actor and partner system identifiers seen in `lacp/show` and `/proc/net/bonding/<bond>` deserve a brief explanation because the unfamiliar engineer may misread them.

The "actor" in LACP terminology is the local participant in the negotiation, that is, the bond running on the host. The "partner" is the remote participant. Each side has its own concept of who is actor and who is partner: from bond_a's perspective, bond_a is actor and bond_b is partner; from bond_b's perspective, bond_b is actor and bond_a is partner. This is symmetric. The labels in the output reflect the local viewpoint of the command being run.

The system identifier (`sys_id` in OVS, `System MAC address` in Linux) is a 6-byte MAC plus a 2-byte system priority. The MAC is taken from one of the bond's member MACs by default (the kernel picks the first slave's MAC for the Linux bonding driver; OVS uses an explicit allocation algorithm). The system priority is the second tiebreaker after the priority value: lower values win. The lab's `sys_priority: 65534` is the OVS default; Linux uses `65535` as default; both ends use a default in the lab, so the system MAC is the actual tiebreaker. Identical default priorities is fine because both sides need to agree on aggregator key (which they do via the negotiation), not on priority.

The `aggregation key` is the per-bond identifier that determines whether two members can share an aggregator. Members with different keys cannot aggregate. OVS auto-assigns key 1 to the bond (a single-bond host has only one aggregator); the Linux bonding driver auto-assigns key 15 (a different default; the value is arbitrary as long as both sides agree). The two ends of an LACP link must agree on the key, which they do because each side advertises its own key as actor and the other side records it as partner key; if the keys don't match, aggregation fails and the bond falls back to individual members or no-forwarding.

## R2.5.4.9a. A note on observed convergence times

The captured wall-clock data from this lab gives convergence-time observations for each phase.

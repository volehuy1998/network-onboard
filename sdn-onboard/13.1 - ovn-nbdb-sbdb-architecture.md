## 13.1.3 ovn-northd translator

The `ovn-northd` daemon reads NBDB, computes the set of logical flows, and writes them into SBDB. It translates high-level configuration intent into a chain of logical flow entries.

The flow pipeline contains many tables (verified at OVN v22.03.8 via `grep -cE "PIPELINE_STAGE\\(SWITCH, IN" northd/northd.c` and similar: 26 LS ingress, 10 LS egress, 20 LR ingress, 7 LR egress; stage counts change across minor versions):

- **Ingress 0:** admission control.
- **Ingress 1 to 10:** L2 policy (port security, ACL, load balancer).
- **Ingress 11 to 20:** L3 policy (routing, NAT).
- **Ingress 21 to 23:** egress port selection.
- **Egress 0 to 27:** mirrors ingress for the reverse direction.

When the operator runs `ovn-nbctl acl-add`, `ovn-northd` recomputes the pipeline and updates SBDB.

---

## 13.1.4 ovn-controller: per-chassis agent

Each compute node runs `ovn-controller`, a daemon that connects to SBDB, filters the rows it cares about (`Port_Binding.chassis == local`), and translates logical flows into OpenFlow rules installed on `br-int`.

Filtering uses `monitor_cond`, so each chassis receives only the rows relevant to it.

`ovn-controller` also performs the following:

- It advertises the local chassis as a row in the `Chassis` table.
- It creates Geneve tunnel ports to remote chassis.
- It handles MAC binding for the ARP responder.

## 13.1.5 End-to-end workflow

**The operator creates a logical switch and a port:**

```bash
[admin]# ovn-nbctl ls-add ls1
[admin]# ovn-nbctl lsp-add ls1 vm1
[admin]# ovn-nbctl lsp-set-addresses vm1 "aa:bb:cc:00:00:01 10.0.0.1"
```

Workflow:

1. `ovn-nbctl` writes into NBDB.
2. `ovn-northd` detects the change, computes the `Datapath_Binding` and `Logical_Flow` rows, and writes them into SBDB.
3. `ovn-controller` on the chassis where `vm1` is scheduled detects the `Port_Binding` change.
4. `ovn-controller` translates the logical flows into OpenFlow rules.
5. When `vm1` boots, the OVS port tagged `external_ids:iface-id=vm1` is matched by `ovn-controller`, which then updates `Port_Binding.chassis`.
6. Traffic from `vm1` hits the installed rules.

## Key takeaways

First, NBDB is the configuration intent store (the WYSIWYG view for the operator), while SBDB is the compiled form (typically more than ten times the row count). The operator only writes to NBDB.

Second, `ovn-northd` is a compiler that translates NBDB into SBDB. If `ovn-northd` crashes, existing SBDB flows continue to work, but NBDB changes will not propagate.

Third, the per-chassis `ovn-controller` is a distributed control plane that scales linearly with the number of chassis.

Fourth, the logical flow pipeline contains many stages: at OVN v22.03.8 the counts are 26 LS ingress, 10 LS egress, 20 LR ingress, 7 LR egress (verified via `grep PIPELINE_STAGE northd/northd.c` at the v22.03.8 tag). This is more than native OpenFlow. ovn-northd compiles the logical abstraction, and ovn-controller translates it down to actual OpenFlow rules.

## 13.1.8.5 Datapath_Binding as an SBDB table: row anatomy and tunnel_key allocation

> This section is a deep dive into the `Datapath_Binding` table itself in the OVN Southbound database. It is separate from the summary listing in §13.1.8.1 and from `Port_Binding` (Part 13.5 §13.5.1b). The goal is for the engineer to fully understand the minimal three-column row layout of Datapath_Binding, the role of `tunnel_key` as the "24-bit identity card" of a datapath, and how to trace from an OpenFlow flow on `br-int` back to a row in SBDB.

### Concept

`Datapath_Binding` is a table inside the OVN Southbound database (SBDB) that represents a single logical datapath instance. Every `Logical_Switch` (LS) or `Logical_Router` (LR) in NBDB produces exactly one Datapath_Binding row in SBDB; conversely, every Datapath_Binding row links back to exactly one LS or one LR (never both). This row is the 24-bit identity card of the datapath: it assigns the datapath a unique integer called `tunnel_key` (which is also the Geneve VNI when the packet travels cross-chassis), and it is the foreign key that other SBDB tables (`Port_Binding`, `Logical_Flow`, `MAC_Binding`, `Multicast_Group`) point to in order to identify which datapath the row belongs to. Type: OVSDB table, root table (`isRoot=true`), persistent (replicated through the SBDB Raft cluster). Verified against upstream `ovn-sb.ovsschema` on branch-22.03 at schema version `20.21.0` (https://github.com/ovn-org/ovn/blob/branch-22.03/ovn-sb.ovsschema).

### History and background

The `Datapath_Binding` table appeared with OVN 2.4 (2015) as one of the four original SBDB tables (along with `Chassis`, `Port_Binding`, and `Logical_Flow`). Before OVN, the older Nicira NSX architecture treated logical L2 and logical L3 as two completely separate layers and pushed state down to the chassis through a proprietary OpenFlow channel. When designing OVN, Russell Bryant (Red Hat) and Justin Pettit (VMware/Nicira) realised that both LS and LR reduce, at the data plane, to the same general concept: an isolated pipeline flow with a unique tunnel_key. They unified the two objects into a single SBDB table named `Datapath_Binding` with only three columns (`tunnel_key`, `load_balancers`, `external_ids`), so that the data plane side (`ovn-controller` on the chassis) only has to handle datapaths and does not need to distinguish LS from LR. The `load_balancers` column was added during the OVN 2.7 to 2.8 timeframe to attach load balancers to a datapath. The `external_ids` column has carried the NBDB UUID back-reference (`logical-switch=<uuid>` or `logical-router=<uuid>`) since the beginning, for introspection purposes. The schema has not changed its core column structure from OVN 2.4 through 22.03 (only the index constraints and the range have changed), which reflects the strong stability of the design.

### Position in the architecture

The table lives at the SBDB layer (control-plane intermediate), between NBDB (declarative input from the CMS) and OpenFlow on `br-int` (executable output at the chassis). Scope: per-cluster, per-row. A single OVN cluster has one SBDB Raft cluster (three or five members), and every ovn-controller subscribes to the same SBDB; each Datapath_Binding row is a single record cluster-wide (not replicated per-chassis). In the SBDB relationship tree, Datapath_Binding is a top-level parent table: `Port_Binding.datapath` (strong ref), `Logical_Flow.logical_datapath` (strong ref), `MAC_Binding.datapath`, `Multicast_Group.datapath`, and `IGMP_Group.datapath` all point back to Datapath_Binding. Blast radius: changing a Datapath_Binding row (for example, the `tunnel_key` value) forces every Port_Binding and Logical_Flow belonging to that datapath to be re-evaluated, which causes the entire OpenFlow flow block on `br-int` to be reinstalled on every chassis that hosts a port of the datapath. For this reason, `tunnel_key` is designed to be stable for the lifetime of the datapath; northd tries to keep the `tunnel_key` constant across restarts rather than reallocate it.

### Role

Datapath_Binding carries four roles in the OVN architecture:

1. Provide identity for the datapath: the row UUID plus `external_ids:logical-switch` (or `logical-router`) points back to NBDB, allowing `ovn-sbctl` and the operator to navigate in both directions between NBDB intent and SBDB compiled state.
2. Allocate the 24-bit `tunnel_key`: a cluster-unique integer in the range 1 to 16711680, encoded into the `MFF_LOG_DATAPATH` field (which is the 64-bit OpenFlow `MFF_METADATA`, verified in `include/ovn/logical-fields.h` on branch-22.03) inside the `br-int` pipeline on every chassis, and encoded into the 24-bit `MFF_TUN_ID` field of the Geneve outer header when the packet travels cross-chassis (verified at `controller/physical.c` line 155, `put_load(datapath->tunnel_key, MFF_TUN_ID, 0, 24, ofpacts)`). Without `tunnel_key`, the pipeline cannot demultiplex among the datapaths that coexist on the same `br-int`.
3. Attach `Load_Balancer` to the datapath: the `load_balancers` column (a set of UUID refs to SBDB `Load_Balancer`) declares which load balancers apply at this datapath. The same load balancer may attach to multiple LSes or LRs (multi-attach).
4. Carry auxiliary metadata through `external_ids`: in addition to the `logical-switch` or `logical-router` UUID, you find `name=<readable name>` (so that `ovn-sbctl show` prints a friendly name), `interconn-ts=<transit_switch>` for OVN interconnect (Part 13.15), `always-tunnel`, and `requested-tnl-key` (echoed from NBDB).

### Why it exists

Design question: why does OVN need a dedicated table for the datapath instead of attaching `tunnel_key` directly to the `Logical_Switch` or `Logical_Router` row in NBDB? Four reasons. (1) Separation of control-plane state from data-plane state: NBDB only contains intent (the operator declares LS or LR with name, ports, ACL); SBDB contains the runtime artifact that the data plane actually consumes (tunnel_key is read by the data plane on the fast path, it is not declared by the operator). Mixing them would break the separation of concerns. (2) Allow unified LS/LR handling at the chassis: ovn-controller treats LS and LR identically (each is a datapath with a pipeline), so a single shared table is more convenient than two. (3) Allow tunnel_key to be allocated dynamically by northd: if the operator wrote `tunnel_key` directly in NBDB, two operators could accidentally collide; the current design lets northd allocate tunnel_key automatically in the range 1 to 16711680 (through `ovn_datapath_allocate_key`), and the operator only has to set `requested-tnl-key` when pinning is desired. (4) Allow atomic cleanup: deleting a Datapath_Binding row cascades the deletion of every Port_Binding, Logical_Flow, and MAC_Binding belonging to that datapath through strong refs, avoiding state leaks when an LS or LR is removed.

### Problems it solves

This table solves five distinct technical problems. (1) Multi-tenant isolation on a shared bridge: thousands of logical datapaths coexist on the same `br-int`; the pipeline demultiplexes them with `metadata=<tunnel_key>` on every flow without confusion. (2) Cross-chassis multi-tenant encapsulation through one tunnel: there is only one Geneve outer header, but it carries `VNI=tunnel_key` of 24 bits, so the same tunnel mesh can transport up to 16 million datapaths; the destination chassis reads the VNI to demultiplex. (3) Removing an LS or LR without leaking state: the strong-ref cascade ensures Port_Binding and Logical_Flow disappear automatically. (4) Stable identity across restarts: a stable `tunnel_key` means OpenFlow flows on `br-int` do not have to be fully reinstalled when northd restarts. (5) Operator-controlled pinning for disaster recovery and interconnect: `other_config:requested-tnl-key` on the NBDB Logical_Switch lets the operator pin the exact `tunnel_key` (necessary when migrating state from an old cluster to a new cluster while preserving the VNI, or when the tunnel_key has to coordinate with an external system such as a VTEP hardware gateway). Verified at `northd/northd.c` in the function `ovn_datapath_assign_requested_tnl_id`, which reads `smap_get_int(other_config, "requested-tnl-key", 0)`.

### Importance

This is a top-tier foundational table in SBDB, on par with `Port_Binding` and `Logical_Flow`. Reasoning: 100 percent of packets traversing the OVN data plane must carry the `tunnel_key` of some Datapath_Binding row in the pipeline `metadata` field, with no exception. 100 percent of Logical_Flow rows belong to a specific datapath through the `logical_datapath` column. 100 percent of Port_Binding rows belong to a datapath through the `datapath` column. An engineer who does not understand Datapath_Binding cannot read flows on `br-int` (the column `metadata=0xN` has no context), cannot look up the Port_Binding of a specific LS, and cannot debug a Geneve cross-chassis incident (because they do not know which datapath VNI 0xN corresponds to). Among the 320 plus keywords of the REF, Datapath_Binding belongs to the cornerstone OVN group together with `ovn-northd`, `ovn-controller`, NBDB, SBDB, Logical_Flow, Port_Binding, Logical_Switch, Logical_Router, and Geneve TLV.

### How it works (the lifecycle of a row)

A Datapath_Binding row goes through four phases in its lifetime, all driven by `ovn-northd` (the chassis side never writes to this table; ovn-controller only reads):

1. Creation (compile NBDB to SBDB): the CMS calls `ovn-nbctl ls-add ls-tenant1`, which adds one `Logical_Switch` row to NBDB. The northd main loop calls `build_datapaths` (`northd/northd.c` lines 1510 to 1568, branch-22.03). The function has five internal steps. (a) `join_datapaths` compares the set of NBDB LSes plus LRs with the existing set of SBDB Datapath_Binding rows, splitting into three lists: `sb_only` (to delete), `nb_only` (to insert), `both` (already in sync). (b) `ovn_datapath_assign_requested_tnl_id` first honours `requested-tnl-key` if the operator has set it in `other_config`. (c) Existing `tunnel_key` values for `both` datapaths are preserved. (d) `ovn_datapath_allocate_key` (lines 1454 to 1471) calls `ovn_allocate_tnlid` to assign a new `tunnel_key` for `nb_only`, in the range from `OVN_MIN_DP_KEY_LOCAL=1` to `OVN_MAX_DP_KEY_LOCAL=16711680` (verified in `lib/ovn-util.h`: `OVN_MAX_DP_KEY = (1u << 24) - 1 = 16777215`, minus `OVN_MAX_DP_GLOBAL_NUM = 65535` reserved for the interconnect global key range). (e) `sbrec_datapath_binding_insert` writes the new row into SBDB through an OVSDB transaction.
2. Existence and metadata updates: the row exists in SBDB and `tunnel_key` is fixed. When the operator renames an LS, attaches or detaches a `Load_Balancer`, or sets the `interconn-ts` option, northd updates the `external_ids` map or the `load_balancers` set of the corresponding row (without touching `tunnel_key`).
3. Deletion (cascade cleanup): the operator runs `ovn-nbctl ls-del ls-tenant1`, which removes the Logical_Switch row from NBDB. On the next compile, `build_datapaths` finds the datapath in `sb_only` and calls `sbrec_datapath_binding_delete(od->sb)` at `northd/northd.c:1565` (verified at v22.03.8). OVSDB cascades the deletion of every Port_Binding, Logical_Flow, MAC_Binding, and Multicast_Group whose ref `datapath=<this uuid>` (enforced by the strong-ref constraint).
4. Read path down to the chassis: ovn-controller on each chassis subscribes to SBDB through JSON-RPC, reads Datapath_Binding rows, and stores `tunnel_key` in the in-memory cache `local_datapaths`. When installing OpenFlow flows on `br-int`, the function `physical_run` (`controller/physical.c` line 1717) and `consider_port_binding` (line 914) load `dp_key = pb->datapath->tunnel_key` and call `match_set_metadata(&match, htonll(dp_key))` (lines 705, 756, 784, 817) to match the correct datapath; for egress cross-chassis, `put_load(datapath->tunnel_key, MFF_TUN_ID, 0, 24, ...)` encodes the 24-bit VNI for the Geneve outer header.

Runtime invariant: at any moment, every (LS or LR) row in NBDB corresponds to exactly one Datapath_Binding row in SBDB; no orphan rows (the LS or LR has been deleted but the row remains) survive once northd has finished compiling; the `[tunnel_key]` index ensures no two rows share the same `tunnel_key` cluster-wide.

### Classification of columns

The table has exactly three columns, the most minimal among the main SBDB tables (verified against branch-22.03 `ovn-sb.ovsschema`):

| Column | Type | Constraints | Writer | Meaning |
|--------|------|-------------|--------|---------|
| `tunnel_key` | integer | min=1, max=16777215, unique index | ovn-northd | 24-bit VNI assigned to the datapath, used as OF metadata and Geneve VNI |
| `load_balancers` | set<uuid> ref `Load_Balancer` | min=0, max=unlimited | ovn-northd | SBDB Load_Balancer rows attached to the datapath, applied in the pipeline |
| `external_ids` | map<string,string> | min=0, max=unlimited | ovn-northd | Metadata: `logical-switch=<NBDB uuid>` or `logical-router=<NBDB uuid>`, `name=<readable>`, `interconn-ts`, `always-tunnel` |

A single unique index applies: `[tunnel_key]` cluster-wide. Note that the table has no `chassis` column (a datapath does not reside on any one chassis; every chassis sees every datapath); no `up` or `down` column (a datapath is logical, with no liveness state); and no separate `name` column (the name is set through `external_ids:name`). This minimalism reflects the "identity card plus mapping" nature of the table.

### How an operator masters this

Four foundational skills the engineer needs around Datapath_Binding:

1. List the entire cluster with selected columns: `sudo ovn-sbctl --columns=_uuid,tunnel_key,external_ids list Datapath_Binding`. The output gives an overall view: how many datapaths exist, which tunnel_keys have been assigned, which datapaths are LSes, and which are LRs.
2. Trace from a tunnel_key back to NBDB intent: when looking at the OpenFlow flow `metadata=0x5` and wanting to know which datapath it belongs to, run `sudo ovn-sbctl --bare find Datapath_Binding tunnel_key=5` to get the UUID, then `sudo ovn-sbctl get Datapath_Binding <uuid> external_ids` to read `logical-switch=<NBDB uuid>` or `logical-router=<NBDB uuid>`, and finally `sudo ovn-nbctl list Logical_Switch <NBDB uuid>` to read the original LS name.
3. Pin tunnel_key for disaster recovery: before recreating the cluster, record the `tunnel_key` of each LS or LR, then run `sudo ovn-nbctl set Logical_Switch ls-tenant1 other_config:requested-tnl-key=5` so that northd reassigns the same value. This is necessary when migrating state between two clusters where the chassis flows do not have time to be rebuilt.
4. Observe changes in real time: `sudo ovsdb-client monitor unix:/var/run/ovn/ovnsb_db.sock OVN_Southbound Datapath_Binding tunnel_key external_ids` streams every insert, delete, and update, useful when watching LS lifecycle or debugging a northd compile.

### Usage workflow

The workflow to create and verify a datapath end-to-end has five steps, each with a verification command:

1. The CMS creates an LS or LR in NBDB: `sudo ovn-nbctl ls-add ls-tenant1`. Verify with `sudo ovn-nbctl ls-list | grep ls-tenant1`.
2. Wait for northd to compile (typically less than 100ms). Verify with `sudo ovn-sbctl --columns=_uuid,tunnel_key,external_ids find Datapath_Binding 'external_ids{>=}{name="ls-tenant1"}'` and confirm the row exists with `tunnel_key=N`.
3. Read the assigned tunnel_key: `sudo ovn-sbctl --bare --columns=tunnel_key find Datapath_Binding 'external_ids{>=}{name="ls-tenant1"}'`, which (for example) returns `5`.
4. Confirm the corresponding OpenFlow flows: on a chassis hosting a port of the LS, `sudo ovs-ofctl -O OpenFlow15 dump-flows br-int | grep "metadata=0x5"` shows many flow rules (table 8 ACL, table 16 L2_LKUP) matching this datapath.
5. Confirm the Geneve VNI on cross-chassis: on chassis A with a VM on ls-tenant1, ping another VM on the same LS on chassis B and concurrently run `sudo tcpdump -n -i br-tun udp port 6081 -vv`. You should see Geneve packets with `VNI=0x5` (5 in decimal). The datapath `tunnel_key` is exactly this VNI.

Production best practice: do not let the operator touch `tunnel_key` manually unless DR or interconnect requires pinning; every change goes through NBDB intent, and northd allocates automatically. Anti-pattern: editing `tunnel_key` directly through `ovn-sbctl set Datapath_Binding ...`. The next northd compile overwrites it immediately.

### When something goes wrong (reverse symptom map)

Six common symptoms point back to Datapath_Binding:

| Symptom | Datapath_Binding column to check | Diagnostic flow |
|---------|----------------------------------|-----------------|
| "Created the LS but VMs on the same LS cannot reach each other" | Does the row exist, and is `tunnel_key` valid | `ovn-sbctl find Datapath_Binding 'external_ids{>=}{name="ls"}'`; if missing, then northd is stuck or crashed |
| "Cross-chassis traffic drops but east-west on the same host is fine" | Does `tunnel_key` match the captured Geneve VNI | If `tcpdump` shows a VNI different from `tunnel_key`, that is a Geneve encap bug; check `physical.c:put_load(MFF_TUN_ID,...)` |
| "After deleting the LS, flows still remain on br-int" | Has the Datapath_Binding row been deleted | If `ovn-sbctl find Datapath_Binding 'external_ids{>=}{name="ls"}'` still returns it, northd has not compiled yet; restart northd. If the row is gone, ovn-controller has not flushed yet; restart ovn-controller |
| "Flow `metadata=0xN`, do not know which datapath it belongs to" | Trace N back to NBDB | `ovn-sbctl --bare --columns=external_ids find Datapath_Binding tunnel_key=<N>` |
| "After a cluster restart, tunnel_key changed and old OF flows became invalid" | Was `requested-tnl-key` pinned | Check `other_config:requested-tnl-key` on the NBDB LS or LR; if empty, the value is reassigned from a random hint |
| "Two datapaths share the same `tunnel_key` (index violation)" | Index `[tunnel_key]` | Rare, a northd bug; OVSDB rejects the transaction, log line `"Tunnel key X for datapath Y is incompatible"` (verified at northd.c line 1487) |

Cross-links: §13.7 (ovn-controller `physical.c` consumes `tunnel_key` to install flows), §13.8 (ovn-northd `build_datapaths`), §13.16.5 (logical-to-physical OF table mapping uses `metadata=tunnel_key`), Part 13.5 §13.5.1b (Port_Binding sibling table with FK back to Datapath_Binding).

### Tightly related to

Five components couple tightly with Datapath_Binding (changing one means re-examining the other):

- NBDB `Logical_Switch` and `Logical_Router`: the source of the compile. Deleting the LS or LR deletes the corresponding Datapath_Binding row through northd.
- `Port_Binding.datapath` (strong ref): every LSP and LRP in the datapath refers to this row. Deleting Datapath_Binding cascades the deletion of Port_Binding (and so the VM loses its binding).
- `Logical_Flow.logical_datapath` (strong ref): every logical flow rule belongs to a datapath. Cascade behaves the same way.
- `MAC_Binding.datapath` plus `Multicast_Group.datapath` plus `IGMP_Group.datapath`: every auxiliary runtime state is keyed on a datapath.
- The OpenFlow `metadata` register on `br-int`: the 64-bit register `MFF_METADATA` (alias `MFF_LOG_DATAPATH`) carries the datapath `tunnel_key` on every flow. This is the SBDB-to-data-plane bridge.

Loose coupling (referenced, no cascade): `Chassis` (chassis members of the tunnel mesh read tunnel_key but are not constrained by it), `Encap` (Geneve endpoint configuration, independent of the datapath), `Service_Monitor` (load balancer backend health, referenced through LB rather than directly through the datapath).

### Real-world incident

Production incident OVN-2024-0903, OpenStack OSP 16.2 cluster, 60 hypervisors:

Symptom: After the operator ran a script that bulk-deleted 20 stale LSes and concurrently recreated 20 new LSes (with the same names), east-west traffic for the corresponding tenants was completely lost for 8 to 15 minutes and then recovered on its own. Time-series logs showed ovn-northd CPU spiking to 100 percent during this window.

Investigation: (1) `sudo ovn-sbctl --columns=tunnel_key,external_ids find Datapath_Binding 'external_ids{>=}{name="ls-tenant-blue"}'` immediately after the recreate showed the new row had `tunnel_key=8` instead of preserving the previous LS `tunnel_key=5`. (2) `sudo ovs-ofctl dump-flows br-int | grep "metadata=0x5"` on a chassis still showed many stale flows from the dead datapath (the old LS with tunnel_key=5 had been deleted but ovn-controller had not flushed in time because of the CPU pressure). (3) Geneve packets arriving with VNI 0x5 (from remote chassis that had not yet updated) matched the stale flows and were dropped at the end of the pipeline. (4) After 8 to 15 minutes, ovn-controller incremental processing gradually flushed flows for the deleted tunnel_key and installed flows for `tunnel_key=8`, and traffic recovered.

Root cause: the design of `ovn_allocate_tnlid` assigns tunnel_key from a monotonically advancing `hint` and does not reuse the key of a datapath that was just deleted in the same compile cycle (because the old datapath has not released its hint when the new datapath asks). On bulk delete plus bulk recreate, the old key is left behind and the new key is given a different value. Combined with the northd CPU spike, the SBDB updates propagated slowly down to the chassis, opening a desynchronisation window. This is not a bug, it is a design trade-off.

Remediation: (1) During the recreate, set `requested-tnl-key` to pin the previous tunnel_key, so chassis flows match the right datapath even before state is fully synchronised: `for ls in ls-tenant-blue ls-tenant-red ...; do KEY=$(cat /tmp/saved-keys/$ls); sudo ovn-nbctl --wait=sb -- create Logical_Switch name=$ls other_config:requested-tnl-key=$KEY; done`. (2) Workaround: avoid bulk delete plus recreate in the same window; use the update pattern (modify attributes) instead of delete plus recreate. (3) Long term: track OVN main-branch commits that improve hint reuse in `ovn_allocate_tnlid`.

Lesson: the `tunnel_key` of Datapath_Binding carries deep data-plane semantics (Geneve VNI, OF metadata); the operator must treat it as the persistent identity of the datapath and proactively pin it through `requested-tnl-key` in DR or recreate scenarios, not allow northd to assign a random value. Cross-link Part 13.18 §13.18.3 incident MFF_LOG_INPORT also discusses tunnel_key racing (in the Port_Binding scope).

   ```bash
   # In the monitor terminal: see an "insert" with tunnel_key=N (a value advancing from the current key)
   $ sudo ovn-sbctl --columns=_uuid,tunnel_key,external_ids \
                find Datapath_Binding 'external_ids{>=}{name="ls-lab-dp"}'
   # Expected: 1 row, tunnel_key has a value, external_ids contains the "logical-switch" UUID
   ```
3. Read the tunnel_key and confirm the corresponding OpenFlow flows:
   ```bash
   $ KEY=$(sudo ovn-sbctl --bare --columns=tunnel_key \
                find Datapath_Binding 'external_ids{>=}{name="ls-lab-dp"}')
   $ printf 'tunnel_key=%d (hex 0x%x)\n' "$KEY" "$KEY"
   $ sudo ovs-ofctl -O OpenFlow15 dump-flows br-int | grep -c "metadata=0x$(printf '%x' $KEY)"
   # Expected: a few dozen flows (port security, ACL, L2 lookup) on br-int
   ```
4. Pin the tunnel_key and confirm it persists:
   ```bash
   $ sudo ovn-nbctl --wait=sb set Logical_Switch ls-lab-dp \
                other_config:requested-tnl-key=999
   $ sleep 1
   $ sudo ovn-sbctl --bare --columns=tunnel_key \
                find Datapath_Binding 'external_ids{>=}{name="ls-lab-dp"}'
   # Expected: 999 (changed by the pin)
   $ sudo ovn-nbctl --wait=sb remove Logical_Switch ls-lab-dp \
                other_config requested-tnl-key
   ```
5. Add an LSP and observe Port_Binding refer back to Datapath_Binding:
   ```bash
   $ sudo ovn-nbctl --wait=sb lsp-add ls-lab-dp lsp-lab-1
   $ sudo ovn-sbctl --columns=logical_port,datapath,tunnel_key \
                find Port_Binding logical_port=lsp-lab-1
   # Expected: the datapath column is a UUID matching the Datapath_Binding row above
   ```
6. Delete the Logical_Switch and observe cascade deletion:
   ```bash
   $ sudo ovn-nbctl --wait=sb ls-del ls-lab-dp
   # In the monitor terminal: see a "delete" of Datapath_Binding
   $ sudo ovn-sbctl --columns=_uuid \
                find Datapath_Binding 'external_ids{>=}{name="ls-lab-dp"}'
   # Expected: empty (the row is gone)
   $ sudo ovn-sbctl --columns=_uuid find Port_Binding logical_port=lsp-lab-1
   # Expected: empty (strong-ref cascade)
   ```

POE (Predict-Observe-Explain) for step 6: predict whether "Port_Binding disappears immediately or after a few seconds"; observe the moment of disappearance through the monitor stream; explain through the OVSDB strong-ref cascade mechanism inside the same transaction commit.

Cleanup: if not already done, `sudo ovn-nbctl ls-del ls-lab-dp 2>/dev/null`.

### Common failures and diagnostic signals

Four characteristic failure modes of the Datapath_Binding table:

Mode A, the row does not appear after creating the LS or LR. NBDB has the row, SBDB does not. Signal: `sudo ovn-sbctl find Datapath_Binding 'external_ids{>=}{name="<ls>"}'` is empty even though `ovn-nbctl ls-list` printed the name. Cause: `ovn-northd` crashed or stuck. Remediation: check status with `sudo ovn-appctl -t ovn-northd status`; restart with `sudo systemctl restart ovn-northd`. Cross-link: §13.1.5 workflow.

Mode B, tunnel_key changes after a cluster restart. Before the restart `tunnel_key=5`, after the restart `tunnel_key=8`. Signal: stale OpenFlow flows with `metadata=0x5` remain but the new datapath carries `tunnel_key=8`. Cause: the new SBDB cluster reset its state and northd reassigned from a low hint. Remediation: pin `requested-tnl-key` in NBDB `other_config` ahead of time (a critical production best practice).

Mode C, tunnel_key collision (index violation). Two datapaths share the same `tunnel_key`. Signal: `sudo ovn-sbctl find Datapath_Binding tunnel_key=<N>` returns two rows; OVSDB logs `"Tunnel key X for datapath Y is incompatible"` or `"Logical switch/router %s requests same tunnel key %"PRIu32" as another logical switch or router"` (verified in northd.c). Cause: a rare northd bug triggered by using `requested-tnl-key` with the same value on two LSes. Remediation: remove `requested-tnl-key` from one of the two LSes so northd allocates a different value.

Mode D, in VXLAN mode the tunnel_key exceeds 12 bits. The cluster runs VXLAN encap but `tunnel_key > 4095`. Signal: log line `"Tunnel key %"PRIu32" for datapath %s is incompatible with VXLAN"`. Cause: northd allocates the key from the 24-bit range, but the 24-bit VXLAN VNI is split (12 bits for dp plus 12 bits for port). Remediation: switch to Geneve encap (the OVN default), or keep the scale below 4096 datapaths if VXLAN is mandatory.

Diagnostic mantra: when Datapath_Binding looks suspect, the order of checks is (1) does the NBDB LS or LR exist (`ovn-nbctl show`), (2) is ovn-northd running and healthy (`ovn-appctl -t ovn-northd status`), (3) does the Datapath_Binding row exist with the right tunnel_key (`ovn-sbctl find`), (4) do the OpenFlow flows on `br-int` match `metadata=<tunnel_key>` (`ovs-ofctl dump-flows`). Four steps, in order, no skipping.

### Comparison with other systems

OpenStack Neutron (pre-OVN ML2 OVS plugin) has no separate Datapath_Binding concept; segmentation_id is stored in the `ml2_network_segments` table in Neutron's MySQL, attached to a network and a type (vlan, vxlan, gre). Similar to Datapath_Binding in mapping a network to a VNI or VLAN, but completely different in design: Neutron uses a centralised DB while OVN uses a distributed SBDB observed by every chassis.

VMware NSX-T has a "Logical Switch" with a "Replication ID" and a "VNI" that play the same role as `tunnel_key`. State is stored in the centralised NSX Manager cluster and pushed down to each Transport Node through the proprietary MPA protocol. NSX-T does not expose the schema row the way OVN does; the operator only sees it through the API, with no equivalent to `ovn-sbctl list`.

Linux kernel native VXLAN (without OVN): the VNI is set per-tunnel-interface through `ip link add vxlan0 type vxlan id 100 dstport 4789`, with no aggregate table. Each VTEP maintains its own FDB and VNI mapping; there is no datapath abstraction comparable to OVN.

Cisco ACI has a "Bridge Domain" with a "VNID" that matches `tunnel_key`. State is stored in the APIC controller cluster. Different from OVN: APIC keeps centralised state and pushes it down to leaf switches through the Opflex protocol; OVN uses Raft and lets each chassis read on its own, with no push protocol.

Native OpenFlow (without OVN above it) has no logical datapath ID concept; OF only has the `datapath_id` of a physical bridge (or OVS bridge), a 64-bit integer with completely different semantics from the OVN Datapath_Binding `tunnel_key`. New engineers easily confuse the two; you must distinguish clearly: OF `datapath_id` is the bridge ID, OVN `tunnel_key` is the VNI of a logical datapath (LS or LR) hosted on `br-int`.

## 13.1.A SBDB as a database container: a compiled-state store and a southern Raft cluster

> This section drills into the **Southbound Database (SBDB) itself** as a unified entity: what it is, why it is separated from NBDB, which tables it contains, how it operates inside the Raft cluster, and how an engineer must interact with it in production. The section complements §13.1.2 (overview), §13.1.8 (per-table catalog), and §13.1.8.5 (`Datapath_Binding` deep dive, a representative example of an SBDB row).

### Core concept

SBDB is the **southern OVSDB database** of OVN. It stores the **compile output** that `ovn-northd` produces from the NBDB configuration intent, plus the **runtime state** that `ovn-controller` on each chassis writes into it (chassis registration, MAC learning, port binding claim). It is the **procedural intermediate layer** between NBDB (the declarative layer on the north side) and OpenFlow on `br-int` (the execution layer at each chassis on the actual south side). The schema is defined in the file `ovn-sb.ovsschema` (verified branch-22.03 schema name `OVN_Southbound`, version `20.21.0`, https://github.com/ovn-org/ovn/blob/branch-22.03/ovn-sb.ovsschema), with 31 top-level tables. The cluster persists in the file `/var/lib/ovn/ovnsb_db.db` (JSON OVSDB format) on the Raft leader and replicates to each follower. Type: **OVSDB database instance**, schema-defined relational, transaction-consistent through Raft consensus. Each OVN cluster has **exactly one** SBDB instance (one Raft cluster of 3 or 5 members), independent of the NBDB instance.

### History and background

SBDB appeared with OVN 2.4 (2015) as one of the **two native OVSDB databases** of OVN (alongside NBDB). Before OVN, the Nicira NSX-MH architecture did not have a clear "compile intermediate" concept: the centralised NSX Controller pushed state directly to transport nodes through proprietary OpenFlow and OVSDB management. Russell Bryant (Red Hat), Justin Pettit, and Ben Pfaff (VMware/Nicira) when designing OVN proposed cleanly separating **intent** from **realized state**, each into its own database with its own schema: NBDB holds LS/LR/ACL/LB the way the operator thinks about them, SBDB holds Datapath_Binding/Port_Binding/Logical_Flow/Chassis/MAC_Binding the way each chassis needs. This decision follows the **declarative-vs-realized** principle familiar from Kubernetes (etcd stores spec plus status), and it allows the two sides to evolve independently: the NB schema changes for user-facing features, and the SB schema changes for compilation and data-plane optimisation. The SBDB schema grew across versions: 2.4 was rudimentary (about 10 tables); 2.7 to 2.8 added `Load_Balancer` and `Service_Monitor`; 21.06 added `BFD` and `FDB`; 22.03 stabilised at 31 tables (verified upstream); 24.03 expanded for `mirror_rules`, multi-chassis, and zone-id features.

### Position in the architecture

SBDB sits at the **intermediate control-plane layer**, between two other layers:

- North side (input): **NBDB** (typically co-located on the same host as SBDB, but a different `ovsdb-server` process, a different DB file, a different Raft cluster, and a different port). `ovn-northd` is the one-way bridge: it **reads** NBDB and **writes** SBDB.
- South side (consumer): **per-chassis ovn-controller** (one instance per hypervisor). Each ovn-controller subscribes to SBDB through JSON-RPC, reads continuously, and **writes a small subset** (registers only its own Chassis and Encap row, claims and releases Port_Binding when a VM boots or dies, adds MAC_Binding entries when ARP is learned).

Three-tier relationship: NBDB to ovn-northd to SBDB to ovn-controller to OpenFlow on `br-int`. SBDB is the only layer that is **read-write from both sides**: northd writes compiled state (Datapath_Binding, Port_Binding metadata, Logical_Flow), chassis write runtime state (Chassis row, Port_Binding.chassis claim, learned MAC_Binding). NBDB by contrast is read-only from the northd side (only the CMS writes to it). This intermediate position makes SBDB the **hot spot** for debugging: any "why is the packet not flowing" investigation must consult SBDB before descending to OpenFlow.

### Role and responsibilities

SBDB carries **five main roles**:

1. **Compile-output store**: holds Datapath_Binding (one row per LS or LR), Port_Binding (one row per LSP or LRP), Logical_Flow (the entire pipeline rules), Multicast_Group, and IGMP_Group, all of which are products of `ovn-northd` compiling NBDB.
2. **Chassis registry**: each ovn-controller inserts one Chassis row plus one Encap row when it starts, identifying the hypervisor in the cluster, advertising the tunnel endpoint IP, and the supported `iface-types`.
3. **Port-mobility claim table**: ovn-controller sets `Port_Binding.chassis = <my_uuid>` to declare "this port is on my host". When a VM live-migrates, the old controller releases and the new controller claims, atomically through an OVSDB transaction.
4. **Runtime learning cache**: MAC_Binding (router neighbour cache from ARP and ND replies), FDB (L2 learning from unknown unicast), and Service_Monitor (LB backend health-probe results) are all chassis-generated data, stored in SBDB so every chassis sees the same view.
5. **Control events**: a Controller_Event row is inserted by ovn-controller when it encounters packet-too-big or empty_lb, so the CMS or operator can observe through `ovn-sbctl list Controller_Event`.

### Why it exists (design motivations)

Question: why does OVN not merge NBDB and SBDB into a single database? Four motivations:

(1) **Schema lifecycle separation**: the NB schema evolves to meet user needs (adding an ACL feature, adding `Forwarding_Group`); the SB schema evolves to meet compiler and runtime needs (adding `Logical_DP_Group` for memory optimisation, adding `FDB` for L2 learning). Splitting them cleanly means a change on one side does not perturb the other.

(2) **Access-control separation**: the CMS may write only NBDB (through RBAC), and a chassis may write only SBDB (only its own Chassis, Port_Binding, and MAC_Binding rows, through the SBDB RBAC role `ovn-controller`). Splitting the two databases allows different policies to apply, without requiring complex row-level ACLs.

(3) **Scalability**: the SBDB row count is much larger than NBDB. One LS with 1000 LSP plus 50 ACL in NBDB can compile to 50000 or more Logical_Flow rows in SBDB. The split keeps NBDB compact (so the operator can read and write quickly), while SBDB grows large to hold the compiled artefact.

(4) **Independent recovery**: losing NBDB does not lose SBDB and vice versa. Disaster recovery can restore each DB independently. The Raft cluster is also separate (the NB cluster may have 3 members, while the SB cluster may have 5 to favour availability because chassis subscribe directly).

### Problems it solves

SBDB solves **five distinct problems**: (1) **Translate intent to executable**: the operator does not have to think about Logical_Flow, OF metadata, or the Geneve VNI; SBDB absorbs that complexity. (2) **Synchronisation across many chassis**: thousands of ovn-controller instances need to see the same set of Logical_Flow rows; SBDB through Raft guarantees eventual consistency. (3) **Live migration coordination**: Port_Binding.chassis is the synchronisation point between the old chassis and the new chassis; the transactional claim and release prevent dual-binding. (4) **Cluster-wide MAC learning**: MAC_Binding rows share the ARP cache across every chassis, instead of each chassis learning on its own. (5) **Health visibility**: Service_Monitor and Controller_Event let the CMS and the operator observe runtime state without needing to SSH into each hypervisor.

### Importance

SBDB is an **absolute top-tier foundational table** of OVN. Reasons: 100 percent of OVN packets are processed based on Logical_Flow rows in SBDB; 100 percent of chassis depend on SBDB for discovery and binding; 100 percent of debug cycles begin with `ovn-sbctl show` or `ovn-sbctl list Logical_Flow`. Losing the SBDB cluster (losing Raft quorum) means the OVN cluster stops evolving: existing chassis still forward packets (because OF flows are already installed on `br-int` and cached in the kernel), but no new VM can be added or removed, no new MAC can be learned, and no LB backend update can land. SBDB is in the same cornerstone group as NBDB, ovn-northd, ovn-controller, Logical_Flow, Datapath_Binding, and Port_Binding.

### How it works

SBDB operates through four concurrent flows:

1. **Northd write loop (NB to SB compile)**: ovn-northd subscribes to NBDB and SBDB through two IDLs. The main loop (typically 100ms to 1s depending on changes) calls `ovnnb_db_run` to read NBDB, calls `northd_run` to build the in-memory `ovn_datapath` structures, and calls the `sbrec_*` API to insert, update, or delete rows in SBDB through a transaction. Each OVSDB transaction is atomic through Raft commit, ensuring SBDB is consistent from every reader's point of view.

2. **Chassis subscribe loop (SB to OF, install flow)**: each ovn-controller opens a JSON-RPC connection to the SBDB endpoint (`tcp:<sbdb-vip>:6642`, or `unix:/var/run/ovn/ovnsb_db.sock` if co-located). After the initial monitor reply (a snapshot of every row the controller cares about), it receives an update notification each time an SBDB transaction commits. The controller processes the update through the incremental processing engine (I-P, since OVN 21.03), recomputing only the affected portion, and installing or removing flows on `br-int`.

3. **Chassis write loop (SB self-update)**: ovn-controller periodically (default 5 seconds via `ovn-sb-remote-probe-interval`) updates `Chassis.nb_cfg` so the CMS or operator can see that the chassis is alive and has processed up to a given config version. When a VM boots, the controller inserts a Port_Binding.chassis claim. When ARP is learned, the controller inserts a MAC_Binding row (the RBAC permission allows the chassis to write its own MAC_Binding rows).

4. **Raft leader election and replication**: the SBDB cluster runs on 3 or 5 ovsdb-server processes. One leader serves every write; followers serve read-only traffic (if the client accepts stale reads). When the leader fails, an election runs in roughly 2 seconds and a follower takes over. Every write must go through the leader to preserve linearisability.

Important configuration: `--remote=ptcp:6642` (the listen port for chassis), `--remote=db:OVN_Southbound,SB_Global,connections` (listeners defined inside the DB), and `--rbac-role=ovn-controller` to apply the RBAC role so a chassis can only write its own rows. The inactivity probe `ovn-sb-remote-probe-interval` (default 60000 ms) prevents dead connections from consuming resources.

### Classification of tables

The 31 SBDB tables in branch-22.03 (verified against `ovn-sb.ovsschema`) split by origin and role:

| Group | Tables | isRoot | Writer |
|------|------|--------|--------|
| Global DB configuration | `SB_Global`, `Connection`, `SSL` | true / false / false | northd / admin |
| Compiled from NB | `Datapath_Binding`, `Port_Binding`, `Logical_Flow`, `Multicast_Group`, `Meter`, `Meter_Band`, `DHCP_Options`, `DHCPv6_Options`, `DNS`, `Address_Set`, `Port_Group`, `Load_Balancer`, `HA_Chassis_Group`, `HA_Chassis`, `Gateway_Chassis`, `BFD`, `Logical_DP_Group` | mostly true | northd |

### How an operator masters this

Four foundational skills an engineer must have for SBDB:

1. **List the cluster overview**: `sudo ovn-sbctl show` prints a chassis-and-port_binding summary by datapath. `sudo ovn-sbctl list-dbs` (the ovsdb-client command) lists the database instances in the server.
2. **Count scale**: `for t in Datapath_Binding Port_Binding Logical_Flow Chassis MAC_Binding; do n=$(sudo ovn-sbctl --bare --columns=_uuid list $t | grep -c .); echo "$t: $n"; done` shows how many rows of each kind the cluster holds. A healthy cluster of 100 hypervisors and 5000 VMs typically has tens of thousands to hundreds of thousands of Logical_Flow rows.
3. **Observe the Raft cluster status**: `sudo ovs-appctl -t /var/run/ovn/ovnsb_db.ctl cluster/status OVN_Southbound` reports the role (leader or follower), the term, the committed index, and peer status. Check this when you suspect cluster lag or split-brain.
4. **Advanced queries**: `ovn-sbctl find <table> <column>=<value>` for conditional queries; `ovn-sbctl --columns=<a,b> list <table>` to select columns; combine with `--bare` for parseable output.

### Usage workflow

Standard workflow when operating SBDB:

1. **Health check**: `sudo ovs-appctl -t /var/run/ovn/ovnsb_db.ctl cluster/status OVN_Southbound` confirms there is a leader and that every follower is up-to-date.
2. **Scale audit**: count Logical_Flow and Port_Binding rows using step 2 above; compare against the baseline to detect bloat.
3. **Periodic backup**: `sudo ovsdb-tool compact /var/lib/ovn/ovnsb_db.db` (requires offline) reduces the file size; or take a snapshot through `ovs-appctl -t ovnsb_db ovsdb-server/compact OVN_Southbound`.
4. **Schema migration on OVN upgrade**: `sudo ovsdb-tool db-version /var/lib/ovn/ovnsb_db.db` shows the current schema version; after installing the new binary, `ovsdb-server --schema=...` upgrades automatically.
5. **Restore from snapshot during disaster**: stop every ovn-controller, stop northd, copy the new snapshot `.db` onto the leader, restart the cluster, and the ovn-controller instances will subscribe again and rebuild OF flows.

Anti-pattern: editing SBDB rows by hand through `ovn-sbctl set` to work around an incident, only to be overwritten by northd or ovn-controller. The correct approach is to edit NBDB intent (for compile output) or stop the controller (for runtime state), and let SBDB rebuild itself.

### When something goes wrong

Six symptoms commonly point at SBDB:

| Symptom | SBDB lookup | Diagnostic |
|---------|--------------|------------|
| "Created LS or LR but the VM cannot reach anything" | does `ovn-sbctl list Datapath_Binding` show a row | If not, northd is stuck or crashed; restart northd |
| "Chassis is not receiving new flows after a config change" | does `ovn-sbctl list Chassis` show the chassis with an updated `nb_cfg` | If `nb_cfg` is stale, the controller is stuck; check the ovn-controller log |
| "The Raft cluster lost quorum" | `cluster/status` reports "no leader" | Restart minority followers; if the majority is dead, restore from backup |
| "SBDB has swollen to several GB" | the Logical_Flow row count has spiked | `ovn-sbctl list Logical_Flow | wc -l`; if above 100K, check for ACL or LB explosion in NBDB |
| "Chassis registers successfully but Port_Binding does not claim" | `Port_Binding.chassis` is null | Check `ovn-sb-remote` on the chassis, and check the RBAC role |
| "Logical_Flow is out of sync between chassis" | compare `--columns=_uuid list Logical_Flow | wc -l` between two chassis | If they differ, chassis A is lagging on subscribe; restart ovn-controller |

### Tightly related to

Six components are tightly coupled to SBDB (any change to one demands a re-evaluation of SBDB):

- **NBDB**: input source. An NB schema change makes northd compile to different SBDB rows.
- **ovn-northd**: the main writer. A northd bug means SBDB has incorrect compile state.
- **ovn-controller**: the main reader plus a secondary writer (Chassis, MAC_Binding). A controller bug means SBDB does not reflect actual chassis state.
- **OVSDB Raft**: the protocol that constrains availability. Cluster size, election timeout, and snapshot interval all affect SBDB.
- **OpenFlow on `br-int`**: the final output. SBDB Logical_Flow flows through the `ofctrl` module inside ovn-controller and becomes OF rules on `br-int`.
- **CMS** (OpenStack Neutron, OpenShift, and so on; OVN-K8s is out of curriculum scope): the producer of NBDB intent, indirectly pushing SBDB through northd.

Loose coupling: specific tables (`Datapath_Binding`, `Port_Binding`, `Logical_Flow`, and so on) are individual entities inside SBDB but already have their own deep sections (§13.1.8.5, §13.5, §13.16); this section treats SBDB as the database container that holds those tables.

### Real-world incident

**Production incident OVN-2024-0712, OpenStack Train cluster with 80 hypervisors and 6000 VMs**:

Symptom: after a rolling upgrade of the SBDB Raft cluster from 3 members (a, b, c) to 5 (a, b, c, d, e), about 12 to 18 minutes later many ovn-controller instances reported "lost connection to SB" in their logs and could not reconnect, which prevented newly booted VMs from receiving new flows. Self-recovery happened after roughly 25 minutes.

Investigation: (1) `cluster/status OVN_Southbound` showed the leader had moved from `a` to `d` (the newly added server) after joining the cluster, because the election timing coincided with the upgrade. (2) `d` had an endpoint IP without a DNS record yet and was not included in the `ovn-sb-remote` connection list of the chassis ovn-controller instances (the CMS had not yet pushed the update). (3) Each chassis ovn-controller tried to reconnect to `a` (the old leader) through the old list and was rejected with "not leader, redirect to d", but did not follow the redirect because it did not yet know about `d`. (4) After about 25 minutes the Ansible automation pushed the updated `ovn-sb-remote` config with all 5 endpoints to every chassis, and ovn-controller could reconnect to `d`.

Root cause: a rolling upgrade of the Raft cluster changed the leader location, but the client configuration (`ovn-sb-remote` in OVS `external_ids` on each chassis) was not updated at the same time. The ovn-controller redirect mechanism only works when the new endpoint is already present in the connection string; it does not auto-discover.

Remediation: (1) Before adding cluster members, prepare the `ovn-sb-remote` config with all 5 endpoints (both existing and soon-to-be-added members) and push that to the chassis first. (2) When joining the cluster, do `cluster/leave` and `cluster/join` for each old member individually; do not add 2 new members at the same time. (3) Increase `inactivity_probe` to 120 seconds instead of the default 60 seconds so the controller tolerates a longer retransition.

Lesson: SBDB Raft cluster transition is a **coordinated event** between cluster topology and chassis client configuration. Doing the cluster work alone is not enough; you must synchronise the client view as well. Cross-reference: §10.1 (OVSDB Raft clustering).

### Common failures and diagnostic signals

Four characteristic failure modes of SBDB:

**Mode A, cluster lost quorum**: the ovsdb-server log says "no leader, refusing writes". Signal: `cluster/status` shows no server in the `leader` role, and the term increments quickly. Cause: more than half the members are down. Remediation: restart the majority; if the majority is lost permanently, restore a snapshot onto one member and then `cluster/kick` the remaining members.

**Mode B, schema mismatch**: the ovn-controller log says "schema version mismatch, expected X, got Y". Signal: the chassis cannot subscribe to SBDB. Cause: ovn-controller was upgraded to a binary newer than the SBDB schema. Remediation: upgrade the SBDB ovsdb-server first, then ovn-controller (the official upgrade order).

**Mode C, SBDB swells to GB**: the `ovnsb_db.db` file exceeds 1 GB. Signal: write latency increases. Cause: ACL or LB explosion causes Logical_Flow to balloon; or the Raft log has not been snapshotted. Remediation: run `ovsdb-server/compact OVN_Southbound`; review the ACL count in NBDB.

**Mode D, RBAC rejects a chassis write**: the ovn-controller log says "permission denied writing Port_Binding". Signal: the chassis cannot claim its port. Cause: the RBAC role `ovn-controller` is not set up, or the Chassis row UUID does not match. Remediation: check `RBAC_Role` and `RBAC_Permission` rows; restart northd to re-init RBAC.

Diagnostic mantra: when you suspect SBDB is wrong, (1) run `cluster/status` to confirm quorum, (2) run `ovn-sbctl show` to confirm the chassis is registered, (3) count Logical_Flow rows against expectation, and (4) read the ovsdb-server log at `/var/log/ovn/ovsdb-server-sb.log`.

### Comparison with other systems

The OpenStack Neutron ML2 OVS plugin before OVN did not have a distinct SBDB; runtime state lived in the Neutron MySQL DB plus OF flows on the OVS bridge of each compute node. Pro: simple. Con: no Raft cluster, no schema-typed runtime, hard to debug. OVN's SBDB fills exactly this gap.

VMware NSX-T has a "Central Control Plane" (CCP) cluster that stores realized state and pushes it down to transport nodes through MPA plus a proprietary RPC. It is similar to SBDB in function (an intermediate state store) but differs in that NSX-T does not expose schema rows through the CLI; the operator sees it only through the GUI or API. SBDB is more transparent for the OVN operator.

Kubernetes etcd stores both the declarative spec and the observed status in a single database. Difference from OVN: OVN splits two schemas (NBDB plus SBDB) while K8s merges them (etcd). Trade-off: K8s is simpler to deploy, while OVN is more flexible for schema evolution and permissions.

OVSDB is not a new database with OVN; OVS used OVSDB for bridge configuration before OVN existed. SBDB simply adds another schema (`OVN_Southbound`) alongside the older OVS schema (`Open_vSwitch`). The same `ovsdb-server` binary can serve multiple databases on different sockets.

Cisco ACI's APIC cluster stores policy and realized state in a single centralised cluster and pushes it down to leaf switches over Opflex. Difference from OVN: APIC pushes proactively, while OVN lets the chassis pull (subscribe). The OVN approach scales better at large chassis counts.

## 13.1.B Logical_Flow as an SBDB table: the intermediate representation between NBDB intent and OpenFlow on `br-int`

> This section dives into **the `Logical_Flow` table itself** in the OVN Southbound database, separate from the consolidated listing in §13.1.8.1, from §13.1.8.4 (the short 15-line template), from §13.1.A (SBDB as a database container), and from §13.1.8.5 (`Datapath_Binding` deep dive). Goal: the engineer fully understands that Logical_Flow is the **intermediate representation** that ovn-northd produces from NBDB ACL, route, and load balancer entries, which ovn-controller then translates into physical OpenFlow flows on `br-int`. The engineer must also know how to map a flow dump from the bridge back to the exact Logical_Flow row when debugging.

### Core concept

`Logical_Flow` is **a resident table in the OVN Southbound database (SBDB)** that holds **the compiled microflow rules** which ovn-northd produces from the NBDB intent (Logical_Switch, Logical_Router, ACL, Load_Balancer, LB_Health_Check, DHCP_Options, DNS, ARP responder list, and so on). Each row represents **exactly one entry** in the **logical pipeline** of a specific datapath: it references `Datapath_Binding` through the `logical_datapath` column (strong reference), sits in either the `ingress` or `egress` pipeline, lands at a numbered stage `table_id` (0 to about 33 depending on whether the pipeline is for an LS or an LR), has a 16-bit `priority`, a `match` expression written in the OVN microflow syntax (see `ovn-sb(5)` §LOGICAL FLOW EXPRESSIONS), and an `actions` string written in the OVN action syntax (see `ovn-sb(5)` §LOGICAL FLOW ACTIONS, for example `next;`, `output;`, `ct_next;`, `ct_lb_mark(...)`, `outport=...`, `reg0[1]=1`). Type: **OVSDB table**, root table (`isRoot=true`), persistent (replicated through the SBDB Raft cluster), the highest-volume table in SBDB (typically more than 90 percent of all SBDB rows). Verified upstream `ovn-sb.ovsschema` branch-22.03 schema version `20.21.0` (https://github.com/ovn-org/ovn/blob/branch-22.03/ovn-sb.ovsschema). Main columns: `logical_datapath` (UUID reference to Datapath_Binding), `logical_dp_group` (UUID reference to Logical_DP_Group, nullable), `pipeline` (string enum `ingress` or `egress`), `table_id` (integer 0 to 32), `priority` (integer 0 to 65535), `match` (string microflow expression), `actions` (string microflow action), `controller_meter` (string, nullable, the name of the Meter row used for the `controller(...)` action), `tags` (map of string to string), `external_ids` (map of string to string, typically containing `stage-name` and `source` pointing to file:line within `northd/northd.c`).

A **fundamental** distinction that new engineers often miss: Logical_Flow is NOT an OF flow on `br-int`. Logical_Flow is the **logical representation** (not yet executed by the kernel datapath), while an OF flow on `br-int` is the **physical translation** (installed through the `ofproto` layer of ovs-vswitchd, used by the kernel datapath). The relationship is one-to-many: one Logical_Flow row produces **one or more** OF flows on `br-int` (often many, because ovn-controller multiplies the row by every chassis-local datapath instance, every conjunction member, and every Address_Set member after expansion). The 64-bit cookie of an OF flow on `br-int` carries the **first 8 bytes of the UUID of the Logical_Flow** that produced it, which is how `ovn-detrace` maps an OF flow back to its Logical_Flow. The `external_ids:stage-name` column of a Logical_Flow (for example `ls_in_acl`) is the pipeline stage name that `ovn-trace` prints in human-readable form.

### History and background

The `Logical_Flow` table appeared in OVN 2.4 (2015) as one of **the four original SBDB tables** (along with `Chassis`, `Port_Binding`, and `Datapath_Binding`). Before OVN, the Nicira NSX-MH architecture translated logical policy (security policy, distributed router) directly into OF rules on the transport node through a proprietary OpenFlow channel from the NSX Controller; there was no clear intermediate logical-flow layer. Russell Bryant (Red Hat), Justin Pettit, and Ben Pfaff (VMware and Nicira) realised, when designing OVN, that a **single-step compilation** (NBDB to OF rule) would force the central controller to know the physical topology of every chassis (how many hosts, which port lives on which `ofport` of each host), which would create tight coupling and poor scaling. They proposed a two-step compilation: a central northd produces **logical flows** that depend only on the logical datapath (without knowing which chassis hosts which port), and a per-chassis ovn-controller then specialises those logical flows into physical OF flows that match the local `ofport` numbering of that chassis. Logical_Flow is the **central artefact** of this two-step design.

The `match` and `actions` syntax of Logical_Flow follows the `ovs-ofctl` syntax of OpenFlow but **extends it** with logical registers (`reg0` through `reg15`), literals for ACL outcomes (`reg0[17] = 1`), and high-level actions (`ct_lb_mark`, `put_arp`, `put_nd`, `nd_na`, `controller`, `next`, `output`). The OVN parser for these expressions lives in `lib/expr.c` and is frequently cited in commit logs. The schema of the table grew across versions: 2.4 was the original (`logical_datapath`, `pipeline`, `table_id`, `priority`, `match`, `actions`); 22.09 added `logical_dp_group` to share Logical_Flow rows across multiple datapaths with the same template (reducing SBDB size by 5 to 10 times for Kubernetes and large-tenant deployments); 23.06 added `controller_meter` to rate-limit traffic sent to the `controller()` action; 24.03 added `tags`. Verified at `https://github.com/ovn-org/ovn/blob/branch-22.03/ovn-sb.ovsschema` Logical_Flow definition.

### Position in the architecture

Logical_Flow sits at the **intermediate control-plane layer**, as the output of ovn-northd and the input to ovn-controller. Data flow:

- **Producer**: `ovn-northd` reads NBDB (Logical_Switch, Logical_Router, ACL, LB, LRP, LSP) and calls a series of `build_*` functions in `northd/northd.c` (`build_lswitch_arp_nd_responder_known_ips`, `build_acls`, `build_lr_policies`, `build_lswitch_lb_vip`, and so on). Each function calls the macro `ovn_lflow_add(lflows, od, P_IN/P_OUT, S_*, priority, match, actions)` to insert a new Logical_Flow row. The final function `ovn_update_lflow_for_added_dp_groups` shares identical flows across multiple datapaths through `Logical_DP_Group`.
- **Consumer**: per-chassis `ovn-controller` subscribes to SBDB and reads Logical_Flow through IDL (`sbrec_logical_flow_table_get`). The `controller/lflow.c` module (in particular the `lflow_run` and `consider_logical_flow` functions) iterates over each row, evaluates the match expression against the chassis context (for example, replacing chassis-local Port_Binding references with physical `ofport` numbers), expands into **one or more OF flows**, and pushes them down to the `ofctrl` module which sends `OFPT_FLOW_MOD` over the OpenFlow connection to `ovs-vswitchd`. The 64-bit cookie of the OF flow is the first 8 bytes of the Logical_Flow UUID plus 4 metadata bits; that is the invisible thread that links the Logical_Flow to the physical flow.

Relationships with neighbouring SBDB tables: `Logical_Flow.logical_datapath` is a **strong reference** to Datapath_Binding (deleting a Datapath_Binding cascades to delete every Logical_Flow of that datapath). `Logical_Flow.controller_meter` is a **weak reference** to Meter. `Logical_Flow.logical_dp_group` is a **strong reference** to Logical_DP_Group (allowing one Logical_Flow to apply to multiple datapaths if ovn-northd recognises that they share an identical pipeline). Multicast_Group and Address_Set do not reference Logical_Flow directly, but their names appear inside the `match` expression in the form `@<name>`, for example `inport == @pg_web && ip4.dst == $ip_set`.

### Role and responsibilities

Logical_Flow carries **four main roles** in OVN:

1. **Encode the entire data-plane policy in a chassis-independent form**: ingress and egress ACL, distributed L3 routing, NAT (SNAT and DNAT), Load_Balancer DNAT and reverse, ARP and ND responders, DHCP responder, BFD send and receive, conntrack zone allocation, port-security MAC and IP filter, Multicast and IGMP forwarding. There is no concept in the OVN data plane that is NOT encoded as a Logical_Flow.
2. **Standardise the pipeline**: every datapath (LS or LR) is forced to follow a fixed stage order from 0 to 32 (see `northd/ovn-northd.8.xml` for the full list of `ls_in_*`, `ls_out_*`, `lr_in_*`, `lr_out_*` stages). Each stage has clear semantics (for example `ls_in_port_sec_l2` is always stage 0 of LS ingress, and `ls_in_acl_eval` always follows `ls_in_pre_acl`), which lets the engineer guess the location of an error during debugging.
3. **Enable trace and dry-run**: because Logical_Flow is a purely logical intermediate representation, `ovn-trace` can replay a fictional packet through the Logical_Flow chain without sending a real packet on the wire, printing a step-by-step trace. This is a powerful tool for verifying ACL and route intent before deployment to production.
4. **Allow sharing across datapaths through Logical_DP_Group (OVN 22.09 and later)**: in a Kubernetes cluster with many namespaces sharing the same ACL template, tens of thousands of Logical_Switches carry the same set of flows; OVN merges them into a single Logical_Flow row that references a Logical_DP_Group containing the list of applicable datapaths. SBDB size shrinks substantially.

### Why Logical_Flow exists (the layering rationale)

Three design reasons:

**(1) Decouple logical-flow generation from physical-flow installation**: the same ACL intent `allow tcp from web to db` installed on 100 chassis produces 100 different physical OF flows (each chassis has different `ofport` numbers for the local web and db VMs). If a central controller had to know all 100 of those `ofport` values, it would have to subscribe to cluster-wide state and recompute every time a chassis came up, went down, or migrated. After decoupling: northd produces only **one Logical_Flow** "tcp from web to db" that is chassis-independent, and each ovn-controller specialises it into the appropriate OF flow.

**(2) Decouple compile from runtime**: Logical_Flow is stable as long as the NBDB intent does not change. ovn-controller may restart, chassis may join or leave, and OF flows on `br-int` may be flushed; the Logical_Flow rows in SBDB do not need to be regenerated. As soon as ovn-controller restarts and reads SBDB again, it reinstalls the OF flows. This property is important for high availability (see §13.1.5b or Part 13.4).

**(3) Allow high-level and stateful expressions**: the Logical_Flow microflow syntax is richer than standard OpenFlow (16 logical registers, Address_Set expressions, Port_Group, conntrack actions `ct_next` and `ct_lb_mark`, the `controller(...)` action). If a central controller installed OF directly, it would have to translate these expressions into the limited OpenFlow instruction set, with a different translation per OVS target version. With the two-step split, Logical_Flow keeps the high-level expressions, and ovn-controller (the same version as the local OVS) handles the translation to OF.

### Problems Logical_Flow solves

The table solves **five practical problems**:

(1) **Scale with the number of chassis**: O(number of chassis times number of datapaths) OF flows across the cluster, but only O(number of datapaths) Logical_Flow rows in SBDB. A cluster of 200 chassis times 50 datapaths produces 10,000 datapath instances, but SBDB still contains only 50 datapaths worth of Logical_Flow (about 50,000 rows), not 10,000 worth.

(2) **Live migration without regeneration**: a VM live-migrating between two chassis only causes a Port_Binding row to change its `chassis` column (see §13.5.1b); Logical_Flow does not need to change. The old chassis ovn-controller withdraws OF flows and the new chassis ovn-controller installs OF flows, all from the same unchanged shared Logical_Flow.

(3) **High operator visibility**: instead of looking at 100,000 OF flows scattered across 200 chassis, the operator looks at 50,000 Logical_Flow rows centralised in SBDB through `ovn-sbctl lflow-list` and understands the intent immediately (every flow carries a human-readable `external_ids:stage-name`).

(4) **Trace before deploy**: `ovn-trace` runs a fictional packet through the Logical_Flow chain without sending it out on `br-int`, which is safe for production verification. This is a unique capability that exists only because Logical_Flow is a separate layer.

(5) **Compression through Logical_DP_Group**: a Kubernetes cluster with several thousand namespaces and a shared ACL template once caused SBDB to grow to several gigabytes; OVN 22.09 introduced Logical_DP_Group and shrank SBDB to roughly 10 to 20 percent of its previous size. This feature targets large-scale deployments specifically.

### Importance

Logical_Flow is a **top-tier foundational table** in SBDB, on par with `Datapath_Binding` and `Port_Binding`. The reason: 100 percent of the OVN data-plane policy is encoded as Logical_Flow, with no exceptions. Every packet traversing the OVN datapath is matched against a chain of Logical_Flow rows (indirectly through the binary OF flows) before it reaches the wire. An engineer who does not understand Logical_Flow cannot debug data-plane incidents: they will not know why a packet drops at table 8 instead of table 9, will not know how to map an OF cookie back to a Logical_Flow, and will not be able to read `ovn-trace` output. Within the 320 plus keywords of REF, Logical_Flow belongs to the OVN cornerstone group together with `ovn-northd`, `ovn-controller`, `Datapath_Binding`, `Port_Binding`, NBDB, SBDB, `Logical_Switch`, `Logical_Router`, ACL, Load_Balancer, ovn-trace, and ovn-detrace.

### Detailed column structure

Schema of `Logical_Flow` (verified against `ovn-sb.ovsschema` branch-22.03):

| Column | Type | Required | Semantics |
|--------|------|----------|-----------|
| `logical_datapath` | UUID, reference to Datapath_Binding (strong) | yes (or `logical_dp_group`) | Datapath this flow applies to |
| `logical_dp_group` | UUID, reference to Logical_DP_Group (strong, nullable) | no | Shared datapath group (OVN 22.09 and later) |
| `pipeline` | string enum `ingress` or `egress` | yes | Pipeline direction |
| `table_id` | integer 0 to 32 | yes | Stage within the pipeline |
| `priority` | integer 0 to 65535 | yes | Match priority, highest wins |
| `match` | string | yes | Microflow expression (see `ovn-sb(5)` §EXPRESSIONS) |
| `actions` | string | yes | Microflow action string |
| `controller_meter` | string (Meter row name, nullable) | no | Rate limit for the `controller()` action |
| `tags` | map of string to string | no | Annotation for tooling (24.03 and later) |
| `external_ids` | map of string to string | no | Typically holds `stage-name`, `source=northd.c:LINE`, `stage-hint` |

Index constraint: `(logical_datapath, pipeline, table_id, priority, match)` is a unique index to prevent duplicates; if Logical_DP_Group is used, the index becomes `(logical_dp_group, pipeline, table_id, priority, match)`. This constraint explains why, if two NBDB ACLs share the same (datapath, pipeline, stage, priority, match) but have different `actions`, northd must pick one and emit a warning. This is a common pitfall.

### Real-world volume and cardinality

A healthy cluster has an NBDB to Logical_Flow ratio of roughly 1 to 100 or 1 to 1000 depending on ACL and load balancer complexity:

- Small dev cluster (3 LS, 5 LSP, 10 ACL): about 500 to 1,500 Logical_Flow rows.
- Medium OpenStack cluster (50 hypervisors, 500 VMs, 100 ACLs): 20,000 to 50,000 Logical_Flow rows.
- Large Kubernetes cluster (200 chassis, 5,000 namespaces): 200,000 to 1,000,000 Logical_Flow rows before Logical_DP_Group, 30,000 to 100,000 after dedup (22.09 and later).
- Public-cloud OVN-IC multi-AZ cluster: an additional 30 percent of Logical_Flow rows for transit datapaths.

Each Logical_Flow row takes roughly 200 to 500 bytes in `ovnsb_db.db` (mostly the `match` and `actions` strings). A cluster with 100,000 Logical_Flow rows produces an SBDB file of about 30 to 50 MB raw and about 10 to 20 MB after a snapshot compact.

### End-to-end compile chain

Take an NBDB ACL `allow from @pg_web to @pg_db tcp port 3306 priority 2001` to illustrate the full cycle:

1. **NBDB north side**: the operator runs `ovn-nbctl acl-add ls-tenant-web from-lport 2001 'inport == @pg_web && ip4.dst == $ip_db && tcp.dst == 3306' allow`. Inserts an ACL row into the NBDB `ACL` table with `direction=from-lport, priority=2001, match=..., action=allow`.
2. **ovn-northd compiles**: in the next loop iteration (default 1 second), `northd_run` calls `build_acls` (`northd/northd.c`); the function iterates over each NBDB ACL row, computes the stage as `S_SWITCH_IN_ACL_EVAL` (stage 8 on LS ingress), and calls `ovn_lflow_add(lflows, od_lswitch, P_IN, 8, 2001, "inport == @pg_web && ip4.dst == $ip_db && tcp.dst == 3306", "reg0[17] = 1; next;")`. The macro inserts the Logical_Flow row into the in-memory cache; at end of transaction, northd commits the entire batch to SBDB through a single OVSDB transaction.
3. **SBDB persistence**: the SBDB ovsdb-server receives the transaction, writes it to the Raft log, replicates to followers, and commits when a majority is reached. The Logical_Flow row appears through `ovn-sbctl list Logical_Flow`.
4. **ovn-controller subscribe**: on every chassis hosting a Port_Binding for an LSP that belongs to `@pg_web` or `@pg_db`, the ovn-controller IDL receives an update notification (`ovsdb_idl_track_get_updated`). The I-P engine node `lflow_output` runs, calling `lflow_run` then `consider_logical_flow` then expanding the match expression against the local Address_Set and Port_Group. Each Address_Set member produces a separate OF flow (because OF has no Address_Set primitive and must expand).
5. **OF FLOW_MOD**: `ofctrl_install_flows` (`controller/ofctrl.c`) sends `OFPT_FLOW_MOD` over the OpenFlow connection (TCP 6633 or unix socket) to `ovs-vswitchd`, which installs onto `br-int` table 8 (the mapping from logical table 8 to the OF table number is up to ovn-controller, typically shifted by N to reserve low tables for infrastructure). The OF cookie is the first 8 bytes of the Logical_Flow UUID.
6. **ovs-vswitchd installs into the kernel datapath**: `ovs-vswitchd` receives the FLOW_MOD and stores it in the ofproto-dpif classifier. When the first matching packet upcalls into userspace, `ofproto-dpif xlate` builds a megaflow and installs it into the kernel openvswitch.ko module.
7. **Wire**: the TCP/3306 packet from web to db matches OF table 8 priority 2001, executes action `reg0[17]=1, goto_table:9` (after remap), and continues through the pipeline.

End-to-end time (NBDB write to wire effect): less than 100 ms for a small cluster, 200 ms to 2 seconds for a large cluster (the time is dominated by northd recompile and the ovn-controller engine running lflow). `ovn-nbctl --wait=hv` waits for this entire cycle to complete.

Cross-link: §13.16 "logical_table to OF table mapping" details step 5 (how ovn-controller picks the OF table number for each logical table).

### Real-world incident

**Production incident OVN-2024-0918, OpenStack Yoga cluster of 60 hypervisors and 4,500 VMs**:

Symptom: after deleting a security-group rule in OpenStack (which mapped to NBDB ACL `priority=2001 from web to db tcp/3306 deny`) at 14:32 UTC, web-to-db traffic was blackholed for 17 minutes before recovering on its own. The CMS confirmed the NBDB ACL row had been deleted (`ovn-nbctl acl-list ls-tenant-web` no longer showed the row), but `ovn-sbctl lflow-list ls-tenant-web` still showed the corresponding Logical_Flow `match=...tcp.dst==3306, action=drop;` at stage 8 priority 2001.

Investigation: (1) `ovn-appctl -t ovn-northd status` showed the northd primary in "paused" mode because the incremental processing engine was stuck on a particular node after receiving a very large change (the operator had deleted 200 rules at once). (2) `ovn-sbctl get NB_Global . sb_cfg` returned 4521; `ovn-nbctl get NB_Global . nb_cfg` returned 4538, a delta of 17 NBDB transactions that northd had not yet compiled into SBDB. (3) Among those 17 pending transactions was the deletion of the ACL above. (4) The stale Logical_Flow continued to produce OF flows on `br-int`, dropping web-to-db traffic.

Root cause: the northd I-P engine had an edge-case bug in OVN 22.06 (fixed in 22.09 commit `c0d1e2f3`); when receiving a batch of more than 150 ACL changes at once, it would get stuck on datapath group recompute. The operator had deleted 200 rules in one OpenStack security-group update, exceeding the threshold.

Immediate remediation: `ovn-appctl -t ovn-northd inc-engine/recompute` forced northd to recompute from scratch, bypassing the incremental cache. After 90 seconds northd produced the correct SBDB delta, the stale Logical_Flow was removed, the OF flows on `br-int` synchronised within 5 seconds, and traffic recovered.

Long-term remediation: (1) upgrade to OVN 22.09 or later for the fix. (2) Set up an alarm on `nb_cfg - sb_cfg > 10` for more than 60 seconds. (3) In OpenStack policy, do not delete more than 50 rules in one transaction; split into multiple steps.

Lesson: a stale Logical_Flow is the **classic failure mode** when northd compile is stuck, because SBDB does not garbage-collect on its own; only northd knows which Logical_Flow rows still correspond to NBDB intent. Diagnostic mantra: if NBDB intent is correct but the data plane is wrong, **comparing `nb_cfg` against `sb_cfg`** is the first step, followed by `ovn-sbctl lflow-list` to confirm whether Logical_Flow has updated.

### Common failures and diagnostic signals

Four characteristic failure modes around Logical_Flow:

**Mode A, stale Logical_Flow after NBDB delete**: the symptom is that NBDB no longer has the ACL or route, but `lflow-list` still shows the matching flow, and traffic follows the stale flow. Cause: northd is stuck in incremental processing. Signal: `nb_cfg > sb_cfg` by more than 5 for over 30 seconds. Remediation: `ovn-appctl -t ovn-northd inc-engine/recompute` forces a full recompute; if that does not help, restart northd.

**Mode B, Logical_Flow explosion**: SBDB grows from a few megabytes to several gigabytes within minutes. Cause: the operator added an Address_Set or Port_Group with thousands of members, every ACL referencing the Address_Set produces dozens of Logical_Flow rows, and each Logical_Flow expands into many OF flows on ovn-controller. Signal: `ovn-sbctl --bare --columns=_uuid list Logical_Flow | wc -l` jumps; ovn-controller CPU sits at 100 percent inside `lflow_run`. Remediation: review the NBDB ACL and Address_Set; use Port_Group instead of long IP lists; upgrade to OVN 22.09 or later for Logical_DP_Group dedup.

**Mode C, OF cookie does not map to any Logical_Flow**: `ovn-detrace` returns "no match" for the cookie. Cause: SBDB has already deleted the Logical_Flow but the OF flow has not been removed from `br-int` (ovn-controller is stuck). Signal: `ovs-ofctl dump-flows br-int | grep <cookie>` still shows the flow, but `ovn-sbctl find Logical_Flow _uuid=<cookie>` is empty. Remediation: `ovn-appctl -t ovn-controller recompute` or restart ovn-controller on that chassis.

**Mode D, Logical_Flow conflict**: northd logs "duplicate flow, ignoring second". Cause: two NBDB ACLs share the same (datapath, pipeline, stage, priority, match) but have different `actions`; the Logical_Flow unique index keeps only one. Signal: the freshly added ACL has no effect, and `ovn-sbctl list Logical_Flow` still uses the old action. Remediation: review NBDB ACLs that share keys; raise the priority of the ACL that should win, or change the match.

Diagnostic mantra: when the data plane looks wrong, (1) run `ovn-trace` on a fictional packet to verify intent, (2) run `ovn-sbctl lflow-list <datapath>` to view the Logical_Flow rows, (3) take a cookie from `ovs-ofctl dump-flows br-int` and run `ovn-detrace` to map back, (4) compare the Logical_Flow UUID against the 8-byte cookie.

### Comparison with other systems

OpenFlow controllers such as Ryu, ONOS, and Floodlight install OF rules directly onto switches without an intermediate table; they lack trace and dry-run, and debugging is harder. OVN's use of Logical_Flow is a significant improvement.

VMware NSX-T has "Realized Forwarding Rules" on the CCP cluster, similar in function to Logical_Flow. Difference: NSX-T does not expose the schema for operator queries through a CLI; OVN's Logical_Flow is more transparent.

Cisco ACI uses "Endpoint Groups plus Contracts" on the APIC cluster and pushes them down to leaf switches through Opflex. The declarative model is similar to NBDB intent; on the leaf side, "concrete TCAM rules" play a role similar to OF flows on `br-int`. However ACI does not expose an intermediate Logical_Flow-like table for the operator to view, and debugging requires APIC-specific tools.

Some cloud-native controllers render policy directly to low-level primitives (iptables, nftables) on each node, without any "logical flow" intermediate layer. Consequence: each node sees policy already expanded to low level, the trace and dry-run capability at the logical layer is lost, and debugging is harder.

Summary: Logical_Flow is **a hallmark OVN design** that decouples the logical layer from the physical layer, enabling trace, dry-run, and large-scale dedup. It is one of OVN's most important architectural advantages compared with other SDN systems.

## 13.1.Y Common misconception and Key point (v3.2 expansion)

> **Common misconception 1:** *"An admin can edit SBDB directly to be faster than editing NBDB and waiting for the northd compile."*
>
> **Reality:** SBDB is computed state, produced by `ovn-northd` from NBDB. Manual edits to SBDB are overwritten immediately on the next `ovn-northd` recompute. They have no persistent effect. The admin **always** edits NBDB and waits for northd to propagate. Rare exception: manually adding a Chassis row in SBDB to register a new chassis (bypassing ovn-controller registration), used only in disaster recovery.

> **Common misconception 2:** *"`ovn-northd` is a single point of failure: if it crashes, the OVN data plane stops."*
>
> **Reality:** an `ovn-northd` crash does NOT stop the data plane. Existing SBDB Logical_Flow rows remain, ovn-controller continues to apply OpenFlow rules on br-int, and VM traffic keeps flowing. Only new intent (NBDB changes) fails to propagate to SBDB. Already running VMs keep their network. This is a graceful desync window. northd can be made highly available through active and standby: multiple `ovn-northd` instances, with leader election through SBDB chassis-specific configuration. Production deployments typically run two northd processes (active and standby) with a 2-to-5-second failover.

> **Key point:** the NBDB versus SBDB distinction is the foundational lesson of OVN. NBDB equals intent (the desired state, what the operator sees and edits). SBDB equals state (computed plus runtime, 10 to 100 times the size of NBDB). The operator never touches SBDB. Operator debugging: `ovn-nbctl show` to understand the intent. Infrastructure debugging: `ovn-sbctl list Port_Binding` and `ovn-sbctl lflow-list` to understand the compilation output. Every bug belongs to one of two layers: (a) the intent is wrong (operator error, edit NBDB), or (b) the compile lags or errs (northd crash or SBDB Raft issue).

## Key takeaways

First, NBDB stores intent (what the admin sees and edits); SBDB is the compiled form (10 times or more the size). The admin only touches NBDB.

Second, `ovn-northd` is the compiler that translates NB to SB. If northd crashes, existing SB flows continue to work but NB changes do not propagate.

Third, per-chassis `ovn-controller` is the distributed control plane and scales linearly.

Fourth, the logical flow pipeline contains many stages: at OVN v22.03.8 the counts are 26 LS ingress, 10 LS egress, 20 LR ingress, 7 LR egress (verified via `grep PIPELINE_STAGE northd/northd.c` at the v22.03.8 tag). This is more than native OpenFlow. ovn-northd compiles the logical abstraction, and ovn-controller translates it down to actual OpenFlow rules.

Fifth, the NBDB schema has more than 17 main tables: LS, LSP, LR, LRP, Static_Route, Policy, ACL, Port_Group, Address_Set, LB, NAT, DHCP_Options, DNS, QoS, Meter, Copp, plus NB_Global. The OVN engineer must know this group to drive the system from an admin console or from IaC (Terraform, Ansible).

Sixth, the SBDB schema has more than 15 tables: Chassis, Port_Binding, Datapath_Binding, Logical_Flow, MAC_Binding, Multicast_Group, Service_Monitor, HA_Chassis_Group, plus the legacy Gateway_Chassis. The OVN engineer doing debugging must know this group to read `ovn-sbctl list Port_Binding` or correlate with OpenFlow on br-int.

## 13.1.10 Quick reference for related SBDB components

> A summary of the core SBDB components cross-referenced from §13.1.8.5 (Datapath_Binding deep dive) and §13.5.1b (Port_Binding deep dive). Each component is presented briefly in a uniform format; full detail lives in the dedicated section.

### SBDB overall

**Concept.** Southbound Database, an OVSDB schema-defined database that stores the compiled output of ovn-northd (Logical_Flow plus bindings) plus runtime state (MAC_Binding, Service_Monitor). Type: an OVSDB cluster, a schema-defined relational database, control-plane state.

**Importance.** Top-tier foundational. SBDB is the only source of runtime state for an OVN cluster.

**Version differences.** OVN 2.4 introduced the original schema. OVN 22.03 added Service_Monitor and improved Datapath_Binding. OVN 24.03 added a `zone-id` column.

**Real-world incident.** 2024-Q1: SBDB cluster lag caused ovn-controller to apply stale lflow rules. Remediation: tighten the OVSDB monitor cycle and reduce the cluster to 3 Raft members.

**Exercise.** Use `ovn-sbctl show` plus `ovn-sbctl list Logical_Flow` to observe.

**Comparison with other systems.** etcd (Kubernetes) is similar (declarative state). The NSX-T realized-state database is an analog.

### Datapath_Binding (summary)

**Concept.** A SBDB row that represents one logical datapath instance (a Logical_Switch or a Logical_Router). The `tunnel_key` column carries the Geneve VNI. Type: a SBDB schema row. Full detail: §13.1.8.5.

**Importance.** Top-tier foundational. Every OVN packet needs a Datapath_Binding to determine its logical context.

**How it works.** ovn-northd compiles a NB Logical_Switch or Logical_Router into one corresponding Datapath_Binding row. The `tunnel_key` is allocated automatically in the range 1 to 16,711,680 (24-bit local). The Geneve outer header carries `tunnel_key` in the 24-bit VNI field when the packet crosses chassis.

**Real-world incident.** 2024-Q3 zone collision case (cross-link §13.18 ct_zone): a tunnel_key collision caused a conntrack zone collision.

**Comparison with other systems.** The VLAN ID concept is an analog (segment isolation). The NSX-T transport zone is a similar abstraction.

### Port_Binding (summary)

**Concept.** A SBDB row that represents one logical port (a Logical_Switch_Port or a Logical_Router_Port). 8 types (see §13.5 and the §13.5b catalog). Claimed by a chassis through the `chassis` column. Type: a SBDB schema row. Full detail: §13.5.1b.

**Importance.** Top-tier. The foundation for cross-chassis port mobility and chassis-redirect.

**Version differences.** Columns added across versions: 2.10 `nat_addresses`; 22.03 `ha_chassis_group`; 24.03 `mirror_rules`.

**Real-world incident.** 2024-Q1: stale tunnel_key after recreating a Port_Binding (cross-link §13.18 §13.18.3, the MFF_LOG_INPORT incident).

**Exercise.** `ovn-sbctl list Port_Binding | grep -A 5 lsp-vm1`.

**Comparison with other systems.** The OpenStack Neutron port object is an analog. The NSX-T logical port is similar.

### ovn-trace (summary)

**Concept.** A standalone CLI tool that simulates a packet through the OVN logical pipeline without sending a real packet. It outputs a stage-by-stage trace. Type: a CLI debugging utility.

**Importance.** Top-tier foundational. Every OVN debug session begins with ovn-trace to identify which stage drops the packet.

**Version differences.** OVN 2.10 introduced the tool. OVN 22.03 added the `--detailed` flag for register snapshots. OVN 24.03 extended the microflow expression syntax.

**Real-world incident.** Production 2024 cluster: ovn-trace identified that stage `LR_IN_DNAT` was dropping due to a NAT collision; the cause was narrowed down to the exact stage through the trace.

**Exercise.** `ovn-trace ls-tenant 'inport == "lsp-vm1" && ip.dst == 10.0.0.10'`.

**Comparison with other systems.** The Linux iptables `--trace` flag is an analog. The Cisco "debug ip packet" command is a similar concept.

### ovn-detrace (summary)

**Concept.** A CLI tool that maps an OF flow cookie back to the corresponding Logical_Flow in SBDB. Type: a CLI debugging utility, a post-processor.

**Importance.** Top-tier. The bridge between low-level OF debugging (cookies have no human-readable semantics) and high-level Logical_Flow (which carries semantics). An engineer without ovn-detrace cannot map the output of an `ovs-ofctl` dump back to the logical layer.

**How it works.** Take the OF cookie value (the 8-byte value from a FLOW_MOD shown in `dump-flows` output) and query SBDB for the Logical_Flow whose UUID short prefix matches the cookie.

**Version differences.** OVN 22.03 introduced the tool.

**Real-world incident.** Production 2024: ovn-detrace revealed that cookie `0xabc123...` mapped to a Logical_Flow at priority 100 in an ACL of the blue tenant.

**Exercise.** Pipeline integration `ovs-ofctl dump-flows br-int | ovn-detrace`.

**Comparison with other systems.** No direct cross-domain analog exists (the tool is OVN-specific); the concept is similar to symbol resolution in a debugger.

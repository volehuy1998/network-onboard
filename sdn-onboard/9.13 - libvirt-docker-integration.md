## 9.13.1 Libvirt + OVS

Compass Ch S pattern: trong domain XML của VM, dùng `<virtualport type='openvswitch'/>`:

```xml
<interface type='bridge'>
  <mac address='52:54:00:aa:bb:01'/>
  <source bridge='br-int'/>
  <virtualport type='openvswitch'/>
  <model type='virtio'/>
</interface>
```

Libvirt tự động:

1. Tạo tap interface (ví dụ `vnet0`).
2. Gọi `ovs-vsctl add-port br-int vnet0`.
3. Gọi `ovs-vsctl set Interface vnet0 external_ids:attached-mac=... external_ids:iface-id=<uuid>`.

`iface-id` là cách OVN (và OpenStack Neutron) discover port của VM cụ thể.

Khắc phục sự cố VIF plug:

```bash
[host]# ovs-vsctl list Interface | grep -A2 vnet
[host]# virsh domiflist <vm-name>
```

## 9.13.2 Docker + OVS

Compass Ch T hai pattern:

**Pattern 1 — ovs-docker helper** (đơn giản):

```bash
[host]# ovs-docker add-port br0 eth0 <container-id> \
         --ipaddress=10.0.0.5/24 --gateway=10.0.0.1
[host]# ovs-docker del-port br0 eth0 <container-id>
[host]# ovs-docker set-vlan br0 eth0 <container-id> 100
```

Script tự tạo veth pair, move vào container namespace, plug vào OVS.

**Pattern 2 — Manual veth** (production):

```bash
[host]# ip link add veth0 type veth peer name veth0-br
[host]# ip link set veth0-br up
[host]# ovs-vsctl add-port br0 veth0-br -- \
         set Interface veth0-br external_ids:container_id=<container-id>
[host]# PID=$(docker inspect -f '{{.State.Pid}}' <container-id>)
[host]# ip link set veth0 netns $PID
[host]# nsenter -t $PID -n ip addr add 10.0.0.5/24 dev veth0
[host]# nsenter -t $PID -n ip link set veth0 up
```

Manual cho control chi tiết (QoS, Mirror, OpenFlow per-veth).


## 9.13.6 libvirt-OVS protocol contract deep

### `external_ids:iface-id` semantics

Khi libvirt plug VM vào OVS bridge với `<virtualport type='openvswitch'/>`, libvirt tự động set `external_ids:iface-id` trên OVS Interface row. Đây là **contract** giữa orchestrator và OVN/Neutron để discover port của VM cụ thể.

Source code anchor (libvirt 8.0+, Ubuntu 22.04):

```c
// src/util/virnetdevopenvswitch.c, function virNetDevOpenvswitchAddPort()
// Pseudo-flow:
//   1. virCommandNew("ovs-vsctl")
//   2. virCommandAddArgList("add-port", brname, ifname, "--", ...)
//   3. virCommandAddArgList("set", "Interface", ifname,
//                            "external-ids:attached-mac=...",
//                            "external-ids:iface-id=...",
//                            "external-ids:vm-uuid=...")
```

**Format `iface-id`:**

- **OVN-Kubernetes**: `<namespace>_<pod-name>` (ví dụ `default_my-app-pod-abc123`).
- **OpenStack Neutron**: UUID Neutron port (ví dụ `12345678-1234-...`).
- **Libvirt direct (no orchestrator)**: thường = MAC hoặc UUID VM.

OVN ovn-controller subscribe SBDB Port_Binding `name = iface-id`. Khi thấy match, claim port + cài flow trên br-int.

### Libvirt event hook (qemu hooks)

Libvirt support qemu hooks tại `/etc/libvirt/hooks/qemu` cho lifecycle event:

```bash
#!/bin/bash
# /etc/libvirt/hooks/qemu
DOMAIN=$1
EVENT=$2

case "$EVENT" in
    prepare|start)
        # VM about to start, ensure OVS bridge exists
        ovs-vsctl --may-exist add-br br-int
        ;;
    stopped|release)
        # VM stopped, clean up orphan OVS port
        for port in $(ovs-vsctl list-ports br-int | grep "vnet"); do
            if ! virsh domiflist "$DOMAIN" 2>/dev/null | grep -q "$port"; then
                ovs-vsctl --if-exists del-port br-int "$port"
            fi
        done
        ;;
esac
```

Hook ensure idempotency: VM start retry không tạo duplicate port; VM stop cleanup orphan veth.

### Hot-plug semantics

VM running, attach NIC mới qua `virsh attach-device <vm> nic.xml`:

```xml
<interface type='bridge'>
  <source bridge='br-int'/>
  <virtualport type='openvswitch'/>
  <model type='virtio'/>
</interface>
```

Libvirt:

1. `qemu-monitor` `device_add virtio-net-pci`.
2. Tạo tap interface (`vnet1`).
3. `ovs-vsctl add-port br-int vnet1 ... external_ids:iface-id=...`.
4. VM kernel detect new NIC, register driver.

Toàn bộ < 1 giây cho cluster healthy. Failure mode: nếu OVS bridge không tồn tại → libvirt warn, NIC vẫn detach. Sửa: tạo bridge trước rồi retry attach.

## 9.13.7 Docker-OVS container netns lifecycle

### Container start workflow

Docker không hiểu OVS. Manual workflow:

```
docker run --network=none → container netns tạo, lo only
                         ↓
        người vận hành/script tạo veth pair
                         ↓
        ovs-vsctl add-port br-int <veth-host>
                         ↓
        ip link set <veth-container> netns <pid>
                         ↓
        nsenter -t <pid> -n ip addr add ...
```

Vấn đề: Docker không biết về veth → restart container → veth orphan trên OVS bridge.

### Container restart pattern

3 restart policy:

- `--restart=no` (default): container exit → docker không restart → orphan veth on OVS.
- `--restart=on-failure`: docker restart container nhưng tạo netns mới (vì PID khác) → orphan veth cũ.
- `--restart=always`: tương tự on-failure.

Healthy production cần cleanup mechanism:

```bash
# Cron daily orphan veth cleanup
[host]# for port in $(ovs-vsctl list-ports br-int | grep -E '^c[0-9a-f]{12}'); do
[host]>     container_id=$(echo "$port" | sed 's/^c//')
[host]>     if ! docker ps -q --no-trunc | grep -q "$container_id"; then
[host]>         echo "Orphan port: $port (container $container_id)"
[host]>         ovs-vsctl --if-exists del-port br-int "$port"
[host]>     fi
[host]> done
```

### Veth orphan failure mode

Symptom production:

```bash
[host]# ovs-vsctl list-ports br-int | wc -l
342

[host]# docker ps -q | wc -l
85

# 342 - 85 = 257 orphan port
```

Cause: container restart loop trong vài tuần, mỗi restart 1 orphan. OVS không tự cleanup.

Fix: cron + monitoring metric. Production OVN-Kubernetes có CNI plugin tự cleanup (ovn-cni ADD/DEL hooks). Standalone Docker thì script tự xây.

### Atomic transaction batch

Khi 100 container restart cùng lúc, race condition khi 100 process chạy `ovs-vsctl add-port` parallel:

```bash
# BAD: race condition
for c in c1 c2 ... c100; do
    ovs-vsctl add-port br-int veth-$c &
done
wait
```

OVSDB transactions có thể conflict, retry, slow.

Healthy pattern: batch single transaction:

```bash
# GOOD: 1 transaction
cmd="ovs-vsctl"
for c in c1 c2 ... c100; do
    cmd="$cmd -- add-port br-int veth-$c"
done
$cmd
```

100 add-port trong 1 transaction commit atomic, < 100ms total thay vì 10s+ với race.

## 9.13.8 Production security baseline expand

### File permission ownership matrix

| Path | Owner:Group | Mode | Lý do |
|------|-------------|------|-------|
| `/etc/openvswitch` | openvswitch:openvswitch | 0750 | Config dir, only daemon + admin read |
| `/etc/openvswitch/conf.db` | openvswitch:openvswitch | 0640 | OVSDB file, contains state |
| `/etc/openvswitch/cacert.pem` | openvswitch:openvswitch | 0644 | Public cert, world-readable |
| `/etc/openvswitch/cert.pem` | openvswitch:openvswitch | 0644 | Public cert |
| `/etc/openvswitch/privkey.pem` | openvswitch:openvswitch | 0400 | Private key, daemon-only read |
| `/var/run/openvswitch` | openvswitch:openvswitch | 0755 | Socket dir |
| `/var/run/openvswitch/db.sock` | openvswitch:openvswitch | 0660 | OVSDB socket, group write |
| `/var/log/openvswitch` | openvswitch:openvswitch | 0750 | Log dir |

Verify:

```bash
[host]# stat -c '%U:%G %a %n' /etc/openvswitch/privkey.pem
openvswitch:openvswitch 400 /etc/openvswitch/privkey.pem
```

### TLS mandatory cho controller/manager

Cross-link Phần 9.10 (TLS PKI tier 2). Quick check:

```bash
[host]# ovs-vsctl list Manager | grep target
target              : "ssl:192.0.2.1:6640"     ← good

[host]# ovs-vsctl list Manager | grep target
target              : "tcp:192.0.2.1:6640"     ← BAD, plaintext
```

Mọi `ptcp:` (passive TCP) hoặc `tcp:` (active TCP) trong production = security violation.

### SELinux module

OVS package ship SELinux policy:

```bash
[host]# rpm -ql openvswitch-selinux | grep '\.pp$'    # RHEL
/usr/share/selinux/packages/openvswitch.pp

[host]# semodule -l | grep openvswitch
openvswitch    1.0
```

Module define context cho:

- `openvswitch_t` cho daemon.
- `openvswitch_var_run_t` cho `/var/run/openvswitch`.
- `openvswitch_etc_t` cho `/etc/openvswitch`.
- Allow daemon `connectto` socket TCP ports 6640, 6641, 6642, 6653.

KHÔNG disable SELinux trong production. Đặc biệt tránh `setenforce 0` "tạm thời".

### AppArmor profile (Ubuntu)

Ubuntu 22.04 ship `/etc/apparmor.d/usr.sbin.ovs-vswitchd`:

```
[host]# aa-status | grep ovs
   /usr/sbin/ovs-vswitchd
   /usr/sbin/ovsdb-server
```

Profile `enforce` mode default. Kiểm chứng:

```bash
[host]# aa-status | head -20
apparmor module is loaded.
Profiles loaded: ...
   In enforce mode: 30
   In complain mode: 0
```

## 9.13.9 Anatomy Template A: `ovs-vsctl list Interface external_ids`

> **Bối cảnh khi chạy:** Kiểm tra external_ids contract trên Interface row, debug khi OVN/orchestrator không claim port hoặc port stuck.

```bash
[host]# ovs-vsctl list Interface vnet0
_uuid               : a1b2c3d4-...
external_ids        : {attached-mac="52:54:00:aa:bb:01", \
                       iface-id="default_my-pod-abc123", \
                       iface-id-ver="789def", \
                       iface-status="active", \
                       neutron:port_id="12345678-...", \
                       vm-uuid="9f8e7d6c-..."}
ifindex             : 23
mac_in_use          : "52:54:00:aa:bb:01"
name                : "vnet0"
ofport              : 15
ofport_request      : []
options             : {}
statistics          : {rx_packets=12345, tx_packets=23456, ...}
status              : {driver_name=tun}
type                : ""
```

| `external_ids` key | Healthy | Warning | Critical | Diagnostic hint |
|--------------------|---------|---------|----------|-----------------|
| `attached-mac` | Match `mac_in_use` | Mismatch | Empty | Libvirt set không đúng |
| `iface-id` | Match SBDB Port_Binding name | Empty (port chưa claim) | Mismatch | Orchestrator-OVN sync issue |
| `iface-id-ver` | Increment khi recreate port | Stale | Empty | Version conflict; OVN không pick up update |
| `iface-status` | `active` cho port live | `inactive` (host disconnect) | Empty | Hot-plug failed |
| `neutron:port_id` | UUID Neutron port | (n/a non-Neutron) | Mismatch với iface-id | OpenStack inconsistency |
| `vm-uuid` | Match libvirt VM UUID | (n/a non-libvirt) | Empty cho libvirt VM | Libvirt didn't set |
| `container_id` | Match docker ID | (n/a non-Docker) | Stale (container exited) | Orphan port |
| `mac_in_use` | Match attached-mac | Mismatch | Empty | NIC driver issue |
| `ofport` | Number > 0 | -1 | Empty | OpenFlow port not assigned |

**Kịch bản bẻ gãy:**

- `iface-id` empty trên VM port: libvirt không set, hoặc XML thiếu `<virtualport type='openvswitch'/>`. Kiểm chứng `virsh dumpxml <vm> | grep virtualport`.
- `iface-id` set nhưng OVN-controller không claim: SBDB Port_Binding `name` field không match. Check `ovn-sbctl list Port_Binding | grep <iface-id>`.
- `container_id` stale (Docker đã rm): orphan port. Cleanup script Phần 9.13.7.
- `iface-id-ver` lệch giữa libvirt và OVN: orchestrator update version nhưng OVN-controller chưa pick up. Force ovn-controller restart hoặc ovsdb-server reload.

## 9.13.10 Capstone POE: ovs-docker helper vs manual veth, đâu production-grade?

> **Bối cảnh:** Production cluster Docker với 100+ container, restart frequency 5-10/giờ. Kỹ sư chọn `ovs-docker` helper vì syntax ngắn, đơn giản hơn manual veth.

**Tuyên bố cần kiểm chứng:** "ovs-docker helper đơn giản hơn manual veth nên ổn định hơn cho production."

**Predict:** Sai. ovs-docker là bash script wrapper, không transactional, không atomic, race condition khi nhiều container restart cùng lúc.

**Observe (giả định):**

- Setup 100 container với restart-policy=on-failure, simulate failure rate 10%/phút (10 container fail+restart per minute).
- Dùng `ovs-docker add-port` per container restart: bash invoke → 5-10 sequential `ovs-vsctl` call → 200-500ms per add-port.
- Race condition: 2 container restart same time → 2 ovs-docker invoke → cả 2 try `add-port` cùng port name (nếu user cấu hình sai) → 1 fail, 1 succeed → orphan state.
- Sau 1 giờ: 30+ orphan port, 5-10 container không có network (silent fail).

Manual veth + atomic batch:

- Script batch 10 container restart vào 1 OVSDB transaction.
- 1 transaction = atomic across all 10 add-port + set external_ids.
- Total time < 100ms cho 10 container.
- 0 race condition (OVSDB internally serialize transactions).
- Sau 1 giờ: 0 orphan, 100% container have network.

**Explain:**

1. **`ovs-docker` là bash convenience script**, suitable cho dev/test (1-2 container). Source code (`utilities/ovs-docker.sh`) ~200 lines bash, không transaction, không retry, không atomic.
2. **Production cần atomic OVSDB transaction**: dùng `ovs-vsctl --no-wait <op1> -- <op2> -- <op3>` để batch multiple ops vào 1 transaction.
3. **Race condition trong bash script**: parallel invoke leads non-deterministic state. Test với 100 container không reveal vì timing happen luck-based; surface ở scale 1000+.
4. **Manual veth pattern** (xem §9.13.2 Pattern 2) cho phép control fine-grained: per-container QoS, mirror, OpenFlow policy.

**Falsification test:**

- Setup 100 container, restart loop 1 giờ với `ovs-docker`. Đo orphan count.
- Setup 100 container, restart loop với manual+batch. Đo orphan count.
- Nếu ovs-docker orphan = 0 → counter-example. Trong thực tế, ovs-docker luôn produce orphan dưới scale.

**Tuyên bố sai ở đâu:** "Đơn giản hơn nên ổn định hơn" sai. Đơn giản về syntax ≠ atomic + transactional. Production cần transactional integrity, không syntax sugar.

> **Quy tắc:** ovs-docker chỉ cho dev/test 1-2 container. Production = manual veth + OVSDB atomic batch transaction.

> **K8S note (per directive ban):** OVN-Kubernetes CNI plugin (Phần 15.x) handle container lifecycle pattern này tự động qua CNI ADD/DEL hooks. Không deep-dive ở Phần 9.13. Cross-link Phần 15.x cho K8S triển khai context.

### ▶ Guided Exercise 14: Plug container vào OVS qua veth manual

**Outcomes:** Tạo container, plug veth vào OVS, traffic qua OVS.

**Before You Begin:**

```bash
[host]# ovs-vsctl add-br br-docker
[host]# docker run -dit --name c1 --network none ubuntu:22.04 sleep 3600
```

**Instructions:**

**1.** Tạo veth pair:

```bash
[host]# ip link add c1-veth type veth peer name c1-br
[host]# ip link set c1-br up
```

**2.** Plug vào OVS:

```bash
[host]# ovs-vsctl add-port br-docker c1-br -- \
         set Interface c1-br external_ids:container_id=c1
```

**3.** Move c1-veth vào container namespace:

```bash
[host]# PID=$(docker inspect -f '{{.State.Pid}}' c1)
[host]# ip link set c1-veth netns $PID
[host]# nsenter -t $PID -n ip addr add 10.99.0.1/24 dev c1-veth
[host]# nsenter -t $PID -n ip link set c1-veth up
```

**4.** Kiểm chứng: `docker exec c1 ip addr show c1-veth`

**5.** Cleanup: `docker rm -f c1 && ovs-vsctl del-br br-docker`

## Điểm cốt lõi cần nhớ

Thứ nhất, libvirt + OVS qua `<virtualport type='openvswitch'/>` là pattern chuẩn — libvirt tự plug VM với external_ids metadata.

Thứ hai, Docker + OVS có hai pattern: `ovs-docker` (dev) và manual veth (production).

Thứ ba, security baseline non-negotiable: file permissions strict, TLS, IPsec, SELinux/AppArmor enabled.

Thứ tư, `ovs-sandbox` là công cụ development quan trọng — test mà không cần kernel module hay physical NIC.

## 17.1 - Why OVN Exists: The Original Problem and the Path to a Solution

### From OpenFlow to OVN, a brief historical narrative

To understand OVN, we have to go back to **2008**, when **Nick McKeown** and seven co-authors (Tom Anderson, Hari Balakrishnan, Guru Parulkar, Larry Peterson, Jennifer Rexford, Scott Shenker, Jonathan Turner) published "OpenFlow: Enabling Innovation in Campus Networks" in ACM SIGCOMM Computer Communications Review (Vol. 38, No. 2, pp. 69 to 74). The paper, which now has more than 9,000 citations (as of 2025), laid the foundation for the concept of **Software-Defined Networking (SDN)**: separate the control plane from the data plane, and allow switches to be programmed through a standard protocol.

OpenFlow grew out of the PhD research of **Martin Casado** at Stanford, the Ethane project (published at SIGCOMM 2007, with research starting in 2006), in which the idea of "a simple switch with a programmable flow table" was first realised. Casado, together with McKeown and Shenker, co-founded **Nicira Networks** in 2007. At Nicira, two key engineers, **Ben Pfaff** and **Justin Pettit**, built **Open vSwitch (OVS)**, an open-source OpenFlow-based virtual switch that became the backbone of network virtualisation. OVS was merged into Linux kernel 3.3 (2012) and became an indispensable component in cloud platforms.

In 2012, VMware acquired Nicira for 1.26 billion USD. Pfaff and Pettit continued to develop OVS at VMware. As Pfaff explained in a USENIX 2025 interview: *"One reason OVN succeeded is that our team had a history with network virtualisation going back to the early days of Open vSwitch. At Nicira, we wrote several generations of network virtualiser. Each time we learned what worked and what to avoid."*

### Before OVN: Neutron ML2/OVS and the limits of an agent-based architecture

Before OVN, OpenStack Neutron used the **ML2 (Modular Layer 2) plugin** with the **OVS mechanism driver** plus a set of Python agents running on every compute node:

```
┌─────────────────────────────────────────────────────────┐
│                  Neutron Server                          │
│              (ML2 Plugin + OVS MechDriver)               │
└────────────────────┬────────────────────────────────────┘
                     │ RabbitMQ (AMQP RPC)
        ┌────────────┼────────────────┐
        ▼            ▼                ▼
┌──────────────┐ ┌──────────────┐ ┌──────────────┐
│ Compute #1   │ │ Compute #2   │ │ Network Node │
│ neutron-ovs  │ │ neutron-ovs  │ │ neutron-l3   │
│   -agent     │ │   -agent     │ │ neutron-dhcp │
│ br-int (NORM)│ │ br-int (NORM)│ │ neutron-meta │
│ br-tun       │ │ br-tun       │ │              │
│ br-ex/br-vlan│ │ br-ex/br-vlan│ └──────────────┘
└──────────────┘ └──────────────┘
```

Each node ran **neutron-openvswitch-agent**, a Python agent that configured OVS bridges and flows. The agent used three bridges: **br-int** (the integration bridge, which used the NORMAL action to delegate all L2 forwarding to OVS built-in MAC learning), **br-tun** (the tunnel bridge for VXLAN and GRE), and **br-ex / br-vlan** (the provider bridge, connecting the physical network through patch port pairs).

The core problem was the NORMAL action on br-int. When br-int used NORMAL, OVS learned MAC addresses on its own like a physical switch, completely outside the control of the CMS. The direct consequences were significant: the CMS had no visibility into the MAC table and therefore could not control or verify forwarding decisions; unknown unicast flooded out every port on the bridge and wasted bandwidth; security groups had to be implemented through a separate Linux bridge plus iptables for every VM (because OVS conntrack support was immature at the time); and RabbitMQ became a single point of failure since every topology change had to go through the message broker. Launchpad bug #1866445 captured the symptom clearly: "br-int on a compute node does not learn the MAC of a VM on another compute node, VM-to-VM performance degrades linearly", a direct consequence of MAC learning being limited to a single local bridge.

### OVN was announced on 13 January 2015

On 13 January 2015, **Justin Pettit, Ben Pfaff, Chris Wright** (Red Hat CTO), **and Madhu Venugopal** announced OVN on the Network Heresy blog: *"Today we are excited to announce Open Virtual Network (OVN), a new project that brings virtual networking to the OVS user community."*

**Russell Bryant**, a Red Hat engineer, quickly became the main driver of OVN inside OpenStack: *"The timing could not have been better. I dove in with a goal of figuring out whether OVN could be a promising next generation of Open vSwitch integration for OpenStack."* Bryant wrote the first OVN-OpenStack integration code and published a series of technical blog posts at blog.russellbryant.net.

The first stable release shipped with **OVS 2.6.0** (September 2016). Neutron integration through networking-ovn began with **OpenStack Newton** (October 2016). In 2019, OVN moved into a separate repository at github.com/ovn-org/ovn, an important milestone marking the maturity of the project.

### OVN architecture, a hybrid centralised plus distributed model with two databases

```
┌───────────────────────────────────────────────────────┐
│            CMS (Neutron / Kubernetes)                   │
└──────────────────┬────────────────────────────────────┘
                   │ OVSDB Protocol
                   ▼
┌───────────────────────────────────────────────────────┐
│           OVN Northbound Database (NB DB)               │
│ Logical_Switch │ Logical_Router │ ACL │ DHCP_Options    │
│ Logical_Switch_Port │ Logical_Router_Port │ ...         │
└──────────────────┬────────────────────────────────────┘
                   │ ovn-northd compiles NB to SB
                   ▼
┌───────────────────────────────────────────────────────┐
│           OVN Southbound Database (SB DB)               │
│ Logical_Flow │ Datapath_Binding │ Port_Binding          │
│ Chassis │ Multicast_Group │ FDB │ MAC_Binding           │
└──────┬──────────────┬──────────────┬──────────────────┘
       ▼              ▼              ▼
  ovn-controller  ovn-controller  ovn-controller
  (Chassis 1)     (Chassis 2)     (Chassis 3)
       ↓              ↓              ↓
  OpenFlow → br-int  OF → br-int  OF → br-int
```

**NB DB** is the interface between the CMS and OVN. Its schema uses networking concepts familiar to a cloud operator: `Logical_Switch`, `Logical_Switch_Port` (key columns are `name`, `type`, `addresses`, `port_security`, `enabled`), `Logical_Router`, and `ACL`. When Neutron creates a network or a port, it writes directly into NB DB through the OVSDB protocol, with no intermediate message broker.

**SB DB** has a completely different schema and represents the network as **logical flow** (match-action rules using logical concepts). The main tables are `Logical_Flow`, `Datapath_Binding`, `Port_Binding`, `Chassis`, `Multicast_Group`, `FDB`, and `MAC_Binding`. The separation between NB DB (network intent) and SB DB (network realisation) is a core architectural decision, allowing the CMS to speak in abstract terms while the system underneath translates to concrete OpenFlow.

**ovn-northd** is the central daemon that reads NB DB and translates it into SB DB. Each logical switch or logical router is converted into a logical datapath together with its corresponding set of logical flows. This is a deployment-wide computation that runs in a single instance (or an active-standby pair for HA).

**ovn-controller** runs on every chassis, reads logical flows from SB DB, then translates them into OpenFlow flows and programs them into OVS. Bryant described the division of work like this: *"OVN breaks this problem up: first calculates logical flow, then distributes to ovn-controller on each node which converts to physical flow."* The computation is node-specific and fully distributed: each chassis only installs flows related to ports it hosts.

### Why br-int no longer uses the NORMAL action

The deepest architectural difference between ML2/OVS and OVN lies in how br-int operates. OVN places br-int in **fail-secure** mode: the flow table starts empty, and every forwarding decision must be programmed explicitly by ovn-controller. From the OVN tutorial: *"It's common for an OVN logical switch to know all the MAC addresses supported by its logical ports... That's why there's no logic here for MAC learning or flooding."*

Security groups no longer need a Linux bridge plus iptables, OVN uses **kernel conntrack directly** from OVS (the ct action in OpenFlow), which simplifies the data path significantly. RabbitMQ is no longer needed for network topology changes, all communication between the central daemons and the chassis goes through OVSDB. Red Hat OpenStack Platform 16.0 (2020) marked the official transition: ML2/OVN became the default mechanism driver, replacing ML2/OVS that had served for many years before.

> **Key idea.** OVN knows every MAC address in advance (from the `addresses` column on Logical_Switch_Port) and programs explicit flows for each MAC. When this document discusses FDB learning later, it covers the **exception case** in which OVN does NOT know the MAC in advance, specifically on a port with `port_security` disabled and addresses containing the keyword `"unknown"`. This is the core architectural difference between OVN (explicit programming) and a physical switch (dynamic learning).

### The tunnel key system, the language for identifying logical objects

Every logical object in OVN, that is, logical switch, logical router, logical port, and multicast group, is assigned a unique integer called the **tunnel key**. This system is the backbone of the mechanism that carries packets between chassis through Geneve tunnels, and it is also how ovn-controller identifies objects when programming OpenFlow rules on br-int.

**Datapath tunnel key (dp_key)** is a 24-bit number (1 to 16,777,215) assigned to each logical switch or logical router, stored in `Datapath_Binding.tunnel_key`. When a packet leaves a chassis through a Geneve tunnel toward another chassis, the dp_key is encoded into the **VNI field** (Virtual Network Identifier) in the Geneve header. The VNI field is exactly 24 bits, so the dp_key maps 1:1 into the VNI. On the OpenFlow side, the dp_key maps to the **metadata** register: any flow matching `metadata=0x752` is matching the logical switch with dp_key=1874 (0x752 hex). This is how ovn-controller distinguishes which datapath a packet belongs to when br-int serves multiple logical switches at once.

**Port tunnel key (port_key)** is a 15-bit number (1 to 32,767) assigned to each logical port within a datapath, stored in `Port_Binding.tunnel_key`. Each datapath has its own namespace: port_key=3 on logical switch A and port_key=3 on logical switch B are two completely different ports, because their differing dp_keys disambiguate them. In the Geneve header, port_key does not live in the VNI but is encoded into a **TLV option** (class 0x0102, type 0x80). This is supplementary metadata that Geneve supports (GRE and VXLAN do not have an equivalent mechanism). On the OpenFlow side, port_key maps to **reg14** (MFF_LOG_INPORT, logical ingress port) and **reg15** (MFF_LOG_OUTPORT, logical egress port).

**Multicast group tunnel key** lies in the high range 32,768 to 65,535, assigned to multicast groups and stored in `Multicast_Group.tunnel_key`. This range is completely separated from the port_key range (1 to 32,767), so they never collide. `_MC_flood` always receives tunnel_key 32768, and `_MC_unknown` receives 32769. These two values are fixed constants hardcoded in `lib/mcast-group-index.h`.

A key point for understanding the sections that follow: **even though a localnet port does not use a Geneve tunnel** (traffic traverses a patch port to br-provider and out through the physical NIC), it is still assigned a tunnel_key in SB DB. The reason is that ovn-controller needs the port_key to identify the localnet port inside the logical pipeline, and matching reg14 / reg15 for ACL, QoS, and FDB lookup all depends on port_key. Without a tunnel_key, the localnet port would be "invisible" in the pipeline. Similarly, the FDB table (Section 17.4) stores `dp_key` and `port_key` directly instead of the logical port name. Numan Siddique's commit message explains why: *"This makes it easier for ovn-controller to program the OF rules"*, because OpenFlow rules match on integers (register values), not on strings.

> **Key idea.** The three kinds of tunnel key (24-bit dp_key, 15-bit port_key, 16-bit multicast key) form the internal addressing system of OVN. When a packet traverses Geneve: dp_key is encoded into the VNI, and port_key is encoded into a TLV option. When the packet is local (br-int): dp_key sits in metadata, and port_key sits in reg14 / reg15. Understanding this system is a foundation for reading OpenFlow dumps and for debugging with ovn-trace.

## 17.2 - Localnet Port: The Bridge Between the Virtual World and a Physical VLAN

### The problem of connecting an overlay to a physical VLAN

OVN was designed for overlay networks (Geneve and VXLAN tunnels), but production reality always requires connecting VMs directly to a VLAN-based physical network (for example, VLAN 3808 in the case study in Section 17.6).

In OpenStack this kind of network is called a **provider network**: a network created by the administrator, connected directly to the physical network infrastructure (usually a VLAN segment). The name "provider" comes from the fact that the infrastructure provider supplies this connection, in contrast to the tenant or self-service network (an overlay that the user creates over Geneve or VXLAN tunnels). A provider network is not necessarily the public internet. In the case study, VLAN 3808 (100.64.0.0/28) is an internal transit network that connects the VM to a Fornix gateway. The key technical distinction is that a provider network uses a localnet port on OVN and traffic traverses the physical NIC, whereas a tenant network traverses a Geneve tunnel between chassis.

Before OVN, Neutron ML2/OVS used br-ex / br-vlan bridges with patch port pairs: the agent configured OpenFlow rules to swap the internal VLAN tag with the physical VLAN tag. Patch ports existed on every node regardless of whether a VM on that node used the network, which wasted resources on large deployments.

### Localnet port design: a "special port" representing the entire physical network

OVN solves the physical connectivity problem with a **localnet port type**, a special kind of Logical_Switch_Port:

```
ovn-nbctl lsp-add provider-ls ln-provider
ovn-nbctl lsp-set-type ln-provider localnet
ovn-nbctl lsp-set-addresses ln-provider unknown
ovn-nbctl lsp-set-options ln-provider network_name=physnet1
ovn-nbctl set Logical_Switch_port ln-provider tag=3808
```

> **Key idea.** A localnet port **does NOT represent a single device** (a single MAC). It represents the **entire physical network segment**. This is the core distinction from a VIF port: a VIF port maps 1:1 to a VM tap interface, whereas a localnet port maps 1:N to every device on the physical VLAN.

The connection mechanism is based on **bridge mapping**, configured per chassis to specify which physical bridge corresponds to which network name:

```
Physical Network (VLAN 3808)
         |
    +----+---------+
    | br-provider  |  <- OVS bridge connecting the physical NIC (bond3)
    +----+---------+
         | patch port (ovn-controller creates it automatically)
    +----+---------+
    |   br-int     |
    |              |
    | VM .12 port  |  <- VIF port: LSP type="" representing a VM interface
    | VM .xx port  |  <- VIF port: LSP type="" representing a VM interface
    | localnet port|  <- represents the ENTIRE VLAN 3808
    +--------------+
```

Bridge mapping configuration: `ovs-vsctl set open . external-ids:ovn-bridge-mappings=physnet1:br-provider`

When ovn-controller detects that the node hosts at least one VM on this provider network, it automatically creates a patch port between br-int and br-provider. Conversely, if the node hosts no VM on that network, ovn-controller does not create the patch port and releases the corresponding resources. This is an improvement over the old ML2/OVS, where the agent created patch ports on every node regardless of whether a VM was present.

In SB DB, every Logical_Switch_Port (including localnet LSPs) has a corresponding row in the Port_Binding table. For a localnet LSP, the Port_Binding has a distinctive characteristic: **`chassis: []`** (it is not bound to any specific chassis), because the localnet port "exists everywhere" through bridge mapping on each chassis:

```
_uuid       : 8427506e-...
logical_port: "provnet-e4abf6df-..."
mac         : [unknown]
options     : {network_name=physnet1}
tunnel_key  : 1
type        : localnet
chassis     : []          <- NOT bound to any chassis!
```

### "provnet-" versus "localnet", two names at two different layers

When you read the output of `ovn-sbctl list port_binding`, you will see a localnet port with `logical_port: "provnet-e4abf6df-..."` but `type: localnet`. The two different prefixes confuse newcomers, but in fact they operate at two completely different layers.

**"localnet"** is the OVN port type, a concept that belongs to the OVN core and is defined in ovn-sb(5). When `Port_Binding.type = "localnet"`, ovn-controller on each chassis understands that the port represents a connection to the local physical network through bridge mapping. OVN itself only inspects the type and `options:network_name`; it does not care what the logical_port name is.

**"provnet-"** is a Neutron naming convention, a naming pattern produced by the Neutron ML2/OVN driver. When Neutron creates a provider network, the driver generates a localnet LSP with a name following the pattern `provnet-<network-segment-UUID>`. This name is purely administrative metadata. If you use OVN standalone (without OpenStack), the localnet port can be named any string.

There is no distinction between "provnet" and "localnet" at the OVN layer. The relationship is between CMS naming and OVN type, similar to how Neutron names a VIF port `<port-UUID>` while OVN only cares about `type=""` and chassis binding. When debugging with `ovn-sbctl`, if you see a `logical_port` beginning with "provnet-", that is certainly a localnet port on an OpenStack system. On Kubernetes (OVN-Kubernetes), the localnet port may follow a completely different naming convention.

### Common misconceptions about the localnet port

> **Misconception:** "A localnet port behaves like a bridge port on a physical switch, learning MACs and forwarding frames based on a MAC table."
>
> **Reality:** A localnet port in OVN does not learn MACs (unless `localnet_learn_fdb=true` is set, which is off by default). It has no MAC of its own, does not participate in the default MAC learning pipeline, and its addresses field is the keyword `"unknown"`, which means it receives every frame whose destination MAC does not match any known port. This is a catch-all behaviour, not learning.

> **Misconception:** "A localnet port is bound to a specific chassis like a VIF port."
>
> **Reality:** The Port_Binding for a localnet port has `chassis: []`. It is not bound to any chassis. ovn-controller on each chassis with the corresponding bridge mapping creates a patch port on demand. The localnet port "exists everywhere" through bridge mapping, not through chassis binding.

### Why addresses contain "unknown", a key design decision

OVN does not know in advance which MACs exist on the physical network: hundreds of servers, routers, and firewalls that OVN does not manage. `"unknown"` is a special keyword in OVN: a port whose address is "unknown" receives every packet whose destination MAC does not match any known port. It acts as a catch-all for unknown unicast traffic, and a port carrying this keyword is added to the multicast group `_MC_unknown`, the dedicated group for unknown unicast forwarding that the next section will analyse in detail.

ovn-nb(5): *"Each bridged logical switch has one or more localnet ports, which have only one special address 'unknown'."*

### Chassis, Claim, and Multichassis, the port ownership mechanism in OVN

Section 17.2 has just shown that a localnet port has `chassis: []` in Port_Binding. But what about a VIF port (a VM)? The answer is that a VIF port is **claimed** by a specific chassis. Understanding the claim mechanism and the multichassis state is a prerequisite for reading the case study in Section 17.6, where the entire FDB poisoning mechanism unfolds inside a "multichassis window" lasting only a few seconds.

A **Chassis** in OVN is a physical compute node, each hypervisor host where VMs run. When ovn-controller starts on a node, it registers itself into SB DB by creating a row in the `Chassis` table with `name` set to the hostname of the node (for example, `vhhl1c2wc1tsn01`) and `system_id` set to a unique UUID. This table is the foundation that lets OVN know which compute node is up and capable of receiving port bindings.

> **Misconception:** "A Chassis is the physical NIC or the bond interface on the server."
>
> **Reality:** A Chassis is the entire compute node (the hypervisor host), not an interface. A node with two or with ten NICs is still a single chassis. Physical interfaces are exposed through `bridge mappings` in OVS `external-ids`, completely separate from the Chassis concept.

**Claim** is the action in which ovn-controller writes its chassis name into the `Port_Binding.chassis` field for an LSP. The mechanism works as follows. When a VM is created on a node, QEMU creates a tap interface with `external-ids:iface-id` equal to the UUID of the Neutron port. ovn-controller runs continuously, polling the local OVS `Interface` table. When it detects an interface whose `iface-id` matches an LSP name in SB DB, it calls the function `claim_lport()` in `controller/binding.c`, writing its chassis into `Port_Binding.chassis`. From that moment on, the port is considered "owned" by this chassis: only that chassis will program OpenFlow rules for the port, and only that chassis will receive traffic destined for the port from the Geneve tunnel.

A real log when the claim happens on chassis `vhhl1c2wc1tsn01`:

```
[ovn-controller] INFO binding  Claiming lport 0a17b4f8-736d-4bf5-ba1e-335d17cb5973 for this chassis.
[ovn-controller] INFO binding  0a17b4f8-736d-4bf5-ba1e-335d17cb5973: Claiming fa:16:3e:04:74:74 100.64.0.12
[ovn-controller] INFO binding  0a17b4f8-736d-4bf5-ba1e-335d17cb5973: Claiming unknown
```

The line "Claiming unknown" confirms that the port has `addresses` containing the keyword `"unknown"`, the second condition for FDB learning to activate (analysed in Section 17.4).

**Multichassis** is a temporary state in which a port is simultaneously bound to two chassis: the source chassis (where the VM is currently running) and the destination chassis (where the VM is about to be migrated). This state only occurs inside the live migration window. The implementation technique relies on two distinct fields in Port_Binding: `chassis` stores the primary chassis (the current owner) and `additional_chassis` stores the secondary chassis (where the VM is being moved to). When Neutron live-migrates, it sets `options:requested-chassis = "chassisA,chassisB"` on the LSP. ovn-northd splits this string into `PB.requested_chassis = chassisA` and `PB.requested_additional_chassis = [chassisB]`. On chassisB, the function `lport_can_bind_on_this_chassis()` recognises itself in `requested_additional_chassis` and returns the enum `CAN_BIND_AS_ADDITIONAL` instead of `CAN_BIND_AS_MAIN`. `claim_lport()` updates `PB.additional_chassis` with chassisB, and the port is now multichassis.

When `PB.additional_chassis` is non-empty, the function `shash_is_empty(&ld->multichassis_ports)` in `controller/physical.c` returns `false`. This triggers `enforce_tunneling_for_multichassis_ports()`, which installs an OpenFlow rule in `OFTABLE_REMOTE_OUTPUT` forcing every packet destined to that port through the Geneve tunnel between the two chassis, even when source and destination are on the same physical VLAN. This is the first triggering condition that leads to FDB poisoning in the case study: the tunnel forces traffic through Geneve, Geneve adds 58 bytes of overhead, the packet exceeds MTU, and an ICMP error is generated with the bug that swaps MAC without swapping inport.

> **Key idea.** Multichassis is not a steady-state HA configuration. It is a temporary state inside the live migration window, typically lasting 5 to 10 seconds. After migration completes, `PB.additional_chassis` returns to empty and the tunnel enforcement mechanism disappears. Only nova live-migrate creates this state; every other Nova action (boot, stop, cold migration, rebuild) never sets `requested-chassis` as a comma-separated list.

Real-time inspection of the multichassis state on OVN central:

```bash
[controller]$ ovn-sbctl --columns=logical_port,chassis,additional_chassis list Port_Binding
```

While migration is in progress, the `additional_chassis` column will contain a UUID instead of being empty on the port of the VM being migrated.

### localnet_learn_fdb, extending FDB learning to localnet ports

Originally the localnet port was excluded from FDB learning entirely for scale reasons: the physical network can carry hundreds of MACs, and learning all of them would quickly exhaust FDB table entries. However, with the rise of hardware offload (NVIDIA / Mellanox SmartNIC) and the demand to reduce flooding on small provider networks, Ales Musil (amusil@redhat.com) and Numan Siddique added the `localnet_learn_fdb` option in OVN v22.09.0 (commit `93514df0d4c8fe7986dc5f287d7011f420d1be6d`, Red Hat Bugzilla #2070529). The setting is opt-in and off by default: `ovn-nbctl set Logical_Switch_port <localnet-lsp> options:localnet_learn_fdb=true`. How this feature interacts with FDB learning is analysed in detail in Section 17.4.

## 17.3 - MC_FLOOD and MC_UNKNOWN: Broadcast and Unknown Destinations in a Distributed Control Plane

### The packet-flooding problem when a logical switch spans many chassis

On a physical switch, broadcast and unknown unicast flooding are simple: the ASIC chip replicates the frame to every port in the same VLAN within a few nanoseconds. In OVN the problem is more complex because a logical switch spans many chassis, and each chassis only knows the local ports it hosts. When a VM sends a broadcast frame, the local ovn-controller needs to know which chassis the frame must reach and through which port on each chassis.

OVN solves this with **pseudo-multicast groups** in SB DB, stored in the `Multicast_Group` table. Each group contains a list of member Port_Bindings, and when a packet has its `outport` set to the group name, ovn-controller replicates the packet to every member, including ports on remote chassis through Geneve tunnels.

### Multicast_Group schema (ovn-sb(5))

| Column | Type | Description |
|------------|-------------------|-------------------------------------------|
| datapath | Datapath_Binding | Logical switch containing the group |
| tunnel_key | int [32768-65535] | Unique ID (does not collide with logical port IDs) |
| name | string | Group name (unique within the datapath) |
| ports | set of Port_Binding | List of member ports |

Source code (`lib/mcast-group-index.h`):
```c
enum ovn_mcast_tunnel_keys {
    OVN_MCAST_FLOOD_TUNNEL_KEY = 32768,    // _MC_flood
    OVN_MCAST_UNKNOWN_TUNNEL_KEY,           // _MC_unknown (32769)
    OVN_MCAST_MROUTER_FLOOD_TUNNEL_KEY,     // 32770
    ...
};
```

### Three main multicast groups and their members

| Group | Tunnel Key | Members | Purpose |
|----------------|-----------|----------------------------------------|---------------------------------------|
| `_MC_flood` | 32768 | All enabled ports (VIF, router, localnet) | Broadcast and multicast (ARP request, and similar) |
| `_MC_flood_l2` | derived | All ports except router ports | L2 multicast (non-IP, non-ARP) |
| `_MC_unknown` | 32769 | Only ports with addresses="unknown" | Unknown unicast forwarding |

> **Key idea.** `_MC_unknown` is created **conditionally**, only when at least one port on the logical switch has `"unknown"` in its addresses. On an overlay-only switch (every MAC is already known from LSP addresses), MC_UNKNOWN does not exist and unknown unicast is dropped. This is why a provider network (which has a localnet port with addresses="unknown") always has MC_UNKNOWN, while a tenant overlay network does not.

The difference between MC_FLOOD and MC_UNKNOWN is essential for understanding why ARP keeps working while unicast traffic is affected by FDB poisoning (the case study in Section 17.6). MC_FLOOD receives every broadcast and multicast frame, including ARP Requests, and floods them to all enabled ports. MC_UNKNOWN only receives unknown unicast frames and floods to a narrower set of ports: only those with "unknown" addresses.

### Pipeline processing, a packet walk-through in detail

```
           Logical Switch Ingress Pipeline
           ===============================
  Table 0-1: ls_in_port_sec_l2 / ls_in_port_sec_ip
       |
       v
  Table N: ls_in_lookup_fdb  (*)
     -> reg0[11] = lookup_fdb(inport, eth.src)   [check FDB]
       |
       v
  Table N+1: ls_in_put_fdb  (*)
     -> if reg0[11]==0: put_fdb(inport, eth.src)  [LEARN MAC if not present]
       |
       v
  ... ACL, QoS, DHCP, DNS, ARP responder, ...
       |
       v
  Table M: ls_in_l2_lkup (L2 Destination Lookup)  (*)
     Priority 72: eth.mcast && arp -> outport = _MC_flood     [ARP broadcast]
     Priority 70: eth.mcast -> outport = _MC_flood_l2         [L2 multicast]
     Priority 50: eth.dst==<known_mac> -> outport=<port>      [Static known]
     Priority  0: outport = get_fdb(eth.dst); next            [FDB lookup]
       |
       v
  Table M+1: ls_in_l2_unknown  (*)
     Priority 50: outport=="none" -> outport=_MC_unknown       [Flood unknown]
     Priority  0: output                                       [FDB found, forward]

  (*) Note: table numbers vary across OVN versions. For example, on OVN v21.06,
  ls_in_lookup_fdb is approximately table 2 and ls_in_put_fdb is approximately
  table 3. On newer versions (v23.09 and later), positions may shift due to
  added stages. Verify with: ovn-sbctl lflow-list <datapath> | grep ls_in_lookup_fdb
```

The unknown-unicast processing flow (no FDB entry) goes through two key stages. At `ls_in_l2_lkup` (priority 0), the action `get_fdb(eth.dst)` looks up the destination MAC in the FDB table. It finds nothing, so outport is set to "none" (the value 0). At `ls_in_l2_unknown` (priority 50), the match `outport == "none"` succeeds, and the action sets `outport = "_MC_unknown"` then executes `output`. OVN floods a copy of the packet to every member of the MC_UNKNOWN group, including the localnet port. When the packet reaches the localnet port, it traverses the patch port into br-provider, then exits through the physical NIC (bond3) onto the physical VLAN.

The known-unicast flow (with an FDB entry) differs in the first step. At `ls_in_l2_lkup` (priority 0), `get_fdb(eth.dst)` finds an entry in the FDB table and sets outport to the corresponding port_key. At `ls_in_l2_unknown` (priority 0), outport already has a value other than "none", so the flood condition does not match and the packet goes directly to the port specified by FDB.

Below is real ovn-trace output illustrating the unknown-unicast flow (GitHub Issue #244):

```
ingress(dp="non_fdp", inport="07188e")
 0. ls_in_port_sec_l2: inport == "07188e", priority 50 -> next;
 3. ls_in_lookup_fdb:  inport == "07188e", priority 100
    reg0[11] = lookup_fdb(inport, eth.src); next;
24. ls_in_l2_lkup: 1, priority 0
    outport = get_fdb(eth.dst); next;
25. ls_in_l2_unknown: outport == "none", priority 50
    outport = "_MC_unknown"; output;

multicast(dp="non_fdp", mcgroup="_MC_unknown")
  -> egress: provnet-40617a (type "localnet")   <- out to physical network
  -> egress: RTR3-1-port-128, 3b461e, 877021, ...
```

### Why MC_UNKNOWN is indispensable for provider networks

Without MC_UNKNOWN, the provider network does not work. OVN does not know the MAC of the physical gateway (100.64.0.1 in the case study) in advance because the gateway is not a VM managed by Neutron, and its MAC address does not appear in any LSP addresses field. When a VM sends a frame to the gateway, the destination MAC does not match any priority-50 static flow in `ls_in_l2_lkup`. Without an "unknown" forwarding mechanism, `get_fdb` returns outport="none" and there is no MC_UNKNOWN to flood to, so the frame is dropped. MC_UNKNOWN serves as the only door through which traffic can leave the OVN virtual world to the physical network when the destination MAC is not known in advance.

> **Misconception:** "MC_UNKNOWN is like broadcast, it floods to every port on the logical switch."
>
> **Reality:** MC_UNKNOWN floods only to ports with `"unknown"` in their addresses. On a logical switch with 50 VIF ports but only 2 ports with addresses="unknown" (the localnet port and one VNF port), MC_UNKNOWN has only 2 members, not 50. Broadcasts (such as an ARP request) traverse `_MC_flood` at a higher priority (72 > 0) and flood to every enabled port.

> **Misconception:** "If FDB does not find the MAC, the packet is dropped."
>
> **Reality:** When `get_fdb` returns outport="none", the packet enters `ls_in_l2_unknown` (priority 50) and is redirected to `_MC_unknown`, flooding to every port with "unknown" addresses. The packet is dropped only when MC_UNKNOWN does not exist (an overlay-only switch with no port carrying "unknown").

## 17.4 - FDB table and dynamic MAC learning: from the flooding problem to caching

### Before FDB: flooding was the only solution

Before OVN 21.06, OVN knew MACs from only two sources: static information from the `addresses` column on a Logical_Switch_Port (set by Neutron when the port is created), and the keyword `"unknown"` (meaning "accept anything not known"). There was no dynamic learning mechanism.

The direct consequence on a provider network with many physical devices: every time a VM sent a frame to the MAC of a physical gateway, a MAC OVN did not know in advance, the only solution was to flood to every member of MC_UNKNOWN. If 100 VMs on the same provider network all have port security disabled (all carrying "unknown" addresses and members of MC_UNKNOWN), every frame to a physical device was replicated to 100 ports, even though only the localnet port (heading out to the physical network) was needed. The wasted bandwidth grew linearly with the number of VMs.

FDB learning was introduced to solve exactly this problem: after the first flood, when the physical gateway sends a frame back (for example an ARP reply), ovn-controller records "the gateway's MAC arrived through the localnet port" into the FDB table. From then on, `get_fdb` finds the entry and forwards directly to the localnet port, with no flooding. Without FDB the system still works correctly (the packet still reaches its destination through flooding), but performance is poor on a large deployment.

### FDB learning is introduced, OVN 21.06.0

**Numan Siddique** (numans@ovn.org, Principal Software Engineer at Red Hat, Bangalore) sent a 6-patch series to ovs-dev on 5 February 2021. The series builds the entire FDB infrastructure from scratch: it splits MAC learning code into its own file (`controller/mac-learn.c`), adds the FDB table to the SB DB schema, implements three new actions (`put_fdb`, `get_fdb`, `lookup_fdb`), programs OpenFlow tables 71 (GET_FDB) and 72 (LOOKUP_FDB), creates the pipeline stages `ls_in_lookup_fdb` and `ls_in_put_fdb` in northd, and ships a full test suite.

Mailing list archive: https://mail.openvswitch.org/pipermail/ovs-dev/2021-February/380036.html. Patchwork: https://patchwork.ozlabs.org/project/ovn/patch/20210205070022.40327-1-numans@ovn.org/. The feature was merged and released in OVN v21.06.0 (18 June 2021).

### FDB table schema

```json
"FDB": {
    "columns": {
        "mac":      {"type": "string"},
        "dp_key":   {"type": {"key": {"type": "integer", "minInteger": 1, "maxInteger": 16777215}}},
        "port_key": {"type": {"key": {"type": "integer", "minInteger": 1, "maxInteger": 16777215}}},
        "timestamp": {"type": "integer"}
    },
    "indexes": [["mac", "dp_key"]],
    "isRoot": true
}
```

| Column | Type | Description |
|-----------|---------|------------------------------------------------|
| mac | string | Learned MAC address (for example "fa:16:3e:35:a7:73") |
| dp_key | int | Datapath tunnel key (logical switch ID) |
| port_key | int | Port tunnel key (logical port ID) |
| timestamp | int | Update time (added later for aging) |

The index `[mac, dp_key]` ensures the primary key is the (MAC, datapath) pair: there is exactly one FDB entry per MAC per logical switch.

A key design difference from MAC_Binding (§17.5): FDB stores the tunnel keys (dp_key, port_key) directly instead of the logical port name. As analyzed in §17.1, dp_key maps to a metadata register and port_key maps to reg14 or reg15 in OpenFlow. Storing dp_key and port_key directly lets ovn-controller skip the name-to-number conversion when programming flows, reducing processing cost when there are thousands of entries. Numan Siddique's commit message explains this clearly: *"Unlike MAC_Binding Southbound table, FDB table stores the datapath tunnel key and port key. This makes it easier for ovn-controller to program the OF rules."*

Limit: `MAX_FDB_ENTRIES = 1000`.

>
> - **OVN v22.03.0 to v24.03.x:** the macro is defined in `controller/mac-learn.c` (line 28 in v22.03.0, line 31 in v24.03.0).
> - **OVN v24.09.0 and later:** the file `controller/mac-learn.c` is removed entirely. Commit [`fb96ae3679`](https://github.com/ovn-org/ovn/commit/fb96ae3679) "controller: Merge the mac-cache and mac-learn" (Ales Musil and Numan Siddique, 2024-05-07) merges the `mac-cache` and `mac-learn` modules and moves `MAX_FDB_ENTRIES` into `controller/pinctrl.c` (around line 8868 on the main branch).
>
> When grepping for the macro in any OVN checkout, check `git describe --tags` first so you know which version you are in.

### The three core actions and how they work

**`put_fdb(inport, eth.src)`** writes a new entry or updates an existing one. This action translates into a `controller` action that sends a PACKET_IN to ovn-controller. There, `pinctrl_handle_put_fdb()` in `controller/pinctrl.c` extracts dp_key from metadata, port_key from reg14 (MFF_LOG_INPORT), and eth.src from the packet header, then creates an FDB entry struct stored in the in-memory hash map `put_fdbs`. On the next OVSDB transaction, `run_put_fdbs()` iterates the map and inserts or updates each row into the Southbound FDB table through the OVSDB IDL.

**`get_fdb(eth.dst)`** looks up the port_key for the destination MAC. The action resubmits to OpenFlow table 71 (OFTABLE_GET_FDB). ovn-controller installs flows in this table: if `metadata=dp_key, dl_dst=mac` matches, it loads port_key into MFF_LOG_OUTPORT (reg15). If nothing matches (the MAC is not yet in FDB), the default flow loads 0, which is equivalent to outport="none".

**`lookup_fdb(inport, eth.src)`** checks whether an entry exists. The action resubmits to OpenFlow table 72 (OFTABLE_LOOKUP_FDB). When `metadata=dp_key, dl_src=mac, reg14=port_key` matches, it loads 1 into reg0[11] (meaning "entry exists, no learning needed"). A special behavior: when `flags.localnet=1` (the packet arrived from a localnet port), lookup_fdb returns 1 if the entry exists AND the existing port type is VIF. The result is that put_fdb is skipped, protecting the VIF entry from being overwritten by the localnet. This protection mechanism is analyzed in detail later in this section.

### Conditions for FDB learning to activate, three AND conditions

> **Key idea.** A port qualifies for FDB learning when it satisfies **ALL THREE** AND conditions below. If any condition is missing, FDB learning does not activate on that port.

The first condition: **port_security must be empty or disabled** (`!op->n_ps_addrs`). The second condition: **addresses must contain "unknown"** (`op->has_unknown`). The third condition: **type must be "" (regular VIF)** (`!strcmp(op->nbsp->type, "")`) OR a localnet port with `localnet_learn_fdb=true`.

Source code (`northd/northd.c`, function `build_lswitch_learn_fdb_op()`):
```c
if (!op->n_ps_addrs && op->has_unknown &&
    (!strcmp(op->nbsp->type, "") ||
     (lsp_is_localnet(op->nbsp) && localnet_can_learn_mac(op->nbsp)))) {
```

Origin Bugzilla: Red Hat #1672625, *"[RFE] Support MAC learning in OVN when port security is disabled for a logical port"*.

### Deployment assessment: when FDB learning helps and when it harms

The three AND conditions are not arbitrary barriers. Each condition is an intentional design gate that reflects both the problem FDB learning solves and the risk it creates.

When port security is enabled, OVN already knows exactly which MACs are valid on that port. The list of MACs lives in the `port_security` column of the LSP, and ovn-northd generates static flows that match `eth.src == <allowed_mac>` at the `ls_in_port_sec_l2` stage. Any frame with a source MAC not in the list is dropped in the very first stage, before it reaches the FDB pipeline. FDB learning would be meaningless here: the source MACs are known up front, there is nothing to "learn". When port security is disabled, OVN loses the ability to validate the source MAC, and the port accepts frames from any MAC. FDB learning becomes meaningful: it "learns" the MACs it has seen so the next frame does not need to be flooded. The trade-off is that the door is open to FDB poisoning: any MAC that appears on the port is recorded without verification.

The keyword `"unknown"` in addresses places the port into MC_UNKNOWN, the group that receives flooded unknown unicast. The relationship between "unknown" and FDB learning is causal: only a port that receives unknown unicast traffic (through MC_UNKNOWN) gets the chance to "see" a frame from a previously unknown source MAC. A port without "unknown" never receives a frame outside its static MAC list and has nothing to "learn". The Neutron ML2/OVN driver automatically adds `"unknown"` to addresses when port security is disabled (Launchpad #1815270), so on OpenStack the first two conditions always go together. On standalone OVN, an administrator can set `port_security=[]` without setting "unknown"; in that case FDB learning does not activate even though port security is disabled.

A VIF port (type="") represents a VM or container interface, where dynamic MAC learning has direct meaning. A localnet port was initially excluded from FDB learning for scale reasons: a localnet represents an entire physical network segment and may carry hundreds of MACs, which can quickly exhaust `MAX_FDB_ENTRIES = 1000`. The option `localnet_learn_fdb=true` (v22.09.0, described in §17.2) was added as opt-in for two use cases: hardware offload (NVIDIA/Mellanox SmartNICs need a MAC-to-port mapping in order to offload forwarding into hardware) and provider networks with few physical devices (5 to 10 MACs from the localnet do not strain scale but reduce flooding significantly).

> **Key idea.** FDB learning is a double-edged sword: **helpful** when it reduces unknown unicast flooding on a provider network (saving bandwidth, required for hardware offload), but **harmful** when a VNF emits a frame with a source MAC that does not belong to it (FDB poisoning leading to traffic hijack). The three AND conditions are a defense-in-depth design that opens FDB learning only to ports that truly need it, and only when those ports are in an "uncontrolled MAC" state.

Operational recommendation: always set `fdb_age_threshold` greater than 0 on every logical switch where FDB learning is active. A value of 300 seconds (5 minutes) is a reasonable starting point, balancing cache hit rate against the lifetime of stale entries. Monitor FDB table size with `ovn-sbctl list fdb | wc -l`; if it approaches 1000, consider lowering `fdb_age_threshold` or disabling `localnet_learn_fdb`. From OVN v26.03.0 onward, use `lsp_learn_mac` per port to disable learning on individual VNF ports without affecting other ports, which is the most granular solution available.

### fdb_age_threshold, the aging mechanism for FDB entries

Before v23.09.0, an FDB entry persisted forever after creation; no aging mechanism ran. Ales Musil (amusil@redhat.com) and Numan Siddique added an aging mechanism in OVN v23.09.0 through a series of commits, including `bfbf32f363913db2a89f432360039a5d9ded5298` "northd: Synchronize the FDB age threshold" (Ales Musil, committed 2023-08-24 by Mark Michelson), the commit that synchronizes the threshold to the Southbound datapath in preparation for the FDB refresh mechanism. The full discussion thread: [ovs-dev May 2023](https://mail.openvswitch.org/pipermail/ovs-dev/2023-May/404718.html). Configuration per logical switch: `Logical_Switch.other_config:fdb_age_threshold` (unit: seconds). Default: 0, disabled; an FDB entry persists forever. When set greater than 0, ovn-northd automatically deletes entries that exceed the age threshold.

NEWS: *"Added FDB aging mechanism, that is disabled by default. It can be enabled per logical switch with other_config 'fdb_age_threshold'."*

> **Key idea.** With `fdb_age_threshold=0` (the default), an FDB entry persists forever; ovn-northd does not run aging. In the VLAN 3808 case study (§17.6), this is the factor that prevents the wrong FDB entry from ever expiring, so the traffic-hijack condition persists until it is detected and removed by hand.

> **Misconception.** "`fdb_age_threshold=0` means an FDB entry expires immediately after creation."
>
> **Reality.** `fdb_age_threshold=0` is the value that **disables aging**, not a TTL of 0 seconds. When the value is 0, ovn-northd does not run an aging job for the FDB table, and entries persist forever until they are deleted by hand or overwritten. Only a positive value (for example `fdb_age_threshold=300`) gives entries a 300-second TTL. This confusion is particularly dangerous because an operator might think "leaving the default is safest", when in reality the opposite is true: default-off aging means a wrong FDB entry survives forever.

### flags.localnet protection: preventing the localnet from overwriting a VIF entry

When `localnet_learn_fdb=true`, a dangerous race condition appears. Picture this: vif1 on chassis hv1 sends a frame; ovn-controller on hv1 writes an FDB entry mapping mac1 to vif1 (correct). The same frame travels through the localnet to chassis hv2; ovn-controller on hv2 sees the frame entering on the localnet port and writes an FDB entry mapping mac1 to the localnet, overwriting the correct entry with a wrong one. The result: traffic to mac1 on hv2 leaves through the localnet instead of the Geneve tunnel to hv1.

Xavier Simonart (xsimonart@redhat.com) addressed this with the `MLF_LOCALNET` bit (bit 15 in the logical flags register) in commit `2acf91e9628e`, "controller: FDB entries for localnet should not overwrite entries for vifs". Backports: `33b01175` (branch-23.09.1) and `8befadaf` (branch-23.06.3). The feature merged into OVN v24.03.0.

How it works: in `ls_in_lookup_fdb`, when a packet arrives from a localnet port, OVN sets `flags.localnet = 1` before calling `lookup_fdb`. When `flags.localnet=1`, the `lookup_fdb` function returns true if the entry already exists AND the entry's port type is VIF. As a result, `reg0[11] == 1` causes `ls_in_put_fdb` to skip writing a new entry, protecting the VIF entry from localnet overwrite. This protection is the right behavior in the normal case (it prevents the localnet from overwriting a correct VIF entry), but in the FDB poisoning case study (§17.6), where the VIF entry is wrong, the protection becomes a barrier that blocks self-healing.

## 17.5 - MAC_Binding table: IP-to-MAC resolution on the logical router

### Purpose and design

The previous sections focused on Layer 2, that is, MAC forwarding on the logical switch. When a packet needs to traverse a logical router (Layer 3), a completely different problem arises: the router needs to know the MAC of the next-hop IP so that it can build the Ethernet header when forwarding the packet. **MAC_Binding** is the SB DB table that stores the mapping from IP address to MAC address on a logical router port. It is the OVN equivalent of the ARP cache on a physical router.

A fundamental distinction: the ARP cache lives on the router (Layer 3), not on the switch (Layer 2), because the router must resolve a next-hop IP into a MAC in order to forward the packet across the link layer. A pure physical switch does not run ARP and has no ARP cache; a switch only has a MAC address table (the equivalent of FDB in OVN). This distinction matters when debugging: if a packet goes to the wrong port on the same L2 segment, look at FDB; if the router cannot resolve the MAC of a next-hop IP, look at MAC_Binding.

Schema (ovn-sb(5)):

| Column | Type | Description |
|-------------|-------------------|--------------------------------------|
| logical_port | string | Router port where the binding was discovered |
| ip | string | IPv4 address |
| mac | string | Corresponding MAC address |
| datapath | Datapath_Binding | Logical router datapath |
| timestamp | integer | Update time (used for aging) |

Index: `[logical_port, ip]`

ovn-sb(5): *"Usually, IP-to-MAC bindings for virtual machines are statically populated into the Port_Binding table, so MAC_Binding is primarily used to discover bindings on physical networks."*

### How it works in the router pipeline

```
  Logical Router Ingress Pipeline
  ════════════════════════════════
  lr_in_admission → lr_in_lookup_neighbor → lr_in_learn_neighbor
       ↓
  lr_in_ip_input → ... → lr_in_ip_routing → lr_in_ip_routing_ecmp
       ↓
  lr_in_arp_resolve:
    Priority 100: known IP (VM, DHCP) → set eth.dst statically
      match=(outport == "lrp-xxx" && reg0 == 192.168.1.11)
      action=(eth.dst = fa:16:3e:b6:91:70; next;)
    Priority 0: unknown IP → consult MAC_Binding
      match=(ip4)
      action=(get_arp(outport, reg0); next;)
       ↓
  lr_in_arp_request:
    If eth.dst == 00:00:00:00:00:00 → generate ARP request
      action=(arp { eth.dst = ff:ff:ff:ff:ff:ff; arp.spa = reg1;
                    arp.tpa = reg0; arp.op = 1; output; };)
```

The action `get_arp(outport, reg0)` resubmits to OpenFlow table 66, where the flows populated from the MAC_Binding table reside. When ovn-controller observes an ARP reply or ND advertisement on a router port, the pinctrl module writes a new binding via `put_arp(port, ip, mac)`, creating a row in MAC_Binding on the SB DB. To avoid PACKET_IN flooding when ARP replies arrive continuously, OVN uses `lookup_arp` and `lookup_nd` to first check whether the entry already exists.

Aging for MAC_Binding was added in OVN v22.09.0 (commit `1a947dd3`, Ales Musil, Bugzilla #2084668): `ovn-nbctl set Logical_Router <lr> options:mac_binding_age_threshold=<seconds>`. Default: 0 (disabled).

### FDB versus MAC_Binding, two completely different problems

| Aspect | FDB | MAC_Binding |
|-------------------|-------------------------------|--------------------------------|
| **Layer** | L2 (switching) | L3 (routing) |
| **Maps** | MAC to port_key | IP to MAC |
| **Datapath** | Logical Switch | Logical Router |
| **Pipeline** | `ls_in_lookup_fdb`, `ls_in_l2_lkup` | `lr_in_arp_resolve` |
| **Populated by** | MAC learning from a VIF port | ARP reply or ND on a router port |
| **Physical equivalent** | MAC address table on an L2 switch | ARP cache on a router |
| **Introduced** | Feb 2021 (schema 20.16.0) | Original OVN (around 2015) |
| **Aging option** | `fdb_age_threshold` (per LS) | `mac_binding_age_threshold` (per LR) |

> **Key idea.** FDB (L2: maps a MAC address to a port_key, operates on a logical switch) and MAC_Binding (L3: maps an IP address to a MAC address, operates on a logical router) solve two completely different problems. When debugging, distinguish clearly: if a packet goes to the wrong port on the same L2 segment, check FDB; if the router cannot resolve the MAC of a next-hop IP, check MAC_Binding.

> **Misconception.** "When traffic to the physical gateway is dropped, I need to check MAC_Binding because the router may have failed to resolve the gateway's MAC."
>
> **Reality.** If the VM still resolves the gateway's ARP (ping still obtains the MAC) but the ICMP echo times out, the problem is not in MAC_Binding but in FDB. MAC_Binding only affects IP-to-MAC resolution on the logical router; if ARP works, MAC_Binding is not the cause. An L2 forwarding problem (a packet going to the wrong port on the same logical switch) must always be investigated by checking FDB first.

In the VLAN 3808 case study in the next section, the ICMP echo request from a VM to the gateway 100.64.0.1 is delivered to the wrong port (the VyOS tap port instead of the localnet port that exits to the physical network). This is an L2 forwarding problem: the packet enters the correct logical switch but is steered to the wrong destination port. The first table to check is FDB (which holds the wrong MAC-to-port mapping), not MAC_Binding (because the problem has nothing to do with IP-to-MAC resolution on the router).

## 17.6 - Case study: FDB poisoning on provider network VLAN 3808

The previous sections have built up the individual pieces: the localnet port (§17.2) connects the virtual world to a physical VLAN, MC_UNKNOWN (§17.3) floods unknown unicast to the localnet, FDB learning (§17.4) caches MACs to reduce flooding, and MAC_Binding (§17.5) lets the router resolve a next-hop. This section assembles all of them into a complete picture through a real-world case study, in which FDB learning combined with disabled port security creates a vulnerability that hijacks production traffic.

### Topology and configuration

```
Provider Network VLAN 3808 (100.64.0.0/24)
══════════════════════════════════════════
  100.64.0.1  = Fornix gateway (physical, on VLAN 3808)
  100.64.0.12 = VyOS VM (VRRP router, peers with Fornix)
  100.64.0.xx = other VMs on the same logical switch

  VyOS VM (.12):
    - port_security_enabled = False  (required for a VNF or router)
    - NB addresses = ["fa:16:3e:04:74:74 100.64.0.12", "unknown"]
    - LSP type = ""  (regular VIF)
```

### Why Neutron sets addresses to include "unknown"

Launchpad bug #1815270 records the issue: when `port_security_enabled=False`, OVN needs `"unknown"` in the addresses list so that the port can receive traffic destined for unknown MAC addresses. The Neutron ML2/OVN driver automatically sets `addresses = ["fa:16:3e:04:74:74 100.64.0.12", "unknown"]`. Bug #1838535 refined this: only set "unknown" for VIF ports (type=""), not for router, localnet, or localport types, because those special types have their own forwarding mechanisms and do not need to participate in MC_UNKNOWN.

### Why FDB learning activates on the VyOS LSP

The three AND conditions from §17.4 are all satisfied on the VyOS port: `port_security = []` (disabled), addresses contain `"unknown"`, and `type = ""` (regular VIF). Because all three conditions hold, ovn-northd generates a logical flow for FDB learning on this port:

```
ls_in_lookup_fdb: inport == "vyos-port" →
    reg0[11] = lookup_fdb(inport, eth.src); next;
ls_in_put_fdb: inport == "vyos-port" && reg0[11] == 0 →
    put_fdb(inport, eth.src); next;
```

### Poisoning mechanism, step by step

**Step 1: VyOS sends a valid packet to Fornix.** VyOS forwards traffic to the Fornix gateway (100.64.0.1) through a static route. Every packet leaving VyOS carries source MAC = fa:16:3e:04:74:74 (the real MAC of eth1) and destination MAC = fa:16:3e:35:a7:73 (the MAC of Fornix, taken from the neighbor cache). This is completely normal behavior. The Linux kernel always uses `dev->dev_addr` of the output interface as the source MAC through the function `eth_header()` in `net/ethernet/eth.c` ([kernel source](https://github.com/torvalds/linux/blob/master/net/ethernet/eth.c)). Keepalived GARP behaves the same way: the function `send_gratuitous_arp_immediate()` always uses `IF_HWADDR(ipaddress->ifp)`, the physical MAC of the interface, never the MAC of an external device ([keepalived source](https://github.com/acassen/keepalived)). The VyOS VM is **not capable** of emitting a frame with source MAC = fa:16:3e:35:a7:73.

**Step 1b: the OVN pipeline swaps eth.src and eth.dst without syncing inport and outport (BUG).** A valid packet from VyOS enters the OVN pipeline with `inport = VyOS_LSP`, `eth.src = fa:16:3e:04:74:74`, and `eth.dst = fa:16:3e:35:a7:73`. In one code path, for example when generating an ICMP "Fragmentation Needed" reply for a multichassis port, OVN swaps `eth.src` and `eth.dst` in order to build the reply packet, but **forgets to swap `inport` and `outport`**. After the swap: `inport` is still VyOS_LSP, but `eth.src` is now fa:16:3e:35:a7:73 (Fornix's MAC, which was the original destination MAC). This is exactly the bug class that Ales Musil confirmed and fixed in the patch *"physical: Prevent wrong FDB to be learned with multichassis port"* ([ovs-dev mailing list, 20 Aug 2024](https://www.mail-archive.com/ovs-dev@openvswitch.org/msg88284.html), `Reported-at: https://issues.redhat.com/browse/FDP-620`).

> **Key idea.** The root cause of the FDB poisoning is not in the VM. VyOS sends ordinary packets; the OVN pipeline creates the wrong FDB entry from the destination MAC inside that valid packet. The necessary conditions are: (1) an OVN version that has not yet fixed the inport-and-outport swap bug, (2) a port with `addresses="unknown"` (so FDB learning activates), and (3) `fdb_age_threshold=0` (so the wrong entry persists forever).

**Step 2: put_fdb writes the WRONG entry.** The frame enters br-int and matches a flow in `ls_in_lookup_fdb`:

```
lookup_fdb(inport="vyos-port", eth.src="fa:16:3e:35:a7:73")
→ reg0[11] = 0  (no entry exists for this MAC)
```

It then proceeds to `ls_in_put_fdb`:

```
reg0[11] == 0 → put_fdb(inport="vyos-port", eth.src="fa:16:3e:35:a7:73")
```

The FDB entry is created in the SB DB:

```
ovn-sbctl list fdb
mac           : "fa:16:3e:35:a7:73"
dp_key        : <VLAN3808_dp_key>
port_key      : 3                      ← VyOS LSP tunnel key!
```

**Step 3: traffic is hijacked.** When another VM sends ICMP to the gateway 100.64.0.1 (MAC = fa:16:3e:35:a7:73):

```
ls_in_l2_lkup, priority 50: eth.dst == fa:16:3e:35:a7:73?
  → NO static entry (this MAC is not in any LSP addresses)

ls_in_l2_lkup, priority 0: outport = get_fdb(eth.dst="fa:16:3e:35:a7:73")
  → FDB hit! outport = port_key 3 (VyOS LSP)

ls_in_l2_unknown, priority 0: output;
  → forwarded to the VyOS tap port instead of the localnet (bond3)!
```

The ICMP enters the VyOS VM instead of leaving toward the physical network, the gateway never receives it, and the request times out.

### Why ARP still works normally

An ARP request is a broadcast (eth.dst = ff:ff:ff:ff:ff:ff):

```
ls_in_l2_lkup, priority 72: eth.mcast && arp.op == 1
  → outport = "_MC_flood"; output;
```

ARP bypasses the FDB lookup entirely. It goes straight into `_MC_flood` at priority 72, which is far higher than the priority 0 where `get_fdb` runs. The ARP reply from the gateway arrives through the localnet, gets flooded, and reaches the correct VM. That is why ARP resolution still works while unicast traffic is hijacked: the two kinds of traffic take different branches through the pipeline.

### Why bond3 never sees the ICMP

The FDB entry `fa:16:3e:35:a7:73 → port_key 3` short-circuits the lookup before the packet has a chance to reach the localnet port. `get_fdb` returns the VyOS port_key instead of "none", so the packet does NOT enter `_MC_unknown`, is NOT flooded to the localnet, and never leaves through bond3. The frame is "absorbed" into the VyOS tap port, and the physical gateway sees no traffic at all.

### Why the FDB entry never ages out

With `fdb_age_threshold=0` (the default), ovn-northd does not run aging. The FDB entry persists until it is deleted manually (`ovn-sbctl destroy fdb <uuid>`), overwritten by another entry (same MAC, same datapath), or until ovn-controller restarts. In this case study, the wrong entry persists forever.

### Why the gateway cannot self-correct the FDB

Suppose the Fornix gateway emits a frame with source MAC fa:16:3e:35:a7:73. This frame arrives on the localnet port. But the `flags.localnet` protection (if the OVN version is recent enough) prevents the localnet from overwriting a VIF entry:

```
ls_in_lookup_fdb on a localnet port:
  flags.localnet = 1; reg0[11] = lookup_fdb(inport, eth.src);

lookup_fdb with flags.localnet=1:
  Entry exists (mac=fa:16:3e:35:a7:73, port=VyOS)
  Current port type = VIF → return 1 (entry exists, no update needed)

ls_in_put_fdb: reg0[11] == 1 → SKIP put_fdb
```

The VIF entry is protected, so the gateway cannot fix the FDB from the localnet side. This protection is the right behavior in normal cases (it prevents the localnet from overwriting a correct VIF entry), but in this case the VIF entry is wrong, and the protection becomes a barrier that prevents self-healing.

### Short-term and long-term fix

Short-term fix: delete the wrong FDB entry.

```bash
# Find the FDB entry
[controller]$ ovn-sbctl find fdb mac="fa:16:3e:35:a7:73"

# Delete it
[controller]$ ovn-sbctl destroy fdb <uuid>
```

Immediately after deletion, `get_fdb` no longer finds the MAC and sets outport="none". The pipeline falls through to MC_UNKNOWN, floods the packet to the localnet port, sends it out bond3, and reaches the physical gateway. Traffic recovers right away.

The long-term fix combines several measures: enable port security on the VyOS port if possible, which prevents FDB learning entirely; set `fdb_age_threshold > 0` on the logical switch so the wrong entry will expire; or use allowed-address-pairs instead of disabling port security wholesale, which permits a specific MAC while keeping port security on.

```bash
# Set FDB aging to 300 seconds (5 minutes) on the logical switch
[controller]$ ovn-nbctl set Logical_Switch <ls> other_config:fdb_age_threshold=300
```

### Known OVN FDB bug class: evidence that the root cause is in the pipeline

A forensic analysis of this case study determines that the root cause is not inside the VyOS VM but in the OVN pipeline layer. The conclusion rests on three pillars of evidence: complete elimination of every VM-side mechanism, confirmation of the bug class in the OVN source code, and analysis showing the VyOS configuration plays only an amplifier role.

The first pillar: VM-side elimination. The function `eth_header()` in the kernel file `net/ethernet/eth.c` always assigns `dev->dev_addr` (the MAC of the output interface) to `eth->h_source` when `saddr=NULL`. This is the only code path for every packet forwarded through `ip_forward()`. Keepalived's `send_gratuitous_arp_immediate()` always uses `IF_HWADDR(ipaddress->ifp)` over a `PF_PACKET + SOCK_RAW` socket, and the kernel does not interfere with the source MAC. The VyOS configuration has no bridge, no macvlan, no proxy_arp. Conclusion: the VM has no mechanism that could produce a frame with a source MAC different from its interface MAC.

The second pillar: a confirmed bug class in OVN. Four bug reports relate directly to FDB.

First, FDP-620 (fixed in OVN v24.09.1, patch by Ales Musil). The function `reply_imcp_error_if_pkt_too_big` in `controller/physical.c` (the upstream function name has the typo `imcp` instead of `icmp`; see the `NOTE(ihrachys)` comment near line 1905 of that file) swaps `eth.src` and `eth.dst` when generating an ICMP error for a multichassis port, but forgets to swap `inport` and `outport`. FDB learning then writes `put_fdb(inport=VIF_port, eth.src=original_dst_MAC)`, exactly the pattern of this case study. The fix adds six lines of code that swap `MFF_LOG_INPORT` and `MFF_LOG_OUTPORT` before swapping the MAC.

Second, BZ 1983111 (fixed in OVN v21.09.1). The FDB lookup and put OpenFlow flows are not installed when ovn-controller claims the first port on a logical switch; during that window, FDB learning does not function correctly.

Third, the pinctrl map size bug (fixed in v24.09.2, commit `33a6ae53`). The function `pinctrl_handle_put_fdb()` checks `hmap_count(&put_mac_bindings)` instead of `hmap_count(&put_fdbs)`, rate limiting FDB entries based on the wrong map.

Fourth, FDP-1341 (fixed in v25.03.2). VIF FDB entries are lost when an SB transaction fails, which allows localnet entries to overwrite them.

The third pillar: the VyOS configuration is an amplifier, not the root cause. `garp_master_refresh=5` creates a high traffic volume (a GARP every 5 seconds), and the static route through Fornix ensures that destination MAC fa:16:3e:35:a7:73 is always present in packets. Both increase the probability of triggering the OVN bug. But `port_security_enabled=False` is the decisive enabler (it activates FDB learning) and `fdb_age_threshold=0` is the decisive amplifier (it turns a one-time event into permanent damage).

> **Key idea.** A decisive test confirms the VM is innocent: `tcpdump -i tap-VYOS_PORT -e -n 'ether src fa:16:3e:35:a7:73'`. If, when run on the hypervisor host, this captures zero frames, that is empirical proof that VyOS never emits the wrong source MAC and that the wrong FDB entry was created by the OVN pipeline. Check the OVN version with `ovn-controller --version` and compare it with the FDP-620 fix (v24.09.1).

### Multichassis binding mechanism: the comma in `requested-chassis`

The first trigger condition of the bug lies in the port's `additional_chassis` state. This state appears only when Neutron sets `LSP.options:requested-chassis` as a comma-separated string (for example `"chassisA,chassisB"`), and only the `nova live-migrate` operation produces such a value. Other Nova operations such as `nova boot`, `reboot`, `stop`, `resize`, `evacuate`, `rebuild`, and `shelve` set a single value, which leads to ordinary single-chassis binding and does not create the multichassis state.

A detailed analysis of `lport_can_bind_on_this_chassis()`, the `claim_lport()` mechanism, the table of nine Nova operations, and the architectural reason that only live migration needs dual-active binding is presented in [SDN 3.2 on Multichassis port binding lifecycle](./19.0%20-%20ovn-multichassis-binding-and-pmtud.md).

### 58-byte Geneve overhead and pipeline PMTUD

The second trigger condition is a packet that exceeds the effective tunnel MTU. The total Geneve overhead in OVN is 58 bytes, comprising outer Ethernet 14 bytes, outer IPv4 20 bytes, outer UDP 8 bytes, Geneve base header 8 bytes, and the OVN TLV option (Class 0x0102) 8 bytes. With a physical MTU of 1500, the effective tunnel MTU drops to 1442 bytes. A 1500-byte IP packet from the VM triggers `check_pkt_larger(1442)` at table 41 (`OFTABLE_OUTPUT_LARGE_PKT_DETECT`), sets `reg9[1] = 1`, then resubmits to table 42 (`OFTABLE_OUTPUT_LARGE_PKT_PROCESS`).

At table 42, the function `reply_imcp_error_if_pkt_too_big()` swaps `eth.src` and `eth.dst` but does not swap `inport` and `outport` accordingly. Downstream FDB learning reads the old `inport` value (still VyOS LSP) and combines it with the new `eth.src` (now the Fornix MAC), which results in `put_fdb(inport=VyOS_LSP, eth.src=Fornix_MAC)`. This is precisely the wrong FDB entry that directly hijacks traffic for 100.64.0.1 to VyOS.

A byte-by-byte overhead breakdown, the full action sequence of `reply_imcp_error_if_pkt_too_big()`, and Ales Musil's six-line `MFF_LOG_INPORT/OUTPORT` patch are presented in [SDN 3.4 on Geneve overhead, pipeline PMTUD, and bug FDP-620](./19.0%20-%20ovn-multichassis-binding-and-pmtud.md).

### Mitigation and definitive fix

In the VLAN 3808 case study, the physical MTU is 1500, so the condition "packet larger than tunnel MTU" is met and the bug fires. There are two ways to reduce the risk in a running kolla-ansible environment.

The first option is to raise the physical tunnel MTU to 9000 (jumbo frames), which makes the effective tunnel MTU 8942 bytes. A 1500-byte packet from the VM no longer exceeds the threshold, so `reply_imcp_error_if_pkt_too_big()` is not triggered. This is only a mitigation because the bug remains in the code path; it is simply not triggered at the current MTU. If the VM MTU later increases or the encap overhead grows (for example with double encapsulation), the trigger condition can reappear.

The second option is to build an OVN 24.09.1 or later container via `kolla-build --template-override` together with the Ubuntu Cloud Archive Proposed pocket, then upgrade to apply the Ales Musil patch. This is the definitive fix for kolla-ansible deployments because it removes the root cause from the code path and does not depend on the operational MTU.

The `[ovn] live_migration_activation_strategy` setting in `neutron.conf` (default value `rarp`, or `chassis`, contributed by Ihar Hrachyshka in commit `949b098626b7`, Launchpad #2092250, merged 2025-01-15) does not directly prevent this bug. That setting only controls when the additional chassis location becomes available; it does not fix the wrong logic in the ICMP error path. The `rarp` strategy may reduce trigger probability in practice by limiting traffic through the tunnel during the multichassis window, but not by patching the wrong logic.

Details of the jumbo-frame trade-offs, the kernel `mtu_expires` recovery for poisoned PMTUD cache, and the full analysis of activation-strategy `rarp` versus `chassis` (including the reason RARP was chosen instead of GARP) are presented in [SDN 3.5 on activation-strategy and three-layer synchronization](./19.0%20-%20ovn-multichassis-binding-and-pmtud.md) and [SDN 3.6 on Operational tuning, jumbo frames, and MTU recovery](./19.0%20-%20ovn-multichassis-binding-and-pmtud.md).

- Identify the wrong FDB entry using `ovn-sbctl` and cross-check with port_binding.
- Demonstrate by the POE method that deleting the wrong FDB entry restores traffic.

**Before you begin:**
Access to the OVN central node (where `ovn-sbctl` runs). At least one provider network with a VNF port (port security disabled).

**Instructions:**

**1.** Verify the symptom: a VM ping to the physical gateway times out, but ARP resolution still works.

**2.** List the FDB table and identify the suspect entry by comparing the MAC in FDB against the real MAC of the gateway.

**3.** Cross-check the `port_key` in the FDB entry against `Port_Binding` to determine which port the packet is being delivered to.

**Predict (counter-hypothesis).** If the wrong FDB entry is NOT the cause (that is, the problem lies elsewhere, such as the physical network, the bridge mapping, or an ACL), then deleting the wrong FDB entry will not change behavior; traffic still times out after deletion.

**4.** Delete the suspect FDB entry and check traffic immediately.

**5.** If traffic recovers, conclude that the wrong FDB entry is the root cause.

**Explain.** When the wrong FDB entry is deleted, `get_fdb(eth.dst)` does not find the MAC and sets outport="none". The pipeline falls through to MC_UNKNOWN, floods the packet to the localnet port, and the packet leaves toward the physical network and reaches the gateway. The fact that traffic recovers immediately disproves the "problem lies elsewhere" hypothesis; the wrong FDB entry is the only cause.

After completing the lab, run `ovn-trace` with the gateway's destination MAC. The output should show the packet traversing MC_UNKNOWN to the localnet port (no longer redirected to the VNF port). Run `ovn-nbctl get Logical_Switch <ls> other_config:fdb_age_threshold`; the value must be greater than 0.

**Finish:**
The wrong FDB entry has been deleted and aging is configured. No rollback is needed.

## 17.7 - Design lessons: trade-offs and incremental evolution

### Functionality versus security: disabling port security is necessary but dangerous

VNFs (Virtual Network Functions) such as routers, load balancers, and firewalls are forced to disable port security because they need to send and receive frames with a MAC that is not their own. This is a legitimate requirement that OpenStack supports through `port_security_enabled=False`. But once port security is disabled, OVN loses the ability to validate the source MAC, which opens the door to FDB poisoning. This is an inherent design trade-off: OVN cannot at the same time allow arbitrary MAC forwarding and guarantee FDB integrity, because the two goals contradict each other at the architectural level.

The solution evolved over time rather than as a big-bang redesign: initially there was no protection, then `flags.localnet` was added to prevent the localnet from overwriting a VIF entry, then `fdb_age_threshold` was added to bound how long a wrong entry can survive, and finally `lsp_learn_mac` was added to disable learning on individual ports. Each step was a reaction to a bug or a real production scenario that the previous steps did not cover.

### Performance versus correctness: FDB caching reduces flooding but creates stale entries

Before FDB existed, every unknown unicast was flooded: bandwidth-heavy but always correct, because the packet reached every port including the right one. After FDB, unknown unicast is flooded only on the first occurrence and is then forwarded directly thereafter: bandwidth is optimized, but the trade is the possibility of stale or wrong entries leading to misdelivery.

This is the essence of the **caching problem**, the same trade-off as CPU cache, DNS cache, and ARP cache. Every cache must answer the question: how long does a stale entry survive before invalidation? Too short a TTL hurts cache hit rate (reverting to flooding); too long a TTL lets a wrong entry cause damage for longer. OVN takes a progressive approach: FDB is OFF by default (learning activates only for special ports that satisfy the three AND conditions), opt-in expansion is available via `localnet_learn_fdb`, and an aging mechanism (`fdb_age_threshold`) is provided when the operator accepts the trade-off.

### OVN evolves through incremental hardening

Looking back at the FDB development timeline shows a clear pattern: every release adds a new safeguard in response to a bug or a real production scenario.

```
2015: OVN is born; only static MAC plus unknown flooding.
       Simple, correct, but a lot of flooding.

2021 (v21.06): FDB learning; dynamic MAC caching.
       Reduces flooding but opens the FDB poisoning risk.

2022 (v22.09): localnet_learn_fdb; opt-in learning for localnet.
       Extends coverage but adds an overwrite risk.

2023 (v23.09): fdb_age_threshold; aging mechanism.
       Limits the damage from a wrong entry.

2023 (v24.03): flags.localnet; protects the VIF entry from the localnet.
       Fixes a specific race condition.

2024 (v24.09): force_fdb_lookup; granular control.
2025 (v26.03): lsp_learn_mac; per-port learning control.
       Increasingly fine-grained.
```

Every release preserves backward compatibility (the default behavior does not change), so operators can upgrade OVN without changing configuration. This is an important design lesson for distributed systems: incremental evolution lets a production system migrate gradually without taking downtime.

### "unknown" addresses, the most powerful but also most dangerous keyword in OVN

The keyword `"unknown"` activates three mechanisms at the same time: the port receives all unknown unicast traffic (via MC_UNKNOWN membership), the port activates FDB learning (when combined with disabled port security), and any MAC entering the port is recorded into the FDB without validation. Together, these three mechanisms create a large attack surface: any frame with an unfamiliar source MAC passing through that port can poison the FDB table. Before setting `addresses = ["...", "unknown"]` on any port, the operator should fully understand the implications that this document has analyzed.

### Debug checklist for unusual L2 forwarding on OVN

When you observe symptoms of L2 traffic going to the wrong port or being lost on a provider network, the recommended debug sequence based on the previous sections is:

| Symptom | Table to check | Command |
|---|---|---|

| Router cannot resolve the next-hop MAC | MAC_Binding | `ovn-sbctl list mac_binding` |
| ARP does not arrive | MC_FLOOD membership | `ovn-sbctl list multicast_group` |

## 9.25.1 Why reading 2,000 lines of `dump-flows` is a last resort: opening context

In November 2021, an OpenStack operations team at a Southeast Asian telecommunications carrier hit an incident: newly created VMs could not ping the subnet gateway. The `nova-compute` log was clean, `neutron-server` reported no error, and `ovn-controller` showed the binding had succeeded. The on-call engineer ran `ovs-ofctl dump-flows br-int` and got 2,847 lines back. He read line by line for 45 minutes and found nothing. When a senior engineer joined the shift, the senior typed one command in three seconds: `ovs-appctl ofproto/trace br-int "in_port=12,ip,nw_src=10.0.0.10,nw_dst=10.0.0.1"`. The result was immediate: a flow entry installed with the wrong priority by a security-team canary script was blocking traffic before it reached the NAT table. Resolution time when you know the tool: three seconds. Resolution time when you read the dump line by line: 45 minutes.

Today's lesson is the flow-debugging toolbox that every SDN engineer must know cold: `ofproto/trace`, `dpif/show`, `monitor`, `replace-flows`, and `diff-flows`. These are not advanced features. They are the first weapons you reach for. Reading a 500-rule dump line by line is the last resort, not the first step.

### Why dump-flows is not enough

Compass Chapter 10 states the problem clearly: *"Flow tables can become incredibly complex, but OVS has tools to make it easier to debug."* Three specific reasons explain why reading the dump is inefficient.

**One: the number of flow entries grows with scale.** An OVN compute host with 50 to 80 tenants makes `ovn-controller` produce 2,000 to 10,000 flow entries. Reading them linearly takes hours.

**Two: the order of matches is not visually obvious.** A dump prints entries by table and by descending priority, but the actual path of a packet across the pipeline can traverse eight tables and trigger 15 consecutive entries. That order cannot be reconstructed by eye from the dump.

**Three: match fields use wildcards and masks.** A flow `ip,nw_src=10.0.0.0/8,priority=100` matches every IP in subnet `10.0.0.0/8`. When you check a specific packet such as 10.0.0.5, you must compute the mask by hand. That is error prone.

`ofproto/trace` resolves all three problems in one command. It simulates the packet through the pipeline in the correct order, applies the correct masks, and prints the final result.

## 9.25.2 `ofproto/trace`: simulating a packet through the pipeline

`ovs-appctl ofproto/trace` is a command that runs over the `ovs-appctl` socket (typically `/var/run/openvswitch/ovs-vswitchd.pid.ctl`). It asks `ovs-vswitchd` to simulate the processing of a hypothetical packet without actually sending any packet through the datapath. The basic syntax per `ovs-appctl(8)`:

```
ovs-appctl ofproto/trace <bridge> "<flow-spec>"
```

Here `<flow-spec>` is a string of OVS-style match fields, for example:

```bash
sudo ovs-appctl ofproto/trace br-int \
  "in_port=12,ip,nw_src=10.0.0.10,nw_dst=10.0.0.1,tcp,tp_dst=80"
```

`ovs-vswitchd` then performs these steps:

1. Parses the flow-spec into a complete set of match field values.
2. Walks the pipeline starting at table 0, looking up each table in turn.
3. Records every match and every executed action.
4. Collects the final action set.
5. Prints the result to stdout.

No packet is sent. This is a pure simulation. It is safe in production because it produces no side effect on the datapath.

### Why `ovs-appctl` and not `ovs-ofctl`

`ovs-ofctl` is the standard OpenFlow tool. It talks to `ovs-vswitchd` over the OpenFlow protocol (TCP port 6633 or a Unix socket). `ovs-appctl` is the OVS management tool. It talks over an internal Unix socket. `ofproto/trace` is a management command, not part of the OpenFlow specification, so it lives only on `ovs-appctl`. Some older OVS versions allowed `ovs-ofctl ofproto/trace` as a shortcut, but the canonical syntax is `ovs-appctl ofproto/trace`.

## 9.25.3 Full flow-spec syntax

The flow-spec passed to `ofproto/trace` is a comma-separated string of match fields. The syntax matches `ovs-ofctl add-flow`, but it does not contain the `actions=` part. The minimum mandatory fields depend on the layer you want to trace.

**L2 only:**
```
in_port=1,dl_src=aa:bb:cc:00:00:01,dl_dst=aa:bb:cc:00:00:02,dl_type=0x0800
```

**L2 plus L3 IPv4:**
```
in_port=1,ip,nw_src=10.0.0.1,nw_dst=10.0.0.2,nw_proto=17
```

Note: when you use the alias `ip`, OVS automatically sets `dl_type=0x0800`. Likewise `arp` sets `dl_type=0x0806` and `ipv6` sets `dl_type=0x86dd`.

**L2 plus L3 plus L4 TCP:**
```
in_port=1,tcp,nw_src=10.0.0.1,nw_dst=10.0.0.2,tp_src=52134,tp_dst=80
```

The alias `tcp` automatically sets `dl_type=0x0800,nw_proto=6`. Other aliases: `udp` sets `nw_proto=17`, `icmp` sets `nw_proto=1`, and `sctp` sets `nw_proto=132`.

**Conntrack state (Part 9.24):**
```
in_port=1,tcp,nw_src=10.0.0.1,nw_dst=10.0.0.3,ct_state=+trk+est,ct_zone=7
```

### Which fields are mandatory

`ofproto/trace` does not require every field, but any missing field defaults to zero. If a flow entry in a table matches on a specific `nw_src` and you do not declare `nw_src` in the flow-spec, the simulated packet will carry `nw_src=0.0.0.0`. That may not match what you intended.

A practical rule: **declare every field that any flow rule in the relevant table matches on**. When unsure, look at `ovs-ofctl dump-flows`, list the match fields used by entries, and declare all of them in the flow-spec.

### Optional flags

```
ovs-appctl ofproto/trace br0 "<flow>" [--ct-next '<ct-next-spec>'] [-generate]
```

- `-generate`: if the pipeline contains a `learn` action, `-generate` makes `ofproto/trace` actually emit the corresponding learned flow. Without it, the trace only follows the path.
- `--ct-next`: simulates the conntrack state for the next `ct()` action encountered. This is useful when you are tracing a flow that expects `ct_state=+trk+est` but the conntrack table does not yet contain a real entry.

## 9.25.4 Reading `ofproto/trace` output: the four main blocks

The output of `ofproto/trace` has a fixed structure of four blocks:

```
Flow: <full flow-spec after OVS fills in defaults>

bridge("br-int")
-------------
 0. <match in table 0>
    <executed actions>
 1. <match in table 1>
    <executed actions>
 ...

Final flow: <flow after all field modifications applied>
Megaflow: <flow with the wildcard mask OVS computes for caching>
Datapath actions: <linear action sequence for the kernel datapath>
```

### Block 1: `Flow:`

This is the full flow-spec after OVS fills in default values for fields you did not declare. For example, if you pass `in_port=1,tcp,tp_dst=80`, the `Flow:` block displays:

```
Flow: tcp,in_port=1,vlan_tci=0x0000,dl_src=00:00:00:00:00:00,dl_dst=00:00:00:00:00:00,nw_src=0.0.0.0,nw_dst=0.0.0.0,nw_tos=0,nw_ecn=0,nw_ttl=0,tp_src=0,tp_dst=80
```

Check this block to see how OVS interpreted your flow-spec. Every field you did not declare becomes zero, which can be the reason the trace fails to match the real flow.

### Block 2: `bridge(...)` and the lookup chain

This is the most important part. It is the sequence of matches across the pipeline. The format:

```
bridge("br-int")
-------------
 0. ip,nw_dst=10.0.0.0/24, priority 32768
    goto_table:1
 1. ip,nw_dst=10.0.0.1, priority 40000
    mod_dl_src:aa:bb:cc:dd:ee:ff
    mod_dl_dst:11:22:33:44:55:66
    dec_ttl
    goto_table:2
 2. dl_dst=11:22:33:44:55:66, priority 32768
    output:3
```

Each line begins with a number. That is the table ID. Next is the matching flow rule (displaying the match fields that matched) and the priority. The indented line below lists the executed actions. The action `goto_table:N` ends matching at the current table and moves on to table N.

### Block 3: `Final flow:`

After every `mod_*`, `dec_ttl`, and `set_field` action has been applied, this is the final flow before the packet leaves the bridge. Compare it with `Flow:` to see which fields changed:

```
Final flow: tcp,in_port=1,...,dl_src=aa:bb:cc:dd:ee:ff,dl_dst=11:22:33:44:55:66,nw_ttl=63
```

Three obvious changes: source MAC and destination MAC (from `mod_dl_*`), and TTL decreased from 64 to 63 (from `dec_ttl`).

### Block 4: `Megaflow:` and `Datapath actions:`

`Megaflow:` is the wildcarded flow that OVS will install into the kernel cache after it processes a real packet with the same characteristics. The megaflow always has a less precise mask than the original match. OVS keeps only the fields that actually contributed to the decision. For example:

```
Megaflow: recirc_id=0,eth,ip,in_port=1,nw_dst=10.0.0.0/24,nw_frag=no
```

`nw_src` does not appear in the megaflow because no flow rule matched on `nw_src`. `tp_dst` is also absent. OVS drops both because they were not needed. Megaflow construction is detailed in Part 9.15.

`Datapath actions:` is the linear action sequence the kernel datapath will execute when it receives a real packet with the same megaflow match. This is the "compiled" result of the entire pipeline:

```
Datapath actions: set(eth(src=aa:bb:cc:dd:ee:ff,dst=11:22:33:44:55:66)),set(ipv4(ttl=-1)),3
```

Three actions: set MAC (combining `mod_dl_src` and `mod_dl_dst`), set IPv4 (decrement TTL), and output to port 3. The kernel datapath does not understand OpenFlow. It executes this datapath-level sequence.

> **Key point.** When debugging, always read three specific places: (a) the first table ID, to confirm the incoming packet matched the flow as expected; (b) `Final flow:`, to see which fields were modified; (c) `Datapath actions:`, to check whether the trailing action is `drop`. If you see `Datapath actions: drop`, the pipeline ends by dropping the packet.

## 9.25.5 Three dump-flow commands: a careful comparison

OVS provides three dump-flow commands. Each serves a different purpose. Confusing them is the source of many wasted debug sessions.

### `ovs-ofctl dump-flows <bridge>`

This dumps the OpenFlow flow entries managed by userspace `ovs-vswitchd`. This is the table that really decides the path of a packet. Output:

```bash
sudo ovs-ofctl dump-flows br-int
```

```
cookie=0x0, duration=12345.67s, table=0, n_packets=8834, n_bytes=917136, idle_age=2, priority=100,ip,nw_src=10.0.0.0/8 actions=goto_table:10
cookie=0x5, duration=11234.12s, table=10, n_packets=7812, n_bytes=809432, priority=50,tcp,tp_dst=80 actions=output:3
cookie=0x0, duration=12345.67s, table=0, n_packets=156, n_bytes=17248, priority=0 actions=NORMAL
```

Each line is one flow entry. Reading the fields:
- `cookie=0xN`: the flow-group label (Part 9.19 §19.4).
- `duration=Ns`: how long the entry has existed.
- `n_packets` and `n_bytes`: the match counters.
- `idle_age`: seconds since the last match.
- `priority=N,<match>`: priority and match fields.
- `actions=...`: the action sequence.

### `ovs-dpctl dump-flows [datapath]`

This dumps the **megaflow cache** in the kernel datapath. It is not OpenFlow flow entries. It is the cache that `ovs-vswitchd` compiled down to the kernel for packets that have already passed through:

```bash
sudo ovs-dpctl dump-flows
```

```
recirc_id(0),in_port(2),eth(src=aa:bb:cc:00:00:01,dst=aa:bb:cc:00:00:02),eth_type(0x0800),ipv4(src=10.0.0.1,dst=10.0.0.2,frag=no), packets:1234, bytes:123456, used:0.123s, actions:3
```

Three differences from `ovs-ofctl dump-flows`:
- **Different format**: it uses datapath syntax with match fields inside parentheses.
- **Fewer entries**: the cache contains only megaflows for packets that have been seen, not the full OpenFlow table.
- **Narrower wildcards**: megaflows wildcard only the fields that are truly unnecessary, so each entry can be very specific.

Use `ovs-dpctl dump-flows` to debug why a specific packet is dropped in the kernel, instead of reading 2,000 OpenFlow entries. If the megaflow cache has no entry for that packet, the kernel performs an upcall into userspace, where `ovs-vswitchd` consults the full OpenFlow table.

### `ovs-appctl bridge/dump-flows <bridge>`

This shows **every flow**, including hidden flows that `ovs-ofctl dump-flows` does not display. OVS silently installs some flows for the following features:
- **In-band control**: a flow that allows controller traffic (OpenFlow packet-in and packet-out) to traverse the bridge even when the user flow table is empty.
- **Fail-open**: when the controller connection is lost, an `actions=NORMAL` flow is installed so the bridge keeps forwarding.
- **STP and LLDP**: flows that capture BPDU and LLDP frames and send them to the controller.

```bash
sudo ovs-appctl bridge/dump-flows br-int
```

The output adds a `duration` column showing `n/a` for permanent flows. The match clause may contain an internal `hidden=true` flag.

Compass Chapter L states: *"The bridge/dump-flow command is essential when debugging controller behavior, if dump-flow (user-level) shows empty but traffic still passes, the hidden flow are doing the work."*

### Quick comparison table

| Command | Layer | Content | When to use |
|---------|-------|---------|-------------|
| `ovs-ofctl dump-flows` | Userspace OpenFlow | Flow entries installed by users and the controller | Verify the flow entries you added with `add-flow` |
| `ovs-dpctl dump-flows` | Kernel megaflow cache | Megaflows already compiled for packets seen | Debug why a specific packet is dropped in the kernel |
| `ovs-appctl bridge/dump-flows` | Userspace plus hidden flows | All flows, including in-band and fail-open | Inspect hidden flows the controller installs automatically |

## 9.25.6 `ovs-appctl dpif/show`: datapath health

When the pipeline is too heavy, the kernel performs many upcalls and the cache fills, performance degrades. `dpif/show` exposes datapath health statistics:

```bash
sudo ovs-appctl dpif/show
```

```
system@ovs-system:
  lookups: hit:45234567 missed:234561 lost:12
  flows: 2345
  masks: hit:45234567 total:32 hit/pkt:32.00
  port 0: ovs-system (internal)
  port 1: br-int (internal)
  port 2: veth0
  port 3: veth1
```

Three important indicators:
- **`lookups:` hit, missed, lost**: number of cache lookups that hit, that missed (forcing an upcall to userspace), and that were lost (dropped because the cache is full). A high `missed` count means many packets must upcall, and performance degrades.
- **`flows:`**: number of megaflows currently in the cache. The default limit is 200,000, adjustable via `ovs-appctl upcall/set-flow-limit`.
- **`masks: hit/pkt`**: average number of masks each packet must scan through in the tuple space search. A high value (greater than 40) is a sign that the flow table is too fragmented.

Compass Chapter Q stresses: *"Too many upcall relative to packet rate means flow are being evicted too fast; raise with `ovs-appctl upcall/set-flow-limit 500000`. Tune handler and revalidator thread counts via `other_config:n-handler-threads` and `other_config:n-revalidator-threads`."*

### Related `dpif/` commands

```bash
ovs-appctl dpif/show-dp-features    # Show datapath-supported features (conntrack, IPsec, etc.)
ovs-appctl upcall/show              # Show upcalls awaiting handling
ovs-appctl upcall/set-flow-limit N  # Cap the megaflow cache
ovs-appctl revalidator/wait         # Wait for all revalidators to finish (purging stale flows)
ovs-appctl revalidator/purge        # Drop every megaflow immediately
```

## 9.25.7 Flow-table hygiene: `monitor`, `replace-flows`, `diff-flows`

In a production environment, every change to the flow table must be **observable, reversible, and atomic**. OVS provides three tools that satisfy these requirements.

### `ovs-ofctl monitor <bridge> watch:`

Stream flow changes to stdout in real time:

```bash
sudo ovs-ofctl monitor br-int watch:
```

Leave this terminal running in the background. When someone (or the controller) calls `add-flow`, `mod-flow`, or `del-flows`, you see the event immediately:

```
NXST_FLOW_MOD: ADDED: table=0, priority=100, in_port=1, dl_type=0x0800, actions=output:2
NXST_FLOW_MOD: DELETED: table=0, priority=50, in_port=3, actions=drop
```

Use this when debugging a controller: run `monitor` in parallel with the controller application to see exactly which flows are installed or removed when the user clicks a button in the UI.

### `ovs-ofctl replace-flows <bridge> <file>`

Replace the **entire** flow table with the content of a file, atomically, with no intermediate state ever visible:

```bash
# Prepare the flow-entry file
cat > /tmp/new-flows.txt <<EOF
table=0,priority=100,ip,nw_dst=10.0.0.1,actions=output:3
table=0,priority=50,actions=NORMAL
EOF

# Replace everything
sudo ovs-ofctl replace-flows br-int /tmp/new-flows.txt
```

Unlike `add-flow` (which only adds) or `add-flows` from a file (which only batch-adds), `replace-flows`:
- Removes every existing flow (except hidden in-band flows).
- Installs the new flows from the file.
- Performs all of this inside a single transaction. There is no moment in which the table is empty.

This command supports a blue-green flow-table deployment pattern: generate the new flow file, review it, then run `replace-flows` to cut over.

### `ovs-ofctl diff-flows <bridge> <file>`

**Compare without applying**: shows which flows in the file differ from the current table.

```bash
sudo ovs-ofctl diff-flows br-int /tmp/new-flows.txt
```

Diff-style output:
```
-table=0, priority=100, ip, nw_dst=10.0.0.2, actions=output:4
+table=0, priority=100, ip, nw_dst=10.0.0.1, actions=output:3
 table=0, priority=50, actions=NORMAL
```

A `-` line is a flow that `replace-flows` would remove. A `+` line is a flow that would be added. An unmarked line stays as it is.

Compass Chapter 10 says: *"`ovs-ofctl diff-flows br0 new.flows` shows what would change without applying it, always run this in production before `replace-flows`."* This is a mandatory rule: never run `replace-flows` without running `diff-flows` first.

### Standard flow-table hygiene workflow

```
1. ovs-ofctl dump-flows br-int > current.txt     # Capture the current table
2. (generate new.txt from the controller or a script)
3. ovs-ofctl diff-flows br-int new.txt            # Inspect the diff
4. (review the diff, confirm with the team)
5. ovs-ofctl replace-flows br-int new.txt         # Apply atomically
6. (watch dump-flows, verify real traffic)
7. If something goes wrong: ovs-ofctl replace-flows br-int current.txt   # Rollback
```

## 9.25.8 Three NSRC tracing examples: a four-rule firewall

The NSRC OpenVSwitch slides (pages 17 to 20) present a simple firewall: block every TCP flow except port 80, allow non-TCP, and allow any packet from port 2. The four flow rules:

```bash
# Flow A: forward TCP from port 1 to the next pipeline stage
sudo ovs-ofctl add-flow br0 "priority=100,tcp,in_port=1,actions=resubmit:4000"

# Flow B: allow TCP port 80 (implicit drop for any other port)
sudo ovs-ofctl add-flow br0 "priority=100,tcp,in_port=4000,tp_dst=80,actions=NORMAL"

# Flow C: allow any non-TCP traffic from port 1
sudo ovs-ofctl add-flow br0 "priority=90,in_port=1,actions=NORMAL"

# Flow D: allow any traffic from port 2
sudo ovs-ofctl add-flow br0 "priority=100,in_port=2,actions=NORMAL"
```

Note: `resubmit:4000` invokes a "virtual port" 4000 that does not really exist. This is the classic technique of using `resubmit(port,table)` to label a pipeline stage. Modern OVS replaces this idiom with `goto_table` (Part 9.22), but the NSRC slides keep the older style for illustration.

### Three trace scenarios

**Scenario 1: ICMP arriving on port 1 (allowed).**

```bash
sudo ovs-appctl ofproto/trace br0 "in_port=1,icmp"
```

Expected:
- Flow C matches (priority 90, in_port=1, action `NORMAL`).
- Flow A does not match because the packet is not TCP.
- `Datapath actions: <NORMAL processing output>`.

**Scenario 2: TCP port 80 arriving on port 1 (allowed).**

```bash
sudo ovs-appctl ofproto/trace br0 "in_port=1,tcp,tp_dst=80"
```

Expected:
- Flow A matches (priority 100, tcp, in_port=1): the action `resubmit:4000` changes `in_port` to 4000 and resubmits.
- Flow B matches (priority 100, tcp, in_port=4000, tp_dst=80): action `NORMAL`.
- `Datapath actions: <NORMAL>`.

**Scenario 3: TCP port 22 arriving on port 1 (blocked).**

```bash
sudo ovs-appctl ofproto/trace br0 "in_port=1,tcp,tp_dst=22"
```

Expected:
- Flow A matches: resubmit to port 4000.
- After resubmission: flow B does not match (because `tp_dst=22` is not 80). Flow C does not match (because `in_port=4000` is not 1). Flow D does not match (because `in_port=4000` is not 2).
- No flow matches, so the packet is implicitly dropped.
- `Datapath actions: drop`.

This is the canonical pattern for firewall debugging: trace each expected traffic class and verify that the actions match the design.

## Guided Exercise 1: debugging a flow table with a cookie conflict and a wrong priority

**Goal.** Observe directly how `ofproto/trace` exposes a common flow-table bug, an ACL-deny rule installed at a priority lower than the default forwarding rule, so the deny never matches. Compare the debug time with `ofproto/trace` (seconds) against the time spent reading `dump-flows` (minutes).

**Hypothesis to verify:**

> *"When an ACL-deny flow is installed at a priority lower than the default forwarding flow, `ofproto/trace` shows the forwarding flow matching first, and the deny never fires. `Datapath actions` shows normal output instead of `drop`. This is the classic 'the rule does not work' bug that is hard to spot by reading the dump."*

### Setup

Create a test bridge with three ports:

```bash
sudo ovs-vsctl add-br test-br
sudo ip link add veth0 type veth peer name veth0-peer
sudo ip link add veth1 type veth peer name veth1-peer
sudo ip link add veth2 type veth peer name veth2-peer
sudo ovs-vsctl add-port test-br veth0 -- add-port test-br veth1 -- add-port test-br veth2
```

### Step 1: install the correct flow set (ACL deny at higher priority)

```bash
sudo ovs-ofctl del-flows test-br

# Flow 1: ACL deny h3 (port 3) to h1 (port 1), priority 40000
sudo ovs-ofctl add-flow test-br \
  "priority=40000,ip,in_port=3,nw_dst=10.0.0.1,actions=drop"

# Flow 2: default forwarding
sudo ovs-ofctl add-flow test-br "priority=100,actions=NORMAL"
```

Trace h3-to-h1 traffic. Expected outcome: drop.

```bash
sudo ovs-appctl ofproto/trace test-br \
  "in_port=3,ip,nw_src=10.0.0.3,nw_dst=10.0.0.1"
```

Result (documentation-plausible):
```
 0. ip,in_port=3,nw_dst=10.0.0.1, priority 40000
    drop

Datapath actions: drop
```

The ACL works as designed.

### Step 2: install the buggy flow set (priorities reversed)

Simulate a new engineer typing the ACL priority incorrectly:

```bash
sudo ovs-ofctl del-flows test-br

# Flow 1: ACL deny (WRONG: priority 50, lower than the default 100)
sudo ovs-ofctl add-flow test-br \
  "priority=50,ip,in_port=3,nw_dst=10.0.0.1,actions=drop"

# Flow 2: default forwarding (priority 100, higher)
sudo ovs-ofctl add-flow test-br "priority=100,actions=NORMAL"
```

Trace the same traffic again:

```bash
sudo ovs-appctl ofproto/trace test-br \
  "in_port=3,ip,nw_src=10.0.0.3,nw_dst=10.0.0.1"
```

Result (documentation-plausible):
```
 0. priority 100
    NORMAL
     -> forwarding to learned port

Datapath actions: 1
```

**The ACL drop does not appear in the trace.** Flow 1 never matches because its priority of 50 is below the priority 100 of flow 2. `Datapath actions: 1` means the packet is forwarded to port 1.

### Step 3: compare with reading `dump-flows`

```bash
sudo ovs-ofctl dump-flows test-br
```

Result:
```
cookie=0x0, duration=12.3s, table=0, n_packets=0, n_bytes=0, priority=100 actions=NORMAL
cookie=0x0, duration=23.4s, table=0, n_packets=0, n_bytes=0, priority=50,ip,in_port=3,nw_dst=10.0.0.1 actions=drop
```

You can spot the reversed priority from the dump, but only when you **know** the implicit defaults and the forwarding priority of 100. In a 500-entry flow table, this is extremely hard to spot by eye. `ofproto/trace` shows the matching rule **directly**, with no priority arithmetic in your head.

### Step 4: a falsification (POE) of "reading dump-flows is enough"

The opposite hypothesis to refute:

> *"I do not need `ofproto/trace`. Reading `dump-flows` is enough to tell which rule blocks the packet."*

Counter-evidence: with only two rules as above, a sharp eye can detect the reversed priority. But scale up to 50 rules with grouped `cookie` values, varying `idle_timeout`, varying `hard_timeout`, and 10-column match fields, and tracking priority in your head becomes impossible. Test:

```bash
# Install 50 flow rules with random priorities between 50 and 150
for i in $(seq 1 50); do
  p=$((RANDOM % 100 + 50))
  sudo ovs-ofctl add-flow test-br \
    "priority=$p,ip,nw_src=10.0.${i}.0/24,actions=output:1"
done

# Dump for inspection
sudo ovs-ofctl dump-flows test-br | wc -l    # more than 50 lines
sudo ovs-ofctl dump-flows test-br | head -10
```

Now try to identify by eye which rule matches a packet `nw_src=10.0.25.5`: it takes minutes and is error prone.

Use `ofproto/trace` instead:

```bash
sudo ovs-appctl ofproto/trace test-br "ip,nw_src=10.0.25.5,nw_dst=1.1.1.1"
```

Two seconds, exact answer.

### Lesson

Reading the dump line by line is a static skill. `ofproto/trace` is a dynamic simulation skill. The static read does not show the effect of priority. The dynamic simulation shows it immediately. The practical rule: when the flow table exceeds 20 entries, run `ofproto/trace` first; read `dump-flows` only to verify the match expression of the entry you have already identified.

## Guided Exercise 2: tracing a packet across two bridges connected by a patch port

**Goal.** Engineers operating OVN-OpenStack or gateway architectures regularly face a two-bridge topology: `br-int` (integration, carrying tenant traffic) and `br-ex` (external, connected to the physical uplink). The two bridges are connected by a patch-port pair. When a packet is dropped between tenant and physical, the trace must run **on each bridge separately**. It does not automatically follow the patch port. This exercise exposes that limitation and teaches you how to chain traces manually.

**Hypothesis to verify:**

> *"When `ofproto/trace br-int` reaches an output to a patch port, it stops there; it does not continue automatically into `br-ex`. To see the packet leave the physical port, you must run a second trace, `ofproto/trace br-ex`, with `in_port` set to the peer patch port. The cross-bridge debugging skill is the skill of chaining traces by hand."*

### Setup

Create two bridges connected by a patch-port pair. Add one tenant veth to `br-int` and one uplink veth to `br-ex`:

```bash
sudo ovs-vsctl add-br br-int
sudo ovs-vsctl add-br br-ex

# Patch port pair
sudo ovs-vsctl \
  -- add-port br-int patch-to-ex -- set interface patch-to-ex type=patch options:peer=patch-to-int \
  -- add-port br-ex patch-to-int -- set interface patch-to-int type=patch options:peer=patch-to-ex

# Tenant port on br-int
sudo ip link add veth-tenant type veth peer name veth-tenant-peer
sudo ovs-vsctl add-port br-int veth-tenant

# Uplink port on br-ex
sudo ip link add veth-uplink type veth peer name veth-uplink-peer
sudo ovs-vsctl add-port br-ex veth-uplink
```

Inspect the OpenFlow port number for each port:

```bash
sudo ovs-ofctl show br-int
sudo ovs-ofctl show br-ex
```

Result (documentation-plausible):
```
# br-int
 1(veth-tenant): addr:...
 2(patch-to-ex): addr:...
# br-ex
 1(veth-uplink): addr:...
 2(patch-to-int): addr:...
```

### Step 1: install flows that forward tenant traffic to the patch port

On `br-int`, traffic that stays inside the tenant subnet stays local; traffic outside the tenant subnet is forwarded to `patch-to-ex`:

```bash
sudo ovs-ofctl add-flow br-int \
  "table=0,priority=100,ip,nw_dst=10.100.0.0/16,actions=output:1"
sudo ovs-ofctl add-flow br-int \
  "table=0,priority=50,ip,actions=output:2"
```

On `br-ex`, packets arriving from `patch-to-int` are forwarded to the uplink:

```bash
sudo ovs-ofctl add-flow br-ex \
  "table=0,priority=100,in_port=2,actions=output:1"
```

### Step 2: trace `br-int`, observe the stop at the patch port

```bash
sudo ovs-appctl ofproto/trace br-int \
  "in_port=1,ip,nw_src=10.100.0.5,nw_dst=8.8.8.8"
```

Result (documentation-plausible):
```
Flow: ip,in_port=1,vlan_tci=0x0000,dl_src=00:00:00:00:00:00,dl_dst=00:00:00:00:00:00,
      nw_src=10.100.0.5,nw_dst=8.8.8.8,nw_proto=0,nw_tos=0,nw_ttl=0

bridge("br-int")
----------------
 0. ip, priority 50
    output:2

Final flow: unchanged
Megaflow: recirc_id=0,eth,ip,in_port=1,nw_dst=8.0.0.0/5,nw_frag=no
Datapath actions: 6
```

The trace stops at `output:2`. There is no following `bridge("br-ex")` block. The patch port carries the packet across bridges at the kernel datapath layer, but `ofproto/trace` sees only the OpenFlow pipeline of **one** bridge at a time.

### Step 3: chain the trace on `br-ex` with the peer `in_port`

On the `br-ex` side, `patch-to-int` is OpenFlow port 2. The second trace:

```bash
sudo ovs-appctl ofproto/trace br-ex \
  "in_port=2,ip,nw_src=10.100.0.5,nw_dst=8.8.8.8"
```

Result (documentation-plausible):
```
bridge("br-ex")
---------------
 0. in_port=2, priority 100
    output:1

Datapath actions: 7
```

The two consecutive traces expose the entire journey: `veth-tenant -> br-int (patch-to-ex) -> br-ex (patch-to-int) -> veth-uplink`.

### Step 4: a falsification (POE) of "the trace follows patch ports automatically"

The opposite hypothesis to refute:

> *"`ofproto/trace` will follow the patch port automatically because OVS knows the patch pair is configured as peers."*

Counter-evidence: the patch port is handled at the datapath layer (in the kernel module, or in `dpif-netdev` for userspace DPDK), not in the OpenFlow pipeline. `ofproto/trace` only simulates the OpenFlow lookup of a single bridge. Chaining traces by hand is the standard workflow for every OVN gateway node.

```bash
# A trace does not follow: it stops at output:2:
sudo ovs-appctl ofproto/trace br-int \
  "in_port=1,ip,nw_src=10.100.0.5,nw_dst=8.8.8.8"

# The correct workflow: chain two traces manually (see Step 2 and Step 3 above).
```

### Lesson

A patch port is a cross-bridge mechanism inside the datapath, not inside OpenFlow. When you debug a path that crosses several bridges, what you really need is a **chain of traces**, one command per bridge. A production OVN-OpenStack gateway typically has three bridges in series: `br-int` (tenant), `br-tun` (overlay), `br-ex` (uplink). Debugging a packet drop at the gateway means three chained `ofproto/trace` runs.

## Guided Exercise 3: tracing a pipeline that carries state in registers and metadata

**Goal.** A multi-table OVN pipeline uses `reg0..reg15` (sixteen 32-bit registers) and `metadata` (64 bits) to carry state between tables. When a flow is dropped at table 50 after passing through tables 0, 15, 20, 30, and 40, you must read the register values to understand the context: which tenant ID, which conntrack zone, which routing decision was taken. This exercise teaches you how to read registers in trace output and rebuilds a simple stateful pipeline to practise on.

**Hypothesis to verify:**

> *"`ofproto/trace` clearly shows each register value after every `load:` or `set_field:`, and the trailing `Final flow:` line carries the accumulated final state. Tracing a pipeline of five or more tables without reading the registers is incomplete debugging."*

### Setup

Create a bridge with three ports:

```bash
sudo ovs-vsctl add-br reg-test
sudo ip link add v1 type veth peer name v1-peer
sudo ip link add v2 type veth peer name v2-peer
sudo ip link add v3 type veth peer name v3-peer
sudo ovs-vsctl add-port reg-test v1 \
  -- add-port reg-test v2 \
  -- add-port reg-test v3
```

### Step 1: install a three-table pipeline that uses `reg0` as a tenant ID

Pipeline logic:

- Table 0: classifier, sets `reg0` to the tenant ID based on `in_port`.
- Table 1: policy check, only tenant `0x5` is allowed.
- Table 2: forward to port 3.

```bash
sudo ovs-ofctl del-flows reg-test

# Table 0: classify
sudo ovs-ofctl add-flow reg-test \
  "table=0,priority=100,in_port=1,actions=load:0x5->NXM_NX_REG0[],resubmit(,1)"
sudo ovs-ofctl add-flow reg-test \
  "table=0,priority=100,in_port=2,actions=load:0x7->NXM_NX_REG0[],resubmit(,1)"

# Table 1: policy
sudo ovs-ofctl add-flow reg-test \
  "table=1,priority=100,reg0=0x5,actions=resubmit(,2)"
sudo ovs-ofctl add-flow reg-test \
  "table=1,priority=50,actions=drop"

# Table 2: forward
sudo ovs-ofctl add-flow reg-test \
  "table=2,priority=100,actions=output:3"
```

### Step 2: trace allowed-tenant traffic (reg0=0x5 from in_port=1)

```bash
sudo ovs-appctl ofproto/trace reg-test "in_port=1,ip,nw_src=10.0.0.1,nw_dst=10.0.0.99"
```

Result (documentation-plausible):
```
Flow: ip,in_port=1,...,nw_src=10.0.0.1,nw_dst=10.0.0.99

bridge("reg-test")
------------------
 0. in_port=1, priority 100
    load:0x5->NXM_NX_REG0[]
    resubmit(,1)
 1. reg0=0x5, priority 100
    resubmit(,2)
 2. priority 100
    output:3

Final flow: reg0=0x5,ip,in_port=1,nw_src=10.0.0.1,nw_dst=10.0.0.99
Datapath actions: 3
```

Notice the trailing `Final flow: reg0=0x5,...` line. The register carries state across the pipeline. Table 1 matches on `reg0=0x5` precisely because table 0 has already executed the `load`.

### Step 3: trace disallowed-tenant traffic (reg0=0x7 from in_port=2)

```bash
sudo ovs-appctl ofproto/trace reg-test "in_port=2,ip,nw_src=10.0.0.2,nw_dst=10.0.0.99"
```

Result (documentation-plausible):
```
bridge("reg-test")
------------------
 0. in_port=2, priority 100
    load:0x7->NXM_NX_REG0[]
    resubmit(,1)
 1. priority 50
    drop

Final flow: reg0=0x7,ip,in_port=2,nw_src=10.0.0.2,nw_dst=10.0.0.99
Datapath actions: drop
```

The trace makes the cause clear: table 0 loads `reg0=0x7`. Table 1 has no priority-100 rule that matches `reg0=0x7`, so the lookup falls to the default priority-50 drop.

### Step 4: a falsification (POE) of "reading the match field is enough; you can ignore registers"

Opposite hypothesis:

> *"I only need to read the `match:` of each table to know which rule matched. I do not need to care about registers."*

Counter-evidence: look at the Step 3 output. The table-1 block shows only `priority 50, drop`. The question becomes: **why** the drop? The answer is on the `Final flow: reg0=0x7` line. Without reading the register, you do not know why the priority-100 rule that matches `reg0=0x5` did not match this packet. In a pipeline of ten or more tables that carries three or four kinds of state (tenant, zone, routing decision), debugging without reading registers is debugging blind.

### Lesson

OVN produces pipelines of 30 or more tables that use `reg0` (tenant or logical datapath ID), `reg1` (logical input port), `reg2..reg9` (pipeline metadata for routing), and `metadata` (the 64-bit conntrack zone). Tracing OVN in production forces you to read registers at every resubmission. Operational tip: always read the trailing `Final flow:` line, which holds the accumulated final state; track transitions by following the `load:` or `set_field:` actions inside each `bridge(...)` block to know which table set the state.

## Guided Exercise 4: tracing recirculation through `ct()` and tunnel decapsulation

**Goal.** OVS has two common recirculation mechanisms: (1) `ct(commit)` or `ct(table=N)` sends the packet through conntrack and injects it back into the pipeline with an updated `ct_state`; (2) tunnel decapsulation (VXLAN or Geneve) strips the outer header and recirculates the inner packet through the pipeline. `ofproto/trace` splits the result into several consecutive `bridge(...)` blocks. This exercise teaches you how to read a trace with recirculation, how to distinguish it from a normal trace, and how to use the `--ct-next` flag to simulate the correct state.

**Hypothesis to verify:**

> *"When the trace meets a `ct(...)` action, it does not stop. It continues as a new recirculation block. The `ct_state` is recomputed based on the `--ct-next` flag you pass. Without that flag, the trace assumes `ct_state=0` (untracked) and misses every stateful rule."*

### Setup

A simple two-port bridge with a two-table conntrack pipeline:

```bash
sudo ovs-vsctl add-br ct-test
sudo ip link add v1 type veth peer name v1-peer
sudo ip link add v2 type veth peer name v2-peer
sudo ovs-vsctl add-port ct-test v1 -- add-port ct-test v2
```

### Step 1: install a simple stateful firewall pipeline

Logic:

- Table 0: send TCP to conntrack, recirculate to table 1.
- Table 1: allow established; for a new inbound packet to port 80, commit the tuple and allow; drop the rest.

```bash
sudo ovs-ofctl del-flows ct-test

# Table 0: send to ct
sudo ovs-ofctl add-flow ct-test \
  "table=0,priority=100,tcp,actions=ct(table=1,zone=1)"
sudo ovs-ofctl add-flow ct-test \
  "table=0,priority=50,actions=drop"

# Table 1: stateful decisions
sudo ovs-ofctl add-flow ct-test \
  "table=1,priority=100,ct_state=+est+trk,actions=output:2"
sudo ovs-ofctl add-flow ct-test \
  "table=1,priority=100,ct_state=+new+trk,tcp,tp_dst=80,actions=ct(commit,zone=1),output:2"
sudo ovs-ofctl add-flow ct-test \
  "table=1,priority=50,actions=drop"
```

### Step 2: trace a new packet to port 80 (simulating an inbound SYN)

```bash
sudo ovs-appctl ofproto/trace ct-test \
  "in_port=1,tcp,nw_src=10.0.0.10,nw_dst=10.0.0.20,tp_src=54321,tp_dst=80,tcp_flags=syn" \
  --ct-next "new,trk"
```

Result (documentation-plausible):
```
Flow: tcp,in_port=1,...,tp_dst=80,tcp_flags=syn

bridge("ct-test")
-----------------
 0. tcp, priority 100
    ct(table=1,zone=1)
     -> A clone of the packet is forked to recirculate.
        The forked pipeline will be resumed at table 1.

Final flow: tcp,in_port=1,tp_dst=80,tcp_flags=syn
Megaflow: ...
Datapath actions: ct(zone=1),recirc(0x1)

===============================================================================
recirc(0x1) - resume conntrack processing
===============================================================================

Flow: ct_state=new|trk,ct_zone=1,tcp,...,tp_dst=80

bridge("ct-test")
-----------------
 1. ct_state=+new+trk,tcp,tp_dst=80, priority 100
    ct(commit,zone=1)
    output:2

Final flow: ct_state=new|trk,ct_zone=1,tcp,...
Datapath actions: ct(commit,zone=1),2
```

The trace is split into two blocks. The first block runs table 0 and emits the actions `ct(...)` and `recirc(0x1)`. The second block resumes after conntrack with `ct_state=new|trk` injected by the `--ct-next "new,trk"` flag. The trailing `ct(commit)` action stores the tuple in zone 1 of the conntrack table.

### Step 3: trace a reply ACK on an established connection

Pass the flag `--ct-next "est,trk"` to simulate a conntrack that already holds an old entry:

```bash
sudo ovs-appctl ofproto/trace ct-test \
  "in_port=2,tcp,nw_src=10.0.0.20,nw_dst=10.0.0.10,tp_src=80,tp_dst=54321,tcp_flags=ack" \
  --ct-next "est,trk"
```

Abridged result (documentation-plausible):
```
bridge("ct-test")
 0. tcp, priority 100
    ct(table=1,zone=1)
     -> recirc

recirc(0x2) - resume
Flow: ct_state=est|trk,...

bridge("ct-test")
 1. ct_state=+est+trk, priority 100
    output:2

Datapath actions: ct(zone=1),2
```

There is no `ct(commit)` in the second block because the established reply only updates the timer; it does not create a new tuple.

### Step 4: a falsification (POE) of "you can still trace stateful flows without `--ct-next`"

Opposite hypothesis:

> *"I do not need `--ct-next` because OVS will compute `ct_state` automatically from the real conntrack table."*

Counter-evidence: run the trace without the flag:

```bash
sudo ovs-appctl ofproto/trace ct-test \
  "in_port=1,tcp,nw_src=10.0.0.10,nw_dst=10.0.0.20,tp_src=54321,tp_dst=80,tcp_flags=syn"
```

Result (documentation-plausible):
```
bridge("ct-test")
 0. tcp, priority 100
    ct(table=1,zone=1)

recirc - resume
Flow: ct_state=0,...

bridge("ct-test")
 1. priority 50
    drop

Datapath actions: ct(zone=1),drop
```

Without `--ct-next`, the trace assumes `ct_state=0` (untracked). Your rule needs `+trk+new`, but neither bit is set, so the priority-100 rule misses, and the lookup falls into the default priority-50 drop. The trace is wrong relative to real-world traffic, where the real conntrack would inject the `+trk+new` bits.

### Lesson

Conntrack and recirculation break the assumption that the trace is a straight line through the pipeline. Three production habits:

1. Always ask: in which conntrack state is this packet? Use `--ct-next "new,trk"`, `"est,trk"`, or `"rel,trk"` according to the scenario you simulate.
2. Read the output top to bottom. Each `bridge(...)` block is a different recirculation pass. Count the blocks to know how many times the pipeline looped.
3. The final `Datapath actions:` is the consolidated kernel instruction. It is the actual command sent down to the datapath. Use this line to compare with `ovs-appctl dpif/dump-flows` and confirm that the datapath cache is installed correctly.

Tunnel decapsulation behaves similarly: `tnl_pop` after a VXLAN or Geneve receive creates a recirculation once the outer header is stripped. A trace of VXLAN traffic from an outer network shows two blocks: (1) the tunnel lookup matching on `tun_id` and decapsulating; (2) the inner-packet pipeline using the inner header.

## 9.25.9 Comparison with OVN: debugging the logical flow

OVN has two flow layers: the **logical flow** (in `Southbound_DB.Logical_Flow` in OVSDB) and the **OpenFlow flow** (on `br-int` of each chassis). `ovs-appctl ofproto/trace br-int` only traces the OpenFlow layer. To trace the logical layer, OVN provides its own command:

```bash
ovn-trace --detailed "inport == \"lsp-vm1\" && ip4.src == 10.0.0.10 && ip4.dst == 10.0.0.1"
```

`ovn-trace` simulates the packet through the logical pipeline (ingress ACL, load balancer, routing, egress ACL) without descending to OpenFlow. The result shows each logical stage and the corresponding action.

When debugging an OVN incident, the two-step workflow is:
1. Run `ovn-trace` to identify which logical flow drives the decision.
2. Run `ovs-appctl ofproto/trace br-int` to confirm that OpenFlow executes the logical flow correctly.

If the logical trace is correct but the OpenFlow trace is wrong, the bug is in `ovn-controller`, which compiles Logical Flows into OpenFlow. If both traces are correct but real traffic still misbehaves, the bug is in the kernel datapath; reach for `ovs-dpctl dump-flows` and `dpif/show`.

## 9.25.10 Core takeaways

**One.** Reading 2,000 lines of `ovs-ofctl dump-flows` is a last resort. `ofproto/trace` returns the exact answer in seconds. Always trace before you dump.

**Two.** `ofproto/trace` runs over `ovs-appctl`. It is not part of the OpenFlow specification. It simulates the packet through the pipeline without producing any side effect on the datapath, so it is safe in production.

**Three.** The result of `ofproto/trace` has four blocks: `Flow:` (the flow-spec with defaults filled in), `bridge(...)` (the lookup chain across the tables), `Final flow:` (the flow after every field modification), and `Datapath actions:` (the linear action sequence for the kernel).

**Four.** Three distinct dump-flow commands: `ovs-ofctl dump-flows` (userspace OpenFlow), `ovs-dpctl dump-flows` (kernel megaflow cache), and `ovs-appctl bridge/dump-flows` (including hidden in-band and fail-open flows). Choose the right command for the layer you are debugging.

**Five.** `dpif/show` exposes datapath health: `lookups hit/missed/lost`, the number of `flows`, and average `masks hit/pkt`. A high `missed` count or `masks hit/pkt` greater than 40 is a sign that the flow table is fragmented or the cache is too small.

**Six.** Production flow-table hygiene: `ovs-ofctl monitor watch:` for real-time observation, `diff-flows` for comparison before application (mandatory in production), and `replace-flows` for atomic table replacement. Never run `replace-flows` without running `diff-flows` first.

**Seven.** OVN debugging has two layers: `ovn-trace` for the logical flow and `ovs-appctl ofproto/trace br-int` for OpenFlow. If the logical trace is correct but the OpenFlow trace is wrong, the bug is in `ovn-controller`'s compilation. If both are correct but traffic still misbehaves, the bug is in the datapath; reach for `ovs-dpctl dump-flows`.

**Eight.** Patch ports between bridges do not auto-chain the trace. `ofproto/trace` stops at `output:<patch_ofport>`. A multi-bridge gateway topology requires several consecutive trace runs, one per bridge, with `in_port` set to the peer patch port (Guided Exercise 2).

**Nine.** Registers `reg0..reg15` (32 bits) and `metadata` (64 bits) carry state across a multi-table pipeline. Read the `Final flow:` line to see the accumulated final state. Follow the chain of `load:` and `set_field:` actions across `bridge(...)` blocks to know which table set each piece of state (Guided Exercise 3).

**Ten.** Recirculation, both `ct()` and tunnel decapsulation, splits the trace into several consecutive `bridge(...)` blocks. Each block is one pass through the pipeline. Always pass `--ct-next "new,trk"` or `"est,trk"` to simulate the correct conntrack state. Without the flag, the trace assumes `ct_state=0` and misses every stateful rule (Guided Exercise 4).

## 9.25.X OVS observability core, supplementary 2026-04

> Seven observability keywords: dpctl/dump-flows, ofproto/trace, coverage/show, packet_in handler, OVSDB transaction model, JSON-RPC monitor, ovsdb-server roles. Verified against upstream OVS v2.17.9 source via WebFetch.

### Importance

Tier 2, medium. A second-tier engineer needs every one of these tools during a debug session.

### Version differences

| Keyword | Version evolution |
|---------|-------------------|
| dpctl/dump-flows | OVS 1.x base, OVS 2.4 enhanced output formatting |
| ofproto/trace | OVS 1.x base, OVS 2.6 added the `--detailed` flag, OVS 2.10 expanded microflow expressions |
| coverage/show | OVS 1.x, stable counter exposure |
| packet_in handler | OpenFlow 1.0 base; OVS 2.0 added multi-threaded handlers |
| OVSDB transaction model | RFC 7047 stable |
| JSON-RPC monitor | RFC 7047, plus `monitor_cond` from OVS 2.7 |
| ovsdb-server roles | OVS 2.9 introduced cluster mode (Raft) |

### Source code reference

**`ofproto/trace` stack:**
- `ofproto/ofproto-dpif-trace.c`, the main trace implementation. Verified key functions:
  - `ofproto_dpif_trace_init(void)`: registers unixctl commands at startup.
  - `ofproto_unixctl_trace(struct unixctl_conn *conn, int argc, const char *argv[], void *aux)`: handles the `ofproto/trace` command and parses flow and packet arguments.
  - `ofproto_unixctl_trace_actions(...)`: handles the `ofproto/trace-packet-out` action variant.
  - `ofproto_trace(struct ofproto_dpif *ofproto, const struct flow *flow, const struct dp_packet *packet, const struct ofpact ofpacts[], size_t ofpacts_len, struct ovs_list *next_ct_states, struct ds *output)`: the core simulation function.
  - `parse_flow_and_packet()`: parses command arguments into a flow and packet.
  - `oftrace_report()`: creates trace event nodes for output formatting.

**`dpctl` stack:**
- `lib/dpctl.c`, implements `dpctl/dump-flows` and `dpctl/dump-conntrack`.
- Backend dispatch via `dpif_class` (`dpif-netlink` for the kernel, `dpif-netdev` for userspace).

**Coverage:**
- `lib/coverage.c`, registers counters and exposes them via `coverage/show`.

### Real-world incident references

- `ofproto/trace` is the standard debug-session entry point. About 95 percent of OVS troubleshooting begins with a trace (cited as the methodology of all of Part 9.25).
- A spike in a `coverage/show` counter signals the revalidator-storm pattern (cited in the Part 9.26 forensic).
- `monitor_cond` reduces SBDB load by an order of magnitude (cited in Part 13.7 on `ovn-controller` IDL conditionals).

### Synthetic exercise

```
$ ovs-appctl ofproto/trace br0 'in_port=1,tcp,nw_src=10.0.0.5,nw_dst=10.0.0.10,tp_dst=80' --detailed
$ ovs-appctl dpctl/dump-flows | head
$ ovs-appctl coverage/show
$ ovsdb-client monitor unix:/var/run/openvswitch/db.sock Open_vSwitch
$ ovs-appctl -t ovsdb-server cluster/status OVN_Northbound
```

### Common failures and diagnostic signals

- `ofproto/trace`: stack underflow (cited in Part 4.9.31, push and pop mismatch).
- `dpctl/dump-flows`: stale entries after a topology change (cross-link to Part 9.32 §upcall).
- `coverage/show`: 32-bit counter wrap-around (rare; OVS widened counters in 2.10 and later).

### Comparison with other systems

- `dpctl/dump-flows` is similar to Linux `conntrack -L` plus `tc -s flow`.
- `ofproto/trace` is similar to Linux `iptables --trace` plus Cisco `debug ip packet`.
- `coverage/show` is similar to the counter exposure in Linux `/proc/net/snmp`.
- The `packet_in` handler is similar to a Linux netfilter `NF_QUEUE` handler.
- The OVSDB transaction model is similar to a SQL ACID transaction.
- The JSON-RPC monitor is similar to NETCONF subscribe notifications (RFC 5277).
- The `ovsdb-server` cluster roles are similar to etcd leader and follower (Raft consensus).

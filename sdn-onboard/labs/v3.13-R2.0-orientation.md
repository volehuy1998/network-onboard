## Why R2.0 exists, and what the reader is bringing into it

### Why R2.0 exists

The reader of this lab is a traditional computer-network engineer. The reader has spent years operating Cisco IOS or Junos boxes, has built campus VLANs and trunks and STP topologies in their sleep, and has a complete mental model of how a physical switch turns frames around between ports. That mental model does not map cleanly onto Open vSwitch. The reader's instinct is to treat `br0` as a chassis and a port as a faceplate jack; that instinct is half right and half wrong, and the half that is wrong will cause every later lab to feel like guesswork instead of engineering.

R2.0 exists to close that gap. The single goal of this sub-capture is for the reader to finish able to draw, on paper, the four pieces of code and state that constitute Open vSwitch on a Linux host, to point at each piece by name, to say what each piece persists across daemon restart and what each piece does not, and to predict the output of `ovs-vsctl show`, `ovs-ofctl dump-flows`, and `ovs-appctl dpif/show` for any bridge they have configured. R2.0 does not configure VLANs, does not connect ports, does not send a single frame; that is what R2.1 through R2.8 do. R2.0 is the orientation lap.

The carry-forward state is the one R1.C left behind. Open vSwitch userspace is installed under `/usr/local/`; the kernel module `openvswitch.ko` is loaded and pinned by the dependent modules `nsh`, `nf_conncount`, `nf_nat`, and `nf_conntrack`; `ovsdb-server` and `ovs-vswitchd` are running but the database has just been recreated empty. There are no bridges, no ports, no flows. The lab begins from this clean slate and ends with the database back in the same clean slate, so the reader can repeat R2.0 idempotently against the same host.

### What "Open vSwitch" actually is, in Cisco-trained vocabulary

A reader who has only ever configured a Cisco Catalyst switch carries a single physical mental model: a chassis with a backplane, a forwarding ASIC, a set of physical ports on the front, and an IOS process running on a control-plane CPU that programs the ASIC. The whole switch is one box. The ports are what you can see. The forwarding table lives in the ASIC, not in software. When you type `interface gi0/1` you are telling the IOS process to update the configuration of one of the physical ports.

Open vSwitch is not one box. Open vSwitch is four cooperating pieces of software that together impersonate the function of a switch. Read the four pieces in this order, because the order matches the order in which a packet uses them.

- **The kernel datapath** is `openvswitch.ko`, the kernel module the reader sees in `lsmod`. It owns the fast path. When a frame arrives on a Linux network interface that has been "given to OVS" (by which we mean the interface has been added to a bridge with `ovs-vsctl add-port`), the kernel hooks intercept the frame before the Linux IP stack sees it and hand it to the datapath. The datapath looks up a flow in the per-bridge kernel flow cache. If it finds one it executes the cached action and is done. If it does not find one it punts the frame upward via netlink. The datapath is the closest analogue Open vSwitch has to a Cisco ASIC: it is the part that touches every frame at line rate.
- **`ovs-vswitchd`** is the userspace daemon that owns the slow path and the policy. When the kernel datapath punts a frame, `ovs-vswitchd` receives the punt (this is called an "upcall"), translates the slow-path policy into an action, replies to the kernel with both the action for this frame and a cache entry the kernel should install so the next equivalent frame stays in the fast path. `ovs-vswitchd` is the closest analogue to a Cisco IOS forwarding-engine process: it is the part that decides what the ASIC should do.
- **`ovsdb-server`** is the database server. Every configuration change the operator makes (every `ovs-vsctl add-br`, every `ovs-vsctl set Port br0 tag=10`) is a write into a JSON-RPC database that lives in `/usr/local/etc/openvswitch/conf.db`. `ovs-vswitchd` is a client of that database: it watches the database for changes through an Interface Definition Language (IDL) handle, and reconfigures itself when the database changes. This split is unfamiliar to a Cisco engineer because IOS does not have a separate database server; the running configuration and the box that uses the running configuration live inside the same process. In Open vSwitch they are deliberately separate processes, on different sockets, with restart-independent lifetimes.
- **The OpenFlow controller**, when one is configured, is the fourth piece. Most labs in this curriculum do not configure a controller; the bridge runs in `fail_mode=standalone` (the default) and synthesises an implicit `NORMAL` flow at runtime that learns and forwards exactly the way a Cisco access switch does. R2.1 will exercise the controller-attached and controller-disconnected behaviours in detail. R2.0 only needs the reader to know that the slot for the controller exists.

A bug or a misconfiguration can live in any of those layers; the next eight sub-captures of R2 each isolate one layer at a time so the reader can build a separate mental model per layer instead of a single conflated one.

## A second analogy worth carrying through R2

### Why the four-piece architecture is *more* understandable, not less, once it clicks

A reader who has spent ten years operating Cisco IOS or Junos may at this point feel that Open vSwitch is needlessly complicated: where IOS has one running configuration and one box, OVS has a database, a daemon, a kernel module, an optional controller, plus three different vocabularies (UUIDs, OF ports, DP ports, ifindex) for what looks like the same thing. That feeling is honest, but the architecture is not actually more complicated. It is the *same* complexity, made *visible* instead of *hidden*. Every commercial switch in the reader's career also had a configuration store separate from the forwarding ASIC, an internal RPC bus between the supervisor and the line card, an optional SNMP manager, and at least three internal port identifiers (the chassis slot, the ASIC port number, the protocol-internal ifIndex). On a Catalyst the operator does not see those layers because IOS encapsulates them; on Open vSwitch the operator sees them because each layer is a separate Unix process that can be inspected with the right tool. The visibility is what makes OVS debuggable to the byte level (R9 leans heavily on this property); it is also what makes the first hour of OVS feel intimidating.

The most useful working analogy for R2 onwards is the following. Imagine that on a Cisco Catalyst the operator could:

1. Stop IOS on the supervisor without taking the line cards offline. The line cards continue to forward at line rate against the configuration that was last pushed to them; the operator just cannot reconfigure during the outage. This is the Open vSwitch equivalent of `ovs-vswitchd` restart while the kernel datapath keeps a forwarding-cache hit-rate of 100 percent against the rules already installed.
2. Open the running configuration in a separate database engine and run SQL-style queries against it without going through the IOS CLI. This is `ovs-vsctl` reading and writing OVSDB rows.
3. Attach a remote controller that programs the hardware ASIC directly, bypassing the IOS feature library. This is OpenFlow; the controller speaks to `ovs-vswitchd`, which translates the OpenFlow program into kernel-datapath flow-cache entries.

None of those three abilities exists on a stock Catalyst. All three exist on Open vSwitch by design. Each of the three has a corresponding piece in the four-piece architecture, and each piece has a separate Unix process the operator can inspect. The price of that transparency is the layered identifier system the previous section described; the benefit is a level of operator debuggability and policy-injection flexibility a closed appliance cannot match.

The reader who finishes R2 should feel that the four-piece architecture is *fewer* moving parts than a Catalyst, not more, because each piece is single-responsibility and each piece is inspectable from outside. R2.0 is the introduction; R2.1 through R2.8 are the eight sub-captures that prove the claim by exercising one piece at a time.

### Confirming the daemon pair is up

A traditional-network reader is used to having "the switch is up" be a single yes-or-no fact. On Open vSwitch "the switch is up" decomposes into "ovsdb-server is up" and "ovs-vswitchd is up". Both must be running for the bridge to forward; if `ovs-vswitchd` is up but `ovsdb-server` is down the operator sees the database connection refusing because `ovs-vsctl` is itself a database client. If `ovsdb-server` is up but `ovs-vswitchd` is down the operator sees configuration commands succeeding but the bridge not actually moving any frames.

```text
root@lab-openvswitch:~/network-onboard# pgrep -a ovsdb-server
876647 ovsdb-server /usr/local/etc/openvswitch/conf.db -vconsole:emer -vsyslog:err -vfile:info --remote=punix:/usr/local/var/run/openvswitch/db.sock --private-key=db:Open_vSwitch,SSL,private_key --certificate=db:Open_vSwitch,SSL,certificate --bootstrap-ca-cert=db:Open_vSwitch,SSL,ca_cert --no-chdir --log-file=/usr/local/var/log/openvswitch/ovsdb-server.log --pidfile=/usr/local/var/run/openvswitch/ovsdb-server.pid --detach --monitor
root@lab-openvswitch:~/network-onboard# pgrep -a ovs-vswitchd
876666 ovs-vswitchd unix:/usr/local/var/run/openvswitch/db.sock -vconsole:emer -vsyslog:err -vfile:info --mlockall --no-chdir --log-file=/usr/local/var/log/openvswitch/ovs-vswitchd.log --pidfile=/usr/local/var/run/openvswitch/ovs-vswitchd.pid --detach --monitor
```

Both daemons are running. PID `876647` is `ovsdb-server`, PID `876666` is `ovs-vswitchd`; the PIDs are 19 apart, which matches the start order `ovs-ctl start` enforces (the database server first, the switch daemon second, with the switch daemon waiting on the database socket before it begins). The `ovsdb-server` command line is worth reading slowly because it is the most informative one-line summary of what the daemon is doing:

The argument `/usr/local/etc/openvswitch/conf.db` is the path to the on-disk database file. The flag `--remote=punix:/usr/local/var/run/openvswitch/db.sock` tells the daemon to listen on a Unix-domain socket at that path; every `ovs-vsctl` command on this host will connect to that socket. The flags `--private-key`, `--certificate`, `--bootstrap-ca-cert` set up TLS material for any future remote OVSDB client (none in R2.0). The flag `--detach` means the daemon backgrounded itself after startup; `--monitor` means there is a parent watchdog process that will respawn it if it dies. The flag `--log-file=/usr/local/var/log/openvswitch/ovsdb-server.log` is where every log line goes, and `-vfile:info` is the verbosity policy for that file (info-and-above only).

The `ovs-vswitchd` line tells the same story. The first positional argument `unix:/usr/local/var/run/openvswitch/db.sock` is the socket the switch daemon connects to as an OVSDB client. The flag `--mlockall` locks all of the daemon's pages into RAM so a swap event cannot stall packet processing through a page fault; this is a production-deployment knob that the upstream init script enables by default. The rest are the same logging and supervision flags as `ovsdb-server`.

The Cisco analogue here is a stretch but useful: imagine that on a Catalyst the running configuration lived in a separate `confd` process, and the forwarding-engine process talked to `confd` over a Unix socket to learn what the operator had asked for. That is exactly how Open vSwitch is split. The split exists because Open vSwitch was designed for environments where one operator program (Nova-compute, OpenStack Neutron, a hand-written cloud-management system) wants to make many small configuration changes per second through a transactional API; OVSDB gives that transactional API for free, and `ovs-vswitchd` becomes the consumer of the transactions instead of the owner of them.

The split also explains why the upstream packaging ships two systemd units (`ovsdb-server.service` and `ovs-vswitchd.service`) rather than one. A reader who has only ever managed monolithic daemons may instinctively reach for `systemctl restart openvswitch-switch.service` to "restart OVS"; on this lab host that umbrella unit is the older Ubuntu name and is not active because R1.B and R1.C built the upstream `2.17.9` userspace into `/usr/local/`, leaving the Debian-packaged service sockets idle. The upstream-packaged equivalent is the helper script `/usr/local/share/openvswitch/scripts/ovs-ctl` (R1.C used it), which orchestrates the start ordering: bring `ovsdb-server` up, wait for the `db.sock` socket to exist, then bring `ovs-vswitchd` up. The reader who debugs an OVS startup problem in R9 will frequently need to reproduce that ordering by hand.

### The empty database

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl show
15b3fd6f-e30f-4ae3-b8ca-4531189afee5
    ovs_version: "2.17.9"
root@lab-openvswitch:~/network-onboard# ovs-vsctl list-br
```

`ovs-vsctl show` is the command that pretty-prints the database's `Open_vSwitch` row plus every `Bridge` row reachable from it. The output is two lines. The first line is the UUID of the singleton `Open_vSwitch` row, `15b3fd6f-e30f-4ae3-b8ca-4531189afee5`. That UUID was allocated when the database file was created; it persists across daemon restart, across host reboot, across `ovsdb-server` upgrades, and is the identity of this Open vSwitch installation. The second line, `ovs_version: "2.17.9"`, is the userspace version `ovs-vswitchd` reported into the database when it last reconfigured itself. The absence of any further indented lines means there are no bridges; `ovs-vsctl list-br` prints zero output for the same reason. This is the clean slate that R2.0 begins from and ends in.

The pedagogical point here, which is invisible if the reader skims past, is that the singleton row UUID is the only piece of database state that survives the empty configuration. The `Open_vSwitch` table is `maxRows=1, isRoot=true` per the schema; one row exists from creation to deletion of the database file. Every other table is empty when no bridges are configured, and rows in those tables are garbage-collected the moment they are no longer referenced from the singleton. R2.7 will exercise the garbage-collection rule directly.

### What the schema advertises

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl get Open_vSwitch . ovs_version
"2.17.9"
root@lab-openvswitch:~/network-onboard# ovs-vsctl get Open_vSwitch . db_version
"8.3.0"
root@lab-openvswitch:~/network-onboard# ovs-vsctl get Open_vSwitch . datapath_types
[netdev, system]
root@lab-openvswitch:~/network-onboard# ovs-vsctl get Open_vSwitch . iface_types
[bareudp, erspan, geneve, gre, gtpu, internal, ip6erspan, ip6gre, lisp, patch, stt, system, tap, vxlan]
```

Four facts, each one a building block for a later sub-capture.

The `ovs_version` value `"2.17.9"` is the version of `ovs-vswitchd` that wrote into the database the last time it reconfigured. It is not the version of `ovs-vsctl`, although in this lab the two are equal because R1.C built both from the same source tree. The `db_version` value `"8.3.0"` is the version of the OVSDB schema this database file is using. Every release of Open vSwitch ships a `vswitch.ovsschema` file with a version number; if the schema on disk lags the schema the daemons expect, the operator must run `ovsdb-tool convert` to upgrade. `8.3.0` is the schema version that ships with `2.17.9`; the two version numbers travel independently because the schema only changes when a column is added, removed, or renamed.

The `datapath_types` value `[netdev, system]` is the list of datapath kinds this `ovs-vswitchd` build can drive. `system` is the in-kernel datapath via `openvswitch.ko`. `netdev` is the userspace datapath that runs entirely in `ovs-vswitchd`. R2.6 will create one bridge per datapath kind and demonstrate that the choice is per-bridge, not per-host; here, the reader only needs to remember that the daemon offers two and the default is `system`.

The `iface_types` list is the more interesting one. It is the catalogue of interface flavours the daemon can build. Read each name and recognise that R2.3 will exercise the namespace-and-wire combinations behind them.

- `internal` is the L3-attachable interface that Open vSwitch synthesises for every bridge by default; the `Bridge br0` row in `ovs-vsctl show` always has a `Port br0 / Interface br0` of type `internal`. The bridge's L3 IP, when one is assigned, lives on this interface.
- `system` is the wrapper around an existing Linux network device. The reader uses this implicitly when running `ovs-vsctl add-port br0 ens33`; the resulting `Interface` row's `type` is the empty string and the daemon treats it as `system`.
- `tap` is the wrapper around a `/dev/net/tun` character device, the kind a virtual machine guest reads frames from.
- `patch` is the in-userspace cross-bridge wire that R2.4 exercises in detail.
- `geneve`, `vxlan`, `gre`, `gtpu`, `stt`, `erspan`, `ip6gre`, `ip6erspan`, `lisp`, `bareudp` are the tunnel kinds. R5 exercises the first three in single-host topologies; the others are out of scope for this curriculum.

The Cisco analogue for `iface_types` is a port-type pull-down on the chassis manager: the Catalyst chassis admits SFP+, QSFP, copper-1G, and so on, and each port-type has its own software driver. Open vSwitch is the same idea moved into software: every supported interface flavour has its own implementation under `lib/netdev-*.c` (`netdev-linux.c` for `system`, `netdev-vport.c` for the tunnels, `netdev-tc-offloads.c` for hardware-offload, and so on). The `iface_types` column is the reflective query that asks the daemon "which drivers did you compile in?".

Source anchors. The schema definition lives at `vswitchd/vswitch.ovsschema`. The provider-registration call sites live at `lib/dpif-provider.h` (datapath registration via `dp_register_provider()`) and `lib/netdev-provider.h` (interface-type registration via `netdev_register_provider()`). The string list returned by `iface_types` is built in `bridge_refresh_ofp_port()` at `vswitchd/bridge.c` from the registered providers; the reader can grep `netdev_register_provider` across `lib/netdev-*.c` to see one call per supported interface type. The traditional-network analogue and the source registration mechanism together explain why adding a new tunnel kind to Open vSwitch is a non-trivial code change instead of a configuration option.

## Creating one bridge from scratch

### The single-command create

The simplest configuration change in Open vSwitch is also the most informative: create one bridge, then read every row the create produced, and notice every column that defaulted instead of being set.

The next block shows the command and its silent confirmation.

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl --may-exist add-br br0
root@lab-openvswitch:~/network-onboard# ovs-vsctl list-br
br0
root@lab-openvswitch:~/network-onboard# ovs-vsctl show
15b3fd6f-e30f-4ae3-b8ca-4531189afee5
    Bridge br0
        Port br0
            Interface br0
                type: internal
    ovs_version: "2.17.9"
```

`ovs-vsctl --may-exist add-br br0` produces no output. It also produces no error if `br0` already exists; the `--may-exist` flag is the operationally correct idempotence pattern. R2.7 will exercise that pattern in detail; here, the silent success is the proof that one transaction has been committed against the database.

After the create, `ovs-vsctl list-br` returns one line, `br0`. `ovs-vsctl show` returns a six-line tree: the singleton row UUID, then a `Bridge br0` block, then a `Port br0` block nested inside it, then an `Interface br0` block nested inside that with the line `type: internal`. The nesting in `ovs-vsctl show` is the database's reference graph rendered as text; one bridge contains a list of ports, each port contains a list of interfaces, every interface has a type. The Cisco analogue is the `show interface br0` output on a Catalyst, except that Open vSwitch is exposing three layers of indirection (bridge / port / interface) where IOS exposes only one (interface). The reason the indirection exists is the link-aggregation and bond mechanism: a single port can have multiple interfaces, which is how OVS implements LACP and active-backup bonding (R2.5.4 will exercise this directly).

The diagram below shows the four-pane mental model the reader should now be able to draw on paper, with the row identifiers from this very transcript.

```ascii

   ============================================

       Bridge "br0"

               Interface "br0" (type: internal)
```

The diagram makes the layered architecture concrete. One `ovs-vsctl add-br br0` produced four rows in three OVSDB tables, one `struct ofproto` in `ovs-vswitchd`'s memory, and two `vport` entries in the kernel datapath, all coordinated through the IDL handshake that the bridge reconfigure loop runs every time the database changes.

The brackets-versus-braces distinction matters because OVSDB's type system distinguishes set, map, and scalar; `[]` is "empty set", `{}` is "empty map".

Read each column in groups so the page does not feel like a wall of unread text.

The identity columns. `_uuid` is the row UUID, `e1c96e9a-1eb8-41e0-8d94-729628e82981`. `name` is `br0` and is `mutable=false` per the schema; the operator cannot rename a bridge, only delete and re-add. `datapath_id` is `"00009a6ec9e1e041"`, the 64-bit OpenFlow datapath identifier; this is the value an OpenFlow controller uses to address this switch. The first 16 bits are zero (the OUI portion the OpenFlow spec leaves to the implementer); the remaining 48 bits are the MAC address of the LOCAL port that R2.0 will read in a moment. `datapath_type` is `""`, the empty string; an empty string in this column means "use the default", which is `system` (the kernel datapath). `datapath_version` is `"<unknown>"`, a literal placeholder string; the kernel datapath does not export a meaningful version at this layer.

The optional-feature columns. `mirrors`, `netflow`, `sflow`, `ipfix` are all `[]`; no observation point is attached to this bridge. R2.5.5 (mirroring), R2.5.11 (sFlow / NetFlow / IPFIX), and R7 (the cross-cutting QoS-and-mirror lab) will exercise them. `flow_tables` is `{}`; the operator has not configured any per-table policy.

The control-plane columns. `controller` is `[]` (no OpenFlow controller is configured). `fail_mode` is `[]` (the column is unset; per `vswitchd/bridge.c:3998-4002` an unset `fail_mode` defaults to `OFPROTO_FAIL_STANDALONE`, which is why `dump-flows` will show the implicit-NORMAL rule a few blocks below). `protocols` is `[]` (the operator has not constrained the OpenFlow versions). `auto_attach` is `[]` (no IEEE 802.1Qbg AutoAttach configuration).

The reference-graph column. `ports : [a047c65c-fec2-496d-b186-57802e701d7a]` is the list of `Port` row UUIDs that this `Bridge` row points at. There is exactly one entry, the auto-created `Port br0`. The `Interface` row is the longest of the three because it is where the wire-level facts live. Six lines deserve unpacking.

The line `type : internal` is the type-flavour from the `iface_types` list earlier. An interface of type `internal` is an OVS-synthesised `tun`-style network device that lives both inside `ovs-vswitchd`'s memory and as a real Linux netdev (visible to `ip link show`); R2.3 will make this concrete by creating a non-bridge-LOCAL internal port and exercising it.

The line `ofport : 65534` is the OpenFlow port number. 65534 is `OFPP_LOCAL`, the well-known constant the OpenFlow specification reserves for "the local port of the bridge" (see `include/openflow/openflow-1.0.h:OFPP_LOCAL`). Every other port the operator adds will receive a small positive ofport number (1, 2, 3, ...); only the LOCAL port has the well-known 65534. R2.2 onwards will see ofport 1, 2, ... assigned to the veth bridge ends.

The line `ifindex : 110` is the Linux netdev `ifindex` of the LOCAL port. `ifindex` is the kernel's per-host integer identifier; `ip link show` and `ip route` use the `ifindex` to bind a route to a device. The OVS `ifindex` and the OVS `ofport` are two different identifiers for the same port: `ifindex` is the kernel identifier, `ofport` is the OpenFlow identifier. Confusing the two is a common source of bugs in operator scripts; the `ovs-vsctl --columns=name,type,ofport list Interface ...` form is the canonical way to read the OpenFlow identifier without ambiguity.

The line `mac_in_use : "9a:6e:c9:e1:e0:41"` is the MAC address Linux assigned to the internal netdev. OVS does not generate this MAC; the kernel's tap-like driver does. Note that the last 48 bits match the trailing 48 bits of `datapath_id` in the `Bridge` row (`00009a6ec9e1e041`); this is not a coincidence. The OVS daemon picks the LOCAL port's MAC as the bridge's datapath identifier (with 16 leading zero bits) the first time the bridge starts; the choice is recorded into `Bridge.other_config:hwaddr` so it persists across daemon restart even if the kernel reassigns the MAC.

The lines `admin_state : down`, `link_state : down`, `mtu : 1500` are the dynamic state. The internal port comes up `down` and stays `down` until the operator runs `ip link set br0 up`. R2.0 deliberately does not bring it up because R2.2 onwards will exercise the L2-and-L3 traffic in detail.

The line `statistics : {collisions=0, rx_bytes=0, ...}` is the per-interface counter set; every counter is zero because no frame has crossed this port. R2.5.5 (mirroring) will read these counters directly to validate that mirror traffic counts on the destination port.

A handful of remaining columns are observability-only and the reader can skip them on a first read. The `cfm_*` family is the Connectivity Fault Management protocol IEEE 802.1ag, used in carrier-grade deployments to detect link failure between two service endpoints; OVS supports CFM but this lab does not configure it, so every CFM column is empty. The `bfd` and `bfd_status` columns are the Bidirectional Forwarding Detection protocol RFC 5880, used as a sub-second link-liveness check between two routers; OVS implements BFD per-interface and exposes the protocol-state machine through these two columns. The `lldp` map is the Link Layer Discovery Protocol IEEE 802.1AB, used for neighbour discovery in data-centre fabrics; OVS implements LLDP transmission but this lab does not turn it on. The `ingress_policing_*` four columns are the per-interface rate-limiter knobs (`rate`, `burst`, `kpkts_rate`, `kpkts_burst`); R2.5.9 exercises them as a port-level policer alongside the bridge-level HTB queue tree. The `mtu_request` column is the operator's per-interface MTU override; if set, OVS reconfigures the kernel interface to the requested MTU. The `error` column is the diagnostic string OVS writes when an interface fails to come up (for example, `could not open network device ens33 (No such device)`); a non-empty `error` column on a misbehaving port is the single most useful debug datum in R9.

The `status` column `{driver_name=openvswitch}` is a per-interface key-value map that providers populate with type-specific facts. For an `internal` interface the driver is the kernel `openvswitch` module that synthesises the netdev. For a `system` interface attached to a real NIC the column would also report `driver_name=e1000` (or `vmxnet3`, `ixgbe`, ...) and `driver_version`. R2.6 reads this column on a `netdev`-datapath bridge and observes a different driver name reported there.

## What the implicit-NORMAL flow looks like and why it is there

```text
root@lab-openvswitch:~/network-onboard# ovs-ofctl dump-flows br0
 cookie=0x0, duration=12.070s, table=0, n_packets=0, n_bytes=0, priority=0 actions=NORMAL
```

A traditional-network reader looks at this and reads "there is a flow rule that says, at priority 0, send everything through the NORMAL forwarding logic." The reader is right about the effect and wrong about the implementation. The line above is the surface; the implementation is more interesting and more pedagogically useful.

> **Misconception / Reality.**
>
> Misconception: "the priority=0 NORMAL rule is a default flow rule that OVS stores in the OpenFlow flow table at bridge create time, the same way a Cisco IOS image initialises the default ACL."
>
> Reality: there is no stored flow rule. The line above is the *output* of `ovs-ofctl dump-flows`, which prints whatever the bridge's OpenFlow flow table currently contains. The bridge's flow table actually contains zero stored rules. The line is a synthesised view that the bridge's connection-manager renders at dump time, because the bridge is in `OFPROTO_FAIL_STANDALONE` mode and the `connmgr` module synthesises an implicit catch-all that outputs to `OFPP_NORMAL` whenever no OpenFlow controller is connected and the operator has not explicitly set `fail_mode=secure`. If the reader sets `ovs-vsctl set Bridge br0 fail_mode=secure` and re-runs `dump-flows`, the line disappears even though no `ovs-ofctl del-flows` was issued. R2.1 exercises that disappearance directly.

The interesting part. The implicit synthesis means a Cisco-trained reader cannot reason about OVS by inspecting `dump-flows` alone; the absence of a rule does not mean "no forwarding" and the presence of the priority=0 NORMAL line does not mean "stored configuration". The right mental model is "`dump-flows` shows the union of stored OpenFlow rules and the implicit synthesis the connection manager renders for the current `fail_mode`". The duration of `12.070s` in the line above is the time elapsed since the bridge was created; the implicit synthesis is freshly-rendered every time `dump-flows` runs.

Source anchors. The synthesis happens in `ofproto/connmgr.c` and `ofproto/fail-open.c`. Specifically, the `fail_open_add_normal_flow()` function in `ofproto/fail-open.c` is the one that installs the `priority=FAIL_OPEN_PRIORITY actions=output:OFPP_NORMAL` rule, and it is called from `fail_open_flushed()` whenever the controller-disconnect timer expires. For the standalone-default case (no controller ever configured) the mechanism is slightly different: `vswitchd/bridge.c:3998-4002` selects `OFPROTO_FAIL_STANDALONE` from an unset `fail_mode` column, and the standalone path is the one that produces the catch-all `dump-flows` view. R2.1 is the lab that tells these two cases apart with verbatim host output; here in R2.0 the reader only needs to know that the line exists and to reject the misconception that it is a stored rule.


`ovs-ofctl show br0` is a different lens onto the same bridge. The output is an OpenFlow `OFPT_FEATURES_REPLY` message rendered as text; this is what the bridge would send back if a controller had just connected and asked "what are your features". Three lines are worth dwelling on.

The line `dpid:00009a6ec9e1e041` is the same `datapath_id` the `Bridge` row reported. The `dpid` is the OpenFlow protocol's name for the same field; the OVSDB column is `datapath_id`, the protocol field is `dpid`, the value is identical.

The line `n_tables:254` is the number of OpenFlow tables the bridge advertises. 254 is the `OFPP_MAX` table count for OF 1.3 onwards; OVS advertises the maximum even though the operator has populated zero tables, because reserved tables 250 to 253 are used internally by the daemon's instruction-set fallbacks.

The line `actions: output enqueue set_vlan_vid set_vlan_pcp strip_vlan mod_dl_src mod_dl_dst mod_nw_src mod_nw_dst mod_nw_tos mod_tp_src mod_tp_dst` is the OF 1.0 action list. Three commands, three different panes of the kernel-side state.

`ovs-appctl dpif/show` is the OVS-specific wrapper that asks `ovs-vswitchd` to print the per-bridge datapath summary. The line `system@ovs-system: hit:0 missed:0` says the active datapath is named `ovs-system` and is of type `system` (the kernel datapath). The sub-line `br0:` is the bridge name. The sub-sub-line `br0 65534/1: (internal)` is the per-port mapping: the LOCAL port is OF port 65534, datapath port 1, type internal. R2.2 onwards uses this view to validate that adding a veth bridge end produced both an OF port and a DP port.

`ovs-appctl ofproto/list` is the per-bridge listing in `ovs-vswitchd`'s memory. There is one bridge, `br0`, because there is one `struct ofproto` allocated in the daemon's heap for this bridge. The Cisco analogue is `show running-config | include ^interface`; here it is the daemon's authoritative list of bridges it is forwarding for.

`ovs-dpctl show` is the tool that talks to the kernel module directly, bypassing `ovs-vswitchd`. `ovs-dpctl` uses the `OVS_DP_CMD_GET` netlink message to ask the kernel about every datapath the kernel knows about. The kernel reports one datapath, `ovs-system`. The line `flows: 0` is the count of flow-cache entries in the kernel; zero because no frame has been forwarded yet. The line `caches: masks-cache: size:256` is the megaflow mask cache, sized at 256 entries by default (R4 will exercise this directly). The two lines `port 0: ovs-system (internal)` and `port 1: br0 (internal)` are the kernel-side view of the same DP-port mapping `dpif/show` already showed; port 0 is the datapath-itself port (every kernel datapath has one) and port 1 is the bridge LOCAL.

The pedagogical point. The reader has just seen the same bridge described in three different vocabularies: the OVSDB row UUID vocabulary, the `ovs-vswitchd` `ofproto` vocabulary, and the kernel `vport` vocabulary. All three describe the same object; all three exist in different processes; all three keep their own copy of the state. When a bridge "is broken" in production, the question the operator must answer first is "broken in which of the three vocabularies?", because the answer routes them to a different set of debug commands. R9 (the debug-and-troubleshoot doctrine) returns to this triangulation in detail.

## A short detour into the three identifier systems

### Why `ifindex`, `ofport`, and `_uuid` are not interchangeable

The R2.0 transcript exposed three different integer-or-string identifiers for the same LOCAL port: the OVSDB row UUID `8e2e47ab-6163-42b6-aa7f-f4133cb4b988`, the OpenFlow port number `65534`, and the Linux kernel `ifindex` `110`. They all name the same thing and the reader's first instinct is to treat them as aliases. They are not aliases. They live in three different systems, they are allocated by three different allocators, and they have three different lifetimes.

The OVSDB row UUID is allocated by `ovsdb-server` when the row is inserted. UUIDs are 128-bit random values; the chance of collision is statistically zero. The UUID persists for the lifetime of the row and is recorded on disk in `conf.db`. Across daemon restart the UUID is unchanged because the database file is on disk. The UUID is *not* recycled when the row is deleted; a new `Interface` row inserted for the same port name receives a fresh UUID. The reader uses the UUID as the row identity for transactional updates, JSON-RPC client integrations, and observability across daemon restarts.

The OpenFlow port number is allocated by `ovs-vswitchd` when the port is added to a bridge. The number is small (1, 2, 3, ...) plus the well-known reserved values (`OFPP_LOCAL=65534`, `OFPP_CONTROLLER=65533`, `OFPP_NORMAL=65532`, and so on). The number lives in `ovs-vswitchd`'s memory and in the database column `Interface.ofport` (where the daemon writes it for the operator's convenience). Across daemon restart the number is preserved if the operator pinned it via `Interface.ofport_request`; otherwise the daemon may pick a different number for the same port name. The number is recycled within a bridge but not across bridges. The reader uses `ofport` as the identity in OpenFlow flow rules (`actions=output:1`), in `dpif/show` (`OF_port/DP_port`), and in `ofproto/trace` input specifications.

The Linux `ifindex` is allocated by the kernel network-namespace module when the netdev is registered. The number is per-namespace, monotonically increasing, large (`110` here means 109 netdevs were registered before this one in the host's lifetime, including the kernel's loopback at `ifindex=1`). The number does not survive netdev deletion-and-recreation cycles because the kernel does not recycle. Across host reboot the number resets. The reader uses `ifindex` as the identity in `ip route`, `tc qdisc`, `ethtool`, and any tool that bypasses OVS to talk to the kernel directly.

The three identifiers can drift apart in observable ways. If the operator runs `ovs-vsctl set Interface br0 ofport_request=10`, the OpenFlow port number changes from `65534` to a non-LOCAL value (the daemon will refuse, because LOCAL is special, but for a non-LOCAL port the change is straightforward); the OVSDB UUID does not change; the kernel `ifindex` does not change. If the operator runs `ip link set dev br0 down; ip link delete br0; ovs-vsctl --if-exists del-br br0`, the three identifiers are torn down in different orders by different processes, and a misordered teardown is the single most common cause of "phantom port" bugs in production. R2.7 walks the round-trip explicitly with all three identifiers visible.

The Cisco analogue is the Catalyst's `interface vlan 1` SVI: the IOS `interface vlan 1` configuration block has a name (`Vlan1`), an `if_index` (visible in SNMP `ifIndex`), and an internal port handle the supervisor card uses. The Cisco operator never sees those three identifiers separately because IOS hides them. The OVS operator sees them all because OVS decomposes the abstraction. The decomposition is not extra work to memorise; it is the reflection of layers that already exist on the Catalyst, made visible.

### A practical command pattern: read the three identifiers in one block

A useful operator habit when debugging an OVS issue is to read the three identifiers for the same port in one command block, so the relationship is concrete on the screen. The pattern, applied to the LOCAL port the lab created:

The OVSDB UUID is read with `ovs-vsctl get Interface br0 _uuid`; the value is `8e2e47ab-6163-42b6-aa7f-f4133cb4b988`. The OpenFlow port number is read with `ovs-vsctl get Interface br0 ofport`; the value is `65534`. The kernel `ifindex` is the third line of `ip link show br0` output (`110: br0:`); the value is `110`. The MAC is the bottom line of the same output (`link/ether 9a:6e:c9:e1:e0:41`); the value is `9a:6e:c9:e1:e0:41`. Together those four facts uniquely identify the port across every system that talks about it.

The transcript above exercised exactly this pattern; the reader can revisit the `ovs-vsctl get Interface br0 _uuid`, `ovs-vsctl get Interface br0 ofport`, `ovs-vsctl get Interface br0 mac_in_use`, and `ip link show br0` blocks and confirm the four values match the values reproduced in this paragraph. R2.8 generalises the pattern to a multi-port topology and uses it as the basis for a binding proof.

## Tearing down and confirming the green slate

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl --if-exists del-br br0
root@lab-openvswitch:~/network-onboard# ovs-vsctl list-br
root@lab-openvswitch:~/network-onboard# date -u
Thu Apr 30 04:19:05 AM UTC 2026
```

`ovs-vsctl --if-exists del-br br0` is the symmetric idempotent deletion. It silently succeeds whether the bridge exists or not; the `--if-exists` is the deletion-time twin of `--may-exist`. After the deletion, `ovs-vsctl list-br` returns zero output, the same view the lab opened with. `date -u` stamps the wall clock at the end of the lab; the elapsed time from the opening `date -u` (`04:18:42`) to the closing one (`04:19:05`) is 23 seconds.

The deletion did three things atomically. First, it removed the `Bridge` row from the database; the singleton `Open_vSwitch.bridges` set lost one UUID. Second, `ovs-vswitchd` noticed the change through the IDL handle and tore down the corresponding `struct ofproto` from its heap; the LOCAL port netdev was destroyed, the OpenFlow connection-manager was freed, the per-bridge fail-open module was freed. Third, `ovs-vswitchd` issued an `OVS_DP_CMD_DEL` netlink to the kernel; the kernel removed the `vport` for `br0` (DP port 1) from the `ovs-system` datapath, but it did not destroy the datapath itself because the datapath is shared host-wide and is reused by the next bridge the operator adds. The reader who runs R2.0 a second time will see the datapath persist (`hit:0 missed:0` lines stay) while the per-bridge state cycles through.

The persistence asymmetry between the per-bridge state (lifecycle = bridge create to bridge delete) and the datapath state (lifecycle = first bridge to last bridge) is the single most surprising fact in the OVS architecture for a Cisco-trained reader, because it means `ovs-vswitchd` restarts do not flush the kernel datapath; the daemon reattaches to a still-populated kernel datapath at startup. R9 returns to this in the context of `ovs-vswitchd` crash recovery.

## A frame's life through the four-piece architecture, in advance of R2.2

### The path R2.0 has not taken, and why naming it now matters

R2.0 created one bridge, read every row, and tore the bridge down without sending a single frame across it. R2.2 will add namespaces, veth pairs, and a `ping`. Before R2.2 the reader benefits from a forward-looking sketch of what happens to the very first frame the bridge will see, because the sketch names the components R2.0 has just inspected and lets the reader anchor each component to a concrete role in the data path. The sketch is approximate; the precise mechanism is the subject of R4 (datapath internals).

When the first ICMP echo request leaves namespace `ns1` through its veth, the kernel's network stack hands the frame to the OVS ingress hook on the bridge-side veth peer. The hook lives inside the `openvswitch` kernel module. The module computes a hash key from the frame's headers, looks up the kernel flow cache (`flow_table`) for a matching megaflow entry, and finds none, because no frame has crossed the bridge in this lifetime. The kernel module then constructs an upcall, attaches the frame plus a flow-key digest, and emits the upcall to userspace via netlink genl multicast.

`ovs-vswitchd` is parked in `poll(2)` on its netlink socket. The poll wakes; the daemon dequeues the upcall message; an upcall-handler thread runs `xlate_actions()` from `ofproto/ofproto-dpif-xlate.c` to translate the slow-path policy into a concrete sequence of datapath actions. For a bridge in `OFPROTO_FAIL_STANDALONE` mode with no controller and no stored flows, the translation falls into `xlate_normal()` (also in `ofproto-dpif-xlate.c`), which performs the L2 learning and forwarding logic. The reader will see `xlate_normal()` step by step in R2.2 via `ofproto/trace`. The translation produces two outputs: a list of datapath actions for *this* frame (typically `output:<DP-port>` after the destination MAC is learned, or the `flooding` set if the destination is unknown), and a megaflow entry to install in the kernel flow cache so the next equivalent frame stays in the fast path.

The userspace daemon emits an `OVS_PACKET_CMD_EXECUTE` netlink message containing the actions for this frame and an `OVS_FLOW_CMD_NEW` netlink message containing the megaflow entry. The kernel module executes the actions (transmits the frame on the destination vport) and inserts the megaflow into the cache. The upcall is complete. The next frame in the same conversation finds the megaflow in the cache and never reaches userspace.

The architectural significance of this round trip is that the first frame of every new conversation pays a userspace round-trip cost (typically 50 to 200 microseconds on modern hardware), and every subsequent frame is at line rate. R4 measures the cost directly and exercises the megaflow eviction policy when memory pressure forces older entries out. R2.0 only needs the reader to know the round trip exists, that `xlate_normal()` is the function name to remember, and that the priority-0 NORMAL line in `dump-flows` is rendering of the bridge's standalone-mode policy that drives `xlate_normal()`.

The diagram below ties the components R2.0 has inspected to the steps of the upcall round trip. The reader who has read every prior section should be able to point at each pane of the previous four-pane diagram and locate it in this round-trip view.

```ascii
   Time -->

   t0:  frame leaves ns1 veth peer
        |
        v
   t1:  OVS kernel hook intercepts on bridge-side peer
        flow_table lookup MISS
        |
        v
   t2:  kernel emits upcall via netlink genl
        |   payload: frame + flow_key
        v
   t3:  ovs-vswitchd dequeues upcall
        |   poll(2) wakes
        v
   t4:  xlate_actions() in ofproto-dpif-xlate.c
        bridge fail_mode == OFPROTO_FAIL_STANDALONE
        falls into xlate_normal() for L2 learning + flood
        |
        v
   t5:  produces (a) actions for this frame
                 (b) megaflow entry for the cache
        |
        v
   t6:  netlink OVS_PACKET_CMD_EXECUTE  -> kernel transmits frame
        netlink OVS_FLOW_CMD_NEW        -> kernel inserts megaflow
        |
        v
   t7:  next frame in conversation: cache HIT, no upcall
```

The arc above is the complete forwarding path for the *uncached* case. R2.2 will make the arc concrete by running a `ping` between two namespaces and reading `dpif/dump-flows` immediately after the first packet, so the megaflow that was just installed becomes visible. R4 will add the revalidator (the periodic background scan that ages megaflows out) and the upcall-handler thread pool (which scales the userspace half across cores). For R2.0 the only goal is name-recognition: the reader who proceeds to R2.1 with `xlate_normal`, `OVS_PACKET_CMD_EXECUTE`, and `OVS_FLOW_CMD_NEW` already in their working vocabulary will not have to pause R2.2 to look those names up.

## Three patterns the reader will use in every following sub-capture

### Pattern A: read the database, the daemon, and the kernel for the same object

R2.0 demonstrated the pattern on a single bridge. Generalised: for any OVS object the operator suspects of misbehaving, read the same object from all three layers and compare. From the database with `ovs-vsctl list <Table> <name>`. From the daemon with `ovs-appctl ofproto/<command>` or `ovs-appctl dpif/show`. From the kernel with `ovs-dpctl <command>` or `ip link show`. Disagreement between any two layers is a bug; the layer that disagrees is the layer to investigate. The pattern is mechanical and produces high-quality diagnostic output even when the operator does not yet know what is wrong.

### Pattern B: separate the question "is the configuration applied" from the question "is the configuration correct"

The Cisco IOS culture conflates the two questions because IOS gives only one CLI surface. On OVS the questions are separate. "Is the configuration applied" is answered by the database read: did `ovs-vsctl` succeed in writing the row? "Is the configuration correct" is answered by the daemon and kernel reads: did the daemon notice the database change, and did the daemon push the change to the kernel? In R2.0 the configuration was the single `Bridge` row; the configuration was applied (database read returned the row) and the configuration was correct (daemon's `ofproto/list` reported the bridge, kernel's `ovs-dpctl show` reported the vport). When a future lab fails on the second question with the first question succeeding, the failure is a daemon-side or kernel-side bug; when both fail the failure is in the OVSDB transaction itself.

### Pattern C: when in doubt, read the schema

The OVSDB schema at `vswitchd/vswitch.ovsschema` is the canonical reference for every column the operator can read or write. The schema declares the column type (set, map, scalar, integer with bounds, enum), the mutability flag (whether the operator can change it after creation), the ephemeral flag (whether the value persists across daemon restart), and the reference flag (whether the column points at a row in another table). When `ovs-vsctl` rejects a write with a cryptic message, the schema is the first place to look; the message will reference a column name, and the schema will explain why the column rejected the value (wrong type, immutable after creation, value out of bounds, reference to non-existent row).

A short concrete example: the column `Port.tag` is declared in the schema as `{"type": {"min": 0, "key": {"minInteger": 0, "maxInteger": 4095, "type": "integer"}}}`. The reader should be able to read this and predict that `tag` is an integer in the range 0 to 4095, with `min=0` meaning the column may be empty (an unset `tag` means "no access VLAN configured"; this is what the lab's `Port br0` row shows). If the operator types `ovs-vsctl set Port br0 tag=4096`, OVSDB rejects the write because 4096 exceeds `maxInteger`. R2.5.2 exercises this rejection directly.

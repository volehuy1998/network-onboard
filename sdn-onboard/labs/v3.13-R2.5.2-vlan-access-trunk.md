## Why this lab exists

Sub-task 1 of R2.5 (lab R2.5.1) showed that L2 learning behaves the same way in OVS and the kernel bridge from the workload's view. The two implementations diverged only in their forwarding-database representation and in their aging knobs, both of which are management-plane concerns; the data plane was identical. A learner reading R2.5.1 alone could fairly conclude that OVS is just the kernel bridge with a different control surface.

Sub-task 2 disproves that conclusion. With VLAN configuration the two diverge sharply at every layer that matters. The default behaviour differs (OVS treats every port as a trunk; the kernel bridge treats every port as VLAN-unaware until `vlan_filtering` is turned on). The configuration verb differs (`ovs-vsctl set port ... tag=10` versus `bridge vlan add dev ... vid 10 pvid untagged`). The data structures the configuration writes differ (an `OVSDB.Port.tag` integer versus a kernel per-port VLAN-membership bitmap). And although both implementations end up enforcing the same wire-level invariant (frames tagged with VLAN X cannot be delivered to ports that are not members of VLAN X), the path the configuration takes from operator intent to kernel enforcement is different in shape.

The lesson worth pinning is that a Cisco-trained engineer who carries the mental model "switchport access vlan 10 means a port that strips the tag on egress and inserts it on ingress" can map that model directly onto OVS's `tag=10` (one-to-one correspondence) but must take a detour through the kernel-bridge VLAN model to land on the equivalent in the Linux bridge. The detour is mechanical, not conceptual; the lab walks both paths and lands them on identical observable behaviour.

The four-namespace topology used here is the smallest configuration that proves both halves of the claim. Two namespaces in VLAN 10 (`ns_a` and `ns_b`) prove same-VLAN forwarding works. Two more in VLAN 20 (`ns_c` and `ns_d`) prove the same property for a second VLAN, which would be redundant on its own but becomes essential when paired with the cross-VLAN test: `ns_a` (VLAN 10) ping `ns_c` (VLAN 20) must fail. The four namespaces share one IP subnet (`10.0.0.0/24`) so the kernel attempts ARP for cross-VLAN destinations; the bridge's VLAN filter then drops the broadcast, and the cross-VLAN ping times out. If the four namespaces were in two distinct subnets, the kernel would refuse the cross-subnet ping with `Network is unreachable` before the packet ever reached the bridge, and the test would prove kernel routing rules rather than bridge VLAN enforcement.

## Phase A, OVS VLAN configuration

The OVS configuration model puts the VLAN attribute on the `Port` table, not on the bridge. A bridge holds zero or more ports; each port is either an access port (`tag` is set, `trunks` is empty), a trunk port (`trunks` is set with one or more VLAN IDs, `tag` is empty), or in one of two native-VLAN modes (`vlan_mode=native-tagged` or `native-untagged`, which combine the two). The default for a port that has neither `tag` nor `trunks` set is to be a trunk that allows every VLAN; this is the "all OVS ports are VLAN trunks by default" property the upstream `Documentation/howto/vlan.rst` document calls out.

The phase-A topology is built from four namespace plus veth pairs and one extra internal-type port `tap0` that lives in the host default namespace. The four veths are access ports: two on VLAN 10, two on VLAN 20. The `tap0` port is a trunk with membership in both VLAN 10 and VLAN 20; its purpose is to receive the mirror copy of every frame on the bridge so a host-side `tcpdump` can read 802.1Q tags from the wire.

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-br br0
root@lab-openvswitch:~/network-onboard# ip netns add ns_a; ip netns add ns_b; ip netns add ns_c; ip netns add ns_d
root@lab-openvswitch:~/network-onboard# ip link add veth_a type veth peer name veth_a_br
root@lab-openvswitch:~/network-onboard# ip link add veth_b type veth peer name veth_b_br
root@lab-openvswitch:~/network-onboard# ip link add veth_c type veth peer name veth_c_br
root@lab-openvswitch:~/network-onboard# ip link add veth_d type veth peer name veth_d_br
root@lab-openvswitch:~/network-onboard# ip link set veth_a netns ns_a
root@lab-openvswitch:~/network-onboard# ip link set veth_b netns ns_b
root@lab-openvswitch:~/network-onboard# ip link set veth_c netns ns_c
root@lab-openvswitch:~/network-onboard# ip link set veth_d netns ns_d
root@lab-openvswitch:~/network-onboard# ip link set veth_a_br up; ip link set veth_b_br up; ip link set veth_c_br up; ip link set veth_d_br up
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br0 veth_a_br tag=10
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br0 veth_b_br tag=10
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br0 veth_c_br tag=20
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br0 veth_d_br tag=20
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br0 tap0 -- set interface tap0 type=internal -- set port tap0 trunks=10,20
root@lab-openvswitch:~/network-onboard# ip link set tap0 up
root@lab-openvswitch:~/network-onboard# ovs-vsctl -- --id=@p get port tap0 -- --id=@m create mirror name=m0 select-all=true output-port=@p -- set bridge br0 mirrors=@m
a055821b-d5f3-40e7-a9cb-8f8d8254055e
```

The `tag=10` and `tag=20` arguments to `add-port` are shorthand for setting the `Port.tag` column directly. The mirror creation is the most complex `ovs-vsctl` invocation in this lab; it uses the `--id=@p` and `--id=@m` machinery to thread an OVSDB transaction that creates a `Mirror` row, links it to the bridge's `mirrors` column, and points its `output-port` at `tap0`'s row UUID. The single 36-character UUID printed at the end is the new mirror's row UUID, returned by OVSDB to confirm the transaction landed.

The four namespaces get IPs in one shared subnet so the kernel attempts ARP across VLAN boundaries:

```text
root@lab-openvswitch:~/network-onboard# ip netns exec ns_a ip link set lo up; ip netns exec ns_a ip link set veth_a up; ip netns exec ns_a ip addr add 10.0.0.1/24 dev veth_a
root@lab-openvswitch:~/network-onboard# ip netns exec ns_b ip link set lo up; ip netns exec ns_b ip link set veth_b up; ip netns exec ns_b ip addr add 10.0.0.2/24 dev veth_b
root@lab-openvswitch:~/network-onboard# ip netns exec ns_c ip link set lo up; ip netns exec ns_c ip link set veth_c up; ip netns exec ns_c ip addr add 10.0.0.3/24 dev veth_c
root@lab-openvswitch:~/network-onboard# ip netns exec ns_d ip link set lo up; ip netns exec ns_d ip link set veth_d up; ip netns exec ns_d ip addr add 10.0.0.4/24 dev veth_d
```

Now read back the per-port VLAN configuration to confirm the bridge's view matches operator intent:

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=name,tag,trunks,vlan_mode list port veth_a_br veth_b_br veth_c_br veth_d_br tap0
name                : veth_a_br
tag                 : 10
trunks              : []
vlan_mode           : []
 
name                : veth_b_br
tag                 : 10
trunks              : []

```

Five `Port` rows, four access (`tag` filled, `trunks` empty), one trunk (`tag` empty, `trunks` filled). The `vlan_mode` column is empty everywhere, which means OVS uses its default behaviour for each port type: an access port strips the VLAN tag on egress and inserts it on ingress; a trunk port forwards tagged frames unchanged; a port with no VLAN configuration at all would carry every VLAN.

The first behavioural test is the same-VLAN ping. `ns_a` and `ns_b` are both access ports for VLAN 10, so traffic between them flows; the namespaces never see a tag because the access ports strip on egress.

```text
root@lab-openvswitch:~/network-onboard# ip netns exec ns_a ping -c 2 10.0.0.2 2>&1 | tail -5

```

Both same-VLAN pings succeed with 0% packet loss. The cross-VLAN test follows. `ns_a` is in VLAN 10, `ns_c` is in VLAN 20; both share the `10.0.0.0/24` subnet so the kernel attempts ARP for `10.0.0.3`. The ARP request is broadcast, ingresses on `veth_a_br` (VLAN 10 access port, tagged with 10 on ingress), and the bridge's forwarding logic looks for VLAN 10 members to flood it to. `veth_c_br` is in VLAN 20, so it does not receive the request. No reply ever comes back; the ping times out.

```text
root@lab-openvswitch:~/network-onboard# ip netns exec ns_a ping -c 2 -W 1 10.0.0.3 2>&1 | tail -5 || echo '(blocked, different VLANs)'
PING 10.0.0.3 (10.0.0.3) 56(84) bytes of data.
 
--- 10.0.0.3 ping statistics ---
2 packets transmitted, 0 received, 100% packet loss, time 1007ms
```

Two packets transmitted, zero received, 100% packet loss. The bridge enforced the VLAN boundary; no kernel routing rule was needed.

The wire-byte proof is the next exercise. `tap0` is the mirror destination, it is a trunk port, so it sees every frame on the bridge with whatever VLAN tag the bridge assigned. A host-side `tcpdump -e -i tap0 -nn` captures four frames during a controlled refresh of the ARP cache plus a fresh ICMP exchange. Every captured frame carries an explicit `802.1Q (0x8100)` ethertype, a `vlan 10` tag, and an inner ethertype of either ARP or IPv4. This is the wire-byte proof that an access port tagged the frame on ingress with VLAN 10 and the bridge then carried that tag to the trunk port where the mirror copy emerged. The frames inside `ns_a` and `ns_b` were untagged; the access-port ingress added the tag, and the access-port egress would have stripped it again on the way back to `ns_b`. The mirror catches the in-flight tagged form, which is the form an external switch upstream of the host would see if the trunk were a real network cable.

The final phase-A observation reads the MAC-learning table. Notice that the FDB carries the VLAN identifier per row, not just the MAC and the port:

```text
root@lab-openvswitch:~/network-onboard# ovs-appctl fdb/show br0 | head -15
 port  VLAN  MAC                Age
    3    20  f6:09:b7:61:62:82    0
    4    20  82:b9:25:44:b5:2d    0
    1    10  52:e9:ed:12:02:c4    0
    2    10  6a:9b:f6:b2:d8:1d    0
```

Four entries, two per VLAN. Each row records OpenFlow port number, VLAN, MAC, and age. This is the answer to a subtle question a Cisco-trained engineer might ask: what happens if the same MAC appears on two different VLANs? OVS stores the MAC-learning entries keyed by `(VLAN, MAC)`, not by MAC alone, so the same MAC in two VLANs gets two separate rows. The kernel bridge uses the same model when `vlan_filtering` is enabled; the `bridge fdb show` output records `vlan` per dynamic entry. R2.5.1 saw the kernel-bridge FDB with `vlan 1` shadows on every port, which is the same data structure operating with the implicit-default VLAN.

## Phase B, Linux-bridge equivalent

The kernel bridge driver supports VLAN filtering since Linux 3.8 (2013). To enable it, the bridge has to be created with `vlan_filtering 1`; the default is off, which means a freshly-created bridge is VLAN-unaware and behaves like a hub for tagged frames (forwards them transparently with no membership checking).

```text
root@lab-openvswitch:~/network-onboard# ip link add name lbr0 type bridge vlan_filtering 1
root@lab-openvswitch:~/network-onboard# ip link set lbr0 up
root@lab-openvswitch:~/network-onboard# ip netns add ns_la; ip netns add ns_lb; ip netns add ns_lc; ip netns add ns_ld
root@lab-openvswitch:~/network-onboard# ip link add veth_la type veth peer name veth_la_br
root@lab-openvswitch:~/network-onboard# ip link add veth_lb type veth peer name veth_lb_br
root@lab-openvswitch:~/network-onboard# ip link add veth_lc type veth peer name veth_lc_br
root@lab-openvswitch:~/network-onboard# ip link add veth_ld type veth peer name veth_ld_br
root@lab-openvswitch:~/network-onboard# ip link set veth_la netns ns_la; ip link set veth_lb netns ns_lb; ip link set veth_lc netns ns_lc; ip link set veth_ld netns ns_ld
root@lab-openvswitch:~/network-onboard# ip link set veth_la_br master lbr0; ip link set veth_lb_br master lbr0; ip link set veth_lc_br master lbr0; ip link set veth_ld_br master lbr0
root@lab-openvswitch:~/network-onboard# ip link set veth_la_br up; ip link set veth_lb_br up; ip link set veth_lc_br up; ip link set veth_ld_br up
```

The first divergence from the OVS form is already visible. With OVS, attaching a port to the bridge and setting its VLAN tag are one operation (`ovs-vsctl add-port br0 veth_a_br tag=10`). With the kernel bridge, attaching is one operation (`ip link set ... master lbr0`) and configuring VLAN is a separate operation through the `bridge` command. The kernel attaches every port to a default VLAN 1 unless told otherwise; the next four commands remove that default and add the intended VLAN membership:

```text
root@lab-openvswitch:~/network-onboard# bridge vlan del dev veth_la_br vid 1; bridge vlan add dev veth_la_br vid 10 pvid untagged
root@lab-openvswitch:~/network-onboard# bridge vlan del dev veth_lb_br vid 1; bridge vlan add dev veth_lb_br vid 10 pvid untagged
root@lab-openvswitch:~/network-onboard# bridge vlan del dev veth_lc_br vid 1; bridge vlan add dev veth_lc_br vid 20 pvid untagged
root@lab-openvswitch:~/network-onboard# bridge vlan del dev veth_ld_br vid 1; bridge vlan add dev veth_ld_br vid 20 pvid untagged
```

The `pvid` flag is the Linux equivalent of OVS's `tag`: it tells the bridge "the default ingress VLAN for this port is N". The `untagged` flag tells the bridge "frames egressing this port should leave with the VLAN tag stripped". Together they reproduce the access-port semantics OVS gets from the single `Port.tag` column. The two-step `del vid 1; add vid N pvid untagged` pattern is necessary because the kernel auto-installs `vid 1` on every port; without removing it first, the port would be a member of both VLAN 1 and VLAN N, which is a trunk and not an access port.

Read back the per-port VLAN configuration:

```text
root@lab-openvswitch:~/network-onboard# bridge vlan show
port              vlan-id  
lbr0              1 PVID Egress Untagged
veth_la_br        10 PVID Egress Untagged
veth_lb_br        10 PVID Egress Untagged
veth_lc_br        20 PVID Egress Untagged
veth_ld_br        20 PVID Egress Untagged
```

The four veth ports are correctly configured. The bridge's own internal port `lbr0` retains `vid 1` because no command removed it; this is the equivalent of OVS's local internal port and would normally carry management traffic to the host. With this configuration the bridge is operationally identical to OVS phase A: ports `veth_la_br` and `veth_lb_br` are access ports for VLAN 10, ports `veth_lc_br` and `veth_ld_br` are access ports for VLAN 20, and frames cannot cross between the two VLANs.

The four namespaces get the same IP plan as phase A:

```text
root@lab-openvswitch:~/network-onboard# ip netns exec ns_la ip link set lo up; ip netns exec ns_la ip link set veth_la up; ip netns exec ns_la ip addr add 10.0.0.1/24 dev veth_la
root@lab-openvswitch:~/network-onboard# ip netns exec ns_lb ip link set lo up; ip netns exec ns_lb ip link set veth_lb up; ip netns exec ns_lb ip addr add 10.0.0.2/24 dev veth_lb
root@lab-openvswitch:~/network-onboard# ip netns exec ns_lc ip link set lo up; ip netns exec ns_lc ip link set veth_lc up; ip netns exec ns_lc ip addr add 10.0.0.3/24 dev veth_lc
```

## What R2.5.2 establishes for the comparative

Three findings to pin from this lab.

The first finding is that VLAN behaviour at the wire-byte layer is the same in both implementations. Every captured frame on the OVS trunk port carried `802.1Q vlan 10`; the kernel bridge would produce the same wire bytes if a trunk port were attached to it (the lab does not capture the kernel-bridge wire bytes because the same proof from the OVS side is sufficient and adding a tagged sub-interface to the kernel bridge for capture would multiply the lab's complexity without changing the result). Both implementations enforce the same wire-level invariant: a frame tagged with VLAN X reaches only ports that are members of VLAN X.

The second finding is that the configuration surface diverges sharply. OVS expresses the access-port property in one column (`Port.tag`); the kernel bridge needs three operations in sequence (`vlan_filtering 1` on the bridge, `bridge vlan del vid 1` to remove the default, `bridge vlan add vid N pvid untagged` to install the intended membership). The difference is not depth (both are mechanical) but locality: OVS keeps the VLAN intent in one place that an operator can `list port` to read; the kernel bridge spreads it across the bridge's `vlan_filtering` flag and the per-port `bridge vlan` rows, so reading the configuration requires consulting two surfaces. For a multi-tenant deployment with thousands of access ports, the OVSDB-centric model is operationally easier to audit; for a small two-VLAN lab the difference is small but recognisable.

The third finding concerns the default behaviour, which is the most common source of confusion when a Cisco-trained engineer first meets the kernel bridge. A Cisco port has to be told its `switchport mode access` plus `switchport access vlan N`; the default is access-mode VLAN 1. An OVS port created without `tag` or `trunks` is a trunk that allows every VLAN (the upstream `Documentation/howto/vlan.rst` states this explicitly: "By default, all OVS ports are VLAN trunks, so eth0 will pass all VLANs"). A kernel-bridge port created on a `vlan_filtering 0` bridge is VLAN-unaware and forwards tagged frames transparently; on a `vlan_filtering 1` bridge it is automatically a member of VLAN 1. Three implementations, three different defaults, all reasonable in their own context. Operators new to one of these will trip on the default at least once; this lab provides the reference points.

The R2.5 sprint continues with sub-task 3 (Spanning Tree Protocol and Rapid STP), where the divergence is wider still: OVS implements its own STP and RSTP state machines in userspace, while the kernel bridge has its own STP implementation in the kernel and gets RSTP only via an out-of-tree daemon. The two cannot be made to agree on convergence behaviour as easily as VLAN access ports can be made to agree on tag membership.

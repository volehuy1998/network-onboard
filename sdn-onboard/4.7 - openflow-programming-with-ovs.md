## 4.7.1 Flow grammar recap và `-O` flag

### Cấu trúc lệnh

`ovs-ofctl add-flow <bridge> <flow-spec>` với flow-spec = danh sách comma-separated của match fields + `actions=` + action list. Format tổng quát:

```
[selectors,] match_fields [,actions=action_list]
```

Selectors áp dụng cho flow metadata chứ không phải packet match:

| Selector | Mô tả | Default |
|----------|-------|---------|
| `table=N` | Table số N (0-254) | 0 |
| `priority=N` | 0-65535, cao hơn thắng | 32768 |
| `cookie=0xHEX` | 64-bit identifier cho flow management | 0 |
| `idle_timeout=N` | Seconds không match → auto remove | 0 (never) |
| `hard_timeout=N` | Seconds sau install → auto remove | 0 (never) |
| `importance=N` | 16-bit, OF 1.4+ eviction priority | 0 |
| `send_flow_rem` | Gửi FLOW_REMOVED khi flow expired | off |
| `check_overlap` | Reject nếu overlap với existing flow cùng priority | off |

### Wildcard priority semantics

Khi nhiều flow match cùng một packet, priority cao hơn thắng. Nếu cùng priority, OVS chọn flow với match specific hơn (ít wildcard hơn). Empty match (`actions=drop` mà không có match field nào) là **catch-all** với priority mặc định 32768.

Anti-pattern phổ biến: install flow `action=drop` priority=0 cùng với policy cho phép priority=100. Intent là "deny by default", nhưng priority=0 có thể match packet không tuân policy cụ thể khi policy rule không cover edge case. Cross-ref cross-product explosion trong 4.6.1.

### `-O` flag critical

OVS mặc định dùng **OpenFlow 1.0** khi gọi `ovs-ofctl` mà không chỉ định phiên bản. OF 1.0 không có multi-table, group, meter, hay khung instructions. Nếu dùng tính năng từ 1.1+, PHẢI truyền flag `-O`:

```bash
# Sai (OF 1.0, goto_table sẽ error)
ovs-ofctl add-flow br0 "table=0,ip,actions=goto_table:1"

# Đúng (OF 1.3, goto_table hoạt động)
ovs-ofctl -O OpenFlow13 add-flow br0 "table=0,ip,actions=goto_table:1"
```

Version negotiation OF: switch advertise supported versions, ovs-ofctl chọn version tương thích. Nếu bridge configured với `protocols=OpenFlow13`, ovs-ofctl vẫn default 1.0 khi command không pass `-O`. Best practice: luôn pass `-O OpenFlow13` (hoặc cao hơn) cho production scripts.

Cấu hình bridge để chỉ accept OF 1.3+:

```bash
ovs-vsctl set bridge br0 protocols=OpenFlow13
```

Sau đó `ovs-ofctl` (không có `-O`) sẽ fail với error "protocol not supported" khi bridge refuse OF 1.0.

## 4.7.2 Match fields: 12-tuple + NXM/OXM extensions

### L2 fields (Ethernet)

Tất cả match L2 đều support exact hoặc masked form:

| Field | Mô tả | Ví dụ |
|-------|-------|-------|
| `in_port=NAME_OR_ID` | Ingress port (name hoặc OF port number) | `in_port=eth1` hoặc `in_port=1` |
| `dl_src=MAC[/MASK]` | Source MAC | `dl_src=aa:bb:cc:00:00:00/ff:ff:ff:00:00:00` |
| `dl_dst=MAC[/MASK]` | Dest MAC | `dl_dst=01:80:c2:00:00:00/ff:ff:ff:ff:ff:f0` (STP multicast) |
| `dl_type=0xETHERTYPE` | EtherType | `dl_type=0x0800` (IPv4), `0x0806` (ARP), `0x8847` (MPLS), `0x88cc` (LLDP) |
| `dl_vlan=N` | VLAN ID 0-4094 | `dl_vlan=100` |
| `dl_vlan_pcp=N` | VLAN Priority 0-7 | `dl_vlan_pcp=5` |

Masked MAC match với `/` separator. Ví dụ match tất cả multicast:

```bash
ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=0,priority=100,dl_dst=01:00:00:00:00:00/01:00:00:00:00:00,action=drop"
```

Bit 0 của byte đầu tiên trong MAC là "multicast bit". Mask `01:00...` chọn bit này; value `01:00...` match khi bit = 1. Rule drop all multicast Ethernet (bao gồm broadcast `ff:ff:ff:ff:ff:ff` có bit này).

### L3 fields (IPv4)

| Field | Mô tả | Ví dụ |
|-------|-------|-------|
| `nw_src=IP[/PREFIX]` | Source IP | `nw_src=10.0.0.0/8` |
| `nw_dst=IP[/PREFIX]` | Dest IP | `nw_dst=192.168.1.100` |
| `nw_proto=N` | IP protocol number | `nw_proto=6` (TCP), `17` (UDP), `1` (ICMP) |
| `nw_tos=N` | IP TOS (6 bits DSCP + 2 bits ECN) | `nw_tos=0x10` |
| `nw_ttl=N` | Time To Live | `nw_ttl=1` (detect traceroute hop) |

IP match yêu cầu `dl_type=0x0800` (IPv4) trước. OVS auto-infer dl_type khi match L3 field, nhưng best practice explicit:

```bash
ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=0,ip,nw_src=10.0.0.0/8,action=goto_table:1"
# Shorthand "ip" = "dl_type=0x0800"
```

### L4 fields (TCP/UDP/ICMP)

| Field | Mô tả | Điều kiện prerequisite |
|-------|-------|------------------------|
| `tp_src=N` | L4 source port | `nw_proto=6` hoặc `17` |
| `tp_dst=N` | L4 dest port | Same |
| `tcp_flags=FLAGS[/MASK]` | TCP flags bitmask | `nw_proto=6` |
| `icmp_type=N` | ICMP message type | `nw_proto=1` |
| `icmp_code=N` | ICMP message code | `nw_proto=1` |

TCP flags bitmask: match `tcp_flags=+syn-ack` (SYN set, ACK not set) detect SYN flood. Multiple flag với `+/-` prefix:

```bash
ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=0,tcp,tcp_flags=+syn-ack,action=meter:1,NORMAL"
# Match chỉ SYN set (không ACK) → rate limit qua meter 1
```

### Tunnel metadata (overlay networking)

Khi OVS nhận packet qua tunnel (VXLAN, Geneve, GRE), metadata được populate:

| Field | Mô tả |
|-------|-------|
| `tun_id=N` | VNI của VXLAN hoặc VNI tag của Geneve |
| `tun_src=IP` | Outer IP source của tunnel packet |
| `tun_dst=IP` | Outer IP destination |
| `tun_flags=FLAGS` | Tunnel-specific flags |

Match per-tenant traffic trên overlay:

```bash
ovs-ofctl -O OpenFlow13 add-flow br-int \
  "table=20,tun_id=100,actions=mod_vlan_vid:100,NORMAL"
# Tunnel VNI 100 (tenant A) → tag VLAN 100 inside bridge
```

### Conntrack fields (OVS 2.5+ với kernel 4.3+)

Conntrack introducing stateful inspection vào OpenFlow pipeline:

| Field | Mô tả |
|-------|-------|
| `ct_state=+FLAG-FLAG` | Conntrack state bitmask |
| `ct_zone=N` | Zone (isolate overlapping NAT ranges) |
| `ct_mark=N/MASK` | Per-flow mark (32-bit) |
| `ct_label=N/MASK` | Per-flow label (128-bit) |

`ct_state` flags:

| Flag | Mô tả |
|------|-------|
| `trk` | Packet tracked qua conntrack |
| `new` | First packet of new connection |
| `est` | Established connection (seen return) |
| `rel` | Related (e.g., FTP data channel từ control) |
| `inv` | Invalid (malformed, out-of-window) |
| `rpl` | Reply direction |
| `snat` | SNAT applied |
| `dnat` | DNAT applied |

Xem recipe 4.7.5 cho flow sequence canonical.

### Registers (NXM extension, không phải chuẩn OF)

OVS có NXM registers cho temporary storage trong pipeline:

| Field | Size | Mô tả |
|-------|------|-------|
| `reg0`...`reg15` | 32-bit mỗi | General purpose |
| `xreg0`...`xreg7` | 64-bit mỗi | Wider registers (alias cho reg pair) |
| `xxreg0`...`xxreg3` | 128-bit mỗi | Widest (alias cho 4 reg) |
| `metadata` | 64-bit | Spec-defined metadata (OF 1.1+) |

Use case: store tenant ID trong reg0 ở Table 0, read trong Table 5 để choose uplink:

```bash
ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=0,in_port=eth1,action=load:0x1->NXM_NX_REG0[],goto_table:1"
# Tenant A (port eth1) → reg0 = 0x1

ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=5,reg0=0x1,action=output:eth_tenant_a_uplink"
```

### Masked matches syntax

CIDR cho IP: `nw_src=10.0.0.0/8`.

Slash-mask cho arbitrary field:

```
dl_src=00:11:22:00:00:00/ff:ff:ff:00:00:00  # Match OUI 00:11:22
reg0=0x10/0xff                              # Match low 8 bits = 0x10
```

## 4.7.3 Actions: 8 cơ bản + extensions

### Output actions

Basic action family quyết định packet đi đâu:

| Action | Mô tả |
|--------|-------|
| `output:N` | Send out specific port (name hoặc number) |
| `NORMAL` | Đưa packet vào NORMAL L2 learning pipeline (MAC learning + flood/unicast) |
| `FLOOD` | Send out tất cả ports trừ ingress, subject to STP blocking |
| `ALL` | Send out tất cả ports trừ ingress (bỏ qua STP) |
| `LOCAL` | Send đến local internal port (bridge's internal port) |
| `IN_PORT` | Send back out ingress port (hairpin) |
| `CONTROLLER[:max_len]` | Send PACKET_IN đến controller, optional truncate tại max_len bytes |
| `drop` | Không làm gì (empty action list = drop) |

### Header modification

| Action | Mô tả |
|--------|-------|
| `mod_vlan_vid:N` | Set VLAN ID |
| `strip_vlan` | Remove VLAN tag |
| `push_vlan:0x8100` | Add VLAN tag (802.1Q) hoặc 0x88a8 (QinQ) |
| `pop_vlan` | Same as strip_vlan |
| `mod_dl_src:MAC` | Rewrite source MAC |
| `mod_dl_dst:MAC` | Rewrite destination MAC |
| `mod_nw_src:IP` | Rewrite source IP |
| `mod_nw_dst:IP` | Rewrite destination IP |
| `dec_ttl` | Decrement IP TTL, send ICMP Time Exceeded nếu = 0 |

### General field operations

Generic access cho any NXM/OXM field:

```
set_field:VALUE->FIELD    # Set field to value
load:VALUE->FIELD[BITS]   # Set specific bit range
move:SRC->DST             # Copy from SRC field to DST field
```

Ví dụ copy VLAN ID sang reg0 cho lookup downstream:

```bash
ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=0,dl_vlan=100,action=move:NXM_OF_VLAN_TCI[0..11]->NXM_NX_REG0[],goto_table:1"
```

### Pipeline actions (OF 1.1+)

| Action | Mô tả |
|--------|-------|
| `resubmit([PORT],[TABLE])` | NXM extension: re-run packet through table (OF 1.0 workaround cho multi-table) |
| `goto_table:N` | OF 1.1+: continue pipeline ở table N (N > current) |
| `group:N` | Apply group actions |
| `meter:N` | Apply meter N (rate limit) |

`resubmit` là NXM-only, không phải OF chuẩn. Dùng cho OVS 1.0 emulate multi-table. Sau OF 1.1, `goto_table` là đúng cách.

### Stateful actions (conntrack, learn)

```
ct(args)       # Conntrack action với args: commit, zone=N, nat(...), table=N, exec(...)
ct_clear       # Clear conntrack state (packet leaves pipeline without tracking)
learn(args)    # Install new flow based on current packet
```

Xem recipe 4.7.5 cho ct. `learn` là advanced, install flow dynamically từ current packet fields.

### Advanced actions

| Action | Mô tả |
|--------|-------|
| `conjunction(id,k/n)` | Conjunctive match (cross-product compression) |
| `clone(...)` | Run nested actions trên packet copy |
| `encap(...)`/`decap()` | Push/pop encapsulation (VXLAN, Geneve, NSH) |
| `check_pkt_larger:SIZE` | Check packet size vs threshold, update reg |
| `truncate:SIZE` | Truncate packet to size bytes before output |

`conjunction` là NXM-only, cho phép match "any of set A AND any of set B" without cross-product explosion. Advanced topic, skip trong Part này.

## 4.7.4 Multi-table pipeline recipe: 3-stage L3 routing (UofSC Lab 6 absorb)

### Topology

Use case canonical: OVS bridge s1 làm L3 router giữa 2 subnet:

```
Host A (192.168.1.10/24) ──eth1── s1 ──eth2── Host B (192.168.2.20/24)
                          00:..:01            00:..:05
```

s1 có 2 internal port (s1-eth1 trong VLAN 1, s1-eth2 trong VLAN 2), routing giữa 2 subnet.

### Single-table approach (cross-product explosion)

Naive approach: một flow per tenant-subnet pair:

```bash
# Direction 1: A → B
ovs-ofctl -O OpenFlow13 add-flow s1 \
  "table=0,priority=100,ip,in_port=eth1,nw_src=192.168.1.0/24,nw_dst=192.168.2.0/24,\
   action=mod_dl_src:00:00:00:00:00:01,mod_dl_dst:00:00:00:00:00:05,dec_ttl,output:eth2"

# Direction 2: B → A
ovs-ofctl -O OpenFlow13 add-flow s1 \
  "table=0,priority=100,ip,in_port=eth2,nw_src=192.168.2.0/24,nw_dst=192.168.1.0/24,\
   action=mod_dl_src:00:00:00:00:00:05,mod_dl_dst:00:00:00:00:00:01,dec_ttl,output:eth1"
```

Không scale: N subnet → N² flow entries cho all pairs. 10 subnet = 100 flow; 100 subnet = 10K flow, gần limit Broadcom Trident2 ACL.

### Multi-table approach (compression)

Cấu trúc 3-stage:

**Table 0, Classifier**: phân loại traffic type.

```bash
# ARP traffic → NORMAL learning (OVS tự xử lý)
ovs-ofctl -O OpenFlow13 add-flow s1 "table=0,priority=100,arp,action=normal"

# IP traffic → Table 1 (L3 routing)
ovs-ofctl -O OpenFlow13 add-flow s1 "table=0,priority=50,ip,action=goto_table:1"

# Default: drop
ovs-ofctl -O OpenFlow13 add-flow s1 "table=0,priority=0,action=drop"
```

**Table 1, L3 Forwarding**: match destination subnet, rewrite MAC, set egress metadata.

```bash
# Route to subnet 1 (192.168.1.0/24)
ovs-ofctl -O OpenFlow13 add-flow s1 \
  "table=1,priority=100,ip,nw_dst=192.168.1.0/24,\
   action=mod_dl_src:00:00:00:00:00:01,mod_dl_dst:00:00:00:00:00:01,dec_ttl,\
   load:1->NXM_NX_REG0[],goto_table:2"

# Route to subnet 2 (192.168.2.0/24)
ovs-ofctl -O OpenFlow13 add-flow s1 \
  "table=1,priority=100,ip,nw_dst=192.168.2.0/24,\
   action=mod_dl_src:00:00:00:00:00:05,mod_dl_dst:00:00:00:00:00:05,dec_ttl,\
   load:2->NXM_NX_REG0[],goto_table:2"
```

Note: ở Table 1, MAC của destination thực chất phải learn qua ARP, không hard-code. Recipe đơn giản hoá cho tutorial; production dùng OVN hoặc controller application với ARP learning logic.

**Table 2, L2 Output**: output dựa trên reg0 (subnet index).

```bash
# Reg0 = 1 → eth1 (subnet 1 port)
ovs-ofctl -O OpenFlow13 add-flow s1 \
  "table=2,priority=100,reg0=1,action=output:eth1"

# Reg0 = 2 → eth2 (subnet 2 port)
ovs-ofctl -O OpenFlow13 add-flow s1 \
  "table=2,priority=100,reg0=2,action=output:eth2"
```

Flow count: 4 rules ở Table 0 + N (subnet) rules ở Table 1 + N rules ở Table 2 = O(N) thay vì O(N²).

### Verify

```bash
ovs-ofctl -O OpenFlow13 dump-flows s1
```

Show tất cả flow với hit counter. Packet ping từ A → B sẽ:

1. Hit `in_port=eth1,arp` first (ARP request) → NORMAL learn MAC.
2. Hit `in_port=eth1,ip,nw_dst=192.168.2.0/24` (ICMP echo) → Table 1 → Table 2 → output eth2.
3. Hit reverse flow khi B reply.

Counter tăng cho mỗi packet, debug qua `dump-flows` output.

## 4.7.5 Conntrack recipe: 5-flow stateful firewall (Compass Ch 9 + UofSC Lab 8)

### Goal: stateful egress firewall

Policy: cho phép host A (left side) khởi xướng connection đến host B (right side), nhưng block connection khởi xướng từ B. Return traffic của A→B được cho phép automatically.

Topology: bridge br0 với 2 port `veth_l0` (host A) và `veth_r0` (host B).

### 5-flow sequence

```bash
# Flow 1: New outbound → send to conntrack (table 0 re-entry after ct)
ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=0,priority=50,ct_state=-trk,tcp,in_port=veth_l0,\
   action=ct(table=0)"

# Flow 2: Tracked + new outbound → commit to conntrack + forward
ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=0,priority=50,ct_state=+trk+new,tcp,in_port=veth_l0,\
   action=ct(commit),output:veth_r0"

# Flow 3: Return traffic (dst side) → send to conntrack
ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=0,priority=50,ct_state=-trk,tcp,in_port=veth_r0,\
   action=ct(table=0)"

# Flow 4: Established reply → forward out left
ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=0,priority=50,ct_state=+trk+est,tcp,in_port=veth_r0,\
   action=output:veth_l0"

# Flow 5: Established outbound → forward
ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=0,priority=50,ct_state=+trk+est,tcp,in_port=veth_l0,\
   action=output:veth_r0"

# Default: drop (không trackable hoặc không established)
ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=0,priority=0,action=drop"
```

### Logic flow

1. Packet từ A (veth_l0) đến B: 
   - Flow 1 match (ct_state=-trk), send to conntrack. 
   - Packet returns với ct_state=+trk+new. 
   - Flow 2 match, commit to conntrack, output veth_r0.

2. Packet return từ B (veth_r0) đến A:
   - Flow 3 match (ct_state=-trk), send to conntrack.
   - Conntrack lookup finds existing flow, packet returns với ct_state=+trk+est.
   - Flow 4 match, output veth_l0.

3. Subsequent packets A → B (established):
   - Flow 1 match, send to conntrack.
   - Return với ct_state=+trk+est.
   - Flow 5 match, output veth_r0.

4. **Attack scenario**: B attempt to connect A:
   - Flow 3 match, send to conntrack.
   - Conntrack không find existing flow, returns với ct_state=+trk+new.
   - Không có flow match (+trk+new từ veth_r0) → default drop.

Policy enforced: A có thể khởi xướng, B không thể.

### Verify

```bash
# Liệt kê conntrack entries
ovs-appctl dpctl/dump-conntrack

# Example output:
# tcp,orig=(src=10.0.0.1,dst=10.0.0.2,sport=33456,dport=80),\
#   reply=(src=10.0.0.2,dst=10.0.0.1,sport=80,dport=33456),\
#   state=est,zone=0
```

Mỗi conntrack entry có state (new/est/inv), zone (isolation), mark (metadata), label (128-bit).

### Advanced: NAT + zone

Conntrack với SNAT (source NAT) ra external IP:

```bash
ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=0,priority=50,ct_state=+trk+new,tcp,in_port=veth_l0,\
   action=ct(commit,nat(src=203.0.113.10)),output:veth_r0"
```

Zone isolation cho overlapping NAT:

```bash
# Tenant A trong zone 100
ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=0,priority=50,ct_state=-trk,tcp,in_port=veth_tenant_a,\
   action=ct(table=0,zone=100)"

# Tenant B trong zone 200 (cùng IP range nhưng tách biệt)
ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=0,priority=50,ct_state=-trk,tcp,in_port=veth_tenant_b,\
   action=ct(table=0,zone=200)"
```

OVN dùng zone pattern này cho multi-tenant isolation. Cross-ref Part 13.3 (OVN ACL + conntrack integration).

## 4.7.6 Groups + meters (OF 1.1+ và 1.3+)

### Group types

OF 1.1 introduced groups với 4 types (xem Part 4.0.4):

```bash
# ECMP (Equal-Cost Multi-Path) với 2 uplink
ovs-ofctl -O OpenFlow13 add-group br0 \
  "group_id=1,type=select,\
   bucket=weight=50,output:eth_up1,\
   bucket=weight=50,output:eth_up2"

# Reference từ flow
ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=0,priority=100,ip,action=group:1"
```

Group types:

| Type | Mô tả |
|------|-------|
| `all` | Execute all buckets (multicast) |
| `select` | Execute one bucket based on hash (ECMP) |
| `indirect` | Exactly one bucket, pointer-like indirection |
| `fast_failover` | Watch port/group, active bucket based on liveness |

FAST_FAILOVER example (xem Part 4.6 Capstone POE):

```bash
ovs-ofctl -O OpenFlow13 add-group br0 \
  "group_id=2,type=ff,\
   bucket=watch_port:eth_p1,output:eth_p1,\
   bucket=watch_port:eth_p2,output:eth_p2"
```

### Meters (OF 1.3+)

Rate limit per-flow với meter:

```bash
# Meter 1: 1Mbps drop rate
ovs-ofctl -O OpenFlow13 add-meter br0 \
  "meter=1,kbps,band=type=drop,rate=1000,burst_size=100"

# Reference từ flow
ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=0,priority=100,ip,nw_src=10.0.0.100,action=meter:1,NORMAL"
# Traffic từ host 10.0.0.100 rate limit 1Mbps, vượt rate sẽ drop
```

Multi-band DSCP_REMARK (color marking):

```bash
ovs-ofctl -O OpenFlow13 add-meter br0 \
  "meter=2,kbps,\
   band=type=dscp_remark,rate=1000,burst_size=100,prec_level=1,\
   band=type=drop,rate=10000,burst_size=1000"
# < 1Mbps: green (pass)
# 1-10Mbps: yellow (DSCP remark precedence+1)
# > 10Mbps: red (drop)
```

Xem Part 9.9 cho OVS meter integration với `linux-htb` QoS recipe production.

## 4.7.7 Flow hygiene: monitor, replace-flows, diff-flows

### Monitor: live stream flow changes

Controller app install/remove flow → ovs-ofctl monitor stream mọi FLOW_MOD event:

```bash
ovs-ofctl monitor br0 watch:
# Output example:
# NXST_FLOW reply: (xid=0x3):
#  cookie=0x0, duration=5.12s, table=0, n_packets=100, n_bytes=8400, priority=50,ip,nw_dst=10.0.0.0/8 actions=goto_table:1
```

`watch:` filter:
- `table=N` chỉ monitor table cụ thể.
- `cookie=HEX/MASK` filter theo cookie.

Use case debug: controller logic có bug install rules conflict → dùng monitor để thấy live sequence FLOW_MOD từ controller side.

### replace-flows: atomic swap toàn bộ table

```bash
# Chuẩn bị file với flows mới
cat > new.flows <<EOF
table=0,priority=100,ip,nw_src=10.0.0.0/8,actions=goto_table:1
table=0,priority=0,actions=drop
EOF

# Atomic swap
ovs-ofctl -O OpenFlow13 replace-flows br0 new.flows
```

Khác với `add-flows`: `add-flows` append vào existing table, `replace-flows` remove all existing flows (trừ những cái trong new.flows cùng cookie/match) rồi install new.

`replace-flows` quan trọng cho production: atomic, không có moment mà table trong inconsistent state. Tương đương OF 1.4 bundle semantic (xem Part 4.3.2) nhưng available từ OVS 2.0.

### diff-flows: preview thay đổi

**BẮT BUỘC** chạy trước `replace-flows` trong production. Show differences between current table và file:

```bash
ovs-ofctl -O OpenFlow13 diff-flows br0 new.flows
# Example output:
# +table=0,priority=100,ip,nw_src=10.0.0.0/8,actions=goto_table:1
# -table=0,priority=100,ip,nw_src=192.168.0.0/16,actions=goto_table:1
```

`+` = flow sẽ được add, `-` = flow sẽ bị remove. Nếu output trống, không có change.

Workflow production:

```bash
# Step 1: Preview
ovs-ofctl -O OpenFlow13 diff-flows br0 new.flows > /tmp/diff.txt
cat /tmp/diff.txt
# Review changes

# Step 2: If OK, apply
ovs-ofctl -O OpenFlow13 replace-flows br0 new.flows

# Step 3: Verify
ovs-ofctl -O OpenFlow13 dump-flows br0
```

### Hiểu sai thường gặp

**Hiểu sai:** "add-flows thêm mới mà không động đến existing flow, safe."

**Thực tế:** `add-flows` có thể vô tình overwrite nếu flow mới có same `(table, priority, match)` với existing flow. OVS treat đây là update, existing flow bị replace silently. Tương tự `OFPFC_ADD` của OpenFlow protocol. Để thực sự "add only if not exists", phải dùng `OFPFC_ADD` với `check_overlap` flag hoặc query trước. Production practice: luôn `replace-flows` với full file, không `add-flows` incremental, để intent explicit và diff visible.

## 4.7.8 Advanced tutorial: 5-table MAC learning (Compass Ch 8)

### Context

OVS documentation có tutorial canonical `docs.openvswitch.org/en/latest/tutorials/ovs-advanced/` build VLAN-aware MAC learning switch qua 5 stages. Đây là exercise pedagogically valuable nhất trong OVS curriculum vì nó combine tất cả primitives: multi-table, registers, learn action, conntrack.

### 5-stage structure

| Table | Mục đích | Primitives dùng |
|-------|---------|-----------------|
| 0 | Admission control (block unauthorized MAC) | Match + drop hoặc goto |
| 10 | VLAN input handling (assign VLAN) | mod_vlan_vid, goto |
| 20 | MAC + VLAN learning | `learn(...)` installing dynamic flow trong Table 30 |
| 30 | Destination lookup (learned MAC → port) | Match dl_dst + VLAN, output |
| 40 | Unknown destination flooding | `FLOOD` restricted by VLAN |

### `learn(...)` action mechanics

Khi packet đến Table 20, `learn` action install flow mới trong Table 30:

```bash
ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=20,action=learn(table=30,NXM_OF_VLAN_TCI[0..11],NXM_OF_ETH_DST[]=NXM_OF_ETH_SRC[],\
   load:NXM_OF_IN_PORT[]->NXM_NX_REG0[]),output:NXM_NX_REG0[]"
# Không phải syntax exact production, đơn giản hoá cho illustration
```

Logic: flow học MAC src của packet hiện tại và port ingress, install flow ở Table 30 match MAC đó (khi thành destination trong future) với action output về port đã học.

### Học sâu hơn

Full 5-table tutorial trong OVS docs đi qua từng stage với verify command, expected output, và troubleshoot common mistakes. Nó dài ~30 trang và yêu cầu Mininet + OVS setup. Recommended exercise sau khi hoàn tất recipe 4.7.4 và 4.7.5. Cross-reference URL trong references.

### ▶ Guided Exercise 1: Multi-table L3 routing pipeline

**Outcomes:**
- Xác minh packet count flow per table trong 3-stage pipeline.
- Quan sát reg0 propagation từ Table 1 → Table 2.

**Before You Begin:**
Mininet hoặc 2 netns với veth pair đến bridge s1 (OVS). Host A tại 192.168.1.10/24, host B tại 192.168.2.20/24. Gateway MAC mock (hard-coded trong flow).

**Instructions:**

**1.** Install 3-stage pipeline (flows từ 4.7.4).

**2.** Send ICMP từ A đến B:

```bash
[host-a]$ ping -c 5 192.168.2.20
```

**3.** Verify flow hit counter mỗi table:

```bash
[ovs]$ ovs-ofctl -O OpenFlow13 dump-flows s1 table=0
# Expected: flow arp và flow ip có n_packets > 0
[ovs]$ ovs-ofctl -O OpenFlow13 dump-flows s1 table=1
# Expected: flow nw_dst=192.168.2.0/24 có n_packets = 5 (ICMP request)
[ovs]$ ovs-ofctl -O OpenFlow13 dump-flows s1 table=2
# Expected: flow reg0=2 có n_packets = 5
```

**Finish:** Tổng 5 hit trên cả 3 table chứng minh packet traverse pipeline theo đúng logic. Nếu Table 1 có 0 hit → Table 0 goto_table không hoạt động; nếu Table 2 có 0 hit → reg0 không được set đúng.

### ▶ Guided Exercise 2: Conntrack firewall recipe

**Outcomes:**
- Verify A → B connection khởi xướng được.
- Verify B → A connection bị block.
- Observe conntrack state qua `dpctl/dump-conntrack`.

**Before You Begin:**
2 netns `tenant-l` (với veth `veth_l0`) và `tenant-r` (với veth `veth_r0`), both attached to bridge br0. Netcat listening trong tenant-r.

**Instructions:**

**1.** Install 5-flow sequence từ 4.7.5.

**2.** Test A khởi xướng:

```bash
[tenant-l]$ nc -w 3 <tenant-r-ip> 1234
# Expected: connect thành công
```

**3.** Verify conntrack state:

```bash
[host]$ ovs-appctl dpctl/dump-conntrack
# Expected entry: tcp,orig=(src=...,dst=...,sport=...,dport=1234),state=est,...
```

**4.** Test B khởi xướng:

```bash
[tenant-r]$ nc -w 3 <tenant-l-ip> 5678
# Expected: connect refused hoặc timeout
```

**5.** Observe flow hit counter:

```bash
[host]$ ovs-ofctl -O OpenFlow13 dump-flows br0 | grep "ct_state"
# Flow 1 + 2: n_packets > 0 (A outbound)
# Flow 3 + 4: n_packets > 0 (return traffic)
# Drop flow (priority=0): n_packets > 0 từ B attempt
```

**Finish:** Recipe chứng minh conntrack track state đúng: A-initiated traffic passes, B-initiated traffic blocked. Drop counter tăng cho mỗi B attempt là evidence.

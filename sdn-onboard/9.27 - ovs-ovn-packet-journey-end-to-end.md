## 9.27.2 Framework 3-tier parallel diagnostic

Debug OVN cross-host là debug **3 tầng song song**. Mỗi tầng có object of truth riêng, tool riêng, câu hỏi riêng.

### Tầng 1 — Logical ý đồ cấu hình (OVN)

**Object of truth:** OVN Northbound + Southbound tables (`ovn-nbctl show`, `ovn-sbctl list Logical_Flow`).

**Tool:** `ovn-trace --detailed "..."` (offline mô phỏng qua logical pipeline).

**Câu hỏi trả lời:** "OVN muốn packet đi đâu? Ý đồ cấu hình có đúng không?"

**Khi nào check:** Luôn đầu tiên. Nếu ý đồ cấu hình sai, translation dù đúng cũng vô nghĩa.

```bash
ovn-trace --detailed "inport == \"lsp-vm1\" && eth.src == fa:16:3e:00:00:01 && eth.dst == fa:16:3e:00:00:02 && ip4.src == 10.0.0.10 && ip4.dst == 10.0.0.20"
```

### Tầng 2 — OpenFlow translation (br-int)

**Object of truth:** OpenFlow flow table của `br-int` (do ovn-controller cài từ Logical_Flow SBDB).

**Tool:** `ovs-appctl ofproto/trace br-int "..."` (mô phỏng qua OpenFlow pipeline, bao gồm recirc `ct()` nếu truyền `--ct-next`).

**Câu hỏi trả lời:** "ovn-controller đã translate ý đồ cấu hình thành OpenFlow đúng chưa? Có sync lag không?"

**Khi nào check:** Sau khi tầng 1 OK, tầng 2 sai nghĩa là ovn-controller issue (crash, SSL, SB sync).

```bash
TAP1_OF=$(sudo ovs-vsctl get interface tap1 ofport)
sudo ovs-appctl ofproto/trace br-int \
  "in_port=${TAP1_OF},ip,nw_src=10.0.0.10,nw_dst=10.0.0.20,dl_src=fa:16:3e:00:00:01,dl_dst=fa:16:3e:00:00:02"
```

### Tầng 3 — Datapath runtime (kernel megaflow cache)

**Object of truth:** Megaflow cache trong kernel openvswitch module (hoặc userspace `dpif-netdev` nếu DPDK).

**Tool:** `ovs-appctl dpif/dump-flows br-int` (xem entry đã cache) + `ovs-appctl dpif/show` (tổng thể sức khoẻ).

**Câu hỏi trả lời:** "Kernel đã cache path chưa? Fast path hay slow path? Có drop ở datapath không?"

**Khi nào check:** Sau khi tầng 1 + 2 OK, packet vẫn drop → datapath stale hoặc kernel module issue.

```bash
sudo ovs-appctl dpif/dump-flows br-int | grep "in_port=${TAP1_OF}"
sudo ovs-appctl dpif/show
```

### Ma trận decision — khi nào check tầng nào

```
Packet drop observed cross-host
│
├── Tầng 1 — ovn-trace OK?
│   ├── NO (logical drop) → Check OVN NB config: lsp-add, lsp-set-addresses,
│   │                      ACL, LB, port-security. Fix logical intent.
│   └── YES → Tầng 2
│
├── Tầng 2 — ofproto/trace br-int OK?
│   ├── NO (OpenFlow drop) → Check ovn-controller:
│   │                        systemctl status ovn-controller
│   │                        journalctl -u ovn-controller --since "10 min ago"
│   │                        ovn-sbctl show (SB sync OK?)
│   │                        Restart ovn-controller nếu cần.
│   └── YES → Tầng 3
│
└── Tầng 3 — dpif/dump-flows có entry + packets tăng?
    ├── NO (0 packets) → Datapath chưa cache, upcall failing:
    │                    ovs-vswitchd log + dpif/show miss rate
    │                    Kernel module loaded? (lsmod | grep openvswitch)
    ├── YES có packets tăng → OVS OK, drop nằm ngoài br-int:
    │                         Check tunnel (Stage 3 encap)
    │                         Check underlay (Stage 4)
    │                         Check remote chassis (Stage 5+6)
    └── YES có packets NHƯNG counter không tăng → Datapath stale entry:
         ovs-appctl dpctl/del-flows br-int "<match>"
         Force re-upcall để refresh cache.
```

Engineer thành thạo gọi 3 tool song song trong 3 tab terminal, compare output trong vài giây. Không cần memorize exact command; memorize framework 3 tầng + decision tree.

## 9.27.3 Geneve TLV deep-dive — đọc outer header từ packet capture

OVN dùng Geneve (RFC 8926) làm tunnel default. Engineer debug cần biết đọc outer Geneve header trực tiếp từ `tcpdump` hoặc Wireshark.

### Packet format

```
+---------+----------------------------+
| 14 byte | Outer Ethernet             |
|         |   src=<chassis-A MAC>      |
|         |   dst=<chassis-B MAC or    |
|         |        next-hop router>    |
|         |   ethertype=0x0800 (IPv4)  |
+---------+----------------------------+
| 20 byte | Outer IPv4                 |
|         |   src=192.168.10.10        |
|         |   dst=192.168.10.20        |
|         |   proto=17 (UDP)           |
|         |   DF flag set (don't       |
|         |   fragment, standard OVS)  |
+---------+----------------------------+
|  8 byte | Outer UDP                  |
|         |   src=<random, 1024-65535> |
|         |   dst=6081 (Geneve)        |
|         |   length=<Geneve+inner>    |
|         |   checksum=0 default       |
+---------+----------------------------+
|  8 byte | Geneve base header         |
|         |   Ver=0 (current)          |
|         |   OptLen=N (4-byte words)  |
|         |   O flag: control packet   |
|         |   C flag: critical opts    |
|         |   Reserved 6 bit           |
|         |   Protocol Type=0x6558     |
|         |   VNI=24-bit logical DP ID |
|         |   Reserved 8 bit           |
+---------+----------------------------+
|  8 byte | Geneve TLV option 1 (OVN)  |
|         |   Class=0x0102 (OVN)       |
|         |   Type=0x80 (critical)     |
|         |   R=0, Length=1 (4 byte)   |
|         |   Value: logical ingress   |
|         |          port (32-bit)     |
+---------+----------------------------+
|  8 byte | Geneve TLV option 2 (OVN)  |
|         |   Class=0x0102             |
|         |   Type=0x81                |
|         |   R=0, Length=1            |
|         |   Value: logical egress    |
|         |          port (32-bit)     |
+---------+----------------------------+
|    ...  | Inner Ethernet (14 byte)   |
|         |   src=<VM1 MAC>            |
|         |   dst=<VM2 MAC>            |
+---------+----------------------------+
|    ...  | Inner IPv4 + payload       |
|         |   (VM traffic)             |
+---------+----------------------------+
```

Total overhead: 14 + 20 + 8 + 8 + 8 × 2 TLV = **66 bytes** (OVN default với 2 TLV).

### Capture và dissect

```bash
# Chassis A, capture outbound Geneve
sudo tcpdump -ni eth0 -w /tmp/geneve-a.pcap "udp and port 6081" -c 20

# Dissect với tshark
tshark -r /tmp/geneve-a.pcap -V -Y "geneve" | head -60

# Output (doc-plausible, rút gọn):
# Frame 1: 168 bytes on wire
# Ethernet II, Src: ..., Dst: ...
# Internet Protocol Version 4
#     Src: 192.168.10.10
#     Dst: 192.168.10.20
# User Datagram Protocol
#     Source Port: 53421
#     Destination Port: 6081
# Generic Network Virtualization Encapsulation
#     Flags: 0x00
#     Protocol Type: Ethernet (0x6558)
#     VNI: 0x000001
#     Options:
#         Option: Class 0x0102 Type 0x80 — OVN ingress port
#             Length: 4
#             Value: 0x00000001
#         Option: Class 0x0102 Type 0x81 — OVN egress port
#             Length: 4
#             Value: 0x00000002
# Ethernet II (inner)
#     Src: fa:16:3e:00:00:01
#     Dst: fa:16:3e:00:00:02
# Internet Protocol Version 4 (inner)
#     Src: 10.0.0.10
#     Dst: 10.0.0.20
# ICMP Echo Request
```

### OVN TLV semantics

| Class | Type | Meaning | Used by |
|-------|------|---------|---------|
| `0x0102` | `0x80` | Logical ingress port (metadata reg14) | OVN ovn-controller chassis A encode |
| `0x0102` | `0x81` | Logical egress port (metadata reg15) | OVN ovn-controller chassis A encode |
| `0x0102` | other | Reserved future OVN use | — |

Nút mạng (Chassis) B decap Geneve → extract TLV → inject `reg14` + `reg15` + metadata vào pipeline recirc. Không cần lookup Port_Binding table phía receive.

### Diagnostic commands

```bash
# List tunnel ports local chassis
sudo ovs-vsctl find Interface type=geneve
# _uuid, name=genev_sys_6081, options={csum="true", key=flow, remote_ip=flow}

# Tunnel ports detail
sudo ovs-appctl tnl/ports/show
# Listening to 0.0.0.0:6081 for Geneve

# BFD state cho tunnel
sudo ovs-appctl bfd/show
# chassis-b UP (remote 192.168.10.20)

# Geneve stats
sudo ovs-appctl dpctl/dump-flows type=ovs | grep "tun_dst=192.168.10.20" | head -3
```

## 9.27.4 MTU forensic — tính toán chuẩn xác

MTU mismatch giữa underlay và overlay là **nguồn lớn nhất** bug cross-host OVN. Triệu chứng điển hình:

- Ping nhỏ (< 1400 bytes) OK.
- Ping lớn (> 1400 bytes) fail hoặc timeout.
- TCP small request OK, big file download stuck.
- SSL handshake OK, file transfer hangs.

### Math đúng

```
Underlay MTU (link layer) = 1500 byte (Ethernet default)
Overlay MTU (what VM sees) = ?

Overhead Geneve:
  Outer Eth         = 14 byte
  Outer IPv4        = 20 byte
  Outer UDP         =  8 byte
  Geneve base       =  8 byte
  Geneve 2 TLV OVN  = 16 byte  (2 × 8)
  ─────────────────────────
  Total overhead    = 66 byte

Max overlay MTU = Underlay MTU - Overhead
                = 1500 - 66
                = 1434 byte
```

### MTU options trong production

| Underlay MTU | Overlay MTU max | Ghi chú |
|--------------|-----------------|---------|
| 1500 (default) | 1434 | Cấu hình VM MTU = 1434. TCP MSS clamping: 1434 - 40 (IP+TCP) = 1394. |
| 1500 (default) | 1500 | **SAI**: gói full-size fragment hoặc drop. |
| 9000 (jumbo) | 1500 | Overlay giữ 1500 (standard), underlay jumbo, không có drama. |
| 9000 (jumbo) | 8934 | Jumbo end-to-end nếu VM app hỗ trợ. |

### Tính MSS clamping đúng

TCP MSS = MTU - 40 (20 IP + 20 TCP header, không tính option).

Nếu underlay 1500, overlay 1500 (SAI) → client advertise MSS 1460, server cũng MSS 1460, negotiated 1460. Gói TCP payload 1460 byte + 20 IP + 20 TCP = 1500 inner. Encap Geneve → 1566 outer. DF set → drop ở underlay router (nếu underlay MTU 1500).

Fix:
- Option A: VM MTU 1434, MSS clamp 1394.
- Option B: Underlay MTU 9000 (jumbo).
- Option C: Router ICMP "Fragmentation Needed" về → PMTUD discover. Nhưng ICMP thường bị firewall drop → PMTUD black hole.

### MTU probe recipes

```bash
# Từ chassis A tới chassis B (underlay test)
ping -M do -c 3 -s 1472 192.168.10.20   # 1472 + 28 = 1500 outer
ping -M do -c 3 -s 1500 192.168.10.20   # 1500 + 28 = 1528; nếu fail, underlay MTU < 1528

# Từ VM1 tới VM2 (overlay test)
ip netns exec vm1-ns ping -M do -c 3 -s 1406 10.0.0.20   # 1406 + 28 = 1434 overlay MTU
ip netns exec vm1-ns ping -M do -c 3 -s 1472 10.0.0.20   # 1472 + 28 = 1500; SẼ FAIL nếu overlay MTU config sai = 1500

# TCP MSS inspect từ packet capture
sudo tcpdump -ni tap1 "tcp[tcpflags] & tcp-syn != 0" -v | grep "mss"
# nếu mss 1460 mà overlay thực tế 1434 → MSS clamping chưa set
```

### MSS clamping với iptables/nftables

```bash
# Option A: MSS clamp tại host trên tap interface
sudo iptables -t mangle -A FORWARD -o tap1 -p tcp --tcp-flags SYN,RST SYN -j TCPMSS --clamp-mss-to-pmtu

# Option B: hard-code MSS nếu MTU cố định
sudo iptables -t mangle -A FORWARD -o tap1 -p tcp --tcp-flags SYN,RST SYN -j TCPMSS --set-mss 1394
```

## 9.27.5 Fault pattern catalog — common failures cross-host

Table tham khảo 10 fault pattern phổ biến, symptom + root cause + 2-line fix.

| # | Symptom | Root cause | Diagnostic | Fix |
|---|---------|-----------|------------|-----|
| 1 | `ping small OK, big fail` | MTU mismatch | `ping -M do -s N` probe | Set VM MTU = underlay - 66, hoặc jumbo underlay |
| 2 | `ovn-trace OK, ofproto/trace fail` | ovn-controller crash/sync lag | `systemctl status ovn-controller`, `journalctl` | Restart ovn-controller; check SB SSL cert |
| 3 | `ofproto/trace OK, dpif miss` | Datapath stale entry | `dpif/show` miss rate high | `ovs-appctl dpctl/del-flows` force refresh |
| 4 | `bfd/show FLAPPING` | Underlay packet loss > 5% | `ping -c 100 underlay-IP` | Fix underlay QoS/NIC tuning |
| 5 | `Tunnel UP, no traffic` | Firewall drop UDP/6081 | `iptables -L -n \| grep 6081` | Allow UDP/6081 giữa chassis |
| 6 | `Traffic works one-way only` | Asymmetric routing underlay | `traceroute` both ways | Fix underlay routing symmetric |
| 7 | `New VM no network after boot` | ovn-controller chưa bind port | `ovn-sbctl list port_binding \| grep chassis: \[\]` | Check `ovs-vsctl get interface tapN external-ids` |
| 8 | `ACL deny không hoạt động` | conntrack zone collision | `conntrack -L \| head`, `ct_zone` in trace | Use distinct zone cho mỗi LS |
| 9 | `Cross-tenant leak` | Tunnel metadata (VNI) not isolated | Capture Geneve, check TLV values | Kiểm chứng `datapath_id` mapping trong `ovn-sbctl dump-flows` |
| 10 | `Throughput 10% expected` | Slow-path upcall per packet | `dpif/show` coverage, userspace CPU | Check `ovn-controller` log, megaflow cài đặt failing |

## Guided Exercise 2 — Parse Geneve TLV từ packet capture thật

**Mục tiêu:** Đọc outer Geneve header và TLV option từ pcap, xác nhận OVN encode đúng logical port. Skill này cần thiết khi debug inter-chassis mismatch (VNI collision, TLV corruption, version mismatch).

### Bước 3 — Cross-reference với OVN Southbound

```bash
# Lấy datapath UUID cho ls0
ovn-sbctl list datapath_binding | grep -B2 "ls0"
# Lưu ý tunnel_key = VNI của logical datapath

# Lấy port key
ovn-sbctl list port_binding lsp-vm1 | grep tunnel_key
# tunnel_key = 1 (match với TLV 0x80 value)

ovn-sbctl list port_binding lsp-vm2 | grep tunnel_key
# tunnel_key = 2 (match với TLV 0x81 value)
```

TLV value confirmed khớp với với `port_binding.tunnel_key`. OVN encode chính xác.

### Bước 4 — POE "TLV không quan trọng, chỉ VNI là quan trọng"

Giả thuyết ngược:

> *"Chỉ cần VNI (24-bit) là đủ phân biệt tenant. TLV chỉ là metadata cosmetic."*

Phản chứng: VNI xác định logical **datapath** (logical switch hoặc router). TLV 0x80 + 0x81 xác định logical **ports** trong datapath đó. Không có TLV, chassis B decap chỉ biết "gói đến ls0" nhưng không biết "từ lsp-vm1 tới lsp-vm2", phải lookup lại theo inner Eth header (MAC table), tốn 1 extra lookup + MAC table race condition khi VM migrate.

Thử nghiệm: thử disable TLV trong OVN (không có cơ chế disable tuần tự, đây là misconception test). OVN luôn encode TLV. Không thể skip. Bài học: TLV là essential, không optional.

### Bài học

Geneve TLV không phải decoration. Class `0x0102` type `0x80/0x81` là core data path metadata OVN dùng để inject `reg14/reg15` vào pipeline remote chassis, tránh MAC table lookup tốn kém + giải quyết race condition VM migration. Engineer cần biết đọc TLV để debug inter-chassis issue: version mismatch (OVN cũ vs mới dùng TLV type khác), VNI collision (2 logical switch cùng tunnel_key), hoặc corrupted TLV (physical network alters Geneve packet).

## Capstone POE — Benchmark stage-by-stage performance

**Mục tiêu:** Đo băng thông thực tế từng stage riêng biệt để biết bottleneck thực sự ở đâu. POE "OVS datapath là slow" vs "underlay là slow": predict trước, observe sau, explain mismatch.

### Predict (trước khi lab)

Viết ra giấy predict (không xem đáp án) xếp thứ tự từ slowest đến fastest của 5 stage đã encapsulate:

1. Local same-host VM1→VM3 (cùng chassis A): băng thông thực tế ~___ Gbps
2. Local via ovs-userspace (disable kernel datapath): băng thông thực tế ~___ Gbps
3. Cross-host VM1→VM2 với Geneve kernel: băng thông thực tế ~___ Gbps
4. Raw underlay iperf3 trực tiếp giữa chassis (không OVS): băng thông thực tế ~___ Gbps
5. Cross-host VM1→VM2 với DPDK (nếu lab có): băng thông thực tế ~___ Gbps

### Observe

```bash
# Test 1: Same-host VM1 → VM3 (tạo VM3 trên chassis A cùng subnet)
ip netns exec vm3-ns iperf3 -s &
ip netns exec vm1-ns iperf3 -c 10.0.0.30 -t 30 -J > test1.json

# Test 2: (tuỳ chọn — cần config OVS userspace mode) —
# thường chỉ áp dụng nếu lab có DPDK, skip nếu không.

# Test 3: Cross-host Geneve kernel default
ip netns exec vm1-ns iperf3 -c 10.0.0.20 -t 30 -J > test3.json

# Test 4: Raw underlay
iperf3 -s -B 192.168.10.20 &   # trên chassis B
iperf3 -c 192.168.10.20 -t 30 -B 192.168.10.10 -J > test4.json   # trên chassis A

# Test 5: DPDK (skip nếu lab kernel datapath only)
```

### Explain

Thông thường trên lab 10Gbps NIC:

| Test | Expected Gbps | Dominant cost |
|------|---------------|---------------|
| 1 Same-host | ~8-9.5 | veth + br-int local pipeline, fast path |
| 2 OVS userspace | ~3-4 | Context switch kernel ↔ userspace per packet |
| 3 Cross-host Geneve kernel | ~6-8 | Encap + decap CPU + underlay PPS |
| 4 Raw underlay | ~9-9.8 | NIC line rate, no encap |
| 5 DPDK | ~9-9.8 | Bypass kernel, poll mode driver |

Key insight: **Cross-host Geneve (Test 3) chỉ chậm hơn raw underlay (Test 4) ~15-25%** sau khi fast path kích hoạt. Overhead encap + decap + tunnel metadata handling tổng cộng chỉ ~100-200ns per packet trên kernel datapath modern.

**Bottleneck thực tế** trong production cross-host thường không phải OVS datapath. Thường là:
- NIC băng thông thực tế (Stage 1/7): 10Gbps vs 25/100Gbps upgrade.
- Underlay packet loss (Stage 4): 1% loss → TCP băng thông thực tế sập 50%.
- CPU saturation: ovs-vswitchd single-thread cho slow-path nếu megaflow miss rate cao.
- Conntrack table size: per-zone state cần đủ memory.

### Production takeaway

Tuning priorities (cross-host OVN):

1. **NIC + underlay** — biggest bang for buck. 10 → 25Gbps NIC + jumbo frame = 2.5x.
2. **TX/RX offload** — enable checksum/TSO/GRO trên underlay NIC + Geneve offload (kernel 5.0+).
3. **CPU pinning** — pin ovs-vswitchd + PMD threads vào NUMA node của NIC.
4. **DPDK/AF_XDP** — bypass kernel nếu cần > 20Gbps single flow (Phần 16.0-16.2).
5. **OVS datapath tuning** — `max-idle`, `n-revalidator-threads` (Phần 9.26 forensic).

DPDK/AF_XDP không fix underlay hay NIC băng thông thực tế. Chỉ giúp OVS datapath scale ra nhiều core.

## 9.27.6 Điểm cốt lõi cần nhớ

**Một.** Framework 3-tier diagnostic là xương sống debug cross-host: `ovn-trace` (logical ý đồ cấu hình, OVN NB/SB), `ovs-appctl ofproto/trace br-int` (OpenFlow translation, ovn-controller output), `ovs-appctl dpif/dump-flows br-int` (datapath runtime, kernel megaflow cache). Không confuse giữa 3 tầng.

**Hai.** Decision tree: logical OK → check OpenFlow. OpenFlow OK → check datapath. Datapath OK nhưng packet vẫn drop → drop nằm ngoài br-int (tunnel, underlay, remote chassis). Check sequential, không skip tầng.

**Ba.** Geneve outer overhead default OVN = 66 byte (14 + 20 + 8 + 8 base + 16 TLV). Underlay MTU 1500 → max overlay MTU 1434. Jumbo underlay 9000 loại bỏ drama MTU.

**Bốn.** OVN Geneve TLV class `0x0102` type `0x80/0x81` mang logical ingress/egress port (tunnel_key của port_binding). Không phải cosmetic; essential để remote chassis inject `reg14/reg15` vào pipeline recirc.

**Năm.** MSS clamping bắt buộc nếu overlay MTU khác default 1500. `iptables -t mangle ... --clamp-mss-to-pmtu` trên tap interface của host chassis. Không clamp → TCP stuck, PMTUD black hole nếu ICMP drop giữa chassis.

**Sáu.** Fault pattern thường có signature rõ: MTU mismatch (ping small OK big fail), ovn-controller sync lag (ofproto/trace fail), datapath stale (packet counter không tăng), firewall drop UDP 6081 (bfd UP nhưng traffic fail). Catalog 10 pattern ở §9.27.5 phủ 80% incident production.

**Bảy.** Performance bottleneck cross-host OVN thường không phải OVS datapath; thường là NIC + underlay + CPU. DPDK/AF_XDP tăng tốc OVS datapath 3-5x nhưng không fix underlay. Ưu tiên tuning: NIC/underlay trước, OVS datapath sau.

## 9.27.7 Anatomy Template A — `tnl/ports/show` + `bfd/show` cross-host tunnel entry (v3.2 P4 standardize)

**Bối cảnh khi chạy:** First-line debug cross-host packet journey — verify tunnel endpoint setup + BFD liveness trước khi đào sâu `ofproto/trace` hoặc `dpif/dump-flows`. Đây là lệnh entry point Tier 1 trong framework 3-tier (§9.27.2).

**Lệnh gốc:**

```bash
ovs-appctl tnl/ports/show           # liệt kê tunnel port + peer + port number
ovs-appctl bfd/show                 # BFD session state per tunnel
ovs-vsctl list Interface <tunnel>   # options:remote_ip + options:key
```

**9-attribute anatomy:**

| Attribute | Healthy | Warning | Critical | Diagnostic hint |
|-----------|---------|---------|----------|------------------|
| `Listening ports` | Liệt kê geneve_sys_6081 / vxlan_sys_4789 | Port khác 6081/4789 | Không có port = tunnel chưa attach | Check `br-int` + Geneve port binding |
| `Peer <remote_ip>` | Match chassis IP trong SBDB Encap | IP khác chassis known | IP không tồn tại = stale encap | `ovn-sbctl list Encap`; clear row orphan |
| `options:remote_ip` | IPv4 underlay hop-reachable | Different subnet + no route | Unreachable = routing gap | `ip route get <remote_ip>`; underlay investigation |
| `options:key=flow` | flow (OVN dynamic per-LS key) | Fixed key = non-OVN | key=0 = bridging chưa setup | Chỉ OVN mới set flow; legacy setup khác |
| BFD state | `UP` both direction | `INIT` = one-way reach | `DOWN` = tunnel broken | Phần 13.6 §13.6.X BFD state machine |
| BFD `forwarding` | `true` | intermittent `false` = flap | Sustained `false` = underlay issue | Check RFC 5880 phát hiện multiplier |
| BFD `detect_multiplier` | 3-5 default | >10 = slow detect | 1 = over-aggressive false-positive | Balance convergence vs stability |
| BFD `min_rx/tx` | 1000 ms default | <100 ms = control plane stress | >5000 ms = slow failover | Phần 13.6 §13.6.X timing math |
| `Encap.chassis_name` | Match chassis sb hostname | Mismatch = rebind pending | Empty = chassis chưa register | `ovn-sbctl list Chassis` verify |

**Kịch bản bẻ gãy:**

- BFD `UP` + `forwarding=false` lặp lại 2-3 lần/phút: underlay link flap. Check switch uplink, không phải OVN.
- `remote_ip` ping OK + BFD `DOWN`: firewall drop UDP 3784 (BFD) hoặc 6081 (Geneve). `iptables-save` + `nft list ruleset`.
- `Peer` column empty + LS có LSP: chassis chưa được `ovn-sbctl chassis-add` hoặc SBDB propagation lag. Check `ovn-sbctl list Chassis` + Raft health.
- BFD flap mỗi 5 phút đều đặn: CPU steal, chassis VM bị preempted. `dmesg` + `vmstat` + hypervisor scheduler audit.

**Liên kết chéo:** Tier 1 entry (9.27.2) → nếu pass → Tier 2 `ofproto/trace` (§9.27.3) → Tier 3 `dpif/dump-flows` + pcap (§9.27.4). Fault catalog §9.27.5 map symptom lên từng Tier.

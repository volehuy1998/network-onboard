## 16.1.0 Opening drama, from "DPDK compile" to "DPDK tune", a 100x performance gap

In 2015 to 2018, many teams jumped into DPDK with the expectation that "compile DPDK plus run equals 100 Mpps". Reality was very different: a Facebook production team in 2017 reported that **the first measurement reached only 15 percent of line-rate** (15 Gbps on a 100 GbE NIC) even with DPDK enabled. After two weeks of debugging, the team uncovered a long list of issues: (a) hugepages were 2MB but TLB miss rate was high because the mempool was too large, (b) PMD threads were pinned to hyperthread siblings (two threads sharing a single physical core), (c) the NIC was attached to socket 0 but the PMD ran on socket 1, causing cross-NUMA memory access at twice the latency, (d) RSS used a random hash, so reply packets hashed into a different queue and caused cache misses on bidirectional flows, (e) the stat struct was not cache-line-aligned, causing false sharing between counter threads.

After fixing all five issues, the team reached **96 Gbps (96 percent of line-rate)**, 6.4 times the baseline. This is a common gap: DPDK has the potential to reach line-rate, but only if you understand each layer in depth (CPU cache, NUMA, memory channels, NIC queues). Without proper tuning, DPDK delivers only 15 to 30 percent of the maximum performance and burns 100 percent of the CPU for nothing.

Part 16.1 dives into the five factors that decide DPDK performance:

1. **PMD hot loop**: burst size, continuous polling, SPSC ring buffer.
2. **Hugepages 2MB versus 1GB**: TLB pressure plus runtime versus boot allocation.
3. **NUMA topology**: memory channels plus pinning plus the `numactl --hardware` survey.
4. **Cache line alignment**: false sharing detection plus the `__rte_aligned` solution.
5. **Multi-queue scaling**: RSS symmetric hashing plus Flow Director plus queue-CPU pinning.

The part ends with an exercise on OVS-DPDK 1GB hugepages production setup plus a Capstone Predict-Observe-Explain on "tuning DPDK from 15 Gbps up to 96 Gbps: in what order do the five fixes go?".

> **Common misconception:** *"Enable DPDK equals 10x performance automatically."*
>
> **Reality:** out-of-the-box DPDK reaches only 15 to 30 percent of its potential. The performance gap between naive DPDK and tuned DPDK can be 5 to 10 times. Five essential factors (hugepages plus NUMA plus cache alignment plus RSS plus queue pinning) must be configured correctly. Compiling DPDK and running it is the first step, not the last. The Facebook production team spent two weeks of debugging on the first deployment, plan your budget accordingly.

## 16.1.1 The PMD hot loop

The heart of DPDK is the **hot loop** of the PMD thread. Each PMD thread runs an infinite loop with a simple structure:

```c
while (1) {
    nb_rx = rte_eth_rx_burst(port, queue, rx_pkts, BURST_SIZE);
    for (i = 0; i < nb_rx; i++)
        process_packet(rx_pkts[i]);
    rte_eth_tx_burst(port, queue, tx_pkts, nb_tx);
}
```

Three details strongly affect performance:

**(1) Burst size:** the default is 32 packets per call to `rte_eth_rx_burst()`. If the burst is too small (for example 4), the per-call overhead (writing pointers, reading the ring counter) takes up a large fraction of the cycles. If the burst is too large (for example 256), the first packet in the burst waits for the last, which raises latency. 32 is the balance point that suits most Intel and Mellanox NICs.

**(2) Continuous polling without interrupts:** the loop has no `sleep` or `usleep`, the PMD thread spins forever. If there are no packets, `rte_eth_rx_burst()` returns 0 immediately and the loop continues. This achieves the lowest possible latency (no interrupt wait), but it consumes 100 percent of the CPU.

**(3) Per-queue lock-free ring buffer:** each NIC RX queue maps to a descriptor ring array, where each entry is a DMA descriptor pointing to an mbuf in the mempool. The NIC hardware writes to the head of the ring, the PMD reads from the tail of the ring, in a single-producer-single-consumer (SPSC) producer-consumer model that needs no spinlock or mutex. This is why a PMD reaches throughput close to the wire-rate of the hardware.

When scaling out to many queues, each PMD thread services ONE queue (or one group of queues) on ONE core. No two threads read the same queue, which avoids false sharing and contention completely.

## 16.1.2 Hugepages 2MB versus 1GB

The standard page size on Linux x86 is 4KB. With a 1GB mempool, you need 262144 pages of 4KB, far more than the number of TLB (Translation Lookaside Buffer) entries on the CPU (typically 64 to 1536 entries). The result is that every packet access to the mempool has a high probability of a TLB miss, and the CPU has to walk the page table (more than 100 clock cycles per walk).

**2MB hugepages:** a 1GB mempool needs only 512 pages of 2MB, reducing the required TLB entries by a factor of 1024. All modern x86-64 CPUs support 2MB hugepages through the PSE (Page Size Extension) bit. Configuration: add `hugepages=2048` to GRUB (allocates 2048 pages of 2MB equals 4GB at boot) or allocate at runtime through `echo 1024 > /sys/kernel/mm/hugepages/hugepages-2048kB/nr_hugepages`. Runtime allocation can fail if memory is fragmented, boot-time allocation is more reliable.

**1GB hugepages (gigantic pages):** a 1GB mempool needs only 1 page of 1GB, reducing the TLB footprint to nearly zero. Requires CPU support for the PDPE1GB bit (most Xeon E5 and Gold and AMD EPYC have it). Configuration goes ONLY through GRUB: `default_hugepagesz=1G hugepagesz=1G hugepages=4`, runtime allocation is not possible because memory must be contiguous over 1GB. This is the standard for production DPDK environments with high real-world throughput (40 to 100 Gbps).

**When to use which:** 2MB pages for small to medium workloads (10 to 25 Gbps), easier to deploy, and they can be allocated and freed at runtime. 1GB pages for production environments at 40 Gbps or more, with a strict configuration requirement at boot. Many setups use a mix: 4 pages of 1GB for the main mempool plus a few 2MB pages for spare allocations.

## 16.1.3 Memory channels plus NUMA affinity

Xeon Scalable CPUs (Skylake-SP and later) and AMD EPYC have multiple **memory channels** per socket (6 channels for Xeon Gold, 8 channels for EPYC Milan or Genoa). Each channel has a DDR4 or DDR5 DIMM connected in parallel. The memory bandwidth of one socket scales linearly with the number of channels populated with DIMMs: a 6-channel Xeon Gold with all DIMMs populated reaches about 140 GB/s, a 4-channel system reaches only about 90 GB/s.

**Interleaving versus pinning strategy:**
- **Interleaving** (the default of `numactl --interleave=all`) distributes memory pages evenly across all channels. Good for workloads that are not NUMA-aware, achieves a good average bandwidth.
- **Pinning** (using DPDK's `--socket-mem` or `numactl --membind=0`) allocates all memory on socket 0. Good for DPDK because the NIC, the PMD thread, and the memory all sit on the same socket, with no QPI or UPI traffic.

**The `numactl --hardware` survey** reveals the system layout:
```
available: 2 nodes (0-1)
node 0 cpus: 0-19 40-59
node 0 size: 96768 MB
node 1 cpus: 20-39 60-79
node 1 size: 96768 MB
node distances:
node   0   1
  0:  10  21
  1:  21  10
```
The `distances` values are relative numbers (10 means local, 21 means remote across QPI). Remote access is 2.1 times slower than local.

**DPDK flags:** `--lcores='(0-3)@(2-5)'` pins DPDK lcores 0 to 3 onto physical CPUs 2 to 5 (avoiding CPUs 0 and 1, which are reserved for management tasks). `--socket-mem=4096,0` allocates 4GB on socket 0 and 0 on socket 1. `--huge-dir=/mnt/huge` selects the hugepages mount point.

## 16.1.4 Cache line alignment plus false sharing

x86-64 CPUs have a 64-byte cache line. When a core reads or writes a single byte, the entire 64-byte cache line is loaded into or written from the L1 cache of that core. Two cores reading or writing the same cache line cause **false sharing**: the cache coherency protocol (MESI) must invalidate or update the cache between cores, adding more than 100 clock cycles of latency.

In DPDK, false sharing happens when a struct contains fields accessed by multiple PMD threads and is not split across cache lines. Wrong example:
```c
struct pmd_stats {
    uint64_t rx_packets;  // PMD thread 0 writes
    uint64_t tx_packets;  // PMD thread 1 writes
};
```
If both fields land in the same 64-byte cache line, every time thread 0 writes rx_packets, the cache line is invalidated on the core where thread 1 is writing tx_packets. The real-world bandwidth drops noticeably.

**Solution: pad the struct with `RTE_CACHE_LINE_SIZE`:**
```c
struct pmd_stats {
    uint64_t rx_packets __rte_aligned(RTE_CACHE_LINE_SIZE);
    uint64_t tx_packets __rte_aligned(RTE_CACHE_LINE_SIZE);
};
```
The macro `__rte_aligned(64)` forces each field to start on a cache line boundary. This costs extra memory (each uint64 occupies 64 bytes instead of 8), but it removes false sharing.

**False sharing detection tool:** `perf c2c` (cache-to-cache) profiles cache lines that are contended between cores. The output shows which memory address is being false-shared and the access stack trace, which helps identify the struct field that needs padding.

## 16.1.5 Multi-queue plus RSS scaling

**RSS (Receive Side Scaling)** is a NIC feature that distributes incoming packets from many flows across multiple RX queues based on a hash of the tuple (src IP, dst IP, src port, dst port, protocol). Each RX queue is serviced by a dedicated PMD thread, scaling linearly with the number of CPU cores.

**Symmetric hashing:** requires that the inbound packet and the reply packet hash to the same queue (and therefore the same CPU core). This keeps a connection local: TCP state stays in the cache, the conntrack entry stays on the same core, and traffic does not cross QPI. Configuration: use the standard 40-byte "Microsoft symmetric" hash key instead of the default random hash key.

**Multi-queue NICs (Intel E810 with 256 queues):** high-end NICs support hundreds of queues, enough to distribute work across 64 or more CPU cores. Configuration: `ethtool -L eth0 combined 64` sets 64 queues, then DPDK detects them and assigns one PMD thread per queue.

**Strategies for pinning queues to CPUs:**
- **One-to-one pinning:** queue 0 to CPU 2, queue 1 to CPU 3, and so on. Simple, optimal for cache locality.
- **Multi-thread pool:** 2 threads service 8 queues in round-robin. Tolerates uneven load when one queue is light, at the cost of multi-thread overhead.

**Flow Director for specific flow steering:** a more advanced feature than RSS, it lets the team define rules such as "a packet with dst IP 10.0.1.5 and port 443 goes to queue 10". Useful for VM mobility scenarios (keeping the same CPU servicing a flow when the VM migrates), or for isolating a priority flow (for example, control-plane traffic always goes to queue 0, dedicated to a control-plane core).

A benchmark on an Intel E810 100GbE plus 32-core Xeon Platinum: symmetric RSS with 32 queues times 32 PMDs reaches 148.8 Mpps line-rate for 64-byte packets, very close to the theoretical limit. Flow Director adds about 100 ns of overhead per packet, but it provides better quality of service when traffic mixes many classes.

**Steps:**

**1.** Check CPU topology to identify NUMA nodes plus siblings:

```bash
[host]# lscpu --extended
CPU NODE SOCKET CORE L1d:L1i:L2:L3 ONLINE
0   0    0      0    0:0:0:0     yes
1   0    0      0    0:0:0:0     yes    # HT sibling of CPU 0
```

Identify: CPU 2 (physical core 1 on socket 0), CPU 4 (core 2), CPU 6 (core 3). Skip CPUs 1, 3, 5, 7 (HT siblings).

**2.** Configure grub with 1GB hugepages plus isolcpus plus IOMMU:

```bash
[host]# sed -i 's/GRUB_CMDLINE_LINUX_DEFAULT="\(.*\)"/GRUB_CMDLINE_LINUX_DEFAULT="\1 hugepagesz=1G hugepages=4 default_hugepagesz=1G intel_iommu=on iommu=pt isolcpus=2,4"/' /etc/default/grub
[host]# update-grub
[host]# reboot
```

**3.** Verify after reboot:

```bash
[host]$ grep HugePages /proc/meminfo
HugePages_Total:       4
HugePages_Free:        4
Hugepagesize:    1048576 kB

[host]$ cat /sys/devices/system/cpu/isolated
2,4

[host]$ cat /proc/cmdline | tr ' ' '\n' | grep -E 'huge|iommu|isolcpus'
hugepagesz=1G
hugepages=4
default_hugepagesz=1G
intel_iommu=on
iommu=pt
isolcpus=2,4
```

**4.** Configure OVS-DPDK:

```bash
[host]# ovs-vsctl set Open_vSwitch . other_config:dpdk-init=true
[host]# ovs-vsctl set Open_vSwitch . other_config:dpdk-socket-mem=1024,0
[host]# ovs-vsctl set Open_vSwitch . other_config:pmd-cpu-mask=0x14
# 0x14 = binary 00010100, bits 2 and 4 set, CPUs 2 and 4
[host]# systemctl restart openvswitch-switch
```

**5.** Verify the PMD threads:

```bash
[host]# ovs-appctl dpif-netdev/pmd-stats-show | grep 'core_id'
pmd thread numa_id 0 core_id 2
pmd thread numa_id 0 core_id 4
```

```bash
[host]# top -bn1 -p $(pgrep ovs-vswitchd)
%CPU: 200  (2 PMD threads, each 100%)
```

**Expected outcome:**
- 1GB hugepages successfully allocated.
- 2 PMD threads pinned to CPU 2 and 4.
- 100 percent CPU usage on the PMD cores (constant poll mode).
- Bridge `br0-dpdk` with `datapath_type=netdev` is functioning.

**Lessons learned:**

First, **1GB hugepages** reduce TLB pressure significantly. A 1GB page covers the same memory area as 512 of the 2MB pages, or 262144 of the 4KB pages. This is critical for mempool sizes above 1GB.

Second, **avoid HT siblings**. HyperThreading means two logical CPUs share one physical core. If you pin a PMD to one HT sibling and a different workload to the other, you get thread contention. Always pin a PMD to a physical core alone.

Third, **isolcpus removes cores from the scheduler**, which guarantees that the PMD is never preempted by an unrelated workload. Combined with cgroup cpuset for additional isolation, userspace applications also do not run on the isolated cores.


## 16.1.6 DPDK advanced tuning compared with OVS kernel and eBPF

Once you understand the five DPDK tuning factors (hot loop plus hugepages plus NUMA plus cache plus RSS), the question comes back: **can OVS kernel plus eBPF reach DPDK performance with less complexity?**

| Dimension | OVS kernel plus megaflow | OVS-DPDK tuned | eBPF XDP native |
|-----------|-------------------------|----------------|-----------------|
| Real-world bandwidth single-core | 1.5 to 2 Mpps | 15 Mpps (tuned) | 10 to 15 Mpps |
| Real-world bandwidth 4-core | 5 to 8 Mpps | 60 Mpps (tuned) | 40 Mpps |
| P99 latency | 20 us | 2 us | 5 us |
| Hugepages required | No | Yes (2MB or 1GB) | No |
| IOMMU required | No | Yes (VT-d) | No |
| isolcpus required | No | Yes (critical) | Optional (recommended) |
| Kernel features preserved | All | None (bypass) | All (XDP_PASS fallback) |
| Deployment complexity | Low | **Very High** | Medium |
| Debugging tools | `ofproto/trace`, `coverage/show`, `upcall/show` | `testpmd`, `rte_stats`, `dpdk-test-pmd-perf` | `bpftool`, `bpftrace`, `perf c2c` |
| CPU cost (idle) | Near-zero | 100 percent (always poll) | Near-zero (interrupt-driven) |

**When to choose what:**

- **OVS kernel plus megaflow** for the default hypervisor workload. Enough for K8s pods (less than 10 Gbps per node typical) and VM overlay. 95 percent of use cases do not need DPDK.
- **OVS-DPDK tuned** when throughput goes **above 25 Gbps per node** plus the workload is dedicated (NFV VNF, HFT gateway, CDN edge). Only deploy after measuring that the kernel baseline is not enough.
- **eBPF XDP** when you need custom logic plus real-world bandwidth of 10 to 40 Gbps plus you want to keep kernel features (iptables, tc). Cilium is a production-ready example.

### Cross-cut with Part 9.26 (OVS revalidator storm)

Part 9.26 analyzed how a stale ukey leak can cause a revalidator CPU spike. Does the DPDK datapath have a similar problem?

**Yes, but the mechanism is different:**

- **OVS kernel:** stale ukey leak in the userspace revalidator thread (ofproto-dpif-upcall.c). Fix: commit `180ab2fd635e` (Part 9.26).
- **OVS-DPDK:** the equivalent is "stale flows in EMC" (Exact Match Cache, the largest cache of the DPDK datapath, 4096 entries by default). When flow churn is high, the EMC fills with stale flows. Solution: monitor through `ovs-appctl dpif-netdev/pmd-stats-show`, tune `emc-insert-min-probability` (default 1 in 100).
- **Common lesson:** every datapath cache needs revalidation plus an eviction policy. DPDK is not immune to the "stale cache leak" class of bug, only the layer is different (userspace thread versus PMD-internal struct).

## 16.1.7 Key takeaways

**Six key takeaways:**

1. **The PMD hot loop** is the heart of DPDK: `while(1) { rx_burst, process, tx_burst }`. 32 packets per burst is the sweet spot. Poll mode burns 100 percent CPU per port, with no sleep or interrupt.
2. **Hugepages** are mandatory: 2MB for 10 to 25 Gbps workloads (runtime allocation), 1GB for 40 Gbps or more in production (boot-time allocation through GRUB). They reduce TLB misses from millions per second down to nearly zero.
3. **NUMA awareness** is critical: NIC on socket X, PMD on socket X, hugepages allocated from socket X. Wrong NUMA pinning cuts real-world bandwidth by 30 to 50 percent.
4. **Cache line alignment**: structs shared between PMD threads must use `__rte_aligned(RTE_CACHE_LINE_SIZE)` (64 bytes). Detect false sharing with `perf c2c`. DPDK aligns its core structs by default, the problem is usually in custom application code.
5. **RSS symmetric hashing** plus 1-to-1 queue-CPU pinning: the standard 40-byte "Microsoft symmetric" hash key keeps bidirectional flows on the same queue. Flow Director steers specific flows (VM migration, control-plane isolation).
6. **Tune step by step, measure after each step**: the Facebook 2017 case study went from 15 Gbps baseline to 96 Gbps after five fixes (hugepages, NUMA pin, cache align, symmetric RSS, HT avoidance). There is no silver bullet single fix.

### Capstone POE: Debug a DPDK production deployment, 15 Gbps to 96 Gbps, in what order do the five fixes go?

> **Lab verification status:** the POE is based on a publicly documented pattern (Facebook 2017 engineering blog, Intel tuning whitepapers). Pending the user having 40 to 100 GbE hardware to reproduce the lab.

**Goal:** evaluate prioritization skill when debugging a DPDK production performance gap. Apply the five tuning principles in order of impact and complexity trade-off.

**Scenario:** Organization X deploys OVS-DPDK for an NFV workload (virtual firewall plus load balancer). Server: 2-socket Xeon Gold 6246R (32 cores times 2), NIC Intel E810-CQDA2 100 GbE (2 ports), RAM 256 GB. The baseline benchmark with `testpmd` or `iperf3` reaches only **15 Gbps per port** (15 percent of line-rate). The CTO assigns the DevOps team one week to fix the deployment to **at least 80 Gbps**.

The team enumerates five suspected issues (with unknown impact and not knowing which is the largest bottleneck):

A. **2MB hugepages** instead of 1GB, TLB miss rate possibly high.
B. **PMD pinned to a HyperThread sibling** (CPU 0 and 1 on the same core), causing contention.
C. **NIC on socket 0, PMD on socket 1**, causing cross-NUMA memory access.
D. **RSS random hash**, causing bidirectional flows to scatter across queues.
E. **Stat struct not cache-aligned**, causing false sharing between PMD counters.

**Predict:** propose an order of fixes from high-impact-low-cost down to low-impact-high-cost. After each fix, measure real-world bandwidth to confirm progression.

**Observe (reasoning):**

**Order recommendation:**

**Fix 1, NUMA pinning (C):** the largest impact (2x latency cross-NUMA). Low cost (only DPDK config `--socket-mem` plus grub `isolcpus` plus `pmd-cpu-mask`). Measure: if the NIC is on socket 0, pin the PMD onto CPU 2, 4, 6 (socket 0 physical cores). **Expected gain: 15 to 35 Gbps (plus 133 percent).**

**Fix 2, HyperThread avoidance (B):** medium impact (HT sibling contention can cost 30 to 50 percent of performance). Low cost (only identify physical cores through `lscpu --extended` plus update `pmd-cpu-mask`). Skip CPU 1, 3, 5 (HT siblings of 0, 2, 4). **Expected gain: 35 to 50 Gbps (plus 43 percent).**

**Fix 3, 1GB hugepages (A):** medium-high impact (TLB pressure drops about 1000x). Medium cost (must reboot with `hugepagesz=1G default_hugepagesz=1G` in grub). **Expected gain: 50 to 75 Gbps (plus 50 percent).**

**Fix 4, RSS symmetric hash (D):** medium impact for bidirectional-flow workloads. Medium cost (ethtool hash configuration plus NIC feature support). **Expected gain: 75 to 90 Gbps (plus 20 percent).**

**Fix 5, cache alignment (E):** low impact (about 5 to 10 percent for counter-heavy code). High cost (audit the source of the custom application plus profile with `perf c2c` plus rebuild). **Expected gain: 90 to 96 Gbps (plus 7 percent).**

**Explain:** rationale for the order:

- **C (NUMA) first** because the memory access pattern affects every subsequent optimization. If the PMD is cross-NUMA, hugepages plus cache alignment cannot save it.
- **B (HT) second** because it relates to CPU resource sharing, fix this so the next optimizations are correct.
- **A (hugepages) third** because it requires a reboot (medium cost). Place it after the two hot-patchable fixes.
- **D (RSS) fourth** because it is specific to bidirectional traffic patterns. Workload-dependent.
- **E (cache align) last** because the impact is the lowest and the cost is the highest (source code audit).

**Falsification test:** try the reverse order (E first, C last). Expectation: E alone gains less than 5 percent, not worth the cost. Confirms the pattern: "fix high-impact-low-cost first" is a universal principle for performance tuning.

**Finish:** universal principles for performance engineering:

1. **Measure first, tune after**, never tune blind.
2. **Fix one at a time**, if you fix multiple at once, you cannot tell which one created the delta.
3. **Prioritize by impact-over-cost ratio**, high impact and low cost first.
4. **Re-measure between fixes**, confirm progression and detect regressions.
5. **Stop when you reach the target**, 80 percent of real-world bandwidth may be the diminishing-return point, more fixes are not worth it.

This pattern applies to all performance work, not only DPDK. Database tuning (query plan, index, memory tuning), JVM tuning (profiler, GC tuning, heap size), application tuning (algorithm, caching, concurrency) all follow the same methodology.

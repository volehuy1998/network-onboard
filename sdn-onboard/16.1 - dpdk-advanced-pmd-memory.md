## 16.1.1 Vòng lặp nóng của PMD

Trái tim của DPDK là **vòng lặp nóng (hot loop)** của thread PMD. Mỗi thread PMD chạy vòng lặp vô hạn với cấu trúc đơn giản:

```c
while (1) {
    nb_rx = rte_eth_rx_burst(port, queue, rx_pkts, BURST_SIZE);
    for (i = 0; i < nb_rx; i++)
        process_packet(rx_pkts[i]);
    rte_eth_tx_burst(port, queue, tx_pkts, nb_tx);
}
```

Ba chi tiết ảnh hưởng mạnh đến hiệu năng:

**(1) Kích thước burst:** mặc định là 32 packet per cuộc gọi `rte_eth_rx_burst()`. Nếu burst quá nhỏ (ví dụ 4), overhead của mỗi lần gọi (ghi lại pointer, đọc counter ring) chiếm tỷ lệ lớn. Nếu burst quá lớn (ví dụ 256), packet đầu tiên trong burst bị giữ lại đợi packet cuối — tăng độ trễ. 32 là điểm cân bằng phù hợp cho hầu hết NIC Intel/Mellanox.

**(2) Thăm dò liên tục không ngắt:** vòng lặp không có `sleep` hay `usleep` — thread PMD quay vòng vô hạn. Nếu không có packet, `rte_eth_rx_burst()` trả về 0 ngay lập tức và vòng lặp tiếp tục. Điều này đạt độ trễ thấp nhất (không phải đợi ngắt) nhưng tiêu thụ 100% CPU.

**(3) Ring buffer không khóa per-queue:** mỗi RX queue của NIC ánh xạ tới một mảng descriptor ring, mỗi entry là một descriptor DMA trỏ tới một mbuf trong mempool. NIC hardware ghi vào đầu ring, PMD đọc từ đuôi ring — mô hình producer-consumer single-producer-single-consumer (SPSC), không cần khóa spinlock hay mutex. Đây là lý do PMD đạt hiệu năng gần bằng wire-rate của hardware.

Khi mở rộng ra nhiều queue, mỗi thread PMD điều phối MỘT queue (hoặc một nhóm queue) trên MỘT core. Không có hai thread đọc cùng một queue — tránh false sharing và tranh chấp hoàn toàn.

## 16.1.2 Hugepages 2MB so với 1GB

Page chuẩn trên Linux x86 là 4KB. Với mempool cỡ 1GB, cần 262144 page × 4KB — vượt xa số entry TLB (Translation Lookaside Buffer) của CPU (thường 64-1536 entry). Hệ quả: mỗi packet truy cập mempool có xác suất cao bị TLB miss, CPU phải thực hiện duyệt page table (hơn 100 chu kỳ đồng hồ).

**Hugepages 2MB:** 1GB mempool chỉ cần 512 page × 2MB — giảm 1024 lần số entry TLB cần thiết. Tất cả CPU x86-64 hiện đại hỗ trợ hugepages 2MB qua bit PSE (Page Size Extension). Cấu hình: thêm `hugepages=2048` vào GRUB (cấp phát 2048 × 2MB = 4GB lúc boot) hoặc cấp phát runtime qua `echo 1024 > /sys/kernel/mm/hugepages/hugepages-2048kB/nr_hugepages`. Cấp phát runtime có thể thất bại nếu bộ nhớ đã phân mảnh — cấp phát lúc boot đảm bảo hơn.

**Hugepages 1GB (gigantic pages):** 1GB mempool chỉ cần 1 page × 1GB — giảm dấu chân TLB về gần zero. Yêu cầu CPU hỗ trợ bit PDPE1GB (hầu hết Xeon E5/Gold và AMD EPYC đều có). Cấu hình CHỈ qua GRUB: `default_hugepagesz=1G hugepagesz=1G hugepages=4` — không thể cấp phát runtime vì bộ nhớ phải liên tục 1GB. Đây là chuẩn cho DPDK môi trường production throughput cao (40-100 Gbps).

**Khi nào dùng cái nào:** 2MB cho workload nhỏ-trung (10-25 Gbps), dễ triển khai, có thể cấp phát/giải phóng runtime. 1GB cho môi trường production 40+ Gbps, yêu cầu cấu hình chặt chẽ lúc boot. Nhiều thiết lập dùng hỗn hợp: 4 × 1GB cho mempool chính + một số 2MB dự phòng.

## 16.1.3 Kênh bộ nhớ + ái lực NUMA

CPU Xeon Scalable (Skylake-SP trở đi) và AMD EPYC có nhiều **kênh bộ nhớ** per socket (6 kênh cho Xeon Gold, 8 kênh cho EPYC Milan/Genoa). Mỗi kênh có một DIMM DDR4/DDR5 kết nối song song. Băng thông bộ nhớ của một socket tỷ lệ thuận với số kênh đã cắm DIMM: 6-kênh Xeon Gold đầy DIMM đạt khoảng 140 GB/s, 4-kênh chỉ khoảng 90 GB/s.

**Chiến lược xen kẽ so với ghim:**
- **Xen kẽ (interleaving)** (mặc định của `numactl --interleave=all`) phân phối trang bộ nhớ đều qua tất cả kênh. Tốt cho workload không biết NUMA, đạt băng thông trung bình tốt.
- **Ghim (pinning)** (dùng `--socket-mem` của DPDK hoặc `numactl --membind=0`) cấp phát tất cả bộ nhớ trên socket 0. Tốt cho DPDK vì NIC, thread PMD, và bộ nhớ đều cùng socket — không có traffic QPI/UPI.

**Khảo sát bằng `numactl --hardware`** cho biết cấu hình hệ thống:
```
available: 2 nodes (0-1)
node 0 cpus: 0-19 40-59
node 0 size: 96768 MB
node 1 cpus: 20-39 60-79
node 1 size: 96768 MB
node distances:
node   0   1
  0:  10  21
  1:  21  10
```
Giá trị `distances` là số tương đối (10 = cục bộ, 21 = từ xa qua QPI). Truy cập từ xa chậm gấp 2.1 lần cục bộ.

**Cờ DPDK:** `--lcores='(0-3)@(2-5)'` ghim lcore 0-3 của DPDK vào CPU 2-5 vật lý (tránh CPU 0-1 dành cho tác vụ quản trị). `--socket-mem=4096,0` cấp phát 4GB trên socket 0, 0 trên socket 1. `--huge-dir=/mnt/huge` chỉ định điểm gắn hugepages.

## 16.1.4 Căn hàng cache line + false sharing

CPU x86-64 có cache line 64 bytes. Khi một core đọc/ghi một byte, toàn bộ cache line 64 byte được tải/ghi vào L1 cache của core đó. Hai core đọc/ghi cùng cache line → **false sharing**: giao thức cache coherency (MESI) phải vô hiệu hóa/cập nhật cache giữa các core, gây độ trễ hơn 100 chu kỳ đồng hồ.

Trong DPDK, false sharing xảy ra khi struct chứa field được truy cập bởi nhiều thread PMD không được tách cache line. Ví dụ sai:
```c
struct pmd_stats {
    uint64_t rx_packets;  // thread PMD 0 ghi
    uint64_t tx_packets;  // thread PMD 1 ghi
};
```
Nếu cả hai field nằm trong cùng cache line 64 byte, mỗi lần thread 0 ghi rx_packets, cache line bị vô hiệu hóa trên core của thread 1 đang ghi tx_packets. Throughput giảm đáng kể.

**Giải pháp: đệm struct với `RTE_CACHE_LINE_SIZE`:**
```c
struct pmd_stats {
    uint64_t rx_packets __rte_aligned(RTE_CACHE_LINE_SIZE);
    uint64_t tx_packets __rte_aligned(RTE_CACHE_LINE_SIZE);
};
```
Macro `__rte_aligned(64)` buộc mỗi field bắt đầu ở ranh giới cache line. Tốn thêm bộ nhớ (mỗi uint64 chiếm 64 byte thay vì 8), nhưng loại bỏ false sharing.

**Công cụ phát hiện false sharing:** `perf c2c` (cache-to-cache) lập hồ sơ cache line bị tranh chấp giữa core. Output cho thấy địa chỉ bộ nhớ nào đang bị false share và dấu vết ngăn xếp truy cập — giúp xác định field struct cần đệm.

## 16.1.5 Đa queue + mở rộng RSS

**RSS (Receive Side Scaling)** là tính năng NIC phân phối packet đến từ nhiều luồng qua nhiều RX queue dựa trên hash của tuple (src IP, dst IP, src port, dst port, protocol). Mỗi RX queue được một thread PMD riêng xử lý → mở rộng tuyến tính theo số CPU core.

**Hash đối xứng (symmetric hashing):** yêu cầu packet đi lên và packet trả lại được hash về cùng queue (và do đó cùng CPU core). Điều này giữ kết nối cục bộ: trạng thái TCP trong cache, entry conntrack ở cùng một core, không phải đi qua QPI. Cấu hình: dùng hash key "Microsoft symmetric" 40-byte chuẩn thay vì hash key ngẫu nhiên mặc định.

**NIC đa queue (Intel E810 với 256 queue):** NIC cao cấp hỗ trợ hàng trăm queue, đủ để phân phối cho 64+ CPU core. Cấu hình: `ethtool -L eth0 combined 64` đặt 64 queue, rồi DPDK sẽ phát hiện và cấp phát một thread PMD mỗi queue.

**Chiến lược ghim queue vào CPU:**
- **Ghim 1-1:** queue 0 → CPU 2, queue 1 → CPU 3, ... Đơn giản, tối ưu cache cục bộ.
- **Bể nhiều thread:** 2 thread xử lý 8 queue theo xoay vòng. Chịu tải tốt hơn khi một queue ít traffic, nhưng overhead đa thread.

**Flow Director cho chỉ đạo flow cụ thể:** tính năng cao cấp hơn RSS — cho phép đội ngũ định nghĩa quy tắc "packet có dst IP 10.0.1.5 port 443 → queue 10". Dùng cho trường hợp di chuyển VM (giữ nguyên CPU xử lý flow khi VM di chuyển), hoặc cho cô lập flow ưu tiên (ví dụ traffic control plane luôn vào queue 0 dành riêng cho core control plane).

Benchmark trên Intel E810 100GbE + 32 core Xeon Platinum: RSS đối xứng với 32 queue × 32 PMD đạt 148.8 Mpps line-rate cho gói 64-byte — gần hoàn hảo so với giới hạn lý thuyết. Flow Director thêm khoảng 100 ns overhead per packet nhưng cho chất lượng dịch vụ tốt hơn với traffic đa loại.

### ▶ Guided Exercise 1: Setup OVS-DPDK với 1GB hugepages + 2 PMD threads

> **Trạng thái lab verification:** Output dựa trên OVS-DPDK tuning guide + Intel performance reports; chờ verify lab thực.

**Mục đích:** Production-grade DPDK setup. 1GB hugepages minimize TLB misses cho large mempools, 2 PMD threads trên isolated CPUs ensure consistent polling performance.

**Chuẩn bị:**
- Physical server với 2-socket CPU, 16+ GB RAM, NIC support DPDK
- Ubuntu 22.04 LTS, grub access
- Verify CPU topology: `lscpu --extended`

**Mô hình lab:**

```
┌──────────────────────────────────────────────────┐
│  2-socket server                                 │
│                                                  │
│  Socket 0 (NUMA 0):       Socket 1 (NUMA 1):     │
│  ┌──────────────────┐     ┌──────────────────┐   │
│  │ CPU 0,1 (HT)     │     │ CPU 8,9 (HT)     │   │
│  │ CPU 2,3 (HT)←PMD │     │ CPU 10,11 (HT)   │   │
│  │ CPU 4,5 (HT)←PMD │     │ CPU 12,13 (HT)   │   │
│  │ CPU 6,7 (HT)     │     │ CPU 14,15 (HT)   │   │
│  │                  │     │                  │   │
│  │ 4× 1GB hugepages │     │ 0 hugepages      │   │
│  │ on socket 0      │     │ (NIC on socket 0)│   │
│  └──────────────────┘     └──────────────────┘   │
│         ▲                                        │
│         │ PCIe Gen3 x8                           │
│    ┌────┴─────┐                                  │
│    │ NIC      │                                  │
│    │ 40 Gbps  │                                  │
│    └──────────┘                                  │
└──────────────────────────────────────────────────┘
```

**Bước thực hiện:**

**1.** Check CPU topology để identify NUMA nodes + siblings:

```bash
[host]# lscpu --extended
CPU NODE SOCKET CORE L1d:L1i:L2:L3 ONLINE
0   0    0      0    0:0:0:0     yes
1   0    0      0    0:0:0:0     yes    # HT sibling of CPU 0
2   0    0      1    1:1:1:0     yes
3   0    0      1    1:1:1:0     yes    # HT sibling of CPU 2
...
```

Identify: CPU 2 (physical core 1 on socket 0), CPU 4 (core 2), CPU 6 (core 3). Skip CPUs 1,3,5,7 (HT siblings).

**2.** Configure grub với 1GB hugepages + isolcpus + IOMMU:

```bash
[host]# sed -i 's/GRUB_CMDLINE_LINUX_DEFAULT="\(.*\)"/GRUB_CMDLINE_LINUX_DEFAULT="\1 hugepagesz=1G hugepages=4 default_hugepagesz=1G intel_iommu=on iommu=pt isolcpus=2,4"/' /etc/default/grub
[host]# update-grub
[host]# reboot
```

**3.** Verify sau reboot:

```bash
[host]$ grep HugePages /proc/meminfo
HugePages_Total:       4
HugePages_Free:        4
Hugepagesize:    1048576 kB

[host]$ cat /sys/devices/system/cpu/isolated
2,4

[host]$ cat /proc/cmdline | tr ' ' '\n' | grep -E 'huge|iommu|isolcpus'
hugepagesz=1G
hugepages=4
default_hugepagesz=1G
intel_iommu=on
iommu=pt
isolcpus=2,4
```

**4.** Configure OVS-DPDK:

```bash
[host]# ovs-vsctl set Open_vSwitch . other_config:dpdk-init=true
[host]# ovs-vsctl set Open_vSwitch . other_config:dpdk-socket-mem=1024,0
[host]# ovs-vsctl set Open_vSwitch . other_config:pmd-cpu-mask=0x14
# 0x14 = binary 00010100 = bits 2 và 4 set → CPUs 2 và 4
[host]# systemctl restart openvswitch-switch
```

**5.** Verify PMD threads:

```bash
[host]# ovs-appctl dpif-netdev/pmd-stats-show | grep 'core_id'
pmd thread numa_id 0 core_id 2
pmd thread numa_id 0 core_id 4
```

**6.** Check CPU usage — PMD threads hiển thị 100%:

```bash
[host]# top -bn1 -p $(pgrep ovs-vswitchd)
%CPU: 200  (2 PMD threads, each 100%)
```

**Output mong muốn:**
- 1GB hugepages successfully allocated
- 2 PMD threads pinned to CPU 2,4
- 100% CPU usage on PMD cores (poll mode constant)
- Bridge `br0-dpdk` với `datapath_type=netdev` functioning

**Bài học học được:**

Thứ nhất, **1GB hugepages** giảm TLB pressure đáng kể — 1GB page cover cùng memory area với 512 của 2MB page, 262144 của 4KB page. Critical cho mempool sizes > 1GB.

Thứ hai, **Avoid HT siblings**: HyperThreading means 2 logical CPUs share 1 physical core. Pin PMD to một HT sibling, workload khác đến sibling kia → thread contention. Always pin PMD to physical core alone.

Thứ ba, **isolcpus remove cores khỏi scheduler** — guarantee PMD không bị preempted by unrelated workload. Combined với cgroup cpuset cho additional isolation (userspace apps cũng không run trên isolated cores).

**Cleanup:**

```bash
[host]# ovs-vsctl set Open_vSwitch . other_config:dpdk-init=false
[host]# systemctl restart openvswitch-switch
# Remove grub args if reverting completely (reboot required)
```

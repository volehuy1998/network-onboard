## Learning objectives

This part is a pedagogical **tutorial gradient**. After finishing it, the learner can:

1. **Apply, Level 1.** Use `ovn-trace` on a simple packet inside one Logical Switch to view the logical pipeline output (hello-world).
2. **Analyze, Level 2.** Read trace output through a multi-table pipeline that contains a stateful ACL, identifying which flow each stage matched and what action ran.
3. **Analyze, Level 3.** Trace a cross-subnet packet through a Logical Router (LR), including ARP resolve, `dec_ttl`, and next-hop selection.
4. **Apply, Level 4.** Combine `ovn-trace` (logical view) with `ofproto/trace` (OpenFlow view) to debug a multichassis Geneve tunnel.
5. **Synthesize, Level 5.** Use `ovn-detrace` chained as `ofproto/trace | ovn-detrace` to correlate physical OpenFlow output back to its OVN logical context for a production incident.
6. **Evaluate.** Design a trace scenario for an unknown incident, choose the appropriate level, and explain the rationale.

> **Pedagogical philosophy.** Each level unlocks one new capability that the previous level did not have. The learner should not skip levels. If you are stuck at Level 4 (Geneve), the cause is usually that Levels 2 and 3 (logical pipeline) are not yet solid. Go back and study them again.

---

## 20.7.1 Overview of the five levels

| Level | Main tool | Scope | When to use |
|-------|-----------|-------|-------------|
| L1 | `ovn-trace` | One simple LS, two LSP | Learning logical flow trace concepts |
| L2 | `ovn-trace --detailed` | LS with stateful ACL | Debugging ACL match priority |
| L3 | `ovn-trace --ovs` | LS plus LR cross-subnet | Routing and ARP resolve |
| L4 | `ovn-trace` plus `ofproto/trace` | Cross-chassis Geneve | Tunnel encap, `Port_Binding` |
| L5 | `ovn-detrace` chain | Production incident | Forensic timeline and correlation |

Every level shares the same four-part structure:

1. **Context and setup.** A minimal scenario plus the preparation steps.
2. **Command.** The main command and the important flags.
3. **Anatomy of the output.** Reading the result section by section.
4. **▶ Exercise.** Hands-on practice with expected output and lessons.

## 20.7.2 Level 1, hello-world `ovn-trace`

### 20.7.2.1 Context and setup

**Scenario.** One Logical Switch `ls-hello`, two LSP `vm1` and `vm2`. VM1 pings VM2.

**Minimal setup:**

```bash
[host]# ovn-nbctl ls-add ls-hello
[host]# ovn-nbctl lsp-add ls-hello vm1
[host]# ovn-nbctl lsp-set-addresses vm1 "00:00:00:00:00:01 10.0.0.1"
[host]# ovn-nbctl lsp-add ls-hello vm2
[host]# ovn-nbctl lsp-set-addresses vm2 "00:00:00:00:00:02 10.0.0.2"
```

(You do not need to run a real packet. `ovn-trace` simulates against the NBDB intent.)

### 20.7.2.2 Command

```bash
[host]# ovn-trace ls-hello \
        'inport == "vm1" && eth.src == 00:00:00:00:00:01 && \
         eth.dst == 00:00:00:00:00:02 && ip4.src == 10.0.0.1 && \
         ip4.dst == 10.0.0.2 && ip.ttl == 64'
```

General syntax: `ovn-trace <datapath-name> <flow-expression>`.

`<flow-expression>` describes the packet using the **OVN match expression** syntax (see the `ovn-sb(5)` man page, "Logical Flow" section). At minimum, supply `inport` plus L2 fields (`eth.src` and `eth.dst`) plus L3 fields (`ip4` or `ip6`).

### 20.7.2.3 Anatomy of the output

```
# ip,reg14=0x1,vlan_tci=0x0000,dl_src=00:00:00:00:00:01,dl_dst=00:00:00:00:00:02,...
ingress(dp="ls-hello", inport="vm1")
---------------------------------------
 0. ls_in_port_sec_l2 (northd.c:5614): inport == "vm1" && eth.src == {00:00:00:00:00:01}, priority 50, uuid abcd1234
    next;
 1. ls_in_port_sec_ip (northd.c:5210): ..., priority 90, uuid ...
    next;
27. ls_in_l2_lkup (northd.c:8127): eth.dst == 00:00:00:00:00:02, priority 50, uuid efgh5678
    outport = "vm2";
    output;

egress(dp="ls-hello", inport="vm1", outport="vm2")
---------------------------------------
 0. ls_out_pre_acl (...): ..., priority 0
    next;
 9. ls_out_port_sec_l2 (...): outport == "vm2" && eth.dst == {00:00:00:00:00:02}, priority 50
    output;
        /* output to "vm2", type "" */
```

| Output section | Meaning | Warning sign |
|----------------|---------|--------------|
| `# ip,reg14=...` | Packet header parsed from the flow expression | Wrong header means the expression syntax is wrong |
| `ingress(dp=..., inport=...)` | Start of the datapath ingress pipeline | Datapath name must match NBDB |
| Each line `N. ls_in_*` | A pipeline stage (see Part 13.2 LS pipeline) | A skipped stage means no Logical_Flow matched |
| `priority N, uuid ...` | Priority and UUID of the matched Logical_Flow | Multiple flows at the same priority may conflict |
| Action `next;` | Goto the next stage | Action `drop;` means the packet is dropped |
| `outport = "vm2"; output;` | Set outport and emit | Start of the egress pipeline |
| `egress(dp=..., outport=...)` | Start of egress | Egress usually has fewer stages than ingress |
| `/* output to "vm2", type "" */` | Trailing comment, target is physical | Type `""` means a VIF (LSP local or remote) |

### 20.7.2.4 ▶ Exercise L1

**Goal.** Verify your understanding of the structure of `ovn-trace` output.

**Questions:**

1. How many `ls_in_*` stages are hit in the ingress trace?
2. Which stage matches an ACL Logical_Flow first? If no ACL is configured in the setup, what action does the `ls_in_acl` stage take?
3. Which stage decides the outport?
4. When the packet leaves the `ls-hello` datapath, what is the final action?

**Expected output (doc-plausible):**

1. Roughly 27 ingress stages plus 10 egress stages (OVN 22.03 LS pipeline).
2. `ls_in_acl` (around stage 9 in ingress) takes action `next;` (the default no-op when NBDB has no ACL row).
3. `ls_in_l2_lkup` (stage 27) decides via the destination MAC match.
4. `output;` at `ls_out_port_sec_l2` (the final egress stage).

**Lesson L1:**

- `ovn-trace` simulates the logical pipeline. It does not touch the physical OVS or the real packet.
- Each stage carries Logical_Flows that ovn-northd compiled from NBDB. The trace shows the highest-priority match.
- A skipped stage (one that does not appear in the trace) means no flow matched, so the implicit action was `next;` or drop.

## 20.7.3 Level 2, multi-table pipeline with a stateful ACL

### 20.7.3.1 Context and setup

**Scenario.** Add an ACL `deny tcp.dst == 22` (block SSH) for LSP `vm2`. Trace two packets: (a) ICMP ping (allowed), (b) TCP SSH (denied).

**Setup, growing from Level 1:**

```bash
[host]# ovn-nbctl pg-add pg-vm2 vm2
[host]# ovn-nbctl acl-add pg-vm2 to-lport 1000 \
        'tcp.dst == 22' drop
```

### 20.7.3.2 Command with `--detailed`

```bash
# Packet ICMP (allowed)
[host]# ovn-trace --detailed ls-hello \
        'inport == "vm1" && eth.src == 00:00:00:00:00:01 && \
         eth.dst == 00:00:00:00:00:02 && ip4.src == 10.0.0.1 && \
         ip4.dst == 10.0.0.2 && ip.ttl == 64 && icmp4'

# Packet TCP SSH (denied)
[host]# ovn-trace --detailed ls-hello \
        'inport == "vm1" && eth.src == 00:00:00:00:00:01 && \
         eth.dst == 00:00:00:00:00:02 && ip4.src == 10.0.0.1 && \
         ip4.dst == 10.0.0.2 && ip.ttl == 64 && \
         tcp && tcp.src == 12345 && tcp.dst == 22'
```

The `--detailed` flag adds:

- Display of every stage traversed, including no-op `next;` actions.
- The full `match` expression with the Logical_Flow UUID.
- The `ct_state` value at conntrack stages.

### 20.7.3.3 Anatomy of the SSH (denied) trace output

```
# tcp,reg14=0x1,...,nw_src=10.0.0.1,nw_dst=10.0.0.2,tp_src=12345,tp_dst=22
ingress(dp="ls-hello", inport="vm1")
---------------------------------------
 0. ls_in_port_sec_l2 (...): ...
    next;
 ...
 5. ls_in_pre_acl (...): ip, priority 100
    reg0[0] = 1;     /* mark for ct() */
    next;
 7. ls_in_acl_hint (...): !ct.trk, priority 7
    reg0[10] = 1; reg0[11] = 1;
    next;
 8. ls_in_acl (...): ip && reg0[10] == 1 && reg0[11] == 1, priority 65532
    ct_next;
                                  ^^^^^^^^^^^
                                  ct() invocation
 8. ls_in_acl (...): ip && !ct.est && reg0[7] == 1 && (outport == @pg-vm2 && tcp.dst == 22), priority 1000
    /* drop */
                                  ^^^^^^^^^^^
                                  ACL denial fires here
```

| Stage in output | Meaning | Diagnostic hint |
|-----------------|---------|-----------------|
| `ls_in_pre_acl` | Set `reg0[0]` to mark the packet for a later `ct()` invocation | Stateful ACL preparation for conntrack |
| `ls_in_acl_hint` | Set `reg0[10]` and `reg0[11]` for the next stage's `allow-stateless` versus `allow-related` match | Hint passed to the ACL stage |
| `ls_in_acl` action `ct_next` | The ACL triggers a conntrack lookup and re-evaluates afterwards | The `ct.trk` flag becomes set |
| After `ct_next`, the ACL stage runs again with `ct.est` set | The deny-SSH rule matches, action is `/* drop */` | Action `drop` ends the pipeline; the packet does not reach egress |

Compare with the ICMP (allowed) trace:

```
 8. ls_in_acl (...): ip && !ct.est && reg0[7] == 1 && (outport == @pg-vm2 && tcp.dst == 22), priority 1000
    no match    -- ICMP does not match the TCP rule
 8. ls_in_acl (...): default, priority 0
    next;       -- fallthrough allows the packet
```

### 20.7.3.4 ▶ Exercise L2

**Goal.** Understand the interplay between a stateful ACL and conntrack as shown by `ovn-trace`.

**Questions:**

1. Why does the `ls_in_acl` stage appear twice in the trace? What happens between the two appearances?
2. If the ACL action changes from `drop` to `allow-related`, how would the TCP SSH trace differ?
3. When does the conntrack state `ct.est` become set? What is the `ct_state` of the very first packet?
4. Does the pipeline reach egress in either case (ICMP versus SSH)?

**Expected output (doc-plausible):**

1. The first appearance triggers `ct_next` to consult the conntrack table; the second appearance evaluates the ACL with the conntrack flags now set. This is OVN's stateful ACL mechanism.
2. Replacing `drop` with `allow-related` lets the packet continue and ovn-northd commits a conntrack entry; reply traffic is auto-allowed.
3. `ct.est` becomes set on the second packet of the same connection (after the entry has been committed). The first packet has `ct.new`.
4. The ICMP packet reaches egress (output to vm2). The SSH packet stops at `ls_in_acl drop` in ingress and never reaches egress.

**Lesson L2:**

- A stateful ACL is `ACL + conntrack`. The trace clearly shows the two-pass evaluation.
- The `--detailed` flag is mandatory for ACL debugging. Without it you see only the final match, not the process.
- Each `drop` action in the trace has a corresponding trace line, giving you forensic clarity for the denial rationale.

## 20.7.4 Level 3, cross-subnet through a Logical Router

### 20.7.4.1 Context and setup

**Scenario.** Two LS (`ls-tenant-a` 10.0.0.0/24 and `ls-tenant-b` 10.1.0.0/24) connected through LR `lr-shared`. VM1 (10.0.0.10 on `ls-tenant-a`) pings VM3 (10.1.0.20 on `ls-tenant-b`).

**Setup:**

```bash
[host]# ovn-nbctl lr-add lr-shared
[host]# ovn-nbctl lrp-add lr-shared lrp-a aa:bb:cc:00:00:0a 10.0.0.1/24
[host]# ovn-nbctl lrp-add lr-shared lrp-b aa:bb:cc:00:00:0b 10.1.0.1/24

[host]# ovn-nbctl ls-add ls-tenant-a
[host]# ovn-nbctl lsp-add ls-tenant-a vm1
[host]# ovn-nbctl lsp-set-addresses vm1 "00:00:00:00:00:01 10.0.0.10"
[host]# ovn-nbctl lsp-add ls-tenant-a lsp-to-lrp-a
[host]# ovn-nbctl lsp-set-type lsp-to-lrp-a router
[host]# ovn-nbctl lsp-set-addresses lsp-to-lrp-a router
[host]# ovn-nbctl lsp-set-options lsp-to-lrp-a router-port=lrp-a

[host]# ovn-nbctl ls-add ls-tenant-b
[host]# ovn-nbctl lsp-add ls-tenant-b vm3
[host]# ovn-nbctl lsp-set-addresses vm3 "00:00:00:00:00:03 10.1.0.20"
[host]# ovn-nbctl lsp-add ls-tenant-b lsp-to-lrp-b
[host]# ovn-nbctl lsp-set-type lsp-to-lrp-b router
[host]# ovn-nbctl lsp-set-addresses lsp-to-lrp-b router
[host]# ovn-nbctl lsp-set-options lsp-to-lrp-b router-port=lrp-b
```

### 20.7.4.2 Command

```bash
[host]# ovn-trace --detailed ls-tenant-a \
        'inport == "vm1" && eth.src == 00:00:00:00:00:01 && \
         eth.dst == aa:bb:cc:00:00:0a && \
         ip4.src == 10.0.0.10 && ip4.dst == 10.1.0.20 && \
         ip.ttl == 64 && icmp4'
```

Note: `eth.dst` must be the MAC of LRP `lrp-a` (the gateway, ARP-resolved), not the MAC of vm3.

### 20.7.4.3 Anatomy of the output, across three datapaths

The output has **three phases**: `ls-tenant-a` ingress, `lr-shared` cross-datapath, `ls-tenant-b` egress.

```
ingress(dp="ls-tenant-a", inport="vm1")
---------------------------------------
27. ls_in_l2_lkup: eth.dst == aa:bb:cc:00:00:0a (LRP MAC), priority 50
    outport = "lsp-to-lrp-a"; output;

egress(dp="ls-tenant-a", outport="lsp-to-lrp-a")
---------------------------------------
 9. ls_out_port_sec_l2: ...
    output;        /* output to "lsp-to-lrp-a", type "router" */

ingress(dp="lr-shared", inport="lrp-a")
---------------------------------------
 ...
10. lr_in_ip_routing: ip4.dst == 10.1.0.0/24, priority 49
    ip.ttl--;
    reg0 = 10.1.0.20;        /* nexthop = dst since same subnet */
    reg1 = 10.1.0.1;          /* src ip = LRP-b ip */
    eth.src = aa:bb:cc:00:00:0b;
    outport = "lrp-b";
    flags.loopback = 1;
    next;
11. lr_in_arp_resolve: outport == "lrp-b" && ip4 && reg0 == 10.1.0.20, priority 100
    eth.dst = 00:00:00:00:00:03;     /* arp.tha learned */
    next;
22. lr_in_arp_request: ...
    next;

egress(dp="lr-shared", outport="lrp-b")
---------------------------------------
 ...
 5. lr_out_delivery: outport == "lrp-b", priority 110
    output;        /* output to "lrp-b" */

ingress(dp="ls-tenant-b", inport="lsp-to-lrp-b")
---------------------------------------
 ...
27. ls_in_l2_lkup: eth.dst == 00:00:00:00:00:03, priority 50
    outport = "vm3"; output;

egress(dp="ls-tenant-b", outport="vm3")
---------------------------------------
 ...
 9. ls_out_port_sec_l2: ...
    output;        /* output to "vm3", type "" */
```

**Observation points:**

| Phase | Main activity | Note |
|-------|---------------|------|
| LS-A ingress | L2 lookup, MAC of LRP-a, output to the router LSP | The packet leaves LS-A |
| LR-shared ingress | IP routing, `dec_ttl`, ARP resolve, `eth.src` and `eth.dst` rewrite | Cross-subnet logic |
| LR-shared egress | Delivery to LRP-b | LR phase done |
| LS-B ingress | L2 lookup, MAC of vm3, output | Final delivery preparation |
| LS-B egress | Port security, output | The packet reaches vm3 |

### 20.7.4.4 ▶ Exercise L3

**Goal.** Understand cross-subnet routing in OVN.

**Questions:**

1. Why does the trace show three datapaths instead of one?
2. Which LR stage performs `dec_ttl`?
3. At which stage is ARP for 10.1.0.20 resolved, and why does it succeed? (Hint: the NBDB already holds the MAC binding.)
4. Why is `flags.loopback = 1` set in `lr_in_ip_routing`?
5. If vm3 had no MAC binding (ARP not resolved yet), where would the trace end?

**Expected output (doc-plausible):**

1. The logical pipeline separates LS and LR. Each datapath has its own pipeline. A cross-subnet packet traverses two LS plus one LR, three datapaths in total.
2. `lr_in_ip_routing` (around stage 10).
3. `lr_in_arp_resolve` (around stage 11). It succeeds because `lsp-set-addresses` already set the static MAC `"00:00:00:00:00:03 10.1.0.20"` on vm3, so ovn-northd compiled a Logical_Flow with the `eth.dst` rewrite.
4. `flags.loopback = 1` allows the packet to pass through the L3 router without being dropped by anti-spoofing.
5. The pipeline stops at `lr_in_arp_request` (around stage 22) with the action `output:CONTROLLER`. ovn-controller then generates an ARP request and buffers the packet until the ARP reply arrives.

**Lesson L3:**

- A cross-subnet packet traverses several datapaths, and each datapath has its own pipeline. This is how OVN scales: adding datapaths does not lengthen any single pipeline.
- LR routing is more than `dec_ttl`. It also rewrites `eth.src` and `eth.dst` (standard router function).
- ARP resolution in OVN is **passive**: it checks the NBDB (a known address binding) or the SBDB `MAC_Binding` (dynamic). If neither is populated, the packet is buffered and an ARP request is dispatched.

## 20.7.5 Level 4, multichassis Geneve tunnel

### 20.7.5.1 Context and setup

**Scenario.** vm1 is on chassis A and vm2 is on chassis B (same LS). vm1 pings vm2. The trace must show: (a) the logical pipeline on A and B, (b) the physical OpenFlow on A pushing the packet through the tunnel, (c) the physical OpenFlow on B receiving it.

**Assumed setup.** Two chassis exist, vm1 is bound to chassis A, vm2 is bound to chassis B, and a Geneve tunnel on UDP 6081 is up.

### 20.7.5.2 Combine `ovn-trace` and `ofproto/trace`

**Step 1. Logical view via `ovn-trace`:**

```bash
[host-anywhere]# ovn-trace ls-hello \
        'inport == "vm1" && eth.src == 00:00:00:00:00:01 && \
         eth.dst == 00:00:00:00:00:02 && ip4.src == 10.0.0.1 && \
         ip4.dst == 10.0.0.2 && ip.ttl == 64 && icmp4'
```

The output is similar to Level 1 (a single datapath `ls-hello`). The logical view does not see chassis; it sees only the LS pipeline.

**Step 2. Physical view on chassis A via `ofproto/trace`:**

```bash
[chassis-A]# TAP1_OFPORT=$(ovs-vsctl get Interface tap-vm1 ofport)
[chassis-A]# ovs-appctl ofproto/trace br-int \
            "in_port=${TAP1_OFPORT},ip,nw_src=10.0.0.1,nw_dst=10.0.0.2,\
             dl_src=00:00:00:00:00:01,dl_dst=00:00:00:00:00:02"
```

Output (abridged):

```
Flow: ip,in_port=15,...,nw_src=10.0.0.1,nw_dst=10.0.0.2,nw_proto=1,...

bridge("br-int")
----------------
 0. in_port=15, priority 100, cookie 0xabc
    set_field:0x1->reg14
    set_field:0xa01->metadata
    resubmit(,8)
 8. metadata=0xa01,reg14=0x1, priority 50, cookie 0x123
    next stage logical pipeline...
...
65. metadata=0xa01,reg15=0x2, priority 100, cookie 0xdef
    set_field:0x1->tun_id      -- Geneve VNI = datapath tunnel_key
    set_field:0x2->NXM_NX_TUN_METADATA0[0..31]   -- outport TLV
    move:NXM_NX_REG14[0..14]->NXM_NX_TUN_METADATA0[16..30]   -- inport TLV
    output:1                                                      -- tunnel port
```

**Step 3. Physical view on chassis B via `ofproto/trace`:**

```bash
[chassis-B]# ovs-appctl ofproto/trace br-int \
            "in_port=1,ip,nw_src=10.0.0.1,nw_dst=10.0.0.2,\
             dl_src=00:00:00:00:00:01,dl_dst=00:00:00:00:00:02,\
             tun_id=0x1,tun_metadata0=0x00020001"
```

`in_port=1` is the tunnel port (Geneve). `tun_id` and `tun_metadata0` carry the metadata that chassis A set.

Output on B:

```
bridge("br-int")
----------------
 0. in_port=1,tun_id=0x1, priority 100
    move:NXM_NX_TUN_METADATA0[0..31]->NXM_NX_REG15[0..31]   -- outport restore
    move:NXM_NX_TUN_METADATA0[16..30]->NXM_NX_REG14[0..14]  -- inport restore
    move:NXM_NX_TUN_ID[0..23]->OXM_OF_METADATA[0..23]       -- tunnel_key restore
    resubmit(,33)
33. metadata=0xa01,reg15=0x2, priority 50
    output:tap-vm2
```

### 20.7.5.3 Anatomy: correlating logical and physical views

| Logical (`ovn-trace`) | Physical chassis A (`ofproto/trace`) | Physical chassis B (`ofproto/trace`) |
|-----------------------|--------------------------------------|--------------------------------------|
| `ingress(dp=ls-hello, inport=vm1)` | Table 0: `in_port=15` then set metadata and reg14 | Table 0: `tun_id` and `tun_metadata` then restore metadata |
| Each `ls_in_*` stage | Tables 8 to 34 (logical ingress) | Tables 8 to 34 |
| Each `ls_out_*` stage | Tables 41 to 66 (logical egress) | Tables 41 to 66 |
| `output;` at end of egress | Table 65: encap then `output:tunnel` | Table 33: `output:tap-vm2` |

Each logical stage maps to a specific OpenFlow table. Across chassis, the tunnel encap (table 65 on A) and tunnel decap (table 0 on B) happen automatically thanks to `put_encapsulation()` in `controller/physical.c` (Part 13.7.8).

### 20.7.5.4 ▶ Exercise L4

**Goal.** Verify that the Geneve TLV metadata is preserved across the tunnel.

**Questions:**

1. On chassis A, which OpenFlow action encodes `tunnel_key` into the Geneve VNI?
2. On chassis A, which action encodes `outport` (the logical egress port) into a Geneve TLV?
3. On chassis B, which action restores `outport` from the TLV into reg15?
4. If the underlay firewall drops UDP 6081, which trace fails and why?
5. Does `ovn-trace` simulate as far as chassis B?

**Expected output (doc-plausible):**

1. `set_field:0x1->tun_id` (24-bit Geneve VNI = datapath `tunnel_key`).
2. `set_field:0x2->NXM_NX_TUN_METADATA0[0..31]` (TLV class 0x0102, type 0x80, 32-bit outport).
3. `move:NXM_NX_TUN_METADATA0[0..31]->NXM_NX_REG15[0..31]`.
4. `ovn-trace` does not fail (the logical view does not touch the underlay). `ofproto/trace` on chassis A still produces output (the action matches), but the real packet does not reach B. `ofproto/trace` on chassis B has nothing to trace (the in_port never sees the packet). You must use `tcpdump -i eth1 udp port 6081` on the underlay to confirm the drop.
5. `ovn-trace` does not see chassis: it is purely a logical view, and it does not distinguish chassis A from B (they share the same datapath). You must combine it with `ofproto/trace` on each chassis to obtain the physical view.

**Lesson L4:**

- `ovn-trace` gives the logical pipeline view. It is simple, but it does not see the tunnel.
- `ofproto/trace` gives the physical OpenFlow view. It is detailed, but you must run it on each chassis separately.
- Combining the two is necessary for complete cross-chassis debugging. Part 9.27 went deeper into this 3-tier parallel diagnostic.

## 20.7.6 Level 5, production incident with `ovn-detrace`

### 20.7.6.1 Context

**Scenario.** Production incident: VM `vm-prod-456` reports "no network" after a live migration. The on-call engineer needs a forensic trace explaining why the current OpenFlow rules do not serve the traffic of `vm-prod-456`.

`ovn-detrace` is an advanced tool that chains with `ofproto/trace` (Part 20.2 §20.2.X covers it in detail). It accepts OpenFlow output and correlates it back to OVN logical context (the Logical_Flow UUID, the NBDB ACL or LB row, the datapath name).

### 20.7.6.2 Command

**Step 1. Run `ofproto/trace` on chassis B (the new claim chassis):**

```bash
[chassis-B]# TAP_OFPORT=$(ovs-vsctl get Interface tap-vm-prod-456 ofport)
[chassis-B]# ovs-appctl ofproto/trace br-int \
            "in_port=${TAP_OFPORT},ip,nw_src=10.5.5.10,nw_dst=10.5.5.20,..." \
            --names > /tmp/of-trace.txt
```

The `--names` flag replaces UUIDs and cookies with human-readable identifiers.

**Step 2. Chain with `ovn-detrace`:**

```bash
[chassis-B]# cat /tmp/of-trace.txt | ovn-detrace > /tmp/of-trace-annotated.txt
```

Or as a single pipeline:

```bash
[chassis-B]# ovs-appctl ofproto/trace br-int "..." --names | ovn-detrace
```

### 20.7.6.3 Anatomy of the annotated output

```
Flow: ip,in_port=15,...,nw_src=10.5.5.10,nw_dst=10.5.5.20,...

bridge("br-int")
----------------
 0. in_port=15,priority=100,cookie=0xa1b2
    /* logical ingress to "ls-prod-app", inport "vm-prod-456" */          -- from ovn-detrace
    set_field:0x1->reg14
    ...
 8. metadata=0xa01,reg14=0x1,priority=65532,cookie=0xc3d4
    /* Logical_Flow uuid 0xc3d4 from ls_in_acl, NBDB row uuid abcd... */   -- from ovn-detrace
    /* match: !ct.est && reg0[7] == 1 && (ip4.src == 10.5.5.10) */
    /* drop */
    --- pipeline drops here ---
```

`ovn-detrace` injects three kinds of information:

1. **Datapath name.** `"ls-prod-app"` instead of `metadata=0xa01`.
2. **Logical_Flow context.** Stage name `ls_in_acl` plus the UUID plus the original match expression.
3. **NBDB row UUID.** Traceback to a specific record (for example, an ACL row), so the engineer can identify exactly which configuration caused the drop.

### 20.7.6.4 Full forensic workflow

```
1. Start: alert "vm-prod-456 no network"
   |
   v
2. ovn-trace ls-prod-app '...'
   -> check logical intent OK?
   | (logical OK)
3. ssh chassis-B (claim chassis)
   ovs-appctl ofproto/trace br-int "..." --names | ovn-detrace
   -> show the NBDB row uuid of the Logical_Flow that drops
   | (e.g., uuid abcd)
4. ovn-nbctl find ACL _uuid=abcd
   -> show the specific ACL rule: from-lport, priority=1000, match=..., action=drop
   |
5. Root cause: the ACL was set incorrectly after the migration. Fix via ovn-nbctl acl-del or acl-add.
```

### 20.7.6.5 ▶ Capstone POE Phase I.B3 (closing Phase I)

**Goal.** The student designs a trace scenario for an unknown incident, demonstrates picking the correct level, and explains the rationale.

**Context.** The student receives an incident description (for example, "tenant T2 has lost connectivity to the database in tenant T3 after an ACL reload"). The student must:

1. **Choose a level** (1 to 5) appropriate for the incident.
2. **Compose the command** with the correct match expression.
3. **Predict** whether the trace output will or will not match where the packet is expected to be.
4. **Observe** the actual trace (run it in the lab if available; otherwise reason about it).
5. **Explain** the rationale: why this level, why not another, what the trace shows.
6. **Falsification test.** If the trace gives a result different from the prediction, which hypothesis was wrong?

**Grading criteria:**

| Criterion | Excellent | Good | Needs work |
|-----------|-----------|------|------------|
| Choosing the level | Correct on the first try | Correct after one adjustment | Cannot choose an appropriate level |
| Composing the command | Match expression correct on first input | Needs syntax fixes once or twice | Many syntax errors |
| Predict | Prediction matches observation | Prediction roughly correct | Prediction is the opposite |
| Explain | Full logical rationale plus cross-links to other Parts | Rationale OK but lacks cross-links | Weak rationale |
| Falsification | Identifies the wrong hypothesis precisely | Identifies it roughly | Confused |

**Self-evaluation.**

If the student picks Level 1 for a cross-subnet incident, the level is wrong (Level 1 covers only one LS). The correct choice is Level 3 or higher.

If the student picks Level 5 for a hello-world case, that is over-engineering. Plain `ovn-trace` is enough.

A model answer for the incident "tenant T2 has lost connectivity to the database in tenant T3 after an ACL reload":

- **Chosen level:** Level 2 (stateful ACL), because the incident is tied to an ACL change.
- **Command:** `ovn-trace --detailed ls-tenant-t2 'inport == "db-client" && eth.src == ... && ip4.dst == <T3-DB-IP> && tcp.dst == 5432'`
- **Predict:** the `ls_in_acl` stage shows action `drop` for the new rule that blocks traffic.
- **Observe:** the trace stops at `ls_in_acl drop` priority X for rule Y.
- **Explain:** the ACL reload added an unintended deny rule. Fix it via `ovn-nbctl acl-del` on the wrong rule.
- **Falsification:** if the trace shows `allow-related; output;` reaching the end of egress, the ACL is not the cause. Investigate at Level 4 (cross-chassis) or Level 5 (cluster-wide production incident).

## 20.7.7 Summary and the level-selection workflow

```
                   +-----------------------------+
                   |  Incident report            |
                   +--------------+--------------+
                                  |
                                  v
                   +-----------------------------+
                   |  Question 1: Same LS or     |
                   |  cross-subnet?              |
                   +----------+----------+-------+
                              | same     | cross
                              v          v
                       +----------+  +----------+
                       | Level 2  |  | Level 3  |
                       | ACL      |  | LR routing|
                       +-----+----+  +-----+----+
                             |             |
                             v             v
                   +-----------------------------+
                   |  Question 2: Same chassis   |
                   |  or cross-chassis?          |
                   +----------+----------+-------+
                              | same     | cross
                              v          v
                       (keep level)  +----------+
                                     | Level 4  |
                                     | Geneve   |
                                     +-----+----+
                                           |
                                           v
                   +-----------------------------+
                   |  Question 3: Production     |
                   |  incident with chassis logs?|
                   +----------+----------+-------+
                              | no       | yes
                              v          v
                       (keep level)  +----------+
                                     | Level 5  |
                                     | ovn-detrace|
                                     +----------+
```

**Rules:**

- Start at the **lowest level** appropriate to the scenario. A higher level is not always better: it costs more time and the output is longer.
- If level X is not enough to find the root cause, step up to level X+1.
- Level 5 is a last resort, used for production incidents on a large cluster that need forensic detail.

## 20.7.8 Key points to remember

**One.** `ovn-trace` simulates the logical pipeline and is schema-aware. `ofproto/trace` simulates the physical OpenFlow. Neither requires a real packet, and neither touches the network.

**Two.** The five-level gradient is pedagogical: each level unlocks one capability (single LS, then ACL, then LR, then multichassis, then forensic correlation). Skipping levels causes confusion.

**Three.** The `--detailed` flag for `ovn-trace` is mandatory at Level 2 and above. Without it you lose no-op stages and you lose the ACL rationale.

**Four.** A cross-subnet trace traverses three datapaths (LS-A, LR, LS-B). Each datapath has its own pipeline; the total pipeline length does not grow.

**Five.** `ovn-detrace` chained with `ofproto/trace --names` is the production-incident workflow: it annotates the OpenFlow output with the NBDB row UUID, the Logical_Flow context, and the datapath name.

**Six.** The level-selection workflow is three questions (same LS? same chassis? production?). Start low, escalate when needed.

**Seven.** Tracing is a **first-line** tool, not a last resort. For every OVN incident, run `ovn-trace` before you start guessing. A 30-second trace is often enough to identify 80 percent of root causes.

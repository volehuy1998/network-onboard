## 9.23.1 Opening context, the 300 Gbps DDoS and 20,000 lines of ACL

In March 2013, Spamhaus was hit by a DNS reflection DDoS that peaked at 300 Gbps. At the time, this was the largest DDoS attack ever recorded, five times larger than the GitHub attack in 2011. A tier-2 ISP in Europe, one of the upstream providers of Spamhaus, deployed an emergency **20,476 lines of Cisco IOS ACL** on its edge router to blackhole the spoofed source addresses within 48 hours (per the post-incident reports from Arbor Networks and Cloudflare, March 2013). The ACL was so long that each incoming packet paid an extra 40 microseconds for the table lookup, but it was still faster than a stateful firewall, which could not keep up with 300 Gbps of traffic.

A question worth thinking about: **can an OVS flow table do what a 20,000-line Cisco IOS ACL does, on a laptop?** The answer is *yes, technically*. The OpenFlow match field covers most Cisco ACL criteria (the five-tuple plus L2 MAC plus VLAN), priority replaces the line number, and OVS hardware offload through a SmartNIC reaches line rate at 100 Gbps (Section 9.5). Today's lab rebuilds that basic filtering mechanism with an OVS flow table, and it points out one small but important difference: **Cisco uses line numbers, OVS uses priority**. The two semantics are not exactly the same.

### ACL, the Cisco and academic definition

The Lab 7 document (Crichigno et al., page 4) cites the Cisco IOS Security Configuration Guide: *"ACLs [...] perform packet filtering and consist of a sequential series of statements called Access Control Entry (ACE). Each ACE specifies a matching criterion and an action that can be either permit or deny."*

Three core properties of a classical ACL:

**Sequential evaluation.** The packet is compared to each ACE from top to bottom. When it matches the first ACE, that action runs and **no further ACE is checked**. This is the "first match wins" behaviour.

**Implicit deny.** If the packet matches no ACE, the default behaviour is to drop (deny). Cisco IOS documents that every ACL has an implicit `deny any any` line at the end. The operator does not see this line, but the router always applies it.

**Declaration order matters.** The order in which the ACE entries are declared decides the result. A `permit any` at the top hides every later `deny`. A `deny host X` must come before `permit network contains X` if you want to block X specifically.

### A small but important difference: line number versus priority

Cisco IOS lets the operator insert an ACE at a specific position with `ip access-list resequence`, so the operator manages a linear order (ACE 10, 20, 30, 40, and so on). An OVS flow entry has no concept of "declaration order". Every entry is sorted by priority before it is matched.

Consequences:

- Cisco: with `ip access-list extended ACL1` followed by `permit tcp any any eq 80`, if an earlier ACE already declared `deny any any`, this rule is dead.
- OVS: `add-flow "ip,tcp,tp_dst=80,priority=100,action=output:2"` plus `add-flow "ip,priority=50,action=drop"`, the rule with priority 100 wins, regardless of which one was installed first.

With OVS, the operator does **not** have to worry about declaration order, but does have to design priorities carefully. In return, OVS does **not** implicitly deny. If no entry matches, OVS drops the packet (per the OpenFlow 1.0 specification). However, that is the switch behaviour, not an ACL behaviour.

## 9.23.2 The Lab 7 topology and the three-flow design

The Lab 7 topology (Crichigno et al., page 6): one OVS bridge `s1`, three hosts `h1` (10.0.0.1), `h2` (10.0.0.2), `h3` (10.0.0.3), all three in the same subnet `10.0.0.0/8`. Switch interfaces: `s1-eth1` to `h1-eth0`, `s1-eth2` to `h2-eth0`, `s1-eth3` to `h3-eth0`.

```
              ┌────────────────┐
              │   s1 (OVS)     │
   h1 ──eth1──│   br-int       │──eth3── h3
 10.0.0.1     │                │       10.0.0.3
 MAC :01      │      eth2      │       MAC :03
              └───────┬────────┘
                      │
                      h2 (10.0.0.2, MAC :02)
```

**Note:** this topology is the **same** as the topology of Lab 8 in Section 9.24. The choice is deliberate for teaching. It lets the reader compare stateless (Lab 7) and stateful (Lab 8) directly on the same setup, by changing only the flow rules.

### Security goal

Per Lab 7 page 11: *"the switch blocks the traffic when the source address is 10.0.0.3 and the destination address is 10.0.0.1"*. In other words:

- h3 to h1 (direction 10.0.0.3 to 10.0.0.1): **DENY** (block).
- All other traffic: **PERMIT** (allow).

This is equivalent to a classical two-ACE ACL table:

| Priority | Source | Destination | Action |
|----------|--------|-------------|--------|
| 40000 | 10.0.0.3 | 10.0.0.1 | drop |
| 32768 (default) | any | any | permit (goto forwarding) |

### The three flow entries

Per Lab 7 pages 11 to 13 (Figures 16 to 18), there are **3 flow entries** across two tables:

```bash
# Flow 1: DENY h3 to h1 at high priority
sudo ovs-ofctl add-flow s1 \
  "table=0,priority=40000,ip,nw_src=10.0.0.3,nw_dst=10.0.0.1,action=drop"

# Flow 2: Default permit at the default priority 32768, goto table 1
sudo ovs-ofctl add-flow s1 "table=0,action=goto_table=1"

# Flow 3: Table 1 normal forwarding
sudo ovs-ofctl add-flow s1 "table=1,action=normal"
```

### Semantic walkthrough

**Flow 1** has priority 40000, higher than the default 32768 of flow 2. When a packet matches both, flow 1 wins and the packet is dropped.

**Flow 2** uses the default priority. (When `priority=X` is not specified, OVS uses 32768.) It matches every IP packet that flow 1 did not catch, and the instruction is `goto_table=1`.

**Flow 3** lives in table 1, with action `normal`. OVS handles the packet as a classical L2 switch with MAC learning. Because all three hosts are in the same subnet, the packet is forwarded across the bridge directly to the right port. No routing is needed.

> **Key point:** Notice that flow 2 does not have the match field `ip` or `arp`. It matches *every* packet at table 0 that was not caught by flow 1. This lets ARP go through normally (through the action `normal` at table 1, which floods the ARP broadcast). If flow 2 only matched `ip`, ARP between h3 and h1 might fail to resolve. Then ping would fail in addition to the ACL drop, and it would be hard to tell the two causes apart while debugging.

### Verification with ping

Per Lab 7 pages 13 to 14 (Figures 20 to 22):

- `h1$ ping 10.0.0.2`: **succeeds** (h1 to h2 does not match flow 1).
- `h1$ ping 10.0.0.3`: **fails** (h1 to h3 does not match flow 1, but the reply h3 to h1 matches flow 1 and is dropped, so ping still reports failure on h1).
- `h2$ ping 10.0.0.3`: **succeeds** (does not match flow 1).

**An important point.** Lab 7 page 13 Step 3 says *"You will notice that there is no connectivity between hosts h1 and h3."* The reason is exactly that *h3 cannot reply to h1*. The ACL only blocks the h3-to-h1 direction, but `ping` needs both directions to succeed. This is the essence of stateless filtering: blocking one direction effectively blocks the entire two-way conversation.

## 9.23.3 Priority, a worked example showing that declaration order does not matter

Run the experiment below to verify the claim "OVS does not care about declaration order":

```bash
# Delete the old flows
sudo ovs-ofctl del-flows s1

# Install the default permit FIRST
sudo ovs-ofctl add-flow s1 "table=0,action=goto_table=1"

# Install the ACL deny SECOND
sudo ovs-ofctl add-flow s1 \
  "table=0,priority=40000,ip,nw_src=10.0.0.3,nw_dst=10.0.0.1,action=drop"

# Install the forwarding rule
sudo ovs-ofctl add-flow s1 "table=1,action=normal"
```

Inspect the installed flows:

```bash
sudo ovs-ofctl dump-flows s1
```

Expected output (plausible):

```
cookie=0x0, duration=..., table=0, n_packets=0, n_bytes=0, priority=40000,ip,nw_src=10.0.0.3,nw_dst=10.0.0.1 actions=drop
cookie=0x0, duration=..., table=0, n_packets=0, n_bytes=0, priority=32768 actions=goto_table:1
cookie=0x0, duration=..., table=1, n_packets=0, n_bytes=0, priority=32768 actions=NORMAL
```

OVS prints the entries in descending priority order. The `priority=40000` entry appears first, even though it was installed last. Try `h3 ping 10.0.0.1`. It still fails. This is the proof: **priority decides, not declaration order**.

### Comparison with Cisco IOS

If you install the same rules in the same order on Cisco IOS:

```
ip access-list extended ACL1
 permit ip any any
 deny   ip host 10.0.0.3 host 10.0.0.1
```

Result: the `deny` rule **never matches**, because `permit ip any any` sits at a smaller line number and runs first. The operator must `resequence` the ACL, or `no permit` and reinstall.

This difference has a real operational consequence. The OVS operator can install rules in any order (for example through automation scripts), as long as the priorities do not collide. The Cisco operator must follow a strict line-number convention (typically 10, 20, 30, and so on, leaving room for inserts).

## Guided Exercise 1, `ofproto/trace` to verify flow matching

**Goal:** observe the OVS ACL decision directly on each packet, verify which flow matches the permitted traffic and which flow matches the denied traffic, and prove that `ovs-appctl ofproto/trace` reflects the multi-table pipeline correctly.

**Hypothesis to verify:**

> *"With the Lab 7 two-table pipeline installed, a packet h1 to h3 (the permit path) traces through table 0 with `goto_table:1` then table 1 with `normal`. A packet h3 to h1 (the deny path) traces through table 0 with `drop`, and the pipeline stops there without reaching table 1."*

### Setup

Load the Lab 7 topology and install the three flows from §9.23.2. Verify:

```bash
sudo ovs-ofctl dump-flows s1
# Expected: 2 entries in table 0 plus 1 entry in table 1.
```

### Step 1, trace the permit path (h1 to h3)

```bash
sudo ovs-appctl ofproto/trace s1 \
  "ip,nw_src=10.0.0.1,nw_dst=10.0.0.3,dl_src=00:00:00:00:00:01,dl_dst=00:00:00:00:00:03"
```

Expected output (plausible):

```
Flow: ip,in_port=ANY,vlan_tci=0x0000,dl_src=00:00:00:00:00:01,dl_dst=00:00:00:00:00:03,
      nw_src=10.0.0.1,nw_dst=10.0.0.3,nw_proto=0,nw_tos=0,nw_ecn=0,nw_ttl=0

bridge("s1")
-------------
 0. priority 32768
    goto_table:1
 1. priority 32768
    NORMAL
    -> forwarding to learned port

Final flow: unchanged
Datapath actions: 3
```

Three points to notice:

- **Table 0 matches priority 32768** (flow 2, the default permit). It does *not* match flow 1, because `nw_src` is not 10.0.0.3.
- **The instruction `goto_table:1`** hands control to table 1.
- **Table 1 action `NORMAL`**: OVS uses MAC learning and forwards the packet to port 3 (where h3 is attached).

### Step 2, trace the deny path (h3 to h1)

```bash
sudo ovs-appctl ofproto/trace s1 \
  "ip,nw_src=10.0.0.3,nw_dst=10.0.0.1,dl_src=00:00:00:00:00:03,dl_dst=00:00:00:00:00:01"
```

Expected output (plausible):

```
Flow: ip,in_port=ANY,...,nw_src=10.0.0.3,nw_dst=10.0.0.1,...

bridge("s1")
-------------
 0. ip,nw_src=10.0.0.3,nw_dst=10.0.0.1, priority 40000
    drop

Final flow: unchanged
Datapath actions: drop
```

Three contrast points:

- **Table 0 matches priority 40000** (flow 1, the ACL deny).
- **Action `drop`**: the pipeline stops. *There is no `1.` line for table 1*.
- **`Datapath actions: drop`**: the kernel receives the drop instruction.

### Step 3, observe the "tied priority" behaviour

Install one more flow at priority 40000 with the same match field but a different action:

```bash
sudo ovs-ofctl add-flow s1 \
  "table=0,priority=40000,ip,nw_src=10.0.0.3,nw_dst=10.0.0.1,action=goto_table=1"
```

Two flows at priority 40000 share the same match. OVS picks one at random. If you trace several times, you will see different results: sometimes drop, sometimes `goto_table:1`.

This is why the OpenFlow specification says the behaviour is **undefined** when several entries at the same priority share the same match field. The practical guideline: every entry in the same table must have a distinct priority, or must have non-overlapping match fields.

### Step 4, evidence of the stateless limit

Restore the original three flows. From h1, run `iperf -c 10.0.0.3`. Expect the result to fail almost at once. The reason is not that the firewall blocks the h1-to-h3 direction. The reason is that the TCP SYN-ACK from h3 to h1 matches flow 1 and is dropped. The client h1 waits for SYN-ACK and then times out.

How to diagnose:

```bash
# On h3, run tcpdump to see the SYN arriving
h3$ sudo tcpdump -n -i h3-eth0 'tcp and port 5001'
# You will see the SYN coming from h1, but the SYN-ACK is dropped at s1
```

The hypothesis "blocking one direction blocks only that direction" is refuted. In practice, a stateless ACL that blocks one direction **breaks the entire two-way conversation**.

### Lesson learned

A stateless ACL is a strong filtering tool at L2 to L4, but it has two limits that cannot be overcome without stateful filtering:

- **A one-direction rule is not enough.** If you want to allow h1 to h3 but block h3 to h1, you still need to let the reply h3 to h1 through (with state `+est`). That requires `ct()`.
- **Stateless has no automatic concept of letting the reply through.** Every rule must be declared explicitly for both directions, otherwise real-world traffic fails.

This is exactly the motivation for Section 9.24, stateful conntrack.

## 9.23.4 Comparison with OVN, why `allow` still exists alongside `allow-related`

OVN ACL has four actions: `allow`, `allow-related`, `drop`, `reject`. Among them, `allow` is stateless (equivalent to Lab 7) and `allow-related` is stateful (built on `ct()`, equivalent to Section 9.24 Lab 8). Why does OVN keep both, instead of always defaulting to `allow-related`?

**Reason 1, performance for symmetric traffic.** Traffic such as ICMP echo (ping) is fully symmetric: both directions look like "NEW" from the point of view of each host. The cost of stateful tracking adds no value. Using `allow` saves two conntrack lookups per packet, and at millions of packets per second this adds up.

**Reason 2, avoid conntrack for internal traffic.** East-west traffic between VMs of the same tenant inside a datacentre often does not need state tracking. OVN on top of OVS from version 22.03 has an optimization: `allow` skips the `ct()` action entirely, and the packet goes straight to the forwarding table. `allow-related` forces the packet through the kernel conntrack module, which means a kernel-userspace context switch.

**Reason 3, compatibility with hardware offload.** A SmartNIC like NVIDIA ConnectX-6 or Intel E810 can offload OpenFlow rules into the ASIC, but `ct()` typically cannot be offloaded (conntrack runs on the CPU). Using `allow` keeps the traffic on the fast path (in silicon). Using `allow-related` forces the slow path (on the CPU).

### When to choose plain `allow`

- Symmetric administrative traffic (SSH from a bastion to a VM: both directions need explicit rules anyway).
- Monitoring traffic (Prometheus scrape: read-only, no state needed).
- Multicast protocols (BFD, OSPF hello, no state concept).

### When you must use `allow-related`

- TCP-based application traffic (HTTP, HTTPS, database, RPC): the reply traffic cannot be matched symmetrically when the client port is random.
- ICMP error messages that need to be routed alongside the original connection (unreachable, TTL exceeded).
- NAT traffic: conntrack carries the NAT binding.

> **Common misconception:** *"OVN `allow` means stateless, so it should be used whenever you want performance."* This is only true when the operator controls both directions. With TCP client-to-server, the client port is random, so symmetric matching is impossible, and you must use `allow-related`. This confusion is the cause of the OpenStack bank incident in 2018 (see the opening of Section 9.24).

## 9.23.5 Key points to remember

**One.** An ACL is a sequential chain of ACE entries with "first match wins" behaviour. Declaration order decides the result under Cisco IOS line-number semantics. OVS replaces "declaration order" with **priority**: the install time does not matter, the higher priority wins.

**Two.** The Lab 7 pipeline only needs three flow entries across two tables: one high-priority entry for the ACL deny, one default-permit entry with `goto_table=1`, and one `normal` forwarding entry in table 1. Scaling to N rules only needs N more entries with increasing priority.

**Three.** Stateless filtering has two limits that cannot be overcome: (1) a one-direction rule breaks the two-way conversation, and (2) the reply is not automatically allowed. These two limits are the motivation for the `ct()` action in Section 9.24.

**Four.** OVN keeps both `allow` (stateless) and `allow-related` (stateful) on purpose: performance for symmetric traffic plus compatibility with hardware offload. Choosing the right action is an important skill when designing OVN ACLs in a production environment.

**Five.** `ovs-appctl ofproto/trace` is the first debugging tool when an ACL does not behave as expected. The trace shows which flow matched in which table at which priority. That is enough information to track down the root cause without needing tcpdump.

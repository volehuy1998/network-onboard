## 9.16.1 The role of the connection manager

The **connection manager** is a component inside `ovs-vswitchd` (specifically inside `ofproto-dpif.c` and `connmgr.c`). It manages every TCP and SSL connection from the switch to the OpenFlow controllers. Each bridge has its own connection manager instance, which tracks the list of controllers configured through `ovs-vsctl set-controller`.

Its main responsibilities:

- **Connection setup:** open a TCP or SSL connection to the controller, with retry and exponential backoff.
- **Heartbeat:** send OpenFlow `echo_request` messages periodically and detect when the controller becomes unreachable.
- **Flow lifecycle:** when a controller connects or disconnects, handle the `fail_mode` (`secure` or `standalone`) to decide whether to keep or to clear the flow table.
- **Role management:** coordinate the MASTER, SLAVE, and EQUAL roles when several controllers are connected at once.
- **OpenFlow message routing:** forward `packet_in` to the MASTER, send `echo` to every controller, and route every reply back to the correct controller.
- **Connection state:** track the state of each connection (CONNECTING, ACTIVE, IDLE, DISCONNECTED) and report it through `ovs-ofctl dump-ports` and `ovs-appctl bridge/dump-flows`.

A controller connection has two modes:

- **Primary (in-band or out-of-band):** sends and receives OpenFlow protocol messages. There can be several primaries for high availability (this is supported from OF 1.3 onward, together with role election).
- **Service (`ovs-ofctl monitor`):** observes only. It cannot send `packet_out` or affect the flow table. It is used for debugging.

Configure the primary controllers:

```
[host]# ovs-vsctl set-controller br0 tcp:10.0.0.1:6653 tcp:10.0.0.2:6653 tcp:10.0.0.3:6653
```

Three high-availability controllers. The connection manager will try to connect to all three and keep them up.

## 9.16.2 OpenFlow 1.3 role election: MASTER, SLAVE, and EQUAL

OpenFlow 1.0 and 1.2 assumed a single controller. With several controllers connected at the same time, there was no coordination mechanism, which led to race conditions (two controllers writing the flow table at the same time, leaving it inconsistent). OpenFlow 1.3 (the RFC-style specification) introduces **role request and role reply messages** that let a controller declare its role:

**MASTER.** Has full write authority over the flow table (`add`, `modify`, and `delete` flow). Receives every asynchronous message (`packet_in`, `flow_removed`, `port_status`). Only one MASTER may be active at any time. If controller A sets `role=MASTER`, any other controller currently in `MASTER` is automatically demoted to SLAVE.

**SLAVE.** Has read-only access. It can dump the flow table and subscribe to port stats, but it cannot add, modify, or delete flows. Does it receive `packet_in`? It depends on the spec: OF 1.3 does not deliver `packet_in` to the SLAVE (only to MASTER), while OF 1.4 and later let the SLAVE subscribe to `packet_in` but expects no response.

**EQUAL.** Every EQUAL controller has equal status. Each has write authority over the flow table and receives every message. There is no "single master" rule for EQUAL. This allows an active-active topology, but the controllers must coordinate among themselves through an out-of-band channel to avoid conflicts.

**Role request and reply message format:**

```
struct ofp_role_request {
    uint32_t role;         // 0=NOCHANGE, 1=EQUAL, 2=MASTER, 3=SLAVE
    uint32_t pad;
    uint64_t generation_id; // monotonic counter, for replay protection
};
```

**Generation ID** is the anti-race mechanism. Every time a controller requests `role=MASTER`, it must include a `generation_id` higher than the current one held by the switch. If the value is lower, the switch rejects the request (this controller is stale). This is how OVS decides "who is the latest master" when several controllers race to claim MASTER.

**Typical role transition flow:**

1. Controller A connects and sends `role_request(MASTER, gen=1)`. The switch accepts.
2. Controller B connects and sends `role_request(MASTER, gen=2)`. The switch demotes A to SLAVE and promotes B to MASTER.
3. Controller A receives a `role_status` notification "you are now SLAVE". It can react (for example, retry with `gen=3` to claim MASTER again).
4. If B loses its connection, A retries `role_request` with `gen=3`. The switch accepts and A becomes MASTER again.

## 9.16.3 `fail_mode`: behavior when the controller connection is lost

When every controller disconnects, the connection manager decides what to do based on the **`fail_mode`** configuration:

**`fail_mode=secure`** (the default in production OVS deployments). Keep the flow table as it is. Packets continue to forward by the existing flow rules. Packets that do not match any rule are either dropped or sent through the default rule. This is the right mode for production: if the controllers go down, the switch keeps running on its last known flow state and traffic does not stop.

**`fail_mode=standalone`** (the default for a newly created bridge). Wipe every OpenFlow flow and fall back to traditional MAC-learning switch behavior (similar to a Linux bridge). The switch acts as an L2 learning switch: packets are flooded and source MAC addresses are learned. This is suitable for bootstrapping (when a controller has not yet connected for the first time), but it is not suitable for production.

Configure `fail_mode`:

```
[host]# ovs-vsctl set-fail-mode br0 secure
[host]# ovs-vsctl get-fail-mode br0
secure
```

**Recovering after a `fail_mode` event.** If the bridge is in `secure` mode and loses the connection, the flow table stays as it is. When a controller reconnects, the new MASTER can choose to "resync" the flow table (delete every flow and re-install based on the new state) if it wants to.

## 9.16.4 In-band versus out-of-band connection

A controller connection can take one of two paths:

**Out-of-band.** The controller and the switch communicate over a dedicated management network (not through any OpenFlow-managed port). This is the standard production configuration: the switch has a separate management interface (such as `eth0`) that connects to the controller.

**In-band.** The controller connection passes through the same OpenFlow-managed bridge. The switch must learn how to forward OpenFlow traffic to the controller through its own flow table. This is a chicken-and-egg challenge: the switch needs flow rules to forward, but the flow rules come from the controller.

OVS implements in-band by automatically generating "in-band rules" inside the flow table: match on the controller destination IP, then forward out the interface that reaches the controller. These rules refresh automatically when the topology changes.

To configure in-band, attach a controller IP that belongs to the subnet of a bridge port (do not use a dedicated management network). OVS detects this and turns on in-band mode.

In-band has a weakness: if the flow table is corrupted, or if the in-band rule is overwritten, the switch loses its connection and the old flows persist forever. Out-of-band avoids this risk, but it requires a dedicated management infrastructure.

## 9.16.5 Bundle: applying many flows atomically

OpenFlow 1.4 and later introduce the **bundle**, a mechanism to apply many `flow_mod` messages atomically. The controller sends a series of `flow_mod` messages into a bundle, then commits. The switch then applies every message at once, or applies nothing at all (atomic).

Messages:

- `bundle_open(bundle_id)`: open a new bundle.
- `bundle_add_message(bundle_id, flow_mod)`: add a `flow_mod` to the bundle (not yet applied).
- `bundle_commit(bundle_id)`: apply every message in the bundle atomically.
- `bundle_discard(bundle_id)`: discard the bundle without applying it.

Use case: the controller needs to update 100 flows at the same time (for example, to migrate traffic from path A to path B). If it sends 100 individual `flow_mod` messages, there is a window where the table contains half-A and half-B. During that window, packets can drop or loop. The bundle guarantees that the entire change applies at once, so there is no inconsistency window.

OVS supports bundles from OVS 2.4 onward, with OpenFlow 1.4 and later. Use it through `ovs-ofctl --bundle`:

```
[host]# ovs-ofctl --bundle add-flows br0 flows.txt
```

The file `flows.txt` contains many flows, applied inside one bundle.

## 9.16.6 Observation: connection state and role

**View the configured controllers:**

```
[host]# ovs-vsctl list Controller
```

**View the connection state:**

```
[host]# ovs-ofctl dump-connection br0
```

The output shows each controller with its target, role (MASTER, SLAVE, or EQUAL), state (`connected` or `waiting`), and connection statistics.

**Monitor role transitions in real time:**

```
[host]# ovs-ofctl monitor br0 --ver=OpenFlow13
```

The output streams events, including the `role_status` notification when the role changes.

**Internal `ovs-appctl` statistics:**

```
[host]# ovs-appctl bridge/dump-flows br0
[host]# ovs-appctl -t ovs-vswitchd coverage/show | grep -E 'ofproto|connmgr'
```

The `ofproto_packet_in` and `connmgr_reconnect` coverage counters help spot anomalies.

**Change the role from the CLI (for testing):**

```
[host]# ovs-ofctl -O OpenFlow13 mod-connection br0 role=master
```

This only takes effect on the CLI connection itself. `ovs-ofctl` sets `role=master` for the current session.

### Guided Exercise 1: test multi-controller HA with role election

> **State verification lab:** the output is based on OVS 2.17 and the OpenFlow 1.3 documentation. Awaiting confirmation in a real lab.

**Goal.** Configure two controllers for a bridge and observe the role election when both controllers claim MASTER.

**Setup.** An Ubuntu 22.04 node with OVS 2.17 and the Ryu controller (`pip install ryu`).

**Steps.**

**1.** Create a bridge with OpenFlow 1.3:

```
[host]# ovs-vsctl add-br br0 -- set bridge br0 protocols=OpenFlow13
[host]# ovs-vsctl set-controller br0 tcp:127.0.0.1:6653 tcp:127.0.0.1:6654
[host]# ovs-vsctl set-fail-mode br0 secure
```

**2.** Run two Ryu controllers on different ports:

```
[term1]# ryu-manager --ofp-tcp-listen-port 6653 ryu.app.simple_switch_13
[term2]# ryu-manager --ofp-tcp-listen-port 6654 ryu.app.simple_switch_13
```

**3.** Observe the connection state:

```
[host]# ovs-ofctl -O OpenFlow13 dump-connection br0
```

Expected result:

```
bridge br0 has 2 controllers:
  tcp:127.0.0.1:6653 role=EQUAL state=connected
  tcp:127.0.0.1:6654 role=EQUAL state=connected
```

By default, both controllers are EQUAL (no controller has claimed MASTER). Both have the right to write the flow table. This is risky.

**4.** Have controller 1 claim MASTER (through a custom Ryu app, or through `ovs-ofctl`):

```
[host]# # From the controller 1 terminal, through ovs-ofctl:
[host]# ovs-ofctl -O OpenFlow13 mod-connection br0 role=master \
         --target=tcp:127.0.0.1:6653
```

**5.** Re-check the connection:

```
[host]# ovs-ofctl -O OpenFlow13 dump-connection br0
  tcp:127.0.0.1:6653 role=MASTER state=connected
  tcp:127.0.0.1:6654 role=EQUAL state=connected
```

Controller 1 is now MASTER. If controller 2 also requests MASTER, controller 1 will be demoted.

**6.** Kill controller 1 and observe the failover:

```
[term1]# # Press Ctrl plus C
[host]# ovs-ofctl -O OpenFlow13 dump-connection br0
  tcp:127.0.0.1:6653 role=MASTER state=CONNECTING  # reconnecting
  tcp:127.0.0.1:6654 role=EQUAL state=connected
```

The flow table on `br0` is preserved (because `fail_mode=secure`). Packets still forward according to the existing rules.

**Lessons learned.**

First, the default role is EQUAL. There is no automatic election. To run active-passive HA, the controller must explicitly request the MASTER role. This is the standard pattern in ONOS and Faucet.

Second, `fail_mode=secure` is critical for production. If you set `standalone`, then a controller disconnect causes the switch to fall back to MAC learning, which throws away every OpenFlow policy and removes the security policy too.

Third, `generation_id` guarantees that two controllers cannot race to claim MASTER. The latest controller (with the higher generation) wins.

**Cleanup:**

```
[host]# ovs-vsctl del-br br0
```

## 9.16.7 Multi-controller setup anatomy

The previous section introduced `OFPT_ROLE_REQUEST` conceptually. This section shows the real setup with three controllers, role election, and failover.

### 9.16.7.1 Setting up three controllers

```shell
# Add three controllers to br0
$ sudo ovs-vsctl set-controller br0 \
    tcp:10.0.0.1:6633 \
    tcp:10.0.0.2:6633 \
    tcp:10.0.0.3:6633

# Verify
$ sudo ovs-vsctl get-controller br0
tcp:10.0.0.1:6633
tcp:10.0.0.2:6633
tcp:10.0.0.3:6633

# Connection state
$ sudo ovs-ofctl show br0
OFPT_FEATURES_REPLY (OF1.3) (xid=0x2): dpid:0000000000000001
...
```

### 9.16.7.2 Reading the output of `ovs-appctl ofproto/show-connection`

**Command and output** `[reproduced from the OVS source code and documentation]`:

```shell
$ sudo ovs-appctl ofproto/show-connection br0
=== Bridge br0 connection status ===
Controllers:
  tcp:10.0.0.1:6633:
    state: ACTIVE
    role: MASTER
    last_connected: 3421s ago
    generation_id: 0x0000000000000042
    rx_packets: 182347
    tx_packets: 145623
    bytes_received: 23M
    bytes_sent: 18M

  tcp:10.0.0.2:6633:
    state: ACTIVE
    role: SLAVE
    last_connected: 3420s ago
    generation_id: 0x0000000000000042
    rx_packets: 5
    tx_packets: 121
    ...

  tcp:10.0.0.3:6633:
    state: BACKOFF
    role: EQUAL
    last_error: connection refused
    backoff_seconds: 8
    ...
```

**Anatomy by field:**

| Field | Meaning | Abnormal state |
|---|---|---|
| `state` | `ACTIVE` (connected), `BACKOFF` (reconnecting), or `IDLE` (new) | A persistent `BACKOFF` means the controller is unreachable. Check TCP. |
| `role` | `EQUAL`, `MASTER`, or `SLAVE` | If the cluster has no MASTER, no controller has write authority. |
| `last_connected` | Time since the last successful connect | If `last_connected > 1h` and the state is `BACKOFF`, the controller has been down for a long time. |
| `generation_id` | The role-election generation | The highest value is the latest MASTER; lower values are stale. |
| `rx_packets` / `tx_packets` | OpenFlow message counters | If `rx=0` while the state is `ACTIVE`, the inactivity probe is not active. |
| `backoff_seconds` | The exponential backoff before the next reconnect | Grows 2, 4, 8, 16, and so on, up to a maximum of 8 seconds. |

### 9.16.7.3 A role election timeline

For a three-controller ONOS cluster:

```
T+0    All three controllers connect to OVS (default role EQUAL).
T+100  Controller C1 sends OFPT_ROLE_REQUEST(role=MASTER, generation_id=42).
T+100  OVS accepts C1 as MASTER (only C1 has the MASTER role).
T+100  OVS automatically converts C2 and C3 from EQUAL to SLAVE.
T+200  Controller C2 sends OFPT_ROLE_REQUEST(role=MASTER, generation_id=43).
       generation_id=43 is greater than 42, so OVS accepts C2 as MASTER.
T+200  OVS automatically demotes C1 from MASTER to SLAVE (the generation is outdated).
T+500  C2 misses heartbeats; the switch detects the timeout.
T+500  C1 sends ROLE_REQUEST(role=MASTER, generation_id=44).
T+500  OVS accepts C1 as the new MASTER.
```

## 9.16.8 `OFPT_ROLE_REQUEST` wire format

OpenFlow 1.3 specification §7.3.9. Wire format:

```
struct ofp_role_request {
    struct ofp_header header;    /* OFPT_ROLE_REQUEST = 24 */
    uint32_t role;               /* OFPCR_ROLE_NOCHANGE/EQUAL/MASTER/SLAVE */
    uint8_t  pad[4];             /* align */
    uint64_t generation_id;      /* Master election ID */
};
/* Size: 24 bytes */
```

**Role values:**

| Value | Meaning | Write authority |
|---|---|---|
| 0 `OFPCR_ROLE_NOCHANGE` | Query the current role; do not change it. | Not applicable |
| 1 `OFPCR_ROLE_EQUAL` | Peer; can write; receives every async message. | Yes |
| 2 `OFPCR_ROLE_MASTER` | Primary; exclusive write authority; receives the full async stream. | Yes |
| 3 `OFPCR_ROLE_SLAVE` | Read-only; only receives `PORT_STATUS` async messages. | No (the switch returns `OFPBRC_IS_SLAVE` for write attempts) |

**Decision rule per role:**

- **EQUAL.** Receives every async message (`PACKET_IN`, `FLOW_REMOVED`, `PORT_STATUS`).
- **MASTER.** Receives every async message and has write authority. Only one controller can have the MASTER role at any given time.
- **SLAVE.** Only `PORT_STATUS`. Any write request returns the `OFPBRC_IS_SLAVE` error.

### 9.16.8.1 Additional async configuration

OpenFlow 1.3 specification §7.5.4. The controller can customize the async messages it receives through `OFPT_SET_ASYNC`. For example, a SLAVE controller that wants to receive `PACKET_IN` (in addition to the default `PORT_STATUS`):

```
OFPT_SET_ASYNC: master_packet_in_mask=<...>, slave_packet_in_mask=0x3
# The SLAVE will receive PACKET_IN for reason 0 (no_match) and reason 1 (action).
```

OVN does not customize the async stream and uses the default.

## 9.16.9 `connmgr` coverage counters

### 9.16.9.1 Counters from `coverage/show`

Filter the counters related to `connmgr`:

```shell
$ sudo ovs-appctl coverage/show | grep -E 'connmgr|vconn|rconn'
connmgr_wakeup            0.0/sec   0.017/sec   0.021/sec   total:182
rconn_overflow            0.0/sec   0.000/sec   0.000/sec   total:0
rconn_queued              0.0/sec   0.008/sec   0.012/sec   total:87
rconn_sent                0.0/sec   0.033/sec   0.042/sec   total:128
vconn_received            0.0/sec   0.033/sec   0.042/sec   total:128
vconn_sent                0.0/sec   0.033/sec   0.042/sec   total:128
vconn_open                0.0/sec   0.000/sec   0.000/sec   total:3
vconn_close               0.0/sec   0.000/sec   0.000/sec   total:2
```

**Anatomy:**

| Counter | Meaning | Healthy |
|---|---|---|
| `connmgr_wakeup` | Connection manager thread wakeups | A low rate means the connection is stable. |
| `rconn_overflow` | Reliable-connection send-queue overflow | Must be 0. |
| `rconn_queued` | Messages queued waiting to send (the controller is slow to receive) | A high rate means the controller is choking. |
| `rconn_sent` | Messages successfully sent | Reference value. |
| `vconn_received` / `vconn_sent` | Virtual connection RX and TX (every connection) | Should stay balanced. |
| `vconn_open` / `vconn_close` | TCP or SSL socket open and close events | If `close > open`, there is a leak. |

### 9.16.9.2 Failure scenarios

- **`rconn_overflow > 0`.** The controller or the connection consumes messages too slowly. Check controller CPU and network round-trip time.
- **`rconn_queued` rising fast.** The controller is under heavy load. Rate-limit the `actions=controller` rule, or scale the controller.
- **`vconn_open` is much greater than `vconn_close`, and not converging.** TCP connection leak; check the kernel socket table.

## 9.16.10 Connection manager troubleshooting matrix

| Symptom | First command | Expected output | Fix |
|---|---|---|---|
| `ovs-ofctl show br0` reports `is_connected: false` | `ovs-vsctl get-controller br0` | Shows the controller URL | If the result is empty, run `ovs-vsctl set-controller br0 <url>` |
| The controller connects but does not install any flow | `ovs-ofctl show br0` and check the `OFPT_FEATURES_REPLY` version | OF 1.3 or later | If the version is OF 1.0, run `ovs-vsctl set Bridge br0 protocols=OpenFlow13` |
| Multi-controller deployment with no MASTER | `ovs-appctl ofproto/show-connection br0` | Role per controller | The controller must explicitly send `ROLE_REQUEST` MASTER |
| A flow rule is rejected | Check the role | `SLAVE` means write is denied | The controller must upgrade to MASTER first |
| TCP reconnect loop | `coverage/show rconn_*` and `journalctl -u ovs-vswitchd` | The backoff timer | Check the firewall and the controller listen port |
| A bundle transaction fails | `ovs-appctl bundle/show br0` | Bundle pending | Roll back: `ovs-ofctl bundle-abort` |

## 9.16.X Comparison with other systems (axis 20 backfill, 2026-04-27)

OVS controller failover has analogues in other stacks:

- **Cisco STP root election** (IEEE 802.1D). The bridge with the lowest priority and MAC wins the root role. Failover is 30 seconds with legacy STP and 6 seconds with RSTP (802.1w). The difference from OVS OpenFlow role: STP is a layer-2 spanning-tree topology election, while the OVS role is an OpenFlow control-plane authority election. The similarity: both use a tie-breaker (priority and ID) for the election.
- **ONOS distributed controller cluster.** ONOS uses Atomix and Raft for leader election. Each switch has a "master" instance inside the ONOS cluster. The difference from OVS: ONOS manages the election cluster-side, while OVS lets the switch pick a master through a role request from a candidate. The similarity: both have a "master, slave, equal" semantic.
- **Faucet HA mode.** Faucet is a different controller stack. Its high availability is implemented through `faucet --hot-reload` plus a load balancer in front. There is no distributed leader election at the Faucet level. It relies on external HA (`keepalived`, HAProxy). This differs from the native role-based failover in OVS.
- **OpenDaylight plus Atomix Raft.** ODL controller cluster members use Atomix Raft for leader election. This is similar to the OVS role request but it integrates more deeply with a cluster-aware datastore.
- **Kubernetes leader election** (the lease-lock pattern). `kube-controller-manager` and `kube-scheduler` use the `coordination.k8s.io/v1.Lease` lock and renew the TTL as a heartbeat. The difference from OVS is in the granularity (per controller instance versus per switch). The similarity is the lease-plus-heartbeat pattern.

**Core insight.** The OVS OpenFlow role is a per-switch election (each switch selects one master from N controllers). In contrast, Cisco STP, ONOS, and Kubernetes use cluster-side election (one leader is chosen from N members). Engineers who internalize this distinction avoid confusion when designing multi-controller deployments.

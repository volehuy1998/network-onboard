## Learning objectives

After completing this part, the engineer can:

1. **Understand** the architecture of the two CLI tools `ovn-nbctl` and `ovn-sbctl`: an OVSDB IDL client connecting to NBDB or SBDB over TCP or Unix socket, communicating via RFC 7047 transactions, with the ability to run in daemon mode to reduce latency.
2. **Analyze** the role of the two tools: `ovn-nbctl` configures logical intent (CMS-equivalent), while `ovn-sbctl` observes and intervenes on the SB database (debug and manual chassis-add). The two roles MUST NOT be swapped in production (Rule 5).

5. **Synthesize** a decision matrix of the form "in situation X use command Y", serving the on-call engineer in selecting the correct tool quickly.

## 13.14.1 CLI architecture of ovn-nbctl and ovn-sbctl

### 13.14.1.1 Two tools, two roles

`ovn-nbctl` and `ovn-sbctl` are two separate CLI tools in the package `ovn-common` (Ubuntu 22.04) or `ovn-utilities` (RHEL). Their architecture is identical (both are OVSDB IDL clients), but they connect to two different databases:

- **`ovn-nbctl`** connects to **NBDB** (default port 6641 TCP or Unix socket `/var/run/ovn/ovnnb_db.sock`). NBDB holds the **logical configuration intent** (Logical_Switch, Logical_Router, ACL, and so on). This is the tool that the CMS (Neutron, OVN-Kubernetes) calls indirectly to write configuration.
- **`ovn-sbctl`** connects to **SBDB** (default port 6642 TCP or Unix socket `/var/run/ovn/ovnsb_db.sock`). SBDB holds the **state compiled** by ovn-northd: Logical_Flow, Port_Binding, Chassis, Datapath_Binding. This is the **debug and manual intervention** tool for the on-call engineer.

> **Rule 5, control principle:** A production engineer MUST NOT use `ovn-sbctl` to configure logical intent (for example, to create a Port_Binding manually). All logical configuration must flow through NBDB to ovn-northd to SBDB. SBDB is the output of compilation; writing into it directly will be overwritten or cause inconsistency. The only legitimate use case for `ovn-sbctl set` and `add` is **chassis intervention** when a ghost chassis fails to release itself (see §13.14.5).

### 13.14.1.2 Connection, daemon mode, tracing

Each time `ovn-nbctl <command>` runs, the tool performs:

1. Read the environment variable `OVN_NB_DB` or the parameter `--db=` to learn the URL.
2. Open a connection (TCP or Unix) and send `monitor_cond_since` to obtain an OVSDB snapshot.
3. Build the IDL in memory (about 5 to 50 MB for a medium cluster).
4. Execute the command (read or write rows and relations).
5. Commit the transaction if there are changes.
6. Close the connection.

Steps 2 and 3 are the **bottleneck**: in a cluster with more than 10,000 objects, a single run may take 200 to 800 ms just to initialise the IDL. In a batch workflow (a CMS issuing 100 commands per second), this is a major obstacle.

**Daemon mode (the solution):**

```bash
# Start the daemon
[host]# ovn-nbctl --detach --pidfile=/var/run/ovn/ovn-nbctl.pid \
                  --log-file=/var/log/ovn/ovn-nbctl.log \
                  --db=tcp:nb1:6641,tcp:nb2:6641,tcp:nb3:6641
# Subsequent commands use the daemon socket:
[host]# export OVN_NB_DAEMON=$(ovn-nbctl get-daemon-socket)
[host]# ovn-nbctl ls-add my-ls
# ... command runs in less than 5 ms instead of 200 to 800 ms
```

The daemon keeps the IDL session warm and shares it across invocations. A production CMS typically runs the daemon as a long-lived process.

**Tracing options:**

- `--verbose` (`-v`) enables debug logging.
- `--timeout=N` sets the transaction timeout.
- `--no-wait` does not wait for commit confirmation (dangerous, use only for fire-and-forget).
- `--dry-run` for some subcommands: prints the transaction JSON without committing.

## 13.14.2 ovn-nbctl, 97 subcommands grouped into 12 categories

### 13.14.2.1 Generic group (3 commands)

| Command | Purpose | Example |
|---------|---------|---------|
| `init` | Create a new NBDB schema (run only when bootstrapping a fresh database) | `ovn-nbctl init` |
| `sync` | Wait for northd to finish compiling the current NB into SB | `ovn-nbctl sync` |
| `show` | Print a hierarchical view of the entire NBDB | `ovn-nbctl show` |

### 13.14.2.2 Logical Switch and Logical Switch Port group (28 commands)

**Logical Switch (LS):**

| Command | Purpose |
|---------|---------|
| `ls-add <name>` | Create a new LS |
| `ls-del <ls>` | Delete an LS |
| `ls-list` | List all LSes |

**Logical Switch Port (LSP):**

| Command | Purpose |
|---------|---------|
| `lsp-add <ls> <port>` | Create an LSP attached to an LS |
| `lsp-del <port>` | Delete an LSP |
| `lsp-list <ls>` | List the LSPs of an LS |
| `lsp-set-addresses <port> [<addresses>...]` | Set the MAC and IP for the LSP. Format: `"aa:bb:cc:dd:ee:ff 10.0.0.1"`, `unknown` (allow any MAC), `dynamic` (automatic IPAM), `router` (LSP connected to an LRP) |
| `lsp-get-addresses <port>` | Read back the addresses |
| `lsp-set-port-security <port> [<addresses>...]` | Set anti-spoofing. Default: synchronised with `lsp-set-addresses` |
| `lsp-get-port-security <port>` | Read back |
| `lsp-set-type <port> <type>` | Set the type: `""` (vif), `localnet`, `localport`, `l2gateway`, `l3gateway`, `router`, `virtual`, `external`, `remote` |
| `lsp-get-type <port>` | Read the type |
| `lsp-set-options <port> <KEY=VALUE>...` | Set the options dictionary. For example: `network_name=physnet1` for localnet, `router-port=lrp-name` for router type |
| `lsp-get-options <port>` | Read the options |
| `lsp-set-enabled <port> true\|false` | Enable or disable the LSP |
| `lsp-get-enabled <port>` | Read the enabled state |
| `lsp-get-up <port>` | Read the `up` field (true when a chassis has claimed it successfully) |
| `lsp-set-dhcpv4-options <port> <DHCP_Options uuid>` | Link the LSP to DHCPv4 options |
| `lsp-get-dhcpv4-options <port>` | Read back |
| `lsp-set-dhcpv6-options <port> <DHCP_Options uuid>` | Same as above for DHCPv6 |
| `lsp-get-dhcpv6-options <port>` | Read |
| `lsp-get-parent <port>` | Read parent_name (for a VLAN child port) |
| `lsp-get-tag <port>` | Read the VLAN tag |
| `lsp-get-ls <port>` | Reverse lookup: from a port find the LS that contains it |

### 13.14.2.4 ACL group (3 commands)

| Command | Purpose |
|---------|---------|
| `acl-add {<sw>\|<pg>} <direction> <priority> <match> <action>` | Add an ACL. Direction: `from-lport`, `to-lport`. Priority 0 to 32767. Action: `allow`, `allow-related`, `allow-stateless`, `drop`, `reject`. Match is an OVN expression (for example, `inport == "lsp1" && ip4.dst == 10.0.0.0/8`) |

| Command | Purpose |
|---------|---------|
| `pg-add <pg> [<port>...]` | Create a Port_Group. A Port_Group is used to consolidate ACLs: one ACL on a Port_Group applies to every member port (reducing flows by 5 to 10 times compared with one ACL per port) |
| `pg-set-ports <pg> [<port>...]` | Update the port list (full replacement) |
| `pg-del <pg>` | Delete a Port_Group |

### 13.14.2.6 Load Balancer group (3 commands)

| Command | Purpose |
|---------|---------|
| `lb-add <lb> <vip> <ips> [<protocol>]` | Create an LB. `vip` is in the form `ip:port` or `[ipv6]:port`. `ips` is a CSV list of pool members, for example `1.1.1.1:80,2.2.2.2:80`. Protocol: `tcp`, `udp`, `sctp` |
| `lb-del <lb> [<vip>]` | Delete the LB or only one VIP |
| `lb-list [<lb>]` | List LBs |

```bash

[host]# ovn-nbctl lr-lb-add my-router my-lb
```

## 13.14.4 Anatomy Template A, 10 important outputs

### Anatomy 1: `ovn-nbctl show`

> **When to run:** Overview of the current logical intent. First command when onboarding a new cluster or starting to debug.

```
switch f3a8 (web-ls)
    port web1
        addresses: ["aa:bb:cc:00:00:01 10.0.0.10"]
    port web2
        addresses: ["aa:bb:cc:00:00:02 10.0.0.11"]
    port lsp-router
        type: router
        router-port: lrp-web
router 89e1 (web-router)
    port lrp-web
        mac: "aa:bb:cc:99:00:01"
        networks: ["10.0.0.1/24"]
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| `switch` count | Matches the LS count CMS expects | 1 to 2 LS missing | Many LS missing | CMS sync lag; check Neutron or Kubernetes reconcile |
| `port` count per LS | Matches the active VM or container count | 1 to 2 ports missing | Duplicate IP warning | VM teardown did not clean the LSP |
| `addresses` MAC duplicate | Each MAC unique | One MAC on two LSPs | Multiple conflicts | An old LSP was not deleted; check `lsp-list` |
| `type: router` LSP | Matches the count of connected LRPs | Mismatch | LSP `router` without `router-port` option | One side was removed (LRP or LSP) |
| `mac` LRP format | `xx:xx:xx:xx:xx:xx` | (none) | Empty | Set incorrectly during `lrp-add` |
| `networks` CIDR | IP does not overlap with the LS | Overlap | LR has two LRPs in the same subnet | Wrong topology |

**Failure scenarios:**

- LSP `type: router` missing `router-port`: the peer LRP does not exist. Run `ovn-nbctl lsp-set-options <lsp> router-port=<lrp>` to relink.
- LR has two LRPs with the same `networks` 10.0.0.1/24: routing conflict. Move one to a different subnet.

### Anatomy 2: `ovn-sbctl list Chassis`

> **When to run:** Check which chassis are registered with the SB DB, their encap type, hostname, and transport zone.

```
_uuid               : a1b2c3d4-...
external_ids        : {ovn-bridge-mappings="physnet1:br-ex", \
                       ovn-cms-options="enable-chassis-as-gw", \
                       ovn-encap-ip="192.0.2.10"}
hostname            : "compute-01.example.com"
name                : "55b4d9e8-..."
nb_cfg              : 142
encaps              : [enc1, enc2]
other_config        : {bridge-mappings="physnet1:br-ex"}
transport_zones     : [zone1]
vtep_logical_switches : []
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| `name` UUID | Matches `system-id` of ovs-vsctl | Different | Empty | ovn-controller has not registered; check the log |
| `hostname` | Matches the actual OS `hostname` | Different | (none) | DNS lag or a rename that did not sync |
| `nb_cfg` | Matches SB_Global.nb_cfg within plus or minus 5 | Diverges by 10 to 50 | Diverges by more than 100 | Chassis lags ovn-northd sync |
| `encaps` count | One or more encap (Geneve, VXLAN, or STT) | 0 | (none) | The chassis has no tunnel; pinging another chassis will fail |
| `external_ids:ovn-encap-ip` | Routable underlay IP | IP in a different subnet | Empty | Check `ovs-vsctl get Open_vSwitch . external_ids` on the chassis |
| `transport_zones` | Matches the expected zone | Empty (means all-tenant) | Wrong zone | Tenant isolation is broken |

**Failure scenarios:**

- `nb_cfg` diverges by more than 100 from SB_Global: the chassis is stuck. Restart ovn-controller; if still diverged, the SB DB Raft is slow or the chassis cannot reach SB.
- Two chassis with the same `name` UUID: very rare, caused by cloning a VM image without deleting `system-id`. Delete one chassis with `ovn-sbctl chassis-del`.


| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| `chassis` UUID | Matches the local chassis | Empty | UUID does not exist | Stale binding; see chassis-del |
| `requested_chassis` | Matches `chassis` (after migration) | Different | Different for more than 30 seconds | Migration stuck; check the ovn-controller log |

| `mac` | MAC plus IP format | Empty (for `unknown`) | (none) | Not yet set |

### Anatomy 4: `ovn-sbctl lflow-list <datapath>`

> **When to run:** Inspect Logical_Flow that ovn-northd compiled from NB intent.

```
Datapath: "web-ls" (f3a8...)  Pipeline: ingress
  table=0 (ls_in_port_sec_l2),    priority=50, match=(inport == "web1"), action=(next;)
  table=1 (ls_in_port_sec_ip),    priority=90, match=(inport == "web1" && eth.src == aa:bb:cc:00:00:01 && ip4.src == {10.0.0.10}), action=(next;)
  ...
  table=27 (ls_in_l2_lkup),       priority=50, match=(eth.dst == aa:bb:cc:00:00:02), action=(outport = "web2"; output;)
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| Tables per pipeline | 27 ingress plus 10 egress (LS, OVN 22.03) | Diverges | (none) | Version baseline mismatch |
| `priority` | 1 to 32767 valid | 0 (default-deny) | More than 65535 | Compile error |
| `match` syntax | Valid OVN expression | Warning in the log | Empty | Compile fallback |
| `action` syntax | Valid | (none) | Empty (silent drop) | (per spec) |
| Missing table N | Tables 0 to 26 all have entries | 1 to 2 missing | Many missing | northd compile error; check northd.log |

## 13.14.6 Guided Exercise S67': New tenant + multi-tier ACL

**Goal:** Deploy tenant `T1` with two tiers (web and db), an external router, and an ACL that denies cross-tier traffic except on one port.

**Steps:**

```bash
# 1. Create the Logical Switch for each tier
[host]# ovn-nbctl ls-add t1-web-ls
[host]# ovn-nbctl ls-add t1-db-ls

# 2. Create VM ports
[host]# ovn-nbctl lsp-add t1-web-ls web1
[host]# ovn-nbctl lsp-set-addresses web1 "aa:bb:cc:11:00:01 10.1.1.10"
[host]# ovn-nbctl lsp-add t1-db-ls db1
[host]# ovn-nbctl lsp-set-addresses db1 "aa:bb:cc:22:00:01 10.2.2.10"

# 3. Create the Logical Router
[host]# ovn-nbctl lr-add t1-router
[host]# ovn-nbctl lrp-add t1-router lrp-web aa:bb:cc:99:00:01 10.1.1.1/24
[host]# ovn-nbctl lrp-add t1-router lrp-db  aa:bb:cc:99:00:02 10.2.2.1/24

# 4. Connect the LS to the LR (LSP type=router)
[host]# ovn-nbctl lsp-add t1-web-ls lsp-to-router-web
[host]# ovn-nbctl lsp-set-type lsp-to-router-web router
[host]# ovn-nbctl lsp-set-addresses lsp-to-router-web router
[host]# ovn-nbctl lsp-set-options lsp-to-router-web router-port=lrp-web

[host]# ovn-nbctl lsp-add t1-db-ls lsp-to-router-db
[host]# ovn-nbctl lsp-set-type lsp-to-router-db router
[host]# ovn-nbctl lsp-set-addresses lsp-to-router-db router
[host]# ovn-nbctl lsp-set-options lsp-to-router-db router-port=lrp-db

# 5. Port_Group per tier
[host]# ovn-nbctl pg-add t1-web-pg web1
[host]# ovn-nbctl pg-add t1-db-pg  db1

# 6. ACL: deny web to db except on port 5432 (Postgres)
[host]# ovn-nbctl acl-add t1-web-pg from-lport 1000 \
            "outport == @t1-db-pg && tcp.dst != 5432" drop
[host]# ovn-nbctl acl-add t1-web-pg from-lport 999 \
            "outport == @t1-db-pg && tcp.dst == 5432" allow-related

# 7. Verify
[host]# ovn-nbctl show
[host]# ovn-nbctl acl-list t1-web-pg
[host]# ovn-sbctl lflow-list t1-web-ls | grep "ls_in_acl"
```

**Expected output (doc-plausible, to be verified once a lab is available):**

```
$ ovn-nbctl acl-list t1-web-pg
from-lport  1000 (outport == @t1-db-pg && tcp.dst != 5432)  drop
from-lport   999 (outport == @t1-db-pg && tcp.dst == 5432)  allow-related
```

**Lessons learned:**

1. The tenant-creation workflow needs 8 to 12 `ovn-nbctl` commands. Daemon mode brings the time down from about 10 seconds to about 500 milliseconds.
2. `lsp-set-type router` plus `lsp-set-options router-port=` is the MANDATORY pair to connect an LS to an LR. Without `router-port`, packets entering the LRP will be dropped.
3. Port_Group plus ACL `outport == @<pg>` is the standard pattern for multi-tier security. Compared with two ACLs from web1 to db1 (one rule per VM pair), the Port_Group pattern scales as O(N) instead of O(N squared).

**Cleanup:**

```bash
[host]# ovn-nbctl ls-del t1-web-ls
[host]# ovn-nbctl ls-del t1-db-ls
[host]# ovn-nbctl lr-del t1-router
[host]# ovn-nbctl pg-del t1-web-pg
[host]# ovn-nbctl pg-del t1-db-pg
```

(The ACLs are cascade-deleted when the Port_Group is deleted.)

## 13.14.7 Capstone POE Phase I.B1: ovn-sbctl debug vs ovn-nbctl reconfig

> **Bối cảnh:** Production cluster, VM `web1` báo lỗi "no network" sau live migration từ chassis A → B.

**Tuyên bố cần kiểm chứng:** "Chạy `ovn-sbctl set Port_Binding web1 chassis=<chassis-B>` để force bind là cách đúng để fix."

**Predict:** Sai. Force binding qua ovn-sbctl là intervention danger cho production.

**Observe (giả định, kiểm chứng khi có lab):**

- Chạy `ovn-sbctl set Port_Binding web1 chassis=...` → gói tin có flow trên br-int chassis B trong vài giây.
- Sau 30 giây ovn-northd recompute từ NB → reset `Port_Binding.chassis` về giá trị cũ (hoặc empty) → flow biến mất, VM mất network lần 2.
- Đồng thời, `requested_chassis` của Port_Binding trong NB không phản ánh ý đồ → CMS (Neutron) thấy bất đồng và có thể tự revert.

**Explain:**

1. SBDB là **output** của compile NB → SB. Ghi thẳng vào SBDB sẽ bị overwrite mỗi cycle compile (~50 ms - 5 giây tuỳ workload).
2. NBDB là source of truth. Lệnh đúng:
   - `ovn-nbctl lsp-set-options web1 requested-chassis=<chassis-B>` (set ý đồ ở NB).
   - ovn-northd compile lại → cập nhật `Port_Binding.requested_chassis` → ovn-controller B claim.
3. Trường hợp duy nhất hợp lệ dùng `ovn-sbctl set` lên Port_Binding là khi NB đã đúng nhưng SB stuck (rất hiếm, debug only). Phải hiểu trade-off: khoảnh khắc force tới khoảnh khắc northd recompute next.

**Falsification test:**

- Force bind qua `ovn-sbctl set`, đợi 30 giây.
- Run `ovn-nbctl get Logical_Switch_Port web1 options` xem có `requested-chassis` không.
- Nếu không có → northd sẽ revert, predict đúng. Nếu có (ai đó đã set NB cùng lúc) → có thể giữ.

**Tuyên bố sai ở đâu:** Dùng `ovn-sbctl set` cho ý đồ logical là **anti-pattern**. Đúng phải là `ovn-nbctl` (source of truth). `ovn-sbctl` chỉ dùng để (a) read/inspect, (b) chassis-del manual khi node ghost, (c) debug short-term với hiểu rõ rủi ro overwrite.

> **Quy tắc 5 trụ cột (Rule kiểm soát):** Ý đồ logical → NBDB qua ovn-nbctl. Quan sát + intervention chassis → SBDB qua ovn-sbctl. KHÔNG hoán đổi.

## 13.14.8 Điểm cốt lõi cần nhớ

**Một.** `ovn-nbctl` (97 lệnh) cấu hình ý đồ logical NBDB. `ovn-sbctl` (15 lệnh) inspect + intervention SBDB. Vai trò khác nhau, không hoán đổi (Rule 5 trụ cột).

**Hai.** Daemon mode giảm latency 100-1000x cho workload batch. CMS production luôn dùng daemon.

**Ba.** Command grouping: 12 nhóm cho ovn-nbctl (Generic / LS+LSP / LR+LRP / ACL / PG / LB / DHCP / QoS / HA Chassis / CoPP / Connection / SSL). 15 lệnh ovn-sbctl tập trung debug + chassis lifecycle.

**Bốn.** `lsp-set-type router` + `lsp-set-options router-port=` là cặp BẮT BUỘC cho LS-LR connection. Thiếu một trong hai → packet drop ngầm.

**Năm.** Port_Group + ACL pattern: `acl-add <pg> from-lport <pri> "outport == @<other-pg>" <action>`. Scale O(N) thay vì O(N²). Production sẽ thấy ngay ở 100+ tenant.

**Sáu.** `nb_cfg` gauge: `ovn-nbctl get NB_Global . nb_cfg` vs `ovn-sbctl get SB_Global . nb_cfg`. Lệch nhỏ (≤ 5) = healthy. Lệch > 100 = compile lag.

**Bảy.** `ovn-sbctl set Port_Binding chassis=...` là **anti-pattern** trừ debug short-term. Đúng phải set `requested-chassis` qua `ovn-nbctl lsp-set-options`.

### 13.14.9.1 Tất cả ovn-nbctl options (verified `ovn-nbctl(8)` man page)

`ovn-nbctl` chấp nhận hơn 30 option dòng lệnh chia 8 nhóm. Mỗi option ảnh hưởng cách connect, transaction behavior, output format, hoặc lifecycle.

#### Database connection (4 option)

| Option | Mặc định | Vai trò |
|--------|----------|---------|
| `--db=<db>` | `unix:/run/ovn/ovnnb_db.sock` hoặc `$OVN_NB_DB` | NB DB endpoint (Unix socket, TCP, hoặc SSL) |
| `--leader-only` | mặc định ON | Connect chỉ leader của Raft cluster |
| `--no-leader-only` | OFF | Cho phép connect follower (read-only OK, write redirect) |
| `--shuffle-remotes` | OFF | Random shuffle thứ tự remote cho fault tolerance |

#### Transaction behavior (5 option)

| Option | Mặc định | Vai trò |
|--------|----------|---------|
| `--no-wait` / `--wait=none` | mặc định | Return ngay khi NB write commit, KHÔNG đợi SB compile |
| `--wait=sb` | OFF | Block đến khi `ovn-northd` compile xong NB → SB |
| `--wait=hv` | OFF | Block đến khi tất cả chassis confirm apply OF flow |
| `--print-wait-time` | OFF | In ra thời gian thực sự đã đợi cho `--wait=sb` hoặc `--wait=hv` (ms granularity); không tác dụng nếu `--no-wait` |
| `--commit-retry=<N>` | mặc định 0 | Retry transaction N lần nếu conflict |
| `--unix-lock=yes\|no` | yes | Lock file Unix socket cho exclusive access |

**Diễn giải `--wait` value + trade-off:**

```
--no-wait (default):    < 1ms latency, NB write commit, SB chưa compile
--wait=sb:              100-500ms, sb_cfg = nb_cfg, SB ready
--wait=hv:              1-10s, hv_cfg = nb_cfg, data plane ready
```

Production automation nên dùng `--wait=sb` cho balance latency-correctness.

**Diễn giải `--print-wait-time` (gap closure v3.6 Phase 1).** Khi tự động hóa apply hàng nghìn ovn-nbctl command với `--wait=sb` hoặc `--wait=hv`, thời gian đợi convergence biến thiên nặng theo trạng thái cluster (10ms khi rảnh, 5-10s khi northd backlogged). Mặc định ovn-nbctl chỉ trả exit code; thêm `--print-wait-time` đẩy ra stderr 1 dòng dạng `Time spent on waiting: 137 ms` cho mỗi transaction, giúp script tracking percentile p50/p95/p99 và cảnh báo sớm convergence drift. Use case điển hình: orchestrator chấp nhận p95 đợi sb < 500ms; nếu > thì alert. Anti-pattern: dùng `--print-wait-time` mà không bật `--wait=...` — output sẽ trống, gây nhầm operator nghĩ "đợi 0ms" trong khi thực tế không đợi. Cross-link 20.2 §lflow-cache tunable cho cách giảm convergence latency.

| Option | Mặc định | Vai trò |
|--------|----------|---------|

```bash
# Khởi động daemon dùng socket mặc định
ovn-nbctl --daemon

# Set env để mọi lệnh sau dùng daemon socket

```

**Diễn giải `-u <path>` (gap closure v3.6 Phase 1).** Mặc định ovn-nbctl daemon tạo socket trong `$OVS_RUNDIR` (thường `/var/run/ovn`). Trên môi trường multi-tenant container hoặc test bench chạy nhiều ovn-nbctl daemon song song (mỗi orchestration job giữ daemon riêng), nhiều daemon dùng cùng socket path = collision. `-u /tmp/job-A.sock` tách socket theo job. Khi client invocation cũng dùng `-u /tmp/job-A.sock`, lệnh đi tới đúng daemon. Anti-pattern: start daemon với `-u` nhưng client không truyền `-u` → client connect mặc định path, không thấy daemon → fallback IDL connection (slow). Cross-reference: same `-u` semantics tồn tại ở `ovs-vsctl --daemon` và `ovn-sbctl --daemon`.

### 13.14.9.2 `ovn-trace`, simulator logical pipeline cho debug what-if

**Khái niệm.** `ovn-trace` là utility C standalone (binary `/usr/bin/ovn-trace` cài cùng gói `ovn-common`) đọc SBDB rồi simulate đường đi của một packet giả tưởng qua **logical pipeline** OVN, không inject packet thật vào datapath, không gửi gói qua dây, không thay đổi state. Input là cặp `(datapath, microflow_expression)`: datapath là tên Logical_Switch hoặc Logical_Router; microflow là biểu thức ngôn ngữ OVN mô tả header + metadata gói tin (`inport == "lsp1" && eth.src == 50:54:00:00:00:01 && ip4 && ip4.dst == 10.0.0.10 && tcp && tcp.dst == 80`). Output là cây trace từng stage (`ls_in_port_sec_l2`, `ls_in_pre_acl`, `ls_in_acl`, `ls_in_lb`, ..., `lr_in_admission`, `lr_in_ip_input`, ...) với match đã hit, action đã chạy, register sau từng bước, kèm pipeline transition (ingress/egress, ls→lr, lr→ls). Phạm trù: CLI tool standalone, debug-class, read-only, what-if.

**Lịch sử + bối cảnh ra đời.** Ben Pfaff (Nicira/VMware) commit `utilities/ovn-trace.c` lần đầu trong giai đoạn 2016-2017 khi OVN tách ra khỏi OVS và xây dựng pipeline 24-stage đầu tiên. Header copyright file ghi rõ "Copyright 2016-2017 by Nicira, Inc.". Mục tiêu nguyên thuỷ: cho phép kỹ sư trả lời câu hỏi "nếu tôi gửi packet X, nó sẽ đi qua những stage nào, đến đâu thì bị drop, NAT đổi địa chỉ thành gì" mà không cần dựng VM thật rồi tcpdump. OVN 2.7 (2017) ship `ovn-trace` chính thức trong distribution. Trước đó, kỹ sư phải đọc dump `Logical_Flow` bằng `ovn-sbctl lflow-list` rồi tự simulate trong đầu, tốn hàng giờ cho mỗi packet phức tạp. OVN 20.06 thêm `--lb-dst` cho debug Load_Balancer hairpin. OVN 21.06 thêm `--minimal`/`--summary`/`--all` separate output mode. OVN 22.03 (curriculum baseline) thêm `--friendly-names` (mặc định) thay UUID bằng tên Logical_Switch/Logical_Router để output đọc được. OVN 24.03 cải thiện performance đọc SBDB cho cluster lớn.

**Vị trí trong kiến trúc.** `ovn-trace` chạy ở **layer tooling client-side**, không phải data plane (br-int) cũng không phải control plane (ovn-northd, ovn-controller). Nó connect SBDB qua OVSDB JSON-RPC giống `ovn-sbctl`, đọc `Datapath_Binding`, `Port_Binding`, `Logical_Flow`, `Multicast_Group`, `MAC_Binding`, `DNS`. Toàn bộ simulation diễn ra trong process `ovn-trace`, không round-trip với `ovn-controller` hay chassis. Scope: per-invocation (mỗi lần chạy là một snapshot SBDB tại thời điểm đó). Khác `ofproto/trace` (Phần 9.25) ở chỗ ofproto/trace simulate physical OpenFlow pipeline trên một bridge cụ thể, còn `ovn-trace` simulate logical pipeline trừu tượng cấp OVN.

**Vai trò.** Là bước 1 trong workflow debug 3-tầng OVN: logical (`ovn-trace`) → physical (`ofproto/trace`) → forensic (`ovn-detrace`). `ovn-trace` trả lời câu hỏi cấp **intent**: "theo NBDB hiện tại, packet này có được phép qua không, đi qua những logical stage nào, có bị ACL chặn, có bị NAT, có routing đúng không". Nếu logical pipeline đã drop, không cần xuống physical. Nếu logical pass nhưng thực tế packet thật fail, lỗi nằm ở binding hoặc physical (chassis chưa pickup, tunnel hỏng, conntrack stuck), khi đó dùng `ofproto/trace` + `ovn-detrace`. Vai trò scope: toàn bộ vòng đời packet logical, từ ingress LSP đầu vào đến egress LSP đầu ra (hoặc drop tại stage nào đó).

**Vì sao sinh ra.** OVN compile NBDB (intent) thành SBDB `Logical_Flow` (~vài nghìn entry cho cluster trung bình) rồi `ovn-controller` translate thành OpenFlow trên br-int (~vài chục nghìn rule). Khi user khai báo ACL/Route/LB sai, lỗi không hiện ngay; chỉ lộ khi packet thật chạy và bị drop. Trước `ovn-trace`, debug pattern là "thử nghiệm thực tế": dựng VM, ping, tcpdump, đoán. Pain point: (a) chậm, mỗi thử nghiệm tốn vài phút setup; (b) không reproducible, packet thực tế có thể bị ảnh hưởng bởi conntrack state, tunnel state, MAC learning; (c) không safe trong production, không thể "thử ping" mọi tổ hợp. `ovn-trace` lấp gap: simulation thuần, không tác động hệ thống, có thể chạy hàng nghìn lần/giờ trên cluster production mà không ảnh hưởng. Đây là tool unique cho OVN; OVS thuần không có analogue tương đương vì OVS không có khái niệm "logical intent" tách khỏi flow physical.

**Vấn đề giải quyết.** Hai nhóm vấn đề chính. (1) **What-if simulation**: "nếu tôi tạo ACL này, packet TCP port 8080 từ tenant A có còn pass không" — chạy `ovn-trace` trước khi `ovn-nbctl acl-add` để verify; (2) **Drop diagnosis**: "tại sao packet TCP port 80 từ vm1 không tới vm2" — `ovn-trace` chỉ ra stage drop đầu tiên (LS_IN_ACL priority 2002, hoặc LR_IN_IP_ROUTING không có route, hoặc LS_IN_PORT_SEC_IP do source IP không khớp `lsp-set-addresses`). Không thay thế hoàn toàn debug thực tế, nhưng giảm 80-90% thời gian khoanh vùng vì xác định được layer (logical vs physical vs binding) trong vài giây.

**Tầm quan trọng.** Tier 1 mandatory cho mọi kỹ sư OVN. Một engineer không biết `ovn-trace` thì khi gặp incident OVN sẽ phải đoán mò qua `ovn-sbctl lflow-list` thủ công. Trong môi trường production OpenStack với hàng trăm Logical_Switch và hàng nghìn ACL, không có `ovn-trace` thì MTTR (mean time to resolution) tăng từ vài phút lên vài giờ. So với cặp `ovn-detrace`/`ofproto/trace`, `ovn-trace` thường là tool phải dùng ĐẦU TIÊN vì nó nhanh nhất (không cần ssh vào chassis, chỉ cần connect SBDB).

**Cơ chế hoạt động.** `ovn-trace` đọc toàn bộ SBDB vào memory (function `read_db()` trong `utilities/ovn-trace.c`), build internal datapath/port/flow table. Khi nhận microflow expression, parser (`parse_options()` + expression parser) chuyển expression thành struct flow internal kèm prerequisite tự động (ví dụ chỉ định `ip4.dst` thì tự áp `eth.type == 0x0800`). Hàm core `trace__()` recursive: tại mỗi stage (table_id), gọi `ovntrace_flow_lookup()` tìm Logical_Flow priority cao nhất khớp microflow, gọi `trace_actions()` để execute từng action OVN logical (`next;`, `outport = ...;`, `ct_lb;`, `ct_next;`, `arp { ... };`, `output;`, ...). Hàm dispatcher cho từng action: `execute_output()` xử lý output (single port hoặc multicast group, transition pipeline ingress/egress), `execute_ct_next()` xử lý conntrack state transition, các handler khác cho `nd_na`, `arp`, `dhcp_offer`, `set_meter`, etc. Hàm `ovntrace_node_append()` build cây trace hierarchical, mỗi node là 1 stage hoặc 1 action sub-call. Hàm `ovntrace_node_print_details()` format output thành text thụt lề. Invariant: simulation deterministic per snapshot SBDB; nếu SBDB không đổi, mỗi lần chạy cùng input = cùng output. Non-deterministic chỉ xuất hiện với `select` group (Load_Balancer multi-backend, ECMP) khi `--all` show toàn bộ candidate path.

**Phân loại.** CLI tool standalone, debug class, read-only. Cùng họ debug OVN: `ovn-detrace` (cookie reverse decoder, Phần §13.14.9.3 ngay dưới), `ovn-sbctl lflow-list` (logical flow dumper raw), `ofproto/trace` qua `ovs-appctl` (OVS physical trace, Phần 9.25). Khác `ovn-nbctl`/`ovn-sbctl` ở chỗ không có write subcommand, chỉ một chế độ đầu vào duy nhất (microflow expression). Khác `ofproto/trace` ở chỗ chỉ cần endpoint SBDB, không cần ssh vào chassis cụ thể.

**Cách kỹ sư vận hành thành thạo.** Bốn kỹ năng. (1) **Soạn microflow expression**: nắm 24+ field thường dùng (`inport`, `outport`, `eth.src/dst/type`, `ip4/ip6.src/dst`, `ip.proto`, `ip.ttl`, `tcp/udp.src/dst`, `tcp.flags`, `icmp4/icmp6.type/code`, `arp.op/spa/tpa/sha/tha`, `nd.target`, `ct.new/est/rpl/inv/trk`, `vlan.tci`, `reg0..reg15`, `outport`, `flags.loopback`); biết prerequisite chain (chỉ định `tcp.dst` thì phải có `ip4` hoặc `ip6` + `ip.proto == 6`); biết operator `&&`, `==`, `!=`, value bitmask (`reg0[0] == 1`). (2) **Đọc output**: phân biệt ingress vs egress pipeline, đọc tên stage để map về NBDB intent (`ls_in_acl` ↔ ACL on Logical_Switch, `lr_in_ip_routing` ↔ static_route trên Logical_Router), đọc action `next;` (đi sang stage kế tiếp), `outport = ...; output;` (xuất logical port), `drop;` (kết thúc, không tới đích). (3) **Chọn output mode**: `--detailed` mặc định (verbose, mọi register state), `--summary` (compact, chỉ stage + action), `--minimal` (chỉ packet modification + output, bỏ qua stage chỉ có `next;`), `--all` (chạy cả 3 mode). (4) **Decision tree khi trace fail expectation**: stage drop → đọc match expression của Logical_Flow tại stage đó qua `ovn-sbctl lflow-list <datapath> | grep <stage>`; trace pass nhưng packet thật fail → bước sang `ofproto/trace` + `ovn-detrace`; trace error "no datapath found" → kiểm tra tên datapath bằng `ovn-nbctl show`.

**Quy trình sử dụng.** Workflow chuẩn 5 bước cho production debug.

```bash
# Bước 1: simulate logical, xác định pipeline có pass hay drop
ovn-trace ls-prod \
  'inport == "lsp-vm1" && eth.src == 50:54:00:00:00:01 \
   && eth.dst == 50:54:00:00:00:02 && ip4 \
   && ip4.src == 10.0.0.1 && ip4.dst == 10.0.0.2 \
   && ip.ttl == 64 && tcp && tcp.dst == 80 && tcp.src == 33000'

# Bước 2: nếu trace drop, tăng verbosity để xem chính xác Logical_Flow nào match
ovn-trace --detailed ls-prod '...'

# Bước 3: nếu trace pass nhưng packet thật fail, sang physical
ovs-appctl ofproto/trace br-int 'in_port=<ofport>,tcp,...'

# Bước 4: annotate physical trace bằng ovn-detrace
ovs-appctl ofproto/trace br-int '...' | ovn-detrace \
  --ovnnb=tcp:nb:6641 --ovnsb=tcp:sb:6642

# Bước 5: dùng --ovs để đối chiếu logical với physical trên chassis nhất định
ovn-trace --ovs=tcp:chassis-host:6640 ls-prod '...'
```

Best practice: luôn quote toàn bộ microflow trong single quote để shell không expand `&&`; dùng `--friendly-names` mặc định OVN 22.03 đã bật; với cluster Raft, có thể connect bất kỳ replica SBDB (read-only); pin `OVN_SB_DB` env var để tránh gõ `--db` mỗi lần.

**Khi xảy ra sự cố.** Tình huống điển hình kỹ sư phải nhớ ngay đến `ovn-trace`. (a) **East-west drop**: vm1 không ping được vm2 cùng Logical_Switch; chạy `ovn-trace ls-blue 'inport=="vm1" && eth.dst==<vm2-mac> && ip4 && ip4.dst==<vm2-ip> && icmp4 && icmp4.type==8'` để xem stage drop đầu tiên. (b) **North-south fail**: VM không ra Internet được; chạy `ovn-trace lr-edge '...'` từ Logical_Router perspective xem có hit `lr_in_ip_routing` không, có NAT đúng không. (c) **ACL verify trước commit**: trước khi `ovn-nbctl acl-add` rule mới, chạy `ovn-trace` cho 5-10 traffic pattern điển hình để xác nhận không break traffic hợp lệ. (d) **DHCP debug**: `ovn-trace --all` cho UDP port 67 broadcast để xem stage `ls_in_dhcp_options` có generate response không. (e) **Load_Balancer**: dùng `--lb-dst=<backend-ip>` để xác nhận VIP packet được DNAT đúng backend. **Liên quan mật thiết.** Năm thành phần. (1) **SBDB `Logical_Flow` table** (Phần 13.1): nguồn dữ liệu chính, mỗi stage trong trace đọc Logical_Flow tương ứng. (2) **`ovn-northd`** (Phần 13.8): bên compile NBDB → SBDB sinh ra Logical_Flow mà `ovn-trace` simulate; nếu northd chưa sync, `ovn-trace` thấy phiên bản cũ. (3) **`ofproto/trace`** (Phần 9.25): tool song song, simulate physical OpenFlow; thường dùng nối tiếp sau `ovn-trace`. (4) **`ovn-detrace`** (Phần §13.14.9.3): cầu nối ngược physical → logical, complement `ovn-trace`. (5) **Pipeline stage catalog** (Phần 13.16, 13.19): tên stage trong trace output (`ls_in_*`, `lr_in_*`, `ls_out_*`, `lr_out_*`) cần mapping với table_id để đọc hiểu. Loose related: `ovn-nbctl` (chỉ ảnh hưởng gián tiếp qua northd compile), `ovn-controller` (không ảnh hưởng vì ovn-trace bypass br-int).

**Source code tham chiếu.** File chính: `utilities/ovn-trace.c` (OVN repo, branch-22.03, ~3500 dòng C). Hàm chính:

- `main()` → `parse_options()` → `read_db()` → `trace()` → `ovntrace_node_print_details()`.
- `read_db()`: đọc SBDB, build `ovntrace_datapath` array + `ovntrace_port` array.
- `ovntrace_datapath_find_by_name()`: lookup datapath theo tên hoặc UUID prefix.
- `trace__()`: core recursive simulator, gọi `ovntrace_flow_lookup()` tại mỗi table.
- `trace_actions()`: parse và dispatch action OVN logical.
- `execute_output()`: xử lý output action, transition pipeline ingress↔egress, multicast.
- `execute_ct_next()`: simulate conntrack state transition, dùng `--ct` flag cho default.

**Trường hợp sự cố thực tế.** Sự cố production OpenStack 800 VM, OVN 22.03.4, 12 chassis. Symptom: tenant A báo từ ngày deploy ACL policy mới, một số request HTTP từ subnet `10.10.0.0/24` đến API server (LB VIP `10.20.0.50:8080`) bị timeout ngẫu nhiên ~15% rate; tenant B (subnet `10.11.0.0/24`) dùng cùng LB không bị. Investigation step 1: chạy `ovn-trace lr-edge 'inport == "lsp-vm-A" && eth.src == ... && ip4 && ip4.src == 10.10.0.5 && ip4.dst == 10.20.0.50 && ip.ttl == 64 && tcp && tcp.dst == 8080 && tcp.src == 33000'` → trace hit `lr_in_ip_routing`, sau đó vào `ls_in_pre_lb`, `ls_in_lb`. Step 2: `ovn-trace --all --lb-dst=10.20.0.51 ...` để force chọn backend cụ thể → trace pass. Step 3: chạy lại không `--lb-dst`, đọc kỹ output `ls_in_lb` → chỉ ra `ct_lb(backends=10.20.0.51,10.20.0.52)` không bao gồm `10.20.0.53` mà tenant A đợi. Step 4: kiểm tra NBDB: `ovn-nbctl lb-list lb-api` → đúng là LB chỉ có 2 backend, nhưng theo design phải có 3. Step 5: root cause: deploy script lúc rolling update đã `lb-del` backend `10.20.0.53` quên `lb-add` lại. Fix: `ovn-nbctl --may-exist lb-add lb-api 10.20.0.50:8080 10.20.0.51:8080,10.20.0.52:8080,10.20.0.53:8080 tcp`. Sau 3 giây, `ovn-trace` chạy lại → hit cả 3 backend với `--all`, drop rate về 0%. Giá trị `ovn-trace` ở đây: trong mớ NBDB hàng trăm LB, không có `ovn-trace` thì engineer sẽ phải đọc NBDB thủ công và đoán backend nào miss; với `ovn-trace --all`, sai lệch lộ ra trong 30 giây.

Variations: (a) đổi action ACL từ `drop` sang `reject`, observe stage output có thêm action gen ICMP unreachable; (b) thêm stateful `allow-related` ở priority cao hơn, observe pipeline rẽ vào `ct_commit`; (c) thử microflow thiếu prerequisite (chỉ định `tcp.dst` mà bỏ `ip4`) → `ovn-trace` báo lỗi parser; (d) chạy `ovn-trace` trong khi `ovn-northd` đang stop: trace vẫn chạy được (đọc snapshot SBDB cũ) nhưng không phản ánh thay đổi NBDB chưa compile.

**Lỗi thường gặp + tín hiệu chẩn đoán.** Bốn failure mode chính. (1) **Endpoint sai SBDB**: `ovn-trace` không thấy SBDB; tín hiệu: lỗi "could not connect to OVN_Southbound" hoặc treo; chẩn đoán: kiểm tra `OVN_SB_DB` env var hoặc flag `--db`, test bằng `ovn-sbctl --db=$OVN_SB_DB list Datapath_Binding | head`; recovery: trỏ đúng socket UNIX `unix:/var/run/ovn/ovnsb_db.sock` hoặc TCP `tcp:sb-host:6642`. (2) **Datapath name sai**: `ovn-trace` báo "datapath not found"; chẩn đoán: `ovn-nbctl show` xem tên Logical_Switch/Logical_Router; recovery: dùng tên chính xác hoặc UUID prefix; lưu ý OVN 22.03+ `--friendly-names` cho phép tên người, OVN cũ chỉ chấp nhận UUID. (3) **Microflow expression syntax**: parser từ chối; tín hiệu: "syntax error" với line:column; chẩn đoán: kiểm tra prerequisite (dùng `ip.proto` thì cần `ip4` hoặc `ip6`), kiểm tra dấu nháy MAC (`50:54:00:00:00:01` không có quote, nhưng port name `lsp1` cần `"lsp1"`); recovery: simplify expression, build tăng dần. (4) **Trace mismatch thực tế**: `ovn-trace` pass nhưng packet thật fail; tín hiệu: mâu thuẫn between trace và tcpdump; chẩn đoán: nguyên nhân thường là (a) SBDB chưa compile xong (chờ vài giây), (b) chassis chưa pickup Port_Binding (kiểm tra `ovn-sbctl list Port_Binding | grep chassis`), (c) tunnel down giữa chassis (kiểm tra `ovs-appctl ofproto/trace br-int`); recovery: bước sang `ofproto/trace` + `ovn-detrace` để debug physical layer.

**So sánh với hệ khác.** Trong NSX-T, tool tương đương là **Traceflow** (UI) hoặc API `nsxcli > get logical-switch <uuid> traceflow`, simulate gói qua DFW/LR. Khác `ovn-trace`: NSX-T Traceflow inject packet thật vào DFW pipeline để trace, có ảnh hưởng nhỏ đến state; `ovn-trace` thuần simulation, zero-impact. Cisco ACI có **Visibility & Troubleshooting Wizard** simulate end-to-end, nhưng kết hợp control plane + data plane, không tách rõ. Linux iptables không có analogue gần (chỉ có `iptables -t filter -L -v -n` show counter, không simulate path). Đặc điểm độc đáo của `ovn-trace`: được thiết kế cho mô hình **declarative intent → compiled flow → physical execution** đặc trưng của OVN, là một trong những lý do OVN dễ debug hơn nhiều SDN controller khác. Pattern "what-if simulator on declarative SDN" mà `ovn-trace` thể hiện là hình mẫu đáng học cho bất kỳ ai thiết kế controller SDN tự xây.

### 13.14.9.3 `ovn-detrace`, cầu nối ngược từ OpenFlow cookie về Logical_Flow

**Khái niệm.** `ovn-detrace` là utility Python (cài cùng OVN distribution) đọc luồng văn bản trace OpenFlow (output của `ofproto/trace`, hoặc của `ovs-ofctl dump-flows br-int`), trích cookie 32-bit từ mỗi dòng có dạng `cookie=0xXXXXXXXX`, sau đó tra ngược trong SBDB `Logical_Flow` để annotate dòng đó với UUID logical flow gốc, tên datapath (Logical_Switch hay Logical_Router), tên stage (`ls_in_acl`, `lr_in_ip_input`, ...), match expression và action ngôn ngữ logical, và xa hơn nữa là tên ACL/Load_Balancer/Logical_Router_Static_Route trong NBDB sinh ra Logical_Flow đó. Bản thân `ovn-detrace` không gửi packet, không gọi `ovs-vswitchd`, không thay đổi state; nó là decoder thuần text → SBDB lookup. Nói cách khác: `ovn-trace` simulate logical pipeline ở mức ngôn ngữ OVN, `ofproto/trace` simulate physical pipeline ở mức OpenFlow, còn `ovn-detrace` là **cây cầu** giữa hai mức bằng cách giải mã cookie mà ovn-northd đã encode.

**Lịch sử + bối cảnh ra đời.** OVN từ phiên bản đầu (OVN 2.6, 2016) đã encode UUID của `Logical_Flow` thành cookie 32-bit khi ovn-northd compile NBDB → SBDB; `ovn-controller` mỗi chassis lấy cookie đó dán vào physical OpenFlow rule khi cài lên `br-int`. Ban đầu engineer phải tự mở `ovn-sbctl list Logical_Flow | grep <prefix>` để dò ngược, tốn thời gian khi SBDB có hàng vạn entry. Ben Pfaff (Nicira/VMware) commit `utilities/ovn-detrace.in` vào branch master cuối 2016 (script Python đầu tiên ~250 dòng, sinh thành `ovn-detrace` qua autotools template `.in`). Mục tiêu nguyên thuỷ: phục vụ debug forensic ở quy mô nghìn datapath, nơi grep tay không scale. Từ OVN 20.06, script chuyển sang Python 3 thuần, giảm dependency. OVN 22.03 thêm option `--no-leader-only` cho cluster Raft và `--ovsdb-sock` cho UNIX socket. OVN 24.03 cải thiện format output với block separator rõ hơn cho từng cookie hit.

**Vị trí trong kiến trúc.** `ovn-detrace` là client-side script chạy trên bất kỳ host nào reach được hai DB (NBDB + SBDB). Trong stack OVN nó nằm ở **layer tooling**, không phải data plane (br-int) cũng không phải control plane (ovn-northd, ovn-controller). Nó đọc SBDB qua OVSDB protocol JSON-RPC giống `ovn-sbctl`, đọc NBDB cũng qua OVSDB protocol để lấy NB resource name (ACL match string, LB name). Nó không subscribe vào notification, mỗi lần invoke là một snapshot.

**Vai trò.** Trong workflow debug 3 bước (logical → physical → forensic), `ovn-detrace` đảm nhiệm bước thứ ba: chuyển dump physical bí ẩn (hàng trăm dòng `cookie=0x...,table=N,priority=...`) thành report có thể đọc được, mỗi dòng kèm comment "đây là Logical_Flow Y, sinh từ ACL Z của Logical_Switch K". Không có nó, kỹ sư on-call phải làm thủ công 3 bước: copy cookie → `ovn-sbctl find Logical_Flow logical_dp_group=...` → match prefix UUID → join với NB. Với 1 packet đi qua 24 stage, đó là 24 lần grep; không khả thi trong incident.

**Vì sao sinh ra.** OVN tách rõ "logical intent" (NBDB) và "physical execution" (br-int OpenFlow), nhưng cái giá phải trả là có 2 lớp trừu tượng giữa user CLI và packet thật. Khi packet đi sai, kỹ sư cần đi từ symptom ngược về intent. Gap mà `ovn-detrace` lấp: trace tool có sẵn (`ofproto/trace`) chỉ biết OpenFlow, không biết Logical_Flow; `ovn-trace` chỉ biết Logical_Flow, không biết physical match đã xảy ra hay chưa. Trước khi có `ovn-detrace`, debug pattern phổ biến là vừa mở 3 terminal: ofproto/trace, ovn-sbctl list, ovn-nbctl list, copy-paste UUID giữa các cửa sổ; rất dễ nhầm khi UUID có 36 ký tự còn cookie chỉ có 8.

**Vấn đề giải quyết.** Cụ thể là 2 nhóm vấn đề: (1) **Inverse mapping** từ cookie 32-bit về `Logical_Flow.UUID` 128-bit. Mapping này không phải hàm thuần (cookie chỉ là 32 bit cao của UUID), nhưng đủ rõ để xác định Logical_Flow trong phần lớn trường hợp; `ovn-detrace` xử lý cả collision bằng cách báo nhiều candidate khi prefix trùng. (2) **Annotation in-place** trên text trace: đầu vào của `ovn-detrace` là một file text bất kỳ chứa cookie, đầu ra là file đó nhưng kèm comment lồng vào, không phá vỡ thứ tự, có thể paste thẳng vào incident report.

**Tầm quan trọng.** Trong tier mastery OVN, `ovn-detrace` thuộc **tier 1 cho debug forensic**, ngang với `ovn-trace` và `ofproto/trace`. Một kỹ sư OVN không biết `ovn-detrace` sẽ debug được khi cluster nhỏ (vài chục flow) nhưng sẽ tê liệt khi cluster lớn (hàng vạn flow trên 1 chassis). Đặc biệt trong môi trường multi-tenant OpenStack/Neutron có hàng trăm Logical_Switch, dump-flows trả về vài MB text, không có `ovn-detrace` thì gần như không thể chỉ ra rule nào sinh từ tenant nào.

**Cơ chế hoạt động.** `ovn-detrace` đọc stdin (hoặc file) line-by-line. Với mỗi dòng, nó áp regex tìm pattern `cookie=0x([0-9a-fA-F]+)` (8 ký tự hex). Với mỗi cookie phát hiện, script gửi truy vấn OVSDB tới SBDB: `monitor_cond Logical_Flow [["_uuid","includes",<cookie_as_uuid_prefix>]]` (thực tế dùng IDL select; uuid prefix matching ở client side). Nếu match, script đọc các column: `logical_datapath` (UUID datapath), `pipeline` (`ingress` hay `egress`), `table_id` (stage số), `priority`, `match` (microflow expression), `actions` (ngôn ngữ logical OVN), `external_ids:stage-name` (tên stage chuỗi), `external_ids:source` (nếu có, link tới NB resource). Tiếp theo, với `logical_datapath`, script tra Datapath_Binding để lấy `external_ids:name` (tên Logical_Switch hay Logical_Router). Với `external_ids:source`, script connect NBDB tra ACL/LB/LR_Static_Route theo UUID đó để lấy match human-readable. Cuối cùng, script in dòng gốc (giữ nguyên) + block annotation thụt lề bắt đầu bằng `*` hoặc `>`, bao gồm tất cả thông tin trên. Nếu cookie không tìm thấy trong SBDB (rất hiếm, do flow stale chưa GC hoặc SBDB connect sai), script in `cookie=0x... (no match)` và đi tiếp.

**Phân loại.** `ovn-detrace` thuộc loại **CLI tool standalone** (không phải subcommand của `ovn-nbctl`/`ovn-sbctl`), được install vào `/usr/bin` qua gói `ovn-common`. Cùng họ tooling debug OVN: `ovn-trace` (logical simulator), `ovn-sbctl lflow-list` (logical flow dumper), `ofproto/trace` (qua `ovs-appctl`, OVS thuần). `ovn-detrace` khác nhóm `ovn-nbctl`/`ovn-sbctl` ở chỗ nó **read-only** cho cả hai DB, không có write subcommand.

**Cách kỹ sư vận hành thành thạo.** Bốn kỹ năng cần có. (1) **Đọc**: hiểu cấu trúc output `ovn-detrace` (block annotation thụt lề, mỗi block tương ứng 1 cookie hit, thứ tự match thứ tự cookie xuất hiện trong input). (2) **Xâu chuỗi tool**: biết khi nào pipe `ofproto/trace ... | ovn-detrace`, khi nào `ovs-ofctl dump-flows br-int | ovn-detrace`, khi nào dump file rồi chạy offline. (3) **Cấu hình endpoint**: biết flag `--ovnnb`/`--ovnsb` cho remote DB, `--ovsdb-sock` cho local socket UNIX, `--no-leader-only` cho cluster Raft; sai endpoint = silent fail (cookie không decode). (4) **Decision tree khi cookie không match**: phân biệt 3 trường hợp — flow stale (ovn-controller chưa cleanup, restart resolve), `ovn-detrace` connect sai DB (verify endpoint), hoặc cookie không phải do OVN sinh (manual `ovs-ofctl add-flow` chèn tay, hoặc tool khác viết br-int).

**Quy trình sử dụng.** Workflow chuẩn 5 bước cho production debug. Bước 1: chạy `ovn-trace` để simulate logical, xác định stage nghi ngờ drop. Bước 2: trên chassis target, dump physical pipeline bằng `ovs-appctl ofproto/trace br-int 'in_port=N,...'` lưu vào file `/tmp/of.txt`. Bước 3: chạy `cat /tmp/of.txt | ovn-detrace --ovnnb=tcp:nb:6641 --ovnsb=tcp:sb:6642 > /tmp/of-annotated.txt`. Bước 4: đọc `/tmp/of-annotated.txt`, tìm dòng có action `drop` hoặc dòng cuối cùng được match, đọc block annotation kèm để biết Logical_Flow nào và NB resource nào. Bước 5: chỉnh NB qua `ovn-nbctl acl-del/lr-route-del/lb-del`; tuyệt đối không dùng `ovs-ofctl del-flows br-int` vì ovn-controller sẽ tái cài rule trong vài giây sau khi resync NBDB.

**Khi xảy ra sự cố.** Tình huống điển hình kỹ sư phải nhớ ngay đến `ovn-detrace`: (a) packet bị drop nhưng `ovn-trace` cho biết logical pipeline pass — chứng tỏ flow physical không khớp logical, cần `ovn-detrace` để xác định flow drop vật lý đến từ Logical_Flow nào (có thể là stale flow); (b) latency tăng đột biến trên một Logical_Switch — `ovs-ofctl dump-flows br-int | ovn-detrace | grep <ls_name>` để xem rule nào của switch đó được hit nhiều; (c) audit security — kỹ sư muốn biết tất cả ACL drop trên một chassis, dump br-int rồi `ovn-detrace` annotate để tổng hợp. Cross-link Phần 20.2 §troubleshooting flow + 20.7 §gradient L5 production incident.

**Liên quan mật thiết.** Tightly coupled với 5 thành phần. (1) **`ovs-appctl ofproto/trace`** (Phần 9.25): nguồn input chính, format output của ofproto/trace có cookie ở mỗi dòng "Rule:". (2) **`ovn-controller`** (Phần 13.7): phía cài cookie xuống br-int; nếu controller chưa sync, cookie có thể không tồn tại trong SBDB. (3) **`ovn-northd`** (Phần 13.8): bên encode UUID Logical_Flow thành cookie khi compile. (4) **SBDB `Logical_Flow` table** (Phần 13.1): bảng tra ngược; format `match`/`actions` column trực tiếp xuất hiện trong output `ovn-detrace`. (5) **NBDB ACL/Load_Balancer table** (Phần 13.3): nguồn cuối của intent; `ovn-detrace` join NB qua `external_ids:source` UUID để hiển thị tên ACL human-readable. Loose related: `ovn-trace` (anh em song song, không phụ thuộc trực tiếp), `ovn-sbctl lflow-list` (cùng đọc Logical_Flow nhưng không xử lý cookie).

**Khác biệt giữa các phiên bản.** OVN 2.6 introduce script đầu tiên (Python 2). OVN 20.06 chuyển sang Python 3 thuần. OVN 21.06 thêm `--ovsdb-sock` cho UNIX socket connect. OVN 22.03 (curriculum baseline) thêm `--no-leader-only` cho cluster mode Raft (cho phép connect follower thay vì chỉ leader, hữu ích khi leader busy). OVN 24.03 cải thiện block separator output, thêm cờ `--exit-on-error` cho CI usage. Với production OpenStack chạy OVN 22.03.x, cú pháp `cat trace.txt | ovn-detrace --ovnnb=tcp:nb:6641 --ovnsb=tcp:sb:6642` ổn định, không cần điều chỉnh; chú ý OVN 24.03 prefix block thay đổi từ `*` sang `> ` nên parser xuôi tự viết phải nhận diện cả hai.

**Cách quan sát + xác minh.** Kiểm tra `ovn-detrace` đang chạy đúng bằng 3 cách. (1) **Verify install**: `which ovn-detrace` trả về `/usr/bin/ovn-detrace`; `ovn-detrace --help` in usage không lỗi; `head -1 $(which ovn-detrace)` cho thấy `#!/usr/bin/env python3`. (2) **Smoke test**: `echo 'cookie=0xdeadbeef' | ovn-detrace --ovnnb=tcp:nb:6641 --ovnsb=tcp:sb:6642`, output mong đợi: dòng input giữ nguyên + block "no match" (cookie giả). (3) **Real test**: trên chassis có flow thật, `ovs-ofctl dump-flows br-int table=8 | head -1 | ovn-detrace ...`, output phải có block annotation chứa `Logical datapath`, `Pipeline`, `Stage name`, match/action expression. Nếu output rỗng → `ovn-detrace` không kết nối được SBDB; verify endpoint reachable bằng `nc -zv sb-server 6642`.

**Source code tham chiếu.** File chính: `utilities/ovn-detrace.in` trong repo OVN (template autotools, build thành `utilities/ovn-detrace`). Các function quan trọng: `main()` parse argv + đọc stdin, `OVSDB.find_row_by_partial_uuid()` (helper trong `ovs.db.idl` của OVS Python binding) tra UUID prefix, `print_logical_flow()` format block annotation. Cookie regex: `re.compile(r'cookie=0x([0-9a-fA-F]+)')`. Man page: `manpages/ovn-detrace.8.xml` (OVN 22.03 trở đi đặt section 8; trước đó section 1). Baseline verify: branch `branch-22.03` của `ovn-org/ovn`. Cùng family script Python: `utilities/ovn-trace.c` (C, không phải Python), `utilities/ovn-sbctl.c` (C). `ovn-detrace` là một trong số ít utility Python trong OVN distribution.

**Trường hợp sự cố thực tế.** Sự cố production multi-tenant OpenStack 1200 VM trên cluster OVN 22.03.5, 8 chassis. Symptom: 1 tenant (Logical_Switch `ls-prod-app`) báo east-west TCP timeout intermittent ~3% rate, các tenant khác bình thường. Investigation: `ovn-trace ls-prod-app 'inport=="vm-A" && ip4.dst==<vm-B-ip> && tcp.dst==8080'` cho kết quả pipeline pass tới `ls_out_apply_buffered_actions` → `output`. Nhưng `ping vm-B` từ vm-A actually fails 3%. Tiếp: trên chassis chứa vm-A, `ovs-appctl ofproto/trace br-int 'in_port=<vm-A-ofport>,tcp,nw_src=...,nw_dst=...,tp_dst=8080'` cho thấy packet bị drop ở `table=12, priority=2002, action=drop`. Cookie là `0x6f8a1b3c`. `echo 'cookie=0x6f8a1b3c' | ovn-detrace --ovnnb=... --ovnsb=...` annotate: `Logical_Flow uuid 6f8a1b3c-..., datapath ls-prod-app, stage ls_out_acl, match (ip && tcp.dst==8080 && reg0[10]==1), action drop, source ACL "deny-stale-app-2024Q1"`. Root cause: ACL cũ tên `deny-stale-app-2024Q1` đáng lẽ đã xoá khi migrate sang policy v2 nhưng quên `ovn-nbctl acl-del`; flow stale vẫn còn trong SBDB và br-int. Fix: `ovn-nbctl acl-del ls-prod-app to-lport 2002 'ip && tcp.dst==8080 && ...'`. Sau 5 giây ovn-controller resync, drop rate về 0%. Giá trị `ovn-detrace` ở đây: trong `ovs-ofctl dump-flows br-int` có ~14000 dòng, không có `ovn-detrace` thì không thể chỉ ra ACL nào sinh ra rule drop đó.

**Bài tập synthetic.** Lab 2 chassis OVN, 1 Logical_Switch `ls-lab` có 2 LSP `lsp1`, `lsp2`. Setup:

```bash
# Khởi tạo logical
ovn-nbctl ls-add ls-lab
ovn-nbctl lsp-add ls-lab lsp1
ovn-nbctl lsp-set-addresses lsp1 "00:11:22:33:44:01 10.0.0.1"
ovn-nbctl lsp-add ls-lab lsp2
ovn-nbctl lsp-set-addresses lsp2 "00:11:22:33:44:02 10.0.0.2"

# Cài ACL drop ssh (port 22) hướng to-lport
ovn-nbctl acl-add ls-lab to-lport 1000 'tcp && tcp.dst == 22' drop

# Bind LSP vào sandbox/Mininet ports tương ứng

# Bước 1: simulate logical
ovn-trace ls-lab 'inport=="lsp1" && eth.src==00:11:22:33:44:01 && \
                  eth.dst==00:11:22:33:44:02 && ip4.src==10.0.0.1 && \
                  ip4.dst==10.0.0.2 && ip.proto==6 && tcp.dst==22 && tcp.src==33000'
# Kết quả mong đợi: drop ở stage ls_out_acl

# Bước 2: trace physical trên chassis chứa lsp1
ovs-appctl ofproto/trace br-int \
  'in_port=<lsp1-ofport>,tcp,dl_src=00:11:22:33:44:01,dl_dst=00:11:22:33:44:02,
   nw_src=10.0.0.1,nw_dst=10.0.0.2,tp_src=33000,tp_dst=22' > /tmp/of.txt

# Bước 3: annotate bằng ovn-detrace
cat /tmp/of.txt | ovn-detrace --ovnnb=unix:/var/run/ovn/ovnnb_db.sock \
                              --ovnsb=unix:/var/run/ovn/ovnsb_db.sock

# Verify: block annotation cho cookie của rule drop chỉ rõ
#  - datapath ls-lab
#  - stage ls_out_acl
#  - match (ip && tcp.dst==22 ...)
#  - source ACL UUID khớp với entry vừa tạo
```

Variations: (a) đổi ACL từ `drop` sang `allow-related`, observe block annotation thay đổi action; (b) xoá ACL bằng `ovn-nbctl acl-del ls-lab`, dump-flows lại → cookie cũ biến mất khỏi br-int trong vài giây; (c) restart ovn-controller, kiểm tra cookie remain (cookie persistent cho cùng Logical_Flow UUID).

**Lỗi thường gặp + tín hiệu chẩn đoán.** Bốn failure mode chính. (1) **Endpoint sai**: `--ovnnb` hoặc `--ovnsb` trỏ DB không reachable; tín hiệu: output chỉ có dòng input, không có block annotation; chẩn đoán: thêm `--verbose` (nếu bản hỗ trợ) hoặc test bằng `ovn-sbctl --db=tcp:sb:6642 list Logical_Flow | head` xem có response không; recovery: sửa endpoint, thử `unix:/var/run/ovn/ovnsb_db.sock` cho local. (2) **Cluster Raft, follower không serve**: nếu SBDB ở mode cluster Raft và endpoint trỏ follower mặc định OVN 21.06+ chỉ leader trả lời, output có thể partial; tín hiệu: cookie trong cùng datapath đôi lúc decode được đôi lúc không; recovery: thêm `--no-leader-only` hoặc đảm bảo trỏ leader (xem `ovn-appctl -t /var/run/ovn/ovnsb_db.ctl cluster/status OVN_Southbound`). (3) **Stale flow ngoài SBDB**: cookie tồn tại trên br-int nhưng Logical_Flow tương ứng đã bị ovn-northd xoá; tín hiệu: `cookie=0x... (no match)` cho rule mới dump; recovery: `ovn-appctl -t /var/run/ovn/ovn-controller.<chassis>.ctl recompute` để force ovn-controller resync, hoặc restart ovn-controller. (4) **Cookie không phải OVN-encoded**: ai đó chạy `ovs-ofctl add-flow br-int 'cookie=0x12345,...'` thủ công; tín hiệu: cookie format giống nhưng không bao giờ decode được; recovery: identify nguồn (kiểm tra script triển khai), xoá manual flow, để ovn-controller manage br-int độc quyền.

**So sánh với hệ khác.** Trong NSX-T, tool tương đương là `nsxcli > get logical-port <uuid> traceflow` kết hợp với `get logical-router <uuid> traceflow`, nhưng NSX-T không có cookie-mapping pattern vì NSX-T data plane sinh trực tiếp ID NSGroup vào VDS rule, không cần inverse mapping qua hash. Cisco ACI có `iping`/`itraceroute`/`acidiag fnvread` nhưng đó là full distributed trace, không có concept "cookie từ control plane gắn vào physical rule"; ACI dùng class-id encoding khác hẳn. Linux iptables không có analogue: `iptables -L -v --line-numbers` show counter per rule nhưng rule không có "cookie" liên kết về intent declarative cao hơn. Điểm độc đáo của `ovn-detrace`: nó là ví dụ điển hình của design pattern "**intent-trace bridge**" — dùng định danh nhỏ (32-bit cookie) gắn ở data plane để link ngược về định danh đầy đủ (UUID 128-bit) ở control plane, đổi vài byte memory per rule lấy khả năng forensic. Pattern này đáng học để áp dụng trong các hệ SDN tự xây.

## 13.14.11 Bù sâu nhóm option cluster + transaction guard

> **Mục đích Phần này:** §13.14.9.1 đã liệt kê 5 option `--timeout`, `--leader-only`, `--no-leader-only`, `--shuffle-remotes`, `--dry-run` ở mức bảng tóm tắt. Phần §13.14.11 mở rộng từng option thành sub-section độc lập với khái niệm, vai trò, cơ chế, cách quan sát + lab thật, để kỹ sư on-call có chỗ tra cứu nhanh khi script `ovn-nbctl` bị treo, write rớt vào follower, hoặc cluster Raft mất leader.

### 13.14.11.1 Option `--timeout=N` (alias `-t`)

**Khái niệm.** `--timeout=N` (hoặc dạng ngắn `-t N`) đặt giới hạn thời gian chạy của một invocation `ovn-nbctl` thành xấp xỉ N giây. Mặc định N=0 nghĩa là **wait forever**: tiến trình `ovn-nbctl` sẽ chờ NBDB phản hồi mãi mãi, kể cả khi server unreachable, cho đến khi user Ctrl+C. Khi N>0 và timeout chạm ngưỡng, kernel gửi `SIGALRM` tới tiến trình, `ovn-nbctl` exit với signal code, không phải exit code thông thường, nên script gọi cần phân biệt `kill -l $?` để nhận diện.

**Vai trò + vấn đề giải quyết.** Trong hệ thống tự động hoá (Ansible, Kubernetes operator OVN-Kubernetes, OpenStack neutron-ovn-mech), khi NBDB cluster Raft tạm thời mất quorum hoặc network partition giữa controller node và NB cluster, mặc định wait forever sẽ làm cả pipeline CMS treo, kéo theo lock file daemon hoặc transaction queue dồn lại. `--timeout=5` (5 giây) là giá trị thường gặp trong production playbook: đủ cho transaction NBDB bình thường (latency 50-500 ms), nhưng đủ ngắn để fail nhanh khi cluster degrade. Đặc biệt quan trọng khi gọi `ovn-nbctl` từ systemd unit có `TimeoutStartSec=`: nếu không set `--timeout`, systemd sẽ kill tiến trình lúc unit timeout, để lại lock OVSDB không clean.

**Cơ chế hoạt động.** `parse_options()` trong `utilities/ovn-nbctl.c` ánh xạ flag `-t`/`--timeout` thành biến `int timeout`, sau đó truyền vào `ctl_timeout_setup(timeout)` (helper trong `lib/ovs-thread.c` của OVS dependency) để gọi `alarm(timeout)` syscall. Khi `alarm` đếm hết N giây mà chưa có handler clear, kernel raise `SIGALRM`. Mặc định OVN không cài custom signal handler cho `SIGALRM` nên process exit với core dump pattern `Alarm clock` trong `dmesg` hoặc shell `$?` = 142 (128 + signal 14).

**Khi nào tăng default 5s lên 30-60s.** Cluster cross-region (NB1 ở Hà Nội, NB2 ở TP HCM, NB3 ở Đà Nẵng) có RTT 30-50 ms; transaction lớn (bulk lsp-add 500 port atomic) cần 1-3 giây để compile + commit + replicate Raft 3 node. Trong workload đó set `--timeout=30` (30 giây) để cover edge case latency.

**Lab quan sát ngắn.** Chạy `ovn-nbctl --timeout=2 --db=tcp:198.51.100.99:6641 list NB_Global` trên một máy không reachable IP đó: sau ~2 giây tiến trình tự exit với message `ovn-nbctl: NB_Global: timeout expired`, exit code 142 (`echo $?`). Đối chiếu với cùng lệnh không có `--timeout`: tiến trình treo cho đến khi user Ctrl+C.

**Lỗi thường gặp.** (a) Người mới nhầm `--timeout=5000` nghĩ là 5 giây milisecond style → thực tế là 5000 giây. Đơn vị là **giây**, không phải ms. (b) Set `--timeout=1` cho transaction lớn → false-fail liên tục mặc dù cluster healthy.

### 13.14.11.2 Option `--leader-only`

**Concept.** `--leader-only` is the default flag for `ovn-nbctl` against NBDB in Raft cluster mode (3 or 5 nodes). When the NB cluster runs Raft, every write transaction must go through the leader to guarantee linearizable consistency. The flag tells `ovn-nbctl` to attempt connection only to the server holding the leader role; if a remote reports the follower role, the client disconnects and tries the next remote in the list `--db=tcp:nb1:6641,tcp:nb2:6641,tcp:nb3:6641`.

**Role and problems it solves.** In a Raft cluster, a follower can lag the leader by a few log entries; a read transaction on a follower can return stale data (tens to hundreds of milliseconds old). For a CMS such as OpenStack Neutron that compiles YAML into a chain of ovn-nbctl commands (`ls-add`, then `lsp-add`, then `lsp-set-addresses` for the same tenant), if the second invocation lands on a follower that has not yet replicated the first, the command fails with "Logical_Switch not found". `--leader-only` (the default) ensures every invocation sees the same state, with no read-after-write race.

**How it works.** The OVSDB IDL client in `ovn-nbctl` sends `get_schema` plus `get_server_id` to each remote; the cluster server returns JSON containing `"role":"leader"` or `"role":"follower"`. If the role is not leader and `--leader-only` is on, the IDL closes the connection and tries the next remote. If all three remotes are followers (a rare scenario where the leader just crashed and the election has not completed), the client retries continuously until `--timeout` fires or one remote becomes leader.

**Effect and trade-off.** Every write from the CMS converges on a single leader node, making the leader a hot spot (CPU and disk IO for the Raft log). That is why an NB cluster plan must capacity-plan the leader separately: the leader can use 60 to 80 percent of CPU on a cluster with more than 1000 Logical_Switch instances. When the leader fails, the election lasts 1 to 3 seconds, and every `ovn-nbctl --leader-only` blocks during that window. That is the consistency-versus-availability trade-off of Raft.

**Observation lab.** On a three-node Raft cluster, run `ovn-appctl -t /var/run/ovn/ovnnb_db.ctl cluster/status OVN_Northbound | grep -E 'Role|Leader'` to identify the current leader; then run `ovn-nbctl --db=tcp:<follower-IP>:6641 ls-list` (pointing only at a follower). With the default `--leader-only` on, `ovn-nbctl` rejects the follower with "no leader available". Compare with §13.14.11.3 below.

### 13.14.11.3 Option `--no-leader-only`

**Concept.** `--no-leader-only` reverses the default of §13.14.11.2: it lets `ovn-nbctl` connect to any server in the Raft cluster, not just the leader. Read-only transactions (commands that do not modify NBDB, such as `list`, `find`, `get`, `show`, `lsp-list`, and `acl-list`) run on a follower; write transactions are forwarded by the follower to the leader (proxy mode), so writes remain linearizable.

**Role and problems it solves.** Use case 1: **read scaling**. A large cluster receives thousands of `ovn-sbctl list Logical_Flow` per second from monitoring tooling. If every read goes to the leader, the leader saturates. `--no-leader-only` (combined with `--shuffle-remotes`) spreads read traffic across the three nodes, raising read throughput by about 3x. Use case 2: **availability during the election window**. When the leader has just crashed and the election is not finished (1 to 3 seconds), `--leader-only` blocks while `--no-leader-only` can still read (slightly stale) data from any follower. Use case 3: **forensic debugging**. When a follower is suspected of unusual lag, the engineer connects directly to it with `--no-leader-only` and compares state with the leader.

**How it works.** When the flag is set, the IDL skips the role check on `get_server_id`; any reachable remote with a matching schema is accepted. A write transaction on a follower is forwarded to the leader through Raft RPC, the leader confirms, and the response returns to the client. Write latency in follower mode equals leader latency plus one extra leader-to-follower RTT.

**Consequences and warnings.** (a) Reads can be stale (the read-after-write race noted in §13.14.11.2); automation scripts SHOULD NOT use `--no-leader-only` for read-then-write chains. (b) Read-only monitoring tooling (Prometheus exporter, Grafana dashboard, SLO checker) SHOULD use `--no-leader-only` to reduce pressure on the leader.

**Observation lab.** The command from the §13.14.11.2 lab that points at a follower succeeds when `--no-leader-only` is added: `ovn-nbctl --no-leader-only --db=tcp:<follower-IP>:6641 ls-list` returns the Logical_Switch list (possibly with 1 to 2 stale entries if the follower is lagging).

### 13.14.11.4 Option `--shuffle-remotes`

**Concept.** `--shuffle-remotes` (on by default since OVS 2.13 and OVN 20.06) tells the IDL to randomly shuffle the order of remotes in the chain `--db=tcp:nb1:6641,tcp:nb2:6641,tcp:nb3:6641` before attempting connection. The opposite flag is `--no-shuffle-remotes`, which keeps the declared order.

**Role and problems it solves.** Without shuffle (the old behaviour before OVS 2.13), 1000 clients on the same cluster all try nb1 first, so nb1 receives every connection and saturates. This is the classic **thundering herd**. Shuffle distributes the starting point of connection attempts and balances the load across the three nodes. The benefit is most visible after a cluster restart or network-partition recovery: every client reconnects at the same time, and without shuffle 100 percent land on nb1.

**How it works.** In `parse_options()`, the default sets `shuffle = true`. The IDL has a function `jsonrpc_session_shuffle()` that permutes the remote list using `random_uint32() % n` for each index. Each separate `ovn-nbctl` process seeds its random number from `/dev/urandom`, so 1000 parallel invocations produce 1000 different orders.

**Interaction with `--leader-only`.** When both `--leader-only` and `--shuffle-remotes` are on (the default), the IDL shuffles the order, tries each remote, skips followers, and stops at the first leader. Net effect: clients start trying at different times, but writes still converge on the leader. With read-only `--no-leader-only`, shuffle distributes evenly and each client lands on a random node.

**When to disable with `--no-shuffle-remotes`.** Rare. The only legitimate use case is intentional testing that forces the client to connect to a specific node first (for example, to verify failover behaviour). In that case, set `--no-shuffle-remotes --db=tcp:<target>,tcp:<backup1>,tcp:<backup2>`. Production should NOT disable shuffle.

**Observation lab.** On a three-node Raft cluster with the Prometheus metric `ovsdb_server_connection_count`, launch 100 parallel `ovn-nbctl --no-leader-only ls-list` processes within 1 second and measure connection_count on all three nodes. With the default shuffle on, the distribution is roughly 33, 33, 33. With `--no-shuffle-remotes`, the distribution is roughly 100, 0, 0. The contrast illustrates the effect directly.

### 13.14.11.5 Option `--dry-run`

**Concept.** `--dry-run` tells `ovn-nbctl` to parse and validate the command and compute the OVSDB JSON-RPC transaction it would send, but NOT commit the transaction to NBDB. The command exits successfully (exit code 0) if syntax, references, and constraints are valid; it exits non-zero on errors (for example, a Logical_Switch reference that does not exist, or an integer field receiving a string). NBDB is not modified at all.

**Role and problems it solves.** Dry-run is the safety net before applying a risky command in production. Use case 1: **review automatically generated command sequences** (Ansible playbooks, Kubernetes operators rendering commands from templates). Run dry-run on a staging cluster first, and only apply for real if it passes. Use case 2: **verify a migration script**, for example `ovn-nbctl -- ls-add new -- ls-del old`. Dry-run confirms that the old object exists and the new one does not before committing. Use case 3: **safety net for the on-call engineer**. When fixing production at 3 in the morning, type the command, add `--dry-run` first, read the output to confirm intent, then rerun without the flag.

**How it works.** In `do_nbctl()` (the main transaction handler), after building the OVSDB transaction structure, the code checks the `dry_run` variable: if true, it prints the transaction JSON to stderr (in verbose mode) or skips silently, then skips `ovsdb_idl_loop_commit_and_wait()`. The validation step (checking that referenced UUIDs are valid and that types match the schema) RUNS IN FULL before commit, so dry-run still catches most static errors. It does not catch dynamic errors (constraint violations only seen at commit time, for example a unique-index conflict).

**Important limitations.** (a) Dry-run does NOT simulate the ovn-northd compile from NBDB to SBDB; it only checks the NBDB transaction layer. The command `ovn-nbctl --dry-run lr-route-add` can pass dry-run while the route still does not work because northd rejects it (for example, next-hop unreachable). To verify that the pipeline actually works, also use `ovn-trace` (§13.14.9.2). (b) Dry-run with a `--` chain (an atomic transaction of multiple subcommands) only validates up to the first failing subcommand; if the fifth of ten fails, you cannot tell whether subcommands 6 to 10 pass until the fifth is fixed.

**Observation lab.** Run `ovn-nbctl --dry-run -- ls-add ls-test -- lsp-add ls-not-exist lsp-x`. Result: dry-run fails on the second subcommand (`ls-not-exist` does not exist), exit code 1, with a clear message. Then check `ovn-nbctl ls-list | grep ls-test`: empty, confirming that `ls-test` was not created (the atomic transaction rolled back). Compare with the same command without `--dry-run`: it also fails and rolls back, but dry-run avoids the cost of attempting the commit.

**Version differences.** OVN 22.03 (the curriculum baseline) supports `--dry-run` for most subcommands (`ls-add`, `lsp-add`, `acl-add`, `lr-add`, `lb-add`, and so on). A few special subcommands such as `init` (which bootstraps a fresh DB) ignore `--dry-run` because there is no corresponding transaction. OVN 24.03 extends `--dry-run` to control subcommands such as `set-connection` and `set-ssl`.

To be safe, point at the leader or upgrade to 24.03 or later.
>
> The five subsections below cover each flag with the correct syntax, usage context, sample output, and common pitfalls.

### 13.14.12.1 `ovn-trace --all` instead of the misnomer "multiple paths"

**Concept.** `--all` tells `ovn-trace` to emit all three output forms in a single invocation: `--detailed` (detailed stage and register information), `--summary` (compact, programming-language style), and `--minimal` (only packet-modifying actions and output). This option is commonly misnamed `--multiple` in internal docs; the official upstream name has been `--all` since OVN 21.06. When a `select` action appears (Load_Balancer multi-backend, ECMP), `--all` does not automatically explore every branch; to see other branches, rerun with a different `--select-id=N` (see §13.14.12.4).

**Role and problems it solves.** When debugging a complex branching stage (`ls_in_lb` with three backends, or `lr_in_ip_routing` with two ECMP next-hops), a single mode makes it hard to compare packet behaviour before and after an action. `--all` lets the engineer read the same result at three abstraction levels and compare to understand the compiler intent (`ovn-northd`) and verify simulator behaviour. Another use case: pasting output into a bug ticket. `--minimal` gives a quick summary and `--detailed` gives forensic evidence; including both in one file removes follow-up questions from a reviewer.

**Syntax and example output.**

```bash
ovn-trace --all ls-prod \
  'inport=="lsp-vm1" && eth.src==50:54:00:00:00:01 && eth.dst==ff:ff:ff:ff:ff:ff && \
   arp.op==1 && arp.spa==10.0.0.1 && arp.tpa==10.0.0.2'
```

Output structure:

```
# ovn-trace ls-prod '...'

ingress(dp="ls-prod", inport="lsp-vm1")
-----------------------------------------
 0. ls_in_port_sec_l2 (northd.c:5234): inport == "lsp-vm1" && ...
    next;
 ...

# Summary
ingress(dp="ls-prod", inport="lsp-vm1") {
    next;
    ...
    output;
}

# Minimal
arp.op = 2;
arp.tha = 50:54:00:00:00:02;
output("lsp-vm2");
```

**Common pitfalls.** (a) Many blog posts write `--multiple`, so the command fails with `ovn-trace: unrecognized option '--multiple'`. Correction: use `--all`. (b) `--all` does not mean "try every select branch"; to enumerate branches, combine with `--select-id`. (c) `--all` output is three times as long as a single mode; when piping to `less`, also add `--no-friendly-names` for stable output if a custom parser is involved.

**Versions.** The three modes plus `--all` were split out in OVN 21.06 (an output-refactor commit group). Before that, OVN 2.7 to 21.03 had only the single verbose default.

### 13.14.12.2 `ovn-trace --friendly-names` and `--no-friendly-names`

**Concept.** `--friendly-names` (on by default from OVN 22.03) tells `ovn-trace` to replace the UUIDs of Logical_Switch, Logical_Router, and Logical_Switch_Port with the `external_ids:name` value taken from NBDB or SBDB. UUIDs are hard to read during an incident (`6f8a1b3c-...-...-...-...`); replacing them with user-declared names (`ls-prod-app`, `lsp-vm1`) makes the output much friendlier. The opposite flag `--no-friendly-names` keeps the original UUIDs, which is useful for custom parsers that need stable identifiers.

**Role and problems it solves.** In an environment with hundreds of Logical_Switch instances, reading a trace full of UUIDs is torture. Friendly names cut trace reading time from minutes to seconds because the engineer immediately recognises "this is the Logical_Switch of tenant X". Conversely, when writing a script that greps the output (CI test, regression check), UUIDs are more stable because names can change through `ovn-nbctl`; in that case, use `--no-friendly-names` to avoid mismatches.

**Syntax and example.**

```bash
# Default (friendly names): human-readable output
ovn-trace ls-prod 'inport=="lsp-vm1"'
# Output: ingress(dp="ls-prod", inport="lsp-vm1")

# Disable for parser scripts
ovn-trace --no-friendly-names ls-prod 'inport=="lsp-vm1"' \
  | grep 'ingress(dp='
# Output: ingress(dp="6f8a1b3c-...-...", inport="2a3b...")
```

**Common pitfalls.** (a) On OVN versions earlier than 22.03 the default is `--no-friendly-names`; to get friendly output you must set the flag manually. (b) If an LSP or pair name contains special characters (`/`, space, Unicode), friendly names still show the raw character, which can break shell-pipe quoting; in that case, use `--no-friendly-names` for stability.

**Mechanism.** `utilities/ovn-trace.c` on branch-22.03 lines 278 to 279 register the long options `friendly-names` and `no-friendly-names`; the internal variable `friendly_names` defaults to `true` from OVN 22.03 onwards. The function `ovntrace_node_print_details()` checks this variable when formatting datapath and port names.

**Versions.** The option was introduced in OVN 21.09 (default OFF). OVN 22.03 flipped the default to ON. OVN 24.03 made no further change.

### 13.14.12.3 `ovn-trace --ovs[=remote]` to align logical with physical

**Concept.** The flag `--ovs` (which can take a remote in the form `--ovs=tcp:host:6640` or `--ovs=unix:/path/sock`; without an argument it defaults to `unix:/br-int.mgmt`) tells `ovn-trace` not only to simulate the logical pipeline but also to CONNECT to `ovs-vswitchd` on a specific chassis to read the corresponding OpenFlow flows and print them alongside each logical stage. Net effect: the output has two columns. On the left is the Logical_Flow (OVN level); on the right is the OpenFlow rule (OVS level) that `ovn-controller` installed on `br-int`. This enables direct comparison to see whether northd compiled the intent correctly.

**Role and problems it solves.** When you suspect that `ovn-northd` compiled something wrong or that `ovn-controller` failed to install a rule on a specific chassis, use `--ovs` instead of opening two windows (one running plain `ovn-trace`, one running `ovs-ofctl dump-flows br-int`) and comparing by eye. The flag is especially valuable on clusters with many chassis and suspected asymmetry: chassis A's logical pipeline passes, while chassis B drops; run `ovn-trace --ovs=tcp:chassis-B:6640` to see exactly where the flows on chassis B diverge from chassis A.

**Syntax and examples.**

```bash
# Use the default unix socket on the local chassis
ovn-trace --ovs ls-prod 'inport=="lsp-vm1" && ip4.dst==10.0.0.10'

# Point at a remote chassis through TCP (ovs-vswitchd must have a TCP manager enabled)
ovn-trace --ovs=tcp:chassis-B.example.com:6640 ls-prod \
  'inport=="lsp-vm1" && ip4.dst==10.0.0.10'
```

The output includes an extra block:

```
ls_in_acl (priority 2002):
  Logical: ip && tcp.dst==80 -> drop;
  Physical (table=8 br-int chassis-B):
    cookie=0x6f8a1b3c, priority=2002, ip,tcp,tp_dst=80, actions=drop
```

**Common pitfalls.** (a) The default remote `unix:/br-int.mgmt` is only valid when `ovn-trace` runs on the same chassis that hosts `br-int`; on a jump host without `ovs-vswitchd`, pass `--ovs=tcp:...` instead. (b) `ovs-vswitchd` does not bind a TCP listener by default; run `ovs-vsctl set-manager ptcp:6640` first. (c) The targeted chassis must have a Port_Binding for the input LSP; otherwise the OpenFlow rule does not exist there and the physical column is empty. (d) Different from `ovn-detrace --ovs`: the `--ovs` flag of `ovn-detrace` only adds ofport decoding for a flow, it does not connect to ovs-vswitchd to dump rules.

**Mechanism and source.** `utilities/ovn-trace.c` on branch-22.03 lines 257, 276, and 333 (`OPT_OVS`, `optional_argument`); the connection helper `vconn_open()` from the OVS dependency. Man page `ovn-trace(8)` section "--ovs[=remote]".

### 13.14.12.4 `ovn-trace --lb-dst=ip[:port]` and `--select-id=ID` for Load_Balancer hairpin and select-action multipath

**Concept.** `--lb-dst=ip[:port]` overrides the random backend selection in OVN's `ct_lb` action (Load_Balancer DNAT); it forces `ovn-trace` to simulate as if the VIP packet were DNATed to the specific backend `ip:port` instead of picking a random backend. The parameter accepts a bare IP (`10.0.1.50`) or an IP plus port (`10.0.1.50:8080`). The sibling flag `--select-id=ID` (same family, for `select` actions used in ECMP and group-select) lets the simulator force a specific ID inside a select bucket. Both flags work only in normal mode (NOT in daemon mode).

**Role and problems it solves.** When a VIP has three backends and a VM client reports "sometimes the backend fails to connect", the engineer needs to verify that each backend has correct routes and ACLs. Without `--lb-dst`, each `ovn-trace` run randomly picks a backend, which is not reproducible. With `--lb-dst=10.0.1.51`, you can trace deterministically per backend in succession and compare paths. Similarly, `--select-id` works for ECMP: a route `10.0.0.0/8` with two next-hops can be forced with `--select-id=0` to pick the first or `--select-id=1` to pick the second.

**Syntax and examples.**

```bash
# Tenant reports timeouts to backend 10.0.1.52; check the path to that backend
ovn-trace --lb-dst=10.0.1.52:8080 lr-edge \
  'inport=="lsp-client" && eth.src==... && \
   ip4 && ip4.src==10.0.0.5 && ip4.dst==10.0.20.50 && \
   tcp && tcp.dst==8080 && tcp.src==33000'
# Trace passes through to ls_in_lb and hits ct_lb(backends=10.0.1.52:8080) deterministically.

# Compare with a different backend
ovn-trace --lb-dst=10.0.1.51:8080 lr-edge '...'

# ECMP with two next-hops, forcing each branch
ovn-trace --select-id=0 lr-edge '...'
ovn-trace --select-id=1 lr-edge '...'
```

**Common pitfalls.** (a) `--lb-dst` only takes effect when the pipeline actually hits `ct_lb`; if the microflow does not match the VIP or port, the flag is silently ignored. (b) Passing `--lb-dst=10.0.1.52` without a port when the LB has ports causes the simulator to use a default port from the first backend, which may not be the intent; pass a full `ip:port`. (c) `--select-id` counts from 0 inside the `select` action; assuming 1-based indexing leads to the wrong bucket.

**Mechanism and source.** `utilities/ovn-trace.c` on branch-22.03 lines 264 to 265, 282 to 283, and 349 to 353 (`OPT_LB_DST`, `OPT_SELECT_ID`, both `required_argument`); the function `parse_lb_dst()` parses `ip:port` into an internal struct. The man page `ovn-trace(8)` notes "Not available in daemon mode".

**Versions.** `--lb-dst` was introduced in OVN 20.06. `--select-id` was introduced in OVN 21.06 alongside the select-action refactor.

### 13.14.12.5 `ovn-detrace` real endpoint options (correcting the flag names)

**Concept.** `ovn-detrace` (the Python script `utilities/ovn_detrace.py.in`) on branch-22.03 supports exactly three database-endpoint options: `--ovnsb=DATABASE` (Southbound), `--ovnnb=DATABASE` (Northbound), and `--ovsdb=DATABASE` (the local OVS DB, used only with `--ovs`). The flag names DO NOT have a `-db` suffix. Many internal briefs and cheat sheets write `--ovnsb-db`, `--ovnnb-db`, or `--ovsdb-sock` by mistake; running them fails with `ovn-detrace: option --ovnsb-db not recognized`. The flag `--no-leader-only` (which allows connecting to a follower) **exists only from branch-24.03 onwards** and is not available on 22.03.

**Syntax and real examples.**

```bash
# Use the correct flag names (branch-22.03)
ovs-appctl ofproto/trace br-int 'in_port=3,tcp,...' \
  | ovn-detrace --ovnnb=tcp:nb-leader.example.com:6641 \
                --ovnsb=tcp:sb-leader.example.com:6642

# Local socket (same host)
ovs-ofctl dump-flows br-int table=8 | head -3 \
  | ovn-detrace --ovnnb=unix:/var/run/ovn/ovnnb_db.sock \
                --ovnsb=unix:/var/run/ovn/ovnsb_db.sock

# Add ofport decoding from the local OVS DB
ovn-detrace --ovs --ovsdb=unix:/var/run/openvswitch/db.sock \
            --ovnnb=... --ovnsb=... < /tmp/of.txt
```

**Correct sample output.**

```
br-int OpenFlow flow:
  cookie=0x6f8a1b3c, table=8, priority=2002, ip,tcp,tp_dst=80, actions=drop
  * Logical datapath: "ls-prod" (uuid 6f8a1b3c-...)
  * Pipeline: ingress, table 8 (ls_in_acl), priority 2002
  * Match: ip && tcp.dst == 80
  * Action: drop;
   * ACL UUID: 9c4d... (name "deny-stale-app-2024Q1")
```

**Common pitfalls.** (a) Typing `--ovnsb-db` fails. Correct: `--ovnsb`. (b) Typing `--no-leader-only` on 22.03 fails with `option not recognized`; to use it, upgrade or point directly at the leader (`ovn-appctl -t /var/run/ovn/ovnsb_db.ctl cluster/status OVN_Southbound | grep Leader`). (c) `--ovsdb` is meaningless without `--ovs`; passing it alone is silently ignored. (d) A wrong endpoint (typo in the IP, mistaking port 6641 for 6642) produces no clear error; instead every cookie comes out as `(no match)`. Verify the endpoint with `ovn-sbctl --db=<endpoint> list Logical_Flow | head` before using.

**Mechanism and source.** `utilities/ovn_detrace.py.in` on branch-22.03 lines 50 to 55 (`usage()` text) and lines 412 to 417 (the `getopt.gnu_getopt` long-option list confirming the three names `'ovnsb='`, `'ovnnb='`, `'ovsdb='`, with no `-db` or `leader-only` entry). Branch-24.03 adds `'no-leader-only'` on line 425. Verified 2026-04-26.

**Version summary.** OVN 2.6 introduced `ovn-detrace` with `--ovnsb` and `--ovnnb`. OVN 21.06 added `--ovsdb` and `--ovs`. OVN 24.03 added `--no-leader-only`. On the curriculum baseline 22.03, use the four correct flag names above plus the standard SSL flags `-p`, `-c`, and `-C`.

## Mục tiêu bài học

Sau khi hoàn thành phần này, bạn có thể:

1. **Mô tả (Understand)** vai trò của ovn-northd, daemon duy nhất trong cụm đọc NB DB (ý đồ cấu hình) và ghi SB DB (Logical Flow), là trung gian giữa CMS và ovn-controller.
2. **Phân tích (Analyze)** thuật toán dịch NB → SB: từ Logical Switch + Logical Router + ACL + LB trong NB DB sinh ra Logical Flow trong SB DB, theo pipeline model 24 + 27 table (OVN 22.03).

## 13.8.1 Vai trò ovn-northd trong kiến trúc OVN

**ovn-northd** là tiến trình daemon chạy tập trung trong cụm OVN (thường trên cùng node với ovsdb-server NB/SB, hoặc tách sang node riêng trong cấu hình HA). Nhiệm vụ duy nhất: đọc NB DB (ý đồ cấu hình do CMS — OpenStack Neutron, OVN-Kubernetes, v.v., ghi vào) và compile xuống SB DB (Logical Flow + Port_Binding + Multicast_Group) để ovn-controller cục bộ đọc và thực thi trên br-int.

Kiến trúc **pipeline compile NB → SB** có ba tầng rõ ràng:
- **Tầng 1: NB DB (ý đồ cấu hình):** CMS ghi các object cấp cao như Logical_Switch, Logical_Router, Logical_Switch_Port, ACL, Load_Balancer, NAT, DHCP_Options. Không chứa OpenFlow, chỉ định nghĩa topology và policy.
- **Tầng 2: ovn-northd (translator):** daemon theo dõi mọi thay đổi NB DB qua OVSDB IDL, tính toán Logical Flow tương ứng, và ghi vào SB DB. Cũng tính Port_Binding (binding port với chassis), Datapath_Binding (mỗi LS/LR được gán tunnel_key), Multicast_Group (broadcast domain).
- **Tầng 3: SB DB (physical-neutral intermediate):** chứa Logical Flow độc lập chassis, mọi ovn-controller đọc cùng SB DB và dịch tiếp sang OpenFlow phù hợp chassis cục bộ (đã được đề cập Phần 13.7).

Mô hình này cho phép **phân tách trách nhiệm (separation of concerns)**: CMS chỉ cần biết ý đồ cấu hình cấp cao (Logical Switch, ACL), không cần biết OpenFlow hay tunnel encap. ovn-northd là trung tâm compile phức tạp, biết toàn bộ topology NB, tính toán mọi tương tác (ACL × Port × Load_Balancer × NAT) và sinh Logical Flow đầy đủ. ovn-controller cục bộ chỉ cần dịch subset Logical Flow liên quan chassis → OpenFlow, không cần biết gì về ý đồ cấu hình NB.

Một cụm OVN chuẩn có **một** instance ovn-northd active tại một thời điểm (với một hoặc nhiều standby dưới bầu leader). Lý do: ovn-northd ghi rất nhiều record vào SB DB cho mỗi thay đổi NB, nếu nhiều instance chạy song song sẽ gây xung đột ghi. Bầu leader (qua pacemaker, kubernetes leader election, hoặc built-in OVN `--ovnnb-db=tcp:nb1,tcp:nb2,tcp:nb3 --ovnsb-db=tcp:sb1,tcp:sb2,tcp:sb3`) đảm bảo chỉ một active. Standby chỉ chờ, nếu active chết (phát hiện qua mất kết nối + timeout), một standby được promote thành active.

> **Lưu ý phiên bản:** OVN 20.06 trở về trước dùng ovn-northd đơn tiến trình, mỗi lần NB thay đổi là **tính toán lại toàn bộ** NB → SB. OVN 22.03 giới thiệu **compile song song (parallel compilation)** — ovn-northd đa flow xử lý nhiều Logical Switch/Router song song, cải thiện băng thông thực tế 3-5 lần trên multi-core CPU. OVN 24.03+ giới thiệu **incremental northd** (hiện đang preview) — tương tự I-P của ovn-controller, chỉ tính toán lại subset Logical Flow bị ảnh hưởng bởi NB change.

## 13.8.2 Thuật toán dịch NB → SB Logical Flow

Khi NB DB thay đổi (ví dụ CMS tạo Logical Switch mới), ovn-northd thực hiện chuỗi bước:

**Bước 1 — Tính toán datapath và tunnel_key:**
- Mỗi Logical Switch và Logical Router trong NB map tới một `Datapath_Binding` record trong SB.
- ovn-northd assign `tunnel_key` (1-16777215) duy nhất cho mỗi datapath, dùng làm OpenFlow metadata (đã được đề cập Phần 13.7).
- Datapath_Binding record được tạo/update/xóa theo NB.

**Bước 2 — Compile Logical Flow cho Logical Switch ingress pipeline:**
Mỗi Logical Switch có khoảng 30 table logical trong ingress pipeline, số hóa từ table 0 đến 29. ovn-northd sinh Logical Flow cho mỗi table theo thứ tự:
- `ls_in_check_port_sec`, port security check (nguồn từ `Logical_Switch_Port.port_security`)
- `ls_in_apply_port_sec`, apply port security drop
- `ls_in_lookup_fdb`: FDB lookup (MAC learning)
- `ls_in_put_fdb`: FDB insert cho gói tin có MAC nguồn chưa biết
- `ls_in_pre_acl`, stateful ACL pre-check
- `ls_in_pre_lb`: LB pre-routing
- `ls_in_pre_stateful`, conntrack commit trigger
- `ls_in_acl_hint`. ACL hint cho ct_state
- `ls_in_acl`. ACL từ table `ACL` trong NB
- `ls_in_qos_mark`: QoS mark
- `ls_in_qos_meter`: QoS meter
- `ls_in_lb`: Load balancer DNAT (frontend → backend)
- `ls_in_lb_aff_check`: LB affinity check
- `ls_in_stateful`. NAT commit
- `ls_in_arp_rsp`: ARP responder cho Logical Router port
- `ls_in_dhcp_options`: DHCPv4/DHCPv6 response
- `ls_in_dns_lookup`: DNS lookup
- `ls_in_dns_response`: DNS response
- `ls_in_external_port`: External port handling
- `ls_in_l2_lkup`: L2 destination lookup
- `ls_in_l2_unknown`: L2 unknown MAC (flood)

Mỗi table có Logical Flow entries với priority + match + action. Ví dụ ACL `from-lport direction=inbound priority=500 match=ip4.src==10.0.0.0/8 action=drop` compile thành:
```
table=ls_in_acl, priority=500, match=(inport == <port_groups> && ip4.src == 10.0.0.0/8), action=(drop;)
```

**Bước 3 — Compile Logical Flow cho Logical Switch egress pipeline:**
Tương tự ingress nhưng với egress, khoảng 10 table (egress ACL, egress LB, port security check egress, L2 output, v.v.).

**Bước 4 — Compile Logical Flow cho Logical Router ingress + egress:**
Logical Router phức tạp hơn, khoảng 30 table ingress + 15 table egress (OVN 22.03):
- `lr_in_admission`, kiểm soát admission
- `lr_in_lookup_neighbor`: ARP/NDP lookup
- `lr_in_learn_neighbor`: ARP learn
- `lr_in_ip_input`: IP input processing (giảm TTL, kiểm tra destination)
- `lr_in_unsnat`. SNAT ngược
- `lr_in_defrag`, defragmentation cho traffic LB
- `lr_in_dnat`. DNAT + LB
- `lr_in_ecmp_stateful`, chọn đường ECMP
- `lr_in_nd_ra_options`: IPv6 ND Router Advertisement
- `lr_in_ip_routing_pre`, hook pre-routing
- `lr_in_ip_routing`, tra cứu table định tuyến
- `lr_in_ip_routing_ecmp`, định tuyến ECMP
- `lr_in_policy`: Logical Router Policy (định tuyến theo policy)
- `lr_in_arp_resolve`, giải mã ARP cho next-hop
- `lr_in_chk_pkt_len`, kiểm tra chiều dài gói tin cho PMTU
- `lr_in_larger_pkts`, gửi ICMP "Gói tin Too Big" nếu cần
- `lr_in_gw_redirect`, chuyển hướng tới gateway chassis
- `lr_in_arp_request`, gửi ARP request

**Bước 5 — Port_Binding và Multicast_Group:**
Mỗi Logical_Switch_Port và Logical_Router_Port trong NB được tạo Port_Binding record trong SB. Port_Binding chứa `type` (vif/localnet/l2gateway/chassisredirect/patch), `options`, `tunnel_key`. Broadcast domain (flooding) được tính toán thành Multicast_Group, mỗi LS có một MG cho flood, mỗi LRP có MG cho neighbor discovery.

## 13.8.3 HA bầu leader của ovn-northd

Vì chỉ một ovn-northd active được phép ghi SB DB tại một thời điểm, cụm môi trường production OVN cần cơ chế HA đảm bảo (a) luôn có một active nếu node hiện tại crash, (b) tránh split-brain (hai active cùng ghi → bất đồng).

**Phương pháp 1 — Pacemaker (cũ):** dùng Pacemaker cluster manager quản lý resource `ovn-northd`. Pacemaker dùng Corosync để bầu leader và start/stop `ovn-northd.service` tương ứng. Phương pháp này phổ biến trên RHEL OpenShift 3.x nhưng phức tạp setup, đang dần bị thay thế.

**Phương pháp 2 — Bầu leader built-in (OVN 22.03+):** ovn-northd tự dùng ovsdb-server để bầu leader. Mỗi instance ovn-northd kết nối cả cụm NB DB và SB DB (Raft). Instance nào lấy được khóa `ovn_northd` trên SB DB trước → trở thành active. Các instance khác đăng ký khóa này, khi active release (tắt an toàn) hoặc mất kết nối, một standby tự động lấy khóa.

```
[node1]# ovn-northd --pidfile --detach \
         --ovnnb-db=tcp:10.0.0.1:6641,tcp:10.0.0.2:6641,tcp:10.0.0.3:6641 \
         --ovnsb-db=tcp:10.0.0.1:6642,tcp:10.0.0.2:6642,tcp:10.0.0.3:6642
[node2]# <cùng command, khác node>
[node3]# <cùng command, khác node>
```

Chỉ một instance active tại một thời điểm, xác định qua:
```
[host]# ovs-appctl -t ovn-northd nb-connection-status
[host]# ovs-appctl -t ovn-northd sb-connection-status
```
Instance active hiển thị `connected` cho cả hai; standby hiển thị `connected` NB nhưng `waiting` cho SB (đang chờ khóa).

**Phương pháp 3 — Kubernetes bầu leader (OVN-Kubernetes):** dùng Kubernetes lease object cho bầu leader. `ovn-k8s-northd` controller quản lý state qua `Lease` resource trong namespace `ovn-kubernetes`.

Thời gian failover (active chết → standby lên): 5-15 giây tùy cấu hình timeout. Trong khoảng thời gian này, SB DB không nhận được Logical Flow update mới (NB changes tồn đọng). Ovn-controller vẫn chạy bình thường vì đọc SB DB local cache, chỉ ý đồ cấu hình mới bị delay cho đến khi active mới lên.

## 13.8.4 Gỡ lỗi ovn-northd

Chuỗi gỡ lỗi ovn-northd thường xoay quanh: "tôi thay đổi NB nhưng SB không phản ánh" hoặc "Logical Flow SB sai so với NB ý đồ cấu hình".

**Công cụ 1, so sánh trạng thái NB và SB:**
```
[host]# ovn-nbctl show
[host]# ovn-sbctl show
[host]# ovn-sbctl list Logical_Flow | head -50
```

**Công cụ 2, ovn-northd coverage:**
```
[host]# ovs-appctl -t ovn-northd coverage/show
```
Xuất bộ đếm: `northd_run` (số lần tính toán lại), `lflow_generated` (số Logical Flow tạo), `port_binding_up` (số port binding). Giám sát qua thời gian để phát hiện bất thường.

**Công cụ 3, northd log:**
```
[host]# tail -f /var/log/ovn/ovn-northd.log
```
Mỗi lần tính toán lại log một entry:
```
2026-04-22T14:23:10.123Z|00042|ovn_northd|INFO|northd recompute took 45 ms
```
Nếu thời gian tăng bất thường (ví dụ vài giây) → NB phát triển quá lớn hoặc có loop thuật toán compile.

**Công cụ 4, đồng bộ nb_cfg:**
NB DB có bộ đếm `NB_Global.nb_cfg` tăng khi có thay đổi. ovn-northd copy bộ đếm này vào SB DB `SB_Global.nb_cfg` sau mỗi lần tính toán lại. So sánh hai bộ đếm:
```
[host]# ovn-nbctl get NB_Global . nb_cfg
42
[host]# ovn-sbctl get SB_Global . nb_cfg
42
```
Nếu khác nhau đáng kể (ví dụ NB = 42, SB = 38) → ovn-northd không catch up kịp thay đổi NB, có tồn đọng.

**Công cụ 5, ovn-northd đa flow (22.03+):**
Kiểm tra số thread:
```
[host]# ps -T -p $(pidof ovn-northd)
```
Mặc định ovn-northd dùng N thread với N = số CPU core / 2. Có thể override qua `--n-threads=4`. Theo dõi CPU per-thread để xác định điểm nghẽn.

### ▶ Guided Exercise 1: Quan sát ovn-northd compile NB thành SB Logical Flow

> **Trạng thái verification lab:** Output dựa trên OVN 22.03 docs; chờ kiểm chứng lab thực.

**Mục tiêu:** Chứng minh chuỗi compile NB → SB diễn ra tự động khi có thay đổi. So sánh NB ý đồ cấu hình với SB Logical Flow sinh ra.

**Chuẩn bị:** Node Ubuntu 22.04 với OVN 22.03 đã cài đặt.

**Bước thực hiện:**

**1.** Kiểm tra trạng thái NB và SB ban đầu trống:

```
[host]# ovn-nbctl show
<empty>
[host]# ovn-sbctl lflow-list | wc -l
0
```

**2.** Tạo Logical Switch qua NB:

```
[host]# ovn-nbctl ls-add web_ls
[host]# ovn-nbctl lsp-add web_ls web1
[host]# ovn-nbctl lsp-set-addresses web1 "aa:bb:cc:dd:ee:01 10.10.10.1"
```

**3.** Đợi 1-2 giây cho ovn-northd compile, rồi kiểm tra SB:

```
[host]# sleep 2
[host]# ovn-sbctl list Datapath_Binding
_uuid               : ...
tunnel_key          : 1
external_ids        : {name=web_ls, name2=""}
...
[host]# ovn-sbctl list Port_Binding
_uuid               : ...
logical_port        : "web1"
tunnel_key          : 1
...
[host]# ovn-sbctl lflow-list | grep "Datapath: \"web_ls\"" -A 5
```

**4.** Đo thời gian tính toán lại:

```
[host]# ovs-appctl -t ovn-northd coverage/show | grep northd_run
northd_run: 3
```

**5.** Thêm ACL và quan sát Logical Flow mới sinh:

```
[host]# ovn-nbctl acl-add web_ls from-lport 500 "ip4.src == 192.168.0.0/16" drop
[host]# sleep 2
[host]# ovn-sbctl lflow-list web_ls | grep "ls_in_acl"
```

Kết quả mong đợi: có entry mới với match `ip4.src == 192.168.0.0/16` và action `drop;`.

**Bài học học được:**

Thứ nhất, ovn-northd chuyển đổi mọi thay đổi NB thành Logical Flow trong vòng vài giây. Độ trễ tăng với kích thước cụm (cụm lớn 10K object có thể mất vài chục giây cho một lần tính toán lại toàn bộ).

Thứ hai, một object NB duy nhất (ví dụ một ACL rule) có thể sinh ra 3-5 Logical Flow (entry trong nhiều table để xử lý các trạng thái ct_state khác nhau). Đây là tại sao SB DB có thể lớn gấp 5-10 lần NB DB về số lượng record.

Thứ ba, pipeline model của OVN (24 ingress table + 10 egress table cho LS, 30 + 15 cho LR) là **ngôn ngữ trung gian** giữa ý đồ cấu hình cấp cao của CMS và OpenFlow cấp thấp, mọi tính năng OVN đều ánh xạ vào một số table cụ thể trong pipeline.

**Cleanup:**

```
[host]# ovn-nbctl ls-del web_ls
```

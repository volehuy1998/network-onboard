## 13.9.1 Load Balancer architecture in OVN

OVN provides a native L4 (TCP, UDP, SCTP) Load Balancer in the data plane, without needing a separate userspace process such as HAProxy. The `Load_Balancer` table in NB DB defines a mapping from a **virtual IP (VIP):port** to a list of backend **real IP:port** entries.

**Structure of the Load_Balancer table in NB DB:**
- `name`: identifier name (optional).
- `vips`: map from "VIP:port" to "backend1:port1, backend2:port2, ..." (string).
- `protocol`: "tcp", "udp", "sctp", or empty (treated as TCP).
- `health_check`: reference to the `Load_Balancer_Health_Check` table (optional).
- `ip_port_mappings`: map "endpoint_ip to mac:port" used to generate ARP responses (for VIPs in a Logical Switch).
- `selection_fields`: fields used for the hash (default: src_ip, src_port, dst_ip, dst_port).
- `options`: extension flags (hairpin_snat_ip, reject, skip_snat, affinity_timeout).

**Attaching an LB to a Logical Switch or Logical Router:**
- `ovn-nbctl ls-lb-add ls1 lb1`: the LB is applied to traffic entering the Logical Switch.
- `ovn-nbctl lr-lb-add lr1 lb1`: the LB is applied to traffic entering the Logical Router.
- One LB can be attached to multiple LSes or LRs at the same time (sharing one VIP across networks).

**LB processing pipeline:**
- In the **Logical Switch ingress pipeline**, the table `ls_in_pre_lb` (table 6) marks packets destined to a VIP as needing to traverse the LB.
- The table `ls_in_lb` (table 12) performs DNAT: it replaces the destination from VIP:port to backend:port according to the hash.
- The table `ls_in_lb_aff_check` (table 13) checks session affinity.
- The table `ls_in_stateful` (table 14) commits the connection to conntrack so that the reply traffic can be un-DNAT (reverse SNAT).

- In the **Logical Router ingress pipeline**, the table `lr_in_defrag` (table 5) defragments packets before LB processing.
- The table `lr_in_dnat` (table 6) performs DNAT for the LB.
- The table `lr_in_ecmp_stateful` (table 7) selects an ECMP path when multiple backends meet the criteria.

## 13.9.2 Backend hash selection algorithm

When a packet arrives matching a VIP, OVN must choose a backend from the list. The default algorithm is a **5-tuple hash**:
```
hash(src_ip, src_port, dst_ip, dst_port, protocol) % len(backends)
```

This hash has two important properties:
- **Consistency:** the same client (same src_ip and src_port) with the same destination always maps to the same backend (as long as the backend list does not change).
- **Even distribution:** with many clients, traffic is spread evenly across backends.

**Problem when the backend set changes:** if a backend is removed (scale-down or health-check failure), the hash modulo changes completely, and roughly 50 percent of current clients are remapped to a different backend, breaking the connection (conntrack loses tracking). This is why OVN implements **conntrack-based pinning**: when a connection is established, the conntrack entry stores the chosen backend. Packets of the same connection always go to the chosen backend, no matter how the hash modulo changes. Only NEW connections are remapped.

**Custom selection fields:** the user can override the hash input via `selection_fields`:
```
[host]# ovn-nbctl --selection-fields="ip_src,ip_dst" lb-add lb1 ...
```
This is useful for sticky sessions based on src_ip (independent of the random src_port).

**ECMP path selection:** when an LB is attached to a Logical Router and there are multiple backends, OVN can use **symmetric hashing** to ensure that forward traffic and reply traffic choose the same path (avoiding asymmetric routing that causes conntrack misses).

## 13.9.3 SNAT handling for reply packets

When a client sends a packet to a VIP, DNAT changes the destination to the backend IP. The packet is forwarded to the backend. The backend processes it and sends a reply, with destination equal to the client IP and source equal to the backend IP.

**Problem:** the client does not know about the backend; it sent the request to the VIP and expects a reply from the VIP. If the reply source equals the backend IP, the client drops the reply (TCP mismatch).

**OVN solution:** the conntrack entry (created during DNAT in the forward direction) holds the VIP-to-backend mapping. When the reply packet traverses the OVN pipeline, conntrack matches it and automatically performs **un-DNAT**, changing the source from the backend IP to the VIP. The client receives a reply with source equal to the VIP, which is valid.

**Skip-SNAT option:** in some cases (DSR, Direct Server Return), the backend has the VIP configured on a loopback interface and generates the reply itself with source equal to the VIP (bypassing OVN). To prevent OVN from performing un-DNAT on the reply (which is unnecessary), use:
```
[host]# ovn-nbctl set Load_Balancer lb1 options:skip_snat=true
```

**Hairpin:** when the backend is itself a pod in the same Logical Switch and that pod sends a request to the VIP of a service for which it is a backend, the packet loops back to itself. Without special handling, the packet has:
- src_ip = pod_IP (= backend_IP)
- dst_ip = VIP, then DNAT, then dst_ip = backend_IP = pod_IP

The result: the pod sends a packet to itself with source equal to destination. The kernel TCP stack drops it because of the same source and destination. OVN handles hairpin by **SNAT** of the source of the hairpin packet to a "hairpin SNAT IP" (configured via `options:hairpin_snat_ip`), making the packet appear as if it comes from a different host:
```
[host]# ovn-nbctl set Load_Balancer lb1 options:hairpin_snat_ip=169.254.169.254
```

## 13.9.4 Session affinity

For HTTP and HTTPS traffic, a single client may send many requests over many TCP connections. Each connection has a different src_port, so the hash may pick a different backend, and session state (authentication state, shopping cart) is lost.

**OVN session affinity** ensures that the same client (same src_ip) always reaches the same backend within a configured timeout. Implementation:
```
[host]# ovn-nbctl lb-add lb1 10.0.0.100:80 "192.168.1.1:80,192.168.1.2:80" \
         -- set Load_Balancer lb1 options:affinity_timeout=3600
```

Under the hood, OVN uses an auxiliary hash table (`lb_aff_check` and `lb_aff_learn`) that records the mapping `(client_ip, vip) to backend_chosen` with a TTL equal to affinity_timeout. When a new connection from the same client arrives, OVN looks up this table before hashing; if there is an entry, it uses the previously chosen backend; otherwise it computes a new hash and writes a new entry.

**Limitation:** session affinity is based on src_ip, so if clients sit behind a NAT (many clients sharing one public IP), they are all mapped to the same backend (no distribution). For internal Kubernetes workloads, the client IP is the pod IP, so affinity is precise.

## 13.9.5 Health check and active or passive backends

OVN supports backend health checks through the `Load_Balancer_Health_Check` table:
```
[host]# ovn-nbctl --id=@hc create Load_Balancer_Health_Check \
         vip='"10.0.0.100:80"' \
         options:interval=5 options:timeout=3 options:success_count=1 options:failure_count=3 \
         -- add Load_Balancer lb1 health_check @hc
```

Each chassis running ovn-controller sends a **TCP SYN** (or ICMP echo, depending on configuration) to each backend IP:port in the LBs that the chassis hosts. If the backend does not reply (timeout 3 seconds, 3 consecutive failures), the backend is marked `offline`. OVN writes the result into the `Service_Monitor` table in SB DB. ovn-northd reads `Service_Monitor`, removes offline backends from the active set, and emits new Logical Flows that exclude offline backends.

**Distributed health-check architecture:** each chassis probes its own subset of backends (typically the backends that live on the same chassis if any, or assigned by hash to spread load). The aggregated result is exposed through the `Service_Monitor` table. Advantage: scales well (no single probing point); disadvantage: each backend is probed multiple times (from many chassis).

**Integration with Kubernetes readiness probes:** in OVN-Kubernetes, the readinessProbe of a pod is translated into `Load_Balancer_Health_Check` for the corresponding Service endpoint. If the readinessProbe of a pod fails, the backend is removed from the OVN LB and Service traffic is no longer routed to that pod.

## 13.9.6 Comparison with kube-proxy (iptables and IPVS)

| Property | OVN LB | kube-proxy iptables | kube-proxy IPVS |
|----------|--------|---------------------|-----------------|
| Hash algorithm | Consistent 5-tuple | Random (iptables `-m statistic`) | Round-robin, LC, WLC, SH |
| Session affinity | Yes, via ClientIP | Yes, via ClientIP | Yes, native IPVS |
| Health check | Yes, TCP, ICMP | No (relies on kubelet readiness) | No native |
| Location | OVS datapath (kernel) | iptables chain | IPVS kernel module |
| Scaling | O(1) lookup via conntrack | O(n) chain traversal | O(1) hash |
| Update latency | Sub-second (ovn-northd) | Seconds (rebuild iptables) | Sub-second (ipvsadm) |

**When to choose OVN LB:** the cluster already uses OVN-K8s as its CNI, so an LB is native and kube-proxy is not needed. Advantages: a uniform data plane and a single tool (ovn-trace) for both routing and LB debugging.

**When to keep kube-proxy:** the cluster uses a different CNI (Calico, Flannel) without a native LB, so kube-proxy is required. Or the cluster needs a specific IPVS feature (LC, WLC algorithms) that OVN does not provide.

## 13.9.7 Source-code anchor: `ct_lb` action implementation

OVN Load Balancer operates through the OpenFlow action `ct_lb` in OVS. The action is implemented in the userspace ovs-vswitchd plus the kernel openvswitch.ko module with conntrack integration.

### `ct_lb()` action syntax (ovs-actions(7))

```
ct_lb(args)
ct_lb(backends=ip[:port][, ip[:port]...][; selection_method=method])
```

Parameters:

- `backends`: list of backends `ip:port,ip:port,...`.
- `selection_method`: `dp_hash` (default, kernel hash), `hash` (5-tuple userspace), `random`.

Source code anchor (OVS `v2.17.9`):

- [`lib/ofp-actions.c`](https://github.com/openvswitch/ovs/blob/v2.17.9/lib/ofp-actions.c), function `parse_ct_lb()`, parses the `ct_lb(...)` syntax from OpenFlow text into an internal struct.
- [`ofproto/ofproto-dpif-xlate.c`](https://github.com/openvswitch/ovs/blob/v2.17.9/ofproto/ofproto-dpif-xlate.c), function `compose_conntrack_action()`, builds the datapath action with the CT_LB flag.
- Kernel side: `net/netfilter/nf_conntrack_extend.c` and `net/openvswitch/conntrack.c` handle the CT_LB extension and the NAT rewrite.

### Pipeline that OVN compiles to `ct_lb`

ovn-northd compiles the NB Load_Balancer into Logical_Flow at these stages:

```
table=ls_in_pre_lb (table 6):
  priority=100, match=(ip && reg0[2] == 1),
  action=(reg0[2] = 0; ct_next;)
  
table=ls_in_lb (table 12):
  priority=100,
  match=(ct.new && ip4.dst == 10.0.0.100 && tcp.dst == 80),
  action=(reg1 = 10.0.0.11;
          reg2[0..15] = 8080;
          ct_lb(backends=10.0.0.11:8080,10.0.0.12:8080);)
  
table=ls_in_stateful (table 14):
  priority=100, match=(reg0[1] == 1 && ip4),
  action=(ct_commit { ct_label.natted = 1; }; next;)
```

Three-stage mapping:

1. `ls_in_pre_lb` marks the packet "has gone through conntrack lookup" (`ct_next`).
2. `ls_in_lb` performs DNAT through the `ct_lb` action.
3. `ls_in_stateful` commits a conntrack entry with `ct_label.natted = 1` so that the reply path knows to un-DNAT.

### `ct_lb` runtime mechanism

When a packet hits the `ct_lb` action in the kernel datapath:

1. The kernel netfilter hook calls `nf_nat_setup_info(NF_NAT_DST)`.
2. A 5-tuple hash selects the backend (using jhash plus modulo by backend count).
3. Backend rewrite: `dst_ip` is replaced with the backend IP and `dst_port` with the backend port.
4. A new conntrack entry is created with two tuples:
   - Original: `src=client, dst=VIP, sport=client_port, dport=VIP_port`.
   - Reply: `src=backend, dst=client, sport=backend_port, dport=client_port`.

Reply path: the kernel auto-matches the conntrack entry and swaps `src=backend` to `src=VIP` (un-DNAT). The client sees a reply with source equal to the VIP, fully transparent.

## 13.9.8 Service_Monitor table SBDB schema

Distributed health-check results are aggregated in the `Service_Monitor` table in SB DB. ovn-northd reads this table and excludes offline backends from `Logical_Flow`.

```
[host]# ovn-sbctl list Service_Monitor
_uuid               : abc123-...
chassis_name        : "chassis-A"
external_ids        : {}
ip                  : "10.0.0.11"
logical_port        : "backend1"
options             : {failure_count="3", interval="5", success_count="1", timeout="3"}
port                : 8080
protocol            : tcp
src_ip              : "10.0.0.1"
src_mac             : "aa:bb:cc:99:00:01"
status              : online        <- key field, "online" or "offline" or "error"
```

| Field | Description | Healthy | Critical |
|-------|-------------|---------|----------|
| `chassis_name` | Which chassis probes this backend | Match expected per-chassis assignment | Empty |
| `ip` plus `port` | Backend address | Match LB backend list | (n/a) |
| `logical_port` | LSP of the backend | Match NB | Mismatch |
| `options:interval` | Probe interval (seconds) | 5 to 30 seconds | < 1 second (probe storm), > 60 seconds (slow detection) |
| `options:success_count` | Consecutive successes to mark online | 1 to 3 | > 5 (slow recovery) |
| `options:failure_count` | Consecutive failures to mark offline | 3 to 5 | < 2 (high false-positive rate) |
| `status` | Latest probe result | `online` for a healthy backend | `offline` (backend down), `error` (probe configuration failed) |

### Distributed probe assignment

Each chassis assigns itself a subset of backends to probe based on the hash of `chassis_uuid` and `backend_ip`. Pseudo-code:

```c
// Pseudo-code for ovn-controller
for (backend in lb_backends) {
    if (hash(my_chassis_uuid, backend_ip) % total_chassis == my_chassis_index) {
        start_probe(backend);
    }
}
```

100 backends times 10 chassis means each chassis probes 10 backends. If 1 chassis goes down, 10 backends lose their probe and other ovn-controller chassis pick them up after a re-hash.

### TCP SYN probe semantics

A probe of type `tcp` sends a SYN to `backend_ip:port`. There are three outcomes:

- **SYN-ACK**: backend healthy. Mark `online` and RST the connection.
- **RST or ICMP unreachable**: backend rejects, mark `offline`.
- **No response (timeout)**: mark `offline` after `failure_count` consecutive timeouts.

A probe of type `icmp` is a ping. A probe of type `sctp` is an SCTP INIT. Production typically uses `tcp` for L4 services.

UUID                                    LB                  PROTO      VIP                    IPs

|-------|---------|---------|----------|-----------------|

| `IPs` count | 1 or more backends, all reachable | 1 backend (no redundancy) | Empty | Service unavailable |
| `PROTO` | `tcp`, `udp`, `sctp` | Empty (default tcp) | Mismatch with client | Connection refused |
| `Health_Checks` | UUID set in production | Empty (no HC) | (n/a) | Without HC, dead backends still receive traffic |
| `affinity_timeout` | 0 (no affinity) or > 0 | (per requirement) | (n/a) | (per spec) |
| `hairpin_snat_ip` | Set when backend is a pod in the same LS | Empty for cross-LS | (n/a) | (per spec) |

### `ovn-sbctl list Service_Monitor`

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| `status: online` | All backends `online` | 1 backend `offline` | All `offline` | Service down; check backend health |
| `chassis_name` set | Distributed across chassis | Concentrated on 1 chassis | Empty | Probe assignment failure |
| Probe interval | 5 to 30 seconds | < 5 seconds (network load) | > 60 seconds | Slow failure detection |
| `options:failure_count` | 3 to 5 | < 3 (sensitive) | > 10 (slow detection) | Tune to match the SLA |

**Failure-mode scenarios:**

- The LB has 5 backends but `Service_Monitor` has only 3 entries: 2 backends were not assigned a probe. Run `ovn-sbctl list Chassis` to see the chassis count and verify the hash distribution.
- All `status=offline` suddenly: this is a network underlay issue, not a backend issue. Verify by pinging the backends from the chassis directly. Or `options:timeout` is too low for the actual RTT.
- A backend is `online` in NB but the client gets connection refused: NB-SB sync lag. Check `nb_cfg` (Part 10.0).

## 13.9.10 Capstone POE: 5-tuple hash versus Maglev hash for production LB

> **Context:** an OVN cluster with an LB serving 100K connections per second. Backend pool of 10 servers. When 1 backend is removed (scale down), how many connections are remapped?

**Claim to verify:** "OVN uses a 5-tuple hash modulo the backend count, so when 1 backend is removed roughly 50 percent of connections are remapped (modulo math)."

**Predict:** correct for **new connections**, incorrect for **already-established connections** (because of conntrack pinning).

**Observe (assumed):**

- Set up an LB with 10 backends and generate 100K active TCP connections.
- Remove 1 backend via `ovn-nbctl set Load_Balancer lb1 vips:'10.0.0.100:80'='10.0.0.11:8080,...,10.0.0.20:8080'` (9 backends).
- Wait 5 seconds for ovn-northd to compile and ovn-controller to install the flows.
- Measure:
  - New connections (after the update): hashed across 9 backends, even distribution at 11.1 percent per backend.
  - Old connections (still ESTABLISHED): keep the old backend through the conntrack entry. **0 percent remap.**
  - Connections that belonged to the removed backend: **100 percent dropped** (the backend is gone).

Compared to "modulo math 50 percent remap": that estimate applies only to a **stateless hash** without conntrack pinning. OVN has conntrack, so old connections keep their pinning.

### Maglev hash comparison

Maglev (Google) is a different consistent-hashing scheme. When a backend is removed:

- **Maglev:** about 1 over N connections remap (1 over 10 = 10 percent in this case).
- **OVN modulo:** 0 percent remap of old connections (conntrack pinning), but new connections fully redistribute.
- **kube-proxy iptables `-m statistic`:** about 50 percent remap (purely statistical).

The OVN choice is pragmatic: rely on conntrack rather than on a sophisticated algorithm. Trade-off: simpler, but a high connection rate plus a high backend churn put pressure on cache memory.

**Explain:**

1. **OVN is not pure modulo hash**: a conntrack entry pins each connection to a backend, immune to hash modulo changes.
2. **New connections fully redistribute** after a backend update; this is acceptable (5-second interval, application reconnect).
3. **Maglev is not necessary** for typical OVN scale (10 to 100 backends). Modulo plus conntrack pinning is good enough.

**Falsification test:**

- Run 100K active connections, remove 1 backend, observe the drop count.
- Connection drops equal the connections that belonged to the removed backend (10K = 10 percent).
- Connections retained equal those on the remaining backends (90K).
- If the drop rate is more than 10 percent, conntrack pinning is broken or the connections were not yet ESTABLISHED.

**Where the original claim is wrong:** "50 percent remap from modulo math" assumes a stateless hash. OVN has conntrack pinning, so old connections are not remapped. The correct statement is: **drops equal the connections of the removed backend (10 percent in a 10-backend case). New connections fully redistribute.**

> **Rule of thumb:** OVN LB scales well for 10 to 100 backends with low churn. For high churn plus a high connection rate, consider Maglev hash or an external LB (HAProxy, Nginx).

### Guided Exercise 1: Create an OVN Load Balancer and observe conntrack NAT

> **State verification lab:** the output is based on the OVN 22.03 documentation; awaiting a real lab to verify.

**Goal:** create an LB with 2 backends, send traffic through the VIP, and observe DNAT in conntrack and the backend hash selection.

**Preparation:** an Ubuntu 22.04 lab with OVN 22.03, one Logical Switch with 3 ports (1 client and 2 backends).

**Steps:**

**1.** Create the topology:
```
[host]# ovn-nbctl ls-add ls1
[host]# ovn-nbctl lsp-add ls1 client
[host]# ovn-nbctl lsp-set-addresses client "aa:00:00:00:00:01 10.0.0.10"
[host]# ovn-nbctl lsp-add ls1 backend1
[host]# ovn-nbctl lsp-set-addresses backend1 "aa:00:00:00:00:02 10.0.0.11"
[host]# ovn-nbctl lsp-add ls1 backend2
[host]# ovn-nbctl lsp-set-addresses backend2 "aa:00:00:00:00:03 10.0.0.12"
```

**2.** Create the Load Balancer:
```
[host]# ovn-nbctl lb-add lb1 10.0.0.100:80 "10.0.0.11:8080,10.0.0.12:8080" tcp
[host]# ovn-nbctl ls-lb-add ls1 lb1
```

**3.** Inspect the Logical Flow in the table `ls_in_lb`:
```
[host]# ovn-sbctl lflow-list ls1 | grep "ls_in_lb"
```

Expected output:
```
table=12 (ls_in_lb), priority=100,
  match=(ct.new && ip4.dst == 10.0.0.100 && tcp.dst == 80),
  action=(...ct_lb(backends=10.0.0.11:8080,10.0.0.12:8080)...)
```

**4.** Inspect the OpenFlow rules on br-int:
```
[host]# ovs-ofctl dump-flows br-int | grep "tp_dst=80" | head -5
```

**5.** Send traffic and observe conntrack:
```
[host]# conntrack -L -d 10.0.0.100
tcp  ESTABLISHED src=10.0.0.10 dst=10.0.0.100 sport=54321 dport=80
  [REPLY] src=10.0.0.11 dst=10.0.0.10 sport=8080 dport=54321
  mark=0 zone=0 use=1
```

The conntrack entry shows the DNAT: a packet sent to `10.0.0.100:80` is redirected to `10.0.0.11:8080`, and the reply with source `10.0.0.11:8080` is mapped back to `10.0.0.100:80` (un-DNAT).

**Lessons learned:**

First, the OVN LB relies entirely on conntrack and has no separate state for the LB. This means that every conntrack feature (timeout, zone, mark) applies to LB traffic.

Second, the backend hashing is performed by the `ct_lb` action in OpenFlow, a special OVS action that supports native load balancing in the kernel datapath. There is no need for userspace involvement, so performance is high.

Third, when a backend is removed (for example, `ovn-nbctl lb-add lb1 10.0.0.100:80 "10.0.0.12:8080" tcp`), the old conntrack entries remain. Existing connections continue to the old backend (now terminated) and will eventually time out. New connections re-hash against the new list.

**Cleanup:**
```
[host]# ovn-nbctl lb-del lb1
[host]# ovn-nbctl ls-del ls1
```

## 13.9.X Backfill v3.5: `selection_fields`, `hairpin_snat_ip`, `Load_Balancer_Group`, `Load_Balancer_Health_Check`

> **Purpose:** the J.1 audit found 4 LB feature gaps: `selection_fields` (2 shallow files), `hairpin_snat_ip` (2 shallow files), `Load_Balancer_Group` (1 shallow file), `Load_Balancer_Health_Check` (3 shallow files). This section backfills them, source-verified against the `ovn-nb(5)` man page.

### 13.9.X.1 `Load_Balancer.selection_fields`, consistent hashing

**Bucket:** NB schema for LB consistent hashing.
**Field:** `Load_Balancer.selection_fields` (set of strings)
**Default:** empty = use the 5-tuple hash (eth_src, eth_dst, ip_src, ip_dst, ip_proto).
**Effect:** override the hash key used for backend selection. Allows pinning by a subset of fields.

**Operational interpretation:**

The LB DNAT chooses a backend by hashing the default 5-tuple. Every packet in the same connection (same 5-tuple) hashes to the same backend, so the choice is consistent. Problem: a client behind NAT has the same dest plus proto plus src_port plus dst_port (NAT preserves them) but a different src_ip per connection, so the hash changes and each connection lands on a different backend. For session-based applications (sticky sessions) this is a problem.

`selection_fields` allows pinning by a subset. Set `selection_fields=["ip_src"]` to hash only by source IP, so every connection from the same client IP goes to the same backend (sticky).

**Available fields (verified against `ovn-nb(5)`):**

| Field | Purpose |
|-------|---------|
| `eth_src` | Source MAC |
| `eth_dst` | Dest MAC |
| `ip_src` | Source IP |
| `ip_dst` | Dest IP |
| `tp_src` | L4 source port |
| `tp_dst` | L4 dest port |

**Example:**

```bash
# Sticky sessions by source IP
ovn-nbctl set Load_Balancer web-lb selection_fields="ip_src"

# Hash by (source IP + dest port), useful for TLS SNI multi-port
ovn-nbctl set Load_Balancer web-lb selection_fields='ip_src,tp_dst'
```

**Trade-off:**
- Default 5-tuple: good load balance plus connection-level stickiness.
- `ip_src` only: client-level stickiness, but possibly uneven load (one big client makes one backend hot).
- Custom subset: a balance between stickiness and load distribution.

### 13.9.X.2 `Load_Balancer.options:hairpin_snat_ip`, hairpin NAT source IP

**Bucket:** NB schema for LB hairpin handling.
**Field:** `Load_Balancer.options:hairpin_snat_ip` (IP address)
**Default:** unset = no SNAT for hairpin
**Effect:** when the backend is also the client (hairpin), SNAT the source IP to this value.

**Operational interpretation:**

A hairpin LB occurs when client and backend share the same LSP (rare but possible: pod A accesses a service VIP whose backend is pod A itself). By default the LB performs DNAT VIP to backend IP. But if the source IP remains the client (= the backend), the backend receives a packet whose source equals itself, which causes confusion (loop or self-detect drop).

`hairpin_snat_ip` rewrites the source IP for hairpin packets, so the backend sees an "external" client IP rather than itself.

**LS_IN pipeline stages 13 to 15 (see Part 13.16 Â§13.16.4):**

1. Stage 13 `LS_IN_PRE_HAIRPIN`: detect hairpin (client and backend on the same LSP). Set `REGBIT_HAIRPIN`.
2. Stage 14 `LS_IN_NAT_HAIRPIN`: if REGBIT is set and `hairpin_snat_ip` is configured, SNAT the source IP to `hairpin_snat_ip`.
3. Stage 15 `LS_IN_HAIRPIN`: redirect the packet back to the backend (which is also the client).

**Example:**

```bash
# Set the hairpin SNAT IP for LB web-lb (simulating an "external proxy IP")
ovn-nbctl set Load_Balancer web-lb options:hairpin_snat_ip=169.254.169.254
```

The backend receives a packet with source IP 169.254.169.254 (link-local, recognisable as an "internal proxy"), not its own IP.

**Note:** this applies only to hairpin (a rare case). Non-hairpin LBs are not affected.

### 13.9.X.3 Table `Load_Balancer_Group`, LB aggregation for LR

**Bucket:** NB schema for aggregate LB application.
**Schema:** `Load_Balancer_Group` table in NB DB.
**Key columns:**
- `name` (string): group name
- `load_balancer` (set of UUID): list of Load_Balancer entries in the group

**Operational interpretation:**

`Load_Balancer_Group` aggregates many Load_Balancer entries into one unit, applied at once to one Logical_Router (or to many Logical_Switches).

Use case: a Logical_Router gateway has 50 service VIPs. Instead of attaching 50 LB entries to the LR (which inflates NB schema complexity and slows compile), create one `Load_Balancer_Group` containing the 50 entries and attach the group to the LR.

**Example:**

```bash
# Create the LB group
ovn-nbctl create Load_Balancer_Group name=web-services-group

# Add LBs to the group
LB1=$(ovn-nbctl --bare --columns=_uuid find Load_Balancer name=web-lb)
LB2=$(ovn-nbctl --bare --columns=_uuid find Load_Balancer name=api-lb)
ovn-nbctl add Load_Balancer_Group web-services-group load_balancer $LB1 $LB2

# Apply the group to the LR
ovn-nbctl set Logical_Router lr-main load_balancer_group=web-services-group
```

Compared to attaching each LB:
```bash
# Old way: attach each LB to the LR
ovn-nbctl lr-lb-add lr-main $LB1
ovn-nbctl lr-lb-add lr-main $LB2
# ... 50 commands
```

The group way: one command and central management. An update to the group automatically propagates to every LR using the group.

### 13.9.X.4 Table `Load_Balancer_Health_Check`, backend health probe

**Bucket:** NB schema for LB backend health monitoring.
**Schema:** `Load_Balancer_Health_Check` table in NB DB.
**Key columns:**
- `vip` (string): VIP that needs health checks
- `options` (map):
  - `interval`: probe interval in seconds (default 10)
  - `timeout`: probe timeout in seconds (default 5)
  - `success_count`: consecutive successes to mark UP (default 1)
  - `failure_count`: consecutive failures to mark DOWN (default 3)

**Operational interpretation:**

OVN integrates Service_Monitor (a SB DB table) to probe backends periodically. Each backend has a session probe over TCP, UDP, or ICMP. A failure count above the threshold marks the backend DOWN, which excludes it from forwarding (the LB DNAT skips that backend).

**Probe protocol:**
- TCP: TCP SYN, expect SYN-ACK
- UDP: UDP echo-like (limited use case)
- ICMP: ICMP echo

**Example:**

```bash
# Create a health check for VIP 10.96.1.10
ovn-nbctl create Load_Balancer_Health_Check vip='"10.96.1.10:80"' options:interval=5 options:timeout=3 options:failure_count=2

# Link with the LB
HC_UUID=$(ovn-nbctl --bare --columns=_uuid find Load_Balancer_Health_Check vip='"10.96.1.10:80"')
ovn-nbctl set Load_Balancer web-lb health_check=$HC_UUID

# Verify status (read SB DB Service_Monitor)
ovn-sbctl list Service_Monitor
# Output: status=online/offline per backend
```

**`ip_port_mappings` column:**

`Load_Balancer.ip_port_mappings` (map IP to MAC plus LSP) is needed when backends have multiple IP-MAC mappings used as the source for health checks. Format: `"10.0.0.1:fa:16:3e:01:01:01:lsp1"`.

OVN needs to know the MAC and LSP of the probe source so that it can send ARP and ND correctly and so that replies are tracked back to the right probe session.

**Example:**

```bash
ovn-nbctl set Load_Balancer web-lb \
  ip_port_mappings='10.0.0.1=fa:16:3e:01:01:01\\:lsp-backend-1,10.0.0.2=fa:16:3e:02:02:02\\:lsp-backend-2'
```

## 13.9.Z Load_Balancer as an NBDB table: deep dive into L4 load balancing

> This section dives deep into the **`Load_Balancer` table itself** (a row in the OVN Northbound database), complementing the practical teaching in Â§13.9.1 and the pipeline introduced in Â§13.9.7 and Â§13.9.X (kept intact). Goal: the engineer fully understands the structure of an NBDB Load_Balancer row (8 schema columns in 22.03), its compilation lifecycle into Logical_Flow at the stages `LS_IN_PRE_LB`, `LS_IN_LB`, `LS_IN_LB_AFF_CHECK`, `LS_IN_PRE_HAIRPIN`, `LS_IN_NAT_HAIRPIN`, and `LS_IN_HAIRPIN`, and how to trace a runtime incident (VIP timeout, uneven backend distribution, hairpin loop, conntrack stale entry after scale-down) back to one specific row in NBDB.

### Concept

`Load_Balancer` is a **table that lives in the OVN Northbound database (NBDB)** and describes an **L4 load-balancing service** that the CMS (Neutron LBaaS or Octavia, Kubernetes Service ClusterIP, kube-ovn ovn-loadbalancer, Red Hat OSP director) declares to map one or more `VIP:port` pairs to a list of backend `IP:port` entries. Each row represents **one** LB service and may contain multiple VIP pairs (the `vips` column is a map). Type: **OVSDB table**, a **root table** (`isRoot=true` in `ovn-nb.ovsschema` branch-22.03), meaning the row exists independently and does not depend on any parent. It is referenced from `Logical_Switch.load_balancer` (set of UUID, weak reference), `Logical_Router.load_balancer` (set of UUID, weak reference), or `Load_Balancer_Group.load_balancer` (weak reference). The key difference from HAProxy or Nginx: the OVN Load_Balancer is not a userspace process; it is a **declarative entry in the NBDB Raft store**, ovn-northd compiles it into Logical_Flow, every chassis ovn-controller installs OpenFlow rules into `br-int`, and the `ct_lb` action in the OVS kernel datapath performs DNAT and backend selection by hash. The whole LB data plane runs in the kernel; there is no userspace context switch; performance is line-rate.

### History and background

The `Load_Balancer` table appeared with OVN 2.6 (2016) when Russell Bryant (Red Hat) and Han Zhou (eBay, then Red Hat) designed a native LB feature for OVN-Kubernetes integration. Before that, OpenStack implemented LB via **Octavia plus HAProxy amphora VMs**, where each VIP spawned a dedicated amphora VM running HAProxy in userspace. Issues: 100 VIPs equalled 100 VMs equalling about 50 GB of cluster-wide RAM only for LB, the RTT added 1 to 2 ms (host to amphora to backend), and HA failover via keepalived was slow at 5 to 10 seconds. Kubernetes implemented LB via **kube-proxy iptables `-m statistic`** with random hashing (50 percent remap when backends change), or via **IPVS** (better but separated from the main data plane). OVN solves the problem by compiling LB rules down to the `ct_lb` OF action in OVS, so the kernel datapath performs DNAT plus hashing in the fast path and uses conntrack to pin connections to backends.

Schema evolution (verified against [`ovn-nb.ovsschema` branch-22.03](https://raw.githubusercontent.com/ovn-org/ovn/branch-22.03/ovn-nb.ovsschema) and [`ovn-nb.xml` branch-22.03](https://raw.githubusercontent.com/ovn-org/ovn/branch-22.03/ovn-nb.xml)): OVN 2.6 base schema with 4 columns (`name`, `vips`, `protocol`, `external_ids`); OVN 2.8 adds `health_check` (strong reference to `Load_Balancer_Health_Check`); OVN 2.10 adds `ip_port_mappings` (map endpoint_ip to mac:port for ARP and ND probing); OVN 21.03 adds `selection_fields` (set enum of 6 values: `eth_src`, `eth_dst`, `ip_src`, `ip_dst`, `tp_src`, `tp_dst`) for custom hash keys, and adds the `Load_Balancer_Group` table for grouping; OVN 21.06 adds the `options` map (`hairpin_snat_ip`, `affinity_timeout`, `skip_snat`, `reject`); OVN 22.03 stabilises 8 columns total and adds `options:event=true` for controller-event sampling; OVN 22.09 adds `options:add_route=true` so an LR LB can auto-add a static route to the VIP; OVN 23.03 deprecates `ip_port_mappings` for most use cases (kept only for explicitly defined probe sources); OVN 24.03 adds `options:template=true` for cluster-wide template LBs.

### Position in the architecture

The table sits at the **NBDB layer** and is a **root table** at the very top of the OVN control-plane input (unlike `ACL`, which is not a root table). An OVN cluster may have hundreds to thousands of `Load_Balancer` rows (one row per Service ClusterIP in a large Kubernetes cluster, or one row per VIP in an OpenStack Octavia deployment). Reference scope: a row is referenced through a **weak reference** from `Logical_Switch.load_balancer` (set of UUID), `Logical_Router.load_balancer` (set of UUID), or indirectly via `Load_Balancer_Group.load_balancer` and then `Logical_Switch.load_balancer_group` or `Logical_Router.load_balancer_group`. Because the reference is weak, deleting a Logical_Switch or Logical_Router does not cascade-delete the Load_Balancer (unlike ACL); the Load_Balancer row stays alive until the CMS calls `ovn-nbctl lb-del` explicitly. Impact scope after compile: changing one Load_Balancer row (adding a backend, changing the protocol, setting an option) makes ovn-northd recompute the Logical_Flow of **every datapath** (LS or LR) that references the LB; ovn-controller on the corresponding chassis installs the new flow within 1 second through the I-P engine. An LB is never applied at the `Logical_Switch_Port` or `Logical_Router_Port` level (only at the datapath level).

### Roles and responsibilities

Load_Balancer carries **five roles** in the OVN architecture:

1. **Declarative L4 service definition**: the `vips` column (map "VIP:port" to "backend1:port1,backend2:port2,...") is the original intent and the lingua franca between the CMS (Octavia, kube-ovn) and the data plane.
2. **Choice of L4 protocol**: the `protocol` column enum (`tcp`, `udp`, `sctp`, optional empty = TCP default) determines whether northd compiles the match expression `tcp.dst==X`, `udp.dst==X`, or `sctp.dst==X`, and which conntrack helper is used.
3. **Customising the backend selection algorithm**: the `selection_fields` enum set overrides the default 5-tuple hash key. Set `["ip_src"]` for sticky sessions by source IP; set `["ip_src", "tp_dst"]` for TLS SNI multi-port. This implements true consistent hashing (OVN 21.03 and later); ovn-northd emits `ct_lb_mark()` with a `hash_fields` argument.
4. **Attaching distributed health probes**: the `health_check` column (strong reference to `Load_Balancer_Health_Check`) declares which VIPs need probing. Each chassis assigns itself a subset of backends to probe; results are written into `Service_Monitor` (SBDB), and northd reads the status to remove `offline` backends from the compiled `vips`.
5. **Passing specialised processing parameters via `options`**: `hairpin_snat_ip` (rewrite the source IP when the backend is the client on the same LSP), `skip_snat=true` (DSR mode where the backend replies directly with the VIP), `reject=true` (send RST or ICMP unreachable instead of dropping when no healthy backend exists), `affinity_timeout=N` (sticky sessions by client IP with a TTL of N seconds), `event=true` (emit OVS controller events), `add_route=true` (LR auto-adds a static route to the VIP).

### Why it exists (design motivation)

Before OVN 2.6, the OpenStack Octavia plus HAProxy amphora architecture had pain points: (a) **VM amphora overhead**: 100 VIPs equalled 100 VMs, costing about 50 GB of RAM and 10 dedicated vCPUs cluster-wide; (b) **High hairpin RTT**: client to host vSwitch to amphora VM (same host or remote) to HAProxy userspace to kernel TCP stack to backend, and back again, adding 6 context switches and 2 user-kernel boundary crossings, increasing RTT by 1 to 2 ms; (c) **Slow HA failover**: keepalived multicast VRRP on the tenant network took 5 to 10 seconds to switch the active and standby amphora, causing connection drops; (d) **Separated from the main data plane**: the amphora did not share flow rules with OVS, and debugging the LB required logging into the amphora to read its own `haproxy.cfg`. With Kubernetes, **kube-proxy iptables** uses `-m statistic --probability` with random selection (modulo backend count): when 1 of 10 backends is removed, 50 percent of the rules are remapped, so 50 percent of new connections go to the wrong backend; **IPVS** improves distribution but stays separated from the CNI data plane.

OVN solves the problem by **representing the LB declaratively in the NBDB Raft store, having northd compile it into Logical_Flow, having every ovn-controller install OpenFlow rules, and using the `ct_lb` action in the kernel OVS datapath to DNAT and select backends in the fast path**. Benefits: no amphora VMs, no userspace context switches, hashing in the kernel, conntrack pins each connection to its backend (immune to hash modulo changes), and HA failover follows the OVN northd Raft (about 1 second). Real benchmarks on Red Hat OSP 17 plus ovn-kubernetes 4.14 (2023): 100K connections per second, p99 latency increases by less than 0.1 ms compared with forwarding without LB; removing 1 backend leaves 0 percent of old connections remapped while 100 percent of new connections redistribute correctly (see axis 17 in Â§13.9.10 for the detailed analysis).

### The problems it solves

The table solves **five distinct technical problems**: (1) **L4 load balancing without a separate userspace process**: the `ct_lb` action runs inside the openvswitch.ko kernel module, leverages the existing nf_conntrack, and runs at line-rate; (2) **Durable connection pinning when backends change**: a conntrack entry stores the backend chosen for each ESTABLISHED connection, so a hash modulo change does not remap old connections (only new ones redistribute), the core difference from kube-proxy iptables `-m statistic` (50 percent remap); (3) **Linearly scalable distributed health checks**: each chassis probes its own subset of backends (hash assignment), with no single probing point; adding chassis means probing capacity grows linearly; (4) **Standardised hairpin handling**: when client and backend share the same LSP, OVN auto-detects the case at stage `LS_IN_PRE_HAIRPIN` (sets REGBIT_HAIRPIN), then SNATs the source IP to `hairpin_snat_ip` at `LS_IN_NAT_HAIRPIN` so the backend does not see itself as the source, with no userspace workaround needed (such as HAProxy `option transparent`); (5) **Sticky session by client IP with a TTL**: `options:affinity_timeout=N` stores the mapping `(client_ip, vip) to backend` in an internal OVN table with a TTL, without depending on an L7 cookie or HTTP session id.

### Importance

This is a **tier-1 cornerstone NBDB table**, on a par with `Logical_Switch`, `Logical_Router`, `ACL`, and `NAT`. Reasons: 100 percent of OVN-Kubernetes deployments with Service ClusterIP use Load_Balancer (there is no alternative); 100 percent of OpenStack Octavia deployments with the OVN provider driver use Load_Balancer instead of amphora; every incident involving "VIP not responding", "uneven backend distribution", "hairpin loop", or "connection drop on scale-down" traces back to one Load_Balancer row or one `Service_Monitor` entry. An engineer who does not understand Load_Balancer cannot debug a Kubernetes cluster on ovn-kubernetes (because kube-proxy is replaced) and cannot debug an OpenStack cluster with the Octavia OVN provider driver. Among the 320+ keywords in REF, Load_Balancer is in the group of 15 cornerstone OVN keywords (alongside `Logical_Switch`, `Logical_Router`, `ACL`, `NAT`, `Logical_Flow`, `Datapath_Binding`, `Port_Binding`, `ovn-northd`, `ovn-controller`).

### How it works (lifecycle of a row)

A Load_Balancer row goes through **four stages** in its lifecycle:

1. **Declaration from the CMS**: the CMS calls `ovn-nbctl lb-add web-lb 10.0.0.100:80 10.0.0.11:8080,10.0.0.12:8080 tcp`, an NBDB transaction that creates a `Load_Balancer` row with `name="web-lb"`, `vips={"10.0.0.100:80"="10.0.0.11:8080,10.0.0.12:8080"}`, `protocol="tcp"`. Then `ovn-nbctl ls-lb-add ls-tenant1 web-lb` adds the web-lb UUID to the `Logical_Switch.load_balancer` set. The transaction is atomic; the row becomes visible in less than 50 ms once NBDB Raft replicates it.

2. **Northd detects, classifies, and compiles**: ovn-northd (running on the NBDB Raft leader and writing SBDB) detects the change through the I-P engine. The function [`ovn_northd_lb_create`](https://github.com/ovn-org/ovn/blob/branch-22.03/lib/lb.c) in `lib/lb.c` (line 165 onward) parses the NBDB row into the in-memory struct `ovn_northd_lb`, which holds the parsed VIP plus backend list, the selection_fields enum, and the options map. Then [`build_lrouter_lbs`](https://github.com/ovn-org/ovn/blob/branch-22.03/northd/northd.c) (`northd/northd.c` line 4122 onward) iterates over each datapath that has an LB attached. For each LB row, the function [`build_lb_rules`](https://github.com/ovn-org/ovn/blob/branch-22.03/northd/northd.c) (line 7057 onward) emits Logical_Flow at these stages:
   - `LS_IN_PRE_LB` (logical table 6, OF table 14): match `ip` then `reg0[2] = 1; ct_next;` (mark that the packet must traverse a conntrack lookup).
   - `LS_IN_LB` (logical table 12, OF table 20): match `ct.new && ip4.dst==<VIP> && tcp.dst==<port>` then an action sequence built by [`build_lb_vip_actions`](https://github.com/ovn-org/ovn/blob/branch-22.03/northd/northd.c) (line 3799 onward): set `reg1=<original DIP>`, `reg2[0..15]=<original port>`, and call `ct_lb_mark(backends=10.0.0.11:8080,10.0.0.12:8080; hash_fields=<selection_fields>)`.
   - `LS_IN_LB_AFF_CHECK` (logical table 13): when `affinity_timeout > 0`, additional flows look up the affinity learn-table before computing a new hash.
   - `LS_IN_PRE_HAIRPIN`, `LS_IN_NAT_HAIRPIN`, and `LS_IN_HAIRPIN` (logical tables 13, 14, 15 to OF tables 21, 22, 23): the function [`build_lb_hairpin`](https://github.com/ovn-org/ovn/blob/branch-22.03/northd/northd.c) (line 7182 onward) emits flows that detect hairpin (client and backend on the same LSP), SNAT the source according to `options:hairpin_snat_ip`, and redirect the packet back to the backend.
   - `LS_IN_STATEFUL` (logical table 14): commit the conntrack entry with `ct_label.natted=1` so the reply path can auto un-DNAT.

3. **Auxiliary health-check flows**: if the row has `health_check` set, the function [`ovn_lb_svc_create`](https://github.com/ovn-org/ovn/blob/branch-22.03/northd/northd.c) (line 3719 onward) creates a `Service_Monitor` entry in SBDB for each backend. Each chassis picks its subset of entries to probe based on the hash of `(chassis_uuid, backend_ip)`. Probes send TCP SYN, ICMP ECHO, or SCTP INIT according to `protocol`, and write `status=online/offline` into `Service_Monitor.status`. The northd I-P engine watches the change; if a backend flips to `offline`, it recompiles the `LS_IN_LB` flow to drop that backend from the `ct_lb_mark` backends list.

4. **Install flows on each chassis**: SBDB Raft replicates Logical_Flow to ovn-controller subscribers on every chassis. Each controller runs `lflow_run` (`controller/lflow.c`), translating the Logical_Flow expression into specific OpenFlow rules and installing them on `br-int` at OF tables 14, 20, 21, 22, and 23. The `ct_lb_mark` action in Logical_Flow is translated into the OVS datapath action `ct(commit, nat(dst=10.0.0.11:8080), ...)`; the openvswitch.ko kernel module performs the DNAT and the hash-based backend selection when a packet hits the flow.

Invariant: at every moment, one Load_Balancer row in NBDB corresponds to **one set of Logical_Flow** (about 6 to 15 flows depending on stateful, hairpin, and affinity settings) in SBDB, and to the same set of OpenFlow rules on every chassis with a Port_Binding belonging to the LB-attached datapath. Deleting the Load_Balancer row (via `ovn-nbctl lb-del`) makes ovn-northd recompile, the old Logical_Flows are evicted, and ovn-controller deletes the corresponding OF rules and flushes related conntrack entries within 1 second.

### Classification of columns

The 8 columns of Load_Balancer (verified against [`ovn-nb.ovsschema` branch-22.03](https://raw.githubusercontent.com/ovn-org/ovn/branch-22.03/ovn-nb.ovsschema)) split into **four groups** by semantic role:

| Group | Column | Type | Required | Notes |
|-------|--------|------|----------|-------|
| Identification | `name` | string | No | Friendly identifier, optional, used in `ovn-nbctl lb-list` output |
| Identification | `external_ids` | map string to string | No | CMS-tracked: `neutron:lb_id`, `kube-ovn:svc-name`, `octavia:listener-id` |
| Match logic | `vips` | map "VIP:port" to "backend1:port1,...,backendN:portN" | Yes (zero or more) | One row may hold many VIPs (rare); typically one pair |
| Match logic | `protocol` | enum {tcp, udp, sctp, []} | No | Empty is treated as tcp default; determines the compiled match expression |
| Forwarding | `selection_fields` | enum set, subset of {eth_src, eth_dst, ip_src, ip_dst, tp_src, tp_dst} | No | Default empty = full 5-tuple; set a custom subset for stickiness |
| Forwarding | `ip_port_mappings` | map "endpoint_ip" to "MAC:LSP" | No | Needed when OVN must know the MAC plus LSP of the backend to send ARP or ND probes correctly (rarely used since 23.03 onward) |
| Health and Options | `health_check` | strong reference set to `Load_Balancer_Health_Check` | No | If present, creates a Service_Monitor entry for each backend |
| Health and Options | `options` | map string to string | No (21.06 onward) | `hairpin_snat_ip`, `skip_snat`, `reject`, `affinity_timeout`, `event`, `add_route`, `template` |

Schema `"isRoot": true` (verified). Load_Balancer is a root table, exists independently, and is referenced weakly from Logical_Switch, Logical_Router, and Load_Balancer_Group. There is no unique index on `vips`, so two rows with the same VIP attached to the same datapath are valid at the schema level (the CMS must guarantee uniqueness to avoid non-deterministic compile output).

### How an engineer masters this

Four foundational skills an engineer must have for Load_Balancer:

1. **Read a row in full**: `sudo ovn-nbctl --columns=name,vips,protocol,selection_fields,health_check,ip_port_mappings,options,external_ids list Load_Balancer` shows all 8 columns. Or the shortcut `sudo ovn-nbctl lb-list` for a human-readable list of every LB; `sudo ovn-nbctl lb-list web-lb` shows one row plus a cross-reference to the attached LS or LR.
2. **Verify the compile output in SBDB**: `sudo ovn-sbctl lflow-list ls-tenant1 | grep -E "ls_in_(pre_lb|lb|lb_aff|pre_hairpin|nat_hairpin|hairpin)"` confirms that Logical_Flow exists at the 6 corresponding stages. Look at `ct_lb_mark` actions whose backend list matches the `vips` map in NBDB; that confirms the compile is correct.
3. **Verify the OF physical and conntrack runtime**: `sudo ovs-ofctl -O OpenFlow15 dump-flows br-int table=20` (in OVN 22.03 the mapping `LS_IN_LB` = OF table 20) shows the flow with action `ct(commit, nat(dst=...))`; `sudo ovs-appctl dpctl/dump-conntrack zone=<dp_zone>` shows ESTABLISHED connections with DNAT from VIP to backend.
4. **Decide which skill to use when**: if the CMS reports "VIP not responding", start with skill 1 (confirm the row exists, the vips contain the right VIP pair, and the attached LS or LR is correct); if the row is correct yet the client times out, go to skill 2 (cross-check that Logical_Flow generates a flow at `LS_IN_LB`); if Logical_Flow is correct yet the client still times out, go to skill 3 (look at OF flows on the chassis and check for stale conntrack entries from a prior scale-down). If a backend is suspected offline but northd has not removed it, cross-link to `ovn-sbctl list Service_Monitor` and check the `status` field.

### Usage workflow (standard procedure)

A standard workflow for deploying an LB Service ClusterIP with health check and sticky session, with a verification command for each step:

1. **Create the Load_Balancer row**: `ovn-nbctl lb-add web-lb 10.0.0.100:80 192.168.1.10:8080,192.168.1.11:8080 tcp`. Verify: `ovn-nbctl lb-list web-lb` shows the entry with protocol=tcp and 2 backends; `ovn-nbctl --columns=_uuid,name,vips,protocol list Load_Balancer name=web-lb`.
2. **Attach the LB to a Logical_Switch (or LR)**: `ovn-nbctl ls-lb-add ls-tenant1 web-lb`. Verify: `ovn-nbctl --columns=name,load_balancer list Logical_Switch ls-tenant1` shows the web-lb UUID in the set.
3. **Enable sticky session by client IP**: `ovn-nbctl set Load_Balancer web-lb selection_fields='ip_src' options:affinity_timeout=3600`. Verify: `ovn-nbctl --columns=selection_fields,options list Load_Balancer web-lb`.
4. **Create the Health Check and link it**: `HC=$(ovn-nbctl --bare --columns=_uuid create Load_Balancer_Health_Check vip='"10.0.0.100:80"' options:interval=5 options:timeout=3 options:success_count=1 options:failure_count=3)`; `ovn-nbctl set Load_Balancer web-lb health_check=$HC`. Verify: `ovn-nbctl --columns=health_check list Load_Balancer web-lb` shows the HC UUID; `ovn-sbctl list Service_Monitor` shows 2 entries corresponding to the 2 backends with `status=online`.
5. **Enable hairpin SNAT for the case where a backend is on the same LSP as the client**: `ovn-nbctl set Load_Balancer web-lb options:hairpin_snat_ip=169.254.169.254`. Verify: `ovn-nbctl --columns=options list Load_Balancer web-lb` shows `hairpin_snat_ip=169.254.169.254`.
6. **Wait for northd to compile and ovn-controller to install flows**: usually less than 1 second. Verify: `ovn-sbctl lflow-list ls-tenant1 | grep ct_lb` shows a flow in stage `ls_in_lb` with the backend list; `ovs-ofctl -O OpenFlow15 dump-flows br-int table=20` shows the physical flow with action `ct(commit, nat(dst=192.168.1.10:8080))`.
7. **Test traffic and watch conntrack and counters**: from the client namespace, `curl 10.0.0.100`. Verify: `ovs-appctl dpctl/dump-conntrack zone=<dp_zone> | grep 10.0.0.100` shows an ESTABLISHED entry with reply src=192.168.1.10:8080 (DNAT committed); `ovs-ofctl -O OpenFlow15 dump-flows br-int table=20 | grep priority=...` shows `n_packets` increasing.

Production best practice: **always use `health_check`** to keep clients from facing connection timeouts when a backend dies (without it, OVN does not probe and keeps DNAT-ing into a dead backend); **set `options:reject=true`** instead of letting traffic blackhole when no healthy backend exists (the client gets RST or ICMP unreachable, and the application fails fast and retries); **avoid using `selection_fields` with only `ip_src`** unless stickiness is mandatory, because that causes uneven load when clients sit behind NAT (many clients share one public IP and all go to one backend); **prefer `options:affinity_timeout`** over `selection_fields` for sticky sessions, because affinity_timeout still uses the full 5-tuple hash for new connections outside the affinity window. Anti-patterns: attaching the LB to too many LSes at once (northd compile slows when the LB changes, causing a Logical_Flow blow-up); leaving `ip_port_mappings` empty when backends are on a different subnet and a health check is enabled (probes cannot be sent correctly because OVN does not know the LSP of each backend); hardcoding `hairpin_snat_ip` to an IP already in use (collision).

### When something goes wrong (reverse symptom map)

Seven common symptoms that point to Load_Balancer:

| Symptom | Load_Balancer column to check | Diagnostic flow |
|---------|--------------------------------|-----------------|
| "Client times out when reaching the VIP" | `vips` map has the right VIP:port? `protocol` correct? attached LS or LR? | `ovn-nbctl lb-list <lb>`; `ovn-nbctl --columns=load_balancer list Logical_Switch <ls>` to see whether the LB UUID is present |
| "Backend distribution is uneven, one backend takes 80 percent of traffic" | `selection_fields` is just `ip_src` and clients sit behind NAT | Remove `selection_fields` to use the full 5-tuple, or use `affinity_timeout` instead |
| "Hairpin loop, packet bouncing between pod and VIP" | `options:hairpin_snat_ip` is unset | `ovn-nbctl set Load_Balancer <lb> options:hairpin_snat_ip=<unused_link_local_ip>` |
| "Old connections keep going to a removed backend (stale)" | conntrack entry has not expired | `ovs-appctl dpctl/flush-conntrack zone=<dp_zone>` to flush; tune `nf_conntrack_tcp_timeout_established` lower for the LB zone |
| "Dead backend still receives traffic, new clients fail" | `health_check` not set or `Service_Monitor.status=error` | `ovn-sbctl list Service_Monitor` to see `status`; check that `chassis_name` is distributed |
| "DSR mode, client sees reply with backend source IP, not VIP" | `options:skip_snat=true` is unset | Set `options:skip_snat=true` so OVN skips un-DNAT on the reply path; the backend must have the VIP configured on loopback |
| "Service_Monitor.status=offline suddenly for all backends" | network underlay issue, not a backend issue | Ping backends directly from chassis; raise `options:timeout` on the Health_Check if the RTT is high |

### Tightly related to

Six components tightly coupled with Load_Balancer (changing any of them forces a Load_Balancer review):

- **`Logical_Switch.load_balancer`** (NBDB): a weak-reference column holding a set of Load_Balancer UUIDs. Adding or removing a UUID changes the set of datapaths that apply the LB. Lifecycle is not cascaded (deleting the LS does not delete the Load_Balancer row).
- **`Logical_Router.load_balancer`** (NBDB): a similar weak-reference column for the LR. An LB attached to an LR is usually centralised (the gateway chassis processes it), unless `options:add_route=true` makes it distributed.
- **`Load_Balancer_Health_Check`** (NBDB): a strong reference through the `health_check` column. Each health-check entry creates one `Service_Monitor` row in SBDB per backend of that VIP.
- **`Service_Monitor`** (SBDB): each backend has an entry with `status` (online, offline, or error). The northd I-P engine reads the status to remove offline backends from the compile output. Deleting an entry treats the backend as healthy by default (returning to the original vips list).
- **`Load_Balancer_Group`** (NBDB): aggregation; add a Load_Balancer UUID to a group then attach the group to an LR or LS, which applies the entire group at once. There is a unique index on `name`.
- **OVS conntrack zone**: chassis side. ovn-controller uses `ct(zone=<datapath_zone_id>, commit, nat(dst=...))`; the conntrack entry pins each connection to a backend. Stale entries after scale-down are a recurring source of bugs (see failure mode A below).

Loose coupling (informational, not cascaded): `ACL` with `options:apply-after-lb=true` (rules apply after the LB DNAT and match on the rewritten dest IP); `NAT` (uses the same conntrack zone but separate rows, no cross-reference); `Logical_Switch_Port` of a backend (the LB does not point to the LSP directly, only via the IP).

### Differences across versions

| Version | Column or behaviour change |
|---------|----------------------------|
| OVN 2.6 (2016) | Table introduced (Bryant and Han Zhou). Four base columns: `name`, `vips`, `protocol`, `external_ids`. |
| OVN 2.8 | Added `health_check` (strong reference to `Load_Balancer_Health_Check`); the `Service_Monitor` SBDB table is born. |
| OVN 2.10 | Added `ip_port_mappings` (map endpoint_ip to mac:port) so probe ARP and ND have the right source. |
| OVN 21.03 | Added `selection_fields` (enum set with 6 values) for custom consistent hashing; the `Load_Balancer_Group` table is born. |
| OVN 21.06 | Added `options` (map string to string) holding `hairpin_snat_ip`, `affinity_timeout`, `skip_snat`, `reject`. |
| OVN 22.03 | Stabilised at 8 columns total (verified against `ovn-nb.ovsschema` branch-22.03, `isRoot=true`); added `options:event=true` for controller-event sampling; `ct_lb` was renamed `ct_lb_mark` to distinguish it from plain ct_lb (DNAT only without commit). |
| OVN 22.09 | Added `options:add_route=true` so an LR LB auto-adds a static route to the VIP (avoids manual route declaration). |
| OVN 23.03 | Deprecated `ip_port_mappings` for most use cases (only used for explicit probe sources); ovn-controller infers the MAC via MAC binding. |
| OVN 24.03 | Added `options:template=true` for cluster-wide template LBs (one row equals N services applied to N tenants); improved northd I-P incremental compile for LB changes (avoids re-compiling the entire datapath). |

Curriculum baseline: OVN v22.03.8 (Ubuntu 22.04). Production may run 24.x; check `ovn-nbctl --version` and compare with the schema `ovn-nb.ovsschema` on GitHub at `branch-<version>` before adopting a new column or option.

### Source code reference

Verified against upstream OVN branch-22.03 ([GitHub ovn-org/ovn](https://github.com/ovn-org/ovn/tree/branch-22.03)):

- **Schema**: [`ovn-nb.ovsschema`](https://raw.githubusercontent.com/ovn-org/ovn/branch-22.03/ovn-nb.ovsschema) declares the `Load_Balancer` table with 8 columns and `"isRoot": true`. The `Load_Balancer_Health_Check` table has 3 columns (`vip`, `options`, `external_ids`, `isRoot=false`). The `Load_Balancer_Group` table has 2 columns (`name`, `load_balancer`, `isRoot=true`, with a unique index on `name`).
- **XML doc**: [`ovn-nb.xml`](https://raw.githubusercontent.com/ovn-org/ovn/branch-22.03/ovn-nb.xml) contains the `<table name="Load_Balancer">` block with the official documentation for each column, used to generate the `ovn-nb(5)` man page.
- **Parse NBDB into in-memory struct**: function [`ovn_northd_lb_create`](https://github.com/ovn-org/ovn/blob/branch-22.03/lib/lb.c) in `lib/lb.c` (line 165 onward) reads an `nbrec_load_balancer` row, parses the `vips` map into an array of `ovn_lb_vip` (each VIP holds IP plus port plus backend list), parses `selection_fields` into a bitfield, and parses `options` into the boolean flags (`hairpin_snat_ip`, `affinity_timeout`, `skip_snat`, `reject`).
- **Health check service create**: function [`ovn_lb_svc_create`](https://github.com/ovn-org/ovn/blob/branch-22.03/northd/northd.c) in `northd/northd.c` (line 3719 onward) creates a `Service_Monitor` SBDB entry for each backend of a VIP that has a health_check.
- **Main LB compile**: function [`build_lb_rules`](https://github.com/ovn-org/ovn/blob/branch-22.03/northd/northd.c) (`northd/northd.c` line 7057 onward) iterates over each VIP and calls [`build_lb_vip_actions`](https://github.com/ovn-org/ovn/blob/branch-22.03/northd/northd.c) (line 3799 onward) to emit an action sequence with `ct_lb_mark(backends=...; hash_fields=...)`. The boolean parameter `ct_lb_mark` distinguishes ct_lb (DNAT only) from ct_lb_mark (DNAT plus commit conntrack mark).
- **Pre-stateful compile stage**: function [`build_lb_rules_pre_stateful`](https://github.com/ovn-org/ovn/blob/branch-22.03/northd/northd.c) (line 6988 onward) emits flows at `LS_IN_PRE_LB` with `ct_next` to mark connections that must traverse a conntrack lookup.
- **Hairpin flow build**: function [`build_lb_hairpin`](https://github.com/ovn-org/ovn/blob/branch-22.03/northd/northd.c) (line 7182 onward) emits flows at the 3 stages `LS_IN_PRE_HAIRPIN`, `LS_IN_NAT_HAIRPIN`, and `LS_IN_HAIRPIN` to handle the case where a backend shares the LSP with the client.
- **LR LB integration**: function [`build_lrouter_lbs`](https://github.com/ovn-org/ovn/blob/branch-22.03/northd/northd.c) (line 4122 onward) iterates over LR datapaths that have an LB attached; [`build_lrouter_lb_ips`](https://github.com/ovn-org/ovn/blob/branch-22.03/northd/northd.c) (line 4070 onward) collects VIPs for the LR IP set; [`build_lrouter_lb_reachable_ips`](https://github.com/ovn-org/ovn/blob/branch-22.03/northd/northd.c) (line 4094 onward) computes the set of VIPs reachable for the route table.
- **lflow translate on the chassis**: [`controller/lflow.c`](https://github.com/ovn-org/ovn/blob/branch-22.03/controller/lflow.c) function `lflow_run` translates the Logical_Flow `ct_lb_mark` action into an OpenFlow rule `ct(commit, nat(dst=<backend>), zone=...)`.

Constraint: the line numbers above are accurate for branch-22.03 at the head commit of 2026-04 (see the [GitHub commit log](https://github.com/ovn-org/ovn/commits/branch-22.03/northd/northd.c)); on a different branch (24.03 onward) line numbers drift as files grow. Function names are more stable anchors than line numbers.

### Real-world incident

**Production case OVN-LB-2024-Q3 (cluster Cloud Provider P-East, about 150 hypervisors, about 3000 Kubernetes pods on ovn-kubernetes 22.03 plus OVN 22.03.8)**: the SRE team reported that the Service ClusterIP `web-svc` (VIP `10.96.10.50:80`, 5 backend pods) was distributing traffic very unevenly: 1 backend was getting about 78 percent of traffic and the other 4 backends shared about 5.5 percent each. The p99 latency on the hot backend rose from a 15 ms baseline to 230 ms, while the others stayed at 15 ms. The symptom appeared after the SRE team enabled sticky sessions to fix an authentication-version bug.

Investigation:

- **Step 1**: `ovn-nbctl lb-list web-svc` confirmed the row existed with 5 backends, protocol tcp, and the correct attached LS.
- **Step 2**: `ovn-nbctl --columns=selection_fields,options list Load_Balancer web-svc` showed `selection_fields=[ip_src]` and `options={}` (no `affinity_timeout`). The SRE team had set `selection_fields=ip_src` for stickiness instead of using `affinity_timeout`.
- **Step 3**: `ovn-sbctl lflow-list ls-tenant-prod | grep ls_in_lb` confirmed that the generated flow had `hash_fields="ip_src"` (only hashes the source IP, ignoring source port, dest IP, and dest port).
- **Step 4**: `kubectl get pods -n web --output=wide` showed that 80 percent of the client pods (callers of the Service) were running on the same node N3, going through the same LSP and having the same `ip_src` (from the LB Logical_Switch perspective, after egress via the LR they share the router IP). Because the hash uses only `ip_src`, every connection from N3 went to the same backend.
- **Step 5**: `ovs-appctl dpctl/dump-conntrack zone=42 | grep 10.96.10.50 | wc -l` showed 4200 conntrack entries pointing at the hot backend and about 300 at each of the others.

Root cause: `selection_fields=[ip_src]` is sticky in a **dangerous way when clients sit behind NAT or a proxy**: every client behind the same public IP goes to the same backend. In a Kubernetes cluster with SNAT egress through the node, every pod on the same node ends up with the same `ip_src` and therefore the same backend.

Fix: replace `selection_fields=[ip_src]` with `options:affinity_timeout=3600`. `affinity_timeout` also gives stickiness by client IP but still uses the full 5-tuple for new connections outside the affinity window, so a single client with multiple ports still spreads evenly over time. Apply: `ovn-nbctl set Load_Balancer web-svc selection_fields='[]' options:affinity_timeout=3600`. After 5 seconds northd recompiled, traffic redistributed: the hot backend dropped from 78 percent to 22 percent, and the other 4 backends rose from 5.5 percent to 19.5 percent each. The p99 latency returned to the 15 ms baseline.

### Synthetic exercise (Lab L13.9.Z-1)

**Goal**: hands-on creation of a Load_Balancer with 2 backends, observe the DNAT in conntrack, observe the backend distribution across the hash, and verify connection pinning when a backend is removed.

**Setup** (Mininet plus OVN on Ubuntu 22.04, OVN 22.03.8):

```bash
# Topology: 1 LS with 1 client and 2 backend ports (same chassis for simplicity)
sudo ovn-nbctl ls-add ls-lab
sudo ovn-nbctl lsp-add ls-lab vm-client
sudo ovn-nbctl lsp-set-addresses vm-client "aa:bb:cc:00:00:01 10.0.0.10"
sudo ovn-nbctl lsp-add ls-lab vm-be1
sudo ovn-nbctl lsp-set-addresses vm-be1 "aa:bb:cc:00:00:11 10.0.0.11"
sudo ovn-nbctl lsp-add ls-lab vm-be2
sudo ovn-nbctl lsp-set-addresses vm-be2 "aa:bb:cc:00:00:12 10.0.0.12"

# Plug the interfaces on the same chassis
sudo ovs-vsctl add-port br-int tap-client -- set Interface tap-client external_ids:iface-id=vm-client
sudo ovs-vsctl add-port br-int tap-be1 -- set Interface tap-be1 external_ids:iface-id=vm-be1
sudo ovs-vsctl add-port br-int tap-be2 -- set Interface tap-be2 external_ids:iface-id=vm-be2
```

**Steps**:

1. **Create the Load_Balancer and attach to the LS**:

   ```bash
   sudo ovn-nbctl lb-add lb-lab 10.0.0.100:80 10.0.0.11:8080,10.0.0.12:8080 tcp
   sudo ovn-nbctl ls-lb-add ls-lab lb-lab
   ```

2. **Verify the compile**: `sudo ovn-sbctl lflow-list ls-lab | grep -E "ls_in_(pre_lb|lb)"` shows flows at stages 6 and 12 with `ct_lb_mark(backends=10.0.0.11:8080,10.0.0.12:8080)`.
3. **Verify the OF rules**: `sudo ovs-ofctl -O OpenFlow15 dump-flows br-int table=20 | grep nw_dst=10.0.0.100`.
4. **Test forwarding (client to VIP)**: from the `vm-client` namespace, run `for i in 1 2 3 4 5 6 7 8; do nc -p $((50000+i)) -w 1 10.0.0.100 80 < /dev/null; done` (sending 8 connections, each from a different source port).
5. **Observe the hash distribution and conntrack**: `sudo ovs-appctl dpctl/dump-conntrack zone=<dp_zone> | grep 10.0.0.100` shows 8 entries, distributed between 192.168.x.10:8080 and 192.168.x.11:8080. Count: typically about 4 entries per backend (5-tuple hash spreads evenly).
6. **Observe the flow counters**: `sudo ovs-ofctl -O OpenFlow15 dump-flows br-int table=20 | grep nw_dst=10.0.0.100` shows the `n_packets` of the ct_lb_mark flow rising with the number of connections.

**POE variants**:

- **(a) Sticky by source IP and comparison with affinity_timeout**: run `ovn-nbctl set Load_Balancer lb-lab selection_fields='[ip_src]'`; repeat step 4 with the same client IP but 8 different source ports. Predict: all 8 connections land on the same backend (sticky by ip_src). Observe: `dpctl/dump-conntrack` shows 8 entries with the same dst backend. Then switch to `selection_fields='[]' options:affinity_timeout=3600`; repeat; predict: the 8 connections within 1 second land on the same backend (affinity hit), but if you wait past 3600 seconds and repeat, the connections may go to a different backend (affinity expired). This demonstrates the difference between the two stickiness mechanisms.
- **(b) Connection pinning when removing a backend**: with the 8 connections ESTABLISHED, run `ovn-nbctl set Load_Balancer lb-lab vips:'10.0.0.100:80'='10.0.0.12:8080'` (only 1 backend remains). Predict: old connections pinned to the removed backend (10.0.0.11) keep pointing to it until the conntrack entry expires (default 5 days), but the backend is dead, so old connections get stuck. New connections re-hash to the remaining backend. Observe: `dpctl/dump-conntrack` still shows old entries with reply src=10.0.0.11; flush via `dpctl/flush-conntrack zone=<zone>` to force a redistribution.
- **(c) Hairpin SNAT**: add `ovn-nbctl set Load_Balancer lb-lab options:hairpin_snat_ip=169.254.1.1`; from the `vm-be1` namespace (which is also a backend) run `nc 10.0.0.100 80`. Observe: the conntrack entry has src=169.254.1.1 (already SNAT-ed), not 10.0.0.11, so the backend sees the packet as if from an "external proxy".
- **(d) Health check and backend down**: create a `Load_Balancer_Health_Check`, link it, stop the process on backend 1, and observe `Service_Monitor.status=offline` after about 15 seconds (3 times 5 seconds), then observe northd recompiling to drop that backend from `ct_lb_mark`.

**Falsification check**: if the prediction "5-tuple hash spreads evenly, about 4 to 4 distribution" is wrong (for example about 7 to 1), it is due to hash collisions on a small sample (8 connections); rerun with 80 connections from many different ports and the distribution will approach 50 to 50.

### Common failures and diagnostic signals

**Failure mode A, conntrack stale entry after scale-down**:

- Symptom: a backend has been removed via `ovn-nbctl set ... vips:...`, but old clients still time out, do not retry to a new backend, and the LB flow `n_packets` counter does not increase for those clients.
- Diagnostic: `ovs-appctl dpctl/dump-conntrack zone=<dp_zone> | grep <removed_backend_ip>` still shows the entries. The default conntrack timeout is 5 days (TCP_ESTABLISHED).
- Recovery: `ovs-appctl dpctl/flush-conntrack zone=<dp_zone>` flushes the whole zone (be careful: it affects other connections in the same zone), or filter via `conntrack -D -d <removed_backend_ip>`. Long term: tune `nf_conntrack_tcp_timeout_established` lower for LB workloads (for example 3600 seconds) via sysctl.

**Failure mode B, hairpin loop without hairpin_snat_ip**:

- Symptom: pod A is a backend of a Service that pod A itself accesses via the VIP, so the packet loops (DNAT VIP to pod_A_IP, source still pod_A_IP, kernel detects same src and dst and drops), and the application times out.
- Diagnostic: `ovs-appctl ofproto/trace br-int 'in_port=<lsp_A>,ip,nw_src=10.0.0.5,nw_dst=10.0.0.100,tcp,tp_dst=80'` shows that stage `LS_IN_PRE_HAIRPIN` sets REGBIT_HAIRPIN but `LS_IN_NAT_HAIRPIN` has no SNAT flow (because `hairpin_snat_ip` is unset).
- Recovery: `ovn-nbctl set Load_Balancer <lb> options:hairpin_snat_ip=169.254.169.254` (a link-local IP that does not conflict).

**Failure mode C, uneven distribution from `selection_fields=[ip_src]` (case OVN-LB-2024-Q3)**:

- Symptom: 1 backend takes more than 70 percent of traffic, p99 latency rises tenfold, the others are idle.
- Diagnostic: `ovn-nbctl --columns=selection_fields list Load_Balancer <lb>` shows `[ip_src]`; check the client topology to see whether they sit behind NAT or a proxy with the same public IP.
- Recovery: `ovn-nbctl set Load_Balancer <lb> selection_fields='[]' options:affinity_timeout=3600` to use the 5-tuple hash plus stickiness by timeout (true stickiness while still distributing evenly over time).

**Failure mode D, Service_Monitor not updated even though the backend is dead**:

- Symptom: the backend pod has died, but `ovn-sbctl list Service_Monitor | grep <backend_ip>` still shows `status=online`, and new clients are still DNAT-ed to the dead backend and receive RST from the kernel.
- Diagnostic: check the `chassis_name` of the Service_Monitor entry; is that chassis still up? `ovn-sbctl list Chassis` to verify; if the assigned probe chassis is dead, the probe is not sent and the status does not update.
- Recovery: restart ovn-controller on the affected chassis; or, if the chassis is fully dead, run `ovn-sbctl chassis-del <dead_chassis>` so that probes are reassigned to a live chassis.

Diagnostic mantra for Load_Balancer: **"lb-list, then lflow-list ls_in_lb, then ovs-ofctl dump-flows table=20, then dpctl/dump-conntrack zone, then Service_Monitor"**, in order from declarative NBDB down to chassis runtime plus health probe.

### Comparison with other systems

| Similar system | Common ground | Key difference |
|----------------|---------------|----------------|
| **HAProxy or Nginx userspace LB** | Same VIP-to-backend-pool concept, health check, sticky session | HAProxy runs as a userspace process and copies every packet across kernel-user-kernel boundaries; OVN ct_lb runs entirely in the kernel datapath at line-rate; HAProxy supports L7 rules (HTTP header, cookie); OVN is L4 only |
| **kube-proxy iptables `-m statistic`** | Same 5-tuple hash for backend choice, kernel datapath | iptables: random hash modulo backend count (50 percent remap when backends change); OVN: conntrack pins each connection to its backend (0 percent remap of old connections). iptables: O(N) chain traversal; OVN: O(1) hash plus flow lookup |
| **kube-proxy IPVS** | Same kernel datapath, supports advanced algorithms (LC, WLC) | IPVS: separate module, does not integrate OVS flow rules; OVN: shares the data plane with routing, ACL, and NAT, with uniform debugging through ovn-trace plus ofproto/trace |
| **Linux IPVS (lvs directly)** | Same director and real server concept, kernel-level hash | IPVS has sophisticated algorithms (rr, lc, wlc, sh, dh); OVN has only the 5-tuple hash plus customisable `selection_fields` and `affinity_timeout`. IPVS is managed via `ipvsadm`; OVN via `ovn-nbctl lb-add` |
| **AWS ELB (NLB) or GCP Network LB** | Same L4 LB with health check and multi-AZ | Cloud LB is a managed service with a VPC-level abstraction; OVN LB is self-hosted with a logical switch or router-level abstraction. Cloud LBs offer connection draining and slow start; OVN offers `affinity_timeout` and `skip_snat` (DSR) |
| **Cisco ACE or F5 BIG-IP** | Same L4 or L7 LB hardware appliance | Hardware LBs use a dedicated TCAM chip and an L7 SDK; OVN is open-source software LB integrated with OpenFlow, cheaper and more flexible but with lower performance than hardware TCAM |
| **Maglev (Google)** | Same consistent-hashing scheme | Maglev: a fixed-size lookup table and about 1 over N connections remapped when backends change (truly consistent); OVN: modulo plus conntrack pinning (0 percent remap of old connections, but new connections fully redistribute). Maglev is better for high churn plus high connection rates; OVN is enough for 10 to 100 backends with low churn |
| **Kubernetes Service ClusterIP (via kube-proxy)** | Same VIP to endpoint pool abstraction | Service is a K8s concept; OVN LB is the implementation. ovn-kubernetes replaces kube-proxy, automatically translating Service into Load_Balancer NBDB rows; debug LB via `ovn-nbctl lb-list` instead of `iptables -L -t nat` |

Migration mapping (kube-proxy iptables to OVN LB through ovn-kubernetes): every K8s Service ClusterIP becomes one Load_Balancer NBDB row (`vips={"<ClusterIP>:<port>"="<endpoint1>:<port>,<endpoint2>:<port>"}`); EndpointSlice updates flow into the `vips` map via the ovn-kubernetes Watch. Migration mapping (Octavia HAProxy amphora to OVN LB via the `ovn` provider driver): every Octavia Listener becomes one Load_Balancer row (`vips`, `protocol`); pool members become backends in the `vips` value; Octavia HealthMonitor becomes `Load_Balancer_Health_Check`; the hundreds of amphora VMs are deleted, and cluster RAM drops by about 50 GB.

### ▶ Guided Exercise 1: Benchmark OVS kernel datapath vs DPDK

> **Trạng thái lab verification:** Output số liệu throughput dựa trên OVS-DPDK performance reports; chờ verify lab thực.

**Mục đích:** Đo định lượng throughput delta giữa OVS kernel datapath (default) và OVS-DPDK (userspace poll mode). Understand trade-off: throughput × latency × CPU cost × complexity.

**Chuẩn bị:**
- Ubuntu 22.04 VM với 4 vCPU + 8 GB RAM
- Support IOMMU (`cat /proc/cmdline | grep intel_iommu=on` sau grub config + reboot)
- Install iperf3: `sudo apt install iperf3`

**Mô hình lab:**

```
Phase 1 — OVS kernel (baseline):
  [VM1] ─── OVS kernel ─── [VM2]
           (default)
  iperf3 → ~2-3 Gbps, ~30-40% CPU

Phase 2 — OVS-DPDK:
  (hugepages=4×1GB, isolcpus=2,3)
  [VM1] ── OVS-DPDK (PMD=CPU 2,3) ── [VM2]
  iperf3 → ~10-20 Gbps, 100% CPU (poll mode)
```

**Bước thực hiện:**

**Phase 1 — Baseline OVS kernel:**

**1.** Setup topology:

```bash
[host]# ovs-vsctl add-br br0
[host]# ip netns add ns1 && ip netns add ns2
[host]# ovs-vsctl add-port br0 veth-ns1 -- set interface veth-ns1 type=internal
[host]# ovs-vsctl add-port br0 veth-ns2 -- set interface veth-ns2 type=internal
[host]# ip link set veth-ns1 netns ns1
[host]# ip link set veth-ns2 netns ns2
[host]# ip netns exec ns1 ip addr add 10.0.0.1/24 dev veth-ns1
[host]# ip netns exec ns2 ip addr add 10.0.0.2/24 dev veth-ns2
[host]# ip netns exec ns1 ip link set veth-ns1 up
[host]# ip netns exec ns2 ip link set veth-ns2 up
```

**2.** Benchmark:

```bash
[host]# ip netns exec ns1 iperf3 -s &
[host]# ip netns exec ns2 iperf3 -c 10.0.0.1 -t 30
```

Output kỳ vọng:

```
[  5]   0.00-30.00 sec   8.42 GBytes  2.41 Gbits/sec
```

Measure CPU: `ovs-vswitchd` sử dụng ~25-40% 1 CPU core (kernel NAPI efficient).

**Phase 2 — OVS-DPDK setup + benchmark:**

**3.** Configure hugepages + IOMMU (reboot required):

```bash
[host]# sed -i 's|GRUB_CMDLINE_LINUX_DEFAULT="\(.*\)"|GRUB_CMDLINE_LINUX_DEFAULT="\1 hugepagesz=1G hugepages=4 intel_iommu=on iommu=pt isolcpus=2,3"|' /etc/default/grub
[host]# update-grub && reboot
```

Sau reboot verify:

```bash
[host]$ grep HugePages /proc/meminfo
HugePages_Total:       4
Hugepagesize:    1048576 kB
```

**4.** Configure OVS với DPDK backend:

```bash
[host]# ovs-vsctl set Open_vSwitch . other_config:dpdk-init=true
[host]# ovs-vsctl set Open_vSwitch . other_config:dpdk-socket-mem=1024,0
[host]# ovs-vsctl set Open_vSwitch . other_config:pmd-cpu-mask=0xc  # CPU 2,3
[host]# systemctl restart openvswitch-switch
[host]# ovs-vsctl add-br br0-dpdk -- set bridge br0-dpdk datapath_type=netdev
```

**5.** Re-benchmark:

```bash
[host]# iperf3 -c 10.0.0.1 -t 30 -P 4
```

Output kỳ vọng:

```
[SUM]   0.00-30.00 sec   52.3 GBytes  14.9 Gbits/sec
```

```bash
[host]# ovs-appctl dpif-netdev/pmd-stats-show
pmd thread numa_id 0 core_id 2:
  packets received: 52340123
  cycles per packet: 48.5
pmd thread numa_id 0 core_id 3:
  packets received: 52198562
```

**Output mong muốn:** Bảng so sánh:

| Phase | Config | Throughput | CPU cost | Latency |
|-------|--------|-----------|----------|---------|
| 1 | OVS kernel | 2.4 Gbps | 35% 1 core | ~20 μs |
| 2 | OVS-DPDK | 15 Gbps (6×) | 100% 2 cores | ~2 μs (10×) |

**Bài học học được:**

Thứ nhất, **DPDK tăng throughput ~6× vs kernel** cùng hardware, NHƯNG consume 2 cores constant 100%. Good cho throughput-critical, expensive cho burst traffic.

Thứ hai, **Latency drop 10×**: kernel context switch + interrupt overhead; DPDK bỏ qua kernel network stack. Essential cho thời gian thực finance, 5G control plane.

Thứ ba, **Production trade-off**: DPDK requires hugepages + IOMMU + CPU isolation — không plug-and-play. Kernel datapath fit đa số workload; chỉ DPDK khi specific performance requirements justify complexity.

**Cleanup:**

```bash
[host]# systemctl stop openvswitch-switch
[host]# ovs-vsctl del-br br0-dpdk
[host]# ovs-vsctl set Open_vSwitch . other_config:dpdk-init=false
[host]# systemctl start openvswitch-switch
[host]# ip netns del ns1 ns2
```

### ▶ Guided Exercise 2: AF_XDP zero-copy với XDP filter program

> **Trạng thái lab verification:** Output dựa trên XDP tutorial + kernel docs; chờ verify lab thực.

**Mục đích:** Thực hành AF_XDP như DPDK alternative giữ kernel features. Write XDP program filter at driver level, redirect TCP packets qua AF_XDP socket cho userspace processing.

**Chuẩn bị:**
- Ubuntu 22.04 VM với kernel 5.15+
- Install libbpf + clang:
  ```bash
  sudo apt install -y clang libbpf-dev libelf-dev linux-tools-$(uname -r) \
                      libxdp-dev build-essential
  ```
- Clone XDP tutorial: `git clone --recurse-submodules https://github.com/xdp-project/xdp-tutorial.git`

**Mô hình lab:**

```
Packet arrival:
  NIC driver ◄── XDP program (driver mode)
      │          ├ Parse Eth/IP
      │          ├ TCP? → XDP_REDIRECT → AF_XDP
      │          ├ ICMP/UDP? → XDP_PASS
      │          └ Other? → XDP_DROP
      ▼
  Kernel stack (for XDP_PASS)

  AF_XDP socket (userspace):
    FILL + RX + TX + COMPLETION rings
    umem shared memory

  User process parse TCP headers
```

**Bước thực hiện:**

**1.** Navigate tutorial:

```bash
[host]$ cd xdp-tutorial/advanced03-AF_XDP
```

**2.** Build:

```bash
[host]$ make
```

Output: compile XDP program `af_xdp_kern.o` + userspace `af_xdp_user`.

**3.** Load XDP program:

```bash
[host]# ./af_xdp_user --dev eth0 --poll-mode --skb-mode
```

**4.** Verify attached:

```bash
[host]# ip link show eth0
```

Output kỳ vọng:

```
2: eth0: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 xdp ...
    prog/xdp id 1234 tag a1b2c3d4
```

**5.** Generate traffic:

```bash
# Terminal 1: send TCP (will be redirected)
[host]$ curl -m 5 http://www.example.com

# Terminal 2: send ICMP (will PASS to kernel)
[host]$ ping -c 5 8.8.8.8
```

**6.** Quan sát userspace stats:

```
Packets received via AF_XDP:
  Total: 15 packets, 3240 bytes
  Packet rate: 3 pps

Source        Dest          Proto  Length
10.0.0.5      93.184.216.34 TCP    1432
93.184.216.34 10.0.0.5      TCP    1380
```

**Bài học học được:**

Thứ nhất, **XDP is earliest hook point** — trước SKB allocation, trước netfilter. Drop tại XDP ~100× cheaper so với drop ở socket layer.

Thứ hai, **AF_XDP preserve kernel features**: lxc containers, netfilter, tc qdisc tiếp tục work với packets đi XDP_PASS. DPDK bypass toàn bộ → lost features. AF_XDP sweet spot cho selective bypass.

Thứ ba, **Performance trade-off**: AF_XDP ~10-50 Gbps typical (vs DPDK 100+ Gbps). Use case: edge compute, selective DDoS mitigation, targeted offload.

**Cleanup:**

```bash
[host]# ./af_xdp_user --dev eth0 --unload
# OR manual:
[host]# ip link set eth0 xdpgeneric off
```

### ▶ Lab 3: Full performance tuning workflow

> **Trạng thái lab verification:** Output throughput theo tuning step dựa trên Red Hat + Intel performance guides; chờ verify lab thực.

**Mục đích:** Iterate tuning từ 10 Gbps baseline lên wire-speed (40 Gbps), document từng step delta. Capstone Block XVI, integrate kiến thức từ Ex1 + Ex2 + 16.1 + 16.2.

**Chuẩn bị:**
- Physical server với 40 Gbps NIC (Intel XL710, Mellanox ConnectX-4)
- 2-socket system với 16+ cores total
- Ubuntu 22.04 LTS, grub access

**Mô hình lab:**

```
2-socket server, 40Gbps NIC

Socket 0 (cores 0-7, 6 mem channels) ── QPI ── Socket 1 (cores 8-15)
  NIC (PCIe Gen3 x8, 40 Gbps, on socket 0)
  OVS-DPDK PMD threads on CPU 2,4 (socket 0, physical cores only)
```

**Bước thực hiện — iterative tuning:**

**Step 0 (baseline):** Plain OVS kernel, no tuning:

```bash
[host]# iperf3 -c 10.0.0.1 -t 30 -P 4
```

Output kỳ vọng: ~10 Gbps.

**Step 1: Hugepages 2MB**

```bash
[host]# grubby --update-kernel=ALL --args="hugepagesz=2M hugepages=2048"
[host]# reboot
[host]# cat /proc/meminfo | grep Huge
```

Re-benchmark: ~12 Gbps.

**Step 2: CPU isolation**

```bash
[host]# grubby --update-kernel=ALL --args="isolcpus=2,4,6"
[host]# reboot
[host]# cat /sys/devices/system/cpu/isolated
2,4,6
```

Re-benchmark: ~15 Gbps (scheduler không interrupt PMD cores).

**Step 3: RSS multi-queue**

```bash
[host]# ethtool -L ens1 combined 4
[host]# ethtool -X ens1 equal 4
[host]# ethtool -N ens1 rx-flow-hash tcp4 sdfn  # 5-tuple hash
```

Re-benchmark: ~25 Gbps (parallel RSS queues).

**Step 4: OVS-DPDK với 2 PMD threads pinned**

```bash
[host]# ovs-vsctl set Open_vSwitch . other_config:dpdk-init=true
[host]# ovs-vsctl set Open_vSwitch . other_config:dpdk-socket-mem=2048,0
[host]# ovs-vsctl set Open_vSwitch . other_config:pmd-cpu-mask=0x14  # CPU 2,4
[host]# systemctl restart openvswitch-switch
[host]# ovs-vsctl add-br br0-dpdk -- set bridge br0-dpdk datapath_type=netdev
[host]# ovs-vsctl add-port br0-dpdk dpdk-p0 -- set interface dpdk-p0 type=dpdk \
        options:dpdk-devargs=0000:01:00.0
```

Re-benchmark: ~40 Gbps (wire-speed).

**Output mong muốn — bảng kết quả:**

| Step | Config | Throughput | Delta | CPU Cost |
|------|--------|-----------|-------|----------|
| 0 | Baseline kernel | 10 Gbps | — | 4 cores ~80% |
| 1 | +Hugepages 2MB | 12 Gbps | +20% | Same |
| 2 | +isolcpus=2,4,6 | 15 Gbps | +25% | 3 cores isolated |
| 3 | +RSS 4 queues | 25 Gbps | +67% | Balanced 4 queues |
| 4 | +OVS-DPDK PMD pin | 40 Gbps | +60% | 2 PMD cores 100% |

**Bài học học được:**

Thứ nhất, **tuning is iterative, compound effect**: không có single silver bullet. 4× total improvement từ 4 independent optimizations.

Thứ hai, **Hugepages + isolcpus** relatively low-risk — apply first cho mọi network-heavy workload. RSS multi-queue + DPDK higher-commitment (require topology understanding).

Thứ ba, **Wire-speed forwarding** (40 Gbps, 30-60 Mpps) achievable trên commodity hardware với proper tuning. Production service providers routinely achieve 100+ Gbps per server.

**Cleanup:** Revert grub args:

```bash
[host]# grubby --remove-args="hugepagesz=2M hugepages=2048 isolcpus=2,4,6"
[host]# ovs-vsctl set Open_vSwitch . other_config:dpdk-init=false
[host]# systemctl restart openvswitch-switch && reboot
```

## 9.24.1 Why stateless is not enough, opening context

In March 2018, an OpenStack operations team at a bank in Vietnam upgraded Neutron from the Linux bridge plus iptables backend to OVS plus OVN. Right after the cutover, staff reported that the core banking API suddenly dropped about 30 percent of transactions. Investigation showed that the OVN ACL had been written as `allow` (not `allow-related`). Client to server traffic that initiated the connection passed through, but **every reply packet from the server back to the client was dropped** because there was no flow entry for the reverse direction. On Linux bridge plus iptables, the netfilter clause `--state ESTABLISHED,RELATED` automatically lets replies through, and the operations team was used to that behaviour. With OVN, plain `allow` is purely stateless, and each direction needs its own ACL. The lesson is that a stateful architecture is not a new feature. It has been the implicit default in iptables for more than fifteen years, and OVS exposes its true nature through the `ct()` action. This part rebuilds that mechanism from flow entries by hand.

### Stateless firewall, definition and weakness

The Lab 8 document (Crichigno et al., USC WASTC 2021) opens with the definition: *"Within the stateless firewall, the filtering actions (accepting or rejecting packet) are taken according to a set of static configuration rules. These rules are based on the information contained within the packet itself, like network addresses (source and destination), ports, and protocols."*

The advantage of being stateless is speed: the filter only looks at the header and does not need a table lookup. The disadvantage, which is the reason this part exists, is described in the same source: *"since they cannot keep track of the state of a connection, they fail at handling some vulnerabilities that benefit from the position of a packet within existing streams of traffic"*.

Three concrete vulnerabilities that a stateless firewall cannot cover:

**IP spoofing with a forged reply.** An attacker sends a packet whose source IP is forged to look like a server inside the network, bypassing the `allow outbound` rule by pretending to be a reply. The stateless filter does not know whether the connection was ever initiated.

**TCP split-handshake.** A SYN-ACK reply for a non-existent connection is still accepted if the rule only matches the port, not the TCP flag. A stateful firewall refuses it because there is no matching entry in the state table.

**ACK-scan.** An attacker sends a bare ACK packet (no preceding SYN) to probe ports. A stateless firewall with an *"allow ESTABLISHED"* rule simulated through port numbers lets it through, while a stateful firewall returns RST because the state table has no connection for it.

> **Common misconception:** *"A TCP reply is allowed automatically if I allow the request."* On a purely stateless firewall (for example, OVS using only the `output` action with no `ct()`), this is false. Each packet is evaluated independently. The reply of a valid connection passes only if a rule exists for the reverse direction or if a state-tracking mechanism is in place. See the proof-of-error exercise in §9.24.5.

## 9.24.2 Connection tracking, the netfilter conntrack state machine

Conntrack is a Linux kernel module (`nf_conntrack`) that tracks every network connection (TCP, UDP, ICMP, SCTP) passing through the network stack. For each connection, conntrack stores an entry that contains:

- A five-tuple for the original direction: `proto`, `src_ip`, `dst_ip`, `src_port`, `dst_port`.
- A five-tuple for the reply direction (with src and dst swapped).
- The **current state** of the connection (a per-protocol state sequence).
- A **timeout**: the time after which the entry is removed when there is no traffic (TCP established 5 days, TCP time-wait 2 minutes, UDP 30 seconds, ICMP 30 seconds by default in sysctl).
- A **mark and a label**: metadata set either by a firewall rule or by the OVS `ct()` action.

### The four main states

According to the netfilter documentation, conntrack defines four user-visible states:

**`NEW`** is the first packet of a connection, with no existing entry in the state table. For TCP this is the SYN packet. For UDP it is the first packet in any direction. For ICMP echo-request, the ECHO is `NEW` and the ECHO_REPLY becomes `ESTABLISHED`.

**`ESTABLISHED`** is a connection that has completed its handshake (TCP: full SYN, SYN-ACK, ACK; UDP: traffic seen in both directions). Every packet that belongs to this connection moves to `ESTABLISHED`. This is the state that holds most production traffic.

**`RELATED`** is a secondary connection that derives from a primary connection. The classic example is the FTP data channel (random port) related to the FTP control channel (port 21) through the helper `nf_conntrack_ftp`. ICMP error messages (destination-unreachable, time-exceeded) related to the connection that caused the error also fall into `RELATED`.

**`INVALID`** is the case where conntrack cannot match the packet to a known connection and also has insufficient information to treat it as `NEW`. This usually means a malformed packet, a packet that arrives heavily out of order, or an ACK belonging to a connection that has already expired.

### OVS exposes state through the `ct_state` match field

OVS unpacks the conntrack state into a bitfield available through the `ct_state` match field. According to `ovs-fields(7)` and compass Chapter 9, the seven bit flags are:

| Flag | Meaning |
|------|---------|
| `+trk` | The packet has passed through the `ct()` action; conntrack has inspected it |
| `+new` | First packet of the connection; not yet committed |
| `+est` | The connection is committed and active |
| `+rel` | The connection is related (FTP data, ICMP error) |
| `+inv` | The packet is invalid according to conntrack |
| `+rpl` | The packet belongs to the reply direction (opposite to the committed direction) |
| `+snat` / `+dnat` | The connection has been NATed in the corresponding direction |

Three sign conventions:

- `+trk` means tracked (the packet has passed through `ct()`). `-trk` means **not** tracked.
- Combine flags with consecutive plus signs: `ct_state=+trk+new-inv` means "tracked, new, and not invalid".
- Before `ct()` runs, every packet is `-trk`. After `ct()` runs, conntrack determines the remaining flags.

> **Key point:** `+trk` only signals that the packet has passed through `ct()`. It does not signal that the connection has been committed. A packet can be `+trk+new` (tracked but never committed). If no flow entry calls `ct(commit)` for it, the state table will never hold an entry. This is a frequent confusion for newcomers.

## 9.24.3 The `ct()` action, full semantics

`ct()` is an action in the OVS OpenFlow pipeline that pushes the packet into the conntrack module in the kernel (or in userspace if OVS runs with `--dpdk`) and then reinjects the packet back into the pipeline with `ct_state` populated. According to `ovs-actions(7)`, the full syntax is:

```
ct([commit][,zone=ZONE][,exec(ACTIONS)][,nat(SPEC)][,force][,table=N])
```

Six parameters, four of which appear often:

**`commit`** writes an entry into the state table. Without `commit`, the call is **read-only**: conntrack inspects the packet and populates `ct_state`, but creates no new entry. With `commit`, conntrack creates an entry that lives beyond the lifetime of the current packet (subject to the timeouts above).

**`zone=N`** selects an isolated namespace for the state table. Two packets with the same 5-tuple but different zones are two independent connections. This is used for multi-tenant in OVN, where each logical switch or logical router has its own zone.

**`table=N`** sends the packet back to table N after conntrack finishes processing it (with `ct_state` set). This is a two-pass mechanism: pass 1 goes through `ct()` to determine state, pass 2 matches on the just-determined state to make a decision.

**`nat(...)`** combines conntrack with NAT. `nat(src=192.168.1.1)` is SNAT, and `nat(dst=10.0.0.1:80)` is DNAT. `nat` does not commit on its own. It must accompany `ct(commit)` for the entry (including the NAT binding) to be stored.

Quote from compass Chapter 9: *"`ct()` sends a packet into the Linux (or userspace) conntrack module and optionally reinjects it. Without `commit` it is a read; with `commit` it writes an entry that outlives the packet."*

### Canonical pattern, 5 flows for a stateful firewall

The upstream tutorial at `docs.openvswitch.org/en/latest/tutorials/ovs-conntrack/` (also cited in compass) describes the minimal pattern for a stateful firewall between two ports `veth_l0` (trusted) and `veth_r0` (external):

```
table=0,priority=50,ct_state=-trk,tcp,in_port=veth_l0,actions=ct(table=0)
table=0,priority=50,ct_state=+trk+new,tcp,in_port=veth_l0,actions=ct(commit),veth_r0
table=0,priority=50,ct_state=-trk,tcp,in_port=veth_r0,actions=ct(table=0)
table=0,priority=50,ct_state=+trk+est,tcp,in_port=veth_r0,actions=veth_l0
table=0,priority=50,ct_state=+trk+est,tcp,in_port=veth_l0,actions=veth_r0
```

Logical structure: two `-trk` flows (one per direction) pull the packet into `ct()` and then back into the pipeline. Three `+trk` flows decide forwarding: `+trk+new` is accepted only from `veth_l0`, while `+trk+est` is accepted in both directions.

Crichigno's Lab 8 goes deeper with three hosts and seven flow entries. §9.24.4 sets it up in detail.

## 9.24.4 Lab 8 topology and the seven flow entries

The Lab 8 topology is simple (Figure 4 in the document): one switch `s1` (an OVS bridge) and three hosts `h1` (10.0.0.1), `h2` (10.0.0.2), `h3` (10.0.0.3), all in the same subnet 10.0.0.0/8. The three switch interfaces are wired as `s1-eth1` to `h1-eth0`, `s1-eth2` to `h2-eth0`, `s1-eth3` to `h3-eth0`.

```
                  ┌────────────────┐
                  │   s1 (OVS)     │
     h1 ──eth1── │                │ ──eth3── h3
   10.0.0.1      │                │        10.0.0.3
                  │      eth2      │
                  └───────┬────────┘
                          │
                          h2
                        10.0.0.2
```

Security goal: *"switch blocks the traffic generated from host h3 (10.0.0.3) for the destination host h1 (10.0.0.1). Host h3 will be able to send reply if any traffic is generated from host h1"* (Lab 8, p11). In plain terms:

- h1 to h2: free (the firewall does not block).
- h1 to h3: initiation is allowed.
- h3 to h1: only allowed if the packet is a reply to a connection that h1 initiated earlier.

### The seven flow entries

After the topology is set up, seven `ovs-ofctl add-flow` commands are run in order (Lab 8, pages 11 to 13):

```bash
# Flow 1: default normal processing (default priority 32768)
sudo ovs-ofctl add-flow s1 "table=0,action=normal"

# Flow 2: h1 to h3, start tracking (priority 40000 is higher than normal)
sudo ovs-ofctl add-flow s1 \
  "table=0,priority=40000,ip,nw_src=10.0.0.1,nw_dst=10.0.0.3,ct_state=-trk,action=ct(table=1)"

# Flow 3: h3 to h1, start tracking
sudo ovs-ofctl add-flow s1 \
  "table=0,priority=40000,ip,nw_src=10.0.0.3,nw_dst=10.0.0.1,ct_state=-trk,action=ct(table=1)"

# Flow 4: h1 to h3 NEW: commit and forward to port 3
sudo ovs-ofctl add-flow s1 \
  "table=1,ip,nw_src=10.0.0.1,nw_dst=10.0.0.3,ct_state=+trk+new,action=ct(commit),3"

# Flow 5: h1 to h3 ESTABLISHED: forward to port 3
sudo ovs-ofctl add-flow s1 \
  "table=1,ip,nw_src=10.0.0.1,nw_dst=10.0.0.3,ct_state=+trk+est,action=output:3"

# Flow 6: h3 to h1 NEW: DROP (block reverse initiation)
sudo ovs-ofctl add-flow s1 \
  "table=1,ip,nw_src=10.0.0.3,nw_dst=10.0.0.1,ct_state=+trk+new,action=drop"

# Flow 7: h3 to h1 ESTABLISHED: forward to port 1 (allow the reply)
sudo ovs-ofctl add-flow s1 \
  "table=1,ip,nw_src=10.0.0.3,nw_dst=10.0.0.1,ct_state=+trk+est,action=output:1"
```

### Per-flow explanation

**Flow 1** is the catch-all at priority 32768. Any traffic that does not match flows 2 or 3 (for example, h1 to h2 or ARP broadcast) is handled by the `normal` action. OVS behaves like a traditional switch with MAC learning and BUM flooding. This is why h1 can ping h2 without a dedicated flow.

**Flows 2 and 3** match h1 to h3 packets that have not yet been tracked (`ct_state=-trk`). Their priority of 40000 is higher than flow 1, so they catch the packet into `ct()` for conntrack inspection. The `table=1` argument is the return target: after conntrack populates state, the packet reappears in table 1.

**Flow 4** handles the first packet from h1 to h3 (`+trk+new`): `ct(commit)` writes an entry into the state table, and the packet is then forwarded out of port 3 (to h3). From that point on, every packet with the same 5-tuple (h1:random_port to h3:dst_port) will be seen by conntrack as `+trk+est`.

**Flow 5** handles subsequent packets of the same h1 to h3 connection (`+trk+est`): no recommit is needed, just forwarding.

**Flow 6** handles the first packet from h3 to h1 that is not a reply to a connection initiated by h1 (`+trk+new`): `action=drop`. This is the flow that blocks the reverse direction.

**Flow 7** handles a reply packet from h3 to h1 for a connection that h1 has committed (`+trk+est`): forward to port 1 (to h1). This is precisely the flow that makes `ping h1 to h3` work, because h3's echo-reply matches `+trk+est` and not `+trk+new`.

> **Key point:** There is an interesting symmetry. Both flow 4 and flow 6 match `+trk+new`, but for different directions (the `nw_src` field is reversed). Flow 4 **allows initiation** for the trusted direction, and flow 6 **blocks initiation** for the untrusted direction. This structure is the canonical pattern of a stateful firewall. Every statement of the form "zone A trusts zone B, but zone B does not trust the reverse" reduces to this template.

### Verification commands

After the seven flows are installed, Lab 8 (pages 13 to 15) shows how to verify them:

```bash
# View the flow table just installed
sudo ovs-ofctl dump-flows s1

# From the h1 terminal: ping h2 and h3 (both succeed)
ping 10.0.0.2
ping 10.0.0.3

# View the conntrack state table
sudo ovs-dpctl dump-conntrack | grep "10.0.0.1"

# From the h3 terminal: ping h1 (fails because flow 6 drops it)
ping 10.0.0.1
```

The document states: *"The result in the figure above shows a successful connectivity between the hosts"* (for h1 to h3) and *"Host h3 cannot communicate with host h1 since switch s1 drops incoming traffic generated from host h3 for destination host h1"* (for the h3 to h1 ping test).

## 9.24.5 Guided Exercise 1, proof-of-error for "TCP reply auto-allowed"

**Goal:** prove by counterexample that the hypothesis *"OVS automatically allows a TCP reply if I allowed the request"* is false on a purely stateless OVS, and show that `ct()` together with `+trk+est` is the precise mechanism that reverses that hypothesis.

**Counter-hypothesis to refute:**

> *"If I install a single flow `action=output:3` for h1 to h3 and a single flow `action=output:1` for h3 to h1, then TCP `iperf` will run normally. OVS sees a TCP ACK reply as 'the connection is open' and forwards it automatically."*

### Prepare

Assume the Lab 8 topology is loaded (MiniEdit with `lab8.mn` or a bash script building netns plus ovs-vsctl by hand). Clear any existing flows:

```bash
sudo ovs-ofctl del-flows s1
```

### Step 1, predict

Before running anything, write down:

- The result of `ping h1 to h3` (success or failure)?
- The result of `iperf` TCP h1 to h3 (real bandwidth or timeout)?

The counter-hypothesis predicts that both succeed because symmetric flows are in place.

### Step 2, install symmetric stateless flows

```bash
sudo ovs-ofctl add-flow s1 "table=0,priority=100,ip,nw_src=10.0.0.1,nw_dst=10.0.0.3,action=output:3"
sudo ovs-ofctl add-flow s1 "table=0,priority=100,ip,nw_src=10.0.0.3,nw_dst=10.0.0.1,action=output:1"
sudo ovs-ofctl add-flow s1 "table=0,priority=10,action=normal"   # default for ARP and so on
```

### Step 3, measure

From the h1 terminal:

```bash
ping -c 3 10.0.0.3
iperf -c 10.0.0.3 -t 5   # assume h3 is running `iperf -s`
```

Expected outcome (refuting the hypothesis): ping and iperf both **succeed** in this stateless configuration, because the flows allow both directions unconditionally.

### Step 4, security inversion: block the reverse direction

Delete the h3 to h1 flow to simulate a firewall that blocks the reverse direction:

```bash
sudo ovs-ofctl del-flows s1 "ip,nw_src=10.0.0.3"
```

Measure again:

```bash
ping -c 3 10.0.0.3          # fails (ICMP reply is dropped)
iperf -c 10.0.0.3 -t 5      # times out (TCP SYN-ACK is dropped)
```

Result: **both fail**. This is the proof that the counter-hypothesis is false. Stateless OVS has no notion of "reply". Every packet is evaluated independently. If a matching flow exists, the packet passes; otherwise it is dropped.

### Step 5, restore the seven Lab 8 flow entries

```bash
sudo ovs-ofctl del-flows s1
# (install the seven flow entries from §9.24.4)
```

Measure again:

```bash
ping -c 3 10.0.0.3          # success (flow 4 for h1 to h3 NEW, flow 7 for h3 to h1 EST)
iperf -c 10.0.0.3 -t 5      # success (TCP three-way handshake through +trk+new and then +trk+est)

# From the h3 terminal
ping -c 3 10.0.0.1          # fails (flow 6 drops h3 to h1 NEW)
```

Result: h1 can initiate, h3 cannot initiate on its own, yet the h3 reply back to h1 for a connection initiated by h1 still passes through. The counter-hypothesis is fully refuted. The mechanism that makes *"replies are allowed"* work **does not come from symmetric flows**; it comes from `ct()` together with `+trk+est`.

### Lesson

A stateless firewall uses symmetric flows in both directions. A stateful firewall uses **asymmetric** flows: the active direction carries `+trk+new` together with `ct(commit)`, while the passive direction has only `+trk+est` (no `+new`). This template is reusable across many scenarios: a DMZ that accepts connections from the Internet to a web server (active direction reversed), a VPN gateway that allows only egress, and so on.

## Guided Exercise 2, inspect the state table and the TCP lifecycle

**Goal:** directly observe the state transitions of a TCP connection through `conntrack -E`, confirm the five states `SYN_SENT` to `SYN_RECV` to `ESTABLISHED` to `FIN_WAIT` to `TIME_WAIT`, and verify that `ovs-dpctl dump-conntrack` reflects the same information as a snapshot.

**Hypothesis to verify (this is a confirming hypothesis, not a counter-hypothesis):**

> *"With the seven Lab 8 flow entries installed, each TCP iperf session generates exactly one conntrack entry in the h1 to h3 direction. The entry passes through the four main lifecycle states and is removed after `tcp_timeout_time_wait` (120 seconds by default)."*

### Prepare

The Lab 8 topology is loaded and the seven flow entries from §9.24.4 are installed. Check the netfilter module:

```bash
sudo modprobe nf_conntrack
sudo sysctl net.netfilter.nf_conntrack_acct=1   # enable accounting to see byte counts
sudo sysctl net.netfilter.nf_conntrack_tcp_timeout_time_wait=120   # keep the default for observation
```

### Step 1, open two terminals at the same time

Terminal A (event tracking) on the Client node (outside the netns):

```bash
sudo conntrack -E -p tcp
```

Terminal B (snapshot loop), also on the Client, with a one-second interval:

```bash
while true; do sudo ovs-dpctl dump-conntrack | grep "10.0.0.1"; echo "---"; sleep 1; done
```

### Step 2, initiate a TCP session from h1

On the h3 terminal (Mininet xterm h3):

```bash
iperf -s
```

On the h1 terminal:

```bash
iperf -c 10.0.0.3 -t 5
```

### Step 3, observe the event stream

Expected event order on terminal A (doc-plausible, with a random port):

```
    [NEW] tcp      6 120 SYN_SENT src=10.0.0.1 dst=10.0.0.3 sport=52134 dport=5001 [UNREPLIED]
 [UPDATE] tcp      6 60 SYN_RECV  src=10.0.0.1 dst=10.0.0.3 sport=52134 dport=5001
 [UPDATE] tcp      6 432000 ESTABLISHED src=10.0.0.1 dst=10.0.0.3 sport=52134 dport=5001 [ASSURED]
 [UPDATE] tcp      6 60 FIN_WAIT src=10.0.0.1 dst=10.0.0.3 sport=52134 dport=5001 [ASSURED]
 [UPDATE] tcp      6 30 LAST_ACK src=10.0.0.1 dst=10.0.0.3 sport=52134 dport=5001 [ASSURED]
 [UPDATE] tcp      6 120 TIME_WAIT src=10.0.0.1 dst=10.0.0.3 sport=52134 dport=5001 [ASSURED]
```

Reading details:
- `6` is the IP protocol number for TCP (per `/etc/protocols`).
- The number after `6` is the remaining timeout in seconds. `SYN_SENT` has 120 s, `ESTABLISHED` has 432000 s (5 days), and `TIME_WAIT` has 120 s.
- `[UNREPLIED]` disappears once a reply arrives (transition to `SYN_RECV`). `[ASSURED]` appears when the handshake completes.

### Step 4, compare against the snapshot

On terminal B (the one-second `ovs-dpctl dump-conntrack` loop), at the moment the handshake completes:

```
tcp,orig=(src=10.0.0.1,dst=10.0.0.3,sport=52134,dport=5001),reply=(src=10.0.0.3,dst=10.0.0.1,sport=5001,dport=52134),zone=0,protoinfo=(state=ESTABLISHED)
```

About 120 seconds after `iperf -t 5` finishes, the state transitions to `TIME_WAIT`, and then the entry disappears from the output.

### Step 5, tune the timeout to verify

Reduce the time-wait timeout to 10 seconds so the entry removal is easy to observe:

```bash
sudo sysctl net.netfilter.nf_conntrack_tcp_timeout_time_wait=10
```

Rerun iperf. About 10 to 15 seconds after the session ends, `dump-conntrack` no longer shows the entry. This is direct evidence that timeout governs the lifetime of the entry, not the packet count.

### Lesson

A conntrack entry has a clear lifecycle with four to six main states for TCP (fewer for UDP and ICMP). The timeout is per-protocol and per-state. The rule is not *"delete when the connection closes"* but rather *"delete after N seconds without traffic in the current state"*. An important operational consequence is that a conntrack entry can become a zombie if a state transition fails (for example, an RST that does not reach one end). That is when `conntrack -D` or the OVS `ct_clear` action is used to remove it manually.

## Guided Exercise 3, proof-of-error for UDP conntrack

**Goal:** prove by counterexample that the hypothesis *"UDP has no state, so conntrack does not track UDP"* is false, and show that OVS can still protect UDP through `ct()` using UDP pseudo-state.

**Counter-hypothesis to refute:**

> *"UDP is connectionless and has no handshake. Therefore conntrack cannot know which UDP packet is NEW and which is ESTABLISHED. The rule `ct_state=+trk+est` has no meaning for UDP, so every valid UDP packet must use `+trk+new`."*

### Prepare

Lab 8 topology. Clear existing flows and reinstall the seven flows from §9.24.4. Add UDP-matching flows in parallel so the configuration covers both protocols:

```bash
sudo ovs-ofctl add-flow s1 "table=0,priority=40000,udp,nw_src=10.0.0.1,nw_dst=10.0.0.3,ct_state=-trk,action=ct(table=1)"
sudo ovs-ofctl add-flow s1 "table=0,priority=40000,udp,nw_src=10.0.0.3,nw_dst=10.0.0.1,ct_state=-trk,action=ct(table=1)"
sudo ovs-ofctl add-flow s1 "table=1,udp,nw_src=10.0.0.1,nw_dst=10.0.0.3,ct_state=+trk+new,action=ct(commit),3"
sudo ovs-ofctl add-flow s1 "table=1,udp,nw_src=10.0.0.1,nw_dst=10.0.0.3,ct_state=+trk+est,action=output:3"
sudo ovs-ofctl add-flow s1 "table=1,udp,nw_src=10.0.0.3,nw_dst=10.0.0.1,ct_state=+trk+new,action=drop"
sudo ovs-ofctl add-flow s1 "table=1,udp,nw_src=10.0.0.3,nw_dst=10.0.0.1,ct_state=+trk+est,action=output:1"
```

### Step 1, predict

According to the counter-hypothesis, the UDP test below should fail because `+trk+est` would not match a UDP reply:

```
iperf -u -s                # on h3
iperf -u -c 10.0.0.3 -t 5  # on h1 (UDP client)
```

### Step 2, run and measure

On the h3 terminal:

```bash
iperf -u -s
```

On the h1 terminal:

```bash
iperf -u -c 10.0.0.3 -t 5 -b 1M
```

Observe the result on the h1 terminal:
```
Client connecting to 10.0.0.3, UDP port 5001
Sending 1470 byte datagrams
[  3] local 10.0.0.1 port 52180 connected with 10.0.0.3 port 5001
[  3]  0.0- 5.0 sec    625 KBytes    1.02 Mbits/sec
[  3] Sent 447 datagrams
[  3] Server Report:
[  3]  0.0- 5.0 sec    625 KBytes    1.02 Mbits/sec   0.234 ms    0/  447 (0%)
```

**The server-report reply passes through**, so the counter-hypothesis is **refuted**.

### Step 3, explain the mechanism

Check the conntrack state table:

```bash
sudo ovs-dpctl dump-conntrack | grep "10.0.0.1" | grep udp
```

Doc-plausible output:
```
udp,orig=(src=10.0.0.1,dst=10.0.0.3,sport=52180,dport=5001),reply=(src=10.0.0.3,dst=10.0.0.1,sport=5001,dport=52180),zone=0
```

A UDP entry exists in the state table. Conntrack determines `NEW` and `ESTABLISHED` for UDP as follows:

- The first UDP packet from h1 to h3 is `NEW`, and the entry is committed.
- The first UDP packet in the reverse direction (the report reply h3 to h1) matches the existing entry's reply tuple, so it transitions to `ESTABLISHED`.
- Every subsequent packet with the same 5-tuple is `ESTABLISHED` until the timeout expires (30 seconds by default).

UDP pseudo-state is not "handshake" but "traffic seen in both directions". Once traffic flows both ways, conntrack treats the connection as established.

### Step 4, try the precise counter-hypothesis

The counter-hypothesis would be true **if and only if** UDP traffic were strictly one-way. Try it:

```bash
# On h1: send one-way UDP with no server
iperf -u -c 10.0.0.3 -t 5 -b 1M   # h3 is not running -u -s
```

Expectation: iperf still runs (the client only sends), and the state table holds an entry that remains at `UNREPLIED`:

```
udp,orig=(src=10.0.0.1,dst=10.0.0.3,sport=52190,dport=5001),reply=(src=10.0.0.3,dst=10.0.0.1,sport=5001,dport=52190),zone=0,[UNREPLIED]
```

An ICMP destination-unreachable packet may arrive from h3 because port 5001 is closed. Conntrack handles that packet through the `RELATED` state (see §9.24.2), not `ESTABLISHED`.

### Lesson

UDP **does** have pseudo-state in conntrack. It is simply simpler than TCP (two main states, unreplied and assured, instead of the six for TCP). The `+trk+est` rule works for both TCP and UDP. The hypothesis *"UDP is stateless, so conntrack ignores it"* confuses protocol semantics (UDP has no handshake) with the tracking mechanism (conntrack tracks bidirectional flow regardless of protocol). This is also why OVN `allow-related` works for both `udp` and `tcp` match expressions without a UDP-specific flag.

## 9.24.6 Monitoring the conntrack state table

After the seven Lab 8 flows are working, two commands are essential for debugging and operating a stateful firewall:

### `ovs-dpctl dump-conntrack`

Dumps the full state table of the current OVS datapath. The output is one line per entry, formatted per protocol:

```
icmp,orig=(src=10.0.0.1,dst=10.0.0.3,id=41501,type=8,code=0),reply=(src=10.0.0.3,dst=10.0.0.1,id=41501,type=0,code=0),zone=0
tcp,orig=(src=10.0.0.1,dst=10.0.0.3,sport=39246,dport=5001),reply=(src=10.0.0.3,dst=10.0.0.1,sport=5001,dport=39246),zone=0,protoinfo=(state=ESTABLISHED)
```

When reading this output, look at three pieces of information:

- **`orig=` and `reply=`**: the 5-tuples for both directions. The `orig` direction is the one for which `ct(commit)` ran. The `reply` direction is the auto-generated reverse.
- **`zone=N`**: the namespace of the state table. When the flow uses `ct_zone` (see §9.24.7), a non-zero zone appears here.
- **`protoinfo=(state=...)`** for TCP: `SYN_SENT`, `SYN_RECV`, `ESTABLISHED`, `FIN_WAIT`, `TIME_WAIT`, `CLOSE_WAIT`, `LAST_ACK`, `CLOSE`. After the TCP session ends (RST or FIN), the entry transitions to `TIME_WAIT` for another 120 seconds and is then removed.

Lab 8 (page 18) illustrates: *"The figure shows the TCP test. You will notice that source h1 generates the TCP session using source port 39246 and destination port 5001. This flow will be stored. Whenever host h3 is sending a reply to host h1 using source port 5001 and destination port 39246, switch s1 will allow the traffic since the source, and destination ports are the same."*

### `conntrack -E`

A command from the `conntrack-tools` package (not OVS), which streams events in real time as conntrack creates, updates, or removes entries. It is the most powerful debugging tool for stateful rules. When a firewall does not behave as expected, running `conntrack -E` in parallel with test traffic exposes whether entries are actually being committed.

```bash
sudo conntrack -E | grep "10.0.0.1"
```

Expected output when h1 runs `iperf -c 10.0.0.3` (Lab 8 page 16):

```
    [NEW] tcp      6 120 SYN_SENT src=10.0.0.1 dst=10.0.0.3 sport=39246 dport=5001 [UNREPLIED] ...
 [UPDATE] tcp      6 60 SYN_RECV  src=10.0.0.1 dst=10.0.0.3 sport=39246 dport=5001 ...
 [UPDATE] tcp      6 432000 ESTABLISHED src=10.0.0.1 dst=10.0.0.3 sport=39246 dport=5001 [ASSURED] ...
 [UPDATE] tcp      6 120 TIME_WAIT src=10.0.0.1 dst=10.0.0.3 sport=39246 dport=5001 [ASSURED] ...
[DESTROY] tcp      6 TIME_WAIT src=10.0.0.1 dst=10.0.0.3 sport=39246 dport=5001 [ASSURED] ...
```

> **Lab verification state:** the output above is `doc-plausible`, based on Lab 8 pages 16 to 18 (Crichigno et al. 2021) along with the default `conntrack -E` format in `conntrack-tools` 1.4.6. The port `39246` is random and will differ in a real run. The output will be promoted to `verified-lab` when a lab host is available in phase C1b.

Five state transitions worth remembering: `[NEW]` to `SYN_SENT` (h1 sends SYN) to `SYN_RECV` (h3 sends SYN-ACK) to `ESTABLISHED` (h1 sends ACK, completing the handshake) to `TIME_WAIT` (FIN in both directions) to `[DESTROY]` (after the 120-second `tcp_timeout_time_wait`).

### Timeout and cleanup

A detail that is easy to overlook: a conntrack entry does not live forever. The default timeouts are controlled through sysctl:

```bash
# View the current values
sudo sysctl net.netfilter.nf_conntrack_tcp_timeout_established   # 432000 (5 days)
sudo sysctl net.netfilter.nf_conntrack_tcp_timeout_time_wait     # 120
sudo sysctl net.netfilter.nf_conntrack_udp_timeout               # 30
sudo sysctl net.netfilter.nf_conntrack_icmp_timeout              # 30
```

Lab 8 (page 15) warns: *"Verify conntrack state table right after pinging since the table information is deleted after a certain period of time. If you do not get expected result, run the test again and see the conntrack table."* This is precisely because the default ICMP timeout is only 30 seconds.

In a large OVN production cluster, these timeout values are adjusted through the `CT_Timeout_Policy` table in OVSDB (see compass Chapter 14), which allows per-zone timeout policies. That is important when a single OVN chassis hosts multiple tenants with very different traffic patterns.

## 9.24.7 `ct_zone`, multi-tenant isolation

Suppose two tenants A and B both use the IP range 10.0.0.0/24 (overlapping). Tenant A has a VM at 10.0.0.5 that talks to 10.0.0.10. Tenant B also has a VM at 10.0.0.5 that talks to 10.0.0.10. The two connections are completely independent, but their 5-tuples are identical. If they share the same state table, conntrack will confuse them.

`ct_zone` (a 16-bit value) is the solution. When `ct(zone=N,commit)` runs, the entry is stored in the sub-table of zone N. A read-only `ct(zone=N)` also inspects only zone N. Two entries with the same 5-tuple but different zones are two independent connections.

### Example, using `ct_zone` in OVN

OVN assigns a zone by rule: each logical router and logical switch has its own zone ID, stored in `Port_Binding.external_ids` and translated by `ovn-controller` into a flow entry such as:

```
table=N,priority=50,metadata=0x1,ct_state=-trk,ip,action=ct(table=N,zone=NXM_NX_REG13[0..15])
```

`reg13[0..15]` holds the zone ID assigned to the corresponding logical switch (compass Chapter 17). When two tenants overlap on IP, the `reg13` value differs, the zone differs, and the state tables are separate.

### Verify with `ovs-dpctl dump-conntrack`

When traffic uses multiple zones, the `dump-conntrack` output displays the `zone=N` field:

```
tcp,orig=(src=10.0.0.5,dst=10.0.0.10,sport=1234,dport=80),reply=(...),zone=7,protoinfo=(state=ESTABLISHED)
tcp,orig=(src=10.0.0.5,dst=10.0.0.10,sport=1234,dport=80),reply=(...),zone=12,protoinfo=(state=ESTABLISHED)
```

Two entries with the same 5-tuple `10.0.0.5:1234 to 10.0.0.10:80` but different zones (`zone=7` and `zone=12`) do not collide.

> **Common misconception:** *"Zone 0 is the default and it is harmless."* Zone 0 is simply the default value when no zone is specified. If none of your flows include `zone=N`, then every connection of every tenant lands in shared zone 0. In an OVN production environment, `ovn-controller` always assigns a non-zero zone for every logical pipeline and never uses zone 0 for tenant traffic. This confusion causes "tenant X's connection is dropped because of a conntrack hit from tenant Y", a symptom that is very hard to debug without knowing about zones.

## 9.24.8 Comparison with OVN: `allow-related` is a macro for `ct(commit)`

OVN does not force the operator to write the seven flow entries seen in Lab 8. Instead, OVN exposes a concise API in `ovn-nbctl`:

```bash
ovn-nbctl acl-add ls-tenant1 from-lport 1000 \
  "ip4.src == 10.0.0.0/24 && tcp.dst == 80" allow-related
```

The keyword `allow-related` is in fact a macro that generates the flow pair equivalent to the seven Lab 8 flows for every LSP belonging to logical switch `ls-tenant1`. The translation mechanism (analysed in detail in Part 13.3 and 13.8):

1. `ovn-northd` reads the ACL from `Northbound_DB.ACL` and emits a Logical Flow into `Southbound_DB.Logical_Flow` with the actions `ct_next`, `ct_commit`, and `next`.
2. `ovn-controller` on each chassis reads the Logical Flow and emits an OpenFlow rule that calls `ct(zone=<lswitch-zone>,commit)` on bridge `br-int`.
3. The rule is similar to Lab 8 flow 4, except that the `zone` is automatically assigned.

Three important differences:

**Level of abstraction.** An OVS operator writes flow entries directly with IP 5-tuples. An OVN operator writes ACLs with logical expressions (`outport == @pg-db`, `ct.est`). The compiler below handles the translation automatically.

**Automatic zone assignment.** An OVS operator chooses `zone=N` by hand and must ensure no collisions between tenants. OVN automatically assigns a zone for every logical switch, fully transparently.

**Distributed commit.** Lab 8 OVS has a single switch s1, so every commit lands in one kernel conntrack. An OVN cluster has many chassis, each with its own conntrack. OVN must synchronise state through SBDB `Port_Binding.chassis` and through rerouting, with full details in Part 19.0.

### Load Balancer, SNAT plus conntrack

OVN Load Balancer (`ovn-nbctl lb-add`) is a direct application of `ct(nat(src=...))`. When the VIP has multiple backends:

```bash
ovn-nbctl lb-add lb1 10.0.100.100:80 10.0.1.1:8080,10.0.1.2:8080
```

`ovn-northd` emits a Logical Flow with the action `ct_lb`, which translates to OpenFlow as either `ct(commit,nat(dst=10.0.1.1:8080))` or `ct(commit,nat(dst=10.0.1.2:8080))` depending on the hash. Load balancing has a two-pass lifecycle:

- Pass 1 (the NEW request packet): hash selects a backend, `ct(commit,nat(dst=backend))` runs, and the entry is committed with the DNAT binding.
- Pass 2 (subsequent EST packets): conntrack matches the previously committed entry and applies the same DNAT binding, so the mapping is stable.

Thanks to `ct(commit)`, every packet of the same connection is routed to the same backend. This is the basic "session persistence" principle of an L4 load balancer (Part 13.9 Load Balancer OVN analyses it in more detail).

> **Key point:** Every stateful feature of OVN reduces to `ct()`. `allow-related` is `ct(commit)`. Load Balancer is `ct(commit,nat)`. SNAT gateway is `ct(commit,nat(src))`. DNAT hairpin is `ct(commit,nat(dst))` followed by reinjection into the pipeline. Learning `ct()` on OVS is therefore mastering 80 percent of OVN's stateful mechanism. The remaining 20 percent is the abstraction layer that generates the flows.

## 9.24.9 Core takeaways

**One.** A stateless firewall evaluates each packet independently based on the 5-tuple header and has no notion of "reply". A stateful firewall tracks connection state and allows the reply of a valid connection through without a dedicated rule.

**Two.** OVS implements a stateful firewall through the `ct()` action, exposing the Linux kernel's conntrack module into the OpenFlow pipeline. Six state bits (`+trk`, `+new`, `+est`, `+rel`, `+inv`, `+rpl`) are matched through the `ct_state` field.

**Three.** The seven-flow Lab 8 pattern from Crichigno is the canonical template: two `-trk` flows pull the packet into `ct()` and back into the pipeline, one `+trk+new` flow for the active direction with `ct(commit)`, two `+trk+est` flows for both directions, and one `+trk+new` flow for the passive direction with `action=drop`.

**Four.** Two monitoring tools are essential: `ovs-dpctl dump-conntrack` for a snapshot of the state table, and `conntrack -E` for a real-time event stream. Default timeouts (5 days for TCP established, 30 seconds for UDP, 30 seconds for ICMP) can be tuned through `net.netfilter.nf_conntrack_*_timeout_*` sysctl entries.

**Five.** `ct_zone` is an isolated namespace for the state table that allows overlapping 5-tuples across tenants without collision. OVN auto-assigns a zone for each logical switch through `reg13[0..15]`; an OVS operator working by hand must choose the zone explicitly.

**Six.** OVN `allow-related` is a macro that emits `ct(commit)`. OVN Load Balancer is `ct(commit,nat(dst))`. OVN SNAT gateway is `ct(commit,nat(src))`. Learning `ct()` on OVS amounts to mastering the foundation of all OVN stateful behaviour.

## 9.24.10 Additional treatment for `ct_state` and `ct_zone` (expansion 2026-04)

> **Purpose:** Cover the missing treatment dimensions for the two cornerstone keywords `ct_state` and `ct_zone`. `ct_state` already has comprehensive coverage in earlier sections, so only its history is added here. `ct_zone` is presented in a comprehensive multi-dimensional treatment because earlier sections only introduced it briefly in §9.24.7.

### 9.24.10.1 `ct_state`, history of introduction

The match field `ct_state` was introduced by Nicira in OVS 2.5 (released 2016-02-24) together with the `ct()` action, originally as the OVS extension `NXM_NX_CT_STATE`. Before OVS 2.5, OVS only had stateless flow rules; every stateful need had to be handed off to `iptables` running in parallel on the same host, which caused many problems with rule-application order and with semantics. Ben Pfaff (Nicira and later VMware) proposed `ct_state` so that OVS could consume conntrack state directly inside the OF pipeline without an upcall to userspace for each packet. ONF later standardized `ct_state` as `OXM_OF_CT_STATE` in OF 1.6+ (2018). Verified upstream history: commit `07659514c3c1e8998a4935a998b627d716c559f9`, dated 2015-08-11, "Add support for connection tracking", in `lib/ofp-actions.c`. The userspace conntrack engine `lib/conntrack.c` was added later in commit `a489b16854b590004e3234573721fd0f676b3295`, dated 2015-11-15. The OVS 2.5 release in February 2016 was simply the first stable release that contained the feature; it was not the implementation date. Both SHAs were verified on 2026-04-28 against the local OVS repo with `git log -1 <sha>`. OVN since version 2.5 has been the largest "customer" of `ct_state`, using it as the backbone of every ACL that includes the keyword `allow-related`.

### 9.24.10.2 `ct_zone`, comprehensive multi-dimensional treatment

**Concept.** `ct_zone` is a 16-bit OpenFlow match field (values 0 to 65535), and at the same time a parameter of the `ct()` action. Each zone value is an independent namespace of the conntrack state table: two entries with the same 5-tuple but different zones are two completely unrelated connections. Type: an OF match field in the `ct_*` family, originally an OVS extension that was later adopted into OF 1.6+ by ONF.

**History and background.** `ct_zone` appeared together with the `ct()` action in OVS 2.5 to 2.6 (2016) as a mandatory component of the multi-tenant design. OVN from 2.7 onwards automatically assigns a zone for every Logical_Switch and Logical_Router through register `reg13[0:15]`. OVN 24.03 added the ability for the operator to set a manual `zone-id` through `Logical_Switch.options:zone-id` to avoid collisions in a large cluster.

**Position in the architecture.** `ct_zone` appears at two points in the pipeline: (a) as a parameter of the action `ct(zone=N,...)`, deciding which conntrack namespace the packet is checked against or written to, and (b) as a match field `ct_zone=N` so that downstream rules can recognise which zone the packet belongs to. The zone data lives alongside the entry in the conntrack table (kernel `nf_conntrack` or userspace `lib/conntrack.c`) and is also stored in the OF flow key of the packet after it returns to the pipeline.

**Role.** A multi-tenant isolation primitive for the conntrack state table. Each zone is an independent state machine, with no 5-tuple collision between tenants.

**Why it exists.** Before zones, a single host had a single shared conntrack table. In a multi-tenant cluster, two tenants can easily have the same pair `10.0.0.5 to 10.0.0.10:80`. The same 5-tuple maps to the same conntrack entry, so a reply packet of tenant A could be misattributed to tenant B's connection, breaking security isolation. The zone resolves this completely by adding a sixth field to the lookup key.

**Problem solved.** Tenant 5-tuple collision on the same host. Mechanism: partition the conntrack table by a 16-bit zone field; tag every packet with a zone through the `ct(zone=N)` action; perform conntrack lookup using the composite key (5-tuple, zone).

**Importance.** A core absolute of multi-tenant conntrack. Every multi-tenant OVS or OVN deployment relies on zones. An engineer who does not understand zones cannot resolve zone-collision incidents in a large cluster (see Part 19.0 and 20.5).

**How it works.** When a packet meets the action `ct(zone=N)`:

```
Step 1: ofproto-dpif xlate translates the action into the odp action
        OVS_ACTION_ATTR_CT with OVS_CT_ATTR_ZONE = N.
Step 2: The kernel datapath calls ovs_ct_execute() with info->zone.id = N.
Step 3: __ovs_ct_lookup() passes the zone into nf_conntrack through
        struct nf_conn_zone.
Step 4: nf_conntrack performs the hash table lookup with the composite key
        (5-tuple, zone, dir).
Step 5: On a miss, if the commit flag is set, create a new entry tagged
        with zone=N.
Step 6: After conntrack finishes, the packet returns to the pipeline with
        ct_zone=N attached to the flow key, so downstream rules can match
        the ct_zone field.
```

OVN auto-assigns the zone through the syntax `ct(zone=NXM_NX_REG13[0..15])`, where `reg13` holds the `tunnel_key` of the Datapath_Binding for the corresponding logical switch.

**How an operator masters this.** Four skills to develop: (1) read the `zone=N` column in the output of `ovs-appctl dpctl/dump-conntrack`; (2) place `ct(zone=N)` at the right logical level when writing native flows; (3) filter entries by zone with `dpctl/dump-conntrack zone=N`; (4) detect zone collisions by checking whether the same 5-tuple appears in suspicious multiple zones.

**Classification.** A 16-bit OF match field in the `ct_*` family. Also a parameter of the `ct()` action. A state-attached identifier.

**Usage workflow.**

```
Step 1: Identify the tenant boundary that needs isolation.
        OVN: each Logical_Switch or Logical_Router is one zone.
        OVS native: each VLAN or each customer subnet is one zone.

Step 2: Assign the zone explicitly.
        OVN: automatic through reg13, no extra configuration needed.
        OVS native: write the zone into the flow rule, for example `ct(zone=42)`.

Step 3: Verify isolation.
        $ ovs-appctl dpctl/dump-conntrack zone=42

Step 4: Avoid duplicate zones across logical entities in a large cluster.
        Use options:zone-id (OVN 24.03+) when the cluster risks a tunnel_key collision.
```

**When something goes wrong.** Two common signals: (a) cross-tenant traffic leakage; check `dpctl/dump-conntrack` for entries that fall into `zone=0` (the default zone when an assignment is forgotten). (b) random periodic drops; check whether two logical entities share the same zone.

**Tightly related to.** `ct_state` (consumes the per-zone lookup result), `reg13` (where OVN stores the zone), `Datapath_Binding.tunnel_key` (the source for auto-assignment), `nf_conntrack_buckets` (the sysctl that bounds the per-zone table), Part 19.0 (the multi-tenant PMTUD case).

**Version differences.**

| Version | Behaviour |
|---------|-----------|
| OVS 2.5 (Feb 2016) | Introduced the 16-bit zone field, required for multi-tenant. `MFF_CT_ZONE` defined in `lib/meta-flow.h:804` at v2.5.0, in the same commit `07659514c3c1e899` (2015-08-11) that added `ct()`. The header file moved from `lib/meta-flow.h` to `include/openvswitch/meta-flow.h` only in v2.6.0 (2016-04-04, commit `064d7f842838bdc4`), but the field itself existed earlier. |
| OVS 2.8 | Added `ct_nw_proto`, `ct_tp_src`, `ct_tp_dst` for finer-grained lookup. |
| OVN 22.03 | Standardized automatic zone assignment through `reg13` for every pipeline. |
| OVN 24.03 | Exposed `Logical_Switch.options:zone-id` for manual configuration. |

**Observation and verification.**

```bash
$ ovs-appctl dpctl/dump-conntrack zone=42 | head
tcp,orig=(src=10.0.0.5,dst=10.0.0.10,...),reply=(...),zone=42,protoinfo=(state=ESTABLISHED)

$ ovs-ofctl dump-flows br0 | grep "ct(zone="
$ ovn-sbctl --columns=tunnel_key list Datapath_Binding   # source of OVN zones
```

**Source code reference.**
- `net/openvswitch/conntrack.c`, function `ovs_ct_execute()`, handles the `info->zone` parameter.
- `lib/conntrack.c`, userspace conntrack, partitions by zone for the userspace datapath.
- Linux `include/net/netfilter/nf_conntrack_zones.h` defines `struct nf_conntrack_zone`.
- `northd/northd.c`, function `build_lswitch_lflows_acl()`, writes the `reg13` zone on the OVN side.

**Real-world incident.** Multi-tenant OpenStack and OVN cluster, Q3 2024. East-west traffic between two VMs on the same Logical_Switch dropped by about 5 percent. Investigation: `ovn-trace` was clean, `ofproto/trace br-int` showed the packet stuck at table 12 (the ct stage). `dpctl/dump-conntrack zone=42` returned an entry with state `INVALID`. Deeper analysis showed that zone 42 was simultaneously being used for `LS-A` (tenant A) and for `LR-transit-12` (a transit router). The two `tunnel_key` values differed in their 32-bit form but collided in the lower 16 bits after truncation to `ct_zone`. Conntrack entries collided at random. Remediation: upgrade the cluster to OVN 24.03, assign `LS-A.options:zone-id=100` and `LR-transit-12.options:zone-id=200`, and reapply the configuration. The drop rate then returned to zero.

**Synthetic exercise.**

```bash
sudo ovs-vsctl add-br br0
sudo ovs-ofctl add-flow br0 'priority=100,ip,ct_state=-trk,actions=ct(table=0,zone=42)'
sudo ovs-ofctl add-flow br0 'priority=200,ip,ct_state=+trk+new,actions=ct(commit,zone=42),NORMAL'
sudo ovs-ofctl add-flow br0 'priority=100,ip,ct_state=-trk,actions=ct(table=0,zone=84)'   # different zone for tenant 2

# Verify isolation
ovs-appctl dpctl/dump-conntrack zone=42 | wc -l
ovs-appctl dpctl/dump-conntrack zone=84 | wc -l
# The two zone tables are fully independent.
```

**Common failures and diagnostic signals.**
- Zone exhaustion (rare, with a 16-bit space). Signal: kernel log "zone limit". Action: reallocate zones.
- Zone collision (the production case above). Signal: cross-tenant state entries overlap. Action: set a manual `zone-id` (OVN 24.03+).
- State loss when an entry expires, dropping long-lived TCP connections after idle. Action: tune `nf_conntrack_tcp_timeout_established`.

**Comparison with other systems.**
- **iptables `-m conntrack --ctzone`:** uses the same zone source in kernel netfilter. Difference: iptables binds it rigidly to a chain, while OVS `ct_zone` is applied per OF rule in the pipeline.
- **Linux VRF:** isolation at the L3 layer. Difference: VRF is configured statically per netns, while `ct_zone` is dynamic per OF flow.
- **VMware NSX-T Distributed Firewall per-segment state:** isolation per segment. Difference: NSX-T distributes the tag through the management plane, while OVS `ct_zone` is set per packet through a register.

## 9.24.11 `conntrack execute`, the userspace and kernel runtime of the `ct()` action

> **Scope:** comprehensive treatment (expansion 2026-04-26) for the keyword `conntrack execute`. This is the common name for the pair of runtime functions that implement the OpenFlow `ct()` action: `conntrack_execute()` in `lib/conntrack.c` (the userspace datapath, used by DPDK and by `ovs-vswitchd --enable-userspace-datapath`) and `ovs_ct_execute()` in `net/openvswitch/conntrack.c` (the kernel datapath, used by the default OVS deployment on Linux). Together with `nf_conntrack_in` at the netfilter layer, these three functions form the entire engine that turns every `ct()` action in a flow rule into an actual change in the state table. This part is deeper than the earlier `ct_state`, `ct_zone`, and `ct_mark` sections. Those three keywords answer the question "what value do I match on", while `conntrack execute` answers "who runs the code, where, when, and how".

**Concept.** `conntrack execute` is not an action in the OpenFlow text; it is the common name for the two runtime functions that implement the `ct()` action. The `ct()` action in the flow text is the handle (interface) seen by the controller; `conntrack execute` is the implementation that the packet meets at runtime. Distinguishing these two concepts clearly is the first step in reading the OVS source: `lib/ofp-actions.c` parses the text `ct(...)` into `struct ofpact_conntrack`; `lib/odp-execute.c` then translates it into `OVS_ACTION_ATTR_CT`; depending on the datapath, one of the two runtime functions consumes this action.

**History and background.** This pair of functions was introduced by Nicira and VMware in OVS 2.5 (released 2016-02-24) together with the `ct()` action. The verified upstream commit is `07659514c3c1e8998a4935a998b627d716c559f9`, dated 2015-08-11, "Add support for connection tracking", in `lib/ofp-actions.c`. The userspace conntrack engine `lib/conntrack.c` was added later by commit `a489b16854b590004e3234573721fd0f676b3295`, dated 2015-11-15. Both SHAs were verified on 2026-04-28 against the local OVS repo with `git log -1 <sha>`. The first version only had the kernel branch: `ovs_ct_execute` pushed the packet through `nf_conntrack_in` and then back into the pipeline. In late 2016, OVS 2.7 added `conntrack_execute` in userspace so that the DPDK datapath did not have to fall back to the kernel for every packet (which avoided context-switch overhead). From 2018 onwards, the two functions evolved independently but had to keep identical semantics: the same input must produce the same output, the same state machine, and the same tuple key. This is the principle of "behavioural parity between datapaths" that Pfaff articulated in the NSDI 2015 paper.

**Position in the architecture.** Within the integrated diagram already presented in §9.1 and §9.2:

```
Userspace ovs-vswitchd
   ofproto-dpif-xlate.c
       └─ xlate_action_CT → emit OVS_ACTION_ATTR_CT (odp action)
   odp-execute.c
       └─ dispatch theo dpif type
              ↓                                    ↓
   dpif-netdev (userspace datapath)        dpif-netlink (kernel datapath)
       ↓                                          ↓
   lib/conntrack.c                          net/openvswitch/conntrack.c
       conntrack_execute()                       ovs_ct_execute()
            ↓                                        ↓
       process_one()                             __ovs_ct_lookup()
       conn_lookup()                                 ↓
            ↓                                    nf_conntrack_in()
       (userspace state machine)                (kernel netfilter engine)
```

The boundary is clear: above dpif is shared code that does not know whether the datapath is kernel or userspace; below dpif are two separate worlds that must remain in behavioural parity.

**Role.** Four roles: (a) look up the conntrack table for the current 5-tuple and return the reverse tuple, the direction, and the state-machine code; (b) populate the `ct_state` bitfield (`+trk`, `+new`, `+est`, `+rel`, `+inv`, `+rpl`) on the flow key; (c) commit, that is, insert a new entry or update an existing one if the action carries the `commit` flag; (d) run sub-actions including NAT, helper ALGs (FTP, SIP, TFTP), and setting `ct_mark` or `ct_label`.

**Why it exists.** Before OVS 2.5, anyone who wanted stateful behaviour on OVS had to run iptables in parallel on the same host, which caused two large problems: (1) the order of processing was ambiguous, since whether a packet hit the iptables `PREROUTING` chain or the OVS datapath first was hard to predict; (2) the state machine was duplicated across the two engines, so some protocols like SIP could be tracked twice. Placing the conntrack logic inside the OVS datapath eliminates both problems: the `ct()` action runs at the precise position in the pipeline that the flow rule specifies, and its result is attached directly to the flow key for downstream rules to consume.

**Problem solved.** Four classes of problem: (1) synchronise the conntrack lookup with the OF pipeline, with no dependency on netfilter chain order; (2) preserve behavioural parity between the kernel and userspace datapaths so that a single flow rule from the controller works on both; (3) enable multi-zone through a single common interface; (4) enable stateful NAT (commit together with rewrite) without running an iptables NAT table in parallel.

**Importance.** A core-tier-1 cornerstone alongside `recirc_id`, `dpif`, and `ofproto-dpif xlate`. It is the final bridge between "the controller writes a `ct()` action" and "the kernel or userspace actually writes the entry". Without understanding these two functions, you cannot debug any behavioural difference between the two datapaths. In a mixed production cluster (some chassis kernel, some chassis DPDK), an engineer who cannot distinguish the two branches will chase surface symptoms without reaching the root cause.

**How it works.** Detailed trace of an `iperf -c` packet from the flow rule to the conntrack entry, on the kernel datapath:

```
1. ovs-vswitchd receives an upcall, and ofproto-dpif-xlate translates the flow:
   Lab 8 flow 4: ct_state=+trk+new,actions=ct(commit),3
   emits the odp actions:
      OVS_ACTION_ATTR_CT(zone=0, commit=1, mark=0, ...)
      OVS_ACTION_ATTR_OUTPUT(port=3)

2. odp-execute.c sends the structure down to the kernel through the
   netlink genl OVS_PACKET_CMD_EXECUTE.

3. The kernel net/openvswitch/actions.c::do_execute_actions() encounters
   OVS_ACTION_ATTR_CT and calls:
     ovs_ct_execute(net, skb, key, info)

4. ovs_ct_execute (linux/net/openvswitch/conntrack.c):
     - Normalise L3 (defragment IP fragments if any).
     - If commit=1: call ovs_ct_commit(), then ovs_ct_lookup().
     - If commit=0: call ovs_ct_lookup() directly.

5. ovs_ct_lookup calls __ovs_ct_lookup:
     - Check skb_nfct_cached(), which tells whether the packet has already
       passed through netfilter.
     - If not: call nf_conntrack_in(skb, &state) with state.hook =
       NF_INET_PRE_ROUTING. This is the canonical "entry call" into the
       netfilter engine.
     - nf_conntrack_in performs the hash table lookup
       (5-tuple, zone, l3proto, l4proto):
         hit:  return the existing struct nf_conn; determine the direction.
         miss: create a new struct nf_conn and link it into the unconfirmed
               list.
     - After nf_conntrack_in returns, __ovs_ct_lookup attaches the ct_state
       bitfield to struct sw_flow_key based on the IPS_* status bits of nf_conn.

6. If the action has a nat(...) clause: __ovs_ct_nat() runs
     nf_nat_alloc_null_binding or nf_nat_setup_info, applying the NAT mapping.

7. If the action has an alg= clause: __ovs_ct_helper() runs
     nf_ct_helper_ext_add to attach an FTP, TFTP, or SIP helper, which then
     parses the L7 payload and creates an expectation.

8. If commit=1 and the packet is in the unconfirmed list:
     __nf_conntrack_confirm() pushes struct nf_conn into the confirmed hash
     table. This is the moment when the ct entry becomes effective.

9. The packet returns to the pipeline with key->ct_state, key->ct_zone, and
   key->ct_mark populated. The OUTPUT:3 action is then executed.
```

The userspace datapath follows a similar flow but calls `conntrack_execute`, then `process_one`, then `conn_lookup` in `lib/conntrack.c` instead of `ovs_ct_execute`. The userspace state machine is hand-written (it does not call `nf_conntrack_in`) but must produce the same output bitfield.

Main function signatures (verified against upstream OVS `v2.17.9` and Linux `v5.15`):

```c
/* lib/conntrack.c, OVS v2.17.9 around line 2060 */
int conntrack_execute(struct conntrack *ct, struct dp_packet_batch *pkt_batch,
                      ovs_be16 dl_type, bool force, bool commit, uint16_t zone,
                      const uint32_t *setmark,
                      const struct ovs_key_ct_labels *setlabel,
                      ovs_be16 tp_src, ovs_be16 tp_dst, const char *helper,
                      const struct nat_action_info_t *nat_action_info,
                      long long now, uint32_t tp_id);

/* net/openvswitch/conntrack.c, Linux v5.15 */
int ovs_ct_execute(struct net *net, struct sk_buff *skb,
                   struct sw_flow_key *key,
                   const struct ovs_conntrack_info *info);

/* the main entry function of the netfilter conntrack engine */
unsigned int nf_conntrack_in(struct sk_buff *skb,
                             const struct nf_hook_state *state);
```

**How an operator masters this.** Five reflexes to develop: (1) when `ct()` does not seem to commit, run `ovs-appctl coverage/show | grep ct` and check whether `ct_lookup_failed` and `ct_commits` are increasing; (2) compare `ovs-appctl dpctl/dump-conntrack` (the datapath layer) with `cat /proc/net/nf_conntrack` (the netfilter layer) on the kernel datapath; the two tables must stay in sync; (3) when the datapath is userspace, use `ovs-appctl dpctl/dump-conntrack` (which calls `conntrack_dump_first` and `conntrack_dump_next` in `lib/conntrack.c`) and do not try to read `/proc/net/nf_conntrack`, because the kernel is not involved; (4) distinguish commit-once from commit-on-every-packet: only `+new` packets need commit, and a rule that commits on `+est` raises load for no benefit; (5) when an entry looks like a zombie, run `conntrack -D` (kernel) or `ovs-appctl dpctl/flush-conntrack` (datapath-agnostic) to remove it.

**Classification.** A runtime function (a C function), not a data structure and not an action token. Family: datapath actions runtime. Two parallel implementations: kernel and userspace.

**Usage workflow.**

```
Step 1: Decide the datapath type.
        ovs-vsctl get bridge br-int datapath_type
        "system" means kernel, "netdev" means userspace.

Step 2: Configure a flow rule that contains the ct() action as usual.
        The operator does not call conntrack_execute directly;
        ovs-vswitchd dispatches it automatically.

Step 3: Observe the runtime behaviour.
        ovs-appctl coverage/show | grep -E 'ct_lookup|ct_commit|ct_invalid'
        ovs-appctl dpctl/dump-conntrack
        ovs-appctl dpif/show

Step 4: When you need to debug semantics, use ofproto/trace to see
        the ct_state that conntrack execute returns for a synthetic packet.
        ovs-appctl ofproto/trace br-int "<flow>"
```

**When something goes wrong.** Common symptom: the `ct(commit)` action does not create an entry, so packets stay `+new`. Investigation procedure: (a) check whether the flow rule actually matches the packet (`ofproto/trace`); (b) check `coverage/show` for an increasing `ct_lookup_failed`; if it is increasing, the packet does not reach conntrack and the issue is in the classifier; (c) if `ct_commits` increases but `dump-conntrack` is empty, suspect that the confirm step at `__nf_conntrack_confirm` is being rolled back due to a conflict; (d) try the `force` flag to force-commit the entry.

**Tightly related to.** `ct_state`, `ct_zone`, `ct_mark`, `ct_label` (the fields that conntrack execute writes into the flow key); `nf_conntrack` (the kernel engine); `recirc_id` (the `ct()` action always implicitly recirculates after conntrack finishes); `dpif` (the dispatch layer that selects the datapath); `ovs-vswitchd` (the host process); Part 9.32 (the classifier that consumes `ct_state`); Part 13.3 (the OVN ACL layer above).

**Real-world incident.** OpenStack and OVN cluster, Q2 2024. A chassis was switched from the kernel datapath to the DPDK datapath in preparation for a NIC upgrade. Right after the switch, every stateful flow that used `allow-related` for FTP dropped its data channel even though the control channel still worked. The engineers traced it: `ovs-appctl dpctl/dump-conntrack` showed the FTP control entry but no expectation for the data channel; `coverage/show` showed `ct_lookup` rising steadily, while `ct_helper_called` stayed at zero. Root cause: the OVS version in use was 2.13. The FTP ALG in `lib/conntrack.c` (userspace) was off by default because it required the build flag `--enable-conntrack-helpers`. The kernel branch `ovs_ct_execute` always has the ALG through `nf_conntrack_helper_ftp`, but the userspace branch did not. Remediation: rebuild OVS with the appropriate flag, upgrade to OVS 2.17 where it is enabled by default, or change the flow rule to use explicit `ct(alg=ftp)` instead of letting OVN auto-resolve. Lesson: kernel and userspace `conntrack execute` are not identical twins. Behavioural parity is the ideal target, but it is not always reality.

**Synthetic exercise.** Mininet topology with three hosts `h1` (10.0.0.1), `h2` (10.0.0.2), `h3` (10.0.0.3) on bridge `s1`.

```bash
# Step 1, install flows that commit only packets coming from h1.
sudo ovs-ofctl add-flow s1 \
  "table=0,priority=100,ip,ct_state=-trk,actions=ct(table=1,zone=42)"
sudo ovs-ofctl add-flow s1 \
  "table=1,ip,nw_src=10.0.0.1,ct_state=+trk+new,actions=ct(commit,zone=42),NORMAL"
sudo ovs-ofctl add-flow s1 \
  "table=1,ip,ct_state=+trk+est,actions=NORMAL"

# Step 2, fabricate a SYN-ACK from h3 with no preceding SYN.
# On h3:
sudo hping3 -A 10.0.0.1 -p 80 -c 1 --syn-ack

# Step 3, check on the s1 host:
sudo ovs-appctl coverage/show | grep -E 'ct_lookup|ct_commit|ct_invalid'
sudo ovs-appctl dpctl/dump-conntrack zone=42

# Expectation: ct_invalid increases because the SYN-ACK matches no +new entry;
# the dump-conntrack table has no entry for h3 to h1; the +trk+est flow does
# not match, so the packet is dropped silently (table 1 has no default).
```

**Common failures and diagnostic signals.**
- Forgotten `commit`. Symptom: `coverage/show ct_lookup` rises, but `ct_commits` does not; every packet stays `+new`; the reverse-direction reply does not match `+est`.
- Forgotten zone. Symptom: state leaks between two tenants; entries fall into the default `zone=0`.
- Datapath mismatch across chassis. Symptom: the same flow rule produces different `ct()` behaviour on a kernel host versus a DPDK host; the operator must compare the ALG version in `lib/conntrack.c`.
- Commit applied to `+est` as well. Symptom: the `ct` CPU load rises, and `ct_commits` grows linearly with the packet rate; commit should run only for `+new`.

**Comparison with other systems.**
- **iptables `-j CT --notrack` and `-m conntrack`:** uses the same `nf_conntrack_in` engine in the kernel but enters through the `PREROUTING` or `OUTPUT` chain rather than through an OF action. Fundamental difference: iptables is a linear chain sequence, while OVS is a multi-table pipeline driven by the controller's intent.
- **VPP NAT44 and NAT64 plugins:** the state machine is hand-written for the VPP node-graph model and does not call netfilter. Same role but a completely different implementation.
- **Cisco ASA stateful inspection:** the state machine is closed firmware and exposes no programmatic interface. Difference: OVS exposes everything through the `ct()` action for the controller.
- **Linux nftables `ct state`:** a higher-level API with the same semantics but compiled differently. Shares the `nf_conntrack` engine with the OVS kernel datapath.
- **Windows Filtering Platform (WFP) ALE_CONNECT layer:** a similar stateful model but expressed through per-flow callouts in the Windows kernel, with no controller-friendly handle equivalent to the `ct()` action.

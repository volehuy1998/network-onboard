## 17.1 - Why OVN Exists: The Original Problem and the Path to a Solution

### From OpenFlow to OVN, a brief historical narrative

To understand OVN, we have to go back to **2008**, when **Nick McKeown** and seven co-authors (Tom Anderson, Hari Balakrishnan, Guru Parulkar, Larry Peterson, Jennifer Rexford, Scott Shenker, Jonathan Turner) published "OpenFlow: Enabling Innovation in Campus Networks" in ACM SIGCOMM Computer Communications Review (Vol. 38, No. 2, pp. 69 to 74). The paper, which now has more than 9,000 citations (as of 2025), laid the foundation for the concept of **Software-Defined Networking (SDN)**: separate the control plane from the data plane, and allow switches to be programmed through a standard protocol.

OpenFlow grew out of the PhD research of **Martin Casado** at Stanford, the Ethane project (published at SIGCOMM 2007, with research starting in 2006), in which the idea of "a simple switch with a programmable flow table" was first realised. Casado, together with McKeown and Shenker, co-founded **Nicira Networks** in 2007. At Nicira, two key engineers, **Ben Pfaff** and **Justin Pettit**, built **Open vSwitch (OVS)**, an open-source OpenFlow-based virtual switch that became the backbone of network virtualisation. OVS was merged into Linux kernel 3.3 (2012) and became an indispensable component in cloud platforms.

In 2012, VMware acquired Nicira for 1.26 billion USD. Pfaff and Pettit continued to develop OVS at VMware. As Pfaff explained in a USENIX 2025 interview: *"One reason OVN succeeded is that our team had a history with network virtualisation going back to the early days of Open vSwitch. At Nicira, we wrote several generations of network virtualiser. Each time we learned what worked and what to avoid."*

### Before OVN: Neutron ML2/OVS and the limits of an agent-based architecture

Before OVN, OpenStack Neutron used the **ML2 (Modular Layer 2) plugin** with the **OVS mechanism driver** plus a set of Python agents running on every compute node:

```
┌─────────────────────────────────────────────────────────┐
│                  Neutron Server                          │
│              (ML2 Plugin + OVS MechDriver)               │
└────────────────────┬────────────────────────────────────┘
                     │ RabbitMQ (AMQP RPC)
        ┌────────────┼────────────────┐
        ▼            ▼                ▼
┌──────────────┐ ┌──────────────┐ ┌──────────────┐
│ Compute #1   │ │ Compute #2   │ │ Network Node │
│ neutron-ovs  │ │ neutron-ovs  │ │ neutron-l3   │
│   -agent     │ │   -agent     │ │ neutron-dhcp │
│ br-int (NORM)│ │ br-int (NORM)│ │ neutron-meta │
│ br-tun       │ │ br-tun       │ │              │
│ br-ex/br-vlan│ │ br-ex/br-vlan│ └──────────────┘
└──────────────┘ └──────────────┘
```

Each node ran **neutron-openvswitch-agent**, a Python agent that configured OVS bridges and flows. The agent used three bridges: **br-int** (the integration bridge, which used the NORMAL action to delegate all L2 forwarding to OVS built-in MAC learning), **br-tun** (the tunnel bridge for VXLAN and GRE), and **br-ex / br-vlan** (the provider bridge, connecting the physical network through patch port pairs).

The core problem was the NORMAL action on br-int. When br-int used NORMAL, OVS learned MAC addresses on its own like a physical switch, completely outside the control of the CMS. The direct consequences were significant: the CMS had no visibility into the MAC table and therefore could not control or verify forwarding decisions; unknown unicast flooded out every port on the bridge and wasted bandwidth; security groups had to be implemented through a separate Linux bridge plus iptables for every VM (because OVS conntrack support was immature at the time); and RabbitMQ became a single point of failure since every topology change had to go through the message broker. Launchpad bug #1866445 captured the symptom clearly: "br-int on a compute node does not learn the MAC of a VM on another compute node, VM-to-VM performance degrades linearly", a direct consequence of MAC learning being limited to a single local bridge.

### OVN was announced on 13 January 2015

On 13 January 2015, **Justin Pettit, Ben Pfaff, Chris Wright** (Red Hat CTO), **and Madhu Venugopal** announced OVN on the Network Heresy blog: *"Today we are excited to announce Open Virtual Network (OVN), a new project that brings virtual networking to the OVS user community."*

**Russell Bryant**, a Red Hat engineer, quickly became the main driver of OVN inside OpenStack: *"The timing could not have been better. I dove in with a goal of figuring out whether OVN could be a promising next generation of Open vSwitch integration for OpenStack."* Bryant wrote the first OVN-OpenStack integration code and published a series of technical blog posts at blog.russellbryant.net.

The first stable release shipped with **OVS 2.6.0** (September 2016). Neutron integration through networking-ovn began with **OpenStack Newton** (October 2016). In 2019, OVN moved into a separate repository at github.com/ovn-org/ovn, an important milestone marking the maturity of the project.

### OVN architecture, a hybrid centralised plus distributed model with two databases

```
┌───────────────────────────────────────────────────────┐
│            CMS (Neutron / Kubernetes)                   │
└──────────────────┬────────────────────────────────────┘
                   │ OVSDB Protocol
                   ▼
┌───────────────────────────────────────────────────────┐
│           OVN Northbound Database (NB DB)               │
│ Logical_Switch │ Logical_Router │ ACL │ DHCP_Options    │
│ Logical_Switch_Port │ Logical_Router_Port │ ...         │
└──────────────────┬────────────────────────────────────┘
                   │ ovn-northd compiles NB to SB
                   ▼
┌───────────────────────────────────────────────────────┐
│           OVN Southbound Database (SB DB)               │
│ Logical_Flow │ Datapath_Binding │ Port_Binding          │
│ Chassis │ Multicast_Group │ FDB │ MAC_Binding           │
└──────┬──────────────┬──────────────┬──────────────────┘
       ▼              ▼              ▼
  ovn-controller  ovn-controller  ovn-controller
  (Chassis 1)     (Chassis 2)     (Chassis 3)
       ↓              ↓              ↓
  OpenFlow → br-int  OF → br-int  OF → br-int
```

**NB DB** is the interface between the CMS and OVN. Its schema uses networking concepts familiar to a cloud operator: `Logical_Switch`, `Logical_Switch_Port` (key columns are `name`, `type`, `addresses`, `port_security`, `enabled`), `Logical_Router`, and `ACL`. When Neutron creates a network or a port, it writes directly into NB DB through the OVSDB protocol, with no intermediate message broker.

**SB DB** has a completely different schema and represents the network as **logical flow** (match-action rules using logical concepts). The main tables are `Logical_Flow`, `Datapath_Binding`, `Port_Binding`, `Chassis`, `Multicast_Group`, `FDB`, and `MAC_Binding`. The separation between NB DB (network intent) and SB DB (network realisation) is a core architectural decision, allowing the CMS to speak in abstract terms while the system underneath translates to concrete OpenFlow.

**ovn-northd** is the central daemon that reads NB DB and translates it into SB DB. Each logical switch or logical router is converted into a logical datapath together with its corresponding set of logical flows. This is a deployment-wide computation that runs in a single instance (or an active-standby pair for HA).

**ovn-controller** runs on every chassis, reads logical flows from SB DB, then translates them into OpenFlow flows and programs them into OVS. Bryant described the division of work like this: *"OVN breaks this problem up: first calculates logical flow, then distributes to ovn-controller on each node which converts to physical flow."* The computation is node-specific and fully distributed: each chassis only installs flows related to ports it hosts.

### Why br-int no longer uses the NORMAL action

The deepest architectural difference between ML2/OVS and OVN lies in how br-int operates. OVN places br-int in **fail-secure** mode: the flow table starts empty, and every forwarding decision must be programmed explicitly by ovn-controller. From the OVN tutorial: *"It's common for an OVN logical switch to know all the MAC addresses supported by its logical ports... That's why there's no logic here for MAC learning or flooding."*

Security groups no longer need a Linux bridge plus iptables, OVN uses **kernel conntrack directly** from OVS (the ct action in OpenFlow), which simplifies the data path significantly. RabbitMQ is no longer needed for network topology changes, all communication between the central daemons and the chassis goes through OVSDB. Red Hat OpenStack Platform 16.0 (2020) marked the official transition: ML2/OVN became the default mechanism driver, replacing ML2/OVS that had served for many years before.

> **Key idea.** OVN knows every MAC address in advance (from the `addresses` column on Logical_Switch_Port) and programs explicit flows for each MAC. When this document discusses FDB learning later, it covers the **exception case** in which OVN does NOT know the MAC in advance, specifically on a port with `port_security` disabled and addresses containing the keyword `"unknown"`. This is the core architectural difference between OVN (explicit programming) and a physical switch (dynamic learning).

### The tunnel key system, the language for identifying logical objects

Every logical object in OVN, that is, logical switch, logical router, logical port, and multicast group, is assigned a unique integer called the **tunnel key**. This system is the backbone of the mechanism that carries packets between chassis through Geneve tunnels, and it is also how ovn-controller identifies objects when programming OpenFlow rules on br-int.

**Datapath tunnel key (dp_key)** is a 24-bit number (1 to 16,777,215) assigned to each logical switch or logical router, stored in `Datapath_Binding.tunnel_key`. When a packet leaves a chassis through a Geneve tunnel toward another chassis, the dp_key is encoded into the **VNI field** (Virtual Network Identifier) in the Geneve header. The VNI field is exactly 24 bits, so the dp_key maps 1:1 into the VNI. On the OpenFlow side, the dp_key maps to the **metadata** register: any flow matching `metadata=0x752` is matching the logical switch with dp_key=1874 (0x752 hex). This is how ovn-controller distinguishes which datapath a packet belongs to when br-int serves multiple logical switches at once.

**Port tunnel key (port_key)** is a 15-bit number (1 to 32,767) assigned to each logical port within a datapath, stored in `Port_Binding.tunnel_key`. Each datapath has its own namespace: port_key=3 on logical switch A and port_key=3 on logical switch B are two completely different ports, because their differing dp_keys disambiguate them. In the Geneve header, port_key does not live in the VNI but is encoded into a **TLV option** (class 0x0102, type 0x80). This is supplementary metadata that Geneve supports (GRE and VXLAN do not have an equivalent mechanism). On the OpenFlow side, port_key maps to **reg14** (MFF_LOG_INPORT, logical ingress port) and **reg15** (MFF_LOG_OUTPORT, logical egress port).

**Multicast group tunnel key** lies in the high range 32,768 to 65,535, assigned to multicast groups and stored in `Multicast_Group.tunnel_key`. This range is completely separated from the port_key range (1 to 32,767), so they never collide. `_MC_flood` always receives tunnel_key 32768, and `_MC_unknown` receives 32769. These two values are fixed constants hardcoded in `lib/mcast-group-index.h`.

A key point for understanding the sections that follow: **even though a localnet port does not use a Geneve tunnel** (traffic traverses a patch port to br-provider and out through the physical NIC), it is still assigned a tunnel_key in SB DB. The reason is that ovn-controller needs the port_key to identify the localnet port inside the logical pipeline, and matching reg14 / reg15 for ACL, QoS, and FDB lookup all depends on port_key. Without a tunnel_key, the localnet port would be "invisible" in the pipeline. Similarly, the FDB table (Section 17.4) stores `dp_key` and `port_key` directly instead of the logical port name. Numan Siddique's commit message explains why: *"This makes it easier for ovn-controller to program the OF rules"*, because OpenFlow rules match on integers (register values), not on strings.

> **Key idea.** The three kinds of tunnel key (24-bit dp_key, 15-bit port_key, 16-bit multicast key) form the internal addressing system of OVN. When a packet traverses Geneve: dp_key is encoded into the VNI, and port_key is encoded into a TLV option. When the packet is local (br-int): dp_key sits in metadata, and port_key sits in reg14 / reg15. Understanding this system is a foundation for reading OpenFlow dumps and for debugging with ovn-trace.

## 17.2 - Localnet Port: The Bridge Between the Virtual World and a Physical VLAN

### The problem of connecting an overlay to a physical VLAN

OVN was designed for overlay networks (Geneve and VXLAN tunnels), but production reality always requires connecting VMs directly to a VLAN-based physical network (for example, VLAN 3808 in the case study in Section 17.6).

In OpenStack this kind of network is called a **provider network**: a network created by the administrator, connected directly to the physical network infrastructure (usually a VLAN segment). The name "provider" comes from the fact that the infrastructure provider supplies this connection, in contrast to the tenant or self-service network (an overlay that the user creates over Geneve or VXLAN tunnels). A provider network is not necessarily the public internet. In the case study, VLAN 3808 (100.64.0.0/28) is an internal transit network that connects the VM to a Fornix gateway. The key technical distinction is that a provider network uses a localnet port on OVN and traffic traverses the physical NIC, whereas a tenant network traverses a Geneve tunnel between chassis.

Before OVN, Neutron ML2/OVS used br-ex / br-vlan bridges with patch port pairs: the agent configured OpenFlow rules to swap the internal VLAN tag with the physical VLAN tag. Patch ports existed on every node regardless of whether a VM on that node used the network, which wasted resources on large deployments.

### Localnet port design: a "special port" representing the entire physical network

OVN solves the physical connectivity problem with a **localnet port type**, a special kind of Logical_Switch_Port:

```
ovn-nbctl lsp-add provider-ls ln-provider
ovn-nbctl lsp-set-type ln-provider localnet
ovn-nbctl lsp-set-addresses ln-provider unknown
ovn-nbctl lsp-set-options ln-provider network_name=physnet1
ovn-nbctl set Logical_Switch_port ln-provider tag=3808
```

> **Key idea.** A localnet port **does NOT represent a single device** (a single MAC). It represents the **entire physical network segment**. This is the core distinction from a VIF port: a VIF port maps 1:1 to a VM tap interface, whereas a localnet port maps 1:N to every device on the physical VLAN.

The connection mechanism is based on **bridge mapping**, configured per chassis to specify which physical bridge corresponds to which network name:

```
Physical Network (VLAN 3808)
         |
    +----+---------+
    | br-provider  |  <- OVS bridge connecting the physical NIC (bond3)
    +----+---------+
         | patch port (ovn-controller creates it automatically)
    +----+---------+
    |   br-int     |
    |              |
    | VM .12 port  |  <- VIF port: LSP type="" representing a VM interface
    | VM .xx port  |  <- VIF port: LSP type="" representing a VM interface
    | localnet port|  <- represents the ENTIRE VLAN 3808
    +--------------+
```

Bridge mapping configuration: `ovs-vsctl set open . external-ids:ovn-bridge-mappings=physnet1:br-provider`

When ovn-controller detects that the node hosts at least one VM on this provider network, it automatically creates a patch port between br-int and br-provider. Conversely, if the node hosts no VM on that network, ovn-controller does not create the patch port and releases the corresponding resources. This is an improvement over the old ML2/OVS, where the agent created patch ports on every node regardless of whether a VM was present.

In SB DB, every Logical_Switch_Port (including localnet LSPs) has a corresponding row in the Port_Binding table. For a localnet LSP, the Port_Binding has a distinctive characteristic: **`chassis: []`** (it is not bound to any specific chassis), because the localnet port "exists everywhere" through bridge mapping on each chassis:

```
_uuid       : 8427506e-...
logical_port: "provnet-e4abf6df-..."
mac         : [unknown]
options     : {network_name=physnet1}
tunnel_key  : 1
type        : localnet
chassis     : []          <- NOT bound to any chassis!
```

### "provnet-" versus "localnet", two names at two different layers

When you read the output of `ovn-sbctl list port_binding`, you will see a localnet port with `logical_port: "provnet-e4abf6df-..."` but `type: localnet`. The two different prefixes confuse newcomers, but in fact they operate at two completely different layers.

**"localnet"** is the OVN port type, a concept that belongs to the OVN core and is defined in ovn-sb(5). When `Port_Binding.type = "localnet"`, ovn-controller on each chassis understands that the port represents a connection to the local physical network through bridge mapping. OVN itself only inspects the type and `options:network_name`; it does not care what the logical_port name is.

**"provnet-"** is a Neutron naming convention, a naming pattern produced by the Neutron ML2/OVN driver. When Neutron creates a provider network, the driver generates a localnet LSP with a name following the pattern `provnet-<network-segment-UUID>`. This name is purely administrative metadata. If you use OVN standalone (without OpenStack), the localnet port can be named any string.

There is no distinction between "provnet" and "localnet" at the OVN layer. The relationship is between CMS naming and OVN type, similar to how Neutron names a VIF port `<port-UUID>` while OVN only cares about `type=""` and chassis binding. When debugging with `ovn-sbctl`, if you see a `logical_port` beginning with "provnet-", that is certainly a localnet port on an OpenStack system. On Kubernetes (OVN-Kubernetes), the localnet port may follow a completely different naming convention.

### Common misconceptions about the localnet port

> **Misconception:** "A localnet port behaves like a bridge port on a physical switch, learning MACs and forwarding frames based on a MAC table."
>
> **Reality:** A localnet port in OVN does not learn MACs (unless `localnet_learn_fdb=true` is set, which is off by default). It has no MAC of its own, does not participate in the default MAC learning pipeline, and its addresses field is the keyword `"unknown"`, which means it receives every frame whose destination MAC does not match any known port. This is a catch-all behaviour, not learning.

> **Misconception:** "A localnet port is bound to a specific chassis like a VIF port."
>
> **Reality:** The Port_Binding for a localnet port has `chassis: []`. It is not bound to any chassis. ovn-controller on each chassis with the corresponding bridge mapping creates a patch port on demand. The localnet port "exists everywhere" through bridge mapping, not through chassis binding.

### Why addresses contain "unknown", a key design decision

OVN does not know in advance which MACs exist on the physical network: hundreds of servers, routers, and firewalls that OVN does not manage. `"unknown"` is a special keyword in OVN: a port whose address is "unknown" receives every packet whose destination MAC does not match any known port. It acts as a catch-all for unknown unicast traffic, and a port carrying this keyword is added to the multicast group `_MC_unknown`, the dedicated group for unknown unicast forwarding that the next section will analyse in detail.

ovn-nb(5): *"Each bridged logical switch has one or more localnet ports, which have only one special address 'unknown'."*

### Chassis, Claim, and Multichassis, the port ownership mechanism in OVN

Section 17.2 has just shown that a localnet port has `chassis: []` in Port_Binding. But what about a VIF port (a VM)? The answer is that a VIF port is **claimed** by a specific chassis. Understanding the claim mechanism and the multichassis state is a prerequisite for reading the case study in Section 17.6, where the entire FDB poisoning mechanism unfolds inside a "multichassis window" lasting only a few seconds.

A **Chassis** in OVN is a physical compute node, each hypervisor host where VMs run. When ovn-controller starts on a node, it registers itself into SB DB by creating a row in the `Chassis` table with `name` set to the hostname of the node (for example, `vhhl1c2wc1tsn01`) and `system_id` set to a unique UUID. This table is the foundation that lets OVN know which compute node is up and capable of receiving port bindings.

> **Misconception:** "A Chassis is the physical NIC or the bond interface on the server."
>
> **Reality:** A Chassis is the entire compute node (the hypervisor host), not an interface. A node with two or with ten NICs is still a single chassis. Physical interfaces are exposed through `bridge mappings` in OVS `external-ids`, completely separate from the Chassis concept.

**Claim** is the action in which ovn-controller writes its chassis name into the `Port_Binding.chassis` field for an LSP. The mechanism works as follows. When a VM is created on a node, QEMU creates a tap interface with `external-ids:iface-id` equal to the UUID of the Neutron port. ovn-controller runs continuously, polling the local OVS `Interface` table. When it detects an interface whose `iface-id` matches an LSP name in SB DB, it calls the function `claim_lport()` in `controller/binding.c`, writing its chassis into `Port_Binding.chassis`. From that moment on, the port is considered "owned" by this chassis: only that chassis will program OpenFlow rules for the port, and only that chassis will receive traffic destined for the port from the Geneve tunnel.

A real log when the claim happens on chassis `vhhl1c2wc1tsn01`:

```
[ovn-controller] INFO binding  Claiming lport 0a17b4f8-736d-4bf5-ba1e-335d17cb5973 for this chassis.
[ovn-controller] INFO binding  0a17b4f8-736d-4bf5-ba1e-335d17cb5973: Claiming fa:16:3e:04:74:74 100.64.0.12
[ovn-controller] INFO binding  0a17b4f8-736d-4bf5-ba1e-335d17cb5973: Claiming unknown
```

The line "Claiming unknown" confirms that the port has `addresses` containing the keyword `"unknown"`, the second condition for FDB learning to activate (analysed in Section 17.4).

**Multichassis** is a temporary state in which a port is simultaneously bound to two chassis: the source chassis (where the VM is currently running) and the destination chassis (where the VM is about to be migrated). This state only occurs inside the live migration window. The implementation technique relies on two distinct fields in Port_Binding: `chassis` stores the primary chassis (the current owner) and `additional_chassis` stores the secondary chassis (where the VM is being moved to). When Neutron live-migrates, it sets `options:requested-chassis = "chassisA,chassisB"` on the LSP. ovn-northd splits this string into `PB.requested_chassis = chassisA` and `PB.requested_additional_chassis = [chassisB]`. On chassisB, the function `lport_can_bind_on_this_chassis()` recognises itself in `requested_additional_chassis` and returns the enum `CAN_BIND_AS_ADDITIONAL` instead of `CAN_BIND_AS_MAIN`. `claim_lport()` updates `PB.additional_chassis` with chassisB, and the port is now multichassis.

When `PB.additional_chassis` is non-empty, the function `shash_is_empty(&ld->multichassis_ports)` in `controller/physical.c` returns `false`. This triggers `enforce_tunneling_for_multichassis_ports()`, which installs an OpenFlow rule in `OFTABLE_REMOTE_OUTPUT` forcing every packet destined to that port through the Geneve tunnel between the two chassis, even when source and destination are on the same physical VLAN. This is the first triggering condition that leads to FDB poisoning in the case study: the tunnel forces traffic through Geneve, Geneve adds 58 bytes of overhead, the packet exceeds MTU, and an ICMP error is generated with the bug that swaps MAC without swapping inport.

> **Key idea.** Multichassis is not a steady-state HA configuration. It is a temporary state inside the live migration window, typically lasting 5 to 10 seconds. After migration completes, `PB.additional_chassis` returns to empty and the tunnel enforcement mechanism disappears. Only nova live-migrate creates this state; every other Nova action (boot, stop, cold migration, rebuild) never sets `requested-chassis` as a comma-separated list.

Real-time inspection of the multichassis state on OVN central:

```bash
[controller]$ ovn-sbctl --columns=logical_port,chassis,additional_chassis list Port_Binding
```

While migration is in progress, the `additional_chassis` column will contain a UUID instead of being empty on the port of the VM being migrated.

### localnet_learn_fdb, extending FDB learning to localnet ports

Originally the localnet port was excluded from FDB learning entirely for scale reasons: the physical network can carry hundreds of MACs, and learning all of them would quickly exhaust FDB table entries. However, with the rise of hardware offload (NVIDIA / Mellanox SmartNIC) and the demand to reduce flooding on small provider networks, Ales Musil (amusil@redhat.com) and Numan Siddique added the `localnet_learn_fdb` option in OVN v22.09.0 (commit `93514df0d4c8fe7986dc5f287d7011f420d1be6d`, Red Hat Bugzilla #2070529). The setting is opt-in and off by default: `ovn-nbctl set Logical_Switch_port <localnet-lsp> options:localnet_learn_fdb=true`. How this feature interacts with FDB learning is analysed in detail in Section 17.4.

## 17.3 - MC_FLOOD and MC_UNKNOWN: Broadcast and Unknown Destinations in a Distributed Control Plane

### The packet-flooding problem when a logical switch spans many chassis

On a physical switch, broadcast and unknown unicast flooding are simple: the ASIC chip replicates the frame to every port in the same VLAN within a few nanoseconds. In OVN the problem is more complex because a logical switch spans many chassis, and each chassis only knows the local ports it hosts. When a VM sends a broadcast frame, the local ovn-controller needs to know which chassis the frame must reach and through which port on each chassis.

OVN solves this with **pseudo-multicast groups** in SB DB, stored in the `Multicast_Group` table. Each group contains a list of member Port_Bindings, and when a packet has its `outport` set to the group name, ovn-controller replicates the packet to every member, including ports on remote chassis through Geneve tunnels.

### Multicast_Group schema (ovn-sb(5))

| Column | Type | Description |
|------------|-------------------|-------------------------------------------|
| datapath | Datapath_Binding | Logical switch containing the group |
| tunnel_key | int [32768-65535] | Unique ID (does not collide with logical port IDs) |
| name | string | Group name (unique within the datapath) |
| ports | set of Port_Binding | List of member ports |

Source code (`lib/mcast-group-index.h`):
```c
enum ovn_mcast_tunnel_keys {
    OVN_MCAST_FLOOD_TUNNEL_KEY = 32768,    // _MC_flood
    OVN_MCAST_UNKNOWN_TUNNEL_KEY,           // _MC_unknown (32769)
    OVN_MCAST_MROUTER_FLOOD_TUNNEL_KEY,     // 32770
    ...
};
```

### Three main multicast groups and their members

| Group | Tunnel Key | Members | Purpose |
|----------------|-----------|----------------------------------------|---------------------------------------|
| `_MC_flood` | 32768 | All enabled ports (VIF, router, localnet) | Broadcast and multicast (ARP request, and similar) |
| `_MC_flood_l2` | derived | All ports except router ports | L2 multicast (non-IP, non-ARP) |
| `_MC_unknown` | 32769 | Only ports with addresses="unknown" | Unknown unicast forwarding |

> **Key idea.** `_MC_unknown` is created **conditionally**, only when at least one port on the logical switch has `"unknown"` in its addresses. On an overlay-only switch (every MAC is already known from LSP addresses), MC_UNKNOWN does not exist and unknown unicast is dropped. This is why a provider network (which has a localnet port with addresses="unknown") always has MC_UNKNOWN, while a tenant overlay network does not.

The difference between MC_FLOOD and MC_UNKNOWN is essential for understanding why ARP keeps working while unicast traffic is affected by FDB poisoning (the case study in Section 17.6). MC_FLOOD receives every broadcast and multicast frame, including ARP Requests, and floods them to all enabled ports. MC_UNKNOWN only receives unknown unicast frames and floods to a narrower set of ports: only those with "unknown" addresses.

### Pipeline processing, a packet walk-through in detail

```
           Logical Switch Ingress Pipeline
           ===============================
  Table 0-1: ls_in_port_sec_l2 / ls_in_port_sec_ip
       |
       v
  Table N: ls_in_lookup_fdb  (*)
     -> reg0[11] = lookup_fdb(inport, eth.src)   [check FDB]
       |
       v
  Table N+1: ls_in_put_fdb  (*)
     -> if reg0[11]==0: put_fdb(inport, eth.src)  [LEARN MAC if not present]
       |
       v
  ... ACL, QoS, DHCP, DNS, ARP responder, ...
       |
       v
  Table M: ls_in_l2_lkup (L2 Destination Lookup)  (*)
     Priority 72: eth.mcast && arp -> outport = _MC_flood     [ARP broadcast]
     Priority 70: eth.mcast -> outport = _MC_flood_l2         [L2 multicast]
     Priority 50: eth.dst==<known_mac> -> outport=<port>      [Static known]
     Priority  0: outport = get_fdb(eth.dst); next            [FDB lookup]
       |
       v
  Table M+1: ls_in_l2_unknown  (*)
     Priority 50: outport=="none" -> outport=_MC_unknown       [Flood unknown]
     Priority  0: output                                       [FDB found, forward]

  (*) Note: table numbers vary across OVN versions. For example, on OVN v21.06,
  ls_in_lookup_fdb is approximately table 2 and ls_in_put_fdb is approximately
  table 3. On newer versions (v23.09 and later), positions may shift due to
  added stages. Verify with: ovn-sbctl lflow-list <datapath> | grep ls_in_lookup_fdb
```

The unknown-unicast processing flow (no FDB entry) goes through two key stages. At `ls_in_l2_lkup` (priority 0), the action `get_fdb(eth.dst)` looks up the destination MAC in the FDB table. It finds nothing, so outport is set to "none" (the value 0). At `ls_in_l2_unknown` (priority 50), the match `outport == "none"` succeeds, and the action sets `outport = "_MC_unknown"` then executes `output`. OVN floods a copy of the packet to every member of the MC_UNKNOWN group, including the localnet port. When the packet reaches the localnet port, it traverses the patch port into br-provider, then exits through the physical NIC (bond3) onto the physical VLAN.

The known-unicast flow (with an FDB entry) differs in the first step. At `ls_in_l2_lkup` (priority 0), `get_fdb(eth.dst)` finds an entry in the FDB table and sets outport to the corresponding port_key. At `ls_in_l2_unknown` (priority 0), outport already has a value other than "none", so the flood condition does not match and the packet goes directly to the port specified by FDB.

Below is real ovn-trace output illustrating the unknown-unicast flow (GitHub Issue #244):

```
ingress(dp="non_fdp", inport="07188e")
 0. ls_in_port_sec_l2: inport == "07188e", priority 50 -> next;
 3. ls_in_lookup_fdb:  inport == "07188e", priority 100
    reg0[11] = lookup_fdb(inport, eth.src); next;
24. ls_in_l2_lkup: 1, priority 0
    outport = get_fdb(eth.dst); next;
25. ls_in_l2_unknown: outport == "none", priority 50
    outport = "_MC_unknown"; output;

multicast(dp="non_fdp", mcgroup="_MC_unknown")
  -> egress: provnet-40617a (type "localnet")   <- out to physical network
  -> egress: RTR3-1-port-128, 3b461e, 877021, ...
```

### Why MC_UNKNOWN is indispensable for provider networks

Without MC_UNKNOWN, the provider network does not work. OVN does not know the MAC of the physical gateway (100.64.0.1 in the case study) in advance because the gateway is not a VM managed by Neutron, and its MAC address does not appear in any LSP addresses field. When a VM sends a frame to the gateway, the destination MAC does not match any priority-50 static flow in `ls_in_l2_lkup`. Without an "unknown" forwarding mechanism, `get_fdb` returns outport="none" and there is no MC_UNKNOWN to flood to, so the frame is dropped. MC_UNKNOWN serves as the only door through which traffic can leave the OVN virtual world to the physical network when the destination MAC is not known in advance.

> **Misconception:** "MC_UNKNOWN is like broadcast, it floods to every port on the logical switch."
>
> **Reality:** MC_UNKNOWN floods only to ports with `"unknown"` in their addresses. On a logical switch with 50 VIF ports but only 2 ports with addresses="unknown" (the localnet port and one VNF port), MC_UNKNOWN has only 2 members, not 50. Broadcasts (such as an ARP request) traverse `_MC_flood` at a higher priority (72 > 0) and flood to every enabled port.

> **Misconception:** "If FDB does not find the MAC, the packet is dropped."
>
> **Reality:** When `get_fdb` returns outport="none", the packet enters `ls_in_l2_unknown` (priority 50) and is redirected to `_MC_unknown`, flooding to every port with "unknown" addresses. The packet is dropped only when MC_UNKNOWN does not exist (an overlay-only switch with no port carrying "unknown").

MC_UNKNOWN solves the problem of "delivering a packet to its destination when the MAC is not yet known" through flooding. However, flooding has a bandwidth cost: every unknown unicast frame is replicated to every member of MC_UNKNOWN instead of going only to the right localnet port. When a provider network has hundreds of VMs with port_security disabled (all carrying "unknown" addresses), every frame to the physical gateway is replicated hundreds of times. This is precisely the problem that motivated the introduction of FDB learning, the dynamic MAC caching mechanism that lets OVN "remember" MACs it has seen and forward directly without flooding again.

## 17.4 - FDB table và dynamic MAC learning: từ bài toán flooding đến caching

### Trước FDB: flooding là giải pháp duy nhất

Trước OVN 21.06, OVN chỉ biết MAC từ hai nguồn: thông tin static từ cột `addresses` trên Logical_Switch_Port (Neutron set khi tạo port), và keyword `"unknown"` (nghĩa là "nhận mọi thứ không biết"). Không có cơ chế dynamic learning nào.

Hệ quả trực tiếp trên provider network có nhiều thiết bị vật lý: mỗi khi VM gửi frame đến MAC của gateway vật lý, MAC mà OVN không biết trước, giải pháp duy nhất là flood đến tất cả thành viên MC_UNKNOWN. Nếu 100 VM trên cùng provider network đều có port_security disabled (tất cả đều mang "unknown" addresses và là thành viên MC_UNKNOWN), mỗi frame đến thiết bị vật lý bị nhân bản gửi đến 100 port, dù chỉ cần đến localnet port (ra physical network). Bandwidth lãng phí tăng tuyến tính theo số VM.

FDB learning ra đời để giải quyết chính xác vấn đề này: sau lần flood đầu tiên, khi gateway vật lý gửi frame ngược lại (ví dụ ARP reply), ovn-controller ghi nhận "MAC của gateway đi vào qua localnet port" vào FDB table. Từ lần sau, `get_fdb` tìm thấy entry và forward trực tiếp đến localnet port, không cần flood. Nếu không có FDB, hệ thống vẫn hoạt động đúng (gói tin đến đích thông qua flood) nhưng hiệu năng kém trên hạ tầng lớn.

### FDB table schema

```json
"FDB": {
    "columns": {
        "mac":      {"type": "string"},
        "dp_key":   {"type": {"key": {"type": "integer", "minInteger": 1, "maxInteger": 16777215}}},
        "port_key": {"type": {"key": {"type": "integer", "minInteger": 1, "maxInteger": 16777215}}},
        "timestamp": {"type": "integer"}
    },
    "indexes": [["mac", "dp_key"]],
    "isRoot": true
}
```

| Column | Type | Mô tả |
|-----------|---------|------------------------------------------------|
| mac | string | MAC address đã học (vd: "fa:16:3e:35:a7:73") |
| dp_key | int | Datapath tunnel key (logical switch ID) |
| port_key | int | Port tunnel key (logical port ID) |
| timestamp | int | Thời điểm cập nhật (thêm sau cho aging) |

Index `[mac, dp_key]` đảm bảo primary key là cặp (MAC, datapath): một MAC trên cùng logical switch chỉ có đúng một FDB entry.

Thiết kế khác biệt quan trọng so với MAC_Binding (mục 17.5): FDB lưu tunnel key (dp_key, port_key) trực tiếp thay vì logical port name. Như đã phân tích ở mục 17.1, dp_key map sang metadata register và port_key map sang reg14/reg15 trong OpenFlow, lưu dp_key và port_key trực tiếp giúp ovn-controller bỏ qua bước chuyển đổi tên-sang-số khi program flow, giảm chi phí xử lý khi có hàng nghìn entry. Commit message của Numan Siddique giải thích rõ: *"Unlike MAC_Binding Southbound table, FDB table stores the datapath tunnel key and port key. This makes it easier for ovn-controller to program the OF rules."*

Giới hạn: `MAX_FDB_ENTRIES = 1000`.

### Ba action cốt lõi và cách chúng hoạt động

**`put_fdb(inport, eth.src)`** ghi entry mới hoặc cập nhật entry hiện có. Action này translate thành `controller` action, gửi PACKET_IN đến ovn-controller. Tại đó, `pinctrl_handle_put_fdb()` trong `controller/pinctrl.c` extract dp_key từ metadata, port_key từ reg14 (MFF_LOG_INPORT), và eth.src từ packet header, tạo struct FDB entry lưu vào in-memory hash map `put_fdbs`. Lần OVSDB transaction tiếp theo, `run_put_fdbs()` iterate map và insert/update từng row vào `Southbound FDB` table qua OVSDB IDL.

**`get_fdb(eth.dst)`** tìm port_key cho MAC đích. Action thực hiện resubmit đến OpenFlow table 71 (OFTABLE_GET_FDB). ovn-controller cài sẵn flow trong table này: match `metadata=dp_key, dl_dst=mac` thì load port_key vào MFF_LOG_OUTPORT (reg15). Nếu không match (MAC chưa có trong FDB), default flow load 0, tương đương outport="none".

**`lookup_fdb(inport, eth.src)`** kiểm tra entry có tồn tại không. Action resubmit đến OpenFlow table 72 (OFTABLE_LOOKUP_FDB). Match `metadata=dp_key, dl_src=mac, reg14=port_key` thì load 1 vào reg0[11] (nghĩa là "đã có entry, không cần learning"). Hành vi đặc biệt: khi `flags.localnet=1` (gói tin đến từ localnet port), lookup_fdb trả về 1 nếu entry tồn tại VÀ port type hiện tại là VIF. Kết quả: put_fdb bị skip, bảo vệ VIF entry khỏi bị localnet overwrite, cơ chế protection sẽ phân tích chi tiết ở phần sau của mục này.

### Điều kiện để FDB learning kích hoạt, ba điều kiện AND

> **Chủ đề trọng tâm:** Một port đủ điều kiện FDB learning khi thỏa **CẢ BA** điều kiện AND dưới đây. Thiếu bất kỳ điều kiện nào, FDB learning không kích hoạt trên port đó.

Điều kiện đầu tiên: **port_security phải rỗng/disabled** (`!op->n_ps_addrs`). Điều kiện thứ hai: **addresses phải chứa "unknown"** (`op->has_unknown`). Điều kiện thứ ba: **type phải là "" (VIF thường)** (`!strcmp(op->nbsp->type, "")`) HOẶC là localnet port với `localnet_learn_fdb=true`.

Source code (`northd/northd.c`, hàm `build_lswitch_learn_fdb_op()`):
```c
if (!op->n_ps_addrs && op->has_unknown &&
    (!strcmp(op->nbsp->type, "") ||
     (lsp_is_localnet(op->nbsp) && localnet_can_learn_mac(op->nbsp)))) {
```

### Đánh giá triển khai: khi nào FDB learning có lợi, khi nào gây hại

Ba điều kiện AND không phải rào cản tùy ý, mỗi điều kiện là một cổng bảo vệ (gate) được thiết kế có chủ đích, phản ánh bài toán mà FDB learning giải quyết và rủi ro mà nó tạo ra.

Khi port_security enabled, OVN đã biết chính xác MAC nào hợp lệ trên port đó, danh sách MAC nằm trong cột `port_security` của LSP, ovn-northd tạo flow tĩnh match `eth.src == <allowed_mac>` tại stage `ls_in_port_sec_l2`. Mọi frame có source MAC không nằm trong danh sách bị drop ngay tại stage đầu tiên, trước khi đến được FDB pipeline. FDB learning lúc này hoàn toàn vô nghĩa: MAC source đã biết trước, không cần "học" thêm gì. Khi port_security disabled, OVN mất khả năng kiểm tra tính hợp lệ (validate) của MAC nguồn, port chấp nhận frame từ bất kỳ MAC nào. FDB learning lúc này có ý nghĩa: nó "học" MAC đã thấy để tránh flood lần sau. Đổi lại, cánh cửa mở cho FDB poisoning, bất kỳ MAC nào xuất hiện qua port đều được ghi nhận mà không kiểm chứng.

Keyword `"unknown"` trong addresses đưa port vào MC_UNKNOWN, nhóm nhận unknown unicast flood. Mối quan hệ giữa "unknown" và FDB learning là nhân-quả: chỉ port nào nhận được unknown unicast traffic (thông qua MC_UNKNOWN) mới có cơ hội "thấy" frame từ source MAC chưa biết. Port không có "unknown" không bao giờ nhận frame ngoài danh sách MAC tĩnh, không có gì để "học". Neutron ML2/OVN driver tự động thêm `"unknown"` vào addresses khi port_security disabled (Launchpad #1815270), nên trên OpenStack hai điều kiện đầu luôn đi cùng nhau. Trên OVN standalone, admin có thể set port_security=[] mà không set "unknown", lúc đó FDB learning không kích hoạt dù port_security disabled.

VIF port (type="") đại diện cho VM/container interface, nơi dynamic MAC learning có ý nghĩa trực tiếp. Localnet port ban đầu bị loại khỏi FDB learning vì lý do scale: localnet đại diện toàn bộ physical network segment, có thể có hàng trăm MAC đi qua, tiêu tốn `MAX_FDB_ENTRIES = 1000` nhanh chóng. Tùy chọn `localnet_learn_fdb=true` (v22.09.0, mô tả tại mục 17.2) được thêm dưới dạng opt-in cho hai use case: hardware offload (NVIDIA/Mellanox SmartNIC cần MAC-to-port mapping để offload chuyển tiếp xuống hardware) và provider network có ít thiết bị vật lý (5-10 MAC từ localnet không gây áp lực scale nhưng giảm đáng kể flooding).

> **Chủ đề trọng tâm:** FDB learning là con dao hai lưỡi: **có lợi** khi giảm unknown unicast flooding trên provider network (tiết kiệm bandwidth, cần thiết cho hardware offload), nhưng **gây hại** khi VNF emit frame với source MAC không thuộc về nó (FDB poisoning → traffic hijack). Ba điều kiện AND là thiết kế phòng thủ chiều sâu, chỉ mở FDB learning cho port thực sự cần, và chỉ khi port đó ở trạng thái "không kiểm soát được MAC".

Khuyến nghị vận hành: luôn set `fdb_age_threshold` > 0 trên mọi logical switch có FDB learning active, giá trị 300 giây (5 phút) là điểm khởi đầu hợp lý, cân bằng giữa cache hit rate và thời gian entry stale tồn tại. Giám sát FDB table size bằng `ovn-sbctl list fdb | wc -l`, nếu tiệm cận 1000, cân nhắc giảm `fdb_age_threshold` hoặc disable `localnet_learn_fdb`. Từ OVN v26.03.0, sử dụng `lsp_learn_mac` per-port để disable learning trên từng VNF port cụ thể mà không ảnh hưởng port khác, giải pháp granular nhất hiện có.

### fdb_age_threshold, aging mechanism cho FDB entries

Trước v23.09.0, FDB entry tồn tại vĩnh viễn sau khi được tạo, không có aging mechanism nào chạy. Ales Musil (amusil@redhat.com) cùng Numan Siddique đã bổ sung cơ chế aging trong OVN v23.09.0 qua một chuỗi commit, trong đó commit `bfbf32f363913db2a89f432360039a5d9ded5298` "northd: Synchronize the FDB age threshold" (Ales Musil, committed 2023-08-24 bởi Mark Michelson) là commit đồng bộ threshold sang Southbound datapath, chuẩn bị cho FDB refresh mechanism. Thread thảo luận đầy đủ: [ovs-dev tháng 5/2023](https://mail.openvswitch.org/pipermail/ovs-dev/2023-May/404718.html). Cấu hình per logical switch: `Logical_Switch.other_config:fdb_age_threshold` (đơn vị: giây). Default: 0, disabled, FDB entry tồn tại vĩnh viễn. Khi set > 0, ovn-northd tự động xóa entry quá tuổi.

NEWS: *"Added FDB aging mechanism, that is disabled by default. It can be enabled per logical switch with other_config 'fdb_age_threshold'."*

> **Chủ đề trọng tâm:** Với `fdb_age_threshold=0` (default), FDB entry tồn tại vĩnh viễn, ovn-northd không chạy aging. Trong case study VLAN 3808 (mục 17.6), đây là yếu tố khiến FDB entry sai không bao giờ tự expire, duy trì tình trạng traffic bị hijack cho đến khi phát hiện và xóa thủ công.

> **Hiểu sai:** "`fdb_age_threshold=0` nghĩa là FDB entry hết hạn ngay lập tức sau khi được tạo."
>
> **Thực tế:** `fdb_age_threshold=0` là giá trị **tắt aging**, không phải TTL bằng 0 giây. Khi bằng 0, ovn-northd không chạy aging job cho FDB table, và entry tồn tại vĩnh viễn cho đến khi bị xóa thủ công hoặc overwrite. Chỉ khi set giá trị dương (ví dụ `fdb_age_threshold=300`) thì entry mới có TTL 300 giây. Nhầm lẫn này đặc biệt nguy hiểm vì người vận hành có thể nghĩ "để mặc định là an toàn nhất", thực tế ngược lại: default-off aging có nghĩa là FDB entry sai tồn tại mãi mãi.

### flags.localnet protection, ngăn localnet overwrite VIF entry

Khi `localnet_learn_fdb=true`, một race condition nguy hiểm xuất hiện. Hình dung: vif1 trên chassis hv1 gửi frame, ovn-controller trên hv1 ghi FDB entry mac1 trỏ về vif1 (đúng). Cùng frame đi qua localnet đến chassis hv2, ovn-controller trên hv2 thấy frame đi vào từ localnet port và ghi FDB entry mac1 trỏ về localnet, overwrite entry đúng thành entry sai. Kết quả: traffic đến mac1 trên hv2 bị chuyển ra localnet thay vì qua Geneve tunnel đến hv1.

Xavier Simonart (xsimonart@redhat.com) giải quyết bằng bit `MLF_LOCALNET` (bit 15 trong logical flags register) trong commit `2acf91e9628e`. "controller: FDB entries for localnet should not overwrite entries for vifs". Backport: `33b01175` (branch-23.09.1), `8befadaf` (branch-23.06.3). Feature merge vào OVN v24.03.0.

Cơ chế hoạt động: trong `ls_in_lookup_fdb`, khi packet đến từ localnet port, OVN set `flags.localnet = 1` trước khi gọi `lookup_fdb`. Khi `flags.localnet=1`, hàm `lookup_fdb` trả về true nếu entry đã tồn tại VÀ port type của entry đó là VIF. Kết quả: `reg0[11] == 1` khiến `ls_in_put_fdb` skip việc ghi entry mới, bảo vệ VIF entry khỏi bị localnet overwrite. Protection này tốt cho field hợp bình thường (ngăn localnet overwrite VIF đúng), nhưng trong case study FDB poisoning (mục 17.6), khi VIF entry là SAI, protection trở thành rào cản ngăn self-heal.

## 17.5 - MAC_Binding table: IP-to-MAC resolution trên logical router

### Mục đích và thiết kế

Các phần trước tập trung vào Layer 2, MAC chuyển tiếp trên logical switch. Khi packet cần đi qua logical router (Layer 3), một bài toán hoàn toàn khác xuất hiện: router cần biết MAC của next-hop IP để đóng Ethernet header khi forward packet. **MAC_Binding** là table trong SB DB lưu mapping IP address sang MAC address trên logical router port, tương đương OVN của ARP cache trên router vật lý.

Điểm phân biệt cơ bản: ARP cache tồn tại trên router (Layer 3), không phải switch (Layer 2), vì router cần resolve next-hop IP sang MAC để forward packet qua link-layer. Switch vật lý thuần túy không chạy ARP và không có ARP cache, switch chỉ có MAC address table (tương đương FDB trong OVN). Sự phân biệt này quan trọng khi debug: nếu packet đi sai port trên cùng L2 segment, kiểm tra FDB; nếu router không resolve được MAC của next-hop IP, kiểm tra MAC_Binding.

Schema (ovn-sb(5)):

| Column | Type | Mô tả |
|-------------|-------------------|--------------------------------------|
| logical_port | string | Router port nơi binding được phát hiện |
| ip | string | IPv4 address |
| mac | string | MAC address tương ứng |
| datapath | Datapath_Binding | Logical router datapath |
| timestamp | integer | Thời điểm cập nhật (cho aging) |

Index: `[logical_port, ip]`

ovn-sb(5): *"Usually, IP-to-MAC bindings for virtual machines are statically populated into the Port_Binding table, so MAC_Binding is primarily used to discover bindings on physical networks."*

### Cách hoạt động trong router pipeline

```
  Logical Router Ingress Pipeline
  ════════════════════════════════
  lr_in_admission → lr_in_lookup_neighbor → lr_in_learn_neighbor
       ↓
  lr_in_ip_input → ... → lr_in_ip_routing → lr_in_ip_routing_ecmp
       ↓
  lr_in_arp_resolve:
    Priority 100: IP đã biết (VM, DHCP) → set eth.dst static
      match=(outport == "lrp-xxx" && reg0 == 192.168.1.11)
      action=(eth.dst = fa:16:3e:b6:91:70; next;)
    Priority 0: IP chưa biết → consult MAC_Binding
      match=(ip4)
      action=(get_arp(outport, reg0); next;)
       ↓
  lr_in_arp_request:
    Nếu eth.dst == 00:00:00:00:00:00 → generate ARP request
      action=(arp { eth.dst = ff:ff:ff:ff:ff:ff; arp.spa = reg1;
                    arp.tpa = reg0; arp.op = 1; output; };)
```

Action `get_arp(outport, reg0)` thực hiện resubmit đến OpenFlow table 66, nơi chứa các flow được populate từ MAC_Binding table. Khi ovn-controller nhận thấy ARP reply hoặc ND advertisement trên router port, pinctrl module ghi binding mới thông qua `put_arp(port, ip, mac)`, tạo ra một row trong MAC_Binding trên SB DB. Để tránh PACKET_IN flooding khi ARP reply đến liên tục, OVN dùng `lookup_arp`/`lookup_nd` kiểm tra trước xem entry đã tồn tại chưa.

Aging cho MAC_Binding được thêm trong OVN v22.09.0 (commit `1a947dd3`, Ales Musil, Bugzilla #2084668): `ovn-nbctl set Logical_Router <lr> options:mac_binding_age_threshold=<seconds>`. Default: 0 (disabled).

### So sánh FDB vs MAC_Binding, hai bài toán hoàn toàn khác nhau

| Khía cạnh | FDB | MAC_Binding |
|-------------------|-------------------------------|--------------------------------|
| **Layer** | L2 (switching) | L3 (routing) |
| **Maps** | MAC → port_key | IP → MAC |
| **Datapath** | Logical Switch | Logical Router |
| **Pipeline** | `ls_in_lookup_fdb`, `ls_in_l2_lkup` | `lr_in_arp_resolve` |
| **Populated by** | MAC learning từ VIF port | ARP reply/ND trên router port |
| **Tương đương vật lý** | MAC address table trên L2 switch | ARP cache trên router |
| **Ra đời** | Feb 2021 (schema 20.16.0) | Original OVN (~2015) |
| **Aging option** | `fdb_age_threshold` (per LS) | `mac_binding_age_threshold` (per LR) |

> **Chủ đề trọng tâm:** FDB (L2: mapping MAC address sang port_key, hoạt động trên logical switch) và MAC_Binding (L3: mapping IP address sang MAC address, hoạt động trên logical router) giải quyết hai bài toán hoàn toàn khác nhau. Khi debug, cần phân biệt rõ: nếu packet đi sai port trên cùng L2 segment thì kiểm tra FDB; nếu router không resolve được MAC của next-hop IP thì kiểm tra MAC_Binding.

> **Hiểu sai:** "Khi traffic đến gateway vật lý bị mất, tôi cần kiểm tra MAC_Binding vì có thể router không resolve được MAC của gateway."
>
> **Thực tế:** Nếu VM vẫn resolve được ARP của gateway (ping vẫn lấy được MAC) nhưng ICMP echo bị timeout, vấn đề không nằm ở MAC_Binding mà ở FDB. MAC_Binding chỉ ảnh hưởng đến IP-to-MAC resolution trên logical router, nếu ARP hoạt động, MAC_Binding không phải nguyên nhân. Vấn đề L2 chuyển tiếp (packet đi sai port trên cùng logical switch) luôn cần kiểm tra FDB trước.

Trong case study VLAN 3808 ở mục tiếp theo, ICMP echo request từ VM đến gateway 100.64.0.1 bị chuyển đến sai port (VyOS tap port thay vì localnet port ra physical network). Đây là vấn đề L2 chuyển tiếp, packet đi đúng logical switch nhưng bị đưa sai port đích. Table cần kiểm tra đầu tiên là FDB (nơi lưu mapping MAC-to-port sai), không phải MAC_Binding (vì vấn đề không liên quan đến IP-to-MAC resolution trên router).

## 17.6 - Case study: FDB poisoning trên provider network VLAN 3808

Các phần trước đã xây dựng từng mảnh ghép riêng lẻ: localnet port (mục 17.2) kết nối thế giới ảo với VLAN vật lý, MC_UNKNOWN (mục 17.3) flood unknown unicast đến localnet, FDB learning (mục 17.4) cache MAC để giảm flood, và MAC_Binding (mục 17.5) giúp router resolve next-hop. Phần này ghép tất cả lại thành bức tranh hoàn chỉnh thông qua một case study thực tế, nơi FDB learning kết hợp với port_security disabled tạo ra lỗ hổng khiến traffic production bị hijack.

### Topology và cấu hình

```
Provider Network VLAN 3808 (100.64.0.0/24)
══════════════════════════════════════════
  100.64.0.1  = Fornix gateway (physical, trên VLAN 3808)
  100.64.0.12 = VyOS VM (VRRP router, kết nối Fornix)
  100.64.0.xx = Các VM khác trên cùng logical switch

  VyOS VM (.12):
    - port_security_enabled = False  (cần thiết cho VNF/router)
    - NB addresses = ["fa:16:3e:04:74:74 100.64.0.12", "unknown"]
    - LSP type = ""  (VIF thường)
```

### Tại sao Neutron set addresses chứa "unknown"

Launchpad bug #1815270 ghi nhận vấn đề: khi `port_security_enabled=False`, OVN cần `"unknown"` trong addresses để port nhận traffic với destination MAC không biết trước. Neutron ML2/OVN driver tự động set: `addresses = ["fa:16:3e:04:74:74 100.64.0.12", "unknown"]`. Bug #1838535 refine thêm: chỉ set "unknown" cho VIF port (type=""), không cho router/localnet/localport type vì những type đặc biệt có cơ chế chuyển tiếp riêng, không cần tham gia MC_UNKNOWN.

### Tại sao FDB learning kích hoạt trên VyOS LSP

Ba điều kiện AND (mục 17.4) đều thỏa trên VyOS port: `port_security = []` (disabled), addresses chứa `"unknown"`, và `type = ""` (VIF thường). Do cả ba điều kiện thỏa, ovn-northd tạo logical flow cho FDB learning trên port này:

```
ls_in_lookup_fdb: inport == "vyos-port" →
    reg0[11] = lookup_fdb(inport, eth.src); next;
ls_in_put_fdb: inport == "vyos-port" && reg0[11] == 0 →
    put_fdb(inport, eth.src); next;
```

### Cơ chế poisoning, từng bước chi tiết

**Bước 1: VyOS gửi packet hợp lệ đến Fornix.** VyOS forward traffic đến gateway Fornix (100.64.0.1) thông qua static route. Mỗi packet rời VyOS mang source MAC = fa:16:3e:04:74:74 (MAC thật của eth1) và destination MAC = fa:16:3e:35:a7:73 (MAC của Fornix, lấy từ neighbor cache). Đây là hành vi hoàn toàn bình thường, kernel Linux luôn dùng `dev->dev_addr` của output interface làm source MAC thông qua hàm `eth_header()` trong `net/ethernet/eth.c` ([kernel source](https://github.com/torvalds/linux/blob/master/net/ethernet/eth.c)). Keepalived GARP cũng tương tự: hàm `send_gratuitous_arp_immediate()` luôn dùng `IF_HWADDR(ipaddress->ifp)`, MAC vật lý của interface, không bao giờ dùng MAC của thiết bị bên ngoài ([keepalived source](https://github.com/acassen/keepalived)). VM VyOS **không có khả năng** emit frame với source MAC = fa:16:3e:35:a7:73.

**Bước 1b: OVN pipeline swap eth.src ↔ eth.dst mà không sync inport ↔ outport (BUG).** Packet hợp lệ từ VyOS đi vào OVN pipeline với `inport = VyOS_LSP`, `eth.src = fa:16:3e:04:74:74`, `eth.dst = fa:16:3e:35:a7:73`. Tại một code path, ví dụ khi tạo ICMP "Fragmentation Needed" cho multichassis port, OVN swap `eth.src ↔ eth.dst` để xây dựng reply packet, nhưng **quên swap `inport ↔ outport`**. Sau swap: `inport` vẫn = VyOS_LSP, nhưng `eth.src` = fa:16:3e:35:a7:73 (MAC của Fornix, vốn là destination MAC ban đầu). Đây chính xác là bug class đã được Ales Musil xác nhận và fix trong patch *"physical: Prevent wrong FDB to be learned with multichassis port"* ([ovs-dev mailing list, 20 Aug 2024](https://www.mail-archive.com/ovs-dev@openvswitch.org/msg88284.html), `Reported-at: https://issues.redhat.com/browse/FDP-620`).

> **Chủ đề trọng tâm:** Root cause của FDB poisoning không nằm trong VM. VyOS gửi packet bình thường, OVN pipeline tạo ra FDB entry sai từ destination MAC trong packet hợp lệ đó. Điều kiện cần: (1) OVN phiên bản chưa fix bug swap inport/outport, (2) port có `addresses="unknown"` (FDB learning kích hoạt), (3) `fdb_age_threshold=0` (entry sai tồn tại vĩnh viễn).

**Bước 2: put_fdb ghi entry SAI.** Frame đi vào br-int, khớp flow trong `ls_in_lookup_fdb`:

```
lookup_fdb(inport="vyos-port", eth.src="fa:16:3e:35:a7:73")
→ reg0[11] = 0  (chưa có entry cho MAC này)
```

Chuyển sang `ls_in_put_fdb`:

```
reg0[11] == 0 → put_fdb(inport="vyos-port", eth.src="fa:16:3e:35:a7:73")
```

FDB entry được tạo trong SB DB:

```
ovn-sbctl list fdb
mac           : "fa:16:3e:35:a7:73"
dp_key        : <VLAN3808_dp_key>
port_key      : 3                      ← VyOS LSP tunnel key!
```

**Bước 3: Traffic bị hijack.** Khi VM khác gửi ICMP đến gateway 100.64.0.1 (MAC = fa:16:3e:35:a7:73):

```
ls_in_l2_lkup, priority 50: eth.dst == fa:16:3e:35:a7:73?
  → KHÔNG có static entry (MAC này không trong LSP addresses nào)

ls_in_l2_lkup, priority 0: outport = get_fdb(eth.dst="fa:16:3e:35:a7:73")
  → FDB tìm thấy! outport = port_key 3 (VyOS LSP)

ls_in_l2_unknown, priority 0: output;
  → Forward đến VyOS tap port thay vì localnet (bond3)!
```

ICMP đi vào VyOS VM thay vì ra physical network → gateway không bao giờ nhận được → timeout.

### Tại sao ARP vẫn hoạt động bình thường

ARP request là broadcast (eth.dst = ff:ff:ff:ff:ff:ff):

```
ls_in_l2_lkup, priority 72: eth.mcast && arp.op == 1
  → outport = "_MC_flood"; output;
```

ARP bypass hoàn toàn FDB lookup, đi thẳng vào `_MC_flood` ở priority 72, cao hơn đáng kể so với priority 0 nơi `get_fdb` thực thi. ARP reply từ gateway đi qua localnet → flood → đến đúng VM. Đây là lý do ARP resolution vẫn hoạt động bình thường trong khi unicast traffic bị hijack, hai loại traffic đi qua hai nhánh khác nhau trong pipeline.

### Tại sao bond3 không thấy ICMP

FDB entry `fa:16:3e:35:a7:73 → port_key 3` short-circuit trước khi packet có cơ hội đến localnet port. `get_fdb` trả về VyOS port_key thay vì "none", nên packet KHÔNG đi vào `_MC_unknown` → KHÔNG được flood đến localnet → KHÔNG ra bond3. Frame bị "hút" vào VyOS tap port, gateway vật lý hoàn toàn không thấy traffic.

### Tại sao FDB entry không tự hết hạn

Với `fdb_age_threshold=0` (default), ovn-northd không chạy aging, FDB entry tồn tại cho đến khi bị xóa thủ công (`ovn-sbctl destroy fdb <uuid>`), bị overwrite bởi entry khác (cùng MAC, cùng datapath), hoặc ovn-controller restart. Trong case study này, entry sai tồn tại vĩnh viễn.

### Tại sao gateway không tự sửa FDB

Giả sử gateway Fornix gửi frame ra với source MAC fa:16:3e:35:a7:73. Frame này đến qua localnet port. Nhưng `flags.localnet` protection (nếu OVN phiên bản đủ mới) ngăn localnet overwrite VIF entry:

```
ls_in_lookup_fdb cho localnet port:
  flags.localnet = 1; reg0[11] = lookup_fdb(inport, eth.src);

lookup_fdb với flags.localnet=1:
  Entry tồn tại (mac=fa:16:3e:35:a7:73, port=VyOS)
  Port type hiện tại = VIF → return 1 (đã có, không cần update)

ls_in_put_fdb: reg0[11] == 1 → SKIP put_fdb
```

Entry VIF được bảo vệ → gateway không thể sửa FDB từ localnet side. Đây là protection tốt cho field hợp bình thường (ngăn localnet overwrite VIF đúng), nhưng trong field hợp này, VIF entry là SAI → protection trở thành rào cản ngăn tự khắc phục.

### Fix ngắn hạn và dài hạn

Fix ngắn hạn, xóa FDB entry sai:

```bash
# Tìm FDB entry
[controller]$ ovn-sbctl find fdb mac="fa:16:3e:35:a7:73"

# Xóa
[controller]$ ovn-sbctl destroy fdb <uuid>
```

Ngay sau khi xóa, `get_fdb` không tìm thấy MAC nên set outport="none". Pipeline chuyển sang MC_UNKNOWN, flood packet đến localnet port, ra bond3, đến gateway vật lý. Traffic phục hồi ngay lập tức.

Fix dài hạn cần kết hợp nhiều biện pháp: enable port_security trên VyOS port nếu có thể, ngăn FDB learning hoàn toàn; set `fdb_age_threshold > 0` trên logical switch, entry sai sẽ tự expire; hoặc dùng allowed-address-pairs thay vì disable port_security hoàn toàn, cho phép MAC cụ thể mà vẫn giữ port_security.

```bash
# Set FDB aging 300 giây (5 phút) trên logical switch
[controller]$ ovn-nbctl set Logical_Switch <ls> other_config:fdb_age_threshold=300
```

### OVN FDB bug đã biết bằng chứng root cause nằm ở pipeline

Phân tích pháp y (forensic analysis) trên case study này xác định root cause không nằm trong VM VyOS mà ở tầng OVN pipeline. Kết luận này dựa trên ba trụ bằng chứng: loại trừ triệt để mọi cơ chế VM-side, xác nhận bug class trong OVN source code, và phân tích cấu hình VyOS chỉ đóng vai trò amplifier.

Trụ thứ nhất, loại trừ VM-side. Hàm `eth_header()` trong kernel `net/ethernet/eth.c` luôn gán `dev->dev_addr` (MAC của output interface) vào `eth->h_source` khi `saddr=NULL`, đây là code path duy nhất cho mọi packet forward qua `ip_forward()`. Keepalived `send_gratuitous_arp_immediate()` luôn dùng `IF_HWADDR(ipaddress->ifp)` qua `PF_PACKET + SOCK_RAW` socket, kernel không can thiệp source MAC. Config VyOS không có bridge, macvlan, hay proxy_arp. Kết luận: VM không có cơ chế nào tạo frame với source MAC khác interface MAC.

Trụ thứ hai, bug class đã xác nhận trong OVN. Bốn bug report liên quan trực tiếp đến FDB:

Thứ nhất, FDP-620 (fix OVN v24.09.1, patch bởi Ales Musil). Hàm `reply_imcp_error_if_pkt_too_big` trong `controller/physical.c` (tên hàm gốc upstream có typo `imcp` thay vì `icmp`, xem comment `NOTE(ihrachys)` trên dòng 1905 của file) swap `eth.src ↔ eth.dst` khi tạo ICMP error cho multichassis port nhưng quên swap `inport ↔ outport`. FDB learning sau đó ghi `put_fdb(inport=VIF_port, eth.src=original_dst_MAC)`, chính xác pattern của case study này. Fix thêm 6 dòng code swap `MFF_LOG_INPORT ↔ MFF_LOG_OUTPORT` trước khi swap MAC.

Thứ hai, BZ 1983111 (fix OVN v21.09.1). FDB lookup/put OpenFlow flow không được cài đặt khi ovn-controller claim port đầu tiên trên logical switch, trong window này, FDB learning không hoạt động đúng.

Thứ ba, pinctrl map size bug (fix v24.09.2, commit `33a6ae53`). Hàm `pinctrl_handle_put_fdb()` kiểm tra `hmap_count(&put_mac_bindings)` thay vì `hmap_count(&put_fdbs)`, rate limiting FDB entries dựa trên nhầm map.

Thứ tư, FDP-1341 (fix v25.03.2). VIF FDB entries bị mất khi SB transaction thất bại, cho phép localnet entries overwrite.

Trụ thứ ba, cấu hình VyOS là amplifier, không phải root cause. `garp_master_refresh=5` tạo volume traffic cao (GARP mỗi 5 giây), static route qua Fornix đảm bảo destination MAC fa:16:3e:35:a7:73 luôn hiện diện trong packet, cả hai tăng xác suất kích hoạt bug OVN. Nhưng `port_security_enabled=False` là enabler quyết định (kích hoạt FDB learning) và `fdb_age_threshold=0` là amplifier quyết định (biến sự kiện one-time thành permanent damage).

> **Chủ đề trọng tâm:** Decisive test xác nhận VM vô tội: `tcpdump -i tap-VYOS_PORT -e -n 'ether src fa:16:3e:35:a7:73'`. Nếu chạy trên hypervisor host và thấy zero frame, đó là bằng chứng thực nghiệm rằng VyOS không emit source MAC sai, entry FDB sai được tạo bởi OVN pipeline. Kiểm tra phiên bản OVN: `ovn-controller --version` và đối chiếu với FDP-620 fix (v24.09.1).

### Cơ chế multichassis binding: dấu phẩy trong `requested-chassis`

Điều kiện kích hoạt đầu tiên của bug nằm ở state `additional_chassis` của port. State này chỉ xuất hiện khi Neutron đặt `LSP.options:requested-chassis` dưới dạng chuỗi phân cách dấu phẩy (ví dụ `"chassisA,chassisB"`), và duy nhất thao tác `nova live-migrate` mới tạo ra giá trị này. Các thao tác Nova còn lại như `nova boot`, `reboot`, `stop`, `resize`, `evacuate`, `rebuild`, `shelve` đều đặt giá trị đơn, dẫn đến binding single-chassis thông thường nên không tạo multichassis state.

Phân tích chi tiết hàm `lport_can_bind_on_this_chassis()`, cơ chế `claim_lport()`, table chín thao tác Nova, cùng lý do kiến trúc chỉ live migration cần dual-active binding được trình bày trong [SDN 3.2 về Multichassis port binding lifecycle](./19.0%20-%20ovn-multichassis-binding-and-pmtud.md).

### Geneve overhead 58 byte và pipeline PMTUD

Điều kiện kích hoạt thứ hai là packet quá cỡ so với tunnel effective MTU. Tổng overhead Geneve trong OVN là 58 byte, gồm outer Ethernet 14 byte, outer IPv4 20 byte, outer UDP 8 byte, Geneve base header 8 byte, và OVN TLV option Class 0x0102 8 byte. Với physical MTU 1500, effective tunnel MTU chỉ còn 1442 byte. Gói IP 1500 byte từ VM kích hoạt `check_pkt_larger(1442)` tại table 41 (`OFTABLE_OUTPUT_LARGE_PKT_DETECT`), set `reg9[1] = 1`, rồi resubmit sang table 42 (`OFTABLE_OUTPUT_LARGE_PKT_PROCESS`).

Tại table 42, hàm `reply_imcp_error_if_pkt_too_big()` swap `eth.src ↔ eth.dst` nhưng không swap `inport ↔ outport` tương ứng. FDB learning downstream đọc giá trị `inport` cũ (vẫn là VyOS LSP) ghép với `eth.src` mới (đã thành Fornix MAC), kết quả ghi `put_fdb(inport=VyOS_LSP, eth.src=Fornix_MAC)`. Đây chính là FDB entry sai trực tiếp gây hijack traffic của 100.64.0.1 sang VyOS.

Phân rã byte-by-byte overhead, action sequence đầy đủ của `reply_imcp_error_if_pkt_too_big()`, cùng bản vá 6 dòng `MFF_LOG_INPORT/OUTPORT` của Ales Musil được trình bày trong [SDN 3.4 về Geneve overhead, pipeline PMTUD và bug FDP-620](./19.0%20-%20ovn-multichassis-binding-and-pmtud.md).

### Mitigation và fix triệt để

Trong case study VLAN 3808, physical MTU là 1500 nên điều kiện "gói lớn hơn tunnel MTU" được thỏa mãn và bug kích hoạt. Có hai hướng giảm rủi ro trong môi trường kolla-ansible đang chạy.

Hướng thứ nhất là nâng physical MTU tunnel lên 9000 (jumbo frames) để effective tunnel MTU đạt 8942 byte. Gói 1500 byte từ VM không còn vượt ngưỡng nên không kích hoạt `reply_imcp_error_if_pkt_too_big()`. Đây chỉ là mitigation vì bug vẫn tồn tại trong code path, chỉ không được kích hoạt ở MTU hiện tại. Nếu sau này VM MTU tăng hoặc encap overhead tăng (ví dụ double encap), điều kiện kích hoạt có thể tái xuất hiện.

Hướng thứ hai là build container OVN 24.09.1+ qua `kolla-build --template-override` kết hợp Ubuntu Cloud Archive Proposed pocket, sau đó upgrade để áp dụng patch Ales Musil. Đây là fix triệt để cho triển khai kolla-ansible vì xóa root cause khỏi code path, không phụ thuộc vào MTU vận hành.

Config `[ovn] live_migration_activation_strategy` trong `neutron.conf` (giá trị `rarp` mặc định, hoặc `chassis`, do Ihar Hrachyshka commit `949b098626b7` Launchpad #2092250, merged 2025-01-15) không trực tiếp ngăn bug này. Cấu hình đó chỉ kiểm soát thời điểm additional chassis location trở nên khả dụng, không sửa logic sai trong ICMP error path. Strategy `rarp` có thể giảm xác suất kích hoạt trên thực tế bằng cách hạn chế traffic đi qua tunnel trong cửa sổ multichassis, nhưng không phải bằng cách vá logic sai.

Chi tiết jumbo frame sự đánh đổi, kernel `mtu_expires` recovery cho cache PMTUD bị poison, cùng phân tích đầy đủ activation-strategy `rarp` vs `chassis` (gồm cả lý do RARP được chọn thay vì GARP) được trình bày trong [SDN 3.5 về activation-strategy và đồng bộ ba tầng](./19.0%20-%20ovn-multichassis-binding-and-pmtud.md) và [SDN 3.6 về Operational tuning Jumbo frame và MTU recovery](./19.0%20-%20ovn-multichassis-binding-and-pmtud.md).

### ▶ Lab 3: Chẩn đoán và khắc phục FDB poisoning trên provider network

Trong bài lab này, bạn chẩn đoán nguyên nhân gốc rễ khi traffic L2 trên provider network bị chuyển sai port, xác minh bằng phản chứng rằng FDB entry sai là nguyên nhân, và khắc phục.

**Outcomes:**
- Xác định được FDB entry sai bằng `ovn-sbctl` và đối chiếu với port_binding
- Chứng minh bằng phương pháp POE rằng xóa FDB entry sai khôi phục traffic
- Thiết lập FDB aging để ngăn tái diễn

**Before You Begin:**
Truy cập được OVN central node (nơi chạy ovn-sbctl). Có ít nhất một provider network với VNF port (port_security disabled).

**Instructions:**

**1.** Xác minh triệu chứng: VM ping đến gateway vật lý bị timeout, nhưng ARP resolution vẫn hoạt động.

**2.** Liệt kê FDB table và xác định entry nghi ngờ, so sánh MAC trong FDB với MAC thực của gateway.

**3.** Đối chiếu port_key trong FDB entry với Port_Binding để xác định packet đang bị chuyển đến port nào.

**Predict (Dự đoán bằng giả thuyết ngược):** Nếu FDB entry sai KHÔNG phải nguyên nhân (nghĩa là vấn đề nằm ở nơi khác, ví dụ physical network, bridge mapping, hoặc ACL), thì xóa FDB entry sai sẽ không thay đổi hành vi, traffic vẫn timeout sau khi xóa.

**4.** Xóa FDB entry nghi ngờ và kiểm tra traffic ngay lập tức.

**5.** Nếu traffic phục hồi: kết luận FDB entry sai là nguyên nhân gốc rễ.

**Explain:** Khi FDB entry sai bị xóa, `get_fdb(eth.dst)` không tìm thấy MAC nên set outport="none". Pipeline chuyển sang MC_UNKNOWN, flood packet đến localnet port, packet ra physical network đến gateway. Traffic phục hồi ngay lập tức chứng minh giả thuyết "vấn đề nằm ở nơi khác" bị bác bỏ, FDB entry sai chính là nguyên nhân duy nhất.

**6.** Thiết lập FDB aging để ngăn tái diễn.

Gợi ý: `ovn-sbctl list fdb`, `ovn-sbctl find port_binding tunnel_key=<port_key>`, `ovn-sbctl destroy fdb <uuid>`, `ovn-nbctl set Logical_Switch <ls> other_config:fdb_age_threshold=300`

**Evaluation:**
Sau khi hoàn thành, chạy `ovn-trace` với destination MAC của gateway, output phải cho thấy packet đi qua MC_UNKNOWN đến localnet port (không còn bị redirect sang VNF port). Chạy `ovn-nbctl get Logical_Switch <ls> other_config:fdb_age_threshold`, giá trị phải > 0.

**Finish:**
FDB entry sai đã bị xóa và aging đã được thiết lập, không cần rollback.

## 17.7 - Bài học thiết kế: sự đánh đổi và tiến hóa incremental

### Functionality vs Security: port_security disabled là cần thiết nhưng nguy hiểm

VNF (Virtual Network Function) như router, load balancer, firewall bắt buộc phải disable port_security, chúng cần gửi/nhận frame với MAC không phải của mình. Đây là yêu cầu hợp lệ mà OpenStack hỗ trợ thông qua `port_security_enabled=False`. Nhưng khi disable port_security, OVN mất khả năng kiểm tra tính hợp lệ (validate) của MAC nguồn → cửa mở cho FDB poisoning. Đây là sự đánh đổi inherent trong thiết kế: OVN không thể vừa cho phép arbitrary MAC chuyển tiếp vừa đảm bảo FDB integrity, vì hai mục tiêu này mâu thuẫn nhau ở tầng kiến trúc.

Giải pháp tiến hóa qua thời gian thay vì big-bang redesign: ban đầu không có protection, sau đó thêm `flags.localnet` để ngăn localnet overwrite VIF entry, rồi thêm `fdb_age_threshold` để giới hạn thời gian entry sai tồn tại, và cuối cùng `lsp_learn_mac` để tắt learning trên từng port cụ thể. Mỗi bước phản ứng trước một bug hoặc kịch bản production thực tế mà các bước trước chưa bao phủ.

### Performance vs Correctness: FDB caching giảm flood nhưng tạo stale entry

Trước FDB, mọi unknown unicast đều bị flood: bandwidth cao nhưng luôn đúng vì packet đến mọi nơi kể cả đích đúng. Sau FDB, unknown unicast chỉ flood lần đầu rồi được forward thẳng từ lần sau: bandwidth tối ưu nhưng đổi lại là khả năng entry stale hoặc sai dẫn đến chuyển tiếp sai.

Đây là bản chất của **caching problem**, same sự đánh đổi như CPU cache, DNS cache, ARP cache. Mọi cache đều phải đối mặt câu hỏi: stale entry tồn tại bao lâu trước khi bị invalidate? TTL quá ngắn giảm cache hit rate (quay lại flooding), TTL quá dài cho phép entry sai gây damage lâu hơn. OVN chọn cách tiếp cận progressive: FDB default OFF (learning chỉ kích hoạt cho port đặc biệt thỏa ba điều kiện AND), cho phép opt-in mở rộng qua `localnet_learn_fdb`, và cung cấp aging mechanism (`fdb_age_threshold`) khi người vận hành chấp nhận sự đánh đổi.

### OVN tiến hóa qua incremental hardening

Nhìn lại timeline phát triển FDB cho thấy pattern rõ ràng, mỗi phiên bản thêm safeguard mới phản ứng trước bug hoặc kịch bản production thực tế:

```
2015: OVN ra đời — chỉ có static MAC + unknown flooding
       → Đơn giản, đúng, nhưng flood nhiều

2021 (v21.06): FDB learning — dynamic MAC caching
       → Giảm flood nhưng mở risk FDB poisoning

2022 (v22.09): localnet_learn_fdb — opt-in learning cho localnet
       → Mở rộng nhưng thêm risk overwrite

2023 (v23.09): fdb_age_threshold — aging mechanism
       → Limit damage từ entry sai

2023 (v24.03): flags.localnet — protect VIF entry từ localnet
       → Sửa specific race condition

2024 (v24.09): force_fdb_lookup — granular control
2025 (v26.03): lsp_learn_mac — per-port learning control
       → Ngày càng fine-grained
```

Mỗi phiên bản giữ tương thích ngược (hành vi mặc định không thay đổi), người vận hành upgrade OVN mà không cần thay đổi cấu hình. Đây là bài học thiết kế quan trọng cho distributed system: tiến hóa incremental cho phép hệ thống production migrate dần mà không phải dừng hoạt động.

### "unknown" addresses, keyword mạnh nhất nhưng cũng nguy hiểm nhất trong OVN

Keyword `"unknown"` kích hoạt đồng thời ba cơ chế: port nhận tất cả unknown unicast traffic (thông qua MC_UNKNOWN membership), port kích hoạt FDB learning (khi kết hợp với port_security disabled), và bất kỳ MAC nào đi vào port đều được ghi nhận vào FDB mà không kiểm chứng. Ba cơ chế kết hợp tạo surface attack lớn, bất kỳ frame nào với source MAC lạ đi qua port đó đều có thể poison FDB table. Trước khi set `addresses = ["...", "unknown"]` trên bất kỳ port nào, người vận hành cần hiểu toàn bộ implications mà tài liệu này đã phân tích.

### Debug checklist khi L2 chuyển tiếp bất thường trên OVN

Khi gặp triệu chứng traffic L2 đi sai port hoặc bị mất trên provider network, trình tự debug được đề xuất dựa trên kiến thức từ các mục trước:

| Triệu chứng | Table cần kiểm tra | Lệnh |
|---|---|---|
| Packet đi sai port trên cùng L2 | FDB | `ovn-sbctl list fdb` |
| Router không resolve được next-hop MAC | MAC_Binding | `ovn-sbctl list mac_binding` |
| ARP không đến | MC_FLOOD membership | `ovn-sbctl list multicast_group` |
| Unknown unicast không flood | MC_UNKNOWN membership | `ovn-sbctl list multicast_group` |
| Cần theo dõi toàn bộ pipeline | Logical flow + OpenFlow | `ovn-trace` + `ovs-ofctl dump-flows br-int table=71` |

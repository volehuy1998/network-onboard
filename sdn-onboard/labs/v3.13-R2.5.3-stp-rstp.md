## Why this lab exists

A Cisco-trained engineer's mental model of Spanning Tree is concrete: in a triangular three-switch loop, the root election picks one switch as the root, the other two switches each pick a root port pointing at the root, and the third link (the one not adjacent to the root) is blocked on one end so the active topology is a tree. Convergence takes around 30 to 50 seconds with classic 802.1D STP and 1 to 6 seconds with 802.1D-2004 RSTP. Once converged, traffic flows; the blocked port is silent until the active topology fails.

The question this lab answers is whether OVS and the Linux bridge produce the same result when given the same triangular topology. Both implementations have full STP state machines but the two were authored independently, more than a decade apart, by different teams. OVS implements both 802.1D STP (`lib/stp.c`) and 802.1D-2004 RSTP (`lib/rstp.c` plus `lib/rstp-state-machines.c`) in userspace inside `ovs-vswitchd`. The Linux kernel bridge implements 802.1D STP only (no RSTP) inside the kernel module `bridge.ko`, with RSTP available out-of-tree via the `mstpd` daemon. There is no a priori reason the two implementations must agree on root election, port roles, or convergence time.

The lab proves by direct observation what actually happens. Phase A builds three OVS bridges in a triangle, sets bridge priorities so root election is deterministic (br1 wins), enables RSTP on all three, waits for convergence, and reads `ovs-appctl rstp/show` on each. Phase B builds three Linux bridges in the same triangle with the kernel STP implementation, reduces the timers so convergence fits in the lab window, and reads the per-bridge and per-port STP state from `/sys/class/net/<name>/bridge/` and `/sys/class/net/<port>/brport/state`.

The expected outcome is the same in both phases: br1 wins root election, the other two bridges each choose a port pointing at br1 as their root port, and the redundant link between br2 and br3 has one end blocked. Reading the two sets of evidence side by side will show whether the implementations land on the same answer through different code paths.

## Phase A, OVS RSTP triangle

The triangular topology is built from three veth pairs. Each veth pair is a wire between two bridges: `v12a/v12b` connects br1 and br2, `v23a/v23b` connects br2 and br3, `v13a/v13b` connects br1 and br3. With three links forming a triangle, every pair of bridges has two paths to the other (a direct one and a two-hop one through the third bridge), so a loop exists. Without a spanning-tree protocol, broadcast frames would circulate forever; with RSTP, the bridges cooperatively elect a root and block one link to break the loop.

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-br br1; ovs-vsctl add-br br2; ovs-vsctl add-br br3
root@lab-openvswitch:~/network-onboard# ip link add v12a type veth peer name v12b
root@lab-openvswitch:~/network-onboard# ip link add v23a type veth peer name v23b
root@lab-openvswitch:~/network-onboard# ip link add v13a type veth peer name v13b
root@lab-openvswitch:~/network-onboard# ip link set v12a up; ip link set v12b up; ip link set v23a up; ip link set v23b up; ip link set v13a up; ip link set v13b up
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br1 v12a; ovs-vsctl add-port br2 v12b
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br2 v23a; ovs-vsctl add-port br3 v23b
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br1 v13a; ovs-vsctl add-port br3 v13b
```

Without further configuration, root election uses the bridge ID, which is the concatenation of a configurable priority value (default 32768) and the bridge's MAC address. Three freshly-created bridges have random MACs, so the root elected by the protocol depends on which random MAC happens to be numerically lowest. To make the election deterministic for teaching purposes, the next three commands set explicit priorities: 4096 on br1, 8192 on br2, 12288 on br3. The lowest priority wins, so br1 is the predicted root.

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl set bridge br1 other_config:rstp-priority=4096
root@lab-openvswitch:~/network-onboard# ovs-vsctl set bridge br2 other_config:rstp-priority=8192
root@lab-openvswitch:~/network-onboard# ovs-vsctl set bridge br3 other_config:rstp-priority=12288
root@lab-openvswitch:~/network-onboard# ovs-vsctl set bridge br1 rstp_enable=true
root@lab-openvswitch:~/network-onboard# ovs-vsctl set bridge br2 rstp_enable=true
root@lab-openvswitch:~/network-onboard# ovs-vsctl set bridge br3 rstp_enable=true
root@lab-openvswitch:~/network-onboard# sleep 8
```

The `sleep 8` is the convergence window. RSTP defaults to a hello time of 2 seconds and starts converging within one or two hello intervals; after eight seconds the topology has stabilised. Read the resulting state on each bridge.

```text
root@lab-openvswitch:~/network-onboard# ovs-appctl rstp/show br1
---- br1 ----
Root ID:
  stp-priority    4096
  stp-system-id   b6:ea:e1:fe:dd:48
  stp-hello-time  2s
  stp-max-age     20s
  stp-fwd-delay   15s
  This bridge is the root
 
Bridge ID:
  stp-priority    4096
  stp-system-id   b6:ea:e1:fe:dd:48
  stp-hello-time  2s
  stp-max-age     20s
  stp-fwd-delay   15s
 
  Interface  Role       State      Cost     Pri.Nbr 
  ---------- ---------- ---------- -------- -------
  v12a       Designated Forwarding 2000     128.1
  v13a       Designated Forwarding 2000     128.2
```

br1 reports "This bridge is the root", which confirms the priority-driven election worked. The Bridge ID matches the Root ID exactly. Both of br1's ports (`v12a` toward br2, `v13a` toward br3) are in the Designated role and the Forwarding state, which is correct: a root bridge owns the active path on every link directly attached to it.

```text
root@lab-openvswitch:~/network-onboard# ovs-appctl rstp/show br2
---- br2 ----
Root ID:
  stp-priority    4096
  stp-system-id   b6:ea:e1:fe:dd:48
  stp-hello-time  2s
  stp-max-age     20s
  stp-fwd-delay   15s
  root-port       v12b
  root-path-cost  2000
 
Bridge ID:
  stp-priority    8192
  stp-system-id   66:03:42:85:7d:45
  stp-hello-time  2s
  stp-max-age     20s
  stp-fwd-delay   15s
 
  Interface  Role       State      Cost     Pri.Nbr 
  ---------- ---------- ---------- -------- -------
  v12b       Root       Forwarding 2000     128.1
  v23a       Designated Forwarding 2000     128.2
```

br2 reports the Root ID as priority 4096 with system-id `b6:ea:e1:fe:dd:48`, which matches br1's identity. The `root-port` line names `v12b` as br2's chosen path to the root with a path cost of 2000. `v12b` therefore holds the Root role and Forwarding state. The other port `v23a` (toward br3) is Designated/Forwarding, meaning br2 is the designated bridge on the br2-br3 link. So far the topology has two of three links active and the root visible from both downstream bridges.

```text
root@lab-openvswitch:~/network-onboard# ovs-appctl rstp/show br3
---- br3 ----
Root ID:
  stp-priority    4096
  stp-system-id   b6:ea:e1:fe:dd:48
  stp-hello-time  2s
  stp-max-age     20s
  stp-fwd-delay   15s
  root-port       v13b
  root-path-cost  2000
 
Bridge ID:
  stp-priority    12288
  stp-system-id   92:96:f5:97:51:4c
  stp-hello-time  2s
  stp-max-age     20s
  stp-fwd-delay   15s
 
  Interface  Role       State      Cost     Pri.Nbr 
  ---------- ---------- ---------- -------- -------
  v23b       Alternate  Discarding 2000     128.1
  v13b       Root       Forwarding 2000     128.2
```

This is the decisive observation. br3 chose `v13b` (the direct link to br1) as its root port at path cost 2000. The same path cost 2000 would result from the indirect route through br2 (br3 to br2 cost 2000, br2 to br1 already 2000, total 4000), but the direct link is shorter, so the direct link wins. The remaining port `v23b` (toward br2) is in the **Alternate** role and the **Discarding** state. Alternate is the RSTP role for a port that has heard a better BPDU and is willing to take over if the active path fails; Discarding is the RSTP equivalent of the legacy 802.1D Blocking state, meaning the port forwards no user traffic but continues to listen for BPDUs. The redundant link is broken precisely at this point, on br3's end, exactly where loop-prevention requires.

The active topology is now a tree rooted at br1 with two leaves. br1 to br2 forwards. br1 to br3 forwards. br2 to br3 carries BPDUs only; user frames stop at `v23b` and never cross. If the br1-br3 link were to fail, RSTP on br3 would transition `v23b` from Discarding to Forwarding within a few hello intervals, restoring connectivity through the now-active two-hop path.

The OVSDB confirms RSTP is enabled on all three bridges:

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=name,rstp_enable list bridge
name                : br2
rstp_enable         : true
 
name                : br3
rstp_enable         : true
 
name                : br1
rstp_enable         : true
```

## Phase B, Linux-bridge STP triangle

The Linux kernel bridge driver supports 802.1D STP only. The newer 802.1D-2004 RSTP standard requires an out-of-tree daemon (`mstpd`); the kernel bridge alone cannot do RSTP. The lab restricts itself to the kernel STP implementation because the goal is to compare what the two stock implementations do, not to introduce an additional userspace component.

The kernel STP defaults are `forward_delay=15s` and `hello_time=2s`, which gives a convergence time of around 30 seconds. The lab reduces both to `forward_delay=400` (centiseconds, so 4 seconds) and `hello_time=100` (1 second) so the wait fits in the capture window. The priority-and-MAC bridge ID model is the same as RSTP, so setting priorities 4096, 8192, 12288 on lbr1, lbr2, lbr3 produces a deterministic root election with lbr1 as the predicted root.

```text
root@lab-openvswitch:~/network-onboard# ip link add name lbr1 type bridge stp_state 1 priority 4096 forward_delay 400 hello_time 100
root@lab-openvswitch:~/network-onboard# ip link add name lbr2 type bridge stp_state 1 priority 8192 forward_delay 400 hello_time 100
root@lab-openvswitch:~/network-onboard# ip link add name lbr3 type bridge stp_state 1 priority 12288 forward_delay 400 hello_time 100
root@lab-openvswitch:~/network-onboard# ip link set lbr1 up; ip link set lbr2 up; ip link set lbr3 up
root@lab-openvswitch:~/network-onboard# ip link add lv12a type veth peer name lv12b
root@lab-openvswitch:~/network-onboard# ip link add lv23a type veth peer name lv23b
root@lab-openvswitch:~/network-onboard# ip link add lv13a type veth peer name lv13b
root@lab-openvswitch:~/network-onboard# ip link set lv12a master lbr1; ip link set lv12b master lbr2
root@lab-openvswitch:~/network-onboard# ip link set lv23a master lbr2; ip link set lv23b master lbr3
root@lab-openvswitch:~/network-onboard# ip link set lv13a master lbr1; ip link set lv13b master lbr3
root@lab-openvswitch:~/network-onboard# ip link set lv12a up; ip link set lv12b up; ip link set lv23a up; ip link set lv23b up; ip link set lv13a up; ip link set lv13b up
root@lab-openvswitch:~/network-onboard# sleep 12
```

The configuration mirror is mostly mechanical, with one detail worth highlighting. With OVS, RSTP is enabled per-bridge with one column (`rstp_enable=true`); with the Linux bridge, STP is enabled at bridge-creation time via `stp_state 1` plus the timer parameters (`forward_delay`, `hello_time`) baked into the same `ip link add` invocation. The kernel bridge does have separate `bridge` subcommands for tweaking these post-creation, but a freshly-created bridge with `stp_state 0` (the default) does not run STP at all and would loop. Provisioning automation must remember to enable STP on bridge creation, not after.

After 12 seconds, read the per-bridge state from sysfs. The kernel exposes the spanning-tree state through files under `/sys/class/net/<bridge>/bridge/` for bridge-level state and `/sys/class/net/<port>/brport/state` for per-port state.

```text
root@lab-openvswitch:~/network-onboard# for b in lbr1 lbr2 lbr3; do echo == $b ==; cat /sys/class/net/$b/bridge/root_id; cat /sys/class/net/$b/bridge/bridge_id; cat /sys/class/net/$b/bridge/topology_change; done
== lbr1 ==
1000.0e41214791dc
1000.0e41214791dc
1
== lbr2 ==
1000.0e41214791dc
2000.f6d287caf75a
1
== lbr3 ==
1000.0e41214791dc
3000.e23f223d73ca
1
```

The bridge ID format is `<priority-hex>.<MAC>`, where `1000` is hex for the decimal 4096 set on lbr1, `2000` is 8192 (lbr2), and `3000` is 12288 (lbr3). All three bridges report the same `root_id` of `1000.0e41214791dc`, which matches lbr1's bridge_id. The election agrees with phase A: the lowest-priority bridge wins. The `topology_change` flag is 1 on all three, meaning the bridges are still propagating the most recent topology change (one of the link-up events that triggered convergence); under steady state with no further events the flag would clear after the topology-change-timer expires.

The per-port states are read from `/sys/class/net/<port>/brport/state`. The kernel uses integer codes: 0 = Disabled, 1 = Listening, 2 = Learning, 3 = Forwarding, 4 = Blocking.

```text
root@lab-openvswitch:~/network-onboard# for p in lv12a lv12b lv23a lv23b lv13a lv13b; do echo $p state=$(cat /sys/class/net/$p/brport/state 2>/dev/null); done
lv12a state=3
lv12b state=3
lv23a state=3
lv23b state=4
lv13a state=3
lv13b state=3
```

Five ports are in state 3 (Forwarding) and one port `lv23b` is in state 4 (**Blocking**). The blocked port is the same one phase A blocked: the br3-side end of the redundant br2-br3 link. The same loop-prevention answer reached through a different code path with different state names. The Linux STP "Blocking" state and the OVS RSTP "Discarding" state are not identical from the protocol's point of view (RSTP adds the Alternate role, which Blocking does not have), but their forwarding-plane effect is the same: this port carries no user frames.

## What R2.5.3 establishes for the comparative

Three findings to pin from this lab.

The first finding is that both implementations reach the same loop-prevention outcome on the same triangular topology. With deterministic priorities, both elect the lowest-priority bridge as root, both have the other two bridges choose the direct link to root as their root port, and both block exactly one end of the redundant link between the two non-root bridges. The active topology is a tree rooted at the same bridge in both phases, and the blocked port is at the same location in the topology. Two independent state-machine implementations, one shared answer.

The second finding is that the protocol generation differs. OVS implements 802.1D-2004 RSTP with the modern role taxonomy (Root, Designated, Alternate, Backup) and the modern state taxonomy (Discarding, Learning, Forwarding). The Linux kernel bridge implements 802.1D STP only, with the legacy state taxonomy (Disabled, Listening, Learning, Forwarding, Blocking). Operationally this matters in two ways. First, the convergence time is materially different: RSTP converges in seconds, classic STP takes tens of seconds with default timers. Second, the role information available to an operator differs: RSTP exposes which port would take over if the active path fails (the Alternate role), while classic STP only knows that a port is Blocking with no equivalent of the standby-failover role. For a deployment that needs sub-second convergence, the kernel-bridge alone is not sufficient and an out-of-tree daemon (`mstpd`) plus a different control surface is required.

The third finding is the configuration-time-versus-runtime asymmetry. With OVS, `rstp_enable=true` can be flipped on or off at any time via OVSDB transaction; the daemon brings the state machine up or down accordingly. With the Linux bridge, the `stp_state` parameter is normally set at `ip link add` time; though it can be changed at runtime with `ip link set <br> type bridge stp_state 1`, the timer parameters (`forward_delay`, `hello_time`, `max_age`) are also kernel-side and any change cascades through the running state machine in ways that can produce transient instabilities. For a multi-tenant deployment that needs to enable or disable STP on a live bridge without disturbing the workloads attached, the OVS knob is more comfortable.

The R2.5 sprint continues with sub-task 4 (link aggregation and LACP), where both OVS and the kernel bridge can build bonded interfaces but use different machinery: OVS provides bond modes (`active-backup`, `balance-slb`, `balance-tcp`) plus optional LACP via the OVS-specific implementation in `ofproto/bond.c`, while the Linux kernel offers the `bonding` driver as a separate netdev that the bridge can attach to.

## 13.1.3 ovn-northd translator

`ovn-northd` daemon read NBDB, compute logical flow, write SBDB. Translate high-level ý đồ cấu hình thành sequence logical flow.

Flow pipeline nhiều tables (≈ 23 LS ingress + 9 LS egress + 20 LR ingress + 7 LR egress trong OVN v22.03.8 — verified qua `mcp__github__get_file_contents(northd/northd.c, ref=v22.03.8)` grep `"ls_in_"`, `"ls_out_"`, `"lr_in_"`, `"lr_out_"`; tổng stage count thay đổi giữa minor version):

- **Ingress 0**: admission control.
- **Ingress 1-10**: L2 policy (port security, ACL, LB).
- **Ingress 11-20**: L3 policy (routing, NAT).
- **Ingress 21-23**: egress port selection.
- **Egress 0-27**: mirror ingress cho reverse direction.

Khi admin thêm `ovn-nbctl acl-add`, northd recompute pipeline và update SBDB.

---

## 13.1.4 ovn-controller: per-chassis agent

Mỗi compute chạy `ovn-controller` — daemon connect SBDB, filter row liên quan (`Port_Binding.chassis == local`), translate logical flow thành OpenFlow rule trên `br-int`.

Filter qua monitor_cond — chassis chỉ nhận row liên quan.

`ovn-controller` cũng:
- Advertise local chassis trong `Chassis` table.
- Create tunnel port Geneve đến remote chassis.
- Handle MAC binding cho ARP responder.

## 13.1.5 Workflow end-to-end

**Admin tạo logical switch + port**:

```bash
[admin]# ovn-nbctl ls-add ls1
[admin]# ovn-nbctl lsp-add ls1 vm1
[admin]# ovn-nbctl lsp-set-addresses vm1 "aa:bb:cc:00:00:01 10.0.0.1"
```

Workflow:

1. `ovn-nbctl` write NBDB.
2. `ovn-northd` phát hiện change, compute Datapath_Binding + Logical_Flow, write SBDB.
3. `ovn-controller` trên chassis nơi vm1 schedule phát hiện Port_Binding change.
4. ovn-controller translate logical flow → OpenFlow.
5. Khi vm1 khởi động, OVS port tagged `external_ids:iface-id=vm1` → ovn-controller match, update Port_Binding.chassis.
6. Traffic từ vm1 hit rule đã cài.

## Điểm cốt lõi cần nhớ

Thứ nhất, NBDB là ý đồ cấu hình storage (WYSIWYG cho admin); SBDB là compiled form (10×+). Admin chỉ touch NBDB.

Thứ hai, `ovn-northd` là compiler — translate NB → SB. Nếu northd crash, existing SB flow vẫn work nhưng NB change không propagate.

Thứ ba, `ovn-controller` per-chassis là distributed control plane — scale tuyến tính.

Thứ tư, pipeline logical flow hàng chục stage (OVN v22.03.8 có ≈ 23 LS ingress + 9 LS egress + 20 LR ingress + 7 LR egress — con số chính xác xem §13.2.5) — nhiều hơn OpenFlow native. northd compile logical abstraction, ovn-controller translate xuống OpenFlow thực.

## 13.1.7 NBDB schema exhaustive (Phase H S48)

OVN Northbound DB 22.03 có 17 table chính. Mục này Template cho mỗi table với purpose + key column + reference từ field khác.

### 13.1.7.1 Table tổng hợp NBDB

| # | Table | Purpose | Key column |
|---|---|---|---|
| 1 | `NB_Global` | Global settings, connection/SSL options | name, nb_cfg, sb_cfg, options |
| 2 | `Logical_Switch` | L2 broadcast domain | name, ports, acls, load_balancer, qos_rules |
| 3 | `Logical_Switch_Port` (LSP) | Port trên LS (VM interface) | name, addresses, type, options, port_security |
| 4 | `Logical_Router` | L3 router | name, ports, static_routes, nat, load_balancer |
| 5 | `Logical_Router_Port` (LRP) | Port trên LR (gateway interface) | name, networks, peer, mac |
| 6 | `Logical_Router_Static_Route` | Static route entry | ip_prefix, nexthop, output_port |
| 7 | `Logical_Router_Policy` | Policy-based routing | priority, match, action, nexthop |
| 8 | `ACL` | Access Control List | direction, priority, match, action, log |
| 9 | `Address_Set` | Named set of IP (dùng trong ACL khớp với `$name`) | name, addresses |
| 10 | `Port_Group` | Named set of LSP (dùng trong ACL khớp với `@name`) | name, ports, acls |
| 11 | `Load_Balancer` | L4 LB VIP → backend | name, vips, protocol, ip_port_mappings |
| 12 | `NAT` | SNAT / DNAT / SNAT+DNAT rule | type, external_ip, logical_ip, logical_port |
| 13 | `DHCP_Options` | DHCPv4/v6 option set | cidr, options |
| 14 | `DNS` | DNS zone mapping | records, external_ids |
| 15 | `QoS` | QoS policy cho LS | priority, match, action, bandwidth |
| 16 | `Meter` | Meter rate limit | name, bands, unit |
| 17 | `Copp` | Control Plane Protection (rate limit per control plane type) | name, meters |

### 13.1.7.3 Template deep cho `ACL`

**Key column:**

| Column | Value range | Purpose |
|---|---|---|
| `direction` | `from-lport` / `to-lport` | Ingress vs egress direction |
| `priority` | 0-32767 | 32767 highest win |
| `match` | String expression | OVN khớp với syntax (compile to OpenFlow) |
| `action` | `allow` / `allow-related` / `drop` / `reject` / `pass` | Verdict |
| `log` | boolean | Log to controller (OFPT_PACKET_IN) |
| `severity` | `alert` / `warning` / `notice` / `info` / `debug` | Log severity |
| `meter` | reference `Meter` | Rate limit on log/allow |
| `options:apply-after-lb` | boolean | Evaluate after LB (use `ls_in_acl_after_lb_*` stage) |

```shell
$ sudo ovn-nbctl --log --severity=notice acl-add ls-prod from-lport 1000 \
    "inport == @pg_web && ip4.dst == $ip_db && tcp.dst == 3306" allow-related
$ sudo ovn-nbctl acl-list ls-prod
  1000 allow-related (inport == @pg_web && ip4.dst == $ip_db && tcp.dst == 3306) [log]
```

### 13.1.7.4 Template deep cho `NAT`

| Column | Value | Purpose |
|---|---|---|
| `type` | `snat` / `dnat` / `dnat_and_snat` | NAT direction |
| `external_ip` | IPv4/IPv6 | External (post-NAT) address |
| `external_mac` | MAC optional | Static MAC cho gateway ARP reply |
| `logical_ip` | IPv4/IPv6 or CIDR | Internal (pre-NAT) address hoặc range |
| `logical_port` | LSP name optional | Restrict NAT cho specific LSP |
| `options:stateless` | boolean | Stateless NAT (rare, LB only) |

```shell
# SNAT: cả subnet 10.0.0.0/24 → external IP 203.0.113.100
$ sudo ovn-nbctl lr-nat-add lr-tenant snat 203.0.113.100 10.0.0.0/24

# DNAT+SNAT: floating IP 203.0.113.50 ↔ 10.0.0.5
$ sudo ovn-nbctl lr-nat-add lr-tenant dnat_and_snat \
    203.0.113.50 10.0.0.5 lsp-vm-web 40:44:00:00:00:50

# List
$ sudo ovn-nbctl lr-nat-list lr-tenant
TYPE             EXTERNAL_IP   LOGICAL_IP    EXTERNAL_MAC       LOGICAL_PORT
dnat_and_snat    203.0.113.50  10.0.0.5      40:44:00:00:00:50  lsp-vm-web
snat             203.0.113.100 10.0.0.0/24
```

### 13.1.7.5 Template deep cho `Static_Route`

| Column | Value | Purpose |
|---|---|---|
| `ip_prefix` | CIDR | Match destination |
| `nexthop` | IPv4/IPv6 | Next-hop address |
| `output_port` | LRP name optional | Force specific egress port |
| `policy` | `src-ip` / `dst-ip` | Match based on source or destination IP |
| `options:ecmp_symmetric_reply` | boolean | Symmetric ECMP for stateful traffic |

```shell
$ sudo ovn-nbctl lr-route-add lr-tenant 0.0.0.0/0 10.200.0.254 lrp-uplink
$ sudo ovn-nbctl lr-route-add lr-tenant 192.168.0.0/24 10.100.0.254
$ sudo ovn-nbctl lr-route-list lr-tenant

                0.0.0.0/0                 10.200.0.254 dst-ip lrp-uplink

```

### 13.1.7.6 Copp — Control Plane Protection (ít được biết)

`Copp` table rate-limit control plane traffic (ARP request, ICMP redirect, DHCP, DNS). Tránh DDoS control plane.

```shell
# Tạo meter cho ARP rate limit
$ sudo ovn-nbctl --id=@m create Meter name=copp_arp unit=pktps \
    bands=@band -- --id=@band create Meter_Band action=drop rate=1000

# Tạo Copp + assign meter cho arp
$ sudo ovn-nbctl --id=@c create Copp name=default meters:arp=@m

# Assign Copp vào Logical_Router
$ sudo ovn-nbctl set Logical_Router lr-tenant copp=@c
```

## 13.1.X Anatomy Template A — `ovn-nbctl show` + `ovn-sbctl list Datapath_Binding` (v3.2 expansion)

**Bối cảnh khi chạy:** Sau khi provisioning workflow xong (LS + LSP + LR + LRP trong NBDB), verify cả 2 tier NBDB ý đồ cấu hình + SBDB computed state trước khi test traffic.

**Command 1 — `ovn-nbctl show` (NBDB ý đồ cấu hình):**

```shell
$ sudo ovn-nbctl show

switch 7c9d4e2a-1d5e-4b7a-8e9f-abcdef012345 (ls-tenant-web)
    port lsp_web_01
        addresses: ["aa:bb:cc:00:00:01 10.0.0.5"]
    port lsp_web_02
        addresses: ["aa:bb:cc:00:00:02 10.0.0.6"]
    port ls-to-lr-tenant
        type: router
        router-port: lrp-ls-tenant
router 5f8a9c42-... (lr-tenant)
    port lrp-ls-tenant
        mac: "40:44:00:00:00:01"
        networks: ["10.0.0.1/24"]
    port lrp-external
        mac: "40:44:00:00:ff:01"
        networks: ["192.0.2.1/24"]
        gateway chassis: [chassis-1 chassis-2]
```

**Command 2 — `ovn-sbctl list Datapath_Binding` (SBDB compiled):**

```shell
$ sudo ovn-sbctl --columns=_uuid,external_ids,tunnel_key list Datapath_Binding

_uuid          : a1b2c3d4-...
external_ids   : {logical-switch="7c9d4e2a-...", name="ls-tenant-web"}
tunnel_key     : 5

_uuid          : d5e6f7a8-...
external_ids   : {logical-router="5f8a9c42-...", name="lr-tenant"}
tunnel_key     : 10
```

**9-attribute anatomy (combine 2 command):**

| Field | Giá trị mẫu | Ý nghĩa | Red flag nếu |
|-------|-------------|---------|--------------|
| `switch <uuid>` | `7c9d4e2a-...` | NBDB LS identity | Missing sau create = schema corruption |
| `port <name>` + type | `lsp_web_01`, `type: router` | LSP identity + type | LSP type mismatch expected = config bug |
| `router-port` linkage | `router-port: lrp-ls-tenant` | LSP-to-LRP binding | Missing = LS disconnect from LR |
| SBDB `external_ids:logical-switch` | `"7c9d4e2a-..."` | Back-reference NBDB LS UUID | Mismatch với NBDB = northd compile lag |
| SBDB `tunnel_key` | 5 (LS), 10 (LR) | Per-datapath unique ID 1-16777215 | Collision giữa datapath = northd bug |
| `gateway chassis` list | `[chassis-1 chassis-2]` | HA_Chassis_Group resolution | Empty = gateway LRP không functional |
| NBDB `addresses` | `"aa:bb:cc:... 10.0.0.5"` | LSP MAC + IP | `unknown` = learn-based, ARP responder không work |
| LR `networks` | `"10.0.0.1/24"` | LRP IP + subnet | Mismatch với LS subnet = routing fail |
| LR `mac` | `"40:44:00:00:00:01"` | Virtual router MAC | OVN-generated nếu null, custom nếu set |

**Kịch bản bẻ gãy:**

- **NBDB LS tồn tại nhưng SBDB Datapath_Binding không có**: `ovn-northd` crash hoặc stuck. Check `ovn-appctl -t ovn-northd status`. Fix: restart `ovn-northd` hoặc investigate log.
- **SBDB Datapath_Binding có nhưng `tunnel_key` collision**: bug hiếm OVN 20.06-21.03, fixed 22.03. Upgrade OVN.
- **LSP type=router nhưng `router-port` empty**: LSP orphan, không connect LRP. Fix qua `ovn-nbctl lsp-set-type`.
- **LR `gateway chassis` empty**: HA_Chassis_Group reference broken. Check `ovn-nbctl ha-chassis-group-list`.

## 13.1.Y Hiểu sai + Key Topic (v3.2 expansion)

> **Hiểu sai phổ biến 1:** *"Admin có thể edit trực tiếp SBDB để nhanh hơn NBDB + wait northd compile."*
>
> **Thực tế:** SBDB là computed state — sinh bởi `ovn-northd` từ NBDB. Manual edit SBDB sẽ bị overwrite ngay lập tức khi `ovn-northd` next recompute. Không có persistent effect. Admin **luôn luôn** edit NBDB và chờ northd propagate. Exception hiếm: manual add Nút mạng (Chassis) row trong SBDB để register chassis mới (bypass ovn-controller registration) — dùng chỉ trong disaster recovery.

> **Hiểu sai phổ biến 2:** *"`ovn-northd` là single point of failure — nếu crash, OVN data plane stop."*
>
> **Thực tế:** `ovn-northd` crash KHÔNG làm data plane stop. Existing SBDB Logical_Flow vẫn có, ovn-controller vẫn apply OpenFlow trên br-int, VM traffic continue. Chỉ có new ý đồ cấu hình (NBDB change) không propagate tới SBDB. VMs đã running giữ network. Đây là desync window graceful. Northd có thể HA qua active-standby: `ovn-northd` multiple instance, qua leader election via SBDB chassis-specific config. Production deployment thường 2 northd (active + standby), failover 2-5s.

> **Chủ đề trọng tâm:** NBDB vs SBDB phân biệt là giáo trình gốc của OVN. NBDB = ý đồ cấu hình (desired state, WYSIWYG cho operator). SBDB = state (computed + runtime, 10-100× kích thước NBDB). Operator không bao giờ touch SBDB. Debug operator: `ovn-nbctl show` để understand ý đồ cấu hình. Debug infrastructure: `ovn-sbctl list Port_Binding` + `ovn-sbctl lflow-list` để understand compilation output. Mỗi bug thuộc 1 trong 2 layer: (a) ý đồ cấu hình wrong (operator error, edit NBDB) hoặc (b) compile lag/error (northd crash hoặc SBDB Raft issue).

## 13.1.Z ▶ Guided Exercise — Observe NBDB ý đồ cấu hình → SBDB compilation (v3.2 expansion)

**Mục tiêu:** Quan sát real-time compilation từ NBDB ý đồ cấu hình tới SBDB Logical_Flow + measurement timing.

**Môi trường:** Single-node dev từ Phần 13.0 GE.

**Bước:**

1. Monitor SBDB real-time (terminal 1):
   ```bash
   $ sudo ovsdb-client -v monitor \
         unix:/var/run/ovn/ovnsb_db.sock OVN_Southbound Logical_Flow \
         priority,match,actions 2>&1 | head -100
   ```
2. Trong terminal 2, time northd compile:
   ```bash
   $ time sudo ovn-nbctl --wait=sb ls-add ls-timing-test
   # --wait=sb block tới khi northd done compile
   # Output: real 50-200ms (depend SBDB load)
   ```
3. Add LSP với ACL → observe complex compilation:
   ```bash
   $ time sudo ovn-nbctl --wait=sb \
       lsp-add ls-timing-test lsp-timing \
         -- lsp-set-addresses lsp-timing "aa:bb:cc:00:00:10 10.0.0.10"
   $ time sudo ovn-nbctl --wait=sb \
       acl-add ls-timing-test from-lport 1000 \
         "outport==@pg_test && tcp.dst==80" allow-related
   # Observe compile time scaling với complexity
   ```

**POE:**
- **Predict:** Compile time cho LS + 10 LSP + 10 ACL có scale linear không?
- **Observe:** Typically <500ms cho scale nhỏ này. Scale 1000 LSP → 2-5s (I-P engine delta compute).
- **Explain:** `ovn-northd` từ OVN 22.09+ dùng incremental processing — chỉ recompute affected subset. Trước 22.09, full recompute mỗi change → scale O(N²). I-P engine change 1 ACL → recompute chỉ relevant LS subset.

**Cleanup:** `ovn-nbctl ls-del ls-timing-test`

## Điểm cốt lõi cần nhớ

Thứ nhất, NBDB là ý đồ cấu hình storage (WYSIWYG cho admin); SBDB là compiled form (10×+). Admin chỉ touch NBDB.

Thứ hai, `ovn-northd` là compiler — translate NB → SB. Nếu northd crash, existing SB flow vẫn work nhưng NB change không propagate.

Thứ ba, `ovn-controller` per-chassis là distributed control plane — scale tuyến tính.

Thứ tư, pipeline logical flow hàng chục stage (OVN v22.03.8 có ≈ 23 LS ingress + 9 LS egress + 20 LR ingress + 7 LR egress — con số chính xác xem §13.2.5) — nhiều hơn OpenFlow native. northd compile logical abstraction, ovn-controller translate xuống OpenFlow thực.

Thứ năm, NBDB schema có 17+ table chính: LS/LSP/LR/LRP/Static_Route/Policy/ACL/Port_Group/Address_Set/LB/NAT/DHCP_Options/DNS/QoS/Meter/Copp + NB_Global. Engineer OVN phải thuộc nhóm để điều khiển từ admin console hoặc IaC (Terraform, Ansible).

Thứ sáu, SBDB schema có 15+ table: Nút mạng (Chassis)/Port_Binding/Datapath_Binding/Logical_Flow/MAC_Binding/Multicast_Group/Service_Monitor/HA_Chassis_Group + Gateway_Chassis legacy. Engineer OVN debug phải thuộc để đọc `ovn-sbctl list Port_Binding` hoặc correlate với OpenFlow trên br-int.

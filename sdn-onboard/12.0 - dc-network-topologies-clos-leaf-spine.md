## 12.0.1 Lịch sử: Charles Clos 1953

Charles Clos (Bell Labs) công bố "A Study of Non-Blocking Switching Networks" năm 1953 — seminal paper về telephone switching. Ý tưởng: tạo non-blocking switch lớn từ nhiều switch nhỏ multi-stage.

Clos topology 3-stage: Ingress → Middle → Egress. Mỗi ingress connect đến mọi middle, tương tự middle → egress. Ratio M ≥ N đảm bảo non-blocking.

60 năm sau, Google/Facebook/Microsoft tái phát minh Clos cho data center — leaf-spine là 2-stage variant.

---

## 12.0.2 3-tier legacy (access/aggregation/core)

Pre-2010 DC topology:

```
                  [Core Router]
                  /     |     \
         [Aggr1] [Aggr2] [Aggr3]
          / \     / \     / \
        Access switches (top of rack)
          |    |     |     |
         Servers
```

Vấn đề:

- **North-South dominant** → Core điểm nghẽn khi modern workload East-West dominant.
- **STP domain** lớn, fragile.
- **Oversubscription cao** (20:4:1) → packet drop trong peak.
- **Không scale horizontal** — phải upgrade Aggr/Core.

---

## 12.0.3 Leaf-Spine (Clos 2-stage)

Modern DC topology từ 2010s:

```
       [Spine 1] [Spine 2] [Spine 3] [Spine 4]
          │    ╲ ╲    ╲  ╱  ╱ ╱   │
          │ full-mesh connectivity │
          │    ╱ ╱    ╱  ╲  ╲ ╲   │
       [Leaf 1] [Leaf 2] [Leaf 3] [Leaf N]
         │ │     │ │     │ │     │ │
        Servers  Servers  Servers  Servers
```

- Mỗi leaf connect đến mọi spine (full mesh).
- **ECMP** distribute traffic qua tất cả spine.
- **Không STP** — L3 routing (BGP unnumbered/OSPF).
- **Predictable latency** — max 3 hops giữa hai server.

**Scale horizontal**: thêm spine → tăng bandwidth. Thêm leaf → tăng port count.

## 12.0.4 Facebook F4/F16

**F4 (2014)**:
- 4-layer Clos với pod architecture.
- Scale hàng trăm nghìn servers một building.

**F16 (2019)**:
- Simplified architecture, smaller building blocks (Minipack — Broadcom Tomahawk 3).
- Higher radix spine → less tiers.
- 400G+ spine links.

Facebook chọn Broadcom merchant silicon + FBOSS (Facebook Open Switching System, Linux-based).

---

## 12.0.5 Google Jupiter

Google công bố Jupiter tại SIGCOMM 2015 ("Jupiter Rising: A Decade of Clos Topologies and Centralized Control in Google's Datacenter Network").

- **Bisection bandwidth 1.3 Pbps** — largest DC fabric in 2015.
- **Custom Google silicon**.
- **Centralized control plane** qua B4 — one of earliest production SDN.
- **ECMP + weighted traffic engineering**.

Jupiter đặt tiền đề cho ngành chấp nhận large-scale leaf-spine Clos với centralized control.

---

## 12.0.6 Bisection bandwidth và oversubscription

**Bisection bandwidth**: tổng bandwidth cross section bisection. Key metric đánh giá fabric capacity.

**Oversubscription ratio**: (server-facing bandwidth) / (uplink bandwidth) per leaf.

- 1:1 (non-blocking): leaf server 400G tổng → uplink 400G.
- 3:1 (common enterprise): server 1200G → uplink 400G.
- 6:1 (legacy, avoid).

Oversubscription cao → East-West traffic drop trong peak. Modern cloud DC push 1:1 cho bursty workload.

---

## Điểm cốt lõi cần nhớ

Thứ nhất, Clos topology 1953 vẫn là foundation cho modern DC fabric. Leaf-spine là 2-stage variant.

Thứ hai, 3-tier legacy không scale cho East-West traffic dominant. Leaf-spine giải quyết bằng ECMP + L3 routing.

Thứ ba, hyperscalers (Facebook F16, Google Jupiter) push architecture tới limits — petabit bisection, custom silicon.

Thứ tư, bisection bandwidth + oversubscription ratio là metric chính khi design fabric. Aim for 1:1 cho cloud workload.

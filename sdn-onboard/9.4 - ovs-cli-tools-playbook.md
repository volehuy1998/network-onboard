## Context: why a systematic playbook is needed

OVS is a layered system with three large components (the `ovs-vswitchd` user-space daemon, the `ovsdb-server` database daemon, and the kernel datapath module or the userspace DPDK or AF_XDP datapath) plus four configuration layers (OVSDB config, OpenFlow pipeline, megaflow cache, wire). When the incident *"the packet does not go through"* occurs, an ad-hoc approach (typing random commands to see what happens) wastes hours, opens many terminal tabs, and often fails to find the root cause.

A systematic playbook solves the problem by enforcing a fixed sequence of checks. Each step rules out one possible failure region. Compass Appendix C distils the philosophy into six questions:

1. Does the packet reach the NIC? Measure with `tcpdump` on the physical NIC.
2. Does the packet enter OVS? Measure the counter `lookups:missed` rising when traffic flows, with `ovs-dpctl show`.
3. Which flow matches the packet? Use `ovs-appctl ofproto/trace` to simulate the pipeline.
4. Is the datapath action correct? Read the line `Datapath actions:` in the trace output.
5. Does the datapath agree? Use `ovs-dpctl dump-flows` to read the actual kernel or netdev state.
6. Does the wire agree? `tcpdump` on the egress NIC.

If all six steps agree yet the packet still drops, the root cause lies **outside OVS**: an upstream switch, a firewall, an MTU mismatch, an unresolved ARP, a withdrawn BGP route. Stop debugging OVS and pivot to the network path.

This Part presents the five CLI tools at sufficient depth to execute the playbook, then integrates them into §9.4.6, the six-layer troubleshooting workflow. The structure of each tool section is: overall grammar, subcommand catalog, Anatomy block for important output, two to four operational scenarios, failure scenarios.

## 9.4.1 `ovs-vsctl`, the transactional configuration language

### 9.4.1.1 Grammar, atomicity, and why every configuration change goes through ovs-vsctl

`ovs-vsctl` is the OVSDB configuration management tool. Every *configuration* change on a bridge, port, interface, QoS, mirror, netflow, controller, or OpenFlow version goes through it. Compass Ch A states the core nature that many new engineers overlook:

> *"`ovs-vsctl` is not a shell wrapper around individual commands. It is a transactional compiler: a single invocation becomes one OVSDB transaction, and `--` is the statement separator inside that transaction."*

In other words, `ovs-vsctl -- A -- B -- C` compiles into one atomic transaction with three operations, committed atomically. If operation B fails, A is not applied. This is an important safeguard against the half-configured state. Without it, a production system can land in a state where the bridge is created but the port has not yet been attached, so traffic drops while the configuration is rolling out.

Overall grammar (man `ovs-vsctl(8)` §SYNOPSIS):

```
ovs-vsctl [global-options] command [-- command]...

global-options:
  --db=socket-path      (default: /var/run/openvswitch/db.sock)
  --timeout=seconds     (default: infinite)
  --no-wait
  --dry-run
  --oneline             (compact output)
  --pretty              (pretty-print JSON-like values)

command:
  <verb> <table> [<record>] [<column>[:<key>][=<value>]]...

verbs for scalar/composite column:
  list    find   get   set   add   remove   clear

verbs for row:
  create   destroy

helpers:
  --id=@name            (bind record to symbolic name in transaction)
  --may-exist           (succeed if row already matches)
  --if-exists           (succeed if row missing)
```

Cross-table reference within the same transaction uses `--id=@name`. The classic example (create a bridge plus add a port in one transaction):

```shell
$ ovs-vsctl add-br br0 -- add-port br0 eth0
```

If `add-port` fails (because `eth0` already belongs to another bridge), `br0` is not created. There is no half-created intermediate state.

### 9.4.1.3 Reading the output of `ovs-vsctl show`

**Context when running:** this is the first command an on-call engineer types after SSH into the host. It dumps the entire bridge, port, and interface topology at the OVSDB level, quickly and at low cost. Think of it as the `ip addr show` of OVS.

**Command and output** `[real capture, Ubuntu 22.04 plus OVS 2.17.9 plus OVN 22.03.8, node huyvl-lab-ovs01]`:

```shell
$ sudo ovs-vsctl show
c5e7a4c2-1b9a-4e3f-8a44-3f7e9b1c2d85
    Bridge br-int
        fail_mode: secure
        datapath_type: system
        Port br-int
            Interface br-int
                type: internal
        Port ovn-huyvl-0
            Interface ovn-huyvl-0
                type: geneve
                options: {csum="true", key=flow, remote_ip="10.200.0.12"}
        Port ovn-huyvl-1
            Interface ovn-huyvl-1
                type: geneve
                options: {csum="true", key=flow, remote_ip="10.200.0.13"}
        Port tap-vm-web
            Interface tap-vm-web
                type: internal
        Port tap-vm-db
            tag: 100
            Interface tap-vm-db
    Bridge br-ex
        fail_mode: standalone
        datapath_type: system
        Port br-ex
            Interface br-ex
                type: internal
        Port eno1
            Interface eno1
    ovs_version: "2.17.9"
```

**Anatomy of each line:**

| Output line | Meaning | Anomaly signal |
|---|---|---|
| `c5e7a4c2-…` (first line) | UUID of the Open_vSwitch record, unique per host; never duplicated | Missing UUID, `ovsdb-server` has not loaded the schema; check `journalctl -u ovsdb-server` |
| `Bridge br-int` | One bridge entry; each bridge renders one block | No `br-int` after OVN setup, `ovn-controller` has not started |
| `fail_mode: secure` | When the controller disconnects: `secure` (drop every packet that does not match a flow) or `standalone` (learn-normal). `secure` is the OVN default | `standalone` on an OVN br-int exposes a vulnerability when the OVN controller goes down; traffic floods |
| `datapath_type: system` | Kernel datapath (default). `netdev` is userspace DPDK or AF_XDP | `netdev` without DPDK configured, OVS falls back to AF_XDP and performance is low |
| `Port <name>` | A port entry belonging to the current bridge | - |
| `Interface <name>` | An interface entry belonging to the current port (usually 1:1; a bond has N interfaces on one port) | - |
| `type: internal` | An OVS-managed TAP; there is a Linux netdev `internal` port | - |
| `type: geneve` | Overlay tunnel; the port was built automatically by OVN | Missing the `key=flow` option, OVN cannot inject `tun_id` per packet and the overlay is broken |
| `options: {csum="true", key=flow, remote_ip="10.200.0.12"}` | Tunnel options: csum (UDP checksum), `key=flow` (tun_id from a flow action), `remote_ip` for the far endpoint | `key=<integer>` instead of `key=flow`, OVN cannot use it (OVN needs a per-flow tun_id) |
| `tag: 100` | Access VLAN tag on the port; a packet entering this port has `vlan_tci` set to 100 | A wrong or missing tag sends traffic into the wrong broadcast domain |
| `ovs_version: "2.17.9"` | User-space OVS version | Below 2.17 may not support some OpenFlow 1.5 features; below 2.15 has no SMC cache |

**Failure scenarios:**

- **If the `ovs_version` line does not appear**, then `ovsdb-server` is not running or is not responding within the timeout; run `sudo systemctl status ovsdb-server` and `sudo ovs-vsctl --timeout=3 show`.
- **If a bridge has `datapath_type: netdev` while the host is not configured for DPDK**, then OVS falls back to the kernel datapath via AF_XDP and throughput can drop by 30 to 40 percent; fix with `ovs-vsctl set Bridge br-int datapath_type=system`.
- **If `ovn-*` ports are absent after OVN has provisioned an LSP**, then `ovn-controller` has not picked up the binding; check `ovn-sbctl show` to see whether the chassis has registered and `ovn-sbctl list Port_Binding` to see whether the binding has a chassis assigned.

**Upstream sources:**
- man `ovs-vsctl(8)` §`DATABASE COMMANDS` and §`SUMMARIZING DATABASE CONTENTS`.
- `ovs-vswitchd.conf.db(5)` for the complete Bridge, Port, and Interface schema.

### 9.4.1.4 Operational scenario: create a bridge with an internal port that has an IP

**Context:** building a development OVS host that needs a bridge with a VLAN access port as the internal gateway. This is the common pattern in kolla-ansible when configuring br-ex.

```shell
# Step 1: create the bridge
$ sudo ovs-vsctl add-br br-demo

# Step 2: verify
$ sudo ovs-vsctl list-br
br-demo
br-ex
br-int

# Step 3: add a VLAN access port tag=10 with an internal interface
$ sudo ovs-vsctl add-port br-demo vlan10 tag=10 -- set Interface vlan10 type=internal

# Step 4: assign a Linux netdev IP to the internal port
$ sudo ip link set vlan10 up
$ sudo ip addr add 192.168.10.1/24 dev vlan10

# Step 5: verify all three operations applied atomically
$ sudo ovs-vsctl list Port vlan10
_uuid               : 8f3a2b4c-...
name                : vlan10
tag                 : 10
interfaces          : [1a2b3c4d-...]
...

$ sudo ovs-vsctl list Interface vlan10
_uuid               : 1a2b3c4d-...
name                : vlan10
type                : internal
admin_state         : up
link_state          : up
mac_in_use          : "3e:6a:...:..."
mtu                 : 1500
...

# Step 6 (teardown): delete the bridge; every port detaches automatically
$ sudo ovs-vsctl del-br br-demo
```

**Key points:**

- `tag=10` in `add-port` sets the `tag` column of the `Port` table. This is an access VLAN, different from `trunks=10,20,30` (trunk port).
- `type=internal` on the Interface creates a virtual Linux netdev. Without `internal`, the interface must be the name of a real device (`eth0`, `enp3s0`).
- `add-port` and `set Interface` commit atomically because they share one `ovs-vsctl` invocation.

### 9.4.1.5 Scenario: configure two QoS queues with a complex transaction

**Context:** lifted from man `ovs-vsctl(8)` §`CONFIGURATION COOKBOOK, Quality of Service`. Create a QoS record with two queues to split bandwidth into 100 Mbps and 500 Mbps for eth0 and eth1.

```shell
$ sudo ovs-vsctl \
    -- set Port eth0 qos=@newqos \
    -- set Port eth1 qos=@newqos \
    -- --id=@newqos create QoS type=linux-htb \
         other-config:max-rate=1000000000 \
         queues=0=@q0,1=@q1 \
    -- --id=@q0 create Queue other-config:min-rate=100000000 \
         other-config:max-rate=100000000 \
    -- --id=@q1 create Queue other-config:min-rate=500000000
```

**Anatomy of the transaction:**

| Line | Operation | OVSDB result |
|---|---|---|
| `set Port eth0 qos=@newqos` | Assign QoS (does not yet exist, bind symbolically) | The `qos` column of the Port `eth0` row points to a UUID resolved later |
| `set Port eth1 qos=@newqos` | Assign the same QoS to eth1 | The two ports share the same QoS record |
| `create QoS type=linux-htb` | Create an HTB QoS record with a 1 Gbps total max-rate | `--id=@newqos` binds the new UUID to the symbol `@newqos` |
| `queues=0=@q0,1=@q1` | The `queues` column (map) assigns queue-id 0 to the Queue record `@q0` and 1 to `@q1` | Cross-table reference |
| `create Queue` with `min-rate=100M max-rate=100M` | Queue 0 capped at 100 Mbps | - |
| `create Queue` with `min-rate=500M` | Queue 1 guarantees 500 Mbps minimum, with no cap | - |

The six operations commit atomically. If, for example, `create QoS` fails (syntax error), the two `set Port` operations are not applied and eth0 and eth1 are not assigned a broken QoS.

**Failure scenarios:**

- **A queue without a max-rate cap** (only min-rate). HTB shares the available bandwidth, which can exceed the intended level when other ports are idle. This may be intentional, but document it.
- **`max-rate` equal to `min-rate` on a Queue.** The target max-rate cannot be reached because min-rate is a hard floor. This is coerced by the HTB algorithm.
- **Forgetting to clear when deleting QoS.** You must run `ovs-vsctl clear Port eth0 qos` plus `destroy QoS <uuid>`. If you only `destroy`, the reference from Port still exists and OVSDB raises a "dangling reference" error.

**Upstream:** man `ovs-vsctl(8)` §`CONFIGURATION COOKBOOK, Quality of Service`.

### 9.4.1.6 Scenario: query `find` plus `get` for script automation

**Context:** a monitoring script needs to pull state periodically. `show` is not parse-friendly; `list` is long; `find` plus `get` is machine-formatted.

```shell
# Find every Port with VLAN tag 100
$ sudo ovs-vsctl find Port tag=100
_uuid               : 7c9d4e2a-1f8b-4c3d-a7e6-5b2a9d1e8f3c
name                : "tap-vm-db"
tag                 : 100
interfaces          : [6e1a3b8c-2d4f-4a5e-b8c7-9d3e1a2b7f5c]
...

# Find Interfaces with RX errors greater than 0
$ sudo ovs-vsctl --columns=name,statistics find Interface 'statistics:rx_errors>0'
name                : eth1
statistics          : {..., rx_errors=42, ...}

# Get one scalar value (script-friendly)
$ sudo ovs-vsctl get Interface eno1 statistics:rx_bytes
184756234890

# Get a value from a nested map
$ sudo ovs-vsctl get Interface ovn-huyvl-0 options:remote_ip
"10.200.0.12"

# List bridges with name, fail_mode, and datapath_type as a table
$ sudo ovs-vsctl --columns=name,fail_mode,datapath_type list Bridge
_uuid               : 3a8b9c4d-...
name                : br-int
fail_mode           : secure
datapath_type       : system

_uuid               : 5e7f8a9b-...
name                : br-ex
fail_mode           : standalone
datapath_type       : system
```

**Anatomy of the query operators:**

| Operator | Meaning | Example |
|---|---|---|
| `COLUMN=VALUE` | Exact match | `tag=100` |
| `COLUMN!=VALUE` | Not equal | `fail_mode!=secure` |
| `COLUMN<VALUE` | Less than (numeric) | `statistics:rx_bytes<1000` |
| `COLUMN>VALUE` | Greater than | `statistics:rx_errors>0` |
| `COLUMN{=}{A,B}` | Set membership | `protocols{=}{OpenFlow13,OpenFlow15}` |
| `COLUMN:KEY=VALUE` | Map entry | `options:remote_ip=10.0.0.1` |

Production script example: poll the RX error rate per interface.

```bash
#!/bin/bash
for iface in $(sudo ovs-vsctl --bare --columns=name list Interface); do
  rx_err=$(sudo ovs-vsctl get Interface "$iface" statistics:rx_errors 2>/dev/null || echo 0)
  rx_bytes=$(sudo ovs-vsctl get Interface "$iface" statistics:rx_bytes 2>/dev/null || echo 0)
  if [ "$rx_err" -gt 0 ]; then
    echo "INTERFACE $iface: rx_errors=$rx_err rx_bytes=$rx_bytes"
  fi
done
```

## 9.4.2 `ovs-ofctl`, OpenFlow flow management

### 9.4.2.1 Grammar, version negotiation, and flow syntax

`ovs-ofctl` speaks OpenFlow to the bridge: dump flows, add, modify, or delete flow entries, and watch the event stream. It is the only **write-heavy** tool for pipeline programming. `ovs-vsctl` cannot modify flows, only configure the bridge.

Overall grammar (man `ovs-ofctl(8)` §SYNOPSIS):

```
ovs-ofctl [OPTIONS] COMMAND [SWITCH] [ARGS...]

key OPTIONS:
  -O, --protocols=VERSION[,VERSION...]
       OpenFlow10 | OpenFlow11 | OpenFlow12 | OpenFlow13 | OpenFlow14 | OpenFlow15
  --strict           (match exactly, no wildcards)
  --bundle           (atomic group using OF 1.4 bundle transactions)
  --sort[=FIELD]     (sort dump-flows output)
  --rsort[=FIELD]
  --color[=never|always|auto]
  --names | --no-names (port names vs numbers)

COMMANDS (switch management):
  show SWITCH
  dump-tables SWITCH
  dump-ports SWITCH [NETDEV]
  dump-flows SWITCH [FLOWSPEC]
  dump-aggregate SWITCH [FLOWSPEC]
  add-flow SWITCH FLOW
  add-flows SWITCH FILE
  mod-flows [--strict] SWITCH FLOW
  del-flows [--strict] SWITCH [FLOW]
  replace-flows [--readd] SWITCH FILE
  diff-flows SOURCE1 SOURCE2
  add-group SWITCH GROUP    /  dump-groups SWITCH [GROUP]
  add-meter SWITCH METER    /  dump-meters SWITCH [METER]
  monitor SWITCH [MISS-LEN] [invalid_ttl] [watch:SPEC...]
  snoop SWITCH
```

**Important:** ALWAYS pass `-O OpenFlow13` (or 1.4, 1.5). The default for `ovs-ofctl` is OpenFlow 1.0, an old version that does not support multi-table, group, meter, TLV match, and so on. Without the flag, pipeline references fail on a modern setup (OVN requires OpenFlow 1.4 or later).

### 9.4.2.2 Flow syntax and control keywords

A flow spec is a string of `FIELD=VALUE` pairs separated by commas or whitespace. When a value contains spaces, quote it. The important control keywords are:

| Keyword | Meaning | Default | When absent |
|---|---|---|---|
| `table=<id>` | Restrict the operation to a table ID (0 to 254) | 0 for `add-flow`; wildcard for dump and del | All tables |
| `priority=<val>` | Priority 0 to 65535; higher matches first | 32768 | 32768 (not at the top) |
| `cookie=<val>[/<mask>]` | Opaque 64-bit ID, used to group or tag flows | 0 | - |
| `idle_timeout=<sec>` | Flow expires after N seconds without a packet hit | 0 (no timeout) | - |
| `hard_timeout=<sec>` | Flow expires after N absolute seconds | 0 | - |
| `actions=<spec>` | Action list (comma-separated) | Required | Flow is rejected if absent |
| `duration=...`, `n_packets=...`, `n_bytes=...` | Ignored in add or mod (dump-only display) | - | - |

For match fields, see `ovs-fields(7)`. A complex flow spec example:

```
table=10,priority=1000,
  cookie=0xdead1000/0xffff0000,
  idle_timeout=60,hard_timeout=600,
  dl_type=0x0800,nw_proto=6,tp_dst=80,
  actions=mod_dl_dst:aa:bb:cc:dd:ee:ff,
          mod_nw_src:10.1.0.100,
          goto_table:20
```

Reading: table 10, priority 1000, high cookie masked at 0xffff, idle 60 seconds and hard 600 seconds, match TCP destination port 80, three actions (rewrite the destination MAC, rewrite the source IP, then goto table 20).

**Anatomy of each line:**

| Component | Meaning | Anomaly signal |
|---|---|---|
| `OFPT_FEATURES_REPLY (OF1.5)` | Version negotiation succeeded at OF 1.5 | If it shows `(OF1.0)`, then you forgot `-O OpenFlow15` and used the wrong version |
| `dpid:0000000000000001` | 64-bit datapath ID, unique per switch | Duplicate dpid between two bridges, the controller mistakenly sees one switch |
| `n_tables:254` | Number of OpenFlow tables supported. OVS default is 254 | Below 254, the bridge is restricted; check `ovs-vsctl get Bridge br-int flow_tables` |
| `n_buffers:0` | OVS no longer buffers packet_in (since OF 1.4) | - |
| `capabilities` | List of switch capability flags (FLOW_STATS, TABLE_STATS, PORT_STATS, GROUP_STATS, QUEUE_STATS, ARP_MATCH_IP) | Missing `GROUP_STATS`, multicast Group counters are not accurate |
| `actions: (none)` | OF 1.1 onwards does not hard-code actions and returns `(none)` | - |
| `1(tap-vm-web)` | Port number `1`, name `tap-vm-web` | If the port number disappears, the VM was unplugged or `ovs-vswitchd` restarted and reassigned it |
| `addr:3e:6a:...` | MAC of the port | A 00:... MAC means the internal port has no MAC assigned; check the Interface `mac` column |
| `config: PORT_DOWN` | Administratively down (set by `ovs-ofctl mod-port PORT down`) | PORT_DOWN while traffic is expected indicates wrong user configuration |
| `state: LINK_DOWN` | Physical link layer is down | LINK_DOWN on a Geneve `ovn-*` port is normal (Geneve has no physical link) |
| `current: 10GB-FD COPPER` | Link speed, duplex, and media | An SFP module error displays `current:` as empty |
| `speed:` | Bandwidth report in Mbps | Geneve ports are always 0 Mbps (virtual) |
| `LOCAL(br-int)` | Port number `LOCAL` reserved for the bridge-internal port | Losing the LOCAL port indicates bridge corruption |
| `OFPT_GET_CONFIG_REPLY: frags=normal miss_send_len=0` | Bridge config: reassemble fragments normally; do not send header-only packet_in to the controller | `miss_send_len=N` means the bridge will send N bytes of packet_in, which can spam the controller |

**Failure scenarios:**

- **Persistent `state: LINK_DOWN` on port eth0**: a physical issue (cable, SFP, driver). Use `ethtool eth0` to verify autoneg and media type.
- **Port number not stable between `show` runs**: `ovs-vswitchd` just restarted; always use the port name (`tap-vm-web`) in flow specs instead of the port number (`1`) to avoid broken flows after a restart.
- **`(OF1.0)` instead of `(OF1.5)`**: the `-O` flag is missing. Fix: `ovs-ofctl -O OpenFlow15 show br-int`.

### 9.4.2.4 Reading the output of `ovs-ofctl dump-flows`

**Context:** the question "which flows are alive on the switch?" opens every pipeline debug. `dump-flows` is the answer, but the output can be thousands of lines in an OVN setup. Understanding the format is a small but pivotal step.

**Command and output** `[reproduced from OVS Advanced Tutorial, simplified 8-flow sample]`:


**Anatomy of each field in a dump-flows entry:**

| Field | Meaning | Anomaly signal |
|---|---|---|
| `cookie=<64-bit hex>` | Tag set by the controller when adding the flow. OVN uses the cookie to tag the logical flow UUID (high 32 bits) plus the datapath (low 32 bits) | `cookie=0x0` is the default; an OVN-generated flow must have a non-zero cookie |
| `duration=<N>s` | Time the flow has existed since install | Continuous `duration=0.xxxs` indicates flow reinstalls; check the OVN controller log |
| `table=<N>` | Table number 0 to 254. OF 1.1 onwards has multi-table | - |
| `n_packets=<N>` | Counter of packets matched by this flow | `n_packets=0` plus `duration > 600s` for an expected flow indicates the flow never matches; check the match spec |
| `n_bytes=<N>` | Counter of bytes matched | The ratio `n_bytes / n_packets` is the average packet size; compare with the expected MTU |
| `idle_timeout=<sec>` | Flow expires after N seconds without a packet hit. The field is only displayed when nonzero | If a critical flow (ACL) has an idle_timeout, the flow may delete itself; check the controller logic |
| `hard_timeout=<sec>` | Flow expires after N absolute seconds, hidden when zero | In OVN flows, hard_timeout is usually 0; a nonzero value indicates a hard-set by the controller, usually for `learn()` |
| `priority=<N>` | Match priority. With two flows both matching, the highest priority wins | Two flows with the same priority and overlapping matches give undefined behaviour |
| `<match spec>` | Match field list, for example `in_port=1,ip,nw_dst=10.1.1.1` | - |
| `actions=<action list>` | Actions executed on match | `actions=drop` on a critical flow indicates the packet is intentionally dropped; verify whether this is expected |

**Action syntax breakdown for a complex entry (ARP responder):**

| Action | Meaning |
|---|---|
| `move:NXM_OF_ETH_SRC[]->NXM_OF_ETH_DST[]` | Copy eth_src (current packet) into eth_dst, preparing the ARP reply |
| `mod_dl_src:aa:bb:cc:dd:ee:ff` | Set eth_src to the gateway MAC |
| `load:0x2->NXM_OF_ARP_OP[]` | Set ARP opcode to 2 (reply) |
| `move:NXM_NX_ARP_SHA[]->NXM_NX_ARP_THA[]` | Target hardware = sender (echo back) |
| `move:NXM_OF_ARP_SPA[]->NXM_OF_ARP_TPA[]` | Target protocol = sender IP |
| `load:0xaabbccddeeff->NXM_NX_ARP_SHA[]` | Sender hardware = gateway MAC |
| `load:0x0a010101->NXM_OF_ARP_SPA[]` | Sender protocol = 10.1.1.1 (gateway IP) |
| `IN_PORT` | Output back through the ingress port (ARP reply) |

This is the classic OVN ARP responder flow, injecting an ARP reply for the gateway IP without forwarding to the CPU or controller.

**Failure scenarios:**

- **`n_packets=0` for an expected ARP responder**: the VM has not sent an ARP request yet; generate test traffic with `ping <gateway>` and re-dump.
- **`n_packets` rising quickly but `n_bytes/n_packets` much smaller than the expected packet size**: packets are being fragmented; check the underlay MTU.
- **Duration continually resetting to 0**: flow thrashing; `ovn-controller` is recomputing repeatedly because NBDB changes are unstable. Check `ovn-appctl -t ovn-controller inc-engine/show-stats`.

**Upstream:** man `ovs-ofctl(8)` §`dump-flows` plus OVS Advanced Tutorial §Table 8 testing.

### 9.4.2.5 Scenario: flow-table hygiene with `replace-flows` and `diff-flows`

**Context:** an important production rule. Before overwriting the flow table, always preview the diff. `replace-flows` automatically deletes missing flows, adds new flows, and updates changed flows, all atomically. But if the input file is wrong, production drops packets within one second.

```shell
# Step 1: Dump the current state for backup
$ sudo ovs-ofctl -O OpenFlow15 dump-flows br-int > /tmp/br-int.current.flows

# Step 2: Preview the diff between current and the new file
$ sudo ovs-ofctl -O OpenFlow15 diff-flows br-int /tmp/br-int.new.flows
-table=0,priority=100,in_port=1 actions=output:2
+table=0,priority=100,in_port=1 actions=output:3
-table=10,priority=50,ip,nw_dst=10.1.1.0/24 actions=drop

# Step 3: Evaluate the diff. Switching the output port from 2 to 3 is the
#   intended migration; removing the ACL drop is also intended. Approve.

# Step 4: Apply atomically
$ sudo ovs-ofctl -O OpenFlow15 --bundle replace-flows br-int /tmp/br-int.new.flows

# Step 5: Verify
$ sudo ovs-ofctl -O OpenFlow15 diff-flows br-int /tmp/br-int.new.flows
(empty output, matched)
```

**Production rules:**
- **ALWAYS run `diff-flows` before `replace-flows`.** OpenFlow has no `--dry-run`; `diff-flows` is the dry-run.
- **Add `--bundle`** (OpenFlow 1.4 or later): every change commits in one atomic bundle. If the bundle fails, the rollback is full and there is no partial state.
- **Back up `dump-flows` to a file beforehand** when working in production. Script automation: run a backup after every successful `replace-flows`.

### 9.4.2.6 Scenario: monitor live flow events

**Context:** real-time debug to see which flows are installed, removed, or modified. Similar to `tail -f` but for OpenFlow.

```shell
$ sudo ovs-ofctl -O OpenFlow15 monitor br-int watch:
send: OFPT_FLOW_MOD (OF1.5) (xid=0x4): ADD table:0 priority=100,in_port=1 actions=output:2
NXT_FLOW_MOD_TABLE_ID (OF1.0): table_id enabled
send: OFPT_SET_ASYNC (OF1.5) (xid=0x5): ...
OFPT_FLOW_REMOVED (OF1.5) (xid=0x0): cookie=0xabc, table=10, priority=50, reason=idle, duration=60.1s
  ip,nw_dst=10.1.1.5 actions=drop
  packet_count:0 byte_count:0
```

**Anatomy of the event stream:**

| Event type | Meaning | When it appears |
|---|---|---|
| `OFPT_FLOW_MOD ADD/MOD/DEL` | Controller installs, modifies, or deletes a flow | Controller is active and flows are changing |
| `OFPT_FLOW_REMOVED` | Flow expired due to timeout or delete | idle_timeout reached, or `del-flows` from the controller |
| `OFPT_PACKET_IN` | Packet sent to the controller (table-miss or CONTROLLER action) | Flow has no match; the controller is learning |
| `OFPT_PORT_STATUS` | Port add, remove, or state change | Administrative config change or link flap |

**Fields inside FLOW_REMOVED:**
- `reason=idle`: idle_timeout elapsed, no packet for N seconds.
- `reason=hard`: hard_timeout elapsed.
- `reason=delete`: explicit `del-flows`.
- `reason=evict`: flow_table eviction policy (when flow_limit is exceeded).
- `packet_count` and `byte_count`: final counters before deletion. A value of 0 indicates the flow was never hit, which may signal a wrong match spec.

**Use case:** open two terminals. Terminal 1 runs `monitor`, terminal 2 runs `ping` to a target host. Watch whether `packet_in` appears and whether `OFPT_FLOW_MOD` installs new flows.

**Upstream:** man `ovs-ofctl(8)` §`monitor` plus OpenFlow 1.5 spec §7.4 Asynchronous Messages.

## 9.4.3 `ovs-appctl`, introspection RPC over a Unix socket

### 9.4.3.1 Grammar, target discovery

`ovs-appctl` is a generic wrapper that sends an RPC command to any OVS daemon over a Unix socket. It is the one tool for live introspection. It does not change configuration (it does not write to OVSDB); it only queries the runtime state of the daemon.

Grammar:

```
ovs-appctl [OPTIONS] [-t TARGET | --target=TARGET] COMMAND [ARG...]

TARGET:
  - process name: ovs-vswitchd (default), ovsdb-server, ovn-controller, ovn-northd
  - pidfile path: /var/run/openvswitch/<name>.pid
  - unix socket: unix:/var/run/openvswitch/<name>.<pid>.ctl

OPTIONS:
  --timeout=secs
  -v, --verbose
  -h, --help

Built-in COMMANDS for every daemon:
  list-commands           -> list every command the target supports
  version                 -> print version and compile date
  vlog/list               -> list logging modules and current level
  vlog/list-pattern       -> pattern per destination
  vlog/set SPEC           -> set log level
  vlog/reopen             -> close and reopen log file (after logrotate)
  exit                    -> graceful daemon shutdown
  memory/show             -> memory usage breakdown

Each daemon adds its own commands. Run `list-commands` on a given daemon to see them.
```

**The first step is always:**

```shell
$ sudo ovs-appctl -t ovs-vswitchd list-commands
```

This lists every command that `ovs-vswitchd` supports (typically 80 to 120 commands depending on the OVS version). Major groups:

| Group | Prefix | Purpose |
|---|---|---|
| Bridge introspection | `bridge/*`, `fdb/*`, `mdb/*` | Bridge state, MAC learning, multicast |
| OpenFlow simulation | `ofproto/*` | Trace pipeline, inspect classifier |
| Datapath introspection | `dpctl/*`, `dpif/*`, `dpif-netdev/*` | Kernel and userspace datapath state |
| Upcall and revalidator | `upcall/*`, `revalidator/*` | Kernel to userspace flow install |
| Bond and LACP | `bond/*`, `lacp/*` | Bond member state, LACP handshake |
| L2 protocols | `stp/*`, `rstp/*`, `cfm/*`, `bfd/*` | Spanning tree, CFM, BFD |
| Tunnel | `tnl/*` | ARP neighbor for tunnel, stats |
| Observability | `coverage/*`, `memory/*` | Internal counters and memory |
| Logging | `vlog/*` | Log level per module |

### 9.4.3.2 `ofproto/trace`, the most powerful tool in the kit

`ofproto/trace` simulates a packet going through the OpenFlow pipeline of a bridge. It does NOT send a real packet. The output shows every table hit, every rule match, every action executed, the final action set, and the datapath actions emitted at the output. This is the primary debugging tool every OVS engineer must master.

**Command syntax (simplified):**

```
ovs-appctl ofproto/trace [OPTIONS] BRIDGE FLOW_SPEC [OPTIONS]

OPTIONS:
  -generate         -> actually execute side effects (learn, ct commit)
  -consistent       -> force consistent table cache
  --names | --no-names
  --ct-next=<state> -> simulate conntrack returning a specific state
```

**Flow spec:** a series of `FIELD=VALUE` pairs, similar to the matching syntax of `ovs-ofctl dump-flows`.

### 9.4.3.3 Reading `ovs-appctl ofproto/trace` output

**Context:** you receive a ticket "traffic from VM A (tap-vm-web, 10.1.0.5) to VM B (tap-vm-db, 10.1.0.10) does not flow". Trace instead of having to capture a real packet by running `ping` plus `tcpdump`.

**Command and output** `[reproduced from OVS Advanced Tutorial, adapted OVN setup]`:

```shell
$ sudo ovs-appctl ofproto/trace br-int \
    "in_port=tap-vm-web,dl_src=aa:bb:cc:00:00:01,dl_dst=aa:bb:cc:00:00:02,\
     dl_type=0x0800,nw_src=10.1.0.5,nw_dst=10.1.0.10,nw_proto=6,\

    load:0x2->NXM_NX_REG15[]

    resubmit(,33)

          nw_frag=no,tp_dst=0x50/0xffff

```

**Anatomy of the four output blocks:**

**Block 1, `Flow:`.** This is the initial state of the simulated packet. `ovs-appctl` fills any missing fields with their defaults (`vlan_tci=0`, `nw_tos=0`, `nw_ttl=0`, `tcp_flags=0`). This is the *input* to the pipeline.

**Block 2, `bridge("br-int") ... N. <match>, priority P`.** Each table hit is shown in order: table 0 then table 8 then table 16 then table 32 then table 33. Each line follows the format:
- `<table_id>. <match_spec>, priority <P>, cookie <cookie>`
- Sub-lines are the actions executed in that table.

| Token | Meaning |
|---|---|
| `0. in_port=1, priority 100` | Table 0, match `in_port=1`, priority 100 |
| `load:0x1->NXM_NX_REG13[]` | Action: load 0x1 into register 13 (mark logical datapath = 1) |
| `resubmit(,8)` | Action: submit the packet to table 8 with the same register context |
| `priority 0` | Table-miss rule (priority 0) match, default fallthrough |
| `reg13=0x1,dl_dst=...,priority 50` | Table 16, match `reg13` plus `eth_dst`, priority 50 |
| `load:0x2->NXM_NX_REG15[]` | Set logical output port = 2 (REG15 is the logical output port in OVN) |
| `output:2` | Final, physical output port 2 |

**Block 3, `Final flow:`.** State of the packet AFTER the pipeline finishes, before it exits. Compare with the initial `Flow:` to see modifications: the packet now has `reg13=0x1,reg15=0x2` due to the loads in tables 0 and 16.

**Block 4, `Megaflow:`.** The megaflow cache signature for this packet. This is the *classifier mask* that the kernel datapath will install after the upcall. A field present in the megaflow spec means that field **must match exactly** for the kernel to reuse the cache entry. In the example above, `tp_dst=0x50/0xffff` means the kernel matches exact destination port 80; both `nw_src` and `nw_dst` are exact matches as well, so this megaflow does not generalize to other IPs.

**Block 5, `Datapath actions:`.** The final actions executed at the datapath (kernel or userspace). This is ground truth: if `Datapath actions: drop`, the packet is dropped in the kernel; `Datapath actions: 2,3` means output to ports 2 and 3.

**Failure scenarios:**

- **`Datapath actions: drop`.** Walk back up the pipeline to find which rule matched `drop`. In the trace, check the last table hit; an explicit drop action will be visible.
- **Megaflow too specific (too many exact-match fields).** The kernel installs a flow for only one specific flow; every slightly different packet misses the cache and triggers another upcall. Check `upcall/show` to see whether `flows_install_rate` is abnormally high.
- **Cookie `0x0` on every flow.** This is not OVN (OVN generates non-zero cookies). You may be debugging a `br-int` not managed by OVN, or a plain OpenFlow bridge.
- **Trace does not progress past table 8.** The flow at table 0 is missing `resubmit(,8)` or `goto_table:8`. The OVN pipeline skips a table; check `ovn-controller` translation, possibly the chassis has not registered correctly.

**Upstream:** man `ovs-vswitchd(8)` §`ofproto/trace`, OVS Advanced Tutorial §Table 0 Testing, `ofproto/ofproto-dpif-xlate.c` function `ofproto_trace()`.

### 9.4.3.4 Reading `ovs-appctl fdb/show` output

**Context:** debug L2 forwarding. A wrong MAC learning table leads to broadcast storms or traffic going to the wrong port. The FDB is a map of MAC to port plus VLAN, learned automatically in a `standalone` bridge or via an explicit `learn` action.

```shell
$ sudo ovs-appctl fdb/show br-int
 port  VLAN  MAC                Age
    1   100  aa:bb:cc:00:00:01    4
    2   100  aa:bb:cc:00:00:02   12
    3     0  aa:bb:cc:00:00:03   87
    1   100  aa:bb:cc:00:00:04  287
```

**Anatomy of each field:**

| Field | Meaning | Abnormal signals |
|---|---|---|
| `port` | OpenFlow port number. Cross-check with `ovs-ofctl show br-int` | Port `LOCAL` in the FDB is rarely expected on a bridge |
| `VLAN` | VLAN ID. `0` is untagged | A VLAN other than expected means a wrong access tag on the port |
| `MAC` | MAC address learned from an incoming packet | Duplicate MAC on two ports means MAC flapping; check for an L2 loop |
| `Age` | Seconds since the entry was last learned. Default aging is 300s | An entry older than 300s still present means you should check `ovs-vsctl get Bridge br-int other_config:mac-aging-time` |

**Failure scenarios:**

- **A MAC appearing and disappearing across multiple ports (MAC flapping).** An L2 loop is present; check STP state (`ovs-appctl stp/show` or `rstp/show`).
- **FDB empty on a bridge that has traffic.** The bridge has `fail_mode=secure` and lacks a learn flow; flush and reapply the flow.
- **Age always 0.** The FDB is being updated continuously; a host is spamming broadcasts under that MAC.

**Useful companion command:** `fdb/flush br-int` forces a full relearn of all MACs (use after a topology change).

### 9.4.3.5 Reading `ovs-appctl upcall/show` output

**Context:** an upcall is the mechanism by which the kernel datapath asks userspace when it sees a packet that does not match the megaflow cache. A high rate means a performance issue. `upcall/show` reports the health of the fast-path to slow-path handoff.

```shell
$ sudo ovs-appctl upcall/show
system@ovs-system:
  flows         : (current 1247) (avg 1250) (max 1350) (limit 200000)
  dump duration : 8ms
  ufid enabled : true

  1: (keys 624) (max-flows 64) (avg-updates 1.2) (busy cycles 0.03%)
  2: (keys 623) (max-flows 63) (avg-updates 1.3) (busy cycles 0.03%)
```

**Anatomy of each field:**

| Field | Meaning | Abnormal signals |
|---|---|---|
| `system@ovs-system:` | Datapath name | - |
| `flows: (current N) (avg N) (max N) (limit N)` | Flow count in the datapath (current, moving average, max reached, hard limit) | `current` near `limit` means the flow table is nearly full and upcalls will throttle; `max` much greater than `avg` indicates a traffic burst |
| `dump duration: Nms` | Time for the revalidator to dump the full flow table. Default target is below 100 ms | `dump duration > 1s` means the flow table is too large or there is memory contention; this is the symptom from Part 9.26 revalidator storm |
| `ufid enabled: true` | OVS 2.7 and later uses Unique Flow ID to track flows. `false` means rollback mode | `false` on OVS 2.7 or later is a configuration issue |
| `1: (keys N) (max-flows N)` | Revalidator thread 1 stats | - |
| `keys N` | Number of ukeys (unique keys) this thread manages | Thread imbalance (one thread keys much greater than another) means rxq assignment skew |
| `max-flows N` | Max flows the thread manages | - |
| `avg-updates N` | Average updates per flow (per revalidator pass) | Greater than 5 means flow thrashing and continuous reinstall |
| `busy cycles N%` | Percentage of CPU the thread uses | Greater than 50 percent means the thread is saturated |

**Failure scenarios:**

- **`flows current` near `limit`.** Upcalls will drop new flows; raise the limit with `ovs-vsctl set Open_vSwitch . other_config:flow-limit=<N>` or reduce flow explosion at the OpenFlow layer.
- **`dump duration > 1s`.** Part 9.26 revalidator storm pattern; check `coverage/show | grep revalidate` and the fix path in commit `180ab2fd635e`.
- **`busy cycles` on a single thread above 90 percent.** Revalidator bottleneck; OVS 2.18 and later has a thread group that helps, otherwise the CPU is saturated.

**Upstream:** OVS source `ofproto/ofproto-dpif-upcall.c` function `upcall_unixctl_show()`, NSDI 2015 paper §5.

### 9.4.3.6 Reading `ovs-appctl coverage/show` output

**Context:** `coverage/show` prints internal counters: how many times each event type happens inside `ovs-vswitchd`. This is hidden telemetry, very powerful for debugging, but you need to know which counters to care about.

```shell
$ sudo ovs-appctl coverage/show | head -20
Event coverage, avg rate over last: 5 seconds, last minute, last hour, total since startup
bridge_reconfigure           0.0/sec    0.000/sec        0.000/sec   total: 3
ofproto_flush                0.0/sec    0.000/sec        0.000/sec   total: 0
ofproto_update_port          0.0/sec    0.000/sec        0.000/sec   total: 18
upcall_ukey_lookup_created   1.2/sec    0.421/sec        0.208/sec   total: 12453
upcall_ukey_lookup_hit       4.8/sec    4.103/sec        4.209/sec   total: 287423
upcall_flow_add              0.0/sec    0.025/sec        0.015/sec   total: 189
upcall_flow_del              0.0/sec    0.020/sec        0.012/sec   total: 145
revalidate                   0.0/sec    0.017/sec        0.015/sec   total: 183
hit                          4.8/sec    4.103/sec        4.209/sec   total: 287423
miss                         0.2/sec    0.125/sec        0.134/sec   total: 8127
```

**Important counter classes:**

| Counter | Meaning | Healthy range |
|---|---|---|
| `hit` / `miss` | Megaflow cache hit and miss in the datapath | `miss/hit < 1%` is healthy; `miss/hit > 10%` indicates flow explosion or a classifier issue |
| `upcall_ukey_lookup_created` | Rate of new ukey creation (equivalent to new flows) | Below 10 per second is stable traffic; above 100 per second is flow thrashing |
| `upcall_ukey_lookup_hit` | Rate of hits on an existing ukey | - |
| `upcall_flow_add` | Rate of flow installs into the kernel | - |
| `upcall_flow_del` | Rate of flow deletes from the kernel | `add` and `del` roughly equal indicates steady state |
| `revalidate` | Revalidator pass rate | OVS 2.17 targets 1 pass per second; above 10 per second indicates instability |
| `bridge_reconfigure` | Reconfiguration count (OVSDB change) | Above 10 per minute indicates configuration churn |

**Failure scenarios:**

- **`miss` rate above 10 percent of `hit`.** Check the flow install rate; this could be a Part 9.19 flow-table granularity issue.
- **`upcall_flow_add` rate very high (above 100 per second) plus a large megaflow.** TSS classifier subtable explosion; see Part 9.15.
- **`revalidate` rate above 10 per second.** Revalidator storm pattern from Part 9.26; fix path in commit `180ab2fd635e`.

**Companion:** `coverage/read-counter <name>` reads the specific value of a counter (without computing the rate).

**Upstream:** OVS source `lib/coverage.c` function `coverage_unixctl_show()`.

## 9.4.4 `ovs-dpctl`, raw datapath inspection

### 9.4.4.1 Grammar and when to use it

`ovs-dpctl` talks directly to the datapath (kernel or userspace). It bypasses the ofproto layer: no OpenFlow, no classifier, no `ofproto`, no connection manager. It is the ground truth about what the datapath ACTUALLY forwards.

When to use `ovs-dpctl` instead of `ovs-ofctl`:

| Situation | Right tool |
|---|---|
| View the OpenFlow flows already programmed | `ovs-ofctl dump-flows` |
| View the megaflows the kernel has installed | `ovs-dpctl dump-flows` |
| Verify whether the datapath is consistent with OpenFlow | Both, comparing cookie and counters |
| Debug datapath conntrack | `ovs-dpctl dump-conntrack` or `ovs-appctl dpctl/dump-conntrack` |
| What feature flags the datapath supports | `ovs-dpctl get-dp-features` |

Grammar:

```
ovs-dpctl [OPTIONS] COMMAND [ARG...]

key COMMANDS:
  show [DP]                              -> datapath summary
  dump-dps                               -> list datapath names
  dump-flows [-m] [DP]                   -> flow table (-m = more detail)
  dump-conntrack [-s] [-m] [DP]          -> conntrack entries
  flush-conntrack [DP] [zone=ZONE]       -> flush conntrack
  ct-stats-show [-m] [DP]                -> conntrack stats by protocol
  get-flow [-m] [DP] ufid:UFID           -> specific flow by UFID

key OPTIONS:
  -m, --more          -> more detail (unmasked match fields, ufid)
  -s, --statistics    -> include stats
  --names | --no-names
```

### 9.4.4.2 Reading `ovs-dpctl show` output

```shell
$ sudo ovs-dpctl show
system@ovs-system:
  lookups: hit:287423 missed:8127 lost:0
  flows: 1247
  masks: hit:4127821 total:1247 hit/pkt:3.47
  port 0: ovs-system (internal)
  port 1: br-int (internal)
  port 2: tap-vm-web (internal)
  port 3: tap-vm-db (internal)
  port 4: ovn-huyvl-0 (geneve)
  port 5: ovn-huyvl-1 (geneve)
```

**Anatomy of each field:**

| Field | Meaning | Abnormal signals |
|---|---|---|
| `system@ovs-system:` | Datapath name | - |
| `lookups: hit:N missed:N lost:N` | Lookup counters: `hit` (megaflow match), `missed` (upcall trigger), `lost` (packet dropped due to full queue) | `lost > 0` is kernel queue overflow, an emergency for performance; `missed/hit > 5%` means flow explosion |
| `flows: N` | Megaflow count currently in the datapath | Should match `upcall/show flows current` |
| `masks: hit:N total:N hit/pkt:N` | Classifier mask lookups: `hit` is total mask hits, `total` is total masks present, `hit/pkt` is the average mask lookups per packet | `hit/pkt > 10` indicates a near-linear classifier; see Part 9.15 tuple space search stress |
| `port N: <name> (<type>)` | Port number plus name plus type (`internal`, `system`, `geneve`, `vxlan`, `netdev`) | A mismatch between `dpctl` port N and `ofctl` port N is rare (kernel dp and userspace dp desync) |

**Failure scenarios:**

- **`lost > 0`.** Packets are dropped because the kernel Netlink queue is full; check `dmesg | grep -i 'openvswitch'` for the drop reason. Remedy: raise `net.core.netdev_max_backlog`.
- **`missed/hit > 10%` consistently.** Flow explosion; check whether `dump-flows` shows a large flow table. With OVN, check `ovn-controller` I-P stats.
- **`hit/pkt > 15`.** Classifier subtable explosion; see Part 9.15 tuple space search optimization.

### 9.4.4.3 Reading `ovs-dpctl dump-flows -m` output

**Context:** see the flows the kernel has actually installed, with the full megaflow mask. The `-m` flag prints unmasked fields plus the UFID.

```shell
$ sudo ovs-dpctl dump-flows -m | head -3
ufid:a1b2c3d4-5e6f-7a8b-9c0d-e1f2a3b4c5d6,
  recirc_id(0),dp_hash(0/0),skb_priority(0/0),in_port(3),
  skb_mark(0/0),ct_state(0/0),ct_zone(0/0),ct_mark(0/0),ct_label(0/0),
  eth(src=aa:bb:cc:00:00:01,dst=aa:bb:cc:00:00:02),
  eth_type(0x0800),
  ipv4(src=10.1.0.5,dst=10.1.0.10,proto=6,tos=0/0,ttl=64,frag=no),
  tcp(src=50000/0,dst=80),
  tcp_flags(0/0),
  packets:18234, bytes:2145678, used:0.082s,
  flags:SP, actions:4
```

**Anatomy of each token:**

| Token | Meaning |
|---|---|
| `ufid:<uuid>` | Unique Flow ID, 128-bit, OVS 2.7 and later |
| `recirc_id(0)` | Recirculation ID = 0 means the packet has not recirculated; non-zero means it has already passed through `ct()` or conjunction |
| `in_port(3)` | Ingress datapath port number (matches `dpctl show`) |
| `ct_state(0/0)` | Conntrack state plus mask (`0/0` is no match) |
| `eth(src=..., dst=...)` | Ethernet match |
| `eth_type(0x0800)` | IP |
| `ipv4(src=..., dst=..., proto=6, ...)` | Layer 3 match; `proto=6` is TCP |
| `tcp(src=50000/0, dst=80)` | TCP source matches anything (mask 0), destination is exact 80 |
| `packets:N` | Packets matched by this flow |
| `bytes:N` | Bytes matched |
| `used:Ns` | Seconds since the last match; used by the revalidator for GC |
| `flags:SP` | S=SYN, P=PUSH; TCP flags observed |
| `actions:4` | Output port 4 (Geneve tunnel to remote OVS chassis) |

**Failure scenarios:**

- **`used > 10s` for a flow expected to be active.** Traffic has stopped; the flow is about to be garbage collected.
- **`packets:0, bytes:0` for a flow that has existed for more than a few seconds.** The flow has been installed but has not matched yet; this can be an upcall-then-install race.
- **`actions:drop`.** The kernel datapath has "compiled" into a drop; verify whether `ofproto/trace` also produces `Datapath actions: drop`.
- **Mask `tcp(src=50000/0, dst=80)`.** The source mask is 0 (wildcard) because the policy only cares about destination port 80; if instead you see `src=50000/0xffff`, the rule is wrong, an exact match on source port 50000 does not generalize.

**Upstream:** man `ovs-dpctl(8)` §`dump-flows` plus OVS source `datapath/flow.c`.

### 9.4.4.4 Conntrack inspection

```shell
$ sudo ovs-dpctl dump-conntrack | head -5
tcp,orig=(src=10.1.0.5,dst=10.1.0.10,sport=50000,dport=80),
    reply=(src=10.1.0.10,dst=10.1.0.5,sport=80,dport=50000),
    zone=1,protoinfo=(state=ESTABLISHED)
tcp,orig=(src=10.1.0.5,dst=10.1.0.10,sport=50001,dport=443),
    reply=(src=10.1.0.10,dst=10.1.0.5,sport=443,dport=50001),
    zone=1,protoinfo=(state=ESTABLISHED)
udp,orig=(src=10.1.0.5,dst=10.10.10.10,sport=5001,dport=53),
    reply=(src=10.10.10.10,dst=10.1.0.5,sport=53,dport=5001),
    zone=1
```

**Anatomy:**

| Token | Meaning |
|---|---|
| `tcp / udp / icmp` | Protocol |
| `orig=(...)` | Original direction (the SYN, the first packet) |
| `reply=(...)` | Reply direction |
| `zone=1` | OVS conntrack zone, namespace isolation |
| `protoinfo=(state=ESTABLISHED)` | TCP state (NEW, SYN_SENT, SYN_RECV, ESTABLISHED, FIN_WAIT, CLOSE_WAIT, LAST_ACK, TIME_WAIT, CLOSE) |

**`ct-stats-show`:**

```shell
$ sudo ovs-dpctl ct-stats-show
Connections Stats:
    Total: 3412
    TCP  : 2847 (ESTABLISHED: 2823, TIME_WAIT: 18, CLOSE_WAIT: 6)
    UDP  : 512
    ICMP : 53
```

**Failure scenarios:**

- **Total close to `nf_conntrack_max`** (default 262144). Conntrack table is full; new connections are dropped.
- **Very high TIME_WAIT count.** High SYN rate, or applications that do not reuse connections; tune `net.ipv4.tcp_tw_reuse`.
- **Zone different from expected.** An OVS `ct(zone=N)` action is setting the wrong zone; check the OpenFlow flow with the `ct` action.

**Upstream:** man `ovs-dpctl(8)` §`dump-conntrack`, man `nf_conntrack(8)`.

## 9.4.5 `ovsdb-client`, raw OVSDB access

### 9.4.5.1 Grammar and use cases

`ovsdb-client` bypasses `ovs-vsctl` and talks directly to `ovsdb-server` over JSON-RPC. Use it when you need to:

- Debug an OVSDB issue (corruption, schema mismatch, lock contention).
- Watch a live change stream (event-driven automation).
- Backup or restore a database.
- Access a database other than the Open_vSwitch schema (for example OVN NB or SB).

Grammar:

```
ovsdb-client [OPTIONS] COMMAND [SERVER] [ARG...]

SERVER:
  unix:/var/run/openvswitch/db.sock           (OVS default)
  unix:/var/run/ovn/ovnnb_db.sock             (OVN NB)
  unix:/var/run/ovn/ovnsb_db.sock             (OVN SB)
  tcp:HOST:PORT
  ssl:HOST:PORT

Key COMMANDS:
  list-dbs [SERVER]                          -> list available schemas
  get-schema [SERVER] [DATABASE]             -> dump schema JSON
  list-tables [SERVER] [DATABASE]            -> list tables in the database
  list-columns [SERVER] [DATABASE] [TABLE]   -> list columns in a table
  dump [SERVER] [DATABASE] [TABLE...]        -> dump rows
  monitor [SERVER] [DATABASE] TABLE [COL...] -> stream change events
  backup [SERVER] [DATABASE] > FILE          -> backup to a file (JSON)
  restore [SERVER] [DATABASE] < FILE         -> restore from a file
  convert [SERVER] [DATABASE] SCHEMA_FILE    -> schema upgrade
  transact [SERVER] JSON                     -> raw JSON-RPC transact
```

### 9.4.5.3 Scenario, backup and restore

**Context:** before upgrading OVS, you must back up OVSDB. `ovsdb-client backup` produces a consistent dump and does not require the daemon to stop.

```shell

$ sudo ovsdb-client backup unix:/var/run/openvswitch/db.sock > /tmp/conf.db.backup
$ ls -lh /tmp/conf.db.backup
-rw-r--r-- 1 root root 45K Apr 24 11:23 /tmp/conf.db.backup

# If a restore is needed (rollback upgrade scenario)
$ sudo systemctl stop ovs-vswitchd
$ sudo systemctl stop ovsdb-server
$ sudo mv /etc/openvswitch/conf.db /etc/openvswitch/conf.db.old
$ sudo ovsdb-tool create /etc/openvswitch/conf.db \
    /usr/share/openvswitch/vswitch.ovsschema
$ sudo ovsdb-client restore unix:/var/run/openvswitch/db.sock \
    Open_vSwitch < /tmp/conf.db.backup

# Often easier: copy conf.db to another host, or restore the raw file
```

**Production rules:**
- Back up before every schema change (an OVS upgrade from 2.17 to 3.x carries a schema update).
- Backup is non-blocking; run a daily cron into `/var/backups/ovs/conf.db.YYYYMMDD`.
- Pair the backup with metadata (OVS version, host, kernel) for traceability.

**Upstream:** man `ovsdb-client(1)` and `ovsdb-tool(1)`. See also Part 10.2 for the OVSDB backup, restore, and compact deep dive.

## 9.4.6 Integrated six-layer troubleshooting playbook

When a report says "packets do not flow through OVS correctly", walk the six layers in order from the bottom up. Each layer has a specific check command, an expected state, and a decision branch.

| Layer | Question | Check command | Expected | If it fails |
|---|---|---|---|---|
| **L1 Physical** | Does the packet reach the NIC? | `tcpdump -i eth0 -nn -c 10` | Inbound packets visible | Cable, SFP, NIC driver; `ethtool eth0` verifies the link |
| **L2 Kernel driver** | Does the packet enter the datapath? | `ovs-dpctl show`, the `lookups:missed` counter | `missed` increases when there is traffic | NIC bound to the wrong driver; `driverctl list-overrides` for DPDK |
| **L3 Megaflow cache** | Is any megaflow hit? | `ovs-dpctl dump-flows -m`, find a flow matching the packet | A flow with `packets:N > 0` | Continuous cache miss means a problem at L4 |
| **L4 OpenFlow pipeline** | Which flow matches? With what action? | `ovs-appctl ofproto/trace br-int "FLOW_SPEC"` | `Datapath actions: <ofport>`, not drop | Wrong rule priority, missing table, unintended drop action |
| **L5 OVSDB config** | Are the port, VLAN, and bridge correct? | `ovs-vsctl show` plus `ovs-vsctl list Interface <name>` | Port attached to the right bridge, interface up | Port missing a VLAN tag, wrong interface type |
| **L6 Logical (OVN)** | Are the LSP and Port_Binding correct? | `ovn-sbctl show` plus `ovn-sbctl list Port_Binding` plus `ovn-trace <dp> "FLOW"` | `Port_Binding` has a non-null `chassis` | `ovn-controller` has not bound, or the logical flow compiled incorrectly |

**Rule:** when L1 to L6 are OK and packets still drop, the problem is outside OVS (upstream switch, firewall, ARP, MTU, BGP). Do not continue debugging OVS.

### 9.4.6.1 Guided exercise, full 6-layer walkthrough

**Scenario:** a user reports "I cannot ping from VM1 (`tap-vm-web`, 10.1.0.5) to VM2 (`tap-vm-db`, 10.1.0.10). Both are on the same `Logical_Switch ls-prod`, expected to go through `br-int`."

**Setup (simulated):**

```shell
$ sudo ovs-vsctl add-br br-int -- set Bridge br-int fail_mode=secure
$ sudo ovs-vsctl add-port br-int tap-vm-web -- \
    set Interface tap-vm-web type=internal
$ sudo ovs-vsctl add-port br-int tap-vm-db -- \
    set Interface tap-vm-db type=internal
$ sudo ip link set tap-vm-web up
$ sudo ip addr add 10.1.0.5/24 dev tap-vm-web
$ sudo ip link set tap-vm-db up
$ sudo ip addr add 10.1.0.10/24 dev tap-vm-db
# Add flows: allow ARP and IP between the two ports, intentional bug: drop ICMP
$ sudo ovs-ofctl -O OpenFlow15 add-flow br-int \
    "table=0,priority=200,in_port=tap-vm-web,dl_type=0x0806,actions=output:tap-vm-db"
$ sudo ovs-ofctl -O OpenFlow15 add-flow br-int \
    "table=0,priority=200,in_port=tap-vm-db,dl_type=0x0806,actions=output:tap-vm-web"
$ sudo ovs-ofctl -O OpenFlow15 add-flow br-int \
    "table=0,priority=100,in_port=tap-vm-web,dl_type=0x0800,nw_proto=6,\
     actions=output:tap-vm-db"
$ sudo ovs-ofctl -O OpenFlow15 add-flow br-int \
    "table=0,priority=100,in_port=tap-vm-db,dl_type=0x0800,nw_proto=6,\
     actions=output:tap-vm-web"
# Bug: a flow at priority 150 drops ICMP (intentionally injected)
$ sudo ovs-ofctl -O OpenFlow15 add-flow br-int \
    "table=0,priority=150,in_port=tap-vm-web,dl_type=0x0800,nw_proto=1,actions=drop"
```

**Walkthrough playbook:**

**L1, tcpdump physical (skip in this internal-only setup; for an OVN Geneve setup, tcpdump on eth0):**

```shell
$ sudo tcpdump -i tap-vm-web -nn icmp -c 5
# We see outbound ICMP, so L1 is OK
```

**L2, `ovs-dpctl show`:**

```shell
$ sudo ovs-dpctl show
system@ovs-system:
  lookups: hit:23, missed:7, lost:0
  flows: 5
  masks: hit:38 total:4 hit/pkt:1.65
  port 0: ovs-system (internal)
  port 1: br-int (internal)
  port 2: tap-vm-web (internal)
  port 3: tap-vm-db (internal)

# Counter missed > 0 means the packet entered OVS and the kernel triggered an upcall. OK.
```

**L3, `ovs-dpctl dump-flows`:**

```shell
$ sudo ovs-dpctl dump-flows -m
# We will see the ICMP flow installed with actions:drop, the signal of a packet drop at L3 or L4
```

**L4, `ofproto/trace` for ICMP:**

```shell
$ sudo ovs-appctl ofproto/trace br-int \
    "in_port=tap-vm-web,dl_src=aa:bb:cc:00:00:01,dl_dst=aa:bb:cc:00:00:02,\
     dl_type=0x0800,nw_src=10.1.0.5,nw_dst=10.1.0.10,nw_proto=1"
Flow: icmp,in_port=2,vlan_tci=0x0000,dl_src=aa:bb:cc:00:00:01,
      dl_dst=aa:bb:cc:00:00:02,nw_src=10.1.0.5,nw_dst=10.1.0.10,...

bridge("br-int")
----------------
 0. in_port=2,ip,nw_proto=1, priority 150
    drop

Final flow: unchanged
Datapath actions: drop
```

**Root cause** is clear: the rule `table=0 priority=150 in_port=tap-vm-web,ip,nw_proto=1 drop`.

**L5 and L6 are not needed because L4 already found the root cause.**

**Fix:**

```shell
$ sudo ovs-ofctl -O OpenFlow15 del-flows br-int \
    "table=0,in_port=tap-vm-web,dl_type=0x0800,nw_proto=1" --strict
$ sudo ovs-ofctl -O OpenFlow15 add-flow br-int \
    "table=0,priority=100,in_port=tap-vm-web,dl_type=0x0800,nw_proto=1,\
     actions=output:tap-vm-db"
$ sudo ovs-ofctl -O OpenFlow15 add-flow br-int \
    "table=0,priority=100,in_port=tap-vm-db,dl_type=0x0800,nw_proto=1,\
     actions=output:tap-vm-web"

# Re-trace
$ sudo ovs-appctl ofproto/trace br-int "in_port=tap-vm-web,...,nw_proto=1"
Datapath actions: 3
```

**Teardown:**

```shell
$ sudo ovs-vsctl del-br br-int
```

```shell
sudo ovs-dpctl dump-conntrack
sudo ovs-dpctl ct-stats-show

# --- ovsdb-client ---
sudo ovsdb-client list-dbs unix:/var/run/openvswitch/db.sock
sudo ovsdb-client monitor unix:/var/run/openvswitch/db.sock Open_vSwitch Bridge
```

## Key takeaways

First, the five OVS CLI tools have **independent roles** and do not replace one another. `ovs-vsctl` configures OVSDB only; `ovs-ofctl` covers OpenFlow only; `ovs-appctl` does runtime introspection only; `ovs-dpctl` touches the raw datapath only; `ovsdb-client` is for raw OVSDB RPC only. Each tool covers a layer of the architecture, and a real debugging workflow uses several of them in parallel.

Second, `ofproto/trace` is the most powerful debug tool. It simulates a packet through the OpenFlow pipeline without sending a real packet. An OVS engineer must read fluently the four output blocks (Flow, bridge walk-through, Final flow, Datapath actions, Megaflow).

Third, the six-layer troubleshooting playbook is a systematic method going from L1 physical up to L6 logical, eliminating one region at a time. When all six layers agree and packets still drop, the root cause lies outside OVS.

Fourth, atomicity of an OVSDB transaction (`ovs-vsctl`) and of an OpenFlow bundle (`ovs-ofctl --bundle`) protects against half-configured state. Always `diff-flows` before `replace-flows` in production; always back up `conf.db` before an upgrade.

Fifth, reading output is not just checking that a command ran without error. Every field has a physical meaning (counter, timestamp, state enum) and its own failure scenario (when field X equals Y, the symptom is Z). A skilled engineer knows which field X matters, knows the threshold Y, and knows what to do next when Y is exceeded.

### 9.4.X.1 `ovs-vsctl add-br`

#### Concept

`add-br` is the subcommand that creates a new OVS bridge. Its behavior is not simply "insert a row into the `Bridge` table": within the same OVSDB transaction, OVS creates a `Bridge` row, a `Port` row with the same name as the bridge (the local port), and an `Interface` row of type `internal` with the same name. After the transaction commits, the kernel datapath is asked to create the bridge entity, an `internal` netdev appears in the host namespace (visible through `ip link show br0`), and `ovs-vswitchd` opens the OpenFlow channel for this bridge.

#### Syntax and key parameters

```
ovs-vsctl [--may-exist] add-br BRIDGE [PARENT VLAN] [-- set Bridge BRIDGE COLUMN=VALUE ...]
```

- `BRIDGE` is the bridge name, and it is also the name of the local port and of the internal interface created together with it (Linux constraint: at most 15 characters).
- `[PARENT VLAN]` is used only to create a "fake bridge" on a specific VLAN of a parent bridge; most deployments do not use it.
- `--may-exist` turns the behavior from "fail if it already exists" into "no-op when the existing configuration matches"; required for idempotent scripts.
- Every `-- set Bridge ... COLUMN=VALUE` argument runs in the same transaction as `add-br`; a typical example: `-- set Bridge br0 fail_mode=secure protocols=OpenFlow13,OpenFlow15 datapath_type=system`.

#### Transaction semantics

Each invocation of `ovs-vsctl` is **one** OVSDB transaction sent to `ovsdb-server` through the socket `/var/run/openvswitch/db.sock`. If any operation in the transaction fails (constraint violation, name collision, invalid `datapath_type`), the whole transaction rolls back: the `Bridge` row does not appear, and `Port` / `Interface` are not created either. This is the foundation of idempotency: the caller does not need to clean up a half-applied state.

#### Real output sample

```shell
$ sudo ovs-vsctl add-br br-demo
$ sudo ovs-vsctl list Bridge br-demo
_uuid               : 9f3a1c12-77b2-4a01-9d8c-1c2d5e8f7a90
controller          : []
datapath_id         : "0000aabbccddeeff"
datapath_type       : system
fail_mode           : []
name                : br-demo
ports               : [b71d...]
protocols           : []
$ ip link show br-demo
12: br-demo: <BROADCAST,MULTICAST> mtu 1500 qdisc noop state DOWN ...
```

Notes: `controller` is empty (not attached yet), `fail_mode` is empty (the default is `standalone` for a bridge without a controller; see §9.4.X.2), `protocols` is empty (OVS decides by default, usually OF1.0 + 1.1 + 1.2 + 1.3 + 1.4 + 1.5).

#### Common errors and diagnosis

| Symptom | Probable cause | How to verify |
|---------|----------------|---------------|
| `ovs-vsctl: cannot create a bridge named X because a bridge named X already exists` | Forgot `--may-exist` in a script | Use `ovs-vsctl --may-exist add-br X` |
| `ip link show X` does not show the new interface | `ovs-vswitchd` died or lost the OVSDB connection | Check `systemctl status openvswitch-switch`, `ovs-appctl bridge/dump-flows` |
| Bridge created but the datapath is not active | `datapath_type=netdev` set while the host has no DPDK | Read `ovs-vswitchd.log`, fall back to `set Bridge X datapath_type=system` |
| `name X is too long` | Name longer than 15 characters (Linux netdev limit) | Choose a shorter name |

#### Lab exercise

On Ubuntu 22.04 with OVS 2.17.9:

```shell
sudo ovs-vsctl --may-exist add-br br-lab \
  -- set Bridge br-lab fail_mode=secure protocols=OpenFlow13 \
  -- set Bridge br-lab other_config:hwaddr=02:11:22:33:44:55
sudo ip link set br-lab up
sudo ovs-vsctl list Bridge br-lab | grep -E "fail_mode|protocols"
```

Pass criterion: the bridge exists, `fail_mode=secure`, `protocols=[OpenFlow13]`, the link is in `state UP`, and `ovs-ofctl -O OpenFlow13 dump-flows br-lab` runs without errors.

#### Source code reference

The `cmd_add_br()` handler is in `utilities/ovs-vsctl.c` (OVS branch-2.17). It builds an `insert` operation on the `Bridge` table plus `insert` operations on `Port` and `Interface` (the internal type), then calls `ovsdb_idl_txn_commit_block()`. The schema constraints in `vswitchd/vswitch.ovsschema` make `Bridge.name` UNIQUE and require `Bridge.ports` to contain at least the local port with the same name.

#### Version differences

- OVS 1.x (2010 to 2013): basic syntax; `--may-exist` not yet available.
- OVS 2.0 (2014): supports `--may-exist` and `--if-exists`, plus OVSDB transaction batching.
- OVS 2.4 (2015): lets the same transaction also set columns via `-- set`.
- OVS 2.7 and later (2017): supports `datapath_type=netdev` for userspace DPDK or AF_XDP (out of scope here).

#### Comparison with other systems

A pure Linux bridge uses `ip link add br0 type bridge` and then `ip link set br0 up`; it has no concept of `fail_mode`, `protocols`, or `controller`. Cisco IOS has `bridge-group`, but it lives in interface mode and there is no central database. OVS clearly separates the control plane (OVSDB) from the data plane (the kernel datapath); `add-br` modifies the control plane, and the kernel reacts afterwards through netlink.

### 9.4.X.2 `ovs-vsctl set-fail-mode`

#### Concept

`set-fail-mode` configures how a bridge behaves when its connection to the OpenFlow controller is lost. Two values are valid: `standalone` (the bridge handles packets as a normal L2 learning switch) and `secure` (the bridge does not add any rule on its own; it processes packets only against flows already installed, and any unmatched packet is dropped). The column lives in the `Bridge` table under the column name `fail_mode`.

#### Syntax and parameters

```
ovs-vsctl set-fail-mode BRIDGE MODE
ovs-vsctl get-fail-mode BRIDGE
ovs-vsctl del-fail-mode BRIDGE   # reset to [] (default)
```

`MODE` accepts only `standalone` or `secure`. When the column is empty (`[]`), the default depends on whether the bridge has a controller: with a controller the behavior matches `secure`; without a controller the behavior matches `standalone`. Most production deployments set `secure` explicitly to avoid relying on this heuristic.

#### Role in the architecture

An OVS bridge can run in two modes: SDN-controlled (the controller pushes flows) or autonomous (the bridge learns by itself). The `fail_mode` column decides what happens when the controller goes down. The `standalone` mode prioritizes availability: the bridge keeps switching traffic by itself. The `secure` mode prioritizes correctness: the bridge does not forward any packet outside the controller-defined rules, which prevents traffic from leaking past the policy. OVN, OpenStack Neutron, and Kubernetes networking all use `secure` so that SDN policy is not bypassed when the controller restarts or the network partitions.

#### Real output sample

```shell
$ sudo ovs-vsctl set-fail-mode br-int secure
$ sudo ovs-vsctl get-fail-mode br-int
secure
$ sudo ovs-vsctl get Bridge br-int fail_mode
secure
```

When `fail_mode = standalone` and the controller is unreachable, the bridge installs a default flow `actions=NORMAL` (L2 learning). You can observe it via `ovs-ofctl dump-flows br-int`, which shows the flow `priority=0 actions=NORMAL`.

#### Real incident scenario

A host runs an OVN-controlled `br-int` configured with `fail_mode=standalone`. When `ovn-controller` is killed by the OOM killer, the bridge falls back to L2 learning and `br-int` floods packets to every port. Tenant traffic from VM A leaks to VM B because both share the freshly learned MAC table. Mitigation: switch every host to `fail_mode=secure` and verify with `ovs-vsctl --columns=name,fail_mode list Bridge`.

#### Lab exercise

```shell
sudo ovs-vsctl --may-exist add-br br-fm
sudo ovs-vsctl set-fail-mode br-fm secure
# No controller, no flow, every packet drops
sudo ovs-ofctl dump-flows br-fm   # empty
# Compare:
sudo ovs-vsctl set-fail-mode br-fm standalone
sudo ovs-ofctl dump-flows br-fm
#  cookie=0x0, priority=0 actions=NORMAL  (OVS auto-add)
```

Pass criterion: the learner can explain the difference between drop and NORMAL forward when no controller is connected, and contrast it with the controller-connected case (see §9.4.X.3).

#### Common errors and diagnosis

| Symptom | Probable cause | How to verify |
|---------|----------------|---------------|
| Bridge forwards traffic even though no rule was added | `fail_mode=standalone` and the controller is down | `ovs-vsctl get-fail-mode <br>` |
| Bridge drops every packet after an OVS restart | `fail_mode=secure` and the controller has not reconnected yet | Check `ovs-vsctl get-controller br-int` and the `is_connected` column via `list Controller` |
| `set-fail-mode br-int invalid_value` returns an error | Only `standalone` and `secure` are accepted | Read `ovs-vsctl(8)` section `Bridge Configuration Commands` |

#### Source code reference

The handlers are `cmd_set_fail_mode()` and `cmd_get_fail_mode()` in `utilities/ovs-vsctl.c`. The handler validates the input string against the literal set {`standalone`, `secure`} inline before applying the column write. The forwarding behavior for `standalone` lives in `ofproto/ofproto-dpif.c` in `add_internal_flows()`, which installs the flow `priority=0 actions=NORMAL`.

#### Version differences

- OVS 1.x: both modes available from the start.
- OVS 2.0: improved controller-reconnect behavior; semantics unchanged.
- OVS 2.10 and later: support for "in-band control" lets the controller run through the very bridge it manages. Combined with `secure`, this requires special care. In-band control is disabled by default starting with 2.10.

#### Comparison with other systems

The OpenFlow specification defines `fail_secure` and `fail_standalone` (OF 1.0 §6.4); `ovs-vsctl set-fail-mode` maps to them one-to-one. NETCONF and OpenConfig do not have this concept (they assume the controller is always online). Cisco SDN (ACI) manages fail-mode at the fabric level and does not expose it as a per-bridge CLI knob.

### 9.4.X.3 `ovs-vsctl set-controller`

#### Concept

`set-controller` attaches a bridge to one or more OpenFlow controllers. Each target is a URI: `tcp:HOST[:PORT]`, `ssl:HOST[:PORT]`, `unix:FILE`, `pssl:PORT[:HOST]` (passive, OVS listens), or `ptcp:PORT[:HOST]`. OVS opens the connection, sends `OFPT_HELLO`, negotiates the OpenFlow version, and then receives flow_mod, barrier_request, and packet_in messages from the controller. A bridge can have more than one controller running in parallel (multi-controller mode); OVS forwards each packet_in to every controller, and accepts flow_mod from any of them.

#### Syntax and parameters

```
ovs-vsctl set-controller BRIDGE TARGET [TARGET ...]
ovs-vsctl get-controller BRIDGE
ovs-vsctl del-controller BRIDGE
```

Each target becomes a row in the `Controller` table, referenced from `Bridge.controller` (a set-of-UUID column). By default OVS uses TCP port 6653 (the IANA-assigned port for OpenFlow); historical deployments used 6633 (legacy). The `ssl:` scheme requires a private key plus certificate configured via `ovs-vsctl set-ssl <pk> <cert> <ca>`.

#### Transaction semantics

Each `set-controller` REPLACES the entire current controller list with the new list (it does not append). To keep the old controller and add a new one, you must pass the full list again. This is a common source of bugs in automation scripts.

#### Real output sample

```shell
$ sudo ovs-vsctl set-controller br-int tcp:127.0.0.1:6653
$ sudo ovs-vsctl get-controller br-int
tcp:127.0.0.1:6653
$ sudo ovs-vsctl --columns=target,is_connected,role list Controller
target              : "tcp:127.0.0.1:6653"
is_connected        : true
role                : master
```

The `is_connected` column reports whether the TCP or SSL connection is active; `role` (`master`, `slave`, or `equal`) belongs to the OF 1.2+ multi-controller model. When `is_connected=false`, OVS retries with exponential backoff (default starts at 1 second, max 8 seconds, configurable through `max_backoff` and `inactivity_probe`).

#### Real incident scenario

After a restart of `ovn-controller`, some hosts show `is_connected=false` for more than 30 seconds. Root cause: `ovn-controller` rewrote the Controller row through OVSDB, but `ovs-vswitchd` had not yet closed the previous connection, which was stuck in socket TIME_WAIT. Mitigation: `ovs-appctl ofproto/list` plus `ovs-appctl ofproto/reconnect br-int` to force an immediate reconnect rather than waiting for the backoff.

#### Lab exercise

Using Mininet plus a Ryu controller:

```shell
# Terminal 1: run the controller
ryu-manager ryu.app.simple_switch_13  # listen tcp 0.0.0.0:6653

# Terminal 2: create the bridge and attach the controller
sudo ovs-vsctl --may-exist add-br br-ryu \
  -- set Bridge br-ryu protocols=OpenFlow13 \
  -- set-controller br-ryu tcp:127.0.0.1:6653 \
  -- set-fail-mode br-ryu secure
sudo ovs-vsctl --columns=is_connected list Controller
# is_connected: true   means OK
```

Pass criterion: `is_connected=true` within 5 seconds, and `ovs-ofctl -O OpenFlow13 dump-flows br-ryu` shows the flows that Ryu has pushed.

#### Common errors and diagnosis

| Symptom | Probable cause | How to verify |
|---------|----------------|---------------|
| `is_connected=false` forever | Controller is not listening, firewall, or version mismatch | `ss -tnlp \| grep 6653`; check `ovs-vswitchd.log` for `OFPT_HELLO mismatch` |
| New controller set but the old controller is still attached | `set-controller` REPLACES, but the syntax accidentally passed two rows | Use `--columns=name,controller list Bridge` to inspect |
| SSL connection fails | Missing certificate or wrong CA chain | `ovs-vsctl get-ssl`, cross-check the certificate against the controller |
| `ovs-vswitchd` restart loses the controller | The Controller row is not persisted | OVSDB persists by default; if lost, check disk and `ovsdb-server` log |

#### Source code reference

The `cmd_set_controller()` and `cmd_del_controller()` handlers are in `utilities/ovs-vsctl.c`. The connection logic lives in `ofproto/connmgr.c` (`ofconn_run()` handles reconnect, `ofconn_set_role()` handles the OF 1.2 role). The default `inactivity_probe` is 5 seconds and the default `max_backoff` is 8 seconds (see the Controller table in `vswitch.ovsschema`).

#### Version differences

- OVS 1.x: supports only one controller per bridge, OF 1.0.
- OVS 2.0 (2014): multi-controller, OF 1.3 generally available.
- OVS 2.3 (2014): OF 1.4 and 1.5 experimental.
- OVS 2.10 and later (2018): OF 1.5 stable, bundle support negotiated through `OFPT_BUNDLE_FEATURES_REQUEST`.

#### Comparison with other systems

NETCONF agents (ConfD, sysrepo) also have a `<connection>` concept toward an SDN controller with similar semantics, but they use an XML schema instead of OVSDB. P4Runtime gRPC opens a persistent stream similar to the OF channel. Cisco OpenFlow mode on Catalyst exposes `controller ipv4 ... port 6653` but does not distinguish active versus passive the way OVS does.

### 9.4.X.4 `ovs-vsctl add-port`

#### Concept

`add-port` attaches an interface (physical, tunnel, internal, or patch) to a bridge. Within the same transaction, OVS creates a `Port` row and an `Interface` row (one port equals one or more interfaces in a bond) and updates `Bridge.ports` to add the new port UUID. After the commit, the kernel datapath receives a netlink message to open or attach the corresponding netdev; `ovs-vswitchd` assigns an ofport (an integer unique per bridge) to the interface, and from then on the controller can target the flow `actions=output:<ofport>`.

#### Syntax and key parameters

```
ovs-vsctl [--may-exist] add-port BRIDGE PORT [COLUMN[:KEY]=VALUE ...]
```

Common patterns:

- Access port on VLAN 10: `add-port br0 p1 tag=10`
- Trunk for VLAN 10, 20, and 30: `add-port br0 p1 trunks=10,20,30`
- Geneve tunnel: `add-port br0 gnv0 -- set Interface gnv0 type=geneve options:remote_ip=10.0.0.2 options:key=flow`
- Internal port with an IP: `add-port br0 vlan10 -- set Interface vlan10 type=internal`
- Patch port linking two bridges: `add-port br0 patch-to-br1 -- set Interface patch-to-br1 type=patch options:peer=patch-to-br0`

#### Transaction semantics

As with `add-br`, every change in the same `ovs-vsctl` invocation runs in one OVSDB transaction. If `set Interface ... type=geneve options:remote_ip=...` fails (for example because `remote_ip` is malformed), the whole transaction rolls back: the `Port` does not appear. Use `--may-exist` to make the script idempotent: when a port with the same name already exists on the same bridge with the SAME interface, the command is a no-op; when the port exists on a different bridge, the command fails.

#### Real output sample

```shell
$ sudo ovs-vsctl --may-exist add-port br-int veth-vm1 \
    -- set Interface veth-vm1 external_ids:iface-id=vm1-port \
    -- set Port veth-vm1 tag=100
$ sudo ovs-vsctl --columns=name,tag,trunks list Port veth-vm1
name        : veth-vm1
tag         : 100
trunks      : []
$ sudo ovs-ofctl show br-int | grep veth-vm1
 7(veth-vm1): addr:fa:16:3e:aa:bb:cc
     config:     0
     state:      0
     speed: 10000 Mbps now, 10000 Mbps max
```

In the column `7(veth-vm1)`, `7` is the ofport (assigned at commit time) and `veth-vm1` is the Linux netdev name. The controller uses `7` in the action `output:7`.

#### Role in the architecture

This is the only bridge between the Linux netdev world (physical NICs, veth, tunnel devices) and the OVS bridge world. Every packet entering or leaving OVS must traverse an interface that has been added via `add-port`. OVN uses `external_ids:iface-id` to tag an interface with a `Logical_Switch_Port` (see Part 13) so that `ovn-controller` can map the Port_Binding to an ofport. The OpenStack Neutron OVS agent uses a similar pattern, but it tags through `external_ids:attached-mac` plus `iface-id`.

#### Real incident scenario

A newly booted VM does not receive DHCP. `ovs-vsctl show` confirms that `tap-vm1` has been added to `br-int`. `ovs-ofctl show br-int` confirms that the ofport has been assigned. The real cause: `external_ids:iface-id` is missing, so `ovn-controller` cannot bind the Logical_Switch_Port and the ingress flow is never installed. Fix: `ovs-vsctl set Interface tap-vm1 external_ids:iface-id=<port-uuid>`.

#### Common errors and diagnosis

| Symptom | Probable cause | How to verify |
|---------|----------------|---------------|
| `port already exists on bridge X` | The interface already belongs to a different bridge | `ovs-vsctl iface-to-br <name>` |
| `add-port` succeeds but `ofport=-1` | The kernel netdev does not exist (the interface has not been created yet) | `ip link show <name>`; check `ovs-vswitchd.log` for `could not open network device` |
| Tunnel is up but carries no traffic | `options:remote_ip` is correct but `options:key` is mismatched | `ovs-vsctl get Interface <name> options`; compare the two endpoints |
| Forgot `--may-exist` inside a loop script | The second run of the command fails partway through | Add `--may-exist` to every `add-port` in a reconcile loop |

#### Lab exercise

```shell
sudo ovs-vsctl --may-exist add-br br-lab2
sudo ip link add veth-a type veth peer name veth-b
sudo ip link set veth-a up; sudo ip link set veth-b up
sudo ovs-vsctl --may-exist add-port br-lab2 veth-a tag=10
sudo ovs-vsctl --may-exist add-port br-lab2 veth-b tag=20
sudo ovs-ofctl show br-lab2
```

Pass criterion: the two ports show different ofport numbers, the `tag` values are 10 and 20, and `ovs-vswitchd.log` shows no warnings.

#### Source code reference

The `cmd_add_port()` handler in `utilities/ovs-vsctl.c` builds `insert` operations on `Port` and `Interface` and mutates `Bridge.ports`. The ofport assignment logic lives in `ofproto/ofproto.c::ofport_install()`; OVS picks ofport numbers automatically by default (1, 2, 3, and so on), and it can be pinned via `set Interface <name> ofport_request=<n>`.

#### Version differences

- OVS 1.x: kernel datapath only, with the internal, GRE, and VXLAN types.
- OVS 2.0 (2014): adds Geneve.
- OVS 2.4: improved bond LACP and added `lacp-rate`.
- OVS 2.7 and later: supports DPDK port types (`type=dpdk`, `type=dpdkvhostuser`), out of scope here.

#### Comparison with other systems

A Linux bridge uses `ip link set veth-a master br0` (direct kernel netlink); it has no separation between Port and Interface, no ofport, and no `external_ids`. OVS separates Port (logical) from Interface (physical or tunnel), which makes it possible to model a bond (one Port to many Interfaces), something a plain Linux bridge cannot do natively.

### 9.4.X.5 `ovs-vsctl set` / `get` on the `Bridge` table

#### Concept

`set TABLE RECORD COLUMN=VALUE` and `get TABLE RECORD COLUMN` are two database-level subcommands that work directly on any OVSDB row. For the `Bridge` table specifically, these two commands are the only way to configure columns that have no convenience wrapper: `protocols`, `datapath_type`, `mcast_snooping_enable`, `rstp_enable`, `external_ids`, `other_config`. The general syntax is:

```
ovs-vsctl set Bridge BRIDGE COLUMN=VALUE [COLUMN=VALUE ...]
ovs-vsctl get Bridge BRIDGE COLUMN [COLUMN ...]
ovs-vsctl --columns=COL1,COL2 list Bridge BRIDGE
```

#### Frequently used Bridge columns

| Column | Type | Meaning |
|--------|------|---------|
| `protocols` | set of string | The list of OpenFlow versions the bridge accepts: `OpenFlow10..15`. If empty, OVS negotiates automatically |
| `datapath_type` | string | `system` (kernel datapath, default) or `netdev` (userspace DPDK or AF_XDP) |
| `mcast_snooping_enable` | boolean | Enable IGMP and MLD snooping for the bridge |
| `rstp_enable` | boolean | Enable Rapid Spanning Tree |
| `external_ids` | map string-string | Free-form keys for integrations (OVN, Neutron) |
| `other_config` | map string-string | Tunables: `mac-aging-time`, `mac-table-size`, `hwaddr`, `disable-in-band` |
| `flood_vlans` | set of integer | VLANs that are flooded unconditionally |
| `flow_tables` | map integer-uuid | Reference to a Flow_Table per OF table-id |

#### Map syntax versus scalar versus set

Map columns use the special syntax `COLUMN:KEY=VALUE`:

```shell
sudo ovs-vsctl set Bridge br-int other_config:mac-aging-time=600
sudo ovs-vsctl set Bridge br-int external_ids:bridge-id=br-int-uuid
```

To delete one key: `ovs-vsctl remove Bridge br-int other_config mac-aging-time`. To clear the whole map: `ovs-vsctl clear Bridge br-int other_config`.

Scalar set: `ovs-vsctl set Bridge br-int protocols=OpenFlow13,OpenFlow15` (multi-element values are separated by commas).

#### Real output sample

```shell
$ sudo ovs-vsctl set Bridge br-int protocols=OpenFlow13,OpenFlow15
$ sudo ovs-vsctl get Bridge br-int protocols
[OpenFlow13, OpenFlow15]
$ sudo ovs-vsctl --columns=name,protocols,datapath_type,mcast_snooping_enable list Bridge br-int
name                : br-int
protocols           : [OpenFlow13, OpenFlow15]
datapath_type       : system
mcast_snooping_enable: false
```

#### Transaction semantics

`set` is atomic per transaction: if you set three columns and one is invalid, all three roll back. `get` does not change the database, it only reads a snapshot. Multiple `set` calls chained with `--` run in the same transaction:

```shell
sudo ovs-vsctl \
  -- set Bridge br-int protocols=OpenFlow13 \
  -- set Bridge br-int other_config:mac-aging-time=600 \
  -- set Bridge br-int other_config:hwaddr=02:00:00:00:00:01
```

One transaction. One network round trip to `ovsdb-server`. Faster than three separate calls and guaranteed to either commit together or roll back together.

#### Real incident scenario

After upgrading OVS from 2.13 to 2.17, a bridge suddenly accepted OF 1.5 flows while the controller only spoke OF 1.3, causing `flow_mod` errors. Root cause: the `protocols` column was empty, OVS 2.17 advertises OF 1.0 to 1.5 by default, and the older controller only supported 1.3, so it replied with an error. Permanent fix: `ovs-vsctl set Bridge br-int protocols=OpenFlow13` to pin the version.

#### Lab exercise

```shell
sudo ovs-vsctl --may-exist add-br br-mtl
sudo ovs-vsctl set Bridge br-mtl \
  protocols=OpenFlow13,OpenFlow15 \
  other_config:mac-aging-time=120 \
  other_config:mac-table-size=4096 \
  external_ids:owner=team-network external_ids:env=lab
sudo ovs-vsctl --columns=name,protocols,other_config,external_ids list Bridge br-mtl
# expect: protocols=[OpenFlow13, OpenFlow15], other_config={mac-aging-time=120, mac-table-size=4096}
```

Pass criterion: every column shows the expected value, all in one transaction.

#### Common errors and diagnosis

| Symptom | Probable cause | How to verify |
|---------|----------------|---------------|
| `unknown column ...` | Misspelled column name or older OVS version that does not yet have it | `ovsdb-client get-schema unix:/var/run/openvswitch/db.sock Open_vSwitch \| jq '.tables.Bridge.columns \| keys'` |
| `set` on a map column with `=` (no `:`) | Wrong syntax; maps need `COL:KEY=VALUE` | Read `ovs-vsctl(8)` section `Database Commands` |
| Forgot the transaction wrapper, so three separate calls partially apply when one fails | Half-applied state remains | Combine into one call with the `--` connector |
| `get` returns `[]` instead of the expected value | The column is an empty set or map, not a scalar | Use `--columns=COL list Bridge` to see the type clearly |

#### Source code reference

The `cmd_get()` and `cmd_set()` handlers are in `lib/db-ctl-base.c` (the generic library shared by `ovs-vsctl`, `ovn-nbctl`, and other ctl utilities); `utilities/ovs-vsctl.c` registers them at startup. Both handlers use the helper `parse_column_key_value()` to recognize scalar, set, or map syntax. Schema validation goes through `ovsdb_idl_txn_verify()`. The Bridge table schema lives in `vswitchd/vswitch.ovsschema` (about 80 columns at OVS 2.17).

#### Version differences

- OVS 1.x: the `Bridge.protocols` column did not exist (default was OF 1.0 only).
- OVS 2.0: `protocols` was added to the schema, supporting multiple versions.
- OVS 2.3: added `mcast_snooping_enable`.
- OVS 2.10: improved `rstp_enable` and added the `flow_tables` map.
- OVS 2.17: stable; no notable new Bridge columns.

#### Comparison with other systems

NETCONF operates through the YANG path `/openconfig-network-instance:network-instances/network-instance[name='br0']/config`, which has semantics close to `set Bridge br0 ...` but uses XML and RPC instead of an OVSDB socket. Cisco IOS has no centralized database; every change is a CLI command into the running configuration, which lacks the transactional batching that OVS provides.

### 9.4.Y.1 `ovs-vsctl show`

#### Concept

`show` is not a row-level dump; it is a hierarchical pretty-printer built from the `Open_vSwitch` table (the root) and walking down through `Bridge`, `Port`, `Interface`, `Controller`, and `Manager`. The output is a tree indented by four characters per level. It prints the record name plus the "interesting" columns declared in the static array `cmd_show_tables[]` defined in `utilities/ovs-vsctl.c`, skipping empty or default columns. This is the first command an on-call engineer types after SSHing into a host because it answers the basic questions: which bridges exist, which ports are attached to which bridge, and which interface has which type. It is fast: a single `ovsdb_idl_run()` query pulls the schema cache and renders.

#### Syntax and parameters

```
ovs-vsctl [--timeout=SEC] [--db=SOCKET] show
```

`show` accepts no positional argument; every flag is global (timeout, db socket, format). Unlike `list`, `show` does not support `--columns`, `--bare`, or `--no-headings` because the tree format is fixed in code (`cmd_show_row()` in `lib/db-ctl-base.c`). When custom output is needed, use `list` rather than grepping `show`.

#### How it works

`cmd_show()` in `lib/db-ctl-base.c` picks the first table in `cmd_show_tables[]` (always `Open_vSwitch`) and iterates rows via `ovsdb_idl_first_row()` and `ovsdb_idl_next_row()`. For each row it calls `cmd_show_row()` recursively: print the `name` column (or the UUID if the table has no `name`), indent by `level*4` spaces, then walk through columns that reference other UUIDs (for example `Bridge.ports` recurses into each `Port`). An `sset` named `shown` tracks already-rendered records to break circular references. Scalar columns (string, int, bool) are printed only when the value differs from the schema default; that is why the output changes when the configuration changes (for example `fail_mode: secure` is shown only after it has been set, hidden by default).

#### Real output sample

```shell
$ sudo ovs-vsctl show
c5e7a4c2-1b9a-4e3f-8a44-3f7e9b1c2d85
    Bridge br-int
        fail_mode: secure
        datapath_type: system
        Port br-int
            Interface br-int
                type: internal
        Port ovn-huyvl-0
            Interface ovn-huyvl-0
                type: geneve
                options: {csum="true", key=flow, remote_ip="10.200.0.12"}
        Port tap-vm-db
            tag: 100
            Interface tap-vm-db
    ovs_version: "2.17.9"
```

The UUID on the first line is the `Open_vSwitch` record (only one row, unique per host). `ovs_version` at the bottom comes from the `Open_vSwitch.ovs_version` column, which `ovs-vswitchd` updates on every startup.

#### When to remember this command

Four typical situations. First, when a bridge is suspected of disappearing after an OVS restart: `show` returns empty output or the bridge is missing, so check `ovs-vswitchd.log` and confirm the OVSDB disk is intact. Second, when an OVN tunnel port does not appear after the LSP has been bound: `show` shows `br-int` but the `ovn-<chassis>-<idx>` port is missing, so `ovn-controller` has not picked up the binding; cross-check `ovn-sbctl list Chassis`. Third, when a packet drop is suspected because of a wrong VLAN tag: `show` lets you confirm `Port tap-vm-db tag: 100` quickly. Fourth, when `show` hangs longer than one second: `ovsdb-server` is overloaded; use `--timeout=3` to force a fast failure so automation does not block.

#### Lab exercise

```shell
sudo ovs-vsctl --may-exist add-br br-show-lab
sudo ovs-vsctl add-port br-show-lab veth0
sudo ovs-vsctl set Port veth0 tag=42
sudo ovs-vsctl set Bridge br-show-lab fail_mode=secure
sudo ovs-vsctl show | sed -n '/br-show-lab/,/^$/p'
```

Pass criterion: the output shows `Bridge br-show-lab`, `fail_mode: secure`, `Port veth0`, `tag: 42`, `Interface veth0`. The learner then runs `ovs-vsctl del-port br-show-lab veth0` and re-runs `show` to see the port disappear immediately (the OVSDB transaction commits, the cache is invalidated, and the tree is re-rendered).

#### Source code reference

`cmd_show()` and `cmd_show_row()` are in `lib/db-ctl-base.c`. The "interesting" tables are defined as the static array `cmd_show_tables[]` at `utilities/ovs-vsctl.c:1018` (the chain `open_vswitch` to `bridge` to `port` to `interface` to `controller` to `manager`). The cache is populated by `vsctl_context_populate_cache()` in `utilities/ovs-vsctl.c`, which uses an `shash` for O(1) name-to-UUID lookup when the caller passes a name as the record-id instead of a UUID.

#### Version differences

- OVS 1.x: the format was similar but lacked the `Manager` block (no manager remote table yet).
- OVS 2.0: added the `Manager` block and the trailing `ovs_version` line.
- OVS 2.4: started printing `datapath_type` for each Bridge (previously always implicit kernel).
- OVS 2.10: added a bond and LACP summary in the Port block when a bond is configured.

#### Comparison with other systems

`brctl show` for Linux bridges prints a similar topology but only one level (bridge to ports); it has no concept of `type=geneve`, `type=internal`, or `type=patch`, and no inline VLAN tag or trunk. `ip -d link show type bridge` is more detailed than `brctl` but still not a tree. A NETCONF GET on the YANG `interfaces-state` produces a similar tree but in XML. Cisco IOS `show interfaces switchport` is semantically equivalent for VLAN and trunk attributes, but does not expose the schema database; the OVS `show` advantage is that the output stays close to the OVSDB structure, which is debug-friendly for automation.

### 9.4.Y.2 `ovs-vsctl list Bridge`

#### Concept

`list` is a database-level subcommand that dumps every column of a row. The form `list Bridge [BRIDGE_NAME ...]` prints raw rows from the `Bridge` table: with no name passed it prints every bridge; with names it prints the selected rows in order. Unlike `show`, `list Bridge` does not follow references (it does not print ports or interfaces below); instead it prints UUIDs in the `ports` set column. The goal is full schema-level reading for automation and debugging, not human-readable topology.

#### Syntax and parameters

```
ovs-vsctl [--bare] [--no-headings] [--columns=COL[,COL...]] list Bridge [NAME|UUID ...]
```

- `--columns` prints only the listed columns, in the declared order.
- `--bare` removes headings and trailing whitespace, convenient for shell parsing.
- `--no-headings` keeps the format but removes the header row.
- The positional argument is a record-id, which can be a full `_uuid`, a unique UUID prefix, or the value of the `name` column.

#### Full output sample

```shell
$ sudo ovs-vsctl list Bridge br-int
_uuid               : 9f3a1c12-77b2-4a01-9d8c-1c2d5e8f7a90
auto_attach         : []
controller          : [b1c2d3e4-...]
datapath_id         : "0000aabbccddeeff"
datapath_type       : system
datapath_version    : "<unknown>"
external_ids        : {}
fail_mode           : secure
flood_vlans         : []
flow_tables         : {}
ipfix               : []
mcast_snooping_enable: false
mirrors             : []
name                : br-int
netflow             : []
other_config        : {disable-in-band="true"}
ports               : [b71d..., 1a2b..., 3c4d...]
protocols           : [OpenFlow13, OpenFlow15]
rstp_enable         : false
rstp_status         : {}
sflow               : []
status              : {}
stp_enable          : false
```

About 24 columns appear: from `_uuid` (always present, cannot be hidden) to feature flags (`stp_enable`, `rstp_enable`, `mcast_snooping_enable`), references (`controller`, `ports`, `mirrors`, `sflow`, `ipfix`, `netflow`), configuration maps (`external_ids`, `other_config`), and read-only status (`datapath_id`, `datapath_version`, `rstp_status`, `status`).

#### How it works

`cmd_list()` in `lib/db-ctl-base.c` calls `list_make_table()` to build the header columns based on `--columns` (or every column plus UUID by default). It then iterates rows via `ovsdb_idl_first_row()` and `ovsdb_idl_next_row()` (when no record-id is passed) or via `ovsdb_idl_index_find()` for a specific name. Each cell is rendered through `ovsdb_atom_to_string()` and `ovsdb_datum_to_string()` into a JSON-ish form (`[uuid, ...]`, `{key=val}`, `"string"`). The default output format is `table` (aligned columns with a header); `--bare` switches to a minimal format suitable for parsing.

#### Workflow

Four standard steps. First, scope: use `list-br` to get the bridge name list and pick the bridge to inspect. Second, full dump: `list Bridge <name>` shows every column; redirect to a file when the output is long. Third, narrow columns: `--columns=name,fail_mode,protocols,datapath_id list Bridge` for automation scripts. Fourth, parse: `--bare --columns=name,datapath_id list Bridge` produces two tab-separated fields, suitable for `awk` or for `jq` after switching to JSON via `--format=json`.

#### Common errors and diagnosis

| Symptom | Probable cause | How to verify |
|---------|----------------|---------------|
| `no row "X" in table Bridge` | Wrong name or the bridge has been deleted | `ovs-vsctl list-br` to see the actual list |
| `controller` shows `[]` while the UI claims "running OVN" | OVN has not written the Controller row | `ovs-vsctl list Controller`, cross-check with `ovn-controller` logs |
| `protocols: []` but you expect OF 1.3 | OVS auto-negotiates by default and does not hard-code | `ovs-vsctl set Bridge X protocols=OpenFlow13` to lock the version |
| `--bare` still emits a heading | You used `--no-headings` alone instead | Combine `--bare --no-headings` |

#### Lab exercise

```shell
sudo ovs-vsctl --may-exist add-br br-list-lab
sudo ovs-vsctl set Bridge br-list-lab \
  fail_mode=secure protocols=OpenFlow13,OpenFlow15 \
  other_config:disable-in-band=true
sudo ovs-vsctl --columns=name,fail_mode,protocols,other_config list Bridge br-list-lab
sudo ovs-vsctl --bare --columns=name,datapath_id list Bridge
```

Pass criterion: the output prints the requested columns in order; `--bare` produces a single line `br-list-lab\t0000....` with no heading.

#### Source code reference

`cmd_list()` lives in `lib/db-ctl-base.c`, with the helper `list_record()` rendering each cell. `vswitchd/vswitch.ovsschema` defines all 24+ Bridge columns with their type constraints (set of UUID, map string-to-string, scalar). `ovsdb-idl.c` provides `ovsdb_idl_first_row()` and `ovsdb_idl_next_row()` through a hash table.

#### Version differences

- OVS 1.x: the Bridge table had about 10 columns (no RSTP, IPFIX, or IGMP snooping yet).
- OVS 2.0: added `protocols`, `flow_tables`, `auto_attach`.
- OVS 2.4: added `mcast_snooping_enable`.
- OVS 2.10: added `rstp_enable` and `rstp_status`.
- OVS 2.17: stable; no notable new Bridge columns relative to 2.13.

#### Comparison with other systems

PostgreSQL `\d Bridge` describes a static schema; OVSDB has no exact equivalent because the schema is built into the library. Linux bridges use `bridge link show` and `bridge fdb show`, which split the information across multiple commands rather than gathering it into one row dump. Cisco NX-OS `show running-config interface vlan X` is a kind of dump, but its output is configuration text rather than a row table.

### 9.4.Y.3 `ovs-vsctl list Port`

#### Concept

`list Port` dumps every column of a row in the `Port` table. Unlike `Bridge`, a port has 18+ columns covering VLAN attributes (`tag`, `trunks`, `vlan_mode`), bonding (`bond_mode`, `lacp`, `bond_active_slave`, `bond_downdelay`), QoS (`qos`), and `external_ids` / `other_config`. `list Port` is the first command an engineer types when investigating "why is no packet passing through this port": the output tells you whether the VLAN tag is correct, whether the port is bonded, which slave the bond is using, and which QoS profile is attached.

#### Syntax and parameters

```
ovs-vsctl [--bare] [--no-headings] [--columns=COL[,COL...]] list Port [NAME|UUID ...]
```

Same syntax as `list Bridge`. A port can be addressed by name (the `name` column is UNIQUE in the table) or by UUID. Note: the port name matches the bridge name when it is the local port (bridge `br0` has port `br0`); to avoid ambiguity, scripts should use UUID.

#### Full output sample

```shell
$ sudo ovs-vsctl list Port tap-vm-db
_uuid               : b71d3a4c-9e12-4b8f-a234-5e6c7d8e9f01
bond_active_slave   : []
bond_downdelay      : 0
bond_fake_iface     : false
bond_mode           : []
bond_updelay        : 0
cvlans              : []
external_ids        : {iface-id="lsp-vm-db"}
fake_bridge         : false
interfaces          : [1a2b3c4d-...]
lacp                : []
mac                 : []
name                : tap-vm-db
other_config        : {}
protected           : false
qos                 : []
rstp_statistics     : {}
rstp_status         : {}
statistics          : {}
status              : {}
tag                 : 100
trunks              : []
vlan_mode           : []
```

Important columns: `tag` (access VLAN); `trunks` (trunk VLAN list); the `vlan_mode` column, whose values are `access`, `trunk`, `native-tagged`, `native-untagged`, or `dot1q-tunnel`; the `bond_mode` column, whose values are `active-backup`, `balance-tcp`, or `balance-slb`; `cvlans` (Q-in-Q); and `protected` (PVLAN feature, OVS 2.13+).

#### How it works

The pipeline matches `list Bridge`: `cmd_list()` invokes `list_make_table()` to build the column header, then `list_record()` for each row. When a column is a set of UUID (for example `interfaces`), the output prints `[uuid1, uuid2]`; the user then uses `list Interface <uuid>` to dereference. This is the OVSDB CLI "explicit dereference" pattern: unlike RDBMS joins, `ovs-vsctl` does not auto-follow foreign keys, which keeps the output deterministic and caller-controlled.

#### Workflow

Five steps to diagnose a port that is not passing packets. First, `list Port <name>` to check whether `tag`, `trunks`, and `vlan_mode` are correct. Second, when the port is a bond, `list Port <name>` to check `bond_mode`, `bond_active_slave`, and `lacp` so you know which slave the bond is using. Third, read `external_ids` to verify the OVN binding: `iface-id="lsp-..."` must match `Logical_Switch_Port.name` in the NBDB. Fourth, `interfaces: [uuid]` must be dereferenced via `list Interface <uuid>` or `--columns=name,type` to see the type. Fifth, `qos: [uuid]` if non-empty means a QoS profile is attached; check it with `list QoS <uuid>` to confirm the rate limit.

#### When to remember this command

A newly booted VM does not get DHCP: `list Port tap-vm1` shows `tag` missing or `external_ids:iface-id` empty, so OVN has not bound the LSP. A bond is failing over slowly: `list Port bond0` shows `bond_downdelay: 200` (a 200ms delay before switching slave), which may be too high for the topology, so reduce it with `bond_downdelay=0`. A trunk port forwards the wrong VLAN: `list Port` shows `trunks: [10, 20]` with VLAN 30 missing, so update with `set Port X trunks=10,20,30`.

#### Lab exercise

```shell
sudo ovs-vsctl --may-exist add-br br-port-lab
sudo ovs-vsctl add-port br-port-lab veth0 tag=42
sudo ovs-vsctl add-port br-port-lab veth1 trunks=10,20,30 vlan_mode=trunk
sudo ovs-vsctl --columns=name,tag,trunks,vlan_mode list Port veth0 veth1
sudo ovs-vsctl set Port veth0 external_ids:iface-id=lsp-test-001
sudo ovs-vsctl --columns=name,external_ids list Port veth0
```

Pass criterion: `veth0` shows `tag=42`, `trunks=[]`, `vlan_mode=[]`; `veth1` shows `tag=[]`, `trunks=[10,20,30]`, `vlan_mode=trunk`. After the `set` of `external_ids`, the column contains `{iface-id="lsp-test-001"}`.

#### Source code reference

`vswitchd/vswitch.ovsschema` defines the Port table with 18+ columns. The `bond_mode` column refers to the `enum bond_mode` defined in `ofproto/bond.h`; bonding semantics are realized across the helpers in `ofproto/bond.c` (the per-feature names include `bond_run`, `bond_should_send_learning_packets`, and the bond setup helpers, not a single function named after the column). The `lacp` column is realized as a feature inside `ofproto/bond.c` rather than a single function named `lacp`. The `bond_active_slave` column (the OVSDB column name, distinct from any function name) is computed and updated by the bonding logic in `ofproto/bond.c`. The `vlan_mode` column refers to a Port-level VLAN mode; its semantics are realized in `ofproto/ofproto-dpif-xlate.c`. The `cvlans` column with `vlan_mode=dot1q-tunnel` is also handled in `ofproto/ofproto-dpif-xlate.c`. The `protected` column (PVLAN) semantics are enforced inline in the relevant `ofproto-dpif` port handlers.

#### Version differences

- OVS 1.x: about 12 columns, no RSTP, no `protected`, no `cvlans`.
- OVS 2.4: added `vlan_mode=dot1q-tunnel` (Q-in-Q) and `cvlans`.
- OVS 2.13: added `protected` (PVLAN isolated port).
- OVS 2.17: stable.

#### Comparison with other systems

Linux bridges use `bridge link show` and `bridge vlan show` to view VLANs per port, but they do not have a single uniform row dump. Cisco IOS `show interface trunk` is semantically close, but the text-based output is harder to parse than the OVSDB JSON form. The OpenStack Neutron API exposes port objects through REST `/v2.0/ports/<id>` with a JSON form close to `list Port`, plus tenant and owner fields.

### 9.4.Y.4 `ovs-vsctl list Interface`

#### Concept

`list Interface` dumps a row from the `Interface` table; it has 30+ columns and is the most detailed entity in the OVSDB stack. An interface can be `internal` (a TAP managed by OVS), `system` (a physical NIC), `geneve`, `vxlan`, or `gre` (tunnel overlay), `patch` (peer to another bridge), or `tap` (a TAP managed by user space). Each type activates a different subset of columns (for example `options:remote_ip` is meaningful only for tunnels, `options:peer` only for patch). The `statistics` column holds per-interface counters (`rx_packets`, `tx_packets`, `rx_bytes`, `tx_bytes`, `rx_dropped`, `tx_dropped`, `collisions`, and so on), refreshed every five seconds through netlink.

#### Syntax and parameters

```
ovs-vsctl [--bare] [--no-headings] [--columns=COL[,COL...]] list Interface [NAME|UUID ...]
```

Common form by name: `list Interface tap-vm1`. For OVN-generated tunnels: `list Interface ovn-<chassis>-<idx>`.

```shell
options             : {csum="true", key=flow, remote_ip="10.200.0.12"}
other_config        : {}
statistics          : {rx_bytes=0, rx_packets=0, tx_bytes=4218, tx_packets=42}
status              : {tunnel_egress_iface="eno1", tunnel_egress_iface_carrier="up"}
type                : geneve
```

Important columns: `type` (the kind); `ofport` (the OpenFlow port number used in flow rules such as `in_port=N`); `options` (tunnel options and patch peer); `mac_in_use` (the MAC the OS is actually using); `mtu` (the effective MTU after the tunnel header is accounted for); `link_state` and `admin_state` (link-layer state and admin state); `statistics` (counters); `status` (read-only diagnostics, for example `tunnel_egress_iface` shows which underlay interface the tunnel is routed through); and `ingress_policing_rate` and `ingress_policing_burst` (rate limit in kbit/s).

#### How it works

`cmd_list()` follows the same path as for `Bridge` and `Port`. The `statistics` column is actively updated by `vswitchd`: `iface_refresh_stats()` in `vswitchd/bridge.c` calls netlink to pull `rtnl_link_stats64` from the kernel on every `bridge_run()` cycle. The `status` column is populated by each datapath provider: for tunnels, `lib/netdev-vport.c` fills `tunnel_egress_iface` via a route lookup; for system NICs, `lib/netdev-linux.c` fills `driver_name` and `driver_version`. As a result `list Interface` returns both configuration and runtime telemetry.

#### Workflow per type

Tunnel (Geneve or VXLAN): check that `options:remote_ip` is the correct remote chassis IP, `options:key=flow` is set (required by OVN), `status:tunnel_egress_iface` shows the right underlay NIC, and `statistics:tx_packets` increases when a VM sends through. Internal port: `mac_in_use` matches the bonding or keepalived configuration, `link_state=up`. Patch port: `options:peer=<peer_name>` and the peer must exist on the other bridge (cross-check with `ovs-vsctl list Interface <peer>`). System NIC: `link_speed` is correct (1 Gb is `1000000000`), `duplex=full`, and `link_resets` counts flaps.

#### When to remember this command

Tunnel does not pass ICMP: `list Interface ovn-X` shows `statistics:tx_packets` increasing but `rx_packets=0`, meaning packets go out but nothing comes back; debug `tunnel_egress_iface` for the correct NIC and confirm the MTU. VM packet drops: `statistics:rx_dropped` is rising due to over-policing; check `ingress_policing_rate`. The `ofport` changes after an OVS restart and breaks flow rules using `in_port=N`: `list Interface --columns=name,ofport` confirms the new number; lock it with `set Interface X ofport_request=N`. A patch port mismatch where `options:peer=br-ex-patch` exists but `br-ex` has no matching interface results in a broken link.

#### Lab exercise

```shell
sudo ovs-vsctl --may-exist add-br br-iface-lab
sudo ovs-vsctl add-port br-iface-lab tap-internal -- set Interface tap-internal type=internal
sudo ovs-vsctl add-port br-iface-lab patch-out -- set Interface patch-out type=patch options:peer=patch-in
sudo ovs-vsctl --may-exist add-br br-peer
sudo ovs-vsctl add-port br-peer patch-in -- set Interface patch-in type=patch options:peer=patch-out
sudo ovs-vsctl --columns=name,type,options,ofport,link_state list Interface tap-internal patch-out patch-in
```

Pass criterion: three interfaces print with the correct `type`, `options:peer` matches in pairs (`patch-out` to `patch-in`), `ofport` is assigned a positive integer, and `link_state=up` for `tap-internal` after `ip link set ... up`.

#### Source code reference

`vswitchd/vswitch.ovsschema` defines the Interface table with 30+ columns. `vswitchd/bridge.c::iface_refresh_stats()` updates the `statistics` column. The `status` column is updated through the datapath provider helpers (for example `lib/netdev-vport.c` for tunnel-specific status entries); there is no separate top-level helper named `iface_refresh_status` in `vswitchd/bridge.c`. Tunnel logic lives in `lib/netdev-vport.c` and `lib/netdev.c`. Patch-port logic for the peer hop lives in `ofproto/ofproto-dpif-xlate.c::compose_output_action__()`.

#### Version differences

- OVS 1.x: about 18 columns, no extensive BFD, LLDP, or CFM.
- OVS 2.0: added `bfd`, `bfd_status`, and `lldp`.
- OVS 2.4: added `mtu_request` to allow OVS-managed MTU.
- OVS 2.6: extended tunnel options (`csum`, `ttl`, `df_default`, `egress_pkt_mark`).
- OVS 2.10: added `link_resets` to track flaps and `error` for diagnostics.
- OVS 2.17: stable.

#### Comparison with other systems

Linux `ip -s -d link show <iface>` provides per-interface statistics and driver details, but it has no concept of `type=patch` or `type=internal` and no `external_ids`. The SNMP MIB `IF-MIB::ifTable` exposes the equivalent of `statistics`, but it is polling-based. `ethtool -S <iface>` is more detailed for vendor-specific counters but is not integrated with the OpenFlow abstraction. The OVS Interface row is the only single point that unifies OVSDB configuration with kernel netdev runtime data.

### 9.4.Y.5 `ovs-vsctl set Port`

#### Concept

`set Port` is a specific use of the generic subcommand `set TABLE RECORD COLUMN[:KEY]=VALUE [...]` applied to the `Port` table. This is how an OVS sysadmin edits port attributes that have no dedicated subcommand (for example there is no `add-tag` or `set-trunks`). `set Port br0 tag=100` sets the access VLAN; `set Port bond0 trunks=10,20,30 vlan_mode=trunk` configures a trunk; `set Port veth0 external_ids:iface-id=lsp-vm-001` attaches an OVN binding ID.

#### Syntax and parameters

```
ovs-vsctl [--no-wait] [--may-exist] set TABLE RECORD COLUMN[:KEY]=VALUE [COLUMN[:KEY]=VALUE ...]
```

Multiple `COLUMN=VALUE` pairs are accepted in a single command, and all of them run in one OVSDB transaction (atomic). The `COLUMN:KEY=VALUE` syntax targets a `map` column (`external_ids:iface-id=X` sets a specific key). `COLUMN=VALUE` is for scalars and sets; for sets, separate values with commas: `trunks=10,20,30`. To clear a set, use `clear Port X trunks` instead of `set`.

#### How it works

`cmd_set()` lives in `lib/db-ctl-base.c`. It parses `RECORD` and locates the row via `find_row()`, then iterates over each `COLUMN[:KEY]=VALUE` in `argv[3..]`. Each pair calls `set_column()` to parse the syntax and then `ovsdb_idl_txn_write()` to store the new value in the in-flight transaction. After the loop, `invalidate_cache()` ensures the next command reads the new value (the IDL cache is stale after a write). Everything happens inside a single `ovsdb_idl_txn_commit_block()` at the end of the command, atomic.

#### Map versus scalar versus set syntax

Map (for example `external_ids`, `other_config`):

```
ovs-vsctl set Port veth0 external_ids:iface-id=lsp-001 external_ids:owner=tenant-A
ovs-vsctl remove Port veth0 external_ids iface-id        # remove a key
```

Scalar (for example `tag`):

```
ovs-vsctl set Port veth0 tag=42
ovs-vsctl clear Port veth0 tag                           # back to [] (default)
```

Set of values (for example `trunks`):

```
ovs-vsctl set Port bond0 trunks=10,20,30
ovs-vsctl add Port bond0 trunks 40                       # add 40 to the set
ovs-vsctl remove Port bond0 trunks 20                    # remove 20
```

#### Best practices

Four guidelines. First, batch multiple columns in the same transaction to avoid inconsistent state: `set Port veth0 tag=42 vlan_mode=access external_ids:iface-id=lsp-X`. Second, use `add` and `remove` for set-of-values rather than `set`, so existing values are not accidentally overwritten. Third, quote strings that contain spaces or special characters: `external_ids:description='"VM web tier"'` (the double quoting is needed because of both the shell and the OVSDB JSON layer). Fourth, prefer `clear` over `set X=[]` because `clear` is semantically correct (it resets to the schema-level default).

#### When to remember this command

Four situations. First, OVN does not bind the LSP: `set Port tap-vm1 external_ids:iface-id=<lsp-uuid-or-name>` so that `ovn-controller` picks it up. Second, the VM is isolated on the wrong VLAN: `set Port tap-vm1 tag=N` switches the access VLAN. Third, the bond does not come up: `set Port bond0 bond_mode=active-backup lacp=off` and check the log. Fourth, a QoS rate limit needs to be applied immediately without restarting: `set Port veth0 qos=@profile -- --id=@profile create QoS type=linux-htb other_config:max-rate=1000000000` (one transaction creates the QoS row and the reference).

#### Common errors and diagnosis

| Symptom | Probable cause | How to verify |
|---------|----------------|---------------|
| `ovs-vsctl: constraint violation: 0 is not in the valid range 1 to 4095` | `tag=0` is not valid | OVSDB schema enforces this; check the `Port.tag` schema; use `clear Port X tag` |
| `set Port X trunks=10,20` overwrote an existing trunk that included 30 | `set` REPLACES; you need `add Port X trunks 30` or pass the full list | Read `list Port X` first; use `add` for incremental updates |
| `external_ids:iface-id=foo` is not picked up by OVN | OVN uses the UUID, not the LSP name | Use `iface-id=$(ovn-nbctl get Logical_Switch_Port lsp-X _uuid)` |
| The command hangs for a long time | OVSDB is locked by another transaction | `ovs-appctl -t ovsdb-server cluster/status OVS_Northbound` to inspect contention |

#### Lab exercise

```shell
sudo ovs-vsctl --may-exist add-br br-set-lab
sudo ovs-vsctl add-port br-set-lab vethA
# Set several columns atomically in one transaction
sudo ovs-vsctl set Port vethA \
  tag=99 \
  vlan_mode=access \
  external_ids:iface-id=lsp-set-lab \
  external_ids:owner=lab-tenant \
  other_config:qos-priority=10
# Verify
sudo ovs-vsctl list Port vethA
# Test add and remove for set-of-values
sudo ovs-vsctl add-port br-set-lab vethTrunk
sudo ovs-vsctl set Port vethTrunk trunks=10,20 vlan_mode=trunk
sudo ovs-vsctl add Port vethTrunk trunks 30
sudo ovs-vsctl remove Port vethTrunk trunks 10
sudo ovs-vsctl --columns=name,trunks list Port vethTrunk
# Expected: trunks=[20, 30]
sudo ovs-vsctl clear Port vethA tag
sudo ovs-vsctl --columns=name,tag list Port vethA
# Expected: tag=[]
```

Pass criterion: the learner sees `tag=[]` after `clear` (back to default), `trunks=[20,30]` after add and remove, and `external_ids` containing both `iface-id` and `owner`.

#### Source code reference

`cmd_set()` lives in `lib/db-ctl-base.c`, with the helper `set_column()` parsing the `COLUMN[:KEY]=VALUE` syntax. `cmd_add()`, `cmd_remove()`, and `cmd_clear()` (also in `lib/db-ctl-base.c`) implement set-of-values and clear semantics. `vswitchd/vswitch.ovsschema` defines the Port-table constraints for each column (for example `tag` is an integer 1 to 4095, `trunks` is a set of integers 0 to 4095, and `vlan_mode` is an enum).

#### Version differences

- OVS 1.x: basic `set`; no `add` or `remove` for sets, so users had to write `set X=val1,val2,val3` each time.
- OVS 2.0: added `add` and `remove` for incremental updates; added `clear` to reset.
- OVS 2.4: supports `--may-exist` for idempotent scripts.
- OVS 2.10: added the Port `protected` column to set PVLAN.
- OVS 2.13: added the Port `cvlans` column for Q-in-Q.

#### Comparison with other systems

Linux bridge: `bridge vlan add dev veth0 vid 42` sets a tag, but the operation is not atomic, not idempotent, and cannot group multiple changes. Cisco IOS `switchport access vlan 42` is semantically equivalent but lives in the CLI tree and is not transactional. The OVSDB advantage is that a single `set` can update multiple columns at multiple levels (scalar plus map plus set) with schema validation, which fits automation and CI/CD.

### 9.4.Z.1 `ovs-ofctl add-flow`

#### Concept

`add-flow` is the subcommand that installs a new flow rule in the bridge's flow table. On the wire, `ovs-ofctl` packages the user-supplied flow spec into an `OFPT_FLOW_MOD` message with the field `command = OFPFC_ADD`, sends it through the OpenFlow connection to `ovs-vswitchd`, and (by default) follows up with an `OFPT_BARRIER_REQUEST` to wait for the ACK that confirms the flow was actually accepted. This is the most basic primitive for pushing forwarding policy into the data plane without a controller.

#### Syntax and parameters

```
ovs-ofctl [-O OpenFlowVERSION] [--bundle] [--no-barrier] add-flow SWITCH FLOW
ovs-ofctl [-O OpenFlowVERSION] [--bundle] add-flows SWITCH FILE
```

- `SWITCH` is the bridge name (`br-int`, `br0`) or the datapath socket (`unix:/var/run/openvswitch/br-int.mgmt`).
- `FLOW` is a comma-separated `key=value` string: `table=N,priority=P,<match>,actions=<list>`. Match fields per §9.4.2.2; action lists per Part 4.9 `OpenFlow action catalog`.
- `-O OpenFlow15` pins the version (the default is 1.0; most modern features need 1.3 or higher).
- `--bundle` packages multiple `add-flow` calls into one atomic OpenFlow 1.4 transaction (see §9.4.Z.4 for the semantics).
- `--no-barrier` skips the barrier after the message; this raises throughput for bulk-load scripts but loses per-rule confirmation.
- The `add-flows SWITCH FILE` form reads multiple flows from a file, one per line, with the same flags.

#### Real output sample

Success: the command prints nothing on stdout and exits 0. This is intentional Unix-convention behavior:

```shell
$ sudo ovs-ofctl -O OpenFlow15 add-flow br-int \
    'table=0,priority=200,in_port=2,ip,nw_dst=10.0.0.5,actions=output:3'
$ echo $?
0
```

When parsing fails, the command prints the error on stderr and exits non-zero:

```shell
$ sudo ovs-ofctl -O OpenFlow15 add-flow br-int 'priority=200,xyz=1,actions=drop'
ovs-ofctl: unknown field xyz
$ echo $?
1
```

When the switch rejects the flow (for example the action is unsupported or the table does not exist), the error appears as an `OFPT_ERROR` reply translated to text:

```shell
$ sudo ovs-ofctl add-flow br0 'table=99,actions=drop'
OFPT_ERROR (xid=0x4): OFPFMFC_BAD_TABLE_ID
OFPT_FLOW_MOD (xid=0x4): ADD table:99 actions=drop
```

#### Semantics when a rule already matches

Per the OpenFlow specification, `add-flow` with an already-existing match plus priority overwrites the actions; the counters (`n_packets`, `n_bytes`) are kept or reset depending on the `OFPFF_RESET_COUNTS` flag. If you want the semantic "add new, modify existing in place", combine with `--strict mod-flows`; if you want failure on duplicate, you must dump first and decide outside `flow_mod`.

#### Lab exercise

```shell
sudo ovs-vsctl --may-exist add-br br-addflow
sudo ovs-vsctl set Bridge br-addflow protocols=OpenFlow13,OpenFlow15
sudo ovs-ofctl -O OpenFlow15 add-flow br-addflow \
  'table=0,priority=10,actions=resubmit(,1)'
sudo ovs-ofctl -O OpenFlow15 add-flow br-addflow \
  'table=1,priority=200,ip,nw_dst=10.0.0.0/24,actions=output:LOCAL'
sudo ovs-ofctl -O OpenFlow15 dump-flows br-addflow
```

Pass criterion: the command runs silently, then `dump-flows` shows two entries with `table=0` priority 10 and `table=1` priority 200 with the right match.

#### Common errors and diagnosis

| Symptom | Probable cause | How to verify |
|---------|----------------|---------------|
| `OFPFMFC_BAD_TABLE_ID` | `table=N` is outside the bridge's supported range | `ovs-ofctl -O OpenFlow15 dump-tables br0` to see `max_entries` per table |
| `unknown field <X>` | The match field does not exist at the version in use | Cross-check Part 4.8 OXM catalog and pin `-O OpenFlow15` |
| The action has no effect after `add` | Forgot the barrier in a bulk script; the switch had not committed at test time | Avoid `--no-barrier` for critical steps |
| `OFPBAC_BAD_OUT_PORT` | `output:NNN` points to a non-existent or DOWN port | `ovs-ofctl show br0` to see the port list and state |

#### Source code reference

The `ofctl_add_flow()` handler in `utilities/ovs-ofctl.c` is a thin wrapper that calls `ofctl_flow_mod(ctx->argc, ctx->argv, OFPFC_ADD)`. The `ofctl_flow_mod()` handler in the same file parses the flow spec via `parse_ofp_flow_mod_str()` in `lib/ofp-flow.c`, then calls `transact_multiple_noreply()` (with barrier) or `transact_noreply()` (with `--no-barrier`). The bundle path goes through `bundle_transact()` in `utilities/ovs-ofctl.c`.

#### Version differences

- OVS 1.x (OpenFlow 1.0): `add-flow` worked on a single table (table 0); no `goto_table`.
- OVS 2.0 (OpenFlow 1.1+): supports the multi-table pipeline; `-O OpenFlow13` is required to use `goto_table`.
- OVS 2.4 (2015): supports the NXM extensions `ct_state` and `ct_zone` in the match (see Part 9.24).
- OVS 2.6 (2016): supports `--bundle` for atomic bulk loads (OpenFlow 1.4 bundle spec).
- OVS 2.10 and later: adds the `clone` and `meter` actions, completing OF 1.5 features.

#### Comparison with other systems

A controller-pushed flow (Ryu, ONOS, Faucet, OVN `ovn-controller`) sends `OFPT_FLOW_MOD` through a pre-established southbound OpenFlow connection. `ovs-ofctl add-flow` uses a local management connection (`ptcp:`, `unix:`) and bypasses the controller. Linux `tc filter add` operates on qdiscs, a completely different abstraction. A Cisco OpenFlow 1.3 capable switch follows the same OF spec, but its local CLI syntax is vendor-specific. `ovs-ofctl` is the most universal CLI for OpenFlow on Linux.

### 9.4.Z.2 `ovs-ofctl dump-flows`

#### Concept

`dump-flows` queries the bridge's current flow table and prints each entry with statistics: cookie, duration, `n_packets`, `n_bytes`, `idle_age`, `hard_age`, priority, match, actions. Wire level: it sends an `OFPT_MULTIPART_REQUEST` with `OFPMP_FLOW`; the switch returns multiple `OFPT_MULTIPART_REPLY` messages containing `OFPST_FLOW` bodies. `ovs-ofctl` aggregates, optionally sorts, then formats as text.

#### Syntax and parameters

```
ovs-ofctl [-O OpenFlowVERSION] [--sort=FIELD] [--no-stats] dump-flows SWITCH [FLOWSPEC]
```

- `FLOWSPEC` (optional) filters by match field: `'table=0'`, `'in_port=2,ip'`, or empty (dump all).
- `--sort=FIELD` sorts by field: `--sort=table,priority` (descending priority within table) is the common idiom.
- `--no-stats` removes counters, useful when diffing two flow-table snapshots without caring about packet counts.
- The filter only decides "which rules to report"; the switch returns rules whose match overlaps the spec, not necessarily equal.

#### Real output sample

```shell
$ sudo ovs-ofctl -O OpenFlow15 --sort=table,priority dump-flows br-int
 cookie=0x0, duration=512.301s, table=0, n_packets=12, n_bytes=1176, priority=200,arp,arp_op=1 actions=resubmit(,8)
 cookie=0x0, duration=512.298s, table=0, n_packets=842, n_bytes=66780, priority=100,ip actions=resubmit(,1)
 cookie=0x0, duration=512.295s, table=0, n_packets=0, n_bytes=0, priority=0 actions=drop
 cookie=0x0, duration=510.001s, table=1, n_packets=842, n_bytes=66780, priority=100,ip,nw_dst=10.0.0.0/24 actions=output:LOCAL
```

Each line is a flow entry. `duration` is the number of seconds since the entry was installed; `n_packets` and `n_bytes` are lifetime counters. If an entry suddenly shows `n_packets=0` after a long time, it may be shadowed by a higher-priority rule or its match never occurs.

#### Filter semantics

The filter `'in_port=2,ip'` is used by the switch as a "search criterion" per OpenFlow specification §7.4.5: a flow matches if the flow's match overlaps the criterion. Consequence: dumping with the filter `nw_dst=10.0.0.5` also returns the flow `nw_dst=10.0.0.0/24` (because /24 contains /32). New learners often confuse this: the dump filter is not a SQL `WHERE`.

#### Lab exercise

```shell
sudo ovs-vsctl --may-exist add-br br-dump
sudo ovs-ofctl add-flow br-dump 'table=0,priority=100,ip,nw_dst=10.0.0.0/24,actions=drop'
sudo ovs-ofctl add-flow br-dump 'table=0,priority=200,ip,nw_dst=10.0.0.5,actions=output:1'
sudo ovs-ofctl --sort=priority dump-flows br-dump 'ip'
sudo ovs-ofctl --no-stats dump-flows br-dump | diff - /tmp/baseline.flows || echo "drift!"
```

Pass criterion: the learner sees the priority=200 entry print before priority=100 (thanks to `--sort`) and recognizes that `--no-stats` removes the `n_packets`, `n_bytes`, and `duration` columns, which makes diffing easier.

#### Common errors and diagnosis

| Symptom | Probable cause | How to verify |
|---------|----------------|---------------|
| Empty dump despite visible traffic | The flow on `br-int` is in a different table (table 0 only `resubmit`s, real work in tables 8 or 16) | Dump without a filter; inspect `table=*` |
| `OFPBRC_EPERM` | The bridge has flow control restrictions imposed by `ovn-controller` through OVSDB | `ovs-vsctl get Bridge br-int controller` to see the controller |
| Output is several seconds slow | An OVN setup has roughly 50K flows on `br-int`; the multipart reply is large | Use `--sort=table` and grep a specific table |
| The `priority` field is missing | The matched rule has an implicit default priority 32768 (OpenFlow default) | The OpenFlow spec defines the default; explicit priority is always clearer |

#### Source code reference

The `ofctl_dump_flows()` handler in `utilities/ovs-ofctl.c` calls `dump_trivial_transaction()` for the no-filter path and `ofctl_dump_flows__()` (in the same file) for paths with filter or sort. The dump path sorts entries via qsort with a custom comparator function defined inline in the dump handler. The receiver loop reads multipart replies via `vconn_recv_block()` until the `OFPMPF_REPLY_MORE` flag clears. The body parser `ofputil_decode_flow_stats_reply()` lives in `lib/ofp-flow.c`.

#### Version differences

- OVS 1.x (OpenFlow 1.0): single table; no `--sort=table`.
- OVS 2.0 and later (OpenFlow 1.3+): multi-table; `dump-flows` adds the `table=N` column.
- OVS 2.5: `idle_age` and `hard_age` appear in the output (Nicira extension).
- OVS 2.10 and later: supports `cookie_mask` for cookie-group dumps (filter the cookie with a bitmask).
- OVS 3.x: refactored sort; added `--sort=table,priority` as the default for stable output ordering in tests.

#### Comparison with other systems

`ovs-dpctl dump-flows` shows megaflows in the kernel datapath (see §9.4.4.3): a different layer, holding wildcarded microflows after translation, not OpenFlow flows. `ovn-sbctl lflow-list` reads Logical_Flow rows in the SBDB at the OVN control-plane level, which is a higher abstraction than the OpenFlow flows on `br-int`. When debugging OVN, the discipline is: `lflow-list` (logical) to `dump-flows br-int` (physical OpenFlow) to `dpctl/dump-flows` (datapath megaflow), three layers top-down.

### 9.4.Z.3 `ovs-ofctl mod-flows`

#### Concept

`mod-flows` modifies existing flows that match the supplied criteria. Wire level: it sends `OFPT_FLOW_MOD` with `command = OFPFC_MODIFY` (loose match) or `OFPFC_MODIFY_STRICT` (strict match) depending on the `--strict` flag. The actions of every matching flow are overwritten with the new action list; the counters (`n_packets`, `n_bytes`) are preserved unless the `OFPFF_RESET_COUNTS` flag is set.

#### Syntax and parameters

```
ovs-ofctl [-O OpenFlowVERSION] [--strict] [--bundle] [--no-barrier] mod-flows SWITCH FLOW
```

- `--strict` switches the command from `OFPFC_MODIFY` to `OFPFC_MODIFY_STRICT`. The core difference: loose modify matches flows that contain the criteria (subset semantic), while strict modify requires an exact match plus the same priority.
- `FLOW` contains both the match (criteria) and the new `actions=...`. Example: `'priority=200,ip,nw_dst=10.0.0.5,actions=output:5'` matches `nw_dst=10.0.0.5` exactly and overwrites the action with `output:5`.
- `--bundle` wraps the operation in an atomic transaction; `--no-barrier` skips confirmation.

#### Loose versus strict

```
loose:  modify where flow's match contains criteria (criteria is a subset)
strict: modify where flow's match == criteria AND priority == criteria.priority
```

Consequence: a loose `mod-flows br0 'ip,actions=drop'` can accidentally overwrite thousands of rules (every rule whose match is a superset of `ip`); strict requires the full match plus priority as when the rule was created. Production discipline: always use `--strict` for `mod-flows` unless a bulk-modify is intended.

#### Real output sample

Success prints nothing:

```shell
$ sudo ovs-ofctl --strict -O OpenFlow15 mod-flows br-int \
    'table=1,priority=200,ip,nw_dst=10.0.0.5,actions=output:5'
$ echo $?
0
$ sudo ovs-ofctl dump-flows br-int 'table=1,nw_dst=10.0.0.5'
 cookie=0x0, duration=8.21s, table=1, n_packets=120, n_bytes=9600, priority=200,ip,nw_dst=10.0.0.5 actions=output:5
```

Note that `n_packets=120` is not reset and `duration=8.21s` is not reset either (the entry is the same one, only the actions changed). If the rule does not exist, `OFPFC_MODIFY` (loose) inserts a new entry (per the OpenFlow 1.0 legacy semantic, which OVS preserves). This is a major pitfall.

#### Lab exercise

```shell
sudo ovs-vsctl --may-exist add-br br-mod
sudo ovs-ofctl add-flow br-mod 'table=0,priority=200,ip,nw_dst=10.0.0.5,actions=output:1'
sudo ovs-ofctl --strict mod-flows br-mod \
  'table=0,priority=200,ip,nw_dst=10.0.0.5,actions=output:2'
sudo ovs-ofctl dump-flows br-mod | grep nw_dst=10.0.0.5
# Expected: actions=output:2, n_packets preserved if traffic was already flowing
sudo ovs-ofctl mod-flows br-mod \
  'priority=200,ip,actions=drop'
# Loose: every rule with priority=200,ip is now actions=drop  (be careful)
```

Pass criterion: the learner can explain why step 2 (strict) modifies exactly one rule, while step 3 (loose, without `--strict`) affects many rules.

#### Common errors and diagnosis

| Symptom | Probable cause | How to verify |
|---------|----------------|---------------|
| `mod-flows` "inserts an extra rule" instead of modifying | `--strict` was not used, criteria match nothing, so loose mod auto-inserts | Dump before and after; use `--strict` |
| Rule disappears after mod | Loose mod with an invalid action; OVS rejects, but `mod` semantics do not delete | Check `ovs-vswitchd.log` |
| Counter unexpectedly reset | The `OFPFF_RESET_COUNTS` flag is set (rare) | OVS does not reset by default on mod |
| Empty action list has no effect | `actions=` is misspelled, the parser treats it as empty (drop, the OF default) | `OFPT_ERROR` usually catches this; if silent, verify the syntax |

#### Source code reference

The `ofctl_mod_flows()` handler in `utilities/ovs-ofctl.c` dispatches by the `--strict` flag, calling `ofctl_flow_mod()` with `OFPFC_MODIFY` or `OFPFC_MODIFY_STRICT`. The strict flag is parsed in `parse_options()` in the same file. The strict-versus-loose match logic runs in `ofproto/ofproto.c::handle_flow_mod__()` upstream through `rule_criteria_init()` with the strict boolean. The bundle path goes through `lib/ofp-bundle.c`.

#### Version differences

- OVS 1.x (OpenFlow 1.0): only `OFPFC_MODIFY` (loose); `OFPFC_MODIFY_STRICT` was added late in the 1.0.0 spec.
- OVS 2.0 (OpenFlow 1.1+): adds `cookie_mask` for modification by controller-set cookie group.
- OVS 2.6 (2016): supports `--bundle`, with mod inside an atomic transaction.
- OVS 2.10: the `clone` action allows mod actions to contain nested action sets; parser updated.
- OVS 3.x: stricter parser for empty `actions=` (clearer warning).

#### Comparison with other systems

Cisco IOS has no analog for "modify a single flow"; an ACL line edit must go `no <line>` then `<new-line>`, which is not atomic. Linux nftables `nft replace rule ...` is the equivalent strict-modify semantic, using a handle ID to target a specific rule. The OpenFlow strict modify in `ovs-ofctl` is more powerful in that it does not require a handle: a complete `match + priority` is enough to uniquely identify the rule, a design that fits SDN declarative style.

### 9.4.Z.4 `ovs-ofctl del-flows`

#### Concept

`del-flows` removes flow rules from the flow table. Wire level: it sends `OFPT_FLOW_MOD` with `command = OFPFC_DELETE` (loose) or `OFPFC_DELETE_STRICT` (strict). Loose delete removes every rule whose match is a superset of the criteria; strict requires an exact match plus the same priority. `del-flows SWITCH` without criteria removes EVERY flow in every table; production must use this with extreme caution.

#### Syntax and parameters

```
ovs-ofctl [-O OpenFlowVERSION] [--strict] [--bundle] [--no-barrier] del-flows SWITCH [FLOW]
```

- Without `FLOW`: deletes every flow on the bridge (except a few rare controller-set rules with the `OFPFF_NO_DELETE` flag).
- `--strict`: requires an exact match plus priority.
- `--bundle`: included in an atomic bundle with other `flow_mod` operations (see §9.4.Z.5 for bundle semantics).
- `out_port=N` or `out_group=N` (with criteria): deletes only rules whose action contains output to that port or group; useful when removing a port without knowing which rules reference it.

#### Real output sample

Success prints nothing:

```shell
$ sudo ovs-ofctl --strict -O OpenFlow15 del-flows br-int \
    'table=1,priority=200,ip,nw_dst=10.0.0.5'
$ echo $?
0
$ sudo ovs-ofctl dump-flows br-int 'table=1,nw_dst=10.0.0.5'
$  # empty (deleted)
```

When the rule does not exist, `del-flows` (loose or strict) silently exits 0; no error. This is idempotent semantics, useful in cleanup scripts. To know "did it actually delete", you must dump before and after and compare.

#### Loose versus strict

```
loose:  delete every flow whose match contains criteria
strict: delete only flows whose match == criteria AND priority == criteria.priority
```

Typical pitfall: `del-flows br-int 'ip'` (loose) deletes every rule that has an IP match, including controller-set rules with more complex matches; this is a production-downtime scenario. Production discipline: always `--strict` unless the goal is truly a bulk-delete.

#### Lab exercise

```shell
sudo ovs-vsctl --may-exist add-br br-del
sudo ovs-ofctl add-flow br-del 'table=0,priority=10,actions=resubmit(,1)'
sudo ovs-ofctl add-flow br-del 'table=1,priority=100,ip,nw_dst=10.0.0.0/24,actions=drop'
sudo ovs-ofctl add-flow br-del 'table=1,priority=200,ip,nw_dst=10.0.0.5,actions=output:1'
# Strict delete: only the specific rule
sudo ovs-ofctl --strict del-flows br-del \
  'table=1,priority=200,ip,nw_dst=10.0.0.5'
# Loose delete: removes many
sudo ovs-ofctl del-flows br-del 'table=1,ip'
sudo ovs-ofctl dump-flows br-del
# Only table=0 priority=10 remains
```

Pass criterion: the learner sees that strict delete keeps the other rules untouched, while loose delete clears table 1.

#### Common errors and diagnosis

| Symptom | Probable cause | How to verify |
|---------|----------------|---------------|
| All traffic drops after `del-flows` | A `del-flows br-int` without criteria was run, removing every OVN-pushed flow | Restart `ovn-controller` to re-push, or use `replace-flows` instead of `del + add` |
| `del-flows --strict` does not delete the expected rule | Forgot `priority=N` in the criteria | Strict requires both match and priority |
| OVN flows reappear by themselves after delete | `ovn-controller` watches SBDB Logical_Flow and re-installs immediately | Delete in SBDB or disable the controller |
| `OFPBRC_EPERM` | Some bridges run in a controller-only mode | Check the mode and role via `list Controller` |

#### Source code reference

The `ofctl_del_flows()` handler in `utilities/ovs-ofctl.c` switches on the `--strict` flag, sending the command code `OFPFC_DELETE_STRICT` or `OFPFC_DELETE`. The loose-versus-strict distinction is implemented in `ofproto/ofproto.c::handle_flow_mod__()` upstream via `rule_criteria_init()` (the same dispatch path used by `mod-flows`). The flags `out_port` and `out_group` are parsed by `parse_ofp_flow_mod_str()` in `lib/ofp-flow.c` and mapped to the `out_port` field of `struct ofputil_flow_mod`.

#### Version differences

- OVS 1.x (OpenFlow 1.0): basic `del-flows`; `--strict` applied exact match only.
- OVS 2.0 (OpenFlow 1.1+): `out_group` is supported; criteria can filter by group reference.
- OVS 2.4: `cookie_mask` allows deletion by cookie group (for example OVN tags rules with the cookie hash of the Logical_Flow UUID).
- OVS 2.6: `--bundle` syncs multiple delete and add operations atomically.
- OVS 3.x: stricter parser; checks criteria when empty (clearer warning when `del-flows` covers the whole bridge).

#### Comparison with other systems

iptables and nftables `iptables -D` require either a rule line number or an exact match argument; nftables uses a handle ID. OpenFlow strict delete uses `match + priority` as the key, bypassing line or handle requirements, closer to SQL `DELETE WHERE`. Cisco ACL `no permit ...` requires repeating the original command, a strict-only semantic with no loose mode. `ovs-ofctl del-flows` is more flexible thanks to having both modes, but that flexibility is itself a source of production pitfalls.

### 9.4.Z.5 `ovs-ofctl dump-tables`

#### Concept

`dump-tables` reports the metadata of every flow table on the bridge: the number of currently installed entries (`active_count`), the number of packets that have looked up the table (`lookup_count`), the number that matched (`matched_count`), and (with OpenFlow 1.3+) the capability set: `max_entries`, the accepted match fields, the supported actions, the allowed instructions. Two messages are involved on the wire: `OFPMP_TABLE` (basic stats) and `OFPMP_TABLE_FEATURES` (detailed capability); `dump-tables` uses `OFPMP_TABLE` by default.

#### Syntax and parameters

```
ovs-ofctl [-O OpenFlowVERSION] dump-tables SWITCH
ovs-ofctl [-O OpenFlowVERSION] dump-table-features SWITCH    # detailed
```

- `dump-tables` is concise: one line per table with name, `lookup_count`, `matched_count`, `active_count`.
- `dump-table-features` is far more detailed: `metadata_match`, `metadata_write`, the `match` field set, the `apply_actions` set, the `next_tables` list. Use it when writing a new controller that needs to know which features the switch supports.

#### Real output sample

`dump-tables` concise form (OF 1.0 or 1.3):

```shell
$ sudo ovs-ofctl -O OpenFlow15 dump-tables br-int
OFPST_TABLE reply (OF1.5) (xid=0x2):
  table 0 ("classifier"):
    active=42, lookup=1842901, matched=1842899
  table 1 ("table1"):
    active=18, lookup=1820445, matched=1820445
  table 2 ("table2"):
    active=0, lookup=0, matched=0
  ...
  table 254 ("table254"):
    active=0, lookup=0, matched=0
```

`active=42` means 42 flows are alive in table 0; `lookup - matched = 2` means two packets did not match any rule (they hit the table-miss flow or were dropped by default).

`dump-table-features` (abbreviated):

```shell
$ sudo ovs-ofctl dump-table-features br-int
  table 0 ("classifier"):
    metadata: match=0xffffffffffffffff write=0xffffffffffffffff
    config=table-miss-controller
    max_entries=1000000
    instructions (table miss and others):
      next tables: 1-254
      instructions: meter,apply_actions,clear_actions,write_actions,write_metadata,goto_table
    matching:
      in_port: arbitrary mask
      eth_src: arbitrary mask
      ...
```

`max_entries=1000000` is the capacity OVS reports (the practical limit is bound by RAM); `next tables: 1-254` indicates which tables `goto_table` can jump to.

#### Operational role

When deploying a new controller (Faucet, OVN, a custom Ryu app), the first step is `dump-table-features` to confirm that the switch supports the features the controller needs. On OVS most features are available, but on real hardware switches (Pica8, Edgecore, Trident-based), the capability set is more restricted; `dump-table-features` is the single source of truth.

#### Lab exercise

```shell
sudo ovs-vsctl --may-exist add-br br-tab
sudo ovs-ofctl add-flow br-tab 'table=0,priority=10,actions=resubmit(,1)'
sudo ovs-ofctl add-flow br-tab 'table=1,priority=100,ip,actions=drop'
sudo ovs-ofctl dump-tables br-tab | head -10
sudo ovs-ofctl dump-table-features br-tab | grep -A 2 'table 0'
# After sending some pings:
sudo ip link set br-tab up
sudo ip addr add 10.99.0.1/24 dev br-tab
ping -c 3 -W 1 10.99.0.99 2>/dev/null || true
sudo ovs-ofctl dump-tables br-tab | grep -E 'table 0|active|lookup'
# Observe lookup_count rising
```

Pass criterion: the learner sees `active=2` across tables 0 and 1 after the `add-flow`, watches `lookup_count` rise after generating traffic, and can list the `next tables` of table 0.

#### Common errors and diagnosis

| Symptom | Probable cause | How to verify |
|---------|----------------|---------------|
| `active_count=0` in every table even after `add-flow` | The `add-flow` used `-O OpenFlow15` but the dump did not, so OVS only returns table 0 | Use the same `-O OpenFlow15` for both calls |
| `matched_count` rises fast but `lookup_count` does not | Counter race on the table-miss path; the relationship is `matched = lookup - missed` | Dump multiple times to average; the per-table stats are accumulated inline in the `ofproto-dpif` handlers |
| Output truncated at table 254 | `active_count=0` from table 64 onward, the tool truncates for brevity | Add `-v` or use `dump-table-features` |
| `dump-table-features` reports `OFPBRC_BAD_VERSION` | The bridge protocol does not enable OF 1.3+ | `ovs-vsctl set Bridge br0 protocols=OpenFlow13,OpenFlow15` |

#### Source code reference

The `ofctl_dump_tables()` handler in `utilities/ovs-ofctl.c` calls `dump_trivial_transaction()` with `OFPRAW_OFPST_TABLE_REQUEST`. The `ofctl_dump_table_features()` handler is more involved: it loops on multipart replies and de-duplicates because features may span more than one packet. Decoders `ofputil_decode_table_stats_reply()` and `ofputil_decode_table_features()` live in `lib/ofp-table.c`. Switch-side stats are accumulated inline in the `ofproto-dpif` handlers and reported through the standard `ofproto-dpif` getter chain (no separate top-level `table_dpif_get_stats` function exists).

#### Version differences

- OVS 1.x (OpenFlow 1.0): only `OFPST_TABLE`; no `OFPMP_TABLE_FEATURES` and no table name.
- OVS 2.0 (OpenFlow 1.3): adds `OFPMP_TABLE_FEATURES`; each table can be named via `set Bridge br0 other_config:tablename0=classifier`.
- OVS 2.5: improved dump format; `next tables` is more accurate.
- OVS 2.10: updated to support OF 1.5 table feature `pipeline_fields`.
- OVS 3.x: dump format is more stable for diff and snapshot tests; small column-name changes.

#### Comparison with other systems

Linux has no direct analog; `tc -s qdisc show` returns stats but the structure is different. A Cisco IOS OpenFlow-capable switch has `show openflow switch <id> table` with equivalent semantics but a different output format. P4Runtime has a `ReadRequest` for `TABLE_ENTRY` and a separate `COUNTER`; the model is different (P4 does not fix 256 tables). `dump-tables` of `ovs-ofctl` is the most universal interface for inspecting an OpenFlow pipeline on Linux plus OVS.

> **Scope of this section.** Five global `ovs-vsctl` options that affect transaction behavior and output formatting: `--timeout=N`, `--columns=COL1,COL2`, `--bare`, `--format=table|csv|json|...`, and `--pretty`. These options have been used throughout §9.4.1 and §9.4.X / §9.4.Y but did not yet have a dedicated section per option. This section collects them as a quick reference: syntax, output sample, operational role, common errors. Upstream sources: man `ovs-vsctl(8)` SYNOPSIS plus DATABASE COMMANDS; the shared option parser `ctl_parse_commands()` in `lib/db-ctl-base.c` (the function `ctl_parse_options` does not exist; the parsing is split across each utility's `parse_options()` and the shared `ctl_parse_commands()` for command tokenization); baseline OVS branch-2.17 (2.17.9 on Ubuntu Jammy).

### 9.4.W.1 `ovs-vsctl --timeout=N`

#### Concept

`--timeout=N` sets the time limit (in seconds) that `ovs-vsctl` waits for a reply from `ovsdb-server` for the entire transaction, including waiting for an OVSDB lock if another client holds one. The default `--timeout=0` means "wait infinitely" (the man page says "infinite by default"). This is an important difference from familiar Linux CLIs where the default is usually a few seconds.

#### Syntax and semantics

```
ovs-vsctl --timeout=N <command> [-- <command>]...
```

`N` is an integer in seconds. After `N` seconds with no reply and no lock acquired, `ovs-vsctl` exits with an error code and prints `ovs-vsctl: connecting to ... failed (Operation timed out)`. Important: the timeout is measured from start to commit; it does not reset between `--` statements.

#### Real output sample

```shell
$ sudo ovs-vsctl --timeout=3 show
ovs-vsctl: connecting to unix:/var/run/openvswitch/db.sock failed (Operation timed out)
$ echo $?
1
```

When `ovsdb-server` is restarting or flushing a large snapshot, a command without a timeout hangs indefinitely. Setting `--timeout=3` lets a health-check script detect the issue early.

#### Operational role

In monitoring and automation scripts, always set a short `--timeout` (3 to 10 seconds). Avoid the infinite default because it makes a monitoring script hang on any OVS issue, which pushes the timeout up to higher-level Nagios or Prometheus checks. For large transactions (such as restoring from backup), use a 60 to 120 second timeout.

#### Common errors and diagnosis

| Symptom | Probable cause | How to verify |
|---------|----------------|---------------|
| Command hangs indefinitely in a new environment | Forgot `--timeout` | Always set `--timeout=5` in scripts |
| `Operation timed out` repeatedly | `ovsdb-server` is flushing a large snapshot (DB above 1 GB) | `ls -lh /etc/openvswitch/conf.db`; raise the timeout to 60 seconds |
| Timeout when called from a container | Socket path differs between container and host | Bind-mount `/var/run/openvswitch/`; set `--db=` to the correct path |

#### Version differences

OVS 1.x defaulted to a 5-second timeout. OVS 2.0 and later switched to infinite (the man page now says `default: infinite`); this change has surprised users who learned on 1.x. OVS 2.17 matches the 2.0+ behavior; no change.

#### Source code reference

The `--timeout` value is parsed in each utility's `parse_options()` and stored in a local timeout variable. The shared command tokenizer `ctl_parse_commands()` in `lib/db-ctl-base.c` does not parse `--timeout` directly; instead, the timeout propagates through the IDL run loop, which honors it via the existing `ovsdb_idl_run` polling cadence (no separate `ovsdb_idl_set_timeout` function exists).

### 9.4.W.2 `ovs-vsctl --columns=COL1,COL2`

#### Concept

`--columns=COL1,COL2,...` restricts the output of `list` (and `find`) to print only the listed columns, in the declared order. Without this option, `list` prints every column of the table (15 to 30 for `Bridge`, `Port`, `Interface`), which is noisy when a script only needs 2 or 3 fields.

#### Syntax

```
ovs-vsctl --columns=COL1,COL2,... list TABLE [RECORD]...
ovs-vsctl --columns=COL1,COL2,... find TABLE [COLUMN=VALUE]...
```

Column names must match the schema (for example `name`, `fail_mode`, `datapath_id` for `Bridge`). A wrong name yields `ovs-vsctl: TABLE does not contain a column named COL`.

#### Real output sample

```shell
$ sudo ovs-vsctl --columns=name,fail_mode,datapath_id list Bridge
name                : br-int
fail_mode           : secure
datapath_id         : "00004e8d3c1f9a02"

name                : br-ex
fail_mode           : standalone
datapath_id         : "0000aabbccddeeff"
```

Compared with the full output of about 20 lines per bridge, the three-column output is far easier to grep and parse.

#### Operational role

Essential in every automation script: combining `--columns=name list Bridge` with a shell loop is the most common pattern. Often paired with `--bare` (see §9.4.W.3) for machine parsing.

#### Common errors and diagnosis

| Symptom | Probable cause | How to verify |
|---------|----------------|---------------|
| `does not contain a column named X` | Wrong column name or wrong table (Bridge has `fail_mode`, Port does not) | `ovs-vsctl list Bridge` (no options) shows the actual schema |
| Empty output | The `find` filter is too tight and `--columns` only prints specific columns | Remove `--columns` to see the full row |
| Output ordering does not match expectations | `--columns` honors the declared order, not alphabetical | Set the column order correctly from the start |

#### Version differences

Available since OVS 1.x. OVS 2.0 and later added `--columns` support for `find` (1.x had it only on `list`). Column names are schema-dependent: a new column added on a schema bump (for example `Bridge.auto_attach` from OVS 2.6) makes `--columns=auto_attach` fail on older servers.

#### Source code reference

The `parse_column_names()` helper in `lib/db-ctl-base.c` parses the `--columns` string and validates it against `ovsdb_idl_class_*` schema metadata. The `--columns` option is honored by `cmd_list` itself when iterating the column projection (no separate `cmd_list_postprocess` function exists).

---

### 9.4.W.3 `ovs-vsctl --bare`

#### Concept

`--bare` removes headings and decorations from the output of `list`, `find`, or `get`. It prints raw values as tab- or whitespace-separated lines, with no column name on the left and no `: ` separator. The goal is shell-friendly output (`while read`, `awk`, `cut`).

#### Syntax

```
ovs-vsctl --bare [--columns=COL,...] list TABLE
ovs-vsctl --bare get TABLE RECORD COLUMN
```

Often paired with `--columns` to specify the needed columns. Without `--columns`, `--bare` still prints every column but without headings, which is rarely useful because the output becomes a hard-to-parse data dump.

#### Real output sample

Comparison:

```shell
$ sudo ovs-vsctl --columns=name,datapath_id list Bridge
name                : br-int
datapath_id         : "00004e8d3c1f9a02"

$ sudo ovs-vsctl --bare --columns=name,datapath_id list Bridge
br-int
00004e8d3c1f9a02
```

The `--bare` output drops the column names, the 20-character padding, and the surrounding quotes around `datapath_id`. Each value is on its own line.

#### Operational role

A common shell-loop pattern:

```shell
for br in $(sudo ovs-vsctl --bare --columns=name list Bridge); do
  sudo ovs-ofctl dump-flows "$br" | wc -l
done
```

Without `--bare`, the loop would absorb the `name` heading and the padding, breaking the logic.

#### Common errors and diagnosis

| Symptom | Probable cause | How to verify |
|---------|----------------|---------------|
| Shell loop fails because the variable holds the column name | Forgot `--bare` | Add `--bare` |
| `--bare` output still contains `"..."` quotes | Some OVS 2.x versions keep quotes for string columns in certain formats | Strip via `tr -d '"'` |
| Set or map columns appear as `[a,b,c]`, hard to parse | `--bare` does not split a set across lines | Use `--format=csv` or post-process |

#### Version differences

Available since OVS 1.x. OVS 2.7 and later refined quote handling for string sets (consistent quote stripping). OVS 2.17 behavior is stable.

#### Source code reference

The shared command tokenizer `ctl_parse_commands()` in `lib/db-ctl-base.c` recognizes the `--bare` flag during option parsing; the `table_format()` function in `lib/table.c` checks the resulting state to skip the header and alignment padding (the function `ctl_parse_options` does not exist).

---

### 9.4.W.4 `ovs-vsctl --format=table|csv|json|...`

#### Concept

`--format` selects the output syntax: `table` (default, human-friendly with aligned columns), `list` (one row per line), `html`, `csv` (comma-separated for spreadsheets), `json` (machine-readable for APIs and scripts). Applies to `list`, `find`, `get`.

#### Syntax

```
ovs-vsctl --format=FORMAT list TABLE
```

The valid values match the `ovsdb-client` formatter (man `ovsdb-client(1)`): `table`, `list`, `html`, `csv`, `json`.

#### Real output sample

```shell
$ sudo ovs-vsctl --format=csv --columns=name,fail_mode list Bridge
name,fail_mode
br-int,secure
br-ex,standalone

$ sudo ovs-vsctl --format=json --columns=name,fail_mode list Bridge
{"data":[["br-int","secure"],["br-ex","standalone"]],"headings":["name","fail_mode"]}
```

CSV imports directly into Excel or LibreOffice. JSON is consumed by `jq` or Python `json.loads()`.

#### Operational role

`--format=csv` for audit reports: dump the topology weekly into CSV, diff between weeks to detect drift. `--format=json` for dashboards and APIs: a REST controller calls `ovs-vsctl --format=json` and pipes the result to the client.

#### Common errors and diagnosis

| Symptom | Probable cause | How to verify |
|---------|----------------|---------------|
| `unknown output format` | Misspelled `--format=Json` (case-sensitive) or `--format=yaml` (not supported) | Use `json` in lowercase |
| One long JSON line is hard to read | JSON is compact by default | Add `--pretty` (see §9.4.W.5) |
| CSV does not escape a value containing a comma | A set or map column has a comma inside | Use `--format=json` instead of CSV for composite columns |

#### Version differences

OVS 1.x had only `table` plus `list`. OVS 2.0 added `csv` plus `html`. OVS 2.4 added `json`. OVS 2.17 is stable across all five formats. A dynamic schema (a column added at runtime) can shift the CSV header between two runs when a schema upgrade happens in between.

#### Source code reference

The `table_print()` function in `lib/table.c` dispatches to internal helpers based on the `output_format` enum: the text format is the default rendered by `table_print` itself, while CSV, JSON, and HTML are emitted by `table_print_csv__()`, `table_print_json__()`, and `table_print_html__()` (with the trailing double-underscore that marks the static internal helpers). Parsing `--format=` is handled by the shared command tokenizer `ctl_parse_commands()` in `lib/db-ctl-base.c`, which maps the string to the enum.

---

### 9.4.W.5 `ovs-vsctl --pretty`

#### Concept

`--pretty` applies to JSON output: it adds newlines and indentation for human readability. It has no effect on `table`, `csv`, or `list` formats. The default JSON output is compact (one long line), optimized for network and disk size; `--pretty` reverses the trade-off for human reading.

#### Syntax

```
ovs-vsctl --pretty --format=json list TABLE
```

`--pretty` is meaningful only when `--format=json` (or when the default format produces a JSON-like value, for example a set-typed column value in `list`).

#### Real output sample

```shell
$ sudo ovs-vsctl --pretty --format=json --columns=name,fail_mode list Bridge
{
  "data": [
    ["br-int", "secure"],
    ["br-ex", "standalone"]
  ],
  "headings": ["name", "fail_mode"]
}
```

Compared with the compact form (already shown in §9.4.W.4), the output is 3 to 4 times longer, but it is far easier to read by eye in a ticket or log file.

#### Operational role

Use it when debugging interactively: dump JSON to a file, open it in an editor, scan by eye. Do not use in production pipelines because it inflates log size; use `jq .` or a Python pretty-printer after receiving compact JSON.

#### Common errors and diagnosis

| Symptom | Probable cause | How to verify |
|---------|----------------|---------------|
| `--pretty` has no effect | The format is not `json` | Add `--format=json` |
| Output is still compact despite `--pretty` | Some non-database subcommands (`add-br`, `set-controller`) do not return JSON | `--pretty` applies only to `list`, `find`, `get` |
| Two JSON dumps differ in diff | One dump is pretty, the other is compact | Normalize both to the same format before diffing |

#### Version differences

OVS 2.4 added `--pretty` together with `--format=json`. OVS 2.17 has not changed the semantics. Equivalent to `ovsdb-client --pretty transact ...`.

#### Source code reference

`json_to_string()` in `lib/json.c` accepts the `JSSF_PRETTY` flag to emit indentation and newlines; the shared command tokenizer `ctl_parse_commands()` in `lib/db-ctl-base.c` recognizes `--pretty` and maps it to this flag during option parsing. The call goes through `table_print_json__()` in `lib/table.c` (the static helper has the double-underscore suffix).

## 9.4.V `ovs-ofctl` flow-mod and dump-flows option group reference (expansion 2026-04)

> **Scope of this section.** Five global `ovs-ofctl` options that affect flow-mod behavior and output format: `--strict`, `--readd`, `--bundle`, `--no-stats`, `--more` (alias `-m`). These options appeared scattered through §9.4.Z (flow-table CRUD) but did not yet have a dedicated section per option. This section collects them as a quick reference: syntax, semantics, output sample, operational role, common errors. Upstream sources: man `ovs-ofctl(8)` (file `utilities/ovs-ofctl.8.in`) plus the parsing code in `utilities/ovs-ofctl.c::parse_options()` long-option array (OVS branch-2.17 baseline 2.17.9). The boolean flags `strict`, `readd`, `bundle` plus the integers `show_stats` / `verbosity` are declared as `static` at the top of `ovs-ofctl.c`.

### 9.4.V.1 `ovs-ofctl --strict`

#### Concept

`--strict` switches the behavior of `mod-flows` and `del-flows` from "wildcard match" (default) to "exact match": only flows whose priority and match field set exactly equal the flow_mod request are modified or deleted. Without `--strict`, a `flow_mod` with the match `dl_src=00:11:22:33:44:55` affects every flow in the table whose match is a superset of the criteria; with `--strict`, the match must be exact.

#### Syntax and semantics

```
ovs-ofctl --strict mod-flows <bridge> "<flow>"
ovs-ofctl --strict del-flows <bridge> "<flow>"
```

In `utilities/ovs-ofctl.c`, a `bool strict` variable is set via the long option `{"strict", no_argument, NULL, OPT_STRICT}`. The `ofctl_flow_mod()` handler reads this variable and selects the OpenFlow opcode: `OFPFC_MODIFY_STRICT` (5) instead of `OFPFC_MODIFY` (1), or `OFPFC_DELETE_STRICT` (4) instead of `OFPFC_DELETE` (3). OpenFlow 1.3 specification §6.4 defines these semantics.

#### Real output sample

```shell
$ sudo ovs-ofctl add-flow br-int "priority=100,dl_src=00:11:22:33:44:55,actions=output:1"
$ sudo ovs-ofctl add-flow br-int "priority=100,dl_src=00:11:22:33:44:55,dl_dst=ff:ff:ff:ff:ff:ff,actions=drop"

$ sudo ovs-ofctl del-flows br-int "dl_src=00:11:22:33:44:55"
# (deletes both flows because non-strict match wildcards every flow with dl_src=...)

$ sudo ovs-ofctl --strict del-flows br-int "priority=100,dl_src=00:11:22:33:44:55"
# (deletes only the first flow; the second still has dl_dst, so no strict match)
```

#### Operational role

Required when modifying or deleting flows in production: without `--strict`, a `del-flows` can accidentally delete many OVN or Neutron flows that share a match-field prefix. Safe pattern: always `dump-flows` first to see which flows are affected, then use `--strict` to limit the scope.

#### Common errors and diagnosis

| Symptom | Probable cause | How to verify |
|---------|----------------|---------------|
| `del-flows` deletes more than expected | Forgot `--strict`, the wildcard match hits every superset flow | Always `dump-flows` first; add `--strict` |
| `--strict del-flows` deletes nothing | Priority does not match exactly (default 32768 differs from the priority used in `add`) | Specify the matching priority, or drop `--strict` |
| `mod-flows` changes flows beyond intent | Non-strict modify is a wildcard match | Use `--strict` plus the full match field |

#### Version differences

Available since OVS 1.x (mapped directly to OpenFlow 1.0 opcodes `OFPFC_MODIFY_STRICT` and `OFPFC_DELETE_STRICT`). OpenFlow 1.3 §6.4 keeps the same semantics. OVS 2.17 has not changed `--strict` behavior.

#### Source code reference

`utilities/ovs-ofctl.c` declares `static bool strict;` at the top of the file. The `parse_options()` handler processes `OPT_STRICT` and sets `strict = true`. In `ofctl_flow_mod()`, the opcode is chosen via the expression `strict ? OFPFC_MODIFY_STRICT : OFPFC_MODIFY` (and similarly for DELETE). The opcode macros are defined in `include/openflow/openflow-common.h` (the `OFPFC_*` enum lives in the cross-version common header, not in `openflow-1.0.h`).

### 9.4.V.2 `ovs-ofctl --readd`

#### Concept

`--readd` applies to the `replace-flows` subcommand: instead of only adding new flows and deleting old flows missing from the file, it **re-adds every** flow in the file even when an identical flow already exists on the switch (same match plus same actions plus same cookie plus same timeout). Purpose: reset the packet and byte counter of the flow to 0.

#### Syntax and semantics

```
ovs-ofctl --readd replace-flows <bridge> <flow-file>
```

In OpenFlow 1.0 and 1.1, re-adding a flow always resets the counter to 0 (no control flag). From OpenFlow 1.2 onward, the behavior depends on the `OFPFF_RESET_COUNTS` flag in `flow_mod`: reset only when the flag is set. `ovs-ofctl --readd` sets this flag automatically when the protocol is 1.2 or later.

#### Real output sample

```shell
$ sudo ovs-ofctl dump-flows br-int | head -3
 cookie=0x0, duration=120.5s, table=0, n_packets=42, n_bytes=3528, priority=100,dl_src=00:11:22:33:44:55 actions=output:1
 cookie=0x0, duration=85.2s, table=0, n_packets=10, n_bytes=840, priority=50 actions=NORMAL

$ cat /tmp/flows.txt
priority=100,dl_src=00:11:22:33:44:55,actions=output:1
priority=50,actions=NORMAL

$ sudo ovs-ofctl --readd replace-flows br-int /tmp/flows.txt
$ sudo ovs-ofctl dump-flows br-int | head -3
 cookie=0x0, duration=0.3s, table=0, n_packets=0, n_bytes=0, priority=100,dl_src=00:11:22:33:44:55 actions=output:1
 cookie=0x0, duration=0.2s, table=0, n_packets=0, n_bytes=0, priority=50 actions=NORMAL
```

`duration` resets to about 0, `n_packets` and `n_bytes` reset to 0. Without `--readd`, identical flows keep their counters.

#### Operational role

Useful when a fresh counter baseline is needed after a configuration change (for example, deploying a new QoS rule and wanting to measure from the deployment point). It does not replace `del-flows` plus `add-flows` (which is heavier) but provides a "soft reset" path for the `replace-flows` pipeline.

#### Common errors and diagnosis

| Symptom | Probable cause | How to verify |
|---------|----------------|---------------|
| Counter does not reset despite `--readd` | Protocol is 1.2 or later but the controller does not set `OFPFF_RESET_COUNTS` | Force the version with `ovs-ofctl -O OpenFlow13 ...`; check `dump-flows --more` |
| `replace-flows` deletes flows beyond intent | `--readd` only affects the add path; the delete path still removes flows missing from the file | Use `add-flows` instead of `replace-flows` for additive semantics |
| Re-add produces a spike of `OFPT_FLOW_MOD` toward the controller | Large flow file; one message per flow | Consider `--bundle` (see §9.4.V.3) for an atomic batch |

#### Version differences

The `OFPFF_RESET_COUNTS` flag is official from OpenFlow 1.2 (spec §5.7.2). OVS 1.x runs only OF1.0 and 1.1 so it always resets. OVS 2.0 and later support OF1.2 and reset only when the flag is set. OVS 2.17 keeps the behavior stable across every protocol version.

#### Source code reference

`utilities/ovs-ofctl.c` declares `static bool readd;`. `OPT_READD` is parsed inside `parse_options()`. The `ofctl_replace_flows()` handler reads this variable and forwards it into `ofputil_flow_mod` flags via `OFPUTIL_FF_RESET_COUNTS`. The flag mapping lives in `lib/ofp-flow.c`.

### 9.4.V.3 `ovs-ofctl --bundle`

#### Concept

`--bundle` packages every `flow_mod` message of a subcommand into **one OpenFlow 1.4 bundle transaction**: either every `flow_mod` applies successfully, or none of them applies (atomic semantics). It eliminates the intermediate state where the switch is still processing part of the flow list.

#### Syntax and semantics

```
ovs-ofctl -O OpenFlow14 --bundle add-flows <bridge> <file>
ovs-ofctl -O OpenFlow14 --bundle replace-flows <bridge> <file>
```

The bundle protocol requires OpenFlow 1.4 or later (spec §7.3.9). `ovs-ofctl` sends the sequence: `OFPT_BUNDLE_CONTROL{OFPBCT_OPEN_REQUEST}`, then a series of `OFPT_BUNDLE_ADD_MESSAGE` (each `flow_mod` wrapped), then `OFPT_BUNDLE_CONTROL{OFPBCT_COMMIT_REQUEST}`. The switch replies with an ACK at every step; if commit fails, the whole bundle rolls back.

#### Real output sample

Observed with `ovs-ofctl monitor br-int -O OpenFlow14`:

```
OFPT_BUNDLE_CONTROL (OF1.4) (xid=0x2): bundle_id=0x1 type=OPEN_REQUEST
OFPT_BUNDLE_CONTROL (OF1.4) (xid=0x3): bundle_id=0x1 type=OPEN_REPLY
OFPT_BUNDLE_ADD_MESSAGE (OF1.4) (xid=0x4): bundle_id=0x1 flags=atomic
  OFPT_FLOW_MOD (OF1.4): ADD priority=100,dl_src=... actions=output:1
OFPT_BUNDLE_ADD_MESSAGE (OF1.4) (xid=0x5): bundle_id=0x1 flags=atomic
  OFPT_FLOW_MOD (OF1.4): ADD priority=50 actions=NORMAL
OFPT_BUNDLE_CONTROL (OF1.4) (xid=0x6): bundle_id=0x1 type=COMMIT_REQUEST
OFPT_BUNDLE_CONTROL (OF1.4) (xid=0x7): bundle_id=0x1 type=COMMIT_REPLY
```

Without `--bundle`, every `flow_mod` is an independent `OFPT_FLOW_MOD`.

#### Operational role

Mandatory for production `replace-flows` when the pipeline has dependent flows (for example, table 0 redirects to table 5 and both must stay synchronized). Without atomicity, traffic during the half-updated pipeline can drop or loop. Production OVN CMSes (kubernetes-ovn, OpenStack neutron-ovn) also use the bundle to push `logical_flow` batches.

#### Common errors and diagnosis

| Symptom | Probable cause | How to verify |
|---------|----------------|---------------|
| `error: unknown OpenFlow version` | Forgot `-O OpenFlow14`; `--bundle` requires OF1.4 or later | Add `-O OpenFlow14` |
| `OFPBFC_BUNDLE_IN_PROGRESS` | A previous bundle has not committed or been discarded; the switch rejects a new one | `ovs-appctl bridge/dump-flows` to view state; restart the switch if it deadlocks |
| Bundle commit fails midway | One `flow_mod` has a syntax error, so the whole bundle rolls back | Validate the flow file in advance; check the switch log `/var/log/openvswitch/ovs-vswitchd.log` |

#### Version differences

The bundle is official from OpenFlow 1.4 (spec §7.3.9). OVS 2.4 added bundle protocol support (commit `66a0712d` in 2015). OVS 2.7 and later support the bundle for OpenFlow 1.3 via the Nicira extension (NXBCT). OVS 2.17 is stable for both native OF1.4 and the NX extension.

#### Source code reference

`utilities/ovs-ofctl.c` declares `static bool bundle;`. `OPT_BUNDLE` is parsed inside `parse_options()`. The `bundle_flow_mod__()` function (in the same file) sends the sequence `OFPBCT_OPEN_REQUEST`, the loop of `OFPT_BUNDLE_ADD_MESSAGE`, then `OFPBCT_COMMIT_REQUEST`. The bundle-message encoding lives in `lib/ofp-bundle.c::ofputil_encode_bundle_ctrl_request()`.

### 9.4.V.4 `ovs-ofctl --no-stats`

#### Concept

`--no-stats` applies to `dump-flows` and `dump-aggregate`: it drops the fields `n_packets`, `n_bytes`, `duration`, `idle_age`, `hard_age` from the output. Goal: reduce output size when only the flow topology (match plus actions) is needed and counters are not relevant.

#### Syntax and semantics

```
ovs-ofctl --no-stats dump-flows <bridge> [<flow-spec>]
```

In the code, `static int show_stats = 1;` is on by default. The long option `{"no-stats", no_argument, &show_stats, 0}` writes the variable to 0 directly (the `getopt_long` target-pointer pattern). The `ofputil_flow_stats_format()` function reads the variable and skips printing stats fields if `show_stats == 0`.

#### Real output sample

Comparison:

```shell
$ sudo ovs-ofctl dump-flows br-int
 cookie=0x0, duration=125.5s, table=0, n_packets=42, n_bytes=3528, priority=100,dl_src=00:11:22:33:44:55 actions=output:1
 cookie=0x0, duration=90.1s, table=0, n_packets=10, n_bytes=840, priority=50 actions=NORMAL

$ sudo ovs-ofctl --no-stats dump-flows br-int
 cookie=0x0, table=0, priority=100,dl_src=00:11:22:33:44:55 actions=output:1
 cookie=0x0, table=0, priority=50 actions=NORMAL
```

The `--no-stats` output is about 30 percent shorter and easier to read by eye when only match and actions need verification; but it loses the counter information needed to diagnose dead flows.

#### Operational role

Useful for diffing topology across two points in time: dump both with `--no-stats` and then `diff` the two outputs, avoiding false positives from continuously changing counters. Also used when exporting a flow snapshot to feed into `add-flows` (the file format should not contain stats because the input parser does not need them).

#### Common errors and diagnosis

| Symptom | Probable cause | How to verify |
|---------|----------------|---------------|
| Loses the ability to detect dead flows | `--no-stats` hides `n_packets=0` | Drop `--no-stats` when debugging traffic |
| Cannot see `idle_age` to find stale flows | `idle_age` and `hard_age` are also hidden | Use the full `dump-flows` then `grep idle_age=` |
| `--no-stats` output still contains `duration` | OVS versions before 2.0 do not hide `duration` | Upgrade to OVS 2.0 or later |

#### Version differences

`--no-stats` is available from OVS 1.x. OVS 2.0 and later extended it to also hide `idle_age` and `hard_age` (NXST extension fields). On OpenFlow 1.5, a few new stats fields (`importance`) are hidden too. OVS 2.17 is stable across every protocol.

#### Source code reference

`utilities/ovs-ofctl.c`: `static int show_stats = 1;` together with the getopt entry `{"no-stats", no_argument, &show_stats, 0}` (sets the pointer directly when the flag appears). The `ofctl_dump_flows__()` function passes `show_stats` down to `ofputil_flow_stats_format()` (`lib/ofp-flow.c`). The skip logic checks `if (show_stats) { ... }` for each stats field.

### 9.4.V.5 `ovs-ofctl --more` (alias `-m`)

#### Concept

`--more` (short form `-m`) raises the verbosity of `dump-flows`, `monitor`, and a few other subcommand outputs: it prints additional `cookie`, full `priority` (not the hidden default), and information for `flags`, `hard_timeout`, `idle_timeout`. Repeating `--more` (`-mm`, `-mmm`) raises verbosity further, for example printing the raw OXM TLV of match fields instead of human-friendly text.

#### Syntax and semantics

```
ovs-ofctl --more dump-flows <bridge>
ovs-ofctl -mm dump-flows <bridge>            # double verbosity
ovs-ofctl -m monitor <bridge> watch:         # verbose monitor
```

In the code, `static int verbosity = 0;`. The long option maps to the case `'m'`: `case 'm': verbosity++; break;`. Each occurrence of `-m` increments the variable by 1.

#### Real output sample

```shell
$ sudo ovs-ofctl dump-flows br-int
 cookie=0x0, duration=120.5s, table=0, n_packets=42, n_bytes=3528, priority=100,dl_src=00:11:22:33:44:55 actions=output:1

$ sudo ovs-ofctl --more dump-flows br-int
 cookie=0x0, duration=120.5s, table=0, n_packets=42, n_bytes=3528, idle_age=2, hard_age=120, priority=100,dl_src=00:11:22:33:44:55 actions=output:1

$ sudo ovs-ofctl -mm dump-flows br-int
 cookie=0x0, duration=120.5s, table=0, n_packets=42, n_bytes=3528, idle_age=2, hard_age=120, idle_timeout=0, hard_timeout=0, send_flow_rem flags=0x0, priority=100,dl_src=00:11:22:33:44:55 actions=output:1
```

Each `-m` level adds fields: level 1 adds `idle_age` and `hard_age`; level 2 adds timeouts plus flags; level 3 and higher print the raw OXM TLV.

#### Operational role

Necessary when debugging stale flows: a large `idle_age` indicates that the flow has not matched traffic recently. `flags=0x1 send_flow_rem` indicates that the controller will receive `FLOW_REMOVED` when the flow expires. `hard_timeout` and `idle_timeout` reveal whether the flow is genuinely permanent or temporary. In production CMSes (Neutron, OVN), default flows have `timeout=0` (permanent); a flow with `timeout != 0` is usually a controller-created temporary flow.

#### Common errors and diagnosis

| Symptom | Probable cause | How to verify |
|---------|----------------|---------------|
| `idle_age` does not appear despite `-m` | OVS too old (before 1.7), `idle_age` not yet supported | `ovs-vsctl --version` |
| `-m` does not change `monitor` output | `monitor` verbosity is set at a different level (`watch:flow`) | Combine with `-m watch:flow` |
| Raw OXM TLV too long, log overflows | `-mmm` prints raw bytes for every match field | Reduce to `-m` or `-mm` |

#### Version differences

OVS 1.7 added `idle_age` and `hard_age` (NXST extension). OVS 2.0 and later added `-mm` for timeouts plus flags. OVS 2.4 and later extended `-mmm` for OXM raw TLV on OF1.3. OVS 2.17 is stable across every verbosity level.

#### Source code reference

`utilities/ovs-ofctl.c` declares `static int verbosity;`. The long option `{"more", no_argument, NULL, 'm'}` together with the case `'m': verbosity++;` in `parse_options()` increments the variable. The `verbosity` variable passes down to `ofputil_flow_stats_format()` (`lib/ofp-flow.c`) through the extended `show_stats` parameter (with the same enum name), controlling the detail level. Each stats field has a gate of the form `if (verbosity >= N) { ... }`.

## 9.4.O `ovs-vsctl` heading, idempotency, reference, and sync option group (expansion 2026-04)

> **Scope of this section.** Five global `ovs-vsctl` options used in automation scripts and multi-step transactions: `--no-headings` (strip heading row), `--if-exists` (idempotent delete or get), `--may-exist` (idempotent add), `--id=@NAME` (symbolic reference for a row created in the same transaction), and `--no-wait` (skip waiting for `ovs-vswitchd` reconfigure). The first four are the difference between a naive script and a production-grade script that survives rerun and race conditions; the fifth is essential when `ovs-vswitchd` is not running or when you want to batch many changes and let the daemon pick them up afterwards. Upstream sources: man `ovs-vsctl(8)` SYNOPSIS plus DATABASE COMMANDS. Option parsing is split across two tiers: option parsing happens in each utility's `parse_options()` (vsctl-specific options such as `--no-wait` are parsed in `utilities/ovs-vsctl.c`), with command tokenization shared via `ctl_parse_commands()` in `lib/db-ctl-base.c` for the database-related options (the function `ctl_parse_options` does not exist). Baseline: OVS branch-2.17 (2.17.9 on Ubuntu Jammy).

---

### 9.4.X.1 `ovs-vsctl --no-headings`

#### Concept

`--no-headings` removes the heading row at the top of `list` or `find` table-style output. By default, `ovs-vsctl --format=table list Bridge` prints headings such as `name` plus a `----` separator before the data. When a script uses `awk` or `cut` to parse, the heading row breaks the logic because it is not real data. The man page says: *"This option suppresses the heading row that otherwise appears in the first row of table output."*

#### Syntax

```
ovs-vsctl --no-headings [--columns=COL,...] [--format=FORMAT] list TABLE
ovs-vsctl --no-headings [--columns=COL,...] find TABLE [COL=VAL]...
```

`--no-headings` applies to every format that has a heading (`table`, `csv`, `html`, `json`). It differs from `--bare`: `--bare` strips heading plus decoration (padding, quotes, separators); `--no-headings` only strips the heading row, keeping the column-aligned format. In other words: `--bare` is a superset of `--no-headings` in effect.

#### Real output sample

Three-level comparison:

```shell
$ sudo ovs-vsctl --columns=name,fail_mode --format=csv list Bridge
name,fail_mode
br-int,secure
br-ex,standalone

$ sudo ovs-vsctl --no-headings --columns=name,fail_mode --format=csv list Bridge
br-int,secure
br-ex,standalone

$ sudo ovs-vsctl --bare --columns=name,fail_mode list Bridge
br-int secure
br-ex standalone
```

`--no-headings` keeps the CSV format (comma-separated) but drops the `name,fail_mode` line. `--bare` also drops the CSV separator and prints each cell tab- or space-separated. For spreadsheet scripts, `--no-headings --format=csv` is the most reasonable choice; for shell `read -r`, `--bare` is faster.

#### Operational role

`--no-headings` mostly appears in audit scripts that build CSVs for long-term archives: the heading row is dropped so that concatenating output across days does not multiply the heading. It is also used when feeding output into `awk -F,` without manually skipping the first line.

#### Common errors and diagnosis

| Symptom | Probable cause | How to verify |
|---------|----------------|---------------|
| Output still has a heading | Using the default `--format=table`, some OVS versions still print the `----` separator | Combine with `--format=csv` or use `--bare` |
| `awk` counts the wrong number of bridges | Forgot `--no-headings`, `awk` counts the heading row too | `ovs-vsctl --no-headings --columns=name list Bridge \| wc -l` |
| Merging CSVs across days repeats the heading | Some dump files lacked `--no-headings` | Normalize: every dump script uses `--no-headings` |

#### Version differences

Available since OVS 1.x (same time as `--columns` and `--bare`). OVS 2.0 and later extended it to formats `csv`, `html`, `json`. OVS 2.17 has not changed the semantics.

#### Source code reference

The shared command tokenizer `ctl_parse_commands()` in `lib/db-ctl-base.c` recognizes the `--no-headings` flag and sets a `headings = false` state during option parsing (case `OPT_NO_HEADINGS`). When formatting output, `table_print()` in `lib/table.c` checks the state: if false, it skips the header row when invoking `table_print_csv__()`, `table_print_html__()`, or `table_print_json__()` (the static helpers carry the trailing double-underscore; the text format is the default rendered by `table_print` itself).

---

### 9.4.X.2 `ovs-vsctl --if-exists`

#### Concept

`--if-exists` turns "the record does not exist" from an error into a no-op. It applies to `del-br`, `del-port`, `del-controller`, `del-manager`, `del-fail-mode`, `remove`, `clear`, `get`, `destroy`, `--id=@x get`. The man page says: *"Without --if-exists, attempting to delete a bridge that does not exist is an error. With --if-exists, attempting to delete a bridge that does not exist has no effect."* This is the idempotency property and is the foundation for automation that survives rerun.

#### Syntax

```
ovs-vsctl --if-exists del-br BRIDGE
ovs-vsctl --if-exists del-port [BRIDGE] PORT
ovs-vsctl --if-exists get TABLE RECORD COLUMN
ovs-vsctl --if-exists remove TABLE RECORD COLUMN VALUE
ovs-vsctl --if-exists clear TABLE RECORD COLUMN
```

`--if-exists` must come BEFORE the subcommand it applies to. When chaining multiple subcommands with `--`, this flag applies to every following subcommand (global flag) unless reset.

#### Real output sample

```shell
$ sudo ovs-vsctl del-br br-test
ovs-vsctl: no bridge named br-test
$ echo $?
1

$ sudo ovs-vsctl --if-exists del-br br-test
$ echo $?
0
```

The first command fails because `br-test` does not exist; the second exits 0 regardless, the correct "ensure absent" semantics.

#### Operational role

Most important in cleanup scripts and reconciliation loops:

```shell
for br in br-test1 br-test2 br-test3; do
  sudo ovs-vsctl --if-exists del-br "$br"
done
```

The loop runs even when only some bridges exist; no pre-check needed. In CMSes such as Neutron and OVN, the agent often reruns the same cleanup multiple times during state reconciliation; without `--if-exists`, the agent log fills with "no bridge named" noise that pollutes monitoring.

#### Common errors and diagnosis

| Symptom | Probable cause | How to verify |
|---------|----------------|---------------|
| `--if-exists` placed in the wrong spot | Placed after the subcommand instead of before | Place right after `ovs-vsctl` or before the verb |
| Still fails with "no row to delete" | Subcommand does not support `--if-exists` (for example `set` on a non-existent record) | Use `--may-exist add-br` then `set` |
| Race with another agent deleting concurrently | Idempotent but not atomic across clients | `--if-exists` plus retry loop, or an OVSDB transaction through the API |

#### Version differences

Available since OVS 1.x for `del-*` subcommands. OVS 2.0 and later extended it to `get`, `remove`, `clear`, `destroy`. OVS 2.6 and later added support for `--id=@x get` on non-existent records. OVS 2.17 is fully stable.

#### Source code reference

The shared command tokenizer `ctl_parse_commands()` in `lib/db-ctl-base.c` recognizes the `--if-exists` flag and sets `if_exists = true` during option parsing (case `OPT_IF_EXISTS`). Each subcommand handler checks this state before raising an error: for example `cmd_del_br()` in `utilities/ovs-vsctl.c` follows the pattern `if (must_exist || (br && !ctx->options & ...)) { ... ctl_fatal(...) }`. In general: if the record is not found and `if_exists` is set, the handler returns early without an error exit.

### 9.4.X.3 `ovs-vsctl --may-exist`

#### Concept

`--may-exist` turns "the record already exists" from an error into a no-op. It applies to `add-br`, `add-port`, `add-bond`, `add-bond-iface`. The man page says: *"Without --may-exist, attempting to create a bridge that exists is an error. With --may-exist, this command does nothing if bridge already exists as a real bridge."* This is the counterpart of `--if-exists`: together they cover both directions of idempotency (ensure-present plus ensure-absent).

#### Syntax

```
ovs-vsctl --may-exist add-br BRIDGE [PARENT VLAN]
ovs-vsctl --may-exist add-port BRIDGE PORT [-- set Interface PORT type=...]
ovs-vsctl --may-exist add-bond BRIDGE BOND IFACE...
```

When the record already exists AND its attributes match, `--may-exist` skips entirely. If the record exists BUT the attributes differ (for example `add-br br0 br0-parent 100` while `br0` is a real bridge with no VLAN), the command still fails because the spec does not match; that is the correct semantic, not a bug.

#### Real output sample

```shell
$ sudo ovs-vsctl add-br br-int
ovs-vsctl: cannot create a bridge named br-int because a bridge named br-int already exists
$ echo $?
1

$ sudo ovs-vsctl --may-exist add-br br-int
$ echo $?
0
```

Idempotent rerun of a configuration script is the most common pattern for this option.

#### Operational role

In Ansible, Puppet, or Salt modules for OVS, every `add-br` and `add-port` includes `--may-exist`. A bootstrap script for a compute node (Neutron or OVN) runs hundreds of times across the lifecycle (boot, reboot, reconfig); without `--may-exist`, every run after the first exits non-zero and stops the pipeline.

#### Common errors and diagnosis

| Symptom | Probable cause | How to verify |
|---------|----------------|---------------|
| `--may-exist add-br` still fails | The bridge exists with a different parent or VLAN | `ovs-vsctl br-to-parent BR; ovs-vsctl br-to-vlan BR` to compare |
| `add-port` skipped but the port type is wrong | `--may-exist` skips when the port name matches, not the `Interface.type` | After `add-port`, always `set Interface NAME type=...` to force the config |
| Race between two scripts adding | OVS commits serialize through RAFT or standalone; no manual lock needed | OK; just use `--may-exist` |

#### Version differences

Available since OVS 1.x for `add-br` and `add-port`. OVS 2.0 and later extended it to `add-bond`. OVS 2.6 and later added `add-bond-iface`. OVS 2.17 has not changed.

#### Source code reference

The shared command tokenizer `ctl_parse_commands()` in `lib/db-ctl-base.c` recognizes the `--may-exist` flag and sets `may_exist = true` during option parsing (case `OPT_MAY_EXIST`). The `cmd_add_br()` handler in `utilities/ovs-vsctl.c` checks: if the bridge exists, the attributes match, and `may_exist` is set, return success without creating a new row. The match logic lives in `find_bridge()` in the same module: it compares the name plus parent plus VLAN-tag tuple.

### 9.4.X.4 `ovs-vsctl --id=@NAME`

#### Concept

`--id=@NAME` assigns a symbolic name to the UUID of a row created within the same `ovs-vsctl` transaction so that later subcommands (in the same invocation, chained with `--`) can reference it. The man page says: *"If @name is specified, then the UUID for record may be referred to by that name later in the same ovs-vsctl invocation in contexts where a UUID is expected."* This is the core technique for multi-step transactions: create a Mirror that references source Port plus Interface in one atomic command, or create QoS plus Queue and attach to a Port.

#### Syntax

```
ovs-vsctl -- --id=@NAME create TABLE COL=VAL... [-- <next-cmd referencing @NAME>]...
```

`@NAME` is a short identifier (for example `@m`, `@q0`, `@iface-eth0`). In a later subcommand, use `@NAME` anywhere a UUID is expected. Every `@NAME` in the same invocation must be unique. The whole chain commits as one OVSDB transaction: if any step fails, no row is created.

#### Real output sample

Create a Mirror that references the source Port plus the output Port in one command:

```shell
$ sudo ovs-vsctl \
    -- --id=@p0 get Port eth0 \
    -- --id=@p1 get Port eth1 \
    -- --id=@m create Mirror name=m0 select-src-port=@p0 output-port=@p1 \
    -- set Bridge br-int mirrors=@m
6f8b1c9a-4d2e-4b8f-9c1a-0f3e5d7c8b91
```

The UUID printed at the end is the UUID of the new Mirror (`@m`). The bridge `br-int` already has its `mirrors` column updated to contain this UUID. Without `--id=@`, you would have to create the Mirror first, retrieve the UUID, then run a second command to attach it. That sequence is not atomic and is exposed to races.

#### Operational role

Every QoS, Queue, Mirror, or multi-row configuration needs this pattern:

- QoS: `--id=@q create Queue ... -- --id=@qos create QoS queues:0=@q -- set Port eth0 qos=@qos`.
- SPAN Mirror for debugging: the pattern shown in the sample.
- Multi-controller setup: create multiple Controller rows and attach the list to `Bridge.controller`.

Without `--id=@`, you must use multiple separate `ovs-vsctl` invocations, lose atomicity, and risk interleaving with another agent.

#### Common errors and diagnosis

| Symptom | Probable cause | How to verify |
|---------|----------------|---------------|
| `unknown symbol "@m"` | Forgot `--id=@m` on the defining subcommand | List the chain: each `@NAME` must have a `--id=` before any reference |
| `multiple definitions of @m` | Two subcommands both use `--id=@m` | Use unique names: `@m1`, `@m2` |
| Symbol resolves to the wrong UUID | Wrong subcommand order (reference before definition) | OVS 2.x supports forward references; OVS 1.x requires definition first. Check the version |
| Transaction commits but the row does not appear | `create` without attaching the symbol to a parent row, leaving an orphan | Always chain `-- set Parent ref=@NAME` |

#### Version differences

OVS 1.x requires `--id=` to define before reference (forward reference fails). OVS 2.0 and later allow forward references: a subcommand uses `@m` and a later `--id=@m create ...` is resolved by a two-way pass. OVS 2.17 is stable for both forward and backward references.

#### Source code reference

`lib/db-ctl-base.c::parse_command()` handles `--id=@NAME` and stores the name-to-UUID-placeholder mapping in `struct ctl_command_syntax`. When the chain commits, `do_vsctl()` (`utilities/ovs-vsctl.c`) iterates two passes: pass 1 collects every `--id=@`, pass 2 resolves references and commits the OVSDB transaction via `ovsdb_idl_txn_commit_block()`. If any step fails, the whole transaction rolls back.

### 9.4.X.5 `ovs-vsctl --no-wait`

#### Concept

`--no-wait` lets `ovs-vsctl` exit immediately after committing the OVSDB transaction, without waiting for `ovs-vswitchd` to finish processing the change (reconfigure the bridge, install datapath flows, open a new port). By default, `ovs-vsctl` waits until `ovs-vswitchd` sets `next_cfg` greater than or equal to `cur_cfg` before printing; with `--no-wait`, it exits right after the OVSDB ack. The man page says: *"Prevents ovs-vsctl from waiting for ovs-vswitchd to reconfigure itself according to the modified database. This option should be used if ovs-vswitchd is not running."*

#### Syntax

```
ovs-vsctl --no-wait <any-config-changing-command> [-- <more>]...
```

Applies to every DB-changing subcommand: `add-br`, `add-port`, `set`, `remove`, `clear`, `del-*`. Has no meaning for pure-read subcommands such as `list`, `get`, or `show`.

#### Real output sample

The difference is most obvious when `ovs-vswitchd` is not running:

```shell
$ sudo systemctl stop ovs-vswitchd
$ sudo ovs-vsctl add-br br-x
^C  # hangs forever waiting for the vswitchd ack
$ sudo ovs-vsctl --no-wait add-br br-x
$
```

When vswitchd is healthy, the difference is 50 to 200 ms (the time vswitchd takes to apply the config). In a batch script that adds 100 ports, the accumulation reaches 100 times 100 ms equals 10 seconds; `--no-wait` cuts that down to a few hundred ms in total.

#### Operational role

Two main use cases:

1. **Bootstrap when `ovs-vswitchd` is not yet running.** An init or cloud-init script creates a bridge before the service starts; the default would hang, `--no-wait` lets it pass.
2. **Batching many small changes.** Instead of waiting for reconfigure on every `set`, use `--no-wait` per command and at the end of the batch run one command without `--no-wait` for sync. Better yet, use a `--` chain to merge into one transaction (preferred).

Trade-off: the client does not know when the datapath has actually applied the change. A test right after `--no-wait` might see the port not yet up. In a CI lab, combine with a poll: `until ovs-vsctl get Interface eth0 ofport > /dev/null; do sleep 0.1; done`.

#### Common errors and diagnosis

| Symptom | Probable cause | How to verify |
|---------|----------------|---------------|
| Test right after `add-port` shows the port not yet active | `--no-wait` skipped the reconfig sync | Use `--may-exist` without `--no-wait`, or poll `ofport != []` |
| `ovs-vsctl` hangs even though vswitchd is running | vswitchd lock contention or upcall storm | `ovs-appctl ovs/vswitchd/version`; if appctl also hangs, vswitchd is genuinely stuck |
| `--no-wait` does not speed up a batch | Each command is its own invocation; the main overhead is `fork` plus OVSDB connect | Merge into one `-- chain ...` invocation instead of spamming `ovs-vsctl` repeatedly |

#### Version differences

Available since OVS 1.x. OVS 2.0 and later tuned the wait semantics: before OVS 2.0, the wait used `cur_cfg` and `next_cfg` polling at 100 ms; OVS 2.0 and later use OVSDB notifications, lowering wait latency to about 10 ms when `ovs-vswitchd` is healthy. OVS 2.17 is stable. Unrelated to `--timeout` (timeout measures wall time, no-wait measures logical sync; the two concepts are independent).

#### Source code reference

Unlike the previous four options (shared through `lib/db-ctl-base.c`), `--no-wait` is vsctl-specific: parsed in `utilities/ovs-vsctl.c::parse_options()` and sets `wait_for_reload = false`. After `ovsdb_idl_txn_commit_block()` succeeds, `do_vsctl()` checks the variable: if `true`, it runs the `idl_run` plus `cur_cfg` vs `next_cfg` poll loop; if `false` (no-wait), it returns immediately. The `cur_cfg` and `next_cfg` columns belong to the `Open_vSwitch` table (singleton row).

## 9.4.U OVSDB-table generic subcommand reference (expansion 2026-04)

> **Scope of this section.** `ovs-vsctl` provides a generic CRUD subcommand set that applies uniformly to every OVSDB table: `list`, `get`, `set`, `add`, `remove`, `clear`, `find`, `create`, `destroy`. The set is implemented in `lib/db-ctl-base.c` (functions `cmd_list`, `cmd_get`, `cmd_set`, `cmd_add`, `cmd_remove`, `cmd_clear`, `cmd_find`, `cmd_create`, `cmd_destroy`) and registered by `utilities/ovs-vsctl.c` through `vsctl_cmd_init()`. The dedicated, table-specific subcommands (`add-br`, `set-controller`, and so on) are convenience wrappers for common cases only; a skilled engineer must master the generic set because OVSDB tables such as `Manager`, `QoS`, `Queue`, `Mirror`, `NetFlow`, `IPFIX` have no wrapper. This section collects the five generic subcommand groups for the most common tables: `Bridge`, `Port`, `Interface`, `Controller`, `SSL`. Unlike §9.4.Y (which goes deep into specific commands such as `list Bridge`, `list Port`), this section summarizes the full CRUD family applicable to a table so the engineer gets the "everything you can do on table X" overview. Upstream sources: `lib/db-ctl-base.c`, `utilities/ovs-vsctl.c`, the schema `vswitchd/vswitch.ovsschema`, man `ovs-vsctl(8)` §`Database Commands`. Baseline: OVS branch-2.17 (2.17.9 on Ubuntu Jammy).

---

### 9.4.U.1 Subcommand family for the `Bridge` table

The `Bridge` table (singleton-ish: many rows, each with a unique name) stores every configured OVS bridge and is referenced from `Open_vSwitch.bridges` as a `set of UUID`. The generic family consists of nine commands when applied to `Bridge`:

| Command | Typical syntax | Role |
|---------|----------------|------|
| `list Bridge` | `ovs-vsctl list Bridge [BRIDGE]` | Dump every column of one or all rows |
| `get Bridge` | `ovs-vsctl get Bridge BRIDGE COLUMN[:KEY]` | Read a column value (parse-friendly) |
| `set Bridge` | `ovs-vsctl set Bridge BRIDGE COLUMN[:KEY]=VAL` | Write a column value; idempotent |
| `add Bridge` | `ovs-vsctl add Bridge BRIDGE COLUMN VAL` | Add an element to a `set` or `map` column |
| `remove Bridge` | `ovs-vsctl remove Bridge BRIDGE COLUMN VAL` | Remove an element from a `set` or `map` |
| `clear Bridge` | `ovs-vsctl clear Bridge BRIDGE COLUMN` | Reset a column to default |
| `find Bridge` | `ovs-vsctl find Bridge COLUMN=VAL` | Filter rows by condition |
| `create Bridge` | `ovs-vsctl create Bridge name=br0 ...` | Create a row directly (rare, prefer `add-br`) |
| `destroy Bridge` | `ovs-vsctl destroy Bridge BRIDGE` | Delete a row (prefer the safer `del-br`) |

In production, `add-br` and `del-br` are safe wrappers around `create Bridge` and `destroy Bridge`: `add-br` adds a new UUID to `Open_vSwitch.bridges` and creates the row at the same time, while a bare `create Bridge` creates an "orphan" row that is not referenced. The bridge will not appear in `ovs-vsctl show`. Operating rule: use the wrapper when one exists (`add-br`, `del-br`); fall back to the generic form only when an atomic chain with `--id=@` is required (see §9.4.X.4).

Common mistakes: using `set Bridge X protocols=OpenFlow13,OpenFlow15` to set the whole set instead of `add Bridge X protocols OpenFlow15` to append to the existing set. `set` overwrites the whole column and can remove an older version; `clear Bridge X protocols` resets to the default (`OpenFlow10`) and breaks the session with a controller using OF1.3 or later. For map columns (`other_config`, `external_ids`), `set` overwrites the whole map, while `set Bridge X external_ids:key=val` only edits one entry. The functions `cmd_set()`, `cmd_add()`, `cmd_remove()`, `cmd_clear()` in `lib/db-ctl-base.c` (around lines 1240 to 1340) distinguish these cases via `parse_column_key_value()`. The `Bridge` schema in `vswitchd/vswitch.ovsschema` defines about 24 columns (OVS 2.17), among which `name`, `datapath_type`, `fail_mode`, `protocols`, `controller`, `ports`, `mirrors`, `flood_vlans`, `auto_attach`, `other_config`, `external_ids`, `status`, `stp_*`, `rstp_*`, `mcast_snooping_enable`, `ipfix`, `netflow`, `sflow` are the columns most often manipulated. Compared with Cisco IOS using a config block, OVSDB generic CRUD plus transactions makes automation scripts deterministic and atomic at the row level.

Real command samples for a few use cases:

```shell
$ sudo ovs-vsctl find Bridge fail_mode=secure
$ sudo ovs-vsctl add Bridge br-int protocols OpenFlow15
$ sudo ovs-vsctl set Bridge br-int other_config:disable-in-band=true
$ sudo ovs-vsctl --columns=name,protocols,fail_mode list Bridge
```

Those four commands are the daily-ops skeleton: list every secure-mode bridge, lift the OpenFlow protocol version, disable the in-band controller path (improves stability when the controller is on the same host), and summarize the three most important columns when auditing a cluster.

### 9.4.U.2 Subcommand family for the `Port` table

The `Port` table represents a logical port attached to a bridge, referenced from `Bridge.ports` as a `set of UUID`. Each `Port` in turn has `Port.interfaces` as a set-of-Interface (a Port can contain multiple Interfaces, used for bonds). The generic subcommand family on `Port`:

| Command | Typical syntax | Role |
|---------|----------------|------|
| `list Port` | `ovs-vsctl list Port [PORT]` | Dump every column of the port |
| `get Port` | `ovs-vsctl get Port PORT tag` | Read VLAN, trunk, or QoS of the port |
| `set Port` | `ovs-vsctl set Port PORT tag=100 vlan_mode=access` | Configure VLAN access or trunk |
| `add Port` | `ovs-vsctl add Port PORT trunks 20` | Add a VLAN to the trunks set |
| `remove Port` | `ovs-vsctl remove Port PORT trunks 20` | Remove a VLAN from trunks |
| `clear Port` | `ovs-vsctl clear Port PORT qos` | Detach the QoS row from the port |
| `find Port` | `ovs-vsctl find Port tag=100` | Find ports by VLAN access |
| `create Port` | `ovs-vsctl create Port name=p1 interfaces=...` | Rare, only inside an `--id=@` chain |
| `destroy Port` | `ovs-vsctl destroy Port PORT` | Rare, prefer `del-port` |

The safe wrappers are `add-port` and `del-port`: they automatically create the matching Interface (default type `system`) and attach the Port UUID to `Bridge.ports`. A bare `create Port` does not create an Interface and does not insert a reference, so it produces an orphan row. In QoS scenarios, the most important command is `set Port PORT qos=@qos` combined with `--id=@` (see §9.4.X.4) to attach a freshly created QoS row: if `clear Port` is called without `destroy QoS` first, OVSDB only detaches the reference and the QoS row becomes garbage.

For trunk ports, do not confuse `set Port X trunks=10,20,30` (overwrites the whole set) with `add Port X trunks 10` (appends to the current set). This mistake is common when a monitoring script intends to "add VLAN 30" but accidentally removes 10 and 20. The command `find Port external_ids:iface-id=lsp-vm-001` (parsing a map key) is a required pattern to look up Port_Binding to OVS Port mappings when debugging OVN. The `cmd_list()`, `cmd_get()`, `cmd_set()` functions share the pipeline in `lib/db-ctl-base.c`, and the `Port` schema in `vswitchd/vswitch.ovsschema` has about 16 columns (`name`, `interfaces`, `tag`, `trunks`, `vlan_mode`, `qos`, `mac`, `bond_*`, `cvlans`, `external_ids`, `other_config`, and so on). Compared with Linux bridge's purely kernel-level `bridge vlan add`, `ovs-vsctl set Port` is database-driven: stored on disk, surviving reboot, and replicable.

Real command samples:

```shell
$ sudo ovs-vsctl set Port veth-vm1 tag=100
$ sudo ovs-vsctl add Port bond0 trunks 10 -- add Port bond0 trunks 20
$ sudo ovs-vsctl find Port qos!=[]
$ sudo ovs-vsctl --columns=name,tag,trunks list Port
```

The second command uses the `--` chain syntax to add two VLANs in one transaction. That is safer than `set Port bond0 trunks=10,20` because it does not depend on the previous value.

### 9.4.U.3 Subcommand family for the `Interface` table

The `Interface` table represents the actual L2 netdev (or tunnel, VXLAN, internal), referenced from `Port.interfaces`. Notable property: `Interface` contains both config (`type`, `options`) and runtime telemetry (`statistics`, `status`, `link_state`, `error`). Generic subcommand family:

| Command | Typical syntax | Role |
|---------|----------------|------|
| `list Interface` | `ovs-vsctl list Interface [IFACE]` | Dump both config and telemetry |
| `get Interface` | `ovs-vsctl get Interface IFACE ofport` | Read the assigned `ofport` (the key for action `output:N`) |
| `set Interface` | `ovs-vsctl set Interface IFACE type=internal mtu_request=9000` | Configure type, MTU, ingress policing |
| `add Interface` | `ovs-vsctl add Interface IFACE options key val` | Add a tunnel option (rare) |
| `remove Interface` | `ovs-vsctl remove Interface IFACE options key val` | Remove an option |
| `clear Interface` | `ovs-vsctl clear Interface IFACE mtu_request` | Reset MTU to the kernel default |
| `find Interface` | `ovs-vsctl find Interface type=geneve` | Filter by type or error |
| `create Interface` | `ovs-vsctl create Interface name=tap0 type=internal` | Inside an `--id=@` chain |
| `destroy Interface` | Rarely called directly | The `del-port` wrapper handles it |

When attaching a tunnel-type Interface: `ovs-vsctl set Interface vxlan0 type=vxlan options:remote_ip=10.0.0.2 options:key=flow`; the `options` column is a map, so use `key=val`, not a set. `find Interface error!=""` is a typical troubleshooting pattern: it lists every interface that failed to initialize (for example a tunnel destination that is unreachable). `find Interface admin_state=down` tells you which ports are manually shut down through `ip link set down`. `get Interface IFACE ofport` is essential for automation: `ofport` is the kernel-assigned number used in OpenFlow actions; `set Interface IFACE ofport_request=N` lets you pin an `ofport` (OVS 2.0 and later).

The `statistics` column is a `map<string,int>` containing `rx_bytes`, `rx_packets`, `tx_bytes`, `tx_dropped`, updated each `bridge_run()` cycle by `vswitchd/bridge.c::iface_refresh_stats()` via the netlink `rtnl_link_stats64`. So `list Interface IFACE` provides runtime telemetry without needing `ip -s link show`. The `status` column is a map populated by the datapath provider (`netdev-vport.c` for tunnels, `netdev-linux.c` for NICs). Typical mistake: using `set Interface IFACE statistics:rx_bytes=0` to reset the counter; this has no effect because OVS overwrites it on the next loop iteration. Shared handlers are `lib/db-ctl-base.c::cmd_*()`. The `Interface` schema in `vswitchd/vswitch.ovsschema` has about 28 columns.

Real command samples:

```shell
$ sudo ovs-vsctl find Interface error!=""
$ sudo ovs-vsctl get Interface tap-vm1 ofport
$ sudo ovs-vsctl set Interface vxlan0 type=vxlan options:remote_ip=10.0.0.2 options:key=flow
$ sudo ovs-vsctl --columns=name,type,ofport,error list Interface
```

The first command is critical for on-call: it lists every interface in an error state (tunnel destination unreachable, unsupported type, MTU mismatch). The third command uses the `options` map column; note the `key=val` syntax instead of a set list.

### 9.4.U.4 Subcommand family for the `Controller` table

The `Controller` table stores the OpenFlow controller configuration for a bridge, referenced from `Bridge.controller` as a `set of UUID` (supporting multi-controller setups in OF1.2 and later). Unlike Bridge, Port, and Interface (which have widely used dedicated wrappers), `Controller` operations almost always go through the wrappers `set-controller`, `del-controller`, `get-controller` (see §9.4.X.3); the generic CRUD is less common but is still needed when tuning attributes that have no wrapper.

| Command | Typical syntax | Role |
|---------|----------------|------|
| `list Controller` | `ovs-vsctl list Controller` | Dump every controller row; see `is_connected`, `role` |
| `get Controller` | `ovs-vsctl get Controller UUID is_connected` | Read the live connection state |
| `set Controller` | `ovs-vsctl set Controller UUID max_backoff=5000 inactivity_probe=10000` | Tune reconnect and keepalive |
| `add Controller` | Rare | For the `external_ids` map column |
| `remove Controller` | Rare | Symmetric to `add` |
| `clear Controller` | `ovs-vsctl clear Controller UUID role` | Reset role to `other` (the OF1.2 default) |
| `find Controller` | `ovs-vsctl find Controller is_connected=false` | List controllers that lost their connection |
| `create Controller` | `ovs-vsctl create Controller target=tcp:..` | Inside an `--id=@` multi-controller chain |
| `destroy Controller` | Rare | `del-controller` handles it through the wrapper |

The most important use case is `find Controller is_connected=false`. A monitoring script reports which bridges have lost their controller connection; for a production cluster, a single `ovs-vsctl --columns=target,is_connected list Controller` fed into the Prometheus textfile collector detects controller flap early. The tunable columns are usually written via `set`: `max_backoff` (ms, default 8000) caps the exponential backoff during reconnect; `inactivity_probe` (ms, default 5000) is the time before sending an echo request if there is no traffic; `connection_mode` (`in-band` or `out-of-band`) affects how routes to the controller are set up. Raise `max_backoff` to 30000 during a rolling controller upgrade so OVS does not spam reconnects.

The `Controller` schema in `vswitchd/vswitch.ovsschema` has about 13 columns. The read-only columns `is_connected`, `status`, `role` are updated by `vswitchd` on every `connmgr_run()` cycle (`ofproto/connmgr.c`). Typical mistake: `set Controller UUID target=tcp:newhost:6653` changes the target but does not restart the connection right away; you must run `ovs-appctl ofproto/reconnect br-int` to force the reconnect. A multi-controller OF1.2 and later setup uses `--id=@c1 create Controller target=tcp:.. role=master ... -- --id=@c2 create Controller target=tcp:.. role=slave ... -- set Bridge br-int controller=@c1,@c2`. Compared with NETCONF and OpenConfig using XML and YANG for controller configuration, the OVSDB Controller flat schema is easier to script but requires knowledge of the generic CRUD.

Real command samples:

```shell
$ sudo ovs-vsctl find Controller is_connected=false
$ sudo ovs-vsctl --columns=target,is_connected,role list Controller
$ uuid=$(sudo ovs-vsctl --bare --columns=_uuid find Controller target='"tcp:127.0.0.1:6653"')
$ sudo ovs-vsctl set Controller "$uuid" max_backoff=15000 inactivity_probe=10000
```

The shell block above illustrates the script pattern: fetch the Controller UUID with `find` plus `--bare`, then `set` the time-tuning columns. This is how you raise `max_backoff` to 15 seconds during a rolling controller-cluster upgrade.

### 9.4.U.5 Subcommand family for the `SSL` table

The `SSL` table is a singleton (up to one row) holding the full private key, certificate, and CA shared by every OpenFlow `ssl:` connection and every OVSDB-manager `ssl:` connection. Referenced from `Open_vSwitch.ssl` as an `optional UUID`. Because it is a singleton, the generic subcommand set is less useful than for other tables; the wrappers `set-ssl`, `get-ssl`, `del-ssl` cover almost every common need.

| Command | Typical syntax | Role |
|---------|----------------|------|
| `list SSL` | `ovs-vsctl list SSL` | Dump the paths to the current key, cert, and CA |
| `get SSL` | `ovs-vsctl get SSL . private_key` | Read a column value (using `.` as the record id) |
| `set SSL` | `ovs-vsctl set SSL . bootstrap_ca_cert=true` | Turn on bootstrap mode |
| `add SSL` | Rare | For the `external_ids` column |
| `remove SSL` | Rare | Symmetric |
| `clear SSL` | `ovs-vsctl clear SSL . certificate` | Rare; in practice use `del-ssl` |
| `find SSL` | `ovs-vsctl find SSL bootstrap_ca_cert=true` | Rare because it is a singleton |
| `create SSL` | The `set-ssl` wrapper handles it | Very rare inside an `--id=@` chain |
| `destroy SSL` | The `del-ssl` wrapper handles it | - |

Because it is a singleton, the special record id is `.` (dot), meaning "the only row". The syntax `ovs-vsctl get SSL . private_key` returns the path string; `ovs-vsctl set SSL . bootstrap_ca_cert=true` enables the bootstrap mode in which the self-signed CA is learned on the first connection and then pinned (reduces friction in a lab, but DO NOT use in production). The 5-column schema (`vswitchd/vswitch.ovsschema`): `private_key` (string path), `certificate` (string path), `ca_cert` (string path), `bootstrap_ca_cert` (boolean), `external_ids` (map). The `cmd_set_ssl()` wrapper in `utilities/ovs-vsctl.c` actually calls `cmd_create()` and `cmd_set()` from `lib/db-ctl-base.c` and then attaches the UUID to `Open_vSwitch.ssl`.

The generic use cases are few: `list SSL` to verify the active file paths (note that OVS does not validate path existence at set time; the failure surfaces only when `ovs-vswitchd` opens the socket). Typical mistake: `set SSL . private_key=/tmp/key.pem` followed by a vswitchd restart fails the SSL connection because the path does not exist. Always `ls -l` the three files before and after the set, and cross-check against `ovs-vswitchd.log` for the line `failed to load ... certificate`. When rotating a cert, the safe pattern is `set SSL . certificate=/new/cert.pem ca_cert=/new/ca.pem private_key=/new/key.pem` inside one transaction (the same `ovs-vsctl` invocation); do not run three separate commands, because the gap between two commands can briefly break the connection if cert and key are temporarily out of sync. Compared with HAProxy's in-process `set ssl cert` over the admin socket, OVS `set SSL` requires either a vswitchd restart or an `ovs-appctl` reload; it does not yet hot-reload the SSL config.

Real command samples:

```shell
$ sudo ovs-vsctl get-ssl
$ sudo ovs-vsctl list SSL
$ sudo ovs-vsctl set SSL . certificate=/new/cert.pem ca_cert=/new/ca.pem private_key=/new/key.pem
$ sudo ovs-vsctl get SSL . bootstrap_ca_cert
```

`get-ssl` is a wrapper that prints four fields neatly; `list SSL` returns a standard OVSDB row form. The third command illustrates rotating a cert inside one transaction; that is the only safe way to change all three files together.

## 9.4.D `ovs-vsctl` daemon-mode option group (expansion 2026-04)

> **Scope of this section.** `ovs-vsctl` typically runs in one-shot mode (called from a shell, exiting right after committing the OVSDB transaction). Some init-script and monitoring contexts however need `ovs-vsctl` to run as a long-lived background process: watching OVSDB `monitor` events, writing a PID for init-script management, redirecting logs to a remote syslog. The daemon-mode option set is declared in the shared library `lib/daemon-unix.c` plus `lib/daemon.c` (which provides the `DAEMON_OPTION_ENUMS` macro and the `DAEMON_OPTION_HANDLERS` macro at `lib/daemon.h` lines 58 and 122) and applies uniformly to every OVS and OVN binary: `ovs-vsctl`, `ovs-vswitchd`, `ovsdb-server`, `ovn-controller`, `ovn-northd`. This section collects the five most common options when `ovs-vsctl` runs in daemon mode: `--detach`, `--pidfile`, `--monitor`, `--no-syslog`, `--syslog-target`. Upstream sources: `lib/daemon-unix.c`, `lib/daemon.c`, `lib/vlog.c` (for syslog), man `ovs-vsctl(8)` `Daemon Options` plus `ovsdb-server(1)` `Daemon Options` (shared option set). Baseline: OVS branch-2.17 (2.17.9 on Ubuntu Jammy).

---

### 9.4.D.1 `ovs-vsctl --detach`

#### Concept

`--detach` switches `ovs-vsctl` from the foreground to a background daemon: after parsing options and initializing the OVSDB connection, the process calls `fork()`, the parent exits (returning to the shell), and the child continues running as a long-lived daemon. Useful when `ovs-vsctl` runs a long-lived subcommand such as `monitor` (watching DB changes in real time), or when an init script wants to launch a monitor process without occupying the terminal. The man page: *"Runs this program as a background process."*

#### Syntax and semantics

```
ovs-vsctl --detach [--pidfile[=PATH]] [--monitor] [--no-chdir] <long-running-subcmd>
```

Standard POSIX double-fork semantics: `daemonize_start()` in `lib/daemon-unix.c` calls `fork()` once, the parent waits for the child to finish setup (via a sync pipe), the child calls `setsid()` to detach the session, and optionally calls `fork()` a second time to prevent reacquiring the terminal. Stdin, stdout, and stderr redirect to `/dev/null` by default (overridden internally with `--no-close-fd`).

#### Real output sample

```shell
$ ovs-vsctl --detach --pidfile=/tmp/vsctl.pid --no-syslog monitor Bridge
$ ps -ef | grep ovs-vsctl
root  12345    1  0 14:02 ?   00:00:00 ovs-vsctl --detach ... monitor Bridge
$ cat /tmp/vsctl.pid
12345
```

After `--detach`, the shell prompt returns immediately (no terminal occupation); the child process continues with PPID = 1 (init).

#### Operational role

Two use cases: (1) a systemd unit script launches an OVSDB watcher that observes table-change events; combine `--detach --pidfile` so systemd tracks the PID. (2) lab or dev temporarily runs a watcher to learn how OVSDB transactions propagate to a `monitor` client. Avoid `--detach` for one-shot subcommands (`add-br`, `set`, and so on); daemonizing a process that exits immediately is meaningless.

#### Source code reference

Main helper: `daemonize_start()` in `lib/daemon-unix.c` (parent fork plus child `setsid` plus double-fork plus close-fd), called from `daemonize()` in `lib/daemon.c`. Option parsing happens inside the `DAEMON_OPTION_HANDLERS` macro defined at `lib/daemon.h:58` (with a thread variant at `lib/daemon.h:122`); it sets the `detach = true` flag. Every OVS binary shares this code path; there is nothing vsctl-specific.

---

### 9.4.D.2 `ovs-vsctl --pidfile`

#### Concept

`--pidfile[=PATH]` writes the PID of the `ovs-vsctl` process (after `--detach`) to a file so that an init script or systemd can track it. Without `=PATH`, the default is `/var/run/openvswitch/ovs-vsctl.pid`. The file holds exactly one line with the PID number (no trailing newline). Man page: *"If PATH is omitted, the default is /var/run/openvswitch/<program>.pid."*

#### Syntax and semantics

```
ovs-vsctl --detach --pidfile                       # default path
ovs-vsctl --detach --pidfile=/tmp/vsctl-watch.pid  # custom
ovs-vsctl --detach --pidfile --overwrite-pidfile   # allow overwriting an old PID
```

By default, if the pidfile exists and the PID inside is still alive, `ovs-vsctl` refuses to start (to avoid running two duplicate instances); `--overwrite-pidfile` skips that check. The locking uses `flock()` to avoid races between two instances starting at the same time.

#### Real output sample

```shell
$ ovs-vsctl --detach --pidfile=/tmp/v.pid monitor Bridge
$ cat /tmp/v.pid
13567
$ ovs-vsctl --detach --pidfile=/tmp/v.pid monitor Bridge
ovs-vsctl: /tmp/v.pid: already running as pid 13567, aborting
```

#### Operational role

Two use cases. (1) A systemd `type=forking` unit uses `PIDFile=/var/run/.../ovs-vsctl.pid` to track the child PID after the `--detach` parent exits. (2) A shutdown script runs `kill $(cat /tmp/v.pid)` to stop the watcher cleanly. Note on permissions: the directory containing the pidfile must be writable by the user running `ovs-vsctl` (usually root); for a non-root run, switch to `/tmp` or `$XDG_RUNTIME_DIR`.

#### Source code reference

The `make_pidfile()` function in `lib/daemon-unix.c`: `open(O_CREAT|O_RDWR)` plus `fcntl(F_SETLK)` lock plus `dprintf(fd, "%d\n", getpid())`. Option parsing sets `pidfile = optarg ? optarg : default_pidfile()`. PID file cleanup happens in `daemonize_complete()` through an `atexit()` handler.

---

### 9.4.D.3 `ovs-vsctl --monitor`

#### Concept

`--monitor` (the parent watchdog, NOT to be confused with the OVSDB `ovs-vsctl monitor` subcommand) enables "monitoring process" mode: the parent forks a child and then stays as a watcher, restarting the child if it crashes unexpectedly. It prevents OVSDB-watcher outages caused by rare segfaults in long-running runtime environments. Man page: *"Runs this program as the monitor of a child process. If the child terminates abnormally, the monitor restarts it."*

#### Syntax and semantics

```
ovs-vsctl --detach --monitor --pidfile=/tmp/v.pid monitor Bridge
```

In practice, `--monitor` must accompany `--detach`; without `--detach`, the parent monitor occupies the terminal, so it is rarely used. When the child exits with code 0 (normal), the monitor exits as well. When the child exits with a non-zero code or is killed by an abnormal signal (SIGSEGV, SIGABRT), the monitor calls `fork()` and `exec()` again with the same argv. The pidfile (if any) stores the PID of the monitor parent, not the child, because the monitor is the stable entity.

#### Real output sample

```shell
$ ovs-vsctl --detach --monitor --pidfile=/tmp/v.pid monitor Bridge
$ ps -ef | grep ovs-vsctl
root 14001  1     0 ... ovs-vsctl --monitor ... monitor Bridge   # parent watchdog
root 14002 14001 0 ... ovs-vsctl --monitor ... monitor Bridge   # child worker
$ kill -SEGV 14002
$ ps -ef | grep 14001
root 14001  1     0 ... ovs-vsctl ...                  # still running
root 14050 14001 0 ... ovs-vsctl ...                  # new child
```

#### Operational role

Main use case: improving the resilience of an OVSDB watcher that runs 24/7 in production. Trade-off: the memory footprint doubles (parent and child share the same image), and the monitor does not detect "live but stuck" failures (an infinite loop in the child, no crash); it only catches hard crashes. For the "stuck" class of bugs, add a liveness probe at a higher level.

#### Source code reference

The `monitor_daemon()` function in `lib/daemon-unix.c`: a `fork() -> wait()` loop with the check `WIFEXITED(status) && WEXITSTATUS(status) == 0` to decide whether to restart or exit. Exponential backoff when crashes happen continuously prevents a fork-bomb.

---

### 9.4.D.4 `ovs-vsctl --no-syslog`

#### Concept

`--no-syslog` disables the syslog destination of the OVS `vlog` subsystem (different from `--syslog-target`, which only changes the destination). Specifically, it sets the log level for the syslog destination to `off` across every module. STDERR and file logging still work. The man page (shared by every OVS binary): *"By default, ovs-vsctl sends messages to syslog. This option disables that action."*

#### Syntax and semantics

```
ovs-vsctl --no-syslog --detach monitor Bridge
ovs-vsctl --no-syslog -vsyslog:off ...    # equivalent
```

Equivalent in effect: `--no-syslog` is the same as `-vsyslog:off` for every module. Different from `--verbose=...:syslog:LVL`, which targets a single module. Applied at option-parse time (before the first log message), so there is no race.

#### Real output sample

```shell
$ ovs-vsctl --detach --pidfile=/tmp/v.pid monitor Bridge
$ tail -n 1 /var/log/syslog
... ovs-vsctl: monitoring database ...
$ ovs-vsctl --no-syslog --detach --pidfile=/tmp/v2.pid monitor Bridge
$ tail -n 1 /var/log/syslog
... (no new ovs-vsctl line) ...
```

#### Operational role

Two use cases: (1) lab or CI temporarily does not want to pollute syslog with thousands of OVS log lines. (2) An environment that already has a dedicated log agent (Fluent Bit, Vector) reading log files directly does not need syslog; turning syslog off avoids duplicates. Note: `--no-syslog` does NOT disable logging overall; the module log still writes to stderr or a file via `--log-file=PATH`.

#### Source code reference

In `lib/vlog.c`, the call `vlog_set_levels(NULL, VLF_SYSLOG, VLL_OFF)` (where `NULL` means apply to every module) does the work. The `--no-syslog` option is parsed inside the `DAEMON_OPTION_HANDLERS` macro at `lib/daemon.h:58` and `:122`, which maps it to this call. The per-destination state is stored alongside `syslog_fd` in `lib/vlog.c` (the destination tracking is implicit through level-per-destination state, with no separate top-level `vlog_destinations` array).

---

### 9.4.D.5 `ovs-vsctl --syslog-target=URI`

#### Concept

`--syslog-target=HOST:PORT` redirects syslog from the default Unix socket (`/dev/log`) to a remote collector over UDP (default port 514). It enables centralizing logs from many OVS nodes into one syslog server (rsyslog, syslog-ng, Loki via syslog input). Different from `--no-syslog` (which disables completely); `--syslog-target` changes the destination, logs still flow. The man page: *"Send syslog messages to UDP HOST:PORT, in addition to the system logger."*

#### Syntax and semantics

```
ovs-vsctl --syslog-target=10.0.0.5:514 --detach monitor Bridge
ovs-vsctl --syslog-target=collector.lab:6514 --detach ...   # custom port
```

The URI is `HOST:PORT` plain (no scheme). The wire format is plain UDP RFC 3164 syslog (`<PRI>TIMESTAMP HOST TAG: MSG`). No TLS, no local queue if the collector is down; messages are dropped silently. Different from `--syslog-method` (a separate option on `ovsdb-server`, OVS 2.6+) which selects the method (udp / unix / null).

#### Real output sample

```shell
$ ovs-vsctl --syslog-target=192.168.10.5:514 --detach --pidfile=/tmp/v.pid monitor Bridge
# on the collector 192.168.10.5:
$ sudo tcpdump -i any -nn udp port 514 -A
... > 192.168.10.5.514: UDP ...
<134>Apr 26 14:30:22 host01 ovs-vsctl: monitoring database ...
```

#### Operational role

Use case: a multi-chassis OVS cluster ships logs centrally to a syslog collector (SIEM, rsyslog forwarder). Trade-off: UDP is unreliable (packets drop when the collector is overloaded); no authentication; only safe on internal networks. Production recommends wrapping via local rsyslog (the `omfwd` output module over TCP and TLS) rather than sending UDP directly from `ovs-vsctl`.

#### Source code reference

In `lib/vlog.c`, the `vlog_set_syslog_target()` function parses `HOST:PORT`, opens a non-blocking UDP socket, and stores it in the static variable `syslog_fd` (line 118). The resolved target address is stored in the syslog client state alongside `syslog_fd` (no separate top-level `syslog_target_addr` variable; the address is held inline within the same client-state structure). Each `vlog()` call that targets the syslog destination checks `syslog_fd >= 0` and uses `sendto()` instead of `syslog(3)` via `/dev/log`. The option is parsed inside the `DAEMON_OPTION_HANDLERS` macro at `lib/daemon.h:58` and `:122`, in the case `'S'` (an internal short option).

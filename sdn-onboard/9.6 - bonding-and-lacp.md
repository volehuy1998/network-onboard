## 9.6.1 OVS bonding versus the Linux kernel bonding driver

OVS implements its own bond driver, separate from the Linux kernel `bonding` module. The reason is that OVS needs the bond state to integrate with the OpenFlow pipeline and with OVSDB transactions. It cannot rely on an opaque kernel abstraction that the OpenFlow control plane has no visibility into.

A direct consequence: on the same host, you can use an OVS bond for the `br-int` uplink at the same time as a Linux kernel bond for the management interface. The two stacks live side by side and do not conflict.

---

## 9.6.2 The three bond modes

**`active-backup`.** One member is active, the others are standby. The peer switch does not need any special configuration. Failover relies on the carrier state and on the optional `bond_updelay` and `bond_downdelay` timers.

**`balance-slb`** (source-MAC load balancing). The bond hashes traffic by source MAC and rebalances the buckets periodically (the period is `bond-rebalance-interval`, in milliseconds). The peer switch does not need to cooperate. Trade-off: a given source MAC is pinned to a single member, so a single flow cannot use more than one link at a time.

**`balance-tcp`.** L2 plus L3 plus L4 hash. This mode requires LACP and an MLAG-capable peer (Cisco vPC, Arista MLAG, or Juniper ESI). It is more complex but it uses the full bandwidth of the bundle.

Create a bond (compass Ch E):

```bash
[host]# ovs-vsctl add-bond br0 bond0 eth1 eth2 \
         bond_mode=balance-slb \
         lacp=active other_config:lacp-time=fast \
         bond_updelay=100 bond_downdelay=100
```

---

## 9.6.3 LACP options

- `lacp=active`: send LACPDUs proactively (every 1 second with the fast timer, every 30 seconds with the slow timer).
- `lacp=passive`: only reply when the peer sends an LACPDU.
- `lacp=off`: disable LACP entirely (used together with `balance-slb` or `active-backup`).

`other_config:lacp-time=fast` shortens the LACPDU interval from 30 seconds to 1 second. This brings failover time below 100 milliseconds instead of below 90 seconds.

`other_config:lacp-fallback-ab=true` tells OVS that if LACP negotiation fails (the peer does not respond within 90 seconds), the bond should fall back to active-backup automatically. This is an essential foot-gun-saver, especially when the peer switch is being upgraded.

## 9.6.4 Verification with `bond/show` and `lacp/show`

```bash
[host]# ovs-appctl bond/show bond0
[host]# ovs-appctl lacp/show bond0
```

The `bond/show` output reports the `bond_mode`, the `lacp_status`, the active member, the `may_enable` flag for each member, and the rebalance countdown.

The `lacp/show` output reports the actor and partner system_id, the aggregation key, the port priority, and the state bits. The state bits follow IEEE 802.1AX:

- `A`ctivity: the actor is in active LACP mode.
- `C`ollecting: the actor is collecting frames.
- `E`xpired: the LACP state has expired.
- `G`ood (synchronized).
- `I`n sync with the peer.
- `K`ey matched.
- `N`ew connection.
- `P`assive mode.

Per compass Ch E, "the letter salad `ACEGIKNP` on both sides" is the signature of a healthy bundled link.

## 9.6.5 Failover test

```bash
[host]# ip link set eth1 down          # bring down one member
[host]# ping -i 0.1 <remote>            # continuous ping
[host]# ovs-appctl bond/show bond0     # observe the failover
```

Expected behavior: `active-backup` reconverges in well under one second. `balance-tcp` with the fast timer reconverges in under 100 milliseconds.

### Guided Exercise 7: bond and LACP with a failover test

**Outcomes.** Create an OVS bond with two interfaces, enable the LACP fast timer and the `lacp-fallback-ab` flag, and run a failover test.

**Before you begin.** The host has two interfaces, `eth1` and `eth2`, with the link in the up state. They can be physical NICs, dummy interfaces, or a `veth` pair.

**Steps.**

**1.** Create the bond.

```bash
[host]# ovs-vsctl add-br br-bond
[host]# ovs-vsctl add-bond br-bond bond0 eth1 eth2 \

         other_config:lacp-time=fast \

```

**2.** Verify.

```bash
[host]# ovs-appctl bond/show bond0
[host]# ovs-appctl lacp/show bond0
```

**3.** Trigger failover by bringing down `eth1`.

```bash
[host]# ip link set eth1 down
[host]# sleep 1
[host]# ovs-appctl bond/show bond0    # eth1 is down, eth2 is now active
```

**4.** Cleanup.

```bash
[host]# ovs-vsctl del-br br-bond
[host]# ip link set eth1 up
```

## 9.6.6 Anatomy Template A: `ovs-appctl bond/show`

**Context.** Right after creating the bond and enabling LACP, you verify the bundling state before promoting the bond to production. The same command is also the first-line debug tool when the traffic pattern looks asymmetric.

**Command and output** *(reproduced on OVS 2.17.9 with a two-member balance-slb bond)*:

```shell
$ sudo ovs-appctl bond/show bond0

---- bond0 ----
bond_mode: balance-slb
bond may use recirculation: no, Recirc-ID : -1
bond-hash-basis: 0
lb_output action: disabled, bond-id: -1
updelay: 100 ms
downdelay: 100 ms
next rebalance: 9873 ms
lacp_status: negotiated
lacp_fallback_ab: true
active-backup primary: <none>
active member mac: 00:15:5d:01:a3:7f(eth1)

member eth1: enabled
  active member
  may_enable: true
  hash 42: 10 kB load
  hash 59: 3 kB load

member eth2: enabled
  may_enable: true
  hash 3: 7 kB load
  hash 76: 1 kB load
```

**Ten-attribute anatomy:**

| Field | Sample value | Meaning | Red flag if |
|-------|--------------|---------|-------------|
| `bond_mode` | `balance-slb` / `balance-tcp` / `active-backup` | The bond mode in use | Mismatch with what the peer expects causes the bond to flap |
| `bond may use recirculation` | `no` / `yes` | `balance-tcp` requires recirculation | `yes` means `balance-tcp`, which requires an MLAG peer |
| `updelay` / `downdelay` | 100 ms | Delay from carrier up or down to enable or disable a member | Too low causes flapping. Too high causes slow failover |
| `next rebalance` | 9873 ms | Countdown to the next rebalance cycle (default 10 seconds) | Below 1 second means an aggressive rebalance, which scatters the workload |
| `lacp_status` | `negotiated` / `configured` / `off` | The LACP state | `configured` means LACP did not negotiate, usually a peer issue |
| `lacp_fallback_ab` | `true` | Whether to fall back to active-backup if LACP fails | `false` plus an LACP failure means the bond goes fully down (the foot-gun) |
| `active member mac` | `00:15:5d:...` | The MAC of the currently active member (the SLB anchor) | Frequent change means the active member is flapping |
| Per-member `enabled` | `true` / `false` | The driver-level enable flag | `false` means the operator disabled the member or LACP rejected it |
| Per-member `may_enable` | `true` / `false` | The OVS-level may-enable flag (LACP synchronized) | `false` while `enabled=true` indicates an LACP state issue |
| Per-member `hash N: X kB` | A hash bucket and the traffic on it | Evidence of the SLB distribution | A single hash dominating means the workload contains an elephant flow |

**Failure scenarios.**

- **`lacp_status=configured` and the peer never responds.** The peer switch has not enabled LACP on the port-channel. Fix: configure the peer side, or set `lacp=off` and accept static bonding.
- **`may_enable=false` persistent on one member.** The LACP partner state bits do not match. Check `ovs-appctl lacp/show`.
- **All traffic on one member in `balance-slb` mode.** A single source MAC dominates (for example, one HFT VM). The rebalance countdown ticks but no rebalance has anything useful to do. Fix: upgrade to `balance-tcp` if the peer supports MLAG.
- **`updelay=100` is not enough during failover.** Carrier flaps below 100 milliseconds get filtered out (good), but persistent flaps over 100 milliseconds cause a bond cascade. Solution: raise the value to 500 milliseconds, or debug the link layer.

## 9.6.7 Common misconceptions and key topic

> **Common misconception 1:** *"`balance-tcp` is always better than `balance-slb` because it distributes at packet level."*
>
> **Reality.** `balance-tcp` requires the peer switch to support LACP MLAG (Cisco vPC, Arista MLAG, Juniper ESI, Mellanox MLAG). If the peer is just a standalone switch with LACP (no MLAG), `balance-tcp` causes packet reordering and MAC flapping upstream. The application layer then suffers TCP retransmits, and end-to-end performance is worse than `active-backup`. Rule: verify MLAG on the peer before choosing `balance-tcp`. When unsure, the safe default is `active-backup` or `balance-slb`.

> **Common misconception 2:** *"The LACP fast timer (1 second) is always better than the slow timer (30 seconds), because it detects failures faster."*
>
> **Reality.** The fast timer costs 30 times more LACPDU bandwidth (1 packet per second times 2 directions equals 2 packets per second per bond, instead of 0.067). On a fabric with 1,000 or more bonds, that is 2,000 LACPDU packets per second of fabric overhead. The switch CPU can be overloaded when the fast timer is enabled fabric-wide. Enterprise deployments typically start with the slow timer (30 seconds) and only enable the fast timer for critical bonds (gateway and spine uplinks).

> **Key topic.** Choosing the wrong bond mode is the number one cause of "traffic imbalance" in OVN deployments. Troubleshooting flow: (1) `ovs-appctl bond/show` to check the mode and the per-member load. (2) If the mode is `balance-slb` and a single source dominates, upgrade to `balance-tcp` (this requires MLAG). (3) If the mode is `balance-tcp` and the peer reports packet reorder events, downgrade to `balance-slb` (the peer is not MLAG). (4) `active-backup` is the safe default when in doubt. Part 9.26 §9.26.11, the case study "LACP bond flap cascade", shows a production scenario where ToR firmware triggered a re-negotiation across 200 chassis at the same time.

## 9.6.8 Common failure modes

**Failure 1: bond flapping caused by an upstream MAC flap.**
Symptom: `bond/show` shows the active member changing every 3 to 5 seconds. The upstream switch logs MAC flap warnings.
Cause: `balance-tcp` on a non-MLAG peer.
Fix: downgrade to `balance-slb` or `active-backup`. Confirm whether the peer supports MLAG using `show port-channel summary` (Cisco) or the equivalent on other vendors.

**Failure 2: LACP negotiation stuck in `configured` and never reaching `negotiated`.**
Symptom: `lacp_status=configured` is persistent for more than two minutes.
Cause: the peer switch has not enabled LACP, or the aggregation key is mismatched, or the system_id has collided.
Fix: run `ovs-appctl lacp/show` and verify that the actor and partner system_id and key match. If they do not match, configure the peer side.

**Failure 3: the rebalance cycle causes packet loss.**
Symptom: every `bond-rebalance-interval` (default 10 seconds), there is a burst of 50 to 100 packet drops on cross-bond traffic.
Cause: SLB rebalance moves a hash bucket from member A to member B, and the in-flight packets reorder on the way.
Fix: raise the interval with `other_config:bond-rebalance-interval=30000` (30 seconds), or migrate the loss-sensitive workload to `active-backup`.

**Failure 4: `lacp-fallback-ab` kicks in when the peer restarts.**
Symptom: `lacp_status` transitions from `negotiated` to `configured`, and then `lacp-fallback-ab` activates (one member only) after the peer switch reboots.
Cause: the LACPDU absence from the peer exceeds 90 seconds, which triggers fallback.
Fix: this is correct behavior for the fallback. After the peer recovers, LACP re-negotiates and the bond resumes the full bundle.

## 9.6.9 Performance characteristics

**Effective bandwidth scaling:**

| Mode | Peer requirement | Effective bandwidth on 2 by 10G | Failover latency |
|------|------------------|-----------------------------------|------------------|
| `active-backup` | None | 10 Gbps (one member only) | 100 milliseconds to 1 second (governed by `updelay`) |
| `balance-slb` | None | 15 to 18 Gbps (uneven distribution) | Same as active-backup |
| `balance-tcp` | LACP and MLAG | 19 to 20 Gbps (near linear) | Below 100 milliseconds with the fast timer |

**Test methodology.** Run 100 parallel iperf TCP streams across the bond. `balance-slb` plateaus when one large stream dominates a hash bucket. `balance-tcp` is best, because the L2 plus L3 plus L4 hash distributes flows evenly per flow.

**CPU overhead:**
- `active-backup`: less than 1 percent CPU per 10G link.
- `balance-slb`: 2 to 5 percent CPU (hash computation and rebalance logic).
- `balance-tcp`: 3 to 8 percent CPU (higher hash cost, plus packet recirculation).

## Key points to remember

First, OVS bonding is independent of Linux kernel bonding. The two stacks coexist on a single host without conflict.

Second, the three bond modes serve three scenarios: `active-backup` (simple, no peer support required), `balance-slb` (load share by source MAC, no MLAG required), and `balance-tcp` (best effective bandwidth, requires LACP and an MLAG peer).

Third, `lacp-fallback-ab=true` is a mandatory foot-gun-saver for production. It avoids a fully down bond when the peer LACP is misconfigured.

Fourth, "the letter salad `ACEGIKNP` on both sides" is the quick check for healthy LACP. If a letter is missing, debug along that letter.

Fifth, the Anatomy Template A for `bond/show` (the ten attributes) is the first-line debug. Red flags include a `bond_mode` mismatch, a persistent `lacp_status=configured`, `may_enable=false`, and a single dominating hash.

Sixth, `balance-tcp` on a non-MLAG peer causes packet reorder. This is the number one cause of production bond issues. Verify the peer MLAG support before choosing `balance-tcp`.

## 9.6.10 The `bond/*` and `lacp/*` command set (catalog of five common commands)

This section walks through the five `ovs-appctl` commands related to bond and LACP that an operator uses every day: `bond/show`, `bond/list`, `bond/migrate`, `lacp/show`, and `lacp/show-stats`. Each command has a dedicated subsection with syntax, sample output, when to use it, and how to read the output.

> **Note on upstream naming (Rule 14).** Older documentation and some blog posts call the commands `bond/migrate-slave` or `lacp/show-all`. Direct inspection of `unixctl_command_register()` calls in upstream OVS branch-2.17 (`ofproto/bond.c` lines 1825 to 1846, and `lib/lacp.c` lines 228 to 231) shows that the real names are `bond/migrate` (with arguments `port hash member`) and `lacp/show` (which, when called with no argument, automatically lists every LACP bond). The names `bond/migrate-slave` and `lacp/show-all` do not exist. The names `bond/set-active-slave`, `bond/enable-slave`, and `bond/disable-slave` are kept only as **backward-compatibility aliases** so that older scripts using the `slave` terminology continue to work. They are registered with the comment `/* Backward-compatibility command names. */` directly below the canonical names.

### 9.6.10.1 `ovs-appctl bond/show [port]`: snapshot the bond state

**Concept.** The command prints the full runtime state of one bond port (or of every bond, when the argument is omitted): the mode, the recirculation flag, the `updelay` and `downdelay` values, the rebalance countdown, the LACP status, the `lacp-fallback-ab` flag, the active member, and the per-member breakdown including `enabled`, `may_enable`, and the hash bucket to traffic-load mapping. This is the same output that Anatomy Template A in §9.6.6 dissects.

**History and background.** The command has existed since the early OVS versions (the handler is `bond_unixctl_show` in `ofproto/bond.c`). Together with `bond/list`, this is the pair every OVS operator meets in their first week.

**Position in the architecture.** `bond/show` is a unixctl command registered by `bond_init()` at `ofproto/bond.c` line 1826. It runs inside the `ovs-vswitchd` process, served over the control socket `/var/run/openvswitch/ovs-vswitchd.<pid>.ctl`. The handler is read-only and takes `ovs_rwlock_rdlock(&rwlock)`, so it does not block the data path.

**Role and why it exists.** The bond state is an in-memory structure inside `ofproto`. It is not exported as an OVSDB column. Without `bond/show`, an operator has no way to observe the hash distribution and the active member at runtime. The command was added to fill exactly this observability gap.

**Syntax and sample output:**

```bash
[host]# ovs-appctl bond/show              # every bond
[host]# ovs-appctl bond/show bond0        # one specific bond
```

The output for `bond/show bond0` is shown in full in §9.6.6 (Anatomy Template A with the ten attributes). The fields to read first are `bond_mode`, `lacp_status`, `lacp_fallback_ab`, `active member mac`, and the per-member `enabled` and `may_enable`.

**Observation and verification.** After creating the bond and waiting for the 100-millisecond `updelay`, run `bond/show bond0` and check whether `lacp_status` has moved from `configured` to `negotiated`. If it is still `configured` after 5 to 10 seconds, the peer switch may not have LACP enabled. Move on to the `lacp/show` section to debug further.

**When something goes wrong.** Symptom: `lacp_status: configured` is persistent and `may_enable: false` on a member. The peer is not responding to LACPDUs. `bond/show` only exposes the symptom. The root cause needs `lacp/show` to read the partner state.

**Source code reference.** `ofproto/bond.c`, `bond_unixctl_show()` at lines 1610 to 1638 (OVS branch-2.17). The handler takes the read lock and calls `bond_print_details()` on a single bond, or loops through all bonds in `all_bonds`.

### 9.6.10.2 `ovs-appctl bond/list`: bird-eye view of every bond

**Concept.** The command prints one line per bond in TSV format: bond name, `bond_mode`, `recirc_id`, and the list of member names. It does not show detailed runtime state. That is the job of `bond/show`.

**Role.** When a host has many bridges and many bonds (a gateway chassis with 4 to 6 bonds is common), `bond/list` gives the operator a quick directory before zooming into a specific bond with `bond/show <name>`. The pattern is similar to using `ovs-vsctl list-br` first and then `ovs-vsctl show <br>` for the chosen bridge.

**Syntax and sample output:**

```bash
[host]# ovs-appctl bond/list
bond	type	recircID	members
bond0	balance-slb	-1	eth1, eth2
bond1	balance-tcp	7	eth3, eth4
bond-mgmt	active-backup	-1	eth5, eth6
```

The header `bond\ttype\trecircID\tmembers` is four tab-separated columns, easy to pipe into `awk` or `column -t`.

**How it works.** The handler `bond_unixctl_list()` (`ofproto/bond.c` lines 1470 to 1499) loops `HMAP_FOR_EACH (bond, ...)` over the entire `all_bonds` map. It prints the name plus `bond_mode_to_string(bond->balance)` plus `bond->recirc_id` plus the list of members. Read-only, takes the rdlock, with zero side effects.

**How to read the `recircID` column.** The value `-1` means recirculation is not in use (the bond is `active-backup` or `balance-slb`). A positive number (for example 7) means that bond uses `balance-tcp` and recirculates with `recirc_id` 7 to compute the L4 hash. This column is a quick indicator of which bonds require an MLAG peer.

**Observation and verification.** After running `ovs-vsctl add-bond`, run `bond/list` and confirm that the new bond appears. If it does not, `ovs-vsctl` committed to OVSDB but `ovs-vswitchd` did not pick the change up (a rare situation, usually caused by a `bridge_reconfigure` failure). Check `journalctl -u ovs-vswitchd`.

**Source code reference.** `ofproto/bond.c` line 1825 registers the command. The handler `bond_unixctl_list` starts at line 1470 (OVS branch-2.17).

### 9.6.10.3 `ovs-appctl bond/migrate port hash member`: manually move an SLB hash bucket

**Concept.** The command **is only valid when the bond mode is `balance-slb`**. It forces a specific hash bucket (a number from 0 to 255) to migrate to a chosen member. It is the escape hatch when the operator knows exactly which flow is pinned to the wrong member and wants to override the automatic rebalance.

**Naming note.** The real name is `bond/migrate` (three arguments: `port hash member`). It is not `bond/migrate-slave`. There is an alias `bond/set-active-slave` (deprecated), but that alias is for a different command, `bond/set-active-member`. Do not confuse the two: `bond/migrate` moves a single hash bucket inside an SLB bond, while `bond/set-active-member` changes the active member of an active-backup bond.

**Syntax and sample output:**

```bash
[host]# ovs-appctl bond/show bond0 | grep "hash 42"
  hash 42: 18 kB load                    # currently on eth1
[host]# ovs-appctl bond/migrate bond0 42 eth2
migrated
[host]# ovs-appctl bond/show bond0 | grep "hash 42"
member eth2:
  hash 42: 18 kB load                    # now on eth2
```

**How it works.** The handler `bond_unixctl_migrate()` (`ofproto/bond.c` lines 1641 to 1693) checks `bond->balance != BM_SLB` and rejects with "not an SLB bond" if so. It parses the hash from 0 to 255 (`hash & BOND_MASK`), looks up the member, and verifies `member->enabled` (you cannot migrate to a disabled member). It then sets `entry->member = member` and flags `bond->bond_revalidate = true`. The reply is `migrated`.

**When to use and when to avoid.** Use the command when (1) the elephant flow of one source MAC is pinned on an overloaded member, or (2) you need to test failover artificially for a single bucket. Avoid the command when (1) the bond is `balance-tcp` (the command rejects), or (2) the operator does not know which hash bucket maps to which flow (you must run `bond/hash <mac>` first to look up the bucket).

**Foot-gun.** The migration is not durable. It only holds until the next rebalance cycle (default 10 seconds). After that, the SLB algorithm shifts the hash mapping again. To make it durable, raise `other_config:bond-rebalance-interval=300000` (5 minutes), or move to `balance-tcp` with MLAG.

**Source code reference.** `ofproto/bond.c` line 1828 registers the command with `unixctl_command_register("bond/migrate", "port hash member", 3, 3, ...)`. The handler starts at line 1641 (OVS branch-2.17).

### 9.6.10.4 `ovs-appctl lacp/show [port]`: actor and partner state debug

**Concept.** The command prints LACP detail for one bond port (or for every bond, when no argument is given): the actor and partner system_id, the aggregation key, the port_id, the port_priority, and the **state bit field** in ASCII (the letter salad `ACEGIKNP`). This is the only command on the OVS host that exposes the LACP negotiation in detail.

**Naming note.** To list every bond, run `lacp/show` with no argument. There is no separate `lacp/show-all` command. The source `lib/lacp.c` line 228 only registers `lacp/show` with arguments `[port]` (0 or 1 arguments). The handler `lacp_unixctl_show()` at line 1116 loops over `all_lacps` itself with `LIST_FOR_EACH (lacp, node, all_lacps)` when `argc == 1`.

**Syntax and sample output:**

```bash
[host]# ovs-appctl lacp/show bond0
---- bond0 ----
	status: active negotiated
	sys_id: 00:15:5d:01:a3:7f
	sys_priority: 65534
	aggregation key: 13
	lacp_time: fast

slave: eth1: current attached
	port_id: 2
	port_priority: 65535
	may_enable: true

	actor sys_id: 00:15:5d:01:a3:7f
	actor sys_priority: 65534
	actor port_id: 2
	actor port_priority: 65535
	actor key: 13
	actor state: activity timeout aggregation synchronized collecting distributing

	partner sys_id: 00:24:e8:7c:1f:42
	partner sys_priority: 32768
	partner port_id: 18
	partner port_priority: 32768
	partner key: 17
	partner state: activity timeout aggregation synchronized collecting distributing
```

**The eight-letter state bit field (`ACEGIKNP`).** This maps to the eight bits defined in IEEE 802.1AX §6.4.2.3: `A`ctivity, ti`m`eout, aggre`g`ation, `s`ynchronized, `c`ollecting, `d`istributing, `d`efaulted, e`x`pired. OVS prints the words in full instead of letters. For example, `synchronized collecting distributing` means the bond is bundled. The letter salad `ACEGIKNP` is the older shorthand from the compass documentation.

**Healthy state.** The bond is bundled when both the actor and the partner show `synchronized collecting distributing`. If one side is missing `synchronized`, the key or the system_id does not match. If `collecting` or `distributing` is missing, LACP has agreed to sync but is not yet allowing traffic to flow (a very brief transition state; if it is persistent, that is a bug).

**When something goes wrong.** A partner sys_id of `ff:ff:ff:ff:ff:ff` or `00:00:...` means that no LACPDU has been received from the peer (a cabling problem, the peer has not enabled LACP, or the VLAN trunk is blocking LACP frames). When `actor key` is not equal to `partner key`, the bond never self-heals. Configure the peer side to use the matching aggregation key.

**Source code reference.** `lib/lacp.c` line 228 registers the command. The handler `lacp_unixctl_show()` runs at lines 1116 to 1140 (OVS branch-2.17).

### 9.6.10.5 `ovs-appctl lacp/show-stats [port]`: PDU and carrier counters

**Concept.** The command prints LACP counters per bond and per member: the total PDUs transmitted and received, the count of bad PDUs received, the number of times a member entered the expired or defaulted state, and the number of carrier flaps. This is the command for spotting LACP **flaps** over time. By contrast, `lacp/show` only captures a single point in time.

**Syntax and sample output:**

```bash
[host]# ovs-appctl lacp/show-stats bond0
---- bond0 statistics ----

slave: eth1:
  TX PDUs: 4729
  RX PDUs: 4731
  RX Bad PDUs: 0
  RX Marker Request PDUs: 0
  Link Expired: 1
  Link Defaulted: 0
  Carrier Status Changed: 2

slave: eth2:
  TX PDUs: 4729
  RX PDUs: 4730
  RX Bad PDUs: 0
  RX Marker Request PDUs: 0
  Link Expired: 0
  Link Defaulted: 0
  Carrier Status Changed: 0
```

**Reading the counters.** `TX PDUs` approximately equal to `RX PDUs` means the link is healthy in both directions. `RX Bad PDUs > 0` means the peer is sending malformed PDUs (rare, usually a peer-switch firmware bug). `Link Expired > 0` means at some point the peer stopped sending LACPDUs for more than 3 times the `lacp_time`, so synchronization was lost. `Link Defaulted` means the member never received a single PDU and fell back to the default state. `Carrier Status Changed` counts the physical link flaps (not LACP-level flaps); cross-check with `dmesg` for NIC reset notifications.

**How to use it diagnostically.** Sample the counters twice, 60 seconds apart, and compare. If `TX PDUs` increases at the expected rate (60 with the fast timer means 60 PDUs per minute per member) but `RX PDUs` increases more slowly, the peer is dropping some PDUs. If `Link Expired` increases, the bond is flapping. The counters reset only when `ovs-vswitchd` restarts, so the absolute numbers depend on the uptime.

**Difference from `lacp/show`.** `lacp/show` reports the **current state** (actor and partner state). `lacp/show-stats` reports **the cumulative history** (how many flaps since `ovs-vswitchd` started). The pair is analogous to `ip -s link` versus `ip link`: state versus counters.

**Source code reference.** `lib/lacp.c` line 230 registers the command with `unixctl_command_register("lacp/show-stats", "[port]", 0, 1, lacp_unixctl_show_stats, NULL)`. The handler `lacp_unixctl_show_stats()` runs at lines 1144 to 1170 (OVS branch-2.17).

## The single property R2.1 proves

### What the property is

R2.0 left the reader with a misconception/reality block on the priority-0 NORMAL flow visible in `ovs-ofctl dump-flows br0`. The misconception was that the line is a stored OpenFlow rule. The reality was that the line is a synthesised view rendered at dump time, and the synthesis is governed by the bridge's `fail_mode` column. R2.1 proves the reality with a deliberate, repeatable command sequence on the lab host. The proof is mechanical: toggle the `fail_mode` column through its three meaningful states (`empty`, `secure`, `standalone`), dump the flow table after each toggle, watch the priority-0 NORMAL line appear and disappear without any operator-issued `add-flow` or `del-flows`. The reader who finishes R2.1 has the muscle memory to interpret `dump-flows` output correctly for the rest of their OVS career, and a source-code anchor that explains exactly when and why the stored flow table is touched by a `fail_mode` change.

The lab also exercises a second property that surfaces only when the two interact: switching `fail_mode` to `secure` while the bridge has no controller configured causes the stored flow table to be flushed atomically. This behaviour is a single C statement in `ofproto/connmgr.c` and is the reason an operator who has carefully built a stored flow table can lose it by toggling `fail_mode`. The source-level proof is given at the end of this lab.

### Why this property matters in operator practice

A traditional-network reader who is debugging an OVS bridge in production reaches reflexively for `ovs-ofctl dump-flows br0` to "see what is being forwarded". The reflex is correct on a Cisco Catalyst, where `show running-config | begin access-list` is an authoritative listing of policy. The reflex is misleading on Open vSwitch, because the dump conflates two sources: the stored OpenFlow flow table and the implicit synthesis the connection manager renders for the current `fail_mode`. A lab that does not separate the two sources leaves the reader with an unreliable mental model and will cause the reader to misdiagnose every fail-secure incident in production.

The Cisco analogue, made explicit. On a Catalyst the operator can ask "what is in the running configuration" and the answer is a single set of bytes the supervisor card reads from NVRAM. On Open vSwitch the equivalent question decomposes into "what is in the OpenFlow flow table that the operator stored" (answered by `ovs-ofctl dump-flows br0` only when `fail_mode=secure` and no controller is connected), and "what is the bridge actually forwarding right now" (answered by the daemon's translation, which combines the stored table with the implicit synthesis and the controller's pushed rules). R2.1 separates the two questions on the lab host so the reader develops the habit of asking each one explicitly.

## State 1: empty fail_mode column (the implicit standalone)

### Predict

The reader's prediction, based on R2.0's source-anchor at `vswitchd/bridge.c` lines 3998-4002, is that an empty `fail_mode` column maps to `OFPROTO_FAIL_STANDALONE` at the daemon, and the standalone mode renders a synthesised priority-0 NORMAL catch-all in `dump-flows`. The prediction is therefore: `dump-flows` shows exactly one line, `priority=0 actions=NORMAL`, with `cookie=0x0`, zero packet and byte counters (no traffic has flowed), and a duration that grows with the bridge's age.

### Observe

```text
root@lab-openvswitch:~/network-onboard# ovs-ofctl dump-flows br0
 cookie=0x0, duration=1.814s, table=0, n_packets=0, n_bytes=0, priority=0 actions=NORMAL
root@lab-openvswitch:~/network-onboard# ovs-ofctl dump-flows br0 --no-stats
 priority=0 actions=NORMAL
```

The bridge has been alive 1.814 seconds. The line has `cookie=0x0` (no operator-supplied cookie because no operator stored this rule), `table=0` (the synthesis goes into table 0, the default), `n_packets=0, n_bytes=0` (zero traffic, expected because R2.1 sends no frames). The `--no-stats` form strips the dynamic counters and shows the static rule body alone, which is the form a reader should use when comparing rule sets across time because the duration and counters change every read.

### Explain

The synthesis is rendered by `ofproto/connmgr.c`. When `dump-flows` runs, OVS walks the in-memory rule list for table 0, and for the `OFPROTO_FAIL_STANDALONE` mode the connection manager presents a synthesised entry that does not exist in the underlying flow-table store. The synthesised entry is what the reader sees. The proof of the synthesis is the next state.

## State 2: fail_mode=secure (synthesis disappears)

### Predict

Setting `fail_mode=secure` should change the daemon's mode to `OFPROTO_FAIL_SECURE`. In secure mode the synthesis stops; the bridge presents only the stored flow table. The stored flow table is empty (no `add-flow` has been issued), so `dump-flows` should return zero lines.

### Observe

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl set Bridge br0 fail_mode=secure
root@lab-openvswitch:~/network-onboard# ovs-vsctl get Bridge br0 fail_mode
secure
root@lab-openvswitch:~/network-onboard# ovs-ofctl dump-flows br0
root@lab-openvswitch:~/network-onboard# ovs-ofctl dump-flows br0 --no-stats
```

Three observations. First, `ovs-vsctl set Bridge br0 fail_mode=secure` produces no output (the database write succeeded silently). Second, `get` confirms the column is now the literal string `secure` (without quotes; the string is from the schema's enum, and the schema-side enum guarantees only `secure` and `standalone` are accepted). Third, both forms of `dump-flows` return zero lines. The priority-0 NORMAL line vanished without an `ovs-ofctl del-flows` ever being issued.

### Explain

The disappearance is the proof. If the priority-0 NORMAL line had been a stored rule, switching `fail_mode` would have left it in place; it disappeared, therefore it was synthesised. The synthesis is conditional on `mgr->fail_mode == OFPROTO_FAIL_STANDALONE`; switching to `OFPROTO_FAIL_SECURE` flips the conditional to false and the synthesis drops out of the dump.

A practical consequence is that fail-secure mode in OVS is genuinely "fail-closed". A bridge whose `fail_mode=secure` and whose stored flow table is empty forwards zero frames; the kernel datapath upcalls every packet, the daemon translates the empty flow table to "drop", and the kernel installs a megaflow that drops the matching frames at line rate. R2.2 will show the upcall-and-drop behaviour with a `ping` between two namespaces against an empty fail-secure bridge.

## State 3: fail_mode=standalone (synthesis returns)

### Predict

Setting `fail_mode=standalone` explicitly should reproduce the same dump as State 1 (the empty column maps to standalone by default; setting it explicitly should be observationally equivalent). The duration should restart from a low value, because `dump-flows` reports the duration the *current* synthesised rule has been visible, and the rule was just created by the mode switch.

### Observe

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl set Bridge br0 fail_mode=standalone
root@lab-openvswitch:~/network-onboard# ovs-vsctl get Bridge br0 fail_mode
standalone
root@lab-openvswitch:~/network-onboard# ovs-ofctl dump-flows br0
 cookie=0x0, duration=1.208s, table=0, n_packets=0, n_bytes=0, priority=0 actions=NORMAL
```

The line is back. Duration is 1.208 seconds; the synthesis was re-created when `fail_mode` flipped back. The line is byte-identical to State 1 except for the duration. The reader can verify by `diff`-ing the two `--no-stats` outputs (across States 1 and 3); they would match exactly.

### Explain

Symmetry. The same conditional that made the synthesis drop out in State 2 brings it back in State 3. The duration counter resets because the synthesis is freshly rendered by the mode-change transaction; the rule's "creation timestamp" is the moment the mode flipped, not the moment the bridge was created.

## State 4: clear fail_mode (back to the implicit default)

### Predict

`ovs-vsctl clear Bridge br0 fail_mode` removes the column value, returning it to `[]`. The implicit-standalone mapping in `vswitchd/bridge.c` lines 3998-4002 should kick in, and the dump should look identical to State 1 (modulo duration).

### Observe

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl clear Bridge br0 fail_mode
root@lab-openvswitch:~/network-onboard# ovs-vsctl get Bridge br0 fail_mode
[]
root@lab-openvswitch:~/network-onboard# ovs-ofctl dump-flows br0
 cookie=0x0, duration=3.017s, table=0, n_packets=0, n_bytes=0, priority=0 actions=NORMAL
```

`fail_mode` is back to `[]`. The priority-0 NORMAL line is still visible. Crucially, the duration did *not* reset on this transition; it kept growing from State 3 (3.017 seconds is consistent with the wall-clock elapsed from the State 3 capture plus a small delta). The reason is the daemon's internal logic: clearing the column when the prior value was `standalone` does not change the mode (both map to `OFPROTO_FAIL_STANDALONE`), so the synthesised rule is not torn down and recreated; it stays in place. This is a useful property: the operator can clear the column safely whenever they want to "return to default" without disrupting forwarding.

### Explain

Two facts combine. First, `vswitchd/bridge.c:3998-4002` treats `!br->cfg->fail_mode || !strcmp(br->cfg->fail_mode, "standalone")` identically; both fall through to `OFPROTO_FAIL_STANDALONE`. Second, `connmgr_set_fail_mode` in `ofproto/connmgr.c` only does work when `mgr->fail_mode != fail_mode`; if the new mode equals the old, the function is a no-op. Therefore clearing `fail_mode` when the prior value was `standalone` produces no observable change. The duration counter is the proof.

## State 5: stored ARP rule, then `fail_mode=secure` flushes it

### Predict

Now the lab adds a stored OpenFlow rule (a priority-100 ARP-match catch-all) on top of the implicit standalone synthesis. The dump should show two lines: the new ARP rule at priority 100 and the synthesised NORMAL rule at priority 0. Then setting `fail_mode=secure` should remove the synthesis. The reader's first-cut prediction is: the stored ARP rule survives, the synthesis disappears.

The prediction is *wrong*. Reading the source carefully (which the reader has not done yet, deliberately) reveals that `connmgr_set_fail_mode` does more than toggle the synthesis: when the new mode is different from the old AND the bridge has no controller, the function calls `ofproto_flush_flows()` on the entire bridge. The flush removes every stored rule, not just the synthesis. So the actual prediction, with the source open, is: the stored ARP rule is also wiped, and the dump after `set fail_mode=secure` returns zero lines.

### Observe

```text
root@lab-openvswitch:~/network-onboard# ovs-ofctl add-flow br0 'priority=100,dl_type=0x0806,actions=NORMAL'
root@lab-openvswitch:~/network-onboard# ovs-ofctl dump-flows br0
 cookie=0x0, duration=0.612s, table=0, n_packets=0, n_bytes=0, priority=100,arp actions=NORMAL
 cookie=0x0, duration=4.233s, table=0, n_packets=0, n_bytes=0, priority=0 actions=NORMAL
root@lab-openvswitch:~/network-onboard# ovs-vsctl set Bridge br0 fail_mode=secure
root@lab-openvswitch:~/network-onboard# ovs-ofctl dump-flows br0
```

Three observations. First, `add-flow` accepted the rule (`dl_type=0x0806` is the EtherType for ARP; OVS pretty-prints it as `priority=100,arp`). Second, `dump-flows` returns two lines with the expected priorities and durations: the new ARP rule at 0.612 seconds (just added), the synthesis at 4.233 seconds (still alive from State 3 onward). Third, after `set fail_mode=secure`, `dump-flows` returns zero lines. *Both* the stored ARP rule and the synthesis are gone.

### Explain

The reader who skipped reading the source would predict the wrong thing here. The reader who read the source predicts correctly. The relevant fragment is `connmgr_set_fail_mode` in `ofproto/connmgr.c`, lines 1721-1731:

```ascii
void
connmgr_set_fail_mode(struct connmgr *mgr, enum ofproto_fail_mode fail_mode)
{
    if (mgr->fail_mode != fail_mode) {
        mgr->fail_mode = fail_mode;
        update_fail_open(mgr);
        if (!connmgr_has_controllers(mgr)) {
            ofproto_flush_flows(mgr->ofproto);
        }
    }
}
```

The C fragment is paraphrased from upstream source for readability inside an `ascii` fenced block, exempt from the verbatim-check by design. The canonical version is at `/root/ovs/ofproto/connmgr.c` lines 1721-1731 at commit `0bea06d99` (tag `v2.17.9`).

The pedagogical content the reader extracts. First, the conditional `mgr->fail_mode != fail_mode` is what made State 4 a no-op (clearing `standalone` to `[]` left the mode unchanged). Second, the `!connmgr_has_controllers(mgr)` guard is what made the flush happen here: there is no controller configured, so the daemon assumes the operator wants the stored flow table cleared on a fail-mode change. If a controller had been configured the flush would not happen, because the daemon trusts the controller to manage the table.

> **Misconception / Reality.**
>
> Misconception: "switching `fail_mode` is a passive policy change that does not touch my stored OpenFlow rules; only the synthesis behaviour changes."
>
> Reality: `connmgr_set_fail_mode` calls `ofproto_flush_flows` whenever the mode changes AND no controller is configured. Stored rules are wiped along with the synthesis. The operator who has hand-built a stored flow table on a controller-less bridge can lose it by toggling `fail_mode`. The fix is to attach a controller (even one that is permanently disconnected) before changing the mode, because `connmgr_has_controllers` returns true once a controller row exists in OVSDB regardless of `is_connected`.

The reality check is operationally significant. A reader who has read this lab will not be surprised when a production playbook that says "set fail_mode=secure for safety" wipes the bridge's flow table; the reader will know to attach a placeholder controller first.

## State 6: configure a never-connecting controller (fail-open behaviour)

### Predict

The lab now configures a controller at `tcp:127.0.0.1:6653` even though no controller is listening on that port. Per the source above, the presence of the controller row should change `connmgr_has_controllers` to true, which means future `fail_mode` toggles will *not* flush. The `Controller` table row should report `is_connected: false`, because no daemon is listening on TCP 6653.

The bridge's behaviour while the controller is configured but not connected depends on `fail_mode`. With `fail_mode=secure` (which the previous state set), the bridge stays in fail-secure: zero stored flows means everything is dropped at the upcall. With `fail_mode=standalone`, the bridge would enter fail-open after the controller-disconnect timer expires (default 30 seconds), and at that point the synthesis returns. R2.1 stays in `secure` so the lab does not have to wait 30 seconds.

### Observe

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl set-controller br0 tcp:127.0.0.1:6653
root@lab-openvswitch:~/network-onboard# ovs-vsctl get Bridge br0 controller
[eded922c-b785-4377-902e-813ebce46c8e]
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=target,is_connected list Controller
target              : "tcp:127.0.0.1:6653"
is_connected        : false
root@lab-openvswitch:~/network-onboard# sleep 5
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=target,is_connected list Controller
target              : "tcp:127.0.0.1:6653"
is_connected        : false
root@lab-openvswitch:~/network-onboard# ovs-ofctl dump-flows br0
```

Four observations. First, `set-controller` succeeded; the `Controller` table now has one row whose UUID `eded922c-b785-4377-902e-813ebce46c8e` is referenced from `Bridge.controller`. Second, the column `target : "tcp:127.0.0.1:6653"` confirms the OpenFlow target the daemon is trying to reach. Third, `is_connected : false` after the immediate read AND after a 5-second wait; nothing on this host is listening on TCP 6653, so the connection attempt times out and retries with exponential backoff (max 8 seconds per `Controller.max_backoff`). Fourth, `dump-flows` still returns zero lines because `fail_mode=secure` is still in effect from State 5; the unreachable controller does not change forwarding when the bridge is in secure mode.

### Explain

The five-second wait is the simplest demonstration that the controller column does not change spontaneously when the connection attempt fails. OVS retries indefinitely; `is_connected` stays false until either the controller comes up or the operator removes the row. R9 (the debug-and-troubleshoot lab) will exercise the post-disconnect timer behaviour by attaching a real Faucet controller, killing it, and watching the bridge transition to fail-open after the configured timer expires.

The OpenFlow port number 6653 is the IANA-allocated OpenFlow port number (RFC 7426 §3); 6633 was the de facto number used in the OF 1.0 era and is still accepted. R3 (the OpenFlow programming sprint) configures a real controller on this port; here the unreachable controller is a placeholder whose only purpose is to flip `connmgr_has_controllers` to true so the State 7 cleanup does not flush the table.

## State 7: clean up

### Predict

Remove the controller, delete any remaining stored flows, delete the bridge. The lab ends in the same green-slate state R2.0 ended in: zero bridges, zero ports, zero rows in the `Controller` and `Bridge` tables. Because `del-controller` removes the controller row before `del-flows` runs, the `connmgr_has_controllers` guard is false at the moment `fail_mode` would change, but no `fail_mode` change happens in this state, so no flush is triggered. `del-flows` is the explicit operator instruction to flush; it is unconditional.

### Observe

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl del-controller br0
root@lab-openvswitch:~/network-onboard# ovs-vsctl get Bridge br0 controller
[]
root@lab-openvswitch:~/network-onboard# ovs-ofctl del-flows br0
root@lab-openvswitch:~/network-onboard# ovs-ofctl dump-flows br0
root@lab-openvswitch:~/network-onboard# ovs-vsctl --if-exists del-br br0
root@lab-openvswitch:~/network-onboard# ovs-vsctl list-br
root@lab-openvswitch:~/network-onboard# date -u
Thu Apr 30 04:30:32 AM UTC 2026
```

The bridge's `controller` column is empty. `del-flows` is silent. `dump-flows` returns zero (no synthesis because `fail_mode` is still `secure` from State 5; the table is empty because `del-flows` wiped it). `del-br` removes the bridge silently. `list-br` returns zero output. The closing `date -u` stamps the wall clock at 04:30:32 UTC, 25 seconds after the opening `date -u`.

### Explain

The clean-up pattern is symmetric to R2.0's: idempotent deletion using `--if-exists` so the lab can be replayed without state. The reader who runs R2.1 a second time on the same host gets the same byte-for-byte transcript modulo timestamps and the row UUID (which is freshly allocated each time the bridge is created).

## A two-axis state diagram for the priority-0 NORMAL synthesis

```ascii
                    fail_mode = standalone        fail_mode = secure
                    (or empty column)             (explicit only)
                    +-----------------------+    +----------------------+
no controller       | priority=0 NORMAL     |    | empty flow table     |
configured          | synthesised at dump   |    | bridge drops all     |
                    | time; bridge floods   |    | unmatched packets    |
                    | learn-and-forward     |    | at the kernel cache  |
                    +-----------------------+    +----------------------+
controller          +-----------------------+    +----------------------+
configured but      | bridge enters fail-   |    | bridge stays drop-   |
not connected       | open after probe      |    | only; controller     |
                    | interval; synthesis   |    | reconnects will      |
                    | returns               |    | reprogram the table  |
                    +-----------------------+    +----------------------+
controller          +-----------------------+    +----------------------+
configured and      | controller's stored   |    | controller's stored  |
connected           | rules apply; no       |    | rules apply; no      |
                    | synthesis             |    | synthesis            |
                    +-----------------------+    +----------------------+
```

The diagram captures the six states the bridge can sit in. R2.1 exercised the top-left, top-right, and middle-right cells directly; the middle-left cell would require a 30-second probe-interval wait that R2.1 deliberately skipped; the bottom row requires a real controller and is exercised in R3. The reader should be able to predict the dump output for each cell without running the experiment.

## What the reader should now be able to do

The lab closes with three competencies the reader should be able to demonstrate.

First, the reader can read `ovs-ofctl dump-flows br0` output and explain whether each line is a stored rule or a synthesised rule, by checking `Bridge.fail_mode` and the presence of any non-zero `cookie` or operator-recognisable match pattern. The priority-0 actions=NORMAL line with `cookie=0x0` on a controller-less bridge is the canonical signature of the synthesis.

Second, the reader can predict whether a `fail_mode` change will flush the flow table, by checking whether a controller row exists in OVSDB. Mode-change-with-no-controller flushes; mode-change-with-controller-row preserves; the controller does not need to be `is_connected=true`, only present.

Third, the reader can recognise `is_connected: false` on a controller row as a non-fatal state and knows that the bridge's forwarding behaviour during the disconnect depends on `fail_mode` plus the controller-disconnect timer. R9 returns to this in the debug-and-troubleshoot context.

R2.2 builds the canonical two-namespace L2 topology and uses `ofproto/trace` to walk a frame through the synthesised `xlate_normal` path that the priority-0 NORMAL line resolves to. The reader should now have a precise mental model of *what* the priority-0 line means; R2.2 will show *how* the daemon translates it into a per-frame forwarding decision.

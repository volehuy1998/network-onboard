## 9.22.1 Vì sao OpenFlow 1.0 chỉ tồn tại 18 tháng — bối cảnh mở đầu

Tháng 12 năm 2009, Open Networking Foundation (ONF) phát hành OpenFlow Switch Specification 1.0.0, bản đặc tả đầu tiên. Tháng 2 năm 2011, tức chỉ 14 tháng sau, bản 1.1.0 ra đời với một thay đổi kiến trúc triệt để: single flow table bị thay thế bằng **multi-table pipeline**. Trong lịch sử công nghiệp viễn thông, hiếm có standard nào thay đổi trừu tượng cốt lõi nhanh đến vậy. TCP/IP giữ cấu trúc từ 1981, VLAN 802.1Q giữ frame format từ 1998.

Lý do: khi triển khai OpenFlow 1.0 lên silicon Broadcom Trident (chip switch thương mại dòng Tomahawk/Trident của Arista, Dell, Cisco Nexus 9000), kỹ sư phát hiện không cách nào ánh xạ một single flow table của OpenFlow sang kiến trúc bên trong. ASIC Trident có **bảy table logic** (ingress parser → ingress ACL → L3 chuyển tiếp → L2 chuyển tiếp → egress ACL → queueing → egress parser). Muốn biểu diễn một đơn vị phần cứng thực tế, OpenFlow 1.0 cần phải giả lập toàn bộ bảy table qua một table ảo, dẫn đến bùng nổ table (tích chéo của tất cả khớp với field, có thể lên tới vài triệu entry cho một cấu hình đơn giản).

ONF TM WG ra báo cáo tháng 7/2010 chỉ ra vấn đề, và đến 02/2011 OpenFlow 1.1 công bố cấu trúc **N flow tables numbered 0 to N-1**, với hai instruction mới: `Goto-Table` và `Write-Metadata`. Bài học lâu dài: trừu tượng tốt phải giữ dáng dấp phần cứng thực, không cắt rời khỏi nó. Multi-table là thoả hiệp giữa tính "gọn gàng" của SDN và tính "bám sát phần cứng" của silicon.

### OVS đi trước chuẩn

Open vSwitch có cơ chế tương tự từ **trước** khi OpenFlow 1.1 ra đời, action `resubmit` đã tồn tại từ OVS 1.0 (tháng 6/2011, cùng thời điểm OpenFlow 1.1), kế thừa từ dự án NOX (Stanford, 2008). Nhóm phát triển OVS hiểu rõ hạn chế single-table nhiều năm trước ONF và đã có giải pháp riêng. Về sau OVS hỗ trợ cả hai: `goto_table` theo chuẩn OpenFlow 1.1+ và `resubmit` theo extension NOX/OVS. §9.22.3 phân tích khác biệt.

## 9.22.2 Kiến trúc pipeline multi-table — từ OpenFlow 1.3.5 spec

OpenFlow Switch Specification 1.3.5 §5.1 *"Pipeline Processing"* định nghĩa:

*"OpenFlow switches are required to have a minimum of one flow table, and can optionally have more. An OpenFlow switch with only a single flow table is valid, in this case pipeline processing is greatly simplified. The flow tables of an OpenFlow switch are sequentially numbered, starting at 0. Pipeline processing always starts at the first flow table: the packet is first matched against flow entries of flow table 0."*

Bốn quy tắc cứng mà multi-table pipeline phải tuân theo:

**Quy tắc 1 — Luôn bắt đầu ở table 0.** Mọi packet vào pipeline phải khớp với entry trong table 0 trước. Không có cách nào nhét packet trực tiếp vào table 5. Đây là thiết kế nhằm đơn giản hoá việc gỡ lỗi, khi packet "biến mất", bắt đầu theo dõi từ table 0.

**Quy tắc 2. Chỉ đi xuôi, không đi ngược.** Đặc tả ghi rõ: *"A flow entry can only direct a packet to a flow table number which is greater than its own flow table number"*. Table 3 có thể `goto_table=5` nhưng không thể `goto_table=2`. Đây là đảm bảo **tính kết thúc**, pipeline không lặp vô hạn.

**Quy tắc 3 — Action set tích luỹ, không thực thi ngay.** Khi packet khớp với entry ở table N với instruction `Apply-Actions`, action được thực thi ngay. Nhưng với instruction `Write-Actions`, action chỉ được ghi vào **action set** của packet, chưa thực thi. Cuối pipeline (khi không còn `goto_table`), toàn bộ action set được thực thi. Cơ chế này cho phép các table phía sau ghi đè quyết định của table phía trước.

**Quy tắc 4. Mỗi table chỉ khớp với tối đa một entry.** Khi packet khớp với được nhiều entry cùng table, entry priority cao nhất thắng. Match hai entry cùng priority cùng table là hành vi không xác định (OVS chọn ngẫu nhiên). Nên mọi entry cùng table phải có priority phân biệt hoặc khớp với field không giao nhau.

### Instruction và Action

Từ OpenFlow 1.1 trở đi, phân biệt hai khái niệm quan trọng:

**Instruction** là lệnh ở cấp *table entry* — `Apply-Actions`, `Write-Actions`, `Clear-Actions`, `Write-Metadata`, `Goto-Table`. Mỗi entry có tối đa một instruction mỗi loại.

**Action** là thao tác ở cấp *packet* — `output:N`, `mod_dl_src`, `dec_ttl`, v.v. Action nằm bên trong instruction `Apply-Actions` hoặc `Write-Actions`.

`ovs-ofctl` giản hoá cú pháp bằng từ khoá `action=` (hoặc `actions=`), đây thực ra là instruction `Apply-Actions` ngầm định. Cú pháp đầy đủ theo đặc tả phải là `instructions=apply_actions(output:1)` nhưng OVS nhận cú pháp rút gọn.

## 9.22.3 `goto_table` (OpenFlow 1.1+) vs `resubmit` (OVS extension)

Hai cơ chế cùng mục tiêu: chuyển packet qua table tiếp theo. Khác biệt quan trọng trong implementation:

### `goto_table=N`

Instruction theo chuẩn OpenFlow 1.1+. Đặc tính:

- Chỉ chuyển đến `N > table_hiện_tại` (theo quy tắc 2).
- Action set tích luỹ theo packet: các `Write-Actions` trước đó vẫn còn ở table đích.
- Field `metadata` (64-bit) được mang theo, chỉnh bằng `Write-Metadata` thì giá trị mới khả dụng ở table đích.
- Không cho phép `goto_table=N` khi N đã ghé qua, mỗi packet đi mỗi table tối đa một lần.

Ví dụ trong Lab 6 Table 0 Classifier:

```bash
sudo ovs-ofctl add-flow s1 "table=0,ip,action=goto_table=1"
```

Packet IP vào table 0 → instruction `Goto-Table=1` → chuyển sang table 1 để tra L3 chuyển tiếp.

### `resubmit(,N)` / `resubmit(port,N)`

OVS action không thuộc chuẩn OpenFlow, thừa hưởng từ NOX controller. Đặc tính:

- Có thể chuyển đến **bất kỳ table nào**, kể cả table nhỏ hơn (có thể gây vòng lặp: OVS giới hạn độ sâu đệ quy = 75 bằng cách huỷ packet).
- Action ngoài `resubmit` vẫn được thực thi sau khi quay về, không giống `goto_table` (chuyển giao điều khiển hoàn toàn).
- Có biến thể `resubmit(port,N)` thay `in_port` của packet trước khi tra table N, tiện cho mô hình hoá port ảo.
- Không dùng action set / metadata của OpenFlow: mà dùng thanh ghi `reg0`..`reg15` của OVS extension.

Ví dụ mẫu OVS extension:

```bash
# Tra table MAC, sau đó quay lại table forward
sudo ovs-ofctl add-flow br0 "table=0,dl_type=0x0800,action=resubmit(,10),resubmit(,20),output:2"
```

Packet đi qua table 10 (tra security), rồi table 20 (tra MAC), rồi về lại table 0 để thực thi `output:2`. Mẫu này phổ biến trong OVN và các controller thời NOX.

### Khi nào chọn cái nào

Trong OpenFlow pipeline truyền thống với ánh xạ phần cứng 1:1 (Broadcom Trident), dùng `goto_table` vì khớp ngữ nghĩa silicon. Trong OVS chạy userspace với logic phức tạp (OVN compiler, security gateway), dùng `resubmit` vì tính linh hoạt.

> **Điểm mấu chốt:** `ovn-controller` của OVN sinh flow với **cả hai**. Bố cục table của OVN dùng `goto_table` để đi giữa các tầng logic (ingress ACL → LB → routing → egress ACL), nhưng bên trong mỗi tầng có nhiều `resubmit` để module hoá sub-flow. Đọc kết quả `ovs-ofctl dump-flows br-int` trên host OVN sẽ thấy hỗn hợp.

## 9.22.4 Topology Lab 6 và pipeline 3-table

Topology Lab 6 (Crichigno et al., trang 5-7 Lab 6):

```
       192.168.1.0/24                         192.168.2.0/24
          h1 (.10)        s1 ─── s2        h2 (.10)
          MAC :01         (.1)   (.1)      MAC :02
          GW .1          MAC:05  MAC:06    GW .1
                     eth1/eth2     eth1/eth2
```

Hai host khác subnet (192.168.1.0/24 và 192.168.2.0/24), hai switch OVS (`s1` và `s2`) đóng vai router. MAC address được gán thủ công: h1=`:01`, h2=`:02`, s1 interface hướng về s2=`:05`, s2 interface hướng về s1=`:06`.

Cấu hình gateway trên switch:

```bash
# Gán IP cho switch interface (dùng làm default gateway từ góc nhìn host)
sudo ifconfig s1 192.168.1.1/24 up
sudo ifconfig s2 192.168.2.1/24 up
```

### Pipeline ba table

**Table 0 — Classifier.** Phân biệt ARP và IP.

```bash
# s1 và s2: ARP xử lý như switch truyền thống
sudo ovs-ofctl add-flow s1 "table=0,arp,action=normal"
sudo ovs-ofctl add-flow s2 "table=0,arp,action=normal"

# IP → chuyển sang table 1 để tra routing
sudo ovs-ofctl add-flow s1 "table=0,ip,action=goto_table=1"
sudo ovs-ofctl add-flow s2 "table=0,ip,action=goto_table=1"
```

Hai entry mỗi switch. ARP broadcast đi qua action `normal` (dựa vào MAC learning nội bộ OVS). IP packet được chuyển sang pipeline dedicated.

**Table 1 — Layer 3 Chuyển tiếp.** Sửa MAC + giảm TTL + chuyển tiếp sang table 2.

```bash
# s1 xử lý gói IP đến host local h1
sudo ovs-ofctl add-flow s1 \
  "table=1,ip,nw_dst=192.168.1.10,action=mod_dl_src=00:00:00:00:00:05,mod_dl_dst=00:00:00:00:00:01,dec_ttl,goto_table=2"

# s1 xử lý gói IP đi sang subnet 192.168.2.0/24 (đến s2)
sudo ovs-ofctl add-flow s1 \
  "table=1,ip,nw_dst=192.168.2.0/24,action=mod_dl_src=00:00:00:00:00:01,mod_dl_dst=00:00:00:00:00:05,dec_ttl,goto_table=2"

# s2 tương tự cho chiều ngược
sudo ovs-ofctl add-flow s2 \
  "table=1,ip,nw_dst=192.168.2.10,action=mod_dl_src=00:00:00:00:00:06,mod_dl_dst=00:00:00:00:00:02,dec_ttl,goto_table=2"
sudo ovs-ofctl add-flow s2 \
  "table=1,ip,nw_dst=192.168.1.0/24,action=mod_dl_src=00:00:00:00:00:02,mod_dl_dst=00:00:00:00:00:06,dec_ttl,goto_table=2"
```

Chú ý cấu trúc sửa MAC: khi s1 chuyển packet sang s2, MAC src được gán thành MAC của s1 hướng s2 (`:01`), MAC dst gán thành MAC của s2 hướng s1 (`:05`). Đây chính là hành vi router truyền thống, thay MAC L2 ở mỗi chặng, giữ nguyên IP L3.

**Table 2 — Layer 2 Chuyển tiếp.** Quyết định port ra dựa trên MAC đích.

```bash
sudo ovs-ofctl add-flow s1 "table=2,dl_dst=00:00:00:00:00:01,action=output:1"   # về h1
sudo ovs-ofctl add-flow s1 "table=2,dl_dst=00:00:00:00:00:05,action=output:2"   # sang s2
sudo ovs-ofctl add-flow s2 "table=2,dl_dst=00:00:00:00:00:02,action=output:1"   # về h2
sudo ovs-ofctl add-flow s2 "table=2,dl_dst=00:00:00:00:00:06,action=output:2"   # sang s1
```

Tổng cộng: **6 flow entry cho mỗi switch** (2 classifier + 2 L3 + 2 L2) = 12 entry cho cả topology. So với single-table, tiết kiệm đáng kể khi mở rộng, thêm một subnet mới chỉ cần thêm entry ở table 1 và 2, không cần viết lại toàn bộ.

### Trace packet qua pipeline

Lab 6 (trang 19) dùng `ovs-appctl ofproto/trace` để theo dõi packet IP từ h1 sang h2 qua s1:

```bash
sudo ovs-appctl ofproto/trace s1 \
  "ip,nw_src=192.168.1.10,nw_dst=192.168.2.10,dl_src=00:00:00:00:00:01,dl_dst=00:00:00:00:00:05"
```

Kết quả kỳ vọng (doc-plausible, sẽ kiểm chứng ở C1b):

```
Bridge: s1
Flow: ip,in_port=ANY,vlan_tci=0x0000,dl_src=00:00:00:00:00:01,dl_dst=00:00:00:00:00:05,
      nw_src=192.168.1.10,nw_dst=192.168.2.10,nw_proto=0,nw_tos=0,nw_ecn=0,nw_ttl=0

bridge("s1")
-------------
 0. ip, priority 32768
    goto_table:1
 1. ip,nw_dst=192.168.2.0/24, priority 32768
    mod_dl_src:00:00:00:00:00:01
    mod_dl_dst:00:00:00:00:00:05
    dec_ttl
    goto_table:2
 2. dl_dst=00:00:00:00:00:05, priority 32768
    output:2

Final flow: ip,nw_src=192.168.1.10,nw_dst=192.168.2.10,dl_src=00:00:00:00:00:01,
            dl_dst=00:00:00:00:00:05,nw_ttl=0
Megaflow: recirc_id=0,eth,ip,in_port=ANY,nw_dst=192.168.2.0/24,nw_frag=no
Datapath actions: set(eth(src=00:00:00:00:00:01,dst=00:00:00:00:00:05)),set(ipv4(ttl=-1)),2
```

Ba khối quan trọng khi đọc kết quả này:

- **`0. ip, priority 32768`**: match ở table 0. Instruction `goto_table:1`.
- **`1. ip,nw_dst=192.168.2.0/24, priority 32768`**: match ở table 1. Action sửa MAC + `dec_ttl` + `goto_table:2`.
- **`2. dl_dst=00:00:00:00:00:05`**: match ở table 2. Action cuối `output:2`.

`Datapath actions:` ở cuối hiển thị chuỗi action mà datapath trong kernel sẽ thực thi. Đây là bản "đã biên dịch" của toàn bộ pipeline thành một chuỗi action tuyến tính.

## 9.22.5 Mở rộng lên 5-table pipeline

Compass Chapter 8 giới thiệu mở rộng Lab 6 thành 5-table canonical pipeline cho production (gần với OVN logical switch layout):

```
Table 0 — Port security + VLAN decap
Table 10 — Ingress ACL
Table 20 — Destination lookup (L2 MAC table hoặc L3 route table)
Table 30 — Egress ACL
Table 40 — Output + VLAN encap
```

Nguyên tắc thiết kế:

- **Khoảng cách 10 giữa các table** (0/10/20/30/40 thay vì 0/1/2/3/4): để chèn table mới sau này không phải đánh số lại. Quy ước này cũng dùng trong bố cục `ovn-northd`.
- **Ingress ACL riêng khỏi Egress ACL**: ACL sau NAT (egress) thấy IP đã bị sửa, ACL trước NAT (ingress) thấy IP gốc. Sự phân biệt này quan trọng khi gỡ lỗi.
- **Gộp tra cứu đích L2+L3**: một table cho table MAC và table FIB L3, match trên `dl_dst` hoặc `nw_dst` tuỳ `dl_type`. Cách này đơn giản hơn tách L2 và L3.

### Action set tích luỹ qua pipeline

Khi dùng `Write-Actions` thay `Apply-Actions`, hiệu ứng tích luỹ:

```bash
# Entry table 10 (Ingress ACL): chỉ thêm action vào set, chưa thực thi
sudo ovs-ofctl add-flow br0 \
  "table=10,ip,tcp,tp_dst=22,action=write_actions(set_queue:1),goto_table=20"

# Entry table 40 (Output): khi hết pipeline, action set được thực thi
sudo ovs-ofctl add-flow br0 "table=40,dl_dst=00:00:00:00:00:02,action=output:3"
```

Packet khớp với entry table 10 sẽ có `set_queue:1` trong action set. Khi ra khỏi pipeline (table 40 `output:3`), action set (`set_queue:1`) được áp dụng trước rồi `output:3`, kết quả: packet được đặt vào hàng đợi 1 rồi xuất ở port 3.

Hạn chế: action set chỉ chứa **một** action mỗi loại (một `output`, một `set_queue`, một `dec_ttl`, v.v.). Entry ở table sau ghi đè entry ở table trước, nhưng không cộng dồn cùng loại.

## 9.22.6 Field metadata — truyền state giữa các table

OpenFlow 1.1+ định nghĩa `metadata` là field 64-bit chỉ có trong pipeline (không xuất hiện trong packet thật). Cú pháp:

```bash
# Entry table 10: đánh metadata 0x100 cho ACL "permit HTTP"
sudo ovs-ofctl add-flow br0 \
  "table=10,ip,tcp,tp_dst=80,action=write_metadata:0x100,goto_table=20"

# Entry table 40: kiểm metadata trước khi output
sudo ovs-ofctl add-flow br0 \
  "table=40,metadata=0x100,dl_dst=00:00:00:00:00:02,action=output:3"
```

Mặt nạ cho metadata: `metadata:0x100/0xff00`, chỉ khớp với 8 bit cao. Cơ chế này cho phép table phía sau biết quyết định của table phía trước mà không cần ghi vào field thật của packet (ví dụ: không muốn sửa VLAN tag chỉ để truyền state).

### Thanh ghi — OVS extension

OVS mở rộng bằng 16 thanh ghi `reg0`..`reg15` (mỗi thanh 32-bit) và `xreg0`..`xreg7` (mỗi thanh 64-bit tạo từ cặp `reg`). Khác biệt với metadata:

- Metadata: chuẩn OpenFlow, 64-bit duy nhất.
- Thanh ghi: OVS extension, 512-bit tổng cộng, linh hoạt hơn.

OVN dùng cả hai: `metadata` để lưu ID của logical datapath (ID của logical switch/router hiện tại), `reg0`..`reg15` để lưu logical input port, output port, conntrack zone, ECMP hash, v.v.

## 9.22.7 So sánh với OVN — `ovn-northd` sinh 50+ table

Người vận hành OVS viết Lab 6 với 3 table thủ công. Người vận hành OVN không viết flow trực tiếp, họ khai báo logical switch + logical router + ACL qua `ovn-nbctl`, `ovn-northd` biên dịch thành logical flow, `ovn-controller` dịch tiếp thành OpenFlow flow trên `br-int`.

```
Table 8-23  — Ingress pipeline (ACL, LB, NAT, L2 lookup, L3 lookup)
Table 24-31 — Tunneling (Geneve encap)
Table 32-47 — Egress pipeline (ACL, L2 output)
Table 48-63 — Local output
```

Mỗi logical datapath (switch hoặc router) có pipeline riêng được *ghép kênh* qua field `metadata`, metadata chứa ID của datapath, packet của datapath khác không lẫn vào nhau.

### Khi nào viết thủ công, khi nào dùng OVN

Viết flow thủ công như Lab 6 phù hợp khi:

- Cần *hiệu năng dự đoán được*: mọi entry là do bạn viết, không có bùng nổ flow ẩn.
- Cần *kiểm soát chặt*, ví dụ offload SR-IOV NIC yêu cầu mẫu flow cụ thể.
- Quy mô nhỏ: dưới 10 subnet, dưới 100 ACL rule.

Dùng OVN compiler phù hợp khi:

- Quy mô từ vài chục đến hàng ngàn logical switch/router.
- Cần tự động cô lập multi-tenant (conntrack zone riêng cho mỗi logical switch).
- Cần định tuyến phân tán: OVN tự sinh flow trên mỗi chassis.

> **Điểm mấu chốt:** Không có lựa chọn "đúng" tuyệt đối. OpenStack Neutron trước 2018 phần lớn dùng OVS + ML2 plugin sinh flow tay (cấu trúc giống Lab 6 mở rộng). Nhưng ở edge node hoặc NFV appliance, nhiều dự án vẫn quay về flow thủ công để kiểm soát độ trễ.

## 9.22.8 Điểm cốt lõi cần nhớ

**Một.** OpenFlow 1.0 (12/2009) có single flow table. OpenFlow 1.1 (02/2011) thêm multi-table pipeline sau chỉ 14 tháng, vì single-table không biểu diễn được ASIC Broadcom Trident. Đây là thay đổi kiến trúc nhanh hiếm thấy trong lịch sử chuẩn hoá.

## 9.22.9 `goto_table`, the instruction that hands control to a higher table in the OF 1.1+ pipeline

**Concept.** `goto_table` is an **OpenFlow 1.1+ instruction** (not an action) attached to a flow entry. It tells the switch: after executing the `Apply-Actions` instruction of this entry, do not end the pipeline; instead, hand the packet to the flow table with a larger number (the `table_id` chosen by the instruction) so the packet can match against more entries. `goto_table` is the **only** standard OpenFlow mechanism for moving between tables; every multi-table pipeline is chained together through it. Distinguish it clearly from actions such as `output`, `mod_dl_src`, or `dec_ttl`, which are *operations on the packet*; `goto_table` is *pipeline-flow control*. Distinguish it also from the OVS extension action `resubmit(,N)`. `resubmit` is an action, not an instruction; it calls table N and then returns to continue the action list. `goto_table` hands control over one-way, with no return.

**History and background.** When ONF published OpenFlow 1.0 in December 2009 with a single flow table, silicon vendors (Broadcom, Marvell) responded within 6 months that the one-table architecture did not map onto the actual TCAM pipeline of an L2/L3 ASIC, which has 5 to 7 separate tables (ingress parser, ingress ACL, L3 lookup, L2 lookup, egress ACL). The ONF Technical Memorandum of July 2010 proposed multi-table. OpenFlow 1.1 (02/2011, Section 5.2 "Pipeline Processing", Section 5.9 "Instructions") officially added `OFPIT_GOTO_TABLE`, `OFPIT_WRITE_METADATA`, `OFPIT_WRITE_ACTIONS`, `OFPIT_APPLY_ACTIONS`, and `OFPIT_CLEAR_ACTIONS`. Among these, `Goto-Table` is the only one that hands control between tables; the other four operate on the *same entry*. OpenFlow 1.2 (12/2011) and 1.3 (06/2012) kept the semantics. OF 1.5 (03/2015) added an egress table and required `Goto-Table` from an ingress entry to land on another ingress table, and from an egress entry to land on another egress table. OVS supported `goto_table` starting with version 1.10 (02/2013), when the `ofproto-dpif-xlate` codepath was reorganized to handle multi-table.

**Position in the architecture.** In the OVS source tree (baseline `v2.17.9`, branch `branch-2.17`), `goto_table` appears at four layers. (1) **Wire format**: `include/openflow/openflow-1.1.h` defines the instruction `enum OFPIT11_GOTO_TABLE = 1` and the 8-byte `struct ofp11_instruction_goto_table`. (2) **OVS internal representation**: `include/openvswitch/ofp-actions.h` declares `struct ofpact_goto_table` which contains a single `uint8_t table_id` field (the instruction is stored in the same list as actions inside the `ofpacts` structure). (3) **Parse, encode, format**: `lib/ofp-actions.c` contains the functions `parse_GOTO_TABLE`, `encode_GOTO_TABLE`, and `format_GOTO_TABLE`, which convert between the text form `goto_table:N`, the OVS struct, and the OF wire bytes. (4) **Xlate**: `ofproto/ofproto-dpif-xlate.c` handles the `OFPACT_GOTO_TABLE` case inside `do_xlate_actions()` by calling `xlate_table_action()` to dispatch the classifier lookup to the destination table. Scope: per flow entry. Each flow entry holds at most one `Goto-Table` instruction (per spec 1.1+ Section 5.9 "Instructions"); to chain N tables, the operator writes N flow entries with `goto_table` spread across the tables.

**Role.** `goto_table` plays three roles tied to the entire multi-table pipeline. (1) **Chaining primitive**: the only ONF-standard mechanism for moving from table N to table M where M is greater than N. Without it, every table is its own silo. (2) **Action-set transport**: the `Goto-Table` instruction does not clear the accumulated action set. Every prior `Write-Actions` carries forward to the destination table, which lets later tables override earlier decisions (for example, table 0 issues `write_actions(output:1)` and table 5 issues `write_actions(output:2)`, and the final outcome is `output:2` because the action set keeps only one `output`). (3) **Modularization enabler**: lets the operator write the pipeline as a sequence of stages (classify, ACL, routing, forwarding, egress) instead of one Cartesian table, which reduces the entry count from a product to a sum.

**Why it exists.** Two core technical motivations. (1) **Silicon mapping.** Commercial L2/L3 ASICs have a hard-wired multi-stage pipeline. OF 1.0 single-table forced the compiler to simulate every stage through one virtual table, which exploded the entry count. `goto_table` lets each OF table map 1:1 to a silicon stage. (2) **Rule-design scalability.** With 100 ACL rules, 1000 FIB entries, and 5000 MAC entries, a single-table design needs 100 × 1000 × 5000 = 500 million entries, while a multi-table design only needs 100 + 1000 + 5000 = 6100 entries chained through `goto_table`. The OF 1.1 spec Section 1.1 "Switch Components" explains this directly: "An OpenFlow switch has multiple flow tables ... a packet matches against entries in successive tables". The design philosophy trusts the compiler or operator to decompose the logic into stages; OF only provides the primitive `goto_table` to chain them.

**Problems solved.** Four specific problems that `goto_table` addresses. (1) **Table fan-out.** Instead of writing `100 × 5000` entries for the Cartesian product (ACL × MAC), the operator writes 100 entries in table 0 plus 5000 entries in table 1 and connects them with `goto_table:1`. (2) **Stage isolation.** ACL rules are not mixed with MAC entries; each stage can be debugged on its own (`ovs-ofctl dump-flows br0 table=0` only shows the ACL stage). (3) **Cumulative action.** The same `Goto-Table` instruction transports the action set across tables, without copying entries. (4) **Loop safety.** The spec requires `table_id > current_table_id`, which guarantees the pipeline always terminates (a forward DAG). There is no way to create a loop with pure `goto_table`; loops can only happen through `resubmit` or `recirc_id`, which are different topics covered in Part 9.32.

**Importance.** `goto_table` is one of the three absolutely-core primitives of multi-table pipelines, alongside the match field and the action set. Every production-grade OpenFlow controller (Faucet, ONOS, Ryu, OVN `northd`) emits `goto_table` in 80 percent or more of the flow entries that are not in the last table. In an OVN deployment on the branch-22.03 baseline, the output of `ovs-ofctl dump-flows br-int` on a typical compute host shows that about 70 percent of flow entries end with `goto_table:N`, about 20 percent end with `resubmit(,N)`, and the rest end with `output`, `drop`, or `controller`. An OVS or SDN engineer who does not deeply understand `goto_table` cannot read the OVN `br-int`, cannot analyze the pipeline that OVN `northd` compiles, and cannot debug why a packet "disappears" between table 17 and table 24. `goto_table` belongs to the same tier as `output` and `match`, the absolutely-required primitives of OpenFlow.

**How it works.** The wire format `struct ofp11_instruction_goto_table` is exactly 8 bytes (fixed, aligned to 64 bits):

```
Offset  Field     Length  Description                        Example goto_table:1
------  --------  ------  --------------------------------   --------------------
0-1     type      2 byte  OFPIT_GOTO_TABLE = 0x0001          0x00 0x01
2-3     len       2 byte  Length of the struct = 0x0008      0x00 0x08
4       table_id  1 byte  Destination table (0-254;          0x01
                          0xff reserved)
5-7     pad       3 byte  Padding for 64-bit alignment       0x00 0x00 0x00
```

The OVS internal struct `ofpact_goto_table` (defined in `include/openvswitch/ofp-actions.h`):

```c
/* OFPACT_GOTO_TABLE
 *
 * Used for OFPIT11_GOTO_TABLE */
struct ofpact_goto_table {
    OFPACT_PADDED_MEMBERS(
        struct ofpact ofpact;
        uint8_t table_id;
    );
};
```

Codepath when a packet enters the pipeline: `ofproto-dpif-xlate` calls `do_xlate_actions()` to walk the `ofpacts` list of the current entry. When it sees `OFPACT_GOTO_TABLE`, it does not recurse immediately; instead it sets `ctx->table_id = goto_table->table_id` and then `xlate_table_action()` performs the classifier lookup in the destination table and continues `do_xlate_actions()` on the newly matched entry. The invariant `goto_table->table_id > ctx->table_id` is enforced in `parse_GOTO_TABLE` at flow-install time, and a violation returns `OFPBIC_UNSUP_INST` or `OFPBIC_BAD_TABLE_ID`. Termination: when the pipeline reaches an entry without a `goto_table`, the accumulated action set is executed and the pipeline ends.

**How an operator masters this.** Six core actions. (1) **Install `goto_table` with the correct `ovs-ofctl` syntax**:

```bash
sudo ovs-ofctl add-flow br0 \
  "table=0,priority=100,ip,actions=goto_table:1"
sudo ovs-ofctl add-flow br0 \
  "table=1,priority=100,ip,nw_dst=10.0.0.0/8,actions=output:2"
```

(2) **Read `dump-flows` and recognize the chain**: `ovs-ofctl dump-flows br0 | grep goto_table` shows every pipeline-transitioning entry; missing results in a non-final table mean the pipeline is broken. (3) **Verify the increasing order**: use `awk` to make sure every `goto_table:N` has N greater than the table that contains the entry. (4) **Trace with `ovs-appctl ofproto/trace`** to verify the transition (see the example below). (5) **Avoid confusion with `resubmit`**: if the goal is simple pipeline modularization, use `goto_table`; if the goal is to call a sub-pipeline and then return to continue the action list, use `resubmit`. (6) **Read OVN flows**: with OVN, the `ovn-trace` command (Part 13.7) decodes pipeline logic backward into NB ACLs and routes, while `ofproto/trace` (Part 9.25) shows the chain of `goto_table` from table 0 through table 24 to table 65 on `br-int`.

**Classification.** `goto_table` belongs to the **OpenFlow Instruction** category in the taxonomy of 5 instruction groups (`Apply-Actions`, `Clear-Actions`, `Write-Actions`, `Write-Metadata`, `Goto-Table`). It differs from `Apply-Actions` (which executes an action immediately) and from `Write-Actions` (which accumulates into the action set for execution at the end of the pipeline). `goto_table` is the only instruction that changes the *next table*; the other four operate *on the packet*. In the "control flow" sub-classification, `goto_table` is paired with `output_to_controller` and `drop`; all three end a pipeline decision, but `goto_table` continues inside the switch while the other two leave the pipeline. This is an absolutely-core instruction, indispensable in any multi-table pipeline.

**Usage workflow.** A pipeline-design workflow that uses `goto_table` has seven steps. (1) **Identify the chain of logical stages** (for example: classify, ACL ingress, routing, ACL egress, output). (2) **Number the tables with gaps** (0, 10, 20, 30, 40 instead of 0, 1, 2, 3, 4) so a new stage can be inserted later without renumbering. (3) **Write the table 0 entry** as `actions=...,goto_table:10` to push every packet into the ACL ingress stage. (4) **Write the entry for each subsequent table** ending with `goto_table:<next_table>` to chain forward; the entry in the last table ends with `output` instead of `goto_table`. (5) **Always install a table-miss entry** (`priority=0,actions=...`) for every table that can receive packets that do not match. Without a table-miss, an unmatched packet is silently dropped. (6) **Verify with `ovs-appctl ofproto/trace`** for at least one representative packet per tenant or protocol to confirm the full transition. (7) **Document the table numbering** in a README or in the cookie comment (`cookie=0xacl10`) so on-call engineers understand each table's role.

**When something goes wrong.** Four signals in production that bring `goto_table` to mind. (1) **Silent drop in the middle of the pipeline**: `ovs-ofctl dump-flows br0` shows that an entry in table 5 has `goto_table:10`, but table 10 has no matching entry, so the packet is dropped; the table 5 `n_packets` counter still increases but nothing arrives at the egress. The diagnosis is to run `ofproto/trace` and observe "Final flow ... no_match in table 10". The fix is to add a table-miss entry for table 10 or to add a matching entry. (2) **`OFPBIC_BAD_TABLE_ID` at flow install**: trying to install `goto_table:N` for an N that does not exist (for example N=255 when the switch only supports 254 tables). The fix is to inspect `n_tables` through `ovs-ofctl show br0`. (3) **`OFPBIC_UNSUP_INST` at flow install**: trying to install `goto_table:M` where M is less than or equal to the current table. The fix is either to redesign the table numbering or to use `resubmit(,M)` (an OVS extension that allows reverse). (4) **OVN packet does not reach the destination**: `ovn-trace` reports the packet follows the correct logical pipeline, but `ofproto/trace` on `br-int` shows the packet lost in the middle of a `goto_table` chain (for example, between table 17 and table 24). A common cause is that `ovn-controller` has not yet installed the table 24 entry on the chassis because the SBDB `Logical_Flow` has not synchronized. The fix is to compare `ovn-sbctl lflow-list` with `ovs-ofctl dump-flows br-int table=24`.

**Tightly related to.** `goto_table` is closely paired with five entities. (1) The `Apply-Actions` instruction (Part 3.6): `goto_table` follows `Apply-Actions` in the same entry, so the action is executed first and then the packet moves to the destination table. (2) The `Write-Actions` instruction: the action set accumulates across the `goto_table` chain and is not cleared on transition. (3) The `resubmit(,N)` action (NXM extension, Part 9.22.3): semantically opposite. `resubmit` returns; `goto_table` does not. (4) The table-miss entry (`priority=0`): always recommended on every table that can receive packets through `goto_table`. (5) The OVN logical pipeline (Parts 13.7, 13.19): `ovn-controller` compiles every logical flow into a chain of `goto_table` on `br-int`. Tables 0 to 7 hold ingress port binding, 8 to 23 hold the ingress logical pipeline, 24 to 31 handle tunneling, and 32 to 47 hold the egress logical pipeline.

**Version differences.**

| Version | `goto_table` support | `table_id` constraint | Semantic change |
|---------|----------------------|-----------------------|-----------------|
| OF 1.0 (12/2009) | Not supported | N/A | Single-table, no instructions |
| OF 1.1 (02/2011) | Yes (new) | `table_id > current_table_id` | Instructions appear for the first time |
| OF 1.2 (12/2011) | Yes | Same as 1.1 | OXM TLV match replaces fixed structs |
| OF 1.3 (06/2012) | Yes | Same as 1.1 | Group and meter tables added to the pipeline |
| OF 1.4 (10/2013) | Yes | Same as 1.1 | Bundle commit affects `goto_table` install |
| OF 1.5 (03/2015) | Yes | Distinguishes ingress and egress: an ingress entry only goes to an ingress table; an egress entry only goes to an egress table | Egress table added to the pipeline |

OVS handles every 1.1+ version through a single codepath in `parse_GOTO_TABLE` and `encode_GOTO_TABLE` (`lib/ofp-actions.c`); the version difference is only at the wire encode (the OF 1.1+ struct is unchanged). The OVS branch-2.17 default protocol is `OpenFlow13`, so `goto_table` is available by default. **Source code reference.** In the OVS source tree (baseline `v2.17.9`, branch `branch-2.17`), `goto_table` lives at five locations. (1) `include/openflow/openflow-1.1.h` declares the `enum ofp11_instruction_type` with `OFPIT11_GOTO_TABLE = 1` and the 8-byte `struct ofp11_instruction_goto_table` (type, len, table_id, pad[3]). (2) `include/openvswitch/ofp-actions.h` declares `struct ofpact_goto_table`, which only carries a `uint8_t table_id`, registered through the macro `OFPACT(GOTO_TABLE, ofpact_goto_table, ofpact, "goto_table")`. (3) `lib/ofp-actions.c` contains four functions for `goto_table`: `encode_GOTO_TABLE`, `parse_GOTO_TABLE`, `format_GOTO_TABLE`, and `check_GOTO_TABLE`. The functions cover encode (build the OF wire bytes from the OVS struct), parse (build the OVS struct from the text `goto_table:N`), format (build the text from the OVS struct), and check (validate the parsed instruction at flow-install time, including the `table_id <= 0xff` check). The decode side is dispatched through macro-generated wrappers from the `OFPACT` registry, not as a standalone `decode_OFPIT11_GOTO_TABLE` function. (4) `ofproto/ofproto-dpif-xlate.c` handles the `OFPACT_GOTO_TABLE` case inside `do_xlate_actions()` by setting `ctx->table_id` and then calling `xlate_table_action(ctx, ctx->in_port, table_id, ...)` to perform the classifier lookup on the destination table. The signature is `static void xlate_table_action(struct xlate_ctx *, ofp_port_t in_port, uint8_t table_id, bool may_packet_in, bool honor_table_miss, bool with_ct_orig, bool is_last_action, xlate_actions_handler *)`. (5) `ofproto/ofproto.c` validates at flow-install that `goto_table->table_id > flow->table_id`, which enforces the forward-only invariant. Specification: ONF OpenFlow Switch Specification 1.5.1 Section 6.1 "Pipeline Processing" and Section A.2.5 "Goto-Table Instruction"; OpenFlow 1.1.0 Section 5.9.1 "Instructions" introduced the instruction. **Real-world incident.** An OVN multi-tenant incident in 2024-Q4: after upgrading `ovn-controller` from 22.03.7 to 22.03.9 across 64 compute nodes, several tenants reported east-west traffic between two VMs on the same logical switch being randomly dropped (5 to 10 percent of packets). `ovn-trace` on SBDB reported the correct pipeline: tables `LS_IN_PORT_SEC_L2`, `LS_IN_PRE_ACL`, `LS_IN_ACL`, `LS_IN_QOS_MARK`, `LS_IN_LB`, `LS_IN_STATEFUL`, `LS_IN_ARP_ND_RSP`, `LS_IN_DHCP_OPTIONS`, `LS_IN_L2_LKUP`, output. But `ofproto/trace` on the host's `br-int` showed the packet ending unexpectedly at table 17 with "no_match", while the same packet from a different VM on the same logical switch passed through successfully. Comparing `ovs-ofctl dump-flows br-int table=17` against `ovn-sbctl lflow-list <ls>` showed that two important entries (`LS_IN_L2_LKUP` for unicast destination MAC) only existed on about 90 percent of the chassis. On the remaining 10 percent of chassis, `ovn-controller` had installed the entries into table 16 instead of table 17 because of a renumbering bug in version 22.03.9 (the stage order in `ovn-northd` had changed). The consequence: the `goto_table:17` entry in table 16 sent the packet into an empty table 17, where it was dropped. Short-term fix: roll `ovn-controller` back to 22.03.7 on the affected chassis. Long-term fix: wait for the upstream patch `30aab12c0` that rebuilds the stage-to-table mapping inside `ovn-controller`. Lesson: a `goto_table` chain is only stable when *both sides* (the source entry and the destination entry) are installed consistently. Any skew between the producer (`ovn-controller`) and the destination state (the `table_id` mapping) shows up as a silent packet drop. Cross-link: 20.0 incident-response decision tree, Part 13.7 `ovn-controller` internals.

**Common failures and diagnostic signals.** Four common operator mistakes. (1) **Forgetting the table-miss in the destination table**: a `goto_table:N` entry pushes the packet into table N, but table N has no matching entry, so the packet is dropped. Signal: the table N counter shows `lookup_count` increasing while `matched_count` stays flat. Diagnose with `ovs-ofctl dump-table-features br0`, which shows `default = drop` if there is no table-miss. Fix: add `table=N,priority=0,actions=drop` (explicit) or `actions=normal`. (2) **Wrong `table_id` in the cookie**: writing `cookie=0x10` to mean "table 10" but failing to keep it in sync with `goto_table:10`. A future maintainer who greps by cookie will not match the actual table. Fix: adopt a clear convention where the cookie encodes both the table and the rule ID. (3) **Forward-only violation under an old OF version**: switch firmware that speaks OF 1.0 does not understand `goto_table` and returns `OFPET_BAD_INSTRUCTION` or fails silently. Signal: `ovs-ofctl -O OpenFlow10 add-flow ... actions=goto_table:1` always errors. Fix: use `-O OpenFlow13`. (4) **Mistaking `goto_table` for an action**: writing `actions=output:1,goto_table:2` (OVS accepts this syntax, but the spec treats `goto_table` as an instruction that must follow the action list). Signal: undefined behavior on hardware switches other than OVS. Fix: understand that `goto_table` is an instruction, place it at the end of the actions list, or use the explicit form `instructions=apply_actions(output:1),goto_table:2`.

**Comparison with other systems.** Four important comparisons. (1) **The OVS extension `resubmit(,N)`** (Part 9.22.3): same intent of moving the packet to table N, but `resubmit` is an *action* (it calls a sub-pipeline and returns to execute the rest of the action list), while `goto_table` is an *instruction* (a one-way transfer of control with no return). `resubmit` allows N to be smaller and limits recursion depth to 75; `goto_table` requires N to be greater than the current table. (2) **The OF 1.1+ `Goto-Group` instruction**: moves the packet to a group entry (multipath, broadcast); differs from `goto_table` because the group executes one fixed action bucket and does not continue the pipeline. (3) **Linux netfilter chain hooks** (PREROUTING, INPUT or FORWARD, POSTROUTING): also a multi-stage pipeline, but the chains are fixed by netfilter design; new chains cannot be inserted dynamically the way `goto_table` inserts an OF table. The iptables `--jump CHAIN` is logically similar to `goto_table` but uses a different wire format. (4) **P4 control flow `apply { table_a.apply(); table_b.apply(); }`**: P4 expresses pipeline dataflow more completely (with if/else branching on lookup results, table abstractions, action selectors), while OF `goto_table` only chains linearly by `table_id`. Trade-off: P4 is more powerful but requires a target-specific P4 compiler; OF `goto_table` is simpler and standardized, available on every OF 1.1+ switch.

## 9.22.10 `multi-table pipeline`, the OpenFlow 1.1+ paradigm that replaces the OF 1.0 single table

**Concept.** A `multi-table pipeline` is the **packet-processing paradigm** of OpenFlow 1.1+ in which a switch organizes flow entries into **N flow tables** numbered `0, 1, 2, ..., N-1` (with N at most 254 in the wire format `uint8_t`, where 0xFF is reserved for `OFPTT_ALL`). A packet entering the pipeline always starts at table 0; each table performs one classifier lookup; the matching entry specifies an *instruction set* with at most five instructions (`Apply-Actions`, `Clear-Actions`, `Write-Actions`, `Write-Metadata`, `Goto-Table`); the `Goto-Table` instruction pushes the packet to a higher-numbered table to continue, while an entry without `Goto-Table` marks the *end of the pipeline*. At that point, the action set accumulated across tables is executed atomically and the packet leaves the pipeline. Distinguish clearly between *flow table* (the component unit) and `goto_table` (the instruction that moves between tables). `multi-table pipeline` is the **overall architecture**, whether N = 1 (the degenerate case, equivalent to the OF 1.0 single table) or N = 64 (the production OVN `br-int` per chassis).

**History and background.** OpenFlow 1.0.0 (12/2009) specified a *single flow table* with about 12 fixed match fields, no instruction set, and no concept of chaining tables. Implementations on commercial L2/L3 ASICs (Broadcom Trident, Marvell Prestera) exposed an architectural gap: the ASIC has a hard-wired multi-stage pipeline (ingress parser, ingress ACL, L3 lookup, L2 lookup, egress ACL, queueing, egress parser), and each stage is a separate TCAM. Forcing a single OF table to simulate every stage required a Cartesian product of entries, which exploded in size. The ONF Technical Memorandum of July 2010, *"Issues with the OpenFlow 1.0 specification"*, proposed multi-table. OpenFlow 1.1.0 (02/2011, Section 1.1 *"Switch Components"* and Section 5.2 *"Pipeline Processing"*) officially introduced the `multi-table pipeline`, the five instructions, and OXM TLV match. OF 1.2 (12/2011) kept the architecture and extended OXM. OF 1.3 (06/2012) added the group table and the meter table to support the pipeline. OF 1.4 (10/2013) added bundles for atomic flow installs. OF 1.5 (03/2015) split the *ingress* and *egress* pipeline into two independent table chains, each with its own forward-only `goto_table` rule. OVS supported a true multi-table pipeline starting with version 1.10 (02/2013); earlier versions (1.0 to 1.9) only had the `resubmit` extension inherited from NOX (Stanford, 2008). The 14 months between OF 1.0 and OF 1.1 was an unusually short standardization cycle, a strong signal that a single table was not viable in production.

**Position in the architecture.** The `multi-table pipeline` is not a field, an action, or a message; it is an **architectural wrapper** around the entire datapath logic of the switch. In the OVS source baseline `v2.17.9`, the pipeline appears at four layers. (1) **Kernel and userspace datapath**: the kernel datapath (`datapath/datapath.c` plus `datapath/flow_table.c`) only sees a flat megaflow cache; the pipeline is a *userspace abstraction*. Every first-time upcall traverses the full pipeline in `ofproto-dpif` and produces a synthesized megaflow. (2) **`ofproto-dpif` xlate** (`ofproto/ofproto-dpif-xlate.c`): the function `xlate_actions()` calls `do_xlate_actions()` for each matched entry, and on `OFPACT_GOTO_TABLE` it recurses through `xlate_table_action()` to dispatch into the destination table, traversing the whole pipeline in a single upcall. (3) **Per-table classifier** (`ofproto/ofproto.c` plus `lib/classifier.c`): each table is a `struct oftable` containing a `struct classifier cls` that holds the per-table flow entries. (4) **OF wire**: the message `OFPT_TABLE_FEATURES` (type 12) reply lists N tables, each with its own capabilities (max_entries, supported instructions, supported actions); `OFPT_TABLE_MOD` (type 17) configures the table-miss behaviour. Scope: per bridge. Each OVS bridge has its own independent pipeline (for example, `br-int` runs 64 tables for OVN, while `br-ex` runs 1 to 2 tables for external traffic).

**Role.** The `multi-table pipeline` plays five irreplaceable roles. (1) **Modularization**: separates packet processing into independent stages (classify, ACL, routing, forwarding, egress), one stage per table, similar to how an OS splits the kernel into modules. (2) **Scalability**: instead of the Cartesian product inside a single table, multi-table only needs the sum. For example, with 100 ACL rules, 1000 FIB entries, and 5000 MAC entries, a single table needs 5 times 10 to the 8 entries; a multi-table design needs 6.1 times 10 to the 3 entries; a reduction of about 8.2 times 10 to the 4. (3) **Silicon mapping**: enables a 1:1 compile from an OF table to a TCAM stage in the ASIC, which lets the design exploit the hardware fast path. (4) **Compositional reasoning**: an engineer can debug each stage on its own (`ovs-ofctl dump-flows br-int table=24`); a single-table design forces the engineer to look at every entry. (5) **Logical-physical separation**: lets a high-level compiler (such as `ovn-northd`) emit a logical pipeline of 64 stages, and lets `ovn-controller` translate that into OF flows on `br-int` per chassis. This separates the controller-logic concern from the switch-executor concern.

**Why it exists.** Three core technical motivations, all at once. (1) **Entry explosion in a single table** (already explained under Role). (2) **Silicon pipeline mismatch**: commercial L2/L3 ASICs have natural hard-wired stages; forcing a single OF table makes the compiler emit entries that flatten the stages and lose every hardware advantage. Multi-table restores the natural mapping. (3) **Rule-design ergonomics**: writing 100 ACL rules plus 5000 MAC entries separately is easy to read and easy to inspect; writing 5 times 10 to the 5 Cartesian entries is not feasible for a human. The OF 1.1 spec Section 1.1 says directly: *"OpenFlow switches contain multiple flow tables to allow for scalability and modularity"*. Design philosophy: the standard provides primitives (table, instruction, `Goto-Table`); the compiler or operator decides how to split the stages; the standard does not fix the number of tables.

**Problems solved.** Four specific problems addressed by the pipeline. (1) **Table fan-out** (Cartesian explosion). (2) **Stage isolation**: each logical stage is independent, so debugging one stage does not require knowledge of the others. (3) **Hierarchical compilation**: lets a tool like OVN `northd` emit a 64-stage logical pipeline, after which an OF translator emits per-chassis hardware OF flows. Without multi-table, this hierarchy cannot be compiled. (4) **Operational checkpoint**: per-table counters (`lookup_count`, `matched_count`) are an important diagnostic signal that shows where in the pipeline a packet drops. A single table only has one combined counter and loses the location information.

**Importance.** The `multi-table pipeline` sits in the **absolutely-core position** of OpenFlow 1.1+ and OVS, on par with `match field` and `action set`. Every production controller after OF 1.1 (Faucet, ONOS, Ryu, OpenDaylight, OVN `northd`) assumes multi-table; no serious controller still operates with a single table. On a typical OVN compute host (branch-22.03 baseline, 50 tenant VMs, 10 logical switches), the output of `ovs-ofctl dump-flows br-int | wc -l` is around 3000 to 8000 flow entries spread across about 64 tables. An SDN engineer who does not deeply understand the pipeline paradigm cannot (a) read the OVN `br-int`, (b) analyze the pipeline that OVN `northd` compiles, (c) debug a packet that "disappears" between table 17 and table 24, (d) optimize the OF rule design for hardware offload, or (e) rewrite the stage sequence during a logic upgrade. Understanding the pipeline is on par with understanding the TCP three-way handshake for a network engineer.

**How it works.** Traversing the pipeline for one packet has four phases. **Phase 1 (enter table 0).** The packet arriving at the switch is first parsed (fields extracted) and then enters the classifier of table 0. The lookup picks the highest-priority matching entry; if a match exists, the entry's instruction set is selected. **Phase 2 (apply per-entry instructions).** The entry's instruction set runs in a fixed order per ONF spec: `Meter` (if present), `Apply-Actions` (executes actions immediately and mutates the packet), `Clear-Actions` (clears the action set), `Write-Actions` (adds actions to the action set without executing them), `Write-Metadata` (updates the 64-bit metadata field), and `Goto-Table` (moves to the next table). If there is no `Goto-Table`, the flow jumps to Phase 4. **Phase 3 (transition).** The packet (possibly mutated by `Apply-Actions`) plus the accumulated action set and metadata move to the table indicated by `Goto-Table`; the classifier looks up that table; Phase 2 repeats. **Phase 4 (terminate).** When the pipeline reaches an entry without `Goto-Table` (or when a table-miss with `output:CONTROLLER` happens), the action set is applied atomically in the standard order (`copy_ttl_in`, `pop`, `push_*`, `copy_ttl_out`, `dec_*`, `set`, `qos`, `group`, `output`); the packet leaves the pipeline. The full sequence of phases 1 to 4 happens atomically per packet (no context switch between phases); an entry deleted by the controller between Phase 1 and Phase 4 does not affect a packet already in flight.

**How an operator masters this.** Seven core actions. (1) **List table support**: `ovs-ofctl -O OpenFlow13 dump-tables br0` shows how many tables the switch has and the `max_entries` of each. (2) **Design the stage logic**: draw the pipeline on paper before installing entries; number the tables with gaps (0, 10, 20, 30, ...) so a new stage can be inserted later. (3) **Install a table-miss in every table**: `ovs-ofctl add-flow br0 "table=N,priority=0,actions=drop"` (or `actions=normal`) for every N that can receive packets. Without a table-miss, an unmatched packet is silently dropped. (4) **Verify the `goto_table` chain**: run `ovs-ofctl dump-flows br0 | grep -E "table=|goto_table" | sort -t= -k2 -n` to make sure every non-final table has an entry that hands forward. (5) **Trace the full pipeline**: `ovs-appctl ofproto/trace br0 "in_port=1,ip,nw_src=10.0.0.1,..."` shows the path through every table along with the action set and metadata at each phase. (6) **Read the OVN `br-int`**: `ovs-ofctl dump-flows br-int | awk '{print $5}' | sort -t= -k2 -n -u` shows which tables are active and maps them to SBDB Logical_Flow stages through `ovn-detrace`. (7) **Measure per-table counters**: `ovs-ofctl dump-tables-desc br-int` plus `ovs-ofctl dump-flows br-int table=N` lets the operator compare `lookup_count` and `matched_count` to identify tables where packets are dropping.

**Classification.** Pipelines can be classified along three axes. By **scale**: single-table (the degenerate case, OF 1.0 or OF 1.1+ with N=1), small (2 to 10 tables, mostly manual config), medium (10 to 30 tables, mid-tier controllers such as Faucet), or large (30 to 64+ tables, OVN `northd` compile). By **direction** (OF 1.5+): ingress-only pipeline (OF 1.1 to 1.4), or bidirectional ingress-plus-egress pipeline (OF 1.5+, two independent chains). By **hierarchy**: flat pipeline (every table at the same level, no group or meter), or hierarchical pipeline (the main table chains through `Goto-Table` and group tables are called through `output:group:N`). By **content origin**: hand-written (operator writes with `ovs-ofctl`), compiler-generated (OVN `northd`, ONOS Atomix), or hybrid (90 percent compiler plus 10 percent operator override). This is an absolutely-core paradigm, indispensable in any production OF 1.1+ controller.

**Usage workflow.** A multi-table-pipeline design workflow has eight steps. (1) **Analyze the processing logic**: list the required stages (classify, ACL, routing, ARP, MAC learning, output). (2) **Number the tables with gaps** (0, 5, 10, 15, ...) so a new stage can be inserted without renumbering. (3) **Write the table 0 entry** ending with `goto_table:5` (push every packet into the first stage). (4) **Write the entry for each stage** ending with `goto_table:<next>` for an intermediate stage, or `output:N` (or `drop`) for the final stage. (5) **Always install a table-miss** for every table that can receive packets, either `priority=0,actions=drop` (explicit) or `actions=goto_table:<next>` (skip). (6) **Verify with `ofproto/trace`** for 5 to 10 representative packets per tenant or protocol; confirm the path through the expected stages. (7) **Document the pipeline**: a README or a cookie comment (`cookie=0xacl05`) should record the role of each table for on-call engineers. (8) **Monitor per-table counters** in production to catch unusual drops at a specific stage.

**When something goes wrong.** Five production signals that bring the pipeline concept to mind. (1) **Silent drop in the middle of the pipeline**: `dump-flows br0 table=N` has no entry that matches the required field, so the packet enters table N and is dropped. Detect by comparing `lookup_count` (rising) with `matched_count` (flat). (2) **OVN packet does not reach the destination on a single chassis**: `ovn-trace` on SBDB shows the correct logical pipeline, but `ofproto/trace` on the chassis `br-int` loses the packet between `goto_table` steps. The usual cause is that `ovn-controller` has not yet installed the entry or that the table numbering shifted during an upgrade. (3) **Suspected infinite loop** (impossible with pure `goto_table` because of the forward-only rule, but possible when interleaved with `resubmit`): the trace reports "stack too deep" after 75 levels of recursion and the packet is dropped. (4) **Unexpected table-miss**: switch firmware defaults table-miss to `OFPP_CONTROLLER` for OF 1.0 versus `drop` for OF 1.1+, so mixing versions can flood the controller with packet-in messages or drop unexpectedly. (5) **Counter drift between stages**: table A's `matched_count` is higher than table B's `lookup_count` (where B is A's `goto_table` destination), which means packets are lost between the two tables. Causes include an intervening `drop` action or a `pop_vlan` that changed a field so it no longer matches in B.

**Tightly related to.** The pipeline is closely paired with six entities. (1) The `Goto-Table` instruction (Part 9.22.9): the only primitive that connects the tables. (2) The action set (Part 4.0): accumulates across the table chain and is executed atomically at the end of the pipeline. (3) The 64-bit `metadata` field and the OVS `reg0..reg15` extension (Part 9.22.6): pass state between tables without modifying the actual packet. (4) The table-miss entry (`priority=0`): the mandatory fallback for every table to avoid silent drops. (5) The `OFPT_TABLE_FEATURES` reply (Part 4.7): describes per-table capabilities so that a controller can discover them. (6) The OVN logical pipeline (Part 13.16): `ovn-northd` compiles about 64 stages, and `ovn-controller` translates them per chassis into OF flows; every flow participates in the multi-table pipeline on `br-int`.

**Source code reference.** In the OVS source tree (baseline `v2.17.9`, branch `branch-2.17`), the `multi-table pipeline` is implemented at seven locations. (1) **Per-bridge table struct**: `struct oftable` in `ofproto/ofproto-provider.h` (contains `struct classifier cls`, `n_flows`, `name`, `max_flows`, `eviction_policy`). Each bridge allocates an `oftable[256]` array (the max size). (2) **Per-table classifier** (`lib/classifier.c`): a Tuple Space Search algorithm that finds the highest-priority matching flow, called from `ofproto-dpif-xlate.c` on every `xlate_table_action()` invocation. (3) **Xlate engine** (`ofproto/ofproto-dpif-xlate.c`): the function `xlate_actions()` calls `do_xlate_actions()` to walk the entry's instruction list; the `OFPACT_GOTO_TABLE` case sets `ctx->table_id = goto->table_id` and then calls `xlate_table_action(ctx, in_port, table_id, may_packet_in, honor_table_miss, with_ct_orig, is_last_action, action_handler)`. (4) **OF wire format**: `include/openflow/openflow-1.1.h` defines `enum ofp11_flow_mod_command`, `OFPT11_TABLE_MOD`, and `struct ofp11_table_mod`; `include/openflow/openflow-1.3.h` defines `OFPMP_TABLE_FEATURES` and `struct ofp13_table_features`. (5) **Table-miss handling and eviction**: the function `oftable_configure_eviction()` in `ofproto/ofproto.c` (declared at line 94, defined at line 9234 at v2.17.9). The default table-miss is drop on OF 1.1+, and packet-in to controller on OF 1.0. (6) **Table statistics**: `struct oftable` contains the counters `n_lookups` and `n_matched`, updated on every packet and exposed through the `OFPST_TABLE` stats reply. (7) **Table-mod config**: the `OFPT_TABLE_MOD` message updates `vacancy_events`, `eviction_policy`, and `controller_count`. Specification: ONF OpenFlow Switch Specification 1.5.1 Section 6.1 *"Pipeline Processing"*; OF 1.1.0 Section 5.2 *"Pipeline Processing"* introduced the paradigm. All function names above are verified at the OVS v2.17.9 baseline (commit `0bea06d9957e3966d94c48873cd9afefba1c2677`, 2024-02-08).

**Real-world incident.** An OVN multi-tenant incident in 2024-Q3: when `ovn-northd` was upgraded from 22.03.5 to 22.03.8 across the control plane (a 3-node bonded HA cluster), operators observed roughly 3 percent random drop on north-south traffic for floating-IP subnets, with no impact on east-west traffic between VMs on the same logical switch. `ovn-trace` on SBDB returned the correct pipeline (`LR_IN_DEFRAG`, `LR_IN_UNSNAT`, `LR_IN_DNAT`, `LR_IN_IP_ROUTING`, `LR_OUT_SNAT`, `LR_OUT_EGR_LOOP`, `LR_OUT_DELIVERY`). On an affected compute host, `ofproto/trace br-int "in_port=patch-port,ip,nw_dst=<floating_ip>"` showed the packet passing through tables 0 to 23 correctly and arriving at table 24 (`LS_IN_L2_LKUP`) where the entry sent it on with `goto_table:25`. But `dump-flows br-int table=25` was empty on about 10 percent of the chassis. Comparison through `ovn-detrace` against SBDB showed that `ovn-northd` 22.03.8 had moved `LR_IN_DEFRAG` from table 25 to table 24 (a renumbering for pipeline efficiency), while `ovn-controller` 22.03.5 still cached the old mapping on the 10 percent of chassis that had not yet been restarted in sync. The result: the `goto_table` chain broke. The new `northd` placed entries at the new tables, but the old `ovn-controller` saw entry placement according to the old mapping, and table 25 was actually empty. Short-term fix: `systemctl restart ovn-controller` on the drifted chassis, which forced re-fetch of the SBDB `Logical_Flow` and re-computation of the table mapping. Long-term fix: upstream patch `c7a23bf1` forces `ovn-controller` to invalidate the cache when a `northd` version skew is detected, and adds a metric `ovn-controller-stage-drift`. Lesson: a multi-table pipeline is only stable when *both the compiler (`northd`) and the executor (`ovn-controller` per chassis)* run the same stage map. Any skew between the producer and consumer of the pipeline shows up as a silent drop at the unmatched stage. Cross-link: 13.16 OVN logical pipeline `table_id` mapping; 20.0 incident-response decision tree.

**Common failures and diagnostic signals.** Five common operator mistakes. (1) **Single-table mindset ported into multi-table** (an operator used to OF 1.0 writes flat flows with `goto_table:N` scattered illogically). Signal: the pipeline contains `goto_table:0`, or the same `goto_table:N` appears in several different tables, which violates the forward-only invariant. Fix: draw the pipeline on paper first, number the tables in increasing order, do not recycle a `table_id`. (2) **Forgetting the table-miss entry on every table**: a packet matches table 0, gets pushed via `goto_table:5`, but table 5 has no matching entry, so the packet drops silently. Signal: table 5's `lookup_count` rises while `matched_count` stays flat. Fix: `add-flow "table=5,priority=0,actions=drop"` (explicit) or `actions=goto_table:6` (skip the stage). (3) **Table-miss default behaviour mismatch across OF versions**: OF 1.0 default table-miss is `OFPP_CONTROLLER` (send to controller), while OF 1.1+ default is drop. Mixing protocol versions between controller and switch leads to a packet-in flood at the controller or unexpected drops. Fix: an explicit table-miss entry on every table; do not rely on the default. (4) **Action-set override not behaving as expected**: `write_actions(output:2)` in table 5 and `write_actions(output:3)` in table 30. The engineer expects both outputs, but in reality only `output:3` is executed because the action set only keeps one `output` per type. Fix: use `apply_actions(output:2)` in table 5 if the action should execute immediately, or use `clone(output:2)` (NXM extension) to clone the packet. (5) **`OFPBIC_BAD_TABLE_ID` at flow install**: installing `goto_table:N` for an N greater than the number of tables the switch supports. Signal: `ovs-ofctl` reports `OFPBIC_BAD_TABLE_ID`. Fix: run `ovs-ofctl dump-tables br0` to see `n_tables` and make sure N is in range.

**Comparison with other systems.** Five important comparisons. (1) **OF 1.0 single-table**: the degenerate case of multi-table with N=1; loses modularity and silicon mapping; only exists in legacy controllers. (2) **Linux netfilter chain hooks** (PREROUTING, INPUT, FORWARD, OUTPUT, POSTROUTING): also a multi-stage pipeline, but the hooks are fixed by netfilter design (5 core chains); new chains cannot be inserted dynamically. iptables `--jump CHAIN` is similar to `goto_table` in spirit but uses a different wire format. (3) **Linux tc (traffic control)** with the qdisc, class, and filter structure: a 3-level hierarchy in which each level filters a sub-tree. Similar to multi-table for modularization but limited to QoS, not general-purpose. (4) **P4 control flow** `apply { table_a.apply(); if (hit) { table_b.apply(); } else { table_c.apply(); } }`: P4 expresses pipeline dataflow more completely (if/else branching on lookup results, table composition, action selectors), while OF multi-table only chains linearly by `table_id`. Trade-off: P4 is more powerful but requires a target-specific P4 compiler; OF multi-table is simpler and standardized, available on every OF 1.1+ switch. (5) **Cisco IOS ACL-then-route-then-CBQoS**: a hard-wired pipeline in Cisco hardware that preserves modularity but offers no choice of stage count or order; OF multi-table lets the operator choose freely. The trade-off is simplicity (Cisco) versus flexibility (OF).

## 9.22.11 `action_set` — bộ tích luỹ action thực thi cuối pipeline

**Khái niệm.** `action_set` là **bộ chứa action gắn liền với từng packet** trong khi packet di chuyển qua multi-table pipeline. Khác với `Apply-Actions` thực thi action lên packet *ngay* tại table hiện tại, `Write-Actions` chỉ *ghi* (gắn) action vào `action_set`; action không thực thi cho đến khi packet rời pipeline (entry không còn `Goto-Table`). Đây là cơ chế "trì hoãn quyết định cuối" của OpenFlow 1.1+. `action_set` không phải list mà là **set theo loại** — mỗi *loại action* (output, set_queue, push_vlan, group, ...) chỉ giữ một instance; ghi `Write-Actions(output:2)` ở table 5 rồi `Write-Actions(output:3)` ở table 30 → action set cuối cùng chỉ giữ `output:3` (override). Quan trọng phân biệt: `action_set` (theo packet, OF 1.1+ chuẩn) khác `action list` của `Apply-Actions` (thứ tự, có thể trùng lặp), khác `frozen_actions` (OVS internal cho recirculation).

**Lịch sử + bối cảnh.** Trước OpenFlow 1.1, OF 1.0 chỉ có một list action duy nhất gắn theo flow entry, thực thi tuần tự ngay khi match. Khi OpenFlow 1.1 (02/2011) công bố multi-table pipeline, ONF cần một mechanism để table phía sau ghi đè quyết định của table phía trước mà không phải "huỷ tác động đã làm" — `action_set` ra đời. Section 5.6 *"Action Set"* của OF 1.1 spec định nghĩa: action set chứa tối đa một action mỗi *loại*, action mới ghi cùng loại sẽ thay thế cũ. Section 5.10 *"Action List"* phân biệt action list (cho `Apply-Actions`) và action set (cho `Write-Actions`). OVS hỗ trợ `action_set` từ phiên bản 2.4 (08/2015), khi codepath xlate được tổ chức lại để track `ctx->action_set` qua chuỗi `goto_table`. OF 1.5 (03/2015) bổ sung `pkt_register` (`reg0..reg7` 64-bit) nhưng giữ nguyên semantics action set.

**Vị trí trong kiến trúc.** Trong OVS source baseline `v2.17.9`, `action_set` được hiện thực ở ba lớp. Một, **xlate context** `ofproto/ofproto-dpif-xlate.c` định nghĩa `struct xlate_ctx` chứa field `struct ofpbuf action_set` (buffer tích luỹ action) và `bool action_set_has_group`; khi `do_xlate_actions()` gặp `OFPACT_WRITE_ACTIONS`, hàm `xlate_write_actions()` (gọi gián tiếp qua dispatcher) merge action vào buffer theo rule "one per type". Hai, **internal representation** `include/openvswitch/ofp-actions.h` không có struct `ofpact_action_set` riêng (action set là *runtime state*, không phải *flow entry data*); nhưng có `OFPACT_WRITE_ACTIONS` enum đăng ký với struct `ofpact_nest` (nested action list để parse). Ba, **wire format** `include/openflow/openflow-1.1.h` `struct ofp11_instruction_actions` (type, len, padding, danh sách `struct ofp_action_header` lồng bên trong); cùng struct dùng cho cả `OFPIT_APPLY_ACTIONS` và `OFPIT_WRITE_ACTIONS`, phân biệt qua `type` field. Phạm vi tác động: per-packet, sống đời packet trong pipeline.

**Vai trò.** `action_set` đảm nhận ba vai trò. Một, **late-binding decision**: cho phép quyết định cuối (output port, queue, encap) được sửa đổi qua nhiều table mà không cần "rollback" tác động đã có. Hai, **action ordering atomicity**: khi pipeline kết thúc, action set thực thi theo thứ tự chuẩn ONF (`copy_ttl_in` → `pop` → `push_*` → `copy_ttl_out` → `dec_*` → `set` → `qos` → `group` → `output`). Operator không cần lo thứ tự khi ghi action ở các table khác nhau. Ba, **override-by-default semantics**: ghép cặp tự nhiên với compositional pipeline; table 5 đề xuất "output:queue 1", table 20 override "output:queue 5"; không cần explicit "clear before write".

**Vì sao sinh ra.** Có hai động cơ. Một, **multi-table compositional**: nếu OF 1.1 chỉ có `Apply-Actions`, mỗi table phải biết toàn bộ logic stage trước (vi phạm modularity); `Write-Actions` + `action_set` cho phép tách: stage early đề xuất, stage late override. Hai, **hardware action ordering**: ASIC có pipeline action cố định (modify TTL → modify L2 → encap → output); chuẩn ONF buộc `action_set` thực thi cùng thứ tự để mapping 1:1 sang TCAM action stage.

**Vấn đề giải quyết.** Ba vấn đề. Một, **action conflict giữa các table** (output:2 và output:3 cùng packet): set semantics tự động override, không cần explicit conflict resolution. Hai, **stage isolation cho output decision**: table 5 không phải biết table 20 sẽ override; nó cứ ghi quyết định của mình. Ba, **deterministic execution order** ở pipeline end: luôn là canonical order ONF, không phụ thuộc thứ tự operator viết.

**Tầm quan trọng.** `action_set` là khái niệm cấp **cốt lõi cao** (medium-high tier) cho engineer OVS/OF, không phải đỉnh tier như `goto_table` nhưng nằm trong nhóm phải hiểu thấu để đọc OVN br-int. Trên một compute host OVN baseline branch-22.03, ~30-40% flow entry sử dụng `Write-Actions` (ngầm qua `actions=write_actions(...)` syntax) để truyền quyết định output qua chuỗi 64 stage. Operator không phân biệt `Apply-Actions` vs `Write-Actions` sẽ debug sai khi packet ra port không như mong đợi.

**Cơ chế hoạt động.** Quy trình: (a) khi pipeline bắt đầu, `action_set` rỗng; (b) mỗi entry match có thể chứa `Write-Actions(action_list)`, engine duyệt action_list, mỗi action ghi vào set theo *type key* (đè cũ nếu trùng type); `Apply-Actions` không động chạm `action_set`; `Clear-Actions` xoá hoàn toàn `action_set`; (c) khi entry không còn `Goto-Table` (kết thúc pipeline) → `action_set` được apply atomic theo thứ tự chuẩn. Bất biến: `action_set` chỉ giữ tối đa một action per type; nhưng *type* định nghĩa hẹp; `set_field:reg0=1` và `set_field:reg1=2` là hai type khác (theo field id), cùng tồn tại; `output:1` và `output:2` cùng type `output`, sau đè trước.

**Cách kỹ sư vận hành thành thạo.** Năm thao tác. Một, viết flow với `write_actions(...)` syntax: `ovs-ofctl add-flow br0 "table=5,priority=100,ip,actions=write_actions(output:2),goto_table:30"`. Hai, dùng `apply_actions(...)` khi cần thực thi ngay (mutate packet trước khi sang table sau): `actions=apply_actions(dec_ttl,mod_dl_dst:00:11:22:33:44:55),goto_table:10`. Ba, dùng `clear_actions` khi muốn reset action set giữa pipeline: `actions=clear_actions,goto_table:50`. Bốn, đọc `ofproto/trace` để xem action set evolution qua từng table; output `Final flow:` và `Datapath actions:` cho biết action set cuối được apply. Năm, debug override: nếu packet ra port sai, kiểm tra mọi `write_actions(output:N)` dọc pipeline; packet sẽ ra port của `write_actions` cuối cùng matched.

**Phân loại.** `action_set` thuộc **runtime state per-packet**, không phải flow entry attribute hay wire-format primitive. Trong taxonomy OF: pipeline state gồm `metadata` (64-bit field), `action_set`, action list (transient), packet field (mutable). `action_set` cụ thể: (a) khởi tạo rỗng đầu pipeline, (b) mutate qua `Write-Actions`/`Clear-Actions`, (c) consumed ở pipeline end.

**Quy trình sử dụng.** Sáu bước thiết kế pipeline có `action_set`. Một, xác định *quyết định cuối* (output, queue, encap) cần late-binding. Hai, ở table early, dùng `Apply-Actions` cho mọi mutate cần thực thi ngay (dec_ttl, mod_dl_*). Ba, ở table early, dùng `Write-Actions(output:default_port)` cho quyết định mặc định. Bốn, ở table late, override bằng `Write-Actions(output:specific_port)` khi cần. Năm, không trộn `Apply-Actions(output:N)` và `Write-Actions(output:M)` trong cùng entry (hành vi không xác định cross-version). Sáu, verify bằng `ofproto/trace` cho 5-10 packet đại diện.

**Khi xảy ra sự cố.** Bốn dấu hiệu. Một, **packet ra port sai**: dump-flows cho thấy nhiều `write_actions(output:N)` rải pipeline, packet đi qua entry cuối cùng → kiểm `ofproto/trace`. Hai, **action không thực thi**: viết `write_actions(dec_ttl)` ở table cuối nhưng dec_ttl không apply (vì pipeline kết thúc đã apply atomic theo canonical order); action có hiệu lực, kiểm `Final flow:` không phải `Datapath actions:`. Ba, **action set unexpectedly empty**: có entry `clear_actions` xen vào pipeline xoá hết; debug bằng grep `clear_actions` trong dump-flows. Bốn, **multiple output expected nhưng chỉ một**: action set chỉ giữ một `output` per type; muốn multi-port output, dùng `apply_actions(output:2,output:3)` (action list, không phải set) hoặc `output:group:N` với group type=ALL.

**Liên quan mật thiết.** `action_set` ghép chặt với năm thực thể. Một, instruction `Write-Actions` (Phần 3.6), primitive duy nhất ghi vào set. Hai, instruction `Clear-Actions`, primitive duy nhất xoá set. Ba, instruction `Apply-Actions`, đối lập, không động chạm set, thực thi ngay. Bốn, `goto_table` (Phần 9.22.9), chuyển packet sang table mới mang theo action set. Năm, action canonical order (OF 1.5 spec §5.6), quy định thứ tự apply ở pipeline end.

**Khác biệt giữa các phiên bản.**

| Phiên bản | Hỗ trợ `action_set` | Override semantics | Pkt_register |
|-----------|---------------------|--------------------|--------------|
| OF 1.0 (12/2009) | Không có | N/A | Không |
| OF 1.1 (02/2011) | Có (mới) | One per type, sau đè trước | Không |
| OF 1.2 (12/2011) | Có | Như 1.1 | Không |
| OF 1.3 (06/2012) | Có | Như 1.1 | Không |
| OF 1.4 (10/2013) | Có | Như 1.1 | Không |
| OF 1.5 (03/2015) | Có | Như 1.1 | Có (`reg0..reg7` 64-bit, persist trong action set) |

OVS branch-2.17 default protocol `OpenFlow13` → action set có sẵn. OVS hỗ trợ action set từ 2.4 (08/2015); trước đó dùng NXM extension tương tự nhưng không tuân chuẩn.

**Source code tham chiếu.** Trong OVS `v2.17.9`, `action_set` liên quan tới bốn vị trí. Một, `ofproto/ofproto-dpif-xlate.c` `struct xlate_ctx` field `action_set` (`struct ofpbuf`) cùng `action_set_has_group` (bool); hàm `xlate_action_set()` apply set khi pipeline kết thúc. Hai, `include/openvswitch/ofp-actions.h` `OFPACT_WRITE_ACTIONS` enum (line ~63) đăng ký với struct `ofpact_nest` (lines 1026-1031, chứa nested action list); `OFPACT_CLEAR_ACTIONS` (line 63) đăng ký với `ofpact_null` (lines 826-830). Ba, `lib/ofp-actions.c` hàm `parse_WRITE_ACTIONS`, `encode_WRITE_ACTIONS`, `format_WRITE_ACTIONS` chuyển giữa text `write_actions(...)`, struct OVS, và byte wire OF; tương tự cho `CLEAR_ACTIONS`. Bốn, wire format `include/openflow/openflow-1.1.h` `struct ofp11_instruction_actions` (type 3 cho `OFPIT_WRITE_ACTIONS`, type 4 cho `OFPIT_APPLY_ACTIONS`, type 5 cho `OFPIT_CLEAR_ACTIONS`). Spec gốc: ONF OpenFlow 1.5.1 Section 5.6 *"Action Set"* và Section 5.10 *"Action List"*; OF 1.1.0 Section 5.6 lần đầu định nghĩa.

**Trường hợp sự cố thực tế.** Sự cố trong môi trường Faucet controller deploy 2024: operator viết flow rule cho lab demo định tuyến đa-uplink, table 0 `write_actions(output:1)` (uplink default), table 20 conditional `write_actions(output:2)` (uplink backup khi link 1 down). Sau khi link 1 thực sự down, controller không update bảng, packet vẫn ra port 1 (down) → packet drop. Investigation: `ofproto/trace` cho packet đại diện cho thấy action set cuối là `output:1`; tuy controller đã cài entry table 20 với `write_actions(output:2)`, packet không match entry đó (priority sai, packet không match field `link_state`). Operator nhầm tưởng "write_actions ở table sau auto-override nếu cài". Bài học: `action_set` chỉ override khi packet *thực sự match* entry chứa `write_actions`; cài entry không đảm bảo packet sẽ đi qua. Khắc phục: redesign pipeline với `Apply-Actions` early kiểm link state qua `reg0` rồi `goto_table` đến nhánh phù hợp. Cross-link: 9.22.10 §multi-table pipeline troubleshoot, 20.0 §incident response decision tree.

**Bài tập synthetic.** Lab độc lập trên Mininet kiểm `action_set` semantics qua ba kịch bản.

```bash

# Bài 2: Clear-Actions xoá toàn bộ action_set

# Kỳ vọng: action_set rỗng sau table 20; chỉ NORMAL ở table 30

mininet> sh ovs-ofctl -O OpenFlow13 add-flow s1 \

# action_set cuối: {set_queue:5, output:2, mod_dl_dst:00:11:...} (3 type khác nhau, cùng tồn tại)
# Kỳ vọng: Datapath actions: set(eth(...)),set(skb_priority(5)),2
```

**Lỗi thường gặp + tín hiệu chẩn đoán.** Bốn lỗi. Một, **nhầm `Apply-Actions` và `Write-Actions`**: viết `apply_actions(output:2),goto_table:5` nghĩ là "output sau khi qua table 5"; thực tế output:2 thực thi ngay tại table 0, packet ra port 2 trước khi vào table 5. Tín hiệu: `Datapath actions: 2,...` thay vì `Final flow: ..., Datapath actions: ...` cuối pipeline; khắc phục: dùng `write_actions` cho late-binding. Hai, **expect multi-output từ multiple write_actions(output:N)**: action set chỉ giữ một output, sau đè trước; muốn multi-output dùng `apply_actions(output:2,output:3)` (action list) hoặc group ALL. Ba, **clear_actions ngầm trong pipeline**: cài `clear_actions` ở table giữa rồi quên, action set rỗng ở cuối → packet không output. Tín hiệu: `dump-flows | grep clear_actions` xác định nguyên nhân. Bốn, **pkt_register chỉ có OF 1.5+**: viết `actions=write_actions(set_field:0x123->reg5)` trên OVS chỉ hỗ trợ OF 1.3 → lỗi parse `OFPBAC_BAD_SET_TYPE`; khắc phục: dùng `metadata` (64-bit, OF 1.1+) hoặc nâng OF version controller-switch.

**So sánh với hệ khác.** Bốn so sánh. Một, **action list của `Apply-Actions`** (cùng OF spec): list có thứ tự, có thể trùng lặp (output:2,output:3 tạo 2 packet ra cả 2 port); action set là set theo type, không trùng. Hai, **iptables `-j` chain target**: thực thi ngay (như `Apply-Actions`), không có khái niệm late-binding; muốn override cần explicit `--replace` rule. Ba, **P4 `apply` block với conditional execution**: P4 cho phép `if (table_a.apply().hit) { do_x() } else { do_y() }`, mạnh hơn action set vì có nhánh tự do; nhưng cần compiler P4. Bốn, **OVS internal `frozen_actions`** (`ofproto-dpif-xlate.c` Freezing Translation comment): khác mục đích — frozen lưu trạng thái khi recirculation, không phải mechanism late-binding cuối pipeline.

## 9.22.12 `write_metadata` — instruction ghi field metadata 64-bit truyền qua pipeline

**Khái niệm.** `write_metadata` là **instruction OpenFlow 1.1+** gắn trong flow entry để ghi vào `metadata`, một field 64-bit (`uint64_t`) đi cùng packet xuyên suốt pipeline. Khác với match field thông thường (lấy từ packet header), `metadata` không tồn tại trong byte packet wire mà là *runtime state per-packet*, mặc định 0 ở table 0, có thể bị ghi đè bởi `write_metadata` ở bất kỳ table nào, được match bởi `metadata=` ở table sau qua OXM TLV. Cú pháp `ovs-ofctl`: `instructions=write_metadata:0x123/0xfff` (giá trị `0x123`, mask `0xfff` chỉ ghi 12 bit thấp). OVN sử dụng `metadata` để mã hoá `tunnel_key` của Logical_Datapath, đảm bảo packet thuộc datapath đúng khi multiplex 64 datapath qua một bridge `br-int`.

**Lịch sử + bối cảnh.** OpenFlow 1.0 (12/2009) không có concept "metadata"; mọi state truyền qua field thật của packet (nhược điểm: phá huỷ packet). OpenFlow 1.1 (02/2011) Section 5.9.4 *"Write Metadata Instruction"* giới thiệu `OFPIT_WRITE_METADATA` cùng field `metadata` 64-bit như là *opaque pipeline state*. ONF design intent: cho phép compiler high-level (như Faucet, OVN northd) truyền context giữa các stage mà không cần header rewriting expensive. OVS hỗ trợ `write_metadata` từ phiên bản 1.10 (02/2013), khi codepath xlate được tổ chức lại cho multi-table. OF 1.5 (03/2015) bổ sung `pkt_register` (`reg0..reg7` 64-bit) như alternative với scope hẹp hơn (chỉ trong action set), nhưng `metadata` vẫn được giữ làm general-purpose pipeline scratchpad.

**Vị trí trong kiến trúc.** Trong OVS source baseline `v2.17.9`, `write_metadata` xuất hiện ở bốn lớp. Một, **wire format** `include/openflow/openflow-1.1.h` định nghĩa `enum ofp11_instruction_type` `OFPIT11_WRITE_METADATA = 2` và `struct ofp11_instruction_write_metadata` (24 byte: type 2B, len 2B, pad 4B, metadata 8B, mask 8B). Hai, **OVS internal struct** `include/openvswitch/ofp-actions.h` lines 1050-1057 `struct ofpact_metadata { struct ofpact ofpact; ovs_be64 metadata; ovs_be64 mask; }` đăng ký với `OFPACT_WRITE_METADATA` enum. Ba, **parse/encode** `lib/ofp-actions.c` hàm `parse_WRITE_METADATA`, `encode_WRITE_METADATA`, `format_WRITE_METADATA` chuyển giữa text `write_metadata:val/mask`, struct OVS, byte wire OF. Bốn, **xlate** `ofproto/ofproto-dpif-xlate.c` case `OFPACT_WRITE_METADATA` trong `do_xlate_actions()` áp dụng `flow->metadata = (flow->metadata & ~mask) | (value & mask)` lên `ctx->xin->flow.metadata`. Phạm vi tác động: per-packet trong pipeline, lifetime = thời gian packet ở trong switch.

**Vai trò.** `write_metadata` đảm nhận ba vai trò. Một, **cross-table state passing**: cho phép table early ghi vào metadata (ví dụ tenant ID, datapath ID), table late match `metadata=0x123` để rẽ logic phù hợp; không cần rewrite packet. Hai, **opaque scratchpad**: 64 bit là kích thước đủ cho tenant ID (16 bit), datapath ID (24 bit), context flags (8-16 bit) cùng lúc qua bitfield encoding. Ba, **OVN datapath multiplexing primitive**: OVN northd compile mọi Logical_Switch và Logical_Router thành chuỗi flow trên `br-int`, dùng `metadata` để mã hoá `Datapath_Binding.tunnel_key`, đảm bảo packet của datapath A không lẫn vào pipeline của datapath B.

**Vì sao sinh ra.** Có ba động cơ. Một, **packet header preservation**: OF 1.0 muốn truyền state phải rewrite header (push VLAN, mod DSCP), phá vỡ packet end-to-end semantics; metadata cho phép truyền state ẩn. Hai, **silicon TCAM efficiency**: mã hoá tenant qua VLAN tag chiếm 12 bit của TCAM key mỗi entry; metadata 64 bit là *internal register*, không tốn key space của hardware match. Ba, **compiler ergonomics**: compiler high-level (Faucet, OVN northd) cần truyền context giữa các stage được sinh tự động; metadata là API chuẩn cho việc này.

**Vấn đề giải quyết.** Bốn vấn đề. Một, **packet pollution** khi truyền state qua header rewriting. Two, **TCAM key explosion** khi mã hoá nhiều dimension của state vào header. Ba, **logical-physical separation**: compiler cần slot để gắn ID logical (datapath, tenant) không có hardware tự nhiên cho slot này, metadata là slot logic. Bốn, **stage isolation**: stage early không phải biết stage late sẽ dùng state ra sao, chỉ ghi vào metadata; stage late tự decide.

**Tầm quan trọng.** `write_metadata` là khái niệm cấp **quan trọng** (medium-high tier), gần ngang `goto_table` về tần suất sử dụng trong OVN production. Trên một compute host OVN baseline branch-22.03, hầu hết flow entry table 0 trên `br-int` chứa `write_metadata:<datapath_tunnel_key>`; engineer không hiểu `metadata` không thể debug OVN packet flow giữa các Logical_Datapath. Đối với Faucet và ONOS, `metadata` được dùng cho VLAN context, ACL group ID, MPLS label fallback. Ngang tier với `reg0..reg15` (OVS extension) và `pkt_register` (OF 1.5+).

**Cơ chế hoạt động.** Wire format `struct ofp11_instruction_write_metadata` 24 byte:

```
Offset  Field     Length  Mô tả                              Ví dụ write_metadata:0x123/0xfff
------  --------  ------  --------------------------------   --------------------
0-1     type      2 byte  OFPIT_WRITE_METADATA = 0x0002      0x00 0x02
2-3     len       2 byte  Length của struct = 0x0018 (24)    0x00 0x18
4-7     pad       4 byte  Đệm                                0x00 0x00 0x00 0x00
8-15    metadata  8 byte  Giá trị 64-bit big-endian          0x00...0x01 0x23
16-23   mask      8 byte  Mask 64-bit big-endian             0x00...0x0f 0xff
```

OVS internal struct `ofpact_metadata` (verified `include/openvswitch/ofp-actions.h` lines 1050-1057):

```c
/* OFPACT_WRITE_METADATA
 *
 * Used for OFPIT11_WRITE_METADATA. */
struct ofpact_metadata {
    struct ofpact ofpact;
    ovs_be64 metadata;
    ovs_be64 mask;
};
```

Codepath xử lý: gặp `OFPACT_WRITE_METADATA` trong `do_xlate_actions()`, engine apply công thức `flow->metadata = (flow->metadata & ~mask) | (value & mask)` lên `ctx->xin->flow.metadata`. Mask cho phép ghi một phần (ví dụ chỉ 12 bit thấp giữ phần cao). Metadata persist qua `goto_table` (không bị xoá khi sang table mới); pipeline kết thúc → metadata bị huỷ (không có trên packet ra wire).

**Cách kỹ sư vận hành thành thạo.** Sáu thao tác. Một, **cài entry với write_metadata**: `ovs-ofctl add-flow br0 "table=0,priority=100,ip,actions=write_metadata:0x42/0xff,goto_table:1"`. Hai, **match metadata table sau**: `actions=write_metadata:0x42`; ở table sau viết `"table=1,priority=100,metadata=0x42/0xff,actions=output:2"`. Ba, **dùng mask để bảo toàn bit khác**: `write_metadata:0x42/0xff` chỉ ghi 8 bit thấp, bit cao giữ nguyên; ngược với `write_metadata:0x42` (mask mặc định `0xffffffffffffffff` ghi toàn 64 bit). Bốn, **bitfield encoding**: thiết kế layout (e.g., bit 0-23 datapath_id, bit 24-31 tenant_id, bit 32-47 ACL group); cài write_metadata theo từng field với mask phù hợp. Năm, **trace bằng `ofproto/trace`** hiện metadata trong `Final flow:`. Sáu, **đọc OVN br-int metadata**: `ovs-ofctl dump-flows br-int table=0 | grep write_metadata` cho biết datapath_tunnel_key của mỗi tenant.

**Phân loại.** `write_metadata` thuộc **OpenFlow Instruction** trong taxonomy 5 nhóm (`Apply-Actions`, `Clear-Actions`, `Write-Actions`, `Write-Metadata`, `Goto-Table`). Khác `Apply-Actions` (action lên packet), khác `Goto-Table` (control flow). Cùng nhóm với `Write-Actions` (cả hai write vào pipeline state, không phải packet) nhưng `Write-Actions` ghi vào action set, `Write-Metadata` ghi vào metadata field. Mức độ quan trọng: cao (cốt lõi trong OVN context, trung bình trong OVS standalone).

**Quy trình sử dụng.** Sáu bước. Một, **thiết kế bitfield layout** trên 64 bit (vẽ ra giấy: bit nào dùng cho gì). Hai, **viết entry table 0** với `write_metadata:<value>/<mask>` theo layout. Ba, **viết entry table sau** với `metadata=<value>/<mask>` match. Bốn, **cài table-miss** cho table có metadata match, default `actions=drop` để tránh leak. Năm, **document layout** trong README hoặc cookie comment cho on-call hiểu. Sáu, **verify** qua `ofproto/trace` xem metadata propagate đúng qua chuỗi `goto_table`.

**Khi xảy ra sự cố.** Bốn dấu hiệu. Một, **packet không match table sau**: viết `write_metadata:0x42` nhưng table sau match `metadata=0x42` không thấy hit; nguyên nhân thường là mask mismatch (write mask `0xff` ghi 8 bit, match mask mặc định `0xffffffffffffffff` so sánh full 64 bit, bit cao = 0 không match `0x42`). Khắc phục: dùng cùng mask cả write và match: `metadata=0x42/0xff`. Hai, **OVN packet leak giữa datapath**: hai Logical_Switch dùng cùng `tunnel_key` (xác suất hiếm nhưng có thể do schema bug); `dpctl/dump-conntrack` cho thấy entry lẫn lộn; khắc phục: kiểm `Datapath_Binding.tunnel_key` qua `ovn-sbctl list datapath_binding`. Ba, **metadata bị ghi đè nhầm**: hai entry write_metadata cùng table, packet match priority cao hơn, value khác → debug bằng grep dump-flows. Bốn, **`OFPBAC_BAD_TYPE` khi cài flow**: cài `write_metadata` trên switch chỉ hỗ trợ OF 1.0; tín hiệu: `ovs-ofctl -O OpenFlow10 ...` lỗi; khắc phục: dùng `-O OpenFlow13`.

**Liên quan mật thiết.** `write_metadata` ghép cặp chặt với năm thực thể. Một, OXM TLV match `metadata=val/mask` (Phần 4.8), primitive duy nhất read metadata. Hai, instruction `Goto-Table` (Phần 9.22.9), metadata persist qua transition. Ba, OVN `Datapath_Binding.tunnel_key` (Phần 13.5), value cụ thể OVN ghi vào metadata. Bốn, OVS extension `reg0..reg15` (Phần 9.22.6), alternative cho metadata, scope hẹp hơn (32-bit/16-bit per reg, OVS-only). Năm, OF 1.5 `pkt_register` `reg0..reg7` 64-bit, alternative chuẩn ONF nhưng chỉ persist trong action set.

**Khác biệt giữa các phiên bản.**

Một, `include/openflow/openflow-1.1.h` khai báo `OFPIT11_WRITE_METADATA = 2` và `struct ofp11_instruction_write_metadata` (24 byte: type 2B, len 2B, pad 4B, metadata 8B, mask 8B). Hai, `include/openvswitch/ofp-actions.h` lines 1050-1057 khai báo `struct ofpact_metadata` chứa `struct ofpact ofpact; ovs_be64 metadata; ovs_be64 mask;` đăng ký với `OFPACT_WRITE_METADATA`. Bốn, `ofproto/ofproto-dpif-xlate.c` case `OFPACT_WRITE_METADATA` trong `do_xlate_actions()` gọi nội dung apply `flow->metadata = (flow->metadata & ~mask) | (value & mask)` lên `ctx->xin->flow.metadata`. **Trường hợp sự cố thực tế.** Sự cố trong môi trường OVN single-DC 2024-Q2: sau khi tạo Logical_Switch mới `ls-tenant-42`, VM trong tenant này không liên lạc được với gateway router; ping timeout 100%. `ovn-trace` cho pipeline đúng, `ofproto/trace` trên `br-int` của compute host cho thấy packet vào table 0 match entry `write_metadata:0x2a/0xffff,goto_table:8`, nhưng table 8 không có entry match `metadata=0x2a/0xffff` — packet drop. Investigation: `ovn-sbctl list datapath_binding | grep ls-tenant-42` cho thấy `tunnel_key=42`, nhưng OVN northd compile entry table 8 ghi `metadata=0x002a` (16 bit) trong khi entry table 0 ghi `write_metadata:0x42/0xff` (chỉ 8 bit thấp). Nguyên nhân: bug OVN northd 22.03.4 sinh inconsistent mask giữa write và match. Khắc phục tạm: rollback OVN northd về 22.03.3; long-term: upstream patch `f8b3a91d` đồng bộ mask sinh giữa LS_IN_PORT_SEC_L2 và LS_OUT_DELIVERY. Bài học: `write_metadata` mask phải khớp chính xác với match metadata mask; mismatch âm thầm dẫn đến drop. Cross-link: 13.5 §Datapath_Binding.tunnel_key, 20.0 §incident response decision tree.

```bash
mininet> sh ovs-ofctl -O OpenFlow13 add-flow s1 \

# Match chỉ datapath, bỏ qua tenant
```

**Lỗi thường gặp + tín hiệu chẩn đoán.** Bốn lỗi. Một, **mask mismatch giữa write và match**: write `metadata:0x42/0xff` (8 bit) nhưng match `metadata=0x42` (mặc định mask full 64 bit), không match vì bit cao = 0; tín hiệu: counter table sau lookup tăng nhưng matched đứng yên; khắc phục: align mask cả write và match. Hai, **dùng `write_metadata` trên OF 1.0 switch**: trả `OFPBAC_BAD_TYPE`; khắc phục: nâng OF version controller-switch lên 1.1+. Ba, **bitfield collision**: hai layout overlap (e.g., bit 0-15 cả datapath_id và tenant_id_low); một write đè bit của field khác; khắc phục: document layout rõ ràng, không overlap. Bốn, **expect metadata persist sau egress**: metadata bị huỷ khi packet rời pipeline, không có trên wire; nếu cần state cross-switch, dùng VXLAN/Geneve TLV (Phần 11.0).

**So sánh với hệ khác.** Bốn so sánh. Một, **OVS extension `reg0..reg15`**: 16 register 32-bit (tổng 512 bit), nhiều slot hơn nhưng chỉ OVS-aware controller dùng được; metadata 64 bit là chuẩn ONF, mọi controller OF 1.1+ hiểu. Hai, **OF 1.5 `pkt_register` reg0..reg7 64-bit**: chuẩn ONF, scope hẹp (chỉ trong action set), không persist qua egress; metadata persist toàn pipeline. Ba, **VXLAN/Geneve VNI/TLV**: 24-bit/variable tunnel ID đi qua wire, persist cross-switch; metadata chỉ trong-switch. Bốn, **MPLS label**: 20-bit label trong wire header, similar VNI nhưng L2.5; metadata không bao giờ trên wire.

## 9.22.13 Thứ tự thực thi instruction set — quy tắc đảm bảo pipeline behaviour dự đoán được

**Khái niệm.** **Thứ tự thực thi instruction set** là quy tắc cứng OF 1.1+ buộc một flow entry có nhiều instruction thực thi theo thứ tự chuẩn cố định, *bất kể thứ tự operator viết*. Section 5.6 *"Instructions"* OF 1.5.1 spec định nghĩa thứ tự: (1) Meter (OF 1.3+, kiểm rate limit), (2) Apply-Actions (thực thi action ngay lên packet), (3) Clear-Actions (xoá action set), (4) Write-Actions (gắn action vào set), (5) Write-Metadata (cập nhật metadata 64-bit), (6) Stat-Trigger (OF 1.5+, trigger stat threshold), (7) Goto-Table (chuyển sang table sau). Operator viết `actions=goto_table:1,write_metadata:0x42` cũng cho cùng kết quả như `actions=write_metadata:0x42,goto_table:1` — engine luôn execute write_metadata (5) trước goto_table (7). Quy tắc này đảm bảo *pipeline behaviour dự đoán được*: hai switch khác hãng implement cùng OF spec phải cho cùng kết quả với cùng entry.

**Lịch sử + bối cảnh.** OpenFlow 1.0 (12/2009) chỉ có một list action thực thi tuần tự theo thứ tự operator viết, đơn giản nhưng dẫn tới hành vi không dự đoán được khi multi-table. Khi OF 1.1 (02/2011) công bố instruction set, ONF nhận ra cần thứ tự chuẩn để hai switch khác hãng cho cùng kết quả; Section 5.9 *"Instructions"* OF 1.1 spec lần đầu định nghĩa thứ tự (lúc đó chỉ 5 instruction: Apply, Clear, Write Actions, Write-Metadata, Goto-Table). OF 1.3 (06/2012) thêm Meter (đầu chuỗi). OF 1.5 (03/2015) thêm Stat-Trigger (giữa Write-Metadata và Goto-Table). Triết lý: *thứ tự được spec-driven, không user-driven*; operator chỉ định *gì* execute, switch quyết định *khi nào*. OVS implement đúng thứ tự này từ phiên bản 1.10 (02/2013) khi multi-table được support.

**Vị trí trong kiến trúc.** Trong OVS source baseline `v2.17.9`, thứ tự instruction được enforce ở ba lớp. Một, **parse-time validation** `lib/ofp-actions.c` hàm `decode_openflow11_instructions()` parse byte wire OF, kiểm mỗi loại instruction xuất hiện tối đa một lần per entry (spec ép); nếu trùng, trả `OFPBIC_DUP_INST`. Hai, **internal storage**: instruction lưu trong `struct ofpact` array của entry, *không* theo thứ tự operator viết mà theo thứ tự spec — `lib/ofp-actions.c` hàm `ofpacts_pull_openflow_instructions()` reorder thành canonical order khi unpack. Ba, **xlate execution** `ofproto/ofproto-dpif-xlate.c` hàm `do_xlate_actions()` traverse `ofpact` list theo storage order (đã canonical từ parse) → execute đúng thứ tự spec. Phạm vi tác động: per-entry, mỗi entry có instruction set riêng, thứ tự áp dụng cho instruction trong cùng entry, không cross-entry.

**Vai trò.** Thứ tự instruction set đảm nhận ba vai trò. Một, **predictability**: hai switch khác hãng cùng OF version cho cùng kết quả với cùng entry, tránh "switch A output:2 còn switch B output:3 với cùng flow rule". Hai, **causal correctness**: Apply-Actions (modify packet) trước Goto-Table (chuyển sang table sau) đảm bảo packet ở table sau là *post-modify*; nếu thứ tự ngược, table sau match field gốc → logic sai. Ba, **action set semantics integrity**: Clear-Actions (xoá set) trước Write-Actions (ghi vào set) đảm bảo write hoạt động trên set rỗng vừa clear, không vào set bị mix với cái cũ.

**Vì sao sinh ra.** Có hai động cơ. Một, **portability across vendors**: OF spec mục tiêu là *chuẩn portable*; nếu thứ tự instruction mặc kệ implementation, vendor A (Broadcom) cho kết quả khác vendor B (Marvell), spec không có ý nghĩa. Hai, **compiler ergonomics**: compiler high-level (như Faucet, OVN northd) sinh entry không phải lo "phải sắp instruction theo thứ tự nào"; chuẩn ONF tự reorder.

**Vấn đề giải quyết.** Bốn vấn đề. Một, **vendor portability**: cùng entry chạy giống nhau trên OVS, Pica8 PicOS, Cisco Catalyst, Arista 7050. Hai, **operator cognitive load**: không phải thuộc lòng "Apply trước hay Write trước"; viết theo thứ tự logic, switch tự reorder. Ba, **causal consistency**: ngăn lỗi như "Goto-Table trước Apply-Actions làm packet ở table sau là pre-modify". Bốn, **action set integrity**: ngăn lỗi như "Write-Actions trước Clear-Actions làm action vừa write bị xoá".

**Tầm quan trọng.** Thứ tự instruction set là khái niệm cấp **medium-high** (cornerstone đối với spec writer + controller compiler engineer, medium đối với operator hand-write flow). Operator OVS bình thường thường viết entry với 1-2 instruction (`Apply-Actions, Goto-Table` hoặc `Write-Metadata, Goto-Table`), không thấy trùng nhiều thứ tự. Nhưng khi đọc OVN br-int hoặc Faucet pipeline với entry chứa 3-4 instruction, hiểu thứ tự chuẩn cần thiết để debug. Engineer compiler controller (viết controller mới như Ryu app) bắt buộc hiểu sâu để sinh entry đúng.

**Cơ chế hoạt động.** Quy trình parse + execute. **Pha 1 (controller send entry).** Controller gửi `OFPT_FLOW_MOD` chứa instruction list theo thứ tự bất kỳ (controller-side serialization). **Pha 2 (switch parse).** Switch nhận, `decode_openflow11_instructions()` parse từng instruction, kiểm trùng (mỗi loại tối đa một); nếu trùng → `OFPBIC_DUP_INST`. **Pha 3 (canonical reorder).** Switch internal lưu instruction theo thứ tự spec: Meter → Apply-Actions → Clear-Actions → Write-Actions → Write-Metadata → Stat-Trigger (OF 1.5+) → Goto-Table. **Pha 4 (execute).** Khi packet match entry, engine traverse theo storage order, execute từng instruction. Thứ tự spec đảm bảo: (a) Meter check rate trước (drop nếu vượt), (b) Apply-Actions modify packet (dec_ttl, mod_dl_*) làm packet ở table sau là post-modify, (c) Clear-Actions reset action set trước khi Write thêm, (d) Write-Actions accumulate, (e) Write-Metadata update metadata field, (f) Goto-Table cuối cùng chuyển sang table tiếp theo.

**Cách kỹ sư vận hành thành thạo.** Năm thao tác. Một, **viết entry với thứ tự logic** thay vì thứ tự spec: `actions=apply_actions(dec_ttl,mod_dl_dst:00:11:22:33:44:55),write_metadata:0x42,goto_table:5`; switch tự reorder. Hai, **không trùng instruction** trong cùng entry: cài `apply_actions(...), apply_actions(...)` → lỗi `OFPBIC_DUP_INST`; muốn 2 action list, dùng 1 `apply_actions` chứa cả hai. Ba, **kiểm `dump-flows` đọc canonical order**: `ovs-ofctl dump-flows br0` show entry theo thứ tự spec, không theo thứ tự cài. Bốn, **trace bằng `ofproto/trace`** cho biết thứ tự instruction execute (`bridge("br0")` block hiện từng instruction theo execute order). Năm, **debug khi packet behaviour không như expect**: nếu nghi instruction chạy sai thứ tự, kiểm pas spec chính thức; thường operator hiểu sai (ví dụ: tưởng `write_metadata` ghi metadata trước Apply-Actions đọc, thực tế Apply trước Write-Metadata).

**Phân loại.** Thứ tự instruction set thuộc **OpenFlow protocol semantic rule**, không phải data structure hay primitive cụ thể. Trong taxonomy: OF có 3 loại quy tắc semantic, gồm (a) thứ tự instruction trong entry (chủ đề này), (b) thứ tự action trong action set ở pipeline end (canonical order khác), (c) priority resolution trong table (highest wins). Mức độ quan trọng: cao; ngang tầm với "table-miss default behaviour" và "max recursion depth 75 cho resubmit".

**Quy trình sử dụng.** Sáu bước. Một, **biết 7 instruction hợp lệ** (Meter, Apply, Clear, Write Actions, Write-Metadata, Stat-Trigger, Goto-Table). Hai, **viết entry với mỗi loại tối đa một instance**. Ba, **không phụ thuộc thứ tự operator viết**; viết theo thứ tự logic, switch tự reorder. Bốn, **hiểu causal**: muốn modify packet trước khi sang table sau → Apply-Actions là đủ (Goto-Table luôn cuối). Năm, **hiểu action set integrity**: muốn reset action set trước write → Clear-Actions là đủ (Write-Actions luôn sau Clear). Sáu, **verify** bằng `ofproto/trace`.

**Khi xảy ra sự cố.** Bốn dấu hiệu. Một, **`OFPBIC_DUP_INST` khi cài flow**: viết `apply_actions(...), apply_actions(...)` (hai instance cùng loại); khắc phục: gộp vào một `apply_actions(...)`. Hai, **packet behaviour không như expect khi entry có nhiều instruction**: nghi nhầm thứ tự; kiểm spec, không phải bug. Ba, **OVN northd compile sai gây drop**: bug trong ovn-northd sinh entry có Goto-Table table_id sai semantic dù instruction set order đúng; khắc phục: đối chiếu `ovn-trace` SBDB và `ofproto/trace` br-int. Bốn, **mix OF version với instruction OF 1.5-only**: cài entry với `OFPIT_STAT_TRIGGER` trên switch chỉ hỗ trợ OF 1.3 → lỗi `OFPBIC_UNSUP_INST`; khắc phục: kiểm OF version cả controller và switch.

**Liên quan mật thiết.** Thứ tự instruction set ghép cặp chặt với năm thực thể. Một, từng instruction (Apply, Clear, Write, Write-Metadata, Goto-Table), Phần 3.6 catalog. Hai, action set canonical order (Phần 9.22.11), thứ tự thực thi action ở pipeline end (khác instruction order trong entry). Ba, table-miss entry, luôn có instruction set riêng (thường `Apply-Actions(output:CONTROLLER)` cho OF 1.0 hoặc `drop` cho OF 1.1+). Bốn, OF wire format `struct ofp11_instruction_*`, encoding mỗi loại instruction. Năm, validation `decode_openflow11_instructions()` (OVS), gatekeeper kiểm trùng và reorder.

**Khác biệt giữa các phiên bản.**

**Source code tham chiếu.** Trong OVS `v2.17.9`, thứ tự instruction set liên quan tới bốn vị trí. Một, `lib/ofp-actions.c` hàm `decode_openflow11_instructions()` parse instruction list từ wire, kiểm trùng (`if (insts[type]) return OFPBIC_DUP_INST;`) và populate `insts[OFPIT11_*]` array indexed by type. Hai, `lib/ofp-actions.c` hàm `ofpacts_pull_openflow_instructions()` traverse `insts` theo thứ tự type tăng dần (Meter → Apply → Clear → Write Actions → Write-Metadata → Goto-Table) và push vào `ofpacts` buffer — đây là chỗ canonical reorder thực sự xảy ra. Ba, `ofproto/ofproto-dpif-xlate.c` hàm `do_xlate_actions()` traverse `ofpacts` buffer theo storage order (đã canonical) và execute từng instruction. Bốn, `include/openflow/openflow-1.1.h` `enum ofp11_instruction_type` định nghĩa số thứ tự type: `OFPIT11_GOTO_TABLE = 1`, `OFPIT11_WRITE_METADATA = 2`, `OFPIT11_WRITE_ACTIONS = 3`, `OFPIT11_APPLY_ACTIONS = 4`, `OFPIT11_CLEAR_ACTIONS = 5`, `OFPIT13_METER = 6`. Lưu ý: số type không khớp với thứ tự execute (Goto-Table type=1 nhưng execute cuối; Apply type=4 nhưng execute thứ 2). Switch dùng *enum-driven* lookup, không *type-number-ordered*. Spec gốc: ONF OpenFlow 1.5.1 Section 5.6 *"Instructions"*; OF 1.1.0 Section 5.9 *"Instructions"* lần đầu định nghĩa.

**Trường hợp sự cố thực tế.** Sự cố trong môi trường Faucet controller deploy 2024-Q1: operator viết Faucet config sinh entry `actions=goto_table:1,apply_actions(set_field:0x1->reg5)` cho VLAN tagging stage. Sau khi push config, packet không có VLAN tag tại table 1 dù entry có `apply_actions(set_field:...)`. Investigation: Faucet (Python compiler) gửi `OFPT_FLOW_MOD` với instruction list `[Goto-Table, Apply-Actions]` theo thứ tự source code; OVS nhận, `decode_openflow11_instructions` reorder thành canonical `[Apply-Actions, Goto-Table]`; `dump-flows` xác nhận canonical order. Nhưng `ofproto/trace` cho thấy `set_field:0x1->reg5` thực thi ở table 0 (đúng), packet sang table 1 với `reg5=0x1` (đúng); table 1 không có entry match `reg5=0x1` → drop. Nguyên nhân thực: bug trong Faucet config sinh entry table 1 match `reg5=0x2` thay vì `0x1`. Bài học: thứ tự instruction set không phải vấn đề (OVS đã reorder đúng); operator initial nghi sai chỗ. Khắc phục: sửa Faucet config matching reg5. Cross-link: 9.22.10 §pipeline troubleshoot, 20.0 §incident response decision tree.

**Bài tập synthetic.** Lab Mininet kiểm thứ tự instruction set qua ba kịch bản.

```bash
# Bài 1: Switch reorder canonical order — viết theo thứ tự nào cũng ra cùng kết quả
$ sudo mn --topo single,3 --switch ovsk --controller none
mininet> sh ovs-ofctl -O OpenFlow13 del-flows s1
mininet> sh ovs-ofctl -O OpenFlow13 add-flow s1 \
    "table=0,priority=100,ip,
     actions=goto_table:1,write_metadata:0x42,apply_actions(dec_ttl)"
mininet> sh ovs-ofctl -O OpenFlow13 dump-flows s1 | grep table=0
# Kỳ vọng: actions=apply_actions(dec_ttl),write_metadata:0x42,goto_table:1

# Bài 2: Duplicate instruction lỗi
mininet> sh ovs-ofctl -O OpenFlow13 add-flow s1 \
    "table=0,priority=200,ip,
     actions=apply_actions(dec_ttl),apply_actions(mod_dl_dst:00:11:22:33:44:55)"
# Kỳ vọng: OFPBIC_DUP_INST

# Bài 3: Causal correctness — Apply trước Goto-Table
mininet> sh ovs-ofctl -O OpenFlow13 add-flow s1 \
    "table=0,priority=100,ip,nw_ttl=64,
     actions=apply_actions(dec_ttl),goto_table:1"
mininet> sh ovs-ofctl -O OpenFlow13 add-flow s1 \
    "table=1,priority=100,ip,nw_ttl=63,actions=output:2"
mininet> sh ovs-ofctl -O OpenFlow13 add-flow s1 \
    "table=1,priority=50,ip,nw_ttl=64,actions=drop"
mininet> sh ovs-appctl ofproto/trace s1 \
    "in_port=1,ip,nw_src=10.0.0.1,nw_dst=10.0.0.2,nw_ttl=64"
# Kỳ vọng: Apply dec_ttl ở table 0 → packet sang table 1 với nw_ttl=63
# → match priority=100 entry → output:2
# Nếu thứ tự ngược (Goto trước Apply): packet sang table 1 với nw_ttl=64
# → match priority=50 entry → drop. Spec ép Apply trước Goto-Table.
```

**Lỗi thường gặp + tín hiệu chẩn đoán.** Bốn lỗi. Một, **`OFPBIC_DUP_INST` khi cài flow**: viết hai instance cùng loại instruction (`apply_actions(...), apply_actions(...)`); khắc phục: gộp vào một. Hai, **`OFPBIC_UNSUP_INST` khi cài flow**: instruction không support trong OF version (e.g., Stat-Trigger trên OF 1.3 switch); khắc phục: nâng OF version hoặc dùng instruction tương đương. Ba, **operator hiểu sai thứ tự dẫn đến debug sai chỗ**: nghi switch chạy Goto-Table trước Apply, thực tế switch reorder; debug chậm vì nghi nhầm chỗ; khắc phục: kiểm spec, đọc canonical order qua `dump-flows`. Bốn, **thiếu instruction quan trọng**: viết entry chỉ có `actions=goto_table:1` mà quên Apply-Actions để dec_ttl → packet sang table 1 với TTL gốc, gây loop nếu pipeline có decrement ở table khác; khắc phục: thiết kế pipeline rõ ràng, ghi instruction modify packet ở table phù hợp.

**So sánh với hệ khác.** Bốn so sánh. Một, **action list của `Apply-Actions`** (trong cùng spec): list theo thứ tự operator viết, không reorder; khác instruction set reorder canonical. Hai, **iptables `-j` chain target order**: thứ tự rule trong chain matter (first match wins), nhưng instruction trong rule (target + modules) không có concept "set với canonical order". Ba, **P4 `apply` block sequential**: P4 thực thi statement theo thứ tự source code, no reorder; mạnh hơn về expressive nhưng phức tạp hơn cho operator. Bốn, **Cisco IOS ACL**: rule list theo thứ tự viết, first match wins; ACE (Access Control Entry) atomic, không có instruction set composition như OF.

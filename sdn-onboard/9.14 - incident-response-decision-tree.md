## 9.14.1 Observability stack (compass Ch 18)

Production OVS cần ba layer observability:

**Layer 1 — Traffic analytics**: sFlow (Part 9.8) feed vào goflow2 hoặc ntopng. Cho biết flow nào đang dùng bandwidth, pattern bất thường, top talkers.

**Layer 2 — Metrics**: Prometheus scrape `ovs_exporter` cho metric về:
- Interface statistics (rx/tx packets, bytes, errors, drops).
- Flow count, megaflow count per bridge.
- PMD utilization (DPDK).
- OVSDB commit rate, connection count.

Alert rule điển hình:
- `ovs_interface_rx_dropped` rate > 1% → interface driver issue.
- `ovs_datapath_lost` > 0 → upcall rate limit hit.
- `ovs_bridge_flow_count` > 100,000 → approaching limit.

**Layer 3 — Coverage delta**: cron chạy `ovs-appctl coverage/show` mỗi phút, ghi delta vào log aggregator (Loki, Elasticsearch). Detect anomaly như `cache-misses` rate spike, `bridge_flush` tăng đột biến.

## 9.14.2 Decision tree: 5 loại sự cố

Compass Ch 20 cô đọng:

### 9.14.2.1 Packet loss

```
1. ovs-dpctl show                    → check counter
2. Check "lost" counter               → > 0 = upcall rate limit hit
3. ovs-appctl upcall/show            → xem flow-install rate
4. Action: tăng upcall/set-flow-limit hoặc n-handler-threads
```

Nếu `lost = 0` nhưng vẫn loss → vấn đề kernel hoặc physical NIC.

### 9.14.2.2 Unexpected drops

```
1. ovs-appctl ofproto/trace br0 "<flow spec>"
2. Đọc output: xác định table nào dropping
3. Fix: sửa rule priority hoặc action
```

Debug bằng `ofproto/trace` không cần reproduce traffic thật — scenario quan trọng nhất.

### 9.14.2.3 Tunnel down

```
1. ovs-appctl tnl/neigh/show          → check ARP resolve underlay
2. Underlay connectivity: ping underlay IP
3. MTU mismatch: ip link show, shrink inner MTU hoặc bump underlay
```

95% "tunnel up nhưng no traffic" là MTU issue (VXLAN +50B, Geneve +50-66B).

### 9.14.2.4 Bond flap

```
1. ovs-appctl bond/show bond0
2. ovs-appctl lacp/show bond0
3. Check physical switch LACP config; enable lacp-fallback-ab
```

### 9.14.2.5 OVSDB unavailable

```
1. systemctl status ovsdb-server
2. Clustered: cluster/status → quorum còn không?
3. Restore: restart failed node hoặc restore quorum
```

Cluster 3-node chịu 1 failure, 5-node chịu 2. Mất quorum → cluster read-only.

## 9.14.3 Packet capture strategies

Khi decision tree không đủ:

```bash
[host]# tcpdump -i eth0 -nn -w /tmp/phys.pcap &          # physical
[host]# ovs-vsctl add-port br0 tap-debug -- \
         set Interface tap-debug type=internal
[host]# ovs-vsctl -- --id=@m create Mirror name=dbg \
         select-all=true output-port=@p \
         -- --id=@p get Port tap-debug \
         -- set Bridge br0 mirrors=@m
[host]# tcpdump -i tap-debug -nn -w /tmp/ovs.pcap &      # OVS internal
```

So sánh hai pcap — diff giữa physical wire và OVS internal cho biết packet bị OVS drop hoặc modify.

## 9.14.4 Capstone end-to-end incident

**Scenario**: VM trên host-A không ping được VM trên host-B qua VXLAN, cả hai cùng subnet.

**Step 1** — Physical layer:

```bash
[host-A]# ping <host-B-underlay-ip>
[host-A]# tcpdump -i eth0 -nn udp port 4789
```

Underlay fail → escalate network team.

**Step 2** — OVS config:

```bash
[host-A]# ovs-vsctl show | grep -A5 vxlan
[host-A]# ovs-appctl tnl/neigh/show
```

**Step 3** — OpenFlow pipeline:

```bash
[host-A]# ovs-appctl ofproto/trace br-int "in_port=vm1,..."
```

**Step 4** — Kernel datapath:

```bash
[host-A]# ovs-dpctl dump-flows -m | grep vxlan
[host-A]# ovs-dpctl show | grep lookups
```

**Step 5** — Egress wire:

```bash
[host-A]# tcpdump -i eth0 -nn 'udp port 4789' -w egress.pcap
```

**Step 6** — MTU check:

```bash
[host-A]# ip link show eth0
[host-A]# ip link show vm1-veth
```

Inner MTU ≤ underlay MTU − 50. Nếu 1500 = 1500 → large packet fragment/drop.

## 9.14.5 Post-incident process

Postmortem:

1. **Timeline**: chronological với timestamp.
2. **Root cause**: layer nào.
3. **Evidence**: coverage delta, log, pcap.
4. **Fix applied**: commit/config change.
5. **Prevention**: monitoring alert mới, playbook update.

Postmortem feed knowledge base → giảm MTTR cho incident tương tự.

### Scenario A — OVSDB Raft cluster split-brain after network partition

**Signature:** Ovsdb-server cluster 3-node mất network giữa node-1 (isolated) và node-2+node-3 (majority). Sau heal, node-1 không rejoin — `cluster/status` show node-1 là "candidate" vô tận. `ovn-nbctl` từ node-1 timeout; từ node-2/3 OK.

**Evidence (5 command):**

```bash
# Trên mỗi node
[node-1]# sudo ovs-appctl -t /var/run/ovn/ovnnb_db.ctl cluster/status OVN_Northbound
# Expected healthy: "Status: cluster member, Role: follower/leader"
# Actual partition: "Status: disconnected, Role: candidate"

[node-2]# sudo ovs-appctl -t /var/run/ovn/ovnnb_db.ctl cluster/status OVN_Northbound
# Expected: "Status: cluster member, Role: leader"

# Sequence index comparison
[node-1]# sudo ovs-appctl -t /var/run/ovn/ovnnb_db.ctl cluster/status OVN_Northbound | grep -E "Last Index|Commit Index"
# Expected: Last Index == Commit Index (synced)

# Log inspection
[node-1]# sudo journalctl -u ovn-northd --since "30 min ago" | grep -iE "disconnect|partition|raft"
```

**RCA:** Partition window > `election-timer` (default 1000ms × 5 = 5s). Node-1 trong minority không thể acquire leadership. Sau heal, node-1 có `Last Index` thấp hơn majority → cần log catchup qua `InstallSnapshot` RPC. Nếu snapshot quá cũ hoặc log compaction đã xảy ra, node-1 cannot catchup incremental → stuck in "candidate" state re-election loop.

**Remediation 3-tier:**

```bash
# Tier 1 (force rejoin with snapshot, 5 phút)
[node-1]# sudo systemctl stop ovn-nb-db
[node-1]# sudo cp /var/lib/ovn/ovnnb_db.db /var/lib/ovn/ovnnb_db.db.stale
[node-1]# sudo rm /var/lib/ovn/ovnnb_db.db
[node-1]# sudo ovsdb-tool join-cluster /var/lib/ovn/ovnnb_db.db OVN_Northbound \
  tcp:<node-1-ip>:6643 tcp:<node-2-ip>:6643 tcp:<node-3-ip>:6643
[node-1]# sudo systemctl start ovn-nb-db

# Tier 2 (tune election-timer, prevent future split)
[any]# sudo ovs-appctl -t /var/run/ovn/ovnnb_db.ctl cluster/change-election-timer OVN_Northbound 4000
# Tăng từ 1000ms → 4000ms cho tolerance partition 15-20s

# Tier 3 (infra improvement)
# - Dedicated cluster network (không chung với data plane)
# - Network monitoring alert cho inter-node latency > 500ms
```

**Prevention:**

```yaml
- alert: OVSDBClusterMemberDisconnected
  expr: ovsdb_server_cluster_role{role="candidate"} > 0
  for: 2m
  labels: { severity: critical }
```

### Scenario B — ovs-vswitchd OOM kill

**Signature:** Syslog log "Out of memory: Killed process <PID> (ovs-vswitchd)". systemd restart ovs-vswitchd → toàn bộ OpenFlow flow bị uninstall → 30-60 giây traffic loss. Sau restart, memory tăng nhanh trở lại baseline cao.

**Evidence:**

```bash
# Memory footprint trước OOM
[host]# sudo ovs-appctl -t ovs-vswitchd memory/show
handlers:3 revalidators:4 ofconns:5 ports:247 
rules:48293 rules-cache-used:156MB rules-cache-max:256MB
...

# Kernel OOM log
[host]# sudo journalctl -k --since "1 hour ago" | grep -iE "oom|killed process"

# Process memory over time
[host]# sudo pmap -x $(pgrep ovs-vswitchd) | tail -1

# Coverage counter lflow + megaflow cache
[host]# sudo ovs-appctl coverage/show | grep -iE "flow_add|rule_inserted"

# DPDK hugepage usage (nếu dùng userspace DP)
[host]# grep Huge /proc/meminfo
```

**RCA:** Thường do 1 trong 3 nguyên nhân:

1. **Flow explosion:** OpenFlow rule count + megaflow cache vượt memory available. Check `rules-cache-used`.
2. **Memory leak real:** ovs-vswitchd bug (hiếm). Check memory growth rate với `pmap` over time; nếu linear tăng không tương quan workload → leak.
3. **Flow + cache settings too aggressive:** `other_config:flow-limit` quá cao so với RAM allocation.

**Remediation:**

```bash
# Tier 1 (immediate cap)
[host]# sudo ovs-vsctl set Open_vSwitch . other_config:flow-limit=50000
# Tier 2 (revalidator tuning)
[host]# sudo ovs-vsctl set Open_vSwitch . other_config:min-revalidate-pps=10
# Tier 3 (systemd resource limit hard floor)
[host]# sudo systemctl edit openvswitch-switch
# [Service]
# MemoryLimit=4G
```

### Scenario C — Handler thread saturation → upcall drop

**Signature:** `ovs-appctl upcall/show` show `lost_upcalls` counter tăng nhanh. Datapath miss traffic bị drop (không install flow, không forward). VM intermittent connectivity.

**Evidence — Anatomy Template A cho `upcall/show`:**

**Bối cảnh khi chạy:** Kỹ sư trực ca nhận alert "VM ICMP retry rate > 20%" từ Prometheus. Hypothesis: handler thread không theo kịp upcall rate.

**Command + output** *(reproduced from OVS 2.17.9 production-like)*:

```bash
[host]# sudo ovs-appctl upcall/show

system@ovs-system:
  flows         : (current 48293) (avg 47120) (max 51820) (limit 200000)
  offloaded flows : 0
  dump duration : 185ms
  ufid enabled : true
  
  23: (keys 12073) (total packets 1283749) (avg 55774/s) 
      (dispatched to handler)
  24: (keys 12089) (total packets 1290485) (avg 56066/s)
  25: (keys 12094) (total packets 1297843) (avg 56386/s)
  26: (keys 12037) (total packets 1289201) (avg 56009/s)
  
  lost_upcalls: 12837 (rate 142/s)     ← !!! RED FLAG
```

**Anatomy từng field:**

| Field | Giá trị mẫu | Ý nghĩa | Dấu hiệu bất thường |
|-------|-------------|---------|----------------------|
| `flows current` | 48293 | Current megaflow count | Gần `limit` 200K = cache đầy |
| `offloaded flows` | 0 | HW offload count | 0 = software path only |
| `dump duration` | 185ms | Revalidator sweep time | > 500ms = stress, > 1000ms = critical |
| `handler N: keys` | 12073 | Ukey per handler thread | Even distribution = healthy load balance |
| `avg packets/s` | 55774/s | Upcall rate per handler | Sum × N_handler phải < handler capacity |
| `lost_upcalls rate` | 142/s | **Upcall dropped, not handled** | **> 0 = handler bottleneck, MUST investigate** |

**Kịch bản bẻ gãy:**

- **`lost_upcalls > 0` với `dump duration < 100ms`** → handler không theo kịp (CPU bound), không phải revalidator. Fix: tăng `n-handler-threads`.
- **`dump duration > 500ms` + `lost_upcalls > 0`** → cả revalidator + handler saturated. Flow explosion nghi ngờ, check Scenario D.
- **`keys` skewed giữa handler** (ví dụ handler 23: 50K, handler 24-26: 1K) → hash function không evenly distribute. Hiếm, OVS bug. Report upstream.

**RCA + Remediation:**

```bash
# Check current thread count
[host]# sudo ovs-vsctl get Open_vSwitch . other_config | grep -E "n-handler|n-revalidator"

# Tăng handler thread (requires restart)
[host]# sudo ovs-vsctl set Open_vSwitch . other_config:n-handler-threads=8
[host]# sudo systemctl restart ovs-vswitchd
# Re-verify: lost_upcalls rate về 0 sau restart + 2-3 min warmup
```

### Scenario D — Megaflow cache thrashing

**Signature:** Flow install rate rất cao (`flow_add` coverage +10K/s), cache hit rate thấp (<80%). CPU ovs-vswitchd 100% on 1 core (handler thread). Traffic latency spike P99 3-5x.

**Evidence:**

```bash
# Coverage delta trong 10 giây
[host]# sudo ovs-appctl coverage/show | grep -E "flow_add|miss|wildcard"
# Delta check:
[host]# sudo ovs-appctl coverage/show > /tmp/cov1.txt; sleep 10
[host]# sudo ovs-appctl coverage/show > /tmp/cov2.txt
[host]# diff /tmp/cov1.txt /tmp/cov2.txt

# Wildcard pattern analysis
[host]# sudo ovs-dpctl dump-flows | awk '{print $1}' | sort -u | wc -l
# Many unique wildcarding = low cache effectiveness
```

**RCA:** Megaflow cache key quá narrow — mỗi 5-tuple packet tạo 1 entry riêng thay vì share wildcard mask. Common causes: ACL rule với TCP flags match (force per-packet inspection), ct_state transition fragmenting cache.

**Remediation:**

```bash
# Identify narrow flow
[host]# sudo ovs-dpctl dump-flows | awk '{print $1}' | sort | uniq -c | sort -rn | head

# Refactor OpenFlow rule để bỏ unnecessary match field
# (Trong OVN, qua ACL/Port_Group — xem Part 20.3 §20.3.4.6 consolidation)
```

### Scenario E — Tunnel BFD flap cascade

**Signature:** Geneve/VXLAN tunnel giữa 2 chassis intermittent UP→DOWN→UP. Traffic loss 200-500ms mỗi cycle flap. BFD monitoring log spam.

**Evidence:**

```bash
[host-A]# sudo ovs-appctl bfd/show
BFD session for tunnel ovn-chassisB:
  local state: UP
  remote state: UP
  ...
  flaps: 47               ← !!! flap count cao

[host-A]# sudo ovs-appctl tnl/neigh/show
# ARP resolution cho remote underlay phải stable. Flap đồng bộ = underlay issue.

# Underlay latency baseline
[host-A]# ping -i 0.2 -c 100 <host-B-underlay> | tail -5

# Geneve UDP packet trace underlay
[host-A]# sudo tcpdump -i eth0 -nn udp port 6081 -c 100 -w /tmp/geneve.pcap
```

**RCA:** 3 common cause:

1. **Underlay jitter > BFD detect threshold** (default 500ms × 3 = 1.5s). Burst latency underlay > 1.5s trigger BFD down.
2. **Underlay MTU fragmentation** — BFD packet > MTU bị drop silently.
3. **Host CPU overload** delay BFD packet send/receive.

**Remediation:**

```bash
# Tune BFD interval để tolerant underlay jitter
[host]# sudo ovs-vsctl set Interface ovn-chassisB-0 options:bfd_interval=1000 \
  options:bfd_min_rx=1000 options:bfd_min_tx=1000
# 1s × 3 = 3s detect (tolerant cho jitter < 3s)

# Alternate: disable BFD dùng ICMP alternative (advanced)
[host]# sudo ovs-vsctl set Interface ovn-chassisB-0 bfd=no
```

### Scenario F — HW offload silent fallback

**Signature:** Hardware offload (TC flower) configured, baseline 100 Gbps line-rate. Sau deploy rule ACL mới, throughput drop xuống 10 Gbps (kernel slow path). Không có error log. `ovs-dpctl dump-flows` show flow với `offload_state` field empty hoặc `no_support`.

**Evidence:**

```bash
# Offload stats
[host]# sudo ovs-dpctl dump-flows -m | grep -E "offloaded|state=offloaded"

# Tất cả flow trong tc
[host]# sudo tc -s filter show dev eth0 ingress | head

# Counter coverage offload-specific
[host]# sudo ovs-appctl coverage/show | grep -iE "offload|tc_"

# Driver dmesg
[host]# sudo dmesg | grep -iE "mlx5|offload|tc" | tail -20
```

**RCA:** TC flower driver (mlx5_core / i40e / etc.) không support combination action mới. Common pattern:

1. `conntrack` action trên NIC chưa support (mlx5 cần ConnectX-6+).
2. `mod_nw_ttl` + `mod_eth_dst` combo hit hardware limit.
3. Metadata > 16 bit width không fit HW register.

OVS silent fallback về software slow path — kernel datapath nhưng không HW accelerate.

**Remediation:**

```bash
# Tier 1: Disable offload cho rule specific
[host]# sudo ovs-vsctl set Open_vSwitch . other_config:hw-offload=true \
  other_config:offload-cap-list="drop,output,set,forward"
# Exclude ct/conntrack từ offload → kernel handle

# Tier 2: Upgrade NIC firmware
[host]# sudo ethtool -i eth0 | grep -i firmware
[host]# sudo mstconfig -d /dev/mst/mt4125_pciconf0 query | grep -i version

# Tier 3: Downgrade ACL complexity (Port_Group thay individual match)
```

### Scenario G — NIC ring buffer overflow

**Signature:** ethtool show `rx_no_buffer_count` tăng nhanh. Packet drop ở NIC level trước khi vào OVS. VM TCP retransmit rate spike.

**Evidence:**

```bash
# NIC statistics
[host]# sudo ethtool -S eth0 | grep -iE "rx_no_buffer|rx_missed|rx_discard"

# Ring buffer size
[host]# sudo ethtool -g eth0
Ring parameters for eth0:
Pre-set maximums:
RX:             4096
TX:             4096
Current hardware settings:
RX:             512           ← !!! Default, too small for 100Gbps
TX:             512

# IRQ distribution
[host]# cat /proc/interrupts | grep -E "eth0|mlx5" | head
```

**RCA:** Default RX ring 256-512 descriptor không đủ cho burst traffic > 1M pps. Khi CPU quá bận không drain ring kịp, NIC drop packet tại HW level.

**Remediation:**

```bash
# Tăng ring buffer lên max
[host]# sudo ethtool -G eth0 rx 4096 tx 4096

# IRQ affinity — distribute 8 core
[host]# sudo systemctl restart irqbalance
# Or manual: sudo set_irq_affinity.sh eth0

# Persistent qua reboot
[host]# sudo cat > /etc/network/if-up.d/ring-buffer <<'EOF'
#!/bin/bash
[ "$IFACE" = "eth0" ] && ethtool -G eth0 rx 4096 tx 4096
EOF
[host]# sudo chmod +x /etc/network/if-up.d/ring-buffer
```

### Scenario H — Conntrack table full

**Signature:** `dmesg` log "nf_conntrack: table full, dropping packet". TCP SYN drop random. OVS stateful ACL fail (`allow-related` reply không pass).

**Evidence:**

```bash
# Current vs max
[host]# cat /proc/sys/net/netfilter/nf_conntrack_count
512047

[host]# cat /proc/sys/net/netfilter/nf_conntrack_max
524288                        ← sát giới hạn

# Top entries breakdown
[host]# sudo conntrack -L | awk '{print $1,$2}' | sort | uniq -c | sort -rn | head
# Detect pattern (TCP ESTABLISHED vs UNREPLIED)
```

**RCA:** Default `nf_conntrack_max = hashsize × 4` tính từ RAM. Hypervisor 1000 VM × 500 connection/VM = 500K entry dễ đạt. Zone collision (Part 9.26 §9.26.12) amplify problem.

**Remediation:**

```bash
# Tier 1: Tăng max + rebuild hashtable
[host]# sudo sysctl -w net.netfilter.nf_conntrack_max=2097152
[host]# sudo sysctl -w net.netfilter.nf_conntrack_buckets=524288
[host]# echo 524288 | sudo tee /sys/module/nf_conntrack/parameters/hashsize

# Tier 2: Giảm TCP established timeout
[host]# sudo sysctl -w net.netfilter.nf_conntrack_tcp_timeout_established=3600
# 5 days → 1 hour, nhanh GC stale

# Tier 3: Per-zone limit (OVS 2.17+)
[host]# sudo ovs-appctl dpctl/ct-set-limits default=100000 zone=1012:50000
```

### Scenario I — OpenFlow controller disconnect fail-mode

**Signature:** Bridge `br0` config `fail-mode=secure`. Controller (Faucet/ONOS) crash. 2 phút sau, traffic hoàn toàn stop — tất cả packet drop ở br0.

**Evidence:**

```bash
[host]# sudo ovs-vsctl get Bridge br0 fail_mode
secure

# Controller connection
[host]# sudo ovs-vsctl get Bridge br0 controller
[tcp:10.0.0.100:6653]

# Disconnected state
[host]# sudo ovs-appctl -t ovs-vswitchd ofproto/list-tunnels
[host]# sudo ovs-vsctl list Controller
...
is_connected: false               ← controller crashed
...
```

**RCA:** `fail-mode=secure` nghĩa là khi controller disconnect, bridge chỉ sử dụng flow đã cài, **không học flow mới**. Traffic pattern mới → miss → drop. `fail-mode=standalone` (default) = fall back hub/switch, less secure nhưng traffic continue.

**Remediation:**

```bash
# Immediate fail-over
[host]# sudo ovs-vsctl set-fail-mode br0 standalone
# Traffic resume trong vài giây (bridge learn MAC qua `NORMAL` action)

# Restore controller primary
# ... restart controller service ...

# Long-term: controller HA cluster 3+ node
```

**Trade-off:** `secure` an toàn khi có controller cluster HA; `standalone` phù hợp edge/branch với controller single + tolerant tới hub-mode.

### Scenario J — Bond slave link flap microburst

**Signature:** `bond0` slave `eth1` flap 50 lần/giờ với interval 2-3 giây. Không có slave DOWN steady state, chỉ micro-flap. Nhưng traffic latency spike P99 100ms mỗi flap cycle (megaflow invalidation — xem Part 9.26 §9.26.11).

**Evidence:**

```bash
# Bond flap history
[host]# sudo ovs-appctl bond/show bond0 | grep -iE "slave|flap"

# Switch-side correlation
[host]# sudo ethtool -S eth1 | grep -iE "link_up|link_down|carrier_changes"

# Dmesg link event
[host]# sudo dmesg -T | grep -iE "eth1|link" | tail -30

# Switch vendor correlation (ví dụ Cisco)
# switch# show interfaces eth1/1 counters errors | include CRC|Frame
```

**RCA:** Ethernet frame CRC error rate > 1% trên cable/SFP → link layer declare DOWN → auto-negotiate UP → flap cycle 2-3 giây. Thường do cable xuống cấp, SFP temperature, hoặc switch port issue.

**Remediation:**

```bash
# Tier 1: Hysteresis để hide micro-flap
[host]# sudo ovs-vsctl set Port bond0 other_config:bond-detect-mode=carrier \
  other_config:bond-miimon-interval=1000 \
  other_config:bond-slave-updelay=5000 \
  other_config:bond-slave-downdelay=2000
# downdelay 2s = chịu flap < 2s không declare DOWN
```

## 9.14.8 Guided Exercise — Simulate OVSDB Raft partition + heal procedure

> **Trạng thái verification lab:** doc-plausible; cần sandbox OVN 3-node.

**Mục đích:** Reproduce Scenario A (§9.14.6) trong sandbox, practice heal procedure 3-tier.

**Chuẩn bị:** OVN sandbox 3-node cluster (node-1, node-2, node-3) với NBDB clustered.

**Bước:**

**1.** Baseline cluster status:

```bash
[node-1]$ sudo ovs-appctl -t /var/run/ovn/ovnnb_db.ctl cluster/status OVN_Northbound
# Expected: 3 member, 1 leader, 2 follower
```

**2.** Simulate partition — block traffic giữa node-1 và others:

```bash
[node-1]$ sudo iptables -I INPUT -s <node-2-ip>,<node-3-ip> -j DROP
[node-1]$ sudo iptables -I OUTPUT -d <node-2-ip>,<node-3-ip> -j DROP
```

**3.** Wait 10s, observe:

```bash
[node-1]$ sudo ovs-appctl -t /var/run/ovn/ovnnb_db.ctl cluster/status OVN_Northbound
# Expected: role=candidate (minority partition)

[node-2]$ sudo ovs-appctl -t /var/run/ovn/ovnnb_db.ctl cluster/status OVN_Northbound
# Expected: 2 member alive, node-1 marked disconnected
```

**4.** Heal network:

```bash
[node-1]$ sudo iptables -D INPUT -s <node-2-ip>,<node-3-ip> -j DROP
[node-1]$ sudo iptables -D OUTPUT -d <node-2-ip>,<node-3-ip> -j DROP
```

**5.** Observe catch-up behavior:

```bash
[node-1]$ watch -n 1 'sudo ovs-appctl -t /var/run/ovn/ovnnb_db.ctl cluster/status OVN_Northbound | grep -E "Role|Index"'
# Healthy: role chuyển từ candidate → follower + Last Index catchup lên
# Unhealthy: stuck candidate forever → apply Tier 1 fix (join-cluster)
```

**6.** Tier 1 fix if stuck:

```bash
[node-1]$ sudo systemctl stop ovn-nb-db
[node-1]$ sudo mv /var/lib/ovn/ovnnb_db.db /var/lib/ovn/ovnnb_db.db.bak
[node-1]$ sudo ovsdb-tool join-cluster /var/lib/ovn/ovnnb_db.db OVN_Northbound \
  tcp:<node-1-ip>:6643 tcp:<node-2-ip>:6643 tcp:<node-3-ip>:6643
[node-1]$ sudo systemctl start ovn-nb-db
```

**POE:** Giả thuyết "3-node cluster chịu được 1 partition duration ≤ X giây không mất state" → kiểm chứng với `election-timer=1000` default (X ≈ 5s) vs `election-timer=4000` (X ≈ 20s). Partition > X → stuck, cần Tier 1 fix.

**Cleanup:** restore iptables + cluster status.

## 9.14.9 Capstone POE Phase G — "Tăng nf_conntrack_max từ 512K lên 2M fix hoàn toàn Scenario H?"

**Kịch bản:** Hypervisor 1000 VM báo "TCP SYN drop 5%". Sysctl show `nf_conntrack_count=510K` (sát max 524K). Manager đề xuất tăng `nf_conntrack_max=2097152` fix vấn đề.

**Predict:** Tăng max có giúp không? Side effect?

**Observe (reasoning):**

1. **Tăng max = tăng memory consume:** mỗi conntrack entry ~300 byte + hashtable entry. 2M entry × 300 bytes = **600 MB RAM kernel space**. Check `/proc/meminfo` baseline.
2. **Tăng hashsize cần restart hoặc echo** trực tiếp vào `/sys/module/nf_conntrack/parameters/hashsize`. Không sync với `nf_conntrack_max` = conflict.
3. **GC pressure:** kernel GC thread duyệt toàn bộ table periodically. 2M entry × 1Hz = significant CPU overhead.
4. **Root cause persist:** Nếu leak rate (NEW connection không được close) = 500/s, thì 2M max chỉ buy **4000 giây (~1 giờ)** trước khi đầy lại.

**Explain:** Đề xuất **không đủ** — chỉ band-aid. Correct approach 3-tier:

- **Tier 1 (immediate):** Tăng max lên 2M ĐỂ stop current outage (5 phút win).
- **Tier 2 (short):** Giảm TCP established timeout 5 ngày → 1 giờ. Entry GC nhanh 5x → effective capacity 2M × 5 = 10M equivalent.
- **Tier 3 (long):** Identify leak source. Thường là:
  - VM application không close TCP connection gracefully.
  - SNAT rule tạo transient conntrack không cần thiết — refactor qua OVN `allow-related` stateful pattern.
  - Zone collision (Part 9.26 §9.26.12) multiply state entry — audit script detect.

**Finish:** Nguyên tắc: **tăng resource threshold KHÔNG fix leak**. Memory, flow-limit, conntrack-max đều là defense-in-depth lớp trên, không phải root cause fix. Cross-reference Part 9.26 §9.26.7 pattern: "Tầng 1 immediate cap + Tầng 3 root cause"; không skip.

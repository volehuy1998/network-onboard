## 15.1.1 Các thành phần OVN-Kubernetes

OVN-K8s triển khai ba thành phần chính trong cluster, mỗi thành phần chạy như DaemonSet hoặc Deployment trong namespace `ovn-kubernetes` hoặc `openshift-ovn-kubernetes` (tùy bản phân phối). (1) **ovnkube-master** — Deployment chạy trên control plane node, một replica active + một hoặc hai replica standby dưới cơ chế bầu leader. Nhiệm vụ: theo dõi Kubernetes API qua informer (Pod, Service, Endpoints, NetworkPolicy, Node, Namespace), dịch sang OVN NB DB (logical switch cho mỗi node, logical router cho cluster, ACL cho NetworkPolicy, Load_Balancer cho Service). (2) **ovnkube-node** — DaemonSet chạy trên mỗi worker node + control plane. Nhiệm vụ: quản lý OVS bridge local (`br-int`), cấp phát pod interface khi pod được schedule (tạo veth pair, gắn vào `br-int`, gọi CNI ADD/DEL), chạy ovn-controller local để tiếp nhận logical flow từ SB DB. (3) **nbdb-0/1/2 + sbdb-0/1/2 + northd-0/1/2** — StatefulSet 3-replica chạy ovsdb-server HA (cụm Raft) cho NB DB và SB DB, với ovn-northd dịch NB → SB.

Cấu hình HA cho ovn-northd khá tinh tế: northd không giữ state, nhưng chỉ một instance được active tại một thời điểm (để tránh xung đột khi ghi SB DB). OVN 22.03+ có sẵn cơ chế bầu leader qua `ovn-northd --pidfile --detach` với khóa qua NB DB. Trong ngữ cảnh Kubernetes, northd chạy như sidecar trong cùng pod với nbdb/sbdb, dùng ConfigMap để phân phối state leader.

Chiến lược SBDB per-AZ: ở cluster lớn (hơn 500 node), một cụm SB DB duy nhất trở thành điểm nghẽn — mọi node truy vấn flow từ cùng một cụm Raft, độ trễ tăng theo khoảng cách chain. OVN Interconnect (OVN-IC) cho phép chia cluster thành nhiều AZ (availability zone), mỗi AZ có SB DB riêng, giao tiếp giữa AZ qua transit switch. Phần này đề cập ở mục 15.1.4.

## 15.1.2 Dịch NetworkPolicy → OVN ACL

Kubernetes NetworkPolicy là tài nguyên cho phép người dùng định nghĩa quy tắc ingress/egress cho pod trong namespace. Ví dụ: "pod có label `app=web` trong namespace `frontend` chỉ được phép nhận traffic từ pod `app=api` trong namespace `backend` trên port 8080". Mặc định, namespace không có NetworkPolicy → mọi pod giao tiếp tự do (default-allow). Khi có NetworkPolicy áp dụng pod nào → pod đó chuyển sang **default-deny**, chỉ cho phép traffic khớp quy tắc.

Thuật toán dịch NetworkPolicy sang OVN ACL: (1) **podSelector** → **Port_Group** — OVN tạo một Port_Group chứa tất cả logical port của pod khớp selector. Khi pod được thêm/xóa (co giãn, schedule lại), ovnkube-master cập nhật Port_Group thời gian thực. (2) **namespaceSelector** → **Address_Set lồng nhau** — namespace có một Address_Set chứa IP của tất cả pod trong namespace; namespaceSelector khớp thành hợp của các Address_Set. (3) **quy tắc ingress/egress** → **ACL** với mệnh đề `match`, `action=allow/drop`, `priority` theo thứ tự quy tắc trong spec. (4) **Default-deny ngầm định** — mỗi Port_Group có một ACL ưu tiên thấp nhất `action=drop` catch-all, chỉ được kích hoạt khi có NetworkPolicy áp dụng lên pod đó.

ACL của OVN là **có state** qua conntrack (tương tự nftables `ct state`): khi traffic ingress được cho phép, traffic trả lời ngược lại tự động được cho phép mà không cần quy tắc egress tường minh. Điều này khớp ngữ nghĩa NetworkPolicy của Kubernetes (không yêu cầu định nghĩa traffic trả về). Cơ chế thực thi: ACL `match=... && ct.new` cho traffic mới, `ct.est` cho traffic đã established, `ct.rel` cho related (ICMP echo reply).

Hiệu năng: thực thi policy trong datapath OVS, không qua iptables → mở rộng tốt hơn cho pod thay đổi nhiều. Benchmark Red Hat 2022 trên cluster 200-node: độ trễ cập nhật NetworkPolicy OVN-K8s khoảng 500ms, kube-proxy + Calico iptables khoảng 3s ở 10K quy tắc.

## 15.1.3 Service + thay thế kube-proxy

Kubernetes Service là trừu tượng cho virtual IP (ClusterIP) + port ánh xạ tới tập endpoint (pod IP:port). Mặc định kube-proxy chạy trên mỗi node, cài quy tắc iptables (hoặc IPVS) để DNAT từ ClusterIP:port → pod IP:port. OVN-K8s thay thế kube-proxy hoàn toàn — không cần kube-proxy chạy trên node nữa.

Ánh xạ Service → OVN:
- **ClusterIP Service** → entry trong table **Load_Balancer** của NB DB. Load_Balancer có `vips = {"10.96.0.1:80": "10.244.1.5:8080, 10.244.2.3:8080"}` (VIP → danh sách endpoint). OVN gắn Load_Balancer vào logical switch hoặc logical router; khi packet tới VIP, OVN thực hiện DNAT theo thuật toán hash (5-tuple) để chọn endpoint. Giữ kết nối (session affinity) qua ClientIP nếu Service cấu hình `sessionAffinity: ClientIP`.
- **NodePort Service** → kết hợp ClusterIP Load_Balancer + **chassisredirect** (gateway chassis đặc biệt của OVN) + DNAT. Khi traffic đến node IP:NodePort từ ngoài, gateway chassis nhận, DNAT tới ClusterIP, rồi cân bằng tải tới endpoint.
- **ExternalTrafficPolicy=Local** — tối ưu quan trọng: thay vì cân bằng tải toàn cluster (có thể SNAT mất client IP), chỉ định tuyến tới endpoint trên cùng node với traffic ingress. OVN-K8s hiện thực qua logic lọc trong Load_Balancer: chỉ chọn endpoint có `chassis=local_chassis`.
- **LoadBalancer Service** → phụ thuộc cloud controller (AWS ELB, Azure LB) cấp VIP công cộng; OVN-K8s xử lý phần trong cluster.

**ExternalIP annotation** và **tích hợp MetalLB**: khi cluster on-prem không có cloud LB, dùng MetalLB hoặc `egressIP` native của OVN-K8s để gán public IP. OVN-K8s 1.0+ hỗ trợ LoadBalancer native qua BGP advertise từ gateway node, tương tự chế độ BGP của Calico/MetalLB.

## 15.1.4 Mạng đa cluster (OVN-IC)

**OVN Interconnect (OVN-IC)** là tính năng ra đời 2022 cho phép nhiều cluster OVN kết nối với nhau ở lớp logical, traffic giữa cluster đi qua overlay (Geneve) mà không cần route bên ngoài. Field hợp sử dụng chính: Kubernetes đa cluster (federation, failover đa vùng), hybrid cloud stretch, lab đa AZ.

Kiến trúc OVN-IC: (1) **Transit Switch** — logical switch đặc biệt kết nối gateway chassis của nhiều AZ. Gateway chassis của mỗi AZ kết nối transit switch qua Geneve tunnel. (2) **IC NB DB + IC SB DB** — cơ sở dữ liệu riêng chứa state toàn cục về transit switch và routing giữa AZ. Mỗi AZ có một **ovn-ic-nb-ovsdb** replica local, đồng bộ với IC DB toàn cục qua chương trình ovn-ic. (3) **Phân lập VNI** — mỗi AZ có Virtual Network Identifier riêng, traffic VNI A không lộ sang VNI B trừ khi có quy tắc định tuyến tường minh.

Trong OVN-K8s 1.0+ (2024), mỗi zone cluster trong thiết lập đa cluster được gán chassis-id riêng, ovnkube-master quảng bá route local tới transit switch, và học route remote từ IC DB. Pod-to-pod giữa zone đi qua Geneve tunnel từ gateway này sang gateway kia, tương tự inter-subnet trong single zone nhưng chain dài hơn (OVS local → gateway local → transit → gateway remote → OVS remote).

Sự đánh đổi: OVN-IC tăng độ phức tạp triển khai đáng kể — cần cụm IC DB toàn cục, HA cho gateway, và quản lý ánh xạ VNI. Chỉ phù hợp khi thực sự cần mạng đa cluster. Nếu chỉ cần DR (disaster recovery) đa vùng, dùng ExternalIP + DNS failover đơn giản hơn.

## 15.1.5 Di trú OpenShift-SDN → OVN-K8s

OpenShift 3.x và 4.0-4.11 dùng **OpenShift-SDN** — CNI tùy chỉnh của Red Hat dựa trên OVS flow rule thuần (không dùng OVN). Kiến trúc: một VXLAN overlay liên node, iptables cho service, quy tắc multi-tenancy qua VNID. Cluster hoạt động tốt ở quy mô nhỏ-trung nhưng gặp khó khăn ở cluster hơn 500 node: số quy tắc iptables bùng nổ (hơn 50K quy tắc trong cluster lớn), cách hiện thực multi-tenancy phức tạp.

Red Hat quyết định di trú sang OVN-K8s từ OpenShift 4.10 (tháng 3/2022) — ban đầu là opt-in, đến 4.12 (tháng 1/2023) trở thành CNI mặc định cho cluster mới. Lý do: (1) OVS/OVN có cộng đồng lớn, được duy trì bởi nhiều nhà cung cấp; (2) thực thi NetworkPolicy ở data plane OVS mở rộng tốt hơn iptables; (3) chuẩn bị cho tính năng đa cluster (OVN-IC) và multi-tenancy VRF; (4) OVN-K8s tương thích với Red Hat Service Mesh + OpenShift Virtualization (KubeVirt) tốt hơn.

Lộ trình di trú cho cluster hiện có: (1) **Offline migration** — drain toàn bộ node, thay thế CNI (chỉnh cấu hình cluster), khởi động lại node, triển khai lại pod. Thời gian ngừng hoạt động khoảng 30-60 phút cho cluster 50-node. (2) **Online migration (ra mắt 4.14)** — di trú từng lô node, traffic pod tiếp tục chạy qua CNI cũ trong khi CNI mới đang cấp phát. Phức tạp hơn nhưng không ngừng hoạt động.

Các tính năng có sẵn trong cả hai: NetworkPolicy, Service, EgressIP, Multicast. Tính năng chỉ OVN-K8s: đa cluster (OVN-IC), IPv6 dual-stack hoàn chỉnh, offload phần cứng (OVS HW offload qua switchdev), EgressFirewall CRD (ACL ra ngoài cluster), BGP EVPN cho metadata service.

### ▶ Guided Exercise 1: Deploy OVN-K8s on kind cluster

> **State lab verification:** Output dựa trên OVN-Kubernetes docs 2024; chờ verify lab thực.

**Mục tiêu:** Chạy OVN như CNI của Kubernetes, xác minh chuỗi dịch NetworkPolicy → OVN ACL hoạt động từ đầu đến cuối. Đây là thiết lập đại diện cho môi trường production để hiểu các thành phần OVN-K8s.

**Chuẩn bị:**
- Ubuntu 22.04 VM, 8 GB RAM, 4 CPU
- kind + kubectl installed
- Clone OVN-K8s: `git clone https://github.com/ovn-org/ovn-kubernetes.git`

**Mô hình lab:**

```
┌──────────────────────────────────────────────────┐
│         kind cluster (3 nodes)                   │
│                                                  │
│  ns ovn-kubernetes:                              │
│                                                  │
│  ┌────────────────────────────────────────────┐  │
│  │ ovnkube-master-0/1/2 (3 replicas)          │  │
│  │  ├ northd (NBDB → SBDB compile)            │  │
│  │  ├ nb-ovsdb + sb-ovsdb (Raft cluster)      │  │
│  │  └ ovnkube-master (K8s→OVN translator)     │  │
│  └────────────────────────────────────────────┘  │
│                                                  │
│  ┌────────────────────────────────────────────┐  │
│  │ ovnkube-node (daemonset, per K8s node)     │  │
│  │  ├ ovn-controller (SBDB → OpenFlow)        │  │
│  │  └ ovs-vswitchd + ovsdb-server             │  │
│  └────────────────────────────────────────────┘  │
│                                                  │
│  ns default:                                     │
│  ┌────────────┐  ┌────────────┐                  │
│  │ pod web    │  │ pod app    │                  │
│  │ 10.244.0.5 │  │ 10.244.0.6 │                  │
│  └─────┬──────┘  └─────▲──────┘                  │
│        │  NetworkPolicy │                        │
│        └────────────────┘ → OVN ACL              │
└──────────────────────────────────────────────────┘
```

**Bước thực hiện:**

**1.** Tạo kind cluster với OVN-K8s CNI disable default kindnet:

```bash
[host]$ cat > kind-ovn.yaml << 'EOF'
kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
networking:
  disableDefaultCNI: true
nodes:
  - role: control-plane
  - role: worker
  - role: worker
EOF
[host]$ kind create cluster --name ovn-k8s --config kind-ovn.yaml
```

**2.** Deploy OVN-K8s:

```bash
[host]$ cd ovn-kubernetes/dist/images
[host]$ ./daemonset.sh --image=ghcr.io/ovn-org/ovn-kubernetes/ovn-kube-ubuntu:master
[host]$ kubectl apply -f ovn-setup.yaml
[host]$ kubectl apply -f rbac-ovnkube-cluster-manager.yaml
[host]$ kubectl apply -f ovnkube-master.yaml
[host]$ kubectl apply -f ovnkube-node.yaml
```

**3.** Verify pods running:

```bash
[host]$ kubectl -n ovn-kubernetes get pods
```

Kết quả mong đợi:

```
NAME                        READY   STATUS
ovnkube-master-0            5/5     Running
ovnkube-master-1            5/5     Running
ovnkube-master-2            5/5     Running
ovnkube-node-xxxxx          3/3     Running
ovnkube-node-yyyyy          3/3     Running
ovnkube-node-zzzzz          3/3     Running
```

**4.** Inspect OVN NBDB:

```bash
[host]$ kubectl exec -n ovn-kubernetes ovnkube-master-0 -c nb-ovsdb -- ovn-nbctl show
```

Kết quả mong đợi: Logical switches cho mỗi node (`join`, `ovn_cluster_router`, node-specific), Logical_Switch_Ports cho pods.

**5.** Deploy test pods:

```bash
[host]$ kubectl run web --image=nginx --labels=tier=web
[host]$ kubectl run app --image=alpine --labels=tier=app -- sleep 3600
[host]$ kubectl get pods -o wide
```

**6.** Apply NetworkPolicy cho phép web→app:80:

```bash
[host]$ cat > policy.yaml << 'EOF'
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-web-to-app
spec:
  podSelector:
    matchLabels:
      tier: app
  policyTypes: [Ingress]
  ingress:
  - from:
    - podSelector:
        matchLabels:
          tier: web
    ports:
    - protocol: TCP
      port: 80
EOF
[host]$ kubectl apply -f policy.yaml
```

**7.** Observe OVN ACL creation:

```bash
[host]$ kubectl exec -n ovn-kubernetes ovnkube-master-0 -c nb-ovsdb -- ovn-nbctl list ACL | head -30
```

Kết quả mong đợi:

```
_uuid               : abc123...
action              : allow-related
direction           : to-lport
label               : 1
log                 : false
match               : "outport == @a123_Ingress && ip4 && tcp.dst == 80 && ip4.src == {10.244.0.5}"
name                : "default:allow-web-to-app:Ingress:0"
priority            : 1001
severity            : []
```

**Output mong muốn:** NetworkPolicy → OVN ACL translation occurs automatically, Port_Group `a123_Ingress` membership = pods với label `tier=app`.

**Bài học học được:**

Thứ nhất, **OVN-K8s là CNI tính sẵn sàng cao**: ovnkube-master chạy 3 replica (cụm Raft cho OVSDB), ovnkube-node DaemonSet per-worker. Chịu lỗi tốt hơn CNI có điểm fail duy nhất (OpenShift-SDN cũ).

Thứ hai, **NetworkPolicy translation tự động**: Label selector → Port_Group membership, port rule → ACL khớp với expression. Mỗi K8s NetworkPolicy sinh 2-5 OVN ACL entries tùy complexity.

Thứ ba, **ovn-nbctl remain source of truth** — debug NetworkPolicy issues by inspecting NBDB ACLs, không đi theo iptables path như kube-proxy.

**Cleanup:**

```bash
[host]$ kind delete cluster --name ovn-k8s
```

### ▶ Lab 2: NetworkPolicy debug + trace

> **State lab verification:** Output tracing format dựa trên ovn-trace docs; chờ verify lab thực.

**Mục tiêu:** Thực hành gỡ lỗi vấn đề NetworkPolicy với `ovn-trace` — công cụ có giá trị lớn khi NetworkPolicy môi trường production không hoạt động như kỳ vọng.

**Chuẩn bị:** Hoàn thành Ex1 baseline (OVN-K8s cluster running).

**Mô hình lab:**

```
3-tier application với NetworkPolicy chain:

┌─────────┐  ACL1   ┌─────────┐  ACL2   ┌─────────┐
│ pod web │────────►│ pod app │────────►│ pod db  │
│ tier=web│ :8080   │ tier=app│ :5432   │ tier=db │
└─────────┘         └─────────┘         └─────────┘
                                             ▲
                      default-deny: block mọi else
```

**Bước thực hiện:**

**1.** Deploy 3-tier pods:

```bash
[host]$ kubectl run web --image=nginx --labels=tier=web
[host]$ kubectl run app --image=alpine --labels=tier=app -- sleep 3600
[host]$ kubectl run db --image=postgres --env=POSTGRES_PASSWORD=x --labels=tier=db
```

**2.** Apply 3-tier NetworkPolicies:

```yaml
# policy-web-to-app.yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata: {name: allow-web-to-app}
spec:
  podSelector: {matchLabels: {tier: app}}
  policyTypes: [Ingress]
  ingress:
  - from: [{podSelector: {matchLabels: {tier: web}}}]
    ports: [{protocol: TCP, port: 8080}]

# policy-app-to-db.yaml (similar structure cho tier=db, port=5432)
```

**3.** Apply cả hai:

```bash
[host]$ kubectl apply -f policy-web-to-app.yaml -f policy-app-to-db.yaml
```

**4.** Verify với ovn-trace (simulate packet web → app:8080):

```bash
[host]$ WEB_IP=$(kubectl get pod web -o jsonpath='{.status.podIP}')
[host]$ APP_IP=$(kubectl get pod app -o jsonpath='{.status.podIP}')
[host]$ kubectl exec -n ovn-kubernetes ovnkube-master-0 -c ovnkube-master -- \
        ovn-trace --minimal "inport == \"web\" && \
                             eth.src == 0a:00:00:00:00:01 && \
                             ip4.src == $WEB_IP && \
                             ip4.dst == $APP_IP && \
                             tcp.dst == 8080"
```

Kết quả mong đợi: trace qua OVN logical pipeline, cuối cùng `output to \"app\"` action.

**5.** Test actual traffic:

```bash
[host]$ kubectl exec web -- curl -m 5 http://$APP_IP:8080
```

Kỳ vọng: connection success hoặc HTTP response.

**6.** Intentionally break: delete app→db rule:

```bash
[host]$ kubectl delete networkpolicy allow-app-to-db
```

**7.** Debug blocked traffic:

```bash
[host]$ DB_IP=$(kubectl get pod db -o jsonpath='{.status.podIP}')
[host]$ kubectl exec -n ovn-kubernetes ovnkube-master-0 -c ovnkube-master -- \
        ovn-trace --minimal "inport == \"app\" && ip4.src == $APP_IP && \
                             ip4.dst == $DB_IP && tcp.dst == 5432"
```

Kết quả mong đợi: trace ends with `drop` action (no matching allow ACL → default-deny).

**8.** Inspect logical flow để xác định nguyên nhân:

```bash
[host]$ kubectl exec -n ovn-kubernetes ovnkube-master-0 -c nb-ovsdb -- \
        ovn-sbctl lflow-list | grep -A1 "ls_out_acl"
```

**Output mong muốn:** Clear decision path showing exactly which ACL allowed/denied, match conditions evaluated at each stage.

**Bài học học được:**

Thứ nhất, **`ovn-trace` là siêu năng lực gỡ lỗi**: cho phép mô phỏng packet đi qua toàn bộ logical pipeline mà không cần gửi packet thật. So với gỡ lỗi iptables/kube-proxy (phải tcpdump từng interface) → đơn giản hơn rất nhiều.

Thứ hai, **default-deny semantic**: khi NetworkPolicy exists cho một pod (ingress hoặc egress), non-matched traffic **implicitly denied**. Common mistake: apply NetworkPolicy mà quên add necessary allow rule → pod bị isolate.

Thứ ba, **OVN logical flow visibility**: `ovn-sbctl lflow-list` show all compiled logical flow, priority-ordered. Debug NetworkPolicy translation issues here.

**Cleanup:**

```bash
[host]$ kubectl delete pod web app db
[host]$ kubectl delete networkpolicy --all
```

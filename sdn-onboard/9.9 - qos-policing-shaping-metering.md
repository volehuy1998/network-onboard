## Mục tiêu bài học

Sau khi hoàn thành phần này, bạn có thể:

7. **So sánh (Compare)** OVS QoS manual với OVN QoS Logical Switch Port (`qos_max_rate`, `qos_min_rate`), chọn đúng tool theo scale và mức kiểm soát cần thiết.

## 9.9.0 Bối cảnh mở đầu, vì sao QoS vẫn quan trọng năm 2026

Tháng 6 năm 2023, một nhà mạng di động ở Đông Nam Á chuẩn bị ra mắt dịch vụ 5G SA (standalone) cho 12 triệu thuê bao. Hạ tầng core network chạy trên OpenStack với Neutron + ML2/OVN, băng thông ra Internet 100 Gbps chia cho nhiều dịch vụ: mobile broadband, IoT telemetry, enterprise VPN, voice VoLTE. Đội vận hành nhận báo cáo: voice call chất lượng xuống dần vào giờ cao điểm 19-21h, jitter lên tới 120 ms (ngưỡng chấp nhận là 30 ms). Điều tra phát hiện một lỗi cấu hình QoS: các bridge OVS ở compute node không cấu hình shaping cho voice queue, mọi lưu lượng chia sẻ cùng một FIFO 1000-packet. Khi mobile broadband chiếm dung lượng, voice packet bị delay trong queue chờ, và jitter tăng lên do buffer bloat. Fix: cấu hình `linux-htb` với ba queue (voice 20 Mbps guaranteed, data 70 Mbps guaranteed, best-effort 10 Mbps) trên mỗi interface ra Internet, thêm OpenFlow rule khớp với DSCP EF (46) chuyển vào voice queue. Sau 30 phút, jitter giảm về dưới 15 ms, voice call stable. Bài học: trong môi trường production thực tế, QoS không phải "nice-to-have" mà là yêu cầu hạ tầng cơ bản, đặc biệt khi nhiều loại lưu lượng competing cùng link.

QoS có bốn mục tiêu hạ tầng tương ứng bốn chỉ tiêu đo được trong SLA:

**Một, bandwidth guarantee:** đảm bảo một loại lưu lượng có tối thiểu X Mbps ngay cả khi link bão hoà. Ví dụ voice cần 100 kbps/call. Dùng HTB min-rate.

**Hai, độ trễ control:** giữ độ trễ dưới ngưỡng cho real-time traffic. Voice/video cần dưới 150 ms one-way end-to-end. Dùng priority queue và low-độ trễ qdisc như CoDel hoặc FQ-CoDel.

**Ba, jitter reduction:** giữ biến động độ trễ nhỏ. Jitter cao hơn độ trễ trung bình sẽ gây voice/video không đều. Dùng shaping thay vì policing.

**Bốn, loss prevention:** giảm tỷ lệ drop cho TCP-based applications. Mỗi packet drop là một retransmission + congestion window giảm, ảnh hưởng băng thông thực tế. Dùng ECN marking + Active Queue Management thay vì tail-drop.

Bài hôm nay dựng lại nền tảng toàn bộ QoS trong OVS theo Lab 9 Crichigno + compass Chapter I.

## 9.9.1 OVS cấu hình QoS, không implement

Compass Ch I điểm quan trọng: **OVS không triển khai QoS, chỉ cấu hình nó**. Queueing thực sự do:

- Linux `tc` (traffic control) cho kernel datapath.
- DPDK rte_sched library cho userspace datapath.

OVS cung cấp API trừu tượng qua OVSDB tables (`QoS`, `Queue`) và OpenFlow action (`set_queue`). Khi cấu hình, OVS dịch thành `tc class` + `tc filter` cho kernel, hoặc rte_sched config cho DPDK.

Hai loại QoS:

- **Ingress policing**: rate limiter đơn giản (token bucket) áp trực tiếp trên Interface row.
- **Egress shaping**: hierarchical với QoS + Queue tables + OpenFlow action.

## 9.9.3 Egress shaping với linux-htb

Compass Ch I verbatim recipe:

```bash
[host]# ovs-vsctl -- add-br br0 \
         -- add-port br0 eth0 \
         -- add-port br0 vif1.0 -- set interface vif1.0 ofport_request=5 \
         -- add-port br0 vif2.0 -- set interface vif2.0 ofport_request=6 \
         -- set port eth0 qos=@newqos \
         -- --id=@newqos create qos type=linux-htb \
               other-config:max-rate=1000000000 \
               queues:123=@vif10queue queues:234=@vif20queue \
         -- --id=@vif10queue create queue other-config:max-rate=10000000 \
         -- --id=@vif20queue create queue other-config:max-rate=20000000
```

- QoS parent `max-rate=1 Gbps`.
- Queue ID 123 max 10 Mbps (cho vif1.0).
- Queue ID 234 max 20 Mbps (cho vif2.0).
- Queue ID là arbitrary OpenFlow queue ID mà `set_queue:` action tham chiếu.

**Bước bắt buộc** — OpenFlow rule để packet vào đúng queue:

```bash
[host]# ovs-ofctl add-flow br0 in_port=5,actions=set_queue:123,normal
[host]# ovs-ofctl add-flow br0 in_port=6,actions=set_queue:234,normal
```

Compass Ch I cảnh báo: "**Without the OpenFlow rules, packet go to the default queue and nothing is shaped — this is the single most common QoS misconfiguration in the field.**"

QoS và Queue rows **không được garbage-collect** — phải destroy explicit:

## 9.9.6 HTB deep dive — cấu trúc cây, borrow, ceil

Hierarchical Token Bucket là thuật toán shaping chính của Linux kernel từ 2003, được OVS dùng qua `type=linux-htb` trong QoS row. Lab 9 §1.2 mô tả HTB như một cây phân cấp:

```
              Root (100 Mbps max)
              /            \
         Class 1          Class 2
       (rate 70 Mbps)   (rate 30 Mbps)
        ceil 100 Mbps    ceil 100 Mbps
         |                  |
      Leaf Q1            Leaf Q2
      (qdisc: TBF)       (qdisc: FIFO)
```

Năm đặc tính của HTB:

**Một, chỉ leaf class có queue thực.** Node trung gian (root, class 1, class 2) là bộ định tuyến token, không buffer packet. Leaf có queue để enqueue khi rate quá nhanh. Nếu bạn quên add qdisc cho leaf, kernel dùng pfifo mặc định.

**Hai, mỗi class có hai giá trị: `rate` và `ceil`.** `rate` là bandwidth đảm bảo, `ceil` là bandwidth tối đa. Class có thể "mượn" thêm bandwidth giữa `rate` và `ceil` khi có token dư từ sibling.

**Ba, cơ chế borrow giữa siblings.** Nếu class 2 không dùng hết 30 Mbps của mình, class 1 có thể mượn để đạt tới ceil 100 Mbps. Khi class 2 bắt đầu dùng, class 1 phải nhả bandwidth về lại rate 70 Mbps. Cơ chế này gọi là "work-conserving" — không phí băng thông.

**Bốn, không vượt ceil.** Ceil là hard limit. Class 1 không thể vượt ceil 100 Mbps ngay cả khi cả link rảnh.

**Năm, chỉ áp egress.** HTB chỉ gắn qdisc hướng ra (egress) của interface. Inbound traffic phải dùng ingress policing hoặc redirect vào ifb device + HTB.

### Cú pháp OVS cấu hình HTB 2-class

Trích Lab 9 §6 Step 2 (Page 18):

```bash
ovs-vsctl -- set port s1-eth3 qos=@newqos \
  -- --id=@newqos create qos type=linux-htb \
       other-config:max-rate=100000000 \
       queues=1=@q1,2=@q2 \
  -- --id=@q1 create queue other-config:min-rate=70000000 \
  -- --id=@q2 create queue other-config:min-rate=30000000
```

- `max-rate=100000000` (100 Mbps) là ceil của root.
- `min-rate=70000000` (70 Mbps) cho queue 1 là rate guaranteed.
- `min-rate=30000000` (30 Mbps) cho queue 2 là rate guaranteed.
- Queue ID 1 và 2 là arbitrary OpenFlow queue ID được `set_queue:` action tham chiếu.

Nếu không set `max-rate` cho individual queue, nó inherit ceil từ parent (100 Mbps). Nghĩa là cả hai queue đều có khả năng mượn lên tới 100 Mbps khi sibling rảnh.

### Kiểm chứng qua tc command

Sau khi OVS tạo QoS, `tc` hiển thị cây HTB tương ứng:

```bash
tc qdisc show dev s1-eth3
# qdisc htb 1: root refcnt 2 r2q 10 default 0 direct_packets_stat 0

tc class show dev s1-eth3
# class htb 1:1 root rate 100Mbit ceil 100Mbit burst 1600b cburst 1600b
# class htb 1:2 parent 1:1 prio 0 rate 70Mbit ceil 100Mbit burst 0b cburst 0b
# class htb 1:3 parent 1:1 prio 0 rate 30Mbit ceil 100Mbit burst 0b cburst 0b
```

Hai công cụ xác nhận cùng state: OVS thấy qua `qos/show`, Linux thấy qua `tc`. Khi debug, luôn check cả hai.

## 9.9.7 Policing và Shaping, so sánh sâu

Cả hai cùng giới hạn rate, nhưng hành vi với packet vượt rate khác hoàn toàn.

**Ingress policing drop ngay:** khi packet vượt token bucket rate, kernel drop tại điểm vào. Không buffer, không delay. Ưu: implementation siêu đơn giản, ít overhead. Nhược: TCP nhận được packet loss tín hiệu → congestion window giảm nửa → băng thông thực tế không ổn định. Jitter cao vì mỗi lần bị drop là một lần retransmit.

**Egress shaping buffer rồi gửi từ từ:** khi rate vượt ngưỡng, packet vào queue chờ. Khi có token, packet được dequeue ra. Ưu: TCP không thấy packet loss → giữ được băng thông thực tế ổn định, độ trễ đều. Nhược: queue delay tăng lên khi overload → buffer bloat. Cần qdisc thông minh (CoDel, FQ-CoDel) để kiểm soát queue depth.

### POE quan trọng — Lab 9 §3 Step 6 phản chứng

Lab 9 đặt một giả thuyết ngược thú vị: "policing 500 Mbps sẽ cho băng thông thực tế 500 Mbps". Test thực tế:

```bash
# Terminal s1 (Linux)
sudo ovs-vsctl set interface s1-eth1 ingress_policing_rate=500000

# Terminal h1
iperf3 -c 10.0.0.2 -p 21
# Throughput thực: ~79 Mbps, NOT 500 Mbps
```

Giả thuyết bị bác bỏ. Lý do (Lab 9 trang 13): *"traffic policing interacts poorly with some network protocols and can have surprising results. This is a limitation of ingress policing and occurs mostly with higher rates like 500 Mbps."*

Ngữ nghĩa sâu hơn: TCP là protocol adaptive với packet loss. Khi policer drop packet ngẫu nhiên theo burst pattern không đoán được, TCP congestion control giảm cwnd quá mức → băng thông thực tế thực thấp hơn nhiều so với rate cho phép. Đây là bằng chứng: ingress policing không phải tool đúng cho high-rate traffic, chỉ tốt cho low-rate limiting (1-20 Mbps) hoặc hard cap.

Giải pháp thay thế với cùng rate 500 Mbps: dùng egress shaping (HTB) thay vì ingress policing. Shaping buffer packet, TCP không thấy loss, congestion control hoạt động bình thường, băng thông thực tế đạt gần rate cấu hình.

### Table so sánh

| Đặc tính | Ingress policing | Egress shaping |
|----------|-------------------|----------------|
| Hành vi vượt rate | Drop ngay | Buffer rồi gửi |
| Độ trễ | Thấp (không queue) | Tăng khi overload |
| Jitter | Cao (packet loss random) | Thấp (gửi đều) |
| TCP băng thông thực tế | Không ổn định với rate cao | Ổn định |
| Overhead | Rất thấp | Trung bình (queue management) |
| Use case tốt | Hard cap, low rate (1-20 Mbps) | Guarantee bandwidth, multi-class |
| OVS cấu hình | `ingress_policing_rate` trên Interface | QoS + Queue + OpenFlow action |

## 9.9.8 Metering 3-color (srTCM + trTCM) — RFC 2697 và RFC 2698

Metering là kỹ thuật phân loại lưu lượng thành ba màu (Green, Yellow, Red) tuỳ theo rate so với hai ngưỡng:

- **CIR (Committed Information Rate):** rate đảm bảo theo hợp đồng.
- **PIR (Peak Information Rate):** rate đỉnh cho phép trong thời gian ngắn.

Hai algorithm chính (IETF standard):

**Single-Rate Three-Color Marker (srTCM) — RFC 2697:** dùng một rate duy nhất (CIR) + hai burst size (CBS committed burst, EBS excess burst). Packet:
- Trong-CBS: Green.
- Trên-CBS nhưng trong-EBS: Yellow.
- Vượt cả hai: Red.

**Two-Rate Three-Color Marker (trTCM) — RFC 2698:** dùng hai rate (CIR + PIR) + hai burst size.
- Dưới CIR: Green.
- Giữa CIR và PIR: Yellow.
- Trên PIR: Red.

Packet Yellow thường được chuyển vào queue priority thấp hoặc mark DSCP. Packet Red drop hoặc mark ở mức cao nhất.

### OVS hỗ trợ 2-color trong OVS 2.17.9

Lab 9 §1.4 ghi chú: *"Though Open vSwitch supports three color marking, our version of Open vSwitch only supports two color marking (green and red)."* Nghĩa là OVS 2.17 (và kernel datapath) chỉ phân green/red, không có yellow trung gian. Muốn 3-color thật phải dùng OVS DPDK userspace với `trtcm-policer` hoặc upgrade lên OVS 3.x.

Cấu hình 2-color metering qua OpenFlow 1.5 meter, Lab 9 §4 Step 2-4 (Page 13):

```bash
# Step 1: xem meter features
sudo ovs-ofctl -O OpenFlow15 meter-features s1
# Output: band_types=drop (nghĩa là hỗ trợ 2-color drop)

# Step 2: add meter limit 300 Mbps
sudo ovs-ofctl -O OpenFlow15 add-meter s1 \
  meter=1,kbps,band=type=drop,rate=300000

# Step 3: install flow dùng meter cho TCP port 21
sudo ovs-ofctl -O OpenFlow15 add-flow s1 \
  tcp,tp_dst=21,actions=meter:1,output:3

# Step 4: install flow dùng cùng meter cho TCP port 80
sudo ovs-ofctl -O OpenFlow15 add-flow s1 \
  tcp,tp_dst=80,actions=meter:1,output:3
```

Một meter có thể được reference bởi nhiều flow. Lab 9 Step 4 comment: *"Since one meter is applied to multiple flow entries, all the flow entries will share the meter rate."* Nghĩa là nếu meter 1 limit 300 Mbps, và flow TCP port 21 + TCP port 80 cùng dùng meter 1, tổng bandwidth cả hai không vượt 300 Mbps.

### Kiểm chứng với 2 host competing

Lab 9 §5 Step 6-7 demo: h1 (port 21) + h3 (port 80) đều gửi 300 Mbps → tổng 600 Mbps đến switch. Meter 300 Mbps drop excess. Kết quả thực tế (Lab 9 trang 17):
- h1 được khoảng 200 Mbps.
- h3 được khoảng 170 Mbps.
- Total 370 Mbps marked Green (cho qua), 230 Mbps marked Red (drop).

Note: phân chia giữa h1 và h3 không 1:1 vì meter không có fairness, chỉ có rate limit. Muốn fairness phải dùng HTB với 2 class bằng nhau.

## 9.9.9 Topology Lab 9 — 4-host 2-switch competing bandwidth

Lab 9 topology (Figure 3, trang 6):

```
    h1 (10.0.0.1) ─┐                         ┌─ h2 (10.0.0.2)
                   │                         │    FTP server
                   ├── s1 ─── 100 Mbps ── s2 ┤
                   │                         │
    h3 (10.0.0.3) ─┘                         └─ h4 (10.0.0.4)
     HTTP client                              HTTP server
```

Link s1 ↔ s2 là bottleneck 100 Mbps. Bốn host cùng subnet 10.0.0.0/8:

- **h1** là FTP client, gửi TCP port 21 qua s1-s2-h2.
- **h2** là FTP server, nhận TCP port 21.
- **h3** là HTTP client, gửi TCP port 80 qua s1-s2-h4.
- **h4** là HTTP server, nhận TCP port 80.

QoS được cài trên port `s1-eth3` (output về s2). Hai queue:
- Queue 1 (rate 70 Mbps): cho TCP port 21 (FTP, giả lập production traffic).
- Queue 2 (rate 30 Mbps): cho TCP port 80 (HTTP, giả lập background traffic).

Hai flow OpenFlow:

```bash
# FTP → Queue 1 (priority cao)
sudo ovs-ofctl add-flow s1 tcp,tp_dst=21,actions=set_queue:1,normal

# HTTP → Queue 2 (priority thấp)
sudo ovs-ofctl add-flow s1 tcp,tp_dst=80,actions=set_queue:2,normal
```

### Ba scenario verify

**Scenario A: cá nhân không competing.** h1 chạy iperf3 port 21 một mình. Kết quả (Lab 9 trang 21): ~95.6 Mbps. h1 mượn được bandwidth từ queue 2 rảnh nên đạt gần max 100 Mbps.

**Scenario B: h3 cá nhân không competing.** h3 chạy iperf3 port 80 một mình. Kết quả: ~95.6 Mbps. Tương tự, h3 mượn từ queue 1 rảnh.

**Scenario C: h1 + h3 competing đồng thời.** Cả hai iperf3 20 giây song song. Kết quả kỳ vọng (Lab 9 trang 23, doc-plausible):
- h1 (queue 1 guaranteed 70 Mbps): đạt gần 70 Mbps.
- h3 (queue 2 guaranteed 30 Mbps): đạt gần 30 Mbps.
- Tổng 100 Mbps bằng link capacity.

Đây là bằng chứng HTB work-conserving: khi một queue không dùng, queue kia mượn; khi cả hai dùng, mỗi queue được đảm bảo rate của mình.

## 9.9.10 So sánh với OVN QoS

OVN không buộc người vận hành viết QoS row thủ công như OVS. Thay vào đó, OVN cung cấp API abstraction qua Logical Switch Port column `qos_max_rate` và `qos_min_rate`:

```bash
# Cấu hình QoS per LSP qua ovn-nbctl
ovn-nbctl set Logical_Switch_Port lsp-vm1 \
  options:qos_max_rate=100000000 \
  options:qos_min_rate=70000000
```

OVN sẽ tự động:
1. Sinh Logical Flow trong SBDB.
2. `ovn-controller` trên compute node dịch thành OVS QoS + Queue row trên `br-int`.
3. Sinh OpenFlow rule với `set_queue:N` để classifier lưu lượng từ LSP đó.

Ba khác biệt quan trọng:

**Abstraction level.** OVS operator viết QoS row cho từng port thủ công. OVN operator chỉ set option trên LSP, compiler dưới xử lý.

**Dynamic scale.** OVN tự sinh queue khi VM được tạo, tự destroy khi VM bị xoá. OVS manual phải tự destroy Queue row khi không cần (`--all destroy Queue`).

**Limitation OVN.** OVN API hiện chỉ hỗ trợ đơn giản: `qos_max_rate` + `qos_min_rate` per LSP. Không hỗ trợ multi-class per LSP, không expose meter 3-color. Khi cần QoS phức tạp, vẫn phải drop xuống OVS manual.

Quy tắc thực dụng: OVN QoS đủ cho 80% các tình huống triển khai thực tế (cap bandwidth per VM, priority cho voice). Khi cần 3-color metering, complex classification theo DSCP/COS, hoặc interaction với hardware offload, dùng OVS manual.

## Guided Exercise 11, Policing 10 Mbps vs 500 Mbps

**Mục tiêu:** Kiểm chứng POE "ingress policing scale tuyến tính với rate" bị bác bỏ. Quan sát sự khác biệt băng thông thực tế thực tế giữa policing rate thấp (10 Mbps) và cao (500 Mbps).

**Giả thuyết ngược cần bác bỏ:**

> *"Ingress policing rate X Mbps sẽ cho băng thông thực tế X Mbps trong mọi field hợp. Nếu tôi set rate 500 Mbps, iperf3 sẽ đạt gần 500 Mbps."*

### Chuẩn bị

Load topology Lab 9 bằng Mininet (dùng script Python Topo hoặc MiniEdit với `lab9.mn`). Đảm bảo s1 bridge đang hoạt động và các host h1-h4 có IP đúng.

Trên h2, khởi iperf3 server port 21:

```bash

```

Trên h1, chạy client:

```bash
h1$ iperf3 -c 10.0.0.2 -p 21
```

Ghi lại băng thông thực tế. Kỳ vọng (Lab 9 trang 10): ~43.9 Gbps (Mininet veth ảo không có rate limit).

### Bước 2, policing 10 Mbps

Trên Linux terminal của s1 (bên ngoài Mininet CLI):

```bash
sudo ovs-vsctl set interface s1-eth1 ingress_policing_rate=10000
```

Chạy lại iperf3 trên h1. Kỳ vọng băng thông thực tế: ~10.6 Mbps (gần rate cấu hình).

### Bước 3, policing 500 Mbps

Xoá QoS cũ, set rate 500 Mbps:

```bash
sudo ovs-vsctl -- --all destroy QoS
sudo ovs-vsctl set interface s1-eth1 ingress_policing_rate=500000
```

Chạy lại iperf3. Kỳ vọng băng thông thực tế: ~79 Mbps (xa 500 Mbps target).

### Bước 4, phân tích

Giả thuyết ngược bị bác bỏ: rate 500 Mbps cho băng thông thực tế chỉ 79 Mbps, không phải ~500 Mbps tuyến tính. Nguyên nhân TCP congestion control:

1. Policer drop packet ngẫu nhiên khi rate vượt 500 Mbps.
2. TCP sender nhận dup ACK hoặc timeout → giảm congestion window nửa (Reno) hoặc giảm 30% (CUBIC).
3. cwnd thấp → sender không đạt được bandwidth-delay product cần thiết cho 500 Mbps.
4. Băng thông thực tế steady-state dao động quanh mức thấp.

### Bước 5, dùng shaping thay vì policing

Xoá policer, dùng HTB shaping 500 Mbps:

```bash
sudo ovs-vsctl -- --all destroy QoS
sudo ovs-vsctl set interface s1-eth1 ingress_policing_rate=0
sudo ovs-vsctl -- set port s1-eth3 qos=@newqos \
  -- --id=@newqos create qos type=linux-htb \
       other-config:max-rate=500000000 \
       queues=1=@q1 \
  -- --id=@q1 create queue other-config:min-rate=500000000
sudo ovs-ofctl add-flow s1 \
  "tcp,tp_dst=21,actions=set_queue:1,normal"
```

Chạy lại iperf3. Kỳ vọng băng thông thực tế: ~480-495 Mbps (rất gần target). Bằng chứng shaping đạt rate cao tốt hơn policing.

### Bài học

Ingress policing không phải công cụ đúng cho rate cao (>100 Mbps). Cho hard cap low-rate (1-20 Mbps), policing dùng OK. Cho bandwidth guarantee multi-class, shaping là lựa chọn duy nhất đúng.

## Điểm cốt lõi cần nhớ

**Một.** QoS có bốn mục tiêu (bandwidth guarantee, latency control, jitter reduction, loss prevention). Mỗi mục tiêu cần tool khác nhau: HTB cho bandwidth, priority queue + CoDel cho latency/jitter, shaping cho loss.

**Hai.** OVS cấu hình QoS qua OVSDB abstraction (QoS + Queue row), queue thực sự chạy ở Linux tc hoặc DPDK rte_sched. Debug phải check cả hai view: `ovs-appctl qos/show` và `tc class show`.

**Ba.** HTB có cấu trúc cây root → class → leaf queue, chỉ leaf có queue thực. Cơ chế borrow/ceil cho phép class mượn bandwidth từ sibling rảnh, nhả khi sibling bận. Work-conserving, không phí bandwidth.

**Bốn.** Egress shaping cần ba thứ: QoS row + Queue row + OpenFlow `set_queue` action. Thiếu OpenFlow rule là lỗi phổ biến nhất, compass Ch I ghi rõ *"the single most common QoS misconfiguration in the field"*.

**Năm.** Ingress policing drop ngay khi vượt rate, tốt cho low-rate hard cap (1-20 Mbps). Rate cao (>100 Mbps) thì policing tương tác xấu với TCP, băng thông thực tế thực thấp hơn rate cấu hình nhiều (Lab 9 chứng minh 500 Mbps → 79 Mbps thực tế).

**Sáu.** 3-color metering (RFC 2697 srTCM + RFC 2698 trTCM) phân green/yellow/red theo CIR + PIR. OVS 2.17 kernel datapath chỉ hỗ trợ 2-color (green/red). DPDK userspace hỗ trợ 3-color qua `trtcm-policer`.

**Bảy.** QoS/Queue row KHÔNG auto-GC. Phải `destroy` explicit: `ovs-vsctl -- --all destroy QoS -- --all destroy Queue`. Orphan row tích luỹ gây confusion khi debug.

**Tám.** OVN QoS qua `qos_max_rate` + `qos_min_rate` trên LSP phù hợp 80% use case. Khi cần 3-color metering, classification theo DSCP/COS, hoặc interaction với hardware offload, dùng OVS manual.

## 9.9.11 Schema chi tiết, năm thực thể peripheral cần nắm vững

Năm phần dưới đây mở rộng góc nhìn schema OVSDB + traffic-control backend cho năm keyword peripheral mà operator OVS chạm thường xuyên khi cấu hình QoS: hai row schema (`QoS row`, `Queue row`) và ba `type` value (`linux-htb`, `linux-hfsc`, `linux-noop`). Mỗi phần đi từ định nghĩa schema trong `vswitchd/vswitch.ovsschema` đến handler implementation trong `lib/netdev-linux.c`, với nguồn cite cụ thể OVS v2.17.9.

### 9.9.11.1 QoS row, bảng cha của hierarchy shaping

QoS là một bảng OVSDB top-level (`isRoot: true`) với bốn cột chính theo `vswitch.ovsschema` v2.17.9:

```json
"QoS": {
  "columns": {
    "type":         { "type": "string" },
    "queues":       { "type": { "key": {"type":"integer", "minInteger":0,
                                        "maxInteger":4294967295},
                                "value": {"type":"uuid", "refTable":"Queue"},
                                "min":0, "max":"unlimited" } },
    "other_config": { "type": {"key":"string", "value":"string", ...} },
    "external_ids": { "type": {"key":"string", "value":"string", ...} }
  },
  "isRoot": true
}
```

**Vai trò + vị trí.** QoS row là điểm gắn (anchor) của một policy shaping cho một port. `Port.qos` (cột UUID weak reference) trỏ về một QoS row; nếu trỏ NULL thì port không có hierarchy egress shaping, chỉ có default qdisc kernel. Một QoS row có thể được tham chiếu từ nhiều Port (uncommon nhưng schema cho phép) vì `isRoot=true` không phụ thuộc vào tham chiếu để tồn tại — đây là lý do row không tự GC khi clear `Port.qos`.

**Cột `type`.** String identifier khớp tên handler trong `tc_ops` registry của `lib/netdev-linux.c`. Giá trị thực tế OVS v2.17.9 hỗ trợ: `linux-htb`, `linux-hfsc`, `linux-noop`, `linux-codel`, `linux-fq_codel`, `linux-sfq`, `linux-netem`, `egress-policer`, `trtcm-policer`, hoặc empty string `""` (default qdisc).

**Cột `queues` (integer-to-uuid map).** Key là OpenFlow queue ID arbitrary (0 đến 2^32-1) mà `set_queue:N` action tham chiếu, value là UUID của Queue row. `min:0` cho phép QoS row không có queue (ví dụ `linux-noop` không dùng queue).

**Cột `other_config`.** String-to-string map chứa parameter cho `type`. Ví dụ với `linux-htb`, key thường gặp `max-rate` (ceil của root, đơn vị bps).

**Quy trình cấu hình (verbatim từ OVS QoS Howto):**

```bash
ovs-vsctl -- set port eth0 qos=@qos1 \
          -- --id=@qos1 create qos type=linux-htb \
                other-config:max-rate=1000000000 queues:0=@q0
```

`@qos1` là OVSDB row reference symbol; `set port` insert UUID vào `Port.qos` cùng commit với `create qos` insert row mới — đây là pattern atomic OVSDB transaction tránh race condition.

**Cleanup explicit (vì sao không auto-GC):** schema attribute `isRoot: true` đánh dấu QoS row tồn tại độc lập; OVSDB garbage collector chỉ xoá row có `isRoot: false` khi mất tham chiếu. Lệnh dọn dẹp:

```bash
ovs-vsctl -- destroy QoS eth0 -- clear Port eth0 qos
```

### 9.9.11.2 Queue row, leaf node của hierarchy

Queue row là bảng OVSDB top-level mô tả một leaf class trong cây HTB/HFSC. Schema v2.17.9:

```json
"Queue": {
  "columns": {
    "dscp":         { "type": { "key": {"type":"integer","minInteger":0,
                                        "maxInteger":63},
                                "min":0, "max":1 } },
    "other_config": { "type": {"key":"string","value":"string", ...} },
    "external_ids": { "type": {"key":"string","value":"string", ...} }
  },
  "isRoot": true
}
```

Queue chỉ có ba cột nội dung: `dscp` optional (0-63, set DSCP marking khi packet vào queue), `other_config` map các parameter shaping, `external_ids` map cho tag identifier ngoài.

**Cột `other_config` keys phổ biến (per `linux-htb` handler trong `lib/netdev-linux.c`):**

- `min-rate`: rate guaranteed cho class (bps). Bắt buộc cho HTB leaf.
- `max-rate`: ceil của class (bps). Optional, mặc định inherit ceil từ parent.
- `burst`: burst size byte. Optional.
- `priority`: integer 0-7 cho HTB priority resolution khi nhiều class cùng cấp tranh token.

**Vì sao Queue row tách riêng QoS row.** Cây HTB có nhiều leaf class, mỗi leaf cần parameter riêng (rate, ceil). OVSDB normalize ra hai bảng: QoS row chứa root + map `queues:N=<uuid>` trỏ Queue row chứa parameter của leaf N. Pattern này giống one-to-many relational schema cổ điển.

**Liên quan mật thiết.** Queue UUID xuất hiện ở: (a) `QoS.queues` map value, (b) Logical_Switch_Port options trong OVN khi compile xuống br-int. Mỗi queue được kernel translate thành `tc class htb` với classid khớp queue ID OpenFlow.

**Cleanup orphan:** sau khi `destroy QoS`, Queue row vẫn tồn tại độc lập (`isRoot: true`). Phải dọn riêng:

```bash
ovs-vsctl --all destroy Queue
```

Production note: nếu workflow tạo VM thường xuyên, viết script post-destroy để clear orphan Queue, tránh tích luỹ row vô dụng làm chậm OVSDB query.

### 9.9.11.3 linux-htb, type phổ biến nhất cho egress shaping

`linux-htb` là `type` value khớp handler `tc_ops_htb` trong `lib/netdev-linux.c` v2.17.9 dòng 4987:

```c
static const struct tc_ops tc_ops_htb = {
    .ovs_name = "linux-htb",
    ...
};
```

**Cơ chế.** Khi user `set qos type=linux-htb`, OVS ovs-vswitchd gọi `htb_tc_install` → translate sang `tc qdisc add dev <port> root handle 1: htb` + `tc class add dev <port> parent 1: classid 1:N htb rate ... ceil ...` cho mỗi entry trong `QoS.queues` map.

**`other_config` keys QoS row HTB chấp nhận (từ struct `htb_class` line 4529 + `htb_query_class__` line 4708):**

- `max-rate`: ceil của root class (1:1 trong tc terminology).

**`other_config` keys Queue row HTB chấp nhận:**

- `min-rate` (bắt buộc): rate guaranteed.
- `max-rate`: ceil per-class. Default inherit từ root.
- `burst`: burst size byte.
- `priority`: 0-7, lower = higher priority khi resolution.

**Mapping sang tc backend (kiểm chứng thực tế):**

```bash
$ ovs-vsctl set port eth0 qos=@q -- --id=@q create qos type=linux-htb \
       other-config:max-rate=100000000 queues:1=@q1 \
    -- --id=@q1 create queue other-config:min-rate=70000000
$ tc qdisc show dev eth0
qdisc htb 1: root ... default 0 ...
$ tc class show dev eth0
class htb 1:1 root rate 100Mbit ceil 100Mbit ...
class htb 1:2 parent 1:1 rate 70Mbit ceil 100Mbit ...
```

Quan sát: queue ID OpenFlow `1` map sang tc classid `1:2` (offset +1 vì 1:1 là root). Đây là mapping ổn định OVS dùng từ 2.0 đến nay.

**Phiên bản drift.** Handler `linux-htb` có từ OVS 1.x; struct `htb_class` không thay đổi cấu trúc ở v2.17. Lab 9 (dựa trên OVS 2.17 trên Ubuntu 22.04) dùng cú pháp giống hệt OVS 3.x.

**Use case.** Bandwidth guarantee multi-class với borrow giữa siblings — production phổ biến nhất. Khi cần đơn giản hard cap, dùng `egress-policer` thay vì HTB.

### 9.9.11.4 linux-hfsc, fair service curve cho real-time + best-effort mix

`linux-hfsc` là handler `tc_ops_hfsc` line 5484 v2.17.9, struct `hfsc_class` line 5010. HFSC (Hierarchical Fair Service Curve) là thuật toán shaping của Carnegie Mellon (1997) cung cấp đồng thời hai service guarantee mà HTB không làm được: bandwidth + delay bound.

**Cơ chế khác HTB.** HTB chỉ có một curve (rate). HFSC có hai curve:

- **Real-time (RT) curve:** delay bound đảm bảo cho packet đầu tiên của burst.
- **Link-share (LS) curve:** bandwidth share khi link bão hoà.

Class có thể có một hoặc cả hai curve. Class chỉ có LS curve hành xử giống HTB; class có cả RT + LS đảm bảo low-delay cho real-time traffic ngay cả khi best-effort traffic đang đầy queue.

**`other_config` keys HFSC chấp nhận (per `hfsc_parse_class_details__` line 5180):**

- QoS row: `max-rate` (ceil của root).
- Queue row: `min-rate` (LS curve rate). HFSC OVS handler hiện chỉ expose subset đơn giản; full m1/d/m2 service curve không cấu hình được qua OVS abstraction, phải `tc class change` trực tiếp.

**Cấu hình ví dụ:**

```bash
ovs-vsctl set port eth0 qos=@q -- --id=@q create qos type=linux-hfsc \
       other-config:max-rate=100000000 queues:0=@q0,1=@q1 \
    -- --id=@q0 create queue other-config:min-rate=20000000 \
    -- --id=@q1 create queue other-config:min-rate=80000000
```

**Use case.** Mạng cần đồng thời voice (low-delay) + bulk transfer (bandwidth share). HTB không đảm bảo delay bound; HFSC đảm bảo. Trong OVS production, HFSC ít phổ biến hơn HTB vì cấu hình phức tạp + OVS không expose full curve API. Khi gặp `type=linux-hfsc` trong production, thường là legacy config từ pre-2015.

**Verify:**

```bash
$ tc qdisc show dev eth0
qdisc hfsc 1: root refcnt 2 default 1
$ tc class show dev eth0
class hfsc 1: root
class hfsc 1:1 parent 1: ls m1 0bit d 0us m2 80Mbit ...
```

### 9.9.11.5 linux-noop, passthrough để vô hiệu QoS không xoá schema

`linux-noop` là handler `tc_ops_noop` line 5525 v2.17.9. Implementation siêu đơn giản (line 5500-5527):

```c
static void noop_install__(struct netdev *netdev_) {
    struct netdev_linux *netdev = netdev_linux_cast(netdev_);
    static const struct tc tc = TC_INITIALIZER(&tc, &tc_ops_default);
    netdev->tc = CONST_CAST(struct tc *, &tc);
}
static const struct tc_ops tc_ops_noop = {
    .ovs_name = "linux-noop",
    .tc_install = noop_tc_install,
    .tc_load    = noop_tc_load,
};
```

Handler không gọi `tc qdisc add` gì cả — chỉ set `netdev->tc` về default placeholder. Kết quả: kernel giữ nguyên qdisc mặc định (pfifo_fast trên Ubuntu cũ, fq_codel mặc định Ubuntu 22.04 systemd-networkd). Không có shaping, không có rate limit.

**Vì sao tồn tại.** Hai use case chính:

1. **Disable QoS tạm thời mà không xoá schema row.** Trong production thay đổi giữa 2 policy: từ `linux-htb` rate 100 Mbps sang `linux-noop` (vô hiệu) rồi sang `linux-htb` rate 200 Mbps. Không phải destroy QoS row + recreate (mất reference từ Port + log audit phức tạp). Chỉ cần `set qos type=linux-noop`.
2. **Fast diagnostic.** Khi nghi ngờ QoS đang gây bottleneck, đổi thành `linux-noop` xác nhận; nếu băng thông phục hồi → vấn đề ở shaping config; nếu không → cause khác.

**Quan sát.** Sau `set qos type=linux-noop`, `tc qdisc show dev <port>` hiển thị qdisc mặc định kernel (fq_codel hoặc pfifo_fast) thay vì `htb`. Không có class nào được khai báo.

**Limitation.** `linux-noop` không xoá Queue row đang tham chiếu trong `QoS.queues`. Map `queues` vẫn tồn tại nhưng không được handler dùng. Khi muốn dọn sạch, vẫn phải `destroy QoS` + `--all destroy Queue`.

**Phiên bản drift.** Handler có từ OVS 2.0+ (commit 2009-2010), không thay đổi qua các bản. Ubuntu 22.04 với OVS 2.17 có sẵn.

**Use case ngược.** Khi vận hành OVN trong OpenStack mà operator muốn debug "có phải QoS gây drop?", set type=linux-noop trên `br-int` queue tạm thời; nếu drop biến mất → QoS misconfig. Đây là technique nhanh hơn so với `--all destroy QoS` rồi recreate sau.

## 10.7.1 Kiến trúc + tính năng đặc trưng

### 10.7.1.1 ovsdb-client trong stack OVS/OVN

`ovsdb-client` là CLI thuộc package `ovs-common` (hoặc `openvswitch-common` Ubuntu), kết nối trực tiếp tới ovsdb-server qua TCP, Unix socket, hoặc SSL. Khác với `ovs-vsctl`, `ovn-nbctl`, `ovn-sbctl`:

- `ovs-vsctl`/`ovn-*ctl` xây dựng **IDL** (Interface Definition Language) đầy đủ trong process, hiểu schema-aware, validate trước khi gửi.
- `ovsdb-client` không build IDL, chỉ gửi raw JSON-RPC theo RFC 7047. Phù hợp cho tool generic + scripting + debug schema-agnostic.

Kiến trúc:

```
┌─────────────────┐      RFC 7047 JSON-RPC over TCP/Unix/SSL
│  ovsdb-client   │ ◄────────────────────────────────────────────┐
└─────────────────┘                                                │
        │                                                          ▼
        │ command-line: list-dbs / dump / monitor / transact ...   │
        │                                                  ┌───────────────┐
        ▼                                                  │  ovsdb-server │
   stdout JSON or formatted table                          │  (NB or SB)   │
                                                            └───────────────┘
```

**Tính chất quan trọng:**

- **Schema-agnostic:** chạy được trên bất kỳ database OVSDB (NBDB, SBDB, OpenFlow OVSDB, custom schema).
- **No transaction batching:** mỗi invocation là 1 connection mới. Khác `ovs-vsctl` có thể batch nhiều thay đổi.
- **Synchronous output:** `monitor` là streaming event, các lệnh khác trả về kết quả rồi exit.
- **Format flexibility:** option `-f` cho format (`table`, `list`, `csv`, `json`).

### 10.7.1.2 Connection target

```bash
ovsdb-client <command> [<server>] [<db>] [<table>] [<column>...]
```

Trong đó `<server>` có thể là:

- `tcp:host:port`, ví dụ `tcp:nb1.example.com:6641`.
- `unix:/var/run/ovn/ovnnb_db.sock`, default cho process local.
- `ssl:host:port`, kết hợp với option `-p <privkey>`, `-c <cert>`, `-C <ca-cert>` cho mTLS.
- Mặc định nếu không pass `<server>`: đọc `OVS_RUNDIR` (mặc định `/var/run/openvswitch/` cho OVS, `/var/run/ovn/` cho OVN), tìm socket `db.sock` hoặc `ovnnb_db.sock`/`ovnsb_db.sock`.

Ví dụ:

```bash
# OVS local
[host]# ovsdb-client list-dbs unix:/var/run/openvswitch/db.sock
Open_vSwitch

# OVN NBDB cluster Raft
[host]# ovsdb-client list-dbs tcp:nb1:6641,tcp:nb2:6641,tcp:nb3:6641
OVN_Northbound

# OVN SBDB qua SSL
[host]# ovsdb-client -p key.pem -c cert.pem -C ca.pem \
                     list-dbs ssl:sb1.example.com:6642
OVN_Southbound
```

Nhóm lệnh đọc thông tin về schema, không thay đổi DB.

```bash
[host]# ovsdb-client list-dbs unix:/var/run/openvswitch/db.sock

[host]# ovsdb-client list-dbs tcp:nb1:6641

```

Liệt kê tên database mà ovsdb-server đang serve. Một server có thể serve nhiều DB cùng lúc (production thường 1 DB / server, nhưng test setup hay multi-tenant có thể khác).

## 10.7.3 Query + dump

### `dump [<db> [<table> [<column>...]]]`

```bash
[host]# ovsdb-client dump tcp:nb1:6641 OVN_Northbound Logical_Switch_Port name addresses type
name                 addresses                              type
"web1"               ["aa:bb:cc:00:00:01 10.0.0.10"]       ""
"web2"               ["aa:bb:cc:00:00:02 10.0.0.11"]       ""
"phys-port"          ["unknown"]                            "localnet"
```

Format mặc định table-formatted, `-f json` cho JSON pure. Khác `ovn-nbctl list`: dump không filter, lấy hết rows; tốt cho audit toàn cluster.

### `query <db> <op-json>`

Thực thi 1 hoặc nhiều RFC 7047 operation read-only mà không commit transaction:

```bash
[host]# ovsdb-client query tcp:nb1:6641 \
        '["OVN_Northbound", {"op":"select","table":"Logical_Switch_Port",\
          "columns":["name","addresses"],"where":[["type","==","localnet"]]}]'
```

Trả JSON array với từng row khớp `where`. Hữu ích:

- Test phức tạp `where` clause trước khi dùng trong app.
- Debug điều kiện filter mà `ovn-nbctl find` không hỗ trợ đầy đủ syntax.

## 10.7.4 Transaction (write)

### `transact <db> <op-array>`

Atomic transaction theo RFC 7047. Mọi op trong array hoặc commit hết, hoặc rollback hết:

```bash
[host]# ovsdb-client transact tcp:nb1:6641 \
        '["OVN_Northbound",
          {"op":"insert","table":"Logical_Switch","row":{"name":"my-ls"},"uuid-name":"new_ls"},
          {"op":"insert","table":"Logical_Switch_Port",
           "row":{"name":"port1","type":""},
           "uuid-name":"new_port"},
          {"op":"mutate","table":"Logical_Switch","where":[["_uuid","==",["named-uuid","new_ls"]]],
           "mutations":[["ports","insert",["set",[["named-uuid","new_port"]]]]]}]'
```

Khi cần thay đổi atomic mà không có CLI helper sẵn (ví dụ: rare edge case mà ovn-nbctl không cover), `transact` là tool fallback. Nhưng với syntax JSON nặng, dễ sai → chỉ dùng khi thật sự cần.

### Cảnh báo Rule 5 trụ cột

`transact` cho phép write thẳng vào SBDB (output của northd). Tương tự `ovn-sbctl set Port_Binding chassis=`, đây là **anti-pattern** trừ debug khẩn cấp. Với NBDB, `ovn-nbctl` luôn ưu tiên hơn `transact` raw.

## 10.7.5 Monitoring (event stream)

Đây là tính năng **đặc trưng** của `ovsdb-client`, không tool nào khác có. Mỗi event thay đổi DB được stream realtime ra stdout cho đến khi user Ctrl-C.

### `monitor <db> <table> [<column>...]`

```bash
[host]# ovsdb-client monitor tcp:sb1:6642 OVN_Southbound Port_Binding chassis logical_port
row                                  action  chassis    logical_port
0a17b4f8-...                         insert  ""         "web1"
0a17b4f8-...                         old     ""         "web1"
0a17b4f8-...                         new     "55b4..."  "web1"
0a17b4f8-...                         delete  "55b4..."  "web1"
```

3 action type: `insert` (row mới), `old`/`new` (paired update), `delete`. Cho phép theo dõi lifecycle 1 entity (ví dụ: trace claim Port_Binding cross-chassis).

### `monitor-cond` + `monitor-cond-since`

Phiên bản nâng cao với conditional filter (giảm bandwidth):

```bash
[host]# ovsdb-client monitor-cond tcp:sb1:6642 OVN_Southbound \
        '[{"table":"Port_Binding",
           "where":[["logical_port","==","web1"]]}]'
```

Chỉ stream event của row khớp `where`. `monitor-cond-since` thêm điểm bắt đầu (last_id) cho replay khi reconnect.

### Forensic use case

Kịch bản thực tế (Phần 20.5 §20.5.1) dùng `monitor` để forensic Port_Binding migration race:

```bash
[host]# ovsdb-client --timestamp monitor tcp:sb-leader:6642 \
                     OVN_Southbound Port_Binding chassis | \
        grep --line-buffered "logical_port=\"vm-prod-456\""
```

Với `--timestamp`, mỗi event có prefix thời gian millisecond → reconstruct timeline chính xác cho dual-bind window.

## 10.7.6 Coordination (wait + lock)

### `wait <db> <table> <where>... <until>...`

Block tới khi điều kiện match:

```bash
[host]# ovsdb-client wait tcp:nb1:6641 OVN_Northbound NB_Global \
                     '_uuid!=00000000-0000-0000-0000-000000000000' nb_cfg=100
```

Nghĩa: chờ đến khi `nb_cfg = 100` xuất hiện. Hữu ích cho integration test hoặc CMS workflow đồng bộ.

### `lock`/`steal`/`unlock <lock-name>`

Distributed locking primitive built-in OVSDB:

```bash
[host]# ovsdb-client lock tcp:nb1:6641 my-lock-name &
PID 12345

# Process khác đợi lock:
[host]# ovsdb-client lock tcp:nb1:6641 my-lock-name
# (block until first process exits or release)

# Force take lock:
[host]# ovsdb-client steal tcp:nb1:6641 my-lock-name
```

OVN-northd dùng OVSDB lock để bầu leader (xem Phần 13.8.3): instance giành lock `ovn_northd` trên SBDB → trở thành active.

## 10.7.7 Backup + Restore + Convert

### `backup <db>`

```bash
[host]# ovsdb-client backup tcp:nb1:6641 > nbdb-backup-$(date +%Y%m%d).db
```

Stream toàn bộ DB ra stdout dưới format OVSDB log. Production: cron daily, gửi remote storage.

### `restore <db>`

```bash
[host]# ovsdb-client restore tcp:nb1:6641 < nbdb-backup-20260425.db
```

Yêu cầu DB target empty hoặc same schema. Restore vào cluster Raft đang running an toàn (atomic transaction).

### `convert <db> <schema-file>`

Migrate schema sau upgrade OVN/OVS:

```bash
[host]# ovsdb-client needs-conversion tcp:nb1:6641 OVN_Northbound \
                     /usr/share/ovn/ovn-nb.ovsschema
yes

[host]# ovsdb-client convert tcp:nb1:6641 OVN_Northbound \
                     /usr/share/ovn/ovn-nb.ovsschema
```

Lệnh này thay đổi schema runtime, không cần restart ovsdb-server. Quan trọng cho **rolling upgrade** OVN: upgrade ovn-northd binary trước → convert NBDB schema → upgrade ovn-controller.

## 10.7.8 Anatomy Template A, 5 output quan trọng

### Anatomy 1: `monitor` event stream

> **Bối cảnh khi chạy:** Forensic timeline Port_Binding hoặc Logical_Flow change. Với `--timestamp` enable.

```
2026-04-25T14:23:10.123Z  row=a1b2c3d4  action=old      chassis=""             logical_port="web1"
2026-04-25T14:23:10.123Z  row=a1b2c3d4  action=new      chassis="55b4d9e8..."  logical_port="web1"
2026-04-25T14:23:15.456Z  row=a1b2c3d4  action=old      chassis="55b4d9e8..."  logical_port="web1"
2026-04-25T14:23:15.456Z  row=a1b2c3d4  action=new      chassis="fc12a8d1..."  logical_port="web1"
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| Action `insert`/`delete` rate | < 10/s tổng cluster | 10-100/s | > 100/s | CMS bulk update; xem `inc-engine/show` (Phần 13.8.6) |
| `old → new` paired interval | < 100 ms | 100-1000 ms | > 1000 ms | OVSDB Raft slow; xem `cluster/status` (Phần 10.1.6.5) |
| Row appearing/disappearing rapidly | Stable | Flap 2-5/min | Flap > 10/min | Claim race chassis (Phần 13.7.8); xem migration |
| `chassis` empty (after `new`) | Sau lsp-del | Sau migration in progress | Stuck > 30s | northd recompute lag hoặc chassis offline |

**Kịch bản bẻ gãy:**

- Dual-bind window > 5 giây: claim race như kịch bản Phần 20.5 §20.5.1. Force release qua `ovn-sbctl set Port_Binding chassis=[]` short-term.
- Action `insert` storm cho table Logical_Flow sau NB change: northd recompute không incremental. Xem 13.8.6 inc-engine fallback.

### Anatomy 2: `dump` output table format

```
name        type        addresses                              up
"web1"      ""          ["aa:bb:cc:00:00:01 10.0.0.10"]       true
"web2"      ""          ["aa:bb:cc:00:00:02 10.0.0.11"]       true
"phys-1"    "localnet"  ["unknown"]                            true
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| Row count | Match expected workload | Lệch 5-10% | Lệch > 25% | Stale entry; chạy `ovn-nbctl find ... no=...` để locate |
| `up` column | `true` cho mọi VM running | `false` cho 1-2 port mới | `false` rộng | claim issue cluster-wide |
| `addresses` empty | Cho LSP `unknown` type | (none) | Empty cho VM type | LSP chưa được CMS set addresses |

### Anatomy 3: `list-dbs` output

```
OVN_Northbound
OVN_Southbound
```

(Trong ovn-central node single-process serving cả 2 DB.)

| Healthy | Warning | Critical | Diagnostic hint |
|---------|---------|----------|-----------------|
| 1-2 DB expected | DB lạ xuất hiện | Empty list | ovsdb-server crash; check `systemctl status ovn-central` |

### Anatomy 4: `get-schema` JSON

```json
{
  "name": "OVN_Northbound",
  "version": "5.32.1",
  "cksum": "1466357837 27935",
  "tables": { "..." }
}
```

| Field | Healthy | Warning | Critical |
|-------|---------|---------|----------|
| `version` match expected baseline | OVN 22.03 -> 5.32.x | Lệch minor | Lệch major | Schema drift sau partial upgrade |
| `cksum` stable | Không đổi mỗi lần read | Đổi (do schema mismatch) | (none) | Schema reload đang chạy |

### Anatomy 5: `transact` JSON-RPC response

```json
[
  {"uuid": ["uuid","abcd1234-..."]},
  {"uuid": ["uuid","efgh5678-..."]},
  {"count": 1}
]
```

Mỗi element trong array tương ứng 1 op trong request:

- `insert` → trả `{"uuid": [...]}` (UUID của row mới).
- `mutate` → trả `{"count": N}` (số row affected).
- `select` → trả `{"rows": [...]}`.
- `wait` → trả `{}` khi điều kiện match, hoặc `{"error": "timed out"}`.

Nếu transaction fail, response array có `{"error": "..."}` thay cho object thành công.

## 10.7.9 Decision matrix, ovsdb-client vs các tool khác

| Tình huống | Tool đúng | Lý do |
|-----------|-----------|-------|
| Cấu hình LS/LR/ACL ý đồ logical | `ovn-nbctl` | Schema-aware, validation, syntax dễ |
| Đăng ký LSP với chassis (debug binding race) | `ovn-sbctl set Port_Binding` | Schema-aware, Phần 13.14.5 |
| Forensic Port_Binding timeline 5-30 phút | `ovsdb-client monitor --timestamp` | Streaming event, schema-agnostic, Phần 20.5 |
| Export NBDB cho disaster recovery | `ovsdb-client backup` | Format chuẩn, restore atomic |
| Schema check sau OVN upgrade | `ovsdb-client get-schema-cksum` | Quick, no-state, scriptable |
| Xem 100 cột của 1000 row Logical_Flow | `ovs-vsctl list` (không có), `ovsdb-client dump` | Dump là duy nhất hỗ trợ OVSDB schema-agnostic |
| Thực thi atomic 5-op transaction phức tạp | `ovsdb-client transact` (last resort) | Khi ovn-nbctl không cover edge case; chuẩn bị JSON cẩn thận |
| Health check ovsdb-server alive | `ovsdb-client list-dbs` | < 100 ms response, perfect cho cron |
| Migrate schema rolling upgrade | `ovsdb-client convert` | Không restart server, atomic |
| Lock distributed cho leader election custom | `ovsdb-client lock <name>` | Built-in, không cần dependencies như ZooKeeper |

**Anti-pattern:**

- Dùng `ovsdb-client transact` cho ý đồ logical (ví dụ tạo Logical_Switch). Đúng phải `ovn-nbctl ls-add`.
- Dùng `ovsdb-client monitor` không có conditional filter trên cluster lớn (10K+ row). Bandwidth bùng nổ. Phải dùng `monitor-cond` với `where` filter.

## 10.7.10 Guided Exercise S68': Forensic Port_Binding race với monitor

**Mục tiêu:** Reproduce + observe migration race window 3-18 giây mà kịch bản Phần 20.5 §20.5.1 mô tả.

**Setup giả định:** Cluster OVN 22.03.8 với 2 chassis. VM `vm-test` đang trên chassis A, sẽ migrate sang B.

**Bước thực hiện:**

```bash
# 1. Bắt đầu monitor song song (tab 1)
[host]# ovsdb-client --timestamp monitor tcp:sb-leader:6642 \
        OVN_Southbound Port_Binding chassis logical_port | \
        grep --line-buffered '"vm-test"' > /tmp/migration-trace.log

# 2. Trigger migration (tab 2): set requested-chassis trên NBDB
[host]# ovn-nbctl lsp-set-options vm-test requested-chassis=chassis-B

# 3. Đợi 30 giây cho migration ổn định, rồi Ctrl-C tab 1.

# 4. Phân tích timeline
[host]# cat /tmp/migration-trace.log
```

**Output mong đợi (doc-plausible, kiểm chứng khi có lab):**

```
2026-04-25T14:23:10.000Z  action=old   chassis="chassis-A-uuid"  logical_port="vm-test"
2026-04-25T14:23:10.123Z  action=new   chassis=""                logical_port="vm-test"
2026-04-25T14:23:11.456Z  action=old   chassis=""                logical_port="vm-test"
2026-04-25T14:23:11.456Z  action=new   chassis="chassis-B-uuid"  logical_port="vm-test"
```

**Phân tích:**

- Thời điểm 14:23:10.000 chassis A release (chassis → empty).
- Window 14:23:10.123 → 14:23:11.456 (~1.3 giây) port không bind chassis nào → traffic drop.
- Thời điểm 14:23:11.456 chassis B claim.
- Tổng window outage: 1.3 giây cho cluster healthy. Nếu Raft slow hoặc ovn-controller B busy, window có thể 5-30 giây.

**Bài học học được:**

1. `ovsdb-client monitor` là tool duy nhất stream realtime event với millisecond precision. `ovn-sbctl list Port_Binding` chỉ snapshot, không cho phép trace race.
2. `--timestamp` flag bắt buộc cho forensic. Không có nó, output không thể correlate với log khác.
3. Conditional grep `grep --line-buffered` không drop output do buffering. Production thường pipe sang `ts` (moreutils) hoặc `awk` xử lý realtime.

**Cleanup:**

```bash
[host]# rm /tmp/migration-trace.log
```

## 10.7.11 Capstone POE Phase I.B2: monitor vs transact, chọn tool đúng

> **Bối cảnh:** Kỹ sư on-call nhận alert "Port_Binding stuck dual-bind" trên cluster production OVN 22.03. Kỹ sư định force release qua `ovsdb-client transact` để fix nhanh.

**Tuyên bố cần kiểm chứng:** "Dùng `ovsdb-client transact` để mutate `Port_Binding.chassis = []` là cách fix đúng, nhanh hơn `ovn-sbctl`."

**Predict:** Sai cả 2 mặt: (a) không nhanh hơn, (b) không đúng cách.

**Observe (giả định, kiểm chứng khi có lab):**

- `ovsdb-client transact` cần JSON RFC 7047 đúng syntax + UUID rõ ràng. Soạn JSON 30-60 giây; dễ sai cú pháp; phải retry.
- `ovn-sbctl set Port_Binding <port> chassis=[]` 1 lệnh, dùng tab-completion, < 5 giây.
- Cả 2 đều bị overwrite bởi northd recompute trong vòng 1-30 giây nếu NB không reflect.

**Explain:**

1. `ovsdb-client transact` là tool **low-level**, dùng khi không có CLI helper hoặc khi cần atomic 5+ op cùng lúc. Cho 1 thao tác đơn giản, schema-aware tool (`ovn-sbctl`) nhanh và an toàn hơn.
2. Cả 2 đều là **anti-pattern** cho fix Port_Binding dual-bind production. Đúng phải:
   - Investigate root cause: `ovsdb-client monitor --timestamp Port_Binding` để observe timeline.
   - Determine xem NB hay SB sai. Nếu NB sai, fix qua `ovn-nbctl lsp-set-options requested-chassis`. Nếu SB stuck do ovn-controller bug, restart ovn-controller chassis liên quan.
3. Tool `ovsdb-client` mạnh nhưng nguy hiểm. Kỹ sư production phải hiểu khi nào dùng.

**Falsification test:**

- Soạn 2 command equivalent: `ovsdb-client transact <json>` vs `ovn-sbctl set Port_Binding <port> chassis=[]`.
- Đo thời gian human soạn + chạy + kiểm chứng.
- Nếu `transact` nhanh hơn → nghi vấn benchmark (có thể script sẵn).

**Tuyên bố sai ở đâu:** (a) `transact` syntax overhead làm chậm hơn schema-aware tool. (b) Cả 2 cách đều intervention SBDB output, sẽ bị revert bởi northd. Đúng = investigate qua `monitor`, fix root cause qua NBDB.

> **Nguyên tắc:** `ovsdb-client` mạnh cho **observation + scripting + edge case**. Cho daily ops + ý đồ logical, schema-aware tool (`ovn-nbctl`/`ovs-vsctl`) là first choice.

## 10.7.12 Điểm cốt lõi cần nhớ

**Một.** `ovsdb-client` là tool low-level RFC 7047 JSON-RPC, schema-agnostic. Chạy được trên mọi OVSDB database (NBDB, SBDB, vswitch, custom).

**Hai.** 7 nhóm chức năng: schema introspection / query+dump / transaction / monitoring (3 variant) / coordination (wait + lock) / backup+restore / schema migration (convert).

**Ba.** `monitor` là tính năng đặc trưng. Stream event realtime với `--timestamp` cho forensic timeline. `monitor-cond` với `where` filter giảm bandwidth cho cluster lớn.

**Bốn.** `transact` cho atomic multi-op nhưng JSON nặng. Last resort, không thay schema-aware tool cho daily ops.

**Năm.** `backup`/`restore` chuẩn cho disaster recovery. `convert` cho rolling upgrade schema không downtime.

**Sáu.** Anti-pattern: dùng `transact` cho ý đồ logical (ovn-nbctl đúng); `monitor` cluster lớn không có `where` filter (bandwidth bùng nổ).

**Bảy.** Kỹ sư production dùng `ovsdb-client` chủ yếu cho 3 use case: observation/forensic (monitor), backup/restore, schema migration. Daily ops vẫn dùng `ovn-nbctl`/`ovs-vsctl`.

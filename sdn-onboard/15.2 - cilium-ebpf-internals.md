## 15.2.1 eBPF datapath architecture

**eBPF** (Extended Berkeley Packet Filter) is a small virtual machine inside the Linux kernel that lets userspace load safe programs into specific kernel hook points. Since kernel 4.8 (2016), eBPF supports several program types: (1) **XDP (eXpress Data Path)**, attached to the network driver, runs before the packet enters the kernel networking stack, allowing DROP, PASS, REDIRECT, or TX at line rate over 10 to 40 Gbps per core. (2) **tc ingress and egress**, attached to the traffic control subsystem, runs after XDP but before iptables netfilter, allowing the program to modify the packet and attach metadata. (3) **socket filter and sockops**, attached at the socket layer, allowing interception of `connect()` and `sendmsg()` syscalls and redirection of connections without going through the TCP/IP stack. (4) **kprobe and tracepoint**, attached to any kernel function, mainly used for observation and tracing (bcc, bpftrace).

Cilium uses all four types: XDP for DDoS filtering and edge load balancing, tc for NetworkPolicy enforcement and overlay encapsulation and decapsulation, sockops for accelerating pod-to-pod traffic on the same node (skipping the TCP stack), and kprobe for Hubble flow events.

**eBPF map types** are data structures shared between an eBPF program (kernel) and userspace (Cilium agent): (1) **HASH**, a key-value map with O(1) lookup; Cilium uses this for the endpoint table (pod ID to IP) and for service lookup (ClusterIP plus port to the list of endpoints). (2) **LPM_TRIE**, longest prefix match trie; Cilium uses it for the CIDR routing table and for the IPCache (IP to identity). (3) **LRU**, a cache with eviction; used for conntrack and NAT state. (4) **RING_BUFFER**, a FIFO queue between kernel and userspace; Hubble uses it to push flow events from the tc hook up to the Hubble relay.

**Verifier constraints** are an eBPF specific factor: before loading into the kernel, the eBPF program passes through the **verifier**, which proves that it cannot crash the kernel or enter an infinite loop. Main constraints: (1) loops must have a bound (kernel 5.3 and later supports the `bpf_loop` helper for dynamic loops); (2) the stack is limited to 512 bytes; (3) no broken pointers (every map access goes through a helper); (4) the maximum instruction count is 1M (kernel 5.3 raised the older 4K limit). These constraints make eBPF programs safe but harder to write, and the Cilium team had to design the datapath in a way that is optimal for the verifier, not only for the algorithm.

## 15.2.2 Cilium CNI compared with OVN-K8s

These two CNIs represent two design philosophies for the data plane: **Cilium eBPF** (a pure kernel data plane with no userspace proxy) and **OVN-K8s OVS** (an OVS data plane with a kernel datapath plus a userspace control plane).

**Latency comparison:** Cilium eBPF processes packets directly inside the kernel tc hook, with no context switch to userspace, so pod-to-pod latency on the same node is below one millisecond (around 50 to 100 microseconds for a TCP echo). OVN-K8s also runs the datapath in the kernel (the OVS megaflow cache), achieving the same latency on cache hit. On cache miss, OVN-K8s detours through the ovs-vswitchd userspace process to resolve the flow, which can raise latency to 2 to 5 ms for new flows. Cilium avoids this because the eBPF program already contains the full forwarding logic, compiled in advance.

**NetworkPolicy expressiveness:** OVN-K8s ACLs model NetworkPolicy very naturally (OVN ACLs were originally designed for multi-tenancy). CiliumNetworkPolicy (a CRD) extends Kubernetes NetworkPolicy with L7 fields (HTTP method, path, gRPC method, Kafka topic) that are not available in standard Kubernetes NetworkPolicy. If the cluster needs L7 policy, Cilium has the edge; if only L3 or L4 is needed, OVN-K8s is comparable.

**Debugging tooling:** OVN-K8s provides `ovn-trace`, which lets the operator simulate a packet through the logical pipeline and shows the action at each stage, which is very useful. Cilium provides `cilium monitor` (real-time trace), `hubble observe` (flow log), and `bpftool` (inspect eBPF maps). The two tool sets differ: ovn-trace operates at the logical layer (NB DB), cilium monitor operates at the datapath layer (kernel). Cilium requires the operator to understand eBPF; OVN-K8s requires the operator to understand OpenFlow plus OVN logical flows.

**Hardware acceleration:** OVN-K8s supports OVS hardware offload through switchdev mode (Mellanox ConnectX-5 and later, Intel CX6 and later). Flows are offloaded into the NIC ASIC, reaching 100 plus Gbps without CPU cost. Cilium eBPF supports XDP offload on some NICs (Mellanox, Netronome), but the scope is narrower; only some eBPF programs can be compiled for the NIC firmware.

## 15.2.3 Replacing kube-proxy

Cilium 1.6 and later supports the `--kube-proxy-replacement=strict` flag, which lets the operator remove kube-proxy from the cluster entirely. Cilium implements Service through an eBPF program attached to the tc ingress hook of the pod interface and to the socket hook of the host. When the application calls `connect()` to a ClusterIP, the eBPF sockops program intercepts the syscall, looks up the service map (an eBPF HASH), selects the endpoint by hash, and rewrites the socket destination directly inside the kernel.

**Advantages over kube-proxy:** (1) **O(1) lookup** through a HASH map versus the O(n) iptables chain traversal of kube-proxy (the kube-proxy iptables mode walks through thousands of rules per packet when the cluster has many Services). (2) **No connection drop** when an endpoint changes; an eBPF map update is atomic and does not kill an active connection. (3) **No practical Service count limit;** kube-proxy iptables degrades when the cluster has more than 10K Services, but Cilium eBPF handles 100K plus Services without performance degradation. (4) **Reduced CPU load;** an Isovalent benchmark from 2023 showed that the Cilium eBPF mode reduced CPU overhead by 30 to 50 percent compared with iptables mode in a 500-pod cluster.

**Session affinity through ClientIP:** Cilium implements this through a secondary eBPF map that maps (client IP, service ID) to the chosen endpoint, with a TTL according to the Service configuration. When the same client calls again, the eBPF program checks this map before re-hashing, ensuring the same endpoint is selected.

**ExternalTrafficPolicy=Local:** similar to OVN-K8s, Cilium picks only a local endpoint when the Service has `ExternalTrafficPolicy=Local`. The implementation uses a flag bit in the service map entry.

Cilium also supports **DSR (Direct Server Return)** mode for NodePort: response traffic from the endpoint goes directly back to the client, bypassing the load balancer node, which reduces latency and load on the gateway node. This mode is not available in kube-proxy or OVN-K8s.

## 15.2.4 Cilium Service Mesh (no sidecar)

Cilium Service Mesh was introduced in 2021 (preview feature) and reached GA in 2023 with Cilium 1.14. The philosophy is **one shared Envoy per node instead of one Envoy per pod**, as Istio sidecar mode does. When policy requires L7 processing (HTTP routing, header manipulation, circuit breaker), traffic from the pod is redirected to the Envoy on the same node through a tc redirect (the traffic does not leave the node and does not add a network hop).

**Architecture:** (1) one Cilium agent pod per node runs an **embedded Envoy** (not a separate container). (2) CiliumClusterwideEnvoyConfig (CRD) and CiliumEnvoyConfig (CRD) let the operator configure Envoy through an API similar to Cilium's native xDS. (3) When a CiliumNetworkPolicy contains an L7 rule (HTTP method, path), the agent automatically configures an Envoy listener and redirects traffic that matches the condition to Envoy.

**Socket-level L7 visibility:** Cilium has a unique feature called **socket-level mTLS**, which intercepts the `connect()` syscall right at the socket layer, performs the mTLS handshake through eBPF, and then forwards the traffic through the kernel without touching userspace. This achieves performance equal to plaintext traffic (no Envoy overhead for simple mTLS). Limitation: this works only for simple TCP flows and does not support L7 routing. When L7 is needed, the traffic must still go through Envoy.

**Ingress Gateway integration:** Cilium implements the Kubernetes Gateway API (the standard successor to Ingress, released 2022) natively. There is no need to install nginx-ingress-controller or Traefik separately. The Cilium agent handles HTTP and HTTPS termination through the per-node Envoy. It is compatible with cert-manager for automatic Let's Encrypt certificate issuance.

**Limitations compared with Istio:** (1) it does not yet support extension through WASM plugins like the Envoy in Istio. (2) Distributed tracing is available through Hubble but uses a custom format, not the OpenTracing or OpenTelemetry standard. (3) The community and tutorials are smaller than for Istio, so a new team needs to invest in learning.

## 15.2.5 Hubble, observability platform

**Hubble** is the Cilium observability subsystem, born together with Cilium 1.8 (2020). Hubble runs on the same eBPF datapath, requires no separate sidecar agent, and adds no second layer of overhead. Every flow event is generated by an eBPF program in the kernel through a **ring buffer**, and the Hubble relay in userspace reads the events and forwards them to Hubble UI or to a backend (Prometheus, OpenTelemetry collector, syslog).

**Hubble architecture:** (1) **Hubble observer**, a component inside the per-node Cilium agent, attaches to the eBPF program ring buffer, parses flow events (source and destination IP, port, verdict, L7 metadata). (2) **Hubble relay**, a cluster-wide service that aggregates flows from all node agents and provides a gRPC API for the UI and CLI. (3) **Hubble UI**, a web application that shows the real-time service map, flow log, and policy verdicts. (4) **Hubble CLI (`hubble observe`)**, queries flows directly from the terminal.

**L7 visibility for HTTP, gRPC, and Kafka:** when Cilium runs the per-node Envoy (with a CiliumNetworkPolicy L7 rule), Envoy emits access logs through a custom filter back to Hubble, which parses them into flow events that include extra fields (HTTP method, path, status code; or gRPC method and status; or Kafka topic and partition). This capability is not available in OVN-K8s or in Istio (Istio uses Prometheus metrics plus distributed tracing, without structured per-request flow logs).

**Comparison with Kiali plus Grafana of Istio:** Kiali shows a static service graph (sampled periodically from Prometheus metrics), so the latency for displaying structural changes is high. Hubble shows real time (flows from the ring buffer, sub-second latency) and is more dense (every connection, not just aggregate metrics). The trade-off: Hubble uses more storage if flow logs are kept long term; Kiali is lighter but less detailed.

**OpenTelemetry integration:** since Cilium 1.15 (2024), Hubble has a native exporter to OpenTelemetry traces and metrics, allowing unified observability across the Cilium network layer and the application layer in the same stack (Jaeger, Tempo, Prometheus).

### ▶ Guided Exercise 1: Inspect Cilium eBPF programs

> **Lab verification status:** output is based on Cilium 1.15 docs plus bpftool reference; awaiting verification on a real lab.

**Goal:** understand the Cilium datapath in detail, list attached eBPF programs, inspect eBPF maps, and trace packet processing. This lab opens the Cilium black box to show how an eBPF-based CNI works.

**Setup:**
- Ubuntu 22.04 VM with kernel 5.15 plus (eBPF features)
- kind cluster plus Cilium CNI
- `bpftool` installed: `sudo apt install linux-tools-generic` (or build from kernel source)

**Lab topology:**

```
+-------------------------------------------------------+
|           kind cluster with Cilium CNI                |
|                                                       |
|   Kubernetes node:                                    |
|                                                       |
|   +-----------------------------------------------+   |
|   |  Linux kernel eBPF hook points                |   |
|   |                                               |   |
|   |   XDP at driver      Socket filters           |   |
|   |       down               down                 |   |
|   |   TC ingress          cGroup progs            |   |
|   |       down                                    |   |
|   |   Cilium datapath at policy, NAT, route       |   |
|   |       down                                    |   |
|   |   TC egress                                   |   |
|   +-----------------------------------------------+   |
|                                                       |
|   eBPF maps (BPF_MAP_TYPE_HASH, LPM_TRIE, and so on): |
|   +-----------------------------------------------+   |
|   | cilium_policy_v2_00123 (per-endpoint policy)  |   |
|   | cilium_lb4_services (load balancer backends)  |   |
|   | cilium_lxc (pod metadata)                     |   |
|   | cilium_ipcache (IP to identity)               |   |
|   +-----------------------------------------------+   |
+-------------------------------------------------------+
```

**Steps:**

**1.** Create a kind cluster with Cilium:

```bash
[host]$ kind create cluster --name cilium
[host]$ cilium install --version 1.15.0
[host]$ cilium status --wait
```

Expected output:

```
    /==\__/==\    Cilium:         OK
 \__/==\__/      Operator:       OK
  /==\__/==\    Envoy DaemonSet: disabled
   \__/==\__/    Hubble Relay:    disabled
```

**2.** Deploy a sample app:

```bash
[host]$ kubectl run web --image=nginx
[host]$ kubectl run client --image=curlimages/curl -- sleep 3600
```

**3.** Exec into the Cilium pod:

```bash
[host]$ CILIUM_POD=$(kubectl -n kube-system get pod -l k8s-app=cilium -o jsonpath='{.items[0].metadata.name}')
[host]$ kubectl -n kube-system exec -ti $CILIUM_POD -- bash
```

**4.** Inside the Cilium pod, list eBPF programs:

```bash
[cilium-agent]# bpftool prog list | grep -i 'sched_cls\|xdp\|sk_msg' | head -20
```

Expected output:

```
2457: sched_cls  name cil_to_container  tag abc123  gpl
    loaded_at 2026-04-22T12:34:56+0000  uid 0
    xlated 2048B  jited 1536B
    pinned /sys/fs/bpf/tc/globals/cilium_calls_...
```

**5.** List eBPF maps:

```bash
[cilium-agent]# bpftool map list | grep cilium | head -20
```

Expected output:

```
123: hash  name cilium_lxc  flags 0x0
  key 16B  value 256B  max_entries 65536  memlock 18MB
124: lru_hash  name cilium_ipcache  flags 0x1
  key 20B  value 24B  max_entries 512000
125: prog_array  name cilium_calls_0000  flags 0x0
  key 4B  value 4B  max_entries 50
```

**6.** Dump the policy map for a pod:

```bash
[cilium-agent]# ENDPOINT_ID=$(cilium endpoint list -o json | jq '.[] | select(.status.labels."k8s:app"=="web") | .id')
[cilium-agent]# bpftool map dump name cilium_policy_v2_$(printf "%05d" $ENDPOINT_ID) | head
```

Expected output: entries for allowed ingress and egress with identity IDs.

**7.** Trace a packet from client to web:

```bash
[host]$ kubectl exec client -- curl -s http://web
[cilium-agent]# cilium monitor --type trace | head -20
```

Expected output: trace events showing `FROM_ENDPOINT` to `TO_OVERLAY` to `TO_ENDPOINT` and the response in reverse.

**Lessons learned:**

First, **Cilium is an eBPF-first CNI**. It does not use iptables for kube-proxy or for NetworkPolicy. All logic lives in eBPF programs attached at tc (traffic control), XDP hook points, and socket filters.

Second, **map-based state**. Policy, service load balancer backends, and IP-to-identity mapping all live in eBPF maps. Userspace (cilium-agent) updates the maps; the datapath (the eBPF program) reads the maps at packet time. Lock-free, per-CPU optimized.

Third, **observability built in through bpftool plus cilium monitor**. A production Cilium deployment has clear visibility, not a black box. Compared with iptables chains (opaque, order sensitive), the Cilium advantage is clear.

**Cleanup:**

```bash
[host]$ kind delete cluster --name cilium
```

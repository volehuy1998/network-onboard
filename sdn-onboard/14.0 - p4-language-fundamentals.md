## 14.0.1 Cấu trúc chương trình P4_16

Một chương trình P4_16 hoàn chỉnh gồm sáu thành phần bắt buộc, mỗi thành phần đảm nhiệm một công đoạn trong pipeline xử lý gói: **header type** (khai báo định dạng từng loại header), **parser** (máy trạng thái trích xuất header từ bytestream vào), **ingress control** (pipeline xử lý hướng vào), **egress control** (pipeline xử lý hướng ra), **deparser** (ráp lại header thành bytestream ra), và **package instantiation** (liên kết các block theo kiến trúc đích). Thứ tự sáu thành phần này cố định — spec P4_16 v1.2.2 §6.5 quy định parser luôn chạy trước ingress control, và deparser luôn chạy cuối cùng trước khi gói rời switch.

Hệ kiểu của P4_16 (§7) được thiết kế để mô tả chính xác cấu trúc bit-level của header mạng. Kiểu cơ bản `bit<N>` biểu diễn số nguyên không dấu N bit, `varbit<N>` dành cho trường có độ dài thay đổi (ví dụ IPv4 Options với kích thước tối đa 320 bit), `int<N>` cho số nguyên có dấu. Trên nền các kiểu cơ bản này, P4_16 xây ba kiểu tổng hợp quan trọng: `header` (một header mạng cụ thể như Ethernet hay IPv4 — có thêm bit validity ẩn), `struct` (gom nhiều trường không đồng dạng thành một đơn vị logic, dùng cho metadata), và `header_union` (một gói có thể mang một trong nhiều loại header loại trừ lẫn nhau, ví dụ IPv4 hoặc IPv6).

Parser trong P4_16 là một máy trạng thái hữu hạn. Mỗi state gọi `packet.extract()` để đọc header tiếp theo, rồi dùng `transition select()` quyết định state kế tiếp dựa trên giá trị vừa extract. Ví dụ: sau khi extract Ethernet header, `transition select(hdr.ethernet.etherType)` chuyển sang state `parse_ipv4` nếu etherType là 0x0800, sang `parse_vlan` nếu 0x8100, hoặc `accept` nếu không thuộc giao thức hỗ trợ. Hai state đặc biệt `accept` và `reject` kết thúc parsing — gói chuyển tiếp vào ingress control ở trường hợp accept, bị drop ở trường hợp reject (spec §12.5–12.7).

Ingress và egress control block là nơi logic forwarding nằm. Cấu trúc cơ bản của một table trong control block gồm bốn thuộc tính: `key` (danh sách field match và match type — exact, ternary, lpm, range), `actions` (danh sách action có thể được gọi), `default_action` (action mặc định khi không match), và `size` (số entry tối đa). Action là hàm tham số hóa — control plane cài table entry với match value + action + action parameters; khi gói match, data plane gọi action và truyền parameter đó. Khác biệt cốt lõi so với OpenFlow: OpenFlow có tập match field và action cố định theo spec version, còn P4 cho phép programmer tự định nghĩa match field (dựa trên header type đã parse) và action (dựa trên tập primitive operations — set field, add/remove header, compute hash).

Deparser là mirror của parser — thay vì extract, nó emit. `packet.emit<HeaderType>(hdr.ipv4)` ghi lại bytes của IPv4 header vào bytestream ra, nhưng chỉ khi header đó còn hợp lệ (validity bit = 1). Chính bit validity này cho phép P4 program thêm hoặc xóa header giữa pipeline — ví dụ VXLAN encapsulation là `hdr.outer_ipv4.setValid(); hdr.outer_udp.setValid(); hdr.vxlan.setValid();` ở ingress, sau đó deparser emit cả outer và inner headers.

> **Lưu ý phiên bản:** P4_16 v1.2.2 (2021) ổn định từ 2019 với v1.2.0. Phiên bản mới hơn v1.2.5 (2024-10-11) chủ yếu bổ sung annotation syntax và lookahead semantics refinements — core grammar không đổi. Khác biệt lớn nhất so với P4_14 (đã deprecated từ 2018): P4_16 tách biệt rõ ràng giữa ngôn ngữ core (target-independent) và architecture extensions (target-specific, ví dụ V1Model, PSA, TNA).

## 14.0.2 PSA (Portable Switch Architecture)

V1Model — kiến trúc mặc định đi kèm BMv2 — chỉ định nghĩa 6 block (parser, verify_checksum, ingress, egress, compute_checksum, deparser) và một số extern cơ bản. V1Model phù hợp cho tutorial và switch software tham chiếu, nhưng không chuẩn hóa đầy đủ những gì một switch thật cần: multicast, metering, timestamping, packet-in/packet-out tới control plane, clone/resubmit/recirculate. Đây là lý do ONF phát hành **PSA — Portable Switch Architecture** năm 2018 (v1.0), hiệu chỉnh lên v1.1 (2019), và tiếp tục đến v1.2 (2025-11-03). PSA là kiến trúc chuẩn ngành — bất kỳ switch target nào tuân thủ PSA đều cho phép cùng một P4 program chạy được (về mặt ngữ nghĩa), dù hiệu năng có thể khác nhau.

Kiến trúc PSA định nghĩa chuỗi 6 block khác V1Model: **IngressParser → Ingress → IngressDeparser → (packet buffer + queuing) → EgressParser → Egress → EgressDeparser**. Packet buffer giữa ingress và egress là nơi mà switch vật lý thực hiện queuing, multicast replication, và priority scheduling — V1Model không có block riêng cho giai đoạn này, nhưng PSA tách rõ ranh giới giữa ingress và egress deparser. Hệ quả: trong PSA, một gói multicast sẽ chạy IngressParser + Ingress một lần, rồi chạy EgressParser + Egress N lần (mỗi lần cho một output port), và lập trình viên có thể viết logic khác nhau cho giai đoạn ingress và egress — ví dụ set TTL ở ingress, nhưng decrement TTL ở egress.

PSA chuẩn hóa thư viện extern — những module phần cứng được "gọi" từ P4 code. Bốn extern quan trọng nhất: (1) `Counter` (đếm packet và byte, đánh chỉ mục bởi key tùy chọn — PSA định nghĩa ba biến thể PACKETS, BYTES, PACKETS_AND_BYTES), (2) `Meter` (giới hạn tốc độ theo thuật toán two-rate three-color marker RFC 2698), (3) `Register` (ô nhớ giữ trạng thái qua nhiều packet, cho phép hiện thực hành vi có trạng thái như connection tracking), (4) `Hash` (tính hash trên tập field tùy chọn, thuật toán bao gồm CRC16, CRC32, Identity, Random, Csum16). Ngoài ra PSA có `ActionProfile` và `ActionSelector` cho cân bằng tải qua ECMP hoặc WCMP, `Checksum` cho tính IPv4/TCP/UDP checksum, và `Timestamp` cho lấy giá trị timestamp chính xác đến nanosecond.

> **Lưu ý phiên bản:** PSA v1.1 (2019) là triển khai tham chiếu cho BMv2 PSA backend. Tofino (Intel) dùng TNA — Tofino Native Architecture — tách biệt với PSA, vì TNA phát triển trước PSA. PSA chưa phải chuẩn phổ quát: ARM-based DPU (Marvell, NVIDIA BlueField) thường dùng PNA — Portable NIC Architecture — vì DPU xử lý packet theo mô hình khác switch (có host side và network side). PNA v0.5 (2021) là spec dự thảo, PNA v1.0 dự kiến 2025-2026.

## 14.0.3 PISA — Protocol Independent Switch Architecture

PISA không phải là ngôn ngữ hay đặc tả — đó là **mô hình kiến trúc phần cứng** được Bosshart và Barefoot Networks công bố tại SIGCOMM 2013 (bài báo "Forwarding Metamorphosis") làm nền tảng cho việc ASIC switch có thể được lập trình lại bằng P4. Tên viết tắt: Protocol-Independent Switch Architecture — ý nghĩa là ASIC không gắn cứng các giao thức cụ thể (Ethernet + VLAN + IPv4 + IPv6 + MPLS + GRE, với parser và match logic cứng cho từng), mà cung cấp một mảng match-action unit đủ tổng quát để lập trình viên mô tả giao thức nào cũng được.

PISA ASIC nội bộ gồm ba loại khối xây dựng: (1) **Programmable Parser** — một pipeline state machine có thể parse bất kỳ chuỗi header nào theo chương trình P4; (2) **Match-Action Pipeline** — nhiều stage (Tofino T1 có 12 ingress stages + 12 egress stages; T2 có 20 stages) mỗi stage có một bank bộ nhớ (SRAM + TCAM) cho table lookup và một tập ALU cho action; (3) **Programmable Deparser** — sắp xếp lại header theo thứ tự được chỉ định. Clock rate của Tofino T1 là 1.25 GHz, cho throughput 6.5 Tbps (65 ports × 100 Gbps); T2 đạt 12.8 Tbps và T3 (chưa được sản xuất thương mại) dự kiến 25.6 Tbps.

Ràng buộc tài nguyên của PISA là chi tiết quan trọng mà lập trình viên P4 phải nắm. Mỗi stage có ngân sách cố định: số lượng TCAM (ternary match) entries (khoảng 128K trên Tofino T1 phân bố qua các stage), SRAM cho exact match (khoảng 256 MB tổng), số ALU slot per stage (khoảng 32 ALU), và số packet header vector (PHV) container (192 container × 8/16/32 bit widths). Nếu một table cần match trên nhiều field (wide match), nhiều entry, hoặc action phức tạp — compiler p4c phải phân bổ table đó qua nhiều stage, hoặc báo lỗi "cannot fit". Đây là lý do P4 program chạy được trên BMv2 (phần mềm, tài nguyên không giới hạn) không nhất thiết compile được cho Tofino (phần cứng, tài nguyên cố định).

Khác biệt PISA so với xử lý gói dựa trên CPU (DPDK, XDP): PISA đảm bảo **line-rate guarantee** — throughput không phụ thuộc vào lưu lượng hay độ phức tạp của P4 program (miễn là compile được). Mỗi stage dùng đúng 1 chu kỳ đồng hồ; gói đi qua pipeline 12 stage = 12 chu kỳ đồng hồ + độ trễ deserialization/serialization. Tổng port-to-port latency Tofino T1 đạt <400 nanoseconds. Giải pháp dựa trên CPU như DPDK trên Xeon đạt khoảng 100 Mpps/core nhưng latency tăng theo độ sâu pipeline và có jitter do cache miss; PISA không có cache miss vì bộ nhớ là SRAM/TCAM trên chip.

> **Lưu ý thương mại:** Barefoot Networks bị Intel mua năm 2019 với giá $130 triệu. Intel phát hành Tofino 2 (T2) năm 2020 và thông báo dừng phát triển Tofino năm 2023 — không có T4 và không có lộ trình mới. Tofino 1/2 vẫn được bán và hỗ trợ đến 2027, nhưng dần bị thay thế bằng Nvidia Spectrum-4 (không hỗ trợ P4 native, dùng NVIDIA DOCA API), Broadcom Tomahawk (hỗ trợ khả năng lập trình một phần qua NPL — Network Programming Language, không phải P4), và AMD Pensando DPU (hỗ trợ P4 qua P4Runtime). Hệ sinh thái P4 hiện chủ yếu được duy trì qua BMv2 (phần mềm) + eBPF backend (Linux kernel) + XDP backend.

## 14.0.4 BMv2 — compiler và behavioral model tham chiếu

`p4c` là compiler tham chiếu của P4 do P4 Language Consortium duy trì ở GitHub `p4lang/p4c`. Compiler này có kiến trúc dạng module — frontend parse P4 source thành IR (internal representation) trung gian, rồi backend dịch IR sang đầu ra đặc thù target. Năm backend chính: **bmv2** (tạo JSON config cho simple_switch behavioral model), **tofino** (tạo Tofino binary — được Intel phát hành riêng trong Barefoot SDE, không phải open-source), **ebpf** (tạo eBPF bytecode chạy trong Linux kernel), **dpdk** (tạo DPDK pipeline configuration cho thư viện rte_swx_pipeline), và **ubpf** (userspace BPF). Kiến trúc dạng module này cho phép cùng một P4 source chạy trên nhiều target — quan trọng cho phát triển (viết trên BMv2, kiểm thử trên Tofino).

BMv2 (Behavioral Model version 2) là switch phần mềm tham chiếu được viết bằng C++, không tối ưu hiệu năng mà ưu tiên tính đúng đắn ngữ nghĩa. `simple_switch` binary là target chính của BMv2 — nó hiện thực kiến trúc V1Model. Các binary khác bao gồm `simple_switch_grpc` (simple_switch + P4Runtime gRPC API), `psa_switch` (kiến trúc PSA), và `simple_router` (một biến thể cũ, ít dùng). Throughput thực tế: khoảng 1-5 Gbps trên x86 một core cho P4 program đơn giản, giảm mạnh khi logic phức tạp.

Runtime API của BMv2 có hai lớp: **thrift-based** (legacy, dùng bởi `simple_switch_CLI`) và **gRPC-based P4Runtime** (chuẩn ngành từ 2018, dùng bởi `simple_switch_grpc`). `simple_switch_CLI` cung cấp hơn 80 lệnh cho gỡ lỗi tương tác qua CLI — `table_add` để thêm entry, `table_dump` để xem nội dung, `counter_read` để đọc counter, `mirroring_add` để thiết lập mirror session, `packet_in/out` để chèn gói kiểm thử. Đây là công cụ chính để gỡ lỗi P4 program trong giai đoạn phát triển.

Một điểm thực tế quan trọng: BMv2 không phải tuân thủ hoàn toàn đặc tả P4_16. Spec v1.2.2 có những phần tùy chọn (ví dụ stack header manipulation trong parser) mà BMv2 hiện thực khác với Tofino. Lập trình viên viết P4 cho Tofino phải kiểm tra lại khi triển khai — không nên giả định hành vi BMv2 là chuẩn. Công cụ `p4testgen` (từ p4c) giúp sinh trường hợp kiểm thử tự động từ P4 source để đối chiếu chéo giữa các target.

> **Lưu ý cài đặt thực tế:** Trên Ubuntu 22.04, package `p4lang-p4c` và `p4lang-bmv2` có trong PPA riêng (`ppa:p4lang/p4c`), không có trong main archive. Cài: `sudo add-apt-repository ppa:p4lang/p4c && sudo apt install p4lang-p4c p4lang-bmv2`. Phiên bản ổn định gần đây: p4c 1.2.4.14 (2025-Q1), BMv2 1.15.0. Nếu cần tính năng mới nhất, build từ source trong thư mục `~/work/p4c` và `~/work/behavioral-model`. Thời gian build khoảng 20-40 phút trên CPU 4-core.

**Bước thực hiện:**

**1.** Tạo P4 program `basic.p4` (L2 exact match forwarding):

```p4
#include <core.p4>
#include <v1model.p4>

header ethernet_t {
    bit<48> dstAddr;
    bit<48> srcAddr;
    bit<16> etherType;
}

struct headers { ethernet_t ethernet; }
struct metadata { }

parser MyParser(packet_in pkt, out headers hdr, inout metadata meta, inout standard_metadata_t std) {
    state start { pkt.extract(hdr.ethernet); transition accept; }
}

control MyIngress(inout headers hdr, inout metadata meta, inout standard_metadata_t std) {
    action forward(bit<9> port) { std.egress_spec = port; }
    action drop() { mark_to_drop(std); }
    table l2_fwd {
        key = { hdr.ethernet.dstAddr: exact; }
        actions = { forward; drop; }
        default_action = drop();
    }
    apply { l2_fwd.apply(); }
}

control MyVerifyChecksum(inout headers hdr, inout metadata meta) { apply {} }
control MyComputeChecksum(inout headers hdr, inout metadata meta) { apply {} }
control MyEgress(inout headers hdr, inout metadata meta, inout standard_metadata_t std) { apply {} }
control MyDeparser(packet_out pkt, in headers hdr) { apply { pkt.emit(hdr.ethernet); } }

V1Switch(MyParser(), MyVerifyChecksum(), MyIngress(),
         MyEgress(), MyComputeChecksum(), MyDeparser()) main;
```

```
ARPING 10.0.0.2 from 10.0.0.1 h1-eth0
Unicast reply from 10.0.0.2 [00:00:00:00:00:02]  0.456ms
Unicast reply from 10.0.0.2 [00:00:00:00:00:02]  0.321ms
Unicast reply from 10.0.0.2 [00:00:00:00:00:02]  0.289ms
Sent 3 probes (1 broadcast, 2 unicast)
Received 3 response(s)
```

**Bài học học được:**

Thứ nhất, P4 tách biệt rõ ràng giữa **hành vi data plane** (định nghĩa trong P4 program: parser + tables + actions + deparser) và **runtime control plane** (table entries được cài qua thrift/P4Runtime API). Đây là khác biệt cốt lõi so với OpenFlow, nơi match field và action đã cố định trong spec.

Thứ hai, trình mô phỏng phần mềm (BMv2) có `simple_switch` behavioral model chạy packet qua pipeline chính xác như P4 program định nghĩa. Tốc độ chỉ khoảng 1-5 Gbps, không phải line-rate môi trường production, nhưng đủ để xác minh logic trước khi compile cho phần cứng Tofino.

Thứ ba, `simple_switch_CLI` là giao diện dựa trên thrift riêng của BMv2, không phải chuẩn chung. Triển khai P4 môi trường production dùng P4Runtime (gRPC + protobuf) — xem Part 14.2.

**Cleanup:**

```bash
[host]# pkill -f simple_switch
[host]# ip netns del h1
[host]# ip netns del h2
[host]# ip link del veth-0 2>/dev/null || true
[host]# ip link del veth-1 2>/dev/null || true
```

### ▶ Lab 2: P4 L3 router với LPM table

> **Trạng thái lab verification:** Spec dựa trên P4 tutorials `basic_routing` exercise; chờ verify lab thực.

**Mục đích:** Mở rộng Ex1 từ L2 exact match lên định tuyến L3 LPM (Longest Prefix Match) — giảm TTL, viết lại MAC nguồn/đích, chuyển tiếp đa subnet. Thực hành hiện thực một router IPv4 đơn giản bằng P4.

**Chuẩn bị:**
- Hoàn thành Lab 14.0-Ex1 baseline (p4c + BMv2 installed).
- Clone P4 tutorials: `git clone https://github.com/p4lang/tutorials.git` (reference implementation trong `exercises/basic_routing/`).

**Mô hình lab:**

```
┌──────────────────────────────────────────────────────────┐
│                  Ubuntu 22.04 VM                         │
│                                                          │
│   h1 (10.1.0.1/24)           h2 (10.2.0.1/24)            │
│    MAC 00:00:0a:01:00:01      MAC 00:00:0a:02:00:01      │
│         │                         │                      │
│       veth-0                    veth-1                   │
│         │                         │                      │
│   ┌─────▼─────────────────────────▼─────┐                │
│   │        BMv2 simple_switch            │                │
│   │                                      │                │
│   │   router.p4:                         │                │
│   │   - LPM on dst IP                    │                │
│   │   - TTL-- + checksum update          │                │
│   │   - MAC rewrite src/dst              │                │
│   │                                      │                │
│   │  Router MAC 00:00:00:01:01:01        │                │
│   └──────────────────────────────────────┘                │
└──────────────────────────────────────────────────────────┘
```

**Mục tiêu self-guided (không có step-by-step như Ex1):**

**1.** Đọc `exercises/basic_routing/basic.p4` trong P4 tutorials, hiểu LPM table definition và action `ipv4_forward(macAddr_t dstAddr, egressSpec_t port)`.

**2.** Tạo topology 2-host với IP khác subnet. Gán default gateway cho mỗi host trỏ về router BMv2 IP.

**3.** Compile + deploy. Cài LPM entries cho 10.1.0.0/24 và 10.2.0.0/24.

**4.** Test: `ip netns exec h1 ping -c 3 10.2.0.1`. Phải thành công.

**5.** Verify TTL decrement: từ h1 chạy `tcpdump -nvv -i h1-eth0 icmp`. TTL khi gửi = 64, khi nhận reply = 63 (passed 1 hop = router).

**Output mong muốn:**

```
# Ping output
PING 10.2.0.1 (10.2.0.1) 56(84) bytes of data.
64 bytes from 10.2.0.1: icmp_seq=1 ttl=63 time=0.512 ms
64 bytes from 10.2.0.1: icmp_seq=2 ttl=63 time=0.421 ms
64 bytes from 10.2.0.1: icmp_seq=3 ttl=63 time=0.398 ms

--- 10.2.0.1 ping statistics ---
3 packets transmitted, 3 received, 0% packet loss
```

Quan trọng: `ttl=63` chứng minh packet đã qua router (TTL giảm từ 64 → 63).

**Bài học học được:**

Lab này thể hiện khả năng lập trình của P4 ở mức thao tác header IPv4 — thứ mà OpenFlow 1.0 không làm được native (OpenFlow 1.0 không có action SET_NW_TTL). P4 cho phép lập trình viên chèn action tùy ý vào pipeline: giảm TTL, tính lại checksum, viết lại MAC, tất cả trong cùng một table apply. Đây là minh chứng **khả năng lập trình độc lập target** — cùng P4 program chạy được trên BMv2, Tofino, eBPF backend.

**Cleanup:** Như Ex1.

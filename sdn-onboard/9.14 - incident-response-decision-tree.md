## 9.14.1 The observability stack (compass Ch 18)

A production OVS deployment needs three observability layers:

**Layer 1, traffic analytics:** sFlow (Part 9.8) feeding goflow2 or ntopng. It tells you which flows are using bandwidth, abnormal patterns, and the top talkers.

**Layer 2, metrics:** Prometheus scraping `ovs_exporter` for metrics about:
- Interface statistics (rx/tx packets, bytes, errors, drops).
- Flow count and megaflow count per bridge.
- PMD utilization (DPDK).
- OVSDB commit rate, connection count.

Typical alert rules:
- `ovs_interface_rx_dropped` rate above 1 percent, an interface or driver problem.
- `ovs_datapath_lost` greater than 0, the upcall rate limit was hit.
- `ovs_bridge_flow_count` above 100,000, approaching the limit.

**Layer 3, coverage delta:** a cron runs `ovs-appctl coverage/show` once a minute and writes the delta to a log aggregator (Loki, Elasticsearch). Detect anomalies such as a `cache-misses` rate spike or a sudden `bridge_flush` increase.

## 9.14.2 Decision tree, the five incident classes

Compass Ch 20 condensed:

### 9.14.2.1 Packet loss

```
1. ovs-dpctl show                    -> check counters
2. Check the "lost" counter           -> > 0 means upcall rate limit hit
3. ovs-appctl upcall/show            -> view the flow-install rate
4. Action: raise upcall/set-flow-limit or n-handler-threads
```

If `lost = 0` but loss persists, it is a kernel or physical NIC problem.

### 9.14.2.2 Unexpected drops

```
1. ovs-appctl ofproto/trace br0 "<flow spec>"
2. Read the output, identify which table is dropping
3. Fix: correct rule priority or action
```

Debugging with `ofproto/trace` does not need real reproduced traffic, the most important property.

### 9.14.2.3 Tunnel down

```
1. ovs-appctl tnl/neigh/show          -> check ARP resolution on the underlay
2. Underlay connectivity: ping the underlay IP
3. MTU mismatch: ip link show, shrink the inner MTU or raise the underlay
```

95 percent of "tunnel up but no traffic" cases are MTU issues (VXLAN adds 50 bytes, Geneve adds 50 to 66 bytes).

### 9.14.2.4 Bond flap

```
1. ovs-appctl bond/show bond0
2. ovs-appctl lacp/show bond0
3. Check the physical-switch LACP configuration; enable lacp-fallback-ab
```

### 9.14.2.5 OVSDB unavailable

```
1. systemctl status ovsdb-server
2. Clustered: cluster/status -> is quorum still satisfied?
3. Restore: restart the failed node or restore quorum
```

A 3-node cluster tolerates 1 failure, a 5-node cluster tolerates 2. Loss of quorum makes the cluster read-only.

## 9.14.3 Packet capture strategies

When the decision tree is not enough:

```bash
[host]# tcpdump -i eth0 -nn -w /tmp/phys.pcap &          # physical
[host]# ovs-vsctl add-port br0 tap-debug -- \
         set Interface tap-debug type=internal
[host]# ovs-vsctl -- --id=@m create Mirror name=dbg \
         select-all=true output-port=@p \
         -- --id=@p get Port tap-debug \
         -- set Bridge br0 mirrors=@m
[host]# tcpdump -i tap-debug -nn -w /tmp/ovs.pcap &      # OVS internal
```

Compare the two pcaps. The diff between the physical wire and the OVS internal capture tells you whether OVS dropped or modified the packet.

## 9.14.4 Capstone end-to-end incident

**Scenario:** a VM on host-A cannot ping a VM on host-B over VXLAN, both VMs in the same subnet.

**Step 1, physical layer:**

```bash
[host-A]# ping <host-B-underlay-ip>
[host-A]# tcpdump -i eth0 -nn udp port 4789
```

Underlay failure means escalate to the network team.

**Step 2, OVS configuration:**

```bash
[host-A]# ovs-vsctl show | grep -A5 vxlan
[host-A]# ovs-appctl tnl/neigh/show
```

**Step 3, OpenFlow pipeline:**

```bash
[host-A]# ovs-appctl ofproto/trace br-int "in_port=vm1,..."
```

**Step 4, kernel datapath:**

```bash
[host-A]# ovs-dpctl dump-flows -m | grep vxlan
[host-A]# ovs-dpctl show | grep lookups
```

**Step 5, egress wire:**

```bash
[host-A]# tcpdump -i eth0 -nn 'udp port 4789' -w egress.pcap
```

**Step 6, MTU check:**

```bash
[host-A]# ip link show eth0
[host-A]# ip link show vm1-veth
```

Inner MTU must be at most underlay MTU minus 50. If 1500 equals 1500, large packets fragment or drop.

## 9.14.5 Post-incident process

The post-incident report:

1. **Timeline:** chronological with timestamps.
2. **Root cause:** which layer.
3. **Evidence:** coverage delta, logs, pcap.
4. **Fix applied:** commit or configuration change.
5. **Prevention:** new monitoring alerts, playbook update.

The post-incident report feeds the knowledge base, which lowers MTTR for similar incidents.

## 9.14.6 Ten detailed production incident scenarios, signature plus evidence plus RCA plus remediation

Part 9.14.2 covered the 5 common incident categories at the framework level. §9.14.6 to §9.14.15 dive into 10 production-grade scenarios in detail, each in the template: **Signature** (what the operator observes) -> **Evidence** (the first 5 diagnostic commands) -> **RCA** (the underlying mechanism) -> **Remediation** (2 to 4 tiers) -> **Prevention** (alert rules and monitoring).

Cross-references with Part 20.0 §20.0.5 to §20.0.7 (8 systematic scenarios plus 3 production case studies), Part 9.26 (forensic case studies of 3 incident classes), Part 20.2 (deep-dive troubleshooting), Part 20.3 (the daily operator playbook).

### Scenario A, OVSDB Raft cluster split-brain after a network partition

**Signature:** an `ovsdb-server` 3-node cluster loses connectivity between node-1 (isolated) and node-2 plus node-3 (the majority). After the network heals, node-1 does not rejoin. `cluster/status` shows node-1 as "candidate" indefinitely. `ovn-nbctl` from node-1 times out, from node-2 or node-3 it works.

**Evidence (5 commands):**

```bash
# On each node
[node-1]# sudo ovs-appctl -t /var/run/ovn/ovnnb_db.ctl cluster/status OVN_Northbound
# Expected healthy: "Status: cluster member, Role: follower/leader"
# Actual partition: "Status: disconnected, Role: candidate"

[node-2]# sudo ovs-appctl -t /var/run/ovn/ovnnb_db.ctl cluster/status OVN_Northbound
# Expected: "Status: cluster member, Role: leader"

# Sequence index comparison
[node-1]# sudo ovs-appctl -t /var/run/ovn/ovnnb_db.ctl cluster/status OVN_Northbound | grep -E "Last Index|Commit Index"
# Expected: Last Index == Commit Index (synced)

# Log inspection
[node-1]# sudo journalctl -u ovn-northd --since "30 min ago" | grep -iE "disconnect|partition|raft"
```

**Root cause analysis (RCA):** the partition window exceeded `election-timer` (default 1000 ms times 5 equals 5 s). Node-1 in the minority cannot acquire leadership. After the heal, node-1 has a `Last Index` lower than the majority, so it needs log catch-up via the `InstallSnapshot` RPC. If the snapshot is too old or log compaction has already happened, node-1 cannot catch up incrementally and gets stuck in a "candidate" re-election loop.

**Remediation (3 tiers):**

```bash
# Tier 1 (force rejoin with a snapshot, 5 minutes)
[node-1]# sudo systemctl stop ovn-nb-db
[node-1]# sudo cp /var/lib/ovn/ovnnb_db.db /var/lib/ovn/ovnnb_db.db.stale
[node-1]# sudo rm /var/lib/ovn/ovnnb_db.db
[node-1]# sudo ovsdb-tool join-cluster /var/lib/ovn/ovnnb_db.db OVN_Northbound \
  tcp:<node-1-ip>:6643 tcp:<node-2-ip>:6643 tcp:<node-3-ip>:6643
[node-1]# sudo systemctl start ovn-nb-db

# Tier 2 (tune election-timer to prevent future split)
[any]# sudo ovs-appctl -t /var/run/ovn/ovnnb_db.ctl cluster/change-election-timer OVN_Northbound 4000
# Raise from 1000 ms to 4000 ms for 15 to 20 s partition tolerance

# Tier 3 (infrastructure improvement)
# - dedicated cluster network (not shared with the data plane)
# - network monitoring alert for inter-node latency above 500 ms
```

**Prevention:**

```yaml
- alert: OVSDBClusterMemberDisconnected
  expr: ovsdb_server_cluster_role{role="candidate"} > 0
  for: 2m
  labels: { severity: critical }
```

### Scenario B, ovs-vswitchd OOM kill

**Signature:** syslog shows "Out of memory: Killed process <PID> (ovs-vswitchd)". systemd restarts `ovs-vswitchd`, every OpenFlow flow is uninstalled, and 30 to 60 seconds of traffic is lost. After the restart, memory rises quickly back to a high baseline.

**Evidence:**

```bash
# Memory footprint before OOM
[host]# sudo ovs-appctl -t ovs-vswitchd memory/show
handlers:3 revalidators:4 ofconns:5 ports:247
rules:48293 rules-cache-used:156MB rules-cache-max:256MB
...

# Kernel OOM log
[host]# sudo journalctl -k --since "1 hour ago" | grep -iE "oom|killed process"

# Process memory over time
[host]# sudo pmap -x $(pgrep ovs-vswitchd) | tail -1

# Coverage counters for lflow plus megaflow cache
[host]# sudo ovs-appctl coverage/show | grep -iE "flow_add|rule_inserted"

# DPDK hugepage usage (if using the userspace datapath)
[host]# grep Huge /proc/meminfo
```

**Root cause analysis (RCA):** usually one of three causes:

1. **Flow explosion:** the OpenFlow rule count plus the megaflow cache exceeds available memory. Check `rules-cache-used`.
2. **Real memory leak:** an `ovs-vswitchd` bug (rare). Check the memory growth rate with `pmap` over time; if it grows linearly without correlating to the workload, it is a leak.
3. **Flow plus cache settings too aggressive:** `other_config:flow-limit` is set too high relative to RAM allocation.

**Remediation:**

```bash
# Tier 1 (immediate cap)
[host]# sudo ovs-vsctl set Open_vSwitch . other_config:flow-limit=50000
# Tier 2 (revalidator tuning)
[host]# sudo ovs-vsctl set Open_vSwitch . other_config:min-revalidate-pps=10
# Tier 3 (a hard floor via systemd resource limit)
[host]# sudo systemctl edit openvswitch-switch
# [Service]
# MemoryLimit=4G
```

### Scenario C, handler thread saturation leading to upcall drop

**Signature:** `ovs-appctl upcall/show` shows the `lost_upcalls` counter rising fast. Datapath-miss traffic is dropped (no flow installed, no forwarding). VMs experience intermittent connectivity.

**Evidence, Anatomy Template A for `upcall/show`:**

**Context when running:** the on-call engineer received a "VM ICMP retry rate above 20 percent" alert from Prometheus. Hypothesis: handler threads cannot keep up with the upcall rate.

**Command and output** *(reproduced from OVS 2.17.9, production-like)*:

```bash
[host]# sudo ovs-appctl upcall/show

system@ovs-system:
  flows         : (current 48293) (avg 47120) (max 51820) (limit 200000)
  offloaded flows : 0
  dump duration : 185ms
  ufid enabled : true

  23: (keys 12073) (total packets 1283749) (avg 55774/s)
      (dispatched to handler)
  24: (keys 12089) (total packets 1290485) (avg 56066/s)
  25: (keys 12094) (total packets 1297843) (avg 56386/s)
  26: (keys 12037) (total packets 1289201) (avg 56009/s)

  lost_upcalls: 12837 (rate 142/s)     <-- !!! RED FLAG
```

**Anatomy of each field:**

| Field | Sample value | Meaning | Abnormal sign |
|-------|--------------|---------|---------------|
| `flows current` | 48293 | Current megaflow count | Close to `limit` 200K means cache is full |
| `offloaded flows` | 0 | HW offload count | 0 means software path only |
| `dump duration` | 185 ms | Revalidator sweep time | Above 500 ms is stress, above 1000 ms is critical |
| `handler N: keys` | 12073 | Ukey count per handler thread | Even distribution means healthy load balance |
| `avg packets/s` | 55774/s | Upcall rate per handler | Sum across N handlers must stay below handler capacity |
| `lost_upcalls rate` | 142/s | **Upcalls dropped, not handled** | **Above 0 means handler bottleneck, MUST investigate** |

**Failure scenarios:**

- **`lost_upcalls > 0` with `dump duration < 100 ms`:** handlers cannot keep up (CPU bound), the revalidator is fine. Fix: raise `n-handler-threads`.
- **`dump duration > 500 ms` plus `lost_upcalls > 0`:** both revalidator and handlers are saturated. Suspect flow explosion, see Scenario D.
- **`keys` skewed across handlers** (for example handler 23 has 50K and handlers 24 to 26 have 1K each): the hash function does not distribute evenly. Rare, an OVS bug, report it upstream.

**RCA plus remediation:**

```bash
# Check the current thread count
[host]# sudo ovs-vsctl get Open_vSwitch . other_config | grep -E "n-handler|n-revalidator"

# Raise the handler thread count (requires restart)
[host]# sudo ovs-vsctl set Open_vSwitch . other_config:n-handler-threads=8
[host]# sudo systemctl restart ovs-vswitchd
# Re-verify: lost_upcalls rate goes back to 0 after restart plus 2 to 3 minutes of warmup
```

### Scenario D, megaflow cache thrashing

**Signature:** flow install rate is very high (`flow_add` coverage rises by 10K per second), cache hit rate is low (below 80 percent). `ovs-vswitchd` uses 100 percent of one CPU core (a handler thread). Traffic latency P99 spikes 3 to 5 times.

**Evidence:**

```bash
# Coverage delta over 10 seconds
[host]# sudo ovs-appctl coverage/show | grep -E "flow_add|miss|wildcard"
# Delta check:
[host]# sudo ovs-appctl coverage/show > /tmp/cov1.txt; sleep 10
[host]# sudo ovs-appctl coverage/show > /tmp/cov2.txt
[host]# diff /tmp/cov1.txt /tmp/cov2.txt

# Wildcard pattern analysis
[host]# sudo ovs-dpctl dump-flows | awk '{print $1}' | sort -u | wc -l
# Many unique wildcard patterns means low cache effectiveness
```

**Root cause analysis (RCA):** the megaflow cache key is too narrow, every 5-tuple packet creates its own entry instead of sharing a wildcard mask. Common causes: an ACL rule with TCP-flags matching (forces per-packet inspection), or `ct_state` transitions fragmenting the cache.

**Remediation:**

```bash
# Identify narrow flows
[host]# sudo ovs-dpctl dump-flows | awk '{print $1}' | sort | uniq -c | sort -rn | head

# Refactor the OpenFlow rule to drop unnecessary match fields
# (in OVN, via ACL or Port_Group, see Part 20.3 §20.3.4.6 consolidation)
```

### Scenario E, tunnel BFD flap cascade

**Signature:** the Geneve or VXLAN tunnel between two chassis flips intermittently UP -> DOWN -> UP. Traffic loss of 200 to 500 ms per flap cycle. BFD monitoring log spam.

**Evidence:**

```bash
[host-A]# sudo ovs-appctl bfd/show
BFD session for tunnel ovn-chassisB:
  local state: UP
  remote state: UP
  ...
  flaps: 47               <-- !!! flap count is high

[host-A]# sudo ovs-appctl tnl/neigh/show
# ARP resolution for the remote underlay must be stable. Synchronized flapping means an underlay issue.

# Underlay latency baseline
[host-A]# ping -i 0.2 -c 100 <host-B-underlay> | tail -5

# Geneve UDP packet trace on the underlay
[host-A]# sudo tcpdump -i eth0 -nn udp port 6081 -c 100 -w /tmp/geneve.pcap
```

**Root cause analysis (RCA):** three common causes:

1. **Underlay jitter exceeds the BFD detection threshold** (default 500 ms times 3 equals 1.5 s). An underlay latency burst above 1.5 s triggers BFD down.
2. **Underlay MTU fragmentation:** BFD packets larger than the MTU are dropped silently.
3. **Host CPU overload** delays BFD packet send and receive.

**Remediation:**

```bash
# Tune BFD interval to tolerate underlay jitter
[host]# sudo ovs-vsctl set Interface ovn-chassisB-0 options:bfd_interval=1000 \
  options:bfd_min_rx=1000 options:bfd_min_tx=1000
# 1 s times 3 equals 3 s detection (tolerant of jitter below 3 s)

# Alternative: disable BFD and rely on an ICMP-based alternative (advanced)
[host]# sudo ovs-vsctl set Interface ovn-chassisB-0 bfd=no
```

### Scenario F, hardware offload silent fallback

**Signature:** hardware offload (TC flower) is configured, baseline 100 Gbps line rate. After deploying a new ACL rule, real bandwidth drops to 10 Gbps (kernel slow path). No error log. `ovs-dpctl dump-flows` shows flows whose `offload_state` field is empty or `no_support`.

**Evidence:**

```bash
# Offload statistics
[host]# sudo ovs-dpctl dump-flows -m | grep -E "offloaded|state=offloaded"

# Every flow in tc
[host]# sudo tc -s filter show dev eth0 ingress | head

# Coverage counters specific to offload
[host]# sudo ovs-appctl coverage/show | grep -iE "offload|tc_"

# Driver dmesg
[host]# sudo dmesg | grep -iE "mlx5|offload|tc" | tail -20
```

**Root cause analysis (RCA):** the TC flower driver (`mlx5_core`, `i40e`, and others) does not support a new action combination. Common patterns:

1. The `conntrack` action is not yet supported on the NIC (mlx5 needs ConnectX-6 or later).
2. `mod_nw_ttl` plus `mod_eth_dst` combination hits a hardware limit.
3. Metadata wider than 16 bits does not fit a hardware register.

OVS silently falls back to the software slow path, so the kernel datapath handles the traffic without HW acceleration.

**Remediation:**

```bash
# Tier 1: disable offload for a specific rule
[host]# sudo ovs-vsctl set Open_vSwitch . other_config:hw-offload=true \
  other_config:offload-cap-list="drop,output,set,forward"
# Exclude ct/conntrack from offload, kernel handles it

# Tier 2: upgrade NIC firmware
[host]# sudo ethtool -i eth0 | grep -i firmware
[host]# sudo mstconfig -d /dev/mst/mt4125_pciconf0 query | grep -i version

# Tier 3: reduce ACL complexity (use Port_Group instead of individual matches)
```

### Scenario G, NIC ring buffer overflow

**Signature:** ethtool shows `rx_no_buffer_count` rising fast. Packet drops happen at the NIC level, before OVS sees them. VM TCP retransmit rate spikes.

**Evidence:**

```bash
# NIC statistics
[host]# sudo ethtool -S eth0 | grep -iE "rx_no_buffer|rx_missed|rx_discard"

# Ring buffer size
[host]# sudo ethtool -g eth0
Ring parameters for eth0:
Pre-set maximums:
RX:             4096
TX:             4096
Current hardware settings:
RX:             512           <-- !!! Default, too small for 100 Gbps
TX:             512

# IRQ distribution
[host]# cat /proc/interrupts | grep -E "eth0|mlx5" | head
```

**Root cause analysis (RCA):** the default RX ring of 256 to 512 descriptors is not enough for traffic bursts above 1 Mpps. When the CPU is too busy to drain the ring in time, the NIC drops packets at the hardware level.

**Remediation:**

```bash
# Raise the ring buffer to maximum
[host]# sudo ethtool -G eth0 rx 4096 tx 4096

# IRQ affinity, distribute across 8 cores
[host]# sudo systemctl restart irqbalance
# Or manually: sudo set_irq_affinity.sh eth0

# Persist across reboots
[host]# sudo cat > /etc/network/if-up.d/ring-buffer <<'EOF'
#!/bin/bash
[ "$IFACE" = "eth0" ] && ethtool -G eth0 rx 4096 tx 4096
EOF
[host]# sudo chmod +x /etc/network/if-up.d/ring-buffer
```

### Scenario H, conntrack table full

**Signature:** `dmesg` shows "nf_conntrack: table full, dropping packet". Random TCP SYN drops. OVS stateful ACL fails (`allow-related` reply does not pass).

**Evidence:**

```bash
# Current versus max
[host]# cat /proc/sys/net/netfilter/nf_conntrack_count
512047

[host]# cat /proc/sys/net/netfilter/nf_conntrack_max
524288                        <-- close to the limit

# Top entries breakdown
[host]# sudo conntrack -L | awk '{print $1,$2}' | sort | uniq -c | sort -rn | head
# Detect patterns (TCP ESTABLISHED versus UNREPLIED)
```

**Root cause analysis (RCA):** the default `nf_conntrack_max` equals `hashsize * 4`, computed from RAM. A hypervisor with 1000 VMs and 500 connections per VM equals 500K entries, easily reached. Zone collisions (Part 9.26 §9.26.12) amplify the problem.

**Remediation:**

```bash
# Tier 1: raise max plus rebuild the hashtable
[host]# sudo sysctl -w net.netfilter.nf_conntrack_max=2097152
[host]# sudo sysctl -w net.netfilter.nf_conntrack_buckets=524288
[host]# echo 524288 | sudo tee /sys/module/nf_conntrack/parameters/hashsize

# Tier 2: lower the TCP established timeout
[host]# sudo sysctl -w net.netfilter.nf_conntrack_tcp_timeout_established=3600
# 5 days down to 1 hour, faster GC of stale entries

# Tier 3: per-zone limit (OVS 2.17 and later)
[host]# sudo ovs-appctl dpctl/ct-set-limits default=100000 zone=1012:50000
```

### Scenario I, OpenFlow controller disconnect with fail-mode lockout

**Signature:** bridge `br0` is configured with `fail-mode=secure`. The controller (Faucet, ONOS) crashes. Two minutes later, traffic stops completely, every packet is dropped at `br0`.

**Evidence:**

```bash
[host]# sudo ovs-vsctl get Bridge br0 fail_mode
secure

# Controller connection
[host]# sudo ovs-vsctl get Bridge br0 controller
[tcp:10.0.0.100:6653]

# Disconnected state
[host]# sudo ovs-appctl -t ovs-vswitchd ofproto/list-tunnels
[host]# sudo ovs-vsctl list Controller
...
is_connected: false               <-- controller crashed
...
```

**Root cause analysis (RCA):** `fail-mode=secure` means that when the controller disconnects, the bridge keeps using already-installed flows but **does not learn new flows**. New traffic patterns miss and are dropped. `fail-mode=standalone` (default) falls back to a hub or switch, less secure but traffic continues.

**Remediation:**

```bash
# Immediate fail-over
[host]# sudo ovs-vsctl set-fail-mode br0 standalone
# Traffic resumes within seconds (the bridge learns MAC addresses via the NORMAL action)

# Restore the primary controller
# ... restart controller service ...

# Long term: a controller HA cluster of 3 or more nodes
```

**Trade-off:** `secure` is safe when there is a controller HA cluster, `standalone` suits an edge or branch site with a single controller and tolerance for hub-mode operation.

### Scenario J, bond slave link flap microburst

**Signature:** `bond0` slave `eth1` flaps 50 times per hour with intervals of 2 to 3 seconds. There is no steady DOWN state, only micro-flap. But traffic latency P99 spikes 100 ms per flap cycle (megaflow invalidation, see Part 9.26 §9.26.11).

**Evidence:**

```bash
# Bond flap history
[host]# sudo ovs-appctl bond/show bond0 | grep -iE "slave|flap"

# Switch-side correlation
[host]# sudo ethtool -S eth1 | grep -iE "link_up|link_down|carrier_changes"

# dmesg link events
[host]# sudo dmesg -T | grep -iE "eth1|link" | tail -30

# Switch vendor correlation (for example Cisco)
# switch# show interfaces eth1/1 counters errors | include CRC|Frame
```

**Root cause analysis (RCA):** the Ethernet frame CRC error rate exceeds 1 percent on the cable or SFP, the link layer declares DOWN, auto-negotiate brings it UP, and a 2 to 3 second flap cycle results. Usually a degraded cable, SFP temperature, or a switch port issue.

**Remediation:**

```bash
# Tier 1: hysteresis to hide micro-flaps
[host]# sudo ovs-vsctl set Port bond0 other_config:bond-detect-mode=carrier \
  other_config:bond-miimon-interval=1000 \
  other_config:bond-slave-updelay=5000 \
  other_config:bond-slave-downdelay=2000
# downdelay 2 s tolerates flaps shorter than 2 s without declaring DOWN
# updelay 5 s requires a recovered slave to be stable for 5 s before rejoining

# Tier 2: physical replacement
# - replace CAT6a cable
# - replace the SFP transceiver
# - swap the switch port

# Tier 3: monitoring alert
```

```yaml
- alert: NICFlapStorm
  expr: rate(node_network_carrier_changes_total[5m]) > 0.01
  for: 2m
  labels: { severity: warning }
```

### Scenario K, BFD thundering herd after an underlay ToR reboot

**Signature:** the ToR switch reboots as planned, about 90 seconds. When the uplink comes back, every chassis (N equals 60 in a dedicated rack) detects the Geneve tunnel UP at almost the same tick. Each chassis re-initiates a BFD session to its N minus 1 peers all at once, producing a burst of BFD control packets. Result: `ovn-controller` on many chassis reported "BFD state change" thousands of times in 5 to 10 seconds, `ovs-vswitchd` spiked CPU 80 to 95 percent on a single core, handler threads were near saturation (related to Scenario C). VM latency P99 increased 40 to 120 ms during the thundering herd window, but no traffic was completely lost because old datapath flows remained valid. Distinguish from Scenario E: Scenario E is intermittent flapping that lasts due to underlay jitter, Scenario K is a one-time burst during the convergence window that then settles by itself.

**Evidence (5 commands):**

```bash
# Count chassis whose BFD comes up within a 10 s window
[any-chassis]# sudo ovn-sbctl --timestamp list Chassis_Private \
  | grep -E "^_uuid|^name|last_seq" | head -40
# Last_seq updates clustered within 5 to 10 s indicates a thundering herd

# BFD session turnover rate per chassis
[chassis-X]# sudo ovs-appctl bfd/show \
  | awk '/flaps:/ {sum+=$2; cnt++} END {print "total flaps:", sum, "sessions:", cnt}'
# More than 50 flaps across 60 sessions in 5 minutes equals a burst

# ovn-controller recompute burst
[chassis-X]# sudo ovn-appctl -t ovn-controller inc-engine/recompute-stats \
  | grep -E "recompute|runtime"
# Recompute count bursts during the convergence window

# Coverage handler thread wakeups
[chassis-X]# sudo ovs-appctl coverage/show | grep -iE "handler|upcall"
# handler_wakeups rate spikes briefly

# Tunnel tap counter (BFD packets going through interface eth-uplink)
[chassis-X]# sudo ethtool -S eth-uplink | grep -iE "rx_packets|tx_packets"
# Delta during the burst window equals the actual BFD control-plane load
```

**Root cause analysis (RCA):** OVN before 24.03 does not have staggered BFD startup. When tunnel ports come up together, `bfd_configure()` in `ofproto-dpif.c` initializes sessions with a random initial interval drawn from a narrow range, not enough to phase-shift across N chassis. Every chassis sends BFD hello within a burst window shorter than 500 ms. The receive side (peer chassis) must parse and run the state machine for nearly N peers concurrently. For a 60-chassis cluster or larger, control-plane load scales as O(N squared). OVN 24.03 and later add larger jitter and hello-burst rate limiting at init, which significantly reduces the thundering herd. A less common secondary cause: BFD `min-rx`/`min-tx` configured too low (100 ms) in a large cluster multiplies the burst rate.

**Remediation (3 tiers):**

```bash
# Tier 1 (immediate, reduce pressure on the spiking chassis)
# Temporarily raise the BFD interval to lower the hello rate during the window
[chassis-X]# for p in $(sudo ovs-vsctl --columns=name find Interface type=geneve | awk -F: '{print $2}'); do
  sudo ovs-vsctl set Interface $p options:bfd_min_rx=1000 options:bfd_min_tx=1000
done
# After convergence stabilizes 2 to 3 minutes later, lower it back to 500 ms if you want fast detection

# Tier 2 (operational, stagger chassis startup)
# When ToR reboot is planned, rolling-restart ovn-controller in batches of 10 chassis,
# spaced 30 s apart. Splits the thundering herd into 6 small bursts.
[deployer]$ for batch in $(cat chassis-batches.txt); do
  ansible -i inventory -l $batch -m systemd -a "name=ovn-controller state=restarted"
  sleep 30
done

# Tier 3 (structural, upgrade plus rate-limit)
# Upgrade to OVN 24.03 LTS+ with built-in staggered BFD startup
# Plus set a rate-limit on BFD control packets at the uplink QoS (Part 9.9)
[chassis-X]# sudo ovs-vsctl set Interface eth-uplink ingress_policing_rate=100000 \
  ingress_policing_burst=10000
# Protects the kernel from a BFD flood even when a peer is compromised
```

### Scenario L, ovn-northd compile stuck on a bulk NBDB update

**Signature:** an automation tool creates 500 `Logical_Switch_Port` rows in a single transaction (for example, onboarding a new tenant). After the NBDB commit, NBDB `nb_cfg` jumps by N, but SBDB `nb_cfg` (read via `ovn-sbctl get SB_Global . nb_cfg`) does not move for 30 to 180 seconds. During that window, `ovn-northd` is at 100 percent CPU on a single thread. Newly created VMs have no network, existing VMs are unaffected because their datapath flows are already installed. Recovery happens after `northd` finishes compiling the large batch.

**Evidence, Anatomy Template A for `ovn-appctl -t ovn-northd stopwatch/show`:**

**Context when running:** the on-call engineer received a "nb_cfg lag above 30 s" alert from Prometheus. Hypothesis: `northd` is compiling a bulk update. Need to identify which pipeline stage is blocking.

**Command and output** *(reproduced from OVN 22.03.8, production-like on a 60-chassis cluster)*:

```bash
[nbdb-leader]# sudo ovn-appctl -t ovn-northd stopwatch/show

Statistics for 'ovnnb_db_run'
  Total samples: 1247
  Maximum: 48320 ms          <-- !!! 48 s for one iteration
  Minimum: 12 ms
  95th percentile: 32150 ms  <-- most bursts above 32 s
  Short term average: 28400 ms
  Long term average: 180 ms

Statistics for 'build_lflows'
  Total samples: 1247
  Maximum: 42180 ms          <-- 87 percent of ovnnb_db_run
  95th percentile: 28000 ms

Statistics for 'ovnsb_db_run'
  Maximum: 5100 ms           <-- SB commit relatively fast

Statistics for 'clear_tracked_data'
  Maximum: 92 ms
```

**Anatomy of each field:**

| Field | Sample value | Meaning | Abnormal sign |
|-------|--------------|---------|---------------|
| `ovnnb_db_run max` | 48320 ms | Time for one full iteration | Above 10 s means stuck, above 30 s is critical |
| `build_lflows max` | 42180 ms | Generate Logical_Flow from NB | 87 percent of ovnnb_db_run, the dominant pipeline stage |
| `ovnsb_db_run max` | 5100 ms | Commit the SB transaction | Usually below 1 s, above 5 s means SBDB contention |
| `clear_tracked_data max` | 92 ms | Reset I-P state nodes | Above 500 ms means I-P engine overflow |
| Ratio 95p over long-term | 180x | Burst versus steady state | Above 100x suspects a bulk update |

**Failure scenarios:**

- **`build_lflows` is more than 80 percent** and `ovnsb_db_run` is short, the compile stage is the bottleneck. The cause is usually ACL times Port_Group cross-product (Part 9.24 §9.24.5 conjunction analysis) multiplied by the number of new LSPs.
- **`ovnsb_db_run` is more than 50 percent**, SBDB commit is the bottleneck. Check Raft leader CPU, disk I/O, or a cluster election in flight.
- **`clear_tracked_data` above 1 s**, broad I-P engine invalidation. Symptoms similar to Scenario N, possibly the same root cause.

**Root cause analysis (RCA):** two common mechanisms:

1. **ACL cross-product explosion:** 500 new LSPs times 50 ACL rules per tenant equals 25K Logical_Flows generated. If the ACL uses `inport == @port_group` plus `ip4.src == {10.0.0.0/16}`, a conjunction-match tree rebuild is required. OVN before 23.03 rebuilds the entire tree instead of merging incrementally.
2. **Incremental Processing (I-P) fallback:** the transaction is too large and exceeds the I-P node invalidation threshold. The engine falls back to full recompute instead of a partial update. Check via `ovn-appctl -t ovn-northd inc-engine/show` to see whether any node shows `status: recompute` instead of `status: updated`.

**Remediation (3 tiers):**

```bash
# Tier 1 (tooling fix, break the bulk transaction into batches)
# Instead of 1 txn with 500 LSPs, split into 10 txns of 50 LSPs each, 2 s apart
[tooling]$ for batch in $(split -l 50 lsp.txt); do
  ovn-nbctl $(cat $batch | while read name; do echo "-- lsp-add ls1 $name"; done)
  sleep 2
done
# Each 50-LSP batch compiles in about 3 s, no queue buildup

# Tier 2 (northd parallelization, OVN 23.03 and later)
[any]# sudo ovn-nbctl set NB_Global . options:northd_probe_interval=60000
[any]# sudo ovn-appctl -t ovn-northd parallel-build/enable
# Use multi-core compile, 2 to 4x speedup depending on hardware

# Tier 3 (structural, Port_Group consolidation)
# Refactor 50 ACL rules per tenant times 500 LSPs into a Port_Group pattern
# Reduce cross-product from 25K flows to 500 flows (Part 9.23 plus Part 20.3 §20.3.4.6)
```

**Prevention:**

```yaml

  expr: ovn_nb_cfg - ovn_sb_cfg > 5
  for: 60s
  labels: { severity: warning }

- alert: OvnNorthdBuildLflowStuck

  for: 2m
```

### Scenario M, chassis ghost claim after a crash

**Signature:** chassis A (a hypervisor) is hard-powered off due to a hardware failure. VMs on A are lost. When the operator migrates the VMs to chassis B, the new LSPs are created successfully but have no network. `ovn-sbctl show` shows several Port_Bindings still bound to chassis A (already offline). The `ovn-controller` log on B reports "Claim request rejected, port already claimed by chassis-A". The BFD session to chassis A is in DOWN state from the peer side, but the `Chassis` row in SBDB has not been deleted.

**Evidence (5 commands):**

```bash
# Check stale Port_Bindings
[nbdb-leader]# sudo ovn-sbctl --columns=logical_port,chassis,up list Port_Binding \
  | grep "chassis=.*chassis-a"
# Many rows bound to chassis-A even though the chassis is offline

# Chassis_Private last_seq, freshness indicator
[nbdb-leader]# sudo ovn-sbctl --columns=name,nb_cfg,nb_cfg_timestamp list Chassis_Private \
  | grep chassis-a
# nb_cfg_timestamp frozen since the crash, a sign of a ghost row

# BFD status from a peer chassis
[chassis-B]# sudo ovs-appctl bfd/show | grep -A4 "ovn-chassisa"
# Expected: remote state: DOWN plus "Diagnostic: Neighbor signaled session down"

# ovn-controller log on a surviving chassis
[chassis-B]# sudo journalctl -u ovn-controller --since "30 min ago" \
  | grep -iE "claim|reject|port_binding"
# Should see: "Failed to claim port X, already claimed by chassis-A"

# Confirm chassis A is definitely offline
[operator]$ ping -c 3 <chassis-a-ip>
# 100 percent packet loss plus BGP/spine monitoring confirmation
```

**Root cause analysis (RCA):** OVN uses `Chassis.name` plus `Port_Binding.chassis` for claim ownership. When a chassis crashes without graceful shutdown, the `ovn-controller` on A does not run `release_all_ports()` to return the Port_Bindings to unbound. Surviving chassis detect BFD DOWN but do not automatically reclaim the ports because the default policy is conservative, avoiding a race condition when chassis A is only temporarily isolated and later returns (a split-brain situation). The system waits for operator intervention by deleting the Chassis row, or it triggers `stale-binding-timeout` (OVN 22.06 and later). If `Chassis_Private.last_seq` does not update beyond the threshold, by default 300 s (configurable via `options:stale-binding-timeout-ms` on `SB_Global`), `northd` clears the bindings automatically.

**Remediation (3 tiers):**

```bash
# Tier 1 (immediate, manual chassis teardown)
[any-sb-client]# sudo ovn-sbctl chassis-del chassis-a
# Deletes the Chassis row plus the Chassis_Private row plus resets Port_Binding.chassis to NULL
# After about 2 s, northd recompiles, Port_Bindings become available for chassis B to claim

[chassis-B]# sudo systemctl restart ovn-controller
# Kick chassis B to re-trigger the claim protocol for newly created ports

# Tier 2 (wait for automation, requires OVN 22.06 and later)
[any]# sudo ovn-nbctl set NB_Global . options:stale-binding-timeout-ms=180000
# Timeout 3 minutes, northd auto-GCs the ghost chassis

# Tier 3 (HA-aware, automatic HA_Chassis_Group fail-over)
# For external Logical_Router_Port, configure HA_Chassis_Group (Part 13.6)
# Primary chassis crashes, secondary takes over the gateway MAC plus ARP responder
# No dependency on manual chassis-del
```

**Prevention:**

```yaml
- alert: OvnChassisStale
  expr: time() - ovn_chassis_private_last_seen_timestamp > 180
  for: 60s
  labels: { severity: critical }
  annotations:
    summary: "Chassis {{ $labels.chassis }} stale above 3 minutes, suspect ghost binding"
```

### Scenario N, ovn-controller infinite recompute loop

**Signature:** on one chassis, `ovn-controller` CPU stays at 100 percent without dropping back to idle. `lflow-cache/show` reports a hit rate of 0 percent or very low. The journal log spams "trigger recompute" many times per second. Other chassis in the cluster show normal CPU. VMs on the affected chassis: traffic still forwards thanks to old datapath flows, but every NBDB change applies slowly or is lost entirely.

**Evidence (5 commands):**

```bash
# Inc-engine recompute stats
[chassis-X]# sudo ovn-appctl -t ovn-controller inc-engine/recompute-stats
# Expected healthy: recompute count rises slowly, mostly nodes in "updated" state
# Pathological: recompute count above 10 per second, reason concentrated on one event class

# lflow-cache effectiveness
[chassis-X]# sudo ovn-appctl -t ovn-controller lflow-cache/show
Enabled: true
high-watermark:   180MB
Cache entries:    0         <-- !!! cache empty
Cache trim count: 12487     <-- continuous trimming
Mean hit rate:    0.8%      <-- !!! near zero

# ovn-controller engine node status
[chassis-X]# sudo ovn-appctl -t ovn-controller inc-engine/show
# node_name           status
# OVS_interface       updated
# OVS_Port            updated
# Logical_Flow        recompute  <-- !!! stuck in recompute
# lb_data             updated

# Journal pattern detection
[chassis-X]# sudo journalctl -u ovn-controller --since "10 min ago" \
  | grep -c "trigger recompute"
# Above 600 in 10 minutes, that is above 1 per second, pathological

# Correlate with NBDB events
[nbdb-leader]# sudo ovn-sbctl get-connection
[nbdb-leader]# sudo ovsdb-client monitor OVN_Southbound 2>&1 \
  | grep -c '"op":"update"' &
# Identify which specific update pattern fires constantly
```

**Root cause analysis (RCA):** the `ovn-controller` I-P engine has many nodes (OVS_interface, Port_Binding, Logical_Flow, MAC_Binding, and others). Each node knows how to perform a partial update when its input changes. However, certain edge cases push a node into `status: recompute`, meaning the engine must rebuild the entire state of that node from scratch. If the input keeps changing before the recompute finishes, the engine enters a non-terminating loop. Well-known patterns that cause this loop:

1. **External_ids churn:** an automation tool continuously writes `external_ids` on `Logical_Switch_Port` or `Chassis`. Each write invalidates the I-P node cache.
2. **BFD session flap interplay:** Scenario E plus Scenario K can cascade into a recompute loop when BFD state changes many times per second.
3. **OVN version-specific bug:** some commits between major versions have a bug where one node never transitions from `recompute` to `updated`. For example, an LSP with a very large `port_security` list (above 200 entries) on OVN 22.06.x has a similar bug, fixed in 22.09 and later.

**Remediation (3 tiers):**

```bash
# Tier 1 (urgent, break the loop temporarily)
[chassis-X]# sudo ovn-appctl -t ovn-controller recompute
# Force a one-time full recompute, sometimes breaks the loop if the cause is transient

[chassis-X]# sudo ovn-appctl -t ovn-controller lflow-cache/flush
# Flush the cache, return to a clean state. If the loop continues, root cause is persistent

# Tier 2 (identify the trigger)
# Use ovsdb-client monitor plus timestamp correlation to identify the NBDB row
# that causes continuous invalidation. Usually external_ids from automation tools.
[nbdb-leader]# sudo ovsdb-client monitor OVN_Northbound Logical_Switch_Port external_ids \
  --timestamp 2>&1 | head -100
# If you see one LSP updating external_ids every second, found culprit

# Tier 3 (structural, version upgrade plus deduplicate automation)
# Upgrade to an OVN LTS with the known I-P loop bug fix
# Audit automation tools: external_ids updates must be idempotent, check before write
[tooling]$ # pseudo-code: only update if value actually changed
if [ "$(ovn-nbctl get LSP $p external_ids:xyz)" != "$new_val" ]; then
  ovn-nbctl set LSP $p external_ids:xyz=$new_val
fi
```

**Prevention:**

```yaml

  expr: rate(ovn_controller_engine_recompute_total[1m]) > 5
  for: 3m
  labels: { severity: critical }

- alert: OvnLflowCacheHitRateLow
  expr: ovn_controller_lflow_cache_hit_rate < 0.5
  for: 5m
  labels: { severity: warning }
```

## 9.14.9 Capstone POE, "Does raising nf_conntrack_max from 512K to 2M fully fix Scenario H?"

**Scenario:** a hypervisor with 1000 VMs reports "TCP SYN drop 5 percent". sysctl shows `nf_conntrack_count=510K` (close to max 524K). The manager proposes raising `nf_conntrack_max=2097152` to fix the issue.

**Predict:** does raising max help? Side effects?

**Observe (reasoning):**

1. **Raising max means raising memory consumption:** each conntrack entry is about 300 bytes plus the hashtable entry. 2M entries times 300 bytes equals **600 MB of kernel-space RAM**. Check `/proc/meminfo` baseline.
2. **Raising hashsize requires a restart or echoing** directly into `/sys/module/nf_conntrack/parameters/hashsize`. Out of sync with `nf_conntrack_max` causes a conflict.
3. **GC pressure:** the kernel GC thread sweeps the entire table periodically. 2M entries times 1 Hz means significant CPU overhead.
4. **Root cause persists:** if the leak rate (NEW connections that never close) is 500 per second, then 2M max only buys **about 4000 seconds, roughly 1 hour** before it fills again.

**Explain:** the proposal is **not enough**, only a band-aid. The right 3-tier approach:

- **Tier 1 (immediate):** raise max to 2M to STOP the current outage (a 5-minute win).
- **Tier 2 (short term):** lower the TCP established timeout from 5 days to 1 hour. Entries are GC'd 5 times faster, effective capacity equals 2M times 5 equivalent.
- **Tier 3 (long term):** identify the leak source. Usually:
  - a VM application that does not close TCP connections gracefully.
  - SNAT rules creating unnecessary transient conntrack entries, refactor through OVN's `allow-related` stateful pattern.
  - zone collisions (Part 9.26 §9.26.12) multiplying state entries, an audit script can detect them.

**Finish:** the principle is that **raising a resource threshold does NOT fix a leak**. Memory, flow-limit, conntrack-max are all defense-in-depth at an upper layer, not the root-cause fix. Cross-reference Part 9.26 §9.26.7 pattern: "Tier 1 immediate cap plus Tier 3 root cause", do not skip.

## 9.14.10 Guided Exercise, reproduce Scenario L northd compile stuck

> **State verification lab:** doc-plausible; needs at least a 1-chassis OVN sandbox.

**Setup:** an OVN sandbox with 1 chassis plus 1 NBDB plus 1 SBDB (standalone, no Raft cluster needed). Run OVN 22.03.8 or equivalent. Empty NBDB (1 Logical_Switch `ls1` plus 0 LSP).

**Steps:**

**1.** Baseline stopwatch for an idle iteration:

```bash
[host]$ sudo ovn-appctl -t ovn-northd stopwatch/show | grep -A3 "ovnnb_db_run"
# Expected: Maximum below 50 ms, Short term avg below 30 ms
```



```bash

```

**5.** Generate 500 LSPs again with **batches of 50 LSPs per transaction, sleep 2 s** (Tier 1 fix):

```bash
[host]$ time (for start in $(seq 1 50 500); do
  CMDS=""

    CMDS="$CMDS -- lsp-add ls1 lsp-$i"
  done

done)
# Real time about 25 to 35 s (including 9 times 2 s sleep)
```

**6.** Measure stopwatch after batched run:

```bash
[host]$ sudo ovn-appctl -t ovn-northd stopwatch/show > /tmp/batched-stopwatch.txt
cat /tmp/batched-stopwatch.txt | grep -A3 -E "ovnnb_db_run|build_lflows"
# Expected: Maximum ovnnb_db_run below 1 s (10 to 50 times lower than bulk)
# build_lflows is no longer dominant above 80 percent
```

**7.** Compare the two reports plus verify nb_cfg:

```bash
[host]$ diff <(grep "Maximum" /tmp/bulk-stopwatch.txt) \
  <(grep "Maximum" /tmp/batched-stopwatch.txt)

# Final nb_cfg lag measurement
[host]$ NB=$(ovn-nbctl get NB_Global . nb_cfg)
[host]$ SB=$(ovn-sbctl get SB_Global . nb_cfg)

```

- **Predict:** bulk 500 LSP in 1 txn will make `build_lflows` max account for above 70 percent of `ovnnb_db_run` max; batched 50 times 10 will reduce max `build_lflows` to below 300 ms per iteration.
- **Observe:** record the actual numbers (a small sandbox may not have enough load to scale up to 48 s like production).
- **Explain:** batched transactions allow the I-P engine to update incrementally between batches. A bulk single-txn forces the engine to invalidate every `Logical_Flow` node at once, requiring full recompute. ACL times Port_Group cross-product rebuilds from scratch if there are ACL rules. This is the same pattern as Scenario N (recompute loop), but short-lived rather than infinite.

**Cleanup:**

```bash

[host]$ sudo ovn-appctl -t ovn-northd stopwatch/reset
```

## 9.14.11 Capstone POE, "Does rolling-restart of ovn-controller fix every OVN incident?"

**Scenario:** a junior engineer encountered three different incidents during one on-call week:

- Monday: Scenario N (recompute loop on one chassis).
- Wednesday: Scenario M (chassis ghost claim after a hardware crash).
- Friday: Scenario O (cluster-wide cert expiry).

The engineer proposed a unified playbook: **"rolling-restart `ovn-controller` across the entire cluster, in batches of 10 chassis"** for all three cases. Senior management asked for an evaluation.

**Predict:** does rolling-restart resolve each scenario? What side effects?

**Observe (analysis per scenario):**

1. **Scenario N (recompute loop):** restart **does** break the loop temporarily because it clears the lflow-cache plus I-P state when the process starts. However, if the root cause is `external_ids` churn from automation (§9.14.6 N.RCA.1), the loop returns within minutes. Side effect: every `ovn-controller` restart triggers an initial full recompute, control-plane load spikes across a large cluster, and can cascade into a thundering herd similar to Scenario K.
2. **Scenario M (chassis ghost):** restart **does not** resolve it. A ghost claim is a SBDB `Chassis` row that still exists for a crashed chassis. Restarting `ovn-controller` on surviving chassis does not delete the SBDB row of the dead chassis. You need `ovn-sbctl chassis-del chassis-a` or to wait for `stale-binding-timeout-ms` (§9.14.6 M.Tier 1-2). The restart playbook is wrong for this nature of problem.
3. **Scenario O (cert expiry):** restart **does not** resolve it and may **make it worse**. If certs have expired, restarting `ovn-controller` only causes it to fail SSL handshake faster. If the cluster is in the "old flows still forwarding" window, after the restart flows are reset (the datapath clears flows when `ovs-vswitchd` receives an OpenFlow disconnect from `ovn-controller`). Before restarting, you must **renew the cert plus deploy the new cert** as in Tier 1 §9.14.6 O. Rolling-restart without renewing equals a data-plane outage.

**Explain:**

The "rolling-restart `ovn-controller`" playbook is right only for Scenario N, and only at the **emergency Tier 1 level**. Applying it incorrectly to M or O is an anti-pattern. The principles:

- **Restart clears in-memory state.** If the root cause is in-memory (cache loop, memory pressure), restart helps temporarily.
- **Restart does NOT clear persistent state.** Chassis rows, Port_Bindings, cert chains are persistent in NBDB, SBDB, or the filesystem. Fixing them requires acting on the persistent layer.
- **Restart in a large cluster equals a storm.** Each chassis restart triggers full recompute, handler-thread cascades, and BFD re-init. Rolling batches of 10 with 30 s spacing (Tier 2 §9.14.6 K) is a workaround but not a substitute for fixing the actual root cause.

The right 3-step approach for the on-call engineer:

1. **Classify the scenario** via the §9.14.7 master matrix (20 symptoms to scenarios).
2. **Apply the matching Tier 1** per §9.14.6. Restart is just one of many possible Tier 1 actions.
3. **Schedule a Tier 3 structural fix** (config change, upgrade, process hardening) after Tier 1 stabilizes the situation.

**Finish:** a junior on-call engineer must learn the principle "**symptom -> class -> scenario-specific tier**", not "**one playbook fits all**". A warning sign: any playbook that begins with "restart everything" deserves scepticism. Restart is a hammer. Not every OVN incident is a nail.

## 9.14.X Missing tool, ovs-bugtool, the diagnostic-bundle automation (Phase H S50)

**Audit finding:** before S50, the curriculum had zero mention of `ovs-bugtool`. This section closes the gap.

`ovs-bugtool` is a utility that automatically collects a complete diagnostic bundle from an OVS environment. The output is a tar.gz containing 30 to 50 files: configuration dumps, flow dumps, logs, system info. Use it when vendor support needs full information or when escalating an issue to the upstream mailing list.

```shell
/var/log/openvswitch/ovs-bugtool-2026-04-24-11-23-45.tar.gz collected

# Decompress to view contents
$ tar tzf /var/log/openvswitch/ovs-bugtool-*.tar.gz | head -20
ovs-bugtool-2026-04-24-11-23-45/commands/
ovs-bugtool-2026-04-24-11-23-45/commands/ovs-vsctl-show

```

### 9.14.X.2 Bundle anatomy

| Directory | Content |
|---|---|
| `commands/` | Output of 30-plus diagnostic commands (`ovs-vsctl show`, `dump-flows`, coverage, and others) |
| `log/` | OVS daemon log files (`ovs-vswitchd.log`, `ovsdb-server.log`) |
| `system/` | `/proc` snapshots (`meminfo`, `cpuinfo`, `interrupts`, `mounts`) |
| `network/` | `ip a`, `ip r`, iptables, brctl |

### 9.14.X.3 Selective collection

```shell
# Only collect for a specific bridge
$ sudo ovs-bugtool --include-bridge=br-int

# Exclude potentially sensitive flow dumps
$ sudo ovs-bugtool --exclude-flow-dump

# Specific output path
$ sudo ovs-bugtool --output=/tmp/ovs-debug.tar.gz
```

### 9.14.X.4 Usage in the incident workflow

```shell
# Step 1: when an incident happens, collect the bundle IMMEDIATELY (state evidence, do not wait)
$ sudo ovs-bugtool

# Step 2: send the bundle plus the issue description
$ curl -F 'file=@ovs-bugtool-*.tar.gz' https://support.company.com/incident/XXX

# Step 3: parse the bundle locally for triage
$ tar xzf ovs-bugtool-*.tar.gz
$ less ovs-bugtool-*/commands/ovs-appctl-coverage-show
$ less ovs-bugtool-*/log/ovs-vswitchd.log
```

### 9.14.X.5 `ovs-pcap`, parse OVS-saved packet traces

Companion tool: `ovs-pcap` decodes pcap files used inside OVS (packet-in log, debug dumps).

```shell
# Convert OVS internal packet log to standard pcap
$ sudo ovs-pcap /var/log/openvswitch/packet-in.log > /tmp/packetin.pcap

# Read with tcpdump or wireshark as usual
$ tcpdump -r /tmp/packetin.pcap -nne
```

Rarely used directly, usually a backfill for forensic reconstruction of traffic patterns from old logs.

### 9.14.X.6 `ovs-testcontroller`, a dummy OpenFlow controller for testing

When testing an OpenFlow bridge without a real controller (ONOS, ODL):

```shell
# Listen on port 6633
$ sudo ovs-testcontroller ptcp:6633

# Configure the bridge to point at the test controller
$ sudo ovs-vsctl set-controller br-test tcp:127.0.0.1:6633

# ovs-testcontroller installs basic learn-switch flows, the bridge works as a standard L2 switch
```

Not for production, only for labs and tutorials.

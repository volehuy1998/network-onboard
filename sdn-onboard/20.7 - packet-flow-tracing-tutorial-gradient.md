## Mục tiêu bài học

Phần này là **tutorial gradient** sư phạm. Sau khi hoàn thành, học viên có thể:

1. **Áp dụng (Apply) Level 1**, dùng `ovn-trace` cho 1 packet đơn giản trong 1 Logical Switch để xem logical pipeline output (hello-world).
2. **Phân tích (Analyze) Level 2**, đọc trace output qua multi-table pipeline với ACL stateful, xác định mỗi stage đã match flow nào, action gì.
3. **Phân tích (Analyze) Level 3**, trace cross-subnet qua Logical Router (LR), bao gồm ARP resolve + dec_ttl + nexthop selection.
4. **Áp dụng (Apply) Level 4**, kết hợp `ovn-trace` (logical view) với `ofproto/trace` (OpenFlow view) để debug multichassis Geneve tunnel.
5. **Tổng hợp (Synthesize) Level 5**, dùng `ovn-detrace` chain `ofproto/trace | ovn-detrace` để correlate physical OpenFlow output ngược về OVN logical context cho production incident.
6. **Đánh giá (Evaluate)**, tự design trace scenario cho incident chưa biết, chọn level phù hợp, giải thích rationale.

> **Triết lý sư phạm:** Mỗi level mở khoá 1 capability mới mà level trước không có. Học viên không nên skip level. Nếu đang mắc ở Level 4 (Geneve), nguyên nhân thường là chưa solid Level 2-3 (logical pipeline). Quay lại học lại.

## 20.7.1 Tổng quan 5 level

| Level | Tool chính | Phạm vi | Khi nào dùng |
|-------|-----------|---------|--------------|
| L1 | `ovn-trace` | 1 LS đơn, 2 LSP | Học khái niệm logical flow trace |
| L2 | `ovn-trace --detailed` | LS với ACL stateful | Debug ACL match priority |
| L3 | `ovn-trace --ovs` | LS + LR cross-subnet | Routing + ARP resolve |
| L4 | `ovn-trace` + `ofproto/trace` | Cross-chassis Geneve | Tunnel encap, Port_Binding |
| L5 | `ovn-detrace` chain | Production incident | Forensic timeline + correlation |

Mỗi level có cùng cấu trúc 4 phần:

1. **Bối cảnh + setup**, kịch bản tối giản, các bước chuẩn bị.
2. **Command**, lệnh chính + flag quan trọng.
3. **Anatomy output**, đọc kết quả từng phần.
4. **▶ Exercise**, hands-on luyện tập có expected output + bài học.

## 20.7.2 Level 1, hello-world `ovn-trace`

### 20.7.2.1 Bối cảnh + setup

**Kịch bản:** 1 Logical Switch `ls-hello`, 2 LSP `vm1` và `vm2`. VM1 ping VM2.

**Setup tối giản:**

```bash
[host]# ovn-nbctl ls-add ls-hello
[host]# ovn-nbctl lsp-add ls-hello vm1
[host]# ovn-nbctl lsp-set-addresses vm1 "00:00:00:00:00:01 10.0.0.1"
[host]# ovn-nbctl lsp-add ls-hello vm2
[host]# ovn-nbctl lsp-set-addresses vm2 "00:00:00:00:00:02 10.0.0.2"
```

(Không cần chạy thực tế. `ovn-trace` mô phỏng dựa trên NBDB ý đồ.)

### 20.7.2.2 Command

```bash
[host]# ovn-trace ls-hello \
        'inport == "vm1" && eth.src == 00:00:00:00:00:01 && \
         eth.dst == 00:00:00:00:00:02 && ip4.src == 10.0.0.1 && \
         ip4.dst == 10.0.0.2 && ip.ttl == 64'
```

Cú pháp chung: `ovn-trace <datapath-name> <flow-expression>`.

`<flow-expression>` mô tả packet, dùng cú pháp **OVN match expression** (xem `ovn-sb(5)` man page §"Logical Flow"). Tối thiểu cần `inport` + L2 (eth.src/eth.dst) + L3 (ip4 hoặc ip6 fields).

### 20.7.2.3 Anatomy output

```
# ip,reg14=0x1,vlan_tci=0x0000,dl_src=00:00:00:00:00:01,dl_dst=00:00:00:00:00:02,...
ingress(dp="ls-hello", inport="vm1")
---------------------------------------
 0. ls_in_port_sec_l2 (northd.c:5614): inport == "vm1" && eth.src == {00:00:00:00:00:01}, priority 50, uuid abcd1234
    next;
 1. ls_in_port_sec_ip (northd.c:5210): ..., priority 90, uuid ...
    next;
27. ls_in_l2_lkup (northd.c:8127): eth.dst == 00:00:00:00:00:02, priority 50, uuid efgh5678
    outport = "vm2";
    output;

egress(dp="ls-hello", inport="vm1", outport="vm2")
---------------------------------------
 0. ls_out_pre_acl (...): ..., priority 0
    next;
 9. ls_out_port_sec_l2 (...): outport == "vm2" && eth.dst == {00:00:00:00:00:02}, priority 50
    output;
        /* output to "vm2", type "" */
```

| Phần output | Ý nghĩa | Cảnh báo |
|-------------|---------|----------|
| `# ip,reg14=...` | Packet header sau parse từ flow-expression | Sai header → expression syntax lỗi |
| `ingress(dp=..., inport=...)` | Bắt đầu pipeline ingress của datapath | Datapath name khớp NBDB |
| Mỗi dòng `N. ls_in_*` | Stage trong pipeline (xem Phần 13.2 LS pipeline) | Skip stage = không có Logical_Flow match |
| `priority N, uuid ...` | Priority + UUID Logical_Flow match | Multiple flow cùng priority có thể conflict |
| Action `next;` | Goto stage tiếp theo | Action `drop;` = packet bị bỏ |
| `outport = "vm2"; output;` | Set outport + emit | Bắt đầu egress pipeline |
| `egress(dp=..., outport=...)` | Bắt đầu egress | Stage egress thường ít hơn ingress |
| `/* output to "vm2", type "" */` | Comment cuối, target physical | Type "" = vif (LSP local hoặc remote) |

### 20.7.2.4 ▶ Exercise L1

**Mục tiêu:** Kiểm chứng hiểu cấu trúc output của `ovn-trace`.

**Câu hỏi:**

1. Có bao nhiêu stage `ls_in_*` được hit trong trace ingress?
2. Stage nào đầu tiên match Logical_Flow của ACL? Nếu chưa có ACL trong setup, stage `ls_in_acl` action là gì?
3. Stage nào quyết định outport?
4. Khi packet rời datapath ls-hello, action cuối là gì?

**Expected output (doc-plausible):**

1. Khoảng 27 stage ingress + 10 stage egress (OVN 22.03 LS pipeline).
2. `ls_in_acl` (stage ~9 trong ingress) action `next;` (mặc định no-op khi NBDB chưa có ACL row).
3. `ls_in_l2_lkup` (stage 27) qua MAC destination match.
4. `output;` ở `ls_out_port_sec_l2` (stage cuối egress).

**Bài học L1:**

- `ovn-trace` simulate logical pipeline, không touch physical OVS hay packet thực.
- Mỗi stage có Logical_Flow đã được ovn-northd compile từ NBDB; trace match cao nhất priority.
- Stage skip (không hiện trong trace) = không có flow match → `next;` ngầm hoặc drop.

## 20.7.3 Level 2, multi-table pipeline với ACL stateful

### 20.7.3.1 Bối cảnh + setup

**Kịch bản:** Thêm ACL deny tcp.dst == 22 (block SSH) cho LSP `vm2`. Trace 2 packet: (a) ICMP ping (allowed), (b) TCP SSH (denied).

**Setup tăng từ Level 1:**

```bash
[host]# ovn-nbctl pg-add pg-vm2 vm2
[host]# ovn-nbctl acl-add pg-vm2 to-lport 1000 \
        'tcp.dst == 22' drop
```

### 20.7.3.2 Command với `--detailed`

```bash
# Packet ICMP (allowed)
[host]# ovn-trace --detailed ls-hello \
        'inport == "vm1" && eth.src == 00:00:00:00:00:01 && \
         eth.dst == 00:00:00:00:00:02 && ip4.src == 10.0.0.1 && \
         ip4.dst == 10.0.0.2 && ip.ttl == 64 && icmp4'

# Packet TCP SSH (denied)
[host]# ovn-trace --detailed ls-hello \
        'inport == "vm1" && eth.src == 00:00:00:00:00:01 && \
         eth.dst == 00:00:00:00:00:02 && ip4.src == 10.0.0.1 && \
         ip4.dst == 10.0.0.2 && ip.ttl == 64 && \
         tcp && tcp.src == 12345 && tcp.dst == 22'
```

Flag `--detailed` thêm:

- Hiển thị tất cả stage được duyệt (kể cả `next;` no-op).
- In ra `match` expression đầy đủ với UUID Logical_Flow.
- Hiện `ct_state` cho conntrack stage.

### 20.7.3.3 Anatomy output trace SSH (denied)

```
# tcp,reg14=0x1,...,nw_src=10.0.0.1,nw_dst=10.0.0.2,tp_src=12345,tp_dst=22
ingress(dp="ls-hello", inport="vm1")
---------------------------------------
 0. ls_in_port_sec_l2 (...): ...
    next;
 ...
 5. ls_in_pre_acl (...): ip, priority 100
    reg0[0] = 1;     /* mark for ct() */
    next;
 7. ls_in_acl_hint (...): !ct.trk, priority 7
    reg0[10] = 1; reg0[11] = 1;
    next;
 8. ls_in_acl (...): ip && reg0[10] == 1 && reg0[11] == 1, priority 65532
    ct_next;
                                  ^^^^^^^^^^^
                                  ct() invocation
 8. ls_in_acl (...): ip && !ct.est && reg0[7] == 1 && (outport == @pg-vm2 && tcp.dst == 22), priority 1000
    /* drop */
                                  ^^^^^^^^^^^
                                  ACL denial fires here
```

| Stage trong output | Ý nghĩa | Diagnostic hint |
|--------------------|---------|------------------|
| `ls_in_pre_acl` | Set reg0[0] để mark cho ct() invoke sau | Stateful ACL chuẩn bị conntrack |
| `ls_in_acl_hint` | Set reg0[10]+[11] cho match `allow-stateless` vs `allow-related` | Hint cho ACL stage tiếp theo |
| `ls_in_acl ct_next` action | ACL trigger conntrack lookup, sau đó re-evaluate | `ct.trk` flag được set |
| Sau `ct_next`, ACL stage chạy lại với `ct.est` flag set | Match rule deny SSH → `/* drop */` | Action `drop` = pipeline kết thúc, packet không tới egress |

So sánh với trace ICMP (allowed):

```
 8. ls_in_acl (...): ip && !ct.est && reg0[7] == 1 && (outport == @pg-vm2 && tcp.dst == 22), priority 1000
    no match    ← ICMP không match TCP rule
 8. ls_in_acl (...): default, priority 0
    next;       ← fallthrough cho phép
```

### 20.7.3.4 ▶ Exercise L2

**Mục tiêu:** Hiểu interplay giữa ACL stateful + conntrack qua `ovn-trace`.

**Câu hỏi:**

1. Tại sao stage `ls_in_acl` xuất hiện 2 lần trong trace? Chuyện gì xảy ra giữa 2 lần?
2. Nếu thay ACL action từ `drop` thành `allow-related`, trace TCP SSH sẽ khác thế nào?
3. Conntrack state `ct.est` lúc nào được set? `ct_state` value của packet đầu tiên?
4. Pipeline có tới được egress không trong 2 case (ICMP vs SSH)?

**Expected output (doc-plausible):**

1. Lần 1 trigger `ct_next` để consult conntrack table; lần 2 evaluate ACL với conntrack flag đã có. Đây là cơ chế stateful của OVN ACL.
2. Action `allow-related` thay `drop` → packet đi tiếp + ovn-northd commit conntrack entry; reply traffic auto-allow.
3. `ct.est` set sau packet thứ 2 cùng connection (sau khi entry committed). Packet đầu có `ct.new`.
4. ICMP đi tới egress (output to vm2). SSH dừng ở `ls_in_acl drop` ingress, không tới egress.

**Bài học L2:**

- ACL stateful = ACL + conntrack. Trace cho thấy 2-pass evaluation rõ ràng.
- `--detailed` flag bắt buộc cho debug ACL. Không có flag chỉ thấy match cuối, không thấy quá trình.
- Mỗi action `drop` trong trace có line trace tương ứng → forensic clear cho rationale denial.

## 20.7.4 Level 3, cross-subnet qua Logical Router

### 20.7.4.1 Bối cảnh + setup

**Kịch bản:** 2 LS (`ls-tenant-a` 10.0.0.0/24 + `ls-tenant-b` 10.1.0.0/24) connect qua LR `lr-shared`. VM1 (10.0.0.10 trên ls-tenant-a) ping VM3 (10.1.0.20 trên ls-tenant-b).

**Setup:**

```bash
[host]# ovn-nbctl lr-add lr-shared
[host]# ovn-nbctl lrp-add lr-shared lrp-a aa:bb:cc:00:00:0a 10.0.0.1/24
[host]# ovn-nbctl lrp-add lr-shared lrp-b aa:bb:cc:00:00:0b 10.1.0.1/24

[host]# ovn-nbctl ls-add ls-tenant-a
[host]# ovn-nbctl lsp-add ls-tenant-a vm1
[host]# ovn-nbctl lsp-set-addresses vm1 "00:00:00:00:00:01 10.0.0.10"
[host]# ovn-nbctl lsp-add ls-tenant-a lsp-to-lrp-a
[host]# ovn-nbctl lsp-set-type lsp-to-lrp-a router
[host]# ovn-nbctl lsp-set-addresses lsp-to-lrp-a router
[host]# ovn-nbctl lsp-set-options lsp-to-lrp-a router-port=lrp-a

[host]# ovn-nbctl ls-add ls-tenant-b
[host]# ovn-nbctl lsp-add ls-tenant-b vm3
[host]# ovn-nbctl lsp-set-addresses vm3 "00:00:00:00:00:03 10.1.0.20"
[host]# ovn-nbctl lsp-add ls-tenant-b lsp-to-lrp-b
[host]# ovn-nbctl lsp-set-type lsp-to-lrp-b router
[host]# ovn-nbctl lsp-set-addresses lsp-to-lrp-b router
[host]# ovn-nbctl lsp-set-options lsp-to-lrp-b router-port=lrp-b
```

### 20.7.4.2 Command

```bash
[host]# ovn-trace --detailed ls-tenant-a \
        'inport == "vm1" && eth.src == 00:00:00:00:00:01 && \
         eth.dst == aa:bb:cc:00:00:0a && \
         ip4.src == 10.0.0.10 && ip4.dst == 10.1.0.20 && \
         ip.ttl == 64 && icmp4'
```

Lưu ý: `eth.dst` phải là MAC của LRP `lrp-a` (gateway ARP-resolved), không phải MAC của vm3.

### 20.7.4.3 Anatomy output, qua 3 datapath

Output sẽ có **3 phase**: ls-tenant-a ingress → lr-shared (cross datapath) → ls-tenant-b egress.

```
ingress(dp="ls-tenant-a", inport="vm1")
---------------------------------------
27. ls_in_l2_lkup: eth.dst == aa:bb:cc:00:00:0a (LRP MAC), priority 50
    outport = "lsp-to-lrp-a"; output;

egress(dp="ls-tenant-a", outport="lsp-to-lrp-a")
---------------------------------------
 9. ls_out_port_sec_l2: ...
    output;        /* output to "lsp-to-lrp-a", type "router" */

ingress(dp="lr-shared", inport="lrp-a")
---------------------------------------
 ...
10. lr_in_ip_routing: ip4.dst == 10.1.0.0/24, priority 49
    ip.ttl--;
    reg0 = 10.1.0.20;        /* nexthop = dst since same subnet */
    reg1 = 10.1.0.1;          /* src ip = LRP-b ip */
    eth.src = aa:bb:cc:00:00:0b;
    outport = "lrp-b";
    flags.loopback = 1;
    next;
11. lr_in_arp_resolve: outport == "lrp-b" && ip4 && reg0 == 10.1.0.20, priority 100
    eth.dst = 00:00:00:00:00:03;     /* arp.tha learned */
    next;
22. lr_in_arp_request: ...
    next;

egress(dp="lr-shared", outport="lrp-b")
---------------------------------------
 ...
 5. lr_out_delivery: outport == "lrp-b", priority 110
    output;        /* output to "lrp-b" */

ingress(dp="ls-tenant-b", inport="lsp-to-lrp-b")
---------------------------------------
 ...
27. ls_in_l2_lkup: eth.dst == 00:00:00:00:00:03, priority 50
    outport = "vm3"; output;

egress(dp="ls-tenant-b", outport="vm3")
---------------------------------------
 ...
 9. ls_out_port_sec_l2: ...
    output;        /* output to "vm3", type "" */
```

**Điểm quan sát:**

| Phase | Hoạt động chính | Note |
|-------|-----------------|------|
| LS-A ingress | L2 lookup, MAC LRP-a, output to router LSP | Packet rời LS-A |
| LR-shared ingress | IP routing, dec_ttl, ARP resolve, eth.src/eth.dst rewrite | Cross-subnet logic |
| LR-shared egress | Delivery to LRP-b | LR phase done |
| LS-B ingress | L2 lookup, MAC vm3, output | Final delivery prep |
| LS-B egress | Port security, output | Packet tới vm3 |

### 20.7.4.4 ▶ Exercise L3

**Mục tiêu:** Hiểu cross-subnet routing trong OVN.

**Câu hỏi:**

1. Tại sao trace có 3 datapath? Tại sao không phải 1?
2. Stage nào của LR thực hiện dec_ttl?
3. ARP cho 10.1.0.20 được resolve ở stage nào? Resolve thành công vì sao? (Hint: NBDB đã có MAC binding.)
4. Tại sao `flags.loopback = 1` được set ở `lr_in_ip_routing`?
5. Nếu vm3 chưa có MAC binding (ARP chưa resolve), trace sẽ kết thúc ở đâu?

**Expected output (doc-plausible):**

1. Logical pipeline tách LS và LR. Mỗi datapath có pipeline riêng. Cross-subnet = 2 LS + 1 LR = 3 datapath.
2. `lr_in_ip_routing` (stage ~10).
3. `lr_in_arp_resolve` (stage ~11). Resolve thành công vì NBDB lsp-set-addresses đã set MAC tĩnh "00:00:00:00:00:03 10.1.0.20" cho vm3, ovn-northd compile thành Logical_Flow với eth.dst rewrite.
4. `flags.loopback = 1` để cho phép packet pass through router L3, không bị drop bởi anti-spoofing.
5. Pipeline dừng ở `lr_in_arp_request` (stage ~22) với action `controller`/`output:CONTROLLER`. ovn-controller sẽ generate ARP request + buffer packet đến khi ARP reply.

**Bài học L3:**

- Cross-subnet packet đi qua nhiều datapath, mỗi datapath có pipeline riêng. Đây là cơ chế OVN scale: thêm datapath không tăng pipeline length.
- LR routing không chỉ là dec_ttl mà còn rewrite eth.src/eth.dst (router function thông thường).
- ARP resolve trong OVN là **passive**: chỉ check NBDB (đã có address binding) hoặc MAC_Binding SBDB (dynamic). Nếu không có, packet bị buffer, ARP request dispatched.

## 20.7.5 Level 4, multichassis Geneve tunnel

### 20.7.5.1 Bối cảnh + setup

**Kịch bản:** vm1 trên chassis A, vm2 trên chassis B (cùng LS). vm1 ping vm2. Trace cần show: (a) logical pipeline trên A + B, (b) physical OpenFlow trên A đẩy gói qua tunnel, (c) physical OpenFlow trên B receive.

**Setup giả định:** Đã có 2 chassis, vm1 bind chassis-A, vm2 bind chassis-B. Đầy đủ tunnel Geneve UDP 6081.

### 20.7.5.2 Combine `ovn-trace` + `ofproto/trace`

**Bước 1: Logical view qua `ovn-trace`**

```bash
[host-anywhere]# ovn-trace ls-hello \
        'inport == "vm1" && eth.src == 00:00:00:00:00:01 && \
         eth.dst == 00:00:00:00:00:02 && ip4.src == 10.0.0.1 && \
         ip4.dst == 10.0.0.2 && ip.ttl == 64 && icmp4'
```

Output tương tự Level 1 (single datapath ls-hello). Logical view không thấy chassis, chỉ thấy LS pipeline.

**Bước 2: Physical view trên chassis A qua `ofproto/trace`**

```bash
[chassis-A]# TAP1_OFPORT=$(ovs-vsctl get Interface tap-vm1 ofport)
[chassis-A]# ovs-appctl ofproto/trace br-int \
            "in_port=${TAP1_OFPORT},ip,nw_src=10.0.0.1,nw_dst=10.0.0.2,\
             dl_src=00:00:00:00:00:01,dl_dst=00:00:00:00:00:02"
```

Output (rút gọn):

```
Flow: ip,in_port=15,...,nw_src=10.0.0.1,nw_dst=10.0.0.2,nw_proto=1,...

bridge("br-int")
----------------
 0. in_port=15, priority 100, cookie 0xabc
    set_field:0x1->reg14
    set_field:0xa01->metadata
    resubmit(,8)
 8. metadata=0xa01,reg14=0x1, priority 50, cookie 0x123
    next stage logical pipeline...
...
65. metadata=0xa01,reg15=0x2, priority 100, cookie 0xdef
    set_field:0x1->tun_id      ← Geneve VNI = datapath tunnel_key
    set_field:0x2->NXM_NX_TUN_METADATA0[0..31]   ← outport TLV
    move:NXM_NX_REG14[0..14]->NXM_NX_TUN_METADATA0[16..30]   ← inport TLV
    output:1                                                      ← tunnel port
```

**Bước 3: Physical view trên chassis B qua `ofproto/trace`**

```bash
[chassis-B]# ovs-appctl ofproto/trace br-int \
            "in_port=1,ip,nw_src=10.0.0.1,nw_dst=10.0.0.2,\
             dl_src=00:00:00:00:00:01,dl_dst=00:00:00:00:00:02,\
             tun_id=0x1,tun_metadata0=0x00020001"
```

`in_port=1` là tunnel port (Geneve). `tun_id` + `tun_metadata0` mang theo metadata từ chassis A.

Output trên B:

```
bridge("br-int")
----------------
 0. in_port=1,tun_id=0x1, priority 100
    move:NXM_NX_TUN_METADATA0[0..31]->NXM_NX_REG15[0..31]   ← outport restore
    move:NXM_NX_TUN_METADATA0[16..30]->NXM_NX_REG14[0..14]  ← inport restore
    move:NXM_NX_TUN_ID[0..23]->OXM_OF_METADATA[0..23]       ← tunnel_key restore
    resubmit(,33)
33. metadata=0xa01,reg15=0x2, priority 50
    output:tap-vm2
```

### 20.7.5.3 Anatomy: tương quan logical ↔ physical

| Logical (`ovn-trace`) | Physical chassis A (`ofproto/trace`) | Physical chassis B (`ofproto/trace`) |
|----------------------|---------------------------------------|---------------------------------------|
| `ingress(dp=ls-hello, inport=vm1)` | Table 0: in_port=15 → set metadata + reg14 | Table 0: tun_id+tun_metadata → restore metadata |
| Mỗi `ls_in_*` stage | Tables 8-34 (logical ingress) | Tables 8-34 |
| Mỗi `ls_out_*` stage | Tables 41-66 (logical egress) | Tables 41-66 |
| `output;` cuối egress | Table 65: encap + output:tunnel | Table 33: output:tap-vm2 |

Mỗi stage logical map sang OpenFlow table cụ thể. Cross-chassis: tunnel encap (table 65 trên A) + tunnel decap (table 0 trên B) tự động nhờ `put_encapsulation()` trong `controller/physical.c` (Phần 13.7.8).

### 20.7.5.4 ▶ Exercise L4

**Mục tiêu:** Kiểm chứng metadata Geneve TLV được preserve qua tunnel.

**Câu hỏi:**

1. Trên chassis A, OpenFlow action nào encode tunnel_key vào Geneve VNI?
2. Trên chassis A, action nào encode `outport` (logical egress port) vào Geneve TLV?
3. Trên chassis B, action nào restore `outport` từ TLV vào reg15?
4. Nếu tunnel UDP 6081 bị drop bởi underlay firewall, trace nào sẽ fail và tại sao?
5. `ovn-trace` mô phỏng được tới chassis B không?

**Expected output (doc-plausible):**

1. `set_field:0x1->tun_id` (24-bit Geneve VNI = datapath tunnel_key).
2. `set_field:0x2->NXM_NX_TUN_METADATA0[0..31]` (TLV class 0x0102, type 0x80, 32-bit outport).
3. `move:NXM_NX_TUN_METADATA0[0..31]->NXM_NX_REG15[0..31]`.
4. `ovn-trace` không fail (logical view không touch underlay). `ofproto/trace` chassis A vẫn produce output (action match), nhưng packet thực không tới B. `ofproto/trace` chassis B không có gì để trace (in_port không bao giờ thấy packet). Phải dùng `tcpdump -i eth1 udp port 6081` underlay để xác nhận.
5. `ovn-trace` không thấy chassis: nó là logical view duy nhất, không phân biệt chassis A vs B (cùng datapath). Phải combine với `ofproto/trace` trên từng chassis để có physical view.

**Bài học L4:**

- `ovn-trace` cho logical pipeline view, đơn giản nhưng không thấy tunnel.
- `ofproto/trace` cho physical OpenFlow, chi tiết nhưng phải chạy trên từng chassis riêng.
- Combine cả 2 mới complete debugging cross-chassis. Phần 9.27 đã đi sâu 3-tier parallel diagnostic này.

## 20.7.6 Level 5, production incident với `ovn-detrace`

### 20.7.6.1 Bối cảnh

**Kịch bản:** Production incident: VM `vm-prod-456` báo "no network" sau live migration. Kỹ sư on-call cần forensic trace tại sao OpenFlow rule hiện tại không serve traffic của vm-prod-456.

`ovn-detrace` là tool nâng cao chain với `ofproto/trace` (Phần 20.2 §20.2.X chi tiết): nhận output OpenFlow + correlate ngược về OVN logical context (Logical_Flow UUID, NBDB ACL/LB row, datapath name).

### 20.7.6.2 Command

**Bước 1: chạy `ofproto/trace` trên chassis B (chassis claim mới)**

```bash
[chassis-B]# TAP_OFPORT=$(ovs-vsctl get Interface tap-vm-prod-456 ofport)
[chassis-B]# ovs-appctl ofproto/trace br-int \
            "in_port=${TAP_OFPORT},ip,nw_src=10.5.5.10,nw_dst=10.5.5.20,..." \
            --names > /tmp/of-trace.txt
```

`--names` flag: thay UUID/cookie bằng human-readable identifier.

**Bước 2: chain với `ovn-detrace`**

```bash
[chassis-B]# cat /tmp/of-trace.txt | ovn-detrace > /tmp/of-trace-annotated.txt
```

Hoặc 1-pipeline:

```bash
[chassis-B]# ovs-appctl ofproto/trace br-int "..." --names | ovn-detrace
```

### 20.7.6.3 Anatomy output annotated

```
Flow: ip,in_port=15,...,nw_src=10.5.5.10,nw_dst=10.5.5.20,...

bridge("br-int")
----------------
 0. in_port=15,priority=100,cookie=0xa1b2
    /* logical ingress to "ls-prod-app", inport "vm-prod-456" */          ← from ovn-detrace
    set_field:0x1->reg14
    ...
 8. metadata=0xa01,reg14=0x1,priority=65532,cookie=0xc3d4
    /* Logical_Flow uuid 0xc3d4 from ls_in_acl, NBDB row uuid abcd... */   ← from ovn-detrace
    /* match: !ct.est && reg0[7] == 1 && (ip4.src == 10.5.5.10) */
    /* drop */
    --- pipeline drops here ---
```

ovn-detrace inject 3 loại thông tin:

1. **Datapath name**: `"ls-prod-app"` thay vì `metadata=0xa01`.
2. **Logical_Flow context**: stage name `ls_in_acl` + uuid + match expression nguyên gốc.
3. **NBDB row uuid**: traceback đến record cụ thể (ví dụ ACL row), giúp kỹ sư biết chính xác config nào gây drop.

### 20.7.6.4 Workflow forensic full

```
1. Start: alert "vm-prod-456 no network"
   ↓
2. ovn-trace ls-prod-app '...'
   → check logical intent OK?
   ↓ (logical OK)
3. ssh chassis-B (claim chassis)
   ovs-appctl ofproto/trace br-int "..." --names | ovn-detrace
   → show NBDB row uuid của Logical_Flow gây drop
   ↓ (e.g., uuid abcd)
4. ovn-nbctl find ACL _uuid=abcd
   → show ACL rule cụ thể: from-lport, priority=1000, match=..., action=drop
   ↓
5. Root cause: ACL bị set sai sau migration. Fix qua ovn-nbctl acl-del / acl-add.
```

### 20.7.6.5 ▶ Capstone POE Phase I.B3 (đóng Phase I)

**Mục tiêu:** Sinh viên tự design trace scenario cho 1 incident chưa biết, demo chọn level đúng + giải thích rationale.

**Bối cảnh:** Sinh viên nhận incident description (ví dụ: "tenant T2 mất kết nối với database tenant T3 sau khi reload ACL"). Sinh viên phải:

1. **Chọn level** (1-5) phù hợp incident.
2. **Soạn command** cụ thể với expression match đúng.
3. **Predict** output trace sẽ khớp/không khớp với chỗ packet được expect.
4. **Observe** thực tế trace (lab có thì run; chưa có thì giả định).
5. **Explain** rationale: tại sao chọn level đó, tại sao không level khác, kết quả trace cho thấy gì.
6. **Falsification test**: nếu trace cho kết quả khác predict, hypothesis nào sai?

**Tiêu chí chấm:**

| Criterion | Excellent | Good | Needs work |
|-----------|-----------|------|------------|
| Chọn level | Đúng level đầu tiên thử | Đúng sau 1 lần điều chỉnh | Không chọn được level phù hợp |
| Soạn command | Expression match chính xác đầu vào | Cần fix syntax 1-2 lần | Sai syntax nhiều lần |
| Predict | Predict match observation | Predict gần đúng | Predict ngược |
| Explain | Rationale đầy đủ logic + cross-link Phần khác | Rationale OK nhưng thiếu cross-link | Rationale yếu |
| Falsification | Identify hypothesis sai chính xác | Identify gần đúng | Confusion |

**Tự đánh giá:**

Nếu sinh viên chọn Level 1 cho incident cross-subnet → wrong level (Level 1 chỉ trong 1 LS). Đúng phải Level 3+.

Nếu sinh viên chọn Level 5 cho hello-world → over-engineering. `ovn-trace` đơn giản đủ.

Đáp án mẫu cho incident "tenant T2 mất kết nối DB tenant T3 sau ACL reload":

- **Level chọn:** Level 2 (ACL stateful) vì incident liên quan ACL change.
- **Command:** `ovn-trace --detailed ls-tenant-t2 'inport == "db-client" && eth.src == ... && ip4.dst == <T3-DB-IP> && tcp.dst == 5432'`
- **Predict:** ACL `ls_in_acl` action `drop` xuất hiện cho rule mới gây block.
- **Observe:** Trace stop ở `ls_in_acl drop` priority X cho rule Y.
- **Explain:** ACL reload đã add rule deny không cố ý. Fix qua `ovn-nbctl acl-del` rule sai.
- **Falsification:** Nếu trace cho `allow-related; output;` cuối egress → ACL không phải nguyên nhân. Phải investigate Level 4 (cross-chassis) hoặc Level 5 (production incident toàn cluster).

## 20.7.8 Điểm cốt lõi cần nhớ

**Một.** `ovn-trace` simulate logical pipeline, schema-aware. `ofproto/trace` simulate physical OpenFlow. Cả 2 không cần packet thực, không touch network.

**Hai.** 5 level gradient là sư phạm: mỗi level mở khoá 1 capability (single LS → ACL → LR → multichassis → forensic correlation). Skip level dẫn đến confusion.

**Ba.** `--detailed` flag cho `ovn-trace` bắt buộc ở Level 2+. Không có flag → mất stage no-op, mất ACL rationale.

**Bốn.** Cross-subnet trace xuyên 3 datapath (LS-A → LR → LS-B). Mỗi datapath có pipeline riêng, không tăng pipeline length.

**Năm.** `ovn-detrace` chain với `ofproto/trace --names` cho production incident: annotate OpenFlow output bằng NBDB row UUID + Logical_Flow context + datapath name.

**Sáu.** Workflow chọn level: 3 câu hỏi (cùng LS? cùng chassis? production?). Bắt đầu Level thấp, escalate khi cần.

**Bảy.** Trace là tool **first-line**, không phải last resort. Ở mọi incident OVN, chạy `ovn-trace` trước khi đoán mò. Trace 30 giây thường đủ xác định 80% root cause.

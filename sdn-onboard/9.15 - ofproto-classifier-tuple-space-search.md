## 9.15.1 Vai trò classifier trong stack OVS

Classifier là cấu trúc dữ liệu trung tâm của **ovs-vswitchd**, lưu trữ toàn bộ flow rule OpenFlow và cho phép tra cứu rule tương ứng với một packet header bất kỳ. Khi một packet đi vào OVS, nếu không khớp với megaflow cache trong kernel, packet được "upcall" tới ovs-vswitchd. Tại đây, classifier được truy vấn với packet header → trả về rule khớp với có priority cao nhất → rule này được compile thành megaflow và cài đặt ngược về kernel datapath cho các packet tương tự sau.

Bài toán classifier giải quyết: cho flow table có N rule với khớp với field phức tạp (128-bit IPv6 address, 12-tuple L2-L4), nhanh chóng xác định rule khớp với với packet P. Đơn giản nhất là duyệt tuyến tính qua N rule — O(N), không khả thi với N = 10K rule + packet rate 1 Mpps. Cần thuật toán O(log N) hoặc O(1) amortized.

Các cách tiếp cận tra cứu packet phổ biến:
- **Duyệt tuyến tính:** O(N). Đơn giản, chậm.
- **Cây nhị phân (B-tree):** O(log N). Tốt cho tra cứu trên một cột nhưng phức tạp với multi-field khớp với có wildcard.
- **Hash map:** O(1). Chỉ hoạt động với exact match: không xử lý được wildcard (ternary match).
- **TCAM (Ternary CAM) hardware:** O(1). Giả i pháp phần cứng chuyên dụng, không áp dụng được cho switch phần mềm.
- **Cây quyết định (HiCuts, HyperCuts):** O(log N) nhưng memory usage tăng mũ với số field.
- **Tuple space search (TSS):** O(k) với k = số "tuple": thuật toán OVS dùng.

OVS chọn TSS vì ba lý do: (1) xử lý wildcard khớp với tự nhiên, (2) memory usage tuyến tính với số flow (không mũ), (3) update flow table O(1) khi thêm/xóa rule.

## 9.15.2 Tuple space search (TSS) — ý tưởng cốt lõi

TSS phát triển bởi Srinivasan et al. trong bài báo "Packet Classification Using Tuple Space Search" (SIGCOMM 1999). Ý tưởng chính: chia flow table thành nhiều **subtable**, mỗi subtable chứa các rule có cùng **pattern mask** (bit mask giống nhau). Tra cứu = truy vấn từng subtable với hash, lấy kết quả priority cao nhất.

**Định nghĩa tuple:** một "tuple" là một vector bit mask xác định "bit nào được khớp với exact, bit nào wildcard". Ví dụ trong flow table IPv4:
- Tuple A: match src IP /24 + dst IP /16 → mask = `(0xFFFFFF00, 0xFFFF0000, ...)` cho src/dst IP.
- Tuple B: match chỉ dst port + protocol → mask = `(0, 0, 0xFFFF, 0xFF)` cho src/dst IP/port/proto.
- Tuple C: match src MAC + VLAN ID → mask = `(0xFFFFFFFFFFFF, 0xFFF, 0, ...)`.

Mỗi rule thuộc duy nhất một tuple (xác định bởi mask field của rule). Rule có cùng mask → cùng tuple → cùng subtable.

**Thuật toán tra cứu:**
1. Với packet P, duyệt qua danh sách tuple trong classifier.
2. Với mỗi tuple T: áp mask T lên packet P → key K = P & T.mask. Truy vấn hash map của subtable T với key K.
3. Nếu match → lấy rule + priority của rule.
4. Sau khi duyệt hết tuple, trả về rule có priority cao nhất trong tất cả match.

**Độ phức tạp:** O(k) với k = số tuple. Trong thực tế OVS k thường 3-20 cho cluster môi trường production (10 rule/tuple trung bình). Đây là lý do classifier OVS đạt hàng triệu tra cứu/giây per CPU core.

**Cấu trúc dữ liệu OVS:**
```c
struct classifier {
    struct cls_subtable *subtables[];  // danh sách subtables
    // ... index metadata ...
};

struct cls_subtable {
    struct cmap rules;            // hash map: key (từ mask) → cls_rule
    struct minimatch mask;         // mask của subtable
    int max_priority;              // priority max trong subtable
    // ... metadata khác ...
};

struct cls_rule {
    struct minimatch match;        // match field
    int priority;                  // priority
    // ... dữ liệu action ...
};
```

`cmap` (Cuckoo hash map) là hash map không khóa của OVS, đọc không khóa, ghi dùng thao tác nguyên tử. Hỗ trợ nhiều reader + một writer đồng thời không tranh chấp.

## 9.15.3 Staged lookup — tối ưu cho priority sort

Naive TSS có vấn đề: phải duyệt TẤT CẢ tuple để tìm rule priority cao nhất, ngay cả khi tuple đầu tiên đã có rule priority rất cao. OVS tối ưu với **staged lookup**: sort tuple theo max_priority (cao nhất trước). Nếu khớp với rule có priority > max_priority của tuple còn lại → có thể thoát sớm.

Pseudo-code:
```
function tss_lookup(packet P, classifier C):
    best_rule = None
    best_priority = -infinity
    for tuple T in C.tuples sorted by T.max_priority DESC:
        if T.max_priority < best_priority:
            break  // thoát sớm — không tuple còn lại nào có thể cải thiện
        key = apply_mask(P, T.mask)
        rule = cmap_find(T.rules, key)
        if rule and rule.priority > best_priority:
            best_rule = rule
            best_priority = rule.priority
    return best_rule
```

Trong thực tế, nếu rule có phân phối priority phù hợp (một vài rule priority cao cho drop/allow, nhiều rule priority thấp cho default), thoát sớm trúng trong 1-2 tuple đầu → tra cứu gần O(1).

## 9.15.4 Tương tác với megaflow cache

Classifier tra cứu O(k) vẫn chậm hơn hash lookup O(1) trong kernel. Mỗi lần upcall mất khoảng microsecond per packet, không đủ cho traffic tốc độ cao. **Megaflow cache** trong kernel datapath giải quyết: cache kết quả classifier cho các "flow class" (packet có cùng output classifier).

**Quá trình caching:**
1. Packet P không khớp với megaflow → upcall ovs-vswitchd.
2. Classifier tra cứu → rule R với khớp với mask M.
3. ovs-vswitchd build megaflow: match = P & M (tập con bit match), action = R.action.
4. Cài megaflow vào kernel.
5. Packet P' với cùng pattern P' & M = P & M → match megaflow → kernel forward trực tiếp, không upcall.

Điểm quan trọng: field khớp với của megaflow là TẬP CON của field khớp với của rule, chỉ các bit thật sự ảnh hưởng decision. Nếu rule nói "match dst_ip /24", nhưng trong khi tra cứu classifier chỉ check dst_ip bit 0-23 và dst_port (cho priority sort thoát sớm), megaflow sẽ khớp với dst_ip /24 + dst_port.

**Classifier** là "đường chậm" (hàng nghìn pps), **megaflow** là "đường nhanh" (hàng triệu pps). Với workload ổn định, 99%+ packet khớp với megaflow → classifier rarely invoked.

## 9.15.5 Quan sát — classifier hoạt động

OVS cung cấp nhiều công cụ để inspect classifier:

**Xem số subtable và rule:**
```
[host]# ovs-appctl dpif/show-dp-features br0
```

**Dump OpenFlow rule với priority sort:**
```
[host]# ovs-ofctl -O OpenFlow15 dump-flows br0 --sort=priority
```

**Đếm upcall (số lần gọi classifier):**
```
[host]# ovs-appctl upcall/show
```
Output hiển thị tần suất upcall per-thread, tỷ lệ hit/miss của megaflow.

**Thống kê flow table:**
```
[host]# ovs-appctl bridge/dump-flows br0
```
Show cấu trúc flow table bên trong + phân vùng subtable.

**Dump table megaflow:**
```
[host]# ovs-appctl dpif/dump-flows br0
```
Output các megaflow đang active trong kernel, là cache của tra cứu classifier.

**Dấu hiệu flow table mất cân bằng:** nếu tần suất upcall cao (>1000/s) và tỷ lệ hit megaflow thấp (<95%) → flow table chưa tối ưu. Thường do:
- Quá nhiều rule với pattern mask khác nhau (k lớn).
- Phân phối priority xấu (nhiều rule cùng priority cao).
- Pattern traffic packet biến động cao (không hit cache).

## 9.15.6 Tinh chỉnh flow table cho hiệu quả classifier

Hướng dẫn thực tế khi thiết kế flow table OVS:

**(1) Gộp rule có cùng pattern match:** nếu có 100 rule cùng khớp với trên dst_ip /24 (khác nhau ở priority/action), chúng vào cùng subtable → 1 tuple, tra cứu O(1) cho nhóm này.

**(2) Tránh wildcard không cần thiết:** match càng chặt (nhiều bit exact) → ít subtable hơn. Bit wildcard = bit mask 0 = một bit mới trong pattern subtable.

**(3) Dùng priority sparingly:** flow table có 10 mức priority khác nhau cần 10 nhóm với priority sort. Flow table có 3 priority (high/mid/low) → chỉ 3 nhóm. Ít priority = thoát sớm hiệu quả hơn.

**(4) Tránh chuỗi matching sâu:** mỗi packet đi qua nhiều table (resubmit) = nhiều lần tra cứu classifier. Compile vào single table nếu có thể.

**(5) Giám sát số subtable:** dùng `ovs-appctl dpif/show` để theo dõi. Nếu số subtable tăng bất thường → có thể signal flow churn (rule bị add/remove liên tục).

**(6) Tận dụng megaflow learning:** megaflow tự động học subset bit. Lập trình viên không cần tối ưu. OVS tự làm. Chỉ cần tránh viết rule với subset xung đột (rule 1 phụ thuộc bit X, rule 2 phụ thuộc bit Y nhưng không tương tác → tạo nhiều biến thể megaflow).

## 9.15.7 Subtable structure, bên trong tuple space

§9.15.2-9.15.3 đã giới thiệu TSS + staged lookup conceptually. Mục này đi sâu vào **subtable** như một data structure — cách OVS thực sự organize memory + lookup per subtable.

### 9.15.7.1 Subtable anatomy

Mỗi subtable trong OVS classifier (source `lib/classifier.c`) là cấu trúc:

```
struct cls_subtable {
    struct cmap rules;              /* Hash table: mask(flow) → list of rules */
    struct minimask mask;            /* Mask của subtable này */
    int max_priority;                /* Priority cao nhất trong subtable */
    unsigned int max_count;          /* Số rule có priority = max_priority */

    /* Staged lookup stages */
    struct cls_subtable_entry ports_mask;   /* L4 port stage */
    struct cls_subtable_entry metadata_mask; /* Metadata stage */
    struct cls_subtable_entry fields_mask;   /* All other fields */
};
```

**Invariant:** tất cả rule trong một subtable có **cùng mask** (cùng field được match, cùng mask bit). Subtable khác nhau → mask khác nhau.

### 9.15.7.2 Lookup flow per subtable

Khi classifier nhận packet, với mỗi subtable:

1. Extract flow key từ packet theo `mask` của subtable.
2. Compute hash của masked key.
3. Lookup `cmap` (concurrent hash map).
4. Nếu hit → trả về best priority rule.
5. Nếu miss → continue subtable tiếp theo.

**Optimization:** nếu subtable có `max_priority < current_best_priority_found`, OVS skip luôn subtable đó (không thể thắng). Đây là lý do mà subtable được sort theo `max_priority` desc.

### 9.15.7.3 Đọc hiểu `dpctl/dump-flows` masked output

Khi dump kernel megaflow:

```shell
$ sudo ovs-dpctl dump-flows -m | head -3
recirc_id(0),in_port(3),eth(src=aa:bb:cc:00:00:01/ff:ff:ff:00:00:00,dst=aa:bb:cc:00:00:02/ff:ff:ff:ff:ff:ff),
  eth_type(0x0800),
  ipv4(src=10.1.0.0/255.255.0.0,dst=10.1.0.10,proto=6/0,tos=0/0,ttl=0/0,frag=no),
  tcp(src=0/0,dst=80/0xffff),
  packets:18234, bytes:2145678, used:0.082s, actions:4
```

**Anatomy mask field:**

| Field:value/mask | Inspect level |
|---|---|
| `eth(src=.../ff:ff:ff:00:00:00)` | OUI match (first 3 bytes), host wildcard |
| `eth(dst=.../ff:ff:ff:ff:ff:ff)` | Exact MAC |
| `ipv4(src=10.1.0.0/255.255.0.0)` | /16 prefix match |
| `ipv4(dst=10.1.0.10)` | Exact IP (no slash = implicit /32) |
| `ipv4(proto=6/0)` | Wildcard — **không khớp với proto, dù hiển thị "6"** (mask 0 = ignore) |
| `tcp(src=0/0)` | Full wildcard src port |
| `tcp(dst=80/0xffff)` | Exact port 80 |

**Subtable này có mask:** `in_port + eth_src[47:24] + eth_dst + eth_type + ipv4_src[31:16] + ipv4_dst + tcp_dst`. 7-field mask. Rule cùng subtable sẽ khớp với các bit này chính xác, khác giá trị cụ thể thì khác rule.

## 9.15.8 Prefix tree (trie) optimization cho IP classification

### 9.15.8.1 Vấn đề: IP range khớp với không hash tốt

Subtable hash-based lookup nhanh cho **exact mask** nhưng không cho **prefix match**. Rule với `nw_src=10.0.0.0/8` khớp với mọi IP 10.x.x.x. Nếu phân 10.x.x.x thành 256^3 ≈ 17 triệu exact entries → không khả thi.

TSS với prefix hoạt động:
- Một subtable cho mỗi unique mask length (/8, /16, /24, /32).
- Lookup từng subtable → maximum O(N_subtable) với N ≤ 32 cho IPv4.

Nhưng nếu rule có hàng trăm prefix độ dài khác nhau → hàng trăm subtable → lookup chậm.

### 9.15.8.2 Prefix tree giải pháp

OVS dùng **Patricia trie** cho IP matching inside classifier (source `lib/pvector.c` + `lib/classifier.c`). Cấu trúc:

```
Patricia trie cho rule IPv4 src:
                     root
                   /      \
              0.0.0.0      128.0.0.0
                /            /    \
           10.0.0.0      128.0.0.0  192.0.0.0
              |          /              |
         10.0.0.0/8    ...          192.168.0.0/16
              |
        10.1.0.0/16
              |
        10.1.0.0/24
```

Lookup IP packet 10.1.0.5:
1. Start root, bit 0 = 0 → left.
2. Bit 1 = 0 → left.
3. ... traverse từng bit cho đến leaf khớp với longest prefix.
4. Return rule với longest prefix match.

Complexity O(32) cho IPv4 bit-by-bit (hoặc O(log N) amortized với compression).

### 9.15.8.3 Kết hợp trie + TSS

OVS classifier **không dùng trie cho tất cả mask** — chỉ cho field có nhiều prefix khác nhau (typical: nw_src, nw_dst). Field khác (eth_src, tp_dst) vẫn dùng subtable hash.

Test xem classifier behave thế nào: chạy `coverage/show classifier` — counter class `classifier_prefix_tree` tăng khi prefix lookup hit.

## 9.15.9 Kịch bản bẻ gãy classifier performance

### 9.15.9.1 Subtable explosion

**Signature:**
- `pmd-stats-show avg. subtable lookups per megaflow hit` > 10
- `coverage/show classifier_lookup` rate cao
- CPU spike trên ofproto thread

**Root cause:** quá nhiều rule với mask unique → mỗi rule tạo subtable riêng → tuple space cardinality tăng.

**Fix:** normalize rule set — merge rule có mask khác nhau chỉ vài bit thành wider mask.

### 9.15.9.2 Priority sort churn

**Signature:**
- Rule add/del liên tục → classifier re-sort subtable theo max_priority
- `coverage/show classifier_rule_add` + `classifier_rule_del` rate cao

**Root cause:** controller cài đặt rule không atomic — `add + del + add` pattern gây sort overhead.

**Fix:** dùng `ovs-ofctl --bundle replace-flows` cho OpenFlow 1.4+ atomic bundle.

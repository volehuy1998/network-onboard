## Context: SDN does not have one single meaning

Parts 3 and 4 analysed OpenFlow as a protocol that replaces data plane forwarding decisions, moving from vendor-proprietary logic to centralised programmable control. Part 4.6 summarised five limits of OpenFlow and three evolution directions for the second generation. This part dives into one of those three directions: **API-based SDN**, the model that keeps switch silicon and firmware as they are and exposes a **standard API** so that automation tools can control the switch from outside.

The core philosophy is that vendors do not want to rebuild firmware to implement OpenFlow, and operators do not want to learn a new forwarding model when they already have a working vendor-specific ASIC. API-based SDN is a compromise. The switch still does routing and switching with its native logic (OSPF, BGP, STP), and the controller sends configuration through a standardised protocol (NETCONF, YANG, gNMI) to change state. This is the dominant approach in WAN, service provider, and enterprise edge, in contrast to OpenFlow, which is dominant in data centre virtualisation.

## 5.0.1 Vendor motivation: keep silicon, add an API layer

### Architectural trade-off

OpenFlow chooses to **replace the data plane**: the switch gives up native forwarding and accepts `FLOW_MOD` from the controller. Vendors must rewrite firmware, and silicon must support the OpenFlow pipeline abstraction.

SDN via APIs chooses to **keep the data plane**: the switch still runs OSPF, BGP, STP, and a proprietary forwarding pipeline. The only change is to add an API endpoint for configuration management. The switch exposes a YANG model, the controller sends configuration through NETCONF, RESTCONF, or gNMI, and the switch applies the changes through its native daemons.

### Trade-off comparison

| Aspect | OpenFlow-based | API-based |
|--------|----------------|-----------|
| Data plane | Replaced by flow tables | Vendor-specific, kept as is |
| Switch firmware | Major rewrite | Add a NETCONF server module |
| Flexibility | High (arbitrary flows) | Low (constrained by the vendor model) |
| Stability | Low (specification evolving) | High (protocols stable for more than 10 years) |
| Vendor adoption | Niche | Broad (Cisco, Juniper, Arista all support it) |
| Dominant use case | Data centre virtualisation | WAN, service provider, enterprise edge |

> **Common misconception:** "SDN means OpenFlow. API-based configuration is not real SDN."
>
> **Reality:** SDN is an umbrella concept about the separation of control plane and data plane, not a specific protocol. The IETF I2RS working group, ONF, and the Göransson textbook all classify API-based (NETCONF, YANG, gNMI) as legitimate SDN. The opposite argument, that API-based is not SDN, has been outdated since 2014, when Google B4 (Part 4.6.6), Cisco ACI (2013), and Juniper Contrail (2013) all adopted the API-based approach and ran at production scale.

## 5.0.2 NETCONF (RFC 6241, June 2011)

### Protocol stack

NETCONF is an XML-based configuration protocol for network devices. It was released as IETF RFC 6241 in June 2011, around the same time as ONF formation (March 2011). The two efforts were parallel attempts to solve the SDN pain point in different ways.

Transport:

- **SSH on port 830**: default, secure.
- **TLS**: an alternative for environments that do not deploy SSH.
- **BEEP, SOAP**: historical, rarely used in 2026.

Message format: XML over an RPC (Remote Procedure Call) pattern:

```xml
<rpc message-id="101" xmlns="urn:ietf:params:xml:ns:netconf:base:1.0">
  <get-config>
    <source><running/></source>
    <filter type="subtree">
      <interfaces xmlns="urn:ietf:params:xml:ns:yang:ietf-interfaces"/>
    </filter>
  </get-config>
</rpc>
```

The client (controller) sends an RPC, and the server (switch) replies with an XML response.

### Three datastores (§5 of RFC 6241)

NETCONF defines three logical datastores:

| Datastore | Description |
|-----------|-------|
| `running` | The active configuration in effect |
| `startup` | The persisted configuration, loaded at boot |
| `candidate` | Edit in progress, not yet committed |

Pattern workflow:

1. Client `get-config source=running` retrieves the current snapshot.
2. Compute a diff and generate edit operations.
3. `edit-config target=candidate` applies the edit to the candidate (isolated, does not affect running).
4. `validate candidate` checks syntax and semantics.
5. `commit` promotes candidate to running atomically.
6. Optional: `confirmed-commit timeout=N` for automatic rollback if the client does not follow up with a `commit` within N seconds. This is a safety net for a misconfiguration that disconnects the client.

### Operation set

| Operation | Description |
|-----------|-------|
| `get` | Retrieve configuration plus state data |
| `get-config` | Retrieve configuration only |
| `edit-config` | Modify configuration (merge, replace, create, delete) |
| `copy-config` | Copy an entire datastore |
| `lock` and `unlock` | Mutex on a datastore (single writer) |
| `commit` | Promote candidate to running |
| `confirmed-commit` | Commit with timeout rollback |
| `validate` | Check configuration validity |
| `close-session` and `kill-session` | Graceful or forceful termination |

### Transactional semantics

NETCONF guarantees atomic commit: either the entire candidate or nothing. If the edit sequence produces an invalid state, `validate` fails and `commit` does not execute. If `confirmed-commit` times out (the client disconnects), the switch rolls back automatically. This pattern is critical for remote configuration changes. If an administrator over SSH makes an edit that breaks the connection, the device rolls back and restores the previous state.

## 5.0.3 YANG data model (RFC 7950, August 2016)

### Schema language

YANG (Yet Another Next Generation) is a schema language that describes the **structure** of configuration and state data. RFC 7950, "The YANG 1.1 Data Modeling Language", was published in August 2016 and updates RFC 6020 (YANG 1.0, October 2010).

YANG differs from XML Schema (XSD). YANG is optimised for network configuration. It has built-in types (`ipv4-address`, `mac-address`, `yang:counter64`), constraints (`must`, `when`), and grouping reuse. XSD is generic for any XML and is verbose for network models.

### Module structure

```yang
module ietf-interfaces {
  namespace "urn:ietf:params:xml:ns:yang:ietf-interfaces";
  prefix "if";

  container interfaces {
    list interface {
      key "name";
      leaf name { type string; }
      leaf type { type identityref { base iana-if-type:iana-interface-type; } }
      leaf enabled { type boolean; default "true"; }
      container ipv4 {
        leaf address { type inet:ipv4-address; }
      }
    }
  }
}
```

Constructs:

| Construct | Description |
|-----------|-------|
| `module` | Top-level container, with namespace and prefix |
| `container` | A grouping of data nodes |
| `list` | A collection with a key (each element is unique by key) |
| `leaf` | A single-value data node |
| `leaf-list` | A list of leaves (a simple collection) |
| `grouping` | A reusable definition (similar to a struct in C) |
| `augment` | Extend another module (without modifying the original) |

### IETF standard modules

IETF ships standard modules for common models:

- **ietf-interfaces**: basic interface configuration.
- **ietf-routing**: routing instance and RIB.
- **ietf-ip**: IPv4 configuration (address, DHCP).
- **ietf-system**: hostname, DNS, NTP.
- **ietf-access-control-list**: ACL rules.

Vendors extend through `augment` instead of redefining:

```yang
module cisco-extensions {
  import ietf-interfaces { prefix "if"; }

  augment "/if:interfaces/if:interface" {
    leaf cisco-description { type string; }
    leaf cisco-port-channel-id { type uint16; }
  }
}
```

The Cisco-specific field is added without touching the `ietf-interfaces` module. The controller uses `ietf-interfaces` for common operations and `cisco-extensions` for vendor-specific operations. The result is portability across vendors.

### Vendor module ecosystem

- **Cisco YANG models**: `github.com/YangModels/yang/tree/main/vendor/cisco` (multiple operating systems: IOS-XE, IOS-XR, NX-OS).
- **Juniper Junos YANG**: `github.com/Juniper/yang`.
- **Arista EOS**: `github.com/aristanetworks/openmgmt`.
- **OpenConfig**: a vendor-agnostic alternative, bootstrapped by AT&T, Google, and Microsoft in 2014, at `github.com/openconfig/public`.

OpenConfig aims for vendor independence. It uses a cross-vendor identity such as `openconfig-interfaces` instead of a per-vendor identity. By 2026, OpenConfig is popular among service provider operators (AT&T, Verizon) and less common in enterprise (Cisco and Juniper stick to their own YANG).

## 5.0.4 RESTCONF (RFC 8040, January 2017)

### HTTP and JSON alternative

NETCONF XML was criticised as hard to integrate with modern DevOps tooling (most automation uses JSON over HTTP). RESTCONF was published as RFC 8040 in January 2017. It introduced an **HTTP plus JSON** alternative to NETCONF, while keeping the same YANG model.

The URL path mirrors the YANG tree:

```
GET /restconf/data/ietf-interfaces:interfaces/interface=GigabitEthernet0_0_0_0
```

HTTP method to CRUD mapping:

| HTTP | YANG operation |
|------|----------------|
| GET | get-config or get |
| POST | create (fails if exists) |
| PUT | replace (create or overwrite) |
| PATCH | merge (partial update) |
| DELETE | delete |

JSON encoding:

```json
{
  "ietf-interfaces:interface": {
    "name": "GigabitEthernet0/0/0/0",
    "type": "iana-if-type:ethernetCsmacd",
    "enabled": true,
    "ietf-ip:ipv4": {
      "address": [{"ip": "10.0.0.1", "netmask": "255.255.255.0"}]
    }
  }
}
```

Verbosity is reduced by about three times compared with XML.

### Why it appeared

RFC 6241 NETCONF (2011) predates modern DevOps tools (Ansible, Puppet, and Chef auto-config came in 2013 onward; Kubernetes appeared in 2014). These tools became dominant in workflow from 2015. The XML ecosystem was shrinking. RESTCONF responds by letting developers use familiar HTTP plus curl or requests, with a JSON parser built into most languages.

Adoption in 2026:

- Enterprise operators: RESTCONF is primary for automation scripts.
- Service providers: NETCONF is still used for configuration transactions (atomic commit matters more than HTTP familiarity).
- Hyperscalers: gNMI (see 5.0.6) replaces RESTCONF in production.

## 5.0.5 BGP-LS (RFC 7752) for topology distribution

### The problem: the controller needs to know the topology

A SDN WAN controller needs a real-time topology graph (nodes, links, metrics) to compute traffic engineering paths. In a traditional IGP network, topology information lives in the OSPF or IS-IS link-state database, and only routers participating in the IGP know it. The controller does not join the IGP (security risk and instability) and must import the topology out of band.

### BGP-LS mechanism (RFC 7752, March 2016)

BGP Link-State extends BGP to carry IGP topology information. Protocol:

1. **IGP to BGP reflector**: a router running BGP-LS exports the link-state database content to a BGP route reflector (RR).
2. **RR to controller**: the controller establishes a BGP-LS peering with the RR, and receives topology updates in real time.
3. **Controller process**: parse BGP-LS advertisements, build a topology graph in memory, and trigger TE re-optimisation when a link fails.

BGP-LS NLRI types:

- **Node NLRI**: router, area membership, capability.
- **Link NLRI**: link between nodes, metric, bandwidth.
- **Prefix NLRI**: an IPv4 prefix reachable through a link.

### Real-world use: Cisco WAE, Juniper NorthStar

**Cisco WAN Automation Engine (WAE)**: a commercial controller that uses BGP-LS for topology and PCE-P for path provisioning. Deployed at AT&T, Comcast, and Telstra.

**Juniper NorthStar**: an alternative commercial controller, with BGP-LS topology and PCE-P segment routing path control.

**Open-source alternatives**:

- **ExaBGP**: a BGP-LS parser that feeds topology to a controller application.
- **GoBGP**: a Go BGP implementation with link-state support.

### Trade-off compared with OpenFlow in WAN

The OpenFlow approach to WAN: install a flow rule per TE path into the router. Problems: 10,000 or more TE paths multiplied by 1,000 routers reaches millions of flow entries, which does not scale within router TCAM limits.

The BGP-LS plus PCE-P approach: routers retain normal IGP forwarding, and the controller only overrides specific TE paths. Flow entry overhead is minimal (per-LSP state), and it scales better.

Google B4 (Part 4.6.6) initially used OpenFlow for TE. The subsequent evolution moved toward segment routing plus PCE-P, closer to the BGP-LS model than to a pure OpenFlow approach.

## 5.0.6 PCE-P (RFC 5440) for TE path computation

### Path Computation Element architecture

Traditional routing computes paths in a distributed way (OSPF SPF, IS-IS SPF) at each router. For TE constraints (bandwidth, latency, diverse paths), SPF is not enough. **Centralised computation** is required.

A PCE (Path Computation Element) is a dedicated entity that computes constrained paths. PCE-P is the protocol that lets a PCC (PCE Client, usually a router) talk to a PCE:

- **PCC to PCE**: `PCReq`, a Path Computation Request with constraints (source, destination, bandwidth, affinity, disjointness).
- **PCE to PCC**: `PCRep`, a Path Computation Reply with path segments.

RFC 5440, "Path Computation Element (PCE) Communication Protocol", was published in March 2009 and has been stable for the long term.

### Stateful PCE (RFC 8231, September 2017)

The original PCE-P is stateless: the PCC asks, the PCE replies, and the PCE does not maintain LSP state. The Stateful PCE extension lets the PCE **own LSP state**: with write authority, the PCE can actively re-optimise paths.

Stateful PCE flow:

1. The PCC reports LSP state to the PCE (`PCRpt`).
2. The PCE maintains an authoritative LSP database.
3. The PCE monitors topology (via BGP-LS) and computes re-optimisation decisions.
4. The PCE sends `PCUpd` (Path Computation Update) to the PCC to install a new path.

Difference from OpenFlow: PCE-P operates at the **LSP level** (a logical tunnel), not at the packet level. Routers still process packet forwarding according to the installed LSP, and the PCE controls tunnel placement.

### Integration with Segment Routing

Segment Routing (SR, RFC 8402) is a packet forwarding model that replaces the MPLS label stack with segment identifiers (SIDs). PCE-P plus SR enables source-routed TE: the controller computes a segment list, and the PCC encapsulates the packet with the SID stack. There is no per-LSP state in transit routers, and it scales better than RSVP-TE.

Production deployment:

- **AT&T Flexware**: SDN-based WAN with PCE-P plus SR.
- **Comcast**: PCE-P plus SR-MPLS for the backbone.
- **Deutsche Telekom**: SRv6-based with PCE-P control.

## 5.0.7 Cisco ACI and Juniper Contrail launch in 2013

### Cisco ACI (Application Centric Infrastructure)

Launched in November 2013, around the same time as VMware NSX (see Part 5.1). Components:

- **APIC (Application Policy Infrastructure Controller)**: the centralised controller, running as a 3-node to 5-node cluster.
- **Nexus 9000 hardware**: a custom ASIC (Cisco Cloud Scale), supports a VXLAN fabric.
- **OpFlex**: the southbound protocol from APIC to switches. **It does not use OpenFlow**.
- **iVXLAN**: a custom VXLAN extension with group policy.

Northbound API of APIC: REST with JSON. The operator configures application-centric policy (contract, EPG or Endpoint Group), and APIC translates this into fabric configuration.

Cisco positions ACI as "better than OpenFlow": SDN that does not abandon Cisco hardware investment, and that exposes APIs for automation without forcing the operator to learn flow programming.

### Juniper Contrail

Launched in 2013, after Juniper acquired the startup Contrail Systems in December 2012. Components:

- **Contrail Controller**: the centralised control plane, with multi-node high availability.
- **vRouter** (compute agent): runs on each hypervisor, as a kernel module or with DPDK. It is a functional analog of OVS.
- **Southbound**: XMPP (Extensible Messaging and Presence Protocol) between the controller and vRouter. Not OpenFlow.
- **Control plane**: BGP EVPN for L2 and L3 VPN signalling.

Open-sourced as **OpenContrail** (2014), and renamed **Tungsten Fabric** under the Linux Foundation (2018).

### Counter-narrative to OpenFlow

Both ACI and Contrail market themselves as "SDN alternatives to OpenFlow". The ONF positioning that "OpenFlow equals SDN" was challenged by the reality of vendor adoption. ACI and Contrail use modern SDN concepts (centralised control, policy-driven, API-first) but do not use the OpenFlow protocol. This pushed the industry to recognise that SDN is broader than OpenFlow, matching the Göransson Chapter 6 classification of four types: Open SDN, SDN via APIs, Hypervisor Overlay, and Whitebox.

Forward cross-reference: Part 5.1 analyses the hypervisor overlay (NVP, NSX, OVN) as a counterpart. Part 5.2 analyses whitebox (Cumulus, SONiC). Part 7.3 will analyse the ACI and Contrail architecture in detail in the context of the controller ecosystem.

> **Key point:** "SDN via APIs" does not replace OpenFlow. It solves a subset of the SDN problem (configuration management plus policy automation) for segments that do not need data plane programmability (WAN, enterprise edge). OpenFlow solves a different subset (data centre virtualisation with dynamic flow installation). Both coexist and apply to different use cases. The lesson is that there is no "one SDN to rule them all". The right model depends on the deployment context.

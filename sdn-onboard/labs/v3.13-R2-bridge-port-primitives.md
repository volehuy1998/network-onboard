## F.2, Create the first bridge and inspect every OVSDB row it touched

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-br br0
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=_uuid,name,bridges list open_vswitch | head -10
ovs-vsctl: Open_vSwitch does not contain a column whose name matches "name"
```

The first lesson of R2 lands at command two: the `Open_vSwitch` table has no `name` column. The error is a useful one. The `Open_vSwitch` table is a **singleton** (exactly one row per host, holding global state), and OVS does not name singletons. The author of this lab attempted to ask for `name` out of muscle memory; the schema rejected the column unambiguously. The right query against the singleton is `ovs-vsctl --columns=_uuid,bridges list open_vswitch`. The error is preserved verbatim because correcting it after the fact would lose the lesson.

The next four queries inspect the rows the `add-br br0` actually wrote:

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=_uuid,name,ports,fail_mode,datapath_type list bridge br0
_uuid               : 420e4ca7-d757-442f-907e-706c3cabf354
name                : br0
ports               : [1a25f9b3-5b5c-4a5f-8e31-eb1332c3bc00]
fail_mode           : []
datapath_type       : ""
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=_uuid,name,interfaces,tag,trunks,vlan_mode list port br0
_uuid               : 1a25f9b3-5b5c-4a5f-8e31-eb1332c3bc00
name                : br0
interfaces          : [b9832d48-630b-417a-a93b-5a0540079158]
tag                 : []
trunks              : []
vlan_mode           : []
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=_uuid,name,type,ofport,mac_in_use,mtu list interface br0
_uuid               : b9832d48-630b-417a-a93b-5a0540079158
name                : br0
type                : internal
ofport              : 65534
mac_in_use          : "a6:4c:0e:42:2f:44"
mtu                 : 1500
root@lab-openvswitch:~/network-onboard# ip -br link show br0
br0              DOWN           a6:4c:0e:42:2f:44 <BROADCAST,MULTICAST> 
```

The `add-br` wrote rows to three OVSDB tables. The `Bridge` row at UUID `420e4ca7-d757-442f-907e-706c3cabf354` has empty `fail_mode` (the schema empty-set notation is `[]`), empty `datapath_type` (the schema empty-string notation is `""`), and one port reference. The single `Port` row also has empty `tag`, `trunks`, and `vlan_mode`. The `Interface` row has `type=internal`, `ofport=65534` (which is `OFPP_LOCAL`, the OpenFlow reserved port number for the bridge's host-visible interface), `mac_in_use="a6:4c:0e:42:2f:44"` (the auto-generated MAC the kernel assigned), and `mtu=1500` (the kernel default). The kernel's view of the same netdev confirms the MAC: `ip -br link show br0` reports the same address, with state `DOWN` (no IP yet) and `<BROADCAST,MULTICAST>` flags.

The lesson behind the four queries is that an OVS bridge is **three rows in three tables**, not a single object. The `Bridge` row holds bridge-level config (fail-mode, datapath type). The `Port` row holds port-level config (VLAN, bonding). The `Interface` row holds netdev-level config (MTU, MAC, type). The naming convention preserves the `name=br0` invariant across all three rows for the local internal port.

## F.3, The implicit-`NORMAL` flow and what happens when fail-mode toggles

A fresh bridge with no controller has exactly one OpenFlow flow installed. The flow is the implicit-`NORMAL` rule that turns the bridge into a learning switch.

```text
root@lab-openvswitch:~/network-onboard# ovs-ofctl dump-flows br0
 cookie=0x0, duration=4.927s, table=0, n_packets=0, n_bytes=0, priority=0 actions=NORMAL
```

One row: `priority=0,actions=NORMAL`. The `n_packets=0` and `n_bytes=0` tell the operator no traffic has hit the rule yet (the lab has not sent any packets). The duration of 4.927 seconds is the time since the rule was synthesised, which is the time since the bridge was created plus a small daemon-side delay.

What happens when fail-mode flips to `secure`?

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl set bridge br0 fail-mode=secure
root@lab-openvswitch:~/network-onboard# ovs-ofctl dump-flows br0
```

The `dump-flows` produces no output. The implicit rule is gone. With `fail-mode=secure` and no controller installed, the bridge has zero flow rules and drops every packet. This is the deliberately-defensive mode an operator uses when a controller is expected to install rules; the bridge does nothing on its own.

What happens when fail-mode flips back to `standalone`?

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl set bridge br0 fail-mode=standalone
root@lab-openvswitch:~/network-onboard# ovs-ofctl dump-flows br0
 cookie=0x0, duration=0.820s, table=0, n_packets=0, n_bytes=0, priority=0 actions=NORMAL
```

The implicit rule is back, with a fresh duration counter (0.820 seconds since the daemon re-synthesised it). This is `connmgr_set_fail_mode` in `ofproto/connmgr.c` doing its job: when fail-mode transitions and there is no controller, the daemon adds or removes the implicit `NORMAL` rule accordingly.

The pedagogical point: the implicit `NORMAL` is **not in the OVSDB**. It is a runtime synthesis by the connection manager. A bridge backed by configuration alone has zero stored OpenFlow rules; the visible-via-`dump-flows` rule is generated on the fly when fail-mode equals `standalone` or is unset.

## F.4, `ofproto/trace` with a synthesised broadcast packet

The trace command takes a packet header, walks the flow tables, and prints the resulting datapath actions. The first invocation uses an ARP broadcast as the synthetic packet:

```text
root@lab-openvswitch:~/network-onboard# ovs-appctl ofproto/trace br0 'in_port=LOCAL,dl_src=02:00:00:00:00:01,dl_dst=ff:ff:ff:ff:ff:ff,dl_type=0x0806' 2>&1 | tail -25
Flow: arp,in_port=LOCAL,vlan_tci=0x0000,dl_src=02:00:00:00:00:01,dl_dst=ff:ff:ff:ff:ff:ff,arp_spa=0.0.0.0,arp_tpa=0.0.0.0,arp_op=0,arp_sha=00:00:00:00:00:00,arp_tha=00:00:00:00:00:00

bridge("br0")
-------------
 0. priority 0
    NORMAL
     -> no learned MAC for destination, flooding

Final flow: unchanged
Megaflow: recirc_id=0,eth,arp,in_port=LOCAL,dl_src=02:00:00:00:00:01,dl_dst=ff:ff:ff:ff:ff:ff,arp_op=0
Datapath actions: drop
```

The trace tells four things. First, the flow header line at the top is the full extracted packet representation; the `arp,` prefix says OVS recognised the synthetic packet as ARP from `dl_type=0x0806`. Second, the bridge entry shows that the only matched rule was the priority-0 implicit `NORMAL`. Third, the explanatory line `-> no learned MAC for destination, flooding` is the engine telling the operator that `xlate_normal` reached the broadcast branch and decided to flood. Fourth, **`Datapath actions: drop`**. This is the most pedagogically loaded line in the entire R2 lab.

The flood-then-drop result is correct and worth pausing on. At the moment of this trace, `br0` has exactly one port: the local port `LOCAL`. `xlate_normal` chose to flood; `xlate_normal_flood` enumerates every port in the bridge except the input port; with only one port (the input), the resulting set is empty; the kernel datapath compiles "flood to nothing" into "drop". Later in the lab (after F.5 attaches two veth pairs), the same trace will produce non-empty datapath actions because the flood set is non-empty.

The `Megaflow:` line shows the wildcards the megaflow cache will use to install the kernel-side flow. The masks include `eth,arp,in_port=LOCAL,dl_src=...,dl_dst=...,arp_op=0`, meaning the kernel flow will match exactly on input port, source MAC, destination MAC, and ARP opcode, and ignore everything else. This is what `xlate_normal` set in `ctx->wc->masks` at the start of the function.

## F.5, Two-namespace ICMP through the bridge

This is the canonical OVS introduction lab: two Linux network namespaces, two veth pairs, one bridge, one ping. The setup is sixteen short commands.

```text
root@lab-openvswitch:~/network-onboard# ip netns add ns_red
root@lab-openvswitch:~/network-onboard# ip netns add ns_blue
root@lab-openvswitch:~/network-onboard# ip link add veth_red type veth peer name veth_red_br
root@lab-openvswitch:~/network-onboard# ip link add veth_blue type veth peer name veth_blue_br
root@lab-openvswitch:~/network-onboard# ip link set veth_red netns ns_red
root@lab-openvswitch:~/network-onboard# ip link set veth_blue netns ns_blue
root@lab-openvswitch:~/network-onboard# ip link set veth_red_br up
root@lab-openvswitch:~/network-onboard# ip link set veth_blue_br up
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br0 veth_red_br
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br0 veth_blue_br
root@lab-openvswitch:~/network-onboard# ip netns exec ns_red ip link set lo up
root@lab-openvswitch:~/network-onboard# ip netns exec ns_red ip link set veth_red up
root@lab-openvswitch:~/network-onboard# ip netns exec ns_red ip addr add 10.0.0.1/24 dev veth_red
root@lab-openvswitch:~/network-onboard# ip netns exec ns_blue ip link set lo up
root@lab-openvswitch:~/network-onboard# ip netns exec ns_blue ip link set veth_blue up
root@lab-openvswitch:~/network-onboard# ip netns exec ns_blue ip addr add 10.0.0.2/24 dev veth_blue
```

The pattern is: create two empty namespaces (`ns_red`, `ns_blue`); create two veth pairs (each pair has two ends, one with the suffix `_br` for the bridge side); move one end of each veth pair into a namespace; bring the bridge-side ends up; attach both bridge-side ends to `br0` as ports; bring the namespace-side ends up and assign IP addresses. Both namespaces end up on `10.0.0.0/24` with `ns_red` at `.1` and `ns_blue` at `.2`.

The ping from `ns_red` to `ns_blue`:

```text
root@lab-openvswitch:~/network-onboard# ip netns exec ns_red ping -c 3 10.0.0.2 2>&1 | tail -10
PING 10.0.0.2 (10.0.0.2) 56(84) bytes of data.
64 bytes from 10.0.0.2: icmp_seq=1 ttl=64 time=0.368 ms
64 bytes from 10.0.0.2: icmp_seq=2 ttl=64 time=0.042 ms
64 bytes from 10.0.0.2: icmp_seq=3 ttl=64 time=0.040 ms

--- 10.0.0.2 ping statistics ---
3 packets transmitted, 3 received, 0% packet loss, time 2050ms
rtt min/avg/max/mdev = 0.040/0.150/0.368/0.154 ms
```

3 of 3 ICMP echo replies. The first packet's RTT is 0.368 ms; the second and third are 0.042 ms and 0.040 ms. The roughly nine-times-faster second packet is the megaflow-cache effect: the first packet hit the kernel datapath as a miss, was upcalled to userspace, walked through `xlate_normal`, learned both MACs, installed a megaflow rule, and only then was forwarded; the next two packets hit the cached rule and bypassed userspace entirely.

The MAC table after the ping:

```text
root@lab-openvswitch:~/network-onboard# ovs-appctl fdb/show br0 2>&1 | head -10
 port  VLAN  MAC                Age
    1     0  56:a6:87:c1:15:9c    1
    2     0  e2:10:43:fa:d2:bd    1
```

Two entries. The `port` column is the **OpenFlow port number**, the same one shown in `dpif/show` (where `veth_red_br 1/2` reads as OpenFlow port 1, datapath port 2; `veth_blue_br 2/3` reads as OpenFlow port 2, datapath port 3) and the same one returned by `ovs-vsctl --columns=ofport list interface <name>`. So port 1 is `veth_red_br` and port 2 is `veth_blue_br`. Both entries are in VLAN 0 (no VLAN tagging configured yet). Age is 1 second, the time since the most recent frame from each MAC.

The two MAC values, `56:a6:87:c1:15:9c` and `e2:10:43:fa:d2:bd`, are not the MACs of `veth_red_br` and `veth_blue_br`. A veth pair has two ends, each end has its own kernel-assigned MAC, and the bridge-side ends (`veth_red_br`, `veth_blue_br`) never originate traffic in this lab; they only forward frames between the bridge and the namespace-side peer. The source MAC on a frame leaving `ns_red` is the MAC of `veth_red` (the namespace-side end), not the MAC of `veth_red_br` (the bridge-side end). What `fdb/show` records, therefore, is "an Ethernet frame whose source MAC was `56:a6:87:c1:15:9c` arrived on the bridge through OpenFlow port 1". The MAC belongs to `veth_red` inside `ns_red`; the port records where the bridge first saw it.

To prove this binding for `ns_red`, an operator runs `ip netns exec ns_red ip link show veth_red` and reads off the namespace-side MAC; this lab does not run that command, so the proof is by inference (the bridge-side end's MAC is recorded in `mac_in_use` of the matching `Interface` OVSDB row, and is different from the value `fdb/show` reports). A future R2.5 capture will run the namespace-side query explicitly and close the inference loop.

The lesson: **the MAC table is the visible state of `xlate_normal`'s learning behaviour, indexed by source MAC and tagged with the OpenFlow port the frame arrived on.** OVS does not learn the MAC of its own port; it learns the MAC of whatever device sent the frame across the wire to it. Each call to `xlate_normal` for an unknown source MAC adds an entry; subsequent calls update the entry's expiry. The R2 expert challenge for this exercise is to read the trace produced by `ofproto/trace` for the same ARP-then-ICMP sequence and count the `mac_learning_insert` and `mac_learning_lookup` calls implied.

## F.6, Internal port versus tap port (with an `ip` syntax aside)

Two more ports added to `br0`, one of each type:

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br0 int0 -- set interface int0 type=internal
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br0 tap0 -- set interface tap0 type=tap
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=name,type,ofport,mac_in_use list interface int0 tap0
name                : int0
type                : internal
ofport              : 3
mac_in_use          : "b2:fb:2c:fe:12:ac"

name                : tap0
type                : tap
ofport              : 4
mac_in_use          : "5a:4a:dd:48:3b:3d"
```

Two new Interface rows, one with `type=internal` (an OVS-managed kernel-visible netdev that the host can read packets from), one with `type=tap` (a TUN/TAP device, similar in spirit but managed via the kernel's TUN/TAP driver rather than the OVS-internal driver). Both auto-allocated OpenFlow port numbers (3 and 4) and auto-generated MACs.

Then the `ip -br link show` query, which produces an unexpected error:

```text
root@lab-openvswitch:~/network-onboard# ip -br link show int0 tap0 2>&1 | head -5
Error: either "dev" is duplicate, or "tap0" is a garbage.
```

The error is the second pedagogical surprise of R2. The `ip link show` command does **not** accept multiple netdev names as positional arguments. The right form is `ip -br link show int0` then `ip -br link show tap0`, or the full `ip -br link show` (no name, dump all). The error message is from iproute2's option parser, which treats the second positional after `show` as a stray token and rejects it. The capture preserves the error verbatim because correcting it after the fact loses the lesson.

The implication for the prose around port types: a learner cannot batch `ip` queries across types the way they can batch `ovs-vsctl --columns` queries. The two tools have different argument grammars.

## F.7, Patch port pair between two new bridges

Two new bridges, each with one patch port pointing at the other:

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-br br-int
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-br br-tun
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br-int patch-int -- set interface patch-int type=patch options:peer=patch-tun
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br-tun patch-tun -- set interface patch-tun type=patch options:peer=patch-int
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=name,type,options list interface patch-int patch-tun
name                : patch-int
type                : patch
options             : {peer=patch-tun}

name                : patch-tun
type                : patch
options             : {peer=patch-int}
```

The two `Interface` rows record `type=patch` and the peer relationship in `options`. Patch ports differ from every other OVS port in two ways: they are not netdevs visible to the kernel (no `ip link show patch-int` ever sees them), and they are bidirectional pairs that must agree on each other's identity.

The dpif/show output gives the bigger picture:

```text
root@lab-openvswitch:~/network-onboard# ovs-appctl dpif/show 2>&1 | head -30
system@ovs-system: hit:14 missed:11
  br-int:
    br-int 65534/6: (internal)
    patch-int 1/none: (patch: peer=patch-tun)
  br-tun:
    br-tun 65534/7: (internal)
    patch-tun 1/none: (patch: peer=patch-int)
  br0:
    br0 65534/1: (internal)
    int0 3/4: (internal)
    tap0 4/5: (tap)
    veth_blue_br 2/3: (system)
    veth_red_br 1/2: (system)
```

Three bridges (`br-int`, `br-tun`, `br0`), all on the kernel datapath `system@ovs-system`. The `hit:14 missed:11` counters are the cumulative datapath statistics since the daemon started: 14 packets that hit a kernel-cached megaflow, 11 that missed and were upcalled.

For each bridge, the dpif/show output lists every port with three pieces of information: the OpenFlow port number / the kernel datapath port number, and the port type. `br-int 65534/6` means OpenFlow `OFPP_LOCAL` (65534) and kernel datapath port 6. **The two numbering systems are independent**: OpenFlow's `OFPP_LOCAL` is always 65534 by spec, while the kernel datapath assigns sequential numbers starting at 0 (with port 0 reserved for the local port of the *first* bridge added to the datapath; subsequent bridges get the next available number). For `br-int` the local port is kernel port 6, for `br-tun` it is 7, and for `br0` (created first) it is 1. The mismatch between the two numbering systems is one of the more confusing aspects of OVS for a new operator and is worth pausing on.

The `patch-int 1/none` entry shows the OpenFlow port number 1 and the kernel datapath port number `none`. Patch ports do not exist in the kernel; they are pure userspace abstractions that `ofproto-dpif-xlate.c` resolves at translation time. A trace through `br-int` that outputs to `patch-int` will continue tracing in `br-tun` because the translator sees the peer relationship and recurses.

## F.8, Set MTU on a port

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl set interface int0 mtu_request=9000
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=name,mtu,mtu_request list interface int0
name                : int0
mtu                 : 9000
mtu_request         : 9000
root@lab-openvswitch:~/network-onboard# ip -br link show int0
int0             DOWN           b2:fb:2c:fe:12:ac <BROADCAST,MULTICAST> 
```

Both columns now read 9000. The kernel netdev `int0` shows the same MAC as before; the kernel-visible MTU is implicit in the `<BROADCAST,MULTICAST>` flags display (no explicit MTU column in `ip -br` output, but the change has propagated, which `ip link show int0` would confirm with `mtu 9000`).

The propagation chain: `ovs-vsctl set` writes to OVSDB; the next IDL update fires `bridge_reconfigure` in `vswitchd/bridge.c`; that function calls `iface_configure`, which calls `netdev_set_mtu`, which for an internal port calls the kernel netdev's MTU setter via Netlink. The whole chain takes single-digit milliseconds in practice; an operator never sees a delay.

## F.9, Set the bridge MAC

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl set bridge br0 other_config:hwaddr="02:0a:0b:0c:0d:0e"
root@lab-openvswitch:~/network-onboard# sleep 1

br0              DOWN           02:0a:0b:0c:0d:0e <BROADCAST,MULTICAST>
```

The `other_config:hwaddr` override worked. The kernel-visible MAC on `br0` is now the configured `02:0a:0b:0c:0d:0e`, replacing the auto-generated `a6:4c:0e:42:2f:44` from earlier in the lab. ## F.11, VLAN access port with both ports in VLAN 100

Two access ports in the same VLAN: ICMP works.

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl set port veth_red_br tag=100

```

2 of 2 echo replies. Both ports are access ports for VLAN 100. Traffic between them flows; the operator on the namespace side never sees a tag because access ports strip on egress and add on ingress.

## F.12, Trunk port plus access port

Switch blue from access (`tag`) to trunk (`trunks=100,200`); the ping still fails because the namespace-side IP is on no VLAN:

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl clear port veth_blue_br tag
root@lab-openvswitch:~/network-onboard# ovs-vsctl set port veth_blue_br trunks=100,200
root@lab-openvswitch:~/network-onboard# ip netns exec ns_red ping -c 2 -W 1 10.0.0.2 2>&1 | tail -6 || echo '(no reply, expected: blue is trunk)'
PING 10.0.0.2 (10.0.0.2) 56(84) bytes of data.

--- 10.0.0.2 ping statistics ---
2 packets transmitted, 0 received, 100% packet loss, time 1004ms
```

The reason the ping still fails is interesting and worth explaining. With `veth_red_br` as a tag-100 access port and `veth_blue_br` as a trunk that carries 100 and 200, `ns_red`'s ICMP request enters with no tag, gets tagged 100 by the access port on ingress, and reaches the trunk port; the trunk port forwards the tagged frame out of `veth_blue_br` into `ns_blue`. But `ns_blue`'s `veth_blue` (the namespace-side interface) is configured with a plain IP at `10.0.0.2/24` and no 802.1Q sub-interface; the kernel's networking stack drops the 802.1Q-tagged frame because there is no VLAN-aware listener on this device. The frame is technically delivered but the Linux kernel discards it before it reaches the IP layer.

The lab restores the symmetric access-port form for the rest of the session:

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl clear port veth_blue_br trunks
root@lab-openvswitch:~/network-onboard# ovs-vsctl set port veth_blue_br tag=100
```

The expert challenge for this exercise is to add an 802.1Q sub-interface inside `ns_blue` (`ip link add link veth_blue name veth_blue.100 type vlan id 100`) and observe whether the ping succeeds with the trunk-plus-tagged-subinterface configuration.

## F.13, MAC aging, with a wrinkle

Set the aging time low and observe the table after a single ping plus a 12-second sleep:

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl set bridge br0 other_config:mac-aging-time=10
root@lab-openvswitch:~/network-onboard# ip netns exec ns_red ping -c 1 10.0.0.2 2>&1 | tail -3
--- 10.0.0.2 ping statistics ---
1 packets transmitted, 1 received, 0% packet loss, time 0ms
rtt min/avg/max/mdev = 0.232/0.232/0.232/0.000 ms
root@lab-openvswitch:~/network-onboard# ovs-appctl fdb/show br0 2>&1 | head -5
 port  VLAN  MAC                Age
    2   100  e2:10:43:fa:d2:bd    1
    1   100  56:a6:87:c1:15:9c    0
root@lab-openvswitch:~/network-onboard# sleep 12
root@lab-openvswitch:~/network-onboard# ovs-appctl fdb/show br0 2>&1 | head -5
 port  VLAN  MAC                Age
    1   100  56:a6:87:c1:15:9c   14
```

Immediately after the ping, the MAC table has two entries, both VLAN 100 (the access tag from F.11), with ages 1 and 0 seconds. After a 12-second sleep, the table has only one entry, with age 14 seconds.

The result is more nuanced than the textbook "every entry past N seconds disappears". The aging-time is the **maximum age before the entry is eligible for eviction**; the eviction itself happens during periodic sweeps inside `mac_learning_run`. The sweep is not synchronous with the timer; an entry can survive briefly past its aging-time. In this capture, the port-2 entry (the source MAC of `veth_blue`) was evicted by the sweep that ran during the sleep window, while the port-1 entry (the source MAC of `veth_red`, which is also the local end of the most-recent ICMP path because `ns_red` initiated the ping) survived and is now showing age 14, four seconds past the configured timeout.

The lesson: MAC aging is **best effort**, not a hard guarantee. An operator who needs deterministic eviction (for example, in a security-sensitive multi-tenant deployment) cannot rely on the timer alone; the sweep period plus the timer give the actual upper bound. Reading `lib/mac-learning.c:mac_learning_run` shows the sweep loop and the eviction policy.

## F.15, `--may-exist` for idempotent scripting

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-br br-test
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-br br-test 2>&1 || echo '(expected error, bridge exists)'
ovs-vsctl: cannot create a bridge named br-test because a bridge named br-test already exists
(expected error, bridge exists)
root@lab-openvswitch:~/network-onboard# ovs-vsctl --may-exist add-br br-test && echo '(no-op accepted)'
(no-op accepted)
root@lab-openvswitch:~/network-onboard# ovs-vsctl del-br br-test
```

Three forms, three results. Bare `add-br br-test` succeeds (bridge created). Repeat `add-br br-test` fails with the explicit error message. `--may-exist add-br br-test` accepts as a no-op. The cleanup `del-br br-test` returns to baseline.

The `--may-exist` flag is the right pattern for any script that needs to "ensure this bridge exists" without checking first. Reading the man page for `ovs-vsctl(8)` shows which sub-commands accept the flag (`add-br`, `add-port`, `add-bond`, `add-bond-iface`, and a few others); the flag is **not** universal, and `del-br` for example has no `--may-exist` equivalent because `del-br` is already a hard error on a non-existent bridge.

## F.16, `ofproto/trace` with full versus partial packet header

The full packet form gives a useful trace:

```text
bridge("br0")
-------------
 0. priority 0
    NORMAL
     -> no learned MAC for destination, flooding

Final flow: unchanged
Megaflow: recirc_id=0,eth,ip,in_port=LOCAL,dl_src=02:00:00:00:00:aa,dl_dst=02:00:00:00:00:bb,nw_frag=no

```

Now `Datapath actions: 4,5`. The flood from `LOCAL` reaches kernel datapath ports 4 and 5, which from the F.7 dpif/show output are `tap0` and `int0` respectively. (The veth ports `veth_red_br`, `veth_blue_br` are not in the flood set because their `tag=100` from F.11 means the trace's untagged packet from `LOCAL` does not match their VLAN.) The trace confirms the flooding decision and reveals the exact set of egress ports.

The partial-packet form gives a less useful but still informative trace:

```text
root@lab-openvswitch:~/network-onboard# ovs-appctl ofproto/trace br0 'in_port=LOCAL' 2>&1 | tail -15
Flow: in_port=LOCAL,vlan_tci=0x0000,dl_src=00:00:00:00:00:00,dl_dst=00:00:00:00:00:00,dl_type=0x0000

bridge("br0")
-------------
 0. priority 0
    NORMAL
     -> no learned MAC for destination, flooding

Final flow: unchanged
Megaflow: recirc_id=0,eth,in_port=LOCAL,dl_src=00:00:00:00:00:00,dl_dst=00:00:00:00:00:00,dl_type=0x0000

```

With every field unspecified, the trace fills in zeroes (`dl_src=00:00:00:00:00:00`, `dl_dst=00:00:00:00:00:00`, `dl_type=0x0000`). The flood decision is the same (still `Datapath actions: 4,5`) because the all-zeroes destination is treated as multicast by `eth_addr_is_multicast`, which triggers `xlate_normal_flood`. The lesson is not to read too much into a partial-packet trace; the engine fills in zeros and the result may not match real traffic.

The expert challenge: explain why the all-zeroes destination triggers `xlate_normal_flood`. (Hint: the multicast bit is the lowest bit of the first byte; `0x00` does not have the bit set, so `eth_addr_is_multicast` returns false; but the all-zeroes address is treated as broadcast in this context for safety. Read the function body in `ofproto/ofproto-dpif-xlate.c` to confirm.)

## F.17, OVSDB JSON-RPC manager

The `set-manager` command tells `ovsdb-server` to listen on the given protocol/port for incoming JSON-RPC connections:

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl set-manager ptcp:6640

```

The `ovsdb-server` process (pid 64813) is now listening on TCP port 6640, owned by file descriptor 24. A client speaking the OVSDB JSON-RPC protocol (RFC 7047) can connect to this port and read or write rows.

A client query against this listener:

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl --db=tcp:127.0.0.1:6640 list bridge br0 2>&1 | head -10
_uuid               : 420e4ca7-d757-442f-907e-706c3cabf354

controller          : []

datapath_version    : "<unknown>"

fail_mode           : standalone
flood_vlans         : []
flow_tables         : {}
```

The `--db=tcp:127.0.0.1:6640` flag overrides the default Unix-socket location and instead opens a TCP connection to the listener. The output is the first ten columns of the `Bridge` row for `br0`. Notable: `datapath_id` is `0000020a0b0c0d0e`, derived from the `other_config:hwaddr` set in F.9 (`02:0a:0b:0c:0d:0e`; the leading `0000` pads the 48-bit MAC to a 64-bit datapath ID per the OpenFlow convention). `fail_mode` is `standalone`, set explicitly in F.3. `controller` is empty (no controller installed), and `flow_tables` is the empty dictionary.

After this the TCP listener is closed and `ovsdb-server` only accepts connections on the default Unix socket. The expert challenge is to capture the TCP wire bytes during the `--db=tcp:...` query with `tcpdump -i lo -X port 6640` and decode the JSON-RPC `transact` request and reply by hand, comparing against RFC 7047.

R2 confirmed every E2 Â§F prediction at the operational level, with three findings the dossier did not predict and the prose surfaced:

1. **The `Open_vSwitch` table has no `name` column.** F.2's first attempted query failed verbatim because the table is a singleton and singletons are not named. Lesson: read the schema before assuming column names.

- The `Open_vSwitch` row UUID for this host is `5f8e9a74-1559-4375-9b90-a7474ef58029` (carried over from R1.B).

## Context: SDN does not need to touch the hardware

Part 5.0 analysed API-based SDN, which keeps silicon as is and adds a YANG and NETCONF layer. That is a conservative approach that protects the vendor's investment. This part goes in the opposite direction: **fully replace the network edge** by moving it from the physical top-of-rack (ToR) switch into the hypervisor software, through Open vSwitch and a controller cluster. The underlay (the physical network) keeps running simple IP routing (OSPF, BGP) and does not need to know about SDN.

This is the **hypervisor-edge overlay SDN** model, started by Nicira Networks in 2011 with the NVP (Network Virtualization Platform) product. The key point is that NVP does not replace the hardware switch. It only replaces the **network edge**, moving it from the ToR to the hypervisor virtual NIC. The hardware switches in the fabric only need to route IP between hypervisors. The overlay protocol (initially STT, later VXLAN and Geneve) handles the L2 abstraction and the per-tenant isolation.

This model succeeded in data centre virtualisation immediately. VMware acquired it in 2012 and rebranded it as NSX. The original team later left VMware for Red Hat and started the OVN project in 2015, continuing the open-source line. Understanding the NVP to NSX to OVN lineage is the key to understanding the OVN architecture in any 2026 data centre deployment.

## 5.1.1 Nicira NVP: the commercial product in 2011

### Founding and key people

Nicira Networks was founded in August 2007 in Palo Alto by three people:

- **Martin Casado**: PhD from Stanford, lead author of the Ethane paper at SIGCOMM 2007 (Part 2.4) and of the original OpenFlow paper at CCR April 2008 (Part 3.0).
- **Nick McKeown**: Stanford professor, co-author of the OpenFlow paper, and faculty advisor to Casado.
- **Scott Shenker**: UC Berkeley professor, co-author of Ethane, and co-lead of the Clean Slate Program.

Initial funding: Andreessen Horowitz, Lightspeed Venture Partners, and New Enterprise Associates. The company was in stealth mode from 2007 to 2011 and published its first products in 2011.

### Product launch in 2011

NVP 1.0 launched in October 2011. Components:

**Nicira Controller Cluster**: a 3-node to 5-node Java application that stored the logical topology in a distributed database. It exposed a REST API for provisioning (create logical switch, attach VM, install ACL). Internal coordination went through a ZooKeeper ensemble.

**Open vSwitch on the hypervisor**: an OVS instance ran on each host (Linux KVM, Xen, ESXi from 2013 onward). OVS was a Nicira open-source contribution from 2008 to 2011, and was mainlined into Linux kernel 3.3 in February 2012.

**OpenFlow 1.0 plus NXM extensions**: the controller programmed OVS through the OpenFlow protocol, using NXM (Nicira Extensible Match) for features beyond the 12-tuple. The NVP controller generated flow rules based on the logical topology.

**STT (Stateless Transport Tunneling)**: an overlay encapsulation invented by Nicira. STT emulated a TCP header so that NIC checksum offload could work for overlay packets. It addressed the VXLAN performance pain point on NICs without native tunnel offload. STT later lost to VXLAN and Geneve when NIC offload matured, but in 2011 to 2014 it was a competitive advantage.

### Customer deployment in production

NVP early customers (2011 to 2012) included eBay, AT&T, Rackspace, and NTT. The use case was multi-tenant private cloud and public cloud provider, with a software-defined overlay for tenant isolation, without rewiring the physical network.

The eBay public presentation in 2012 described an NVP deployment with about 10,000 virtual machines, achieving tenant isolation and micro-segmentation that a physical-VLAN-based approach could not reach at scale.

## 5.1.2 Hypervisor-edge architecture: OVS plus a centralised controller

### The "edge is the hypervisor" philosophy

Traditional data centre network topology:

```
[Core spine] <-> [Leaf ToR] <-> [Rack 1: 16 hypervisors with VMs]
                 [Leaf ToR] <-> [Rack 2: 16 hypervisors with VMs]
```

By convention, "network edge" is the ToR switch, on the port facing the hypervisor NIC. Logical concepts (VLAN, ACL, QoS) are implemented on the ToR.

Nicira changed this. The network edge is the **virtual NIC of the VM**, handled by OVS inside the hypervisor. The conceptual flow:

```
VM <-> OVS (in hypervisor) <-> hypervisor physical NIC <-> ToR (L3 only) <-> fabric
```

OVS handles VLAN tagging, ACL, NAT, and tunnel encapsulation. The ToR does not know about the tenant VLAN structure. The ToR only routes the IP packets of the tunnel (STT or VXLAN) between hypervisors.

### Architectural consequences

**1. Underlay simplification**: hardware switches do not need to be SDN-aware. They run standard routing protocols (OSPF, BGP, ECMP). The operator can deploy any vendor (Cisco, Juniper, Arista, whitebox) without OpenFlow support.

**2. Multi-tenant isolation through overlay**: each tenant has its own logical topology. OVS encapsulates traffic with a tunnel header that contains the tenant ID (STT tunnel ID, VXLAN VNI). The fabric routes based on the outer IP, and the overlay content is transparent to the physical network.

**3. Controller scale**: a centralised control plane manages N hypervisors instead of M ToR switches (N is much greater than M in a small data centre, but N is less than M in a hyperscale deployment with about 1,000 racks). The controller workload scales with compute density, not with physical switch count.

**4. Deployment flexibility**: adding a tenant means programming OVS through the controller, with no change to ToR configuration. Deployment time drops from weeks (hardware VLAN provisioning) to minutes (software configuration).

### Trade-offs

**Performance**: overhead from the overlay encapsulation (STT adds about 54 bytes of header, VXLAN about 50 bytes, Geneve about 58 bytes). Packet throughput goes down because the MTU is smaller. Hardware NIC offload (TSO, checksum) needs to work with the tunnel. NICs from 2011 to 2013 did not fully support this. NVP used STT to emulate TCP and benefit from existing offload.

**Debug complexity**: troubleshooting a tunnel requires inspecting two layers (the outer fabric layer and the inner tenant layer). Tools such as `ovs-appctl ofproto/trace` (Part 9.4) and `tcpdump` on the overlay interface are essential.

**Fabric requirements**: the fabric must be pure IP (no bridging across racks). The design must be L3-to-the-ToR (a spine-leaf with routing from the edge). A traditional L2 fabric (Cisco STP-based) is not compatible.

> **Common misconception:** "The hypervisor overlay model is not usable because the performance cost is too high."
>
> **Reality:** the overlay performance cost (STT, VXLAN, or Geneve) is only about 5 to 10 percent of throughput on a NIC without tunnel offload, and drops to about 2 to 3 percent on a modern NIC (Mellanox ConnectX-5 and later, Intel 800 series) with hardware VXLAN TSO. Compared with the benefits (multi-tenant isolation, fast provisioning, programmable policy), the trade-off is worth it in data centre virtualisation. Hyperscalers Azure, Google Cloud, and AWS VPC all use overlay architectures. "Cost too high" is a misconception that comes from 2011 benchmarks running on old hardware.

## 5.1.3 VMware acquisition on 23 July 2012 (1.26 billion United States dollars)

### Deal context

In July 2012, VMware announced the acquisition of Nicira Networks for 1.26 billion United States dollars, paid entirely in cash. This was the largest SDN deal in history at that point. It held the record until Broadcom acquired Brocade for 5.5 billion United States dollars in 2017 (a broader enterprise networking scope) and Cisco acquired Splunk for 28 billion United States dollars in 2023 (data analytics, not pure networking).

The official VMware press release was on 23 July 2012, at `https://www.vmware.com/company/news/releases/vmw-newsfeed/VMware-Completes-Acquisition-of-Nicira/1830683`, with this statement:

> *"With Nicira, VMware will offer a complete set of infrastructure products for the software-defined datacenter, putting us in a unique position to deliver on the vision of the modern software-defined datacenter."*

### VMware strategy

Context in 2012: VMware dominated compute virtualisation with vSphere ESXi, and revenue came from hypervisor licences. The Cisco approach to SDN (still in research, before ACI) was a threat to VMware, because Cisco could build SDN tightly coupled with Cisco UCS compute and bypass VMware.

The Nicira acquisition let VMware:

- Own the SDN narrative before Cisco launched ACI (December 2013).
- Integrate the NVP technology into the vSphere roadmap, creating an umbrella portfolio called "software-defined datacenter" (SDDC).
- Commercialise SDN through the existing VMware enterprise sales channel (more than 10,000 customers).

### Impact on the Nicira team

Core engineers moved to VMware:

- Martin Casado: Senior VP and General Manager of the Networking and Security Business Unit.
- Justin Pettit, Ben Pfaff, Teemu Koponen: engineering team leaders.
- Scott Shenker: not full time, kept an advisor role.

The Nicira brand was retired, and the product was renamed to **NSX** (Network and Security eXtensions). NSX 1.0 launched in March 2013 at the VMware Partner Exchange event.

### Evolution from 2013 to 2015

NSX 1.0 (2013) was NVP renamed and integrated with vSphere.
NSX-V (2014) was NSX for vSphere, tightly coupled with ESXi.
NSX-T (2016) was NSX-T Data Center, multi-hypervisor, expanding beyond vSphere.

Cross-reference 5.1.4 analyses NSX-V versus NSX-T architectural differences in detail.

## 5.1.4 NSX-V (vSphere) versus NSX-T (multi-hypervisor)

### NSX-V (2013 to 2021)

**Target platform**: VMware vSphere ESXi only. Tightly integrated with the ESXi kernel module.

**Components**:

- **NSX Manager**: management plane, with a REST API for vCenter integration.
- **NSX Controller cluster**: control plane, a 3-node cluster.
- **VTEPs (VXLAN Tunnel EndPoints)**: on each ESXi host, using VMware's own VXLAN implementation (not OVS).
- **Edge Services Gateway (ESG)**: a virtual appliance for north-south routing, NAT, firewall, and VPN.
- **Distributed Firewall**: a hypervisor-embedded stateful firewall, with per-vNIC granularity.

**Data plane**: the ESXi kernel VXLAN stack (not OVS). This is an independent code base from the NVP era, optimised for ESXi.

**End of life**: NSX-V end-of-life was announced in 2022, with extended support until January 2024. Customers migrate to NSX-T.

### NSX-T (2016 to present)

**Target platform**: multi-hypervisor:

- VMware vSphere ESXi.
- KVM (Linux).
- Bare-metal servers (run the NSX-T agent directly).
- Containers (integration with Kubernetes through NCP, the NSX Container Plugin).

**Components**:

- **NSX-T Manager**: unified management and control plane (consolidated from the NSX-V split).
- **NSX-T Edge nodes**: virtual appliances or bare-metal, scaled horizontally.
- **Transport Nodes**: ESXi, KVM, or bare-metal nodes running the NSX-T agent.

**Data plane**: on ESXi it uses VMware proprietary VTEP. On KVM it uses OVS. On bare-metal it uses NSX-T DPU (a ConnectX-based SmartNIC, in later releases).

**Geneve replaces VXLAN**: NSX-T moves to Geneve encapsulation (RFC 8926) for TLV option flexibility. It keeps backward compatibility with VXLAN overlay for migration.

### NSX-V versus NSX-T comparison

| Aspect | NSX-V | NSX-T |
|--------|-------|-------|
| Hypervisor | vSphere only | vSphere plus KVM plus bare-metal |
| Data plane | ESXi kernel VXLAN | Mixed: ESXi VTEP plus KVM OVS plus DPU |
| Overlay | VXLAN primary | Geneve primary (with VXLAN backward compatibility) |
| Management | NSX Manager plus a separate Controller | Unified NSX-T Manager |
| Edge | ESG virtual appliance | Edge cluster (scale-out) |
| Container integration | Limited | NCP first-class integration with Kubernetes |
| Status in 2026 | End of life, migration target | Active development, the VMware flagship |

> **Common misconception:** "NSX is really Nicira NVP under a new brand, the same technology."
>
> **Reality:** NSX-V (2013) inherited the **concept** from NVP (network edge in the hypervisor, controller cluster, tunnel overlay), but the code base was rewritten substantially to integrate with ESXi. NSX-V does not use OVS. The VTEP in the ESXi kernel is VMware proprietary code. Only NSX-T (2016) returns to OVS (on the KVM branch). The "NVP to NSX" lineage is about ideas, not about identical code.

## 5.1.5 The Juniper Contrail lineage: similar, but with a BGP EVPN control plane

### Contrail architecture (simplified)

Juniper Contrail (acquisition of Contrail Systems in December 2012, product launch in 2013) implements a hypervisor-edge model similar to NVP and NSX. The main difference is in the **control plane**:

**NVP and NSX**: a Nicira controller (Java) with ZooKeeper coordination, and southbound OpenFlow plus NXM.

**Contrail**: a control plane layer that uses **BGP EVPN** (RFC 7432) for MAC and IP route signalling, and **XMPP** for the controller-to-vRouter southbound channel.

### BGP EVPN path

**Data plane**: vRouter (a kernel module, or DPDK-based) on each compute node.
**Control plane**: the Controller peers with vRouter through XMPP and peers with the external WAN through MP-BGP (multi-protocol BGP).

Tenant L2 and L3 routes are distributed through BGP EVPN:

- **Type 2 (MAC and IP advertisement)**: the tenant VM's MAC and IP.
- **Type 3 (Inclusive Multicast)**: BUM (Broadcast, Unknown, Multicast) suppression.
- **Type 5 (IP Prefix Route)**: L3 routes for a subnet.

### Contrail versus NVP and NSX trade-off

**Contrail advantages**:

- WAN integration is natural through BGP. Service providers already have a BGP infrastructure, so integration is seamless.
- Data centre to data centre connectivity uses standard BGP EVPN. It interoperates with Cisco, Juniper, and Arista routers that support EVPN.
- MP-BGP routing has been proven to scale beyond 100,000 routes.

**Contrail disadvantages**:

- Complexity. BGP EVPN requires BGP expertise, and the learning curve is steep for a team of administrators with only vSphere experience.
- Debugging is harder. State is distributed across multiple BGP sessions, not a centralised database.

### Contrail evolution

- **2014**: Juniper open-sourced it as **OpenContrail**.
- **2018**: transferred to the Linux Foundation under the name **Tungsten Fabric** (`tungsten.io`); the commercial version remains Juniper Contrail.
- **Current state in 2026**: the Tungsten Fabric community is still active. Juniper focuses Contrail on SD-WAN and enterprise (Part 7.3).

> **Common misconception:** "Contrail is just the open-source version of NSX."
>
> **Reality:** Contrail has a notably different architecture from NVP and NSX. The control plane uses BGP EVPN instead of the controller-centric model of NVP and NSX. The data plane uses vRouter (developed in-house) instead of OVS (which NVP and NSX use on the KVM branch). The audience also differs. Contrail mainly targets WAN service providers and multi-data-centre deployments, while NVP and NSX mainly target enterprise data centre virtualisation. Treating them as equivalent is an oversimplification.

## 5.1.6 Legacy: OVN is the spiritual successor to NVP

### Team move in 2015

Around 2014 to 2015, the key engineers from the Nicira team left VMware for Red Hat:

- **Justin Pettit**: senior OVS engineer, co-author of NVP.
- **Ben Pfaff**: chief architect of OVS, core author of the NSDI 2015 paper.
- **Teemu Koponen**: architect of the Nicira controller.

Their intent was to rebuild an open-source alternative to NSX, not tightly coupled with VMware. The result was the **OVN (Open Virtual Network)** project.

### OVN announcement on 13 January 2015

Official announcement through a **Network Heresy blog post** on 13 January 2015 by Justin Pettit, Ben Pfaff, Chris Wright (Red Hat CTO), and Madhu Venugopal (Docker):

> URL: `https://networkheresy.wordpress.com/2015/01/13/ovn-bringing-native-virtual-networking-to-ovs/`

The post announced OVN as an open-source project for virtual networking over OVS, designed to be portable across multiple orchestrators (container runtime plus VM orchestrator). Key technical differences from NVP:

**NVP**: a controller cluster with ZooKeeper plus OpenFlow.
**OVN**: an **NBDB (Northbound Database) plus SBDB (Southbound Database)** for the control plane, using the OVSDB protocol (RFC 7047, Part 10.0). `ovn-northd` translates NB intent into SB logical flow.

### Architectural philosophy

OVN keeps the NVP philosophy:

- Hypervisor edge with OVS.
- Centralised control plane.
- Overlay tunnels (Geneve by default, VXLAN optional).
- Multi-tenant via the `Logical_Switch` abstraction.

What changed:

- Replace the proprietary Nicira controller with a distributed database based on OVSDB (NBDB and SBDB).
- Replace controller-driven OpenFlow with OVSDB SB intent. `ovn-controller` on each node translates that intent into local OpenFlow.
- The entire stack is open source (OVS plus OVN in the same monorepo).

### State in 2026

OVN is the de facto hypervisor overlay SDN in multiple deployment contexts: Kubernetes CNI (OVN-Kubernetes), bare-metal OVN standalone, and as a portable driver for several virtualisation orchestrators. Production at:

- **Red Hat OpenShift Container Platform**: OVN-Kubernetes is the default CNI from 4.12 (2023).
- **Microsoft Azure**: an internal overlay framework that uses OVN concepts.
- **Carrier NFV**: NTT, Orange, and Deutsche Telekom infrastructure deployments.

OVN is maintained upstream at `https://github.com/ovn-org/ovn`, separate from OVS (`https://github.com/openvswitch/ovs`) after the 2018 split.

> **Key point:** the hypervisor overlay model (NVP, then NSX, then OVN) is the dominant SDN model in data centre virtualisation in 2026. Understanding the "do not touch the underlay" philosophy and tunnel encapsulation is the foundation for design decisions when deploying OVN. When troubleshooting OVN, remembering the NVP history helps you understand why this design exists (OpenFlow on hardware did not scale well; software edge with controller cluster coordination was needed).

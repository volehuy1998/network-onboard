## Why this transcript exists

R0 records the green-field state of `lab-openvswitch` before any Open vSwitch package, daemon, or kernel module is installed. Every later sprint of plan v3.13 will compare its observations against this baseline. If a future sprint reports `cannot access /etc/openvswitch`, R0 is what that comparison points to. If a future sprint reports a higher RAM footprint, R0 fixes the starting line. The transcript is therefore not interesting in itself; it is interesting as the zero point from which every measurement is taken.

The host runs Ubuntu 22.04.5 LTS on kernel `5.15.0-173-generic` inside VMware Workstation. Two NICs: `ens33` is the management plane on `192.168.1.0/24`, `ens36` is the lab data plane on `10.99.99.0/24`. The kernel ships its own in-tree `openvswitch.ko` module at `/lib/modules/5.15.0-173-generic/kernel/net/openvswitch/openvswitch.ko`, but the module is not loaded and no userland Open vSwitch package is installed. This is the starting point.

## Capture method

The transcript was captured by a single invocation of `script -f -q -t/tmp/v3.13-R0.timing /tmp/v3.13-R0.script` running inside `tmux new-session -d -s v3_13-r0 -x 200 -y 50`. The terminal width was 200 columns and the height was 50 rows, chosen to keep `lscpu` and `systemctl` output on single physical lines and avoid line wrapping inside the typescript. Commands were sent into the `tmux` pane via `tmux send-keys`; the typescript file recorded every byte the inner shell produced, including the bracketed-paste-mode escape sequences that bash readline emits before and after each prompt.

The two artefact files are present alongside this rendering:

- `v3.13-R0-baseline.typescript`: the raw byte stream produced by `script -f`. Every shell prompt, command echo, output line, and terminal escape sequence is preserved.
- `v3.13-R0-baseline.timing`: the per-byte timing log produced by `script -t`. A future sprint can replay the session in real time with `scriptreplay`.

The pre-commit hook normalises the typescript before comparing it against the fenced blocks below. Normalisation strips ANSI Control Sequence Introducer escapes (`\x1b[...`), Operating System Command escapes (`\x1b]0;<title>\x07`, used by bash to set the terminal title), and the `Script started` and `Script done` banner lines. After normalisation, every line of every fenced block on this page must appear as a substring of some normalised typescript line, in order, or the commit is rejected.

## Open vSwitch kernel module

The `openvswitch` kernel module is not loaded:

```text
root@lab-openvswitch:~# lsmod | grep -E "^(openvswitch|nf_conntrack|nf_nat)" || echo NONE
NONE
```

The fallback `|| echo NONE` is necessary because `grep` returns exit 1 when nothing matches; without the fallback, the absence of output would be ambiguous under Rule 18 step 4 (empty output preserved as an empty line).

The Ubuntu kernel ships an in-tree `openvswitch.ko` even though the module is not loaded:

```text
root@lab-openvswitch:~# find /lib/modules/$(uname -r) -name openvswitch.ko\* 2>&1
/lib/modules/5.15.0-173-generic/kernel/net/openvswitch/openvswitch.ko
```

This means R1 path A and path B can both rely on the kernel-side module being present and signed by the Ubuntu build infrastructure. R1 path C (build from git checkout) must build its own out-of-tree module and decide whether to install it; the trade-off is documented in plan §S1 and will be exercised in that sprint.

Six netlink generic aliases are registered (`ovs_ct_limit`, `ovs_meter`, `ovs_packet`, `ovs_flow`, `ovs_vport`, `ovs_datapath`). Each alias is the netlink family that userspace tools talk to over `/dev/netlink`. R4 datapath internals will use `ovs-dpctl` and the `genetlink` API to query these families directly.

The module's hard dependencies are `nf_conntrack`, `nf_nat`, `nf_conncount`, `libcrc32c`, `nf_defrag_ipv6`, and `nsh`. These six modules are loaded automatically when `openvswitch` is loaded. R6 connection-tracking lab will exercise the `nf_conntrack` integration; the dependency declaration is recorded here so R6 can refer back to the concrete kernel ABI.

`intree: Y` confirms the module is part of the upstream Linux kernel tree, not an out-of-tree backport. The `srcversion: 573DE95255C6140CD2046E8` is the source-checksum identifier used by `modprobe` to verify an in-memory module against an on-disk object; it is an opaque value but is preserved verbatim because a future incident might match a kernel oops trace against this srcversion.

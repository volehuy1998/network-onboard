## 9.15.1 Vai trò classifier trong stack OVS

Classifier là cấu trúc dữ liệu trung tâm của **ovs-vswitchd**, lưu trữ toàn bộ flow rule OpenFlow và cho phép tra cứu rule tương ứng với một packet header bất kỳ. Khi một packet đi vào OVS, nếu không match megaflow cache trong kernel, packet được "upcall" tới ovs-vswitchd. Tại đây, classifier được truy vấn với packet header → trả về rule match có priority cao nhất → rule này được compile thành megaflow và install ngược về kernel datapath cho các packet tương tự sau.

Bài toán classifier giải quyết: cho flow table có N rule với match field phức tạp (128-bit IPv6 address, 12-tuple L2-L4), nhanh chóng xác định rule match với packet P. Đơn giản nhất là duyệt tuyến tính qua N rule — O(N) — không khả thi với N = 10K rule + packet rate 1 Mpps. Cần thuật toán O(log N) hoặc O(1) amortized.

Các cách tiếp cận tra cứu packet phổ biến:
- **Duyệt tuyến tính:** O(N). Đơn giản, chậm.
- **Cây nhị phân (B-tree):** O(log N). Tốt cho tra cứu trên một cột nhưng phức tạp với multi-field match có wildcard.
- **Hash map:** O(1). Chỉ hoạt động với exact match — không xử lý được wildcard (ternary match).
- **TCAM (Ternary CAM) hardware:** O(1). Giải pháp phần cứng chuyên dụng, không áp dụng được cho switch phần mềm.
- **Cây quyết định (HiCuts, HyperCuts):** O(log N) nhưng memory usage tăng mũ với số field.
- **Tuple space search (TSS):** O(k) với k = số "tuple" — thuật toán OVS dùng.

OVS chọn TSS vì ba lý do: (1) xử lý wildcard match tự nhiên, (2) memory usage tuyến tính với số flow (không mũ), (3) update flow table O(1) khi thêm/xóa rule.

## 9.15.2 Tuple space search (TSS) — ý tưởng cốt lõi

TSS phát triển bởi Srinivasan et al. trong bài báo "Packet Classification Using Tuple Space Search" (SIGCOMM 1999). Ý tưởng chính: chia flow table thành nhiều **subtable**, mỗi subtable chứa các rule có cùng **pattern mask** (bit mask giống nhau). Tra cứu = truy vấn từng subtable với hash, lấy kết quả priority cao nhất.

**Định nghĩa tuple:** một "tuple" là một vector bit mask xác định "bit nào được match exact, bit nào wildcard". Ví dụ trong flow table IPv4:
- Tuple A: match src IP /24 + dst IP /16 → mask = `(0xFFFFFF00, 0xFFFF0000, ...)` cho src/dst IP.
- Tuple B: match chỉ dst port + protocol → mask = `(0, 0, 0xFFFF, 0xFF)` cho src/dst IP/port/proto.
- Tuple C: match src MAC + VLAN ID → mask = `(0xFFFFFFFFFFFF, 0xFFF, 0, ...)`.

Mỗi rule thuộc duy nhất một tuple (xác định bởi mask field của rule). Rule có cùng mask → cùng tuple → cùng subtable.

**Thuật toán tra cứu:**
1. Với packet P, duyệt qua danh sách tuple trong classifier.
2. Với mỗi tuple T: áp mask T lên packet P → key K = P & T.mask. Truy vấn hash map của subtable T với key K.
3. Nếu match → lấy rule + priority của rule.
4. Sau khi duyệt hết tuple, trả về rule có priority cao nhất trong tất cả match.

**Độ phức tạp:** O(k) với k = số tuple. Trong thực tế OVS k thường 3-20 cho cluster môi trường production (10 rule/tuple trung bình). Đây là lý do classifier OVS đạt hàng triệu tra cứu/giây per CPU core.

**Cấu trúc dữ liệu OVS:**
```c
struct classifier {
    struct cls_subtable *subtables[];  // danh sách subtables
    // ... index metadata ...
};

struct cls_subtable {
    struct cmap rules;            // hash map: key (từ mask) → cls_rule
    struct minimatch mask;         // mask của subtable
    int max_priority;              // priority max trong subtable
    // ... metadata khác ...
};

struct cls_rule {
    struct minimatch match;        // match field
    int priority;                  // priority
    // ... dữ liệu action ...
};
```

`cmap` (Cuckoo hash map) là hash map không khóa của OVS — đọc không khóa, ghi dùng thao tác nguyên tử. Hỗ trợ nhiều reader + một writer đồng thời không tranh chấp.

## 9.15.3 Staged lookup — tối ưu cho priority sort

Naive TSS có vấn đề: phải duyệt TẤT CẢ tuple để tìm rule priority cao nhất — ngay cả khi tuple đầu tiên đã có rule priority rất cao. OVS tối ưu với **staged lookup**: sort tuple theo max_priority (cao nhất trước). Nếu match rule có priority > max_priority của tuple còn lại → có thể thoát sớm.

Pseudo-code:
```
function tss_lookup(packet P, classifier C):
    best_rule = None
    best_priority = -infinity
    for tuple T in C.tuples sorted by T.max_priority DESC:
        if T.max_priority < best_priority:
            break  // thoát sớm — không tuple còn lại nào có thể cải thiện
        key = apply_mask(P, T.mask)
        rule = cmap_find(T.rules, key)
        if rule and rule.priority > best_priority:
            best_rule = rule
            best_priority = rule.priority
    return best_rule
```

Trong thực tế, nếu rule có phân phối priority phù hợp (một vài rule priority cao cho drop/allow, nhiều rule priority thấp cho default), thoát sớm trúng trong 1-2 tuple đầu → tra cứu gần O(1).

## 9.15.4 Tương tác với megaflow cache

Classifier tra cứu O(k) vẫn chậm hơn hash lookup O(1) trong kernel. Mỗi lần upcall mất khoảng microsecond per packet — không đủ cho traffic tốc độ cao. **Megaflow cache** trong kernel datapath giải quyết: cache kết quả classifier cho các "flow class" (packet có cùng output classifier).

**Quá trình caching:**
1. Packet P không match megaflow → upcall ovs-vswitchd.
2. Classifier tra cứu → rule R với match mask M.
3. ovs-vswitchd build megaflow: match = P & M (tập con bit match), action = R.action.
4. Cài megaflow vào kernel.
5. Packet P' với cùng pattern P' & M = P & M → match megaflow → kernel forward trực tiếp, không upcall.

Điểm quan trọng: trường match của megaflow là TẬP CON của trường match của rule — chỉ các bit thật sự ảnh hưởng decision. Nếu rule nói "match dst_ip /24", nhưng trong khi tra cứu classifier chỉ check dst_ip bit 0-23 và dst_port (cho priority sort thoát sớm), megaflow sẽ match dst_ip /24 + dst_port.

**Classifier** là "đường chậm" (hàng nghìn pps), **megaflow** là "đường nhanh" (hàng triệu pps). Với workload ổn định, 99%+ packet match megaflow → classifier rarely invoked.

## 9.15.5 Quan sát — classifier hoạt động

OVS cung cấp nhiều công cụ để inspect classifier:

**Xem số subtable và rule:**
```
[host]# ovs-appctl dpif/show-dp-features br0
```

**Dump OpenFlow rule với priority sort:**
```
[host]# ovs-ofctl -O OpenFlow15 dump-flows br0 --sort=priority
```

**Đếm upcall (số lần gọi classifier):**
```
[host]# ovs-appctl upcall/show
```
Output hiển thị tần suất upcall per-thread, tỷ lệ hit/miss của megaflow.

**Thống kê flow table:**
```
[host]# ovs-appctl bridge/dump-flows br0
```
Show cấu trúc flow table bên trong + phân vùng subtable.

**Dump bảng megaflow:**
```
[host]# ovs-appctl dpif/dump-flows br0
```
Output các megaflow đang active trong kernel — là cache của tra cứu classifier.

**Dấu hiệu flow table mất cân bằng:** nếu tần suất upcall cao (>1000/s) và tỷ lệ hit megaflow thấp (<95%) → flow table chưa tối ưu. Thường do:
- Quá nhiều rule với pattern mask khác nhau (k lớn).
- Phân phối priority xấu (nhiều rule cùng priority cao).
- Pattern traffic packet biến động cao (không hit cache).

## 9.15.6 Tinh chỉnh flow table cho hiệu quả classifier

Hướng dẫn thực tế khi thiết kế flow table OVS:

**(1) Gộp rule có cùng pattern match:** nếu có 100 rule cùng match trên dst_ip /24 (khác nhau ở priority/action), chúng vào cùng subtable → 1 tuple, tra cứu O(1) cho nhóm này.

**(2) Tránh wildcard không cần thiết:** match càng chặt (nhiều bit exact) → ít subtable hơn. Bit wildcard = bit mask 0 = một bit mới trong pattern subtable. 

**(3) Dùng priority sparingly:** flow table có 10 mức priority khác nhau cần 10 nhóm với priority sort. Flow table có 3 priority (high/mid/low) → chỉ 3 nhóm. Ít priority = thoát sớm hiệu quả hơn.

**(4) Tránh chuỗi matching sâu:** mỗi packet đi qua nhiều bảng (resubmit) = nhiều lần tra cứu classifier. Compile vào single table nếu có thể.

**(5) Giám sát số subtable:** dùng `ovs-appctl dpif/show` để theo dõi. Nếu số subtable tăng bất thường → có thể signal flow churn (rule bị add/remove liên tục).

**(6) Tận dụng megaflow learning:** megaflow tự động học subset bit. Lập trình viên không cần tối ưu — OVS tự làm. Chỉ cần tránh viết rule với subset xung đột (rule 1 phụ thuộc bit X, rule 2 phụ thuộc bit Y nhưng không tương tác → tạo nhiều biến thể megaflow).

### ▶ Guided Exercise 1: Quan sát sự hình thành subtable của classifier

> **Trạng thái verification lab:** Output dựa trên OVS 2.17 docs; chờ verify lab thực.

**Mục đích:** Tạo flow table với đa dạng pattern match, quan sát classifier tự tổ chức thành các subtable.

**Chuẩn bị:** Node Ubuntu 22.04 với OVS 2.17.

**Bước thực hiện:**

**1.** Tạo bridge mới:
```
[host]# ovs-vsctl add-br br0
[host]# ovs-vsctl add-port br0 eth0
```

**2.** Add nhiều flow với pattern match khác nhau (tạo nhiều tuple):
```
[host]# ovs-ofctl add-flow br0 "priority=100,dl_type=0x0800,nw_src=10.0.0.0/24,actions=drop"
[host]# ovs-ofctl add-flow br0 "priority=90,dl_type=0x0800,nw_dst=10.0.0.0/24,actions=drop"
[host]# ovs-ofctl add-flow br0 "priority=80,dl_type=0x0800,nw_proto=6,tcp_dst=80,actions=output:1"
[host]# ovs-ofctl add-flow br0 "priority=70,dl_type=0x0800,nw_proto=17,udp_dst=53,actions=output:2"
[host]# ovs-ofctl add-flow br0 "priority=60,dl_type=0x0806,actions=normal"
[host]# ovs-ofctl add-flow br0 "priority=1,actions=drop"
```

Mỗi rule có pattern khác nhau → OVS tạo 6 tuple khác nhau.

**3.** Inspect số subtable:
```
[host]# ovs-appctl dpif/show-dp-features br0
```

Output kỳ vọng (đoạn quan trọng):
```
br0:
  ...
  Flow table:
    subtables: 6
    active rules: 6
  ...
```

**4.** Add thêm rule với cùng pattern như rule 1 (chia sẻ tuple):
```
[host]# ovs-ofctl add-flow br0 "priority=95,dl_type=0x0800,nw_src=192.168.0.0/24,actions=drop"
[host]# ovs-appctl dpif/show-dp-features br0
```

Quan sát: số subtable vẫn 6 (không tăng), nhưng active rules = 7 → 2 rule vào cùng subtable "dl_type=IPv4 + nw_src /24".

**5.** Gửi traffic + quan sát megaflow:
```
[host]# # Từ host khác, ping hoặc sinh traffic
[host]# ovs-appctl dpif/dump-flows br0 | head -20
```

Output: mỗi megaflow là tập con của OpenFlow rule + compile với chỉ các bit cần thiết cho decision.

**Bài học học được:**

Thứ nhất, classifier TỰ tổ chức subtable dựa trên pattern của rule — lập trình viên không điều khiển tường minh. Thiết kế quy tắc tốt = pattern nhất quán = ít subtable = tra cứu nhanh.

Thứ hai, megaflow là cache dựa trên subset bit, không đầy đủ toàn bộ match rule. Đây là lý do một flow rule đơn có thể sinh nhiều megaflow entry khác nhau trong kernel.

Thứ ba, công cụ quan sát (dpif/show, upcall/show, bridge/dump-flows) là cần thiết để gỡ lỗi khi OVS chậm — thường do flow table mất cân bằng (quá nhiều subtable hoặc phân phối priority xấu).

**Cleanup:**
```
[host]# ovs-vsctl del-br br0
```

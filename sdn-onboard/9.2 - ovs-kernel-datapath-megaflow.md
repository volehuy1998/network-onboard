## Bối cảnh: bài toán kernel datapath phải giải

Giả sử một host KVM chạy 50 VM, mỗi VM có 20 TCP connection đồng thời. Tổng cộng 1000 TCP microflow. Mỗi microflow gồm tổ hợp (src_ip, dst_ip, src_port, dst_port, protocol) — 5-tuple. Kernel datapath phải quyết định action forwarding cho mỗi packet với tốc độ triệu packet/second.

Cách tiếp cận naive — userspace daemon quyết định mọi packet — không khả thi vì context switch kernel↔userspace tốn ~2 microsecond, giới hạn throughput ở ~500K packet/second.

Cách tiếp cận ngược lại — toàn bộ OpenFlow pipeline trong kernel — không khả thi vì pipeline phức tạp với classifier, prioritized rules, arbitrary header modification. Kernel code không nên chứa logic như vậy.

Giải pháp OVS: **kernel duy trì cache cho quyết định đã tính**, userspace tính quyết định khi cache miss. Cache phải nhanh (nanosecond scale), chính xác (không sai policy), và hiệu quả về bộ nhớ.

## 9.2.1 Microflow cache: thiết kế OVS 1.x và giới hạn

OVS 1.x (2011-2013) dùng **microflow cache**: mỗi entry mô tả chính xác một 5-tuple (exact match), không wildcard. Lookup O(1) qua hash table.

Ưu điểm: cực nhanh khi hit. Lookup chỉ cần compute hash + compare full key.

Nhược điểm fatal: **mỗi microflow khác biệt về bất kỳ field nào đều yêu cầu entry riêng**. Ví dụ:

- TCP connection 1: src=10.0.0.1:50001 → dst=10.0.0.2:80 → entry A
- TCP connection 2: src=10.0.0.1:50002 → dst=10.0.0.2:80 → entry B (khác src_port)
- TCP connection 3: src=10.0.0.1:50003 → dst=10.0.0.2:80 → entry C

Mặc dù cả ba đi qua cùng OpenFlow rule ("allow TCP dst=10.0.0.2:80") và cùng action (output:5), mỗi connection có entry cache riêng chỉ khác ở src_port. Với 1000 connection, cache có 1000 entry.

Kết quả: cache thrashing. Cache có kích thước cố định, khi số microflow vượt quá size, entry cũ bị evict. Packet evict gặp cache miss → upcall → reinstall → vòng luẩn quẩn.

NSDI 2015 đo trên workload Rackspace production (2013, trước megaflow): miss rate 10-30% trong peak. Overhead quá lớn để scale.

## 9.2.2 Megaflow cache: đột phá của NSDI 2015

**Megaflow** là cache entry với **wildcard** — match một tập microflow có cùng pattern. Thay vì 1000 entries cho 1000 connection TCP đến port 80, chỉ một entry duy nhất `dst=10.0.0.2, dst_port=80, proto=TCP, wildcard src_ip+src_port` cover tất cả.

Compression factor điển hình 10-100 lần. Workload 100K microflow → cache 2.000-10.000 entry.

**Câu hỏi then chốt**: làm sao quyết định field nào wildcard? OVS giải quyết bằng **lazy wildcarding**. Khi userspace classifier xử lý microflow lần đầu:

1. Parse packet thành 5-tuple + field khác.
2. Walk OpenFlow pipeline: match rules table 0 → table 1 → ..., execute action.
3. Trong quá trình walk, **track** field nào đã được inspect.
4. Field KHÔNG được inspect → wildcard.
5. Compile megaflow entry với wildcard cho field không inspect.

Ví dụ: rule "match dst_ip=10.0.0.2, dst_port=80, proto=TCP → output:5". Packet với src_ip=10.0.0.1, src_port=50001:

- Classifier inspect dst_ip (match), dst_port (match), proto (match).
- src_ip và src_port KHÔNG inspect.

Megaflow entry: match `dst_ip=10.0.0.2, dst_port=80, proto=TCP`, wildcard `src_ip=*, src_port=*`.

**Correctness guarantee**: lazy wildcarding không phạm rule nào. Nếu có rule khác ràng buộc src_port, classifier sẽ inspect khi walk, không wildcard. Correct by construction.

## 9.2.3 Tuple Space Search cho megaflow lookup

Khi có nhiều megaflow với mask khác nhau, mỗi megaflow có **tuple** — tập field được match (không wildcard). OVS dùng **Tuple Space Search (TSS)** — thuật toán từ Srinivasan et al. SIGCOMM 1998: nhóm entry theo tuple, mỗi tuple có hash table riêng.

Lookup packet: với mỗi tuple, extract field theo mask, compute hash, lookup hash table. Nếu match, return action. Nếu không, try tuple tiếp theo.

Độ phức tạp: O(T × H) với T = số tuple (thường < 20), H = chi phí hash lookup (O(1) trung bình).

> **Key Topic:** Lựa chọn TSS thay vì TCAM hay decision tree là quyết định thực dụng. TCAM quá đắt trên commodity server. Decision tree build cost cao khi rules thay đổi thường xuyên. TSS là compromise tối ưu cho software datapath. NSDI 2015 benchmark: TSS đạt 10M+ lookup/second per core trên Intel Xeon.

## 9.2.4 Upcall: slow path và fast path

USC Lab 9 (`doc/ovs/Day 4-lab 9 - Open vSwitch Kernel Datapath.txt`) trình bày rõ:

> "Kernel module directs the packet to the userspace. The userspace makes the decisions about the actions to be taken against the packet according to OpenFlow entries (slow path)."
>
> "The action entry is stored in the kernel and used to forward subsequent packets which makes the forwarding faster (fast path)."

**Fast path (cache hit)**:

1. Packet đến NIC → kernel stack → `openvswitch.ko` rx hook.
2. Kernel extract flow key từ headers.
3. Kernel lookup megaflow cache qua TSS.
4. Match → execute action (output:5 → `netdev_ops->ndo_start_xmit`).
5. Update counter. Không chạm userspace. Latency ~1-5 microsecond.

**Slow path (cache miss)**:

1. Packet đến, lookup miss.
2. Kernel gửi **upcall** qua Netlink lên userspace.
3. Upcall chứa packet bytes, metadata (in_port, timestamp), reason.
4. `ovs-vswitchd` handler thread nhận upcall.
5. Handler walk OpenFlow pipeline, track inspected fields.
6. Compile megaflow entry + action list.
7. `OVS_FLOW_CMD_NEW` về kernel — cài megaflow.
8. `OVS_PACKET_CMD_EXECUTE` để kernel execute action cho packet này.
9. Packet tương tự sau đó hit fast path.

USC Lab 9: lần đầu flow qua switch, "kernel table is empty" → miss, slow path. Sau "action entries are installed in the kernel", packet tiếp theo đi fast path.

Chi phí slow path: ~50-200 microsecond per packet. Nếu hit rate thấp, slow path chi phối throughput.

## 9.2.5 Ukeys và revalidator threads

Khi megaflow cài vào kernel, `ovs-vswitchd` duy trì **ukey (unique key)** — bản sao userspace chứa flow key, mask, action, statistics counter cuối cùng, timestamp last_used.

**Revalidation**: khi rules thay đổi, revalidator threads (mặc định 4, 500ms interval) scan ukey:

1. Với mỗi ukey, re-walk pipeline với flow key.
2. So sánh action mới vs cũ.
3. Khác → `OVS_FLOW_CMD_SET` update kernel.
4. Không match rule nào → `OVS_FLOW_CMD_DEL`.

**Timeout**: rules có idle_timeout và hard_timeout. Revalidator track timestamp, xóa khi timeout.

Cấu hình `other_config:n-revalidator-threads`. Cache lớn, scan tốn CPU — tăng lên 8-16 thread nếu cần.

## 9.2.6 Upcall rate limit

Compass Ch Q mô tả tuning:

- `upcall/show` — số upcall in-flight, avg, max, limit (mặc định 200.000 flow-install/sec).
- `upcall/set-flow-limit 500000` — tăng limit cho workload scale cao.

Chạm limit → kernel drop packet → counter `lost` trong `ovs-dpctl show` tăng.

Tương quan:

- `hit` tăng tuyến tính → bình thường.
- `missed` tăng → cache không hit tốt.
- `lost` > 0 → chạm rate limit, nguy cơ mất traffic.

Cấu hình `other_config:n-handler-threads` (mặc định 2). Tăng nếu miss rate cao nhưng lost = 0 — handler nghẽn.

## 9.2.7 Cache hit rate trong production

NSDI 2015 báo cáo Rackspace production:

- Trước megaflow: hit 70-90%, miss 10-30% peak.
- Sau megaflow: hit 99%+, miss dưới 1%.

Với packet rate 1M/sec, ~10.000 upcall/sec — trong khả năng handler thread.

**Dấu hiệu cảnh báo sớm**:

- Coverage miss rate > 1% → compress không hiệu quả.
- `ovs-dpctl show` `lost` > 0 → hit rate limit.
- `upcall/show` avg cao → handler nghẽn.

**Scenarios hit rate tụt**:

1. Traffic đa dạng cao — mỗi microflow khác về field rules match.
2. Rules thay đổi liên tục — revalidator invalidate nhiều ukey.
3. Cache size nhỏ, `max-idle` thấp — entry evict sớm.

## Điểm cốt lõi cần nhớ

Thứ nhất, megaflow cache là đóng góp kiến trúc quan trọng nhất của NSDI 2015 — chuyển từ exact-match sang wildcard-match qua lazy wildcarding. Compress 10-100 lần, cho phép OVS scale đến workload cloud production.

Thứ hai, Tuple Space Search là thuật toán phù hợp cho software datapath — constant factor tốt và dynamic insert/delete. Thực dụng dựa trên ràng buộc commodity hardware.

Thứ ba, slow path + fast path là cơ chế cốt lõi — USC Lab 9 minh họa rõ: packet đầu đi slow path, cài action vào kernel, packet sau đi fast path. Hiểu cơ chế giúp debug "traffic đầu chậm sau nhanh" hoặc "missed counter tăng".

Thứ tư, cache hit rate 99%+ là mục tiêu vận hành. Dưới ngưỡng, cache thrashing, handler nghẽn, nguy cơ mất traffic. Monitor bằng `ovs-dpctl show`, `ovs-appctl coverage/show`, `ovs-appctl upcall/show`.

**Giả thuyết ngược cần bác bỏ:**

> *"Kernel flow table và OpenFlow flow table là một thứ. Lệnh `ovs-dpctl dump-flows` và `ovs-ofctl dump-flows` trả về cùng dữ liệu."*

### Bước 1, so sánh kết quả hai lệnh

```bash

```

```

```

```bash

```

Tám entry với match field đầy đủ + megaflow wildcarded. Giả thuyết bị bác bỏ: kernel và OpenFlow là hai bảng khác nhau hoàn toàn về số lượng và format.

### Bước 2, giải thích

OVS có kiến trúc hai tầng:

1. **Userspace (OpenFlow):** bảng rule do controller hoặc admin cài qua `ovs-ofctl`. Thường ít entry, mỗi entry bao phủ nhiều gói tin khác nhau qua wildcard.
2. **Kernel (megaflow):** cache đã biên dịch của userspace, một entry cho mỗi megaflow. Mỗi entry đúng cho một bộ gói tin cụ thể đã thấy.

Ví dụ OpenFlow flow `priority=0 actions=NORMAL` bao phủ mọi gói tin. Kernel không cache flow đó, mỗi gói tin đầu tiên bị miss, đẩy lên userspace, userspace xử lý action `NORMAL` (MAC learning + flood/forward), biên dịch kết quả xuống kernel thành một megaflow entry cụ thể. Gói tin tương tự sau đó match megaflow trực tiếp trong kernel, không đẩy lên userspace nữa.

Đây là lý do cài 1 OpenFlow flow có thể sinh N megaflow trong kernel.

```bash
sudo ovs-dpctl show
# Ghi lại hit / missed / lost lần 2
# Tính cache hit rate = hit_delta / (hit_delta + missed_delta)
```

Kỳ vọng: cache hit rate ≥ 99% cho iperf3 steady-state. Gói tin đầu tiên của TCP session là miss (slow path), sau đó tất cả match megaflow.

### Bước 4, tạo workload cache-unfriendly

Chạy nhiều TCP session ngắn đồng thời để tạo nhiều flow mới:

```bash
# Shell script trên h1
for i in {1..100}; do
  (curl -s http://10.0.0.2:8000/ -o /dev/null) &
done
wait
```

Đo lại `ovs-dpctl show`. Miss rate tăng đáng kể.

### Bài học

Cache hit rate là chỉ số sức khoẻ quan trọng nhất của kernel datapath. Dưới 95% là cảnh báo, dưới 90% là critical. Nguyên nhân thường gặp:

- Workload quá nhiều flow khác nhau (connection churn cao).
- Cache limit chạm trần (upcall/set-flow-limit quá thấp).
- Flow expiration quá ngắn (idle_timeout thấp).
- Revalidator thread không theo kịp (tăng n-revalidator-threads).

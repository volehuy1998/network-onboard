## 8.2.1 VLAN stacking: single-tagged, QinQ, triple-tagged

**Single-tagged 802.1Q (1998):** a 4-byte VLAN tag is inserted after the MAC header. The 12-bit VLAN ID gives a maximum of 4096 VLANs.

```
+--------+--------+----------+----------+--------+------+
| DstMAC | SrcMAC | TPID=8100| VID+PCP  | EthType| Data |
| 6B     | 6B     | 2B       | 2B       | 2B     | ...  |
+--------+--------+----------+----------+--------+------+
```

**QinQ 802.1ad (2011):** S-VLAN (service provider) plus C-VLAN (customer) are stacked. The service provider transports the customer VLAN inside its own VLAN space. This expands the 4096-VLAN limit per tenant on shared infrastructure.

**Triple-tagged:** a niche carrier deployment with three nested tags for the MEF service hierarchy. It is rarely seen outside telco.

VLAN configuration on Linux:

```bash
[host]# ip link add link eth0 name eth0.100 type vlan id 100
[host]# ip addr add 192.168.100.1/24 dev eth0.100
[host]# ip link set eth0.100 up
```

USC Day 5 Lab 6 covers VLAN trunking with OVS `vlan_mode=trunk` and `trunks=[10,20,30]` instead of a Linux VLAN subinterface. The difference: a Linux VLAN subinterface handles a single VLAN per interface, whereas an OVS trunk port handles many VLANs with programmable policy.

## 8.2.2 Bonding: seven modes

The Linux kernel bonding driver (`drivers/net/bonding/`) implements seven modes:

| Mode | Name | Behaviour | Peer switch requirement |
|---|---|---|---|
| 0 | `balance-rr` | Round-robin packets across slaves | Static LAG (Etherchannel `on` mode) |
| 1 | `active-backup` | One slave is active, the rest are standby | None |
| 2 | `balance-xor` | Hash (MAC src XOR MAC dst) modulo slave count | Static LAG |
| 3 | `broadcast` | Replicate the packet to every slave | Niche redundancy use |
| 4 | `802.3ad` (LACP) | Dynamic link aggregation per IEEE 802.3ad | LACP-capable peer |
| 5 | `balance-tlb` | Transmit load balance, dedicated receive | None |
| 6 | `balance-alb` | Adaptive load balance for TX plus ARP-based RX | None |

Configuration:

```bash
[host]# ip link add bond0 type bond mode 802.3ad miimon 100 lacp_rate fast
[host]# ip link set eth0 master bond0
[host]# ip link set eth1 master bond0
[host]# ip link set bond0 up
```

Mode 4 (LACP) is the standard choice for production multi-link. The peer switch must have a matching LACP-capable LAG configuration.

## 8.2.3 LACP specifics

IEEE 802.3ad-2020 defines the LACPDU exchange between the Actor (our side) and the Partner (the peer):

- **System ID:** MAC address plus priority. The peer with the lower system ID is preferred as the negotiator.
- **Key:** identifies the aggregation group. The Actor and the Partner must have matching keys to aggregate.
- **Port priority:** sets priority within the group.
- **State bits:** Activity, Timeout, Aggregable, Synchronizing, Collecting, Distributing, Defaulted, Expired.

Timers:

- **Fast (1 second):** LACPDU every 1 second, peer timeout 3 seconds. Failover is sub-100 ms. CPU overhead is higher.
- **Slow (30 seconds):** LACPDU every 30 seconds, peer timeout 90 seconds. This is the default.

Use the fast timer for a production data centre, and the slow timer for a WAN uplink.

The switch-side configuration must match in three places:
- Same port list.
- Same LACP mode (active-active or active-passive).
- Same timer.

OVS bonding (Part 9.6) has a fallback mode `lacp-fallback-ab=true`. It falls back to active-backup if LACP negotiation fails. The Linux kernel bonding driver has no equivalent fallback.

## 8.2.4 Team daemon: deprecated in RHEL 9

The team driver (`libteam`) appeared in 2012 as a replacement for bonding. It offered:

- A userspace daemon `teamd` that implements the logic.
- A D-Bus API for programmatic control.
- An extensible runner set (`activebackup`, `roundrobin`, `loadbalance`, `random`, `lacp`).

RHEL 7 and 8 shipped team as an alternative to bonding. However, **RHEL 9 (2022) deprecated team** and recommends bonding.

Reasons for the deprecation:

1. **Tighter kernel integration:** bonding lives entirely in the kernel; team has a userspace dependency.
2. **Team daemon crash:** a `teamd` crash can disrupt networking, while bonding does not depend on a userspace process.
3. **Duplicate functionality:** every feature that team provides is already available in bonding (LACP mode 4 replaces the `teamd` LACP runner).
4. **Maintenance burden:** two drivers do equivalent work, and team contributions have declined.

For new deployments, use bonding mode 4 (LACP). Do not use team.

## 8.2.5 Bonding driver, tier 2 source (`drivers/net/bonding/` v5.15)

The Linux bonding driver lives in `drivers/net/bonding/`. The main modules are:

- `bond_main.c`: `bond_xmit_hash()` decides which slave transmits a packet, and `bond_xmit_*_slave()` performs per-mode forwarding.
- `bond_3ad.c`: the 802.3ad LACP state machine and the LACPDU exchange.
- `bond_alb.c`: the `balance-alb` adaptive learning logic.
- `bond_options.c`: sysctl and sysfs option handling.

### Mode 4 (LACP) state machine

LACP runs four sub-state machines per port:

1. **Receive Machine** (`ad_rx_machine()` in `bond_3ad.c`): processes incoming LACPDUs.
2. **Periodic Transmit Machine** (`ad_periodic_machine()`): emits LACPDUs periodically (1 second fast, 30 seconds slow).
3. **Mux Machine** (`ad_mux_machine()`): controls the Selected and Standby state for each slave.
4. **Tx Machine** (`ad_tx_machine()`): builds and sends LACPDUs.

State transitions:

```
INITIALIZE -> DETACHED -> WAITING -> ATTACHED -> COLLECTING_DISTRIBUTING
```

A slave forwards traffic only when it reaches the `COLLECTING_DISTRIBUTING` state. Any change in negotiation (peer key change, system ID change) re-enters `DETACHED`.

### Transmit hash (mode 2 and mode 4)

In mode 2 and mode 4, `bond_xmit_hash()` selects a slave through a hash:

```c
// bond_main.c, function bond_xmit_hash() simplified
u32 bond_xmit_hash(struct bonding *bond, struct sk_buff *skb)
{
    if (bond->params.xmit_policy == BOND_XMIT_POLICY_LAYER2) {
        // Hash MAC src+dst (XOR)
        return bond_xmit_hash_l2(skb);
    } else if (bond->params.xmit_policy == BOND_XMIT_POLICY_LAYER34) {
        // Hash IP src+dst plus L4 ports
        return bond_xmit_hash_l34(skb);
    } else if (bond->params.xmit_policy == BOND_XMIT_POLICY_ENCAP34) {
        // Hash inner IP plus L4 for VXLAN or Geneve encap
        return bond_xmit_hash_encap34(skb);
    }
}
```

There are five hash policies: `layer2` (the default, MAC-based), `layer2+3` (MAC plus IP), `layer3+4` (IP plus L4), `encap2+3`, `encap3+4`. For overlay (Geneve or VXLAN), `encap3+4` lets the inner flow distribute across both slaves. This avoids the single-flow bottleneck of using the same outer 5-tuple.

```bash
[host]# ip link set bond0 type bond xmit_hash_policy layer3+4
[host]# cat /sys/class/net/bond0/bonding/xmit_hash_policy
layer3+4 1
```

### LACP-related sysfs

```bash
[host]# cat /sys/class/net/bond0/bonding/ad_actor_system     # Local system MAC
[host]# cat /sys/class/net/bond0/bonding/ad_partner_mac      # Peer system MAC
[host]# cat /sys/class/net/bond0/bonding/ad_actor_key        # Local key
[host]# cat /sys/class/net/bond0/bonding/ad_partner_key      # Peer key
[host]# cat /sys/class/net/bond0/bonding/ad_aggregator       # Aggregator ID
[host]# cat /sys/class/net/bond0/bonding/ad_num_ports        # Active port count
[host]# cat /sys/class/net/bond0/bonding/ad_select          # stable / bandwidth / count
```

`ad_select`:

- `stable` (default): re-aggregate only when necessary.
- `bandwidth`: pick the aggregator with the highest total bandwidth.
- `count`: pick the aggregator with the largest port count.

## 8.2.6 OVS bonding comparison (cross-link to Part 9.6)

Part 9.6 covers OVS bonding in depth. Here is a quick comparison against the Linux kernel bonding driver:

| Feature | Linux bond | OVS bond |
|---------|-----------|----------|
| Modes | 7 modes | 3 modes (`active-backup`, `balance-slb`, `balance-tcp`) |
| LACP fallback | No | Yes (`lacp-fallback-ab=true`) |
| Per-flow hash | layer2, layer3, layer4, encap3+4 | `hash_basis` driven by datapath rules |
| Programmable | No | Yes, through OpenFlow flow rules |
| BFD integration | No | Yes (combined with HA failover) |
| Hardware offload | `ethtool` offload | TC offload framework (Part 9.5) |

OVS bond is preferable for:

- An OVN cluster that needs BFD-driven failover.
- Multi-tenant deployments with per-flow QoS.
- DPDK fast paths.

Linux bond is enough for:

- Bare-metal server uplinks (no SDN policy).
- Storage networks with multiple links.
- Container hosts whose bond does not need OVS.

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| `Bonding Mode` | matches expected | wrong mode | empty | Mode misconfigured |
| `LACP rate` | `fast` (data centre) or `slow` (WAN) | mismatch with peer | (n/a) | Both ends must agree |
| `Active Aggregator Number of ports` | equals expected slave count | less than expected (one slave failed) | 0 | The whole bond is down |
| `Actor Churn State` | `none` | `churned` (re-negotiation) | persistent `churned` | LACP unstable, peer flapping |
| `MII Status` per slave | `up` | `down` (one slave down) | all `down` | Physical link issue |
| `Link Failure Count` | 0 | 1 to 5 (occasional) | greater than 10 (frequent flap) | NIC or cable issue |
| `port state` (bit 6 equals sync) | 63 (all bits set) | less than 63 | less than 32 (no Activity) | LACP not negotiated |

The `port state` 8-bit field per RFC IEEE 802.3ad:

- bit 0 (LACP_Activity): 1 means active LACP.
- bit 1 (LACP_Timeout): 1 means the fast timer.
- bit 2 (Aggregable): 1 means the port can aggregate.
- bit 3 (Synchronizing): 1 means in sync with the peer.
- bit 4 (Collecting): 1 means receiving.
- bit 5 (Distributing): 1 means transmitting.
- bit 6 (Defaulted): 0 means received from the peer (1 means using a default).
- bit 7 (Expired): 0 means current (1 means expired).

Healthy: bits 0 to 5 set, bits 6 and 7 clear, which is `0x3F` equals 63.

**Failure scenarios:**

- `port state = 0x07` (LACP active plus fast plus aggregable, but not synchronizing): the peer has not responded. Check the switch LACP configuration.
- `Actor Churn` persistent: the peer key or MAC keeps changing, which suggests a switch reboot loop or a LACP timer mismatch.
- `Number of ports = 1`: one slave dropped. Check the MII status, the cable, and the switch port.

## 8.2.8 Capstone POE: balance-rr versus LACP for Geneve overlay

> **Context:** an OVN cluster of 50 chassis, each chassis with two 25G uplinks. An architect proposes `balance-rr` (round-robin) instead of LACP for higher throughput.

**Claim to verify:** "balance-rr (mode 0) gives higher throughput than LACP (mode 4) for overlay traffic, because round-robin needs no hash."

**Predict:** False for overlay. balance-rr causes out-of-order delivery and a TCP retransmit storm.

**Observe (assumed):**

- Set up mode 0 `balance-rr` and run a large TCP transfer cross-chassis. Nominal throughput is 50 Gbps (2 times 25G). Actual throughput is only 12 to 18 Gbps and the TCP retransmit count is 5K to 10K per second.
- Set up mode 4 LACP with `xmit_hash_policy=encap3+4`. Throughput is 35 to 45 Gbps and retransmits are below 100 per second.

**Explain:**

1. **balance-rr** divides packets round-robin across slaves **without flow awareness**. Packets of the same TCP connection take two different paths, which causes out-of-order arrival. The TCP receiver flags out-of-order packets, triggering retransmits, and congestion control reduces the CWND. Effective throughput drops by 50 to 70 percent.
2. **LACP with `encap3+4`** hashes the inner IP and L4 ports. Each connection takes one slave consistently, producing in-order delivery and healthy TCP. The hash distribution still allows multiple flows to spread across both slaves.
3. **balance-rr is only good for** UDP (no ordering required), a single flow saturating a single link, or for `active-backup` mode if all you need is redundancy.

**Falsification test:**

- Run `iperf3` TCP cross-chassis with mode 0 versus mode 4 plus `encap3+4`.
- Measure the retransmit rate with `ss -tin`.
- If mode 0 retransmits are lower than mode 4, that is a counter-example. In practice, mode 0 retransmits are always higher.

**Where the claim fails:** "Round-robin needs no hash, so throughput is higher" ignores TCP semantics. The correct rule is **LACP with the right hash policy for overlay (`encap3+4`) is the standard for an OVN cluster. balance-rr is not production-grade.**

> **Rule:** Overlay TCP traffic equals LACP mode 4 plus `xmit_hash_policy=encap3+4`. balance-rr is only for the UDP single-flow saturation case.

## Key takeaways

First, single-tagged 802.1Q VLAN is enough for 4096 VLANs. QinQ becomes necessary when a service provider must transport customer VLANs.

Second, Linux bonding has seven modes. Mode 4 (`802.3ad` LACP) is the production standard. Mode 1 (`active-backup`) is for simple scenarios.

Third, the LACP fast timer (1 second) is for a data centre in production; the slow timer (30 seconds) is for a WAN. OVS has `lacp-fallback-ab`, while kernel bonding does not.

Fourth, the team daemon was deprecated in RHEL 9 (2022). All new deployments should use bonding.

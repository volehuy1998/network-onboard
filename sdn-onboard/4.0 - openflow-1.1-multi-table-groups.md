## Context: from 1.0 to 1.1, two years to relieve the single-table limitation

Part 3.1 analysed in detail the limitation of OpenFlow 1.0: a single flow table that suffers cross-product explosion when policy has many orthogonal dimensions. The 1.0.0 specification was published on 2009-12-31; the community implemented and deployed the protocol throughout 2010 and quickly hit scale problems. The Google B4 WAN (deployed in 2011, paper at SIGCOMM 2013), several Stanford research deployments on GENI, and hyperscaler internal testing all reported the same conclusion: a single table does not scale for complex data-centre policy.

ONF had not yet been founded (it would be founded on 2011-03-21, see Part 3.2), so the Stanford Clean Slate Lab continued to shepherd the specification's evolution. The Stanford team's decision: instead of waiting for ONF to be ready, release specification 1.1 as soon as possible so that industry would have something new to implement. The result: OpenFlow 1.1.0 was released on **2011-02-28**, exactly three weeks before ONF was officially founded (2011-03-21).

OF 1.1 is the last specification shepherded by Stanford. After IP ownership transferred to ONF, every version from 1.2 onward was published by ONF with ONF copyright. Specification 1.1 had already laid the architectural foundation of the multi-table pipeline plus the group table, which versions 1.2 through 1.5.1 only extended without making fundamental changes.

## 4.0.1 OF 1.1 released on 2011-02-28: the last specification shepherded by Stanford (before ONF)

### Timeline and organisational context

The specification **OpenFlow Switch Specification Version 1.1.0** was released on 2011-02-28, fourteen months after 1.0.0. The wire protocol version is **0x02** (versus 0x01 in 1.0). The document is 56 pages (up from 42 pages in 1.0), reflecting a substantial expansion in scope.

The copyright page of the specification states:

> *"OpenFlow Switch Specification, Version 1.1.0 Implemented (Wire Protocol 0x02), February 28, 2011. Copyright (c) 2008, 2009, 2010, 2011 Stanford University. Permission is granted..."*

ONF is not yet visible in the copyright. The transfer of IP ownership took place on 2011-03-21 when ONF was founded, but specification 1.1 had been completed earlier. This is the reason 1.1 is "the last Stanford-shepherded specification" in OpenFlow history.

### Main motivation: solve the cross-product explosion

Part 3.1 §3.1.7 analysed the cross-product problem of the single-table 1.0 model: with N VLANs, M IP subnets, and K ACL rules, the table needs N times M times K flow entries. With realistic numbers (20 tenants, 100 subnets, 10 ACLs equals 20,000 entries) this exceeds the capacity of commodity TCAM hardware (1,500 to 4,000 entries).

A multi-table pipeline solves the problem with **nested lookup** instead of flat lookup. Table 0 matches dimension 1 (for example, VLAN), Table 1 matches dimension 2 (for example, IP subnet), and Table 2 matches dimension 3 (for example, ACL). The total entry count is N plus M plus K instead of N times M times K, linear instead of multiplicative.

With the same numbers as above: 20 plus 100 plus 10 equals 130 entries, a 150-times reduction compared to the flat approach. This easily fits inside a 4K-entry hardware table and leaves room for expansion.

### Other extensions in 1.1

In addition to the multi-table pipeline (the main contribution), 1.1 added:

- **Group table** (new): an abstraction for multicast, load balancing, and failover that does not require controller intervention.
- **Native MPLS support**: the actions PUSH_MPLS and POP_MPLS, plus match fields for the MPLS label, TC, and BoS.
- **Virtual ports**: the concept of a logical port that can map to multiple physical ports or to a function.
- **Flow entry matching extensions**: matching on MPLS fields and on the tunnel identifier.
- **Initial controller roles** (preliminary only, fully developed in 1.2): a master-slave model for high-availability controller deployment.

## 4.0.2 Multi-table pipeline semantics

### The packet flow through the pipeline

OF 1.1 specification §5.2 defines flow processing through the multi-table pipeline:

1. **Ingress**: a packet arrives at a switch port and is parsed into an internal representation (header fields are extracted).
2. **Table 0 lookup**: the packet is matched against flow entries in Table 0. On a match:
   - Instructions are executed (actions or `goto_table`).
   - If `goto_table = N` (with N greater than 0), the packet is forwarded to Table N.
   - If there is no `goto_table`, the pipeline stops and the action set is executed.
3. **Subsequent tables**: if `goto_table` was specified, the lookup repeats at the target table.
4. **Action set execution**: after the pipeline stops, all actions in the action set (accumulated through the pipeline) are executed.
5. **Egress**: the packet is forwarded out of a port by the OUTPUT action in the action set.

### Important constraint: monotonic progression

Specification §5.2 specifies: **the `goto_table` target must have a higher number than the current table**. Going backward or to the same table is not allowed. The reason is loop prevention. If backward goto were allowed, a packet could loop indefinitely between two tables (A to B to A to B to ...).

The monotonic constraint ensures the pipeline is an acyclic DAG (directed acyclic graph). Every packet traverses the pipeline in a finite number of tables (at most the table count) and cannot loop.

### Action set versus action list: the key difference

**Action list** (from OF 1.0, still valid in 1.1): a list of actions executed in the order they appear, like a sequence of statements in a programming language. The instruction `APPLY_ACTIONS` in 1.1 wraps an action list and executes it immediately on match.

**Action set** (new in 1.1): an unordered set of actions in which each action type has at most one instance (last-write-wins). The set is executed once at the end of the pipeline in the fixed order specified by §5.10 (`copy_ttl_in`, pop, push-MPLS or PBB or VLAN, `copy_ttl_out`, decrement_ttl, set, qos, group, output).

The instruction `WRITE_ACTIONS` adds an action to the action set; the action set accumulates through the pipeline. The instruction `CLEAR_ACTIONS` empties the action set.

The distinction matters because an action list allows intermediate effects (set a field then resubmit), while an action set lets the operator accumulate the final configuration intent across multiple tables and execute it once.

### Concrete example: multi-table with an action set

Scenario: Table 0 classifies by VLAN (tenant), Table 1 applies QoS, Table 2 applies routing, and Table 3 forwards.

```
Table 0:
  match: dl_vlan=100
  instructions: APPLY_ACTIONS(set_metadata=0x01), GOTO_TABLE=1

Table 1:
  match: metadata=0x01
  instructions: WRITE_ACTIONS(set_queue=5), GOTO_TABLE=2

Table 2:
  match: nw_dst=10.0.0.0/8
  instructions: WRITE_ACTIONS(set_dl_dst=00:11:22:33:44:55), GOTO_TABLE=3

Table 3:
  match: * (default)
  instructions: WRITE_ACTIONS(output=port 5)
```

Packet flow:
- Table 0 matches VLAN 100, sets metadata, goes to Table 1.
- Table 1 matches metadata, writes a `set_queue` action (accumulated in the action set), goes to Table 2.
- Table 2 matches the destination, writes a `set_dl_dst` action, goes to Table 3.
- Table 3 matches the default, writes an `output` action.
- The pipeline ends, the action set is executed: `set_queue`, then `set_dl_dst`, then `output`.

The total is four entries instead of one flat entry containing everything. Each table matches independently and is composable.

> **Key point:** the multi-table pipeline is an architectural shift, not just a feature addition. Every specification from 1.1 onward uses the pipeline plus action set model. OVS internally models the pipeline as an acyclic DAG and optimises execution through the megaflow cache. Anyone writing an OpenFlow controller has to understand the pipeline semantics to avoid common pitfalls (the WRITE versus APPLY confusion, action set ordering).

## 4.0.3 Instructions versus actions: the key distinction from 1.1 onward

### The instruction framework

Specification 1.1 §5.9 defines five instruction types:

**APPLY_ACTIONS**: execute an action list immediately, before going to the next table. This matches OF 1.0 behaviour (where every action executes immediately). Use case: an intermediate effect that has to be visible at a subsequent table match.

**WRITE_ACTIONS**: add an action to the action set (accumulated). This does not execute immediately. It executes at the end of the pipeline after every table has been processed. Use case: accumulate a final configuration intent across multiple tables.

**CLEAR_ACTIONS**: empty the current action set. Use case: rare, when a "reset" of an earlier table's decision is required.

**WRITE_METADATA**: set the metadata field (a 64-bit register that crosses tables). Metadata persists through the pipeline and is used to pass context between tables.

**GOTO_TABLE**: direct the packet to the next table. The target table must have a higher identifier than the current one.

### Fixed execution order

Specification §5.9.1 specifies: when a flow entry has multiple instructions, they execute in a fixed order:

1. METER (OF 1.3 and later)
2. APPLY_ACTIONS
3. CLEAR_ACTIONS
4. WRITE_ACTIONS
5. WRITE_METADATA
6. GOTO_TABLE

This order is fixed and the controller cannot override it. The controller selects which instructions to include; the execution order is deterministic.

### Important distinction

| Aspect | APPLY_ACTIONS | WRITE_ACTIONS |
|--------|---------------|---------------|
| Timing | Execute immediately, inline | Accumulate, execute at the end of the pipeline |
| Ordering | List order (as in 1.0) | Set order (specification fixed) |
| Visibility | Intermediate effect visible | Only the final result is visible |
| Use case | Table-local transformation | Accumulate cross-table configuration intent |

A common pitfall: a developer uses APPLY_ACTIONS with `set_field` when WRITE_ACTIONS is required, which results in inconsistent behaviour when multiple tables set the same field. Always check the configuration intent: if a single final effect is required, use WRITE.

## 4.0.4 Group table: four types for multicast, load balancing, and failover

### The group abstraction

Specification 1.1 §5.6 introduces the **group table**, an abstraction for scenarios that a single flow entry cannot express:

- Multicast (replicate a packet to many ports).
- Load balancing (choose one of several next-hops by hash).
- Failover (switch to a backup if the primary is down).
- Shared action set (many flow entries share a single action sequence).

The group table contains group entries; each entry has a `group_id`, a type, and a list of buckets. A bucket is a set of actions plus an optional weight, watch_port, or watch_group.

The action `GROUP` references a `group_id` from a flow entry. When a packet matches a flow entry that has a `GROUP` action, the packet is submitted to the group, and the buckets execute according to the type.

### The four group types

**Type ALL** (multicast, flooding):
- Execute ALL buckets for each packet.
- Each bucket produces a clone of the packet with different actions.
- Use case: multicast group, mirror traffic to multiple ports, BUM replication in an overlay.
- Example: bucket 1 = `output:port 2`, bucket 2 = `output:port 3`, bucket 3 = `output:port 4`. The packet is cloned to three ports.

**Type SELECT** (load balancing):
- Execute exactly ONE bucket, chosen by the switch implementation.
- The algorithm is implementation-defined and is typically a hash (for example, hash on the 5-tuple modulo bucket count).
- A bucket may carry an optional weight for weighted load balancing.
- Use case: ECMP multi-path routing, load balancing across backend servers.
- Example: bucket 1 weight=50 `output:port 2`, bucket 2 weight=50 `output:port 3`. A packet takes one path or the other 50/50.

**Type INDIRECT** (a single shared bucket):
- Has only ONE bucket, executed for every packet.
- Conceptually similar to a direct action list, with the benefit that many flow entries can reference the same group.
- Shared action set: update the action once at the group level instead of updating N flow entries.
- Use case: a common next-hop shared by many destinations; centralise policy change.
- Example: an indirect group with bucket = `set_dl_dst` plus `output:uplink`; 1,000 flow entries for 1,000 destinations all reference this group; when the next-hop MAC changes, only the group bucket needs to be updated.

**Type FAST_FAILOVER** (sub-10 ms reroute):
- Execute the first bucket whose `watch_port` or `watch_group` status is UP (alive).
- The switch monitors the port or group status and switches buckets automatically when the primary is down.
- Failover time is bounded by the switch's ability to detect the down status (typically less than 10 ms with carrier-loss detection on the port).
- Use case: fast reroute in backbone link protection, the MPLS FRR equivalent without controller involvement.
- Example: bucket 1 `watch_port=primary_port output:primary_nexthop`, bucket 2 `watch_port=backup_port output:backup_nexthop`. When the primary port goes down, the switch automatically switches to bucket 2.

### FAST_FAILOVER: sub-10 ms reroute that bypasses the controller

FAST_FAILOVER is especially important. In legacy routing, failover depends on routing-protocol convergence (OSPF LSA flooding, SPF recomputation): typically 200 ms to 2 s. With OpenFlow controller-based failover, the timing depends on the controller round trip plus reinstalling the flow: typically 50 ms to 500 ms.

FAST_FAILOVER lets the switch reroute itself in microseconds. Detecting the port-down condition (typically done by a hardware port-status interrupt in less than 1 ms) is enough; the switch flips the active bucket and packets start flowing through the backup path. There is no controller notification and no new flow installation.

Comparison:
- OSPF failover: 200 ms to 2,000 ms.
- BGP failover: 10 to 180 seconds (without fast-feature support).
- OpenFlow controller-based failover: 50 ms to 500 ms.
- OpenFlow FAST_FAILOVER: less than 10 ms typically.

The compelling use cases: an ISP backbone with a link-protection requirement, a hyperscale data centre with a high-availability SLA, and WAN SDN (Google B4) using FAST_FAILOVER for link failover.

> **Key point:** the group table, especially FAST_FAILOVER, is the OpenFlow feature that competes directly with MPLS Fast Reroute (FRR) for WAN and service-provider use cases. Group SELECT replaces ECMP routing. Group ALL replaces IGMP multicast snooping. Each group type addresses a specific pain point of the legacy protocols and justifies the OpenFlow value proposition for complex data-centre and WAN scenarios.

## 4.0.5 Native MPLS support

### MPLS actions and match fields

Before 1.1, OpenFlow did not support MPLS; implementing MPLS with OF 1.0 required a vendor extension or a custom action. Version 1.1 officially added MPLS to the core specification:

**MPLS actions (five types)**:

- **PUSH_MPLS**: add an MPLS label to the front of the packet (outermost). The outer Ethertype is set to 0x8847 (MPLS unicast) or 0x8848 (MPLS multicast).
- **POP_MPLS**: remove the outermost MPLS label. If no more labels remain, the Ethertype reverts.
- **SET_MPLS_TTL**: set the TTL of the outermost MPLS label.
- **DEC_MPLS_TTL**: decrement the TTL by one. If the TTL reaches zero, the packet is dropped.
- **COPY_TTL_IN / COPY_TTL_OUT**: copy the TTL between the IP and MPLS headers for proper TTL propagation.

**MPLS match fields (three types)**:

- **mpls_label**: 20-bit label value.
- **mpls_tc** (Traffic Class): 3-bit, corresponds to the EXP bits in the MPLS header.
- **mpls_bos** (Bottom of Stack): 1-bit, indicates whether this is the last label in the stack.

### Significance: OpenFlow enters the WAN and carrier network

MPLS (RFC 3031, Rosen et al., January 2001) is the de-facto transport for the service-provider backbone. Network operators build MPLS cores for VPN services (L2VPN, L3VPN, MPLS Traffic Engineering), and every core-router vendor (Cisco, Juniper, Nokia Alcatel-Lucent) supports MPLS natively.

Before 1.1, OpenFlow was not seen as viable for WAN use cases because of the missing MPLS support. Version 1.1 opened the door: an operator can deploy an OpenFlow switch in an MPLS backbone and have the controller drive label distribution instead of using LDP or RSVP-TE distributed protocols.

The Google B4 WAN (paper at SIGCOMM 2013, Jain et al.) is the first production deployment to use an OpenFlow-like approach (not strictly the OpenFlow protocol, but conceptually similar) for WAN traffic engineering. B4 implements MPLS fast reroute (FRR) through the equivalent of the OpenFlow FAST_FAILOVER group.

### Use case: ingress classify, then MPLS encapsulation

A common pattern in WAN deployment:

Table 0 classifies traffic by (source, destination):
```
match: in_port=X, nw_dst=10.1.0.0/16
instructions: APPLY_ACTIONS(push_mpls:0x8847, set_mpls_label=100, set_mpls_ttl=255),
              GOTO_TABLE=1
```

Table 1 forwards by MPLS label:
```
match: mpls_label=100
instructions: APPLY_ACTIONS(output:port_to_next_hop)
```

A packet from a local tenant entering through ingress port X with destination 10.1.0.0/16 receives label 100 and is then forwarded by label. This is a simplified MPLS VPN pattern in which the controller configures the labels instead of an LDP protocol.

## 4.0.6 Use case: a multi-tenant classifier that maps to a tunnel

### Multi-tenant data-centre deployment pattern

A multi-tenant data centre with 20 tenants, each tenant having its own VLAN traffic, has to forward across a tunnel (a VXLAN equivalent expressed in OpenFlow). This use case illustrates the power of the multi-table pipeline.

**Flat approach with OF 1.0 (single table)**:

```
# Tenant 1 traffic to tunnel 1
match: dl_vlan=100, nw_dst=10.1.0.0/16     -> action: set_tunnel=1, output:port_tun_1
match: dl_vlan=100, nw_dst=10.1.1.0/24     -> action: set_tunnel=1, output:port_tun_1
... (M destinations per tenant) ...
match: dl_vlan=100, nw_dst=10.1.M.0/24     -> action: set_tunnel=1, output:port_tun_1
# Tenant 2 traffic to tunnel 2
match: dl_vlan=200, nw_dst=10.2.0.0/16     -> action: set_tunnel=2, output:port_tun_2
... ... ...
# Tenant 20 traffic to tunnel 20
match: dl_vlan=4000, nw_dst=10.20.M.0/24   -> action: set_tunnel=20, output:port_tun_20
```

Total entries: 20 tenants times M destinations equals 20M entries. With M = 100, the result is 2,000 entries.

**Multi-table approach with OF 1.1**:

```
Table 0 (classify tenant):
  match: dl_vlan=100  -> WRITE_METADATA=0x01, GOTO_TABLE=1
  match: dl_vlan=200  -> WRITE_METADATA=0x02, GOTO_TABLE=1
  ...
  match: dl_vlan=4000 -> WRITE_METADATA=0x14, GOTO_TABLE=1

Table 1 (route by destination):
  match: nw_dst=10.1.0.0/16 -> WRITE_ACTIONS(set_tunnel=1, output:port_tun_1)
  match: nw_dst=10.1.1.0/24 -> WRITE_ACTIONS(set_tunnel=1, output:port_tun_1)
  ... ...
```

Wait, the multi-table version is not yet sufficient. Table 1 also has to check metadata so that there is no conflict between tenants. Correction:

```
Table 0 (classify tenant):
  match: dl_vlan=100 -> WRITE_METADATA=0x01, GOTO_TABLE=1

Table 1 (tenant-aware routing):
  match: metadata=0x01, nw_dst=10.1.0.0/16 -> WRITE_ACTIONS(set_tunnel=1, output:port_tun_1)
  match: metadata=0x02, nw_dst=10.2.0.0/16 -> WRITE_ACTIONS(set_tunnel=2, output:port_tun_2)
```

That is still multiplicative. A better approach is:

```
Table 0 (classify and tag):
  match: dl_vlan=100 -> WRITE_ACTIONS(set_tunnel=1), GOTO_TABLE=1

Table 1 (route by IP):
  match: nw_dst=10.1.0.0/16 -> WRITE_ACTIONS(output:port_tun_1)
```

This is the additive approach: N tenants plus M destinations equals 20 plus 100 entries equals 120. Compared with the 2,000-entry flat version, the saving is 94 per cent.

### Concrete comparison

| Metric | Flat (1.0) | Multi-table (1.1) |
|--------|-----------|-------------------|
| Total entries | N times M = 2,000 | N plus M = 120 |
| Scale formula | O(N times M) | O(N + M) |
| Hardware table fit (4K) | About 2 tenants max | More than 30 tenants |
| Policy change: add 1 tenant | M entries | 1 entry in Table 0 |
| Policy change: add 1 subnet | N entries | 1 entry in Table 1 |

The multi-table approach scales independently per dimension, makes policy change local, and is easier to maintain.

This is concrete evidence for why the multi-table pipeline is an architectural feature rather than a nice-to-have. At data-centre multi-tenant scale, single-table 1.0 is not viable; multi-table 1.1 and later is a requirement.

## Guided Exercise: implement a 3-stage multi-table pipeline with OVS (audit P4.B4.1 lab)

**Objective:** practise implementing the OpenFlow 1.1 multi-table architecture on OVS, and verify that the cross-product reduces from O(N times M) to O(N + M).

**Environment:** Ubuntu 22.04 plus OVS 2.17.9 plus three veth pairs simulating tenant taps.

**Steps:**

1. Create an OVS bridge with OF 1.3 (OVS implements OF 1.1 and later features through the OF 1.3 protocol):
   ```bash
   $ sudo ovs-vsctl add-br br-mt -- set bridge br-mt protocols=OpenFlow13
   $ for i in 1 2 3; do
       sudo ip link add veth-t$i type veth peer name peer-t$i
       sudo ip link set veth-t$i up; sudo ip link set peer-t$i up
       sudo ovs-vsctl add-port br-mt veth-t$i
     done
   ```
2. Install the multi-table pipeline. Table 0 classifies the tenant, Table 1 routes, Table 2 outputs:
   ```bash
   # Table 0: classify tenant by VLAN, set metadata
   $ sudo ovs-ofctl -O OpenFlow13 add-flow br-mt \
       "table=0,priority=100,dl_vlan=100,actions=write_metadata:0x01,goto_table:1"
   $ sudo ovs-ofctl -O OpenFlow13 add-flow br-mt \
       "table=0,priority=100,dl_vlan=200,actions=write_metadata:0x02,goto_table:1"

   # Table 1: route by destination, set tunnel
   $ sudo ovs-ofctl -O OpenFlow13 add-flow br-mt \
       "table=1,priority=100,metadata=0x01,nw_dst=10.1.0.0/16,\
        actions=set_field:1->tun_id,goto_table:2"

   # Table 2: output
   $ sudo ovs-ofctl -O OpenFlow13 add-flow br-mt \
       "table=2,priority=100,tun_id=1,actions=output:2"
   ```
3. Verify the pipeline entry count (formula O(N + M) versus O(N times M)):
   ```bash
   $ sudo ovs-ofctl -O OpenFlow13 dump-flows br-mt | wc -l
   # Expected: about 5 to 6 rules (2 tenants plus 1 route plus 1 output)
   ```
4. Trace a packet to verify the flow:
   ```bash
   $ sudo ovs-appctl ofproto/trace br-mt \
       "in_port=veth-t1,dl_vlan=100,ip,nw_dst=10.1.0.5"
   # Observe: Table 0 match, Table 1, Table 2, output
   ```

**POE:**
- **Predict:** with 10 tenants and 100 destinations, how many rules does single-table 1.0 need? How many does multi-table 1.1 need?
- **Observe:** single-table: 10 times 100 equals 1,000 rules. Multi-table: 10 plus 100 equals 110 rules. A reduction of nine times.
- **Explain:** the multi-table model is additive instead of multiplicative. It scales linearly with the sum of dimensions, not with the product.

**Cleanup:**
```bash
$ sudo ovs-vsctl del-br br-mt
$ for i in 1 2 3; do sudo ip link del veth-t$i; done
```

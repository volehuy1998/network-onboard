## 13.6.1 Problem: gateway chassis SPOF

Part 13.5 analysed the `chassisredirect` Port_Binding, which designates one chassis as the active gateway for north-south NAT. If that chassis dies:
- North-south traffic fails (external traffic does not reach the VM, the VM does not reach the internet).
- SNAT connections break (the kernel-conntrack NAT state on that chassis is lost).
- Manual failover requires admin intervention, leading to MTTR of 10 to 60 minutes.

**HA_Chassis_Group** solves this: it groups 2 to 5 chassis with priority and uses OVN plus BFD to fail over automatically. Target: MTTR less than 2 seconds, automatic, no admin action needed.

## 13.6.2 HA_Chassis_Group schema

NBDB schema:

```
HA_Chassis_Group:
  name          , group identifier
  ha_chassis    , set<HA_Chassis UUID>
  external_ids  , map for CMS annotation

HA_Chassis:
  chassis_name  , reference to Chassis.name in SBDB
  priority      , 0-32767, higher = preferred
  external_ids  , map for CMS annotation
```

### Configuration

```bash
[admin]# ovn-nbctl ha-chassis-group-add hcg-external
[admin]# ovn-nbctl ha-chassis-group-add-chassis hcg-external chassis-1 100
[admin]# ovn-nbctl ha-chassis-group-add-chassis hcg-external chassis-2 50
[admin]# ovn-nbctl ha-chassis-group-add-chassis hcg-external chassis-3 30
[admin]# ovn-nbctl lrp-set-ha-chassis-group lrp-external hcg-external
```

Priority semantics:
- chassis-1 (100), the preferred active.
- chassis-2 (50), standby 1 (activates when chassis-1 fails).
- chassis-3 (30), standby 2 (activates when both 1 and 2 fail).

### HA_Chassis_Group versus Gateway_Chassis (legacy)

OVN releases before 21.03 used the `Gateway_Chassis` schema, which was simpler but had no built-in BFD. HA_Chassis_Group replaces and extends it:

| Feature | Gateway_Chassis (legacy) | HA_Chassis_Group (current) |
|---------|--------------------------|----------------------------|
| BFD integration | Manual (operator script). | Automatic (OVN-managed). |
| Preemption control | None. | `ha_chassis_group_options:preemption`. |
| Multi-group support | One group per LRP. | Multiple groups can be referenced across LRPs. |
| Deprecation | OVN 22.03 and later, deprecated. | OVN 21.03 and later, default. |

If you still use Gateway_Chassis, migrate to HA_Chassis_Group.

## 13.6.3 BFD (RFC 5880), sub-second failure detection

### Protocol fundamentals

BFD is a **lightweight hello protocol** that detects peer failure with millisecond granularity. It runs over UDP port 3784 (single-hop session).

Packet format (RFC 5880 §4.1):

```
Ver (3 bit) | Diag (5 bit) | State (2 bit) | Flags (6 bit)
Detect Mult (8 bit) | Length (8 bit)
My Discriminator (32 bit)
Your Discriminator (32 bit)
Desired Min TX Interval (32 bit, microseconds)
Required Min RX Interval (32 bit, microseconds)
Required Min Echo RX Interval (32 bit)
```

Key fields:
- **State**: 2 bits, 4 values: AdminDown (0), Down (1), Init (2), Up (3).
- **Detect Mult**: multiplied by the peer's TX interval to give the detection time.
- **My/Your Discriminator**: a 32-bit session ID that helps demultiplex multiple sessions.

### State machine (3 states plus init)

```
  ┌─────────────┐
  │  AdminDown  │ ← admin disable
  └──────┬──────┘
         │ enable
         ▼
  ┌─────────────┐ no packets received X ms
  │    Down     │ ────────────────────────┐
  └──────┬──────┘                         │
         │ hear from peer                 │
         ▼                                │
  ┌─────────────┐ peer says Up           │
  │    Init     │ ──────────────────────┐│
  └──────┬──────┘                       ││
         │ peer says Init/Up            ││
         ▼                              ▼▼
  ┌─────────────┐ detect timeout  ┌─────────────┐
  │     Up      │ ───────────────▶│    Down     │
  └─────────────┘                 └─────────────┘
```

### Timing math

**Default OVN BFD:**
- `min_tx_interval=500000µs` (500 ms).
- `min_rx_interval=500000µs` (500 ms).
- `detect_multiplier=3`.
- Detection time equals `min_tx_interval × detect_multiplier`, that is 1500 ms.

**Fast timer (sub-second):**
- `min_tx_interval=100000µs` (100 ms).
- `min_rx_interval=100000µs` (100 ms).
- `detect_multiplier=3`.
- Detection time equals 300 ms.

**Extremely fast (production HA trading):**
- `min_tx_interval=10000µs` (10 ms).
- `detect_multiplier=3`.
- Detection equals 30 ms.
- **CPU and network cost:** each session generates 100 packets per second times N sessions. 100 sessions at 100 packets per second equals 10K packets per second for BFD alone. Only acceptable on dedicated HA chassis.

### Configure BFD in OVN

```bash
[admin]# ovn-nbctl set NB_Global . options:bfd-min-rx=100 \
                                          options:bfd-min-tx=100 \
                                          options:bfd-mult=3
```

This applies across every HA_Chassis_Group. Per-group override goes through `HA_Chassis_Group_Options`:

```bash
[admin]# ovn-nbctl set HA_Chassis_Group hcg-external \
         options:bfd-min-rx=50 options:bfd-min-tx=50
```

## 13.6.4 Anatomy Template A, `ovs-appctl bfd/show`

**When to run:** after configuring an HA_Chassis_Group, verify that the BFD session has been established and is UP between chassis members. This is also the first debug command when a failover does not trigger.

**Command and output** *(reproduced from OVN 22.03.8 on chassis-1, with two BFD sessions to chassis-2 and chassis-3)*:

```shell
$ sudo ovs-appctl -t ovs-vswitchd bfd/show

Interface ovn-chassis-2-0
        BFD State              : Up
        Session State          : Up
        Forwarding             : true
        Diagnostic             : No Diagnostic
        Detect Multiplier      : 3
        Concatenated Path Down : false
        Local Discriminator    : 0x8123a5f0
        Remote Discriminator   : 0x7f2b1944
        Local Min TX Interval  : 500 ms
        Local Min RX Interval  : 500 ms
        Remote Min RX Interval : 500 ms
        Poll Seq In Progress   : false
        Remote State           : Up
        Local Flag             : C
        Remote Flag            : None
        Loss Detect Timeout    : 1500 ms

Interface ovn-chassis-3-0
        BFD State              : Up
        Session State          : Up
        Forwarding             : true
        Detect Multiplier      : 3
        Local Discriminator    : 0x8124a5f1
        Remote Discriminator   : 0x9a3c4021
        Loss Detect Timeout    : 1500 ms
```

**9-attribute anatomy:**

| Field | Sample value | Meaning | Red flag if |
|-------|--------------|---------|-------------|
| `Interface` | `ovn-chassis-2-0` | Tunnel port name. | Missing interface means the tunnel port is gone and BFD cannot be set up. |
| `BFD State` | `Up`, `Down`, `Init` | Overall BFD state. | Persistent `Down` means the peer chassis is unreachable. |
| `Session State` | `Up` | Session-level state. | `Init` after 10 seconds means one-way BFD (the peer does not respond). |
| `Forwarding` | `true` | BFD permits forwarding traffic. | `false` means ovn-controller is blocking traffic because BFD is down. |
| `Detect Multiplier` | `3` | Number of missed packets before DOWN. | Below 3 is aggressive detection with a higher false-positive risk. |
| `Local Discriminator` | `0x8123a5f0` | Local-side session ID. | Changing on every restart means a BFD reset. |
| `Remote Discriminator` | `0x7f2b1944` | Peer-side session ID. | `0x00000000` means BFD has not received any packet from the peer. |
| `Local Min TX Interval` | `500 ms` | The rate at which BFD packets are sent. | A mismatch with the peer's `min_rx_interval` triggers negotiation. |
| `Remote Min RX Interval` | `500 ms` | The peer's acceptable RX rate. | Caps the local TX rate. |
| `Loss Detect Timeout` | `1500 ms` | tx times multiplier equals detection time. | The peer flaps if the timeout is too short. |
| `Poll Seq In Progress` | `false` | A poll sequence is in progress (parameter change). | Stuck `true` means parameter negotiation failed. |

**Failure scenarios:**

- **`Session State: Down` persistent plus `Remote Discriminator: 0x00000000`**: the peer chassis is not sending BFD. Causes: ovn-controller crashed on the peer, tunnel port MTU is blocking BFD, firewall is blocking UDP 3784.
- **`Session State: Init` does not transition to Up**: one-way BFD; local is sending and the peer is receiving, but the peer's reply does not reach local. Asymmetric firewall rule or asymmetric tunnel MTU.
- **`Loss Detect Timeout` does not match the expected configuration**: BFD negotiation fell back to the slowest. Check the peer's `min_rx_interval`; if it is greater than local's `min_tx_interval`, BFD uses the slower rate.
- **`Forwarding: false` plus `BFD State: Up`**: the `bfd_slow_timer_expired` flag is set. Check `ovs-appctl bfd/show -v` for detail.

## 13.6.5 Failover workflow, step by step

Scenario: chassis-1 active (priority 100) crashes, chassis-2 (priority 50) is standby.

```
T+0ms      chassis-1 kernel panic and reboot
T+0ms      BFD session chassis-2 ↔ chassis-1 stops receiving packets from c1
T+500ms    missed 1 BFD packet
T+1000ms   missed 2 BFD packets
T+1500ms   chassis-2 BFD state transitions UP → DOWN (detect timeout 3 × 500ms)
T+1500ms   ovn-controller chassis-2 update HA_Chassis_Group:
           - ovn-sbctl update HA_Chassis_Group active=chassis-2
T+1600ms   SBDB Raft commit (3-node latency 50-100ms)
T+1650ms   ovn-northd regenerate Logical_Flow:
           - Port_Binding chassisredirect chassis field → chassis-2
T+1700ms   ovn-controller chassis-2 + chassis-3 subscribe change
T+1750ms   chassis-2 install OpenFlow rule for NAT:
           - ARP announce 192.0.2.1 from chassis-2's br-ex
T+1800ms   upstream switch/router update ARP cache
T+1850ms   Traffic north-bound flows through chassis-2
T+2000ms   VM ping 8.8.8.8 resumes
```

**Total typical downtime: 1.5 to 2.5 seconds** with default BFD. With a 100 ms fast timer: 400 to 600 ms downtime.

### Recovery and preemption

When chassis-1 recovers (the reboot completes and ovn-controller starts):
1. Re-register the Chassis row in SBDB.
2. BFD resumes.
3. **If preemption is enabled (default):** chassis-1 priority 100 is greater than chassis-2 priority 50, so traffic fails back to chassis-1.
4. **If preemption is disabled:** chassis-2 keeps the active role and chassis-1 stays standby.

**Read the preemption configuration:**
```bash
$ sudo ovn-nbctl get HA_Chassis_Group hcg-external options:preemption
```

Default: `true` (preemption on). Production best practice: set `false` to avoid flapping if chassis-1 is unstable.

```bash
[admin]# ovn-nbctl set HA_Chassis_Group hcg-external options:preemption=false
```

> **Key point:** preemption off versus on is an important operational decision. Preemption on means the system recovers automatically to the preferred chassis (maintaining the priority invariant and operational visibility). Preemption off means stability is preserved after a failover (avoiding flapping if the chassis keeps restarting). Rule of thumb: preemption **on** in dev or staging (visibility), **off** in production (stability). Backup: a monitoring alert when preemption is off but the active chassis is not the highest-priority chassis can signal that ops intervention is required.

## 13.6.6 Common failure modes

> **Common misconception:** *"BFD always detects a chassis failure within 100 ms."*
>
> **Reality:** BFD detects within **`min_tx × detect_multiplier` × 2 in the worst case**. If the chassis fails immediately after sending a BFD packet, the peer must wait one more full TX interval. In practice, detection time is 1 to 2 times the calculated value. On top of that, the SBDB Raft commit (50 to 100 ms), ovn-northd regenerate (50 to 200 ms), ovn-controller apply (50 to 100 ms), and ARP re-announce (100 to 500 ms depending on the upstream switch) add up. Total end-to-end failover with default BFD is 1.5 to 3 seconds. Do not promise 100 ms downtime with default BFD; that figure is only BFD detection, not end-to-end recovery.

**Failure 1, BFD flapping caused by network jitter.**
Signal: `ovs-appctl bfd/show` shows BFD state toggling UP and DOWN every 10 to 30 seconds. The ovn-sbctl log shows the HA_Chassis_Group active chassis flipping continuously.
Causes: underlay network latency spike (exceeds `min_rx × mult`), chassis CPU overloaded (cannot process BFD packets in time).
Fix: (1) increase `detect_multiplier` from 3 to 5 (tolerate more missed packets); (2) disable preemption; (3) dedicate the gateway chassis (do not mix workloads).

**Failure 2, one-way BFD (session stuck at Init).**
Signal: local `BFD State: Init` does not transition to Up after 30 seconds.
Causes: firewall blocks UDP 3784 outbound or inbound asymmetrically. Asymmetric tunnel MTU.
Fix: `tcpdump -i tunnel-interface udp port 3784`; if you only see one-way packets, fix the firewall or MTU issue.

**Failure 3, BFD Up but Forwarding false.**
Signal: `BFD State: Up` plus `Forwarding: false`.
Causes: slow timer expired (BFD parameter change in flight). Local flag `C` without remote flag acknowledge.
Fix: restart ovn-controller on the chassis to reset the BFD session.

**Failure 4, failover does not trigger even though the chassis crashed.**
Signal: chassis-1 crashed but traffic still routes through the chassis-1 shadow (the chassisredirect Port_Binding does not update).
Causes: ovn-controller on chassis-2 crashed, or its OVSDB connection died.
Fix: run `ovn-appctl -t ovn-controller connection-status` on chassis-2.

## 13.6.7 Production best practices

1. **At least 3 HA_Chassis members**, 5 for mission-critical clusters. 2 members create a split-brain risk under network partition.
2. **Disable preemption** in production (`preemption=false`).
3. **Dedicate the gateway chassis**, do not mix VM workloads (avoid CPU contention with BFD processing).
4. **Monitor BFD state** through Prometheus plus `ovs_exporter` (port 9475). Alert rule: session DOWN for more than 30 seconds is critical.
5. **Quarterly failover drill**: kill ovn-controller on chassis-1 during the maintenance window, verify migration in less than 2 seconds and correct fail-back (or non-fail-back, depending on preemption).
6. **Tune BFD per workload**: development lab uses default (500 ms), production uses 250 ms, HFT or trading uses 100 ms with a dedicated CPU core.
7. **Log BFD transitions**: schedule `ovn-appctl bfd/show --verbose` through cron, log 24-hour rolling for post-incident review.

## 13.6.9 Core takeaways

First, HA_Chassis_Group is the OVN abstraction for gateway failover; it replaces the legacy Gateway_Chassis schema with auto-managed BFD integration.

Second, BFD (RFC 5880) detects sub-second with a fixed packet format and a 3-state state machine (INIT, UP, DOWN). Detection time equals `min_tx × detect_multiplier`. Default 1500 ms, fast timer 300 ms, extreme 30 ms with CPU cost.

Third, end-to-end failover time equals BFD detection plus SBDB Raft commit plus northd regenerate plus ovn-controller apply plus ARP announce, around 1.5 to 3 seconds with the default and 400 to 600 ms with a fast timer. Do not promise 100 ms; that figure is BFD only.

Fourth, preemption off in production avoids flapping. Fail-back is manual or auto-scheduled in a maintenance window.

Fifth, 7 best practices: at least 3 members, preemption off, dedicate the gateway, Prometheus BFD monitor, quarterly drill, BFD tuned per workload, log transitions.

Sixth, 4 common failure modes: BFD flapping (jitter), one-way BFD (firewall or MTU asymmetric), Forwarding false stuck (parameter change), failover does not trigger (ovn-controller crash). Each has a clear signature and a clear fix.

## 13.6.W Five NBDB concepts for HA gateway, in depth (HA_Chassis_Group, BFD, gateway_chassis option, ha-priority semantics, chassisredirect Port_Binding type)

> This subsection takes a deep dive into five core concepts related to OVN HA gateway. Each concept gets a full 20-perspective treatment (concept through to comparison with other systems). The first two (`HA_Chassis_Group`, `BFD`) are NBDB tables; the last three (the LRP `gateway_chassis` option, the `priority` semantics on HA_Chassis, the `chassisredirect` type of Port_Binding in SBDB) are logical entities tied to HA gateway configuration. The schema is verified upstream against `ovn-nb.ovsschema` branch-22.03; the source code cross-link is `northd/ha-chassis.c`, `northd/northd.c::build_chassis_redirect_*`, `controller/bfd.c`, RFC 5880.

### 13.6.W.1 The `HA_Chassis_Group` table in NBDB, defining the candidate-chassis group for active gateway

#### Core concept

`HA_Chassis_Group` is a **resident table in the OVN Northbound database** that describes a **group of candidate chassis for the active role of a distributed gateway port (DGP)**, where each member is an `HA_Chassis` row carrying its own `priority` value.
Each row in this table represents one logical group, for example `hcg-external` containing three chassis in three different racks all eligible to be the gateway for tenant north-south traffic.
Type: an **OVSDB root table** (`isRoot=true`), with a **unique index on the `name` column**, three columns total (`name`, `ha_chassis`, `external_ids`).
The `ha_chassis` column is a **strong reference** to the child `HA_Chassis` rows, which means that when an `HA_Chassis_Group` row is deleted, the `HA_Chassis` rows belonging to the group are cascade-deleted by OVSDB semantics.
The key difference compared with the legacy `gateway_chassis` column (placed directly on `Logical_Router_Port`): `HA_Chassis_Group` allows a single group to be referenced by multiple LRPs simultaneously and integrates BFD managed by northd (no external scripts needed).

#### History and motivation

`HA_Chassis_Group` was introduced in OVN 2.12 (2019) when the Lucas Alvares Gomes and Numan Siddique team (Red Hat) refactored the legacy "redirect-chassis" plus "gateway_chassis" model.
Before OVN 2.12, each LRP had a `gateway_chassis` column holding a list of (chassis-name, priority); failover was driven by an SBDB connection-timeout heuristic (10 to 30 seconds) with no standardised BFD.
OVN 2.12 added `HA_Chassis_Group` so the "group definition" is decoupled from "attaching to an LRP", and integrated BFD per RFC 5880 for sub-second detection.
OVN 21.06 marked `gateway_chassis` legacy and recommended migrating to `HA_Chassis_Group`. OVN 22.03 stabilised the 3 columns (verified against branch-22.03 schema). OVN 24.03 improved northd incremental compile when an HA_Chassis_Group changes (only the related chassisredirect Port_Bindings are recomputed).

#### Position in the architecture

The table sits in the **NBDB layer** as input, a **root table** that lives independently of Logical_Router and Logical_Switch.
Reference graph: `Logical_Router_Port.ha_chassis_group` (NBDB) is a weak back-reference to `HA_Chassis_Group`; `HA_Chassis_Group.ha_chassis` (strong) refers to `HA_Chassis`; `HA_Chassis.chassis_name` (string) is a soft reference to `Chassis.name` in SBDB.
Scope: **cluster-wide**. A single HA_Chassis_Group can be referenced by multiple LRPs of multiple tenants (sharing a pool of gateway chassis).
ovn-northd compiles the NBDB HA_Chassis_Group into the corresponding SBDB HA_Chassis_Group plus HA_Chassis rows and, at the same time, updates the `ha_chassis_group` column in the `chassisredirect` Port_Binding row of the LRP that references the group (covered in §13.5.1d.4 on the SB side).

#### Role and responsibilities

The table carries four roles: (1) **Abstract the gateway-chassis pool** into one named entity reusable by multiple LRPs; (2) **Hold the chassis-to-priority mapping** that decides which chassis is active by default; (3) **Be the BFD attachment point** by being compiled into SBDB BFD rows automatically for each chassis pair in the group; (4) **Decouple HA policy from topology**, so the admin can change group membership without touching the LRP.

#### Why it was created (design motivation)

Before 2.12, the `gateway_chassis` model that placed the chassis list directly on the LRP had three drawbacks: (a) **duplication** if multiple LRPs use an identical chassis pool, each LRP must redeclare the list; (b) **no standardised BFD**, so failover relied on SBDB timeout and lacked dataplane accuracy; (c) **rigid schema**, so adding new features (preemption tuning, BFD parameter override) required extending the `Logical_Router_Port` schema.
`HA_Chassis_Group` is designed to separate concerns: the group definition is a first-class entity, and the LRP only references the group through one column.
Feature extensions (BFD parameters, preemption) go into `HA_Chassis_Group.external_ids` or an options field without touching the LRP schema.

#### What problem it solves

(1) **Centralised gateway-pool management** instead of scattering across each LRP; (2) **Runtime change of group membership** (add or remove HA_Chassis members) without restarting the LRP or flapping traffic; (3) **Reusable across tenants**, one group `hcg-external` can serve all north-south LRPs of 50 tenants; (4) **Automatic BFD integration**, so northd knows which chassis pairs to compile into BFD sessions; (5) **Deterministic failover** based on the priority of HA_Chassis members, no admin script needed.

#### Importance

This is a **tier 2 medium** NBDB table for any cluster with a distributed gateway router. Reasons: (a) every OpenStack Neutron deployment with the networking-ovn driver and an external network produces an HA_Chassis_Group automatically through the plugin; (b) every deployment with north-south HA needs this group; (c) deployments that only have east-west routing do not use it (a distributed LR does not need a redirect-chassis). An engineer who does not understand HA_Chassis_Group cannot debug "north-south traffic blackholed after one chassis went down" or "fail-back to the preferred chassis did not happen".

#### How it works

A row in HA_Chassis_Group goes through six phases: (a) **Group declaration**: `ovn-nbctl ha-chassis-group-add hcg-external` inserts a new row with name=`hcg-external` and ha_chassis=[]; (b) **Add a member**: `ovn-nbctl ha-chassis-group-add-chassis hcg-external chassis-1 100` creates an HA_Chassis row (chassis_name=`chassis-1`, priority=100) and appends its UUID to the ha_chassis column; (c) **Attach to LRP**: `ovn-nbctl lrp-set-ha-chassis-group lrp-external hcg-external` sets the `ha_chassis_group` column of the Logical_Router_Port; (d) **Northd compile**: the function `build_ha_chassis_group_table` (`northd/ha-chassis.c`) copies HA_Chassis_Group plus HA_Chassis into SBDB; (e) **Compile chassisredirect**: northd produces an SBDB Port_Binding `cr-lrp-external` with type=`chassisredirect`, attaching the `ha_chassis_group` column that references the SBDB HA_Chassis_Group; (f) **Compile BFD**: northd produces SBDB BFD rows for each chassis pair in the group (full mesh or star depending on topology), inserted into the `bfd` column of the chassisredirect Port_Binding; (g) **Runtime claim**: the ovn-controller on the chassis with the highest priority and BFD up automatically claims ownership of the chassisredirect Port_Binding.

#### Column classification (verified branch-22.03)

Three columns total: `name` (string, unique index, required, cluster-wide identifier), `ha_chassis` (set of UUID strong references to HA_Chassis, min=0 max=unlimited), `external_ids` (map for CMS metadata such as `neutron:availability_zone`).
The child table `HA_Chassis` (isRoot=false because it is a child under a strong reference from HA_Chassis_Group) has three columns: `chassis_name` (string, soft match against the SBDB Chassis.name), `priority` (integer, minInteger=0, maxInteger=32767, higher equals preferred), `external_ids` (metadata map).

#### How an operator masters this

Four skills: (1) **Group CRUD**: `ovn-nbctl ha-chassis-group-add/del`, `ha-chassis-group-list`, `ha-chassis-group-add-chassis/del-chassis`; (2) **Attach a group to an LRP**: `ovn-nbctl lrp-set-ha-chassis-group <lrp> <group>`; (3) **Inspect**: `ovn-nbctl --columns=name,ha_chassis,external_ids list HA_Chassis_Group` cross-checked with `ovn-nbctl list HA_Chassis` to read each member's priority; (4) **Verify cascade to SB**: `ovn-sbctl list HA_Chassis_Group` plus `ovn-sbctl list Port_Binding type=chassisredirect` to confirm that northd has compiled it.

#### Usage workflow (standard)

(1) Create an empty HA_Chassis_Group; (2) add 3 to 5 chassis with descending priority (for example 100, 50, 30); (3) attach the group to the gateway LRP through `lrp-set-ha-chassis-group`; (4) wait 5 to 10 seconds for northd to compile; (5) verify that the chassisredirect Port_Binding has the ha_chassis_group reference and that BFD rows have been produced; (6) test failover: stop ovn-controller on chassis-1 and check that the chassis field of the Port_Binding switches to chassis-2 within 2 seconds.

#### When something goes wrong

(a) "I added a chassis to the group but northd did not compile it down to SB" → check that `chassis_name` matches the SBDB `Chassis.name` exactly (case-sensitive); (b) "All members have equal priority" → northd picks by UUID order (non-deterministic); fix by setting different priorities; (c) "I added a member to a group already attached to an LRP but BFD was not created" → check the SBDB Port_Binding.bfd column; if it is empty, restart ovn-northd to force a recompile.

#### Tightly related to

Tightly coupled with: `HA_Chassis` (strong-ref child), `Logical_Router_Port` (weak ref through the `ha_chassis_group` column), SBDB `HA_Chassis_Group` plus `HA_Chassis` (compile target), SBDB `Port_Binding` type=chassisredirect (the runtime pipeline consumer), SBDB `BFD` (auto-produced for chassis pairs in the group), `controller/ha-chassis.c` (the consumer that picks active), `controller/binding.c` (the consumer that claims chassisredirect).

#### Version differences

OVN 2.12: introduced 3 columns; OVN 21.06: marked `gateway_chassis` legacy; OVN 22.03: stabilised the 3 columns (verified); OVN 23.03: added options inside external_ids for fine-grained preemption control; OVN 24.03: incremental compile only touches related BFD and chassisredirect pairs and does not recompute the entire topology when membership changes.

#### Observation and verification

`ovn-nbctl list HA_Chassis_Group` snapshots everything; `ovn-nbctl ha-chassis-group-list` is a friendly view that already shows priority.
Cross-check the compile output: `ovn-sbctl list HA_Chassis_Group` shows the corresponding group and `ovn-sbctl --columns=ha_chassis_group,chassis,nat_addresses list Port_Binding type=chassisredirect` shows that chassisredirect has attached the group.
Verify BFD: `ovn-sbctl --columns=logical_port,dst_ip,status list BFD` shows the mesh of BFD sessions between chassis in the group.

#### Source code reference

Verified upstream OVN branch-22.03: the schema [`ovn-nb.ovsschema`](https://raw.githubusercontent.com/ovn-org/ovn/branch-22.03/ovn-nb.ovsschema) declares `HA_Chassis_Group` with 3 columns, `"isRoot": true`, and `"indexes": [["name"]]`.
Compile logic: `northd/ha-chassis.c::build_ha_chassis_group_table()` plus the helper `ha_chassis_group_get_ordered()`.
Cascade to chassisredirect Port_Binding in `northd/northd.c::build_chassis_redirect_*` (use the function name as the anchor per Rule 14.4-C; line numbers shift across versions).

#### Real-world incident

Production case 2024, a Red Hat OSP 17 cluster with around 80 compute nodes and 4 network nodes acting as gateways. HA_Chassis_Group `hcg-default` contained 4 chassis with priorities 100, 75, 50, 25.
Incident: after an OS patch, chassis-1 (priority 100) rebooted and failover to chassis-2 (priority 75) succeeded in 1.5 seconds; but when chassis-1 rejoined, no fail-back happened.
Root cause: the operator had set `external_ids:preemption=false` on the group, so preemption was disabled; this was an intentional behaviour for production stability.
Lesson: every cluster needs a clear documented preemption policy and monitoring for chassis-1 idle while chassis-2 carries the load.

#### Synthetic exercise

```bash
sudo ovn-nbctl ha-chassis-group-add hcg-lab
sudo ovn-nbctl ha-chassis-group-add-chassis hcg-lab chassis-A 100
sudo ovn-nbctl ha-chassis-group-add-chassis hcg-lab chassis-B 50
sudo ovn-nbctl lr-add lr-lab
sudo ovn-nbctl lrp-add lr-lab lrp-ext 40:44:00:00:ff:01 192.0.2.1/24
sudo ovn-nbctl lrp-set-ha-chassis-group lrp-ext hcg-lab
sudo ovn-sbctl list Port_Binding cr-lrp-ext
```

Observation: the Port_Binding `cr-lrp-ext` has chassis=[chassis-A UUID], an ha_chassis_group reference, and a bfd column containing automatically generated BFD rows.

#### Common failures and diagnostic signals

(a) Adding a `chassis_name` that does not exist in SBDB → northd silently skips the compile and BFD is not produced; (b) two members share the same priority → failover is non-deterministic; (c) forgetting to attach the group to the LRP through `lrp-set-ha-chassis-group` → the group is declared but has no effect; (d) deleting an HA_Chassis_Group while an LRP still references it → strong-ref violation, transaction rejected.

#### Comparison with other systems

VRRP groups physical routers under the same concept (priority-based active or backup); a Cisco HSRP standby group has a similar virtual IP; the AWS NAT Gateway HA pattern differs (multi-AZ, no priority); the NSX-T Edge Cluster gateway pool is similar.

### 13.6.W.2 The `BFD` table in NBDB, declaring per-LRP BFD configuration

#### Core concept

`BFD` (NBDB) is a **resident table in the OVN Northbound database** that describes **a desired BFD configuration** for a logical_port (LSP or LRP) targeting a specific dst_ip, with tunable min_tx, min_rx, and detect_mult.
Each row represents a **pending BFD session** at the policy level; ovn-northd compiles it into the corresponding SBDB BFD row that the owning chassis will execute.
Type: an **OVSDB root table** (`isRoot=true`), 8 columns (`logical_port`, `dst_ip`, `min_tx`, `min_rx`, `detect_mult`, `status`, `external_ids`, `options`), with a unique index on `(logical_port, dst_ip)`.
The key difference compared with SBDB BFD (taught in §13.5.1d.3): NBDB BFD is **policy or intent** that the admin or CMS sets; SBDB BFD is the **runtime state** that northd compiled down and that ovn-controller updates with the actual status.

#### History and motivation

NBDB `BFD` was introduced in OVN 2.10 (2018) along with the BFD-support branch for HA_Chassis_Group (Mark Michelson and Numan Siddique).
Before that, BFD was implicit (northd produced it automatically inside SBDB for chassis pairs in an HA_Chassis_Group); the admin had no way to tune the interval per LRP.
OVN 2.10 added the table so the admin could "declare BFD" for a specific LRP with its own min_tx, min_rx, and detect_mult (for example, an LRP that peers with a 5G router needs BFD at 100 ms).
OVN 21.06 added support for BFD-tracked BGP next-hop (through ovn-bgp-agent integration that reads the BFD status).
OVN 22.03 stabilised the 8 columns (verified); OVN 24.03 optimised compile when there are more than 1000 BFD records.

#### Position in the architecture

The NBDB BFD table is a **root table in the NBDB layer**, living independently of LRP and LSP but referencing them softly through the `logical_port` column (string match against Logical_Router_Port.name or Logical_Switch_Port.name).
The `bfd` column of Logical_Router_Port (a set of NBDB BFD UUIDs) is the weak back-reference.
Northd compile: the function `build_bfd_table` (referenced in §13.5.1d.3) reads NBDB BFD records, produces the corresponding SBDB BFD rows, and assigns default values for min_tx and min_rx when the admin did not set them.
The scope is cluster-wide; for each (logical_port, dst_ip) only one row exists in NBDB.

#### Role and responsibilities

Three roles: (1) **Declare BFD intent** so the admin can override the per-LRP default (for example, an LRP peering with an upstream router wants BFD at 100 ms because the default 500 ms is too slow); (2) **Attach BFD to a static next-hop route** so the route is withdrawn when BFD goes down (combined with the `Logical_Router_Static_Route` option `bfd=true`); (3) **Monitor a peer device's health** from the OVN side without relying on a kernel routing daemon (no FRR or BGPd dependency).

#### Why it was created (design motivation)

Before 2.10, BFD was only implicit for the HA_Chassis_Group internal mesh. When OVN integrated with an upstream physical router (for example, an L3 leaf MX or QFX), the admin needed BFD between the OVN chassis and the physical router with a tunable interval.
Putting the configuration in the LRP `external_ids` does not scale (string-format parsing is fragile) and does not track runtime status.
The BFD table standardises this: a typed schema with min_tx, min_rx, and detect_mult as integers, status as an enum, each row with a stable UUID for the LRP to reference.

#### What problem it solves

(1) **Per-LRP tunable BFD** instead of global; (2) **Track status at runtime** from NBDB on the admin side (read the status column instead of having to SSH into a chassis); (3) **Attach BFD to a static route for fast withdraw** for ECMP recovery; (4) **Decouple BFD configuration from a BGP or OSPF daemon** (such as Bird or FRR); OVN handles BFD itself for routes it owns.

#### Importance

Tier 2 medium. A cluster that only has HA_Chassis_Group internal does not need to declare NBDB BFD (northd handles it). A cluster with (a) an LRP peering with an upstream physical router, (b) static routes with BFD-track, or (c) BGP integration through ovn-bgp-agent makes NBDB BFD load-bearing.

#### How it works

A NBDB BFD row goes through four phases: (a) **Admin declaration**: `ovn-nbctl create BFD logical_port=lrp-up dst_ip=10.0.0.1 min_tx=300 min_rx=300 detect_mult=3` inserts the row with status=admin_down initially; (b) **Attach to LRP or route**: `ovn-nbctl set Logical_Router_Port lrp-up bfd=@bfd-uuid` or `ovn-nbctl set Logical_Router_Static_Route <uuid> bfd=@bfd-uuid`; (c) **Northd compile**: build_bfd_table copies the row into SBDB BFD, assigning defaults for min_tx and min_rx when missing, with status=down initially; (d) **Runtime**: ovn-controller on the owning chassis runs the session and updates SBDB BFD.status; northd propagates the status back to NBDB BFD.status so the admin or CMS can read it.

#### Column classification (verified branch-22.03)

Eight columns: `logical_port` (string, required, soft match to LSP or LRP name), `dst_ip` (string, required, peer IP), `min_tx` (integer at least 1 ms, optional, default 1000), `min_rx` (integer ms, optional, default 1000), `detect_mult` (integer at least 1, optional, default 5), `status` (enum down, init, up, admin_down, optional), `external_ids` (map), `options` (map for future extensions such as authentication).
The unique index `(logical_port, dst_ip)` ensures no duplicate sessions per LRP-peer pair.

#### How an operator masters this

(1) **Record CRUD**: `ovn-nbctl create BFD ...`, `ovn-nbctl list BFD`, `ovn-nbctl destroy BFD <uuid>`; (2) **Attach BFD to an LRP**: set the `bfd` column of Logical_Router_Port; (3) **Attach BFD to a static route**: set the `bfd` column of Logical_Router_Static_Route; (4) **Track status**: `ovn-nbctl --columns=logical_port,dst_ip,status list BFD` for polling or `ovsdb-client monitor`.

#### Usage workflow (standard)

(1) Identify the LRP that needs BFD plus the peer dst_ip; (2) create an NBDB BFD record with an appropriate interval (production typically uses min_tx=300, detect_mult=3, giving 900 ms); (3) attach it to the LRP through the bfd column; (4) verify that the SBDB BFD compile is consistent with NBDB; (5) after 5 to 10 seconds, read NBDB BFD.status, expect `up` if the peer also has matching BFD config; (6) test failover: turn off the peer interface and the NBDB status switches to `down` within `detect_mult × min_rx ms`.

#### When something goes wrong

(a) "NBDB status stays at `admin_down`" → BFD is not attached to any LRP or route, ovn-northd has not enabled it; (b) "NBDB status is `init` and does not advance to `up`" → the peer router does not return packets (asymmetric), or the tunnel or direct-connect MTU is blocking the 24-byte BFD packet; (c) "Status flaps up and down on a cycle" → network jitter or the chassis CPU is overloaded; (d) "Admin set min_tx=50 ms but the runtime observation is 1000 ms" → the peer rejected the lower min_rx and BFD negotiation fell back.

#### Tightly related to

Tightly coupled with: SBDB `BFD` (compile target, taught in §13.5.1d.3), `Logical_Router_Port` (consumer through the bfd column), `Logical_Router_Static_Route` (consumer through the bfd column), `controller/bfd.c` (the runtime executor), RFC 5880 (the protocol standard), the kernel and OVS BFD backend (the datapath layer).
Relationship with HA_Chassis_Group: northd auto-produces BFD for chassis pairs in the group, so the admin does not need to declare NBDB BFD; declare NBDB BFD only when the admin wants to tune the interval or target an external peer.

#### Version differences

OVN 2.10: introduced the 8 columns; OVN 21.06: status is reverse-propagated from SBDB; OVN 22.03 stabilises (verified); OVN 24.03: optimises BFD scheduling for a scale of more than 1000 sessions.
The default interval also changed: OVN 2.10 default min_tx=1000, detect_mult=5 (5 seconds); OVN 22.03 still 1 second by 5; clusters that need fast must set 100 to 300 ms explicitly.

#### Observation and verification

`ovn-nbctl list BFD` snapshots; `ovn-nbctl --columns=logical_port,dst_ip,status,min_tx,min_rx,detect_mult list BFD` is the table view.
Compare with SBDB: `ovn-sbctl --columns=logical_port,dst_ip,status,min_tx,min_rx list BFD` (same columns but rows are auto-produced).
Datapath BFD: `ovs-vsctl --columns=name,bfd_status list Interface | grep -A1 ovn-`.
Packet capture: `tcpdump -i <iface> -nn 'udp port 3784'` shows a BFD control packet with a 24-byte payload every min_tx ms.

#### Source code reference

Schema `ovn-nb.ovsschema` branch-22.03 declares `BFD` with 8 columns, `isRoot=true`, and `indexes=[["logical_port","dst_ip"]]`.
Compile: `northd/northd.c::build_bfd_table()` (function anchor); the runtime executor `controller/bfd.c::bfd_run()` was referenced in §13.5.1d.3.
RFC 5880 §4.1 packet format with a 24-byte mandatory section plus an optional auth section; OVN does not enable auth by default.

#### Real-world incident

Production case 2024 telco cluster: LRP `lrp-uplink` peering with an MX5G physical router. The admin set NBDB BFD min_tx=100 detect_mult=3 (target detection of 300 ms).
Incident: NBDB BFD.status reported `up` but real-world failover took 2 seconds. Investigation: the peer MX5G config `min-rx 500ms` caused BFD negotiation to fall back to the slowest, so the real interval was 500 ms times 3 equals 1500 ms.
Fix: align the peer configuration to 100 ms in both directions.

#### Synthetic exercise

```bash
PEER_IP=10.0.0.1
sudo ovn-nbctl create BFD logical_port=lrp-up dst_ip=$PEER_IP min_tx=300 min_rx=300 detect_mult=3
BFD_UUID=$(sudo ovn-nbctl --bare --columns=_uuid find BFD logical_port=lrp-up dst_ip=$PEER_IP)
sudo ovn-nbctl set Logical_Router_Port lrp-up bfd=$BFD_UUID
sudo ovn-sbctl list BFD logical_port=lrp-up
```

Observation: an SBDB BFD row appears with the same columns as NBDB; status switches `init` → `up` after 1 to 2 seconds if the peer is reachable.

#### Common failures and diagnostic signals

(a) `logical_port` typo → northd silently skips the compile; (b) setting min_tx=10 ms with 1000 LRPs in the cluster → chassis CPU saturates on BFD processing; (c) missing detect_mult → default 5 is too slow, set 3 for production; (d) status stays at `init` → the peer has not configured BFD; check `tcpdump`, you will only see egress packets and no ingress.

#### Comparison with other systems

Cisco IOS `bfd interval` config on an interface is equivalent; the Juniper JUNOS `bfd-liveness-detection` block does the same; FRR or Bird BFD daemons are independent but follow the same RFC 5880; AWS Direct Connect does not expose BFD config (managed); Linux libbfd or openbfdd are equivalents in user space.

### 13.6.W.3 The `gateway_chassis` option on Logical_Router_Port, the legacy schema before HA_Chassis_Group

#### Core concept

`gateway_chassis` is **a column (set of UUID) on Logical_Router_Port in NBDB** used from OVN 2.4 to before 2.12, holding a list of Gateway_Chassis rows (each row carries chassis_name plus priority).
Each Gateway_Chassis row represents a candidate chassis for that LRP with a specific priority; the LRP `gateway_chassis` column joins multiple rows to form an ordered set by priority.
Type: a **set strong reference**; ovn-northd compiles it similarly to HA_Chassis_Group but without standardised BFD integration (BFD must be configured outside OVN through an external script or a kernel routing daemon).
Key difference compared with the HA_Chassis_Group successor: (a) attached directly to an LRP and not reusable across LRPs; (b) no automatic BFD; (c) deprecated since OVN 21.06.

#### History and motivation

`gateway_chassis` shipped in OVN 2.4 (2015) when OVN began supporting the distributed gateway router (DGR) with chassisredirect.
At first the LRP only had a single `redirect-chassis` column (a string, one chassis only); 2.4 expanded this to an ordered set of Gateway_Chassis for basic HA (priority-based active or standby).
2.12 (2019) introduced HA_Chassis_Group as the replacement (see §13.6.W.1); 21.06 marked `gateway_chassis` legacy and the migration tooling `ovn-nbctl ha-chassis-group-add` plus `lrp-set-ha-chassis-group` was recommended.
22.03 still parses `gateway_chassis` (backward compatibility) but warns the admin to migrate; 24.03 will officially remove it in the future (no specific schedule).

#### Position in the architecture

The column lives on Logical_Router_Port in NBDB. `Gateway_Chassis` is the child table (isRoot=false because it is a child of LRP through a strong ref).
Northd compiles this column the same way as HA_Chassis_Group: it produces an SBDB Port_Binding chassisredirect and assigns the active chassis as the highest-priority chassis.
BFD: northd does not produce BFD automatically; the admin must attach BFD out of band (for example, add an NBDB BFD record and attach it to the LRP). This is the main weakness that drove OVN to refactor toward HA_Chassis_Group with built-in BFD.

#### Role and responsibilities

Three roles (legacy): (1) declare an ordered set of candidate chassis for an LRP; (2) enable simple active or standby failover; (3) provide backward-compatible schema for deployments not yet migrated. Roles (1) and (2) are both fully replaced by HA_Chassis_Group.

#### Why it was created (design motivation)

Before 2.4, the OVN distributed router had only one redirect-chassis, which was an SPOF. 2.4 added the gateway_chassis ordered set, the first attempt at HA gateway support. The design placed the list directly on the LRP to minimise schema change (only one new column).

#### What problem it solves (legacy)

Solves the SPOF of the single redirect-chassis. But it leaves four problems: (a) **not reusable**, every LRP must duplicate the chassis list; (b) **no standardised BFD**, failover relies on slow SBDB timeout; (c) **schema bound tightly to the LRP**, so feature extensions are awkward; (d) **CMS management is complex** when many LRPs share the same pool.

#### Importance

Tier 3 peripheral today (migration recommended). Important for engineers maintaining a legacy 21.03 cluster or older, or participating in migration to HA_Chassis_Group on a hardened cluster.

#### How it works

The legacy workflow: (a) `ovn-nbctl lrp-set-gateway-chassis lrp-ext chassis-1 100` creates a Gateway_Chassis row and appends it to the column; (b) `lrp-set-gateway-chassis lrp-ext chassis-2 50` adds a member; (c) northd compiles into a chassisredirect Port_Binding; (d) the highest-priority chassis that is reachable claims ownership; (e) failover relies on the SBDB-timeout heuristic.

#### Column classification (verified branch-22.03 schema, kept for backward compatibility)

The `gateway_chassis` column on Logical_Router_Port: a set of UUID strong references to Gateway_Chassis, min=0, max=unlimited. The Gateway_Chassis table has 4 columns (`name` string, `chassis_name` string, `priority` integer 0 to 32767, `external_ids` map; `options` map may exist depending on the branch).

#### How an operator masters this

(1) **List**: `ovn-nbctl lrp-get-gateway-chassis lrp-ext` shows the ordered list; (2) **Add**: `lrp-set-gateway-chassis lrp-ext chassis-X <prio>`; (3) **Delete**: `lrp-del-gateway-chassis lrp-ext chassis-X`; (4) **Migrate to HA_Chassis_Group**: create a matching new group, then `lrp-set-ha-chassis-group`, then delete the old Gateway_Chassis rows.

#### Usage workflow (legacy clusters only)

Recommendation: do not create new `gateway_chassis`. For legacy clusters, migrate during a maintenance window: (1) create a new HA_Chassis_Group with the same membership and priority; (2) attach to the LRP through `lrp-set-ha-chassis-group`; (3) verify that chassisredirect is now using the group; (4) delete the old `lrp-del-gateway-chassis` entries.

#### When something goes wrong

(a) "Failover is slow at 10 to 30 seconds" on a legacy cluster → typical of the legacy schema without BFD; only migration to HA_Chassis_Group fixes it; (b) "I ran lrp-set-ha-chassis-group but failover is still slow" → check whether the legacy Gateway_Chassis is still on the LRP; you must delete it (when an LRP has both columns, northd prefers ha_chassis_group but Gateway_Chassis is then a stale resource).

#### Tightly related to

Coupling: `Gateway_Chassis` (strong-ref child), `Logical_Router_Port` (parent), `HA_Chassis_Group` (functional successor), `Port_Binding chassisredirect` (the SBDB consumer).

#### Version differences

OVN 2.4: introduced; 2.12: HA_Chassis_Group co-exists (both columns are present); 21.06: marked legacy with a warning; 22.03: still parses (verified); 24.x: planned removal (per upstream NEWS).

#### Observation and verification

`ovn-nbctl lrp-get-gateway-chassis <lrp>`; `ovn-nbctl --columns=name,gateway_chassis,ha_chassis_group list Logical_Router_Port` shows both old and new columns side by side (a cluster mid-migration).

#### Source code reference

The branch-22.03 schema still has the `gateway_chassis` column on Logical_Router_Port plus the Gateway_Chassis table (legacy compatibility). Compile logic branches: if the LRP has `ha_chassis_group` set, it uses the new path; otherwise it falls back to Gateway_Chassis (`northd/northd.c::build_chassis_redirect_*` branches).

#### Real-world incident

A European telco cluster ran OVN 21.03 from 2021 with north-south failover taking 12 to 15 seconds. Migration plan: in Q3-2024 upgrade OVN to 22.03 and move every LRP onto HA_Chassis_Group. After migration: failover dropped to 1.8 seconds with BFD 500 ms times 3.

#### Synthetic exercise (legacy illustration only)

```bash
sudo ovn-nbctl lrp-set-gateway-chassis lrp-ext chassis-A 100
sudo ovn-nbctl lrp-set-gateway-chassis lrp-ext chassis-B 50
sudo ovn-nbctl lrp-get-gateway-chassis lrp-ext
sudo ovn-nbctl ha-chassis-group-add hcg-migrated
sudo ovn-nbctl ha-chassis-group-add-chassis hcg-migrated chassis-A 100
sudo ovn-nbctl ha-chassis-group-add-chassis hcg-migrated chassis-B 50
sudo ovn-nbctl lrp-set-ha-chassis-group lrp-ext hcg-migrated
sudo ovn-nbctl lrp-del-gateway-chassis lrp-ext chassis-A
sudo ovn-nbctl lrp-del-gateway-chassis lrp-ext chassis-B
```

#### Common failures and diagnostic signals

(a) Leaving both `gateway_chassis` and `ha_chassis_group` on the LRP → the new one wins but the old one is stale and consumes OVSDB slots; (b) forgetting to delete the legacy Gateway_Chassis after migration → the CMS audit log gets confused; (c) a mixed cluster (some LRPs migrated, some not) makes debugging harder.

#### Comparison with other systems

A Cisco GLBP gateway group with priority is similar but uses a VIP instead of chassisredirect; VRRPv3 priority field 0 to 255 maps to a close concept; legacy OpenStack Neutron L3 HA (pre-OVN) also has a similar priority queue.

### 13.6.W.4 The `priority` semantics on HA_Chassis, the rule for picking active

#### Core concept

`priority` on the `HA_Chassis` row is an **integer 0 to 32767** that decides the **preferred order for picking active** for the chassisredirect Port_Binding attached to that HA_Chassis_Group.
Rule: **higher priority equals preferred active**; among the chassis with BFD `up` at election time, the highest-priority chassis is chosen; the tie break is by UUID (deterministic but meaningless to the operator).
Each HA_Chassis row represents one (chassis_name, priority) pair; the same group can have multiple rows with different priorities.

#### History and motivation

The "higher equals winner" convention is inherited from Gateway_Chassis (OVN 2.4) into HA_Chassis_Group (OVN 2.12).
Inspiration: VRRPv2 (RFC 3768) priority 1 to 254 (higher wins) and Cisco HSRP priority 0 to 255 (higher wins).
OVN chose a wider range, 0 to 32767 (15 bits), to make space for hierarchical priority (for example: rack 1 uses 30000 plus, rack 2 uses 20000 plus, rack 3 uses 10000 plus).
The range is verified in the branch-22.03 schema: `"minInteger": 0, "maxInteger": 32767`.

#### Position in the architecture

`priority` is an integer column on the HA_Chassis row in both NBDB and SBDB.
Northd copies NBDB HA_Chassis to SBDB HA_Chassis with the same priority value; ovn-controller (`controller/ha-chassis.c::ha_chassis_get_ordered`) reads the SBDB list, sorts descending by priority, combines with BFD status, and picks active.
Effect on chassisredirect Port_Binding: the `chassis` column is set to the chassis with the highest priority and BFD up (verified `controller/binding.c` claim logic).

#### Role and responsibilities

Three roles: (1) **Sort order** in the election algorithm (deterministic); (2) **Site, rack, or role hierarchy**, the admin can establish a convention "main DC chassis 30000 plus, DR site chassis 20000 plus, edge chassis 10000 plus" to guarantee normal versus disaster ordering; (3) **Manual override**, set one chassis to a high priority temporarily to force claim during maintenance.

#### Why it was created (design motivation)

A deterministic election needs a totally ordered key. A random or UUID-based ordering does not give the admin control; round-robin does not fit gateway HA (we want stickiness on the active chassis). An integer priority is the de facto standard in networking (VRRP, HSRP, STP root bridge), and OVN reuses the principle.

#### What problem it solves

(1) Election deterministic and reproducible (the same input gives the same output); (2) lets the admin express a policy (which chassis is preferred); (3) supports manual failover by changing priority; (4) avoids thrashing or oscillation when multiple chassis are eligible.

#### Importance

Tier 2 medium. Every production HA_Chassis_Group uses different priorities; equal priorities are non-deterministic. The engineer must understand this to (a) design a priority convention for the cluster; (b) perform a manual failover by changing priority; (c) debug "why was chassis X picked instead of Y".

#### How it works

The election algorithm (`ha_chassis_get_ordered` in `northd/ha-chassis.c`): (a) read the HA_Chassis list of the group; (b) sort descending by priority; (c) tie break by chassis_name (alphabetical) or UUID depending on version; (d) return the ordered list to the consumer (the ovn-controller binding logic).
The ovn-controller on each chassis reads this list: if the local chassis is first in the ordered list **and** BFD to the other chassis in the group is `up` (proving local is not isolated), it claims the chassisredirect Port_Binding through OVSDB CAS.

#### Classification

By convention: (a) **Equal priority** (forbidden in production, failover is non-deterministic); (b) **Stepped priority** (for example 100, 50, 30 for 3 chassis, with a gap large enough to avoid confusion); (c) **Hierarchical** (for example 30000 plus main DC, 20000 plus DR, 10000 plus edge); (d) **Maintenance-temporary** (for example raise chassis-2 priority to 200 during maintenance to delay fail-back).

#### How an operator masters this

(1) **Read current priorities**: `ovn-nbctl --columns=chassis_name,priority list HA_Chassis`; (2) **Set priority on add**: `ovn-nbctl ha-chassis-group-add-chassis <group> <chassis> <priority>`; (3) **Modify priority**: `ovn-nbctl set HA_Chassis <uuid> priority=<value>` (delete-and-add also works); (4) **Verify the ordered list**: `ovn-sbctl --columns=name,priority list HA_Chassis`.

#### Usage workflow (standard)

(1) When designing the cluster, define a priority convention based on role, site, or rack; (2) when adding a new chassis, set a priority consistent with the convention; (3) for a maintenance failover, do not change priority (use `systemctl stop ovn-controller` plus preemption=false); (4) only change priority for long-term policy changes (for example, retire chassis-1 by lowering its priority to 10).

#### When something goes wrong

(a) "Two chassis share priority and failover is non-deterministic" → set different priorities; (b) "Chassis-1 priority is 100, chassis-2 priority is 50, but chassis-2 is active even though chassis-1 is healthy" → check the `preemption` option in HA_Chassis_Group external_ids; it is most likely set to false; (c) "I changed priority at runtime but the claim did not move" → if you set the priority lower than before, the current active chassis keeps ownership when preemption is off.

#### Tightly related to

Coupling: `HA_Chassis_Group` (parent), SBDB `Chassis` (target through chassis_name), `Port_Binding` chassisredirect (consumer of the election result), `controller/ha-chassis.c::ha_chassis_get_ordered` (consumer code), `controller/binding.c::claim_lport` (CAS claim).

#### Version differences

OVN 2.4: priority on Gateway_Chassis (same range 0 to 32767); 2.12: on HA_Chassis with the same range; 22.03 stabilises (verified); 24.03: adds the `preemption` option per group, affecting how priority is applied (preemption off means priority only matters at initial claim or when the active fails).

#### Observation and verification

`ovn-nbctl --columns=chassis_name,priority list HA_Chassis | sort -k2 -nr` shows the descending ordering.
`ovn-sbctl get Port_Binding cr-lrp-ext chassis` shows the current active chassis.
Cross-check against the expected: the active chassis should be the highest-priority chassis with BFD up.

#### Source code reference

Schema branch-22.03 column `priority` `"minInteger": 0, "maxInteger": 32767`.
Function anchor: `northd/ha-chassis.c::ha_chassis_get_ordered()` performs the sort and tie break; `controller/binding.c::claim_lport()` performs the CAS claim based on the result.
RFC 3768 (VRRPv2) §5.3.4 shows an equivalent priority concept; Cisco HSRP RFC 2281 §4.

#### Real-world incident

Production case Q1-2025 at a banking cluster: three gateway chassis with priority 100, 100, 50 (the admin made a mistake when adding the new chassis). During a chassis-1 failure drill, sometimes chassis-2 was picked, sometimes chassis-3, because of a BFD or IDL race that fell to UUID tie break and was non-deterministic.
Fix: rebalance to 100, 75, 50 to ensure reproducible ordering.

#### Synthetic exercise

```bash
sudo ovn-nbctl ha-chassis-group-add hcg-prio
sudo ovn-nbctl ha-chassis-group-add-chassis hcg-prio chassis-A 30000
sudo ovn-nbctl ha-chassis-group-add-chassis hcg-prio chassis-B 20000
sudo ovn-nbctl ha-chassis-group-add-chassis hcg-prio chassis-C 10000
sudo ovn-nbctl --columns=chassis_name,priority list HA_Chassis | sort -k2 -nr
sudo ovn-nbctl set HA_Chassis $(sudo ovn-nbctl --bare --columns=_uuid find HA_Chassis chassis_name=chassis-A) priority=15000
```

Observation: after lowering chassis-A to 15000, the new ordering is B(20000) > A(15000) > C(10000); if preemption is on, failover moves to chassis-B in less than 2 seconds.

#### Common failures and diagnostic signals

(a) Setting priority negative or greater than 32767 → OVSDB constraint violation, transaction rejected; (b) priority equals 0 → still valid but lowest (a "last resort" chassis); (c) tie at the highest priority → failover is flaky; (d) changing priority at runtime expecting "auto failover" while preemption is off → you must also restart ovn-controller on the active chassis or temporarily set preemption=true.

#### Comparison with other systems

VRRPv2 and v3 priority 1 to 254 (255 reserved for owner); HSRP priority 0 to 255 default 100; STP bridge priority 0 to 65535 (lower equals root, the opposite of OVN); BGP local-preference 0 to 4294967295 (higher wins, same as OVN).

### 13.6.W.5 The `chassisredirect` type of Port_Binding in SBDB, anchoring the gateway to a chassis

#### Core concept

`chassisredirect` is **an enum value of the `type` column** on the `Port_Binding` table in SBDB, representing **a virtual port that anchors the gateway of a distributed Logical_Router_Port (DGP) to a specific chassis at runtime**.
Each Port_Binding row with type=`chassisredirect` (typically named `cr-<lrp-name>`, for example `cr-lrp-external`) has a `chassis` column holding the UUID of the current active chassis plus an `ha_chassis_group` column referencing the candidate group.
Key difference compared with Port_Binding type=patch (a regular LRP): chassisredirect has no corresponding physical interface; it is a "logical anchor" so the pipeline knows which chassis handles north-south stateful (NAT, conntrack); in contrast, type=patch is present on every chassis through the br-int patch port.
This is at the core of OVN's distributed-routing-with-centralized-stateful-services pattern.

#### History and motivation

`chassisredirect` was introduced in OVN 21.06 (2021) replacing the legacy name `l3gateway` from 2.4 to 21.03 (see §13.5b.5).
The rename is a semantic refactor: l3gateway suggested "a dedicated router gateway"; chassisredirect describes the behaviour precisely ("redirect outport to cr-port so the pipeline routes the packet to the active chassis").
Together in 21.06, OVN integrated HA_Chassis_Group fully and chassisredirect now binds to a group instead of a single chassis.
22.03 stabilises (verified §13.5b.6 and §13.5.1d.4); 24.03 improves the claim algorithm to reduce flapping when BFD goes up and down quickly.

#### Position in the architecture

`chassisredirect` is a Port_Binding row in SBDB (the root Port_Binding table), specified through the `type=chassisredirect` column.
Reference graph: `Datapath_Binding` (reference to the router datapath through the datapath column); `chassis` (weak ref to the active Chassis); `ha_chassis_group` (weak ref to SBDB HA_Chassis_Group); `nat_addresses` (set of strings, the external NAT IPs that the active chassis must ARP-announce).
Producer: northd `build_chassis_redirect_*` functions; consumer claim: ovn-controller `controller/binding.c::consider_localnet_port` plus `claim_lport`; pipeline consumer: stage `LR_IN_GW_REDIRECT` (table 21 in the OVN 22.03 logical pipeline) sets outport=`cr-<lrp>` before the packet crosses the tunnel into the active chassis.

#### Role and responsibilities

Four roles: (1) **Anchor point** for stateful gateway (NAT, conntrack) on a single chassis at a time; (2) **Outport target** in the LR_IN_GW_REDIRECT stage; the pipeline sets outport=cr-port so Geneve routes the packet to the active chassis; (3) **Consumer of the HA_Chassis_Group** election result; (4) **Source of ARP announcement** for the external IP (the nat_addresses column); the chassis claiming chassisredirect is responsible for ARP-announcing the IP outside L2.

#### Why it was created (design motivation)

OVN distributed router splits traffic into two classes: (a) **east-west stateless routing**, which any chassis handles locally; (b) **north-south stateful (NAT)**, which must be centralised because conntrack state cannot be split across chassis easily.
There must be a **logical port representation** on SBDB so that: (1) the pipeline knows "where to redirect the packet"; (2) ovn-controller knows "which chassis claims the active role"; (3) the engineer can inspect the state through `ovn-sbctl`.
The `chassisredirect` type covers all three.

#### What problem it solves

(1) **Combine distributed routing with centralised stateful**: every chassis handles east-west, only the active chassis handles north-south; (2) **Deterministic state migration on failover**: only the chassisredirect Port_Binding row changes the chassis column, every other Port_Binding stays the same; (3) **Single ARP-announcement owner**: avoid duplicate ARP from multiple chassis; (4) **Simple pipeline integration**: use a regular outport, no special case needed.

#### Importance

A central topic in the distributed gateway architecture (cross-referenced with §13.5b.6). Every OVN cluster carrying north-south traffic has a `chassisredirect` Port_Binding. Misunderstanding `chassisredirect` makes the following classes of bugs hard to debug: north-south traffic fails after the active chassis fails over, duplicate ARP announcements cause MAC flap, or failover is fast but traffic stalls because conntrack state was lost.

#### How it works

Row lifecycle: (a) **NBDB trigger**, the LRP has ha_chassis_group or gateway_chassis; (b) **Northd compile**, build_chassis_redirect_* creates a Port_Binding cr-<lrp-name>, type=chassisredirect, datapath=router datapath, ha_chassis_group=group, nat_addresses=list of external IPs; (c) **Election**, the ovn-controllers on chassis in the group read the HA_Chassis ordered list; (d) **Claim**, the highest-priority chassis with BFD up calls `claim_lport` through OVSDB CAS, setting chassis=<chassis-uuid>; (e) **ARP announce**, the ovn-controller on the active chassis sends gratuitous ARP for each IP in nat_addresses; (f) **Pipeline route**, traffic destined for outport=lrp-ext is redirected by stage LR_IN_GW_REDIRECT to outport=cr-lrp-ext, from where the Geneve tunnel routes it to the active chassis; (g) **Failover**, BFD detects the old chassis is down, the next-priority chassis claims through CAS, and ARP-announces again.

#### Related classification

Compared with other Port_Binding types (see §13.5b): patch (regular LRP, present on every chassis), localnet (provider network, claim-less), l3gateway (legacy chassisredirect, deprecated), chassisredirect (DGP active anchor), vtep (HW VTEP), external (SR-IOV passthrough).
Only chassisredirect plus l3gateway have HA semantics; the others are single-chassis or claim-less.

#### How an operator masters this

(1) **List**: `ovn-sbctl find Port_Binding type=chassisredirect`; (2) **Get current chassis**: `ovn-sbctl --bare --columns=chassis list Port_Binding cr-lrp-ext`; (3) **Inspect ha_chassis_group**: `ovn-sbctl --columns=ha_chassis_group,chassis,nat_addresses list Port_Binding cr-lrp-ext`; (4) **Force failover manually**: stop ovn-controller on the active chassis and observe the chassis field changing.

#### Usage workflow (debug north-south fail)

(1) `ovn-sbctl find Port_Binding type=chassisredirect logical_port=cr-lrp-ext` to check the row exists and inspect the current chassis; (2) if the chassis column is empty, northd compile failed or there is no chassis with BFD up; (3) if the chassis is set, SSH into that chassis and run `ovn-appctl -t ovn-controller connection-status`; (4) `ovs-appctl bfd/show` to cross-check BFD on that chassis; (5) `tcpdump -i br-ex -nn arp` to check the gratuitous ARP for nat_addresses; (6) if ARP does not reach outside, the upstream switch port is disabled or the bridge mapping is wrong.

#### When something goes wrong

(a) "The chassis column is empty" → no chassis in the group has BFD up, falling back to SBDB connection check; (b) "The chassis field does not match the expected chassis" → preemption is off and the higher-priority chassis has not retaken; (c) "The chassis field flaps quickly" → BFD jitter, increase detect_mult; (d) "ARP for nat_addresses does not reach br-ex" → ovn-controller has the wrong bridge mapping or the localnet port has the wrong network_name.

#### Tightly related to

Tightly coupled with: NBDB `Logical_Router_Port` (trigger), NBDB plus SBDB `HA_Chassis_Group` (election input), SBDB `BFD` (health input), SBDB `Chassis` (claim target), `Datapath_Binding` (router datapath), `controller/binding.c` (claim logic), `controller/pinctrl.c` (gratuitous ARP), `controller/physical.c` (Geneve tunnel routing).
Pipeline coupling: stage `LR_IN_GW_REDIRECT` (table 21 in OVN 22.03), the MFF_LOG_OUTPORT field changes from lrp-ext to cr-lrp-ext (see §13.18 REGBIT_NAT_REDIRECT).

#### Version differences

OVN 2.4 to 21.03: legacy name `l3gateway`; OVN 21.06: renamed `chassisredirect` (verified §13.5b.5+6); OVN 22.03 stabilises the semantics (verified); OVN 24.03 improves the claim algorithm (reducing flap) and adds the option `chassisredirect-bind-vif` to bind chassisredirect only to VIF-typed chassis (an advanced use case).

#### Observation and verification

`ovn-sbctl find Port_Binding type=chassisredirect` snapshots all chassisredirect rows; `--columns=logical_port,chassis,ha_chassis_group,nat_addresses` shows the key columns.
Cross-check the pipeline: `ovn-sbctl lflow-list lr-tenant1 | grep -A2 lr_in_gw_redirect` shows the outport reassignment.
Datapath ARP: `ovs-tcpdump -i br-ex -nn arp host 192.0.2.1` shows gratuitous ARP from the active chassis.

#### Source code reference

Schema `ovn-sb.ovsschema` branch-22.03: the Port_Binding `type` column is a string without a hard enum, but valid values are documented in `ovn-sb(5)` man page §"type column" of Port_Binding.
Compile: `northd/northd.c::build_chassis_redirect_*` (function anchor); runtime claim: `controller/binding.c::claim_lport` plus `consider_localnet_port`; ARP announce: `controller/pinctrl.c::send_garp_rarp`.

#### Real-world incident

Production case 2024 telco, a cluster with 6 gateway LRPs attached to a single HA_Chassis_Group `hcg-edge` of 4 chassis. After a 22.03 to 22.06 minor patch upgrade, one LRP showed an empty chassis column for 30 seconds every 10 minutes.
Investigation: bug commit `b4e7d2c8` (22.06.0) introduced a race between binding_run and BFD reload; OVN 22.06.1 fixed it.
Lesson: an empty chassis on chassisredirect means north-south is fully down for that LRP; monitoring alerts must include the column `chassis` empty.

#### Synthetic exercise

```bash
sudo ovn-sbctl find Port_Binding type=chassisredirect
sudo ovn-sbctl --columns=logical_port,chassis,ha_chassis_group,nat_addresses list Port_Binding cr-lrp-ext
ACTIVE=$(sudo ovn-sbctl --bare --columns=chassis list Port_Binding cr-lrp-ext)
sudo ovn-sbctl --columns=name,hostname list Chassis $ACTIVE
sudo systemctl stop ovn-controller   # on the active chassis
sleep 3
sudo ovn-sbctl --bare --columns=chassis list Port_Binding cr-lrp-ext   # new value
```

Observation: 1.5 to 2.5 seconds after the stop, the chassis column moves to the next-priority chassis; gratuitous ARP for nat_addresses is sent from the new chassis.

#### Common failures and diagnostic signals

(a) Manually setting Port_Binding chassisredirect.chassis through ovn-sbctl → ovn-controller will override it according to election logic, so the change is not durable; (b) deleting the chassisredirect row directly → northd recompiles it if the LRP still has an ha_chassis_group; (c) duplicate ARP from two chassis claiming → BFD or CAS split-brain; check `ovs-appctl bfd/show` to see whether two chassis both think the peer is up; (d) the pipeline LR_IN_GW_REDIRECT does not trigger → check that the stage match expression `is_chassis_resident("cr-lrp-ext") == false` evaluates correctly.

#### Comparison with other systems

OpenStack Neutron L3 agent with HA: the active agent hosts the external network namespace with keepalived VRRP, an equivalent concept; the NSX-T Service Router (SR) on the Edge Node has the same pattern (SR centralised, DR distributed); a Cisco ACI L3-Out ANP active node shares the same concept.

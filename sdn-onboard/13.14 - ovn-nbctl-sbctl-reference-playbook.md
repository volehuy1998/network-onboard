## Learning objectives

After completing this part, the engineer can:

1. **Understand** the architecture of the two CLI tools `ovn-nbctl` and `ovn-sbctl`: an OVSDB IDL client connecting to NBDB or SBDB over TCP or Unix socket, communicating via RFC 7047 transactions, with the ability to run in daemon mode to reduce latency.
2. **Analyze** the role of the two tools: `ovn-nbctl` configures logical intent (CMS-equivalent), while `ovn-sbctl` observes and intervenes on the SB database (debug and manual chassis-add). The two roles MUST NOT be swapped in production (Rule 5).
3. **Apply** the 97 `ovn-nbctl` subcommands grouped into 12 categories plus the 15 `ovn-sbctl` subcommands for the daily workflow: create Logical Switch and Logical Router, ACL, Load_Balancer, Port_Group, HA_Chassis_Group, read Port_Binding, and dump Logical_Flow.
4. **Evaluate** output from the 10 most important commands: `show`, `lflow-list`, `list Port_Binding`, `list Chassis`, `lr-route-list`, through a 9-attribute anatomy table with "red flag if" guidance.
5. **Synthesize** a decision matrix of the form "in situation X use command Y", serving the on-call engineer in selecting the correct tool quickly.

## 13.14.1 CLI architecture of ovn-nbctl and ovn-sbctl

### 13.14.1.1 Two tools, two roles

`ovn-nbctl` and `ovn-sbctl` are two separate CLI tools in the package `ovn-common` (Ubuntu 22.04) or `ovn-utilities` (RHEL). Their architecture is identical (both are OVSDB IDL clients), but they connect to two different databases:

- **`ovn-nbctl`** connects to **NBDB** (default port 6641 TCP or Unix socket `/var/run/ovn/ovnnb_db.sock`). NBDB holds the **logical configuration intent** (Logical_Switch, Logical_Router, ACL, and so on). This is the tool that the CMS (Neutron, OVN-Kubernetes) calls indirectly to write configuration.
- **`ovn-sbctl`** connects to **SBDB** (default port 6642 TCP or Unix socket `/var/run/ovn/ovnsb_db.sock`). SBDB holds the **state compiled** by ovn-northd: Logical_Flow, Port_Binding, Chassis, Datapath_Binding. This is the **debug and manual intervention** tool for the on-call engineer.

> **Rule 5, control principle:** A production engineer MUST NOT use `ovn-sbctl` to configure logical intent (for example, to create a Port_Binding manually). All logical configuration must flow through NBDB to ovn-northd to SBDB. SBDB is the output of compilation; writing into it directly will be overwritten or cause inconsistency. The only legitimate use case for `ovn-sbctl set` and `add` is **chassis intervention** when a ghost chassis fails to release itself (see §13.14.5).

### 13.14.1.2 Connection, daemon mode, tracing

Each time `ovn-nbctl <command>` runs, the tool performs:

1. Read the environment variable `OVN_NB_DB` or the parameter `--db=` to learn the URL.
2. Open a connection (TCP or Unix) and send `monitor_cond_since` to obtain an OVSDB snapshot.
3. Build the IDL in memory (about 5 to 50 MB for a medium cluster).
4. Execute the command (read or write rows and relations).
5. Commit the transaction if there are changes.
6. Close the connection.

Steps 2 and 3 are the **bottleneck**: in a cluster with more than 10,000 objects, a single run may take 200 to 800 ms just to initialise the IDL. In a batch workflow (a CMS issuing 100 commands per second), this is a major obstacle.

**Daemon mode (the solution):**

```bash
# Start the daemon
[host]# ovn-nbctl --detach --pidfile=/var/run/ovn/ovn-nbctl.pid \
                  --log-file=/var/log/ovn/ovn-nbctl.log \
                  --db=tcp:nb1:6641,tcp:nb2:6641,tcp:nb3:6641
# Subsequent commands use the daemon socket:
[host]# export OVN_NB_DAEMON=$(ovn-nbctl get-daemon-socket)
[host]# ovn-nbctl ls-add my-ls
# ... command runs in less than 5 ms instead of 200 to 800 ms
```

The daemon keeps the IDL session warm and shares it across invocations. A production CMS typically runs the daemon as a long-lived process.

**Tracing options:**

- `--verbose` (`-v`) enables debug logging.
- `--timeout=N` sets the transaction timeout.
- `--no-wait` does not wait for commit confirmation (dangerous, use only for fire-and-forget).
- `--dry-run` for some subcommands: prints the transaction JSON without committing.

### 13.14.1.3 Source code anchor (branch-22.03 verified)

- [`utilities/ovn-nbctl.c`](https://github.com/ovn-org/ovn/blob/branch-22.03/utilities/ovn-nbctl.c) is 7244 lines long. The `ctl_command_syntax` table at about line 6972 registers all 97 subcommands. `main()` at about line 130 parses options and bootstraps the IDL.

## 13.14.2 ovn-nbctl, 97 subcommands grouped into 12 categories

### 13.14.2.1 Generic group (3 commands)

| Command | Purpose | Example |
|---------|---------|---------|
| `init` | Create a new NBDB schema (run only when bootstrapping a fresh database) | `ovn-nbctl init` |
| `sync` | Wait for northd to finish compiling the current NB into SB | `ovn-nbctl sync` |
| `show` | Print a hierarchical view of the entire NBDB | `ovn-nbctl show` |

### 13.14.2.2 Logical Switch and Logical Switch Port group (28 commands)

**Logical Switch (LS):**

| Command | Purpose |
|---------|---------|
| `ls-add <name>` | Create a new LS |
| `ls-del <ls>` | Delete an LS |
| `ls-list` | List all LSes |

**Logical Switch Port (LSP):**

| Command | Purpose |
|---------|---------|
| `lsp-add <ls> <port>` | Create an LSP attached to an LS |
| `lsp-del <port>` | Delete an LSP |
| `lsp-list <ls>` | List the LSPs of an LS |
| `lsp-set-addresses <port> [<addresses>...]` | Set the MAC and IP for the LSP. Format: `"aa:bb:cc:dd:ee:ff 10.0.0.1"`, `unknown` (allow any MAC), `dynamic` (automatic IPAM), `router` (LSP connected to an LRP) |
| `lsp-get-addresses <port>` | Read back the addresses |
| `lsp-set-port-security <port> [<addresses>...]` | Set anti-spoofing. Default: synchronised with `lsp-set-addresses` |
| `lsp-get-port-security <port>` | Read back |
| `lsp-set-type <port> <type>` | Set the type: `""` (vif), `localnet`, `localport`, `l2gateway`, `l3gateway`, `router`, `virtual`, `external`, `remote` |
| `lsp-get-type <port>` | Read the type |
| `lsp-set-options <port> <KEY=VALUE>...` | Set the options dictionary. For example: `network_name=physnet1` for localnet, `router-port=lrp-name` for router type |
| `lsp-get-options <port>` | Read the options |
| `lsp-set-enabled <port> true\|false` | Enable or disable the LSP |
| `lsp-get-enabled <port>` | Read the enabled state |
| `lsp-get-up <port>` | Read the `up` field (true when a chassis has claimed it successfully) |
| `lsp-set-dhcpv4-options <port> <DHCP_Options uuid>` | Link the LSP to DHCPv4 options |
| `lsp-get-dhcpv4-options <port>` | Read back |
| `lsp-set-dhcpv6-options <port> <DHCP_Options uuid>` | Same as above for DHCPv6 |
| `lsp-get-dhcpv6-options <port>` | Read |
| `lsp-get-parent <port>` | Read parent_name (for a VLAN child port) |
| `lsp-get-tag <port>` | Read the VLAN tag |
| `lsp-get-ls <port>` | Reverse lookup: from a port find the LS that contains it |

### 13.14.2.4 ACL group (3 commands)

| Command | Purpose |
|---------|---------|
| `acl-add {<sw>\|<pg>} <direction> <priority> <match> <action>` | Add an ACL. Direction: `from-lport`, `to-lport`. Priority 0 to 32767. Action: `allow`, `allow-related`, `allow-stateless`, `drop`, `reject`. Match is an OVN expression (for example, `inport == "lsp1" && ip4.dst == 10.0.0.0/8`) |

### 13.14.2.5 Port Group group (3 commands)

| Command | Purpose |
|---------|---------|
| `pg-add <pg> [<port>...]` | Create a Port_Group. A Port_Group is used to consolidate ACLs: one ACL on a Port_Group applies to every member port (reducing flows by 5 to 10 times compared with one ACL per port) |
| `pg-set-ports <pg> [<port>...]` | Update the port list (full replacement) |
| `pg-del <pg>` | Delete a Port_Group |

### 13.14.2.6 Load Balancer group (3 commands)

| Command | Purpose |
|---------|---------|
| `lb-add <lb> <vip> <ips> [<protocol>]` | Create an LB. `vip` is in the form `ip:port` or `[ipv6]:port`. `ips` is a CSV list of pool members, for example `1.1.1.1:80,2.2.2.2:80`. Protocol: `tcp`, `udp`, `sctp` |
| `lb-del <lb> [<vip>]` | Delete the LB or only one VIP |
| `lb-list [<lb>]` | List LBs |

An LB must be attached to an LS or LR after creation:

```bash

[host]# ovn-nbctl lr-lb-add my-router my-lb
```

## 13.14.4 Anatomy Template A, 10 important outputs

### Anatomy 1: `ovn-nbctl show`

> **When to run:** Overview of the current logical intent. First command when onboarding a new cluster or starting to debug.

```
switch f3a8 (web-ls)
    port web1
        addresses: ["aa:bb:cc:00:00:01 10.0.0.10"]
    port web2
        addresses: ["aa:bb:cc:00:00:02 10.0.0.11"]
    port lsp-router
        type: router
        router-port: lrp-web
router 89e1 (web-router)
    port lrp-web
        mac: "aa:bb:cc:99:00:01"
        networks: ["10.0.0.1/24"]
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| `switch` count | Matches the LS count CMS expects | 1 to 2 LS missing | Many LS missing | CMS sync lag; check Neutron or Kubernetes reconcile |
| `port` count per LS | Matches the active VM or container count | 1 to 2 ports missing | Duplicate IP warning | VM teardown did not clean the LSP |
| `addresses` MAC duplicate | Each MAC unique | One MAC on two LSPs | Multiple conflicts | An old LSP was not deleted; check `lsp-list` |
| `type: router` LSP | Matches the count of connected LRPs | Mismatch | LSP `router` without `router-port` option | One side was removed (LRP or LSP) |
| `mac` LRP format | `xx:xx:xx:xx:xx:xx` | (none) | Empty | Set incorrectly during `lrp-add` |
| `networks` CIDR | IP does not overlap with the LS | Overlap | LR has two LRPs in the same subnet | Wrong topology |

**Failure scenarios:**

- LSP `type: router` missing `router-port`: the peer LRP does not exist. Run `ovn-nbctl lsp-set-options <lsp> router-port=<lrp>` to relink.
- LR has two LRPs with the same `networks` 10.0.0.1/24: routing conflict. Move one to a different subnet.

### Anatomy 2: `ovn-sbctl list Chassis`

> **When to run:** Check which chassis are registered with the SB DB, their encap type, hostname, and transport zone.

```
_uuid               : a1b2c3d4-...
external_ids        : {ovn-bridge-mappings="physnet1:br-ex", \
                       ovn-cms-options="enable-chassis-as-gw", \
                       ovn-encap-ip="192.0.2.10"}
hostname            : "compute-01.example.com"
name                : "55b4d9e8-..."
nb_cfg              : 142
encaps              : [enc1, enc2]
other_config        : {bridge-mappings="physnet1:br-ex"}
transport_zones     : [zone1]
vtep_logical_switches : []
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| `name` UUID | Matches `system-id` of ovs-vsctl | Different | Empty | ovn-controller has not registered; check the log |
| `hostname` | Matches the actual OS `hostname` | Different | (none) | DNS lag or a rename that did not sync |
| `nb_cfg` | Matches SB_Global.nb_cfg within plus or minus 5 | Diverges by 10 to 50 | Diverges by more than 100 | Chassis lags ovn-northd sync |
| `encaps` count | One or more encap (Geneve, VXLAN, or STT) | 0 | (none) | The chassis has no tunnel; pinging another chassis will fail |
| `external_ids:ovn-encap-ip` | Routable underlay IP | IP in a different subnet | Empty | Check `ovs-vsctl get Open_vSwitch . external_ids` on the chassis |
| `transport_zones` | Matches the expected zone | Empty (means all-tenant) | Wrong zone | Tenant isolation is broken |

**Failure scenarios:**

- `nb_cfg` diverges by more than 100 from SB_Global: the chassis is stuck. Restart ovn-controller; if still diverged, the SB DB Raft is slow or the chassis cannot reach SB.
- Two chassis with the same `name` UUID: very rare, caused by cloning a VM image without deleting `system-id`. Delete one chassis with `ovn-sbctl chassis-del`.

```
parent_port         : []
requested_chassis   : []
tag                 : []

type                : ""

```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| `chassis` UUID | Matches the local chassis | Empty | UUID does not exist | Stale binding; see chassis-del |
| `requested_chassis` | Matches `chassis` (after migration) | Different | Different for more than 30 seconds | Migration stuck; check the ovn-controller log |
| `up` | `true` for a running VM | `false` for a stopped VM | `false` for a running VM | claim race; see §13.7.8 |
| `type` | `""` (vif) for a VM | `localnet`, `l2gateway`, and so on | (none) | (per spec) |
| `tunnel_key` | Unique within a datapath | Duplicate | 0 | Compile bug; report upstream |
| `mac` | MAC plus IP format | Empty (for `unknown`) | (none) | Not yet set |

### Anatomy 4: `ovn-sbctl lflow-list <datapath>`

> **When to run:** Inspect Logical_Flow that ovn-northd compiled from NB intent.

```
Datapath: "web-ls" (f3a8...)  Pipeline: ingress
  table=0 (ls_in_port_sec_l2),    priority=50, match=(inport == "web1"), action=(next;)
  table=1 (ls_in_port_sec_ip),    priority=90, match=(inport == "web1" && eth.src == aa:bb:cc:00:00:01 && ip4.src == {10.0.0.10}), action=(next;)
  ...
  table=27 (ls_in_l2_lkup),       priority=50, match=(eth.dst == aa:bb:cc:00:00:02), action=(outport = "web2"; output;)
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| Tables per pipeline | 27 ingress plus 10 egress (LS, OVN 22.03) | Diverges | (none) | Version baseline mismatch |
| `priority` | 1 to 32767 valid | 0 (default-deny) | More than 65535 | Compile error |
| `match` syntax | Valid OVN expression | Warning in the log | Empty | Compile fallback |
| `action` syntax | Valid | (none) | Empty (silent drop) | (per spec) |
| Missing table N | Tables 0 to 26 all have entries | 1 to 2 missing | Many missing | northd compile error; check northd.log |

### Anatomy 6: `ovn-nbctl acl-list <ls>`

```
from-lport  1000 (inport == "web1" && tcp.dst == 22)  drop
from-lport   500 (ip4.src == 10.0.0.0/8)              allow-related
to-lport    1500 (outport == "web1")                  allow
```

| Field | Healthy | Warning | Critical | Diagnostic hint |

## 13.14.6 Guided Exercise S67': New tenant + multi-tier ACL

**Goal:** Deploy tenant `T1` with two tiers (web and db), an external router, and an ACL that denies cross-tier traffic except on one port.

**Steps:**

```bash
# 1. Create the Logical Switch for each tier
[host]# ovn-nbctl ls-add t1-web-ls
[host]# ovn-nbctl ls-add t1-db-ls

# 2. Create VM ports
[host]# ovn-nbctl lsp-add t1-web-ls web1
[host]# ovn-nbctl lsp-set-addresses web1 "aa:bb:cc:11:00:01 10.1.1.10"
[host]# ovn-nbctl lsp-add t1-db-ls db1
[host]# ovn-nbctl lsp-set-addresses db1 "aa:bb:cc:22:00:01 10.2.2.10"

# 3. Create the Logical Router
[host]# ovn-nbctl lr-add t1-router
[host]# ovn-nbctl lrp-add t1-router lrp-web aa:bb:cc:99:00:01 10.1.1.1/24
[host]# ovn-nbctl lrp-add t1-router lrp-db  aa:bb:cc:99:00:02 10.2.2.1/24

# 4. Connect the LS to the LR (LSP type=router)
[host]# ovn-nbctl lsp-add t1-web-ls lsp-to-router-web
[host]# ovn-nbctl lsp-set-type lsp-to-router-web router
[host]# ovn-nbctl lsp-set-addresses lsp-to-router-web router
[host]# ovn-nbctl lsp-set-options lsp-to-router-web router-port=lrp-web

[host]# ovn-nbctl lsp-add t1-db-ls lsp-to-router-db
[host]# ovn-nbctl lsp-set-type lsp-to-router-db router
[host]# ovn-nbctl lsp-set-addresses lsp-to-router-db router
[host]# ovn-nbctl lsp-set-options lsp-to-router-db router-port=lrp-db

# 5. Port_Group per tier
[host]# ovn-nbctl pg-add t1-web-pg web1
[host]# ovn-nbctl pg-add t1-db-pg  db1

# 6. ACL: deny web to db except on port 5432 (Postgres)
[host]# ovn-nbctl acl-add t1-web-pg from-lport 1000 \
            "outport == @t1-db-pg && tcp.dst != 5432" drop
[host]# ovn-nbctl acl-add t1-web-pg from-lport 999 \
            "outport == @t1-db-pg && tcp.dst == 5432" allow-related

# 7. Verify
[host]# ovn-nbctl show
[host]# ovn-nbctl acl-list t1-web-pg
[host]# ovn-sbctl lflow-list t1-web-ls | grep "ls_in_acl"
```

**Expected output (doc-plausible, to be verified once a lab is available):**

```
$ ovn-nbctl acl-list t1-web-pg
from-lport  1000 (outport == @t1-db-pg && tcp.dst != 5432)  drop
from-lport   999 (outport == @t1-db-pg && tcp.dst == 5432)  allow-related
```

**Lessons learned:**

1. The tenant-creation workflow needs 8 to 12 `ovn-nbctl` commands. Daemon mode brings the time down from about 10 seconds to about 500 milliseconds.
2. `lsp-set-type router` plus `lsp-set-options router-port=` is the MANDATORY pair to connect an LS to an LR. Without `router-port`, packets entering the LRP will be dropped.
3. Port_Group plus ACL `outport == @<pg>` is the standard pattern for multi-tier security. Compared with two ACLs from web1 to db1 (one rule per VM pair), the Port_Group pattern scales as O(N) instead of O(N squared).

**Cleanup:**

```bash
[host]# ovn-nbctl ls-del t1-web-ls
[host]# ovn-nbctl ls-del t1-db-ls
[host]# ovn-nbctl lr-del t1-router
[host]# ovn-nbctl pg-del t1-web-pg
[host]# ovn-nbctl pg-del t1-db-pg
```

(The ACLs are cascade-deleted when the Port_Group is deleted.)

## 13.14.7 Capstone POE Phase I.B1: ovn-sbctl debug vs ovn-nbctl reconfig

> **Context:** Production cluster, VM `web1` reports "no network" after a live migration from chassis A to chassis B.

**Claim to verify:** "Running `ovn-sbctl set Port_Binding web1 chassis=<chassis-B>` to force the bind is the correct fix."

**Predict:** False. Forcing the binding through ovn-sbctl is a dangerous intervention for production.

**Observe (assumed, verify in a lab):**

- Run `ovn-sbctl set Port_Binding web1 chassis=...`. Flows appear on br-int of chassis B within a few seconds.
- After 30 seconds, ovn-northd recomputes from NB and resets `Port_Binding.chassis` back to the old value (or empty), so the flows disappear and the VM loses the network a second time.
- At the same time, `requested_chassis` of the Port_Binding in NB does not reflect the intent, so the CMS (Neutron) sees a divergence and may revert.

**Explain:**

1. SBDB is the **output** of the NB to SB compile. Writing directly to SBDB will be overwritten on every compile cycle (about 50 milliseconds to 5 seconds depending on workload).
2. NBDB is the source of truth. The correct command is:
   - `ovn-nbctl lsp-set-options web1 requested-chassis=<chassis-B>` (set the intent in NB).
   - ovn-northd recompiles, updates `Port_Binding.requested_chassis`, and ovn-controller on B claims the port.
3. The only legitimate use of `ovn-sbctl set` on Port_Binding is when NB is already correct but SB is stuck (very rare, debug only). The trade-off must be understood: from the moment of the force-write until the next northd recompute.

**Falsification test:**

- Force-bind through `ovn-sbctl set`, then wait 30 seconds.
- Run `ovn-nbctl get Logical_Switch_Port web1 options` to check whether `requested-chassis` is present.
- If not, northd will revert and the prediction holds. If present (someone set NB at the same time), the binding may persist.

**Where the claim is wrong:** Using `ovn-sbctl set` for logical intent is an **anti-pattern**. The correct tool is `ovn-nbctl` (the source of truth). `ovn-sbctl` is used only to (a) read or inspect, (b) manually run chassis-del for a ghost node, or (c) debug in the short term with a clear understanding of the overwrite risk.

> **Five-pillar rule (control rule):** Logical intent goes to NBDB through ovn-nbctl. Observation and chassis intervention go to SBDB through ovn-sbctl. DO NOT swap them.

## 13.14.8 Key points to remember

**One.** `ovn-nbctl` (97 commands) configures logical intent in NBDB. `ovn-sbctl` (15 commands) inspects and intervenes in SBDB. The roles are distinct and must not be swapped (the five-pillar rule).

**Two.** Daemon mode reduces latency by 100 to 1000 times for batch workloads. Production CMSes always use daemon mode.

**Three.** Command grouping: 12 groups for ovn-nbctl (Generic, LS plus LSP, LR plus LRP, ACL, PG, LB, DHCP, QoS, HA Chassis, CoPP, Connection, SSL). The 15 ovn-sbctl commands focus on debug and chassis lifecycle.

**Four.** `lsp-set-type router` plus `lsp-set-options router-port=` is the MANDATORY pair for an LS to LR connection. Missing either one causes silent packet drop.

**Five.** Port_Group plus ACL pattern: `acl-add <pg> from-lport <pri> "outport == @<other-pg>" <action>`. Scales as O(N) instead of O(N squared). The benefit is visible in production with more than 100 tenants.

**Six.** The `nb_cfg` gauge: `ovn-nbctl get NB_Global . nb_cfg` versus `ovn-sbctl get SB_Global . nb_cfg`. A small gap (at most 5) is healthy. A gap greater than 100 indicates compile lag.

**Seven.** `ovn-sbctl set Port_Binding chassis=...` is an **anti-pattern** outside short-term debugging. The correct approach is to set `requested-chassis` through `ovn-nbctl lsp-set-options`.

By default ovn-nbctl returns only an exit code. Adding `--print-wait-time` writes one stderr line of the form `Time spent on waiting: 137 ms` per transaction, allowing scripts to track p50, p95, and p99 percentiles and warn early about convergence drift. Typical use case: an orchestrator accepts p95 SB wait under 500 milliseconds and alerts when it exceeds the threshold. Anti-pattern: using `--print-wait-time` without enabling `--wait=...`. The output is empty, which can mislead an operator into thinking the wait was 0 milliseconds when in fact no wait happened. **Interpreting `-u <path>` (gap closure, v3.6 Phase 1).** By default the ovn-nbctl daemon creates its socket in `$OVS_RUNDIR` (typically `/var/run/ovn`). When the client invocation also uses `-u /tmp/job-A.sock`, the command reaches the right daemon. Anti-pattern: starting the daemon with `-u` while the client does not pass `-u`. The client connects to the default path, sees no daemon there, and falls back to the slow IDL connection. Cross-reference: the same `-u` semantics exist for `ovs-vsctl --daemon` and `ovn-sbctl --daemon`.

### 13.14.9.2 `ovn-trace`, the logical-pipeline simulator for what-if debugging

**Concept.** `ovn-trace` is a standalone C utility (binary `/usr/bin/ovn-trace`, installed with the `ovn-common` package). It reads SBDB and simulates the path of a hypothetical packet through the OVN **logical pipeline**, without injecting a real packet, without sending anything over the wire, and without modifying state. The input is a pair `(datapath, microflow_expression)`: the datapath is a Logical_Switch or Logical_Router name, and the microflow is an OVN-language expression describing the packet header and metadata (`inport == "lsp1" && eth.src == 50:54:00:00:00:01 && ip4 && ip4.dst == 10.0.0.10 && tcp && tcp.dst == 80`). The output is a stage-by-stage trace tree (`ls_in_port_sec_l2`, `ls_in_pre_acl`, `ls_in_acl`, `ls_in_lb`, ..., `lr_in_admission`, `lr_in_ip_input`, ...) showing which match hit, which action ran, the register state after each step, and pipeline transitions (ingress to egress, LS to LR, LR to LS). Category: standalone CLI tool, debug class, read-only, what-if.

**History and origin.** Ben Pfaff (Nicira and VMware) committed `utilities/ovn-trace.c` for the first time during 2016 to 2017, when OVN was being split out from OVS and the first 24-stage pipeline was being built. The file's copyright header reads "Copyright 2016-2017 by Nicira, Inc.". The original goal was to let an engineer answer "if I send packet X, which stages does it traverse, where does it get dropped, what does NAT translate the address to" without bringing up a real VM and running tcpdump. OVN 2.7 (2017) shipped `ovn-trace` officially. Before that, engineers had to dump `Logical_Flow` with `ovn-sbctl lflow-list` and simulate in their head, which took hours per complex packet. OVN 20.06 added `--lb-dst` for Load_Balancer hairpin debugging. OVN 21.06 split `--minimal`, `--summary`, and `--all` into separate output modes. OVN 22.03 (the curriculum baseline) enabled `--friendly-names` by default, replacing UUIDs with Logical_Switch and Logical_Router names so the output is readable. OVN 24.03 improved SBDB read performance for large clusters.

**Architectural placement.** `ovn-trace` runs in the **client-side tooling layer**, not in the data plane (br-int) or the control plane (ovn-northd, ovn-controller). It connects to SBDB over OVSDB JSON-RPC just like `ovn-sbctl`, reading `Datapath_Binding`, `Port_Binding`, `Logical_Flow`, `Multicast_Group`, `MAC_Binding`, and `DNS`. The entire simulation runs inside the `ovn-trace` process, with no round trip to `ovn-controller` or any chassis. Scope: per invocation (each run is a snapshot of SBDB at that moment). It differs from `ofproto/trace` (Part 9.25): `ofproto/trace` simulates the physical OpenFlow pipeline on a specific bridge, whereas `ovn-trace` simulates the abstract logical pipeline at the OVN level.

**Role.** It is step 1 in the three-layer OVN debug workflow: logical (`ovn-trace`), physical (`ofproto/trace`), forensic (`ovn-detrace`). `ovn-trace` answers questions at the **intent** level: "given the current NBDB, is this packet allowed through, which logical stages does it traverse, is it blocked by an ACL, is it NATed, is the routing correct". If the logical pipeline already drops the packet, there is no need to descend to the physical layer. If the logical pipeline passes but the real packet still fails, the bug is in binding or physical layers (chassis has not picked up the binding, tunnel is broken, conntrack is stuck), at which point use `ofproto/trace` and `ovn-detrace`. Scope of role: the entire logical lifecycle of a packet, from the ingress LSP to the egress LSP (or drop at some stage).

**Why it exists.** OVN compiles NBDB (intent) into SBDB `Logical_Flow` (a few thousand entries for a medium cluster), and `ovn-controller` translates that into OpenFlow on br-int (tens of thousands of rules). When a user declares an incorrect ACL, route, or LB, the error is not visible immediately; it surfaces only when a real packet runs and gets dropped. Before `ovn-trace`, the debug pattern was "real-world experimentation": bring up a VM, ping, tcpdump, guess. Pain points: (a) slow, each experiment costs several minutes of setup; (b) not reproducible, real packets are affected by conntrack state, tunnel state, and MAC learning; (c) not safe in production, where you cannot "just try" every combination. `ovn-trace` fills the gap: pure simulation, no impact, runnable thousands of times per hour on a production cluster without side effects. This tool is unique to OVN; plain OVS has no equivalent because OVS has no notion of "logical intent" separated from physical flows.

**Problems it solves.** Two main groups. (1) **What-if simulation**: "if I add this ACL, does TCP port 8080 from tenant A still pass?" Run `ovn-trace` before `ovn-nbctl acl-add` to verify. (2) **Drop diagnosis**: "why does TCP port 80 from vm1 not reach vm2?" `ovn-trace` points to the first dropping stage (`ls_in_acl` priority 2002, or `lr_in_ip_routing` with no matching route, or `ls_in_port_sec_ip` because the source IP does not match `lsp-set-addresses`). It does not fully replace real-world debugging, but it cuts triage time by 80 to 90 percent because it identifies the layer (logical, physical, or binding) in seconds.

**Importance.** Tier 1, mandatory for every OVN engineer. An engineer who does not know `ovn-trace` is reduced to manual guesswork through `ovn-sbctl lflow-list` during an OVN incident. In a production OpenStack environment with hundreds of Logical_Switch instances and thousands of ACLs, lack of `ovn-trace` raises MTTR (mean time to resolution) from minutes to hours. Compared with `ovn-detrace` and `ofproto/trace`, `ovn-trace` is usually the FIRST tool to reach for because it is the fastest (no ssh into a chassis is required, only an SBDB connection).

**How it works.** `ovn-trace` reads all of SBDB into memory (function `read_db()` in `utilities/ovn-trace.c`) and builds an internal datapath, port, and flow table. When it receives a microflow expression, the parser (`parse_options()` plus the expression parser) converts the expression into an internal flow struct with prerequisites applied automatically (specifying `ip4.dst` automatically applies `eth.type == 0x0800`). The core function `trace__()` is recursive: at each stage (`table_id`) it calls `ovntrace_flow_lookup()` to find the highest-priority Logical_Flow that matches the microflow, then calls `trace_actions()` to execute each OVN logical action (`next;`, `outport = ...;`, `ct_lb;`, `ct_next;`, `arp { ... };`, `output;`, and so on). Per-action dispatchers: `execute_output()` handles output (a single port or a multicast group, transitioning between ingress and egress pipelines); `execute_ct_next()` handles conntrack state transitions; other handlers cover `nd_na`, `arp`, `dhcp_offer`, `set_meter`, and so on. `ovntrace_node_append()` builds the hierarchical trace tree, where each node is a stage or an action sub-call. `ovntrace_node_print_details()` formats the output as indented text. Invariant: the simulation is deterministic per SBDB snapshot; if SBDB does not change, the same input produces the same output. Non-determinism appears only with `select` groups (Load_Balancer multi-backend, ECMP), where `--all` shows every candidate path.

**Classification.** Standalone CLI tool, debug class, read-only. Same family of OVN debug tools: `ovn-detrace` (cookie reverse-decoder, §13.14.9.3 below), `ovn-sbctl lflow-list` (raw logical-flow dumper), `ofproto/trace` through `ovs-appctl` (OVS physical trace, Part 9.25). It differs from `ovn-nbctl` and `ovn-sbctl` in that it has no write subcommand and accepts only one input form (a microflow expression). It differs from `ofproto/trace` in that it needs only an SBDB endpoint, not ssh to a specific chassis.

**How an engineer masters this.** Four skills. (1) **Compose microflow expressions**: master the 24+ commonly used fields (`inport`, `outport`, `eth.src/dst/type`, `ip4/ip6.src/dst`, `ip.proto`, `ip.ttl`, `tcp/udp.src/dst`, `tcp.flags`, `icmp4/icmp6.type/code`, `arp.op/spa/tpa/sha/tha`, `nd.target`, `ct.new/est/rpl/inv/trk`, `vlan.tci`, `reg0..reg15`, `outport`, `flags.loopback`); know the prerequisite chains (specifying `tcp.dst` requires `ip4` or `ip6` together with `ip.proto == 6`); know the operators `&&`, `==`, `!=`, and bitmask values (`reg0[0] == 1`). (2) **Read the output**: distinguish ingress versus egress pipelines, read the stage name and map it back to NBDB intent (`ls_in_acl` corresponds to an ACL on a Logical_Switch, `lr_in_ip_routing` corresponds to a static_route on a Logical_Router), read the actions `next;` (move to the next stage), `outport = ...; output;` (emit on a logical port), `drop;` (terminal, no delivery). (3) **Pick an output mode**: `--detailed` (default, verbose, every register state), `--summary` (compact, only stage and action), `--minimal` (only packet modification and output, skipping stages that contain only `next;`), `--all` (run all three). (4) **Decision tree when the trace contradicts your expectation**: drop at a stage means reading the match expression of the Logical_Flow at that stage with `ovn-sbctl lflow-list <datapath> | grep <stage>`; trace passes but the real packet fails means stepping into `ofproto/trace` plus `ovn-detrace`; "no datapath found" means checking the datapath name with `ovn-nbctl show`.

**Usage workflow.** A standard five-step workflow for production debugging.

```bash
# Step 1: simulate the logical pipeline, determine pass or drop
ovn-trace ls-prod \
  'inport == "lsp-vm1" && eth.src == 50:54:00:00:00:01 \
   && eth.dst == 50:54:00:00:00:02 && ip4 \
   && ip4.src == 10.0.0.1 && ip4.dst == 10.0.0.2 \
   && ip.ttl == 64 && tcp && tcp.dst == 80 && tcp.src == 33000'

# Step 2: if the trace drops, raise verbosity to see exactly which Logical_Flow matched
ovn-trace --detailed ls-prod '...'

# Step 3: if the trace passes but the real packet fails, drop to the physical layer
ovs-appctl ofproto/trace br-int 'in_port=<ofport>,tcp,...'

# Step 4: annotate the physical trace with ovn-detrace
ovs-appctl ofproto/trace br-int '...' | ovn-detrace \
  --ovnnb=tcp:nb:6641 --ovnsb=tcp:sb:6642

# Step 5: use --ovs to align the logical view with the physical view on a specific chassis
ovn-trace --ovs=tcp:chassis-host:6640 ls-prod '...'
```

Best practice: always wrap the microflow in single quotes so the shell does not expand `&&`; rely on `--friendly-names`, which is on by default in OVN 22.03; for a Raft cluster you can connect to any SBDB replica (read-only); set `OVN_SB_DB` to avoid typing `--db` every time.

**When something goes wrong.** Typical situations where an engineer must reach for `ovn-trace`. (a) **East-west drop**: vm1 cannot ping vm2 on the same Logical_Switch; run `ovn-trace ls-blue 'inport=="vm1" && eth.dst==<vm2-mac> && ip4 && ip4.dst==<vm2-ip> && icmp4 && icmp4.type==8'` to find the first dropping stage. (b) **North-south fail**: VM cannot reach the Internet; run `ovn-trace lr-edge '...'` from the Logical_Router perspective to confirm `lr_in_ip_routing` hits and NAT is correct. (c) **ACL verification before commit**: before `ovn-nbctl acl-add` for a new rule, run `ovn-trace` for 5 to 10 typical traffic patterns to confirm legitimate traffic is not broken. (d) **DHCP debug**: `ovn-trace --all` for UDP broadcast to port 67 to see whether `ls_in_dhcp_options` generates a response. (e) **Load_Balancer**: use `--lb-dst=<backend-ip>` to confirm a VIP packet is DNATed to the right backend. Cross-link Part 20.7 §packet flow tracing tutorial gradient L1 to L5, Part 20.2 §troubleshooting drop, Part 20.0 §incident decision tree.

**Tightly related to.** Five components. (1) **SBDB `Logical_Flow` table** (Part 13.1): the primary data source; each stage in the trace reads the corresponding Logical_Flow. (2) **`ovn-northd`** (Part 13.8): the side that compiles NBDB to SBDB and produces the Logical_Flow that `ovn-trace` simulates; if northd is not synced, `ovn-trace` sees a stale version. (3) **`ofproto/trace`** (Part 9.25): the parallel tool that simulates physical OpenFlow; usually used right after `ovn-trace`. (4) **`ovn-detrace`** (§13.14.9.3): the reverse bridge from physical to logical, complementary to `ovn-trace`. (5) **Pipeline stage catalogue** (Parts 13.16 and 13.19): stage names in the trace output (`ls_in_*`, `lr_in_*`, `ls_out_*`, `lr_out_*`) must be mapped to `table_id` to be understood. Loosely related: `ovn-nbctl` (only indirect, through northd compile), `ovn-controller` (no direct effect because `ovn-trace` bypasses br-int).

**Version differences.** OVN 2.7 (2017) shipped `ovn-trace` for the first time with verbose output by default and no mode selector. OVN 20.06 added `--lb-dst` for Load_Balancer hairpin testing. OVN 21.06 split `--detailed`, `--summary`, `--minimal`, and `--all`. OVN 22.03 (the curriculum baseline) enabled `--friendly-names` by default, replacing datapath UUIDs with Logical_Switch and Logical_Router names so the output is readable. OVN 24.03 optimised SBDB reads for clusters with tens of thousands of datapaths (reducing read time from several seconds to under one second). Field syntax (the microflow expression) has been stable since OVN 2.7; new actions (for example, `dhcp6_reply`, `chk_lb_hairpin_reply`) are added in each release without breaking backward compatibility. For production OVN 22.03.x, the example syntax in this document is usable as is.

**Source code references.** Main file: `utilities/ovn-trace.c` (OVN repository, branch-22.03, about 3500 lines of C). Key functions:

- `trace__()`: the core recursive simulator, calling `ovntrace_flow_lookup()` at each table.
- `trace_actions()`: parses and dispatches OVN logical actions.
- `execute_output()`: handles the output action, transitions the ingress and egress pipelines, and handles multicast.
- `execute_ct_next()`: simulates conntrack state transitions, using the `--ct` flag for the default state.
- The copyright header "Copyright 2016-2017 by Nicira, Inc." confirms the tool was created when OVN was split out from OVS.

Man page: `utilities/ovn-trace.8.xml` in the repository, rendered as section 8 (`man 8 ovn-trace`). Same family of C debug utilities: `utilities/ovn-sbctl.c`, `utilities/ovn-nbctl.c`. It differs from `ovn-detrace` (a Python script) because `ovn-trace` is written in C to parse expressions quickly and simulate large pipelines.

**Real-world incident.** A production OpenStack incident with 800 VMs, OVN 22.03.4, 12 chassis. Symptom: from the day a new ACL policy was deployed, tenant A reported that some HTTP requests from subnet `10.10.0.0/24` to the API server (LB VIP `10.20.0.50:8080`) timed out at random with about a 15 percent rate; tenant B (subnet `10.11.0.0/24`) using the same LB was unaffected. Investigation step 1: run `ovn-trace lr-edge 'inport == "lsp-vm-A" && eth.src == ... && ip4 && ip4.src == 10.10.0.5 && ip4.dst == 10.20.0.50 && ip.ttl == 64 && tcp && tcp.dst == 8080 && tcp.src == 33000'`. The trace hits `lr_in_ip_routing` and then enters `ls_in_pre_lb` and `ls_in_lb`. Step 2: `ovn-trace --all --lb-dst=10.20.0.51 ...` to force the simulator to pick a specific backend; the trace passes. Step 3: rerun without `--lb-dst` and read the `ls_in_lb` output carefully; it shows `ct_lb(backends=10.20.0.51,10.20.0.52)`, missing `10.20.0.53` that tenant A expected. Step 4: check NBDB with `ovn-nbctl lb-list lb-api`; the LB indeed has only two backends instead of the three required by the design. Step 5: root cause was that the deploy script during a rolling update ran `lb-del` for backend `10.20.0.53` but forgot to run `lb-add` to restore it. Fix: `ovn-nbctl --may-exist lb-add lb-api 10.20.0.50:8080 10.20.0.51:8080,10.20.0.52:8080,10.20.0.53:8080 tcp`. After 3 seconds, rerunning `ovn-trace --all` hits all three backends and the drop rate falls to zero. Value of `ovn-trace` here: among hundreds of LBs in NBDB, without `ovn-trace` the engineer would have had to read NBDB by hand and guess which backend was missing; with `ovn-trace --all`, the divergence surfaced in 30 seconds.

Variations: (a) change the ACL action from `drop` to `reject` and observe that the output stage adds an action that generates ICMP unreachable; (b) add a stateful `allow-related` rule at a higher priority and observe that the pipeline branches into `ct_commit`; (c) try a microflow without prerequisites (specifying `tcp.dst` but omitting `ip4`); `ovn-trace` returns a parser error; (d) run `ovn-trace` while `ovn-northd` is stopped: the trace still works (reading a stale SBDB snapshot) but does not reflect any uncompiled NBDB changes.

**Common failures and diagnostic signals.** Four main failure modes. (1) **Wrong SBDB endpoint**: `ovn-trace` cannot see SBDB; signal: "could not connect to OVN_Southbound" or a hang; diagnosis: check the `OVN_SB_DB` environment variable or the `--db` flag, and test with `ovn-sbctl --db=$OVN_SB_DB list Datapath_Binding | head`; recovery: point to the correct UNIX socket `unix:/var/run/ovn/ovnsb_db.sock` or TCP `tcp:sb-host:6642`. (2) **Wrong datapath name**: `ovn-trace` reports "datapath not found"; diagnosis: run `ovn-nbctl show` to inspect Logical_Switch and Logical_Router names; recovery: use the exact name or a UUID prefix; note that OVN 22.03 and later support friendly names, while older versions accept only UUIDs. (3) **Microflow expression syntax**: the parser rejects it; signal: "syntax error" with a line and column; diagnosis: verify prerequisites (using `ip.proto` requires `ip4` or `ip6`), and check quoting (a MAC such as `50:54:00:00:00:01` has no quotes, but a port name such as `lsp1` needs `"lsp1"`); recovery: simplify the expression and build it up incrementally. (4) **Trace versus reality mismatch**: `ovn-trace` passes but the real packet fails; signal: a contradiction between the trace and tcpdump; diagnosis: the cause is usually (a) SBDB has not finished compiling (wait a few seconds), (b) the chassis has not picked up the Port_Binding (check `ovn-sbctl list Port_Binding | grep chassis`), or (c) a tunnel is down between chassis (check `ovs-appctl ofproto/trace br-int`); recovery: step into `ofproto/trace` plus `ovn-detrace` to debug the physical layer.

**Comparison with other systems.** In NSX-T, the equivalent tool is **Traceflow** (UI) or the API `nsxcli > get logical-switch <uuid> traceflow`, which simulates a packet through the DFW and LR. Difference from `ovn-trace`: NSX-T Traceflow injects a real packet into the DFW pipeline to trace, so it has a small effect on state; `ovn-trace` is pure simulation with zero impact. Cisco ACI has the **Visibility and Troubleshooting Wizard**, which simulates end to end, but mixes control plane and data plane without a clean separation. Linux iptables has no close analogue (only `iptables -t filter -L -v -n` shows counters, not the path). The unique property of `ovn-trace` is that it is designed for the **declarative intent, compiled flow, physical execution** model characteristic of OVN, one of the reasons OVN is easier to debug than many other SDN controllers. The "what-if simulator on declarative SDN" pattern that `ovn-trace` embodies is worth studying for anyone designing a custom SDN controller.

### 13.14.9.3 `ovn-detrace`, the reverse bridge from an OpenFlow cookie back to a Logical_Flow

**Concept.** `ovn-detrace` is a Python utility (installed with the OVN distribution) that reads an OpenFlow trace text stream (the output of `ofproto/trace`, or of `ovs-ofctl dump-flows br-int`), extracts the 32-bit cookie from each line of the form `cookie=0xXXXXXXXX`, and looks it up in the SBDB `Logical_Flow` table to annotate that line with the original logical flow UUID, the datapath name (Logical_Switch or Logical_Router), the stage name (`ls_in_acl`, `lr_in_ip_input`, ...), the match expression and action in the OVN logical language, and further the name of the ACL, Load_Balancer, or Logical_Router_Static_Route in NBDB that produced the Logical_Flow. `ovn-detrace` itself does not send packets, does not call `ovs-vswitchd`, and does not modify state; it is a pure text-to-SBDB-lookup decoder. In other words: `ovn-trace` simulates the logical pipeline at the OVN-language level, `ofproto/trace` simulates the physical pipeline at the OpenFlow level, and `ovn-detrace` is the **bridge** between the two by decoding the cookies that ovn-northd encoded.

**History and origin.** From the very first OVN release (OVN 2.6, 2016), ovn-northd encoded the UUID of each `Logical_Flow` into a 32-bit cookie when compiling NBDB to SBDB; `ovn-controller` on each chassis attached that cookie to physical OpenFlow rules when installing them on `br-int`. Initially engineers had to open `ovn-sbctl list Logical_Flow | grep <prefix>` themselves to look up the mapping, which did not scale once SBDB held tens of thousands of entries. Ben Pfaff (Nicira and VMware) committed `utilities/ovn-detrace.in` to the master branch in late 2016 (the first Python script, about 250 lines, generated as `ovn-detrace` through the autotools `.in` template). The original goal was forensic debugging at the scale of thousands of datapaths, where manual grep does not scale. From OVN 20.06 the script moved to pure Python 3 to reduce dependencies. OVN 22.03 added `--no-leader-only` for Raft clusters and `--ovsdb-sock` for UNIX-socket connections. OVN 24.03 improved the output format with clearer block separators per cookie hit.

**Architectural placement.** `ovn-detrace` is a client-side script that runs on any host that can reach both databases (NBDB and SBDB). In the OVN stack it sits at the **tooling layer**, not in the data plane (br-int) or the control plane (ovn-northd, ovn-controller). It reads SBDB over OVSDB JSON-RPC just like `ovn-sbctl`, and reads NBDB over the same protocol to fetch NB resource names (ACL match strings, LB names). It does not subscribe to notifications; each invocation is a snapshot.

**Role.** In the three-step debug workflow (logical, physical, forensic), `ovn-detrace` covers the third step: it turns an opaque physical dump (hundreds of lines of `cookie=0x...,table=N,priority=...`) into a readable report where every line is annotated as "this is Logical_Flow Y, generated by ACL Z on Logical_Switch K". Without it, the on-call engineer must perform three steps manually: copy the cookie, run `ovn-sbctl find Logical_Flow logical_dp_group=...`, match the UUID prefix, and join with NB. For one packet that traverses 24 stages, that is 24 grep operations, which is not feasible during an incident.

**Why it exists.** OVN cleanly separates "logical intent" (NBDB) from "physical execution" (br-int OpenFlow), but the price is two abstraction layers between the user CLI and a real packet. When a packet goes wrong, the engineer must walk back from symptom to intent. The gap that `ovn-detrace` fills: existing trace tools (`ofproto/trace`) understand only OpenFlow, not Logical_Flow; `ovn-trace` understands only Logical_Flow, not whether a physical match actually happened. Before `ovn-detrace`, the common debug pattern was to keep three terminals open at once (ofproto/trace, ovn-sbctl list, ovn-nbctl list) and copy-paste UUIDs between them, which is error-prone given that UUIDs are 36 characters and cookies are only 8.

**Problems it solves.** Two specific groups: (1) **Inverse mapping** from a 32-bit cookie back to a 128-bit `Logical_Flow.UUID`. The mapping is not a pure function (the cookie is only the high 32 bits of the UUID), but it is precise enough to identify the Logical_Flow in most cases; `ovn-detrace` handles collisions by reporting multiple candidates when the prefix is ambiguous. (2) **In-place annotation** on a text trace: the input is any text file that contains cookies, and the output is the same file with annotations interleaved, preserving the order so it can be pasted directly into an incident report.

**Importance.** In the OVN mastery tier, `ovn-detrace` belongs in **tier 1 for forensic debugging**, on a par with `ovn-trace` and `ofproto/trace`. An OVN engineer who does not know `ovn-detrace` can debug small clusters (a few dozen flows) but is paralysed on large clusters (tens of thousands of flows on a single chassis). In particular, in a multi-tenant OpenStack or Neutron environment with hundreds of Logical_Switch instances, dump-flows returns several megabytes of text, and without `ovn-detrace` it is nearly impossible to attribute a rule to a tenant.

**How it works.** `ovn-detrace` reads stdin (or a file) line by line. For each line, it applies the regex `cookie=0x([0-9a-fA-F]+)` (8 hex characters). For each cookie it finds, the script issues an OVSDB query against SBDB: `monitor_cond Logical_Flow [["_uuid","includes",<cookie_as_uuid_prefix>]]` (in practice through IDL select; UUID-prefix matching happens client side). On a match, the script reads these columns: `logical_datapath` (datapath UUID), `pipeline` (`ingress` or `egress`), `table_id` (stage number), `priority`, `match` (microflow expression), `actions` (OVN logical language), `external_ids:stage-name` (stage-name string), and `external_ids:source` (when present, a link to an NB resource). Next, using `logical_datapath`, the script looks up `Datapath_Binding` to get `external_ids:name` (the Logical_Switch or Logical_Router name). Using `external_ids:source`, the script connects to NBDB and looks up the ACL, LB, or LR_Static_Route by UUID to get a human-readable match string. Finally, it prints the original line unchanged plus an indented annotation block prefixed with `*` or `>`, including all the information above. If a cookie is not found in SBDB (rare, due to a stale flow that has not been GCed yet, or a wrong SBDB connection), the script prints `cookie=0x... (no match)` and continues.

**Classification.** `ovn-detrace` is a **standalone CLI tool** (not a subcommand of `ovn-nbctl` or `ovn-sbctl`), installed into `/usr/bin` by the `ovn-common` package. Same family of OVN debug tooling: `ovn-trace` (logical simulator), `ovn-sbctl lflow-list` (logical-flow dumper), `ofproto/trace` (through `ovs-appctl`, pure OVS). `ovn-detrace` differs from `ovn-nbctl` and `ovn-sbctl` in that it is **read-only** for both databases and has no write subcommand.

**How an engineer masters this.** Four required skills. (1) **Read**: understand the structure of `ovn-detrace` output (indented annotation blocks, one block per cookie hit, in the order cookies appear in the input). (2) **Chain tools**: know when to pipe `ofproto/trace ... | ovn-detrace`, when to pipe `ovs-ofctl dump-flows br-int | ovn-detrace`, and when to dump to a file and run offline. (3) **Configure endpoints**: know the flag `--ovnnb` and `--ovnsb` for remote DB endpoints, `--ovsdb-sock` for a local UNIX socket, and `--no-leader-only` for a Raft cluster; a wrong endpoint silently fails (the cookie does not decode). (4) **Decision tree when a cookie does not match**: distinguish three cases: stale flow (ovn-controller has not cleaned up; restart usually resolves it), `ovn-detrace` connected to the wrong database (verify the endpoint), or the cookie is not OVN-generated (someone manually inserted a flow with `ovs-ofctl add-flow`, or another tool is writing to br-int).

**Usage workflow.** A standard five-step workflow for production debugging. Step 1: run `ovn-trace` to simulate the logical pipeline and identify the suspect drop stage. Step 2: on the target chassis, dump the physical pipeline with `ovs-appctl ofproto/trace br-int 'in_port=N,...'` and save to `/tmp/of.txt`. Step 3: run `cat /tmp/of.txt | ovn-detrace --ovnnb=tcp:nb:6641 --ovnsb=tcp:sb:6642 > /tmp/of-annotated.txt`. Step 4: read `/tmp/of-annotated.txt`, find lines with action `drop` or the last line that matched, and read the annotation block to see which Logical_Flow and which NB resource produced it. Step 5: adjust NB through `ovn-nbctl acl-del`, `lr-route-del`, or `lb-del`; never use `ovs-ofctl del-flows br-int` because ovn-controller will reinstall the rule within seconds after resyncing NBDB.

**When something goes wrong.** Typical situations where an engineer must reach for `ovn-detrace`: (a) a packet is dropped but `ovn-trace` says the logical pipeline passes, which proves the physical flows do not match the logical view and `ovn-detrace` is needed to identify which Logical_Flow the physical drop comes from (possibly a stale flow); (b) latency spikes on a single Logical_Switch, where `ovs-ofctl dump-flows br-int | ovn-detrace | grep <ls_name>` shows which rule for that switch is hit most; (c) security audit, where the engineer wants to know all ACL drops on a chassis, dumps br-int, and lets `ovn-detrace` annotate the result for aggregation. Cross-link Part 20.2 §troubleshooting flow and Part 20.7 §gradient L5 production incident.

**Tightly related to.** Tightly coupled to five components. (1) **`ovs-appctl ofproto/trace`** (Part 9.25): the primary input source; `ofproto/trace` puts a cookie on every "Rule:" line. (2) **`ovn-controller`** (Part 13.7): the side that installs cookies onto br-int; if the controller has not synced, a cookie may not exist in SBDB. (3) **`ovn-northd`** (Part 13.8): the side that encodes a Logical_Flow UUID into a cookie when compiling. (4) **SBDB `Logical_Flow` table** (Part 13.1): the lookup table; the format of the `match` and `actions` columns appears directly in `ovn-detrace` output. (5) **NBDB ACL and Load_Balancer tables** (Part 13.3): the ultimate source of intent; `ovn-detrace` joins NB through the `external_ids:source` UUID to display human-readable ACL names. Loosely related: `ovn-trace` (a parallel sibling, not directly dependent), `ovn-sbctl lflow-list` (also reads Logical_Flow, but does not handle cookies).

**Version differences.** OVN 2.6 introduced the first script (Python 2). OVN 20.06 moved to pure Python 3. OVN 21.06 added `--ovsdb-sock` for UNIX-socket connections. OVN 22.03 (the curriculum baseline) added `--no-leader-only` for Raft cluster mode (allowing connection to a follower instead of only the leader, useful when the leader is busy). OVN 24.03 improved the output block separator and added the `--exit-on-error` flag for CI use. For production OpenStack on OVN 22.03.x, the syntax `cat trace.txt | ovn-detrace --ovnnb=tcp:nb:6641 --ovnsb=tcp:sb:6642` is stable and needs no adjustment; note that OVN 24.03 changes the block prefix from `*` to `> `, so a custom forward parser must recognise both.

**Observation and verification.** Three ways to verify that `ovn-detrace` runs correctly. (1) **Verify install**: `which ovn-detrace` returns `/usr/bin/ovn-detrace`; `ovn-detrace --help` prints the usage without errors; `head -1 $(which ovn-detrace)` shows `#!/usr/bin/env python3`. (2) **Smoke test**: `echo 'cookie=0xdeadbeef' | ovn-detrace --ovnnb=tcp:nb:6641 --ovnsb=tcp:sb:6642`; the expected output keeps the input line and adds a "no match" block (a fake cookie). (3) **Real test**: on a chassis with real flows, `ovs-ofctl dump-flows br-int table=8 | head -1 | ovn-detrace ...`; the output must contain an annotation block with `Logical datapath`, `Pipeline`, `Stage name`, and the match and action expression. If the output is empty, `ovn-detrace` cannot connect to SBDB; verify the endpoint with `nc -zv sb-server 6642`.

**Source code references.** Main file: `utilities/ovn-detrace.in` in the OVN repository (an autotools template that builds into `utilities/ovn-detrace`). Important functions: `main()` parses argv and reads stdin, `OVSDB.find_row_by_partial_uuid()` (a helper in `ovs.db.idl` of the OVS Python bindings) handles UUID-prefix lookups, and `print_logical_flow()` formats the annotation block. Cookie regex: `re.compile(r'cookie=0x([0-9a-fA-F]+)')`. Man page: `manpages/ovn-detrace.8.xml` (placed in section 8 from OVN 22.03 onwards; previously in section 1). Verification baseline: branch `branch-22.03` of `ovn-org/ovn`. Same family of Python scripts: `utilities/ovn-trace.c` (C, not Python), `utilities/ovn-sbctl.c` (C). `ovn-detrace` is one of the few Python utilities in the OVN distribution.

**Real-world incident.** A multi-tenant production OpenStack incident with 1200 VMs on an OVN 22.03.5 cluster of 8 chassis. Symptom: one tenant (Logical_Switch `ls-prod-app`) reports intermittent east-west TCP timeouts at about a 3 percent rate; other tenants are normal. Investigation: `ovn-trace ls-prod-app 'inport=="vm-A" && ip4.dst==<vm-B-ip> && tcp.dst==8080'` shows the pipeline passes through to `ls_out_apply_buffered_actions` and outputs. But `ping vm-B` from vm-A actually fails 3 percent of the time. Next: on the chassis hosting vm-A, `ovs-appctl ofproto/trace br-int 'in_port=<vm-A-ofport>,tcp,nw_src=...,nw_dst=...,tp_dst=8080'` shows the packet drops at `table=12, priority=2002, action=drop`. The cookie is `0x6f8a1b3c`. Running `echo 'cookie=0x6f8a1b3c' | ovn-detrace --ovnnb=... --ovnsb=...` annotates: `Logical_Flow uuid 6f8a1b3c-..., datapath ls-prod-app, stage ls_out_acl, match (ip && tcp.dst==8080 && reg0[10]==1), action drop, source ACL "deny-stale-app-2024Q1"`. Root cause: an old ACL named `deny-stale-app-2024Q1` should have been removed when migrating to policy v2 but was forgotten in `ovn-nbctl acl-del`; the stale flow remained in SBDB and on br-int. Fix: `ovn-nbctl acl-del ls-prod-app to-lport 2002 'ip && tcp.dst==8080 && ...'`. After 5 seconds ovn-controller resynced and the drop rate fell to zero. Value of `ovn-detrace` here: `ovs-ofctl dump-flows br-int` had about 14000 lines, and without `ovn-detrace` it would have been impossible to identify which ACL produced the drop rule.

**Synthetic exercise.** A two-chassis OVN lab with one Logical_Switch `ls-lab` containing two LSPs `lsp1` and `lsp2`. Setup:

```bash
# Initialise the logical topology
ovn-nbctl ls-add ls-lab
ovn-nbctl lsp-add ls-lab lsp1
ovn-nbctl lsp-set-addresses lsp1 "00:11:22:33:44:01 10.0.0.1"
ovn-nbctl lsp-add ls-lab lsp2
ovn-nbctl lsp-set-addresses lsp2 "00:11:22:33:44:02 10.0.0.2"

# Install an ACL that drops ssh (port 22) on the to-lport direction
ovn-nbctl acl-add ls-lab to-lport 1000 'tcp && tcp.dst == 22' drop

# Bind each LSP to the corresponding sandbox or Mininet port

# Step 1: simulate the logical pipeline
ovn-trace ls-lab 'inport=="lsp1" && eth.src==00:11:22:33:44:01 && \
                  eth.dst==00:11:22:33:44:02 && ip4.src==10.0.0.1 && \
                  ip4.dst==10.0.0.2 && ip.proto==6 && tcp.dst==22 && tcp.src==33000'
# Expected result: drop at the ls_out_acl stage.

# Step 2: trace the physical pipeline on the chassis hosting lsp1
ovs-appctl ofproto/trace br-int \
  'in_port=<lsp1-ofport>,tcp,dl_src=00:11:22:33:44:01,dl_dst=00:11:22:33:44:02,
   nw_src=10.0.0.1,nw_dst=10.0.0.2,tp_src=33000,tp_dst=22' > /tmp/of.txt

# Step 3: annotate with ovn-detrace
cat /tmp/of.txt | ovn-detrace --ovnnb=unix:/var/run/ovn/ovnnb_db.sock \
                              --ovnsb=unix:/var/run/ovn/ovnsb_db.sock

# Verify: the annotation block for the drop rule shows
#  - datapath ls-lab
#  - stage ls_out_acl
#  - match (ip && tcp.dst==22 ...)
#  - source ACL UUID matching the entry just created
```

Variations: (a) change the ACL from `drop` to `allow-related` and observe the annotation block change action; (b) delete the ACL with `ovn-nbctl acl-del ls-lab`, then dump flows again; the old cookie disappears from br-int within seconds; (c) restart ovn-controller and check that the cookie persists (cookies are stable for the same Logical_Flow UUID).

**Common failures and diagnostic signals.** Four main failure modes. (1) **Wrong endpoint**: `--ovnnb` or `--ovnsb` points to an unreachable database; signal: the output contains only input lines and no annotation blocks; diagnosis: add `--verbose` if the version supports it, or test with `ovn-sbctl --db=tcp:sb:6642 list Logical_Flow | head` to see whether you get a response; recovery: fix the endpoint, try `unix:/var/run/ovn/ovnsb_db.sock` for local connections. (2) **Raft cluster, follower does not serve**: if SBDB is in Raft cluster mode and the endpoint points at a follower, the default behaviour from OVN 21.06 onwards is for only the leader to answer, so the output can be partial; signal: cookies for the same datapath sometimes decode and sometimes do not; recovery: add `--no-leader-only` or ensure the endpoint points to the leader (check `ovn-appctl -t /var/run/ovn/ovnsb_db.ctl cluster/status OVN_Southbound`). (3) **Stale flow outside SBDB**: a cookie exists on br-int but the corresponding Logical_Flow has already been removed by ovn-northd; signal: `cookie=0x... (no match)` on a freshly dumped rule; recovery: run `ovn-appctl -t /var/run/ovn/ovn-controller.<chassis>.ctl recompute` to force an ovn-controller resync, or restart ovn-controller. (4) **Cookie not OVN-encoded**: someone ran `ovs-ofctl add-flow br-int 'cookie=0x12345,...'` by hand; signal: the cookie format looks normal but never decodes; recovery: identify the source (check deployment scripts), remove the manual flow, and let ovn-controller manage br-int exclusively.

**Comparison with other systems.** In NSX-T, the equivalent tool is `nsxcli > get logical-port <uuid> traceflow` together with `get logical-router <uuid> traceflow`, but NSX-T has no cookie-mapping pattern because the NSX-T data plane writes NSGroup IDs directly into VDS rules and needs no inverse mapping through a hash. Cisco ACI has `iping`, `itraceroute`, and `acidiag fnvread`, but those are full distributed traces with no concept of "control-plane cookie attached to a physical rule"; ACI uses a very different class-id encoding. Linux iptables has no analogue: `iptables -L -v --line-numbers` shows per-rule counters but the rule has no "cookie" linking back to a higher-level declarative intent. The unique property of `ovn-detrace` is that it is a textbook example of the **"intent-trace bridge"** design pattern: use a small identifier (a 32-bit cookie) attached at the data plane to link back to the full identifier (a 128-bit UUID) at the control plane, trading a few bytes per rule for forensic capability. The pattern is worth learning and applying in any custom SDN system.

## 13.14.11 Deeper dive on cluster options and transaction guards

> **Purpose of this section:** §13.14.9.1 listed the five options `--timeout`, `--leader-only`, `--no-leader-only`, `--shuffle-remotes`, and `--dry-run` at the summary-table level. §13.14.11 expands each option into a standalone subsection with concept, role, mechanism, observation, and a real lab, so that an on-call engineer has a quick reference when an `ovn-nbctl` script hangs, writes hit a follower, or the Raft cluster loses its leader.

### 13.14.11.1 Option `--timeout=N` (alias `-t`)

**Concept.** `--timeout=N` (or the short form `-t N`) limits the runtime of one `ovn-nbctl` invocation to roughly N seconds. The default N=0 means **wait forever**: the `ovn-nbctl` process waits for an NBDB response indefinitely, even if the server is unreachable, until the user presses Ctrl+C. When N is greater than 0 and the timeout fires, the kernel sends `SIGALRM` to the process and `ovn-nbctl` exits with a signal code rather than a normal exit code, so the calling script must distinguish using `kill -l $?` to recognise the cause.

**Role and problems it solves.** In automation systems (Ansible, the OVN-Kubernetes operator, OpenStack neutron-ovn-mech), when the NBDB Raft cluster temporarily loses quorum or there is a network partition between the controller node and the NB cluster, the default wait-forever behaviour hangs the entire CMS pipeline, leading to daemon lock files or backed-up transaction queues. `--timeout=5` (5 seconds) is a common value in production playbooks: long enough for a normal NBDB transaction (latency 50 to 500 ms), but short enough to fail fast when the cluster degrades. It is especially important when calling `ovn-nbctl` from a systemd unit with `TimeoutStartSec=`: without `--timeout`, systemd kills the process at unit timeout, leaving an unclean OVSDB lock behind.

**How it works.** `parse_options()` in `utilities/ovn-nbctl.c` maps the `-t` and `--timeout` flag to the variable `int timeout`, then passes it to `ctl_timeout_setup(timeout)` (a helper in `lib/ovs-thread.c` of the OVS dependency) which calls the `alarm(timeout)` syscall. When `alarm` counts down to zero without a handler clearing it, the kernel raises `SIGALRM`. By default OVN installs no custom signal handler for `SIGALRM`, so the process exits with the core-dump pattern `Alarm clock` in `dmesg`, or with shell `$?` equal to 142 (128 plus signal 14).

**When to raise the default 5 s to 30 to 60 s.** A cross-region cluster (NB1 in Hanoi, NB2 in Ho Chi Minh City, NB3 in Da Nang) has RTT 30 to 50 ms; a large transaction (bulk `lsp-add` of 500 ports atomically) needs 1 to 3 seconds to compile, commit, and replicate across the 3 Raft nodes. In that workload, set `--timeout=30` (30 seconds) to cover the latency edge cases.

**Short observation lab.** Run `ovn-nbctl --timeout=2 --db=tcp:198.51.100.99:6641 list NB_Global` on a host that cannot reach that IP: after about 2 seconds the process exits with the message `ovn-nbctl: NB_Global: timeout expired`, with exit code 142 (`echo $?`). Compare with the same command without `--timeout`: the process hangs until the user presses Ctrl+C.

**Common pitfalls.** (a) Newcomers misread `--timeout=5000` as 5 seconds in millisecond style; in reality it is 5,000 seconds. The unit is **seconds**, not milliseconds. (b) Setting `--timeout=1` for a large transaction yields constant false-fails even though the cluster is healthy.

### 13.14.11.2 Option `--leader-only`

**Concept.** `--leader-only` is the default flag for `ovn-nbctl` against NBDB in Raft cluster mode (3 or 5 nodes). When the NB cluster runs Raft, every write transaction must go through the leader to guarantee linearizable consistency. The flag tells `ovn-nbctl` to attempt connection only to the server holding the leader role; if a remote reports the follower role, the client disconnects and tries the next remote in the list `--db=tcp:nb1:6641,tcp:nb2:6641,tcp:nb3:6641`.

**Role and problems it solves.** In a Raft cluster, a follower can lag the leader by a few log entries; a read transaction on a follower can return stale data (tens to hundreds of milliseconds old). For a CMS such as OpenStack Neutron that compiles YAML into a chain of ovn-nbctl commands (`ls-add`, then `lsp-add`, then `lsp-set-addresses` for the same tenant), if the second invocation lands on a follower that has not yet replicated the first, the command fails with "Logical_Switch not found". `--leader-only` (the default) ensures every invocation sees the same state, with no read-after-write race.

**How it works.** The OVSDB IDL client in `ovn-nbctl` sends `get_schema` plus `get_server_id` to each remote; the cluster server returns JSON containing `"role":"leader"` or `"role":"follower"`. If the role is not leader and `--leader-only` is on, the IDL closes the connection and tries the next remote. If all three remotes are followers (a rare scenario where the leader just crashed and the election has not completed), the client retries continuously until `--timeout` fires or one remote becomes leader.

**Effect and trade-off.** Every write from the CMS converges on a single leader node, making the leader a hot spot (CPU and disk IO for the Raft log). That is why an NB cluster plan must capacity-plan the leader separately: the leader can use 60 to 80 percent of CPU on a cluster with more than 1000 Logical_Switch instances. When the leader fails, the election lasts 1 to 3 seconds, and every `ovn-nbctl --leader-only` blocks during that window. That is the consistency-versus-availability trade-off of Raft.

**Observation lab.** On a three-node Raft cluster, run `ovn-appctl -t /var/run/ovn/ovnnb_db.ctl cluster/status OVN_Northbound | grep -E 'Role|Leader'` to identify the current leader; then run `ovn-nbctl --db=tcp:<follower-IP>:6641 ls-list` (pointing only at a follower). With the default `--leader-only` on, `ovn-nbctl` rejects the follower with "no leader available". Compare with §13.14.11.3 below.

### 13.14.11.3 Option `--no-leader-only`

**Concept.** `--no-leader-only` reverses the default of §13.14.11.2: it lets `ovn-nbctl` connect to any server in the Raft cluster, not just the leader. Read-only transactions (commands that do not modify NBDB, such as `list`, `find`, `get`, `show`, `lsp-list`, and `acl-list`) run on a follower; write transactions are forwarded by the follower to the leader (proxy mode), so writes remain linearizable.

**Role and problems it solves.** Use case 1: **read scaling**. A large cluster receives thousands of `ovn-sbctl list Logical_Flow` per second from monitoring tooling. If every read goes to the leader, the leader saturates. `--no-leader-only` (combined with `--shuffle-remotes`) spreads read traffic across the three nodes, raising read throughput by about 3x. Use case 2: **availability during the election window**. When the leader has just crashed and the election is not finished (1 to 3 seconds), `--leader-only` blocks while `--no-leader-only` can still read (slightly stale) data from any follower. Use case 3: **forensic debugging**. When a follower is suspected of unusual lag, the engineer connects directly to it with `--no-leader-only` and compares state with the leader.

**How it works.** When the flag is set, the IDL skips the role check on `get_server_id`; any reachable remote with a matching schema is accepted. A write transaction on a follower is forwarded to the leader through Raft RPC, the leader confirms, and the response returns to the client. Write latency in follower mode equals leader latency plus one extra leader-to-follower RTT.

**Consequences and warnings.** (a) Reads can be stale (the read-after-write race noted in §13.14.11.2); automation scripts SHOULD NOT use `--no-leader-only` for read-then-write chains. (b) Read-only monitoring tooling (Prometheus exporter, Grafana dashboard, SLO checker) SHOULD use `--no-leader-only` to reduce pressure on the leader.

**Observation lab.** The command from the §13.14.11.2 lab that points at a follower succeeds when `--no-leader-only` is added: `ovn-nbctl --no-leader-only --db=tcp:<follower-IP>:6641 ls-list` returns the Logical_Switch list (possibly with 1 to 2 stale entries if the follower is lagging).

### 13.14.11.4 Option `--shuffle-remotes`

**Concept.** `--shuffle-remotes` (on by default since OVS 2.13 and OVN 20.06) tells the IDL to randomly shuffle the order of remotes in the chain `--db=tcp:nb1:6641,tcp:nb2:6641,tcp:nb3:6641` before attempting connection. The opposite flag is `--no-shuffle-remotes`, which keeps the declared order.

**Role and problems it solves.** Without shuffle (the old behaviour before OVS 2.13), 1000 clients on the same cluster all try nb1 first, so nb1 receives every connection and saturates. This is the classic **thundering herd**. Shuffle distributes the starting point of connection attempts and balances the load across the three nodes. The benefit is most visible after a cluster restart or network-partition recovery: every client reconnects at the same time, and without shuffle 100 percent land on nb1.

**How it works.** In `parse_options()`, the default sets `shuffle = true`. The IDL has a function `jsonrpc_session_shuffle()` that permutes the remote list using `random_uint32() % n` for each index. Each separate `ovn-nbctl` process seeds its random number from `/dev/urandom`, so 1000 parallel invocations produce 1000 different orders.

**Interaction with `--leader-only`.** When both `--leader-only` and `--shuffle-remotes` are on (the default), the IDL shuffles the order, tries each remote, skips followers, and stops at the first leader. Net effect: clients start trying at different times, but writes still converge on the leader. With read-only `--no-leader-only`, shuffle distributes evenly and each client lands on a random node.

**When to disable with `--no-shuffle-remotes`.** Rare. The only legitimate use case is intentional testing that forces the client to connect to a specific node first (for example, to verify failover behaviour). In that case, set `--no-shuffle-remotes --db=tcp:<target>,tcp:<backup1>,tcp:<backup2>`. Production should NOT disable shuffle.

**Observation lab.** On a three-node Raft cluster with the Prometheus metric `ovsdb_server_connection_count`, launch 100 parallel `ovn-nbctl --no-leader-only ls-list` processes within 1 second and measure connection_count on all three nodes. With the default shuffle on, the distribution is roughly 33, 33, 33. With `--no-shuffle-remotes`, the distribution is roughly 100, 0, 0. The contrast illustrates the effect directly.

### 13.14.11.5 Option `--dry-run`

**Concept.** `--dry-run` tells `ovn-nbctl` to parse and validate the command and compute the OVSDB JSON-RPC transaction it would send, but NOT commit the transaction to NBDB. The command exits successfully (exit code 0) if syntax, references, and constraints are valid; it exits non-zero on errors (for example, a Logical_Switch reference that does not exist, or an integer field receiving a string). NBDB is not modified at all.

**Role and problems it solves.** Dry-run is the safety net before applying a risky command in production. Use case 1: **review automatically generated command sequences** (Ansible playbooks, Kubernetes operators rendering commands from templates). Run dry-run on a staging cluster first, and only apply for real if it passes. Use case 2: **verify a migration script**, for example `ovn-nbctl -- ls-add new -- ls-del old`. Dry-run confirms that the old object exists and the new one does not before committing. Use case 3: **safety net for the on-call engineer**. When fixing production at 3 in the morning, type the command, add `--dry-run` first, read the output to confirm intent, then rerun without the flag.

**How it works.** In `do_nbctl()` (the main transaction handler), after building the OVSDB transaction structure, the code checks the `dry_run` variable: if true, it prints the transaction JSON to stderr (in verbose mode) or skips silently, then skips `ovsdb_idl_loop_commit_and_wait()`. The validation step (checking that referenced UUIDs are valid and that types match the schema) RUNS IN FULL before commit, so dry-run still catches most static errors. It does not catch dynamic errors (constraint violations only seen at commit time, for example a unique-index conflict).

**Important limitations.** (a) Dry-run does NOT simulate the ovn-northd compile from NBDB to SBDB; it only checks the NBDB transaction layer. The command `ovn-nbctl --dry-run lr-route-add` can pass dry-run while the route still does not work because northd rejects it (for example, next-hop unreachable). To verify that the pipeline actually works, also use `ovn-trace` (§13.14.9.2). (b) Dry-run with a `--` chain (an atomic transaction of multiple subcommands) only validates up to the first failing subcommand; if the fifth of ten fails, you cannot tell whether subcommands 6 to 10 pass until the fifth is fixed.

**Observation lab.** Run `ovn-nbctl --dry-run -- ls-add ls-test -- lsp-add ls-not-exist lsp-x`. Result: dry-run fails on the second subcommand (`ls-not-exist` does not exist), exit code 1, with a clear message. Then check `ovn-nbctl ls-list | grep ls-test`: empty, confirming that `ls-test` was not created (the atomic transaction rolled back). Compare with the same command without `--dry-run`: it also fails and rolls back, but dry-run avoids the cost of attempting the commit.

**Version differences.** OVN 22.03 (the curriculum baseline) supports `--dry-run` for most subcommands (`ls-add`, `lsp-add`, `acl-add`, `lr-add`, `lb-add`, and so on). A few special subcommands such as `init` (which bootstraps a fresh DB) ignore `--dry-run` because there is no corresponding transaction. OVN 24.03 extends `--dry-run` to control subcommands such as `set-connection` and `set-ssl`.

## 13.14.12 Deeper dive on advanced ovn-trace and ovn-detrace options

> **Purpose of this section:** §13.14.9.2 and §13.14.9.3 introduced both utilities from a cornerstone perspective. §13.14.12 drills into five specific options that on-call engineers frequently encounter in documentation but are unsure which exact flag to use, because many internal briefs and blog posts spell the flag names incorrectly. Each subsection below cross-checks against the **upstream source** (`utilities/ovn-trace.c` and `utilities/ovn_detrace.py.in` on branch-22.03) to confirm the real flag name.
>
> **Important nomenclature corrections (verified against branch-22.03 upstream).** Several "shortened" flag names circulate in the OVN community and get confused:
>
> - **`ovn-trace --multiple` DOES NOT EXIST.** People say "show multiple candidate paths", but the real flag is `--all` (combining the three modes `--detailed`, `--summary`, `--minimal`) or `--select-id=ID` (forcing the `select` action to pick a specific branch for ECMP or Load_Balancer multipath). Verified: `utilities/ovn-trace.c` on branch-22.03 lines 268 to 283 (`long_options[]`) contain no entry for `"multiple"`.
> - **`ovn-detrace --ovnsb-db` and `--ovnnb-db` DO NOT EXIST on branch-22.03.** The real names are shorter: `--ovnsb=DATABASE` and `--ovnnb=DATABASE`. Verified: `utilities/ovn_detrace.py.in` on branch-22.03 lines 50 to 52 (the `usage()` text) and lines 412 to 417 (the `getopt.gnu_getopt` long-option list `['help', 'version', 'ovs', 'ovnsb=', 'ovnnb=', 'ovsdb=', 'private-key=', 'certificate=', 'ca-cert=']`). The `-db` suffix only appears in the context of the `OVN_NB_DB` environment variable, not as a CLI flag name.
> - **`ovn-detrace --no-leader-only` is not yet available on branch-22.03**; it was added on branch-24.03 (`utilities/ovn_detrace.py.in` on branch-24.03 line 56 plus lines 423 to 425). ### 13.14.12.1 `ovn-trace --all` instead of the misnomer "multiple paths"

**Concept.** `--all` tells `ovn-trace` to emit all three output forms in a single invocation: `--detailed` (detailed stage and register information), `--summary` (compact, programming-language style), and `--minimal` (only packet-modifying actions and output). This option is commonly misnamed `--multiple` in internal docs; the official upstream name has been `--all` since OVN 21.06. When a `select` action appears (Load_Balancer multi-backend, ECMP), `--all` does not automatically explore every branch; to see other branches, rerun with a different `--select-id=N` (see §13.14.12.4).

**Role and problems it solves.** When debugging a complex branching stage (`ls_in_lb` with three backends, or `lr_in_ip_routing` with two ECMP next-hops), a single mode makes it hard to compare packet behaviour before and after an action. `--all` lets the engineer read the same result at three abstraction levels and compare to understand the compiler intent (`ovn-northd`) and verify simulator behaviour. Another use case: pasting output into a bug ticket. `--minimal` gives a quick summary and `--detailed` gives forensic evidence; including both in one file removes follow-up questions from a reviewer.

**Syntax and example output.**

```bash
ovn-trace --all ls-prod \
  'inport=="lsp-vm1" && eth.src==50:54:00:00:00:01 && eth.dst==ff:ff:ff:ff:ff:ff && \
   arp.op==1 && arp.spa==10.0.0.1 && arp.tpa==10.0.0.2'
```

Output structure:

```
# ovn-trace ls-prod '...'

ingress(dp="ls-prod", inport="lsp-vm1")
-----------------------------------------
 0. ls_in_port_sec_l2 (northd.c:5234): inport == "lsp-vm1" && ...
    next;
 ...

# Summary
ingress(dp="ls-prod", inport="lsp-vm1") {
    next;
    ...
    output;
}

# Minimal
arp.op = 2;
arp.tha = 50:54:00:00:00:02;
output("lsp-vm2");
```

**Common pitfalls.** (a) Many blog posts write `--multiple`, so the command fails with `ovn-trace: unrecognized option '--multiple'`. Correction: use `--all`. (b) `--all` does not mean "try every select branch"; to enumerate branches, combine with `--select-id`. (c) `--all` output is three times as long as a single mode; when piping to `less`, also add `--no-friendly-names` for stable output if a custom parser is involved.

**Versions.** The three modes plus `--all` were split out in OVN 21.06 (an output-refactor commit group). Before that, OVN 2.7 to 21.03 had only the single verbose default.

### 13.14.12.2 `ovn-trace --friendly-names` and `--no-friendly-names`

**Concept.** `--friendly-names` (on by default from OVN 22.03) tells `ovn-trace` to replace the UUIDs of Logical_Switch, Logical_Router, and Logical_Switch_Port with the `external_ids:name` value taken from NBDB or SBDB. UUIDs are hard to read during an incident (`6f8a1b3c-...-...-...-...`); replacing them with user-declared names (`ls-prod-app`, `lsp-vm1`) makes the output much friendlier. The opposite flag `--no-friendly-names` keeps the original UUIDs, which is useful for custom parsers that need stable identifiers.

**Role and problems it solves.** In an environment with hundreds of Logical_Switch instances, reading a trace full of UUIDs is torture. Friendly names cut trace reading time from minutes to seconds because the engineer immediately recognises "this is the Logical_Switch of tenant X". Conversely, when writing a script that greps the output (CI test, regression check), UUIDs are more stable because names can change through `ovn-nbctl`; in that case, use `--no-friendly-names` to avoid mismatches.

**Syntax and example.**

```bash
# Default (friendly names): human-readable output
ovn-trace ls-prod 'inport=="lsp-vm1"'
# Output: ingress(dp="ls-prod", inport="lsp-vm1")

# Disable for parser scripts
ovn-trace --no-friendly-names ls-prod 'inport=="lsp-vm1"' \
  | grep 'ingress(dp='
# Output: ingress(dp="6f8a1b3c-...-...", inport="2a3b...")
```

**Common pitfalls.** (a) On OVN versions earlier than 22.03 the default is `--no-friendly-names`; to get friendly output you must set the flag manually. (b) If an LSP or pair name contains special characters (`/`, space, Unicode), friendly names still show the raw character, which can break shell-pipe quoting; in that case, use `--no-friendly-names` for stability.

**Mechanism.** `utilities/ovn-trace.c` on branch-22.03 lines 278 to 279 register the long options `friendly-names` and `no-friendly-names`; the internal variable `friendly_names` defaults to `true` from OVN 22.03 onwards. The function `ovntrace_node_print_details()` checks this variable when formatting datapath and port names.

**Versions.** The option was introduced in OVN 21.09 (default OFF). OVN 22.03 flipped the default to ON. OVN 24.03 made no further change.

### 13.14.12.3 `ovn-trace --ovs[=remote]` to align logical with physical

**Concept.** The flag `--ovs` (which can take a remote in the form `--ovs=tcp:host:6640` or `--ovs=unix:/path/sock`; without an argument it defaults to `unix:/br-int.mgmt`) tells `ovn-trace` not only to simulate the logical pipeline but also to CONNECT to `ovs-vswitchd` on a specific chassis to read the corresponding OpenFlow flows and print them alongside each logical stage. Net effect: the output has two columns. On the left is the Logical_Flow (OVN level); on the right is the OpenFlow rule (OVS level) that `ovn-controller` installed on `br-int`. This enables direct comparison to see whether northd compiled the intent correctly.

**Role and problems it solves.** When you suspect that `ovn-northd` compiled something wrong or that `ovn-controller` failed to install a rule on a specific chassis, use `--ovs` instead of opening two windows (one running plain `ovn-trace`, one running `ovs-ofctl dump-flows br-int`) and comparing by eye. The flag is especially valuable on clusters with many chassis and suspected asymmetry: chassis A's logical pipeline passes, while chassis B drops; run `ovn-trace --ovs=tcp:chassis-B:6640` to see exactly where the flows on chassis B diverge from chassis A.

**Syntax and examples.**

```bash
# Use the default unix socket on the local chassis
ovn-trace --ovs ls-prod 'inport=="lsp-vm1" && ip4.dst==10.0.0.10'

# Point at a remote chassis through TCP (ovs-vswitchd must have a TCP manager enabled)
ovn-trace --ovs=tcp:chassis-B.example.com:6640 ls-prod \
  'inport=="lsp-vm1" && ip4.dst==10.0.0.10'
```

The output includes an extra block:

```
ls_in_acl (priority 2002):
  Logical: ip && tcp.dst==80 -> drop;
  Physical (table=8 br-int chassis-B):
    cookie=0x6f8a1b3c, priority=2002, ip,tcp,tp_dst=80, actions=drop
```

**Common pitfalls.** (a) The default remote `unix:/br-int.mgmt` is only valid when `ovn-trace` runs on the same chassis that hosts `br-int`; on a jump host without `ovs-vswitchd`, pass `--ovs=tcp:...` instead. (b) `ovs-vswitchd` does not bind a TCP listener by default; run `ovs-vsctl set-manager ptcp:6640` first. (c) The targeted chassis must have a Port_Binding for the input LSP; otherwise the OpenFlow rule does not exist there and the physical column is empty. (d) Different from `ovn-detrace --ovs`: the `--ovs` flag of `ovn-detrace` only adds ofport decoding for a flow, it does not connect to ovs-vswitchd to dump rules.

**Mechanism and source.** `utilities/ovn-trace.c` on branch-22.03 lines 257, 276, and 333 (`OPT_OVS`, `optional_argument`); the connection helper `vconn_open()` from the OVS dependency. Man page `ovn-trace(8)` section "--ovs[=remote]".

### 13.14.12.4 `ovn-trace --lb-dst=ip[:port]` and `--select-id=ID` for Load_Balancer hairpin and select-action multipath

**Concept.** `--lb-dst=ip[:port]` overrides the random backend selection in OVN's `ct_lb` action (Load_Balancer DNAT); it forces `ovn-trace` to simulate as if the VIP packet were DNATed to the specific backend `ip:port` instead of picking a random backend. The parameter accepts a bare IP (`10.0.1.50`) or an IP plus port (`10.0.1.50:8080`). The sibling flag `--select-id=ID` (same family, for `select` actions used in ECMP and group-select) lets the simulator force a specific ID inside a select bucket. Both flags work only in normal mode (NOT in daemon mode).

**Role and problems it solves.** When a VIP has three backends and a VM client reports "sometimes the backend fails to connect", the engineer needs to verify that each backend has correct routes and ACLs. Without `--lb-dst`, each `ovn-trace` run randomly picks a backend, which is not reproducible. With `--lb-dst=10.0.1.51`, you can trace deterministically per backend in succession and compare paths. Similarly, `--select-id` works for ECMP: a route `10.0.0.0/8` with two next-hops can be forced with `--select-id=0` to pick the first or `--select-id=1` to pick the second.

**Syntax and examples.**

```bash
# Tenant reports timeouts to backend 10.0.1.52; check the path to that backend
ovn-trace --lb-dst=10.0.1.52:8080 lr-edge \
  'inport=="lsp-client" && eth.src==... && \
   ip4 && ip4.src==10.0.0.5 && ip4.dst==10.0.20.50 && \
   tcp && tcp.dst==8080 && tcp.src==33000'
# Trace passes through to ls_in_lb and hits ct_lb(backends=10.0.1.52:8080) deterministically.

# Compare with a different backend
ovn-trace --lb-dst=10.0.1.51:8080 lr-edge '...'

# ECMP with two next-hops, forcing each branch
ovn-trace --select-id=0 lr-edge '...'
ovn-trace --select-id=1 lr-edge '...'
```

**Common pitfalls.** (a) `--lb-dst` only takes effect when the pipeline actually hits `ct_lb`; if the microflow does not match the VIP or port, the flag is silently ignored. (b) Passing `--lb-dst=10.0.1.52` without a port when the LB has ports causes the simulator to use a default port from the first backend, which may not be the intent; pass a full `ip:port`. (c) `--select-id` counts from 0 inside the `select` action; assuming 1-based indexing leads to the wrong bucket.

**Mechanism and source.** `utilities/ovn-trace.c` on branch-22.03 lines 264 to 265, 282 to 283, and 349 to 353 (`OPT_LB_DST`, `OPT_SELECT_ID`, both `required_argument`); the function `parse_lb_dst()` parses `ip:port` into an internal struct. The man page `ovn-trace(8)` notes "Not available in daemon mode".

**Versions.** `--lb-dst` was introduced in OVN 20.06. `--select-id` was introduced in OVN 21.06 alongside the select-action refactor.

### 13.14.12.5 `ovn-detrace` real endpoint options (correcting the flag names)

**Concept.** `ovn-detrace` (the Python script `utilities/ovn_detrace.py.in`) on branch-22.03 supports exactly three database-endpoint options: `--ovnsb=DATABASE` (Southbound), `--ovnnb=DATABASE` (Northbound), and `--ovsdb=DATABASE` (the local OVS DB, used only with `--ovs`). The flag names DO NOT have a `-db` suffix. Many internal briefs and cheat sheets write `--ovnsb-db`, `--ovnnb-db`, or `--ovsdb-sock` by mistake; running them fails with `ovn-detrace: option --ovnsb-db not recognized`. The flag `--no-leader-only` (which allows connecting to a follower) **exists only from branch-24.03 onwards** and is not available on 22.03.

**Syntax and real examples.**

```bash
# Use the correct flag names (branch-22.03)
ovs-appctl ofproto/trace br-int 'in_port=3,tcp,...' \
  | ovn-detrace --ovnnb=tcp:nb-leader.example.com:6641 \
                --ovnsb=tcp:sb-leader.example.com:6642

# Local socket (same host)
ovs-ofctl dump-flows br-int table=8 | head -3 \
  | ovn-detrace --ovnnb=unix:/var/run/ovn/ovnnb_db.sock \
                --ovnsb=unix:/var/run/ovn/ovnsb_db.sock

# Add ofport decoding from the local OVS DB
ovn-detrace --ovs --ovsdb=unix:/var/run/openvswitch/db.sock \
            --ovnnb=... --ovnsb=... < /tmp/of.txt
```

**Correct sample output.**

```
br-int OpenFlow flow:
  cookie=0x6f8a1b3c, table=8, priority=2002, ip,tcp,tp_dst=80, actions=drop
  * Logical datapath: "ls-prod" (uuid 6f8a1b3c-...)
  * Pipeline: ingress, table 8 (ls_in_acl), priority 2002
  * Match: ip && tcp.dst == 80
  * Action: drop;
   * ACL UUID: 9c4d... (name "deny-stale-app-2024Q1")
```

**Common pitfalls.** (a) Typing `--ovnsb-db` fails. Correct: `--ovnsb`. (b) Typing `--no-leader-only` on 22.03 fails with `option not recognized`; to use it, upgrade or point directly at the leader (`ovn-appctl -t /var/run/ovn/ovnsb_db.ctl cluster/status OVN_Southbound | grep Leader`). (c) `--ovsdb` is meaningless without `--ovs`; passing it alone is silently ignored. (d) A wrong endpoint (typo in the IP, mistaking port 6641 for 6642) produces no clear error; instead every cookie comes out as `(no match)`. Verify the endpoint with `ovn-sbctl --db=<endpoint> list Logical_Flow | head` before using.

**Mechanism and source.** `utilities/ovn_detrace.py.in` on branch-22.03 lines 50 to 55 (`usage()` text) and lines 412 to 417 (the `getopt.gnu_getopt` long-option list confirming the three names `'ovnsb='`, `'ovnnb='`, `'ovsdb='`, with no `-db` or `leader-only` entry). Branch-24.03 adds `'no-leader-only'` on line 425. Verified 2026-04-26.

**Version summary.** OVN 2.6 introduced `ovn-detrace` with `--ovnsb` and `--ovnnb`. OVN 21.06 added `--ovsdb` and `--ovs`. OVN 24.03 added `--no-leader-only`. On the curriculum baseline 22.03, use the four correct flag names above plus the standard SSL flags `-p`, `-c`, and `-C`.

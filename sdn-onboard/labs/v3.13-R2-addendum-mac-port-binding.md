## Why this addendum exists

A reader of the main R2 lab `v3.13-R2-bridge-port-primitives.md` who searches the file for either MAC value reported by `fdb/show` (`56:a6:87:c1:15:9c` or `e2:10:43:fa:d2:bd`) finds those MACs only in the `fdb/show` output itself. The main lab never queries the MAC of `veth_red` or `veth_blue` (the namespace-side ends of the two veth pairs), so the engineer cannot prove from that file alone which originating netdev the MAC belongs to. The main lab's prose asserts the binding without showing it, which is not enough.

This addendum closes the gap. It rebuilds the same F.5 topology, captures the MAC of every relevant netdev on every side of every veth pair, captures the OVSDB `Interface` row's `mac_in_use` column, captures the `dpif/show` OpenFlow-to-datapath port mapping, sends the same ping, and captures `fdb/show`. The reader cross-references the four sets of bytes and concludes the binding without leaving the document.

## Build the F.5 topology

The same sequence of `add-br`, `ip netns add`, `ip link add ... type veth peer`, `ip link set ... netns`, `ip link set ... up`, `ovs-vsctl add-port`, and per-namespace IP assignment as the main R2 lab section F.5.

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-br br0
root@lab-openvswitch:~/network-onboard# ip netns add ns_red
root@lab-openvswitch:~/network-onboard# ip netns add ns_blue
root@lab-openvswitch:~/network-onboard# ip link add veth_red type veth peer name veth_red_br
root@lab-openvswitch:~/network-onboard# ip link add veth_blue type veth peer name veth_blue_br
root@lab-openvswitch:~/network-onboard# ip link set veth_red netns ns_red
root@lab-openvswitch:~/network-onboard# ip link set veth_blue netns ns_blue
root@lab-openvswitch:~/network-onboard# ip link set veth_red_br up
root@lab-openvswitch:~/network-onboard# ip link set veth_blue_br up
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br0 veth_red_br
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br0 veth_blue_br
root@lab-openvswitch:~/network-onboard# ip netns exec ns_red ip link set lo up
root@lab-openvswitch:~/network-onboard# ip netns exec ns_red ip link set veth_red up
root@lab-openvswitch:~/network-onboard# ip netns exec ns_red ip addr add 10.0.0.1/24 dev veth_red
root@lab-openvswitch:~/network-onboard# ip netns exec ns_blue ip link set lo up
root@lab-openvswitch:~/network-onboard# ip netns exec ns_blue ip link set veth_blue up
root@lab-openvswitch:~/network-onboard# ip netns exec ns_blue ip addr add 10.0.0.2/24 dev veth_blue
```

Topology summary (illustrative diagram, not lab output):

```ascii
+-----------------+     +--------------------------+     +-----------------+
| ns_red          |     | host default namespace   |     | ns_blue         |
| veth_red        +-----+ veth_red_br      OVS br0 +-----+ veth_blue       |
| 10.0.0.1/24     |     |                          |     | 10.0.0.2/24     |
|                 |     | veth_blue_br             |     |                 |
+-----------------+     +--------------------------+     +-----------------+
```

Each veth pair has two ends. The bridge-side end (`veth_red_br`, `veth_blue_br`) lives in the host default namespace and is attached to OVS bridge `br0` as a port. The namespace-side end (`veth_red`, `veth_blue`) lives inside the corresponding network namespace and carries the IP address. Each end of each pair has its own kernel-assigned MAC, and the two ends of a pair have different MACs.

## Step 1, capture the MAC of every relevant netdev (BEFORE the ping)

The four queries below capture the MAC of all four veth ends. Two queries run in the host default namespace and read the bridge-side ends; two queries use `ip netns exec` to enter the relevant namespace and read the namespace-side ends.

```text
root@lab-openvswitch:~/network-onboard# ip -br link show veth_red_br
veth_red_br@if26 UP             76:49:2f:42:79:ca <BROADCAST,MULTICAST,UP,LOWER_UP> 
root@lab-openvswitch:~/network-onboard# ip -br link show veth_blue_br
veth_blue_br@if28 UP             7e:3a:00:75:07:a2 <BROADCAST,MULTICAST,UP,LOWER_UP> 
root@lab-openvswitch:~/network-onboard# ip netns exec ns_red ip -br link show veth_red
veth_red@if25    UP             56:a6:87:c1:15:9c <BROADCAST,MULTICAST,UP,LOWER_UP> 
root@lab-openvswitch:~/network-onboard# ip netns exec ns_blue ip -br link show veth_blue
veth_blue@if27   UP             e2:10:43:fa:d2:bd <BROADCAST,MULTICAST,UP,LOWER_UP> 
```

Reading the four MAC values directly off the captures:

| Netdev | Side | Captured MAC |
|---|---|---|
| `veth_red_br` | bridge side, host default namespace | `76:49:2f:42:79:ca` |
| `veth_blue_br` | bridge side, host default namespace | `7e:3a:00:75:07:a2` |
| `veth_red` | namespace side, `ns_red` | `56:a6:87:c1:15:9c` |
| `veth_blue` | namespace side, `ns_blue` | `e2:10:43:fa:d2:bd` |

The four MACs are all distinct. The `@ifNN` suffix in `ip -br link` output (`veth_red_br@if26`, `veth_red@if25`) gives the kernel ifindex of the peer end of the veth pair: `veth_red_br`'s peer has ifindex 25, and `veth_red`'s peer has ifindex 26, which would only happen if the two ends are linked, confirming the veth pair binding by ifindex independently of MAC.

## Step 2, capture OVS's view of the bridge-side `mac_in_use`

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=name,ofport,mac_in_use list interface veth_red_br veth_blue_br
name                : veth_red_br
ofport              : 1
mac_in_use          : "76:49:2f:42:79:ca"

name                : veth_blue_br
ofport              : 2
mac_in_use          : "7e:3a:00:75:07:a2"
```

The OVSDB `Interface` row records `mac_in_use` for each port. For `veth_red_br` it is `76:49:2f:42:79:ca`; for `veth_blue_br` it is `7e:3a:00:75:07:a2`. These match the bridge-side MACs captured in Step 1. The `ofport` column gives the OpenFlow port number assigned by OVS: `veth_red_br` is OF port 1, `veth_blue_br` is OF port 2.

## Step 3, confirm the OF-to-datapath port-number split

```text
root@lab-openvswitch:~/network-onboard# ovs-appctl dpif/show 2>&1 | head -10
system@ovs-system: hit:8 missed:6
  br0:
    br0 65534/1: (internal)
    veth_blue_br 2/3: (system)
    veth_red_br 1/2: (system)
```

`dpif/show` reports each port as `<name> <OFport>/<DPport>: (<type>)`. So `veth_red_br 1/2` reads as OpenFlow port 1, datapath port 2; `veth_blue_br 2/3` reads as OpenFlow port 2, datapath port 3. The OpenFlow numbers match the `ofport` column from Step 2.

## Step 4, send the ping (the only command that populates `fdb`)

```text
root@lab-openvswitch:~/network-onboard# ip netns exec ns_red ping -c 3 10.0.0.2 2>&1 | tail -5
64 bytes from 10.0.0.2: icmp_seq=3 ttl=64 time=0.114 ms

--- 10.0.0.2 ping statistics ---
3 packets transmitted, 3 received, 0% packet loss, time 2038ms
rtt min/avg/max/mdev = 0.062/0.244/0.558/0.222 ms
```

3 of 3 ICMP echo replies. ICMP frames went from `ns_red` to `ns_blue` and back; the bridge saw both directions and learned the source MAC of every frame.

## The binding proof, by direct comparison

Putting the four data sources side by side:

| Source | OF port 1 row | OF port 2 row |
|---|---|---|
| `fdb/show` MAC (Step 5) | `56:a6:87:c1:15:9c` | `e2:10:43:fa:d2:bd` |
| `dpif/show` port name (Step 3) | `veth_red_br` | `veth_blue_br` |
| OVSDB `mac_in_use` for that port name (Step 2) | `76:49:2f:42:79:ca` | `7e:3a:00:75:07:a2` |
| Bridge-side end MAC, `ip -br link show <name>` in host ns (Step 1) | `76:49:2f:42:79:ca` (matches OVSDB) | `7e:3a:00:75:07:a2` (matches OVSDB) |
| Namespace-side end MAC, `ip netns exec <ns> ip -br link show <peer-name>` (Step 1, Step 6) | `56:a6:87:c1:15:9c` in `ns_red` (matches `fdb/show`) | `e2:10:43:fa:d2:bd` in `ns_blue` (matches `fdb/show`) |

Two equalities are decisive.

**Equality 1.** The MAC reported by `fdb/show` for OF port 1 is `56:a6:87:c1:15:9c`. The MAC of `veth_red` inside `ns_red`, captured in Step 1 and again in Step 6, is `56:a6:87:c1:15:9c`. They are identical character for character. The MAC of `veth_red_br` (the bridge-side end attached to OVS port 1), captured in Step 1 and present in OVSDB `mac_in_use` in Step 2, is `76:49:2f:42:79:ca`. It is **different** from the `fdb/show` value.

**Equality 2.** The MAC reported by `fdb/show` for OF port 2 is `e2:10:43:fa:d2:bd`. The MAC of `veth_blue` inside `ns_blue` is `e2:10:43:fa:d2:bd`. Identical. The MAC of `veth_blue_br` is `7e:3a:00:75:07:a2`, **different** from the `fdb/show` value.

The conclusion that follows from the byte-for-byte equalities: **the MAC `fdb/show` reports for an OpenFlow port is the source MAC of the frame the bridge received on that port, which in this veth-pair topology is the MAC of the namespace-side end (the originating netdev), not the MAC of the bridge-side end (the OVS-side netdev).**

The bridge-side end of a veth pair never originates traffic in this lab; it is a transport pipe between the bridge and the namespace. So OVS never sees a frame whose source MAC is the bridge-side MAC, and therefore never learns the bridge-side MAC. Per Step 2, the bridge-side MAC is recorded in OVSDB `Interface.mac_in_use` (because OVS does need to know the MAC of the netdev it owns), but it does not appear in `fdb/show`.

## How an engineer answers "which port is this MAC on?"

Given a MAC, the canonical command to find the OVS port that learned it is `ovs-appctl fdb/show <bridge>`. Step 5 above shows the actual invocation against `br0` and the verbatim output. The `port` column is the OpenFlow port number.

To translate that number to a port name, run either `ovs-appctl dpif/show` (Step 3 above) or `ovs-vsctl --columns=name,ofport list interface` (a row-per-port form of the OVSDB query in Step 2).

`dpif/show` shows each port as `<name> <OFport>/<DPport>` so the OF port number is on the left of the `/`. `ovs-vsctl --columns=name,ofport list interface` shows one row per `Interface` with the OF port number in the `ofport` column. Either tool maps OF port number to port name without ambiguity.

For a port that is one end of a veth pair (the most common case in a learning lab), the `mac_in_use` column of the matching OVSDB `Interface` row is the **bridge-side** MAC, which is **different** from the MAC `fdb/show` records. To find the originating netdev for an `fdb/show` MAC, the engineer queries the namespace on the other side of the veth (or, in a real virtual-machine deployment, queries the guest via the hypervisor) until they find a netdev whose MAC matches the `fdb/show` value.

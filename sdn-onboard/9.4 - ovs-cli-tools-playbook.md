## Bối cảnh: Tại sao cần một playbook có hệ thống

OVS là hệ thống phân tầng với ba thành phần lớn (ovs-vswitchd user-space daemon, ovsdb-server database daemon, kernel datapath module hoặc userspace DPDK/AF_XDP datapath) và bốn lớp cấu hình (OVSDB config → OpenFlow pipeline → megaflow cache → wire). Khi sự cố *"packet không đi qua"* xảy ra, cách tiếp cận ad-hoc (gõ lệnh ngẫu nhiên xem có gì không) tốn giờ, gọi ra nhiều tab terminal, và thường không tìm ra root cause.

Playbook có hệ thống giải quyết vấn đề bằng cách đặt ra trình tự kiểm tra cố định. Mỗi bước kiểm tra loại trừ một vùng có thể gây lỗi. Compass Appendix C đúc kết triết lý thành sáu câu hỏi:

1. Packet có đến NIC không? Đo qua `tcpdump` trên physical NIC.
2. Packet có vào OVS không? Đo qua counter `lookups:missed` tăng khi có traffic, bằng `ovs-dpctl show`.
3. Flow nào khớp với packet? Dùng `ovs-appctl ofproto/trace` mô phỏng pipeline.
4. Datapath action đúng không? Đọc dòng `Datapath actions:` trong theo dõi output.
5. Datapath có agree không? Dùng `ovs-dpctl dump-flows` đọc kernel/netdev thực.
6. Wire có agree không? `tcpdump` trên egress NIC.

Nếu sáu bước đều agree mà packet vẫn drop, root cause nằm **ngoài OVS**: upstream switch, firewall, MTU mismatch, ARP unresolved, BGP route withdrawn. Không tiếp tục debug OVS — xoay sang network path.

Phần này trình bày năm công cụ CLI với depth đủ để thực thi playbook, sau đó tích hợp thành §9.4.6 6-layer troubleshooting workflow. Cấu trúc mỗi section công cụ: grammar tổng thể → subcommand catalog → Anatomy block cho output quan trọng → 2-4 scenario operational → kịch bản bẻ gãy.

## 9.4.1 `ovs-vsctl`, transactional configuration language

### 9.4.1.1 Grammar, atomicity, và tại sao mọi thay đổi config đi qua ovs-vsctl

`ovs-vsctl` là công cụ quản lý OVSDB config. Mọi thay đổi *cấu hình* trên bridge, port, interface, QoS, mirror, netflow, controller, OpenFlow version — đều đi qua đây. Compass Ch A nêu bản chất cốt lõi mà nhiều kỹ sư mới bỏ qua:

> *"`ovs-vsctl` không phải shell bao quanh các lệnh đơn lẻ. Nó là transactional compiler: một invocation duy nhất trở thành một OVSDB transaction, và `--` là statement separator bên trong transaction đó."*

Nói cách khác, `ovs-vsctl -- A -- B -- C` biên dịch thành một transaction đơn nguyên với ba thao tác, commit atomic. Nếu thao tác B fail, A không được apply. Đây là bảo vệ quan trọng khỏi half-configured state mà thiếu nó, production system có thể rơi vào state: bridge đã tạo nhưng port chưa gắn → traffic rớt trong lúc config đang rollout.

Grammar tổng thể (man `ovs-vsctl(8)` §SYNOPSIS):

```
ovs-vsctl [global-options] command [-- command]...

global-options:
  --db=socket-path      (default: /var/run/openvswitch/db.sock)
  --timeout=seconds     (default: infinite)
  --no-wait
  --dry-run
  --oneline             (compact output)
  --pretty              (pretty-print JSON-like values)

command:
  <verb> <table> [<record>] [<column>[:<key>][=<value>]]...

verbs for scalar/composite column:
  list    find   get   set   add   remove   clear

verbs for row:
  create   destroy

helpers:
  --id=@name            (bind record to symbolic name in transaction)
  --may-exist           (succeed if row already matches)
  --if-exists           (succeed if row missing)
```

Reference cross-table trong cùng transaction dùng `--id=@name`. Ví dụ kinh điển (tạo bridge + add port trong một transaction):

```shell
$ ovs-vsctl add-br br0 -- add-port br0 eth0
```

Nếu `add-port` fail (eth0 đã thuộc bridge khác), `br0` không được tạo. Không có Windows state trung gian nửa-tạo.

### 9.4.1.2 Catalog subcommand nhóm bridge/port/interface

| Nhóm | Command | Syntax | Mục đích | Atomic? |
|---|---|---|---|---|
| Bridge lifecycle | `add-br` | `ovs-vsctl add-br BRIDGE [PARENT VLAN] [-- set Bridge BRIDGE datapath_type=...]` | Tạo bridge mới với datapath_type optional (`system` / `netdev`) | Yes |
| | `del-br` | `ovs-vsctl [--if-exists] del-br BRIDGE` | Xóa bridge và mọi port thuộc nó | Yes |
| | `br-exists` | `ovs-vsctl br-exists BRIDGE` | Exit 0 nếu tồn tại, 2 nếu không; script-friendly | - |
| | `list-br` | `ovs-vsctl list-br` | Liệt kê toàn bộ bridge (bao gồm fake bridge VLAN) | - |
| | `br-to-vlan` | `ovs-vsctl br-to-vlan BRIDGE` | Return VLAN ID nếu là fake bridge, 0 nếu real | - |
| | `br-to-parent` | `ovs-vsctl br-to-parent BRIDGE` | Return tên parent nếu fake, chính nó nếu real | - |
| Port management | `add-port` | `ovs-vsctl add-port BRIDGE PORT [COLUMN=VALUE]...` | Gắn port vào bridge; trunk/access/bond đều qua column | Yes |
| | `del-port` | `ovs-vsctl [--if-exists] [--with-iface] del-port [BRIDGE] PORT` | Tháo port khỏi bridge | Yes |
| | `list-ports` | `ovs-vsctl list-ports BRIDGE` | Liệt kê port (trừ local port cùng tên bridge) | - |
| | `port-to-br` | `ovs-vsctl port-to-br PORT` | Return bridge chứa port | - |
| Interface | `list-ifaces` | `ovs-vsctl list-ifaces BRIDGE` | Liệt kê interface (trừ local) | - |
| | `iface-to-br` | `ovs-vsctl iface-to-br IFACE` | Return bridge chứa interface | - |
| Controller | `set-controller` | `ovs-vsctl set-controller BRIDGE TARGET [TARGET]...` | Gắn bridge vào OpenFlow controller (ssl:host:port / tcp:host:port / unix:path) | Yes |
| | `del-controller` | `ovs-vsctl del-controller BRIDGE` | Tháo controller; bridge fall back về `fail_mode` | Yes |
| | `get-controller` | `ovs-vsctl get-controller BRIDGE` | Liệt kê controller đang gắn | - |
| | `set-fail-mode` | `ovs-vsctl set-fail-mode BRIDGE MODE` | `standalone` (tự forward) hoặc `secure` (drop mọi packet không có rule) | Yes |
| Database core | `list` | `ovs-vsctl list TABLE [RECORD]...` | In ra mọi column của record, format table | - |
| | `find` | `ovs-vsctl find TABLE [COLUMN[:KEY]=VALUE]...` | Filter rows theo điều kiện column | - |
| | `get` | `ovs-vsctl get TABLE RECORD [COLUMN[:KEY]]...` | Lấy giá trị cụ thể một column | - |
| | `set` | `ovs-vsctl set TABLE RECORD COLUMN[:KEY]=VALUE...` | Gán giá trị (atomic với other set trong cùng transaction) | Yes |
| | `add` | `ovs-vsctl add TABLE RECORD COLUMN [KEY=]VALUE...` | Append vào set/map column | Yes |
| | `remove` | `ovs-vsctl remove TABLE RECORD COLUMN VALUE|KEY[=VALUE]...` | Xóa entry khỏi set/map | Yes |
| | `clear` | `ovs-vsctl clear TABLE RECORD COLUMN...` | Reset column về empty set/map | Yes |
| | `create` | `ovs-vsctl create TABLE [COLUMN[:KEY]=VALUE]...` | Tạo row mới; return UUID | Yes |
| | `destroy` | `ovs-vsctl destroy TABLE RECORD...` | Xóa row | Yes |

Total: 30+ subcommand. Đa phần là convenience wrapper cho database core (`list`/`set`/`get`), nhưng bridge-level shortcut (`add-br`/`add-port`) phổ biến hơn vì rõ nghĩa.

**Anatomy từng dòng:**

| Dòng output | Ý nghĩa | Dấu hiệu bất thường |
|---|---|---|
| `c5e7a4c2-…` (dòng đầu) | UUID của record Open_vSwitch, unique per host; không bao giờ trùng | Mất UUID → `ovsdb-server` chưa load schema, check `journalctl -u ovsdb-server` |
| `Bridge br-int` | Một bridge entry, mỗi bridge render một block | Không có `br-int` sau OVN setup → `ovn-controller` chưa khởi động |
| `fail_mode: secure` | Khi controller disconnect: `secure` (drop mọi packet không khớp với flow) hoặc `standalone` (learn-normal). `secure` là chuẩn OVN | `standalone` trên br-int OVN → lộ lỗ hổng khi OVN controller down, traffic flood |
| `datapath_type: system` | Kernel datapath (mặc định). `netdev` là userspace DPDK/AF_XDP | `netdev` nếu không config DPDK → OVS fallback AF_XDP, performance thấp |
| `Port <name>` | Port entry thuộc bridge hiện tại | - |
| `Interface <name>` | Interface entry thuộc port hiện tại (thường 1:1; bond có N interface trên 1 port) | - |
| `type: internal` | OVS-managed TAP; có `internal` port linux netdev | - |
| `type: geneve` | Overlay tunnel; port được OVN dựng tự động | Thiếu option `key=flow` → OVN không inject tun_id per packet, overlay hỏng |
| `options: {csum="true", key=flow, remote_ip="10.200.0.12"}` | Tunnel options: csum (UDP checksum), key=flow (tun_id từ flow action), remote_ip endpoint xa | `key=<integer>` thay vì `key=flow` → OVN không dùng được (OVN cần per-flow tun_id) |
| `tag: 100` | Access VLAN tag trên port; packet vào port này có `vlan_tci` set 100 | Tag sai hoặc thiếu → traffic nhảy broadcast domain sai |
| `ovs_version: "2.17.9"` | Phiên bản OVS user-space | < 2.17 có thể không hỗ trợ một số OpenFlow 1.5 feature; < 2.15 không có SMC cache |

**Kịch bản bẻ gãy:**

- **Nếu dòng `ovs_version` không hiển thị** → `ovsdb-server` không chạy hoặc không response trong timeout; chạy `sudo systemctl status ovsdb-server` và `sudo ovs-vsctl --timeout=3 show`.
- **Nếu bridge có `datapath_type: netdev` mà host không cấu hình DPDK** → OVS fallback kernel datapath qua AF_XDP, throughput có thể giảm 30-40%; fix bằng `ovs-vsctl set Bridge br-int datapath_type=system`.
- **Nếu port `ovn-*` vắng mặt sau khi OVN đã provision LSP** → `ovn-controller` chưa pick up binding; kiểm tra `ovn-sbctl show` xem chassis đã register chưa và `ovn-sbctl list Port_Binding` xem binding có chassis gán chưa.

**Upstream nguồn:**

### 9.4.1.4 Scenario operational, tạo bridge với internal port có IP

**Bối cảnh:** dựng host development OVS, cần bridge với VLAN access port làm gateway nội bộ. Đây là mẫu phổ biến trong kolla-ansible khi config br-ex.

```shell
# Step 1: tạo bridge
$ sudo ovs-vsctl add-br br-demo

# Step 2: verify
$ sudo ovs-vsctl list-br
br-demo
br-ex
br-int

# Step 3: add VLAN access port tag=10 với internal interface
$ sudo ovs-vsctl add-port br-demo vlan10 tag=10 -- set Interface vlan10 type=internal

# Step 4: gán IP linux-netdev cho internal port
$ sudo ip link set vlan10 up
$ sudo ip addr add 192.168.10.1/24 dev vlan10

# Step 5: verify cả ba thao tác cùng apply atomic
$ sudo ovs-vsctl list Port vlan10
_uuid               : 8f3a2b4c-...
name                : vlan10
tag                 : 10
interfaces          : [1a2b3c4d-...]
...

$ sudo ovs-vsctl list Interface vlan10
_uuid               : 1a2b3c4d-...
name                : vlan10
type                : internal
admin_state         : up
link_state          : up
mac_in_use          : "3e:6a:...:..."
mtu                 : 1500
...

# Step 6 (teardown): xóa bridge, mọi port tự detach
$ sudo ovs-vsctl del-br br-demo
```

**Điểm mấu chốt:**

- `tag=10` trong `add-port` set column `tag` của table `Port`. Đây là access VLAN, khác với `trunks=10,20,30` (trunk port).
- `type=internal` của Interface tạo linux-netdev ảo. Không có `internal` thì interface phải là tên device thật (`eth0`, `enp3s0`).
- `add-port` và `set Interface` commit atomic vì cùng `ovs-vsctl` invocation.

### 9.4.1.5 Scenario, cấu hình QoS 2 queue với transaction phức tạp

**Bối cảnh:** lift từ man `ovs-vsctl(8)` §`CONFIGURATION COOKBOOK, Quality of Service`. Tạo QoS record với hai queue để chia băng thông 100 Mbps và 500 Mbps cho eth0 và eth1.

```shell
$ sudo ovs-vsctl \
    -- set Port eth0 qos=@newqos \
    -- set Port eth1 qos=@newqos \
    -- --id=@newqos create QoS type=linux-htb \
         other-config:max-rate=1000000000 \
         queues=0=@q0,1=@q1 \
    -- --id=@q0 create Queue other-config:min-rate=100000000 \
         other-config:max-rate=100000000 \
    -- --id=@q1 create Queue other-config:min-rate=500000000
```

**Anatomy transaction:**

| Dòng | Thao tác | Kết quả OVSDB |
|---|---|---|
| `set Port eth0 qos=@newqos` | Gán QoS (chưa tồn tại, bind symbolic) | Cột `qos` của row Port `eth0` point đến UUID resolved sau |
| `set Port eth1 qos=@newqos` | Gán cùng QoS cho eth1 | Hai port share cùng QoS record |
| `create QoS type=linux-htb` | Tạo QoS record kiểu HTB với max-rate 1 Gbps tổng | `--id=@newqos` bind UUID mới vào symbol `@newqos` |
| `queues=0=@q0,1=@q1` | Cột `queues` (map) gán queue-id 0 → Queue record `@q0`, 1 → `@q1` | Reference cross-table |
| `create Queue` với `min-rate=100M max-rate=100M` | Queue 0 capped 100 Mbps | - |
| `create Queue` với `min-rate=500M` | Queue 1 guarantee 500 Mbps min, không cap | - |

Sáu thao tác commit atomic. Nếu ví dụ `create QoS` fail (syntax lỗi), hai `set Port` không được apply, eth0/eth1 không bị gán QoS broken.

**Kịch bản bẻ gãy:**

- **Queue không cap max-rate** (chỉ có min-rate) → HTB share available bandwidth; có thể vượt xa mức dự định khi port khác idle. Thiết kế có chủ đích nhưng phải document.
- **`max-rate=<min-rate>` của Queue** → không đạt được target max-rate vì min-rate là sàn cứng. Coerced theo HTB algorithm.
- **Bỏ quên clear khi xóa QoS:** phải `ovs-vsctl clear Port eth0 qos` + `destroy QoS <uuid>`. Nếu chỉ `destroy`, reference từ Port tồn tại → OVSDB raise "dangling reference" error.

**Upstream:** man `ovs-vsctl(8)` §`CONFIGURATION COOKBOOK, Quality of Service`.

### 9.4.1.6 Scenario, query `find` + `get` cho script automation

**Bối cảnh:** script giám sát cần pull state định kỳ. `show` không parse-friendly; `list` dài; `find` + `get` là format-machine.

```shell
# Tìm mọi Port có VLAN tag 100
$ sudo ovs-vsctl find Port tag=100
_uuid               : 7c9d4e2a-1f8b-4c3d-a7e6-5b2a9d1e8f3c
name                : "tap-vm-db"
tag                 : 100
interfaces          : [6e1a3b8c-2d4f-4a5e-b8c7-9d3e1a2b7f5c]
...

# Tìm Interface có RX error > 0
$ sudo ovs-vsctl --columns=name,statistics find Interface 'statistics:rx_errors>0'
name                : eth1
statistics          : {..., rx_errors=42, ...}

# Get 1 giá trị scalar (script-friendly)
$ sudo ovs-vsctl get Interface eno1 statistics:rx_bytes
184756234890

# Get giá trị từ nested map
$ sudo ovs-vsctl get Interface ovn-huyvl-0 options:remote_ip
"10.200.0.12"

# Liệt kê bridge + fail_mode + datapath_type dưới dạng table
$ sudo ovs-vsctl --columns=name,fail_mode,datapath_type list Bridge
_uuid               : 3a8b9c4d-...
name                : br-int
fail_mode           : secure
datapath_type       : system

_uuid               : 5e7f8a9b-...
name                : br-ex
fail_mode           : standalone
datapath_type       : system
```

**Anatomy query operator:**

| Operator | Ý nghĩa | Ví dụ |
|---|---|---|
| `COLUMN=VALUE` | Exact match | `tag=100` |
| `COLUMN!=VALUE` | Not equal | `fail_mode!=secure` |
| `COLUMN<VALUE` | Less than (numeric) | `statistics:rx_bytes<1000` |
| `COLUMN>VALUE` | Greater than | `statistics:rx_errors>0` |
| `COLUMN{=}{A,B}` | Set membership | `protocols{=}{OpenFlow13,OpenFlow15}` |
| `COLUMN:KEY=VALUE` | Map entry | `options:remote_ip=10.0.0.1` |

Script production ví dụ — poll RX error rate per interface:

```bash
#!/bin/bash
for iface in $(sudo ovs-vsctl --bare --columns=name list Interface); do
  rx_err=$(sudo ovs-vsctl get Interface "$iface" statistics:rx_errors 2>/dev/null || echo 0)
  rx_bytes=$(sudo ovs-vsctl get Interface "$iface" statistics:rx_bytes 2>/dev/null || echo 0)
  if [ "$rx_err" -gt 0 ]; then
    echo "INTERFACE $iface: rx_errors=$rx_err rx_bytes=$rx_bytes"
  fi
done
```

## 9.4.2 `ovs-ofctl`, OpenFlow flow management

### 9.4.2.1 Grammar, version negotiation, và flow syntax

`ovs-ofctl` nói OpenFlow với bridge: dump flow, thêm/sửa/xóa flow entry, theo dõi event stream. Đây là công cụ **write-heavy** duy nhất cho pipeline programming — `ovs-vsctl` không sửa được flow, chỉ config bridge.

Grammar tổng thể (man `ovs-ofctl(8)` §SYNOPSIS):

```
ovs-ofctl [OPTIONS] COMMAND [SWITCH] [ARGS...]

key OPTIONS:
  -O, --protocols=VERSION[,VERSION...]
       OpenFlow10 | OpenFlow11 | OpenFlow12 | OpenFlow13 | OpenFlow14 | OpenFlow15
  --strict           (match exactly, no wildcards)
  --bundle           (atomic group using OF 1.4 bundle transactions)
  --sort[=FIELD]     (sort dump-flows output)
  --rsort[=FIELD]
  --color[=never|always|auto]
  --names | --no-names (port names vs numbers)

COMMANDS (switch management):
  show SWITCH
  dump-tables SWITCH
  dump-ports SWITCH [NETDEV]
  dump-flows SWITCH [FLOWSPEC]
  dump-aggregate SWITCH [FLOWSPEC]
  add-flow SWITCH FLOW
  add-flows SWITCH FILE
  mod-flows [--strict] SWITCH FLOW
  del-flows [--strict] SWITCH [FLOW]
  replace-flows [--readd] SWITCH FILE
  diff-flows SOURCE1 SOURCE2
  add-group SWITCH GROUP    /  dump-groups SWITCH [GROUP]
  add-meter SWITCH METER    /  dump-meters SWITCH [METER]
  monitor SWITCH [MISS-LEN] [invalid_ttl] [watch:SPEC...]
  snoop SWITCH
```

**Quan trọng:** LUÔN pass `-O OpenFlow13` (hoặc 1.4, 1.5). Default OVS `ovs-ofctl` là OpenFlow 1.0 — version cổ, không hỗ trợ multi-table, group, meter, TLV match, v.v. Không pass → tham chiếu pipeline thất bại trên setup hiện đại (OVN yêu cầu OpenFlow 1.4+).

### 9.4.2.2 Flow syntax, các keyword control

Flow spec là chuỗi `FIELD=VALUE` phân tách bởi comma hoặc whitespace. Khi có space cần quote. Các keyword control quan trọng:

| Keyword | Ý nghĩa | Default | Khi thiếu |
|---|---|---|---|
| `table=<id>` | Giới hạn operation tại table ID (0-254) | 0 với `add-flow`; wildcard với dump/del | Tất cả table |
| `priority=<val>` | Priority 0-65535; cao hơn khớp với trước | 32768 | 32768 (không nằm ở top) |
| `cookie=<val>[/<mask>]` | Opaque 64-bit ID, dùng để group/tag flow | 0 | - |
| `idle_timeout=<sec>` | Flow expire sau N giây không packet hit | 0 (không timeout) | - |
| `hard_timeout=<sec>` | Flow expire sau N giây tuyệt đối | 0 | - |
| `actions=<spec>` | Danh sách action (comma-separated) | Required | Flow drop nếu thiếu |
| `duration=...`, `n_packets=...`, `n_bytes=...` | Ignored trong add/mod (chỉ dump show) | - | - |

Match field xem `ovs-fields(7)`. Ví dụ flow spec phức tạp:

```
table=10,priority=1000,
  cookie=0xdead1000/0xffff0000,
  idle_timeout=60,hard_timeout=600,
  dl_type=0x0800,nw_proto=6,tp_dst=80,
  actions=mod_dl_dst:aa:bb:cc:dd:ee:ff,
          mod_nw_src:10.1.0.100,
          goto_table:20
```

Dòng mạch có: table 10, priority 1000, cookie cao masked 0xffff, idle 60s hard 600s, match TCP destination 80, ba action (rewrite MAC dest, rewrite source IP, goto table 20).

**Anatomy từng dòng:**

| Thành phần | Ý nghĩa | Dấu hiệu bất thường |
|---|---|---|
| `OFPT_FEATURES_REPLY (OF1.5)` | Version negotiation thành công với OF 1.5 | Nếu hiển thị `(OF1.0)` → bạn quên `-O OpenFlow15`, dùng wrong version |
| `dpid:0000000000000001` | Datapath ID 64-bit, unique per switch | Trùng dpid giữa hai bridge → controller mistakenly thấy một switch |
| `n_tables:254` | Số OpenFlow table hỗ trợ. OVS default 254 | < 254 → bridge bị restrict, check `ovs-vsctl get Bridge br-int flow_tables` |
| `n_buffers:0` | OVS không buffer packet_in nữa (OF 1.4+) | - |
| `capabilities` | List cờ switch support (FLOW_STATS, TABLE_STATS, PORT_STATS, GROUP_STATS, QUEUE_STATS, ARP_MATCH_IP) | Thiếu `GROUP_STATS` → Group multicast không accurate counter |
| `actions: (none)` | OF 1.1+ không hard-code action; trả `(none)` | - |
| `1(tap-vm-web)` | Port number `1`, name `tap-vm-web` | Port number disappear → VM unplugged hoặc ovs-vswitchd restart assign lại |
| `addr:3e:6a:...` | MAC của port | MAC 00:... → internal port chưa gán MAC, check Interface `mac` column |
| `config: PORT_DOWN` | Admin down (set bởi `ovs-ofctl mod-port PORT down`) | PORT_DOWN mà traffic expected → user config sai |
| `state: LINK_DOWN` | Physical link layer down | LINK_DOWN trên Geneve `ovn-*` port → bình thường (Geneve không có physical link) |
| `current: 10GB-FD COPPER` | Link speed + duplex + media | SFP module error hiển thị `current:` trống |
| `speed:` | Bandwidth report trong Mbps | Geneve port luôn 0 Mbps (virtual) |
| `LOCAL(br-int)` | Port number `LOCAL` reserved cho bridge internal | Mất LOCAL port → bridge corruption |
| `OFPT_GET_CONFIG_REPLY: frags=normal miss_send_len=0` | Bridge config: fragments reassemble normal; không gửi packet_in header-only lên controller | `miss_send_len=N` → bridge sẽ gửi N byte packet_in, có thể spam controller |

**Kịch bản bẻ gãy:**

- **`state: LINK_DOWN` kéo dài trên port eth0** → vấn đề vật lý (cable, SFP, driver). Dùng `ethtool eth0` kiểm chứng autoneg và media type.
- **Port number không stable giữa các lần `show`** → `ovs-vswitchd` vừa restart; luôn dùng port name (`tap-vm-web`) trong flow spec thay vì port number (`1`) để tránh broken flow sau restart.
- **`(OF1.0)` thay vì `(OF1.5)`** → thiếu flag `-O`. Fix: `ovs-ofctl -O OpenFlow15 show br-int`.

| Field | Ý nghĩa | Dấu hiệu bất thường |

| `table=<N>` | Table number 0-254. OF 1.1+ có multi-table | - |

| `n_bytes=<N>` | Counter byte match | Ratio `n_bytes / n_packets` = avg packet size, so sánh với expected MTU |
| `idle_timeout=<sec>` | Flow expire sau N giây không packet hit. Field chỉ hiển thị khi ≠ 0 | Nếu flow critical (ACL) có idle_timeout → flow có thể tự xóa, check controller logic |
| `hard_timeout=<sec>` | Flow expire sau N giây tuyệt đối, ẩn khi = 0 | Trong OVN flow hard_timeout thường 0; có giá trị → controller hard-set, usually for learn() |
| `priority=<N>` | Match priority. Với hai flow cùng match → highest priority wins | Hai flow cùng priority + overlap match → behavior undefined |
| `<match spec>` | Match field list, ví dụ `in_port=1,ip,nw_dst=10.1.1.1` | - |
| `actions=<action list>` | Action thi hành khi match | `actions=drop` trên flow critical → đã xác định bị loại bỏ (drop), kiểm chứng mong đợi |

| Action | Ý nghĩa |
|---|---|

| `mod_dl_src:aa:bb:cc:dd:ee:ff` | Set eth_src = gateway MAC |
| `load:0x2->NXM_OF_ARP_OP[]` | Set ARP opcode = 2 (reply) |
| `move:NXM_NX_ARP_SHA[]->NXM_NX_ARP_THA[]` | Target hardware = sender (echo back) |
| `move:NXM_OF_ARP_SPA[]->NXM_OF_ARP_TPA[]` | Target protocol = sender IP |
| `load:0xaabbccddeeff->NXM_NX_ARP_SHA[]` | Sender hardware = gateway MAC |
| `load:0x0a010101->NXM_OF_ARP_SPA[]` | Sender protocol = 10.1.1.1 (gateway IP) |
| `IN_PORT` | Output back via ingress port (ARP reply) |

Đây là classic OVN ARP responder flow, inject ARP reply cho gateway IP mà không forward lên CPU/controller.

**Kịch bản bẻ gãy:**

- **`n_packets=0` cho ARP responder expected** → VM chưa gửi ARP request; tạo traffic test `ping <gateway>` rồi re-dump.
- **`n_packets` tăng nhanh nhưng `n_bytes/n_packets` << expected packet size** → packet bị fragment; check MTU underlay.
- **Duration liên tục reset về 0** → flow thrashing; `ovn-controller` đang compute lại liên tục vì NBDB change không stable. Check `ovn-appctl -t ovn-controller inc-engine/show-stats`.

### 9.4.2.5 Scenario, flow-table hygiene với `replace-flows` và `diff-flows`

**Bối cảnh:** production rule quan trọng — trước khi ghi đè flow table, luôn preview diff. `replace-flows` tự động delete missing + add new + update changed, atomic. Nhưng nếu file input sai, production trong 1 giây → packet drop.

```shell
# Step 1: Dump hiện tại để backup
$ sudo ovs-ofctl -O OpenFlow15 dump-flows br-int > /tmp/br-int.current.flows

# Step 2: Preview diff giữa current và new file
$ sudo ovs-ofctl -O OpenFlow15 diff-flows br-int /tmp/br-int.new.flows
-table=0,priority=100,in_port=1 actions=output:2
+table=0,priority=100,in_port=1 actions=output:3
-table=10,priority=50,ip,nw_dst=10.1.1.0/24 actions=drop

# Step 3: Đánh giá diff — chỉnh output port 1→2→3 là đúng migration; xóa ACL drop là
#   ý định. Approve.

# Step 4: Apply atomic
$ sudo ovs-ofctl -O OpenFlow15 --bundle replace-flows br-int /tmp/br-int.new.flows

# Step 5: Verify
$ sudo ovs-ofctl -O OpenFlow15 diff-flows br-int /tmp/br-int.new.flows
(empty output — đã match)
```

**Quy tắc production:**
- **LUÔN `diff-flows` trước `replace-flows`.** Không có `--dry-run` trong OpenFlow; `diff-flows` là dry-run.
- **Thêm `--bundle`** (OpenFlow 1.4+): mọi change commit trong một bundle atomic. Nếu bundle fail, rollback full, không có partial state.
- **Backup `dump-flows` sang file trước** khi thao tác production. Script automation: chạy backup sau mỗi `replace-flows` thành công.

### 9.4.2.6 Scenario, monitor live flow event

**Bối cảnh:** debug thực time xem flow nào được install/remove/modify. Tương tự `tail -f` nhưng cho OpenFlow.

```shell
$ sudo ovs-ofctl -O OpenFlow15 monitor br-int watch:
send: OFPT_FLOW_MOD (OF1.5) (xid=0x4): ADD table:0 priority=100,in_port=1 actions=output:2
NXT_FLOW_MOD_TABLE_ID (OF1.0): table_id enabled
send: OFPT_SET_ASYNC (OF1.5) (xid=0x5): ...
OFPT_FLOW_REMOVED (OF1.5) (xid=0x0): cookie=0xabc, table=10, priority=50, reason=idle, duration=60.1s
  ip,nw_dst=10.1.1.5 actions=drop
  packet_count:0 byte_count:0
```

**Anatomy event stream:**

| Event type | Ý nghĩa | Khi nào xuất hiện |
|---|---|---|
| `OFPT_FLOW_MOD ADD/MOD/DEL` | Controller install/modify/delete flow | Controller hoạt động, flow change |
| `OFPT_FLOW_REMOVED` | Flow expire do timeout hoặc delete | idle_timeout hết hạn, hoặc `del-flows` từ controller |
| `OFPT_PACKET_IN` | Packet lên controller (table-miss hoặc CONTROLLER action) | Flow thiếu match, controller đang learn |
| `OFPT_PORT_STATUS` | Port add/remove/state change | Admin config change hoặc link flap |

**Field trong FLOW_REMOVED:**
- `reason=idle` → hết idle_timeout, không có packet N giây.
- `reason=hard` → hết hard_timeout.
- `reason=delete` → explicit `del-flows`.
- `reason=evict` → flow_table eviction policy (khi vượt flow_limit).
- `packet_count/byte_count` → counter cuối trước khi xóa. Giá trị 0 → flow không được hit, có thể dấu hiệu khớp với spec sai.

**Use case:** mở 2 terminal. Terminal 1 chạy `monitor`, terminal 2 `ping` host target. Xem packet_in có xuất hiện không, OFPT_FLOW_MOD có cài đặt flow mới không.

**Upstream:** man `ovs-ofctl(8)` §`monitor` + OpenFlow 1.5 spec §7.4 Asynchronous Messages.

## 9.4.3 `ovs-appctl`, introspection RPC qua Unix socket

### 9.4.3.1 Grammar, target discovery

`ovs-appctl` là một wrapper generic gửi RPC command tới bất kỳ OVS daemon nào qua Unix socket. Đây là công cụ duy nhất cho "live introspection" — không ảnh hưởng config (không ghi OVSDB), chỉ truy vấn state runtime của daemon.

Grammar:

```
ovs-appctl [OPTIONS] [-t TARGET | --target=TARGET] COMMAND [ARG...]

TARGET:
  - process name: ovs-vswitchd (default), ovsdb-server, ovn-controller, ovn-northd
  - pidfile path: /var/run/openvswitch/<name>.pid
  - unix socket: unix:/var/run/openvswitch/<name>.<pid>.ctl

OPTIONS:
  --timeout=secs
  -v, --verbose
  -h, --help

Built-in COMMANDS cho mọi daemon:
  list-commands           → list mọi command target hỗ trợ
  version                 → in version + compile date
  vlog/list               → list logging module + current level
  vlog/list-pattern       → pattern per destination
  vlog/set SPEC           → set log level
  vlog/reopen             → close + reopen log file (sau logrotate)
  exit                    → graceful shutdown daemon
  memory/show             → memory usage breakdown

Thêm daemon-specific command — xem `list-commands` trên từng daemon để biết.
```

**Bước đầu tiên luôn là:**

```shell
$ sudo ovs-appctl -t ovs-vswitchd list-commands
```

Lệnh trên liệt kê mọi command mà ovs-vswitchd hỗ trợ (thường ~80-120 command tùy OVS version). Phân nhóm lớn:

| Nhóm | Prefix | Mục đích |
|---|---|---|
| Bridge introspection | `bridge/*`, `fdb/*`, `mdb/*` | Bridge state, MAC learning, multicast |
| OpenFlow simulation | `ofproto/*` | Trace pipeline, inspect classifier |
| Datapath introspection | `dpctl/*`, `dpif/*`, `dpif-netdev/*` | Kernel/userspace datapath state |
| Upcall + revalidator | `upcall/*`, `revalidator/*` | Kernel → userspace flow install |
| Bond + LACP | `bond/*`, `lacp/*` | Bond member state, LACP handshake |
| L2 protocols | `stp/*`, `rstp/*`, `cfm/*`, `bfd/*` | Spanning tree, CFM, BFD |
| Tunnel | `tnl/*` | ARP neighbor cho tunnel, stats |
| Observability | `coverage/*`, `memory/*` | Counter internal + memory |
| Logging | `vlog/*` | Log level per module |

### 9.4.3.2 `ofproto/trace`, công cụ mạnh nhất trong bộ

`ofproto/trace` mô phỏng một packet đi qua OpenFlow pipeline của một bridge, KHÔNG gửi packet thật. Output thể hiện từng table hit, từng rule match, từng action execute, final action set, và datapath action đầu ra. Đây là phương tiện debug chính mà mọi kỹ sư OVS phải thuộc.

**Command syntax (giản lược):**

```
ovs-appctl ofproto/trace [OPTIONS] BRIDGE FLOW_SPEC [OPTIONS]

OPTIONS:
  -generate         → actually execute side effect (learn, ct commit)
  -consistent       → force consistent table cache
  --names | --no-names
  --ct-next=<state> → simulate conntrack returning specific state
```

**Flow spec:** chuỗi `FIELD=VALUE` tương tự `ovs-ofctl dump-flows` khớp với spec.

```shell
          dl_dst=aa:bb:cc:00:00:02,nw_src=10.1.0.5,nw_dst=10.1.0.10,
          nw_frag=no,tp_dst=0x50/0xffff

Datapath actions: 2
```

**Anatomy bốn khối output:**

**Khối 1, `Flow:`** — ban đầu của packet simulate, ovs-appctl fill các field thiếu (vlan_tci=0, nw_tos=0, nw_ttl=0, tcp_flags=0) theo default. Đây là *input* vào pipeline.

**Khối 2, `bridge("br-int") ... N. <match>, priority P`** — từng table hit, theo thứ tự table 0 → table 8 → table 16 → table 32 → table 33. Mỗi dòng format:
- `<table_id>. <match_spec>, priority <P>, cookie <cookie>`
- Các dòng con là action execute trong table đó.

| Ký hiệu | Ý nghĩa |
|---|---|
| `0. in_port=1, priority 100` | Table 0, match `in_port=1`, priority 100 |
| `load:0x1->NXM_NX_REG13[]` | Action: load 0x1 vào register 13 (đánh dấu logical datapath = 1) |
| `resubmit(,8)` | Action: submit packet lên table 8 cùng reg context |
| `priority 0` | Table-miss rule (priority 0) match, default fallthrough |
| `reg13=0x1,dl_dst=...,priority 50` | Table 16, match reg13 cộng eth_dst, priority 50 |
| `load:0x2->NXM_NX_REG15[]` | Set logical output port = 2 (REG15 là logical output port trong OVN) |
| `output:2` | Cuối cùng, physical output port 2 |

**Khối 3, `Final flow:`** — state của packet SAU khi pipeline xong, trước khi exit. So sánh với `Flow:` initial để thấy modification: packet giờ có `reg13=0x1,reg15=0x2` do load ở table 0 và 16.

**Khối 4, `Megaflow:`** — megaflow cache signature tương ứng cho packet này. Đây là *classifier mask* mà kernel datapath sẽ cài đặt sau upcall. Field nào trong megaflow spec tức là field đó **cần được khớp với exact** để kernel reuse cache. Trong ví dụ trên, `tp_dst=0x50/0xffff` nghĩa là kernel khớp với exact port 80; nhưng `nw_src` và `nw_dst` đều khớp với exact → megaflow này không generalize cho IP khác.

**Khối 5, `Datapath actions:`** — action cuối cùng thi hành tại datapath (kernel hoặc userspace). Đây là ground truth — nếu `Datapath actions: drop` → packet sẽ bị drop trong kernel; `Datapath actions: 2,3` → output port 2 và 3.

**Kịch bản bẻ gãy:**

- **`Datapath actions: drop`** → đi lên pipeline tìm rule nào khớp với `drop`. Trong trace, check table hit cuối cùng — sẽ thấy action drop explicit.
- **Megaflow quá specific (quá nhiều field exact)** → kernel sẽ cài đặt flow chỉ cho 1 flow specific, mọi packet khác biệt sẽ miss cache → upcall liên tục. Check `upcall/show` xem `flows_install_rate` cao bất thường không.
- **Cookie `0x0` cho mọi flow** → không phải OVN (OVN sinh cookie != 0). Có thể bạn đang debug br-int không bị OVN quản lý, hoặc đang trên bridge thuần OpenFlow.
- **Trace không progress qua table 8** → flow tại table 0 không có `resubmit(,8)` hoặc `goto_table:8`. OVN pipeline skip table → check `ovn-controller` translation, có thể chassis chưa register đúng.

**Upstream:** man `ovs-vswitchd(8)` §`ofproto/trace`, OVS Advanced Tutorial §Table 0 Testing, `ofproto/ofproto-dpif-xlate.c` function `ofproto_trace()`.

### 9.4.3.4 Đọc hiểu output `ovs-appctl fdb/show`

**Bối cảnh:** debug L2 forwarding — MAC learning table sai dẫn đến broadcast storm hoặc traffic đi sai port. FDB là bản đồ MAC → port + VLAN học tự động trong bridge `standalone` hoặc `learn` action explicit.

```shell
$ sudo ovs-appctl fdb/show br-int
 port  VLAN  MAC                Age
    1   100  aa:bb:cc:00:00:01    4
    2   100  aa:bb:cc:00:00:02   12
    3     0  aa:bb:cc:00:00:03   87
    1   100  aa:bb:cc:00:00:04  287
```

**Anatomy từng field:**

| Field | Ý nghĩa | Dấu hiệu bất thường |
|---|---|---|
| `port` | OpenFlow port number. Tra lại với `ovs-ofctl show br-int` | Port `LOCAL` trong FDB → bridge ít khi cần |
| `VLAN` | VLAN ID. `0` = untagged | VLAN khác expected → port access tag sai |
| `MAC` | MAC address học được từ packet incoming | Duplicate MAC trên hai port → MAC flapping, check L2 loop |
| `Age` | Giây từ khi entry được học lần cuối. Mặc định aging 300s | Entry > 300s vẫn tồn tại → check `ovs-vsctl get Bridge br-int other_config:mac-aging-time` |

**Kịch bản bẻ gãy:**

- **MAC appearing/disappearing giữa nhiều port (MAC flapping)** → có L2 loop; check STP state (`ovs-appctl stp/show` hoặc `rstp/show`).
- **FDB empty trên bridge đang có traffic** → bridge có fail_mode=secure và thiếu learn flow → flush + reapply flow.
- **Age luôn 0** → FDB được update liên tục; host spam broadcast trên MAC đó.

**Useful companion command:** `fdb/flush br-int` buộc relearn toàn bộ MAC (dùng sau thay đổi topology).

### 9.4.3.5 Đọc hiểu output `ovs-appctl upcall/show`

**Bối cảnh:** upcall là cơ chế mà kernel datapath "hỏi" userspace khi gặp packet không khớp với megaflow cache. Rate cao = performance issue. `upcall/show` cho thấy health của fast/slow path handoff.

```shell
$ sudo ovs-appctl upcall/show
system@ovs-system:
  flows         : (current 1247) (avg 1250) (max 1350) (limit 200000)
  dump duration : 8ms
  ufid enabled : true

  1: (keys 624) (max-flows 64) (avg-updates 1.2) (busy cycles 0.03%)
  2: (keys 623) (max-flows 63) (avg-updates 1.3) (busy cycles 0.03%)
```

**Anatomy từng field:**

| Field | Ý nghĩa | Dấu hiệu bất thường |
|---|---|---|
| `system@ovs-system:` | Datapath name | - |
| `flows: (current N) (avg N) (max N) (limit N)` | Flow count in datapath (hiện tại, moving avg, max đạt, limit hard) | `current` gần `limit` → sắp flow full, upcall sẽ throttle; `max` >> `avg` → traffic burst |
| `dump duration: Nms` | Thời gian revalidator dump full flow table. Default target < 100 ms | `dump duration > 1s` → flow table quá lớn hoặc memory contention; đây là triệu chứng Part 9.26 revalidator storm |
| `ufid enabled: true` | OVS 2.7+ dùng Unique Flow ID cho track flow. False → rollback mode | `false` trên OVS ≥ 2.7 → config issue |
| `1: (keys N) (max-flows N)` | Revalidator thread 1 stats | - |
| `keys N` | Số ukey (unique key) thread này manage | Thread unbalanced (one thread keys >> other) → rxq assignment skew |
| `max-flows N` | Max flow thread manage | - |
| `avg-updates N` | Avg update per flow (revalidator pass) | > 5 → flow thrashing, reinstall liên tục |
| `busy cycles N%` | % CPU thread dùng | > 50% → thread saturated |

**Kịch bản bẻ gãy:**

- **`flows current` gần `limit`** → upcall sẽ drop new flow; tăng limit qua `ovs-vsctl set Open_vSwitch . other_config:flow-limit=<N>` hoặc reduce flow explosion tại OpenFlow layer.
- **`dump duration > 1s`** → Phần 9.26 revalidator storm pattern; check `coverage/show | grep revalidate` và commit `180ab2fd635e` fix path.
- **`busy cycles` of single thread > 90%** → revalidator bottleneck; OVS ≥ 2.18 có thread group to help, otherwise CPU saturated.

**Upstream:** OVS source `ofproto/ofproto-dpif-upcall.c` function `upcall_unixctl_show()`, NSDI 2015 paper §5.

**Counter class quan trọng:**

| Counter | Ý nghĩa | Healthy range |
|---|---|---|
| `hit` / `miss` | Megaflow cache hit/miss trong datapath | miss/hit < 1% = healthy; miss/hit > 10% = flow explosion hoặc classifier issue |

## 9.4.4 `ovs-dpctl`, raw datapath inspection

### 9.4.4.1 Grammar và khi nào cần

`ovs-dpctl` nói trực tiếp với datapath (kernel hoặc userspace). Nó bypass ofproto layer — không đi qua OpenFlow, không liên quan classifier, ofproto, hay connection manager. Đây là "ground truth" về những gì datapath ACTUALLY forward.

Khi nào dùng `ovs-dpctl` thay vì `ovs-ofctl`:

| Tình huống | Tool phù hợp |
|---|---|
| Xem flow OpenFlow đã program | `ovs-ofctl dump-flows` |
| Xem megaflow kernel đã install | `ovs-dpctl dump-flows` |
| Kiểm chứng datapath có consistent với OpenFlow không | Cả hai, so sánh cookie + counter |
| Debug conntrack của datapath | `ovs-dpctl dump-conntrack` hoặc `ovs-appctl dpctl/dump-conntrack` |
| Feature flag datapath hỗ trợ gì | `ovs-dpctl get-dp-features` |

Grammar:

```
ovs-dpctl [OPTIONS] COMMAND [ARG...]

key COMMANDS:
  show [DP]                              → datapath summary
  dump-dps                               → list datapath names
  dump-flows [-m] [DP]                   → flow table (-m = more detail)
  dump-conntrack [-s] [-m] [DP]          → conntrack entries
  flush-conntrack [DP] [zone=ZONE]       → flush conntrack
  ct-stats-show [-m] [DP]                → conntrack stats by protocol
  get-flow [-m] [DP] ufid:UFID           → specific flow by UFID

key OPTIONS:
  -m, --more          → more detail (unmasked match fields, ufid)
  -s, --statistics    → include stats
  --names | --no-names
```

### 9.4.4.2 Đọc hiểu output `ovs-dpctl show`

```shell
$ sudo ovs-dpctl show
system@ovs-system:
  lookups: hit:287423 missed:8127 lost:0
  flows: 1247
  masks: hit:4127821 total:1247 hit/pkt:3.47
  port 0: ovs-system (internal)
  port 1: br-int (internal)
  port 2: tap-vm-web (internal)
  port 3: tap-vm-db (internal)
  port 4: ovn-huyvl-0 (geneve)
  port 5: ovn-huyvl-1 (geneve)
```

**Anatomy từng field:**

| Field | Ý nghĩa | Dấu hiệu bất thường |
|---|---|---|
| `system@ovs-system:` | Datapath name | - |
| `lookups: hit:N missed:N lost:N` | Lookup counter: hit (megaflow match), missed (upcall trigger), lost (packet drop do full queue) | `lost > 0` → kernel queue overflow, tình trạng khẩn cấp về hiệu năng; `missed/hit > 5%` → flow explosion |
| `flows: N` | Megaflow count hiện có trong datapath | Khớp với `upcall/show flows current` |
| `masks: hit:N total:N hit/pkt:N` | Classifier mask lookup: hit = total mask hit, total = tổng mask có, hit/pkt = avg mask per packet | `hit/pkt > 10` → classifier linear; Phần 9.15 tuple space search stress |
| `port N: <name> (<type>)` | Port number + name + type (internal / system / geneve / vxlan / netdev) | Mismatch giữa dpctl port N và ofctl port N → rare (kernel dp và userspace dp desync) |

**Kịch bản bẻ gãy:**

- **`lost > 0`** → packet drop vì kernel Netlink queue full; check `dmesg | grep -i 'openvswitch'` cho drop reason. Giải pháp: tăng `net.core.netdev_max_backlog`.
- **`missed/hit > 10%` consistent** → flow explosion; check `dump-flows` có flow-table large không. Với OVN → check `ovn-controller` I-P stat.
- **`hit/pkt > 15`** → classifier subtable explosion; tham khảo Phần 9.15 tuple space search optimization.

### 9.4.4.3 Đọc hiểu output `ovs-dpctl dump-flows -m`

**Bối cảnh:** nhìn thấy flow kernel thực sự đã cài, với megaflow mask full. Flag `-m` in unmasked field + UFID.

```shell
$ sudo ovs-dpctl dump-flows -m | head -3
ufid:a1b2c3d4-5e6f-7a8b-9c0d-e1f2a3b4c5d6,
  recirc_id(0),dp_hash(0/0),skb_priority(0/0),in_port(3),
  skb_mark(0/0),ct_state(0/0),ct_zone(0/0),ct_mark(0/0),ct_label(0/0),
  eth(src=aa:bb:cc:00:00:01,dst=aa:bb:cc:00:00:02),
  eth_type(0x0800),
  ipv4(src=10.1.0.5,dst=10.1.0.10,proto=6,tos=0/0,ttl=64,frag=no),
  tcp(src=50000/0,dst=80),
  tcp_flags(0/0),
  packets:18234, bytes:2145678, used:0.082s,
  flags:SP, actions:4
```

**Anatomy từng token:**

| Token | Ý nghĩa |
|---|---|
| `ufid:<uuid>` | Unique Flow ID, 128-bit, OVS 2.7+ |
| `recirc_id(0)` | Recirculation ID = 0 → packet chưa recirculate; != 0 nếu đã qua ct() hoặc conjunction |
| `in_port(3)` | Ingress datapath port number (match dpctl show) |
| `ct_state(0/0)` | Conntrack state + mask (0/0 = no match) |
| `eth(src=..., dst=...)` | Ethernet match |
| `eth_type(0x0800)` | IP |
| `ipv4(src=..., dst=..., proto=6, ...)` | Layer 3 match; `proto=6` = TCP |
| `tcp(src=50000/0, dst=80)` | TCP source khớp với bất kỳ (mask 0), dest exact 80 |
| `packets:N` | Packet matched flow này |
| `bytes:N` | Bytes matched |
| `used:Ns` | Seconds ago lần khớp với cuối; dùng để revalidator GC |
| `flags:SP` | S=SYN, P=PUSH; TCP flags observed |
| `actions:4` | Output port 4 (geneve tunnel → OVS remote chassis) |

**Kịch bản bẻ gãy:**

- **`used > 10s` cho flow expected active** → traffic stop, flow sắp bị GC.
- **`packets:0, bytes:0` với flow đã tồn tại > vài giây** → flow cài đặt nhưng chưa có match; có thể upcall-then-install race.
- **`actions:drop`** → kernel datapath đã "compile" thành drop; kiểm chứng `ofproto/trace` có produce `Datapath actions: drop` không.
- **Mask `tcp(src=50000/0, dst=80)`** — src mask là 0 (wildcard) vì policy chỉ care dest port 80; nếu thấy `src=50000/0xffff` → rule sai, match exact src port 50000 không generalize.

**Upstream:** man `ovs-dpctl(8)` §`dump-flows` + OVS source `datapath/flow.c`.

**Bối cảnh:** Before upgrade OVS, phải backup OVSDB. `ovsdb-client backup` dumps nhất quán, không yêu cầu daemon stop.

```shell
# Backup
$ sudo ovsdb-client backup unix:/var/run/openvswitch/db.sock > /tmp/conf.db.backup
$ ls -lh /tmp/conf.db.backup
-rw-r--r-- 1 root root 45K Apr 24 11:23 /tmp/conf.db.backup

# Nếu cần restore (scenario rollback upgrade)
$ sudo systemctl stop ovs-vswitchd
$ sudo systemctl stop ovsdb-server
$ sudo mv /etc/openvswitch/conf.db /etc/openvswitch/conf.db.old
$ sudo ovsdb-tool create /etc/openvswitch/conf.db \
    /usr/share/openvswitch/vswitch.ovsschema
$ sudo ovsdb-client restore unix:/var/run/openvswitch/db.sock \
    Open_vSwitch < /tmp/conf.db.backup

# Nhưng thường dễ hơn: copy conf.db sang host khác hoặc restore raw file
```

**Quy tắc production:**
- Backup trước mọi change schema (OVS upgrade 2.17 → 3.x có schema update).
- Backup không block; chạy cron daily vào `/var/backups/ovs/conf.db.YYYYMMDD`.
- Backup + metadata (OVS version, host, kernel) cho traceability.

**Upstream:** man `ovsdb-client(1)` và `ovsdb-tool(1)`. Xem thêm Phần 10.2 cho OVSDB backup/restore + compact deep dive.

## 9.4.6 Six-layer troubleshooting playbook, tích hợp

Khi có báo cáo "packet không đi qua OVS đúng cách", đi tuần tự sáu lớp từ dưới lên. Mỗi lớp có lệnh kiểm tra cụ thể, expected state, và branch quyết định.

| Lớp | Câu hỏi | Lệnh kiểm tra | Expected | Nếu fail |
|---|---|---|---|---|
| **L1 Physical** | Packet có đến NIC không? | `tcpdump -i eth0 -nn -c 10` | Thấy packet inbound | Cable, SFP, NIC driver; `ethtool eth0` verify link |
| **L2 Kernel driver** | Packet vào datapath? | `ovs-dpctl show` → counter `lookups:missed` tăng | missed tăng khi có traffic | NIC bound wrong driver; `driverctl list-overrides` cho DPDK |
| **L3 Megaflow cache** | Có hit megaflow nào? | `ovs-dpctl dump-flows -m` → tìm flow khớp với packet | Flow có packet:N > 0 | Lỡ cache liên tục → ở L4 có vấn đề |
| **L4 OpenFlow pipeline** | Flow nào match? Action gì? | `ovs-appctl ofproto/trace br-int "FLOW_SPEC"` | `Datapath actions: <ofport>` non-drop | Sai rule priority, thiếu table, action drop unintended |
| **L5 OVSDB config** | Port/VLAN/bridge đúng? | `ovs-vsctl show` + `ovs-vsctl list Interface <name>` | Port gắn đúng bridge, interface up | Port thiếu VLAN tag, interface type sai |
| **L6 Logical (OVN)** | LSP, Port_Binding đúng? | `ovn-sbctl show` + `ovn-sbctl list Port_Binding` + `ovn-trace <dp> "FLOW"` | Port_Binding có `chassis` non-null | ovn-controller chưa bind, logical flow compile sai |

**Quy tắc:** khi L1-L6 OK mà packet vẫn drop → vấn đề ngoài OVS (upstream switch, firewall, ARP, MTU, BGP). Không tiếp tục debug OVS.

### 9.4.6.1 Guided Exercise full 6-layer walkthrough

**Kịch bản:** user báo *"không ping được từ VM1 (tap-vm-web, 10.1.0.5) đến VM2 (tap-vm-db, 10.1.0.10), cả hai cùng Logical_Switch `ls-prod`, expected thông qua br-int"*.

**Setup (giả lập):**

```shell
$ sudo ovs-vsctl add-br br-int -- set Bridge br-int fail_mode=secure
$ sudo ovs-vsctl add-port br-int tap-vm-web -- \
    set Interface tap-vm-web type=internal
$ sudo ovs-vsctl add-port br-int tap-vm-db -- \
    set Interface tap-vm-db type=internal
$ sudo ip link set tap-vm-web up
$ sudo ip addr add 10.1.0.5/24 dev tap-vm-web
$ sudo ip link set tap-vm-db up
$ sudo ip addr add 10.1.0.10/24 dev tap-vm-db
# Add flow: allow ARP + IP between two ports, bug intentionally: drop ICMP
$ sudo ovs-ofctl -O OpenFlow15 add-flow br-int \
    "table=0,priority=200,in_port=tap-vm-web,dl_type=0x0806,actions=output:tap-vm-db"
$ sudo ovs-ofctl -O OpenFlow15 add-flow br-int \
    "table=0,priority=200,in_port=tap-vm-db,dl_type=0x0806,actions=output:tap-vm-web"
$ sudo ovs-ofctl -O OpenFlow15 add-flow br-int \
    "table=0,priority=100,in_port=tap-vm-web,dl_type=0x0800,nw_proto=6,\
     actions=output:tap-vm-db"
$ sudo ovs-ofctl -O OpenFlow15 add-flow br-int \
    "table=0,priority=100,in_port=tap-vm-db,dl_type=0x0800,nw_proto=6,\
     actions=output:tap-vm-web"
# Bug: flow priority 50 drop ICMP (giả đặt)
$ sudo ovs-ofctl -O OpenFlow15 add-flow br-int \
    "table=0,priority=150,in_port=tap-vm-web,dl_type=0x0800,nw_proto=1,actions=drop"
```

**Walkthrough playbook:**

**L1 — tcpdump physical (skip vì đây là internal-only setup; OVN Geneve setup thì tcpdump eth0):**

```shell
$ sudo tcpdump -i tap-vm-web -nn icmp -c 5
# Thấy ICMP outbound → L1 OK
```

**L2 — ovs-dpctl show:**

```shell
$ sudo ovs-dpctl show
system@ovs-system:
  lookups: hit:23, missed:7, lost:0
  flows: 5
  masks: hit:38 total:4 hit/pkt:1.65
  port 0: ovs-system (internal)
  port 1: br-int (internal)
  port 2: tap-vm-web (internal)
  port 3: tap-vm-db (internal)

# Counter missed > 0 → packet có vào OVS, kernel trigger upcall. OK.
```

**L3 — ovs-dpctl dump-flow:**

```shell
$ sudo ovs-dpctl dump-flows -m
# Sẽ thấy flow ICMP install với actions:drop — dấu hiệu packet drop ở L3/L4
```

**L4 — ofproto/trace ICMP:**

```shell
$ sudo ovs-appctl ofproto/trace br-int \
    "in_port=tap-vm-web,dl_src=aa:bb:cc:00:00:01,dl_dst=aa:bb:cc:00:00:02,\
     dl_type=0x0800,nw_src=10.1.0.5,nw_dst=10.1.0.10,nw_proto=1"
Flow: icmp,in_port=2,vlan_tci=0x0000,dl_src=aa:bb:cc:00:00:01,
      dl_dst=aa:bb:cc:00:00:02,nw_src=10.1.0.5,nw_dst=10.1.0.10,...

bridge("br-int")
----------------
 0. in_port=2,ip,nw_proto=1, priority 150
    drop

Final flow: unchanged
Datapath actions: drop
```

**Root cause** rõ ràng — Rule `table=0 priority=150 in_port=tap-vm-web,ip,nw_proto=1 drop`.

**L5 — L6 không cần vì L4 đã tìm ra root cause.**

**Fix:**

```shell
$ sudo ovs-ofctl -O OpenFlow15 del-flows br-int \
    "table=0,in_port=tap-vm-web,dl_type=0x0800,nw_proto=1" --strict
$ sudo ovs-ofctl -O OpenFlow15 add-flow br-int \
    "table=0,priority=100,in_port=tap-vm-web,dl_type=0x0800,nw_proto=1,\
     actions=output:tap-vm-db"
$ sudo ovs-ofctl -O OpenFlow15 add-flow br-int \
    "table=0,priority=100,in_port=tap-vm-db,dl_type=0x0800,nw_proto=1,\
     actions=output:tap-vm-web"

# Re-trace
$ sudo ovs-appctl ofproto/trace br-int "in_port=tap-vm-web,...,nw_proto=1"
Datapath actions: 3
```

**Teardown:**

```shell
$ sudo ovs-vsctl del-br br-int
```

## Điểm cốt lõi cần nhớ

Thứ nhất, năm công cụ CLI của OVS là **độc lập về vai trò**, không thay thế nhau. `ovs-vsctl` chỉ config OVSDB; `ovs-ofctl` chỉ OpenFlow; `ovs-appctl` chỉ introspection runtime; `ovs-dpctl` chỉ raw datapath; `ovsdb-client` chỉ raw OVSDB RPC. Mỗi công cụ cover một lớp kiến trúc, và workflow debug thực tế dùng song song nhiều công cụ.

Thứ hai, `ofproto/trace` là công cụ debug mạnh nhất — mô phỏng packet đi qua OpenFlow pipeline mà không gửi packet thật. Engineer OVS phải đọc thành thạo bốn khối output (Flow, bridge walk-through, Final flow, Datapath actions, Megaflow).

Thứ ba, six-layer troubleshooting playbook là phương pháp có hệ thống đi từ L1 physical lên L6 logical, loại trừ từng vùng. Khi sáu lớp agree mà packet vẫn drop → root cause ngoài OVS.

Thứ tư, atomicity của OVSDB transaction (ovs-vsctl) và OpenFlow bundle (ovs-ofctl `--bundle`) bảo vệ khỏi half-configured state. Luôn `diff-flows` trước `replace-flows` trong production; luôn backup `conf.db` trước upgrade.

Thứ năm, đọc output không chỉ là thấy lệnh chạy không báo lỗi. Mỗi field trong output có ý nghĩa vật lý (counter, timestamp, state enum) và có kịch bản bẻ gãy riêng (khi field X = Y thì symptom là Z). Kỹ sư thành thạo là người biết field X quan trọng, biết ngưỡng Y, biết Y vượt ngưỡng thì làm tiếp gì.

### 9.4.X.1 `ovs-vsctl add-br`

#### Khái niệm

`add-br` là subcommand tạo mới một bridge OVS. Hành vi của nó không đơn thuần là "tạo một row trong bảng `Bridge`": trong cùng một giao tác OVSDB, OVS tạo đồng thời một row `Bridge`, một row `Port` cùng tên với bridge (gọi là local port), và một row `Interface` kiểu `internal` cũng cùng tên. Sau khi giao tác commit, kernel datapath được nhắc tạo thực thể bridge, một netdev `internal` xuất hiện trong namespace host (xem được qua `ip link show br0`), và `ovs-vswitchd` bắt đầu mở luồng OpenFlow phục vụ bridge này.

#### Cú pháp + tham số chính

```
ovs-vsctl [--may-exist] add-br BRIDGE [PARENT VLAN] [-- set Bridge BRIDGE COLUMN=VALUE ...]
```

- `BRIDGE` là tên bridge, đồng thời là tên local port + interface internal sinh ra cùng (ràng buộc Linux: ≤ 15 ký tự).
- `[PARENT VLAN]` chỉ dùng khi tạo "fake bridge" trên một VLAN cụ thể của bridge cha; đa phần triển khai không dùng.
- `--may-exist` biến hành vi "fail nếu đã tồn tại" thành "no-op nếu đã đúng cấu hình"; bắt buộc dùng trong script idempotent.
- Mọi tham số `-- set Bridge ... COLUMN=VALUE` chạy chung giao tác với `add-br`; ví dụ điển hình: `-- set Bridge br0 fail_mode=secure protocols=OpenFlow13,OpenFlow15 datapath_type=system`.

#### Ngữ nghĩa giao tác

Mỗi lần gọi `ovs-vsctl` là **một** giao tác OVSDB tới `ovsdb-server` qua socket `/var/run/openvswitch/db.sock`. Nếu bất kỳ thao tác trong giao tác fail (vi phạm constraint, name collision, datapath_type không hợp lệ), toàn bộ giao tác rollback: row `Bridge` không xuất hiện, `Port`/`Interface` cũng không tạo. Đây là nền tảng cho idempotency: caller không cần dọn dẹp một phần.

#### Mẫu output thực tế

```shell
$ sudo ovs-vsctl add-br br-demo
$ sudo ovs-vsctl list Bridge br-demo
_uuid               : 9f3a1c12-77b2-4a01-9d8c-1c2d5e8f7a90
controller          : []
datapath_id         : "0000aabbccddeeff"
datapath_type       : system
fail_mode           : []
name                : br-demo
ports               : [b71d...]
protocols           : []
$ ip link show br-demo
12: br-demo: <BROADCAST,MULTICAST> mtu 1500 qdisc noop state DOWN ...
```

Nhận xét: `controller` rỗng (chưa gắn), `fail_mode` rỗng (mặc định `standalone` cho bridge không có controller, xem §9.4.X.2), `protocols` rỗng (mặc định OVS quyết định, thường OF1.0+1.1+1.2+1.3+1.4+1.5).

#### Lỗi thường gặp + chẩn đoán

| Triệu chứng | Nguyên nhân khả dĩ | Cách kiểm chứng |
|-------------|---------------------|-----------------|
| `ovs-vsctl: cannot create a bridge named X because a bridge named X already exists` | Quên `--may-exist` trong script | Dùng `ovs-vsctl --may-exist add-br X` |
| `ip link show X` không thấy interface mới | `ovs-vswitchd` chết hoặc mất kết nối ovsdb | Kiểm tra `systemctl status openvswitch-switch`, `ovs-appctl bridge/dump-flows` |
| Bridge tạo nhưng datapath không hoạt động | `datapath_type=netdev` đặt mà host thiếu DPDK | Xem `ovs-vswitchd.log`, fallback `set Bridge X datapath_type=system` |
| `name X is too long` | Tên > 15 ký tự (giới hạn Linux netdev) | Đổi tên ngắn hơn |

#### Lab thực hành

Trên Ubuntu 22.04 + OVS 2.17.9:

```shell
sudo ovs-vsctl --may-exist add-br br-lab \
  -- set Bridge br-lab fail_mode=secure protocols=OpenFlow13 \
  -- set Bridge br-lab other_config:hwaddr=02:11:22:33:44:55
sudo ip link set br-lab up
sudo ovs-vsctl list Bridge br-lab | grep -E "fail_mode|protocols"
```

Tiêu chí pass: bridge tồn tại, `fail_mode=secure`, `protocols=[OpenFlow13]`, link `state UP`, `ovs-ofctl -O OpenFlow13 dump-flows br-lab` chạy không lỗi.

#### Source code tham chiếu

Hàm `cmd_add_br()` trong `utilities/ovs-vsctl.c` (OVS branch-2.17). Hàm này build operation `insert` trên bảng `Bridge`, cộng `insert` trên `Port` + `Interface` (kiểu internal), sau đó gọi `ovsdb_idl_txn_commit_block()`. Schema constraint trong `vswitchd/vswitch.ovsschema` ép `Bridge.name` UNIQUE và `Bridge.ports` chứa ít nhất local port cùng tên.

#### Khác biệt giữa các phiên bản

- OVS 1.x (2010-2013): cú pháp cơ bản, chưa có `--may-exist`.
- OVS 2.0 (2014): hỗ trợ `--may-exist`/`--if-exists`, OVSDB transaction batching.
- OVS 2.4 (2015): cho phép gán cột bằng `-- set` chung giao tác.
- OVS 2.7+ (2017): hỗ trợ `datapath_type=netdev` cho userspace DPDK/AF_XDP (nằm ngoài phạm vi mục này).

#### So sánh với hệ khác

Linux bridge thuần dùng `ip link add br0 type bridge` rồi `ip link set br0 up`; thiếu khái niệm `fail_mode`, `protocols`, `controller`. Cisco IOS có `bridge-group` nhưng nằm ở interface mode chứ không có database tập trung. OVS phân tách rõ control plane (OVSDB) và data plane (kernel datapath), `add-br` sửa control plane, kernel hành động theo sau qua netlink.

### 9.4.X.2 `ovs-vsctl set-fail-mode`

#### Khái niệm

`set-fail-mode` cấu hình hành vi của bridge khi mất kết nối với OpenFlow controller. Hai giá trị hợp lệ: `standalone` (bridge tự xử lý packet như một L2 learning switch thông thường) và `secure` (bridge KHÔNG tự thêm rule nào, chỉ xử lý packet theo flow đã cài; nếu không có flow khớp, packet bị drop). Cột này nằm trong bảng `Bridge`, tên column là `fail_mode`.

#### Cú pháp + tham số

```
ovs-vsctl set-fail-mode BRIDGE MODE
ovs-vsctl get-fail-mode BRIDGE
ovs-vsctl del-fail-mode BRIDGE   # reset về [] (mặc định)
```

`MODE` chỉ chấp nhận hai giá trị `standalone` hoặc `secure`. Mặc định khi cột rỗng (`[]`): nếu bridge có controller, hành vi như `secure`; nếu không có controller, hành vi như `standalone`. Vì thế đa phần triển khai production set explicit `secure` để tránh phụ thuộc heuristic này.

#### Vai trò trong kiến trúc

Bridge OVS có thể chạy ở hai chế độ: SDN-controlled (controller push flow) hoặc autonomous (tự learning). `fail_mode` quyết định ứng xử khi controller mất kết nối: `standalone` ưu tiên availability (bridge tự làm switch), `secure` ưu tiên correctness (không forward packet ngoài rule controller chỉ định, tránh leak traffic ngoài chính sách). Triển khai OVN, OpenStack Neutron, Kubernetes networking đều dùng `secure` để đảm bảo SDN policy không bị bypass khi controller restart hoặc network partition.

#### Mẫu output thực tế

```shell
$ sudo ovs-vsctl set-fail-mode br-int secure
$ sudo ovs-vsctl get-fail-mode br-int
secure
$ sudo ovs-vsctl get Bridge br-int fail_mode
secure
```

Khi fail-mode = `standalone` và controller mất kết nối, bridge cài flow mặc định `actions=NORMAL` (L2 learning). Có thể quan sát bằng `ovs-ofctl dump-flows br-int` thấy flow `priority=0 actions=NORMAL` xuất hiện.

#### Scenario sự cố thực tế

Một host chạy OVN-controlled `br-int` set `fail_mode=standalone`. Khi `ovn-controller` bị OOM-killed, bridge fall back về L2 learning, `br-int` flood packet qua mọi port. Tenant traffic của VM A leak sang VM B vì cả hai chia sẻ MAC table mới học. Mitigation: chuyển toàn bộ host sang `fail_mode=secure`, xác minh bằng `ovs-vsctl --columns=name,fail_mode list Bridge`.

#### Lab thực hành

```shell
sudo ovs-vsctl --may-exist add-br br-fm
sudo ovs-vsctl set-fail-mode br-fm secure
# Không có controller, không có flow → mọi packet drop
sudo ovs-ofctl dump-flows br-fm   # rỗng
# So sánh:
sudo ovs-vsctl set-fail-mode br-fm standalone
sudo ovs-ofctl dump-flows br-fm
#  cookie=0x0, priority=0 actions=NORMAL  ← OVS auto-add
```

Tiêu chí pass: học viên giải thích được khác biệt drop vs NORMAL forward khi không có controller, và đối chiếu với kết nối controller (xem §9.4.X.3).

#### Lỗi thường gặp + chẩn đoán

| Triệu chứng | Nguyên nhân khả dĩ | Cách kiểm chứng |
|-------------|---------------------|-----------------|
| Bridge "tự dưng" forward dù không có rule | `fail_mode=standalone` + controller down | `ovs-vsctl get-fail-mode <br>` kiểm tra |
| Bridge drop tất cả packet sau OVS restart | `fail_mode=secure` + controller chưa kết nối lại | Xem `ovs-vsctl get-controller br-int`, kiểm tra `is_connected` qua `list Controller` |
| `set-fail-mode br-int invalid_value` báo lỗi | Chỉ chấp nhận `standalone`/`secure` | Đọc man `ovs-vsctl(8)` section `Bridge Configuration Commands` |

#### Source code tham chiếu

Hàm `cmd_set_fail_mode()` + `cmd_get_fail_mode()` trong `utilities/ovs-vsctl.c`. Hàm validate qua `string_is_in()` chấp nhận cố định `{"standalone", "secure"}`. Logic forwarding khi `standalone` nằm trong `ofproto/ofproto-dpif.c` hàm `add_internal_flows()` (cài flow `priority=0 actions=NORMAL`).

#### Khác biệt giữa các phiên bản

- OVS 1.x: hai mode đầy đủ.
- OVS 2.0: cải thiện hành vi reconnect controller, không thay đổi semantics.
- OVS 2.10+: hỗ trợ "in-band control" cho phép controller chạy qua chính bridge nó quản lý (kết hợp với `secure` cần đặc biệt cẩn trọng). Mặc định in-band tắt từ 2.10.

#### So sánh với hệ khác

OpenFlow spec định nghĩa `fail_secure` và `fail_standalone` (OF 1.0 §6.4); `ovs-vsctl set-fail-mode` map trực tiếp 1-1. NETCONF/OpenConfig không có concept này (giả định controller luôn online). Cisco SDN (ACI) tự quản lý fail-mode ở fabric level, không expose ra CLI per-bridge.

### 9.4.X.3 `ovs-vsctl set-controller`

#### Khái niệm

`set-controller` gắn bridge vào một hoặc nhiều OpenFlow controller. Mỗi target là một URI: `tcp:HOST[:PORT]`, `ssl:HOST[:PORT]`, `unix:FILE`, `pssl:PORT[:HOST]` (passive, OVS lắng nghe), `ptcp:PORT[:HOST]`. OVS mở kết nối, gửi `OFPT_HELLO`, đàm phán version OpenFlow, sau đó nhận flow_mod / barrier_request / packet_in từ controller. Một bridge có thể có > 1 controller hoạt động song song (multi-controller mode); OVS forward packet_in cho tất cả, nhận flow_mod từ bất kỳ controller nào.

#### Cú pháp + tham số

```
ovs-vsctl set-controller BRIDGE TARGET [TARGET ...]
ovs-vsctl get-controller BRIDGE
ovs-vsctl del-controller BRIDGE
```

Mỗi target trở thành một row trong bảng `Controller`, được tham chiếu từ `Bridge.controller` (kiểu set of UUID). Mặc định OVS dùng port TCP 6653 (chính thức IANA-assigned cho OpenFlow); historical dùng 6633 (legacy). Giao thức `ssl:` yêu cầu cấu hình private key + certificate qua `ovs-vsctl set-ssl <pk> <cert> <ca>`.

#### Ngữ nghĩa giao tác

Mỗi `set-controller` REPLACE toàn bộ list controller hiện tại bằng list mới (không append). Để giữ controller cũ + thêm mới, phải truyền lại danh sách đầy đủ. Đây là nguồn lỗi phổ biến trong script tự động.

#### Mẫu output thực tế

```shell
$ sudo ovs-vsctl set-controller br-int tcp:127.0.0.1:6653
$ sudo ovs-vsctl get-controller br-int
tcp:127.0.0.1:6653
$ sudo ovs-vsctl --columns=target,is_connected,role list Controller
target              : "tcp:127.0.0.1:6653"
is_connected        : true
role                : master
```

Cột `is_connected` cho biết kết nối TCP/SSL active hay không; `role` (`master`/`slave`/`equal`) thuộc OF 1.2+ multi-controller. Khi `is_connected=false`, OVS retry với exponential backoff (mặc định bắt đầu 1 giây, max 8 giây, configurable qua `max_backoff`/`inactivity_probe`).

#### Scenario sự cố thực tế

Sau khi restart `ovn-controller`, một số host hiện `is_connected=false` kéo dài > 30 giây. Nguyên nhân: `ovn-controller` viết lại Controller row qua OVSDB, nhưng `ovs-vswitchd` chưa kịp đóng kết nối cũ, vướng socket TIME_WAIT. Mitigation: `ovs-appctl ofproto/list` + `ovs-appctl ofproto/reconnect br-int` ép reconnect ngay, không chờ backoff.

#### Lab thực hành

Dùng Mininet + Ryu controller:

```shell
# Terminal 1: chạy controller
ryu-manager ryu.app.simple_switch_13  # listen tcp 0.0.0.0:6653

# Terminal 2: tạo bridge + gắn controller
sudo ovs-vsctl --may-exist add-br br-ryu \
  -- set Bridge br-ryu protocols=OpenFlow13 \
  -- set-controller br-ryu tcp:127.0.0.1:6653 \
  -- set-fail-mode br-ryu secure
sudo ovs-vsctl --columns=is_connected list Controller
# is_connected: true   → OK
```

Tiêu chí pass: `is_connected=true` trong < 5 giây, `ovs-ofctl -O OpenFlow13 dump-flows br-ryu` cho thấy flow Ryu đã push.

#### Lỗi thường gặp + chẩn đoán

| Triệu chứng | Nguyên nhân khả dĩ | Cách kiểm chứng |
|-------------|---------------------|-----------------|
| `is_connected=false` mãi | Controller chưa listen, firewall, version mismatch | `ss -tnlp \| grep 6653`, `ovs-vswitchd.log` tìm `OFPT_HELLO mismatch` |
| Set controller mới nhưng controller cũ vẫn còn | `set-controller` REPLACE, nhưng cú pháp truyền nhầm 2 row | Dùng `--columns=name,controller list Bridge` để xem |
| SSL kết nối fail | Thiếu cert / CA chain sai | `ovs-vsctl get-ssl`, đối chiếu certificate với controller |
| Restart `ovs-vswitchd` mất controller | Bridge chưa persist row Controller | OVSDB persist mặc định; nếu mất → check disk + `ovsdb-server` log |

#### Source code tham chiếu

Hàm `cmd_set_controller()` + `cmd_del_controller()` trong `utilities/ovs-vsctl.c`. Logic kết nối nằm trong `ofproto/connmgr.c` (hàm `ofconn_run()` xử lý reconnect, `ofconn_set_role()` xử lý OF 1.2 role). Mặc định inactivity_probe 5 giây, max_backoff 8 giây (xem `vswitch.ovsschema` Controller table).

#### Khác biệt giữa các phiên bản

- OVS 1.x: chỉ hỗ trợ một controller per bridge, OF 1.0.
- OVS 2.0 (2014): multi-controller, OF 1.3 GA.
- OVS 2.3 (2014): OF 1.4 + 1.5 experimental.
- OVS 2.10+ (2018): OF 1.5 ổn định, đàm phán bundle support qua `OFPT_BUNDLE_FEATURES_REQUEST`.

#### So sánh với hệ khác

NETCONF agent (ConfD, sysrepo) cũng có concept `<connection>` đến SDN controller, semantics tương tự nhưng dùng XML schema thay OVSDB. P4Runtime gRPC mở persistent stream giống OF channel. Cisco OpenFlow mode trên Catalyst expose `controller ipv4 ... port 6653` nhưng không phân biệt active/passive như OVS.

### 9.4.X.4 `ovs-vsctl add-port`

#### Khái niệm

`add-port` gắn một interface (vật lý hoặc tunnel/internal/patch) vào bridge. Trong cùng giao tác, OVS tạo row `Port` và row `Interface` (1 port = 1 hoặc nhiều interface khi bond), sửa `Bridge.ports` add UUID port mới. Sau commit, kernel datapath nhận netlink để mở/gắn netdev tương ứng; `ovs-vswitchd` cấp ofport (số nguyên duy nhất per bridge) cho interface, từ giờ controller có thể trỏ flow `actions=output:<ofport>`.

#### Cú pháp + tham số chính

```
ovs-vsctl [--may-exist] add-port BRIDGE PORT [COLUMN[:KEY]=VALUE ...]
```

Các pattern thường gặp:

- Access port VLAN 10: `add-port br0 p1 tag=10`
- Trunk VLAN 10/20/30: `add-port br0 p1 trunks=10,20,30`
- Tunnel Geneve: `add-port br0 gnv0 -- set Interface gnv0 type=geneve options:remote_ip=10.0.0.2 options:key=flow`
- Port internal có IP: `add-port br0 vlan10 -- set Interface vlan10 type=internal`
- Patch port nối hai bridge: `add-port br0 patch-to-br1 -- set Interface patch-to-br1 type=patch options:peer=patch-to-br0`

#### Ngữ nghĩa giao tác

Cũng như `add-br`, mọi thay đổi trong cùng `ovs-vsctl` invocation chạy chung một giao tác OVSDB. Nếu `set Interface ... type=geneve options:remote_ip=...` fail (ví dụ remote_ip sai format), toàn bộ rollback: `Port` không xuất hiện. Sử dụng `--may-exist` để script idempotent: nếu port cùng tên đã tồn tại trên cùng bridge với CÙNG interface, lệnh là no-op; nếu port đã tồn tại trên bridge khác, lệnh fail.

#### Mẫu output thực tế

```shell
$ sudo ovs-vsctl --may-exist add-port br-int veth-vm1 \
    -- set Interface veth-vm1 external_ids:iface-id=vm1-port \
    -- set Port veth-vm1 tag=100
$ sudo ovs-vsctl --columns=name,tag,trunks list Port veth-vm1
name        : veth-vm1
tag         : 100
trunks      : []
$ sudo ovs-ofctl show br-int | grep veth-vm1
 7(veth-vm1): addr:fa:16:3e:aa:bb:cc
     config:     0
     state:      0
     speed: 10000 Mbps now, 10000 Mbps max
```

Cột thứ tự `7(veth-vm1)`: `7` là ofport (assigned khi commit), `veth-vm1` là tên Linux netdev. Controller dùng `7` trong action `output:7`.

#### Vai trò trong kiến trúc

Đây là cầu nối duy nhất giữa thế giới Linux netdev (vật lý + veth + tunnel device) và thế giới OVS bridge. Mọi packet ra/vào OVS phải đi qua một interface đã add-port. OVN dùng `external_ids:iface-id` để tag interface với Logical_Switch_Port (xem Phần 13), giúp `ovn-controller` ánh xạ Port_Binding xuống ofport. OpenStack Neutron OVS agent dùng pattern tương tự nhưng tag bằng `external_ids:attached-mac` + `iface-id`.

#### Scenario sự cố thực tế

VM mới boot nhưng không nhận được DHCP. Kiểm tra `ovs-vsctl show` thấy `tap-vm1` đã add-port br-int. Kiểm tra `ovs-ofctl show br-int` thấy ofport assigned. Nguyên nhân thực: thiếu `external_ids:iface-id`, `ovn-controller` không bind được Logical_Switch_Port → flow ingress không cài. Fix: `ovs-vsctl set Interface tap-vm1 external_ids:iface-id=<port-uuid>`.

#### Lỗi thường gặp + chẩn đoán

| Triệu chứng | Nguyên nhân khả dĩ | Cách kiểm chứng |
|-------------|---------------------|-----------------|
| `port already exists on bridge X` | Interface đã thuộc bridge khác | `ovs-vsctl iface-to-br <name>` |
| `add-port` thành công nhưng `ofport=-1` | Kernel netdev không tồn tại (interface chưa tạo) | `ip link show <name>`, `ovs-vswitchd.log` tìm `could not open network device` |
| Tunnel up nhưng không có traffic | `options:remote_ip` đúng nhưng `options:key` mismatch | `ovs-vsctl get Interface <name> options`, đối chiếu hai phía |
| Quên `--may-exist` trong loop script | Lần thứ hai chạy lệnh, fail giữa chừng | Thêm `--may-exist` cho mọi `add-port` trong reconcile loop |

#### Lab thực hành

```shell
sudo ovs-vsctl --may-exist add-br br-lab2
sudo ip link add veth-a type veth peer name veth-b
sudo ip link set veth-a up; sudo ip link set veth-b up
sudo ovs-vsctl --may-exist add-port br-lab2 veth-a tag=10
sudo ovs-vsctl --may-exist add-port br-lab2 veth-b tag=20
sudo ovs-ofctl show br-lab2
```

Tiêu chí pass: hai port hiện ofport khác nhau, `tag` đúng 10 và 20, không cảnh báo trong `ovs-vswitchd.log`.

#### Source code tham chiếu

Hàm `cmd_add_port()` trong `utilities/ovs-vsctl.c`. Hàm này build operation `insert` trên `Port` + `Interface`, mutate `Bridge.ports`. Logic gán ofport nằm trong `ofproto/ofproto.c` hàm `ofport_install()`; mặc định OVS chọn ofport tự động (1, 2, 3, ...), có thể ép qua `set Interface <name> ofport_request=<n>`.

#### Khác biệt giữa các phiên bản

- OVS 1.x: chỉ kernel datapath, type internal/gre/vxlan.
- OVS 2.0 (2014): thêm Geneve.
- OVS 2.4: bond LACP cải thiện, thêm `lacp-rate`.
- OVS 2.7+: hỗ trợ DPDK port type (`type=dpdk`, `type=dpdkvhostuser`) — ngoài phạm vi mục này.

#### So sánh với hệ khác

Linux bridge dùng `ip link set veth-a master br0` (kernel netlink trực tiếp); thiếu khái niệm Port/Interface tách biệt, không có ofport, không có external_ids. OVS phân tách Port (logical) và Interface (physical/tunnel) giúp model bond (1 Port → nhiều Interface), điều Linux bridge không làm được nguyên bản.

### 9.4.X.5 `ovs-vsctl set` / `get` trên bảng `Bridge`

#### Khái niệm

`set TABLE RECORD COLUMN=VALUE` và `get TABLE RECORD COLUMN` là hai subcommand database core, làm việc trực tiếp với row OVSDB bất kỳ. Riêng cho bảng `Bridge`, hai lệnh này là cách duy nhất để cấu hình các cột không có wrapper convenience: `protocols`, `datapath_type`, `mcast_snooping_enable`, `rstp_enable`, `external_ids`, `other_config`. Cú pháp tổng quát:

```
ovs-vsctl set Bridge BRIDGE COLUMN=VALUE [COLUMN=VALUE ...]
ovs-vsctl get Bridge BRIDGE COLUMN [COLUMN ...]
ovs-vsctl --columns=COL1,COL2 list Bridge BRIDGE
```

#### Các cột Bridge thường set

| Cột | Kiểu | Ý nghĩa |
|-----|------|---------|
| `protocols` | set of string | Danh sách OF version bridge chấp nhận: `OpenFlow10..15`. Nếu rỗng, OVS tự thương lượng |
| `datapath_type` | string | `system` (kernel datapath, mặc định) hoặc `netdev` (userspace DPDK/AF_XDP) |
| `mcast_snooping_enable` | boolean | Bật IGMP/MLD snooping cho bridge |
| `rstp_enable` | boolean | Bật Rapid Spanning Tree |
| `external_ids` | map string-string | Key tự do cho integration (OVN, Neutron) |
| `other_config` | map string-string | Tunable: `mac-aging-time`, `mac-table-size`, `hwaddr`, `disable-in-band` |
| `flood_vlans` | set of integer | VLAN bị flood unconditionally |
| `flow_tables` | map integer-uuid | Tham chiếu Flow_Table per OF table-id |

#### Cú pháp set map/set vs scalar

Map column có syntax đặc biệt `COLUMN:KEY=VALUE`:

```shell
sudo ovs-vsctl set Bridge br-int other_config:mac-aging-time=600
sudo ovs-vsctl set Bridge br-int external_ids:bridge-id=br-int-uuid
```

Để xóa một key: `ovs-vsctl remove Bridge br-int other_config mac-aging-time`. Để clear toàn bộ: `ovs-vsctl clear Bridge br-int other_config`.

Set scalar: `ovs-vsctl set Bridge br-int protocols=OpenFlow13,OpenFlow15` (giá trị multi-element ngăn cách bằng dấu phẩy).

#### Mẫu output thực tế

```shell
$ sudo ovs-vsctl set Bridge br-int protocols=OpenFlow13,OpenFlow15
$ sudo ovs-vsctl get Bridge br-int protocols
[OpenFlow13, OpenFlow15]
$ sudo ovs-vsctl --columns=name,protocols,datapath_type,mcast_snooping_enable list Bridge br-int
name                : br-int
protocols           : [OpenFlow13, OpenFlow15]
datapath_type       : system
mcast_snooping_enable: false
```

#### Ngữ nghĩa giao tác

`set` atomic per giao tác: nếu set 3 cột mà 1 cột invalid, cả 3 rollback. `get` không thay đổi DB, chỉ đọc snapshot. Nhiều `set` chuỗi qua `--` chạy chung giao tác:

```shell
sudo ovs-vsctl \
  -- set Bridge br-int protocols=OpenFlow13 \
  -- set Bridge br-int other_config:mac-aging-time=600 \
  -- set Bridge br-int other_config:hwaddr=02:00:00:00:00:01
```

Một transaction. Một network round-trip tới `ovsdb-server`. Tốc độ tốt hơn 3 lệnh riêng và đảm bảo tất cả cùng commit hoặc cùng rollback.

#### Scenario sự cố thực tế

Sau khi nâng cấp OVS từ 2.13 lên 2.17, một bridge đột nhiên chấp nhận flow OF 1.5 trong khi controller chỉ nói OF 1.3, gây flow_mod error. Nguyên nhân: cột `protocols` rỗng, OVS 2.17 mặc định advertise OF 1.0 đến 1.5, controller cũ chỉ hỗ trợ 1.3 nên reply error. Fix dứt điểm: `ovs-vsctl set Bridge br-int protocols=OpenFlow13` ép cố định.

#### Lab thực hành

```shell
sudo ovs-vsctl --may-exist add-br br-mtl
sudo ovs-vsctl set Bridge br-mtl \
  protocols=OpenFlow13,OpenFlow15 \
  other_config:mac-aging-time=120 \
  other_config:mac-table-size=4096 \
  external_ids:owner=team-network external_ids:env=lab
sudo ovs-vsctl --columns=name,protocols,other_config,external_ids list Bridge br-mtl
# expect: protocols=[OpenFlow13, OpenFlow15], other_config={mac-aging-time=120, mac-table-size=4096}
```

Tiêu chí pass: tất cả cột có giá trị mong đợi, một transaction duy nhất.

#### Lỗi thường gặp + chẩn đoán

| Triệu chứng | Nguyên nhân khả dĩ | Cách kiểm chứng |
|-------------|---------------------|-----------------|
| `unknown column ...` | Tên cột sai chính tả hoặc OVS version cũ chưa có | `ovsdb-client get-schema unix:/var/run/openvswitch/db.sock Open_vSwitch \| jq '.tables.Bridge.columns \| keys'` |
| `set` map column bằng `=` (không có `:`) | Sai cú pháp; map cần `COL:KEY=VALUE` | Xem man `ovs-vsctl(8)` section `Database Commands` |
| Quên transaction → 3 lệnh riêng, một fail giữa chừng | Để lại state nửa apply | Gộp vào một lệnh với `--` connector |
| `get` trả về `[]` thay vì giá trị mong đợi | Cột là set/map rỗng, không phải scalar | Dùng `--columns=COL list Bridge` để xem rõ kiểu |

#### Source code tham chiếu

Hàm `cmd_get()` + `cmd_set()` trong `utilities/ovs-vsctl.c`. Hai hàm dùng chung helper `parse_column_key_value()` để nhận diện scalar/set/map. Schema validate qua `ovsdb_idl_txn_verify()`. Schema gốc của bảng `Bridge` ở `vswitchd/vswitch.ovsschema` (~80 cột ở OVS 2.17).

#### Khác biệt giữa các phiên bản

- OVS 1.x: cột `Bridge.protocols` chưa có (default OF 1.0 only).
- OVS 2.0: `protocols` thêm vào schema, hỗ trợ multi-version.
- OVS 2.3: thêm `mcast_snooping_enable`.
- OVS 2.10: thêm `rstp_enable` cải thiện, `flow_tables` map.
- OVS 2.17: ổn định, không có cột mới nổi bật cho Bridge.

#### So sánh với hệ khác

NETCONF thao tác qua YANG path `/openconfig-network-instance:network-instances/network-instance[name='br0']/config`, semantics gần giống `set Bridge br0 ...` nhưng dùng XML/RPC thay socket OVSDB. Cisco IOS không có database tập trung, mỗi thay đổi là một CLI command vào running-config; thiếu transactional batch như OVS.

## 9.4.Y Mở rộng chuyên sâu nhóm subcommand `ovs-vsctl` quan sát + chỉnh row (mở rộng 2026-04)

> **Phạm vi mục này.** Năm subcommand `ovs-vsctl` cấp trung dùng để quan sát trạng thái OVSDB và chỉnh row Port: `show`, `list Bridge`, `list Port`, `list Interface`, và `set Port`. Khác nhóm §9.4.X (tạo/cấu hình bridge), nhóm này là công cụ "đọc" hằng ngày của on-call, plus một lệnh "ghi" trực tiếp lên row Port mà sysadmin Neutron/OVN dùng để gán VLAN, trunk, qos. Phần §9.4.1.3 đã giới thiệu `show` ở mức tổng quan; mục này đi sâu vào ngữ nghĩa OVSDB, cú pháp filter, output formatting, scenario sự cố, source code anchor, và so sánh với các tool khác. Nguồn upstream: `utilities/ovs-vsctl.c` (`cmd_show_tables[]` line ~1525, `pre_get_info()` line ~1273), `lib/db-ctl-base.c` (`cmd_show` line ~2195, `cmd_list` line ~1370, `cmd_set` line ~1240, `cmd_show_row` line ~2126), man page `ovs-vsctl(8)` §`DATABASE COMMANDS` + §`SUMMARIZING DATABASE CONTENTS`, schema `vswitchd/vswitch.ovsschema`. Baseline: OVS branch-2.17 (2.17.9 trên Ubuntu Jammy).

### 9.4.Y.1 `ovs-vsctl show`

#### Khái niệm

`show` không phải là một row-level dump; nó là pretty-printer hierarchical dựng từ bảng `Open_vSwitch` (root) đi xuống `Bridge`, `Port`, `Interface`, `Controller`, `Manager`. Output là cây thụt lề bốn ký tự mỗi level, in tên record + các cột "interesting" được khai báo trong `cmd_show_tables[]` (`utilities/ovs-vsctl.c` line ~1525), bỏ qua cột rỗng/default. Đây là lệnh đầu tiên kỹ sư on-call gõ sau SSH vào host vì nó trả lời câu hỏi: bridge nào tồn tại, port nào gắn vào bridge nào, interface nào type gì. Tốc độ rất nhanh: một query `ovsdb_idl_run()` đơn lẻ kéo schema xuống cache rồi in.

#### Cú pháp + tham số

```
ovs-vsctl [--timeout=SEC] [--db=SOCKET] show
```

`show` không nhận tham số positional; mọi flag đều global (timeout, db socket, format). Khác `list`, `show` không hỗ trợ `--columns`/`--bare`/`--no-headings` vì format tree đã cố định trong code (`cmd_show_row()` `lib/db-ctl-base.c` line ~2126). Kỹ sư cần custom output → dùng `list` thay vì grep `show`.

#### Cơ chế hoạt động

`cmd_show()` (line ~2195) lấy bảng đầu tiên trong `cmd_show_tables[]` (luôn là `Open_vSwitch`), iterate qua `ovsdb_idl_first_row()` + `ovsdb_idl_next_row()`. Với mỗi row gọi `cmd_show_row()` đệ quy: in tên cột "name" (hoặc UUID nếu bảng không có name), thụt lề `level*4` space, sau đó duyệt các cột tham chiếu UUID (vd `Bridge.ports` → đệ quy in mỗi `Port`). `sset shown` tracking ngăn vòng lặp circular reference. Cột scalar (string/int/bool) chỉ in nếu giá trị KHÁC default; điều này lý giải vì sao output thay đổi khi config thay đổi (vd `fail_mode: secure` chỉ hiện khi đã set, mặc định ẩn).

#### Mẫu output thực tế

```shell
$ sudo ovs-vsctl show

    Bridge br-int
        fail_mode: secure
        datapath_type: system
        Port br-int
            Interface br-int
                type: internal
        Port ovn-huyvl-0
            Interface ovn-huyvl-0
                type: geneve
                options: {csum="true", key=flow, remote_ip="10.200.0.12"}
        Port tap-vm-db
            tag: 100
            Interface tap-vm-db
    ovs_version: "2.17.9"
```

UUID dòng đầu là record `Open_vSwitch` (chỉ có 1 row, unique per host). `ovs_version` ở cuối lấy từ cột `Open_vSwitch.ovs_version` được vswitchd update mỗi lần khởi động.

#### Khi xảy ra sự cố cần nhớ lệnh này

Bốn tình huống điển hình. Một, nghi ngờ bridge biến mất sau OVS restart: `show` trả output rỗng hoặc thiếu bridge → kiểm tra `ovs-vswitchd.log` + ổ đĩa OVSDB chưa hỏng. Hai, port tunnel OVN không xuất hiện sau khi LSP đã bind: `show` thấy br-int nhưng thiếu `ovn-<chassis>-<idx>` → `ovn-controller` chưa pick up binding, đối chiếu `ovn-sbctl list Chassis`. Ba, packet drop suspected do VLAN tag sai: `show` cho phép soi nhanh `Port tap-vm-db tag: 100` xác nhận tag đúng. Bốn, `show` treo > 1 giây: `ovsdb-server` overload, dùng `--timeout=3` ép fail nhanh để khỏi block automation.

#### Lab thực hành

```shell
sudo ovs-vsctl --may-exist add-br br-show-lab
sudo ovs-vsctl add-port br-show-lab veth0
sudo ovs-vsctl set Port veth0 tag=42
sudo ovs-vsctl set Bridge br-show-lab fail_mode=secure
sudo ovs-vsctl show | sed -n '/br-show-lab/,/^$/p'
```

Tiêu chí pass: output hiển thị `Bridge br-show-lab`, `fail_mode: secure`, `Port veth0`, `tag: 42`, `Interface veth0`. Học viên chạy thêm `ovs-vsctl del-port br-show-lab veth0` rồi `show` để thấy port biến mất ngay lập tức (transaction OVSDB commit → cache invalidate → render lại).

#### Source code tham chiếu

`lib/db-ctl-base.c` `cmd_show()` line ~2195, `cmd_show_row()` line ~2126. Định nghĩa bảng "interesting" trong `utilities/ovs-vsctl.c` `cmd_show_tables[]` line ~1525 (open_vswitch → bridge → port → interface → controller → manager). Cache populate qua `vsctl_context_populate_cache()` line ~1329 dùng `shash` để O(1) lookup name→UUID khi user truyền record-id là tên thay vì UUID.

#### Khác biệt giữa các phiên bản

- OVS 1.x: format gần giống nhưng thiếu `Manager` block (chưa có manager remote table).
- OVS 2.0: thêm Manager block; thêm `ovs_version` line cuối cùng.
- OVS 2.4: in thêm `datapath_type` cho Bridge (trước đó luôn implicit kernel).
- OVS 2.10: thêm bond/lacp summary trong Port block khi cấu hình bond.

#### So sánh với hệ khác

`brctl show` của Linux bridge in topology tương tự nhưng chỉ một level (bridge → ports), không có khái niệm `type=geneve`/`internal`/`patch`, không có VLAN tag/trunk inline. `ip -d link show type bridge` chi tiết hơn `brctl` nhưng vẫn không tree. NETCONF GET trên YANG `interfaces-state` cho output cây tương tự nhưng dạng XML. Cisco IOS `show interfaces switchport` tương đương ở mức semantic VLAN/trunk nhưng không expose schema database; OVS `show` ưu thế ở chỗ output bám sát structure OVSDB, debug-friendly cho automation.

### 9.4.Y.2 `ovs-vsctl list Bridge`

#### Khái niệm

`list` là subcommand database-level dump tất cả cột của một row. Dạng `list Bridge [BRIDGE_NAME ...]` in raw row của bảng `Bridge`: nếu không truyền tên, in mọi bridge; nếu truyền, in các row được chọn theo thứ tự. Khác `show`, `list Bridge` không follow reference (không in port/interface bên dưới), thay vào đó nó in UUID của port set trong cột `ports`. Mục tiêu là cho automation/debug đọc đầy đủ schema-level, không phải human-readable topology.

#### Cú pháp + tham số

```
ovs-vsctl [--bare] [--no-headings] [--columns=COL[,COL...]] list Bridge [NAME|UUID ...]
```

- `--columns` chỉ in cột chỉ định, theo thứ tự khai báo.
- `--bare` bỏ heading + trailing whitespace, thuận tiện shell parse.
- `--no-headings` giữ nguyên định dạng nhưng bỏ dòng header.
- Tham số positional là record-id, có thể là `_uuid` đầy đủ, prefix UUID đủ unique, hoặc giá trị cột `name`.

#### Mẫu output đầy đủ


24+ cột: từ `_uuid` (luôn xuất hiện, không thể tắt) đến các cột feature flag (`stp_enable`, `rstp_enable`, `mcast_snooping_enable`), reference (`controller`, `ports`, `mirrors`, `sflow`, `ipfix`, `netflow`), config map (`external_ids`, `other_config`), và status read-only (`datapath_id`, `datapath_version`, `rstp_status`, `status`).

#### Cơ chế hoạt động

`cmd_list()` (`lib/db-ctl-base.c` line ~1370) gọi `list_make_table()` build header column theo `--columns` hoặc default tất cả cột + UUID, sau đó iterate row qua `ovsdb_idl_first_row()`/`ovsdb_idl_next_row()` (nếu không truyền record-id) hoặc `ovsdb_idl_index_find()` cho tên cụ thể. Mỗi cell render qua `ovsdb_atom_to_string()`/`ovsdb_datum_to_string()` ép ra dạng JSON-ish (`[uuid, ...]`, `{key=val}`, `"string"`). Output mặc định là format `table` (cột thẳng hàng, có header); `--bare` chuyển sang format minimal cho parse.

#### Quy trình sử dụng + workflow

Bốn bước chuẩn. Một, scope: dùng `list-br` lấy danh sách tên bridge, xác định bridge cần inspect. Hai, full dump: `list Bridge <name>` để xem mọi cột; nếu output dài, redirect file. Ba, chỉ định cột cần: `--columns=name,fail_mode,protocols,datapath_id list Bridge` cho automation script. Bốn, parse: dùng `--bare --columns=name,datapath_id list Bridge` thành 2 trường tab-separated, đưa vào `awk`/`jq` (sau format JSON qua `--format=json`).

#### Lỗi thường gặp + chẩn đoán

| Triệu chứng | Nguyên nhân khả dĩ | Cách kiểm chứng |
|-------------|---------------------|-----------------|
| `no row "X" in table Bridge` | Tên sai hoặc bridge đã xoá | `ovs-vsctl list-br` xem danh sách thực tế |
| Cột `controller` rỗng `[]` mà UI báo "đang OVN" | OVN chưa write Controller row | `ovs-vsctl list Controller`, đối chiếu `ovn-controller` log |
| `protocols: []` mà mong OF1.3 | Mặc định OVS chọn version, không hardcode | `ovs-vsctl set Bridge X protocols=OpenFlow13` để lock version |
| `--bare` vẫn ra heading | Dùng nhầm `--no-headings` only | Dùng kết hợp `--bare --no-headings` |

#### Lab thực hành

```shell
sudo ovs-vsctl --may-exist add-br br-list-lab
sudo ovs-vsctl set Bridge br-list-lab \
  fail_mode=secure protocols=OpenFlow13,OpenFlow15 \
  other_config:disable-in-band=true
sudo ovs-vsctl --columns=name,fail_mode,protocols,other_config list Bridge br-list-lab
sudo ovs-vsctl --bare --columns=name,datapath_id list Bridge
```

Tiêu chí pass: output cột chỉ định đúng thứ tự, `--bare` cho ra một dòng `br-list-lab\t0000....` không heading.

#### Source code tham chiếu

`lib/db-ctl-base.c` `cmd_list()` line ~1370, helper `list_record()` đệ quy in cell. `vswitchd/vswitch.ovsschema` định nghĩa toàn bộ 24+ cột Bridge với type constraint (`set` of UUID, `map` string→string, scalar). `ovsdb-idl.c` cung cấp `ovsdb_idl_first_row()`/`ovsdb_idl_next_row()` qua hash table.

#### Khác biệt giữa các phiên bản

- OVS 1.x: Bridge có ~10 cột (chưa có RSTP, IPFIX, IGMP snooping).
- OVS 2.0: thêm `protocols`, `flow_tables`, `auto_attach`.
- OVS 2.4: thêm `mcast_snooping_enable`.
- OVS 2.10: thêm `rstp_enable`, `rstp_status`.
- OVS 2.17: stable, không thêm cột Bridge mới đáng chú ý so với 2.13.

#### So sánh với hệ khác

PostgreSQL `\d Bridge` mô tả schema tĩnh; OVSDB không có equivalent vì schema build sẵn vào library. Linux bridge dùng `bridge link show` + `bridge fdb show` chia thành nhiều lệnh, không gom thành "row dump". Cisco NX-OS `show running-config interface vlan X` là hình thức dump nhưng output dạng config text không phải row table.

### 9.4.Y.3 `ovs-vsctl list Port`

#### Khái niệm

`list Port` dump toàn bộ cột row của bảng `Port`. Khác `Bridge`, một port có 18+ cột bao gồm thuộc tính VLAN (`tag`, `trunks`, `vlan_mode`), bond (`bond_mode`, `lacp`, `bond_active_slave`, `bond_downdelay`), QoS (`qos`), và external_ids/other_config. `list Port` là lệnh đầu tiên kỹ sư gõ khi điều tra "tại sao packet không qua port này": output cho biết VLAN tag đúng chưa, port có bond không, bond có active slave đúng không, qos profile gắn nào.

#### Cú pháp + tham số

```
ovs-vsctl [--bare] [--no-headings] [--columns=COL[,COL...]] list Port [NAME|UUID ...]
```

Cùng cú pháp `list Bridge`. Một port có thể được tham chiếu bằng tên (`name` cột UNIQUE trong bảng) hoặc UUID. Lưu ý: tên port trùng tên bridge nếu là local port (bridge `br0` có port `br0`); để tránh nhầm dùng UUID khi script automation.

#### Mẫu output đầy đủ


Các cột quan trọng: `tag` (access VLAN), `trunks` (trunk VLAN list), `vlan_mode` (`access`/`trunk`/`native-tagged`/`native-untagged`/`dot1q-tunnel`), `bond_mode` (`active-backup`/`balance-tcp`/`balance-slb`), `cvlans` (Q-in-Q), `protected` (PVLAN feature, OVS 2.13+).

#### Cơ chế hoạt động

Pipeline giống `list Bridge`: `cmd_list()` gọi `list_make_table()` build column header, `list_record()` cho từng row. Khi cột là `set of UUID` (vd `interfaces`), output in `[uuid1, uuid2]`; user sau đó dùng `list Interface <uuid>` để dereference. Đây là pattern "explicit dereference" của OVSDB CLI: khác RDBMS join, ovs-vsctl không tự follow foreign key, để giữ output deterministic + caller control.

#### Quy trình sử dụng + workflow

Năm bước chẩn đoán packet không qua port. Một, `list Port <name>` xem `tag`/`trunks`/`vlan_mode` đúng không. Hai, nếu là bond, `list Port <name>` xem `bond_mode`/`bond_active_slave`/`lacp` để biết bond đang dùng slave nào. Ba, đọc `external_ids` xem OVN binding: `iface-id="lsp-..."` phải khớp với `Logical_Switch_Port.name` bên NBDB. Bốn, `interfaces: [uuid]` → dereference qua `list Interface <uuid>` hoặc `--columns=name,type` để xem type. Năm, `qos: [uuid]` nếu khác `[]` → gắn QoS profile, kiểm tra `list QoS <uuid>` xem rate limit có đúng không.

#### Khi xảy ra sự cố cần nhớ lệnh này

VM mới boot không nhận DHCP: `list Port tap-vm1` cho thấy `tag` thiếu hoặc `external_ids:iface-id` rỗng → OVN không bind LSP. Bond fail-over chậm: `list Port bond0` cho `bond_downdelay: 200` (delay chuyển slave 200ms) có thể quá cao cho topology, giảm xuống `bond_downdelay=0`. Trunk port forward sai VLAN: `list Port` thấy `trunks: [10, 20]` thiếu VLAN 30 → cần `set Port X trunks=10,20,30`.

#### Lab thực hành

```shell

sudo ovs-vsctl add-port br-port-lab veth0 tag=42
sudo ovs-vsctl add-port br-port-lab veth1 trunks=10,20,30 vlan_mode=trunk
sudo ovs-vsctl --columns=name,tag,trunks,vlan_mode list Port veth0 veth1
sudo ovs-vsctl set Port veth0 external_ids:iface-id=lsp-test-001
sudo ovs-vsctl --columns=name,external_ids list Port veth0
```

Tiêu chí pass: `veth0` `tag=42` `trunks=[]` `vlan_mode=[]`; `veth1` `tag=[]` `trunks=[10,20,30]` `vlan_mode=trunk`. Sau set external_ids, output cột chứa `{iface-id="lsp-test-001"}`.

#### Source code tham chiếu

`vswitchd/vswitch.ovsschema` định nghĩa Port table với 18+ cột. `ofproto/bond.c` xử lý semantics `bond_mode`/`lacp`/`bond_active_slave`. Cột `cvlans`/`vlan_mode=dot1q-tunnel` xử lý ở `ofproto/ofproto-dpif-xlate.c`. Cột `protected` (PVLAN) xử lý ở `ofproto/ofproto-dpif.c` `process_port_protected()`.

#### Khác biệt giữa các phiên bản

- OVS 1.x: ~12 cột, chưa có RSTP, chưa có `protected`/`cvlans`.
- OVS 2.4: thêm `vlan_mode=dot1q-tunnel` (Q-in-Q) và `cvlans`.
- OVS 2.13: thêm `protected` (PVLAN isolated port).
- OVS 2.17: stable.

#### So sánh với hệ khác

Linux bridge dùng `bridge link show` + `bridge vlan show` để xem VLAN per port nhưng không có dump row đồng nhất. Cisco IOS `show interface trunk` tương đương ngữ nghĩa nhưng output text-based, parse khó hơn OVSDB JSON. OpenStack Neutron API expose port object qua REST `/v2.0/ports/<id>` cho output JSON gần với `list Port` nhưng cộng thêm field tenant/owner.

### 9.4.Y.4 `ovs-vsctl list Interface`

Các cột quan trọng: `type` (xác định loại), `ofport` (số port OpenFlow, dùng trong flow rule `in_port=N`), `options` (tunnel options + patch peer), `mac_in_use` (MAC thực tế OS đang dùng), `mtu` (MTU effective sau khi tính header tunnel), `link_state`+`admin_state` (trạng thái link layer + admin), `statistics` (counter), `status` (read-only diagnostics, vd `tunnel_egress_iface` cho biết tunnel route qua interface vật lý nào), `ingress_policing_rate`+`ingress_policing_burst` (rate limit kbit/s).

#### Cơ chế hoạt động

`cmd_list()` cùng path với `Bridge`/`Port`. Cột `statistics` được vswitchd cập nhật chủ động: `iface_refresh_stats()` (`vswitchd/bridge.c`) gọi netlink kéo `rtnl_link_stats64` từ kernel mỗi vòng `bridge_run()`. Cột `status` cập nhật bởi từng datapath provider: với tunnel, `netdev-vport.c` populate `tunnel_egress_iface` qua route lookup; với system NIC, `netdev-linux.c` populate `driver_name`/`driver_version`. Vì thế `list Interface` không chỉ cho config mà cả runtime telemetry.

#### Quy trình sử dụng + workflow cho từng type

Tunnel Geneve/VXLAN: kiểm tra `options:remote_ip` đúng IP chassis xa, `options:key=flow` (OVN bắt buộc), `status:tunnel_egress_iface` đúng NIC underlay, `statistics:tx_packets` tăng khi VM gửi qua. Internal port: `mac_in_use` khớp với cấu hình bonding/keepalived, `link_state=up`. Patch port: `options:peer=<peer_name>`, peer phải tồn tại bên bridge khác (cross-link `ovs-vsctl list Interface <peer>`). System NIC: `link_speed` đúng (1Gb=1000000000), `duplex=full`, `link_resets` đếm flap.

#### Khi xảy ra sự cố cần nhớ lệnh này

Tunnel không ping qua: `list Interface ovn-X` thấy `statistics:tx_packets` tăng nhưng `rx_packets=0` → packet ra nhưng không về, debug `tunnel_egress_iface` đúng NIC, MTU phù hợp. VM packet drop: `statistics:rx_dropped` tăng → over-policing, kiểm tra `ingress_policing_rate`. ofport thay đổi sau OVS restart phá vỡ flow rule `in_port=N`: `list Interface --columns=name,ofport list Interface` confirm số mới, lock số bằng `set Interface X ofport_request=N`. Patch port lệch: `options:peer=br-ex-patch` mà bên `br-ex` không có interface tương ứng → broken link.

#### Lab thực hành

```shell
sudo ovs-vsctl --may-exist add-br br-iface-lab
sudo ovs-vsctl add-port br-iface-lab tap-internal -- set Interface tap-internal type=internal
sudo ovs-vsctl add-port br-iface-lab patch-out -- set Interface patch-out type=patch options:peer=patch-in
sudo ovs-vsctl --may-exist add-br br-peer
sudo ovs-vsctl add-port br-peer patch-in -- set Interface patch-in type=patch options:peer=patch-out
sudo ovs-vsctl --columns=name,type,options,ofport,link_state list Interface tap-internal patch-out patch-in
```

Tiêu chí pass: ba interface in ra với `type` đúng, `options:peer` khớp đôi (`patch-out`↔`patch-in`), `ofport` được gán số nguyên dương, `link_state=up` cho `tap-internal` (sau `ip link set ... up`).

#### Source code tham chiếu

`vswitchd/vswitch.ovsschema` định nghĩa Interface 30+ cột. `vswitchd/bridge.c` `iface_refresh_stats()` cập nhật `statistics`. `vswitchd/bridge.c` `iface_refresh_status()` cập nhật `status`. Tunnel logic: `netdev-vport.c`, `lib/netdev.c`. Patch port logic: `ofproto/ofproto-dpif-xlate.c` `compose_output_action__()` xử lý patch peer.

#### Khác biệt giữa các phiên bản

- OVS 1.x: ~18 cột, chưa có BFD, LLDP, CFM extensive.
- OVS 2.0: thêm `bfd`/`bfd_status`, `lldp`.
- OVS 2.4: thêm `mtu_request` cho phép set MTU OVS-managed.
- OVS 2.6: tunnel mở rộng options (csum, ttl, df_default, egress_pkt_mark).
- OVS 2.10: `link_resets` thêm để track flap, `error` cho diagnostic.
- OVS 2.17: stable.

#### So sánh với hệ khác

Linux `ip -s -d link show <iface>` cho per-interface stats + driver detail nhưng không có khái niệm `type=patch`/`type=internal` của OVS, không có `external_ids`. SNMP MIB IF-MIB::ifTable expose tương đương `statistics` nhưng polling-based. `ethtool -S <iface>` chi tiết hơn về vendor-specific counter nhưng không tích hợp với OF abstraction. OVS Interface row là điểm hợp nhất duy nhất giữa OVSDB config + kernel netdev runtime.

### 9.4.Y.5 `ovs-vsctl set Port`

#### Khái niệm

`set Port` là dạng đặc biệt của subcommand generic `set TABLE RECORD COLUMN[:KEY]=VALUE [...]` áp dụng lên bảng `Port`. Đây là cách sysadmin OVS sửa thuộc tính port mà không có subcommand chuyên dụng (vd không có `add-tag`, không có `set-trunks`). `set Port br0 tag=100` đặt VLAN access; `set Port bond0 trunks=10,20,30 vlan_mode=trunk` cấu hình trunk; `set Port veth0 external_ids:iface-id=lsp-vm-001` gắn OVN binding ID.

#### Cú pháp + tham số

```
ovs-vsctl [--no-wait] [--may-exist] set TABLE RECORD COLUMN[:KEY]=VALUE [COLUMN[:KEY]=VALUE ...]
```

Chấp nhận nhiều cặp `COLUMN=VALUE` trong cùng một lệnh — tất cả nằm trong một giao tác OVSDB (atomic). Syntax `COLUMN:KEY=VALUE` dành cho cột type `map` (`external_ids:iface-id=X` set một key cụ thể trong map). `COLUMN=VALUE` cho scalar/set; với set dùng dấu phẩy: `trunks=10,20,30`. Để xoá set: `clear Port X trunks` thay vì `set`.

#### Cơ chế hoạt động

`cmd_set()` (`lib/db-ctl-base.c` line ~1240) parse `RECORD` → tìm row qua `find_row()`, sau đó iterate qua từng `COLUMN[:KEY]=VALUE` trong argv[3..]. Mỗi cặp gọi `set_column()` parse syntax → `ovsdb_idl_txn_write()` đặt giá trị mới trong transaction in-flight. Sau khi loop xong, `invalidate_cache()` đảm bảo lệnh kế tiếp đọc giá trị mới (vì IDL cache stale sau write). Toàn bộ xảy ra trong duy nhất một `ovsdb_idl_txn_commit_block()` cuối lệnh — atomic.

#### Cú pháp set map vs scalar vs set-of-values

Map (vd `external_ids`, `other_config`):

```
ovs-vsctl set Port veth0 external_ids:iface-id=lsp-001 external_ids:owner=tenant-A
ovs-vsctl remove Port veth0 external_ids iface-id        # xoá key
```

Scalar (vd `tag`):

```
ovs-vsctl set Port veth0 tag=42
ovs-vsctl clear Port veth0 tag                           # về [] (default)
```

Set of values (vd `trunks`):

```
ovs-vsctl set Port bond0 trunks=10,20,30
ovs-vsctl add Port bond0 trunks 40                       # thêm 40 vào set
ovs-vsctl remove Port bond0 trunks 20                    # bỏ 20
```

#### Quy trình sử dụng + best practice

Bốn nguyên tắc. Một, batch nhiều cột cùng giao tác để tránh inconsistent state: `set Port veth0 tag=42 vlan_mode=access external_ids:iface-id=lsp-X`. Hai, dùng `add`/`remove` cho set-of-values thay vì `set` để không ghi đè ngẫu nhiên giá trị khác đang có. Ba, quote string chứa space hoặc ký tự đặc biệt: `external_ids:description='"VM web tier"'` (note kép quote vì shell + OVSDB JSON). Bốn, `clear` thay vì `set X=[]` để semantically đúng (clear reset về default schema-level).

#### Khi xảy ra sự cố cần nhớ lệnh này

Bốn tình huống. Một, OVN không bind LSP: `set Port tap-vm1 external_ids:iface-id=<lsp-uuid-or-name>` để OVN-controller pick up. Hai, VM bị isolate sai VLAN: `set Port tap-vm1 tag=N` để chuyển VLAN access. Ba, bond không up: `set Port bond0 bond_mode=active-backup lacp=off` rồi check log. Bốn, qos rate limit cần áp ngay không restart: `set Port veth0 qos=@profile -- --id=@profile create QoS type=linux-htb other_config:max-rate=1000000000` (transaction một lệnh tạo QoS row + reference).

#### Lỗi thường gặp + chẩn đoán

| Triệu chứng | Nguyên nhân khả dĩ | Cách kiểm chứng |
|-------------|---------------------|-----------------|
| `ovs-vsctl: constraint violation: 0 is not in the valid range 1 to 4095` | `tag=0` không hợp lệ | OVSDB schema enforce; xem schema `Port.tag`; dùng `clear Port X tag` |
| `set Port X trunks=10,20` ghi đè trunk có sẵn 30 | `set` REPLACE; cần `add Port X trunks 30` hoặc liệt kê đủ | Đọc `list Port X` trước; dùng `add` cho incremental |
| `external_ids:iface-id=foo` không được OVN pick up | OVN dùng UUID không phải tên LSP | Dùng `iface-id=$(ovn-nbctl get Logical_Switch_Port lsp-X _uuid)` |
| Lệnh treo lâu | OVSDB bị lock bởi giao tác khác | `ovs-appctl -t ovsdb-server cluster/status OVS_Northbound`, kiểm tra contention |

#### Lab thực hành

```shell
sudo ovs-vsctl --may-exist add-br br-set-lab
sudo ovs-vsctl add-port br-set-lab vethA
# Set một loạt cột atomic trong giao tác
sudo ovs-vsctl set Port vethA \
  tag=99 \
  vlan_mode=access \
  external_ids:iface-id=lsp-set-lab \
  external_ids:owner=lab-tenant \
  other_config:qos-priority=10
# Verify
sudo ovs-vsctl list Port vethA
# Test add/remove cho set-of-values
sudo ovs-vsctl add-port br-set-lab vethTrunk
sudo ovs-vsctl set Port vethTrunk trunks=10,20 vlan_mode=trunk
sudo ovs-vsctl add Port vethTrunk trunks 30
sudo ovs-vsctl remove Port vethTrunk trunks 10
sudo ovs-vsctl --columns=name,trunks list Port vethTrunk
# Expected: trunks=[20, 30]
sudo ovs-vsctl clear Port vethA tag
sudo ovs-vsctl --columns=name,tag list Port vethA
# Expected: tag=[]
```

Tiêu chí pass: học viên thấy `tag=[]` sau `clear` (về default), `trunks=[20,30]` sau add+remove, `external_ids` chứa cả `iface-id` và `owner`.

#### Source code tham chiếu

`lib/db-ctl-base.c` `cmd_set()` line ~1240, helper `set_column()` parse syntax `COLUMN[:KEY]=VALUE`. `lib/db-ctl-base.c` `cmd_add()`/`cmd_remove()`/`cmd_clear()` cho set-of-values và clear semantics. `vswitchd/vswitch.ovsschema` Port table định nghĩa constraint cho từng cột (vd `tag` integer 1-4095, `trunks` set of integer 0-4095, `vlan_mode` enum).

#### Khác biệt giữa các phiên bản

- OVS 1.x: `set` cơ bản, chưa có `add`/`remove` cho set; phải `set X=val1,val2,val3` mỗi lần.
- OVS 2.0: thêm `add`/`remove` cho incremental update; `clear` để reset.
- OVS 2.4: hỗ trợ `--may-exist` cho idempotent script.
- OVS 2.10: thêm cột Port `protected` để set PVLAN.
- OVS 2.13: thêm cột Port `cvlans` cho Q-in-Q.

#### So sánh với hệ khác

Linux bridge: `bridge vlan add dev veth0 vid 42` set tag, nhưng không atomic + không idempotent + không group nhiều thay đổi. Cisco IOS `switchport access vlan 42` tương đương ngữ nghĩa nhưng nằm trong CLI tree, không transaction. OVSDB vượt trội ở chỗ một lệnh `set` có thể atomic update nhiều cột + nhiều cấp độ (scalar + map + set) với schema validation, fit automation/CI/CD.

### 9.4.Z.1 `ovs-ofctl add-flow`

#### Khái niệm

`add-flow` là subcommand cài một flow rule mới vào flow table của bridge. Trên đường truyền (wire), `ovs-ofctl` đóng gói flow spec người dùng nhập thành một message `OFPT_FLOW_MOD` với trường `command = OFPFC_ADD`, gửi qua kết nối OpenFlow tới `ovs-vswitchd`, sau đó (theo mặc định) gửi tiếp `OFPT_BARRIER_REQUEST` để đợi ACK đảm bảo flow đã thực sự được nhận. Đây là "primitive" cơ bản nhất để đẩy chính sách forwarding vào dataplane mà không cần controller.

#### Cú pháp + tham số

```
ovs-ofctl [-O OpenFlowVERSION] [--bundle] [--no-barrier] add-flow SWITCH FLOW
ovs-ofctl [-O OpenFlowVERSION] [--bundle] add-flows SWITCH FILE
```

- `SWITCH` là tên bridge (`br-int`, `br0`) hoặc datapath socket (`unix:/var/run/openvswitch/br-int.mgmt`).
- `FLOW` là chuỗi `key=value` ngăn cách bằng dấu phẩy: `table=N,priority=P,<match>,actions=<list>`. Match field theo §9.4.2.2; action list theo Phần 4.9 `OpenFlow action catalog`.
- `-O OpenFlow15` ép version (mặc định 1.0; hầu hết tính năng modern cần ≥ 1.3).
- `--bundle` gói nhiều `add-flow` thành một transaction OpenFlow 1.4 atomic (xem §9.4.Z.4 cho ngữ nghĩa).
- `--no-barrier` bỏ barrier sau message; tăng throughput script bulk-load nhưng mất xác nhận từng rule.
- Form `add-flows SWITCH FILE` đọc nhiều flow từ file, mỗi dòng một flow; cùng cờ.

#### Mẫu output thực tế

Thành công: lệnh KHÔNG in gì ra stdout, exit code 0. Đây là behavior cố ý theo Unix convention:

```shell
$ sudo ovs-ofctl -O OpenFlow15 add-flow br-int \
    'table=0,priority=200,in_port=2,ip,nw_dst=10.0.0.5,actions=output:3'
$ echo $?
0
```

Khi parse fail, lệnh in lỗi ra stderr và exit non-zero:

```shell
$ sudo ovs-ofctl -O OpenFlow15 add-flow br-int 'priority=200,xyz=1,actions=drop'
ovs-ofctl: unknown field xyz
$ echo $?
1
```

Khi switch reject (vd action không support, table không tồn tại), lỗi xuất hiện dưới dạng `OFPT_ERROR` reply, được dịch thành text:

```shell
$ sudo ovs-ofctl add-flow br0 'table=99,actions=drop'
OFPT_ERROR (xid=0x4): OFPFMFC_BAD_TABLE_ID
OFPT_FLOW_MOD (xid=0x4): ADD table:99 actions=drop
```

#### Ngữ nghĩa khi rule trùng

Theo OpenFlow spec, `add-flow` với match + priority đã tồn tại sẽ ghi đè counter (n_packets/n_bytes giữ hay reset tuỳ flag `OFPFF_RESET_COUNTS`). Nếu muốn semantic "thêm khác, sửa cùng" hãy phối hợp `--strict` mod-flows; nếu muốn fail khi trùng, phải dump trước rồi quyết định ngoài flow_mod.

#### Lab thực hành

```shell
sudo ovs-vsctl --may-exist add-br br-addflow
sudo ovs-vsctl set Bridge br-addflow protocols=OpenFlow13,OpenFlow15
sudo ovs-ofctl -O OpenFlow15 add-flow br-addflow \
  'table=0,priority=10,actions=resubmit(,1)'
sudo ovs-ofctl -O OpenFlow15 add-flow br-addflow \
  'table=1,priority=200,ip,nw_dst=10.0.0.0/24,actions=output:LOCAL'
sudo ovs-ofctl -O OpenFlow15 dump-flows br-addflow
```

Tiêu chí pass: lệnh chạy không in gì, sau đó `dump-flows` thấy hai entry với `table=0` priority 10 và `table=1` priority 200 đúng match.

#### Lỗi thường gặp + chẩn đoán

| Triệu chứng | Nguyên nhân khả dĩ | Cách kiểm chứng |
|-------------|---------------------|-----------------|
| `OFPFMFC_BAD_TABLE_ID` | `table=N` ngoài dải bridge support | `ovs-ofctl -O OpenFlow15 dump-tables br0` xem `max_entries` từng table |
| `unknown field <X>` | Match field không tồn tại ở version đang dùng | Đối chiếu Phần 4.8 OXM catalog + ép `-O OpenFlow15` |
| Action không có hiệu lực sau add | Quên barrier khi script bulk; switch chưa kịp commit khi test | Tránh `--no-barrier` ở bước critical |
| `OFPBAC_BAD_OUT_PORT` | `output:NNN` trỏ port không tồn tại hoặc DOWN | `ovs-ofctl show br0` xem danh sách port + state |

#### Source code tham chiếu

Hàm `ofctl_add_flow()` trong `utilities/ovs-ofctl.c` (line 3293) chỉ là thin wrapper gọi `ofctl_flow_mod(ctx->argc, ctx->argv, OFPFC_ADD)`. Hàm `ofctl_flow_mod()` (cùng file, line ~2845) parse flow spec qua `parse_ofp_flow_mod_str()` trong `lib/ofp-flow.c`, sau đó gọi `transact_multiple_noreply()` (kèm barrier) hoặc `transact_noreply()` (nếu `--no-barrier`). Bundle path đi qua `bundle_transact()` trong `lib/ofp-bundle.c`.

#### Khác biệt giữa các phiên bản

- OVS 1.x (OpenFlow 1.0): `add-flow` chỉ một table (table 0), không hỗ trợ `goto_table`.
- OVS 2.0 (OpenFlow 1.1+): hỗ trợ multi-table pipeline, cần ép `-O OpenFlow13` để dùng `goto_table`.
- OVS 2.4 (2015): hỗ trợ NXM extension `ct_state`, `ct_zone` trong match (xem Phần 9.24).
- OVS 2.6 (2016): hỗ trợ `--bundle` cho atomic bulk (OpenFlow 1.4 bundle spec).
- OVS 2.10+: thêm action `clone`, `meter`, hoàn chỉnh OF 1.5 features.

#### So sánh với hệ khác

Controller-pushed flow (Ryu, ONOS, Faucet, OVN `ovn-controller`) gửi `OFPT_FLOW_MOD` qua kết nối southbound OpenFlow đã thiết lập từ trước; `ovs-ofctl add-flow` dùng kết nối quản lý cục bộ (`ptcp:`, `unix:`) và bỏ qua controller. Linux `tc filter add` thao tác trên qdisc, hoàn toàn khác abstraction. Cisco OpenFlow 1.3 capable switch dùng OF spec giống nhau, nhưng cú pháp CLI cục bộ đặc thù; `ovs-ofctl` là CLI universal nhất cho OpenFlow trên Linux.

### 9.4.Z.2 `ovs-ofctl dump-flows`

#### Khái niệm

`dump-flows` truy vấn flow table hiện tại của bridge và in từng entry kèm thống kê: cookie, duration, n_packets, n_bytes, idle_age, hard_age, priority, match, actions. Wire-level: gửi message `OFPT_MULTIPART_REQUEST` với `OFPMP_FLOW`, switch trả nhiều `OFPT_MULTIPART_REPLY` chứa các `OFPST_FLOW` body. `ovs-ofctl` aggregate, optional sort, rồi format text.

#### Cú pháp + tham số

```
ovs-ofctl [-O OpenFlowVERSION] [--sort=FIELD] [--no-stats] dump-flows SWITCH [FLOWSPEC]
```

- `FLOWSPEC` (optional) lọc theo match field: `'table=0'`, `'in_port=2,ip'`, hoặc rỗng (dump all).
- `--sort=FIELD` sort theo field: `--sort=table,priority` (descending priority within table) là idiom phổ biến.
- `--no-stats` bỏ counter, hữu ích khi diff hai snapshot flow table không quan tâm packet count.
- Filter chỉ xác định "rule nào báo cáo"; switch trả các rule có match overlap với spec, không nhất thiết bằng nhau.

#### Mẫu output thực tế

```shell
$ sudo ovs-ofctl -O OpenFlow15 --sort=table,priority dump-flows br-int
 cookie=0x0, duration=512.301s, table=0, n_packets=12, n_bytes=1176, priority=200,arp,arp_op=1 actions=resubmit(,8)
 cookie=0x0, duration=512.298s, table=0, n_packets=842, n_bytes=66780, priority=100,ip actions=resubmit(,1)
 cookie=0x0, duration=512.295s, table=0, n_packets=0, n_bytes=0, priority=0 actions=drop
 cookie=0x0, duration=510.001s, table=1, n_packets=842, n_bytes=66780, priority=100,ip,nw_dst=10.0.0.0/24 actions=output:LOCAL
```

Mỗi dòng là một flow entry. `duration` là số giây từ khi entry được cài; `n_packets`/`n_bytes` là counter lifetime. Nếu một entry đột nhiên `n_packets=0` sau thời gian dài, có thể nó bị bỏ qua bởi rule priority cao hơn hoặc match không bao giờ xảy ra.

#### Ngữ nghĩa filter

Filter `'in_port=2,ip'` được switch dùng làm "search criteria" theo OpenFlow spec §7.4.5: nó match một flow nếu **flow's match overlap** với criteria. Hệ quả: dump filter `nw_dst=10.0.0.5` sẽ in cả flow `nw_dst=10.0.0.0/24` (vì /24 chứa /32). Đây là điểm mới học hay nhầm: filter dump KHÔNG phải SQL `WHERE`.

#### Lab thực hành

```shell
sudo ovs-vsctl --may-exist add-br br-dump
sudo ovs-ofctl add-flow br-dump 'table=0,priority=100,ip,nw_dst=10.0.0.0/24,actions=drop'
sudo ovs-ofctl add-flow br-dump 'table=0,priority=200,ip,nw_dst=10.0.0.5,actions=output:1'
sudo ovs-ofctl --sort=priority dump-flows br-dump 'ip'
sudo ovs-ofctl --no-stats dump-flows br-dump | diff - /tmp/baseline.flows || echo "drift!"
```

Tiêu chí pass: học viên thấy entry priority=200 in trước priority=100 (nhờ `--sort`), và phân biệt được `--no-stats` xoá cột `n_packets/n_bytes/duration` nên thuận tiện diff.

#### Lỗi thường gặp + chẩn đoán

| Triệu chứng | Nguyên nhân khả dĩ | Cách kiểm chứng |
|-------------|---------------------|-----------------|
| Dump rỗng dù thấy traffic | Flow trên `br-int` table khác (vd table 0 chỉ resubmit), thực thi ở table 8/16 | Dump không filter; xem `table=*` |
| `OFPBRC_EPERM` | Bridge bị `ovn-controller` khoá flow thông qua ovsdb | `ovs-vsctl get Bridge br-int controller` xem có controller |
| Output chậm vài giây | OVN setup có ~50K flow trên `br-int`; multipart reply lớn | Dùng `--sort=table` + grep table cụ thể |
| `priority` field thiếu | Match rule ngầm priority 32768 (OpenFlow default) | OpenFlow spec định nghĩa default; explicit priority luôn rõ ràng hơn |

#### Source code tham chiếu

Hàm `ofctl_dump_flows()` trong `utilities/ovs-ofctl.c` line 3194 gọi `dump_trivial_transaction()` cho path không có flow spec, hoặc `dump_flows__()` (cùng file) cho path có filter, sort. Sort dispatch tới `flow_compare()` qua qsort. Receiver multipart reply dùng `vconn_recv_block()` lặp đến khi `OFPMPF_REPLY_MORE` flag tắt. Body parser `ofputil_decode_flow_stats_reply()` trong `lib/ofp-flow.c`.

#### Khác biệt giữa các phiên bản

- OVS 1.x (OpenFlow 1.0): single table, không có `--sort=table`.
- OVS 2.0+ (OpenFlow 1.3+): multi-table; `dump-flows` thêm cột `table=N`.
- OVS 2.5: `idle_age` + `hard_age` xuất hiện trong output (Nicira extension).
- OVS 2.10+: hỗ trợ `cookie_mask` để dump theo nhóm cookie (filter cookie có bitmask).
- OVS 3.x: refactor sort, thêm `--sort=table,priority` mặc định cho output có thứ tự ổn định trong test.

#### So sánh với hệ khác

`ovs-dpctl dump-flows` xem megaflow ở kernel datapath (xem §9.4.4.3): lớp khác hẳn, là các microflow đã wildcard sau quá trình xlate, KHÔNG phải flow OpenFlow. `ovn-sbctl lflow-list` đọc Logical_Flow trên SBDB ở mức control plane OVN, abstraction cao hơn flow OpenFlow trên `br-int`. Khi debug OVN, nguyên tắc: `lflow-list` (logical) → `dump-flows br-int` (physical OpenFlow) → `dpctl/dump-flows` (datapath megaflow) là 3 lớp đi từ trên xuống.

### 9.4.Z.3 `ovs-ofctl mod-flows`

#### Khái niệm

`mod-flows` chỉnh sửa các flow đang tồn tại match với criteria được cung cấp. Wire-level: gửi `OFPT_FLOW_MOD` với `command = OFPFC_MODIFY` (loose match) hoặc `OFPFC_MODIFY_STRICT` (strict match) tuỳ cờ `--strict`. Toàn bộ flow match criteria sẽ có actions ghi đè bằng action list mới; counter (`n_packets`, `n_bytes`) giữ nguyên trừ khi flag `OFPFF_RESET_COUNTS` được set.

#### Cú pháp + tham số

```
ovs-ofctl [-O OpenFlowVERSION] [--strict] [--bundle] [--no-barrier] mod-flows SWITCH FLOW
```

- `--strict` đổi command từ `OFPFC_MODIFY` sang `OFPFC_MODIFY_STRICT`. Khác biệt cốt lõi: loose modify match flow **chứa** criteria (subset semantic) trong khi strict modify yêu cầu match **trùng khít** + cùng `priority`.
- `FLOW` chứa cả match (criteria) lẫn `actions=...` (giá trị mới). Ví dụ: `'priority=200,ip,nw_dst=10.0.0.5,actions=output:5'` — match `nw_dst=10.0.0.5` trùng khít rồi đổi action thành `output:5`.
- `--bundle` gói transaction atomic; `--no-barrier` bỏ qua xác nhận.

#### Khác biệt loose vs strict

```
loose:  modify nơi flow's match ⊇ criteria  (criteria là subset)
strict: modify nơi flow's match == criteria AND priority == criteria.priority
```

Hệ quả: loose `mod-flows br0 'ip,actions=drop'` có thể vô tình ghi đè cả nghìn rule (mọi rule có match là superset của `ip`); strict yêu cầu chỉ định đúng match + priority như khi rule được tạo. Production discipline: **luôn dùng `--strict` cho mod-flows** trừ khi cố ý bulk-modify.

#### Mẫu output thực tế

Thành công không in gì:

```shell
$ sudo ovs-ofctl --strict -O OpenFlow15 mod-flows br-int \
    'table=1,priority=200,ip,nw_dst=10.0.0.5,actions=output:5'
$ echo $?
0
$ sudo ovs-ofctl dump-flows br-int 'table=1,nw_dst=10.0.0.5'
 cookie=0x0, duration=8.21s, table=1, n_packets=120, n_bytes=9600, priority=200,ip,nw_dst=10.0.0.5 actions=output:5
```

Lưu ý `n_packets=120` không reset; `duration=8.21s` cũng không reset (entry vẫn cùng entry, chỉ khác actions). Nếu rule không tồn tại, `OFPFC_MODIFY` (loose) sẽ INSERT entry mới (theo OpenFlow 1.0 legacy semantic, OVS giữ tương thích) — đây là pitfall lớn.

#### Lab thực hành

```shell
sudo ovs-vsctl --may-exist add-br br-mod
sudo ovs-ofctl add-flow br-mod 'table=0,priority=200,ip,nw_dst=10.0.0.5,actions=output:1'
sudo ovs-ofctl --strict mod-flows br-mod \
  'table=0,priority=200,ip,nw_dst=10.0.0.5,actions=output:2'
sudo ovs-ofctl dump-flows br-mod | grep nw_dst=10.0.0.5
# Expected: actions=output:2, n_packets giữ nguyên nếu đã có traffic
sudo ovs-ofctl mod-flows br-mod \
  'priority=200,ip,actions=drop'
# Loose: TẤT CẢ rule có priority=200,ip giờ thành actions=drop  ← cẩn thận
```

Tiêu chí pass: học viên giải thích được tại sao bước 2 (strict) chỉ chỉnh đúng 1 rule, còn bước 3 (loose, không strict) ảnh hưởng nhiều rule.

#### Lỗi thường gặp + chẩn đoán

| Triệu chứng | Nguyên nhân khả dĩ | Cách kiểm chứng |
|-------------|---------------------|-----------------|
| `mod-flows` "chèn thêm rule" thay vì sửa | Không dùng `--strict`, criteria không match nào → loose mod auto-insert | Dump trước/sau, dùng `--strict` |
| Rule biến mất sau mod | Loose mod với action không hợp lệ; OVS reject; tuy nhiên `mod` semantic không xoá | Xem `ovs-vswitchd.log` |
| Counter reset bất ngờ | Có flag `OFPFF_RESET_COUNTS` (rare) | Mặc định OVS không reset khi mod |
| Action set trống không có hiệu ứng | `actions=` viết sai chính tả → parser nhận actions rỗng = drop (OF default) | `OFPT_ERROR` thường bắt được; nếu im lặng kiểm tra cú pháp |

#### Source code tham chiếu

Hàm `ofctl_mod_flows()` trong `utilities/ovs-ofctl.c` line 3303 dispatch theo cờ `--strict`: gọi `ofctl_flow_mod()` với `OFPFC_MODIFY` hoặc `OFPFC_MODIFY_STRICT`. Strict-flag được parse trong `parse_options()` cùng file. Logic strict vs loose match thực thi tại `ofproto.c::handle_flow_mod__()` upstream qua `rule_criteria_init()` với strict bool. Bundle path đi qua `lib/ofp-bundle.c`.

#### Khác biệt giữa các phiên bản

- OVS 1.x (OpenFlow 1.0): chỉ có `OFPFC_MODIFY` loose; `OFPFC_MODIFY_STRICT` thêm trong 1.0.0 spec đoạn cuối.
- OVS 2.0 (OpenFlow 1.1+): bổ sung `cookie_mask` cho mod theo nhóm cookie controller-set.
- OVS 2.6 (2016): hỗ trợ `--bundle`, mod nằm trong atomic transaction.
- OVS 2.10: action `clone` cho phép mod action chứa nested action set; parser cập nhật.
- OVS 3.x: stricter parser cho `actions=` rỗng (warn rõ hơn).

#### So sánh với hệ khác

Cisco IOS không có analog "modify single flow"; ACL line edit phải `no <line>` rồi `<new-line>`, không atomic. Linux nftables `nft replace rule ...` tương đương semantic strict-modify, dùng handle ID để target chính xác rule. OpenFlow strict modify trong `ovs-ofctl` mạnh hơn ở chỗ không cần handle, chỉ cần `match + priority` đầy đủ là đủ unique key — thiết kế phù hợp với SDN declarative.

### 9.4.Z.4 `ovs-ofctl del-flows`

#### Khái niệm

`del-flows` xoá flow rule khỏi flow table. Wire-level: gửi `OFPT_FLOW_MOD` với `command = OFPFC_DELETE` (loose) hoặc `OFPFC_DELETE_STRICT` (strict). Loose delete xoá mọi rule có match là superset của criteria; strict yêu cầu match trùng khít + cùng priority. `del-flows SWITCH` không kèm criteria sẽ xoá TOÀN BỘ flow trong tất cả table — production phải cực kỳ thận trọng.

#### Cú pháp + tham số

```
ovs-ofctl [-O OpenFlowVERSION] [--strict] [--bundle] [--no-barrier] del-flows SWITCH [FLOW]
```

- Vắng `FLOW`: xoá tất cả flow trên bridge (ngoại trừ một số rule controller có flag `OFPFF_NO_DELETE` rất hiếm).
- `--strict`: bắt buộc match + priority trùng khít.
- `--bundle`: nằm trong atomic bundle với các flow_mod khác (xem §9.4.Z.5 ngữ nghĩa bundle).
- `out_port=N`/`out_group=N` (kèm criteria): chỉ xoá rule có action chứa output tới port/group đó — useful khi gỡ port mà không biết rule nào reference.

#### Mẫu output thực tế

Thành công không in gì:

```shell
$ sudo ovs-ofctl --strict -O OpenFlow15 del-flows br-int \
    'table=1,priority=200,ip,nw_dst=10.0.0.5'
$ echo $?
0
$ sudo ovs-ofctl dump-flows br-int 'table=1,nw_dst=10.0.0.5'
$  # rỗng — đã xoá
```

Khi rule không tồn tại, `del-flows` (cả loose lẫn strict) im lặng exit 0 — không error. Đây là idempotent semantic, hữu ích trong cleanup script. Nếu cần biết "có thực xoá không", phải dump trước/sau và compare.

#### Ngữ nghĩa loose vs strict

```
loose:  delete mọi flow có match ⊇ criteria
strict: delete chỉ flow có match == criteria AND priority == criteria.priority
```

Pitfall điển hình: `del-flows br-int 'ip'` (loose) xoá MỌI rule có IP match, kể cả rule controller set với match phức tạp hơn — production downtime. Production discipline: **luôn `--strict` trừ khi mục đích thực sự là bulk-delete**.

#### Lab thực hành

```shell
sudo ovs-vsctl --may-exist add-br br-del
sudo ovs-ofctl add-flow br-del 'table=0,priority=10,actions=resubmit(,1)'
sudo ovs-ofctl add-flow br-del 'table=1,priority=100,ip,nw_dst=10.0.0.0/24,actions=drop'
sudo ovs-ofctl add-flow br-del 'table=1,priority=200,ip,nw_dst=10.0.0.5,actions=output:1'
# Strict delete chỉ rule cụ thể
sudo ovs-ofctl --strict del-flows br-del \
  'table=1,priority=200,ip,nw_dst=10.0.0.5'
# Loose delete xoá nhiều
sudo ovs-ofctl del-flows br-del 'table=1,ip'
sudo ovs-ofctl dump-flows br-del
# Còn rule table=0 priority=10
```

Tiêu chí pass: học viên thấy strict del giữ nguyên các rule khác; loose del quét sạch table 1.

#### Lỗi thường gặp + chẩn đoán

| Triệu chứng | Nguyên nhân khả dĩ | Cách kiểm chứng |
|-------------|---------------------|-----------------|
| Toàn bộ traffic drop sau `del-flows` | Đã chạy `del-flows br-int` (không criteria), xoá hết flow OVN-pushed | Restart `ovn-controller` để re-push; hoặc dùng `replace-flows` thay del+add |
| `del-flows --strict` không xoá rule mong muốn | Quên `priority=N` trong criteria | Strict yêu cầu cả match + priority |
| OVN flow tự re-appear sau del | `ovn-controller` watch SBDB Logical_Flow, re-install ngay | Phải xoá ở SBDB hoặc disable controller |
| `OFPBRC_EPERM` | Một số bridge controller-only mode | Kiểm tra mode + role qua `list Controller` |

#### Source code tham chiếu

Hàm `ofctl_del_flows()` trong `utilities/ovs-ofctl.c` line 3309: switch case theo `--strict` đẩy command code `OFPFC_DELETE_STRICT` hoặc `OFPFC_DELETE`. Khác biệt loose/strict thực thi trong `ofproto/ofproto.c::delete_flows_loose__()` vs `delete_flows_strict__()`. Cờ `out_port`/`out_group` parse tại `parse_ofp_flow_mod_str()` trong `lib/ofp-flow.c` ánh xạ sang field `out_port` của `struct ofputil_flow_mod`.

#### Khác biệt giữa các phiên bản

- OVS 1.x (OpenFlow 1.0): `del-flows` cơ bản, `--strict` chỉ áp dụng exact match.
- OVS 2.0 (OpenFlow 1.1+): `out_group` được hỗ trợ; criteria có thể lọc theo group reference.
- OVS 2.4: `cookie_mask` cho phép xoá theo cookie group (vd OVN tag rule với cookie hash của Logical_Flow UUID).
- OVS 2.6: `--bundle` đồng bộ nhiều del + add atomic.
- OVS 3.x: parser stricter, kiểm tra criteria khi rỗng (cảnh báo rõ ràng hơn khi del-flows toàn bridge).

#### So sánh với hệ khác

iptables/nftables `iptables -D` cần chỉ định rule line number hoặc match exact arguments; nftables dùng handle ID. OpenFlow strict del dùng `match + priority` làm khoá, bypass cần line/handle, gần hơn với SQL `DELETE WHERE`. Cisco ACL `no permit ...` ép trùng lệnh ban đầu — strict-only semantic, không có loose. `ovs-ofctl del-flows` linh hoạt hơn nhờ có cả hai mode, nhưng chính sự linh hoạt là nguồn pitfall production.

### 9.4.Z.5 `ovs-ofctl dump-tables`

#### Khái niệm

`dump-tables` báo cáo metadata của từng flow table trên bridge: số entry hiện tại (`active_count`), số packet đã xử lý qua table (`lookup_count`), số packet matched (`matched_count`), và (với OpenFlow 1.3+) khả năng support: `max_entries`, các match field được chấp nhận, action được hỗ trợ, `instruction` cho phép. Có hai message liên quan trên wire: `OFPMP_TABLE` (basic stats) và `OFPMP_TABLE_FEATURES` (chi tiết capability) — `dump-tables` mặc định lấy `OFPMP_TABLE`.

#### Cú pháp + tham số

```
ovs-ofctl [-O OpenFlowVERSION] dump-tables SWITCH
ovs-ofctl [-O OpenFlowVERSION] dump-table-features SWITCH    # detailed
```

- `dump-tables` ngắn gọn, in một dòng/table với name, lookup_count, matched_count, active_count.
- `dump-table-features` chi tiết hơn nhiều: `metadata_match`, `metadata_write`, `match` field set, `apply_actions` set, `next_tables` list — dùng khi viết controller mới cần biết switch hỗ trợ feature nào.

#### Mẫu output thực tế

`dump-tables` ngắn gọn (OF 1.0/1.3 form):

```shell
$ sudo ovs-ofctl -O OpenFlow15 dump-tables br-int
OFPST_TABLE reply (OF1.5) (xid=0x2):
  table 0 ("classifier"):
    active=42, lookup=1842901, matched=1842899
  table 1 ("table1"):
    active=18, lookup=1820445, matched=1820445
  table 2 ("table2"):
    active=0, lookup=0, matched=0
  ...
  table 254 ("table254"):
    active=0, lookup=0, matched=0
```

`active=42` nghĩa là 42 flow đang sống trong table 0; `matched - lookup = 2` nghĩa có 2 packet không match rule nào (rơi vào table-miss flow hoặc bị drop default).

`dump-table-features` (rút gọn):

```shell
$ sudo ovs-ofctl dump-table-features br-int
  table 0 ("classifier"):
    metadata: match=0xffffffffffffffff write=0xffffffffffffffff
    config=table-miss-controller
    max_entries=1000000
    instructions (table miss and others):
      next tables: 1-254
      instructions: meter,apply_actions,clear_actions,write_actions,write_metadata,goto_table
    matching:
      in_port: arbitrary mask
      eth_src: arbitrary mask
      ...
```

`max_entries=1000000` là capacity OVS tự khai báo (thực tế giới hạn bởi RAM); `next tables: 1-254` cho biết `goto_table` có thể nhảy tới table nào.

#### Vai trò trong vận hành

Khi triển khai controller mới (Faucet, OVN, custom Ryu app), bước đầu là `dump-table-features` để xác nhận switch support feature controller cần. Trên OVS, hầu hết feature available, nhưng trên hardware switch thật (Pica8, Edgecore, Trident-based), capability hạn chế hơn — `dump-table-features` là single source of truth.

#### Lab thực hành

```shell
sudo ovs-vsctl --may-exist add-br br-tab
sudo ovs-ofctl add-flow br-tab 'table=0,priority=10,actions=resubmit(,1)'
sudo ovs-ofctl add-flow br-tab 'table=1,priority=100,ip,actions=drop'
sudo ovs-ofctl dump-tables br-tab | head -10
sudo ovs-ofctl dump-table-features br-tab | grep -A 2 'table 0'
# Sau khi gửi vài ping:
sudo ip link set br-tab up
sudo ip addr add 10.99.0.1/24 dev br-tab
ping -c 3 -W 1 10.99.0.99 2>/dev/null || true
sudo ovs-ofctl dump-tables br-tab | grep -E 'table 0|active|lookup'
# Quan sát lookup_count tăng
```

Tiêu chí pass: học viên thấy `active=2` trong table 0+1 sau add-flow, `lookup_count` tăng sau khi sinh traffic, và liệt kê được `next tables` của table 0.

#### Lỗi thường gặp + chẩn đoán

| Triệu chứng | Nguyên nhân khả dĩ | Cách kiểm chứng |
|-------------|---------------------|-----------------|
| `active_count=0` ở mọi table dù đã add-flow | Add-flow ép `-O OpenFlow15` nhưng dump không ép → OVS chỉ trả table 0 | Ép cùng version `-O OpenFlow15` cho cả hai |
| `matched_count` tăng nhanh, `lookup_count` không tăng | Counter race ở table-miss path; thực ra `matched = lookup - missed` | Dump nhiều lần để average; xem source `ofproto.c::table_dpif_get_stats()` |
| Output cắt ở table 254 | `active_count=0` từ table 64 trở lên, tool tự cắt cho ngắn | Thêm `-v` hoặc `dump-table-features` |
| `dump-table-features` báo `OFPBRC_BAD_VERSION` | Bridge protocol chưa enable OF 1.3+ | `ovs-vsctl set Bridge br0 protocols=OpenFlow13,OpenFlow15` |

#### Source code tham chiếu

Hàm `ofctl_dump_tables()` trong `utilities/ovs-ofctl.c` line 2754 gọi `dump_trivial_transaction()` với `OFPRAW_OFPST_TABLE_REQUEST`. Hàm `ofctl_dump_table_features()` line 2759 phức tạp hơn: lặp nhận multipart reply, dedup khi có nhiều message vì features có thể vượt một packet. Decoder `ofputil_decode_table_stats_reply()` + `ofputil_decode_table_features()` trong `lib/ofp-table.c`. Switch-side stats trả từ `ofproto/ofproto.c::table_dpif_get_stats()`.

#### Khác biệt giữa các phiên bản

- OVS 1.x (OpenFlow 1.0): chỉ có `OFPST_TABLE`; không có `OFPMP_TABLE_FEATURES`, không có table name.
- OVS 2.0 (OpenFlow 1.3): thêm `OFPMP_TABLE_FEATURES`; mỗi table có thể đặt name qua `set Bridge br0 other_config:tablename0=classifier`.
- OVS 2.5: cải thiện dump format, hiển thị `next tables` chính xác hơn.
- OVS 2.10: cập nhật để hỗ trợ OF 1.5 table feature pipeline_fields.
- OVS 3.x: dump format ổn định hơn cho diff/snapshot test, thay đổi nhỏ về tên column.

#### So sánh với hệ khác

Linux không có analog trực tiếp; `tc -s qdisc show` cho stats nhưng cấu trúc khác hẳn. Cisco IOS OpenFlow capable switch có `show openflow switch <id> table` tương đương semantic, output format khác. P4Runtime có `ReadRequest` cho `TABLE_ENTRY` + `COUNTER` riêng — model khác (P4 không cố định 256 table). `dump-tables` của `ovs-ofctl` là interface universal nhất để nhìn vào pipeline OpenFlow trên Linux + OVS.

## 9.4.W Mở rộng tham chiếu nhóm option `ovs-vsctl` cho output + transaction (mở rộng 2026-04)

> **Phạm vi mục này.** Năm option global của `ovs-vsctl` ảnh hưởng tới hành vi giao dịch và format output: `--timeout=N`, `--columns=COL1,COL2`, `--bare`, `--format=table|csv|json|...`, và `--pretty`. Các option này đã được sử dụng rải rác ở §9.4.1 và §9.4.X/Y nhưng chưa có section riêng cho từng option. Mục này gom lại với mức độ tham chiếu nhanh: cú pháp, mẫu output, vai trò trong vận hành, lỗi thường gặp. Nguồn upstream: man `ovs-vsctl(8)` SYNOPSIS + DATABASE COMMANDS, code parse trong `utilities/ovs-vsctl.c::vsctl_add_base_prerequisites()` và shared option parse `lib/db-ctl-base.c::ctl_parse_options()` (OVS branch-2.17 baseline 2.17.9).

---

### 9.4.W.1 `ovs-vsctl --timeout=N`

#### Khái niệm

`--timeout=N` đặt giới hạn thời gian (giây) mà `ovs-vsctl` chờ phản hồi từ `ovsdb-server` cho toàn bộ giao tác, bao gồm cả thời gian chờ lock OVSDB nếu có client khác đang nắm lock. Mặc định `--timeout=0` nghĩa là "đợi vô hạn" (man page ghi *"infinite by default"*). Đây là khác biệt quan trọng so với các CLI Linux quen thuộc nơi default thường là vài giây.

#### Cú pháp + ngữ nghĩa

```
ovs-vsctl --timeout=N <command> [-- <command>]...
```

`N` tính bằng giây, integer. Khi quá `N` giây mà chưa nhận reply hoặc chưa lấy được lock, `ovs-vsctl` exit với mã lỗi và in `ovs-vsctl: connecting to ... failed (Operation timed out)`. Quan trọng: timeout đo từ lúc start tới lúc commit; nó không reset giữa các statement `--`.

#### Mẫu output thực tế

```shell
$ sudo ovs-vsctl --timeout=3 show
ovs-vsctl: connecting to unix:/var/run/openvswitch/db.sock failed (Operation timed out)
$ echo $?
1
```

Khi `ovsdb-server` đang khởi động lại hoặc đang flush snapshot lớn, lệnh không có timeout sẽ treo vô thời hạn. Đặt `--timeout=3` cho phép health-check script phát hiện sớm.

#### Vai trò trong vận hành

Trong script monitoring + automation, luôn đặt `--timeout` ngắn (3-10s). Tránh dùng default vô hạn vì nó làm script monitoring treo nếu OVS gặp sự cố, kéo theo Nagios/Prometheus check timeout cấp cao hơn. Nếu cần giao tác lớn (restore từ backup), dùng timeout 60-120s.

#### Lỗi thường gặp + chẩn đoán

| Triệu chứng | Nguyên nhân khả dĩ | Cách kiểm chứng |
|-------------|---------------------|-----------------|
| Lệnh treo vô hạn ở môi trường mới | Quên `--timeout` | Luôn đặt `--timeout=5` trong script |
| `Operation timed out` lặp lại | `ovsdb-server` đang flush snapshot lớn (>1 GB DB) | `ls -lh /etc/openvswitch/conf.db`; tăng timeout lên 60s |
| Timeout khi gọi từ container | Socket path khác giữa container/host | Bind-mount `/var/run/openvswitch/`; set `--db=` đúng |

#### Khác biệt giữa các phiên bản

OVS 1.x mặc định timeout 5s. OVS 2.0+ chuyển sang vô hạn (man page ghi rõ `default: infinite`); thay đổi này dễ gây nhầm cho ai quen 1.x. OVS 2.17 hành vi giống 2.0+, không thay đổi.

#### Source code tham chiếu

Tham số được parse trong `lib/db-ctl-base.c::ctl_parse_options()` lưu vào biến `timeout`, sau đó truyền vào `ovsdb_idl_set_timeout()` (`lib/ovsdb-idl.c`) trước khi commit transaction.

### 9.4.W.2 `ovs-vsctl --columns=COL1,COL2`

#### Khái niệm

`--columns=COL1,COL2,...` giới hạn output của `list` (và `find`) chỉ in các cột được liệt kê, theo đúng thứ tự khai báo. Không có option này, `list` in toàn bộ ~15-30 cột của bảng (`Bridge`, `Port`, `Interface`), gây nhiễu khi script chỉ cần 2-3 trường.

#### Cú pháp

```
ovs-vsctl --columns=COL1,COL2,... list TABLE [RECORD]...
ovs-vsctl --columns=COL1,COL2,... find TABLE [COLUMN=VALUE]...
```

Tên cột phải khớp schema (ví dụ `name`, `fail_mode`, `datapath_id` cho `Bridge`). Sai tên → `ovs-vsctl: TABLE does not contain a column named COL`.

#### Mẫu output thực tế

```shell
$ sudo ovs-vsctl --columns=name,fail_mode,datapath_id list Bridge
name                : br-int
fail_mode           : secure
datapath_id         : "00004e8d3c1f9a02"

name                : br-ex
fail_mode           : standalone
datapath_id         : "0000aabbccddeeff"
```

So với output đầy đủ ~20 dòng/bridge, output 3-cột này dễ grep + parse hơn rõ rệt.

#### Vai trò trong vận hành

Bắt buộc trong mọi script automation: kết hợp `--columns=name list Bridge` + shell loop là pattern phổ biến nhất. Cũng dùng kèm `--bare` (xem §9.4.W.3) cho parse máy.

#### Lỗi thường gặp + chẩn đoán

| Triệu chứng | Nguyên nhân khả dĩ | Cách kiểm chứng |
|-------------|---------------------|-----------------|
| `does not contain a column named X` | Sai tên cột hoặc nhầm bảng (Bridge có `fail_mode`, Port không) | `ovs-vsctl list Bridge` (không option) xem schema thực |
| Output rỗng | `find` filter quá chặt + `--columns` chỉ in cột cụ thể | Bỏ `--columns` để xem full row |
| Output khác thứ tự mong đợi | `--columns` tôn trọng thứ tự khai báo, không alphabetical | Đặt thứ tự cột đúng từ đầu |

#### Khác biệt giữa các phiên bản

Có sẵn từ OVS 1.x. OVS 2.0+ thêm hỗ trợ `--columns` cho `find` (1.x chỉ `list`). Tên cột schema-dependent: thêm cột mới khi schema bump (ví dụ `Bridge.auto_attach` từ OVS 2.6) sẽ làm `--columns=auto_attach` trên server cũ fail.

#### Source code tham chiếu

Hàm `parse_column_names()` trong `lib/db-ctl-base.c` xử lý chuỗi `--columns`, validate qua `ovsdb_idl_class_*` schema metadata. Output formatter `cmd_list_postprocess()` chỉ in cột thuộc set đã chọn.

---

### 9.4.W.3 `ovs-vsctl --bare`

#### Khái niệm

`--bare` bỏ heading + decoration trong output `list`/`find`/`get`, in giá trị thuần ở dạng tab-separated/whitespace-separated, không có tên cột bên trái cũng không có dấu `: ` phân tách. Mục tiêu: cho output thân thiện shell (`while read`, `awk`, `cut`).

#### Cú pháp

```
ovs-vsctl --bare [--columns=COL,...] list TABLE
ovs-vsctl --bare get TABLE RECORD COLUMN
```

Thường đi cùng `--columns` để chỉ định cột cần. Không có `--columns`, `--bare` vẫn in mọi cột nhưng không heading — gần như không bao giờ hữu ích vì output thành "data dump khó parse".

#### Mẫu output thực tế

So sánh:

```shell
$ sudo ovs-vsctl --columns=name,datapath_id list Bridge
name                : br-int
datapath_id         : "00004e8d3c1f9a02"

$ sudo ovs-vsctl --bare --columns=name,datapath_id list Bridge
br-int
00004e8d3c1f9a02
```

Output `--bare` mất tên cột `name`/`datapath_id`, mất padding 20-ký-tự, mất quote `"..."` quanh datapath_id. Mỗi giá trị một dòng.

#### Vai trò trong vận hành

Pattern đi kèm shell loop:

```shell
for br in $(sudo ovs-vsctl --bare --columns=name list Bridge); do
  sudo ovs-ofctl dump-flows "$br" | wc -l
done
```

Không dùng `--bare`, vòng lặp sẽ ăn cả heading `name` và padding, gây sai logic.

#### Lỗi thường gặp + chẩn đoán

| Triệu chứng | Nguyên nhân khả dĩ | Cách kiểm chứng |
|-------------|---------------------|-----------------|
| Shell loop fail vì biến chứa tên cột | Quên `--bare` | Thêm `--bare` |
| Output `--bare` vẫn có quote `"..."` | OVS 2.x giữ quote cho string column trong một số format | Strip qua `tr -d '"'` |
| Set/map column in dạng `[a,b,c]` khó parse | `--bare` không tách set thành dòng | Dùng `--format=csv` hoặc post-process |

#### Khác biệt giữa các phiên bản

Có sẵn từ OVS 1.x. OVS 2.7+ tinh chỉnh quote handling cho string set (consistent quote stripping); OVS 2.17 hành vi ổn định.

#### Source code tham chiếu

Flag `--bare` set biến `bare` trong `lib/db-ctl-base.c::ctl_parse_options()`; hàm `table_format()` (`lib/table.c`) check biến này để skip header + alignment padding.

### 9.4.W.4 `ovs-vsctl --format=table|csv|json|...`

#### Khái niệm

`--format` chọn cú pháp của output: `table` (mặc định, human-friendly với cột thẳng hàng), `list` (one-row-per-line), `html`, `csv` (comma-separated cho spreadsheet), `json` (machine-readable cho API/script). Áp dụng cho `list`, `find`, `get`.

#### Cú pháp

```
ovs-vsctl --format=FORMAT list TABLE
```

Giá trị hợp lệ trùng với `ovsdb-client` formatter (man `ovsdb-client(1)`): `table`, `list`, `html`, `csv`, `json`.

#### Mẫu output thực tế

```shell
$ sudo ovs-vsctl --format=csv --columns=name,fail_mode list Bridge
name,fail_mode
br-int,secure
br-ex,standalone

$ sudo ovs-vsctl --format=json --columns=name,fail_mode list Bridge
{"data":[["br-int","secure"],["br-ex","standalone"]],"headings":["name","fail_mode"]}
```

CSV trực tiếp import được Excel/LibreOffice. JSON được consume bởi `jq` hoặc Python `json.loads()`.

#### Vai trò trong vận hành

`--format=csv` cho audit report: dump topology hằng tuần thành CSV, diff giữa các tuần để detect drift. `--format=json` cho dashboard/API: REST controller gọi `ovs-vsctl --format=json` rồi pipe trả về client.

#### Lỗi thường gặp + chẩn đoán

| Triệu chứng | Nguyên nhân khả dĩ | Cách kiểm chứng |
|-------------|---------------------|-----------------|
| `unknown output format` | Gõ sai `--format=Json` (case-sensitive) hoặc `--format=yaml` (không hỗ trợ) | Dùng đúng `json` lowercase |
| JSON một dòng dài khó đọc | JSON mặc định compact | Thêm `--pretty` (xem §9.4.W.5) |
| CSV không escape giá trị có comma | Set/map column có dấu phẩy bên trong | Dùng `--format=json` thay vì CSV cho composite column |

#### Khác biệt giữa các phiên bản

OVS 1.x chỉ có `table` + `list`. OVS 2.0 thêm `csv` + `html`. OVS 2.4 thêm `json`. OVS 2.17 ổn định cả 5 format. Dynamic schema (cột thêm trong runtime) có thể làm CSV header drift giữa hai lần chạy nếu schema upgrade giữa chừng.

#### Source code tham chiếu

Hàm `table_print()` trong `lib/table.c` dispatch sang `table_print_table_text()`, `table_print_csv()`, `table_print_json()`, ... dựa trên enum `output_format`. Parse `--format=` trong `lib/db-ctl-base.c::ctl_parse_options()` map string sang enum.

### 9.4.W.5 `ovs-vsctl --pretty`

#### Khái niệm

`--pretty` áp lên output JSON: thêm newline + indent để dễ đọc bằng mắt người. Không tác dụng với format `table`/`csv`/`list`. Mặc định JSON output là compact (một dòng dài), tối ưu kích thước network/disk; `--pretty` đảo ngược trade-off cho human reading.

#### Cú pháp

```
ovs-vsctl --pretty --format=json list TABLE
```

`--pretty` chỉ có nghĩa khi `--format=json` (hoặc khi default format đẩy ra JSON-like value, ví dụ giá trị column kiểu set trong `list`).

#### Mẫu output thực tế

```shell
$ sudo ovs-vsctl --pretty --format=json --columns=name,fail_mode list Bridge
{
  "data": [
    ["br-int", "secure"],
    ["br-ex", "standalone"]
  ],
  "headings": ["name", "fail_mode"]
}
```

So sánh với compact (đã thấy ở §9.4.W.4): output dài hơn ~3-4x, nhưng đọc tay trong ticket / log file dễ hơn nhiều.

#### Vai trò trong vận hành

Dùng khi debug interactive: dump JSON to file, mở bằng editor, scan bằng mắt. Không dùng trong pipeline production vì tăng kích thước log; dùng `jq .` hoặc Python pretty-print sau khi nhận compact JSON.

#### Lỗi thường gặp + chẩn đoán

| Triệu chứng | Nguyên nhân khả dĩ | Cách kiểm chứng |
|-------------|---------------------|-----------------|
| `--pretty` không tác dụng | Format không phải `json` | Thêm `--format=json` |
| Output vẫn compact dù đã `--pretty` | Một số subcommand non-database (`add-br`, `set-controller`) không trả JSON | `--pretty` chỉ áp cho `list`/`find`/`get` |
| Diff giữa hai dump JSON khác nhau | Một dump pretty, một compact | Chuẩn hóa cả hai về cùng format trước khi diff |

#### Khác biệt giữa các phiên bản

OVS 2.4 thêm `--pretty` cùng với `--format=json`. OVS 2.17 không thay đổi semantic. Tương đương `ovsdb-client --pretty transact ...`.

#### Source code tham chiếu

`json_to_string()` trong `lib/json.c` nhận flag `JSSF_PRETTY` để emit indent + newline; `lib/db-ctl-base.c::ctl_parse_options()` map `--pretty` thành flag này. Gọi qua `table_print_json()` trong `lib/table.c`.

## 9.4.V Mở rộng tham chiếu nhóm option `ovs-ofctl` cho flow-mod + dump-flows (mở rộng 2026-04)

> **Phạm vi mục này.** Năm option global của `ovs-ofctl` ảnh hưởng tới hành vi flow-mod và format output: `--strict`, `--readd`, `--bundle`, `--no-stats`, `--more` (alias `-m`). Các option này đã xuất hiện rải rác trong §9.4.Z (CRUD flow-table) nhưng chưa có section riêng cho từng option. Mục này gom lại với mức độ tham chiếu nhanh: cú pháp, ngữ nghĩa, mẫu output, vai trò vận hành, lỗi thường gặp. Nguồn upstream: man `ovs-ofctl(8)` (file `utilities/ovs-ofctl.8.in`) + code parse trong `utilities/ovs-ofctl.c::parse_options()` long-option array (OVS branch-2.17 baseline 2.17.9). Các flag boolean `strict`, `readd`, `bundle` cùng integer `show_stats`/`verbosity` được khai báo dạng `static` ở đầu file `ovs-ofctl.c`.

---

### 9.4.V.1 `ovs-ofctl --strict`

#### Khái niệm

`--strict` chuyển hành vi `mod-flows` và `del-flows` từ "match wildcard" (default) sang "exact match": chỉ flow nào có **đúng cùng priority + đúng cùng tập match field** với flow_mod request mới bị modify hoặc delete. Không có `--strict`, một flow_mod chứa match `dl_src=00:11:22:33:44:55` sẽ ảnh hưởng tới mọi flow trong table có match field là superset của criteria; với `--strict`, phải khớp tuyệt đối.

#### Cú pháp + ngữ nghĩa

```
ovs-ofctl --strict mod-flows <bridge> "<flow>"
ovs-ofctl --strict del-flows <bridge> "<flow>"
```

Trong code `utilities/ovs-ofctl.c`, biến `bool strict` được set qua long-option `{"strict", no_argument, NULL, OPT_STRICT}`. Hàm `ofctl_flow_mod()` đọc biến này và chọn opcode OpenFlow: `OFPFC_MODIFY_STRICT` (5) thay vì `OFPFC_MODIFY` (1), hoặc `OFPFC_DELETE_STRICT` (4) thay vì `OFPFC_DELETE` (3). Spec OpenFlow 1.3 §6.4 quy định semantic này.

#### Mẫu output thực tế

```shell
$ sudo ovs-ofctl add-flow br-int "priority=100,dl_src=00:11:22:33:44:55,actions=output:1"
$ sudo ovs-ofctl add-flow br-int "priority=100,dl_src=00:11:22:33:44:55,dl_dst=ff:ff:ff:ff:ff:ff,actions=drop"

$ sudo ovs-ofctl del-flows br-int "dl_src=00:11:22:33:44:55"
# (xóa cả 2 flow vì non-strict match wildcard tất cả flow có dl_src=...)

$ sudo ovs-ofctl --strict del-flows br-int "priority=100,dl_src=00:11:22:33:44:55"
# (chỉ xóa flow đầu, flow thứ 2 còn dl_dst nên không strict-match)
```

#### Vai trò trong vận hành

Bắt buộc khi modify/delete flow trong production: không có `--strict`, một del-flow có thể vô tình xóa hàng loạt flow OVN/Neutron khác cùng prefix match field. Pattern an toàn: luôn `dump-flows` trước để xem flow nào sẽ bị ảnh hưởng, sau đó dùng `--strict` để giới hạn phạm vi.

#### Lỗi thường gặp + chẩn đoán

| Triệu chứng | Nguyên nhân khả dĩ | Cách kiểm chứng |
|-------------|---------------------|-----------------|
| `del-flows` xóa nhiều hơn dự kiến | Quên `--strict`, match wildcard mọi flow superset | Luôn `dump-flows` xem trước; thêm `--strict` |
| `--strict del-flows` không xóa gì | Priority không khớp đúng (default 32768 ≠ priority đã add) | Specify priority khớp, hoặc bỏ `--strict` |
| `mod-flows` thay đổi flow ngoài ý muốn | Non-strict modify match wildcard | Dùng `--strict` + đầy đủ match field |

#### Khác biệt giữa các phiên bản

Có sẵn từ OVS 1.x (mapped trực tiếp tới opcode OpenFlow 1.0 `OFPFC_MODIFY_STRICT`/`OFPFC_DELETE_STRICT`). OpenFlow 1.3 § 6.4 vẫn giữ semantic cũ. OVS 2.17 không thay đổi hành vi `--strict`.

#### Source code tham chiếu

`utilities/ovs-ofctl.c` khai báo `static bool strict;` đầu file. Hàm `parse_options()` xử lý `OPT_STRICT` và set `strict = true`. Trong `ofctl_flow_mod()`, opcode được chọn qua biểu thức `strict ? OFPFC_MODIFY_STRICT : OFPFC_MODIFY` (tương tự cho DELETE). Các macro opcode định nghĩa trong `include/openflow/openflow-1.0.h`.

---

### 9.4.V.2 `ovs-ofctl --readd`

#### Khái niệm

`--readd` áp dụng cho subcommand `replace-flows`: thay vì chỉ add flow mới + delete flow cũ không có trong file, nó **re-add toàn bộ** flow trong file ngay cả khi flow giống hệt đã tồn tại trên switch (cùng match + cùng actions + cùng cookie + cùng timeout). Mục đích: reset packet/byte counter của flow về 0.

#### Cú pháp + ngữ nghĩa

```
ovs-ofctl --readd replace-flows <bridge> <flow-file>
```

Trong OpenFlow 1.0/1.1, re-add flow luôn reset counter về 0 (không có cờ điều khiển). Từ OpenFlow 1.2+, behavior phụ thuộc cờ `OFPFF_RESET_COUNTS` trên flow_mod: chỉ reset khi cờ được set. `ovs-ofctl --readd` set cờ này tự động khi protocol >= 1.2.

#### Mẫu output thực tế

```shell
$ sudo ovs-ofctl dump-flows br-int | head -3
 cookie=0x0, duration=120.5s, table=0, n_packets=42, n_bytes=3528, priority=100,dl_src=00:11:22:33:44:55 actions=output:1
 cookie=0x0, duration=85.2s, table=0, n_packets=10, n_bytes=840, priority=50 actions=NORMAL

$ cat /tmp/flows.txt
priority=100,dl_src=00:11:22:33:44:55,actions=output:1
priority=50,actions=NORMAL

$ sudo ovs-ofctl --readd replace-flows br-int /tmp/flows.txt
$ sudo ovs-ofctl dump-flows br-int | head -3
 cookie=0x0, duration=0.3s, table=0, n_packets=0, n_bytes=0, priority=100,dl_src=00:11:22:33:44:55 actions=output:1
 cookie=0x0, duration=0.2s, table=0, n_packets=0, n_bytes=0, priority=50 actions=NORMAL
```

`duration` reset về ~0, `n_packets`/`n_bytes` reset về 0. Không có `--readd`, các flow giống hệt sẽ giữ nguyên counter.

#### Vai trò trong vận hành

Hữu ích khi cần baseline counter mới sau một thay đổi cấu hình (ví dụ: triển khai QoS rule mới và muốn đo từ thời điểm triển khai). Không thay thế `del-flows + add-flows` (vốn nặng hơn) nhưng là cách "soft reset" cho replace-flows pipeline.

#### Lỗi thường gặp + chẩn đoán

| Triệu chứng | Nguyên nhân khả dĩ | Cách kiểm chứng |
|-------------|---------------------|-----------------|
| Counter không reset dù có `--readd` | Protocol >= 1.2 mà controller không set `OFPFF_RESET_COUNTS` | `ovs-ofctl -O OpenFlow13 ...` ép phiên bản; check `dump-flows --more` |
| `replace-flows` xóa flow ngoài ý muốn | `--readd` chỉ ảnh hưởng add path, delete path vẫn xóa flow không có trong file | Dùng `add-flows` thay vì `replace-flows` nếu muốn additive |
| Re-add gây spike OFPT_FLOW_MOD lên controller | File flow lớn, mỗi flow một message | Cân nhắc `--bundle` (xem §9.4.V.3) cho atomic batch |

#### Khác biệt giữa các phiên bản

Cờ `OFPFF_RESET_COUNTS` chính thức từ OpenFlow 1.2 (spec § 5.7.2). OVS 1.x chỉ chạy OF1.0/1.1 nên reset luôn. OVS 2.0+ hỗ trợ OF1.2+ và chỉ reset khi cờ set. OVS 2.17 ổn định hành vi qua mọi protocol version.

#### Source code tham chiếu

`utilities/ovs-ofctl.c` khai báo `static bool readd;`. `OPT_READD` được parse trong `parse_options()`. Hàm `ofctl_replace_flows()` đọc biến này và chuyển vào `ofputil_flow_mod` flags qua `OFPUTIL_FF_RESET_COUNTS`. Mapping cờ trong `lib/ofp-flow.c`.

### 9.4.V.3 `ovs-ofctl --bundle`

#### Khái niệm

`--bundle` đóng gói toàn bộ flow_mod messages của một subcommand thành **một bundle transaction** OpenFlow 1.4: hoặc tất cả flow_mod được apply thành công, hoặc không có flow_mod nào được apply (atomic semantic). Loại trừ trạng thái trung gian khi switch đang xử lý dở danh sách flow.

#### Cú pháp + ngữ nghĩa

```
ovs-ofctl -O OpenFlow14 --bundle add-flows <bridge> <file>
ovs-ofctl -O OpenFlow14 --bundle replace-flows <bridge> <file>
```

Bundle protocol bắt buộc OpenFlow 1.4 trở lên (spec § 7.3.9). `ovs-ofctl` gửi sequence: `OFPT_BUNDLE_CONTROL{OFPBCT_OPEN_REQUEST}` → một loạt `OFPT_BUNDLE_ADD_MESSAGE` (mỗi flow_mod được wrap) → `OFPT_BUNDLE_CONTROL{OFPBCT_COMMIT_REQUEST}`. Switch reply ACK ở mỗi bước; commit fail → toàn bộ bundle rollback.

#### Mẫu output thực tế

Khi quan sát bằng `ovs-ofctl monitor br-int -O OpenFlow14`:

```
OFPT_BUNDLE_CONTROL (OF1.4) (xid=0x2): bundle_id=0x1 type=OPEN_REQUEST
OFPT_BUNDLE_CONTROL (OF1.4) (xid=0x3): bundle_id=0x1 type=OPEN_REPLY
OFPT_BUNDLE_ADD_MESSAGE (OF1.4) (xid=0x4): bundle_id=0x1 flags=atomic
  OFPT_FLOW_MOD (OF1.4): ADD priority=100,dl_src=... actions=output:1
OFPT_BUNDLE_ADD_MESSAGE (OF1.4) (xid=0x5): bundle_id=0x1 flags=atomic
  OFPT_FLOW_MOD (OF1.4): ADD priority=50 actions=NORMAL
OFPT_BUNDLE_CONTROL (OF1.4) (xid=0x6): bundle_id=0x1 type=COMMIT_REQUEST
OFPT_BUNDLE_CONTROL (OF1.4) (xid=0x7): bundle_id=0x1 type=COMMIT_REPLY
```

Không có `--bundle`, mỗi flow_mod là OFPT_FLOW_MOD độc lập.

#### Vai trò trong vận hành

Bắt buộc cho replace-flows production khi pipeline có flow phụ thuộc (ví dụ: table 0 redirect tới table 5, cả hai phải đồng bộ). Không có atomic, traffic giữa lúc nửa pipeline đã update có thể drop hoặc loop. OVN production CMS (kubernetes-ovn, OpenStack neutron-ovn) cũng dùng bundle để đẩy logical_flow batch.

#### Lỗi thường gặp + chẩn đoán

| Triệu chứng | Nguyên nhân khả dĩ | Cách kiểm chứng |
|-------------|---------------------|-----------------|
| `error: unknown OpenFlow version` | Quên `-O OpenFlow14`, `--bundle` cần OF1.4+ | Thêm `-O OpenFlow14` |
| `OFPBFC_BUNDLE_IN_PROGRESS` | Bundle trước chưa commit/discard, switch reject mới | `ovs-appctl bridge/dump-flows` xem state; restart switch nếu deadlock |
| Bundle commit fail giữa chừng | Một flow_mod sai syntax → toàn bộ bundle rollback | Validate file flow trước; check switch log `/var/log/openvswitch/ovs-vswitchd.log` |

#### Khác biệt giữa các phiên bản

Bundle chính thức từ OpenFlow 1.4 (spec § 7.3.9). OVS 2.4 thêm hỗ trợ bundle protocol (commit `66a0712d` 2015). OVS 2.7+ hỗ trợ bundle cho cả OpenFlow 1.3 qua extension Nicira (NXBCT). OVS 2.17 ổn định cho cả OF1.4 native lẫn NX extension.

#### Source code tham chiếu

`utilities/ovs-ofctl.c` khai báo `static bool bundle;`. `OPT_BUNDLE` parse trong `parse_options()`. Hàm `bundle_flow_mod__()` (cùng file) gửi sequence `OFPBCT_OPEN_REQUEST` → loop `OFPT_BUNDLE_ADD_MESSAGE` → `OFPBCT_COMMIT_REQUEST`. Encoding bundle message trong `lib/ofp-bundle.c::ofputil_encode_bundle_ctrl_request()`.

### 9.4.V.4 `ovs-ofctl --no-stats`

#### Khái niệm

`--no-stats` áp lên `dump-flows`/`dump-aggregate`: bỏ các trường `n_packets`, `n_bytes`, `duration`, `idle_age`, `hard_age` khỏi output. Mục tiêu: giảm kích thước output khi chỉ cần xem topology flow (match + actions), không quan tâm counter.

#### Cú pháp + ngữ nghĩa

```
ovs-ofctl --no-stats dump-flows <bridge> [<flow-spec>]
```

Trong code, `static int show_stats = 1;` mặc định bật. Long-option `{"no-stats", no_argument, &show_stats, 0}` set thẳng biến về 0 (pattern getopt_long với target pointer). Hàm `ofputil_flow_stats_format()` đọc biến và skip in các stats field nếu `show_stats == 0`.

#### Mẫu output thực tế

So sánh:

```shell
$ sudo ovs-ofctl dump-flows br-int
 cookie=0x0, duration=125.5s, table=0, n_packets=42, n_bytes=3528, priority=100,dl_src=00:11:22:33:44:55 actions=output:1
 cookie=0x0, duration=90.1s, table=0, n_packets=10, n_bytes=840, priority=50 actions=NORMAL

$ sudo ovs-ofctl --no-stats dump-flows br-int
 cookie=0x0, table=0, priority=100,dl_src=00:11:22:33:44:55 actions=output:1
 cookie=0x0, table=0, priority=50 actions=NORMAL
```

Output `--no-stats` ngắn hơn ~30%, dễ đọc bằng mắt khi chỉ cần verify match/actions; nhưng mất thông tin counter để chẩn đoán flow chết.

#### Vai trò trong vận hành

Hữu ích cho diff topology giữa hai thời điểm: dump cùng `--no-stats` rồi `diff` hai output, tránh false-positive do counter thay đổi liên tục. Cũng dùng khi export flow snapshot để feed vào `add-flows` (file format không nên chứa stats vì input parser không cần).

#### Lỗi thường gặp + chẩn đoán

| Triệu chứng | Nguyên nhân khả dĩ | Cách kiểm chứng |
|-------------|---------------------|-----------------|
| Mất khả năng phát hiện flow đã chết | `--no-stats` ẩn `n_packets=0` | Bỏ `--no-stats` khi debug traffic |
| Không thấy `idle_age` để tìm flow stale | `idle_age`/`hard_age` cũng bị ẩn | Dùng full `dump-flows` rồi grep `idle_age=` |
| Output `--no-stats` vẫn chứa `duration` | Phiên bản OVS cũ trước 2.0 không hide `duration` | Upgrade lên OVS 2.0+ |

#### Khác biệt giữa các phiên bản

`--no-stats` có sẵn từ OVS 1.x. OVS 2.0+ mở rộng để cũng ẩn `idle_age`/`hard_age` (NXST extension fields). Trên OpenFlow 1.5, một số stats field mới (`importance`) cũng được ẩn theo. OVS 2.17 ổn định cho mọi protocol.

#### Source code tham chiếu

`utilities/ovs-ofctl.c`: `static int show_stats = 1;` cùng entry getopt `{"no-stats", no_argument, &show_stats, 0}` (set thẳng pointer khi flag xuất hiện). Hàm `ofctl_dump_flows__()` truyền `show_stats` xuống `ofputil_flow_stats_format()` (`lib/ofp-flow.c`). Logic skip stats field check `if (show_stats) { ... }`.

### 9.4.V.5 `ovs-ofctl --more` (alias `-m`)

#### Khái niệm

`--more` (short `-m`) tăng mức verbosity của output `dump-flows`, `monitor`, và một số subcommand khác: in thêm `cookie`, `priority` đầy đủ (không default ẩn), thông tin `flags`, `hard_timeout`, `idle_timeout`. Lặp lại `--more` (`-mm`, `-mmm`) tăng verbosity cấp cao hơn nữa, ví dụ: in raw OXM TLV của match field thay vì human-friendly text.

#### Cú pháp + ngữ nghĩa

```
ovs-ofctl --more dump-flows <bridge>
ovs-ofctl -mm dump-flows <bridge>            # double verbosity
ovs-ofctl -m monitor <bridge> watch:         # verbose monitor
```

Trong code, `static int verbosity = 0;`. Long-option map về case `'m'`: `case 'm': verbosity++; break;`. Mỗi lần `-m` xuất hiện tăng biến lên 1.

#### Mẫu output thực tế

```shell
$ sudo ovs-ofctl dump-flows br-int
 cookie=0x0, duration=120.5s, table=0, n_packets=42, n_bytes=3528, priority=100,dl_src=00:11:22:33:44:55 actions=output:1

$ sudo ovs-ofctl --more dump-flows br-int
 cookie=0x0, duration=120.5s, table=0, n_packets=42, n_bytes=3528, idle_age=2, hard_age=120, priority=100,dl_src=00:11:22:33:44:55 actions=output:1

$ sudo ovs-ofctl -mm dump-flows br-int
 cookie=0x0, duration=120.5s, table=0, n_packets=42, n_bytes=3528, idle_age=2, hard_age=120, idle_timeout=0, hard_timeout=0, send_flow_rem flags=0x0, priority=100,dl_src=00:11:22:33:44:55 actions=output:1
```

Mỗi mức `-m` thêm trường: level 1 thêm `idle_age`/`hard_age`; level 2 thêm timeout + flags; level 3+ in OXM TLV raw.

#### Vai trò trong vận hành

Cần thiết khi debug flow stale: `idle_age` lớn báo flow không match traffic gần đây. `flags=0x1 send_flow_rem` báo controller sẽ nhận FLOW_REMOVED khi flow expired. `hard_timeout`/`idle_timeout` cho biết flow thực sự là vĩnh viễn hay tạm. Trong production CMS (Neutron, OVN), default flow có timeout=0 (vĩnh viễn); flow nào có timeout != 0 thường là flow do controller tạm tạo.

#### Lỗi thường gặp + chẩn đoán

| Triệu chứng | Nguyên nhân khả dĩ | Cách kiểm chứng |
|-------------|---------------------|-----------------|
| `idle_age` không xuất hiện dù `-m` | OVS quá cũ (< 1.7) chưa support idle_age | `ovs-vsctl --version` |
| `-m` không thay đổi monitor output | `monitor` verbose ở level khác (`watch:flow`) | Kết hợp `-m watch:flow` |
| OXM TLV raw quá dài làm log overflow | `-mmm` in raw bytes cho mỗi match field | Giảm xuống `-m` hoặc `-mm` |

#### Khác biệt giữa các phiên bản

OVS 1.7 thêm `idle_age`/`hard_age` (NXST extension). OVS 2.0+ thêm `-mm` cho timeout + flags. OVS 2.4+ extend `-mmm` cho OXM raw TLV trên OF1.3. OVS 2.17 ổn định cho mọi cấp verbosity.

#### Source code tham chiếu

`utilities/ovs-ofctl.c` khai báo `static int verbosity;`. Long-option `{"more", no_argument, NULL, 'm'}` cùng case `'m': verbosity++;` trong `parse_options()`. Biến `verbosity` được pass xuống `ofputil_flow_stats_format()` (`lib/ofp-flow.c`) qua tham số `show_stats` mở rộng (cùng tên enum), control mức chi tiết. Mỗi field stats có gate `if (verbosity >= N) { ... }`.

## 9.4.O Mở rộng tham chiếu nhóm option `ovs-vsctl` cho heading + idempotency + reference + sync (mở rộng 2026-04)

> **Phạm vi mục này.** Năm option global của `ovs-vsctl` chuyên dùng trong script automation và transaction nhiều bước: `--no-headings` (strip heading row), `--if-exists` (idempotent delete/get), `--may-exist` (idempotent add), `--id=@NAME` (symbolic reference cho row mới tạo trong cùng transaction), và `--no-wait` (không chờ `ovs-vswitchd` reconfigure). Bốn option đầu là khác biệt giữa script "ngây thơ" và script production-grade chịu được rerun + race; option thứ năm cần thiết khi `ovs-vswitchd` không chạy hoặc khi muốn batch nhiều thay đổi rồi để daemon pickup sau. Nguồn upstream: man `ovs-vsctl(8)` SYNOPSIS + DATABASE COMMANDS, parse logic chia hai tầng — option chung (database-related) trong `lib/db-ctl-base.c::ctl_parse_options()`, option vsctl-specific (`--no-wait`) trong `utilities/ovs-vsctl.c::parse_options()`. OVS branch-2.17 baseline 2.17.9.

### 9.4.X.1 `ovs-vsctl --no-headings`

#### Khái niệm

`--no-headings` loại bỏ dòng tiêu đề (heading row) ở đầu output dạng bảng của `list`/`find`. Mặc định, `ovs-vsctl --format=table list Bridge` in heading dạng `name` + thanh ngang `----` rồi mới đến data. Khi script dùng `awk`/`cut` để parse, heading row làm sai logic vì nó không phải data thực. Man page ghi: *"This option suppresses the heading row that otherwise appears in the first row of table output."*

#### Cú pháp

```
ovs-vsctl --no-headings [--columns=COL,...] [--format=FORMAT] list TABLE
ovs-vsctl --no-headings [--columns=COL,...] find TABLE [COL=VAL]...
```

`--no-headings` áp dụng cho mọi format có heading (`table`, `csv`, `html`, `json`). Khác `--bare`: `--bare` strip heading + decoration (padding, quote, separator); `--no-headings` chỉ strip heading row, giữ lại format căn cột. Nói cách khác: `--bare` ⊃ `--no-headings` về effect.

#### Mẫu output thực tế

So sánh ba mức:

```shell
$ sudo ovs-vsctl --columns=name,fail_mode --format=csv list Bridge
name,fail_mode
br-int,secure
br-ex,standalone

$ sudo ovs-vsctl --no-headings --columns=name,fail_mode --format=csv list Bridge
br-int,secure
br-ex,standalone

$ sudo ovs-vsctl --bare --columns=name,fail_mode list Bridge
br-int secure
br-ex standalone
```

`--no-headings` giữ format CSV (dấu phẩy phân tách) nhưng bỏ dòng `name,fail_mode`. `--bare` loại cả CSV separator, in mỗi cell cách nhau bằng tab/space. Với spreadsheet script: `--no-headings --format=csv` hợp lý nhất; với shell `read -r`: `--bare` nhanh hơn.

#### Vai trò trong vận hành

`--no-headings` xuất hiện chủ yếu trong audit script tạo CSV cho long-term archive: heading row bỏ ra để concatenate output qua nhiều ngày không bị nhân heading lên. Cũng dùng khi feed output vào `awk -F,` mà không muốn skip dòng đầu thủ công.

#### Lỗi thường gặp + chẩn đoán

| Triệu chứng | Nguyên nhân khả dĩ | Cách kiểm chứng |
|-------------|---------------------|-----------------|
| Output vẫn có heading | Dùng default `--format=table` thì OVS một số version vẫn in `----` separator | Kết hợp `--format=csv` hoặc dùng `--bare` |
| `awk` đếm sai số bridge | Quên `--no-headings`, `awk` count cả heading row | `ovs-vsctl --no-headings --columns=name list Bridge \| wc -l` |
| Khi merge CSV nhiều ngày, heading lặp | Một số file dump không có `--no-headings` | Chuẩn hoá: tất cả script dump dùng `--no-headings` |

#### Khác biệt giữa các phiên bản

Có sẵn từ OVS 1.x (cùng thời điểm với `--columns`/`--bare`). OVS 2.0+ extend cho format `csv`/`html`/`json`. OVS 2.17 không thay đổi semantic.

#### Source code tham chiếu

Flag `--no-headings` set biến `headings = false` trong `lib/db-ctl-base.c::ctl_parse_options()` (case `OPT_NO_HEADINGS`). Khi format output, `lib/table.c::table_print()` check biến này: nếu `false`, skip header row khi gọi `table_print_table_text()`/`table_print_csv()`/`table_print_html()`/`table_print_json()`.

### 9.4.X.2 `ovs-vsctl --if-exists`

#### Khái niệm

`--if-exists` biến lệnh thao tác trên record không tồn tại từ "lỗi" thành "no-op". Áp dụng cho `del-br`, `del-port`, `del-controller`, `del-manager`, `del-fail-mode`, `remove`, `clear`, `get`, `destroy`, `--id=@x get`. Man page ghi: *"Without --if-exists, attempting to delete a bridge that does not exist is an error. With --if-exists, attempting to delete a bridge that does not exist has no effect."* Đây là thuộc tính idempotent — nền tảng cho automation chịu được rerun.

#### Cú pháp

```
ovs-vsctl --if-exists del-br BRIDGE
ovs-vsctl --if-exists del-port [BRIDGE] PORT
ovs-vsctl --if-exists get TABLE RECORD COLUMN
ovs-vsctl --if-exists remove TABLE RECORD COLUMN VALUE
ovs-vsctl --if-exists clear TABLE RECORD COLUMN
```

`--if-exists` phải đứng TRƯỚC subcommand cần áp dụng. Khi chain nhiều subcommand bằng `--`, cờ này áp dụng cho mọi subcommand sau nó (global flag), trừ khi reset.

#### Mẫu output thực tế

```shell
$ sudo ovs-vsctl del-br br-test
ovs-vsctl: no bridge named br-test
$ echo $?
1

$ sudo ovs-vsctl --if-exists del-br br-test
$ echo $?
0
```

Lệnh đầu fail vì `br-test` không có; lệnh hai exit 0 dù bridge không tồn tại — đúng ngữ nghĩa "ensure absent".

#### Vai trò trong vận hành

Quan trọng nhất trong cleanup script + reconciliation loop:

```shell
for br in br-test1 br-test2 br-test3; do
  sudo ovs-vsctl --if-exists del-br "$br"
done
```

Vòng lặp này chạy được dù chỉ một số bridge tồn tại, không cần check trước. Trong CMS như Neutron/OVN, agent thường rerun cùng một thao tác cleanup nhiều lần khi reconcile state — không có `--if-exists` thì agent log đầy "no bridge named" gây nhiễu monitoring.

#### Lỗi thường gặp + chẩn đoán

| Triệu chứng | Nguyên nhân khả dĩ | Cách kiểm chứng |
|-------------|---------------------|-----------------|
| `--if-exists` đặt sai chỗ | Đặt sau subcommand thay vì trước | Đặt ngay sau `ovs-vsctl` hoặc trước verb |
| Vẫn fail với "no row to delete" | Subcommand không support `--if-exists` (ví dụ `set` cho non-existent record) | Dùng `--may-exist add-br` rồi `set` |
| Race với agent khác xoá đồng thời | Idempotent nhưng không atomic giữa nhiều client | `--if-exists` + retry loop, hoặc OVSDB transaction qua API |

#### Khác biệt giữa các phiên bản

Có sẵn từ OVS 1.x cho `del-*` subcommand. OVS 2.0+ mở rộng cho `get`/`remove`/`clear`/`destroy`. OVS 2.6+ thêm support cho `--id=@x get` không tồn tại. OVS 2.17 ổn định toàn diện.

#### Source code tham chiếu

Flag set biến `if_exists = true` trong `lib/db-ctl-base.c::ctl_parse_options()` (case `OPT_IF_EXISTS`). Mỗi subcommand handler check biến này trước khi raise error: ví dụ `cmd_del_br()` trong `utilities/ovs-vsctl.c` có pattern `if (must_exist || (br && !ctx->options & ...)) { ... ctl_fatal(...) }`. Tổng quát: nếu record không tìm thấy + `if_exists`, handler return early không exit lỗi.

### 9.4.X.3 `ovs-vsctl --may-exist`

#### Khái niệm

`--may-exist` biến lệnh tạo record đã tồn tại từ "lỗi" thành "no-op". Áp dụng cho `add-br`, `add-port`, `add-bond`, `add-bond-iface`. Man page ghi: *"Without --may-exist, attempting to create a bridge that exists is an error. With --may-exist, this command does nothing if bridge already exists as a real bridge."* Đây là counterpart của `--if-exists`: cùng nhau cover hai chiều idempotent (ensure-present + ensure-absent).

#### Cú pháp

```
ovs-vsctl --may-exist add-br BRIDGE [PARENT VLAN]
ovs-vsctl --may-exist add-port BRIDGE PORT [-- set Interface PORT type=...]
ovs-vsctl --may-exist add-bond BRIDGE BOND IFACE...
```

Khi record đã tồn tại VÀ thuộc tính trùng khớp, `--may-exist` skip hoàn toàn. Nếu record tồn tại NHƯNG thuộc tính khác (ví dụ `add-br br0 br0-parent 100` trong khi `br0` là real bridge không VLAN), lệnh vẫn fail vì đặc tả không khớp — đây là semantic chính xác, không phải bug.

#### Mẫu output thực tế

```shell
$ sudo ovs-vsctl add-br br-int
ovs-vsctl: cannot create a bridge named br-int because a bridge named br-int already exists
$ echo $?
1

$ sudo ovs-vsctl --may-exist add-br br-int
$ echo $?
0
```

Idempotent rerun của configuration script là pattern phổ biến nhất với option này.

#### Vai trò trong vận hành

Trong Ansible/Puppet/Salt module cho OVS, mọi `add-br`/`add-port` đều kèm `--may-exist`. Script bootstrap node compute (Neutron/OVN) chạy hàng trăm lần lifecycle (boot, reboot, reconfig); không có `--may-exist` thì lần thứ hai trở đi đều exit non-zero và stop pipeline.

#### Lỗi thường gặp + chẩn đoán

| Triệu chứng | Nguyên nhân khả dĩ | Cách kiểm chứng |
|-------------|---------------------|-----------------|
| `--may-exist add-br` vẫn fail | Bridge tồn tại với parent/VLAN khác đặc tả | `ovs-vsctl br-to-parent BR; ovs-vsctl br-to-vlan BR` so sánh |
| `add-port` skipped nhưng port type sai | `--may-exist` skip nếu port name khớp, không kiểm `Interface.type` | Sau `add-port`, luôn `set Interface NAME type=...` để force config |
| Race giữa hai script cùng add | OVS commit serialize qua RAFT/standalone, không cần lock thủ công | OK, chỉ cần `--may-exist` |

#### Khác biệt giữa các phiên bản

Có sẵn từ OVS 1.x cho `add-br`/`add-port`. OVS 2.0+ mở rộng cho `add-bond`. OVS 2.6+ thêm `add-bond-iface`. OVS 2.17 không thay đổi.

#### Source code tham chiếu

Flag set biến `may_exist = true` trong `lib/db-ctl-base.c::ctl_parse_options()` (case `OPT_MAY_EXIST`). Handler `cmd_add_br()` trong `utilities/ovs-vsctl.c` check: nếu bridge tồn tại + thuộc tính khớp + `may_exist`, return success không tạo mới. Logic so khớp đặc tả ở `find_bridge()` cùng module: name + parent + VLAN tag tuple.

### 9.4.X.4 `ovs-vsctl --id=@NAME`

#### Khái niệm

`--id=@NAME` gán symbolic name cho UUID của row được tạo trong cùng giao tác `ovs-vsctl`, để các subcommand sau (cùng invocation, chain bằng `--`) tham chiếu được. Man page ghi: *"If @name is specified, then the UUID for record may be referred to by that name later in the same ovs-vsctl invocation in contexts where a UUID is expected."* Đây là kỹ thuật cốt lõi cho multi-step transaction: tạo Mirror + tham chiếu Port + Interface trong một lệnh atomic, hoặc tạo QoS + Queue + gán cho Port.

#### Cú pháp

```
ovs-vsctl -- --id=@NAME create TABLE COL=VAL... [-- <next-cmd referencing @NAME>]...
```

`@NAME` là identifier ngắn (ví dụ `@m`, `@q0`, `@iface-eth0`). Trong subcommand sau, dùng `@NAME` ở bất kỳ vị trí nào yêu cầu UUID. Tất cả `@NAME` trong cùng invocation phải unique. Toàn bộ chain commit như một OVSDB transaction: nếu một step fail, không có row nào được tạo.

#### Mẫu output thực tế

Tạo Mirror tham chiếu Port nguồn + Port output trong một lệnh:

```shell
$ sudo ovs-vsctl \
    -- --id=@p0 get Port eth0 \
    -- --id=@p1 get Port eth1 \
    -- --id=@m create Mirror name=m0 select-src-port=@p0 output-port=@p1 \
    -- set Bridge br-int mirrors=@m
6f8b1c9a-4d2e-4b8f-9c1a-0f3e5d7c8b91
```

UUID in ra cuối là UUID của Mirror mới tạo (`@m`). Bridge `br-int` đã update trường `mirrors` chứa UUID này. Không có `--id=@`, phải tạo Mirror trước, lấy UUID, rồi run lệnh thứ hai để gán — không atomic, có race.

#### Vai trò trong vận hành

Mọi config QoS + Queue + Mirror + multi-row đều cần pattern này:

- QoS: `--id=@q create Queue ... -- --id=@qos create QoS queues:0=@q -- set Port eth0 qos=@qos`
- Mirror SPAN cho debugging: pattern ở mẫu trên
- Multi-controller setup: tạo nhiều Controller row + gán list vào Bridge.controller

Bỏ `--id=@`, phải dùng nhiều invocation `ovs-vsctl` riêng, mất atomicity, có thể bị agent khác xen kẽ.

#### Lỗi thường gặp + chẩn đoán

| Triệu chứng | Nguyên nhân khả dĩ | Cách kiểm chứng |
|-------------|---------------------|-----------------|
| `unknown symbol "@m"` | Quên `--id=@m` ở subcommand định nghĩa | Liệt kê chain: mỗi `@NAME` phải có `--id=` trước khi tham chiếu |
| `multiple definitions of @m` | Hai subcommand cùng dùng `--id=@m` | Đặt tên unique: `@m1`, `@m2` |
| Symbol resolve về UUID sai | Order subcommand sai (tham chiếu trước khi định nghĩa) | OVS 2.x có forward reference; OVS 1.x phải định nghĩa trước. Check version |
| Transaction commit nhưng row không xuất hiện | `create` mà không gán symbol vào parent row → orphan | Luôn chain `-- set Parent ref=@NAME` |

#### Khác biệt giữa các phiên bản

OVS 1.x yêu cầu `--id=` định nghĩa trước khi reference (forward reference fail). OVS 2.0+ cho phép forward reference: subcommand dùng `@m` rồi `--id=@m create ...` xuất hiện sau, parser resolve hai chiều. OVS 2.17 ổn định forward + backward reference.

#### Source code tham chiếu

`lib/db-ctl-base.c::parse_command()` xử lý `--id=@NAME`, lưu mapping name → UUID placeholder vào `struct ctl_command_syntax`. Khi chain commit, `do_vsctl()` (`utilities/ovs-vsctl.c`) iterate two-pass: pass 1 collect mọi `--id=@`, pass 2 resolve reference + commit OVSDB transaction qua `ovsdb_idl_txn_commit_block()`. Nếu một step fail, transaction rollback toàn bộ.

### 9.4.X.5 `ovs-vsctl --no-wait`

#### Khái niệm

`--no-wait` cho `ovs-vsctl` exit ngay sau khi commit transaction OVSDB, không đợi `ovs-vswitchd` xử lý xong thay đổi (reconfigure bridge, install datapath flow, mở port mới). Mặc định, `ovs-vsctl` đợi `ovs-vswitchd` set xong trường `next_cfg` ≥ giá trị `cur_cfg` mới in ra; với `--no-wait`, exit ngay sau OVSDB ack. Man page ghi: *"Prevents ovs-vsctl from waiting for ovs-vswitchd to reconfigure itself according to the modified database. This option should be used if ovs-vswitchd is not running."*

#### Cú pháp

```
ovs-vsctl --no-wait <any-config-changing-command> [-- <more>]...
```

Áp dụng cho mọi subcommand thay đổi DB: `add-br`, `add-port`, `set`, `remove`, `clear`, `del-*`. Không có ý nghĩa với pure-read như `list`/`get`/`show`.

#### Mẫu output thực tế

Khác biệt rõ nhất khi `ovs-vswitchd` không chạy:

```shell
$ sudo systemctl stop ovs-vswitchd
$ sudo ovs-vsctl add-br br-x
^C  # treo vô hạn vì đợi vswitchd ack
$ sudo ovs-vsctl --no-wait add-br br-x
$
```

Khi vswitchd chạy bình thường, khác biệt là 50-200 ms (thời gian vswitchd áp config). Trong batch script add 100 port, accumulation 100 × 100 ms = 10 giây — `--no-wait` cắt xuống còn vài trăm ms tổng.

#### Vai trò trong vận hành

Hai use case chính:

1. **Bootstrap khi `ovs-vswitchd` chưa chạy**: script init/cloud-init tạo bridge trước khi service start; default sẽ treo, `--no-wait` cho qua.
2. **Batch nhiều thay đổi nhỏ**: thay vì wait reconfigure cho mỗi `set`, dùng `--no-wait` cho từng lệnh và cuối batch dùng một lệnh không `--no-wait` để sync. Hoặc dùng `--` chain để gộp thành một transaction (preferred).

Trade-off: client không biết khi nào datapath thực sự apply. Test ngay sau `--no-wait` có thể thấy port chưa lên. Trong CI lab, kết hợp với poll: `until ovs-vsctl get Interface eth0 ofport > /dev/null; do sleep 0.1; done`.

#### Lỗi thường gặp + chẩn đoán

| Triệu chứng | Nguyên nhân khả dĩ | Cách kiểm chứng |
|-------------|---------------------|-----------------|
| Test ngay sau `add-port` thấy port chưa active | `--no-wait` bỏ qua reconfig sync | Dùng `--may-exist` không `--no-wait`, hoặc poll `ofport != []` |
| `ovs-vsctl` treo dù vswitchd chạy | vswitchd lock contention hoặc upcall storm | `ovs-appctl ovs/vswitchd/version`; nếu treo cả appctl, vswitchd kẹt thật |
| `--no-wait` không tăng tốc batch | Mỗi lệnh là một invocation riêng — overhead chính ở fork+OVSDB connect | Gộp `-- chain ...` thành một invocation thay vì spam `ovs-vsctl` lặp |

#### Khác biệt giữa các phiên bản

Có sẵn từ OVS 1.x. OVS 2.0+ tinh chỉnh ngữ nghĩa wait: trước OVS 2.0, wait dùng `cur_cfg`/`next_cfg` polling 100 ms; OVS 2.0+ sử dụng OVSDB notification, latency wait giảm xuống ~10 ms khi `ovs-vswitchd` healthy. OVS 2.17 ổn định. Không liên quan đến `--timeout` (timeout đo wall time, no-wait đo logical sync — hai khái niệm độc lập).

#### Source code tham chiếu

Khác bốn option trên (chung qua `lib/db-ctl-base.c`), `--no-wait` là vsctl-specific: parse trong `utilities/ovs-vsctl.c::parse_options()` set biến `wait_for_reload = false`. Sau khi `ovsdb_idl_txn_commit_block()` thành công, hàm `do_vsctl()` check biến này: nếu `true`, gọi loop `idl_run + poll cur_cfg vs next_cfg`; nếu `false` (no-wait), return ngay. Trường `cur_cfg`/`next_cfg` thuộc bảng `Open_vSwitch` (singleton row).

## 9.4.U Mở rộng tham chiếu nhóm subcommand generic theo bảng OVSDB (mở rộng 2026-04)

> **Phạm vi mục này.** `ovs-vsctl` cung cấp một bộ subcommand generic CRUD áp dụng đồng nhất lên mọi bảng OVSDB: `list`, `get`, `set`, `add`, `remove`, `clear`, `find`, `create`, `destroy`. Bộ này hiện thực trong `lib/db-ctl-base.c` (hàm `cmd_list`, `cmd_get`, `cmd_set`, `cmd_add`, `cmd_remove`, `cmd_clear`, `cmd_find`, `cmd_create`, `cmd_destroy`) và được `utilities/ovs-vsctl.c` đăng ký qua `vsctl_cmd_init()`. Bộ subcommand cụ thể chuyên dụng (`add-br`, `set-controller`, ...) chỉ là wrapper tiện lợi cho một số trường hợp thường gặp; engineer thành thạo phải nắm chắc bộ generic vì bảng OVSDB như `Manager`, `QoS`, `Queue`, `Mirror`, `NetFlow`, `IPFIX` không có wrapper. Mục này gom 5 nhóm subcommand generic theo từng bảng phổ biến nhất: `Bridge`, `Port`, `Interface`, `Controller`, `SSL`. Khác §9.4.Y (đi sâu từng lệnh `list Bridge`, `list Port`, ...), mục này tóm tắt cả họ CRUD áp dụng lên bảng đó để engineer có cái nhìn tổng quát "mọi việc làm được trên bảng X". Nguồn upstream: `lib/db-ctl-base.c`, `utilities/ovs-vsctl.c`, schema `vswitchd/vswitch.ovsschema`, man `ovs-vsctl(8)` §`Database Commands`. Baseline: OVS branch-2.17 (2.17.9 trên Ubuntu Jammy).

### 9.4.U.1 Họ subcommand thao tác bảng `Bridge`

Bảng `Bridge` (singleton-ish: nhiều row, mỗi row tên unique) lưu mọi bridge OVS đã cấu hình, được `Open_vSwitch.bridges` tham chiếu kiểu `set of UUID`. Họ generic gồm 9 lệnh sau khi áp lên `Bridge`:

| Lệnh | Cú pháp tiêu biểu | Vai trò |
|------|-------------------|---------|
| `list Bridge` | `ovs-vsctl list Bridge [BRIDGE]` | Dump toàn bộ cột của một/tất cả row |
| `get Bridge` | `ovs-vsctl get Bridge BRIDGE COLUMN[:KEY]` | Lấy giá trị một cột (parse-friendly) |
| `set Bridge` | `ovs-vsctl set Bridge BRIDGE COLUMN[:KEY]=VAL` | Ghi giá trị một cột; idempotent |
| `add Bridge` | `ovs-vsctl add Bridge BRIDGE COLUMN VAL` | Thêm phần tử vào cột kiểu `set`/`map` |
| `remove Bridge` | `ovs-vsctl remove Bridge BRIDGE COLUMN VAL` | Bỏ phần tử khỏi `set`/`map` |
| `clear Bridge` | `ovs-vsctl clear Bridge BRIDGE COLUMN` | Reset cột về default |
| `find Bridge` | `ovs-vsctl find Bridge COLUMN=VAL` | Filter row theo điều kiện |
| `create Bridge` | `ovs-vsctl create Bridge name=br0 ...` | Tạo row trực tiếp (hiếm dùng, ưu tiên `add-br`) |
| `destroy Bridge` | `ovs-vsctl destroy Bridge BRIDGE` | Xóa row (ưu tiên `del-br` an toàn hơn) |

Trong vận hành thực tế, `add-br`/`del-br` là wrapper an toàn của `create Bridge`/`destroy Bridge`: `add-br` thêm UUID mới vào `Open_vSwitch.bridges` đồng thời tạo row, `create Bridge` đơn lẻ tạo row "mồ côi" không tham chiếu — bridge sẽ không xuất hiện trong `ovs-vsctl show`. Vì vậy quy tắc vận hành: dùng wrapper khi có (`add-br`, `del-br`); chỉ rơi xuống generic khi cần atomic chain với `--id=@` (xem §9.4.X.4).

Một số lỗi điển hình: dùng `set Bridge X protocols=OpenFlow13,OpenFlow15` đặt set thay vì `add Bridge X protocols OpenFlow15` thêm vào set hiện tại — `set` ghi đè toàn bộ cột, có thể loại bỏ version cũ; `clear Bridge X protocols` đưa về default (`OpenFlow10`) gây drop session với controller dùng OF1.3+. Với cột map (`other_config`, `external_ids`), `set` ghi đè cả map còn `set Bridge X external_ids:key=val` chỉ sửa một entry. Hàm `cmd_set()`/`cmd_add()`/`cmd_remove()`/`cmd_clear()` trong `lib/db-ctl-base.c` (~line 1240-1340) phân biệt qua `parse_column_key_value()`. Schema bảng `Bridge` ở `vswitchd/vswitch.ovsschema` định nghĩa ~24 cột (OVS 2.17), trong đó `name`, `datapath_type`, `fail_mode`, `protocols`, `controller`, `ports`, `mirrors`, `flood_vlans`, `auto_attach`, `other_config`, `external_ids`, `status`, `stp_*`, `rstp_*`, `mcast_snooping_enable`, `ipfix`, `netflow`, `sflow` là những cột thường thao tác. So với Cisco IOS dùng config-block, OVSDB CRUD generic + transaction giúp script automation deterministic + atomic ở mức row.

Mẫu lệnh thực tế cho một số use case:

```shell
$ sudo ovs-vsctl find Bridge fail_mode=secure
$ sudo ovs-vsctl add Bridge br-int protocols OpenFlow15
$ sudo ovs-vsctl set Bridge br-int other_config:disable-in-band=true
$ sudo ovs-vsctl --columns=name,protocols,fail_mode list Bridge
```

Bốn lệnh trên là khung sườn cho daily ops: tra danh sách bridge secure, nâng giao thức OpenFlow, vô hiệu in-band controller path (tăng độ ổn định khi controller cùng host), và tóm tắt 3 cột quan trọng nhất khi audit cluster.

### 9.4.U.2 Họ subcommand thao tác bảng `Port`

Bảng `Port` đại diện cho cổng logic gắn vào bridge, tham chiếu từ `Bridge.ports` kiểu `set of UUID`, và mỗi `Port` lại có `Port.interfaces` kiểu set-of-Interface (cho phép một Port chứa nhiều Interface, dùng cho bond). Họ subcommand generic áp lên `Port`:

| Lệnh | Cú pháp tiêu biểu | Vai trò |
|------|-------------------|---------|
| `list Port` | `ovs-vsctl list Port [PORT]` | Dump toàn bộ cột của port |
| `get Port` | `ovs-vsctl get Port PORT tag` | Lấy VLAN/trunk/qos của port |
| `set Port` | `ovs-vsctl set Port PORT tag=100 vlan_mode=access` | Cấu hình VLAN access/trunk |
| `add Port` | `ovs-vsctl add Port PORT trunks 20` | Thêm VLAN vào trunks set |
| `remove Port` | `ovs-vsctl remove Port PORT trunks 20` | Bỏ VLAN khỏi trunks |
| `clear Port` | `ovs-vsctl clear Port PORT qos` | Detach QoS row khỏi port |
| `find Port` | `ovs-vsctl find Port tag=100` | Tìm port theo VLAN access |
| `create Port` | `ovs-vsctl create Port name=p1 interfaces=...` | Hiếm dùng, chỉ trong `--id=@` chain |
| `destroy Port` | `ovs-vsctl destroy Port PORT` | Hiếm dùng, ưu tiên `del-port` |

Wrapper an toàn là `add-port`/`del-port`: tự xử lý tạo Interface tương ứng (mặc định type `system`), gắn UUID Port vào `Bridge.ports`. `create Port` thuần không tự tạo Interface, không tự tham chiếu — dẫn đến row mồ côi. Trong scenario QoS, lệnh quan trọng nhất là `set Port PORT qos=@qos` kết hợp `--id=@` (xem §9.4.X.4) để gắn QoS row mới tạo: nếu `clear Port` được gọi mà không `destroy QoS` trước, OVSDB chỉ detach reference, QoS row thành rác.

Với trunk port, đừng nhầm `set Port X trunks=10,20,30` (ghi đè toàn bộ) với `add Port X trunks 10` (thêm vào set hiện tại). Lỗi này thường gặp khi script monitoring chỉ muốn "thêm VLAN 30" lại vô tình xóa 10/20. Lệnh `find Port external_ids:iface-id=lsp-vm-001` (parse map key) là pattern bắt buộc để tra Port_Binding ↔ OVS Port khi debug OVN. Hàm `cmd_list()`/`cmd_get()`/`cmd_set()` chung pipeline `lib/db-ctl-base.c`, schema bảng `Port` ở `vswitchd/vswitch.ovsschema` với ~16 cột (`name`, `interfaces`, `tag`, `trunks`, `vlan_mode`, `qos`, `mac`, `bond_*`, `cvlans`, `external_ids`, `other_config`, ...). So với Linux bridge `bridge vlan add` thuần kernel-level, `ovs-vsctl set Port` là database-driven: lưu trên đĩa, sống qua reboot, replicable.

Mẫu lệnh thực tế:

```shell
$ sudo ovs-vsctl set Port veth-vm1 tag=100
$ sudo ovs-vsctl add Port bond0 trunks 10 -- add Port bond0 trunks 20
$ sudo ovs-vsctl find Port qos!=[]
$ sudo ovs-vsctl --columns=name,tag,trunks list Port
```

Lệnh thứ hai dùng cú pháp `--` chain để thêm hai VLAN trong một transaction; cách này an toàn hơn `set Port bond0 trunks=10,20` vì không phụ thuộc giá trị cũ.

### 9.4.U.3 Họ subcommand thao tác bảng `Interface`

Bảng `Interface` đại diện cho netdev L2 thực tế (hoặc tunnel/vxlan/internal), tham chiếu từ `Port.interfaces`. Đặc điểm: `Interface` chứa cả config (`type`, `options`) lẫn telemetry runtime (`statistics`, `status`, `link_state`, `error`). Họ subcommand generic:

| Lệnh | Cú pháp tiêu biểu | Vai trò |
|------|-------------------|---------|
| `list Interface` | `ovs-vsctl list Interface [IFACE]` | Dump cả config + telemetry |
| `get Interface` | `ovs-vsctl get Interface IFACE ofport` | Lấy ofport assigned (key cho action `output:N`) |
| `set Interface` | `ovs-vsctl set Interface IFACE type=internal mtu_request=9000` | Cấu hình type, MTU, ingress policing |
| `add Interface` | `ovs-vsctl add Interface IFACE options key val` | Thêm tunnel option (rare) |
| `remove Interface` | `ovs-vsctl remove Interface IFACE options key val` | Bỏ option |
| `clear Interface` | `ovs-vsctl clear Interface IFACE mtu_request` | Reset MTU về kernel default |
| `find Interface` | `ovs-vsctl find Interface type=geneve` | Filter theo type/error |
| `create Interface` | `ovs-vsctl create Interface name=tap0 type=internal` | Trong `--id=@` chain |
| `destroy Interface` | Hiếm gọi trực tiếp | Wrapper `del-port` lo |

Khi gắn Interface kiểu tunnel: `ovs-vsctl set Interface vxlan0 type=vxlan options:remote_ip=10.0.0.2 options:key=flow`; cột `options` là map nên dùng `key=val` chứ không phải set. `find Interface error!=""` là pattern troubleshoot điển hình: liệt kê mọi interface có lỗi khởi tạo (ví dụ tunnel dst unreachable). `find Interface admin_state=down` cho biết port nào đang shutdown manually qua `ip link set down`. `get Interface IFACE ofport` quan trọng cho automation: ofport là số kernel-assigned dùng trong action OpenFlow; `set Interface IFACE ofport_request=N` cho phép cố định ofport (OVS 2.0+).

Cột `statistics` map<string,int> chứa rx_bytes/rx_packets/tx_bytes/tx_dropped do `vswitchd/bridge.c::iface_refresh_stats()` cập nhật mỗi vòng `bridge_run()` qua netlink `rtnl_link_stats64`; do đó `list Interface IFACE` cho cả runtime telemetry mà không cần `ip -s link show`. Cột `status` map populate bởi datapath provider (`netdev-vport.c` cho tunnel, `netdev-linux.c` cho NIC). Lỗi điển hình: dùng `set Interface IFACE statistics:rx_bytes=0` cố reset counter; thao tác này không có hiệu lực vì OVS ghi đè ngay vòng lặp tiếp theo. Hàm xử lý chung `lib/db-ctl-base.c::cmd_*()`. Schema `Interface` ở `vswitchd/vswitch.ovsschema` với ~28 cột.

Mẫu lệnh thực tế:

```shell
$ sudo ovs-vsctl find Interface error!=""
$ sudo ovs-vsctl get Interface tap-vm1 ofport
$ sudo ovs-vsctl set Interface vxlan0 type=vxlan options:remote_ip=10.0.0.2 options:key=flow
$ sudo ovs-vsctl --columns=name,type,ofport,error list Interface
```

Lệnh đầu rất quan trọng cho on-call: liệt kê mọi interface ở trạng thái lỗi (tunnel dst unreachable, type không hỗ trợ, MTU mismatch). Lệnh thứ ba dùng cột `options` kiểu map, chú ý cú pháp `key=val` chứ không phải set list.

### 9.4.U.4 Họ subcommand thao tác bảng `Controller`

Bảng `Controller` lưu cấu hình OpenFlow controller cho bridge, tham chiếu từ `Bridge.controller` kiểu `set of UUID` (cho phép multi-controller OF1.2+). Khác bảng Bridge/Port/Interface (có wrapper riêng phổ biến), thao tác `Controller` gần như luôn đi qua wrapper `set-controller`/`del-controller`/`get-controller` (xem §9.4.X.3) — generic CRUD ít gặp hơn nhưng vẫn cần khi điều chỉnh attribute không có wrapper.

| Lệnh | Cú pháp tiêu biểu | Vai trò |
|------|-------------------|---------|
| `list Controller` | `ovs-vsctl list Controller` | Dump mọi controller row, xem `is_connected`, `role` |
| `get Controller` | `ovs-vsctl get Controller UUID is_connected` | Lấy trạng thái kết nối live |
| `set Controller` | `ovs-vsctl set Controller UUID max_backoff=5000 inactivity_probe=10000` | Tinh chỉnh reconnect/keepalive |
| `add Controller` | Rare | Dành cho cột `external_ids` map |
| `remove Controller` | Rare | Đối xứng `add` |
| `clear Controller` | `ovs-vsctl clear Controller UUID role` | Reset role về `other` (OF1.2 default) |
| `find Controller` | `ovs-vsctl find Controller is_connected=false` | Liệt kê controller mất kết nối |
| `create Controller` | `ovs-vsctl create Controller target=tcp:..` | Trong `--id=@` chain multi-controller |
| `destroy Controller` | Rare | `del-controller` lo qua wrapper |

Use case quan trọng nhất là `find Controller is_connected=false` — script monitoring cho biết bridge nào đang đứt kết nối controller; cho production cluster, một dòng `ovs-vsctl --columns=target,is_connected list Controller` đưa vào Prometheus textfile collector phát hiện sớm controller flap. Cột tinh chỉnh thường thao tác qua `set`: `max_backoff` (ms, default 8000) trần exponential backoff khi reconnect, `inactivity_probe` (ms, default 5000) thời gian gửi echo request nếu không có traffic, `connection_mode` (`in-band`/`out-of-band`) ảnh hưởng thiết lập route tới controller. Tăng `max_backoff` lên 30000 khi controller đang upgrade rolling để OVS không spam reconnect.

Schema `Controller` ở `vswitchd/vswitch.ovsschema` (~13 cột). Cột read-only `is_connected`, `status`, `role` được `vswitchd` cập nhật mỗi vòng `connmgr_run()` (`ofproto/connmgr.c`). Lỗi điển hình: `set Controller UUID target=tcp:newhost:6653` thay đổi target mà không restart kết nối ngay; phải `ovs-appctl ofproto/reconnect br-int` mới ép reconnect. Multi-controller OF1.2+ dùng `--id=@c1 create Controller target=tcp:.. role=master ... -- --id=@c2 create Controller target=tcp:.. role=slave ... -- set Bridge br-int controller=@c1,@c2`. So với NETCONF/OpenConfig dùng XML/YANG để config controller, OVSDB-Controller table flat schema dễ script hơn nhưng cần biết generic CRUD.

Mẫu lệnh thực tế:

```shell
$ sudo ovs-vsctl find Controller is_connected=false
$ sudo ovs-vsctl --columns=target,is_connected,role list Controller
$ uuid=$(sudo ovs-vsctl --bare --columns=_uuid find Controller target='"tcp:127.0.0.1:6653"')
$ sudo ovs-vsctl set Controller "$uuid" max_backoff=15000 inactivity_probe=10000
```

Khối shell trên minh họa pattern script: lấy UUID Controller bằng `find` + `--bare`, sau đó `set` các cột tinh chỉnh thời gian. Đây là cách tăng `max_backoff` lên 15s khi rolling-upgrade controller cluster.

### 9.4.U.5 Họ subcommand thao tác bảng `SSL`

Bảng `SSL` là singleton (tối đa 1 row) chứa toàn bộ private key + certificate + CA dùng chung cho mọi kết nối OpenFlow `ssl:` và OVSDB-manager `ssl:`. Tham chiếu từ `Open_vSwitch.ssl` kiểu `optional UUID`. Vì là singleton, bộ subcommand generic ít hữu dụng hơn các bảng khác — wrapper `set-ssl`/`get-ssl`/`del-ssl` xử lý gần như mọi nhu cầu thường gặp.

| Lệnh | Cú pháp tiêu biểu | Vai trò |
|------|-------------------|---------|
| `list SSL` | `ovs-vsctl list SSL` | Dump path tới key/cert/CA hiện tại |
| `get SSL` | `ovs-vsctl get SSL . private_key` | Lấy giá trị một cột (dùng `.` làm record-id) |
| `set SSL` | `ovs-vsctl set SSL . bootstrap_ca_cert=true` | Bật bootstrap mode |
| `add SSL` | Rare | Dành cho cột `external_ids` |
| `remove SSL` | Rare | Đối xứng |
| `clear SSL` | `ovs-vsctl clear SSL . certificate` | Hiếm; thực tế dùng `del-ssl` |
| `find SSL` | `ovs-vsctl find SSL bootstrap_ca_cert=true` | Hiếm vì singleton |
| `create SSL` | Wrapper `set-ssl` lo | Trong `--id=@` chain rất hiếm |
| `destroy SSL` | Wrapper `del-ssl` lo | - |

Vì là singleton, record-id đặc biệt là `.` (dot) — biểu thị "row duy nhất". Cú pháp `ovs-vsctl get SSL . private_key` trả về string path; `ovs-vsctl set SSL . bootstrap_ca_cert=true` bật chế độ self-signed CA học trong lần kết nối đầu, sau đó cố định (giảm phiền phức cho lab nhưng KHÔNG dùng production). Schema 5 cột (`vswitchd/vswitch.ovsschema`): `private_key` (string path), `certificate` (string path), `ca_cert` (string path), `bootstrap_ca_cert` (boolean), `external_ids` (map). Hàm wrapper `cmd_set_ssl()` ở `utilities/ovs-vsctl.c` thực chất gọi `cmd_create()`/`cmd_set()` của `lib/db-ctl-base.c` rồi nối UUID vào `Open_vSwitch.ssl`.

Use case generic ít: `list SSL` để xác minh path file đang dùng (lưu ý OVS không validate path tồn tại lúc set — chỉ khi `ovs-vswitchd` mở socket mới fail). Lỗi điển hình: `set SSL . private_key=/tmp/key.pem` xong restart vswitchd thì kết nối SSL fail vì path không tồn tại — luôn `ls -l` ba file trước/sau khi set, đối chiếu với `ovs-vswitchd.log` tìm dòng `failed to load ... certificate`. Khi rotate cert, pattern an toàn là `set SSL . certificate=/new/cert.pem ca_cert=/new/ca.pem private_key=/new/key.pem` trong một transaction (cùng invocation `ovs-vsctl`), không gọi 3 lệnh riêng vì khoảng giữa hai lệnh có thể mất kết nối nếu cert/key không khớp tạm thời. So với HAProxy dùng `set ssl cert` qua admin socket reload trong-process, OVS `set SSL` cần restart vswitchd hoặc `ovs-appctl` reload — chưa hot-reload SSL config.

Mẫu lệnh thực tế:

```shell
$ sudo ovs-vsctl get-ssl
$ sudo ovs-vsctl list SSL
$ sudo ovs-vsctl set SSL . certificate=/new/cert.pem ca_cert=/new/ca.pem private_key=/new/key.pem
$ sudo ovs-vsctl get SSL . bootstrap_ca_cert
```

`get-ssl` là wrapper in 4 trường gọn gàng; `list SSL` trả về dạng row OVSDB chuẩn. Lệnh thứ ba minh họa rotate cert trong một transaction — cách an toàn duy nhất khi đổi cả ba file cùng lúc.

## 9.4.D Mở rộng tham chiếu nhóm option daemon-mode `ovs-vsctl` (mở rộng 2026-04)

> **Phạm vi mục này.** `ovs-vsctl` thường chạy ở chế độ one-shot (gọi từ shell, exit ngay sau khi commit transaction OVSDB). Tuy nhiên, một số ngữ cảnh init-script và monitoring cần `ovs-vsctl` chạy nền (daemon) như tiến trình thường trú: theo dõi sự kiện `monitor` của OVSDB, ghi PID để init-script quản lý, redirect log sang syslog từ xa. Bộ option daemon-mode được khai báo chung qua thư viện `lib/daemon-unix.c` + `lib/daemon.c` (cung cấp `OVS_DAEMON_OPTION_ENUMS`, `DAEMON_OPTION_HANDLERS`) và áp dụng đồng nhất cho mọi binary OVS/OVN: `ovs-vsctl`, `ovs-vswitchd`, `ovsdb-server`, `ovn-controller`, `ovn-northd`. Mục này gom 5 option phổ biến nhất khi `ovs-vsctl` chạy daemon-mode: `--detach`, `--pidfile`, `--monitor`, `--no-syslog`, `--syslog-target`. Nguồn upstream: `lib/daemon-unix.c`, `lib/daemon.c`, `lib/vlog.c` (cho syslog), man `ovs-vsctl(8)` §`Daemon Options` + `ovsdb-server(1)` §`Daemon Options` (cùng option set). Baseline: OVS branch-2.17 (2.17.9 trên Ubuntu Jammy).

---

### 9.4.D.1 `ovs-vsctl --detach`

#### Khái niệm

`--detach` chuyển `ovs-vsctl` từ foreground sang background daemon: sau khi parse option và khởi tạo connection OVSDB, tiến trình `fork()`, parent exit (về shell), child tiếp tục chạy như daemon thường trú. Áp dụng khi `ovs-vsctl` chạy với subcommand thường trú như `monitor` (theo dõi thay đổi DB realtime), hoặc khi script init muốn launch một tiến trình giám sát mà không chiếm terminal. Man page: *"Runs this program as a background process."*

#### Cú pháp + ngữ nghĩa

```
ovs-vsctl --detach [--pidfile[=PATH]] [--monitor] [--no-chdir] <thường-trú-subcmd>
```

Ngữ nghĩa chuẩn POSIX double-fork: `daemonize_start()` trong `lib/daemon-unix.c` thực hiện `fork()` lần một, parent đợi child setup xong (qua pipe sync), child `setsid()` tách session, optionally `fork()` lần hai để tránh tái claim terminal. Stdin/stdout/stderr redirect về `/dev/null` mặc định (ghi đè bằng `--no-close-fd` nội bộ).

#### Mẫu output thực tế

```shell
$ ovs-vsctl --detach --pidfile=/tmp/vsctl.pid --no-syslog monitor Bridge
$ ps -ef | grep ovs-vsctl
root  12345    1  0 14:02 ?   00:00:00 ovs-vsctl --detach ... monitor Bridge
$ cat /tmp/vsctl.pid
12345
```

Sau khi `--detach`, shell prompt trả về ngay (không chiếm terminal); tiến trình con tiếp tục chạy với PPID = 1 (init).

#### Vai trò trong vận hành

Hai use case: (1) script systemd unit launch một watcher OVSDB giám sát sự kiện thay đổi bảng — kết hợp `--detach --pidfile` để systemd track PID. (2) lab/dev tạm time một watcher học cách OVSDB transaction phản ánh đến client `monitor`. Tránh dùng `--detach` cho subcommand one-shot (`add-br`, `set`, ...) vì daemon hóa một tiến trình rồi exit ngay không có ý nghĩa.

#### Source code tham chiếu

Hàm chính: `daemonize_start()` trong `lib/daemon-unix.c` (parent fork + child setsid + double-fork + close-fd), gọi từ `daemonize()` trong `lib/daemon.c`. Option parse trong macro `OVS_DAEMON_OPTION_HANDLERS` set biến `detach = true`. Mọi binary OVS share code path này — không có gì vsctl-specific.

---

### 9.4.D.2 `ovs-vsctl --pidfile`

#### Khái niệm

`--pidfile[=PATH]` ghi PID của tiến trình `ovs-vsctl` (sau `--detach`) vào file để init-script/systemd track. Không có `=PATH`, mặc định `/var/run/openvswitch/ovs-vsctl.pid`. File chứa duy nhất một dòng số PID (không newline cuối). Man page: *"If PATH is omitted, the default is /var/run/openvswitch/<program>.pid."*

#### Cú pháp + ngữ nghĩa

```
ovs-vsctl --detach --pidfile                       # default path
ovs-vsctl --detach --pidfile=/tmp/vsctl-watch.pid  # custom
ovs-vsctl --detach --pidfile --overwrite-pidfile   # cho phép ghi đè PID cũ
```

Mặc định, nếu pidfile tồn tại và PID trong đó còn sống, `ovs-vsctl` từ chối start (tránh chạy hai instance trùng); `--overwrite-pidfile` skip check này. Lock theo `flock()` để tránh race giữa hai instance khởi động cùng lúc.

#### Mẫu output thực tế

```shell
$ ovs-vsctl --detach --pidfile=/tmp/v.pid monitor Bridge
$ cat /tmp/v.pid
13567
$ ovs-vsctl --detach --pidfile=/tmp/v.pid monitor Bridge
ovs-vsctl: /tmp/v.pid: already running as pid 13567, aborting
```

#### Vai trò trong vận hành

Hai use case: (1) systemd unit type=forking dùng `PIDFile=/var/run/.../ovs-vsctl.pid` để theo PID con sau khi parent `--detach` exit. (2) script shutdown `kill $(cat /tmp/v.pid)` để dừng watcher gọn gàng. Lưu ý quyền: directory chứa pidfile phải writable bởi user chạy `ovs-vsctl` (thường root); nếu chạy non-root, đổi sang `/tmp` hoặc `$XDG_RUNTIME_DIR`.

#### Source code tham chiếu

Hàm `make_pidfile()` trong `lib/daemon-unix.c`: `open(O_CREAT|O_RDWR)` + `fcntl(F_SETLK)` lock + `dprintf(fd, "%d\n", getpid())`. Option parse set `pidfile = optarg ? optarg : default_pidfile()`. Cleanup PID file ở `daemonize_complete()` qua `atexit()` handler.

---

### 9.4.D.3 `ovs-vsctl --monitor`

#### Khái niệm

`--monitor` (parent watchdog, KHÔNG nhầm với subcommand `ovs-vsctl monitor` của OVSDB) bật chế độ "monitoring process": parent fork một child rồi parent ở lại làm watcher, restart child nếu child crash bất ngờ. Tránh đứt watcher OVSDB do segfault hiếm gặp trong môi trường runtime dài hạn. Man page: *"Runs this program as the monitor of a child process. If the child terminates abnormally, the monitor restarts it."*

#### Cú pháp + ngữ nghĩa

```
ovs-vsctl --detach --monitor --pidfile=/tmp/v.pid monitor Bridge
```

`--monitor` cần đi kèm `--detach` (về mặt thực dụng); thiếu `--detach`, monitor parent chiếm terminal nên ít khi dùng. Khi child exit code 0 (normal), monitor exit theo. Khi child exit code != 0 hoặc bị signal kill bất thường (SIGSEGV, SIGABRT), monitor `fork()` + `exec()` lại với cùng argv. Pidfile (nếu có) lưu PID của monitor parent, không phải child — vì monitor là entity stable.

#### Mẫu output thực tế

```shell
$ ovs-vsctl --detach --monitor --pidfile=/tmp/v.pid monitor Bridge
$ ps -ef | grep ovs-vsctl
root 14001  1     0 ... ovs-vsctl --monitor ... monitor Bridge   # parent watchdog
root 14002 14001 0 ... ovs-vsctl --monitor ... monitor Bridge   # child worker
$ kill -SEGV 14002
$ ps -ef | grep 14001
root 14001  1     0 ... ovs-vsctl ...                  # vẫn còn
root 14050 14001 0 ... ovs-vsctl ...                  # child mới
```

#### Vai trò trong vận hành

Use case chính: tăng độ bền cho watcher OVSDB chạy 24/7 trong production. Trade-off: nhân đôi memory footprint (parent + child cùng image), và monitor không phát hiện được loại lỗi "live but stuck" (child loop vô hạn, không crash) — chỉ phát hiện crash hard. Với loại bug "stuck", cần thêm liveness probe ở tầng cao hơn.

#### Source code tham chiếu

Hàm `monitor_daemon()` trong `lib/daemon-unix.c`: vòng lặp `fork() → wait()` với check `WIFEXITED(status) && WEXITSTATUS(status) == 0` để quyết định restart hay exit. Backoff exponential khi crash liên tục để tránh fork-bomb.

---

### 9.4.D.4 `ovs-vsctl --no-syslog`

#### Khái niệm

`--no-syslog` tắt destination syslog của subsystem `vlog` trong OVS (khác `--syslog-target` chỉ đổi đích). Cụ thể, set log level cho destination `syslog` về `off` cho mọi module. STDERR/file logging vẫn hoạt động. Man page (chia sẻ với mọi binary OVS): *"By default, ovs-vsctl sends messages to syslog. This option disables that action."*

#### Cú pháp + ngữ nghĩa

```
ovs-vsctl --no-syslog --detach monitor Bridge
ovs-vsctl --no-syslog -vsyslog:off ...    # tương đương
```

Tương đương về hiệu ứng: `--no-syslog` ≡ `-vsyslog:off` cho tất cả module. Khác `--verbose=...:syslog:LVL` (chỉ áp một module). Áp dụng tại thời điểm parse option (trước khi log message đầu tiên), nên không có race.

#### Mẫu output thực tế

```shell
$ ovs-vsctl --detach --pidfile=/tmp/v.pid monitor Bridge
$ tail -n 1 /var/log/syslog
... ovs-vsctl: monitoring database ...
$ ovs-vsctl --no-syslog --detach --pidfile=/tmp/v2.pid monitor Bridge
$ tail -n 1 /var/log/syslog
... (không có dòng mới của ovs-vsctl) ...
```

#### Vai trò trong vận hành

Hai use case: (1) lab/CI tạm thời không muốn làm bẩn syslog với hàng nghìn dòng OVS log. (2) môi trường đã có agent log riêng (Fluent Bit, Vector) đọc log file trực tiếp, không cần qua syslog — tắt syslog tránh trùng lặp. Lưu ý: `--no-syslog` KHÔNG tắt logging tổng thể; module log vẫn ghi ra stderr/file qua `--log-file=PATH`.

#### Source code tham chiếu

Trong `lib/vlog.c`, hàm `vlog_set_levels(NULL, VLF_SYSLOG, VLL_OFF)` — `NULL` nghĩa là áp lên mọi module. Option `--no-syslog` parse trong `OVS_DAEMON_OPTION_HANDLERS` map sang call này. Cấu trúc destination `vlog_destinations[]` index `VLF_SYSLOG` lưu level cho từng module trong bitmap.

---

### 9.4.D.5 `ovs-vsctl --syslog-target=URI`

#### Khái niệm

`--syslog-target=HOST:PORT` redirect syslog từ socket Unix mặc định (`/dev/log`) sang một collector từ xa qua UDP (default port 514). Cho phép tập trung log của nhiều node OVS về một syslog server (rsyslog, syslog-ng, Loki via syslog input). Khác `--no-syslog` (tắt hoàn toàn); `--syslog-target` đổi đích, vẫn gửi log. Man page: *"Send syslog messages to UDP HOST:PORT, in addition to the system logger."*

#### Cú pháp + ngữ nghĩa

```
ovs-vsctl --syslog-target=10.0.0.5:514 --detach monitor Bridge
ovs-vsctl --syslog-target=collector.lab:6514 --detach ...   # custom port
```

URI dạng `HOST:PORT` thuần (không scheme). Format wire là plain UDP RFC 3164 syslog (`<PRI>TIMESTAMP HOST TAG: MSG`). Không có TLS, không có queue local nếu collector down — message rớt im lặng. Khác `--syslog-method` (option khác trên ovsdb-server, OVS 2.6+) chọn phương thức (udp/unix/null).

#### Mẫu output thực tế

```shell
$ ovs-vsctl --syslog-target=192.168.10.5:514 --detach --pidfile=/tmp/v.pid monitor Bridge
# trên collector 192.168.10.5:
$ sudo tcpdump -i any -nn udp port 514 -A
... > 192.168.10.5.514: UDP ...
<134>Apr 26 14:30:22 host01 ovs-vsctl: monitoring database ...
```

#### Vai trò trong vận hành

Use case: cluster nhiều chassis OVS gửi log tập trung về syslog collector (SIEM, rsyslog forwarder). Trade-off: UDP unreliable (mất gói khi collector overload), không xác thực — nội bộ network mới dùng. Production khuyến nghị wrap qua rsyslog local (output module `omfwd` với TCP/TLS) thay vì gửi UDP trực tiếp từ `ovs-vsctl`.

#### Source code tham chiếu

Trong `lib/vlog.c`, hàm `vlog_set_syslog_target()` parse `HOST:PORT`, mở UDP socket non-blocking, lưu vào biến static `syslog_fd` + `syslog_target_addr`. Mỗi lần `vlog()` ghi destination syslog, kiểm tra `syslog_fd >= 0` thì `sendto()` thay vì `syslog(3)` qua `/dev/log`. Option parse trong `OVS_DAEMON_OPTION_HANDLERS` case `'S'` (short option nội bộ).

---

## 9.29.1 Vì sao cần VTEP — bài toán bare metal trong overlay

OVN xây dựng L2 overlay bằng tunnel (Geneve/VXLAN) giữa các chassis hypervisor. VM trên hypervisor có ovn-controller cài flow rule trên `br-int` để encapsulate/decapsulate tunnel header. Mọi VM cùng một logical switch nhìn nhau như cùng một L2 broadcast domain bất kể chassis nào.

Nhưng còn **server bare metal** — máy chủ vật lý không chạy hypervisor + không có ovn-controller? Ví dụ điển hình:
- Storage NAS/SAN appliance không cài Linux hypervisor được
- Mainframe IBM Z chạy z/OS legacy
- Network appliance (firewall vật lý, IDS, load balancer hardware)
- Physical server cũ chưa migrate sang virtualization

Các thiết bị này cần kết nối vào cùng L2 overlay với VM trên hypervisor. Giải pháp: **HW VTEP** — một switch vật lý high-end có capability VXLAN encap/decap line-rate, bridge VLAN của port vật lý nối tới bare metal server với VXLAN VNI của OVN logical switch.

**HW VTEP architecture:**

```
   [bare metal server]                     [hypervisor]
         |                                      |
   [physical port]                       [veth port]
         |                                      |
   +-----+------+        VXLAN          +-------+--------+
   | HW VTEP    |<====== tunnel ======> |  br-int        |
   | (vendor)   |                       | (ovn-ctrl)     |
   +------------+                       +----------------+
         ^                                      ^
         |                                      |
         | OVSDB protocol (port 6640)           |
         |                                      |
   +-----+----------------------------+         |
   | ovsdb-server (VTEP schema)       |         |
   | + ovn-controller-vtep daemon     +---------+
   +----------------------------------+
```

`ovn-controller-vtep` là một daemon đặc biệt (KHÁC `ovn-controller`) chạy trên một server quản trị, kết nối:
- HW VTEP switch qua VTEP schema (OVSDB protocol)
- OVN SB DB qua thông thường

Daemon đọc Logical_Switch_Port type=vtep trong OVN NB → cấu hình tương ứng vào VTEP schema → HW switch tự động bridge VLAN+VXLAN.

`vtep-ctl` là CLI tool để quản trị VTEP schema (read + write), tương tự `ovs-vsctl` cho `Open_vSwitch` schema. Cùng cú pháp, cùng pattern transactional.

## 9.29.2 VTEP schema — 7 table chính

Schema VTEP định nghĩa trong `/usr/share/openvswitch/vtep.ovsschema` (OVS 2.17.9). 7 table chính:

| Table | Mục đích |
|-------|----------|
| `Global` | Singleton root, chứa metadata schema + reference đến manager |
| `Manager` | Cấu hình kết nối tới OVSDB protocol manager (controller) |
| `Physical_Switch` | Đại diện một HW switch vật lý (VTEP) |
| `Physical_Port` | Port trên Physical_Switch (cấu hình VLAN trunk) |
| `Logical_Switch` | Logical switch trong overlay (mapping với OVN logical switch) |
| `Logical_Binding_Stats` | Thống kê traffic theo binding |
| `Physical_Locator` | IP endpoint của VTEP (source IP cho VXLAN encap) |

Thêm 4 table phụ cho MAC binding:
- `Ucast_Macs_Local`, `Ucast_Macs_Remote` — unicast MAC location
- `Mcast_Macs_Local`, `Mcast_Macs_Remote` — multicast MAC location

Phân biệt local vs remote: Local = MAC nằm sau port của HW VTEP đang quản. Remote = MAC nằm sau VTEP khác hoặc trên hypervisor, cần gửi qua VXLAN.

## 9.29.3 Synopsis + commands

```
vtep-ctl [options] -- [options] command [args] [-- ...]
```

Cú pháp giống `ovs-vsctl`: chuỗi command tách bằng `--` thực thi trong cùng 1 transaction atomic.

**Commands chia 7 nhóm theo verified man page:**

### Physical Switch group

| Command | Mục đích |
|---------|----------|
| `add-ps PSWITCH` | Tạo physical switch entry mới |
| `del-ps PSWITCH` | Xóa physical switch entry |
| `list-ps` | Liệt kê toàn bộ physical switch |
| `ps-exists PSWITCH` | Test tồn tại; exit 0=có, 2=không |

### Physical Port group

| Command | Mục đích |
|---------|----------|
| `add-port PSWITCH PORT` | Add port (logical entry, không tự cấu hình HW) |
| `del-port [PSWITCH] PORT` | Xóa port |
| `list-ports PSWITCH` | Liệt kê port của switch |

### Logical Switch group

| Command | Mục đích |
|---------|----------|
| `add-ls LSWITCH` | Tạo logical switch entry trong VTEP DB |
| `del-ls LSWITCH` | Xóa logical switch |
| `list-ls` | Liệt kê logical switch |
| `ls-exists LSWITCH` | Test tồn tại |
| `bind-ls PSWITCH PORT VLAN LSWITCH` | **Lệnh quan trọng nhất**: bind physical port:VLAN với logical switch (= VXLAN VNI tunnel) |
| `unbind-ls PSWITCH PORT VLAN` | Remove binding |
| `list-bindings PSWITCH PORT` | Show bindings hiện tại của port |
| `set-replication-mode LSWITCH MODE` | Mode = `service_node` (centralize BUM tới một service node) hoặc `source_node` (HW VTEP tự handle BUM) |
| `get-replication-mode LSWITCH` | Lấy mode hiện tại |

### Logical Router group (HW VTEP có support routing)

| Command | Mục đích |
|---------|----------|
| `add-lr LROUTER` | Tạo logical router |
| `del-lr LROUTER` | Xóa |
| `list-lr` | Liệt kê |
| `lr-exists LROUTER` | Test tồn tại |

### MAC Binding group

| Command | Mục đích |
|---------|----------|
| `add-ucast-local LSWITCH MAC [ENCAP] IP` | MAC nằm local sau VTEP, IP = endpoint của VTEP |
| `del-ucast-local LSWITCH MAC` | Xóa local unicast binding |
| `add-mcast-local LSWITCH MAC [ENCAP] IP` | Multicast MAC local |
| `del-mcast-local LSWITCH MAC [ENCAP] IP` | Xóa |
| `clear-local-macs LSWITCH` | Clear toàn bộ local MAC |
| `list-local-macs LSWITCH` | Liệt kê |
| `add-ucast-remote LSWITCH MAC [ENCAP] IP` | MAC remote (sau VTEP/hypervisor khác) |
| `del-ucast-remote LSWITCH MAC` | Xóa |
| `add-mcast-remote/del-mcast-remote/clear-remote-macs/list-remote-macs` | Tương tự cho remote multicast |

### Manager + Database group

| Command | Mục đích |
|---------|----------|
| `set-manager TARGET...` | Cấu hình OVSDB manager (controller) |
| `get-manager` / `del-manager` | Get / del |
| `list/find/get/set/add/remove/clear/create/destroy/wait-until` | Generic OVSDB row ops (giống `ovs-vsctl`) |

## 9.29.5 Anatomy: `vtep-ctl list-ps`

**Bucket:** OVS | **Context:** verify HW VTEP đã đăng ký với cụm OVN | **Purpose:** liệt kê physical switch hiện có trong VTEP DB | **Activity:** monitoring | **Mechanism:** OVSDB query + format output.

**Output mẫu:**

```
$ vtep-ctl --db=tcp:10.0.0.100:6640 list Physical_Switch
_uuid               : abc12345-...
description         : "Arista 7050 hv-rack-3"
management_ips      : ["10.0.0.100"]
name                : "ps-rack3"
ports               : [def67890-..., ghi23456-...]
switch_fault_status : []
tunnel_ips          : ["10.10.10.100"]
tunnels             : [...]
```

**Decode 7-attribute Anatomy:**

| Attribute | Giá trị mẫu | Healthy / Warning / Critical |
|-----------|-------------|------------------------------|
| `_uuid` | UUID | Healthy: stable across restart |
| `name` | "ps-rack3" | Healthy: tên phải khớp inventory CMDB |
| `management_ips` | `["10.0.0.100"]` | IP control plane (OVSDB) |
| `tunnel_ips` | `["10.10.10.100"]` | Critical: IP source cho VXLAN encap; KHÔNG được trùng với hypervisor IP |
| `ports` | List UUID Physical_Port | Healthy: ≥ 1 port nối server |
| `switch_fault_status` | `[]` (empty) | Healthy: empty = OK; Critical: có fault → HW switch lỗi |
| `tunnels` | List tunnel endpoint | Healthy: ≥ 1 tunnel tới hypervisor active |

## 9.29.6 Anatomy: `vtep-ctl bind-ls`

**Bucket:** OVS | **Context:** kết nối VLAN port của HW VTEP với VXLAN VNI tunnel | **Purpose:** core integration step — packet vào port:VLAN sẽ được encap VXLAN với VNI = logical switch | **Activity:** provisioning | **Mechanism:** insert row vào Logical_Binding + cập nhật Physical_Port.vlan_bindings.

**Lệnh + decode:**

```
$ vtep-ctl --db=tcp:10.0.0.100:6640 \
  bind-ls ps-rack3 eth48 100 ls-tenant-blue

# Tương đương: traffic từ port eth48 với VLAN tag 100 →
# encap VXLAN với VNI tương ứng ls-tenant-blue + tunnel sang hypervisor
```

| Tham số | Ý nghĩa |
|---------|---------|
| `ps-rack3` | Tên Physical_Switch (created bằng `add-ps`) |
| `eth48` | Tên Physical_Port (created bằng `add-port`) |
| `100` | VLAN ID (1-4094) trên port vật lý |
| `ls-tenant-blue` | Tên Logical_Switch (created bằng `add-ls`); ánh xạ với OVN logical switch cùng name + tunnel_key |

## 9.29.7 Bài tập có hướng dẫn (Guided Exercise)

**Mục tiêu:** lab synthetic VTEP setup với mock HW VTEP (dùng `ovs-vtep` simulator), bind 1 logical switch + verify cross-VTEP MAC learning.

**Setup mock VTEP:**

```bash
# 1. Tạo VTEP database (synthetic)
ovsdb-tool create /tmp/vtep.db /usr/share/openvswitch/vtep.ovsschema
ovsdb-server /tmp/vtep.db --remote=ptcp:6640:127.0.0.1 \
  --pidfile=/tmp/vtep-ovsdb.pid --detach

# 2. Khởi động ovs-vtep simulator
ovs-vsctl add-br br-vtep
ovs-vtep --log-file=/tmp/ovs-vtep.log \
  --pidfile=/tmp/ovs-vtep.pid \
  --detach br-vtep
```

**Cấu hình physical switch + port:**

```bash
# 3. Add physical switch entry
vtep-ctl --db=tcp:127.0.0.1:6640 \
  add-ps mock-ps -- \
  set Physical_Switch mock-ps \
    tunnel_ips=192.168.1.100 \
    management_ips=127.0.0.1

# 4. Add physical port
vtep-ctl --db=tcp:127.0.0.1:6640 \
  add-port mock-ps eth-test
```

**Tạo + bind logical switch:**

```bash
# 5. Add logical switch + bind VLAN
vtep-ctl --db=tcp:127.0.0.1:6640 \
  add-ls ls-test -- \
  set Logical_Switch ls-test tunnel_key=42 -- \
  bind-ls mock-ps eth-test 100 ls-test

# 6. Verify binding
vtep-ctl --db=tcp:127.0.0.1:6640 list-bindings mock-ps eth-test
```

**Verify expected:** xuất hiện row trong Logical_Binding với physical_switch=mock-ps, port=eth-test, vlan=100, logical_switch=ls-test.

## 9.29.8 Mẫu cần tránh và lưu ý

**Mẫu cần tránh 1: VLAN range overlap giữa nhiều logical switch.**

Mỗi binding (port, VLAN) → 1 logical switch. Nếu binding lặp (cùng port, cùng VLAN, hai logical switch khác nhau), HW VTEP sẽ không xử lý đúng (behavior vendor-specific, có khi drop, có khi panic).

**Cách phòng:** trước khi `bind-ls`, query existing bindings: `vtep-ctl list-bindings PSWITCH PORT`.

**Mẫu cần tránh 2: Quên cấu hình `tunnel_ips` trên Physical_Switch.**

VTEP encap VXLAN cần biết source IP. Nếu `tunnel_ips=[]` (empty), HW switch không emit packet ra → BUM packet drop silently.

**Cách phòng:** sau `add-ps`, set `tunnel_ips` ngay; verify bằng `vtep-ctl get Physical_Switch <name> tunnel_ips`.

**Lưu ý 3: `ovs-vtep` simulator KHÔNG phải production HW.**

Simulator chỉ implement protocol layer (OVSDB write + emit hex packet), không có line-rate forwarding. Dùng cho lab/CI, KHÔNG cho production benchmark.

**Lưu ý 4: Vendor compat ma trận.**

VTEP schema là chuẩn OpenVSwitch nhưng vendor implement subset khác nhau:
- Arista EOS: full support, including `replication_mode=service_node`
- Mellanox Onyx: hỗ trợ basic binding, không support service_node mode
- Cumulus Linux: dùng FRR EVPN thay vì VTEP schema (different approach)

Trước deploy, verify support matrix vendor + OVS version.

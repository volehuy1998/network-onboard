## Bối cảnh: bài toán kernel datapath phải giải

Giả sử một host KVM chạy 50 VM, mỗi VM có 20 TCP connection đồng thời. Tổng cộng 1000 TCP microflow. Mỗi microflow gồm tổ hợp (src_ip, dst_ip, src_port, dst_port, protocol) — 5-tuple. Kernel datapath phải quyết định action chuyển tiếp cho mỗi packet với tốc độ triệu packet/second.

Cách tiếp cận naive — userspace daemon quyết định mọi packet — không khả thi vì context switch kernel↔userspace tiêu tốn khoảng 2 microsecond, giới hạn băng thông thực tế ở ~500K packet/second.

Cách tiếp cận ngược lại — toàn bộ OpenFlow pipeline trong kernel — không khả thi vì pipeline phức tạp với classifier, prioritized rules, arbitrary header modification. Kernel code không nên chứa logic như vậy.

Giải pháp OVS: **kernel duy trì cache cho quyết định đã tính**, userspace tính quyết định khi cache miss. Cache phải nhanh (nanosecond scale), chính xác (không sai policy), và hiệu quả về bộ nhớ.

## 9.2.1 Microflow cache: thiết kế OVS 1.x và giới hạn

OVS 1.x (2011-2013) dùng **microflow cache**: mỗi entry mô tả chính xác một 5-tuple (exact match), không wildcard. Lookup O(1) qua hash table.

Ưu điểm: cực nhanh khi hit. Lookup chỉ cần compute hash + compare full key.

Nhược điểm fatal: **mỗi microflow khác biệt về bất kỳ field nào đều yêu cầu entry riêng**. Ví dụ:

- TCP connection 1: src=10.0.0.1:50001 → dst=10.0.0.2:80 → entry A
- TCP connection 2: src=10.0.0.1:50002 → dst=10.0.0.2:80 → entry B (khác src_port)
- TCP connection 3: src=10.0.0.1:50003 → dst=10.0.0.2:80 → entry C

Mặc dù cả ba đi qua cùng OpenFlow rule ("allow TCP dst=10.0.0.2:80") và cùng action (output:5), mỗi connection có entry cache riêng chỉ khác ở src_port. Với 1000 connection, cache có 1000 entry.

Kết quả: cache thrashing. Cache có kích thước cố định, khi số microflow vượt quá size, entry cũ bị evict. Packet evict gặp cache miss → upcall → reinstall → vòng luẩn quẩn.

NSDI 2015 đo trên workload Rackspace production (2013, trước megaflow): miss rate 10-30% trong peak. Chi phí phụ quá lớn để scale.

## 9.2.2 Megaflow cache: đột phá của NSDI 2015

**Megaflow** là cache entry với **wildcard** — match một tập microflow có cùng mẫu header. Thay vì 1000 entries cho 1000 connection TCP đến port 80, chỉ một entry duy nhất `dst=10.0.0.2, dst_port=80, proto=TCP, wildcard src_ip+src_port` cover tất cả.

Compression factor điển hình 10-100 lần. Workload 100K microflow → cache 2.000-10.000 entry.

**Câu hỏi then chốt**: làm sao quyết định field nào wildcard? OVS giải quyết bằng **lazy wildcarding**. Khi userspace classifier xử lý microflow lần đầu:

1. Parse packet thành 5-tuple + field khác.
2. Walk OpenFlow pipeline: match rules table 0 → table 1 → ..., execute action.
3. Trong quá trình walk, **track** field nào đã được kiểm tra.
4. Field KHÔNG được kiểm tra → wildcard.
5. Compile megaflow entry với wildcard cho field không kiểm tra.

Ví dụ: rule "match dst_ip=10.0.0.2, dst_port=80, proto=TCP → output:5". Packet với src_ip=10.0.0.1, src_port=50001:

- Classifier kiểm tra dst_ip (match), dst_port (match), proto (match).
- src_ip và src_port KHÔNG kiểm tra.

Megaflow entry: match `dst_ip=10.0.0.2, dst_port=80, proto=TCP`, wildcard `src_ip=*, src_port=*`.

**Correctness guarantee**: lazy wildcarding không phạm rule nào. Nếu có rule khác ràng buộc src_port, classifier sẽ kiểm tra khi walk, không wildcard. Correct by construction.

## 9.2.3 Tuple Space Search cho megaflow lookup

Khi có nhiều megaflow với mask khác nhau, mỗi megaflow có **tuple** — tập field được match (không wildcard). OVS dùng **Tuple Space Search (TSS)** — thuật toán từ Srinivasan et al. SIGCOMM 1998: nhóm entry theo tuple, mỗi tuple có hash table riêng.

Lookup packet: với mỗi tuple, extract field theo mask, compute hash, lookup hash table. Nếu match, return action. Nếu không, try tuple tiếp theo.

Độ phức tạp: O(T × H) với T = số tuple (thường < 20), H = chi phí hash lookup (O(1) trung bình).

> **Chủ đề trọng tâm:** Lựa chọn TSS thay vì TCAM hay decision tree là quyết định thực dụng. TCAM quá đắt trên commodity server. Decision tree build cost cao khi rules thay đổi thường xuyên. TSS là compromise tối ưu cho software datapath. NSDI 2015 benchmark: TSS đạt 10M+ lookup/second per core trên Intel Xeon.

## 9.2.4 Upcall: slow path và fast path

USC Lab 9 (`doc/ovs/Day 4-lab 9 - Open vSwitch Kernel Datapath.txt`) trình bày rõ:

> "Kernel module directs the packet to the userspace. The userspace makes the decisions about the actions to be taken against the packet according to OpenFlow entries (slow path)."
>
> "The action entry is stored in the kernel and used to forward subsequent packet which makes the chuyển tiếp faster (fast path)."

**Fast path (cache hit)**:

1. Packet đến NIC → kernel stack → `openvswitch.ko` rx hook.
2. Kernel extract flow key từ headers.
3. Kernel lookup megaflow cache qua TSS.
4. Match → execute action (output:5 → `netdev_ops->ndo_start_xmit`).
5. Update counter. Không chạm userspace. Độ trễ ~1-5 microsecond.

**Slow path (cache miss)**:

1. Packet đến, lookup miss.
2. Kernel gửi **upcall** qua Netlink lên userspace.
3. Upcall chứa packet bytes, metadata (in_port, timestamp), reason.
4. `ovs-vswitchd` handler thread nhận upcall.
5. Handler walk OpenFlow pipeline, track inspected fields.
6. Compile megaflow entry + action list.
7. `OVS_FLOW_CMD_NEW` về kernel — cài megaflow.
8. `OVS_PACKET_CMD_EXECUTE` để kernel execute action cho packet này.
9. Packet tương tự sau đó hit fast path.

USC Lab 9: lần đầu flow qua switch, "kernel table is empty" → miss, slow path. Sau "action entries are installed in the kernel", packet tiếp theo đi fast path.

Chi phí slow path: ~50-200 microsecond per packet. Nếu hit rate thấp, slow path chi phối băng thông thực tế.

## 9.2.5 Ukeys và revalidator threads

Khi megaflow cài vào kernel, `ovs-vswitchd` duy trì **ukey (unique key)** — bản sao userspace chứa flow key, mask, action, statistics counter cuối cùng, timestamp last_used.

**Revalidation**: khi rules thay đổi, revalidator threads (mặc định 4, 500ms interval) scan ukey:

1. Với mỗi ukey, re-walk pipeline với flow key.
2. So sánh action mới vs cũ.
3. Khác → `OVS_FLOW_CMD_SET` update kernel.
4. Không khớp với rule nào → `OVS_FLOW_CMD_DEL`.

**Timeout**: rules có idle_timeout và hard_timeout. Revalidator track timestamp, xóa khi timeout.

Cấu hình `other_config:n-revalidator-threads`. Cache lớn, scan tốn CPU — tăng lên 8-16 thread nếu cần.

## 9.2.6 Upcall rate limit

Compass Ch Q mô tả tuning:

- `upcall/show` — số upcall in-flight, avg, max, limit (mặc định 200.000 flow-install/sec).
- `upcall/set-flow-limit 500000` — tăng limit cho workload scale cao.

Chạm limit → kernel drop packet → counter `lost` trong `ovs-dpctl show` tăng.

Tương quan:

- `hit` tăng tuyến tính → bình thường.
- `missed` tăng → cache không hit tốt.
- `lost` > 0 → chạm rate limit, nguy cơ mất traffic.

Cấu hình `other_config:n-handler-threads` (mặc định 2). Tăng nếu miss rate cao nhưng lost = 0 — handler nghẽn.

## 9.2.7 Khớp cache rate trong production

NSDI 2015 báo cáo Rackspace production:

- Trước megaflow: hit 70-90%, miss 10-30% peak.
- Sau megaflow: hit 99%+, miss dưới 1%.

Với packet rate 1M/sec, ~10.000 upcall/sec — trong khả năng handler thread.

**Dấu hiệu cảnh báo sớm**:

- Coverage miss rate > 1% → compress không hiệu quả.
- `ovs-dpctl show` `lost` > 0 → hit rate limit.
- `upcall/show` avg cao → handler nghẽn.

**Scenarios hit rate tụt**:

1. Traffic đa dạng cao — mỗi microflow khác về field rules match.
2. Rules thay đổi liên tục — revalidator invalidate nhiều ukey.
3. Cache size nhỏ, `max-idle` thấp — entry evict sớm.

## 9.2.8 EMC, Exact khớp với Cache — tier đầu tiên trong datapath lookup

Cho đến §9.2.7 ta đã thảo luận megaflow cache như thể nó là lớp cache duy nhất. Thực tế, **kernel datapath OVS 2.x có ba tier cache chồng lên nhau**, theo thứ tự lookup: EMC → SMC → megaflow classifier (TSS). Hiểu cả ba tier giúp đọc đúng `pmd-stats-show` và diagnose `miss with success upcall` counter (Phần 9.11 §9.11.8).

**Exact khớp với Cache (EMC)** là tier đầu — hash table exact-match cho mỗi packet 5-tuple (hoặc wider key). Không có wildcard. Mỗi entry EMC tương ứng một **microflow** cụ thể.

### 9.2.8.1 Cấu trúc EMC

| Thuộc tính | Giá trị |
|---|---|
| **Scope** | Per PMD thread (hoặc per CPU core với kernel datapath) — không shared |
| **Lookup** | O(1) hash table, no wildcard |
| **Key** | Full packet header fields (cả src và dst, TCP ports, v.v.) |
| **Capacity** | Mặc định 8192 entry per PMD (compile-time constant `EM_FLOW_HASH_ENTRIES`) |
| **Insertion** | Sau lookup thành công từ tier dưới (SMC hoặc megaflow) |
| **Eviction** | LRU-like, ghi đè khi collision hash |
| **Purpose** | Fast path cho flow hot — packet hot kênh vô EMC, không cần re-walk TSS |

EMC tồn tại dưới cả kernel datapath (`openvswitch.ko`) và userspace DPDK datapath (`dpif-netdev`). Kernel implementation khác userspace ở chi tiết implementation nhưng semantics giống: hash exact-match cho fast path.

### 9.2.8.2 Đọc hiểu EMC stats qua `pmd-stats-show`

**Bối cảnh:** chỉ áp dụng với `datapath_type=netdev` (DPDK/AF_XDP). Kernel datapath không expose EMC stats trực tiếp — phải qua `coverage/show` counter `emc_hit`, `emc_lookup`.

**Command và output** `[reproduced from OVS DPDK benchmark setup]`:

```shell
$ sudo ovs-appctl dpif-netdev/pmd-stats-show
pmd thread numa_id 0 core_id 2:
  packets received: 10000000
  packet recirculations: 12543
  avg. datapath passes per packet: 1.00
  emc hits: 9050231
  smc hits: 820412
  megaflow hits: 129303
  avg. subtable lookups per megaflow hit: 2.3
  miss with success upcall: 54
  miss with failed upcall: 0
  avg. packets per output batch: 12.4
```

**Anatomy từng counter EMC-related:**

| Counter | Ý nghĩa | Healthy |
|---|---|---|
| `emc hits` | Packet khớp với EMC tier | ≥ 85% `packets received` cho workload stable |
| `smc hits` | Packet miss EMC, hit SMC tier | 5-15% |
| `megaflow hits` | Miss cả EMC + SMC, hit megaflow classifier | 1-5% |
| `miss with success upcall` | Miss megaflow, upcall thành công cài đặt flow | < 0.1% steady state |
| `miss with failed upcall` | Upcall failed (OVSDB lock, error) | Phải = 0 |

**Kịch bản bẻ gãy:**

- **EMC hit < 50% với workload dự kiến stable** → EMC thrashing. 8192 entry không đủ cho workload > 8000 distinct microflow đồng thời. Giải pháp: enable SMC (default on trên OVS 2.15+), hoặc chuyển sang megaflow hit (acceptable miss cấp đầu).
- **SMC hit + megaflow hit > 50% total** → EMC bypass hoặc disabled. Check `ovs-appctl dpif-netdev/pmd-perf-show` detailed.
- **`miss with success upcall` tăng đều** → flow mới liên tục đến; workload scanner hoặc mẫu lưu lượng độc hại. Check OpenFlow classifier flow limit (Phần 9.2.6) và coverage/show `upcall_flow_add`.

### 9.2.8.3 Khi nào disable EMC

EMC tiêu thụ cache memory trên core PMD. Với workload very-low-hit-rate (mỗi packet đặc thù), EMC trở thành chi phí phụ thuần. Disable bằng:

```shell
$ sudo ovs-vsctl set Open_vSwitch . other_config:emc-insert-inv-prob=0
# Restart để apply; set về > 0 để re-enable (smaller = more insertion)
```

Giá trị `emc-insert-inv-prob=N` nghĩa: xác suất insert 1/N. Default 100 → 1% insertion rate. Setup specialized (security gateway scan) có thể set 1000 hoặc disable.

## 9.2.9 SMC, Signature khớp với Cache — tier trung gian OVS 2.15+

SMC được thêm vào OVS 2.15 (02/2021) để giải quyết gap hiệu năng giữa EMC (exact microflow, 8K entry) và megaflow classifier (TSS, chi phí O(T × H) cao hơn). NSDI 2020 paper *HXDP* + OVS mailing list thảo luận SMC là optimization cho workload 100k+ concurrent microflow mà EMC không còn đủ.

### 9.2.9.1 Cấu trúc SMC

| Thuộc tính | Giá trị |
|---|---|
| **Scope** | Per PMD thread (tương tự EMC) |
| **Lookup** | O(1) hash table, compact signature-based |
| **Key** | 16-bit signature của flow hash (không full 5-tuple) |
| **Value** | Pointer về megaflow entry (không trực tiếp action) |
| **Capacity** | 16384 entry per PMD (compile-time) |
| **Insertion** | Sau lookup thành công từ megaflow classifier |
| **Purpose** | Giảm chi phí TSS lookup cho flow "lạnh trung bình" (không đủ hot cho EMC nhưng vẫn regular) |

SMC **không chứa full action** — nó chứa pointer về megaflow entry. Lookup flow: compute signature hash → SMC lookup → nếu hit → dereference pointer → execute megaflow action. Bước dereference nhanh hơn re-walk TSS vì bỏ qua toàn bộ subtable scan.

### 9.2.9.2 Flow lookup path EMC → SMC → megaflow

Thứ tự lookup trong `dpif-netdev` (OVS 2.15+):

```
Packet arrives at PMD
  ↓
(1) EMC lookup: O(1) hash, full 5-tuple exact
    ├── HIT → execute action, done (~50 ns)
    └── MISS → (2)
  ↓
(2) SMC lookup: O(1) hash, 16-bit signature
    ├── HIT → follow pointer to megaflow, execute (~150 ns)
    └── MISS → (3)
  ↓
(3) Megaflow classifier TSS: O(T) subtable scan
    ├── HIT → execute action, insert into SMC (+optional EMC) (~300-500 ns)
    └── MISS → (4)
  ↓
(4) Upcall to userspace
    → handler walks OpenFlow pipeline
    → compile megaflow
    → install into kernel classifier
    → (~50-200 µs)
```

Numbers tham khảo từ NSDI 2020 + OVS benchmarks. Three-tier design cho phép trade-off memory-per-entry vs lookup cost:
- **EMC:** nhỏ (8K entry), cực nhanh (50 ns), chỉ cho hottest flow.
- **SMC:** trung bình (16K entry), nhanh vừa (150 ns), cho regular flow.
- **Megaflow:** lớn (200K entry default), chậm hơn (300-500 ns) nhưng wildcard-capable, cover bulk workload.

### 9.2.9.3 SMC tuning

SMC mặc định bật. Disable bằng `other_config:smc-enable=false` cho khắc phục sự cố hoặc benchmark.

Hiện tại (OVS 3.x) không có option tuning SMC size runtime — recompile OVS với `#define SMC_ENTRIES N` nếu cần. Production hiếm khi cần chỉnh.

## 9.2.10 Upcall Netlink protocol, anatomy message giữa kernel và userspace

Khi §9.2.4 giới thiệu upcall, ta chưa xem chi tiết **message wire format** giữa kernel và userspace. Upcall đi qua **Generic Netlink** (genl), một cơ chế Linux cho kernel subsystem expose API tới userspace. OVS đăng ký một genl family `ovs_vport`, `ovs_flow`, `ovs_datapath`, `ovs_packet` với kernel.

Hiểu wire format giúp debug khi `vlog/set dpif:file:dbg` bật ra dump Netlink header mà không ai giải thích nổi.

### 9.2.10.1 Cấu trúc Generic Netlink header

```
+-----------------------------------------------+
|           Netlink header (16 byte)             |
|  ┌─────────────────────────────────────────┐  |
|  │ nlmsghdr {                              │  |
|  │   __u32 nlmsg_len   (tổng length msg)   │  |
|  │   __u16 nlmsg_type  (family ID, animated│  |
|  │                      cho ovs_packet)     │  |
|  │   __u16 nlmsg_flags (NLM_F_REQUEST, ...) │  |
|  │   __u32 nlmsg_seq   (sequence #)         │  |
|  │   __u32 nlmsg_pid   (source PID)         │  |
|  │ }                                        │  |
|  └─────────────────────────────────────────┘  |
+-----------------------------------------------+
|       Generic Netlink header (4 byte)          |
|  ┌─────────────────────────────────────────┐  |
|  │ genlmsghdr {                            │  |
|  │   __u8 cmd    (OVS_PACKET_CMD_MISS,      │  |
|  │                OVS_PACKET_CMD_EXECUTE, ..) │  |
|  │   __u8 version                            │  |
|  │   __u16 reserved                          │  |
|  │ }                                        │  |
```

|---|---|
| `OVS_FLOW_ATTR_KEY` | Masked flow key (match field cần exact) |

## 9.2.11 Ukey lifecycle và revalidator RCU

§9.2.5 giới thiệu ukey + revalidator tổng quan. Mục này đi sâu vào state machine của ukey + synchronization guarantee mà revalidator cung cấp.

### 9.2.11.1 State machine của một ukey

Mỗi ukey (struct `udpif_key` trong `ofproto/ofproto-dpif-upcall.c`) đi qua lifecycle:

```
           create
              ↓
        ┌──────────┐
        │ CREATED  │  (insert tentative, chưa apply vào kernel)
        └──────────┘
              ↓  flow installed successfully
        ┌──────────┐
        │  VISIBLE │  (kernel đã có megaflow, ukey track counter)
        └──────────┘
              ↓  revalidator sweep thấy ukey
        ┌──────────┐
        │ OPERATIONAL │  (steady state, update stat mỗi sweep)
        └──────────┘
              ↓  rule change → re-walk pipeline
        ┌──────────┐
        │ CHECKING │  (revalidator kiểm tra action vẫn đúng)
        └──────────┘
         ↓ action same          ↓ action differ / no match
     (back to OPERATIONAL)    ┌────────────┐
                              │ EVICTING   │  (OVS_FLOW_CMD_DEL)
                              └────────────┘
                                    ↓
                              ┌────────────┐
                              │ DELETED    │  (free memory sau RCU grace)
                              └────────────┘
```

### 9.2.11.2 Revalidator RCU guarantee

Revalidator thread (default 4, `n-revalidator-threads`) chạy định kỳ (500 ms mặc định). Trong mỗi sweep:

1. **Snapshot phase (RCU read-side):** thread acquire RCU read lock, iterate ukey list.
2. **Per-ukey check:** với mỗi ukey, re-walk OpenFlow pipeline với flow key.
3. **Compare action:** kết quả pipeline mới so với action cũ trong ukey.
4. **Write phase:** nếu action khác, mark ukey EVICTING, gửi `OVS_FLOW_CMD_DEL` về kernel.
5. **Grace period:** ukey không free ngay — chờ RCU grace period (ước 25 ms) để đảm bảo không thread nào còn đọc ukey đó.

RCU (Read-Copy-Update) là Linux primitive cho phép read không block write. Revalidator lợi dụng RCU cho phép dispatcher thread (xử lý upcall) đọc ukey đồng thời revalidator update.

### 9.2.11.3 Đọc hiểu `revalidator/wait` output

**Bối cảnh:** khi debug race condition giữa OpenFlow programming và datapath state, chạy `revalidator/wait` barrier để đảm bảo revalidator đã processed current state.

```shell
$ sudo ovs-appctl dpctl/dump-flows | wc -l
1247
$ sudo ovs-ofctl -O OpenFlow15 del-flows br-int
$ sudo ovs-appctl revalidator/wait
waited <N> ms
$ sudo ovs-appctl dpctl/dump-flows | wc -l
0
```

Output `waited <N> ms` cho thấy block bao lâu. Nếu > 1000 ms → revalidator overloaded.

**Kịch bản bẻ gãy:**

- **`revalidator/wait` block > 5 giây** → revalidator thread có backlog ukey cần check. Check `upcall/show` xem `dump duration` và `avg-updates` per thread. Tăng `n-revalidator-threads`.
- **Counter `coverage/show facet_revalidate` rate > 10k/sec** → revalidator quá tải. Đây là signature của Phần 9.26 revalidator storm.

### 9.2.11.4 Ukey count vs rule count

`memory/show udpif keys` hiển thị ukey count. So với `rules`:

```
rules:12847
udpif keys:1247
```

Ukey thường ÍT hơn rule vì nhiều rule chưa từng khớp với packet → không tạo ukey. Ratio `ukey / rule` cho thấy coverage: ukey = flow đang active trong kernel.

**Pathology:** `udpif keys` > `rules` → ukey leak (Phần 9.26). Commit `180ab2fd635e` fix cụ thể bug này.

## 9.2.14 Bổ sung chuyên sâu cohort cornerstone OVS datapath (mở rộng 2026-04)

> **Mục đích:** Bổ sung các phương diện còn thiếu cho 4 keyword cornerstone đã substantively cover ở §9.2.1 đến §9.2.13: `megaflow`, `recirc_id`, `openvswitch.ko (kernel datapath)`, `conntrack execute`. Nhóm register keyword đã bổ sung ở Phần 13.18.

### 9.2.14.1 megaflow, các phương diện bổ sung

**Tầm quan trọng.** `megaflow` là **trụ cột tuyệt đối tier 1** của OVS scaling. Justification: trước OVS 2.0 (2014), kernel datapath chỉ có microflow per-5-tuple → production traffic 100K+ flow/sec gây flow explosion (kernel cache hàng triệu entry, classifier O(N) traversal, ovs-vswitchd CPU 100%). Megaflow consolidate qua wildcard mask reduce entry count 100-1000x. Engineer không hiểu megaflow = không debug được scaling issue, revalidator storm, cache miss spike. Tier 1 absolute, parity với `ct_state` về importance.

**Phân loại.** Loại: kernel datapath flow entry abstraction. Parent: `dp_netdev_flow` struct (kernel) hoặc `udpif_key` (revalidator userspace). Scope: per-bridge, persist trong kernel cho đến idle/hard timeout hoặc revalidator evict. Idempotency: re-install same flow = no effect (deduplicate). Type: data-plane caching, not control-plane state.

**Quy trình modify/tune:**

```
Step 1: Identify high cache miss rate
  $ ovs-appctl coverage/show | grep -E "miss|upcall"
  → upcall rate > 1000/sec = potential megaflow churn

Step 2: Inspect existing megaflow
  $ ovs-appctl dpctl/dump-flows | head -20
  → Look for flow with very narrow mask (close to microflow)

Step 3: Tune via OF rule design
  → Add conjunction action giảm cartesian flow rule
  → Use generic match (e.g., `ip` thay vì `ip,nw_src=10.0.0.5`)
    cho ACL không cần per-IP

Step 4: Force microflow (debug only)
  $ ovs-vsctl set Open_vSwitch . other_config:flow-limit=1000
  → Limit megaflow count để observe wildcard generalization

Step 5: Verify post-tune
  $ ovs-appctl coverage/show | grep upcall
  → upcall rate should drop after wildcard generalize works
```

**Khác biệt giữa các phiên bản:**

| Version | Behavior |
|---------|----------|
| Pre-OVS 2.0 | Microflow only, flow explosion problem |
| OVS 2.0 (2014) | Megaflow introduce, TSS classifier |
| OVS 2.4 | Subtable mask consolidation |
| OVS 2.6 | SMC (Signature Match Cache) thêm tier giữa EMC và megaflow |
| OVS 2.9 | Depth-wise lookup optimization |
| OVS 2.15 | SMC tunable via other_config |
| OVS 3.0+ | Lookup optimization continues, behavior stable |

Curriculum baseline: OVS v2.17.9. Production có thể chạy 3.3+; check version cho SMC tunable.

**So sánh với hệ khác.**

- **Linux bridge MAC learning:** wildcard concept (1 entry per MAC, not per-flow). Khác: bridge L2 only, không có L3/L4 mask. OVS megaflow flexible.
- **Cisco TCAM ACL aggregation:** TCAM cells aggregate similar pattern. Khác: hardware-level (fast, fixed), OVS megaflow software (slow, flexible).
- **Intel DPDK rte_flow API:** modern hardware offload similar wildcard. OVS hardware offload (xem Phần 9.5) kết hợp megaflow + rte_flow.

### 9.2.14.2 recirc_id, định danh recirculation cho multi-pass pipeline

> **Phạm vi:** Treatment chuyên sâu (mở rộng 2026-04-26) cho keyword `recirc_id`. Các phần liên quan: §9.2.10.2 (megaflow row có `recirc_id(0)` trong dump-flows), §9.22 multi-table pipeline (so sánh với `goto_table`/`resubmit`), §9.24 conntrack (`ct()` action sinh recirc), §9.32 classifier xlate.

**Khái niệm.** `recirc_id` là một metadata field 32-bit của OVS đánh dấu *round* recirculation mà gói đang đi qua trong datapath. Mỗi gói khi vừa nhận từ port (chưa qua bất kỳ `ct()`, `recirc()`, hoặc translation breakpoint nào) có `recirc_id = 0`. Khi xlate gặp tình huống không thể "compose" tiếp action set tĩnh trong một lần lookup duy nhất (ví dụ phải biết kết quả conntrack trước khi quyết định OF table tiếp theo), userspace `ofproto-dpif-xlate` gắn cho gói một `recirc_id` mới (uint32 khác 0, do `recirc_alloc_id()` cấp), đóng băng frozen state lại, và datapath sẽ "re-enter" classifier với key đã cập nhật `recirc_id = <id mới>`. Loại: pseudo-field metadata kernel datapath, NXM extension match field (NXM-only, không có OXM tương ứng), và là chìa khóa của mọi hành vi multi-pass trong OVS.

**Lịch sử + bối cảnh.** Trước OVS 2.2 (2014), kernel datapath không có cơ chế recirculation; mọi pipeline phải "linearize" thành một chuỗi action duy nhất. Khi conntrack được lên kế hoạch tích hợp vào OVS, vấn đề lập tức xuất hiện: kết quả `ct()` (NEW vs ESTABLISHED, NAT mapping) chỉ biết *sau khi* gói đi qua hook netfilter, nhưng OF rule cần *match trên kết quả đó* để quyết định forward hay drop. NXM_NX_RECIRC_ID(36) được introduce trong OVS 2.2 (commit pool nguồn `ofproto/ofproto-dpif-rid.c`, copyright Nicira 2014-2017) như một cơ chế chung để "bookmark" trạng thái dịch và resume từ một bảng OF khác sau khi gói đã được tái xử lý. OVS 2.5 (2016) chính thức đưa `ct()` vào kernel datapath, biến recirc_id thành nền tảng của toàn bộ stateful firewall pipeline. OVS 2.6+ mở rộng sang tunnel decap, push/pop MPLS, và `dec_ttl` exception path.

**Vị trí trong kiến trúc.** `recirc_id` nằm ở giao điểm của ba lớp: (1) classifier khoá megaflow theo `recirc_id` cùng với `in_port` và header fields; (2) `ofproto-dpif-xlate` userspace dịch OF flow gặp action không-thể-linearize sẽ `recirc_alloc_id()` và sinh frozen state; (3) datapath (kernel `net/openvswitch/actions.c` hoặc userspace `lib/dpif-netdev.c`) thực thi action `OVS_ACTION_ATTR_RECIRC` bằng cách set `key->recirc_id = N` rồi `ovs_execute_actions()` đệ quy. Megaflow trong `ovs-dpctl dump-flows` hiển thị `recirc_id(0x<hex>)` ngay đầu match list, vì đó là dimension đầu tiên TSS classifier phân nhánh.

**Vai trò.** Bốn vai trò cốt lõi: (a) **state continuation token**, gắn frozen translation state vào pool `ofproto-dpif-rid` để resume sau khi packet quay lại classifier; (b) **classifier partition**, mỗi `recirc_id` khác nhau tạo một "logical sub-table" độc lập, packet pass đầu (`recirc_id=0`) không bao giờ match flow của pass sau, ngăn cross-contamination; (c) **action chain breakpoint**, khi action set không thể compose tĩnh (vì phụ thuộc kết quả ct/decap), recirc tạo một "join point" mới; (d) **megaflow caching key**, `recirc_id` là một field trong key được cache, nên cùng một flow tuple ở pass-0 và pass-1 sinh ra hai megaflow entry độc lập, không bị aliasing.

**Vì sao sinh ra.** Động cơ thiết kế là giải bài toán "stateful match phụ thuộc trạng thái runtime" mà mô hình OpenFlow gốc không giải được. OF chuẩn cho phép `goto_table` linearize forward nhưng không có khái niệm "trở về classifier sau khi đã modify packet bằng kết quả ngoài flow table" (ví dụ kết quả nf_conntrack, kết quả tunnel decap thay đổi inner header). Nicira phát hiện rằng nếu để controller quản lý multi-pass thông qua barrier message + nhiều flow_mod, latency và state divergence sẽ phá hỏng fast path. Recirc_id là giải pháp dataplane-internal: controller cài static OF rule một lần, datapath tự động lặp lại classifier với metadata khác nhau cho từng pass.

**Vấn đề giải quyết.** Cụ thể recirc_id giải quyết bốn lớp vấn đề: (1) **Stateful firewall**, biết một gói thuộc connection ESTABLISHED hay NEW chỉ có thể sau khi conntrack lookup, mà conntrack lookup là hành động tốn time và có side-effect; phải recirc để match `ct_state` ở pass tiếp theo. (2) **Tunnel decapsulation**, khi gói VXLAN/Geneve được decap, inner header trở thành "gói mới"; recirc cho phép classifier phân loại lại theo inner header thay vì outer. (3) **MPLS/NSH push-pop**, thao tác trên label stack thay đổi packet structure; recirc tạo classifier checkpoint để xử lý phần sau đã thay đổi. (4) **`dp_hash`/`select_via_hash` group bucket**, kernel cần tính hash trước khi chọn bucket, recirc kết hợp với `recirc_update_dp_hash` đẩy việc chọn bucket vào pass tiếp theo.

**Tầm quan trọng.** Đây là cornerstone tier 1 của OVS internals. Engineer mà không nắm recirc_id sẽ không debug được: (a) bất kỳ pipeline nào dùng `ct()` (tức 100% production OVS có security group); (b) bất kỳ pipeline OVN nào (OVN logical pipeline thường tạo 5-15 lần recirc per packet do `lr_in_arp_resolve`, `ls_in_acl`, `ls_in_pre_acl`, `ls_in_pre_lb`, `ls_in_lb_aff_check`, etc.); (c) megaflow explosion debugging, vì một trong những anti-pattern phổ biến nhất là OF rule sinh recirc tổ hợp gây tăng entry theo cấp số nhân. Importance ngang `ct_state`, `megaflow`, `OFPACT_OUTPUT` trong tier cốt lõi OVS.

**Cơ chế hoạt động.** Trình tự per-packet khi pipeline gặp action cần recirc (ví dụ `ct(zone=5)` ở table 0):

1. Userspace xlate nhận packet, dịch flow table 0. Đụng `OFPACT_CT`.
2. `ofproto-dpif-xlate.c` gọi `recirc_alloc_id_ctx(frozen_state)` để cấp `recirc_id = 0xabcd1234` (uint32 khác 0). `frozen_state` chứa metadata cần cho pass tiếp theo: ofproto pointer, table_id resume, register values, stack values.
3. State được lưu vào hash table trong `ofproto/ofproto-dpif-rid.c` với khóa `recirc_id`.
4. xlate emit datapath action sequence: `ct(zone=5,commit) → set(recirc_id=0xabcd1234) → recirc()` (dạng concrete trong odp action format `OVS_ACTION_ATTR_RECIRC`).
5. Datapath (kernel `net/openvswitch/actions.c` hoặc userspace `lib/dpif-netdev.c`) thực thi: chạy conntrack hook → set `key->recirc_id = 0xabcd1234` → enqueue gói vào FIFO defer (kernel) hoặc đệ quy gọi `dp_netdev_recirculate()` (userspace).
6. Pass mới: classifier lookup lại với key có `recirc_id=0xabcd1234`. Nếu hit megaflow đã cache → fast path. Nếu miss → upcall, userspace tra `frozen_state` qua `recirc_find_id()` và resume xlate từ table_id đã lưu, lúc này `ct_state` đã có giá trị thật.
7. Pass mới tiếp tục đến output hoặc gặp recirc tiếp theo. Mỗi pass có một megaflow riêng, kernel phân biệt qua `recirc_id` field trong key.

Lưu ý quan trọng: `recirc_id` không phải counter tăng dần mỗi pass cho packet đó, mà là *handle* tới một frozen state cụ thể. Hai packet cùng đi qua cùng `ct()` action có thể nhận cùng `recirc_id` nếu frozen state giống hệt (hash dedup), tiết kiệm pool slot.

**Cách kỹ sư vận hành thành thạo.** Một engineer thành thạo `recirc_id` cần phản xạ năm thao tác: (1) Đọc `ovs-dpctl dump-flows` và lập tức nhận diện "recirc_id(0)" (pass đầu) versus "recirc_id(0x<hex>)" (pass sau), không nhầm với register match. (2) Chạy `ofproto/trace` với option `--no-megaflow` để thấy đường recirc đầy đủ kèm dòng "thawed from frozen state at table N". (3) Đếm số pass per packet trong production qua `ovs-appctl coverage/show | grep -i recirc` và hiểu ngưỡng cảnh báo (rule of thumb: trung bình > 6 recirc per packet là dấu hiệu pipeline OF rule chưa optimize). (4) Đọc `ovs-appctl dpif/show -s` để thấy "packet recirculations: N" cumulative counter, kết hợp với packet count để tính tỷ lệ. (5) Khi viết OF rule mới, tự dự đoán recirc cost trước khi commit: mỗi `ct()`, mỗi `dec_ttl()` exception, mỗi `pop_mpls`, mỗi tunnel `decap` đều tạo ít nhất một recirc.

**Phân loại.** Có thể phân loại recirc_id theo nguồn sinh: (a) **conntrack-induced**, phổ biến nhất, do `ct()`, `ct(commit)`, `ct(nat)`, hoặc `ct_clear` sinh; (b) **decap-induced**, `decap()` Geneve/VXLAN/STT/NSH, `pop_mpls`, `pop_vlan` (chỉ một số trường hợp); (c) **hash-induced**, `dp_hash`, `hash_l4` action cho group `select_via_hash`; (d) **clone/sample-induced**, `clone()`, `sample()` action tách flow để cả nhánh chính và nhánh sample tiếp tục độc lập; (e) **OVN-pipeline-induced**, không phải mechanism mới mà là pattern OVN logical flow generate hàng loạt recirc qua `ct_*` và `lr_*` stages. Theo tier importance: nhóm (a) > (e) > (b) > (c) > (d).

**Khi xảy ra sự cố.** `recirc_id` là keyword bắt buộc nhớ trong các tình huống: (a) OVS upcall rate đột biến không rõ nguyên nhân, kiểm tra trước tiên xem có rule mới sinh recirc không. (b) Megaflow count tăng phi tuyến tính so với traffic mới, recirc tổ hợp là thủ phạm số một. (c) Packet bị drop sau khi cài rule ct(), frozen state pool có thể full hoặc rule sinh recirc loop. (d) Latency p99 nhảy vọt mà bandwidth không tăng, mỗi recirc thêm một lần classifier traversal, nếu pass sau miss megaflow sẽ thêm upcall latency. (e) Kernel oops với stack trace chứa `do_execute_actions` đệ quy, recursion limit `OVS_RECURSION_LIMIT=5` (Linux kernel `net/openvswitch/actions.c`) có thể bị hit nếu rule sinh recirc loop không có exit condition.

**Liên quan mật thiết.** `recirc_id` đan xen với: (a) `ct_state`, `ct_zone`, `ct_mark`, `ct_label`, tất cả đều chỉ "có giá trị thật" sau pass-0 recirc do `ct()` (xem §9.24). (b) `frozen_state`, cấu trúc lưu trong pool `ofproto-dpif-rid` (xem §9.32 classifier xlate). (c) `goto_table` và `resubmit`, KHÁC recirc về bản chất: hai cái sau là *cùng một pass* (linearize trong xlate), recirc là *pass mới* với key cập nhật (xem §9.22). (d) `megaflow`, recirc_id là field đầu tiên trong key (xem §9.2.2). (e) `dp_hash`, `select_via_hash`, group bucket selection cần recirc (xem §9.18, §9.22). (f) OVN logical flow, mỗi stage trong logical pipeline thường tương đương 1 recirc (xem §13.7, §13.8).

**Source code tham chiếu.** Theo Rule 14 dùng function name anchor (line drift across version). Baseline OVS v2.17.9:

- `include/openvswitch/meta-flow.h`, định nghĩa MFF_RECIRC_ID. Comment block ghi `NXM: NXM_NX_RECIRC_ID(36) since v2.2`, `OXM: none`, `Access: read-only`, `Internal use only, not programmable from controller`.
- `ofproto/ofproto-dpif-rid.h`, public API: `recirc_alloc_id()`, `recirc_alloc_id_ctx(struct frozen_state *)`, `recirc_find_id()`, `recirc_free_id()`. File header copyright Nicira 2014-2017.
- `ofproto/ofproto-dpif-rid.c`, implementation pool hash table. Doc nội bộ: "A recirculation id can be any uint32_t value except 0, which is reserved for the no-recirculation case".
- `ofproto/ofproto-dpif-xlate.c`, `xlate_ctx` struct chứa fields liên quan recirc: `freezing` (bool), `recirc_update_dp_hash` (bool), `dp_hash_alg`, `dp_hash_basis`, `frozen_actions` (struct ofpbuf). Function `ctx_trigger_recirculate_with_hash()` (~line 766 v2.17.9). Hằng số `MAX_DEPTH = 64` cho recursion translation.
- `lib/odp-execute.c`, userspace datapath thực thi `OVS_ACTION_ATTR_RECIRC`.
- `lib/dpif-netdev.c`, `dp_netdev_recirculate()` đệ quy gọi lại classifier per-packet.
- Linux kernel `net/openvswitch/actions.c`, `OVS_ACTION_ATTR_RECIRC` handler trong `do_execute_actions()`. Hằng số `OVS_RECURSION_LIMIT = 5` và `DEFERRED_ACTION_FIFO_SIZE = 10` định giới hạn recirc depth + buffer FIFO defer.
- Linux kernel `net/openvswitch/flow.h`, struct `sw_flow_key` chứa field `recirc_id` ở vị trí đầu key.

**Trường hợp sự cố thực tế.** Dựa trên pattern revalidator storm forensic phổ biến trong production OpenStack/OVN environment (xem §9.26 cho anatomy template chi tiết): một deployment Neutron với security group dùng `ct()` cho stateful firewall, sau khi tăng số tenant từ 200 lên 1000, `ovs-vswitchd` CPU lên 100% và packet drop rate tăng. Triệu chứng: `ovs-appctl dpif/show -s` báo `packet recirculations` từ 50K/sec lên 800K/sec; `ovs-dpctl dump-flows | wc -l` cho số megaflow tăng từ ~5K lên ~80K. Root cause thường là OF rule generated bởi neutron OVS agent có pattern `ct(commit,zone=N)` với zone khác nhau per port, kết hợp với recirc đầu pipeline cho ARP resolve, tạo tổ hợp `(zone × in_port × ct_state)` blow up megaflow. Mitigation: gộp zone về per-tenant thay vì per-port, giảm recirc point từ 4 xuống 2 trong pipeline. Lesson: mỗi recirc point thêm vào nhân số megaflow theo (số giá trị metadata mà pass tiếp theo match), không phải cộng.

**Lỗi thường gặp + tín hiệu chẩn đoán.**

- *Triệu chứng* Megaflow count tăng đột biến sau commit OF rule mới. *Nguyên nhân* Recirc tổ hợp với metadata khác nhau per-port/per-zone. *Chẩn đoán* `ovs-dpctl dump-flows | awk '{print $1}' | sort | uniq -c | sort -rn`, nhóm theo `recirc_id` xem pass nào có nhiều entry nhất. *Khắc phục* Gộp zone hoặc tách rule để giảm số metadata dimension trong pass đó.
- *Triệu chứng* Kernel log `nf_conntrack: table full, dropping packet` kèm OVS upcall storm. *Nguyên nhân* Mỗi recirc qua `ct()` tạo conntrack lookup; nếu pipeline có 2-3 recirc qua ct, nf_conntrack table chịu áp lực gấp 2-3 lần expected. *Chẩn đoán* `cat /proc/sys/net/netfilter/nf_conntrack_count` so với `nf_conntrack_max`. *Khắc phục* Tăng `nf_conntrack_max` hoặc loại bỏ recirc qua `ct()` thừa.
- *Triệu chứng* `ofproto/trace` báo `recursion too deep` hoặc kernel oops `OVS_RECURSION_LIMIT exceeded`. *Nguyên nhân* OF rule sinh recirc loop không có exit condition (pass N gọi recirc về table cũ pass N-1). *Chẩn đoán* Trace từng pass, tìm rule có action recirc về table sớm hơn current. *Khắc phục* Thêm match condition để pass sau không trigger lại cùng recirc point.
- *Triệu chứng* Frozen state pool grows unbounded, memory leak ovs-vswitchd. *Nguyên nhân* Bug cũ pre-OVS 2.13 khi ofproto bị remove mà không free recirc_id. *Chẩn đoán* `ovs-appctl memory/show` so sánh trước và sau `ovs-vsctl del-br`. *Khắc phục* Upgrade OVS lên ≥ 2.13.
- *Triệu chứng* Latency p99 tăng đột biến sau khi enable conntrack ACL. *Nguyên nhân* Mỗi packet thêm 1-2 recirc, nếu cache miss thì thêm 1-2 upcall latency. *Chẩn đoán* `ovs-appctl coverage/show | grep upcall` so trước/sau enable. *Khắc phục* Pre-warm cache hoặc giảm số recirc point.

**So sánh với hệ khác.** Các hệ khác giải bài "multi-pass packet processing" theo cách khác:

- **Linux netfilter NF_HOOK** chained (PREROUTING → FORWARD → POSTROUTING): mỗi hook là một "pass" theo nghĩa pipeline, nhưng metadata không persist explicit như `recirc_id`. Khác biệt cơ bản: netfilter pipeline tĩnh do kernel define, OVS recirc động do OF rule define.
- **Cisco IOS-XR feature pipeline** trên ASIC (egress recirculation cho VXLAN encap): tương đương về nguyên tắc nhưng implementation hardware. recirc_id trong OVS gần với khái niệm "loopback FIFO" trên chip Trident/Tomahawk.
- **P4 architecture parser-deparser-parser** (re-parse sau header rewrite): khác chỗ P4 cần programmer khai báo explicit recirc trong P4 code, OVS recirc là implicit do xlate sinh.
- **VPP graph node re-enqueue** (FD.io): tương đương. Mỗi graph node là một "stage", packet được re-enqueue vào graph với metadata cập nhật. recirc_id ≈ VPP `vnet_buffer` opaque field.
- **OpenFlow `goto_table` thuần** (không recirc): chỉ linearize trong cùng một xlate pass, không thay đổi packet key. Đây là điểm dễ nhầm lẫn nhất với recirc, vì `goto_table` rẻ còn recirc đắt.

### 9.2.14.3 openvswitch.ko (kernel datapath), các phương diện bổ sung

**Khái niệm.** `openvswitch.ko` là Linux kernel module implementing fast-path packet forwarding cho OVS. Loại: Linux kernel module (.ko file).

**Lịch sử + bối cảnh.** Initial 2009 by Nicira. Mainlined Linux 3.3 (2012). Stable since.

**Tầm quan trọng.** Tier 1 cornerstone. Without openvswitch.ko, OVS chỉ chạy userspace (slow path), không thể production-grade.

**Cơ chế hoạt động.** Module register hooks tại Linux network stack (`net_device` interception). Per-packet:
1. Receive packet from netif_rx
2. Extract flow key (5-tuple + metadata)
3. Lookup megaflow table (TSS classifier)
4. If miss → upcall to userspace via Netlink
5. If hit → execute action (forward, drop, modify)

**Cách quan sát + xác minh.** `lsmod | grep openvswitch`, `cat /proc/net/openvswitch/*` (if available), `ovs-dpctl show`.

**Source code tham chiếu.** Linux kernel `net/openvswitch/datapath.c`, `flow_table.c`, `actions.c`. Userspace counterpart `lib/dpif-linux.c`.

**Trường hợp sự cố thực tế.** Production case 2023: kernel module crash sau Linux upgrade từ 5.4 → 5.15 do API drift; `openvswitch.ko` fail to load. Fix: rebuild against new kernel headers.

**So sánh với hệ khác.** Linux native bridge (`bridge` module): simpler L2-only equivalent. (Các datapath thay thế ngoài kernel module truyền thống nằm ngoài phạm vi curriculum.)

### 9.2.14.4 conntrack execute, các phương diện bổ sung

**Khái niệm.** `conntrack_execute()` là kernel function trong `net/openvswitch/conntrack.c` execute conntrack action (ct() in OF flow). Per-packet conntrack lookup + state update.

**Tầm quan trọng.** Tier 1 cornerstone OVS conntrack pipeline. Cross-link `9.24 ct_state` (consumer of state set by execute).

**Cơ chế hoạt động.** Triggered by `OFPACT_CONNTRACK` action:
1. Lookup nf_conntrack table với 5-tuple
2. If first packet → create new entry (state=NEW)
3. If existing → update state (NEW→ESTABLISHED, etc.)
4. Set packet's ct_state, ct_zone, ct_mark, ct_label registers
5. Optionally commit entry, NAT rewrite, helper invoke

**Source code tham chiếu.** `net/openvswitch/conntrack.c` `ovs_ct_execute()`. Linux `nf_conntrack_in()` underlying. `lib/conntrack.c` userspace fallback (DPDK datapath).

**Lỗi thường gặp + tín hiệu chẩn đoán.**
- Mode A: Conntrack table full → diagnostic `nf_conntrack_count > nf_conntrack_max`. Recovery: increase max sysctl.
- Mode B: Conntrack zone collision → ct_state INVALID. Recovery: zone partition.

**So sánh với hệ khác.** Linux iptables `-m conntrack` consumes same kernel conntrack. Khác: iptables expression chain hook; OVS conntrack action OF flow stage.

revalidator + handler thread OVS 2.0 introduce, 2.4 configurable. ukey OVS 2.0 + UFID OVS 2.4. fast/slow path conceptual. action_set + write_metadata OF 1.1.


### So sánh với hệ khác

- revalidator/handler thread ≈ Linux net/core RX/TX softirq

- fast/slow path ≈ Linux conntrack fast path / netfilter slow path
- write_metadata ≈ Cisco IOS forwarding-class
- br-int patch port ≈ Linux veth pair

## 9.12.1 Khái niệm

`Upgrade choreography` là quy trình production đổi version OVS userspace + kernel module + OVSDB schema **đồng thời** trên một fleet hosts mà KHÔNG drop data plane traffic. Khác với "package update" thông thường (apt upgrade kèm restart): rolling restart phối hợp 3 thành phần (ovsdb-server, ovs-vswitchd, openvswitch.ko) theo thứ tự nghiêm ngặt để fast-path forwarding tiếp tục chạy ngay cả lúc daemon restart. Phạm trù: operational procedure + state-machine choreography (không phải CLI tool, không phải OVSDB schema row).

---

## 9.12.2 Lịch sử + bối cảnh

OVS 1.x (2009-2013) yêu cầu reboot host khi đổi kernel module → mọi VM mất kết nối nhiều giây. OVS 2.0 (Jan 2014) introduce `--monitor` cho ovs-vswitchd graceful restart (auto-respawn khi crash), giảm downtime restart từ giây xuống dưới giây. OVS 2.6 (Sep 2016) thêm OVSDB cluster Raft + `cluster/leave` + `ovsdb-tool join-cluster` enable rolling upgrade cluster theo từng node. OVS 2.10+ (~2018) refine schema-conversion atomicity. OVS 2.13 (Feb 2020) optimize megaflow eviction để minimize cache invalidation lúc reload. OVS 2.17.0 (17 Feb 2022) hiện là LTS phổ biến cho OpenStack Yoga/Zed và OVN 22.03. OVS 3.0.0 (15 Aug 2022) đổi container iteration macro UB-safe + OVSDB relay transaction history; tương thích ABI nhưng cần recompile clients. Nhu cầu upgrade rolling sinh ra từ thực tế production: cluster OpenStack/OVN với 100+ compute node không thể full-fleet restart đồng thời (mất quorum, mất tenant traffic).

---

## 9.12.3 Vị trí trong kiến trúc

Upgrade choreography là cross-cutting concern, tác động đến cả 3 thành phần OVS đồng thời:

- **OVSDB layer (`ovsdb-server`)**: schema conversion + state migration; persist data tại `/etc/openvswitch/conf.db`.
- **Userspace daemon (`ovs-vswitchd`)**: process lifecycle + ofproto state restore; reload từ OVSDB sau restart.
- **Kernel module (`openvswitch.ko`)**: kernel data plane; module reload ảnh hưởng megaflow cache nhưng không ảnh hưởng conntrack state (state nằm netfilter `nf_conntrack` table độc lập).

Trong stack lifecycle: upgrade choreography đứng ở tầng "operational procedure" trên các binary nhị phân và file system artifacts (package manager, systemd unit, kernel module file).

---

## 9.12.4 Vai trò

Upgrade choreography đóng 4 vai trò chính:

1. **Atomic version transition**: đảm bảo ovsdb-server new + ovs-vswitchd new + openvswitch.ko new chuyển trạng thái nhất quán, không có window trung gian không tương thích.
2. **State preservation**: bảo toàn flow rule, port, conntrack state, mirror config, QoS queue qua restart cycle.
3. **Data plane continuity**: fast-path forwarding tiếp tục cho existing flow khi daemon đang restart (sub-second window).
4. **Cluster coordination** (Raft cluster): upgrade một node tại một thời điểm trong khi cluster vẫn đạt quorum (N/2+1).

---

## 9.12.5 Vì sao sinh ra

Trước OVS rolling restart, production OpenStack/OVN deployment đối mặt 4 pain point thực tế: (1) **maintenance window quá dài**, full-fleet restart 100+ compute node mất 30-60 phút mất kết nối, vi phạm SLA. (2) **Quorum loss khi restart đồng thời cluster Raft** ovsdb cluster 3 node restart đồng thời = mất quorum = read-only mode = OpenStack neutron block-create. (3) **VM connectivity loss khi kernel module reload** ảnh hưởng SSH session, NFS mount, distributed application replication. (4) **Schema migration race**, khi binary mới start trước schema convert → crash loop. Pfaff + Pettit thiết kế rolling pattern năm 2014-2016 để address cả 4 pain point đồng thời, tham chiếu kinh nghiệm từ telco/cloud production.

---

## 9.12.6 Vấn đề giải quyết

Cụ thể bốn vấn đề kỹ thuật được rolling upgrade address: (1) **Schema-binary mismatch deadlock**: nếu binary mới đọc DB cũ schema chưa convert, server fail validate schema, exit; nếu binary cũ đọc DB mới đã convert, fail forward-compatible parse, exit. Giải pháp: convert TRƯỚC start binary mới (Rule 1 schema-first). (2) **Kernel-userspace ABI mismatch**: Netlink genl family struct version mismatch → `genlmsg_parse` fail; fix: kernel module + userspace cùng major version (Rule 3 không đồng thời). (3) **Megaflow cache invalidation cascade**: restart ovs-vswitchd flush all cached megaflow → first packet lookup miss → upcall → revalidator install → 100K megaflow burst trong 1-2 giây → CPU spike; fix: `--monitor` graceful + revalidator pacing. (4) **OVSDB Raft majority loss**: restart 2/3 node đồng thời = lost quorum = no leader; fix: rolling per-node + wait `cluster/status` "leader: yes" + lag <50ms.

## 9.12.7 Tầm quan trọng

Upgrade choreography là kỹ năng cốt lõi tuyệt đối cho mọi engineer vận hành OVS/OVN production. Engineer không nắm choreography sẽ không thực hiện được: (a) bất kỳ security patch khẩn cấp nào (CVE OVS xuất hiện 2-3 lần/năm); (b) major version migration cho OpenStack release cycle (Yoga → Zed → Antelope mỗi 6 tháng); (c) kernel upgrade chu kỳ (Ubuntu LTS HWE rotation 6 tháng); (d) downgrade emergency khi binary mới có regression. Mọi distro release cycle (RHEL/Ubuntu LTS) + OpenStack release + OVN release đều có gắn version OVS cụ thể cần engineer hiểu choreography để rollout mà không gây outage. Mức độ quan trọng ngang `megaflow`, `ct_state`, `dpif` trong nhóm cốt lõi OVS.

## 9.12.8 Cơ chế hoạt động

Quy trình 5 bước upgrade single-node atomic theo thứ tự nghiêm ngặt (compass Ch P):

**Bước 1 — Install packages KHÔNG restart daemon**:

```bash
[host]# apt install openvswitch-switch openvswitch-common
```

Package manager cài binary mới vào filesystem, NHƯNG daemon đang chạy (`ovsdb-server`, `ovs-vswitchd`) vẫn dùng binary cũ đã loaded vào memory. systemd unit không restart. Step này safe trên live host.

**Bước 2 — Restart ovsdb-server graceful**:

```bash
[host]# ovs-appctl -t ovsdb-server exit --cleanup
[host]# systemctl start ovsdb-server
```

`exit --cleanup` flush pending transaction trước exit, đảm bảo file `conf.db` consistent. systemd start ovsdb-server với binary mới. Down time ~100-500ms tùy schema size. Trong window này, ovs-vswitchd vẫn forwarding (data plane continue).

**Bước 3 — Schema conversion nếu cần**:

```bash
[host]# ovsdb-tool needs-conversion /etc/openvswitch/conf.db \
         /usr/share/openvswitch/vswitch.ovsschema
```

Nếu output "yes":

```bash
[host]# ovsdb-tool convert /etc/openvswitch/conf.db \
         /usr/share/openvswitch/vswitch.ovsschema
```

Source: `ovsdb/ovsdb-tool.c::do_convert` (line 449 v2.17.9) implement schema migration: parse old DB JSON, validate against new schema, transform record format (column add/remove/type change), atomic rename `conf.db.new` → `conf.db`. Backward-compatible cho minor version; major version có thể yêu cầu manual data transformation.

**Bước 4 — Restart ovs-vswitchd atomic**:

```bash
[host]# systemctl reload-or-restart openvswitch-switch
```

Compass Ch P pattern: `--monitor` mode cho ovs-vswitchd. New instance fork, attach OVSDB connection, restore ofproto state từ DB, sau đó signal old instance exit. Data plane không drain vì kernel megaflow cache độc lập với userspace process; megaflow tiếp tục match cho existing flow trong window restart 2-5 giây.

**Bước 5 — Kernel module reload (chỉ khi kernel module upgrade)**:

```bash
[host]# rmmod openvswitch
[host]# modprobe openvswitch
```

Compass Ch P: reload kernel module **brief evicts megaflow cache** (cache nằm trong kernel module memory) NHƯNG **không drop existing connections** vì conntrack state lives separately trong `nf_conntrack` netfilter table. Window: 1-3 giây fast-path miss → upcall → reinstall megaflow.

## 9.12.9 Cách kỹ sư vận hành thành thạo

Kỹ sư thành thạo upgrade choreography cần phản xạ 6 thao tác:

1. **Read pre-upgrade state**: `ovs-vsctl get Open_vSwitch . ovs-version` + `ovs-vsctl get Open_vSwitch . db-version` cho biết current version OVSDB schema. Combine với `ovs-vswitchd --version` cho userspace + `cat /sys/module/openvswitch/version` cho kernel module.
2. **Drain orchestrator scheduler**: pre-upgrade, mark host "scheduling=disabled" trong OpenStack Nova hoặc cordon Kubernetes node. Lý do: tránh new VM/Pod được schedule lên host trong window upgrade.
3. **Backup DB pre-upgrade**: `ovsdb-client backup unix:/var/run/openvswitch/db.sock OVN_Northbound > /backup/conf.db.$(date +%F).bak`. Lý do: insurance cheap, recover được khi schema convert fail.
4. **Apply upgrade 5-step**: theo thứ tự nghiêm ngặt §9.12.8.
5. **Verify post-upgrade**: `ovs-vsctl show` (bridge list intact?), `ovs-ofctl dump-flows br-int | wc -l` (rule count match pre-upgrade?), `ovs-appctl dpif/show` (datapath state OK?), `journalctl -fu openvswitch-switch | tail -20` (no error log?).
6. **Re-enable scheduling**: mark host "scheduling=enabled". Done.

Decision tree khi gặp issue:
- Pre-upgrade `ovsdb-tool needs-conversion` returns "no" → skip schema convert step (Bước 3).
- Kernel module version match userspace → skip kernel reload (Bước 5).
- Cluster Raft mode → use cluster choreography §9.12.20 thay 5-step single-node.

## 9.12.10 Phân loại

`Upgrade choreography` là **operational procedure**, không phải:
- CLI tool standalone (chỉ là sequence of commands)
- OVSDB schema row (không có row trong database)
- OpenFlow primitive
- Daemon hoặc kernel feature

Phân loại theo scope:
- **Single-node rolling**: 1 host upgrade theo §9.12.8 5-step (single-host scope).
- **Cluster Raft rolling**: N node OVSDB cluster upgrade từng node leave-rejoin (cluster scope, §9.12.20).
- **Fleet rolling**: N host trong OpenStack/Kubernetes fleet upgrade tuần tự, mỗi host theo single-node procedure (fleet scope).

Phân loại theo trigger:
- **Security patch** (CVE): minor version, schema thường không đổi, nhanh.
- **Minor version**: schema có thể có column add (compat), feature add.
- **Major version** (2.17 → 3.0): schema breaking change có thể, ABI re-compile required, planned maintenance window.

## 9.12.12 Khi xảy ra sự cố

Operator on-call cần nhớ upgrade choreography khi gặp 4 triệu chứng điển hình:

**Symptom A: Schema conversion fail**
- Trigger: `ovsdb-tool convert` báo lỗi "constraint violation" hoặc "schema mismatch"
- Cause: custom column hoặc breaking schema change
- Recovery: `ovsdb-client restore unix:/var/run/openvswitch/db.sock < /backup/...` (restore pre-upgrade DB), rollback binary version, investigate

**Symptom B: VM/container mất kết nối > 10s sau restart**
- Trigger: SSH session drop, NFS mount stale, application timeout
- Cause: kernel module mismatch (Rule 3 violation), conntrack state lost, hoặc bond flap
- Recovery: check `dmesg | grep openvswitch`, `lsmod | grep openvswitch`, restart bond peer

**Symptom C: ovn-controller disconnect SBDB sau upgrade**
- Trigger: `ovn-controller` log "ssl: peer certificate verification failed"
- Cause: TLS cert expiry coincide với upgrade window
- Recovery: rotate cert per Phần 20.1 §20.1.15, restart ovn-controller

**Symptom D: Megaflow CPU spike sustained > 5 phút post-restart**
- Trigger: `top` cho thấy ovs-vswitchd CPU > 80%, `coverage/show upcall_flow_add` rate > 1000/s
- Cause: revalidator storm post-cache-flush (Phần 9.26)
- Recovery: tune `revalidator/wait`, throttle rule install rate, hoặc accept transient

## 9.12.13 Liên quan mật thiết tới

`Upgrade choreography` tightly coupled với 6 thành phần:

1. **`ovsdb-tool`** (Phần 9.31): `convert` + `needs-conversion` + `compact` + `join-cluster` đều invoke trong choreography.
2. **`ovsdb-server`** + Raft cluster (Phần 10.1): cluster mode yêu cầu rolling per-node thay single-node 5-step.
3. **`ovs-vswitchd`** monitor mode (Phần 9.1): `--monitor` graceful restart cơ chế chính cho data plane continuity.
4. **`openvswitch.ko`** kernel module (Phần 9.2): module reload state lifecycle, megaflow cache eviction.
5. **`mTLS` cert** (Phần 9.10 + 20.1): cert expiry check là pre-upgrade checklist mandatory.
6. **OpenStack Nova / Kubernetes scheduler**: drain + cordon coordination.

## 9.12.17 Trường hợp sự cố thực tế

Production case (curated từ public bug pattern): OpenStack Yoga deployment Q4-2023, 50-compute-node fleet, plan upgrade OVS 2.17.4 → 2.17.7 (security CVE patch). Bước 4 (restart ovs-vswitchd) trên compute02 báo timeout 30s; systemd unit fail. Investigation:

```
[compute02]# journalctl -fu openvswitch-switch --since "5 min ago"
ovs-vswitchd: ovsdb-server connection lost, reconnecting
ovs-vswitchd: ofproto: failed to register controller
ovs-vswitchd: bridge: failed to set up bridge
systemd: openvswitch-switch.service: Main process exited, code=exited, status=1
```

Root cause: `ovs-vswitchd` post-restart không kết nối được `ovsdb-server` vì `/var/run/openvswitch/db.sock` permission bị thay đổi sau bước 2 graceful exit. Cụ thể: systemd unit cho `ovsdb-server` 2.17.7 set socket permission `0660 root:openvswitch`, nhưng `ovs-vswitchd` chạy user `openvswitch`; 2.17.4 set `0666` permissive. Đây là regression trong distro packaging (Ubuntu 22.04 Yoga maintainer adjustment), không phải upstream OVS bug.

Diagnostic chain:
- `ls -la /var/run/openvswitch/db.sock` → permission 0660
- `ps -ef | grep ovs-vswitchd` → running as user `openvswitch`
- `id openvswitch` → groups `openvswitch` only, không thuộc group `root`

Fix: tạm thời `chmod 0666 /var/run/openvswitch/db.sock` để unblock fleet upgrade, sau đó adjust systemd unit overlay `/etc/systemd/system/ovsdb-server.service.d/permission.conf` đặt `ExecStartPost=chmod 0666 /var/run/openvswitch/db.sock`.

Lesson: distro packaging có thể introduce regression không có trong upstream NEWS; pre-upgrade staging environment test mandatory, không trust release notes mặc định.

## 9.12.19 Lỗi thường gặp + tín hiệu chẩn đoán

5 failure mode điển hình và recovery path:

**Failure 1 — Schema conversion fail (constraint violation)**
- Signal: `ovsdb-tool convert` exits non-zero, log `constraint violation: column 'X' type mismatch`
- Cause: custom column trong production DB, hoặc breaking schema change
- Recovery: restore backup DB; nếu cần convert, contact distro maintainer cho migration script

**Failure 2 — Bond flap during restart (LACP timeout)**
- Signal: `ovs-vsctl list Interface | grep -E "lacp_status|name"` báo `expired`; ifup/down log
- Cause: LACP slow-timer 90s vs daemon restart 5-30s window làm peer mark "down"
- Recovery: pre-upgrade switch LACP fast-timer; post-upgrade switch lại slow-timer

**Failure 3 — Flow rule loss (grammar removed)**
- Signal: `ovs-ofctl dump-flows br-int | wc -l` post-upgrade < pre-upgrade
- Cause: OF action/match grammar deprecated trong major version
- Recovery: patch controller (OVN northd hoặc Faucet) trước OVS, regenerate rules

**Failure 4 — OVN-controller disconnect SBDB (TLS expiry)**
- Signal: `ovn-controller` log `ssl: peer certificate verification failed: certificate has expired`
- Cause: cert expiry coincide với upgrade maintenance window
- Recovery: rotate cert per Phần 20.1 §20.1.15, restart ovn-controller

**Failure 5 — Kernel module verification reject (signing strict)**
- Signal: `dmesg | grep openvswitch` report `module verification failed: signature and/or required key missing`
- Cause: kernel 5.15+ strict signing; OVS module compile out-of-tree không signed bởi distro key
- Recovery: install `linux-modules-extra-$(uname -r)` từ Ubuntu repo (đã signed), HOẶC disable signing strict via boot param `module.sig_enforce=0` (KHÔNG khuyến khích production)

## 9.12.20 Cluster upgrade choreography (OVSDB Raft HA)

Cho OVSDB Raft clustered (Phần 10.1), single-node 5-step KHÔNG áp dụng. Cluster choreography theo per-node sequence:

1. **Identify non-leader node**: `ovs-appctl -t ovsdb-server cluster/status Open_vSwitch` cho biết role; chọn follower trước, leader cuối cùng (giảm election event).
2. **Drain client connection**: optional, redirect client sang node khác qua DNS hoặc load balancer config.
3. **Leave cluster**: `ovs-appctl -t ovsdb-server cluster/leave Open_vSwitch`. Wait đến khi `cluster/status` báo node "left".
4. **Stop ovsdb-server**: `systemctl stop ovsdb-server`.
5. **Install binary mới**: `apt install openvswitch-common`.
6. **Reset DB**: **xoá** `/etc/openvswitch/conf.db` (compass Ch N: "Once a server leaves a cluster it cannot rejoin; create a fresh member").
7. **Rejoin cluster**: `ovsdb-tool join-cluster /etc/openvswitch/conf.db Open_vSwitch \
    tcp:<this-node-ip>:6643 tcp:<remaining-leader-ip>:6643`.
8. **Start ovsdb-server**: `systemctl start ovsdb-server`. Wait sync via `cluster/status` "leader: <leader-name>" + lag < 50ms.
9. **Repeat per node**: until all N node upgraded.
10. **Verify quorum + leader stable**: `cluster/status` from any node show consistent.

Source: `ovsdb/ovsdb-tool.c::do_join_cluster` (line 326 v2.17.9) implements join logic. Critical path: maintain N/2+1 quorum throughout; với N=3 cluster, không bao giờ stop > 1 node đồng thời.

## 9.12.21 Downgrade strategy

Downgrade phức tạp hơn upgrade vì schema forward conversion thường irreversible. Best practice:

```bash
# Pre-upgrade: backup DB explicit
[host]# ovsdb-client backup unix:/var/run/openvswitch/db.sock Open_vSwitch \
  > /backup/conf.db.pre-upgrade-$(date +%F).bak

# Downgrade if needed:
[host]# systemctl stop ovs-vswitchd ovsdb-server
[host]# apt install openvswitch-switch=<old-version>
[host]# rm /etc/openvswitch/conf.db
[host]# ovsdb-tool create /etc/openvswitch/conf.db /usr/share/openvswitch/vswitch.ovsschema
[host]# ovsdb-client restore unix:/var/run/openvswitch/db.sock Open_vSwitch < /backup/conf.db.pre-upgrade-...bak
[host]# systemctl start ovsdb-server ovs-vswitchd
```

Cluster downgrade thậm chí phức tạp hơn: dùng `ovsdb-tool::do_convert_to_standalone` (line 1673 v2.17.9) chuyển cluster DB → standalone DB → reinstall old binary → recreate cluster.

## 9.12.22 So sánh với hệ khác

`Upgrade choreography` của OVS có analogue ở các stack khác:

- **Linux bridge native**: KHÔNG có rolling restart. Kernel bridge driver upgrade cùng kernel reboot. Downtime: full reboot ~30-60s. OVS rolling là lợi thế lớn so với native.
- **VPP graph node version**: VPP có graph node version trong control thread; upgrade graph node không cần restart toàn bộ vpp-agent. Khác OVS ở chỗ VPP focus là performance (DPDK polling); upgrade phức tạp hơn vì PMD thread phải coordinate.
- **Cisco IOS-XE In-Service Software Upgrade (ISSU)**: ISSU là benchmark industrial cho rolling upgrade; sử dụng standby supervisor module + state replication. Khác OVS ở chỗ ISSU yêu cầu hardware redundancy (dual supervisor); OVS rolling chỉ cần single-host nhờ kernel-userspace separation.
- **Juniper Junos NSR/NSSU** (Non-Stop Routing / Non-Stop Software Upgrade): tương đương ISSU; routing engine A serve trong khi B upgrade. Hardware redundancy.
- **Kubernetes kubelet rolling**: kubelet drain + cordon + upgrade là pattern tương đồng cho compute node; OVS choreography piggy-back trên Kubernetes lifecycle khi deploy OVN-as-CNI.
- **PostgreSQL major upgrade (`pg_upgrade`)**: schema convert qua `pg_upgrade` tool tương đồng với `ovsdb-tool convert` ý tưởng nhưng PostgreSQL yêu cầu downtime full database; OVSDB convert sub-second.

## Hiểu sai phổ biến + Key Topic

> **Hiểu sai phổ biến 1:** *"Skip major version (2.17 → 3.3) safe với `apt upgrade`."*
>
> **Thực tế:** OVS release cycle 6-12 tháng, migration path từ 1-2 major previous. Skip nhiều version đụng untested schema conversion chain. Pattern safe: incremental 2.17 → 2.18 → ... → 3.3 qua distro backport hoặc manual build.

> **Hiểu sai phổ biến 2:** *"Rolling restart = no downtime."*
>
> **Thực tế:** Sub-second packet loss khi flow cache invalidate. SLA 99.99% budget 5-second/restart. 10 chassis × 5s = 50s cumulative. Tunnel pre-warmed + CT state preserved (OVS 2.15+) giảm impact, nhưng KHÔNG zero-downtime.

> **Hiểu sai phổ biến 3:** *"Cluster Raft node leave-rejoin tương đương restart binary."*
>
> **Thực tế:** Compass Ch N: "Once a server leaves a cluster it cannot rejoin, create a fresh member." Sau `cluster/leave`, phải xoá DB và `ovsdb-tool join-cluster` recreate, không thể chỉ restart binary giữ nguyên DB.

> **Chủ đề trọng tâm:** "Golden 3 rules + 5-step choreography" zero data plane loss. Golden rules: (1) Schema first trước binary. (2) One daemon at a time. (3) Không đồng thời OVS + kernel. 5-step: install / restart-ovsdb / convert-schema / restart-ovs-vswitchd / reload-kernel-module. Cluster mode: per-node leave-recreate-rejoin với quorum maintained.

## Điểm cốt lõi cần nhớ

Thứ nhất, OVS upgrade được thiết kế không mất data plane qua rolling restart 5-step phối hợp 3 thành phần (OVSDB + ovs-vswitchd + kernel module).

Thứ hai, ba golden rule (schema first, one daemon at a time, không đồng thời OVS + kernel) tránh phần lớn upgrade failure.

Thứ ba, cluster Raft upgrade khác single-node: node rời cluster KHÔNG thể rejoin với DB cũ, phải recreate qua `ovsdb-tool::do_join_cluster`.

Thứ tư, backup DB trước mọi production change là insurance cheap; restore qua `ovsdb-client restore` với pre-upgrade backup file.

Thứ năm, Anatomy pre-upgrade checklist 9-attribute (version + schema + rules + ofconns + upcall + backup + peer compat + window). Go/no-go decision.

Thứ sáu, skip major version = risk untested schema conversion chain. Incremental safe pattern: 2.17 → 2.18 → ... → 3.x qua distro backport hoặc manual.

Thứ bảy, rolling restart sub-second data plane loss. Budget SLA math nếu 99.99%: 5-second × N-host cumulative.

Thứ tám, version drift table 2.17.0 (Feb 2022) → 3.0.0 (Aug 2022): OVS 3.0 thay đổi container iteration macro UB-safe, recompile clients required; OVSDB schema 8.3.x → 8.4.x.

Thứ chín, source code anchor `ovsdb/ovsdb-tool.c::do_convert` (line 449 v2.17.9) là entry point schema conversion; `do_join_cluster` (line 326) cho cluster choreography.

Thứ mười, downgrade phức tạp hơn upgrade do schema forward conversion irreversible; cluster mode dùng `do_convert_to_standalone` (line 1673) làm bridge cho downgrade path.

## Learning objectives

After completing this part, the engineer can:

1. **Understand** the role of the VTEP schema and `vtep-ctl` for integrating a **hardware VXLAN gateway** (Arista, Mellanox, Cumulus) into an OVN cluster. This integration lets bare-metal servers that run outside any hypervisor (legacy storage, mainframe, network appliance) share the same L2 overlay as VMs running on a hypervisor.
2. **Apply** the workflow `add-ps`, then `add-ls`, then `bind-ls` to bind one logical switch in OVN to a specific VLAN port on a physical hardware switch.
3. **Analyze** the seven main tables in the VTEP schema (Global, Manager, Logical_Switch, Logical_Binding_Stats, Physical_Switch, Physical_Port, Physical_Locator) and the relationships between them.
4. **Evaluate** the trade-off between a hardware VTEP gateway (line-rate VXLAN, vendor lock-in) and a software gateway (a chassis running `ovn-controller` on a server, higher flexibility, lower throughput).

Prerequisite knowledge: understand the VXLAN packet format from Part 11.0, understand the OVSDB schema from Part 10.0, and have a basic concept of an L2 gateway.

## 9.29.1 Why VTEP exists, the bare-metal-in-overlay problem

OVN builds an L2 overlay using tunnels (Geneve or VXLAN) between hypervisor chassis. A VM on a hypervisor relies on `ovn-controller` to install flow rules on `br-int` that encapsulate and decapsulate the tunnel header. Every VM on the same logical switch sees one another as part of one L2 broadcast domain, regardless of which chassis hosts them.

But what about **bare-metal servers**, that is, physical machines without a hypervisor and without `ovn-controller`? Typical examples include:

- A storage NAS or SAN appliance that cannot run a Linux hypervisor.
- An IBM Z mainframe running legacy z/OS.
- A network appliance (a hardware firewall, an IDS, a hardware load balancer).
- An older physical server not yet migrated to virtualization.

These devices need to connect to the same L2 overlay as VMs on a hypervisor. The solution is a **hardware VTEP**, a high-end physical switch with line-rate VXLAN encapsulation and decapsulation capability. The hardware VTEP bridges the VLAN of the physical port that connects to the bare-metal server with the VXLAN VNI of the OVN logical switch.

**Hardware VTEP architecture:**

```
   [bare metal server]                     [hypervisor]
         |                                      |
   [physical port]                       [veth port]
         |                                      |
   +-----+------+        VXLAN          +-------+--------+
   | HW VTEP    |<====== tunnel ======> |  br-int        |
   | (vendor)   |                       | (ovn-ctrl)     |
   +------------+                       +----------------+
         ^                                      ^
         |                                      |
         | OVSDB protocol (port 6640)           |
         |                                      |
   +-----+----------------------------+         |
   | ovsdb-server (VTEP schema)       |         |
   | + ovn-controller-vtep daemon     +---------+
   +----------------------------------+
```

`ovn-controller-vtep` is a special daemon (DIFFERENT from `ovn-controller`). It runs on a management server and connects to:

- The hardware VTEP switch through the VTEP schema (the OVSDB protocol).
- The OVN SB DB through the usual interface.

The daemon reads `Logical_Switch_Port` rows of `type=vtep` in OVN NB, then writes the matching configuration into the VTEP schema. The hardware switch then bridges VLAN to VXLAN automatically.

`vtep-ctl` is the CLI tool to administer the VTEP schema (read and write). It is the VTEP-schema counterpart of `ovs-vsctl` for the `Open_vSwitch` schema. The two tools share syntax and the same transactional pattern.

## 9.29.2 The VTEP schema, seven main tables

The VTEP schema is defined in `/usr/share/openvswitch/vtep.ovsschema` (OVS 2.17.9). The seven main tables are:

| Table | Purpose |
|-------|---------|
| `Global` | Singleton root, holds schema metadata and a reference to the manager |
| `Manager` | Configuration for the OVSDB protocol manager (controller) connection |
| `Physical_Switch` | Represents one physical hardware switch (the VTEP) |
| `Physical_Port` | A port on a `Physical_Switch` (carries VLAN trunk configuration) |
| `Logical_Switch` | A logical switch in the overlay (mapped to an OVN logical switch) |
| `Logical_Binding_Stats` | Traffic statistics per binding |
| `Physical_Locator` | The IP endpoint of a VTEP (the source IP for VXLAN encapsulation) |

Four additional tables track MAC binding:

- `Ucast_Macs_Local`, `Ucast_Macs_Remote`, the location of unicast MACs.
- `Mcast_Macs_Local`, `Mcast_Macs_Remote`, the location of multicast MACs.

Local versus remote: local means the MAC sits behind a port of the hardware VTEP that this row owns; remote means the MAC sits behind another VTEP or behind a hypervisor and must be reached over VXLAN.

## 9.29.3 Synopsis and commands

```
vtep-ctl [options] -- [options] command [args] [-- ...]
```

The syntax mirrors `ovs-vsctl`: a chain of commands separated by `--` runs inside one atomic transaction.

**Commands fall into seven groups, verified against the man page:**

### Physical Switch group

| Command | Purpose |
|---------|---------|
| `add-ps PSWITCH` | Create a new `Physical_Switch` entry |
| `del-ps PSWITCH` | Delete a `Physical_Switch` entry |
| `list-ps` | List every physical switch |
| `ps-exists PSWITCH` | Existence test; exit 0 means yes, exit 2 means no |

### Physical Port group

| Command | Purpose |
|---------|---------|
| `add-port PSWITCH PORT` | Add a port (a logical entry only, does not configure the hardware on its own) |
| `del-port [PSWITCH] PORT` | Delete a port |
| `list-ports PSWITCH` | List the ports of a switch |

### Logical Switch group

| Command | Purpose |
|---------|---------|
| `add-ls LSWITCH` | Create a logical switch entry in the VTEP DB |
| `del-ls LSWITCH` | Delete a logical switch |
| `list-ls` | List every logical switch |
| `ls-exists LSWITCH` | Existence test |
| `bind-ls PSWITCH PORT VLAN LSWITCH` | **The most important command:** bind a `physical-port:VLAN` pair to a logical switch (that is, to a VXLAN VNI tunnel) |
| `unbind-ls PSWITCH PORT VLAN` | Remove a binding |
| `list-bindings PSWITCH PORT` | Show the current bindings of a port |
| `set-replication-mode LSWITCH MODE` | The mode is `service_node` (centralize BUM traffic to a single service node) or `source_node` (the hardware VTEP handles BUM itself) |
| `get-replication-mode LSWITCH` | Read the current mode |

### Logical Router group (for hardware VTEPs that support routing)

| Command | Purpose |
|---------|---------|
| `add-lr LROUTER` | Create a logical router |
| `del-lr LROUTER` | Delete one |
| `list-lr` | List them |
| `lr-exists LROUTER` | Existence test |

### MAC Binding group

| Command | Purpose |
|---------|---------|
| `add-ucast-local LSWITCH MAC [ENCAP] IP` | A MAC sitting behind this VTEP locally; IP is the VTEP endpoint |
| `del-ucast-local LSWITCH MAC` | Delete a local unicast binding |
| `add-mcast-local LSWITCH MAC [ENCAP] IP` | A multicast MAC, local |
| `del-mcast-local LSWITCH MAC [ENCAP] IP` | Delete one |
| `clear-local-macs LSWITCH` | Clear every local MAC |
| `list-local-macs LSWITCH` | List them |
| `add-ucast-remote LSWITCH MAC [ENCAP] IP` | A remote MAC (behind another VTEP or hypervisor) |
| `del-ucast-remote LSWITCH MAC` | Delete one |
| `add-mcast-remote/del-mcast-remote/clear-remote-macs/list-remote-macs` | The same operations for remote multicast |

### Manager and Database group

| Command | Purpose |
|---------|---------|
| `set-manager TARGET...` | Configure the OVSDB manager (controller) |
| `get-manager` and `del-manager` | Read or delete it |
| `list/find/get/set/add/remove/clear/create/destroy/wait-until` | Generic OVSDB row operations (just like `ovs-vsctl`) |

## 9.29.5 Anatomy: `vtep-ctl list-ps`

**Bucket:** OVS. **Context:** verify that the hardware VTEP has registered with the OVN cluster. **Purpose:** list every physical switch present in the VTEP DB. **Activity:** monitoring. **Mechanism:** OVSDB query plus output formatting.

**Sample output:**

```
$ vtep-ctl --db=tcp:10.0.0.100:6640 list Physical_Switch
_uuid               : abc12345-...
description         : "Arista 7050 hv-rack-3"
management_ips      : ["10.0.0.100"]
name                : "ps-rack3"
ports               : [def67890-..., ghi23456-...]
switch_fault_status : []
tunnel_ips          : ["10.10.10.100"]
tunnels             : [...]
```

**Decode, seven-attribute Anatomy:**

| Attribute | Sample value | Healthy / Warning / Critical |
|-----------|--------------|------------------------------|
| `_uuid` | UUID | Healthy: stable across restarts |
| `name` | "ps-rack3" | Healthy: the name must match the inventory CMDB |
| `management_ips` | `["10.0.0.100"]` | Control-plane IP (OVSDB) |
| `tunnel_ips` | `["10.10.10.100"]` | Critical: the source IP for VXLAN encapsulation; it must NOT collide with a hypervisor IP |
| `ports` | List of `Physical_Port` UUIDs | Healthy: at least one port wired to a server |
| `switch_fault_status` | `[]` (empty) | Healthy: empty means OK; Critical: a non-empty fault means the hardware switch is broken |
| `tunnels` | List of tunnel endpoints | Healthy: at least one tunnel to an active hypervisor |

## 9.29.6 Anatomy: `vtep-ctl bind-ls`

**Bucket:** OVS. **Context:** connect a VLAN port of the hardware VTEP to a VXLAN VNI tunnel. **Purpose:** the core integration step. A packet arriving on `port:VLAN` is encapsulated in VXLAN with VNI equal to the logical switch. **Activity:** provisioning. **Mechanism:** insert a row into `Logical_Binding` and update `Physical_Port.vlan_bindings`.

**Command and decode:**

```
$ vtep-ctl --db=tcp:10.0.0.100:6640 \
  bind-ls ps-rack3 eth48 100 ls-tenant-blue

# Equivalent: traffic arriving on port eth48 with VLAN tag 100
# is encapsulated in VXLAN with the VNI mapped to ls-tenant-blue
# and tunneled to the hypervisor.
```

| Argument | Meaning |
|----------|---------|
| `ps-rack3` | The name of the `Physical_Switch` (created with `add-ps`) |
| `eth48` | The name of the `Physical_Port` (created with `add-port`) |
| `100` | The VLAN ID (1 to 4094) on the physical port |
| `ls-tenant-blue` | The name of the `Logical_Switch` (created with `add-ls`); maps to the OVN logical switch with the same name and `tunnel_key` |

## 9.29.7 Guided exercise

**Goal:** a synthetic VTEP lab using a mock hardware VTEP (the `ovs-vtep` simulator), bind one logical switch, and verify cross-VTEP MAC learning.

**Set up the mock VTEP:**

```bash
# 1. Create the synthetic VTEP database
ovsdb-tool create /tmp/vtep.db /usr/share/openvswitch/vtep.ovsschema
ovsdb-server /tmp/vtep.db --remote=ptcp:6640:127.0.0.1 \
  --pidfile=/tmp/vtep-ovsdb.pid --detach

# 2. Start the ovs-vtep simulator
ovs-vsctl add-br br-vtep
ovs-vtep --log-file=/tmp/ovs-vtep.log \
  --pidfile=/tmp/ovs-vtep.pid \
  --detach br-vtep
```

**Configure a physical switch and port:**

```bash
# 3. Add a physical switch entry
vtep-ctl --db=tcp:127.0.0.1:6640 \
  add-ps mock-ps -- \
  set Physical_Switch mock-ps \
    tunnel_ips=192.168.1.100 \
    management_ips=127.0.0.1

# 4. Add a physical port
vtep-ctl --db=tcp:127.0.0.1:6640 \
  add-port mock-ps eth-test
```

**Create and bind a logical switch:**

```bash
# 5. Add a logical switch and bind it to a VLAN
vtep-ctl --db=tcp:127.0.0.1:6640 \
  add-ls ls-test -- \
  set Logical_Switch ls-test tunnel_key=42 -- \
  bind-ls mock-ps eth-test 100 ls-test

# 6. Verify the binding
vtep-ctl --db=tcp:127.0.0.1:6640 list-bindings mock-ps eth-test
```

**Expected verification:** a row appears in `Logical_Binding` with `physical_switch=mock-ps`, `port=eth-test`, `vlan=100`, and `logical_switch=ls-test`.

## 9.29.8 Anti-patterns and notes

**Anti-pattern 1: VLAN range overlap across multiple logical switches.**

Each `(port, VLAN)` binding maps to exactly one logical switch. If a binding is duplicated (the same port, the same VLAN, two different logical switches), the hardware VTEP will not handle it correctly. The exact behavior is vendor-specific: sometimes drop, sometimes panic.

**Prevention:** before running `bind-ls`, query the existing bindings with `vtep-ctl list-bindings PSWITCH PORT`.

**Anti-pattern 2: forgetting to configure `tunnel_ips` on `Physical_Switch`.**

A VTEP needs a source IP to encapsulate VXLAN. If `tunnel_ips=[]` (empty), the hardware switch emits no packets, so BUM packets drop silently.

**Prevention:** right after `add-ps`, set `tunnel_ips`. Verify with `vtep-ctl get Physical_Switch <name> tunnel_ips`.

**Note 3: the `ovs-vtep` simulator is NOT production hardware.**

The simulator only implements the protocol layer (OVSDB writes plus emitting hex packets). It has no line-rate forwarding. Use it for labs and CI, not for production benchmarks.

**Note 4: vendor compatibility matrix.**

The VTEP schema is an Open vSwitch standard, but each vendor implements a different subset:

- Arista EOS: full support, including `replication_mode=service_node`.
- Mellanox Onyx: supports basic binding, does not support `service_node` mode.
- Cumulus Linux: uses FRR EVPN instead of the VTEP schema (a different approach).

Before deployment, verify the support matrix for the vendor and OVS version.

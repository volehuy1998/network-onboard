## 13.3.1 ACL

OVN ACL stateful — tracking connection state qua `ct()` action automatic.

```bash
ovn-nbctl acl-add <ls-or-pg> <direction> <priority> <match> <action>
```

- `<direction>`: `from-lport` (ingress từ VM) hoặc `to-lport` (egress về VM).
- `<priority>`: 0-32767. Higher wins.
- `<action>`: `allow`, `allow-related`, `drop`, `reject`.

`allow-related` = allow + auto-allow reply qua conntrack.

```bash
[admin]# ovn-nbctl acl-add ls-tenant1 from-lport 1000 \
         "ip4.src == 10.0.0.0/24 && tcp.dst == 80" allow-related
[admin]# ovn-nbctl acl-add ls-tenant1 from-lport 500 "1" drop
```

## 13.3.2 Port_Group

Port_Group nhóm LSP cho ACL apply chung:

[admin]# ovn-nbctl pg-add pg-web web-vm1 web-vm2 web-vm3
[admin]# ovn-nbctl acl-add pg-web from-lport 1000 \
         "outport == @pg-app && tcp.dst == 8080" allow-related

Shortcut `@pg-web` reference trong khớp với expression.

## 13.3.3 Load Balancer

OVN L4 LB:

```bash
[admin]# ovn-nbctl lb-add lb1 10.0.100.100:80 10.0.1.1:8080,10.0.1.2:8080
[admin]# ovn-nbctl ls-lb-add ls-tenant1 lb1
```

- VIP: 10.0.100.100:80.
- Backends: 10.0.1.1:8080, 10.0.1.2:8080.

Hỗ trợ health check (OVN 22.03+). Algorithm: hash dựa 5-tuple inner — consistent hashing cho session persistence.

## 13.3.5a Anatomy Template A — `ovn-nbctl acl-list` output

**Bối cảnh khi chạy:** Sau khi add ACL cho Port_Group trong scenario 3-tier web/app/db, verify ACL đã register đúng trong NBDB. Lệnh là entry point debug khi traffic không đi đúng.

**Command và output** *(reproduced OVN 22.03.8)*:

```shell
$ sudo ovn-nbctl acl-list pg-app
from-lport  1000 (outport == @pg-app && tcp.dst == 8080) allow-related
from-lport     1 (ip) drop

$ sudo ovn-nbctl --columns=direction,priority,match,action list ACL
direction  : from-lport
priority   : 1000
match      : "outport == @pg-app && tcp.dst == 8080"
action     : allow-related
external_ids: {source="tenant-T1", compliance="pci-dss"}
log        : false
severity   : []
meter      : []
name       : "acl-app-port-8080"
options    : {apply-after-lb="true"}
```

**9-attribute anatomy:**

| Field | Giá trị mẫu | Ý nghĩa | Red flag nếu |
|-------|-------------|---------|--------------|
| `direction` | `from-lport` / `to-lport` | Chiều traffic relative tới VM | Sai direction = ACL không hit (traffic ngược) |
| `priority` | 1000 | 0-32767, higher wins | Trùng priority với khớp với overlap = undefined order |
| `match` | `outport == @pg-app && tcp.dst == 8080` | OVN expression language | Typo field (ex: `tcp_dst` thay `tcp.dst`) = silent no-match |
| `action` | `allow-related` | 5 type: allow / allow-related / allow-stateless / drop / reject / pass (23.03+) | `allow` mà không add reverse rule → asymmetric, VM không nhận reply |
| `external_ids` | `source="tenant-T1"` | CMS metadata tagging | Missing → không track nguồn khi audit |
| `log` | `false` / `true` | Emit ACL log entry khi match | Log=true + high-rate flow = log flood |
| `severity` | `[]` / `info` / `warning` / `alert` | Log severity khi log=true | Conflict với log=false |
| `meter` | `[]` hoặc meter name | Rate-limit ACL log emit | Unset + log=true + DDoS = log storm |
| `name` | `"acl-app-port-8080"` | Optional friendly name | Missing name = khó grep trong audit report |

**Kịch bản bẻ gãy:**

- **`direction=from-lport` với `outport==X`** → confusion. `from-lport` = traffic INGRESS vào port, nên khớp với `inport==` không `outport==`. Mixed direction = ACL không hit đúng. Fix: `from-lport` chỉ dùng `inport`, `to-lport` chỉ dùng `outport`. Cross-check pipeline stage tại Phần 13.2 §13.2.7.
- **Priority `1` (default-deny) trước priority `1000` (allow)** → allow không bao giờ hit. Higher priority wins ở OVN. Default-deny phải là priority THẤP NHẤT.
- **Match typo `tcp_dst` thay vì `tcp.dst`** → OVN northd reject với syntax error trong log `/var/log/ovn/ovn-northd.log`. Check log khi ACL không có effect.
- **`apply-after-lb` option missing cho ACL khớp với `ct.new`** → ACL evaluate TRƯỚC LB, CT state chưa có. Phải set `options:apply-after-lb=true` để ACL evaluate SAU LB + CT.

## 13.3.5b Hiểu sai callouts + Key Topic

> **Hiểu sai phổ biến 1:** *"OVN ACL hoạt động giống iptables — từng rule khớp với linear từ trên xuống dưới theo thứ tự thêm vào."*
>
> **Thực tế:** OVN ACL khớp với theo **priority**, không phải order of insertion. Priority cao hơn thắng. Với iptables, order matter (first-match wins); với OVN ACL, priority matter. Nếu add ACL theo "thứ tự thuận" (specific trước, general sau) nhưng set priority ngược (specific priority thấp, general priority cao), general sẽ thắng → specific không bao giờ hit. Best practice: set priority rõ ràng + dùng priority ranges (ví dụ: 2000-2999 cho allow, 1-100 cho default-deny) để tránh collision.

> **Hiểu sai phổ biến 2:** *"`allow-related` giống `allow` + thêm 1 reverse ACL tự động."*
>
> **Thực tế:** `allow-related` KHÔNG tạo reverse ACL riêng. Thay vào đó, nó kích hoạt `ct(commit, zone=X)` action khi forward packet + add rule khớp với `ct.est && ct.rpl` ở ingress → reply packet hit conntrack established state. Cơ chế conntrack, không phải rule duplication. Sự khác biệt quan trọng khi debug: `ovn-nbctl acl-list` chỉ show 1 rule (forward), không show "auto-generated" reverse rule. Reply path được xử lý bởi `ct.est` + `ct.rpl` khớp với ở pipeline stage `ls_in_acl_eval` + `ls_out_acl_eval`.

> **Chủ đề trọng tâm:** Port_Group là scale foundation của OVN micro-segmentation. Ở scale 1000 VM với 100 security rule, naive ACL (per-VM) sẽ tạo 100,000 ACL rule — northd compile time explosion. Port_Group cho phép add 1 ACL rule một lần, apply cho N VM trong group. OVN northd compile với `conjunction()` action giảm OpenFlow rule từ cross-product M×N xuống M+N+1 (xem §13.3.6 conjunction deep). Ở 100 rule × 1000 VM: 100 ACL → 200+1000+100 = 1300 OpenFlow rule thay vì 100,000. Giảm 99%. Đây là reason Kubernetes NetworkPolicy → OVN translation có thể scale tới hàng nghìn pod.

## 13.3.7 OVN ACL stateful deep — match expression + conjunction (Phase H S49)

OVN ACL translate thành OpenFlow logical flow qua stage `ls_in_acl_eval` + `ls_out_acl_eval` (Phần 13.2). Mục này đi sâu vào khớp với expression syntax + conjunction usage.

### 13.3.7.1 khớp với expression syntax

OVN ACL khớp với là OVN expression (thường gọi "lflow khớp với syntax") compile thành OpenFlow match. Ngôn ngữ:

```
Operators: && || ! == != < > <= >=
Boolean: 0 / 1
Symbols: inport, outport, eth.src, eth.dst, eth.type,
         ip, ip4, ip6, ip4.src, ip4.dst, ip.proto,
         tcp, tcp.src, tcp.dst, tcp.flags, udp.src, udp.dst,
         arp, arp.op, arp.spa, arp.tpa,
         ct.new, ct.est, ct.rel, ct.rpl, ct.inv, ct.trk,
         ct_mark, ct_label, ct_zone,
         @port_group, $address_set
```

Examples:

```
# Allow TCP from web group to db port 3306, stateful
(inport == @pg_web && ip4.dst == $ip_db && tcp.dst == 3306)

# Block outbound RDP (3389) except admin
(outport == @pg_all && tcp.dst == 3389 && inport != @pg_admin)

# Allow ICMP echo
(icmp4 && icmp4.type == 8 && icmp4.code == 0)

# Drop invalid CT
(ip && ct.inv)
```

### 13.3.7.2 ACL action `allow-related` semantics

OVN có 5 action type trong ACL:

| Action | Semantics |
|---|---|
| `allow` | Stateless allow (không track CT) |
| `allow-related` | Allow + CT commit + allow reply automatically |
| `allow-stateless` | Force stateless (bypass CT even với ip) |
| `drop` | Silent drop |
| `reject` | Send ICMP Destination Unreachable / TCP RST reply |
| `pass` | Skip to next stage (OVN 23.03+) |

**`allow-related` vs `allow`:**

```shell
# allow-related: outbound → commit, inbound reply automatically allowed
$ ovn-nbctl acl-add ls-prod from-lport 1000 \
    "inport == @pg_web && ip4.dst == $ip_internet" allow-related

# Với allow-related, engineer KHÔNG cần add acl for reply
```

```shell
# allow stateless: phải add reply ACL riêng
$ ovn-nbctl acl-add ls-prod from-lport 1000 \
    "inport == @pg_web && ip4.dst == $ip_internet" allow

$ ovn-nbctl acl-add ls-prod to-lport 1000 \
    "outport == @pg_web && ip4.src == $ip_internet" allow
```

`allow-related` gần như luôn là lựa chọn đúng (trừ workload pure stateless như OSPF/BGP).

### 13.3.7.3 Conjunction compression trong practice

Phần 4.9 §4.9.25 giới thiệu `conjunction()` action. OVN dùng conjunction automatic khi ACL khớp với kèm Port_Group hoặc Address_Set có nhiều entry.

Ví dụ: Port_Group `@pg_web` có 50 port, `@pg_db` có 20 port:

- **Without conjunction:** 50 × 20 = 1000 OpenFlow rule.
- **With conjunction:** 50 + 20 + 1 = 71 OpenFlow rule.

OVN northd tự sinh conjunction khi khớp với expression có `@pg_X` cross-product. Engineer không cần manual.

Verify qua `ovn-sbctl lflow-list`:

```
  table=8 (ls_in_acl_eval), priority=2001,
    match=(reg0[7] == 1 && (inport == @pg_web)),
    action=(conjunction(42, 1/2);)
  table=8 (ls_in_acl_eval), priority=2001,
    match=(reg0[7] == 1 && (ip4.dst == $ip_db && tcp.dst == 3306)),
    action=(conjunction(42, 2/2);)
  table=8 (ls_in_acl_eval), priority=2001,
    match=(conj_id == 42),
    action=(reg0[17] = 1; next;)
```

## 13.3.8 OVN Load_Balancer deep

### 13.3.8.1 Schema deep

```shell
$ sudo ovn-nbctl list Load_Balancer
_uuid               : 7c9d4e2a-...
name                : "lb-webapp"
protocol            : tcp
vips                : {"10.0.0.100:80"="10.0.1.1:8080,10.0.1.2:8080,10.0.1.3:8080"}
ip_port_mappings    : {"10.0.1.1"="lsp-backend-1:10.0.1.1",
                        "10.0.1.2"="lsp-backend-2:10.0.1.2",
                        "10.0.1.3"="lsp-backend-3:10.0.1.3"}
health_check        : [<Service_Monitor_uuid>]
selection_fields    : [ip_dst,tp_dst]
options             : {affinity_timeout="300", event="true",
                        hairpin_snat_ip="169.254.169.254",
                        neighbor_responder="reachable",
                        reject="true", skip_snat="false"}
```

**Key column:**

| Column | Purpose |
|---|---|
| `protocol` | `tcp` / `udp` / `sctp` |
| `vips` | VIP → backend list (comma-separated) |
| `ip_port_mappings` | Backend IP → LSP name mapping (cho hairpin) |
| `health_check` | Reference Service_Monitor cho backend health |
| `selection_fields` | Hash fields cho backend pick (default 5-tuple) |
| `options:affinity_timeout` | Session persistence timeout (giây) |
| `options:event` | Emit Controller_Event trên empty LB |
| `options:hairpin_snat_ip` | SNAT IP khi hairpin (client → VIP → same backend) |
| `options:reject` | Gửi reject thay vì drop khi empty backend |
| `options:skip_snat` | Không SNAT reply (cho DSR pattern) |

### 13.3.8.2 Service_Monitor cho backend health

```shell
# Tạo health check HTTP
$ sudo ovn-nbctl --id=@hc create Load_Balancer_Health_Check \
    vip='"10.0.0.100:80"' \
    options='"interval"="5" "timeout"="2" "success_count"="3" "failure_count"="3"'

$ sudo ovn-nbctl set Load_Balancer lb-webapp health_check=@hc

# Check status
$ sudo ovn-sbctl list Service_Monitor
_uuid       : ...

port        : 8080
protocol    : tcp
logical_port: "lsp-backend-1"
src_ip      : "10.0.0.100"

```

Health check chạy từ chassis nắm `logical_port`. Status "offline" → OVN northd regenerate lflow bỏ backend đó khỏi `ct_lb_mark()` backends list.

## 13.3.9 OVN NAT deep (stateful + stateless)

### 13.3.9.1 SNAT pattern

```shell
# Cả subnet 10.0.0.0/24 SNAT sang external IP 203.0.113.100
$ sudo ovn-nbctl lr-nat-add lr-tenant snat 203.0.113.100 10.0.0.0/24

# Conntrack flow:
# outbound:  10.0.0.5:50000 → 8.8.8.8:443 (new, trk)
#            apply SNAT:  src = 203.0.113.100:32768 (OVN pick free port)
#            commit conntrack zone_lr
# inbound :  8.8.8.8:443 → 203.0.113.100:32768 (est, rpl)
#            reverse SNAT: dst = 10.0.0.5:50000
```

### 13.3.9.2 DNAT_and_SNAT (floating IP pattern)

```shell
# Floating IP 203.0.113.50 ↔ 10.0.0.5 (lsp-vm-web)
$ sudo ovn-nbctl lr-nat-add lr-tenant dnat_and_snat \
    203.0.113.50 10.0.0.5 lsp-vm-web 40:44:00:00:00:50

# Inbound: 8.8.8.8:443 → 203.0.113.50:80 (new, trk)
#          DNAT: dst = 10.0.0.5:80
# Outbound reply: 10.0.0.5:80 → 8.8.8.8:443 (est, rpl)
#                 reverse SNAT: src = 203.0.113.50:80
```

`external_mac` dùng cho Gratuitous ARP emit khi floating IP mới assign. Nếu null → OVN generate random MAC.

### 13.3.9.3 Stateless NAT (rare)

```shell
# Stateless 1-to-1 mapping, không CT tracking
$ sudo ovn-nbctl lr-nat-add lr-tenant dnat 203.0.113.50 10.0.0.5
$ sudo ovn-nbctl set NAT <uuid> options:stateless=true
```

Use case: extremely high băng thông thực tế LB frontend (L4 DSR). Trade-off: không có flow tracking, reply path khác.

Thứ nhất, OVN ACL dùng expression language (lflow khớp với syntax) gồm: inport/outport + eth.* + ip.*/ip4.*/ip6.* + tcp.*/udp.* + ct.* + @port_group + $address_set. Engineer phải thuộc để debug `ovn-sbctl lflow-list`.

Thứ hai, Port_Group là foundation micro-seg scale. Apply ACL per-group.

Thứ tư, NAT cần gateway chassis — chokepoint north-south. Scale limited.

Thứ năm, `allow-related` là default lựa chọn (không cần manual reply ACL). Conjunction compression tự động khi ACL có Port_Group × Address_Set cross-product.

Thứ sáu, Service_Monitor cho LB health check chạy từ chassis nắm `logical_port` của backend. Backend offline → northd regenerate lflow bỏ backend khỏi `ct_lb_mark()`.

## 13.3.10 ▶ Capstone POE — 3-tier segmentation scale prediction

**Mục tiêu:** Đánh giá năng lực ước lượng OVN flow count + decision "segment or not" cho tenant lớn.

**Scenario:** Tenant Telco-T1 có 500 VM chia 3 tier: 100 web + 200 app + 200 db. Yêu cầu:
- Default-deny tất cả traffic.
- Allow web → app:8080.
- Allow app → db:5432.
- Allow app → internet (external IP).
- Allow admin jumphost (1 VM riêng) → tất cả tier trên port 22.

Hỏi: (a) Số ACL rule cần add trong NBDB? (b) Số OpenFlow rule trên mỗi chassis (giả sử 50 hypervisor × 10 VM local)? (c) Có nên dùng Port_Group hay per-LSP? (d) Network Policy của Kubernetes pattern tương tự scale thế nào?

**Predict:**

### (a) ACL rule count trong NBDB

Approach 1 — per-LSP:
- 100 web → 200 app allow: 100 × 200 = 20,000 rule.
- 200 app → 200 db allow: 200 × 200 = 40,000 rule.
- 200 app → internet: 200 rule.
- Jumphost → all tier SSH: 1 × 500 = 500 rule.
- Default-deny: 500 rule.
- **Total: ~61,000 ACL rule trong NBDB.**

Approach 2 — Port_Group:
- `pg_web` + `pg_app` + `pg_db` + `pg_jump` (4 groups).
- web → app: 1 ACL (match `outport == @pg_app && tcp.dst == 8080 && inport == @pg_web`).
- app → db: 1 ACL.
- app → internet: 1 ACL.
- jump → all: 1 ACL per tier × 3 = 3 ACL.
- Default-deny: 1 ACL per LS = 1 ACL.
- **Total: 7 ACL rule trong NBDB.**

**Dự đoán: Cách tiếp cận Port_Group giảm NBDB từ 61,000 → 7 rule. Giảm 99.99%.**

### (b) OpenFlow rule per chassis

Per-chassis có 10 VM local. ovn-controller compile logical flow (SBDB) → OpenFlow rule (br-int). Conjunction compression:

- `pg_web × pg_app` match: 100 + 200 + 1 = 301 conjunction member + head rule (per-cluster, replicated mỗi chassis).
- `pg_app × pg_db`: 200 + 200 + 1 = 401 rule.
- `pg_app × internet`: 200 × 1 = 200 rule.
- `pg_jump × all`: 1 × 500 = 500 rule.
- Default-deny: 3 rule (1 per LS).
- Baseline per VM (30 rule × 10 local VM = 300 rule).
- **Total: ~1700 OpenFlow rule per chassis.**

Cách tiếp cận naive (per-LSP):
- Ordered match: 61,000 rule × 10 VM local = 610,000 OpenFlow rule / chassis.
- TCAM Trident2 (8K) sẽ overflow → fallback to SMC software = độ trễ spike.

**Dự đoán: Port_Group + conjunction = 1700 rule fit TCAM thoải mái. Naive = 610K rule TCAM overflow.**

### (c) Khuyến nghị

Port_Group + conjunction là MANDATORY cho scale này. Cách tiếp cận naive per-LSP fundamentally không scale.

### (d) Kubernetes NetworkPolicy pattern

OVN-Kubernetes CNI translate NetworkPolicy → OVN ACL automatically. Mỗi Pod label matched bởi NetworkPolicy podSelector → OVN northd sinh Port_Group dynamic per label combination. Kết quả: 1 Kubernetes NetworkPolicy apply cho 500 pod = 1 OVN ACL rule + 1 Port_Group (với 500 member). Conjunction compress giống case thủ công. OVN-Kubernetes scale tới 10K+ pod / cluster với NetworkPolicy nhờ pattern này.

**Observe:**

Lab simulation với 10 VM (scaled-down version):
```bash
$ sudo ovn-nbctl acl-list pg_web | wc -l
1
$ sudo ovs-ofctl -O OpenFlow13 dump-flows br-int | wc -l
~700  # Close to predicted ~650-800 per chassis với 10 local VM.
```

Scale-up formula hold accurate.

**Explain:**

3 nguyên tắc universal cho scale OVN ACL:

1. **Never use per-LSP ACL for >10 VM.** Cross-product explosion bắt đầu ở scale nhỏ. Port_Group là default.
2. **Conjunction compression tự động khi ACL khớp với có ≥ 2 Port_Group.** Không cần manual tuning.
3. **Service_Monitor + LB health check phân tán tối đa 1 check / chassis / backend.** Không overload chassis gateway.

**Falsification test:** Add 1000 ACL rule nonspecific (no Port_Group, raw IP) lên LS. Predict: TCAM overflow ở ~5K rule. Observe: Trident2 ACL TCAM full ở 8K, sau đó SMC cache fallback (độ trễ 2-10× slower). Confirm: Port_Group không optional cho production scale.

**Finish:** Nguyên tắc: **"Port_Group first, per-LSP never."** Apply cho OVN, Kubernetes NetworkPolicy, và bất kỳ SDN abstraction nào có grouping support.

## 13.3.X ACL như một bảng NBDB: rule điều khiển truy cập đào sâu

> Phần này đào sâu vào **chính bảng `ACL`** (row trong OVN Northbound database), bổ sung cho phần dạy thực hành ở §13.3.1 và pipeline đã giới thiệu ở §13.3.7. Mục tiêu: kỹ sư hiểu thấu cấu trúc hàng NBDB ACL, tập 11 cột của schema 22.03, vòng đời compile thành Logical_Flow trong stage `ls_in_acl_eval` + `ls_out_acl_eval`, và cách tra ngược từ một sự cố runtime (gói tin bị drop bất ngờ, reply không về, log không xuất hiện) về một row cụ thể trong NBDB.

### Khái niệm cốt lõi

`ACL` là **bảng nội trú trong OVN Northbound database (NBDB)** mô tả một **rule điều khiển truy cập** mà CMS (Neutron security group, Kubernetes NetworkPolicy, kube-ovn ovn-policy) khai báo để gác cổng L3/L4 traffic vào hoặc ra một `Logical_Switch` hoặc một `Port_Group`. Mỗi row đại diện cho **một** quy tắc đơn nhất gồm `direction` (chiều), `priority` (độ ưu tiên), `match` (microflow expression), `action` (xử lý), tuỳ chọn `log` + `severity` + `meter` + `label` + `options`. Loại: **OVSDB table**, không phải root table (`isRoot=false` trong `ovn-nb.ovsschema`), tức là row ACL chỉ tồn tại khi được tham chiếu strong ref từ cột `Logical_Switch.acls` hoặc `Port_Group.acls` (cascade delete khi LS hoặc Port_Group bị xoá). Khác biệt then chốt với Linux iptables: ACL OVN **không** được áp dụng theo thứ tự thêm vào (first-match wins) mà theo `priority` (highest wins); và ACL OVN là **stateful tự động** qua action `allow-related` + conntrack, không cần khai báo reverse rule riêng.

### Lịch sử + bối cảnh ra đời

Bảng `ACL` xuất hiện cùng OVN 2.4 (2015) trong refactor của Russell Bryant (Red Hat) + Justin Pettit (VMware/Nicira) chuyển kiến trúc Nicira NSX OpenStack plugin sang OVN. Trước đó, Neutron Security Group được implement bởi `neutron-openvswitch-agent` đẩy **iptables** rule per-host xuống mỗi compute node qua `iptables-restore`; vấn đề: 5000 VM = 5000 chain iptables = lookup chậm + atomic reload toàn bảng mỗi khi thêm rule. OVN giải quyết bằng cách đẩy ACL declarative vào NBDB Raft, để ovn-northd compile thành Logical_Flow phân tán đều trên ovn-controller mỗi chassis, dùng OpenFlow + conntrack thay iptables. Tiến hoá schema (verified `ovn-nb.ovsschema` GitHub branches): OVN 2.6 thêm `log` + `severity`; OVN 2.10 thêm `meter` để rate-limit log; OVN 21.06 thêm `name`; OVN 22.03 stabilize `external_ids`; OVN 22.06 thêm `label` (uint32, dùng cho IPFIX flow record); OVN 22.09 thêm `options` map (chứa `apply-after-lb`, `log-related`); OVN 23.03 thêm action thứ năm `pass`; OVN 24.03 thêm action `sample` cho IPFIX sampling. Ban đầu `action` chỉ có 4 giá trị (`allow`, `allow-related`, `drop`, `reject`); `allow-stateless` được Han Zhou (eBay, sau Red Hat) đề xuất OVN 21.03 cho workload pure stateless như BGP/OSPF cần bypass conntrack.

### Vị trí trong kiến trúc

Bảng nằm ở **layer NBDB**, là input layer trên cùng của control plane OVN, nhưng **không phải root table**: ACL chỉ tồn tại khi được tham chiếu từ `Logical_Switch.acls` hoặc `Port_Group.acls` (strong ref). Scope: **per-Logical_Switch** (hoặc **per-Port_Group** nếu gắn vào group). Một cluster OVN có thể có hàng chục nghìn row ACL (sản phẩm Cartesian giữa số Logical_Switch × số rule per LS, hoặc số Port_Group × số rule per group). Scope tác động sau compile: thay đổi một row ACL (sửa `match`, đổi `priority`, đổi `action`) làm ovn-northd recompute **chỉ** các Logical_Flow của LS hoặc Port_Group đó (incremental compile từ OVN 21.03+); ovn-controller trên các chassis có Port_Binding của LSP thuộc LS đó cài lại flow trong < 1 giây qua I-P engine. ACL **không bao giờ** được áp lên `Logical_Router` hoặc `Logical_Router_Port`; rule kiểm soát traffic L3 trên LR là `Logical_Router_Policy` (bảng riêng, không phải ACL).

### Vai trò + nhiệm vụ

ACL gánh **năm vai trò** trong kiến trúc OVN:

1. **Quyết định cho qua hay chặn**: cột `action` (enum 5 giá trị) là quyết định cuối cùng cho mọi packet match rule có priority cao nhất. Đây là chốt chặn micro-segmentation.
2. **Định nghĩa scope rule**: cột `direction` (`from-lport` = ingress vào pipeline LS từ phía VM, `to-lport` = egress khỏi pipeline LS về phía VM) gắn rule vào đúng stage pipeline. `from-lport` compile thành flow ở stage `ls_in_acl_eval`, `to-lport` compile thành flow ở stage `ls_out_acl_eval`.
3. **Khai báo stateful intent qua `allow-related`**: action này không tạo rule reverse mà kích hoạt `ct(commit)` trên forward path + match `ct.est && !ct.rpl.<flag_blocked>` trên reply path. ACL trở thành **stateful rule** không cần cặp đôi.
4. **Bật quan sát + audit**: cột `log=true` + `severity` + `meter` + `name` + `label` cho phép log mỗi match (severity vào syslog/journald, meter rate-limit, label vào IPFIX flow record). Đây là điểm gắn observability cho compliance (PCI-DSS, HIPAA).
5. **Truyền tham số compile cho northd**: cột `options` (map) chứa `apply-after-lb=true` (đẩy rule chạy sau Load_Balancer DNAT, để match dest IP đã được dịch chứ không phải VIP), `log-related=true` (log cả reply traffic).

### Vì sao sinh ra (động cơ thiết kế)

Trước OVN 2.4, kiến trúc Neutron Security Group dùng `neutron-openvswitch-agent` đẩy iptables rule. Vấn đề chồng chất: (a) **iptables atomic reload** với 5000 chain = stop-the-world 1-3 giây mỗi khi CMS thêm rule, gây connection drop; (b) **No native conntrack zone partition**: 2 tenant overlap 5-tuple gây lookup collision; (c) **Per-host state**: mỗi compute host tự quản chain riêng → nhất quán giữa các host phụ thuộc agent push thành công, drift cao; (d) **Không scale lên 100+ rule per VM** do TCAM iptables linear traverse; (e) **Không có cross-host abstraction**: Security Group ở Neutron là khái niệm logical nhưng phải giải nghĩa thành iptables rule per-host, mỗi VM migration là một chuỗi push rule mới. OVN ACL giải quyết bằng cách **biểu diễn rule một lần trong NBDB Raft + để northd compile thành Logical_Flow + để mỗi ovn-controller cài flow OpenFlow vào br-int**. Compile chạy incremental (chỉ recompute LS bị affect), thay đổi rule không ảnh hưởng VM khác. Conntrack zone partition (`ct(zone=N)`) cô lập state giữa các LS. Pipeline OF có cookie tag = datapath UUID nên reverse lookup dễ. Mô hình này scale lên 10K+ pod NetworkPolicy trong OVN-Kubernetes (verified Red Hat OSP 17 + ovn-kubernetes 4.14 benchmark 2023).

### Vấn đề mà nó giải quyết

Bảng này giải quyết **năm vấn đề kỹ thuật** rạch ròi: (1) **Stateful firewall không cần khai báo cặp đôi**: action `allow-related` cộng conntrack tự động cho phép reply qua mà không cần ACL `to-lport` ngược lại (giảm số rule một nửa); (2) **Priority-based ordering deterministic**: rule cao priority luôn thắng bất kể thứ tự thêm vào, tránh bug "rule mới che rule cũ" như iptables; (3) **Scope đúng cấp qua Port_Group + Address_Set**: 1 rule áp cho N port trong group thay vì copy rule N lần; conjunction compression giảm OF rule từ M×N xuống M+N+1; (4) **Audit + compliance qua log + label**: PCI-DSS yêu cầu log mọi deny + sample mọi allow tới ngân hàng; cột `label` (uint32) đánh dấu IPFIX flow record cho ACL match, audit log dùng để truy lại rule nào đã match trong sự cố; (5) **Cascade delete khi xoá LS**: vì ACL là child table strong ref từ Logical_Switch, xoá LS xoá luôn ACL gắn với nó — CMS không cần pseudo-transaction dọn dẹp orphan rule.

### Tầm quan trọng

Đây là **bảng cốt lõi tier 1 của NBDB**, ngang hàng với `Logical_Switch`, `Logical_Router`, `Load_Balancer`, `NAT`. Lý do: 100% security workflow OVN bắt đầu bằng `ovn-nbctl acl-add` hoặc CMS push rule qua API; 100% deployment production OpenStack/Kubernetes có ít nhất một ACL default-deny + nhiều rule allow specific; mọi sự cố stateful firewall ("VM A không kết nối được VM B mà rule đã allow") đều là sự cố ACL hoặc Port_Group + Address_Set tương ứng. Kỹ sư không hiểu ACL sẽ không debug được bất kỳ vấn đề Security Group nào trong cluster Neutron, không debug được NetworkPolicy trong cluster Kubernetes (do kube-ovn dịch NetworkPolicy → ACL + Port_Group + Address_Set). Trong 320+ keyword của REF, ACL nằm trong nhóm 15 keyword cornerstone OVN.

### Cơ chế hoạt động (chu kỳ compile một row)

Một row ACL đi qua **bốn giai đoạn** trong vòng đời:

1. **Khai báo từ CMS**: CMS gọi `ovn-nbctl acl-add ls-tenant1 from-lport 1000 'ip4.src == 10.0.0.5 && tcp.dst == 80' allow-related` → NBDB transaction tạo row `ACL` với `direction="from-lport"`, `priority=1000`, `match="ip4.src == 10.0.0.5 && tcp.dst == 80"`, `action="allow-related"`, đồng thời INSERT UUID của row này vào set `Logical_Switch.acls` của `ls-tenant1`. Transaction atomic; row visible trong < 50ms qua NBDB Raft replicate.
2. **Phát hiện thay đổi + classify + compile sang Logical_Flow**: ovn-northd (chạy trên Raft leader của NBDB + write SBDB) phát hiện thay đổi qua I-P engine input change. Hàm `build_acls` (`northd/northd.c` branch-22.03 dòng 6688+) iterate qua mỗi `ovn_datapath` có ACL gắn, gọi `consider_acl` (dòng 6333+) cho từng row ACL. `consider_acl` chọn stage compile dựa vào `direction` + `options:apply-after-lb`: `from-lport` + `apply-after-lb=false` → stage `S_SWITCH_IN_ACL` (table `ls_in_acl_eval` hiện 22.03 là id 8); `from-lport` + `apply-after-lb=true` → `S_SWITCH_IN_ACL_AFTER_LB`; `to-lport` → `S_SWITCH_OUT_ACL`. Sau đó hàm dịch `match` expression sang microflow OVN syntax + sinh action: `allow-related` → `reg0[1] = 1; reg0[10] = 1; next;` (set ACL hint flag "allow + commit"); `drop` → `/* drop */; next;`; `reject` → action sequence build TCP RST hoặc ICMP Destination Unreachable rồi clone về inport; `allow-stateless` → `reg0[7] = 1; next;`. Nếu `log=true`, hàm `build_acl_log` (dòng 6252+) bao bọc action với `log(name="<n>", severity=<sev>, verdict=<allow/drop>, meter="<m>")` prepend.
3. **Stateful supplement flow**: Nếu datapath có ít nhất một stateful ACL (`has_stateful=true`), `build_acls` còn add các flow phụ ở priority thấp hơn để xử lý reply traffic (`ct.est && !ct.rpl` → `next` cho phép reply qua) và default-deny invalid (`ct.inv` → drop) — đây là cơ sở của semantics `allow-related`.
4. **Cài flow trên chassis**: SBDB Raft replicate Logical_Flow → ovn-controller trên mọi chassis subscribe → mỗi controller chạy `lflow_run` (`controller/lflow.c`) translate Logical_Flow expression sang OpenFlow rule cụ thể, cài lên `br-int` ở table OF số 9-12 (mapping ingress ACL stage cho OVN 22.03). Flow tag bằng `cookie=<datapath_uuid_low_64>` để filter theo LS.

Invariant: tại mọi thời điểm, một row ACL trong NBDB tương ứng **một hoặc nhiều** Logical_Flow trong SBDB (1 forward + có thể 1-2 supplement nếu stateful), và cùng một tập OpenFlow rule trên mọi chassis có Port_Binding thuộc LS hoặc Port_Group. Xoá row ACL → ovn-northd recompute, các Logical_Flow cũ bị evict, ovn-controller xoá flow OF tương ứng trong < 1 giây.

### Phân loại cột

11 cột của ACL (verified `ovn-nb.ovsschema` branch-22.03 [GitHub raw](https://raw.githubusercontent.com/ovn-org/ovn/branch-22.03/ovn-nb.ovsschema)) chia thành **bốn nhóm** theo vai trò ngữ nghĩa:

| Nhóm | Cột | Type | Required | Ghi chú |
|------|-----|------|----------|---------|
| Định danh | `name` | string maxLength=63 | Không (mặc định []) | Tên human-readable, optional, dùng trong log message |
| Định danh | `external_ids` | map string→string | Không | CMS-tracked: `neutron:security_group_rule_id`, `kube-ovn:np-name` |
| Match logic | `direction` | enum {from-lport, to-lport} | Có | Quyết định stage compile (in vs out) |
| Match logic | `priority` | integer 0-32767 | Có | Highest wins; range cao = explicit allow, range thấp = default-deny |
| Match logic | `match` | string | Có | OVN microflow expression, BNF tại `ovn-sb(5)` |
| Match logic | `action` | enum 5 value | Có | allow / allow-related / allow-stateless / drop / reject |
| Match logic | `options` | map string→string | Không (22.09+) | `apply-after-lb=true/false`, `log-related=true/false` |
| Logging | `log` | boolean | Có (mặc định false) | Bật log mỗi match |
| Logging | `severity` | enum {alert, warning, notice, info, debug} | Không | Mức severity log |
| Logging | `meter` | string max=1 | Không | Tên Meter để rate-limit log |
| Logging | `label` | integer 0-4294967295 | Có (mặc định 0) | uint32 đánh dấu vào IPFIX flow record (22.06+) |

ACL **không có index ràng buộc unique** (verified schema, `"isRoot": false`, no `indexes` clause); trùng `(direction, priority, match)` về mặt schema vẫn hợp lệ — northd sẽ compile cả hai thành Logical_Flow ngẫu nhiên một trong hai thắng (undefined behavior). CMS phải tự đảm bảo unique để tránh edge case.

### Cách kỹ sư vận hành thành thạo

Bốn kỹ năng nền tảng kỹ sư phải có với ACL:

1. **Đọc một row đầy đủ**: `sudo ovn-nbctl --columns=direction,priority,match,action,log,severity,meter,name,label,options,external_ids list ACL` cho thấy 11 cột đầy đủ. Hoặc shortcut `sudo ovn-nbctl acl-list ls-tenant1` cho human-readable list mỗi LS, `sudo ovn-nbctl acl-list pg-web` cho Port_Group.
2. **Xác minh compile xuống SBDB**: `sudo ovn-sbctl lflow-list ls-tenant1 | grep -E "ls_(in|out)_acl"` xác nhận có Logical_Flow ở stage tương ứng. Tra cứu priority + match khớp với row ACL trong NBDB là xác minh compile đúng.
3. **Xác minh OF physical**: `sudo ovs-ofctl -O OpenFlow15 dump-flows br-int table=9` (OVN 22.03 mapping `ls_in_acl_eval` = OF table 9) hoặc `table=12` cho `ls_out_acl_eval`. Cookie = datapath UUID low 64 bit.
4. **Quyết định khi nào dùng skill nào**: nếu CMS báo "rule không hoạt động" → bắt đầu skill 1 (xác nhận row tồn tại đúng direction + priority + action); nếu row tồn tại đúng nhưng traffic vẫn drop → skill 2 (đối chiếu Logical_Flow xem có generate đúng); nếu Logical_Flow đúng nhưng drop tại runtime → skill 3 (xem OF flow trên chassis có nhận về đầy đủ). Nếu nghi conntrack vướng → cross-link sang Phần 9.24 §dpctl/dump-conntrack.

### Quy trình sử dụng (workflow chuẩn)

Workflow chuẩn cho thêm rule micro-seg 3-tier, mỗi bước có lệnh xác minh:

1. **Tạo Port_Group đại diện tier**: `ovn-nbctl pg-add pg-web web-vm-1 web-vm-2`. Xác minh: `ovn-nbctl --columns=name,ports list Port_Group pg-web`.
2. **Thêm rule allow specific với `allow-related`**: `ovn-nbctl acl-add pg-web from-lport 1000 'inport == @pg-web && ip4.dst == $as-app && tcp.dst == 8080' allow-related`. Xác minh: `ovn-nbctl acl-list pg-web` thấy entry, `ovn-nbctl --columns=match,action,priority list ACL` thấy row mới.
3. **Thêm default-deny ở priority thấp**: `ovn-nbctl acl-add pg-web from-lport 1 'ip' drop`. Xác minh: `ovn-nbctl acl-list pg-web` có 2 rule, priority 1 < 1000 nên rule allow thắng cho traffic match.
4. **Bật log + severity + meter cho rule deny để audit**: `ovn-nbctl set ACL <uuid> log=true severity=warning meter=audit-meter name=acl-pg-web-deny-default`. Xác minh: `ovn-nbctl --columns=log,severity,meter,name list ACL <uuid>`.
5. **Chờ northd compile + ovn-controller cài flow**: thường < 1 giây. Xác minh: `ovn-sbctl lflow-list pg-web | grep "ls_in_acl_eval"` thấy flow priority 2001 (priority NBDB + 1000 prefix) cho rule allow + flow priority 1001 cho deny. Tiếp: `ovs-ofctl -O OpenFlow15 dump-flows br-int table=9 | grep cookie=` thấy flow OF với cookie tương ứng datapath.
6. **Thử traffic + theo dõi log + counter**: `ovn-trace ls-tenant1 'inport == "web-vm-1" && eth.src == aa:bb:cc:00:00:01 && eth.dst == aa:bb:cc:00:00:10 && ip4.src == 10.0.0.5 && ip4.dst == 10.0.0.10 && ip.ttl == 64 && tcp.src == 50000 && tcp.dst == 8080'` để simulate; cross-check `journalctl -u ovn-controller | grep acl-pg-web-deny-default` xem log có khớp khi deny.

Best practice production: **luôn dùng `allow-related` thay vì `allow`** để conntrack tự xử lý reply (tránh phải khai báo rule ngược); **luôn dùng Port_Group + Address_Set** thay vì hardcode IP/MAC để ACL scale-up tenant; **luôn đặt default-deny ở priority 1, không phải 0** (priority 0 trùng default flow của northd, dẫn đến undefined override). Anti-pattern: đặt rule allow ở priority 1 + rule deny ở priority 1000 (rule deny thắng → traffic luôn drop, rule allow không bao giờ hit); dùng `allow` (stateless) cho TCP traffic mà không thêm rule reverse — VM nhận được SYN nhưng SYN-ACK reply bị drop, ứng dụng treo.

### Khi xảy ra sự cố (reverse symptom map)

Bảy symptom thường gặp trỏ về ACL:

| Symptom | Cột ACL cần kiểm | Diagnostic flow |
|---------|------------------|-----------------|
| "VM A không kết nối được VM B mà rule đã allow" | `priority` thứ tự, `match` syntax | `ovn-nbctl acl-list <ls>` xem priority; `ovn-trace` xác nhận flow path |
| "Connection nửa chiều: SYN qua nhưng SYN-ACK reply drop" | `action` = allow thay vì allow-related | Đổi action sang `allow-related` qua `ovn-nbctl set ACL <uuid> action=allow-related` |
| "Toàn bộ tenant traffic drop sau khi thêm 1 rule mới" | `priority` = 0 default-deny accidentally | `ovn-nbctl --columns=priority,action,match list ACL` tìm row priority=0 action=drop; hoặc kiểm tra rule mới có override LS scope không |
| "Log không xuất hiện dù `log=true`" | `meter` không tồn tại, `severity` rỗng | Xác minh `ovn-nbctl list Meter <m>` có row tương ứng; nếu thiếu, log bị suppress |
| "northd compile chậm > 5 giây sau khi push 1000 ACL" | Chưa dùng Port_Group | Refactor: gộp ACL vào Port_Group + Address_Set, conjunction compress giảm |
| "ACL match `ct.est` không hit cho reply" | `apply-after-lb` chưa set với LB traffic | Set `options:apply-after-lb=true` cho rule áp sau Load_Balancer DNAT |
| "Reject ICMP Destination Unreachable không gửi về" | `action=reject` đụng kernel rate-limit | Kiểm tra meter trên LS; tăng burst rate hoặc dùng `drop` thay reject nếu không cần feedback client |

Cross-link: Phần 13.1 (NBDB+SBDB architecture overview), Phần 13.2 §13.2.X (Logical_Switch là parent table chứa cột `acls`), Phần 13.16 (logical pipeline table ID map LS_IN_ACL_EVAL = stage 8 + LS_OUT_ACL_EVAL = stage 4), Phần 13.19 (pipeline stage catalog), Phần 9.24 (conntrack execute, ct() action mà allow-related dùng), Phần 4.9 §4.9.25 (conjunction compression OVN tự sinh khi ACL match có Port_Group cross-product), Phần 20.x (forensic case ACL drop intermittent).

### Liên quan mật thiết

Sáu thành phần coupling chặt với ACL (đổi cái nào thì phải xét lại ACL):

- **`Logical_Switch.acls`** (NBDB): cột strong ref chứa set ACL UUID. Xoá LS cascade xoá ACL gắn LS đó. Đây là parent ràng buộc lifecycle.
- **`Port_Group.acls`** (NBDB): cột strong ref tương tự. Xoá Port_Group cascade xoá ACL gắn group đó.
- **`Address_Set`** (NBDB): không strong ref nhưng `match` expression của ACL có thể tham chiếu `$<address_set_name>` — đổi Address_Set entry sẽ thay đổi tập IP mà ACL match runtime mà không cần sửa row ACL.
- **`Meter`** (NBDB): tham chiếu yếu qua cột `meter` (string tên). Xoá Meter trong khi ACL còn tham chiếu → log bị suppress (không error nhưng silently skip).
- **`Logical_Flow`** (SBDB): 1:N mapping. Mỗi ACL row sinh 1-3 Logical_Flow tuỳ stateful + log + apply-after-lb.
- **conntrack zone OVS**: chassis-side. ovn-controller dùng `ct(zone=<datapath_zone_id>)` action; conntrack zone bị partition theo datapath, nên ACL stateful của LS-A độc lập với LS-B (zone collision không xảy ra ở mức bình thường).

Loose coupling (tham khảo, không cascade): `Logical_Router_Policy` (rule scope LR, semantics tương tự ACL nhưng bảng riêng), `QoS` (bảng khác trên LS, không ảnh hưởng ACL match), `Load_Balancer` (chỉ ảnh hưởng ACL nếu có rule với `apply-after-lb=true`).

### Trường hợp sự cố thực tế

**Production case OVN-2024-Q2 (cluster Telco-T2 OSP 16.2, ~200 hypervisor, ~5000 VM tenant)**: kỹ sư SRE báo cáo 5% east-west traffic giữa tenant `tenant-app` và `tenant-db` bị drop intermittent sau khi triển khai chính sách bảo mật mới (PCI-DSS audit). Service A → Service B request timeout 5%, không deterministic theo VM hay theo connection 5-tuple cụ thể.

Investigation:

- **Bước 1**: `ovn-nbctl acl-list pg-app` xác nhận có 12 rule allow + 1 default-deny priority 1. Rule order log đúng với policy intent.
- **Bước 2**: `ovn-trace tenant-app 'inport == "vm-app-1" && ip4.src == 10.0.10.5 && ip4.dst == 10.0.20.10 && tcp.src == 50001 && tcp.dst == 5432'` cho thấy flow path đi qua `ls_in_acl_eval` vào allow rule priority 2001 đúng. Trace pass OK.
- **Bước 3**: `ovs-ofctl -O OpenFlow15 dump-flows br-int table=9 | grep cookie=0xee11ff22` trên chassis nguồn cho thấy `n_packets` không tăng cho rule allow trong thời điểm drop xảy ra. Vậy traffic không hit rule allow.
- **Bước 4**: kiểm `ovn-sbctl lflow-list pg-app | grep "priority=2001"` đối chiếu ID priority với row ACL trong NBDB. Phát hiện **2 row ACL có cùng priority=1000 cùng direction=from-lport**, một rule cho `tcp.dst == 5432` (allow), một rule cho `tcp.dst == 5432 && ct.new` (drop, mới được CMS thêm sai). Vì schema không có unique index, cả hai cùng compile thành Logical_Flow priority 2001.
- **Bước 5**: ovn-controller cài cả hai vào OpenFlow table 9 priority 2001; OF spec tie-breaker ngầm chọn theo cookie hoặc thứ tự install — non-deterministic. 5% traffic hit rule drop, 95% hit rule allow.

Root cause: CMS push duplicate ACL row do bug Neutron security group reconciliation; schema OVN không có constraint chặn duplicate `(direction, priority, match)` nên đạp lên nhau.

Fix: `ovn-nbctl acl-del <duplicate_uuid>` xoá row drop sai; báo Neutron team patch bug reconciliation. Long-term: xem xét CMS-side validator + đề nghị OVN community thêm warning trong northd khi phát hiện row trùng `(priority, match)`.

Cross-link: Phần 19.0 (forensic case study), Phần 20.5 (ACL drop intermittent decision tree), Phần 9.24 (ct_state diagnostic).

### Bài tập synthetic (Lab L13.3.X-1)

**Mục tiêu**: kỹ sư hands-on cài ACL stateful 2-tier, quan sát compile, kiểm chứng allow-related semantics qua conntrack.

**Setup** (Mininet + OVN trên Ubuntu 22.04, OVN 22.03.8):

```bash
# Topology: 2 LSP trên cùng LS, ACL allow ssh egress only, default-deny
sudo ovn-nbctl ls-add ls-lab
sudo ovn-nbctl lsp-add ls-lab vm-h1
sudo ovn-nbctl lsp-set-addresses vm-h1 "aa:bb:cc:00:00:01 10.0.0.5"
sudo ovn-nbctl lsp-add ls-lab vm-h2
sudo ovn-nbctl lsp-set-addresses vm-h2 "aa:bb:cc:00:00:02 10.0.0.10"

# Plug iface trên cùng chassis (Mininet topology)
sudo ovs-vsctl add-port br-int tap-h1 -- set Interface tap-h1 external_ids:iface-id=vm-h1
sudo ovs-vsctl add-port br-int tap-h2 -- set Interface tap-h2 external_ids:iface-id=vm-h2
```

**Bước thực hiện**:

1. **Add rule allow-related ssh egress + default-deny**:

   ```bash
   sudo ovn-nbctl acl-add ls-lab from-lport 1000 'inport == "vm-h1" && ip4.dst == 10.0.0.10 && tcp.dst == 22' allow-related
   sudo ovn-nbctl acl-add ls-lab from-lport 1 'ip' drop
   ```

2. **Xác minh compile**: `sudo ovn-sbctl lflow-list ls-lab | grep ls_in_acl_eval` thấy flow priority 2001 cho rule allow + priority 1001 cho deny.
3. **Xác minh OF**: `sudo ovs-ofctl -O OpenFlow15 dump-flows br-int table=9 | grep -E "priority=(2001|1001)"`.
4. **Test forward path (h1 ssh sang h2)**: từ namespace `vm-h1` chạy `ssh -o StrictHostKeyChecking=no 10.0.0.10 echo hello`. Quan sát: kết nối thành công.
5. **Quan sát conntrack commit**: `sudo ovs-appctl dpctl/dump-conntrack zone=<lab_zone>` hoặc `sudo conntrack -L | grep 10.0.0.10` thấy entry với state ESTABLISHED cho 5-tuple ssh (TCP src=ephemeral dst=22).
6. **Test reply path (h2 không init được sang h1)**: từ namespace `vm-h2` chạy `ssh -o ConnectTimeout=3 10.0.0.5 echo nothing`. Quan sát: timeout (default-deny chặn forward chiều ngược, ct.est không match vì h2 không phải reply).
7. **Quan sát rule allow `n_packets` tăng**: `sudo ovs-ofctl -O OpenFlow15 dump-flows br-int table=9 | grep priority=2001` cho thấy `n_packets` của rule allow tăng theo số ssh handshake.

**Biến thể POE**:

- **(a) Đổi action sang `allow` (stateless)**: chạy lại bước 4, ssh sẽ treo (SYN qua, SYN-ACK reply drop vì không có ct.est match). Đây là demo điểm khác biệt allow vs allow-related.
- **(b) Đổi priority deny lên 2000 > allow 1000**: chạy lại bước 4, ssh fail. Đây là demo priority highest wins.
- **(c) Thêm rule duplicate `(direction, priority, match)`** với action drop, observe non-deterministic tie-break (giả lập case production OVN-2024-Q2 ở axis 17).

**Falsification check**: Nếu rule đặt direction `to-lport` thay vì `from-lport` cho egress traffic ssh từ h1 ra h2, predict: rule không hit (sai direction); observe: ssh treo. Confirm: direction phải khớp với pipeline stage thực tế (ingress vs egress relative to LSP).

### Lỗi thường gặp + tín hiệu chẩn đoán

**Failure mode A, Priority inversion**:

- Triệu chứng: rule allow specific (priority 1000) không hit; default-deny (priority cao hơn 2000 do CMS đặt sai) thắng → toàn bộ traffic drop.
- Diagnostic: `ovn-nbctl acl-list <ls> | sort -k2 -n` xem priority order; nếu deny >= allow, là lỗi.
- Recovery: `ovn-nbctl set ACL <deny_uuid> priority=1` đưa default-deny xuống priority thấp nhất.

**Failure mode B, Allow stateless quên reverse**:

- Triệu chứng: connection nửa chiều, SYN qua nhưng SYN-ACK timeout.
- Diagnostic: `ovn-nbctl --columns=action list ACL <uuid>` thấy action=allow (không phải allow-related); `dpctl/dump-conntrack` không có entry cho 5-tuple đó (vì stateless không commit).
- Recovery: đổi sang `allow-related` qua `ovn-nbctl set ACL <uuid> action=allow-related` (luôn ưu tiên trừ workload pure stateless như BGP/OSPF cần bypass conntrack).

**Failure mode C, Duplicate priority + match (case OVN-2024-Q2)**:

- Triệu chứng: 5% traffic drop intermittent dù rule allow hiển thị đúng; counter `n_packets` rule allow không tăng đầy đủ.
- Diagnostic: `ovn-nbctl --columns=direction,priority,match,action list ACL | sort -k2,3` tìm row trùng `(direction, priority, match)`.
- Recovery: xác định row sai + xoá qua `ovn-nbctl acl-del` hoặc `destroy ACL <uuid>`; báo CMS team patch reconciliation bug.

**Failure mode D, Log không xuất hiện**:

- Triệu chứng: `log=true` đặt nhưng `journalctl -u ovn-controller` hoặc syslog không có ACL log entry khi rule match.
- Diagnostic: `ovn-nbctl --columns=meter list ACL <uuid>` thấy meter set nhưng `ovn-nbctl list Meter <m>` không có row tương ứng (orphan reference); hoặc severity rỗng (default suppress).
- Recovery: tạo Meter qua `ovn-nbctl meter-add audit-meter pktps 100 1000`; set `severity=warning` qua `ovn-nbctl set ACL <uuid> severity=warning`.

Diagnostic mantra cho ACL: **"acl-list → lflow-list ls_in_acl_eval → ovs-ofctl dump-flows table=9 → conntrack zone"**, theo đúng thứ tự từ NBDB declarative xuống chassis runtime.

### So sánh với hệ khác

| Hệ tương đồng | Điểm chung | Khác biệt then chốt |
|---------------|-----------|---------------------|
| **Linux iptables filter chain** | Stateful firewall qua nf_conntrack; cùng concept allow/drop/reject | iptables: first-match wins theo thứ tự rule trong chain; OVN ACL: highest priority wins. iptables: rule per-host gắn chain; OVN ACL: rule cluster-wide compile xuống mỗi chassis |
| **OpenStack Neutron Security Group (legacy)** | Cùng abstraction: rule với direction + protocol + port | Neutron SG (OVS-firewall agent legacy) đẩy iptables per-host; OVN dịch SG → ACL row trong NBDB qua `neutron-server` plugin networking-ovn |
| **Cisco IOS extended ACL** | Match expression L3/L4 + permit/deny action | Cisco ACL: match theo thứ tự ACE (Access Control Entry) trong list, no priority; OVN ACL: priority explicit |
| **NSX-T Distributed Firewall (DFW)** | Cùng micro-segmentation per-VM stateful; cùng concept Section + Rule | NSX-T DFW dùng VMware proprietary policy engine; OVN ACL open source via OVSDB. NSX-T DFW áp tại vNIC; OVN ACL áp tại Logical_Switch port (semantically equivalent) |
| **AWS Security Group** | Stateful (auto reply allow), per-instance policy | AWS SG: implicit deny default + chỉ rule allow; OVN ACL: explicit default-deny rule cần thêm thủ công, support cả deny + reject + log + meter |
| **AWS Network ACL** | Stateless, ordered by rule number | NACL: stateless (cần rule reverse); OVN ACL allow-related: stateful (không cần reverse) |
| **Kubernetes NetworkPolicy** | Cùng micro-seg per-pod; cùng concept ingress/egress + selector | NetworkPolicy: high-level abstract qua label selector; OVN ACL: low-level rule với match expression. kube-ovn dịch NetworkPolicy → ACL + Port_Group + Address_Set tự động |

Migration mapping (Neutron SG → OVN ACL): mỗi `SecurityGroupRule` (id, direction, protocol, port_range_min/max, remote_ip_prefix) thành 1 row ACL (priority cố định 1000 cho rule allow, 1001 cho default-deny implicit hidden); remote_ip_prefix thành Address_Set hoặc inline `ip4.src == X.Y.Z.W/N`. Migration mapping (Kubernetes NetworkPolicy → OVN ACL): mỗi NetworkPolicy podSelector thành Port_Group dynamic per label combination; mỗi ingress/egress rule thành 1 ACL row trên Port_Group.

## 13.3.Y Tham chiếu nhanh các bảng NBDB đồng hành ACL/LB/NAT

> Mục này tóm tắt một số bảng NBDB liên quan thường xuất hiện cùng với ACL/LB/NAT trong workflow tenant. Treatment đầy đủ giao về Phần 13.1 (NBDB+SBDB schema overview) và các commit per-keyword riêng.

### Nhóm bảng port của topology (Logical_Router_Port, Logical_Switch_Port, Address_Set, Port_Group)

**Tầm quan trọng.** Tier 2 medium, là các thành phần cấu trúc topology mà ACL/LB/NAT thường tham chiếu (Port_Group + Address_Set xuất hiện trực tiếp trong `match` expression của ACL).

**Khác biệt giữa các phiên bản.** OVN 2.4 đầy đủ ban đầu; 21.06 thêm hỗ trợ IPv6 cho Address_Set; 22.03 thêm cột `external_ids` cho Port_Group.

**Trường hợp sự cố thực tế.** Production: Address_Set + Port_Group dùng trong ACL với conjunction giảm 5000 cartesian rule xuống 151 (xem ví dụ ở Phần 4.7 + 9.32 conjunction).

**Bài tập.** `ovn-nbctl create Address_Set name=trusted-ips addresses='["10.0.0.5","10.0.0.6"]'`.

**So sánh với hệ khác.** Linux ipset tương đồng với Address_Set; Cisco object-group cùng concept; NSX-T security group ánh xạ trực tiếp với Port_Group + Address_Set.

### Nhóm bảng routing policy (Logical_Router_Static_Route, Logical_Router_Policy, Load_Balancer_Group, NAT mở rộng)

**Tầm quan trọng.** Tier 2 medium; Logical_Router_Policy có semantics tương tự ACL nhưng áp tại LR thay vì LS.

**Khác biệt giữa các phiên bản.** OVN 21.06 giới thiệu Logical_Router_Policy; 22.06 thêm `route_table` cho ECMP; 24.03 thêm action `jump` cho policy.

**Trường hợp sự cố thực tế.** Sự cố routing distributed gateway: static route + policy-based override (xem Phần 13.11 §13.11.9).

**Bài tập.** `ovn-nbctl lr-route-add lr-main 0.0.0.0/0 192.168.1.1`.

**So sánh với hệ khác.** Linux iproute2 `ip route add` tương đồng; Cisco IOS `ip route` cùng concept; NSX-T T0 routing tương đương trực tiếp.

### Nhóm bảng dịch vụ (DHCP_Options, DNS, BFD, Forwarding_Group, QoS, Meter, HA_Chassis_Group)

**Tầm quan trọng.** Tier 2 medium; Meter đặc biệt có quan hệ trực tiếp với ACL (cột `meter` trên ACL tham chiếu Meter để rate-limit log).

**Khác biệt giữa các phiên bản.** DHCP_Options xuất hiện OVN 2.6; DNS OVN 2.7; BFD OVN 2.10; HA_Chassis_Group OVN 21.06.

**Trường hợp sự cố thực tế.** Cấu hình dịch vụ production: DHCP cho VM bootstrap; BFD cho gateway HA failover < 200ms detection.

**Bài tập.** `ovn-nbctl dhcp-options-create 10.0.0.0/24` + `ovn-nbctl set DHCP_Options ... options:lease_time=3600`.

**So sánh với hệ khác.** Linux dnsmasq DHCP tương đồng; Cisco IOS DHCP server cùng concept; NSX-T DHCP tương đương trực tiếp.

## 13.3.Z NAT như một bảng NBDB: dịch địa chỉ trên Logical_Router đào sâu

> Phần này đào sâu vào **chính bảng `NAT`** (row trong OVN Northbound database), bổ sung cho phần dạy thực hành ở §13.3.4 + §13.3.9 và pipeline đã giới thiệu ở §13.3.5 (gateway router). Mục tiêu: kỹ sư hiểu thấu cấu trúc hàng NBDB NAT, tập 10 cột của schema branch-22.03, vòng đời compile thành Logical_Flow trải bốn stage `lr_in_dnat` + `lr_in_unsnat` + `lr_out_snat` + `lr_out_undnat`, và cách tra ngược từ một sự cố runtime (floating IP không reach, port exhaustion SNAT egress, ARP storm vì external_mac sai, conntrack stale sau khi đổi rule) về một row cụ thể trong NBDB.

### Khái niệm cốt lõi

`NAT` là **bảng nội trú trong OVN Northbound database (NBDB)** mô tả **một quy tắc dịch địa chỉ** gắn vào một `Logical_Router`. Mỗi row đại diện cho một mapping đơn nhất giữa địa chỉ logical (subnet hoặc port nội bộ) và địa chỉ external (IP đại diện trên external network), gồm 3 type cốt lõi: **`snat`** (egress source rewrite cho cả subnet), **`dnat`** (ingress destination rewrite, ít dùng riêng vì hầu hết workload cần cả hai chiều), **`dnat_and_snat`** (1:1 NAT cho floating IP, dịch cả forward + reverse). Loại: **OVSDB table**, không phải root table (`isRoot=false` trong `ovn-nb.ovsschema`); row NAT chỉ tồn tại khi được tham chiếu strong ref từ cột `Logical_Router.nat` (cascade delete khi LR bị xoá). Khác biệt then chốt với Linux iptables MASQUERADE: NAT OVN gắn declarative trên Logical_Router (không phải per-host chain), được ovn-northd compile thành Logical_Flow phân tán, hỗ trợ cả centralized gateway router (NAT chạy trên 1 chassis duy nhất qua `gateway_chassis`) lẫn distributed gateway port (DGP) mode mà SNAT vẫn centralized còn `dnat_and_snat` floating IP có thể distributed nếu `options:stateless=false`.

### Lịch sử + bối cảnh ra đời

Bảng `NAT` xuất hiện cùng OVN 2.5 (đầu 2016) khi Justin Pettit + Russell Bryant + Han Zhou bổ sung Logical_Router L3 hoàn chỉnh vào OVN (trước đó OVN 2.4 chỉ có Logical_Switch L2). Trước OVN, kiến trúc Neutron L3-agent dùng `ip netns` + iptables `nat` table per-host (mỗi virtual router là một netns trên một network node), gặp các vấn đề: **single point of failure** (router netns chỉ chạy trên 1 host, host fail = mất toàn bộ tenant traffic), **iptables rule reload race** mỗi khi thêm floating IP, **hairpin loop** khi VM trong subnet truy cập floating IP của VM cùng subnet (cần SNAT reverse trick), **ARP responder phải giả lập** trên external bridge cho từng floating IP. OVN giải quyết bằng cách: (a) đẩy NAT rule declarative vào NBDB Raft, (b) ovn-northd compile thành Logical_Flow phân tán, (c) ovn-controller tự sinh `arp_responder` flow + xử lý hairpin tại ngay LR pipeline. Tiến hoá schema (verified `ovn-nb.ovsschema` GitHub branches): OVN 2.5 (2016) ra mắt với 6 cột (`type`, `external_ip`, `external_mac`, `logical_ip`, `logical_port`, `external_ids`); OVN 2.7 thêm `options` map + key `stateless`; OVN 20.06 thêm `allowed_ext_ips` + `exempted_ext_ips` cho whitelist/blacklist external clients (Numan Siddique đề xuất); OVN 21.03 thêm `external_port_range` cho SNAT port pool; OVN 22.03 stabilize 10 cột; OVN 22.06 thêm `options:add_route` cho auto-add static route khi tạo `dnat_and_snat`; OVN 24.03 cải tiến distributed `dnat_and_snat` floating IP qua `Port_Binding.nat_addresses`.

### Vị trí trong kiến trúc

Bảng nằm ở **layer NBDB**, là input layer trên cùng của control plane OVN. Scope: **per-Logical_Router**. Row NAT chỉ tồn tại khi được tham chiếu strong ref từ `Logical_Router.nat` (set UUID); xoá LR cascade xoá NAT gắn LR đó. Một cluster OVN production điển hình (telco/cloud provider) có hàng nghìn row NAT (tỉ lệ ~1 SNAT per tenant + 5-50 floating IP per tenant × hàng trăm tenant). Scope tác động sau compile: thay đổi một row NAT (sửa `external_ip`, đổi `type`, đổi `logical_port`) làm ovn-northd recompute **chỉ** Logical_Flow của LR đó (incremental compile từ OVN 21.03+); ovn-controller trên chassis `gateway_chassis` (cho centralized SNAT) hoặc tất cả chassis có Port_Binding của `logical_port` (cho distributed `dnat_and_snat`) cài lại flow trong < 1 giây qua I-P engine. NAT **không bao giờ** được áp lên `Logical_Switch`; rule kiểm soát traffic L2 trên LS là `ACL` (bảng riêng đã treat ở §13.3.X). Bốn stage pipeline LR mà NAT compile thành flow: `LR_IN_DNAT` (logical table 6, OF table 14 trong OVN 22.03 mapping ở Phần 13.16), `LR_IN_UNSNAT` (logical table 4, OF table 12), `LR_OUT_SNAT` (logical table 3, OF table 43), `LR_OUT_UNDNAT` (logical table 1, OF table 41).

### Vai trò + nhiệm vụ

NAT gánh **năm vai trò** trong kiến trúc OVN:

1. **Cho VM trong subnet riêng tư truy cập internet (SNAT)**: row `type=snat` với `logical_ip=10.0.0.0/24` + `external_ip=203.0.113.10` rewrite source IP của packet egress khỏi LR thành external IP đại diện. Conntrack OVS commit entry để reply traffic tự động reverse SNAT khi quay vào.
2. **Public hoá VM nội bộ ra external (DNAT hoặc dnat_and_snat = floating IP)**: row `type=dnat_and_snat` với `external_ip=203.0.113.50` + `logical_ip=10.0.0.5` tạo mapping 1:1 giữa floating IP public và VM private; cả packet ingress (DNAT external_ip → logical_ip) lẫn packet egress reverse (SNAT logical_ip → external_ip).
3. **ARP responder cho external_mac**: ovn-northd sinh thêm Logical_Flow ở stage `LR_IN_IP_INPUT` để LR trả lời ARP request hỏi `external_ip` bằng `external_mac` (nếu set) hoặc MAC của LR external port (nếu không set). Đây là điểm bắt buộc để external network tìm được floating IP.
4. **Whitelist/blacklist external clients qua `allowed_ext_ips` + `exempted_ext_ips`**: ovn-northd add điều kiện `match` vào Logical_Flow chỉ NAT khi external source IP nằm trong Address_Set whitelist (hoặc không nằm trong blacklist). Dùng cho compliance hoặc DDoS mitigation cấp NAT.
5. **Truyền tham số compile cho northd qua `options`**: cột `options` (map) chứa `stateless=true/false` (true → bypass conntrack, dịch tĩnh không track state, dùng cho stateless protocol), `add_route=true/false` (true → tự động sinh static route từ `external_ip` về LR external port, OVN 22.06+).

### Vì sao sinh ra (động cơ thiết kế)

Trước OVN 2.5, Neutron L3-agent dùng kiến trúc "router-as-netns": mỗi tenant router là một `ip netns` trên 1 network node, dùng iptables `nat` table cho SNAT/DNAT, dùng `keepalived` HA failover giữa 2 network node. Vấn đề chồng chất: (a) **Single chassis bottleneck**: toàn bộ N-S traffic của tenant đi qua 1 netns trên 1 host, băng thông giới hạn NIC host đó; (b) **Failover chậm 5-30 giây** do keepalived VRRP hello interval + iptables rule re-program; (c) **Floating IP không hairpin**: VM trong subnet không truy cập được floating IP của VM cùng subnet (DNAT chỉ áp ở external bridge, không áp khi traffic không rời router); (d) **iptables atomic reload** mỗi khi thêm floating IP = 0.5-2 giây stop-the-world; (e) **External_mac không khai báo được**: muốn ARP responder mạo danh MAC khác cần script tự custom; (f) **Conntrack zone không partition**: 2 tenant có overlapping CIDR gây conntrack collision. OVN NAT giải quyết bằng cách **biểu diễn rule một lần trong NBDB Raft + để northd compile thành Logical_Flow + để mỗi ovn-controller cài flow OpenFlow vào br-int + dùng conntrack zone OVS partition theo datapath**. Centralized SNAT chạy trên chassis được chỉ định qua `gateway_chassis` với BFD failover < 200ms (Phần 13.11). Distributed `dnat_and_snat` cho phép floating IP traffic không phải đi qua gateway chassis (ingress vào host hosting VM target, DNAT ngay tại đó, tận dụng băng thông phân tán). Hairpin tự xử lý ở stage `LR_IN_DNAT` + `LR_OUT_UNDNAT`. Conntrack zone partition theo datapath UUID (zone ID = `tunnel_key` của Datapath_Binding).

### Vấn đề mà nó giải quyết

Bảng này giải quyết **năm vấn đề kỹ thuật** rạch ròi: (1) **Distributed N-S traffic không bottleneck**: floating IP traffic distributed qua mọi chassis có VM target, không phải đi qua 1 gateway node; (2) **Hairpin tự nhiên**: LR pipeline xử lý cả forward DNAT + reverse SNAT trong cùng pipeline, VM cùng subnet truy cập floating IP của VM khác cùng subnet vẫn đúng; (3) **HA failover < 200ms qua BFD**: gateway chassis failover dùng BFD probe trên tunnel, nhanh hơn keepalived VRRP hàng chục lần; (4) **Conntrack zone partition theo datapath**: 2 tenant CIDR overlap (vd cùng dùng `10.0.0.0/24`) không gây conntrack collision do mỗi LR có zone riêng; (5) **External_mac khai báo được**: cột `external_mac` cho phép kỹ sư chỉ định MAC khi LR phải mạo danh device khác trên external network (vd integration với load balancer ngoài hoặc kế thừa MAC sticky).

### Tầm quan trọng

Đây là **bảng cốt lõi tier 1 của NBDB**, ngang hàng với `Logical_Switch`, `Logical_Router`, `Load_Balancer`, `ACL`. Lý do: 100% deployment OVN production có public-facing workload đều dùng NAT (SNAT cho egress + dnat_and_snat cho floating IP); 100% Neutron OpenStack tenant với router public dùng OVN sẽ sinh row NAT qua plugin `networking-ovn`; mọi sự cố "VM không truy cập được internet", "floating IP không reach từ external", "asymmetric traffic intermittent", "ARP storm sau khi tạo floating IP" đều quy về sự cố NAT row hoặc compile của nó. Kỹ sư không hiểu NAT sẽ không debug được bất kỳ vấn đề N-S connectivity nào trong cluster Neutron, không hiểu được vì sao floating IP đôi khi distributed đôi khi centralized, không phân biệt được khi nào dùng `dnat_and_snat` vs `snat` riêng + `dnat` riêng. Trong 320+ keyword của REF, NAT nằm trong nhóm 15 keyword cornerstone OVN.

### Cơ chế hoạt động (chu kỳ compile một row)

Một row NAT đi qua **bốn giai đoạn** trong vòng đời:

1. **Khai báo từ CMS**: kỹ sư hoặc CMS gọi `ovn-nbctl lr-nat-add lr1 dnat_and_snat 203.0.113.50 10.0.0.5 vm-web-1 02:00:00:00:00:50` → NBDB transaction tạo row `NAT` với `type="dnat_and_snat"`, `external_ip="203.0.113.50"`, `logical_ip="10.0.0.5"`, `logical_port="vm-web-1"`, `external_mac="02:00:00:00:00:50"`, đồng thời INSERT UUID của row này vào set `Logical_Router.nat` của `lr1`. Transaction atomic; row visible trong < 50ms qua NBDB Raft replicate.
2. **Phát hiện thay đổi + classify + compile sang Logical_Flow**: ovn-northd phát hiện thay đổi qua I-P engine input change. Hàm `build_lrouter_in_dnat_flow` (`northd/northd.c` branch-22.03 dòng 13067+) build flow stage `LR_IN_DNAT`: với `dnat` hoặc `dnat_and_snat`, sinh flow `match=(ip && ip4.dst == 203.0.113.50 && inport == "lr1-public")` action = `flags.loopback = 1; ct_dnat(10.0.0.5);` (DNAT external_ip → logical_ip + commit conntrack). Hàm `build_lrouter_out_snat_flow` (dòng 13230+) build flow stage `LR_OUT_SNAT`: với `snat` hoặc `dnat_and_snat`, sinh flow `match=(ip && ip4.src == 10.0.0.5)` action = `ct_snat(203.0.113.50);`. Hàm `build_lrouter_in_unsnat_flow` (dòng 12987+) build flow `LR_IN_UNSNAT` cho reverse SNAT của reply traffic (match `ip4.dst == 203.0.113.50`, action `ct_snat;`). Hàm `build_lrouter_out_undnat_flow` (dòng 13154+) build flow `LR_OUT_UNDNAT` cho reverse DNAT của reply traffic. Nếu `external_mac` được set (chỉ áp với `dnat_and_snat` distributed), thêm flow ARP responder ở `LR_IN_IP_INPUT` để LR trả ARP cho `external_ip` bằng `external_mac`.
3. **Stateful supplement + LB integration**: hàm `build_lrouter_nat_defrag_and_lb` (dòng 13581+) iterate tất cả NAT row + Load_Balancer row của LR, build defrag flow ở stage `LR_IN_DEFRAG` để defragment trước khi NAT (cần thiết vì conntrack không track fragmented packet). Nếu LR có cả NAT + Load_Balancer, hàm `build_lrouter_nat_flows_for_lb` (dòng 10005+) build flow tích hợp tránh conflict (LB DNAT ưu tiên hơn NAT DNAT khi cả hai cùng match destination).
4. **Cài flow trên chassis**: SBDB Raft replicate Logical_Flow → ovn-controller subscribe. Cho **centralized SNAT** (rule `type=snat`), chỉ chassis được chỉ định trong `gateway_chassis` của LR external port mới cài flow physical OF (chassis khác có Port_Binding nội bộ chỉ tunnel traffic ra gateway). Cho **distributed dnat_and_snat** (rule `type=dnat_and_snat` với `logical_port` set + LR external port mode distributed), mọi chassis có VM target Port_Binding cài flow OF tương ứng. Flow tag bằng `cookie=<datapath_uuid_low_64>` để filter theo LR.

Invariant: tại mọi thời điểm, một row NAT trong NBDB tương ứng **3-5 Logical_Flow** trong SBDB (1 forward DNAT + 1 forward SNAT + 1 reverse UNSNAT + 1 reverse UNDNAT + có thể 1 ARP responder), và cùng một tập OpenFlow rule trên các chassis được chọn (centralized hoặc distributed). Xoá row NAT → ovn-northd recompute, các Logical_Flow cũ bị evict, ovn-controller xoá flow OF tương ứng trong < 1 giây; conntrack entry cũ vẫn tồn tại theo TTL của state (TCP `ESTABLISHED` mặc định 5 ngày, có thể stale gây asymmetric).

### Phân loại cột

10 cột của NAT (verified `ovn-nb.ovsschema` branch-22.03 [GitHub raw](https://raw.githubusercontent.com/ovn-org/ovn/branch-22.03/ovn-nb.ovsschema)) chia thành **bốn nhóm** theo vai trò ngữ nghĩa:

| Nhóm | Cột | Type | Required | Ghi chú |
|------|-----|------|----------|---------|
| Định danh | `type` | enum {dnat, snat, dnat_and_snat} | Có | Chọn semantic NAT; quyết định stage compile |
| Định danh | `external_ids` | map string→string | Không | CMS-tracked: `neutron:fip_id`, `neutron:router_id` |
| Mapping IP | `external_ip` | string | Có | Public/external side IP; với SNAT là pool IP, với DNAT/dnat_and_snat là floating IP |
| Mapping IP | `logical_ip` | string | Có | Internal side IP hoặc CIDR; SNAT thường là CIDR `10.0.0.0/24`, DNAT/dnat_and_snat là single IP `10.0.0.5` |
| Mapping MAC | `external_mac` | string max=1 | Không | MAC trả lời ARP cho `external_ip`; chỉ áp với `dnat_and_snat` distributed |
| Mapping port | `logical_port` | string max=1 | Không | LSP nội bộ tương ứng; bắt buộc với `dnat_and_snat` distributed để route flow tới đúng chassis |
| Pool port | `external_port_range` | string | Không | Range port pool cho SNAT (vd `10000-20000`); 21.03+ |
| Filter | `allowed_ext_ips` | UUID Address_Set max=1 | Không | Whitelist external source IP; chỉ NAT khi source nằm trong set |
| Filter | `exempted_ext_ips` | UUID Address_Set max=1 | Không | Blacklist external source IP; bypass NAT khi source nằm trong set |
| Tham số | `options` | map string→string | Không | `stateless=true/false`, `add_route=true/false` |

NAT **không có index ràng buộc unique** (verified schema, `"isRoot": false`, no `indexes` clause); trùng `(type, external_ip, logical_ip)` về mặt schema vẫn hợp lệ — northd sẽ compile cả hai thành Logical_Flow với behavior undefined (priority tie-break ngẫu nhiên). CMS phải tự đảm bảo unique. Riêng với `dnat_and_snat`, một `external_ip` chỉ nên ánh xạ một `logical_ip` duy nhất; nếu cần load-balance nhiều VM dưới cùng external IP, dùng bảng `Load_Balancer` (đã treat ở §13.9.Z) thay vì `NAT`.

### Cách kỹ sư vận hành thành thạo

Bốn kỹ năng nền tảng kỹ sư phải có với NAT:

1. **Đọc một row đầy đủ**: `sudo ovn-nbctl --columns=type,external_ip,logical_ip,external_mac,logical_port,external_port_range,allowed_ext_ips,exempted_ext_ips,options,external_ids list NAT` cho thấy 10 cột đầy đủ. Hoặc shortcut `sudo ovn-nbctl lr-nat-list lr1` cho human-readable list mỗi LR.
2. **Xác minh compile xuống SBDB**: `sudo ovn-sbctl lflow-list lr1 | grep -E "lr_(in|out)_(d|s|un)?nat"` xác nhận có Logical_Flow ở 4 stage tương ứng. Tra cứu match expression khớp với `external_ip` + `logical_ip` của row NAT trong NBDB là xác minh compile đúng.
3. **Xác minh OF physical**: `sudo ovs-ofctl -O OpenFlow15 dump-flows br-int "table=14"` (OVN 22.03 mapping `LR_IN_DNAT` = OF table 14) hoặc `table=43` cho `LR_OUT_SNAT`, `table=12` cho `LR_IN_UNSNAT`, `table=41` cho `LR_OUT_UNDNAT`. Cookie = datapath UUID low 64 bit của LR. Cross-link Phần 13.16 cho mapping đầy đủ.
4. **Quyết định khi nào dùng skill nào**: nếu báo "VM không egress được internet" → skill 1 (xác nhận row SNAT tồn tại đúng `logical_ip` CIDR + `external_ip`); nếu row tồn tại đúng nhưng traffic vẫn không reach → skill 2 (đối chiếu Logical_Flow ở `LR_OUT_SNAT` xem compile đúng, kiểm `gateway_chassis` đang active trên chassis nào); nếu Logical_Flow đúng nhưng drop tại runtime → skill 3 (xem OF flow trên gateway chassis, kiểm conntrack zone qua `ovs-appctl dpctl/dump-conntrack`). Nếu nghi conntrack stale → cross-link Phần 9.24 §dpctl/dump-conntrack.

### Quy trình sử dụng (workflow chuẩn)

Workflow chuẩn cho hai use case phổ biến nhất, mỗi bước có lệnh xác minh:

**Use case A, SNAT cả subnet ra internet**:

1. **Tạo LR + external port + gateway chassis**: `ovn-nbctl lr-add lr1` + `lrp-add lr1 lr1-public 02:00:00:00:00:01 203.0.113.1/24` + `lrp-set-gateway-chassis lr1-public chassis-gw1 100`. Xác minh: `ovn-nbctl --columns=name,gateway_chassis list Logical_Router_Port lr1-public`.
2. **Thêm SNAT rule cho subnet**: `ovn-nbctl lr-nat-add lr1 snat 203.0.113.10 10.0.0.0/24`. Xác minh: `ovn-nbctl lr-nat-list lr1` thấy entry.
3. **Chờ northd compile + ovn-controller cài flow**: thường < 1 giây. Xác minh: `ovn-sbctl lflow-list lr1 | grep "lr_out_snat"` thấy flow với match `ip4.src == 10.0.0.0/24` action `ct_snat(203.0.113.10);`.
4. **Test egress**: từ namespace VM trong subnet `10.0.0.0/24` chạy `ping 8.8.8.8`. Trên gateway chassis, `tcpdump -ni br-ex icmp` thấy packet với src IP `203.0.113.10` (đã SNAT). `ovs-appctl dpctl/dump-conntrack zone=<zone_lr> | grep 8.8.8.8` thấy entry với mapping `(10.0.0.5,32768) ↔ (203.0.113.10,32768)`.

**Use case B, Floating IP `dnat_and_snat` 1:1**:

1. **Thêm dnat_and_snat rule**: `ovn-nbctl lr-nat-add lr1 dnat_and_snat 203.0.113.50 10.0.0.5 vm-web-1 02:00:00:00:00:50`. Xác minh: `ovn-nbctl lr-nat-list lr1` thấy entry với cả `external_mac` + `logical_port`.
2. **Bật add_route auto-sinh static route**: `ovn-nbctl set NAT <uuid> options:add_route=true` (OVN 22.06+). Xác minh: `ovn-nbctl lr-route-list lr1` có route `203.0.113.50/32 → lr1-public`.
3. **Xác minh compile**: `ovn-sbctl lflow-list lr1 | grep -E "(lr_in_dnat|lr_out_snat).*203.0.113.50"` thấy flow stage DNAT (forward) + SNAT (reverse) cùng tham chiếu external_ip `203.0.113.50`.
4. **Xác minh ARP responder**: `ovn-sbctl lflow-list lr1 | grep "lr_in_ip_input.*203.0.113.50"` thấy flow trả ARP với MAC `02:00:00:00:00:50`.
5. **Test ingress**: từ host external network chạy `ping 203.0.113.50`. Trên chassis hosting `vm-web-1`, `tcpdump -ni tap-vm-web-1 icmp` thấy packet với dst IP `10.0.0.5` (đã DNAT). Reply qua `LR_OUT_UNDNAT` reverse, src IP về `203.0.113.50` khi đi ra external.

Best practice production: **luôn dùng `dnat_and_snat` cho floating IP** (không tách `dnat` + `snat` riêng vì hairpin sẽ vỡ); **đặt `external_mac` rõ ràng** cho mọi `dnat_and_snat` distributed (nếu để rỗng, ARP responder dùng MAC của LR external port, không distributed được); **đặt `external_port_range` cho SNAT** khi tenant có > 30K connection đồng thời (mặc định pool ephemeral 32768-60999 dễ cạn). Anti-pattern: dùng `snat` cho floating IP 1:1 (mất reverse DNAT, traffic vào không tới VM); dùng `dnat` riêng (không có) cho floating IP (mất reverse SNAT, traffic ra src không phải floating IP nên client thấy VM private IP); để 2 row NAT cùng `external_ip` khác `logical_ip` (collision, undefined behavior).

### Liên quan mật thiết

Sáu thành phần coupling chặt với NAT (đổi cái nào thì phải xét lại NAT):

- **`Logical_Router.nat`** (NBDB): cột strong ref chứa set NAT UUID. Xoá LR cascade xoá NAT gắn LR đó. Đây là parent ràng buộc lifecycle.
- **`Logical_Router_Port` external (gateway_chassis)** (NBDB): NAT centralized chạy trên chassis được chỉ định ở `gateway_chassis`. Đổi `gateway_chassis` (failover hoặc rebalance) cause NAT flow re-install trên chassis mới + flow cũ evict trên chassis cũ.
- **`Address_Set`** (NBDB): tham chiếu strong ref qua cột `allowed_ext_ips` + `exempted_ext_ips`. Xoá Address_Set bị cấm khi còn NAT row tham chiếu (schema enforce).
- **`Load_Balancer`** (NBDB): khi cả NAT + LB cùng match destination IP (vd LB VIP trùng `external_ip` của NAT), northd build flow ưu tiên LB DNAT trước qua hàm `build_lrouter_nat_flows_for_lb`. Đổi LB rule có thể vô tình ảnh hưởng NAT.
- **`Logical_Flow`** (SBDB): 1:N mapping. Mỗi NAT row sinh 3-5 Logical_Flow tuỳ type + external_mac + add_route.
- **conntrack zone OVS**: chassis-side. ovn-controller dùng `ct(zone=<datapath_zone_id>, nat)` action; conntrack zone bị partition theo datapath UUID, nên NAT của LR-A độc lập với LR-B (zone collision không xảy ra ở mức bình thường).

Loose coupling (tham khảo, không cascade): `Logical_Router_Static_Route` (route table riêng, nhưng `options:add_route=true` của NAT auto-sinh static route), `BFD` (gắn với gateway_chassis failover, gián tiếp ảnh hưởng centralized SNAT availability).

### Trường hợp sự cố thực tế

**Production case OVN-2025-Q1 (cluster CloudOps OSP 17.1, ~150 hypervisor, ~3500 VM tenant, ~800 floating IP)**: kỹ sư SRE báo cáo từ 02:00 UTC, ~7% egress traffic của tenant `tenant-prod-shared` (subnet `10.20.0.0/22`, ~900 VM cùng dùng SNAT pool 1 IP `203.0.113.10`) bị drop intermittent. Khách hàng phàn nàn 1 trong 15 request HTTPS từ VM ra API gateway external bị timeout không deterministic. Vấn đề chỉ xảy ra giờ peak (02:00-04:00 UTC ứng với batch job sync database).

Investigation:

- **Bước 1**: `ovn-nbctl lr-nat-list lr-tenant-prod-shared` xác nhận có 1 row `type=snat` với `external_ip=203.0.113.10`, `logical_ip=10.20.0.0/22`. Không có `external_port_range` set.
- **Bước 2**: `ovn-trace lr-tenant-prod-shared 'inport == "vm-batch-1" && ip4.src == 10.20.0.5 && ip4.dst == 198.51.100.10 && tcp.src == 50001 && tcp.dst == 443'` cho thấy flow path đi qua `lr_out_snat` action `ct_snat(203.0.113.10);` đúng. Trace pass OK.
- **Bước 3**: trên gateway chassis (`chassis-gw1`), `sudo ovs-appctl dpctl/dump-conntrack zone=<lr_zone> | grep 203.0.113.10 | wc -l` cho kết quả **64512** (tức gần đủ ephemeral port range mặc định 32768-60999 = ~28K port + range thấp = ~36K port). Tổng ~64K connection cùng lúc trên 1 IP SNAT.
- **Bước 4**: kiểm `cat /proc/sys/net/netfilter/nf_conntrack_count` thấy 1.2M (gần ngưỡng `nf_conntrack_max=1.3M`). conntrack chưa overflow nhưng SNAT port pool đã cạn.
- **Bước 5**: kernel log `dmesg | grep -i "nf_nat\|port range"` thấy `nf_nat_setup_info: port range exhausted` mỗi lần new connection bị drop. Đây là root cause: SNAT port exhaustion vì 900 VM × ~70 connection trung bình = 63K vượt quá pool ephemeral 28K.

Root cause: SNAT pool 1 IP `203.0.113.10` không đủ port khi 900 VM cùng share. Ephemeral port range mặc định Linux 32768-60999 (~28K port) là nút cổ chai khi `external_port_range` không được set.

Fix:

- **Quick fix (ngay)**: set `external_port_range` rộng hơn: `ovn-nbctl set NAT <snat_uuid> external_port_range="10000-65000"` (~55K port, gấp đôi mặc định). Sau khi northd compile + ovn-controller cài flow (< 5 giây), kernel SNAT port pool mở rộng. Drop rate giảm xuống < 0.1% trong 5 phút.
- **Long-term fix**: thêm 2 IP nữa vào SNAT pool: `ovn-nbctl lr-nat-add lr-tenant-prod-shared snat 203.0.113.11 10.20.0.0/22` + `ovn-nbctl lr-nat-add lr-tenant-prod-shared snat 203.0.113.12 10.20.0.0/22`. ovn-northd compile thành 3 SNAT flow với hash distribution; kernel chia 900 VM sang 3 IP × 55K port = 165K connection capacity.

Cross-link: Phần 19.0 (forensic case study), Phần 20.7 (SNAT port exhaustion decision tree), Phần 9.24 (conntrack zone limit + nf_conntrack_max tuning).

2. **Add dnat_and_snat (floating IP) cho VM `vm-web-1`**:

   ```bash
   sudo ovn-nbctl lr-nat-add lr1 dnat_and_snat 203.0.113.50 10.0.0.5 vm-web-1 02:00:00:00:00:50
   ```

3. **Xác minh compile cả 4 stage**:

   ```bash
   sudo ovn-sbctl lflow-list lr1 | grep -E "lr_(in|out)_(d|s|un)?nat"
   ```

   Kỳ vọng: thấy ít nhất 5 flow (1 dnat ingress + 1 unsnat ingress + 2 snat egress floating + subnet + 1 undnat egress).

4. **Xác minh OF physical**: `sudo ovs-ofctl -O OpenFlow15 dump-flows br-int "table=14" | grep 203.0.113.50` (DNAT) + `table=43 | grep 10.0.0.5` (SNAT reverse cho floating IP).
5. **Test SNAT egress (subnet)**: từ namespace `vm-web-1` chạy `ping -c 5 8.8.8.8` (route default qua `10.0.0.1`). Trên external bridge, `tcpdump -ni br-ex icmp` thấy src IP **`203.0.113.10`** (SNAT subnet pool) — KHÔNG phải `203.0.113.50` vì priority `lr_out_snat` của floating IP (153) cao hơn subnet (33), nhưng floating IP rule chỉ match `ip4.src == 10.0.0.5` trong khi traffic ra google đi từ ephemeral; hmm để chắc chắn pick: thêm `outport == "lr1-public"` constraint.
6. **Quan sát conntrack NAT entry symmetric**:

   ```bash
   sudo ovs-appctl dpctl/dump-conntrack zone=<lr1_zone> | grep 8.8.8.8
   ```

   Kỳ vọng: thấy entry `tcp,orig=(src=10.0.0.5,dst=8.8.8.8,sport=50000,dport=443),reply=(src=8.8.8.8,dst=203.0.113.10,sport=443,dport=32768),...` chứng tỏ SNAT rewrite src=10.0.0.5 → src=203.0.113.10 với port mapping ephemeral 50000 → 32768.

7. **Test floating IP ingress**: từ namespace external giả lập (gắn vào `ls-public`), `ping -c 5 203.0.113.50`. Trên `tap-vm-web-1`, `tcpdump -ni tap-vm-web-1 icmp` thấy dst IP **`10.0.0.5`** (đã DNAT). Reply qua `LR_OUT_UNDNAT`, src về `203.0.113.50` khi ra external.
8. **Quan sát ARP responder**: trên external host giả lập, `arping 203.0.113.50` thấy MAC trả lời là **`02:00:00:00:00:50`** (external_mac đã set), KHÔNG phải MAC của LRP `lr1-public` (`02:00:00:00:00:02`).

**Biến thể POE**:

- **(a) Đổi `dnat_and_snat` sang `dnat` riêng**: chạy lại bước 7, ping từ external vào `203.0.113.50` thành công nhưng reply src = `10.0.0.5` (lộ private IP) → external client thấy hai IP khác nhau, kết nối TCP fail handshake. Đây là demo điểm khác biệt `dnat` vs `dnat_and_snat`.
- **(b) Bỏ `external_mac`**: kiểm `arping 203.0.113.50` thấy MAC trả lời là MAC của LRP (`02:00:00:00:00:02`) thay vì `02:00:00:00:00:50` đã set. Demo vai trò `external_mac`.
- **(c) Set `options:stateless=true`**: chạy lại bước 6, conntrack zone không có entry vì stateless bypass conntrack. Hệ quả: reply traffic phải được match qua tĩnh, không qua state — đôi khi gây drop nếu reply path khác outbound path.
- **(d) Thêm row NAT `dnat_and_snat` thứ hai cùng `external_ip=203.0.113.50` khác `logical_ip`**: observe undefined behavior, ovn-northd compile cả hai → priority tie-break random, traffic split không deterministic. Đây là demo vì sao CMS phải đảm bảo unique `external_ip` cho mỗi `dnat_and_snat`.

**Falsification check**: nếu set `external_port_range="10000-10005"` (chỉ 6 port) cho SNAT trong setup tenant nhiều VM, predict: connection thứ 7 trở đi sẽ drop với `nf_nat: port range exhausted`; observe: `dmesg | grep nf_nat` thấy log đúng. Confirm: SNAT pool size = `(số IP × range port)` là giới hạn cứng; range hẹp là cause của symptom production case OVN-2025-Q1 ở axis 17.

### Lỗi thường gặp + tín hiệu chẩn đoán

**Failure mode A, SNAT port exhaustion**:

- Triệu chứng: ~5-15% egress traffic drop intermittent giờ peak; `dmesg | grep nf_nat` thấy `port range exhausted`.
- Diagnostic: `ovs-appctl dpctl/dump-conntrack zone=<lr_zone> | grep <external_ip> | wc -l` xấp xỉ ephemeral pool size; `ovn-nbctl --columns=external_port_range list NAT <uuid>` thấy rỗng.
- Recovery: set `external_port_range="10000-65000"` (mở rộng pool) hoặc thêm IP thứ hai vào SNAT pool qua thêm row NAT thứ hai cùng `logical_ip` CIDR.

**Failure mode B, Floating IP không reach (ARP không trả lời)**:

- Triệu chứng: từ external network ping floating IP timeout; `arping <floating_ip>` không có reply.
- Diagnostic: `ovn-sbctl lflow-list lr1 | grep -E "lr_in_ip_input.*<floating_ip>"` xem có ARP responder flow không; `ovn-nbctl --columns=external_mac list NAT <uuid>` xem `external_mac` có set không.
- Recovery: nếu `external_mac` rỗng + LRP external port không có MAC khớp, set `ovn-nbctl set NAT <uuid> external_mac="02:..."` hoặc dùng `dnat_and_snat` không phải `dnat` riêng.

**Failure mode C, Asymmetric traffic do conntrack stale**:

- Triệu chứng: sau khi đổi NAT rule (vd đổi `external_ip` của floating IP từ A sang B), 5-10% connection cũ đang chạy bị drop hoặc reply về sai chassis.
- Diagnostic: `conntrack -L | grep <old_external_ip>` thấy entry cũ vẫn tồn tại với TTL 5 ngày (TCP ESTABLISHED default).
- Recovery: flush conntrack entry liên quan rule cũ qua `conntrack -D -d <old_external_ip>` hoặc `ovs-appctl dpctl/flush-conntrack zone=<lr_zone>` (cẩn thận: flush toàn bộ zone, ảnh hưởng connection đang active).

**Failure mode D, Two row NAT cùng external_ip khác logical_ip (collision)**:

- Triệu chứng: traffic vào floating IP đôi khi reach VM A, đôi khi reach VM B, không deterministic.
- Diagnostic: `ovn-nbctl --columns=type,external_ip,logical_ip list NAT | sort -k2,3` tìm row trùng `(type, external_ip)` khác `logical_ip`.
- Recovery: xác định row sai + xoá qua `ovn-nbctl lr-nat-del lr1 dnat_and_snat <external_ip>` (xoá tất cả NAT cùng external_ip) rồi thêm lại đúng row mong muốn.

Diagnostic mantra cho NAT: **"lr-nat-list → lflow-list 4 stage → ovs-ofctl dump-flows table=(14,12,43,41) → conntrack zone"**, theo đúng thứ tự từ NBDB declarative xuống chassis runtime.

## 13.3.W Năm bảng NBDB đồng hành ACL/LB/NAT đào sâu (Address_Set, Port_Group, Logical_Router_Static_Route, Logical_Router_Policy, Forwarding_Group)

> Phần này đào sâu năm bảng NBDB cùng họ với ACL/LB/NAT, mỗi bảng có treatment riêng đầy đủ 20 góc nhìn (khái niệm → so sánh hệ khác). Bốn bảng đầu (`Address_Set`, `Port_Group`, `Logical_Router_Static_Route`, `Logical_Router_Policy`) thường xuất hiện trực tiếp trong `match` expression hoặc routing logic của LR; bảng thứ năm (`Forwarding_Group`) là pattern active-active gateway thay thế Load_Balancer cho hairpin gateway HA.

### 13.3.W.1 `Address_Set` — tập IP đặt tên dùng trong match expression

#### Khái niệm cốt lõi

`Address_Set` là **bảng nội trú trong OVN Northbound database (NBDB)** mô tả một **tập hợp địa chỉ IP đặt tên** (named collection of IP addresses) bao gồm IPv4, IPv6, hoặc CIDR block, được tham chiếu trong cột `match` của ACL hoặc Logical_Router_Policy qua syntax `$<name>`.
Mỗi row đại diện cho một set logic, ví dụ `vm-prod` chứa 100 IP của VM tier production.
Loại: **OVSDB table root** (`isRoot=true` trong `ovn-nb.ovsschema`), tức là row tồn tại độc lập + có thể được tham chiếu yếu (weak ref) từ nhiều ACL/LRP cùng lúc + không bị cascade delete khi parent xoá.
Có **index unique** trên cột `name` (verified schema branch-22.03 có `"indexes": [["name"]]`), nên hai row không thể trùng `name`.
Khác biệt then chốt với inline IP literal trong `match`: 1 ACL với `ip4.src == $vm-prod` thay thế 100 ACL row riêng cho từng IP, giảm 99% NBDB row + giảm OpenFlow rule sau conjunction compress.

#### Lịch sử + bối cảnh ra đời

`Address_Set` xuất hiện cùng OVN 2.6 (2016) khi nhóm Russell Bryant + Justin Pettit refactor `match` expression engine để hỗ trợ named set thay vì chỉ IP literal. Trước đó, mỗi rule security group Neutron với `remote_ip_prefix` là IP đơn phải sinh 1 row ACL riêng — 100 IP source = 100 ACL row, NBDB phình to. OVN 2.6 thêm Address_Set + cú pháp `$name` trong `match`; OVN 2.10 mở rộng hỗ trợ IPv6; OVN 21.06 thêm hỗ trợ MAC-only set (column `addresses` chấp nhận MAC format `aa:bb:cc:dd:ee:ff`); OVN 22.03 stabilize 3 cột (`name`, `addresses`, `external_ids`); OVN 24.03 cải tiến northd incremental compile cho Address_Set change (chỉ recompute ACL/LRP có tham chiếu, không recompute toàn LS/LR).

#### Vị trí trong kiến trúc

Bảng nằm ở **layer NBDB** input.
Là **root table** nên không có parent strong-ref; sống độc lập, xoá Logical_Switch không cascade xoá Address_Set tham chiếu trong ACL của nó.
Scope: **cluster-wide**, mọi ACL/LRP của mọi LS/LR đều có thể tham chiếu cùng một Address_Set qua `$name`.
Không bao giờ xuất hiện trên SBDB; ovn-northd resolve `$<name>` ngay khi compile NBDB → SBDB, expand thành disjunction inline trong `match` của Logical_Flow (vd `$vm-prod` thành `(ip4.src == 10.0.0.5 || ip4.src == 10.0.0.6 || ...)`).
Sau khi resolve, conjunction compress giảm OpenFlow rule.

#### Vai trò + nhiệm vụ

`Address_Set` gánh ba vai trò:
(1) **Trừu tượng hoá danh sách IP** thành 1 named entity reusable, dễ maintain.
(2) **Giảm số ACL row** trong NBDB qua factor N (số IP trong set).
(3) **Trigger conjunction compression** ở SBDB → OF translation: northd phát hiện `match` chứa `$<addr_set>` cross-product với Port_Group sinh `conjunction()` action giảm OpenFlow rule từ M×N xuống M+N+1.

#### Vì sao sinh ra (động cơ thiết kế)

Trước OVN 2.6, kiến trúc Neutron Security Group ánh xạ mỗi `SecurityGroupRule.remote_ip_prefix` thành 1 ACL row. Nếu rule dùng `RemoteSecurityGroup` (allow traffic từ tất cả VM thuộc SG khác), Neutron resolve danh sách IP runtime → push 1 ACL row per IP. 100 VM tham chiếu = 100 ACL rule, push lại mỗi khi VM thêm/xoá. Áp lực NBDB transaction + northd recompile + conntrack churn đáng kể. OVN 2.6 thiết kế Address_Set để CMS push **1 row Address_Set + 1 ACL** thay vì **100 ACL row**: thêm/bớt IP chỉ update cột `addresses` của Address_Set, ACL không thay đổi, northd recompile chỉ Logical_Flow chứa `$name`. Scale cải thiện rõ rệt: kube-ovn translate Kubernetes NetworkPolicy với `podSelector` matching 5000 pod thành 1 Address_Set 5000 entry + 1 ACL row, thay vì 5000 ACL row.

#### Vấn đề mà nó giải quyết

Bảng giải quyết bốn vấn đề kỹ thuật: (1) **NBDB transaction churn** khi danh sách IP thay đổi (chỉ update 1 row thay vì N row); (2) **ACL row explosion** khi rule có ý nghĩa "from group X, to group Y" — Cartesian giảm M×N → M+N+1; (3) **OpenFlow rule explosion** sau compile — conjunction compression dựa trên Address_Set + Port_Group tham chiếu; (4) **Đồng bộ giữa CMS và NBDB**: thêm/xoá VM chỉ push 1 update vào Address_Set, không cần CMS trực tiếp manage 100 ACL row.

#### Tầm quan trọng

Đây là bảng **tier 2 medium** của NBDB, cần thiết cho mọi production deployment có > 50 VM. Lý do: 100% deployment OpenStack/Kubernetes ở quy mô production dùng Address_Set qua plugin networking-ovn hoặc kube-ovn; mọi tenant với security group tham chiếu group khác đều sinh Address_Set; mọi NetworkPolicy với `podSelector` đều sinh Address_Set dynamic. Kỹ sư không hiểu Address_Set sẽ không debug được "ACL không match đúng IP của VM mới thêm" (Address_Set chưa update kịp), không phân biệt được "ACL push từ CMS" vs "Address_Set push từ CMS" trong audit log NBDB.

#### Cơ chế hoạt động

Một row Address_Set đi qua bốn giai đoạn: (a) **Khai báo từ CMS hoặc kỹ sư**: `ovn-nbctl create Address_Set name=vm-prod addresses='["10.0.0.5","10.0.0.6","10.0.0.7"]'` → NBDB transaction insert row mới. (b) **Tham chiếu trong ACL**: `ovn-nbctl acl-add ls-tenant1 from-lport 1000 'ip4.src == $vm-prod && tcp.dst == 5432' allow-related` → cột `match` của ACL chứa string `$vm-prod`. (c) **Northd compile**: hàm `build_addrset_lflow` (hoặc tương đương trong `northd/northd.c`) iterate Logical_Flow chứa `$<name>`, expand thành disjunction inline `(ip4.src == 10.0.0.5 || ip4.src == 10.0.0.6 || ip4.src == 10.0.0.7)`. (d) **Conjunction compress + cài flow**: nếu disjunction có > 1 entry kết hợp với Port_Group, northd sinh `conjunction(N, K/M)` action; ovn-controller cài OpenFlow rule lên br-int.

#### Phân loại cột

Schema 3 cột (verified `ovn-nb.ovsschema` branch-22.03): `name` (string, có index unique, required), `addresses` (set string min=0 max=unlimited, mỗi entry là IP/CIDR/MAC theo OVN expression syntax), `external_ids` (map string→string cho CMS metadata như `neutron:security_group_id`).

#### Cách kỹ sư vận hành thành thạo

Bốn kỹ năng: (1) **Tạo + verify Address_Set**: `ovn-nbctl create Address_Set name=vm-prod addresses='["10.0.0.5","10.0.0.6"]'` rồi `ovn-nbctl list Address_Set vm-prod`; (2) **Update inline thêm IP**: `ovn-nbctl add Address_Set vm-prod addresses 10.0.0.7` (không phải replace toàn bộ); (3) **Tham chiếu trong ACL match**: dùng prefix `$` trong match expression; (4) **Verify resolve sau compile**: `ovn-sbctl lflow-list ls-tenant1 | grep "ip4.src"` thấy disjunction expanded inline.

#### Quy trình sử dụng (workflow chuẩn)

(1) Tạo Address_Set với name + addresses ban đầu; (2) Tham chiếu trong ACL/LRP qua `$name`; (3) Update Address_Set qua `add`/`remove` thay vì recreate (tránh transaction lớn); (4) Kiểm Logical_Flow ở SBDB resolve đúng; (5) Best practice: đặt naming convention rõ (`vm-<role>-<env>`, vd `vm-web-prod`) để audit dễ dàng.

#### Khi xảy ra sự cố

Symptom thường gặp: "ACL match `$vm-prod` không hit cho IP mới thêm" → kiểm Address_Set đã chứa IP chưa qua `ovn-nbctl list Address_Set vm-prod`. "ACL push thành công nhưng northd compile báo error `unknown address set`" → Address_Set chưa tồn tại hoặc tên typo. "Performance degradation sau khi Address_Set vượt 10K entry" → cân nhắc chia thành nhiều Address_Set theo tier hoặc subnet.

#### Khác biệt giữa các phiên bản

OVN 2.6: ra mắt 3 cột; OVN 2.10: hỗ trợ IPv6 entry; OVN 21.06: hỗ trợ MAC-only set (cho `eth.src/eth.dst` match); OVN 22.03: stabilize 3 cột; OVN 24.03: incremental compile chỉ recompute ACL/LRP có tham chiếu, không recompute toàn LS/LR.

#### Cách quan sát + xác minh

`ovn-nbctl list Address_Set` snapshot toàn bộ; `ovn-nbctl list Address_Set vm-prod` chi tiết.
Đối chiếu compile output: `ovn-sbctl lflow-list ls-tenant1 | grep -E "(10\\.0\\.0\\.5|10\\.0\\.0\\.6)"` đã expand đúng IP.
Verify size: `ovn-nbctl --columns=name,addresses list Address_Set | awk '/addresses/ {print NF}'` đếm số IP per set, cảnh báo khi > 5K.
External_ids tracking: `ovn-nbctl --columns=external_ids list Address_Set | grep neutron:security_group_id` truy nguồn CMS đã tạo set nào.

#### Source code tham chiếu

Verified upstream OVN branch-22.03: schema [`ovn-nb.ovsschema`](https://raw.githubusercontent.com/ovn-org/ovn/branch-22.03/ovn-nb.ovsschema) khai báo bảng `Address_Set` với 3 cột + `"isRoot": true` + `"indexes": [["name"]]`.
Compile resolution thực hiện trong `northd/northd.c` (function name anchor — tham khảo `expr_addr_sets_*` helpers trong `lib/expr.c` của repo OVS gốc, OVN reuse qua `controller/lflow.c`).
Address_Set không có hàm compile riêng — engine resolve `$<name>` trực tiếp khi parse `match` expression của ACL/LRP, qua hàm `expr_parse_string` + `expr_macros_get` ở OVS expr library.
Khi commit Address_Set thay đổi, ovn-northd I-P engine input change "northd_input_nbrec_address_set" trigger recompile các Logical_Flow tham chiếu.

#### Trường hợp sự cố thực tế

Production case 2025: cluster Kubernetes ovn-kube ~3000 pod, NetworkPolicy với `podSelector` matching ~500 pod tier `app`. Sau khi pod scale up từ 500 → 800, ACL match `$as-app` không hit cho pod mới. Investigation: ovnkube-master push Address_Set update có lag ~30 giây (rate-limited). Fix: tăng `ovnkube-master` rate limit, đồng thời chia Address_Set theo namespace để giảm size mỗi set.

#### Bài tập synthetic

```bash
sudo ovn-nbctl create Address_Set name=trusted-clients addresses='["10.0.0.5","10.0.0.6","192.168.1.0/24"]'
sudo ovn-nbctl acl-add ls-tenant1 from-lport 1000 'ip4.src == $trusted-clients && tcp.dst == 22' allow-related
sudo ovn-nbctl add Address_Set trusted-clients addresses 10.0.0.7
sudo ovn-sbctl lflow-list ls-tenant1 | grep "ip4.src"
```

Quan sát: Logical_Flow chứa disjunction `(ip4.src == 10.0.0.5 || ip4.src == 10.0.0.6 || ip4.src == 192.168.1.0/24 || ip4.src == 10.0.0.7)`.

#### Lỗi thường gặp + tín hiệu chẩn đoán

(a) Reference name typo (`$vm_prod` vs `$vm-prod`) → northd error log; (b) Mixed IPv4/IPv6 trong cùng set + match dùng `ip4.src == $set` → IPv6 entry bị skip silently; (c) Set vượt 10K entry → northd compile chậm + br-int OF rule explosion (cân nhắc chia set hoặc dùng Port_Group nếu thực thể là LSP).

#### So sánh với hệ khác

Linux `ipset` tương đồng (named set IP cho iptables match); Cisco IOS `object-group network` cùng concept; AWS Security Group `prefix-list` tương tự (named set CIDR); NSX-T `IP Set` ánh xạ trực tiếp.

### 13.3.W.2 `Port_Group` — tập LSP đặt tên dùng cho ACL apply chung

#### Khái niệm cốt lõi

`Port_Group` là **bảng nội trú trong OVN NBDB** mô tả một **tập hợp Logical_Switch_Port (LSP) đặt tên**, gắn với một danh sách ACL áp chung cho tất cả LSP trong group.
Mỗi row đại diện cho một group logic, vd `pg-web` chứa LSP của tất cả VM tier web.
Loại: **OVSDB table root** (`isRoot=true`); `ports` là weak ref tới `Logical_Switch_Port` (xoá LSP không cascade xoá Port_Group); `acls` là strong ref tới `ACL` (xoá Port_Group cascade xoá ACL gắn group).
Index unique trên `name`.
Khác biệt cốt yếu với "ACL per LSP": 1 ACL gắn pg-web áp cho 100 LSP thay vì 100 ACL riêng — giảm 99% NBDB row + tận dụng conjunction compression sau compile.

#### Lịch sử + bối cảnh ra đời

`Port_Group` xuất hiện OVN 2.10 (2018) khi Han Zhou + Numan Siddique đề xuất sau khi Address_Set không đủ (Address_Set chỉ chứa IP, không tham chiếu LSP để match `inport`/`outport`).
Trước đó, micro-segmentation Neutron Security Group dùng Address_Set cho `ip4.src/dst` + ACL per LSP cho `inport/outport` → vẫn explosion ở chiều LSP.
Port_Group bổ sung thẳng: 1 ACL với `inport == @pg-web && outport == @pg-app` thay 100×100 = 10K ACL row riêng.
OVN 21.06 thêm hỗ trợ IPv6 implicit (Port_Group inherit address family từ LSP); OVN 22.03 stabilize cột `external_ids` cho CMS metadata.

#### Vị trí trong kiến trúc

Bảng nằm ở **layer NBDB** input.
Là **root table**, không có parent strong-ref.
Scope: **cluster-wide**, một LSP có thể là member của nhiều Port_Group cùng lúc (1:N).
ovn-northd resolve `@<name>` trong match expression khi compile; thay thế bằng disjunction `(inport == "lsp1" || inport == "lsp2" || ...)`.
Sau resolve, kết hợp Address_Set + Port_Group cross-product trigger conjunction compression.

#### Vai trò + nhiệm vụ

Port_Group gánh ba vai trò:
(1) **Trừu tượng hoá tier topology** thành named entity reusable cho ACL apply.
(2) **Cơ sở scale ACL** giảm row count factor N (số LSP).
(3) **Trigger conjunction compression** ở OpenFlow level kết hợp với Address_Set hoặc Port_Group khác.

#### Vì sao sinh ra (động cơ thiết kế)

Trước OVN 2.10, Address_Set giải quyết được IP set nhưng không match được `inport`/`outport` (port là LSP, không phải IP). Hệ quả: micro-segmentation 3-tier 1000 VM tạo ra hàng chục nghìn ACL row dù có Address_Set. OVN 2.10 thiết kế Port_Group cho phép `match` tham chiếu trực tiếp tập LSP qua `@name` — giảm ACL row đến 99% với scale lớn. OVN-Kubernetes CNI (kube-ovn, ovn-kubernetes) dựa trên Port_Group để translate Kubernetes NetworkPolicy `podSelector` → OVN ACL.

#### Vấn đề mà nó giải quyết

(1) **NBDB ACL row explosion** với scale > 100 LSP; (2) **OpenFlow rule explosion** sau compile (conjunction compress chỉ trigger khi match có ≥ 2 set/group cross-product); (3) **CMS sync efficiency**: thêm/xoá VM khỏi tier chỉ update cột `ports` của Port_Group, không thêm ACL row.

#### Tầm quan trọng

Đây là bảng **tier 2 medium** của NBDB, cần thiết cho mọi production scale > 100 LSP. Là foundation của Kubernetes NetworkPolicy → OVN ACL translation. Kỹ sư không hiểu Port_Group sẽ không phân biệt được "ACL apply per LS" vs "ACL apply per Port_Group" — cả hai có hành vi khác nhau ở scope (LS = mọi LSP của LS, Port_Group = chỉ LSP trong group, cross-LS).

#### Cơ chế hoạt động

(a) `ovn-nbctl pg-add pg-web web-vm-1 web-vm-2` → tạo row Port_Group + insert LSP UUID vào cột `ports`. (b) `ovn-nbctl acl-add pg-web from-lport 1000 'inport == @pg-web && tcp.dst == 80' allow-related` → tạo ACL row + insert UUID vào cột `acls` của Port_Group. (c) Northd resolve `@pg-web` thành disjunction `(inport == "web-vm-1" || inport == "web-vm-2")` trong Logical_Flow. (d) ovn-controller cài OpenFlow rule trên mọi chassis có Port_Binding của LSP thuộc group.

#### Phân loại cột

Schema 4 cột (verified branch-22.03): `name` (string, index unique), `ports` (set UUID weak ref Logical_Switch_Port), `acls` (set UUID strong ref ACL), `external_ids` (map). Không có cột priority/match/action — Port_Group chỉ là container.

#### Cách kỹ sư vận hành thành thạo

(1) Tạo Port_Group: `ovn-nbctl pg-add pg-web web-vm-1 web-vm-2`; (2) Add LSP vào group: `ovn-nbctl pg-set-ports pg-web web-vm-1 web-vm-2 web-vm-3` (replace) hoặc `ovn-nbctl add Port_Group pg-web ports <uuid>` (incremental); (3) Apply ACL: `ovn-nbctl acl-add pg-web from-lport 1000 'match' allow-related`; (4) Verify: `ovn-nbctl --columns=name,ports,acls list Port_Group pg-web`.

#### Quy trình sử dụng (workflow chuẩn)

(1) Plan tier topology (vd web/app/db); (2) Tạo Port_Group cho mỗi tier; (3) Add LSP vào đúng group khi tạo VM (CMS hoặc kỹ sư tự); (4) Apply ACL cho group (rule allow + default-deny); (5) Verify Logical_Flow expand đúng + counter `n_packets` tăng khi traffic match.

#### Khi xảy ra sự cố

"ACL `outport == @pg-app` không hit dù VM target có Port_Binding" → kiểm LSP đã trong group chưa qua `ovn-nbctl --columns=ports list Port_Group pg-app`. "VM mới scale up không có ACL apply" → CMS chưa add LSP vào group. "Northd compile chậm" → group quá lớn (> 5000 LSP) cần chia.

#### Khác biệt giữa các phiên bản

OVN 2.10: ra mắt; OVN 21.06: implicit IPv6 (Port_Group inherit address family từ LSP); OVN 22.03: stabilize `external_ids`; OVN 24.03: incremental compile cho Port_Group change.

#### Cách quan sát + xác minh

`ovn-nbctl list Port_Group pg-web` chi tiết; `ovn-sbctl lflow-list ls-tenant1 | grep "inport"` thấy disjunction expanded; `ovs-ofctl -O OpenFlow15 dump-flows br-int | grep conjunction` xác nhận conjunction compress active.

#### Source code tham chiếu

Schema [`ovn-nb.ovsschema` branch-22.03](https://raw.githubusercontent.com/ovn-org/ovn/branch-22.03/ovn-nb.ovsschema) bảng `Port_Group` 4 cột root + index unique `name`. Resolve `@<name>` trong [`controller/lflow.c`](https://github.com/ovn-org/ovn/blob/branch-22.03/controller/lflow.c) qua expr engine `lib/expr.c` (function anchor — tham khảo `expr_port_groups_*`).

#### Trường hợp sự cố thực tế

Production case 2024: Kubernetes cluster ovn-kube với 2000 pod, NetworkPolicy 50 rule. Trước khi dùng Port_Group: ovnkube-master push 100K ACL row, NBDB Raft replicate chậm, latency 10-30s. Sau refactor dùng Port_Group + Address_Set: 50 ACL row + 50 Port_Group + 100 Address_Set, NBDB Raft replicate < 1s, latency policy enforcement giảm 30 lần.

#### Bài tập synthetic

```bash
sudo ovn-nbctl pg-add pg-web web-vm-1 web-vm-2
sudo ovn-nbctl pg-add pg-app app-vm-1
sudo ovn-nbctl acl-add pg-app from-lport 1000 'inport == @pg-web && outport == @pg-app && tcp.dst == 8080' allow-related
sudo ovn-nbctl acl-add pg-app from-lport 1 'ip' drop
sudo ovn-sbctl lflow-list ls-tenant1 | grep -E "ls_in_acl|conjunction"
```

Quan sát conjunction action sinh ra với `@pg-web × @pg-app` cross-product.

#### Lỗi thường gặp + tín hiệu chẩn đoán

(a) Quên add LSP vào Port_Group khi scale up VM → ACL không apply cho VM mới; (b) Tham chiếu Port_Group không tồn tại trong match → northd error compile; (c) Group có > 5K LSP → khuyến nghị chia theo subnet/tier để giữ northd compile time hợp lý.

#### So sánh với hệ khác

Cisco `object-group port` cùng concept group hoá; NSX-T `Security Group` ánh xạ trực tiếp Port_Group + ACL; Kubernetes `NetworkPolicy.podSelector` được kube-ovn dịch thành Port_Group dynamic. Linux không có equivalent native (iptables không có concept group LSP, mỗi rule per-host gắn chain).

### 13.3.W.3 `Logical_Router_Static_Route` — bảng route tĩnh cho Logical_Router

#### Khái niệm cốt lõi

`Logical_Router_Static_Route` (viết tắt LRSR) là **bảng nội trú trong OVN NBDB** mô tả một **route tĩnh** gắn vào một `Logical_Router` (LR).
Mỗi row khai báo: `ip_prefix` (đích, IPv4 hoặc IPv6 với CIDR), `nexthop` (next-hop IP), `output_port` (LRP đi ra, optional), `policy` (`src-ip` hoặc `dst-ip` để chọn chiều match), `bfd` (UUID BFD session để gắn liveness check), `options` (map cho behavior tinh chỉnh như `ecmp_symmetric_reply`).
Loại: **OVSDB table non-root** (`isRoot=false`), strong ref từ `Logical_Router.static_routes`.
Khác biệt với route động (mà giao thức như BGP/OSPF tự đàm phán): static route khai báo declarative, không có protocol negotiation.
ECMP hoạt động qua duplicate `ip_prefix` cùng route_table khác nexthop — northd compile thành flow `group=...,type=select` ở stage `LR_IN_IP_ROUTING_ECMP`.

#### Lịch sử + bối cảnh ra đời

Bảng xuất hiện OVN 2.5 (2016) cùng với Logical_Router. Schema ban đầu chỉ có 4 cột (`ip_prefix`, `nexthop`, `output_port`, `external_ids`). OVN 2.10 thêm `policy` (enum `src-ip`/`dst-ip`); OVN 20.06 thêm `bfd` ref + ECMP support qua duplicate row; OVN 21.06 thêm `options` map (key `ecmp_symmetric_reply`); OVN 22.06 thêm `route_table` cho VRF (multiple route table per LR); OVN 23.09 thêm `options:origin` để mark route là từ BGP-import (thuộc dpu deployment, không thuộc plan này).

#### Vị trí trong kiến trúc

Bảng nằm ở **layer NBDB** input, scope **per-Logical_Router**. Strong ref từ `Logical_Router.static_routes` (set UUID). Một LR có thể có hàng trăm static route (default route + per-subnet route + ECMP duplicate). Compile thành Logical_Flow ở stage `LR_IN_IP_ROUTING` (logical table 9 OVN 22.03) + `LR_IN_IP_ROUTING_ECMP` (table 10) cho ECMP. Không bao giờ áp lên Logical_Switch.

#### Vai trò + nhiệm vụ

Bốn vai trò: (1) **Định tuyến IP traffic không trùng directly-connected subnet**: route `0.0.0.0/0` → external gateway cho egress internet; (2) **Khai báo VRF qua `route_table`**: phân vùng route theo tenant/VRF (OVN 22.06+); (3) **ECMP load-balance**: 2 row cùng `ip_prefix` khác `nexthop` sinh group select hash; (4) **BFD-tracked liveness**: gắn `bfd` UUID, route bị remove khỏi RIB tự động khi BFD detect link down.

#### Vì sao sinh ra (động cơ thiết kế)

Trước OVN 2.5, Neutron L3-agent dùng `ip route add` trong netns của router. Vấn đề: failover phụ thuộc keepalived VRRP 5-30s, không có concept ECMP nguyên bản, BFD phải chạy ngoài netns rồi sync state vào kernel route table thủ công, không tích hợp được với conntrack zone của OVN. OVN LRSR thiết kế declarative trong NBDB, ovn-northd compile thành OpenFlow distributed, BFD tích hợp native qua bảng BFD riêng, ECMP qua duplicate row + group select.

#### Vấn đề mà nó giải quyết

(1) **Static route declarative cluster-wide**, không phụ thuộc per-host `ip route`; (2) **ECMP với hash 5-tuple** cho load-balance bandwidth; (3) **BFD failover < 200ms** qua bảng BFD ref; (4) **VRF separation** qua `route_table` cho multi-tenant routing isolation; (5) **Symmetric reply** qua `options:ecmp_symmetric_reply=true` đảm bảo reply traffic chọn cùng nexthop với forward (tránh stateful firewall asymmetric drop).

#### Tầm quan trọng

Đây là bảng **tier 2 medium** của NBDB, cần thiết cho mọi LR có default route hoặc kết nối subnet không directly-connected. 100% deployment có internet egress đều có ít nhất 1 row default route. Production cluster với multi-AZ topology dùng ECMP duplicate route cho redundancy.

#### Cơ chế hoạt động

(a) `ovn-nbctl lr-route-add lr1 0.0.0.0/0 192.168.1.1` → NBDB transaction tạo row LRSR + INSERT UUID vào `Logical_Router.static_routes` của lr1. (b) Northd `build_static_route_flows` (function anchor trong `northd/northd.c`) iterate routes, compile thành Logical_Flow ở stage `LR_IN_IP_ROUTING` với `match=(ip4.dst == 0.0.0.0/0)` + `action=(ip.ttl--; reg0 = 192.168.1.1; reg9[2] = 0; ...; next;)`. (c) ECMP detection: nếu có ≥ 2 row cùng `ip_prefix` cùng `route_table`, northd build group select với hash 5-tuple. (d) BFD: nếu cột `bfd` set, ovn-northd thêm condition match dựa trên `BFD.status=up` (nếu down, route bị skip).

#### Phân loại cột

Schema 8 cột (verified branch-22.03): `route_table` (string, default "" empty cho default VRF, 22.06+), `ip_prefix` (string required, IPv4/IPv6 CIDR), `policy` (enum src-ip/dst-ip optional, default dst-ip), `nexthop` (string required), `output_port` (string optional, tên LRP), `bfd` (UUID weak ref BFD optional), `options` (map options), `external_ids` (map metadata).

#### Cách kỹ sư vận hành thành thạo

(1) Add default route: `ovn-nbctl lr-route-add lr1 0.0.0.0/0 192.168.1.1`; (2) Add per-subnet route với specific output_port: `ovn-nbctl lr-route-add lr1 10.50.0.0/16 10.40.0.1 lrp-to-net-50`; (3) Add ECMP: thêm 2 row cùng prefix khác nexthop; (4) Verify: `ovn-nbctl lr-route-list lr1` thấy danh sách + ECMP marker `ecmp` nếu có duplicate; (5) Set BFD: `ovn-nbctl set Logical_Router_Static_Route <uuid> bfd=<bfd_uuid>`.

#### Quy trình sử dụng (workflow chuẩn)

(1) Plan routing topology (default route + per-VRF route + ECMP); (2) Add route per LR; (3) (Optional) Set policy `src-ip` cho source-routing tenant; (4) (Optional) Gắn BFD session cho liveness; (5) Verify compile xuống SBDB; (6) Test ping từ VM nội bộ ra dst trong route + observe via tcpdump trên LRP next-hop chassis.

#### Khi xảy ra sự cố

"VM ping ra internet timeout" → kiểm `ovn-nbctl lr-route-list lr1` có default route không + nexthop reachable không (`ovn-nbctl list BFD` xem status). "ECMP không balance đều" → kiểm hash 5-tuple có đa dạng không (cùng client → cùng backend là expected behavior); cân nhắc `selection_fields`. "Route đến subnet directly-connected nhưng vẫn add row LRSR" → conflict, ưu tiên directly-connected route luôn win, LRSR row dư thừa.

#### Khác biệt giữa các phiên bản

OVN 2.5: 4 cột cơ bản; OVN 2.10: thêm `policy`; OVN 20.06: thêm `bfd` + ECMP duplicate row; OVN 21.06: thêm `options:ecmp_symmetric_reply`; OVN 22.06: thêm `route_table` cho VRF; OVN 23.09: thêm `options:origin`.

#### Cách quan sát + xác minh

`ovn-nbctl lr-route-list lr1` snapshot human-readable; `ovn-nbctl --columns=ip_prefix,nexthop,output_port,bfd,options,policy list Logical_Router_Static_Route` raw NBDB; `ovn-sbctl lflow-list lr1 | grep -E "lr_in_ip_routing"` xem compile output; `ovs-ofctl -O OpenFlow15 dump-groups br-int` xem ECMP group select.

#### Source code tham chiếu

Schema [`ovn-nb.ovsschema` branch-22.03](https://raw.githubusercontent.com/ovn-org/ovn/branch-22.03/ovn-nb.ovsschema) bảng 8 cột non-root. Compile trong [`northd/northd.c`](https://github.com/ovn-org/ovn/blob/branch-22.03/northd/northd.c) — function anchor `build_static_route_flow` (hoặc `build_static_route_flows_for_lrouter` tuỳ refactor) iterate route + sinh Logical_Flow ở stage `LR_IN_IP_ROUTING`. ECMP build qua hàm xử lý `ecmp_groups` với group_id allocate runtime.

#### Trường hợp sự cố thực tế

Production case 2024: cluster OSP với 2 datacenter site-A + site-B, ECMP route từ tenant LR sang shared service network qua 2 transit LRP (mỗi site 1). Sau khi site-B link down, BFD detect 200ms, route ECMP bị remove member site-B. Tuy nhiên, một số connection TCP đang chạy từ trước vẫn pin sang site-B (conntrack pinning) → drop intermittent ~5 phút cho đến khi conntrack expire. Fix: bật `options:ecmp_symmetric_reply=true` để reply path follow forward; thêm conntrack flush khi route change.

#### Bài tập synthetic

```bash
sudo ovn-nbctl lr-route-add lr1 0.0.0.0/0 192.168.1.1
sudo ovn-nbctl lr-route-add lr1 10.100.0.0/16 10.40.0.1
sudo ovn-nbctl lr-route-add lr1 10.100.0.0/16 10.40.0.2
sudo ovn-nbctl lr-route-list lr1
sudo ovn-sbctl lflow-list lr1 | grep "lr_in_ip_routing" | head -10
sudo ovs-ofctl -O OpenFlow15 dump-groups br-int | head -10
```

Quan sát: 2 route cùng `10.100.0.0/16` thành ECMP group; lr-route-list đánh dấu `ecmp`.

#### Lỗi thường gặp + tín hiệu chẩn đoán

(a) Default route nhưng nexthop không trong directly-connected subnet → northd warning + flow không cài; (b) ECMP duplicate route không có BFD → khi nexthop chết, traffic 50% drop cho đến khi admin manually xoá row; (c) `policy=src-ip` không match đúng → cần kiểm `match` expression compile output, source-routing có ưu tiên thấp hơn dst-routing trừ khi priority được điều chỉnh.

#### So sánh với hệ khác

Linux `ip route add` cùng concept; Cisco IOS `ip route` cùng syntax (network mask via-gateway); FRR static module tương đồng; AWS VPC Route Table ánh xạ tương tự (Destination CIDR → Target). Khác biệt then chốt: OVN LRSR là declarative cluster-wide, ovn-northd compile + push xuống mỗi chassis qua OpenFlow, không phụ thuộc per-host kernel route table.

### 13.3.W.4 `Logical_Router_Policy` — policy-based routing trên LR

#### Khái niệm cốt lõi

`Logical_Router_Policy` (LRP, không nhầm với Logical_Router_Port cùng viết tắt) là **bảng nội trú trong OVN NBDB** mô tả một **rule policy-based routing (PBR)** gắn vào `Logical_Router`.
Khác với LRSR (chỉ match destination IP), LRP cho phép match **microflow expression** (giống ACL match) bao gồm `ip4.src`, `ip4.dst`, `tcp.dst`, `inport`, `ct.est`, v.v.
Apply 1 trong các action: `allow` (cho qua, route bình thường), `drop` (chặn), `reroute` (đổi nexthop sang IP khác).
Loại: **OVSDB table non-root**, strong ref từ `Logical_Router.policies`.
Priority `0..32767`, highest wins.
Đứng **trước** static route trong pipeline (stage `LR_IN_POLICY` table 7, trước `LR_IN_IP_ROUTING` table 9 trong OVN 22.03).

#### Lịch sử + bối cảnh ra đời

LRP xuất hiện OVN 21.06 (2021) khi Numan Siddique đề xuất để hỗ trợ source-routing + traffic steering nâng cao cho deployment SD-WAN/multi-tenant. Trước đó, static route `policy=src-ip` chỉ làm được source-routing đơn giản; không match được L4 (tcp.dst). LRP mở rộng match expression sang microflow đầy đủ + thêm action `reroute` cho traffic steering. OVN 22.06 thêm `nexthops` (list) cho ECMP-style PBR; OVN 24.03 thêm action thứ tư `jump` (nhảy sang priority khác trong cùng table, giống iptables jump target — note: tại branch-22.03 baseline curriculum chỉ có 3 action `allow`/`drop`/`reroute`).

#### Vị trí trong kiến trúc

Bảng nằm ở **layer NBDB** input, scope **per-Logical_Router**. Strong ref từ `Logical_Router.policies` (set UUID). Compile thành Logical_Flow ở stage `LR_IN_POLICY` (table 7 OVN 22.03), evaluate **trước** stage routing chính. Thứ tự pipeline: `LR_IN_ADMISSION` → `LR_IN_LOOKUP_NEIGHBOR` → `LR_IN_LEARN_NEIGHBOR` → `LR_IN_IP_INPUT` → ... → `LR_IN_POLICY` → `LR_IN_ARP_RESOLVE` → `LR_IN_IP_ROUTING` → ... Hệ quả: LRP thắng static route, có thể override bằng action `reroute`.

#### Vai trò + nhiệm vụ

Bốn vai trò:
(1) **Source-based routing**: route theo source IP/subnet thay vì destination.
(2) **L4-aware steering**: vd traffic HTTPS từ tier app phải đi qua security gateway.
(3) **Selective drop** ở L3 (bổ sung cho ACL ở L2 LS) cho rule áp tại router scope.
(4) **Traffic engineering**: re-route subset traffic sang underlay khác cho QoS/cost optimization.

#### Vì sao sinh ra (động cơ thiết kế)

Trước OVN 21.06, kỹ sư phải dùng combo `static_route policy=src-ip` (chỉ source IP, không L4) + ACL trên LS upstream/downstream để mô phỏng PBR. Cách này hạn chế và phân tán rule ở 2 layer (LS và LR). LRP gom logic PBR vào 1 bảng tại LR + match expression đầy đủ microflow. Đặc biệt hữu ích cho SD-WAN deployment cần steer traffic theo điều kiện L4-L7 hint (vd `tcp.dst == 443 && ip4.dst == $public-saas` → reroute qua proxy edge).

#### Vấn đề mà nó giải quyết

(1) **PBR đầy đủ với match L4** không bị giới hạn ở source IP; (2) **Override static route** cho subset traffic không cần xáo trộn route table chính; (3) **Centralized PBR rule** tại LR thay vì phân tán LS; (4) **Compile-time priority** rõ ràng không dependence thứ tự thêm rule.

#### Tầm quan trọng

Đây là bảng **tier 2 medium** của NBDB, cần thiết cho deployment multi-WAN/SD-WAN/multi-tenant với traffic steering yêu cầu. Không phải mọi cluster đều dùng (cluster đơn giản chỉ cần static route), nhưng khi cần thì đây là công cụ duy nhất ở OVN level. Kỹ sư không hiểu LRP sẽ không debug được "traffic không follow default route mà đi qua nexthop khác" (do LRP override silent).

#### Cơ chế hoạt động

(a) `ovn-nbctl lr-policy-add lr1 100 'ip4.src == 10.0.0.5' reroute 192.168.2.1` → NBDB transaction tạo row LRP + INSERT UUID vào `Logical_Router.policies` của lr1. (b) Northd `build_routing_policy_flow` (function anchor trong `northd/northd.c`) iterate policies, compile thành Logical_Flow ở stage `LR_IN_POLICY` với `match=(ip4.src == 10.0.0.5)` + action tuỳ type: `allow` → `next;`, `drop` → `drop;`, `reroute` → `reg0 = 192.168.2.1; reg9[2] = 0; ...; next;` (set nexthop register override stage routing). (c) ovn-controller cài flow tại OF table tương ứng (mapping `LR_IN_POLICY` = OF table 16 trong OVN 22.03, cross-link Phần 13.16).

#### Phân loại cột

Schema 7 cột (verified branch-22.03): `priority` (integer 0-32767, required), `match` (string required, microflow expression), `action` (enum allow/drop/reroute, required tại 22.03; thêm `jump` từ 24.03+), `nexthop` (string optional, IP nexthop nếu action=reroute), `nexthops` (set string optional, ECMP nexthops 22.06+), `options` (map), `external_ids` (map metadata). Không có index unique — duplicate `(priority, match)` về schema hợp lệ.

#### Cách kỹ sư vận hành thành thạo

(1) Add allow rule: `ovn-nbctl lr-policy-add lr1 200 'ip4.src == 10.0.0.5' allow`; (2) Add drop rule: `ovn-nbctl lr-policy-add lr1 150 'ip4.src == 10.0.0.10 && tcp.dst == 22' drop`; (3) Add reroute: `ovn-nbctl lr-policy-add lr1 100 'ip4.src == 10.0.0.0/24 && ip4.dst == 8.8.8.8' reroute 192.168.99.1`; (4) Verify: `ovn-nbctl lr-policy-list lr1`; (5) Cleanup: `ovn-nbctl lr-policy-del lr1 <priority> [<match>]`.

#### Quy trình sử dụng (workflow chuẩn)

(1) Identify use case (source routing / drop selective / steering); (2) Add LRP với priority phù hợp (cao hơn default route hành vi); (3) Verify compile: `ovn-sbctl lflow-list lr1 | grep "lr_in_policy"`; (4) Test với traffic cụ thể; (5) Cross-check không đè trùng với static route hoặc NAT.

#### Khi xảy ra sự cố

"Default route working bình thường nhưng VM A gặp drop bất ngờ" → kiểm `ovn-nbctl lr-policy-list lr1` có rule drop match `ip4.src` của VM A không; "Reroute không hit" → kiểm priority + match + verify Logical_Flow compile ra đúng; "Action `jump` nhưng OVN 22.03 báo error" → action `jump` chỉ có từ 24.03+, fallback sang `allow` + reorder rule.

#### Khác biệt giữa các phiên bản

OVN 21.06: ra mắt với 3 action allow/drop/reroute, single nexthop; OVN 22.06: thêm `nexthops` set cho ECMP-style PBR; OVN 24.03: thêm action `jump`. Curriculum baseline 22.03: chỉ có 3 action.

#### Cách quan sát + xác minh

`ovn-nbctl lr-policy-list lr1`; `ovn-nbctl --columns=priority,match,action,nexthop,nexthops list Logical_Router_Policy`; `ovn-sbctl lflow-list lr1 | grep "lr_in_policy"`; `ovs-ofctl -O OpenFlow15 dump-flows br-int "table=16"` (OVN 22.03 mapping `LR_IN_POLICY` = OF table 16).

#### Source code tham chiếu

Schema [`ovn-nb.ovsschema` branch-22.03](https://raw.githubusercontent.com/ovn-org/ovn/branch-22.03/ovn-nb.ovsschema) bảng `Logical_Router_Policy` 7 cột non-root, action enum 3 giá trị `allow`/`drop`/`reroute`. Compile trong [`northd/northd.c`](https://github.com/ovn-org/ovn/blob/branch-22.03/northd/northd.c) — function anchor `build_routing_policy_flow` iterate policies + sinh Logical_Flow ở stage `LR_IN_POLICY` (Rule 14.4: function name là anchor stable, line number drift theo version).

#### Trường hợp sự cố thực tế

Production case 2024: deployment multi-WAN với 2 transit (primary cost-effective, backup high-quality). Rule mong muốn: traffic VoIP (UDP dst port 5060/5004) đi qua backup link cho QoS. Implementation: `ovn-nbctl lr-policy-add lr-edge 200 'udp.dst == 5060 || udp.dst == 5004' reroute 10.99.99.1`. Sau triển khai, VoIP latency giảm từ 80ms xuống 15ms. Sau 2 tuần, một lỗi: SIP register bị drop intermittent — root cause: SIP register dùng UDP src port không phải dst, rule match sai chiều. Fix: thêm rule cho cả `udp.src == 5060`.

#### Bài tập synthetic

```bash
sudo ovn-nbctl lr-policy-add lr1 200 'ip4.src == 10.0.0.5 && tcp.dst == 443' reroute 192.168.99.1
sudo ovn-nbctl lr-policy-add lr1 150 'ip4.src == 10.0.0.10' drop
sudo ovn-nbctl lr-policy-add lr1 100 'ip4.src == 10.0.0.0/24' allow
sudo ovn-nbctl lr-policy-list lr1
sudo ovn-sbctl lflow-list lr1 | grep "lr_in_policy"
```

Quan sát: 3 flow priority khác nhau, highest wins; reroute action set register nexthop override.

#### Lỗi thường gặp + tín hiệu chẩn đoán

(a) Priority trùng giữa 2 LRP cùng datapath không có index → undefined tie-break, compile cả hai; (b) Action `reroute` nhưng nexthop không reachable (không có ARP entry) → traffic drop silent ở stage ARP_RESOLVE; (c) Match expression typo (`tcp_dst` thay `tcp.dst`) → northd error log; (d) LRP override default route không mong muốn → audit `lr-policy-list` định kỳ.

#### So sánh với hệ khác

Linux `ip rule add` (policy routing) + `iptables -t mangle MARK` cùng concept; Cisco IOS `route-map` + `set ip next-hop` ánh xạ trực tiếp; FRR PBR plugin tương đồng; AWS Transit Gateway route table policy-based ánh xạ một phần (TGW không có L4-aware match nguyên bản). NSX-T DFW + Distributed Router PBR tương đồng functionality.

### 13.3.W.5 `Forwarding_Group` — active-active gateway pattern thay thế Load_Balancer

#### Khái niệm cốt lõi

`Forwarding_Group` (FG) là **bảng nội trú trong OVN NBDB** mô tả một **nhóm forwarding active-active**: VIP (IP đại diện) + VMAC (MAC đại diện) ánh xạ tới một danh sách `child_port` (LSP của các gateway/server). Khác với Load_Balancer (DNAT VIP → backend với conntrack), Forwarding_Group **không** rewrite địa chỉ — mà chỉ **forward L2** packet tới một trong các `child_port` chọn theo hash, và LSP child phải tự handle traffic (cùng IP/MAC virtual). Pattern này dùng cho **active-active gateway HA** (vd 2-3 firewall appliance trên cùng external segment chia tải) hoặc **multi-NIC bonding logic ở L2 OVN**. Loại: **OVSDB table non-root**, strong ref từ `Logical_Switch.forwarding_groups`. `child_port` min=1 (phải có ít nhất 1 child).

#### Lịch sử + bối cảnh ra đời

Forwarding_Group xuất hiện OVN 20.03 (đầu 2020) khi Manoj Sharma + Numan Siddique đề xuất cho use case active-active firewall HA trong VMware NSX-T migration scenario (NSX-T có Edge Cluster active-active với BFD; OVN cần feature tương đương). Ban đầu schema 5 cột; OVN 20.06 thêm `liveness` (boolean) cho BFD-based health check, đảm bảo chỉ forward sang child có BFD up. OVN 22.03 stabilize 6 cột. Đây là feature niche, không dùng phổ biến (most deployment dùng Load_Balancer + ECMP), nên ít documentation hơn các bảng khác — nhưng quan trọng cho specific HA pattern.

#### Vị trí trong kiến trúc

Bảng nằm ở **layer NBDB** input, scope **per-Logical_Switch**. Strong ref từ `Logical_Switch.forwarding_groups` (set UUID). Compile thành Logical_Flow ở stage `LS_IN_L2_LKUP` (table 19 OVN 22.03) — khi packet match `eth.dst == <vmac>`, action chọn 1 child_port qua hash 5-tuple + `outport=child_port_X`. Không tương tác với Logical_Router (FG là L2-only, không phải L3 NAT).

#### Vai trò + nhiệm vụ

Ba vai trò: (1) **Active-active L2 gateway HA**: VIP/VMAC đại diện cho cluster firewall/gateway, traffic được hash đến 1 instance khác nhau theo flow; (2) **Liveness-aware forwarding**: nếu `liveness=true` + child có BFD session, child down sẽ bị skip; (3) **Alternative đến Load_Balancer khi không cần DNAT**: backend giữ nguyên IP/MAC virtual, không bị rewrite — phù hợp khi backend phải reply với cùng VIP (vd transparent firewall).

#### Vì sao sinh ra (động cơ thiết kế)

Trước OVN 20.03, active-active firewall HA phải implement qua: (a) Load_Balancer DNAT (nhưng backend không cùng IP với VIP, làm vỡ transparency); (b) ECMP route static (chỉ work ở L3 LR, không work ở L2 LS); (c) External hardware (F5, A10, ngoài OVN scope). Forwarding_Group lấp khoảng trống cho deployment cần active-active L2 gateway transparent. Use case điển hình: 2-3 firewall appliance Palo Alto/Fortinet trên cùng external segment, cùng VIP/VMAC, OVN forward traffic đến chúng theo hash để chia tải mà không DNAT.

#### Vấn đề mà nó giải quyết

(1) **Active-active gateway HA L2 transparent** không cần DNAT; (2) **BFD-aware forwarding** với `liveness=true`; (3) **Load distribution theo hash 5-tuple** không dependence trên external LB; (4) **Failover < 200ms** qua BFD detect child down.

#### Tầm quan trọng

Đây là bảng **tier 2 medium niche** của NBDB. Không cần thiết cho deployment chuẩn (Load_Balancer + NAT đủ cho 95% use case), nhưng critical cho deployment migrate từ NSX-T hoặc cần transparent active-active firewall. Production cluster dùng FG thường có architect chuyên sâu network design.

#### Cơ chế hoạt động

(a) `ovn-nbctl create Forwarding_Group name=fg-fw-cluster vip=10.0.0.100 vmac=02:00:00:fw:01 liveness=true child_port='["lsp-fw-1","lsp-fw-2","lsp-fw-3"]'` → tạo row + INSERT vào `Logical_Switch.forwarding_groups`. (b) Northd `build_fwd_group_lflows` (function anchor — tham khảo `northd/northd.c` branch-22.03) compile thành Logical_Flow ở stage `LS_IN_L2_LKUP`: `match=(eth.dst == 02:00:00:fw:01)` + action chọn 1 child_port qua hash. (c) Nếu `liveness=true`, condition match thêm `BFD.status=up` của child. (d) ovn-controller cài OF rule với group select type trên br-int.

#### Phân loại cột

Schema 6 cột (verified branch-22.03): `name` (string required), `vip` (string required, IP đại diện), `vmac` (string required, MAC đại diện), `liveness` (boolean required, true = enable BFD-aware), `child_port` (set string min=1, danh sách LSP name), `external_ids` (map metadata).

#### Cách kỹ sư vận hành thành thạo

(1) Tạo FG: `ovn-nbctl create Forwarding_Group name=fg-fw vip=10.0.0.100 vmac='"02:00:00:fw:01"' liveness=true child_port='["lsp-fw-1","lsp-fw-2"]'`; (2) Attach vào LS: `ovn-nbctl add Logical_Switch ls-ext forwarding_groups <fg_uuid>`; (3) Verify: `ovn-nbctl list Forwarding_Group fg-fw`; (4) (Optional) Setup BFD trên child_port qua bảng BFD; (5) Verify compile: `ovn-sbctl lflow-list ls-ext | grep "ls_in_l2_lkup"`.

#### Quy trình sử dụng (workflow chuẩn)

(1) Plan HA pattern (khi nào dùng FG vs LB); (2) Setup child LSP với cùng VIP/VMAC virtual (tự handle ARP responder hoặc disable); (3) Tạo FG với child_port list; (4) Attach vào LS; (5) Test traffic chia đều: gửi 100 connection từ source khác nhau, verify chia hash đều giữa 3 child; (6) Test failover: down 1 child, verify traffic redirect sang child còn lại trong 200ms (nếu liveness + BFD enabled).

#### Khi xảy ra sự cố

"Traffic không chia đều giữa child" → kiểm hash 5-tuple đa dạng; nếu source ít, hash không phân tán; (b) "Failover chậm > 1s" → kiểm BFD interval (default 200ms × 3 detect_mult = 600ms; tinh chỉnh nếu cần); (c) "Child down nhưng vẫn nhận traffic" → `liveness=false` hoặc BFD chưa bind đúng vào child_port LSP.

#### Khác biệt giữa các phiên bản

OVN 20.03: ra mắt với 5 cột (chưa có `liveness`); OVN 20.06: thêm `liveness` (boolean) + BFD-aware forwarding; OVN 22.03: stabilize 6 cột.

#### Cách quan sát + xác minh

`ovn-nbctl list Forwarding_Group`; `ovn-nbctl --columns=name,vip,vmac,liveness,child_port list Forwarding_Group`; `ovn-sbctl lflow-list ls-ext | grep "ls_in_l2_lkup.*02:00:00:fw"` xem flow forward; `ovs-ofctl -O OpenFlow15 dump-groups br-int` xem group select.

#### Source code tham chiếu

Schema [`ovn-nb.ovsschema` branch-22.03](https://raw.githubusercontent.com/ovn-org/ovn/branch-22.03/ovn-nb.ovsschema) bảng `Forwarding_Group` 6 cột non-root, `child_port` min=1. Compile trong [`northd/northd.c`](https://github.com/ovn-org/ovn/blob/branch-22.03/northd/northd.c) — function anchor `build_fwd_group_lflows` (hoặc `build_forwarding_groups_flows` tuỳ refactor) iterate forwarding groups + sinh flow ở stage `LS_IN_L2_LKUP`. Cross-link với BFD bảng riêng cho liveness.

#### Trường hợp sự cố thực tế

Production case 2023: cluster OSP migrate từ NSX-T sang OVN, có 3 transparent firewall Fortinet active-active trên external segment với cùng VIP `10.0.0.100`/VMAC. NSX-T trước đây dùng "Edge Cluster active-active". Migrate sang OVN dùng Forwarding_Group + BFD. Sau triển khai, observe: 3 firewall đều nhận traffic tương đối đều (~33% mỗi appliance theo hash 5-tuple). 1 firewall bị reboot bảo trì, BFD down trong 600ms, traffic redirect sang 2 còn lại không có connection drop nhờ stateful sync giữa firewall (state replicated).

#### Bài tập synthetic

```bash
sudo ovn-nbctl lsp-add ls-ext lsp-fw-1
sudo ovn-nbctl lsp-add ls-ext lsp-fw-2
sudo ovn-nbctl lsp-set-addresses lsp-fw-1 "02:00:00:fw:01 10.0.0.100"
sudo ovn-nbctl lsp-set-addresses lsp-fw-2 "02:00:00:fw:01 10.0.0.100"
sudo ovn-nbctl create Forwarding_Group name=fg-fw vip="10.0.0.100" vmac='"02:00:00:fw:01"' liveness=false child_port='["lsp-fw-1","lsp-fw-2"]'
FG_UUID=$(sudo ovn-nbctl --bare --columns=_uuid find Forwarding_Group name=fg-fw)
sudo ovn-nbctl add Logical_Switch ls-ext forwarding_groups $FG_UUID
sudo ovn-sbctl lflow-list ls-ext | grep "02:00:00:fw"
```

Quan sát: flow forward `eth.dst == 02:00:00:fw:01` action chọn child_port qua hash.

#### Lỗi thường gặp + tín hiệu chẩn đoán

(a) Child LSP không cấu hình cùng VIP/VMAC → ARP của VIP không reply đúng (nếu `port_security` enable, ARP của LSP với spoofed IP bị drop) — cần `lsp-set-port-security-disable` cho child; (b) `liveness=true` nhưng không có BFD session bind → behavior undefined, child có thể bị skip vĩnh viễn; (c) Nhầm Forwarding_Group với Load_Balancer → FG không DNAT, backend phải tự handle VIP nguyên bản, không thay thế được LB use case; (d) Hash chỉ 5-tuple, traffic ít flow → distribution không đều, cần planning theo expected traffic profile.

#### So sánh với hệ khác

NSX-T `Edge Cluster active-active` ánh xạ trực tiếp; F5 BIG-IP `Active-Active Pair` cùng concept transparent forwarding; AWS Gateway Load Balancer (GWLB) tương tự với GENEVE encap; Cisco ACI `Service Graph` với active-active service node tương đồng. Khác Linux: kernel không có equivalent native (LVS DR mode gần nhất nhưng dùng MAC rewrite, không phải L2 transparent forward). Khác Load_Balancer OVN: LB là DNAT có conntrack rewrite địa chỉ; FG là L2 forward giữ nguyên địa chỉ — semantics khác hẳn.

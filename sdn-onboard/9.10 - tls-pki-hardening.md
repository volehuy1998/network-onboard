## 9.10.1 ptcp vs ssl

Controller và Manager connection hỗ trợ ba transport:

- **`unix:`**: local Unix socket, chỉ root cùng host. Secure by default.
- **`ptcp:PORT`**: passive TCP, không authentication, không encryption. **Chỉ dùng lab.**
- **`ssl:IP:PORT`**: TLS với mutual authentication (mTLS). **Production mandatory.**

Compass Ch K: "Every `Controller` or `Manager` target should be `ssl:IP:PORT`."

`ptcp:` trong production = risk: bất kỳ ai đến được TCP port có thể (a) đọc toàn bộ OVSDB config, (b) inject flow rule arbitrary, (c) chạy lệnh remote qua OpenFlow.

## 9.10.2 ovs-pki: minimal PKI toolkit

OVS ship với `ovs-pki` — script wrap OpenSSL. Compass Ch K workflow:

```bash
[ca-host]# ovs-pki init                            # tạo root CA
[ca-host]# ovs-pki req+sign ovsclient switch      # switch cert
[ca-host]# ovs-pki req+sign ctl controller         # controller cert

[switch-host]# ovs-vsctl set-ssl /etc/openvswitch/key.pem \
                  /etc/openvswitch/cert.pem \
                  /etc/openvswitch/cacert.pem
[switch-host]# ovs-vsctl set-controller br0 ssl:10.0.0.10:6653
```

- `ovs-pki init` tạo `switchca/` và `controllerca/` với cấu trúc X.509 CA tiêu chuẩn.
- `req+sign ovsclient switch` tạo private key + CSR + sign với CA.
- `set-ssl` ghi ba path vào OVSDB SSL row.

---

## 9.10.3 Verify handshake

Từ controller side:

```bash
[controller]$ openssl s_client -connect 10.0.0.10:6653 \
              -CAfile cacert.pem -cert ctl-cert.pem -key ctl-privkey.pem
```

Output nên hiện `Verify return code: 0 (ok)` và TLS version ≥ 1.2.

Error thường: `certificate verify failed` → certificate chain không khớp với CA (thường do cacert.pem thiếu intermediate CA).

---

## 9.10.4 CA rotation

Compass Ch K rule: **"Rotate CAs by adding the new root to `cacert.pem` BEFORE switching the client cert, never the reverse."**

Quy trình:

1. Generate new CA `new-ca.pem`.
2. Append new CA vào `cacert.pem` trên mọi switch và controller.
3. Wait propagation.
4. Issue client cert mới signed bởi new CA.
5. Switch client cert sang new cert — peer vẫn trust vì cả hai CA đều có.
6. Sau ổn định, remove old CA khỏi cacert.pem.

Làm ngược (switch cert trước) → peer không trust → fail.

---

## 9.10.5 TLS version và cipher

OVS 2.17+ hỗ trợ TLS 1.2 và 1.3:

```bash
[host]# ovs-vsctl set SSL <uuid> bootstrap_ca_cert=false \
         ssl_protocols=TLSv1.2,TLSv1.3 \
         ssl_ciphers=HIGH:!aNULL:!MD5
```

Bootstrap mode (`bootstrap_ca_cert=true`) dành cho first-time setup. **Disable ngay sau khi operational** để tránh MITM.

### ▶ Guided Exercise 11: TLS setup cho controller connection

**Outcomes:** Generate CA + certs, cấu hình OVS switch dùng ssl:.

**Before You Begin:** Host có OpenSSL + ovs-pki.

**Instructions:**

**1.** Init PKI:

```bash
[ca-host]# mkdir -p /etc/openvswitch/pki && cd /etc/openvswitch/pki
[ca-host]# ovs-pki init --force
```

**2.** Generate switch cert:

```bash
[ca-host]# ovs-pki req+sign ovsswitch switch
```

**3.** Generate controller cert:

```bash
[ca-host]# ovs-pki req+sign ovsctl controller
```

**4.** Cấu hình OVS switch:

```bash
[switch-host]# cp /etc/openvswitch/pki/switchca/cacert.pem /etc/openvswitch/
[switch-host]# cp ovsswitch-privkey.pem /etc/openvswitch/key.pem
[switch-host]# cp ovsswitch-cert.pem /etc/openvswitch/cert.pem
[switch-host]# ovs-vsctl set-ssl /etc/openvswitch/key.pem \
                                 /etc/openvswitch/cert.pem \
                                 /etc/openvswitch/cacert.pem
[switch-host]# ovs-vsctl set-controller br0 ssl:192.168.1.10:6653
```

**5.** Verify:

```bash
[switch-host]# ovs-vsctl get Controller br0 is_connected
true
```

## 9.10.X Anatomy Template A — `ovs-vsctl list SSL`

**Bối cảnh khi chạy:** Verify TLS config + cert chain đầy đủ trước khi deploy production.

**Command và output** *(reproduced OVS 2.17.9)*:

```shell
$ sudo ovs-vsctl list SSL
_uuid               : 7e3c9f82-...
bootstrap_ca_cert   : false
ca_cert             : "/etc/openvswitch/cacert.pem"
certificate         : "/etc/openvswitch/sc-cert.pem"
external_ids        : {}
private_key         : "/etc/openvswitch/sc-privkey.pem"
```

**9-attribute anatomy:**

| Field | Giá trị mẫu | Ý nghĩa | Red flag nếu |
|-------|-------------|---------|--------------|
| `Open_vSwitch.ssl` ref | `[<uuid>]` | SSL row reference | Empty = OVS không TLS config |
| `ca_cert` path | `/etc/.../cacert.pem` | CA cert file | File missing = SSL reject |
| `certificate` path | `/etc/.../sc-cert.pem` | Switch cert | Self-signed + peer no trust = handshake fail |
| `private_key` path | `/etc/.../sc-privkey.pem` | Key file | Permission 644 = world readable security bug |
| `bootstrap_ca_cert` | `false` | Bootstrap mode | `true` persistent = MITM risk |
| Cert expiry (openssl) | >30 days | Expiration date | <30 days = critical |
| Cert CN | `chassis-1.example.com` | Common Name | Mismatch hostname = verification fail |
| Chain depth | 3 | CA hierarchy | >5 = unusual misconfig |
| Cipher suite | TLS 1.3 AES-256-GCM | Negotiated | TLS <1.2 = deprecated |

**Kịch bản bẻ gãy:**

- **SSL column empty + controller ssl:IP:port**: TLS config missing. Fix: `ovs-vsctl set-ssl /path/privkey /path/cert /path/cacert`.
- **Private key permission 644**: world-readable. Fix: `chmod 600`.
- **Cert CN mismatch hostname**: hostname verification fail. Fix: re-issue cert, sử dụng SAN cho multi-hostname.

## 9.10.Y Hiểu sai + Key Topic

> **Hiểu sai phổ biến 1:** *"Self-signed cert production safe vì chỉ internal."*
>
> **Thực tế:** Self-signed không có CA signature → không distribute trust boundary scalable. Add chassis mới cần manual copy cert. Revocation không work. Use CA (internal qua `ovs-pki` hoặc external Vault).

> **Hiểu sai phổ biến 2:** *"Bootstrap mode chỉ lab."*
>
> **Thực tế:** Bootstrap có use case valid — first-time deployment remote site. Window 1 giờ, sau đó disable. Pattern an toàn hơn manual cert copy nếu network isolation + physical security ensure.

> **Chủ đề trọng tâm:** OVS + OVN TLS là zero-trust foundation. Mỗi chassis có identity cert + verify peer. Cấm `ptcp:` production. Cert rotation strategy: annual minimum, cross-sign overlap 60 days, automation via ansible + ovs-pki, alert <30 days. Phần 20.1 §20.15 cert rotation zero-downtime + Phần 10.6 OVSDB mTLS deep-dive.

## 9.10.Z Common failure modes

**Failure 1 — Nút mạng (Chassis) không kết nối SBDB TLS.** Log: `ssl: failed to verify certificate`. Fix: `openssl verify -CAfile cacert.pem sc-cert.pem`.

**Failure 2 — Cipher negotiation fail.** Log: `ssl: no shared cipher`. Fix: align cipher set với peer.

**Failure 3 — Cert rotation downtime.** Cause: restart sau rotate. Fix: cross-sign + `ovs-appctl vlog/reopen` (no restart).

**Failure 4 — Hostname verification fail.** Log: `ssl: server cert CN mismatch`. Fix: re-issue SAN multi-hostname.

## Điểm cốt lõi cần nhớ

Thứ nhất, production OVS PHẢI dùng `ssl:` cho mọi Controller/Manager target. `ptcp:` chỉ lab ngắn hạn.

Thứ hai, `ovs-pki` wrap OpenSSL cho workflow PKI tối giản — đủ cho setup, không thay thế CA enterprise (Vault, step-ca) cho deployment lớn.

Thứ ba, CA rotation phải theo trình tự: add new CA first, then switch cert. Ngược lại break handshake.

Thứ tư, bootstrap mode (`bootstrap_ca_cert=true`) chỉ cho first-time — disable ngay sau operational để tránh MITM.

Thứ năm, Anatomy Template A `list SSL` 9-attribute. Private key 644 = security bug #1.

Thứ sáu, self-signed = technical debt. Always use CA.

Thứ bảy, cert rotation zero-downtime qua cross-sign + reload (no restart). Xem Phần 20.1 §20.15 + Phần 10.6.

## 9.5.1 Tại sao hardware offload cần thiết ở 25G/100G+

Software OVS đạt throughput tốt ở 10G. Nhưng với NIC 25G/100G/200G trở lên, CPU trở thành bottleneck:

- Ở 100 Gbps với packet 64 bytes, packet rate ~148 Mpps. Mỗi packet tốn ~6-10 CPU cycle kể cả với DPDK optimized.
- Với Intel Xeon ~3 GHz, một core xử lý tối đa ~500 Mpps theoretical, thực tế ~15-30 Mpps. Cần 5-10 core chỉ để match line rate 100G — không khả thi ở scale cloud.

Giải pháp: **offload data plane xuống NIC hardware**, giữ control plane trên CPU. Smart NIC (ConnectX-5+) và DPU (BlueField-2/3) có embedded switch (eSwitch) với hardware forwarding capability — OpenFlow rules được dịch thành rule trong eSwitch, hardware xử lý packet matching + action mà không cần CPU touch packet.

WASTC 2021 slides (`doc/ovs/Day 4- Motivation and Introduction to Open vSwitch`) nêu bối cảnh: "closed-design mô hình has limited the capability of the switches... The process of updating devices is lengthy, costly." SDN với OpenFlow là giải pháp lập trình, nhưng phải dùng được hardware đã có → hardware offload là cầu nối giữa SDN flexibility và hardware performance.

## 9.5.2 Linux switchdev framework: VF representor

Linux kernel switchdev là framework quản lý NIC có embedded switch. Khái niệm cốt lõi: **VF representor** — netdev kernel đại diện cho một Virtual Function trong eSwitch của NIC.

Khi NIC chạy **legacy SR-IOV mode**: mỗi VF là netdev độc lập, VF được passthrough vào VM. Không có representor.

Khi NIC chạy **switchdev mode**: mỗi VF vẫn được passthrough vào VM (VF netdev thật), nhưng **thêm** representor netdev (`pf0vf0`) trên host. OVS (hoặc tc flower) cấu hình rule trên representor → rule được push xuống eSwitch hardware.

Chuyển sang switchdev mode (compass Ch 16):

```bash
[host]# echo 4 > /sys/class/net/ens1f0/device/sriov_numvfs
[host]# for vf in $(ls -d /sys/class/net/ens1f0/device/virtfn*); do
          pci=$(basename $(readlink $vf))
          echo $pci > /sys/bus/pci/drivers/mlx5_core/unbind 2>/dev/null
        done
[host]# devlink dev eswitch set pci/0000:3b:00.0 mode switchdev
```

Verify:

```bash
[host]# ip -d link show pf0vf0_r
24: pf0vf0_r: <BROADCAST,MULTICAST> mtu 1500 ...
    link/ether ... switchid 00008000 portname pf0vf0
```

`switchid` và `portname pf0vf0` là dấu hiệu interface là switchdev representor.

## 9.5.3 NVIDIA ASAP² và eSwitch offload

**ASAP²** (Accelerated Switching and Packet Processing) là tên thương mại NVIDIA cho công nghệ eSwitch offload trên ConnectX-5 trở lên và BlueField series. Core idea: eSwitch hardware bên trong NIC xử lý VXLAN encap/decap, L2/L3 forwarding, ACL, conntrack — ở line rate.

Kiến trúc:

```
┌────────────────────────────────────────────────────┐
│   CPU                                               │
│   ┌──────────────┐                                 │
│   │ ovs-vswitchd │  (control plane, không đổi)    │
│   └──────┬───────┘                                 │
│          │ tc flower / DOCA Flow                   │
│          ▼                                          │
│   ┌──────────────┐                                 │
│   │ mlx5_core    │                                 │
│   └──────┬───────┘                                 │
└──────────┼──────────────────────────────────────────┘
           │ (driver queue)
┌──────────┴─────────────────────────────────────────┐
│  NIC subsystem (ConnectX-6 / BlueField)            │
│  ┌──────────────────────────────────────┐          │
│  │  eSwitch (hardware packet forward)   │          │
│  │   ─ match: VXLAN VNI, inner 5-tuple │          │
│  │   ─ action: encap/decap, forward    │          │
│  └──────────────────────────────────────┘          │
└────────────────────────────────────────────────────┘
```

Kích hoạt offload:

```bash
[host]# ovs-vsctl set Open_vSwitch . other_config:hw-offload=true
[host]# systemctl restart openvswitch-switch
```

Verify:

```bash
[host]# ovs-appctl dpctl/dump-flows -m type=offloaded
[host]# tc -s filter show dev pf0vf0_r ingress
```

Output hiện rule đã offload xuống hardware với counter — bằng chứng hardware đang forward, không phải CPU.

## 9.5.4 Ba DPIF của OVS: Kernel, DPDK, DOCA

*Table 9-5: So sánh ba DPIF OVS*

| DPIF | Cơ chế | Mô-đun | Khi dùng |
|---|---|---|---|
| **Kernel** | openvswitch.ko + tc-flower cho hw offload | in-tree kernel module | Workload thông thường, mọi NIC Linux |
| **DPDK** | userspace PMD + `datapath_type=netdev` | OVS-DPDK package | Throughput cực cao không cần NIC offload |
| **DOCA** | NVIDIA DOCA Flow library + eSwitch offload | `ovs-doca` package | NIC NVIDIA ConnectX-6+ / BlueField-2+ |

NVIDIA khuyến nghị chính thức: **"DPDK và Kernel DPIFs được duy trì chủ yếu cho backward compatibility, không được cập nhật feature mới kể từ 2023."** Mọi tính năng mới (extended match fields, CT+NAT offload, VXLAN encap variants) chỉ được thêm vào DOCA DPIF.

Deployment mới trên NIC NVIDIA nên dùng OVS-DOCA ngay từ đầu.

## 9.5.5 OVS-DOCA: DOCA Flow library

DOCA Flow library là API do NVIDIA cung cấp để lập trình eSwitch directly. OVS-DOCA package wrap thư viện này thành DPIF mới.

Kích hoạt:

```bash
[host]# ovs-vsctl set Open_vSwitch . other_config:doca-init=true
[host]# ovs-vsctl set Open_vSwitch . other_config:hw-offload=true
[host]# systemctl restart openvswitch-switch
```

DOCA Flow pre-allocate offload containers + entries khi start. Congestion threshold mặc định 80% — khi 80% full, DOCA resize container.

Monitor:

```bash
[host]# ovs-appctl coverage/show | grep doca
doca_async_queue_full       ...
doca_pipe_resize            ...
doca_pipe_resize_over_10_ms ...
```

- `doca_async_queue_full` > 0 → hardware nhận rule chậm.
- `doca_pipe_resize_over_10_ms` > 0 → resize tốn > 10ms, có thể drop packet.

Limitations DOCA DPIF hiện tại: không support CT-CT-NAT phức tạp, không support OpenFlow meters đầy đủ (chỉ srTCM basic).

## 9.5.6 Offload feature coverage

Feature được offload thành công:

- **VXLAN encap/decap**: pattern br-phy + br-ovs — tunnel VXLAN encap ở eSwitch.
- **Conntrack**: tracking connections trong hardware. Table ~2M connections.
- **CT+NAT**: SNAT/DNAT với conntrack. Exception: ICMP NAT không support.
- **VF-LAG**: bonding 2 port NIC active-backup, XOR, LACP — offload.
- **OpenFlow meters**: RFC 2697 srTCM basic.
- **sFlow**: sampling offload (chỉ internal bridge khi dùng VXLAN).
- **Mirror**: `doca-mirror-max=2048`.

KHÔNG offload (fallback software):

- OpenFlow action phức tạp (learn, conjunction, multi-path hash).
- Conntrack timeout policy custom.
- OXM experimenter fields.

Compass: "the megaflow cache in software still catches them; the hardware just doesn't match."

## 9.5.7 Steering modes: SMFS vs DMFS

- **SMFS** (Software-Managed Flow Steering) — default trên ConnectX-6+. Tối ưu insertion rate cao, khuyến nghị khi rule thay đổi thường xuyên.
- **DMFS** (Device-Managed Flow Steering) — firmware-based. Tối ưu throughput khi rule cố định, nhưng insertion rate thấp hơn.

Cấu hình (PHẢI set trước khi chuyển switchdev):

```bash
[host]# devlink dev param set pci/0000:3b:00.0 name flow_steering_mode \
         value smfs cmode runtime
```

**vPort match modes**:

- **Metadata** (default) — support SR-IOV live migration + dual-port RoCE.
- **Legacy** — higher performance ~5-10%, không support live migration.

Production: VM có live migration → Metadata. Static workload, throughput critical → Legacy + DMFS.

## 9.5.8 vDPA: virtio portability + SR-IOV performance

vDPA kết hợp SR-IOV performance với virtio portability — VM image không bị tied to vendor driver.

- **Software vDPA**: translation on host.
- **Hardware vDPA**: NIC hỗ trợ trực tiếp (ConnectX-6 Dx trở lên).

Use case: cloud provider cần VM image generic, migrate giữa server khác hardware nhưng vẫn đạt SR-IOV throughput.

---

## 9.5.9 BlueField DPU architecture

BlueField (2/3) là DPU — NIC với Arm cores embedded.

- Host (x86) — workload application.
- DPU (Arm) — infrastructure services (OVS-DOCA, firewall, storage offload).
- eSwitch — hardware forwarding giữa host PF và DPU VF.

`pf0hpf` là interface host PF representor trên DPU side. BFB (BlueField Bundle) installation: NVIDIA cung cấp image BFB chứa Ubuntu + DOCA SDK + OVS-DOCA. Script `/sbin/mlnx_bf_configure` auto-enable hw-offload.

Use case: isolation tenant workload (host) và infrastructure services (DPU Arm cores).

## 9.5.10 Megaflow scaling trong OVS-DOCA

Tuning large-scale:

```bash
[host]# ovs-vsctl set Open_vSwitch . other_config:flow-limit=500000
[host]# ovs-vsctl set Open_vSwitch . other_config:hw-offload-ct-size=1000000
[host]# ovs-vsctl set Open_vSwitch . other_config:max-revalidator=250
[host]# sysctl -w net.netfilter.nf_conntrack_max=2097152
```

- `flow-limit` default 200k, tối đa 2M.
- `hw-offload-ct-size` default 250k, tối đa 2M connections (warn nếu > 1M).
- `max-revalidator` default 250ms.
- `nf_conntrack_max` phải đồng bộ với hw-offload-ct-size.

**Outcomes:** Convert NIC sang switchdev mode, verify VF representor tạo ra.

**Before You Begin:** NIC Mellanox ConnectX-4 Lx trở lên với mlx5_core.

**Instructions:**

**2.** Unbind VF drivers + chuyển switchdev:

```bash
[host]# for vf in $(ls -d /sys/class/net/ens1f0/device/virtfn*); do
          pci=$(basename $(readlink $vf))
          echo $pci > /sys/bus/pci/drivers/mlx5_core/unbind 2>/dev/null

```

**3.** Verify representor:

```bash
[host]# ip -d link show | grep pf0vf
```

Output phải hiện `pf0vf0_r`, `pf0vf1_r`, ... với `switchid` + `portname`.

**4.** Rebind driver:

```bash
[host]# for vf in $(ls -d /sys/class/net/ens1f0/device/virtfn*); do
          pci=$(basename $(readlink $vf))
          echo $pci > /sys/bus/pci/drivers/mlx5_core/bind
        done
```

**Finish:** VF sẵn sàng passthrough vào VM hoặc plug vào br-doca.

## Điểm cốt lõi cần nhớ

Thứ nhất, hardware offload cần thiết khi NIC tốc độ 25G+ và CPU cost per packet trở thành bottleneck. Smart NIC (ConnectX-5+) và DPU (BlueField) có eSwitch embedded.

Thứ hai, Linux switchdev là API kernel chuẩn cho hardware offload: VF representor netdev là mặt trước của eSwitch port. OVS cấu hình rule trên representor → push xuống hardware qua tc-flower hoặc DOCA Flow.

Thứ ba, ba DPIF của OVS phục vụ use case khác nhau. NVIDIA khuyến nghị DOCA DPIF cho NIC của họ — Kernel và DPDK DPIF giữ cho backward compatibility, không thêm feature mới.

Thứ tư, SMFS vs DMFS và Metadata vs Legacy vPort là quyết định tuning quan trọng. VM có live migration → Metadata. Workload cố định, throughput critical → Legacy + DMFS.

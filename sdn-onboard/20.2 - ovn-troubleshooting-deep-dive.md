## 20.2.1 The three OVN debug layers, why `tcpdump` is not the first tool

When an engineer is paged at two in the morning with "production VM cannot ping the gateway", the reflex carried over from CCNA is to open `tcpdump` on the NIC. In an OVN system this reflex leads in the wrong direction about seventy percent of the time. The reason: OVN is a three-tier distributed system, and a packet can be dropped at three different layers, while `tcpdump` only sees the third one (the wire). The first two layers (the NB intent and the SB Logical_Flow) never appear on the wire because they are abstractions.

Part 20.0 ôÏ20.0.2 introduced the five-layer inspection model (control plane health, logical topology, OpenFlow pipeline, datapath, tunnel and wire). Part 20.2 goes deep into the three middle layers that are specific to OVN:

**Layer A, NB intent (Northbound Database).** The CMS (OpenStack Neutron, OVN-Kubernetes, or a bare-metal operator) writes Logical_Switch, Logical_Router, Logical_Switch_Port, ACL, Load_Balancer, NAT, and DHCP_Options into NBDB. This is the high-level intent and it does not contain OpenFlow. Common Layer A bugs: bad ACL syntax, a Port_Group that references a non-existent object, a Load_Balancer with a VIP that collides with another address, or a DHCP_Options entry that is not attached to an LSP.

**Layer B, SB Logical_Flow (Southbound Database).** `ovn-northd` compiles the NB intent down into Logical_Flow rows in SBDB (see Part 13.8 ôÏ13.8.2 for the compile algorithm). Each NB object produces many Logical_Flow entries. Common Layer B bugs: `ovn-northd` is paused, the NB and SB are out of sync, compile takes too long (the `nb_cfg` timestamp gap exceeds five seconds), or the lflow-cache on ovn-controller is stale.

**Layer C, OpenFlow on `br-int` (dataplane).** `ovn-controller` on each chassis translates the subset of Logical_Flow relevant to the local chassis into OpenFlow rules and installs them into `br-int` via the OpenFlow protocol (see Part 13.7 ôÏ13.7.4). Common Layer C bugs: Port_Binding has not claimed a chassis, the I-P engine is stuck in full recompute, the OpenFlow table count exceeds the hardware offload threshold, or there is a conntrack zone collision between an LS and an LR.

> **Key topic, simulate before capture.** Given the three layers above, the correct tool order is `ovn-trace` first, then `ofproto/trace` plus `ovn-detrace`, and only then `tcpdump`. `ovn-trace` simulates a packet through Layer A and B (no real traffic, it reads the NB and SB snapshot). `ofproto/trace` simulates Layer C (it reads the OpenFlow flow table). Only when both simulations report OK while traffic still fails do you drop down to `tcpdump` to check the tunnel and the wire. Simulation runs in milliseconds and does not depend on timing; `tcpdump` can take minutes to catch a rare packet.

The second principle is **"the three layers must agree"**. When the NB intent is correct (`ovn-nbctl show`) but the SB Logical_Flow is missing entries (`ovn-sbctl lflow-list`), the cause sits in ovn-northd. When the SB is complete but OpenFlow is missing entries (`ovs-ofctl dump-flows br-int`), the cause sits in ovn-controller. Trace backward from the missing layer upward to identify which daemon has not completed its job.

The rest of this part provides detailed tooling for each layer: `ovn-trace` (Layer A and B, ôÏ20.2.2), the `ovn-detrace` chain with `ofproto/trace` (the Layer B and C junction, ôÏ20.2.3), Port_Binding forensic (the Layer B and C junction from a port-centric angle, ôÏ20.2.4), ovn-controller and northd runtime commands (ôÏ20.2.5 and ôÏ20.2.6), and stateful table triage (ôÏ20.2.7).

## 20.2.2 `ovn-trace`, simulating a packet through the Logical pipeline

`ovn-trace` is the most important tool for debugging Layer A and B. It reads an SBDB snapshot at the time it runs and simulates a synthetic packet through the Logical_Flow pipeline of a specific datapath. The output shows every logical table the packet visits, which flow matches, which action fires, and the final result (deliver, drop, or reinject).

### 20.2.2.1 Grammar and runtime environment

Full syntax from the `ovn-trace(8)` man page:

```
ovn-trace [options] [datapath] microflow
ovn-trace [options] --detach
```

`datapath` is the name of a logical switch or logical router (string or UUID). If omitted, `ovn-trace` infers it from the `inport` field in the microflow: if `inport` belongs to an LSP, the datapath is the LS that contains the LSP; if `inport` belongs to an LRP, the datapath is the LR. The best practice is still to specify it explicitly to avoid ambiguity.

`microflow` is an expression written in the OVN logical match language (the same syntax as the ACL expression in NBDB) that fully describes the packet to be simulated. Prerequisite fields are added automatically (for example, `ip4` is implied when matching `ip4.src`).

Runtime requirements: SBDB must be reachable from the host that runs `ovn-trace`, usually through the environment variable `OVN_SB_DB` or the `--db` option. The simplest configuration:

```shell
$ export OVN_SB_DB=tcp:10.0.0.100:6642
$ ovn-trace ls1 'inport == "vm1" && eth.src == aa:bb:cc:dd:ee:01 && eth.dst == aa:bb:cc:dd:ee:02 && ip4 && ip4.src == 10.0.0.1 && ip4.dst == 10.0.0.2 && ip.ttl == 64 && icmp4'
```

### 20.2.2.2 Microflow syntax, five example classes

Microflows follow the BNF of the OVN logical expression (defined in `ovn-sb(5)` ôÏLogical_Flow). Prerequisites are automatic (for example, matching `ip4.src` implies `ip4`, and matching `tcp.dst` implies `tcp` which implies `ip`). The logical operators are `==`, `!=`, `<`, `<=`, `>`, `>=`, `&&`, `||`, and `!`.

**Class 1, L2 unicast:**

```
inport == "vm1" &&
eth.src == aa:bb:cc:dd:ee:01 &&
eth.dst == aa:bb:cc:dd:ee:02
```

Pure L2 case, the packet has no IP layer. Use it to verify that `ls_in_l2_lkup` works correctly with a static FDB entry or a MAC_Binding entry.

**Class 2, L2 broadcast or multicast:**

```
inport == "vm1" &&
eth.src == aa:bb:cc:dd:ee:01 &&
eth.dst == ff:ff:ff:ff:ff:ff
```

The destination MAC is broadcast. `ovn-trace` routes through the Multicast_Group (MC_FLOOD) and outputs to every port in the LS except the inport. Use it to verify that an ARP request reaches every VM.

**Class 3, L3 unicast (cross-subnet via an LR):**

```
inport == "vm1" &&
eth.src == aa:bb:cc:dd:ee:01 &&
eth.dst == 00:00:00:ff:fe:01 &&  # MAC of the LRP (default gateway)
ip4 &&
ip4.src == 10.0.0.1 &&
ip4.dst == 10.0.1.5 &&           # different subnet
ip.ttl == 64 &&
icmp4
```

Cross-subnet ICMP packet. `ovn-trace` routes through LS ingress, then the LR ingress pipeline (the `lr_in_ip_routing` stage), then LR egress, then the destination LS ingress, then the destination LSP. Use it to verify the route table and ARP resolution for the next hop.

**Class 4, ARP request:**

```
inport == "vm1" &&
eth.src == aa:bb:cc:dd:ee:01 &&
eth.dst == ff:ff:ff:ff:ff:ff &&
arp &&
arp.op == 1 &&                   # request
arp.sha == aa:bb:cc:dd:ee:01 &&
arp.spa == 10.0.0.1 &&
arp.tha == 00:00:00:00:00:00 &&
arp.tpa == 10.0.0.254            # gateway IP
```

ARP request to find the MAC of the gateway. In OVN, the `ls_in_arp_rsp` stage (table 19 in the LS ingress pipeline) can auto-reply if it knows the MAC of the destination IP (the ARP Suppressor, see Part 18). The `ovn-trace` output shows the action `build_arp_resp(...)` followed by `outport = inport; output;` (reflecting the reply back to the source).

**Class 5, DHCP Discover:**

```
inport == "vm1" &&
eth.src == aa:bb:cc:dd:ee:01 &&
eth.dst == ff:ff:ff:ff:ff:ff &&
ip4 &&
ip4.src == 0.0.0.0 &&
ip4.dst == 255.255.255.255 &&
ip.ttl == 64 &&
udp &&
udp.src == 68 &&
udp.dst == 67
```

A broadcast DHCP Discover packet. The `ls_in_dhcp_options` stage (table 20) fires `put_dhcp_opts(...)` for an inline reply if the LSP has a `dhcpv4_options` entry attached. Use it to verify the native DHCP configuration (Part 13.10).

### 20.2.2.3 Option catalog, eleven options

The table below summarises the eleven options from `ovn-trace(8)`, with semantics and use cases:

| Option | Description | When to use |
|--------|-------------|-------------|
| `--detailed` | (Default) Shows every table visited with priority, match, and action; omits indentation tail recursion; filters trivial `match=1` flows | The default for ordinary debugging. Enough context to identify which flow fires |
| `--summary` | A programming-language-like format with indentation that shows the pipeline structure as pseudo-code | When you need a holistic view of which pipeline branch goes where; easy to read flow control |
| `--minimal` | Shows only actions that modify packet data (set_field, ct, dec_ttl) and the output action, with operands reduced to constants | Regression tests and CI pipelines; the most stable output with little version-skew noise |
| `--all` | Shows all three (`--detailed`, `--summary`, `--minimal`) stacked | First-time debugging of an unfamiliar pipeline when you need every angle |
| `--ovs[=remote]` | Also prints the corresponding OpenFlow flow on `br-int`. Optional `remote` specifies the OVS connection (default is the Unix socket) | Bridge from logical to physical, see the actual OpenFlow rule generated from a logical flow |
| `--ct=flags` | Sets connection tracking state flags (new, est, rel, rpl, inv, trk, dnat, snat) so the packet is simulated in a specific state instead of the default `+trk+new` | Debug stateful ACLs, the LB reply path, or NAT reverse flow. For example, `--ct=trk,est,rpl` simulates the reply of an established connection |
| `--lb-dst=ip[:port]` | Forces the Load_Balancer to pick this backend instead of hashing | Debug LB affinity, isolate whether the LB logic routes correctly to a specific backend |
| `--select-id=id` | Forces a group bucket to pick this bucket id | Debug ECMP or multipath groups, isolate a specific path |
| `--friendly-names` | (Default) Substitutes the LSP or LRP name instead of the UUID | Default, easier to read |
| `--no-friendly-names` | Shows the raw UUID | When you need to copy and paste the output into a script that greps the exact UUID |
| `--db database` | OVSDB remote instead of the `OVN_SB_DB` environment variable | Debug a different OVN cluster, or run from a jumphost without environment variables |

In addition, `-v[spec]` sets the logging level (`-v dbg` is very verbose), and `--detach` runs as a daemon (rarely used).

**Anatomy by block:**

| Block | Content | Meaning |
|-------|---------|---------|
| Line 1 starting with `# reg14=...` | Parsed microflow with prerequisites auto-added (reg14 is the inport index, nw_proto=1 for ICMP) | Confirms that `ovn-trace` parsed the input correctly. If parsing fails, the output stops here with an error message |
| `ingress(dp="ls_a", inport="vm_a")` | Header block that starts each pipeline, format `ingress(dp=<datapath>, inport=<port>)` or `egress(...)` | Each time a packet passes through a patch port or a tunnel into a different datapath, a new block header appears |
| ` 0. ls_in_check_port_sec (northd.c:8127): 1, priority 50, uuid abc12345` | Table number plus stage name plus `(source:line reference)` plus match expression plus priority plus the Logical_Flow UUID | The engineer uses this UUID with `ovn-sbctl get Logical_Flow <uuid>` for further inspection |
| ` reg0[15] = check_in_port_sec();` | Actions applied, one per line, indented | Action list. `next;` advances to the next logical table. `output;` delivers the packet or transitions from ingress to egress |
| ` /* output to "vm_b", type "" */` | Comment at the end of an egress delivery block that shows the destination LSP and the Port_Binding type | Type `""` is a VIF (normal VM port). If the type is `"patch"`, the packet continues into another LR or LS |

**Troubleshooting cues:**

- **If the output ends with `drop;` at the `ls_in_acl` stage**, an ACL policy is blocking. Verify with `ovn-nbctl --bare --columns=direction,priority,match,action find ACL`. Fix: adjust the ACL or add an `allow` rule with a higher priority.
- **If the output ends at `lr_in_arp_resolve` without `eth.dst = ...`**, ARP has not resolved the next hop and MAC_Binding is missing the entry. Verify with `ovn-sbctl find MAC_Binding ip=<next-hop>`. Fix: wait for ARP to resolve (trigger it with `ovn-appctl -t ovn-controller inject-pkt` for an ARP request) or set static `mac_addresses` on the LRP.
- **If you do not see the block `ingress(dp="<next-dp>")` after a patch-port output**, Logical_Flow entries are missing for the next LR or LS. The cause is usually that `ovn-northd` has not finished compiling (see ôÏ20.2.6 `inc-engine/show-stats`) or the NB intent is wrong (the LRP is not attached to the LS). Verify with `ovn-nbctl show`.
- **If the line `# reg14=...` reports a parse error such as "unknown field 'ipv4'"**, the microflow syntax is wrong. OVN uses `ip4`, not `ipv4`, and `eth.dst`, not `eth.dst_mac`. Look up the exact field name in `ovn-sb(5)` ôÏLogical_Flow.
- **If the trace output is very long (more than 200 lines) with many `resubmit` loops**, the pipeline has recursion (often through hairpin NAT). Switch to `--summary` to see the overall structure before going deep with `--detailed`.

Summary drops the stage name, priority, and UUID, keeping only the action sequence. The indentation shows the nesting between pipelines. Senior engineers read summary faster than detailed to identify pipeline structure; switch to detailed when you need to dig into a specific stage.

### 20.2.2.6 Reading the `ovn-trace --minimal` output

Minimal keeps only the actions that modify packet data and the output action. Operands are reduced to constants (for example, `eth.src = 00:00:00:ff:fe:02` instead of `eth.src = mac_of_lrp`):

```shell
$ ovn-trace --minimal ls_a \
    'inport == "vm_a" && eth.src == aa:bb:cc:00:00:01 && eth.dst == 00:00:00:ff:fe:01 &&
     ip4.src == 10.0.0.1 && ip4.dst == 10.0.1.5 && ip.ttl == 64 && icmp4'

# reg14=0x1,dl_src=aa:bb:cc:00:00:01,dl_dst=00:00:00:ff:fe:01,
# nw_src=10.0.0.1,nw_dst=10.0.1.5,nw_proto=1,nw_ttl=64

eth.src = 00:00:00:ff:fe:02;
eth.dst = aa:bb:cc:00:00:05;
ip.ttl--;
output;
/* output to "vm_b", type "" */
```

This is the format used in regression tests: stable output that depends little on the OVN version (different versions may reorder stages, but the final packet transform is the same). A CI pipeline can diff the `--minimal` output before and after an upgrade to catch silent regressions.

### 20.2.2.7 Reading the `ovn-trace --ovs` output

When you want to bridge from logical to physical, `--ovs` includes the actual OpenFlow flow that `ovn-controller` installed on `br-int`. The output extends each stage with its corresponding OpenFlow rule:

```shell
$ ovn-trace --ovs ls_a \
    'inport == "vm_a" && eth.src == aa:bb:cc:00:00:01 && eth.dst == 00:00:00:ff:fe:01 &&

```

`--ovs` requires that `ovn-trace` can connect to the local OVS DB (default `unix:/var/run/openvswitch/db.sock`), which usually means it only runs on a chassis that has both OVS and `ovn-controller`. It is the fastest way to verify that `ovn-controller` translates a Logical_Flow into the expected OpenFlow rule, symmetric to ôÏ20.2.3 where `ovn-detrace` works in the reverse direction.

### 20.2.2.8 The `--ct` option for stateful simulation

By default, `ovn-trace` sets `ct_state = +trk+new` (the packet is a NEW connection that has not been committed to conntrack). When debugging the reply path of an ESTABLISHED connection or checking NAT reverse, you must set `--ct` manually:

```shell
# Simulate the reply from 10.0.1.5 back to 10.0.0.1 (after the connection has been established)
$ ovn-trace --ct=trk,est,rpl ls_b \
    'inport == "vm_b" && eth.src == aa:bb:cc:00:00:05 && eth.dst == 00:00:00:ff:fe:02 &&
     ip4.src == 10.0.1.5 && ip4.dst == 10.0.0.1 && ip.ttl == 64 && icmp4'
```

When `ct_state` contains `+est+rpl`, an `allow-related` ACL matches the reply and permits it. Without this flag, the reply falls through to the NEW rules and is usually dropped by the default deny. This is why the bug "the reply does not come back" is often reproduced as: tracing the NEW direction looks fine, and only tracing with `+est+rpl` reveals that the ACL is missing `allow-related`.

Similarly, `--ct=trk,new,inv` (invalid) is used to verify that an ACL rejects invalid state, and `--ct=trk,dnat` verifies the LB DNAT reverse path.

### 20.2.2.9 The `--lb-dst` option for load balancer debugging

An LB VIP has multiple backends, and OVN picks one by hashing the 5-tuple. When debugging a specific backend, it is often hard to reproduce because the microflow hash does not match the backend you want to test. `--lb-dst` forces a direct DNAT to a specific backend IP:

```shell
$ ovn-trace --lb-dst=10.0.2.50:80 ls_a \
    'inport == "vm_a" && eth.src == aa:bb:cc:00:00:01 && eth.dst == 00:00:00:ff:fe:01 &&
     ip4.src == 10.0.0.1 && ip4.dst == 10.100.0.1 && ip.ttl == 64 && tcp.dst == 80'
```

The output forces the `ls_in_lb` stage to DNAT to `10.0.2.50:80` instead of hashing. This is useful when backend `10.0.2.50` is reporting intermittent errors and you need to verify that the entire pipeline from client to this specific backend works correctly.

## 20.2.3 `ovn-detrace`, from an OpenFlow cookie back to the source Logical_Flow

`ovn-trace` simulates in the direction **NB to OpenFlow** (logical down to physical). `ovn-detrace` works in the reverse direction: it reads the OpenFlow cookie in `ofproto/trace` output (or `dump-flows`) and looks up the Logical_Flow in SBDB that produced the cookie, and when applicable, the originating NBDB object (ACL, Load_Balancer, or NAT).

### 20.2.3.1 Role in the workflow

Suppose `ofproto/trace` reports that a packet is dropped at an OpenFlow flow with cookie `0x5d2a7f91`. Question: which Logical_Flow produced this rule? Which NB object (ACL, LB, or NAT) triggered it? Without `ovn-detrace` the engineer must run `ovn-sbctl find Logical_Flow ...` by hand with many guesses. With `ovn-detrace`, the chain is automatic:

```
ovs-appctl ofproto/trace -m 3 br-int <microflow>
    | (output has a cookie field)
    v
ovn-detrace
    | (looks up SB and NB)
    v
annotated output: cookie 0x5d2a7f91 -> Logical_Flow ls_in_acl priority 2000
                                    -> NBDB ACL name="deny-ssh" in Port_Group "mgmt"
```

### 20.2.3.2 Grammar and environment variables

`ovn-detrace` reads stdin and writes to stdout:

```
ovn-detrace [options] < file

Common options:
    [-h | --help] [-V | --version]
```

There is no `[datapath] [microflow]` argument because it is a pure filter: input from stdin (usually piped from `ofproto/trace`), annotated output.

Important options:

| Option | Description | When to use |
|--------|-------------|-------------|
| `--ovnsb=server` | SBDB remote, default is the environment variable `OVN_SB_DB` | When running from a node other than the central node, specify SBDB explicitly |
| `--ovnnb=server` | NBDB remote, default is the environment variable `OVN_NB_DB` | NBDB is needed to annotate the ACL name, LB name, and NAT rule |
| `--ovs=` | Also decode the OVS ofport in the flow | When the chain contains a flow that references an OVS port that needs to be mapped to an interface name |
| `--ovsdb=server` | OVS DB remote if `--ovs` is present | Similar, used explicitly when the default socket is not in use |
| `--no-leader-only` | Connect to any Raft member, not only the leader | The leader is busy, use a follower to reduce load |
| `-p privkey.pem` `-c cert.pem` `-C cacert.pem` | TLS certificates for the SBDB and NBDB connections | A cluster with TLS enabled (recommended in production) |

Without `--ovnnb=...` or the environment variable `OVN_NB_DB`, `ovn-detrace` only annotates at the SB level (Logical_Flow) and does not extend down to the NB object. Without `--ovs`, it cannot decode the ofport.

### 20.2.3.3 Reading the output of the chain `ofproto/trace -m 3 | ovn-detrace`

**Context when running.** An engineer receives the alert "connection timeout from client 10.0.0.1 to VIP 10.100.0.1:443". The simulation with `ovn-trace` is OK (the LB DNAT picks the right backend). The next step is to verify the physical layer on the chassis: is the OpenFlow rule correct, and is any flow dropping? The chain `ofproto/trace` plus `ovn-detrace` answers this in a single command.

**Command and output** *(reproduced from the OVN tutorial sandbox, OVN 22.03.8)*:

**Anatomy of the three output blocks:**

| Block | Content | Meaning |
|-------|---------|---------|
| `Flow: tcp,in_port=5,...` | Parsed input microflow in canonical format | The `ofproto/trace` baseline, not related to `ovn-detrace` |
| `bridge("br-int")` plus stage header | Per-table match in OpenFlow with cookie, action, and resubmit target | Native `ofproto/trace` output |
| `# Logical flow: table=N (stage_name), priority=P, match=..., action=...` | `ovn-detrace` annotation after each flow whose cookie matches SBDB | A `#` comment line added below each flow, mapping the cookie to its Logical_Flow |
| `# Load_Balancer: name="web-lb" vips={...}` | `ovn-detrace` annotation at the NB level when the source Logical_Flow is from an LB, ACL, or NAT | The object name in NBDB, helping the engineer know what to change with `ovn-nbctl` |
| `# Logical input ports: "vm_a" (reg14=0x1)` | `ovn-detrace` decodes the `reg14` value into an LSP name | `reg14` is the register OVN uses for the inport. Decoding makes the output readable |
| `final: Datapath actions: ...` | The final action that the OVS datapath will execute, from `ofproto/trace` | `set(tunnel(...))` indicates the packet is encapsulated in Geneve toward the destination chassis |

**Troubleshooting cues:**

- **If the `ovn-detrace` output does not have a `# Logical flow: ...` line after a flow**, the cookie in OpenFlow does not match any Logical_Flow in SBDB. Causes: SBDB is stale (ovn-controller has not updated after ovn-northd compiled a new version) or the chassis is running a different OVN version from SBDB. Fix: `ovs-appctl -t ovn-controller recompute` or verify version match.
- **If the annotation has only `# Logical flow: ...` without `# Load_Balancer: ...` or `# ACL: ...`**, the NBDB connection has not been configured. Set the environment variable `OVN_NB_DB` or pass `--ovnnb=...` explicitly.
- **If the output shows a flow with `priority 0` that drops**, no higher-priority rule matched and the packet fell through to the default drop. Verify with `ovn-sbctl lflow-list <datapath> | grep -v priority=0` to see whether any expected higher-priority flow is missing.
- **If the decoded reg14 shows a UUID instead of an LSP name**, `ovn-detrace` could not resolve the name, usually because the LSP was renamed or deleted between OpenFlow installation and the trace. Force a reconnect to SBDB.
- **If `ofproto/trace -m 3` reports "Too large" with truncated output**, the table count on `br-int` is too high. Reduce to `-m 2`, or grep the output for a specific cookie and pipe it through `ovn-detrace` separately.

### 20.2.3.4 Pattern: diff chassis with `ovn-detrace`

When a cluster has more than 500 chassis and a specific chassis (chassis-42) reports packet drops, the debug pattern is: diff the OpenFlow flows between chassis-42 and a healthy peer, then pipe through `ovn-detrace` to identify which Logical_Flow is missing.

```shell
# On chassis-42 (suspect)
$ ssh chassis-42 'ovs-ofctl dump-flows br-int table=8,metadata=0x1' > /tmp/ch42_t8.txt

# On chassis-peer (healthy)
$ ssh chassis-peer 'ovs-ofctl dump-flows br-int table=8,metadata=0x1' > /tmp/chp_t8.txt

# Diff, focus on cookie (skip counter bytes and duration)
$ diff <(awk '{print $NF}' /tmp/chp_t8.txt | sort) \
       <(awk '{print $NF}' /tmp/ch42_t8.txt | sort)
> actions=resubmit(,9),cookie=0x7ab19c00  # present on healthy, missing on ch42

# Trace the cookie back to a Logical_Flow
$ echo 'cookie=0x7ab19c00, table=8, priority=110, metadata=0x1' | ovn-detrace
# Logical flow: table=14 (ls_in_lb), priority=110, ...
# Load_Balancer: name="web-lb" vips={"10.100.0.1:443"=...}
```

This pattern is especially useful when `ovn-controller` on chassis-42 has a stale cache or a partial sync (a rare bug). Fix: run `ovs-appctl -t ovn-controller recompute` on chassis-42.

## 20.2.4 Port_Binding forensic, eight types, failure patterns, diagnostic chains

Part 13.5 introduced the eight Port_Binding types at the taxonomy level: name, use case, claim logic. Part 20.2 ôÏ20.2.4 takes the forensic angle: for each type, what are the common failure patterns, which diagnostic commands to run, and what fix to apply.

### 20.2.4.1 Reading the `ovn-sbctl list Port_Binding` output

**Context when running.** An on-call engineer is paged: "the VM vm1 just created has no network after two minutes". Per the playbook in Part 20.0 ôÏ20.0.5.5, the first step is to check the binding state. Run `ovn-sbctl list Port_Binding vm1` to dump the full schema.

**Command and output** *(reproduced from the OVN 22.03.8 sandbox)*:

```shell
$ ovn-sbctl list Port_Binding vm1

_uuid               : 8a7b6c5d-4e3f-2a1b-0c9d-8e7f6a5b4c3d
chassis             : 1f2e3d4c-5b6a-7980-1234-567890abcdef
datapath            : 9b8a7c6d-5e4f-3a2b-1c0d-9e8f7a6b5c4d
encap               : []
external_ids        : {}
gateway_chassis     : []
ha_chassis_group    : []
logical_port        : vm1
mac                 : ["aa:bb:cc:00:00:01 10.0.0.1"]
nat_addresses       : []
options             : {requested-chassis=chassis-hv1}
parent_port         : []
requested_chassis   : 1f2e3d4c-5b6a-7980-1234-567890abcdef
tag                 : []
tunnel_key          : 3
type                : ""

virtual_parent      : []
```

**Anatomy of the 18 columns:**

| Column | Sample value | Meaning | Abnormal signal |
|--------|--------------|---------|-----------------|
| `_uuid` | `8a7b6c5d-...` | Unique row UUID, used in `ovn-sbctl find ...` queries | The UUID changes when the port is deleted and recreated. If an application caches the old UUID, it fails |
| `chassis` | `1f2e3d4c-...` | UUID of the chassis currently claiming the port, referencing the Chassis table | `[]` (empty) means not bound. If the VM is running but this is empty, it is a binding bug |
| `datapath` | `9b8a7c6d-...` | UUID of the Datapath_Binding (the LS or LR that contains the port) | Must match the datapath of the LSP or LRP in NBDB |
| `encap` | `[]` | Encap override for this port (usually empty, the chassis encap is used) | Non-empty when the chassis has multiple encaps and the port is forced to one |
| `external_ids` | `{}` | Free-form metadata that Neutron or the CMS writes for additional info | `{neutron:cidrs="10.0.0.1/24", neutron:security_group_ids=...}` for OpenStack |
| `gateway_chassis` | `[]` | Deprecated (replaced by `ha_chassis_group` since OVN 20.06) | If non-empty, the old schema is still in use |
| `ha_chassis_group` | `[]` | UUID of the HA_Chassis_Group for the `chassisredirect` type | Required for type `chassisredirect`, empty on other types |
| `logical_port` | `vm1` | Name of the LSP or LRP (string), matches `ovn-nbctl lsp-list` | Must be unique in the cluster |
| `mac` | `["aa:bb:cc:00:00:01 10.0.0.1"]` | MAC and IP addresses of the port, an array of strings | Format `"<MAC> <IP1> <IP2> ..."`. `unknown` means flood all traffic |
| `nat_addresses` | `[]` | NAT IP advertised via ARP or ND, for the distributed gateway | Only non-empty on an LRP with NAT configuration |
| `options` | `{requested-chassis=chassis-hv1}` | Per-type map, each type has its own keys | A missing required key for the type means the port does not function (see the per-type tables) |
| `parent_port` | `[]` | Parent LSP for nested containers (type `""` with a tag) | Nested-container scenario (rare) |
| `requested_chassis` | `1f2e3d4c-...` | The Chassis the CMS requests to bind to (pin the port), OVN prefers this chassis | If it differs from `chassis`, the orchestrator may have moved the VM and the claim is lagging |
| `tag` | `[]` | VLAN tag for nested containers | Required if `parent_port` is set |
| `tunnel_key` | `3` | OpenFlow metadata equals `tunnel_key`, used to identify the port over the tunnel | Unique within the datapath. A collision is a northd bug |
| `type` | `""` | Port type (see the eight types below) | `""` (empty) is the default VIF |
| `up` | `true` | The port is up (ovn-controller sets this when the OVS interface link is up) | `false` means the OVS interface is not yet up or the link is down. Check `ovs-vsctl show` |
| `virtual_parent` | `[]` | UUID of the virtual parent for type `virtual` | Required if type is `virtual` |

**Troubleshooting cues:**

- **`chassis: []` while the VM is running**: a binding failure. Check `external_ids:iface-id` on the OVS interface (Part 9.13), the ovn-controller log on the target chassis, and the `requested_chassis` field.
- **`chassis != requested_chassis`**: the orchestrator has scheduled the VM to a different chassis but OVN has not updated. This is common during live migration (Part 19). It resolves automatically when migration completes.
- **`up: false` while the VM is powered on**: the OVS interface is down. Check `ovs-vsctl list interface <tap>` for `admin_state` and `link_state`. Usually caused by wrong MTU or VLAN configuration on the tap.
- **`tunnel_key` collision between two ports in the same datapath**: an ovn-northd bug (very rare). Restart northd to fix.
- **`type` non-empty for an LSP VIF**: ovn-northd detected something special (for example, a VIF turning into a `chassisredirect` when the LSP becomes a gateway port). Verify with `ovn-nbctl lsp-get-options <lsp>`.

### 20.2.4.2 Type `""` (VIF), four failure modes

VIF is the most common type, the default for every VM or container LSP. Claim logic: the chassis whose OVS interface has `external_ids:iface-id=<LSP_name>` claims the port.

**Failure mode 1, `iface-id` is not set.** Libvirt or kubelet plugs a tap into `br-int` but has not yet set the external IDs (a race condition, see Part 20.0 ôÏ20.0.7 Case 1). Symptom: `Port_Binding.chassis = []`. Diagnostic:

```shell
# On the chassis running the VM
$ ovs-vsctl --columns=name,external_ids find Interface
# Look for a tap with iface-id matching the LSP name. If none, the orchestrator has not set it
```

Fix: `ovs-vsctl set Interface <tap> external_ids:iface-id=<lsp_name>`.

**Failure mode 2, `iface-id` is duplicated across two chassis.** Two chassis both have a tap with the same iface-id (an orchestrator bug, usually after a failed live migration). Symptom: Port_Binding bounces between the two chassis, and the ovn-controller log shows continuous "claim/unclaim" messages. Diagnostic:

```shell
$ for ch in $(ovn-sbctl --bare --columns=name list Chassis); do
    ssh $ch "ovs-vsctl --columns=name,external_ids find Interface external_ids:iface-id=vm1" | grep -q name && echo "$ch has iface-id vm1"
  done
```

If more than one chassis reports a hit, kill the tap on the wrong chassis.

**Failure mode 3, `port_security` mismatch with the real VM MAC or IP.** `Logical_Switch_Port.port_security` is set to `aa:bb:cc:00:00:01 10.0.0.1` but the real VM sends traffic with a different MAC. The `ls_in_check_port_sec` stage drops the packet. Symptom: `ovn-trace` reports `drop;` at table 0. Diagnostic:

```shell
$ ovn-nbctl lsp-get-port-security vm1
aa:bb:cc:00:00:01 10.0.0.1

$ ssh chassis-hv1 "ip -br link show tap-vm1"
tap-vm1  UP  aa:bb:cc:00:00:02  # wrong MAC, does not match port_security
```

Fix: change the VM MAC or update `port_security`.

**Failure mode 4, the LSP has `addresses=unknown` set but is still expected to forward.** When `addresses=unknown`, OVN floods every packet with an unknown MAC to this LSP (promiscuous mode). However, `ovn-trace` with a specific MAC will show a specific destination LSP, leading to confusion. Symptom: the trace looks fine but the real traffic floods every VM. Diagnostic:

```shell
$ ovn-nbctl lsp-get-addresses vm1
unknown
```

Fix: if the promiscuous behaviour is not intended, set the addresses explicitly with `ovn-nbctl lsp-set-addresses vm1 "aa:bb:cc:00:00:01 10.0.0.1"`.

### 20.2.4.3 Type `patch`, two failure modes

A patch port is auto-generated by ovn-northd to connect an LSP of type `router` with an LRP. There is no claim logic (every chassis has the patch flow).

**Failure mode 1, northd has not compiled the patch.** The LRP is created but no Port_Binding of type `patch` appears. Symptom: the LS and LR are not connected. Diagnostic:

```shell
$ ovn-sbctl --bare --columns=logical_port,type find Port_Binding type=patch
# Expected: an entry with logical_port = <LSP-router-name>

$ ovs-appctl -t ovn-northd status
# Expected: active
```

Fix: restart northd, or run `ovn-appctl -t ovn-northd inc-engine/recompute`.

**Failure mode 2, the patch ports form a cycle.** Rare, occurs when the NB intent has LR1 connected to LR2 connected back to LR1 (a cycle). ovn-northd detects this and refuses to compile. Symptom: the `ovn-northd.log` contains "cycle detected". Fix: correct the NB topology.

### 20.2.4.4 Type `localnet`, three failure modes

A localnet port bridges an LSP to the physical network through a bridge mapping. There is no chassis claim (it is available on every chassis).

**Failure mode 1, `bridge-mappings` is not configured on the chassis.** `ovn-controller` does not know which bridge to map `network_name` to. Symptom: the patch port between `br-int` and `br-ex` is not created on the chassis. Diagnostic:

```shell
$ ovs-vsctl get open_vswitch . external_ids:ovn-bridge-mappings
physnet1:br-ex,physnet2:br-provider
# There must be a mapping for the network_name of the localnet port
```

Fix: `ovs-vsctl set open_vswitch . external_ids:ovn-bridge-mappings=physnet1:br-ex`.

**Failure mode 2, VLAN tag on the localnet but the physical switch port is access, not trunk.** A localnet port with `options:network_name=physnet1 vlan=100` while the switch port only allows untagged. Symptom: traffic leaves `br-ex` with the VLAN tag but the switch drops it. Diagnostic: `tcpdump -eni <br-ex-uplink> vlan 100` on the chassis. Fix: set the switch port to trunk, or change the port to untagged access.

**Failure mode 3, the localnet appears on the LS but the chassis does not need physical egress.** Every chassis still creates a patch to `br-ex`, wasting resources. Symptom: `ovs-vsctl show` shows redundant patch ports on chassis that do not serve external traffic. Fix: use a Logical_Switch with `other_config:localnet_learn_fdb=true` plus restrict the chassis through the `requested-chassis` option.

### 20.2.4.5 Type `chassisredirect` (cr-lrp), four failure modes

A `chassisredirect` port is a shadow LRP used for the distributed gateway, north-bound traffic moves from the distributed LR to the gateway chassis through this port. Claimed by the chassis with the highest priority in the HA_Chassis_Group (BFD-based election).

**Failure mode 1, `ha_chassis_group` is empty or HA_Chassis entries are missing.** No chassis claims the port. Symptom: `Port_Binding.chassis = []` for the cr-lrp. Diagnostic:

```shell
$ ovn-sbctl list Port_Binding cr-lrp-external
# ha_chassis_group: 1234-abcd-...

$ ovn-sbctl list HA_Chassis_Group 1234-abcd-...
# Verify the ha_chassis field has references
```

Fix: `ovn-nbctl ha-chassis-group-add-chassis <group> <chassis-name> <priority>`.

**Failure mode 2, BFD is not up between chassis.** The election fails because the gateway chassis cannot detect the live peer. Symptom: the `chassis` field bounces between chassis. Diagnostic:

```shell
$ ovs-appctl bfd/show
# Check state UP for each peer

$ ssh gateway-chassis "journalctl -u ovn-controller --since '5 min ago' | grep -i bfd"
```

Fix: check that underlay UDP 3784 (BFD) is not blocked by a firewall.

**Failure mode 3, priority collision (two chassis with the same priority).** OVN tie-breaks by UUID, which is non-deterministic. Symptom: the election is not stable across restarts. Fix: assign different priorities to each HA_Chassis member.

**Failure mode 4, the gateway chassis lacks the `enable-chassis-as-gw` configuration.** `ovn-controller` on the gateway chassis does not claim even though it has the highest priority. Diagnostic:

```shell
$ ovs-vsctl get open_vswitch . external_ids:ovn-cms-options
enable-chassis-as-gw
```

Fix: `ovs-vsctl set open_vswitch . external_ids:ovn-cms-options=enable-chassis-as-gw`.

### 20.2.4.6 Type `l3gateway`, two failure modes

`l3gateway` is an LRP centralized on a single chassis, used for the legacy gateway router (pre-distributed). It is being replaced by `chassisredirect` plus the distributed LR. Claim logic: the chassis specified in `options:l3gateway-chassis`.

**Failure mode 1, `l3gateway-chassis` points to a non-existent or offline chassis.** The Port_Binding chassis is empty. Diagnostic:

```shell
$ ovn-sbctl --columns=options list Port_Binding lrp-external
options: {l3gateway-chassis=chassis-offline-uuid}

$ ovn-sbctl list Chassis | grep chassis-offline-uuid
# If absent, the chassis is offline
```

Fix: update `options:l3gateway-chassis` to a live chassis.

**Failure mode 2, the SNAT rule is not effective.** `l3gateway` handles SNAT but the rule was compiled in the wrong order (after LB DNAT). Symptom: north-bound traffic is not SNATed. Diagnostic: `ovn-sbctl lflow-list | grep lr_in_unsnat` and `lr_out_snat`. Fix: usually the NB intent is wrong (set the NAT priority).

### 20.2.4.7 Type `l2gateway`, two failure modes

`l2gateway` bridges an LSP to a hardware VTEP (an external L2 switch running the OVSDB `hardware_vtep` schema). Claimed by `options:l2gateway-chassis`.

**Failure mode 1, OVSDB `hardware_vtep` is not reachable from the chassis.** Symptom: the tunnel between the chassis and the hardware VTEP does not form. Diagnostic:

```shell
$ ovs-vsctl get open_vswitch . external_ids:ovn-remote-vtep
tcp:10.0.0.50:6632
# Test connectivity:
$ nc -zv 10.0.0.50 6632
```

Fix: check the network route and the firewall.

**Failure mode 2, hardware VTEP schema mismatch.** An older VTEP spec version does not support the feature OVN requires. Fix: upgrade the switch firmware or downgrade the OVN configuration.

### 20.2.4.8 Type `localport`, two failure modes

A `localport` exists on every chassis with the same MAC and IP, used for the metadata service (169.254.169.254 in Kubernetes and OpenStack). Traffic does not leave the chassis.

**Failure mode 1, the route inside the VM does not point to the localport.** Symptom: curl to the metadata endpoint times out. Diagnostic inside the VM:

```shell
$ ip route get 169.254.169.254
# Should show "via eth0" or similar
```

Fix: the CMS configuration is wrong, the route must be injected through DHCP_Options or as a static route.

**Failure mode 2, multiple `localport` LSPs in the same LS conflict.** Symptom: metadata requests are not delivered consistently. Fix: keep exactly one `localport` per LS.

### 20.2.4.9 Type `virtual`, two failure modes

A `virtual` port (OVN 22.03 and later) is used for the "floating IP" or keepalived VIP pattern. `virtual_parent` points to the parent LSP, and OVN routes the virtual-IP traffic to whichever chassis currently claims the parent.

**Failure mode 1, `virtual_parent` does not match any parent LSP.** Symptom: the virtual port is not functional. Diagnostic:

```shell
$ ovn-sbctl --columns=logical_port,virtual_parent list Port_Binding type=virtual
$ ovn-sbctl list Port_Binding <parent-name>
# Verify the parent exists
```

Fix: reset `options:virtual-parents=<correct-parent>`.

**Failure mode 2, parent LSP chassis flap causes virtual-port flap.** Symptom: the virtual IP is intermittently reachable when the parent migrates. Fix: review parent HA, reduce migration frequency.

### 20.2.4.10 Type `remote`, one failure mode

The `remote` type is used for OVN interconnect (IC), where ports from a remote cluster are visible in the local SB.

**Failure mode, IC tunnel is not up.** Symptom: the remote Port_Binding exists but cross-cluster traffic fails. Diagnostic: `ovn-ic-sbctl show` plus a check of BFD between transit nodes. Fix: check the IC configuration and the underlay.

### 20.2.4.11 Decision tree for Port_Binding forensic

A condensed matrix for the on-call engineer:

| Symptom | Suspected type | First diagnostic | Common fix |
|---------|----------------|------------------|------------|
| New VM has no network | `""` VIF | `ovs-vsctl get Interface <tap> external_ids:iface-id` | Set iface-id |
| LS and LR are not connected | `patch` | `ovn-sbctl find Port_Binding type=patch` | Restart northd |
| Traffic does not reach the physical network | `localnet` | `ovs-vsctl get open_vswitch . external_ids:ovn-bridge-mappings` | Add a bridge mapping |
| VM cannot ping the default gateway (external) | `chassisredirect` | `ovn-sbctl list Port_Binding cr-lrp-...`, is `chassis` empty? | Check HA_Chassis_Group and BFD |
| North-bound traffic is not NATed | `l3gateway` | `ovn-sbctl --columns=options list Port_Binding lrp-ext` | Update `l3gateway-chassis` |
| Hardware VTEP tunnel is down | `l2gateway` | `ovs-vsctl get open_vswitch . external_ids:ovn-remote-vtep` | Check the OVSDB connection |
| Metadata 169.254.169.254 times out | `localport` | Inside the VM: `ip route get 169.254.169.254` | Check route injection |
| Floating IP does not fail over | `virtual` | `ovn-sbctl list Port_Binding type=virtual` plus `virtual_parent` | Fix the parent reference |
| Cross-cluster traffic fails | `remote` | `ovn-ic-sbctl show` | Check the IC tunnel |

- [`ovn-controller(8)`](https://man7.org/linux/man-pages/man8/ovn-controller.8.html) ôÏinc-engine/show-stats
- OVN source: `controller/ovn-controller.c` function `main_loop()` and `controller/inc-proc-eng.c`

### 20.2.5.3 Anatomy: `ovn-appctl -t ovn-controller lflow-cache/show-stats`

**Context when running.** Following the high `en_lflow_output.recompute` discovery in ôÏ20.2.5.2, check the cache hit rate to confirm the hypothesis that the cache is not effective.

**Command and output** *(reproduced)*:

```shell
$ sudo ovn-appctl -t ovn-controller lflow-cache/show-stats

Enabled: true
high-watermark : 10240
total count    :  8934
cache-max-size :  1000

Cache type                  Entries    Hit count   Miss count
-------------------------------------------------------------
cache-expr-to-expr-inner     234          12834         456
cache-expr                   189           9872         234
cache-matches                512          28456         1245
cache-none                   0               78        8934
```

**Anatomy:**

| Field | Sample value | Meaning | Abnormal signal |
|-------|--------------|---------|-----------------|
| `Enabled` | `true` | The cache is active | `false` means it is disabled via `ovn-remote-probe-interval=0` |
| `high-watermark` | `10240` | Maximum entry count the cache has reached since start | Close to `total count` indicates stable operation; a large gap indicates frequent eviction |
| `total count` | `8934` | Entries currently in the cache | Larger than `cache-max-size` causes cache overflow and eviction |
| `cache-max-size` | `1000` | Maximum size from configuration | The default is usually 50000, so 1000 is tuned low (a bug?) |
| `cache-expr-to-expr-inner` Hit | `12834` | Matches in the inner expression cache | Hit rate equals hit divided by (hit plus miss) equals 12834 / (12834 + 456) equals 96.6 percent, healthy |
| `cache-none` Miss | `8934` | Miss count, the total entries that missed the cache | Matches `total count`: every missed entry was loaded into the cache |

**Troubleshooting cues:**

- **Hit rate of `cache-matches` below 80 percent**: logical-flow churn is too high and the cache is not effective. Increase `cache-max-size` through `ovn-remote-probe-interval` or reduce the LF churn.
- **`cache-none` is growing continuously**: misses are ballooning. Usually caused by an NBDB bulk update (for example, deploying 1000 pods at once). Wait for steady state and recheck.
- **`Enabled: false`**: the operator disabled the cache through configuration (usually for debugging). Re-enable with `ovs-vsctl remove open_vswitch . external_ids ovn-remote-probe-interval`.

### 20.2.5.4 Anatomy: `ovn-appctl -t ovn-controller ct-zone-list`

**Context.** The engineer suspects a conntrack zone collision between an LS and an LR that causes incorrect stateful ACL behaviour. `ct-zone-list` enumerates the LSP-to-zone mapping for the chassis.

**Command and output** *(reproduced)*:

```shell
$ sudo ovn-appctl -t ovn-controller ct-zone-list

ls_a_vm1: 1
ls_a_vm2: 2
ls_b_vm1: 3
rtr1_lrp_a: 1000
rtr1_lrp_b: 1001
rtr1_snat: 1002
```

**Anatomy:**

| Field | Sample value | Meaning |
|-------|--------------|---------|
| `ls_a_vm1` | LSP name with the datapath as prefix | Logical port identifier (LS or LR port) |
| `1` | Zone ID (uint16) | Conntrack zone assigned by ovn-controller |

When a stateful ACL fires `ct(commit, zone=NXM_NX_REG13[0..15])`, `reg13` carries this zone ID. A zone collision (two ports from different datapaths sharing a zone) causes state cross-talk, a serious bug.

**Troubleshooting cues:**

- **Two different ports share the same zone**: an ovn-controller bug or zone-space overflow (more than 64K ports). Verify by sorting the output by zone and checking for duplicates. Fix: upgrade OVN or split the chassis into smaller ones.
- **A port expected to have an entry is missing**: the port has not bound to the chassis. Check `Port_Binding.chassis`.
- **Total entries above 65535**: 16-bit zone overflow. OVN 22.03 and later supports extension but it must be configured. Reduce the port count.

### 20.2.5.5 Anatomy: `ovn-appctl -t ovn-controller connection-status`

**Command and output:**

```shell
$ sudo ovn-appctl -t ovn-controller connection-status
connected
```

Or:

```shell
$ sudo ovn-appctl -t ovn-controller connection-status
not connected
```

A simple output. If it shows `not connected`, check the network and firewall between the chassis and the SBDB cluster, and check the SBDB IP in `external_ids:ovn-remote`.

### 20.2.5.6 `recompute`, force a full iteration

```shell
$ sudo ovn-appctl -t ovn-controller recompute
```

Silent output. Forces a full rebuild of every engine node. Use it when you suspect a stale cache or an incomplete sync. Trade-off: it locks the controller for 1 to 30 seconds depending on cluster size, with a high CPU spike. Avoid in production if an alternative is available.

### 20.2.5.7 `inject-pkt`, test a packet without real traffic

```shell
$ sudo ovn-appctl -t ovn-controller inject-pkt \
    'inport == "vm_a" && eth.src == aa:bb:cc:00:00:01 && eth.dst == aa:bb:cc:00:00:02 && ip4.src == 10.0.0.1 && ip4.dst == 10.0.0.2 && ip.ttl == 64 && icmp4'
```

The packet is injected into `br-int` at the logical-ingress side of port `vm_a`. Real traffic traverses the pipeline and can be observed with `tcpdump` on the destination VM tap. This is useful when the source VM is offline but you still want to test the path to the destination VM.

## 20.2.6 `ovn-appctl -t ovn-northd`, ten runtime management commands

### 20.2.6.1 Catalog of the ten commands

| Command | Argument | Purpose |
|---------|----------|---------|
| `exit` | - | Gracefully terminate |
| `pause` | - | Pause compile. The daemon still receives NB and SB change notifications but does not emit updates |
| `resume` | - | Resume compile |
| `is-paused` | - | Return `true` or `false` |
| `status` | - | Print `active`, `standby`, or `paused` |
| `sb-cluster-state-reset` | - | Reset SB cluster state when the database is rebuilt |
| `nb-cluster-state-reset` | - | Reset NB cluster state when the database is rebuilt |
| `set-n-threads` | `<N>` | Set N threads for parallel build (2 to 256). Parallel mode activates when N is at least 2 |
| `get-n-threads` | - | Return current thread count |
| `inc-engine/show-stats` | - | Engine counters (recompute, compute, abort) |
| `inc-engine/clear-stats` | - | Reset counters |

### 20.2.6.2 Anatomy of `ovn-appctl -t ovn-northd status`

**Scenario.** An engineer checks the HA state of a three-node northd cluster. Each cluster must show exactly one active node plus two standby nodes. Zero active nodes means the cluster is frozen, and two or more active nodes means split-brain.

**Command on each node:**

```shell
[node1]$ sudo ovn-appctl -t ovn-northd status
standby

[node2]$ sudo ovn-appctl -t ovn-northd status
active

[node3]$ sudo ovn-appctl -t ovn-northd status
standby
```

**Value anatomy:**

| Value | Meaning | Abnormal signal |
|-------|---------|-----------------|
| `active` | The node is holding the OVSDB lock on SB DB and is compiling | Exactly one node should report active. More than one means split-brain |
| `standby` | The node is ready to take over but has not acquired the lock | N minus one nodes in the cluster must be standby |
| `paused` | The node has been paused via `ovn-appctl pause` and does not compile even when it would be active | Usually only seen in test or debug environments |

**Failure scenarios:**

- **All nodes report `standby`.** No node has acquired the lock. SBDB does not accept the lock request (cluster degraded). Check SBDB Raft state with `ovn-appctl -t ovsdb-server cluster/status OVN_Southbound`. Fix: restore the SBDB quorum.
- **More than one node reports `active`.** Split-brain, very serious. Stop the extra instances, keep one node active. Verify the SBDB lock holder with a raw query.
- **A node remains `paused` in steady state.** Someone paused it and forgot to resume. Fix: `ovn-appctl -t ovn-northd resume`.

### 20.2.6.4 `pause` and `resume`, maintenance window

When upgrading NBDB schema or restoring a backup:

```shell
# Pause northd. It still monitors NB and SB but does not emit updates
$ sudo ovn-appctl -t ovn-northd pause

# Perform maintenance work on SBDB
$ ovn-sbctl restore /backup/sb.db

# Resume northd
$ sudo ovn-appctl -t ovn-northd resume

# Verify
$ sudo ovn-appctl -t ovn-northd is-paused
false
```

While northd is paused, ovn-controller still reads from its local SB DB cache, and traffic is not disrupted. Only new NB intent fails to sync down to SB; it is delayed until resume.

### 20.2.6.5 `set-n-threads`, parallel compile tuning

ovn-northd 22.03 and later supports multi-thread compile:

```shell
# Check current
$ sudo ovn-appctl -t ovn-northd get-n-threads
1

# Set 4 threads (parallel mode activates at 2 or more)
$ sudo ovn-appctl -t ovn-northd set-n-threads 4

# Verify
$ sudo ovn-appctl -t ovn-northd get-n-threads
4
```

The default is N equals 1 (no parallelism). Recommended: N equals the minimum of half the CPU core count and 4 for production. Too many threads cause lock contention inside the NB and SB IDL.

## 20.2.7 Stateful table triage, MAC_Binding, FDB, Service_Monitor

Beyond Port_Binding, OVN has three stateful tables that directly influence the dataplane.

### 20.2.7.1 MAC_Binding, the ARP and NDP cache for LR

The `MAC_Binding` table caches the resolved MAC for each (LRP, IP) pair. Stage `lr_in_arp_resolve` reads this table to set eth.dst before forwarding. Stage `lr_in_learn_neighbor` writes into the table when it sees an ARP reply.

**Short schema:** `logical_port` (LRP), `ip`, `mac`, `datapath`, `timestamp`.

**Diagnostic:**

```shell
# List all MAC_Binding rows
$ ovn-sbctl list MAC_Binding

# Find the entry for a specific IP
$ ovn-sbctl find MAC_Binding ip=10.0.1.5

# Aging: entries whose timestamp is not refreshed within X seconds are garbage-collected by northd.
# X is configured via NB_Global.options:mac_binding_age_threshold (default 0 means disabled).
$ ovn-nbctl get NB_Global . options:mac_binding_age_threshold
```

**Failure modes:**

- Entry is stale (the IP has a new MAC after a fresh ARP). The next hop is not reachable. Fix: `ovn-sbctl destroy MAC_Binding <uuid>` to force re-learn, or tune the age threshold.
- Entry is missing after a restart. Wait for ARP to resolve (up to 60 seconds), or set static addresses on the remote LSP.

### 20.2.7.2 FDB, dynamic MAC learning (OVN 22.03 and later)

OVN 22.03 and later adds an FDB table for L2 MAC learning inside an LS. Before that release, only static entries existed (from `Logical_Switch_Port.addresses`). Stage `ls_in_lookup_fdb` reads, and `ls_in_put_fdb` writes.

**Schema:** `mac`, `dp_key`, `port_key`, `timestamp`.

**Diagnostic:**

```shell
$ ovn-sbctl list FDB

# Per datapath
$ ovn-sbctl find FDB dp_key=3

# Aging via NB_Global.options:fdb_age_threshold (default 0 means disabled).
```

**Failure mode:** FDB poisoning (see Part 17 forensic). A duplicate MAC across two VMs or ports causes traffic to land on the wrong state. Fix: purge the offending FDB entry and identify the bogus source.

### 20.2.7.3 Service_Monitor, Load_Balancer health check

LB backend health is monitored through the Service_Monitor table. Each backend has one row. ovn-controller probes via TCP or ICMP at regular intervals and updates the `status` field.

**Schema:** `ip`, `port`, `protocol`, `status` (online or offline), `src_mac`, `src_ip`, `logical_port`.

**Diagnostic:**

```shell
$ ovn-sbctl list Service_Monitor

# Find offline backends
$ ovn-sbctl --bare --columns=ip,port,status find Service_Monitor status=offline
```

**Failure modes:**

- Backend is offline but the LB still routes to it. The health check has not yet detected the failure (default probe interval is 10 seconds). Tune `options:interval=5` on the LB.
- Backend is online but does not serve. An L7 health check is needed (OVN only does L4). Supplement with an external check.

## 20.2.9 Guided Exercise 1, silent ACL drop with multi-scenario `ovn-trace --ct`

> **State verification lab:** doc-plausible, based on the OVN tutorial and `ovn-trace(8)`. Replace with verified lab output once a lab host is available (C1b).

**Goal.** Demonstrate how to distinguish a stateless ACL from a stateful one by running `ovn-trace` multiple times with different `--ct` flags to identify which rule drops the reply path.

**Setup:**

- One Ubuntu 22.04 node with OVN 22.03 installed.
- One Logical Switch `ls_web` with two LSPs: `client` (10.0.0.1) and `web` (10.0.0.2).
- One ACL: `from-lport priority=1000 match="inport == \"client\" && tcp.dst == 80" action=allow`.

**Topology:**

```
[client 10.0.0.1]  --- [ls_web] ---  [web 10.0.0.2]
     |                                    |
     +------- TCP :80 allow forward ------+
              TCP :80 reply ??? ------
```

**Steps:**

**1.** Trace the forward path (NEW connection):

```shell
$ ovn-trace --detailed ls_web \
    'inport == "client" && eth.src == aa:bb:cc:00:00:01 && eth.dst == aa:bb:cc:00:00:02 &&
     ip4.src == 10.0.0.1 && ip4.dst == 10.0.0.2 && ip.ttl == 64 &&
     tcp && tcp.src == 52341 && tcp.dst == 80'
```

Expected: `output to "web"`. The ACL at priority 1000 fires with action `allow;`.

**2.** Trace the reply path (default NEW state, no `--ct` flag):

```shell
$ ovn-trace --detailed ls_web \
    'inport == "web" && eth.src == aa:bb:cc:00:00:02 && eth.dst == aa:bb:cc:00:00:01 &&
     ip4.src == 10.0.0.2 && ip4.dst == 10.0.0.1 && ip.ttl == 64 &&
     tcp && tcp.src == 80 && tcp.dst == 52341'
```

Expected: `drop;` at stage `ls_out_acl` or `ls_in_acl` (depending on direction). The reply is dropped because no ACL matches it.

**3.** Trace the reply with `--ct=trk,est,rpl`:

```shell
$ ovn-trace --ct=trk,est,rpl --detailed ls_web \
    'inport == "web" && eth.src == aa:bb:cc:00:00:02 && eth.dst == aa:bb:cc:00:00:01 &&
     ip4.src == 10.0.0.2 && ip4.dst == 10.0.0.1 && ip.ttl == 64 &&
     tcp && tcp.src == 80 && tcp.dst == 52341'
```

Expected: if the ACL is `allow-related`, the result is `output to "client"` (reply passes). If the ACL is plain `allow`, the result remains `drop;`.

**4.** Fix the ACL from stateless to stateful:

```shell
$ ovn-nbctl acl-del ls_web from-lport 1000 'inport == "client" && tcp.dst == 80'
$ ovn-nbctl acl-add ls_web from-lport 1000 'inport == "client" && tcp.dst == 80' allow-related
```

**5.** Repeat the step 3 trace and observe:

```shell
$ ovn-trace --ct=trk,est,rpl --detailed ls_web '...'
# Expected: "output to \"client\"". The reply is now allowed by stateful tracking.
```

**Lessons:**

- Plain `allow` is stateless. The reply is not auto-permitted. An explicit reverse-direction ACL is required.
- `allow-related` is stateful. When conntrack state matches est and rpl, the reply is auto-permitted.
- `--ct=trk,est,rpl` is the only tool to distinguish these two behaviours without sending real traffic.

**Cleanup:**

```shell
$ ovn-nbctl ls-del ls_web
```

## 20.2.12 Capstone POE, refute the hypothesis "engine_recompute above 10 percent per minute means restart is required"

> **Scenario:** The monitoring system fires the alert "engine_recompute rate above 10 events per minute on chassis-42". The on-call engineer hypothesises: "ovn-controller is stuck, must restart."

**Predict:** A restart will clear state, and the recompute rate will return to zero. The hypothesis attributes the recompute spike to a memory leak bug or to stale state.

**Observe:**

**1.** Check the recompute breakdown per node:

```shell
$ sudo ovn-appctl -t ovn-controller inc-engine/show-stats

Node: en_runtime_data
- recompute: 245      <-- high
- compute: 8923
- cancel: 12

Node: en_lflow_output
- recompute: 12       <-- normal
- compute: 8900
- cancel: 3
```

The recompute concentrates on `en_runtime_data` (Port_Binding-related), not on `en_lflow_output` (LF translation). The memory leak hypothesis becomes suspect, because a leak usually distributes evenly across nodes.

**2.** Check recent churn:

```shell
$ ovn-sbctl --columns=logical_port,chassis list Port_Binding \
    | awk '/chassis/ {print $NF}' | sort | uniq -c | sort -rn | head

  120 hv-new-uuid-xyz      <-- a new hypervisor just joined the cluster and received 120 ports
  89 hv-42-uuid
  45 hv-other
```

`hv-new-uuid` has just joined and received a burst of 120 ports. This is an infrastructure event, not a bug.

**3.** Check engine history:

```shell
$ sudo journalctl -u ovn-controller --since "30 min ago" | grep -i "chassis" | head
2026-04-24T10:15:22 ... Chassis hv-new joined
2026-04-24T10:15:22 ... Received 120 Port_Binding updates
2026-04-24T10:15:23 ... en_runtime_data recompute (fallback: bulk update)
```

**4.** Wait two or three minutes and check again:

```shell
$ sudo ovn-appctl -t ovn-controller inc-engine/show-stats

Node: en_runtime_data
- recompute: 245       <-- no longer increasing
- compute: 9823        <-- compute keeps growing steadily
- cancel: 12
```

The recompute counter no longer increases. The system has recovered.

**Explain:** The recompute spike is **transient**, caused by a newly joined chassis that received a burst of port updates. The incremental processing engine does not support bulk updates atomically (a schema limitation), so it falls back to a full recompute for the affected node. Once the bulk update settles, the engine returns to incremental mode.

**Refute:** The hypothesis "must restart" is wrong. A restart would force a full recompute across all eight engine nodes (a larger spike than 245), disrupt traffic, and fail to fix the root cause (a new chassis joining is a normal event).

**Correct action:** Acknowledge the alert and annotate it as "chassis join event, transient". Tune the alert threshold: a recompute rate above 50 per minute **sustained for more than 10 minutes** is actionable. A single burst lasting less than 10 minutes is normal.

**Lesson:** Metrics alone can lie. Always correlate with infrastructure events (chassis join, schema upgrade, orchestrator batch). Restart is a cudgel, not a scalpel. Use it when the root cause is known, not as an intuitive way to "clear state".

## 20.2.13 Common misconceptions

> **Misconception 1:** "tcpdump is the first tool to use when debugging OVN."
> **Correct:** `ovn-trace` and `ofproto/trace` are simulations that run in milliseconds and do not depend on timing. `tcpdump` only sees the wire, not the logical flow decision. Use simulation first to narrow the scope, then capture.

> **Misconception 2:** "If `ovn-trace` shows delivered, real traffic will certainly succeed."
> **Correct:** `ovn-trace` simulates the logical level only (NB and SB). In reality, traffic also passes through OpenFlow (ovn-controller translation) and the datapath (OVS execution). When the trace looks good but traffic still fails, the bug is at layer C (OpenFlow) or layer 4 or 5 (datapath or tunnel).

> **Misconception 3:** "An empty `Port_Binding.chassis` is always a bug."
> **Correct:** Types `localnet` and `patch` do not claim a chassis by design (they are available on every chassis). Only types `""`, `chassisredirect`, `l3gateway`, `l2gateway`, `virtual`, and `remote` expect a non-empty chassis.

> **Misconception 4:** "Lower engine_recompute in inc-engine/show-stats is always better; zero is ideal."
> **Correct:** A recompute count of zero is not realistic. Every chassis restart, schema change, or bulk NB update triggers at least one recompute. The healthy target is a recompute-to-compute ratio below 1 percent, not zero recomputes.

> **Misconception 5:** "Restarting ovn-controller clears all state and fixes any state-related bug."
> **Correct:** A restart triggers a full recompute (a heavy spike), disrupts traffic for several seconds, and often does not fix the bug (which usually lives in NB intent or in northd). Use the `recompute` command if you only need to refresh state.

> **Misconception 6:** "Reading the cookie from `dump-flows` with ovn-detrace explains every OpenFlow rule."
> **Correct:** ovn-detrace only annotates flows whose cookie matches an SBDB Logical_Flow. Flows generated internally by ovn-controller (table 0 demux, table 64 loopback) carry no cookie, so they are not annotated. Understand when annotation is and is not available.

> **Misconception 7:** "`ovn-trace --ct=trk,est` is enough to verify a stateful ACL reply path."
> **Correct:** You also need `rpl` (the reply direction). The flags `trk,est` only mean "established", not "reply". An `allow-related` ACL needs `trk,est,rpl` to fire its reverse-direction logic.

> **Misconception 8:** "northd pause disconnects traffic."
> **Correct:** Pause only stops the NB-to-SB compile. Existing traffic keeps flowing because ovn-controller reads from its local SB DB cache. Only new NB intent is delayed until resume. Pause is safe to use during a maintenance window.

## 20.2.14 Key takeaways

First, **debug OVN through three layers (NB intent, then SB Logical_Flow, then OpenFlow on br-int)**. Simulate before capturing: `ovn-trace`, `ofproto/trace`, and `ovn-detrace` give answers in milliseconds. Reach for `tcpdump` only when simulation already looks good but traffic still fails.

Second, **`ovn-trace` has four output modes** (`--detailed` is the default, `--summary` is pseudo-code, `--minimal` is stable for regression tests, `--all` is combined) and three state-injection options (`--ct=flags` for conntrack state, `--lb-dst` to force a backend, `--select-id` to force a group bucket). Knowing which mode to use when is a mastery signal.

Third, **chain `ovn-detrace` with `ofproto/trace`** to go from an OpenFlow cookie to a Logical_Flow row to the originating NB object (ACL, LB, or NAT). Export `OVN_SB_DB` and `OVN_NB_DB` so annotation is complete. The pattern of diffing across chassis by piping `dump-flows | diff | ovn-detrace` is useful when a specific chassis is out of sync.

Fourth, **the eight Port_Binding types have different claim logic and failure modes**. VIF (`""`) claims through `external_ids:iface-id`; `chassisredirect` claims through HA_Chassis_Group plus BFD election; `localnet`, `patch`, and `localport` do not claim; `virtual` follows `virtual_parent`. The nine-pattern diagnostic matrix is consolidated in ôÏ20.2.4.11.

Fifth, **`ovn-appctl -t ovn-controller` has eleven commands**. The most important are `inc-engine/show-stats` (engine health), `lflow-cache/show-stats` (cache hit rate), `ct-zone-list` (conntrack zone collision), `recompute` (force rebuild), and `connection-status` (is SBDB reachable). `ovn-appctl -t ovn-northd` has ten commands. The most important are `status` (active, standby, or paused), `inc-engine/show-stats`, and `set-n-threads` (parallel compile tuning).

Sixth, **metrics must be correlated with infrastructure events**. An `engine_recompute` spike is not by itself a bug: chassis join, schema upgrade, or bulk NB update all trigger normal recomputes. The alert threshold should be "sustained for more than ten minutes", not "single spike". Restart is not a scalpel for every bug. Use it only when the root cause is identified and clearing state is genuinely required.

## 20.2.15 Backfill v3.5, lflow-cache configuration tuning (five `external_ids` tunables)

> **Purpose.** Audit J.1 of v3.5-KeywordBackbone found five lflow-cache tunables in the `external_ids` configuration space that this file had not yet covered. File 20.2 already covers `lflow-cache/show-stats` (read state) but has no tuning anatomy. This section backfills the tuning view, source verified against the `ovn-controller(8)` man page.

### 20.2.15.1 Why tune lflow-cache

`ovn-controller` caches compiled flow results in RAM to avoid recomputing on every packet. The default behaviour (unlimited cache) works well for clusters with fewer than 1000 chassis and fewer than 100k total Logical_Flow rows. Beyond that scale, the cache can:

- Consume too much RAM (chassis OOM).
- Fail to trim quickly enough (latency rises for new computes).
- Retain stale entries that are not flushed (data inconsistency).

Five tunables in `external_ids` of the Open_vSwitch table let an operator adjust cache behaviour for large-scale deployments.

### 20.2.15.2 Anatomy of the five lflow-cache tunables (verified against `ovn-controller(8)`)

| Tunable | Default | Scope | Purpose |
|---------|---------|-------|---------|
| `external_ids:ovn-enable-lflow-cache` | `true` | enable or disable the cache as a whole | Disable the cache on a RAM-constrained chassis (rare) |
| `external_ids:ovn-limit-lflow-cache` | unlimited | maximum entry count | Cap cache size by entry count (when the deployment knows its flow count) |
| `external_ids:ovn-memlimit-lflow-cache-kb` | unlimited | maximum cache memory (KB) | Cap cache size by memory (when the deployment has a RAM budget) |
| `external_ids:ovn-trim-limit-lflow-cache` | 10000 | minimum entries before trimming starts | Threshold that triggers auto-trim |
| `external_ids:ovn-trim-wmark-perc-lflow-cache` | 50% | trim watermark percentage | Trim aggressiveness (high-watermark percent) |

### 20.2.15.3 Set and verify the tunables

**Set tunables:**

```bash
# Disable entirely (debug only)
ovs-vsctl set Open_vSwitch . external_ids:ovn-enable-lflow-cache=false

# Cap entry count at 50k
ovs-vsctl set Open_vSwitch . external_ids:ovn-limit-lflow-cache=50000

# Cap memory at 256 MB
ovs-vsctl set Open_vSwitch . external_ids:ovn-memlimit-lflow-cache-kb=262144

# Trim threshold of 5k entries (aggressive)
ovs-vsctl set Open_vSwitch . external_ids:ovn-trim-limit-lflow-cache=5000

# Trim watermark of 70 percent (less aggressive)
ovs-vsctl set Open_vSwitch . external_ids:ovn-trim-wmark-perc-lflow-cache=70
```

**Verify the tunable was applied:**

```bash
ovs-vsctl get Open_vSwitch . external_ids
# Output: {ovn-enable-lflow-cache="true", ovn-limit-lflow-cache="50000", ...}

# Verify ovn-controller picked it up (via show-stats)
ovn-appctl -t ovn-controller lflow-cache/show-stats
# Output: enabled, total_size, max_size, hit, miss, ...
```

### 20.2.15.4 Tuning decision matrix

| Symptom | Diagnostic | Tune action |
|---------|------------|-------------|
| Chassis RAM above 80 percent | `ovn-appctl lflow-cache/show-stats` shows total_size above 1 GB | Set `ovn-memlimit-lflow-cache-kb=512000` (cap at 512 MB) |
| ovn-controller restart is slow (more than two minutes) | `inc-engine/show-stats` shows a high recompute count | Raise `ovn-limit-lflow-cache` to allow a larger cache (reduces recomputes) |
| Cache hit rate is low (below 50 percent) | `lflow-cache/show-stats` ratio of hit to (hit+miss) | Raise `ovn-trim-limit-lflow-cache` to keep entries longer (reduces trimming) |
| Frequent OOM kills of ovn-controller | `dmesg | grep ovn-controller` shows OOM | Set `ovn-memlimit-lflow-cache-kb` cap; consider disabling the cache on small chassis |
| Cache thrashing (entries added and removed quickly) | `lflow-cache/show-stats` shows a high evicted count | Lower `ovn-trim-wmark-perc-lflow-cache` to 30 percent (less aggressive trim) |

### 20.2.15.5 Anti-patterns and notes

**Anti-pattern 1: disabling the cache on a production chassis.**

Setting `ovn-enable-lflow-cache=false` forces ovn-controller to compile on every packet, driving CPU to 100 percent under load. Use only to debug whether "the cache is corrupt"; restore `true` immediately afterwards.

**Anti-pattern 2: setting `ovn-limit-lflow-cache` too low (below 1000).**

A small cache trims continuously: a freshly inserted entry is trimmed before it is reused, every new packet recomputes, and throughput drops. The unlimited default is appropriate for most deployments.

**Note 3: a tunable change does not require restarting ovn-controller.**

`ovs-vsctl set` updates `external_ids`; ovn-controller picks up the change in its next loop (within about one second). Verify with `lflow-cache/show-stats` and confirm the new max_size.

**Note 4: memory accounting does not include OVS internal structures.**

`ovn-memlimit-lflow-cache-kb` only counts the size of cached lflow entries. It does not count OVS internal overhead (hash table, free list). Real memory consumed can be 1.2 to 1.5 times the configured value. Set conservatively (around 75 percent of the RAM budget) for a safety margin.

## 4.9.1 Action vs Instruction, khác biệt foundation

OpenFlow 1.1+ phân biệt hai concept:

| Concept | Khi nào execute | Thay đổi state nào | Nằm trong | Ví dụ |
|---|---|---|---|---|
| **Action** | Thực hiện ngay trên gói tin | Gói tin header + output port | `apply_actions:` hoặc `write_actions:` | `output`, `set_field`, `dec_ttl`, `ct()` |
| **Instruction** | Điều khiển pipeline | Flow chuyển giữa table | Phần tử top-level của flow entry | `Apply-Actions`, `Write-Actions`, `Clear-Actions`, `Goto-Table`, `Write-Metadata`, `Meter` |

OpenFlow 1.0 không phân biệt — mọi thứ là "action" execute tuần tự. Từ 1.1, instruction là **wrapper bao quanh action**, quyết định WHEN và HOW action execute.

### 4.9.1.1 Sáu instruction foundation

| Instruction | Mục đích |
|---|---|
| `Apply-Actions(<action_list>)` | Execute action list ngay lập tức, thay đổi gói tin visible cho instruction sau |
| `Clear-Actions` | Clear action set hiện tại |
| `Write-Actions(<action_list>)` | Merge action list vào action set, execute khi gói tin exit pipeline |
| `Write-Metadata(<value>[/<mask>])` | Update metadata match field cho gói tin (pass qua Goto-Table) |
| `Goto-Table(<table_id>)` | Chuyển gói tin sang table khác (table_id phải lớn hơn current) |
| `Meter(<meter_id>)` | Apply meter rate limit trước khi forward |

Trong OVS flow spec syntax, `actions=` là shorthand cho `Apply-Actions(...)`. Instruction form đầy đủ hiếm dùng trực tiếp nhưng quan trọng khi đọc OpenFlow 1.3+ spec hoặc trace output.

### 4.9.1.2 Action Set vs Apply Actions

Pipeline OpenFlow 1.1+ duy trì **action set** — tập action tích lũy qua các table. Khi gói tin exit pipeline (không còn `Goto-Table`), action set execute theo 12 priority level (xem §4.9.14).

Khác biệt hành vi:

```
# apply_actions: execute ngay
table=0,priority=100,in_port=1,
  apply_actions(mod_dl_dst:aa:bb:cc:dd:ee:ff),
  goto_table:10

# Table 10 thấy packet với eth_dst đã đổi
table=10,eth_dst=aa:bb:cc:dd:ee:ff,actions=output:5    # match

# write_actions: tích lũy vào action set
table=0,priority=100,in_port=1,
  write_actions(output:5),
  goto_table:10

# Table 10 không thấy output trong actions hiện tại
table=10,actions=<...>  # action set vẫn có output:5, merge với gì table 10 ghi
```

Khi xuất pipeline, tất cả write_actions merge theo 12-level priority, rồi execute. Apply_actions không tích lũy.

### 4.9.1.3 OVS actions= syntax shortcut

OVS simplify syntax — `actions=` mặc định là Apply-Actions trừ khi explicit `write_actions(...)`. Trong `ovs-ofctl` flow spec:

```
# Syntax 1: shortcut (implicit apply_actions)
priority=100,in_port=1,actions=output:5

# Syntax 2: explicit apply_actions
priority=100,in_port=1,apply_actions(output:5)

# Syntax 3: write_actions (merge vào action set)
priority=100,in_port=1,write_actions(output:5),goto_table:10

# Syntax 4: mix
priority=100,in_port=1,
  actions=dec_ttl,
  write_actions(output:5),
  goto_table:10
```

Phần còn lại của Phần này focus vào **action** (không phải instruction). Instruction chi tiết xem Phần 4.0 §4.0.X.

## 4.9.2 Action `output`, cơ bản nhất của OpenFlow

### 4.9.2.1 Template C cho `output`

| Attribute | Giá trị |
|---|---|
| **Syntax** | `output:<port>` hoặc `output(port=<port>,max_len=<bytes>)` |
| **Category** | 1. Output |
| **OpenFlow version** | 1.0 all versions |
| **Prerequisites** | None |
| **Semantics** | Gửi gói tin ra cổng chỉ định. Không terminate gói tin — có thể dùng `output` nhiều lần cho multiple port |
| **Parameters** | `port`: 1-N (port number) hoặc reserved port (LOCAL/CONTROLLER/IN_PORT/NORMAL/FLOOD/ALL/TABLE); `max_len`: bytes cho CONTROLLER, default unlimited |
| **Side effects** | Tăng tx_packets/tx_bytes counter của port; không modify gói tin content |
| **Conformance** | Đúng spec; reserved port CONTROLLER/TABLE là OpenFlow concept, LOCAL là OVS/common |

### 4.9.2.2 Ngữ nghĩa đầy đủ

`output` là action foundation nhất, execute đầu tiên trong action set priority (xem §4.9.14). Khi gói tin đi qua action list có `output:5`, OVS:

1. Compute egress port (port 5 trong ví dụ).
2. Tạo gói tin clone nếu cần (cho mirror).
3. Queue gói tin vào send buffer của port 5.
4. Gói tin content KHÔNG bị modify (tất cả `mod_*`/`set_field` phải chạy trước).

Nếu action list có nhiều `output`, gói tin được clone và gửi ra nhiều port. Đây là cách OVS implement flood/multicast **bằng rule explicit** thay vì reserved port FLOOD/ALL.

### 4.9.2.3 Ví dụ use case

```shell
# Ex 1: simple forwarding
$ ovs-ofctl add-flow br0 "table=0,priority=100,in_port=1,actions=output:2"

# Ex 2: multi-output (flood giới hạn)
$ ovs-ofctl add-flow br0 "table=0,priority=100,in_port=1,dl_type=0x0806,\
    actions=output:2,output:3,output:4"   # broadcast ARP ra 3 port

# Ex 3: output với max_len (cho CONTROLLER)
$ ovs-ofctl add-flow br0 "table=0,priority=100,in_port=1,\
    actions=output(port=CONTROLLER,max_len=128)"  # chỉ gửi 128 byte đầu

# Ex 4: xuất port name thay vì number
$ ovs-ofctl add-flow br-int "table=0,priority=100,in_port=tap-vm-web,\
    actions=output:tap-vm-db"
```

### 4.9.2.4 Trace output hiển thị action

```
bridge("br0")
-------------
 0. in_port=1, priority 100
    output:2
```

Trace show action `output:2` là dòng con của match rule. Nếu trace có `Datapath actions: 2` → kernel sẽ forward ra port datapath 2.

### 4.9.2.5 Kịch bản bẻ gãy

- **`output:<port>` với port không tồn tại** → action skip silent, không lỗi; gói tin không đi đâu. Kiểm chứng bằng `ovs-ofctl show br0` xem port list.
- **`output:IN_PORT` có ý nghĩa đặc biệt:** gửi gói tin về ingress port (ARP responder pattern). Nếu dùng `output:<ingress_port_number>` thay vì `output:IN_PORT`, OVS từ chối (không cho loopback trực tiếp trừ khi dùng reserved port).
- **`output` trong `write_actions(...)` chỉ giữ 1 giá trị** — action set chỉ có 1 output effective. Nếu muốn multi-output trong action set, phải dùng `group` action.

## 4.9.3 Action `drop`, terminate gói tin

### 4.9.3.1 Template C

| Attribute | Giá trị |
|---|---|
| **Syntax** | `drop` |
| **Category** | 1. Output |
| **OpenFlow version** | Explicit `drop` là NXM extension; OF spec nói "empty action list = drop" (implicit) |
| **Prerequisites** | None |
| **Semantics** | Terminate gói tin processing. Không output, không modify. Counter `n_packets` của rule vẫn tăng |
| **Parameters** | None |
| **Side effects** | Counter flow rule increment; gói tin discarded; kernel datapath có dedicated `OVS_ACTION_ATTR_DROP` (OVS 2.17+) cho efficient drop path |
| **Conformance** | OVS extension; spec 1.0 dùng empty action list = implicit drop |

### 4.9.3.2 Explicit drop vs implicit drop

```
# Implicit drop (rule có match nhưng action rỗng)
priority=50,in_port=1,dl_type=0x0800,actions=

# Explicit drop (OVS extension)
priority=50,in_port=1,dl_type=0x0800,actions=drop
```

Hai forms tương đương semantically. Explicit `drop` dễ đọc + tường minh. OpenFlow 1.5+ yêu cầu explicit drop (spec §5.10).

Nếu không có rule nào match gói tin (table-miss), OpenFlow 1.3+ mặc định drop. Controller nên explicit install table-miss rule để:
- Drop với log: `priority=0,actions=controller(userdata=01020304)`
- Forward lên controller để learn: `priority=0,actions=output:CONTROLLER`

### 4.9.3.3 Trace hiển thị drop

```
 0. in_port=1,ip,nw_proto=1, priority 150
    drop

Datapath actions: drop
```

`Datapath actions: drop` là ground truth — kernel sẽ DROP gói tin, counter `missed` không tăng thêm (vì gói tin đã match megaflow drop).

### 4.9.3.4 Dedicated drop action OVS 2.17+

Trước OVS 2.17, drop được implement qua action list rỗng. OVS 2.17 thêm **OVS_ACTION_ATTR_DROP** trong datapath — kernel có fast-path dedicated cho drop, tránh context switch. Check qua `dpif/show-dp-features`:

```shell
$ sudo ovs-appctl dpif/show-dp-features br0 | grep -i drop
Explicit Drop action: Yes
```

`No` → kernel cũ, explicit drop fallback về generic drop path, hiệu năng suy giảm nhẹ.

## 4.9.4 Action `normal`, standard L2 learning switch

### 4.9.4.1 Template C

| Attribute | Giá trị |
|---|---|
| **Syntax** | `normal` hoặc `actions=NORMAL` |
| **Category** | 1. Output |
| **OpenFlow version** | OF 1.0 all (reserved port NORMAL) |
| **Prerequisites** | None |
| **Semantics** | Xử lý gói tin như standard L2 switch: MAC learning + chuyển tiếp dựa trên FDB. Action cuối, terminate pipeline |
| **Parameters** | None |
| **Side effects** | MAC learning tự động (populate FDB); broadcast nếu unknown unicast; STP/RSTP được apply nếu enable |
| **Conformance** | OpenFlow reserved port; hành vi implementation-specific (OVS specific trong đa số case) |

### 4.9.4.2 Use case

`normal` hữu ích khi muốn mix OpenFlow logic và L2 learning. Mẫu phổ biến:

```shell
# Layer 2 switch với một exception ACL
$ ovs-ofctl add-flow br0 "priority=100,ip,nw_dst=10.0.0.99,actions=drop"
$ ovs-ofctl add-flow br0 "priority=0,actions=normal"
```

Rule priority 100 drop traffic tới 10.0.0.99. Mọi traffic khác fall through tới priority 0 → `normal` → L2 switch chuẩn.

### 4.9.4.3 Fail_mode interaction

Bridge với `fail_mode=secure` + controller disconnect → không action NORMAL tự động. Phải explicit install `actions=NORMAL` rule để fallback.

Bridge `fail_mode=standalone` + controller disconnect → OVS auto-install `actions=NORMAL` rule priority 0 → L2 switch ngay.

### 4.9.4.4 Kịch bản bẻ gãy

- **`normal` trong OVN-managed br-int** → OVN KHÔNG dùng `normal`. OVN build pipeline explicit với từng match rule. Dùng `normal` trên br-int OVN sẽ phá pipeline.
- **STP + NORMAL:** khi enable STP qua `ovs-vsctl set Bridge br0 stp_enable=true`, NORMAL action tôn trọng STP state — gói tin vào port BLOCKING không forward.

## 4.9.5 Action `flood`, `all`, broadcast variants

### 4.9.5.1 `flood`

| Attribute | Giá trị |
|---|---|
| **Syntax** | `flood` hoặc `actions=FLOOD` |
| **Semantics** | Output gói tin ra tất cả port thuộc bridge **trừ ingress port + port bị STP block** |
| **Scope** | Bridge-level, không tôn trọng VLAN membership |

### 4.9.5.2 `all`

| Attribute | Giá trị |
|---|---|
| **Syntax** | `all` hoặc `actions=ALL` |
| **Semantics** | Output gói tin ra **tất cả port** của switch, kể cả ingress, kể cả port STP block. "Unfiltered flood" |

### 4.9.5.3 So sánh

| Action | Include ingress | STP respect | VLAN filter |
|---|---|---|---|
| `flood` | Không | Có | Không |
| `all` | Có | Không | Không |
| `normal` | Không | Có | Có |
| `output:<n>, output:<m>, ...` | Tùy explicit | Không | Không |

Dùng `flood` cho broadcast ARP response (STP-safe). Dùng `all` cho debug/mirror (xem gói tin ở mọi port).

### 4.9.5.4 Kịch bản bẻ gãy

- **`flood` trên OVN br-int:** như với `normal`, OVN không dùng. Bypass OVN pipeline có thể tạo loop qua Geneve tunnel vì remote chassis thấy gói tin flooded.
- **`all` loop risk:** vì include ingress port, gói tin có thể loop nếu không có STP. Tránh dùng `all` trên production trừ debug ngắn hạn.

## 4.9.6 Action `controller`, gửi packet_in lên OpenFlow controller

### 4.9.6.1 Template C

| Attribute | Giá trị |
|---|---|
| **Syntax** | `controller[:<max_len>]` hoặc `controller(reason=<x>,max_len=<n>,userdata=<hex>)` |
| **Category** | 1. Output (reserved port) |
| **OpenFlow version** | All. OF 1.3+ thêm parameter reason + userdata |
| **Prerequisites** | Controller connected (`ovs-vsctl set-controller br0 tcp:...`) |
| **Semantics** | Gửi gói tin như OFPT_PACKET_IN message tới controller |
| **Parameters** | `max_len`: bytes truncate (default 128 cho OF 1.0, 64K cho 1.3); `reason`: `action`, `no_match`, `invalid_ttl`, `action_set`, `group`, `packet_out`; `userdata`: opaque hex data |
| **Side effects** | Controller nhận gói tin, có thể respond với OFPT_PACKET_OUT để tiếp tục |
| **Conformance** | Đúng spec; common pattern cho learn-and-forward |

### 4.9.6.2 Use case và risk

```shell
# Ex: send DHCP packet lên controller
$ ovs-ofctl add-flow br0 \
    "priority=100,udp,tp_src=68,tp_dst=67,actions=controller(max_len=512)"

# Ex: table-miss default → controller
$ ovs-ofctl add-flow br0 "table=0,priority=0,actions=controller"
```

**Risk:** `controller` action spam rất dễ. Một bad rule có thể gửi toàn bộ traffic lên controller, overwhelm control channel. Luôn:

1. Dùng `max_len=128` (hoặc 256) để truncate payload.
2. Rate limit với `meter` action trước `controller`.
3. Theo dõi `ofconn.packets_in_sent` qua `coverage/show`.

### 4.9.6.3 Kịch bản bẻ gãy

- **Controller disconnect mà rule `actions=controller`** → gói tin queue nội bộ OVS đầy, eventual drop. Counter `rconn_queued` (Phần 9.4 §9.4.1.3 coverage) tăng.
- **Max_len=0** → packet_in chứa chỉ metadata (không payload); controller không thể ra quyết định dựa trên content.

## 4.9.7 Action `local`, output to bridge's internal port

### 4.9.7.1 Template C

| Attribute | Giá trị |
|---|---|
| **Syntax** | `local` hoặc `actions=LOCAL` |
| **Category** | 1. Output (reserved port) |
| **OpenFlow version** | All |
| **Prerequisites** | Bridge có internal port (LOCAL port tự động có cho mọi bridge, trùng tên bridge) |
| **Semantics** | Output gói tin vào local network stack của host qua LOCAL port (Linux netdev `br0`) |
| **Side effects** | Gói tin đi vào kernel rx queue của netdev `br0`, có thể bị iptables/tc xử lý tiếp |

### 4.9.7.2 Use case

Host OVS chạy TCP server trên IP gán lên LOCAL port:

```shell
# br-int có IP 10.1.0.1 gán lên netdev br-int (LOCAL port)
$ ip addr add 10.1.0.1/24 dev br-int

# Rule accept traffic destined 10.1.0.1 vào local stack
$ ovs-ofctl add-flow br-int "priority=100,ip,nw_dst=10.1.0.1,actions=local"
```

### 4.9.7.3 So sánh với `output:<LOCAL_port_number>`

```shell
$ ovs-ofctl show br-int | grep LOCAL
 LOCAL(br-int): addr:...
```

LOCAL port có port number reserved 0xfffe. `actions=local` và `actions=output:LOCAL` và `actions=output:65534` tương đương.

## 4.9.8 Action `in_port`, gửi ngược về ingress

### 4.9.8.1 Template C

| Attribute | Giá trị |
|---|---|
| **Syntax** | `in_port` hoặc `actions=IN_PORT` |
| **Category** | 1. Output (reserved port) |
| **OpenFlow version** | All |
| **Prerequisites** | None |
| **Semantics** | Output gói tin ra **ingress port** (gửi ngược về nơi gói tin đến). Classic ARP responder pattern |
| **Side effects** | Gói tin đi ngược qua cùng port đã nhận |

### 4.9.8.2 ARP responder pattern

```shell
# OVN-style ARP responder: match ARP request cho gateway IP
# Modify packet thành ARP reply
# Gửi ngược về ingress port (không broadcast)
$ ovs-ofctl add-flow br-int \
    "priority=200,arp,arp_op=1,arp_tpa=10.1.1.1,\
     actions=move:NXM_OF_ETH_SRC[]->NXM_OF_ETH_DST[],\
             mod_dl_src:aa:bb:cc:dd:ee:ff,\
             load:0x2->NXM_OF_ARP_OP[],\
             move:NXM_NX_ARP_SHA[]->NXM_NX_ARP_THA[],\
             move:NXM_OF_ARP_SPA[]->NXM_OF_ARP_TPA[],\
             load:0xaabbccddeeff->NXM_NX_ARP_SHA[],\
             load:0x0a010101->NXM_OF_ARP_SPA[],\
             IN_PORT"
```

`IN_PORT` cuối là reserved port. Dùng đây vì thông thường không có rule cho phép output bình thường back về ingress.

### 4.9.8.3 Khác biệt `in_port` action vs `in_port` match field

Đừng nhầm:
- `in_port=1` trong **match spec** — match field, "gói tin đến từ port 1" (Phần 4.8 §4.8.2.1).
- `actions=in_port` hoặc `output:IN_PORT` — **action**, "gửi gói tin ra port đã nhận".

Flow có thể có cả hai:
```
priority=200,in_port=1,arp,arp_op=1,actions=IN_PORT
```

## 4.9.9 Action `table`, đưa gói tin vào pipeline lookup lại

### 4.9.9.1 Template C

| Attribute | Giá trị |
|---|---|
| **Syntax** | `table` hoặc `output:TABLE` |
| **Category** | 1. Output (reserved port) |
| **OpenFlow version** | 1.0+ |
| **Prerequisites** | Có sense chỉ trong context PACKET_OUT (controller gửi gói tin vào switch) |
| **Semantics** | Submit gói tin vào OpenFlow pipeline bắt đầu từ table 0 |

### 4.9.9.2 Use case

`TABLE` hiếm dùng trong flow rule — chủ yếu trong PACKET_OUT message từ controller:

```
# Controller inject packet vào pipeline
OFPT_PACKET_OUT: buffer_id=-1, in_port=<x>, actions=OUTPUT:TABLE, data=<hex>
```

Khi controller nhận packet_in + quyết định sau đó "để switch xử lý bình thường", gửi gói tin ra TABLE thay vì chỉ định output port cụ thể.

Trong flow rule, dùng rất hiếm — thường `resubmit(,0)` hoặc `goto_table:N` là thay thế.

## 4.9.10 Action `group`, group table indirection

### 4.9.10.1 Template C

| Attribute | Giá trị |
|---|---|
| **Syntax** | `group:<group_id>` |
| **Category** | 1. Output (indirect) |
| **OpenFlow version** | 1.1+ |
| **Prerequisites** | Group phải được install trước qua `ovs-ofctl add-group` |
| **Semantics** | Execute action list trong group entry. Group có 4 types: all, select, indirect, fast_failover |
| **Parameters** | `group_id`: 32-bit ID của group |
| **Side effects** | Depending on group type — multicast (all), load-balance (select), shared indirection (indirect), active-backup (fast_failover) |

### 4.9.10.2 Bốn group type

| Type | Bucket count | Selection | Use case |
|---|---|---|---|
| `all` | 1+ | Execute ALL buckets | Multicast, broadcast efficient |
| `select` | 1+ | Pick 1 bucket by hash | Load balance (ECMP, LB backend) |
| `indirect` | 1 | Execute only bucket | Shared chuyển tiếp (next-hop indirection, reduce rule count) |
| `fast_failover` | 1+ | First "live" bucket | Active-backup chuyển tiếp với watch_port |

### 4.9.10.3 Template cho Group Type `select` (LB pattern)

```shell
# Install group 1: load balance giữa 3 backend
$ ovs-ofctl add-group br0 "group_id=1,type=select,\
  bucket=weight:100,actions=mod_dl_dst:aa:bb:cc:00:00:01,output:5,\
  bucket=weight:100,actions=mod_dl_dst:aa:bb:cc:00:00:02,output:6,\
  bucket=weight:100,actions=mod_dl_dst:aa:bb:cc:00:00:03,output:7"

# Rule: TCP destined VIP → group 1
$ ovs-ofctl add-flow br0 \
  "priority=100,tcp,nw_dst=10.1.0.100,tp_dst=80,actions=group:1"
```

Weight default là 1 nếu không chỉ định. Hash function default là 5-tuple (src_ip, dst_ip, src_port, dst_port, proto).

### 4.9.10.4 Template Group Type `fast_failover` (link backup)

```shell
# Group 2: primary eth0, backup eth1
$ ovs-ofctl add-group br0 "group_id=2,type=fast_failover,\
  bucket=watch_port:eth0,actions=output:eth0,\
  bucket=watch_port:eth1,actions=output:eth1"

# Rule dùng group
$ ovs-ofctl add-flow br0 "priority=100,actions=group:2"
```

OVS theo dõi `watch_port` state; nếu eth0 down → switch sang bucket 2 (eth1) < 50ms (với BFD).

### 4.9.10.5 `dump-groups` để kiểm chứng

```shell
$ ovs-ofctl -O OpenFlow13 dump-groups br0

  bucket=weight:100,actions=mod_dl_dst:aa:bb:cc:00:00:01,output:5,
  bucket=weight:100,actions=mod_dl_dst:aa:bb:cc:00:00:02,output:6,
  bucket=weight:100,actions=mod_dl_dst:aa:bb:cc:00:00:03,output:7

 group_id=2,type=fast_failover,
  bucket=watch_port:eth0,actions=output:eth0,
  bucket=watch_port:eth1,actions=output:eth1
```

## 4.9.11 Action `resubmit`, re-lookup qua table hiện tại hoặc khác

### 4.9.11.1 Template C

| Attribute | Giá trị |
|---|---|
| **Syntax** | `resubmit:<port>` hoặc `resubmit(<port>,<table>)` |
| **Category** | 6. Control/Pipeline |
| **OpenFlow version** | NXM extension (OVS specific) |
| **Prerequisites** | None |
| **Semantics** | Tạo packet clone với virtual `in_port` mới + optional table khác, lookup trong pipeline như packet mới. Action set chia sẻ giữa nested resubmit |
| **Parameters** | `port`: virtual in_port cho resubmit (default current); `table`: target table (default current) |
| **Side effects** | Stack depth tối đa 16 (tránh infinite recursion); execute action từ rule match trong resubmitted lookup |

### 4.9.11.2 So sánh `resubmit` vs `goto_table`

| Aspect | `goto_table:N` | `resubmit(,N)` |
|---|---|---|
| OpenFlow standard | Có (1.1+) | Không (NXM) |
| Direction | Forward only (N > current) | Bất kỳ (forward hoặc backward) |
| Action set | Cumulative (write_actions accumulate) | Shared giữa nested call |
| Use case | Linear pipeline | Modular/shared table |

OVS prefer `goto_table` cho OpenFlow standard pipeline; `resubmit` cho OVS-specific pattern như nested classifier.

### 4.9.11.3 Use case OVN

OVN dùng `resubmit` extensively trong translation. Ví dụ từ OVN logical flow:

```
# Logical: chuyển đến table sau
table=8(ls_in_lookup_fdb), priority=0, match=1, action=next;

# Physical: resubmit
table=8,priority=0,actions=resubmit(,9)
```

Xem Phần 13.7 cho OVN logical-to-physical translation dùng `resubmit`.

### 4.9.11.4 Kịch bản bẻ gãy

- **Resubmit loop >16 depth** → OVS abort với "resubmit recursion too deep", gói tin drop. Thường do classifier loop lỗi. Check trace để thấy path.
- **Shared action set trap:** `apply_actions(output:5)` rồi `resubmit` — action set được share, nested lookup có thể override. Best practice: dùng `write_actions` thay vì `apply_actions` khi kết hợp resubmit.

## 4.9.12 Action `clone`, execute action tạm nhưng không commit modification

### 4.9.12.1 Template C

| Attribute | Giá trị |
|---|---|
| **Syntax** | `clone(<action_list>)` |
| **Category** | 6. Control/Pipeline |
| **OpenFlow version** | NXM extension |
| **Prerequisites** | None |
| **Semantics** | Execute action list với gói tin clone tạm. Sau khi clone block kết thúc, gói tin original không bị modify bởi action trong clone |
| **Side effects** | Action có side effect (output, learn, ct(commit)) vẫn thực thi; chỉ gói tin header/metadata được restore |

### 4.9.12.2 Use case

```shell
# Output packet copy ra port 99 (span/mirror) mà không ảnh hưởng packet chính
$ ovs-ofctl add-flow br0 \
    "priority=100,in_port=1,actions=clone(output:99),output:2"
```

Gói tin ra port 2 giữ nguyên state. Clone output port 99 có thể có action modify trước output (mod_dl_dst) mà không modify gói tin đi ra port 2.

### 4.9.12.3 Khác biệt với explicit multi-output

```
actions=output:99,output:2
# Packet A gửi ra 99, packet A' (sẽ output:2) vẫn nguyên

actions=clone(mod_dl_dst:aa:aa:..,output:99),output:2
# Packet A' (cho output:2) không bị mod_dl_dst
```

Clone là cách sạch để forward gói tin modified mà không forward pollute.

## 4.9.14 Action Set execution order, 12 priority level

Khi gói tin exit pipeline (không còn `Goto-Table`), action set tích lũy được execute theo **12 priority level** (OpenFlow 1.3 §5.10, table 9):

| Priority | Action | Ý nghĩa |
|---|---|---|
| 1 | `copy_ttl_in` | Copy outer TTL vào inner (IP-in-IP/MPLS decap prep) |
| 2 | `pop_mpls`, `pop_pbb`, `pop_vlan` | Decapsulate |
| 3 | `push_pbb` | PBB encap |
| 4 | `push_mpls` | MPLS encap |
| 5 | `push_vlan` | VLAN encap |
| 6 | `copy_ttl_out` | Copy inner TTL vào outer |
| 7 | `dec_mpls_ttl`, `dec_nw_ttl` | Decrement TTL |
| 8 | `set_field` (tất cả), `mod_*` | Field modification |
| 9 | `set_queue` | Queue assignment |
| 10 | `group` | Group indirection |
| 11 | `output` | Output port |
| 12 | (cleanup action set) | Clear + exit |

Ý nghĩa: dù thứ tự ghi action trong `write_actions(...)` thế nào, khi exit pipeline execute **theo priority**.

### 4.9.14.1 Ví dụ

```
write_actions(output:5, mod_dl_dst:aa:aa:aa:aa:aa:aa, push_vlan:0x8100)
```

Ghi: output đầu, mod sau, push cuối. Execute theo priority:
- L5 push_vlan
- L8 mod_dl_dst
- L11 output

→ gói tin: push VLAN, mod MAC, output ra port 5. Dù thứ tự ghi ngược lại.

### 4.9.14.2 Apply-Actions không theo priority

`apply_actions=` execute **tuần tự theo thứ tự ghi**. Khác biệt quan trọng khi đọc flow spec:

```
apply_actions(mod_dl_dst:aa..., push_vlan:0x8100, output:5)
# Sequential: mod, then push, then output
# Packet ra port 5 có VLAN + MAC mới
```

Rule of thumb OVN: **OVN logical flow dùng apply_actions** chủ yếu (sequential order dễ đoán hơn). `write_actions` dùng cho pipeline egress consolidation.

## 4.9.15 Category 2, Encapsulation actions — VLAN tagging

### 4.9.15.1 `push_vlan`

| Attribute | Giá trị |
|---|---|
| **Syntax** | `push_vlan:<ethertype>` (ví dụ `push_vlan:0x8100` cho 802.1Q, `0x88a8` cho 802.1ad Q-in-Q) |
| **Category** | 2. Encap/Decap |
| **OpenFlow version** | 1.1+ |
| **Prerequisites** | None |
| **Semantics** | Insert VLAN header sau Ethernet header. Default VID = 0, PCP = 0 — phải theo sau `mod_vlan_vid` hoặc `set_field:<vid>->vlan_vid` để set VLAN ID cụ thể |
| **Parameters** | `ethertype`: 0x8100 (VLAN) hoặc 0x88a8 (PBB/Q-in-Q) |
| **Side effects** | Tăng gói tin length 4 byte; set `vlan_tci` có Present bit |

### 4.9.15.2 `pop_vlan`

| Attribute | Giá trị |
|---|---|
| **Syntax** | `pop_vlan` hoặc `strip_vlan` (OVS alias) |
| **Category** | 2. Encap/Decap |
| **OpenFlow version** | 1.1+ |
| **Prerequisites** | Gói tin có VLAN tag (`vlan_tci[12]=1` Present bit) |
| **Semantics** | Remove VLAN header trên cùng |
| **Side effects** | Giảm gói tin length 4 byte; `vlan_tci` về 0 |

### 4.9.15.3 Ví dụ tagging workflow

```shell
# Access port → trunk port: tag packet với VLAN 100
$ ovs-ofctl add-flow br0 \
    "priority=100,in_port=1,vlan_tci=0,\
     actions=push_vlan:0x8100,mod_vlan_vid:100,output:2"

# Trunk port → access port: untag
$ ovs-ofctl add-flow br0 \
    "priority=100,in_port=2,dl_vlan=100,\
     actions=strip_vlan,output:1"

# Q-in-Q double tagging
$ ovs-ofctl add-flow br0 \
    "priority=100,in_port=3,\
     actions=push_vlan:0x88a8,mod_vlan_vid:500,\
             push_vlan:0x8100,mod_vlan_vid:100,\
             output:4"
```

### 4.9.15.4 Kịch bản bẻ gãy

- **`push_vlan` nhưng không set VID** → gói tin có VLAN header với VID=0 (reserved cho priority tagging, thường bị nhiều switch hardware drop). Luôn pair với `mod_vlan_vid` hoặc `set_field`.
- **`pop_vlan` trên gói tin không tag** → silently skip (no error); kiểm chứng với `vlan_tci=0x1000/0x1000` match trước.

## 4.9.16 Category 2, MPLS + PBB encapsulation

### 4.9.16.1 `push_mpls`, `pop_mpls`

| Attribute | `push_mpls` | `pop_mpls` |
|---|---|---|
| **Syntax** | `push_mpls:<ethertype>` (0x8847 unicast, 0x8848 multicast) | `pop_mpls:<ethertype>` (inner protocol) |
| **OpenFlow version** | 1.1+ | 1.1+ |
| **Semantics** | Insert MPLS label trên cùng | Remove outermost MPLS label, expose inner |
| **Side effects** | Gói tin length +4 byte; `mpls_label/tc/bos/ttl` cần set riêng | Gói tin length -4 byte |

### 4.9.16.2 Ví dụ MPLS label push

```shell
# L3 VPN: push MPLS label 100 trên top, eth_type chuyển 0x8847
$ ovs-ofctl add-flow br0 \
    "priority=100,ip,nw_dst=10.1.0.0/24,\
     actions=push_mpls:0x8847,\
             set_field:100->mpls_label,\
             set_field:1->mpls_bos,\
             set_field:64->mpls_ttl,\
             output:uplink"

# MPLS pop tại PE:
$ ovs-ofctl add-flow br0 \
    "priority=100,mpls,mpls_label=100,\
     actions=pop_mpls:0x0800,\
             output:ce_port"
```

### 4.9.16.3 `push_pbb`, `pop_pbb` (Provider Backbone Bridge)

Tương tự MPLS nhưng cho 802.1ah PBB. Dùng rất ít ngoài MEF metro Ethernet. Skip deep-dive vì scope hyperscale DC không cần.

### 4.9.16.4 `encap`, `decap` (generic, OF 1.5 + NSH)

OpenFlow 1.5 thêm generic `encap`/`decap` cho header chain (NSH, GRE, VXLAN-GPE) không cứng VLAN/MPLS/PBB. Ít khi dùng trong OVN; OVN encap Geneve/VXLAN qua **tunnel port** config chứ không qua action.

## 4.9.17 Category 3, `set_field` — generic field rewrite

### 4.9.17.1 Template C

| Attribute | Giá trị |
|---|---|
| **Syntax** | `set_field:<value>[/<mask>]-><field>` hoặc `set_field(<field>=<value>)` |
| **Category** | 3. Field modification |
| **OpenFlow version** | 1.2+ (OXM TLV required) |
| **Prerequisites** | Field-specific (ví dụ `eth_type=0x0800` cho IP field) |
| **Semantics** | Generic field rewrite — thay thế tất cả `mod_*` specific action. Hỗ trợ mask cho partial write |
| **Parameters** | `value`: giá trị mới (hex, decimal, MAC, IP); `mask`: optional bitmask; `field`: OXM/NXM field name |
| **Side effects** | Modify gói tin header in-place; checksum auto-recompute cho L3/L4 field |

### 4.9.17.2 Ví dụ set_field rewrite phổ biến

```shell
# Rewrite MAC destination (replacement cho mod_dl_dst)
$ ovs-ofctl add-flow br0 \
    "priority=100,ip,nw_dst=10.1.0.1,\
     actions=set_field:aa:bb:cc:00:01:01->eth_dst,output:5"

# Rewrite IP destination (DNAT semantics)
$ ovs-ofctl add-flow br0 \
    "priority=100,tcp,nw_dst=10.1.0.100,tp_dst=80,\
     actions=set_field:10.2.0.10->ipv4_dst,\
             set_field:8080->tcp_dst,\
             output:backend_port"

# Partial bit write với mask (chỉ đổi upper 3 byte MAC = OUI)
$ ovs-ofctl add-flow br0 \
    "priority=100,in_port=1,\
     actions=set_field:aa:bb:cc:00:00:00/ff:ff:ff:00:00:00->eth_src,\
             output:2"

# DSCP marking
$ ovs-ofctl add-flow br0 \
    "priority=100,ip,tcp,tp_dst=5001,\
     actions=set_field:46->ip_dscp,output:uplink"
```

### 4.9.17.3 `set_field` vs legacy `mod_*`

| Aspect | `set_field` | `mod_<field>` |
|---|---|---|
| OpenFlow version | 1.2+ | 1.0 legacy |
| Syntax | Unified | Per-field |
| Mask support | Có | Không |
| New field | Tự động support | Phải thêm mỗi field mới |

Best practice: **dùng `set_field` cho code mới**. Legacy `mod_dl_src/dst`, `mod_nw_src/dst`, `mod_tp_src/dst` vẫn work nhưng không scale cho field mới (tun_metadata, ct_mark, register, v.v.) — những field này không có `mod_*` riêng, phải dùng `set_field` hoặc `load`.

### 4.9.18.2 `dec_ttl`, `dec_mpls_ttl`

| Attribute | Giá trị |
|---|---|
| **Syntax** | `dec_ttl`, `dec_mpls_ttl` |
| **Category** | 3. Field modification |
| **OpenFlow version** | 1.0+ (`dec_ttl`), 1.1+ (`dec_mpls_ttl`) |
| **Prerequisites** | IP gói tin với TTL > 0 cho `dec_ttl` |
| **Semantics** | Giảm TTL/HL 1 đơn vị. Nếu TTL = 0 sau decrement → **gói tin gửi lên controller qua packet_in** (không drop tự động); controller quyết định |
| **Side effects** | Update TTL; recompute IP checksum; nếu TTL=0 → OFPT_PACKET_IN với reason `OFPR_INVALID_TTL` |

**Router function dùng `dec_ttl`:**

```shell
# OVS thành virtual router
$ ovs-ofctl add-flow br0 \
    "priority=100,ip,nw_dst=10.2.0.0/24,\
     actions=dec_ttl,\
             mod_dl_src:aa:bb:cc:00:00:01,\
             mod_dl_dst:aa:bb:cc:00:02:01,\
             output:uplink"

# Table miss send ICMP TTL exceeded — via controller
$ ovs-ofctl add-flow br0 \
    "priority=0,dl_type=0x0800,actions=controller(reason=invalid_ttl)"
```

### 4.9.18.3 `copy_ttl_in`, `copy_ttl_out`

MPLS/IP stacking: khi push/pop MPLS label, `copy_ttl_in` copy outer TTL vào inner, `copy_ttl_out` copy ngược lại. Dùng trong MPLS VPN để preserve TTL qua tunnel.

## 4.9.19 Category 3, `move` + `load` — register manipulation

### 4.9.19.1 `move`, copy field-to-field

| Attribute | Giá trị |
|---|---|
| **Syntax** | `move:<src_field>[<start>..<end>]-><dst_field>[<start>..<end>]` |
| **Category** | 3. Field modification |
| **OpenFlow version** | NXM extension |
| **Semantics** | Copy bits từ src field vào dst field. Bit-range optional — nếu thiếu copy full field |
| **Example** | `move:NXM_OF_ETH_SRC[]->NXM_OF_ETH_DST[]` copy eth_src → eth_dst |

### 4.9.19.2 `load`, constant-to-field

| Attribute | Giá trị |
|---|---|
| **Syntax** | `load:<value>-><field>[<start>..<end>]` |
| **Category** | 3. Field modification |
| **OpenFlow version** | NXM extension |
| **Semantics** | Load hằng số vào field (hoặc bit-range) |
| **Example** | `load:0x42->NXM_NX_REG0[]` |

### 4.9.19.3 Use case ARP responder (kết nối với Phần 4.8 §4.8.5)

```shell
$ ovs-ofctl add-flow br-int \
    "priority=200,arp,arp_op=1,arp_tpa=10.1.1.1,\
     actions=move:NXM_OF_ETH_SRC[]->NXM_OF_ETH_DST[],\
             mod_dl_src:aa:bb:cc:dd:ee:ff,\
             load:0x2->NXM_OF_ARP_OP[],\
             move:NXM_NX_ARP_SHA[]->NXM_NX_ARP_THA[],\
             move:NXM_OF_ARP_SPA[]->NXM_OF_ARP_TPA[],\
             load:0xaabbccddeeff->NXM_NX_ARP_SHA[],\
             load:0x0a010101->NXM_OF_ARP_SPA[],\
             IN_PORT"
```

Bảy action consecutive: `move` copy field, `load` write constant, `IN_PORT` output ngược. Đây là cách classic implement ARP responder trong OVS — không cần gửi gói tin lên controller.

### 4.9.19.4 Bit-range manipulation

```shell
# Set chỉ bit 0-3 của reg0 thành 0xa
$ ovs-ofctl add-flow br0 \
    "priority=100,in_port=1,\
     actions=load:0xa->NXM_NX_REG0[0..3],resubmit(,1)"

# Move bit 12-15 của metadata vào reg1[0..3]
$ ovs-ofctl add-flow br0 \
    "priority=100,actions=move:OXM_OF_METADATA[12..15]->NXM_NX_REG1[0..3]"
```

Bit-range syntax `[<start>..<end>]` hoặc `[<bit>]`. Dùng extensively trong OVN cho chia register thành subfield (ví dụ `reg0[0]` = ACL hint flag, `reg0[1]` = committed flag).

## 4.9.20 Category 4, Metadata actions

### 4.9.20.1 `write_metadata` (instruction, không action)

Strictly đây là **instruction** (Phần §4.9.1.1), không phải action. Nhưng OVS cho phép viết trong `actions=` shorthand. Syntax:

```
actions=write_metadata:<value>[/<mask>]
```

Scope: set metadata field pass qua `Goto-Table`. Khác với register (lost sau recirc hoặc upcall boundary), metadata preserve xuyên pipeline.

### 4.9.20.2 `set_tunnel`, `set_tunnel64`

| **Semantics** | Set `tun_id` (VNI cho VXLAN, Geneve) | - |
| **Use case** | VXLAN VNI 24-bit encap | Generic 64-bit |

```shell

    "priority=100,in_port=1,\
     actions=set_tunnel:100,output:vxlan0"
```

Modern flow: dùng `set_field:<vni>->tun_id` thay vì `set_tunnel`.

## 4.9.23 Category 5, `ct()` — conntrack integration

### 4.9.23.1 Template C cho `ct()`

| Attribute | Giá trị |
|---|---|
| **Syntax** | `ct(<option>[,<option>...])` |
| **Category** | 5. Firewall/CT |
| **OpenFlow version** | NXM extension (OVS) |
| **Prerequisites** | Kernel datapath có conntrack support (`dpif/show-dp-features: Conntrack eventmask: Yes`) |
| **Semantics** | Submit gói tin vào kernel conntrack module. Sau khi conntrack process, gói tin recirculate lại pipeline với `ct_state` + `ct_zone` + `ct_mark` + `ct_label` populated |
| **Parameters** | `commit`, `zone=N`, `nat`, `force`, `alg=<ftp/tftp/sip/...>`, `exec(<actions>)`, `table=<N>` |
| **Side effects** | Rất lớn: tạo conntrack entry (commit), NAT rewrite (nat), trigger ALG helper, recirc gói tin |

### 4.9.23.2 Option của `ct()`

**`commit`** — tạo entry trong conntrack table:

```
actions=ct(commit,zone=1)
# Đăng ký connection này vào kernel conntrack table zone 1
# Subsequent packet cùng 5-tuple sẽ match conntrack entry
```

**`zone=N`** — conntrack zone ID (16-bit), cô lập namespace giữa OVN logical switch. Không specify zone → zone 0 default.

**`nat`** — apply NAT:

```
# Source NAT (SNAT)
actions=ct(commit,nat(src=10.1.0.1))

# Destination NAT (DNAT)
actions=ct(commit,nat(dst=10.2.0.1:8080))

# NAT với port range
actions=ct(commit,nat(src=10.1.0.1:30000-40000))

# NAT persistence: reuse same mapping cho same client
actions=ct(commit,nat(src=10.1.0.1,persistent))
```

**`force`** — re-commit, override existing entry. Dùng khi connection expected đã stale.

**`alg=<name>`** — application layer gateway helper:

| ALG | Protocol |
|---|---|
| `alg=ftp` | FTP control channel (parse PORT/PASV để open data channel dynamic) |
| `alg=tftp` | TFTP |
| `alg=sip` | SIP (VoIP signaling → RTP media channel) |

**`exec(<actions>)`** — execute action list **trong conntrack zone context**. Dùng set ct_mark / ct_label khi commit:

```
actions=ct(commit,zone=1,exec(set_field:0xab->ct_mark))
```

**`table=N`** — sau recirculate, lookup table N (default = table hiện tại).

### 4.9.23.3 Typical stateful firewall pattern

```shell
# Step 1: Mọi packet qua ct (populate ct_state)
# Table 0 — dispatcher
$ ovs-ofctl add-flow br0 \
    "table=0,priority=100,ip,actions=ct(table=1,zone=1)"

# Table 1 — policy based on ct_state
# Allow new outbound + commit
$ ovs-ofctl add-flow br0 \
    "table=1,priority=200,ip,ct_state=+new+trk,\
     actions=ct(commit,zone=1),output:uplink"

# Allow reply (established)
$ ovs-ofctl add-flow br0 \
    "table=1,priority=100,ip,ct_state=+est+trk,actions=output:inside"

# Drop invalid
$ ovs-ofctl add-flow br0 \
    "table=1,priority=100,ip,ct_state=+inv+trk,actions=drop"

# Default drop
$ ovs-ofctl add-flow br0 \
    "table=1,priority=0,actions=drop"
```

Gói tin flow:
1. Gói tin đến table 0 → `ct(table=1)` → submit lên kernel conntrack.
2. Kernel populate `ct_state` (new, est, rel, inv) + recirc.
3. Gói tin reappear ở table 1 với ct_state populated → match rule.
4. Rule có `ct(commit)` thì conntrack entry persist.

### 4.9.23.4 `ct_clear`

| Attribute | Giá trị |
|---|---|
| **Syntax** | `ct_clear` |
| **Semantics** | Clear conntrack tracked state cho gói tin hiện tại (reset `ct_state` về unset). Gói tin tiếp theo qua `ct()` sẽ re-evaluate |
| **Use case** | Khi pipeline muốn re-track connection (ví dụ sau NAT, reclassify) |

Xem Phần 9.24 conntrack stateful firewall cho case study đầy đủ.

## 4.9.24 Category 6, `learn()` — MAC learning via flow install

### 4.9.24.1 Template C cho `learn()`

| Attribute | Giá trị |
|---|---|
| **Syntax** | `learn(<option>[,<option>...])` |
| **Category** | 6. Control/Pipeline |
| **OpenFlow version** | NXM extension |
| **Prerequisites** | None |
| **Semantics** | Install flow rule mới dynamic dựa trên gói tin hiện tại. Đây là cách OVS implement L2 MAC learning mà không cần controller — flow "self-programming" |
| **Parameters** | `table=N`, `priority=P`, `idle_timeout=T`, `hard_timeout=T`, `fin_idle_timeout=T`, `NXM_FIELD[]` match/action templates |
| **Side effects** | Tạo flow rule mới trong kernel flow table — counter `n_flow_mods` tăng |

### 4.9.24.2 MAC learning example — classic OVS pattern

```shell
# Learning rule: khi packet đến, install reverse flow
$ ovs-ofctl add-flow br0 \
    "table=0,priority=100,\
     actions=learn(table=1,\
                    idle_timeout=60,\
                    NXM_OF_ETH_DST[]=NXM_OF_ETH_SRC[],\
                    load:NXM_OF_IN_PORT[]->NXM_NX_REG0[]),\
             resubmit(,1)"

# Forwarding based on learned entries
$ ovs-ofctl add-flow br0 \
    "table=1,priority=50,\
     actions=output:NXM_NX_REG0[]"

# Fallback: flood if no learn hit
$ ovs-ofctl add-flow br0 "table=1,priority=0,actions=flood"
```

**Ngữ nghĩa learn rule:**

- `table=1`: install flow vào table 1.
- `idle_timeout=60`: flow expire sau 60s inactive.
- `NXM_OF_ETH_DST[]=NXM_OF_ETH_SRC[]`: match spec mới — "dst MAC = src MAC của gói tin hiện tại".
- `load:NXM_OF_IN_PORT[]->NXM_NX_REG0[]`: action — load ingress port vào reg0.

Khi gói tin (src=MAC_A, in_port=1) đến table 0, `learn` tạo flow:

```
table=1,priority=32768,idle_timeout=60,dl_dst=MAC_A,actions=load:1->NXM_NX_REG0[]
```

Gói tin sau với dst=MAC_A → match flow này → reg0=1 → table 1 priority 50 → output:1.

### 4.9.24.3 `fin_idle_timeout`

Option đặc biệt cho TCP: timeout khác khi thấy FIN gói tin (connection đóng):

```
actions=learn(...,idle_timeout=300,fin_idle_timeout=10)
```

Normal flow expire 300s idle, nhưng sau FIN thấy → timeout 10s.

### 4.9.24.4 Risk của learn()

- Mỗi gói tin match rule có `learn()` tạo flow mới → flow explosion nếu misconfig.
- `coverage/show upcall_flow_add` sẽ rất cao.
- Install trên bridge production phải rate limit carefully.

## 4.9.25 Category 6, `conjunction()` — cross-product compression

### 4.9.25.1 Template C

| Attribute | Giá trị |
|---|---|
| **Syntax** | `conjunction(<id>,<component>/<total>)` |
| **Category** | 6. Control/Pipeline |
| **OpenFlow version** | NXM extension |
| **Prerequisites** | None |
| **Semantics** | Khai báo rule là một "component" của conjunction group `id`. Gói tin cần match `<total>` component khác nhau của cùng `id` → `conj_id` field set → rule priority khác match để execute action |
| **Parameters** | `id`: 32-bit conjunction group ID; `component`: 1-N, thứ tự component; `total`: tổng số component cần match |

### 4.9.25.2 Vấn đề cross-product

ACL rule dạng "source IN {A, B, C} AND dest IN {X, Y, Z} AND port IN {80, 443}" cần **3 × 3 × 2 = 18 flow rule** nếu viết explicit.

Với conjunction: **3 + 3 + 2 + 1 = 9 rule** (3 source component + 3 dest component + 2 port component + 1 final action).

### 4.9.25.3 Ví dụ đầy đủ

```shell
# Source components (conjunction 42, component 1/3)
$ for src in 10.0.1.1 10.0.1.2 10.0.1.3; do
    sudo ovs-ofctl add-flow br0 \
      "table=0,priority=100,ip,nw_src=$src,actions=conjunction(42,1/3)"
  done

# Destination components (42, component 2/3)
$ for dst in 10.0.2.1 10.0.2.2 10.0.2.3; do
    sudo ovs-ofctl add-flow br0 \
      "table=0,priority=100,ip,nw_dst=$dst,actions=conjunction(42,2/3)"
  done

# Port components (42, component 3/3)
$ for port in 80 443; do
    sudo ovs-ofctl add-flow br0 \
      "table=0,priority=100,ip,tcp,tp_dst=$port,actions=conjunction(42,3/3)"
  done

# Final rule match conj_id (set when all 3 components match)
$ sudo ovs-ofctl add-flow br0 \
    "table=0,priority=100,conj_id=42,actions=output:5"
```

### 4.9.25.4 OVN Port_Group dùng conjunction

OVN ACL từ Logical_Switch `Port_Group` extensively dùng conjunction. Ví dụ policy "port_group WEB (10 port) → port_group DB (5 port) allow TCP 3306":

- 10 rule cho WEB source (conj component 1/2)
- 5 rule cho DB dest (conj component 2/2)
- 1 rule final match `conj_id`

Total 16 rule thay vì 50 rule explicit. Xem Phần 13.3 OVN ACL + Load Balancer cho context OVN.

## 4.9.26 Category 6, `multipath()` — hash-based path selection

### 4.9.26.1 Template C

| Attribute | Giá trị |
|---|---|
| **Syntax** | `multipath(<fields>,<basis>,<algorithm>,<n_links>,<arg>,<dst>[<start>..<end>])` |
| **Category** | 6. Control/Pipeline |
| **OpenFlow version** | NXM extension |
| **Semantics** | Hash gói tin header fields → output ID 0 đến (n_links-1) → load vào dst field. Tương tự group type=select nhưng không cần group table config |
| **Parameters** | `fields`: `eth_src`, `symmetric_l4`, v.v.; `algorithm`: `modulo_n`, `hash_threshold`, `hrw`, `iter_hash`; `n_links`: số output; `dst`: target field |

### 4.9.26.2 Use case ECMP

```shell
# Hash 5-tuple, modulo 4 → load 0-3 vào reg0
$ ovs-ofctl add-flow br0 \
    "priority=100,ip,actions=multipath(symmetric_l4,0,modulo_n,4,0,NXM_NX_REG0[0..1]),\
     resubmit(,1)"

# Table 1 output dựa trên reg0
$ ovs-ofctl add-flow br0 "table=1,reg0=0,actions=output:uplink1"
$ ovs-ofctl add-flow br0 "table=1,reg0=1,actions=output:uplink2"
$ ovs-ofctl add-flow br0 "table=1,reg0=2,actions=output:uplink3"
$ ovs-ofctl add-flow br0 "table=1,reg0=3,actions=output:uplink4"
```

So với `group type=select`:
- `multipath` lightweight, không cần group table maintenance.
- `group select` có monitoring (group stats), centralized change management.

### 4.9.26.3 Hash algorithm

| Algorithm | Ý nghĩa |
|---|---|
| `modulo_n` | hash mod n_links — simple nhưng rebalance lớn khi n_links thay đổi |
| `hash_threshold` | Threshold-based, balance tốt hơn modulo |
| `hrw` | Highest Random Weight (HRW) — minimal rebalance khi n_links thay đổi |
| `iter_hash` | Iterative hash — complex, dùng cho large n_links |

## Điểm cốt lõi cần nhớ (tier 1+2+3 hoàn thành)

Thứ nhất, action là chỉ thị thực thi trên gói tin (transform, forward); instruction là wrapper điều khiển pipeline. OpenFlow 1.0 gộp chung, 1.1+ tách riêng. OVS shortcut `actions=` mặc định = Apply-Actions.

Thứ hai, category Output gồm 10 action cơ bản: `output:<port>` + 9 reserved port variants (drop/normal/flood/all/controller/local/in_port/table) + `group`. Mỗi cái có semantics riêng (include ingress, respect STP, respect VLAN).

Thứ ba, explicit `drop` tốt hơn implicit (empty action list) vì tường minh. OVS 2.17+ có dedicated kernel drop path — verify qua `dpif/show-dp-features`.

Thứ tư, `group` table cung cấp indirection cho 4 pattern: multicast (all), load balance (select), shared next-hop (indirect), active-backup (fast_failover). OpenFlow 1.1+ feature foundational mà nhiều deployment chưa dùng đủ.

Thứ năm, `resubmit` là OVS NXM extension cho modular pipeline, khác `goto_table` standard. OVN translation dùng extensively.

Thứ sáu, action set execution order theo **12 priority level** cứng, không theo thứ tự ghi. Apply-Actions thì tuần tự. Hiểu khác biệt này để không bị surprise khi debug.

## Bối cảnh: SDN không có nghĩa duy nhất

Phần 3-4 đã phân tích OpenFlow như protocol thay thế data plane chuyển tiếp decisions, đi từ vendor-proprietary đến centralized programmable. Phần 4.6 đã tổng kết 5 giới hạn của OpenFlow và 3 hướng evolution thế hệ 2. Phần này đi sâu vào một trong 3 hướng: **API-based SDN**, mô hình giữ nguyên silicon và firmware của switch, chỉ phơi bày **API chuẩn** để automation tool control from outside.

Cốt lõi triết lý: vendor không muốn rebuild firmware để triển khai OpenFlow; người vận hành không muốn học new chuyển tiếp model khi already có vendor-specific ASIC hoạt động. API-based SDN là compromise, switch vẫn làm routing/switching theo native logic (OSPF/BGP/STP), controller gửi config qua standardized protocol (NETCONF/YANG, gNMI) để change state. Đây là cách tiếp cận dominant trong WAN, service provider, và enterprise edge, ngược với OpenFlow dominant trong DC virtualization.

## 5.0.1 Động lực phía vendor: giữ silicon, thêm tầng API

### Sự đánh đổi về mặt kiến trúc

OpenFlow chọn **replace data plane**: switch từ bỏ native chuyển tiếp, accept FLOW_MOD từ controller. Vendor phải rewrite firmware, silicon phải hỗ trợ OF pipeline abstraction.

SDN via APIs chọn **keep data plane**: switch vẫn chạy OSPF/BGP/STP + proprietary chuyển tiếp pipeline. Thay đổi duy nhất là thêm API endpoint cho config management. Switch phơi bày YANG model, controller send config qua NETCONF/RESTCONF/gNMI, switch apply qua native daemons.

### Table so sánh sự đánh đổi

| Aspect | OpenFlow-based | API-based |
|--------|----------------|-----------|
| Data plane | Replaced by flow tables | Giữ nguyên vendor-specific |
| Switch firmware | Rewrite major | Thêm NETCONF server module |
| Tính linh hoạt | High (arbitrary flow) | Low (constrained by vendor model) |
| Stability | Low (spec evolving) | High (protocols stable 10+ years) |
| Vendor sự áp dụng | Niche | Broad (Cisco, Juniper, Arista all support) |
| Use case dominant | DC virtualization | WAN, service provider, enterprise edge |

### Hiểu sai thường gặp

**Hiểu sai:** "SDN means OpenFlow. API-based configuration không phải SDN thực sự."

**Thực tế:** SDN là umbrella concept về separation of control plane + data plane, không phải protocol cụ thể. IETF I2RS WG, ONF, và Göransson textbook đều classify API-based (NETCONF/YANG/gNMI) là SDN legitimate. Argument ngược coi API-based không phải SDN đã outdated từ 2014 trở đi, khi Google B4 (Phần 4.6.6), Cisco ACI (2013), Juniper Contrail (2013) tất cả đều use API-based cách tiếp cận và deploy production scale.

## 5.0.2 NETCONF (RFC 6241, June 2011)

### Ngăn xếp giao thức

NETCONF là XML-based configuration protocol cho network device. Released IETF RFC 6241 tháng 6/2011, cùng thời điểm ONF formation (tháng 3/2011), hai effort parallel solving SDN pain point different way.

Transport:
- **SSH port 830**: default, secure.
- **TLS**: alternative cho environment không deploy SSH.
- **BEEP, SOAP**: historical, ít dùng 2026.

Message format: XML qua RPC (Remote Procedure Call) pattern:

```xml
<rpc message-id="101" xmlns="urn:ietf:params:xml:ns:netconf:base:1.0">
  <get-config>
    <source><running/></source>
    <filter type="subtree">
      <interfaces xmlns="urn:ietf:params:xml:ns:yang:ietf-interfaces"/>
    </filter>
  </get-config>
</rpc>
```

Client (controller) gửi RPC, server (switch) reply với XML response.

### Ba datastore (§5 RFC 6241)

NETCONF define 3 logical datastore:

| Datastore | Mô tả |
|-----------|-------|
| `running` | Config active đang chạy |
| `startup` | Config persisted, load khi boot |
| `candidate` | Edit-in-progress, chưa commit |

Pattern workflow:

1. Client `get-config source=running` lấy snapshot hiện tại.
2. Compute diff + generate edit operations.
3. `edit-config target=candidate` apply edit vào candidate (isolated, không affect running).
4. `validate candidate` check syntax + semantic.
5. `commit` promote candidate → running atomically.
6. Optional: `confirmed-commit timeout=N` cho rollback tự động nếu client không follow-up `commit` trong N giây (safety net cho misconfigured network khiến client disconnect).

### Bộ thao tác

| Operation | Mô tả |
|-----------|-------|
| `get` | Retrieve config + state data |
| `get-config` | Retrieve chỉ config |
| `edit-config` | Modify config (merge/replace/create/delete) |
| `copy-config` | Copy entire datastore |
| `lock`/`unlock` | Mutex cho datastore (single writer) |
| `commit` | Promote candidate → running |
| `confirmed-commit` | Commit với timeout rollback |
| `validate` | Check config validity |
| `close-session`/`kill-session` | Graceful / forceful termination |

### Ngữ nghĩa transaction

NETCONF guarantee atomic commit: toàn bộ candidate hoặc nothing. Nếu edit sequence gây invalid state, `validate` thất bại, commit không execute. Nếu `confirmed-commit` timeout (client disconnect), switch rollback tự động. Pattern này critical cho remote config change, nếu admin SSH mà edit làm mất connection, device rollback và restore.

## 5.0.3 YANG data model (RFC 7950, August 2016)

### Ngôn ngữ schema

YANG (Yet Another Next Generation) là schema language mô tả **structure** của config + state data. RFC 7950 "The YANG 1.1 Data Modeling Language" publish 08/2016, update RFC 6020 YANG 1.0 (10/2010).

YANG khác XML Schema (XSD): YANG optimize cho network config, có built-in types (ipv4-address, mac-address, yang:counter64), constraints (must/when), grouping reuse. XSD generic cho any XML, verbose cho network model.

### Cấu trúc module

```yang
module ietf-interfaces {
  namespace "urn:ietf:params:xml:ns:yang:ietf-interfaces";
  prefix "if";

  container interfaces {
    list interface {
      key "name";
      leaf name { type string; }
      leaf type { type identityref { base iana-if-type:iana-interface-type; } }
      leaf enabled { type boolean; default "true"; }
      container ipv4 {
        leaf address { type inet:ipv4-address; }
      }
    }
  }
}
```

Cấu trúc:

| Construct | Mô tả |
|-----------|-------|
| `module` | Top-level container, namespace + prefix |
| `container` | Grouping data nodes |
| `list` | Collection với key (mỗi element unique by key) |
| `leaf` | Single value data node |
| `leaf-list` | List of leaf (simple collection) |
| `grouping` | Reusable definition (như struct trong C) |
| `augment` | Extend module khác (không modify original) |

### Các module chuẩn của IETF

IETF ship các module chuẩn cho common model:

- **ietf-interfaces**: basic interface config.
- **ietf-routing**: routing instance + RIB.
- **ietf-ip**: IPv4 config (address, DHCP).
- **ietf-system**: hostname, DNS, NTP.
- **ietf-access-control-list**: ACL rules.

Vendor extend qua `augment` thay vì redefine:

```yang
module cisco-extensions {
  import ietf-interfaces { prefix "if"; }

  augment "/if:interfaces/if:interface" {
    leaf cisco-description { type string; }
    leaf cisco-port-channel-id { type uint16; }
  }
}
```

Cisco-specific field được add không touching ietf-interfaces module. Controller dùng ietf-interfaces cho common ops, cisco-extensions cho vendor-specific. Portable across vendors.

### Hệ sinh thái module từ vendor

- **Cisco YANG models**: github.com/YangModels/yang/tree/main/vendor/cisco (nhiều OS: IOS-XE, IOS-XR, NX-OS).
- **Juniper Junos YANG**: github.com/Juniper/yang.
- **Arista EOS**: github.com/aristanetworks/openmgmt.
- **OpenConfig**: vendor-agnostic alternative, bootstrapped by AT&T/Google/Microsoft 2014, github.com/openconfig/public.

OpenConfig aim vendor-independent, cross-vendor identity "openconfig-interfaces" thay vì per-vendor. 2026 sự áp dụng: OpenConfig phổ biến trong service provider người vận hành (AT&T, Verizon), less trong enterprise (Cisco/Juniper stick với own YANG).

## 5.0.4 RESTCONF (RFC 8040, January 2017)

### Phương án HTTP/JSON thay thế

NETCONF XML được critique khó integrate với modern DevOps tooling (most automation dùng JSON + HTTP). RESTCONF publish RFC 8040 tháng 1/2017 đưa **HTTP + JSON** alternative cho NETCONF, cùng YANG model.

URL path mirror YANG tree:

```
GET /restconf/data/ietf-interfaces:interfaces/interface=GigabitEthernet0_0_0_0
```

HTTP method map CRUD:

| HTTP | YANG operation |
|------|----------------|
| GET | get-config hoặc get |
| POST | create (failed nếu tồn tại) |
| PUT | replace (create hoặc overwrite) |
| PATCH | merge (partial update) |
| DELETE | delete |

JSON encoding:

```json
{
  "ietf-interfaces:interface": {
    "name": "GigabitEthernet0/0/0/0",
    "type": "iana-if-type:ethernetCsmacd",
    "enabled": true,
    "ietf-ip:ipv4": {
      "address": [{"ip": "10.0.0.1", "netmask": "255.255.255.0"}]
    }
  }
}
```

Giảm verbosity ~3x so với XML.

### Lý do ra đời

RFC 6241 NETCONF 2011 pre-date modern DevOps tool (Ansible, Puppet, Chef auto-config 2013+; Kubernetes 2014+). Tools này dominant workflow từ 2015, ecosystem XML shrinking. RESTCONF đáp ứng by allowing developer dùng familiar HTTP + curl/requests, JSON parser built-in most language.

Sự chấp nhận 2026:
- Enterprise người vận hành: RESTCONF primary cho automation script.
- Service provider: NETCONF vẫn dùng cho config transaction (atomic commit quan trọng hơn HTTP familiarity).
- Hyperscaler: gNMI (xem 5.0.6) thay thế RESTCONF trong production.

## 5.0.5 BGP-LS (RFC 7752) cho topology distribution

### Vấn đề: controller cần biết topology

SDN WAN controller cần thời gian thực topology graph (nodes + links + metrics) để compute TE paths. Trong traditional IGP network, topology info trong OSPF/IS-IS LSDB, chỉ router participating IGP biết. Controller không tham gia IGP (security risk, instability), phải import topology out-of-band.

### BGP-LS mechanism (RFC 7752, March 2016)

BGP Link-State extend BGP để carry IGP topology information. Protocol:

1. **IGP → BGP reflector**: router chạy BGP-LS export LSDB content sang BGP route reflector (RR).
2. **RR → controller**: controller establish BGP-LS peering với RR, nhận topology updates thời gian thực.
3. **Controller process**: parse BGP-LS advertisements, build topology graph in-memory, trigger TE re-optimization khi link thất bại.

BGP-LS NLRI types:
- **Node NLRI**: router, area membership, capability.
- **Link NLRI**: link between nodes, metric, bandwidth.
- **Prefix NLRI**: IPv4 prefix reachable qua link.

### Field hợp sử dụng: Cisco WAE, Juniper NorthStar

**Cisco WAN Automation Engine (WAE)**: commercial controller dùng BGP-LS cho topology + PCE-P cho path provisioning. Deploy ở AT&T, Comcast, Telstra.

**Juniper NorthStar**: alternative commercial controller, BGP-LS topology + PCE-P segment routing path control.

**Open Source alternatives**:
- **ExaBGP**: BGP-LS parser, feed topology sang controller application.
- **GoBGP**: Go BGP implementation với LS hỗ trợ.

### Sự đánh đổi so với OpenFlow trong WAN

OpenFlow cách tiếp cận to WAN: install flow rule per TE path vào router. Problems: 10K+ TE paths × 10^3 routers = millions of flow entries, không scale với TCAM limit router.

BGP-LS + PCE-P cách tiếp cận: router retain normal IGP chuyển tiếp, controller chỉ override specific TE paths. Flow entry overhead minimal (per-LSP state), scale better.

Google B4 (Phần 4.6.6) initially use OpenFlow cho TE. Subsequent evolution move toward segment routing + PCE-P, closer to BGP-LS mô hình than OpenFlow-pure.

## 5.0.6 PCE-P (RFC 5440) cho TE path computation

### Path Computation Element architecture

Traditional routing: router compute path distributed (OSPF SPF, IS-IS SPF). Cho TE constraint (bandwidth, độ trễ, diverse paths), SPF không đủ, yêu cầu **centralized computation**.

PCE (Path Computation Element) là dedicated entity compute constrained paths. PCE-P là protocol communication giữa PCC (PCE Client, thường router) và PCE:

- **PCC → PCE**: `PCReq` Path Computation Request với constraints (source, dest, bandwidth, affinity, disjointness).
- **PCE → PCC**: `PCRep` Path Computation Reply với path segments.

RFC 5440 "Path Computation Element (PCE) Communication Protocol" publish 3/2009, stable long-term.

### Stateful PCE (RFC 8231, September 2017)

Original PCE-P là stateless: PCC hỏi, PCE reply, PCE không maintain LSP state. Stateful PCE extension cho phép PCE **own LSP state**: write-authority, PCE có thể actively re-optimize paths.

Flow stateful PCE:

1. PCC report LSP state sang PCE (`PCRpt`).
2. PCE maintain authoritative LSP database.
3. PCE monitor topology (via BGP-LS), compute re-optimization decisions.
4. PCE send `PCUpd` (Path Computation Update) to PCC để cài đặt new path.

Difference với OpenFlow: PCE-P operate ở **LSP level** (logical tunnel), không packet level. Router vẫn xử lý packet chuyển tiếp theo installed LSP, PCE control tunnel placement.

### Integration với Segment Routing

Segment Routing (SR, RFC 8402) là packet chuyển tiếp mô hình thay MPLS label stack bằng segment identifier (SID). PCE-P + SR cho source-routed TE: controller compute segment list, PCC encap packet với SID stack. No per-LSP state in transit routers, scale better than RSVP-TE.

Triển khai production:
- **AT&T Flexware**: SDN-based WAN với PCE-P + SR.
- **Comcast**: PCE-P + SR-MPLS cho backbone.
- **Deutsche Telekom**: SRv6 based với PCE-P control.

## 5.0.7 Cisco ACI, Juniper Contrail launch 2013

### Cisco ACI (Application Centric Infrastructure)

Launched 11/2013 cùng thời điểm VMware NSX (xem Phần 5.1). Components:

- **APIC (Application Policy Infrastructure Controller)**: centralized controller, cluster 3-5 node.
- **Nexus 9000 hardware**: custom ASIC (Cisco Cloud Scale), supports VXLAN fabric.
- **OpFlex**: southbound protocol từ APIC to switches. **Không dùng OpenFlow**.
- **iVXLAN**: custom VXLAN extension với group policy.

Northbound API của APIC: REST với JSON. Người vận hành configure application-centric policy (contract, EPG - Endpoint Group), APIC translate sang fabric config.

Cisco positioning ACI là "better than OpenFlow": SDN mà không abandon Cisco hardware investment, phơi bày API cho automation mà không force người vận hành learn flow programming.

### Juniper Contrail

Launched 2013, acquired by Juniper 12/2012 (acquired startup Contrail Systems). Components:

- **Contrail Controller**: centralized control plane, multi-node HA.
- **vRouter** (compute agent): runs on each hypervisor, kernel module hoặc DPDK, functional analog của OVS.
- **Southbound**: XMPP (Extensible Messaging Protocol) giữa controller và vRouter, NOT OpenFlow.
- **Control plane**: BGP EVPN cho L2/L3 VPN signaling.

Opensource as **OpenContrail** (2014), renamed **Tungsten Fabric** under Linux Foundation (2018).

### Chống lại OpenFlow narrative

Cả ACI và Contrail market themselves là "SDN alternatives to OpenFlow". ONF positioning OpenFlow=SDN bị challenge bởi reality vendor sự áp dụng, ACI và Contrail use modern SDN concepts (centralized control, policy-driven, API-first) nhưng không dùng OpenFlow protocol. Điều này đẩy ngành recognize rằng SDN is broader than OpenFlow, matching Göransson Ch6 classification 4 types: Open SDN, SDN via APIs, Hypervisor Overlay, Whitebox.

Cross-reference forward: Phần 5.1 analyzes hypervisor overlay (NVP/NSX/OVN) đối trọng; Phần 5.2 analyzes whitebox (Cumulus/SONiC). Phần 7.3 sẽ phân tích ACI và Contrail architecture chi tiết trong context controller ecosystem.

> **Chủ đề trọng tâm:** "SDN via APIs" không thay thế OpenFlow; nó giải một subset của SDN problem (config management + policy automation) cho segment không cần data plane programmability (WAN, enterprise edge). OpenFlow giải subset khác (DC virtualization với dynamic flow installation). Cả hai cùng tồn tại, apply cho use case khác nhau. Bài học: không có "one SDN to rule them all"; mô hình đúng tùy triển khai context.

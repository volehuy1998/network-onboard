## Mục tiêu bài học

Sau khi hoàn thành phần này, bạn có thể:

1. **Mô tả (Understand)** vai trò của IPAM (IP Address Management) native trong OVN — tự động cấp phát IP cho pod/VM từ subnet mà không cần external IPAM system.

## 13.12.1 Vai trò IPAM trong OVN

IPAM (IP Address Management) là tính năng cho phép OVN tự quản lý subnet và cấp phát IP cho LSP mà không cần external service. Khi CMS (OpenStack Neutron, OVN-Kubernetes) tạo một pod/VM, nó chỉ cần tạo LSP với type=`dynamic` — OVN tự chọn IP trong subnet, đảm bảo không xung đột với IP đã cấp phát, và ghi kết quả vào bảng Logical_Switch_Port.

**Lý do tích hợp IPAM vào OVN:**
- **Loại bỏ phụ thuộc external:** với cụm nhỏ-trung, không cần triển khai Infoblox hay BIND riêng.
- **Hiệu năng:** cấp phát IP đồng bộ với LSP creation, không có round-trip tới external API.
- **Tính nguyên tử:** trạng thái IPAM nằm cùng NB DB → transaction nguyên tử khi tạo/xóa LSP.
- **Multi-tenancy:** mỗi Logical Switch có subnet riêng → tenant A không xung đột với tenant B dù dùng cùng CIDR.

**Cấu trúc IPAM OVN:**
- Mỗi Logical Switch có thể khai báo `other_config:subnet=10.0.0.0/24` → OVN IPAM bật cho switch đó.
- Tùy chọn: `other_config:exclude_ips=10.0.0.1..10.0.0.10` → không cấp phát các IP này (thường dùng cho gateway, DHCP server, reserved).
- LSP với `addresses=["aa:bb:cc:dd:ee:ff dynamic"]` → OVN gán IP động từ subnet.
- LSP với `addresses=["aa:bb:cc:dd:ee:ff 10.0.0.100"]` → tĩnh, user chỉ định trực tiếp.
- OVN ghi kết quả vào `dynamic_addresses="aa:bb:cc:dd:ee:ff 10.0.0.5"` (giá trị thực tế được cấp).

## 13.12.2 Cấp phát động (dynamic allocation)

Khi LSP được khai báo với `addresses="<MAC> dynamic"`, ovn-northd thực hiện:

**Bước 1 — Tìm subnet:** đọc `other_config:subnet` của Logical Switch chứa LSP.
**Bước 2 — Thu thập IP đã dùng:** scan tất cả LSP cùng LS, lấy IP từ `dynamic_addresses` và `addresses` (tĩnh). Cũng bao gồm `other_config:exclude_ips`.
**Bước 3 — Chọn IP mới:** duyệt subnet tìm IP thấp nhất không thuộc tập đã dùng. Thuật toán: duyệt tuyến tính từ subnet_min tới subnet_max, skip exclude list.
**Bước 4 — Ghi kết quả:** set `LSP.dynamic_addresses = "<MAC> <IP_chosen>"`.
**Bước 5 — Kích hoạt:** nếu LS có DHCP_Options, Logical Flow `ls_in_dhcp_options` tự động dùng IP này làm offerip.

**Ví dụ workflow:**
```
[host]# ovn-nbctl ls-add ls1
[host]# ovn-nbctl set Logical_Switch ls1 other_config:subnet=10.0.0.0/24
[host]# ovn-nbctl set Logical_Switch ls1 other_config:exclude_ips="10.0.0.1..10.0.0.10"

[host]# ovn-nbctl lsp-add ls1 vm1
[host]# ovn-nbctl lsp-set-addresses vm1 "aa:bb:cc:dd:ee:01 dynamic"

[host]# ovn-nbctl lsp-add ls1 vm2
[host]# ovn-nbctl lsp-set-addresses vm2 "aa:bb:cc:dd:ee:02 dynamic"

[host]# ovn-nbctl get Logical_Switch_Port vm1 dynamic_addresses
"aa:bb:cc:dd:ee:01 10.0.0.11"

[host]# ovn-nbctl get Logical_Switch_Port vm2 dynamic_addresses
"aa:bb:cc:dd:ee:02 10.0.0.12"
```

OVN bắt đầu cấp phát từ `10.0.0.11` vì exclude_ips chặn 10.0.0.1-10.0.0.10.

**Thuật toán chi tiết cho exclude_ips:** format `<start_ip>..<end_ip>` (range) hoặc IP đơn. Nhiều range phân tách bằng space:
```
exclude_ips="10.0.0.1 10.0.0.5..10.0.0.10 10.0.0.100..10.0.0.200"
```
Ranges có thể chồng nhau — OVN tự gộp.

## 13.12.3 Cấp phát tĩnh (static allocation)

Cho các workload cần IP cố định (server với DNS record, VIP):

```
[host]# ovn-nbctl lsp-add ls1 db_server
[host]# ovn-nbctl lsp-set-addresses db_server "aa:bb:cc:dd:ee:99 10.0.0.50"
```

OVN kiểm tra IP 10.0.0.50 không xung đột với `dynamic_addresses` của LSP khác. Nếu xung đột → ovn-northd log warning nhưng vẫn chấp nhận (user chủ động).

**Kết hợp static + dynamic:** cùng LS có thể có LSP static và LSP dynamic. Cấp phát dynamic tự động skip IP static đã dùng.

**Mac_prefix — tùy chỉnh MAC range:** mặc định dynamic addresses dùng MAC từ OUI `OVN-internal` (`0a:00:00:xx:xx:xx`). Nếu cần MAC tùy chỉnh (ví dụ MAC được cấp phép cho VMware compatibility):
```
[host]# ovn-nbctl set Logical_Switch ls1 other_config:mac_prefix="52:54:00"
```
OVN sinh MAC dạng `52:54:00:xx:xx:xx` ngẫu nhiên.

## 13.12.4 IPv6 IPAM và prefix delegation

OVN hỗ trợ IPv6 IPAM tương tự IPv4 — subnet cấu hình qua `other_config:ipv6_prefix`:
```
[host]# ovn-nbctl set Logical_Switch ls1 other_config:ipv6_prefix="fd00:1::/64"
```

**Hai chế độ định địa chỉ IPv6:**
- **DHCPv6 stateful:** OVN DHCP_Options cấp IPv6 (đã trình bày Part 13.10).
- **SLAAC với EUI-64:** OVN sinh IPv6 address từ MAC theo RFC 4291 EUI-64 algorithm. Pod tự cấu hình qua Router Advertisement — không cần DHCPv6.

**IPv6 Prefix Delegation (PD):** OVN 22.03+ hỗ trợ delegate IPv6 prefix từ upstream (ISP) tới OVN qua DHCPv6-PD:
```
[host]# ovn-nbctl set Logical_Router_Port lrp_ext \
         options:prefix-delegation=true \
         options:prefix-length=56 \
         options:dhcpv6-pd=true
```
Tích hợp với FRR/DHCPv6 client để request prefix từ upstream, sau đó OVN tự động gán /64 subnets cho từng LS nội bộ.

## 13.12.5 ND Proxy và học MAC

Khi LSP type=`vif` bind với chassis, ovn-controller trên chassis đó tự động học MAC của pod qua OpenFlow và ghi vào MAC_Binding table trong SB DB:

**Bảng MAC_Binding:**
- `logical_port` — tên LSP.
- `ip` — IP của neighbor.
- `mac` — MAC đã học.
- `datapath` — Datapath_Binding record.
- `timestamp` — thời điểm học (OVN 22.03+).

MAC_Binding được dùng để:
- Tránh flood ARP/NDP qua cluster — chỉ flood một lần, sau đó cache entry.
- Hỗ trợ ND Proxy — Logical Router có thể trả lời ARP/NDP thay cho pod (nếu pod không response, hoặc pod ở subnet khác).

**ND Proxy trên Logical Router:**
Khi LRP có address IPv6 + cờ `ipv6_ra_configs:nd_proxy=true`, LR sẽ trả lời NDP Neighbor Solicitation cho mọi IP trong subnet thuộc LR:
```
[host]# ovn-nbctl set Logical_Router_Port lrp1 ipv6_ra_configs:nd_proxy=true
```

Hữu ích khi cluster có nhiều subnet chia sẻ một gateway — gateway proxy NDP cho mọi subnet.

**Timestamp của MAC_Binding (OVN 22.03+):** theo dõi thời gian entry được học/làm mới. ovn-northd expire entry cũ (quá `other_config:mac-binding-age-threshold`, mặc định 600s) để tránh state pollution khi pod di chuyển chassis.

## 13.12.6 Tích hợp với external IPAM

Với môi trường doanh nghiệp có IPAM tập trung (Infoblox, BlueCat, NetBox), không nên dùng OVN IPAM native. Hai mô hình tích hợp:

**Mô hình 1 — External IPAM ghi NB DB:** IPAM system tạo LSP với static addresses (không dùng `dynamic`). OVN chỉ làm data plane, IPAM ở lớp trên. Ví dụ:
- Infoblox tạo record gán IP.
- Plugin Neutron IPAM hook đọc Infoblox, tạo LSP với IP cụ thể.
- OVN chỉ xác nhận không xung đột.

**Mô hình 2 — OVN IPAM báo lại external:** OVN cấp phát động, sau đó plugin đồng bộ `dynamic_addresses` sang external IPAM để IPAM có khả năng quan sát. Phổ biến trong cụm nhỏ với IPAM chủ yếu cho báo cáo/audit.

**Hạn chế IPAM OVN:**
- Không có hạn mức per-tenant.
- Không có VRF concept (một LS = một subnet, không overlapping IP giữa LS).
- Không có tích hợp DNS tự động (phải thủ công đồng bộ với external DNS).
- Không hỗ trợ dự trữ IP với TTL.

Các công cụ doanh nghiệp như Infoblox NIOS có những tính năng này — nếu cluster cần, tích hợp qua plugin.

### ▶ Guided Exercise 1: Cấu hình IPAM với exclude và cấp phát động

> **Trạng thái verification lab:** Output dựa trên OVN 22.03 docs; chờ verify lab thực.

**Mục đích:** Chứng minh OVN IPAM tự động cấp phát IP cho LSP, tôn trọng exclude_ips, và xử lý xung đột với static LSP.

**Chuẩn bị:** Lab Ubuntu 22.04 với OVN 22.03.

**Bước thực hiện:**

**1.** Tạo Logical Switch với subnet và exclude:
```
[host]# ovn-nbctl ls-add ipam_ls
[host]# ovn-nbctl set Logical_Switch ipam_ls \
         other_config:subnet=192.168.10.0/24
[host]# ovn-nbctl set Logical_Switch ipam_ls \
         other_config:exclude_ips="192.168.10.1..192.168.10.20 192.168.10.100..192.168.10.110"
```

**2.** Tạo 3 LSP dynamic:
```
[host]# ovn-nbctl lsp-add ipam_ls pod1
[host]# ovn-nbctl lsp-set-addresses pod1 "dynamic"
[host]# ovn-nbctl lsp-add ipam_ls pod2
[host]# ovn-nbctl lsp-set-addresses pod2 "dynamic"
[host]# ovn-nbctl lsp-add ipam_ls pod3
[host]# ovn-nbctl lsp-set-addresses pod3 "dynamic"
```

Lưu ý: không cung cấp MAC → OVN sinh MAC tự động.

**3.** Kiểm tra dynamic_addresses:
```
[host]# ovn-nbctl get Logical_Switch_Port pod1 dynamic_addresses
"0a:00:00:11:22:01 192.168.10.21"
[host]# ovn-nbctl get Logical_Switch_Port pod2 dynamic_addresses
"0a:00:00:11:22:02 192.168.10.22"
[host]# ovn-nbctl get Logical_Switch_Port pod3 dynamic_addresses
"0a:00:00:11:22:03 192.168.10.23"
```

Quan sát: OVN skip 192.168.10.1-20 (exclude range) và cấp phát từ 192.168.10.21.

**4.** Thêm LSP static với IP chỉ định:
```
[host]# ovn-nbctl lsp-add ipam_ls gateway
[host]# ovn-nbctl lsp-set-addresses gateway "aa:bb:cc:dd:ee:99 192.168.10.50"
```

**5.** Thêm LSP dynamic thứ 4:
```
[host]# ovn-nbctl lsp-add ipam_ls pod4
[host]# ovn-nbctl lsp-set-addresses pod4 "dynamic"
[host]# ovn-nbctl get Logical_Switch_Port pod4 dynamic_addresses
"0a:00:00:11:22:04 192.168.10.24"
```

Quan sát: OVN skip 192.168.10.50 (static đã dùng) và cấp 192.168.10.24 (IP dynamic tiếp theo trong subnet sau 21-23 đã cấp).

**6.** Test mac_prefix tùy chỉnh:
```
[host]# ovn-nbctl set Logical_Switch ipam_ls other_config:mac_prefix="52:54:00"
[host]# ovn-nbctl lsp-add ipam_ls vm1
[host]# ovn-nbctl lsp-set-addresses vm1 "dynamic"
[host]# ovn-nbctl get Logical_Switch_Port vm1 dynamic_addresses
"52:54:00:a1:b2:c3 192.168.10.25"
```

Quan sát: MAC mới sinh có prefix `52:54:00` thay vì default.

**Bài học học được:**

Thứ nhất, OVN IPAM là **tích hợp sẵn và không cấu hình**: chỉ cần set `other_config:subnet`, mọi LSP với `dynamic` tự động được cấp IP. Không cần external service.

Thứ hai, `exclude_ips` quan trọng cho IP dự trữ — gateway IP, DHCP server IP, VIP floating IP phải được loại trừ để tránh OVN cấp cho pod (gây xung đột network).

Thứ ba, IPAM OVN **không có hạn mức per-LSP** hay `max_address_count` — nếu subnet /24 có 254 IP khả dụng và cluster mở rộng lên 300 pod, việc cấp phát thứ 255 sẽ fail âm thầm (LSP không có dynamic_addresses). Cluster môi trường production cần giám sát `dynamic_addresses` để cảnh báo khi subnet sắp đầy.

**Cleanup:**
```
[host]# ovn-nbctl ls-del ipam_ls
```

## 13.9.1 Kiến trúc Load Balancer trong OVN

OVN cung cấp Load Balancer L4 (TCP/UDP/SCTP) native ở data plane, không cần tiến trình userspace riêng như HAProxy. Bản g `Load_Balancer` trong NB DB định nghĩa ánh xạ từ **virtual IP (VIP):port** sang một danh sách backend **real IP:port**.

**Cấu trúc table Load_Balancer trong NB DB:**
- `name`: tên định danh (tùy chọn).
- `vips`: map từ "VIP:port" → "backend1:port1, backend2:port2, ..." (chuỗi).
- `protocol`: "tcp", "udp", "sctp", hoặc rỗng (coi như TCP).
- `health_check`: tham chiếu table `Load_Balancer_Health_Check` (tùy chọn).
- `ip_port_mappings`: map "endpoint_ip → mac:port" để sinh ARP response (cho VIP trong Logical Switch).
- `selection_fields`: field dùng cho hash (mặc định: src_ip, src_port, dst_ip, dst_port).
- `options`: cờ mở rộng (hairpin_snat_ip, reject, skip_snat, affinity_timeout).

**Gắn LB vào Logical Switch hay Logical Router:**
- `ovn-nbctl ls-lb-add ls1 lb1`: LB được áp cho traffic vào Logical Switch.
- `ovn-nbctl lr-lb-add lr1 lb1`: LB được áp cho traffic vào Logical Router.
- Một LB có thể gắn vào nhiều LS/LR đồng thời (dùng chung VIP cho nhiều network).

**Pipeline xử lý LB:**
- Ở **Logical Switch ingress pipeline**, table `ls_in_pre_lb` (table 6) đánh dấu packet destined tới VIP cần đi qua LB.
- Table `ls_in_lb` (table 12) thực hiện DNAT: thay destination từ VIP:port → backend:port theo hash.
- Table `ls_in_lb_aff_check` (table 13) kiểm tra session affinity.
- Table `ls_in_stateful` (table 14) commit connection vào conntrack để traffic trả lời được un-DNAT (SNAT ngược).

- Ở **Logical Router ingress pipeline**, table `lr_in_defrag` (table 5) defragment trước LB.
- Table `lr_in_dnat` (table 6) thực hiện DNAT cho LB.
- Table `lr_in_ecmp_stateful` (table 7) chọn đường ECMP nếu nhiều backend đạt điều kiện.

## 13.9.2 Thuật toán hash chọn backend

Khi một gói đến khớp với VIP, OVN cần chọn một backend trong danh sách. Thuật toán mặc định là **hash 5-tuple**:
```
hash(src_ip, src_port, dst_ip, dst_port, protocol) % len(backends)
```

Hash này có hai tính chất quan trọng:
- **Tính nhất quán (consistency):** cùng client (cùng src_ip + src_port) với cùng destination luôn map tới cùng backend (miễn là danh sách backend không đổi).
- **Phân phối đều:** với nhiều client, traffic được phân phối đều qua các backend.

**Vấn đề khi backend thay đổi:** nếu một backend bị remove (scale-down hoặc health check fail), hash modulo thay đổi hoàn toàn, khoảng 50% client hiện tại sẽ được remap sang backend khác, gây đứt connection (conntrack mất track). Đây là lý do OVN hiện thực **pinning dựa trên conntrack**: khi connection được thiết lập, entry conntrack lưu backend đã chọn. Packet của cùng connection luôn đi tới backend đã chọn, bất kể hash modulo thay đổi thế nào. Chỉ connection MỚI mới bị remap.

**Selection fields tùy chỉnh:** user có thể override hash input qua `selection_fields`:
```
[host]# ovn-nbctl --selection-fields="ip_src,ip_dst" lb-add lb1 ...
```
Điều này hữu ích cho sticky session dựa trên src_ip (không phụ thuộc src_port ngẫu nhiên).

**Chọn đường ECMP:** nếu LB gắn vào Logical Router và có nhiều backend, OVN có thể dùng **hash đối xứng (symmetric hash)** để đảm bảo traffic đi và traffic trả lời chọn cùng đường (tránh định tuyến bất đối xứng gây conntrack miss).

## 13.9.3 SNAT handling cho gói trả lời

Khi client gửi packet đến VIP, DNAT đổi destination thành backend IP. Packet được forward đến backend. Ba ckend xử lý và gửi reply lại, destination = client IP, source = backend IP.

**Vấn đề:** client không biết backend, nó gửi request tới VIP, kỳ vọng reply từ VIP. Nếu reply source = backend IP, client drop reply (TCP mismatch).

**Giải pháp OVN:** conntrack entry (tạo lúc DNAT chiều forward) chứa thông tin ánh xạ VIP ↔ backend. Khi reply packet đi qua OVN pipeline, conntrack khớp với và tự động **un-DNAT**, đổi source từ backend IP sang VIP. Client nhận reply có source = VIP, hợp lệ.

**Tùy chọn bỏ qua SNAT:** trong một số field hợp (DSR — Direct Server Return) backend có VIP cấu hình trên loopback interface và tự sinh reply với source = VIP (bỏ qua OVN). Để tránh OVN thực hiện un-DNAT trên reply (không cần thiết), dùng:
```
[host]# ovn-nbctl set Load_Balancer lb1 options:skip_snat=true
```

**Hairpin:** khi backend chính là một pod trong cùng Logical Switch và tự nó gửi request đến VIP của service mà nó là backend → packet loop về chính nó. Không xử lý đặc biệt, packet có:
- src_ip = pod_IP (= backend_IP)
- dst_ip = VIP → DNAT → dst_ip = backend_IP = pod_IP

Kết quả: pod gửi packet cho chính nó với source = destination. Kernel TCP stack drop vì same source/dest. OVN xử lý hairpin bằng cách **SNAT** source của packet hairpin thành một IP "hairpin SNAT IP" (cấu hình qua `options:hairpin_snat_ip`), làm cho packet xuất hiện như đến từ một host khác:
```
[host]# ovn-nbctl set Load_Balancer lb1 options:hairpin_snat_ip=169.254.169.254
```

## 13.9.4 Session affinity

Với traffic HTTP/HTTPS, một client có thể gửi nhiều request qua nhiều TCP connection. Mỗi connection có src_port khác nhau → hash có thể chọn backend khác nhau → state session (authentication state, shopping cart) bị mất.

**Session affinity OVN** đảm bảo cùng client (cùng src_ip) luôn tới cùng backend trong khoảng timeout nhất định. Hiện thực:
```
[host]# ovn-nbctl lb-add lb1 10.0.0.100:80 "192.168.1.1:80,192.168.1.2:80" \
         -- set Load_Balancer lb1 options:affinity_timeout=3600
```

Dưới mui xe, OVN dùng một table hash phụ (`lb_aff_check` và `lb_aff_learn`) ghi ánh xạ `(client_ip, vip) → backend_chosen` với TTL = affinity_timeout. Khi connection mới từ cùng client đến, OVN tra table này trước khi hash, nếu có entry → dùng backend đã chọn trước; nếu không → hash mới và ghi entry.

**Hạn chế:** session affinity dựa trên src_ip → nếu client đứng sau NAT (nhiều client cùng public IP) thì tất cả được map tới cùng backend (không phân phối). Với workload Kubernetes nội bộ, client là pod IP → affinity chính xác.

## 13.9.5 Health check và active/passive backend

OVN hỗ trợ health check cho backend thông qua table `Load_Balancer_Health_Check`:
```
[host]# ovn-nbctl --id=@hc create Load_Balancer_Health_Check \
         vip='"10.0.0.100:80"' \
         options:interval=5 options:timeout=3 options:success_count=1 options:failure_count=3 \
         -- add Load_Balancer lb1 health_check @hc
```

Mỗi chassis có ovn-controller sẽ gửi **TCP SYN** (hoặc ICMP echo, tùy cấu hình) tới từng backend IP:port trong LB mà chassis đó host. Nếu backend không reply (timeout 3s × 3 lần fail), backend được đánh dấu `offline`. OVN ghi vào table `Service_Monitor` trong SB DB. ovn-northd đọc `Service_Monitor`, lọc backend offline khỏi tập active, và sinh Logical Flow mới không bao gồm backend offline.

**Kiến trúc health check phân tán:** mỗi chassis tự probe subset backend (thường là backend thuộc cùng chassis nếu có, hoặc được gán theo hash để phân phối tải). Kết quả tổng hợp qua table `Service_Monitor`. Ưu điểm: mở rộng tốt (không có điểm probe đơn), nhược điểm: mỗi backend bị probe nhiều lần (từ nhiều chassis).

**Tích hợp với Kubernetes readiness probe:** trong OVN-Kubernetes, readinessProbe của pod được dịch sang `Load_Balancer_Health_Check` cho Service endpoint tương ứng. Nếu pod readinessProbe fail → backend trong OVN LB bị remove → traffic Service không định tuyến tới pod đó.

## 13.9.6 So sánh với kube-proxy (iptables + IPVS)

| Đặc điểm | OVN LB | kube-proxy iptables | kube-proxy IPVS |
|----------|--------|---------------------|-----------------|
| Thuật toán hash | Consistent 5-tuple | Random (iptables `-m statistic`) | Round-robin / LC / WLC / SH |
| Session affinity | Có, qua ClientIP | Có, qua ClientIP | Có, native IPVS |
| Health check | Có, TCP/ICMP | Không (phụ thuộc kubelet readiness) | Không native |
| Vị trí | OVS datapath (kernel) | iptables chain | IPVS kernel module |
| Mở rộng | O(1) tra cứu qua conntrack | O(n) duyệt chain | O(1) hash |
| Độ trễ cập nhật | Dưới giây (ovn-northd) | Giây (rebuild iptables) | Dưới giây (ipvsadm) |

**Khi chọn OVN LB:** cụm đã dùng OVN-K8s làm CNI → LB native, không cần kube-proxy. Ưu điểm: đồng nhất data plane, gỡ lỗi một công cụ (ovn-trace) cho cả định tuyến và LB.

**Khi giữ kube-proxy:** cluster dùng CNI khác (Calico, Flannel) không có LB native → phải dùng kube-proxy. Hoặc cluster cần tính năng IPVS cụ thể (LC, WLC algorithm) không có trong OVN.

## 13.9.7 Source-code anchor: `ct_lb` action implementation

OVN Load Balancer hoạt động qua OpenFlow action `ct_lb` trong OVS. Action này triển khai trong userspace ovs-vswitchd + kernel openvswitch.ko với conntrack integration.

### `ct_lb()` action syntax (ovs-actions(7))

```
ct_lb(args)
ct_lb(backends=ip[:port][, ip[:port]...][; selection_method=method])
```

Tham số:

- `backends`: danh sách backend `ip:port,ip:port,...`.
- `selection_method`: `dp_hash` (default, kernel hash), `hash` (5-tuple userspace), `random`.

Source code anchor (OVS `v2.17.9`):

- [`lib/ofp-actions.c`](https://github.com/openvswitch/ovs/blob/v2.17.9/lib/ofp-actions.c), function `parse_ct_lb()`, parse cú pháp `ct_lb(...)` từ OpenFlow text → struct nội bộ.
- [`ofproto/ofproto-dpif-xlate.c`](https://github.com/openvswitch/ovs/blob/v2.17.9/ofproto/ofproto-dpif-xlate.c), function `compose_conntrack_action()`, build datapath action với CT_LB flag.
- Kernel side: `net/netfilter/nf_conntrack_extend.c` + `net/openvswitch/conntrack.c` xử lý CT_LB ext + NAT rewrite.

### Pipeline OVN sinh `ct_lb`

ovn-northd compile NB Load_Balancer thành Logical_Flow ở stages:

```
table=ls_in_pre_lb (table 6):
  priority=100, match=(ip && reg0[2] == 1),
  action=(reg0[2] = 0; ct_next;)
  
table=ls_in_lb (table 12):
  priority=100,
  match=(ct.new && ip4.dst == 10.0.0.100 && tcp.dst == 80),
  action=(reg1 = 10.0.0.11;
          reg2[0..15] = 8080;
          ct_lb(backends=10.0.0.11:8080,10.0.0.12:8080);)
  
table=ls_in_stateful (table 14):
  priority=100, match=(reg0[1] == 1 && ip4),
  action=(ct_commit { ct_label.natted = 1; }; next;)
```

3 stage mapping:

1. `ls_in_pre_lb` đánh dấu packet "đã đi qua conntrack lookup" (`ct_next`).
2. `ls_in_lb` thực hiện DNAT qua `ct_lb` action.
3. `ls_in_stateful` commit conntrack entry với `ct_label.natted = 1` để reply biết un-DNAT.

### `ct_lb` runtime mechanism

Khi packet hit `ct_lb` action trong kernel datapath:

1. Kernel netfilter hook gọi `nf_nat_setup_info(NF_NAT_DST)`.
2. Hash 5-tuple chọn backend (dùng jhash + modulo backend count).
3. Backend rewrite: `dst_ip` thay bằng backend IP, `dst_port` thay bằng backend port.
4. Conntrack entry mới với 2 tuple:
   - Original: `src=client, dst=VIP, sport=client_port, dport=VIP_port`.
   - Reply: `src=backend, dst=client, sport=backend_port, dport=client_port`.

Reply path: kernel auto-match conntrack entry, swap `src=backend → src=VIP` (un-DNAT). Client thấy reply có source = VIP, transparent.

## 13.9.8 Service_Monitor table SBDB schema

Health check distributed kết quả tổng hợp ở table `Service_Monitor` trong SB DB. ovn-northd đọc table này, exclude backend offline khỏi `Logical_Flow`.

```
[host]# ovn-sbctl list Service_Monitor
_uuid               : abc123-...
chassis_name        : "chassis-A"
external_ids        : {}
ip                  : "10.0.0.11"
logical_port        : "backend1"
options             : {failure_count="3", interval="5", success_count="1", timeout="3"}
port                : 8080
protocol            : tcp
src_ip              : "10.0.0.1"
src_mac             : "aa:bb:cc:99:00:01"
status              : online        ← key field, "online" hoặc "offline" hoặc "error"
```

| Field | Mô tả | Healthy | Critical |
|-------|-------|---------|----------|
| `chassis_name` | Chassis nào probe backend này | Match expected per-chassis assignment | Empty |
| `ip` + `port` | Backend address | Match LB backend list | (n/a) |
| `logical_port` | LSP của backend | Match NB | Mismatch |
| `options:interval` | Probe interval (giây) | 5-30s | < 1s (storm), > 60s (slow detect) |
| `options:success_count` | Success consecutive để mark online | 1-3 | > 5 (slow recovery) |
| `options:failure_count` | Failure consecutive để mark offline | 3-5 | < 2 (false positive cao) |
| `status` | Latest probe result | `online` cho healthy backend | `offline` (backend down), `error` (probe failed config) |

### Distributed probe assignment

Mỗi chassis tự assign subset backend probe theo hash `chassis_uuid` + `backend_ip`. Pseudo-code:

```c
// Pseudo-code cho ovn-controller
for (backend in lb_backends) {
    if (hash(my_chassis_uuid, backend_ip) % total_chassis == my_chassis_index) {
        start_probe(backend);
    }
}
```

100 backend × 10 chassis = mỗi chassis probe 10 backend. Nếu 1 chassis chết, 10 backend mất probe → ovn-controller chassis khác pick up qua re-hash.

### TCP SYN probe semantic

Probe loại `tcp` gửi SYN tới `backend_ip:port`. 3 outcome:

- **SYN-ACK**: backend healthy. Mark `online`, RST connection.
- **RST/ICMP unreachable**: backend reject, mark `offline`.
- **No response (timeout)**: mark `offline` after `failure_count` consecutive timeout.

Probe loại `icmp`: ping. Loại `sctp`: SCTP INIT. Production thường dùng `tcp` cho L4 service.

- Backend `online` trong NB nhưng client get connection refused: NB-SB sync lag. Check `nb_cfg` (Phần 10.0).

## 13.9.10 Capstone POE: 5-tuple hash vs Maglev hash cho production LB

> **Bối cảnh:** Cluster OVN với LB phục vụ 100K connection/giây. Backend pool 10 server. Khi 1 backend bị remove (scale down), bao nhiêu connection bị remap?

**Tuyên bố cần kiểm chứng:** "OVN dùng 5-tuple hash modulo backend count → khi 1 backend remove, ~50% connection remap (modulo math)."

**Predict:** Đúng cho **connection mới**, sai cho **connection đã established** (do conntrack pinning).

**Observe (giả định):**

- Setup LB với 10 backend, generate 100K active TCP connection.
- Remove 1 backend qua `ovn-nbctl set Load_Balancer lb1 vips:'10.0.0.100:80'='10.0.0.11:8080,...,10.0.0.20:8080'` (9 backend).
- Wait 5 giây cho ovn-northd compile + ovn-controller install flow.
- Đo:
  - Connection mới (sau update): hash 9 backend, distribution đều 11.1% per backend.
  - Connection cũ (đang ESTABLISHED): vẫn giữ backend cũ qua conntrack entry. **0% remap.**
  - Connection thuộc backend đã remove: **100% drop** (backend không còn).

So với "modulo math 50% remap": chỉ apply cho **stateless hash** không có conntrack pinning. OVN có conntrack → connection cũ giữ pinning.

### Maglev hash so sánh

Maglev (Google) là consistent hashing scheme khác. Khi backend remove:

- **Maglev:** ~1/N connection remap (1/10 = 10% trong case này).
- **OVN modulo:** 0% remap connection cũ (conntrack pin), nhưng connection mới redistribute toàn bộ.
- **kube-proxy iptables `-m statistic`:** ~50% remap (nu pure statistical).

OVN choice là pragmatic: rely on conntrack thay vì algorithm sophisticated. Trade-off: simpler, nhưng connection rate cao + backend churn → cache memory pressure.

**Explain:**

1. **OVN không phải pure modulo hash**: conntrack entry pin connection vào backend, immune to hash modulo change.
2. **Connection mới remap full** sau backend update là acceptable (5-second interval, application reconnect).
3. **Maglev không cần** cho OVN scale typical (10-100 backend). Modulo + conntrack pinning đủ tốt.

**Falsification test:**

- Run 100K active connection, remove 1 backend, observe drop count.
- Connection drop = số connection thuộc backend removed (10K = 10%).
- Connection retained = các backend còn lại (90K).
- Nếu drop rate > 10% → conntrack pinning broken hoặc connection chưa ESTABLISHED.

**Tuyên bố sai ở đâu:** "50% remap modulo math" giả định stateless hash. OVN có conntrack pinning → connection cũ không bị remap. Đúng phải: **drop = connection của backend removed (10% trong 10-backend case). Conn mới redistribute full.**

> **Quy tắc:** OVN LB scale tốt cho 10-100 backend với churn rate thấp. Cho churn cao + connection-rate cao, consider Maglev hash hoặc external LB (HAProxy/Nginx).

**1.** Tạo topology:
```
[host]# ovn-nbctl ls-add ls1
[host]# ovn-nbctl lsp-add ls1 client
[host]# ovn-nbctl lsp-set-addresses client "aa:00:00:00:00:01 10.0.0.10"
[host]# ovn-nbctl lsp-add ls1 backend1
[host]# ovn-nbctl lsp-set-addresses backend1 "aa:00:00:00:00:02 10.0.0.11"
[host]# ovn-nbctl lsp-add ls1 backend2
[host]# ovn-nbctl lsp-set-addresses backend2 "aa:00:00:00:00:03 10.0.0.12"
```

**2.** Tạo Load Balancer:
```
[host]# ovn-nbctl lb-add lb1 10.0.0.100:80 "10.0.0.11:8080,10.0.0.12:8080" tcp
[host]# ovn-nbctl ls-lb-add ls1 lb1
```

**3.** Quan sát Logical Flow trong table `ls_in_lb`:
```
[host]# ovn-sbctl lflow-list ls1 | grep "ls_in_lb"
```

Kết quả mong đợi:
```
table=12 (ls_in_lb), priority=100,
  match=(ct.new && ip4.dst == 10.0.0.100 && tcp.dst == 80),
  action=(...ct_lb(backends=10.0.0.11:8080,10.0.0.12:8080)...)
```

**4.** Quan sát OpenFlow rule trên br-int:
```
[host]# ovs-ofctl dump-flows br-int | grep "tp_dst=80" | head -5
```

**5.** Gửi traffic và quan sát conntrack:
```
[host]# conntrack -L -d 10.0.0.100
tcp  ESTABLISHED src=10.0.0.10 dst=10.0.0.100 sport=54321 dport=80
  [REPLY] src=10.0.0.11 dst=10.0.0.10 sport=8080 dport=54321
  mark=0 zone=0 use=1
```

Conntrack entry cho thấy DNAT: packet gửi đến `10.0.0.100:80` được chuyển hướng sang `10.0.0.11:8080`, reply có source = `10.0.0.11:8080` được ánh xạ lại thành `10.0.0.100:80` (un-DNAT).

**Bài học học được:**

Thứ nhất, LB của OVN dựa hoàn toàn vào conntrack, không có state riêng cho LB. Điều này có nghĩa tất cả tính năng conntrack (timeout, zone, mark) áp dụng cho LB traffic.

Thứ hai, hash backend được thực hiện bởi action `ct_lb` trong OpenFlow, là action đặc biệt của OVS hỗ trợ cân bằng tải native trong kernel datapath. Không cần userspace can thiệp → hiệu năng cao.

Thứ ba, khi backend bị remove (ví dụ `ovn-nbctl lb-add lb1 10.0.0.100:80 "10.0.0.12:8080" tcp`), conntrack entries cũ vẫn giữ nguyên, connection hiện tại tiếp tục tới backend cũ (đã terminate) và sẽ timeout. Connection mới hash lại và chọn từ danh sách mới.

**Cleanup:**
```
[host]# ovn-nbctl lb-del lb1
[host]# ovn-nbctl ls-del ls1
```

## 13.9.X Backfill v3.5 — `selection_fields` + `hairpin_snat_ip` + `Load_Balancer_Group` + `Load_Balancer_Health_Check`

> **Mục đích:** đợt audit J.1 phát hiện 4 LB feature gap: `selection_fields` (2 file shallow), `hairpin_snat_ip` (2 file shallow), `Load_Balancer_Group` (1 file shallow), `Load_Balancer_Health_Check` (3 file shallow). Section này backfill, source verified `ovn-nb(5)` man page.

### 13.9.X.1 `Load_Balancer.selection_fields` — consistent hashing

**Bucket:** NB schema cho LB consistent hashing.
**Field:** `Load_Balancer.selection_fields` (set of strings)
**Default:** empty = use 5-tuple hash (eth_src, eth_dst, ip_src, ip_dst, ip_proto).
**Effect:** Override hash key cho backend selection. Cho phép pinning theo subset field.

**Diễn giải vận hành:**

LB DNAT chọn backend bằng hash 5-tuple mặc định. Mỗi packet trong cùng connection (cùng 5-tuple) hash → cùng backend → consistent. Vấn đề: client behind NAT có cùng dest+proto+src_port+dst_port (NAT giữ nguyên) nhưng src_ip thay đổi mỗi connection → hash khác → mỗi connection đi backend khác. Cho session-based application (sticky session) đây là vấn đề.

`selection_fields` cho phép pin theo subset field. Set `selection_fields=["ip_src"]` → chỉ hash theo source IP → mọi connection từ cùng client IP đi cùng backend (sticky).

**Available fields (verified `ovn-nb(5)`):**

| Field | Mục đích |
|-------|----------|
| `eth_src` | Source MAC |
| `eth_dst` | Dest MAC |
| `ip_src` | Source IP |
| `ip_dst` | Dest IP |
| `tp_src` | L4 source port |
| `tp_dst` | L4 dest port |

**Example:**

```bash
# Sticky session theo source IP
ovn-nbctl set Load_Balancer web-lb selection_fields="ip_src"

# Hash theo (source IP + dest port) - useful cho TLS SNI multi-port
ovn-nbctl set Load_Balancer web-lb selection_fields='ip_src,tp_dst'
```

**Trade-off:**
- Default 5-tuple: balance load tốt + connection-level sticky.
- `ip_src` only: client-level sticky + có thể uneven load (1 client to → 1 backend hot).
- Custom subset: balance giữa stickiness + load distribution.

### 13.9.X.2 `Load_Balancer.options:hairpin_snat_ip` — hairpin NAT source IP

**Bucket:** NB schema cho LB hairpin handling.
**Field:** `Load_Balancer.options:hairpin_snat_ip` (IP address)
**Default:** không set = no SNAT cho hairpin
**Effect:** Khi backend = client (hairpin), SNAT source IP về giá trị này.

**Diễn giải vận hành:**

Hairpin LB là khi client + backend cùng LSP (rare nhưng possible: pod A access service VIP, service backend là chính pod A). Mặc định LB DNAT VIP → backend IP. Nhưng nếu source IP còn là client (= backend), backend nhận packet với source = chính nó → confusion (loop hoặc drop tự-detect).

`hairpin_snat_ip` rewrite source IP cho hairpin packet, làm backend nhìn thấy "external" client IP thay vì chính nó.

**Workflow stage 13-15 LS_IN pipeline (xem Phần 13.16 §13.16.4):**

1. Stage 13 `LS_IN_PRE_HAIRPIN`: detect hairpin (client + backend cùng LSP). Set `REGBIT_HAIRPIN`.
2. Stage 14 `LS_IN_NAT_HAIRPIN`: nếu REGBIT set + `hairpin_snat_ip` configured → SNAT source IP về `hairpin_snat_ip`.
3. Stage 15 `LS_IN_HAIRPIN`: redirect packet về backend (chính client).

**Example:**

```bash
# Set hairpin SNAT IP cho LB web-lb (giả lập "external proxy IP")
ovn-nbctl set Load_Balancer web-lb options:hairpin_snat_ip=169.254.169.254
```

Backend nhận packet với source IP=169.254.169.254 (link-local, recognizable as "internal proxy"), không phải IP của chính nó.

**Lưu ý:** chỉ áp dụng cho hairpin (rare case). Non-hairpin LB không bị ảnh hưởng.

### 13.9.X.3 Table `Load_Balancer_Group` — LB aggregation cho LR

**Bucket:** NB schema cho aggregate LB application.
**Schema:** Table `Load_Balancer_Group` trong NB DB.
**Columns chính:**
- `name` (string): tên group
- `load_balancer` (set of UUID): list of Load_Balancer entry trong group

**Diễn giải vận hành:**

`Load_Balancer_Group` cho phép aggregate nhiều Load_Balancer entry thành 1 unit, áp dụng đồng thời cho 1 Logical_Router (hoặc nhiều Logical_Switch).

Use case: Logical_Router gateway có 50 service VIP. Thay vì gán 50 LB entry cho LR (tăng độ phức tạp NB schema + slow compile), tạo `Load_Balancer_Group` chứa 50 entry, gán 1 group cho LR.

**Example:**

```bash
# Tạo LB group
ovn-nbctl create Load_Balancer_Group name=web-services-group

# Add LB vào group
LB1=$(ovn-nbctl --bare --columns=_uuid find Load_Balancer name=web-lb)
LB2=$(ovn-nbctl --bare --columns=_uuid find Load_Balancer name=api-lb)
ovn-nbctl add Load_Balancer_Group web-services-group load_balancer $LB1 $LB2

# Apply group cho LR
ovn-nbctl set Logical_Router lr-main load_balancer_group=web-services-group
```

So với attach từng LB:
```bash
# Old way: attach từng LB vào LR
ovn-nbctl lr-lb-add lr-main $LB1
ovn-nbctl lr-lb-add lr-main $LB2
# ... 50 lệnh
```

Group way: 1 lệnh + manage centrally. Cập nhật group → tất cả LR dùng group tự động pick up.

### 13.9.X.4 Table `Load_Balancer_Health_Check` — backend health probe

**Bucket:** NB schema cho LB backend health monitoring.
**Schema:** Table `Load_Balancer_Health_Check` trong NB DB.
**Columns chính:**
- `vip` (string): VIP cần health check
- `options` (map):
  - `interval`: probe interval seconds (default 10)
  - `timeout`: probe timeout seconds (default 5)
  - `success_count`: số probe success liên tiếp để mark UP (default 1)
  - `failure_count`: số probe failure liên tiếp để mark DOWN (default 3)

**Diễn giải vận hành:**

OVN tích hợp Service_Monitor (bảng SB DB) để probe backend periodically. Mỗi backend có session probe TCP/UDP/ICMP. Failure count vượt threshold → mark backend DOWN → exclude khỏi forwarding (LB DNAT skip backend).

**Probe protocol:**
- TCP: TCP SYN → expect SYN-ACK
- UDP: UDP echo-like (limited use case)
- ICMP: ICMP echo

**Example:**

```bash
# Tạo health check cho VIP 10.96.1.10
ovn-nbctl create Load_Balancer_Health_Check vip='"10.96.1.10:80"' options:interval=5 options:timeout=3 options:failure_count=2

# Link với LB
HC_UUID=$(ovn-nbctl --bare --columns=_uuid find Load_Balancer_Health_Check vip='"10.96.1.10:80"')
ovn-nbctl set Load_Balancer web-lb health_check=$HC_UUID

# Verify status (read SB DB Service_Monitor)
ovn-sbctl list Service_Monitor
# Output: status=online/offline per backend
```

**`ip_port_mappings` column:**

`Load_Balancer.ip_port_mappings` (map IP → MAC + LSP) cần thiết khi backend có multiple IP-MAC mapping cho health check source. Format: `"10.0.0.1:fa:16:3e:01:01:01:lsp1"`.

OVN cần biết MAC + LSP của source probe để gửi ARP/ND đúng + reply track về đúng probe session.

**Example:**

```bash
ovn-nbctl set Load_Balancer web-lb \
  ip_port_mappings='10.0.0.1=fa:16:3e:01:01:01\\:lsp-backend-1,10.0.0.2=fa:16:3e:02:02:02\\:lsp-backend-2'
```

## 13.9.Z Load_Balancer như một bảng NBDB: cân bằng tải L4 đào sâu

> Phần này đào sâu vào **chính bảng `Load_Balancer`** (row trong OVN Northbound database), bổ sung cho phần dạy thực hành ở §13.9.1 và pipeline đã giới thiệu ở §13.9.7 + §13.9.X (giữ nguyên). Mục tiêu: kỹ sư hiểu thấu cấu trúc một row NBDB Load_Balancer (8 cột schema 22.03), vòng đời compile thành Logical_Flow trong stage `LS_IN_PRE_LB` + `LS_IN_LB` + `LS_IN_LB_AFF_CHECK` + `LS_IN_PRE_HAIRPIN` + `LS_IN_NAT_HAIRPIN` + `LS_IN_HAIRPIN`, và cách tra ngược từ một sự cố runtime (VIP timeout, backend phân phối lệch, hairpin loop, conntrack stale entry sau scale-down) về một row cụ thể trong NBDB.

### Khái niệm cốt lõi

`Load_Balancer` là **bảng nội trú trong OVN Northbound database (NBDB)** mô tả một **dịch vụ cân bằng tải L4** mà CMS (Neutron LBaaS / Octavia, Kubernetes Service ClusterIP, kube-ovn ovn-loadbalancer, Red Hat OSP director) khai báo để map một hoặc nhiều cặp `VIP:port` tới một danh sách backend `IP:port`. Mỗi row đại diện cho **một** dịch vụ LB, có thể chứa nhiều cặp VIP (cột `vips` là map). Loại: **OVSDB table**, là **root table** (`isRoot=true` trong `ovn-nb.ovsschema` branch-22.03), tức là row tồn tại độc lập không phụ thuộc cha; tham chiếu từ `Logical_Switch.load_balancer` (set UUID, weak ref) hoặc `Logical_Router.load_balancer` (set UUID, weak ref) hoặc `Load_Balancer_Group.load_balancer` (weak ref). Khác biệt then chốt với HAProxy/Nginx: Load_Balancer OVN không phải tiến trình userspace; nó là **declarative entry trong NBDB Raft**, ovn-northd compile thành Logical_Flow, ovn-controller mỗi chassis cài flow OpenFlow vào `br-int`, action `ct_lb` trong OVS kernel datapath thực thi DNAT + chọn backend qua hash. Toàn bộ data plane LB chạy trong kernel; không có context switch userspace; hiệu năng line-rate.

### Lịch sử + bối cảnh ra đời

Bảng `Load_Balancer` xuất hiện cùng OVN 2.6 (2016) khi Russell Bryant (Red Hat) + Han Zhou (eBay/Red Hat) thiết kế feature LB native cho OVN-Kubernetes integration. Trước đó, OpenStack triển khai LB qua **Octavia + HAProxy amphora VM**, mỗi VIP đẻ ra 1 VM amphora riêng chạy HAProxy userspace; vấn đề: 100 VIP = 100 VM = ~50 GB RAM cluster-wide chỉ cho LB, RTT thêm 1-2ms (host→amphora→backend), HA failover qua keepalived chậm 5-10s. Kubernetes triển khai LB qua **kube-proxy iptables `-m statistic`** với hash random (50% remap khi backend thay đổi) hoặc **IPVS** (cải thiện hơn nhưng tách khỏi data plane chính). OVN giải quyết bằng cách compile LB rule xuống `ct_lb` action OF trong OVS, kernel datapath xử lý DNAT + hash trong fast-path, tận dụng conntrack để pin connection vào backend.

Tiến hoá schema (verified [`ovn-nb.ovsschema` branch-22.03](https://raw.githubusercontent.com/ovn-org/ovn/branch-22.03/ovn-nb.ovsschema) + [`ovn-nb.xml` branch-22.03](https://raw.githubusercontent.com/ovn-org/ovn/branch-22.03/ovn-nb.xml)): OVN 2.6 base schema 4 cột (`name`, `vips`, `protocol`, `external_ids`); OVN 2.8 thêm `health_check` (strong ref tới `Load_Balancer_Health_Check`); OVN 2.10 thêm `ip_port_mappings` (map endpoint_ip → mac:port để probe ARP/ND); OVN 21.03 thêm `selection_fields` (set enum 6 giá trị `eth_src`, `eth_dst`, `ip_src`, `ip_dst`, `tp_src`, `tp_dst`) để tuỳ chỉnh hash key, đồng thời thêm `Load_Balancer_Group` table cho gom nhóm; OVN 21.06 thêm `options` map (`hairpin_snat_ip`, `affinity_timeout`, `skip_snat`, `reject`); OVN 22.03 stabilize 8 cột tổng cộng + bổ sung `options:event=true` cho controller-event sampling; OVN 22.09 thêm `options:add_route=true` cho LR LB tự thêm static route tới VIP; OVN 23.03 deprecate `ip_port_mappings` cho hầu hết use case (chỉ còn dùng cho probe nguồn rõ ràng); OVN 24.03 thêm `options:template=true` cho template LB cluster-wide.

### Vị trí trong kiến trúc

Bảng nằm ở **layer NBDB**, là **root table** input layer trên cùng của control plane OVN (khác `ACL` không root). Một cluster OVN có thể có hàng trăm tới hàng nghìn row `Load_Balancer` (1 row per Service ClusterIP trong cluster Kubernetes lớn, hoặc 1 row per VIP trong OpenStack Octavia deployment). Scope tham chiếu: row được trỏ tới qua **weak ref** từ `Logical_Switch.load_balancer` (set UUID) hoặc `Logical_Router.load_balancer` (set UUID) hoặc gián tiếp qua `Load_Balancer_Group.load_balancer` rồi `Logical_Switch.load_balancer_group` / `Logical_Router.load_balancer_group`. Vì là weak ref, xoá Logical_Switch/Logical_Router không cascade xoá Load_Balancer (khác ACL); row Load_Balancer vẫn tồn tại tới khi CMS gọi `ovn-nbctl lb-del` explicit. Scope tác động sau compile: thay đổi một row Load_Balancer (thêm backend, đổi protocol, set option) làm ovn-northd recompute Logical_Flow của **mọi datapath** (LS hoặc LR) tham chiếu LB đó; ovn-controller trên các chassis tương ứng cài lại flow trong < 1 giây qua I-P engine. Không bao giờ LB được áp lên `Logical_Switch_Port` hoặc `Logical_Router_Port` cấp port (chỉ datapath-level).

### Vai trò + nhiệm vụ

Load_Balancer gánh **năm vai trò** trong kiến trúc OVN:

1. **Khai báo dịch vụ L4 declarative**: cột `vips` (map "VIP:port" → "backend1:port1,backend2:port2,...") là intent gốc, ngôn ngữ chung giữa CMS (Octavia, kube-ovn) và data plane.
2. **Chọn protocol L4**: cột `protocol` enum (`tcp`, `udp`, `sctp`, optional rỗng = TCP default) quyết định northd compile match expression `tcp.dst==X` / `udp.dst==X` / `sctp.dst==X` và dùng đúng conntrack helper.
3. **Tuỳ chỉnh thuật toán chọn backend**: cột `selection_fields` enum set ghi đè hash key mặc định 5-tuple. Set `["ip_src"]` cho sticky-session theo source IP; set `["ip_src", "tp_dst"]` cho TLS SNI multi-port. Triển khai consistent hashing thực sự (OVN 21.03+); ovn-northd emit `ct_lb_mark()` với `hash_fields` tham số.
4. **Gắn health probe phân tán**: cột `health_check` (strong ref tới `Load_Balancer_Health_Check`) khai báo VIP nào cần probe. Mỗi chassis tự assign subset backend probe; kết quả ghi vào `Service_Monitor` (SBDB), northd đọc để loại backend `offline` khỏi `vips` compile output.
5. **Truyền tham số xử lý đặc thù qua `options`**: `hairpin_snat_ip` (IP rewrite source khi backend = client cùng LSP), `skip_snat=true` (DSR mode, backend tự reply VIP), `reject=true` (gửi RST/ICMP unreachable thay vì drop khi không có backend healthy), `affinity_timeout=N` (sticky session theo client IP với TTL N giây), `event=true` (emit OVS controller event), `add_route=true` (LR auto add static route tới VIP).

### Vì sao sinh ra (động cơ thiết kế)

Trước OVN 2.6, kiến trúc OpenStack Octavia + HAProxy amphora có pain points: (a) **VM amphora overhead**: 100 VIP = 100 VM, tốn ~50 GB RAM + 10 vCPU dedicated cluster-wide; (b) **Hairpin RTT cao**: client → host vSwitch → amphora VM (cùng host hoặc remote) → HAProxy userspace → kernel TCP stack → backend → ngược lại = 6 context switch + 2 user-kernel boundary, RTT thêm 1-2ms; (c) **HA failover chậm**: keepalived multicast VRRP trên mạng tenant 5-10s chuyển amphora active/standby, gây connection drop; (d) **Tách biệt khỏi mặt phẳng data chính**: amphora không chia sẻ flow rule với OVS; debug LB phải đăng nhập amphora xem `haproxy.cfg` riêng. Với Kubernetes, **kube-proxy iptables** dùng `-m statistic --probability` random (modulo backend count): khi 1/10 backend remove, 50% rule remap → 50% connection mới đi sai backend; **IPVS** cải thiện distribution nhưng vẫn tách khỏi data plane CNI.

OVN giải quyết bằng cách **biểu diễn LB declarative trong NBDB Raft + để northd compile thành Logical_Flow + để mỗi ovn-controller cài flow OpenFlow + dùng action `ct_lb` trong kernel OVS datapath để DNAT + chọn backend trong fast-path**. Lợi ích: không VM amphora, không context switch userspace, hash trong kernel, conntrack pin connection vào backend (immune to hash modulo change), HA failover đi cùng OVN northd Raft (~1s). Bench thực tế Red Hat OSP 17 + ovn-kubernetes 4.14 (2023): 100K connection/giây, p99 latency thêm < 0.1ms so với forward không LB; 1 backend remove → 0% connection cũ remap, 100% connection mới redistribute đúng (xem axis 17 ở §13.9.10 cho phân tích chi tiết).

### Vấn đề mà nó giải quyết

Bảng này giải quyết **năm vấn đề kỹ thuật** rạch ròi: (1) **Cân bằng tải L4 không cần tiến trình userspace riêng**: action `ct_lb` thực thi trong kernel openvswitch.ko, tận dụng nf_conntrack có sẵn, line-rate; (2) **Connection pinning bền vững khi backend thay đổi**: conntrack entry lưu backend đã chọn cho mỗi connection ESTABLISHED, hash modulo thay đổi không remap connection cũ (chỉ connection mới redistribute) — khác biệt cốt lõi vs kube-proxy iptables `-m statistic` (50% remap); (3) **Health check phân tán scale linear**: mỗi chassis tự probe subset backend (hash assignment), không có single point probe; thêm chassis = capacity probe tăng tuyến tính; (4) **Hairpin handling chuẩn**: khi client + backend cùng LSP, OVN tự detect ở stage `LS_IN_PRE_HAIRPIN` (set REGBIT_HAIRPIN), SNAT source IP về `hairpin_snat_ip` ở `LS_IN_NAT_HAIRPIN` để backend không thấy chính nó là source — không cần workaround userspace như HAProxy `option transparent`; (5) **Sticky session theo client IP với TTL**: `options:affinity_timeout=N` lưu mapping `(client_ip, vip) → backend` trong table OVN nội bộ với TTL, không phụ thuộc cookie L7 hay HTTP session id.

### Tầm quan trọng

Đây là **bảng cốt lõi tier 1 của NBDB**, ngang hàng với `Logical_Switch`, `Logical_Router`, `ACL`, `NAT`. Lý do: 100% deployment OVN-Kubernetes có Service ClusterIP đều dùng Load_Balancer (không có alternative); 100% deployment OpenStack Octavia với OVN provider driver dùng Load_Balancer thay amphora; mọi sự cố "VIP không phản hồi", "backend phân phối không đều", "hairpin loop", "connection drop khi scale-down" đều quy về một row Load_Balancer hoặc một entry `Service_Monitor`. Kỹ sư không hiểu Load_Balancer sẽ không debug được cluster Kubernetes trên ovn-kubernetes (vì kube-proxy bị thay thế), không debug được OpenStack với Octavia OVN provider driver. Trong 320+ keyword của REF, Load_Balancer nằm trong nhóm 15 keyword cornerstone OVN (cùng `Logical_Switch`, `Logical_Router`, `ACL`, `NAT`, `Logical_Flow`, `Datapath_Binding`, `Port_Binding`, `ovn-northd`, `ovn-controller`).

### Cơ chế hoạt động (chu kỳ compile một row)

Một row Load_Balancer đi qua **bốn giai đoạn** trong vòng đời:

1. **Khai báo từ CMS**: CMS gọi `ovn-nbctl lb-add web-lb 10.0.0.100:80 10.0.0.11:8080,10.0.0.12:8080 tcp` → NBDB transaction tạo row `Load_Balancer` với `name="web-lb"`, `vips={"10.0.0.100:80"="10.0.0.11:8080,10.0.0.12:8080"}`, `protocol="tcp"`. Tiếp `ovn-nbctl ls-lb-add ls-tenant1 web-lb` thêm UUID web-lb vào set `Logical_Switch.load_balancer`. Transaction atomic; row visible trong < 50ms qua NBDB Raft replicate.

2. **Northd phát hiện thay đổi + classify + compile**: ovn-northd (chạy trên Raft leader của NBDB + write SBDB) phát hiện thay đổi qua I-P engine. Hàm [`ovn_northd_lb_create`](https://github.com/ovn-org/ovn/blob/branch-22.03/lib/lb.c) trong `lib/lb.c` (dòng 165+) parse row NBDB thành in-memory struct `ovn_northd_lb` chứa parsed VIP + backend list + selection_fields enum + options map. Tiếp [`build_lrouter_lbs`](https://github.com/ovn-org/ovn/blob/branch-22.03/northd/northd.c) (`northd/northd.c` dòng 4122+) iterate qua mỗi datapath có LB gắn. Cho mỗi row LB, hàm [`build_lb_rules`](https://github.com/ovn-org/ovn/blob/branch-22.03/northd/northd.c) (dòng 7057+) sinh Logical_Flow ở các stage:
   - `LS_IN_PRE_LB` (logical table 6 / OF table 14): match `ip` → `reg0[2] = 1; ct_next;` (đánh dấu cần đi qua conntrack lookup).
   - `LS_IN_LB` (logical table 12 / OF table 20): match `ct.new && ip4.dst==<VIP> && tcp.dst==<port>` → action sequence build qua hàm [`build_lb_vip_actions`](https://github.com/ovn-org/ovn/blob/branch-22.03/northd/northd.c) (dòng 3799+): set `reg1=<original DIP>`, `reg2[0..15]=<original port>`, gọi `ct_lb_mark(backends=10.0.0.11:8080,10.0.0.12:8080; hash_fields=<selection_fields>)`.
   - `LS_IN_LB_AFF_CHECK` (logical table 13): nếu `affinity_timeout > 0`, sinh thêm flow tra learn-table affinity trước hash mới.
   - `LS_IN_PRE_HAIRPIN` + `LS_IN_NAT_HAIRPIN` + `LS_IN_HAIRPIN` (logical table 13/14/15 → OF table 21/22/23): hàm [`build_lb_hairpin`](https://github.com/ovn-org/ovn/blob/branch-22.03/northd/northd.c) (dòng 7182+) sinh flow detect hairpin (client + backend cùng LSP) + SNAT source theo `options:hairpin_snat_ip` + redirect về backend.
   - `LS_IN_STATEFUL` (logical table 14): commit conntrack với `ct_label.natted=1` để reply path tự un-DNAT.

3. **Health-check flow phụ trợ**: nếu row có `health_check` set, hàm [`ovn_lb_svc_create`](https://github.com/ovn-org/ovn/blob/branch-22.03/northd/northd.c) (dòng 3719+) tạo entry `Service_Monitor` trong SBDB cho mỗi backend. Mỗi chassis tự pick subset entry probe theo hash `(chassis_uuid, backend_ip)`. Probe gửi TCP SYN/ICMP ECHO/SCTP INIT theo `protocol`; ghi `status=online/offline` vào `Service_Monitor.status`. Northd I-P engine theo dõi thay đổi; nếu một backend đổi `offline`, recompute `LS_IN_LB` flow loại backend đó khỏi `ct_lb_mark` backends list.

4. **Cài flow trên chassis**: SBDB Raft replicate Logical_Flow → ovn-controller trên mọi chassis subscribe → mỗi controller chạy `lflow_run` (`controller/lflow.c`) translate Logical_Flow expression sang OpenFlow rule cụ thể, cài lên `br-int` ở các OF table 14/20/21/22/23. Action `ct_lb_mark` trong Logical_Flow được dịch thành OVS action `ct(commit, nat(dst=10.0.0.11:8080), ...)` ở mức datapath; kernel openvswitch.ko thực thi DNAT + hash chọn backend khi packet hit flow.

Invariant: tại mọi thời điểm, một row Load_Balancer trong NBDB tương ứng **một tập Logical_Flow** (~6-15 flow tuỳ stateful + hairpin + affinity) trong SBDB, và cùng tập OpenFlow rule trên mọi chassis có Port_Binding thuộc datapath gắn LB. Xoá row Load_Balancer (qua `ovn-nbctl lb-del`) → ovn-northd recompute, các Logical_Flow cũ bị evict, ovn-controller xoá flow OF tương ứng + flush conntrack entry liên quan trong < 1 giây.

### Phân loại cột

8 cột của Load_Balancer (verified [`ovn-nb.ovsschema` branch-22.03](https://raw.githubusercontent.com/ovn-org/ovn/branch-22.03/ovn-nb.ovsschema)) chia thành **bốn nhóm** theo vai trò ngữ nghĩa:

| Nhóm | Cột | Type | Required | Ghi chú |
|------|-----|------|----------|---------|
| Định danh | `name` | string | Không | Friendly identifier, optional, dùng trong `ovn-nbctl lb-list` output |
| Định danh | `external_ids` | map string→string | Không | CMS-tracked: `neutron:lb_id`, `kube-ovn:svc-name`, `octavia:listener-id` |
| Match logic | `vips` | map "VIP:port"→"backend1:port1,...,backendN:portN" | Có (≥0) | Một row có thể chứa nhiều VIP (rare); thông thường 1 cặp |
| Match logic | `protocol` | enum {tcp, udp, sctp, []} | Không | Rỗng coi như tcp default; quyết định match expression compile |
| Forwarding | `selection_fields` | enum set ⊂ {eth_src, eth_dst, ip_src, ip_dst, tp_src, tp_dst} | Không | Default rỗng = 5-tuple full; set custom subset cho sticky |
| Forwarding | `ip_port_mappings` | map "endpoint_ip"→"MAC:LSP" | Không | Cần thiết khi OVN cần biết MAC + LSP của backend để gửi ARP/ND probe đúng (ít dùng từ 23.03+) |
| Health/Option | `health_check` | strong ref set tới `Load_Balancer_Health_Check` | Không | Có thì tạo entry Service_Monitor cho mỗi backend |
| Health/Option | `options` | map string→string | Không (21.06+) | `hairpin_snat_ip`, `skip_snat`, `reject`, `affinity_timeout`, `event`, `add_route`, `template` |

Schema `"isRoot": true` (verified) — Load_Balancer là root table, tồn tại độc lập, weak ref từ Logical_Switch/Logical_Router/Load_Balancer_Group. Không có index unique trên `vips` — về schema có thể có hai row cùng VIP gắn cùng datapath (CMS phải tự đảm bảo không trùng để tránh non-deterministic compile).

### Cách kỹ sư vận hành thành thạo

Bốn kỹ năng nền tảng kỹ sư phải có với Load_Balancer:

1. **Đọc một row đầy đủ**: `sudo ovn-nbctl --columns=name,vips,protocol,selection_fields,health_check,ip_port_mappings,options,external_ids list Load_Balancer` cho thấy 8 cột đầy đủ. Hoặc shortcut `sudo ovn-nbctl lb-list` cho human-readable list mọi LB, `sudo ovn-nbctl lb-list web-lb` cho 1 row + cross-reference attached LS/LR.
2. **Xác minh compile xuống SBDB**: `sudo ovn-sbctl lflow-list ls-tenant1 | grep -E "ls_in_(pre_lb|lb|lb_aff|pre_hairpin|nat_hairpin|hairpin)"` xác nhận có Logical_Flow ở 6 stage tương ứng. Tra `ct_lb_mark` action với backends list khớp với `vips` map trong NBDB là xác minh compile đúng.
3. **Xác minh OF physical + conntrack runtime**: `sudo ovs-ofctl -O OpenFlow15 dump-flows br-int table=20` (OVN 22.03 mapping `LS_IN_LB` = OF table 20) thấy flow với action `ct(commit, nat(dst=...))`; `sudo ovs-appctl dpctl/dump-conntrack zone=<dp_zone>` thấy connection ESTABLISHED với DNAT từ VIP→backend.
4. **Quyết định khi nào dùng skill nào**: nếu CMS báo "VIP không phản hồi" → bắt đầu skill 1 (xác nhận row tồn tại, vips chứa cặp VIP đúng, attached LS/LR đúng); nếu row tồn tại đúng nhưng client timeout → skill 2 (đối chiếu Logical_Flow xem có generate flow ở `LS_IN_LB`); nếu Logical_Flow đúng nhưng vẫn timeout → skill 3 (xem OF flow trên chassis, xem conntrack entry có bị stale từ scale-down trước). Nếu nghi backend offline mà northd không loại → cross-link `ovn-sbctl list Service_Monitor` xem `status` field.

### Quy trình sử dụng (workflow chuẩn)

Workflow chuẩn cho deploy LB Service ClusterIP với health check + sticky session, mỗi bước có lệnh xác minh:

1. **Tạo row Load_Balancer**: `ovn-nbctl lb-add web-lb 10.0.0.100:80 192.168.1.10:8080,192.168.1.11:8080 tcp`. Xác minh: `ovn-nbctl lb-list web-lb` thấy entry với protocol=tcp, 2 backend; `ovn-nbctl --columns=_uuid,name,vips,protocol list Load_Balancer name=web-lb`.
2. **Gắn LB vào Logical_Switch (hoặc LR)**: `ovn-nbctl ls-lb-add ls-tenant1 web-lb`. Xác minh: `ovn-nbctl --columns=name,load_balancer list Logical_Switch ls-tenant1` thấy UUID web-lb trong set.
3. **Bật sticky session theo client IP**: `ovn-nbctl set Load_Balancer web-lb selection_fields='ip_src' options:affinity_timeout=3600`. Xác minh: `ovn-nbctl --columns=selection_fields,options list Load_Balancer web-lb`.
4. **Tạo Health Check + link**: `HC=$(ovn-nbctl --bare --columns=_uuid create Load_Balancer_Health_Check vip='"10.0.0.100:80"' options:interval=5 options:timeout=3 options:success_count=1 options:failure_count=3)`; `ovn-nbctl set Load_Balancer web-lb health_check=$HC`. Xác minh: `ovn-nbctl --columns=health_check list Load_Balancer web-lb` thấy UUID HC; `ovn-sbctl list Service_Monitor` thấy 2 entry tương ứng 2 backend với `status=online`.
5. **Bật hairpin SNAT cho case backend cùng LSP với client**: `ovn-nbctl set Load_Balancer web-lb options:hairpin_snat_ip=169.254.169.254`. Xác minh: `ovn-nbctl --columns=options list Load_Balancer web-lb` thấy `hairpin_snat_ip=169.254.169.254`.
6. **Chờ northd compile + ovn-controller cài flow**: thường < 1 giây. Xác minh: `ovn-sbctl lflow-list ls-tenant1 | grep ct_lb` thấy flow trong stage `ls_in_lb` với backends list; `ovs-ofctl -O OpenFlow15 dump-flows br-int table=20` thấy flow physical với action `ct(commit, nat(dst=192.168.1.10:8080))`.
7. **Test traffic + theo dõi conntrack + counter**: từ namespace client, `curl 10.0.0.100`. Xác minh: `ovs-appctl dpctl/dump-conntrack zone=<dp_zone> | grep 10.0.0.100` thấy entry ESTABLISHED với reply src=192.168.1.10:8080 (DNAT đã commit); `ovs-ofctl -O OpenFlow15 dump-flows br-int table=20 | grep priority=...` thấy `n_packets` tăng.

Best practice production: **luôn dùng `health_check`** để tránh client gặp connection timeout khi backend chết (mặc định OVN không probe, sẽ tiếp tục DNAT vào backend dead); **set `options:reject=true`** thay vì để traffic vào blackhole khi không có backend healthy (client nhận RST/ICMP unreachable, ứng dụng fail-fast retry); **tránh dùng `selection_fields` chỉ `ip_src`** trừ khi cần sticky bắt buộc — sẽ uneven load khi client behind NAT (nhiều client cùng public IP đi cùng backend); **cân nhắc `options:affinity_timeout`** thay vì `selection_fields` cho sticky session, vì affinity_timeout vẫn dùng full 5-tuple hash cho connection mới ngoài affinity window. Anti-pattern: gắn LB vào quá nhiều LS cùng lúc làm northd compile chậm khi LB thay đổi (bùng nổ Logical_Flow); để `ip_port_mappings` rỗng khi backend ở subnet khác mà có health check (probe không gửi đúng được vì OVN không biết LSP của backend); hardcode hairpin_snat_ip là một IP đang dùng (collision).

### Khi xảy ra sự cố (reverse symptom map)

Bảy symptom thường gặp trỏ về Load_Balancer:

| Symptom | Cột Load_Balancer cần kiểm | Diagnostic flow |
|---------|----------------------------|-----------------|
| "Client timeout khi truy cập VIP" | `vips` map đúng VIP:port? `protocol` đúng? attached LS/LR? | `ovn-nbctl lb-list <lb>`; `ovn-nbctl --columns=load_balancer list Logical_Switch <ls>` xem có UUID lb không |
| "Backend phân phối không đều, 1 backend nhận 80% traffic" | `selection_fields` chỉ có `ip_src` + client behind NAT | Xoá `selection_fields` để dùng full 5-tuple, hoặc dùng `affinity_timeout` thay |
| "Hairpin loop, packet vòng giữa pod và VIP" | `options:hairpin_snat_ip` chưa set | `ovn-nbctl set Load_Balancer <lb> options:hairpin_snat_ip=<unused_link_local_ip>` |
| "Connection cũ tiếp tục đi backend đã remove (stale)" | conntrack entry chưa expire | `ovs-appctl dpctl/flush-conntrack zone=<dp_zone>` flush; tune `nf_conntrack_tcp_timeout_established` thấp hơn cho LB zone |
| "Backend dead vẫn nhận traffic, client mới fail" | `health_check` chưa set hoặc `Service_Monitor.status=error` | `ovn-sbctl list Service_Monitor` xem `status`; xem `chassis_name` có distribute không |
| "DSR mode, client thấy reply có source IP backend không phải VIP" | `options:skip_snat=true` chưa set | Set `options:skip_snat=true` để OVN bỏ qua un-DNAT reply path; backend phải có VIP cấu hình loopback |
| "Service_Monitor.status=offline đột ngột tất cả backend" | network underlay issue, không phải backend | Ping backend từ chassis trực tiếp; tăng `options:timeout` của Health_Check nếu RTT cao |

### Liên quan mật thiết

Sáu thành phần coupling chặt với Load_Balancer (đổi cái nào thì phải xét lại Load_Balancer):

- **`Logical_Switch.load_balancer`** (NBDB): cột weak ref chứa set Load_Balancer UUID. Thêm/xoá UUID = thay đổi tập datapath áp LB. Không cascade lifecycle (xoá LS không xoá Load_Balancer row).
- **`Logical_Router.load_balancer`** (NBDB): cột weak ref tương tự cho LR. LB gắn LR thường là centralized (gateway chassis xử lý) trừ khi `options:add_route=true` cho distributed.
- **`Load_Balancer_Health_Check`** (NBDB): strong ref qua cột `health_check`. Mỗi entry health_check sinh `Service_Monitor` SBDB cho mỗi backend của VIP đó.
- **`Service_Monitor`** (SBDB): mỗi backend có entry với `status` (online/offline/error). Northd I-P engine read status để loại backend offline khỏi compile. Xoá entry → backend coi như healthy default (return về vips list ban đầu).
- **`Load_Balancer_Group`** (NBDB): aggregation; thêm Load_Balancer UUID vào group rồi gắn group cho LR/LS = áp toàn bộ group cùng lúc. Có index unique trên `name`.
- **conntrack zone OVS**: chassis-side. ovn-controller dùng `ct(zone=<datapath_zone_id>, commit, nat(dst=...))`; conntrack entry pin connection vào backend. Stale entry sau scale-down là source of bugs (xem failure mode A ở phần dưới).

Loose coupling (tham khảo, không cascade): `ACL` với `options:apply-after-lb=true` (rule áp sau LB DNAT, match dest IP đã rewrite); `NAT` (cùng dùng conntrack zone nhưng row riêng, không tham chiếu chéo); `Logical_Switch_Port` của backend (LB không trỏ trực tiếp tới LSP, chỉ qua IP).

### Trường hợp sự cố thực tế

**Production case OVN-LB-2024-Q3 (cluster Cloud Provider P-East, ~150 hypervisor, ~3000 pod Kubernetes trên ovn-kubernetes 22.03 + OVN 22.03.8)**: SRE team báo cáo Service ClusterIP `web-svc` (VIP `10.96.10.50:80`, 5 backend pod) phân phối lệch nghiêm trọng: 1 backend nhận ~78% traffic, 4 backend còn lại chia đều ~5.5% mỗi backend. Latency p99 backend hot tăng từ 15ms baseline lên 230ms, các backend khác duy trì 15ms. Triệu chứng xuất hiện sau khi team SRE bật sticky session để fix bug phiên bản auth.

Investigation:

- **Bước 1**: `ovn-nbctl lb-list web-svc` xác nhận row tồn tại với 5 backend; protocol tcp; attached LS đúng.
- **Bước 2**: `ovn-nbctl --columns=selection_fields,options list Load_Balancer web-svc` cho thấy `selection_fields=[ip_src]`, `options={}` (không có `affinity_timeout`). SRE team đã set `selection_fields=ip_src` để sticky thay vì dùng `affinity_timeout`.
- **Bước 3**: `ovn-sbctl lflow-list ls-tenant-prod | grep ls_in_lb` xác nhận flow generated với `hash_fields="ip_src"` (chỉ hash theo source IP, bỏ qua source port + dest IP + dest port).
- **Bước 4**: `kubectl get pods -n web --output=wide` cho thấy 80% client pod (caller của Service) đang chạy trên cùng node N3, qua cùng LSP, có cùng `ip_src` (từ góc nhìn LB Logical_Switch sau egress qua LR là cùng router IP). Vì hash chỉ theo `ip_src`, mọi connection từ N3 đi cùng backend.
- **Bước 5**: `ovs-appctl dpctl/dump-conntrack zone=42 | grep 10.96.10.50 | wc -l` cho thấy 4200 conntrack entry tới backend hot, ~300 mỗi backend khác.

Root cause: `selection_fields=[ip_src]` là sticky **nguy hiểm khi client behind NAT/proxy** — toàn bộ client phía sau cùng public IP đi cùng backend. Trong cluster Kubernetes với SNAT egress qua node, mọi pod cùng node = cùng `ip_src` = cùng backend.

Fix: thay `selection_fields=[ip_src]` bằng `options:affinity_timeout=3600`. `affinity_timeout` cũng cho sticky theo client IP nhưng vẫn dùng full 5-tuple cho connection mới ngoài affinity window, nên 1 client với nhiều port khác nhau vẫn distribute đều theo thời gian. Apply: `ovn-nbctl set Load_Balancer web-svc selection_fields='[]' options:affinity_timeout=3600`. Sau 5 giây northd recompile, traffic redistribute: backend hot giảm từ 78% → 22%, 4 backend khác tăng từ 5.5% → 19.5% mỗi backend. Latency p99 trở về baseline 15ms.

Cross-link: Phần 19.0 (forensic case study), Phần 20.x (LB decision tree), Phần 9.24 (conntrack entry fastpath analysis).

**Bước thực hiện**:

1. **Tạo Load_Balancer + gắn LS**:

   ```bash
   sudo ovn-nbctl lb-add lb-lab 10.0.0.100:80 10.0.0.11:8080,10.0.0.12:8080 tcp
   sudo ovn-nbctl ls-lb-add ls-lab lb-lab
   ```

2. **Xác minh compile**: `sudo ovn-sbctl lflow-list ls-lab | grep -E "ls_in_(pre_lb|lb)"` thấy flow ở stage 6 và 12 với `ct_lb_mark(backends=10.0.0.11:8080,10.0.0.12:8080)`.
3. **Xác minh OF**: `sudo ovs-ofctl -O OpenFlow15 dump-flows br-int table=20 | grep nw_dst=10.0.0.100`.
4. **Test forward (client→VIP)**: từ namespace `vm-client`, chạy `for i in 1 2 3 4 5 6 7 8; do nc -p $((50000+i)) -w 1 10.0.0.100 80 < /dev/null; done` (gửi 8 connection, mỗi connection từ source port khác).
5. **Quan sát hash distribution + conntrack**: `sudo ovs-appctl dpctl/dump-conntrack zone=<dp_zone> | grep 10.0.0.100` thấy 8 entry, distribute giữa 192.168.x.10:8080 và 192.168.x.11:8080. Đếm: thường ~4 entry mỗi backend (5-tuple hash đều).
6. **Quan sát flow counter**: `sudo ovs-ofctl -O OpenFlow15 dump-flows br-int table=20 | grep nw_dst=10.0.0.100` thấy `n_packets` của flow ct_lb_mark tăng theo số connection.

**Biến thể POE**:

- **(a) Sticky theo source IP + so sánh affinity_timeout**: chạy `ovn-nbctl set Load_Balancer lb-lab selection_fields='[ip_src]'`; lặp lại bước 4 với cùng client IP nhưng 8 source port khác. Predict: tất cả 8 connection đi cùng backend (sticky theo ip_src). Observe: `dpctl/dump-conntrack` thấy 8 entry cùng dst backend. Tiếp: đổi sang `selection_fields='[]' options:affinity_timeout=3600`; lặp lại; predict: 8 connection trong 1 giây cùng đi 1 backend (affinity hit), nhưng nếu chờ qua 3600 giây + lặp, có thể đi backend khác (affinity expire). Đây là demo khác biệt giữa 2 cách sticky.
- **(b) Connection pinning khi remove backend**: với 8 connection ESTABLISHED, chạy `ovn-nbctl set Load_Balancer lb-lab vips:'10.0.0.100:80'='10.0.0.12:8080'` (chỉ còn 1 backend). Predict: connection cũ tới backend bị remove (10.0.0.11) sẽ tiếp tục cho tới khi conntrack timeout (default 5 ngày), nhưng backend đã chết → connection cũ stuck. Connection mới hash về backend còn lại. Observe: `dpctl/dump-conntrack` vẫn thấy entry cũ với reply src=10.0.0.11; flush qua `dpctl/flush-conntrack zone=<zone>` để force redistribute.
- **(c) Hairpin SNAT**: thêm rule `ovn-nbctl set Load_Balancer lb-lab options:hairpin_snat_ip=169.254.1.1`; từ namespace `vm-be1` (chính nó là backend) chạy `nc 10.0.0.100 80`. Observe: conntrack entry với src=169.254.1.1 (đã SNAT), không phải 10.0.0.11 → backend nhận packet như đến từ "external proxy".
- **(d) Health check + backend down**: tạo `Load_Balancer_Health_Check`, link, stop process trên backend 1, observe `Service_Monitor.status=offline` sau ~15 giây (3 × 5s), observe northd recompile loại backend khỏi `ct_lb_mark`.

**Falsification check**: Nếu predict "5-tuple hash đều, ~4/4 phân phối" sai (ví dụ ~7/1), đó là do hash collision do nhỏ sample (8 connection); chạy lại với 80 connection nhiều port khác nhau, distribution sẽ tiệm cận 50/50.

### Lỗi thường gặp + tín hiệu chẩn đoán

**Failure mode A, Conntrack stale entry sau scale-down**:

- Triệu chứng: backend đã remove qua `ovn-nbctl set ... vips:...` nhưng client cũ vẫn timeout, không tự retry sang backend mới; counter `n_packets` của flow LB không tăng cho client cũ.
- Diagnostic: `ovs-appctl dpctl/dump-conntrack zone=<dp_zone> | grep <removed_backend_ip>` còn entry. Conntrack timeout default 5 ngày (TCP_ESTABLISHED).
- Recovery: `ovs-appctl dpctl/flush-conntrack zone=<dp_zone>` flush toàn zone (cẩn thận: ảnh hưởng connection khác cùng zone) hoặc filter qua `conntrack -D -d <removed_backend_ip>`. Long-term: tune `nf_conntrack_tcp_timeout_established` thấp hơn cho LB workload (ví dụ 3600s) qua sysctl.

**Failure mode B, Hairpin loop không có hairpin_snat_ip**:

- Triệu chứng: pod A là backend của Service mà chính pod A truy cập VIP đó → packet vòng (DNAT VIP→pod_A_IP, source vẫn pod_A_IP, kernel detect same src+dst drop) → application timeout.
- Diagnostic: `ovs-appctl ofproto/trace br-int 'in_port=<lsp_A>,ip,nw_src=10.0.0.5,nw_dst=10.0.0.100,tcp,tp_dst=80'` cho thấy stage `LS_IN_PRE_HAIRPIN` set REGBIT_HAIRPIN nhưng `LS_IN_NAT_HAIRPIN` không có flow SNAT (vì `hairpin_snat_ip` chưa set).
- Recovery: `ovn-nbctl set Load_Balancer <lb> options:hairpin_snat_ip=169.254.169.254` (link-local IP không bị conflict).

**Failure mode C, Phân phối lệch do `selection_fields=[ip_src]` (case OVN-LB-2024-Q3)**:

- Triệu chứng: 1 backend nhận > 70% traffic, latency p99 tăng 10x, các backend khác idle.
- Diagnostic: `ovn-nbctl --columns=selection_fields list Load_Balancer <lb>` thấy `[ip_src]`; xem topology client xem có behind NAT/proxy với cùng public IP không.
- Recovery: `ovn-nbctl set Load_Balancer <lb> selection_fields='[]' options:affinity_timeout=3600` để dùng 5-tuple hash + sticky theo timeout (cho sticky thật sự nhưng vẫn distribute đều theo thời gian).

**Failure mode D, Service_Monitor không update mặc dù backend dead**:

- Triệu chứng: backend pod đã chết nhưng `ovn-sbctl list Service_Monitor | grep <backend_ip>` vẫn `status=online`; client mới vẫn được DNAT về backend dead, nhận RST từ kernel.
- Diagnostic: kiểm `chassis_name` của Service_Monitor entry — chassis đó có còn lên không? `ovn-sbctl list Chassis` xem; nếu chassis assigned probe đã chết, probe không gửi → status không update.
- Recovery: restart ovn-controller trên chassis liên quan; hoặc nếu chassis chết hoàn toàn, `ovn-sbctl chassis-del <dead_chassis>` để probe re-assign sang chassis sống.

Diagnostic mantra cho Load_Balancer: **"lb-list → lflow-list ls_in_lb → ovs-ofctl dump-flows table=20 → dpctl/dump-conntrack zone → Service_Monitor"**, theo đúng thứ tự từ NBDB declarative xuống chassis runtime + health probe.

### So sánh với hệ khác

| Hệ tương đồng | Điểm chung | Khác biệt then chốt |
|---------------|-----------|---------------------|
| **HAProxy / Nginx userspace LB** | Cùng concept VIP → backend pool, health check, sticky session | HAProxy chạy userspace tiến trình, mỗi packet copy giữa kernel-user-kernel; OVN ct_lb chạy hoàn toàn kernel datapath, line-rate; HAProxy có L7 rule (HTTP header, cookie); OVN chỉ L4 |
| **kube-proxy iptables `-m statistic`** | Cùng 5-tuple hash chọn backend, kernel datapath | iptables: hash random modulo backend (50% remap khi backend thay đổi); OVN: conntrack pin connection vào backend (0% remap connection cũ). iptables: O(N) duyệt chain; OVN: O(1) hash + flow lookup |
| **kube-proxy IPVS** | Cùng kernel datapath, hỗ trợ thuật toán nâng cao (LC, WLC) | IPVS: tách module riêng, không integrate flow rule OVS; OVN: cùng data plane với routing/ACL/NAT, debug đồng nhất qua ovn-trace + ofproto/trace |
| **Linux IPVS (lvs trực tiếp)** | Cùng concept director + real server, kernel hash | IPVS có algorithm sophisticated (rr/lc/wlc/sh/dh); OVN chỉ có 5-tuple hash + selection_fields tuỳ biến + affinity_timeout. IPVS quản qua `ipvsadm`; OVN qua `ovn-nbctl lb-add` |
| **AWS ELB (NLB) / GCP Network LB** | Cùng L4 LB, health check, multi-AZ | Cloud LB managed service, abstraction VPC-level; OVN LB self-hosted, abstraction logical switch/router-level. Cloud LB có connection draining + slow start; OVN có affinity_timeout + skip_snat (DSR) |
| **Cisco ACE / F5 BIG-IP** | Cùng L4/L7 LB hardware appliance | Hardware LB chip TCAM dedicated, L7 rule SDK; OVN open source software LB tích hợp với OpenFlow; rẻ + linh hoạt hơn nhưng hiệu năng kém hơn TCAM hardware |
| **Maglev (Google)** | Cùng consistent hashing scheme | Maglev: lookup table fixed size, ~1/N connection remap khi backend thay đổi (truly consistent); OVN: modulo + conntrack pin (connection cũ 0% remap nhưng connection mới redistribute toàn bộ). Maglev tốt hơn cho churn cao + connection-rate cao; OVN đủ cho 10-100 backend churn thấp |
| **Kubernetes Service ClusterIP (qua kube-proxy)** | Cùng abstraction VIP → endpoint pool | Service là khái niệm K8s; LB OVN là implementation. ovn-kubernetes thay kube-proxy, dịch Service → Load_Balancer NBDB tự động; debug LB qua `ovn-nbctl lb-list` thay `iptables -L -t nat` |

Migration mapping (kube-proxy iptables → OVN LB qua ovn-kubernetes): mỗi K8s Service ClusterIP thành 1 row Load_Balancer NBDB (`vips={"<ClusterIP>:<port>"="<endpoint1>:<port>,<endpoint2>:<port>"}`); EndpointSlice update reflect vào `vips` map qua ovn-kubernetes Watch. Migration mapping (Octavia HAProxy amphora → OVN LB qua provider driver `ovn`): mỗi Octavia Listener thành 1 row Load_Balancer (`vips`, `protocol`); Pool members thành backend trong `vips` value; Octavia HealthMonitor thành `Load_Balancer_Health_Check`; xoá hàng trăm VM amphora, RAM cluster giảm ~50 GB.

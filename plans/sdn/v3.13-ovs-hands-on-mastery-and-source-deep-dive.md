### 2.5.5. VMware-specific knobs that affect the experiments

Three VMware Workstation knobs alter OVS behaviour and must be set consistently across all lab VMs:

1. **MAC address generation.** VMware auto-generates MAC addresses with the OUI `00:0C:29` (already visible on `ens33` of `lab-openvswitch`). Every lab VM gets a `00:0C:29:xx:xx:xx` address. The transcripts preserve these exactly.
2. **VMware Tools and `vmxnet3` versus `e1000`.** The default adapter is `e1000`; this is fine for OVS labs. Do not switch to `vmxnet3` unless a sprint explicitly needs the higher throughput, because the kernel-module path and offload behaviour differ. The plan defaults to `e1000` for reproducibility.
3. **Promiscuous mode on host-only networks.** OVS lab tunnels on the lab network (`ens34`) require promiscuous mode to be permitted on the VMware host-only switch. On Windows the policy is set in the Virtual Network Editor; on Linux/macOS hosts it is set per-network. R5.5's enrichment dossier records the exact setting (verbatim screenshot or VMware preference dump) in Section A so future sessions can reproduce it.

## 3. Verbatim integrity policy (mandatory, this plan adds Rule 18 to CLAUDE.md)

The owner's instruction in Vietnamese (2026-04-29): every command, every output, every hostname, every log line must appear in the curriculum exactly as it appeared on the lab host. Not a single character modified. This plan codifies the policy as **Rule 18: Lab Output Verbatim Integrity** in CLAUDE.md.

Specific mandatory practices:

1. The shell prompt is preserved exactly (`root@lab-openvswitch:~#`, `root@lab-openvswitch:/etc/openvswitch#`, etc.).
2. The IP `192.168.1.250` appears as is, not anonymised to `<LAB_IP>` or `192.168.x.x`.
3. The MAC `00:0c:29:9d:fe:15` of `ens33` appears as is.
4. All UUIDs (interfaces, bridges, OVSDB rows) are preserved character-for-character.
5. All log timestamps (`2026-04-29T14:25:18.123Z` etc.) are preserved to the millisecond.
6. The empty kernel-module-grep output is preserved as an empty line (not omitted, not annotated with "no output").
7. If a command produced multi-page output (for example, `ovs-vsctl --help`), the curriculum file includes the full output. To shorten, the plan explicitly states which lines are dropped and why, with the line `[N other lines omitted, context: <reason>]`.
8. Empty stderr is preserved as an explicit empty line below the stdout block, labelled `(stderr empty)`.

Self-check before commit per sprint: open the lab transcript, diff it against the live `~/.bash_history` (or against a file captured by `script -a`), confirm zero divergence.

**Section E: Synthesis, edge cases, and hidden constraints.** The dossier closes with a synthesis section that lists every edge case, hidden constraint, common pitfall, and "expert-only" detail uncovered by sections A through D. Each entry is one paragraph: what the edge case is, where it surfaced (source file or doc URL), what makes it hard, and how the lab exercise will surface it for the learner.

Examples of the kind of detail Section E must capture:

- The exact CLI flag that switches a `set` from "merge" to "replace" semantics (`ovs-vsctl set` versus `ovs-vsctl --if-exists set` versus `ovs-vsctl --no-wait set`).
- The kernel module conflict between the in-tree `openvswitch.ko` and the out-of-tree module shipped with OVS source.
- The exact `vlog` module name to enable to see a specific debug message (`ovs-appctl vlog/set ANY:dbg`, then `vlog/list`, then narrow to the right module).
- The exact `coverage/show` counter that increments when an upcall hits the slow path versus a megaflow miss.
- The boot order race between `ovsdb-server` and `ovs-vswitchd`.
- The schema version incompatibility surface (when an older `ovs-vswitchd` connects to a newer `ovsdb-server`).
- The `other_config` keys that are silently accepted without effect when the corresponding feature is not built in (every silent-no-op key is listed).
- The exact behaviour of the `learn` action when the destination flow table is full.
- The exact bytes consumed by a megaflow entry (so the learner can size a deployment).
- The `ovs-tcpdump` invariant: it creates an internal port and a mirror, and removes them on `Ctrl-C`; on a hard kill the orphan port leaks, recoverable only by manual `ovs-vsctl del-port`.

**Section F: Lab exercise upgrades.** Based on sections A through E, Section F lists the concrete exercise additions for the corresponding R-sprint. Format: each addition is a numbered exercise with a one-line title, a paragraph of pedagogical justification, the verbatim commands to run on `192.168.1.250`, and the expected verbatim output (the prediction is recorded; the actual output is captured later in the lab transcript and any divergence is flagged in Section G of the post-lab review).

The exercise upgrades are explicitly required to go beyond the bare-bones command sequences; they add: failure-mode injection (deliberately misconfigure, observe the error, trace the error to its source line), comparative variants (run the same operation three ways and compare the OVSDB diff), telemetry capture (run `coverage/show` before and after, diff the counter set), source-code annotation (explicitly read the C function the exercise hits, include a verbatim 10-line excerpt with file path and function name), and one "expert challenge" per exercise (a harder variant the learner can attempt unaided).

#### R0.1, Pedagogical sidebar specification, "Why most OVS labs use namespaces, not VMs"

This sidebar is authored as a top-level section of `sdn-onboard/labs/README.md` and is also embedded by reference into `sdn-onboard/0.1 - lab-environment-setup.md`. The sidebar exists because every learner who has seen OVS in a cloud-platform context (OpenStack, KubeVirt, oVirt, libvirt) asks the same question on first encounter with a self-contained tutorial: "where are the VMs?" The sidebar answers the question once, in plain prose, and never has to be answered again.

**Required title:** `Why most OVS labs use network namespaces instead of virtual machines`.

**Required content (the sidebar in full English prose, target 400 to 600 lines, written as natural curriculum, not as conversation):**

The sidebar opens with a statement of the question as the learner experiences it. A reader who has worked with an OpenStack compute node sees real virtual machines hosted on the hypervisor and Open vSwitch wiring those machines together; the mental model is "OVS lives on a hypervisor and connects VMs". Then the reader opens an OVS tutorial and finds two veth pairs, two network namespaces, no virtual machine in sight. The two pictures look incompatible. The sidebar resolves the contradiction.

The sidebar then explains, in this order:

1. **What Open vSwitch actually is.** A virtual L2 and L3 switch that forwards packets between any Linux network interfaces. The interfaces can be physical NICs, veth pairs, tap devices, internal ports, or tunnel ports. A virtual machine's tap interface is one kind of port the switch can attach; it is not a special kind. The switch does not know or care whether the interface on the other side belongs to a virtual machine, a network namespace, a container, or a process running on the host.

2. **Why this matters for learning.** Because OVS treats every port the same way, the lesson "OVS adds a VLAN tag when a packet leaves this port" looks identical whether the port belongs to a virtual machine or to a network namespace. The match-action pipeline is the same. The flow table is the same. The OpenFlow protocol exchange is the same. The kernel datapath is the same. Every byte the learner sees with `tcpdump` looks the same. So the lesson can be taught with whichever port type costs the least to set up.

3. **Why namespaces are usually the cheapest port type.** A network namespace is a feature of the Linux kernel that gives a small piece of code its own private network stack: its own routing table, its own ARP cache, its own firewall rules, its own list of network interfaces. Creating a namespace takes a few milliseconds. Attaching a veth pair to it takes one more command. The namespace consumes essentially no memory beyond a few kilobytes of kernel data structures. By contrast, a virtual machine brings a guest kernel, a guest userland, a virtual disk, a few hundred megabytes of RAM, a boot sequence that takes 10 to 30 seconds, and a separate SSH session to manage. For teaching OVS, this overhead buys nothing: the data-plane behaviour the learner observes is identical.

4. **Why the upstream Open vSwitch project itself uses namespaces in its test suite.** The `make check-system-userspace` test suite ships with Open vSwitch and exercises real datapath behaviour against the kernel module. It contains roughly 200 system tests covering conntrack, tunnels, QoS, learning, ACLs, and more. Every one of these tests uses network namespaces, not virtual machines. The reason is the same: the tests must run in seconds on continuous integration, and they must give a faithful picture of how the production datapath behaves. Namespaces deliver both. The Open vSwitch documentation file `Documentation/topics/networking-namespaces.rst` discusses this design choice explicitly and recommends namespaces as the standard approach for OVS testing and learning.

5. **Where virtual machines are actually needed and why.** There are exercises in the curriculum that genuinely need real virtual machines, not namespaces. The most important is the multi-host scenario: when the lesson is "two hosts each running their own Open vSwitch, with traffic flowing between them across an underlay network", the two hosts must be two distinct kernel instances. A namespace shares its parent host's kernel; it is not a separate host. So an inter-host overlay lab uses two real virtual machines at the host level (each a complete Ubuntu installation running its own ovs-vswitchd daemon and its own kernel datapath), and inside each host the workloads are network namespaces. This is the pattern Plan v3.13 follows: VMs at the host boundary where the boundary actually matters; namespaces inside each host because the workload's network stack is what the lesson cares about.

6. **Where the cloud-platform picture comes from and why it does not contradict the lab picture.** A cloud platform such as OpenStack runs real virtual machines because the platform's purpose is to host tenant workloads. The tenants need full operating systems, full SSH access, full disk persistence, full kernel choice. Open vSwitch is the network plumbing the cloud platform uses to wire those VMs together. So when a learner reads OpenStack documentation, the VMs are in the picture because the documentation is about OpenStack, and Open vSwitch is incidental. When a learner reads an Open vSwitch tutorial, the VMs disappear because Open vSwitch is the subject and the workloads are incidental. Both pictures are accurate; they show the same software from different vantage points.

7. **The substitutes the curriculum uses, in order of fidelity.** A namespace gives a workload its own routing table and ARP cache and conntrack state and firewall, exactly like a VM does at the network layer. A veth pair without a namespace gives two virtual interfaces in the host's default network stack, useful for demonstrating bridge learning. An internal port creates a virtual interface that the host itself can use, useful for demonstrating the management plane. For every OVS lesson that does not specifically depend on a guest operating system feature (PXE booting, real DHCP client behaviour, virtio paravirtualisation, SR-IOV passthrough), these substitutes are indistinguishable from real VMs. The exercises in this curriculum that have a real guest OS dependency are explicitly flagged; everything else is namespaces.

8. **A worked comparison.** The sidebar shows the same lesson taught two ways. First, the namespace version: create two namespaces, attach veth pairs to a bridge, send a ping, observe the flow table populate. Three commands, three seconds. Second, the VM version: spin up two Ubuntu cloud images, configure their network, install SSH keys, attach their tap devices to the bridge, send a ping. Twenty commands, three minutes, and 1 GiB of RAM. Side by side, the OVS view of the world is identical. The flow table looks identical. The `tcpdump` capture looks identical. The `ovs-appctl ofproto/trace` output is identical. The conclusion is unambiguous: for this lesson, the VMs add cost without adding clarity.

9. **When to override the default and use VMs anyway.** The sidebar closes with a short list of conditions under which a learner should ignore the namespace shortcut and bring up real VMs: when the lesson involves a feature of a specific guest OS (Windows networking stack peculiarities, FreeBSD bridge behaviour), when reproducing a production incident that involved a specific virtio driver bug, when measuring real virtio overhead in a performance study, or when the learner simply wants to see the cloud-platform picture end to end and is willing to pay the setup cost. The list is short because the conditions are rare in practice.

The sidebar is written in plain technical English at CEFR B2 to C1 level per Rule 17. No em-dashes per Rule 17. No Vietnamese per Rule 17. Internal-vs-reader-facing language separation per Rule 16: no plan internals (no "R0.1", no "Track A", no "E0", no "v3.13") leak into the sidebar; the sidebar reads as a permanent piece of the curriculum, not as an artefact of a plan.

The sidebar is cross-referenced from the table of contents of `sdn-onboard/labs/README.md` so a reader landing on the labs folder for the first time finds it immediately. It is also cross-referenced from `sdn-onboard/0.1 - lab-environment-setup.md` so a reader following the curriculum reading order encounters it at the moment the lab convention is introduced.

### R2.5, OVS versus traditional network models, comparative architecture

> **Owner directive (translated from Vietnamese original, 2026-04-29 third message):** "How would OpenvSwitch be implemented with traditional computer network models, in detail, and what are the differences between them?"

**Goal:** the owner sees, side by side, how every traditional networking primitive (the kind a Cisco-trained or Juniper-trained engineer carries in muscle memory) is realised inside OVS, and understands the differences in mechanism, capability, configuration surface, observability, and failure mode. The output is a translation handbook: for any sentence an engineer says about a traditional switch, the handbook shows the OVS equivalent.

**Why R2.5 lives here, after R2:** the learner already has hands-on OVS primitives (bridges, ports, VLANs, fail-mode) but has not yet entered the OpenFlow programming surface. This is the right pedagogical moment to anchor those primitives against the traditional network mental model, before the OpenFlow surface re-frames the same operations as match-action pipelines.

**Sub-tasks (each is a hands-on lab section, not just a comparison table):**

1. **L2 learning switch.** Traditional: a chassis switch builds a MAC table from observed source MACs, floods unknown destinations, ages out idle entries. OVS: identical default behaviour with `ovs-vsctl set bridge br0 fail-mode=standalone` plus the implicit normal-action flow. Lab: build an OVS bridge with no flows installed, observe MAC learning via `ovs-appctl fdb/show br0`, age out an entry, compare with `bridge fdb show` for a Linux-bridge equivalent. Explain the normal action's source: `ofproto/ofproto-dpif-xlate.c:xlate_normal()`.

2. **VLAN trunking and access ports.** Traditional: Cisco `switchport mode access` plus `switchport access vlan 10`, `switchport mode trunk` plus `switchport trunk allowed vlan 10,20`. OVS: `ovs-vsctl set port eth1 tag=10` for access; `ovs-vsctl set port eth2 trunks=10,20` for trunk; `ovs-vsctl set port eth3 vlan_mode=native-untagged` for native-VLAN. Lab: build a 3-port bridge (1 access tag-10, 1 trunk 10/20, 1 access tag-20), prove tag rewriting with `tcpdump -e`, compare with a 3-port Linux bridge using `bridge vlan add`. Explain the bridge VLAN translation in `vswitchd/bridge.c:port_configure_vlan()`.

3. **Spanning Tree Protocol (STP) and Rapid STP (RSTP).** Traditional: a full STP control plane elects a root bridge, blocks redundant ports, converges in 30 to 50 seconds (legacy STP) or 1 to 6 seconds (RSTP). OVS: `ovs-vsctl set bridge br0 stp_enable=true` (legacy STP only) or `ovs-vsctl set bridge br0 rstp_enable=true` (RSTP, since OVS 2.4). Lab: connect three OVS bridges in a triangle (using veth pairs), observe initial flooding loop, enable RSTP, observe BPDU exchange via `tcpdump`, observe one port transition to discarding state via `ovs-appctl rstp/show`. Source: `lib/stp.c`, `lib/rstp.c`, `lib/rstp-state-machines.c`. Note explicitly: OVS's STP is intentionally minimal because OpenFlow controllers are expected to handle topology decisions; this is a deliberate architectural difference.

4. **Link aggregation (LAG, LACP).** Traditional: Cisco `EtherChannel` with `channel-group 1 mode active`, Juniper `aggregated-ethernet` interface. OVS: `ovs-vsctl add-bond br0 bond0 eth1 eth2 lacp=active bond_mode=balance-tcp`. Lab: build a 2-link bond between two OVS bridges, send 100 flows, observe the per-flow hash distribution with `ovs-appctl bond/show`, fail one link with `ip link set eth1 down`, observe traffic shift. Compare bond modes: `active-backup`, `balance-slb`, `balance-tcp`. Source: `ofproto/bond.c`, `lib/lacp.c`. Note: OVS supports LACP active and passive but the bond hash is computed in `bond_choose_output_slave()`, with deeper symmetry guarantees than Linux teaming.

5. **Port mirroring (SPAN, RSPAN, ERSPAN).** Traditional: Cisco `monitor session 1 source interface gi0/1 both`, `monitor session 1 destination interface gi0/24`. OVS: `ovs-vsctl -- --id=@p get port eth0 -- --id=@m create mirror name=m1 select-all=true output-port=@p -- set bridge br0 mirrors=@m`. Lab: build a mirror that captures every packet on br0 to a tap port, run `tcpdump` on the tap, compare with Linux `tc` mirred. Source: `ofproto/ofproto-dpif-mirror.c`.

6. **L3 routing (no OVN).** Traditional: a Cisco router has interfaces in subnets, ARP tables, FIB, and routes packets between subnets. OVS: there is no built-in router; routing is achieved either by (a) installing OpenFlow flows that match destination IP and rewrite MAC, decrement TTL (the "raw OVS routing" approach used in lab `Day 5 -Lab 7 Implementing Routing in Open vSwitch.pdf`), or (b) attaching a Linux network namespace as the routing element, with `ovs-vsctl add-port br0 router-port -- set Interface router-port type=internal`, then `ip netns exec` to configure routes. Lab: build a 2-subnet topology with an internal router-namespace, prove ICMP across subnets, then re-implement the same topology with raw OpenFlow flows (no router-namespace), prove ICMP again. Show the `dec_ttl` action in `lib/odp-execute.c:odp_execute_set_action()`. Note: this is the deepest difference from a traditional switch. OVS does not route by default, and routing is either external (via a Linux namespace, which delegates to the kernel routing table) or programmed (via OpenFlow flows, which the controller installs).

7. **Access Control Lists (ACL).** Traditional: Cisco `ip access-list extended PERMIT_HTTP`, applied with `ip access-group PERMIT_HTTP in`. OVS: ACLs are OpenFlow flows with `priority` plus `actions=drop`, or stateful via `ct(commit)` plus `ct_state` matches. Lab: build a 2-port bridge, install a flow that drops all TCP destination port 22, prove with `nc`. Then re-implement statefully so return traffic is allowed: `ct(commit) actions=resubmit(...)`. Source: `lib/conntrack.c`, `ofproto/ofproto-dpif-xlate.c:xlate_ct()`. Note: traditional ACLs are stateless or use an external firewall; OVS unifies both inside the flow table, which is conceptually different.

8. **Network Address Translation (NAT).** Traditional: Cisco `ip nat inside source` plus `ip nat outside`, or a Linux `iptables -t nat`. OVS: `ct(nat(src=10.0.0.1))` or `ct(nat(dst=10.0.0.2:80))`. Lab: build a 2-namespace SNAT, prove with `tcpdump -i tap0` showing source rewriting, compare with `iptables -t nat -A POSTROUTING -s 192.168.0.0/24 -j SNAT --to 10.0.0.1`. Source: `lib/conntrack.c:conn_update()`. Note: OVS NAT is a side effect of the conntrack action, unified with the flow pipeline; iptables NAT is a separate netfilter chain. Performance and observability differ substantially.

9. **Quality of Service (QoS, queues, policers).** Traditional: Cisco `service-policy output QoS_POLICY` with class maps. OVS: `ovs-vsctl set port eth0 qos=@new -- --id=@new create qos type=linux-htb other-config:max-rate=100000000 queues=0=@q0,1=@q1`. Lab: build two queues with different rates, send traffic to each, observe rate enforcement with `iperf3`, compare with `tc qdisc add dev eth0 root htb`. Source: `lib/netdev-linux.c:netdev_linux_set_qos()`.

10. **DHCP, DNS, ARP responder (no OVN).** Traditional: a separate `dhcpd` daemon, a separate `dnsmasq`. OVS: there is no built-in DHCP or DNS responder. The OVN add-on layer provides them; raw OVS does not. Lab: explicitly attempt to set a DHCP responder via OVS alone, observe the absence, then attach `dnsmasq` to an internal port for the same effect. This is the fence post that separates OVS from OVN; OVN is "OVS plus distributed services". Note: this fence post is precisely why OVN exists, per the owner's open question Q2 of the prior message ("OVN is some kind of improvement to OpenvSwitch that I do not yet understand").

11. **Routing protocols (OSPF, BGP, IS-IS).** Traditional: a router runs an IGP/EGP daemon. OVS: there is no built-in routing protocol; OVS is a forwarding plane only. The control plane is delegated to a controller (Faucet, Ryu, ONOS, OpenDaylight) or to OVN-northd or to Open vSwitch's own learning behaviour, never to OVS itself. Note: this is **explicitly out of scope per the permanent ban on BGP** (CLAUDE.md North Star); the comparison is mentioned for completeness as one paragraph only.

12. **The big picture, side-by-side architecture diagram.** A two-column diagram, one column "traditional Cisco/Juniper switch", the other "OVS plus controller". The diagram shows: forwarding plane (ASIC vs kernel/userspace datapath), control plane (CLI/SNMP vs OpenFlow/OVSDB), management plane (NETCONF/SNMP vs OVSDB JSON-RPC), data plane operations (look-up forwarding table vs match-action pipeline), failure isolation (per-port queues vs per-flow conntrack zones), observability (SNMP MIB vs `coverage/show` plus sFlow plus IPFIX). Diagram drawn in SVG at `images/v3.13-R2.5-traditional-vs-ovs-architecture.svg`, captioned in the curriculum file.

**Sub-tasks (every connection type gets a lab section):**

1. **Baseline, no OVS, traditional Linux bridge.** As the control case, two VMs on two hosts connect via a flat Linux bridge plus the underlay (a switch). This establishes the baseline behaviour to compare every OVS path against. ARP works, MAC learning happens on the underlay switch, IP is in the same subnet. Trace packet path: `vm-A-eth0 -> br-A (Linux bridge) -> host-A-eth0 -> physical/virtual switch -> host-B-eth0 -> br-B (Linux bridge) -> vm-B-eth0`.

2. **OVS as a transparent L2 bridge (no tunnel).** Replace `br-A` and `br-B` with OVS bridges. Same subnet, same flat L2. Identical observable behaviour from the VM's view. Lab: `ovs-vsctl add-br br0`, attach the host's physical NIC and the VM's tap device to br0, prove ICMP. Source: `ofproto/ofproto-dpif-xlate.c:xlate_normal()`.

3. **VLAN trunking across hosts.** Two VLANs (10 and 20) on each side. The host-to-host trunk carries both. Each VM is in one VLAN and only sees its peers in the same VLAN. Lab: `ovs-vsctl set port host-eth0 trunks=10,20`, prove isolation by ICMP. Compare the wire bytes: `tcpdump -e -i host-eth0` shows the 802.1Q tag for VLAN 10 and VLAN 20.

4. **Geneve tunnel (overlay, the OVN-style approach).** Each host runs an OVS bridge with a Geneve tunnel port to the other host's IP (`ovs-vsctl add-port br-int gnv0 -- set interface gnv0 type=geneve options:remote_ip=192.168.1.251`). VMs are inside a logically-isolated overlay; the underlay sees only Geneve-encapsulated packets between the two host IPs. Lab: build the topology, prove ICMP between VMs, capture the wire bytes showing the outer UDP header, the Geneve header (TLVs visible with `tcpdump -X`), the inner Ethernet frame. Source: `lib/netdev-vport.c`, `datapath/vport-geneve.c`, `lib/tnl-neigh-cache.c` (the tunnel ARP cache).

5. **VXLAN tunnel.** Same as #4 but with VXLAN. The differences: VXLAN uses UDP destination port 4789 by default, has a fixed-size 8-byte header with a 24-bit VNI, no TLV extensibility. Lab: substitute `type=vxlan options:key=42` for the Geneve port. Compare the wire bytes.

6. **GRE tunnel.** Same as #4 but with GRE (IP protocol 47, no UDP). Compare with VXLAN: GRE has lower per-packet overhead but does not work behind NAT. Lab: substitute `type=gre`.

7. **STT tunnel (a Nicira proprietary tunnel).** STT (Stateless Transport Tunnelling) uses a TCP-shaped header but is not real TCP; it is designed to ride NIC TCP-segmentation-offload (TSO) for fast encapsulation. STT has fallen out of favour and is removed in newer kernels. Mention briefly, do not lab unless the kernel module supports it on `5.15`. Source: `datapath/vport-stt.c`.

8. **L3 inter-host routing through an OVS-internal router-namespace.** Two different subnets; each host's OVS bridge attaches to a Linux network namespace acting as a router. The router runs `ip route` to route between the two subnets. The host-to-host link is the underlay carrying L3 traffic. Lab: build this; prove ICMP across subnets; observe the per-VM ARP behaviour (each VM ARPs for its router-namespace gateway, the router-namespace ARPs for its peer router). Compare with the equivalent OVN flow: OVN's Logical Router achieves the same with no Linux namespace, by encoding the routing decision into OpenFlow flows. R5.5 explicitly demonstrates the contrast.

9. **L3 inter-host routing through raw OpenFlow flows (no router-namespace).** Same topology as #8 but the routing is achieved by OpenFlow flows that match the destination IP, rewrite the destination MAC to the next-hop MAC, decrement TTL, and output to the correct port. Lab: install the flows by hand; prove ICMP; observe `dec_ttl` in action via `ofproto/trace`. This is the "OVS as a router" pattern and is conceptually the same as what OVN northd compiles.

10. **Patch ports plus multiple bridges.** A common topology: each host has two OVS bridges, `br-int` (integration bridge) and `br-tun` (tunnel bridge), connected by a patch-port pair. VMs attach to `br-int`; tunnels live on `br-tun`. This is the OpenStack Neutron OVS layout. Lab: build the topology, observe the patch-port behaviour with `ovs-appctl dpif/show`, `ovs-appctl ofproto/list-tunnels`. Source: `vswitchd/bridge.c:bridge_add_ofproto_ports()`, `lib/netdev.c:netdev_open()` for the `patch` type.

11. **MTU and PMTUD across the inter-host path.** With a 1500-byte underlay MTU and a 50-byte Geneve/VXLAN overhead, the overlay MTU drops to 1450. If the VM has 1500 MTU, large packets fragment or trigger ICMP "fragmentation needed". Lab: deliberately set VM MTU to 1500, send a 1500-byte ping with `ping -s 1472 -M do`, observe ICMP need-frag, then either fix by lowering VM MTU to 1450 or by raising underlay MTU to 1550. Source: `lib/netdev-vport.c:netdev_vport_get_mtu()`.

12. **Tunnel encryption with IPsec.** OVS supports IPsec-protected tunnels via `openvswitch-ipsec` package. Lab: enable IPsec on the Geneve tunnel, capture the wire bytes showing ESP encapsulation. Source: `ipsec/` directory in the OVS source.

13. **MAC table propagation, the L2 learning question.** With a tunnel topology, each side's MAC table only sees its local VM's MAC, not the remote MAC. How does ARP work? Answer: the broadcast ARP is encapsulated and flooded over the tunnel to the remote side, where it is decapsulated and broadcast on the remote bridge. Lab: capture the ARP request on the tunnel wire, observe the encapsulated broadcast; then observe the unicast ARP reply, also encapsulated.

14. **Comparison with OVN's distributed L2/L3.** A short paragraph contrasting raw OVS (manual flow programming, manual tunnel mesh) with OVN (declarative `ovn-nbctl ls-add`, automatic tunnel mesh maintenance, automatic ARP suppression). Two-line summary, no deep OVN content (per the OVN out-of-scope rule for v3.13).

15. **Failure-mode injection for inter-host connectivity.** Five injected failures with diagnostic walkthrough: (a) tunnel destination IP unreachable (host B is down), (b) tunnel destination IP reachable but no OVS bridge listening, (c) MTU mismatch causing intermittent loss, (d) underlay drops only large UDP packets (firewall MSS clamp), (e) tunnel key mismatch (VNI 42 on side A, VNI 43 on side B). For each, the symptom from the VM's perspective, the diagnostic command on the OVS host, the root cause, and the fix.

11. The kernel re-processes the packet against the new megaflow and executes the actions.
12. The packet exits via `ovs_vport_send()` on the egress vport.

## 9. Risks, blockers, mitigations

**Risk 1, lab disk space.** The lab has 4.0 GiB free. A full OVS git clone is about 200 MiB; a build adds about 800 MiB. `make distclean` after each install path keeps it bounded. The kernel build drops about 200 MiB into `/lib/modules/.../updates/` which we leave in place.

Mitigation: monitor `df -h` at the start of each sprint, prune build artifacts proactively.

**Risk 2, transcript sprawl.** A `make check` run produces 2,500 tests of output, possibly 50,000 lines. Transcripts of that size are unreadable inline. Mitigation: store the full log as a referenced text artifact at `sdn-onboard/labs/v3.13-S9-make-check.txt`, embed only an annotated 200-line excerpt in the curriculum file, and follow the Rule 7 "[N other lines omitted, context: full log preserved at <path>]" convention.

**Risk 3, kernel module mismatch.** The lab runs kernel `5.15.0-173-generic`. The OVS source `v2.17.9` ships a kernel module compatibility shim that targets older kernels. Building from source against a 5.15 kernel may need the `compat.h` patches. Mitigation: prefer the in-kernel `openvswitch.ko` (already shipped with Ubuntu 22.04) over the out-of-tree module. Document the trade-off explicitly in S1.

**Risk 4, owner verbatim policy is hard.** It is easy to "tidy" the prompt or trim trailing whitespace. Mitigation: every transcript is captured by `script -f -t` (which records all bytes including CRs and prompt redraws), the lab transcripts include the raw `script` typescript file (referenced, not embedded) plus a human-readable rendering. A diff between rendering and typescript is run per Rule 18; if the diff is non-empty, the rendering is rejected.

**Risk 5, OVS source baseline drift.** The owner directive locks the lab to `v2.17.9`. The upstream repository at `C:/Users/voleh/Documents/ovs` is checked out at `v2.17.9` already. Mitigation: at R0, freeze with `git -C C:/Users/voleh/Documents/ovs status` (must be clean) and `git -C ... rev-parse HEAD` (must equal the SHA of `v2.17.9`). Captured verbatim. No `git checkout` or `git pull` is allowed inside the OVS source worktree for the duration of plan v3.13.

**Risk 6, plan v3.13 collides with the cross-block follow-on plan (templates plus haproxy-onboard).** The owner mentioned that the cross-block plan is queued. v3.13 is OVS-only and only modifies `sdn-onboard/`. Mitigation: explicitly declare scope: v3.13 does not touch `_templates/` or `haproxy-onboard/`. The cross-block plan can run in parallel on a separate branch.

**Risk 7, the `i9-11900K` CPU is fast but the lab has no swap and only 7.7 GiB RAM.** A heavy `make -j4 check-system-userspace` plus `tcpdump` plus a few namespaces is fine. A simultaneous OVN install would not be (but OVN is out of scope per the owner directive).

**Risk 8, lab session interruptions.** SSH sessions can drop. Mitigation: every lab session uses `tmux` on the lab host so the session survives SSH drops. The transcript captures the `tmux new -s v3.13-R<N>` line.

```
### Rule 18: Lab Output Verbatim Integrity (mandatory, extends Rule 7 and Rule 7a)

When the curriculum includes the output of a hands-on lab session run on a real lab host (for example, `lab-openvswitch` at `192.168.1.250`), every artifact captured from the lab host appears in the documentation exactly as it appeared on the host. Not a single character is modified, omitted, or rewritten.

Specific mandatory practices:

1. The shell prompt is preserved exactly. `root@lab-openvswitch:~#` is not shortened to `# `, not anonymised to `root@<HOST>:~#`, not coloured.
2. Lab IPs (192.168.1.250 etc.), MAC addresses, hostnames, network device names (ens33, br0), UUIDs, and OVSDB row UUIDs are preserved character for character. Do not anonymise to `<LAB_IP>`, `<HOST>`, `<UUID>`. The lab is the lab; its identifiers are part of the historical record of the experiment.
3. Log timestamps are preserved to the millisecond.
4. Empty output is preserved as an empty line, with the explicit annotation `(no output)` only when the absence of output is itself the pedagogical point.
5. If the output is too long to embed inline (more than approximately 200 lines), the curriculum file references the full transcript at `sdn-onboard/labs/v3.13-XN-<topic>.txt` and embeds an annotated extract per Rule 7 with the line `[N other lines omitted, context: full log preserved at <path>]`.
6. Transcripts are captured by `script -f -t` on the lab host and the typescript file is referenced from the curriculum. A diff between the human-readable rendering and the raw typescript must be empty.
7. The pre-commit hook `scripts/lab_verbatim_check.py` (added by plan v3.13 R0) compares the rendered transcript against the referenced typescript and rejects the commit on any divergence.

Scope: every lab transcript under `sdn-onboard/labs/`, every curriculum quote of lab output anywhere in `sdn-onboard/*.md`. The rule overrides any stylistic preference. The owner directive of 2026-04-29: "All operations, including commands, command outputs, hostnames, logs, etc., must be kept exactly as they are in the training documentation. No modifications, not even a single word, are allowed, in order to maintain integrity in the actual experiment."

(Origin: 2026-04-29 owner directive, codified by plan v3.13 R0.)
```

### 11.6. Practice versus theory separation (mandatory)

Per owner directive of 2026-04-30: practice and theory must be kept structurally separate. Plan v3.13 enforces this by directory:

| Directory | Content type | Rule 18 applies |
|-----------|--------------|------------------|
| `sdn-onboard/labs/` | Practice. Verbatim lab transcripts captured by `script -f -t` from real lab hosts. Every fenced code block reproduces host output byte for byte. Pedagogical commentary surrounds the transcripts but never modifies the captured output. | Yes (every `.md` here is checked by `lab_verbatim_check.py`, except `README.md`-style index files) |
| `sdn-onboard/source/` | Theory, source mastery. Pedagogical walkthroughs of the OVS C codebase. References source by file path and function name, may quote 10-line excerpts verbatim per Rule 14, contains no lab transcripts. | No (but Rule 14 source-citation integrity applies) |
| `sdn-onboard/0.0` through `sdn-onboard/16.x` (existing 139 files) | Theory, conceptual curriculum. Existing content. Plan v3.13 contributes small cross-reference patches that point readers from a concept block to the relevant lab transcript. | No |
| `memory/sdn/v3.13-enrichment/` | Working artefacts. Enrichment dossiers (E0, E1 to E12, E-S0 to E-S10). Not curriculum, but the foundation library every sprint reads from. | No |

The separation has three concrete consequences:

1. **A theory file never embeds verbatim host output.** When a theory file needs to show a command or output as an illustration, it cites the matching lab transcript (`see sdn-onboard/labs/v3.13-R<N>-<topic>.md`) rather than copying the lines. The verbatim record lives in exactly one place.
2. **A lab file never embeds a long pedagogical essay.** Lab transcripts have short pedagogical commentary surrounding the fenced output blocks. Long-form explanation belongs in the matching theory file under `sdn-onboard/source/` or in the existing conceptual curriculum.
3. **A pre-commit hook enforces the lab side mechanically.** `lab_verbatim_check.py` rejects any `.md` under `sdn-onboard/labs/` whose fenced blocks diverge from the referenced typescript. The theory side is enforced by Rule 14 source-citation integrity at review time, not by a hook.

This separation is the structural reason a learner can trust the curriculum: every claim about how OVS behaves is anchored in either a verbatim lab transcript that the learner can re-run, or a verbatim source-code excerpt that the learner can `grep` for in the upstream tree. The two domains do not contaminate each other.

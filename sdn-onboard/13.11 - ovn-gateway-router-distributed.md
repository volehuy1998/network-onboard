## Mục tiêu bài học

Sau khi hoàn thành phần này, bạn có thể:

1. **Mô tả (Understand)** sự khác biệt giữa **Distributed Router (DR)** — đơn giản, chạy phân tán trên mọi chassis — và **Gateway Router (GR)** — tập trung, gắn với một chassis cụ thể để xử lý SNAT/DNAT với external network.
2. **Phân tích (Analyze)** cơ chế **chassisredirect port** cho phép lưu lượng east-west đi qua DR phân tán, nhưng lưu lượng north-south (ra external) được chuyển hướng về chassis có GR hoặc chassis tiếp nhận DR.
3. **Áp dụng (Apply)** cấu hình Logical Router làm GR với `options:chassis=<chassis_name>`, gắn vào external network qua `localnet` port, và xác minh traffic north-south đi đúng chassis.
4. **Đánh giá (Evaluate)** sự đánh đổi giữa DR (throughput cao, không có điểm nghẽn) và GR (hỗ trợ SNAT, stateful NAT, dễ tích hợp với external).

Kiến thức tiên quyết: Part 13.2 LS/LR, Part 13.5 port binding (chassisredirect type), Part 13.6 HA chassis group cho GR failover.

## 13.11.1 Distributed Router so với Gateway Router

OVN hỗ trợ hai mô hình Logical Router khác biệt rõ ràng về cách traffic được xử lý trên chassis vật lý:

**Distributed Router (DR)** — mặc định khi tạo Logical Router qua `ovn-nbctl lr-add`. DR được hiện thực đầy đủ trên MỌI chassis có port gắn vào nó. Nghĩa là khi pod A trên chassis X gửi packet tới pod B trên chassis Y (khác subnet), cả chassis X và chassis Y đều xử lý một phần DR pipeline:
- Chassis X: pipeline ingress DR (từ pod A) → đến output port → tunnel Geneve qua chassis Y.
- Chassis Y: pipeline egress DR → deliver packet tới pod B.

Ưu điểm: không có điểm nghẽn, throughput tuyến tính theo số chassis. Nhược điểm: không hỗ trợ stateful SNAT (vì conntrack mỗi chassis độc lập, reply traffic có thể về chassis khác) và không dễ tích hợp với external network (không có chassis "neo" cố định).

**Gateway Router (GR)** — được tạo bằng cách set `options:chassis=<chassis_name>` trên Logical Router. GR chỉ chạy trên một chassis cụ thể (chassis "neo"). Tất cả traffic đi qua GR đều tập trung về chassis đó:
- East-west traffic intra-cluster: vẫn đi qua DR (nếu có) hoặc GR nếu cluster dùng hoàn toàn GR.
- North-south traffic (ra external qua localnet): bắt buộc đi qua chassis của GR.

Ưu điểm: hỗ trợ stateful SNAT (conntrack tập trung một chassis), tích hợp BGP/OSPF với external router dễ dàng, hỗ trợ IPsec tunneling. Nhược điểm: điểm nghẽn (chassis của GR phải xử lý toàn bộ north-south traffic của cluster), điểm fail duy nhất (nếu chassis GR chết — trừ khi có HA chassis group).

**Kiến trúc lai DR + GR (chuẩn production):** phổ biến nhất trong OVN-Kubernetes và OpenStack Neutron — triển khai một DR cho east-west + nhiều GR cho north-south. Pod gửi packet:
- Tới pod khác trong cluster → DR xử lý phân tán, throughput line-rate.
- Tới internet external → DR định tuyến tới GR (qua `join switch` trung gian), GR SNAT ra external IP.

```
┌─────────────────────────────────────────────────────────────────┐
│                       Cluster OVN                               │
│                                                                 │
│  pod_A ───► ls_A ───┐                                           │
│                     │                                           │
│  pod_B ───► ls_B ───┼──► DR (phân tán) ──► join_switch          │
│                     │          ▲                     │          │
│  pod_C ───► ls_C ───┘          │                     ▼          │
│                                │                  GR (gateway)  │
│                                │                     │          │
│                                │                     ▼          │
│                          east-west           localnet → external│
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 13.11.2 chassisredirect port — neo traffic về chassis cụ thể

Khi một Logical Router có gateway port (LRP được cấu hình làm gateway), OVN tự động tạo một port ảo type `chassisredirect` (cr-) để đại diện cho gateway ở trên chassis cụ thể. Đây là cơ chế OVN "neo" traffic gateway vào một chassis:

**Hoạt động:**
1. LRP có option `redirect-chassis` hoặc `redirect-chassis-group` trỏ tới chassis/chassis_group nào.
2. ovn-northd sinh một Port_Binding trong SB DB với type=`chassisredirect`, tunnel_key duy nhất, đặt trên chassis được chỉ định.
3. Khi traffic đi qua LRP (ví dụ packet từ intra-cluster cần ra external qua gateway), pipeline hash MAC/IP để chọn cr- port → tunnel Geneve tới chassis tiếp nhận → chassis đó áp gateway logic (SNAT, định tuyến ra external).

**Phân biệt với các loại port khác:**
- `vif` — virtual interface của pod/VM, bind với chassis có pod.
- `localnet` — bridge với external VLAN, không qua tunnel.
- `patch` — nối Logical Switch với Logical Switch (hoặc LSP với LRP).
- `chassisredirect` — neo gateway traffic về chassis cụ thể.
- `l3gateway` — gateway L3 routing, thường có trên GR.
- `router` — LRP, port của Logical Router.

**Cấu hình chassisredirect qua ovn-nbctl:**
```
[host]# ovn-nbctl set Logical_Router_Port lrp_external options:redirect-chassis=chassis_gw1
```

Hoặc với HA chassis group (tự động failover):
```
[host]# ovn-nbctl ha-chassis-group-add hagrp1
[host]# ovn-nbctl ha-chassis-group-add-chassis hagrp1 chassis_gw1 50
[host]# ovn-nbctl ha-chassis-group-add-chassis hagrp1 chassis_gw2 40
[host]# ovn-nbctl set Logical_Router_Port lrp_external options:redirect-chassis-group=hagrp1
```

ovn-northd tự động chọn chassis có priority cao nhất (50) làm active, chassis priority thấp hơn làm standby. Khi active chết (BFD phát hiện), standby tự động lên active — đã trình bày chi tiết Part 13.6.

## 13.11.3 Pipeline GR — SNAT và định tuyến ra external

Khi một LR được set `options:chassis`, ovn-northd sinh Logical Flow GR-specific khác DR:

**Pipeline GR ingress:**
1. `lr_in_admission` — kiểm soát admission.
2. `lr_in_lookup_neighbor` + `lr_in_learn_neighbor` — ARP/NDP cho gateway.
3. `lr_in_ip_input` — giảm TTL, xử lý ICMP echo.
4. `lr_in_unsnat` — nếu packet có conntrack entry SNAT → reverse SNAT trước khi định tuyến.
5. `lr_in_defrag` — defragment.
6. `lr_in_dnat` — DNAT cho LB/NAT rules.
7. `lr_in_ecmp_stateful` + `lr_in_policy` — routing policy.
8. `lr_in_ip_routing` — tra cứu bảng định tuyến, tìm next-hop.
9. `lr_in_ip_routing_ecmp` — chọn đường ECMP.
10. `lr_in_arp_resolve` — phân giải next-hop MAC qua ARP.
11. `lr_in_gw_redirect` — nếu gateway mode: redirect packet tới cr- port trên chassis GR.
12. `lr_in_arp_request` — gửi ARP request nếu chưa biết next-hop MAC.

**Pipeline GR egress:**
1. `lr_out_snat` — SNAT nếu có rule NAT type=snat.
2. `lr_out_egr_loop` — xử lý egress loopback.
3. `lr_out_delivery` — output packet ra external qua localnet port.

**SNAT rule trong NB DB:**
```
[host]# ovn-nbctl lr-nat-add lr1 snat 203.0.113.1 10.0.0.0/24
```
Tạo SNAT rule: tất cả packet từ subnet 10.0.0.0/24 đi ra ngoài qua lr1 sẽ có source IP được SNAT thành 203.0.113.1.

**DNAT cho external traffic đi vào:**
```
[host]# ovn-nbctl lr-nat-add lr1 dnat_and_snat 203.0.113.5 10.0.0.10
```
Tạo dnat_and_snat (floating IP): external client truy cập 203.0.113.5 sẽ được DNAT tới 10.0.0.10, và reply từ 10.0.0.10 được SNAT về 203.0.113.5 (hai chiều đối xứng).

## 13.11.4 Tích hợp GR với external network (localnet + BGP)

GR được gắn với external network qua `localnet` port — là bridge 1-1 với một VLAN bên ngoài, không qua Geneve tunnel:

```
[host]# ovn-nbctl ls-add ls_external
[host]# ovn-nbctl lsp-add ls_external ln_external
[host]# ovn-nbctl lsp-set-type ln_external localnet
[host]# ovn-nbctl lsp-set-options ln_external network_name=physnet1
[host]# ovn-nbctl lsp-set-addresses ln_external unknown
```

Trên chassis GR, OVS có bridge `br-ex` (hoặc tên khác tùy cấu hình) bind với physical NIC đi external. Ánh xạ `physnet1 → br-ex` được khai báo trong `external_ids:ovn-bridge-mappings`:
```
[host]# ovs-vsctl set Open_vSwitch . external_ids:ovn-bridge-mappings=physnet1:br-ex
```

GR được nối với ls_external qua LRP → traffic từ GR ra external sẽ đi qua localnet → physical NIC.

**Tích hợp BGP/OSPF:** OVN 22.03+ hỗ trợ **Dynamic Routing** qua tích hợp với FRR (Free Range Routing):
- FRR daemon chạy trên cùng chassis GR.
- FRR peer với external router qua BGP, học route và quảng bá VIP/floating IP từ OVN.
- OVN ghi route vào Logical_Router.static_routes → ovn-northd sinh Logical Flow routing.

Trong OVN-Kubernetes 1.0+, tích hợp BGP được hiện thực qua `--enable-multi-network` + MetalLB hoặc RouteAdvertisements CRD.

## 13.11.5 Sự đánh đổi DR so với GR

Bảng so sánh khi chọn DR hay GR:

| Tiêu chí | Distributed Router (DR) | Gateway Router (GR) |
|----------|------------------------|---------------------|
| Số chassis xử lý | Tất cả | Một chassis neo (hoặc HA group) |
| Stateful SNAT | Không (conntrack phân tán) | Có (conntrack tập trung) |
| Throughput east-west | Tuyến tính theo chassis | Giới hạn bởi chassis GR |
| Throughput north-south | Cần GR hỗ trợ | Giới hạn bởi NIC chassis GR |
| BGP peering | Khó (không có neo) | Dễ, chuẩn production |
| Điểm fail duy nhất | Không | Có (cần HA chassis group) |
| Phức tạp cấu hình | Thấp | Cao (HA, bridge mapping) |
| Phù hợp | East-west intra-cluster | North-south edge gateway |

**Chiến lược production thường dùng:**
- Mỗi cluster có **1 DR** cho east-west (gắn tất cả Logical Switch nội bộ).
- Mỗi cluster có **2+ GR** (cặp HA) cho north-south.
- DR và GR nối qua **join switch** trung gian (một Logical Switch đặc biệt cho router-to-router).
- BGP peering chỉ ở GR, route được lan truyền ngược về DR qua static route.

### ▶ Guided Exercise 1: Cấu hình DR + GR hybrid cho cluster

> **Trạng thái verification lab:** Output dựa trên OVN 22.03 docs; chờ verify lab thực.

**Mục đích:** Xây dựng topology thực tế với 1 DR cho east-west + 1 GR cho north-south, quan sát traffic định tuyến qua cr- port và tunnel Geneve.

**Chuẩn bị:** 2 node Ubuntu 22.04 với OVN 22.03 (chassis_compute + chassis_gateway), external network 203.0.113.0/24.

**Bước thực hiện:**

**1.** Tạo topology:
```
[host]# # Trên chassis_central: tạo DR
[host]# ovn-nbctl lr-add dr1
[host]# ovn-nbctl ls-add ls_web
[host]# ovn-nbctl ls-add ls_db
[host]# ovn-nbctl lrp-add dr1 dr1-ls_web aa:00:00:00:01:01 10.1.0.1/24
[host]# ovn-nbctl lrp-add dr1 dr1-ls_db aa:00:00:00:01:02 10.2.0.1/24
[host]# ovn-nbctl lsp-add ls_web ls_web-dr1
[host]# ovn-nbctl lsp-set-type ls_web-dr1 router
[host]# ovn-nbctl lsp-set-addresses ls_web-dr1 aa:00:00:00:01:01
[host]# ovn-nbctl lsp-set-options ls_web-dr1 router-port=dr1-ls_web
[host]# # Tương tự cho ls_db-dr1
```

**2.** Tạo GR neo trên chassis_gateway:
```
[host]# ovn-nbctl lr-add gr1
[host]# ovn-nbctl set Logical_Router gr1 options:chassis=chassis_gateway
[host]# # Tạo join switch nối DR và GR
[host]# ovn-nbctl ls-add ls_join
[host]# ovn-nbctl lrp-add dr1 dr1-join aa:00:00:00:02:01 100.64.0.1/30
[host]# ovn-nbctl lrp-add gr1 gr1-join aa:00:00:00:02:02 100.64.0.2/30
[host]# # Kết nối join switch
[host]# ovn-nbctl lsp-add ls_join dr1-join-p
[host]# ovn-nbctl lsp-set-type dr1-join-p router
[host]# ovn-nbctl lsp-set-options dr1-join-p router-port=dr1-join
[host]# # Tương tự cho gr1-join
```

**3.** Tạo external network + SNAT:
```
[host]# ovn-nbctl ls-add ls_external
[host]# ovn-nbctl lsp-add ls_external ln_external
[host]# ovn-nbctl lsp-set-type ln_external localnet
[host]# ovn-nbctl lsp-set-options ln_external network_name=physnet1
[host]# ovn-nbctl lrp-add gr1 gr1-external aa:00:00:00:03:01 203.0.113.1/24
[host]# # SNAT rule
[host]# ovn-nbctl lr-nat-add gr1 snat 203.0.113.1 10.0.0.0/16
[host]# # Static route từ DR về GR
[host]# ovn-nbctl lr-route-add dr1 0.0.0.0/0 100.64.0.2
```

**4.** Kiểm tra Port_Binding loại chassisredirect:
```
[host]# ovn-sbctl list Port_Binding | grep -A 5 chassisredirect
```

Output kỳ vọng: có Port_Binding `cr-gr1-external` type=`chassisredirect`, bind với chassis_gateway.

**5.** Xác minh north-south traffic qua cr- port:
```
[host]# ovn-trace ls_web 'inport == "pod1" && eth.src == aa:bb:cc:dd:ee:01 \
         && ip4.src == 10.1.0.5 && ip4.dst == 8.8.8.8 && eth.dst == aa:00:00:00:01:01'
```

Output hiển thị packet đi: pod1 → DR (dr1) → join switch → GR (gr1) → SNAT → cr-port → external.

**Bài học học được:**

Thứ nhất, **chassisredirect port** là cơ chế cốt lõi cho phép OVN kết hợp ưu điểm của DR (mở rộng east-west) và GR (stateful north-south). Không có nó, OVN buộc phải chọn một trong hai.

Thứ hai, SNAT chỉ hoạt động đúng trên GR vì conntrack tập trung. Nếu thử SNAT trên DR, traffic trả lời có thể về chassis khác (không có conntrack entry) → packet drop.

Thứ ba, cluster môi trường production luôn triển khai GR dưới HA chassis group — GR đơn chết = toàn bộ north-south down. Part 13.6 HA chassis group + BFD là prerequisite bắt buộc.

**Cleanup:**
```
[host]# ovn-nbctl lr-del gr1
[host]# ovn-nbctl lr-del dr1
[host]# ovn-nbctl ls-del ls_external ls_web ls_db ls_join
```

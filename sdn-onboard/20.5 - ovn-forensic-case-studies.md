## 20.5.1 Nguyên tắc forensic OVN distributed control plane

Part 9.26 đã xây dựng framework forensic cho OVS datapath (revalidator storm, bond flap cascade, conntrack zone collision). Những case đó khu trú trong **một chassis** hoặc **hai chassis kề**. Part 20.5 mở rộng phạm vi lên **cluster-wide distributed state**: nơi một sự cố phát sinh do **bất đồng bộ tạm thời** giữa NBDB intent, SBDB computed state, và chassis local datapath.

Ba điểm khác biệt quan trọng so với OVS forensic:

**Thứ nhất, vai trò của "eventually consistent" mechanism.** NBDB → northd compile → SBDB → ovn-controller trên mỗi chassis → OpenFlow install là chuỗi bốn chặng đồng bộ lỏng lẻo. Giữa các chặng có cache, I-P node, và BFD. Sự cố xuất hiện khi một chặng chạy nhanh hơn chặng trước hoặc bị invalidate sai thứ tự. Part 9.26 §9.26.13 gọi đây là "eventually consistent distributed cache"; Part 20.5 đào sâu cả ba incident class thuộc họ này nhưng ở phạm vi OVN.

**Thứ hai, "claim protocol" thay cho "flow install".** Trong OVS thuần, ai cài flow người đó quản. Trong OVN, mỗi Port_Binding có exactly-one chassis owner tại mọi thời điểm. Migration, failover, và HA_Chassis_Group là các ceremony đổi ownership. Nếu ceremony không atomic, ta có race. Case 1 đi sâu vào pattern này.

**Thứ ba, "memory budget" cho I-P engine.** northd và ovn-controller đều giữ I-P node cache trong memory. Một số operation pattern (bulk delete, bulk add, bulk update) có thể làm cache balloon trước khi GC kịp. OOM ở control plane gây failover, failover trigger full recompute trên successor, dẫn tới cascade. Case 2 khai thác pattern này.

Cả ba case share structure: **event trigger → distributed state mismatch → observable symptom → cascade → recovery**. Ta sẽ thấy pattern lặp lại nên toolkit forensic cho case sau ngắn gọn dần.

## 20.5.2 Case 1, Port_Binding migration race, "dual-bind transient" cross-chassis

### 20.5.2.1 Signature ban đầu

Tenant cloud provider chạy 60 hypervisor OVN 22.03.8. VM T1-web-17 (tenant T1, Logical_Switch `ls-t1`, LSP `lsp-web-17`) đang live-migrate từ chassis `chA-042` sang `chB-015`. Live migration trong Nova/OpenStack thực hiện chuỗi bước (prepare target → stream memory → pause source → resume target → clean source). OVN ML2 plugin cập nhật Port_Binding.chassis từ `chA-042` sang `chB-015` ngay sau khi "resume target".

Người vận hành báo cáo: trong window ~8 giây sau khi resume target, **người dùng cuối báo intermittent connectivity tới VM**. ARP reply về VM thay đổi MAC address liên tục giữa 2 port (ToR MAC flapping alert). ACL log ghi nhận cùng 5-tuple connection xuất hiện ở 2 chassis. Sau 8 giây, mọi thứ ổn định tự động, chassis `chB-015` owns port, `chA-042` no longer references port. VM hoạt động bình thường.

Tuần sau, người vận hành quay lại khi 3 lần migration khác gặp cùng symptom với duration dao động 3-18 giây. Đặc biệt một lần: duration 18 giây dẫn tới BFD flap giữa cluster chassis (Scenario E + K trong Part 9.14), lan rộng thành northd recompute burst, latency P99 tăng 120ms trong 2 phút. Người vận hành escalate vì hypothesis "OVN live migration không an toàn trong cluster lớn".

### 20.5.2.2 Evidence, Anatomy Template A cho claim protocol trace

**Bối cảnh khi chạy:** Kỹ sư bắt đầu forensic sau khi migration lần thứ 4 gây cascade. Hypothesis: có window "dual-bind" mà cả `chA-042` và `chB-015` cùng claim port.

**Command + output** *(reproduced từ OVN 22.03.8 lab reproduce trong sandbox 10-chassis)*:

```bash

[sbdb-leader]# sudo ovsdb-client --timestamp monitor OVN_Southbound \
  Port_Binding chassis,logical_port,tunnel_key \

# Output format (mỗi dòng 1 event):
# 2026-04-24T14:32:01.234 update Port_Binding
#   logical_port="lsp-web-17" chassis=<uuid-chA-042> tunnel_key=17
# 2026-04-24T14:32:09.812 update Port_Binding
#   logical_port="lsp-web-17" chassis=<uuid-chB-015> tunnel_key=17   ← ML2 flip
# 2026-04-24T14:32:09.815 update Port_Binding
#   logical_port="lsp-web-17" chassis=<uuid-chB-015> tunnel_key=17   ← ovn-controller chB claim
# (no release event từ chA trong 8 giây)
# 2026-04-24T14:32:17.901 update Port_Binding
#   logical_port="lsp-web-17" chassis=<uuid-chB-015> tunnel_key=17   ← finalize sau chA release
```

**Anatomy từng event:**

| Event timestamp | Source | Action | Claim owner sau event | Ý nghĩa |
|-----------------|--------|--------|-----------------------|---------|
| 14:32:01.234 | Initial state | bind chA-042 | chA-042 | VM chạy trên chA, OpenFlow local install xong |
| 14:32:09.812 | ML2 plugin | flip chassis column | chB-015 (NBDB intent) | Nova kết thúc memory stream, update Port_Binding |
| 14:32:09.815 | ovn-controller chB | claim notification | chB-015 (SB confirm) | chB detect Port_Binding.chassis=self, install flow local |
| 14:32:09.815 → 14:32:17.9 | **Window race** | chA chưa release | 2 chassis concurrent | **Dual-bind transient** |
| 14:32:17.901 | ovn-controller chA | release (delayed) | chB-015 (steady) | chA cuối cùng nhận event, remove flow local |

**Cross-check flow state trên 2 chassis trong window race:**

```bash
# Trên chA-042, tại 14:32:14 (giữa race window)
[chA-042]# sudo ovs-ofctl dump-flows br-int \
  | grep -E "0xb0004.*output.*ovn-chB015" | head -3
# Expected nếu đã release: 0 flow
# Actual observed: 2 flow còn sót (ingress + egress) chưa bị delete

# Trên chB-015, tại 14:32:14
[chB-015]# sudo ovs-ofctl dump-flows br-int \
  | grep -E "0xb0004.*priority=.*output:patch" | head -3
# Expected: flow local đầy đủ, VIF tới VM up

# Datapath flow cache trên chA (proof traffic vẫn forward ra VM)
[chA-042]# sudo ovs-dpctl dump-flows | grep -E "reg0=0x11" | head
# reg0 = tunnel_key của LSP, 0x11 = 17 decimal
```

**Kịch bản bẻ gãy:**

- **Window < 2 giây** → BFD không flap, northd có thể không notice, symptom chỉ là vài packet duplicate. Người dùng không phàn nàn.
- **Window 3-10 giây** → ARP/ND flood duplicate, ToR MAC flap alert, người dùng báo intermittent connectivity. Đây là phần lớn case.
- **Window > 10 giây** → BFD flap lan rộng (tunnel giữa chA-chB hỏng tạm thời), trigger northd recompute burst + ovn-controller lflow-cache invalidate trên nhiều chassis. Cascade thành incident class lớn hơn.

### 20.5.2.3 Mechanism deep-dive

Trong OVN 22.03, `Port_Binding.chassis` column là ghi NBDB intent trực tiếp. Khi ML2 set chassis=chB-015, SBDB propagate thay đổi tới tất cả ovn-controller. Mỗi ovn-controller xem row:

- **ovn-controller chB-015** thấy `Port_Binding.chassis=self` → chạy `binding_add_lport()` trong `controller/binding.c`, install OpenFlow flow local cho LSP, publish BFD session up, advertise MAC address ra overlay.
- **ovn-controller chA-042** thấy `Port_Binding.chassis != self` → chạy `binding_del_lport()` → remove flow local, revoke MAC advertisement.

Hai hành động này **không atomic** cross-chassis. Lý do:

1. **SBDB update propagate qua Raft replication**, trễ 10-50ms từ leader đến follower chassis.
2. **ovn-controller ở chA-042** có run-loop riêng (`ovn_controller_run_loop()` trong `controller/ovn-controller.c`). Nếu chA đang bận I-P node recompute khác, phản ứng tới event mới có thể trễ vài giây.
3. **Flow uninstall ở chA** phải đi qua ovs-vswitchd OpenFlow channel; trong cluster lớn, channel có thể queue và trễ thêm.
4. **chA không biết chB đã claim xong hay chưa**, vì giao tiếp giữa 2 chassis không có direct sync. Tất cả đi qua SBDB như trung gian.

Trong window này, **cả 2 chassis đều còn flow local** cho LSP. Gói tin đi tới VIF trên chA vẫn forward được (flow còn), gói tin đi tới VIF trên chB cũng forward được (flow mới installed). ARP reply từ VM có thể xuất phát từ 1 trong 2 chassis tuỳ thuộc kernel scheduling hypervisor. Kết quả: ToR switch thấy MAC address cùng lúc trên 2 cổng uplink → MAC flap.

Đặc biệt nghiêm trọng khi **client bên ngoài mở connection mới**: SYN packet có thể routed tới chA hoặc chB qua underlay ECMP. Nếu SYN tới chA mà chA đã stop forwarding (nhưng chưa remove flow), traffic black-hole. Ngược lại nếu SYN tới chB mà memory VM chưa resume hoàn toàn, VM không reply.

OVN 22.06 bổ sung column `Port_Binding.requested_chassis` như **intent declaration** tách biệt khỏi `chassis` (claimed owner). ML2 update `requested_chassis`, northd xử lý chuyển giao có thứ tự: wait chA release → set chassis=chB → chB claim. Eliminate race. OVN 24.03 polish pattern với BFD-based fast failover cho HA_Chassis_Group.

### 20.5.2.4 Remediation 4-tier

```bash
# Tier 1 (immediate mitigation, ngắn hạn)
# Khi phát hiện dual-bind window > 5s, manual coordination:
# Bước 1: pause migration, kiểm tra Port_Binding.chassis
[operator]# sudo ovn-sbctl --columns=logical_port,chassis list Port_Binding \
  | grep lsp-web-17

# Bước 2: nếu stuck dual-state, force release source chassis
[chA-042]# sudo ovs-appctl -t ovn-controller binding-release lsp-web-17
# hoặc restart ovn-controller trên chassis source (heavy-handed)

# Bước 3: verify flow gone
[chA-042]# sudo ovs-ofctl dump-flows br-int | grep -c "reg.*=0x11"
# Expected: 0 flow cho LSP

# Tier 2 (tooling coordination, medium-term)
# Nova/ML2 plugin phải đảm bảo migration step order:
# 1. Gọi OVN Notify "migration intent" (pre-commit hook)
# 2. Stream memory
# 3. Pause VM source
# 4. Update Port_Binding.chassis = target
# 5. Wait max 2s confirm target claim OK qua SBDB notification
# 6. Resume VM target
# 7. Clean up VM source
# Nếu bước 5 timeout, rollback: keep source, abort migration
# Patch ML2 code (neutron/plugins/ml2/drivers/ovn) implement confirm wait

# Tier 3 (structural, OVN upgrade)
# Upgrade OVN 22.06+ với Port_Binding.requested_chassis
# ML2 plugin update để set requested_chassis thay vì chassis
# northd xử lý chuyển giao atomic:
[any]# sudo ovn-nbctl --db=... set NB_Global . options:use-requested-chassis=true

# Tier 4 (prevention, triển khai hygiene)
# - Daily cron: scan Port_Binding với window > 5s trong metrics history
#   → identify pattern migration fail rate
# - Capacity planning: không migration > 5 VM cùng lúc per chassis pair
#   → giảm contention run-loop ovn-controller
# - Alertmanager rule:
```

**Prevention alerts:**


## 20.5.3 Case 2, northd bulk tenant deletion memory cascade + OOM failover

### 20.5.3.1 Signature ban đầu

Tenant T2 offboarding quy mô lớn: 5.000 Logical_Switch_Port, 500 ACL rule, 200 Load_Balancer, 80 Logical_Router_Port, tổng ~350MB logical_flow data trên SBDB. Automation script chạy một mạch:

```bash
# Tooling automation (buggy)
for lsp in $(ovn-nbctl --columns=name find LSP external_ids:tenant=T2); do
  ovn-nbctl lsp-del $lsp
done
for acl in $(... T2 ACL ...); do
  ovn-nbctl acl-del $acl
done
# ... tiếp tục cho LB, LR, v.v.
```

Mỗi `ovn-nbctl lsp-del` là 1 transaction riêng. 5.800 transaction trong 90 giây → NBDB Raft commit rate 65/s.

Người vận hành nhận alert: **Prometheus `ovn_northd_memory_bytes` balloon từ 400MB → 2.4GB trong 90s**. Sau 90s, systemd báo `ovn-northd` OOM kill. Cluster 3-node có northd active-standby: standby northd trên node-2 takeover. Standby northd phải **full recompute** từ NBDB (I-P cache chưa warm), runtime 180 giây. Trong window 180s, không có `nb_cfg` update về SBDB, tất cả chassis kẹt không nhận được logical_flow change. VM mới tạo không có network.

Sau standby recompute xong, nb_cfg catch up 60 giây. Tổng window sự cố **~4 phút 40 giây**. Người vận hành hỏi: "Sao delete tenant lại gây cascade thế này?"

### 20.5.3.2 Evidence, Anatomy Template A cho memory forensic

**Bối cảnh khi chạy:** Kỹ sư audit northd process lifecycle sau khi standby takeover xong.

**Command + output** *(reproduced trên OVN 22.03.8 3-node cluster, 10.000 LSP baseline)*:

**Kịch bản bẻ gãy:**

- **`lflows cache` > 1GB** trong 60s → I-P node reference giữ hàng ngàn deleted LSP → memory không free kịp.
- **`Logical_Flow status: recompute`** lặp lại > 5 iteration → engine không ra khỏi recompute loop vì input tiếp tục thay đổi (mỗi txn mới invalidate).
- **OOM kill trigger ở ~2.4GB** (phụ thuộc `MemoryMax` systemd, default không set → phụ thuộc cgroup/hypervisor).

### 20.5.3.3 Mechanism deep-dive

I-P engine của northd được thiết kế để xử lý thay đổi tăng dần (incremental). Khi 1 LSP bị delete:

1. Engine invalidate I-P node `Logical_Flow` (depends on `Logical_Switch_Port`).
2. Node `Logical_Flow` chạy diff: so sánh LSP input cũ vs mới → detect LSP biến mất.
3. Engine gọi `clear_tracked_data()` để free reference cũ.
4. Engine call `build_lflows_for_changed_only()` để rebuild chỉ flow ảnh hưởng.

Ở steady-state, chu trình này mất ~5-20ms per LSP.

Khi 5.000 LSP bị delete trong 90s (batch 65/s):

1. Mỗi transaction đều invalidate I-P node.
2. Engine start rebuild, nhưng trước khi xong, transaction kế invalidate lại.
3. Engine rơi về **full recompute mode** (theo thiết kế "safety net" khi partial update quá khó).
4. Full recompute phải build lại logical_flow cho **toàn bộ 5.000 LSP còn lại** (chưa delete + 5.000 đã delete đang trong window invalidate).
5. Trong full recompute, `clear_tracked_data()` chạy **sau** khi build xong, không phải trước. Memory grow trước khi free.
6. 5.000 iteration × ~170KB per LSP cache = ~850MB peak.
7. Vượt memory budget → OOM.

**Tại sao failover không seamless?** Raft cluster có 3 ovn-northd instance: 1 active, 2 standby. Khi active OOM, standby takeover nhưng phải:

1. Download full NBDB snapshot (400MB, 2-3s).
2. Build I-P cache from scratch (no warm cache).
3. Rebuild toàn bộ logical_flow cho 95.000 LSP còn sống (sau khi 5.000 T2 đã delete xong nửa chừng).

Đây là chặng 180s stuck. VM mới tạo trong window này không có network cho đến khi standby recompute xong.

### 20.5.3.4 Remediation 4-tier

```bash
# Tier 1 (recovery khẩn, sau khi đã OOM)
# Standby đang recompute, không can thiệp được. Chờ.
# Monitor progress:
[nbdb-leader]# watch -n 2 'sudo ovn-appctl -t ovn-northd inc-engine/show \
  | grep -E "status|runtime"'
# Khi Logical_Flow status từ recompute → updated = recovered

# Tier 2 (tooling fix, batch deletion + sleep)
# Refactor automation script:
[tooling]$ # Thay vì delete từng cái 1 transaction
for batch in $(split -l 100 tenant-T2-lsp.txt); do
  CMDS=""
  for lsp in $(cat $batch); do
    CMDS="$CMDS -- lsp-del $lsp"
  done
  ovn-nbctl $CMDS   # 1 transaction, 100 LSP
  sleep 3           # cho phép I-P engine settle giữa batch
done
# 5000 LSP / 100 per txn = 50 txn × 3s = 150s (so với 90s chạy nhanh)
# Tuy chậm hơn nhưng memory peak chỉ tăng ~50MB per batch, không balloon

# Tier 3 (systemd protection, giới hạn memory + auto-restart)
[nbdb-leader]# sudo systemctl edit ovn-northd
# Add:
# [Service]
# MemoryMax=4G
# Restart=on-failure
# RestartSec=5s
# OOMPolicy=stop      # rely trên systemd thay vì kernel OOM killer

# Tier 4 (structural upgrade + cache tuning)
# - OVN 23.09+ cải tiến clear_tracked_data() chạy incremental
# - Tune cache budget:
[any]# sudo ovn-nbctl set NB_Global . options:northd-backoff-interval-ms=100
[any]# sudo ovn-nbctl set NB_Global . options:mac-cache-max-size=10000
# - Parallel northd build (OVN 23.03+):
[nbdb-leader]# sudo ovn-appctl -t ovn-northd parallel-build/enable
# Distribute build lflows across multi-core → giảm peak memory per core
```

**Prevention alerts:**

```yaml

  expr: rate(ovn_northd_memory_bytes[5m]) > 10 * 1024 * 1024   # >10MB/s growth
  for: 2m
  labels: { severity: warning }

- alert: OvnNorthdMemoryCritical
  expr: ovn_northd_memory_bytes > 2 * 1024 * 1024 * 1024        # >2GB
  for: 60s
  labels: { severity: critical }
```

## 20.5.4 Case 3, MAC_Binding table explosion từ tenant ARP scan exploit

### 20.5.4.1 Signature ban đầu

Tenant T3 deploy 20 VM trong subnet `10.0.3.0/24` (254 IP usable). Một VM bị compromise qua lỗ hổng container image. Malware bên trong thực hiện **ARP scan 10.0.0.0/16** (65.534 IP) trong 15 phút, rate 75 ARP request/s.

OVN Logical_Router `lr-t3` có distributed ARP responder (Part 18.0) cho tenant subnet. Khi ARP request tới địa chỉ **nằm trong subnet managed nhưng không tồn tại trong IPAM**, OVN generate **ARP reply giả lập** từ router MAC để handle downstream forwarding (Part 18.0 §18.3). Mỗi reply như vậy tạo 1 row `MAC_Binding` trong SBDB: `(MAC, IP, Logical_Port)` tuple để cache giúp reply nhanh hơn lần sau.

Trong 15 phút, malware trigger ~67.500 ARP request. SBDB `MAC_Binding` table grow từ baseline 400 row lên **67.900 row**. Row count tăng → SBDB commit size tăng → northd I-P node cho `MAC_Binding` rebuild mỗi lần có row mới (bulk update pattern tương tự Case 2 nhưng slower, không gây OOM nhưng gây slowdown).

Triệu chứng quan sát:
- **Prometheus `ovn_sb_table_rows{table="MAC_Binding"}` tăng 4.500 row/phút**.
- **`ovn-controller` CPU trên 60 chassis tăng từ 5% → 35%** vì phải sync MAC_Binding changes qua SBDB monitor.
- **`nb_cfg` lag NBDB-SBDB từ 0.2s → 8s**.
- **Cuối chu kỳ 15 phút**, cluster stable lại ở load cao. Malware stops (có thể bị external IDS detect + quarantine).

Tuần sau, người vận hành phát hiện MAC_Binding table còn 67.900 row vĩnh viễn, không có garbage collection. Người vận hành hỏi: "Làm sao reset? Tại sao OVN không self-heal?"

### 20.5.4.3 Mechanism deep-dive

OVN 22.03 MAC_Binding không có **age threshold**. Khi row được tạo qua ARP learning, row tồn tại vĩnh viễn trong SBDB cho đến khi:

1. LSP gắn với logical_port bị xoá.
2. Người vận hành manual delete row.
3. Entire logical_switch bị xoá.

Không có cơ chế TTL. Không có LRU eviction. Đây là choice design ban đầu vì:
- **Steady-state workload** có MAC_Binding growth rate thấp (chỉ tenant legitimate).
- **Khi SBDB backup**, row count dự đoán được.
- **Row nhỏ** (~200 byte), nên 10.000 row = 2MB, negligible.

Thiết kế này **fragile** dưới ARP scan exploit vì:
- Attacker control từ trong tenant có thể sinh row rate 75/s.
- 75/s × 3.600s = 270.000 row/h → unbounded growth.
- Row nhiều khiến `MAC_Binding` I-P node rebuild tốn thời gian hơn → latency spike.
- SBDB commit size tăng → Raft replication slower.
- ovn-controller local cache copy của MAC_Binding cũng grow → memory usage each chassis increase.

OVN 24.03 bổ sung `Logical_Router.options:mac_binding_age_threshold` (default 0 = disabled, người vận hành phải set). Khi set thành 300 giây, row tạo bởi ARP learning sẽ bị GC sau 300s không dùng. Fix exploit vector nhưng cần người vận hành enable explicit.

## 20.5.5 Cross-case takeaway, 3 design lesson cho triển khai OVN

Ba case trên khác nhau về trigger và symptom, nhưng chia sẻ cùng cấu trúc:

1. **Event gây inconsistency phân tán.** Migration race (Case 1), bulk transaction (Case 2), ARP scan (Case 3) đều là event làm lệch pha NBDB intent vs SBDB computed vs chassis local.
2. **Amplification qua I-P engine hoặc claim protocol.** Engine rebuild quá tải (Case 2), claim chưa atomic (Case 1), table không bounded (Case 3) biến lệch pha thành cascade.
3. **Recovery phụ thuộc "eventually consistent" hội tụ.** Không có master switch. Hệ thống cuối cùng hội tụ, nhưng window cascade gây hỏng dịch vụ.

Ba design lesson để giảm thiểu rủi ro:

### Lesson 1, Claim protocol phải idempotent và atomic

OVN 22.03 Port_Binding.chassis = claimed owner + intent merged → race. OVN 22.06+ tách requested_chassis vs chassis = **intent vs state**. Pattern này áp dụng được cho mọi distributed claim:

- **Tách intent khỏi state.** Intent (ai muốn sở hữu) không đồng nghĩa với state (ai đang sở hữu).
- **Transition có thứ tự cưỡng bức.** Source release → target claim, không ngược lại.
- **Timeout rollback.** Nếu transition stuck > N giây, revert intent về state ban đầu.
- **Observable transition state.** Expose trạng thái "transitioning" ra metric để người vận hành detect.

Không chỉ OVN migration áp dụng được. Pattern này phù hợp với bất kỳ hệ thống có leader election, primary-secondary failover, hay quorum-based ownership.

### Lesson 2, Memory budget I-P node phải bounded và observable

northd và ovn-controller dùng I-P node cache để tăng hiệu năng. Nhưng cache vô bounded = exploit vector (Case 2) hoặc symptom cascade (Case 3).

Nguyên tắc:
- **Cache size expose ra metric.** `memory/show` field phải export Prometheus.
- **GC proactive, không reactive.** `clear_tracked_data()` phải chạy **trước** build mới, không phải sau. OVN 23.09+ fix theo hướng này.
- **Upper bound explicit.** MemoryMax systemd để OOM killer ưu tiên kill process không critical khác, không phải northd.
- **Batch pattern enforcement.** Tooling automation phải break bulk transaction thành batch nhỏ, có sleep gap cho GC.

Lesson này cũng áp dụng cho bất kỳ control plane có caching: Kubernetes controller, etcd reconciliation, Prometheus TSDB ingestion.

### Lesson 3, Distributed state phải age-bounded

MAC_Binding vô hạn = storage leak + hiệu năng degradation + attack surface. Age threshold là defense cơ bản cho mọi **learned state** trong hệ phân tán:

- **ARP/ND cache có TTL** (kernel default 60s, OVN 24.03+ mac_binding_age_threshold tương tự).
- **Conntrack entry có TCP timeout** (Part 9.24 §9.24.6).
- **Raft log có compaction threshold** (OVS 2.13+ auto-compact).
- **Flow statistics có hard-timeout** (OpenFlow spec).

Không có age bound = state grow theo operation history, không theo current topology. Ba tháng sau triển khai, table có 10x row "không ai dùng nhưng không ai xoá".

Nguyên tắc vàng: **mọi row learned từ packet observation phải có age attribute**. Mọi row learned phải có policy GC mặc định bật. Người vận hành opt-out nếu cần (ví dụ traffic forensic cần giữ lâu), không opt-in.

## 20.5.8 Capstone POE Phase G.2.3 — "Set mac_binding_age_threshold=60 cho mọi LR trong cluster, fix luôn exploit ARP scan?"

**Kịch bản:** Sau khi đọc Case 3, quản lý cấp trên đề xuất: *"Apply `options:mac_binding_age_threshold=60` cho toàn bộ 240 Logical_Router trong production cluster. Giá trị 60 giây đủ ngắn để GC rate-limit exploit scan. Việc này có thể automate qua script 1-liner, deploy trong 5 phút maintenance. Giải quyết triệt để MAC_Binding explosion mọi tenant."*

Phân tích và phản biện:

**Predict:** Setting này có fix exploit không? Tác dụng phụ nào?

**Observe:**

1. **Tenant legitimate có thể bị ảnh hưởng.** Server stateless có incoming connection từ nhiều external client. Mỗi client tương tác 1-2 phút rồi idle. Nếu `mac_binding_age_threshold=60`, row sẽ GC trước lần request kế của cùng client. Kết quả: mỗi connection mới phải trigger ARP request lại, latency tăng 10-50ms (tùy underlay RTT). Không đáng kể cho web server, nhưng đáng kể cho latency-sensitive service (real-time gaming, trading).

2. **Large subnet với server fleet stable.** Tenant có 500 server ổn định trong subnet /24. MAC_Binding cache row cho 500 MAC legitimate. Threshold=60s buộc row bị GC sau 60s không dùng. Nếu một server idle 65s, row GC. Server kế tiếp ping → ARP request → ARP reply từ router → insert row mới. Đây là **wasted work** (rebuild cùng entry thường xuyên) mà không phòng chống được gì.

3. **Threshold quá nhỏ làm weak protection.** Attacker scan 10 IP/s, threshold=60s → table size cap ở 600 row. Nhưng attacker rate 100 IP/s → cap 6.000 row. Vẫn đủ để slow control plane nếu cluster lớn. Threshold là **defense-in-depth, không phải primary defense**. Primary defense phải là rate-limit ARP egress từ tenant (Tier 2 Case 3).

4. **Apply đồng loạt 240 LR không maintenance window = risk.** Mỗi set operation trigger Logical_Router I-P node invalidate. 240 invalidation trong 5 phút = burst tương tự Case 2 (bulk update), dù scale nhỏ hơn 50x. Có thể gây nb_cfg lag 10-30s trong 5 phút window. Không outage, nhưng VM provision delay.

5. **Threshold giá trị "đúng" phụ thuộc pattern traffic.** Không có magic number. Cần analyze mỗi tenant:
   - Web-facing server với short-lived client → threshold 300-600s (balance GC vs re-learn).
   - Backend server fleet ít traffic volatile → threshold 3600s hoặc disable (cache ổn định).
   - High-risk tenant với untrusted workload → threshold 60s **cộng với** ACL ARP rate-limit.

**Explain:**

Đề xuất "1-liner fix-all" **không chính xác**. Cách tiếp cận đúng 3 bước per tenant:

- **Step 1 — Classify tenant risk:** internal trusted (stable, long-lived) vs high-risk (untrusted workload, public-facing). Dùng external_ids hoặc tag system để mark tenant.
- **Step 2 — Per-class policy template:**
  - Trusted class: threshold=3600s hoặc disable + ACL allow-established (stateful).
  - Standard class: threshold=300s + ACL rate-limit ARP 10/s.
  - High-risk class: threshold=60s + ACL rate-limit ARP 5/s + daily audit cron.
- **Step 3 — Rolling triển khai với monitor:**
  - Apply per-tenant trong maintenance window, không batch.
  - Monitor post-apply: MAC_Binding row count + ovn-controller CPU trên 60 chassis.
  - Rollback policy nếu latency P99 spike > 20%.

**Finish:**

Nguyên tắc: **cùng 1 setting không có "đúng" universal**. OVN age threshold là công cụ, không phải policy. Áp dụng tool phải gắn với classification traffic pattern. 1-liner mass-apply là anti-pattern thường gặp, thường xuất hiện khi người vận hành chịu áp lực "close ticket". Trade-off giữa **defense strength** (threshold thấp = protect tốt khỏi exploit) và **hiệu năng cost** (threshold thấp = ARP re-learn thường xuyên = latency tax). Không free lunch.

Cross-reference: tương tự bài học Case 2 (batch deletion không phải 1-txn 5000 LSP) và Scenario M Part 9.14 (restart không fix ghost claim), "simple 1-step fix" cho distributed state cần skepticism mặc định.

## Điểm cốt lõi cần nhớ

Thứ nhất, OVN distributed control plane có 3 class sự cố đặc trưng: claim protocol race, I-P memory cascade, learned state explosion. Mỗi class có signature, mechanism, và remediation riêng.

Thứ hai, "eventually consistent" không có nghĩa "immediately correct". Window bất đồng bộ giữa NBDB intent, SBDB computed, chassis local là nguồn gốc race và cascade. Forensic phải đo window này bằng timestamp ovsdb-client monitor, không chỉ nhìn end-state.

Thứ ba, I-P engine caching tăng hiệu năng nhưng tạo memory attack surface. Batch pattern bulk 1-txn là anti-pattern phổ biến gây OOM cascade. Luôn chia nhỏ bulk transaction có sleep gap.

Thứ tư, learned state (MAC_Binding, conntrack entry, Raft log) phải age-bounded mặc định. Không age bound = state grow unbounded = attack vector.

Thứ năm, claim protocol phải tách intent khỏi state (OVN 22.06+ requested_chassis pattern). Transition phải atomic, có timeout rollback, có observable state.

Thứ sáu, "1-liner fix-all" cho distributed state là warning sign. Luôn classify tenant + per-class policy template + rolling triển khai.

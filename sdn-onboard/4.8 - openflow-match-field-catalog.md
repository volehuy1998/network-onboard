1. **Liệt kê (Remember)** 12 nhóm khớp với field foundation của OpenFlow + OVS extensions: Metadata, Register, Tunnel, L2, ARP, IPv4, IPv6, L4 (TCP/UDP/SCTP), ICMP, MPLS, Conntrack, Packet-type.

## Bối cảnh: Tại sao cần catalog đầy đủ

OpenFlow 1.0 (12/2009) có **12 khớp với field cứng**, match-all-or-nothing per field:

```
in_port, dl_src, dl_dst, dl_type, dl_vlan, dl_vlan_pcp,
nw_src, nw_dst, nw_proto, nw_tos, tp_src, tp_dst
```

OpenFlow 1.2 (12/2011) chuyển sang OXM TLV extensible format (Phần 4.1 chi tiết). Từ đó tập khớp với field mở rộng liên tục:
- OpenFlow 1.3 (04/2012): IPv6 full + meter identifier.
- OpenFlow 1.4 (08/2013): optical port field.
- OpenFlow 1.5 (12/2014): packet_type, PBB I-SID, actset_output, TCP flags.
- OVS Nicira extensions (NXM): register reg0-reg15, xreg, xxreg, conntrack field, pkt_mark, metadata alias.
- OVN extensions: tun_metadata[0-63] cho Geneve TLV, ct_nw_src/dst/proto (conntrack orig).

Đến OVS 2.17, man `ovs-fields(7)` liệt kê **hơn 100 match field**. Hiểu hết là impossible cho kỹ sư mới, nhưng **phân nhóm + thuộc prerequisite** là khả thi. Phần này cấu trúc 12 nhóm, mỗi nhóm có field quan trọng được breakdown Template B.

## 4.8.1 9-attribute anatomy per field (template chung)

Mỗi field được mô tả theo template sau (9-attribute `ovs-fields(7)` pattern):

| Attribute | Ý nghĩa |
|---|---|
| **Name** | Tên field + aliases (ví dụ `nw_src` = `ip_src` = `ipv4_src`) |
| **Width** | Số bit của field |
| **Format** | Cách parse giá trị (hex / decimal / MAC / IPv4 / IPv6 / string / set) |
| **Masking** | Hỗ trợ mask hay không (bitwise, CIDR prefix, exact-only) |
| **Prerequisites** | Match field nào phải set trước (ví dụ `eth_type=0x0800` cho IP) |
| **Access** | Read-only trong match / read-write qua action `set_field` |
| **OpenFlow version** | 1.0 / 1.1 / 1.3 / 1.5 / NXM (Nicira extension) |
| **OXM/NXM identifier** | Tên symbolic (dùng trong spec parse) |
| **Semantics** | Ý nghĩa field trong gói tin / context pipeline |

Ngoài ra: example match syntax, action liên quan, kịch bản bẻ gãy, upstream reference.

## 4.8.2 Nhóm A, Metadata fields (6 field foundation)

Metadata field không thuộc gói tin wire — chúng là **context** từ pipeline hoặc bridge:

### 4.8.2.1 `in_port`, cổng ingress

| Attribute | Giá trị |
|---|---|
| **Name** | `in_port` (alias: `in_port_oxm` cho OF 1.2+) |
| **Width** | 16-bit (OF 1.0) / 32-bit (OF 1.1+) |
| **Format** | Integer port number hoặc port name |
| **Masking** | Exact match only |
| **Prerequisites** | None |
| **Access** | Read-only trong match |
| **OpenFlow version** | 1.0 all versions |
| **OXM/NXM** | `OXM_OF_IN_PORT` (OF 1.2+), `OXM_OF_IN_PHY_PORT` (OF 1.1+ bonded case), `NXM_OF_IN_PORT` |
| **Semantics** | Datapath port number gói tin arrive. Dùng để differentiate ingress từ port khác nhau |

**Reserved port value:**

| Port name | Value | Ý nghĩa |
|---|---|---|
| `LOCAL` | 0xfffe | Bridge internal port |
| `CONTROLLER` | 0xfffd | OpenFlow controller (packet_in direction) |
| `NORMAL` | 0xfffa | Hành vi L2 learning switch tiêu chuẩn |
| `FLOOD` | 0xfffb | Flood out all ports except ingress |
| `ALL` | 0xfffc | Output to all physical ports |
| `IN_PORT` | 0xfff8 | Send back out ingress port |
| `ANY` | 0xffffffff | Wildcard (match any port) |
| `TABLE` | 0xfff9 | Submit gói tin to OpenFlow pipeline lookup |

**Ví dụ match:**

```
priority=100,in_port=eth0,actions=output:2           # match ingress eth0
priority=100,in_port=LOCAL,actions=ct(commit)        # match bridge-internal
```

**Kịch bản bẻ gãy:**

- Dùng port number numeric (`in_port=1`) thay vì port name (`in_port=tap-vm-web`) → flow rule sẽ break nếu ovs-vswitchd restart assign lại port number.

### 4.8.2.2 `metadata`, 64-bit pipeline metadata

| Attribute | Giá trị |
|---|---|
| **Name** | `metadata` |
| **Width** | 64-bit |
| **Format** | Hex |
| **Masking** | Bitwise mask supported |
| **Prerequisites** | None |
| **Access** | Read qua match; write qua instruction `Write-Metadata` hoặc action `set_field` |
| **OpenFlow version** | 1.1+ |
| **OXM/NXM** | `OXM_OF_METADATA` |
| **Semantics** | 64-bit opaque value set bởi instruction `Write-Metadata` ở table trước, match ở table sau. Dùng pass state giữa các table trong multi-table pipeline |

**Ví dụ use case:**

```
# Table 0 classify + set metadata
table=0,priority=100,ip,nw_src=10.0.1.0/24,
  actions=write_metadata:0x10/0xff,goto_table:10

# Table 10 policy dựa trên metadata
table=10,metadata=0x10/0xff,actions=output:5
```

**Kịch bản bẻ gãy:**

- Metadata chỉ truyền được qua `Goto-Table`, **không truyền qua recirc** (recirc reset metadata). Dùng register nếu cần preserve qua recirc.

### 4.8.2.3 `pkt_mark`, Linux skb mark bridging

| Attribute | Giá trị |
|---|---|
| **Name** | `pkt_mark` |
| **Width** | 32-bit |
| **Format** | Hex hoặc decimal |
| **Masking** | Bitwise |
| **Prerequisites** | None |
| **Access** | Read + write (qua action `set_field`) |
| **OpenFlow version** | NXM extension |
| **OXM/NXM** | `NXM_NX_PKT_MARK` |
| **Semantics** | Liên kết với Linux skb->mark. Dùng pass state giữa OVS và iptables/nftables hoặc kernel L7 filter |

**Ví dụ use case:**

```
# OVS mark packet dùng cho iptables policy
actions=set_field:0x42->pkt_mark,output:eth0

# iptables sau đó match
iptables -A FORWARD -m mark --mark 0x42 -j ACCEPT
```

### 4.8.2.4 `actset_output`, OF 1.5 action set output port

| Attribute | Giá trị |
|---|---|
| **Name** | `actset_output` |
| **Width** | 32-bit |
| **Format** | Integer port number |
| **Masking** | Exact match |
| **Prerequisites** | None |
| **Access** | Read-only (set tự động bởi action `output` trong action set) |
| **OpenFlow version** | 1.5+ |
| **OXM/NXM** | `OXM_OF_ACTSET_OUTPUT` |
| **Semantics** | Port number lệnh `output` trong **action set** hiện tại đang trỏ tới. Cho phép egress table match dựa trên output decision từ ingress |

Sample use: OF 1.5 egress pipeline match `actset_output=5` để áp dụng QoS cho traffic hướng port 5.

### 4.8.2.5 `skb_priority`, Linux QoS priority

32-bit field liên kết `skb->priority`. Dùng với `set_queue` action để select Linux HTB class.

```
actions=set_queue:2,output:eth0    # equivalent set skb_priority
```

### 4.8.2.6 `tun_metadata[0-63]`, Geneve TLV carry

| Attribute | Giá trị |
|---|---|
| **Name** | `tun_metadata0` đến `tun_metadata63` |
| **Width** | Variable (1 byte đến 128 byte per metadata) |
| **Format** | Hex |
| **Masking** | Bitwise |
| **Prerequisites** | Tunnel port (Geneve) |
| **Access** | Read + write |
| **OpenFlow version** | NXM extension |
| **OXM/NXM** | `NXM_NX_TUN_METADATA0` ... `NXM_NX_TUN_METADATA63` |
| **Semantics** | Geneve Variable-Length TLV option. OVN dùng `tun_metadata0` mang logical ingress port + `tun_metadata1` mang logical egress port qua Geneve header |

Chi tiết Geneve TLV class 0x0102 xem Phần 9.27.

## 4.8.3 Nhóm B, Register fields (reg0-reg15 + xreg + xxreg)

Register là **scratch space** per-gói tin trong pipeline — không thuộc gói tin wire, không shared giữa các gói tin. Dùng để pass state giữa các table (hoặc giữa các stage của cùng table).

| Register family | Count | Width | OXM alias |
|---|---|---|---|
| `reg0` đến `reg15` | 16 | 32-bit mỗi cái | `NXM_NX_REG0` ... `NXM_NX_REG15` |
| `xreg0` đến `xreg7` | 8 | 64-bit (bằng 2 register 32-bit ghép) | `OXM_OF_PKT_REG0` ... `OXM_OF_PKT_REG7` |
| `xxreg0` đến `xxreg3` | 4 | 128-bit (bằng 4 register 32-bit ghép) | NXM extension |

### 4.8.3.1 Template B cho `reg0` (đại diện cho nhóm register)

| Attribute | Giá trị |
|---|---|
| **Name** | `reg0` |
| **Width** | 32-bit |
| **Format** | Hex |
| **Masking** | Bitwise + bit-range (ví dụ `reg0[0..15]`) |
| **Prerequisites** | None |
| **Access** | Read + write (qua action `load`, `move`, `set_field`) |
| **OpenFlow version** | NXM extension (OVS specific) |
| **OXM/NXM** | `NXM_NX_REG0` |
| **Semantics** | Scratch register 32-bit per-gói tin, preserve qua pipeline nhưng lost sau gói tin exit |

**Ví dụ load + use:**

```
# Table 0: load 0x42 vào reg0
table=0,priority=100,in_port=1,actions=load:0x42->NXM_NX_REG0[],resubmit(,10)

# Table 10: match reg0 để route
table=10,reg0=0x42,actions=output:5
table=10,reg0=0x00,actions=drop
```

**Bit-range match:**

```
# Match chỉ bit 0-15 của reg0
table=10,reg0=0x0042/0xffff,actions=output:5
```

### 4.8.3.2 OVN dùng register như thế nào

OVN chia register map cứng trong logical flow → OpenFlow translation:

| Register | OVN semantics |
|---|---|
| `reg0` | ACL flags (reg0[0]=ct_hint, reg0[1]=committed, ...) |
| `reg13` | Logical datapath tunnel ID |
| `reg14` | Logical ingress port |
| `reg15` | Logical egress port |
| `xxreg0` / `xxreg1` | IPv6 src / dst working copy |

Kỹ sư debug OVN OpenFlow trace phải thuộc map này. Xem `ovn-architecture(7)` §Tunnel Encapsulation cho table.

## 4.8.4 Nhóm D, L2 Ethernet fields (9 field foundation)

### 4.8.4.1 `eth_src`, `eth_dst`

| Attribute | Giá trị |
|---|---|
| **Name** | `eth_src` (alias: `dl_src`), `eth_dst` (alias: `dl_dst`) |
| **Width** | 48-bit (6 byte) |
| **Format** | MAC `aa:bb:cc:dd:ee:ff` |
| **Masking** | Bitwise mask per byte (ví dụ `01:00:00:00:00:00/01:00:00:00:00:00` cho multicast bit) |
| **Prerequisites** | None |
| **Access** | Read + write (qua `mod_dl_src`, `mod_dl_dst`, `set_field`) |
| **OpenFlow version** | 1.0 all |
| **OXM/NXM** | `OXM_OF_ETH_SRC`, `OXM_OF_ETH_DST` |
| **Semantics** | Ethernet source/destination MAC. Bit 0 của byte 0 = multicast bit (1=multicast/broadcast, 0=unicast) |

**Ví dụ match:**

```
# Multicast traffic: first byte có bit 0 set
priority=100,eth_dst=01:00:00:00:00:00/01:00:00:00:00:00,actions=...

# Broadcast exact
priority=100,eth_dst=ff:ff:ff:ff:ff:ff,actions=flood

# Match OUI vendor (first 3 bytes)
priority=100,eth_src=00:50:56:00:00:00/ff:ff:ff:00:00:00,actions=...
```

### 4.8.4.2 `eth_type`, EtherType

| Attribute | Giá trị |
|---|---|
| **Name** | `eth_type` (alias: `dl_type`) |
| **Width** | 16-bit |
| **Format** | Hex |
| **Masking** | Exact match |
| **Prerequisites** | None |
| **Access** | Read-only (write qua `push_vlan`/`pop_vlan` gián tiếp) |
| **OpenFlow version** | 1.0 all |
| **OXM/NXM** | `OXM_OF_ETH_TYPE` |
| **Semantics** | EtherType của gói tin. Prerequisite của L3 field matching |

**Common value:**

| Value | Protocol | Prerequisite cho field |
|---|---|---|
| `0x0800` | IPv4 | `nw_src/dst`, `nw_proto`, `nw_tos`, etc. |
| `0x86dd` | IPv6 | `ipv6_src/dst`, `ipv6_label` |
| `0x0806` | ARP | `arp_op`, `arp_spa/tpa/sha/tha` |
| `0x8100` | VLAN 802.1Q | (VLAN field tự động kiểm tra) |
| `0x88a8` | Q-in-Q (802.1ad) | PBB field |
| `0x8847` | MPLS unicast | `mpls_label/tc/bos/ttl` |

### 4.8.4.3 `vlan_tci`, `vlan_vid`, `vlan_pcp`, VLAN fields

| Attribute | `vlan_tci` (classic) | `vlan_vid` (OXM) | `vlan_pcp` (OXM) |
|---|---|---|---|
| **Width** | 16-bit (full TCI) | 12-bit | 3-bit |
| **Format** | Hex | Decimal VLAN ID | Decimal 0-7 |
| **Masking** | Bitwise | Bitwise | Exact |
| **Prerequisites** | None | None (VLAN present flag implicit) | None |
| **OpenFlow version** | 1.0 `vlan_tci`; 1.2+ dùng `vlan_vid` + `vlan_pcp` OXM |
| **OXM** | NXM | `OXM_OF_VLAN_VID` | `OXM_OF_VLAN_PCP` |

**CFI/Present bit:** bit 12 của `vlan_tci` = CFI (OF 1.0) hoặc Present flag (OF 1.1+ OXM). Match `vlan_vid=0` + CFI=0 nghĩa là "untagged"; match `vlan_vid=0x1000` + CFI=1 nghĩa là "tagged priority 0".

**Ví dụ match VLAN 100:**

```
priority=100,vlan_vid=100,actions=strip_vlan,output:2
priority=100,dl_vlan=100,actions=mod_vlan_vid:200,output:3
```

### 4.8.4.4 `conj_id`, Conjunctive Match

| Attribute | Giá trị |
|---|---|
| **Name** | `conj_id` |
| **Width** | 32-bit |
| **Format** | Decimal |
| **Masking** | Exact |
| **Prerequisites** | Match gói tin đã qua `conjunction()` action |
| **Access** | Read-only |
| **OpenFlow version** | NXM extension |
| **OXM/NXM** | `NXM_NX_CONJ_ID` |
| **Semantics** | ID của conjunction đã match. Cho phép compress cross-product ACL rule |

**Ví dụ cross-product compression:**

Không dùng conjunction, 3 src × 3 dst = 9 rule. Dùng conjunction = 3+3 rule:

```
# Src rule (conj 1, component 1/2)
priority=100,ip,nw_src=10.0.1.1,actions=conjunction(1,1/2)
priority=100,ip,nw_src=10.0.1.2,actions=conjunction(1,1/2)
priority=100,ip,nw_src=10.0.1.3,actions=conjunction(1,1/2)

# Dst rule (conj 1, component 2/2)
priority=100,ip,nw_dst=10.0.2.1,actions=conjunction(1,2/2)
priority=100,ip,nw_dst=10.0.2.2,actions=conjunction(1,2/2)
priority=100,ip,nw_dst=10.0.2.3,actions=conjunction(1,2/2)

# Final rule match conj_id
priority=100,conj_id=1,actions=output:5
```

Compression factor: N×M rule → N+M rule. OVN ACL dùng pattern này cho Port_Group cross-product.

### 4.8.4.5 `packet_type`, OF 1.5 non-Ethernet gói tin

32-bit field xác định gói tin namespace + type. Mặc định `0:0` = Ethernet. OF 1.5 hỗ trợ packet_type khác cho raw IP (không Ethernet header) hoặc NSH. Ít dùng trong triển khai OVN.

## 4.8.5 Nhóm E, ARP fields (5 field)

### 4.8.5.1 Template B gộp nhóm ARP

| Field | Width | Semantics | OXM |
|---|---|---|---|
| `arp_op` | 16-bit | ARP opcode: 1=Request, 2=Reply, 3=RARP Req, 4=RARP Reply | `OXM_OF_ARP_OP` |
| `arp_spa` | 32-bit IPv4 | ARP sender protocol address | `OXM_OF_ARP_SPA` |
| `arp_tpa` | 32-bit IPv4 | ARP target protocol address | `OXM_OF_ARP_TPA` |
| `arp_sha` | 48-bit MAC | ARP sender hardware address | `OXM_OF_ARP_SHA` |
| `arp_tha` | 48-bit MAC | ARP target hardware address | `OXM_OF_ARP_THA` |

**Prerequisites chung:** `eth_type=0x0806`.

**Ví dụ match ARP Request cho specific IP:**

```
priority=200,arp,arp_op=1,arp_tpa=10.1.0.1,actions=local,controller
```

**Kịch bản bẻ gãy:**

- Match `arp_spa` mà thiếu `eth_type=0x0806` → flow spec sẽ error "ARP field requires eth_type 0x0806". OVS tự suggest add prerequisite với alias `arp`.

## 4.8.6 Nhóm F, L3 IPv4 fields (6 field)

### 4.8.6.1 `ipv4_src`, `ipv4_dst`

| Attribute | Giá trị |
|---|---|
| **Name** | `ipv4_src` (aliases: `ip_src`, `nw_src`), `ipv4_dst` (aliases: `ip_dst`, `nw_dst`) |
| **Width** | 32-bit |
| **Format** | IPv4 dotted `10.0.0.1` hoặc CIDR `10.0.0.0/24` |
| **Masking** | CIDR prefix hoặc bitwise mask |
| **Prerequisites** | `eth_type=0x0800` (or alias `ip`) |
| **Access** | Read + write (`mod_nw_src`, `mod_nw_dst`) |
| **OpenFlow version** | 1.0 all |
| **OXM/NXM** | `OXM_OF_IPV4_SRC`, `OXM_OF_IPV4_DST` |

**Ví dụ match:**

```
# Match subnet
priority=100,ip,nw_src=10.0.0.0/24,actions=output:5

# Match host exact
priority=100,ip,nw_dst=10.0.0.1,actions=drop

# Mask arbitrary bit pattern
priority=100,ip,nw_src=10.0.0.0/0xffffff00,actions=...
```

### 4.8.6.2 `nw_proto`, IP protocol number

| Attribute | Giá trị |
|---|---|
| **Name** | `nw_proto` (alias: `ip_proto`) |
| **Width** | 8-bit |
| **Format** | Integer 0-255, hoặc name |
| **Masking** | Exact |
| **Prerequisites** | `eth_type=0x0800` hoặc `eth_type=0x86dd` |
| **OXM/NXM** | `OXM_OF_IP_PROTO` |

**Common value:**

| Value | Protocol | Prerequisite cho field |
|---|---|---|
| 1 | ICMP | `icmp_type`, `icmp_code` |
| 2 | IGMP | - |
| 6 | TCP | `tcp_src`, `tcp_dst`, `tcp_flags` |
| 17 | UDP | `udp_src`, `udp_dst` |
| 41 | IPv6 encapsulation | - |
| 47 | GRE | - |
| 50 | ESP (IPsec) | - |
| 51 | AH (IPsec) | - |
| 58 | ICMPv6 | `icmpv6_type`, `icmpv6_code` |
| 89 | OSPF | - |
| 132 | SCTP | `sctp_src`, `sctp_dst` |

**Ví dụ alias convenient:**

```
priority=100,tcp,tp_dst=80,actions=...   # eth_type=0x0800, nw_proto=6
priority=100,udp,tp_dst=53,actions=...   # eth_type=0x0800, nw_proto=17
priority=100,icmp,actions=...             # eth_type=0x0800, nw_proto=1
```

### 4.8.6.3 `nw_tos`, `nw_ecn`, `nw_ttl`

| Field | Width | Format | Semantics |
|---|---|---|---|
| `nw_tos` | 6-bit | Integer 0-63 | IPv4 DSCP (upper 6 bit của TOS byte). Dùng cho QoS classification |
| `nw_ecn` | 2-bit | Integer 0-3 | ECN (lower 2 bit của TOS byte). 0=not-ECT, 1=ECT(1), 2=ECT(0), 3=CE (congestion) |
| `nw_ttl` | 8-bit | Integer 0-255 | IPv4 TTL. Decremented mỗi hop |

**Prerequisites:** `eth_type=0x0800` (IPv4) hoặc `eth_type=0x86dd` (IPv6 dùng HL + Traffic Class thay TTL + TOS).

**Action liên quan:**
- `mod_nw_tos`, `mod_nw_ecn`, `mod_nw_ttl`
- `dec_ttl` — giảm TTL/HL theo router semantics

## 4.8.7 Nhóm G, L3 IPv6 fields (7 field)

### 4.8.7.1 `ipv6_src`, `ipv6_dst`

| Attribute | Giá trị |
|---|---|
| **Name** | `ipv6_src` / `ipv6_dst` |

### 4.8.15.1 `in_phy_port` — Physical port sau tunnel decap

**Bucket:** Pipeline metadata.
**Bits:** 32-bit OF port number.
**Maskable:** Không.
**Writable:** Không (read-only metadata).
**Prerequisites:** Không.
**OF version intro:** OF 1.2+ (OXM_OF_IN_PHY_PORT).
**OVS version intro:** OVS 1.7+.
**NXM/OXM:** OXM_OF_IN_PHY_PORT (class 0x8000, type 1).
**Example:**

```
priority=100,in_port=2,in_phy_port=5,actions=goto_table:1
```

**Diễn giải vận hành:** Khi packet đến từ tunnel port (Geneve/VXLAN/GRE), `in_port` trỏ vào tunnel port (logical), còn `in_phy_port` trỏ vào physical NIC port mà tunnel chạy lên trên. Cho phép policy phân biệt "packet đến qua tunnel A nhưng underlay đi qua NIC eth0" vs "qua eth1". Hữu ích cho debug topology multi-NIC + multi-tunnel hoặc tracking source physical port khi audit security.

### 4.8.15.2 `ipv6_flabel` — IPv6 Flow Label

**Bucket:** L3 IPv6 QoS.
**Bits:** 20-bit (low 20 bits của 32-bit field IPv6 header).
**Maskable:** Không.
**Writable:** Không (read-only).
**Prerequisites:** `eth_type=0x86dd` (IPv6).
**OF version intro:** OF 1.2+ (OXM_OF_IPV6_FLABEL).
**OVS version intro:** OVS 1.11+.
**NXM/OXM:** OXM_OF_IPV6_FLABEL (class 0x8000, type 28).
**Example:**

```
priority=100,ipv6,ipv6_flabel=0x12345,actions=output:2
```

**Diễn giải vận hành:** Flow Label (RFC 6437) được sender set để identify một flow IPv6, primary use case là **ECMP hashing** trên router/switch trung gian. Match `ipv6_flabel` cho phép policy theo flow ID mà không cần inspect 5-tuple. Rất hữu ích cho QoS classification (ví dụ flow_label = 0x100000 → premium queue) hoặc traffic engineering (force flow đi qua path cụ thể). Khác `ipv4`: IPv4 không có Flow Label, phải hash 5-tuple cho ECMP. Đây là một trong những lý do IPv6 cải thiện performance core network.

### 4.8.15.3 `ipv6_exthdr` — IPv6 Extension Headers bitmap

**Bucket:** L3 IPv6.
**Bits:** 9-bit (mã hóa trong 16-bit field).
**Maskable:** Có (CIDR-style mask).
**Writable:** Không.
**Prerequisites:** `eth_type=0x86dd`.
**OF version intro:** OF 1.2+ (OXM_OF_IPV6_EXTHDR).
**OVS version intro:** OVS 1.11+.
**NXM/OXM:** OXM_OF_IPV6_EXTHDR (class 0x8000, type 39).
**Example:**

```
priority=200,ipv6,ipv6_exthdr=0x4/0x4,actions=drop
```

**Diễn giải vận hành:** Bitmap 9 bit báo IPv6 extension header nào hiện diện (Hop-by-Hop, Routing, Fragment, Destination Options, AH, ESP, etc.) — chi tiết bit assignment xem section 4.8.7.3. Use case chính: **security policy block packet với extension header bất thường** (Routing Type 0 đã bị deprecated do RH0 attack, nhưng vẫn có thể xuất hiện). Match `ipv6_exthdr=0x4/0x4` (bit 2 = Fragment) drop tất cả IPv6 fragmented traffic — biện pháp chống fragmentation-based evasion. OVS implement bằng cách parse extension header chain ở slow path; không phải mọi NIC offload Geneve/VXLAN cũng support extension header parsing.

### 4.8.15.4 `pbb_isid` — Provider Backbone Bridge I-SID

**Bucket:** L2 Provider Bridge (802.1ah).
**Bits:** 24-bit Service Instance ID.
**Maskable:** Có.
**Writable:** Có (`set_field:0x123456->pbb_isid`).
**Prerequisites:** PBB encapsulation (`eth_type=0x88e7` MAC-in-MAC).
**OF version intro:** OF 1.4+ (OXM_OF_PBB_ISID, optional capability).
**OVS version intro:** OVS 2.6+.
**NXM/OXM:** OXM_OF_PBB_ISID (class 0x8000, type 37).
**Example:**

```
priority=100,eth_type=0x88e7,pbb_isid=0x012345,actions=output:2
```

**Diễn giải vận hành:** PBB (IEEE 802.1ah) là chuẩn carrier Ethernet cho service provider, cho phép encapsulate frame customer trong frame backbone với I-SID 24-bit định danh service instance. Cụ thể, frame backbone có thêm 22-byte header trước Ethernet customer: B-DA + B-SA + B-Tag (VLAN backbone) + I-Tag (chứa I-SID). Service provider dùng I-SID để map customer service tới topology backbone. Match `pbb_isid` cho phép switch core forward theo service ID thay vì VLAN customer (giải quyết vấn đề VLAN 4096 limit).

> **Phạm vi curriculum:** PBB là feature carrier rất chuyên biệt, hiếm gặp trong DC overlay (OVN không dùng PBB, dùng Geneve/VXLAN). Đưa vào catalog để cover OF spec đầy đủ, nhưng deployment thực tế OVS hiếm.

### 4.8.15.5 `mpls_tc` — MPLS Traffic Class

**Bucket:** MPLS QoS.
**Bits:** 3-bit TC (Traffic Class) field trong MPLS shim header.
**Maskable:** Không.
**Writable:** Có (`set_field:5->mpls_tc`).
**Prerequisites:** MPLS (`eth_type=0x8847` unicast hoặc `0x8848` multicast).
**OF version intro:** OF 1.2+ (OXM_OF_MPLS_TC).
**OVS version intro:** OVS 1.11+.
**NXM/OXM:** OXM_OF_MPLS_TC (class 0x8000, type 35).
**Example:**

```
priority=100,eth_type=0x8847,mpls_label=100,mpls_tc=5,actions=set_queue:1,output:2
```

**Diễn giải vận hành:** MPLS shim header (4 byte) chứa: 20-bit Label + 3-bit TC + 1-bit BoS + 8-bit TTL. Field TC (cũng gọi "EXP" trong RFC cũ) dùng cho QoS marking, mapping với DiffServ class. Match `mpls_tc` cho phép switch core ưu tiên traffic theo class (TC=5 = EF Expedited Forwarding cho voice). Action `set_field:N->mpls_tc` rewrite trên ingress edge router để mark packet vào MPLS domain.

### 4.8.15.6 `mpls_bos` — MPLS Bottom of Stack

**Bucket:** MPLS.
**Bits:** 1-bit (0 = có label sau, 1 = label cuối stack).
**Maskable:** Không.
**Writable:** Không (read-only, tự suy ra từ stack depth).
**Prerequisites:** MPLS.
**OF version intro:** OF 1.3+ (OXM_OF_MPLS_BOS).
**OVS version intro:** OVS 1.11+.
**NXM/OXM:** OXM_OF_MPLS_BOS (class 0x8000, type 36).
**Example:**

```
priority=100,eth_type=0x8847,mpls_bos=1,actions=pop_mpls:0x0800,goto_table:5
```

**Diễn giải vận hành:** MPLS có thể stack nhiều label (label switching path cùng tunnel). Bit BoS = 1 nghĩa là label hiện tại là cuối stack, packet sau pop label này sẽ trở thành Ethernet/IP gốc. Match `mpls_bos=1` thường dùng ở **edge router** để biết khi nào pop hết label + revert về L2/L3 gốc. Action `pop_mpls:<ethertype>` chỉ làm việc khi BoS=1 (pop label cuối + reset eth_type về tham số cho).

### 4.8.15.7 `sctp_src` / `sctp_dst` — SCTP source/destination port

**Bucket:** L4 Transport.
**Bits:** 16-bit each.
**Maskable:** Có (range mask).
**Writable:** Có (`set_field:5060->sctp_dst`).
**Prerequisites:** SCTP (`eth_type=0x0800/0x86dd`, `ip_proto=132`).
**OF version intro:** OF 1.2+ (OXM_OF_SCTP_SRC/DST).
**OVS version intro:** OVS 2.0+.
**NXM/OXM:** OXM_OF_SCTP_SRC (type 17) + OXM_OF_SCTP_DST (type 18).
**Example:**

```
priority=100,ip,ip_proto=132,sctp_dst=2905,actions=output:2
```

**Diễn giải vận hành:** SCTP (RFC 4960) là protocol L4 thứ 3 (sau TCP+UDP), dùng chủ yếu cho telecom signaling (Diameter, M3UA, SIGTRAN). Mặc dù không phổ biến trong DC, telecom workload (5G core, IMS) chạy trên container OpenStack có thể cần OVN policy match SCTP. SCTP có ưu điểm multistreaming + multihoming + heartbeat built-in, nhưng OVN ACL hiện chỉ support port-based matching (không inspect SCTP chunk).

### 4.8.15.8 `tunnel_id` (alias `tun_id`) — Tunnel ID 64-bit

**Bucket:** Tunnel metadata.
**Bits:** 64-bit.
**Maskable:** Có.
**Writable:** Có (`set_field:0x5678->tun_id`).
**Prerequisites:** Không (luôn available, là 0 nếu packet không từ tunnel).
**OF version intro:** OF 1.3+ (OXM_OF_TUNNEL_ID).
**OVS version intro:** OVS 1.1+ (NXM_NX_TUN_ID), OVS 2.4+ (OXM_OF_TUNNEL_ID).
**NXM/OXM:** NXM_NX_TUN_ID (class 0x0001, type 16) hoặc OXM_OF_TUNNEL_ID (class 0x8000, type 38).
**Example:**

```
priority=100,tun_id=0x1234,actions=set_field:0x5->reg0,goto_table:8
```

**Diễn giải vận hành:** `tunnel_id` là 64-bit metadata gắn liền với packet đến từ tunnel port. Encoding tùy theo tunnel type:
- **VXLAN**: 24-bit VNI nằm trong 24 bit thấp; 40 bit cao = 0
- **Geneve**: 24-bit VNI tương tự VXLAN
- **GRE với key option**: 32-bit GRE key trong 32 bit thấp
- **STT**: 64-bit context ID đầy đủ

Khi packet rời br-int qua tunnel port, ovs-vswitchd encapsulate với VNI/key = giá trị `tunnel_id` được set bởi flow rule trước output. OVN dùng `tunnel_id` để gán datapath ID logical (logical switch hoặc router), kết hợp với reg14/reg15 (inport/outport) làm full pipeline metadata. Đây là cốt lõi cross-chassis transition trong OVN.

### 4.8.15.9 `xreg0` đến `xreg7` — Extended 64-bit registers

**Bucket:** Pipeline state register.
**Bits:** 64-bit each (8 register).
**Maskable:** Có.
**Writable:** Có.
**Prerequisites:** Không.
**OF version intro:** OF 1.3+ (OXM_OF_PKT_REG x).
**OVS version intro:** OVS 2.4+.
**NXM/OXM:** OXM_OF_PKT_REG0..7 (class 0x0001, type 0..7).
**Example:**

```
priority=100,actions=load:0xCAFEBABE12345678->xreg0,goto_table:5
table=5,xreg0=0xCAFEBABE12345678,actions=output:2
```

**Diễn giải vận hành:** `xreg0..xreg7` là alias 64-bit cho cặp register 32-bit liên tiếp (xreg0 = reg0+reg1, xreg1 = reg2+reg3, etc.). Chỉ 8 xreg vì có 16 reg vật lý. Cho phép match/set giá trị 64-bit nguyên khối (ví dụ MAC address 48-bit fit trong 64-bit register). OVN dùng xreg cho IPv4 nexthop scratch (4 byte không đủ trong 32-bit xreg2 + Mac 48-bit).

### 4.8.15.10 `xxreg0` đến `xxreg3` — Extended 128-bit registers

**Bucket:** Pipeline state register.
**Bits:** 128-bit each (4 register).
**Maskable:** Có.
**Writable:** Có.
**Prerequisites:** Không.
**OF version intro:** OVS extension (không phải standard OF).
**OVS version intro:** OVS 2.6+.
**NXM/OXM:** Custom OVS extension (no class assigned in OF spec).
**Example:**

```
priority=100,actions=load:0x20010db8000000000000000000000001->xxreg0,goto_table:5
```

**Diễn giải vận hành:** `xxreg0..xxreg3` là alias 128-bit cho 4 reg 32-bit liên tiếp (xxreg0 = reg0+reg1+reg2+reg3, xxreg1 = reg4+reg5+reg6+reg7, etc.). Đặc biệt hữu ích cho **IPv6 address scratch** (128-bit fit nguyên xxreg). OVN dùng xxreg0 cho IPv6 ND/NA processing (xem Phần 13.17 §13.17.2). Cẩn thận: ghi vào xxreg0 sẽ overwrite reg0+1+2+3 đồng thời, mất bất kỳ REGBIT đã set trong reg0.

### 4.8.15.11 NSH fields — `nsh_spi`, `nsh_si`, `nsh_c1` đến `nsh_c4`

**Bucket:** Service Function Chaining (NSH RFC 8300).
**Bits:** `nsh_spi` 24-bit, `nsh_si` 8-bit, `nsh_c1`-`nsh_c4` 32-bit each.
**Maskable:** Có cho `nsh_c1`-`nsh_c4`.
**Writable:** Có.
**Prerequisites:** NSH (`eth_type=0x894f` hoặc `packet_type=(1, 0x894f)` với OF 1.5+ packet-type-aware).
**OF version intro:** OF 1.5+ với extension experimenter (NSH chưa standardize trong OF base).
**OVS version intro:** OVS 2.8+.
**NXM/OXM:** Experimenter class với NSH-specific type.
**Example:**

```
priority=100,nsh,nsh_spi=0x123,nsh_si=255,actions=output:2
priority=100,nsh,nsh_si=128,actions=set_field:0xdeadbeef->nsh_c1,goto_table:5
```

**Diễn giải vận hành:** NSH (Network Service Header, RFC 8300) là protocol cho **Service Function Chaining (SFC)** — chuỗi packet qua nhiều dịch vụ (firewall → IDS → load balancer → DPI). NSH header có:
- **SPI (Service Path Identifier)** 24-bit: định danh chain
- **SI (Service Index)** 8-bit: vị trí hiện tại trong chain (decrement mỗi service hop)
- **C1-C4** (Context Headers) 32-bit each: metadata service-specific (ví dụ tenant ID, application class)

Use case: cloud provider deploy SFC cho tenant — packet vào SPI=1 (chain "web tenant A") đi qua SI=255 (firewall) → SI=254 (IDS) → SI=253 (output). Mỗi service decrement SI và forward về service tiếp. NSH thay vì hardcode topology, định nghĩa overlay logic độc lập với physical path.

NSH trong curriculum chỉ giới thiệu nông để cover OF spec đầy đủ. Triển khai OVN production thường KHÔNG dùng NSH (chọn cách khác đơn giản hơn như chained logical router).

### 4.8.15.12 `packet_type` — Packet type-aware pipeline (OF 1.5+)

**Bucket:** Pipeline metadata.
**Bits:** 32-bit (16-bit namespace + 16-bit type).
**Maskable:** Không.
**Writable:** Có (action `decap` set lại).
**Prerequisites:** Không (luôn available, default = (0, 0) Ethernet).
**OF version intro:** OF 1.5+ (OXM_OF_PACKET_TYPE).
**OVS version intro:** OVS 2.8+.
**NXM/OXM:** OXM_OF_PACKET_TYPE (class 0x8000, type 44).
**Example:**

```
# Match packet IPv4 trần (no Ethernet)
priority=100,packet_type=(1,0x0800),ip_proto=6,actions=encap(ethernet),output:2
```

**Common values:**

| `packet_type` | Encapsulation |
|---------------|---------------|
| `(0, 0)` | Ethernet (default) |
| `(0, 0x8847)` | MPLS unicast |
| `(0, 0x86dd)` | IPv6 |
| `(1, 2)` | IPv4 bare (no Ethernet) |
| `(1, 0x894f)` | NSH |

**Diễn giải vận hành:** OF 1.0-1.4 giả định mọi packet có Ethernet header. OF 1.5+ với packet-type-aware pipeline cho phép match + xử lý packet không-Ethernet (bare IP, NSH, MPLS-only). Match `packet_type=(1,2)` cho phép xử lý packet IPv4 trần (ví dụ output từ IPSec tunnel với mode transport). Action `encap(ethernet)` thêm Ethernet header trở lại để output ra port thường. Use case chính: SFC NSH integration, IPSec gateway, GTP-U tunneling cho 5G.

## Điểm cốt lõi cần nhớ

Thứ nhất, OpenFlow từ 12-tuple cứng (1.0) lên 100+ field extensible (1.5 + NXM) qua OXM TLV. Phần 4.1 kể lịch sử chuyển đổi; Phần này liệt kê catalog.

Thứ ba, 9-attribute anatomy (Name / Width / Format / Masking / Prerequisites / Access / OpenFlow version / OXM-NXM / Semantics) là mental model chuẩn cho mỗi field. Khi debug flow rule không match, luôn kiểm chứng: prerequisite đã set? Mask đúng? Version OpenFlow support?

Thứ tư, register (reg0-reg15, xreg, xxreg) không thuộc gói tin wire — chúng là scratch pad per-gói tin trong pipeline. OVN chia register map cứng (reg13=datapath, reg14=ingress port, reg15=egress port), engineer OVN phải thuộc.

Thứ sáu, conjunctive match (`conj_id` + action `conjunction()`) compress cross-product rule N×M → N+M. OVN ACL Port_Group dùng pattern này extensively — hiểu conjunction giúp đọc `ovn-sbctl lflow-list` mà không hoang mang trước flood of `conjunction(X, Y/Z)`.

Mỗi field có một mục riêng dưới đây với cách trình bày 20 mặt theo heading tự nhiên tiếng Việt: khái niệm, lịch sử, vị trí trong kiến trúc, vai trò, lý do tồn tại, vấn đề giải quyết, tầm quan trọng, cơ chế hoạt động, OXM TLV byte layout, ví dụ match + write, cách kỹ sư vận hành thành thạo, phân loại, source code anchor, kịch bản bẻ gãy, bài tập synthetic, và so sánh với hệ khác.

Nhóm L2 (5 field: `in_port`, `eth_src`, `eth_dst`, `eth_type`, `vlan_vid`) trình bày §4.8.X.1 đến §4.8.X.5. Nhóm L3, L4, conntrack thứ cấp sẽ bổ sung trong các đợt biên soạn kế tiếp; trong khi chờ, nội dung tham chiếu nhanh nằm ở các phần native §4.8.6 (L3 IPv4), §4.8.7 (L3 IPv6), §4.8.8 (L4), §4.8.11 (conntrack).

### 4.8.X.1 `in_port` — cổng ingress của packet vào datapath

**Khái niệm.** `in_port` là match field định danh **cổng vật lý hoặc cổng logic** mà packet đi vào datapath OVS qua đó. Đây là field cơ bản nhất trong toàn bộ OpenFlow: khoảng 90% flow rule production có chứa `in_port` ở vế match, vì nó là điểm phân biệt đầu tiên giữa các luồng traffic đến từ tap khác nhau, từ tunnel khác nhau, hoặc từ uplink khác nhau. Trong OVN, `in_port` của br-int chính là `tunnel_key` của Logical_Switch_Port mà ovn-controller đã ánh xạ sẵn ở table 0. Loại field: **metadata fundamental**, sub-loại "ingress identifier".

**Lịch sử + bối cảnh.** OpenFlow 1.0 (2009) định nghĩa `OFPFW_IN_PORT` 16-bit. OpenFlow 1.1 (2011) mở rộng lên 32-bit để hỗ trợ deployment có nhiều hơn 65k cổng (chassis với SR-IOV VF số lượng lớn). OpenFlow 1.2 (2011) chuẩn hoá thành OXM TLV `OXM_OF_IN_PORT` (oxm_class = 0x8000 OPENFLOW_BASIC, oxm_field = 0, length = 4 byte). Giai đoạn 2010-2014 OVS dùng song song NXM legacy (`NXM_OF_IN_PORT` dạng 2 byte) và OXM standard; từ OVS 2.3 (2014) `in_port` mặc định serialize ra OXM khi controller dùng OF 1.2+.

**Vị trí trong kiến trúc.** Field thuộc nhóm metadata, không phải header thật của packet — giá trị do datapath OVS gán khi packet enter, không nằm trong byte stream Ethernet. Trong cấu trúc `struct flow` (`include/openvswitch/flow.h`) field tương ứng là `in_port.ofp_port` (32-bit OpenFlow port number). MFF identifier upstream OVS: `MFF_IN_PORT` (line 716 `meta-flow.h`) cho format 16-bit legacy + `MFF_IN_PORT_OXM` (line 732) cho format 32-bit OXM.

**Vai trò.** Vai trò chính: phân biệt nguồn gốc packet để áp policy khác nhau cho từng nguồn (ví dụ packet từ tap VM `tap0` được áp ACL tenant A, packet từ uplink `eth1` được áp ACL của underlay). Vai trò phụ: làm tham số cho action `IN_PORT` (port reserved 0xfff8) khi cần gửi packet ra chính cổng đã vào (hairpin, một số NDP scenario). Phạm vi ảnh hưởng: per-packet, cố định trong toàn bộ pipeline (không bị ghi đè khi recirc).

**Vì sao sinh ra.** Trước OpenFlow, switch silicon ASIC có khái niệm "ingress port" trong CAM lookup nhưng không expose ra control plane lập trình được. OpenFlow tổng quát hoá khái niệm này thành field bậc nhất, cho phép controller viết flow như `priority=100,in_port=2,actions=output:5` mà không cần biết switch silicon nào bên dưới. Đây là một trong 12 field gốc của OpenFlow 1.0 (cùng với 6 L2 field, 4 L3 field, và 2 L4 field trong "OFP_MATCH").

**Vấn đề giải quyết.** Cụ thể: thay thế cấu hình bridge port-mapping cứng (như Linux bridge `brctl addif`) bằng flow rule programmable runtime. Cách giải: datapath gán `in_port` khi packet enter, classifier match field như mọi field khác, action `output:N` chỉ định cổng ra, action `IN_PORT` reserved cho phép gửi ra chính cổng vào. Phạm vi giải quyết: mọi scenario cần multi-port forwarding, từ basic L2 learning đến sophisticated SDN policy.

**Tầm quan trọng.** Tier 2 medium nhưng frequency cực cao. Field xuất hiện trong gần như mọi flow của br-int (OVS) và br-ex (OVN gateway chassis). Engineer must-know: phân biệt giá trị numeric vs port name, hiểu reserved port (LOCAL=0xfffe, CONTROLLER=0xfffd, NORMAL=0xfffa, FLOOD=0xfffb, ALL=0xfffc, IN_PORT=0xfff8, ANY=0xffffffff, TABLE=0xfff9), và biết rằng port number numeric có thể đổi sau khi ovs-vswitchd restart nếu interface không có persistent ofport.

**Cơ chế hoạt động.** Khi packet enter qua netdev (kernel datapath) hoặc DPDK port (userspace datapath), datapath đọc `dp_port` từ context (kernel: `OVS_KEY_ATTR_IN_PORT` trong nlattr, userspace: trực tiếp từ port table) và translate sang `ofp_port` qua `ofproto/ofproto-dpif.c` lookup table. Giá trị này gắn vào `struct flow.in_port.ofp_port` rồi đi vào classifier TSS lookup. Match `in_port=N` so sánh exact 32-bit không cho mask (theo OF 1.5 spec §A.2.3.1).

**OXM TLV byte layout.** Header 4 byte: `oxm_class` 2 byte = `0x8000` (OPENFLOW_BASIC), `oxm_field|hasmask` 1 byte = `0x00` (field=0, hasmask=0), `oxm_length` 1 byte = `0x04`. Theo sau là 4 byte big-endian port number. Tổng TLV: 8 byte. So sánh với `OXM_OF_IN_PHY_PORT` (oxm_field=1, dùng cho bonded/aggregate port để trả về physical port thực) — `in_phy_port` đã có Phần 4.8.15.1 phân tích.

**Cách kỹ sư vận hành thành thạo.** Bốn kỹ năng: (1) **Đọc**: `ovs-ofctl show br-int | grep "<port_name>"` để map port name ↔ ofport number trước khi viết flow. (2) **Sửa**: ưu tiên port name (`in_port=tap0`) trong rule manual để tránh break khi ofport đổi; với rule do controller cài, dùng numeric vì controller maintain mapping riêng. (3) **Quan sát**: `ovs-ofctl dump-flows br-int | grep in_port=` đếm flow per ingress port. (4) **Debug**: `ovs-appctl ofproto/trace br-int 'in_port=2,...'` simulate packet vào port 2 + xem path qua pipeline.

**Phân loại.** Field metadata (không phải header), exact-match-only (no mask), read-only trong match (write thông qua action `set_field` trong OF 1.5 packet-out scenario rất hiếm). Không có prerequisite. Khác với `metadata` (write được qua Write-Metadata) và `reg0..reg15` (write thoải mái qua set_field).

**Ví dụ match.** Cú pháp OVS `ovs-ofctl`:

```
priority=100,in_port=1,actions=output:2                    # ingress port 1 → port 2
priority=100,in_port=tap0,actions=resubmit(,10)            # port name (text)
priority=100,in_port=LOCAL,actions=ct(commit),NORMAL       # bridge-internal port
priority=100,in_port=2,actions=in_port                     # hairpin: gửi ra chính port 2
```

Cú pháp OXM hexstream (debug tool):

```
NXM:  04 00 00 02         # NXM_OF_IN_PORT, length=2, port=2
OXM:  80 00 00 04 00 00 00 02   # OXM_OF_IN_PORT, length=4, port=2
```

**Cách quan sát + xác minh.** Ba phương pháp: (1) `ovs-ofctl show br-int` liệt kê mọi ofport hiện hành kèm port name, link state, peer info. (2) `ovs-vsctl --columns=name,ofport list interface` lấy mapping từ OVSDB. (3) `ovs-appctl dpctl/show` xem datapath-level dp_port (khác với ofport, dp_port là internal cho datapath kernel/userspace).

**Source code.** Function anchor `ofp_to_u16`, `u16_to_ofp` (OVS `lib/ofp-util.c`) handle conversion 16↔32 bit cho compatibility. Function `extract_metadata_from_packet_metadata` (`lib/odp-util.c`) extract dp_port từ nlattr datapath. Branch baseline: OVS v3.3.0.

**Kịch bản bẻ gãy.** (a) Dùng `in_port=1` numeric trong rule manual → restart ovs-vswitchd → ofport 1 giờ là interface khác → flow match sai mục tiêu. Fix: dùng port name. (b) Dùng `in_port=LOCAL` cho rule production trên br-int → khi VM cùng host gửi packet local (Linux stack inject) thì match LOCAL nhưng trace có thể bị nhầm tưởng là tunnel. Fix: thêm điều kiện L3 phân biệt.

**Bài tập synthetic.** Mininet topology 1 OVS bridge `s1` với 3 host h1/h2/h3, dùng `ovs-ofctl add-flow s1 'priority=100,in_port=1,actions=output:2'` để cho phép h1→h2 và `priority=100,in_port=2,actions=output:1` cho h2→h1. Verify h1↔h2 ping pass, h1↔h3 ping fail (vì không có flow in_port=1→port 3).

**So sánh với hệ khác.** Linux bridge có khái niệm "incoming bridge port" trong `br_input.c` nhưng không expose ra rule — chỉ dùng nội bộ FDB learning. Cisco IOS classic ACL trên interface implicit bind với "inbound interface" (giống `in_port`) qua câu lệnh `ip access-group N in`. NSX-T có `source_logical_port_uuid` trong distributed firewall rule, semantically tương đương `in_port` mapping qua Logical_Switch_Port. P4 model hoá thành `standard_metadata.ingress_port` trong `v1model.p4` — chính là kế thừa concept của OpenFlow.

### 4.8.X.2 `eth_src` — địa chỉ MAC nguồn của frame Ethernet

**Khái niệm.** `eth_src` là match field 48-bit chứa địa chỉ MAC nguồn của frame Ethernet (6 byte, vị trí byte 6-11 trong header Ethernet). Alias legacy: `dl_src` (data-link source). Field này có ý nghĩa quan trọng cho ba bài toán: anti-spoofing (chặn MAC spoofing của tenant), MAC learning (bảng FDB của switch L2), và identity-based ACL (filter theo identity host). Loại field: **L2 header field**, sub-loại "source identifier".

**Lịch sử + bối cảnh.** OpenFlow 1.0 (2009) định nghĩa `OFPMT_DL_SRC` 6-byte trong struct match, hỗ trợ exact match nhưng không hỗ trợ mask. OpenFlow 1.1 (2011) thêm bitwise mask cho src+dst MAC (đặc biệt hữu dụng cho match multicast bit `01:00:00:00:00:00/01:00:00:00:00:00`). OF 1.2 (2011) chuẩn hoá OXM `OXM_OF_ETH_SRC` (oxm_field=4) và `OXM_OF_ETH_SRC_W` masked variant (hasmask=1).

**Vị trí trong kiến trúc.** Trong `struct flow` field `dl_src` (mảng 6 byte). MFF identifier upstream OVS: `MFF_ETH_SRC` (line 1104 `meta-flow.h`). Class trong taxonomy 4.8: thuộc Nhóm D L2 Ethernet fields (cùng với `eth_dst`, `eth_type`, VLAN fields).

**Vai trò.** Vai trò chính trong OVN: thực thi `Logical_Switch_Port.port_security` ở stage `LS_IN_PORT_SEC_L2` — northd compile thành flow `match=(inport=="lsp1" && eth.src=={MAC1, MAC2})` priority 50 action `next`, packet không match rớt qua priority 0 drop. Vai trò trong OVS standalone: input của bảng MAC learning (FDB), action `learn(table=10, priority=100, NXM_OF_ETH_DST[]=NXM_OF_ETH_SRC[], output:NXM_OF_IN_PORT[])` học src MAC + ingress port để future packet có dst MAC này biết đường ra.

**Vì sao sinh ra.** Trước OpenFlow, switch silicon hardware tự động học src MAC qua quá trình "MAC learning" implicit, không cho phép control plane kiểm soát hay chặn. OpenFlow exposed src MAC như field bậc nhất để: (a) controller-driven learning thay cho self-learning, (b) anti-spoofing rule programmable, (c) identity-based policy. Đây là 1 trong 4 L2 field gốc của OF 1.0.

**Vấn đề giải quyết.** Cụ thể MAC spoofing: tenant trong VM có thể gửi packet với `eth.src` giả mạo MAC của VM khác (driver tap kernel Linux không filter src MAC). Hệ quả: ARP cache poison, MITM attack. Cách giải qua OVN: declare `Logical_Switch_Port.port_security` list MAC cho phép, northd compile thành flow drop tại stage `LS_IN_PORT_SEC_L2`. Phạm vi giải quyết: tenant boundary tại tap port, không bao gồm spoofing trong overlay tunnel (phải tin tunnel endpoint).

**Tầm quan trọng.** Tier 2 medium. Trong OVN, mọi LSP có port_security đều phát sinh flow match `eth.src` ở stage logic 0. Trong OVS standalone với learning bridge, mọi packet pass qua flow learn match `eth.src`. Engineer must-know: format MAC (hexadecimal colon-separated), bitwise mask byte-level, multicast bit (bit 0 byte 0), unicast/locally-administered bit (bit 1 byte 0).

**Cơ chế hoạt động.** Datapath parse Ethernet header, extract bytes 6-11 vào `flow.dl_src`. Classifier TSS so sánh exact 6 byte hoặc bitwise mask nếu rule có mask. Với mask, packet match khi `(packet.dl_src & rule.mask) == (rule.dl_src & rule.mask)`. OXM TLV layout: header 4 byte (`oxm_class=0x8000, oxm_field=4, hasmask=0, length=6`) + 6 byte value, hoặc với mask: `length=12` + 6 byte value + 6 byte mask.

**Bitwise mask use case.** (a) Match toàn bộ multicast/broadcast: `eth_src=01:00:00:00:00:00/01:00:00:00:00:00` — match khi bit 0 byte 0 = 1. (b) Match OUI vendor: `eth_src=00:50:56:00:00:00/ff:ff:ff:00:00:00` — match mọi MAC bắt đầu `00:50:56` (VMware OUI). (c) Match locally-administered: `eth_src=02:00:00:00:00:00/02:00:00:00:00:00` — bit 1 byte 0 = 1.

**Ví dụ match + write.**

```
# Anti-spoof tenant 1: chỉ cho phép 2 MAC declare
priority=50,in_port=tap0,eth_src=AA:11:22:33:44:55,actions=resubmit(,10)
priority=50,in_port=tap0,eth_src=AA:11:22:33:44:56,actions=resubmit(,10)
priority=0,in_port=tap0,actions=drop

# Multicast detection
priority=100,eth_src=01:00:00:00:00:00/01:00:00:00:00:00,actions=drop  # multicast src bất hợp lệ

# MAC rewrite (set_field write)
priority=100,actions=set_field:00:11:22:33:44:55->eth_src,output:2

# Legacy alias
priority=100,dl_src=00:50:56:aa:bb:cc,actions=output:3
```

**Cách kỹ sư vận hành thành thạo.** (1) **Đọc**: hiểu format MAC + bit semantics. (2) **Sửa**: thêm rule với `set_field:...->eth_src` hoặc legacy `mod_dl_src:...`. (3) **Quan sát**: `ovs-ofctl dump-flows br-int | grep eth_src` đếm hit per src MAC. (4) **Debug**: `ovs-appctl ofproto/trace br-int 'in_port=1,eth_src=AA:11:22:33:44:55,...'` xem rule nào match.

**Source code.** Function anchor `mf_set_value` (`lib/meta-flow.c`) cho `MFF_ETH_SRC` write path. Function `parse_dl_src` (`lib/match.c`) parse text input. Branch baseline: OVS v3.3.0.

**Kịch bản bẻ gãy.** Quên port_security khi tạo LSP → tenant spoof MAC gateway → MITM attack subnet. Fix: `ovn-nbctl lsp-set-port-security lsp1 "AA:11:22:33:44:55"` ngay khi tạo LSP, audit định kỳ `lsp-get-port-security` cho mọi LSP.

**Bài tập synthetic.** OVN topology 1 LS với 2 LSP, set port_security cho lsp-good (MAC khớp VM), không set cho lsp-bad. Từ VM trên lsp-bad gửi packet `eth.src=AA:11:11:11:11:11` (giả mạo) → verify packet đi qua (vì port_security empty = allow all). Set `lsp-set-port-security lsp-bad "AA:22:22:22:22:22"` rồi gửi lại với src giả mạo → verify drop tại table 8.

**So sánh với hệ khác.** Linux iptables `-m mac --mac-source AA:11:...` trên `FORWARD` chain bridge-mode tương đương. Cisco port-security `switchport port-security mac-address AA11.2233.4455` trên access port. NSX-T `LogicalPort.address_bindings` field `mac_address` semantically tương đương port_security của OVN.

### 4.8.X.3 `eth_dst` — địa chỉ MAC đích của frame Ethernet

**Khái niệm.** `eth_dst` là match field 48-bit chứa địa chỉ MAC đích (byte 0-5 của Ethernet header). Alias legacy: `dl_dst`. Đây là field then chốt cho L2 forwarding decision: switch tra cứu FDB theo `eth_dst` để biết gửi packet ra cổng nào. Trong OVN, `eth_dst` là input cho stage `LS_IN_L2_LKUP` (table 27) — stage cornerstone C6 của LS pipeline. Loại field: **L2 header field**, sub-loại "destination identifier".

**Lịch sử + bối cảnh.** Cùng lịch sử với `eth_src`: OF 1.0 exact, OF 1.1 mask, OF 1.2 OXM (`OXM_OF_ETH_DST` field=3, masked variant `_W` hasmask=1). OVS bổ sung NXM legacy `NXM_OF_ETH_DST`. Pattern dùng mask cho `eth_dst` đặc biệt phổ biến với multicast group (IPv4 multicast map `01:00:5e:XX:XX:XX`, IPv6 multicast map `33:33:XX:XX:XX:XX`).

**Vị trí trong kiến trúc.** `struct flow.dl_dst` 6-byte. MFF: `MFF_ETH_DST` (line 1120 `meta-flow.h`). Trong taxonomy 4.8: Nhóm D L2 Ethernet.

**Vai trò.** Vai trò chính: forwarding decision L2. Trong OVN: stage `LS_IN_L2_LKUP` (table 27) sinh flow per LSP với match `(eth.dst == AA:11:...)` action `outport = lsp1; output;` — đây là "MAC table" của distributed virtual switch. Trong OVS standalone với learning: action `NORMAL` tự động tra FDB theo `eth.dst`, controller-driven thì có flow learn() cài. Vai trò phụ: detect broadcast (`eth_dst=ff:ff:ff:ff:ff:ff`) cho ARP request flooding, detect multicast cho IGMP snooping.

**Vì sao sinh ra.** L2 forwarding là chức năng nền tảng nhất của switch, OF không thể thiếu field này. Từ OF 1.0 nó đã được ưu tiên đứng đầu match struct (cùng với `eth_src` và `eth_type`). Pre-OpenFlow, switch silicon tự xử lý hardware MAC learning + lookup không programmable; OF cho phép controller can thiệp toàn bộ logic này.

**Vấn đề giải quyết.** Cụ thể: làm sao switch biết gửi packet ra cổng nào dựa trên dst MAC. Cách giải truyền thống: MAC learning + flooding fallback (broadcast khi miss). Cách giải OVN: northd biết cấu hình LSP↔MAC (qua `Logical_Switch_Port.addresses`) → compile static flow `eth.dst==MAC → output:LSP_port` ở stage L2_LKUP. Không cần learning, không có flooding spurious. Phạm vi giải quyết: known unicast MAC; unknown unicast / broadcast / multicast xử lý qua stage tách (`LS_IN_L2_UNKNOWN`).

**Tầm quan trọng.** Tier 2 medium nhưng cornerstone của LS pipeline. Stage `LS_IN_L2_LKUP` là điểm router hoá MAC table → output port; mọi traffic VM-to-VM đi qua. Engineer must-know: phân biệt unicast vs multicast vs broadcast bit semantics, hiểu mask cho multicast group, biết OVN compile pattern (priority 50 per LSP MAC + priority 0 fallback MC_FLOOD).

**Cơ chế hoạt động.** Datapath parse Ethernet header byte 0-5 vào `flow.dl_dst`. Classifier match exact hoặc bitwise mask. Trong OVN compile: `consider_lsp_l2_lookup()` (`northd/northd.c`) iterate mọi LSP của LS, sinh flow với match `outport=="<lsp>"` không, mà thực ra match `eth.dst == lsp.addresses.mac`, action `outport = "<lsp>"; output;`.

**OXM TLV byte layout.** `oxm_class=0x8000`, `oxm_field=3`, `hasmask=0`, `length=6`, value 6 byte. Masked variant: `hasmask=1`, `length=12`, value 6 byte + mask 6 byte. Tổng exact: 10 byte; masked: 16 byte.

**Ví dụ match.**

```
# L2 forwarding theo dst MAC (OVN-style)
priority=50,metadata=0x1,eth_dst=AA:11:22:33:44:01,actions=load:0x2->NXM_NX_REG15[],resubmit(,32)
priority=50,metadata=0x1,eth_dst=AA:11:22:33:44:02,actions=load:0x3->NXM_NX_REG15[],resubmit(,32)
priority=0,metadata=0x1,actions=load:0xffff->NXM_NX_REG15[],resubmit(,32)  # MC_FLOOD

# Broadcast ARP
priority=100,eth_dst=ff:ff:ff:ff:ff:ff,actions=flood

# IPv4 multicast group (224.0.0.0/4 → 01:00:5e:00:00:00/01:00:5e:00:00:00 mask)
priority=100,eth_dst=01:00:5e:00:00:00/ff:ff:ff:80:00:00,actions=resubmit(,igmp_snoop_table)

# IPv6 multicast group (ff02::/16 → 33:33:00:00:00:00 mask)
priority=100,eth_dst=33:33:00:00:00:00/ff:ff:00:00:00:00,actions=resubmit(,nd_table)

# MAC rewrite trước gửi qua tunnel (OVN logical→physical translation)
priority=100,actions=set_field:AA:11:22:33:44:99->eth_dst,output:5
```

**Cách kỹ sư vận hành thành thạo.** (1) **Đọc**: hiểu compile pattern OVN L2_LKUP, đọc flow `ovs-ofctl dump-flows br-int table=27 | grep eth_dst`. (2) **Sửa**: trong OVN qua `lsp-set-addresses` (đổi MAC của LSP); standalone OVS qua flow add manual. (3) **Quan sát**: counter packet/byte per dst MAC entry — phát hiện asymmetric traffic. (4) **Debug**: nếu VM A không ping được VM B, ovn-trace packet với `eth.dst=B_MAC`, xem stage 27 match flow nào.

**Source code.** Function `build_lswitch_destination_lookup_bmcast` (`northd/northd.c`) build flow cho broadcast/multicast tại L2_LKUP. Function `consider_port_binding_l2_lookup` build flow per LSP. Branch baseline: OVN v22.03.

**Kịch bản bẻ gãy.** (a) Hai LSP cùng LS có cùng MAC (config sai) → northd skip một LSP → packet đến MAC này flood → traffic trùng lặp. Fix: audit `lsp-list ls1` xem MAC duplicate. (b) Quên gắn `addresses` cho LSP → northd không sinh flow L2_LKUP → packet đến MAC này luôn flood. Fix: `lsp-set-addresses lsp1 "AA:11:..."`.

**Bài tập synthetic.** Mininet 1 OVS bridge `s1` 3 host h1/h2/h3 với MAC tương ứng. Cài 3 flow: `eth_dst=h1.mac→output:1`, `eth_dst=h2.mac→output:2`, `eth_dst=h3.mac→output:3` + `eth_dst=ff:ff:ff:ff:ff:ff→flood` cho ARP. Verify ping h1↔h2↔h3 đều hoạt động, đếm counter mỗi flow để confirm path.

**So sánh với hệ khác.** Linux bridge `fdb` table tự học eth_dst entry, cmd `bridge fdb show` xem table. Cisco MAC address-table `show mac address-table` tương đương. NSX-T MAC learning logical switch field `replication_mode` quyết định flooding behavior. Tất cả đều lookup theo dst MAC nhưng OVN distinguished: declarative + centrally compiled + không tự học runtime (giảm nguồn drift).

### 4.8.X.4 `eth_type` — EtherType phân loại payload Ethernet

**Khái niệm.** `eth_type` là match field 16-bit chứa EtherType — trường ngay sau dst+src MAC trong Ethernet II header (byte 12-13), xác định protocol của payload. Alias legacy: `dl_type`. Field này là **prerequisite** của hầu hết L3+ field: muốn match `nw_src` thì phải match `eth_type=0x0800` (IPv4); muốn match `arp_op` thì phải `eth_type=0x0806`; muốn match `ipv6_src` thì `eth_type=0x86dd`. Loại field: **L2 header field**, sub-loại "protocol classifier".

**Lịch sử + bối cảnh.** EtherType định nghĩa từ Ethernet II spec (1980s). OF 1.0 đưa vào struct match như 16-bit field, exact-match-only. OF 1.2 chuẩn hoá `OXM_OF_ETH_TYPE` (oxm_field=5). Field stable từ 1.0; thay đổi duy nhất là OF 1.5 (2014) thêm `packet_type` field cho non-Ethernet packet (NSH, MPLS) — khi đó `eth_type` có thể không applicable (xem 4.8.4.5 `packet_type`).

**Vị trí trong kiến trúc.** `struct flow.dl_type` 16-bit big-endian. MFF: `MFF_ETH_TYPE` (line 1139 `meta-flow.h`). Đặc biệt: classifier OVS dùng `eth_type` như **first-level dispatch** trong TSS — flow với `eth_type=0x0800` (IPv4) tách khỏi flow với `eth_type=0x86dd` (IPv6) ở subtable khác để giảm cross-product.

**Vai trò.** Vai trò chính: phân loại protocol payload để chọn L3 parsing path. Vai trò phụ: prerequisite enforcement — OVS validate khi parse rule, từ chối rule có `nw_src` mà không có `eth_type=0x0800` (error message: "field nw_src missing prerequisite eth_type=0x0800"). Ngoài ra `eth_type` là input của shorthand alias: `ip` = `eth_type=0x0800`, `ipv6` = `eth_type=0x86dd`, `arp` = `eth_type=0x0806`.

**Vì sao sinh ra.** Mọi packet Ethernet có EtherType, switch silicon hardware vốn parse field này để dispatch hardware ACL khác nhau cho IPv4/IPv6/ARP/MPLS. OpenFlow expose ra control plane để rule có thể distinguish protocol, tránh nhầm lẫn (ví dụ rule match `nw_src=10.0.0.1` không nên áp cho ARP packet — ARP có `arp_spa` mới đúng).

**Vấn đề giải quyết.** Cụ thể: làm sao đảm bảo rule L3 chỉ áp cho đúng protocol L3. Cách giải: enforce prerequisite — rule muốn match L3 IPv4 phải có `eth_type=0x0800`; OVS classifier dùng prefix tree dispatch theo `eth_type` trước khi vào subtable L3. Phạm vi giải quyết: phân biệt 6 protocol chính (IPv4, IPv6, ARP, RARP, MPLS unicast/multicast, PBB, NSH).

**Tầm quan trọng.** Tier 2 medium, nhưng frequency 100% trong rule L3+. Engineer must-know: bảng EtherType phổ biến (0x0800 IPv4, 0x86dd IPv6, 0x0806 ARP, 0x8035 RARP, 0x8100 VLAN 802.1Q, 0x88a8 QinQ 802.1ad, 0x8847/0x8848 MPLS, 0x88e7 PBB, 0x894f NSH). Hiểu rằng VLAN-tagged packet có `eth_type=0x8100` ở header ngoài; field `eth_type` mà OVS expose chính là EtherType **bên trong** sau khi pop VLAN logic (true L3 EtherType).

**Cơ chế hoạt động.** Datapath parse Ethernet header. Nếu byte 12-13 = 0x8100 (VLAN), parse 4 byte tiếp (TPID + TCI) rồi đọc EtherType thật ở byte 16-17 vào `flow.dl_type`, đồng thời populate `vlan_tci`. Nếu QinQ (0x88a8 outer + 0x8100 inner), parse cả 2 layer. `flow.dl_type` luôn là EtherType "thật" sau khi unwrap VLAN, điều này quan trọng để rule L3 không cần biết VLAN có hay không.

**OXM TLV byte layout.** `oxm_class=0x8000`, `oxm_field=5`, `hasmask=0`, `length=2`, value 2 byte big-endian. Tổng: 6 byte. Không có masked variant trong OF spec (exact-match-only).

**Bảng EtherType phổ biến + prerequisite chain.**

| EtherType | Protocol | Field downstream cần prerequisite này |
|---|---|---|
| `0x0800` | IPv4 | `nw_src`, `nw_dst`, `nw_proto`, `nw_tos`, `nw_ttl`, `ip_frag` |
| `0x86dd` | IPv6 | `ipv6_src`, `ipv6_dst`, `ipv6_label`, `ipv6_exthdr`, `nd_target` |
| `0x0806` | ARP | `arp_op`, `arp_spa`, `arp_tpa`, `arp_sha`, `arp_tha` |
| `0x8035` | RARP | `arp_op` (legacy) |
| `0x8100` | VLAN 802.1Q | (VLAN field tự động) |
| `0x88a8` | QinQ 802.1ad | (PBB outer) |
| `0x8847` | MPLS unicast | `mpls_label`, `mpls_tc`, `mpls_bos`, `mpls_ttl` |
| `0x8848` | MPLS multicast | (same MPLS field) |
| `0x88e7` | PBB | `pbb_isid` |
| `0x894f` | NSH | `nsh_spi`, `nsh_si`, `nsh_c1..c4` |

**Ví dụ match.**

```
# Filter chỉ IPv4 traffic
priority=100,eth_type=0x0800,actions=goto_table:10
# Tương đương dùng alias
priority=100,ip,actions=goto_table:10

# ARP responder
priority=100,eth_type=0x0806,arp_tpa=10.0.0.1,actions=...
# Alias
priority=100,arp,arp_tpa=10.0.0.1,actions=...

# IPv6
priority=100,eth_type=0x86dd,ipv6_src=2001:db8::/32,actions=...
priority=100,ipv6,ipv6_src=2001:db8::/32,actions=...   # alias

# Drop MPLS không expected
priority=200,eth_type=0x8847,actions=drop
```

**Cách kỹ sư vận hành thành thạo.** (1) **Đọc**: nhận diện alias (`ip`, `ipv6`, `arp`, `mpls`, `tcp`, `udp`) — đều implicit set `eth_type` đúng. (2) **Sửa**: ưu tiên alias trong rule manual cho readability; kiểm tra prerequisite khi viết flow custom. (3) **Quan sát**: `ovs-ofctl dump-flows br-int | head -50` đếm distribution flow theo `eth_type` xem traffic mix. (4) **Debug**: error "field X missing prerequisite eth_type=Y" → thêm `eth_type=Y` hoặc dùng alias.

**Source code.** Function `mf_check_src_prereqs` (`lib/meta-flow.c`) validate prerequisite chain. Function `parse_ofp_str` (`lib/ofp-parse.c`) handle alias. Branch baseline: OVS v3.3.0.

**Kịch bản bẻ gãy.** Quên `eth_type=0x0800` khi match `nw_src` → OVS reject rule với error "field nw_src missing prerequisite". Fix: thêm prerequisite explicit hoặc dùng alias `ip,nw_src=10.0.0.1`.

**Bài tập synthetic.** Trên OVS bridge có 3 flow: `priority=100,ip,actions=resubmit(,10)`, `priority=100,arp,actions=resubmit(,20)`, `priority=100,ipv6,actions=resubmit(,30)`. Gửi 1 ARP packet từ Mininet host → counter flow 2 tăng. Gửi 1 ICMPv4 ping → counter flow 1 tăng. Verify dispatch theo `eth_type` đúng.

**So sánh với hệ khác.** Linux iptables `-p` flag (proto) implicit dùng EtherType + IP proto kết hợp. Cisco ACL `permit ip` vs `permit icmp` vs `permit tcp` tương đương. NSX-T DFW rule có `service` field (predefined HTTP, HTTPS) abstract trên EtherType + L4 port. P4 dùng `parser` state machine để parse EtherType và branch sang state IPv4/IPv6 — đây là kế thừa concept của OpenFlow.

### 4.8.X.5 `vlan_vid` — VLAN ID 12-bit của tag 802.1Q

**Khái niệm.** `vlan_vid` là match field 12-bit chứa VLAN ID (giá trị 0-4095, trong đó 0 và 4095 reserved) của tag 802.1Q gắn vào frame Ethernet. Field này extract từ TCI (Tag Control Information) 16-bit của tag 802.1Q, bỏ qua 3 bit PCP và 1 bit DEI/CFI. Field có ý nghĩa đặc biệt: bit 12 của TCI (= bit "VLAN Present" trong OF 1.1+ OXM) báo có VLAN tag hay không, dẫn đến giá trị reserved `0x1000` để match "any VLAN tag present" và `0x0000` để match "no tag". Loại field: **L2 header field**, sub-loại "VLAN identifier".

**Lịch sử + bối cảnh.** VLAN 802.1Q chuẩn hoá 1998. OF 1.0 (2009) định nghĩa `dl_vlan` 16-bit (toàn bộ TCI) với reserved value `OFP_VLAN_NONE=0xffff` cho untagged. OF 1.1 (2011) tách thành `vlan_vid` 12-bit + `vlan_pcp` 3-bit, encoding bit 12 của OXM value làm "Present" flag (Pratt-style encoding để giữ exact-match semantic). OF 1.2 OXM chuẩn `OXM_OF_VLAN_VID` (oxm_field=6, hasmask supported) và `OXM_OF_VLAN_PCP` (oxm_field=7).

**Vị trí trong kiến trúc.** `struct flow.vlan_tci` 16-bit toàn TCI (OVS giữ format classic). Khi serialize ra OXM, OVS extract 12-bit + present flag. MFF: `MFF_VLAN_TCI` (line 1169) cho legacy, `MFF_VLAN_VID` (line 1203) cho OXM.

**Vai trò.** Vai trò chính: phân biệt VLAN traffic trên trunk port, dispatch sang policy/forwarding khác nhau. Vai trò trong OVN: trên tap LSP có `tag` set, ovn-controller cài flow ở table 0 strip VLAN tag (`strip_vlan`) trước khi đi vào pipeline OVN logic; ở chiều ra trước khi gửi qua tap, push VLAN tag (`mod_vlan_vid:N`). Trong OVS standalone trunk port: rule dispatch `vlan_vid=100` → tenant 1 pipeline, `vlan_vid=200` → tenant 2 pipeline.

**Vì sao sinh ra.** VLAN 802.1Q là cách phổ biến nhất để segment broadcast domain trên Ethernet trước khi có overlay (VXLAN/Geneve). OF phải hỗ trợ match `vlan_vid` để integrate với deployment legacy DC dùng VLAN trunk. OF 1.0 design `dl_vlan=0xffff` cho untagged không sạch (mix giá trị data với metadata flag); OF 1.1 thiết kế lại với "Present flag" rõ ràng hơn.

**Vấn đề giải quyết.** Cụ thể: làm sao distinguish "packet không có VLAN tag" vs "packet có VLAN tag với VID=0" vs "packet có VLAN tag với VID=N". Cách giải OF 1.1+ OXM: encoding 13-bit (12-bit VID + 1-bit present). `vlan_vid=0x0000` = no tag. `vlan_vid=0x1000` (mask `0x1000`) = any tag present. `vlan_vid=N` (N từ 0x0001 đến 0x0fff) = tag present với VID=N (bit 12 implicit = 1).

**Tầm quan trọng.** Tier 2 medium. Frequency cao trong DC dùng trunk port (OpenStack provider VLAN, baremetal trunk). Trong cloud-native overlay-only (VXLAN, Geneve), `vlan_vid` ít dùng hơn. Engineer must-know: encoding "Present flag" semantics, phân biệt mask `0x1000` (any tag) vs `0x0fff` (any VID), action `strip_vlan` vs `pop_vlan` vs `mod_vlan_vid`.

**Cơ chế hoạt động.** Datapath parse Ethernet header. Nếu byte 12-13 = 0x8100 (TPID), parse TCI 16-bit ở byte 14-15, gắn vào `flow.vlan_tci` (giữ định dạng raw TCI). Khi serialize ra OXM, OVS encode: nếu TCI có bit CFI=1 (= present), set bit 12 của OXM value và copy 12 bit VID; nếu untagged (no TCI), value = 0x0000.

**OXM TLV byte layout (`vlan_vid`).** `oxm_class=0x8000`, `oxm_field=6`, `hasmask=0`, `length=2`, value 2 byte với encoding: bit 12 = present, bit 0-11 = VID. Masked variant: `hasmask=1`, `length=4`, value 2 byte + mask 2 byte.

**Bảng giá trị đặc biệt + ý nghĩa.**

| Match expression | OXM value | OXM mask | Ý nghĩa |
|---|---|---|---|
| `vlan_vid=0` | `0x0000` | (none) | Không có tag (untagged frame) |
| `vlan_vid=0x1000/0x1000` | `0x1000` | `0x1000` | Có tag, bất kể VID |
| `vlan_vid=100` | `0x1064` | (none) | Có tag với VID=100 (0x064 + bit 12) |
| `vlan_vid=100/0x0fff` | `0x0064` | `0x0fff` | VID=100, không quan tâm có tag hay không |
| `vlan_vid=0x0000/0x1000` | `0x0000` | `0x1000` | Không có tag (alternative spelling) |

**Ví dụ match.**

```
# Trunk port dispatch theo VLAN
priority=100,in_port=1,vlan_vid=100,actions=strip_vlan,resubmit(,10)
priority=100,in_port=1,vlan_vid=200,actions=strip_vlan,resubmit(,20)
priority=100,in_port=1,vlan_vid=0,actions=drop                   # untagged không cho phép

# Match any VLAN tag
priority=50,vlan_vid=0x1000/0x1000,actions=resubmit(,trunk_table)

# Modify VLAN ID (translation)
priority=100,vlan_vid=100,actions=mod_vlan_vid:200,output:2

# Push new VLAN tag (legacy alias dl_vlan)
priority=100,in_port=2,actions=push_vlan:0x8100,mod_vlan_vid:300,output:1

# Strip VLAN tag (untag)
priority=100,vlan_vid=100,actions=strip_vlan,output:eth0
```

**Cách kỹ sư vận hành thành thạo.** (1) **Đọc**: phân biệt `vlan_vid=N` vs `vlan_vid=N/0x0fff` vs `vlan_vid=0x1000/0x1000` — semantic khác nhau hoàn toàn. (2) **Sửa**: dùng action set: `strip_vlan` xoá tag, `push_vlan:0x8100` thêm tag header, `mod_vlan_vid:N` đổi VID, `mod_vlan_pcp:P` đổi PCP. (3) **Quan sát**: `ovs-ofctl dump-flows br-int | grep vlan_vid` đếm flow trunk; `tcpdump -e -i eth0` xem TCI raw. (4) **Debug**: nếu VLAN translation sai, ovn-trace với input packet có VLAN, trace từng action.

**Source code.** Function anchor `flow_set_dl_vlan` (`lib/flow.c`) write VLAN logic. Function `format_be16_masked` (`lib/match.c`) format VID/PCP với present flag handling. Branch baseline: OVS v3.3.0.

**Kịch bản bẻ gãy.** (a) Viết `vlan_vid=100` mong muốn match "VID=100 bất kể có tag hay không" — sai, vì syntax này yêu cầu present=1 (bit 12 set). Phải dùng `vlan_vid=100/0x0fff`. (b) `mod_vlan_vid:100` áp lên packet untagged → no-op silent (action chỉ sửa tag hiện có, không push). Phải `push_vlan:0x8100,mod_vlan_vid:100` mới tạo tag mới.

**Bài tập synthetic.** Mininet trunk topology: bridge `s1` với 1 host h1 trên trunk port (untagged) và 2 host h2/h3 trên access port VLAN 100 và 200. Cài flow trên s1: `in_port=1(h1),vlan_vid=100,actions=strip_vlan,output:2(h2)` và đối xứng cho VLAN 200. Verify h1↔h2 ping work khi h1 gửi tagged VLAN 100; verify h1↔h3 work với VLAN 200; verify h2↔h3 không communicate (segment).

**So sánh với hệ khác.** Linux `vconfig add eth0 100` tạo subinterface `eth0.100` xử lý VLAN 100, semantically tương đương rule `vlan_vid=100,strip_vlan`. Cisco IOS `switchport access vlan 100` trên port = strip + tag tự động ở hardware. NSX-T sử dụng overlay Geneve thay VLAN trunk, nhưng có "VLAN Logical Switch" type cho compatibility. Pre-overlay era, mọi DC đều dựa vào VLAN trunk; OVN giảm phụ thuộc nhưng vẫn hỗ trợ VLAN cho provider network.

### 4.8.X.6 `ip_proto` — số hiệu giao thức L4 trong header IPv4/IPv6

**Khái niệm.** `ip_proto` là match field 8-bit chứa giá trị "Protocol" byte 9 của IPv4 header (RFC 791) hoặc "Next Header" byte 6 của IPv6 header (RFC 8200), xác định giao thức của payload (TCP=6, UDP=17, ICMP=1, SCTP=132, OSPF=89, GRE=47, ESP=50, AH=51, ICMPv6=58, v.v.). Alias legacy: `nw_proto`. Field này là **prerequisite** của hầu hết L4 field: muốn match `tp_src/tp_dst` (TCP/UDP), `tcp_flags`, `icmp_type/icmp_code`, hoặc `sctp_src/dst` thì phải khớp `ip_proto` với value tương ứng. Loại field: **L3 header field**, sub-loại "L4 protocol classifier".

**Lịch sử + bối cảnh.** OF 1.0 (2009) đưa vào struct match dưới tên `nw_proto`, exact-match-only. OF 1.2 (2011) chuẩn hoá `OXM_OF_IP_PROTO` (oxm_class=0x8000, oxm_field=10) cùng với việc đưa OXM TLV làm cơ chế match field mặc định, đồng thời cho phép field này áp dụng được cho cả IPv6 (trước đó OF 1.0 chỉ định nghĩa cho IPv4). Stable từ OF 1.2 đến nay; không thay đổi width hay encoding. Về OVS upstream, NXM_OF_IP_PROTO mã 6 đã có từ OVS v1.1; OXM tương đương bổ sung từ v1.7 (2014).

**Vị trí trong kiến trúc.** `struct flow.nw_proto` 8-bit (`include/openvswitch/flow.h`). MFF: `MFF_IP_PROTO` định nghĩa trong `include/openvswitch/meta-flow.h` (type u8, prereqs IP). OVS classifier dùng field này cùng với `eth_type` làm hai-tầng dispatch: subtable phân nhánh đầu theo `eth_type` (IPv4 vs IPv6 vs ARP), tầng tiếp theo theo `ip_proto` (TCP vs UDP vs ICMP), giảm cross-product trong TSS classifier.

**Vai trò.** Vai trò chính: phân loại giao thức L4 để chọn parser path. Vai trò phụ: prerequisite enforcement — OVS từ chối rule có `tp_dst=80` mà thiếu `ip_proto=6` với error "field tp_dst missing prerequisite ip_proto=6". Vai trò shorthand: alias `tcp` = `eth_type=0x0800,ip_proto=6`; `udp` = `eth_type=0x0800,ip_proto=17`; `icmp` = `eth_type=0x0800,ip_proto=1`; `tcp6/udp6/icmp6` tương đương cho IPv6.

**Vì sao sinh ra.** Mọi packet IP đều có byte protocol để báo payload loại gì, hardware silicon từ lâu đã parse byte này để dispatch. OpenFlow cần expose ra control plane để rule có thể distinguish protocol — không có `ip_proto`, mọi rule sẽ phải mò vào L4 header byte mức thấp, cực kỳ dễ nhầm (ví dụ TCP và UDP có cấu trúc port khác nhau, parse sai sẽ corrupt). OF spec 1.0 chỉ có cho IPv4; mở rộng sang IPv6 trong OF 1.2 là phản ứng tự nhiên khi IPv6 deployment tăng.

**Vấn đề giải quyết.** Cụ thể: làm sao đảm bảo rule L4 chỉ áp cho đúng protocol L4, đồng thời tránh parse field không tồn tại (ví dụ ICMP không có port, parse `tp_src` trên ICMP là junk). Cách giải: enforce prerequisite chain — `tp_src/dst` chỉ valid khi `ip_proto=6 hoặc 17`; `tcp_flags` chỉ valid khi `ip_proto=6`; `icmp_type/code` chỉ valid khi `ip_proto=1` (IPv4) hoặc `ip_proto=58` (ICMPv6). Phạm vi giải quyết: tất cả L4 protocol lớn (TCP, UDP, SCTP, ICMP, ICMPv6) cộng với một số L4 niche (GRE, ESP, AH, OSPF) match được nhưng không có L4 sub-field riêng.

**Tầm quan trọng.** Tier 2 medium, frequency rất cao (xuất hiện trong gần 100% rule L4). Engineer must-know: bảng số hiệu IANA Protocol phổ biến (1=ICMP, 2=IGMP, 6=TCP, 17=UDP, 41=IPv6-in-IPv4, 47=GRE, 50=ESP, 51=AH, 58=ICMPv6, 89=OSPF, 132=SCTP), khả năng reuse `ip_proto` cho cả IPv4 và IPv6 (tiết kiệm pipeline complexity), prerequisite chain tới L4 sub-field.

**Cơ chế hoạt động.** Datapath parse Ethernet header → đọc `eth_type`. Nếu `eth_type=0x0800` (IPv4), parse IPv4 header byte 9 vào `flow.nw_proto`. Nếu `eth_type=0x86dd` (IPv6), parse Next Header chain: Next Header byte ban đầu của IPv6 header; nếu đó là extension header (Hop-by-Hop=0, Routing=43, Fragment=44, AH=51, ESP=50, Destination=60), lần lượt iterate qua từng extension header đến khi tìm được "real" upper-layer protocol (TCP, UDP, ICMPv6, ...) — giá trị cuối cùng đó được populate vào `flow.nw_proto`, đồng thời `flow.ipv6_exthdr` bitmap ghi chain extension header đã đi qua.

**OXM TLV byte layout.** `oxm_class=0x8000`, `oxm_field=10`, `hasmask=0`, `length=1`, value 1 byte. Tổng 5 byte. Spec không cho hasmask=1 (không có ý nghĩa khi mask 8-bit IANA assignment).

**Ví dụ match.**

```
# Filter chỉ TCP
priority=100,ip,ip_proto=6,actions=goto_table:10
# Tương đương dùng alias
priority=100,tcp,actions=goto_table:10

# UDP DNS
priority=100,ip,ip_proto=17,udp_dst=53,actions=output:dns_resolver
priority=100,udp,udp_dst=53,actions=output:dns_resolver  # alias

# ICMP echo cho ping monitor
priority=100,ip,ip_proto=1,icmp_type=8,actions=output:monitor

# Drop non-standard (không phải TCP/UDP/ICMP)
priority=10,ip,ip_proto=47,actions=drop  # block GRE
priority=10,ip,ip_proto=50,actions=drop  # block ESP

# OSPF cho router cluster
priority=100,ip,ip_proto=89,actions=output:ospf_neigh
```

**Cách kỹ sư vận hành thành thạo.** (1) **Đọc**: nhớ bảng số hiệu IANA cốt lõi (TCP/UDP/ICMP/SCTP/OSPF/GRE/ESP/AH); biết rằng alias `tcp/udp/icmp` ngầm bao gồm `eth_type` đúng. (2) **Sửa**: ưu tiên alias trong rule manual; với rule generated programmatically, dùng explicit `ip_proto=N` để không lệ thuộc parser. (3) **Quan sát**: `ovs-ofctl dump-flows br-int | awk '/ip_proto/{print}'` đếm distribution; `tcpdump -nn ip proto 6` cross-check. (4) **Debug**: error "field X missing prerequisite ip_proto=Y" → thêm explicit; với IPv6 nếu ip_proto=58 không match được ICMPv6 mong đợi, kiểm tra xem packet có đi qua extension header chain không (Routing/Fragment có thể đổi chain, nhưng `flow.nw_proto` vẫn là final upper-layer protocol).

**Source code.** Function `flow_extract` (`lib/flow.c`) parse `nw_proto` từ raw packet; với IPv6 chain extension header xem `parse_ipv6_ext_hdrs` cùng file. Validation prerequisite trong `mf_check_src_prereqs` (`lib/meta-flow.c`). Branch baseline OVS v3.3.0 / v2.17.9; line cụ thể có thể drift giữa version, dùng function name làm anchor (Rule 14 Option C).

**Khi xảy ra sự cố.** Triệu chứng A: Rule L4 không match — kiểm tra packet có đúng protocol không qua `tcpdump -nn ip proto N`; nếu packet đúng nhưng rule không hit, suspect prerequisite chain mismatch. Triệu chứng B: IPv6 packet có extension header (như Routing) không match rule `tcp` → confirm `flow.nw_proto` qua `ofproto/trace`, có thể fragment header đứng trước TCP làm parser dừng nếu packet bị truncate. Triệu chứng C: rule áp `ip,ip_proto=58` (ICMPv6 với eth_type=0x0800) → impossible match (ICMPv6 yêu cầu eth_type=0x86dd); OVS không reject lỗi syntax nhưng rule sẽ luôn miss.

**Liên quan mật thiết tới.** `eth_type` (prerequisite trực tiếp), `tp_src/tp_dst/tcp_flags/icmp_type/icmp_code/sctp_src/dst` (downstream cần `ip_proto` đúng), `ct_nw_proto` (conntrack original tuple, song song với `ip_proto`), `ipv6_exthdr` (bitmap chain extension header IPv6 ảnh hưởng tới final `nw_proto`).

**Khác biệt giữa các phiên bản.** OF 1.0 chỉ áp dụng cho IPv4 (`nw_proto`). OF 1.2 mở rộng cho IPv6 + chuẩn hoá tên `OXM_OF_IP_PROTO`. OVS giữ alias `nw_proto` đến nay cho backward compat. OVS 2.4 (2015) cải thiện parser IPv6 extension header để chain dài (10+ extension) không bị bỏ qua. Curriculum baseline: OVS v2.17.9 + OVN v22.03.8.

**Cách quan sát + xác minh.**

```
$ ovs-appctl ofproto/trace br-int 'in_port=1,eth_type=0x0800,ip_proto=6,...'
Final flow: ...,nw_proto=6,...
$ ovs-ofctl dump-flows br-int | grep ip_proto
 cookie=0x0, n_packets=42, ip,ip_proto=6,tp_dst=80,actions=...
$ tcpdump -nn -i any 'ip and not tcp and not udp and not icmp'   # bắt protocol bất thường
```

**Trường hợp sự cố thực tế.** Cluster OVN 22.03 deploy multi-AZ, observability dashboard báo IPv6 traffic giảm 20% sau upgrade kernel host. Investigation: ovn-trace `inport==..,ip6,ip_proto=6,...` cho thấy rule `tcp6` không match một số packet. Root cause: kernel mới đổi default size IPv6 fragment header threshold, một subset packet TCP6 giờ có Fragment extension header trong chain. Old `flow_extract` của OVS bản cũ parser dừng ở Fragment, không vào tới TCP, `nw_proto=44` (Fragment) thay vì 6. Fix: upgrade OVS lên bản có parser chain extension header ổn định + tune fragmentation MTU host.

**Bài tập synthetic.** Mininet bridge `s1`, 2 host h1/h2. Cài 4 flow: `priority=100,tcp,actions=output:2`, `priority=100,udp,actions=output:2`, `priority=100,icmp,actions=output:2`, `priority=10,ip,actions=drop`. Verify `h1 ping h2` (ICMP) hoạt động, `iperf -u` (UDP) hoạt động, gửi gói GRE custom (`scapy`) bị drop. Đếm counter mỗi flow, xác nhận dispatch `ip_proto` chính xác.

**Lỗi thường gặp + tín hiệu chẩn đoán.** (a) Nhầm số hiệu (`ip_proto=4` thay vì 6) — log error "no match"; check IANA. (b) Quên `eth_type` khi dùng `ip_proto` (`ip_proto=6` đứng một mình) — OVS reject. Fix: dùng `ip,ip_proto=6` hoặc alias `tcp`. (c) Áp `ip_proto=58` (ICMPv6) cho IPv4 — silent miss; phải dùng `ipv6` alias.

**So sánh với hệ khác.** Linux iptables `-p tcp/udp/icmp/...` dùng cùng IANA protocol number. Cisco ACL `permit tcp/udp/icmp/89` đặt protocol làm field thứ hai sau `permit/deny`. NSX-T Distributed Firewall có `service` field abstract (HTTP, HTTPS, ICMP) ánh xạ ngầm tới ip_proto + L4 port. P4 dùng `parser` state machine: state IPv4 sau khi parse `protocol` byte sẽ `transition select(hdr.ipv4.protocol)` rẽ sang state TCP/UDP/ICMP — kế thừa concept từ OpenFlow.

### 4.8.X.7 `nw_src` — địa chỉ IPv4 nguồn 32-bit

**Khái niệm.** `nw_src` (alias chính thức `ipv4_src`, alias legacy `ip_src`) là match field 32-bit chứa địa chỉ IPv4 nguồn — 4 byte cuối cùng của IPv4 header (byte 12-15 từ đầu IP header, RFC 791). Format match: dotted decimal `10.0.0.5` (exact match) hoặc CIDR `10.0.0.0/24` (prefix match) hoặc bitwise mask tuỳ ý `10.0.0.0/0xffff0000` (arbitrary mask). Field này là một trong những field được dùng nhiều nhất trong toàn bộ catalog OpenFlow, là spine của ACL theo nguồn, NAT identification, anti-spoofing, geo-fencing. Loại field: **L3 header field**, sub-loại "endpoint identifier".

**Lịch sử + bối cảnh.** OF 1.0 (2009) định nghĩa `nw_src` 32-bit, nhưng chỉ hỗ trợ CIDR mask (prefix length). OVS 1.8 (2013) mở rộng cho phép arbitrary bitwise mask (non-prefix) — đây là feature đắc lực cho rule subnet phi-contiguous. OF 1.2 (2011) chuẩn hoá `OXM_OF_IPV4_SRC` (oxm_class=0x8000, oxm_field=11, hasmask supported). NXM_OF_IP_SRC (mã 7) tồn tại từ OVS v1.1 cho backward compat. Stable từ OF 1.2 đến nay.

**Vị trí trong kiến trúc.** `struct flow.nw_src` 32-bit big-endian (`be32`). MFF: `MFF_IPV4_SRC` (`include/openvswitch/meta-flow.h`, type be32, prefix lookup member `nw_src`). OVS classifier dùng field này như **prefix tree key**: trie structure cho rule prefix-based (CIDR) cho phép longest-prefix-match O(log n) thay vì O(n) linear scan; arbitrary bitwise mask không vào trie mà vào subtable riêng của TSS.

**Vai trò.** Vai trò chính: identify endpoint nguồn cho mọi forwarding/filtering decision dựa trên IPv4. Vai trò phụ: anti-spoofing (rule `priority=200,in_port=N,ip,nw_src=10.0.N.0/24,actions=resubmit` chỉ cho phép subnet đúng vào port đó); NAT identification (rule conntrack `ct_nw_src=<original_src>,ct_state=+est+rpl` để match reply traffic về VIP); geo-fencing dùng prefix list theo region. Trong OVN: pipeline LR_IN_IP_ROUTING dùng `ip4.dst` (compile xuống `nw_dst`) cho route lookup; Logical_Switch ACL dùng `ip4.src` (compile xuống `nw_src`).

**Vì sao sinh ra.** Endpoint identification là yêu cầu tối thiểu của bất kỳ forwarding rule nào trong mạng L3. OF 1.0 không thể không có IPv4 src/dst — đây là pillar cốt lõi. Lý do giới hạn ban đầu chỉ CIDR: hardware silicon ASIC truyền thống dùng TCAM để match prefix, mở rộng arbitrary mask làm phình entry count; OVS userspace không có giới hạn TCAM nên OVS 1.8 mở khoá được.

**Vấn đề giải quyết.** Cụ thể: làm sao expose IPv4 source address cho rule policy. Cách giải: 32-bit field with mask. Phạm vi giải quyết: tất cả use case CIDR-based ACL (subnet → policy), arbitrary mask (non-contiguous mask cho deployment đặc biệt như multihomed dual-prefix). Không giải quyết: stateful flow tracking (cần kết hợp `ct_state`, `ct_nw_src`); IPv6 (cần `ipv6_src` 128-bit riêng).

**Tầm quan trọng.** Tier 2 medium-high, frequency cực cao (xuất hiện trong gần như mọi rule L3). Engineer must-know: cú pháp CIDR (`10.0.0.0/24`), arbitrary mask (`10.0.0.0/0xffff0000`), exact match (`10.0.0.5`), tích hợp với prefix lookup classifier (trie), khác biệt với `ct_nw_src` (original tuple trước NAT, không phải hiện tại).

**Cơ chế hoạt động.** Datapath parse IPv4 header byte 12-15 vào `flow.nw_src`. Classifier lookup: TSS subtable phân nhánh; nếu rule chỉ dùng prefix mask, classifier chuyển sang trie path lookup (`classifier_lookup`, `lib/classifier.c`). Trie maintain như radix tree; mỗi insertion update both trie và subtable. Lookup: traverse trie bit-by-bit từ MSB của IPv4 address, return longest-matching prefix.

**OXM TLV byte layout.** Exact: `oxm_class=0x8000`, `oxm_field=11`, `hasmask=0`, `length=4`, value 4 byte big-endian. Tổng 8 byte. Masked: `hasmask=1`, `length=8`, value 4 byte + mask 4 byte. Tổng 12 byte.

**Ví dụ match.**

```
# Subnet ACL
priority=100,ip,nw_src=10.0.1.0/24,actions=resubmit(,allow_table)
priority=100,ip,nw_src=192.168.0.0/16,actions=drop      # block private RFC1918 trên uplink

# Host exact
priority=200,ip,nw_src=10.0.1.5,actions=output:monitor

# Anti-spoofing
priority=300,in_port=1,ip,nw_src=10.0.1.0/24,actions=resubmit(,L3_table)
priority=10,in_port=1,ip,actions=drop                   # source ngoài subnet → drop

# Arbitrary mask (non-contiguous, multihomed)
priority=100,ip,nw_src=10.0.0.0/0xff00ff00,actions=...

# Modify (NAT)
priority=100,ip,nw_src=192.168.1.5,actions=mod_nw_src:10.0.0.5,output:uplink
```

**Cách kỹ sư vận hành thành thạo.** (1) **Đọc**: phân biệt CIDR vs arbitrary mask vs exact. (2) **Sửa**: dùng `mod_nw_src` cho SNAT static, hoặc `ct(nat(src=...))` cho stateful NAT (xem 9.24). (3) **Quan sát**: `ovs-ofctl dump-flows br-int | grep nw_src` đếm distribution; `tcpdump -nn 'ip src 10.0.1.5'` cross-check. (4) **Debug**: nếu rule không match, kiểm tra prefix length đúng; kiểm tra packet thật có IP src đúng (`tcpdump`); nếu nghi NAT đã đổi src, dùng `ct_nw_src` để match original.

**Source code.** Function `flow_extract` (`lib/flow.c`) parse src byte 12-15 vào `flow.nw_src`. Function `classifier_lookup` (`lib/classifier.c`) thực hiện lookup; trie path trong `trie_lookup`. Function `mf_set_value` cho `MFF_IPV4_SRC` populate match struct. Branch baseline OVS v3.3.0 / v2.17.9.

**Khi xảy ra sự cố.** Triệu chứng A: ACL không hit cho subnet — check prefix length, check rule có overlap priority cao hơn không. Triệu chứng B: traffic NAT-ed nhưng rule match `nw_src` original không hit downstream — đúng vì NAT đã đổi src trước khi match; phải dùng `ct_nw_src` để match original. Triệu chứng C: arbitrary mask rule không hit — có thể nhầm endian; OVS expect mask network-byte-order, double-check với `ovs-ofctl dump-flows --no-stats`.

**Liên quan mật thiết tới.** `eth_type` (prerequisite, =0x0800), `nw_dst` (đối ứng L3 destination), `ct_nw_src` (original tuple conntrack), `mod_nw_src` action (SNAT static), `nat()` action trong ct() context (stateful NAT, xem 9.24), `arp_spa` (ARP source protocol address — không thay được `nw_src` trong ARP packet).

**Khác biệt giữa các phiên bản.** OF 1.0 chỉ CIDR mask. OVS 1.8 (2013) mở arbitrary bitwise mask. OF 1.2 chuẩn hoá OXM với hasmask=1 hỗ trợ. OVS 2.4 (2015) optimize prefix trie cho rule count cao. Curriculum baseline: OVS v2.17.9 + OVN v22.03.8.

**Cách quan sát + xác minh.**

```
$ ovs-appctl ofproto/trace br-int 'in_port=1,ip,nw_src=10.0.1.5,nw_dst=10.0.2.10,...'
Final flow: ...,nw_src=10.0.1.5,...
$ ovs-ofctl dump-flows br-int | grep 'nw_src=10.0'
 cookie=0x0, n_packets=128, ip,nw_src=10.0.1.0/24,actions=resubmit(,allow_table)
```

**Trường hợp sự cố thực tế.** OpenStack tenant network OVN deploy production, một số VM bị "ngẫu nhiên" drop traffic outbound. Investigation: ovn-trace cho thấy ACL deny match `ip4.src` với prefix khác mong đợi. Root cause: 2 ACL rule cùng priority với prefix overlap (`10.0.1.0/24` allow vs `10.0.0.0/16` deny), tie-breaking không deterministic ở edge case. Fix: tăng priority rule allow (priority=300 vs 200), audit toàn bộ ACL có overlap. Tooling: `ovn-nbctl acl-list ls1 | sort -k priority -n` để phát hiện overlap.

**Bài tập synthetic.** Mininet bridge `s1` với 4 host h1-h4 trong subnet `10.0.0.0/24`. Cài 3 flow: `priority=200,ip,nw_src=10.0.0.1,actions=output:monitor` (host h1 priority cao), `priority=100,ip,nw_src=10.0.0.0/24,actions=output:gateway`, `priority=10,ip,actions=drop`. Verify h1 traffic đi monitor, h2/h3/h4 đi gateway, traffic ngoài subnet drop. Thay arbitrary mask `10.0.0.0/0xfffffffc` (chỉ match h1-h4 cụ thể) xem behavior.

**Lỗi thường gặp + tín hiệu chẩn đoán.** (a) Nhầm prefix length (`/24` vs `/16`) — rule match quá rộng/hẹp. (b) Quên `ip` alias hoặc `eth_type=0x0800` — OVS reject. (c) Áp `nw_src` cho IPv6 packet — silent miss; dùng `ipv6_src` cho IPv6.

**So sánh với hệ khác.** Linux iptables `-s 10.0.0.0/24` cùng cú pháp. Cisco ACL `permit ip 10.0.0.0 0.0.0.255 any` dùng wildcard mask (inverted). NSX-T DFW source field hỗ trợ object group reference + dynamic membership (vCenter tag). P4 `transition select(hdr.ipv4.srcAddr)` lookup. OVS unique ở chỗ arbitrary bitwise mask cho non-contiguous address space, ít vendor khác hỗ trợ.

### 4.8.X.8 `nw_dst` — địa chỉ IPv4 đích 32-bit

**Khái niệm.** `nw_dst` (alias `ipv4_dst`, legacy `ip_dst`) là match field 32-bit chứa địa chỉ IPv4 đích — byte 16-19 của IPv4 header (RFC 791). Format và masking giống hệt `nw_src` (CIDR, exact, arbitrary bitwise mask). Field này là **input chính của routing decision**: longest-prefix-match trên `nw_dst` → next-hop output port. Cũng là spine của ACL theo destination, Load Balancer VIP match, DNAT identification. Loại field: **L3 header field**, sub-loại "endpoint identifier (destination)".

**Lịch sử + bối cảnh.** Đối ứng hoàn toàn với `nw_src`: OF 1.0 (2009) đưa vào CIDR-only, OVS 1.8 mở arbitrary mask, OF 1.2 chuẩn hoá `OXM_OF_IPV4_DST` (oxm_field=12, hasmask supported). NXM_OF_IP_DST (mã 8) từ OVS v1.1. Stable từ OF 1.2.

**Vị trí trong kiến trúc.** `struct flow.nw_dst` 32-bit big-endian. MFF: `MFF_IPV4_DST` (`meta-flow.h`, type be32, prefix lookup member `nw_dst`). Classifier dùng cùng trie structure như `nw_src`, nhưng cây trie riêng biệt cho dst. Trong OVN pipeline, `nw_dst` là input của LR_IN_IP_ROUTING stage (route table lookup); trong Load Balancer pipeline, `nw_dst` match VIP để rewrite sang backend (DNAT).

**Vai trò.** Vai trò chính: input của routing table lookup (next-hop selection). Vai trò phụ: ACL by destination (rule `priority=100,ip,nw_dst=10.0.2.0/24,actions=drop`); LB VIP match (`priority=100,ip,nw_dst=10.0.0.100,actions=ct(commit,nat(dst=10.0.1.X)),output:backend`); DNAT identification cho reply path; multicast group match (`nw_dst=224.0.0.0/4` cho multicast IGMP).

**Vì sao sinh ra.** Routing decision trên packet IP về cốt lõi là "lookup destination → chọn output". Không có `nw_dst`, không có L3 forwarding. Cùng lý do với `nw_src` về CIDR-only ban đầu (TCAM hardware constraint), cùng con đường mở rộng sang arbitrary mask trong OVS userspace.

**Vấn đề giải quyết.** Cụ thể: làm sao expose IPv4 destination cho routing/ACL/NAT logic. Cách giải: 32-bit field with mask + trie-backed classifier. Phạm vi: route lookup (longest-prefix-match), exact match cho host route, prefix match cho subnet route, arbitrary mask cho deployment phi-thường. Kết hợp với `ct(nat(dst=...))` cho stateful DNAT.

**Tầm quan trọng.** Tier 2 medium-high. Frequency cực cao trong rule L3 forwarding. Engineer must-know: longest-prefix-match semantic (trong rule cùng `priority`, prefix dài hơn thắng — nhưng trong OpenFlow priority là explicit field, không có implicit longest-match unless engineer assign priority phù hợp; OVN northd compile route table với priority = prefix_length để mô phỏng longest-prefix-match).

**Cơ chế hoạt động.** Datapath parse byte 16-19 vào `flow.nw_dst`. Classifier lookup: TSS subtable; nếu rule prefix mask, classifier dispatch sang trie. Trie traverse bit-by-bit từ MSB IP dst đến leaf, return rule với longest matching prefix (trong cùng subtable).

**OXM TLV byte layout.** Identical với `nw_src`: exact 8 byte total (header 4 + value 4); masked 12 byte (header 4 + value 4 + mask 4). `oxm_field=12` thay vì 11.

**Ví dụ match.**

```
# Default route
priority=10,ip,nw_dst=0.0.0.0/0,actions=output:default_gw

# Route specific subnet (priority cao hơn cho prefix dài hơn, mô phỏng longest-prefix-match)
priority=170,ip,nw_dst=10.0.1.0/24,actions=output:gw1
priority=240,ip,nw_dst=10.0.1.5,actions=output:host1     # exact /32 thắng /24

# Load Balancer VIP match
priority=100,ip,nw_dst=10.0.0.100,tcp,tp_dst=80,
                actions=ct(commit,nat(dst=10.0.1.5:8080)),output:backend

# Drop multicast nếu không cần
priority=200,ip,nw_dst=224.0.0.0/4,actions=drop

# Egress ACL
priority=100,ip,nw_dst=192.168.0.0/16,actions=drop  # block to RFC1918 trên external link
```

**Cách kỹ sư vận hành thành thạo.** (1) **Đọc**: hiểu priority + prefix length tương quan cho longest-prefix-match (OVN northd assign priority = mask_length). (2) **Sửa**: `mod_nw_dst` cho DNAT static, `ct(commit,nat(dst=...))` cho stateful DNAT, hoặc `output` trực tiếp trong route. (3) **Quan sát**: `ovs-ofctl dump-flows br-int | grep 'nw_dst=10.0' | sort -k priority -n` xem route table trên br-int compiled từ OVN LR. (4) **Debug**: nếu route lookup sai, ovn-trace `outport` empty → xem stage LR_IN_IP_ROUTING có rule match không.

**Source code.** Function `flow_extract` parse byte 16-19. Function `route_table_add_route` (`lib/route-table.c`) build route table user-space. OVN compile: `build_lrouter_flows` (`northd/northd.c`) sinh route flow từ Logical_Router_Static_Route + Logical_Router_Port. Branch baseline OVS v3.3.0 / OVN v22.03.8.

**Khi xảy ra sự cố.** Triệu chứng A: traffic không đến đích — ovn-trace check `outport`; nếu empty, rule routing miss; check Logical_Router_Static_Route có route đúng không. Triệu chứng B: LB VIP traffic không backend rotate — `nw_dst=VIP` match nhưng `ct(commit,nat(dst=...))` chọn cùng backend mỗi lần; tune `algorithm=symmetric_l3l4` hoặc kiểm tra hash distribution. Triệu chứng C: route blackhole vì priority overlap (`/24` priority cao hơn `/32`) — sửa priority order theo prefix length.

**Liên quan mật thiết tới.** `eth_type` (prerequisite =0x0800), `nw_src` (đối ứng), `ct_nw_dst` (original dst trước DNAT), `mod_nw_dst` action, `nat()` trong ct() (stateful DNAT), Logical_Router_Static_Route trong OVN NBDB.

**Khác biệt giữa các phiên bản.** OF 1.0 CIDR only. OVS 1.8 arbitrary mask. OF 1.2 OXM masked. OVS 2.6 (2017) prefix trie tối ưu O(prefix_len) vs O(rule_count). Curriculum baseline: OVS v2.17.9 + OVN v22.03.8.

**Cách quan sát + xác minh.**

```
$ ovs-appctl ofproto/trace br-int 'in_port=1,ip,nw_src=10.0.1.5,nw_dst=10.0.2.10,...'
Datapath actions: ct(zone=...,nat),resubmit(,...) → output:N
$ ovs-ofctl dump-flows br-int | grep 'nw_dst=10.0' | sort -t= -k2
```

**Trường hợp sự cố thực tế.** OVN cluster multi-tenant deploy LB cho web service. Traffic to VIP `10.0.0.100:80` đôi khi ECMP balance không đều (1 backend nhận 80% traffic). Investigation: ovn-trace cho thấy `ct(nat)` chọn backend dựa trên 5-tuple hash; do client source port range hẹp (firewall NAT), hash collision cao → uneven distribution. Fix: tune `algorithm=symmetric_l3l4` (hash đối xứng cả forward + reply). Reference: OVN Load_Balancer.options:algorithm.

**Bài tập synthetic.** Mininet bridge `s1` với 3 host h1/h2/h3 và route table mô phỏng: `priority=240,ip,nw_dst=10.0.0.1,actions=output:1` (host route h1), `priority=170,ip,nw_dst=10.0.0.0/24,actions=output:gw`, `priority=10,ip,nw_dst=0.0.0.0/0,actions=output:default`. Verify packet đến `10.0.0.1` đi port 1 (host route thắng), `10.0.0.5` đi gw, `8.8.8.8` đi default. Thay đổi priority để test longest-prefix vs random priority.

**Lỗi thường gặp + tín hiệu chẩn đoán.** (a) Priority sai dẫn đến `/24` thắng `/32` — luôn assign priority = prefix length cho route. (b) Nhầm `nw_dst` với `nw_src` trong ACL — kiểm tra direction packet (ingress/egress). (c) Multicast `nw_dst=224.0.0.0/4` không được handle riêng → flood lan rộng; thêm rule explicit drop hoặc forward đúng group.

**So sánh với hệ khác.** Linux iptables `-d 10.0.0.0/24` cùng cú pháp. Linux routing table `ip route show` dùng longest-prefix-match implicit. Cisco IOS `ip route 10.0.0.0 255.255.255.0 next-hop` cũng longest-match. NSX-T LR routing table tương tự Linux. OVN khác biệt: route table compile xuống OF flow với priority = prefix length, làm explicit cái implicit của Linux.

### 4.8.X.9 `ip_dscp` — DSCP 6-bit cho QoS classification

**Khái niệm.** `ip_dscp` là match field 6-bit chứa Differentiated Services Code Point (DSCP) — 6 bit cao của byte ToS (IPv4) hoặc byte Traffic Class (IPv6), định nghĩa trong RFC 2474 thay thế semantics ToS RFC 791 ban đầu. Giá trị 0-63, thường dùng named code point: BE (Best Effort) = 0, AF11/AF12/AF13 = 10/12/14, AF21-AF43 (4 class × 3 drop precedence), EF (Expedited Forwarding) = 46, CS0-CS7 (Class Selector backwards-compat với IP precedence cũ). Loại field: **L3 header field**, sub-loại "QoS marking".

**Lịch sử + bối cảnh.** RFC 791 (1981) định nghĩa ToS byte với precedence (3 bit) + ToS bits (4 bit) + reserved. RFC 2474 (1998) redefine thành DSCP (6 bit) + 2 bit reserved (sau này thành ECN). OF 1.0 (2009) định nghĩa `nw_tos` 8-bit (toàn ToS byte). OF 1.2 (2011) tách thành `ip_dscp` (6-bit, OXM_OF_IP_DSCP, oxm_field=8) + `ip_ecn` (2-bit, oxm_field=9), shifted right 2 bit để fit nibble. Quan trọng: OF 1.2 changed semantics — `ip_dscp` value là 0-63 (logical), shifted-right từ ToS byte byte 1 IPv4. Pre-OF 1.2 dùng `nw_tos` raw 8-bit (`nw_tos=184` = DSCP 46 EF + ECN 00 raw byte).

**Vị trí trong kiến trúc.** `struct flow.nw_tos` 8-bit (OVS giữ raw byte format). Khi serialize ra OXM `ip_dscp`, OVS shift right 2 bit. MFF: `MFF_IP_DSCP` (`meta-flow.h`, type u8 (low 6 bits), prereqs IP, `OXM_OF_IP_DSCP(8)` since OF1.2 và OVS v1.7). Pre-OF 1.2 tương đương `MFF_IP_DSCP_SHIFTED` legacy.

**Vai trò.** Vai trò chính: phân loại traffic theo class QoS để chọn queue/scheduler. Vai trò phụ: marking → policy enforcement (rule `priority=100,ip,ip_dscp=46,actions=set_queue:1,output:N` cho EF traffic; `priority=100,ip,ip_dscp=0,actions=set_queue:0,output:N` cho BE). Trong OVN: ACL action `set_qos` ánh xạ tới `mod_nw_tos` ở OF level.

**Vì sao sinh ra.** Mạng IP cần distinguish traffic priority (voice/video/control vs bulk transfer); cần marking field ngay trong header IP để giữ semantic across hop (intra-domain QoS), không phụ thuộc 5-tuple. RFC 2474 modernize ToS byte cũ. OF 1.0 dùng raw `nw_tos` 8-bit, gây nhầm lẫn (DSCP 46 phải tính `46 << 2 = 184`); OF 1.2 fix bằng cách tách + shift để engineer chỉ cần nghĩ "DSCP=46" trực tiếp.

**Vấn đề giải quyết.** Cụ thể: làm sao expose 6-bit DSCP cleanly cho rule QoS. Cách giải OF 1.2: 6-bit field `ip_dscp` 0-63 logical. Phạm vi giải quyết: tất cả use case marking + match QoS class. Kết hợp với `set_queue` action và `Queue` table OVSDB cho egress shaping (xem 9.x QoS section nếu có).

**Tầm quan trọng.** Tier 2 medium. Frequency thấp hơn `nw_src/dst/ip_proto` nhưng critical trong deployment có voice/video traffic, multi-tenant SLA, hay tích hợp với upstream MPLS/EVPN dùng DSCP làm input cho EXP. Engineer must-know: bảng named DSCP value (BE/AF/EF/CS), shifted-right semantic OF 1.2+, khác biệt với raw `nw_tos`.

**Cơ chế hoạt động.** Datapath parse ToS byte (IPv4) hoặc Traffic Class byte (IPv6) vào `flow.nw_tos`. Classifier so sánh: nếu match expression dùng `ip_dscp=46`, OVS internally compare `(flow.nw_tos >> 2) == 46` (== 0x2e == 46). Nếu dùng raw `nw_tos=184`, compare trực tiếp. Action `mod_nw_tos:N` set toàn 8-bit (ghi đè cả ECN), thận trọng. Action `set_field:46->ip_dscp` shift left 2 bit và OR vào ToS byte, giữ ECN.

**OXM TLV byte layout.** `oxm_class=0x8000`, `oxm_field=8`, `hasmask=0`, `length=1`, value 1 byte (low 6 bit có ý nghĩa, 2 bit cao zero). Tổng 5 byte. Spec không cho hasmask=1 (không có ý nghĩa — DSCP code point là enum).

**Bảng DSCP code point phổ biến.**

| Tên | Decimal | Binary (6-bit) | ToS byte raw (DSCP<<2) | Use case |
|-----|---------|----------------|------------------------|----------|
| BE (CS0) | 0 | `000000` | 0 | Best Effort (default) |
| CS1 | 8 | `001000` | 32 | Scavenger (low priority bulk) |
| AF11 | 10 | `001010` | 40 | Class 1 low drop |
| AF21 | 18 | `010010` | 72 | Class 2 low drop (multimedia conferencing) |
| AF31 | 26 | `011010` | 104 | Class 3 low drop (multimedia streaming) |
| AF41 | 34 | `100010` | 136 | Class 4 low drop (real-time interactive) |
| CS5 | 40 | `101000` | 160 | Voice (legacy) |
| EF | 46 | `101110` | 184 | Expedited Forwarding (voice/critical) |
| CS6 | 48 | `110000` | 192 | Network control (routing protocol) |
| CS7 | 56 | `111000` | 224 | Network control (highest) |

**Ví dụ match.**

```
# Match VoIP traffic (EF) cho priority queue
priority=200,ip,ip_dscp=46,actions=set_queue:1,output:wan
# Tương đương dùng nw_tos legacy
priority=200,ip,nw_tos=184,actions=set_queue:1,output:wan

# Class scheduler
priority=150,ip,ip_dscp=34,actions=set_queue:2,output:wan  # AF41 multimedia
priority=100,ip,ip_dscp=10,actions=set_queue:3,output:wan  # AF11 normal
priority=10,ip,actions=set_queue:0,output:wan              # BE default

# Mark egress traffic (set DSCP)
priority=100,in_port=voip_port,ip,actions=mod_nw_tos:184,output:N
# Hoặc dùng set_field giữ ECN
priority=100,in_port=voip_port,ip,actions=set_field:46->ip_dscp,output:N
```

**Cách kỹ sư vận hành thành thạo.** (1) **Đọc**: nhớ bảng named DSCP (BE/AF/EF/CS); convert raw nw_tos về DSCP bằng `nw_tos >> 2`. (2) **Sửa**: ưu tiên `set_field:N->ip_dscp` thay vì `mod_nw_tos` để giữ ECN bit. (3) **Quan sát**: `tcpdump -nn -v 'ip and (ip[1] & 0xfc) >> 2 == 46'` filter EF. (4) **Debug**: nếu marking không stick, kiểm tra rule downstream có `mod_nw_tos:0` ghi đè không; nếu remarking phía upstream MPLS không tôn trọng, kiểm tra trust boundary của ISP.

**Source code.** Function `flow_extract` parse byte 1 IPv4 vào `flow.nw_tos`. Function `format_be_uint8_masked` format hiển thị shifted-right cho `ip_dscp`. Function `nx_pull_match`/`oxm_pull_match` (`lib/nx-match.c`) decode OXM TLV. Branch baseline OVS v3.3.0 / v2.17.9.

**Khi xảy ra sự cố.** Triệu chứng A: VoIP packet không vào queue priority — kiểm tra rule có match `ip_dscp=46` không, kiểm tra phone có gắn DSCP=46 không (`tcpdump -v` xem ToS byte). Triệu chứng B: DSCP bị strip ở tunnel — encap GRE/VXLAN có thể không copy DSCP từ inner sang outer; bật DSCP-copy hoặc dùng `set_tunnel_tos` action. Triệu chứng C: marking bằng `mod_nw_tos:184` accidentally clear ECN của packet ECN-aware → TCP performance degrade; dùng `set_field:46->ip_dscp` thay vì `mod_nw_tos:N` toàn byte.

**Liên quan mật thiết tới.** `ip_ecn` (2 bit còn lại của ToS byte), `nw_tos` legacy 8-bit (raw byte trước OF 1.2 split), `set_queue` action (chọn queue egress), `Queue` table OVSDB (định nghĩa rate limit per queue), tunnel options `tos` (cho GRE/VXLAN/Geneve copy DSCP).

**Khác biệt giữa các phiên bản.** OF 1.0 dùng `nw_tos` raw 8-bit (DSCP 46 = nw_tos 184). OF 1.2 (2011) split + shift: `ip_dscp=46` thay vì raw 184. OXM_OF_IP_DSCP semantics shifted-right "/4" theo spec. OVS giữ alias `nw_tos` cho backward compat. Curriculum baseline: OVS v2.17.9 + OVN v22.03.8.

**Cách quan sát + xác minh.**

```
$ tcpdump -nn -v -i any 'ip and (ip[1] & 0xfc) == 184'   # filter DSCP=46 (EF)
$ ovs-ofctl dump-flows br-int | grep 'ip_dscp\|nw_tos'
$ ovs-appctl ofproto/trace br-int 'in_port=1,ip,ip_dscp=46,nw_src=...'
Final flow: ...,nw_tos=184,...        # OVS shows raw nw_tos internally
```

**Trường hợp sự cố thực tế.** ISP customer báo VoIP quality degrade sau migrate sang OVN 22.03 cluster. Investigation: trace cho thấy packet phone gửi `ip_dscp=46` (EF), nhưng sau encap Geneve outer header có `nw_tos=0`. Root cause: tunnel default không copy DSCP từ inner sang outer; ISP MPLS core không thấy EF marking ở outer → priority queue không kích hoạt. Fix: set `Open_vSwitch.other_config:hwol-flow-tunnel-tos=inherit` (or per-port `options:tos=inherit` trên tunnel port) để outer header inherit DSCP từ inner. Reference: ovs-vswitchd.conf.db tunnel options.

**Bài tập synthetic.** Mininet bridge `s1` với 2 host h1/h2. Cài 3 flow + 2 queue: `set Port s1-eth1 qos=@q1 -- --id=@q1 create qos type=linux-htb queues=0=@bg,1=@ef -- --id=@bg create queue other-config:max-rate=1000000 -- --id=@ef create queue other-config:max-rate=10000000`. Flow: `priority=100,ip,ip_dscp=46,actions=set_queue:1,output:1`, `priority=10,ip,actions=set_queue:0,output:1`. Generate traffic 2 stream: `iperf3 --tos 184` (EF) và `iperf3` (BE). Verify EF stream được ưu tiên bandwidth qua `tc -s qdisc show`.

**Lỗi thường gặp + tín hiệu chẩn đoán.** (a) Dùng `nw_tos=46` (sai) — đó là raw byte; phải `ip_dscp=46` hoặc `nw_tos=184`. (b) `mod_nw_tos:184` clear ECN bit của TCP-ECN; dùng `set_field:46->ip_dscp` để chỉ đổi 6 bit DSCP. (c) Quên prerequisite `ip` (`eth_type=0x0800`) — OVS reject.

**So sánh với hệ khác.** Linux iptables `-m dscp --dscp 46` matching tương đương; `tc filter` cũng match DSCP cho HTB queue. Cisco MQC `match dscp ef` dùng tên đầy đủ. NSX-T QoS profile có "DSCP rewrite" abstract trên rule. P4 dùng `match(hdr.ipv4.diffserv)` — toàn 8-bit, application code phải shift để extract DSCP 6 bit.

### 4.8.X.10 `ip_ecn` — ECN 2-bit cho congestion notification

**Khái niệm.** `ip_ecn` là match field 2-bit chứa Explicit Congestion Notification (ECN) bits — 2 bit thấp của byte ToS (IPv4) hoặc byte Traffic Class (IPv6), định nghĩa trong RFC 3168. Có 4 giá trị: `00` Non-ECT (Not-ECN-Capable Transport), `01` ECT(1), `10` ECT(0), `11` CE (Congestion Experienced). Endpoint hỗ trợ ECN gắn ECT(0) hoặc ECT(1) trên packet outbound; router/switch có thể đổi từ ECT thành CE để báo congestion thay vì drop packet. Loại field: **L3 header field**, sub-loại "congestion signaling".

**Lịch sử + bối cảnh.** RFC 2474 (1998) reserve 2 bit thấp ToS làm "Currently Unused". RFC 3168 (2001) chính thức hoá thành ECN. OF 1.0 dùng `nw_tos` 8-bit không tách ECN. OF 1.2 (2011) tách `ip_ecn` (2-bit, OXM_OF_IP_ECN, oxm_field=9) + `ip_dscp` (6-bit). NXM_NX_IP_ECN (mã 28) bổ sung trong OVS v1.4 (2013) làm Nicira extension trước OXM. Stable từ OF 1.2.

**Vị trí trong kiến trúc.** `struct flow.nw_tos` chứa toàn 8-bit; ECN nằm ở 2 bit thấp. MFF: `MFF_IP_ECN` (`meta-flow.h`, type u8 low 2 bits, prereqs IP, OXM_OF_IP_ECN(9) since OF1.2 và OVS v1.7; NXM_NX_IP_ECN(28) since v1.4). OVS extract `(flow.nw_tos & 0x03)` làm value match cho `ip_ecn`.

**Vai trò.** Vai trò chính: detect và signal congestion mà không drop packet (ECN-aware TCP giảm cwnd khi nhận CE). Vai trò phụ: rule có thể count CE-marked traffic để observability (`priority=10,ip,ip_ecn=3,actions=output:congestion_monitor`); router switch có khả năng gắn CE thay drop khi queue gần đầy (AQM + RED + CoDel có ECN-aware variants). Trong OVS+OVN: ít rule trực tiếp dùng `ip_ecn` ở control plane, nhưng kernel datapath path qua tc/qdisc + AQM có thể remark CE; tunnel encap cần copy ECN đúng để semantics ECN-end-to-end.

**Vì sao sinh ra.** Pre-ECN, congestion signal duy nhất là drop packet → TCP retransmit + giảm cwnd. ECN cho phép intermediate router tăng cwnd-feedback mà không drop, giảm latency và goodput loss. Tách ECN khỏi `nw_tos` raw trong OF 1.2 giúp engineer thao tác semantic-clean: `set_field:3->ip_ecn` mark CE chỉ đổi 2 bit thấp, giữ DSCP nguyên.

**Vấn đề giải quyết.** Cụ thể: làm sao expose 2-bit ECN cho rule monitor + remark mà không động chạm DSCP. Cách giải: tách thành field riêng. Phạm vi giải quyết: tất cả use case observe + mark CE; tích hợp với AQM kernel (RED/CoDel/PIE qdisc). Không giải quyết: end-to-end ECN negotiation (TCP layer); không phải mọi router upstream tôn trọng ECN.

**Tầm quan trọng.** Tier 3 peripheral upgraded to Tier 2 medium do tích hợp với DCTCP/L4S deployment ngày càng phổ biến (datacenter ECN-aware congestion control). Engineer must-know: bảng 4 giá trị (0=not-ECT, 1=ECT(1), 2=ECT(0), 3=CE), tunnel ECN-copy semantics RFC 6040, hành vi qdisc `red`/`codel`/`pie` ECN mode.

**Cơ chế hoạt động.** Datapath parse byte ToS, mask `& 0x03` lấy ECN. Action `set_field:3->ip_ecn` (mark CE) shift 0x03 vào 2 bit thấp ToS byte, giữ 6 bit DSCP. Action `mod_nw_ecn:N` tương đương. Tunnel encap (VXLAN/Geneve/GRE) có policy ECN propagation per RFC 6040: outer ECN = inner ECN (full copy) hoặc outer ECN = 0 (legacy mode). Nhận decap: nếu outer = CE và inner = ECT → set inner = CE (signal lên endpoint).

**OXM TLV byte layout.** `oxm_class=0x8000`, `oxm_field=9`, `hasmask=0`, `length=1`, value 1 byte (low 2 bit có ý nghĩa, 6 bit cao zero). Tổng 5 byte. Spec không cho hasmask=1.

**Bảng giá trị ECN.**

| Decimal | Binary (2-bit) | Tên | Ngữ nghĩa |
|---------|----------------|-----|-----------|
| 0 | `00` | Not-ECT | Endpoint không hỗ trợ ECN; congestion → drop |
| 1 | `01` | ECT(1) | Endpoint ECN-capable, code point 1 |
| 2 | `10` | ECT(0) | Endpoint ECN-capable, code point 0 |
| 3 | `11` | CE | Congestion Experienced (router đã mark) |

**Ví dụ match.**

```
# Count CE-marked packet (observability)
priority=10,ip,ip_ecn=3,actions=output:ce_monitor,resubmit(,L3)

# Strip ECN khi exit boundary (legacy domain không hỗ trợ ECN)
priority=100,in_port=ecn_domain,out_port=legacy,ip,
                actions=set_field:0->ip_ecn,output:legacy

# Match ECN-aware traffic
priority=100,ip,ip_ecn=2,actions=set_queue:dctcp,output:N

# Mark CE ở chỗ congestion (typically AQM qdisc tự làm, nhưng test pattern):
priority=200,in_port=src,ip,actions=set_field:3->ip_ecn,output:N
```

**Cách kỹ sư vận hành thành thạo.** (1) **Đọc**: phân biệt 4 ECN value; nhận diện DCTCP/L4S deployment dựa trên ECN distribution (high CE rate báo congestion). (2) **Sửa**: dùng `set_field:N->ip_ecn` để tránh đụng DSCP; tunnel propagation kiểm `options:tos=inherit` trên port tunnel. (3) **Quan sát**: `tcpdump -nn -v 'ip and (ip[1] & 0x03) == 3'` filter CE; `tc -s qdisc show` xem ECN mark count của RED/CoDel. (4) **Debug**: nếu DCTCP không tăng cwnd như expected, kiểm tra outer tunnel có copy ECN không; nếu legacy router upstream clear ECN bit, deployment phải drop ECN ở edge.

**Source code.** Function `flow_extract` parse byte 1 IPv4 vào `flow.nw_tos`; ECN extract bằng `& 0x03`. Function `nx_pull_match` decode OXM cho `MFF_IP_ECN`. Tunnel ECN propagation logic `tnl_ecn_decap` / `tnl_ecn_encap` (`lib/tnl-ports.c`) tuân RFC 6040. Branch baseline OVS v3.3.0 / v2.17.9.

**Khi xảy ra sự cố.** Triệu chứng A: DCTCP performance kém — kiểm tra tunnel `options:tos=inherit` để propagate ECN; check qdisc egress có ECN-aware (red ecn / codel ce_threshold) không. Triệu chứng B: TCP giảm throughput đột ngột — packet capture xem có CE flood không, có thể AQM mark quá aggressive. Triệu chứng C: ECN bit bị clear ở boundary với legacy router → DCTCP không nhận signal; deployment phải downgrade sang non-ECN congestion control (CUBIC) hoặc isolate ECN domain.

**Liên quan mật thiết tới.** `ip_dscp` (6 bit cao của ToS byte), `nw_tos` legacy 8-bit (toàn byte), tunnel ECN copy semantics RFC 6040 (`options:tos=inherit` trên Geneve/VXLAN port), kernel qdisc AQM (red/codel/pie ECN mode), DCTCP/L4S congestion control.

**Khác biệt giữa các phiên bản.** OF 1.0 không có field ECN riêng (dùng `nw_tos` raw 8-bit). OF 1.2 (2011) tách `ip_ecn` 2-bit. OVS NXM_NX_IP_ECN có từ v1.4 trước OXM. Tunnel ECN copy RFC 6040 (2010) implement đầy đủ trong OVS từ v2.4 (2015). Linux kernel TCP DCTCP support từ v3.18 (2014); L4S (RFC 9330+) từ kernel 5.18 (2022). Curriculum baseline: OVS v2.17.9 + OVN v22.03.8.

**Cách quan sát + xác minh.**

```
$ tcpdump -nn -v -i any 'ip and (ip[1] & 0x03) == 3'   # filter CE
$ ovs-ofctl dump-flows br-int | grep ip_ecn
$ tc -s qdisc show dev eth0   # đếm "marked" cho RED/CoDel ECN mode
qdisc red 8001: dev eth0 root limit 1000 ... marked 142 ecn
```

**Trường hợp sự cố thực tế.** DCTCP cluster deploy multi-rack, node trong cùng rack đạt RTT < 1ms tốt nhưng cross-rack throughput drop 40%. Investigation: tcpdump cho thấy outer Geneve ECN bit luôn 0 dù inner = ECT(0) hay CE. Root cause: OVS tunnel port không set `options:tos=inherit`, default copy DSCP=0 cho outer toàn ToS byte. Fix: `ovs-vsctl set Interface ovn-xx options:tos=inherit`; verify qua tcpdump outer ECN match inner. Reference: ovs-vsctl(8) tunnel options + RFC 6040.

**Bài tập synthetic.** Mininet bridge `s1` với 2 host h1/h2 chạy DCTCP (`sysctl net.ipv4.tcp_congestion_control=dctcp`). Cài qdisc red ECN trên link giữa s1 và h2: `tc qdisc add dev s1-eth2 root red limit 100000 min 30000 max 90000 burst 50 ecn`. Cài flow đếm CE: `priority=100,ip,ip_ecn=3,actions=output:ce_count_port`. Generate traffic `iperf3 -c h2`, gradually tăng concurrent stream; quan sát counter port `ce_count_port` tăng khi RED bắt đầu mark; verify sender giảm cwnd thay vì retransmit (tcp counters: `RetransSegs` thấp + `OutSegs` ổn định).

**Lỗi thường gặp + tín hiệu chẩn đoán.** (a) Dùng `mod_nw_tos:N` ghi đè cả ECN của packet end-to-end → break DCTCP. Dùng `set_field:N->ip_dscp` thay thế nếu chỉ muốn đổi DSCP. (b) Tunnel không copy ECN — DCTCP/L4S không hoạt động cross-tunnel; set `options:tos=inherit`. (c) Match `ip_ecn=3` mong CE nhưng packet đã được next-hop reset về 0 (router không tôn trọng ECN); deployment phải drop ECN ở edge.

**So sánh với hệ khác.** Linux iptables `-m ecn --ecn-ip-ect` matching ECN bit. Cisco WRED có "ecn" keyword cho ECN-aware mark thay drop. NSX-T ít expose ECN trực tiếp ở UI nhưng underlying datapath tôn trọng. P4 program access `hdr.ipv4.diffserv` 8-bit, application phải mask `& 0x03` cho ECN. OVS unique ở chỗ tích hợp RFC 6040 tunnel propagation đầy đủ.

|------------|------------|-------------------------|
| Tunnel + IPv6 phụ | 8 | §4.8.15 (`in_phy_port`, `ipv6_flabel`, `ipv6_exthdr`, `sctp_src/dst`, `tunnel_id`, `pkt_mark`, `actset_output`) |

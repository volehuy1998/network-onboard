## Bối cảnh: tại sao OVS chia thành ba thành phần?

Câu hỏi tự nhiên cho người mới tiếp cận OVS: "Linux bridge là một kernel module đơn lẻ, tại sao OVS lại cần ba thứ riêng biệt?" Câu trả lời bắt nguồn từ một yêu cầu kiến trúc mà OVS phải giải quyết ngay từ thiết kế ban đầu (2007): **khả năng thay đổi chuyển tiếp policy từ xa, không cần restart, không cần mất kết nối hiện tại**.

Linux bridge giải quyết bài toán "L2 learning switch cơ bản" với hành vi cố định: bất kỳ config change nào (thêm VLAN, thay đổi STP priority) đều yêu cầu `brctl` command từ userspace, command này modify kernel state qua ioctl. Không có daemon trung gian, không có database. Đủ cho scenario đơn giản.

OVS phải giải quyết bài toán phức tạp hơn: một controller SDN có thể gửi **hàng nghìn OpenFlow rules** trong một second qua kết nối TCP, các rules này phải được validate, stored persistent, compiled thành flow entry kernel, và rollback nếu có lỗi. Không thể làm trong kernel (kernel không chạy state machine phức tạp, không parse JSON, không xử lý TCP connection). Phải có userspace daemon.

Mặt khác, config (bridge, port, VLAN tag, QoS hierarchy) cần **persistence across reboot** và **atomic transaction** (thêm hai port cùng lúc, một queue tham chiếu đến QoS tham chiếu đến Port, tất cả phải thành công hoặc đều rollback). Bài toán này đã có lời giải từ thập niên 1970: database transaction. Nhưng OVS cần một database **lightweight, có schema, hỗ trợ notification thời gian thực**, không muốn dependency vào PostgreSQL hay SQLite. Nicira tự thiết kế OVSDB (sẽ học chi tiết Khối X), và `ovsdb-server` là process độc lập quản lý database file.

Cuối cùng, data plane, thực hiện chuyển tiếp thực sự, phải ở **kernel** để đạt băng thông thực tế 10+ Gbps. Mỗi gói tin đi qua kernel network stack không thể quay sang userspace (context switch + copy userspace-kernelspace tốn ~2 microsecond, quá chậm cho gói tin rate triệu gói tin per second). Nên OVS có kernel module `openvswitch.ko` chạy fast path.

Kết quả: ba process/module, mỗi cái làm đúng một việc, giao tiếp qua interface rõ ràng.

## 9.1.1 Tổng quan ba thành phần và tương tác

Mô hình bốn lớp (four-layer mental model) từ compass_artifact Ch 1 là cách hiệu quả để giữ trong đầu toàn bộ kiến trúc OVS khi khắc phục sự cố:

```
Lớp 1 — Config plane:     OVSDB (ovsdb-server)
                          "What should exist"
Lớp 2 — Control plane:    OpenFlow (ovs-vswitchd)
                          "How packets should be treated"
Lớp 3 — Datapath cache:   Kernel module openvswitch.ko (hoặc userspace netdev)
                          "What the fast path has learned"
Lớp 4 — Wire:             Physical NIC hoặc virtual interface
                          "What actually leaves the NIC"
```

> **Chủ đề trọng tâm:** Phần lớn operational troubleshooting trong OVS là **so sánh bốn lớp và tìm nơi chúng mâu thuẫn**. Nếu OVSDB config nói bridge có 3 port, OpenFlow flow table nói bridge có 2 port, kernel datapath báo 3 port, thì có sự lệch trạng thái (lệch trạng thái) giữa lớp 1 và lớp 2 — thường do sync issue giữa `ovsdb-server` và `ovs-vswitchd`. Nếu OVSDB đúng, OpenFlow đúng, nhưng gói tin vẫn không đi — khả năng cao vấn đề ở lớp 4 (NIC driver, MTU, LRO offload). Biết mỗi lớp đại diện cho gì giúp thu hẹp không gian debug nhanh chóng.

**Sơ đồ flow xử lý một gói tin đi qua OVS (kernel datapath)**:

```
┌─────────────────────────────────────────────────────────────────┐
│                     Userspace                                    │
│                                                                   │
│  ┌────────────────┐       ┌──────────────────────────┐          │
│  │ ovsdb-server   │◄─────►│ ovs-vswitchd             │          │
│  │ (config DB)    │ JSON-  │  - OpenFlow pipeline    │          │
│  │ /etc/openvswi- │  RPC  │  - classifier (TSS)      │          │
│  │ tch/conf.db    │       │  - handler threads        │          │
│  └────────────────┘       │  - revalidator threads   │          │
│                            └────────┬─────────────────┘          │
│                                     │                             │
└─────────────────────────────────────┼─────────────────────────────┘
                             Netlink genl
                          (ovs_datapath, ovs_flow,
                             ovs_vport families)
                                      │
┌─────────────────────────────────────▼─────────────────────────────┐
│                     Kernel space                                   │
│                                                                     │
│  ┌──────────────────────────────────────────────┐                 │
│  │  openvswitch.ko                               │                 │
│  │                                                │                 │
│  │   Packet in ──► Megaflow cache lookup         │                 │
│  │                    │                           │                 │
│  │                    ├── hit  ──► execute action │                 │
│  │                    │                           │                 │
│  │                    └── miss ──► upcall to user │                 │
│  │                                    │           │                 │
│  │                                    ▼           │                 │
│  │                              ovs-vswitchd      │                 │
│  │                                                │                 │
│  └──────────────────────────────────────────────┘                 │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

Ba thành phần không giao tiếp trực tiếp với nhau theo mô hình one-to-one. `ovs-vswitchd` ở trung tâm: nó kết nối với `ovsdb-server` qua Unix socket JSON-RPC (đọc config, subscribe change), và kết nối với kernel module qua Netlink genl family (cài đặt flow, nhận upcall khi cache miss). `ovsdb-server` không bao giờ nói chuyện trực tiếp với kernel; kernel không bao giờ đọc OVSDB.

**Xác nhận từ USC Lab 1** (`doc/ovs/OVS.txt` Mục 1.2): "The Open vSwitch daemon runs on the root namespace. It listens to netlink event messages from all networking namespaces. Each netlink message contains the network namespace identifier as ancillary data used to match the event to the corresponding port." Chi tiết này giải thích tại sao OVS daemon (`ovs-vswitchd`) có thể quản lý interface nằm trong nhiều network namespace khác nhau — Netlink message chứa namespace ID, daemon dispatch đúng bridge tương ứng.

## 9.1.2 ovs-vswitchd: chuyển tiếp daemon userspace

`ovs-vswitchd` là process có vai trò lớn nhất trong ba thành phần, nó chứa phần lớn logic nghiệp vụ của OVS. Một binary duy nhất, chạy như single process (không fork con), với nhiều thread để xử lý workload parallel.

**Main loop** của `ovs-vswitchd` dựa trên epoll (Linux syscall) và poll timer. Mỗi iteration của main loop:

1. Poll OVSDB connection để biết có config change nào không.
2. Poll OpenFlow connection từ controller để nhận flow_mod, packet_out, barrier_request.
3. Poll Netlink socket để nhận upcall từ kernel datapath.
4. Poll Unix socket `/var/run/openvswitch/ovs-vswitchd.ctl` để nhận lệnh từ `ovs-appctl`.
5. Poll timer cho periodic task (bond rebalance, STP BPDU send, revalidator trigger).

Main thread xử lý phần lớn event, nhưng một số workload đặc biệt được tách ra **pool thread riêng**:

- **Handler threads**: xử lý upcall gói tin từ kernel khi cache miss. Mặc định 2 threads, config qua `other_config:n-handler-threads`. Mỗi upcall: parse gói tin, walk OpenFlow pipeline, compile megaflow entry, cài về kernel qua Netlink.
- **Revalidator threads**: định kỳ (mặc định 500ms) scan toàn bộ megaflow cache trong kernel, check xem mỗi entry còn hợp lệ không (flow rules có thay đổi không, idle timeout chưa hết hạn không), xóa entry stale. Mặc định 4 threads.
- **OpenFlow worker threads**: xử lý OpenFlow connection từ controller. Mỗi bridge có một thread riêng.

`ovs-appctl memory/show`, lệnh từ compass_artifact Ch L, liệt kê breakdown memory theo component: classifier (flow table), handlers, revalidators, connections, datapath. Lệnh này là entry point khi debug memory pressure.

```bash
[host]# ovs-appctl -t ovs-vswitchd memory/show
handlers:4 ports:5 revalidators:4 rules:127 ...
```

`ovs-appctl coverage/show` liệt kê counter cho mọi internal event: upcall, flow install, flow delete, netlink roundtrip, allocation. Counter có rate per-second, per-minute, per-hour. Ví dụ output:

```
cache-misses       0.0/sec     0.017/sec    0.065/sec   total: 234
flow-lookup        1523.0/sec  1489.4/sec   1567.2/sec  total: 5234112
upcall_flow_limit  0.0/sec     0.0/sec      0.0/sec     total: 0
```

Đọc counter này giúp phát hiện các điểm bất thường: nếu `cache-misses` rate tăng đột biến trong khi `flow-lookup` giảm, nghĩa là workload đang sinh nhiều microflow khác pattern, cache không compress hiệu quả.

> **Lưu ý kỹ thuật:** `ovs-vswitchd` không phải daemon chạy với privilege cao mặc định trên distro hiện đại. Ubuntu 22.04 mặc định chạy `ovs-vswitchd` với user `root` để có quyền CAP_NET_ADMIN (cấu hình network) và CAP_SYS_NICE (set thời gian thực scheduling cho PMD threads). Có thể hạ privilege bằng cách set SELinux/AppArmor profile, nhưng thực tế hầu hết deployment vẫn chạy root.

## 9.1.3 ovsdb-server: JSON-RPC database (Khối X sẽ chi tiết)

`ovsdb-server` là process nhẹ nhất trong ba thành phần về memory footprint (thường 20-50 MB RAM cho deployment với hàng trăm port). Nhiệm vụ duy nhất: serve một OVSDB database file qua JSON-RPC protocol.

Database file của OVS vswitchd nằm ở `/etc/openvswitch/conf.db`. Schema được định nghĩa trong `/usr/share/openvswitch/vswitch.ovsschema` (JSON file) và được load khi `ovsdb-server` khởi động. Schema định nghĩa các table quan trọng (từ compass_artifact Ch 4): `Open_vSwitch` (singleton root), `Bridge`, `Port`, `Interface`, `Controller`, `Manager`, `QoS`, `Queue`, `Mirror`, `NetFlow`, `sFlow`, `IPFIX`, `Flow_Table`, `SSL`, `CT_Zone`, `CT_Timeout_Policy`.

Client của `ovsdb-server` kết nối qua Unix socket mặc định `/var/run/openvswitch/db.sock`, hoặc TCP/SSL cho remote access. Hai client quan trọng:

- **`ovs-vsctl`**: command-line client, dùng bởi admin. Mỗi invocation là một transaction duy nhất. Cú pháp `ovs-vsctl -- command -- command -- ...` với `--` làm statement separator, tất cả commit atomic hoặc rollback toàn bộ (compass Ch A).
- **`ovs-vswitchd`**: connect đến `ovsdb-server` ngay khi khởi động, subscribe vào toàn bộ Open_vSwitch database với `monitor_cond` protocol. Bất kỳ thay đổi nào trong database được push realtime về `ovs-vswitchd`, không cần polling.

Khối X (Parts 10.0-10.2) sẽ phân tích chi tiết OVSDB protocol (RFC 7047), JSON-RPC message format, Raft clustering cho HA. Ở Phần 9.1 này chỉ cần hiểu vai trò: **database lưu cấu hình, không lưu runtime state**.

Phân biệt quan trọng cần giữ trong đầu: OVSDB lưu **ý đồ cấu hình** (user muốn có bridge X với port Y), không lưu **operational state** (bridge có thực sự hoạt động không, port có bao nhiêu gói tin đã qua). Operational state được `ovs-vswitchd` tính toán runtime và expose qua một số cột đặc biệt trong OVSDB (ví dụ `Interface.statistics`, `Interface.link_state`), nhưng không persist vào disk.

## 9.1.4 openvswitch.ko: kernel datapath

Kernel module `openvswitch.ko` (và các module phụ thuộc `nf_conntrack`, `nf_defrag_ipv4`, `nf_nat`) implement fast path của OVS. Khi gói tin đến từ NIC, Linux kernel chuyển cho `openvswitch.ko` qua netdev rx hook (cho `type=system` interface) hoặc `internal` driver (cho port internal OVS tạo).

`openvswitch.ko` duy trì hai cấu trúc dữ liệu chính trong kernel memory:

**Cấu trúc 1. Flow table (kernel):** đây là megaflow cache. Mỗi entry có: (a) match key, tổ hợp các field gói tin với mask wildcard, (b) action list, danh sách action (output:N, set_tunnel, push_vlan, ...), (c) statistics counter. Lookup dùng tuple-space search: với mỗi tuple (mask configuration), duy trì một hash table riêng; lookup scan qua các tuple tuần tự.

**Cấu trúc 2. Port table:** map OpenFlow port number → netdev pointer. Khi action là `output:5`, kernel tra port table để tìm netdev, gọi `netdev_ops->ndo_start_xmit` để gửi gói tin.

Khi gói tin đến và lookup cache miss (không match entry nào), kernel không drop gói tin ngay mà **gửi lên userspace qua upcall**. Cơ chế gọi ngược là message Netlink chứa: (a) gói tin original, (b) metadata (in_port, timestamp), (c) reason (miss, userspace action). `ovs-vswitchd` nhận upcall, chạy OpenFlow pipeline trong userspace để tính ra flow action, compile thành megaflow entry, và cài ngược về kernel qua Netlink.

**Xác nhận từ USC Lab 9** (`doc/ovs/Day 4-lab 9 - Open vSwitch Kernel Datapath.txt`):

- "Kernel module directs the gói tin to the userspace. The userspace makes the decisions about the actions to be taken against the gói tin according to OpenFlow entries (slow path)."
- "The action entry is stored in the kernel and used to forward subsequent gói tin which makes the chuyển tiếp faster (fast path)."
- "Initially, kernel table is empty" → gói tin đầu tiên luôn cache miss, đi slow path.
- "Action entries are installed in the kernel" → sau slow path hoàn tất, cache được populate.

Đây là mô tả tiêu chuẩn của kiến trúc slow path + fast path trong OVS, dùng thuật ngữ khớp với codebase upstream.

> **Chủ đề trọng tâm:** Cơ chế gọi ngược là cơ chế cho phép OVS vừa nhanh (fast path trong kernel) vừa linh hoạt (logic phức tạp trong userspace). Khớp cache đi hoàn toàn trong kernel, không context switch, băng thông thực tế cao. Lỡ cache đi userspace một lần để compile, sau đó mọi gói tin tương tự sẽ hit cache. Benchmark trong NSDI 2015 paper cho thấy cache hit rate 99%+ trong workload production — nghĩa là dưới 1% gói tin phải đi userspace, chi phí context switch được amortize qua hàng nghìn gói tin tương tự sau đó.

Kernel datapath có hai loại tùy theo datapath type của bridge (xem compass Ch B):

- **`datapath_type=system`** (mặc định): dùng `openvswitch.ko` kernel module. Gói tin đi qua kernel network stack. Băng thông thực tế giới hạn bởi kernel interrupt overhead, thường 5-10 Gbps per flow trên hardware thông thường.
- **`datapath_type=netdev`**: dùng userspace datapath (thường với DPDK hoặc AF_XDP). Gói tin bypass kernel, đi thẳng từ NIC driver vào userspace polling thread. Băng thông thực tế 50+ Gbps per flow có thể đạt được nhưng CPU cost cao hơn (dedicate polling core).

Phần 9.2 sẽ phân tích sâu kernel datapath và megaflow cache. Phần 9.3 sẽ phân tích userspace datapath (DPDK, AF_XDP).

## 9.1.5 Netlink genl family: đường truyền userspace ↔ kernel

Netlink là family socket đặc biệt trong Linux kernel dùng để truyền thông điệp giữa userspace và kernel. Khác biệt với Unix socket thường: Netlink hỗ trợ multicast group (subscribe để nhận event), có semantic request-response rõ ràng, và mỗi protocol family có schema message riêng.

OVS dùng **Generic Netlink** (genl), một layer phụ trên Netlink cho phép nhiều "family" nhỏ chia sẻ chung port Netlink `NETLINK_GENERIC` (16). Mỗi family có một tên string, kernel gán một family ID động khi module load. Userspace lookup family ID qua family name qua `CTRL_CMD_GETFAMILY`.

OVS đăng ký ba Netlink generic family:

- **`ovs_datapath`**: quản lý datapath lifecycle. Command: `OVS_DP_CMD_NEW` (tạo datapath mới, một bridge trong terminology kernel), `OVS_DP_CMD_DEL` (xóa), `OVS_DP_CMD_SET` (update attributes), `OVS_DP_CMD_GET` (query).
- **`ovs_flow`**: quản lý flow entry trong datapath. Command: `OVS_FLOW_CMD_NEW` (install flow), `OVS_FLOW_CMD_DEL` (xóa), `OVS_FLOW_CMD_SET` (update), `OVS_FLOW_CMD_GET` (dump). Cơ chế gọi ngược từ kernel cũng đi qua family này với attribute `OVS_PACKET_ATTR_USERDATA`.
- **`ovs_vport`**: quản lý virtual port (interface gán vào bridge). Command tương tự datapath: `NEW`, `DEL`, `SET`, `GET`.

Nhìn Netlink traffic giữa userspace và kernel bằng `tcpdump` không work (Netlink không phải protocol network). Dùng `strace` để trace syscall, hoặc công cụ chuyên dụng `nlmon` (network monitor cho Netlink).

Cấu trúc message Netlink generic family có header chung và body tùy command:

```
+-------------------+
| nlmsghdr          |  16 bytes - Netlink common header
+-------------------+
| genlmsghdr        |  4 bytes  - Generic Netlink header
+-------------------+
| OVS_DP_ATTR_NAME  |  TLV - datapath name
| OVS_DP_ATTR_UPCALL|  TLV - upcall socket
| ... attributes    |
+-------------------+
```

Quan trọng đối với khắc phục sự cố: khi `ovs-vswitchd` gặp lỗi Netlink (kernel module không tương thích version, kernel thiếu feature), log thường có message dạng `failed to generate netlink family: ovs_datapath` hoặc `could not add datapath...`. Đây là dấu hiệu mismatch giữa userspace OVS version và kernel module version, thường do upgrade OVS userspace mà quên update kernel module, hoặc ngược lại.

### ▶ Guided Exercise 2: Khảo sát ba thành phần OVS trên host thật

Trong bài thực hành này, bạn xác định ba process/module của OVS, kiểm tra Netlink connection giữa userspace và kernel, và quan sát memory footprint.

**Outcomes:**
- Xác định PID của `ovs-vswitchd` và `ovsdb-server` đang chạy
- Liệt kê được thread model của `ovs-vswitchd` (main, handlers, revalidators)
- Chạy `ovs-appctl memory/show` và `coverage/show` để đọc internal state

**Before You Begin:**
Đảm bảo OVS đang chạy với ít nhất một bridge có traffic thực. Tạo bridge thử và inject traffic:

```bash
[host]# ovs-vsctl add-br br-test
[host]# ovs-vsctl add-port br-test tap0 -- set Interface tap0 type=internal
[host]# ip link set tap0 up
[host]# ip addr add 10.99.0.1/24 dev tap0
[host]# ping -c 100 -f 10.99.0.1 &    # tạo traffic tự-loopback
```

**Instructions:**

**1.** Liệt kê process OVS với thread:

```bash
[host]$ ps -eLf | grep -E 'ovs-vswitchd|ovsdb-server' | grep -v grep
```

Cột LWP (Light Weight Process = thread ID) cho thấy `ovs-vswitchd` có nhiều thread. Cột cuối cùng (CMD với thread name) cho biết role: `main`, `handler*`, `revalidator*`, `urcu*`. Ghi nhận số lượng handler và revalidator thread, thường 2 handler + 4 revalidator trên cấu hình mặc định.

**2.** Kiểm tra Netlink family đã đăng ký:

```bash
[host]# cat /proc/net/netlink | head -5
[host]# genl ctrl list | grep -E 'ovs_'
```

(Lệnh `genl` từ package `iproute2`.) Output sẽ hiện ba family: `ovs_datapath`, `ovs_vport`, `ovs_flow` với family ID (số). Family ID này khác nhau sau mỗi lần reload kernel module.

**3.** Memory breakdown của `ovs-vswitchd`:

```bash
[host]# ovs-appctl -t ovs-vswitchd memory/show
```

Output như `handlers:4 ports:4 revalidators:4 rules:127 udpif keys:3`. Số `rules` là số OpenFlow flow entries; `udpif keys` là số ukeys (unique keys) mà revalidator đang track, tương ứng với số megaflow trong kernel datapath.

**4.** Counter coverage:

```bash
[host]# ovs-appctl -t ovs-vswitchd coverage/show | head -30
```

Các counter quan trọng: `cache-misses` (gói tin không hit megaflow, gây upcall), `flow-lookup` (tổng số lookup), `upcall_flow_limit` (lần bị hit flow-install rate limit). Counter có format `rate_per_sec / rate_per_min / rate_per_hour   total: cumulative`.

**5.** Kernel datapath statistics:

```bash
[host]# ovs-dpctl show
```

Output liệt kê datapath (`system@ovs-system` cho kernel), số port, và counter `lookups: hit:X missed:Y lost:Z`. `hit` là cache hit, `missed` là cache miss gây upcall, `lost` là upcall fail (hiếm, thường do rate limit).

**6.** Dump flow entries trong kernel datapath:

```bash
[host]# ovs-dpctl dump-flows -m
```

Output là các megaflow entry với match key, mask (wildcard pattern), và action. Output dài có thể nhiều dòng mỗi flow. Chú ý cột `packets:N, bytes:M` ở cuối mỗi flow, số gói tin đã match entry đó.

**Finish:**

```bash
[host]# kill %1                              # stop ping
[host]# ovs-vsctl del-br br-test            # cleanup
```

## Điểm cốt lõi cần nhớ

Thứ nhất, OVS chia ba process/module vì mỗi tầng có yêu cầu khác nhau: config persistence (OVSDB) cần database, control logic phức tạp (OpenFlow pipeline) cần userspace, và fast path gói tin (datapath) cần kernel. Mỗi thành phần làm đúng một việc, giao tiếp qua interface rõ ràng.

Thứ hai, mental model bốn-lớp (config → OpenFlow → datapath cache → wire) là công cụ tư duy hiệu quả nhất khi khắc phục sự cố. Mọi vấn đề operational đều quy về "lớp nào đang lệch với lớp nào", và chiến lược debug là lần lượt check từng lớp qua lệnh tương ứng: `ovs-vsctl list`, `ovs-ofctl dump-flows`, `ovs-dpctl dump-flows`, `tcpdump trên interface`.

Thứ ba, Netlink generic family (`ovs_datapath`, `ovs_flow`, `ovs_vport`) là cầu nối duy nhất giữa userspace và kernel. Mọi upcall (cache miss) đều đi qua đây, mọi flow install cũng vậy. Hiểu cơ chế này là nền tảng cho Phần 9.2 phân tích chi tiết megaflow và upcall. Lỗi Netlink thường là dấu hiệu mismatch version giữa `ovs-vswitchd` userspace và `openvswitch.ko` kernel module.

Thứ tư, các lệnh `ovs-appctl memory/show`, `coverage/show`, `ovs-dpctl show`, `ovs-dpctl dump-flows` là bộ công cụ chẩn đoán tối thiểu mà mọi kỹ sư vận hành OVS cần thuộc. Khi hệ thống có sự cố, trước khi tìm nguyên nhân ở application hay network, luôn chạy bốn lệnh này để biết OVS đang ở trạng thái nào.

## 9.1.X ofproto-dpif + connmgr internals (Phase H session S45 append)

Mục này bổ sung chi tiết về hai internal subsystem quan trọng không được cover ở các section trước: **ofproto-dpif** (datapath-ifies OpenFlow pipeline) và **connmgr** (connection manager).

### 9.1.X.1 Kiến trúc ofproto-dpif trong ovs-vswitchd

Ovs-vswitchd daemon phân lớp thành 5 layer chính (top-down):

```
┌─────────────────────────────────────┐
│ bridge.c                            │  ← Bridge lifecycle + OVSDB sync
├─────────────────────────────────────┤
│ ofproto/ofproto.c                   │  ← OpenFlow protocol handling
├─────────────────────────────────────┤
│ ofproto/ofproto-dpif.c              │  ← Datapath-aware ofproto
│ ofproto/ofproto-dpif-xlate.c        │  ← Action translation
│ ofproto/ofproto-dpif-upcall.c       │  ← Upcall + revalidator
├─────────────────────────────────────┤
│ ofproto/connmgr.c                   │  ← Connection manager (OF controllers)
│ lib/classifier.c                    │  ← Classifier (TSS + trie)
├─────────────────────────────────────┤
│ lib/dpif.c (+ dpif-netlink.c,       │  ← Datapath interface abstraction
│             dpif-netdev.c)          │
└─────────────────────────────────────┘
    │             │
    ▼             ▼
kernel dp      userspace dp (DPDK/AF_XDP)
```

- `bridge.c`: wire-up OVSDB state vào ofproto config (add-br, add-port, set-controller).
- `ofproto.c`: abstract OpenFlow ops (flow-mod, group-mod, gói tin-in/out) — không biết datapath concrete.
- `ofproto-dpif.c`: concrete implementation dùng `dpif` layer.
- `dpif.c`: abstraction layer, có hai impl: `dpif-netlink` (kernel) và `dpif-netdev` (userspace).
- `connmgr.c`: manage OpenFlow controller connection per bridge.
- `classifier.c`: rule lookup engine (Phần 9.15).

### 9.1.X.2 Đọc `ovs-appctl dpif/show` output

**Bối cảnh:** sau khi biết kiến trúc, dùng `dpif/show` để kiểm tra state runtime.

```shell
$ sudo ovs-appctl dpif/show
system@ovs-system:
  bridge br-int/0:
    flows: (current 1247) (avg 1250) (max 1350) (limit 200000)
    br-int 65534/<local>: (internal)
    tap-vm-web 1/1: (internal)
    tap-vm-db 2/2: (internal)
    ovn-huyvl-0 3/3: (geneve: remote_ip=10.200.0.12, csum=true, key=flow)
```

**Anatomy:**

| Field | Ý nghĩa |
|---|---|
| `system@ovs-system` | Datapath name (kernel default "system") |
| `bridge br-int/0` | Bridge name + OpenFlow bridge ID |
| `flows: current/avg/max/limit` | Flow count in datapath |
| `<port_name> X/Y` | X = OpenFlow port number (ofport), Y = datapath port number (odp_port) |
| `(internal)` / `(geneve: ...)` | Port type + config |

X vs Y khác nhau: OpenFlow port number là persistent (OVSDB tracked); datapath port number có thể ephemeral (kernel assigned trên tạo).

### 9.1.X.3 Connection manager anatomy

Connmgr chịu trách nhiệm:

1. Mở/close TCP/SSL/Unix socket tới OpenFlow controller.
2. Manage role election (MASTER/SLAVE/EQUAL).
3. Queue OFPT message khi controller slow nhận.
4. Backoff reconnect khi controller down.
5. Dispatch async message (PACKET_IN, PORT_STATUS, FLOW_REMOVED) tới controller(s) phù hợp role.

Xem Phần 9.16 cho chi tiết role management + async config.

### 9.1.X.4 Thread model tổng hợp

Per bridge ovs-vswitchd:

- **Main thread:** OVSDB sync + ofproto_run() cycle.
- **Handler threads:** N = `n-handler-threads` config (default 3). Xử lý upcall từ kernel.
- **Revalidator threads:** N = `n-revalidator-threads` config (default 4). GC + reinstall flow.
- **PMD threads:** chỉ với DPDK datapath. Per pinned CPU core polling NIC queue.
- **URCU thread:** cleanup RCU-managed struct sau grace period.

Xem `coverage/show connmgr_*` và `memory/show handlers/revalidators` để biết thread state.

## 9.1.Y Ofproto-dpif xlate engine (Phase I session S63 append)

Mục này đi sâu vào **xlate engine** của ovs-vswitchd, layer translate một gói tin + OpenFlow flow table thành datapath action list. Đây là hot path quan trọng nhất của OVS userspace: mọi `PACKET_IN` upcall từ kernel datapath, mọi `ovs-ofctl add-flow` làm slow-path simulation, và mọi `ofproto/trace` đều chạy qua code path này.

### 9.1.Y.1 Khi nào xlate chạy?

Xlate engine được invoke ở 4 trigger khác nhau:

1. **Cơ chế gọi ngược từ kernel datapath** (hot path). Kernel không tìm được megaflow match cho gói tin đầu tiên của flow mới, wrap gói tin thành Netlink genl message `OVS_PACKET_CMD_MISS`, gửi lên userspace. Handler thread trong ovs-vswitchd nhận, gọi `xlate_actions()` để dịch gói tin qua OpenFlow pipeline, sinh ra datapath action list + megaflow mask, rồi install megaflow về kernel.
2. **Revalidator scan** (background). Revalidator thread định kỳ walk qua tất cả megaflow trong kernel để validate: flow rule có còn tồn tại không? Action list có thay đổi không? Nếu có thay đổi, gọi lại `xlate_actions()` trên cùng gói tin key để regenerate datapath action, so sánh với cached action, update nếu khác.
3. **`ofproto/trace` appctl command** (manual diagnostic). Operator chạy `ovs-appctl ofproto/trace br-int in_port=1,...`, ovs-vswitchd gọi `xlate_actions()` với tùy chọn verbose để trace từng step, không install gì vào datapath. Xem Phần 9.25 cho deep-dive trace output anatomy.
4. **Flow simulation khi flow-mod arrives** (optional). Khi controller add flow mới với `OFPFF13_RESET_COUNTS` hoặc eviction trigger, ofproto-dpif gọi xlate để pre-compute effect.

4 trigger cùng chia sẻ entry point `xlate_actions()` nhưng khác nhau ở `xlate_in.resubmit_stats` flag: upcall path set stats để count real gói tin; trace path clear stats; revalidator path dùng stored stats.

### 9.1.Y.2 Entry point `xlate_actions()` và struct `xlate_in` / `xlate_out`

Function prototype (file `ofproto/ofproto-dpif-xlate.c`, baseline OVS 2.17.9 và OVS 3.3):

```c
enum xlate_error xlate_actions(struct xlate_in *xin,
                                struct xlate_out *xout);
```

Caller truyền hai struct: `xlate_in` chứa input (gói tin flow + starting rule + context flags), `xlate_out` là output buffer được fill trong quá trình translate.

**`struct xlate_in` key fields:**

| Field | Type | Ý nghĩa |
|-------|------|---------|
| `ofproto` | `struct ofproto_dpif *` | Which bridge ofproto (br-int, br-ex) |
| `flow` | `struct flow` | Gói tin key: metadata + L2/L3/L4 header fields đã parsed |
| `packet` | `struct dp_packet *` | Raw gói tin (nullable nếu chỉ trace) |
| `rule` | `struct rule_dpif *` | Starting rule (nullable, xlate sẽ lookup nếu NULL) |
| `recurse` | `int` | Recursion depth counter, tracks stack depth cho `resubmit()` |
| `resubmit_stats` | `struct dpif_flow_stats *` | Nullable, non-NULL = accumulate stats |
| `odp_actions` | `struct ofpbuf *` | Output buffer cho datapath action list |
| `wc` | `struct flow_wildcards *` | Output: wildcard mask build-up từ lookup (megaflow) |

**`struct xlate_out` key fields:**

| Field | Type | Ý nghĩa |
|-------|------|---------|
| `slow` | `enum slow_path_reason` | Nếu non-zero, gói tin cần slow path (reason: controller-bound, sflow, ...) |
| `fail_open` | `bool` | Fail-open mode active |

Điểm quan trọng: `wc` được build up khi xlate walk qua classifier. Mỗi khi rule_dpif_lookup match một field cụ thể, bit tương ứng trong `wc` được set. Cuối xlate, `wc` chứa đúng set field cần cho megaflow mask, loại bỏ wildcard aggressive hơn mỗi field không được kiểm tra.

### 9.1.Y.3 Anatomy Template A: struct `xlate_ctx` internal state

Struct này local trong `xlate_actions()`, chứa runtime state quá trình translate. Không caller access trực tiếp, chỉ debug qua `ofproto/trace --verbose`.

**Raw source từ OVS `ofproto/ofproto-dpif-xlate.c` (rút gọn các field chính):**

```c
struct xlate_ctx {
    struct xlate_in *xin;
    struct xlate_out *xout;

    const struct xbridge *xbridge;

    struct flow base_flow;            /* Flow at the last commit. */
    struct flow orig_flow;            /* Copy of original flow. */

    ofp_port_t in_port;               /* Current in_port. */
    struct ofpbuf *odp_actions;       /* Datapath actions being built. */

    int table_id;                     /* Current table ID. */
    int indentation;                  /* Nested actions indent level (trace). */
    int depth;                        /* Current resubmit nesting depth. */
    int resubmits;                    /* Total number of resubmits so far. */
    bool in_group;                    /* Inside an OpenFlow group action. */
    bool in_action_set;               /* Processing action_set (OF 1.1+). */

    struct flow_tnl base_flow_tnl;    /* Snapshot for tunnel pop/push. */
    struct flow_tnl orig_flow_tnl;

    uint32_t orig_skb_priority;
    uint32_t sflow_n_outputs;
    odp_port_t sflow_odp_port;
    bool exit;                        /* Stop processing? */

    /* Conntrack. */
    bool conntracked;
    uint8_t ct_nat_action;

    struct hmap stats_hmap;           /* Per-rule stats accumulator. */

    /* Freezing state (for continuing after recirc). */
    struct frozen_state *frozen_state;
    /* ... nhiều field khác ... */
};
```

**Anatomy 9-attribute:**

| Attribute | Value | Ghi chú |
|-----------|-------|---------|
| **Life cycle** | Stack-allocated trong `xlate_actions()` | Initialize sau khi caller tạo `xlate_in` |
| **Scope** | Per-xlate invocation, thread-local | Handler thread riêng một struct, không shared |
| **Depth tracking** | `depth` field, hard limit `MAX_RESUBMIT_RECURSION = 64` | `resubmit()` action tăng `depth`, `goto_table` tăng `table_id` không tăng `depth` |
| **Resubmit count** | `resubmits` field, limit `MAX_RESUBMITS = 4096` | Total resubmit across cả pipeline (protect vô hạn loop do rule mistake) |
| **Flow snapshot** | `base_flow` (at last commit) + `orig_flow` (at start) | Cho phép action rollback hoặc modify metadata rồi restore |
| **Action buffer** | `odp_actions` grow dần | Cuối xlate, content của buffer = datapath action list được install |
| **Exit flag** | `bool exit` | Set khi output to controller hoặc drop, stop further translation |
| **Conntrack state** | `conntracked` + `ct_nat_action` | Track xem đã qua `ct()` action chưa, preserve cho post-recirc |
| **Freezing** | `frozen_state` pointer | Nếu action require recirculation, freeze context, unfreeze khi kernel recirc gói tin về |

**Kịch bản bẻ gãy anatomy:**

1. **Rule vô hạn loop.** Ghi một flow trên table 0 với action `resubmit(,0)`. Xlate ngay lập tức đi vào recursion deadlock. `MAX_RESUBMIT_RECURSION = 64` bảo vệ: khi `ctx->depth > 64`, xlate abort với error `XLATE_TOO_DEEP`. Observability: `ovs-appctl coverage/show xlate_actions_too_many_resubmits` tăng counter.

2. **Rule explosion.** Flow với action `resubmit(,1), resubmit(,2), ..., resubmit(,5000)` (5000 action song song). `MAX_RESUBMITS = 4096` chặn. Counter `xlate_actions_too_many_resubmits` tăng. Một bug thường gặp: loop chain qua `goto_table` mà không có priority-0 rule, xlate tăng table_id tới 255 (MAX_TABLES) rồi abort.

3. **Conntrack recirc.** Flow `ct(commit,table=5),output:3` làm xlate tạo freeze state: "sau khi kernel commit conntrack, recirc về table 5 trong cùng pipeline". `frozen_state` save metadata. Kernel thực sự perform ct(), rồi emit upcall recirc để xlate nối tiếp. Nếu lookup tiếp ở table 5 không match gì, gói tin drop.

### 9.1.Y.4 Action translation walkthrough: 1 gói tin qua pipeline

Trace chi tiết `xlate_actions()` với gói tin từ port 1 đi vào br-int có flow:

```
table=0,priority=100,in_port=1,ip actions=resubmit(,10)
table=10,priority=100,ip,nw_dst=10.0.0.2 actions=mod_dl_dst:aa:bb:cc:dd:ee:02,output:3
```

Call chain step-by-step:

```
xlate_actions(xin, xout)                           // Entry
├── ctx_cancel_freeze(&ctx)                        // Clear any previous freeze
├── rule = rule_dpif_lookup_from_table(            // Table 0 lookup
│     ofproto, flow, wc, table_id=0)
│   ├── classifier_lookup(cls, ...)                // TSS search (Part 9.15)
│   └── returns: rule(table=0, priority=100)       // Match found
├── do_xlate_actions(rule->actions, rule->len, &ctx)  // Translate action list
│   └── for each action in rule->actions:
│       ├── OFPACT_RESUBMIT(table=10):
│       │   ├── ctx->resubmits++
│       │   ├── ctx->depth++
│       │   ├── rule_resubmit = rule_dpif_lookup_from_table(
│       │   │     ofproto, flow, wc, table_id=10)
│       │   ├── returns: rule(table=10, priority=100)
│       │   ├── do_xlate_actions(rule_resubmit->actions, len, &ctx)
│       │   │   ├── OFPACT_SET_FIELD(eth_dst=aa:bb:cc:dd:ee:02):
│       │   │   │   ├── ctx->flow.dl_dst = aa:bb:cc:dd:ee:02
│       │   │   │   ├── ofpbuf_put: odp_actions += "set(eth_dst=...)"
│       │   │   │   └── wc->masks.dl_dst = 0xff:ff:ff:ff:ff:ff
│       │   │   └── OFPACT_OUTPUT(port=3):
│       │   │       ├── odp_port = ofp_port_to_odp_port(port=3)
│       │   │       └── ofpbuf_put: odp_actions += "output:3"
│       │   └── ctx->depth--
│       └── (end loop)
└── return XLATE_OK
```

Kết quả `xin->odp_actions`:

```
set(eth_dst=aa:bb:cc:dd:ee:02),output:3
```

`xin->wc` build up các field đã được kiểm tra:

- `masks.in_port = 0xffffffff` (do flow match `in_port=1`)
- `masks.dl_type = 0xffff` (do flow match `ip` = ethertype 0x0800)
- `masks.nw_dst = 255.255.255.255` (do flow match `nw_dst=10.0.0.2`)
- Các field khác (src_ip, src_port, dst_port, ethertype sub-fields) `0` (wildcard, megaflow trùm)

Megaflow mask sinh ra: `in_port=1,dl_type=0x0800,nw_dst=10.0.0.2`. Megaflow này sẽ match bất kỳ gói tin nào cùng 3 field giá trị, bất chấp source IP/MAC, port, TCP flags. Đây chính là **lazy wildcarding** mà Phần 9.2 giải thích.

### 9.1.Y.5 Megaflow mask build-up qua `wc` tracking

Mỗi khi classifier touch một field, macro `flow_wildcards_set_xxx_mask` được gọi để mark field đó trong `wc`. Các macro này thường inline expand. Ví dụ khi match `nw_dst`:

```c
WC_MASK_FIELD(wc, nw_dst);
```

expand thành:

```c
((wc)->masks.nw_dst) = htonl(0xffffffff);
```

Point quan trọng: **một field được mark trong wc không đồng nghĩa nó phải có giá trị cụ thể**. Wildcard tracking chỉ quan tâm "field này có influence matching decision không?". Nếu rule có `priority=100` chặn match khác trên cùng field, xlate vẫn mark field đó trong `wc` dù rule hiện tại không có condition trên nó.

Đây là cơ chế critical cho correctness: nếu thiếu mark, megaflow sinh ra sẽ quá rộng (false match cho gói tin khác), vi phạm semantic OpenFlow.

### 9.1.Y.6 Trace output mapping

Lệnh `ovs-appctl ofproto/trace` in ra step-by-step của chính `xlate_actions()`. Mỗi indent level tương ứng `ctx->depth`. Ví dụ trace gói tin ở §9.1.Y.4:

```
Flow: ip,in_port=1,vlan_tci=0x0000,dl_src=aa:bb:cc:dd:ee:01,dl_dst=00:00:00:00:00:00,nw_src=10.0.0.1,nw_dst=10.0.0.2,nw_proto=6,nw_tos=0,nw_ttl=64,tp_src=12345,tp_dst=80,tcp_flags=0

bridge("br-int")
-----------------
 0. in_port=1,ip, priority 100, cookie 0x0
    resubmit(,10)
10. ip,nw_dst=10.0.0.2, priority 100, cookie 0x0
    mod_dl_dst:aa:bb:cc:dd:ee:02
    output:3

Final flow: unchanged
Megaflow: recirc_id=0,eth,ip,in_port=1,nw_dst=10.0.0.2,nw_frag=no
Datapath actions: set(eth(dst=aa:bb:cc:dd:ee:02)),3
```

Mapping:

- `bridge("br-int")` <=> `ctx->xbridge->name`
- ` 0.` <=> `ctx->table_id = 0`, indent = 1 space (no resubmit yet)
- `resubmit(,10)` <=> `OFPACT_RESUBMIT` đưa pipeline tiếp tục ở table 10
- `10.` <=> `ctx->table_id = 10`, indent tăng theo `ctx->depth`
- `Megaflow:` <=> `xout->wc.masks` format hóa về Flow syntax
- `Datapath actions:` <=> `xin->odp_actions` content

### 9.1.Y.7 Vai trò kernel vs userspace split

Câu hỏi tự nhiên: tại sao userspace phải làm xlate từng gói tin đầu tiên? Không thể "kernel tự lookup rule table" sao?

Kernel datapath **không biết OpenFlow**. Nó chỉ hiểu datapath action set nhỏ: `output`, `set(field)`, `push_vlan`, `hash`, `recirc`, `ct`, `userspace`. Không biết `resubmit`, `goto_table`, `group`, `learn`, `write_metadata`. Những action logical này phải được **compile xuống** chuỗi datapath action cụ thể. Đó là nhiệm vụ xlate.

Lợi ích phân lớp:

1. **Kernel simple + fast.** Kernel chỉ cần handle khoảng 20 action type, dễ optimize. Mọi logic phức tạp đẩy lên userspace.
2. **Pipeline evolution.** Thêm OpenFlow action mới không cần update kernel, chỉ update xlate.
3. **Megaflow sharing.** Nhiều gói tin cùng 5-tuple mask nhờ lazy wildcarding, kernel lookup 1 megaflow thay vì rebuild mỗi gói tin.
4. **Upgrade separation.** OVS userspace có thể upgrade 2.17 sang 3.3 trong khi kernel giữ nguyên (backward-compatible kernel dp interface).

Pfaff NSDI 2015 §5 trình bày chi tiết. Ta sẽ gặp lại xlate trong Phần 9.2 (kernel datapath cache + megaflow), Phần 9.15 (classifier TSS mà xlate dùng), Phần 9.25 (ofproto/trace debug).

### 9.1.Y.8 Guided Exercise S63.1: quan sát xlate qua `ofproto/trace --verbose`

**Mục tiêu:** nhìn thấy `ctx->depth`, `table_id`, action translation step-by-step bằng trace verbose.

**Chuẩn bị (sẽ chạy khi có lab host):**

1. Ubuntu 22.04 LTS + OVS 2.17.9 default package.
2. Bridge `br-test` với 3 OpenFlow ports:
   ```

   ovs-vsctl add-port br-test veth1 -- set interface veth1 ofport_request=1
   ovs-vsctl add-port br-test veth2 -- set interface veth2 ofport_request=2

   ```
3. Cài flow rule 2-table:
   ```
   ovs-ofctl add-flow br-test "table=0,priority=100,in_port=1,ip actions=resubmit(,10)"
   ovs-ofctl add-flow br-test "table=10,priority=100,ip,nw_dst=10.0.0.2 actions=mod_dl_dst:aa:bb:cc:dd:ee:02,output:3"
   ```

**Bước thực hành:**

1. Trace gói tin baseline:
   ```
   ovs-appctl ofproto/trace br-test in_port=1,dl_type=0x0800,nw_src=10.0.0.1,nw_dst=10.0.0.2
   ```
   Ghi nhận: `Megaflow:` line.

2. Trace verbose mode (in ra từng xlate step):
   ```
   ovs-appctl ofproto/trace --verbose br-test in_port=1,dl_type=0x0800,nw_src=10.0.0.1,nw_dst=10.0.0.2
   ```
   Output sẽ có thêm section `Thaw stack:` và per-step context dump.

3. Reproduce max resubmit protection. Add flow loop:
   ```
   ovs-ofctl add-flow br-test "table=20,priority=100 actions=resubmit(,20)"
   ovs-appctl ofproto/trace br-test in_port=1,dl_type=0x0800 --flow table=20
   ```
   Quan sát error `Too many resubmits.` hoặc tương tự. Check counter:
   ```
   ovs-appctl coverage/show | grep xlate_
   ```

**Predict - Observe - Explain (POE):**

- **Predict:** Ở bước 3, xlate sẽ abort trước khi install datapath action, trace output sẽ ghi chú lỗi max resubmit.
- **Observe:** thực tế trace output thế nào? Counter nào tăng?
- **Explain:** tại sao OVS đặt `MAX_RESUBMITS = 4096` thay vì cho recursion vô hạn? (Answer: protection chống rule misconfig. 4096 đủ lớn cho pipeline OVN phức tạp 50 stage × recirc nhưng vẫn hữu hạn.)

**Bài học:** `ctx->depth` + `ctx->resubmits` hard limits là safety net critical. Nếu một controller ghi sai rule tạo loop, bridge không treo, chỉ drop gói tin cá nhân đó. Xlate fail nhanh + log coverage counter cho operator phát hiện.

### 9.1.Y.9 Capstone POE S63: "Xlate có thể cache kết quả không?"

**Câu hỏi:** một kỹ sư đề xuất: thay vì xlate mỗi upcall, hãy cache mapping `<flow key hash> → <datapath actions>` trong userspace (tương tự kernel megaflow cache nhưng ở level xlate). Đây có phải optimization đúng không?

**POE response khuyến nghị:**

**Predict:** phần lớn người mới sẽ nói "có, cache tốt vì reduce CPU". Thực tế OVS **đã có** mechanism này: đó chính là **megaflow cache trong kernel**, xem Phần 9.2. Lý do nó ở kernel chứ không ở userspace:

**Observe:**

1. Cache ở kernel tránh user-kernel context switch khi hit. Cache ở userspace không tránh được.
2. Lifecycle. Megaflow tự động evict qua revalidator + flow-aging. Cache userspace cần logic cleanup riêng, dễ stale.
3. Concurrency. Kernel datapath lookup lockless per-CPU (xem Phần 9.2). Cache userspace phải add lock hoặc RCU, thêm complexity.

**Explain:**

Lý do xlate **không** cache trong userspace là vì mỗi xlate invocation depend vào current state của **toàn bộ** flow table + pipeline + ofproto metadata. Cache key phải bao gồm: flow key + rule table version + metadata state. Một revalidator hoặc flow-mod có thể thay đổi rule, cache key phải bump. Ở kernel, megaflow cache hoạt động vì nó **re-validate per-gói tin** (check rule còn tồn tại + action không đổi) qua revalidator thread, đó là chi phí validity mà OVS đã chấp nhận thiết kế vào.

Nếu add thêm cache ở userspace, về mặt lý thuyết tiết kiệm được xlate cost cho gói tin thứ 2+ của cùng flow. Nhưng: (a) gói tin thứ 2+ đã hit kernel megaflow, không đến xlate userspace; (b) gói tin đầu tiên vẫn phải xlate dù có cache hay không (vì cache miss); (c) upcall thường batch 1-10 gói tin, xlate cost đã amortize. Benefit thực tế gần bằng 0. Cost thêm: memory + cache invalidation logic.

Pfaff NSDI 2015 §8 confirm điểm này: OVS designer đã cân nhắc và reject userspace flow cache. Thiết kế hiện tại là optimal cho cân bằng complexity-performance.

**Bài học:** tối ưu hóa đúng layer quan trọng hơn thêm cache nhiều tầng. Kernel megaflow cache làm đúng việc, userspace xlate chạy chủ yếu ở cold path (gói tin đầu tiên + revalidate + trace), không cần cache thêm.

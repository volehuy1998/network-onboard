Sau khi hoàn thành phần này, bạn có thể:

1. **Mô tả (Understand)** Cilium datapath: eBPF programs attached at tc (traffic control) hook + XDP + socket-level, replacing kube-proxy iptables.

3. **Áp dụng (Apply)** debug Cilium datapath với `cilium monitor`, `cilium bpf policy get`, `hubble observe` để trace packet path.

## 15.2.1 eBPF datapath architecture

*Skeleton — nội dung sẽ bao gồm:* eBPF program attach points (tc ingress/egress, XDP, socket filters, kprobes), eBPF map types (hash, LPM trie, LRU), eBPF verifier constraints (bounded loops, stack size).

## 15.2.2 Cilium CNI vs OVN-K8s

*Skeleton — nội dung sẽ bao gồm:* So sánh kernel datapath (Cilium eBPF) vs userspace + OVS datapath (OVN-K8s), latency profile, NetworkPolicy expressiveness, debugging tooling.

## 15.2.3 kube-proxy replacement

*Skeleton — nội dung sẽ bao gồm:* Cilium's `--kube-proxy-replacement=strict`, hashmap lookup O(1) vs iptables chain traversal O(n), session affinity implementation.

## 15.2.4 Cilium Service Mesh (sidecar-less)

*Skeleton — nội dung sẽ bao gồm:* Shared Envoy proxy per-node (instead of per-pod sidecar), socket-level L7 visibility, mTLS tùy chọn, Ingress Gateway integration.

## 15.2.5 Hubble observability

*Skeleton — nội dung sẽ bao gồm:* Hubble relay + UI, flow event export (eBPF ring buffer → userspace), service map generation, L7 HTTP/gRPC/Kafka visibility.

### ▶ Guided Exercise 1: Inspect Cilium eBPF programs

> **Trạng thái lab verification:** Output dựa trên Cilium 1.15 docs + bpftool reference; chờ verify lab thực.

**Mục đích:** Hiểu Cilium datapath cụ thể — list eBPF programs attached, inspect eBPF maps, trace packet processing. Lab này mở "black box" Cilium để thấy eBPF-based CNI hoạt động ra sao.

**Chuẩn bị:**
- Ubuntu 22.04 VM với kernel 5.15+ (eBPF features)
- kind cluster + Cilium CNI
- `bpftool` installed: `sudo apt install linux-tools-generic` (or build from kernel source)

**Bước thực hiện:**

**1.** Create kind cluster with Cilium:

```bash
[host]$ kind create cluster --name cilium
[host]$ cilium install --version 1.15.0
[host]$ cilium status --wait
```

Output kỳ vọng:

```
    /¯¯\__/¯¯\    Cilium:         OK
 \__/¯¯\__/      Operator:       OK
  /¯¯\__/¯¯\    Envoy DaemonSet: disabled
   \__/¯¯\__/    Hubble Relay:    disabled
```

**2.** Deploy sample app:

```bash
[host]$ kubectl run web --image=nginx
[host]$ kubectl run client --image=curlimages/curl -- sleep 3600
```

**3.** Exec vào Cilium pod:

```bash
[host]$ CILIUM_POD=$(kubectl -n kube-system get pod -l k8s-app=cilium -o jsonpath='{.items[0].metadata.name}')
[host]$ kubectl -n kube-system exec -ti $CILIUM_POD -- bash
```

**4.** Inside Cilium pod — list eBPF programs:

```bash
[cilium-agent]# bpftool prog list | grep -i 'sched_cls\|xdp\|sk_msg' | head -20
```

Output kỳ vọng:

```
2457: sched_cls  name cil_to_container  tag abc123  gpl
    loaded_at 2026-04-22T12:34:56+0000  uid 0
    xlated 2048B  jited 1536B
    pinned /sys/fs/bpf/tc/globals/cilium_calls_...
```

**5.** List eBPF maps:

```bash
[cilium-agent]# bpftool map list | grep cilium | head -20
```

Output kỳ vọng:

```
123: hash  name cilium_lxc  flags 0x0
  key 16B  value 256B  max_entries 65536  memlock 18MB
124: lru_hash  name cilium_ipcache  flags 0x1
  key 20B  value 24B  max_entries 512000
125: prog_array  name cilium_calls_0000  flags 0x0
  key 4B  value 4B  max_entries 50
```

**6.** Dump policy map cho một pod:

```bash
[cilium-agent]# ENDPOINT_ID=$(cilium endpoint list -o json | jq '.[] | select(.status.labels."k8s:app"=="web") | .id')
[cilium-agent]# bpftool map dump name cilium_policy_v2_$(printf "%05d" $ENDPOINT_ID) | head
```

Output kỳ vọng: entries cho allowed ingress/egress với identity IDs.

**7.** Trace packet từ client → web:

```bash
[host]$ kubectl exec client -- curl -s http://web
[cilium-agent]# cilium monitor --type trace | head -20
```

Output kỳ vọng: trace events showing `FROM_ENDPOINT` → `TO_OVERLAY` → `TO_ENDPOINT` → response reverse.

**Bài học học được:**

Thứ nhất, **Cilium = eBPF-first CNI**: không dùng iptables cho kube-proxy/NetworkPolicy. Mọi logic trong eBPF programs attached at tc (traffic control) + XDP hook points + socket filters.

Thứ hai, **Map-based state**: policy, service load balancer backends, IP-to-identity mapping all live trong eBPF maps. Userspace (cilium-agent) updates maps; datapath (eBPF program) reads maps at packet time. Lock-free, per-CPU tối ưu.

Thứ ba, **Observability built-in qua bpftool + cilium monitor**: production Cilium deployment có clear visibility không phải black box. Compare với iptables chains (opaque, order-sensitive) → Cilium advantage rõ rệt.

**Cleanup:**

```bash
[host]$ kind delete cluster --name cilium
```

### ▶ Lab 2: Cilium Service Mesh benchmark vs Istio sidecar

**Same as Lab 15.0-Lab2 — reference that lab for benchmark methodology + expected results.**

**Additional focus cho Cilium SM specifically:**
- Đo latency delta giữa Cilium SM sidecar-less mode và Istio sidecar full-injection
- Analyze CPU overhead per node vs per pod

## 16.2.0 Drama mở đầu, từ "DPDK hoặc nothing" đến "AF_XDP compromise kernel 4.18 (2018)"

Năm 2015-2018, network engineering ở Linux bị mắc kẹt giữa 2 lựa chọn cực đoan:

- **Kernel stack**: Đầy đủ feature (iptables, tc, netfilter, routing, bridging) nhưng throughput limit ~2 Mpps per core.
- **DPDK**: Bypass toàn bộ kernel, đạt 15+ Mpps per core, **nhưng mất hết kernel features** + yêu cầu hugepages + IOMMU + CPU dedication + complexity cao.

Không có middle ground. Cloudflare, Facebook, Netflix muốn filter DDoS ở line-rate 40+ Gbps trên commodity hardware, không muốn pay DPDK complexity, nhưng kernel quá chậm cho packet drop 10M+ pps.

Năm 2014-2016, **Alexei Starovoitov** (Facebook, nay Meta) phát triển **eBPF** (extended BPF), JIT-compiled virtual machine trong kernel cho phép lập trình viên viết code C, verified safe, chạy với near-native performance trong kernel context. Năm 2016, **Jesper Dangaard Brouer** (Red Hat) + **Daniel Borkmann** (Isovalent) propose **XDP (eXpress Data Path)**: hook point cho eBPF program chạy **TRƯỚC khi kernel cấp phát sk_buff**, tại ngay driver NIC. Kernel 4.8 (2016) merge XDP BPF infrastructure.

Năm 2018, **Björn Töpel** + **Magnus Karlsson** (Intel) đóng góp **AF_XDP** socket family cho kernel 4.18: kết hợp XDP hook với shared memory ring (UMEM) cho phép userspace nhận packet zero-copy, **giữ kernel driver + kernel features intact**. Đây là **middle ground** giữa DPDK và kernel stack: throughput ~10-30 Mpps per core (80% của DPDK), giữ kernel stack cho packet không redirect, dễ deploy hơn DPDK.

Sự chấp nhận nhanh chóng: **Cilium** (2018+) dùng XDP cho K8s service LB + network policy enforcement; **Facebook Katran** (2018, open-source) dùng XDP_TX cho stateless LB; **Suricata IDS** (2019+) dùng AF_XDP cho deep packet inspection line-rate; **OVS-AFXDP** (2020) cho OVS userspace datapath alternative. Năm 2024-2026, XDP + AF_XDP đã trở thành standard cho DDoS mitigation, network policy enforcement, custom LB.

Part 16.2 này đi sâu vào AF_XDP 4-ring architecture + 3 attach mode + 4 return code + libbpf/libxdp API + eBPF map types + production pattern. Kết thúc với 2 exercise hands-on (XDP_PASS minimal, AF_XDP redirect TCP) + Capstone POE "DDoS mitigation 200 Gbps: XDP hay DPDK?".

> **Hiểu sai phổ biến:** *"AF_XDP chỉ là DPDK chậm hơn."*
>
> **Thực tế:** AF_XDP có 3 ưu điểm KHÁC biệt so với DPDK: (a) giữ kernel features (iptables, tc, routing) cho packet không redirect, (b) không cần hugepages + IOMMU + isolcpus (dễ deploy), (c) chia sẻ NIC với kernel traffic thay vì độc chiếm. Throughput kém DPDK ~20-30% cho gói nhỏ, nhưng use case khác nhau. DPDK cho workload dedicated (NFV, HFT), AF_XDP cho selective bypass (Cilium K8s, DDoS mitigation, Suricata IDS).

## 16.2.1 Kiến trúc AF_XDP (4 ring)

AF_XDP socket triển khai giao tiếp userspace-kernel qua bốn **ring** shared memory, tổ chức theo mô hình producer-consumer:

**(1) FILL ring**, userspace là producer: đẩy frame index (offset trong UMEM) của các buffer trống vào ring để báo cho kernel "các buffer này sẵn sàng nhận packet mới". Kernel là consumer. Khi có packet đến, kernel dequeue một frame index từ FILL ring, DMA packet từ NIC vào buffer tương ứng, rồi enqueue frame index đó vào RX ring.

**(2) RX ring**, kernel là producer, userspace là consumer. Mỗi entry chứa {frame_idx, offset, len}. Userspace thăm dò RX ring qua syscall `recvmsg()` (chế độ thường) hoặc thăm dò liên tục (chế độ busy-poll). Sau khi xử lý packet, userspace phải đẩy frame index trả lại FILL ring để tái sử dụng.

**(3) TX ring**, userspace là producer, kernel là consumer. Userspace muốn gửi packet: ghi dữ liệu vào một buffer UMEM trống, enqueue {frame_idx, offset, len} vào TX ring, gọi `sendto()` để kích hoạt kernel xử lý ring. Kernel dequeue từ TX ring, DMA từ UMEM tới NIC, rồi enqueue frame index đó vào COMPLETION ring.

**(4) COMPLETION ring**, kernel là producer, userspace là consumer. Báo hiệu cho userspace biết buffer đã được truyền xong và có thể tái sử dụng. Userspace dequeue từ COMPLETION ring, trả frame index về FILL ring nếu muốn tái dùng cho RX.

**UMEM (User Memory)** là vùng bộ nhớ liên tục (thường vài MB - vài GB) được cấp phát bằng `mmap()` + đăng ký với kernel qua `setsockopt(XDP_UMEM_REG)`. UMEM chia thành các frame cố định kích thước (thường 2048 hoặc 4096 bytes). Kernel DMA TRỰC TIẾP từ NIC vào UMEM khi driver hỗ trợ chế độ zero-copy, không có memcpy giữa kernel buffer và UMEM.

Bốn ring này là shared memory giữa userspace và kernel, đồng bộ qua chỉ số producer/consumer nguyên tử. Không có khóa, không có chuyển đổi ngữ cảnh, đây là lý do AF_XDP đạt hiệu năng gần DPDK.

## 16.2.2 Chế độ driver so với SKB

**XDP_DRIVER (chế độ native):** chương trình XDP chạy trong ngữ cảnh ngắt của driver NIC, TRƯỚC khi kernel cấp phát sk_buff. Điểm gắn hiệu năng cao nhất, khoảng 24 Mpps per core trên Intel XXV710, hơn 30 Mpps trên Mellanox ConnectX-6. Yêu cầu driver hỗ trợ native XDP: Intel `i40e`/`ice`/`ixgbe`, Mellanox `mlx4`/`mlx5`, Broadcom `bnxt`, Netronome `nfp`, virtio `virtio_net`, TAP device `tun`. Kiểm tra: `ethtool -i eth0 | grep driver` rồi tra cứu driver trong `kernel/drivers/net/ethernet/*/`.

**XDP_SKB (chế độ generic):** chương trình XDP chạy SAU khi kernel đã cấp phát sk_buff. Hoạt động với MỌI driver, kể cả những driver không có XDP native. Nhưng hiệu năng giảm đáng kể, khoảng 3-5 Mpps per core. Lý do: đã trả phí cấp phát sk_buff (hàm `build_skb()` tốn khoảng 100-200 ns), nhưng không có lợi ích zero-copy. Chế độ này dành cho phát triển/gỡ lỗi hoặc workload có nhiều driver loại cũ.

**XDP_HW (offload phần cứng):** compile chương trình eBPF cho firmware NIC, chạy trong ASIC NIC, hoàn toàn không tốn CPU. Hỗ trợ bởi Netronome Agilio CX (driver nfp) và Mellanox ConnectX-5+ (với card hỗ trợ eBPF offload). Hạn chế tập lệnh: một số helper eBPF không có sẵn (ví dụ tra cứu map chậm), và chương trình phải đủ đơn giản để vừa trong bộ nhớ của NIC ASIC. Phù hợp cho trường hợp sử dụng đơn giản (lọc DDoS đơn giản, drop trước firewall) nơi tiết kiệm CPU quan trọng hơn linh hoạt.

**Cách chọn chế độ:** luôn thử **XDP_DRIVER** trước, kiểm tra driver hỗ trợ qua `bpftool net show`. Nếu driver không hỗ trợ, chuyển về XDP_SKB (chấp nhận được cho traffic dưới 5 Gbps). XDP_HW chỉ dùng khi có NIC đặc thù và workload đơn giản.

## 16.2.3 Loại và hành động của chương trình XDP

Sau khi chương trình XDP xử lý một packet, nó trả về một trong bốn mã hành động chuẩn:

**XDP_PASS:** cho phép packet tiếp tục đi vào kernel networking stack như bình thường. Dùng khi chương trình XDP chỉ quan sát (telemetry, đếm) mà không can thiệp. Đây là hành vi mặc định nếu chương trình không có logic drop/redirect.

**XDP_DROP:** drop packet ngay lập tức ở điểm sớm nhất trong pipeline (trước khi kernel cấp phát sk_buff). Chi phí dưới 100 ns per packet. Đây là cơ chế cho giảm thiểu DDoS hiệu quả: khi phát hiện traffic độc hại, XDP_DROP hàng chục triệu packet per second mà không tốn CPU của kernel stack. Cloud flare và AWS Shield sử dụng chiến lược này để giảm thiểu DDoS 100+ Gbps.

**XDP_REDIRECT:** chuyển hướng packet sang đích khác. Hai chế độ con: (1) **chuyển hướng tới AF_XDP socket** qua `bpf_redirect_map(&xsk_map, queue_id, 0)`, packet được đẩy vào UMEM của userspace (đã trình bày mục 16.2.1). (2) **chuyển hướng tới NIC khác** qua `bpf_redirect(ifindex, 0)`, packet đi thẳng ra NIC xuất mà không qua kernel routing (dùng cho software switch XDP như ovs-afxdp, Katran của Facebook). XDP_REDIRECT là chế độ mạnh nhất, đạt line-rate cho nhiều workload chuyển tiếp.

**XDP_TX:** gửi packet ra cùng interface mà nó đến, thường sau khi sửa đổi header. Dùng cho load balancer không trạng thái (Katran), NAT nhanh, hoặc trình trả lời ARP/ICMP. Không cần entry bảng định tuyến, packet đi thẳng ra queue TX của cùng interface.

Chương trình XDP được viết bằng C (tập con hạn chế, không malloc, không số thực, không vòng lặp không biên), compile bằng clang với target `bpf`: `clang -O2 -target bpf -c xdp_prog.c -o xdp_prog.o`. Verifier kiểm tra bytecode trước khi tải.

## 16.2.4 libbpf + libxdp ở userspace

**libbpf** là thư viện C chuẩn cho ứng dụng userspace tương tác với eBPF, open-source tại `github.com/libbpf/libbpf`. API chính:

- `bpf_object__open(path)`: tải file ELF chứa chương trình eBPF đã compile.
- `bpf_object__load(obj)`: tải chương trình vào kernel qua syscall `bpf(BPF_PROG_LOAD)`.
- `bpf_program__attach_xdp(prog, ifindex)`: gắn chương trình XDP vào interface.
- `bpf_map__find_by_name(obj, name)`: lấy handle của eBPF map để thao tác từ userspace.
- `bpf_map_update_elem(map_fd, key, value, flags)` / `bpf_map_lookup_elem()`, cập nhật/đọc map.

**libxdp** là wrapper cao cấp hơn libbpf, tập trung riêng vào trường hợp sử dụng XDP, open-source tại `github.com/xdp-project/xdp-tools`. Cung cấp API:

- `xdp_program__open_file(path)`: tải chương trình XDP với metadata (tên XDP prog, ưu tiên chạy trong thiết lập nhiều prog).
- `xdp_program__attach(prog, ifindex, mode)`: gắn với chế độ tường minh (XDP_MODE_NATIVE, XDP_MODE_SKB, XDP_MODE_HW).
- `xdp_multiprog__add_prog()`: hỗ trợ nhiều chương trình XDP trên cùng interface (ra mắt kernel 5.10), theo thứ tự ưu tiên.
- Helper cho AF_XDP socket: `xsk_socket__create()`, `xsk_ring_prod__reserve()`, v.v.

**Các loại map chung giữa chương trình XDP và userspace:**

- **BPF_MAP_TYPE_PERCPU_ARRAY**: mảng per-CPU, mỗi CPU có bản sao riêng, không cần khóa. Dùng cho đếm packet per-CPU mà chương trình XDP cập nhật và userspace tổng hợp qua tất cả CPU.
- **BPF_MAP_TYPE_HASH**: bản đồ key-value tổng quát, tra cứu O(1). Dùng cho bảng policy (src IP → allow/deny), bảng session.
- **BPF_MAP_TYPE_LPM_TRIE**: cho bảng định tuyến CIDR (subnet nguồn → hành động).
- **BPF_MAP_TYPE_XSKMAP**: ánh xạ queue_id → AF_XDP socket, dùng cho XDP_REDIRECT sang userspace.

Quy trình phát triển chuẩn: viết chương trình XDP (.c) → compile với clang → viết code userspace loader dùng libbpf/libxdp → kiểm thử với traffic tổng hợp (pktgen, trex) → đo bằng perf/bpftrace.

## 16.2.5 Đặc tính hiệu năng

**Benchmark AF_XDP thực tế** (nguồn: OVS-AFXDP tuning guide Intel 2023, Linux Plumbers Conference 2023 AF_XDP presentations):

- NIC 25GbE Intel XXV710, 1 core: AF_XDP zero-copy đạt 12 Mpps với gói 64-byte (gần line-rate 37 Mpps lý thuyết).
- NIC 100GbE Mellanox ConnectX-6, 4 core: AF_XDP zero-copy đạt 60 Mpps tổng (gần line-rate 148 Mpps lý thuyết cho 64-byte).
- Gói lớn hơn (1500-byte): AF_XDP đạt line-rate trên hầu hết NIC 10-100GbE với chỉ 1-2 core.

**Độ trễ** (packet round-trip trong cùng node): AF_XDP khoảng 10 microseconds, so với DPDK khoảng 2 microseconds. Khác biệt đến từ: (1) AF_XDP vẫn qua bộ xử lý IRQ của driver kernel; (2) packet phải đi qua tc hook trước khi vào XDP (chế độ generic). Với chế độ native, độ trễ giảm xuống khoảng 5 microseconds, gần DPDK hơn.

**Ưu điểm so với DPDK:** (1) **Giữ tính năng kernel**, netfilter, tc qdisc, bảng định tuyến vẫn hoạt động bình thường cho traffic không bị chuyển hướng. Hệ thống vẫn có cấu hình IP, quy tắc iptables. (2) **Không cần tách NIC** khỏi kernel driver, triển khai đơn giản, không có nguy cơ mất kết nối lúc tái cấu hình. (3) **Không cần hugepages** bắt buộc, dùng page thường cũng chạy được (hiệu năng thấp hơn một chút). (4) **Triển khai cùng với container** dễ dàng, không xung đột với Kubernetes CNI.

**Nhược điểm so với DPDK:** (1) **PPS thuần thấp hơn** khoảng 20-30% cho gói nhỏ (64-byte) do vẫn qua một số đường dẫn code kernel. (2) **Cộng đồng nhỏ hơn DPDK**, ít ví dụ, ít thư viện sẵn có. (3) **Yêu cầu kernel 5.10+** cho tính năng đầy đủ (XDP metadata, XDP multi-prog), một số workload trên kernel cũ (RHEL 7, Ubuntu 18.04 stock) không chạy được.

**Lựa chọn giữa AF_XDP và DPDK:** AF_XDP cho workload cần PPS trung-cao (10-60 Mpps) với triển khai đơn giản, như Cilium, Suricata, OVS-AFXDP. DPDK cho workload PPS cực cao (hơn 100 Mpps), 5G UPF, packet gateway viễn thông, nơi hiệu năng là yếu tố quyết định và có đội ngũ sẵn sàng đầu tư vào triển khai phức tạp.

### ▶ Guided Exercise 1: Attach minimal XDP_PASS program với bpftool

> **Trạng thái lab verification:** Output dựa trên kernel XDP docs + bpftool reference 2024; chờ verify lab thực.

**Mục đích:** Entry-level XDP lab, viết một XDP program đơn giản nhất (return XDP_PASS cho mọi packet), compile, load, attach, verify. Ba seline cho mọi XDP development later.

**Chuẩn bị:**
- Ubuntu 22.04 VM với kernel 5.15+
- Install toolchain:
  ```bash
  sudo apt install -y clang llvm libbpf-dev libelf-dev gcc-multilib \
                      linux-headers-$(uname -r) linux-tools-$(uname -r)
  ```
- Driver support XDP native: e1000e, virtio_net, mlx5_core, i40e, ice đều support. Check: `ethtool -i eth0 | grep driver`.

**Mô hình lab:**

```
┌────────────────────────────────────────────────┐
│    Ubuntu 22.04 VM                             │
│                                                │
│    Packet inbound ───► NIC (eth0)              │
│                          │                     │
│                          ▼                     │
│                   ┌──────────────────┐         │
│                   │  XDP program     │         │
│                   │  xdp_pass_prog() │         │
│                   │  return XDP_PASS │         │
│                   └──────┬───────────┘         │
│                          │                     │
│                          ▼                     │
│                   Kernel network stack         │
│                   (normal processing)          │
└────────────────────────────────────────────────┘
```

**Bước thực hiện:**

**1.** Viết XDP program `xdp_pass.c`:

```c
#include <linux/bpf.h>
#include <bpf/bpf_helpers.h>

SEC("xdp")
int xdp_pass_prog(struct xdp_md *ctx) {
    return XDP_PASS;
}

char _license[] SEC("license") = "GPL";
```

**2.** Compile:

```bash
[host]$ clang -O2 -g -target bpf -c xdp_pass.c -o xdp_pass.o
[host]$ ls -la xdp_pass.o
-rw-r--r-- 1 user user 1024 Apr 22 12:34 xdp_pass.o
```

**3.** Load program vào kernel:

```bash
[host]# bpftool prog load xdp_pass.o /sys/fs/bpf/xdp_pass
[host]# ls /sys/fs/bpf/
xdp_pass
```

**4.** Get program ID + attach to interface:

```bash
[host]# PROG_ID=$(bpftool prog show pinned /sys/fs/bpf/xdp_pass | awk -F: '/^[0-9]/ {print $1}')
[host]# echo $PROG_ID
42
[host]# bpftool net attach xdp id $PROG_ID dev eth0
```

**5.** Verify attached:

```bash
[host]# ip link show eth0
```

Output kỳ vọng:

```
2: eth0: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 xdp/id:42 qdisc ...
    link/ether aa:bb:cc:dd:ee:ff brd ff:ff:ff:ff:ff:ff
    prog/xdp id 42 tag 123abc jited
```

Key indicators: `xdp/id:42` trong ip link output, `jited` = program was JIT-compiled to native machine code cho performance.

**6.** Measure performance impact (sanity check, should be negligible):

```bash
# Baseline (no XDP)
[host]# bpftool net detach xdp dev eth0
[host]# iperf3 -c 10.0.0.1 -t 10

# With XDP_PASS
[host]# bpftool net attach xdp id $PROG_ID dev eth0
[host]# iperf3 -c 10.0.0.1 -t 10
```

Output kỳ vọng: throughput deltas within ±1% (XDP_PASS is near-zero overhead since it just falls through to kernel stack).

**Bài học học được:**

Thứ nhất, **XDP program architecture**: C source → clang -target bpf → eBPF bytecode → kernel verifier → JIT compile → native execution. Pipeline này ensure safety (verifier reject unbounded loops, invalid memory access) + performance (JIT to native instructions).

Thứ hai, **XDP_PASS is no-op**: packet continues to kernel stack unchanged. Useful cho initial development, attach program without breaking traffic, gradually add logic.

Thứ ba, **bpftool vs ip link**: cả hai có thể attach XDP. `bpftool` advantage: cross-platform, native BPF concepts. `ip link set dev eth0 xdp obj xdp_pass.o` là alternative.

**Cleanup:**

```bash
[host]# bpftool net detach xdp dev eth0
[host]# rm /sys/fs/bpf/xdp_pass
```

Thứ hai, **AF_XDP zero-copy** preserve kernel features cho selected packets. ICMP vẫn reach kernel (ping works), TCP redirect vào userspace (custom processing), UDP dropped silently. Tính linh hoạt không cần full DPDK bypass.

Thứ ba, **Production use case**: DDoS mitigation (XDP drop malicious traffic), custom TCP accelerator (AF_XDP redirect for in-house processing), monitoring tap (XDP mirror to AF_XDP without impact). Cilium + Katran (Facebook LB) real-world examples.

## 16.2.6 So sánh XDP/AF_XDP với DPDK, OVS, và Cilium production

Trong hệ sinh thái 2024-2026, XDP/AF_XDP đã vượt DPDK về sự chấp nhận cho K8s + cloud native workload. Bảng so sánh 4 stack chính:

| Dimension | Kernel stack + iptables | OVS-DPDK userspace | XDP + AF_XDP | Cilium eBPF (XDP + tc) |
|-----------|------------------------|--------------------|--------------|-----------------------|
| Throughput single-core | 1.5-2 Mpps | 15 Mpps | 12 Mpps (zero-copy) | 10-15 Mpps |
| Latency P99 | 20-30 µs | 2 µs | 5-10 µs | 5-8 µs |
| Hugepages required | No | Yes (2MB/1GB) | No (optional) | No |
| IOMMU + isolcpus required | No | Yes | No | No |
| Kernel features (iptables, tc, routing) | Yes | No (bypass) | Yes (for XDP_PASS) | Yes (replace iptables) |
| Deployment complexity | Low | Very High | Medium | Medium-High |
| K8s CNI integration | Native (kube-proxy iptables) | N/A (hypervisor only) | Native (Cilium, KubeArmor) | Production (Cilium) |
| Debug tools | iptables -L, conntrack -L | testpmd, rte_stats | bpftool, bpftrace, xdpdump | cilium debug, hubble |
| Main production use case | Traditional enterprise | NFV VNF, 5G UPF, HFT | DDoS mitigation, IDS (Suricata), custom LB (Katran) | K8s cloud native, replace kube-proxy + CNI |

**Khi nào dùng gì:**

- **Kernel + iptables**: Traditional workload < 5 Gbps/node, simple firewall + NAT.
- **OVS-DPDK**: Dedicated NFV workload > 25 Gbps, 5G UPF, Telecom.
- **XDP + AF_XDP**: Selective bypass — DDoS mitigation, IDS deep packet inspection, custom LB với XDP_TX.
- **Cilium eBPF**: K8s cloud native — replace kube-proxy + Calico iptables, identity-based NetworkPolicy, service mesh data plane.

### Cross-cut với Part 9.3 OVS-AFXDP

Part 9.3 đã trình bày OVS có thể chạy với AF_XDP datapath (`other_config:datapath-type=netdev` + `type=afxdp` cho port). Quan hệ:

- **OVS + kernel datapath** (default, Part 9.2): OVS.ko module, throughput ~2 Mpps/core.
- **OVS + DPDK datapath** (Part 9.3): `datapath-type=netdev` + `type=dpdk`, throughput ~15 Mpps/core.
- **OVS + AF_XDP datapath** (Part 9.3): `datapath-type=netdev` + `type=afxdp`, throughput ~10 Mpps/core.

OVS-AFXDP kết hợp OVS OpenFlow pipeline với AF_XDP zero-copy driver path. Ưu điểm: giữ OVS features (OpenFlow, conntrack via `ct()` action, OVN integration) + có performance hybrid (không cần DPDK complexity). Triển khai trong Red Hat OpenShift Container Platform 4.11+ làm alternative cho workload latency-sensitive.

### Cross-cut với Part 9.26 (revalidator storm)

Part 9.26 đã phân tích OVS kernel datapath có stale ukey leak. OVS-AFXDP có vấn đề tương tự không? **Có, nhưng ít phổ biến hơn** vì:

- OVS-AFXDP dùng `dpif-netdev` (userspace) thay `dpif-netlink` (kernel). Stale ukey vẫn có thể xảy ra nhưng revalidator pattern khác.
- Flow eviction trong `dpif-netdev` trực tiếp (no kernel round-trip), giảm delay.
- Debug tools khác: `ovs-appctl dpif-netdev/pmd-stats-show` thay `ovs-dpctl dump-flows`.

## 16.2.7 Điểm cốt lõi cần nhớ

**Sáu điểm cốt lõi:**

1. **AF_XDP 4 rings (FILL/RX/TX/COMPLETION) + UMEM**: shared memory zero-copy giữa userspace và kernel driver. Không memcpy, không chuyển đổi ngữ cảnh. Throughput 10-30 Mpps/core gần DPDK.
2. **3 attach mode XDP**: native (driver mode, fastest ~24 Mpps), SKB (generic, compatible but slow ~3-5 Mpps), hardware offload (Netronome/ConnectX-5+, zero CPU cost, hạn chế tập lệnh).
3. **4 return code**: `XDP_PASS` (tiếp tục kernel stack), `XDP_DROP` (drop ngay tại driver, DDoS mitigation), `XDP_REDIRECT` (chuyển tới AF_XDP socket hoặc NIC khác), `XDP_TX` (gửi ra cùng interface, load balancer).
4. **libbpf vs libxdp**: libbpf là low-level eBPF API general, libxdp là wrapper cao cấp hơn cho XDP specifically, hỗ trợ multi-prog + lifecycle management.
5. **eBPF map types**: PERCPU_ARRAY (lock-free counter), HASH (key-value general), LPM_TRIE (CIDR routing), XSKMAP (AF_XDP socket redirect).
6. **Production use case**: **Cilium** K8s CNI (replace kube-proxy), **Katran** Facebook LB, **Suricata** IDS AF_XDP capture, **OVS-AFXDP** hybrid OVS pipeline. XDP/AF_XDP đã mature cho cloud native workload.

### ▶ Capstone POE: DDoS mitigation 200 Gbps — XDP hay DPDK?

> **Trạng thái lab verification:** POE dựa trên publicly documented Cloudflare L4Drop + Facebook Shiv architecture + AWS Shield Advanced. Chờ user có setup production để verify reproducibility.

**Mục đích:** Đánh giá stack lựa chọn cho DDoS mitigation ở scale hyperscale, so sánh XDP vs DPDK.

**Scenario:** Organization Y là CDN cung cấp dịch vụ edge cho 500 customer. Attack traffic trong năm 2025 đã lên đến **200 Gbps UDP flood + 50 Mpps SYN flood**. Hiện tại dùng iptables + conntrack với throughput limit ~10 Gbps/node, không scale. CTO cần chọn stack mới cho anti-DDoS layer.

**3 phương án:**

1. **XDP + AF_XDP**: XDP drop tại driver cho attack pattern, AF_XDP redirect legitimate traffic lên userspace inspection. Pattern Cloudflare L4Drop.
2. **OVS-DPDK pipeline**: OVS-DPDK với custom ACL + rate limiting. Pattern Arbor Networks appliance.
3. **Dedicated hardware appliance**: Radware DefensePro hoặc Arbor APS. Vendor-specific, không open-source.

**Predict:** Đánh giá 3 phương án theo 5 tiêu chí: (a) throughput capability, (b) latency impact on legitimate traffic, (c) deployment cost, (d) scalability, (e) flexibility for new attack pattern.

**Observe (reasoning + data check):**

**Phương án 1 (XDP + AF_XDP):**
- Throughput: 40+ Gbps XDP_DROP per core × 8 core = 320 Gbps drop capacity. Met 200 Gbps requirement.
- Latency legitimate traffic: < 10 µs (XDP_PASS → kernel stack). Minimal impact.
- Deployment cost: Medium (eBPF programming skill + commodity hardware).
- Scalability: Horizontal scale qua thêm commodity server. Linear cost.
- Flexibility: High (viết custom XDP program cho attack mới trong vài giờ, deploy qua `bpftool attach`).

**Phương án 2 (OVS-DPDK):**
- Throughput: 15 Mpps/core × 4 core PMD = 60 Mpps packet processing. Met 50 Mpps SYN flood requirement.
- Latency: 2 µs (DPDK poll). Ngang hoặc tốt hơn XDP.
- Deployment cost: Very High (hugepages + IOMMU + isolcpus + OVS config complexity).
- Scalability: Horizontal nhưng expensive (DPDK yêu cầu dedicated core per port).
- Flexibility: Medium. OVS OpenFlow rules có thể cập nhật nhanh nhưng limited match field.

**Phương án 3 (dedicated hardware appliance):**
- Throughput: Vendor-specified (1-10 Tbps for top-tier). Over-provision.
- Latency: <1 µs (hardware accelerated).
- Deployment cost: **Very Very High** ($500K-$5M per appliance + recurring license).
- Scalability: Vertical (buy bigger appliance) — limited.
- Flexibility: Low (vendor-controlled, custom pattern phải chờ vendor update).

**Explain:** Recommendation:

**Primary: XDP + AF_XDP (phương án 1).** Lý do:

- Throughput 320 Gbps drop capacity trên commodity hardware — meeting 200 Gbps requirement với 60% headroom.
- Flexibility cao nhất: Cloudflare có ghi nhận viết custom XDP filter cho new attack pattern trong vài giờ, deploy tới edge POP fleet trong vài phút. Dedicated appliance mất weeks/months.
- Cost/performance ratio tốt nhất: commodity server + open-source toolchain (libbpf, libxdp, bpftool). Không recurring license cost.
- Production-proven: Cloudflare L4Drop (2018+), Facebook Shiv (XDP-based DDoS mitigation), AWS Shield Advanced (XDP-based tier) đều deploy thực tế ở hyperscale.

**KHÔNG chọn:**
- OVS-DPDK: Over-engineering cho anti-DDoS. OVS OpenFlow flexibility không needed cho drop-or-pass logic. DPDK complexity waste for 3× cost.
- Dedicated appliance: Over-priced. CAPEX $5M cho 1 site vs $100K commodity server × 50 site = $5M với better flexibility + distributed resilience.

**Falsification test:** Thử chạy anti-DDoS với iptables + conntrack baseline. Expected: fail at 30-40 Gbps limit (iptables O(n) rule scan + conntrack table explosion under SYN flood). Confirm iptables KHÔNG đủ cho hyperscale attack, XDP essential.

**Finish:** Nguyên tắc universal: **"Move packet drop to earliest possible point"**. Traditional iptables drop happens AFTER skb allocation + netfilter chain traversal. XDP drop happens BEFORE skb allocation. Difference: 10-100× CPU efficiency. Cùng principle áp dụng cho:

- **Database query**: filter early (WHERE trên index) thay vì filter late (sort all + filter).
- **API rate limiting**: reject trước authentication thay vì sau.
- **CDN caching**: serve static content tại edge thay vì origin.

XDP không chỉ là networking concept — là **general pattern "early rejection"** trong system design.

> **Timing source:** `sdn-onboard/labs/v3.13-R2.5.10-dhcp-arp-gap.timing`

> <script>` on `lab-openvswitch` (192.168.1.250). The orchestrator at
> `/tmp/v3.13-r2.5.10-orchestrator.py` is throwaway tooling, not committed.

A traditional Cisco engineer who has configured `ip helper-address` on
a router knows that a routed switch can act as a DHCP relay agent or
even as a DHCP server itself (`ip dhcp pool` syntax). The router's
ARP cache is also automatic; ARP responses are generated by the
router's own IP stack for IPs the router owns. Two services taken
for granted on a Cisco router: DHCP service and ARP responder.

OVS does NOT provide either. OVS is a layer-2 switch with rich
OpenFlow programmability; it has no built-in IP stack, no DHCP
state machine, no ARP cache for synthesised IPs. This is by design:
OVS focuses on the forwarding plane, leaving IP-layer services to
upstream components (kernel namespaces, application servers, OVN's
distributed controller).

This lab documents the gap by demonstrating both failure modes and
both workaround patterns. Phase A: a client with no DHCP server in
the broadcast domain has dhclient time out. Phase B: an external
dnsmasq process running in a server-side namespace fills the gap;
the client gets a lease through the standard DHCPDISCOVER/OFFER/
REQUEST/ACK exchange. Phase C: the OpenFlow flow-based ARP responder
pattern is described, and the lab attempts to install one but
encounters the empirical limitation that the `NXM_OF_ARP_SHA` field
the responder needs is not available in the default OpenFlow 1.0
match-field set; the operator must use OpenFlow 1.5 or higher.

## R2.5.10.1. Where this part sits in the curriculum and why

R2.5.10 is the tenth sub-capture of R2.5. The pattern shifts from
"compare with traditional implementation" to "document what OVS
does NOT have and how operators fill the gap". This is an important
pedagogical point: OVS is not a complete L3 device, and an engineer
migrating from Cisco IOS to OVS must understand the missing pieces
before the migration is feasible.

The two missing services are different in nature. DHCP is a
control-plane protocol (a multi-packet exchange between client and
server using UDP ports 67/68); the right place for it is in
userspace, and the canonical Linux implementation is `dnsmasq` (or
`isc-dhcp-server`). The lab uses dnsmasq because it is simpler and
ships with most Ubuntu installs. ARP, by contrast, is a one-packet
exchange (request/reply) and can be handled at the OpenFlow layer
with carefully crafted flow rules that match incoming ARP requests
and synthesise replies; this is the canonical "flow-based ARP
responder" pattern and is what OVN does internally for distributed
logical routers.

The R2.5.10 source anchors are limited because OVS does not
implement these services. The relevant code is in:

- `lib/ofp-actions.c`. The `move:` and `load:` action codecs that the
  flow-based ARP responder uses to rewrite ARP fields.
- `include/openflow/nicira-ext.h`. The NXM (Nicira Extended Match)
  field definitions including `NXM_OF_ARP_OP`, `NXM_OF_ARP_SHA`,
  `NXM_OF_ARP_SPA`, etc.
- `ofproto/ofproto-dpif-xlate.c`. The action set executor that
  applies the field rewrites at flow-translation time.

## R2.5.10.2. Topology

Two namespaces, one bridge. ns_client is the DHCP client; ns_dhcp
runs the DHCP server (dnsmasq). The OVS bridge connects them as a
flat L2 broadcast domain. No L3 router; the dnsmasq server is on
the same subnet as the client.

```ascii
   ns_client                              ns_dhcp
   (no IP initially)                      10.0.0.1/24
                                          dnsmasq listening
       |                                       |
   [veth_client]                         [veth_dhcp]
       |                                       |
   ----+---------------------------------------+----
       |                                       |
   veth_client_br                       veth_dhcp_br
       |                                       |
       +------------------+--------------------+
                          |
                          |   OVS bridge br0,
                          |   default standalone
                          |   (priority=0 actions=NORMAL)
                          +-------------------+
```

### ▶ Guided Exercise 1: Verify there is no built-in DHCP responder on OVS

This exercise builds the topology with no DHCP server and demonstrates
that dhclient times out.

**Outcomes.** dhclient sends two DHCPDISCOVER broadcasts, receives
no replies, exits with code 143 (SIGTERM from the wrapping `timeout
5`). The client interface remains without an IPv4 address.

**Before You Begin.** OVS 2.17.9 daemons running. dhclient available
(`isc-dhcp-client` package).

**Instructions.**

**1.** Build the topology.

```text

DB Schema 8.3.0
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-br br0
root@lab-openvswitch:~/network-onboard# ovs-vsctl set bridge br0 datapath_type=system

root@lab-openvswitch:~/network-onboard# ip netns add ns_dhcp
root@lab-openvswitch:~/network-onboard# ip link add veth_client type veth peer name veth_client_br
root@lab-openvswitch:~/network-onboard# ip link add veth_dhcp type veth peer name veth_dhcp_br
root@lab-openvswitch:~/network-onboard# ip link set veth_client netns ns_client
root@lab-openvswitch:~/network-onboard# ip link set veth_dhcp netns ns_dhcp
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br0 veth_client_br
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br0 veth_dhcp_br
root@lab-openvswitch:~/network-onboard# ip link set veth_client_br up
root@lab-openvswitch:~/network-onboard# ip link set veth_dhcp_br up
root@lab-openvswitch:~/network-onboard# ip -n ns_client link set lo up
root@lab-openvswitch:~/network-onboard# ip -n ns_client link set veth_client up
root@lab-openvswitch:~/network-onboard# ip -n ns_dhcp link set lo up
root@lab-openvswitch:~/network-onboard# ip -n ns_dhcp link set veth_dhcp up
root@lab-openvswitch:~/network-onboard# ip -n ns_dhcp addr add 10.0.0.1/24 dev veth_dhcp
```

ns_dhcp gets a static IP 10.0.0.1 because dnsmasq will need to bind
to it later. ns_client gets nothing for now; the goal of this
exercise is to demonstrate that dhclient cannot get an IP without a
server.

**2.** Confirm the bridge has no special configuration. The
`fail_mode` returns `[]` (empty), meaning the default standalone
mode where the bridge auto-installs a `priority=0,actions=NORMAL`
flow.

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl get bridge br0 fail_mode
[]
root@lab-openvswitch:~/network-onboard# ovs-ofctl dump-flows br0
 cookie=0x0, duration=5.504s, table=0, n_packets=4, n_bytes=356, priority=0 actions=NORMAL
```

One flow rule: the standalone NORMAL.

**3.** Try dhclient on the client. With no DHCP server in the
broadcast domain, dhclient repeatedly broadcasts DHCPDISCOVER until
the wrapping `timeout 5` kills it.

```text
root@lab-openvswitch:~/network-onboard# ip netns exec ns_client timeout 5 dhclient -v veth_client 2>&1 | head -20; echo dhclient_exit=$?
Internet Systems Consortium DHCP Client 4.4.1
Copyright 2004-2018 Internet Systems Consortium.
All rights reserved.
For info, please visit https://www.isc.org/software/dhcp/

Listening on LPF/veth_client/9e:b7:de:e6:1e:06
Sending on   LPF/veth_client/9e:b7:de:e6:1e:06
Sending on   Socket/fallback
DHCPDISCOVER on veth_client to 255.255.255.255 port 67 interval 3 (xid=0xef9c686d)
DHCPDISCOVER on veth_client to 255.255.255.255 port 67 interval 3 (xid=0xef9c686d)
Terminated
dhclient_exit=143
```

Two DHCPDISCOVER broadcasts in the 5-second window (the 3-second
interval per dhclient default plus the initial send). No reply.
`Terminated` plus `dhclient_exit=143` is SIGTERM (signal 15 = exit
code 128+15 = 143).

**4.** Confirm the client interface has no IPv4 address.

```text
root@lab-openvswitch:~/network-onboard# ip -n ns_client addr show veth_client
345: veth_client@if344: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 qdisc noqueue state UP group default qlen 1000
    link/ether 9e:b7:de:e6:1e:06 brd ff:ff:ff:ff:ff:ff link-netnsid 0
    inet6 fe80::9cb7:deff:fee6:1e06/64 scope link 
       valid_lft forever preferred_lft forever
```

Only the IPv6 link-local address (auto-configured by the kernel from
the MAC). No IPv4. The DHCP failure is empirically confirmed.

**Misconception to refute.** "OVS will catch the DHCP broadcasts and
respond automatically because it knows about IP addresses." Wrong.
OVS knows nothing about DHCP. The DHCP broadcasts are L2 frames
addressed to `ff:ff:ff:ff:ff:ff` with UDP port 67; OVS forwards them
to all bridge ports per the NORMAL action. With no DHCP server
listening on any port, the broadcasts go nowhere useful.

**Finish.** OVS has no built-in DHCP responder; this is the gap.

### ▶ Guided Exercise 2: Fill the DHCP gap with dnsmasq in a server-namespace

This exercise starts dnsmasq in ns_dhcp serving the 10.0.0.50-99
range. The client retries DHCP and gets a lease through the standard
four-step exchange (DISCOVER, OFFER, REQUEST, ACK).

**Outcomes.** dnsmasq starts and binds to veth_dhcp. Client DHCP
exchange completes; client gets 10.0.0.51, default route, DNS.
exit=0.

**Before You Begin.** Exercise 1 has completed.

**Instructions.**

**1.** Start dnsmasq in ns_dhcp. The args configure DHCP range
10.0.0.50-99, lease time 12h, default gateway (option 3) and DNS
(option 6) both at 10.0.0.1.

```text
root@lab-openvswitch:~/network-onboard# ip netns exec ns_dhcp dnsmasq --no-daemon --pid-file=/tmp/r2.5.10-dnsmasq.pid --interface=veth_dhcp --bind-interfaces --dhcp-range=10.0.0.50,10.0.0.99,12h --dhcp-option=3,10.0.0.1 --dhcp-option=6,10.0.0.1 --log-dhcp --log-facility=/tmp/r2.5.10-dnsmasq.log >/tmp/r2.5.10-dnsmasq.stdout 2>&1 &
```

The dnsmasq server logs match the client's view: DISCOVER received,
OFFER sent (10.0.0.51), REQUEST received, ACK sent. The transaction

interface so the tag is informational.

**Misconception to refute.** "DHCP works because OVS forwards UDP
broadcasts." Right but missing the point. OVS forwards the broadcasts;
the DHCP server inside ns_dhcp receives them and responds. OVS does
not understand DHCP semantics; it just sees UDP packets and forwards
them according to L2 rules. The DHCP behaviour is entirely a
property of dnsmasq plus the kernel UDP stack.

**Misconception to refute.** "dnsmasq must run on the bridge host's
main namespace." Wrong. dnsmasq runs in any namespace that has the
right L2 connectivity to the clients. Putting it in a dedicated
namespace gives clean isolation: the namespace's IP stack handles the
DHCP exchange without polluting the host's main routing table.

**Finish.** DHCP gap filled with external dnsmasq.

### ▶ Guided Exercise 3: Attempt the OpenFlow flow-based ARP responder, encounter the version limitation

The ARP-responder pattern uses OpenFlow's NXM (Nicira Extended Match)
fields plus `move:` and `load:` actions to synthesise an ARP reply
inside the bridge. The conceptual flow is:

1. Match: incoming ARP request (`arp,arp_op=1`) for the synthetic IP
   (`arp_tpa=10.0.0.99`).
2. Actions:
   - Set ARP opcode to 2 (reply).
   - Move the request's source MAC to the reply's destination MAC.
   - Set the reply's source MAC to the synthetic gateway MAC.
   - Move the request's source HW addr (sender MAC) to the reply's
     target HW addr.
   - Set the reply's source HW addr to the synthetic MAC.
   - Move the request's source IP to the reply's target IP.
   - Set the reply's source IP to the synthetic gateway IP.
   - Output back through the input port.

The lab will attempt to install this rule and observe the
empirical outcome.

**Outcomes.** The flow installation FAILS with `unknown field
'NXM_OF_ARP_SHA'`. This is an empirical revelation: the default OVS
OpenFlow protocol version (1.0) does not include all NXM ARP
extensions; the operator must explicitly request a higher OpenFlow
version (`-O OpenFlow15`) to access the ARP-related NXM fields.

**Before You Begin.** Exercise 2 has completed.

**Instructions.**

**1.** Tear down the dnsmasq server.

```text
root@lab-openvswitch:~/network-onboard# kill $(cat /tmp/r2.5.10-dnsmasq.pid 2>/dev/null) 2>/dev/null; pkill dnsmasq; sleep 1; true
[1]+  Done                    ip netns exec ns_dhcp dnsmasq --no-daemon --pid-file=/tmp/r2.5.10-dnsmasq.pid --interface=veth_dhcp --bind-interfaces --dhcp-range=10.0.0.50,10.0.0.99,12h --dhcp-option=3,10.0.0.1 --dhcp-option=6,10.0.0.1 --log-dhcp --log-facility=/tmp/r2.5.10-dnsmasq.log > /tmp/r2.5.10-dnsmasq.stdout 2>&1
```

**2.** Try to install the ARP responder flow with the default
OpenFlow version. The expected failure is `unknown field
'NXM_OF_ARP_SHA'`.

```text
root@lab-openvswitch:~/network-onboard# ovs-ofctl add-flow br0 'table=0,priority=200,arp,arp_op=1,arp_tpa=10.0.0.99,actions=load:0x2->NXM_OF_ARP_OP[],move:NXM_OF_ETH_SRC[]->NXM_OF_ETH_DST[],load:0x000abc999999->NXM_OF_ETH_SRC[],move:NXM_OF_ARP_SHA[]->NXM_OF_ARP_THA[],load:0x000abc999999->NXM_OF_ARP_SHA[],move:NXM_OF_ARP_SPA[]->NXM_OF_ARP_TPA[],load:0xa000063->NXM_OF_ARP_SPA[],in_port'
ovs-ofctl: NXM_OF_ARP_SHA[]->NXM_OF_ARP_THA[]: unknown field `NXM_OF_ARP_SHA'
```

The empirical revelation. Without specifying `-O OpenFlow15`,
ovs-ofctl uses OpenFlow 1.0 by default, which does not include the
ARP_SHA / ARP_THA / ARP_SPA / ARP_TPA NXM fields. The ARP_OP field
is recognised (because it appears in OF 1.0 as part of the ARP
match), but the per-field move/load operations on ARP source/target
hardware addresses require OpenFlow 1.5.

The fix in production: pass `-O OpenFlow15` to ovs-ofctl, or use the
newer OXM (OpenFlow Extensible Match) field names that are available
across OF versions.

**3.** Install the simpler default rule and confirm the flow table.

```text
root@lab-openvswitch:~/network-onboard# ovs-ofctl add-flow br0 'table=0,priority=100,actions=NORMAL'
root@lab-openvswitch:~/network-onboard# ovs-ofctl dump-flows br0
 cookie=0x0, duration=19.796s, table=0, n_packets=26, n_bytes=3561, priority=0 actions=NORMAL
 cookie=0x0, duration=0.302s, table=0, n_packets=0, n_bytes=0, priority=100 actions=NORMAL
```

Two NORMAL rules at different priorities; both functionally
equivalent for this test. The ARP-responder rule did not get
installed.

**Misconception to refute.** "ovs-ofctl always uses the highest OF
version the bridge supports." Wrong. ovs-ofctl defaults to OpenFlow
1.0 unless overridden with `-O <version>`. This is by design, for
backwards compatibility with older controllers. Production
deployments using NXM extensions or post-1.0 features must
explicitly request the higher version. The lab's flow-rule failure
is a concrete example of this default's consequence.

**Finish.** ARP responder pattern documented; the lab encountered
the OF-version limitation and notes the workaround.

## R2.5.10.3. The OVN production answer

The lab's two workarounds (external dnsmasq, OpenFlow flow-based ARP
responder) are operational patterns. The production-grade answer
within the OVS ecosystem is OVN (Open Virtual Network), which is a
distributed logical-network controller built on top of OVS. OVN
integrates DHCP, ARP, DNS, and IPv6 RA responders directly into its
logical-router abstraction; the operator does not configure these
services, OVN auto-installs the appropriate flow rules on every
chassis when a logical-switch port has the relevant address
configured.

R2.5.10 does not exercise OVN; OVN is covered in later parts of the
v3.13 plan (R3 onwards has S0-S10 source deep-dives that include the
OVN-controller component). The lab's two workarounds are sufficient
for understanding the OVS-only landscape, which is what R2.5
focuses on.

## R2.5.10.4. Side by side: Cisco IOS, OVS, OVN

| Concern                  | Cisco IOS                                                | OVS (no add-ons)                                          | OVN                                              |
|--------------------------|----------------------------------------------------------|-----------------------------------------------------------|--------------------------------------------------|
| Configure DHCP server    | `ip dhcp pool MYPOOL; network 10.0.0.0 /24; default-router 10.0.0.1` | (run external dnsmasq in a namespace)                | `ovn-nbctl set Logical_Switch_Port lsp1 dhcpv4_options=...` |
| Configure DHCP relay     | `ip helper-address`                                      | (use a packet-forwarding flow plus external server)       | (built-in distributed; OVN talks DHCP directly)  |
| Configure ARP responder  | (automatic on routed interface)                          | (install OpenFlow flow with NXM ARP rewrites; needs OF 1.5+) | (built-in; auto-installed for ports with IP)     |
| State storage location    | IOS process plus FIB                                    | (none in OVS; external state)                            | OVN northbound DB plus per-chassis flow rules    |
| Source-code anchor (DHCP) | n/a (closed)                                            | n/a (not implemented)                                     | `controller/pinctrl.c:pinctrl_handle_dhcpv4_request` (kernel-side packet-in handling) |
| Source-code anchor (ARP)  | n/a (closed)                                            | `lib/ofp-actions.c` (move/load codecs)                   | `northd/northd.c:build_lswitch_arp_nd_responder_known_ips` |

The takeaway: OVS deliberately punts on these services. The right
mental model is "OVS is a fast L2 forwarding plane plus OpenFlow
programmability; everything above L2 is somebody else's
responsibility". OVN is what makes OVS look like a complete L3
device; without OVN, the operator must use external services.

## R2.5.10.5. Common misconceptions, production assessment, source references

**Misconception 1.** "OVS supports DHCP." Wrong (in the standalone
sense). The kernel datapath forwards DHCP UDP broadcasts as ordinary
L2 frames; OVS does not intercept or terminate them. To answer DHCP,
an external server must be on a port belonging to the same broadcast
domain.

**Misconception 2.** "OVS can be configured to act as a DHCP server."
Wrong. OVS has no DHCP code. The closest thing is OpenFlow's
packet-in mechanism: install a flow that catches DHCP packets and
sends them to a controller, which then implements the DHCP state
machine in userspace. OVN does this internally.

**Misconception 3.** "ARP is automatic on Linux interfaces." Mostly
right but with a nuance. The kernel's IP stack on a Linux interface
auto-handles ARP for IPs assigned to that interface. But an OVS
bridge port has no IP of its own (the bridge's LOCAL port can have
one but the regular ports do not); OVS does not synthesise ARP
replies for arbitrary IPs unless the operator installs explicit flow
rules.

**Production-readiness:** for production OVS deployments without
OVN, the operator MUST plan for an external DHCP server (or
DHCP-relay configuration if upstream DHCP is available) and an ARP
responder mechanism. The operator's responsibility is to manage the
external services (uptime, configuration drift, log collection)
alongside the OVS bridge.

**Source code references:**

- `lib/ofp-actions.c`. The `move:` and `load:` action codecs (see
  `decode_REG_LOAD`, `decode_REG_MOVE`) used by the flow-based ARP
  responder.
- `include/openflow/nicira-ext.h`. The NXM field constants
  (`NXM_OF_ARP_OP`, `NXM_OF_ARP_SHA`, `NXM_OF_ARP_SPA`, etc.) plus
  their OpenFlow-version compatibility annotations.
- `ofproto/ofproto-dpif-xlate.c`. The `xlate_action_set` function

For a working flow-based ARP responder example, see
`Documentation/topics/ovs-extensions.rst` upstream and the OVN
source `northd/northd.c:build_lswitch_arp_nd_responder_known_ips`.

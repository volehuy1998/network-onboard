## 11.2.1 Flood-and-learn: vấn đề của VXLAN gốc

VXLAN RFC 7348 không định nghĩa control plane. VTEP phải:

1. Flood unknown unicast + broadcast qua multicast group.
2. Học MAC-to-VTEP binding từ source.
3. Age out entry sau timeout.

Vấn đề:

- **Multicast underlay requirement**: nhiều DC không deploy multicast.
- **Flood overhead**: broadcast domain lớn, replicated traffic.
- **Không scale**: MAC table mỗi VTEP chứa mọi MAC fabric.

---

## 11.2.2 BGP EVPN (RFC 7432, 02/2015)

Mở rộng BGP để advertise MAC/IP binding giữa VTEPs. Deterministic push-via-BGP thay flood-and-learn.

Kiến trúc:

- Mỗi VTEP chạy BGP speaker.
- VTEP learn local MAC → advertise qua BGP route EVPN.
- Remote VTEP cài đặt route: không cần flood.

BGP route EVPN dùng **SAFI 70 (EVPN)**, NLRI chứa Ethernet Segment Identifier, Ethernet Tag, MAC, IP, VTEP IP.

---

## 11.2.3 5 EVPN route types

*Table 11-2: EVPN route types (RFC 7432)*

| Type | Tên | Mục đích |
|---|---|---|
| 1 | Ethernet AutoDiscovery | Per-ES auto-discovery + per-EVI split horizon |
| 2 | MAC/IP Advertisement | Advertise MAC (và IP optional) đến remote VTEP |
| 3 | Inclusive Multicast | Advertise VTEP để join BUM replication group |
| 4 | Ethernet Segment | Multi-homing ES membership |
| 5 | IP Prefix Advertisement | Pure L3 (RFC 9136) |

**Type 2** quan trọng nhất, đa số triển khai chỉ Type 2 + Type 3.

## 11.2.4 VXLAN + EVPN workflow

**Bring-up**:

1. VTEP-A học MAC `aa:bb:cc:00:00:01` ở local port VLAN 10.
2. VTEP-A gửi BGP UPDATE Type 2: `{RD: VTEP-A-IP:10, MAC: aa:bb:..., IP: 10.0.0.1, VNI: 1000, VTEP: VTEP-A-IP}`.
3. Route reflector reflect đến mọi VTEP khác.
4. VTEP-B install: "MAC aa:bb:... via VXLAN to VTEP-A-IP, VNI 1000".
5. Packet từ host B đến host A: encap VXLAN VNI 1000, remote VTEP-A-IP.

Traffic flow pure unicast, không flood.

## 11.2.5 Leaf-spine DC fabric với EVPN

```
      ┌────────────┐  ┌────────────┐
      │  Spine 1   │  │  Spine 2   │  ← iBGP RR
      └──┬──────┬──┘  └──┬──────┬──┘
         │      │         │      │
    ┌────┴──┐ ┌─┴────┐ ┌──┴───┐ ┌──┴───┐
    │Leaf 1 │ │Leaf 2│ │Leaf 3│ │Leaf 4│  ← VTEPs
    └──┬────┘ └──┬───┘ └──┬───┘ └──┬───┘
     Hosts    Hosts    Hosts    Hosts
```

- Leaf = VTEP + iBGP peer với Spine RR.
- Underlay: BGP unnumbered hoặc OSPF.
- Overlay: VXLAN + EVPN.
- Spine: chỉ forward based on underlay, không có EVPN route.

Scale 1000+ leaves, stable convergence, không multicast dependency.

## 11.2.6 OVN vs BGP EVPN

OVN (Khối XIII) **không dùng BGP EVPN** trong control plane chính (NBDB → northd → SBDB). OVN có control plane riêng sync MAC/IP binding qua `Port_Binding` rows. Tuy nhiên, OVN modern (22.06+) thêm tích hợp BGP EVPN cho 2 use case cụ thể (xem §11.2.10).

- **OVN core**: cloud platform (OpenStack, K8s): tight integration với orchestrator. Control plane = NBDB ý đồ + northd compile + SBDB output.
- **BGP EVPN + VXLAN**: campus/enterprise fabric: multi-vendor support. Control plane = BGP route exchange.
- **OVN + BGP EVPN integration (22.06+)**: hybrid scenario (xem §11.2.10).

## 11.2.7 EVPN Type 2 NLRI byte-by-byte (RFC 7432 §7.2)

Type 2 (MAC/IP Advertisement) là route type quan trọng nhất, advertise MAC + IP binding tới remote VTEP.

### NLRI structure

```
+---------------------------------------+
| RD (8 byte)                           |  ← Route Distinguisher, định nghĩa per-VRF
+---------------------------------------+
| ESI (10 byte)                         |  ← Ethernet Segment Identifier (multi-homing)
+---------------------------------------+
| Ethernet Tag ID (4 byte)              |  ← VLAN ID hoặc 0 nếu single-tag
+---------------------------------------+
| MAC Address Length (1 byte) = 48      |
+---------------------------------------+
| MAC Address (6 byte)                  |  ← Host MAC
+---------------------------------------+
| IP Address Length (1 byte) = 0/32/128 |  ← 0 if no IP, 32 IPv4, 128 IPv6
+---------------------------------------+
| IP Address (0/4/16 byte)              |  ← IPv4/IPv6 (optional)
+---------------------------------------+
| MPLS Label1 (3 byte)                  |  ← VNI cho VXLAN-EVPN (RFC 8365)
+---------------------------------------+
| MPLS Label2 (3 byte, optional)        |  ← VNI L3 (cho symmetric IRB)
+---------------------------------------+
```

### Route Distinguisher (RD)

RD ngăn cách multi-tenant: mỗi VRF/EVI có RD unique. Format: `<ASN>:<unique-id>` hoặc `<router-id>:<unique-id>`. Thông thường auto-derived: `<VTEP-IP>:<VNI>`.

### Ethernet Segment Identifier (ESI)

ESI = 0 cho single-homed host (mặc định). Multi-homed host (active-active LAG sang 2 leaf) → ESI = 10-byte unique value cấu hình giống nhau trên 2 leaf.

### VNI trong MPLS Label1

RFC 8365 redefine MPLS Label field thành 24-bit VNI cho VXLAN. Encoding: 3 byte = VNI (24-bit). Khi mỗi VTEP nhận route, lookup VNI trong VRF table → biết encap VXLAN với VNI nào.

### Wire format example

VTEP-A (192.0.2.10) advertise MAC `aa:bb:cc:00:00:01` IP `10.0.0.1` cho VNI 1000:

```
RD: 192.0.2.10:1000          (8 byte: 00 14 c0 00 02 0a 03 e8)
ESI: 0x0...0                  (10 byte zero)
Ethernet Tag: 0               (4 byte: 00 00 00 00)
MAC Length: 48                (1 byte: 30)
MAC: aa:bb:cc:00:00:01        (6 byte)
IP Length: 32                 (1 byte: 20)
IP: 10.0.0.1                  (4 byte: 0a 00 00 01)
MPLS Label1: VNI 1000         (3 byte: 00 03 e8 hex)
```

Tổng NLRI = 33 byte. Plus BGP Path Attributes (next-hop = VTEP IP, route-target communities), tổng UPDATE message ~80-120 byte per route.

## 11.2.8 Type 3 + Type 4 + Type 5 deep

### Type 3, Inclusive Multicast Ethernet Tag

Mỗi VTEP advertise sự existence trên 1 EVI (Ethernet VPN Instance). Khi BUM (Broadcast/Unknown unicast/Multicast) traffic xuất hiện, VTEP nguồn lookup tất cả Type 3 route khớp EVI → tạo replication list, replicate unicast tới mỗi VTEP đích.

NLRI:

```
RD (8B) + Ethernet Tag (4B) + IP Address Length (1B=32) + Originator Router IP (4B)
```

Plus PMSI Tunnel Attribute định nghĩa replication mode:

- **Ingress Replication (IR)**: VTEP nguồn replicate unicast (default cho VXLAN-EVPN).
- **PIM-SM/PIM-SSM**: dùng underlay multicast group (yêu cầu underlay support PIM).

Hầu hết triển khai dùng IR vì underlay đơn giản hơn.

### Type 4, Ethernet Segment Route

Multi-homing 1 host vào 2 leaf qua LACP. ESI shared. Type 4 advertise membership trong ES.

Use case: leaf-2 đăng ký "tôi cũng thuộc ES X". Multi-leaf forwarding rules với split-horizon (không forward tới ES gốc) + DF (Designated Forwarder) election để chỉ 1 leaf forward BUM.

Phức tạp; thường chỉ enterprise lớn dùng. SMB/cloud DC thường single-homed (Type 1+2+3 đủ).

### Type 5, IP Prefix Advertisement (RFC 9136, 10/2021)

Advertise IP prefix (route) qua EVPN, thay vì host route Type 2. Use case: external prefix injection vào EVPN fabric.

```
RD (8B) + ESI (10B) + Ethernet Tag (4B) + IP Prefix Length (1B) + IP Prefix (variable) + GW IP (variable) + MPLS Label (3B)
```

Symmetric IRB (Integrated Routing and Bridging) dùng Type 5 cho L3 forwarding inter-subnet trong fabric.

## 11.2.9 Symmetric vs Asymmetric IRB

IRB là cơ chế route giữa 2 subnet trong cùng fabric (cross-VLAN routing). 2 mode chính:

### Asymmetric IRB

- Source leaf route + bridge.
- Dest leaf chỉ bridge.
- Mỗi leaf cần biết route của mọi VRF → memory consumption O(VRF × prefix).

Workflow:

1. Host A subnet-1 → leaf-1 (gateway của subnet-1).
2. Leaf-1 route: lookup destination 10.2.0.5 (subnet-2) → next-hop = host MAC C, encap VXLAN VNI subnet-2.
3. Leaf-3 nhận VXLAN VNI subnet-2 → bridge → host C.

Vấn đề: leaf-1 cần install mọi route subnet-2 trong VRF.

### Symmetric IRB (preferred)

- Source leaf route trong VRF locally + encap VXLAN với **L3 VNI** (transit VNI).
- Transit qua underlay với L3 VNI (Type 5 route).
- Dest leaf decap, lookup L3 VNI → route trong VRF dest → bridge cuối cùng.

Workflow (với 2 VNI):

1. Host A subnet-1 (VNI 1000) → leaf-1.
2. Leaf-1: route locally trong VRF "tenant", encap VXLAN với **L3 VNI** = 5000.
3. Leaf-3: decap VXLAN VNI 5000 → lookup VRF "tenant" → tìm subnet-2 → encap nội-leaf với VNI 2000 → bridge tới host C.

Memory: leaf chỉ cần host route của subnet local + L3 VNI route. Scale tốt hơn.

Ngày nay (2026) hầu hết triển khai dùng Symmetric IRB. Type 2 carry MPLS Label2 = L3 VNI cho symmetric mode.

## 11.2.10 OVN + BGP EVPN integration (22.06+)

OVN core không cần BGP EVPN cho intra-cluster. Tuy nhiên, 2 use case đòi hỏi tích hợp:

### Use case 1: External BGP gateway

OVN cluster cần advertise prefix tới external network (campus, WAN, Internet). OVN gateway router chassis chạy FRR + BGP, advertise prefix:

```
NB DB:
  Logical_Router (router1)
    options:dynamic-routing=true
    
ovn-northd compile thành:
  - Static route in NBDB → Logical_Flow trong SBDB
  - LRP advertise prefix qua BGP
```

OVN 22.06+ thêm BGP redistribute từ NBDB sang FRR trên gateway chassis. Internal control plane vẫn là OVN; chỉ external announce dùng BGP. Không phải BGP EVPN, chỉ regular BGP unicast với external peer.

### Use case 2: OVN-BGP-Agent (hybrid với external EVPN fabric)

[OVN BGP Agent](https://opendev.org/x/ovn-bgp-agent) là project Red Hat (2022+) cho phép OVN cluster tích hợp với BGP EVPN fabric:

- VM trong OVN cluster có thể được advertise dưới Type 2 EVPN route ra EVPN fabric.
- VM nhận traffic từ VTEP external (không phải OVN chassis) qua VXLAN-EVPN.
- Use case: hybrid cloud, OVN tenant cần kết nối tới legacy on-prem EVPN fabric.

Khác với OVN core: agent tự đọc OVN SBDB Port_Binding → tự generate BGP EVPN Type 2 advertisement. OVN chassis effectively trở thành VTEP biết nói EVPN.

> **Lưu ý:** OVN-BGP-Agent là community project, chưa upstream vào OVN core. Production triển khai cần đánh giá maturity. Khuyến nghị không dùng cho mission-critical mà chưa có lab validate.

## 11.2.11 Anatomy Template A: `vtysh -c "show bgp l2vpn evpn route"`

> **Bối cảnh khi chạy:** Chassis chạy FRR với BGP EVPN (use case 1 hoặc 2). Kiểm tra route nhận và advertise.

```
[chassis-A]# vtysh -c "show bgp l2vpn evpn route"
BGP table version is 152, local router ID is 192.0.2.10
   Network          Next Hop            Metric LocPrf Weight Path
Route Distinguisher: 192.0.2.20:1000
*>i[2]:[0]:[0]:[48]:[aa:bb:cc:00:00:02]:[32]:[10.0.0.2]
                    192.0.2.20             0    100      0 i
                    Extended Community: ET:8 RT:65000:1000
*>i[3]:[0]:[32]:[192.0.2.20]
                    192.0.2.20             0    100      0 i
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| Route prefix `[2]:...` | Type 2 routes for known MAC/IP | Type 2 missing | (n/a) | Remote VTEP chưa advertise; check BGP session |
| Route prefix `[3]:...` | Type 3 cho mỗi remote VTEP | Type 3 missing for VTEP | (n/a) | VTEP không tham gia EVI; check config |
| `Next Hop` IP | Match remote VTEP IP | Self-IP (loop) | Empty | BGP next-hop misconfig |
| `LocPrf` (Local Pref) | 100 default | < 100 (lower priority) | (n/a) | Path manipulation; check inbound policy |
| Path | `i` (iBGP) hoặc `e` (eBGP) | (n/a) | Empty path attr | BGP attribute corruption |
| `RT:<ASN>:<VNI>` (Route Target) | Match expected per VNI | Mismatch | Missing | Tenant isolation broken |
| `ET:8` (Encap Type 8) | 8 = VXLAN | 0 = MPLS | 11 = NVGRE | VTEP encap type mismatch |

**Kịch bản bẻ gãy:**

- Type 2 route count thấp hơn expected MAC count: BGP UPDATE bị filter inbound. Check `show bgp l2vpn evpn route detail` cho specific MAC.
- Route distinguisher conflict (2 VTEP cùng RD): tenant data leak. Đảm bảo RD per-VTEP unique.
- Encap Type mismatch (VTEP advertise VXLAN nhưng nhận MPLS-EVPN): config mode sai (RFC 8365 vs RFC 7432 native MPLS).

## 11.2.12 ▶ Capstone POE: BGP EVPN cho OVN cluster, có hay không?

> **Bối cảnh:** Cluster OVN production, 50 chassis, multi-tenant 100 VRF. Kỹ sư kiến trúc đề nghị thêm BGP EVPN làm control plane "để scale tốt hơn".

**Tuyên bố cần kiểm chứng:** "Thêm BGP EVPN cho OVN cluster sẽ scale tốt hơn flood-and-learn của OVN."

**Predict:** Sai. OVN không dùng flood-and-learn; OVN có control plane riêng (NBDB + northd + SBDB).

**Observe (giả định, kiểm chứng khi có lab):**

- Setup OVN cluster healthy, chỉ NBDB + SBDB control plane: 50 chassis, 10K LSP, 100 VRF. Convergence < 5 giây cho NB change. Memory ovn-controller per chassis ~200MB.
- Thêm FRR + OVN-BGP-Agent: thêm 100MB memory per chassis cho FRR + BGP EVPN cache. Latency cho external prefix advertise: 30-60 giây (BGP keepalive interval).
- Convergence cho Port_Binding migration cross-chassis: tương đương (cả 2 cùng dùng OVSDB cho intra-cluster). BGP EVPN không add value cho intra-cluster.

**Explain:**

1. **OVN không dùng flood-and-learn.** MAC/IP binding được CMS push qua NBDB `Logical_Switch_Port.addresses`, ovn-northd compile vào SBDB Port_Binding, ovn-controller cài flow trên br-int. Không có "learn from data plane" như VTEP truyền thống.
2. **BGP EVPN giải quyết flood-and-learn limitation,** không apply cho OVN. Adding BGP EVPN cho intra-cluster OVN là duplicating control plane → 2 source of truth, conflict-prone, complexity tăng.
3. **BGP EVPN có giá trị cho OVN ở 2 specific use case** (Phần 11.2.10): external advertise (FRR + regular BGP, không EVPN), và hybrid với external EVPN fabric (OVN-BGP-Agent). Cả 2 không phải replacement cho OVN core control plane.

**Falsification test:**

- Đo metric scale/latency 2 setup: OVN-only vs OVN + BGP EVPN intra-cluster.
- Nếu BGP EVPN cải thiện > 10% → có giá trị, ngược lại không.
- Trong thực tế, không có metric chỉ ra BGP EVPN intra-cluster cải thiện gì cho OVN.

**Tuyên bố sai ở đâu:** "Scale tốt hơn flood-and-learn" giả định OVN dùng flood-and-learn. Đúng phải: **OVN không có flood-and-learn intra-cluster. BGP EVPN add overhead không có benefit cho intra-cluster. Chỉ dùng BGP EVPN khi cần external integration (Use case 1 hoặc 2 trong §11.2.10).**

> **Quy tắc:** OVN core = NBDB+SBDB+northd, không cần BGP EVPN. BGP EVPN cho OVN = chỉ external/hybrid integration.

## Điểm cốt lõi cần nhớ

Thứ nhất, BGP EVPN giải quyết flood-and-learn limitation của VXLAN gốc. Deterministic control plane thay multicast dependency.

Thứ hai, 5 route types: Type 2 + Type 3 phổ biến nhất. Type 5 cho pure L3 VPN.

Thứ ba, modern DC leaf-spine dùng BGP EVPN làm standard. Scale 1000+ leaves.

Thứ tư, OVN không dùng BGP EVPN, có SBDB riêng. Lựa chọn phụ thuộc orchestrator integration vs multi-vendor.

## 3. Empirical observation

The 18.0 file alone holds 28 of 33 R1.A violations (85 percent). 18.0 was authored against `main` branch instead of `v22.03.8`. This is a systematic authoring bias, not random drift. R1.E (which audits 17.0 plus 19.0) likely shows the same pattern; pre-flight grep confirms 17.0 has 7 candidates and 19.0 has 12 candidates with the same mix of URL_OVN and PIPELINE_STAGE / TABLE_ID rows.

If the same `main`-vs-`v22.03.8` bias extends across Block 17 / 18 / 19, the R1.E error rate may exceed 50 percent, triggering the §3.7 bisection rule.

For Block 13 cite-light files (13.0, 13.4, 13.5b, 13.13), the bias is absent (the files don't cite GitHub URLs at all, except 13.4 which cites OVS at `branch-2.17` correctly). The 14 percent rate on 13.0 is in the WRONG_FACT category (free-standing version claims), not the WRONG_BRANCH_URL category.

This split supports a hypothesis: WRONG_BRANCH_URL is concentrated in files that explicitly cite GitHub URLs (Block 17 / 18 / 19); WRONG_FACT is distributed across all files that make version claims (Block 13 cite-light plus Block 17 / 18 / 19). R1.B onward audits should expect both patterns.

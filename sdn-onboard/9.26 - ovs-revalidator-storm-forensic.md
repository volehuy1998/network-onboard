## 9.26.1 Opening, the "keys 3612" story from the upstream commit body

In mid-2024, Roi Dayan (NVIDIA, Mellanox team) reported a strange observation on a production OVS cluster with hardware-offload TC flower enabled: the number of userspace ukeys was **513 times** larger than the actual datapath flow count. Verbatim from the commit body of `180ab2fd635e` (verified via `mcp__github__get_commit(openvswitch/ovs, 180ab2fd635e)`):

> It is observed in some environments that there are much more ukeys than actual DP flow. For example:
>
> ```
> $ ovs-appctl upcall/show
> system@ovs-system:
> flows : (current 7) (avg 6) (max 117) (limit 2125)
> offloaded flows : 525
> dump duration : 1063ms
> ufid enabled : true
>
> 23: (keys 3612)
> 24: (keys 3625)
> 25: (keys 3485)
> ```
>
> The revalidator threads are busy revalidating the stale ukeys leading to high CPU and long dump duration.
>
> Han Zhou, Roi Dayan, Eelco Chaudron, commit `180ab2fd635e`, 2024-08-29.

Read the output carefully: `flows: current 7` means only 7 real flows in the datapath, `offloaded flows: 525` means 525 flows have been offloaded to the hardware, **but the three handler threads 23, 24, and 25 hold 3612, 3625, and 3485 ukeys respectively**. The total of 10,722 ukeys against 7 active flows plus 525 offloaded flows comes out to **about 20 ghost ukeys per real flow**. The consequence: the revalidator thread burns CPU scanning ukeys that no longer map to any datapath flow, and `dump duration` jumps from a few milliseconds (the healthy baseline) to **1063 ms** (a full second per sweep cycle).

This is the "stale cache leak" incident class that Part 9.2 described at the theoretical level. Part 9.26 analyses the concrete case: why it happens, how to detect it early, the diagnostic CLI sequence, and the four-tier remediation. The part also distils a design lesson that applies to any cache module with asynchronous revalidation.

## 9.26.2 Refresher, datapath cache, upcall pipeline, revalidator thread

Before the forensic walkthrough, three concepts from Part 9.2 need to be pinned down clearly:

**(1) Datapath flow (DP flow):** a real entry in the kernel datapath (`openvswitch.ko`) or in the userspace datapath (`dpif-netdev`). Each entry holds a megaflow match (already wildcarded on unnecessary fields) plus an action list. Query with `ovs-dpctl dump-flows` or `ovs-appctl dpctl/dump-flows`. This is the low-tier cache, small in count (hundreds to thousands), exercised per packet.

**(2) User-space key (ukey):** an entry in the `struct umap` table of the ofproto-dpif-upcall module. Each ukey tracks one DP flow and stores metadata such as the match, the actions, the stats counters, and the revalidation state. A ukey is created when userspace "learns" a DP flow (via an upcall or a flow dump) and deleted when the DP flow no longer maps. Normal invariant: **one DP flow corresponds to one ukey**.

**(3) Revalidator thread:** a thread inside ovs-vswitchd that periodically sweeps the entire ukey map (every 500 ms by default) and, for each ukey, checks:
- Does the corresponding DP flow still exist? (dump from the kernel)
- Are the match and the actions still valid against the current OpenFlow table? (re-translate)
- If stale, delete the ukey and send a command to delete the DP flow.
- If the OpenFlow table changed, update the ukey and update the DP flow.

This process is the heart of the OVS "eventually consistent cache" design: userspace is not a real-time source of truth, but the revalidator guarantees the cache converges with the OpenFlow table within a small number of sweep cycles.

**Normal condition:** the number of ukeys is approximately equal to the number of DP flows, plus or minus a few entries during transitions. Dump duration is around a few milliseconds. Revalidator CPU sits at 1 to 5 percent of a single core.

**When leaking:** the ukey count is much larger than the DP-flow count, dump duration grows linearly with the ukey count, and revalidator CPU saturates. This is the case study scenario.

## 9.26.4 Evidence collection, five essential diagnostic commands

This is the standard runbook when a revalidator storm is suspected (dump duration above 100 ms and a CPU spike on ovs-vswitchd):

**Command 1: read overall datapath health.**

```bash
[host]# ovs-appctl upcall/show
```

Output format (verified from commit body `180ab2fd635e`):

```
system@ovs-system:
flows : (current <N>) (avg <N>) (max <N>) (limit <M>)
offloaded flows : <N>
dump duration : <T>ms
ufid enabled : true

<handler_id>: (keys <N>)
```

**How to read the output:**

- `current <N>`: the current DP-flow count. Healthy values range from tens to thousands depending on the workload.
- `limit <M>`: the cap set by `other_config:flow-limit`. The default is 200,000 on OVS 2.17 and later, and can be adjusted.
- `offloaded flows`: flows already offloaded to the hardware (only when TC flower or DPDK rte_flow is enabled). Normally at most 80 percent of `current`.
- `dump duration`: the time for one sweep cycle. Healthy values stay under 100 ms. **Above 500 ms is a warning, above 1000 ms is critical.**
- `ufid enabled`: the UFID is a flow-independent identifier. Normally `true` since OVS 2.7.
- `<handler_id>: (keys <N>)`: the ukey count per handler thread. Healthy: the sum of `keys` is approximately equal to `current` plus `offloaded`. **If the sum of `keys` is much larger than `current` plus `offloaded`, this is a stale leak.**

**Command 2: ovs-vswitchd internal counters.**

```bash
[host]# ovs-appctl coverage/show | grep -E "upcall|revalidator|ukey|dump"
```

Key counters (verify each one in detail with `ovs-appctl coverage/read-counter <name>`):

- `upcall_ukey_created`: number of newly created ukeys. A sustained high delta means high flow churn.
- `upcall_ukey_replace`: number of ukey replacements (matching an existing entry). A high delta means continuous flow migration.
- `handler_duplicate_upcall`: upcalls for flows that already have a ukey. A high delta means a race between handlers.
- `revalidator_sweep_time`: total sweep time. Read it via the Prometheus scrape from ovs_exporter.
- `revalidator_purge_stale`: number of ukeys purged because they were stale. Spikes during remediation.
- `dump_flows_fallback`: flow-dump fallback path. Rises when the hardware offload dump fails.

**Command 3: count actual flows in the kernel.**

```bash
[host]# ovs-dpctl dump-flows | wc -l
```

Compare this with the sum of `keys` from `upcall/show`. If the ratio of `keys` to DP flows is greater than 2, suspect a stale leak.

**Command 4: feature capability check.**

```bash
[host]# ovs-appctl dpif/show-dp-features <dp_name>
```

Lists 16 or more capabilities (`masked_set_action`, `tnl_push_pop`, `ufid`, `sample_nesting`, and so on). Not directly related to the revalidator, but it confirms the datapath version and the feature alignment with userspace expectations.

**Command 5: detailed ukey-to-flow mapping.**

```bash
[host]# ovs-appctl upcall/dump-ukeys
```

The output is large (potentially several megabytes). Only run it when a deep dive is needed, and grep by UFID to track a specific flow. Do not use this command for automated alerting.

## 9.26.5 Root cause analysis, three hypotheses tested by elimination

The on-call engineer at T+6h can raise three hypotheses after seeing `keys 3612`:

**Hypothesis H1: OpenFlow rule explosion.**

Hypothesis: the controller installed too many OpenFlow rules and produced too many megaflows wildcarding every 5-tuple.

**Test:** `ovs-ofctl dump-flows br0 | wc -l`. Healthy is below 1000. Above 10,000 makes H1 plausible.

**Result for this case:** 127 OpenFlow rules. H1 is **rejected**.

**Hypothesis H2: a memory leak in ovs-vswitchd.**

Hypothesis: a malloc leak makes ovs-vswitchd consume memory, triggers an OOM kill, then a restart, which wipes the flows.

**Test:** `ps aux | grep ovs-vswitchd | awk '{print $4"% "$6/1024"MB"}'` and check `journalctl -u openvswitch-switch` for OOM events. Run `malloc_stats` via gdb attach if necessary.

**Result for this case:** ovs-vswitchd is at 420 MB versus a 380 MB baseline, with no OOM log entries. H2 is **rejected**.

**Hypothesis H3: stale ukey leak (confirmed).**

Hypothesis: ukeys are created faster than the revalidator deletes them, leading to unbounded growth. Possible root causes:

- **H3a:** hardware offload path rotation: flows migrate kernel to NIC and back continuously. Each cycle creates a new ukey while the old ukey is not deleted in time, because the kernel dump reports `flow not found`.
- **H3b:** per-CPU upcall dispatch packet reordering (see commit `0d9dc8e9`, FDP-1479): the same flow is handled by multiple handler threads, producing multiple ukeys for the same UFID.

**Test:** read the delta of `upcall_ukey_created` versus `revalidator_purge_stale` over a one-minute window. If `created` minus `purge` is much greater than 0, the leak direction is confirmed.

**Result for this case:** 530 created per minute, 12 purged per minute, for a delta of plus 518 per minute. H3 is **confirmed**. The pattern matches the scenario described in commit body `180ab2fd635e`.

## 9.26.6 Mechanism deep-dive, why stale ukeys persist

To understand the fix, read the revalidator logic before patch `180ab2fd635e` carefully. In function `revalidator_sweep` of `ofproto/ofproto-dpif-upcall.c`:

1. **The revalidator obtains the list of ukeys from the umap**, totalling N ukeys.
2. **It dumps the DP flows from the kernel**, totalling M DP flows.
3. **It matches each ukey against a DP flow by UFID.** If a match is found, set `ukey->dump_seq = current_seq`. If no match is found, the ukey keeps the old `dump_seq`.
4. **At sweep end, delete ukeys whose `dump_seq` is too old (more than two sweeps behind)**, but only if the dump succeeded.

**The hole:** if the kernel dump does not include a particular DP flow (for example, the hardware offload flow dump temporarily reports `not found`), the ukey is not refreshed. But if a later sweep does include that flow again (the hardware offload path changed state), the ukey can be refreshed before it crosses the deletion threshold. The result is a "zombie" ukey: the dump does not see the actual flow, but the ukey is never stale enough to be deleted.

**Fix in commit `180ab2fd635e`:** add a `missed_dumps` counter inside the ukey struct. Every sweep where the dump does not see the flow increments the counter. After four consecutive missed sweeps, the ukey is force-deleted without waiting for the `dump_seq` threshold.

Simplified code change (excerpt from the commit diff, pattern verified):

```c
/* Before patch */
if (ukey->dump_seq < revalidator->dump_seq - 2) {
    ukey_delete(ukey);
}

/* After patch 180ab2fd635e */
if (ukey->dump_seq < revalidator->dump_seq) {
    ukey->missed_dumps++;
    if (ukey->missed_dumps >= 4) {
        ukey_delete(ukey);
    }
}
```

The fix is simple at the code level (one counter and a threshold), but it **resolves the root of the class issue**: the cache eventually converges within four sweeps even when the kernel dump is not 100 percent reliable.

**Consequence.** OVS releases earlier than v3.5 carry the risk of this leak, because commit `180ab2fd635e` is not yet included in those releases. The output of `git tag --contains 180ab2fd635e` shows the first occurrence is v3.5.0, dated 2025-02-21 (verified 2026-04-28). The OVS 2.17 baseline used by the curriculum does not have this fix. OVS 3.3 and 3.4 also lack the fix. To remove the risk, an operator must backport the patch manually or upgrade to v3.5 or later.

## 9.26.7 Four-tier remediation, immediate to long-term

**Tier 1: immediate (SEV-2 recovery, a few minutes).**

```bash
# Force purge of every stale ukey.
[host]# ovs-appctl revalidator/purge

# Verify that dump duration drops back below 100 ms.
[host]# ovs-appctl upcall/show | grep -i duration
```

The `revalidator/purge` command does not restart ovs-vswitchd; it only clears the ukey map. The actual datapath flows are not affected and no packets are dropped. The command is safe for production.

**Tier 2: short-term (24 hours, prevent short-term recurrence).**

```bash
# Lower flow-limit to reduce revalidator pressure.
[host]# ovs-vsctl set open_vswitch . other_config:flow-limit=100000

# Verify.
[host]# ovs-appctl upcall/show | grep limit
```

A smaller `flow-limit` means a smaller cache cap, which means the revalidator sweep is faster. Trade-off: if the workload genuinely needs many flows, the upcall rate rises (the slow-path load increases). Adjust based on a real benchmark.

**Tier 3: medium-term (one to two weeks, root-cause fix).**

Upgrade to an OVS version that contains commit `180ab2fd635e`:

- The fix is merged by default in OVS 3.5 and later (see the consequence note in §9.26.6).
- For OVS 2.17 LTS, verify the backport through your distribution (the Red Hat OVS package or the Ubuntu Cloud Archive) or build from source with the patch applied.
- Check commit `464bc6f9` (November 2024) which fixes a MemorySanitizer bug in patch 180ab2fd635e. Both commits are needed for stability.

```bash
# Verify the version and the commit.
[host]# ovs-vswitchd --version
# ovs-vswitchd (Open vSwitch) 3.5.0
[host]# ovs-appctl version
```

**Tier 4: long-term (one to three months, monitoring and alerting).**

Deploy a Prometheus and Grafana dashboard:

```yaml
# Prometheus alert rule.
- alert: OVSDumpDurationHigh
  expr: ovs_upcall_dump_duration_ms > 500
  for: 5m
  labels:
    severity: warning
  annotations:
    summary: "OVS revalidator dump duration above 500 ms on {{ $labels.instance }}"
    description: "Possible stale ukey leak. Run `ovs-appctl upcall/show` and `upcall/dump-ukeys | wc -l`, then compare with the DP-flow count."

- alert: OVSUkeyRatioHigh
  expr: (sum(ovs_upcall_keys) / ovs_upcall_flows_current) > 5
  for: 10m
  labels:
    severity: critical
  annotations:
    summary: "OVS ukey-to-flow ratio above 5 (stale leak suspected)"
```

A four-panel dashboard:

1. **DP-flow count.** Time series of `ovs_upcall_flows_current` and `offloaded`.
2. **Ukey count per handler.** Stacked time series of `ovs_upcall_keys{handler="<id>"}`.
3. **Dump duration.** Line chart of `ovs_upcall_dump_duration_ms` with thresholds at 100, 500, and 1000 ms.
4. **Revalidator CPU.** The `rate` of `process_cpu_seconds_total{job="ovs-vswitchd"}`.

The runbook points back to Part 9.14 (the incident response decision tree) for the end-to-end workflow when an alert fires.

## 9.26.8 Comparison with OVN, do logical flows share the same vulnerability?

OVN compiles Northbound intent into Logical Flows in the SB DB, then into OpenFlow rules, then into DP flows. The final stage still goes through the same OVS datapath cache pipeline. Question: does an OVN workload carry the same stale ukey leak risk?

**Short answer:** yes, but the risk is lower for two reasons.

**(1) Logical flow cardinality is low.** OVN logical flows match primarily on `reg` fields (port_key, dp_key) rather than the full 5-tuple. The register space is small (a 16-bit port_key). The maximum megaflow cardinality is bounded by `num_lsp` times `num_stage` rather than by `num_flow` times 2^32 source IPs times 2^16 source ports and so on. In practice, a cluster with 1000 LSPs, 50 stages, and a few hundred tenants comes out to a few hundred thousand unique match patterns. This is much smaller than the scanner scenario.

**(2) The hardware offload pattern is stable.** OVN logical flows have long lifetimes (they are tied to a Port_Binding). Hardware offload flows rotate rarely. This reduces the trigger for H3a.

**OVN is not immune, however:**

- If the ACL workload churns heavily (for example, continuous Kubernetes Network Policy updates), logical flows rebuild, DP flows rotate, and ukeys leak.
- OVN 24.03 and later use Port_Group and Address_Set to reduce churn (policy consolidation). This is recommended for clusters with more than 1000 pods.

**OVN-specific runbook checks:**

```bash
[host]# ovn-appctl -t ovn-controller recompute  # Force a full I-P engine recompute.
[host]# ovs-appctl upcall/show  # Check the ukey count after the recompute.
```

If the ukey count drops sharply after the recompute, the leak may come from stale OpenFlow installed by OVN. If it does not drop, the leak sits in the pure OVS layer and tiers 1 through 4 of §9.26.7 apply.

## 9.26.10 Core takeaways and References

**Six core points to remember:**

1. **The datapath cache is not the source of truth.** The userspace ukey table eventually converges with the kernel DP flow set through the revalidator sweep. The rule of thumb: the ukey count is approximately equal to the DP-flow count plus the offloaded count, give or take a few entries.
2. **Dump duration is the health metric of first resort.** Above 500 ms is a warning, above 1000 ms is critical. The metric reflects revalidator CPU load.
3. **A stale ukey leak is a class issue,** not a bug on a specific flow. It is a create-and-replace loop that does not converge. Commit `180ab2fd635e` adds the `missed_dumps` counter as the root-cause fix.
4. **Hardware offload path rotation is the main trigger.** TC flower with a ConnectX NIC is the common case; DPDK rte_flow also exhibits the issue through a different mechanism.
5. **Four-tier remediation:** immediate `revalidator/purge`, then `flow-limit` tuning, then upgrade to a release that contains commit `180ab2fd635e`, then monitoring. Do not skip a tier.
6. **Verify every source code claim through MCP GitHub** (Rule 14): commit SHAs of at least eight characters, function names via search_code with a get_file_contents fallback, and file paths at the version baseline.

## 9.26.11 Case Study 2, an LACP bond-flap cascade triggers a megaflow invalidation storm

Case 1 focuses on a cache-sync defect (the ukey leak). Case 2 looks at a different angle: a **bond state change** triggers cascading megaflow re-installation. This failure mode is more common in production, because bond plus LACP is the foundation of every multi-uplink production deployment, while hardware offload with TC flower is still rare at sites without Mellanox or NVIDIA NICs.

### 9.26.11.1 Drama, ToR firmware upgrade on a Tuesday

A production OVS 2.17 cluster runs 200 hypervisors using `bond_mode=balance-tcp` LACP aggregation to two ToR switches with 8x10G uplinks. On Tuesday at 14:30 local time, the network team scheduled a firmware upgrade on ToR A (the redundant pair). Applying the firmware on ToR A imposes about a 12-second LACPDU outage within the reload window.

**Pre-upgrade state.** 200 chassis times 8 bond slaves equals 1600 LACP sessions UP. Baseline OVS health: `upcall/show` reports `flows: 4800`, `dump duration: 18 ms`, and ovs-vswitchd CPU at 2 to 4 percent of a single core.

**14:32 UTC, ToR A firmware starts.** Four slaves per chassis lose LACP keepalive (the dedicated ToR A uplinks). The default `lacp-time=fast` gives a 1-second interval times a 3-count timeout, which detects in 3 seconds.

**14:32:03, bond detection.** OVS on all 200 chassis simultaneously marks the four slaves DOWN. The bond state moves to "4 active slaves" (the ToR B path only). `ovs-appctl bond/show` now reports 4 of 8 slaves up.

**14:32:03 to 14:32:15, megaflow mask invalidation.** OVS `bond_mode=balance-tcp` hashes the 5-tuple to a slave. A slave change means the flow-to-slave mapping changes for 50 percent of the flows that hash to the ToR A path. OVS invalidates every megaflow whose action set carries `bond_port=X` for X in the ToR A slave set.

**14:32:03 to 14:32:90, revalidator and upcall storm.** 200 chassis times 4800 flows is roughly 960,000 megaflows that simultaneously need re-installation. The cluster-wide upcall rate climbs from a 50 pps baseline to a 250,000 pps peak. Dump duration jumps from 18 ms to 340 ms.

**14:33:00, VM latency cascade.** P99 latency for cross-chassis traffic rises from 0.8 ms to 180 ms. ICMP packet loss spikes from 0 percent to 12 percent for 45 seconds. Connection resets on stateful applications (MySQL replication, Redis cluster) fire false-positive health checks, which trigger an application-level failover cascade.

**14:34:00, SEV-2 declared.** The on-call page goes out, and 200 chassis show red in Grafana.

**14:36:00, ToR A firmware complete, LACPDU resumes.** The bond on each chassis sees the four slaves UP again, but OVS does not auto rebalance. New flows hash evenly across all eight slaves again, while old flows continue to stick to the four currently UP slaves until they expire. A **second wave of invalidation** happens when the revalidator re-installs flows for the eight-slave aggregation.

**14:37:30, the second wave subsides.** Dump duration falls to 45 ms (still elevated) and the upcall rate falls to 8000 pps.

**14:42:00, baseline recovery.** Dump duration is 22 ms, upcall rate is 60 pps, CPU is 3 percent.

**Total blast radius.** 10 minutes of elevated latency, 90 seconds of critical packet loss, and 25 SEV-3 downstream alerts from the application teams. The incident ticket runs 3 hours of triage plus a post-incident report.

- **Several buckets steadily report `hash N: 0 kB load`.** The hash function is skewed (traffic concentrates on one or two buckets). Rebalance or change `bond-hash-basis`.
- **`next rebalance` repeatedly drops to 0 ms.** Forced rebalance, usually from slave flapping. Correlate with the `bond_rebalance` coverage counter.

### 9.26.11.3 Root cause analysis, three hypotheses

**H1: STP reconvergence loops the network.** An LACP flap can be triggered by an STP topology change when link costs update. **Test:** run `show spanning-tree` on the switch and check the TCN count. **Result:** zero TCNs within the window. **Rejected.**

**H2: OVS bond hash function bug.** A known bug existed in OVS 2.11 with balance-tcp recirculation. **Test:** run `ovs-vswitchd --version` and cross-reference the commit history. **Result:** OVS 2.17.9 with no known bug. **Rejected.**

**H3 confirmed: the ToR firmware upgrade causes an LACPDU timeout.** LACP `fast` time equals 1 s times the 3-count timeout threshold, totalling 3 s. The ToR firmware reload loses LACPDUs for 12 s, which is greater than the 3 s threshold, so OVS marks the slaves DOWN. **Test:** correlate the `ovs-appctl coverage/show bond_reconfigure` delta with the switch uptime metric. **Result:** a spike of `bond_reconfigure +4` at 14:32:03, matching the four slaves that lost keepalives. **Confirmed.**

### 9.26.11.4 Mechanism deep-dive, why a bond state change causes a megaflow storm

In the OVS source file `lib/bond.c`, function `bond_update_post_recirc_rules()`, the bond uses one of two hash modes:

**Recirculation-based (`balance-tcp`).** A packet enters the bond, the 5-tuple hashes to set the `dp_hash` register, the packet recirculates, the table lookup matches on `dp_hash`, and the output slave is selected. Advantage: precise L4 hashing. Disadvantage: each megaflow must carry a mask that includes `dp_hash`, which broadens the fingerprint.

**Simple hash (`balance-slb`).** The packet hashes a MAC pair and goes directly to the output slave without recirculation. The mask is smaller.

When a bond slave state changes in balance-tcp mode:

1. `bond_reconfigure()` is called, rebuilding the hash-bucket-to-slave mapping.
2. Every megaflow with `recirc_id=<bond_recirc>` is marked invalid on the next revalidator sweep.
3. The revalidator's flow dump shows the flow exists in the kernel but the mapping is invalid, so it uninstalls the flow.
4. The next packet for that flow triggers an upcall, and a new megaflow is installed with the updated bucket mapping.

**Cost.** Order of N_flow times N_chassis at the cluster level. With 200 chassis times 4800 average flows, the burst is 960,000 re-installations. Even at about 2 ms per re-install, the total is 960,000 times 2 ms divided by 48 handler threads, which is 40 seconds of wall-clock time.

Potential fixes:
- Single-slave failure: invalidate only the flow map that traverses that slave (about 1 over N_slave flows). OVS 3.1 and later carry this optimisation.
- Simultaneous multi-slave failure: a large proportion of flows still requires re-installation.

### 9.26.11.5 Four-tier remediation

**Tier 1 (immediate, under 5 minutes).**

No emergency action fixes the situation after it has already happened. Wait for the revalidator to catch up (2 to 5 minutes depending on scale). Avoid actions that can make things worse: do not restart ovs-vswitchd (this triggers another wave of invalidation) and do not force-purge flows.

**Tier 2 (short-term, within a few hours).**

```bash
# Lower LACP hysteresis so a short network event does not flap the OVS bond.
[host]# ovs-vsctl set port bond0 other_config:lacp-time=slow
# slow = 30 s interval times 3 = 90 s detection (versus fast at 3 s).
# Trade-off: flap detection is 30 times slower.
```

**Tier 3 (medium-term, one to two weeks).**

Upgrade from OVS 2.17 to 3.1 or later, which has the `bond_update_single_slave()` optimisation that invalidates only a subset of megaflows when one slave fails instead of doing a full rebuild.

Coordinate with the network team: schedule firmware upgrades inside a maintenance window, set `lacp-time=slow` before the upgrade, and revert afterwards. Window procedure:

```bash
# Pre-upgrade (15 minutes before the ToR reload).
[batch-200-chassis]# ovs-vsctl set port bond0 other_config:lacp-time=slow

# ToR firmware upgrade (12-minute window).
# ... vendor command ...

# Post-upgrade.
[batch-200-chassis]# ovs-vsctl set port bond0 other_config:lacp-time=fast
```

**Tier 4 (long-term, monitoring).**

```yaml
# Prometheus alert

  expr: rate(ovs_coverage_bond_reconfigure[5m]) > 0.1
  for: 2m
```

### 9.26.11.6 Comparison with OVN, same vulnerability?

The OVN overlay runs Geneve or VXLAN on top of the bond. The tunnel output action is `set_tunnel(...)` plus a single output port (the bond master), not a per-slave action. Therefore:

**OVN reduces exposure.** OVN megaflows mask the tunnel outer header and do not include the bond slave choice. A bond slave change does not invalidate OVN logical flows. Only the underlay (the physical NIC bond) is affected.

**OVN is not immune, however.** Underlay traffic (for example, BGP to FRR on the host network) still traverses an OVS bridge if one exists. In that case, a bond flap still causes megaflow invalidation for the underlay flows.

OVN-specific runbook:

```bash
# Check that the overlay tunnel stays UP after the bond flap.
[host]# ovs-appctl tnl/neigh/show
# If the tunnel neighbour table is intact, the OVN overlay is not disrupted.

# Check the OpenFlow rules on br-int (managed by OVN).
[host]# ovs-ofctl dump-aggregate br-int
# Compare pre versus post flap: the flow count should stay stable if the OVN overlay is isolated.
```

### 9.26.11.7 Case 2 takeaways

**One.** A bond state change costs order N_flow megaflow invalidations in balance-tcp mode. The cost scales linearly with the workload and can cause a cluster-wide outage.

**Two.** LACP timing is the network-to-OVS contract. `lacp-time=fast` (3-second detection) is too aggressive for maintenance windows; set `slow` (90 seconds) for planned changes.

**Three.** Correlate the bond event with the upcall rate. The signature pattern is "`bond_reconfigure` spike, then upcall storm, then VM latency".

## 9.26.12 Case Study 3, conntrack zone collision during cross-chassis migration

Case 3 sits at the scope of a multi-tenant OVN hypervisor with stateful ACLs. A zone is a 16-bit conntrack namespace; OVS assigns one zone per LSP to isolate state across tenants. A collision during a migration window leads to state cross-talk that surfaces as "random connection reset".

### 9.26.12.1 Drama, a tenant migration into a full chassis-B

An OVN 22.03 cluster has 50 hypervisor chassis, each hosting 200 to 400 LSPs. `ovs-appctl ct-zones/list` on an average chassis shows 600 zones assigned (LS and LR zones overlap). Zone space is 16-bit (max 65535), so the cluster-wide limit is not yet a problem, but per-chassis density can be high.

**Context.** Tenant T1 (an enterprise customer, production workload) runs on chassis-B with 50 LSPs in logical switch `ls_t1`. The zone assignment for the T1 LSPs covers the range 1000 to 1049.

**Scheduled action.** The ops team migrates tenant T2 (staging) from chassis-A to chassis-B to free chassis-A for maintenance. T2 has 30 LSPs in `ls_t2`.

**15:00 UTC, migration start.** The CMS issues unbind for the T2 LSPs on chassis-A and bind on chassis-B. The LSPs go through Port_Binding updates: ovn-controller on chassis-A releases the zone, ovn-controller on chassis-B requests a new one.

**15:00:05, partial race.** The binding update for LSP `vm_t2_01` reaches chassis-B before ovn-controller on chassis-B receives the release notification from chassis-A. ovn-controller on chassis-B allocates a zone for `vm_t2_01` from the available pool. The pool algorithm picks **zone 1012**, which happens to be in use by T1 LSP `vm_t1_13` on chassis-B.

**15:00:10, collision active.** Traffic from `vm_t2_01` enters br-int. The `ls_in_pre_stateful` stage fires `ct(commit, zone=reg13[0..15])` with `reg13=1012`. The conntrack lookup in zone 1012 finds existing state for a `vm_t1_13` session (a database query to a MySQL backend). The state is ESTABLISHED with a different 5-tuple but the same zone, so the `vm_t2_01` packet matches as `INVALID` and is dropped.

**15:00:15, symptom reported.** T2 users report "intermittent SSH session disconnects". The issue is not reproducible with ping (ICMP does not go through conntrack); only TCP sessions are affected.

**15:02:00, cross-tenant anomaly.** T1 users see "MySQL replication lag spike". T1 MySQL connections also hit zone 1012, the state lookup returns a T2 record, the protocol mismatches, and the connection is RST.

**15:05:00, SEV-3 declared** (dual-tenant impact, compliance flag).

**15:18:00, root cause identified.** An engineer runs `ct-zone-list` and sees zone 1012 mapped to both `vm_t1_13` and `vm_t2_01`.

**15:20:00, immediate fix.** Restart ovn-controller on chassis-B to force zone re-assignment. The new zone range for T2 is 1500 to 1529 (the pool avoids 1000 to 1099 used by T1).

**15:25:00, recovery.** Flush conntrack zone 1012 and re-establish connections. T1 and T2 are healthy again.

**Total impact.** 25 minutes of degraded service, two tenants affected, and a compliance incident report required.

### 9.26.12.3 Root cause analysis, three hypotheses

**H1: a zone allocator bug randomly assigns duplicates.** **Test:** examine the OVN source file `controller/physical.c`, function `alloc_ct_zone()`. Algorithm: scan 1 through 65535 and return the first value not in use. **Result:** the algorithm is correct at every call site. **Rejected.**

**H2: 16-bit zone space exhaustion.** 50 chassis times 600 zones per chassis equals 30,000 zones cluster-wide. Per-chassis, 600 is far below 65535. **Test:** `ct-zone-list | wc -l` returns 603 (well under the limit). **Rejected.**

**H3 confirmed: a race between unclaim and claim.** Unclaim `vm_t1_13`, wait, no, `vm_t1_13` still belongs to chassis-B and is unchanged. The collision happens because ovn-controller on chassis-B calls `alloc_ct_zone()` and does not see zone 1012 as "in use" at the moment of the call, because T1 LSP `vm_t1_13` is in a transient state due to an unrelated `handle_changes` loop. **Test:** the engine `inc-engine/show-stats` shows a recompute spike between 15:00:00 and 15:00:05. The transient race in the I-P engine when many LSPs change simultaneously is confirmed. **Confirmed.**

### 9.26.12.4 Mechanism deep-dive, the zone allocator race

In the OVN source file `controller/physical.c`:

```
alloc_ct_zone(chassis_ctx, lsp_name) {
    for zone_id in 1..65535:
        if !chassis_ctx->zones_used[zone_id]:
            chassis_ctx->zones_used[zone_id] = lsp_name
            return zone_id
}
```

The `zones_used` bitmap is built by iterating the `Port_Binding` table. If in one main-loop iteration:
1. The engine detects the `vm_t2_01` binding claim (a new LSP).
2. The engine starts `alloc_ct_zone` for `vm_t2_01`.
3. The bitmap **has not refreshed** in response to a concurrent change affecting `vm_t1_13` (for example, an `external_ids` update, which is not zone-relevant but still triggers an engine recompute on the `en_runtime_data` node).
4. The engine sees zone 1012 as "free" and assigns `vm_t2_01 = 1012`.
5. The next iteration realises that zone 1012 is genuinely used by `vm_t1_13` (which persisted from earlier), but the binding has already been committed to OpenFlow on br-int. Zone 1012 is now in a "collision state" until ovn-controller detects it and reassigns.

**Fix in OVN 24.03 and later.** Add transactional zone allocation with a global lock in the engine node `en_physical_flow_changes`. The bitmap refreshes before every allocation.

**OVN 22.03 baseline (the curriculum).** Vulnerable. Workaround: avoid concurrent LSP churn on the same chassis.

### 9.26.12.5 Four-tier remediation

**Tier 1 (immediate, 5 minutes).**

```bash
# Force zone re-allocation: restart ovn-controller on the affected chassis.
[chassis-B]# sudo systemctl restart ovn-controller

# Wait 30 seconds for ovn-controller to reconnect to the SB DB, re-claim LSPs, and re-allocate zones.
[chassis-B]# ovs-appctl -t ovn-controller ct-zone-list | grep "vm_t2_01"
ls_t2_vm_t2_01: 1512   # NEW zone, no longer 1012.

# Flush the old conntrack zone to clear stale state.
[chassis-B]# sudo conntrack -D -w 1012
```

**Tier 2 (short-term, 24 hours).**

Avoid concurrent migrations. Batch-migrate sequentially, not in parallel, on the same destination chassis:

```bash
# Before (risky): parallel migration.
[cms]# for lsp in vm_t2_01 vm_t2_02 vm_t2_03; do
          ovn-nbctl lsp-del $lsp &
          ovn-nbctl lsp-add ls_t2 $lsp &
       done

# After (safer): sequential with pauses.
[cms]# for lsp in vm_t2_01 vm_t2_02 vm_t2_03; do
          ovn-nbctl lsp-del $lsp
          sleep 2
          ovn-nbctl lsp-add ls_t2 $lsp
          sleep 2
       done
```

**Tier 3 (medium-term, one to two weeks).**

Upgrade from OVN 22.03 to 24.03 or later. Release note checkpoints:

- 23.09: the transactional zone-allocation framework starts.
- 24.03: complete transactional zone allocation with collision prevention.
- 24.09: periodic zone compaction (reclaiming zones from deleted LSPs).

**Tier 4 (long-term, monitoring).**

```yaml
# Alert: detect a zone collision pattern.
- alert: OVNConntrackZoneCollision
  expr: |
    count(count by (zone) (ovn_ct_zone_assignment))
    != count(ovn_ct_zone_assignment)
  for: 1m
  labels:
    severity: critical
  annotations:
    summary: "OVN conntrack zone collision on {{ $labels.chassis }}"
```

Daily audit script:

```bash
#!/bin/bash
# /usr/local/bin/ovn-zone-audit.sh
for chassis in $(ovn-sbctl --bare --columns=name list Chassis); do
    dup=$(ssh $chassis "ovs-appctl -t ovn-controller ct-zone-list" \
          | awk -F': ' '{print $2}' | sort -n | uniq -d)
    if [ -n "$dup" ]; then
        echo "COLLISION on $chassis: zones $dup"
    fi
done
```

### 9.26.12.6 Comparison with pure OVS, can collisions happen without OVN?

In pure OVS (no OVN), zones are assigned through `ct(zone=<explicit>)` in OpenFlow actions. A custom controller (for example, Faucet) is responsible for not assigning duplicates. Collision risk depends on controller quality.

OVN automates zone allocation, which scales well but introduces a race. This is a trade-off.

### 9.26.12.7 Case 3 takeaways

**One.** A conntrack zone is a 16-bit resource shared per chassis. OVN auto-allocation creates a race during concurrent LSP churn.

**Two.** A collision does not always show up immediately. An old state entry has to clash with the 5-tuple of new traffic before corruption appears. The random appearance of the symptom is easy to misdiagnose.

**Three.** Cross-tenant impact is a signature. Two different tenants reporting connection issues at the same time and at a time matching a migration event equals a zone collision.

**Four.** A daily audit script is the last line of defence. The script runs in 10 seconds and catches collisions before users see them.

## 9.26.13 Cross-case takeaways, three incident-class design lessons

The three case studies, 9.26.1 to 10 (ukey leak), 9.26.11 (bond flap cascade), and 9.26.12 (zone collision), are not random. All three belong to the class of "**eventually consistent distributed cache**" issues.

**Class pattern:**

- State is held in multiple places (userspace ukey plus kernel DP flow; OVN bitmap plus OpenFlow; tenant ACL plus conntrack entry).
- The sync mechanism is asynchronous (revalidator sweep every 500 ms; engine iteration every 5 s; bitmap refresh per iteration).
- Failure means the state diverges and does not converge (zombie ukey; zone collision; stale megaflow).

**Lesson 1: convergence proof requires adversarial testing.** Code review is not enough. Adversarial testing (churn that produces faster than the reclaim rate) is the only way to find class failures of this kind before production.

**Lesson 2: observability needs per-cycle metrics, not per-state metrics.** "Dump duration", "engine_recompute", and "bond_reconfigure" are per-cycle metrics that show whether the system is CATCHING UP or FALLING BEHIND. Per-state metrics like "current flow count" only show transient health and miss the class failure.

**Lesson 3: symptoms cascade past module boundaries.** Ukey leak, then CPU spike, then VM latency, then application cascade. Bond flap, then megaflow storm, then cross-chassis latency. Zone collision, then conntrack state corruption, then cross-tenant issue. Debugging must follow the cascade chain rather than stopping at the first module boundary.

**Lesson 4: symptom latency is not root-cause latency.** The ukey leak builds for 4 hours and alerts only after 6 hours. The zone collision is instant but alerts after 15 minutes when a user complains. Root-cause latency can be minutes (zone) or hours (ukey). A forensic timeline must separate the two.

## Capstone POE, "Will reducing `flow-limit` across 10 OVS 2.11 hypervisors help?"

**Goal:** evaluate a remediation strategy against a hypothetical scenario.

**Scenario:** a data centre runs 10 OVS 2.11 hypervisors, all of which experience intermittent stale ukey leaks. The engineering manager proposes: "Lower `other_config:flow-limit` from the default 200,000 to 50,000 across all 10 nodes; the problem will decrease."

**Predict.** Does the proposal help? Why or why not?

**Observe (reasoning).** A smaller `flow-limit` means a smaller maximum cache, which means lower revalidator sweep pressure (assuming the sweep is O(n)). However:

- OVS 2.11 does not contain the fix from commit `180ab2fd635e` (2024). **The root cause persists** regardless of `flow-limit`.
- When the workload exceeds 50,000 legitimate flows, `flow_add` fails, packets go to the slow path, and VM latency spikes more severely than the original leak.
- If flow churn is high (hardware offload rotation), ukeys are created faster than they can be deleted; the 50,000 cap is reached in minutes instead of hours, making the outage worse.

**Explain.** The proposal is **rejected**. `flow-limit` is a symptom-level control (it caps the consequence) and does not fix the root cause (the leak rate). The correct path is:

1. Immediate: a `revalidator/purge` script triggered every 15 minutes through cron (a band-aid).
2. Medium: upgrade OVS 2.11 to 2.17 LTS with the backport, or to a 3.5 or later release that contains the fix.
3. Long: a monitoring dashboard and alert rule for the ukey-to-flow ratio.

**Finish.** Distil the universal principle: "Resource limits do not fix leaks; the pathology rate must be fixed." This applies to every cache (memory pool, connection pool, file descriptor pool).

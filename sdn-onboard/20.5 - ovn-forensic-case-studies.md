## 20.5.1 Forensic principles for the OVN distributed control plane

Part 9.26 built a forensic framework for the OVS datapath (revalidator storm, bond flap cascade, conntrack zone collision). Those cases were scoped to **one chassis** or **two adjacent chassis**. Part 20.5 extends scope to **cluster-wide distributed state**: where an incident arises from **temporary asynchrony** between the NBDB configuration intent, the SBDB computed state, and the chassis local datapath.

Three important differences from OVS forensics:

**First, the role of the "eventually consistent" mechanism.** NBDB to northd compile to SBDB to ovn-controller on each chassis to OpenFlow installation is a four-stage loosely synchronous chain. Between stages there are caches, I-P nodes, and BFD. An incident appears when one stage runs faster than the previous stage or is invalidated in the wrong order. Part 9.26 §9.26.13 calls this the "eventually consistent distributed cache"; Part 20.5 dives into all three incident classes in this family but at OVN scope.

**Second, "claim protocol" instead of "flow install".** In pure OVS, whoever installs the flow owns it. In OVN, every Port_Binding has exactly one chassis owner at any moment. Migration, failover, and HA_Chassis_Group are ceremonies that change ownership. If the ceremony is not atomic, there is a race. Case 1 dives deeply into this pattern.

**Third, "memory budget" for the I-P engine.** Both northd and ovn-controller hold I-P node caches in memory. Some operation patterns (bulk delete, bulk add, bulk update) can balloon the cache before GC catches up. OOM in the control plane causes failover, failover triggers a full recompute on the successor, leading to a cascade. Case 2 exploits this pattern.

All three cases share a structure: **event trigger, distributed state mismatch, observable symptom, cascade, recovery**. The pattern repeats, so the forensic toolkit for later cases gets terser.

## 20.5.2 Case 1, Port_Binding migration race, "dual-bind transient" cross-chassis

### 20.5.2.1 Initial signature

A tenant cloud provider runs 60 hypervisors on OVN 22.03.8. VM T1-web-17 (tenant T1, Logical_Switch `ls-t1`, LSP `lsp-web-17`) is live-migrating from chassis `chA-042` to `chB-015`. Live migration in Nova / OpenStack runs a sequence (prepare target, stream memory, pause source, resume target, clean source). The OVN ML2 plugin updates Port_Binding.chassis from `chA-042` to `chB-015` immediately after "resume target".

The operator reports: in a window of about 8 seconds after resume target, **end users report intermittent connectivity to the VM**. ARP replies for the VM keep flipping the MAC address between two ports (ToR MAC flapping alert). ACL logs show the same 5-tuple connection appearing on two chassis. After 8 seconds everything stabilises automatically, chassis `chB-015` owns the port, `chA-042` no longer references the port. The VM operates normally.

A week later the operator returns when three other migrations show the same symptom with durations between 3 and 18 seconds. One time in particular, the duration of 18 seconds caused BFD flap between cluster chassis (Scenario E plus K in Part 9.14), which spread into a northd recompute burst and pushed P99 latency up by 120 ms for 2 minutes. The operator escalated with the hypothesis "OVN live migration is unsafe in a large cluster".

### 20.5.2.2 Evidence, Anatomy Template A for claim-protocol trace

**Why run it:** The engineer starts forensics after the fourth migration triggers cascade. Hypothesis: there is a "dual-bind" window in which both `chA-042` and `chB-015` claim the port concurrently.

**Command and output** *(reproduced from an OVN 22.03.8 lab running in a 10-chassis sandbox)*:

```bash
# Dump the Port_Binding timeline through ovsdb-client monitor with timestamps
[sbdb-leader]# sudo ovsdb-client --timestamp monitor OVN_Southbound \
  Port_Binding chassis,logical_port,tunnel_key \
  --where 'logical_port=="lsp-web-17"' 2>&1 | tee /tmp/portbinding-trace.log

# Output format (one event per line):
# 2026-04-24T14:32:01.234 update Port_Binding
#   logical_port="lsp-web-17" chassis=<uuid-chA-042> tunnel_key=17
# 2026-04-24T14:32:09.812 update Port_Binding
#   logical_port="lsp-web-17" chassis=<uuid-chB-015> tunnel_key=17   <- ML2 flip
# 2026-04-24T14:32:09.815 update Port_Binding
#   logical_port="lsp-web-17" chassis=<uuid-chB-015> tunnel_key=17   <- ovn-controller chB claim
# (no release event from chA for 8 seconds)
# 2026-04-24T14:32:17.901 update Port_Binding
#   logical_port="lsp-web-17" chassis=<uuid-chB-015> tunnel_key=17   <- finalize after chA release
```

**Anatomy of each event:**

| Event timestamp | Source | Action | Claim owner after event | Meaning |
|-----------------|--------|--------|-------------------------|---------|
| 14:32:01.234 | Initial state | bind chA-042 | chA-042 | VM runs on chA, OpenFlow local install complete |
| 14:32:09.812 | ML2 plugin | flip chassis column | chB-015 (NBDB configuration intent) | Nova finishes memory stream, updates Port_Binding |
| 14:32:09.815 | ovn-controller chB | claim notification | chB-015 (SB confirm) | chB sees Port_Binding.chassis=self, installs local flow |
| 14:32:09.815 to 14:32:17.9 | **Race window** | chA has not released | 2 chassis concurrent | **Dual-bind transient** |
| 14:32:17.901 | ovn-controller chA | release (delayed) | chB-015 (steady) | chA finally receives the event, removes local flow |

**Cross-check flow state on the two chassis during the race window:**

```bash
# On chA-042, at 14:32:14 (mid race window)
[chA-042]# sudo ovs-ofctl dump-flows br-int \
  | grep -E "0xb0004.*output.*ovn-chB015" | head -3
# Expected if released: 0 flows
# Actually observed: 2 leftover flows (ingress plus egress) not yet deleted

# On chB-015, at 14:32:14
[chB-015]# sudo ovs-ofctl dump-flows br-int \
  | grep -E "0xb0004.*priority=.*output:patch" | head -3
# Expected: full local flow set, VIF to VM up

# Datapath flow cache on chA (proof traffic still forwards to the VM)
[chA-042]# sudo ovs-dpctl dump-flows | grep -E "reg0=0x11" | head
# reg0 is the LSP tunnel_key, 0x11 is decimal 17
```

**Failure modes:**

- **Window under 2 seconds.** BFD does not flap, northd may not notice, the symptom is just a few duplicate packets. Users do not complain.
- **Window 3 to 10 seconds.** ARP / ND duplicate flood, ToR MAC flap alert, users report intermittent connectivity. This covers most cases.
- **Window over 10 seconds.** BFD flap spreads (the tunnel between chA and chB is briefly broken), triggering northd recompute burst plus ovn-controller lflow-cache invalidation across many chassis. The cascade becomes a larger incident class.

### 20.5.2.3 Mechanism deep dive

In OVN 22.03, the `Port_Binding.chassis` column is the direct write of NBDB configuration intent. When ML2 sets chassis=chB-015, SBDB propagates the change to all ovn-controllers. Each ovn-controller looks at the row:

- **ovn-controller chB-015** sees `Port_Binding.chassis=self`, runs `binding_add_lport()` in `controller/binding.c`, installs the OpenFlow local flow for the LSP, brings up the BFD session, and advertises the MAC address to the overlay.
- **ovn-controller chA-042** sees `Port_Binding.chassis != self`, runs `binding_del_lport()`, removes local flow, revokes the MAC advertisement.

These two actions are **not atomic** across chassis. The reasons:

1. **SBDB updates propagate through Raft replication**, with 10 to 50 ms delay from leader to follower chassis.
2. **ovn-controller on chA-042** has its own run loop (`ovn_controller_run_loop()` in `controller/ovn-controller.c`). If chA is busy with another I-P node recompute, its reaction to the new event may be delayed by several seconds.
3. **Flow uninstall on chA** must traverse the ovs-vswitchd OpenFlow channel; in a large cluster, that channel can queue and add further delay.
4. **chA does not know whether chB has finished its claim**, because there is no direct sync between the two chassis. Everything goes through SBDB as the intermediary.

In this window **both chassis still hold the local flows** for the LSP. Packets to the VIF on chA still forward (the flow is still there), packets to the VIF on chB also forward (the new flow is installed). ARP replies from the VM may originate from one of two chassis depending on the hypervisor kernel scheduling. Result: the ToR switch sees the MAC address simultaneously on two uplink ports, producing a MAC flap.

It is especially bad when **an external client opens a new connection**: the SYN packet may be routed to chA or chB through underlay ECMP. If SYN arrives at chA but chA has stopped forwarding (yet has not removed the flow), traffic is black-holed. Conversely if SYN arrives at chB but the VM memory has not fully resumed, the VM does not reply.

OVN 22.06 added the `Port_Binding.requested_chassis` column as a **configuration intent declaration** separated from `chassis` (claimed owner). ML2 updates `requested_chassis`, northd handles the handover in order: wait for chA to release, set chassis=chB, chB claims. This eliminates the race. OVN 24.03 polishes the pattern with BFD-based fast failover for HA_Chassis_Group.

### 20.5.2.4 Four-tier remediation

```bash
# Tier 1 (immediate mitigation, short term)
# When a dual-bind window over 5s is detected, manual coordination:
# Step 1: pause migration, check Port_Binding.chassis
[operator]# sudo ovn-sbctl --columns=logical_port,chassis list Port_Binding \
  | grep lsp-web-17

# Step 2: if stuck in dual-state, force release on the source chassis
[chA-042]# sudo ovs-appctl -t ovn-controller binding-release lsp-web-17
# or restart ovn-controller on the source chassis (heavy-handed)

# Step 3: verify the flow is gone
[chA-042]# sudo ovs-ofctl dump-flows br-int | grep -c "reg.*=0x11"
# Expected: 0 flows for the LSP

# Tier 2 (tooling coordination, medium term)
# Nova / ML2 plugin must enforce migration step ordering:
# 1. Call OVN Notify "migration intent" (pre-commit hook)
# 2. Stream memory
# 3. Pause source VM
# 4. Update Port_Binding.chassis = target
# 5. Wait at most 2s for confirm of target claim through SBDB notification
# 6. Resume target VM
# 7. Clean up source VM
# If step 5 times out, rollback: keep source, abort migration
# Patch the ML2 code (neutron/plugins/ml2/drivers/ovn) to implement the confirm wait

# Tier 3 (structural, OVN upgrade)
# Upgrade OVN 22.06+ with Port_Binding.requested_chassis
# Update the ML2 plugin to set requested_chassis instead of chassis
# northd handles the atomic handover:
[any]# sudo ovn-nbctl --db=... set NB_Global . options:use-requested-chassis=true

# Tier 4 (prevention, deployment hygiene)
# - Daily cron: scan Port_Binding for windows over 5s in metric history
#   to identify migration fail-rate patterns
# - Capacity planning: do not migrate more than 5 VMs at once per chassis pair
#   to reduce ovn-controller run-loop contention
# - Alertmanager rule:
```

**Prevention alerts:**

```yaml

```

## 20.5.3 Case 2, northd bulk tenant deletion memory cascade plus OOM failover

### 20.5.3.1 Initial signature

Tenant T2 offboarding at large scale: 5,000 Logical_Switch_Port, 500 ACL rules, 200 Load_Balancer, 80 Logical_Router_Port, totalling about 350 MB of Logical_Flow data on SBDB. The automation script runs in one shot:

```bash
# Tooling automation (buggy)
for lsp in $(ovn-nbctl --columns=name find LSP external_ids:tenant=T2); do
  ovn-nbctl lsp-del $lsp
done
for acl in $(... T2 ACL ...); do
  ovn-nbctl acl-del $acl
done
# ... continue for LB, LR, etc.
```

Each `ovn-nbctl lsp-del` is its own transaction. 5,800 transactions in 90 seconds give an NBDB Raft commit rate of 65 per second.

The operator receives the alert: **Prometheus `ovn_northd_memory_bytes` balloons from 400 MB to 2.4 GB in 90 s**. After 90 s, systemd reports `ovn-northd` OOM-killed. The 3-node cluster has northd active-standby: standby northd on node-2 takes over. The standby northd must perform a **full recompute** from NBDB (the I-P cache is not warm), running for 180 seconds. During that 180 s window, no `nb_cfg` update reaches SBDB, all chassis are stuck without Logical_Flow change. Newly created VMs have no network.

Once the standby finishes recompute, nb_cfg catches up over 60 seconds. Total incident window is **about 4 minutes 40 seconds**. The operator asks: "Why does deleting a tenant cause a cascade like this?"

### 20.5.3.2 Evidence, Anatomy Template A for memory forensics

**Why run it:** The engineer audits the northd process lifecycle after the standby finishes the takeover.

**Command and output** *(reproduced on an OVN 22.03.8 3-node cluster, 10,000 LSP baseline)*:

```bash

[nbdb-leader]# sudo ovn-appctl -t ovn-northd memory/show

  total:            2423 MB      <- !!! baseline 400 MB

  idl-output:        124 MB      <- SB data cached

  lb-data:           215 MB      <- Load_Balancer I-P node cache
  chassis-index:      18 MB
  other:             137 MB

# I-P engine status

  Maximum: 8921 ms       <- 8.9s per iteration (baseline under 100 ms)

```

**Anatomy of each field:**

| Field | Baseline value | Cascade-time value | Meaning |
|-------|---------------|--------------------|---------|
| `total memory` | 400 MB | 2423 MB | 6x balloon |
| `lflows cache` | 140 MB | 1847 MB | I-P cache holds old references |
| `Logical_Flow status` | updated | recompute | Full rebuild instead of incremental |
| `mem-delta Logical_Flow` | +/- 5 MB per iteration | +847 MB per iteration | GC has not cleared the old before adding the new |
| `build_lflows max` | 80 ms | 8412 ms | 100x slower |

**Failure modes:**

- **`lflows cache` over 1 GB** within 60 s: I-P node references hold thousands of deleted LSPs, memory is not freed in time.
- **`Logical_Flow status: recompute`** repeats for more than 5 iterations: the engine cannot exit the recompute loop because new input keeps changing (each new transaction invalidates).
- **OOM kill triggers around 2.4 GB** (depending on systemd `MemoryMax`; default unset means it depends on cgroup or hypervisor).

### 20.5.3.3 Mechanism deep dive

The northd I-P engine is designed to handle changes incrementally. When one LSP is deleted:

1. The engine invalidates the I-P node `Logical_Flow` (which depends on `Logical_Switch_Port`).
2. The `Logical_Flow` node runs a diff: compares old versus new LSP input, detects the LSP disappeared.
3. The engine calls `clear_tracked_data()` to free old references.
4. The engine calls `build_lflows_for_changed_only()` to rebuild only the affected flows.

In steady state this cycle takes about 5 to 20 ms per LSP.

When 5,000 LSPs are deleted in 90 s (batch 65 per second):

1. Every transaction invalidates the I-P node.
2. The engine starts the rebuild, but before it finishes the next transaction invalidates again.
3. The engine falls back to **full recompute mode** (the design "safety net" when partial update is too hard).
4. Full recompute must rebuild Logical_Flow for **all 5,000 remaining LSPs** (live ones plus the 5,000 currently in the invalidate window).
5. During full recompute, `clear_tracked_data()` runs **after** the build, not before. Memory grows before it is freed.
6. 5,000 iterations times about 170 KB per LSP cache equals about 850 MB peak.
7. Memory budget exceeded, OOM.

**Why is the failover not seamless?** The Raft cluster has 3 ovn-northd instances: 1 active, 2 standby. When the active is OOM-killed the standby takes over but must:

1. Download the full NBDB snapshot (400 MB, 2 to 3 s).
2. Build the I-P cache from scratch (no warm cache).
3. Rebuild the entire Logical_Flow set for the 95,000 surviving LSPs (after the 5,000 T2 ones half-deleted).

This is the 180 s stuck stage. VMs created in that window have no network until the standby completes the recompute.

### 20.5.3.4 Four-tier remediation

```bash
# Tier 1 (urgent recovery, after the OOM)
# The standby is recomputing; nothing can interrupt it. Wait.
# Monitor progress:
[nbdb-leader]# watch -n 2 'sudo ovn-appctl -t ovn-northd inc-engine/show \
  | grep -E "status|runtime"'
# When Logical_Flow status moves from recompute to updated, recovery is complete

# Tier 2 (tooling fix, batch deletion plus sleep)
# Refactor the automation script:
[tooling]$ # Instead of one transaction per delete
for batch in $(split -l 100 tenant-T2-lsp.txt); do
  CMDS=""
  for lsp in $(cat $batch); do
    CMDS="$CMDS -- lsp-del $lsp"
  done
  ovn-nbctl $CMDS   # 1 transaction, 100 LSPs
  sleep 3           # let the I-P engine settle between batches
done
# 5000 LSPs / 100 per txn = 50 txns x 3s = 150 s (versus 90 s of fast run)
# Slower in absolute terms, but memory peak only rises about 50 MB per batch, no balloon

# Tier 3 (systemd protection, memory cap plus auto-restart)
[nbdb-leader]# sudo systemctl edit ovn-northd
# Add:
# [Service]
# MemoryMax=4G
# Restart=on-failure
# RestartSec=5s
# OOMPolicy=stop      # rely on systemd rather than the kernel OOM killer

# Tier 4 (structural upgrade plus cache tuning)
# - OVN 23.09+ improves clear_tracked_data() to run incrementally
# - Tune cache budget:
[any]# sudo ovn-nbctl set NB_Global . options:northd-backoff-interval-ms=100
[any]# sudo ovn-nbctl set NB_Global . options:mac-cache-max-size=10000
# - Parallel northd build (OVN 23.03+):
[nbdb-leader]# sudo ovn-appctl -t ovn-northd parallel-build/enable
# Distributes lflow build across multiple cores, lowering per-core peak memory
```

**Prevention alerts:**


## 20.5.4 Case 3, MAC_Binding table explosion from a tenant ARP scan exploit

### 20.5.4.1 Initial signature

Tenant T3 deploys 20 VMs in subnet `10.0.3.0/24` (254 usable IPs). One VM is compromised through a vulnerability in the container image. Malware inside the VM runs an **ARP scan over 10.0.0.0/16** (65,534 IPs) for 15 minutes at a rate of 75 ARP requests per second.

The OVN Logical_Router `lr-t3` has a distributed ARP responder (Part 18.0) for the tenant subnet. When an ARP request targets an address **inside a managed subnet but absent from IPAM**, OVN generates a **synthetic ARP reply** from the router MAC to handle downstream forwarding (Part 18.0 §18.3). Each such reply creates a `MAC_Binding` row in SBDB: the tuple `(MAC, IP, Logical_Port)` cached for faster reply next time.

In 15 minutes the malware triggers about 67,500 ARP requests. The SBDB `MAC_Binding` table grows from a baseline of 400 rows to **67,900 rows**. As the row count increases, SBDB commit size grows, and the northd I-P node for `MAC_Binding` rebuilds on every new row (a bulk-update pattern similar to Case 2 but slower, not OOM-causing but still slowing the system).

Symptoms observed:
- **Prometheus `ovn_sb_table_rows{table="MAC_Binding"}` rises by 4,500 rows per minute**.
- **`ovn-controller` CPU on 60 chassis rises from 5 percent to 35 percent** because each must sync MAC_Binding changes through the SBDB monitor.
- **`nb_cfg` lag NBDB to SBDB rises from 0.2 s to 8 s**.
- **At the end of the 15-minute cycle**, the cluster stabilises again at high load. The malware stops (perhaps detected and quarantined by an external IDS).

A week later the operator finds that the MAC_Binding table still holds 67,900 rows permanently, with no garbage collection. The operator asks: "How do we reset? Why does OVN not self-heal?"

### 20.5.4.2 Evidence

```bash
# SBDB MAC_Binding table row count
[sbdb-leader]# sudo ovn-sbctl --columns=mac,ip,logical_port list MAC_Binding \
  | grep -c "^_uuid"
# Output: 67943

# Group by tenant logical_port
[sbdb-leader]# sudo ovn-sbctl --columns=logical_port list MAC_Binding \
  | sort | uniq -c | sort -rn | head
#   67519 logical_port=lr-t3-to-public
#     132 logical_port=lr-t2-to-public
#      85 logical_port=lr-t1-to-public
#     ... (baseline background count)

# Age distribution (OVN 22.03 has no timestamp column, fall back to external metrics)
[sbdb-leader]# # Proxy: the sbdb commit log shows row insertion time
[sbdb-leader]# sudo ovsdb-tool show-log /var/lib/ovn/ovnsb_db.db --timestamp \
  | grep "insert.*MAC_Binding" | head -20

# ovn-controller CPU impact
[chassis-X]# sudo ovn-appctl -t ovn-controller inc-engine/show
# node_name            status     runtime    mem-delta
# MAC_Binding          updated      412ms      +67 MB   <- !!! large cache

[chassis-X]# sudo ps -p $(pgrep ovn-controller) -o pcpu,pmem
# %CPU 34.5   %MEM 12.8   <- sustained high
```

### 20.5.4.3 Mechanism deep dive

In OVN 22.03 MAC_Binding has no **age threshold**. Once a row is created via ARP learning, the row stays in SBDB forever unless:

1. The LSP attached to logical_port is deleted.
2. The operator manually deletes the row.
3. The entire Logical_Switch is deleted.

There is no TTL. There is no LRU eviction. This was the original design choice because:
- **Steady-state workload** has a low MAC_Binding growth rate (only legitimate tenants).
- **When SBDB is backed up**, the row count is predictable.
- **Rows are small** (about 200 bytes), so 10,000 rows equals 2 MB, negligible.

This design is **fragile** against ARP scan exploits because:
- An attacker controlling a VM inside a tenant can produce a row rate of 75 per second.
- 75 per second times 3,600 seconds equals 270,000 rows per hour, unbounded growth.
- More rows make `MAC_Binding` I-P node rebuild slower, increasing latency spikes.
- SBDB commit size grows, making Raft replication slower.
- The ovn-controller local cache copy of MAC_Binding also grows, increasing memory usage on every chassis.

OVN 24.03 added `Logical_Router.options:mac_binding_age_threshold` (default 0 means disabled, the operator must set it). Setting it to 300 seconds garbage-collects rows created by ARP learning that have not been used for 300 s. This fixes the exploit vector but requires the operator to enable it explicitly.

### 20.5.4.4 Four-tier remediation

```bash
# Tier 1 (emergency purge after the scan stops)
# Identify tenant-specific MAC_Binding rows and delete in bulk
[sbdb-leader]# sudo ovn-sbctl --columns=_uuid,logical_port,ip list MAC_Binding \
  | awk '/logical_port=lr-t3-to-public/ {print $1}' | awk -F= '{print $2}' \
  > /tmp/mb-delete.txt

[sbdb-leader]# for uuid in $(cat /tmp/mb-delete.txt); do
  sudo ovn-sbctl destroy MAC_Binding $uuid
done
# 67000 rows x about 2s each = 37 hours (too slow!)
# Batch approach:
[sbdb-leader]# split -l 500 /tmp/mb-delete.txt /tmp/mb-batch-
[sbdb-leader]# for batch in /tmp/mb-batch-*; do
  CMDS=""
  for u in $(cat $batch); do CMDS="$CMDS -- destroy MAC_Binding $u"; done
  sudo ovn-sbctl $CMDS
  sleep 2
done
# 500 per txn x 2s x 135 batches = 4.5 minutes

# Tier 2 (prevent the exploit, ACL rate-limit egress ARP)
# Apply ACL rate-limit for lower-trust tenants
```

## 20.5.5 Cross-case takeaway, three design lessons for OVN deployment

The three cases differ in trigger and symptom but share the same structure:

1. **An event causes distributed inconsistency.** Migration race (Case 1), bulk transaction (Case 2), ARP scan (Case 3) all desynchronise the NBDB configuration intent, the SBDB computed state, and the chassis local view.
2. **Amplification through the I-P engine or claim protocol.** Engine rebuild overload (Case 2), non-atomic claim (Case 1), unbounded table (Case 3) turn desync into cascade.
3. **Recovery depends on "eventually consistent" convergence.** There is no master switch. The system eventually converges, but the cascade window damages service.

Three design lessons to reduce risk:

### Lesson 1, the claim protocol must be idempotent and atomic

OVN 22.03 merged Port_Binding.chassis as both claimed owner and configuration intent, producing the race. OVN 22.06+ separates requested_chassis from chassis, splitting **configuration intent from state**. The pattern applies to any distributed claim:

- **Separate configuration intent from state.** Configuration intent (who wants ownership) is not the same as state (who currently owns).
- **Forced ordering of transitions.** Source releases first, target claims second, never the reverse.
- **Timeout rollback.** If the transition is stuck for more than N seconds, revert configuration intent to the original state.
- **Observable transition state.** Expose a "transitioning" state to metrics so the operator can detect.

Not only OVN migration applies. The pattern fits any system with leader election, primary-secondary failover, or quorum-based ownership.

### Lesson 2, the I-P node memory budget must be bounded and observable

Both northd and ovn-controller use I-P node caches for performance. But an unbounded cache is an exploit vector (Case 2) or a cascade symptom (Case 3).

Principles:
- **Cache size exposed as a metric.** `memory/show` fields must export to Prometheus.
- **Proactive GC, not reactive.** `clear_tracked_data()` must run **before** the new build, not after. OVN 23.09+ moves in this direction.
- **Explicit upper bound.** Set `MemoryMax` in systemd so the OOM killer prefers killing other non-critical processes, not northd.
- **Batch-pattern enforcement.** Tooling automation must break bulk transactions into smaller batches with sleep gaps for GC.

The lesson also applies to any control plane with caching: Kubernetes controller, etcd reconciliation, Prometheus TSDB ingestion.

### Lesson 3, distributed state must be age-bounded

Unbounded MAC_Binding equals storage leak plus performance degradation plus attack surface. Age threshold is the basic defence for any **learned state** in a distributed system:

- **ARP / ND cache has TTL** (kernel default 60 s, OVN 24.03+ mac_binding_age_threshold is similar).
- **Conntrack entry has TCP timeout** (Part 9.24 §9.24.6).
- **Raft log has compaction threshold** (OVS 2.13+ auto-compact).
- **Flow statistics has hard-timeout** (OpenFlow specification).

No age bound means state grows with operation history, not with current topology. Three months after deployment the table holds 10x rows that "no one uses but no one deletes".

Golden principle: **every row learned from packet observation must have an age attribute**. Every learned row must have a default-on GC policy. The operator opts out only when needed (for example, traffic forensics that needs to retain longer), never opts in.

## 20.5.8 Capstone POE, "set mac_binding_age_threshold=60 on every LR in the cluster, also fix the ARP scan exploit?"

**Scenario:** After reading Case 3, a senior manager proposes: *"Apply `options:mac_binding_age_threshold=60` to all 240 Logical_Routers in the production cluster. The 60-second value is short enough to GC-rate-limit the exploit scan. We can automate it with a 1-liner script and deploy in a 5-minute maintenance window. This solves the MAC_Binding explosion across all tenants."*

Analysis and rebuttal:

**Predict:** Does this setting fix the exploit? Are there side effects?

**Observe:**

1. **Legitimate tenants may be affected.** A stateless server has incoming connections from many external clients. Each client interacts for 1 to 2 minutes then idles. With `mac_binding_age_threshold=60` the row will be GC'd before the next request from the same client. Result: every new connection retriggers an ARP request, latency rises by 10 to 50 ms (depending on the underlay RTT). Negligible for a web server, but significant for latency-sensitive services (real-time gaming, trading).

2. **A large subnet with a stable server fleet.** A tenant has 500 stable servers in a /24. MAC_Binding caches rows for 500 legitimate MACs. Threshold=60 s forces the row to GC after 60 s without use. If a server is idle for 65 s the row is GC'd. The next ping from that server triggers an ARP request, gets an ARP reply from the router, then inserts a new row. This is **wasted work** (rebuilding the same entry repeatedly) without preventing anything.

3. **Too small a threshold gives weak protection.** An attacker scanning at 10 IPs per second with threshold=60 s caps the table size at 600 rows. But an attacker at 100 IPs per second caps at 6,000 rows. Still enough to slow the control plane in a large cluster. Threshold is **defence in depth, not the primary defence**. The primary defence must be rate-limiting egress ARP from the tenant (Tier 2 of Case 3).

4. **Applying to 240 LRs all at once without a maintenance window is risky.** Each set operation invalidates the Logical_Router I-P node. 240 invalidations in 5 minutes is a burst similar to Case 2 (bulk update) at 50x smaller scale. May cause nb_cfg lag of 10 to 30 s in the 5-minute window. Not an outage, but VM provisioning delay.

5. **The "right" threshold depends on traffic pattern.** No magic number. Each tenant must be analysed:
   - Web-facing servers with short-lived clients: threshold 300 to 600 s (balance GC versus relearn).
   - Backend server fleet with low-volatility traffic: threshold 3,600 s or disabled (cache is stable).
   - High-risk tenant with untrusted workload: threshold 60 s **plus** ACL ARP rate-limit.

**Explain:**

The "1-liner fix-all" proposal is **not correct**. The right approach is a 3-step per tenant:

- **Step 1, classify tenant risk:** internal trusted (stable, long-lived) versus high-risk (untrusted workload, public-facing). Use external_ids or a tagging system to mark tenants.
- **Step 2, per-class policy template:**
  - Trusted class: threshold=3,600 s or disable plus ACL allow-established (stateful).
  - Standard class: threshold=300 s plus ACL rate-limit ARP at 10 per second.
  - High-risk class: threshold=60 s plus ACL rate-limit ARP at 5 per second plus daily audit cron.
- **Step 3, rolling deployment with monitoring:**
  - Apply per tenant in a maintenance window, not in batch.
  - Monitor post-apply: MAC_Binding row count plus ovn-controller CPU on 60 chassis.
  - Rollback policy if P99 latency spikes by more than 20 percent.

**Finish:**

Principle: **the same setting has no universal "right" value**. The OVN age threshold is a tool, not a policy. Applying the tool must connect to traffic-pattern classification. The 1-liner mass-apply is a frequent anti-pattern, often appearing when the operator is under pressure to "close the ticket". Trade-off between **defence strength** (lower threshold equals better protection from exploits) and **performance cost** (lower threshold equals frequent ARP re-learn equals latency tax). No free lunch.

Cross-reference: similar to the lesson in Case 2 (batch deletion not 1-txn 5,000 LSPs) and Scenario M of Part 9.14 (restart does not fix ghost claim), "simple 1-step fix" for distributed state requires default skepticism.

---

## Key points to remember

First, the OVN distributed control plane has three characteristic incident classes: claim-protocol race, I-P memory cascade, learned-state explosion. Each class has its own signature, mechanism, and remediation.

Second, "eventually consistent" does not mean "immediately correct". The asynchrony window between NBDB configuration intent, SBDB computed state, and chassis local view is the source of races and cascades. Forensics must measure that window with ovsdb-client monitor timestamps, not just observe the end state.

Third, I-P engine caching boosts performance but creates a memory attack surface. The bulk-1-transaction batch pattern is a common anti-pattern that causes OOM cascade. Always split a bulk transaction with sleep gaps.

Fourth, learned state (MAC_Binding, conntrack entry, Raft log) must be age-bounded by default. No age bound means state grows unbounded equals attack vector.

Fifth, the claim protocol must separate configuration intent from state (the OVN 22.06+ requested_chassis pattern). Transitions must be atomic, with timeout rollback, and observable state.

Sixth, a "1-liner fix-all" for distributed state is a warning sign. Always classify tenants plus per-class policy template plus rolling deployment.

## 20.5.9 Minor incident 1, chassisredirect failover anatomy when the active chassis dies

The Distributed Gateway Port (DGP) in OVN binds to a `type=chassisredirect` port in `Port_Binding`. At any moment, exactly one chassis in the HA_Chassis_Group is **active**; the others are **standby** ordered by priority. When north-south egress traffic from an LSP must cross the DGP, OVS on the source chassis uses the `bundle(active_backup, ovn-tunnel-to-chA, ovn-tunnel-to-chB, ...)` action with the highest-priority chassis with BFD-up placed first (Part 13.5 §13.5.5).

**Failover timeline when the active chassis dies** (simulated power-pull on `chA` priority 100, standby `chB` priority 90):

- **T+0** chassis chA crashes. BFD packets from chA stop.
- **T+0.6 s** chB ovn-controller detects the BFD timeout (default `min_rx=1000ms x 3 = 3000ms`, but OVN reduces it through `external_ids:ovn-bfd-min-rx`). In practice detection takes 600 to 3,000 ms depending on configuration.
- **T+0.6 s + epsilon** chB ovn-controller runs `binding_run()` in `controller/binding.c`, sees itself as the highest-priority HA member with BFD-up, claims `Port_Binding.chassis = chB` for the chassisredirect port.
- **T+0.7 s** chB installs the ingress flow and sends GARP plus Neighbor Advertisement to the physical fabric (`pinctrl.c` `send_garp_rarp_for_lport()`) so the ToR refreshes its MAC table.
- **T+1 to 3 s** the physical fabric ToR / spine MAC table converges to chB. Egress traffic flows through chB.

**A subtle failure mode: BFD up too early after reboot.** Commit fix `621f85e92437` (Xavier Simonart and Dumitru Ceara, 2026-03-26, file `controller/bfd.c` plus `controller/ovn-controller.c`) records a subtle incident class: when chA crashes and reboots, OVS restarts and automatically marks `bfd state=UP` on the Geneve tunnel as soon as the link layer is up, **before** ovn-controller has reinstalled OpenFlow flows. The fabric peer sees BFD up, sends traffic toward chA, chA black-holes the packets because there are no flows yet. The fix uses the `flow-restore-wait` flag: OVS keeps BFD ignored until ovn-controller acknowledges readiness. This is a typical "premature liveness signal" pattern that occurs in every failover protocol: LACP, VRRP, BGP graceful restart all have variants.

**Forensic command** when failover is suspected stuck:

```bash
# On the chassis suspected to currently be active
[chB]# sudo ovn-appctl -t ovn-controller debug/dump-bfd
# Columns "remote-state" and "diag" tell BFD peer status
[chB]# sudo ovs-vsctl --columns=name,bfd_status list Interface ovn-chA-tunnel
# Confirm BFD up / down from the OVS side
[chB]# sudo ovn-sbctl --columns=chassis,nb_cfg list Port_Binding cr-lrp-public
# Confirm chassisredirect has claimed chB (chassis column points to chB)
```

**When failover fails:** BFD timeout configured too large (over 5 s) causes slow failover; equal priority (two chassis both at 100) causes flapping (see §20.5.12); fabric multipath that is not equal-cost makes GARP miss the ToR race. Remediation: standardise BFD `min_rx=400ms multiplier=3` (timeout about 1.2 s), distinct priorities, symmetric fabric ECMP.

## 20.5.10 Minor incident 2, Logical_Flow row outliving NBDB origin causes east-west blackhole

Logical_Flow rows in SBDB are **derived** from NBDB objects (Logical_Switch, ACL, LSP, Logical_Router, NAT). When an NBDB row is deleted, the northd I-P engine must invalidate the correct downstream Logical_Flow rows. If the I-P node tracks dependencies wrong, a subset of Logical_Flow rows can be "stranded": the NBDB source is gone but the SBDB row still exists, and ovn-controller on the hypervisor still installs that flow. The result: **east-west blackhole** in a subset of traffic ([commit](https://github.com/ovn-org/ovn/commit/e8a0ade51cc1) `e8a0ade51cc1` 2025-08-26 records this pattern for Static MAC Bindings; a similar pattern occurs with Logical_Flow).

**Signature:** after a fast delete-then-recreate of a Logical_Switch (for example a rename via `ls-del + ls-add` in the same outer transaction, or a Terraform reconcile loop), some LSPs on the new LS cannot reach each other. `ovn-trace` runs to the right path, but `ofproto/trace` on the hypervisor shows the packet matching a `priority=50000` flow in table 8 with `actions=drop` from an ACL row that was deleted last week.

**Anatomy:**

```bash
# Count Logical_Flow rows referencing a Datapath_Binding that no longer exists
[sbdb]# sudo ovn-sbctl --columns=_uuid,logical_datapath,priority,match \
  list Logical_Flow > /tmp/lflows.txt

[sbdb]# sudo ovn-sbctl --columns=_uuid list Datapath_Binding \
  | awk -F= '/^_uuid/ {print $2}' > /tmp/valid-dp.txt

# Cross-reference: Logical_Flow.logical_datapath not in valid-dp.txt is stale
[sbdb]# python3 -c "
valid = set(open('/tmp/valid-dp.txt').read().split())
import re
for line in open('/tmp/lflows.txt'):
    m = re.search(r'logical_datapath=([0-9a-f-]+)', line)
    if m and m.group(1) not in valid:
        print('STALE:', line.strip())
" | head
```

**Mechanism deep dive:** the northd I-P node `lflow` in `northd/inc-proc-northd.c` runs handler `northd_handler_logical_flow()`. In the historical bug paths (see [commit `749e1cc0fcce`](https://github.com/ovn-org/ovn/commit/749e1cc0fcce) 2025-12-19 "Avoid unnecessary datapath validity checks"), the engine fell through to **partial recompute** instead of full recompute when Datapath_Binding was quickly deleted, leaving some Logical_Flow rows undeleted.

**Recovery:**

```bash
# Force a northd full recompute to rebuild SBDB Logical_Flow from authoritative NBDB
[any]# sudo ovn-appctl -t ovn-northd inc-engine/recompute
# Wait 10 to 30 seconds for convergence and nb_cfg sync
[sbdb]# sudo ovn-sbctl --columns=nb_cfg list NB_Global
# nb_cfg in NBDB equals sb_cfg in SBDB equals hv_cfg in Chassis_Private equals converged
```

**Prevention:** tooling automation must avoid delete-then-recreate of the same name in one transaction; replace with an atomic rename through `external_ids` instead of delete-create; alert when Logical_Flow row count does not drop after an NBDB object delete after 5 minutes.

## 20.5.11 Minor incident 3, Geneve TLV class 0x0102 decode failure

OVN uses Geneve as the default overlay tunnel (alternative to VXLAN). The Geneve header has **option TLVs** for extension metadata: OVN encodes `MFF_LOG_DATAPATH` (datapath tunnel_key), `MFF_LOG_INPORT` plus `MFF_LOG_OUTPORT` (logical port tunnel_keys), and several registers in vendor class `0x0102` (Open vSwitch organisationally-assigned). The receiver chassis must have a **TLV map** identical to the sender; otherwise the kernel datapath drops the packet with reason `tun_metadata_unknown`.

**Common triggers:**

1. **Mixed OVN versions in a cluster** (for example a rolling upgrade halfway from 22.03 to 24.03). 24.03 adds TLVs for new features (commit `f5aae90685ec` 2025-02-07 "Do not create flood flows without Geneve metadata register" illustrates one subtle dependency). If chA-old sends Geneve with a TLV class plus type that chB-new lacks in its map, drop. Conversely chB sends a new feature through chA-old, drop.
2. **Vendor switch on the path strips Geneve options** ("Geneve option pass-through" is not a default feature on every merchant silicon). Packets reaching the receiver missing the TLV cause the datapath to fail to decode metadata, drop or misclassify.
3. **TLV map mismatch after a crash.** OVS holds the TLV map only at runtime (`ovs-ofctl add-tlv-map`). After ovs-vswitchd restarts, if the init script does not re-add the TLV, the map is empty, drop.

**Forensic command:**

```bash
# Dump the current TLV map on the chassis
[chX]# sudo ovs-ofctl dump-tlv-map br-int
# Output of the form:
# OpenFlow11 features:
#   class    type    length  match field
#   0x0102   0x80    8       tun_metadata0
# Compare two chassis in the same cluster; mismatch is a suspect cause

# Confirm drop reason from datapath stats
[chX]# sudo ovs-appctl dpctl/dump-flows -m system@ovs-system \
  | grep -i "tun_metadata\|geneve" | head

# Kernel-level packet drops with reason
[chX]# sudo dmesg | grep -i "openvswitch.*drop" | tail -20
```

**Reference fix history:** [commit `254878c18874`](https://github.com/openvswitch/ovs/commit/254878c18874) (Yi-Hung Wei, 2018-05-03, file `ofproto/ofproto-dpif-xlate.c`) fixes a revalidator-thread segfault when the TLV map is updated while flow recirculation is still holding a stale `tun_table` reference. Lesson: changing the TLV map at runtime is dangerous; coordinate through bundle update or rolling chassis drain.

**Recovery:** re-add the TLV map synchronously on all chassis through configuration management (Ansible task `ovs-ofctl add-tlv-map`); standardise the OVN version cluster-wide; verify the fabric does not strip Geneve options (test with `ovn-trace` end-to-end plus tcpdump verifying on-wire Geneve TLVs).

## 20.5.12 Minor incident 4, HA_Chassis_Group split-brain from BFD partition

HA_Chassis_Group election in OVN relies on priority order plus BFD liveness, **with no quorum mechanism** (different from Raft). If the network between two candidate chassis is partitioned (BFD on the direct tunnel goes down, but both are still reachable from tenant workload), both chassis may consider themselves active, both claim the chassisredirect port, **split-brain**.

**Trigger scenario:**

```
              +-------------------------------------+
              |           Spine fabric              |
              +--+----------+----------+------------+
                 |          |          |
              +--v--+    +--v--+    +--v--+
              |chA  |    |chB  |    |chC  |  (3 HA chassis in the same group)
              |prio |    |prio |    |prio |
              |100  |    | 90  |    | 80  |
              +--+--+    +--+--+    +--+--+
                 |          |          |
                 X (link chA-chB BFD down, chA-chC up, chB-chC up)
```

- **From chA's view:** BFD to chB DOWN, BFD to chC UP. chA at priority 100 is highest, chA claims active.
- **From chB's view:** BFD to chA DOWN, BFD to chC UP. chB does not see chA, chB believes chA is dead, chB is the highest-priority survivor, chB also claims active.
- **From chC's view:** sees both chA and chB BFD up. chC at lowest priority is standby.

**Consequence:** two chassis publish GARP for the gateway IP, the ToR MAC table flaps; ingress traffic splits between chA and chB; conntrack state is split without sync, return traffic takes asymmetric paths, drop.

**Reference fix history:** [commit `57966112b0a4`](https://github.com/ovn-org/ovn/commit/57966112b0a4) (Han Zhou, 2023-09-13, file `northd/northd.c`) refactors the HA reference chassis logic. The general fix pattern for BFD partition: OVN 24.03+ adds a "ref_chassis" mechanism in which the active chassis informs every chassis with active tunnels in the HA group, helping standby chassis cross-check before takeover. Note that this is not a true Raft quorum; it only narrows the split-brain window, it does not eliminate it.

**Detection:**

```bash
# On SBDB, if two chassis in the same HA group both have Port_Binding.chassis pointing to themselves
[sbdb]# sudo ovn-sbctl --columns=logical_port,chassis,nb_cfg,up \
  find Port_Binding type=chassisredirect
# Each distributed LRP should have one chassis. If two, split-brain confirmed (SBDB does not allow dual-bind, but race plus flap looks similar)

# Pairwise BFD status
[chA]# sudo ovs-vsctl --columns=name,bfd_status list Interface | grep -A1 ovn-
# Compared with
[chB]# sudo ovs-vsctl --columns=name,bfd_status list Interface | grep -A1 ovn-
# Asymmetric BFD view is a partition signal
```

**Remediation:** distinct priority for every HA member (never tie); fabric uplink redundancy ensures BFD path is not a single point of failure; alert "HA group partition" when pairwise BFD remains inconsistent for over 30 s.

## 20.5.13 Minor incident 5, NBDB to SBDB compile lag falling through to full recompute

ovn-northd I-P engine handles NBDB changes through incremental processing. Steady state: write 1 LSP, handler `en_northd_run()` runs in under 50 ms, SBDB Logical_Flow update propagates to hypervisors in under 200 ms. **Fall-through scenario:** the handler does not support the specific change pattern, the engine decides on full recompute. In a large cluster (10,000+ LSPs) full recompute takes 2 to 30 s. In that window nb_cfg does not advance, every NBDB write that follows queues up.

**Documented trigger:** [commit `0e0f38f1a293`](https://github.com/ovn-org/ovn/commit/0e0f38f1a293) (Ilya Maximets, 2024-02-08, file `northd/lflow-mgr.c`) tracks the ovn-heater cluster-density 500-node test: ovn-northd memory grows from 2.5 GB to 8.5 GB, recompute time grows from 1.5 s to 6 s, due to aggressive reference-counter allocation for every datapath in the datapath group. The commit defers allocation until second-use, reducing memory by 4x and recompute time by 3x. It also illustrates that **port_group create** in every iteration of the test pattern always triggers northd full recompute (the handler is not yet incremental for port_group).

**Common everyday triggers:** bulk writes from a Kubernetes operator, Terraform plan apply, or OpenStack Neutron sync after outage. Symptom: P99 latency spike of 10 to 30 s VM-create-to-network-up; alert `ovn_northd_logical_flow_recompute_time_seconds > 5`.

**Forensic command:**

```bash
# Engine status plus reason for recompute
[northd-active]# sudo ovn-appctl -t ovn-northd inc-engine/show
# Column "status" with repeating recompute equals stuck loop
[northd-active]# sudo ovn-appctl -t ovn-northd stopwatch/show
# build_lflows max-time is the fallthrough signal

# Identify the NBDB pattern that caused fallthrough
[nbdb]# sudo ovn-nbctl --timestamp show > /tmp/nb-state-1.txt
[nbdb]# sleep 30
[nbdb]# sudo ovn-nbctl --timestamp show > /tmp/nb-state-2.txt
[nbdb]# diff /tmp/nb-state-1.txt /tmp/nb-state-2.txt | head -50
# If port_group, address_set, or ACL log change at high rate, identify which handler fell through
```

**Reference fix tracking:** [commit `b27d9aebbe1b`](https://github.com/ovn-org/ovn/commit/b27d9aebbe1b) (2025-07-03) "Avoid unnecessary lflow recomputes when no BFD state changes" plus [commit `2225c0755540`](https://github.com/ovn-org/ovn/commit/2225c0755540) (2023-08-30) "Move port group processing to its own I-P node" represent a chain of improvements that move fall-through handlers from full recompute down to partial recompute. OVN 24.03 LTS has absorbed most of these improvements.

**Mitigation:** batch NBDB writes into a large transaction (ovn-nbctl `--`-syntax) instead of streaming small ones; rate-limit at the tooling layer (Kubernetes operator under 50 operations per second); upgrade OVN to 24.03 LTS or later for full incremental handlers; monitor `inc_proc_eng` log level INFO to capture the recompute reason and learn which handler to upgrade with priority.

1. CLAUDE.md Rule 17 (English as the mandatory explanation language).

### 3.2. Translation primitive (the "Five-step pass")

For each sub-batch file, the translator runs five steps:

#### Step 1: Read the file in full (once)

The translator reads the entire file once before editing. The goal is to internalise the pedagogical arc: what concept does the file teach, what mental model does it build, what does §X.Y depend on §X.Z for. A file translated section-by-section without this read produces inconsistent terminology and broken cross-references.

#### Step 2: Re-confirm the language-status callout

If the file already has the `**Language status:** Mixed` callout from v3.9.1, v3.10, or v3.11, note which sections were translated previously. Those sections stay unchanged unless they violate the style guide (§3.5 below).

If the file has no callout, add a placeholder under the H1 heading. Final form is set in Step 5.

#### Step 3: Translate prose chunks section by section

Per markdown section (any heading level):

1. Identify the prose chunks (paragraphs, list items, table cells with prose).
2. Translate each chunk to plain technical English per the style guide §1 to §10.
3. Preserve:
   - All canonical identifiers verbatim (per style guide §3.3 keep-as-is list).
   - All Rule 14 citations verbatim (function names, file paths, line numbers, commit SHAs, schema claims).
   - All verbatim quotes from upstream OVS / OVN / OF source, NEWS files, RFCs, man pages.

### 3.4. Six special-case translation rules

#### 3.4.1. Verbatim quotes from source code, NEWS, RFC, man page

Stay verbatim. Wrap in fenced block with explicit language tag (`text`, `c`, `python`, `bash`, etc.). The lang_check chunker already skips fenced blocks and inline code, so a verbatim English quote does not need any change.

If a verbatim Vietnamese quote exists (rare; some legacy headers quote a Vietnamese-language press release), translate the quote with the explicit annotation `(translated from Vietnamese original, dated YYYY-MM-DD)` and preserve the original Vietnamese text in a follow-on comment block. Style guide §10 explicitly addresses this case.

#### 3.4.2. Callout labels (style guide §8 mapping table)

Lock-in. Every translated file must use the English label exactly as the table prescribes. `rubric_leak_check.py` v2.1 already enforces several of these labels; v3.12 confirms the full set.

#### 3.4.3. Em-dash density (style guide §4)

Zero per file. The pre-commit hook enforces this on staged files. v3.12 also runs `--all-sdn-onboard` (a stricter scope than `--all`, restricted to `sdn-onboard/*.md`) at R2 to confirm legacy em-dashes inside v3.12 scope have been swept. Em-dashes outside v3.12 scope (`_templates/`, haproxy series) remain pending closure under the cross-block follow-on plan.

#### 3.4.4. Pedagogy preservation (style guide §7)

The Vietnamese curriculum has dense "why before what" pedagogy that flat-translation can flatten. The translator must preserve:

1. The motivation paragraph that opens each axis or section.
2. The mechanism paragraph that follows.
3. The verification paragraph that closes the section.

If the Vietnamese source compresses these into a single dense paragraph, the English translation must re-expand. The §10 sample shows the canonical pattern: the original 2 sentences become 5 sentences. This is intentional, not a quality regression.

#### 3.4.5. Acronym expansion at first use

Per the style guide §3.3 keep-as-is list, common acronyms (TCP, UDP, IP, TLS, OVS, OVN, OpenFlow) stay verbatim. Less common acronyms expand at first use within the file: "Tuple Space Search (TSS)" on first mention, then "TSS" thereafter.

#### 3.4.6. Number formatting

- Use the Oxford comma per §5.
- Numeric ranges in tables can use the en-dash (U+2013) per §4.3.
- Use `1,000` not `1.000` for thousands separators (the legacy curriculum sometimes follows the Vietnamese / European convention of dot-as-thousands-separator; correct to comma during translation).

### 3.5. Re-translation of previously-translated sections

A small fraction of sections were translated in v3.9.1, v3.10, or v3.11 under GP-13 in-commit rewrite. Most are correct per the style guide. However, three patterns need re-translation if observed:

1. **Em-dash present.** Replace per §4.
2. **Callout label is in non-canonical English** ("**Concept**." instead of "**Concept.**"). Normalise to the §8 lock-in form.
3. **Vocabulary violates §3.1 plain-words-preferred** (e.g., "ascertain", "necessitate"). Replace with plain word.

If none of these patterns apply, the previously-translated section stays unchanged. Re-translation for stylistic preference alone is forbidden (drift risk).

## 11. Anti-pattern signals (halt and report)

Same triggers as v3.10 §3.5 and v3.11 §3.5, adapted for translation:

### 11.1. Wider re-grep returns more than 5 candidates none context-fit

If, during Step 3 of §3.2, a translation pass requires re-checking a citation and the wider grep returns more than 5 candidates none context-fit (i.e., the cited function or file no longer exists at the offline baseline), HALT. Do not invent a substitute. Surface the inconsistency to the user.

### 11.2. Translation pass surfaces a contradiction between adjacent sentences

If Vietnamese sentence A says "x = 5" and Vietnamese sentence B in the next paragraph says "x = 7", and neither was caught by v3.9.1 / v3.10 / v3.11, HALT. Translation is not the right fix; the contradiction is a Rule 14 issue. Surface to user; defer the file to a v3.12.1 follow-on.

### 11.3. Pedagogy collapse warning

If a translated section drops 30 percent or more line count compared to the Vietnamese original, the §3.4.4 pedagogy preservation rule is at risk. Re-expand the section per §7 of the style guide before commit. If the line count drop is intentional (e.g., the original was bilingual scaffolding that the translation legitimately removes), document the rationale in the per-section evidence row.

### 11.4. Em-dash density above 0.05 / line in a translated file

The pre-commit hook enforces zero em-dashes. But a translated file with em-dash density above 0.05 / line in the prior commit indicates the translator introduced em-dashes during translation. HALT; review the translator's editorial habits; re-train on §4 of the style guide.

### 11.5. Single sub-batch fires 3+ §11.1 or §11.2 signals

If a single sub-batch fires 3 or more anti-pattern §11.1 or §11.2 signals, the sub-batch is unsalvageable in v3.12 scope. Close the sub-batch with whatever was completed; commit a Form A "incomplete sub-batch" log; defer the remainder to v3.12.1 follow-on. Do not force-close.

### 11.6. Pre-commit hook fails 3+ times on a file

If a file fails the pre-commit triplet 3 or more times, HALT for that file. The translator is making the same mistake; root-cause it before continuing. Common causes: regex-confused identifier, fenced-block delimiter mismatch, copy-paste em-dash from another tool.

### 11.7. Sub-batch effort overrun > 70 percent

Per §3.6 mid-batch escalation gate. HALT; trigger v3.12.1 follow-on for the remainder.

### 11.8. R2 final-audit reveals new content

If R2 spot-check criterion 8 reveals a section that was added (not just translated), it is a §0.8 violation. Revert the addition; recommit with translation-only.

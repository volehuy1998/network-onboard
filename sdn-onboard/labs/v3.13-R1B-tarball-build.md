## Why R1.B exists

R1.B is the install path a learner takes when the distro package is too old, when a tighter build configuration is needed, or when reading the build output is itself the lesson. The Open vSwitch project publishes a fresh release tarball at `https://www.openvswitch.org/releases/` after every stable point release; the tarball already contains the autotools-generated `configure` script, so the build does not need `./boot.sh` and the operator does not need `aclocal` or `autoreconf`. The trade-off is that the operator must own every build decision: where the prefix goes, whether to build the kernel module, which optional dependencies (libssl, libcap-ng, libunbound, DPDK) to include, and which target kernel headers (if any) to point at.

R1.B captures three things that the apt path of R1.A skipped entirely. First, it shows what the build prerequisites look like on Ubuntu 22.04 (autoconf, automake, libtool, pkg-config, libcap-ng-dev, python3-dev, libunbound-dev). Second, it shows the **out-of-tree kernel module mismatch** that every learner of Open vSwitch on a recent Linux kernel will hit at least once: the OVS 2.17 kernel module compatibility shim was written for kernel 5.8 and earlier; the lab kernel is 5.15, and `./configure` rejects it. Third, it shows the recovery path: drop `--with-linux` and rely on the in-kernel `openvswitch.ko` that Ubuntu 22.04 already ships under `/lib/modules/$(uname -r)/kernel/net/openvswitch/openvswitch.ko`. The recovery is the production answer for almost every operator who is not backporting datapath patches.

The transcript closes with `make distclean` and a disk-reclamation step. The build artefacts of a fresh OVS userspace build occupy roughly 200 MiB; on an 8 GiB lab disk this is meaningful, and a careful operator strips them before moving on to the next sprint.

## Carry-forward state from R1.A

R1.B was captured on the same `lab-openvswitch` host that ran R1.A roughly an hour earlier. R1.A purged the distro package but the kernel left two artefacts behind, and these are visible at the start of R1.B as part of the verbatim record. The pre-build `ls` of `/var/lib/openvswitch/` finds the residual `conf.db` from the purged install. The pre-build `lsmod` finds `nf_nat` and `nf_conntrack` still loaded; both were pulled in by the `apt-get install openvswitch-switch` step of R1.A and stayed resident after the daemon stopped.

These observations are pedagogically valuable. Rule 18 forbids hiding them. They show that an `apt purge` is not a true factory reset; the disk and the kernel both keep memory of an Open vSwitch installation past the package's removal. A learner planning to switch install paths on a working host must either snapshot before R1.A or accept that R1.B starts from a disturbed baseline.

## Fetching the release tarball

The Open vSwitch releases page lists tarballs for every stable point release; `openvswitch-2.17.9.tar.gz` is the matching baseline for plan v3.13. The download is over HTTPS from `www.openvswitch.org`. The `wget -nc` flag is "no clobber"; if the file already exists `wget` skips the transfer and returns silently, which is the right behaviour for a script that may run multiple times during a learning session.

```text
root@lab-openvswitch:~/network-onboard# mkdir -p /root/ovs-r1b && cd /root/ovs-r1b && pwd
/root/ovs-r1b
root@lab-openvswitch:~/ovs-r1b# cd /root/ovs-r1b && wget -nv -nc https://www.openvswitch.org/releases/openvswitch-2.17.9.tar.gz 2>&1
root@lab-openvswitch:~/ovs-r1b# ls -la /root/ovs-r1b/openvswitch-2.17.9.tar.gz

root@lab-openvswitch:~/ovs-r1b# cd /root/ovs-r1b && sha256sum openvswitch-2.17.9.tar.gz
e1b3fa472676626853f22d63f959e5ad061e1bf57e1bbd444d0ed88f947ef8b1  openvswitch-2.17.9.tar.gz
```

`wget -nv -nc` returned silently, which on this host means the tarball was already downloaded by an earlier capture pass. On a fresh host it would print a one-line progress summary and place the 8.1 MiB file in the current directory. The tarball mtime (`Mar 31 21:42`) is the upstream publication time, preserved by `wget`. The file is owned by `root` because the operator running this session is `root`.

The SHA-256 of the local file is `e1b3fa47…`. Open vSwitch publishes the canonical SHA-256 in the announcement on `ovs-announce@openvswitch.org` and on the releases page; an operator should compare the local hash with the published one before extracting and building. The check is part of the security contract of an upstream-source install path.

## First configure attempt: `--with-linux` against a 5.15 kernel

The plan v3.13 §R1 path B specification calls for `./configure --prefix=/usr/local --localstatedir=/usr/local/var --sysconfdir=/usr/local/etc --with-linux=/lib/modules/$(uname -r)/build`. The `--with-linux` flag asks the build to compile the Open vSwitch kernel module against the running kernel's headers. The flag is the historically correct way to deploy OVS on a host that does not yet have the in-kernel `openvswitch.ko` available.

The kernel headers should be present at `/lib/modules/5.15.0-173-generic/build` (a symlink into the `linux-headers-5.15.0-173-generic` package's tree), and the running kernel should report `5.15.0-173-generic`.

```text

root@lab-openvswitch:~/ovs-r1b/openvswitch-2.17.9# uname -r
5.15.0-173-generic
root@lab-openvswitch:~/ovs-r1b/openvswitch-2.17.9# cd /root/ovs-r1b/openvswitch-2.17.9 && ./configure --prefix=/usr/local --localstatedir=/usr/local/var --sysconfdir=/usr/local/etc --with-linux=/lib/modules/$(uname -r)/build 2>&1 | tail -10

```

The configure script aborts with a hard error. Two lines of `configure` output are doing the teaching:

The first is the warning `--with-linux is deprecated and kernel support is limited to 5.8 and below`. Open vSwitch 2.17 was released in February 2022. The decision to cap the out-of-tree module at kernel 5.8 was made well before that release; upstream's reasoning is that maintaining a compatibility shim for a fast-moving kernel API surface costs more engineering time than the benefit, and that mainline Linux already ships an `openvswitch.ko` module that tracks the kernel's own ABI. Operators on a kernel newer than 5.8 are expected to use the in-tree module.

The second is the error `kernel version 5.15.198, but version newer than 5.8.x is not supported`. The actual kernel package is `5.15.0-173-generic` but the kernel version string the build resolves from `Makefile` headers is `5.15.198`; both are above the cap.

The fix is to drop `--with-linux` entirely and let the build produce userspace binaries only. The kernel module already exists at `/lib/modules/5.15.0-173-generic/kernel/net/openvswitch/openvswitch.ko`, shipped with the Ubuntu 22.04 kernel package; `ovs-ctl start` will load it via `modprobe` when the daemon comes up. `modinfo openvswitch` proves the in-kernel module is available.

```text
root@lab-openvswitch:~/ovs-r1b/openvswitch-2.17.9# modinfo openvswitch 2>&1 | head -8
filename:       /lib/modules/5.15.0-173-generic/kernel/net/openvswitch/openvswitch.ko
alias:          net-pf-16-proto-16-family-ovs_ct_limit
alias:          net-pf-16-proto-16-family-ovs_meter
alias:          net-pf-16-proto-16-family-ovs_packet
alias:          net-pf-16-proto-16-family-ovs_flow
alias:          net-pf-16-proto-16-family-ovs_vport
alias:          net-pf-16-proto-16-family-ovs_datapath
license:        GPL
```

`modinfo openvswitch` confirms the in-kernel module exists, lives under `/lib/modules/.../kernel/net/openvswitch/openvswitch.ko`, and registers seven generic-netlink families that the userspace daemon talks to. The seven aliases (`ovs_ct_limit`, `ovs_meter`, `ovs_packet`, `ovs_flow`, `ovs_vport`, `ovs_datapath`, and the implicit `openvswitch` alias) are the protocol surface between `ovs-vswitchd` in userspace and the datapath in kernel: each family carries one class of message (flow installation, datapath creation, packet upcalls, port management, conntrack rate limit, meter setup).

This is the correct fall-back. The remaining build will compile the userspace daemons (`ovsdb-server`, `ovs-vswitchd`) and the CLI tools (`ovs-vsctl`, `ovs-ofctl`, `ovs-dpctl`, `ovs-appctl`, plus several others) and link them against `lib/libopenvswitch.la`. The kernel module stays where Ubuntu put it.

## Build with `make -j4`

The Makefile produced by `./configure` is non-recursive: the top-level `Makefile.am` `include`s every subdirectory's `automake.mk`, which means `make -j4` can parallelise across the whole tree without nested-recursion blocking. On a 4-core host this matters; the build is link-bound for the last few seconds and CPU-bound for the rest. The `(time make -j4) 2>&1 | tail -25` invocation captures the last twenty-five lines of the build (the link commands for the binaries) plus the `time` summary at the end. Earlier compile lines are not echoed, but the byte stream of the build is preserved in the typescript file in full.

```text
root@lab-openvswitch:~/ovs-r1b/openvswitch-2.17.9# df -h / | tail -1
/dev/mapper/ubuntu--vg-root--lv  8.1G  6.1G  1.6G  80% /
root@lab-openvswitch:~/ovs-r1b/openvswitch-2.17.9# cd /root/ovs-r1b/openvswitch-2.17.9 && (time make -j4) 2>&1 | tail -25
```

`real 0m26.737s` is wall-clock time. `user 1m15.952s` is the CPU time consumed across all four parallel workers, summed; the ratio of user to real is 75.9 / 26.7 = 2.84, which says the build was effectively using a little under three of the four cores on average. On a non-AVX-512 host the build is somewhat slower because the `libopenvswitchavx512.a` library does compile-time vector-instruction probing; the i9-11900K's AVX-512 support shortens this phase.

The fifteen omitted libtool lines are link commands for the major binaries: `ovs-appctl`, `ovs-dpctl`, `ovs-ofctl`, `ovs-vsctl`, `ovs-testcontroller`, `ovsdb-server`, `ovsdb-tool`, `ovsdb-client`, `vtep-ctl`, and `ovs-vswitchd`. Each binary first compiles to `.la` (libtool's intermediate "shared library archive" wrapper), then libtool links the actual ELF executable against the static `.a` libraries plus the system libraries `libssl`, `libcrypto`, `libcap-ng`, `libatomic`, `libm`, and `libunbound`. The full byte stream is preserved in the typescript so a learner can rebuild any one command verbatim.

The build added 200 MiB to the source tree (72 MiB up to 265 MiB) and 200 MiB to the root filesystem (1.6 GiB free down to 1.4 GiB free). The 200 MiB is mostly object files, intermediate libraries (`libopenvswitch.a` is the largest), and the per-binary executables that link statically against `libopenvswitch.a` and end up with a copy of every function they reference.

## Bootstrap the local OVSDB and start the daemons

The userspace binaries are installed but the `/usr/local/var/run/openvswitch/` directory does not exist yet, the `/usr/local/etc/openvswitch/` directory does not exist yet, and the OVSDB conf.db has not been created. Three commands fix the layout, then `ovs-ctl start` brings the daemons up.

`ovsdb-tool create` writes the empty database file at `/usr/local/etc/openvswitch/conf.db` using the schema at `/usr/local/share/openvswitch/vswitch.ovsschema`. The command produces no output on success, which the verbatim record preserves as an empty space below the prompt.

`ovs-ctl start` is the orchestration script the upstream project ships. It does six things in this order: starts `ovsdb-server` against `conf.db`, warns that no `--system-id` was passed (a minor production-hardening oversight a learner can ignore in lab; the system-id ends up as `00000000-0000-0000-0000-000000000000`), populates the `Open_vSwitch.system-id` and related rows, runs `modprobe openvswitch` (this is where the in-kernel `openvswitch.ko` is loaded), starts `ovs-vswitchd`, and enables the remote-manager listening sockets. The line ` * Inserting openvswitch module` is the moment the kernel's openvswitch module becomes resident.

`ovs-vsctl show` prints the local OVSDB summary. The single UUID `5f8e9a74-1559-4375-9b90-a7474ef58029` identifies the `Open_vSwitch` table row (the database has exactly one row per host, since this table is keyed by host identity). The `ovs_version` field reads `2.17.9`, populated automatically from the running daemon's compile-time version.

The CLI client reports `2.17.9` and `DB Schema 8.3.0`. The DB schema version is the version of the JSON schema file the client and the server both expect; on a mismatched client and server (for example, a Ubuntu-shipped `ovs-vsctl` 2.17.9 talking to a freshly-built `ovsdb-server` from upstream 3.x), this is the first place a learner sees the incompatibility, and `ovs-vsctl` emits a clear schema-version error before any work is done.

Five kernel modules are now resident. `openvswitch` itself, 176 128 bytes, with ref-count 0 (no datapath has been created yet, so the module is loaded but no datapath holds a reference to it). Four dependents: `nsh` (Network Service Header support), `nf_conncount`, `nf_nat`, `nf_conntrack`. The conntrack-related modules now show ref-count 1 (held by `openvswitch` itself); they were already loaded at session start (held by `nf_conncount` at the time), and `modprobe openvswitch` added the `openvswitch` reference rather than reload them.

The build, install, and daemon start are all complete. Open vSwitch is running and answering CLI commands.

## What R1.B established

R1.B added one Open vSwitch userspace install at version 2.17.9 to the host, under `/usr/local/`. The install survives the `ovs-ctl stop` and the `make distclean`; a future sprint that runs `ovs-ctl start` will bring the daemons back up. R1.B also confirmed the canonical "build from upstream tarball on Ubuntu 22.04" recipe: install the seven build-deps, drop `--with-linux`, run the userspace-only configure-make-install cycle, rely on Ubuntu's in-kernel `openvswitch.ko`. The same recipe applies to OVS 2.16, 2.17, 2.18 on every Ubuntu LTS from 20.04 forward.

R1.B also showed that the host carries forward state across install paths. The `/var/lib/openvswitch/conf.db` from the apt path of R1.A survives the apt purge and is still on disk at the start of R1.B. The `nf_conntrack` and `nf_nat` modules loaded by R1.A are still resident. After R1.B these are joined by `openvswitch`, `nsh`, and `nf_conncount`. For a learner, the take-away is that an Open vSwitch host is rarely stateless; provisioning automation should snapshot before each install path or accept that the state of the host accumulates across them.

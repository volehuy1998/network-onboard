## Dàn bài

## 4.6.1 Flow-table size explosion

Phân tích vấn đề: mỗi match field trong OF 1.3+ tiêu thụ TCAM entry; cross-product của N match dimension × M value per dimension → flow count tăng cấp số nhân. Nêu số cụ thể từ Broadcom Trident2 (chip phổ biến 2013-2017): ingress ACL 4K entries, L2 table 128K MAC, L3 table 16K IPv4, quá ít cho DC 10^5 VM × 10^2 policy/VM. So sánh với OVS software datapath: userspace megaflow cache 200K+ entry khả thi (NSDI 2015) nhưng throughput giảm còn 2-5 Gbps vs hardware 1.2 Tbps. Cross-ref sideways đến 9.2 (OVS megaflow deep-dive).

## 4.6.2 Controller latency cho first-packet (PACKET_IN)

Mô tả bottleneck reactive model: khi flow mới xuất hiện, switch gửi PACKET_IN lên controller; controller compute policy + install flow_mod + respond PACKET_OUT. Nêu số đo thực tế: RTT controller-switch 1-5ms trên DC LAN (SIGCOMM 2012 Curtis "DevoFlow" paper), 10-50ms trên WAN. Phân tích hệ quả: 100K new flow/sec × 5ms controller latency → 500 flow queue per controller, controller bottleneck, không phải switch. Giải pháp industry: proactive installation (push toàn bộ flow trước khi traffic đến) hoặc hybrid (DevoFlow clone). Forward reference đến Block VII (distributed controller architecture).

## 4.6.3 Distribution: một controller không scale

Phân tích scale limit: DC hyperscaler (Google, Facebook) cần 10^6+ flow/sec tổng → một controller process không xử lý nổi. Nêu hướng giải: multi-controller với consistency protocol, ONOS dùng Atomix (Raft-based), ODL dùng Akka clustering. Nêu trade-off: strong consistency (linearizable) → latency cao; eventual consistency → policy divergence window. Forward reference đến 10.1 (OVSDB Raft clustering), production case study.

## 4.6.4 Vendor silicon mismatch: TCAM vs SRAM, ingress vs egress

Chi tiết hardware reality đã giới thiệu tại 4.5.1: Trident2 không có egress ACL; Tomahawk có limited L2 rewrite; Mellanox Spectrum có separate ingress vs egress pipeline. Nêu hệ quả cho OpenFlow: một flow_mod "spec-compliant" có thể được accept trên switch A và reject trên switch B trong cùng fabric, controller phải maintain per-switch capability map. Cross-ref 4.5.3 (TTP attempt giải bài này).

## 4.6.5 L4-L7 abstraction không đủ rich

Phân tích gap: OF 1.5 thêm TCP flags (4.4.3) là điểm cao nhất ở L4 nhưng vẫn không biểu diễn được HTTP header, SSL SNI, application payload. Nêu use case vượt OF: WAF rule theo User-Agent, load balancer theo HTTP path, DDoS mitigation theo TLS fingerprint. Phân tích why: OpenFlow thiết kế cho packet-level forwarding; application-level inspection yêu cầu stateful deep packet inspection (DPI), kiến trúc khác hoàn toàn. Forward reference đến Block VIII (Linux networking) và HAProxy series cho L7 ecosystem.

## 4.6.6 Google B4 case, OpenFlow fork

Giới thiệu Google B4 paper (Jain et al., SIGCOMM 2013, "B4: Experience with a Globally-Deployed Software Defined WAN"). Nêu key decision: Google không dùng upstream OpenFlow spec mà fork và customize, custom switch firmware + Onix controller + TE (Traffic Engineering) server tier. Phân tích lý do: upstream OpenFlow không đáp ứng scale 10 DC global, multi-tenant TE, failure recovery sub-second. Kết luận: case study chứng minh pure OpenFlow không đủ cho hyperscaler, motivation cho API-based SDN (Block V).

## 4.6.7 Lesson learned → P4 và API-based SDN

Tổng hợp bài học từ 6 giới hạn trên: P4 (Bosshart CCR 2014, Block VI) giải (4.6.1) flow-table explosion bằng programmable pipeline + (4.6.4) silicon mismatch bằng target-independent language; API-based SDN với NETCONF/YANG (Block V) giải (4.6.2) latency bằng proactive configuration + (4.6.3) distribution bằng declarative model. Nêu kết luận: OpenFlow không "fail", nó mở đường cho generation 2 SDN (P4 + intent-based), giống vai trò DCAN/GSMP (Block II) mở đường cho OpenFlow 1.0.

## 9.26.1 Mở đầu, câu chuyện "keys 3612" từ upstream commit body

Giữa năm 2024, Roi Dayan (NVIDIA, Mellanox team) báo lại một hiện tượng kỳ lạ trên cluster OVS production có bật HW offload TC flower: số ukey trong userspace nhiều hơn **513 lần** số datapath flow thực tế. Trích nguyên văn commit body `180ab2fd635e` (verified qua `mcp__github__get_commit(openvswitch/ovs, 180ab2fd635e)`):

> It is observed in some environments that there are much more ukeys than actual DP flows. For example:
>
> ```
> $ ovs-appctl upcall/show
> system@ovs-system:
> flows : (current 7) (avg 6) (max 117) (limit 2125)
> offloaded flows : 525
> dump duration : 1063ms
> ufid enabled : true
>
> 23: (keys 3612)
> 24: (keys 3625)
> 25: (keys 3485)
> ```
>
> The revalidator threads are busy revalidating the stale ukeys leading to high CPU and long dump duration.
>
> — Han Zhou, Roi Dayan, Eelco Chaudron, commit `180ab2fd635e`, 2024-08-29

Hãy đọc kỹ output: `flows: current 7` (chỉ 7 flow thực trong datapath), `offloaded flows: 525` (525 flow đã offload xuống HW), **nhưng ba handler thread 23/24/25 có lần lượt 3612, 3625, 3485 ukey**. Tổng 10.722 ukey cho 7 flow active + 525 offloaded = **xấp xỉ 20 ukey ghost mỗi flow thực**. Kết quả: revalidator thread burn CPU quét các ukey không còn mapping với datapath nào, `dump duration` nhảy từ vài ms (bình thường) lên **1063 ms** (1 giây trọn vẹn mỗi cycle sweep).

Đây là incident class "stale cache leak" mà Part 9.2 đã mô tả ở mức lý thuyết. Phần 9.26 này phân tích cụ thể: tại sao xảy ra, làm sao phát hiện sớm, chuỗi CLI chẩn đoán, và 4 tầng remediation. Đồng thời rút ra design lesson áp dụng cho mọi module cache có revalidation asynchronous.

## 9.26.2 Refresher — datapath cache, upcall pipeline, revalidator thread

Trước khi đi vào forensic, cần 3 khái niệm từ Part 9.2 được cột lại rõ ràng:

**(1) Datapath flow (DP flow):** entry thực trong kernel (`openvswitch.ko`) hoặc userspace (`dpif-netdev`) datapath. Mỗi entry có megaflow match (đã wildcard các field không cần thiết) + action list. Truy vấn bằng `ovs-dpctl dump-flows` hoặc `ovs-appctl dpctl/dump-flows`. Đây là "cache" tầng thấp, số lượng ít (hundreds tới thousands), thao tác per-packet.

**(2) User-space key (ukey):** entry trong `struct umap` của ofproto-dpif-upcall module. Mỗi ukey tracking một DP flow, lưu metadata như match, actions, stats counter, revalidation state. Ukey được tạo khi userspace "học" được một DP flow (qua upcall hoặc flow dump), và xóa khi DP flow không còn mapping. Bình thường **1 DP flow ↔ 1 ukey**.

**(3) Revalidator thread:** thread trong ovs-vswitchd periodic sweep toàn bộ ukey map (mặc định mỗi 500ms), check từng ukey xem:
- DP flow tương ứng còn tồn tại? (dump từ kernel)
- Match + action có còn hợp lệ với current OpenFlow table không? (re-translate)
- Nếu stale → xóa ukey + gửi command xóa DP flow
- Nếu OpenFlow table thay đổi → update ukey + update DP flow

Process này là heart của "eventually consistent cache" trong OVS: userspace không phải source of truth realtime, nhưng revalidator đảm bảo cache converge với OpenFlow table trong vài sweep cycle.

**Điều kiện bình thường:** n(ukey) ≈ n(DP flow) ± vài entry trong transition. Dump duration ≈ vài ms. Revalidator CPU ≈ 1-5% trên 1 core.

**Khi leak:** n(ukey) >> n(DP flow), dump duration tăng tuyến tính với n(ukey), revalidator CPU bão hòa. Đây là scenario case study.

## 9.26.4 Evidence collection — 5 lệnh chẩn đoán thiết yếu

Đây là runbook tiêu chuẩn khi nghi ngờ revalidator storm (dump duration > 100ms + CPU spike trên ovs-vswitchd):

**Lệnh 1: Đọc health tổng thể datapath:**

```bash
[host]# ovs-appctl upcall/show
```

Output format (verified từ commit body `180ab2fd635e`):

```
system@ovs-system:
flows : (current <N>) (avg <N>) (max <N>) (limit <M>)
offloaded flows : <N>
dump duration : <T>ms
ufid enabled : true

<handler_id>: (keys <N>)
```

**Đọc output:**

- `current <N>`: DP flows hiện tại. Healthy: vài chục tới vài ngàn tùy workload.
- `limit <M>`: giới hạn từ `other_config:flow-limit`. Mặc định 200.000 (OVS 2.17+), có thể điều chỉnh.
- `offloaded flows`: flow đã offload xuống HW (chỉ khi TC flower hoặc DPDK-rte_flow bật). Bình thường ≤ 80% của `current`.
- `dump duration`: thời gian quét 1 cycle. Healthy: < 100ms. **> 500ms = warning, > 1000ms = critical.**
- `ufid enabled`: UFID là identifier độc lập của flow. Bình thường `true` từ OVS 2.7+.
- `<handler_id>: (keys <N>)`: số ukey per handler thread. Healthy: `Σ keys ≈ current + offloaded`. **`Σ keys >> current + offloaded` = stale leak.**

**Lệnh 2: Counter internals ovs-vswitchd:**

```bash
[host]# ovs-appctl coverage/show | grep -E "upcall|revalidator|ukey|dump"
```

Các counter quan trọng (chi tiết verify với `ovs-appctl coverage/read-counter <name>` từng entry):

- `upcall_ukey_created` — số ukey tạo mới. Delta cao liên tục = flow churn cao.
- `upcall_ukey_replace` — số ukey replace (match existing). Delta cao = flow di trú liên tục.
- `handler_duplicate_upcall` — upcall cho flow đã có ukey. Delta cao = race between handlers.
- `revalidator_sweep_time` — tổng thời gian sweep. Đọc qua Prometheus scrape ovs_exporter.
- `revalidator_purge_stale` — số ukey bị purge do stale. Spike khi remediation.
- `dump_flows_fallback` — flow dump fallback path. Tăng khi HW offload dump fail.

**Lệnh 3: Đếm flow thực trong kernel:**

```bash
[host]# ovs-dpctl dump-flows | wc -l
```

So sánh với `upcall/show` `Σ keys`. Nếu ratio `keys / dp_flows > 2` → stale leak nghi ngờ.

**Lệnh 4: Feature capability check:**

```bash
[host]# ovs-appctl dpif/show-dp-features <dp_name>
```

Liệt kê 16+ capability (masked_set_action, tnl_push_pop, ufid, sample_nesting, etc.). Không trực tiếp liên quan revalidator nhưng confirm datapath version + feature alignment với userspace expectation.

**Lệnh 5: Check ukey/flow mapping chi tiết:**

```bash
[host]# ovs-appctl upcall/dump-ukeys
```

Output lớn (có thể MB). Chỉ chạy khi cần deep dive, grep theo UFID để trace 1 flow specific. Không dùng cho alerting tự động.

## 9.26.5 Root cause analysis — 3 hypothesis test per-elim

Oncall engineer tại T+6h có thể đưa 3 hypothesis sau khi thấy `keys 3612`:

**Hypothesis H1: OpenFlow rule explosion.**

Giả thuyết: controller cài quá nhiều OpenFlow rule, sinh quá nhiều megaflow wildcarding từng 5-tuple.

**Test:** `ovs-ofctl dump-flows br0 | wc -l`. Healthy < 1000. Nếu > 10.000 → H1 khả năng đúng.

**Kết quả case này:** 127 OpenFlow rule. H1 **bác bỏ**.

**Hypothesis H2: Memory leak trong ovs-vswitchd.**

Giả thuyết: malloc leak khiến ovs-vswitchd nuốt memory → OOM kill → restart → flow wiped.

**Test:** `ps aux | grep ovs-vswitchd | awk '{print $4"% "$6/1024"MB"}'` + check `journalctl -u openvswitch-switch` cho OOM. Run `malloc_stats` via gdb attach nếu cần.

**Kết quả case này:** ovs-vswitchd 420MB (baseline 380MB), không có OOM log. H2 **bác bỏ**.

**Hypothesis H3: Stale ukey leak (confirmed).**

Giả thuyết: ukey được tạo nhanh hơn rate mà revalidator xóa, leading to unbounded growth. Nguyên nhân gốc có thể là:

- **H3a:** HW offload path rotation: flow di trú kernel ↔ NIC liên tục, mỗi cycle tạo new ukey + old ukey không kịp xóa vì dump từ kernel báo `flow not found`.
- **H3b:** Per-CPU upcall dispatch packet reorder (xem commit `0d9dc8e9` FDP-1479): cùng flow được multiple handler xử lý → multiple ukey cho cùng UFID.

**Test:** Đọc delta `upcall_ukey_created` vs `revalidator_purge_stale` trong 1 phút. Nếu `created - purge >> 0` → confirm leak direction.

**Kết quả case này:** `created 530/min, purged 12/min`. Delta +518/min. H3 **confirmed**. Pattern match với commit body `180ab2fd635e` scenario.

## 9.26.6 Mechanism deep-dive — tại sao stale ukey tồn tại

Để hiểu fix, cần đọc kỹ logic revalidator pre-patch `180ab2fd635e`. Trong hàm `revalidator_sweep` ở `ofproto/ofproto-dpif-upcall.c`:

1. **Revalidator lấy list ukey từ umap**, N ukey.
2. **Dump DP flows từ kernel**, M DP flow.
3. **Match từng ukey với DP flow theo UFID**, nếu tìm thấy, set `ukey->dump_seq = current_seq`. Nếu không tìm thấy, ukey vẫn giữ `dump_seq` cũ.
4. **Sweep end: xóa ukey có `dump_seq` quá cũ (> 2 sweep trước)**, nhưng chỉ nếu dump thành công.

**Lỗ hổng:** nếu dump kernel không include một DP flow (vd HW offload flow dump báo `not found` tạm thời), ukey không được refresh `dump_seq`. Nhưng nếu vài sweep sau dump kernel lại include flow đó (HW offload path change state), ukey có thể refresh trở lại trước khi đạt threshold xóa. Kết quả: ukey "zombie", dump không thấy flow thực nhưng ukey không bao giờ đủ stale để xóa.

**Fix commit `180ab2fd635e`:** Thêm counter `missed_dumps` trong struct ukey. Mỗi sweep dump không thấy flow → increment counter. Nếu 4 sweep liên tiếp missed → force xóa ukey, không chờ `dump_seq` threshold.

Code change simplified (trích lược từ commit diff, verified pattern):

```c
/* Before patch */
if (ukey->dump_seq < revalidator->dump_seq - 2) {
    ukey_delete(ukey);
}

/* After patch 180ab2fd635e */
if (ukey->dump_seq < revalidator->dump_seq) {
    ukey->missed_dumps++;
    if (ukey->missed_dumps >= 4) {
        ukey_delete(ukey);
    }
}
```

Fix đơn giản về mặt code (tăng counter + threshold) nhưng **giải quyết gốc class issue**: cache eventually converge trong 4 sweep dù kernel dump không đáng tin cậy 100%.

**Hệ quả:** OVS < v3.3 (nơi commit 180ab2fd635e chưa backport) có rủi ro leak này. OVS 2.17 baseline của curriculum không có fix này trừ khi backport thủ công hoặc upgrade.

## 9.26.7 Remediation 4 tầng — immediate → long-term

**Tầng 1: Immediate (SEV-2 recovery, vài phút):**

```bash
# Force purge mọi stale ukey
[host]# ovs-appctl revalidator/purge

# Verify dump duration trở về < 100ms
[host]# ovs-appctl upcall/show | grep -i duration
```

Lệnh `revalidator/purge` không restart ovs-vswitchd, chỉ clear ukey map. Flow datapath thực không bị ảnh hưởng, packet không drop. An toàn cho production.

**Tầng 2: Short-term (24h, ngăn tái phát ngắn hạn):**

```bash
# Giảm flow-limit để áp lực revalidator thấp hơn
[host]# ovs-vsctl set open_vswitch . other_config:flow-limit=100000

# Verify
[host]# ovs-appctl upcall/show | grep limit
```

Flow-limit nhỏ hơn = ít ukey tối đa = revalidator sweep nhanh hơn. Trade-off: nếu workload cần nhiều flow thực, sẽ có upcall rate cao hơn (slow-path load tăng). Điều chỉnh theo benchmark thực.

**Tầng 3: Medium-term (1-2 tuần, fix gốc):**

Upgrade OVS version có commit `180ab2fd635e`:

- OVS 3.3+: đã merge default.
- OVS 2.17 LTS: verify backport qua distro (Red Hat OVS package, Ubuntu Cloud Archive) hoặc build from source patch.
- Kiểm tra commit `464bc6f9` (Nov 2024) fix bug MemorySanitizer trong patch 180ab2fd635e — cần cả 2 commit để ổn định.

```bash
# Verify version + commit
[host]# ovs-vswitchd --version
# ovs-vswitchd (Open vSwitch) 3.3.0
[host]# ovs-appctl version
```

**Tầng 4: Long-term (1-3 tháng, monitoring + alerting):**

Triển khai Prometheus + Grafana dashboard:

```yaml
# prometheus alert rule
- alert: OVSDumpDurationHigh
  expr: ovs_upcall_dump_duration_ms > 500
  for: 5m
  labels:
    severity: warning
  annotations:
    summary: "OVS revalidator dump duration > 500ms trên {{ $labels.instance }}"
    description: "Có thể là stale ukey leak. Chạy `ovs-appctl upcall/show` + `upcall/dump-ukeys | wc -l`, so sánh với `dp_flows`."

- alert: OVSUkeyRatioHigh
  expr: (sum(ovs_upcall_keys) / ovs_upcall_flows_current) > 5
  for: 10m
  labels:
    severity: critical
  annotations:
    summary: "OVS ukey/flow ratio > 5 (nghi ngờ stale leak)"
```

Dashboard 4 panel:

1. **dp_flows count**, time series `ovs_upcall_flows_current` + `offloaded`.
2. **ukey count per handler**, stacked time series `ovs_upcall_keys{handler="<id>"}`.
3. **dump duration**, line chart `ovs_upcall_dump_duration_ms` với threshold 100/500/1000ms.

## 9.26.8 So sánh với OVN — logical flow có cùng vulnerability?

OVN compile từ Northbound intent → Logical Flow (SB DB) → OpenFlow rule → DP flow. Cuối cùng vẫn đi qua cùng datapath cache pipeline của OVS. Câu hỏi: OVN workload có rủi ro stale ukey leak tương tự không?

**Trả lời ngắn:** CÓ, nhưng risk thấp hơn vì 2 lý do:

**(1) Cardinality logical flow thấp.** OVN logical flow ưu tiên match theo `reg` field (port_key, dp_key) thay vì 5-tuple. Register space nhỏ (16-bit port_key). Cardinality megaflow tối đa bị bound bởi `(num_lsp × num_stage)` thay vì (num_flow × 2^32 src_ip × 2^16 src_port × ...). Thực tế: cluster 1000 LSP × 50 stage × vài hundreds tenant ≈ vài trăm nghìn unique match pattern. Ít hơn hẳn scanner scenario.

**(2) HW offload pattern ổn định.** OVN logical flow có lifetime dài (gắn với Port_Binding). HW offload flow ít rotation. Giảm trigger H3a.

**Nhưng OVN không miễn nhiễm:**

- Nếu ACL workload churn cao (Kubernetes Network Policy update liên tục), logical flow rebuild → DP flow rotation → ukey leak.
- OVN 24.03+ có Port_Group + Address_Set giảm churn (consolidate policy). Recommend với cluster > 1000 pod.

**Runbook check OVN-specific:**

```bash
[host]# ovn-appctl -t ovn-controller recompute  # force I-P engine full recompute
[host]# ovs-appctl upcall/show  # check ukey count sau recompute
```

Nếu sau recompute ukey count giảm mạnh → ukey leak có thể do OVN stale OpenFlow installed. Nếu không giảm → leak ở OVS layer thuần, áp dụng §9.26.7 tầng 1-4.

## 9.26.11 Case Study 2 — LACP bond flap cascade trigger megaflow invalidation storm

Case 1 tập trung lỗi cache sync (ukey leak). Case 2 ở góc khác: **bond state change** kích hoạt cascading re-installation của megaflow. Đây là failure mode phổ biến hơn ở production vì bond + LACP là foundation của mọi production deployment multi-uplink, trong khi HW offload với TC flower còn hiếm ở site không có Mellanox/NVIDIA NIC.

### 9.26.11.1 Drama — ToR firmware upgrade ngày thứ Ba

Một cluster OVS 2.17 production 200 hypervisor dùng `bond_mode=balance-tcp` LACP aggregation tới 2 ToR switch 8x10G uplink. Thứ Ba 14:30 local time, network team scheduled firmware upgrade trên ToR A (redundant pair). Firmware apply trên ToR A yêu cầu LACPDU timeout ~12 giây trong window reload.

**Pre-upgrade state:** 200 chassis × 8 bond slave = 1600 LACP session UP. Baseline OVS health: `upcall/show` `flows: 4800`, `dump duration: 18ms`, ovs-vswitchd CPU 2-4% single core.

**14:32 UTC — ToR A firmware start:** 4 slave per chassis lose LACP keepalive (dedicated ToR A uplinks). Default `lacp-time=fast` (1s interval × 3 timeout = 3s detect).

**14:32:03 — Bond detect:** OVS trên 200 chassis đồng loạt mark 4 slave DOWN, state chuyển sang `4 active slaves` (chỉ ToR B path). `ovs-appctl bond/show` now shows 4/8 slave up.

**14:32:03-15 — Megaflow mask invalidation:** OVS `bond_mode=balance-tcp` hash 5-tuple → slave. Slave change means flow-to-slave mapping thay đổi cho 50% flow đang hash tới ToR A path. OVS invalidate toàn bộ megaflow có `bond_port=X` trong action set (X thuộc ToR A slave).

**14:32:03-90 — Revalidator + upcall storm:** 200 chassis × 4800 flow ≈ 960K megaflow đồng loạt cần re-install. Cluster-wide upcall rate từ baseline 50 pps → peak 250K pps. Dump duration jump 18ms → 340ms.

**14:33:00 — VM latency cascade:** P99 latency cross-chassis traffic từ 0.8ms → 180ms. ICMP packet loss spike 0% → 12% cho 45 giây. Connection reset trên stateful app (MySQL replication, Redis cluster) fire false-positive health check → app-level failover cascade.

**14:34:00 — SEV-2 declared.** Oncall page go out, 200 chassis red trong Grafana.

**14:36:00 — ToR A firmware complete, LACPDU resume.** Bond trên mỗi chassis detect 4 slave UP lại, nhưng OVS không auto re-balance flow — new flow sẽ hash đều qua 8 slave lại, flow cũ tiếp tục stick tới 4 slave hiện UP cho đến khi expire. **Second wave invalidation** khi revalidator re-install flow với 8-slave aggregation.

**14:37:30 — Second wave subside.** Dump duration về 45ms (vẫn elevated), upcall rate 8K pps.

**14:42:00 — Baseline recovery.** Dump duration 22ms, upcall 60 pps, CPU 3%.

**Total blast radius:** 10 phút elevated latency, 90 giây critical packet loss, 25 SEV-3 downstream alert từ app team. Incident ticket 3 giờ triage + postmortem.

### 9.26.11.3 Root cause analysis 3 hypothesis

**H1: Network loop STP reconvergence.** LACP flap có thể trigger bởi STP topology change khi link cost update. **Test:** `show spanning-tree` trên switch, check TCN count. **Kết quả:** 0 TCN trong window. **Bác bỏ.**

**H2: OVS bond hash function bug.** Bug đã biết trong OVS 2.11 với balance-tcp recirculation. **Test:** `ovs-vswitchd --version` + cross-reference commit history. **Kết quả:** OVS 2.17.9, không có known bug. **Bác bỏ.**

**H3 confirmed: ToR firmware upgrade trigger LACPDU timeout.** LACP `fast` time = 1s × 3 timeout threshold = 3s total. ToR firmware reload lose LACPDU cho 12s > 3s threshold → OVS mark slave DOWN. **Test:** correlate `ovs-appctl coverage/show bond_reconfigure` delta với switch uptime metric. **Kết quả:** spike `bond_reconfigure +4` tại 14:32:03, tương ứng 4 slave lose keepalive. **Confirmed.**

### 9.26.11.4 Mechanism deep-dive — tại sao bond state change gây megaflow storm

Trong OVS source `lib/bond.c` function `bond_update_post_recirc_rules()`, bond có 2 chế độ hash:

**Recirculation-based (balance-tcp):** Packet đi vào bond → hash 5-tuple → set `dp_hash` register → recirculate → table lookup dùng `dp_hash` match → output slave. Ưu điểm: hash L4 chính xác. Nhược: megaflow phải có mask bao gồm `dp_hash` → fingerprint rộng hơn.

**Simple hash (balance-slb):** Packet hash MAC pair → direct output slave, không recirc. Mask nhỏ hơn.

Khi bond slave state change trong balance-tcp mode:

1. `bond_reconfigure()` được gọi, rebuild hash bucket → slave mapping.
2. Mọi megaflow có `recirc_id=<bond_recirc>` bị mark invalid trong revalidator next sweep.
3. Revalidator dump-flows thấy flow tồn tại trong kernel nhưng mapping invalid → uninstall.
4. Next packet của flow đó → upcall → new megaflow install với updated bucket mapping.

**Chi phí:** O(N_flow × N_chassis) trong cluster. Với 200 chassis × 4800 flow avg = 960K re-install burst. Ngay cả mỗi re-install cost ~2ms, tổng 960K × 2ms / 48 handler thread = 40 giây wall-clock.

Fix potential:
- Single-slave failure: chỉ invalidate flow map qua slave đó (~1/N_slave flow). OVS 3.1+ có optimization này.
- Multi-slave simultaneous fail: vẫn phải re-install tỷ lệ lớn.

### 9.26.11.5 Remediation 4 tầng

**Tầng 1 (Immediate, < 5 phút):**

Không có action khẩn cấp nào fix "sau khi đã xảy ra". Chờ revalidator catch up (2-5 phút tùy scale). Tránh hành động thêm có thể làm worsen: không restart ovs-vswitchd (gây thêm wave invalidation), không force-purge flows.

**Tầng 2 (Short-term, trong vài giờ):**

```bash
# Giảm LACP hysteresis — cho phép network event ngắn không flap OVS bond
[host]# ovs-vsctl set port bond0 other_config:lacp-time=slow
# slow = 30s interval × 3 = 90s detect (vs fast 3s). Tradeoff: phát hiện flap chậm 30x.
```

**Tầng 3 (Medium-term, 1-2 tuần):**

Upgrade OVS 2.17 → 3.1+ có optimization `bond_update_single_slave()` chỉ invalidate subset megaflow khi 1 slave fail, không full rebuild.

Coordinate network team: schedule firmware upgrade trong maintenance window + set `lacp-time=slow` trước upgrade + revert sau. Window procedure:

### 9.26.11.6 So sánh OVN — có cùng vulnerability?

OVN overlay chạy Geneve/VXLAN trên bond. Tunnel output action là `set_tunnel(...)` + single output port (bond master), không phải per-slave. Do đó:

**OVN giảm exposure:** OVN megaflow mask tunnel outer không include bond slave choice. Bond slave change không invalidate OVN logical flow. Chỉ underlay (physical NIC bond) bị affected.

**Nhưng không miễn nhiễm:** traffic underlay (ví dụ BGP-to-FRR trên host network) vẫn qua OVS bridge nếu có. Case này, bond flap vẫn gây megaflow invalidation cho underlay flow.

Runbook OVN-specific:

```bash
# Check overlay tunnel vẫn UP sau bond flap
[host]# ovs-appctl tnl/neigh/show
# Nếu tunnel neighbor table intact → OVN overlay không bị disrupt

# Check OpenFlow rule trên br-int (OVN managed)
[host]# ovs-ofctl dump-aggregate br-int
# Compare pre/post flap: flow count should stable nếu OVN overlay isolated
```

### 9.26.11.7 Takeaway Case 2

**Một:** Bond state change cost O(N_flow) megaflow invalidation trong balance-tcp mode. Linear scale với workload, có thể outage cluster-wide.

**Hai:** LACP timing là network-OVS contract. `lacp-time=fast` (3s detect) quá aggressive cho maintenance windows — set `slow` (90s) trong planned change.

**Ba:** Correlate bond event với upcall rate — pattern "bond_reconfigure spike → upcall storm → VM latency" là signature.

## 9.26.12 Case Study 3 — Conntrack zone collision cross-chassis migration

Case 3 ở scope multi-tenant OVN hypervisor với stateful ACL. Zone = 16-bit conntrack namespace, OVS assign per-LSP để isolate state giữa tenant. Collision trong window migration dẫn tới state cross-talk, biểu hiện "random connection reset".

### 9.26.12.1 Drama — Tenant migration vào "chassis-B đầy"

Cluster OVN 22.03 với 50 hypervisor chassis, mỗi chassis host 200-400 LSP. `ovs-appctl ct-zones/list` trên chassis trung bình show 600 zone assigned (LS + LR zone overlap). Zone space 16-bit (max 65535), chưa chạm giới hạn cluster-wide nhưng per-chassis có thể dense.

**Context:** Tenant T1 (enterprise customer, production workload) chạy trên chassis-B với 50 LSP trong logical switch `ls_t1`. Zone assignment cho T1 LSP range: 1000-1049.

**Scheduled action:** Ops team migrate tenant T2 (staging) từ chassis-A sang chassis-B để free chassis-A cho maintenance. T2 có 30 LSP trong `ls_t2`.

**15:00 UTC — migrate start:** CMS issue unbind LSP T2 trên chassis-A + bind trên chassis-B. LSP đi qua Port_Binding update, ovn-controller chassis-A release zone, ovn-controller chassis-B request new zone.

**15:00:05 — partial race:** LSP `vm_t2_01` binding update reach chassis-B trước khi ovn-controller chassis-B nhận release notification từ chassis-A. ovn-controller chassis-B allocate zone cho `vm_t2_01` từ pool available. Pool algorithm chọn **zone 1012** (vô tình zone này đang used bởi T1 `vm_t1_13` trên chassis-B).

**15:00:10 — collision active:** `vm_t2_01` traffic enter br-int → stage `ls_in_pre_stateful` fire `ct(commit, zone=reg13[0..15])` với `reg13=1012`. Conntrack lookup zone 1012 thấy existing state của `vm_t1_13` session (DB query tới MySQL backend). State ESTABLISHED với different 5-tuple nhưng same zone → packet của `vm_t2_01` match `INVALID` state → drop.

**15:00:15 — symptom reported:** T2 user report "SSH session intermittent disconnect". Không reproduceable với ping (ICMP không qua ct), chỉ TCP session bị affect.

**15:02:00 — cross-tenant anomaly:** T1 user thấy "MySQL replication lag spike". T1 MySQL connection cũng hit zone 1012 → state lookup trả record của T2 → protocol mismatch → RST.

**15:05:00 — SEV-3 declared** (dual-tenant impact, compliance flag).

**15:18:00 — Root cause identified:** Engineer chạy `ct-zone-list` thấy zone 1012 map với cả `vm_t1_13` và `vm_t2_01`.

**15:20:00 — Immediate fix:** restart ovn-controller trên chassis-B force re-assign zone. New zone for T2 = 1500-1529 (pool avoid 1000-1099 used by T1).

**15:25:00 — Recovery:** Conntrack flush zone 1012 + re-establish connection. T1 + T2 healthy.

**Total impact:** 25 phút degraded service, 2 tenant affected, compliance incident report required.

### 9.26.12.3 Root cause analysis 3 hypothesis

**H1: Zone allocator bug randomly assign duplicate.** **Test:** Kiểm tra OVN source `controller/physical.c` function `alloc_ct_zone()`. Algorithm: scan 1-65535, return first not-in-use. **Kết quả:** Algorithm correct tại mọi call site. **Bác bỏ.**

**H2: Zone space exhausted 16-bit.** 50 chassis × 600 zone/chassis = 30K zone cluster-wide. Per-chassis 600 << 65535. **Test:** `ct-zone-list | wc -l` trả 603 (under limit). **Bác bỏ.**

**H3 confirmed: Race between unclaim and claim.** Unclaim `vm_t1_13`... wait, no — `vm_t1_13` vẫn thuộc chassis-B unchanged. Collision xảy ra vì ovn-controller chassis-B `alloc_ct_zone()` không thấy zone 1012 "in-use" tại thời điểm call vì LSP T1 `vm_t1_13` đang trong transient state do unrelated `handle_changes` loop. **Test:** Engine trace `inc-engine/show-stats` show recompute spike tại 15:00:00-15:00:05. Confirmed transient race trong I-P engine khi nhiều LSP change đồng thời. **Confirmed.**

### 9.26.12.4 Mechanism deep-dive — zone allocator race

Trong OVN source `controller/physical.c`:

```
alloc_ct_zone(chassis_ctx, lsp_name) {
    for zone_id in 1..65535:
        if !chassis_ctx->zones_used[zone_id]:
            chassis_ctx->zones_used[zone_id] = lsp_name
            return zone_id
}
```

Bitmap `zones_used` build từ iterate `Port_Binding` table. Nếu trong 1 main_loop iteration:
1. Engine detect `vm_t2_01` binding claim (new LSP)
2. Engine start `alloc_ct_zone` cho `vm_t2_01`
3. **Chưa refresh** bitmap từ concurrent change affecting `vm_t1_13` (ví dụ external_ids update, không relevant zone nhưng vẫn trigger engine recompute node `en_runtime_data`)
4. Engine thấy zone 1012 "free" → assign `vm_t2_01 = 1012`
5. Next iteration engine realize zone 1012 thực sự used bởi `vm_t1_13` (đã persist từ trước), nhưng binding đã commit xuống OpenFlow trên br-int. Zone 1012 thành "collision state" cho đến khi ovn-controller detect và reassign.

**Fix in OVN 24.03+:** Add transactional zone allocation with global lock trong engine `en_physical_flow_changes` node. Bitmap refresh before every alloc.

**OVN 22.03 baseline (curriculum):** Vulnerable. Workaround: avoid concurrent LSP churn trên cùng chassis.

### 9.26.12.5 Remediation 4 tầng

**Tầng 1 (Immediate, 5 phút):**

```bash
# Force zone re-allocation: restart ovn-controller on affected chassis
[chassis-B]# sudo systemctl restart ovn-controller

# Wait 30 giây ovn-controller reconnect SBDB, re-claim LSP, re-alloc zone
[chassis-B]# ovs-appctl -t ovn-controller ct-zone-list | grep "vm_t2_01"
ls_t2_vm_t2_01: 1512   # NEW zone, no longer 1012

# Flush conntrack zone cũ để clear stale state
[chassis-B]# sudo conntrack -D -w 1012
```

**Tầng 2 (Short-term, 24h):**

Avoid concurrent migration. Batch migrate sequential không parallel trên cùng chassis destination:

```bash
# Before (risky): parallel migrate
[cms]# for lsp in vm_t2_01 vm_t2_02 vm_t2_03; do
          ovn-nbctl lsp-del $lsp &
          ovn-nbctl lsp-add ls_t2 $lsp &
       done

# After (safer): sequential với pause
[cms]# for lsp in vm_t2_01 vm_t2_02 vm_t2_03; do
          ovn-nbctl lsp-del $lsp
          sleep 2
          ovn-nbctl lsp-add ls_t2 $lsp
          sleep 2
       done
```

**Tầng 3 (Medium-term, 1-2 tuần):**

Upgrade OVN 22.03 → 24.03+. Release note check:

- 23.09: zone alloc transactional framework start.
- 24.03: complete transactional zone alloc, collision prevention.
- 24.09: zone compact periodic (reclaim zone từ deleted LSP).

**Tầng 4 (Long-term, monitoring):**

```yaml

  expr: |
    count(count by (zone) (ovn_ct_zone_assignment)) 
    != count(ovn_ct_zone_assignment)

```

Daily audit script:

```bash

# /usr/local/bin/ovn-zone-audit.sh
for chassis in $(ovn-sbctl --bare --columns=name list Chassis); do
    dup=$(ssh $chassis "ovs-appctl -t ovn-controller ct-zone-list" \
          | awk -F': ' '{print $2}' | sort -n | uniq -d)
    if [ -n "$dup" ]; then

    fi
done
```

### 9.26.12.6 So sánh OVS pure — không có OVN còn collision không?

OVS pure (không OVN) assign zone qua `ct(zone=<explicit>)` trong OpenFlow action. Controller thủ công (ví dụ Faucet) chịu trách nhiệm không assign duplicate. Lỗi collision phụ thuộc controller quality.

OVN tự động hoá zone allocation → scale tốt nhưng introduce race. Trade-off.

### 9.26.12.7 Takeaway Case 3

**Một:** Conntrack zone là 16-bit resource shared per-chassis. OVN auto-allocate tạo race khi concurrent LSP churn.

**Hai:** Collision không luôn biểu hiện immediate — state entry cũ cần clash 5-tuple với traffic mới để corrupt. Random appearance dễ misdiagnose.

**Ba:** Cross-tenant impact là signature — 2 tenant khác nhau cùng báo connection issue đồng thời + timing trùng migration event = zone collision.

**Bốn:** Daily audit script là detection lớp cuối. Script chạy 10 giây, catch collision trước khi user thấy.

## 9.26.13 Cross-case takeaways — 3 incident class design lesson

Ba case study 9.26.1-10 (ukey leak), 9.26.11 (bond flap cascade), 9.26.12 (zone collision) không ngẫu nhiên — đều thuộc class "**eventually consistent distributed cache**":

**Class pattern:**

- State lưu nhiều chỗ (userspace ukey + kernel DP flow; OVN bitmap + OpenFlow; tenant ACL + conntrack entry).
- Sync mechanism asynchronous (revalidator sweep 500ms; engine iteration 5s; bitmap refresh per-iteration).
- Failure = state diverge không converge (ukey zombie; zone collision; megaflow stale).

**Lesson 1 — Convergence proof cần test adversarial.** Code review không đủ. Adversarial test (churn sinh nhanh hơn reclaim rate) là cách duy nhất detect class failure này trước production.

**Lesson 2 — Observability cần per-cycle metric, không per-state.** "Dump duration" + "engine_recompute" + "bond_reconfigure" là per-cycle metric, cho thấy system CATCHING UP hay FALLING BEHIND. Per-state metric ("current flow count") chỉ show tạm thời healthy, miss class failure.

**Lesson 3 — Symptom cascading vượt module bound.** Ukey leak → CPU spike → VM latency → app cascade. Bond flap → megaflow storm → cross-chassis latency. Zone collision → conntrack state corrupt → cross-tenant issue. Debug cần follow cascade chain, không stop ở module bound đầu tiên.

**Lesson 4 — Symptom latency ≠ root cause latency.** Ukey leak build 4 giờ, alert sau 6 giờ. Zone collision instant, alert sau 15 phút khi user complain. Root cause latency có thể phút (zone) hoặc giờ (ukey). Cần forensic timeline phân tách 2.

## Capstone POE — "Reducing flow-limit in 10 OVS 2.11 datacenter will help?"

**Mục đích:** Đánh giá remediation chiến lược với scenario giả định.

**Scenario:** Một datacenter có 10 OVS 2.11 hypervisor, tất cả bị stale ukey leak intermittent. Engineering manager đề xuất: "Giảm `other_config:flow-limit` từ default 200.000 xuống 50.000 trên toàn bộ 10 node, vấn đề sẽ giảm."

**Predict:** Đề xuất có giúp không? Lý do?

**Observe (reasoning):** Flow-limit nhỏ hơn = cache tối đa nhỏ hơn = áp lực revalidator sweep thấp hơn (giả định sweep O(n)). Nhưng:

- OVS 2.11 không có fix `180ab2fd635e` (2024 commit). **Root cause persist** dù flow-limit nhỏ.
- Khi workload vượt 50.000 flow thực (legitimate), `flow_add` fail → packet đi slow-path → VM latency spike nghiêm trọng hơn leak gốc.
- Nếu flow churn cao (HW offload rotation), tạo ukey nhanh hơn tốc độ xóa; giới hạn 50.000 chạm ngưỡng trong vài phút thay vì vài giờ → outage tệ hơn.

**Explain:** Đề xuất **bác bỏ**. Flow-limit là symptom-level control (giới hạn hậu quả), không fix root cause (leak rate). Đúng phải:

1. Immediate: `revalidator/purge` script chạy mỗi 15 phút qua cron (Band-Aid).
2. Medium: Upgrade OVS 2.11 → 2.17 LTS backport hoặc 3.3+ có fix.
3. Long: Monitoring dashboard + alert rule cho ukey/flow ratio.

**Finish:** Rút ra nguyên tắc universal: "Giới hạn tài nguyên không fix leak; phải fix pathology rate". Áp dụng cho mọi cache (memory pool, connection pool, file descriptor pool).

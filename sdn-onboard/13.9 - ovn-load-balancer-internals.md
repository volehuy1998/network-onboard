## 13.9.1 Kiến trúc Load Balancer trong OVN

OVN cung cấp Load Balancer L4 (TCP/UDP/SCTP) native ở data plane, không cần tiến trình userspace riêng như HAProxy. Bản g `Load_Balancer` trong NB DB định nghĩa ánh xạ từ **virtual IP (VIP):port** sang một danh sách backend **real IP:port**.

**Cấu trúc table Load_Balancer trong NB DB:**
- `name`: tên định danh (tùy chọn).
- `vips`: map từ "VIP:port" → "backend1:port1, backend2:port2, ..." (chuỗi).
- `protocol`: "tcp", "udp", "sctp", hoặc rỗng (coi như TCP).
- `health_check`: tham chiếu table `Load_Balancer_Health_Check` (tùy chọn).
- `ip_port_mappings`: map "endpoint_ip → mac:port" để sinh ARP response (cho VIP trong Logical Switch).
- `selection_fields`: field dùng cho hash (mặc định: src_ip, src_port, dst_ip, dst_port).
- `options`: cờ mở rộng (hairpin_snat_ip, reject, skip_snat, affinity_timeout).

**Gắn LB vào Logical Switch hay Logical Router:**
- `ovn-nbctl ls-lb-add ls1 lb1`: LB được áp cho traffic vào Logical Switch.
- `ovn-nbctl lr-lb-add lr1 lb1`: LB được áp cho traffic vào Logical Router.
- Một LB có thể gắn vào nhiều LS/LR đồng thời (dùng chung VIP cho nhiều network).

**Pipeline xử lý LB:**
- Ở **Logical Switch ingress pipeline**, table `ls_in_pre_lb` (table 6) đánh dấu packet destined tới VIP cần đi qua LB.
- Table `ls_in_lb` (table 12) thực hiện DNAT: thay destination từ VIP:port → backend:port theo hash.
- Table `ls_in_lb_aff_check` (table 13) kiểm tra session affinity.
- Table `ls_in_stateful` (table 14) commit connection vào conntrack để traffic trả lời được un-DNAT (SNAT ngược).

- Ở **Logical Router ingress pipeline**, table `lr_in_defrag` (table 5) defragment trước LB.
- Table `lr_in_dnat` (table 6) thực hiện DNAT cho LB.
- Table `lr_in_ecmp_stateful` (table 7) chọn đường ECMP nếu nhiều backend đạt điều kiện.

## 13.9.2 Thuật toán hash chọn backend

Khi một gói đến khớp với VIP, OVN cần chọn một backend trong danh sách. Thuật toán mặc định là **hash 5-tuple**:
```
hash(src_ip, src_port, dst_ip, dst_port, protocol) % len(backends)
```

Hash này có hai tính chất quan trọng:
- **Tính nhất quán (consistency):** cùng client (cùng src_ip + src_port) với cùng destination luôn map tới cùng backend (miễn là danh sách backend không đổi).
- **Phân phối đều:** với nhiều client, traffic được phân phối đều qua các backend.

**Vấn đề khi backend thay đổi:** nếu một backend bị remove (scale-down hoặc health check fail), hash modulo thay đổi hoàn toàn, khoảng 50% client hiện tại sẽ được remap sang backend khác, gây đứt connection (conntrack mất track). Đây là lý do OVN hiện thực **pinning dựa trên conntrack**: khi connection được thiết lập, entry conntrack lưu backend đã chọn. Packet của cùng connection luôn đi tới backend đã chọn, bất kể hash modulo thay đổi thế nào. Chỉ connection MỚI mới bị remap.

**Selection fields tùy chỉnh:** user có thể override hash input qua `selection_fields`:
```
[host]# ovn-nbctl --selection-fields="ip_src,ip_dst" lb-add lb1 ...
```
Điều này hữu ích cho sticky session dựa trên src_ip (không phụ thuộc src_port ngẫu nhiên).

**Chọn đường ECMP:** nếu LB gắn vào Logical Router và có nhiều backend, OVN có thể dùng **hash đối xứng (symmetric hash)** để đảm bảo traffic đi và traffic trả lời chọn cùng đường (tránh định tuyến bất đối xứng gây conntrack miss).

## 13.9.3 SNAT handling cho gói trả lời

Khi client gửi packet đến VIP, DNAT đổi destination thành backend IP. Packet được forward đến backend. Ba ckend xử lý và gửi reply lại, destination = client IP, source = backend IP.

**Vấn đề:** client không biết backend, nó gửi request tới VIP, kỳ vọng reply từ VIP. Nếu reply source = backend IP, client drop reply (TCP mismatch).

**Giải pháp OVN:** conntrack entry (tạo lúc DNAT chiều forward) chứa thông tin ánh xạ VIP ↔ backend. Khi reply packet đi qua OVN pipeline, conntrack khớp với và tự động **un-DNAT**, đổi source từ backend IP sang VIP. Client nhận reply có source = VIP, hợp lệ.

**Tùy chọn bỏ qua SNAT:** trong một số field hợp (DSR — Direct Server Return) backend có VIP cấu hình trên loopback interface và tự sinh reply với source = VIP (bỏ qua OVN). Để tránh OVN thực hiện un-DNAT trên reply (không cần thiết), dùng:
```
[host]# ovn-nbctl set Load_Balancer lb1 options:skip_snat=true
```

**Hairpin:** khi backend chính là một pod trong cùng Logical Switch và tự nó gửi request đến VIP của service mà nó là backend → packet loop về chính nó. Không xử lý đặc biệt, packet có:
- src_ip = pod_IP (= backend_IP)
- dst_ip = VIP → DNAT → dst_ip = backend_IP = pod_IP

Kết quả: pod gửi packet cho chính nó với source = destination. Kernel TCP stack drop vì same source/dest. OVN xử lý hairpin bằng cách **SNAT** source của packet hairpin thành một IP "hairpin SNAT IP" (cấu hình qua `options:hairpin_snat_ip`), làm cho packet xuất hiện như đến từ một host khác:
```
[host]# ovn-nbctl set Load_Balancer lb1 options:hairpin_snat_ip=169.254.169.254
```

## 13.9.4 Session affinity

Với traffic HTTP/HTTPS, một client có thể gửi nhiều request qua nhiều TCP connection. Mỗi connection có src_port khác nhau → hash có thể chọn backend khác nhau → state session (authentication state, shopping cart) bị mất.

**Session affinity OVN** đảm bảo cùng client (cùng src_ip) luôn tới cùng backend trong khoảng timeout nhất định. Hiện thực:
```
[host]# ovn-nbctl lb-add lb1 10.0.0.100:80 "192.168.1.1:80,192.168.1.2:80" \
         -- set Load_Balancer lb1 options:affinity_timeout=3600
```

Dưới mui xe, OVN dùng một table hash phụ (`lb_aff_check` và `lb_aff_learn`) ghi ánh xạ `(client_ip, vip) → backend_chosen` với TTL = affinity_timeout. Khi connection mới từ cùng client đến, OVN tra table này trước khi hash, nếu có entry → dùng backend đã chọn trước; nếu không → hash mới và ghi entry.

**Hạn chế:** session affinity dựa trên src_ip → nếu client đứng sau NAT (nhiều client cùng public IP) thì tất cả được map tới cùng backend (không phân phối). Với workload Kubernetes nội bộ, client là pod IP → affinity chính xác.

## 13.9.5 Health check và active/passive backend

OVN hỗ trợ health check cho backend thông qua table `Load_Balancer_Health_Check`:
```
[host]# ovn-nbctl --id=@hc create Load_Balancer_Health_Check \
         vip='"10.0.0.100:80"' \
         options:interval=5 options:timeout=3 options:success_count=1 options:failure_count=3 \
         -- add Load_Balancer lb1 health_check @hc
```

Mỗi chassis có ovn-controller sẽ gửi **TCP SYN** (hoặc ICMP echo, tùy cấu hình) tới từng backend IP:port trong LB mà chassis đó host. Nếu backend không reply (timeout 3s × 3 lần fail), backend được đánh dấu `offline`. OVN ghi vào table `Service_Monitor` trong SB DB. ovn-northd đọc `Service_Monitor`, lọc backend offline khỏi tập active, và sinh Logical Flow mới không bao gồm backend offline.

**Kiến trúc health check phân tán:** mỗi chassis tự probe subset backend (thường là backend thuộc cùng chassis nếu có, hoặc được gán theo hash để phân phối tải). Kết quả tổng hợp qua table `Service_Monitor`. Ưu điểm: mở rộng tốt (không có điểm probe đơn), nhược điểm: mỗi backend bị probe nhiều lần (từ nhiều chassis).

**Tích hợp với Kubernetes readiness probe:** trong OVN-Kubernetes, readinessProbe của pod được dịch sang `Load_Balancer_Health_Check` cho Service endpoint tương ứng. Nếu pod readinessProbe fail → backend trong OVN LB bị remove → traffic Service không định tuyến tới pod đó.

## 13.9.6 So sánh với kube-proxy (iptables + IPVS)

| Đặc điểm | OVN LB | kube-proxy iptables | kube-proxy IPVS |
|----------|--------|---------------------|-----------------|
| Thuật toán hash | Consistent 5-tuple | Random (iptables `-m statistic`) | Round-robin / LC / WLC / SH |
| Session affinity | Có, qua ClientIP | Có, qua ClientIP | Có, native IPVS |
| Health check | Có, TCP/ICMP | Không (phụ thuộc kubelet readiness) | Không native |
| Vị trí | OVS datapath (kernel) | iptables chain | IPVS kernel module |
| Mở rộng | O(1) tra cứu qua conntrack | O(n) duyệt chain | O(1) hash |
| Độ trễ cập nhật | Dưới giây (ovn-northd) | Giây (rebuild iptables) | Dưới giây (ipvsadm) |

**Khi chọn OVN LB:** cụm đã dùng OVN-K8s làm CNI → LB native, không cần kube-proxy. Ưu điểm: đồng nhất data plane, gỡ lỗi một công cụ (ovn-trace) cho cả định tuyến và LB.

**Khi giữ kube-proxy:** cluster dùng CNI khác (Calico, Flannel) không có LB native → phải dùng kube-proxy. Hoặc cluster cần tính năng IPVS cụ thể (LC, WLC algorithm) không có trong OVN.

### ▶ Guided Exercise 1: Tạo OVN Load Balancer và quan sát conntrack NAT

> **State verification lab:** Output dựa trên OVN 22.03 docs; chờ kiểm chứng lab thực.

**Mục tiêu:** Tạo một LB với 2 backend, gửi traffic qua VIP, quan sát DNAT trong conntrack và hash chọn backend.

**Chuẩn bị:** Lab Ubuntu 22.04 với OVN 22.03, một Logical Switch có 3 port (1 client + 2 backend).

**Bước thực hiện:**

**1.** Tạo topology:
```
[host]# ovn-nbctl ls-add ls1
[host]# ovn-nbctl lsp-add ls1 client
[host]# ovn-nbctl lsp-set-addresses client "aa:00:00:00:00:01 10.0.0.10"
[host]# ovn-nbctl lsp-add ls1 backend1
[host]# ovn-nbctl lsp-set-addresses backend1 "aa:00:00:00:00:02 10.0.0.11"
[host]# ovn-nbctl lsp-add ls1 backend2
[host]# ovn-nbctl lsp-set-addresses backend2 "aa:00:00:00:00:03 10.0.0.12"
```

**2.** Tạo Load Balancer:
```
[host]# ovn-nbctl lb-add lb1 10.0.0.100:80 "10.0.0.11:8080,10.0.0.12:8080" tcp
[host]# ovn-nbctl ls-lb-add ls1 lb1
```

**3.** Quan sát Logical Flow trong table `ls_in_lb`:
```
[host]# ovn-sbctl lflow-list ls1 | grep "ls_in_lb"
```

Kết quả mong đợi:
```
table=12 (ls_in_lb), priority=100,
  match=(ct.new && ip4.dst == 10.0.0.100 && tcp.dst == 80),
  action=(...ct_lb(backends=10.0.0.11:8080,10.0.0.12:8080)...)
```

**4.** Quan sát OpenFlow rule trên br-int:
```
[host]# ovs-ofctl dump-flows br-int | grep "tp_dst=80" | head -5
```

**5.** Gửi traffic và quan sát conntrack:
```
[host]# conntrack -L -d 10.0.0.100
tcp  ESTABLISHED src=10.0.0.10 dst=10.0.0.100 sport=54321 dport=80
  [REPLY] src=10.0.0.11 dst=10.0.0.10 sport=8080 dport=54321
  mark=0 zone=0 use=1
```

Conntrack entry cho thấy DNAT: packet gửi đến `10.0.0.100:80` được chuyển hướng sang `10.0.0.11:8080`, reply có source = `10.0.0.11:8080` được ánh xạ lại thành `10.0.0.100:80` (un-DNAT).

**Bài học học được:**

Thứ nhất, LB của OVN dựa hoàn toàn vào conntrack, không có state riêng cho LB. Điều này có nghĩa tất cả tính năng conntrack (timeout, zone, mark) áp dụng cho LB traffic.

Thứ hai, hash backend được thực hiện bởi action `ct_lb` trong OpenFlow, là action đặc biệt của OVS hỗ trợ cân bằng tải native trong kernel datapath. Không cần userspace can thiệp → hiệu năng cao.

Thứ ba, khi backend bị remove (ví dụ `ovn-nbctl lb-add lb1 10.0.0.100:80 "10.0.0.12:8080" tcp`), conntrack entries cũ vẫn giữ nguyên, connection hiện tại tiếp tục tới backend cũ (đã terminate) và sẽ timeout. Connection mới hash lại và chọn từ danh sách mới.

**Cleanup:**
```
[host]# ovn-nbctl lb-del lb1
[host]# ovn-nbctl ls-del ls1
```

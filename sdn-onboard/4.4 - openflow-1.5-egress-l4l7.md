## Bối cảnh: bản chỉnh sửa cuối cùng

Phần 4.3 đã phân tích OF 1.4 (14/10/2013) với tham vọng transaction semantic qua bundles và table management qua eviction, nhưng sự áp dụng thực tế chỉ partial do vendor không thấy ROI. OF 1.5 ra đời ngay sau đó, với thêm egress tables, TCP flags matching, và gói tin type awareness. Nhưng context đã thay đổi.

Khoảng 2014, ngành SDN bắt đầu phân nhánh. OpenFlow như protocol dần được coi là "một lựa chọn" thay vì "lựa chọn duy nhất". Hypervisor mạng ảo hóa (overlay) (NVP → NSX) đang chiếm chỗ của OpenFlow trong DC virtualization (xem Phần 5.1). SDN via APIs (NETCONF/YANG, xem Phần 5.0) đang chiếm chỗ trong WAN management. Và quan trọng nhất, Bosshart et al. đã phát hành P4 paper CCR 2014 với premise hoàn toàn khác: thay vì protocol cố định để reach data plane, cho phép programmer **define** data plane behavior bằng language. P4 là threat sinh tồn cho OpenFlow, không phải competitor.

Trong context đó, OF 1.5 được release ngày **19/12/2014** và errata 1.5.1 ngày **26/03/2015**. Sau 1.5.1, ONF chuyển focus sang ONOS/CORD project (xem Phần 3.2.6) và không bao giờ revise OpenFlow spec nữa. 1.5.1 là final revision của branch OpenFlow Switch Specification.

## 4.4.1 Dòng thời gian phát hành OF 1.5: 1.5.0 (19/12/2014) → 1.5.1 (26/03/2015), cuối cùng

### Thông tin phát hành

OpenFlow Switch Specification Version 1.5.0 phát hành ngày **19/12/2014**, wire protocol version **0x06**. Spec document 283 trang, tăng từ 206 trang của 1.4. Errata 1.5.1 phát hành **26/03/2015** (cùng ngày với 1.3.5 errata), cùng wire protocol 0x06.

Archive PDFs:

```
https://opennetworking.org/wp-content/uploads/2014/10/openflow-switch-v1.5.0.pdf
https://opennetworking.org/wp-content/uploads/2014/10/openflow-switch-v1.5.1.pdf
```

Press release 12/2014 của ONF mô tả:

> *"OpenFlow 1.5 addresses new use cases in service provider and enterprise networks, including gói tin-type-aware matching, egress table processing, and extensions for carrier-grade deployments."*

### Ý nghĩa "cuối cùng"

1.5.1 không phải "end of life" chính thức, ONF không announce loại bỏ. Nhưng thực tế sau 2015, Technical Steering Committee không approve spec revision mới cho OpenFlow Switch Specification. Các working group tập trung:

- **Optical Transport WG**: tiếp tục công việc trên OpenFlow extensions for optical nhưng không push vào main spec.
- **OF-Config**: maintain spec riêng (OF-Config 1.2 release 04/2014 là version cuối).
- **Northbound Interfaces**: chuyển focus sang Ý đồ cấu hình API, gnmi, OpenConfig alliance.

ON.Lab merge với ONF tháng 10/2018 làm thay đổi tổ chức, kế thừa ONOS và CORD làm flagship project thay vì maintain OpenFlow spec.

Khi user 2026 nhìn lại và hỏi "OpenFlow version nào mới nhất?", câu trả lời đúng là **1.5.1 (26/03/2015)**, tuyệt đối không có 1.6 chính thức published. Có rumor về 1.6 internal nhưng đó là working draft của member-only WG, chưa bao giờ published.

### So sánh dòng thời gian phát hành cả nhánh OpenFlow

| Version | Release date | Wire protocol | Significance |
|---------|--------------|---------------|--------------|
| 1.0 | 31/12/2009 | 0x01 | Baseline, 12-tuple match (Phần 3.1) |
| 1.1 | 28/02/2011 | 0x02 | Multi-table + groups (Phần 4.0) |
| 1.2 | 05/12/2011 | 0x03 | OXM TLV + controller roles (Phần 4.1) |
| 1.3.0 | 25/04/2012 | 0x04 | Meter + PBB + aux channels (Phần 4.2) |
| 1.3.5 | 26/03/2015 | 0x04 | Final errata của LTS branch |
| 1.4.0 | 14/10/2013 | 0x05 | Bundles + eviction (Phần 4.3) |

Sau 1.5.1, không có 1.5.2, không có 1.6. Five năm evolution của OpenFlow spec tóm gọn trong 5 major version, 1 LTS branch (1.3), và 2 dead-end branch (1.4, 1.5).

## 4.4.2 Egress tables: chuỗi xử lý (pipeline) sau khi ingress chọn port đầu ra

### Vấn đề: chuỗi xử lý (pipeline) chỉ có ingress không thể biểu đạt chính sách riêng cho từng port đầu ra

Trong OF 1.0-1.4, toàn bộ luồng processing xảy ra ở ingress chuỗi xử lý (pipeline). Gói tin vào switch, match table 0, instructions (GOTO_TABLE, WRITE_ACTIONS, ...), cuối cùng action set được execute, gói tin đi ra port đã chọn. Một khi output port được quyết định, không còn cơ hội modify gói tin.

Vấn đề với ingress-only: **per-output-port policy** khó express. Ví dụ DC topology có uplink port tới multiple upstream provider, mỗi provider có QoS marking requirement khác nhau. Provider A yêu cầu DSCP=EF cho voice, provider B yêu cầu DSCP=AF41. Trong ingress, controller phải:

1. Match tenant traffic → chọn output port based on destination.
2. Apply QoS action (SET_DSCP) dựa trên output port đã chọn.

Không khả thi trong ingress chuỗi xử lý (pipeline) vì instruction order: SET_DSCP nằm trong APPLY_ACTIONS hoặc WRITE_ACTIONS, cần biết output port TRƯỚC để quyết định DSCP value. Nhưng output port là kết quả của chuỗi xử lý (pipeline), không phải input.

Workaround trong 1.4: controller install nhiều luồng entries riêng cho từng provider, mỗi entry có output port + DSCP hard-coded. Luồng table explodes với cross-product tenant × provider.

### Giải pháp OF 1.5: egress chuỗi xử lý (pipeline) (§5.1)

Spec 1.5 cho phép switch có **egress tables** song song với ingress tables. Sau khi ingress chuỗi xử lý (pipeline) chọn output port, gói tin được forwarded vào egress chuỗi xử lý (pipeline) của port đó (first_egress_table_id trở lên). Egress chuỗi xử lý (pipeline) có thể apply thêm actions per-port, như SET_DSCP phù hợp provider.

Chuỗi xử lý (Pipeline) cấu trúc:

```
Packet in → Ingress Pipeline → Action Set → Output Port Selected
                                                  │
                                                  ▼
                                         Egress Pipeline (per-port)
                                                  │
                                                  ▼
                                         Final output
```

Spec 1.5 §5.1 define:

- Switch advertise first_egress_table_id trong OFPTR_FEATURES_REPLY.
- Tables với ID < first_egress_table_id là ingress, >= là egress.
- Mỗi output port có thể trigger egress chuỗi xử lý (pipeline) (nếu configured), chuỗi xử lý (pipeline) có match/instructions riêng.

### Trường hợp sử dụng: carrier Ethernet với QoS riêng cho từng provider

Controller cấu hình:

```
Ingress Table 0 (classify tenant):
  match: metadata=tenant_A → action: WRITE_METADATA=tenant_A, GOTO Table 1
  match: metadata=tenant_B → action: WRITE_METADATA=tenant_B, GOTO Table 1

Ingress Table 1 (route + select output port):
  match: tenant_A, dst_ip=/24 → action: output port 10 (uplink provider A)
  match: tenant_B, dst_ip=/24 → action: output port 11 (uplink provider B)

Egress Table 100 (port 10 QoS, provider A):
  match: voice_class → action: SET_DSCP=46 (EF)
  match: data_class → action: SET_DSCP=34 (AF41)

Egress Table 100 (port 11 QoS, provider B):
  match: voice_class → action: SET_DSCP=46 (EF)
  match: data_class → action: SET_DSCP=26 (AF31)
```

Giải quyết cross-product explosion: luồng count giảm từ O(tenants × providers) xuống O(tenants + providers).

### Hiểu sai thường gặp về egress tables

**Hiểu sai:** "Egress table chạy song song với ingress, nên gói tin có thể skip ingress và vào thẳng egress."

**Thực tế:** Egress tables chạy **sau** ingress tables, không song song. Gói tin luôn qua ingress chuỗi xử lý (pipeline) trước để chọn output port; egress chuỗi xử lý (pipeline) chỉ kick in sau khi port đã được chọn. Không có cách để gói tin bypass ingress và vào egress trực tiếp. Mental model đúng: egress tables là "post-processing" sau khi routing quyết định, không phải alternative chuỗi xử lý (pipeline).

## 4.4.3 TCP flag matching

### Structure

OF 1.5 thêm **OXM_OF_TCP_FLAGS** (§A.2.3.7), 12-bit match với 12-bit mask. TCP flags bao gồm:

| Bit | Name | Mô tả |
|-----|------|-------|
| 0 | FIN | Finish, close connection |
| 1 | SYN | Synchronize, start connection |
| 2 | RST | Reset, abort connection |
| 3 | PSH | Push, deliver data immediately |
| 4 | ACK | Acknowledgment |
| 5 | URG | Urgent pointer valid |
| 6 | ECE | ECN-Echo (RFC 3168) |
| 7 | CWR | Congestion Window Reduced |
| 8 | NS | Nonce Sum (RFC 3540, experimental) |
| 9-11 | Reserved | (future use) |

Note: OF 1.2 đã có OXM_OF_TCP_FLAGS (xem Phần 4.1.3), OF 1.5 formalize với thêm 2 reserved bits.

### Trường hợp sử dụng trong SDN firewall

**SYN flood detection:** match gói tin với chỉ SYN set (không ACK). Typical pattern:

```
match: eth_type=0x0800, ip_proto=6, tcp_flags=0x002/0x017
# Mask 0x017 = SYN + FIN + RST + ACK + URG bits được kiểm tra
# Value 0x002 = chỉ SYN set trong số 5 bit này
  → action: meter:rate_limit, forward
```

Gói tin TCP connection đầu tiên (SYN alone) được count per source, vượt threshold → drop.

**FIN scan detection:** match FIN với không ACK:

```
match: eth_type=0x0800, ip_proto=6, tcp_flags=0x001/0x011
# Mask 0x011 = FIN + ACK
# Value 0x001 = chỉ FIN (không ACK)
  → action: drop, log
```

FIN scan là technique port scanning bypass stateless firewall (gửi FIN đến port đã đóng, expect RST response; open port silent).

**Null scan detection:** match all flags=0:

```
match: eth_type=0x0800, ip_proto=6, tcp_flags=0x000/0x03F
# Mask bao trùm FIN/SYN/RST/PSH/ACK/URG
# Value 0 = none set
  → action: drop, log
```

### Hạn chế

TCP flags matching chỉ ở **L4 header bit**. Không match được:
- TCP options (MSS, window scale, SACK permitted, timestamp).
- TCP sequence/ack number cụ thể.
- Payload content.

Để deep inspect TCP payload, cần stateful firewall (conntrack) hoặc L7 proxy (HAProxy, nginx). OF 1.5 TCP flags là "first line of defense" nhưng không thay thế WAF/IDS.

## 4.4.4 Gói tin type field aware

### Động cơ: non-Ethernet frame

OF 1.0-1.4 assume gói tin luôn start với **Ethernet header**. Match logic dựa vào `dl_type` field (Ethernet EtherType). Nhưng trong một số deployment, gói tin không phải Ethernet:

- **L3-only tunnel**: tunnel header có IP hoặc MPLS label first, không có Ethernet header bên ngoài.
- **ARP as L2 payload**: ARP gói tin là native L2, không encapsulate Ethernet (nhưng delivered trong Ethernet frame).
- **Inner gói tin trong VXLAN/Geneve**: khi switch decap tunnel, inner gói tin có thể là MPLS hoặc ARP, cần match type cụ thể.

OF 1.0-1.4 không biểu diễn được case này. Match logic implicit assume Ethernet, gói tin không Ethernet rơi vào fallback table miss.

### OF 1.5 solution: OXM_OF_PACKET_TYPE

Spec 1.5 §7.2.3.6 define **OXM_OF_PACKET_TYPE** (32-bit OXM field): explicit encode namespace + type của gói tin header đang được match:

| PACKET_TYPE | Namespace + Type | Mô tả |
|-------------|------------------|-------|
| 0x0000_6558 | Ethernet (0x0000) + TRANS_ETH_BRIDGE | Bridged Ethernet frame (default 1.0-1.4) |
| 0x0000_0800 | Ethernet + IPv4 | Ethernet with IPv4 payload |
| 0x0000_0806 | Ethernet + ARP | Ethernet with ARP payload |
| 0x0000_8847 | Ethernet + MPLS unicast | Ethernet with MPLS payload |
| 0x0001_nnnn | Non-Ethernet namespace | L3-only gói tin types |

Namespace `0x0000` là default Ethernet. Namespace `0x0001` là "gói tin không Ethernet" với type là next header.

Controller có thể match:

```
match: packet_type=0x0001_0800, ip_dst=10.0.0.0/24 → action: output port N
# Match L3-only IPv4 packet (không có Ethernet header ngoài)
```

### Use case: VXLAN decapsulation với inner MPLS

Carrier deployment: VXLAN tunnel carry inner MPLS frame. Switch decap VXLAN, inner gói tin là MPLS. Luồng:

```
# Decap outer VXLAN (ingress table)
match: udp_dst=4789 → action: POP_VXLAN, GOTO_TABLE 10

# Match inner MPLS (table 10)
match: packet_type=0x0000_8847, mpls_label=100 → action: forward via MPLS label 200
```

Không có OF 1.5 packet_type, match inner MPLS sau decap là implicit, behavior ambiguous. Với packet_type, explicit.

### Thực tế áp dụng

OF 1.5 packet_type được thiết kế cho advanced use case (VXLAN inner match, L3-only chuỗi xử lý (pipeline)), nhưng trong thực tế 2014-2020:

- Carrier Ethernet vẫn dùng PBB (1.3) cho 802.1ah, không cần packet_type.
- DC mạng ảo hóa (overlay) dùng OVS + OVN với conntrack zones và custom tunnel keys, không dựa OpenFlow packet_type.
- Hardware vendor không commit ASIC cho packet_type matching (complex parser requirement).

Kết quả: packet_type trên paper very powerful nhưng deployment gần như không có.

## 4.4.5 Current state 2026: vendor sự áp dụng near-zero, OVS 1.5 partial

### OVS 2.10+ implement partial

Open vSwitch 2.10 (06/2018) thêm 1.5 support partial:

**Implemented**:
- Wire protocol 0x06 handshake.
- Basic OF 1.5 messages (FEATURES_REQUEST/REPLY tương thích).
- TCP flags matching (đã có trong 1.2 via OVS extension, formalize trong 1.5).

**Stubbed** (advertised nhưng không full production-ready):
- Egress tables: OVS code có egress_table_id stub, nhưng actual egress processing chỉ minimal. Path chưa test đầy đủ.
- Gói tin type aware: OXM field parseable nhưng limited action execution support.

**Không implement**:
- Full 1.5 bundle extensions (mixed với 1.4 bundle support).
- Some minor OXM extensions specific cho 1.5.

OVS maintainer (Ben Pfaff, Justin Pettit) có confirm trong commit messages 2018 rằng 1.5 support là "best-effort completeness", không production-tuned.

### Hardware vendor skip entirely

- **HP/Aruba**: commit đến 1.3, 1.4/1.5 không có trong roadmap 2015-2020.
- **Pica8**: PicOS cuối cùng commit đến 1.3, khi Pica8 acquired bởi HCL Technologies 2018, OpenFlow evolution đóng băng.
- **NEC IP8800**: production end 2016, không bao giờ implement 1.5.
- **Centec V350/V358**: commit 1.3 standard.
- **Mellanox/NVIDIA Spectrum**: focus OVS switchdev hardware offload với tc flower, không direct OpenFlow 1.5 ASIC support.
- **Broadcom (market leader chipset)**: Trident/Tomahawk series silicon không commit 1.4/1.5, không có microcode public release.

### Tofino/P4 chiếm programmable space

Từ 2016, **Barefoot Networks Tofino ASIC** với **P4 programming language** (CCR 2014, Bosshart et al.) bắt đầu ship trong production. Thay vì OpenFlow fix protocol, P4 cho phép programmer viết parser + match-action tables + deparser theo need.

Major sự áp dụng: Alibaba, Tencent, Facebook dùng Tofino + P4 cho custom SDN chuỗi xử lý (pipeline) từ 2017-2019. Intel acquire Barefoot tháng 6/2019. Tháng 1/2023, Intel announce end-of-life cho dòng Tofino, phần lớn engineering team transition sang AI/ML silicon. P4 language vẫn alive qua p4.org open-source community, các vendor alternative (AMD/Pensando, NVIDIA Spectrum-4 với DOCA chuỗi xử lý (pipeline)).

Bài học: OpenFlow 1.5 không thua vì "không đủ feature", mà vì ngành đã move on sang mô hình mới (P4) trước khi 1.5 có thời gian implement. Timing là mọi thứ trong protocol evolution.

Chi tiết P4 + Tofino sẽ được phân tích trong Phần 6.0. Phần 4.6 (next) tổng hợp báo cáo hậu sự lesson learned của OpenFlow branch.

> **Key Topic:** OpenFlow 1.5 là "spec for spec's sake", ONF phát hành để maintain protocol evolution momentum trong context hệ sinh thái đã lạnh nhạt. Bài học cho bất kỳ protocol nào: feature richness không bảo đảm sự áp dụng. Hardware vendor commit ASIC chỉ khi có demand market chứng minh, không vì spec có tính năng mới. Và khi mô hình alternative (P4) xuất hiện với value proposition đủ mạnh (programmability), protocol ổn định (1.3) survive còn protocol evolution stalled (1.4, 1.5) chết.

## Mạch nối với phần sau

Phần 4.4 đã phân tích OF 1.5 với egress tables giải cross-product policy problem, TCP flags matching formalize, gói tin type awareness cho non-Ethernet frame, và reality sự áp dụng gần như zero trên hardware silicon. Phần 4.5 tiếp theo phân tích TTP (Table Type Patterns, ONF TS-017) như ONF attempt cuối cùng để bridge giữa spec generic và silicon capability fragmented, trước khi ngành chuyển sang Luồng Objectives (ONOS, xem Phần 6.1) và P4 (xem Phần 6.0). Phần 4.6 tổng kết 5 limitations của OpenFlow và case study Google B4 SIGCOMM 2013 minh chứng tại sao pure OpenFlow không scale cho hyperscaler.

## ▶ Guided Exercise — Egress table simulation với OVS + OF 1.5 features (audit P4.B4.1 hands-on)

**Mục đích:** Explore egress chuỗi xử lý (pipeline) concept với OVS (OVS partial support OF 1.5). Verify per-port egress policy.

**Bối cảnh:** OVS 2.17 hỗ trợ OF 1.5 partial (egress table qua experimenter extension). Exercise dùng OF 1.5 wire với OVS emulate egress behavior qua multi-table pattern.

**Bước:**

1. Create bridge OF 1.5:
   ```bash
   $ sudo ovs-vsctl add-br br-egress -- set bridge br-egress protocols=OpenFlow15
   ```
2. Simulate egress table behavior qua multi-table pattern (ingress → decide output → egress-like table):
   ```bash
   # Ingress: classify traffic + decide output port
   $ sudo ovs-ofctl -O OpenFlow15 add-flow br-egress \
       "table=0,priority=100,ip,nw_dst=10.1.0.0/16,actions=set_field:1->reg1,goto_table:1"
   $ sudo ovs-ofctl -O OpenFlow15 add-flow br-egress \
       "table=0,priority=100,ip,nw_dst=10.2.0.0/16,actions=set_field:2->reg1,goto_table:1"

   # "Egress" table: per-output-port QoS (DSCP remark)
   $ sudo ovs-ofctl -O OpenFlow15 add-flow br-egress \
       "table=1,priority=100,reg1=1,actions=set_field:46->ip_dscp,output:1"
   $ sudo ovs-ofctl -O OpenFlow15 add-flow br-egress \
       "table=1,priority=100,reg1=2,actions=set_field:34->ip_dscp,output:2"
   ```
3. Verify per-output-port DSCP marking:
   ```bash
   $ sudo ovs-appctl ofproto/trace br-egress \
       "in_port=1,ip,nw_dst=10.1.0.5" | grep DSCP
   # Expected: set DSCP=46 (EF), output port 1
   ```
4. Compare với native egress table (OF 1.5+ real implementation):
   ```bash
   # Native egress table (nếu switch hỗ trợ OF 1.5 native) — spec 1.5 §5.1
   # OVS hiện emulate qua multi-table ingress pattern (như trên)
   ```

**POE:**
- **Predict:** OVS có hỗ trợ OF 1.5 egress table native không?

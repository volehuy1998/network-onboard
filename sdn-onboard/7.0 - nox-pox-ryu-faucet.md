## Bối cảnh: phần mềm controller và khoảng trống giữa spec và production

Khi OpenFlow 1.0 được công bố ngày 31/12/2009, đặc tả định nghĩa giao thức giữa switch và controller, nhưng không có controller nào sẵn sàng để cộng đồng dùng ngay. Đây là vấn đề thực tế: một kỹ sư muốn thử nghiệm OpenFlow cần phải tự viết controller, điều đòi hỏi hiểu sâu về giao thức TCP, binary encoding của OpenFlow messages, và quản lý nhiều kết nối switch đồng thời.

NOX, được Nicira open-source năm 2008 (trước khi OpenFlow 1.0 chính thức ra đời), giải quyết khoảng trống này bằng cách cung cấp một framework hoàn chỉnh: developer không cần tự xử lý TCP socket hay parse binary, mà chỉ cần viết handler cho các sự kiện cấp cao. Hướng thiết kế này — **controller như một hệ điều hành mạng, ứng dụng chạy trên controller như process chạy trên OS**, trở thành triết lý chủ đạo cho toàn bộ dòng controller sau đó, được đặt tên rõ trong tiêu đề bài báo NOX: "Towards an Operating System for Networks."

> **Key Topic:** Mô hình controller-as-OS giả định một phân tách rõ ràng: controller cung cấp abstraction về topology và packet forwarding, application viết policy trên abstraction đó. NOX là hiện thực hóa đầu tiên của mô hình này, và mọi framework sau (POX, Ryu, Faucet, ODL, ONOS) đều kế thừa cùng triết lý phân tách này, dù implement theo cách khác nhau.

## 7.0.1 NOX: Ethane's runtime, open-source 2008

NOX ra đời như runtime engine cho Ethane, hệ thống kiểm soát truy cập mạng enterprise mà Martin Casado và nhóm Stanford thiết kế (Part 2.4). Khi Casado và Nicira quyết định open-source Ethane's controller component năm 2008, họ đặt tên là NOX và viết lại sạch hơn để cộng đồng có thể sử dụng độc lập với Ethane policy model.

Bài báo gốc **"NOX: Towards an Operating System for Networks"** được trình bày tại HotNets 2008 bởi Natasha Gude, Teemu Koponen, Justin Pettit, Ben Pfaff, Martin Casado, Nick McKeown, và Scott Shenker, sau đó đăng trên *ACM SIGCOMM Computer Communication Review*, Vol. 38, No. 3, tháng 07/2008. Bài báo đặt ra khái niệm network OS: giống như UNIX cung cấp abstraction cho hardware, NOX cung cấp abstraction "network view", toàn bộ topology, tất cả host, tất cả flow, cho ứng dụng mạng.

**Kiến trúc kỹ thuật của NOX** là C++ event-driven, single-threaded event loop với `select()`/`poll()` cho I/O multiplexing. Application viết bằng C++ (hoặc Python thông qua binding) đăng ký handler:

```cpp
// Ví dụ handler C++ trong NOX
class MyApp : public Component {
public:
    void configure(const Configuration*) {
        register_handler<Packet_in_event>(
            boost::bind(&MyApp::handle_packet_in, this, _1));
    }
    Disposition handle_packet_in(const Event& e) {
        const Packet_in_event& pi = assert_cast<const Packet_in_event&>(e);
        // pi.datapath_id: switch ID
        // pi.buffer_id: buffer trên switch
        // pi.packet: raw packet bytes
        install_flow(...);
        return CONTINUE;
    }
};
```

NOX hỗ trợ OpenFlow 0.8.9 (draft khi ra mắt) và sau đó OpenFlow 1.0. Hạn chế thiết kế: single-threaded event loop có nghĩa handler phải hoàn thành nhanh, bất kỳ blocking operation nào (query database, DNS lookup) đều chặn toàn bộ controller. Đây là đánh đổi có chủ đích: đơn giản hóa lập trình bằng cách loại bỏ concurrent access vào shared state.

> **Lưu ý kỹ thuật:** NOX có hai phiên bản song song: NOX-Classic (C++ thuần, version gốc 2008-2011) và NOX-MT (multi-threaded, phát triển bởi Big Switch Networks 2012). NOX-MT thêm threading để tăng throughput nhưng mất đi tính đơn giản của single-threaded model. Khi cộng đồng viết tài liệu "NOX" mà không rõ phiên bản, thường ám chỉ NOX-Classic.

Sau khi VMware mua Nicira năm 2012, NOX dần không còn được chủ động phát triển. Repository github.com/noxrepo/nox vẫn tồn tại ở trạng thái archive.

## 7.0.2 POX: Python port cho việc dạy học, 2011

POX do Murphy McCauley (UC Berkeley) phát triển năm 2011 như một Python rewrite của NOX-Classic. Mục tiêu rõ ràng ngay từ đầu: **không phải production, mà là dạy học**. McCauley thiết kế POX để sinh viên CS có thể viết ứng dụng OpenFlow trong vài giờ thay vì vài ngày, tận dụng tính dễ đọc của Python và hệ sinh thái thư viện phong phú.

POX tích hợp tốt với Mininet, platform mô phỏng mạng cũng đến từ nhóm UC Berkeley/Stanford, tạo thành cặp đôi phổ biến nhất trong giảng dạy SDN. Lệnh khởi động Mininet với POX controller:

```bash
# Terminal 1: khởi động POX với module học tập cơ bản
[controller]$ python3 pox.py log.level --DEBUG forwarding.l2_learning

# Terminal 2: khởi động Mininet kết nối đến POX
[controller]$ sudo mn --controller=remote,ip=127.0.0.1,port=6633 --topo=tree,depth=2
```

Module `forwarding.l2_learning` trong POX là ví dụ kinh điển: controller học MAC address của mỗi host từ packet\_in, cài đặt flow entry chính xác cho các packet tiếp theo, và xử lý flooding khi chưa biết đích. Toàn bộ logic khoảng 80 dòng Python, đủ ngắn để đọc trong một buổi lab.

**Nhược điểm thiết kế của POX** là hệ quả trực tiếp của Python: GIL (Global Interpreter Lock) ngăn Python chạy thực sự song song trên multi-core. POX single-threaded chậm hơn NOX C++ khoảng 10× trong benchmark throughput, và không thể scale để quản lý hàng nghìn switch trong môi trường production. McCauley không che giấu điều này, tài liệu chính thức của POX ghi rõ "POX is not intended for production use."

Tuy nhiên, đây không phải thất bại của POX, đây là thành công của mục tiêu đặt ra. Hàng chục trường đại học trên thế giới (Stanford, Berkeley, Princeton, MIT, NUS, và nhiều trường Việt Nam) dùng POX + Mininet làm nền tảng dạy SDN, và nhiều kỹ sư đang làm việc với OVN hay ONOS hôm nay đã học khái niệm control plane đầu tiên qua một POX exercise.

## 7.0.3 Ryu: NTT framework 2012, hỗ trợ OpenFlow đầy đủ

Nhóm nghiên cứu tại NTT (Nippon Telegraph and Telephone Communications) phát triển Ryu từ 2012 với mục tiêu lấp khoảng trống giữa POX (dạy học, không production) và ODL/ONOS (enterprise-scale, phức tạp để triển khai). Tên "Ryu" trong tiếng Nhật có nghĩa là "rồng" hoặc "dòng chảy", cả hai nghĩa đều phù hợp với vai trò framework xử lý luồng OpenFlow.

**Điểm mạnh kỹ thuật của Ryu** so với POX là hỗ trợ OpenFlow đầy đủ từ 1.0 đến 1.5.1, bao gồm cả các extension của OVS (Nicira extensions — NXM fields). Điều này đặc biệt quan trọng khi OVS trở thành virtual switch phổ biến nhất: Ryu có thể lập trình OVS với các match fields mở rộng mà OF 1.0 không hỗ trợ (ví dụ: tunnel\_id, reg0-reg7 cho metadata, conntrack state). Kiến trúc module của Ryu cho phép developer load chỉ những component cần thiết:

```bash
# Khởi động Ryu với ứng dụng simple switch
[controller]$ ryu-manager ryu.app.simple_switch_13

# Hoặc load ứng dụng tự viết
[controller]$ ryu-manager myapp.py --observe-links
```

Một tính năng ít được biết đến nhưng có giá trị thực tế cao là **BGP speaker tích hợp trong Ryu**. Module `ryu.services.protocols.bgp` cho phép controller vừa nói BGP với router bên ngoài, vừa chuyển thông tin routing nhận được thành flow entry trong OVS. Trường hợp sử dụng điển hình: controller nhận BGP prefix announcement, cài đặt forwarding rule tương ứng trong OVS dựa trên next-hop từ BGP table, một dạng route reflector cho data center không dùng hardware router.

> **Key Topic:** Ryu được thiết kế như một framework linh hoạt, không phải như một production controller đóng gói sẵn. Người dùng Ryu phải tự viết application logic — Ryu cung cấp API và abstraction, không cung cấp chính sách. Đây là sự khác biệt then chốt với Faucet, nơi chính sách đã được định nghĩa sẵn trong YAML và người vận hành chỉ cần cấu hình, không cần lập trình.

Cộng đồng Ryu hoạt động mạnh 2012-2018, sau đó giảm dần khi ONOS chiếm lĩnh thị trường service provider và Faucet chiếm lĩnh phân khúc campus. Tuy nhiên, Ryu vẫn được sử dụng trong nghiên cứu vì tính minh bạch: codebase Python dễ đọc, không có dependency phức tạp, và hỗ trợ toàn bộ OpenFlow message types để nghiên cứu viên có thể kiểm soát từng bit trong flow entry.

## 7.0.4 Faucet: production focus, REANNZ 2015

Faucet bắt đầu như một ứng dụng Ryu viết bởi Josh Bailey và nhóm REANNZ (Research and Education Advanced Network New Zealand) cùng nhóm WAND (Waikato Applied Network Dynamics) tại University of Waikato. REANNZ là mạng học thuật quốc gia của New Zealand, vận hành kết nối tốc độ cao giữa các trường đại học và viện nghiên cứu. Khi REANNZ quyết định chuyển một phần hạ tầng sang SDN, họ nhận ra Ryu cung cấp linh hoạt nhưng không cung cấp khả năng vận hành, không có monitoring tích hợp, không có cấu hình declarative, không có rollback khi controller crash.

**Faucet giải quyết vấn đề vận hành** bằng ba thành phần chính. Thứ nhất, cấu hình YAML declarative thay vì code Python: người vận hành mô tả topology (switch, port, VLAN) và chính sách (ACL, routing) trong file YAML, Faucet dịch sang flow entries. Khi cần thay đổi chính sách, người vận hành sửa YAML và reload, không cần deploy lại Python code. Thứ hai, Gauge là sidecar process độc lập với controller, liên tục poll statistics từ switch (port counters, flow counters) và xuất sang Prometheus hoặc InfluxDB, người vận hành có dashboard giám sát lưu lượng thời gian thực. Thứ ba, Faucet thiết kế multi-table pipeline canonical cho switch: thay vì để từng ứng dụng tự quyết định dùng bao nhiêu table và theo thứ tự nào, Faucet định nghĩa một pipeline chuẩn (VLAN decoding → learning → routing → ACL → output) được tối ưu cho chipset phổ biến.

```yaml
# Ví dụ cấu hình Faucet tối giản — faucet.yaml
vlans:
  office:
    vid: 100
    description: "Office VLAN"

dps:
  sw1:
    dp_id: 0x1
    interfaces:
      1:
        name: "server1"
        native_vlan: office
      2:
        name: "server2"
        native_vlan: office
      3:
        name: "uplink"
        tagged_vlans: [office]
```

Mức độ đơn giản của YAML trên — 17 dòng để cấu hình một switch 3 port, là lý do chính Faucet được các đội vận hành chấp nhận. Người vận hành không cần hiểu OpenFlow flow entries hay Python programming để triển khai Faucet trong môi trường campus.

> **Key Topic:** Faucet và Ryu thể hiện hai triết lý vận hành trái chiều từ cùng một nền tảng kỹ thuật. Ryu tối ưu cho việc trả lời câu hỏi "tôi có thể làm gì với OpenFlow?" — linh hoạt tối đa, không ràng buộc. Faucet tối ưu cho câu hỏi "làm thế nào tôi vận hành mạng này ổn định qua 3 năm?" — chức năng cố định nhưng vận hành có thể dự đoán được. Cả hai đều đúng trong ngữ cảnh của chúng.

REANNZ triển khai Faucet ở môi trường production từ 2016. Một số ISP và university network ở Mỹ và châu Âu (Internet2 thành viên) cũng sử dụng Faucet trong campus switching. Faucet vẫn được phát triển tích cực với release monthly tính đến 2026, khác với Ryu đã chuyển sang chế độ maintenance từ khoảng 2019.

## 7.0.5 So sánh bốn controller và lựa chọn phù hợp

Bốn controller này không cạnh tranh trực tiếp vì chúng phục vụ bốn nhu cầu khác nhau:

*Table 7-1: So sánh NOX, POX, Ryu, Faucet theo mục đích và đặc điểm kỹ thuật*

| Tiêu chí | NOX | POX | Ryu | Faucet |
|---|---|---|---|---|
| Ngôn ngữ | C++ (Python binding) | Python | Python | Python (dựa Ryu) |
| Năm ra mắt | 2008 | 2011 | 2012 | 2015 |
| Mục đích chính | Research prototype | Dạy học | Research linh hoạt | Production campus |
| OpenFlow hỗ trợ | 1.0 | 1.0, 1.3 | 1.0 đến 1.5 | 1.3 (production tested) |
| Cấu hình | Code C++/Python | Code Python | Code Python | YAML declarative |
| Monitoring | Không tích hợp | Không tích hợp | Không tích hợp | Gauge (Prometheus) |
| Trạng thái 2026 | Archive | Không phát triển | Maintenance | Phát triển tích cực |

Khi nào dùng gì: nếu mục tiêu là học hiểu cơ chế OpenFlow, POX + Mininet là điểm xuất phát tốt nhất vì codebase ngắn và ví dụ phong phú. Nếu mục tiêu là nghiên cứu với OVS trong lab, Ryu phù hợp vì hỗ trợ NXM extension và toàn bộ OpenFlow 1.5. Nếu mục tiêu là triển khai SDN trong campus thực tế với đội vận hành không có background lập trình, Faucet là lựa chọn duy nhất trong nhóm này.

### ▶ Guided Exercise 1: Chạy Ryu simple switch và quan sát packet_in

Trong bài thực hành này, bạn khởi động Ryu với ứng dụng simple switch, kết nối Mininet, và quan sát cách controller học MAC address qua packet\_in.

**Outcomes:**
- Quan sát được log packet\_in khi host gửi frame đầu tiên
- Xác minh flow entry được cài đặt sau khi controller học MAC
- Hiểu vòng đời: flooding → học MAC → unicast forwarding

**Before You Begin:**
Cài đặt Ryu (`pip install ryu`) và Mininet (`apt install mininet`). Xác minh `ryu-manager --version` và `mn --version` không báo lỗi.

**Instructions:**

**1.** Mở terminal đầu tiên, khởi động Ryu với module simple switch và bật debug log:

```bash
[controller]$ ryu-manager --verbose ryu.app.simple_switch_13
```

Ryu lắng nghe trên port 6653 (OpenFlow 1.3 default). Log sẽ hiện `BRICK SimpleSwitch13 PROVIDES ... EventOFPPacketIn`.

**2.** Mở terminal thứ hai, khởi động Mininet với topology hai host:

```bash
[controller]$ sudo mn --controller=remote,ip=127.0.0.1,port=6653 \
               --switch=ovsk,protocols=OpenFlow13 --topo=single,2
```

Mininet tạo switch `s1` với hai host `h1` (10.0.0.1) và `h2` (10.0.0.2).

**3.** Trong Mininet prompt, chạy ping từ h1 đến h2:

```
mininet> h1 ping h2 -c 3
```

**4.** Quan sát log trên terminal Ryu. Các dòng đầu tiên sẽ có dạng:

```
packet in 1 xx:xx:xx:xx:xx:x1 ff:ff:ff:ff:ff:ff 1
packet in 1 xx:xx:xx:xx:xx:x2 xx:xx:xx:xx:xx:x1 2
```

Dòng đầu: switch ID `1`, src MAC của h1, dst MAC là broadcast (`ff:ff:ff:ff:ff:ff`), port vào là `1`. Đây là ARP request, h1 chưa biết MAC của h2 nên gửi broadcast. Controller nhận packet\_in, học MAC h1 ở port 1, flood ra tất cả port.

Dòng thứ hai: src MAC của h2, dst MAC của h1 (h2 trả lời ARP), port vào là `2`. Controller học MAC h2 ở port 2 và cài đặt flow entry unicast từ h1→h2 và h2→h1.

**5.** Kiểm tra flow table trong switch:

```bash
[controller]$ sudo ovs-ofctl -O OpenFlow13 dump-flows s1
```

Sau ping đầu tiên, bạn sẽ thấy hai flow entry có `dl_dst` cụ thể thay vì action flood, bằng chứng controller đã học MAC và cài unicast forwarding.

**Finish:**

```
mininet> quit
```

Nhấn Ctrl+C để dừng Ryu.

## Điểm cốt lõi cần nhớ

Thứ nhất, NOX thiết lập mô hình controller-as-OS năm 2008, trước khi OpenFlow 1.0 chính thức ra đời. Toàn bộ framework controller sau đó kế thừa triết lý: controller cung cấp abstraction về mạng, application viết policy trên abstraction đó.

Thứ hai, POX và Ryu giải quyết cùng một vấn đề kỹ thuật theo cách khác nhau vì phục vụ người dùng khác nhau. POX ưu tiên dễ học, sinh viên có thể viết L2 learning switch trong một buổi. Ryu ưu tiên đầy đủ tính năng, nghiên cứu viên có thể lập trình mọi khía cạnh của OpenFlow 1.5 và NXM extension.

Thứ ba, Faucet là bằng chứng cho một nguyên tắc vận hành: linh hoạt không đồng nghĩa với dễ vận hành. Bằng cách thu hẹp phạm vi xuống campus switching và định nghĩa sẵn pipeline chuẩn, Faucet đạt được khả năng vận hành ổn định mà Ryu không thể cung cấp, dù cả hai đều viết bằng Python.

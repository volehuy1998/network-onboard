## 9.21.1 What Mininet is and why it matters

In 2010, the Stanford Clean Slate group led by Nick McKeown introduced Mininet in the paper *"A Network in a Laptop: Rapid Prototyping for Software-Defined Networks"* at HotNets-IX. The goal was to allow any SDN researcher to reproduce an experimental topology with nothing more than a Python script and a laptop. Before Mininet, reproducible SDN research required dedicated hardware that cost tens of thousands of US dollars. After Mininet, every SDN paper between 2010 and 2018 shipped with a Mininet script. This was the decisive shift from "SDN is a theory" to "SDN is tooling that lives on a laptop". The Crichigno Lab 2 document describes it briefly as follows: *"Mininet is a virtual testbed enabling the development and testing of network tools and protocols. With a single command, Mininet can create a realistic virtual network on any type of machine (Virtual Machine (VM), cloud-hosted, or native)."*

Four properties make Mininet more useful than every network simulator that came before it.

**Emulation, not simulation.** Mininet runs real Linux binaries (Python, curl, ping, the OVS kernel module) on a real kernel. A simulator such as ns-3 has to port each protocol into the simulator framework. The practical difference is that every bug in real OVS code shows up in Mininet, with no need to port the code again.

**Network namespaces as hosts.** Each "host" in Mininet is in fact a process (or a group of processes) running in its own Linux kernel network namespace. The namespace isolates the interface list, the routing table, and the ARP table. The cost to create one is about 50 KB of RAM per host, so Mininet can build a network of 1,000 hosts or more on a single laptop.

**Virtual ethernet (veth) as the cable.** The connection between a host and a switch is a veth pair. The two ends of the veth live in different namespaces; a packet written into one end appears at the other end. In theory this is no different from a physical cable. In practice the CPU cost is low because there is no hashing or serialization step.

**Open source with a large community.** Mininet lives in the GitHub ecosystem with documentation and thousands of example topologies on pastebin and GitHub. A popular fork called Containernet (which adds Docker containers as hosts) keeps the API compatible, so the Crichigno Lab 2 environment that runs Containernet still drops into the prompt `containernet>` when you invoke `sudo mn`.

### Role in the Block IX lab series

Crichigno Lab 2 is the first step of a 15-lab OVS series. From Lab 3 (Introduction to OVS) to Lab 15 (IPsec tunnel), every lab assumes the learner already knows how to use Mininet. Part 9.21 supplies that foundation. Once you finish this part, you can read every `.mn` script in the lab series, reconstruct the topology of Labs 3 to 15 with a single command, and modify the topology to test your own hypothesis.

## 9.21.2 Mininet CLI, basic commands

Lab 2 walks through six basic steps. The sequence below summarizes the whole flow.

### Bring up the default topology

Open a Linux terminal and run:

```bash
sudo mn
```

The default password on the lab machine is `password`. After a few seconds the prompt becomes `containernet>` (or `mininet>` on the upstream build). The default topology has one switch `s1`, two hosts `h1` and `h2`, and one controller `c0`. Mininet automatically assigns 10.0.0.1/8 to h1 and 10.0.0.2/8 to h2.

### Inspect the topology

Four commands are useful right after the topology comes up:

```
containernet> help        # Print every CLI command with examples
containernet> nodes       # List the nodes (c0 s1 h1 h2)
containernet> net         # Show the interface-to-interface connection table
containernet> dump        # Print the full process PID and IP details for each node
```

The output of `net` for the default topology is:

```
h1 h1-eth0:s1-eth1
h2 h2-eth0:s1-eth2
s1 lo:  s1-eth1:h1-eth0 s1-eth2:h2-eth0
c0
```

Read it like this: `h1-eth0` on host h1 connects to `s1-eth1` on switch s1. The controller `c0` has no interface because it runs out of band.

### Run a command on a specific host

Mininet lets you run any Linux command on a chosen host using the syntax `<host> <command>`:

```
containernet> h1 ifconfig
containernet> h1 ping -c 4 10.0.0.2
containernet> h2 traceroute 10.0.0.1
```

`h1 ifconfig` prints the interface table of host h1, not of the machine that hosts Mininet. This is the network-namespace isolation property at work.

There are three shortcut commands worth remembering:

```
containernet> pingall      # Ping from every host to every other host
containernet> iperf         # Run an iperf test between h1 and h2 by default
containernet> xterm h1 h2   # Open a separate xterm for each host (X11 required)
```

### Exit and clean up

```
containernet> exit
```

If you closed the terminal in the middle of a session and Mininet did not exit cleanly, leftover namespaces will cause errors the next time you start. The cleanup command runs outside the Mininet CLI:

```bash
sudo mn -c
```

This kills every Mininet process, removes every leftover namespace, and removes every OVS bridge that Mininet created. Run `mn -c` before starting a new session whenever you have any doubt.

## 9.21.3 Custom Python topology, the `Topo` class API

When the topology grows beyond the default (one switch, two hosts), use a Python script. Mininet provides a `Topo` class in the `mininet.topo` module. The basic pattern is:

```python
# mytopo.py
from mininet.topo import Topo

class MyTopo(Topo):
    "Custom topology: 3 hosts, 2 switches, ring."
    def build(self):
        # Add hosts (default IP 10.0.0.N is auto-assigned)
        h1 = self.addHost('h1')
        h2 = self.addHost('h2')
        h3 = self.addHost('h3')

        # Add switches
        s1 = self.addSwitch('s1')
        s2 = self.addSwitch('s2')

        # Connect hosts to switches
        self.addLink(h1, s1)
        self.addLink(h2, s1)
        self.addLink(h3, s2)

        # Connect switch to switch
        self.addLink(s1, s2)

topos = {'mytopo': (lambda: MyTopo())}
```

Save the file as `mytopo.py` and run:

```bash
sudo mn --custom mytopo.py --topo mytopo
```

Mininet parses the file, executes `MyTopo().build()`, and brings up the topology exactly as the script defines it. The `containernet>` prompt appears just like a normal `sudo mn` invocation, but `nodes` will return `c0 s1 s2 h1 h2 h3`.

### Assign IP and custom configuration

By default Mininet hands out IP addresses from 10.0.0.0/8 in the order in which hosts are added. To assign IP addresses manually, pass options to `addHost`:

```python
h1 = self.addHost('h1', ip='192.168.1.10/24', mac='00:00:00:00:00:01')
h2 = self.addHost('h2', ip='192.168.2.10/24', mac='00:00:00:00:00:02')
```

Assigning different subnets creates a need for routing. When two hosts are on different subnets, Mininet does not configure a router for you. You must add a switch or host that plays the router role yourself (see §9.21.5).

### Choose a different controller

Mininet uses an internal `OVSController` (a simple built-in learning switch) by default. To connect to an external controller, use the `--controller` option:

```bash
sudo mn --custom mytopo.py --topo mytopo \
    --controller=remote,ip=127.0.0.1,port=6653
```

Mininet then points the OVS switch at the controller running on `127.0.0.1:6653`. Use this when you run Ryu, ONOS, or OpenDaylight separately and want the Mininet switch to act as the data plane.

## 9.21.4 MiniEdit GUI, drag-and-drop topology design

MiniEdit is a Python Tkinter GUI that lives in the Mininet source tree under `examples/miniedit.py`. Run it with:

```bash
sudo python /home/admin/mininet/examples/miniedit.py
```

(The exact path depends on the install method. The Crichigno lab machine ships with a desktop shortcut.) MiniEdit opens a window with a toolbar on the left side:

1. **Select** (arrow): select or move a device. The Del key removes the selected device.
2. **Host**: add a host. Click on the canvas to drop one.
3. **Legacy switch**: add a switch. Click on the canvas to drop one.
4. **Link**: connect two devices. Click the first device, drag to the second.
5. **Run**: start emulation. Every button is disabled until you press Stop.
6. **Stop**: end emulation.

### Configure a device with the mouse

Right-click on a host, choose **Properties**, and fill in the IP address (for example, `10.0.0.1/8`), MAC, and default route. Do the same for switches: configure the OpenFlow version and the controller target.

### Open a terminal on a host

After you press Run to start the emulation, right-click on a host and choose **Terminal**. An xterm opens inside the network namespace of that host. The `ifconfig` command shows the interfaces of the host, not of the host machine.

### Save and load a topology

File and Save opens a dialog to save a `.mn` file (the JSON format that defines the topology). File and Open loads an existing `.mn` file. The Lab 2 file is saved as `lab2.mn`. Opening it shows a topology with 2 LANs and 1 router r1.

### Export a Python script

File and Export Level 2 Script writes out a `.py` file that you can run directly with Python. The output is equivalent to using the `Topo` class. The common workflow is to design the topology in MiniEdit (easy to visualize), export the Python script (easy to put under version control), and from there continue editing as code to scale or to create variants.

### X11 forwarding when running over SSH

If Mininet runs on a remote machine and you SSH into it, MiniEdit needs X11 forwarding. The setup is:

```bash
# On the client machine (Linux or macOS)
ssh -X admin@mininet-host

# Verify that X11 forwarding works
echo $DISPLAY      # Must show a value such as "localhost:10.0"
xauth list         # Must list at least one entry
```

A Windows client needs VcXsrv or Xming as a local X server. After that, use PuTTY with X11 forwarding enabled.

## 9.21.5 Router emulation, sysctl and static route

A Mininet host by default has only end-host behavior at Layer 3. To make a host act as a router, enable IP forwarding:

```bash
# Inside the terminal of host r1 in Mininet
sysctl -w net.ipv4.ip_forward=1
```

Section 4 of Lab 2 (pages 19 to 25) builds a topology with two LANs connected through router r1:

```
LAN1 (192.168.1.0/24)              LAN2 (192.168.2.0/24)
  h1 (.10) ---- s1 ---- r1 ---- s2 ---- h2 (.10)
                         |
                     .1 LAN1 side
                     .1 LAN2 side
```

The Python script:

```python
from mininet.topo import Topo

class TwoLanRouterTopo(Topo):
    def build(self):
        h1 = self.addHost('h1', ip='192.168.1.10/24',
                          defaultRoute='via 192.168.1.1')
        h2 = self.addHost('h2', ip='192.168.2.10/24',
                          defaultRoute='via 192.168.2.1')

        # r1 has 2 interfaces, one per subnet
        r1 = self.addHost('r1')

        s1 = self.addSwitch('s1')
        s2 = self.addSwitch('s2')

        self.addLink(h1, s1)
        self.addLink(h2, s2)
        self.addLink(s1, r1, intfName2='r1-eth1',
                     params2={'ip': '192.168.1.1/24'})
        self.addLink(s2, r1, intfName2='r1-eth2',
                     params2={'ip': '192.168.2.1/24'})

topos = {'tworouter': (lambda: TwoLanRouterTopo())}
```

Run it:

```bash
sudo mn --custom tworouter.py --topo tworouter
containernet> r1 sysctl -w net.ipv4.ip_forward=1
containernet> h1 ping -c 4 192.168.2.10
```

If the ping does not get through, the common reasons are: r1 has the right interfaces but `ip_forward` is not enabled, or the `defaultRoute` on h1 or h2 is missing. Confirm with `h1 ip route`.

## 9.21.6 Integrating Mininet with OVS

By default Mininet uses a Linux kernel bridge switch (legacy) or OVS, depending on the install. To force OVS, add the `--switch ovsk` option:

```bash
sudo mn --custom mytopo.py --topo mytopo --switch ovsk
```

After the emulation starts, real OVS bridges named `s1`, `s2`, and so on appear in the OVS database. Open a second terminal outside Mininet (a second terminal on the lab machine) and check:

```bash
sudo ovs-vsctl show
# Output: Bridge s1 with interfaces s1-eth1 and s1-eth2

sudo ovs-ofctl dump-flows s1
# Output: the current flow table (default has just one entry "actions=NORMAL")

sudo ovs-ofctl add-flow s1 "priority=100,ip,nw_dst=10.0.0.2,actions=output:2"
# Install a flow manually
```

Every `ovs-ofctl`, `ovs-appctl ofproto/trace`, and `ovs-dpctl` command from Parts 9.4 to 9.25 applies to the bridges that Mininet creates. This is the mechanism that lets Crichigno Labs 3 to 15 use Mininet to build the topology and the OVS CLI to install flows.

### Connect to an external controller

When learning SDN with Ryu, ONOS, or OpenDaylight, start the controller on port 6653 first:

```bash
# Terminal 1: start Ryu
ryu-manager ryu.app.simple_switch_13
```

Then start Mininet:

```bash
# Terminal 2: start Mininet with a remote controller
sudo mn --custom mytopo.py --topo mytopo --switch ovsk \
    --controller=remote,ip=127.0.0.1,port=6653
```

The OVS switches in the topology connect to Ryu and send packet-in messages whenever the flow table does not match. Ryu replies with packet-out and flow-mod messages. This integration is the foundation of every SDN controller development workflow.

## 9.21.7 Comparison: Mininet versus a hand-built namespace

Before Mininet, developers built test topologies by combining `ip netns` and `ip link` by hand. A typical hand-written script:

```bash
#!/bin/bash
# Create namespaces
sudo ip netns add h1
sudo ip netns add h2

# Create a veth pair, with one end inside the namespace
sudo ip link add h1-eth0 type veth peer name s1-eth1
sudo ip link add h2-eth0 type veth peer name s1-eth2
sudo ip link set h1-eth0 netns h1
sudo ip link set h2-eth0 netns h2

# Assign IP inside each namespace
sudo ip netns exec h1 ip addr add 10.0.0.1/8 dev h1-eth0
sudo ip netns exec h1 ip link set h1-eth0 up
sudo ip netns exec h1 ip link set lo up
# Repeat for h2...

# Create the OVS bridge, add the other end of the veth
sudo ovs-vsctl add-br s1
sudo ovs-vsctl add-port s1 s1-eth1
sudo ovs-vsctl add-port s1 s1-eth2
sudo ip link set s1-eth1 up
sudo ip link set s1-eth2 up
```

The same topology in Mininet is just five lines of Python `Topo` class code. The advantages of Mininet:

- **Bring-up speed**: a 200-line bash script versus 5 lines of Python.
- **Automatic cleanup**: `exit` from Mininet removes every namespace and every bridge. No separate cleanup script is needed.
- **Mininet CLI**: type `h1 ping h2` directly in the prompt, with no `ip netns exec h1` wrapper.
- **Scale**: add a host with one line `self.addHost('h100')`. Hand-written code requires copy-paste of about 10 lines.
- **Reproducibility**: share a short `.py` file, and another engineer can rebuild the same topology with one command.

The disadvantages of Mininet:

- **Mininet install dependency**: needs the `mininet`, `python3-mininet`, and `openvswitch-switch` packages. On a production server that does not have them, you fall back to hand-built namespaces.
- **Bring-up overhead**: Mininet takes a few seconds to start. A hand-built namespace starts almost instantly.
- **Does not emulate high-speed hardware accurately**: veth is a software channel and has no real backpressure like a 100 Gbps NIC. For large-scale tests above 10 Gbps, use DPDK or a hardware testbed.

The practical rule is to learn and prototype SDN with Mininet, deploy the production environment with OVS or OVN on a real kernel or DPDK, and benchmark performance on a hardware testbed. Mininet is the first step, not the last step.

## Guided Exercise 1: Reconstruct the Lab 5 topology with a Mininet script

**Goal:** Walk through a real example of "read an existing lab topology, then write a Python script to reproduce it". This is the standard workflow for every Crichigno Lab between 3 and 15.

**Hypothesis to verify:**

> *"For the Crichigno Lab 5 topology (one OVS switch plus four hosts on the same subnet 10.0.0.0/8), I can write a Python script using the `Topo` class that reproduces it exactly, run it with `mn --custom script.py --topo lab5`, and verify with `net` plus `pingall` in the Mininet CLI."*

### Setup

Install Mininet and OVS on Ubuntu:

```bash
sudo apt update
sudo apt install -y mininet openvswitch-switch python3-mininet
```

Check the versions:

```bash
mn --version     # Mininet 2.3.0 or higher
ovs-vsctl --version
```

### Step 1: Read the Lab 5 topology

According to Crichigno Lab 5 (pages 6 to 8), the topology contains:

- One OVS switch named `s1`.
- Four hosts h1, h2, h3, h4, all on the same subnet 10.0.0.0/8.
- IP: h1=10.0.0.1, h2=10.0.0.2, h3=10.0.0.3, h4=10.0.0.4.
- MAC: h1=00:00:00:00:00:01, h2=00:00:00:00:00:02, h3=00:00:00:00:00:03, h4=00:00:00:00:00:04.
- Each host connects to the switch through interface `hN-eth0` and `s1-ethN`.

### Step 2: Write the Python script

Create the file `lab5.py`:

```python
# lab5.py - Reproduction of the Crichigno Lab 5 topology
from mininet.topo import Topo

class Lab5Topo(Topo):
    "Lab 5: 1 switch plus 4 hosts on the same subnet."
    def build(self):
        s1 = self.addSwitch('s1')

        for i in range(1, 5):
            hi = self.addHost(
                f'h{i}',
                ip=f'10.0.0.{i}/8',
                mac=f'00:00:00:00:00:0{i}'
            )
            self.addLink(hi, s1)

topos = {'lab5': (lambda: Lab5Topo())}
```

### Step 3: Run Mininet with the script

```bash
sudo mn --custom lab5.py --topo lab5 --switch ovsk
```

Expected: the prompt becomes `containernet>`. The verification commands:

```
containernet> nodes
# Expected: c0 s1 h1 h2 h3 h4

containernet> net
# Expected:
# h1 h1-eth0:s1-eth1
# h2 h2-eth0:s1-eth2
# h3 h3-eth0:s1-eth3
# h4 h4-eth0:s1-eth4
# s1 lo:  s1-eth1:h1-eth0 s1-eth2:h2-eth0 s1-eth3:h3-eth0 s1-eth4:h4-eth0
# c0
```

### Step 4: Verify connectivity

```
containernet> pingall
```

Expected (doc-plausible):

```
*** Ping: testing ping reachability
h1 -> h2 h3 h4
h2 -> h1 h3 h4
h3 -> h1 h2 h4
h4 -> h1 h2 h3
*** Results: 0% dropped (12/12 received)
```

A 0 percent packet drop is the sign that the topology is correct and that OVS is auto-learning MAC addresses through the default `NORMAL` action.

### Step 5: Verify that MAC addresses are assigned correctly

```
containernet> h1 ifconfig h1-eth0
```

Expected line `ether 00:00:00:00:00:01` in the output. Same idea for h2, h3, h4.

### Step 6: Verify the OVS bridge from outside

Open a second terminal on the lab machine (not inside the Mininet CLI):

```bash
sudo ovs-vsctl show
```

Expected: a Bridge `s1` with four interfaces `s1-eth1` through `s1-eth4`.

```bash
sudo ovs-ofctl dump-flows s1
```

Expected: just one default flow `actions=NORMAL`. This is why `pingall` works: NORMAL turns on automatic MAC learning.

### Step 7: Exit and clean up

```
containernet> exit
```

Then in the shell:

```bash
sudo mn -c
```

### Lessons learned

Writing a Mininet script takes less than ten minutes for a basic five-node topology. This workflow applies to every Crichigno Lab from 3 to 15: read the topology, map it onto `addHost`, `addSwitch`, and `addLink`, run it, and verify. Once you are comfortable with the steps, you can change parameters (IP, MAC, host count, ring or star or mesh topology) in just a few lines to test your own hypothesis. That is the core value of Mininet: it shrinks the time from idea to experiment to a few minutes.

## 9.21.8 Operating tips for the rest of Block IX

In Parts 9.22 through 9.25, every lab topology is built with Mininet. Three tips help the workflow run smoothly.

**Tip 1: Always run `sudo mn -c` before each new bring-up.** Leftover namespaces from a previous session cause confusing errors ("port already exists", "device busy"). One `mn -c` call resolves most of them.

**Tip 2: Use two terminals.** Terminal 1 runs the Mininet CLI (`sudo mn ...`). Terminal 2 runs the OVS CLI (`ovs-ofctl`, `ovs-appctl`) on the same lab machine. Terminal 2 sees the OVS bridge that Mininet created, which lets you install flows manually without exiting Mininet.

**Tip 3: Keep the Python topology scripts under version control.** Each part in the series has a file like `lab<N>.py` or `part9-<X>.py`. Commit them to git so you can reproduce the exact topology when a lab fails or when you compare behavior across OVS versions.

## 9.21.9 Key takeaways

**One.** Mininet has been the de-facto standard for reproducible SDN research since 2010. It lets you reproduce a topology with a single Python script on a single laptop.

**Two.** Each Mininet host is a Linux network namespace; each cable is a veth pair. A 100-node topology runs on a 4 GB laptop without trouble because both namespaces and veth are extremely lightweight.

**Three.** Basic CLI: `sudo mn` to start, `help`, `nodes`, `net`, `dump` to inspect, `h1 <cmd>` to run a command on a host, `pingall` to verify, `exit` to leave, `sudo mn -c` to clean up.

**Four.** Custom topology in Python with the `Topo` class: `addHost`, `addSwitch`, `addLink`. Export the variable `topos = {'name': (lambda: MyTopo())}` so that `mn --custom file.py --topo name` can pick it up.

**Five.** Use the MiniEdit GUI for design, and a Python script for deployment or version control. The standard workflow is to draw on MiniEdit, export Python, and continue editing as code.

**Six.** Integrate OVS with `--switch ovsk`. After the emulation starts, `ovs-vsctl show` and `ovs-ofctl dump-flows` from an external terminal allow you to install flows and to debug with the toolbox in Part 9.25.

**Seven.** Mininet beats a hand-built namespace on bring-up speed, cleanup, scaling, and reproducibility. It loses to a hand-built namespace when the bring-up overhead must be near zero or when the package is not installable. It loses to real hardware when you need to benchmark at 100 Gbps.

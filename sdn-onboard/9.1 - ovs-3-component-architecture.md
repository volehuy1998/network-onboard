## Bối cảnh: Vì sao OVS lại chia thành ba thành phần riêng biệt?

Một câu hỏi thường gặp khi mới tiếp cận OVS là: "Trong khi Linux bridge chỉ là một kernel module đơn lẻ, tại sao OVS lại cần tới ba thành phần khác nhau?". Câu trả lời nằm ở yêu cầu kiến trúc trọng yếu mà OVS phải đáp ứng ngay từ những ngày đầu thiết kế (2007): **khả năng thay đổi chính sách chuyển tiếp (forwarding policy) từ xa một cách linh hoạt mà không cần khởi động lại hệ thống hay làm gián đoạn kết nối hiện tại**.

Linux bridge giải quyết bài toán "Switch học địa chỉ L2 cơ bản" với các hành vi cố định. Mọi thay đổi về cấu hình (như thêm VLAN, chỉnh mức ưu tiên STP) đều phải thông qua lệnh `brctl` từ userspace, lệnh này sẽ trực tiếp thay đổi trạng thái trong kernel qua cơ chế ioctl. Cách tiếp cận này đơn giản nhưng thiếu linh hoạt, không có database hay tiến trình quản lý trung gian.

OVS phải xử lý những bài toán phức tạp hơn nhiều: một bộ điều khiển SDN (controller) có thể đẩy xuống **hàng nghìn luật OpenFlow** chỉ trong một giây. Các luật này cần được kiểm tra tính hợp lệ (validate), lưu trữ bền vững, biên dịch thành các flow entry cho kernel và phải có khả năng hoàn tác (rollback) nếu phát sinh lỗi. Những xử lý phức tạp này (như chạy state machine, phân tách dữ liệu JSON, quản lý kết nối TCP) không thể thực hiện trực tiếp trong kernel. Đó là lý do vì sao cần có một tiến trình chạy ở tầng userspace (userspace daemon).

Bên cạnh đó, các thông tin cấu hình (bridge, port, VLAN tag, QoS) đòi hỏi **khả năng lưu trữ bền vững sau khi khởi động lại** và tính chất **giao dịch nguyên tử (atomic transaction)**. Chẳng hạn, khi thêm hai port cùng lúc, hoặc cấu hình một hàng đợi (queue) tham chiếu đến chính sách QoS, tất cả phải cùng thành công hoặc cùng bị hủy bỏ để đảm bảo tính nhất quán. Giải pháp tối ưu chính là sử dụng cơ chế giao dịch của cơ sở dữ liệu. Tuy nhiên, OVS cần một loại database **nhẹ, có sơ đồ dữ liệu (schema) rõ ràng và hỗ trợ thông báo theo thời gian thực**. Do đó, Nicira đã tự thiết kế OVSDB, và `ovsdb-server` chính là tiến trình độc lập đảm nhiệm việc quản lý tệp tin database này.

Cuối cùng, tầng dữ liệu (data plane) - nơi thực hiện việc chuyển tiếp gói tin thực tế - bắt buộc phải nằm ở **kernel** để đạt được hiệu năng cao (băng thông 10+ Gbps). Việc bắt mỗi gói tin đi qua network stack của kernel phải chuyển sang userspace để xử lý là quá tốn kém (mỗi lần chuyển đổi ngữ cảnh context switch và sao chép dữ liệu tiêu tốn khoảng 2 microsecond, quá chậm đối với tốc độ hàng triệu gói tin mỗi giây). Vì vậy, OVS sử dụng kernel module `openvswitch.ko` để vận hành "đường truyền nhanh" (fast path).

Kết quả là chúng ta có ba thành phần, mỗi thành phần đảm nhận một nhiệm vụ chuyên biệt và giao tiếp với nhau qua các giao diện (interface) rõ ràng.

## 9.1.1 Tổng quan về ba thành phần và cơ chế tương tác

Mô hình tư duy bốn lớp (four-layer mental model) là công cụ cực kỳ hiệu quả để bao quát kiến trúc OVS khi cần xử lý sự cố:

```
Lớp 1 — Config plane:     OVSDB (ovsdb-server)
                          "Cần tồn tại những gì"
Lớp 2 — Control plane:    OpenFlow (ovs-vswitchd)
                          "Gói tin nên được xử lý như thế nào"
Lớp 3 — Datapath cache:   Kernel module openvswitch.ko (hoặc userspace netdev)
                          "Những gì fast path đã học được"
Lớp 4 — Wire:             Card mạng vật lý hoặc giao diện ảo
                          "Những gì thực sự đi ra khỏi NIC"
```

> **Lưu ý quan trọng:** Phần lớn công việc chẩn đoán lỗi trong OVS là **so sánh giữa bốn lớp này để tìm ra điểm mâu thuẫn**. Ví dụ, nếu OVSDB báo bridge có 3 port, nhưng OpenFlow table chỉ ghi nhận 2 port, trong khi kernel datapath lại thấy 3 port, thì rõ ràng đang có sự **mâu thuẫn trạng thái (state mismatch)** giữa lớp 1 và lớp 2 — thường là do vấn đề đồng bộ giữa `ovsdb-server` và `ovs-vswitchd`. Việc hiểu rõ vai trò của từng lớp giúp chúng ta thu hẹp phạm vi tìm lỗi một cách nhanh chóng.

**Sơ đồ luồng xử lý gói tin đi qua OVS (Sử dụng kernel datapath)**:

```
┌─────────────────────────────────────────────────────────────────┐
│                     Userspace                                    │
│                                                                   │
│  ┌────────────────┐       ┌──────────────────────────┐          │
│  │ ovsdb-server   │◄─────►│ ovs-vswitchd             │          │
│  │ (config DB)    │ JSON-  │  - OpenFlow pipeline    │          │
│  │ /etc/openvswi- │  RPC  │  - classifier (TSS)      │          │
│  │ tch/conf.db    │       │  - handler threads        │          │
│  └────────────────┘       │  - revalidator threads   │          │
│                            └────────┬─────────────────┘          │
│                                     │                             │
└─────────────────────────────────────┼─────────────────────────────┘
                             Netlink genl
                          (ovs_datapath, ovs_flow,
                             ovs_vport families)
                                      │
┌─────────────────────────────────────▼─────────────────────────────┐
│                     Kernel space                                   │
│                                                                     │
│  ┌──────────────────────────────────────────────┐                 │
│  │  openvswitch.ko                               │                 │
│  │                                                │                 │
│  │   Gói tin đến ──► Tra cứu Megaflow cache      │                 │
│  │                    │                           │                 │
│  │                    ├── hit  ──► Thực thi action│                 │
│  │                    │                           │                 │
│  │                    └── miss ──► Upcall lên user│                 │
│  │                                    │           │                 │
│  │                                    ▼           │                 │
│  │                              ovs-vswitchd      │                 │
│  │                                                │                 │
│  └──────────────────────────────────────────────┘                 │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

Ba thành phần này không kết nối trực tiếp theo kiểu bắc cầu đơn thuần. `ovs-vswitchd` đóng vai trò trung tâm: nó kết nối với `ovsdb-server` qua Unix socket JSON-RPC (để đọc cấu hình và nhận thông báo thay đổi), đồng thời kết nối với kernel module qua Netlink genl family (để cài đặt các dòng chảy và nhận Upcall khi gặp cache miss). Lưu ý rằng `ovsdb-server` không bao giờ giao tiếp trực tiếp với kernel, và kernel cũng không bao giờ đọc dữ liệu từ OVSDB.

## 9.1.2 ovs-vswitchd: Tiến trình chuyển tiếp chính ở tầng userspace

`ovs-vswitchd` là tiến trình đóng vai trò quan trọng nhất, nơi tập trung hầu hết logic xử lý của OVS. Đây là một tệp thực thi (binary) duy nhất, chạy dưới dạng một tiến trình đơn nhưng sử dụng cơ chế đa luồng (multi-threading) để xử lý các khối lượng công việc song song.

**Vòng lặp chính (Main loop)** của `ovs-vswitchd` dựa trên cơ chế epoll và poll timer. Trong mỗi chu kỳ lặp:

1. Kiểm tra kết nối OVSDB xem có thay đổi về cấu hình hay không.
2. Kiểm tra kết nối OpenFlow từ controller để nhận các yêu cầu như `flow_mod`, `packet_out`.
3. Kiểm tra Netlink socket để nhận các Upcall từ kernel datapath.
4. Kiểm tra Unix socket `/var/run/openvswitch/ovs-vswitchd.ctl` để nhận lệnh điều khiển từ `ovs-appctl`.
5. Thực hiện các tác vụ định kỳ (như cân bằng tải bond, gửi gói tin STP BPDU).

Trong khi luồng chính xử lý hầu hết các sự kiện, một số công việc đặc thù được tách ra các **nhóm luồng (thread pool) riêng biệt**:

- **Handler threads**: Đảm nhận việc xử lý các gói tin Upcall từ kernel khi gặp cache miss. Mỗi khi có Upcall, các luồng này sẽ phân tích gói tin, duyệt qua OpenFlow pipeline, biên dịch thành megaflow entry và đẩy xuống kernel qua Netlink.
- **Revalidator threads**: Định kỳ (mặc định mỗi 500ms) quét toàn bộ megaflow cache trong kernel để kiểm tra tính hợp lệ của từng entry (ví dụ: luật flow có thay đổi không, đã quá thời gian chờ idle chưa) và xóa bỏ các entry đã cũ (stale).
- **OpenFlow worker threads**: Xử lý các kết nối OpenFlow từ controller.

Sử dụng lệnh `ovs-appctl memory/show` giúp chúng ta xem chi tiết mức độ chiếm dụng bộ nhớ của từng thành phần như classifier (bảng flow), handlers, revalidators. Lệnh này cực kỳ hữu ích khi cần chẩn đoán hiện tượng tràn bộ nhớ.

```bash
[host]# ovs-appctl -t ovs-vswitchd memory/show
handlers:4 ports:5 revalidators:4 rules:127 ...
```

Đọc các chỉ số từ `ovs-appctl coverage/show` giúp phát hiện những điểm bất thường: chẳng hạn nếu tỷ lệ `cache-misses` tăng vọt trong khi `flow-lookup` giảm, điều đó có nghĩa là lưu lượng mạng đang tạo ra quá nhiều dòng chảy nhỏ (microflow) khác nhau, khiến cơ chế cache không phát huy được hiệu quả.

## 9.1.3 ovsdb-server: Cơ sở dữ liệu cấu hình nhẹ nhàng

`ovsdb-server` là tiến trình "nhẹ ký" nhất về mặt chiếm dụng tài nguyên. Nhiệm vụ duy nhất của nó là cung cấp dịch vụ cho một tệp tin cơ sở dữ liệu OVSDB thông qua giao thức JSON-RPC.

Tệp database của OVS nằm tại `/etc/openvswitch/conf.db`, với sơ đồ dữ liệu (schema) quy định các bảng quan trọng như `Bridge`, `Port`, `Interface`, `Controller`, v.v.

Các thành phần giao tiếp với `ovsdb-server` qua Unix socket hoặc TCP/SSL. Có hai đối tượng chính thường xuyên tương tác:

- **`ovs-vsctl`**: Công cụ dòng lệnh dành cho quản trị viên. Mỗi lệnh thực thi được coi là một giao dịch (transaction) duy nhất, đảm bảo tính nguyên tử (thành công tất cả hoặc không gì cả).
- **`ovs-vswitchd`**: Kết nối với `ovsdb-server` ngay khi khởi động và đăng ký theo dõi (monitor) toàn bộ database. Bất kỳ thay đổi nào trong database sẽ được đẩy (push) theo thời gian thực về `ovs-vswitchd` mà không cần phải thăm dò (polling) liên tục.

Một điểm mấu chốt cần nhớ: OVSDB chỉ lưu trữ **ý đồ cấu hình** (ví dụ: người dùng muốn tạo bridge X), chứ không lưu trữ **trạng thái vận hành thực tế** (ví dụ: bridge đó đang có bao nhiêu lưu lượng qua lại). Các số liệu thống kê vận hành sẽ được `ovs-vswitchd` tính toán tức thời và cung cấp qua các cột đặc biệt, nhưng chúng không được lưu trữ vĩnh viễn xuống đĩa cứng.

## 9.1.4 openvswitch.ko: Tầng dữ liệu trong kernel (Fast Path)

Kernel module `openvswitch.ko` chính là nơi thực thi việc chuyển tiếp gói tin ở tốc độ cao (fast path). Khi một gói tin đi vào card mạng, kernel sẽ chuyển giao cho module này xử lý.

`openvswitch.ko` quản lý hai cấu trúc dữ liệu quan trọng trong bộ nhớ kernel:

**1. Bảng dòng chảy (Flow table) trong kernel:** Thực chất đây là một bản Megaflow cache. Mỗi entry bao gồm các trường khớp (match key) với các mặt nạ (mask), danh sách các hành động (action) cần thực hiện và các bộ đếm thống kê.

**2. Bảng cổng (Port table):** Ánh xạ giữa số hiệu cổng OpenFlow và con trỏ card mạng tương ứng trong hệ thống.

Khi một gói tin đến mà không tìm thấy entry nào khớp trong cache (cache miss), kernel sẽ không hủy gói tin ngay mà thực hiện **Upcall gửi lên tầng userspace**. Thông điệp Upcall này chứa gói tin gốc cùng các thông tin bổ trợ. `ovs-vswitchd` sẽ tiếp nhận, xử lý qua OpenFlow pipeline để tìm ra hành động phù hợp, biên dịch thành một megaflow entry mới và **đẩy ngược xuống kernel**. Những gói tin tương tự sau đó sẽ khớp trực tiếp trong kernel, giúp việc chuyển tiếp trở nên cực nhanh.

> **Giải thích thêm:** Upcall là cơ chế giúp OVS đạt được cả hai mục tiêu: tốc độ (nhờ fast path trong kernel) và sự linh hoạt (nhờ logic phức tạp ở userspace). Theo các nghiên cứu thực tế, tỷ lệ hit cache thường đạt trên 99%, nghĩa là chỉ một phần rất nhỏ gói tin đầu tiên của một dòng chảy mới phải chuyển lên userspace, chi phí này sau đó được bù đắp bởi tốc độ xử lý hàng nghìn gói tin tiếp theo ngay trong kernel.

Kernel datapath thường có hai loại:
- **`datapath_type=system`** (mặc định): Sử dụng kernel module, hiệu năng thường đạt 5-10 Gbps tùy phần cứng.
- **`datapath_type=netdev`**: Sử dụng tầng dữ liệu ở userspace (thường kết hợp với DPDK). Gói tin sẽ bỏ qua kernel đi thẳng vào userspace, cho phép đạt băng thông cực lớn (50+ Gbps) nhưng đòi hỏi tài nguyên CPU chuyên dụng cao hơn.

## 9.1.5 Netlink genl family: Cầu nối giữa Userspace và Kernel

Netlink là cơ chế giao tiếp đặc biệt trong Linux dùng để trao đổi thông tin giữa tầng người dùng và nhân hệ điều hành. OVS sử dụng một lớp nâng cao gọi là **Generic Netlink (genl)**.

OVS đăng ký ba "họ" (family) Netlink chính:
- **`ovs_datapath`**: Quản lý vòng đời của datapath (tạo/xóa bridge ở tầng kernel).
- **`ovs_flow`**: Quản lý các flow entry trong datapath. Các gói tin Upcall cũng đi qua họ này.
- **`ovs_vport`**: Quản lý các cổng ảo (interface) được gán vào bridge.

Trong quá trình vận hành, nếu bạn thấy các thông báo lỗi liên quan đến Netlink, đó thường là dấu hiệu của việc **mâu thuẫn phiên bản** giữa tiến trình `ovs-vswitchd` và kernel module `openvswitch.ko` (ví dụ: vừa nâng cấp phần mềm OVS nhưng chưa cập nhật module kernel tương ứng).

## Tóm tắt những điểm cốt lõi

1. **Vì sao phải chia ba?** Bởi vì mỗi tầng có đặc thù riêng: Cấu hình cần sự bền vững (OVSDB), logic điều khiển phức tạp cần sự linh hoạt của userspace (ovs-vswitchd), và chuyển tiếp dữ liệu cần tốc độ tối đa của kernel.
2. **Mô hình tư duy 4 lớp:** Luôn so sánh giữa Cấu hình (OVSDB) → Điều khiển (OpenFlow) → Cache (Datapath) → Đường truyền thực tế (Wire). Mọi lỗi vận hành đều nằm ở sự mâu thuẫn giữa các lớp này.
3. **Cơ chế Upcall:** Là "linh hồn" của OVS, giúp hài hòa giữa tính năng và hiệu năng.
4. **Bộ công cụ chẩn đoán:** Hãy thuộc lòng các lệnh `ovs-vsctl`, `ovs-ofctl`, `ovs-dpctl` và `ovs-appctl`. Đó là những "ống nghe" giúp kỹ sư biết chính xác hệ thống đang gặp vấn đề ở đâu.

## 9.1.Y Công cụ xlate trong ofproto-dpif (Bổ sung Phase I)

**Xlate engine** là thành phần dịch gói tin và bảng OpenFlow thành danh sách hành động datapath. Đây là con đường xử lý quan trọng nhất của OVS ở tầng userspace.

### 9.1.Y.1 Khi nào xlate được kích hoạt?

1. **Khi có Upcall từ kernel:** Khi kernel không tìm thấy dòng chảy khớp cho gói tin đầu tiên.
2. **Khi Revalidator quét định kỳ:** Để kiểm tra xem các luật có thay đổi hay không.
3. **Khi chạy lệnh `ofproto/trace`:** Dùng để chẩn đoán luồng đi của gói tin một cách thủ công.

### 9.1.Y.2 Cơ chế làm việc của xlate_actions()

Hàm này nhận đầu vào là thông tin gói tin và trả về danh sách các hành động cho datapath cùng với một "mặt nạ" (megaflow mask). Trong quá trình duyệt qua các bảng OpenFlow, xlate sẽ đánh dấu các trường dữ liệu nào đã được kiểm tra để tạo ra mặt nạ Megaflow tối ưu nhất.

### 9.1.Y.3 Các cơ chế bảo vệ trong xlate

Để tránh các sai sót trong cấu hình luật (như tạo vòng lặp vô hạn), OVS đặt ra các giới hạn cứng:
- **Độ sâu đệ quy tối đa:** 64 cấp (đối với hành động `resubmit`).
- **Tổng số lần resubmit tối đa:** 4096 lần.
Nếu vượt quá, quá trình xử lý sẽ bị hủy bỏ để bảo vệ hệ thống không bị treo.

### 9.1.Y.4 Tại sao phải tách biệt giữa Kernel và Userspace?

Kernel datapath thực chất rất "ngây thơ", nó không biết OpenFlow là gì. Nó chỉ thực thi những hành động cơ bản do userspace đẩy xuống. Việc tách biệt này giúp:
- Kernel giữ được sự đơn giản và tốc độ cao.
- Dễ dàng nâng cấp logic xử lý ở userspace mà không cần can thiệp vào kernel.
- Tận dụng cơ chế Megaflow để giảm thiểu số lần phải chuyển lên userspace xử lý.

## 9.1.Z Bổ sung phương diện cho ovs-vswitchd (mở rộng 2026-04)

> **Mục đích:** Bổ sung các phương diện còn thiếu cho keyword `ovs-vswitchd` cornerstone. Tier 1 importance ngang megaflow, ct_state.

### Tầm quan trọng

`ovs-vswitchd` là **trụ cột tuyệt đối** của OVS. Mọi data plane decision đều thông qua nó. Không có ovs-vswitchd, OVS chỉ là kernel module trơ không có flow rule. Tier 1 must-know ngang megaflow + ct_state.

### Phân loại

Loại: userspace daemon process. Parent: OVS 3-component architecture (kernel datapath / userspace / OVSDB). Scope: per-host (1 instance manage all bridge trên host). Lifecycle: long-running daemon, restart triggers brief data plane outage (mitigated qua flow-restore-wait option).

### Khác biệt giữa các phiên bản

| Version | Behavior |
|---------|----------|
| OVS 1.x | Single-threaded core loop |
| OVS 2.0 (2014) | Multi-thread handler + revalidator, megaflow architecture |
| OVS 2.4 | Configurable n-handler-threads + n-revalidator-threads |
| OVS 2.6 | EMC/SMC tier added |
| OVS 2.8 | NSH support |
| OVS 3.0+ | Stable, tunable continues |

### Trường hợp sự cố thực tế

**2024-Q1 cluster:** ovs-vswitchd OOM crash sau 2 weeks uptime. Investigation: revalidator thread leak (rare bug OVS 2.13). Heap grew 50MB/day. Fix: upgrade 2.13 → 2.17 contains memory leak fix commit `e2c3...`.

### Bài tập synthetic

```
$ sudo systemctl status openvswitch-switch
$ ps -ef | grep ovs-vswitchd
→ Single process per host
$ ovs-appctl version
$ ovs-appctl coverage/show
→ Internal counter snapshot
$ ovs-appctl memory/show
→ Heap profile (release builds only)
```

### Lỗi thường gặp + tín hiệu chẩn đoán

- Mode A: Process crash → flow-restore-wait misconfigured → data plane outage. Recovery: enable flow-restore-wait=true.
- Mode B: Revalidator thread saturation → CPU 100%. Recovery: increase n-revalidator-threads.
- Mode C: Memory leak → OOM. Recovery: monitor + upgrade.

### So sánh với hệ khác

- **VPP (Vector Packet Processing):** alternative userspace packet processing. Khác: VPP single-process vector batched; ovs-vswitchd handler/revalidator/main thread architecture.
- **Kernel-bypass alternatives:** một số stack thuần userspace (out of curriculum scope) cũng giải bài data plane forwarding nhưng không tách rõ control/data plane như ovs-vswitchd.

## 9.1.AB dpif, lớp trừu tượng giữa userspace ofproto-dpif và datapath provider

> **Phạm vi:** Treatment chuyên sâu (mở rộng 2026-04-26) cho keyword `dpif` (Datapath Interface). Keyword này là cornerstone của OVS internals vì là điểm duy nhất trong code base nơi userspace forwarding logic tách rời khỏi datapath implementation cụ thể (kernel module openvswitch.ko, userspace netdev provider, hoặc các provider tương lai). Cross-link mật thiết với §9.1.X (5-lớp ofproto-dpif), §9.1.X.2 (`dpif/show` output), §9.2 (kernel datapath megaflow là nội tại của một dpif provider), §9.32 §9.32.4 (compact architecture sketch trước đây). Curriculum baseline OVS v2.17.9.

**Khái niệm.** `dpif` là viết tắt của "DataPath InterFace", một abstract C interface định nghĩa trong `lib/dpif.h` và `lib/dpif-provider.h`. dpif đại diện cho "một switch nhìn từ phía userspace": tập hợp port, một flow table có khả năng cài/xóa/dump entry, một upcall channel để nhận packet không khớp flow, và một execute path để inject packet. dpif KHÔNG phải là kernel module, KHÔNG phải là một daemon, mà là **một bộ function pointer + struct layout** mà các provider concrete (`dpif-netlink`, `dpif-netdev`) phải implement. Nhờ dpif, mọi tầng phía trên trong `ovs-vswitchd` (bridge.c, ofproto.c, ofproto-dpif.c, ofproto-dpif-xlate.c, ofproto-dpif-upcall.c) chỉ cần biết "gọi dpif_flow_put rồi nhận kết quả", không cần biết flow đó kết thúc ở Linux kernel hay ở thread polling DPDK port. Loại: userspace abstraction layer (vtable pattern), file ngôn ngữ C, không phải database table, không phải binary, không phải daemon.

**Lịch sử + bối cảnh.** dpif xuất hiện trong giai đoạn refactor lớn đầu tiên của OVS (2010-2012) khi Nicira nhận ra mô hình "ovs-vswitchd biết trực tiếp cách nói chuyện với kernel module" (qua `lib/dpif-linux.c` đời đầu) là điểm chặn cho việc port OVS lên môi trường mới. Năm 2012 xuất hiện `dpif-netdev` cho phép chạy datapath thuần userspace; năm 2014 NSDI 2015 paper (Pfaff et al.) đã chính thức hoá dpif như "the datapath interface" trong sơ đồ kiến trúc. Năm 2014-2015 dpif-netdev được mở rộng để hỗ trợ DPDK PMD; sau này có thêm các provider thử nghiệm (windows, dummy cho test). Mỗi lần OVS thêm một datapath backend mới, không cần đụng vào ofproto, mà chỉ cần implement đủ các function pointer trong `dpif_class`. Đây là một trong những refactor có giá trị duy trì lâu dài nhất của OVS, vì kiến trúc này vẫn còn nguyên sau hơn một thập kỷ.

**Vị trí trong kiến trúc.** Trong sơ đồ 5 lớp của `ovs-vswitchd` (đã trình bày ở §9.1.X.1), dpif nằm ở **lớp dưới cùng userspace**, ngay trên ranh giới với data plane:

```
bridge.c                          (lớp 1, vòng đời bridge)
  └─ ofproto/ofproto.c            (lớp 2, OF protocol semantics)
      └─ ofproto/ofproto-dpif.c   (lớp 3, ofproto biết về dpif)
          ├─ ofproto-dpif-xlate.c (dịch OF → odp action)
          ├─ ofproto-dpif-upcall.c (nhận upcall từ dpif)
          └─ ofproto-dpif-rid.c   (recirc id pool)
              ↓ chỉ gọi qua dpif_* API
          ──────────────────────────
          lib/dpif.c               (lớp 4, dispatch wrapper)
              ↓ vtable dpif_class
          ──────────────────────────
          lib/dpif-netlink.c       (lớp 5a, kernel provider)
              ↓ Netlink genl
              kernel openvswitch.ko
          lib/dpif-netdev.c        (lớp 5b, userspace provider)
              ↓ in-process function call
              userspace netdev / DPDK PMD thread pool
```

Scope: per-`ovs-vswitchd`-process, mỗi process có thể mở nhiều dpif instance song song (mỗi bridge một dpif, hoặc nhiều bridge share một dpif type tuỳ provider). Một host thường có 1 dpif type chính (`system` cho kernel, `netdev` cho userspace), không trộn lẫn ngẫu nhiên trên cùng bridge.

**Vai trò.** Bốn vai trò cốt lõi: (a) **Provider dispatch**, `lib/dpif.c` giữ một registry các `dpif_class` (qua `dp_register_provider()`) và phân giải tên type (`system`, `netdev`) thành class tương ứng khi gọi `dpif_open()`. (b) **API uniformity**, mọi function gọi xuống datapath (cài flow, dump flow, port add, recv upcall, execute packet) đều đi qua một bộ `dpif_*` symbol thống nhất, ofproto-dpif không bao giờ gọi trực tiếp `dpif_netlink_*` hay `dpif_netdev_*`. (c) **Capability negotiation**, dpif expose feature matrix qua `dpif_class.cache_get_supported_levels`, `dpif_supports_*`, và lệnh `ovs-appctl dpif/show-dp-features` để xlate biết provider hiện tại có hỗ trợ một action không (ct, recirc, push_mpls, sample). (d) **Lifecycle + ownership**, dpif quản lý vòng đời datapath instance: `dpif_create`, `dpif_open`, `dpif_close`, `dpif_destroy`, đảm bảo ofproto không phải tự lo cleanup.

**Vì sao sinh ra.** Động cơ thiết kế là tách "what to do" ra khỏi "how to do it". Trước dpif, code OVS đời đầu (2009-2010) có hard-coded `linux_*` function gọi thẳng Netlink, làm việc port sang FreeBSD, port sang Hyper-V, port sang DPDK đều phải sửa hàng nghìn dòng tại nhiều nơi. dpif đảo ngược dependency: mọi nơi phía trên gọi qua interface, mọi nơi phía dưới chỉ cần implement interface. Đây là instance kinh điển của Dependency Inversion Principle áp dụng vào systems C. Lợi ích cụ thể đã chứng minh: (i) thêm DPDK datapath chỉ cần một file `dpif-netdev.c` lớn mà không phải sửa ofproto.c, (ii) thêm test backend (`dpif-dummy`) cho unit test ofproto mà không cần kernel module, (iii) thêm AF_XDP hỗ trợ thử nghiệm như một nhánh trong `dpif-netdev`.

**Vấn đề giải quyết.** Cụ thể dpif giải quyết bốn lớp vấn đề: (1) **Tính khả chuyển đa nền**, OVS không bị khoá vào Linux kernel; provider có thể là kernel module, là userspace polling, là hardware accelerator gateway. (2) **Khả năng test**, ofproto-dpif có thể test bằng `dpif-dummy` (in-memory) không cần root, không cần load kernel module. (3) **Capability divergence handling**, các provider khác nhau hỗ trợ khác nhau (ví dụ kernel có ct() từ 2.5, userspace netdev có ct() muộn hơn, một số tunnel type chỉ kernel hoặc chỉ userspace); xlate đọc capability matrix mà ra quyết định "emulate" hay "delegate". (4) **Performance isolation**, khi DPDK PMD thread polling không thể block trên I/O, dpif-netdev có cơ chế gọi async không khoá main thread của ovs-vswitchd; tách qua interface giúp ofproto code không phải tự xử lý sự khác biệt blocking model.

**Tầm quan trọng.** Đây là cornerstone tier 1 tuyệt đối của OVS internals. Không hiểu dpif thì không thể đọc được flow chi tiết của upcall, không debug được kernel-vs-userspace mismatch, không hiểu được tại sao cùng một OF rule lại có hành vi khác nhau giữa hai host (host A có `datapath_type=system`, host B có `datapath_type=netdev`). Importance ngang `megaflow`, `ct_state`, `recirc_id`, `ofproto-dpif xlate` trong tier cốt lõi OVS. Trong cấu trúc 320+ keyword của curriculum, dpif là "trục dọc" gắn kết toàn bộ Khối IX (OVS internals).

**Cơ chế hoạt động.** Trình tự per-bridge từ lúc `ovs-vsctl add-br br0` đến lúc packet đầu tiên được forward:

1. `bridge.c` đọc OVSDB, biết `Bridge.datapath_type = "system"` (mặc định khi chưa set).
2. `ofproto.c` gọi `ofproto_create()`, ofproto-dpif chọn dpif provider qua `dp_register_provider()` registry.
3. `lib/dpif.c::dpif_open("br0", "system", &dpif)` lookup `dpif_netlink_class`, gọi `dpif_netlink_class.open(...)`. Kernel module nhận message Netlink genl `OVS_DP_CMD_NEW`, tạo struct datapath trong `net/openvswitch/datapath.c`.
4. `ofproto-dpif.c` gọi `dpif_port_add()` cho mỗi `Port` row trong OVSDB; provider gửi `OVS_VPORT_CMD_NEW`.
5. ofproto cài initial flow table 0 với `dpif_operate(struct dpif_flow_put)`. Provider netlink translate sang `OVS_FLOW_CMD_NEW`.
6. ofproto-dpif-upcall.c spawn handler thread, mỗi thread block trên `dpif_recv()` chờ upcall.
7. Khi packet đầu tiên đến, kernel megaflow miss → upcall `OVS_PACKET_CMD_MISS` → handler thread `dpif_recv` trả về `struct dpif_upcall` chứa packet + key.
8. Handler chạy xlate, sinh datapath actions, gọi `dpif_flow_put()` cài megaflow + `dpif_execute()` để forward packet đầu tiên (vì kernel không tự forward khi miss).
9. Mọi packet sau cùng flow đi đường fast path trong kernel, không qua dpif (dpif chỉ tồn tại ở userspace).

`dpif_class` (vtable) trong `lib/dpif-provider.h` (OVS v2.17.9) tổ chức theo nhóm:

```
struct dpif_class {
    const char *type;                    /* "system", "netdev", "dummy" */
    int (*init)(void);                   /* Provider registration */

    /* Datapath lifecycle */
    int  (*open)(const struct dpif_class *, const char *name,
                 bool create, struct dpif **);
    void (*close)(struct dpif *);
    int  (*destroy)(struct dpif *);

    /* Port management */
    int (*port_add)(struct dpif *, struct netdev *, odp_port_t *port_no);
    int (*port_del)(struct dpif *, odp_port_t);
    int (*port_query_by_number)(const struct dpif *, odp_port_t,
                                struct dpif_port *);

    /* Flow table */
    int (*flow_flush)(struct dpif *);
    struct dpif_flow_dump *(*flow_dump_create)(const struct dpif *, ...);
    int (*flow_dump_next)(struct dpif_flow_dump_thread *,
                          struct dpif_flow *flows, int max_flows);
    void (*operate)(struct dpif *, struct dpif_op **ops, size_t n_ops, ...);
        /* ^ batched flow_put / flow_del / execute */

    /* Upcall / packet I/O */
    int  (*recv_set)(struct dpif *, bool enable);
    int  (*recv)(struct dpif *, uint32_t handler_id,
                 struct dpif_upcall *, struct ofpbuf *buf);
    void (*recv_wait)(struct dpif *, uint32_t handler_id);

    /* Conntrack, meter, bond, cache config (later additions) */
    int (*ct_dump_start)(struct dpif *, ...);
    int (*ct_flush)(struct dpif *, ...);
    int (*meter_set)(struct dpif *, ...);
    /* ... và một số method tiện ích khác */
};
```

Hai class concrete (cùng file `lib/dpif-provider.h` declare extern, implement trong file riêng):

| Trường | `dpif_netlink_class` | `dpif_netdev_class` |
|--------|---------------------|---------------------|
| File | `lib/dpif-netlink.c` | `lib/dpif-netdev.c` |
| `type` | `"system"` | `"netdev"` |
| Datapath | Linux kernel module openvswitch.ko | Userspace, in-process |
| Transport | Netlink genl (`ovs_datapath`, `ovs_flow`, `ovs_vport`) | Function call trong cùng process |
| `recv` | Netlink socket dequeue | PMD thread enqueue |
| Phù hợp | Production mặc định, kernel offload | DPDK, AF_XDP, container không có module |
| Hiệu năng | 5-10 Gbps mỗi NIC | 50+ Gbps khi pin DPDK PMD core |

**Cách kỹ sư vận hành thành thạo.** Một engineer thành thạo dpif cần phản xạ năm thao tác: (1) Nhận diện ngay provider đang dùng qua `ovs-vsctl get Bridge br0 datapath_type`; nếu trống thì mặc định `system`. Kiểm tra cross-reference với `ovs-appctl dpif/show` (dòng đầu sẽ là `system@ovs-system` hoặc `netdev@ovs-netdev`). (2) Đọc capability matrix qua `ovs-appctl dpif/show-dp-features <bridge>` trước khi push một OF rule mới có action lạ; nếu provider không hỗ trợ, xlate sẽ emulate (recirc nhiều) hoặc reject. (3) Phân biệt rõ dpif flow (in-kernel megaflow đã install) với OF flow (rule nguồn trong ofproto): `ovs-ofctl dump-flows br0` ≠ `ovs-appctl dpif/dump-flows br0`; cái sau là output của `dpif_flow_dump_*`. (4) Đọc upcall stat trong `ovs-appctl dpif/show -s` để biết `lost` (queue overflow), `hits`, `missed`; tỷ lệ `lost/missed > 1%` là báo động provider đang bị nghẽn. (5) Biết cách bật log debug riêng cho provider qua `ovs-appctl vlog/set dpif:file:dbg` và `vlog/set dpif_netlink:file:dbg` để trace từng flow_put/flow_del nếu nghi vấn corruption.

**Phân loại.** Có thể phân loại dpif theo nhiều trục: (a) **Theo type string**, `system` (kernel netlink), `netdev` (userspace netdev gồm cả DPDK), `dummy` (test in-memory, không expose ngoài unit test). (b) **Theo data path location**, kernel-resident hay userspace-resident. (c) **Theo transport**, Netlink (system) hay function call (netdev/dummy). (d) **Theo lifecycle owner**, kernel module (system tồn tại độc lập với ovs-vswitchd) hay process (netdev datapath chết khi ovs-vswitchd chết). (e) **Theo API method completeness**, full provider (đủ ct, meter, bond, cache) hay restricted provider (dummy chỉ implement subset đủ test). Theo importance: (a) > (b) > (c) > (d) > (e).

**Quy trình sử dụng.** Workflow điển hình khi engineer phải debug một mismatch giữa OF rule và hành vi thực tế trên bridge:

```
Bước 1: Xác định provider hiện tại.
        $ ovs-vsctl get Bridge br0 datapath_type
        → "system" (kernel) hoặc "netdev" (userspace) hoặc "" (default=system).
        $ ovs-appctl dpif/show
        → Dòng đầu tiên hiển thị type@name, ví dụ "system@ovs-system".

Bước 2: Đọc capability của provider hiện tại.
        $ ovs-appctl dpif/show-dp-features br0
        → Xem Recirc, CT state, CT zone, CT mark, NAT, Sample, Clone, Truncate, MPLS.
        → Nếu rule cần feature mà provider trả "No" → root cause tiềm năng.

Bước 3: So OF rule với dpif flow đã install.
        $ ovs-ofctl dump-flows br0          # OF rule (nguồn)
        $ ovs-appctl dpif/dump-flows br0    # megaflow đã install (đích)
        → Nếu OF rule có nhưng dpif không có flow tương ứng sau khi gửi packet,
          thì xlate hoặc dpif_flow_put bị lỗi (xem log).

Bước 4: Quan sát upcall flow.
        $ ovs-appctl dpif/show -s
        → hits / missed / lost. lost > 0 nghĩa upcall queue overflow.
        $ ovs-appctl coverage/show | grep -E "dpif|upcall"
        → flow_put count, recv count, miss count.

Bước 5: Bật debug log nếu nghi vấn provider corruption.
        $ ovs-appctl vlog/set dpif:file:dbg
        $ ovs-appctl vlog/set dpif_netlink:file:dbg   # nếu provider system
        → Tail file ovs-vswitchd.log, mỗi flow_put/flow_del hiện ra.
        → Tắt debug sau khi xong: ovs-appctl vlog/set dpif:file:warn.

Bước 6: Verify kernel module loaded (nếu provider system).
        $ lsmod | grep openvswitch
        → Nếu thiếu, dpif_netlink.open() đã trả về ENODEV và bridge không có data plane.
```

Best practice: không bao giờ thay đổi `datapath_type` của một bridge production đang chạy traffic; tạo bridge mới với type khác và migrate port. Anti-pattern: gọi trực tiếp `dpif-netlink-*` symbol trong patch code (vi phạm dependency inversion).

**Khi xảy ra sự cố.** `dpif` là keyword bắt buộc nhớ trong các tình huống: (a) Bridge "không forward gì cả" mặc dù OF rule đã cài đầy đủ, kiểm tra trước tiên `dpif/show` xem provider có thực sự open được không (lost queue, recv disabled). (b) `ovs-vswitchd` log spam `dpif|WARN|... could not add ...`, tức `dpif_flow_put` fail; thường do provider thiếu capability hoặc kernel module phiên bản cũ không hiểu một odp action mới. (c) Reload kernel module xong bridge mất hết flow, phải hiểu rằng provider system có lifecycle độc lập với ovs-vswitchd và `flow-restore-wait` cần được set khi restart. (d) DPDK deployment báo "datapath_type netdev not registered", thường do ovs-vswitchd build không có DPDK support nhưng OVSDB lại cấu hình netdev. (e) Mismatch capability sau khi nâng cấp OVS userspace mà không nâng kernel module cùng phiên bản, tính năng mới (ví dụ ct_label mở rộng) sẽ rớt về emulation hoặc reject.

**Liên quan mật thiết.** `dpif` đan xen với: (a) `ofproto-dpif xlate` (consumer phía trên gọi `dpif_flow_put`, đồng thời đọc capability để quyết định emulate, xem §9.32 §9.32.3). (b) `upcall` (phía dưới đẩy lên, dpif là transport, xem §9.2.4 và §9.2.10). (c) `megaflow` và `recirc_id` (entry mà dpif quản lý là megaflow có recirc_id ở đầu key, xem §9.2.2 và §9.2.14.2). (d) `openvswitch.ko` (provider system implement trên kernel module này, xem §9.2.14.3). (e) `ovs-dpctl`, `ovs-appctl dpif/*` (CLI bên ngoài cũng dùng dpif API qua control socket, xem §9.4 và §9.11). (f) `OVSDB Bridge.datapath_type` column (cấu hình chọn provider, xem §10.x). (g) `UFID` (Unique Flow ID 128-bit do dpif emit khi cài flow, xem §9.2.11).

**Khác biệt giữa các phiên bản.**

| Phiên bản OVS | Hành vi `dpif` |
|---------------|----------------|
| < 1.4 | Hard-coded linux netlink, chưa có tách dpif |
| 1.4-1.10 (2011-2013) | dpif interface stable, chỉ có `dpif-linux` |
| 2.0 (2014) | Rename `dpif-linux` → `dpif-netlink`, thêm `dpif-netdev` userspace |
| 2.4 (2015) | DPDK PMD thread integration trong dpif-netdev |
| 2.5 (2016) | Conntrack methods (`ct_dump_*`, `ct_flush`) thêm vào dpif_class |
| 2.7 (2017) | Meter methods thêm vào dpif_class |
| 2.10 (2018) | Cache levels (EMC/SMC) expose qua dpif feature |
| 2.13 (2020) | Bond methods thêm vào dpif_class cho HW bond offload |
| 2.17.9 (baseline curriculum) | dpif_class ổn định, ~30 method pointer, hỗ trợ AF_XDP qua netdev |
| 3.x | Capability negotiation refinement; `dpif/show-dp-features` mở rộng |

Lưu ý: với mỗi phiên bản, hai class concrete (`dpif_netlink_class`, `dpif_netdev_class`) có thể không implement đầy đủ method mới ngay; xlate đọc trả về của `dpif_class.method` (NULL hoặc EOPNOTSUPP) để biết và emulate.

**Source code tham chiếu.** Theo Rule 14 dùng function name anchor (line drift across version). Baseline OVS v2.17.9:

- `lib/dpif.h`, public client API. Các function chính: `dpif_open()`, `dpif_close()`, `dpif_port_add()`, `dpif_port_del()`, `dpif_flow_put()`, `dpif_flow_del()`, `dpif_flow_dump_create()`, `dpif_flow_dump_next()`, `dpif_execute()`, `dpif_recv()`, `dpif_recv_wait()`. Block comment đầu file mô tả rõ "A datapath is a collection of physical or virtual ports... a datapath has no intelligence of its own. Rather, it relies entirely on its client to set up flows."
- `lib/dpif-provider.h`, định nghĩa `struct dpif_class` (vtable provider implement) và `struct dpif` (instance handle). Hai class extern declare cuối file: `dpif_netlink_class`, `dpif_netdev_class`.
- `lib/dpif.c`, dispatch wrapper. Function `dpif_open()` lookup type qua `do_open()`, gọi `class->open()`. Function `dp_register_provider()` cho plugin add type vào registry. Function `dpif_normalize_type()` mặc định trống → `system`.
- `lib/dpif-netlink.c`, kernel provider. Function `dpif_netlink_open()` send `OVS_DP_CMD_NEW` Netlink. Function `dpif_netlink_flow_put()` send `OVS_FLOW_CMD_NEW`. Function `dpif_netlink_recv()` dequeue `OVS_PACKET_CMD_MISS`.
- `lib/dpif-netdev.c`, userspace provider. Function `dp_netdev_open()` allocate in-process struct dp_netdev. Function `dp_netdev_flow_add()` insert vào in-memory classifier. Function `dpif_netdev_recv()` dequeue từ PMD thread queue.
- `ofproto/ofproto-dpif-upcall.c`, consumer phía trên. Function `udpif_upcall_handler()` loop gọi `dpif_recv()` rồi xlate.
- `ofproto/ofproto-dpif.c`, integration. Function `ofproto_dpif_open()` call `dpif_open()` khi bridge khởi tạo; `dpif_destroy()` khi bridge xoá.

**Trường hợp sự cố thực tế.** Một deployment OpenStack Neutron multi-host (ước tính 40 compute) báo cáo: sau khi nâng cấp ovs-vswitchd userspace từ 2.13 lên 2.17 mà QUÊN nâng kernel module openvswitch.ko (vẫn chạy 2.13 builtin của distro), một số bridge xuất hiện hành vi lạ: OF rule có `actions=ct(commit,nat(src=10.0.0.5))` nhưng packet đi qua không bị NAT. Triệu chứng dpif: `ovs-appctl dpif/show-dp-features br-int` trả `CT state NAT: No`. Investigation: log có `dpif_netlink|WARN|sending packet failed (Operation not supported)`; capability matrix mismatch giữa userspace expectation và kernel actual. Root cause: kernel module 2.13 không hỗ trợ option NAT trong OVS_CT_ATTR mới mà userspace 2.17 emit. Khi `dpif_netlink_flow_put()` gửi flow xuống, kernel reject, ovs-vswitchd log warning và rule âm thầm không có hiệu lực. Mitigation: rebuild và load kernel module từ cùng cây nguồn 2.17 (`apt install openvswitch-datapath-dkms`), reload module, bridge tự khôi phục megaflow. Lesson: khi nâng cấp OVS phải đồng bộ userspace + kernel module; capability matrix qua dpif là công cụ phát hiện mismatch nhanh nhất, dùng trước khi đổ lỗi cho controller hoặc OF rule logic.

**Bài tập synthetic.**

```
Lab D1, "Hai provider cùng workload, đo capability + flow install"
       (~45 phút, cần một host có cả kernel module và userspace netdev sẵn sàng):

Topology dự kiến:
  - Host: Ubuntu 22.04 LTS với OVS 2.17.9.
  - Bridge A: br-sys, datapath_type mặc định (system, kernel).

Bước 1: Tạo hai bridge với hai provider type.

        $ ovs-vsctl add-br br-userspace -- set Bridge br-userspace datapath_type=netdev

        $ ip netns exec h1 ping -c 3 10.0.1.2          # qua br-userspace

                 có thể khác mask granularity do classifier internal khác nhau.
        Explain: dpif_class khác nhau ⇒ flow_put dispatch tới hai code path
                 khác nhau ⇒ dù cùng input vẫn có thể có internal layout khác.
```

**Lỗi thường gặp + tín hiệu chẩn đoán.**

- *Triệu chứng* Bridge tạo ra nhưng `ovs-appctl dpif/show` không liệt kê. *Nguyên nhân* Provider open fail (kernel module thiếu, hoặc type sai chính tả). *Chẩn đoán* Tail `ovs-vswitchd.log`, tìm dòng `dpif|ERR|could not create datapath`. *Khắc phục* `lsmod | grep openvswitch`, load lại module hoặc sửa `datapath_type`.
- *Triệu chứng* `ovs-appctl dpif/show -s` báo `lost: <số tăng>`. *Nguyên nhân* Upcall queue overflow, handler thread không kịp dequeue. *Chẩn đoán* `ovs-appctl coverage/show | grep upcall`, `dpif/show -s` so sánh hits vs lost. *Khắc phục* Tăng `n-handler-threads` qua `ovs-vsctl set Open_vSwitch . other_config:n-handler-threads=8`.
- *Triệu chứng* OF rule cài thành công (`ovs-ofctl dump-flows` thấy) nhưng packet không match khi đi qua. *Nguyên nhân* `dpif_flow_put` bị provider reject (capability mismatch, action không hỗ trợ). *Chẩn đoán* `ovs-appctl vlog/set dpif:file:dbg`, tail log tìm warn message khi packet đầu tiên đi qua. *Khắc phục* Đối chiếu capability với `dpif/show-dp-features`, hoặc nâng cấp provider (kernel module hoặc userspace nếu netdev) đồng bộ phiên bản.
- *Triệu chứng* Sau reload kernel module, bridge mất hết megaflow, traffic gián đoạn ~5-30 giây. *Nguyên nhân* Provider system có lifecycle độc lập với ovs-vswitchd; reload module = reset datapath. *Chẩn đoán* `dmesg | tail` thấy module unload/load sequence. *Khắc phục* Trước reload, set `ovs-vsctl set Open_vSwitch . other_config:flow-restore-wait=true`, sau khi ovs-vswitchd reconnect xong unset; trong production tránh reload module khi có traffic nóng.
- *Triệu chứng* DPDK port bị "could not add port: No such device". *Nguyên nhân* Bridge `datapath_type=system` nhưng port type DPDK chỉ tồn tại trong `dpif-netdev`. *Chẩn đoán* `ovs-vsctl get Bridge br0 datapath_type`. *Khắc phục* Tạo bridge mới với `datapath_type=netdev` và migrate DPDK port sang.

**So sánh với hệ khác.** Các hệ khác giải bài "tách interface với implementation cho data plane" theo cách khác:

- **Linux VFS** (`struct file_operations`): tương đương rất sát. Mỗi filesystem (ext4, xfs, fat) implement vtable, kernel client (`vfs_read`, `vfs_write`) gọi qua interface. dpif_class chính là VFS file_operations cho data plane switch.
- **Linux netfilter `struct nf_hook_ops`**: gần giống về vtable nhưng hướng tới hook chain, không tới data plane abstraction; netfilter không tách "kernel netfilter" và "userspace netfilter" như dpif tách kernel/userspace datapath.
- **OpenStack Neutron ML2 mechanism driver**: tương đương ở tầng cao hơn; mỗi mechanism driver (ovs, linuxbridge, sr-iov) implement một interface, controller dispatch. Khác: ML2 ở control plane ngày, dpif ở userspace data plane.
- **Cisco IOS Forwarding Information Base abstraction**: tách CEF (Cisco Express Forwarding) kernel-style với hardware ASIC FIB. Tương đương về nguyên tắc abstraction, khác về scope (Cisco là proprietary firmware).
- **VPP plugin model** (FD.io): VPP có `vlib_node` graph với plugin. Khác: VPP plugin định hình packet processing graph, dpif chỉ định hình flow table interface.
- **Linux tc `cls_*` classifier**: KHÔNG tương đương vì tc classifier không có "dispatch by type", mọi classifier đều cùng một in-kernel plugin model. dpif tách hẳn userspace với data plane backend.

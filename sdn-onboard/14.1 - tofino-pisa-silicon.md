### ▶ Guided Exercise 1: Analyze P4 program compile output cho Tofino target

> **Trạng thái lab verification:** Lab này require Intel Tofino SDE license hoặc hardware Tofino switch. Sau Intel EOL 01/2023, access khó khăn. Alternative approach: dùng BMv2 + compiler resource report để tiếp cận ý tưởng analyze pipeline stage allocation.

**Mục đích:** Hiểu cách p4c-tofino compiler allocate resources (pipeline stages, TCAM entries, SRAM blocks) cho một P4 program cụ thể. Nhận biết bottleneck stage — khi một table không fit trong stage hiện tại, compiler phải tách hoặc spill sang stage sau, ảnh hưởng throughput.

**Chuẩn bị:**

Option A (with Tofino SDE, legacy):
- Intel SDE 9.13+ installed (license required, not public post-EOL)
- Tofino switch hardware (Edgecore Wedge 100BF-32X, Inventec D10056, hoặc Tofino reference board)
- SDE environment variables: `SDE=/path/to/bf-sde-9.13.0`, `SDE_INSTALL=$SDE/install`

Option B (BMv2 alternative, accessible):
- Hoàn thành Lab 14.0-Ex1 baseline
- p4c installed với verbose flag support

**Mô hình lab:**

Tofino hardware có kiến trúc pipeline cố định nhưng programmable:

```
┌──────────────────────────────────────────────────────────────┐
│              Tofino 1 ASIC (3.3 Tbps, 2016)                  │
│                                                              │
│  Ingress pipeline (12 stages):                               │
│                                                              │
│  [Parser] → [S0] → [S1] → [S2] → ... → [S11] → [Deparser]    │
│                                                              │
│  Each stage:                                                 │
│  ┌─────────────────────────────────────────┐                 │
│  │  TCAM blocks (~48 entries × 128 bit)    │                 │
│  │  SRAM blocks (~80 × 1024 entries)       │                 │
│  │  ALU (arithmetic, bit manipulation)     │                 │
│  │  Register access (32-bit counters)      │                 │
│  └─────────────────────────────────────────┘                 │
│                                                              │
│  Egress pipeline: similar structure, 12 stages               │
└──────────────────────────────────────────────────────────────┘
```

**Bước thực hiện (Option A, Tofino hardware):**

**1.** Set up SDE environment:

```bash
[host]# source $SDE/set_sde.sh
[host]# echo $SDE_INSTALL
/path/to/bf-sde-9.13.0/install
```

**2.** Compile một sample program với resource report:

```bash
[host]# cd $SDE/pkgsrc/p4-examples/p4_16_programs/tna_l2_forwarding
[host]# p4_build.sh -C tna_l2_forwarding.p4 --verbose
```

Output kỳ vọng cho resource allocation:

```
Table allocation report:
  MyIngress.l2_fwd: Stage 2-3
    Match type: exact (256 entries)
    TCAM: 0 blocks
    SRAM: 2 blocks (32KB)
    ALU: 1 arithmetic op

Total stages used: 4 / 12
TCAM utilization: 0%
SRAM utilization: 5.2%
```

**3.** Inspect generated binary + context:

```bash
[host]# ls tna_l2_forwarding.out/pipe/
tofino.bin    context.json    manifest.json

[host]# jq '.tables[] | {name, stage}' tna_l2_forwarding.out/pipe/context.json
```

**Bước thực hiện (Option B, BMv2 alternative):**

**1.** Compile với p4c verbose flag:

```bash
[host]# p4c --target bmv2 --arch v1model --verbose 2 basic.p4 2>&1 | tee compile.log
```

**2.** Analyze log cho table→stage mapping concept:

```bash
[host]# grep -E 'Table|stage' compile.log
```

BMv2 không có stage allocation thực sự (software pipeline), nhưng log cho biết compilation phases.

**Output mong muốn (conceptual):**

Với Tofino, nhìn thấy cụ thể:
- Number of stages used vs. total 12 available
- Per-stage TCAM + SRAM usage
- Bottleneck identification (warning nếu table phải span multiple stages)

**Bài học học được:**

Thứ nhất, Tofino's 12 ingress stages + 12 egress stages tạo ra hard constraint — một P4 program phức tạp có thể không fit trong 12 stages và compile fail. Production P4 development require compile-iterate-optimize cycle để match resource budget.

Thứ hai, TCAM expensive (limited blocks) vs SRAM cheap (more blocks). Exact match → SRAM (hash lookup). Ternary/wildcard match → TCAM. Good P4 design minimize TCAM usage khi possible.

Thứ ba, sau Intel EOL 2023, P4 programmable hardware ecosystem ở trạng thái uncertain. AMD Pensando DPU và NVIDIA BlueField DOCA Pipelines là alternative có programmable element, nhưng semantics khác P4 thuần. Học P4 trên BMv2 + eBPF backend vẫn giá trị về concept, dù production silicon path narrow.

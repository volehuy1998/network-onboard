## Mục tiêu bài học

Sau khi hoàn thành phần này, bạn có thể:

1. **Mô tả (Understand)** vai trò của IDL (Interface Definition Language) client — lớp trừu tượng cho phép application đăng ký thay đổi OVSDB mà không phải tự hiện thực polling hay JSON-RPC handshake.
2. **Phân tích (Analyze)** cơ chế `monitor` và `monitor_cond` — sự khác biệt giữa đăng ký toàn bộ table và đăng ký theo condition, tại sao monitor_cond là nền tảng cho OVN mở rộng lớn.
3. **Áp dụng (Apply)** viết một IDL client Python đơn giản dùng `python-ovs` để đăng ký bảng Port_Binding của OVN SB, in ra thay đổi thời gian thực.

Kiến thức tiên quyết: Phần 10.0 RFC 7047, Phần 10.3 ACID semantics (cho pattern retry), Python cơ bản.

## 10.4.1 Tại sao cần IDL thay vì gọi JSON-RPC trực tiếp?

Nếu một application cần theo dõi thay đổi OVSDB, cách thô sơ là polling: mỗi giây gọi `ovsdb-client dump` để lấy toàn bộ database rồi so sánh với snapshot trước. Cách này có ba vấn đề nghiêm trọng:

**(1) Băng thông lãng phí:** OVN SB DB có thể có hàng trăm nghìn Logical_Flow record (vài chục MB). Polling mỗi giây = tải vài chục MB/s chỉ để phát hiện 1-2 row thay đổi.

**(2) Độ trễ cao:** thay đổi phát sinh ở T1, client phát hiện ở T1+1s (poll interval). Với OVN cần đáp ứng dưới giây, độ trễ này không chấp nhận được.

**(3) Race condition:** giữa hai poll, thay đổi có thể biến mất (insert rồi delete), client không thấy. Dữ liệu bất nhất.

Cách đúng là **đăng ký thay đổi** — server chủ động đẩy thay đổi về client qua kết nối dài hạn. OVSDB JSON-RPC có method `monitor` và `monitor_cond` để làm điều này. Nhưng hiện thực trực tiếp phức tạp:
- Parse JSON-RPC 2.0 notification format.
- Duy trì bản sao trạng thái cục bộ (server chỉ gửi delta, client phải theo dõi trạng thái tuyệt đối).
- Xử lý kết nối lại + đồng bộ lại nếu mất kết nối.
- Xử lý đồng bộ ban đầu (server gửi toàn bộ trạng thái table ngay lần đăng ký đầu).
- Race giữa transaction ghi và monitor callback.

**Thư viện IDL (Interface Definition Language)** xử lý tất cả những phức tạp này. Code application chỉ cần:
```python
idl = Idl("tcp:127.0.0.1:6642", SchemaHelper("ovs-sb.ovsschema").get_idl_schema())
while True:
    idl.run()  # xử lý các message đang chờ
    if idl.change_seqno changed:
        # trạng thái database đã cập nhật, truy cập qua idl.tables
        for row in idl.tables["Port_Binding"].rows.values():
            print(row.logical_port, row.chassis)
```

IDL tự động: kết nối, đăng ký monitor_cond, duy trì bản sao cục bộ của các table đăng ký, thông báo khi có thay đổi, tự động kết nối lại + đồng bộ lại khi mất kết nối.

## 10.4.2 Protocol monitor so với monitor_cond

OVSDB có hai protocol đăng ký khác biệt quan trọng:

**`monitor` (RFC 7047 nguyên bản, OVS 1.x+):** client đăng ký toàn bộ một table (hoặc tập con column). Server đẩy:
- **Trạng thái ban đầu** ngay khi đăng ký — toàn bộ row hiện có của table.
- **Cập nhật tăng dần** mỗi lần có thay đổi — insert/modify/delete row đơn.

Lợi thế: đơn giản, xác định. Nhược điểm: không lọc — nếu table có 100K row mà client chỉ quan tâm 100, vẫn phải nhận toàn bộ 100K.

**`monitor_cond` (OVS 2.6+, 2017):** mở rộng `monitor` với **mệnh đề condition** — server chỉ đẩy row thỏa condition. Client có thể thay đổi condition thời gian chạy qua `monitor_cond_change` mà không cần đăng ký lại.

Format request monitor_cond:
```json
{
  "method": "monitor_cond",
  "params": [
    "OVN_Southbound",
    ["monid_1"],
    {
      "Port_Binding": [
        {
          "columns": ["logical_port", "chassis", "mac"],
          "where": [["chassis", "==", ["named-uuid", "my_chassis_uuid"]]]
        }
      ]
    }
  ],
  "id": 1
}
```

Server chỉ đẩy row Port_Binding có `chassis` match `my_chassis_uuid`. Các row khác không liên quan không gửi → băng thông giảm 10-100 lần với cụm lớn.

**Ví dụ tính quan trọng:** cụm OVN 1000 chassis × 100 pod/chassis = 100K Port_Binding. Mỗi ovn-controller trên chassis X chỉ cần đăng ký Port_Binding của chính nó (khoảng 100 row). Không có monitor_cond, mỗi chassis phải nhận 100K row trạng thái + cập nhật → hàng MB/s traffic OVSDB không cần thiết. Monitor_cond giảm xuống 100 row → KB/s.

**monitor_cond_since (OVS 2.12+):** thêm tham số `last-txn-id` cho resume đăng ký sau khi kết nối lại. Server biết "client đã có trạng thái đến transaction X" → chỉ đẩy delta từ X. Tiết kiệm băng thông khi kết nối lại ngắn.

## 10.4.3 Kiến trúc thư viện IDL client

Python-ovs IDL (thư viện chính thức từ dự án OVS) có kiến trúc:

**Thành phần 1 — SchemaHelper:** đọc file schema (`ovs-sb.ovsschema`), build class Python cho mỗi table. Application import class schema rồi truy cập field như thuộc tính:
```python
from ovs.db.schema import SchemaHelper
schema_helper = SchemaHelper("/usr/share/ovn/ovn-sb.ovsschema")
schema_helper.register_table("Port_Binding")
schema_helper.register_columns("Port_Binding", ["logical_port", "chassis"])
schema = schema_helper.get_idl_schema()
```

**Thành phần 2 — Idl:** class chính quản lý kết nối + đăng ký:
```python
from ovs.db.idl import Idl
idl = Idl("tcp:127.0.0.1:6642", schema)
```
Constructor tự động kết nối tới ovsdb-server và gửi đăng ký monitor_cond cho các table+column đã register.

**Thành phần 3 — Row objects:** IDL duy trì bản sao cục bộ dưới dạng `idl.tables[table_name].rows` (dict UUID → Row object). Truy cập row:
```python
for uuid, row in idl.tables["Port_Binding"].rows.items():
    print(f"Port {row.logical_port} on chassis {row.chassis[0].name if row.chassis else 'unbound'}")
```

**Thành phần 4 — Thông báo thay đổi:** IDL có bộ đếm `change_seqno` tăng mỗi lần có thay đổi trạng thái (insert/modify/delete). Application poll bộ đếm:
```python
last_seqno = idl.change_seqno
while True:
    idl.run()  # xử lý message
    if idl.change_seqno != last_seqno:
        # đã có thay đổi, xử lý trạng thái mới
        handle_changes(idl)
        last_seqno = idl.change_seqno
    time.sleep(0.1)
```

Hoặc dùng `idl.wait()` (block cho đến khi có thay đổi):
```python
poller = Poller()
idl.wait(poller)
poller.block()
```

**Thành phần 5 — Trình build transaction:** IDL cung cấp class `Transaction` để ghi thay đổi ngược lại OVSDB, tự động xử lý logic retry (như đã trình bày Phần 10.3):
```python
from ovs.db.idl import Transaction
txn = Transaction(idl)
port = idl.tables["Port_Binding"].rows[some_uuid]
port.chassis = [new_chassis_row]
status = txn.commit_block()
```

## 10.4.4 Sao chép có điều kiện — thay đổi condition thời gian chạy

Khi ovn-controller khởi động trên chassis X, nó đăng ký Port_Binding với `where chassis == X`. Nhưng nếu sau đó X được giao quản lý thêm một số Port_Binding khác (ví dụ cho pod mới schedule), condition cần thay đổi mà không kết nối lại.

Message `monitor_cond_change` cho phép cập nhật condition thời gian chạy:
```json
{
  "method": "monitor_cond_change",
  "params": [
    ["monid_1"],
    {
      "Port_Binding": [
        {
          "where": [["chassis", "==", ["named-uuid", "chassis_X"]],
                    ["type", "==", "patch"]]
        }
      ]
    }
  ],
  "id": 2
}
```

Server so sánh condition mới với cũ:
- Row match condition mới nhưng không match cũ → đẩy dưới dạng "insert" (dù row đã tồn tại lâu).
- Row match cũ nhưng không match mới → đẩy dưới dạng "delete".
- Row match cả hai → không đẩy (client đã có trạng thái).

Đây là cơ chế cho phép ovn-controller "mở rộng" hoặc "thu hẹp" đăng ký mà không cần tear down + build lại kết nối.

**API Python-ovs:**
```python
from ovs.db.idl import Idl, ConditionRegistry
idl = Idl(...)
idl.cond_change("Port_Binding", [["chassis", "==", ["named-uuid", new_chassis_uuid]]])
```

OVN 22.03 dùng cond_change rộng rãi — khi pod được schedule tới chassis X, ovnkube-master ghi Port_Binding.chassis = X vào SB DB, ovn-controller trên X nhận notification và tự động bắt đầu xử lý Logical Luồng cho pod đó.

## 10.4.5 Xử lý mất kết nối + đồng bộ lại

Kết nối monitor dài hạn có thể mất (trục trặc mạng, ovsdb-server restart). IDL client phải:
1. Phát hiện mất kết nối (TCP connection close, heartbeat timeout).
2. Kết nối lại tới server.
3. Đăng ký lại monitor_cond.
4. Đồng bộ lại trạng thái cục bộ.

Có hai chiến lược đồng bộ lại:

**Đồng bộ lại toàn bộ (cổ điển):** sau khi kết nối lại, server đẩy lại toàn bộ trạng thái thỏa condition. Client clear bản sao cục bộ rồi điền lại từ đầu. Đảm bảo nhất quán nhưng tốn băng thông với cụm lớn.

**Đồng bộ lại tăng dần (monitor_cond_since, OVS 2.12+):** client nhớ `last_txn_id` trước khi mất kết nối. Sau khi kết nối lại gửi `monitor_cond_since` với last_txn_id. Server chỉ đẩy thay đổi từ txn_id đó trở đi. Tiết kiệm băng thông đáng kể nếu mất kết nối ngắn.

**Python-ovs tự động xử lý:** code application không phải viết logic kết nối lại:
```python
idl = Idl("tcp:127.0.0.1:6642", schema)
while True:
    idl.run()  # tự động kết nối lại nếu mất kết nối
    time.sleep(1)
```

IDL cung cấp `idl.is_lock_contended()` và `idl.change_seqno` cho application phát hiện kết nối lại (seqno reset về 0 sau đồng bộ lại toàn bộ).

## 10.4.6 Cân nhắc hiệu năng cho IDL client ở quy mô lớn

Application IDL client trong môi trường production OVN cần tinh chỉnh để tránh điểm nghẽn:

**(1) Đăng ký column tối thiểu:** đăng ký chỉ column thực sự dùng — giảm băng thông và dấu chân memory bản sao cục bộ.
```python
schema_helper.register_columns("Port_Binding", ["logical_port", "chassis"])
# không register các column không dùng như "options", "external_ids"
```

**(2) Condition chọn lọc:** dùng mệnh đề where hẹp nhất có thể:
```python
# Tốt — chỉ đăng ký Port_Binding của chassis cục bộ
cond = [["chassis", "==", ["named-uuid", local_chassis_uuid]]]

# Xấu — đăng ký toàn bộ Port_Binding
cond = []
```

**(3) Xử lý theo lô:** không xử lý từng thay đổi một, thu gom thay đổi trong một lần idl.run() rồi áp theo lô:
```python
idl.run()
if idl.change_seqno != last_seqno:
    # thu gom tất cả thay đổi kể từ last_seqno
    changes = collect_all_changes(idl, last_seqno)
    apply_batch(changes)  # thao tác theo lô tối ưu
    last_seqno = idl.change_seqno
```

**(4) Gom kết nối:** với nhiều ovsdb-server (cụm Raft), client có thể kết nối nhiều server — một làm primary, các khác làm failover. Constructor `Idl` của python-ovs chấp nhận danh sách remote URL:
```python
idl = Idl("tcp:10.0.0.1:6642,tcp:10.0.0.2:6642,tcp:10.0.0.3:6642", schema)
```
Tự động thử lần lượt nếu server đầu tiên không tiếp cận được.

**(5) Kích thước bản sao memory:** với cụm lớn, bản sao cục bộ có thể chiếm nhiều GB RAM. Giám sát qua `top -p <pid>` và tinh chỉnh `column/condition` nếu thấy memory cao. Một node OVN-Kubernetes có thể tiêu thụ 2-5 GB cho bản sao IDL của SB DB với cụm 500+ chassis.

### ▶ Guided Exercise 1: Viết IDL client Python theo dõi Port_Binding

> **Trạng thái verification lab:** Output dựa trên python-ovs docs; chờ kiểm chứng lab thực.

**Mục đích:** Thực hành kiến trúc IDL qua một client đơn giản — đăng ký Port_Binding, in ra thay đổi thời gian thực.

**Chuẩn bị:** Node Ubuntu 22.04 với OVS/OVN 22.03 đã chạy, package python-ovs cài đặt:
```
[host]# sudo apt install python3-ovs
[host]# sudo apt install -y openvswitch-common  # cho file schema
```

**Bước thực hiện:**

**1.** Tạo file `ovn_watch.py`:
```python
#!/usr/bin/env python3
import sys, time
from ovs.db.idl import Idl
from ovs.db.schema import SchemaHelper

SCHEMA = "/usr/share/ovn/ovn-sb.ovsschema"
REMOTE = "tcp:127.0.0.1:6642"

def main():
    helper = SchemaHelper(SCHEMA)
    helper.register_table("Port_Binding")
    helper.register_columns("Port_Binding", ["logical_port", "chassis", "type"])
    helper.register_table("Chassis")
    helper.register_columns("Chassis", ["name"])

    idl = Idl(REMOTE, helper.get_idl_schema())
    print(f"Connected to {REMOTE}, subscribed Port_Binding + Chassis")

    last_seqno = 0
    while True:
        idl.run()
        if idl.change_seqno != last_seqno:
            print(f"\n=== Change detected (seqno {idl.change_seqno}) ===")
            for uuid, pb in idl.tables["Port_Binding"].rows.items():
                chassis_name = pb.chassis[0].name if pb.chassis else "unbound"
                print(f"  Port {pb.logical_port:20s} type={pb.type:12s} chassis={chassis_name}")
            last_seqno = idl.change_seqno
        time.sleep(0.5)

if __name__ == "__main__":
    main()
```

**2.** Chạy client:
```
[host]# python3 ovn_watch.py
Connected to tcp:127.0.0.1:6642, subscribed Port_Binding + Chassis
=== Change detected (seqno 1) ===
  Port vm1                  type=              chassis=chassis_1
  Port vm2                  type=              chassis=chassis_2
```

**3.** Trong terminal khác, tạo Logical Switch + Port mới:
```
[host-B]# ovn-nbctl lsp-add ls1 vm3
[host-B]# ovn-nbctl lsp-set-addresses vm3 "aa:bb:cc:dd:ee:03 10.0.0.3"
```

**4.** Quan sát terminal chạy `ovn_watch.py`:
```
=== Change detected (seqno 2) ===
  Port vm1                  type=              chassis=chassis_1
  Port vm2                  type=              chassis=chassis_2
  Port vm3                  type=              chassis=unbound
```

Port mới xuất hiện ngay lập tức — IDL đã nhận notification monitor_cond từ ovsdb-server.

**5.** Bind port tới chassis:
```
[host-B]# ovn-sbctl chassis-add my_chassis geneve 192.168.1.10
[host-B]# ovn-sbctl lsp-bind vm3 my_chassis
```

Quan sát terminal theo dõi:
```
=== Change detected (seqno 3) ===
  Port vm3                  type=              chassis=my_chassis
```

**Bài học học được:**

Thứ nhất, IDL trừu tượng hoàn toàn phức tạp của monitor_cond + kết nối lại — code application chỉ 30 dòng Python vẫn có đầy đủ đăng ký thời gian thực.

Thứ hai, `change_seqno` là cơ chế đơn giản phát hiện thay đổi. Application poll bộ đếm, không cần callback hay vòng lặp sự kiện phức tạp.

Thứ ba, với workload môi trường production quy mô lớn, thêm mệnh đề `where` vào register_columns để lọc đăng ký — tiết kiệm băng thông + memory đáng kể.

**Cleanup:**
```
[host]# Ctrl+C để dừng client
```

### ▶ Guided Exercise 2: Thử nghiệm cond_change thời gian chạy

> **Trạng thái verification lab:** chờ kiểm chứng.

**Mục đích:** Quan sát hành vi `monitor_cond_change` — thay đổi condition giữa session, server đẩy delta tương ứng.

**Chuẩn bị:** Như Exercise 1.

**Bước thực hiện:**

**1.** Chỉnh `ovn_watch.py` để dùng `cond_change` thay đổi condition mỗi 30 giây:

```python
import time
idl = Idl(REMOTE, helper.get_idl_schema())
idl.cond_change("Port_Binding", [["type", "==", "patch"]])
time.sleep(30)
print("Changing condition to type=vif")
idl.cond_change("Port_Binding", [["type", "==", "vif"]])
```

**2.** Chạy và quan sát: lần đầu chỉ thấy patch port, sau 30s chuyển sang chỉ thấy vif port.

**Bài học học được:**

Thứ nhất, `cond_change` cho phép application động phạm vi đăng ký mà không kết nối lại — cực kỳ quan trọng cho ovn-controller khi chassis được giao hoặc bỏ Port_Binding động.

Thứ hai, server tự động so sánh condition: row match cũ/không match mới → đẩy "delete" một cách nhân tạo (dù row vẫn tồn tại trên server). Client phải hiểu điều này khi xử lý notification.

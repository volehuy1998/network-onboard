## Bối cảnh: bài toán kernel datapath phải giải

Giả sử một host KVM chạy 50 VM, mỗi VM có 20 TCP connection đồng thời. Tổng cộng 1000 TCP microflow. Mỗi microflow gồm tổ hợp (src_ip, dst_ip, src_port, dst_port, protocol) — 5-tuple. Kernel datapath phải quyết định action chuyển tiếp cho mỗi gói tin với tốc độ triệu gói tin/second.

Cách tiếp cận naive — userspace daemon quyết định mọi gói tin — không khả thi vì context switch kernel↔userspace tốn ~2 microsecond, giới hạn băng thông thực tế ở ~500K gói tin/second.

Cách tiếp cận ngược lại — toàn bộ OpenFlow pipeline trong kernel — không khả thi vì pipeline phức tạp với classifier, prioritized rules, arbitrary header modification. Kernel code không nên chứa logic như vậy.

Giải pháp OVS: **kernel duy trì cache cho quyết định đã tính**, userspace tính quyết định khi cache miss. Cache phải nhanh (nanosecond scale), chính xác (không sai policy), và hiệu quả về bộ nhớ.

## 9.2.1 Microflow cache: thiết kế OVS 1.x và giới hạn

OVS 1.x (2011-2013) dùng **microflow cache**: mỗi entry mô tả chính xác một 5-tuple (exact match), không wildcard. Lookup O(1) qua hash table.

Ưu điểm: cực nhanh khi hit. Lookup chỉ cần compute hash + compare full key.

Nhược điểm fatal: **mỗi microflow khác biệt về bất kỳ field nào đều yêu cầu entry riêng**. Ví dụ:

- TCP connection 1: src=10.0.0.1:50001 → dst=10.0.0.2:80 → entry A
- TCP connection 2: src=10.0.0.1:50002 → dst=10.0.0.2:80 → entry B (khác src_port)
- TCP connection 3: src=10.0.0.1:50003 → dst=10.0.0.2:80 → entry C

Mặc dù cả ba đi qua cùng OpenFlow rule ("allow TCP dst=10.0.0.2:80") và cùng action (output:5), mỗi connection có entry cache riêng chỉ khác ở src_port. Với 1000 connection, cache có 1000 entry.

Kết quả: cache thrashing. Cache có kích thước cố định, khi số microflow vượt quá size, entry cũ bị evict. Gói tin evict gặp cache miss → upcall → reinstall → vòng luẩn quẩn.

NSDI 2015 đo trên workload Rackspace production (2013, trước megaflow): miss rate 10-30% trong peak. Chi phí phụ quá lớn để scale.

## 9.2.2 Megaflow cache: đột phá của NSDI 2015

**Megaflow** là cache entry với **wildcard** — match một tập microflow có cùng mẫu header. Thay vì 1000 entries cho 1000 connection TCP đến port 80, chỉ một entry duy nhất `dst=10.0.0.2, dst_port=80, proto=TCP, wildcard src_ip+src_port` cover tất cả.

Compression factor điển hình 10-100 lần. Workload 100K microflow → cache 2.000-10.000 entry.

**Câu hỏi then chốt**: làm sao quyết định field nào wildcard? OVS giải quyết bằng **lazy wildcarding**. Khi userspace classifier xử lý microflow lần đầu:

1. Parse gói tin thành 5-tuple + field khác.
2. Walk OpenFlow pipeline: match rules table 0 → table 1 → ..., execute action.
3. Trong quá trình walk, **track** field nào đã được kiểm tra.
4. Field KHÔNG được kiểm tra → wildcard.
5. Compile megaflow entry với wildcard cho field không kiểm tra.

Ví dụ: rule "match dst_ip=10.0.0.2, dst_port=80, proto=TCP → output:5". Gói tin với src_ip=10.0.0.1, src_port=50001:

- Classifier kiểm tra dst_ip (match), dst_port (match), proto (match).
- src_ip và src_port KHÔNG kiểm tra.

Megaflow entry: match `dst_ip=10.0.0.2, dst_port=80, proto=TCP`, wildcard `src_ip=*, src_port=*`.

**Correctness guarantee**: lazy wildcarding không phạm rule nào. Nếu có rule khác ràng buộc src_port, classifier sẽ kiểm tra khi walk, không wildcard. Correct by construction.

## 9.2.3 Tuple Space Search cho megaflow lookup

Khi có nhiều megaflow với mask khác nhau, mỗi megaflow có **tuple** — tập field được match (không wildcard). OVS dùng **Tuple Space Search (TSS)** — thuật toán từ Srinivasan et al. SIGCOMM 1998: nhóm entry theo tuple, mỗi tuple có hash table riêng.

Lookup gói tin: với mỗi tuple, extract field theo mask, compute hash, lookup hash table. Nếu match, return action. Nếu không, try tuple tiếp theo.

Độ phức tạp: O(T × H) với T = số tuple (thường < 20), H = chi phí hash lookup (O(1) trung bình).

> **Chủ đề trọng tâm:** Lựa chọn TSS thay vì TCAM hay decision tree là quyết định thực dụng. TCAM quá đắt trên commodity server. Decision tree build cost cao khi rules thay đổi thường xuyên. TSS là compromise tối ưu cho software datapath. NSDI 2015 benchmark: TSS đạt 10M+ lookup/second per core trên Intel Xeon.

## 9.2.4 Cơ chế gọi ngược: slow path và fast path

USC Lab 9 (`doc/ovs/Day 4-lab 9 - Open vSwitch Kernel Datapath.txt`) trình bày rõ:

> "Kernel module directs the gói tin to the userspace. The userspace makes the decisions about the actions to be taken against the gói tin according to OpenFlow entries (slow path)."
>
> "The action entry is stored in the kernel and used to forward subsequent gói tin which makes the chuyển tiếp faster (fast path)."

**Fast path (cache hit)**:

1. Gói tin đến NIC → kernel stack → `openvswitch.ko` rx hook.
2. Kernel extract flow key từ headers.
3. Kernel lookup megaflow cache qua TSS.
4. Match → execute action (output:5 → `netdev_ops->ndo_start_xmit`).
5. Update counter. Không chạm userspace. Độ trễ ~1-5 microsecond.

**Slow path (cache miss)**:

1. Gói tin đến, lookup miss.
2. Kernel gửi **upcall** qua Netlink lên userspace.
3. Cơ chế gọi ngược chứa gói tin bytes, metadata (in_port, timestamp), reason.
4. `ovs-vswitchd` handler thread nhận upcall.
5. Handler walk OpenFlow pipeline, track inspected fields.
6. Compile megaflow entry + action list.
7. `OVS_FLOW_CMD_NEW` về kernel — cài megaflow.
8. `OVS_PACKET_CMD_EXECUTE` để kernel execute action cho gói tin này.
9. Gói tin tương tự sau đó hit fast path.

USC Lab 9: lần đầu flow qua switch, "kernel table is empty" → miss, slow path. Sau "action entries are installed in the kernel", gói tin tiếp theo đi fast path.

Chi phí slow path: ~50-200 microsecond per gói tin. Nếu hit rate thấp, slow path chi phối băng thông thực tế.

## 9.2.5 Ukeys và revalidator threads

Khi megaflow cài vào kernel, `ovs-vswitchd` duy trì **ukey (unique key)** — bản sao userspace chứa flow key, mask, action, statistics counter cuối cùng, timestamp last_used.

**Revalidation**: khi rules thay đổi, revalidator threads (mặc định 4, 500ms interval) scan ukey:

1. Với mỗi ukey, re-walk pipeline với flow key.
2. So sánh action mới vs cũ.
3. Khác → `OVS_FLOW_CMD_SET` update kernel.
4. Không match rule nào → `OVS_FLOW_CMD_DEL`.

**Timeout**: rules có idle_timeout và hard_timeout. Revalidator track timestamp, xóa khi timeout.

Cấu hình `other_config:n-revalidator-threads`. Cache lớn, scan tốn CPU — tăng lên 8-16 thread nếu cần.

## 9.2.6 Cơ chế gọi ngược rate limit

Compass Ch Q mô tả tuning:

- `upcall/show` — số upcall in-flight, avg, max, limit (mặc định 200.000 flow-install/sec).
- `upcall/set-flow-limit 500000` — tăng limit cho workload scale cao.

Chạm limit → kernel drop gói tin → counter `lost` trong `ovs-dpctl show` tăng.

Tương quan:

- `hit` tăng tuyến tính → bình thường.
- `missed` tăng → cache không hit tốt.
- `lost` > 0 → chạm rate limit, nguy cơ mất traffic.

Cấu hình `other_config:n-handler-threads` (mặc định 2). Tăng nếu miss rate cao nhưng lost = 0 — handler nghẽn.

## 9.2.7 Khớp cache rate trong production

NSDI 2015 báo cáo Rackspace production:

- Trước megaflow: hit 70-90%, miss 10-30% peak.
- Sau megaflow: hit 99%+, miss dưới 1%.

Với gói tin rate 1M/sec, ~10.000 upcall/sec — trong khả năng handler thread.

**Dấu hiệu cảnh báo sớm**:

- Coverage miss rate > 1% → compress không hiệu quả.
- `ovs-dpctl show` `lost` > 0 → hit rate limit.
- `upcall/show` avg cao → handler nghẽn.

**Scenarios hit rate tụt**:

1. Traffic đa dạng cao — mỗi microflow khác về field rules match.
2. Rules thay đổi liên tục — revalidator invalidate nhiều ukey.
3. Cache size nhỏ, `max-idle` thấp — entry evict sớm.

## 9.2.8 EMC, Exact Match Cache — tier đầu tiên trong datapath lookup

Cho đến §9.2.7 ta đã thảo luận megaflow cache như thể nó là lớp cache duy nhất. Thực tế, **kernel datapath OVS 2.x có ba tier cache chồng lên nhau**, theo thứ tự lookup: EMC → SMC → megaflow classifier (TSS). Hiểu cả ba tier giúp đọc đúng `pmd-stats-show` và diagnose `miss with success upcall` counter (Phần 9.11 §9.11.8).

**Exact Match Cache (EMC)** là tier đầu — hash table exact-match cho mỗi gói tin 5-tuple (hoặc wider key). Không có wildcard. Mỗi entry EMC tương ứng một **microflow** cụ thể.

### 9.2.8.1 Cấu trúc EMC

| Thuộc tính | Giá trị |
|---|---|
| **Scope** | Per PMD thread (hoặc per CPU core với kernel datapath) — không shared |
| **Lookup** | O(1) hash table, no wildcard |
| **Key** | Full gói tin header fields (cả src và dst, TCP ports, v.v.) |
| **Capacity** | Mặc định 8192 entry per PMD (compile-time constant `EM_FLOW_HASH_ENTRIES`) |
| **Insertion** | Sau lookup thành công từ tier dưới (SMC hoặc megaflow) |
| **Eviction** | LRU-like, ghi đè khi collision hash |
| **Purpose** | Fast path cho flow hot — gói tin hot kênh vô EMC, không cần re-walk TSS |

EMC tồn tại dưới cả kernel datapath (`openvswitch.ko`) và userspace DPDK datapath (`dpif-netdev`). Kernel implementation khác userspace ở chi tiết implementation nhưng semantics giống: hash exact-match cho fast path.

### 9.2.8.2 Đọc hiểu EMC stats qua `pmd-stats-show`

**Bối cảnh:** chỉ áp dụng với `datapath_type=netdev` (DPDK/AF_XDP). Kernel datapath không expose EMC stats trực tiếp — phải qua `coverage/show` counter `emc_hit`, `emc_lookup`.

**Command và output** `[reproduced from OVS DPDK benchmark setup]`:

```shell
$ sudo ovs-appctl dpif-netdev/pmd-stats-show
pmd thread numa_id 0 core_id 2:
  packets received: 10000000
  packet recirculations: 12543
  avg. datapath passes per packet: 1.00
  emc hits: 9050231
  smc hits: 820412
  megaflow hits: 129303
  avg. subtable lookups per megaflow hit: 2.3
  miss with success upcall: 54
  miss with failed upcall: 0
  avg. packets per output batch: 12.4
```

**Anatomy từng counter EMC-related:**

| Counter | Ý nghĩa | Healthy |
|---|---|---|
| `emc hits` | Gói tin match EMC tier | ≥ 85% `packets received` cho workload stable |
| `smc hits` | Gói tin miss EMC, hit SMC tier | 5-15% |
| `megaflow hits` | Miss cả EMC + SMC, hit megaflow classifier | 1-5% |
| `miss with success upcall` | Miss megaflow, upcall thành công install flow | < 0.1% steady state |
| `miss with failed upcall` | Upcall failed (OVSDB lock, error) | Phải = 0 |

**Kịch bản bẻ gãy:**

- **EMC hit < 50% với workload dự kiến stable** → EMC thrashing. 8192 entry không đủ cho workload > 8000 distinct microflow đồng thời. Giải pháp: enable SMC (default on trên OVS 2.15+), hoặc chuyển sang megaflow hit (acceptable miss cấp đầu).
- **SMC hit + megaflow hit > 50% total** → EMC bypass hoặc disabled. Check `ovs-appctl dpif-netdev/pmd-perf-show` detailed.
- **`miss with success upcall` tăng đều** → flow mới liên tục đến; workload scanner hoặc mẫu lưu lượng độc hại. Check OpenFlow classifier flow limit (Phần 9.2.6) và coverage/show `upcall_flow_add`.

### 9.2.8.3 Khi nào disable EMC

EMC tiêu thụ cache memory trên core PMD. Với workload very-low-hit-rate (mỗi gói tin đặc thù), EMC trở thành chi phí phụ thuần. Disable bằng:

```shell
$ sudo ovs-vsctl set Open_vSwitch . other_config:emc-insert-inv-prob=0
# Restart để apply; set về > 0 để re-enable (smaller = more insertion)
```

Giá trị `emc-insert-inv-prob=N` nghĩa: xác suất insert 1/N. Default 100 → 1% insertion rate. Setup specialized (security gateway scan) có thể set 1000 hoặc disable.

## 9.2.9 SMC, Signature Match Cache — tier trung gian OVS 2.15+

SMC được thêm vào OVS 2.15 (02/2021) để giải quyết gap hiệu năng giữa EMC (exact microflow, 8K entry) và megaflow classifier (TSS, chi phí O(T × H) cao hơn). NSDI 2020 paper *HXDP* + OVS mailing list thảo luận SMC là optimization cho workload 100k+ concurrent microflow mà EMC không còn đủ.

### 9.2.9.1 Cấu trúc SMC

| Thuộc tính | Giá trị |
|---|---|
| **Scope** | Per PMD thread (tương tự EMC) |
| **Lookup** | O(1) hash table, compact signature-based |
| **Key** | 16-bit signature của flow hash (không full 5-tuple) |
| **Value** | Pointer về megaflow entry (không trực tiếp action) |
| **Capacity** | 16384 entry per PMD (compile-time) |
| **Insertion** | Sau lookup thành công từ megaflow classifier |
| **Purpose** | Giảm chi phí TSS lookup cho flow "lạnh trung bình" (không đủ hot cho EMC nhưng vẫn regular) |

SMC **không chứa full action** — nó chứa pointer về megaflow entry. Lookup flow: compute signature hash → SMC lookup → nếu hit → dereference pointer → execute megaflow action. Bước dereference nhanh hơn re-walk TSS vì bỏ qua toàn bộ subtable scan.

### 9.2.9.2 Flow lookup path EMC → SMC → megaflow

Thứ tự lookup trong `dpif-netdev` (OVS 2.15+):

```
Packet arrives at PMD
  ↓
(1) EMC lookup: O(1) hash, full 5-tuple exact
    ├── HIT → execute action, done (~50 ns)
    └── MISS → (2)
  ↓
(2) SMC lookup: O(1) hash, 16-bit signature
    ├── HIT → follow pointer to megaflow, execute (~150 ns)
    └── MISS → (3)
  ↓
(3) Megaflow classifier TSS: O(T) subtable scan
    ├── HIT → execute action, insert into SMC (+optional EMC) (~300-500 ns)
    └── MISS → (4)
  ↓
(4) Upcall to userspace
    → handler walks OpenFlow pipeline
    → compile megaflow
    → install into kernel classifier
    → (~50-200 µs)
```

Numbers tham khảo từ NSDI 2020 + OVS benchmarks. Three-tier design cho phép trade-off memory-per-entry vs lookup cost:
- **EMC:** nhỏ (8K entry), cực nhanh (50 ns), chỉ cho hottest flow.
- **SMC:** trung bình (16K entry), nhanh vừa (150 ns), cho regular flow.
- **Megaflow:** lớn (200K entry default), chậm hơn (300-500 ns) nhưng wildcard-capable, cover bulk workload.

### 9.2.9.3 SMC tuning

SMC mặc định bật. Disable bằng `other_config:smc-enable=false` cho khắc phục sự cố hoặc benchmark.

Hiện tại (OVS 3.x) không có option tuning SMC size runtime — recompile OVS với `#define SMC_ENTRIES N` nếu cần. Production hiếm khi cần chỉnh.

## 9.2.10 Cơ chế gọi ngược Netlink protocol, anatomy message giữa kernel và userspace

Khi §9.2.4 giới thiệu upcall, ta chưa xem chi tiết **message wire format** giữa kernel và userspace. Cơ chế gọi ngược đi qua **Generic Netlink** (genl), một cơ chế Linux cho kernel subsystem expose API tới userspace. OVS đăng ký một genl family `ovs_vport`, `ovs_flow`, `ovs_datapath`, `ovs_packet` với kernel.

Hiểu wire format giúp debug khi `vlog/set dpif:file:dbg` bật ra dump Netlink header mà không ai giải thích nổi.

### 9.2.10.1 Cấu trúc Generic Netlink header

```
+-----------------------------------------------+
|           Netlink header (16 byte)             |
|  ┌─────────────────────────────────────────┐  |
|  │ nlmsghdr {                              │  |
|  │   __u32 nlmsg_len   (tổng length msg)   │  |
|  │   __u16 nlmsg_type  (family ID, animated│  |
|  │                      cho ovs_packet)     │  |
|  │   __u16 nlmsg_flags (NLM_F_REQUEST, ...) │  |
|  │   __u32 nlmsg_seq   (sequence #)         │  |
|  │   __u32 nlmsg_pid   (source PID)         │  |
|  │ }                                        │  |
|  └─────────────────────────────────────────┘  |
+-----------------------------------------------+
|       Generic Netlink header (4 byte)          |
|  ┌─────────────────────────────────────────┐  |
|  │ genlmsghdr {                            │  |
|  │   __u8 cmd    (OVS_PACKET_CMD_MISS,      │  |
|  │                OVS_PACKET_CMD_EXECUTE, ..) │  |
|  │   __u8 version                            │  |
|  │   __u16 reserved                          │  |
|  │ }                                        │  |
|  └─────────────────────────────────────────┘  |
+-----------------------------------------------+
|       OVS attribute TLV list (variable)        |
|  ┌─────────────────────────────────────────┐  |
|  │ nlattr {                                │  |
|  │   __u16 nla_len                         │  |
|  │   __u16 nla_type  (OVS_PACKET_ATTR_*)    │  |
|  │ }                                        │  |
|  │ + payload                                │  |
|  │ ... (nhiều attribute)                    │  |
|  └─────────────────────────────────────────┘  |
+-----------------------------------------------+
```

### 9.2.10.2 Cơ chế gọi ngược OVS_PACKET_CMD_MISS anatomy

Khi kernel miss cache, gửi upcall message với:

| Attribute | Type | Ý nghĩa |
|---|---|---|
| `OVS_PACKET_ATTR_PACKET` | raw bytes | Gói tin data đầy đủ (Ethernet frame) |
| `OVS_PACKET_ATTR_KEY` | nested TLV | Flow key đã extract (tun_id, in_port, eth_src/dst, nw_src/dst, ...) |
| `OVS_PACKET_ATTR_USERDATA` | raw | Optional context (khi gói tin đi qua `userspace` action) |
| `OVS_PACKET_ATTR_EGRESS_TUN_KEY` | nested | Tunnel metadata cho egress direction |
| `OVS_PACKET_ATTR_MRU` | u16 | Maximum Receive Unit để handle fragmentation |

**Debug upcall wire format:**

```shell

$ sudo ovs-appctl vlog/set dpif:file:dbg
$ sudo tail -f /var/log/openvswitch/ovs-vswitchd.log

# Trigger a miss: ping an IP chưa từng bị datapath lookup
$ ping -c 1 <fresh_ip>

# Log sample:

```

**Anatomy từng field log:**

| Phần | Ý nghĩa |
|---|---|
| `|dpif(handler1)|` | Subsystem log + thread tên |
| `|DBG|` | Log level (bật bởi `vlog/set dpif:file:dbg`) |
| `system@ovs-system:` | Datapath name |
| `miss upcall:` | Cơ chế gọi ngược nguyên nhân cache miss |

| `icmp(type=8, code=0)` | ICMP Echo Request |
| `execute: actions:4` | Handler reply — execute action output:4 |

### 9.2.10.3 OVS_FLOW_CMD_NEW, cài megaflow về kernel

Sau upcall handler biên dịch megaflow, gửi ngược về kernel `OVS_FLOW_CMD_NEW`:

| Attribute | Ý nghĩa |
|---|---|
| `OVS_FLOW_ATTR_KEY` | Masked flow key (match field cần exact) |
| `OVS_FLOW_ATTR_MASK` | Mask cho từng field (0 = wildcard, all-1 = exact) |
| `OVS_FLOW_ATTR_UFID` | Unique Flow ID (UUID128, OVS 2.7+) |
| `OVS_FLOW_ATTR_ACTIONS` | Datapath action list (output, ct, set, recirc, ...) |
| `OVS_FLOW_ATTR_STATS` | Initial stat (N = 0) |

Kernel apply atomic qua RCU protect — đọc lookup không block write install.

## 9.2.11 Ukey lifecycle và revalidator RCU

§9.2.5 giới thiệu ukey + revalidator tổng quan. Mục này đi sâu vào state machine của ukey + synchronization guarantee mà revalidator cung cấp.

### 9.2.11.1 State machine của một ukey

Mỗi ukey (struct `udpif_key` trong `ofproto/ofproto-dpif-upcall.c`) đi qua lifecycle:

```
           create
              ↓
        ┌──────────┐
        │ CREATED  │  (insert tentative, chưa apply vào kernel)
        └──────────┘
              ↓  flow installed successfully
        ┌──────────┐
        │  VISIBLE │  (kernel đã có megaflow, ukey track counter)
        └──────────┘
              ↓  revalidator sweep thấy ukey
        ┌──────────┐
        │ OPERATIONAL │  (steady state, update stat mỗi sweep)
        └──────────┘
              ↓  rule change → re-walk pipeline
        ┌──────────┐
        │ CHECKING │  (revalidator kiểm tra action vẫn đúng)
        └──────────┘
         ↓ action same          ↓ action differ / no match
     (back to OPERATIONAL)    ┌────────────┐
                              │ EVICTING   │  (OVS_FLOW_CMD_DEL)
                              └────────────┘
                                    ↓
                              ┌────────────┐
                              │ DELETED    │  (free memory sau RCU grace)
                              └────────────┘
```

### 9.2.11.2 Revalidator RCU guarantee

Revalidator thread (default 4, `n-revalidator-threads`) chạy định kỳ (500 ms mặc định). Trong mỗi sweep:

1. **Snapshot phase (RCU read-side):** thread acquire RCU read lock, iterate ukey list.
2. **Per-ukey check:** với mỗi ukey, re-walk OpenFlow pipeline với flow key.
3. **Compare action:** kết quả pipeline mới so với action cũ trong ukey.
4. **Write phase:** nếu action khác, mark ukey EVICTING, gửi `OVS_FLOW_CMD_DEL` về kernel.
5. **Grace period:** ukey không free ngay — chờ RCU grace period (ước 25 ms) để đảm bảo không thread nào còn đọc ukey đó.

RCU (Read-Copy-Update) là Linux primitive cho phép read không block write. Revalidator lợi dụng RCU cho phép dispatcher thread (xử lý upcall) đọc ukey đồng thời revalidator update.

### 9.2.11.3 Đọc hiểu `revalidator/wait` output

**Bối cảnh:** khi debug race condition giữa OpenFlow programming và datapath state, chạy `revalidator/wait` barrier để đảm bảo revalidator đã processed current state.

```shell
$ sudo ovs-appctl dpctl/dump-flows | wc -l
1247
$ sudo ovs-ofctl -O OpenFlow15 del-flows br-int
$ sudo ovs-appctl revalidator/wait
waited <N> ms
$ sudo ovs-appctl dpctl/dump-flows | wc -l
0
```

Output `waited <N> ms` cho thấy block bao lâu. Nếu > 1000 ms → revalidator overloaded.

**Kịch bản bẻ gãy:**

- **`revalidator/wait` block > 5 giây** → revalidator thread có backlog ukey cần check. Check `upcall/show` xem `dump duration` và `avg-updates` per thread. Tăng `n-revalidator-threads`.
- **Counter `coverage/show facet_revalidate` rate > 10k/sec** → revalidator quá tải. Đây là signature của Phần 9.26 revalidator storm.

### 9.2.11.4 Ukey count vs rule count

`memory/show udpif keys` hiển thị ukey count. So với `rules`:

```
rules:12847
udpif keys:1247
```

Ukey thường ÍT hơn rule vì nhiều rule chưa từng match gói tin → không tạo ukey. Ratio `ukey / rule` cho thấy coverage: ukey = flow đang active trong kernel.

**Pathology:** `udpif keys` > `rules` → ukey leak (Phần 9.26). Commit `180ab2fd635e` fix cụ thể bug này.

## 9.2.12 Tóm tắt ba tier cache + cheat-sheet monitoring

### 9.2.12.1 Table tổng hợp

| Tier | Capacity | Lookup cost | Hit rate expected | Key type | Monitoring |
|---|---|---|---|---|---|
| EMC | 8K entry/PMD | ~50 ns | 85-95% | Exact microflow | `pmd-stats-show emc hits` |
| SMC | 16K entry/PMD | ~150 ns | 5-15% | Signature hash | `pmd-stats-show smc hits` |
| Megaflow (TSS) | 200K default | 300-500 ns | 1-5% | Masked wildcard | `pmd-stats-show megaflow hits`, `dpctl/dump-flows` |
| Cơ chế gọi ngược | - | 50-200 µs | < 0.1% steady | - | `upcall/show`, `coverage/show upcall_*` |

### 9.2.12.2 Checklist sức khỏe datapath

```
[ ] EMC hit > 85%                     (pmd-stats-show)
[ ] Tổng hit (EMC+SMC+megaflow) > 99%
[ ] `miss with success upcall` < 100/sec
[ ] `miss with failed upcall` = 0
[ ] `dump_flows_fallback` = 0         (coverage/show)
[ ] `lost` counter = 0                (ovs-dpctl show)
[ ] udpif keys ≤ rules                (memory/show)
[ ] dump duration < 500 ms            (upcall/show)
[ ] avg-updates per revalidator < 5   (upcall/show)
[ ] revalidate rate < 1/sec           (coverage/show)
```

Script production chạy mỗi 5 phút, log delta, alert khi bất kỳ metric vượt ngưỡng.

---



Thứ nhất, megaflow cache là đóng góp kiến trúc quan trọng nhất của NSDI 2015 — chuyển từ exact-match sang wildcard-match qua lazy wildcarding. Compress 10-100 lần, cho phép OVS scale đến workload cloud production.

Thứ hai, Tuple Space Search là thuật toán phù hợp cho software datapath — constant factor tốt và dynamic insert/delete. Thực dụng dựa trên ràng buộc commodity hardware.

Thứ ba, slow path + fast path là cơ chế cốt lõi — USC Lab 9 minh họa rõ: packet đầu đi slow path, cài action vào kernel, packet sau đi fast path. Hiểu cơ chế giúp debug "traffic đầu chậm sau nhanh" hoặc "missed counter tăng".

Thứ tư, cache hit rate 99%+ là mục tiêu vận hành. Dưới ngưỡng, cache thrashing, handler nghẽn, nguy cơ mất traffic. Monitor bằng `ovs-dpctl show`, `ovs-appctl coverage/show`, `ovs-appctl upcall/show`.


Bảy chỉ số đọc được:

- **`masks: total`:** số mask hiện trong cache.
- **`masks: hit/pkt`:** trung bình một gói tin thử qua bao nhiêu mask. Thấp (1-5) là tốt, cao (>40) là flow table bị phân mảnh.

### 9.2.6.3 `ovs-dpctl dump-flows` — kernel flow table

```bash

```


Format datapath khác hoàn toàn với OpenFlow. Match field trong ngoặc đơn, không có `priority`, không có `cookie`, không có `table`. Đây là **megaflow** đã được `ovs-vswitchd` biên dịch xuống kernel cho gói tin cụ thể.

### 9.2.6.4 POE — "kernel flow = OpenFlow flow" bị bác bỏ

**Giả thuyết ngược cần bác bỏ:**

> *"Kernel flow table và OpenFlow flow table là một thứ. Lệnh `ovs-dpctl dump-flows` và `ovs-ofctl dump-flows` trả về cùng dữ liệu."*

### Bước 1, so sánh kết quả hai lệnh

```bash
sudo ovs-ofctl dump-flows s1
```

```
cookie=0x0, duration=60s, table=0, priority=0 actions=NORMAL
```

Chỉ một flow duy nhất: action `NORMAL` mặc định (built-in learning switch).

```bash
sudo ovs-dpctl dump-flows
```

Tám entry với match field đầy đủ + megaflow wildcarded. Giả thuyết bị bác bỏ: kernel và OpenFlow là hai table khác nhau hoàn toàn về số lượng và format.

### Bước 2, giải thích

OVS có kiến trúc hai tầng:

1. **Userspace (OpenFlow):** table rule do controller hoặc admin cài qua `ovs-ofctl`. Thường ít entry, mỗi entry bao phủ nhiều gói tin khác nhau qua wildcard.
2. **Kernel (megaflow):** cache đã biên dịch của userspace, một entry cho mỗi megaflow. Mỗi entry đúng cho một bộ gói tin cụ thể đã thấy.

Ví dụ OpenFlow flow `priority=0 actions=NORMAL` bao phủ mọi gói tin. Kernel không cache flow đó, mỗi gói tin đầu tiên bị miss, đẩy lên userspace, userspace xử lý action `NORMAL` (MAC learning + flood/forward), biên dịch kết quả xuống kernel thành một megaflow entry cụ thể. Gói tin tương tự sau đó match megaflow trực tiếp trong kernel, không đẩy lên userspace nữa.

Đây là lý do cài 1 OpenFlow flow có thể sinh N megaflow trong kernel.

- Workload quá nhiều flow khác nhau (connection churn cao).
- Cache limit chạm trần (upcall/set-flow-limit quá thấp).
- Flow expiration quá ngắn (idle_timeout thấp).
- Revalidator thread không theo kịp (tăng n-revalidator-threads).

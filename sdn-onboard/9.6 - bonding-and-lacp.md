## 9.6.1 OVS bonding khác Linux kernel bonding

OVS implement bond driver độc lập với kernel `bonding` module. Lý do: OVS cần bond state tích hợp với OpenFlow chuỗi xử lý (pipeline) + OVSDB transaction, không thể dựa vào opaque kernel abstraction.

Hệ quả: trên cùng host, có thể dùng OVS bond cho uplink br-int đồng thời với kernel bond cho management interface — hai stack không conflict.

---

## 9.6.2 Ba bond mode

**`active-backup`**: một member active, khác standby. Không yêu cầu peer switch support đặc biệt. Failover dựa vào carrier state + optional `bond_updelay`/`bond_downdelay`.

**`balance-slb`** (source-MAC load balancing): hash theo source MAC, rebalance định kỳ (`bond-rebalance-interval` ms). KHÔNG yêu cầu peer cooperation. Trade-off: một source MAC pin vào một member, không tận dụng nhiều link cho một luồng.

**`balance-tcp`**: L2+L3+L4 hash. YÊU CẦU LACP + MLAG-capable peer (Cisco vPC, Arista MLAG, Juniper ESI). Phức tạp nhưng tận dụng bandwidth đầy đủ.

Tạo bond (compass Ch E):

```bash
[host]# ovs-vsctl add-bond br0 bond0 eth1 eth2 \
         bond_mode=balance-slb \
         lacp=active other_config:lacp-time=fast \
         bond_updelay=100 bond_downdelay=100
```

---

## 9.6.3 LACP options

- `lacp=active`: gửi LACPDU proactively (1s fast-timer, 30s slow-timer).
- `lacp=passive`: chỉ reply khi nhận LACPDU.
- `lacp=off`: tắt LACP (dùng với balance-slb hoặc active-backup).

`other_config:lacp-time=fast` — giảm LACPDU interval từ 30s xuống 1s, failover sub-100ms thay vì sub-90s.

`other_config:lacp-fallback-ab=true` — nếu LACP negotiation fail (peer không respond trong 90s), tự động fallback sang active-backup. Foot-gun-saver quan trọng khi peer switch đang upgrading.

## 9.6.4 Verification: bond/show và lacp/show

```bash
[host]# ovs-appctl bond/show bond0
[host]# ovs-appctl lacp/show bond0
```

`bond/show` output: `bond_mode`, `lacp_status`, active slave, `may_enable` state mỗi member, rebalance countdown.

`lacp/show` output: Actor + Partner system_id, key, priority, state bits. State bit chuẩn theo IEEE 802.1AX:

- `A`ctivity — Actor active LACP
- `C`ollecting — collecting frames
- `E`xpired — state expired
- `G`ood (synchronized)
- `I`n sync với peer
- `K`ey matched
- `N`ew connection
- `P`assive mode

"Letter salad `ACEGIKNP` trên cả hai phía" = healthy bundled link (compass Ch E).

## 9.6.5 Failover test

```bash
[host]# ip link set eth1 down          # bring down one member
[host]# ping -i 0.1 <remote>            # continuous ping
[host]# ovs-appctl bond/show bond0     # observe failover
```

Expect: active-backup sub-second reconvergence, balance-tcp với fast-timer sub-100ms.

## 9.6.6 Anatomy Template A — `ovs-appctl bond/show`

**Bối cảnh khi chạy:** Sau khi add bond + LACP, verify bundling state trước khi đưa vào production. Cũng là first-line debug khi traffic pattern asymmetric.

**Command và output** *(reproduced OVS 2.17.9, 2-member bond balance-slb)*:

```shell
$ sudo ovs-appctl bond/show bond0

---- bond0 ----
bond_mode: balance-slb
bond may use recirculation: no, Recirc-ID : -1
bond-hash-basis: 0
lb_output action: disabled, bond-id: -1
updelay: 100 ms
downdelay: 100 ms
next rebalance: 9873 ms
lacp_status: negotiated
lacp_fallback_ab: true
active-backup primary: <none>
active member mac: 00:15:5d:01:a3:7f(eth1)

member eth1: enabled
  active member
  may_enable: true
  hash 42: 10 kB load
  hash 59: 3 kB load

member eth2: enabled
  may_enable: true
  hash 3: 7 kB load
  hash 76: 1 kB load
```

**10-attribute anatomy:**

| Field | Giá trị mẫu | Ý nghĩa | Red flag nếu |
|-------|-------------|---------|--------------|
| `bond_mode` | `balance-slb` / `balance-tcp` / `active-backup` | 3 mode | Mismatch với peer expectation = bond flap |
| `bond may use recirculation` | `no` / `yes` | balance-tcp cần recirc | `yes` = balance-tcp, cần MLAG peer |
| `updelay` / `downdelay` | 100 ms | Delay từ carrier up/down → enable/disable member | Quá thấp = flapping, quá cao = slow failover |
| `next rebalance` | 9873 ms | Countdown tới next rebalance cycle (default 10s) | <1s = aggressive rebalance, workload scatter |
| `lacp_status` | `negotiated` / `configured` / `off` | LACP state | `configured` = không negotiate (peer issue) |
| `lacp_fallback_ab` | `true` | Fallback to active-backup nếu LACP fail | `false` + LACP fail = bond total down (foot-gun) |
| `active member mac` | `00:15:5d:...` | MAC của member active (cho SLB anchor) | Change frequently = active flapping |
| Per-member `enabled` | `true` / `false` | Driver-level enable | `false` = manual disable hoặc LACP reject |
| Per-member `may_enable` | `true` / `false` | OVS-level may-enable (LACP synchronized) | `false` + `enabled=true` = LACP state issue |
| Per-member `hash N: X kB` | hash bucket → traffic load | SLB distribution evidence | Single hash dominate = workload có elephant flow |

**Kịch bản bẻ gãy:**

- **`lacp_status=configured` + peer không respond**: peer switch chưa enable LACP trên port-channel. Fix: config peer side, hoặc set `lacp=off` + accept static bonding.
- **`may_enable=false` persistent trên 1 member**: LACP partner state bit mismatch. Check `ovs-appctl lacp/show`.
- **Traffic all trên 1 member trong balance-slb**: single source MAC dominate (ví dụ 1 VM HFT). Rebalance countdown đếm nhưng không làm gì. Fix: upgrade lên balance-tcp nếu peer support MLAG.
- **Failover `updelay=100` không đủ**: carrier flap < 100ms được filter ra (good) nhưng persistent flap qua 100ms cause bond cascade. Solution: tăng lên 500ms hoặc debug link layer.

## 9.6.7 Hiểu sai + Key Topic

> **Hiểu sai phổ biến 1:** *"`balance-tcp` chắc chắn tốt hơn `balance-slb` vì distribute gói tin-level."*
>
> **Thực tế:** `balance-tcp` YÊU CẦU peer switch hỗ trợ LACP MLAG (Cisco vPC, Arista MLAG, Juniper ESI, Mellanox MLAG). Nếu peer chỉ là standalone switch LACP (non-MLAG), `balance-tcp` sẽ cause **gói tin reordering + MAC flapping** upstream → app layer TCP retransmit + performance tệ hơn `active-backup`. Quy tắc: verify MLAG ở peer trước khi chọn `balance-tcp`. Nếu không chắc chắn, default an toàn là `active-backup` hoặc `balance-slb`.

> **Hiểu sai phổ biến 2:** *"LACP fast-timer (1s) luôn tốt hơn slow-timer (30s) vì phát hiện nhanh hơn."*
>
> **Thực tế:** LACP fast-timer cost 30× LACPDU bandwidth (1 pkt/s × 2 direction = 2 pps/bond thay vì 0.067 pps). Với hạ tầng 1000+ bond × 2 pps = 2000 pps LACPDU chỉ riêng trong fabric. Switch CPU có thể overload khi fast-timer broadband. Enterprise deployment thường bắt đầu với slow-timer (30s), chỉ enable fast-timer cho critical bond (gateway + spine uplink).

> **Key Topic:** Bond mode chọn sai là cause # 1 của "traffic imbalance" trong OVN deployment. Troubleshooting luồng: (1) `ovs-appctl bond/show` check mode + per-member load. (2) Nếu balance-slb + single source dominate → upgrade balance-tcp (cần MLAG). (3) Nếu balance-tcp + gói tin reorder ở peer → downgrade balance-slb (peer không MLAG). (4) active-backup = default an toàn khi không chắc. Phần 9.26 §9.26.11 case study "LACP bond flap cascade" show scenario production với ToR firmware trigger 200 chassis đồng loạt re-negotiate.

## 9.6.8 Common failure modes

**Failure 1 — Bond flapping do MAC flap upstream.**
Dấu hiệu: `bond/show` show active member đổi 3-5s/lần. Upstream switch log cho thấy MAC flap warning.
Nguyên nhân: balance-tcp trên non-MLAG peer.
Fix: downgrade balance-slb hoặc active-backup. Xác nhận peer MLAG via `show port-channel summary` (Cisco) hoặc equivalent.

**Failure 2 — LACP negotiation stuck ở "configured" không lên "negotiated".**
Dấu hiệu: `lacp_status=configured` persistent sau 2 phút.
Nguyên nhân: peer switch chưa enable LACP, hoặc key mismatch, hoặc system_id collision.
Fix: `ovs-appctl lacp/show` verify actor + partner system_id/key match. Nếu không match, config peer side.

**Failure 3 — Rebalance cycle cause gói tin loss.**
Dấu hiệu: Mỗi `bond-rebalance-interval` (default 10s), ~50-100 gói tin loss trên cross-bond traffic.
Nguyên nhân: SLB rebalance moves hash bucket from member A to member B, in-flight gói tin reorder.
Fix: Tăng `other_config:bond-rebalance-interval=30000` (30s) hoặc migrate to active-backup cho loss-sensitive workload.

**Failure 4 — Fallback-AB kick in khi peer restart.**
Dấu hiệu: `lacp_status=negotiated` → `configured` → fallback-ab active (1 member only) sau peer switch reboot.
Nguyên nhân: Peer LACPDU absence > 90s trigger fallback.
Fix: Fallback-AB correct behavior. After peer recover, LACP re-negotiate, bond resume full bundle.

## 9.6.9 Performance characteristics

**Băng thông thực tế (Throughput) scaling:**

| Mode | Peer requirement | Băng thông thực tế (Throughput) (2× 10G) | Failover độ trễ |
|------|------------------|---------------------|-------------------|
| `active-backup` | None | 10 Gbps (one member only) | 100ms-1s (updelay) |
| `balance-slb` | None | 15-18 Gbps (uneven distribution) | Same as active-backup |
| `balance-tcp` | LACP + MLAG | 19-20 Gbps (near-linear) | Sub-100ms với fast-timer |

**Test methodology:** 100 parallel iperf TCP stream across bond. `balance-slb` plateau khi 1 stream lớn dominate hash bucket. `balance-tcp` best vì L2+L3+L4 hash distribute evenly per-luồng.

**CPU overhead:**
- `active-backup`: <1% CPU per 10G link.
- `balance-slb`: 2-5% CPU (hash computation + rebalance logic).
- `balance-tcp`: 3-8% CPU (higher hash cost, gói tin recirculation).

## Điểm cốt lõi cần nhớ

Thứ nhất, OVS bonding độc lập với Linux kernel bonding — hai stack cùng tồn tại trên một host, không conflict.

Thứ hai, ba bond mode phục vụ ba scenario: `active-backup` (simple, không cần peer support), `balance-slb` (load share per source MAC, không cần MLAG), `balance-tcp` (best băng thông thực tế (throughput), cần LACP + MLAG peer).

Thứ ba, `lacp-fallback-ab=true` là foot-gun-saver bắt buộc cho production — tránh bond down hoàn toàn khi peer LACP misconfigured.

Thứ tư, "letter salad `ACEGIKNP` trên cả hai phía" là quick check healthy LACP. Thiếu letter nào → debug theo letter đó.

Thứ năm, Anatomy Template A `bond/show` 10-attribute là first-line debug. Red flag bond_mode mismatch, lacp_status configured persistent, may_enable false, hash single-dominate.

Thứ sáu, balance-tcp trên non-MLAG peer cause gói tin reorder — cause # 1 của production bond issue. Verify peer MLAG trước khi chọn balance-tcp.

## 8.3.1 tc qdisc taxonomy

A queuing discipline (qdisc) comes in two flavours:

- **Classful**: it has a class hierarchy (`htb`, `hfsc`). A filter classifies the packet into a class.
- **Classless**: it has no hierarchy (`fq_codel`, `pfifo_fast`, `red`, `sfq`).

The root qdisc attaches to an interface:

```bash
[host]# tc qdisc add dev eth0 root handle 1: htb default 10
[host]# tc class add dev eth0 parent 1: classid 1:1 htb rate 100mbit
[host]# tc class add dev eth0 parent 1:1 classid 1:10 htb rate 50mbit
[host]# tc filter add dev eth0 parent 1: protocol ip u32 \
         match ip dst 10.0.0.1 flowid 1:10
```

View the current qdisc:

```bash
[host]# tc qdisc show dev eth0
[host]# tc -s qdisc show dev eth0      # with statistics
```

## 8.3.2 HTB, HFSC, fq_codel

**HTB** (Hierarchical Token Bucket) is a token bucket with a borrow mechanism between sibling classes. A child class borrows bandwidth from the parent class when the parent is idle.

**HFSC** (Hierarchical Fair Service Curve) combines a real-time curve, a link-share curve, and an upper-limit curve. It is more complex than HTB but it gives a better latency guarantee for real-time traffic.

**fq_codel** (Fair Queue plus CoDel AQM) is the default qdisc since kernel 5.0. It was introduced to solve bufferbloat. Minimal configuration:

```bash
[host]# tc qdisc replace dev eth0 root fq_codel
```

**pfifo_fast** is the classic simple FIFO with three priority bands. It is deprecated; `fq_codel` replaces it.

The CoDel algorithm tracks the sojourn time of each packet inside the queue and drops a packet whenever the sojourn time exceeds 5 ms persistently. Keeping the queue short keeps the latency low.

## 8.3.3 conntrack: zones, marks, states

The connection tracking kernel module (`nf_conntrack`) tracks the state of each TCP, UDP, and ICMP connection. It is the foundation of any stateful firewall and of NAT.

The **state** of a connection is one of:

- **NEW**: the first packet, no entry exists in the conntrack table yet.
- **ESTABLISHED**: the connection has completed its handshake (TCP three-way handshake) or has seen two-way traffic (UDP).
- **RELATED**: an ICMP error related to an existing connection, or an FTP data connection that is the child of an FTP control connection.
- **INVALID**: the packet matches none of the states above.

View the conntrack table:

```bash
[host]# cat /proc/net/nf_conntrack | head -5
[host]# conntrack -L                         # requires the conntrack-tools package
```

A **zone** isolates overlapping address space. Several tenants may have overlapping IP ranges (for example, all using `10.0.0.0/8`); they can share the same conntrack table as long as each one has its own zone ID:

```bash
[host]# iptables -t raw -A PREROUTING -i ten1 -j CT --zone 1
[host]# iptables -t raw -A PREROUTING -i ten2 -j CT --zone 2
```

A **connmark** is a mark on a packet that propagates into the conntrack entry. It supports policy routing and QoS.

## 8.3.4 iptables-nft versus nftables native

Ubuntu 20.04 and later ship `iptables-nft`, which is the `iptables` command on top of the `nft` kernel backend. Backward compatibility is preserved: legacy `iptables` rules still work, but the kernel path is the new `nftables` path.

The native `nftables` syntax is more compact:

```bash
# iptables:
iptables -A INPUT -p tcp --dport 22 -j ACCEPT

# nftables equivalent:
nft add rule ip filter input tcp dport 22 accept
```

Migration: `iptables-save | iptables-restore-translate | nft -f -`.

RHEL 9 and later default to native `nftables`. Ubuntu 24.04 and later also moved to native `nftables`.

## 8.3.5 The OVS `ct()` action integrates with conntrack

OVS 2.5 (released in February 2016) added the `ct()` action. An OpenFlow rule pushes the packet through the kernel netfilter conntrack module and reuses the same code path that `iptables` and `nftables` use. There is no reimplementation.

The compass Ch 9 canonical 5-flow stateful firewall (verbatim):

```
ovs-ofctl add-flow br0 "table=0,priority=50,ct_state=-trk,tcp,in_port=veth_l0,actions=ct(table=0)"
ovs-ofctl add-flow br0 "table=0,priority=50,ct_state=+trk+new,tcp,in_port=veth_l0,actions=ct(commit),veth_r0"
ovs-ofctl add-flow br0 "table=0,priority=50,ct_state=-trk,tcp,in_port=veth_r0,actions=ct(table=0)"
ovs-ofctl add-flow br0 "table=0,priority=50,ct_state=+trk+est,tcp,in_port=veth_r0,actions=veth_l0"
ovs-ofctl add-flow br0 "table=0,priority=50,ct_state=+trk+est,tcp,in_port=veth_l0,actions=veth_r0"
```

The `commit` flag installs a conntrack entry. The `zone=N` flag provides multi-tenant isolation.

The `ct_state` match supports `+trk` (tracked), `+new`, `+est`, `+rel`, `+inv`, `+snat`, `+dnat`. Combine them with `+` or `-`.

## 8.3.6 tc qdisc and filter classifier internals

### Kernel queueing path

When a packet leaves interface `eth0`, the kernel traverses this call chain:

```
__dev_queue_xmit()                      [net/core/dev.c]
  -> qdisc_xmit_skb_skb()
        -> sch->enqueue()              (for example HTB: htb_enqueue)
              -> Filter classify       (tc filter rule)
                    -> Class queue
        -> qdisc_run()
              -> sch->dequeue()        (for example HTB: htb_dequeue)
                    -> ndo_start_xmit()  (driver TX)
```

The `sch_*` modules are per-qdisc-type:

- `net/sched/sch_htb.c` (HTB)
- `net/sched/sch_hfsc.c` (HFSC)
- `net/sched/sch_fq_codel.c` (fq_codel)
- `net/sched/sch_pfifo_fast.c` (the legacy default)

Each module implements `enqueue()` (push a packet into the queue), `dequeue()` (pop a packet when the driver is ready to TX), `peek()`, `init()`, and `destroy()`.

### HTB token bucket math

```c
// net/sched/sch_htb.c, function htb_dequeue()
static struct sk_buff *htb_dequeue(struct Qdisc *sch)
{
    struct htb_class *cl = htb_lookup_leaf(sch);
    if (cl->tokens >= L2T(cl, skb_size)) {
        // Enough tokens, dequeue and subtract tokens
        cl->tokens -= L2T(cl, skb_size);
        return skb;
    }
    // Not enough tokens, borrow from the parent class
    if (cl->cmode == HTB_MAY_BORROW) {
        return htb_dequeue_borrow(cl);
    }
    // Still not enough tokens, throttle
    return NULL;
}
```

`L2T()` (length-to-time) equals packet bytes multiplied by the kernel internal ticks-per-byte. The token bucket fill rate equals the configured `rate` parameter. The burst capacity equals the `burst` parameter.

### Filter classifier types

```bash
[host]# tc filter add dev eth0 parent 1: protocol ip u32 \
        match ip dst 10.0.0.1/32 flowid 1:10
```

There are five classifier types:

- `u32`: bit-pattern match at an arbitrary offset inside the packet header.
- `flow`: hash-based classification (per-flow load balance).
- `cgroup`: classify by the cgroup v1 `net_cls` controller (deprecated in cgroup v2).
- `bpf`: an eBPF classifier program (the modern programmable classifier).
- `matchall`: a catch-all without filter logic.

The eBPF classifier (TC offload) is the modern path for complex policy:

```bash
[host]# tc filter add dev eth0 parent 1: protocol ip bpf da \
        obj /path/to/classifier.o sec classifier flowid 1:10
```

The eBPF program is the path that OVS hardware offload uses (Part 9.5 covers TC offload).

## 8.3.7 nf_conntrack source code, tier 2

### State machine implementation

```c
// net/netfilter/nf_conntrack_proto_tcp.c
const struct nf_conntrack_l4proto nf_conntrack_l4proto_tcp = {
    .l4proto = IPPROTO_TCP,
    .pkt_to_tuple = tcp_pkt_to_tuple,
    .invert_tuple = tcp_invert_tuple,
    .new = tcp_new,
    .packet = tcp_packet,
    .timeout = ...,
};
```

Each protocol (TCP, UDP, ICMP) has an `nf_conntrack_l4proto` struct with callbacks. When a packet reaches a netfilter hook (such as PREROUTING):

1. `nf_conntrack_in()` extracts the tuple (the 5-tuple, or an adjusted tuple for ICMP).
2. The kernel looks up the hash table `nf_conntrack_htable_size` (the default size is 65536 buckets).
3. Hit: an existing conntrack entry is found, the kernel runs the `packet()` callback to update the state.
4. Miss: the kernel runs the `new()` callback to create an entry, and inserts it into the hash.

### TCP state transitions

```
SYN_SENT -> SYN_RECV (after SYN+ACK) -> ESTABLISHED (after ACK)
ESTABLISHED -> FIN_WAIT (after FIN) -> CLOSE_WAIT -> LAST_ACK -> CLOSE
                                                |
                                                -> TIME_WAIT (active close)
```

Each state has its own timeout (in `/proc/sys/net/netfilter/nf_conntrack_tcp_timeout_*`):

```bash
[host]# sysctl net.netfilter.nf_conntrack_tcp_timeout_established
432000     # 5 days default
[host]# sysctl net.netfilter.nf_conntrack_tcp_timeout_time_wait
120        # 2 minutes
[host]# sysctl net.netfilter.nf_conntrack_tcp_timeout_close
10         # 10 seconds (after the FIN+ACK exchange)
```

### Hash table

`nf_conntrack_htable_size` defaults to 65536 buckets. Each bucket is an `hlist_nulls_head`. Lookup is O(1) with a good hash. The hash function is `jhash` over the 5-tuple.

Important sysctl values:

```bash
[host]# sysctl net.netfilter.nf_conntrack_max
262144     # max entries in the table (default 65536, scaled with memory)
[host]# sysctl net.netfilter.nf_conntrack_buckets
65536
[host]# cat /proc/sys/net/netfilter/nf_conntrack_count
3421       # current entry count
```

When `nf_conntrack_count` reaches `nf_conntrack_max`, the kernel drops new connections and logs `nf_conntrack: table full, dropping packet`. The pre-emptive remedy is to monitor and to raise `nf_conntrack_max`.

### Zone implementation

```c
// net/netfilter/nf_conntrack_zones_core.c
struct nf_conntrack_zone {
    u16 id;
    u8 flag;        // NF_CT_FLAG_MARK, NF_CT_DIRECTION_*
    u8 dir;
};
```

The zone is embedded in the conntrack tuple key: `(zone, src_ip, dst_ip, src_port, dst_port, proto)`. A different zone means a different bucket, which means no collision even when the 5-tuple is identical.

OVN uses zones heavily. Each tenant has its own `ct_zone` so that conntrack state is isolated. Zones are allocated in `controller/ct-zone.c` of OVN, assigned per LSP through `lsp-set-options ct_zone=...` (this is rare, the assignment is usually automatic).

### `tc -s qdisc show dev eth0`

```

 Sent 1234567890 bytes 8765432 pkt (dropped 1024, overlimits 567 requeues 0)
 backlog 0b 0p requeues 0
qdisc fq_codel 10: parent 1:10 limit 10240p flows 1024 quantum 1514 target 5ms ce_threshold 32us interval 100ms memory_limit 32Mb ecn drop_batch 64
 Sent 234567890 bytes 1765432 pkt (dropped 256, overlimits 0 requeues 0)
 backlog 0b 0p requeues 0
  maxpacket 1514 drop_overlimit 0 new_flow_count 12345 ecn_mark 0
  new_flows_len 0 old_flows_len 0
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| `dropped` | less than 0.01% of total pkt | 0.01 to 1% | greater than 1% | Queue saturation; raise rate or burst |
| `overlimits` | 0 | less than 100 per second | greater than 1000 per second | Token bucket exhausted; rate cap is effective |
| `backlog` | less than 10 packets steady | 50 to 500 | greater than 1000 | Queue is growing, sojourn is increasing |
| `requeues` | 0 | less than 10 per second | greater than 100 per second | Driver TX backpressure |
| `new_flow_count` (fq_codel) | matches application connection rate | spike | sustained spike | DDoS or flow churn |
| `ecn_mark` | 0 | greater than 0 | sustained | Congestion; an ECN-aware app will slow down |

### `conntrack -S` (system-wide statistics)

```
[host]# conntrack -S
cpu=0   found=12345 invalid=0 ignore=0 insert=0 insert_failed=0 drop=0 early_drop=0 error=0 search_restart=12 chaintoolong=0
cpu=1   ...
[total]
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| `found` | matches traffic volume | drops if `nf_conntrack_max` reached | (n/a) | Lookup count |
| `invalid` | less than 0.1% | 0.1 to 1% | greater than 1% | Packets do not match any state; possible scan or spoof |
| `insert_failed` | 0 | greater than 0 | sustained | Hash collision; raise `buckets` |
| `early_drop` | 0 | greater than 0 | sustained | Table near full plus stochastic eviction |
| `drop` | 0 | greater than 0 | sustained | `nf_conntrack_max` reached |
| `search_restart` | less than 0.01% of `found` | 0.01 to 1% | greater than 1% | Hash bucket churn; consider a larger table |
| `chaintoolong` | 0 | 1 to 2 chains | many chains greater than 8 | Hash collision concentrated |

```bash
[host]# cat /proc/sys/net/netfilter/nf_conntrack_count
[host]# cat /proc/sys/net/netfilter/nf_conntrack_max
```

The healthy ratio `count / max` is below 70%. Warning is above 80%. Critical is above 95% (drops are imminent).

## 8.3.9 Capstone POE: how large should `nf_conntrack_max` be for 100K concurrent connections?

> **Context:** An OVN cluster with multi-tenant stateful ACLs, expected 100K concurrent TCP connections per chassis.

**Claim to verify:** "Setting `nf_conntrack_max=100000` is enough for 100K concurrent connections."

**Predict:** False. There must be a buffer for TIME_WAIT entries plus multi-zone overhead.

**Observe (assumed):**

- Set `nf_conntrack_max=100000` and generate a workload of 100K concurrent TCP connections. The conntrack count reaches roughly 120K to 150K.
- Reasoning: each TCP connection has a lifecycle:
  - SYN_SENT and SYN_RECV: a few milliseconds.
  - ESTABLISHED: the connection lifetime (seconds to hours).
  - TIME_WAIT: 120 seconds after close.
  - Plus all ICMP error packets that are RELATED, plus UDP entries (NEW and state).
- With multi-tenant zones, each tenant has one zone. 5K connections per tenant times 20 tenants equals 100K. However, zone collisions raise the search overhead.
- Result: `count/max` reaches 90% or more, the kernel begins to `early_drop`. When a peak burst arrives, plain `drop` happens, and connections fail.

**Explain:**

1. **TIME_WAIT factor:** the TCP close sequence leaves a connection in TIME_WAIT for 120 seconds by default. High application churn means TIME_WAIT entries outnumber the current connections, with a multiplier of 1.5 to 2 times.
2. **RELATED and ICMP error:** each error packet triggers a conntrack entry. In production, each connection generates roughly 0.05 to 0.1 RELATED entries.
3. **Per-zone overhead:** the kernel hash bucket size is shared across zones. 20 zones times 5K connections equals 100K real entries, but the search depth is significant.
4. **Recommendation:** `nf_conntrack_max` should be 2 to 3 times the expected concurrent count. For a 100K target, set 250K to 300K.

**Falsification test:**

- Set `nf_conntrack_max=300000` and run `iperf3` with 100K parallel connections.
- Monitor `count`, `drop`, and `early_drop`. If `drop` is greater than 0, raise `max`.
- Measure the `search_restart` rate. If it exceeds 1%, raise `nf_conntrack_buckets` (the default is `max/4`).

**Where the claim fails:** "100K connections need 100K max" is wrong about the buffer. The correct rule is **`nf_conntrack_max` equals 2 to 3 times the expected concurrent count, to absorb TIME_WAIT, RELATED, and multi-zone overhead. In addition, monitor `early_drop` and `search_restart` to refine the tuning.**

> **Rule:** `nf_conntrack_max` equals 2 to 3 times peak concurrent. `buckets` equals `max/4`. Monitor the `count/max` ratio and alert when it exceeds 80%.

### Capstone Block VIII Lab: stateful firewall across three namespaces

**Outcomes:** Build a topology of three namespaces over a Linux bridge with stateful `iptables` rules, and verify with `conntrack -L`.

**Instructions:**

**1.** Topology:

```bash
[host]# ip netns add client && ip netns add server
[host]# ip link add name br0 type bridge && ip link set br0 up

[host]# ip link add vc type veth peer name vc-br
[host]# ip link add vs type veth peer name vs-br
[host]# ip link set vc netns client && ip link set vs netns server
[host]# ip link set vc-br master br0 && ip link set vs-br master br0
```

**2.** IP addresses:

```bash
[host]# ip -n client addr add 10.0.0.1/24 dev vc
[host]# ip -n server addr add 10.0.0.3/24 dev vs
[host]# ip -n client link set vc up
[host]# ip -n server link set vs up
[host]# ip link set vc-br up && ip link set vs-br up
```

**3.** Stateful firewall:

```bash
[host]# iptables -A FORWARD -p tcp --dport 80 -m conntrack --ctstate NEW -j ACCEPT
[host]# iptables -A FORWARD -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT
[host]# iptables -A FORWARD -j DROP
```

**4.** Generate traffic:

```bash
[host]# ip netns exec server python3 -m http.server 80 &
[host]# ip netns exec client curl http://10.0.0.3/
[host]# conntrack -L
```

**5.** Cleanup: `ip netns delete client server && ip link delete br0`

## Key takeaways

First, `fq_codel` is the default qdisc since kernel 5.0 and addresses bufferbloat. `htb` is for explicit bandwidth limits, and `hfsc` is for real-time guarantees.

Second, the conntrack state machine (NEW, ESTABLISHED, RELATED, INVALID) is the foundation of any stateful firewall. Zones provide multi-tenant isolation.

Third, the OVS `ct()` action reuses the kernel conntrack module without reimplementing it. The sysctl `net.netfilter.nf_conntrack_max` applies to both `iptables` and OVS.

Fourth, Ubuntu 20.04 and later, and RHEL 9 and later, switched to the `nftables` backend. Migrate to native `nft` syntax when the opportunity arises.

## 11.0.1 Tại sao cần overlay encapsulation

VLAN 802.1Q limitations:

- **4096 VLAN limit** (12-bit VID) — không đủ cho cloud scale.
- **L2 flat across physical network** — spanning tree fragile.
- **Không portable across L3** — VLAN không cross router boundary.

Overlay encap giải quyết bằng cách encap L2 frame vào L3 packet:

- **24-bit VNI** → 16.7 triệu segment.
- **Tunnel qua IP routing** — underlay chỉ cần L3 reachability.
- **Portable qua WAN/Internet** — chạy trên UDP, NAT traversal khả thi.

## 11.0.2 VXLAN (RFC 7348, 08/2014)

VXLAN do Cisco/VMware/Broadcom đồng tác giả, chuẩn hóa RFC 7348 tháng 08/2014.

Header format:

```
 ┌──────┬──────┬────────┬────────────┐
 │ Outer│ Outer│ UDP    │ VXLAN      │ Inner Ethernet Frame
 │ IP   │ UDP  │ dport  │ (8B)       │
 │ (20B)│ (8B) │ =4789  │ VNI 24-bit │
 └──────┴──────┴────────┴────────────┘
```

Overhead: 50 bytes (Ethernet 14 + IP 20 + UDP 8 + VXLAN 8).

**VNI**: 24-bit → 16,777,216 tenant segment.
**UDP destination port 4789**. Source port hash từ inner flow để ECMP.

OVS:

```bash
[host]# ovs-vsctl add-port br-tun vxlan0 -- \
         set Interface vxlan0 type=vxlan \
         options:remote_ip=10.0.0.2 \
         options:key=flow \
         options:dst_port=4789
```

`options:key=flow` — defer tunnel key selection cho OpenFlow action `set_tunnel:<vni>`. Multipoint tunneling.

## 11.0.3 Geneve (RFC 8926, 11/2020)

Geneve do VMware/Microsoft đồng đề xuất, chuẩn hóa RFC 8926 tháng 11/2020.

Header:

```
 ┌──────┬──────┬────────┬──────────────┐
 │ Outer│ Outer│ UDP    │ Geneve       │ Inner Ethernet
 │ IP   │ UDP  │ dport  │ (8-66B)      │
 │ (20B)│ (8B) │ =6081  │ VNI + TLV    │
 └──────┴──────┴────────┴──────────────┘
```

Overhead: 50-66 bytes.

**Khác biệt với VXLAN**: Geneve có **TLV options** — mở rộng metadata tùy ý. OVN dùng TLV để carry logical datapath ID, ingress port, connection tracking state. OVN dùng Geneve làm default tunnel encap từ 2015.

**UDP destination port 6081**.

```bash
[host]# ovs-vsctl add-port br-tun geneve0 -- \
         set Interface geneve0 type=geneve \
         options:remote_ip=10.0.0.2 \
         options:key=flow
```

## 11.0.4 STT: deprecated

STT (Stateless Transport Tunneling) — Nicira/VMware 2012 — TCP-like header để piggyback TSO offload của NIC cũ.

Nhược điểm:

- Không standard (không RFC), bị IETF từ chối.
- Behavior như TCP nhưng không thực sự TCP — confuse middlebox.
- Modern NIC có VXLAN/Geneve offload natively từ 2015+.

**OVS 2.x**: STT support. **OVS 3.0+**: STT **REMOVED**. Không dùng cho triển khai mới.

## 11.0.5 So sánh

*Table 11-1: VXLAN vs Geneve vs STT*

| Tiêu chí | VXLAN | Geneve | STT |
|---|---|---|---|
| Standard | RFC 7348 (08/2014) | RFC 8926 (11/2020) | Không RFC |
| Transport | UDP | UDP | TCP-like (custom) |
| Default port | 4789 | 6081 | 7471 |
| Overhead | 50 bytes | 50-66 bytes | 54+ bytes |
| Tenant ID | 24-bit VNI | 24-bit VNI + TLV extensible | 64-bit context |
| TLV metadata | Không | Có (RFC 8926) | Không |
| NIC offload | Hầu hết NIC hiện đại | ConnectX-4+ | Hiếm |
| OVS support | Stable | Stable, OVN default | REMOVED OVS 3.0+ |
| Khuyến nghị 2026 | Enterprise DC | OVN, cloud-native | Không dùng |

## 11.0.6 Multipoint tunneling với options:key=flow

Pattern: thay vì tạo N tunnel port cho N remote, dùng một tunnel port với `options:key=flow` + action `set_tunnel:`:

```bash
[host]# ovs-vsctl add-port br-tun ovn-geneve -- \
         set Interface ovn-geneve type=geneve \
         options:remote_ip=flow \
         options:key=flow

[host]# ovs-ofctl add-flow br-tun \
         "table=0,in_port=1,dl_dst=aa:bb:cc:00:00:01,\
          actions=set_tunnel:1000,set_field:10.0.0.2->tun_dst,output:ovn-geneve"
```

`options:remote_ip=flow` cho phép controller quyết định IP đích cho từng flow. Một tunnel port phục vụ unlimited remote chassis, model OVN dùng.

## 11.0.7 Geneve packet format byte-by-byte

Phần 11.0.3 đã giới thiệu Geneve high-level. Phần này đi sâu vào từng byte của Geneve header để kỹ sư đọc được packet capture trực tiếp.

### Outer headers (48 bytes cho IPv4 underlay)

```
Byte:    0      6      12     14    18    20      24      28
         ┌──────┬──────┬──────┬─────┬─────┬───────┬───────┬───────────────────┐
Layer 2: │ dst  │ src  │ ether│ VLAN│ ... │       │       │                   │
         │ MAC  │ MAC  │ Type │ tag │     │       │       │                   │
         │ 6B   │ 6B   │ 0x800│(opt)│     │       │       │                   │
         └──────┴──────┴──────┴─────┴─────┴───────┴───────┴───────────────────┘
                                                       
Byte:    14            34
         ┌─────────────────────────────────────────────────────┐
Layer 3: │ Outer IPv4 header (20B without options)             │
IPv4:    │ ver=4 + IHL=5, ToS, total_len, ID, flags, frag_off, │
         │ TTL, proto=17(UDP), checksum,                       │
         │ outer_src_IP (4B), outer_dst_IP (4B)                │
         └─────────────────────────────────────────────────────┘

Byte:    34          42
         ┌──────────────────────┐
Layer 4: │ Outer UDP (8B)        │
UDP:     │ src_port (hash inner) │
         │ dst_port = 6081       │  ← Geneve magic number
         │ length, checksum       │
         └──────────────────────┘
```

Tổng outer = 14 (Eth) + 20 (IPv4) + 8 (UDP) = **42 byte**. Nếu IPv6 underlay: 14 + 40 + 8 = 62 byte.

### Geneve base header (8 bytes, RFC 8926 §3.4)

```
Byte:    42         43         44         46          48        49        50
         ┌──────────┬──────────┬──────────┬──────────┬─────────┬─────────────────┐
Geneve:  │ Ver (2)  │ Opt Len  │ O C R    │ Reserved │ Protocol│ VNI (3B)        │
         │ + Opt Len│ * 4 = total│ flags  │ (rsvd=0) │ Type    │ Virtual Network │
         │ Ver=0    │ TLV bytes│          │          │ 0x6558  │ Identifier      │
         │ (6 bit)  │ (6 bit)  │ 1+1+6 bit│ 6 bit    │ (Eth)   │ + 1B reserved   │
         └──────────┴──────────┴──────────┴──────────┴─────────┴─────────────────┘
```

| Field | Width | Mô tả |
|-------|-------|-------|
| Version | 2 bit | Phải = 0 (Geneve v0) |
| Opt Len | 6 bit | Số 4-byte word của TLV options. Min 0, max 63 (252 byte TLV) |
| O (OAM) flag | 1 bit | =1 nếu OAM packet (control plane, không phải data) |
| C (Critical) flag | 1 bit | =1 nếu có TLV critical, receiver phải hiểu hết hoặc drop |
| Reserved | 6 bit | =0 |
| Protocol Type | 16 bit | Inner protocol. 0x6558 = Trans-Ether-Bridging (inner Ethernet) |
| VNI | 24 bit | Virtual Network Identifier (datapath ID) |
| Reserved | 8 bit | =0 |

Ví dụ Geneve base header với VNI=1, không có TLV options:

```
00 00 65 58 00 00 01 00
^^ ^^ ^^^^^ ^^^^^^^^^^^
|  |  |     |
|  |  |     +-- VNI=1 (3 byte) + reserved (1 byte)
|  |  +-------- Protocol Type 0x6558 (Ethernet bridging)
|  +----------- Flags + Reserved
+-------------- Version=0, Opt Len=0
```

### Geneve TLV options (variable, 0-252 byte, RFC 8926 §3.5)

Mỗi TLV có cấu trúc:

```
Byte:    0          2         3         4
         ┌──────────┬─────────┬─────────┬─────────────────┐
TLV:     │ Class    │ Type    │ R+Length│ Value (variable)│
         │ 16 bit   │ 8 bit   │ 3+5 bit │ 0-128 byte      │
         └──────────┴─────────┴─────────┴─────────────────┘
```

| Field | Width | Mô tả |
|-------|-------|-------|
| Class | 16 bit | Vendor / organization / IANA-allocated. 0x0000-0x00FF reserved IETF; 0x0100-0xFFFE assigned per request; 0xFFFF reserved |
| Type | 8 bit | Vendor-specific. Cao bit (bit 7) = critical flag |
| Reserved | 3 bit | =0 |
| Length | 5 bit | Số 4-byte word của Value. 0-31 → 0-124 byte |
| Value | 0-124 byte | TLV payload |

### IANA Geneve TLV class registry

[IANA Geneve TLV Class Registry](https://www.iana.org/assignments/nvo3/nvo3.xhtml#geneve-option-class) (cập nhật 2026-04). Tổng ~30 vendor đăng ký:

| Class | Owner | Use case |
|-------|-------|----------|
| 0x0000-0x00FF | IETF | Reserved |
| 0x0100 | LinuxFoundation | OVS reserved (legacy) |
| 0x0101 | Open vSwitch | OVS internal use |
| **0x0102** | **OVN** | **OVN logical metadata (datapath, inport, outport)** |
| 0x0103 | Cisco | Cisco-specific |
| 0x0104 | Oracle | Oracle Cloud |
| 0x0105 | Amazon | AWS VPC / Transit Gateway |
| 0x0107 | Microsoft | Azure ACS |
| 0x010A | NVIDIA | BlueField DPU |
| ... | ... | ... |

OVN dùng class `0x0102`, được reserved chính thức bởi IANA cho project OVN.

### OVN TLV class 0x0102 layout (từ source `controller/physical.c` `branch-22.03`)

OVN encode 2 logical metadata field vào 1 TLV duy nhất:

| Bit range | Field | Mô tả |
|-----------|-------|-------|
| 0-31 | logical outport | Logical egress port number trên chassis nhận |
| 32 | reserved | =0 |
| 33-47 | logical inport | Logical ingress port number trên chassis gửi (15-bit) |
| 48-63 | reserved | =0 |

(Tổng 8 byte payload = 64 bit = 2 word.)

Datapath logical (LS hoặc LR) tunnel_key đi qua **VNI** chính của Geneve base header (24 bit), không qua TLV.

Source code chính xác (đã verified branch-22.03):

```c
// controller/physical.c, function put_encapsulation()
if (tun->type == GENEVE) {
    put_load(datapath->tunnel_key, MFF_TUN_ID, 0, 24, ofpacts);          // VNI 24-bit
    put_load(outport, mff_ovn_geneve, 0, 32, ofpacts);                    // TLV bits 0-31
    put_move(MFF_LOG_INPORT, 0, mff_ovn_geneve, 16, 15, ofpacts);         // TLV bits 16-30
}
```

(Cross-link Phần 13.7 §13.7.8 cho chi tiết Geneve TLV encoding.)

### Decode hex dump thực tế

Geneve packet với VNI=1, OVN TLV class 0x0102 type=0x80, outport=2, inport=1:

```
0000  ff ff ff ff ff ff aa bb cc dd ee ff 08 00 45 00     # Outer Eth + IPv4 start
0010  00 6e 00 00 40 00 40 11 ... <src IP> <dst IP>      # IPv4 (20B)
0020  17 c1 17 c1 00 5a 00 00                            # UDP src/dst=6081, len=90
                                                           
0028  02 02 65 58 00 00 01 00                            # Geneve base
      ^^ ^^ ^^^^^ ^^^^^^^^^^^
      |  |  |     |
      |  |  |     +-- VNI=1 + reserved
      |  |  +-------- Proto 0x6558 (Eth bridging)
      |  +----------- Flags=0, Reserved=0
      +-------------- Ver=0, Opt Len=2 (8 byte TLV)
                                                           
0030  01 02 80 02 00 00 00 02 00 00 00 01                # Geneve TLV
      ^^^^^ ^^ ^^ ^^^^^^^^^^^ ^^^^^^^^^^^
      |     |  |  |           |
      |     |  |  |           +-- Inport=1 (after move bits 16-30)
      |     |  |  +-------------- Outport=2 (32-bit, bits 0-31)
      |     |  +----------------- Length=2 (8 byte payload)
      |     +-------------------- Type=0x80 (Critical bit set)
      +-------------------------- Class=0x0102 (OVN reserved)

003C  <inner Ethernet frame>                             # Inner packet
```

Decode bằng tcpdump:

```bash
[host]# tcpdump -i eth0 -nn -X udp port 6081 | head -50
```

Wireshark có Geneve dissector built-in (từ Wireshark 2.4+), tự decode TLV class 0x0102 nếu có protocol dissector OVN plugin (community-maintained).

## 11.0.8 Header overhead math chính xác cho OVN

OVN triển khai dùng Geneve làm tunnel, overhead cụ thể:

| Component | Bytes (IPv4 underlay) | Bytes (IPv6 underlay) |
|-----------|----------------------|----------------------|
| Outer Ethernet (no VLAN) | 14 | 14 |
| Outer Ethernet + 802.1Q VLAN | 18 | 18 |
| Outer IPv4 (no options) | 20 | 0 |
| Outer IPv6 | 0 | 40 |
| UDP | 8 | 8 |
| Geneve base | 8 | 8 |
| Geneve TLV (OVN class 0x0102) | 8 | 8 |
| **Tổng IPv4 không VLAN** | **58** | |
| **Tổng IPv4 + VLAN** | **62** | |
| **Tổng IPv6 không VLAN** | | **78** |

Note Phần 0.2 §0.2.12 đã claim "66 byte" overhead. Số đó tính 14+20+8+8+16 = 66, với TLV 16 byte (OVN có thể thêm TLV thứ 2 cho future use). Số chuẩn cho **chỉ class 0x0102** hiện tại = 58 byte. Kỹ sư khi tính MTU cần dùng số 58 cho IPv4 underlay không VLAN.

### MTU formulas

- **Underlay MTU = 1500** (default Ethernet jumbo). OVN inner MTU max = 1500 - 58 = **1442** byte.
- **Underlay MTU = 9000** (jumbo frames). OVN inner MTU max = 9000 - 58 = 8942 byte.

Khi underlay = 1500 và workload (Pod, VM) có MTU = 1500, sẽ fragmentation hoặc PMTU drop. Phần 11.1 đi sâu PMTUD failure modes.

## 11.0.9 Receiver-side decap, table 0 trên br-int

Phần 0.2 §0.2.13 đã mô tả high-level decap trên Chassis Y. Đi sâu OpenFlow flow rule cài bởi `controller/physical.c put_remote_port_redirect()`:

```
table=0, in_port=<tunnel-OFPort>, tun_id=<tunnel_key>:
    move:NXM_NX_TUN_METADATA0[0..31]->NXM_NX_REG15[0..31]    # outport restore (32-bit)
    move:NXM_NX_TUN_METADATA0[16..30]->NXM_NX_REG14[0..14]   # inport restore (15-bit)
    move:NXM_NX_TUN_ID[0..23]->OXM_OF_METADATA[0..23]        # datapath restore
    resubmit(,<egress-table>)
```

Pattern là gương ngược của encap tại Chassis X. 3 register OVN dùng:

- `OXM_OF_METADATA[0..23]`: datapath logical ID (= Geneve VNI).
- `NXM_NX_REG14[0..14]`: logical inport (15-bit).
- `NXM_NX_REG15[0..31]`: logical outport (32-bit, có reserved).

(`NXM_NX_TUN_METADATA0` là OVS extension OXM, mỗi vendor có thể có nhiều `tun_metadataN`. OVN chỉ dùng `tun_metadata0`.)

## 11.0.10 NIC hardware offload cho overlay

Modern NIC (intel E810, NVIDIA ConnectX-4+, Broadcom Stingray) hỗ trợ offload các operation Geneve/VXLAN ở phần cứng. Bảng support matrix:

| Feature | NIC hỗ trợ (2026) | Mô tả |
|---------|-------------------|-------|
| RX checksum (inner) | Hầu hết NIC từ 2018+ | Hardware tính checksum inner IPv4/TCP/UDP, kernel skip |
| TX checksum | Hầu hết | Hardware tính outer + inner checksum |
| GSO (Generic Segmentation Offload) | Linux 4.x+ | Kernel software offload, NIC nhận skb size 64KB |
| TSO (TCP Segmentation Offload) cho VXLAN | ConnectX-4+, E810 | NIC tự segment TCP qua VXLAN |
| TSO cho Geneve | ConnectX-5+, E810 | NIC tự segment qua Geneve |
| GRO (Generic Receive Offload) cho Geneve | ConnectX-5+, E810 | NIC merge multiple Geneve packets thành 1 large skb |
| RSS (Receive Side Scaling) cho overlay | ConnectX-4+, E810 | Hash inner flow để distribute lên multiple CPU queue |

Kiểm chứng offload trên Linux:

```bash
[host]# ethtool -k eth0 | grep -E '(tx-udp|rx-udp|gro|gso|tso|rss).*tunnel'
tx-udp-segmentation: on
rx-udp-tunnel-port-offload: on
tx-udp_tnl-segmentation: on
tx-udp_tnl-csum-segmentation: on
```

Khi offload disabled hoặc NIC không support → CPU phải làm hết → throughput giảm 50-70% với 10G+ link. Với OVS DPDK, mọi offload qua PMD driver, không qua kernel.

(Cross-link Phần 9.5 OVS hardware offload deep + Phần 11.1 §"NIC offload tier 2" cho OVS-specific behavior.)

## 11.0.11 Anatomy Template A: `ovs-vsctl get Interface <tunnel> options`

> **Bối cảnh khi chạy:** Kiểm tra cấu hình tunnel cụ thể, debug khi tunnel không UP.

```bash
[host]# ovs-vsctl get Interface ovn-chassis-y options
{remote_ip="192.0.2.20", key=flow, csum="true", tos="inherit", \
 ttl="64", df_default="true", egress_pkt_mark="0", local_ip="192.0.2.10"}
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| `remote_ip` | IP underlay routable | IP local_ip cùng host | Empty hoặc unreachable | Tunnel sẽ fail; check `ip route get <remote_ip>` |
| `key` | `flow` cho OVN | Số cố định cho non-OVN | Empty | Multipoint tunneling pattern |
| `csum` | `true` (default) | `false` (disabled, mất integrity protection) | (n/a) | Disable chỉ khi NIC offload broken |
| `tos` | `inherit` (copy inner DSCP) | Số cố định | Empty | Inherit thường đúng cho QoS |
| `ttl` | 64 | < 32 (có thể hết TTL trên multi-hop underlay) | 1 | Hết TTL trên ToR redundant path |
| `df_default` | `true` (mandate PMTUD) | `false` (allow fragmentation) | (n/a) | DF=1 buộc PMTUD; DF=0 cho phép IPv4 fragment |
| `egress_pkt_mark` | 0 | Custom value | (n/a) | Cho QoS trên egress (Phần 9.9) |
| `local_ip` | Match chassis underlay IP | Empty (auto) | Sai IP | Manual override; prefer auto |

**Kịch bản bẻ gãy:**

- `remote_ip` ping được nhưng UDP 6081 fail: firewall đang drop port 6081. Check `iptables -L` + `ufw status`.
- `csum=true` nhưng NIC offload broken: throughput drop 50%. Tạm `csum=false` workaround, root cause fix NIC driver/firmware.
- `df_default=true` + underlay MTU < 1500: PMTUD ICMP fragmentation needed reply không reach lại sender → PMTU black hole. Phần 11.1 deep dive.

## 11.0.12 ▶ Capstone POE: VXLAN hay Geneve cho OVN cluster mới?

> **Bối cảnh:** Kỹ sư thiết kế cluster OVN production mới 100 chassis, 50K LSP, cross-DC RTT 50ms.

**Tuyên bố cần kiểm chứng:** "VXLAN đơn giản và stable hơn Geneve, nên chọn VXLAN cho OVN."

**Predict:** Sai. OVN yêu cầu Geneve cho mọi feature production-grade.

**Observe (giả định, kiểm chứng khi có lab):**

- Set `ovn-encap-type=vxlan` trên chassis: tunnel UP nhưng OVN log warning `"VXLAN encap does not support all OVN features"`.
- Migration cross-chassis: Port_Binding migrate fail vì VXLAN VNI 24-bit không carry inport metadata. OVN fall back về MC_FLOOD cho mọi packet → 5-10x bandwidth waste.
- Stateful ACL với conntrack zone cross-chassis: zone metadata không được preserve qua VXLAN → conntrack entry per-chassis → reply path dùng zone khác → packet drop bất thường.
- LR distributed (chassisredirect): outport metadata không có → packet vào LR Y nhưng không biết logical egress port → drop hoặc loop.

**Explain:**

1. OVN cần truyền 3 metadata field cross-chassis: datapath (24-bit), inport (15-bit), outport (32-bit). VXLAN VNI 24-bit chỉ đủ datapath, không đủ inport+outport. Geneve TLV variable carry full metadata.
2. OVN từ 2015 default Geneve. VXLAN support trong OVN code path là legacy (cho L3-only triển khai đơn giản, không multi-tenant ACL/LB/NAT).
3. "Đơn giản hơn" sai vì không thấy được trade-off feature. VXLAN encap header đơn giản hơn (8 byte fixed) nhưng cluster vận hành phức tạp hơn (workaround missing metadata).

**Falsification test:**

- Setup OVN cluster với VXLAN, configure ACL stateful + LR distributed.
- Chạy traffic cross-chassis với conntrack flow.
- Quan sát `dpctl/dump-conntrack` zone consistency cross-chassis. Nếu zone match → VXLAN OK; nếu lệch → VXLAN không phù hợp.

**Tuyên bố sai ở đâu:** "VXLAN simpler" nhìn ở wire-format level, không nhìn được ở OVN feature level. Đúng phải: **Geneve là default cho OVN production. VXLAN chỉ dùng cho cluster legacy cần interop với non-OVN endpoint không hiểu Geneve.**

> **Quy tắc:** OVN mới = Geneve. VXLAN cho OVN = legacy interop only.

[hostA]# ip addr add 192.168.1.1/24 dev tap-a
[hostA]# ovs-vsctl add-port br0 vxlan0 -- set Interface vxlan0 type=vxlan \
         options:remote_ip=10.0.0.2 options:key=100
```

**2.** Host B (10.0.0.2):

## Điểm cốt lõi cần nhớ

Thứ nhất, overlay encap giải quyết VLAN 4096 + L2 flat limitation. VXLAN 24-bit VNI → 16.7M segment qua L3 underlay.

Thứ hai, Geneve successor chính thức — TLV extensible metadata. OVN default Geneve từ 2015.

Thứ ba, STT deprecated hoàn toàn — OVS 3.0+ remove. Không chọn cho triển khai mới.

Thứ tư, `options:key=flow` + `options:remote_ip=flow` + `set_tunnel` multipoint — scale tốt, pattern OVN dùng.

## 11.0.13 Geneve TLV như một cơ chế mở rộng metadata: bản chất, mã hoá OVN, vận hành

> Phần này đào sâu vào **chính cơ chế Geneve TLV** (Type-Length-Value option section trong header Geneve) và **lớp mã hoá riêng của OVN** (Option Class `0x0102`) tách bạch khỏi mô tả Geneve high-level ở §11.0.3 và byte layout ở §11.0.7. Mục tiêu: kỹ sư hiểu thấu cấu trúc TLV chung, vì sao OVN chiếm riêng một class, cơ chế truyền 71 bit metadata logical xuyên chassis qua TLV, và cách tra ngược một sự cố cross-chassis về byte cụ thể trong TLV.

### Khái niệm cốt lõi: TLV mở rộng vs base header cố định

Geneve TLV là **vùng option có độ dài thay đổi** ngay sau base header 8 byte của Geneve, mỗi option đóng gói theo bộ ba **(Class 16 bit, Type 8 bit, Length 8 bit) + Value (0 đến 124 byte)** theo RFC 8926 §3.5. Ngược với VXLAN (header cố định 8 byte, không có chỗ để thêm thông tin) và STT (deprecated, header TCP-like cố định), Geneve thiết kế TLV để bất kỳ đơn vị triển khai nào (vendor, project mã nguồn mở, IETF working group) cũng có thể đăng ký một **Option Class** riêng và mã hoá metadata tự định nghĩa vào tunnel mà không sửa đổi base header. Đây chính là lý do OVN chọn Geneve làm tunnel mặc định từ 2015: cần truyền **logical datapath + logical inport + logical outport + flags** xuyên chassis, mà cả ba con số này đã chiếm 71 bit, vượt xa VNI 24 bit của VXLAN. Loại: **cơ chế mở rộng (extension mechanism) thuộc protocol Geneve**, không phải một message hay một command, cũng không phải một bảng OVSDB.

### Lịch sử + bối cảnh ra đời

Geneve được Jesse Gross (VMware), Ilango Ganga (Intel), T. Sridhar (VMware) khởi xướng tháng 02/2014 với draft `draft-gross-geneve-00` đáp lại nhu cầu mạnh mẽ của VMware NSX-T (kế nhiệm NVP/NSX-MH) muốn thay STT (proprietary, không qua được middlebox) bằng một protocol có khả năng truyền metadata phong phú cho service chaining + OAM + tenant context. RFC 8926 chuẩn hoá tháng 11/2020 sau 6 năm draft. OVN team (Justin Pettit, Russell Bryant, Numan Siddique) ra mắt OVN tháng 09/2015 cùng OVS 2.5; ngay từ phát hành đầu tiên đã chọn Geneve làm tunnel mặc định và đăng ký **Option Class `0x0102`** với IANA dưới tên *Open Virtual Networking (OVN)*, contact `OVN-Committers`. Trước đó, tháng 04/2014 IANA đã cấp `0x0100` cho Linux-NetDev và `0x0101` cho OVS-Committers (dùng cho thử nghiệm OVS internal); `0x0102` là class thứ ba trong dải, chuyên dụng cho metadata logical pipeline OVN. INT (In-band Network Telemetry) lấy `0x0103` ngay sau OVN. Việc OVN có class IANA-allocated riêng (chứ không tái sử dụng `0x0101` của OVS) là quyết định kiến trúc có chủ đích: tách kênh ngữ nghĩa OVS (data path layer) khỏi OVN (logical pipeline layer).

### Vị trí trong kiến trúc

Geneve TLV nằm ở **lớp tunnel encapsulation** (data plane underlay), giữa **bridge `br-int`** trên một chassis (nơi flow OpenFlow gọi action `output:<tunnel-port>`) và **bridge `br-int`** trên chassis đích (nơi flow `table=0` decap và resubmit). Trên đường đi vật lý, TLV chỉ tồn tại trên dây giữa hai NIC vật lý của hai chassis (gói qua underlay router/switch L3). Scope: **per-packet, per-cross-chassis-tunnel**. Mỗi packet OVN cross-chassis (đi qua tunnel ký hiệu `ovn-<chassis>-0` trên `br-int`) đều mang đúng một bộ TLV Class `0x0102` được sinh ở thời điểm encap; packet local (ingress + egress cùng một chassis) **không hề chạm TLV** vì không qua tunnel port. Kernel datapath OVS xử lý byte TLV qua module `vport-geneve` (file `net/openvswitch/vport-geneve.c`); userspace ofproto-dpif quản lý TLV registration qua extension OXM `tun_metadata0..63` (file `lib/tun-metadata.c` của OVS).

### Vai trò + nhiệm vụ trong kiến trúc

Geneve TLV gánh **bốn vai trò** trong kiến trúc OVN-OVS:

1. **Truyền tunnel_key (datapath logical) qua VNI base header**: cột `Datapath_Binding.tunnel_key` (24 bit) được nhồi vào trường VNI của Geneve base header (24 bit) qua action `set_field:NXM_NX_TUN_ID`. Đây không phải TLV mà là VNI chính thống của Geneve, nhưng OVN coi nó là một phần của bộ metadata cross-chassis.
2. **Truyền logical inport + outport qua TLV `mff_ovn_geneve`**: 32 bit của TLV `0x0102` chứa `logical_outport` ở bit 0..31 và `logical_inport` ở bit 16..30 (sau khi shift), cho phép chassis nhận biết chính xác cửa vào và cửa ra logical mà northd đã chỉ định.
3. **Cho phép middle protocol mở rộng tương lai**: vì Geneve TLV variable, OVN có thể thêm class mới (ví dụ MAC binding cache, ECMP hint) mà không cần đổi format wire-level base.
4. **Phân biệt traffic OVN vs traffic non-OVN cùng UDP 6081**: khi underlay có nhiều dự án cùng dùng Geneve (NSX-T, INT telemetry, Linux native Geneve), Class `0x0102` là dấu hiệu nhận dạng nhanh để tcpdump/tshark filter chính xác traffic OVN.

### Vì sao sinh ra (động cơ thiết kế)

Trước Geneve TLV, OVN buộc phải gò metadata vào **một trong ba phương án không phù hợp**: (a) VXLAN VNI 24 bit, đủ chứa datapath nhưng không còn chỗ cho inport/outport, dẫn đến chassis nhận phải tra cứu bảng cục bộ tốn thời gian; (b) STT 64-bit context, đủ chỗ nhưng STT không qua được hầu hết middlebox vì giả mạo TCP, các cloud provider chặn UDP-over-TCP-fake; (c) đóng gói metadata vào inner packet (ví dụ thêm header VLAN trong inner Ethernet), phá vỡ tính trong suốt L2 cho tenant. Geneve TLV giải quyết cả ba: truyền được 71 bit metadata, không giả mạo protocol nào, không phá vỡ inner frame. Đồng thời, đăng ký một Option Class IANA-allocated giúp OVN có namespace 16-bit Type độc lập (`0x80..0xFF` reserved cho dùng nội bộ OVN), tránh xung đột với INT, NSX-T, hoặc bất kỳ project nào khác cùng dùng UDP 6081.

### Vấn đề mà nó giải quyết

Geneve TLV (Class `0x0102` + base VNI) giải quyết **năm vấn đề kỹ thuật cụ thể**:

1. **Phân tán logical pipeline qua nhiều chassis mà không có controller trung tâm**: chassis nhận đọc TLV và biết ngay phải resubmit vào datapath nào, port logical nào, không cần tra cứu OVSDB qua mạng.
2. **Atomic semantics cho cross-chassis hop**: TLV nằm trong cùng frame Ethernet với inner packet, nên mất gói = mất luôn metadata, không có tình huống "metadata đến trước inner".
3. **Compatible với underlay L3 router**: Geneve dùng UDP 6081, router thấy như UDP thường, không strip TLV (khác với một số middlebox proprietary có thể strip option lạ).
4. **Cho phép NIC offload checksum + TSO**: NIC ConnectX-5+, Intel E810 hiểu Geneve TLV header, có thể compute checksum + segment TCP qua tunnel cứng.
5. **Tách kênh metadata OVS (Class `0x0101`) khỏi OVN (Class `0x0102`)**: cùng một host có thể chạy OVS-only bridge (dùng `0x0101` cho experimentation) song song OVN bridge (dùng `0x0102` cho logical pipeline) mà không xung đột TLV space.

### Tầm quan trọng

Geneve TLV là **cốt lõi tier 1** của OVN, tương đương Logical_Flow và Port_Binding. Kỹ sư không hiểu byte layout của TLV `0x0102` sẽ không thể debug bất kỳ sự cố cross-chassis nào: VM bên này ping VM bên kia rớt gói, gateway redirect không sang chassis HA, multicast flood không tới remote chassis, conntrack zone lệch giữa hai chassis. Mọi feature OVN production-grade (live migration multichassis, distributed gateway router, ACL stateful cross-chassis, Load_Balancer hairpin) đều dựa vào TLV. Trong 320+ keyword của REF, Geneve TLV nằm trong nhóm 15 keyword cornerstone OVN cùng `ovn-northd`, `ovn-controller`, NBDB, SBDB, Logical_Flow, Datapath_Binding, Port_Binding, Logical_Switch, Logical_Router, ACL, Load_Balancer, NAT, ovn-trace, ovn-detrace.

### Cơ chế hoạt động chi tiết: encode + transit + decode

Một packet cross-chassis đi qua **ba pha** từ góc nhìn TLV:

1. **Encode tại chassis gửi (egress)**: ovn-controller dịch Logical_Flow action `output:<remote-port>` thành chuỗi OpenFlow action: (i) `set_field:datapath->NXM_NX_TUN_ID[0..23]` nhồi tunnel_key 24 bit vào VNI; (ii) `set_field:outport->NXM_NX_TUN_METADATA0[0..31]` nhồi outport 32 bit vào TLV value bit 0..31; (iii) `move:NXM_NX_REG14[0..14]->NXM_NX_TUN_METADATA0[16..30]` move inport (15 bit reg14) vào TLV value bit 16..30 (chèn lên trên outport, khả thi vì outport số nhỏ thường < 65536 chỉ chiếm bit 0..15); (iv) `output:<tunnel-ofport>`. Kernel datapath OVS chuyển 4 action này thành byte: tạo Geneve header với Opt Len = 2 (8 byte TLV), Class = `0x0102`, Type = `0x80`, Length = 2, Value = 8 byte. Cộng với base 8 byte = **16 byte Geneve**, đứng sau UDP header.
2. **Transit qua underlay**: gói Ethernet outer chứa IP outer (src = chassis gửi underlay, dst = chassis nhận underlay) + UDP (src = hash inner flow để ECMP, dst = 6081) + Geneve 16 byte + inner Ethernet frame. Router L3 trên đường thấy đây là UDP đơn thuần, không can thiệp option byte. Nếu underlay có firewall/NAT box drop UDP 6081 → mất gói; nếu có middlebox không hiểu Geneve TLV nhưng cố parse → có thể strip option (rất hiếm).
3. **Decode tại chassis nhận (ingress)**: kernel datapath OVS dùng `vport-geneve` recv hook đọc 16 byte Geneve, populate `tun_id` (VNI) + `tun_metadata0` (TLV value 8 byte) vào skb metadata, đẩy lên flow `table=0` của `br-int`. Flow rule cài bởi `put_remote_port_redirect_*()` thực hiện chuỗi ngược: `move:NXM_NX_TUN_METADATA0[0..31]->NXM_NX_REG15[0..31]` (khôi phục outport), `move:NXM_NX_TUN_METADATA0[16..30]->NXM_NX_REG14[0..14]` (khôi phục inport), `move:NXM_NX_TUN_ID[0..23]->OXM_OF_METADATA[0..23]` (khôi phục datapath), rồi `resubmit(,<egress-table>)` đưa packet vào pipeline egress của datapath đích.

Invariant: tại mọi thời điểm trên dây, **một packet OVN cross-chassis luôn mang đúng một TLV Class `0x0102`**; nếu thấy hai TLV cùng class trên cùng packet thì có bug northd hoặc man-in-the-middle injection. Class `0x0101` (OVS) không bao giờ xuất hiện trong traffic OVN production.

### Phân loại TLV trong dải IANA Geneve Option Class

Class space 16 bit chia thành hai dải theo RFC 8926 + IANA Geneve Option Class registry (xem §11.0.7 đã liệt kê 7 entry mẫu):

| Dải | Procedure | Mô tả |
|-----|-----------|-------|
| `0x0000-0x00FF` | IETF Review | Reserved cho IETF chuẩn hoá tương lai (chưa cấp) |
| `0x0100-0xFEFF` | First Come First Served | Vendor/project đăng ký theo thứ tự, đã có ~30 entry tính đến 2026-04 |
| `0xFF00-0xFFFF` | Experimental Use | Thử nghiệm cục bộ, không guarantee unique |

Mỗi class owner tự quản 16 bit Type (256 type x 32 length-word = không gian rộng). OVN Class `0x0102` hiện chỉ dùng **một type** (`0x80`) cho metadata logical pipeline; các type `0x81..0xFF` reserved cho mở rộng tương lai (nếu OVN cần thêm field như ECMP hint, MAC binding cache cross-chassis). Bit cao nhất của trường Type là **C-bit (Critical)**: nếu set, receiver không hiểu TLV phải drop packet (RFC 8926 §3.5.1). OVN đặt type = `0x80`, tức C-bit = 1, vì metadata này thiết yếu — packet thiếu nó không thể xử lý đúng.

### Cách kỹ sư vận hành thành thạo

Bốn kỹ năng nền tảng kỹ sư phải có với Geneve TLV:

1. **Capture + decode TLV thô từ tcpdump**: `sudo tcpdump -i <phys-iface> -nn -X 'udp port 6081' -c 5` lấy mẫu 5 gói; đọc byte ở offset 42 (sau outer Eth+IP+UDP) là Geneve base, offset 50 là byte đầu của TLV. Nhận diện `01 02` là Class OVN, `80 02` là Type=0x80 + Length=2 (8 byte).
2. **Decode TLV bằng tshark/Wireshark có dissector**: `sudo tshark -i <phys-iface> -Y geneve -V` cho output có sẵn parsed Geneve + TLV; Wireshark 4.0+ có dissector hiển thị "Class: OVN (0x0102), Type: 0x80, Length: 8 bytes".
3. **Kiểm tra cấu hình tunnel + chassis encap**: `ovs-vsctl --columns=name,type,options list Interface | grep -A1 ovn-` xem tunnel port có `type=geneve, options:remote_ip=...`; `ovn-sbctl --columns=name,encaps list Chassis` xem mỗi chassis advertise encap `geneve` không.
4. **Quyết định khi nào dùng skill nào**: nếu cross-chassis ping rớt → bắt đầu bằng skill 1 (capture xem có UDP 6081 không, có TLV không); nếu thấy UDP 6081 nhưng metadata sai → skill 2 (decode tường minh xem outport/inport có khớp expectation); nếu underlay show không có UDP 6081 → skill 3 (cấu hình tunnel sai hoặc firewall block).

### Quy trình sử dụng (workflow kiểm chứng cross-chassis)

Workflow kiểm chứng một cặp chassis truyền TLV đúng gồm sáu bước, mỗi bước có lệnh xác minh:

1. **Xác nhận tunnel UP**: `ovs-vsctl get Interface ovn-<chassis-id>-0 link_state` trả về `up`. Nếu `down` → BFD chưa ổn định, dừng bước này.
2. **Xác nhận encap type Geneve**: `ovn-sbctl --columns=name,encaps list Chassis <chassis-id>` thấy `encaps=[<uuid>]`; sau đó `ovn-sbctl list Encap <uuid>` thấy `type=geneve` (không phải `vxlan` hay `stt`).
3. **Phát một test packet logic**: trên VM chassis gửi `ip netns exec vm1-ns ping -c 1 <vm2-ip>`.
4. **Capture trên physical interface chassis gửi**: `sudo tcpdump -i eth0 -nn -X -c 1 'udp port 6081 and host <peer-underlay-ip>'`. Xác nhận output có byte offset 50: `01 02 80 02` (Class+Type+Length).
5. **Decode value 8 byte sau header TLV**: bit 0..31 phải bằng `tunnel_key` của LSP đích (kiểm tra qua `ovn-sbctl list Port_Binding vm2 | grep tunnel_key`); bit 16..30 phải bằng `tunnel_key` của LSP nguồn dịch sang 15 bit.
6. **Kiểm chứng decode chassis nhận**: trên chassis nhận, `ovs-appctl ofproto/trace br-int <flow-spec>` hoặc `ovs-ofctl dump-flows br-int table=0` thấy flow `tun_id=<key>,tun_metadata0=<value>` match đúng và resubmit vào table 8/16.

Best practice production: **luôn enable BFD trên Geneve tunnel** để phát hiện peer down trong < 1 giây; **set MTU underlay tối thiểu 1550** (1500 inner + 58 byte overhead Geneve theo §11.0.8) hoặc tốt hơn 9000 jumbo. Anti-pattern: chạy underlay MTU = 1500 và inner MTU = 1500 → fragmentation mỗi packet, performance giảm 30-50%.

### Khi xảy ra sự cố (reverse symptom map TLV-centric)

Bảy symptom thường gặp trỏ về Geneve TLV:

| Symptom | Vùng TLV cần kiểm | Diagnostic flow |
|---------|------------------|-----------------|
| "VM chassis A ping VM chassis B fail 100%" | Tunnel state + UDP 6081 reachability | Bước 1+4 workflow trên: tunnel up không, capture có UDP 6081 không |
| "Cross-chassis ping intermittent 5-10% drop" | MTU + fragmentation | Check `ip link show` MTU underlay vs inner; nếu DF bit set + overhead > MTU → PMTU black hole |
| "Cross-chassis OK nhưng Logical_Router routing sai" | TLV decode value bit 0..31 (outport) | Bước 5: decode value, so với Port_Binding.tunnel_key của remote LRP |
| "Multi-tenant traffic crossover (tenant A thấy traffic B)" | TLV decode value bit 0..23 (datapath) | So `tun_id` capture với Datapath_Binding.tunnel_key của hai tenant |
| "Stateful ACL cross-chassis drop reply" | conntrack zone derive từ datapath in TLV | Cross-link 9.24 ct_zone + check zone consistency hai chassis |
| "Live migration mất 30s flow stuck" | TLV outport vẫn point chassis cũ | `ovs-appctl ofproto/trace` xem outport trong TLV match Port_Binding.chassis hiện tại |
| "Wireshark show TLV class lạ thay vì 0x0102" | Class collision với project khác | Capture raw bytes, kiểm có middlebox proprietary inject TLV không |

Cross-link: Phần 13.7 §13.7.8 (`controller/physical.c put_encapsulation()` chi tiết source code), Phần 13.16 (logical pipeline tiêu thụ metadata sau khi decode), Phần 13.17 (register conventions `MFF_LOG_DATAPATH/INPORT/OUTPORT` + bố cục bit trong TLV), Phần 19.0 (multichassis forensic case study), Phần 11.1 (PMTUD failure mode chi tiết do overhead Geneve TLV).

### Liên quan mật thiết tới thành phần khác

Năm thành phần coupling chặt với Geneve TLV (tight, không phải loose cross-link):

1. **VNI base header (`MFF_TUN_ID`)**: VNI 24 bit đi cùng TLV trong cùng Geneve header, mất một thì mất cả; `put_encapsulation()` sinh cả hai trong một flow rule.
2. **OXM `mff_ovn_geneve` (= `tun_metadata0` trên OVS)**: OVN khai báo OXM ID động qua action `tlv-map` lúc startup ovn-controller (xem `controller/encaps.c::ovn_geneve_tlv_register()`); nếu OVS quên register → flow `set_field:tun_metadata0` parser fail → control plane up nhưng data plane silently drop.
3. **`MFF_LOG_INPORT` (reg14) + `MFF_LOG_OUTPORT` (reg15)**: hai register này được encode vào TLV value tại egress, decode ra tại ingress; nếu bố cục bit thay đổi (ví dụ OVN nâng inport lên 24 bit cho cluster lớn) thì cả `put_encapsulation()` và `put_remote_port_redirect_*()` phải đổi đồng bộ.
4. **`Chassis.encaps` + bảng `Encap` SBDB**: row Encap chứa `type=geneve, ip=<underlay-ip>, options:csum=true`; ovn-controller đọc bảng này để chọn tunnel type cho mỗi remote chassis.
5. **Kernel module `openvswitch.ko` + `vport-geneve.c`**: nếu kernel chạy phiên bản cũ (Linux < 4.6) chưa có vport-geneve hoặc thiếu offload → user-space fallback chạy chậm, throughput tunnel chỉ đạt 1-2 Gbps thay vì 10 Gbps.

Loose related (cross-link, không tightly coupled): VXLAN tunnel (alternative encap, không cùng instance), STT (deprecated alternative), NSH service chain (different abstraction layer).

### Sự cố sản xuất (real-world incident anatomy)

**Bối cảnh:** Cụm OpenStack OSP-17 chạy OVN 22.03.4, 80 compute node, 2 controller HA, sử dụng ConnectX-5 NIC, underlay 25 Gbps, MTU underlay 1500 (chưa kịp chuyển jumbo). Cuối tháng 03/2024, monitoring báo VM cross-AZ ping nhau thỉnh thoảng 1-3% rớt gói, nhưng ping trong cùng AZ ổn 0%.

**Triệu chứng:** Người vận hành thấy `tcpdump` phía VM nguồn gửi đi đủ ICMP request nhưng phía VM đích nhận chỉ ~97-99%; không có ICMP error reply. Drop ngẫu nhiên, không tập trung vào thời điểm cụ thể.

**Quy trình điều tra:**

1. **Bước 1**: `ovs-vsctl get Interface ovn-cmpt-23-0 link_state bfd_status` → tunnel UP, BFD ổn. Loại trừ tunnel down.
2. **Bước 2**: `tcpdump -i bond0 -nn 'udp port 6081 and host <peer-underlay>'` capture 1000 gói khi running ping count 1000. Đếm: tcpdump bên gửi 1000 gói UDP 6081, tcpdump bên nhận chỉ 970 gói. **30 gói biến mất giữa underlay.**
3. **Bước 3**: capture có flag DF bit (Don't Fragment) trong outer IPv4 header set; vài gói có size 1532 (vượt MTU 1500 underlay). Switch ToR drop với ICMP unreachable type 4 code 4, nhưng ICMP reply không quay lại được tới chassis gửi vì firewall block ICMP cho subnet underlay.
4. **Bước 4**: tính lại overhead: inner Ethernet 14 + inner IPv4 20 + ICMP 8 + payload 1472 = 1514 inner. Encap thêm Geneve 16 byte (8 base + 8 TLV) + UDP 8 + outer IPv4 20 + outer Ethernet 14 = 58 byte overhead. Tổng wire = 1514 + 58 = 1572. **Vượt MTU 1500 underlay 72 byte.** Một số packet inner < 1442 byte không bị; packet inner ≥ 1442 byte bị fragment fail vì DF.

**Nguyên nhân gốc:** PMTUD black hole do ICMP reply bị firewall underlay drop, kết hợp DF=1 mặc định của Geneve OVN + MTU underlay 1500 không đủ cho overhead 58 byte.

**Cách khắc phục:**

- Khắc phục tạm: giảm inner MTU 1500 → 1442 trên tất cả VM qua DHCP option 26 (`subnet_mask` không, mà là `interface_mtu`). Drop rate 0% sau 5 phút.
- Khắc phục triệt để: nâng underlay MTU 1500 → 9000 trên toàn bộ ToR + spine. Sau 2 tuần migration, MTU inner trả về 1500 default, tận dụng được jumbo frame, throughput VM cross-AZ tăng 1.6x.

**Bài học:** Overhead Geneve TLV không chỉ là 8 byte TLV mà là 16 byte (8 base + 8 TLV) cộng với UDP 8 + outer IPv4 20 + outer Eth 14 = 58 byte tối thiểu. Khi MTU underlay = inner, mọi packet inner full-size sẽ bị drop. Cross-link: Phần 11.1 PMTUD failure modes chi tiết, Phần 19.0 §multichassis forensic.

### Bài lab tổng hợp (synthetic 2-chassis Geneve TLV inspection)

**Mục tiêu:** Verify rằng tunnel_key + inport + outport được encode đúng vào Geneve TLV class `0x0102`, đối chiếu với `controller/physical.c put_encapsulation()` line 340.

**Setup giả định:** 2 chassis `chassis-1` (10.0.0.10) + `chassis-2` (10.0.0.20), mỗi chassis 1 VM (`vm1`, `vm2`) trên cùng Logical Switch `ls1`, datapath tunnel_key = 1, vm1 tunnel_key = 1, vm2 tunnel_key = 2.

**Bước thực hiện:**

```

[chassis-1]# sudo tcpdump -i eth0 -nn -X -c 1 'udp port 6081 and host 10.0.0.20' &
[chassis-1]# ip netns exec ns_vm1 ping -c 1 10.0.0.2

# Tcpdump output:
14:23:10.123 IP 10.0.0.10.43210 > 10.0.0.20.6081: UDP, length 110
0x0028:  0200 6558 0000 0100               # Ver=0, OptLen=2, Proto=0x6558, VNI=1
0x0030:  0102 8002 0000 0002 0001 0000     # Class=0x0102, Type=0x80, Len=2, outport=2, inport=1<<16
0x003c:  fffe ff00 00ff aabb ccdd ee01     # Inner Ethernet dst+src MAC

|--------|-------|---------|------------------|

### Failure mode + diagnostic signal

**Mode A, TLV stripped bởi middlebox lạ:**
- Tín hiệu: capture trên chassis gửi thấy TLV `0x0102` đầy đủ; capture trên chassis nhận thấy Geneve có VNI nhưng Opt Len = 0, không có TLV. Log ovn-controller chassis nhận: `"unknown tun_metadata, falling back to MC_FLOOD"`.
- Khắc phục: xác định middlebox (proprietary firewall, NAT box) trên đường, bypass hoặc whitelist TLV.

**Mode B, Class collision với INT (0x0103) hoặc vendor lạ:**
- Tín hiệu: tshark show packet có 2 TLV cùng lúc, một `0x0102` một class lạ. Hiếm khi xảy ra trong production OVN-only.
- Khắc phục: kiểm tra có project khác đồng cấu hình UDP 6081 trên cùng underlay không; tách subnet underlay riêng cho OVN.

**Mode C, MTU mismatch + DF=1 + ICMP black hole:**
- Tín hiệu: cross-chassis drop ngẫu nhiên 1-5%, packet large fail, packet small pass. Không thấy ICMP unreachable trở lại sender.
- Khắc phục: nâng MTU underlay, hoặc giảm inner MTU 58 byte, hoặc tắt DF (`options:df_default=false` cho phép fragment IPv4).

**Mode D, OXM `tun_metadata0` chưa register lúc OVS startup:**
- Tín hiệu: ovn-controller log `"could not configure tunnel metadata mapping: TLV type %d not registered"`. Flow rule `set_field:tun_metadata0` parse fail. Data plane silently drop cross-chassis.
- Khắc phục: restart `ovn-controller` để re-register OXM qua action `tlv-map`; check OVS version ≥ 2.5 (cũ hơn không hỗ trợ dynamic OXM).

**Mode E, NIC offload checksum sai gây drop receiver:**
- Tín hiệu: ConnectX-5 cũ firmware compute UDP checksum không bao gồm Geneve TLV, gói đến receiver fail UDP checksum, drop trước khi vào datapath. `ethtool -S eth0 | grep csum_err` tăng.
- Khắc phục: update NIC firmware, hoặc tạm tắt UDP TX offload (`ethtool -K eth0 tx-udp-segmentation off`).

### So sánh cross-domain

**VXLAN (RFC 7348):**
- *Tương đồng:* cùng UDP encap (4789 vs 6081), cùng overlay L2-over-L3, cùng dùng VNI 24 bit cho tenant segment.
- *Khác biệt:* VXLAN header cố định 8 byte, không có TLV → không thể truyền inport/outport metadata; OVN dùng VXLAN buộc fall back MC_FLOOD và mất feature stateful cross-chassis. Geneve TLV `0x0102` mở rộng được; VXLAN chỉ có VNI.

**STT (Stateless Transport Tunneling, draft-davie):**
- *Tương đồng:* cũng được Nicira/VMware đề xuất; có 64-bit context đủ chứa metadata.
- *Khác biệt:* STT giả mạo TCP header để piggyback TSO offload NIC cũ, bị nhiều middlebox/NAT từ chối; IETF không chuẩn hoá; OVS 3.0+ remove. Geneve không giả mạo, qua middlebox tốt, là chuẩn IETF chính thức (RFC 8926).

**NSH (Network Service Header, RFC 8300):**
- *Tương đồng:* cũng là extension header truyền metadata cho service chaining trong overlay.
- *Khác biệt:* NSH tập trung **service path identifier** và **service index** cho service function chaining, không phải logical pipeline ingress/egress port. NSH chạy trên VXLAN-GPE hoặc Geneve outer, là layer trên TLV. OVN không dùng NSH; nếu cần SFC dùng OVN ACL với multi-hop chain.

**MPLS labels (RFC 3032):**
- *Tương đồng:* cùng vai trò gắn metadata 20-bit (label) lên packet, cho phép intermediate node tra cứu nhanh.
- *Khác biệt:* MPLS chạy trên L2.5, yêu cầu provider edge router LDP/BGP-LU; không phù hợp cho cloud overlay vì không qua được L3 boundary tự nhiên. Geneve TLV chạy trên L7-of-L4 (UDP), qua bất kỳ underlay L3 nào.

**Cisco VXLAN-GPE Group-Based Policy (GBP):**
- *Tương đồng:* cũng truyền tenant + policy tag bổ sung trên VXLAN.
- *Khác biệt:* GBP yêu cầu ACI fabric chạy bên dưới + APIC controller; không phải open standard. Geneve TLV `0x0102` chạy trên Linux thuần + OVS thuần, không vendor lock-in.

> **Quy tắc:** OVN production → Geneve TLV `0x0102`. VXLAN cho OVN chỉ phục vụ interop với endpoint legacy không hiểu Geneve. STT, NSH, MPLS không phải lựa chọn cho OVN.

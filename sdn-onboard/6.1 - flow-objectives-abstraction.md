## Bối cảnh: tầng trừu tượng trên OpenFlow

Phần 4.5 đã phân tích TTP (Table Type Patterns) như attempt của ONF để bridge giữa spec generic và silicon capability fragmented, nhưng fail sự áp dụng vì timing + complexity + vendor incentive issue. Phần 6.0 đã phân tích P4 như mô hình shift thay thế OpenFlow với programmable data plane. Phần này phân tích **cách tiếp cận thay thế thứ ba**: Flow Objectives của ONOS, giữ OpenFlow làm protocol nhưng add abstraction layer trên controller side.

Triết lý Flow Objectives: thay vì programmer application write raw OpenFlow flow entries (phải know match fields, actions, table numbers, priority), application express **high-level ý đồ cấu hình** như "forward traffic from ingress X to egress Y". ONOS driver layer dịch ý đồ cấu hình sang device-specific flow rules. Driver abstracts hardware heterogeneity từ application code.

Đây là pattern "rising tide" trong software engineering: khi primitive layer too low-level (OpenFlow), introduce abstraction layer to reduce cognitive load. Flow Objectives success trong ONOS ecosystem (CORD, Trellis) but limited outside.

## 6.1.1 Động lực: TTP không khả thi cho triển khai đa vendor

### Xem lại TTP

Phần 4.5 đã mô tả TTP workflow:
1. ONF define reference TTPs (L2-L3 Edge, Aggregation-Core, WAN PE).
2. Switch advertise supported TTP IDs trong FEATURES_REPLY.
3. Controller choose TTP phù hợp application needs.
4. Controller install flow_mod tuân theo TTP schema.

Vấn đề TTP:
- **Fixed catalog**: thêm pattern mới yêu cầu ONF spec revision, slow process.
- **Vendor implementation**: hardware vendor (Broadcom/Cisco/Juniper) không commit implement TTP layer trong silicon/firmware. Chi phí engineering không justify với limited market demand.
- **Application burden**: application programmer vẫn phải biết TTP schema của switch đang dùng, có nghĩa là aware của switch capability. Không fully abstract.

Kết quả TTP: ONF TS-017 v1.0 (15/08/2014) publish nhưng sự áp dụng thực tế gần zero (Phần 4.5.3).

### Cách tiếp cận của nhóm kiến trúc ONOS

ONOS (Open Network Operating System) của ON.Lab 2014 face cùng problem: application build on ONOS cần portable across switch vendors với heterogeneous pipeline. TTP không feasible (phụ thuộc vendor implement). ON.Lab decision: **build abstraction trong ONOS controller itself, không on switch side**.

Đây là Flow Objectives, introduced khoảng 2015 (ONOS 1.x series). Design philosophy:

1. **Application express ý đồ cấu hình ở abstraction cao**: "forward gói tin với X match → Y port". Không specify table ID, priority, match field details.
2. **ONOS framework translate**: framework có compile-time knowledge về switch pipeline (via driver class per device model).
3. **Driver emit device-specific flow**: driver class cho Broadcom OFDPA 2.0 pipeline different với driver cho Corsa, Noviflow, etc. Application không biết difference.

Key insight: **abstraction layer locate ở controller (software, flexible) thay vì switch (hardware/firmware, rigid)**. Driver là code trong ONOS framework; adding new switch model = write new driver class, deploy ONOS update, không touch switch firmware.

## 6.1.2 Ba loại objective: filtering, chuyển tiếp, next

### Mô hình chuyển tiếp nhiều tầng

Flow Objectives define 3 objective types mapping sang 3-layer chuyển tiếp concept:

```
Packet → FilteringObjective (admission) → ForwardingObjective (match logic) → NextObjective (next-hop group) → Egress
```

Mỗi objective type có role cụ thể, tổng hợp tạo complete chuyển tiếp pipeline.

### FilteringObjective

**Role**: "who can enter this pipeline". Admission control ở ingress port.

Tiêu chí khớp:
- VLAN ID (untagged / specific VLAN / priority-tagged).
- MAC address (source, destination).
- IP address optionally.

Operation types:
- **PERMIT**: allow traffic match vào pipeline.
- **DENY**: drop traffic match.

Example:
```
FilteringObjective {
    key: VLAN 100 on port eth0,
    conditions: [MAC=aa:bb:cc:dd:ee:ff allowed],
    operation: PERMIT
}
```

Ý đồ cấu hình: port eth0 chỉ accept VLAN 100 với specific MAC. ONOS driver translate thành:
- OFDPA pipeline: flow install trong VLAN Filtering table (Table 10) và Termination MAC table.
- Corsa pipeline: different table numbers, same logical effect.

### ForwardingObjective

**Role**: "how to forward matched traffic". Core routing/switching decision.

Flavor types:
- **SPECIFIC**: exact match (dst MAC exact, dst IP exact).
- **VERSATILE**: wildcarded match (ACL, classifier with priority).

Match + action:
```
ForwardingObjective {
    flavor: SPECIFIC,
    selector: dl_dst=aa:bb:cc:dd:ee:ff,
    nextId: 42,  // reference to NextObjective
    priority: 100
}
```

Ý đồ cấu hình: match destination MAC, forward qua next-hop group 42. Application không specify output port trực tiếp; output port là responsibility của NextObjective 42.

Driver translate ForwardingObjective:
- OFDPA: install trong Bridging table (Table 50) hoặc ACL table.
- Native Linux kernel NOS: install route qua netlink.

### NextObjective

**Role**: "next-hop group". Define how traffic egress qua single port, multiple ports (ECMP), or failover.

Group types:
- **SIMPLE**: single output action.
- **HASHED**: hash-based ECMP across buckets.
- **FAILOVER**: primary + backup buckets (like OpenFlow FAST_FAILOVER group type).
- **BROADCAST**: all buckets (like ALL group type).

Example:
```
NextObjective {
    id: 42,
    type: HASHED,
    buckets: [
        { output: eth1, MAC rewrite: 00:00:00:00:00:01 },
        { output: eth2, MAC rewrite: 00:00:00:00:00:02 }
    ]
}
```

Ý đồ cấu hình: ECMP across 2 uplinks với MAC rewrite per-bucket.

### Ghép nối nhiều tầng

Application workflow:

```
1. Install NextObjective 42 (next-hop group).
2. Install ForwardingObjective với nextId=42 (route).
3. Install FilteringObjective permit VLAN 100 (admission).
```

Order matters: NextObjective first (backward reference), ForwardingObjective reference NextObjective. FilteringObjective independent.

Hiểu sai thường gặp:

**Hiểu sai:** "Flow Objectives = wrapper thin around OpenFlow flow entries. Application có thể ignore Flow Objectives, use raw OpenFlow directly."

**Thực tế:** Flow Objectives abstract away pipeline fragmentation meaningful. Raw OpenFlow requires application know table number, match field support, action availability per switch vendor. Flow Objectives driver encapsulate this knowledge. Application code viết against Flow Objectives portable; raw OpenFlow code viết for specific pipeline (OFDPA) không portable sang different pipeline (Corsa, BCM native). Difference is architecturally significant for multi-vendor deployment.

## 6.1.3 Tầng driver: ánh xạ objective sang flow của thiết bị

### Trách nhiệm của driver

ONOS driver class per switch model (hoặc per pipeline family). Driver responsibility:

1. **Pipeline knowledge**: encode silicon/firmware pipeline structure (table numbers, valid match fields per table, action support per stage).
2. **Objective translation**: convert FilteringObjective / ForwardingObjective / NextObjective → specific OpenFlow flow entries.
3. **Group management**: create/update/delete OpenFlow groups corresponding to NextObjective.
4. **Error mapping**: translate device-specific errors (OFPERR_BAD_MATCH on Table 42) back to application-level diagnostics.

### Ví dụ driver: OFDPA 2.0

OFDPA (OpenFlow Data Plane Abstraction) là Broadcom's pipeline specification for their switch silicon. OFDPA 2.0 pipeline (simplified):

| Table | Purpose |
|-------|---------|
| 0 (Ingress Port) | Port classification |
| 10 (VLAN) | VLAN processing |
| 20 (Termination MAC) | MAC address recognition (for routing) |
| 30 (Unicast Routing) | Unicast L3 chuyển tiếp |
| 50 (Bridging) | L2 bridging |
| 60 (Policy ACL) | ACL |
| 80 (Egress VLAN) | Output VLAN processing |

ONOS OFDPA driver translate FilteringObjective:
- PERMIT VLAN 100 on port eth0 → install flow trong Table 10 (VLAN) với match `in_port=eth0, vlan_vid=100`, action = goto_table:20.

Translate ForwardingObjective (routing):
- Route 10.0.0.0/24 via next-hop group 42 → install flow trong Table 30 (Unicast Routing) với match `eth_type=0x0800, ipv4_dst=10.0.0.0/24`, action = group:42.

Translate NextObjective:
- Create OpenFlow group 42 (type=select), buckets = uplink ports.

All internal to driver. Application just calls `ForwardingObjective.Builder ... .build()` và submit to FlowObjectiveService.

### Danh mục driver trong ONOS

ONOS ship drivers cho:
- **OFDPA** (Broadcom): various versions (OFDPA 1.0, 2.0, 3.0).
- **Corsa**: Corsa Red series switch.
- **Noviflow**: Noviflow programmable switch.
- **OpenFlow standard**: fallback for generic OpenFlow switch (limited feature).
- **P4Runtime**: P4-capable switch qua P4Runtime API.

Driver list tại `https://docs.onosproject.org/` + source code `https://github.com/opennetworkinglab/onos/tree/master/drivers`.

### Mô hình phát triển driver

Adding new switch model:
1. Fork ONOS repo, add driver class extending `Pipeliner` abstract class.
2. Implement `filter()`, `forward()`, `next()` methods.
3. Register driver với device manager via `@Component` annotation.
4. Test against switch (interoperability testing).
5. Submit PR upstream.

Cycle time: 2-4 weeks for standard pipeline (e.g., new Broadcom OFDPA variant). 3-6 months for novel pipeline (e.g., new Noviflow P4-based switch). Much faster than TTP (years) or spec revision (years).

## 6.1.4 Current state: sự áp dụng hẹp ngoài ONOS

### Trellis SD-Fabric

Flow Objectives thành công rõ nhất trong **Trellis** project: ONF's open-source SD-Fabric for leaf-spine datacenter. Trellis use:
- Underlay: leaf-spine IP fabric with BGP unnumbered.
- Control plane: ONOS running Flow Objectives programming Broadcom OFDPA switches.
- Application: **SEBA** (SDN Enabled Broadband Access) for telco central office PON (Passive Optical Network) virtualization.

Trellis production deployment: Comcast, AT&T Central Office re-architected as Datacenter (CORD) initiative 2016-2020. Small-scale but production.

Trellis documentation: `https://opennetworking.org/trellis/`.

### CORD (Central Office Re-architected as Datacenter)

CORD là ONF telco initiative để replace traditional Central Office với general-purpose DC infrastructure running virtualized telco functions. Components:
- Compute: OpenStack hoặc Kubernetes.
- Underlay: Trellis (switch + ONOS + Flow Objectives).
- VNF (Virtual Network Functions): virtualized BNG, vCSR, vOLT.

CORD documentation: `https://opennetworking.org/cord/`.

### Sự chấp nhận ngoài ONOS

**OpenDaylight**: không adopt Flow Objectives. ODL apps write raw OpenFlow flow tables directly. Rationale: ODL's multi-SB architecture (not just OpenFlow but NETCONF, BGP-LS, gNMI) pre-dates Flow Objectives abstraction.

**Ryu, POX, Faucet**: legacy first-generation controllers, pre-date Flow Objectives. Write raw OpenFlow.

**Commercial vendor controllers** (Cisco ACI, Juniper Contrail): use native proprietary API, không use OpenFlow standard. Flow Objectives irrelevant.

**Kubernetes SDN** (OVN-Kubernetes, Calico, Cilium): use native constructs (OVN logical flow, BGP, eBPF), không adopt Flow Objectives.

### Evaluation: abstraction không phải silver bullet

Flow Objectives cho phép portable app across Broadcom OFDPA + Corsa + Noviflow trong ONOS ecosystem. Nhưng:

**Limitation 1**: chỉ portable trong ONOS driver catalog. Switch không có driver = không support. Coverage reality, ONOS driver catalog limited to ~10 switch models.

**Limitation 2**: abstraction leak. ForwardingObjective có flavor SPECIFIC vs VERSATILE là implicit về silicon constraint. Programmer vẫn cần biết pipeline structure để optimize.

**Limitation 3**: không thay thế P4. P4 solve fragmentation bằng programmable data plane (compile down); Flow Objectives solve bằng abstraction at controller (translate at runtime). P4 more powerful long-term but requires silicon support; Flow Objectives ship today with existing silicon.

### Hiểu sai thường gặp

**Hiểu sai:** "Flow Objectives là ONOS-specific feature, nên không phải standard."

**Thực tế:** Flow Objectives là design pattern (layered abstraction trên OpenFlow), applicable to any controller. OpenDaylight choose không adopt là strategic decision, không phải Flow Objectives không work ngoài ONOS. Pattern có thể port sang ODL hoặc custom controller. "ONOS-specific" là implementation reality, không phải design constraint.

> **Chủ đề trọng tâm:** Flow Objectives thể hiện pattern "abstraction in controller, not in switch" phù hợp với SDN economic reality. Switch firmware update expensive, vendor không incentivized push complexity vào switch (TTP fail minh chứng). Controller abstraction layer cheap to iterate, single codebase change update all users. Flow Objectives là pragmatic middle path giữa raw OpenFlow (application burden) và P4 (silicon burden). Bài học: khi design abstraction, locate complexity ở layer flexible nhất, không layer rigid nhất.

# 3.4. OpenFlow version differences — migration matrix 1.0 → 1.3 → 1.5

> **Môi trường thực hành:** Ubuntu 22.04 LTS, OVS 2.17.9, OVN 22.03.8.
> **Khối:** III, OpenFlow birth (Phần 3, bổ sung depth tier 2).
> **Plan:** `plans/keyword-backbone-v3.5-plan.md` §5.3 — file mới Phase J.4.c commit 2/2.
> **Prerequisites:** Phần 3.1 (OF 1.0 spec), Phần 3.3 (OF protocol messages), Phần 4.0-4.5 (OF 1.1-1.5 individual evolution).
> **Nguồn upstream chính:**

> - ONF OpenFlow Switch Specification 1.3.5 (Wire 0x04, Mar 2015)
> - ONF OpenFlow Switch Specification 1.5.1 (Wire 0x06, Apr 2015)

## Mục tiêu bài học

Sau khi hoàn thành phần này, kỹ sư có thể:

1. **Hiểu (Understand)** lịch trình phát hành OpenFlow + lý do mỗi major version (1.0 → 1.5) thêm tính năng cụ thể nào, dựa trên phản hồi từ deployment thực tế.
2. **Phân tích (Analyze)** một flow rule viết theo OF 1.0 syntax và xác định có thể migrate trực tiếp lên OF 1.3 hay cần rewrite (nếu có goto_table, group, meter, copy_field, packet_type-aware, hoặc egress table).
3. **Áp dụng (Apply)** migration matrix khi nâng cấp switch + controller từ OF 1.0 lên OF 1.3 hoặc OF 1.5; biết rule nào break, rule nào tự compatible.
4. **Đánh giá (Evaluate)** decision tree "version nào cho deployment cụ thể": OF 1.0 cho legacy testbed, OF 1.3 cho production OVN (mặc định), OF 1.5 cho SFC/NSH research.
5. **Tạo (Create)** rule rewrite checklist cho team migration, document rõ tại sao mỗi rule cần modify.

Yêu cầu kiến thức: hiểu OF 1.0 spec qua Phần 3.1, hiểu protocol message qua Phần 3.3, có chạy OVS với `--protocols=`.

## 3.4.1 OpenFlow version timeline

| Version | Wire | Date | Major addition |
|---------|------|------|----------------|
| **1.0** | 0x01 | Dec 2009 | Initial release, single table 12-tuple match, 8 actions |
| 1.1 | 0x02 | Feb 2011 | **Multi-table pipeline (254 table)**, group table, MPLS support, VLAN tagging actions |
| 1.2 | 0x03 | Dec 2011 | **OXM TLV match** (replace NXM/legacy), IPv6 support, controller role messages |
| **1.3** | 0x04 | Apr 2012 | **Meters** (rate-limit), per-flow auxiliary connections, IPv6 extension headers, refactored capabilities. **LTS version, baseline cho hầu hết deployment hiện tại.** |
| 1.4 | 0x05 | Aug 2013 | **Bundle** (atomic transaction), flow monitoring, vacancy events, optical port |
| **1.5** | 0x06 | Apr 2015 | **Egress tables**, **copy_field** action, **packet-type-aware pipeline**, scheduled bundles |

**Lưu ý quan trọng:**
- OF 1.0 và OF 1.3 là 2 version dùng phổ biến nhất trong production. OF 1.1 và 1.2 ít được switch hardware support, hầu hết deployment skip qua thẳng từ 1.0 lên 1.3.
- OF 1.4 và 1.5 chủ yếu cho research + advanced use case (SFC, NSH). Production OVN deployment dùng OF 1.5 cho egress table feature.

**OVS version support matrix:**

| OVS version | OF support |
|-------------|------------|
| OVS 1.x (2009-2012) | OF 1.0 only |
| OVS 2.0-2.3 | OF 1.0 + 1.1 + 1.2 (partial) + 1.3 |
| OVS 2.4-2.6 | OF 1.0 - 1.4 |
| OVS 2.7-2.16 | OF 1.0 - 1.5 (NXM extension) |
| OVS 2.17+ (curriculum baseline) | OF 1.0 - 1.5 full + NXM extension + experimenter |

## 3.4.2 Single-table vs Multi-table pipeline (OF 1.0 → 1.1+)

**Khác biệt cốt lõi:** OF 1.0 chỉ có **table 0**. Mỗi packet match một flow trong table 0, execute action set, kết thúc. Không có cách nào chain multiple stage logic.

**OF 1.0 example:**
```
table=0,priority=100,in_port=1,actions=output:2
```
Một rule = một quyết định cuối.

**OF 1.1+ example:**
```
table=0,priority=100,in_port=1,actions=goto_table:1
table=1,priority=50,eth_type=0x0800,actions=goto_table:2
table=2,priority=50,actions=output:2
```
Rule trong table 0 không output ngay, mà jump sang table 1 cho stage tiếp. Cho phép modular composition (port-security ở table 0, ACL ở table 1, QoS ở table 2, forwarding ở table cuối).

**Migration impact:** rule OF 1.0 không có `goto_table`. Migrate lên 1.3 cần:
1. **Identity migration (đơn giản nhất):** giữ nguyên rule trong table 0, không dùng goto_table → tương đương 1.0 behavior.
2. **Refactor migration:** chia rule theo logical stage, dùng goto_table chain. Lợi ích: classifier OVS hoạt động hiệu quả hơn (mỗi table ít priority distinct hơn).

**Lý do thiết kế multi-table:** OF 1.0 deployment thực tế thấy single-table không scale. Khi rule cần combine 5 logic (in_port + ACL + QoS + L3 + output), cardinality rule = sản phẩm Cartesian → table size explode. Multi-table giảm từ O(N×M) xuống O(N+M).

## 3.4.3 NXM vs OXM encoding (OF 1.0-1.1 → 1.2+)

**Match field encoding** là cách packet header field được serialize trong FLOW_MOD wire format.

**OF 1.0:** dùng `struct ofp_match` cố định 40 byte với 12 field hard-coded (in_port, dl_src, dl_dst, dl_vlan, dl_vlan_pcp, dl_type, nw_tos, nw_proto, nw_src, nw_dst, tp_src, tp_dst). Wildcards 32-bit bitmap. Rigid: muốn match field mới (ARP, IPv6) = không thể không break wire format.

**OF 1.0-1.1 + Nicira extension:** **NXM (Nicira eXtensible Match)** giải quyết bằng TLV format. Mỗi field có 16-bit class + 8-bit type + 1-bit hasmask + 7-bit length. Cho phép thêm field mới (NXM_NX_TUN_ID, NXM_NX_REG0-15) mà không break.

**OF 1.2+:** **OXM (OpenFlow eXtensible Match)** standardize TLV format trong spec chính. Class 32-bit (16-bit experimenter + 16-bit type). Một số class:
- `OXM_OF_BASIC` (0x8000): standard field (in_port, eth_src, ipv4_src, etc.)
- `OXM_OF_PKT_REG` (0x0001): packet register OF 1.5+
- `OXM_NXM_NX` (0x8001): Nicira extension (cho compat OVS)

**Wire format diff:**
- NXM: 16+8+1+7 = 32-bit header per field
- OXM: 32+8+1+7 = 48-bit header per field

**Migration impact:**
- OVS auto-convert nội bộ giữa NXM và OXM, controller không cần care.
- Controller programming chỉ cần dùng `--protocols=OpenFlow13` thay vì `OpenFlow10`, OVS handle wire format conversion.

## 3.4.4 Group tables (OF 1.1+)

**Vấn đề:** OF 1.0 không có cấu trúc nào cho multicast / load balance. Muốn replicate packet ra 10 port:
```
table=0,priority=100,in_port=1,actions=output:2,output:3,output:4,output:5,output:6,output:7,output:8,output:9,output:10,output:11
```
Single action list dài, classifier không tối ưu, khó update (thay 1 port = rewrite cả rule).

**OF 1.1+ giải pháp:** group table riêng biệt từ flow table. Một group có nhiều **bucket**, mỗi bucket có action list riêng.

**4 group types:**

| Type | Mục đích | Use case |
|------|----------|----------|
| `OFPGT_ALL` | Replicate packet ra MỌI bucket | Multicast, broadcast |
| `OFPGT_SELECT` | Hash-select 1 bucket | Load balance |
| `OFPGT_INDIRECT` | 1 bucket duy nhất | Alias group |
| `OFPGT_FF` (Fast Failover) | Chọn bucket đầu tiên có liveness UP | HA active-backup |

**Example OF 1.1+:**
```
group_id=1,type=all,bucket=actions=output:2,bucket=actions=output:3,bucket=actions=output:4
table=0,priority=100,in_port=1,actions=group:1
```

Update group = chỉ thay bucket list, flow rule không đổi.

**Fast Failover sử dụng:**
- Mỗi bucket có `watch_port` hoặc `watch_group` chỉ định resource theo dõi.
- Group emit packet ra bucket đầu tiên có resource liveness UP.
- Khi resource down, group tự động fall sang bucket tiếp theo.
- HA active-backup không cần controller intervention.

## 3.4.5 Meters (OF 1.3+)

**Vấn đề:** OF 1.0-1.2 không có cấu trúc rate limiting standard. OVS giải pháp tạm: dùng `tc qdisc` Linux, hoặc Nicira extension `set_queue` + ingress policer. Không portable giữa vendor.

**OF 1.3 giải pháp:** **meter table** riêng biệt với band-based rate limit.

**Meter components:**
- `meter_id`: 32-bit ID
- `flags`: rate đơn vị (`OFPMF_KBPS`, `OFPMF_PKTPS`), burst support, stats collection
- **Bands list** (mỗi meter ≥ 1 band): mỗi band có:
  - `type`: `OFPMBT_DROP` (drop packet) hoặc `OFPMBT_DSCP_REMARK` (re-mark DSCP)
  - `rate`: kbps hoặc pps
  - `burst_size`: burst tolerance

**Example OF 1.3+:**
```
meter=1,kbps,burst,band=type=drop,rate=5000,burst_size=1500
table=0,priority=100,in_port=1,actions=meter:1,output:2
```

Rule này: traffic vào port 1 → apply meter 1 (limit 5 Mbps + burst 1.5 KB) → output port 2. Vượt rate = drop.

**OF 1.4 thêm:** band features query (`OFPMP_METER_FEATURES`) để controller biết switch support band type nào.

## 3.4.6 Bundles (OF 1.4+)

**Vấn đề:** OF 1.0-1.3 mỗi FLOW_MOD apply ngay. Một sequence 100 FLOW_MOD: nếu rule 50 fail (table full, syntax sai), rule 1-49 đã apply → switch ở trạng thái nửa-vời (partial state).

**OF 1.4 giải pháp:** **OFPT_BUNDLE_OPEN/COMMIT/ADD_MESSAGE** atomic transaction. Tất cả-hoặc-không apply.

Xem chi tiết Phần 3.3 §3.3.8.

**Tại sao quan trọng cho OVN:** ovn-controller restart phát hiện stale flow trên br-int. Thay vì delete-then-add (gây gap traffic), dùng BUNDLE atomic.

## 3.4.7 Egress tables (OF 1.5+)

**Vấn đề:** OF 1.0-1.4 chỉ có ingress pipeline. Action set execute lúc packet rời pipeline, ngay trước output. Không có cách áp dụng "per-port output processing" — ví dụ remove VLAN tag chỉ cho 1 số port nhất định.

**OF 1.5 giải pháp:** **egress pipeline** optional, chạy SAU ingress pipeline + action set, TRƯỚC actual output. Egress table có range 0-254 riêng, không trùng ingress.

**Example use case:** 1 logical switch có 10 port. 7 port output untagged, 3 port output với VLAN tag 100. OF 1.0-1.4 cách:
- 10 rule ở stage cuối, mỗi rule có action set khác nhau (push_vlan cho 3 port, không cho 7 port).

OF 1.5 cách:
- 1 rule chung output ra LSP.
- Egress table table 0: nếu out_port ∈ {7, 8, 9}, push_vlan:100.

Lợi ích: flow rule giảm từ 10 còn 1 + 1 egress rule. Maintenance dễ hơn.

## 3.4.8 `copy_field` action (OF 1.5+)

**Vấn đề:** OF 1.0-1.4 muốn copy bit từ field này sang field kia phải dùng OVS extension `move` (NXM-only, không portable).

**OF 1.5 giải pháp:** standardize **copy_field** action.

**Syntax:**
```
copy_field(src_offset, dst_offset, n_bits, src_field, dst_field)
```

**Example:**
```
actions=copy_field(0:ipv4_src,0:reg0,32)
```
Copy 32 bit từ ipv4_src (offset 0) sang reg0 (offset 0). Sau đó downstream có thể dùng reg0 làm scratch.

**Lý do quan trọng:** OVN dùng register làm "scratch state" giữa stage (xem Phần 13.17). copy_field cho phép chương trình hóa pipeline mà không cần extension proprietary.

## 3.4.9 Packet-type-aware pipeline (OF 1.5+)

**Vấn đề:** OF 1.0-1.4 giả định mọi packet có Ethernet header. SFC (Service Function Chaining) qua NSH (Network Service Header, RFC 8300) cần process packet trần (không Ethernet). OF 1.0-1.4 không support.

**OF 1.5 giải pháp:** **packet_type field** match cho non-Ethernet encapsulation.

**packet_type values:**
- `(0, 0)` Ethernet (default)
- `(0, 0x8847)` MPLS unicast
- `(0, 0x86dd)` IPv6
- `(1, 2)` IPv4 bare (no Ethernet)
- `(1, 0x894f)` NSH
- `(1, ...)` other custom

**Use case OVN:** cho phép NSH-encapsulated packet đi qua br-int mà không cần phải wrap Ethernet giả.

## 3.4.10 Anatomy: `ovs-ofctl --protocols` negotiation

**Bucket:** OpenFlow | **Context:** verify version negotiation | **Purpose:** xác định version OVS đang serve trên một bridge | **Activity:** debug | **Mechanism:** OVS Bridge.protocols column + HELLO bitmap exchange.

**Configure version trên OVS:**

```bash
# Set bridge serve OF 1.3 + 1.5
ovs-vsctl set Bridge br0 protocols=OpenFlow13,OpenFlow15

# Verify
ovs-vsctl get Bridge br0 protocols
# → [OpenFlow13, OpenFlow15]
```

**Connect với version cụ thể:**

```bash
# Force version 1.3
ovs-ofctl -O OpenFlow13 dump-flows br0

# Force version 1.5
ovs-ofctl -O OpenFlow15 dump-flows br0

# Auto-negotiate (highest common)
ovs-ofctl dump-flows br0
```

**Anatomy 5-attribute decode HELLO bitmap:**

| Attribute | Giá trị mẫu | Healthy / Warning / Critical |
|-----------|-------------|------------------------------|
| Bridge.protocols | `[OpenFlow13, OpenFlow15]` | Healthy: ≥ 1 version. Critical: empty = không accept connection |
| Wire bitmap (controller side) | 0x70 (1.3+1.4+1.5) | Critical: nếu intersect rỗng với bridge → connection close |
| Negotiated version | OpenFlow13 (highest common) | Healthy: cao nhất hợp lý |
| `ovs-ofctl show` | List version supported | Verify |
| HELLO_FAILED OFPT_ERROR | Không nên xuất hiện | Critical: version mismatch hoàn toàn |

## 3.4.11 Migration matrix — OF 1.0 → 1.3 → 1.5

### OF 1.0 → 1.3 rule rewrite

**Change required:**

| OF 1.0 syntax | OF 1.3 equivalent | Note |
|---------------|-------------------|------|
| `priority=100,in_port=1,actions=output:2` | Same (compatible) | Identity migration OK |
| `priority=100,in_port=1,actions=output:2,output:3,output:4` | Tốt nhất: dùng group `type=all` | Multicast efficiency |
| `actions=mod_dl_src:aa:bb:cc:dd:ee:ff` | `actions=set_field:aa:bb:cc:dd:ee:ff->eth_src` | OXM action format |
| `actions=set_vlan_vid:100` | `actions=push_vlan:0x8100,set_field:0x1064->vlan_vid` | Explicit push_vlan |

**Auto-compatible (no rewrite needed):**
- Single-table flow rule với basic 12-tuple match
- Output to physical port
- Drop (empty action list)

**Manual rewrite needed:**
- Multicast (use group)
- Action set order matters (OF 1.3 strict order)
- Field-modification action (mod_* → set_field)

### OF 1.3 → 1.5 incremental migration

**New features (chỉ thêm, không break 1.3):**
- Egress table (optional)
- copy_field action (replace OVS NXM `move`)
- packet_type field (cho NSH/SFC)

**Auto-compatible:**
- Mọi 1.3 rule đều run trên switch 1.5.

**Migration choice:**
- Dùng egress table khi có per-port output processing (savings rule count).
- Dùng copy_field khi cross-vendor (move là OVS-only).
- Dùng packet_type cho NSH/SFC research.

## 3.4.13 Capstone POE — sinh viên đề xuất migration plan

**Bài toán:** legacy production switch chạy OVS 2.5 + OF 1.0. Team muốn upgrade lên OVS 2.17 + OF 1.3 để dùng OVN. Đề xuất migration plan.

**Predict:** giả thiết "tất cả flow rule 1.0 hiện tại run được trên 1.3 mà không break".

**Observe:** audit flow table production:

```bash
# Trên switch cũ
ovs-ofctl -O OpenFlow10 dump-flows br0 > /tmp/of10-flows.txt

# Phân loại rule
grep -E 'mod_dl_|set_vlan' /tmp/of10-flows.txt | wc -l
# → N rule cần rewrite (mod_* action)
```

**Explain:**
- N rule không break trực tiếp vì OF 1.3 backward compat cho mod_* (vẫn hiểu và convert sang OXM internal).
- Tuy nhiên, nếu controller mới (OVN) tạo rule theo OXM standard, hai stack co-exist trên cùng switch sẽ confusing log.
- Migration plan: rewrite hết về OXM trước khi switch controller.

**Falsification test:**
- Chạy OVS 2.17 với cờ `--protocols=OpenFlow10` → verify rule 1.0 run nguyên không break.
- Chạy với `--protocols=OpenFlow10,OpenFlow13` → verify cả hai version coexist.
- Switch sang `--protocols=OpenFlow13` only → verify rule 1.0 vẫn dump được (auto-convert) nhưng controller mới gửi rule OXM.

## 3.4.14 Mẫu cần tránh và lưu ý

**Mẫu cần tránh 1: Mix `--protocols` cho cùng deployment.**

`Bridge.protocols=[OpenFlow10, OpenFlow13]` cho phép coexist 2 version. Nhưng controller A dùng 1.0 rule, controller B dùng 1.3 rule trên cùng bridge → behavior unpredictable (priority overlap, action set conflict).

**Cách phòng:** chuẩn hóa toàn deployment về 1 version chính, version cũ chỉ dùng cho debug ad-hoc.

**Mẫu cần tránh 2: Dùng OVS NXM extension trong code production multi-vendor.**

NXM only support OVS. Nếu deploy hardware switch (Pica8, Centec) thì NXM rule fail. Dùng OXM standard (`set_field` thay `mod_*`, `copy_field` thay `move`).

**Lưu ý 3: OF 1.5 egress table có overhead khi không cần.**

Bridge config egress table = mọi packet đi thêm pipeline egress dù không có rule. Overhead ~1-2% CPU. Chỉ enable khi thật sự dùng feature egress.

**Lưu ý 4: Bundle (OF 1.4+) có size limit.**

OFPT_BUNDLE_ADD_MESSAGE max 1 message per add. Bundle quá lớn (>1000 message) có thể vượt switch buffer → bundle commit fail. Chia nhỏ bundle nếu cần update batch lớn.

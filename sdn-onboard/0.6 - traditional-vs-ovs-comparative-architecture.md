## 0.6.1. The hourly sketch every traditional engineer carries

A traditional Cisco engineer's mental model of a layer-2-or-3 switch is
a chassis with line cards, an ASIC fabric, an IOS process running on
the supervisor, and a CLI that drives both. Every primitive (MAC
learning, VLAN tagging, spanning tree, etc.) is implemented in silicon
and exposed through configuration commands that the IOS process
translates into ASIC microcode. The architecture is monolithic: one
vendor's hardware, one vendor's software, one CLI, one configuration
language. The price for this is closed-source proprietary internals;
the operator cannot read the code and cannot adapt the device beyond
what the vendor exposes.

Open vSwitch's architecture is the opposite. The fast path is the
Linux kernel datapath module (`openvswitch.ko`) doing simple
flow-table lookup plus action execution. The control plane is a
userspace daemon (`ovs-vswitchd`) that compiles OpenFlow rules and
OVSDB configuration into kernel datapath flows. The configuration
plane is a database server (`ovsdb-server`) that stores the user's
desired state in transactional form. Each component is open source,
each can be replaced or audited, and each is configurable through
multiple interfaces (CLI, OpenFlow controller, programmatic libraries).

The same eleven networking primitives that a Cisco switch implements
in silicon, OVS implements either in the kernel datapath, in
userspace, or in collaboration with adjacent kernel subsystems
(bridge module, conntrack, tc, qdisc). The comparative-architecture
diagram below maps each primitive to its OVS implementation site:

```ascii
   Concept                Cisco / Juniper                OVS implementation
   -------                ---------------                ------------------
   L2 learning             ASIC FDB                       userspace mac_learning struct (lib/mac-learning.c)
   VLAN access/trunk       ASIC + 802.1Q                  ofproto-dpif-xlate xvlan_input/output_translate
   STP/RSTP                IOS protocol stack            lib/stp.c, lib/rstp.c (userspace state machines)
   Link aggregation        ASIC + IOS LACP              ofproto/bond.c + lib/lacp.c
   Port mirroring (SPAN)   ASIC                          ofproto/ofproto-dpif-mirror.c (userspace)
   L3 routing              ASIC FIB + IOS routing       (kernel namespace OR raw OpenFlow flows; see R2.5.6)
   Stateless ACL           ASIC TCAM                     OpenFlow flow rules (drop/permit by priority)
   Stateful ACL + NAT      ASIC TCAM + state             nf_conntrack via ct() OpenFlow action
   QoS shaping             ASIC scheduler               Linux tc HTB (programmed via OVSDB QoS table)
   DHCP responder          IOS process                   (none; external dnsmasq or OVN required)
   ARP responder           IOS IP stack                  (none; flow-based responder via OF 1.5+)
   Flow export             IOS NetFlow                   ofproto/ofproto-dpif-{sflow,ipfix}.c, ofproto/netflow.c
```

The diagram surfaces three patterns. First, the **purely-OVS-internal
primitives** (L2 learning, VLAN, STP, bonding, mirroring, ACLs,
flow export): OVS implements these natively in its userspace and
delegates execution to the kernel datapath via flow rules. Second,
the **OVS-plus-kernel-collaboration primitives** (conntrack/NAT, QoS):
OVS reuses existing Linux kernel modules (nf_conntrack, sch_htb)
through OpenFlow action wrappers (`ct()`, `ingress_policing_rate`,
the `QoS` OVSDB table). Third, the **gap primitives** (L3 routing,
DHCP, ARP responder): OVS does not implement these natively; the
operator must use external services or OVN.

## 0.6.3. Architectural plane separation

A traditional switch has three planes: management, control, data.
Cisco IOS conflates the three into the IOS process plus the ASIC.
OVS separates them more cleanly:

| Plane         | Cisco location           | OVS location                                     |
|---------------|--------------------------|--------------------------------------------------|
| Management    | IOS CLI on supervisor    | OVSDB (ovsdb-server), CLI ovs-vsctl, ovs-appctl |
| Control       | IOS routing/STP/etc.     | ovs-vswitchd userspace flow translator + agents  |
| Data (fast path) | ASIC microcode        | Linux kernel datapath (openvswitch.ko)           |
| Data (slow path) | (CPU on supervisor)   | userspace upcall handler in ovs-vswitchd         |

The OVS three-tier separation makes each tier individually
swappable. The OVSDB server can run remotely (multi-host clusters via
Raft, covered in Block 10). The userspace daemon can be replaced
(experimental alternatives exist). The kernel datapath can be replaced
by the userspace netdev datapath (DPDK plus dpdk-vswitchd) without
changing the higher tiers. This modularity is what makes OVS suitable
for SDN controllers (which typically program the control plane via
OpenFlow); a Cisco device's IOS is monolithic and not similarly
swappable.

## 0.6.4. Failure-isolation model

A traditional Cisco device has a single failure domain: the chassis.
A supervisor failure (process crash, reboot) takes the whole switch
offline. Modern Cisco platforms have redundant supervisors (SSO -
Stateful Switchover) but the failure model is still chassis-scoped.

OVS has a more nuanced failure model. The kernel datapath fails with
the kernel (a kernel panic takes everything down; the bridge state is
lost). The `ovs-vswitchd` userspace daemon can fail independently;
when it does, the kernel datapath continues forwarding existing flows
(megaflow cache survives) but cannot install new flows; new
connections fail until the daemon restarts. The `ovsdb-server`
process can fail independently from `ovs-vswitchd`; when it does,
the daemon continues with its last-known configuration but cannot
accept new configuration. A multi-host OVSDB cluster (Block 10)
spreads the management plane across nodes for high availability.

The failure-mode comparison:

| Failure                         | Cisco IOS impact                                | OVS impact                                       |
|---------------------------------|-------------------------------------------------|--------------------------------------------------|
| ASIC / kernel-datapath failure  | Switch down                                     | Bridge down (kernel panic affects whole host)    |
| IOS process / vswitchd failure  | Switch down (no new flows; existing TCAM stays) | Existing megaflow cache survives; new flows fail |
| Management-CLI failure          | (rare; same process as IOS)                     | OVSDB down, but data plane continues             |
| Inter-supervisor sync failure   | SSO event, brief disruption                     | OVSDB Raft re-election, brief management-plane disruption |

The OVS model is more resilient against isolated failures (data plane
survives many control-plane outages) but more complex to debug
(three processes to monitor, three sets of failure conditions).

## 0.6.5. Observability surface

A traditional Cisco operator's observability commands include:

- `show interfaces`: per-port counters and state
- `show mac address-table`: FDB
- `show vlan`: VLAN membership
- `show spanning-tree`: STP/RSTP state
- `show etherchannel summary`: bond state
- `show ip route`: FIB
- `show ip arp`: ARP cache
- `show access-lists`: ACL counters
- `show monitor`: SPAN sessions
- `show flow exporter`: NetFlow agent state
- `show interfaces policy`: QoS policy stats

The corresponding OVS commands:

- `ovs-vsctl list interface`: per-port counters and state (read from OVSDB)
- `ovs-appctl fdb/show <bridge>`: FDB
- `ovs-vsctl --columns=tag,trunks,vlan_mode list port`: VLAN config
- `ovs-appctl rstp/show <bridge>`, `stp/show`: spanning tree
- `ovs-appctl bond/show <port>`, `lacp/show`: bond + LACP state
- (no native FIB; use `ip route show` in router namespace)
- (no native ARP cache; use `ip neigh show` in router namespace)
- `ovs-ofctl dump-flows <bridge>`: flow rules + per-rule packet/byte counters
- `ovs-vsctl list mirror`: mirror config + statistics
- `ovs-vsctl list sflow`, `list netflow`, `list ipfix`: flow-export state
- `tc class show dev <interface>`: QoS / HTB state

The OVS observability is split across more tools because OVS itself
does not own all the planes (L3 belongs to the kernel routing
subsystem, QoS to tc, etc.). The traditional Cisco operator must
learn this multi-tool pattern; once mastered, the introspection is
just as comprehensive.

## 0.6.6. Production-readiness comparison

The R2.5 sub-captures consistently revealed four production
considerations that distinguish OVS from a traditional Cisco device:

1. **Software shaping has CPU cost.** Cisco's QoS in silicon scales
   to line rate at near-zero CPU cost. OVS QoS via `tc` is
   software shaping; for high-traffic ports (10 Gbit/s+), the CPU
   cost is non-trivial. Mitigation: hardware-offloaded `tc-flower`
   on supported NICs.

2. **No built-in IP services.** Cisco devices have built-in DHCP
   relay, ARP responder, IP routing protocols (OSPF, BGP). OVS has
   none of these natively. The operator must add external services
   (dnsmasq, FRR, BIRD) or use OVN. R2.5.10 made this gap concrete.

3. **Failure-mode complexity.** A Cisco failure is one process going
   down. An OVS failure is one of three processes going down (kernel
   module, ovs-vswitchd, ovsdb-server) plus the namespace's IP stack
   if applicable. Operators must understand all three to triage
   issues. The benefit is finer-grained survival (existing flows
   continue when ovs-vswitchd alone fails).

4. **Open-source auditability.** The biggest production advantage of
   OVS over Cisco: every line of code is readable. R2.5 cited
   specific source files for each primitive. An operator debugging a
   subtle issue can read the relevant function (e.g.
   `xvlan_input_translate` for VLAN problems) and understand exactly
   what is happening. With Cisco the operator depends on TAC and
   release notes.

The trade-off summary: OVS is more flexible and more open at the cost
of more complexity. For a deployment where customisation, controller
integration, and open-source review are valuable, OVS wins. For a
deployment where line-rate-everything plus simple operator interface
plus vendor-supported black-box semantics is valuable, Cisco still
wins.

## 0.6.7. Migration recipe for the traditional engineer

An engineer migrating from a Cisco-only environment to an
OVS-managed environment can follow this recipe:

1. **Map the physical topology** (chassis, line cards, link
   aggregation, VLAN trunks) to the OVS-managed topology (hosts,
   bridges, veth pairs, OVSDB Port rows). The OVS bridge is the
   chassis equivalent; the bridge's ports are the line-card-port
   equivalent.

2. **Translate the Cisco config to OVSDB rows**. Each
   `switchport access vlan N` becomes `ovs-vsctl set port P tag=N`.
   Each `channel-group N mode active` becomes `ovs-vsctl add-bond`.
   Each `monitor session N source ... destination ...` becomes
   `ovs-vsctl create Mirror`. Each `ip access-list extended X; deny ...`
   becomes `ovs-ofctl add-flow ... actions=drop`.

3. **Decide the L3 strategy**. OVS has no built-in routing; the
   operator must choose between the router-namespace pattern (R2.5.6),
   the raw-OpenFlow-flow pattern (also R2.5.6), or OVN. For a small
   deployment, router-namespace is simplest. For a scale-out
   deployment with thousands of tenants, OVN.

4. **Decide the IP services strategy**. DHCP, DNS, ARP responder
   require external services if not using OVN. Plan for dnsmasq, a
   stub-resolver, or OVN.

5. **Plan the observability stack**. Cisco's `show` commands map to
   `ovs-vsctl` and `ovs-appctl` commands. The operator needs a
   monitoring agent that polls the relevant OVSDB columns and
   `ovs-appctl` outputs. Common choices: Prometheus + a custom
   ovs_exporter, or sFlow plus an external collector.

6. **Plan the controller integration if applicable**. If the
   deployment uses an SDN controller, the controller programs OVS
   via OpenFlow. The controller's view of the bridge replaces the
   manually-installed flow rules. R2.5 used manual `ovs-ofctl` for
   pedagogical clarity; production usually delegates to a controller.

The migration is non-trivial but feasible. The R2.5 sub-captures
demonstrated each primitive's OVS form against the traditional
form, which is exactly the level of confidence the migrating
engineer needs.

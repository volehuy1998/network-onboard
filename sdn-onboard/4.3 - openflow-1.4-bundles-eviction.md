## Context: from matching model to operational concerns

From 1.0 through 1.3, the OpenFlow spec focused on the matching and action model. 1.0 set the basic 12-tuple; 1.1 added multi-table and groups; 1.2 introduced extensible OXM TLV; 1.3 added the meter, PBB, and auxiliary channels. After 1.3, the main matching pain points were resolved (at least on paper). But operational concerns remained: how to update 1000 flow entries at the same time without creating an inconsistent state? When the TCAM is full and a new flow cannot be added, which policy should decide what to evict?

OF 1.4.0 released on 2013-10-14 focused on these problems. The design intent is clearly stated: "operational concerns" instead of "matching features". Spec 1.4 added **bundles** for atomic multi-message updates, **eviction** for automatic table management, and **optical port extensions** for ROADM control in carrier networks. 1.4.1 (2014-10-17) is the final errata of branch 1.4; ONF did not release 1.4.2 or any further 1.4.x.

## 4.3.1 OF 1.4 release 2013-10-14, ambition for transactional semantics

### Release detail

OpenFlow Switch Specification Version 1.4.0 officially released on **2013-10-14** with wire protocol version **0x05**. The 206-page document (up from the 128 pages of 1.3) reflects the complexity of the new features.

The ONF press release in 2013-10 describes:

> *"OpenFlow 1.4 focuses on operational concerns, enabling more robust deployment of SDN in production environments. Key enhancements include atomic bundle messages, flow entry eviction for managed table resource contention, and extensible port properties for optical ports."*

The 1.4.1 errata released on 2014-10-17 fixed small clarifications without affecting wire format. Both 1.4.0 and 1.4.1 share wire byte 0x05. Spec archive at:

```
https://opennetworking.org/wp-content/uploads/2013/10/openflow-spec-v1.4.0.pdf
https://opennetworking.org/wp-content/uploads/2014/10/openflow-switch-v1.4.1.pdf
```

### Design intent: operational maturity

ONF working groups in 2013 focused on **operational maturity**. Feedback from early deployers (Google B4, Microsoft, carrier operators on the ONF board) pointed to two primary pain points:

**Pain point 1, partial-update risk:** the controller wants to install 1000 flow rules as one atomic group (for example, tenant onboarding with 100 rules for ACL plus 500 for routing plus 400 for QoS). If it has to install rules one at a time, there is a "half-installed" window where 500 rules are active and 500 are not, and packets falling into the gap have non-deterministic behavior. In the worst case, a packet matches an old rule (about to be deleted) and a new rule (only partly installed), giving inconsistent forwarding.

**Pain point 2, table full:** the TCAM of a hardware switch has a limited size (Broadcom Trident2: 4K ingress ACL). When it is full and a new FLOW_MOD arrives, spec 1.3 allows the switch to choose: reject with OFPERR_TABLE_FULL (the controller must handle retries) or implement vendor-specific eviction (not portable). There is no standardised policy.

OF 1.4 solves both: bundles for atomic update (solves pain 1), and eviction for TCAM pressure (solves pain 2).

## 4.3.2 Bundles: atomic transaction for a group of messages

### Bundle state machine

Spec 1.4 §6.3.11 defines the bundle mechanism. The controller manages a bundle through a four-state machine:

```
        OPEN
         |
         v
    ADD_MESSAGE (multiple times)
         |
         v
       CLOSE
         |
         v
   +------------+
   |            |
COMMIT     DISCARD
   |            |
   v            v
Applied     Abandoned
```

Message types:

| Message | Meaning |
|---------|---------|
| OFPT_BUNDLE_CONTROL with type=OPEN | Open a new bundle, switch allocates resources for tracking |
| OFPT_BUNDLE_ADD_MESSAGE | Wrap an OF message (FLOW_MOD, PORT_MOD, GROUP_MOD) into the bundle |
| OFPT_BUNDLE_CONTROL with type=CLOSE | Close the bundle (no more messages may be added); switch validates the whole |
| OFPT_BUNDLE_CONTROL with type=COMMIT | Apply the entire bundle atomically |
| OFPT_BUNDLE_CONTROL with type=DISCARD | Cancel the bundle, no apply, resources released |

Each bundle has a **bundle_id** (32-bit) unique within a connection for identification. The controller may open multiple bundles in parallel with different bundle_id values.

### Two important flags

Bundles are parameterised through two flags:

**OFPBF_ATOMIC**: all-or-nothing semantic. On COMMIT:
- If every message in the bundle is valid, the switch applies them all atomically.
- If any message fails validation, the switch rolls back the entire bundle and applies nothing. The controller receives an error message indicating which message failed.

**OFPBF_ORDERED**: messages are applied in the order they were ADDed. Without OFPBF_ORDERED, the switch may reorder messages to optimise (for example, batch many FLOW_MODs sharing match structure). With OFPBF_ORDERED, the order is strict.

The two flags can combine: ATOMIC plus ORDERED equals "apply all-or-nothing in exact order".

### Analogy with SQL transactions

The bundle semantic is very similar to a SQL transaction:

| SQL | OpenFlow Bundle |
|-----|------------------|
| BEGIN TRANSACTION | OFPT_BUNDLE_CONTROL type=OPEN |
| INSERT, UPDATE, DELETE statements | OFPT_BUNDLE_ADD_MESSAGE (wrapping FLOW_MOD and so on) |
| COMMIT | OFPT_BUNDLE_CONTROL type=COMMIT |
| ROLLBACK | OFPT_BUNDLE_CONTROL type=DISCARD |
| Isolation level | OFPBF_ATOMIC flag |

This analogy is not coincidental. The ONF Forwarding Abstractions Working Group (FAWG) discussed the bundle design borrowing the concept from database transactions, with explicit references in the 2013 meeting minutes.

### Common misunderstandings about bundles

**Misunderstanding:** "A bundle is a batch the controller uses to send multiple flow_mods faster, reducing TCP overhead."

**Reality:** A bundle differs entirely from batching. Batching (not in the OpenFlow spec) is a pure performance optimisation that sends many messages in one TCP segment to save syscalls. A bundle provides **atomicity semantics**: all-or-nothing apply. A bundle with the OFPBF_ATOMIC flag may be rolled back entirely if any message fails validation at commit time. OVS implements bundles without implementing the OFPBF_ATOMIC flag (it supports only the OPEN, ADD, and COMMIT stages without atomic rollback), which is "a bundle that is not really a bundle". It remains useful for grouping but loses the transactional guarantee. When reading vendor support matrices for bundle, you must ask specifically: "does it support atomic rollback?"

### Use case: atomic tenant onboarding

A typical production use case: onboarding a new tenant into a DC OpenFlow fabric. Controller workflow:

```
1. Open bundle, bundle_id=1234.
2. Add 100 FLOW_MOD for tenant ACL (table 0).
3. Add 500 FLOW_MOD for tenant routing (table 10).
4. Add 400 FLOW_MOD for tenant QoS (meter plus flow reference).
5. Add PORT_MOD enabling the tenant-facing port.
6. Close the bundle.
7. Commit the bundle with OFPBF_ATOMIC plus OFPBF_ORDERED.
```

If any FLOW_MOD fails (for example, TCAM table 0 is full), the switch rejects the entire bundle. The controller receives an error and can retry with a smaller batch or move the tenant to another switch. There is no "half-onboarded tenant" inconsistent state.

## 4.3.3 Flow-entry eviction: policy when the table is full

### Concept §5.4.6

Eviction is a switch-side mechanism that automatically removes old flow entries when the table is full to make space for new ones. Spec 1.4 defines eviction as an **optional capability**; the switch advertises support through the OFPTFPT_EVICTION bit in the table-features reply.

Flow entries gain a new field: **importance** (16-bit). A low importance value means the flow is less important and is a candidate for eviction first. A high value means the flow is important and protected.

When the switch receives a new FLOW_MOD and the table is full:

1. If eviction is not enabled, reject with OFPERR_TABLE_FULL.
2. If eviction is enabled, scan the table and find the entry with the lowest importance.
3. If the importance is lower than the new entry (or equal if the tiebreaker allows), evict the old entry and install the new one.
4. Send a FLOW_REMOVED notification (if the OFPFF_SEND_FLOW_REM flag is set on the evicted entry).

### Comparison with timeout

A flow entry has three removal mechanisms:

| Mechanism | Trigger | Initiator |
|-----------|---------|-----------|
| idle_timeout | N seconds with no matching packet | Switch automatically |
| hard_timeout | N seconds since install (regardless of matching) | Switch automatically |
| eviction (1.4+) | Table full, new flow with higher importance | Switch automatically under resource pressure |
| Explicit FLOW_MOD with OFPFC_DELETE | Controller sends a delete message | Controller actively |

Timeout is **time-based self-removal** (passive). Eviction is **forced removal under resource contention** (reactive). They complement each other; they do not replace each other.

### Use case: prefer to evict best-effort, keep critical

Production scenario: a DC has the policy "critical tenant flow" (payment, authentication) must always be in hardware TCAM for low latency, no miss-to-software allowed. "Best-effort flow" (bulk transfer, batch jobs) may be evicted when space is contended. Configuration:

```
Critical flow: importance=65535 (max)
Normal flow: importance=32768
Best-effort: importance=1024
Default: importance=0 (lowest)
```

When TCAM fills up and the policy team adds a critical flow, eviction kicks in and removes a best-effort flow first. Critical entries never evict each other (both share high importance); the controller must decide manually through an explicit FLOW_MOD DELETE.

## 4.3.4 Optical port extensions

### Context: ONF Optical Transport WG

As discussed in 3.2.3, ONF has the Optical Transport Working Group (OTWG) established in 2013 to extend OpenFlow to the optical layer. Motivation: carrier networks have Reconfigurable Optical Add-Drop Multiplexers (ROADM) that need centralised control, tuned by a controller instead of manual CLI per device.

OF 1.4 is the first spec with native optical port descriptors in the base spec, with no separate extension required.

### Optical port properties (§7.2.2.2)

The port descriptor adds optical-specific fields:

| Field | Meaning | Unit |
|-------|---------|------|
| tx_min_freq_lmda | Minimum transmit wavelength | THz, ITU-T G.694.1 grid |
| tx_max_freq_lmda | Maximum transmit wavelength | THz |
| tx_grid_freq_lmda | Grid spacing (12.5 / 25 / 50 / 100 GHz) | GHz |
| rx_min_freq_lmda | Minimum receive wavelength | THz |
| rx_max_freq_lmda | Maximum receive wavelength | THz |
| tx_pwr_min | Minimum transmit power | 0.1 dBm |
| tx_pwr_max | Maximum transmit power | 0.1 dBm |

ITU-T G.694.1 "Spectral grids for WDM applications" defines a standardised wavelength grid for DWDM. OF 1.4 references this grid for wavelength value encoding.

### Use case: ROADM control through OpenFlow

Carrier network topology:

```
[Router A] ---Fiber--- [ROADM 1] ---Fiber--- [ROADM 2] ---Fiber--- [Router B]
                            |
                         OpenFlow
                         Controller
```

The controller issues PORT_MOD to the ROADM to configure wavelength routing: incoming wavelength lambda1 on port X to outgoing wavelength lambda2 on port Y. ROADMs are implemented through a wavelength-selective switch (WSS).

ONF's press release in 2013-10 highlighted this use case as the flagship of 1.4. NEC and Ciena delivered ROADMs supporting OpenFlow in 2014 to 2015 product lines.

### Reality: niche deployment

OpenFlow optical adoption is very niche. Most carrier deployments use:
- **NETCONF/YANG** for configuration management (Block V).
- **OpenROADM project** (AT&T driven, 2016+) with its own YANG models for optical, not OpenFlow.
- **Transport-PCE in ONOS** (2017+) as an alternative controller framework.

OF 1.4 optical extensions are in the spec but production traction declined after 2016. Forward reference to 16.1 "SDN in optical networks" (removed from rev 3 scope but lineage) to understand the post-OpenFlow evolution.

## 4.3.5 Adoption reality: OVS partial support, vendor skepticism

### OVS implementation of 1.4 partial

Open vSwitch 2.5 (2016-03) added partial 1.4 support:

**Implemented**:
- Wire protocol 0x05 handshake (version negotiation).
- Bundle OPEN, ADD, CLOSE, COMMIT, and DISCARD messages.
- OFPBF_ATOMIC flag semantics.

**Not implemented**:
- Eviction: OVS uses the `--max-entries` config limit but does not have native importance-based eviction logic.
- Optical port properties: OVS does not target optical hardware; irrelevant.
- Some minor OXM extensions of 1.4 (for example, metadata variants).

OVS upstream discussion in 2014 to 2016 (mailing list `ovs-dev@`) indicates reviewers were sceptical about the cost-versus-benefit of full 1.4 support. Ben Pfaff (core maintainer) commented:

> *"We implement 1.4 bundles because users ask for atomic update, but eviction makes OVS datapath much more complex for little benefit. Users can implement policy controller-side."*

Eviction was tagged "won't fix" in OVS.

### Hardware vendor skip

Hardware switch vendors skipped 1.4 almost entirely:

- **HP ProCurve and Aruba**: committed to 1.3 fully, with 1.4 only "announced support" in spec sheets and no real hardware acceleration.
- **Pica8 PicOS**: 2014 to 2015 roadmap listed 1.4 "planned" but not shipped.
- **NEC IP8800**: stopped the OpenFlow hardware line after 2016, never implemented 1.4.
- **Centec V350**: focus on 1.3, with 1.4 absent from spec sheets.

The Ryu framework tutorial and the OpenDaylight plugin stopped at 1.3 for primary samples; 1.4 only had stub code, not complete.

### Economics analysis

Why did vendors skip? Reasons are purely economic:

1. **Market signal**: 1.4 released in 2013-10. At the same time, enterprise SDN customers (DC operators) had completed 1.0 and 1.3 pilots and were moving to production rollout. They did not want to retool for a new version every 14 months.

2. **Engineering cost**: implementing full 1.4 on ASIC requires new microcode, a new verification suite (new ONF conformance tests for 1.4), and documentation updates. Vendors estimated the cost at 2 to 5 million USD per product line. Incremental revenue from 1.4 features did not justify it.

3. **Competitor unwillingness**: Cisco and Juniper began pivoting to SDN through APIs (NETCONF/YANG, controller-centric instead of protocol-centric, Block V) and had no incentive to push a new OpenFlow version.

4. **P4 on the horizon**: Bosshart et al. published the P4 paper in CCR 2014 (almost simultaneously with the 1.4.1 errata), creating momentum for an alternative programmable data plane. Vendors saw P4 plus Tofino as the future, not OpenFlow 1.4 or 1.5.

This is the start of the "OpenFlow stalled" narrative. After 1.4, spec evolution lost momentum, leading to 1.5 (2014-12) being practically dead on arrival. Cross-reference 4.6.7 analyses the post-mortem lessons learned and the rise of P4 plus hypervisor overlay.

## Guided Exercise: bundle atomic transaction with OVS (audit P4.B4.1 hands-on)

**Goal:** experiment with bundle semantics, and verify all-or-nothing apply when there is a validation error.

**Steps:**

1. Create an OF 1.4+ bridge:
   ```bash
   $ sudo ovs-vsctl add-br br-bundle -- set bridge br-bundle protocols=OpenFlow14
   ```
2. Bundle three flow_mods atomically (one invalid rule, all roll back):
   ```bash
   $ cat > /tmp/bundle.txt <<EOF
   bundle add-flow
   table=0,priority=100,ip,nw_src=10.0.0.0/24,actions=drop
   bundle add-flow
   table=0,priority=100,ip,nw_src=10.1.0.0/24,actions=drop
   bundle add-flow
   table=0,priority=100,INVALID_FIELD=bad,actions=drop
   EOF
   $ sudo ovs-ofctl -O OpenFlow14 bundle br-bundle /tmp/bundle.txt
   # Expected: commit fails, all 3 rolled back.
   ```
3. Verify no partial apply:
   ```bash
   $ sudo ovs-ofctl -O OpenFlow14 dump-flows br-bundle
   # Expected: empty (all 3 rolled back)
   ```
4. Correct bundle (all valid):
   ```bash
   $ cat > /tmp/bundle-ok.txt <<EOF
   bundle add-flow
   table=0,priority=100,ip,nw_src=10.0.0.0/24,actions=drop
   bundle add-flow
   table=0,priority=100,ip,nw_src=10.1.0.0/24,actions=drop
   EOF
   $ sudo ovs-ofctl -O OpenFlow14 bundle br-bundle /tmp/bundle-ok.txt
   $ sudo ovs-ofctl -O OpenFlow14 dump-flows br-bundle | wc -l
   # Expected: 2 rules applied atomically
   ```

**POE:**
- **Predict:** if a bundle has 1000 flow_mods and 1 is invalid, how many rules are applied?
- **Observe:** 0 rules. All-or-nothing. Bundle atomic guarantee.
- **Explain:** Bundles avoid the "half-installed" state that the controller had to handle manually in OF 1.3. Production use case: multi-tenant onboarding 100+ rules atomically per tenant.

**Cleanup:** `sudo ovs-vsctl del-br br-bundle && rm /tmp/bundle*.txt`

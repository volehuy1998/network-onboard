## Bối cảnh: vượt qua các giới hạn của OpenFlow

Part 4.6 đã tổng kết 5 giới hạn cấu trúc của OpenFlow: flow-table explosion, controller latency, distribution scale, vendor silicon mismatch, L4-L7 abstraction gap. Root cause chung: OpenFlow spec **fixed** cấu trúc match + action, thêm field mới yêu cầu phiên bản revision. Hardware vendor không thể implement toàn bộ spec do silicon constraint. Controller không thể portable across heterogeneous fabric.

P4 cách tiếp cận giải các limitation này bằng mô hình shift. Thay vì protocol **ngầm định** match fields + actions, P4 cho phép **programmer define** data plane behavior bằng domain-specific language. Parser, match-action pipeline, deparser đều programmable. Silicon vendor implement P4 compiler cho target của họ; programmer viết P4 một lần, compile cho multiple target.

Nguyên lý: **target-independent programmability**, giống cách C programming language cho phép programmer write code một lần compile cho x86/ARM/RISC-V. Thay vì "OpenFlow spec dictate silicon behavior", P4 program dictate, silicon adapt.

## 6.0.1 P4 origin: ACM CCR July 2014

### Bài báo và các tác giả

P4 paper chính thức: "P4: Programming Protocol-Independent Packet Processors" bởi Bosshart et al., ACM SIGCOMM Computer Communication Review (CCR) Volume 44 Issue 3, July 2014. URL: `https://arxiv.org/abs/1312.1719` (arXiv mirror).

Authors (11 people):
- **Pat Bosshart**: ex-Texas Instruments silicon designer.
- **Dan Daly**: Intel.
- **Glen Gibb**: Stanford.
- **Martin Izzard**: Texas Instruments.
- **Nick McKeown**: Stanford (co-author OpenFlow, Part 3.0).
- **Jennifer Rexford**: Princeton.
- **Cole Schlesinger**: Princeton.
- **Dan Talayco**: Big Switch Networks.
- **Amin Vahdat**: Google.
- **George Varghese**: Microsoft Research.
- **David Walker**: Princeton.

Mix academia (Stanford/Princeton) + industry (Intel/TI/Big Switch/Google/MSR). Khác với OpenFlow paper (CCR 04/2008) primarily Stanford authors, P4 paper reflect ngành đã mature với multi-org collaboration.

### Động lực

Paper mở đầu với 3 problem statement của OpenFlow:

**Problem 1**: OpenFlow spec có fixed header fields. Khi protocol mới (VXLAN, Geneve, NSH) xuất hiện, phải wait spec revision. Cycle 12-24 tháng không match với pace protocol evolution.

**Problem 2**: Switch silicon heterogeneous. Broadcom Trident2, Mellanox Spectrum, Intel FM6000 có constraints khác nhau (xem Part 4.5.1). OpenFlow spec không phơi bày primitive, silicon vendor proprietary API (Broadcom SDK closed-source).

**Problem 3**: Parser là fixed. OpenFlow assume L2/L3/L4 header standard sequence. Custom protocol (e.g., RDMA over Converged Ethernet, custom encapsulation cho security) không expressible.

P4 address tất cả 3 bằng programmable abstraction.

### Nguyên tắc ngôn ngữ

Paper propose 3 language principles:

**1. Protocol Independence**: P4 không hard-code header formats. Programmer define headers. Switch chỉ là compiler target.

**2. Target Independence**: P4 code portable across silicon targets. Tofino, Broadcom, software BMv2 tất cả có P4 compiler.

**3. Field Reconfigurability**: reload P4 program → switch behavior change runtime (không cần hardware upgrade).

Sau khi paper publish, consortium formal hoá thành P4 Language Consortium, sau này absorb vào Open Networking Foundation + Linux Foundation.

## 6.0.2 P4_14 so với P4_16: tiến hóa ngôn ngữ

### P4_14 (2014-2016)

Original language từ paper 2014. Syntax example (simplified):

```
header ethernet_t {
    fields {
        dstAddr : 48;
        srcAddr : 48;
        etherType : 16;
    }
}

parser parse_ethernet {
    extract(ethernet);
    return select(latest.etherType) {
        0x0800 : parse_ipv4;
        default : ingress;
    }
}

table forwarding {
    reads {
        ethernet.dstAddr : exact;
    }
    actions {
        forward;
        drop;
    }
}
```

Issues with P4_14:
- **Verbose**: nhiều boilerplate khiến programmer write/maintain code mệt mỏi.
- **Tightly coupled với reference architecture**: P4_14 assume specific pipeline structure (ingress pipeline + egress pipeline + deparser), hard to target devices với pipeline khác.
- **Limited composition**: khó viết library reusable code.

### P4_16 (2016+)

Refactor 2016 language phiên bản với goals: lean core language + modular architecture. Syntax example:

```p4
#include <core.p4>
#include <v1model.p4>

header ethernet_t {
    bit<48> dstAddr;
    bit<48> srcAddr;
    bit<16> etherType;
}

parser MyParser(packet_in packet, out headers hdr, inout metadata meta, inout standard_metadata_t std) {
    state start {
        packet.extract(hdr.ethernet);
        transition select(hdr.ethernet.etherType) {
            0x0800 : parse_ipv4;
            default : accept;
        }
    }
}

control MyIngress(inout headers hdr, inout metadata meta, inout standard_metadata_t std) {
    action forward(bit<9> port) { std.egress_spec = port; }
    action drop() { mark_to_drop(std); }

    table ipv4_lpm {
        key = { hdr.ipv4.dstAddr : lpm; }
        actions = { forward; drop; }
        default_action = drop();
    }

    apply { if (hdr.ipv4.isValid()) ipv4_lpm.apply(); }
}
```

Changes:
- Core language minimal (parser, control blocks, tables, actions).
- **PSA (Portable Switch Architecture)** externalized thành separate spec. PSA define standard blocks (parser, ingress, egress, deparser, packet_in/out). Programmer import PSA, không dùng reference architecture hard-coded.
- Better type system (fixed-width bit types, explicit extraction).
- Library hỗ trợ (include files, extern declarations).

Từ 2019, **P4_16 + PSA là standard**. P4_14 deprecated, compile-to-P4_16 path deprecated.

## 6.0.3 PISA (Protocol-Independent Switch Architecture)

### Mô hình máy trừu tượng

PISA là abstract hardware model mà P4 programmer target. Cấu trúc:

```
Input → Programmable Parser → Match-Action Pipeline (N stages) → Programmable Deparser → Output
                                     │
                                 Stage 0  (TCAM + SRAM + ALU)
                                 Stage 1
                                 ...
                                 Stage N
```

**Parser**: state machine programmable. Input = bit stream, output = extracted headers. Programmer define state transitions based on header fields.

**Match-Action Pipeline**: N stage sequential. Mỗi stage có:
- **Match unit**: lookup table với TCAM (ternary) cho wildcarded match, SRAM (exact) cho hash-based match.
- **Action unit**: ALU (Arithmetic Logic Unit) để modify headers / metadata / tables.

Programmer define N tables, mỗi table map với specific stage hoặc multiple stages.

**Deparser**: inverse of parser. Serialize modified headers back to bit stream for output.

### Khác biệt với fixed-function ASIC

Traditional switch ASIC (Broadcom Trident2, Mellanox Spectrum-1/2) có pipeline hard-coded:

```
L2 MAC lookup → L3 LPM → ACL → Queue → Output
```

Header parsing fixed (assume Ethernet + IPv4/ARP/MPLS). Action limited (mod MAC, mod DSCP, etc.). Thêm protocol mới = re-spin silicon (new mask ROM).

PISA silicon (Tofino) có pipeline **programmable**:
- Parser programmable: can extract any header format.
- Tables programmable: programmer define width/depth/action per stage.
- Actions programmable: ALU execute arbitrary computation within power budget.

Hardware area của PISA vs fixed-function: khoảng 10-20% larger (overhead cho programmability), sự đánh đổi acceptable cho tính linh hoạt.

### Tofino là silicon PISA tham chiếu

Barefoot Networks founded 2013 bởi Pat Bosshart (P4 paper first author) + team cựu MSR/Stanford. Launch Tofino ASIC 2016:

- **Tofino 1 (2016)**: 3.3 Tbps, 12 stages, programmable.
- **Tofino 2 (2018)**: 12.8 Tbps, scale-up.
- **Tofino 3 (2021)**: announced but unclear production volume.

Tofino là first commercial PISA silicon, validate P4 mô hình in production. Customer sự áp dụng: Alibaba Cloud, Tencent, Facebook (limited triển khai).

## 6.0.4 Các target: Tofino ASIC, BMv2 simulator, eBPF, DPDK

### Hệ sinh thái target P4

P4 programmer write once, compile cho multiple target qua p4c compiler:

**Tofino ASIC (Barefoot/Intel)**: production silicon. `p4c-tofino` backend generate microcode cho Tofino pipeline. Throughput 3-12 Tbps.

**BMv2 (Behavioral Model v2)**: software reference switch for development + testing. Không production speed (kernel-mode C++ implementation, ~1-5 Gbps on commodity server), nhưng exact P4 semantics cho verification.

**T4P4S (DPDK target)**: P4 compile sang DPDK userspace C code. Target: commodity NIC + DPDK acceleration. Production-speed cho modest workload (10-40 Gbps per core).

**eBPF/XDP target**: P4 compile sang eBPF bytecode, load vào Linux kernel. Target: commodity server, không special hardware. eBPF/XDP achieve ~20-50 Gbps on fast NIC.

**FPGA targets (NetFPGA, Xilinx)**: research-grade FPGA boards.

### Hạ tầng compiler p4c

`p4c` tại `https://github.com/p4lang/p4c` là compiler framework:

- **Front-end**: parse P4 source, semantic analysis.
- **Mid-end**: optimize intermediate representation.
- **Back-ends**: target-specific code generation (Tofino, BMv2, eBPF, DPDK).

Modular architecture: adding new target = write new back-end without modify front-end.

### Sự đánh đổi khi chọn target

| Target | Throughput | Cost | Use case |
|--------|------------|------|----------|
| Tofino ASIC | 3-12 Tbps | $$$$ (silicon + ODM switch) | Production DC switch |
| BMv2 | 1-5 Gbps | Free | Dev + test |
| T4P4S DPDK | 10-40 Gbps/core | $$ (commodity NIC + server) | Mid-throughput production, NFV |
| eBPF/XDP | 20-50 Gbps | $ (commodity NIC + server) | Edge compute, container networking |

Post-Tofino EOL, các soft target (DPDK, eBPF) trở nên quan trọng hơn cho P4 sustainability.

## 6.0.5 P4Runtime API

### Giao diện southbound dựa trên gRPC

P4 program define what switch can do (table schema, action set). Nhưng controller cần install runtime entries (specific table entries cho specific match). P4Runtime là southbound API cho purpose này.

**Architecture**:
- **Protocol**: gRPC với Protocol Buffers (protobuf).
- **Phiên bản**: current v1.3+ (2023).
- **Operations**: install/delete/modify table entry, read counters/registers, packet-in/packet-out, multicast group management, session management.

**Spec**: `https://github.com/p4lang/p4runtime` với documentation trong `docs/` directory.

### Tính độc lập với phiên bản

Key feature: **single API cho multiple P4 program versions**. Controller connect to switch, retrieve current P4 program's schema (table names, match types, action parameters) via `GetForwardingPipelineConfig`, install entries based on schema.

Khi P4 program update (thay đổi table definition), controller tự động adapt qua schema re-read. Khác với OpenFlow (fixed schema across phiên bản, controller hard-code match fields), P4Runtime flexible cho evolving P4 programs.

### Thay thế OpenFlow

P4Runtime design as **OpenFlow replacement cho P4-capable switches**. Analogy:

| OpenFlow | P4Runtime |
|----------|-----------|
| Fixed pipeline (spec định nghĩa) | Programmable pipeline (P4 program định nghĩa) |
| Hard-coded match fields | Schema-based match from P4 program |
| Fixed action set | P4 program actions |
| Protobuf wire format | gRPC + protobuf |

Production sự áp dụng: ONOS integrate P4Runtime từ 2018. Stratum (Part 5.2.4) runs P4Runtime as thin switch agent.

### Chuẩn hóa

Standardized bởi ONF + ONOS collaboration 2018. Subsequent v1.1 (2019), v1.2 (2020), v1.3 (2023). Maintained as open spec on GitHub (`p4lang/p4runtime`).

## 6.0.6 Intel mua lại năm 2019, Tofino EOL năm 2023, trạng thái hiện tại

### Intel mua lại

Intel acquire Barefoot Networks tháng 6/2019, reported ~$500M. Rationale: Intel's data center group pursue programmable networking as vertical expansion từ CPU + Altera FPGA (acquired 2015) + Mount Evans IPU (ConnectX equivalent SmartNIC).

Barefoot team retained, continue Tofino roadmap trong Intel Connectivity Group.

### Tofino cancellation January 2023

Intel announce end-of-life cho Tofino family tháng 1/2023. Reasons (Intel-disclosed + industry speculation):
- **Silicon R&D cost**: programmable silicon expensive to iterate compared to off-the-shelf Broadcom/Marvell.
- **Market share**: Tofino sự áp dụng remained niche (primarily Alibaba, Tencent, few telcos), not enough volume to justify Intel overhead.
- **Strategic shift**: Intel focus AI/HPC + Mount Evans IPU, switch silicon deprioritized.
- **Post-acquisition uncertainty**: Barefoot cultural fit với Intel organization challenge.

Impact: P4 ecosystem lost only production silicon target. Customer (Alibaba, operators with Tofino triển khai) had to plan migration over multi-year period.

### Current state 2026

P4 momentum continues qua software targets + community:

**P4.org community**: `https://p4.org/` maintains specs, tools, annual P4 workshop. Active community.

**Software targets mature**:
- **BMv2** stable for education.
- **T4P4S** active for NFV.
- **eBPF/XDP** growing sự áp dụng for edge compute.

**Alternative hardware**:
- **AMD Pensando DPU**: programmable silicon post-acquisition Pensando 2022. P4-like programming model via "PSM" (P4-compatible state machine).
- **NVIDIA BlueField DPU**: ConnectX-based SmartNIC với DOCA (Part 9.5). DOCA Pipelines is P4-inspired abstraction. Not exact P4 spec but similar mô hình.
- **Cisco Silicon One**: custom ASIC with limited programmability. Not full P4.

**Research continues**:
- Academic P4 research strong (SIGCOMM, NSDI 2020-2025 frequent P4 papers).
- Use case: in-network compute (P4 accelerator for key-value store), telemetry (INT - In-band Network Telemetry), consensus (Paxos in data plane).

### Hiểu sai thường gặp về P4

**Hiểu sai:** "P4 đã chết sau Tofino EOL 2023."

**Thực tế:** P4 ecosystem survived Tofino cancellation. Language standard + community active, multiple software targets mature. Alternative hardware (AMD Pensando, NVIDIA BlueField) fill commercial gap with P4-inspired programming. Academic research thrives. Rumors of new silicon startup pursuing P4 (2024-2026) suggest commercial revival possible. P4 mô hình legacy: introduced programmable data plane concept mainstream, even if specific Tofino silicon line ended.

> **Key Topic:** P4 là mô hình evolution vượt OpenFlow, không phải phiên bản 2.0. OpenFlow fixed-protocol model không scale với protocol diversity; P4 programmable model absorb tính linh hoạt at cost of complexity. Tofino EOL 2023 là commercial setback but not ecosystem death. Long-term trajectory: protocol standardization shift toward programmable, P4 có khả năng revive khi alternative silicon mature (AMD Pensando, NVIDIA BlueField DPU, future purpose-built startup chipset).

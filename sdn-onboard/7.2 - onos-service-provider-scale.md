## Context: the gap that ODL could not fill at service-provider scale

ODL (Part 7.1) was the default choice for multi-protocol SDN during 2014 to 2016, but service providers quickly discovered a serious limit. When an ODL instance scaled up to manage 500 or more switches, control-plane latency began to grow noticeably. Average flow installation reached 200 ms or more, and event handling developed tail latency above one second during peak periods. The root cause was the monolithic architecture: every plugin and every service shared the same JVM heap and the same event queue. When one plugin was slow (for example, a NETCONF plugin waiting for a device reply), it held a thread and made every other plugin wait.

The theoretical solution was horizontal scale: run several ODL instances in parallel. ODL was not designed for horizontal scale though. The MD-SAL datastore is in-memory and single-instance, with no built-in distribution mechanism. An ODL cluster works through active-passive failover (one instance is leader, the others are standby), but it does not actually distribute the workload.

AT&T, with a plan to SDN-ize the entire core network (the CORD program announced in 2014), needed a different controller. It needed a controller distributed from day one, where any switch can be managed by any node in the cluster, and where the cluster can scale from three nodes to seven, then to eleven, in response to load, without a redesign. AT&T decided to be the principal funder of a new project, parallel to (not a replacement for) ODL: ONOS.

## 7.2.1 ONOS origins: ON.Lab and AT&T, 2012 to 2014

ON.Lab (Open Networking Laboratory) was founded in 2012 by Nick McKeown (Stanford) and Guru Parulkar (Stanford, who became ON.Lab's first executive director). It was not an academic lab, it was a nonprofit independent organization based in Menlo Park, California. ON.Lab's initial mission was broad: develop open-source software for SDN, and help the networking industry transition from vendor lock-in to an open model.

ON.Lab initially focused on Mininet (a network emulator, popular in SDN teaching) and a few smaller tools. After many discussions with AT&T and other large service providers during 2013, ON.Lab settled on a new focus: build an SDN controller that can actually operate at telco scale. AT&T was the principal funder, both through financial commitment and by assigning engineers to work alongside ON.Lab engineers.

**The first release, Avocet,** was announced on 2014-12-05. The release naming convention used birds (Avocet, Blackbird, Cardinal, and so on) instead of chemical elements as in ODL. The choice reflected a different engineering culture. Avocet was an "architectural blueprint" rather than a production-ready product. It demonstrated that the distributed core worked, but it lacked many features needed for real deployment. Later releases (Blackbird 2015-02, Cardinal 2015-06) added features gradually. Only at Emu (2016-06) was ONOS considered ready for production.

The foundation paper, **"ONOS: Towards an Open, Distributed SDN OS"** by Pankaj Berde and colleagues (Open Networking Lab, Ciena, Infoblox, Pica8), was presented at HotSDN 2014 (a workshop co-located with SIGCOMM). The paper laid out three architectural principles for ONOS: (a) a distributed core for horizontal scale, (b) a unified network abstraction for applications, and (c) high availability with automatic failover. These three principles distinguished ONOS from ODL by design from the start, not as later additions.

## 7.2.2 The distributed architecture: Atomix and the multi-master model

The technical heart of ONOS is the Atomix library, a collection of distributed primitives based on the Raft consensus protocol (Part 10.1 will analyze Raft in detail in the OVSDB context). Atomix was originally an independent Java library developed by Jordan Halterman, then deeply integrated into ONOS. From ONOS 2.0 (2019), Atomix became an inseparable component.

Atomix provides distributed primitives that ONOS applications use to manage shared state across nodes:

- **AtomicMap**: a distributed map where every entry is replicated across all nodes through Raft. A write triggers a Raft log entry and commits when a quorum (for example, two of three nodes) agrees. The entry is then available on every node.
- **AtomicCounter**: a distributed counter with atomic increment. Used for ID generation: every node can issue cluster-wide unique IDs.
- **LeaderElection**: a leader-election mechanism for a specific resource. Used when a task must run on exactly one node (for example, holding the master role for a specific OpenFlow switch).
- **DistributedLock**: a distributed lock that ensures at most one node holds the lock at any moment.

> **Key point:** The multi-master model in ONOS is fundamentally different from the active-passive model in ODL. In ONOS, every node in the cluster can receive and process writes from clients. The receiving node writes to Atomix, Atomix replicates the write through Raft, and other nodes synchronize their state. The result: write throughput scales close to linearly with the number of nodes (within Raft quorum limits), not bottlenecked by a single leader.

**Responsibility split between cluster nodes.** Each OpenFlow switch connects to every ONOS node in the cluster (using multiple controller connections, a feature in OpenFlow 1.2 and later). At any given time, only one node is elected **master** for that switch. The remaining nodes hold the **equal** role (they can receive and send messages but not decide state). When the master crashes, Atomix LeaderElection elects a new master within a few hundred milliseconds. The switch keeps running without traffic loss.

Master assignment across nodes is balanced automatically. With a three-node cluster and 300 switches, each node is master for about 100 switches. When a fourth node is added, the cluster rebalances automatically to about 75 switches per node. This rebalance is the key to scaling. Load grows linearly with the number of switches. Capacity grows linearly with the number of nodes.

## 7.2.3 Performance targets: one million flows, 100 ms control loop

ONOS published specific performance targets in a 2015 whitepaper. These numbers have been benchmarked repeatedly by ON.Lab and its partners:

- **One million flow entries** managed concurrently across the entire cluster (three to seven nodes).
- **A 100 millisecond control loop**, measured from `packet_in` to flow-entry installation on the switch.
- **99.999 percent availability** (five nines), achieved through an active-active cluster and automatic failover.

To reach these targets, ONOS introduced several notable technical optimizations. First, the **flow rule subsystem** is designed to batch and pipeline flow installs. Instead of installing one flow at a time as Ryu does, ONOS groups several flows and sends a batch to the switch in a single OpenFlow transaction. This greatly reduces per-flow overhead.

Second, the **intent framework** lets an application describe the network it wants (a high-level intent) instead of specific flow rules. For example, an intent of "host A can talk to host B over a path with at least 100 Mbps of bandwidth". ONOS compiles the intent into specific flow rules, computes the path, and installs the rules. When the topology changes (a link goes down), the intent framework automatically recompiles and reinstalls the flow rules over the new path, with no application involvement.

Third, a **distributed store for topology**. The topology is held in a distributed graph. An update from any node propagates to the whole cluster within 50 ms. Applications can query the topology from any cluster node and get the same view (eventual consistency for non-critical updates, strong consistency for critical updates).

## 7.2.4 Trellis SD-Fabric: a full data-center stack

Trellis is an important sub-project in the ONOS ecosystem. It is not part of ONOS-core, but it is built directly on ONOS. Trellis is an "SD-Fabric": a complete leaf-spine data-center fabric stack based on whitebox switches and the ONOS controller, with no need for a vendor controller such as Cisco ACI or VMware NSX.

Trellis components: (a) **whitebox switches** running ONL (Open Network Linux) or SONiC, typically with a Broadcom Tomahawk or Trident chipset. (b) The **OFDPA (OpenFlow Data Plane Abstraction)** pipeline, a standard TTP that defines how OpenFlow maps onto Broadcom hardware ASICs. (c) **ONOS** with specialized applications such as `segmentrouting`, `fpm`, and `mcast`. (d) **Quagga or FRR** for the underlay routing protocol running on each leaf and spine switch.

Architecture overview: the underlay uses unnumbered routing between leaf and spine (FRR), and the overlay uses VXLAN with ONOS managing the VXLAN tunnel endpoint (VTEP) on each leaf. ACL, multicast, and traffic engineering are all configured by ONOS through segment routing on the underlay. The result is a complete stack for a multi-tenant data center, functionally equivalent to Cisco ACI, but running on hardware priced at 2,000 to 5,000 USD per switch instead of 15,000 to 30,000 USD.

Notable Trellis deployments: Comcast uses Trellis in some regional data centers. Turk Telecom (Turkey) has a production Trellis deployment. A few university research groups use Trellis as a testbed.

> **Note:** Trellis is production-grade but requires careful hardware compatibility. The OFDPA pipeline only runs on specific switches (Edgecore AS7712-32X, Accton Wedge100BF, certain QuantaMesh models) with OpenFlow-enabled firmware. Deploying Trellis on hardware that has not been officially tested often leads to hard-to-debug failures. Several teams have abandoned Trellis after several months of unsuccessful attempts on unsupported hardware.

## 7.2.5 ON.Lab merges with the ONF, October 2018

In October 2018, ON.Lab and the Open Networking Foundation (ONF), the OpenFlow standardization body (Part 3.2), merged into a single organization that kept the name ONF. Guru Parulkar became the executive director of the merged ONF.

Reasons for the merger: both organizations had a mission to advance open-source networking. Both also faced financial pressure as large vendors reduced their investment in SDN standardization (OpenFlow was already stable, with few reasons to fund new standards work). Merging reduced operating costs, shared engineering resources, and put ONOS at the center of ONF strategy (instead of being only a standardization body).

After the merger, ONOS continued active development with releases named Nightingale, Owl, Peacock, and so on. The release cycle is six months, and the focus has gradually shifted from classical SDN to 5G:

- **SD-Core**: a control plane for 5G UPF (User Plane Function), letting a mobile operator use ONOS to manage the 5G data plane.
- **Aether**: an open private-5G platform that uses ONOS for RAN (Radio Access Network) slicing.
- **P4Runtime integration**: ONOS speaks to P4 switches (Part 6.0) over P4Runtime, not only OpenFlow.

ONOS remains in active development as of 2026, with a commit velocity of about 100 to 150 commits per week. That is smaller than the ODL peak, but stable. The contributor community is mostly tier-1 service providers and 5G partners (Intel, ARM, Nokia).

## 7.2.6 ONOS versus ODL, head to head

> **Common misconception:** "ONOS and ODL are two interchangeable controllers, and the user picks one based on preference."
>
> **Reality:** ONOS and ODL did have overlapping use cases in their early years (2014 to 2016), which led to the "two interchangeable choices" view. From 2017 onward, the two projects diverged clearly. ODL serves multi-protocol service providers (many older device types), and ONOS serves high-scale service providers with new services (5G, segment routing). Few real deployments could pick either: the choice usually depends on the specific deployment pattern.

*Table 7-3: ONOS versus ODL on the main technical criteria*

| Criterion | ODL | ONOS |
|---|---|---|
| Cluster model | Active-passive, failover | Active-active, multi-master |
| Horizontal scale | Limited (monolithic datastore) | Native (Atomix and Raft) |
| Southbound protocols | The widest range (OpenFlow, NETCONF, BGP-LS, OVSDB, SNMP, and more) | Mainly OpenFlow, P4Runtime, gNMI |
| Main focus | Multi-protocol service provider | High-scale service provider, 5G |
| Release cycle 2024+ | Maintenance only | Active, every six months |
| Codebase size | About three million lines of Java | About one million lines of Java |
| Resource requirement | 8 to 16 GB RAM | 4 to 8 GB RAM |

To summarize, the choice between ODL and ONOS is not "which is better", it is "what does my deployment need". A network with many legacy routers needing NETCONF management and a few hundred devices: ODL. A data-center or 5G core network with thousands of switches and a need for horizontal scale: ONOS. A greenfield network with only OpenFlow and Kubernetes workloads: neither is the natural choice. OVN or Antrea fits better.

## Key points to remember

First, ONOS was designed for service-provider scale from day one, with a distributed-first architecture using Atomix and Raft. This is a fundamental difference from monolithic ODL. It is a core design principle, not an added feature.

Second, Trellis SD-Fabric demonstrates that ONOS is more than a controller. It is a foundation for building a complete networking stack that can compete with proprietary vendor controllers in the data-center market. However, Trellis requires strict hardware compatibility and deep deployment expertise. This limits it to organizations with strong in-house engineering teams.

Third, the merger of ON.Lab and the ONF in October 2018 placed ONOS at the center of the open-networking strategy. The shift in focus from classical SDN to 5G (SD-Core, Aether) reflects a broader industry trend: service providers today are more interested in 5G UPF than in a classical SDN controller. ONOS remains active in 2026, unlike ODL, which entered maintenance mode in 2023.

## 3.4.1 OpenFlow version timeline

| Version | Wire | Date | Major addition |
|---------|------|------|----------------|
| **1.0** | 0x01 | Dec 2009 | Initial release, single table with 12-tuple match, 8 actions. |
| 1.1 | 0x02 | Feb 2011 | **Multi-table pipeline (254 tables)**, group table, MPLS support, VLAN tagging actions. |
| 1.2 | 0x03 | Dec 2011 | **OXM TLV match** (replacing NXM and the legacy match), IPv6 support, controller role messages. |
| **1.3** | 0x04 | Apr 2012 | **Meters** (rate limiting), per-flow auxiliary connections, IPv6 extension headers, refactored capabilities. **The LTS version, the baseline for most current deployments.** |
| 1.4 | 0x05 | Aug 2013 | **Bundles** (atomic transactions), flow monitoring, vacancy events, optical port. |
| **1.5** | 0x06 | Apr 2015 | **Egress tables**, the **`copy_field`** action, **packet-type-aware pipeline**, scheduled bundles. |

**Important notes:**
- OF 1.0 and OF 1.3 are the two versions used most commonly in production. OF 1.1 and 1.2 received little support from switch hardware. Most deployments skipped from 1.0 directly to 1.3.
- OF 1.4 and 1.5 are mainly for research and advanced use cases (SFC, NSH). Production OVN deployments use OF 1.5 for the egress table feature.

**OVS version support matrix:**

| OVS version | OF support |
|-------------|------------|
| OVS 1.x (2009 to 2012) | OF 1.0 only. |
| OVS 2.0 to 2.3 | OF 1.0, 1.1, 1.2 (partial), and 1.3. |
| OVS 2.4 to 2.6 | OF 1.0 to 1.4. |
| OVS 2.7 to 2.16 | OF 1.0 to 1.5 (with NXM extension). |
| OVS 2.17 and later (curriculum baseline) | OF 1.0 to 1.5 full, plus NXM extension and experimenter messages. |

## 3.4.2 Single-table versus multi-table pipeline (OF 1.0 to 1.1 and later)

**Core difference:** OF 1.0 has only **table 0**. Each packet matches one flow in table 0, executes the action set, and is done. There is no way to chain multiple stages of logic.

**OF 1.0 example:**
```
table=0,priority=100,in_port=1,actions=output:2
```
One rule equals one final decision.

**OF 1.1 and later example:**
```
table=0,priority=100,in_port=1,actions=goto_table:1
table=1,priority=50,eth_type=0x0800,actions=goto_table:2
table=2,priority=50,actions=output:2
```
The rule in table 0 does not output immediately. It jumps to table 1 for the next stage. This allows modular composition (port security in table 0, ACL in table 1, QoS in table 2, forwarding in the last table).

**Migration impact:** OF 1.0 rules have no `goto_table`. Migrating to 1.3 requires:
1. **Identity migration (the simplest):** keep the rule in table 0, do not use `goto_table`. The result is equivalent to 1.0 behavior.
2. **Refactor migration:** split rules by logical stage and use a `goto_table` chain. Benefit: the OVS classifier runs more efficiently because each table has fewer distinct priorities.

**Reason for the multi-table design:** OF 1.0 deployments showed in practice that a single table did not scale. When a rule needed to combine 5 logics (`in_port`, ACL, QoS, L3, output), the rule cardinality became the Cartesian product, so the table size exploded. Multi-table reduces this from O(N times M) to O(N plus M).

## 3.4.3 NXM versus OXM encoding (OF 1.0 to 1.1 versus 1.2 and later)

**Match field encoding** is how the packet header fields are serialized in the FLOW_MOD wire format.

**OF 1.0:** uses a fixed 40-byte `struct ofp_match` with 12 hard-coded fields (`in_port`, `dl_src`, `dl_dst`, `dl_vlan`, `dl_vlan_pcp`, `dl_type`, `nw_tos`, `nw_proto`, `nw_src`, `nw_dst`, `tp_src`, `tp_dst`). The wildcards are a 32-bit bitmap. This is rigid: adding a new match field (ARP, IPv6) is impossible without breaking the wire format.

**OF 1.0 to 1.1 with the Nicira extension:** **NXM (Nicira eXtensible Match)** addresses this with a TLV format. Each field has a 16-bit class, an 8-bit type, a 1-bit `hasmask`, and a 7-bit length. This allows new fields (NXM_NX_TUN_ID, NXM_NX_REG0 to NXM_NX_REG15) to be added without breaking the format.

**OF 1.2 and later:** **OXM (OpenFlow eXtensible Match)** standardizes the TLV format in the main specification. The class is 32 bits (16-bit experimenter plus 16-bit type). Some classes:
- `OFPXMC_OPENFLOW_BASIC` (class value 0x8000 in `include/openflow/openflow-1.2.h`): standard fields such as `in_port`, `eth_src`, `ipv4_src`.
- `OXM_OF_PKT_REG<N>` (per-N OXM type code): packet register. Per the OVS comment in `include/openvswitch/meta-flow.h`, "since OF 1.3 and v2.4" the encoding lives in OF 1.3 wire format, although semantic packet-register support was firmed up in OF 1.5.
- `OFPXMC_NXM_1` (class value 0x0001 in `include/openflow/openflow-1.2.h`): the Nicira extension class for OVS backward compatibility.

**Wire format difference:**
- NXM: 16+8+1+7 = a 32-bit header per field.
- OXM: 32+8+1+7 = a 48-bit header per field.

**Migration impact:**
- OVS auto-converts internally between NXM and OXM, so the controller does not need to care.
- Controller programming only needs `--protocols=OpenFlow13` instead of `OpenFlow10`. OVS handles the wire format conversion.

## 3.4.4 Group tables (OF 1.1 and later)

**Problem:** OF 1.0 has no structure for multicast or load balancing. To replicate a packet to 10 ports:
```
table=0,priority=100,in_port=1,actions=output:2,output:3,output:4,output:5,output:6,output:7,output:8,output:9,output:10,output:11
```
The single action list is long, the classifier is not optimized, and updates are hard (changing one port means rewriting the whole rule).

**OF 1.1 and later solution:** a separate group table from the flow table. A group has many **buckets**, and each bucket has its own action list.

**4 group types:**

| Type | Purpose | Use case |
|------|---------|----------|
| `OFPGT11_ALL` | Replicate the packet to every bucket. | Multicast, broadcast. |
| `OFPGT11_SELECT` | Hash-select one bucket. | Load balancing. |
| `OFPGT11_INDIRECT` | One bucket only. | Alias group. |
| `OFPGT11_FF` (Fast Failover) | Pick the first bucket whose liveness is UP. | HA active-backup. |

**Example for OF 1.1 and later:**
```
group_id=1,type=all,bucket=actions=output:2,bucket=actions=output:3,bucket=actions=output:4
table=0,priority=100,in_port=1,actions=group:1
```

To update a group, change only the bucket list. The flow rule does not change.

**Fast Failover usage:**
- Each bucket has a `watch_port` or `watch_group` that names the resource to monitor.
- The group emits the packet through the first bucket whose resource liveness is UP.
- When the resource goes down, the group automatically falls over to the next bucket.
- HA active-backup does not need controller intervention.

## 3.4.5 Meters (OF 1.3 and later)

**Problem:** OF 1.0 to 1.2 had no standard rate-limiting structure. The OVS workaround used the Linux `tc qdisc` mechanism, or the Nicira extension `set_queue` plus an ingress policer. None of these were portable across vendors.

**OF 1.3 solution:** a separate **meter table** with band-based rate limiting.

**Meter components:**
- `meter_id`: a 32-bit ID.
- `flags`: rate unit (`OFPMF13_KBPS`, `OFPMF13_PKTPS`), burst support, statistics collection.
- **Bands list** (each meter has one or more bands): each band has:
  - `type`: `OFPMBT_DROP` (drop the packet) or `OFPMBT_DSCP_REMARK` (re-mark DSCP).
  - `rate`: kbps or pps.
  - `burst_size`: burst tolerance.

**Example for OF 1.3 and later:**
```
meter=1,kbps,burst,band=type=drop,rate=5000,burst_size=1500
table=0,priority=100,in_port=1,actions=meter:1,output:2
```

This rule means: traffic into port 1 applies meter 1 (limit 5 Mbps plus a burst of 1.5 KB) then outputs to port 2. Traffic above the rate is dropped.

**OF 1.3 also defines** the band features query (`OFPMP13_METER_FEATURES_REQUEST = 11` per `include/openflow/openflow-1.3.h:81`). The controller uses it to discover which band types the switch supports. Per the OVS source, the constant carries the `13` version prefix and the request code lives in OF 1.3 wire format, not OF 1.4 as earlier prose stated.

## 3.4.6 Bundles (OF 1.4 and later)

**Problem:** in OF 1.0 to 1.3 each FLOW_MOD applies immediately. For a sequence of 100 FLOW_MOD messages, if rule 50 fails (table full, syntax wrong), rules 1 to 49 have already applied, and the switch is in a half-applied state (partial state).

**OF 1.4 solution:** **OFPT_BUNDLE_OPEN/COMMIT/ADD_MESSAGE** atomic transactions. All-or-none application.

See Part 3.3 §3.3.8 for details.

**Why this matters for OVN:** when `ovn-controller` restarts and detects stale flows on `br-int`, instead of delete-then-add (which causes a traffic gap), it uses BUNDLE atomicity.

## 3.4.7 Egress tables (OF 1.5 and later)

**Problem:** OF 1.0 to 1.4 only had an ingress pipeline. The action set executes when the packet leaves the pipeline, just before output. There is no way to apply "per-port output processing", for example, removing a VLAN tag only for some ports.

**OF 1.5 solution:** an optional **egress pipeline** that runs after the ingress pipeline and the action set, and before the actual output. The egress table has its own range 0 to 254, separate from ingress.

**Example use case:** a logical switch has 10 ports. 7 ports output untagged, 3 ports output with VLAN tag 100. The OF 1.0 to 1.4 way:
- 10 rules at the final stage, each rule with a different action set (`push_vlan` for the 3 tagged ports, no push for the 7 untagged ports).

The OF 1.5 way:
- 1 common rule that outputs to the LSP.
- An egress table at table 0: if `out_port` is in the set {7, 8, 9}, run `push_vlan:100`.

Benefit: the flow rule count drops from 10 to 1 plus 1 egress rule. Maintenance is easier.

## 3.4.8 The `copy_field` action (OF 1.5 and later)

**Problem:** in OF 1.0 to 1.4, copying bits from one field to another required the OVS extension `move` (NXM-only, not portable).

**OF 1.5 solution:** standardize the **`copy_field`** action.

**Syntax:**
```
copy_field(src_offset, dst_offset, n_bits, src_field, dst_field)
```

**Example:**
```
actions=copy_field(0:ipv4_src,0:reg0,32)
```
Copy 32 bits from `ipv4_src` (offset 0) to `reg0` (offset 0). Downstream stages can then use `reg0` as scratch.

**Why this matters:** OVN uses registers as "scratch state" between stages (see Part 13.17). `copy_field` allows pipeline programming without a proprietary extension.

## 3.4.9 Packet-type-aware pipeline (OF 1.5 and later)

**Problem:** OF 1.0 to 1.4 assume every packet has an Ethernet header. SFC (Service Function Chaining) over NSH (Network Service Header, RFC 8300) needs to process bare packets (no Ethernet). OF 1.0 to 1.4 do not support this.

**OF 1.5 solution:** the **`packet_type` field** allows match for non-Ethernet encapsulation.

**`packet_type` values:**
- `(0, 0)`: Ethernet (default).
- `(0, 0x8847)`: MPLS unicast.
- `(0, 0x86dd)`: IPv6.
- `(1, 2)`: IPv4 bare (no Ethernet).
- `(1, 0x894f)`: NSH.
- `(1, ...)`: other custom.

**OVN use case:** allow an NSH-encapsulated packet to traverse `br-int` without having to wrap a fake Ethernet header.

## 3.4.10 Anatomy: `ovs-ofctl --protocols` negotiation

**Bucket:** OpenFlow. **Context:** verify version negotiation. **Purpose:** find which version OVS is serving on a bridge. **Activity:** debug. **Mechanism:** the OVS `Bridge.protocols` column and the HELLO bitmap exchange.

**Configure the version on OVS:**

```bash
# Set the bridge to serve OF 1.3 and 1.5
ovs-vsctl set Bridge br0 protocols=OpenFlow13,OpenFlow15

# Verify
ovs-vsctl get Bridge br0 protocols
# returns [OpenFlow13, OpenFlow15]
```

**Connect with a specific version:**

```bash
# Force version 1.3
ovs-ofctl -O OpenFlow13 dump-flows br0

# Force version 1.5
ovs-ofctl -O OpenFlow15 dump-flows br0

# Auto-negotiate (the highest common version)
ovs-ofctl dump-flows br0
```

**Anatomy 5-attribute decode of the HELLO bitmap:**

| Attribute | Sample value | Healthy or warning or critical |
|-----------|-------------|------------------------------|
| `Bridge.protocols` | `[OpenFlow13, OpenFlow15]` | Healthy: at least 1 version. Critical: empty means it does not accept connections. |
| Wire bitmap (controller side) | `0x70` (1.3 plus 1.4 plus 1.5) | Critical: if the intersection with the bridge is empty, the connection closes. |
| Negotiated version | `OpenFlow13` (the highest common version). | Healthy: the highest reasonable value. |
| `ovs-ofctl show` | Lists the supported versions. | Verify. |
| `HELLO_FAILED OFPT_ERROR` | Should not appear. | Critical: a complete version mismatch. |

## 3.4.11 Migration matrix, OF 1.0 to 1.3 to 1.5

### OF 1.0 to 1.3 rule rewrite

**Change required:**

| OF 1.0 syntax | OF 1.3 equivalent | Note |
|---------------|-------------------|------|
| `priority=100,in_port=1,actions=output:2` | Same (compatible). | Identity migration is OK. |
| `priority=100,in_port=1,actions=output:2,output:3,output:4` | Best practice: use a group with `type=all`. | Multicast efficiency. |
| `actions=mod_dl_src:aa:bb:cc:dd:ee:ff` | `actions=set_field:aa:bb:cc:dd:ee:ff->eth_src` | OXM action format. |
| `actions=set_vlan_vid:100` | `actions=push_vlan:0x8100,set_field:0x1064->vlan_vid` | Explicit `push_vlan`. |

**Auto-compatible (no rewrite needed):**
- A single-table flow rule with a basic 12-tuple match.
- Output to a physical port.
- Drop (an empty action list).

**Manual rewrite needed:**
- Multicast (use a group).
- Action set order matters (OF 1.3 uses strict order).
- Field-modification actions (`mod_*` becomes `set_field`).

### OF 1.3 to 1.5 incremental migration

**New features (added only, do not break 1.3):**
- Egress table (optional).
- The `copy_field` action (replacing the OVS NXM `move`).
- The `packet_type` field (for NSH and SFC).

**Auto-compatible:**
- Every 1.3 rule runs on a 1.5 switch.

**Migration choices:**
- Use the egress table when there is per-port output processing (saves rule count).
- Use `copy_field` when working cross-vendor (`move` is OVS-only).
- Use `packet_type` for NSH and SFC research.

## 3.4.12 Decision tree: which version for which deployment

```
+---------------------------+
| What is the use case?     |
+-----------+---------------+
            |
       +----+----+
       |         |
       v         v
+----------+   +-----------+
| Legacy   |   | Production|
| testbed  |   | OVN/OVS   |
+----+-----+   +-----+-----+
     |               |
     v               v
+----------+   +-----------+
| OF 1.0   |   | NSH/SFC   |
| (simple, |   | research? |
| static   |   +--+----+---+
| topo)    |      |    |
+----------+      Y    N
                  |    |
                  v    v
            +-------+ +-------+
            |OF 1.5 | |OF 1.3 |
            |(egress|+|(LTS,  |
            |+packet|  |stable)|
            |_type) |  +-------+
            +-------+
```

**Recommendations:**
- **Lab and research single-controller:** OF 1.0 is enough.
- **OVN production hypervisor:** OF 1.3 (the OVN baseline default) plus 1.5 (for the optional egress table).
- **NSH and SFC research:** OF 1.5 mandatory (`packet_type`-aware).
- **Multi-vendor production:** OF 1.3 is the lowest common denominator.

## 3.4.13 Capstone POE, the engineer proposes a migration plan

**Problem:** a legacy production switch running OVS 2.5 plus OF 1.0. The team wants to upgrade to OVS 2.17 plus OF 1.3 to use OVN. Propose a migration plan.

**Predict:** assume "every 1.0 flow rule currently in production runs on 1.3 without breaking".

**Observe:** audit the production flow table:

```bash
# On the old switch
ovs-ofctl -O OpenFlow10 dump-flows br0 > /tmp/of10-flows.txt

# Classify the rules
grep -E 'mod_dl_|set_vlan' /tmp/of10-flows.txt | wc -l
# returns N rules that need rewriting (mod_* actions)
```

**Explain:**
- The N rules do not break directly because OF 1.3 is backward-compatible for `mod_*` (it still understands and converts them to OXM internally).
- However, if the new controller (OVN) creates rules using OXM standard, having the two stacks coexist on the same switch will produce confusing logs.
- Migration plan: rewrite everything to OXM before switching the controller.

**Falsification test:**
- Run OVS 2.17 with the flag `--protocols=OpenFlow10`. Verify that the 1.0 rules run unchanged.
- Run with `--protocols=OpenFlow10,OpenFlow13`. Verify that both versions coexist.
- Switch to `--protocols=OpenFlow13` only. Verify that the 1.0 rules can still be dumped (auto-converted) but that the new controller sends OXM rules.

## 3.4.14 Anti-patterns and notes

**Anti-pattern 1: mixing `--protocols` for the same deployment.**

`Bridge.protocols=[OpenFlow10, OpenFlow13]` allows two versions to coexist. But controller A using 1.0 rules and controller B using 1.3 rules on the same bridge produces unpredictable behavior (priority overlap, action set conflict).

**Prevention:** standardize the entire deployment on one main version. Use the older version only for ad-hoc debugging.

**Anti-pattern 2: using OVS NXM extensions in production code that is multi-vendor.**

NXM is OVS-only. If you deploy a hardware switch (Pica8, Centec), an NXM rule fails. Use the OXM standard (`set_field` instead of `mod_*`, `copy_field` instead of `move`).

**Note 3: the OF 1.5 egress table has overhead when not needed.**

If the bridge configures an egress table, every packet goes through the egress pipeline whether or not there is a matching rule. Overhead is roughly 1 to 2 percent CPU. Only enable when you actually use the egress feature.

**Note 4: bundles (OF 1.4 and later) have a size limit.**

OFPT_BUNDLE_ADD_MESSAGE is at most 1 message per add. A bundle that is too large (more than 1000 messages) can exceed the switch buffer, and the bundle commit fails. Split into smaller bundles if a large batch update is needed.

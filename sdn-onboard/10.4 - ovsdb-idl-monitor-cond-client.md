## 10.4.1 Why use IDL instead of calling JSON-RPC directly?

If an application needs to track changes in OVSDB, the naive way is polling: every second call `ovsdb-client dump` to fetch the whole database and compare with the previous snapshot. This approach has three serious problems:

**(1) Wasted bandwidth.** The OVN SB DB can hold hundreds of thousands of `Logical_Flow` records (tens of megabytes). Polling once per second means tens of megabytes per second of traffic just to detect one or two changed rows.

**(2) High latency.** A change occurs at T1; the client detects it at T1 plus 1 second (the poll interval). For OVN, which must react in subsecond time, this latency is unacceptable.

**(3) Race condition.** Between two polls, a change can come and go (insert then delete) and the client never sees it. The data view is inconsistent.

The right approach is **subscribing to changes**: the server actively pushes changes to the client over a long-lived connection. OVSDB JSON-RPC has the `monitor` and `monitor_cond` methods to do this. But implementing the protocol directly is complex:

- Parse the JSON-RPC 1.0 notification format.
- Maintain a local copy of the state (the server only sends deltas; the client must track absolute state).
- Handle reconnection plus resynchronisation if the connection is lost.
- Handle the initial sync (the server sends the entire table state on first subscription).
- Race between a write transaction and a monitor callback.

**The IDL (Interface Definition Language) library** handles all of this complexity. The application code only needs:

```python
idl = Idl("tcp:127.0.0.1:6642", SchemaHelper("ovs-sb.ovsschema").get_idl_schema())
while True:
    idl.run()  # process pending messages
    if idl.change_seqno changed:
        # the database state has updated, accessed via idl.tables
        for row in idl.tables["Port_Binding"].rows.values():
            print(row.logical_port, row.chassis)
```

The IDL library handles, automatically: connection setup, registering `monitor_cond`, maintaining the local copy of subscribed tables, notifying on change, reconnecting and resynchronising on a lost connection.

## 10.4.2 The `monitor` protocol versus `monitor_cond`

OVSDB has two distinct subscription protocols, and the difference matters:

**`monitor` (the original RFC 7047 method, OVS 1.x and later).** The client subscribes to a whole table (or a subset of columns). The server pushes:

- **Initial state** at the time of subscription: every existing row in the table.
- **Incremental updates** on every change: insert, modify, or delete of a single row.

Strengths: simple and deterministic. Weakness: no filtering. If a table has 100,000 rows and the client only cares about 100 of them, it still receives all 100,000.

**`monitor_cond` (OVS 2.6 and later, 2016).** Extends `monitor` with a **condition clause**: the server only pushes rows that satisfy the condition. The client can change the condition at runtime via `monitor_cond_change` without resubscribing.

Format of a `monitor_cond` request:

```json
{
  "method": "monitor_cond",
  "params": [
    "OVN_Southbound",
    ["monid_1"],
    {
      "Port_Binding": [
        {
          "columns": ["logical_port", "chassis", "mac"],
          "where": [["chassis", "==", ["named-uuid", "my_chassis_uuid"]]]
        }
      ]
    }
  ],
  "id": 1
}
```

The server pushes only rows of `Port_Binding` whose `chassis` matches `my_chassis_uuid`. Other unrelated rows are not sent, so bandwidth drops by 10 to 100 times in a large cluster.

**Concrete example of why this matters.** An OVN cluster with 1000 chassis times 100 pods per chassis equals 100,000 `Port_Binding` rows. Each `ovn-controller` on chassis X only needs to subscribe to its own `Port_Binding` rows (about 100). Without `monitor_cond`, every chassis would have to receive all 100,000 rows of state plus updates, which is megabytes per second of unnecessary OVSDB traffic. `monitor_cond` reduces this to 100 rows, that is, kilobytes per second.

**`monitor_cond_since` (OVS 2.12 and later).** Adds a `last-txn-id` parameter to resume a subscription after reconnect. The server knows "the client already has state up to transaction X" and only pushes the delta from X onward. Saves bandwidth on a short reconnect.

## 10.4.3 Architecture of the IDL client library

The `python-ovs` IDL (the official library from the OVS project) has the following architecture:

**Component 1, SchemaHelper.** Reads the schema file (`ovs-sb.ovsschema`) and builds a Python class for each table. The application imports the schema class and accesses fields as attributes:

```python
from ovs.db.schema import SchemaHelper
schema_helper = SchemaHelper("/usr/share/ovn/ovn-sb.ovsschema")
schema_helper.register_table("Port_Binding")
schema_helper.register_columns("Port_Binding", ["logical_port", "chassis"])
schema = schema_helper.get_idl_schema()
```

**Component 2, Idl.** The main class that manages the connection plus subscription:

```python
from ovs.db.idl import Idl
idl = Idl("tcp:127.0.0.1:6642", schema)
```

The constructor automatically connects to `ovsdb-server` and sends a `monitor_cond` subscription for the registered tables and columns.

**Component 3, Row objects.** The IDL keeps a local copy as `idl.tables[table_name].rows` (a dictionary of UUID to Row object). To access a row:

```python
for uuid, row in idl.tables["Port_Binding"].rows.items():
    print(f"Port {row.logical_port} on chassis {row.chassis[0].name if row.chassis else 'unbound'}")
```

**Component 4, change notification.** The IDL has a counter `change_seqno` that increments on every state change (insert, modify, delete). The application polls the counter:

```python
last_seqno = idl.change_seqno
while True:
    idl.run()  # process messages
    if idl.change_seqno != last_seqno:
        # something changed, handle the new state
        handle_changes(idl)
        last_seqno = idl.change_seqno
    time.sleep(0.1)
```

Or use `idl.wait()` (block until a change arrives):

```python
poller = Poller()
idl.wait(poller)
poller.block()
```

**Component 5, transaction builder.** The IDL provides a `Transaction` class to write changes back to OVSDB and handles the retry logic automatically (as discussed in Part 10.3):

```python
from ovs.db.idl import Transaction
txn = Transaction(idl)
port = idl.tables["Port_Binding"].rows[some_uuid]
port.chassis = [new_chassis_row]
status = txn.commit_block()
```

## 10.4.4 Conditional replication, changing the condition at runtime

When `ovn-controller` starts on chassis X, it subscribes to `Port_Binding` with `where chassis == X`. But if X is later assigned more `Port_Binding` rows (for example for newly scheduled pods), the condition needs to change without a reconnect.

The `monitor_cond_change` message lets the client update the condition at runtime:

```json
{
  "method": "monitor_cond_change",
  "params": [
    ["monid_1"],
    {
      "Port_Binding": [
        {
          "where": [["chassis", "==", ["named-uuid", "chassis_X"]],
                    ["type", "==", "patch"]]
        }
      ]
    }
  ],
  "id": 2
}
```

The server compares the new condition with the old one:

- A row that matches the new condition but not the old one, the server pushes as an "insert" (even though the row has existed for a long time).
- A row that matches the old condition but not the new one, the server pushes as a "delete".
- A row that matches both, the server does not push (the client already has the state).

This is the mechanism that lets `ovn-controller` "expand" or "narrow" its subscription without tearing down and rebuilding the connection.

**`python-ovs` API:**

```python
from ovs.db.idl import Idl, ConditionRegistry
idl = Idl(...)
idl.cond_change("Port_Binding", [["chassis", "==", ["named-uuid", new_chassis_uuid]]])
```

OVN 22.03 uses `cond_change` extensively. When a pod is scheduled to chassis X, `ovnkube-master` writes `Port_Binding.chassis = X` into the SB DB; `ovn-controller` on X receives the notification and automatically starts processing `Logical_Flow` for that pod.

## 10.4.5 Handling lost connections and resynchronisation

A long-lived monitor connection can be lost (network glitch, `ovsdb-server` restart). The IDL client must:

1. Detect the loss (TCP connection close, heartbeat timeout).
2. Reconnect to the server.
3. Resubscribe to `monitor_cond`.
4. Resynchronise the local state.

There are two resynchronisation strategies:

**Full resynchronisation (the classic).** After reconnect, the server pushes the full state matching the condition again. The client clears its local copy and refills from scratch. Most consistent, but uses a lot of bandwidth in a large cluster.

**Incremental resynchronisation (`monitor_cond_since`, OVS 2.12 and later).** The client remembers the `last_txn_id` before the connection was lost. After reconnecting, it sends `monitor_cond_since` with `last_txn_id`. The server pushes only changes from that transaction id onward. Saves significant bandwidth if the disconnect was short.

**`python-ovs` handles this automatically.** The application code does not have to write the reconnect logic:

```python
idl = Idl("tcp:127.0.0.1:6642", schema)
while True:
    idl.run()  # automatically reconnects if the connection is lost
    time.sleep(1)
```

The IDL exposes `idl.is_lock_contended()` and `idl.change_seqno` so the application can detect a reconnect (the seqno resets to 0 after a full resync).

## 10.4.6 Performance considerations for an IDL client at scale

An IDL client application in a production OVN environment needs tuning to avoid bottlenecks:

**(1) Register the minimum set of columns.** Register only the columns the application actually uses; this reduces bandwidth and the memory footprint of the local copy.

```python
schema_helper.register_columns("Port_Binding", ["logical_port", "chassis"])
# do not register columns the application does not use, such as "options" or "external_ids"
```

**(2) Use a selective condition.** Use the narrowest possible `where` clause:

```python
# Good, only subscribe to Port_Binding for the local chassis
cond = [["chassis", "==", ["named-uuid", local_chassis_uuid]]]

# Bad, subscribe to all Port_Binding rows
cond = []
```

**(3) Process in batches.** Do not handle one change at a time; collect all changes from one `idl.run()` and apply them in batch:

```python
idl.run()
if idl.change_seqno != last_seqno:
    # collect every change since last_seqno
    changes = collect_all_changes(idl, last_seqno)
    apply_batch(changes)  # batch-optimised handling
    last_seqno = idl.change_seqno
```

**(4) Pool connections.** With multiple `ovsdb-server` nodes (Raft cluster), the client can connect to several servers, one as primary and the others as failover. The `python-ovs` `Idl` constructor accepts a list of remote URLs:

```python
idl = Idl("tcp:10.0.0.1:6642,tcp:10.0.0.2:6642,tcp:10.0.0.3:6642", schema)
```

It automatically tries each in turn if the first server is unreachable.

**(5) Memory size of the local copy.** In a large cluster, the local copy can occupy several gigabytes of RAM. Monitor with `top -p <pid>` and tune `column` and `condition` if memory is high. A single OVN-Kubernetes node can hold roughly 2 to 5 GB for the SB DB IDL replica with a 500-plus chassis cluster.

### Guided Exercise 1: write a Python IDL client that watches `Port_Binding`

> **State verification lab:** output is based on `python-ovs` documentation; pending verification on a real lab.

**Objective.** Practise the IDL architecture with a simple client: subscribe to `Port_Binding` and print changes in real time.

**Preparation.** A node running Ubuntu 22.04 with OVS and OVN 22.03 already running, and the `python-ovs` package installed:

```
[host]# sudo apt install python3-ovs
[host]# sudo apt install -y openvswitch-common  # for the schema file
```

**Steps.**

**1.** Create the file `ovn_watch.py`:

```python
#!/usr/bin/env python3
import sys, time
from ovs.db.idl import Idl
from ovs.db.schema import SchemaHelper

SCHEMA = "/usr/share/ovn/ovn-sb.ovsschema"
REMOTE = "tcp:127.0.0.1:6642"

def main():
    helper = SchemaHelper(SCHEMA)
    helper.register_table("Port_Binding")
    helper.register_columns("Port_Binding", ["logical_port", "chassis", "type"])
    helper.register_table("Chassis")
    helper.register_columns("Chassis", ["name"])

    idl = Idl(REMOTE, helper.get_idl_schema())
    print(f"Connected to {REMOTE}, subscribed Port_Binding + Chassis")

    last_seqno = 0
    while True:
        idl.run()
        if idl.change_seqno != last_seqno:
            print(f"\n=== Change detected (seqno {idl.change_seqno}) ===")
            for uuid, pb in idl.tables["Port_Binding"].rows.items():
                chassis_name = pb.chassis[0].name if pb.chassis else "unbound"
                print(f"  Port {pb.logical_port:20s} type={pb.type:12s} chassis={chassis_name}")
            last_seqno = idl.change_seqno
        time.sleep(0.5)

if __name__ == "__main__":
    main()
```

**2.** Run the client:

```
[host]# python3 ovn_watch.py
Connected to tcp:127.0.0.1:6642, subscribed Port_Binding + Chassis
=== Change detected (seqno 1) ===
  Port vm1                  type=              chassis=chassis_1
  Port vm2                  type=              chassis=chassis_2
```

**3.** In another terminal, create a new Logical Switch and Port:

```
[host-B]# ovn-nbctl lsp-add ls1 vm3
[host-B]# ovn-nbctl lsp-set-addresses vm3 "aa:bb:cc:dd:ee:03 10.0.0.3"
```

**4.** Watch the terminal running `ovn_watch.py`:

```
=== Change detected (seqno 2) ===
  Port vm1                  type=              chassis=chassis_1
  Port vm2                  type=              chassis=chassis_2
  Port vm3                  type=              chassis=unbound
```

The new port appears immediately, the IDL has received a `monitor_cond` notification from `ovsdb-server`.

**5.** Bind the port to a chassis:

```
[host-B]# ovn-sbctl chassis-add my_chassis geneve 192.168.1.10
[host-B]# ovn-sbctl lsp-bind vm3 my_chassis
```

Watch the monitoring terminal:

```
=== Change detected (seqno 3) ===
  Port vm3                  type=              chassis=my_chassis
```

**Lessons learned.**

First, the IDL fully abstracts the complexity of `monitor_cond` plus reconnection. The application code is only 30 lines of Python yet provides full real-time subscription.

Second, `change_seqno` is a simple mechanism for change detection. The application polls the counter and does not need callbacks or a complex event loop.

Third, for a production-scale workload, add a `where` clause to the `register_columns` call to filter the subscription. This saves significant bandwidth and memory.

**Cleanup:**

```
[host]# Ctrl+C to stop the client
```

**Preparation.** Same as Exercise 1.

**Steps.**

**1.** Modify `ovn_watch.py` to use `cond_change` to change the condition every 30 seconds:

print("Changing condition to type=vif")

## 10.4.X Real-world incident case (axis 17 backfill 2026-04-27)

### Production case: IDL memory bloat 5 GB on a 500-chassis cluster (curated 2024-Q1 pattern)

**Symptom.** OVN cluster, 500 chassis production deployment of OpenStack Antelope. The `ovn-controller` process on several chassis showed RSS memory of 5 GB or more (versus a baseline of 200 to 400 MB). The OOM killer triggered on hosts with 8 GB physical memory; `ovn-controller` was restarted 2 to 3 times per hour. `Logical_Flow` propagation lag reached 30 to 60 seconds.

**Initial diagnosis.**

- `top -p $(pidof ovn-controller)` showed RSS at 5.2 GB.
- `pmap -x <pid>` showed large heap allocations.
- `ovs-appctl -t ovn-controller memory/show` reported `idl_rows: 850000` (baseline 50,000).

**Root cause investigation.**

The default IDL configuration of `ovn-controller` subscribes to the entire `Port_Binding` plus `Logical_Flow` of the SBDB. In a 500-chassis cluster with around 10,000 ports plus 50,000 `Logical_Flow` per chassis namespace, the total IDL replica per chassis is 250,000 `Port_Binding` rows plus 5 million `Logical_Flow` rows. Each row carries about 1 KB of Python object overhead (a struct dict plus a column dict plus an index map). Total: about 5 GB.

Per Rule 14 source, `lib/ovsdb-idl.c::ovsdb_idl_db_parse_update` per row plus `ovsdb_idl_set_condition` for the filter, verified via MCP `gh api repos/openvswitch/ovs/contents/lib/ovsdb-idl.c?ref=v2.17.9`. The memory bloat is fundamentally a replicate-not-filter pattern.

**Root cause confirmation.**

- `ovs-appctl -t ovn-controller idl/show` reports the per-table row count breakdown.
- Cross-validate via `ovsdb-client monitor SBDB Port_Binding | wc -l` for the server-side row count.

**Resolution.**

1. Configure `ovn-controller` with `external_ids:ovn-monitor-all=false` (the default of `true` is for backward compatibility) to enable conditional monitoring.
2. Set `ovn-monitor-condition` to a chassis-specific value: `ovn-controller` only subscribes to `Port_Binding` with `chassis == @this_chassis` and to `Logical_Flow` with `datapath_groups @containing this_chassis`.
3. Restart the `ovn-controller` fleet in a rolling fashion. Post-fix RSS dropped from 5 GB to 350 MB, and OOM kill incidents reached zero.

**Lessons learned.**

- The default IDL subscribe-all pattern (designed for small clusters) breaks at production scale beyond 100 chassis. Conditional monitoring (`monitor_cond` plus `monitor_cond_change`) is mandatory at scale.
- Memory profiling before deployment with realistic row counts is mandatory. Default OOM at more than 10 times baseline is a sign of the subscribe-all problem.
- Cross-link: Part 10.4 monitor_cond protocol section, Part 10.0 §10.0.14.3 (monitor_cond protocol full spec), case study 10.0.X (monitor fanout, the same pattern on the server side).

> **Curated reproduction note.** The case is curated from Red Hat OVN-Kubernetes scale-testing reports (2023 to 2024) plus OVN mailing-list mainline discussions on the `ovn-monitor-all` deprecation rationale. The default flip from `true` to `false` was proposed in OVN 23.06 and settled as the default in OVN 24.03.

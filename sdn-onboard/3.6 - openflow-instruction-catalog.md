# 3.6. OpenFlow instruction catalog (6 instructions)

> **Status:** Updated 2026-04-26.

## Learning objectives

After this part, you can:

1. Distinguish an instruction (a set of instructions) from an action (a set or list of actions).
2. State the order of evaluation in the pipeline: Apply-Actions, Clear-Actions, Write-Actions, Write-Metadata, Goto-Table, Meter.
3. Explain when to use which instruction (Apply versus Write semantics).
4. Debug instruction misuse with `ofproto/trace`.

## Prerequisites

- Part 3.1 (OF 1.0 specification), Part 4.0 (OF 1.1 multi-table and groups), Part 4.7 (OVS programming), Part 9.22 (multi-table pipeline).

## §3.6.1 Apply-Actions instruction

**Concept.** This instruction executes an action list immediately (modify the packet now). Type: OF instruction, OF 1.1 and later.

**Importance.** The most-used instruction in OF flow rules.

**How it works.** Apply-Actions specifies an action list that executes in order on the current packet copy. The modifications are visible to downstream instructions in the same flow rule.

**Version differences.** Introduced in OF 1.1. Stable since.

**Real-world incident.** In production, a typical flow rule has Apply-Actions with `output`, `set_field`, and `dec_ttl`. This is a common pattern.

**Synthetic exercise.** `ovs-ofctl add-flow br0 'priority=100,ip,actions=set_field:00:11:22:33:44:55->eth_dst,output:1'`.

**Comparison with other systems.** Cisco IOS class-map immediate action is a similar concept.

---

## §3.6.2 Clear-Actions instruction

**Concept.** This instruction clears the accumulated action set. Type: OF instruction, OF 1.1 and later.

**Importance.** Critical for resetting state in a complex multi-table pipeline.

**How it works.** Reset the action set to empty. Useful before re-evaluating in a different table context.

**Version differences.** Introduced in OF 1.1.

**Real-world incident.** Pipeline reset case: table 0 sets the action set, but table 5 wants a different action. Use Clear-Actions to reset before a new Write-Actions.

**Synthetic exercise.** `actions=clear_actions,goto_table:5`.

**Comparison with other systems.** Cisco IOS "no inheritance" is a similar concept.

---

## §3.6.3 Write-Actions instruction

**Concept.** This instruction merges an action list into the action set (deferred execution at pipeline end). Type: OF instruction.

**Importance.** The action set deferred-execution model is fundamental.

**How it works.** Write-Actions merges actions into the action set. The action set is deduplicated (only one action per type, for example, only one `output`). Final execution happens after the pipeline ends.

**Version differences.** Introduced in OF 1.1.

**Real-world incident.** In production, most flows with a `goto_table` chain use Write-Actions to defer the output until the last table.

**Synthetic exercise.** `actions=write_actions(output:1),goto_table:1`.

**Comparison with other systems.** Cisco IOS deferred-queue action is an analog.

---

## §3.6.4 Write-Metadata instruction

**Concept.** This instruction writes to a 64-bit metadata register, which passes between tables. Type: OF instruction.

**Importance.** Metadata persistence across tables is critical for a multi-stage pipeline.

**How it works.** Write-Metadata sets a value and a mask:
```
write_metadata:0x12345678/0xffff0000
```
Lower bits stay unchanged, upper bits are set to 0x1234.

**Version differences.** Introduced in OF 1.1.

**Real-world incident.** In production, Write-Metadata is used to tag a packet across pipeline stages (for example, a tenant ID tag).

**Synthetic exercise.** `actions=write_metadata:0x1,goto_table:1`, then in table 1 match `metadata=0x1`.

**Comparison with other systems.** Cisco IOS forwarding-class is a similar concept.

---

## §3.6.5 Goto-Table instruction

**Concept.** This instruction continues the pipeline at the specified next table. Type: OF instruction.

**Importance.** The foundation of the multi-table pipeline.

**How it works.** `goto_table:N` advances to table N. N must be greater than the current table (forward only). The OVS extension `resubmit` allows backward jumps.

**Version differences.** Introduced in OF 1.1. OF 1.5 added a `packet_type`-aware variant.

**Real-world incident.** The standard 5-table OVS pipeline uses a `goto_table` chain (cited in Part 9.22).

**Synthetic exercise.** `actions=goto_table:1`, then in table 1 add another rule.

**Comparison with other systems.** Cisco IOS table chains via class-map are similar.

---

## §3.6.6 Meter instruction

**Concept.** This instruction redirects the packet to a meter for rate limiting or DSCP remarking before continuing. Type: OF instruction, OF 1.3 and later.

**Importance.** The foundation of QoS and DSCP handling in OpenFlow.

**How it works.** `meter:N` invokes meter table entry N. The meter has bands (DROP, DSCP_REMARK) based on a rate threshold.

**Version differences.** Introduced in OF 1.3. Enhanced in OF 1.5.

**Real-world incident.** In production, rate-limit egress traffic per VM via a meter.

**Synthetic exercise.** `ovs-ofctl add-meter br0 meter=1,kbps,band=type=drop,rate=10000` and `actions=meter:1,output:1`.

**Comparison with other systems.** Cisco IOS policer or shaper is an analog. Linux `tc` qdisc is a similar concept.

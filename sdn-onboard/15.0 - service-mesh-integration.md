## 15.0.1 Mô hình kiến trúc service mesh

Service mesh xuất hiện khoảng 2016-2017 từ nhu cầu cụ thể của kiến trúc microservice: khi một ứng dụng monolith bị chia thành 50-500 microservice, mỗi service gọi sang service khác qua mạng, tổng số lượt đi qua mạng tăng lên gấp trăm lần so với monolith. Mỗi lần gọi cần: mTLS (mã hóa + xác thực), retry với exponential backoff, timeout, circuit breaker (ngắt khi downstream hỏng nhiều), cân bằng tải, distributed tracing, và metrics. Nếu mỗi service tự hiện thực logic này (dùng thư viện như Hystrix, Resilience4j), đội ngũ phải duy trì tính nhất quán qua nhiều ngôn ngữ (Go + Java + Python + Node.js), không khả thi.

Giải pháp service mesh: **tách logic network policy khỏi mã ứng dụng, đưa vào sidecar proxy chạy song song với app**. Mô hình này gồm hai thành phần: (1) **Data plane**, proxy (Envoy, Linkerd2-proxy, hoặc chương trình eBPF) chặn tất cả traffic vào/ra service, áp dụng policy, chuyển tiếp. (2) **Control plane**, service (istiod, destination controller của Linkerd) phát cấu hình xDS hoặc policy xuống data plane, tổng hợp metrics từ data plane lên stack quan sát.

Các tính năng cốt lõi mà mọi service mesh cung cấp: **mTLS termination** (xác thực dựa trên danh tính giữa service, thay thế firewall mức mạng), **định tuyến L7** (định tuyến traffic theo HTTP header, path, method, không chỉ IP/port), **circuit breaker** (ngắt outbound khi tỷ lệ lỗi vượt ngưỡng, bảo vệ downstream khỏi sự cố dây chuyền), **retry với backoff kèm jitter** (tránh thundering herd), **giới hạn tốc độ** (hạn ngạch per service/per consumer), **distributed tracing** (truyền trace context qua chuỗi gọi, sinh span cho quan sát), **metrics** (tương thích Prometheus theo chuẩn RED — Rate, Error, Duration).

Khác biệt service mesh so với API gateway: API gateway là **north-south** (ngoài → cluster), service mesh là **east-west** (service-to-service trong cluster). Do anh nghiệp ở môi trường production thường triển khai cả hai — API gateway ở biên cho traffic bên ngoài, service mesh cho traffic nội bộ.

## 15.0.2 Istio với Envoy sidecar + xDS

**Istio** là service mesh phổ biến nhất hiện nay, bắt đầu bởi Google + IBM + Lyft năm 2017 (dựa trên Envoy proxy của Lyft). Istio tốt nghiệp CNCF năm 2022. Kiến trúc gồm hai phần: (1) **Istiod**, control plane hợp nhất từ 3 binary cũ (Pilot + Citadel + Galley) thành một binary từ Istio 1.5 (2020). Istiod phát cấu hình xDS, quản lý chứng chỉ mTLS, kiểm chứng CRD. (2) **Envoy sidecar**, proxy data plane được chèn vào mỗi pod dưới dạng container thứ hai. Traffic vào pod bị `iptables` chuyển hướng sang Envoy port 15001; Envoy áp dụng policy rồi chuyển tiếp sang app container trên localhost.

**Giao thức xDS** là nền tảng giao tiếp control plane ↔ data plane của Envoy. xDS gồm nhiều sub-protocol: **LDS** (Listener Discovery Service, cấu hình port Envoy lắng nghe), **RDS** (Route Discovery, quy tắc route HTTP), **CDS** (Cluster Discovery, danh sách upstream cluster), **EDS** (Endpoint Discovery — IP của pod trong mỗi cluster), **SDS** (Secret Discovery, chứng chỉ TLS), **HCM** (cấu hình HTTP Connection Manager). Envoy kết nối ban đầu → yêu cầu cấu hình xDS → control plane đẩy cấu hình mới khi có thay đổi (pod co giãn, service được thêm) qua luồng gRPC. Streaming giúp cập nhật cấu hình ở quy mô hàng chục nghìn endpoint đạt độ trễ dưới giây.

**CRD (Custom Resource Definition)** mà Istio cung cấp cho người dùng gồm: (1) **VirtualService**, định nghĩa quy tắc route (match HTTP header → destination); (2) **DestinationRule**, cấu hình subset, policy cân bằng tải, connection pool per destination; (3) **Gateway**, cấu hình proxy biên ingress/egress; (4) **PeerAuthentication**, bật/tắt mTLS per namespace; (5) **AuthorizationPolicy** — RBAC L7 (ai được gọi ai). Các CRD này được biên dịch sang cấu hình Envoy qua istiod.

**Istio Ambient Mesh** (ra mắt 2022, GA 2024) là chế độ sidecar-less: thay vì mỗi pod có một Envoy, Istio triển khai hai loại proxy dùng chung — **ztunnel** (per-node, xử lý L4 + mTLS) và **waypoint** (per-namespace hoặc per-service-account, xử lý L7). Ưu điểm: giảm overhead sidecar (một node với 100 pod không cần 100 tiến trình Envoy), nâng cấp mesh không cần khởi động lại pod. Nhược điểm: độ phức tạp cao hơn, gỡ lỗi khó hơn.

## 15.0.3 Linkerd với linkerd2-proxy

**Linkerd** là service mesh đầu tiên đặt ra thuật ngữ này (2016, ban đầu dựa trên Finagle của Twitter chạy trên JVM, sau đó viết lại hoàn toàn bằng Rust năm 2018 thành Linkerd 2.x). Tốt nghiệp CNCF năm 2021, trước Istio. Linkerd triết lý khác Istio: ưu tiên **sự đơn giản hơn tính năng**.

Kiến trúc Linkerd: (1) **linkerd2-proxy**, sidecar proxy viết bằng Rust, tiêu thụ khoảng 30 MB memory per pod (so với Envoy khoảng 100 MB); (2) **Control plane**, component destination, identity, proxy-injector; (3) **Linkerd CLI**, phần lớn thao tác qua CLI, không qua CRD phức tạp.

So sánh cụ thể với Istio: (1) **Dấu chân tài nguyên**, sidecar Linkerd khoảng 30 MB so với Envoy khoảng 100 MB (nhỏ hơn 3-4 lần), overhead độ trễ Linkerd khoảng 1ms p99 so với Envoy khoảng 5ms p99. Lý do: binary Rust compile không có GC, Envoy dùng C++ nhưng có overhead khả năng mở rộng plugin. (2) **Độ phức tạp CRD** — Linkerd chỉ có ServiceProfile + TrafficSplit (khoảng 2 CRD chính); Istio có hơn 10 CRD. (3) **Phạm vi tính năng** — Linkerd tập trung L7 HTTP/gRPC với knob tối thiểu, Istio hỗ trợ cả TCP không phải HTTP, mở rộng WASM, plugin Istio CNI. (4) **Độ dốc học tập** — Linkerd dễ học (vài giờ), Istio cần học vài tuần.

**Linkerd-proxy** không hỗ trợ plugin WASM (khác Envoy), sự đánh đổi để có ổn định và hiệu năng. Tinh thần Linkerd: nếu bạn cần tính năng không có, bạn có thể không cần service mesh cho trường hợp đó. Linkerd phù hợp startup và cluster cỡ trung; Istio phù hợp doanh nghiệp có yêu cầu policy phức tạp.

**Buoyant Cloud** là SaaS do Buoyant (công ty tạo Linkerd) cung cấp quan sát đa cluster, audit log, và hỗ trợ thương mại. Linkerd open-source vẫn miễn phí dưới giấy phép Apache 2.0.

## 15.0.4 Service mesh dựa trên eBPF của Cilium

**Cilium** không bắt đầu như service mesh, nó là **plugin CNI của Kubernetes** ra đời 2016 bởi Isovalent, tập trung network policy với eBPF thay vì iptables. Năm 2021, Cilium giới thiệu **Cilium Service Mesh**, biến thể không cần sidecar, thay vào đó dùng chương trình eBPF gắn vào socket layer của kernel (socket redirect, tc ingress/egress, XDP).

Kiến trúc khác biệt so với Istio/Linkerd: (1) **Không có sidecar**, traffic L4 (TCP/UDP) được xử lý hoàn toàn bởi chương trình eBPF trong kernel, không qua userspace proxy. Nghĩa là packet TCP từ app → eBPF trực tiếp → app khác, không quay vòng qua userspace. Độ trễ thấp hơn đáng kể (dưới một mili giây) và không tốn memory per pod cho sidecar. (2) **L7 chỉ khi cần**, khi cần L7 (định tuyến HTTP, circuit breaker), Cilium triển khai **Envoy dùng chung** per-node, không per-pod. Traffic app được chuyển hướng lên Envoy chỉ khi policy yêu cầu xử lý L7.

**Hubble** là stack quan sát của Cilium. Hubble chạy trên cùng data plane (eBPF), cung cấp flow log chi tiết đến giây (từng connection, từng HTTP request), bản đồ service thời gian thực, log phán quyết network policy. Hubble UI cung cấp trực quan hóa tương tự Kiali của Istio nhưng phạm vi rộng hơn vì Cilium quản lý cả traffic L3/L4 không qua service mesh.

**Khi nào dùng Cilium Service Mesh:** (1) cluster yêu cầu cao về hiệu năng (độ trễ p99 dưới 2ms cho service call), (2) đã dùng Cilium CNI, muốn hợp nhất data plane, (3) không cần nhiều policy L7 (chủ yếu mTLS + quan sát), (4) không cần plugin WASM. **Khi nào không dùng:** cần quy tắc định tuyến L7 phức tạp (canary theo header), cần hệ sinh thái plugin Envoy rộng, hoặc đội ngũ chưa quen gỡ lỗi eBPF (bpftool, Hubble CLI).

Cilium tốt nghiệp CNCF tháng 10/2023, cùng thời điểm với Linkerd. Công ty Isovalent (thành lập 2017) được **Cisco mua lại tháng 4/2024** với giá không công bố.

## 15.0.5 OVN-Kubernetes CNI

**OVN-Kubernetes** (thường viết tắt **ovn-kubernetes** hoặc **ovn-k8s**) là plugin CNI dùng OVN (từ Block VII-X) làm hạ tầng mạng cho Kubernetes. Khởi đầu 2017 bởi Red Hat + Canonical, hiện là CNI mặc định của OpenShift (Kubernetes doanh nghiệp của Red Hat) từ phiên bản 4.12 (2023), thay thế OpenShift SDN cũ hơn.

Kiến trúc OVN-Kubernetes gồm: (1) **ovnkube-master**, pod control plane chạy trên master node, theo dõi Kubernetes API (Pod, Service, NetworkPolicy, Namespace) và dịch sang OVN northbound DB (logical switch, logical router, ACL). (2) **ovnkube-node** — DaemonSet chạy trên mỗi worker node, quản lý OVS bridge + ovn-controller local, cấp phát pod interface (veth pair + OVS port). (3) **ovn-northd + ovn-sb**, stack OVN chuẩn chạy trong cluster, northd dịch NB → SB flow.

**Dịch NetworkPolicy → OVN ACL:** khi người dùng áp dụng `NetworkPolicy` (tài nguyên K8s), ovnkube-master parse quy tắc và sinh OVN ACL entries trong NB DB. OVN ACL được phân phối tới SB DB, ovn-controller trên node cài OpenFlow rule vào OVS. Kết quả: thực thi policy ở data plane OVS + conntrack, không qua iptables của kube-proxy. Hiệu năng tốt hơn iptables khi pod thay đổi nhiều (iptables mất vài giây để build lại khi có hơn 10K quy tắc, OVS cập nhật gia tăng dưới giây).

**Thay thế kube-proxy:** OVN-K8s có thể thay thế hoàn toàn kube-proxy — ClusterIP Service được hiện thực qua OVN load balancer (bảng Load_Balancer trong NB DB), không cần iptables NAT chain. Đây là tính năng của nhiều CNI hiện đại (Cilium không cần kube-proxy, Calico eBPF).

**Khả năng tương tác với service mesh:** OVN-K8s cung cấp network policy L3/L4, nhưng **không phải service mesh**, không có mTLS, định tuyến L7, circuit breaker. Có thể triển khai Istio/Linkerd/Cilium Service Mesh **trên nền** OVN-K8s — L2/L3 nền tảng do OVN xử lý, policy L7 mesh do Istio/Linkerd xử lý. Kết hợp này phổ biến trong OpenShift: OVN-K8s CNI + Red Hat Service Mesh (bản phân phối Istio). Cilium Service Mesh không tương thích với OVN-K8s vì Cilium cần data plane riêng (eBPF) trong khi OVN-K8s dùng datapath OVS.

### ▶ Guided Exercise 1: Deploy Istio + Envoy sidecar trên minikube

> **Trạng thái lab verification:** Output dựa trên Istio docs 1.20+; chờ verify lab thực.

**Mục đích:** Triển khai Istio service mesh vào Kubernetes cluster local, chèn Envoy sidecar vào ứng dụng mẫu (bookinfo), quan sát đồ thị service với Kiali, chứng minh mTLS tự động chèn giữa các sidecar.

**Chuẩn bị:**
- Ubuntu 22.04 VM, 8 GB RAM, 4 CPU, 20 GB disk
- Install dependencies: Docker, kubectl, minikube (Install commands in production lab setup).

**Mô hình lab:**

```
┌────────────────────────────────────────────────────┐
│             minikube VM (Ubuntu 22.04)             │
│                                                    │
│  Kubernetes cluster:                               │
│                                                    │
│  ns istio-system:                                  │
│    istiod + Kiali + Prometheus                     │
│                                                    │
│  ns default (istio-injection=enabled):             │
│  ┌──────────────────┐  ┌────────────────┐          │
│  │ productpage pod  │  │ details pod    │          │
│  │  ├ app container │  │  ├ app         │          │
│  │  └ istio-proxy   │  │  └ istio-proxy │          │
│  │   (Envoy SC)     │  │   (Envoy SC)   │          │
│  └────────┬─────────┘  └────────▲───────┘          │
│           │   mTLS auto-injected │                 │
│           └──────────────────────┘                 │
│                                                    │
│  Similar cho reviews-v1/v2/v3, ratings             │
└────────────────────────────────────────────────────┘
```

**Bước thực hiện:**

**1.** Start minikube:

```bash
[host]$ minikube start --cpus=4 --memory=8192 --driver=docker
```

Output kỳ vọng: `kubectl is now configured to use "minikube" cluster`.

**2.** Install istioctl + Istio:

```bash
[host]$ curl -L https://istio.io/downloadIstio | sh -
[host]$ cd istio-1.20.*/bin && export PATH=$PWD:$PATH
[host]$ istioctl install --set profile=demo -y
```

Output kỳ vọng: `✔ Installation complete`.

**3.** Enable sidecar injection:

```bash
[host]$ kubectl label namespace default istio-injection=enabled
```

**4.** Deploy bookinfo:

```bash
[host]$ kubectl apply -f samples/bookinfo/platform/kube/bookinfo.yaml
[host]$ kubectl apply -f samples/bookinfo/networking/bookinfo-gateway.yaml
```

**5.** Verify sidecar injected (mỗi pod 2/2):

```bash
[host]$ kubectl get pods
```

Output kỳ vọng:

```
NAME                             READY   STATUS
details-v1-xxxxx                 2/2     Running
productpage-v1-xxxxx             2/2     Running
reviews-v1-xxxxx                 2/2     Running
reviews-v2-xxxxx                 2/2     Running
reviews-v3-xxxxx                 2/2     Running
ratings-v1-xxxxx                 2/2     Running
```

`2/2` = app container + istio-proxy sidecar ✓

**6.** Install Kiali + generate traffic:

```bash
[host]$ kubectl apply -f samples/addons/kiali.yaml
[host]$ istioctl dashboard kiali &
[host]$ export INGRESS_HOST=$(minikube ip)
[host]$ for i in $(seq 1 100); do curl -s http://$INGRESS_HOST/productpage > /dev/null; done
```

**Output mong muốn:** Kiali Graph view tại `http://localhost:20001/kiali`:
- 6 nodes: productpage → reviews-v1/v2/v3 → ratings, productpage → details
- mTLS padlock icons trên mỗi edge
- Traffic ~10-50 req/s, response time p50 ~5-20ms

**Bài học học được:**

Thứ nhất, **chèn sidecar trong suốt** với developer, chỉ gán nhãn namespace, không sửa code. Envoy chặn traffic qua quy tắc iptables được đặt bởi container istio-init.

Thứ hai, **mTLS tự động**: Istio bootstrap certs qua SDS (Secret Discovery Service) từ istiod. Zero-config encryption giữa services.

Thứ ba, **overhead cost**: mỗi pod +50-100 MB RAM + ~1 CPU core per sidecar. Scale 1000+ pods → overhead compound. Đây là động lực cho Cilium SM (Part 15.2) và Istio ambient mode.

**Cleanup:**

```bash
[host]$ kubectl delete -f samples/bookinfo/platform/kube/bookinfo.yaml
[host]$ istioctl uninstall --purge -y
[host]$ minikube delete
```

### ▶ Lab 2: Benchmark Istio vs Cilium vs no-mesh latency

> **Trạng thái lab verification:** Output số liệu dựa trên Cilium benchmark reports 2021-2023; chờ verify lab thực.

**Mục đích:** Đo định lượng overhead sidecar, so sánh 3 chế độ mesh (không mesh, Istio sidecar, Cilium SM) trên cùng workload (bookinfo).

**Chuẩn bị:**
- 16 GB RAM VM (3 kind clusters parallel)
- Install `kind`: `curl -Lo ./kind https://kind.sigs.k8s.io/dl/v0.20.0/kind-linux-amd64 && sudo install kind /usr/local/bin/`
- Install `wrk`: `sudo apt install wrk`

**Mô hình lab:**

```
┌───────────────────────────────────────────────────┐
│       Ubuntu 22.04 VM (16 GB RAM)                 │
│                                                   │
│  kind cl A       kind cl B       kind cl C        │
│  (no mesh)       (Istio SC)      (Cilium SM)      │
│  bookinfo        bookinfo        bookinfo         │
│  1/1 per pod     2/2 per pod     1/1 + node-level │
│     │                │                │           │
│     └────────────────┴────────────────┘           │
│                      │                            │
│                wrk benchmark                      │
│          (4 threads, 100 conn, 30s)               │
└───────────────────────────────────────────────────┘
```

**Bước thực hiện:**

**1.** Create 3 kind clusters:

```bash
[host]$ for name in a b c; do kind create cluster --name $name; done
```

**2.** Cluster A (baseline no-mesh):

```bash
[host]$ kubectl --context kind-a apply -f istio-*/samples/bookinfo/platform/kube/bookinfo.yaml
```

**3.** Cluster B (Istio sidecar):

```bash
[host]$ kubectl config use-context kind-b
[host]$ istioctl install --set profile=default -y
[host]$ kubectl label namespace default istio-injection=enabled
[host]$ kubectl apply -f istio-*/samples/bookinfo/platform/kube/bookinfo.yaml
```

**4.** Cluster C (Cilium SM):

```bash
[host]$ kubectl config use-context kind-c
[host]$ cilium install --version 1.15.0 --set mesh.enabled=true
[host]$ cilium status --wait
[host]$ kubectl apply -f istio-*/samples/bookinfo/platform/kube/bookinfo.yaml
```

**5.** Port-forward mỗi cluster:

```bash
# 3 terminals, 1 per cluster
[host]$ kubectl --context kind-a port-forward svc/productpage 9080 &
[host]$ kubectl --context kind-b port-forward svc/productpage 9081:9080 &
[host]$ kubectl --context kind-c port-forward svc/productpage 9082:9080 &
```

**6.** Benchmark:

```bash
[host]$ for port in 9080 9081 9082; do
         echo "=== Port $port ==="
         wrk -t4 -c100 -d30s http://localhost:$port/productpage
       done
```

**Output mong muốn:**

```
=== Port 9080 (no-mesh) ===
  Latency   15.23ms  ±5.18ms
  Requests/sec:   6428.34

=== Port 9081 (Istio sidecar) ===
  Latency   48.9ms   ±18.3ms
  Requests/sec:   2128.12

=== Port 9082 (Cilium SM) ===
  Latency   22.4ms   ±7.9ms
  Requests/sec:   4412.51
```

Bảng kết quả:

| Mesh Mode | Avg Latency | Req/sec | Overhead vs no-mesh |
|-----------|-------------|---------|---------------------|
| No-mesh | 15 ms | 6,400 | baseline |
| Istio SC | 49 ms | 2,100 | **3.3× latency, 67% req/s loss** |
| Cilium SM | 22 ms | 4,400 | 1.5× latency, 31% req/s loss |

**Bài học học được:**

Thứ nhất, **Istio sidecar overhead thực tế rất đáng kể** với bookinfo (7 services, 10+ RPC/request). Mỗi request đi qua 2 Envoy hops (out + in) → latency compound.

Thứ hai, **Cilium SM lợi thế rõ**: shared Envoy per-node, không duplicate state, eBPF L7 skip cho simple traffic. Tiết kiệm 30-50% overhead.

Thứ ba, **decision framework**: chỉ adopt mesh khi benefits (mTLS, observability, traffic mgmt) outweigh overhead. Latency-sensitive apps (trading, gaming) → sidecar-less approach essential.

**Cleanup:**

```bash
[host]$ pkill kubectl
[host]$ for name in a b c; do kind delete cluster --name $name; done
```

2. [Envoy Documentation](https://www.envoyproxy.io/docs), CNCF Envoy project. xDS protocol + filter chain reference.
3. [Linkerd2 Documentation](https://linkerd.io/2/overview/), Buoyant Linkerd project. Linkerd2 architecture.
4. [Cilium Service Mesh](https://cilium.io/use-cases/service-mesh/), Cilium project. eBPF-based service mesh without sidecar.
5. [OVN-Kubernetes Documentation](https://ovn-kubernetes.io/), OVN-Kubernetes project. OVN CNI for K8s với NetworkPolicy translation.
6. [CNCF Service Mesh Landscape](https://landscape.cncf.io/), CNCF. Current state of service mesh ecosystem.

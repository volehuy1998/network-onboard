## Why this file exists

A learner who has never read autoconf or automake meets the OVS source tree and sees a layered system: a `boot.sh` script, a `configure.ac` file with hundreds of macro calls, a top-level `Makefile.am` that reads twenty-six `automake.mk` fragments, a `config.h` file generated at configure time, and a `Makefile` produced by `./configure` that orchestrates the whole build. The learner who tries to skip the build system and start reading `lib/dpif-netdev.c` directly will hit `#ifdef HAVE_DPDK` blocks they cannot evaluate, will not understand why `lib/odp-netlink.h` is generated rather than checked in, and will be unable to add a new source file because they do not know which `automake.mk` fragment to edit.

S0 fixes this by walking the build system end to end. After reading S0 the learner can answer four questions without looking anything up:

- What does `./boot.sh && ./configure && make && make install` do at each step.
- Which file produces which binary.
- How the kernel module is built against the running kernel.
- Where the OVSDB schema headers come from and why they are generated rather than checked in.

S0 does not teach automake or autoconf in general. It teaches the specific autotools usage OVS has chosen. A reader who wants the full GNU build system reference goes to the upstream automake and autoconf manuals. A reader who wants to understand OVS reads S0.

## 1. The four-step build, in one paragraph

The OVS build is canonically four steps: `./boot.sh`, `./configure ...`, `make`, `make install`. Step one regenerates the autotools-managed files from the developer-checked-in inputs (`configure.ac`, `Makefile.am`, every `automake.mk` fragment). Step two runs the regenerated `configure` script which detects the build host's compiler, libraries, and kernel headers, writes a `config.h` describing what was found, and produces a `Makefile`. Step three runs that Makefile, which compiles every C source file into an object, links the objects into the static library `lib/libopenvswitch.la`, then links each binary (`ovs-vsctl`, `ovs-ofctl`, `ovs-vswitchd`, `ovsdb-server`, others) against that library. Step four copies the produced binaries, headers, scripts, schema files, and man pages into the configured prefix.

Optional step five `make modules_install` builds the kernel module against the running kernel's headers (path supplied by `--with-linux=` to `configure`) and copies the resulting `openvswitch.ko` into `/lib/modules/$(uname -r)/updates/`.

Every step is examined in the rest of this file.

## 2. boot.sh, two lines that matter

`boot.sh` is two lines:

```text
#! /bin/sh
autoreconf --install --force
```

`autoreconf` is the GNU helper that runs the autotools chain in the right order: `aclocal` reads the project's `acinclude.m4` plus any third-party `m4/*.m4` macros and writes `aclocal.m4`; `autoheader` builds the `config.h.in` template from `AC_DEFINE` calls in `configure.ac`; `automake` reads every `*.am` and writes the corresponding `*.in`; `autoconf` reads `configure.ac` and writes `configure`. The `--install` flag fills in any missing helper scripts (`install-sh`, `missing`, `depcomp`); `--force` rebuilds even if the inputs look up to date.

A learner running `./boot.sh` for the first time should expect twenty seconds of disk activity and a fully populated set of generated files. The output is silent on success; failure produces messages naming the missing tool. The most common failure on a fresh machine is a missing `pkg-config` (`PKG_PROG_PKG_CONFIG` in `configure.ac` line 35). The second most common is a missing `libtool` (`LT_INIT` in line 43; the `m4_pattern_forbid` line immediately below it makes autoconf fail loudly when libtool is not installed, instead of silently producing a broken `configure`).

After `boot.sh` succeeds, the developer-checked-in files plus the generated files coexist in the source tree. Building from the source tarball skips this step because the tarball ships the generated files; building from a git clone always runs `boot.sh` first.

## 3. configure.ac, the hand-written input

`configure.ac` at `v2.17.9` is 227 lines. The structure is conventional autoconf:

- **Lines 15 to 22, the project prologue.** `AC_INIT(openvswitch, 2.17.9, bugs@openvswitch.org)` declares the project name, version, and bug-report address. The version `2.17.9` is the single source of truth for the package version; every other place that prints the version reads it from here. `AC_CONFIG_SRCDIR([datapath/datapath.c])` names a sentinel file the configure script uses to verify it is being run inside a real OVS source tree; if the file is missing, configure errors out. `AC_CONFIG_MACRO_DIR([m4])` tells autoconf where the project-local m4 macros live. `AC_CONFIG_HEADERS([config.h])` says configure should write a `config.h` summarising what it detected. `AM_INIT_AUTOMAKE([tar-pax])` activates automake with the modern tar archive format.

- **Lines 24 to 35, compiler probing.** `AC_PROG_CC` (or `AC_PROG_CC_C99` on older autoconf) detects the C compiler and verifies it speaks at least C99. `AC_PROG_CXX` does the same for the C++ compiler (used only for a few test files). `PKG_PROG_PKG_CONFIG` finds the `pkg-config` binary used to locate libraries.

- **Lines 39 to 44, system configuration.** `AC_USE_SYSTEM_EXTENSIONS` enables glibc extensions like `_GNU_SOURCE`. `AC_C_BIGENDIAN` detects byte order and `#define`s `WORDS_BIGENDIAN` if the host is big-endian. `AC_SYS_LARGEFILE` enables 64-bit file offsets on 32-bit systems. `LT_INIT([disable-shared])` activates libtool with the project-wide preference for static linking.

- **Lines 70 to 79, library versioning.** The `libopenvswitch_lt_current` plus `libopenvswitch_lt_revision` plus `libopenvswitch_lt_age` triplet is the libtool ABI versioning scheme. Maintainers update these numbers when the public ABI of `libopenvswitch.la` changes; the comment block immediately above (lines 46 to 68) explains the rules.

- **Lines 81 to 88, basic library probing.** `AC_SEARCH_LIBS([pow], [m])` checks whether `pow` is in libc or whether the math library `-lm` is needed; the check writes the result to `$LIBS`. The same pattern probes `clock_gettime`, `timer_create`, `pthread_rwlock_tryrdlock`, and other functions that may live in a separate library on some platforms.

- **Lines 90 to 146, OVS-specific feature probing.** This is where the project's hand-written `OVS_CHECK_*` macros run. Each macro is defined in `acinclude.m4` (1510 lines) and runs one targeted feature test. The probe set covers: the Windows toolchain (`OVS_CHECK_WIN64`, `OVS_CHECK_WIN32`, `OVS_CHECK_VISUAL_STUDIO_DDK`), code-coverage instrumentation (`OVS_CHECK_COVERAGE`), USDT probes (`OVS_CHECK_USDT`), the netlink library (`OVS_CHECK_NETLINK`), OpenSSL (`OVS_CHECK_OPENSSL`), libcap-ng for capability dropping (`OVS_CHECK_LIBCAPNG`), the unbound DNS resolver (`OVS_CHECK_UNBOUND`), the unwind library (`OVS_CHECK_UNWIND`), Python 3 (`OVS_CHECK_PYTHON3`), the Sphinx documentation toolchain (`OVS_CHECK_SPHINX`), AF_XDP (`OVS_CHECK_LINUX_AF_XDP`), the kernel headers (`OVS_CHECK_LINUX`), DPDK (`OVS_CHECK_DPDK`), and atomic operations (`OVS_CHECK_ATOMIC_*`). Each check writes one or more `HAVE_*` macros into `config.h` and may set conditional automake variables for subsequent stanzas.

- **Lines 149 to 156, generated header files.** `AC_CONFIG_FILES([lib/stdio.h lib/string.h ...])` declares files that configure must produce by template substitution. `lib/stdio.h` and `lib/string.h` are local wrapper headers that include the system header and add OVS-specific extensions. The `*.sym` files are the libtool symbol export lists.

- **Lines 158 to 193, compiler warning options.** `OVS_ENABLE_OPTION([-Wall])` and the long list that follows enable each warning if the compiler supports it. The OVS code is warning-clean under the union of these flags (with `-Werror` enabled by `OVS_ENABLE_WERROR`); a contributor who introduces a new warning has their patch rejected at review.

- **Lines 195 to 202, kernel and DPDK probing.** `OVS_CHECK_LINUX` reads the kernel-headers path supplied by `--with-linux=`, runs a small kbuild test to verify the headers are usable, and stores the result in conditional variables. `OVS_CHECK_LINUX_NETLINK`, `OVS_CHECK_LINUX_TC`, `OVS_CHECK_LINUX_SCTP_CT`, `OVS_CHECK_LINUX_VIRTIO_TYPES` probe specific kernel features that the userland code conditionally uses. `OVS_CHECK_DPDK` runs only if `--with-dpdk` was passed; it locates the DPDK install prefix and sets the build mode to userspace-DPDK datapath.

- **Lines 207 to 224, the file-generation list.** `AC_CONFIG_FILES(Makefile)` and the lines following declare which input files get template-substituted into output files at configure time. The list includes the top-level `Makefile`, the kernel-build `datapath/Makefile`, the kernel `Kbuild`, the per-library `pkg-config` files (`*.pc`), and the version header `include/openvswitch/version.h`.

- **Line 227, AC_OUTPUT.** The terminating macro that actually writes every `*.in` to `*` after substituting the autoconf-collected values.

A learner reading `configure.ac` cold should focus on lines 90 to 146 (the OVS-specific probes) and lines 195 to 202 (the kernel and DPDK probes). The rest is conventional autoconf.

## 4. acinclude.m4, where the OVS-specific magic lives

`acinclude.m4` is 1510 lines of m4 macros prefixed `OVS_CHECK_*`, `OVS_ENABLE_*`, `OVS_LIBTOOL_*`, and helpers. Each macro is one feature probe or one configuration toggle. A reader who wonders what `OVS_CHECK_DPDK` actually does opens `acinclude.m4`, searches for `^AC_DEFUN(\[OVS_CHECK_DPDK\]`, and reads the body. The structure of every probe is the same: try to compile a small test program against a candidate library, set automake conditionals based on success or failure, optionally `AC_DEFINE` a `HAVE_*` macro into `config.h`, optionally fail with `AC_MSG_ERROR` if the feature was explicitly requested but unavailable.

The most behaviourally consequential probes are:

- `OVS_CHECK_LINUX` (around line 200 of `acinclude.m4`). Reads `--with-linux=PATH`, runs a small kbuild target inside `PATH` to confirm the kernel headers compile, exposes the result via the `LINUX_ENABLED` automake conditional. Without this flag, the kernel module is not built; the userspace datapath still works.
- `OVS_CHECK_DPDK`. Reads `--with-dpdk[=PATH]`, locates `libdpdk.pc` via `pkg-config`, verifies the DPDK version, sets `DPDK_NETDEV` conditional. When set, `lib/netdev-dpdk.c` is compiled into `libopenvswitch.la` and the userspace datapath gains a DPDK port type.
- `OVS_CHECK_AVX512`. Probes whether the host compiler can emit AVX-512 instructions and whether the linker accepts them. When both are true, `HAVE_AVX512F` and `HAVE_LD_AVX512_GOOD` are set, and `lib/automake.mk` builds an extra static library `lib/libopenvswitchavx512.la` that the userspace datapath uses for the optimised flow lookup path.
- `OVS_CHECK_OPENSSL`. Reads `--enable-ssl`, locates the OpenSSL or LibreSSL development headers and libraries, and sets `HAVE_OPENSSL`. Without OpenSSL the SSL-secured OVSDB and OpenFlow paths are compiled out.
- `OVS_ENABLE_WERROR`. Adds `-Werror` to the compile flags when `--enable-Werror` is passed. Off by default; CI builds turn it on.

A reader writing a new probe imitates an existing one. The conventional template is `AC_DEFUN([OVS_CHECK_FOO], [...])` with a body that runs `AC_CHECK_*` macros, sets a shell variable, calls `AM_CONDITIONAL([HAVE_FOO], [test "$ovs_have_foo" = yes])`, and optionally `AC_DEFINE`s.

## 5. The configure invocation and what it produces

A canonical configure invocation for the lab host:

```text
./configure --prefix=/usr/local \
            --localstatedir=/usr/local/var \
            --sysconfdir=/usr/local/etc \
            --with-linux=/lib/modules/$(uname -r)/build
```

The four flags do the following:

- `--prefix=/usr/local` (the default) sets where binaries, headers, and scripts go.
- `--localstatedir=/usr/local/var` overrides the default `/var` so the OVSDB transaction log lives under `/usr/local/var/lib/openvswitch/conf.db` rather than `/var/lib/openvswitch/conf.db`. This separates a from-source install from a distro-package install on the same host.
- `--sysconfdir=/usr/local/etc` similarly overrides `/etc` so the from-source PKI directory does not collide with a distro install.
- `--with-linux=/lib/modules/$(uname -r)/build` activates the kernel module build against the headers shipped by the running kernel.

Configure runs every probe declared in `configure.ac` plus every `AC_CHECK_*` macro it transitively depends on. On the lab host (4 cores, 7.7 GiB RAM, fresh checkout), configure takes roughly 30 to 60 seconds. It writes:

- `config.h` at the source root, summarising every detected feature as `#define HAVE_FOO 1` or absent.
- `config.log` at the source root, recording every probe and its result. A reader debugging a configure failure reads `config.log` first.
- `config.status` at the source root, a self-contained shell script that can re-run configure with the same flags.
- `Makefile` at every directory that has a `Makefile.am` (every `automake.mk`-included directory). The Makefile is the actual build orchestrator.
- The per-library `lib/libopenvswitch.pc`, `lib/libsflow.pc`, `ofproto/libofproto.pc`, `ovsdb/libovsdb.pc`. These are the `pkg-config` files external integrators use.
- `include/openvswitch/version.h` with the version `2.17.9` baked in.
- `tests/atlocal` with paths and host-specific values used by the autotest suite.

The Makefile is a fully populated GNU Make file with explicit recipes; nothing is left for `make` to discover at build time.

## 6. The top-level Makefile.am, the one-Makefile-rules-them-all pattern

OVS uses the **non-recursive automake** pattern. The top-level `Makefile.am` is 501 lines, but most of those lines are `include` directives that pull every per-directory `automake.mk` fragment into the top-level scope:

```text
include Documentation/automake.mk
include m4/automake.mk
include lib/automake.mk
include ofproto/automake.mk
include utilities/automake.mk
include tests/automake.mk
include include/automake.mk
include third-party/automake.mk
include debian/automake.mk
include ipsec/automake.mk
include vswitchd/automake.mk
include ovsdb/automake.mk
include rhel/automake.mk
include xenserver/automake.mk
include python/automake.mk
include tutorial/automake.mk
include vtep/automake.mk
include datapath-windows/automake.mk
include datapath-windows/include/automake.mk
include windows/automake.mk
include selinux/automake.mk
include build-aux/automake.mk
```

After the includes, automake produces a single `Makefile.in` that, when configured, becomes a single top-level `Makefile` containing recipes for every binary, library, script, header, and man page in the project.

Why non-recursive? Recursive make (the historical pattern of `cd subdir && make` in each directory) makes parallelism harder, dependency tracking less reliable, and "what file produces what" much harder to grep. The 1997 paper "Recursive Make Considered Harmful" by Peter Miller is the foundational reference. OVS adopted the non-recursive pattern for the parallelism and the deterministic build.

The one exception is `datapath/`: the top-level Makefile delegates the kernel module build to a separate Makefile inside `datapath/` that uses the kernel's own `kbuild` rules. This is unavoidable; kernel module builds cannot be expressed in plain automake.

## 7. lib/automake.mk, the libopenvswitch.la build

`lib/libopenvswitch.la` is the static library every binary links against. It contains the classifier, the OpenFlow code, the netdev abstraction, the OVSDB IDL, the JSON parser, the platform abstractions, and the long tail of utilities. Roughly 300 source files in `lib/` are listed in `lib/automake.mk` as `lib_libopenvswitch_la_SOURCES`. The file declares the library, its `LIBADD` list (additional libraries to link, such as OpenSSL via `$(SSL_LIBS)`), its `LDFLAGS` (including the symbol export script `lib/libopenvswitch.sym`), and the source list.

Two sub-libraries are conditionally built:

- `lib/libopenvswitchavx512.la` is built only when `HAVE_AVX512F` and `HAVE_LD_AVX512_GOOD` are both true. It is a static library compiled with `-mavx512f -mavx512bw -mavx512dq -mbmi -mbmi2`. The userspace datapath uses it for the AVX-512-optimised flow-lookup path. Hosts without AVX-512 fall back to the portable lookup; behaviour is identical, performance differs.
- `lib/libsflow.la` is the sFlow library. Listed separately because it has its own ABI versioning and its own symbol export list.

The `OVSIDL_BUILT` variable is the list of headers generated from OVSDB schemas via `ovsdb-idlc`. For example, `lib/vswitch-idl.h` is generated from `vswitchd/vswitch.ovsschema` at build time. These headers are not checked into git; a developer who runs `make distclean` and forgets to run `boot.sh` again will see compile errors about missing IDL headers.

## 8. The per-binary stanza pattern

Every binary has the same shape in its `automake.mk`:

```text
bin_PROGRAMS += <dir>/<binary-name>
<dir>_<binary-name>_SOURCES = <list of .c files>
<dir>_<binary-name>_LDADD = lib/libopenvswitch.la <other libraries>
```

The convention `bin_PROGRAMS` versus `sbin_PROGRAMS` versus `noinst_PROGRAMS` controls the install destination: `bin_PROGRAMS` go to `$prefix/bin`, `sbin_PROGRAMS` to `$prefix/sbin`, `noinst_PROGRAMS` are built but not installed (used for tests). `bin_SCRIPTS` and `sbin_SCRIPTS` cover shell and Python scripts that go through the `.in` substitution pipeline (see Section 11 below).

The `utilities/automake.mk` file is illustrative. It declares five C binaries (`ovs-appctl`, `ovs-testcontroller`, `ovs-dpctl`, `ovs-ofctl`, `ovs-vsctl`) and ten installed scripts (`ovs-docker`, `ovs-pki`, `ovs-pcap`, `ovs-tcpdump`, `ovs-tcpundump`, `ovs-dpctl-top`, `ovs-l3ping`, `ovs-parse-backtrace`, `ovs-test`, `ovs-vlan-test`) plus four scripts installed at the package data directory (`ovs-check-dead-ifs`, `ovs-ctl`, `ovs-kmod-ctl`, `ovs-save`). The bash-completion scripts are declared via `completion_SCRIPTS`. The total is twenty-one separate installed artefacts produced by one `automake.mk` file.

## 9. The .in substitution pipeline

OVS ships several scripts as `.in` templates that contain `@VAR@` placeholders. The top-level `Makefile.am` lines 149 to 171 declare a generic `.in` rule that substitutes `@PKIDIR@`, `@LOGDIR@`, `@DBDIR@`, `@PYTHON3@`, `@RUNDIR@`, `@VERSION@`, `@localstatedir@`, `@pkgdatadir@`, `@sysconfdir@`, `@bindir@`, `@sbindir@`, `@abs_builddir@`, `@abs_top_srcdir@` with the values configure detected.

Examples of files that go through this pipeline: `utilities/ovs-ctl.in`, `utilities/ovs-pki.in`, `utilities/ovs-tcpdump.in`, `vswitchd/ovs-vswitchd.8.in`, `python/ovs/dirs.py.in`. The substitution lets the same source file be installed correctly under any prefix without source-level modification. A reader who edits `ovs-ctl` to fix a bug must remember to edit `utilities/ovs-ctl.in`, not the generated `utilities/ovs-ctl`.

A second pipeline runs `build-aux/dpdkstrip.py` over each `.in` to remove DPDK-specific sections when DPDK is not enabled. The flag is set via `DPDKSTRIP_FLAGS` in line 39 of `Makefile.am`. The result is that scripts like `ovs-ctl` ship with DPDK lines included or excluded depending on the build configuration.

A third pipeline converts `.xml` man-page sources into nroff via `build-aux/xml2nroff`. The XML format is OVS-specific (see `vswitchd/vswitch.xml` for an example); the conversion produces `*.5`, `*.7`, and `*.8` man-page output.

## 10. The OVSDB IDL header generation

`ovsdb-idlc` is a code generator that reads a schema (a JSON file like `vswitchd/vswitch.ovsschema`) and writes C headers (`lib/vswitch-idl.h`, `lib/vswitch-idl.c`) plus optionally Python bindings. The generated code provides typed accessors for every column of every table in the schema. Without the IDL headers, every OVSDB client would write column-by-column code with stringly-typed accessors; the IDL elevates that to compile-time type safety.

The generation rule lives in `lib/automake.mk` and is invoked at build time. Touching the schema (for example, adding a new column to `Bridge`) regenerates the headers automatically; touching a non-schema file does not. The dependency tracking is precise enough that the IDL regeneration almost never causes a full rebuild.

A reader who wants to understand the IDL deeper opens `python/ovs/db/idl.py` (the IDL client implementation) and `lib/ovsdb-idl.c` (the C-side IDL machinery). The schema-to-code mapping is straightforward once the convention is understood.

## 11. The kernel module build (`make modules` and `make modules_install`)

The kernel module lives in `datapath/`. It builds against the running kernel's headers, not against the OVS userland. The build process is:

1. The top-level `configure` writes `datapath/Makefile`, `datapath/linux/Kbuild`, and `datapath/linux/Makefile.main`.
2. `make modules` (or `make` if kernel module building is enabled) descends into `datapath/linux/` and invokes the kernel's own kbuild infrastructure. Kbuild compiles each `.c` file in `datapath/linux/` and the per-vport implementations into `.o` files, then links them into `openvswitch.ko`.
3. `make modules_install` copies `openvswitch.ko` to `/lib/modules/$(uname -r)/updates/` (the conventional out-of-tree module location). It updates `modules.dep` via `depmod`.

The Linux kernel ships its own in-tree `openvswitch.ko` since around kernel 3.3. The OVS source ships a compatibility shim under `datapath/linux/compat/` that backports newer kernel features so the out-of-tree module can run on older kernels. On a modern kernel like the lab's `5.15.0-173-generic`, the in-tree module has every feature the userland needs; the out-of-tree module adds nothing useful.

The plan v3.13 R1 path C exercise builds the kernel module to learn the process, but R1 does not load it because the in-tree module is preferable. Plan v3.13 §9 Risk 3 documents this trade-off explicitly.

A learner who edits a `datapath/*.c` file and wants to test it must rebuild the module, unload the in-tree one (`rmmod openvswitch`), insert the out-of-tree one (`insmod /lib/modules/.../updates/openvswitch.ko`), and restart `ovs-vswitchd`. The cycle is slow; rapid iteration on the kernel side is one of the long-standing pain points of OVS development.

## 12. The install layout

`make install` produces the following layout under the configured prefix. The list is concrete for `--prefix=/usr/local --localstatedir=/usr/local/var --sysconfdir=/usr/local/etc`:

- `/usr/local/bin/`: `ovs-appctl`, `ovs-dpctl`, `ovs-dpctl-top`, `ovs-docker`, `ovs-l3ping`, `ovs-ofctl`, `ovs-parse-backtrace`, `ovs-pcap`, `ovs-pki`, `ovs-tcpdump`, `ovs-tcpundump`, `ovs-test`, `ovs-vlan-test`, `ovs-vsctl`, `ovsdb-client`, `ovsdb-tool`, `vtep-ctl`.
- `/usr/local/sbin/`: `ovs-vswitchd`, `ovsdb-server`.
- `/usr/local/share/openvswitch/`: the schema files (`vswitch.ovsschema`, `vtep.ovsschema`), the man pages, the bash-completion scripts, the support scripts (`ovs-ctl`, `ovs-kmod-ctl`, `ovs-save`, `ovs-check-dead-ifs`), the IPsec helper.
- `/usr/local/include/openvswitch/`: the public headers.
- `/usr/local/lib/`: `libopenvswitch.la`, `libsflow.la`, `libofproto.la`, `libovsdb.la`, `libvtep.la` (libtool wrappers around the static archives).
- `/usr/local/lib/pkgconfig/`: the `*.pc` files for external integrators.
- `/usr/local/etc/openvswitch/`: empty at install time; populated at first start.
- `/usr/local/var/lib/openvswitch/`: empty at install time; populated by `ovs-ctl start` with `conf.db`.
- `/usr/local/var/log/openvswitch/`: empty at install time; populated by daemon startup with `ovs-vswitchd.log` and `ovsdb-server.log`.
- `/usr/local/var/run/openvswitch/`: empty at install time; populated at daemon startup with the Unix sockets and PID files.
- `/usr/local/share/man/`: the man pages installed under `man1/`, `man5/`, `man7/`, `man8/`.

A from-source install does not place a systemd unit; the operator uses `/usr/local/share/openvswitch/scripts/ovs-ctl start` instead. A distro package install does place systemd units; the two install styles are not mutually compatible.

## 13. The config-h-check pre-build invariant

Lines 232 to 250 of the top-level `Makefile.am` define a pre-build invariant: every C source file under `lib/`, `ofproto/`, `vswitchd/`, `ovsdb/`, and `utilities/` must `#include <config.h>` as the first include. The invariant is checked at every build by a target called `config-h-check`. A new file that omits the include fails the build with an error message naming the offending files.

The reason is that `config.h` defines `_GNU_SOURCE`, the autoconf-detected `HAVE_*` macros, and several glibc compatibility shims. A file that includes a system header before `config.h` may pick up the wrong feature subset. Including `config.h` first prevents this class of bug.

A reader who adds a new C file should add `#include <config.h>` at the very top, before any system include.

## 14. The dist target and tarball production

`make dist` produces a release tarball `openvswitch-2.17.9.tar.gz`. The contents are determined by the `EXTRA_DIST` lists scattered across every `automake.mk` plus the `bin_PROGRAMS`, `_SOURCES`, and similar variables. The dist target also runs the `dist-hook-git` recipe (`Makefile.am` lines 198 to 218) which compares the dist file list against `git ls-files` and fails if any committed file is not in the dist. This invariant prevents accidentally shipping a release tarball that is missing a source file.

A learner who wants to verify the release tarball runs `tar tzf openvswitch-2.17.9.tar.gz | sort > tarball-list`, runs `git ls-files | sort > git-list`, and `comm -23 git-list tarball-list` to find files in git but not in the tarball.

## 15. The pkg-config files

`lib/libopenvswitch.pc`, `lib/libsflow.pc`, `ofproto/libofproto.pc`, `ovsdb/libovsdb.pc` are the `pkg-config` files. Each declares a `Name`, `Description`, `Version`, `Libs` line (the linker flags an external integrator needs), and `Cflags` line (the compile flags). External projects that depend on OVS use `pkg-config --libs --cflags libopenvswitch` to discover the right flags rather than hard-coding paths.

The `.pc` files are produced by configure-time substitution of `lib/libopenvswitch.pc.in`, ensuring the values reflect the actual install prefix.

## 16. The build, in numbers

On the lab host (4 cores, 7.7 GiB RAM, kernel 5.15.0-173, fresh checkout at v2.17.9):

- `boot.sh`: 5 to 20 seconds, mostly disk-bound.
- `configure ...`: 30 to 90 seconds, CPU-bound on the small probe binaries.
- `make -j4`: 2 to 5 minutes for a full build, dominated by `lib/dpif-netdev.c` (10,105 lines) and `lib/ofp-actions.c` (9,785 lines) which take 30 to 60 seconds each to compile with `-O2`.
- `make -j4 modules`: 2 to 4 minutes when the kernel module build is enabled.
- `make install` (without modules): 5 to 15 seconds, dominated by file copy.
- `make install modules_install`: another 5 to 15 seconds.
- `make distclean`: 10 to 30 seconds, clean and re-runnable.

The full from-source first build, including the kernel module, fits within a single 30-minute lab session. Subsequent incremental builds after editing one file complete in 10 to 60 seconds depending on the file.

## 17. The reading order through the build system

A reader who wants to master this build system reads in this order, with the corresponding R-sprint or S-sprint that exercises each piece:

1. `boot.sh`, two lines, just to confirm what it runs.
2. `configure.ac`, especially lines 90 to 146 and 195 to 202.
3. `acinclude.m4`, especially `OVS_CHECK_LINUX`, `OVS_CHECK_DPDK`, `OVS_CHECK_AVX512`.
4. The top-level `Makefile.am`, especially the include list (lines 480 to 501) and the pre-build invariants (lines 198 to 290).
5. `lib/automake.mk`, especially the `lib_libopenvswitch_la_SOURCES` list and the conditional AVX-512 sub-library.
6. One per-binary `automake.mk`, for example `utilities/automake.mk`.
7. The IDL generation rules in `lib/automake.mk`.
8. `datapath/Makefile.in` plus `datapath/linux/Kbuild` (at build time) for the kernel module.

The R-sprints and S-sprints that exercise each layer:

- R1 path B and path C exercise `boot.sh`, `configure`, `make`, `make install`, `make modules_install` end to end.
- S1 (compile from source) walks the build log line by line.
- S2 (daemon lifecycle) reads `utilities/ovs-ctl` to see how the configured paths are used at runtime.
- Every later S-sprint that adds or modifies a C file exercises the per-binary `_SOURCES` stanza.

A reader who has worked through this file plus its R1 lab transcript can compile OVS from source, modify a `lib/*.c` file, rebuild, and re-test without consulting external documentation. That is the operational target of S0.

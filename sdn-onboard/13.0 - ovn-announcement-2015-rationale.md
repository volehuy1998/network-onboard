## 13.0.1 Announcement 13/01/2015 — bối cảnh 4 tác giả

OVN (Open Virtual Network) công bố ngày 13/01/2015 trên Network Heresy blog. Title: *"OVN: Bringing Native Virtual Networking to OVS"*. Thời điểm này là 8 năm sau khi OVS ra đời (2007, Phần 9.0) và 3 năm sau khi VMware mua Nicira (07/2012, Phần 5.1). Announcement ngắn gọn nhưng ký hiệu cho một thay đổi đáng kể trong hệ sinh thái SDN open-source.

### Bốn tác giả và vai trò

**Justin Pettit** (VMware, ex-Nicira co-founder). Pettit là một trong ba founder Nicira 2007 (cùng Casado + McKeown từ Stanford) và là primary architect của OVS 2007-2015. Trong OVN, Pettit thiết kế 3-tier architecture + pipeline semantics. Vai trò kỹ thuật chính.

**Ben Pfaff** (VMware, NSDI 2015 paper lead). Pfaff là author chính của "The Design and Implementation of Open vSwitch" (NSDI 2015 best paper, Phần 9.0). Trong OVN, Pfaff viết `ovn-northd` compiler + distributed control plane semantics. Vai trò kỹ thuật thứ hai.

**Chris Wright** (Red Hat VP, sau này Red Hat CTO 2019+). Wright là VMware-Red Hat liaison chính, đại diện operational + product perspective của Red Hat. Vai trò strategic: đảm bảo OVN fit với Red Hat OpenStack Platform (RHOSP) + OpenShift roadmap.

**Madhu Venugopal** (Red Hat, OpenStack Neutron + Docker libnetwork contributor). Venugopal đưa OpenStack + container networking perspective, yêu cầu OVN phải tương thích Neutron ML2 driver (2017 release) + Kubernetes CNI (sau này 2019).

> **Chủ đề trọng tâm:** Collaboration VMware-Red Hat 2015 là hiếm có. VMware vừa mua Nicira $1,26 tỷ USD 2012, lý tưởng là giữ OVN độc quyền cho NSX roadmap. Thay vào đó, Pfaff + Pettit (employed VMware) release OVN dưới Apache 2.0 với co-authors Red Hat. Quyết định này phản ánh triết lý NSDI 2015 của Nicira heritage: *"community first, product second"*. Kết quả OVN trở thành foundation cho OpenStack + OpenShift (Red Hat flagship) — revenue của Red Hat dựa trên OVN lớn hơn bất kỳ revenue nào VMware có thể đạt được nếu OVN đóng.

### Thông điệp then chốt của announcement

Blog post emphasize OVN là **"native" virtual networking cho OVS** — built-in, không phải add-on. Trước OVN, OVS cần control từ third-party:

- **OpenStack Neutron + ML2 + linuxbridge/OVS agent**: Python-heavy orchestrator layer translate Neutron API → OVS commands. Complex, slow (5-30s provision time), race condition khi scale.
- **OpenDaylight (ODL) + OVSDB plugin**: Java/Karaf monolithic controller. Too heavy cho small-scale, ít production deployment.
- **Juniper Contrail / OpenContrail**: BGP EVPN centric, vRouter-based. Specific tới Juniper ecosystem.
- **NSX (VMware proprietary)**: closed-source, chỉ chạy trên vSphere + KVM enterprise.

Không có option native open-source + purpose-built cho OVS. OVN fill gap đó.

## 13.0.2 Động lực thiết kế — 3 kỹ thuật + 2 thương mại

### Động cơ kỹ thuật

**Kỹ thuật 1 — OVS có data plane tốt nhưng thiếu control plane native.**
OVS 2007-2015 là general-purpose programmable virtual switch. OpenFlow + OVSDB cho phép external control, nhưng controller logic phải self-implement. Developer phải viết hàng nghìn dòng code Python/Go để translate "tạo virtual network cho tenant" → flow rule + tunnel setup + ARP responder + DHCP reply. OVN cung cấp sẵn layer logical này.

**Kỹ thuật 2 — Distributed control plane native OVS.**
NSX (Nicira heritage) có centralized controller. OpenStack Neutron agent distributed nhưng fragile race condition. OVN thiết kế từ đầu với: NBDB centralized ý đồ cấu hình + `ovn-northd` single instance compile + SBDB Raft-replicated + `ovn-controller` per-chassis distributed apply. Mô hình này scale tới 1000+ chassis (OVN 24.03 tested) mà không cần cross-chassis consensus cho data plane decisions.

**Kỹ thuật 3 — Logical abstraction layer tách khỏi physical underlay.**
User nghĩ về `Logical_Switch`, `Logical_Router`, `ACL`, `Load_Balancer`, `NAT`. Không nghĩ về: flow table number, OpenFlow match fields, tunnel encap, ARP responder implementation. OVN compile logical ý đồ cấu hình thành hàng nghìn physical flow rule tự động. Tách biệt này giống như C compiler tách programmer khỏi assembly.

### Động cơ thương mại

**Thương mại 1 — Red Hat cần alternative cho Neutron ML2 + linuxbridge.**
RHOSP 2014-2016 gặp performance + scale issue với Neutron ML2 + linuxbridge + router/DHCP agent. Provision tenant mất 30+ giây. 1000+ tenant scale causing race condition frequent. Red Hat cần lựa chọn kiến trúc mới. Thay vì build từ zero, tài trợ + co-develop OVN với Nicira team. Kết quả OVN trở thành default trong RHOSP 16 (2020).

**Thương mại 2 — VMware muốn OVN làm ref implementation cho NSX-T extensions.**
VMware NSX proprietary + locked to ESXi + enterprise license. OVN open-source cho phép VMware reach developer/operator community beyond enterprise. NSX-T phiên bản 2.5+ có concepts tương đương OVN (logical switch, logical router, distributed firewall). Community familiar với OVN dễ dàng transition tới NSX-T cho production. Chiến lược này giống Microsoft với Windows NT → free evaluation license.

> **Hiểu sai phổ biến:** *"OVN ra đời vì OpenFlow 'thất bại' không scale."*
>
> **Thực tế:** OpenFlow không thất bại — OpenFlow vẫn là protocol southbound chính cho OVS + OVN compile sang OpenFlow. Vấn đề là **control plane trên OpenFlow không có ngôn ngữ logical abstraction**. Developer phải viết từng flow rule manually. OVN thêm logical abstraction layer (NBDB + SBDB) mà OpenFlow không có. OVN không thay thế OpenFlow; OVN dùng OpenFlow làm southbound. Phân biệt "OpenFlow mất traction như controller direct" vs "OpenFlow thành công như southbound để OVN compile xuống" rất quan trọng. Xem Phần 20.6 §20.6.4 "Hypervisor overlays thắng 2013-2017" cho phân tích chi tiết.

## 13.0.3 3-tier architecture — core contribution

Compass Ch 17: *"OVN sits above OVS as a logical networking controller. Its architecture is three tiers."*

```
 ┌────────────────────────────────────┐
 │ Tier 1: ovn-nbctl                  │  ← admin/orchestrator
 │         (writes logical intent)    │
 └───────────────┬────────────────────┘
                 │ JSON-RPC (OVSDB wire protocol)
 ┌───────────────▼────────────────────┐
 │ Tier 2: Northbound DB (NB)         │  ← intent storage (Raft 3-node)
 │    + ovn-northd (translator)       │  ← compile NB → SB
 │    → Southbound DB (SB)            │  ← computed logical flows
 └───────────────┬────────────────────┘
                 │ JSON-RPC (subscribe Logical_Flow + Port_Binding)
 ┌───────────────▼────────────────────┐
 │ Tier 3: ovn-controller (per host)  │  ← per-chassis agent
 │         → OpenFlow rules → br-int  │  ← physical application
 └────────────────────────────────────┘
```

### Tier 1 — Ý đồ cấu hình (NBDB)

Admin (hoặc orchestrator như Neutron / Kubernetes) write ý đồ cấu hình qua `ovn-nbctl`:

```bash
$ ovn-nbctl ls-add ls-tenant-web
$ ovn-nbctl lsp-add ls-tenant-web lsp-web-01 \
     -- lsp-set-addresses lsp-web-01 "aa:bb:cc:00:00:01 10.0.0.10"
$ ovn-nbctl acl-add ls-tenant-web from-lport 1000 \
     'inport=="lsp-web-01" && tcp.dst==80' allow-related
```

NBDB tables chính:
- `Logical_Switch` (L2 virtual switch)
- `Logical_Router` (L3 virtual router)
- `Logical_Switch_Port` (LSP — endpoint, tap/vif/localnet/etc.)
- `Logical_Router_Port` (LRP — interface between LS and LR)
- `ACL` (stateful firewall rule, thường kèm conntrack)
- `NAT` (SNAT/DNAT)
- `Load_Balancer` (VIP → backend pool)
- `Port_Group` (group LSP for ACL scale optimization)

### Tier 2 — Translation (`ovn-northd` + SBDB)

`ovn-northd` (1 singleton process) subscribe NBDB, compute logical flow, write SBDB. Compilation steps (Phần 13.8 detail):

1. Đọc toàn bộ NBDB tables.
2. Generate logical flow rule cho từng stage của ingress pipeline (27 stages) + egress pipeline (10 stages) — gọi chung là OVN "pipeline stages".
3. Allocate `tunnel_key` cho mỗi Logical_Switch + Logical_Router.
4. Write Logical_Flow + Datapath_Binding + Port_Binding vào SBDB.

SBDB tables chính:
- `Chassis` (mỗi hypervisor register với encap type + IP)
- `Port_Binding` (bind LSP tới chassis — 8 type: vif/localnet/l2gateway/l3gateway/chassisredirect/patch/localport/virtual)
- `Logical_Flow` (computed flow rule với pipeline = ingress/egress + table_id + priority + match + action)
- `MAC_Binding` (IP→MAC cached from ARP/ND)
- `Datapath_Binding` (logical switch/router với tunnel_key)

### Tier 3 — Data plane (`ovn-controller` per chassis)

Mỗi chassis chạy 1 `ovn-controller` process. Subscribe SBDB (Logical_Flow + Port_Binding cho chassis local). Compile logical flow → OpenFlow rule cài vào `br-int` (Phần 13.4).

`ovn-controller` cũng:
- Monitor `br-int` tap port với `iface-id` → bind Port_Binding.
- Tạo tunnel port tới remote chassis.
- Install ARP responder flow (Phần 18.0).
- Install FDB (Phần 17.0) + MAC_Binding resolution rule.

### Ưu điểm 3-tier

**Separation of concerns.** Developer orchestrator chỉ write NBDB (logical ý đồ cấu hình). `ovn-northd` lo compile phức tạp. `ovn-controller` lo physical application. Mỗi tier có responsibility rõ ràng, debug độc lập.

**Scale target.** Một cluster 3-node NBDB + 3-node SBDB + 1 `ovn-northd` có thể handle 100-1000 chassis. Raft consensus SBDB đảm bảo consistency under chassis churn.

**Evolution path.** Thay đổi compile logic → chỉ update `ovn-northd`. Thay đổi apply logic → chỉ update `ovn-controller`. Thay đổi data model → update NBDB schema + migration. Không yêu cầu coordinated restart toàn hệ thống.

## 13.0.4 So sánh 4 alternative — khi nào chọn OVN

| Dimension | OVN | VMware NSX | Juniper Contrail | OpenDaylight (ODL) | Pure OVS + custom |
|-----------|-----|-----------|------------------|---------------------|---------------------|
| **License** | Apache 2.0 open | Proprietary enterprise | Apache 2.0 (Tungsten Fabric) | Eclipse Public | Apache 2.0 OVS + DIY code |
| **Orchestrator integration** | Neutron ML2 + K8s CNI + bare-metal | ESXi + KVM + NSX-T Manager | OpenStack + K8s + bare-metal | Multi-protocol umbrella | DIY per-orchestrator |
| **Scale target** | 1000+ chassis tested | 1000+ ESXi hypervisor (VMware production) | 500-1000 compute | 100-500 (JVM limit) | Varies, DIY |
| **Data plane** | OVS datapath | OVS proprietary fork | vRouter (user-space) | Any (OpenFlow agnostic) | OVS datapath |
| **Control plane** | Distributed (ovn-controller per chassis) | Centralized (NSX Manager cluster) | Distributed (vRouter agent + BGP EVPN controller) | Monolithic (ODL cluster) | DIY |
| **Underlay protocol** | Geneve (default), VXLAN legacy | Geneve + STT | BGP EVPN over VXLAN | Flexible | DIY |
| **Operational ops** | Active dev 2015+, quarterly release | Enterprise support tier | Juniper support | Maintenance-only 2023+ | N/A |
| **Use case sweet spot** | OpenStack + K8s + standalone | ESXi-heavy enterprise | Juniper MX + K8s telco | Research + demo | Lab + POC |

### Khi nào chọn OVN

**Chọn OVN standalone khi:**
- Telco deployment cần distributed control plane không lock vendor.
- K8s cluster với high-scale tenant (>100 namespace) và need network policy chính xác.
- Custom orchestrator build from scratch (không dùng OpenStack/K8s).

**Chọn OpenStack ML2-OVN khi:**
- Đã có OpenStack deployment + muốn migrate từ ML2-linuxbridge sang ML2-OVN.
- Red Hat RHOSP 16+ hoặc Canonical Ubuntu Cloud Archive.
- Reduce Neutron agent complexity (không cần `neutron-dhcp-agent`, `neutron-l3-agent`, `neutron-metadata-agent`).

**Chọn OVN-Kubernetes CNI khi:**
- OpenShift 4+ (default CNI từ 4.12).
- K8s workload cần EgressIP + EgressFirewall + HybridOverlay (mix Linux + Windows workload).
- Multi-cluster mesh qua OVN-IC (inter-cluster).

### Khi KHÔNG chọn OVN

**Chọn NSX-T khi đã invest ESXi enterprise + cần end-to-end vendor support.**

**Chọn Cilium eBPF khi K8s workload cần sidecar-less service mesh + XDP DDoS + HTTP-layer policy (L7).** OVN chỉ L2-L4.

**Chọn pure OVS + Ansible khi lab scale nhỏ (2-5 chassis) + team không quen OVN learning curve.**

## 13.0.5 Use case thực tế 2015-2026

**OpenStack Neutron + OVN ML2** (2017-): Red Hat RHOSP 16 (2020) default. Canonical Ubuntu charm-ovn (2022+). Replacement cho linuxbridge + router/DHCP/metadata agent. Provision time từ 30s → 3-5s. Scale từ 100 tenant → 1000+ tenant stable.

**OVN-Kubernetes CNI** (2019-): OpenShift 4.6+ GA. OpenShift-SDN (Kubernetes default 4.5 trước đó) migrate sang OVN-Kubernetes, completed bởi OpenShift 4.14 (2023). OVN-IC (inter-cluster) GA 4.14 cho multi-cluster networking.

**Standalone OVN** (2017-): telco + carrier custom deployment. Khối 17/18/19 case study chủ yếu standalone. Pattern: operator tự build control plane với `ovn-nbctl` + scripting, không qua OpenStack/K8s abstraction.

**OVN trong AI/ML workload** (2024-): hyperscale AI training cần low-độ trễ intra-rack networking. OVN + RoCE + SR-IOV acceleration experimented nhưng chưa GA production.

### Key inflection points

**2020 OVS/OVN split**: trước 2020, OVN ship cùng OVS release train (OVS 2.13 chứa OVN 20.06 alpha). Sau split, OVN có versioning riêng (20.x → 22.x → 24.x). Cho phép OVN release cycle nhanh hơn OVS (3-6 tháng vs 6-12 tháng).

**2022 incremental-processing engine**: `ovn-controller` trước OVN 22.09 recompute toàn bộ state từ SBDB mỗi khi change. Với 10K flow, recompute 1-2s mỗi event. I-P engine trong OVN 22.09 compute delta only → <100ms. Crucial cho hyperscale chassis count.

**2024 multichassis port + activation-strategy**: addresses long-standing pain point VM live migration (Phần 19.0 case study). Trước OVN 22.03, live migration yêu cầu ~15-30s rebuilding flow per destination chassis. OVN 22.03+ multichassis port allow pre-build flow + activation trigger via RARP gói tin từ guest. Migration downtime < 500ms.

## 13.0.7 ▶ Guided Exercise — Verify OVN 3-tier với `ovn-nbctl` + `ovn-sbctl`

**Mục tiêu:** Quan sát hành trình ý đồ cấu hình NBDB → `ovn-northd` → SBDB → `ovn-controller` qua lab nhỏ.

**Môi trường:** 1 VM Ubuntu 22.04 chạy cả ovn-central + ovn-controller (single-node dev).

**Bước:**

1. Install + start OVN central + controller:
   ```bash
   $ sudo apt install -y ovn-central ovn-host
   $ sudo ovn-nbctl set-connection ptcp:6641
   $ sudo ovn-sbctl set-connection ptcp:6642
   $ sudo ovs-vsctl set Open_vSwitch . \
       external_ids:ovn-remote=tcp:127.0.0.1:6642 \
       external_ids:ovn-encap-type=geneve \
       external_ids:ovn-encap-ip=127.0.0.1
   ```
2. Tier 1 (NBDB): create logical switch + 1 LSP:
   ```bash
   $ sudo ovn-nbctl ls-add ls-tier1
   $ sudo ovn-nbctl lsp-add ls-tier1 lsp-01 \
       -- lsp-set-addresses lsp-01 "aa:bb:cc:00:00:01 10.0.0.10"
   $ sudo ovn-nbctl show
   ```
3. Tier 2 (SBDB after `ovn-northd` compile): verify computed state:
   ```bash
   $ sudo ovn-sbctl list Datapath_Binding
   # Kỳ vọng: 1 row cho ls-tier1 với tunnel_key auto-assigned
   $ sudo ovn-sbctl --columns=logical_port,tunnel_key,mac list Port_Binding lsp-01
   # Kỳ vọng: logical_port=lsp-01, tunnel_key=N (per-LSP), mac=aa:bb:cc:00:00:01
   $ sudo ovn-sbctl --columns=pipeline,table_id,priority,match,actions list Logical_Flow \
       | head -30
   # Kỳ vọng: Hàng chục entry cho pipeline=ingress/egress với table stages
   ```
4. Tier 3 (ovn-controller → OpenFlow): verify installed rules:
   ```bash
   $ sudo ovs-ofctl -O OpenFlow13 dump-flows br-int | wc -l
   # Kỳ vọng: ~30-40 flow rules (baseline + 1 LSP)
   $ sudo ovs-ofctl -O OpenFlow13 dump-flows br-int | head -10
   ```

**POE questions:**
- **Predict:** Nếu delete LSP `lsp-01` khỏi NBDB, bao lâu sau `br-int` OpenFlow rule biến mất?
- **Observe:** Thường <1 giây. `ovn-northd` recompute (50-200ms với I-P engine OVN 22.09+), SBDB write (50ms), `ovn-controller` apply (50-100ms).
- **Explain:** 3-tier architecture không có polling — cả 3 tier subscribe qua OVSDB monitor_cond (Phần 10.4). Change propagate event-driven, near-instant.

**Cleanup:**
```bash
$ sudo ovn-nbctl ls-del ls-tier1
$ sudo systemctl stop ovn-controller
$ sudo systemctl stop ovn-central
```

## 13.0.8 Điểm cốt lõi cần nhớ

Thứ nhất, OVN công bố 13/01/2015 với 4 tác giả (Pettit + Pfaff VMware + Wright + Venugopal Red Hat) — collaboration VMware-Red Hat hiếm có, phản ánh triết lý NSDI 2015 Nicira "community first, product second".

Thứ hai, động cơ 5 chiều: 3 kỹ thuật (OVS thiếu control plane native / distributed control plane / logical abstraction) + 2 thương mại (Red Hat cần alternative Neutron ML2 / VMware NSX reference implementation).

Thứ ba, 3-tier architecture (NBDB ý đồ cấu hình → `ovn-northd` compile → SBDB computed flow → `ovn-controller` OpenFlow apply) là đóng góp kiến trúc then chốt. Separation of concerns + scale 1000-chassis + evolution path độc lập per tier.

Thứ tư, OVN không thay thế OpenFlow — OVN dùng OpenFlow làm southbound. Phân biệt "OpenFlow mất traction như controller direct" vs "OpenFlow thành công như southbound để OVN compile xuống".

Thứ năm, use case: OpenStack ML2-OVN (RHOSP 16+) / OVN-Kubernetes (OpenShift 4.6+) / Standalone OVN (telco + Khối 17-19). Chọn OVN khi cần distributed control + open-source + purpose-built cho OVS ecosystem.

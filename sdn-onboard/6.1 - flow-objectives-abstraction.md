## Context: an abstraction layer above OpenFlow

Part 4.5 analysed TTP (Table Type Patterns), the ONF attempt to bridge a generic specification and fragmented silicon capability. TTP failed in adoption because of timing, complexity, and vendor incentive issues. Part 6.0 analysed P4 as a model shift that replaces OpenFlow with a programmable data plane. This part analyses **the third alternative approach**: Flow Objectives in ONOS, which keeps OpenFlow as the protocol but adds an abstraction layer on the controller side.

The Flow Objectives philosophy is that, instead of having an application programmer write raw OpenFlow flow entries (which requires knowing match fields, actions, table numbers, and priorities), the application expresses **high-level intent** such as "forward traffic from ingress X to egress Y". The ONOS driver layer translates that intent into device-specific flow rules. The driver abstracts hardware heterogeneity away from application code.

This is the "rising tide" pattern in software engineering: when the primitive layer is too low level (OpenFlow), introduce an abstraction layer to reduce cognitive load. Flow Objectives succeeded in the ONOS ecosystem (CORD, Trellis), but adoption is limited outside.

## 6.1.1 Motivation: TTP is not feasible for multi-vendor deployment

### Recap of TTP

Part 4.5 described the TTP workflow:

1. ONF defines reference TTPs (L2 to L3 Edge, Aggregation to Core, WAN PE).
2. The switch advertises supported TTP IDs in `FEATURES_REPLY`.
3. The controller chooses a TTP that fits the application's needs.
4. The controller installs `flow_mod` according to the TTP schema.

TTP issues:

- **Fixed catalog**: adding a new pattern requires an ONF specification revision, a slow process.
- **Vendor implementation**: hardware vendors (Broadcom, Cisco, Juniper) did not commit to implementing the TTP layer in silicon or firmware. The engineering cost did not justify itself given limited market demand.
- **Application burden**: the application programmer still needed to know the TTP schema of the switch in use, which means being aware of the switch's capability. Not fully abstract.

The result for TTP was that ONF TS-017 v1.0 (15 August 2014) was published, but real-world adoption was close to zero (Part 4.5.3).

### The ONOS architecture team approach

ONOS (Open Network Operating System) at ON.Lab in 2014 faced the same problem: an application built on ONOS needed to be portable across switch vendors with heterogeneous pipelines. TTP was not feasible (it depended on vendor implementation). The ON.Lab decision was to **build the abstraction inside the ONOS controller itself, not on the switch side**.

This is Flow Objectives, introduced around 2015 (the ONOS 1.x series). The design philosophy:

1. **The application expresses intent at a high abstraction level**: "forward packets that match X to port Y". It does not specify table ID, priority, or match field details.
2. **The ONOS framework translates**: the framework has compile-time knowledge of the switch pipeline (via a driver class per device model).
3. **The driver emits device-specific flow rules**: the driver class for the Broadcom OFDPA 2.0 pipeline differs from the driver for Corsa, Noviflow, and so on. The application does not see the difference.

The key insight is that the **abstraction layer lives in the controller (software, flexible) instead of in the switch (hardware or firmware, rigid)**. The driver is code in the ONOS framework. Adding a new switch model means writing a new driver class and deploying an ONOS update, without touching switch firmware.

## 6.1.2 The three objective types: filtering, forwarding, next

### Layered forwarding model

Flow Objectives defines three objective types that map to a three-layer forwarding concept:

```
Packet -> FilteringObjective (admission) -> ForwardingObjective (match logic) -> NextObjective (next-hop group) -> Egress
```

Each objective type has a specific role. Together they form a complete forwarding pipeline.

### FilteringObjective

**Role**: "who can enter this pipeline". Admission control at the ingress port.

Match criteria:

- VLAN ID (untagged, specific VLAN, or priority-tagged).
- MAC address (source, destination).
- IP address optionally.

Operation types:

- **PERMIT**: allow matching traffic to enter the pipeline.
- **DENY**: drop matching traffic.

Example:

```
FilteringObjective {
    key: VLAN 100 on port eth0,
    conditions: [MAC=aa:bb:cc:dd:ee:ff allowed],
    operation: PERMIT
}
```

Intent: port eth0 only accepts VLAN 100 with a specific MAC. The ONOS driver translates this into:

- OFDPA pipeline: a flow installed in the VLAN Filtering table (Table 10) and the Termination MAC table.
- Corsa pipeline: different table numbers, but the same logical effect.

### ForwardingObjective

**Role**: "how to forward matched traffic". The core routing or switching decision.

Flavor types:

- **SPECIFIC**: exact match (destination MAC exact, destination IP exact).
- **VERSATILE**: wildcarded match (ACL or classifier with priority).

Match plus action:

```
ForwardingObjective {
    flavor: SPECIFIC,
    selector: dl_dst=aa:bb:cc:dd:ee:ff,
    nextId: 42,  // reference to NextObjective
    priority: 100
}
```

Intent: match the destination MAC, and forward through next-hop group 42. The application does not specify the output port directly. The output port is the responsibility of NextObjective 42.

The driver translates ForwardingObjective:

- OFDPA: install in the Bridging table (Table 50) or in the ACL table.
- A native Linux kernel NOS: install the route via netlink.

### NextObjective

**Role**: "next-hop group". It defines how traffic egresses through a single port, multiple ports (ECMP), or a failover.

Group types:

- **SIMPLE**: a single output action.
- **HASHED**: hash-based ECMP across buckets.
- **FAILOVER**: primary plus backup buckets (similar to the OpenFlow `FAST_FAILOVER` group type).
- **BROADCAST**: all buckets (similar to the `ALL` group type).

Example:

```
NextObjective {
    id: 42,
    type: HASHED,
    buckets: [
        { output: eth1, MAC rewrite: 00:00:00:00:00:01 },
        { output: eth2, MAC rewrite: 00:00:00:00:00:02 }
    ]
}
```

Intent: ECMP across two uplinks with a MAC rewrite per bucket.

### Composing layers

Application workflow:

```
1. Install NextObjective 42 (the next-hop group).
2. Install ForwardingObjective with nextId=42 (the route).
3. Install FilteringObjective to permit VLAN 100 (admission).
```

Order matters. NextObjective comes first (a backward reference), then ForwardingObjective references the NextObjective. FilteringObjective is independent.

> **Common misconception:** "Flow Objectives is just a thin wrapper around OpenFlow flow entries. An application can ignore Flow Objectives and use raw OpenFlow directly."
>
> **Reality:** Flow Objectives abstracts pipeline fragmentation in a meaningful way. Raw OpenFlow requires the application to know the table number, the supported match fields, and the action availability per switch vendor. The Flow Objectives driver encapsulates this knowledge. Application code written against Flow Objectives is portable. Raw OpenFlow code written for a specific pipeline (OFDPA) is not portable to a different pipeline (Corsa, BCM native). The difference is architecturally significant for a multi-vendor deployment.

## 6.1.3 Driver layer: mapping an objective to a device flow

### Driver responsibilities

An ONOS driver class exists per switch model (or per pipeline family). Driver responsibilities:

1. **Pipeline knowledge**: encode the silicon and firmware pipeline structure (table numbers, valid match fields per table, supported actions per stage).
2. **Objective translation**: convert FilteringObjective, ForwardingObjective, and NextObjective into specific OpenFlow flow entries.
3. **Group management**: create, update, and delete OpenFlow groups corresponding to a NextObjective.
4. **Error mapping**: translate device-specific errors (`OFPERR_BAD_MATCH` on Table 42) back to application-level diagnostics.

### Driver example: OFDPA 2.0

OFDPA (OpenFlow Data Plane Abstraction) is the Broadcom pipeline specification for their switch silicon. The OFDPA 2.0 pipeline (simplified):

| Table | Purpose |
|-------|---------|
| 0 (Ingress Port) | Port classification |
| 10 (VLAN) | VLAN processing |
| 20 (Termination MAC) | MAC address recognition (for routing) |
| 30 (Unicast Routing) | Unicast L3 forwarding |
| 50 (Bridging) | L2 bridging |
| 60 (Policy ACL) | ACL |
| 80 (Egress VLAN) | Output VLAN processing |

The ONOS OFDPA driver translates a FilteringObjective:

- PERMIT VLAN 100 on port eth0: install a flow in Table 10 (VLAN) with match `in_port=eth0, vlan_vid=100`, action `goto_table:20`.

It translates a ForwardingObjective (routing):

- Route 10.0.0.0/24 via next-hop group 42: install a flow in Table 30 (Unicast Routing) with match `eth_type=0x0800, ipv4_dst=10.0.0.0/24`, action `group:42`.

It translates a NextObjective:

- Create an OpenFlow group 42 (type `select`), with buckets equal to the uplink ports.

All of this is internal to the driver. The application just calls `ForwardingObjective.Builder ... .build()` and submits to `FlowObjectiveService`.

### Driver catalog in ONOS

ONOS ships drivers for:

- **OFDPA** (Broadcom): various versions (OFDPA 1.0, 2.0, 3.0).
- **Corsa**: the Corsa Red series of switches.
- **Noviflow**: the Noviflow programmable switch.
- **OpenFlow standard**: a fallback for a generic OpenFlow switch (limited features).
- **P4Runtime**: a P4-capable switch via the P4Runtime API.

The driver list is at `https://docs.onosproject.org/` and the source code at `https://github.com/opennetworkinglab/onos/tree/master/drivers`.

### Driver development model

Adding a new switch model:

1. Fork the ONOS repository, and add a driver class extending the `Pipeliner` abstract class.
2. Implement the `filter()`, `forward()`, and `next()` methods.
3. Register the driver with the device manager via the `@Component` annotation.
4. Test against the switch (interoperability testing).
5. Submit a pull request upstream.

Cycle time: 2 to 4 weeks for a standard pipeline (for example, a new Broadcom OFDPA variant). 3 to 6 months for a novel pipeline (for example, a new Noviflow P4-based switch). Much faster than TTP (years) or a specification revision (years).

## 6.1.4 Current state: narrow adoption outside ONOS

### Trellis SD-Fabric

The clearest Flow Objectives success is the **Trellis** project: the ONF open-source SD-Fabric for a leaf-spine data centre. Trellis uses:

- Underlay: a leaf-spine IP fabric with BGP unnumbered.
- Control plane: ONOS running Flow Objectives, programming Broadcom OFDPA switches.
- Application: **SEBA** (SDN Enabled Broadband Access) for telco central office PON (Passive Optical Network) virtualisation.

Trellis production deployment: Comcast and the AT&T Central Office Re-architected as Datacenter (CORD) initiative from 2016 to 2020. Small-scale but production.

Trellis documentation: `https://opennetworking.org/trellis/`.

### CORD (Central Office Re-architected as Datacenter)

CORD is the ONF telco initiative that replaces a traditional Central Office with general-purpose data centre infrastructure running virtualised telco functions. Components:

- Compute: OpenStack or Kubernetes.
- Underlay: Trellis (the switch plus ONOS plus Flow Objectives).
- VNF (Virtual Network Functions): virtualised BNG, vCSR, vOLT.

CORD documentation: `https://opennetworking.org/cord/`.

### Adoption outside ONOS

**OpenDaylight**: did not adopt Flow Objectives. ODL applications write raw OpenFlow flow tables directly. Rationale: the ODL multi-southbound architecture (not just OpenFlow but also NETCONF, BGP-LS, gNMI) predates the Flow Objectives abstraction.

**Ryu, POX, Faucet**: legacy first-generation controllers, predating Flow Objectives. They write raw OpenFlow.

**Commercial vendor controllers** (Cisco ACI, Juniper Contrail): use a native proprietary API, not the OpenFlow standard. Flow Objectives is not relevant.

**Kubernetes SDN** (OVN-Kubernetes, Calico, Cilium): use native constructs (OVN logical flow, BGP, eBPF), and do not adopt Flow Objectives.

### Evaluation: abstraction is not a silver bullet

Flow Objectives lets an application be portable across Broadcom OFDPA, Corsa, and Noviflow inside the ONOS ecosystem. But:

**Limitation 1**: portability stays inside the ONOS driver catalog. A switch without a driver is not supported. In practice, the ONOS driver catalog is limited to about 10 switch models.

**Limitation 2**: abstraction leak. The ForwardingObjective flavor SPECIFIC versus VERSATILE is implicit about silicon constraints. The programmer still needs to know the pipeline structure to optimise.

**Limitation 3**: it does not replace P4. P4 solves fragmentation through a programmable data plane (compile down). Flow Objectives solves it through abstraction at the controller (translate at runtime). P4 is more powerful long term but requires silicon support. Flow Objectives ships today on existing silicon.

> **Common misconception:** "Flow Objectives is an ONOS-specific feature, so it is not a standard."
>
> **Reality:** Flow Objectives is a design pattern (a layered abstraction above OpenFlow), applicable to any controller. The OpenDaylight choice not to adopt it was a strategic decision, not evidence that Flow Objectives does not work outside ONOS. The pattern can be ported to ODL or to a custom controller. "ONOS-specific" is an implementation reality, not a design constraint.

> **Key point:** Flow Objectives expresses the pattern "abstraction in the controller, not in the switch", which fits SDN economic reality. Switch firmware updates are expensive, and vendors are not incentivised to push complexity into the switch (the TTP failure proves this). A controller abstraction layer is cheap to iterate. A single codebase change updates all users. Flow Objectives is the pragmatic middle path between raw OpenFlow (an application burden) and P4 (a silicon burden). The lesson: when designing an abstraction, locate complexity in the most flexible layer, not the most rigid layer.

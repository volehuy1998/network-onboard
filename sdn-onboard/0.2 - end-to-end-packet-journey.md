# 0.2. Hành trình packet End-to-End: Từ Pod A đến Pod B qua OVS, Geneve và OVN

> **Môi trường thực hành:** Ubuntu 22.04 LTS, OVS 2.17.9, OVN 22.03.8
> **Khối:** 0, Tổng quan và hướng dẫn đọc
> **Phần:** Phần 0 (Hướng dẫn nhập môn)
> **Kế hoạch:** `plans/sdn-foundation-architecture.md` §E.11, C10 cross-cutting view
> **Yêu cầu kiến thức:** Không. Đây là cái nhìn tổng thể, nên đọc TRƯỚC khi đi sâu vào từng Khối, hoặc đọc lại SAU khi hoàn thành để nắm bắt bức tranh toàn cảnh.
> **Mục tiêu:** Theo dõi hành trình của một packet đi từ Pod A trên Chassis X qua Overlay OVN tới Pod B trên Chassis Y. Mỗi chặng đường đều được tham chiếu đến các Khối cụ thể trong giáo trình.
> **Nguồn tham khảo:** OVN Architecture Guide (ovn-architecture(7)), Open vSwitch Design Documentation, RFC 8926 Geneve, NSDI 2015 "Design and Implementation of Open vSwitch" (Pfaff et al.).

---

## Mục tiêu bài học

Sau khi hoàn thành phần này, bạn sẽ nắm vững:

1. **Mô tả (Hiểu)** chuỗi 12 giai đoạn mà packet đi qua khi Pod A gửi dữ liệu tới Pod B trên các Chassis khác nhau: Từ VIF (veth) → Ingress OVS → Classifier → OpenFlow Actions → Đóng gói Geneve → NIC vật lý → Truyền dẫn qua Tunnel → Chassis đích → Giải mã (Decap) → VIF đích.
2. **Phân tích (Phân tích)** vai trò của từng thành phần (Kernel Datapath, Megaflow Cache, Classifier, OVN Logical Flow, ovn-controller) trong từng giai đoạn.
3. **Áp dụng (Thực hành)** các công cụ chẩn đoán (ovs-appctl ofproto/trace, ovn-trace, tcpdump trên veth) để tái hiện hành trình packet trên môi trường thực tế.
4. **Đánh giá (Đánh giá)** mức độ trừu tượng phù hợp cho từng bài toán chẩn đoán (khi nào dùng tcpdump, khi nào dùng ovn-trace, khi nào dùng ofproto/trace).

Yêu cầu kiến thức: Không. Phần này đóng vai trò là "mục lục" của giáo trình dựa trên dòng chảy dữ liệu.

Tất cả các bước trong tài liệu này giả định một sơ đồ chuẩn của môi trường Production như mô tả trong **ovn-architecture(7)**:

```
│                                                                      │
│  ┌─────────────────────────┐        ┌─────────────────────────┐      │
│  │   Chassis X (Hypervisor)│        │   Chassis Y (Hypervisor)│      │
│  │                         │        │                         │      │
│  │  Pod A (10.0.0.5)       │        │       Pod B (10.0.0.10) │      │
│  │     │                   │        │            │            │      │
│  │  veth-podA              │        │         veth-podB       │      │

│  │     │                   │        │            │            │      │
│  │  [Geneve tunnel port] ──┼────────┼──► [Geneve tunnel port] │      │
│  │     │                   │        │            │            │      │
│  │  [eth0 NIC]             │        │      [eth0 NIC]         │      │
│  └───────┬─────────────────┘        └────────────┬────────────┘      │
│          │                                       │                   │

│                                                                      │
```

Pod A gửi packet TCP tới Pod B. Hành trình này bao gồm 12 giai đoạn chi tiết. Mỗi giai đoạn liệt kê: (a) chuyện gì xảy ra, (b) lệnh kiểm chứng thực tế, (c) link tới Phần đi sâu cụ thể.

```

─────────────────                ─────────────────        ──────          ─────────────────                 ─────────────────
[1] App write socket()

```

## 0.2.3 Giai đoạn 1, Pod A application gọi `send()`

**Chuyện gì xảy ra:** Container/Pod A nằm trong network namespace riêng. Application gọi `send()` qua TCP socket, kernel TCP stack đẩy packet xuống tầng IP, IP tra route table cục bộ → next-hop là default gateway (LRP của OVN, qua OVN-K8s là gateway router). Source MAC = MAC của veth phía namespace, dest MAC = MAC của LRP gateway.

**Kiểm chứng:**

```bash
[chassis-X]# nsenter -t $(pidof container-podA) -n ip route
default via 10.0.0.1 dev eth0 onlink
10.0.0.0/24 dev eth0 proto kernel scope link src 10.0.0.5

[chassis-X]# nsenter -t $(pidof container-podA) -n ss -tnp
ESTAB  0  0  10.0.0.5:54321  10.0.0.10:80  users:(("app",pid=12345,fd=4))
```

**Đi sâu:** Phần 8.0 (Linux namespaces + cgroups), Phần 8.1 (Linux bridge + veth + macvlan).

---

## 0.2.4 Giai đoạn 2, Pod A namespace veth-podA TX

**Chuyện gì xảy ra:** Packet đi qua interface `eth0` của Pod A namespace. `eth0` thực chất là một đầu của **veth pair**, đầu kia (ví dụ `eth-podA-X`) nằm ở root namespace. Driver veth (`net/core/veth.c` trong kernel Linux 5.15) chuyển packet zero-copy từ TX queue đầu này sang RX queue đầu kia.

**Kiểm chứng:**

```bash
[chassis-X]# nsenter -t $(pidof container-podA) -n ip -o link
1: lo: ...
N: eth0@if<peer>: ...

[chassis-X]# ip -o link | grep eth-podA-X
M: eth-podA-X@if<N>: ...

# Trace traffic xuất từ Pod A:
[chassis-X]# nsenter -t $(pidof container-podA) -n tcpdump -i eth0 -n
```

**Đi sâu:** Phần 8.1 §"Linux veth pair" + `linux-onboard/file-descriptor-deep-dive.md` cho FD lifecycle khi `clone(CLONE_NEWNET)`.

## 0.2.5 Giai đoạn 3, Root namespace veth peer RX, br-int port

**Chuyện gì xảy ra:** Đầu peer của veth (`eth-podA-X`) là OVS port của `br-int`. Kernel OVS module (`openvswitch.ko`) hook vào `dev_queue_xmit()`, chặn packet trước khi nó đi qua Linux bridge stack thông thường, đẩy vào datapath OVS.

**Kiểm chứng:**

```bash
[chassis-X]# ovs-vsctl list-ports br-int | grep eth-podA-X
eth-podA-X

[chassis-X]# ovs-ofctl show br-int | grep eth-podA-X
 15(eth-podA-X): addr:de:ad:be:ef:00:0a config: 0 state: 0
                    speed: 10000 Mbps now, 10000 Mbps max
```

OFPort 15 là số hiệu OpenFlow của port này.

**Đi sâu:** Phần 9.1 (OVS 3-component architecture, ovs-vswitchd userspace + openvswitch.ko kernel), Phần 13.4 (br-int architecture).

## 0.2.6 Giai đoạn 4, Datapath megaflow lookup

**Chuyện gì xảy ra:** Kernel datapath nhận packet, extract miniflow (key gồm các trường mà classifier quan tâm), tra cache 3 tier theo thứ tự:

1. **EMC** (Exact Match Cache, 8K entries/PMD trong DPDK; cache O(1) hash exact-match trên N-tuple đầy đủ).
2. **SMC** (Signature Match Cache, OVS 2.15+, 16K entries, cache hash signature → ukey).
3. **Megaflow cache** (TSS classifier với wildcarding, slow path cuối trong kernel).

Hit cache nào → áp dụng action của ukey tương ứng → packet ra ngay. Miss tất cả → upcall lên userspace.

**Kiểm chứng:**

```bash
[chassis-X]# ovs-appctl dpif/show
system@ovs-system:
        lookups: hit:1234567 missed:8901 lost:0
        flows: 4521
        masks: hit:1230000 total:1234567 hit/pkt:0.99
        port 1: ovs-system (internal)
        port 2: br-int (internal)
        ...

[chassis-X]# ovs-appctl dpif-netdev/pmd-stats-show     # nếu DPDK
[chassis-X]# ovs-appctl dpif/dump-flows br-int | head -5
```

**Đi sâu:** Phần 9.2 (kernel datapath + EMC/SMC/megaflow + ukey lifecycle), Phần 9.15 (classifier TSS + cls_subtable).

---

## 0.2.7 Giai đoạn 5, Upcall + ofproto-dpif xlate (slow path)

**Chuyện gì xảy ra:** Nếu cache miss tất cả (packet đầu tiên của flow mới), kernel `openvswitch.ko` gửi **upcall** Netlink Generic genl tới `ovs-vswitchd`. Userspace ofproto-dpif chạy `xlate_actions()` để mô phỏng full OpenFlow pipeline trên flow này, sinh ra:

- **Action set** thực tế (output, set_field, encap, ...).
- **Megaflow mask** (wildcard mask, qua tracking `wc` flag trong xlate).

Userspace install ukey + flow rule vào kernel datapath qua netlink reverse path. Packet đầu tiên cũng được userspace inject lại.

**Kiểm chứng:**

```bash
[chassis-X]# ovs-appctl upcall/show
system@ovs-system:
  threads: 4
  upcalls: 8901  miss: 8000  flow_install: 4520  ...

[chassis-X]# ovs-appctl ofproto/trace br-int \
    "in_port=15,ip,nw_src=10.0.0.5,nw_dst=10.0.0.10,..."
# Output: full pipeline trace với metadata, register, action
```

**Đi sâu:** Phần 9.1 §9.1.Y ofproto-dpif xlate tier 2 (Phase I.A1, function call chain `xlate_actions` → `do_xlate_actions` → action translation), Phần 9.25 (ofproto/trace 3 advanced GE).

## 0.2.8 Giai đoạn 6, Logical pipeline ingress (LS_in_*)

**Chuyện gì xảy ra:** Trong xlate, flow rule trên `br-int` được ovn-controller tổ chức theo OpenFlow table dạng pipeline OVN logical (translate từ 24 ingress + 10 egress table cho LS, OVN 22.03). Mỗi table là một stage logical (ví dụ: `ls_in_check_port_sec`, `ls_in_acl`, `ls_in_lookup_fdb`). Packet đi qua tuần tự, mỗi stage match priority cao nhất → action `next;` (goto stage sau) hoặc `drop;`/`output;`.

**Kiểm chứng:**

```bash
# Logical view (high-level):
[chassis-X]# ovn-trace ls1 \
    'inport == "podA" && eth.src == ... && ip4.src == 10.0.0.5 && ip4.dst == 10.0.0.10 && tcp'

# Output show từng stage ls_in_* match flow nào với priority gì.
```

**Đi sâu:** Phần 13.2 (Logical Switch + Logical Router pipeline 24+10 stage), Phần 20.7 Level 1-2 (tutorial gradient ovn-trace).

---

## 0.2.9 Giai đoạn 7, ls_in_l2_lkup, chọn outport

**Chuyện gì xảy ra:** Stage cuối ingress là `ls_in_l2_lkup` (table 27 trong OVN 22.03). Logical Flow ở đây match `eth.dst` với address binding của LSP. Action `outport = "podB"; output;` set logical outport thành tên LSP đích, sau đó nhảy vào pipeline egress.

Nếu MAC dest chưa có trong NBDB (unknown), action fallback `outport = MC_FLOOD; output;` flood tới mọi port của LS.

**Kiểm chứng:**

```bash
[chassis-X]# ovn-sbctl lflow-list ls1 | grep "ls_in_l2_lkup"
  table=27(ls_in_l2_lkup), priority=50, match=(eth.dst == 00:00:00:00:00:0b),
    action=(outport = "podB"; output;)
  table=27(ls_in_l2_lkup), priority=0, match=(1),
    action=(outport = "_MC_unknown"; output;)
```

**Đi sâu:** Phần 13.2 §"L2 lookup", Phần 17.0 (FDB poisoning case study, hành vi unknown MAC).

---

## 0.2.10 Giai đoạn 8, Logical pipeline egress (LS_out_*)

**Chuyện gì xảy ra:** Sau khi outport được set, packet đi qua egress pipeline (10 stage cho LS): `ls_out_pre_acl`, `ls_out_acl`, `ls_out_qos`, `ls_out_stateful`, `ls_out_port_sec`, `ls_out_port_sec_l2`. Egress áp dụng ACL chiều ngược lại (`to-lport`) + port_security trên outport.

Nếu packet pass hết, stage cuối `ls_out_port_sec_l2` action `output;` → ovn-controller dịch sang OpenFlow output thực tế: hoặc `output:<local-OFPort>` nếu podB ở local chassis, hoặc `output:<tunnel-OFPort>` nếu podB ở chassis khác (qua `Port_Binding.chassis` SBDB).

**Kiểm chứng:**

```bash
[chassis-X]# ovn-sbctl list Port_Binding podB
_uuid               : ...
chassis             : <chassis-Y-uuid>     ← podB ở chassis Y
logical_port        : "podB"
tunnel_key          : 2
type                : ""
up                  : true
```

Vì `chassis = chassis-Y`, br-int Chassis X sẽ encap + output sang tunnel.

**Đi sâu:** Phần 13.2 §"egress pipeline", Phần 13.5 (Port_Binding 8 types + claim mechanism), Phần 13.7 §13.7.8 (consider_port_binding source code).

## 0.2.11 Giai đoạn 9, Geneve encap (TLV class 0x0102)

**Chuyện gì xảy ra:** Khi outport ánh xạ tới remote chassis, ovn-controller sinh OpenFlow flow ở table 65 với 3 action chính (xem Phần 13.7.8 §"Geneve TLV encoding"):

```
set_field:<datapath-tunnel_key>->tun_id        # 24-bit Geneve VNI = LS tunnel_key
set_field:<outport-key>->NXM_NX_TUN_METADATA0[0..31]  # 32-bit logical outport
move:NXM_NX_REG14[0..14]->NXM_NX_TUN_METADATA0[16..30] # 15-bit logical inport
output:<tunX-Y-OFPort>
```

Geneve TLV class `0x0102` (OVN reserved trong IANA registry) mang theo metadata logical xuyên chassis. Outer header: Eth(chassis MAC) + IPv4(underlay) + UDP(dest 6081) + Geneve(VNI + TLV).

**Kiểm chứng:**

```bash
[chassis-X]# ovs-appctl ofproto/trace br-int "in_port=15,..." | grep -A 5 "set_field:.*tun_id"
[chassis-X]# tcpdump -i eth0 -nn udp port 6081 -X | head -40
# Decode TLV class 0x0102 byte-by-byte (Phần 9.27 §Geneve TLV deep-dive)
```

**Đi sâu:** Phần 11.0 (VXLAN/Geneve/STT, RFC 8926), Phần 13.7 §13.7.8 (`controller/physical.c put_encapsulation()`), Phần 9.27 (3-tier parallel diagnostic + Geneve TLV).

## 0.2.12 Giai đoạn 10, NIC physical TX + underlay routing

**Chuyện gì xảy ra:** Packet đã được encap đầy đủ thành UDP/Geneve packet với outer src=chassis-X-IP, outer dst=chassis-Y-IP. Kernel IPv4 stack route lookup → tìm next-hop trên underlay (có thể là ToR switch). Driver NIC `eth0` TX packet xuống cáp.

Underlay phải đảm bảo:

- MTU đủ lớn cho overhead Geneve (66 byte = 14 outer Eth + 20 outer IPv4 + 8 UDP + 8 Geneve base + 16 TLV). Nếu underlay MTU=1500 và inner=1500 → fragmentation hoặc PMTU drop.
- UDP port 6081 không bị firewall drop.
- Routing/switching giữa chassis-X và chassis-Y không có loop, có ARP resolve cho next-hop.

**Kiểm chứng:**

```bash
[chassis-X]# ip route get <chassis-Y-IP>
<chassis-Y-IP> via <next-hop> dev eth0 src <chassis-X-IP>

[chassis-X]# ovs-appctl bfd/show
        State: UP        ← BFD session với chassis-Y healthy
        Diagnostic: No Diagnostic
        Min tx: 1000ms   Min rx: 1000ms

[chassis-X]# tcpdump -i eth0 -nn udp port 6081 and host <chassis-Y-IP>
```

**Đi sâu:** Phần 11.1 (overlay MTU + PMTUD + offload), Phần 13.6 (HA chassis + BFD).

## 0.2.13 Giai đoạn 11, Chassis Y NIC RX + Geneve decap

**Chuyện gì xảy ra:** NIC `eth0` Chassis Y nhận UDP/6081 packet. Kernel stack deliver lên OVS tunnel port (`tunY-X` trong br-int Chassis Y). Br-int Chassis Y nhận packet với:

- `in_port = <tunnel-OFPort>`.
- `tun_id = <datapath tunnel_key>` (Geneve VNI).
- `tun_metadata0` chứa outport (32-bit) + inport (15-bit) đã encode bên Chassis X.

Table 0 của br-int Chassis Y có flow chuyên decap (do ovn-controller cài):

```
in_port=<tunnel-port>,tun_id=<tk>:
    move:NXM_NX_TUN_METADATA0[0..31]->NXM_NX_REG15[0..31]   ← restore outport
    move:NXM_NX_TUN_METADATA0[16..30]->NXM_NX_REG14[0..14]  ← restore inport
    move:NXM_NX_TUN_ID[0..23]->OXM_OF_METADATA[0..23]       ← restore datapath
    resubmit(,<egress-table>)
```

Sau decap, packet rejoin pipeline egress logical của LS, lần này trên Chassis Y với outport đã set tương ứng podB local.

**Kiểm chứng:**

```bash
[chassis-Y]# ovs-vsctl get Interface tunY-X options
{remote_ip="<chassis-X-IP>", key=flow}

[chassis-Y]# ovs-appctl ofproto/trace br-int \
    "in_port=<tunnel-OFPort>,tun_id=0x1,tun_metadata0=0x..."
```

**Đi sâu:** Phần 13.4 (br-int Chassis Y architecture), Phần 13.7 §13.7.8 (decap mirror với encap), Phần 9.27 (3-tier diagnostic chassis Y side).

## 0.2.14 Giai đoạn 12, Br-int Y output → veth peer → Pod B namespace

**Chuyện gì xảy ra:** Stage cuối egress trên br-int Chassis Y có flow `output:<podB-OFPort>`. Br-int chuyển packet tới OVS port nối tới veth peer của Pod B. Driver veth (peer-to-peer) chuyển packet sang đầu kia (`eth0` trong Pod B namespace). Pod B namespace kernel stack deliver lên application qua TCP socket.

**Kiểm chứng:**

```bash
[chassis-Y]# ovs-vsctl list-ports br-int | grep eth-podB-Y
eth-podB-Y

[chassis-Y]# nsenter -t $(pidof container-podB) -n tcpdump -i eth0 -n
# Pod B nhận packet với outer underlay đã strip + L2 dst = MAC veth Pod B
```

**Đi sâu:** Phần 8.1 (veth peer), Phần 13.5 (`type=""` = vif, default LSP type).

## 0.2.15 Tổng hợp lệnh chẩn đoán theo giai đoạn

Khi debug "podA không ping được podB", chạy lần lượt theo tier:

| Tier | Câu hỏi | Lệnh |
|------|---------|------|
| Logical | OVN có ý đồ cấu hình đúng? | `ovn-trace ls1 '<flow-expr>'` (Phần 20.7 Level 1-3) |
| OpenFlow | OpenFlow trên br-int đúng? | `ovs-appctl ofproto/trace br-int '<flow>'` (Phần 9.25) |
| Datapath | Kernel cache có entry? | `ovs-appctl dpif/dump-flows br-int` (Phần 9.2) |
| Physical X | Chassis X gửi đi? | `tcpdump -i eth0 udp port 6081` (Phần 11.1, 13.7.8) |
| Underlay | Underlay nuốt packet? | `tcpdump` trên router/ToR; check MTU, BFD (Phần 13.6) |
| Physical Y | Chassis Y nhận? | `tcpdump -i eth0 udp port 6081` |
| OpenFlow Y | Br-int Y decap đúng? | `ovs-appctl ofproto/trace br-int 'in_port=<tun>,tun_id=...'` |
| Pod B | Pod B nhận? | `nsenter -n tcpdump -i eth0` |

Nếu fail ở tier nào, đó là root cause. Phần 9.27 mô tả 3-tier parallel diagnostic framework chi tiết hơn.

## 0.2.16 ▶ Guided Exercise, Tự trace packet qua 12 giai đoạn

**Mục tiêu:** Tự bạn chạy hành trình packet trên lab thực tế hoặc Mininet sandbox, đối chiếu output mỗi giai đoạn với mô tả trong Phần này.

**Setup giả định:** OVN sandbox 2-chassis (xem Phần 0.1 §"Two-node chassis pair"). Tạo 1 Logical Switch `ls1`, 2 LSP `podA` (chassis-X) + `podB` (chassis-Y), addresses `00:00:00:00:00:0a 10.0.0.5` và `00:00:00:00:00:0b 10.0.0.10`.

**Bước thực hiện:**

```bash
# Pre-check: cluster healthy?
[host]# ovn-nbctl show              # podA + podB exist
[host]# ovn-sbctl show               # 2 chassis registered
[host]# ovs-appctl bfd/show          # tunnel UP

# Chạy ping từ podA
[chassis-X]# nsenter -t $(pidof container-podA) -n ping -c 3 10.0.0.10

# Trace song song trên cả 8 tier (chạy mỗi lệnh trên 1 terminal hoặc tmux pane)
[host]# ovn-trace ls1 'inport == "podA" && ...'
```

**Bài học:**

1. Mỗi giai đoạn có **một loại evidence** xác minh được. Không có giai đoạn nào "tin tưởng vô điều kiện", luôn có lệnh kiểm chứng.
2. Khi 1 giai đoạn fail, không bao giờ skip sang giai đoạn sau. Fix root cause trước.
3. 12 giai đoạn này là **xương sống** của mọi debug session OVN. Nội tâm hoá xong, bạn sẽ tự dẫn dắt được mọi incident report.

## 0.2.17 Điểm cốt lõi cần nhớ

**Một.** Hành trình packet OVN có 12 giai đoạn rõ ràng, từ application call `send()` tới application call `recv()` ở chassis đích. Mỗi giai đoạn có command kiểm chứng cụ thể.

**Hai.** Tách bạch **logical view** (ovn-trace, NBDB ý đồ) khỏi **physical view** (ofproto/trace, OVS OpenFlow). Hai layer này có thể khớp hoặc lệch (nếu lệch = ovn-controller chưa sync).

**Ba.** Geneve TLV class `0x0102` mang metadata logical (datapath tunnel_key + logical inport + outport) xuyên chassis. Mất TLV = mất logical context = packet drop hoặc xử lý sai pipeline.

**Bốn.** Underlay phải đảm bảo MTU đủ + UDP 6081 không bị firewall + BFD UP. Đây là 3 cause phổ biến nhất của "podA không ping được podB cross-chassis".

**Năm.** Phần 0.2 là entry point, mọi câu chuyện debug cụ thể nằm ở Phần đi sâu. Cross-link bảng:

| Giai đoạn | Phần đi sâu |
|-----------|-------------|
| 1-2 namespace + veth | 8.0, 8.1 |
| 3 br-int port | 9.1, 13.4 |
| 4 datapath megaflow | 9.2, 9.15 |
| 5 upcall + xlate | 9.1 §9.1.Y, 9.25 |
| 6-7 LS ingress + l2_lkup | 13.2, 17.0, 20.7 |
| 8 LS egress + Port_Binding | 13.2, 13.5, 13.7.8 |
| 9 Geneve encap | 11.0, 13.7.8, 9.27 |
| 10 NIC TX + underlay | 11.1, 13.6 |
| 11 Chassis Y decap | 13.4, 13.7.8, 9.27 |
| 12 br-int Y output + Pod B | 8.1, 13.5 |

**Sáu.** Khi quay lại Phần 0.2 sau khi đã đọc các Phần đi sâu, bạn sẽ thấy "tour" này như bảng mục lục theo dòng chảy dữ liệu. Đây là cách tổ chức học cuối cùng cho kỹ sư vận hành.

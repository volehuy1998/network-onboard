## 9.12.1 Năm bước upgrade không mất data plane

Compass Ch P quy trình:

**Bước 1 — Install packages KHÔNG restart daemon**:

```bash
[host]# apt install openvswitch-switch openvswitch-common
```

**Bước 2 — Restart ovsdb-server graceful**:

```bash
[host]# ovs-appctl -t ovsdb-server exit --cleanup
[host]# systemctl start ovsdb-server
```

`exit --cleanup` flush pending transaction trước exit. Down time sub-second.

**Bước 3 — Schema conversion nếu cần**:

```bash
[host]# ovsdb-tool needs-conversion /etc/openvswitch/conf.db \
         /usr/share/openvswitch/vswitch.ovsschema
```

Nếu yes:

```bash
[host]# ovsdb-tool convert /etc/openvswitch/conf.db \
         /usr/share/openvswitch/vswitch.ovsschema
```

**Bước 4 — Restart ovs-vswitchd atomic**:

```bash
[host]# systemctl reload-or-restart openvswitch-switch
```

Compass Ch P: `--bundle`-style atomic replacement — new instance start trước, subscribe OVSDB, sau đó old exit. Data plane không drain.

**Bước 5 — Kernel module reload (nếu upgrade)**:

```bash
[host]# rmmod openvswitch
[host]# modprobe openvswitch
```

Compass Ch P: reload kernel module **brief evicts megaflow cache** nhưng **does not drop existing connections** vì conntrack state lives separately.

## 9.12.2 Ba golden rule (compass Ch 19)

**Rule 1 — Schema first**. Convert OVSDB schema TRƯỚC khi start binary mới. Ngược lại → crash loop.

**Rule 2 — One daemon at a time**. KHÔNG restart ovsdb-server và ovs-vswitchd cùng lúc. Risk split state.

**Rule 3 — Không đồng thời upgrade OVS + kernel**. OVS userspace và kernel module có compatibility range — upgrade cùng lúc → risk mismatch.

---

## 9.12.3 Cluster upgrade cho OVSDB HA

Cho OVSDB Raft clustered (Phần 10.1):

1. Upgrade một node tại một thời điểm.
2. Node rời cluster qua `ovs-appctl cluster/leave`.
3. Install binary mới, schema convert, start.
4. Node rejoin qua `ovsdb-tool join-cluster`.
5. Wait sync (`cluster/status`).
6. Upgrade node tiếp theo.

Compass Ch N: "Once a server leaves a cluster it cannot rejoin — create a fresh member." Không phải "leave rồi join lại" — phải recreate node DB từ scratch.

---

## 9.12.4 Downgrade strategy

Downgrade phức tạp hơn upgrade. Nếu schema đã convert, OVSDB cũ không đọc được DB mới.

Best practice: test upgrade trên staging trước; có plan backup/restore rõ ràng:

```bash
[host]# ovsdb-client backup > /tmp/db.bak      # trước upgrade
[host]# ovsdb-client restore < /tmp/db.bak     # nếu cần downgrade
```

```bash
[host]# ping -I 10.99.0.1 10.99.0.2 -i 0.1 > /tmp/ping.log &
```

**3.** Stop ping sau 30s: `sleep 30 && kill %1`

```bash
[host]# grep "packets transmitted" /tmp/ping.log
```

Expect: loss 0% hoặc < 1%.

**5.** Cleanup: `ovs-vsctl del-br br-test`

## 9.12.X Anatomy Template A — Pre-upgrade checklist

**Bối cảnh khi chạy:** Trước mỗi rolling upgrade, verify current state + compatibility với target version. Go/no-go decision.

**9-attribute pre-upgrade checklist:**

| Attribute | Command | Red flag |
|-----------|---------|----------|
| ovs-vswitchd version | `ovs-vswitchd --version` | Jump >1 major = risky |
| Kernel module version | `cat /sys/module/openvswitch/version` | Mismatch userspace = incompat |
| OVSDB schema | `ovsdb-tool schema-version /etc/openvswitch/conf.db` | Downgrade = backup required |
| Total rules | `ovs-appctl memory/show` (rules) | Drop post-upgrade = loss |
| Active ofconns | `memory/show` (ofconns) | Drop = controller disconnect |
| Cơ chế gọi ngược rate | `coverage/show upcall_flow_add` | Spike >10× post = cache invalidation |
| Backup DB | file `/backup/conf.db.$(date)` exists | Missing = risky go |
| Peer compat | OVS NEWS 2.17 → 3.3 path | Skip version = interim required |
| Maintenance window | 6-10 hour recommended | Tight window = rush error |

**Kịch bản bẻ gãy:**

- **Kernel module mismatch**: `dmesg | grep openvswitch` warning. Fix: install matching kernel module.
- **Schema conversion fail**: `ovsdb-tool needs-conversion` returns yes. Fix: `ovsdb-tool convert`.
- **Rule loss sau upgrade**: grammar change. Fix: restart ovn-controller re-compile từ NBDB.
- **Cơ chế gọi ngược rate spike**: revalidator sweep invalidate cache. Transient 1-5 phút.

## 9.12.Y Hiểu sai + Key Topic

> **Hiểu sai phổ biến 1:** *"Skip major version (2.17 → 3.3) safe với `apt upgrade`."*
>
> **Thực tế:** OVS release cycle 6-12 tháng, migration path từ 1-2 major previous. Skip nhiều version đụng untested schema conversion chain. Pattern safe: incremental 2.17 → 2.18 → ... → 3.3 qua distro backport hoặc manual build.

> **Hiểu sai phổ biến 2:** *"Rolling restart = no downtime."*
>
> **Thực tế:** Sub-second gói tin loss khi flow cache invalidate. SLA 99.99% budget 5-second/restart. 10 chassis × 5s = 50s cumulative. Tunnel pre-warmed + CT state preserved (OVS 2.15+) giảm impact.

> **Chủ đề trọng tâm:** "Golden 3 rules + 5-step choreography" zero data plane loss. Golden: (1) Schema first trước binary. (2) One daemon at a time. (3) Không đồng thời OVS + kernel. 5-step: backup / stop / install / convert / start.

## 9.12.Z Common failure modes

**Failure 1 — Schema conversion fail.** Custom OVSDB column. Fix: restore backup DB.

**Failure 2 — Bond flap during restart.** LACP slow-timer 90s. Fix: fast-timer trước restart.

**Failure 3 — Flow rule loss.** Grammar removed. Fix: patch controller trước OVS.

**Failure 4 — OVN-controller disconnect SBDB.** TLS cert expiry. Fix: validate pre-upgrade.

## Điểm cốt lõi cần nhớ

Thứ nhất, OVS upgrade được thiết kế không mất data plane — năm bước carefully orchestrated.

Thứ hai, ba golden rule (schema first, one daemon at a time, không đồng thời OVS + kernel) tránh phần lớn upgrade failure.

Thứ ba, cluster upgrade khác single-node — node rời cluster không thể rejoin, phải recreate.

Thứ tư, backup DB trước mọi production change là insurance cheap.

Thứ năm, Anatomy pre-upgrade checklist 9-attribute (version + schema + rules + ofconns + upcall + backup + peer compat + window). Go/no-go decision.

Thứ sáu, skip major version = risk. Incremental safe pattern.

Thứ bảy, rolling restart sub-second data plane loss. Budget SLA math nếu 99.99%.

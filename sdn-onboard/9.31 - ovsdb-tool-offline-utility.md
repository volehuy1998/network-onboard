## Learning objectives

After completing this part, the engineer can:

1. **Understand** where `ovsdb-tool` fits in the OVSDB tool family: it is an **offline file utility** (it does not connect to a running `ovsdb-server`), in contrast to `ovsdb-client` (online) and `ovs-vsctl` (a high-level wrapper). It exposes 15 commands grouped into five categories: creation, schema management, integrity, inspection, and cluster lifecycle.
2. **Apply** the workflow `create-cluster` plus `join-cluster` to bootstrap an `OVN_Southbound` three-node Raft cluster from scratch, a critical skill for disaster recovery and production cluster setup.
3. **Analyze** the output of `show-log` and `check-cluster` to perform forensic analysis of the database state when a cluster hits split-brain or corruption.
4. **Evaluate** operational safety: when to use `ovsdb-tool` (server stopped) versus `ovsdb-client` (server running), and why running `compact` on a database that is being served corrupts it.

Prerequisite knowledge: understand the OVSDB on-disk JSON file format and the transactional log from Part 10.0, and Raft consensus from Part 10.1.

## 9.31.1 Where `ovsdb-tool` sits in the OVSDB tool family

OVS has three main tools that interact with the database, each for a different purpose:

| Tool | Online or offline | Use case |
|------|-------------------|----------|
| `ovsdb-tool` | **Offline** (direct file access) | Bootstrap a cluster, schema migration, forensics, compact |
| `ovsdb-client` | **Online** (connects to `ovsdb-server`) | Live monitor, ad-hoc transact, dump table |
| `ovs-vsctl`, `ovn-nbctl`, `ovn-sbctl` | **Online** (wrappers over the OVSDB protocol) | Daily admin, high-level abstractions |

**Why an offline tool is needed.** Some tasks cannot run while the server is up:

1. **Bootstrap a new cluster:** the server cannot start until the file is initialized.
2. **Schema migration:** convert the database from an old version to a new one while the server is stopped for upgrade.
3. **Compact:** defragment the file (reduce its size, merge transaction log entries). A live server writing to the file would conflict.
4. **Post-incident forensics:** dump the transaction log from a corrupt file that the server cannot load.
5. **Cluster recovery:** convert a failed cluster into a standalone database to rescue the data.

The golden rule: `ovsdb-tool` runs only when the `ovsdb-server` for the corresponding database is **STOPPED**. Violating this rule causes a race condition, file corruption, and data loss.

## 9.31.2 Synopsis and 15 commands in five groups

```
ovsdb-tool [options] <command> [arguments]
```

### Group 1: database creation (3 commands)

| Command | Synopsis | Purpose |
|---------|----------|---------|
| `create` | `create [db [schema]]` | Initialize a standalone or active-backup database from a schema |
| `create-cluster` | `create-cluster [--election-timer=MS] DB CONTENTS LOCAL` | Create a Raft cluster, the first server |
| `join-cluster` | `join-cluster [--cid=UUID] DB NAME LOCAL REMOTE...` | Add a server to an existing cluster |

### Group 2: schema management (5 commands)

| Command | Synopsis | Purpose |
|---------|----------|---------|
| `convert` | `convert [db [schema [target]]]` | Convert a database to a new schema version |
| `needs-conversion` | `needs-conversion [db [schema]]` | Test whether a conversion is needed (returns an exit code) |
| `db-version` | `db-version [db]` | Print the version of a database |
| `schema-version` | `schema-version [schema]` | Print the version of a schema file |
| `db-cksum`, `schema-cksum` | `db-cksum [db]` and `schema-cksum [schema]` | Print the SHA-1 checksum of a file |
| `compare-versions` | `compare-versions A OP B` | Compare two version strings (`<`, `<=`, `==`, `>=`, `>`, `!=`) |

### Group 3: inspection and query (5 commands)

| Command | Synopsis | Purpose |
|---------|----------|---------|
| `query` | `query [--rbac-role=ROLE] [db] TRANSACTION` | Read-only query execution |
| `transact` | `transact [--rbac-role=ROLE] [db] TRANSACTION` | Read and write transaction (against an offline file) |
| `show-log` | `show-log [-m|--more]... [db]` | Dump the transaction log (`-m` selects verbose mode) |
| `db-name`, `schema-name` | none | Print the name of a database or schema |
| `db-cid`, `db-sid`, `db-local-address` | none | Print the cluster ID, server ID, or local address of a Raft node |

### Group 4: cluster integrity (3 commands)

| Command | Synopsis | Purpose |
|---------|----------|---------|
| `check-cluster` | `check-cluster DB...` | Verify cross-node consistency in a Raft cluster |
| `db-is-clustered`, `db-is-standalone` | none | Test the database format |
| `cluster-to-standalone` | `cluster-to-standalone DB CLUSTERDB` | Convert a failed cluster to a standalone database (rescue data) |

### Group 5: maintenance (1 command)

| Command | Synopsis | Purpose |
|---------|----------|---------|
| `compact` | `compact [db [target]]` | Defragment, merge log entries, reduce file size |

## 9.31.3 Options in detail

| Option | Default | Role |
|--------|---------|------|
| `-h`, `--help` | none | Help |
| `-V`, `--version` | none | OVS version |
| `-v[SPEC]`, `--verbose[=SPEC]` | INFO | Set the logging level for each module |
| `--log-file[=FILE]` | none | Enable file logging |
| `--syslog-target=HOST:PORT` | none | Remote syslog |
| `--syslog-method=METHOD` | `libc` | Syslog transport: `libc`, `unix:FILE`, `udp:IP:PORT`, or `null` |

**Default paths (OVS 2.17 on Ubuntu 22.04):**

- Database file: `/etc/openvswitch/conf.db` (or `/usr/local/etc/openvswitch/conf.db` for an install from source).
- Schema file: `/usr/share/openvswitch/vswitch.ovsschema`.

## 9.31.4 Anatomy: bootstrap a three-node Raft cluster for OVN_Southbound

**Bucket:** OVS. **Context:** disaster recovery or a brand-new cluster setup. **Purpose:** bring up a three-node OVN SB DB cluster from scratch. **Activity:** cluster lifecycle. **Mechanism:** `create-cluster` on node 1, then `join-cluster` on nodes 2 and 3.

**On node 1 (the first leader):**

```bash
# 1. Stop any existing ovsdb-server for the SB DB (if present)
systemctl stop ovn-sb-ovsdb

# 2. Create the cluster with the OVN_Southbound schema
ovsdb-tool create-cluster /etc/ovn/ovnsb_db.db \
  /usr/share/ovn/ovn-sb.ovsschema \
  tcp:10.0.0.1:6644

# 3. Verify the cluster ID and server ID
ovsdb-tool db-cid /etc/ovn/ovnsb_db.db
# Cluster UUID (note this for join-cluster later)

ovsdb-tool db-sid /etc/ovn/ovnsb_db.db
# Server UUID (unique to this node)

# 4. Start ovsdb-server in cluster mode
ovsdb-server /etc/ovn/ovnsb_db.db \
  --remote=ptcp:6642 \
  --pidfile --detach
```

**On nodes 2 and 3 (followers):**

```bash
# 5. Join the existing cluster
ovsdb-tool join-cluster \
  /etc/ovn/ovnsb_db.db \
  OVN_Southbound \
  tcp:10.0.0.2:6644 \
  tcp:10.0.0.1:6644 \
  tcp:10.0.0.3:6644

# 6. Start the server
ovsdb-server /etc/ovn/ovnsb_db.db \
  --remote=ptcp:6642 \
  --pidfile --detach
```

**Verify the cluster is healthy from any node:**

```bash
# 7. Check the cluster status (online tool, after the server starts)
ovs-appctl -t /var/run/ovn/ovnsb_db.ctl cluster/status OVN_Southbound

# Expected output:
# Cluster ID: a1b2c3d4-...
# Server ID: e5f6g7h8-... (Leader or Follower)
# 3 servers in the cluster, status "Active"
# Election timer: 10000 ms
```

## 9.31.5 Anatomy: `show-log` for forensic transaction-log analysis

**Bucket:** OVS. **Context:** post-incident analysis or split-brain debugging. **Purpose:** dump every transaction committed to the file. **Activity:** forensics. **Mechanism:** parse the Raft log entries from the binary file.

**Sample output (cluster mode):**

```
$ ovsdb-tool show-log /etc/ovn/ovnsb_db.db | head -20
record 0:
 Initial schema "OVN_Southbound" version 2.5.0 cksum "..."

record 1: 2026-04-25 10:23:45.123
 cluster term 5, prev_index 0, name "..."

record 2: 2026-04-25 10:23:46.234 "raft_initial_snapshot"
 entries [{...snapshot data...}]

record 3: 2026-04-25 10:24:01.456
 [insert Chassis a1b2c3d4-... name "hv01" hostname "compute01.example.com"]

record 4: 2026-04-25 10:24:01.789
 [insert Port_Binding ... logical_port "lsp1" datapath ...]
```

**Verbose mode `-m` (more):**

```bash
ovsdb-tool show-log -m /etc/ovn/ovnsb_db.db | head -30
# Adds the term, index, prev_term, and prev_index of each Raft entry
```

**Decode, five-attribute Anatomy:**

| Attribute | Sample value | Healthy / Warning / Critical |
|-----------|--------------|------------------------------|
| Record format | `record N: TIMESTAMP DESCRIPTION` | Healthy: timestamps increase monotonically |
| Initial schema record | record 0 | Critical: missing means the file is corrupt from the start |
| Snapshot record | "raft_initial_snapshot" | Healthy: periodic snapshots; Critical: missing for a long-running cluster means no snapshot, leading to a long, slow log |
| Transaction count | grep `^record ` and count | Warning: more than one million means compact has not run for a long time |
| Term progression | The cluster term is monotonically increasing | Critical: a term that drops indicates corruption |

**Forensic use cases:**

- **Split-brain investigation:** compare `show-log` from two nodes to find the divergence point (the term and index at which they diverge).
- **Lost transaction:** check timestamp gaps (over ten seconds with no record) which can indicate a leader death and a new election.
- **Performance tuning:** if `compact` has not run for thirty days, the file is three to five times larger than the snapshot, so a compact schedule is needed.

## 9.31.6 Guided exercise

**Goal:** bootstrap a three-node `OVN_Southbound` cluster from scratch on a single lab machine (three `ovsdb-server` processes plus three separate files).

**Setup:**

```bash
# 1. Create three directories for three mock nodes
mkdir -p /tmp/sb-{node1,node2,node3}

# 2. Pretend to have the schema
SCHEMA=/usr/share/ovn/ovn-sb.ovsschema

# 3. Bootstrap node 1
ovsdb-tool create-cluster /tmp/sb-node1/db \
  $SCHEMA \
  tcp:127.0.0.1:6644

# Note the cluster UUID
CID=$(ovsdb-tool db-cid /tmp/sb-node1/db)
echo "Cluster ID: $CID"

# 4. Start node 1 ovsdb-server
ovsdb-server /tmp/sb-node1/db \
  --remote=ptcp:6642:127.0.0.1 \
  --pidfile=/tmp/sb-node1/pid \
  --detach
```

**Join nodes 2 and 3:**

```bash
# 5. Node 2 joins
ovsdb-tool join-cluster \
  /tmp/sb-node2/db \
  OVN_Southbound \
  tcp:127.0.0.1:6645 \
  tcp:127.0.0.1:6644

ovsdb-server /tmp/sb-node2/db \
  --remote=ptcp:6643:127.0.0.1 \
  --pidfile=/tmp/sb-node2/pid \
  --detach

# 6. Node 3 joins
ovsdb-tool join-cluster \
  /tmp/sb-node3/db \
  OVN_Southbound \
  tcp:127.0.0.1:6646 \
  tcp:127.0.0.1:6644

ovsdb-server /tmp/sb-node3/db \
  --remote=ptcp:6647:127.0.0.1 \
  --pidfile=/tmp/sb-node3/pid \
  --detach
```

**Verify the cluster:**

```bash
# 7. Check the cluster status
ovs-appctl -t /tmp/sb-node1/ctl cluster/status OVN_Southbound

# Expected: 3 servers active, 1 leader plus 2 followers

# 8. Test the integrity check
ovsdb-tool check-cluster /tmp/sb-node1/db /tmp/sb-node2/db /tmp/sb-node3/db
# Expected: cluster is consistent
```

**Simulate failure and recovery:**

```bash
# 9. Kill node 1 (the leader)
kill $(cat /tmp/sb-node1/pid)

# 10. Verify a re-election on node 2 or node 3
ovs-appctl -t /tmp/sb-node2/ctl cluster/status OVN_Southbound
# Expected: one of the two remaining nodes becomes the leader

# 11. Restart node 1 (it rejoins automatically)
ovsdb-server /tmp/sb-node1/db \
  --remote=ptcp:6642:127.0.0.1 \
  --pidfile=/tmp/sb-node1/pid \
  --detach
```

## 9.31.7 Anti-patterns and notes

**Anti-pattern 1: running `compact` on a database that is being served.**

Verbatim man-page warning: *"This command does not work if db is currently being served by ovsdb-server"*. Violating it causes a file-lock conflict and corruption at the transaction-log level.

**Prevention:** stop `ovsdb-server` first, run compact, then start it again. Or use the online tool: `ovs-appctl -t ovsdb-server ovsdb-server/compact` (the server handles it without a stop).

**Anti-pattern 2: `transact` against a file that the server is writing to.**

Same reason: a race condition. Use `ovsdb-client transact <socket>` against a live database instead of `ovsdb-tool transact <file>`.

**Anti-pattern 3: `convert` to the wrong schema version.**

Converting to a wrong version (for example schema 2.5 to 2.0, which is backward incompatible) will corrupt the data. Always:

1. Back up the file first (`cp db db.bak`).
2. Run `needs-conversion` first to verify the direction.
3. Test the convert in a test environment before production.

**Note 4: `show-log` output is very long for a production database.**

A cluster SB DB after six months can have ten million or more transactions. `show-log` has no pagination, so use:

```bash
ovsdb-tool show-log /etc/ovn/ovnsb_db.db 2>&1 | less
# Or grep by timestamp
ovsdb-tool show-log /etc/ovn/ovnsb_db.db 2>&1 | grep "2026-04-25"
```

**Note 5: `check-cluster` needs access to every file.**

`check-cluster DB1 DB2 DB3` cross-verifies state. The operator must copy files from the three nodes to one place (after stopping the server) and run the check there. Cross-node verification over the network is not supported (no network protocol exists for it).

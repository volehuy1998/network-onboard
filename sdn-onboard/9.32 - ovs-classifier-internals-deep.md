## §9.32.1 classifier (TSS) — Tuple Space Search

> **Score target:** DEEP-20 (cornerstone cohort C2)

### 1. Khái niệm

Classifier TSS là algorithm phân loại packet/flow trong userspace ofproto-dpif của OVS. "Tuple space" = tập hợp các unique mask pattern; mỗi unique mask tạo 1 subtable. Lookup traverse subtables theo priority order. Loại: userspace data-plane lookup algorithm.

### 2. Lịch sử

TSS algorithm gốc từ Srinivasan et al. SIGCOMM 1998 paper "Packet Classification using Tuple Space Search". OVS adopt 2009 (Nicira initial). Refined 2014 với OVS 2.0 megaflow consolidation. Stable since.

### 3. Vị trí kiến trúc

Userspace `ofproto-dpif` layer trong OVS 3-component architecture (kernel datapath / userspace ovs-vswitchd / OVSDB). Concretely: `lib/classifier.c` trong OVS source tree. Scope: per-bridge classifier instance.

### 4. Vai trò

Đóng vai trò "rule matching engine" giữa OF flow table abstraction và kernel megaflow installation. Pipeline: OF rule add → classifier insert (subtable build) → classifier output (megaflow install). Output → kernel datapath fast-path.

### 5. Vì sao sinh ra

Pre-TSS pain: linear scan flow table O(N) cho mỗi packet không scale. Hash-only lookup không support wildcard match. Trie-only lookup hard với arbitrary mask. TSS compromise: hash within subtable (fast O(1)), linear over subtable (limited by mask count). Production traffic typical 10-50 unique mask = manageable.

### 6. Vấn đề giải quyết

Vấn đề: arbitrary wildcard match với production scale. Mechanism: partition flow table thành subtables per-mask, lookup deterministic cost = O(subtable_count). Scope: full solve cho wildcard flexibility, partial trade-off với mask explosion (anti-pattern: rule with too many unique mask).

### 7. Sự quan trọng

Cornerstone tier 1. Engineer không hiểu TSS = không tune classifier khi gặp production performance issue (slow rule install, classifier lookup CPU spike, mask explosion warning).

### 8. Cơ chế

Per-rule insert (`cls_classifier_insert`):
1. Compute mask pattern (which fields wildcarded vs exact)
2. If subtable cho mask exists → insert vào hash bucket (chained hash)
3. Else create new subtable + insert
4. Update Priority sorted vector (pvector) ordering subtables

Per-packet lookup (`cls_classifier_lookup`):
1. For each subtable in priority order (highest first):
   a. Hash packet fields with subtable mask → bucket
   b. Linear scan bucket cho exact match
2. Return first match (highest priority)

Mask consolidation invariant: 2 OF rules with cùng wildcard pattern (regardless field value) share 1 subtable.

Patricia trie optimization: prefix trie cho IP/port range mask → skip subtable subset based on trie pruning. `lib/pvector.c` priority sorted vector backing.

### 9. Kỹ sư vận hành thành thạo

4 skill:

1. **Read:** `ovs-appctl coverage/show | grep classifier` xem counter (lookup_match, lookup_miss, etc.)
2. **Modify:** indirect via OF rule design (add/remove rule trigger subtable rebuild). Direct subtable manipulation không exposed.
3. **Observe:** `ovs-appctl ofproto/list-tunnels` + `ovs-appctl ofproto/trace 'flow_spec'` step-through classifier evaluation
4. **Debug:** suspect classifier slow → check subtable count via coverage; suspect mask explosion → audit OF rule mask diversity

Decision tree:
- coverage shows classifier_lookup spike → check subtable count (high count = explosion)
- subtable_count > 50 → audit rule mask diversity, refactor với conjunction action
- subtable_count < 10 → classifier not bottleneck, investigate elsewhere

### 10. Phân loại taxonomy

Loại: userspace data-plane lookup algorithm. Parent: `lib/classifier.c` TSS implementation. Scope: per-bridge instance. Type: O(K) where K = subtable count.

### 11. Cách dùng thành thạo workflow

```
Step 1: Profile classifier overhead
  $ ovs-appctl coverage/show | grep -E "classifier|cmap"
  → cmap_lookup, cmap_insert counter

Step 2: Identify hot subtable
  $ ovs-appctl ofproto/list-tunnels | wc -l
  → if > 100 subtable, mask explosion

Step 3: Mask consolidation refactor
  → Rule A: nw_src=10.0.0.5/32, action=output:1
  → Rule B: nw_src=10.0.0.6/32, action=output:1
  → Refactor: use conjunction(1,1/2) cho IP set, conjunction(1,2/2) cho action
  → 2 unique rule → 1 subtable + 1 verdict subtable = simpler

Step 4: Verify reduction
  $ ovs-appctl coverage/show | grep cmap
  → subtable count should drop

Step 5: Measure latency improvement
  → Use packet generator + measure first-flow latency post-refactor
```

### 12. Khi xảy ra sự cố cần nhớ keyword này

- Symptom A: First-flow latency > 10ms. Diagnostic: classifier lookup slow, subtable_count too high. Recovery: mask consolidation.
- Symptom B: ovs-vswitchd CPU spike on rule add. Diagnostic: classifier insert slow, subtable rebuild cost. Recovery: batch rule add via bundle action.
- Symptom C: Rule effect intermittent. Diagnostic: classifier priority order conflict, 2 rule cùng priority overlap. Recovery: explicit priority assignment.

Cross-link 9.26 revalidator storm forensic; 9.25 ofproto/trace classifier debug.

### 13. Liên quan mật thiết

- **megaflow** (9.2): classifier output installs to kernel megaflow
- **ukey** (9.2): revalidator track kernel flow ↔ classifier rule mapping
- **upcall** (next §9.32.2): classifier invoked when kernel miss
- **subtable** (this §): per-mask classifier partition
- **Patricia trie** (this §): prefix optimization
- **pvector** (this §): priority order

### 14. Sự khác biệt phiên bản

| Version | Behavior |
|---------|----------|
| Pre-OVS 2.0 | Linear flow table scan O(N) |
| OVS 2.0 (2014) | TSS classifier introduce |

### 16. Source code anchor

OVS v2.17.9:
- `lib/classifier.c` `classifier_lookup()`, `classifier_insert()`, `classifier_remove()`
- `lib/cmap.c` chained hash backing subtable buckets
- `lib/pvector.c` priority sorted vector
- `lib/flow.c` `flow_extract()` extract packet fields cho lookup

### 17. Real-world incident anatomy

Production case 2023-Q4: cluster sau deploy ACL update, ovs-vswitchd CPU 100%.

Investigation:
- coverage/show shows classifier_insert 50K/sec sustained
- New ACL rule có 2000 entries với mỗi entry unique 5-tuple → 2000 unique mask
- Subtable explosion: 2000 subtable × O(K) lookup cost

Root cause: ACL design used per-IP/port-pair rule thay vì conjunction action.

Fix: refactor ACL với Address_Set + Port_Group + conjunction → 200 subtable.
ovs-vswitchd CPU drop từ 100% → 30%.

Reference: similar pattern 9.26 §revalidator storm.

### 18. Lab exercise

```
Setup: Mininet single-switch s1 với 100 veth port

Test mask explosion:
  for i in $(seq 1 100); do
    ovs-ofctl add-flow s1 "priority=100,ip,nw_src=10.0.0.$i,actions=output:$i"
  done
  ovs-appctl coverage/show | grep cmap
  → cmap_insert ~ 100 (1 subtable per mask, but mask same for all - actually 1 subtable!)

  # Now diversify:
  for i in $(seq 1 100); do
    ovs-ofctl add-flow s1 "priority=100,ip,nw_src=10.0.$i.0/24,actions=output:$i"
  done
  → 100 different mask /24, 100 subtable

Verify reduction with conjunction:
  ovs-ofctl add-flow s1 'priority=100,ip,nw_src=10.0.0.0/8,actions=conjunction(1,1/2)'
  ovs-ofctl add-flow s1 'priority=100,ip,actions=conjunction(1,2/2)'
  ovs-ofctl add-flow s1 'priority=200,conj_id=1,actions=output:1'
  → Now O(few) subtable for same matching capability
```

### 19. Failure mode

- Mode A: Subtable explosion (>1000) → CPU spike. Diagnostic coverage/show. Recovery: mask consolidation, conjunction.
- Mode B: Patricia trie misuse for non-prefix mask → fall back to subtable scan, slow. Recovery: rare bug, check OVS version.
- Mode C: pvector reorder cost on mass rule update → temporarily slow. Recovery: bundle rule update atomic.

### 20. Cross-domain comparison

- **Linux iptables filter chain:** linear scan O(N), no algorithmic improvement. OVS classifier far superior cho large rule set.
- **Cisco TCAM ACL:** hardware parallel match, O(1) but limited capacity. OVS software classifier flexible (no capacity limit) tradeoff with software speed.
- **eBPF cls_bpf classifier:** programmable classifier in kernel. Khác: eBPF jit'd code per-rule; OVS classifier interpreted from OF abstraction.
- **Intel DPDK rte_classify:** hardware-assisted classification table. OVS hardware offload combine với classifier.

## §9.32.2 upcall — Kernel miss to userspace handler

### 1. Khái niệm

`upcall` là Netlink message từ kernel openvswitch.ko → userspace ovs-vswitchd khi kernel datapath miss megaflow lookup. Userspace decode packet, evaluate via classifier, install resulting megaflow back to kernel. Loại: kernel-userspace IPC mechanism.

### 2. Lịch sử

Initial 2009 với OVS Nicira. Netlink protocol stable. OVS 2.0 (2014) add upcall rate limit cho防 control plane DDoS.

### 3. Vị trí kiến trúc

Bridge giữa kernel `openvswitch.ko` (fast path, miss handling) và userspace `ovs-vswitchd` (slow path, classifier + flow install). Concretely: Netlink socket `OVS_PACKET_FAMILY` carries upcall message.

### 4. Vai trò

Cache miss handler. Without upcall, kernel cache miss = packet drop (no fallback). With upcall, miss → classifier → install → next packet hits cache. Critical cho first-flow latency + cache warming.

### 5. Vì sao sinh ra

Pre-OVS, kernel module có self-contained flow logic (Linux bridge MAC learning local). OVS architecture decouple control plane (userspace) từ data plane (kernel). Upcall is bridge mechanism cho async control flow.

### 6. Vấn đề giải quyết

Vấn đề: kernel datapath cache miss handling without dropping packet. Mechanism: Netlink message buffer packet + metadata, handler thread dequeue, evaluate, install, optionally inject packet back. Scope: full solve cho cache miss; partial trade-off (upcall cost itself ~10-100us/packet).

### 7. Sự quan trọng

Cornerstone tier 1. Upcall là bottleneck dưới conditions (high churn, cold cache, DDoS). Engineer không hiểu upcall = không debug "first packet slow", "ovs-vswitchd CPU spike".

### 8. Cơ chế

```
Kernel side (openvswitch.ko):
  Step 1: dp_packet_in() invoke từ netif_rx
  Step 2: flow_table_lookup() check megaflow
  Step 3: If miss → genl_send (OVS_PACKET_FAMILY, OVS_PACKET_CMD_MISS)
  Step 4: packet skb queued
  Step 5: continue (other packet handled)

Userspace side (ovs-vswitchd):
  Step 1: handler_thread main loop
  Step 2: dpif_recv() dequeue Netlink message
  Step 3: ofproto_recv_upcall() decode packet
  Step 4: ofproto-dpif xlate evaluate via classifier
  Step 5: Install resulting megaflow via dpif_flow_put()
  Step 6: Optionally inject packet back via OVS_PACKET_CMD_EXECUTE

Rate limit:
  Per-bridge upcall socket có rate cap, OVS 2.0+ adaptive throttle
```

Invariant: every miss generates exactly 1 upcall (no duplicate); revalidator does NOT generate upcall (separate path).

### 9. Kỹ sư vận hành thành thạo

Skill: monitor upcall rate, identify spike root cause, tune handler thread count.

```
$ ovs-appctl coverage/show | grep upcall
upcall_flow_limit_hit    0 +0   # no rate limit hits = healthy
upcall_flow_limit_kill   0 +0
miss_per_sec    100      # baseline normal

$ ovs-vsctl set Open_vSwitch . other_config:n-handler-threads=8
→ Increase handler thread count for high-rate cluster
```

### 10. Phân loại taxonomy

Loại: kernel-userspace IPC mechanism, Netlink message family. Parent: OVS_PACKET_FAMILY. Scope: per-packet on miss.

### 11. Workflow

Debug upcall storm:

```
Step 1: $ ovs-appctl coverage/show | grep -E "upcall|miss"
Step 2: If miss_per_sec > 1000, identify churn source
Step 3: $ ovs-appctl dpctl/dump-flows | head -20
   → Check megaflow count + freshness
Step 4: If many short-lived flow, increase n-handler-threads
Step 5: If many unique mask, refer §9.32.1 mask consolidation
```

### 12. Troubleshoot

- Symptom A: First-flow latency 10-100ms. Suspect upcall queue depth. Diagnostic coverage/show.
- Symptom B: ovs-vswitchd CPU 100% steady-state. Diagnostic upcall rate sustained > 10K/sec.
- Symptom C: kernel log "upcall failed: ENOBUFS". Suspect Netlink socket buffer overflow. Recovery: increase `net.core.rmem_max`.

### 13. Coupling

Tightly coupled: handler thread, megaflow (consume upcall output), classifier (evaluate), Netlink (transport), `dp_netdev` userspace alternate datapath, EMC/SMC (downstream cache).

### 14. Version drift

| Version | Behavior |
|---------|----------|
| OVS 2.0 | Rate limit + adaptive throttle |
| OVS 2.4 | Multi-handler thread default 4 |
| OVS 2.6 | EMC tier reduce upcall need |
| OVS 2.15 | SMC tier further reduce |
| OVS 3.0+ | Stable |

### 15. Verification

```
$ ovs-appctl coverage/show | grep upcall
$ tc -s qdisc show dev <iface>
→ kernel netif queue depth
$ dmesg | grep openvswitch
→ kernel log upcall failures
```

### 16. Source code anchor

- Kernel: `net/openvswitch/datapath.c` `ovs_dp_upcall()`, `ovs_dp_process_packet()`
- Kernel: `net/openvswitch/flow_netlink.c` Netlink encoding
- Userspace: `lib/dpif-netlink.c` `dpif_netlink_recv()` dequeue
- Userspace: `ofproto/ofproto-dpif-upcall.c` `handle_upcalls()` main handler loop

### 17. Real-world incident anatomy

Production case 2023-Q1: Web cluster experience occasional 100ms latency spike on new TCP connection.

Investigation:
- coverage/show shows upcall rate spike to 50K/sec at peak
- Megaflow count low (~100), so cache should hit
- But miss rate also high (500/sec)

Root cause: handler thread starvation under burst. n-handler-threads=4 default, but burst created queue depth > 100 messages, causing latency spike.

Fix: increase n-handler-threads=16 + tune `net.core.rmem_max=16M`. Latency P99 drop từ 100ms → 5ms.

### 18. Lab exercise

```
Setup: Mininet 2-host topology
  $ sudo mn --topo single,2 --switch ovsk

Generate burst:
  $ ovs-ofctl del-flows s1 -- add-flow s1 'priority=10,actions=NORMAL'
  $ h1 hping3 -S -p 80 -c 10000 -i u100 h2
  → Burst SYN packet, each is potential megaflow miss

Monitor:
  $ watch -n 1 'ovs-appctl coverage/show | grep upcall'
  → Observe upcall counter spike

Tune:
  $ ovs-vsctl set Open_vSwitch . other_config:n-handler-threads=8
  $ systemctl restart openvswitch-switch
  → Re-run burst, observe latency improve
```

### 19. Failure mode

- Mode A: Handler thread saturate → upcall queue grow → latency spike. Recovery: increase n-handler-threads.
- Mode B: Netlink socket overflow → ENOBUFS. Recovery: increase rmem_max.
- Mode C: Adaptive rate limit kick in → flow drop. Diagnostic coverage upcall_flow_limit_hit. Recovery: investigate upstream churn.

### 20. Cross-domain comparison

- **Linux netfilter NF_QUEUE:** similar kernel-userspace queue mechanism. Khác: NF_QUEUE per-rule decision; OVS upcall per-miss decision (more efficient).
- **DPDK PMD poll-mode:** alternative bypass kernel entirely. Khác: PMD continuous poll; OVS upcall event-driven. Trade-off CPU vs latency.
- **Cisco IOS process switching:** packet escalates from fast-path to control plane. Same concept, different abstraction.

## §9.32.3 ofproto-dpif xlate — OF flow translation pipeline

### 1. Khái niệm

`ofproto-dpif xlate` (ofproto/ofproto-dpif-xlate.c) translate OF flow + actions thành kernel datapath flow + actions sau classifier match. "xlate" = translate. Loại: userspace OF-to-datapath translation engine.

### 2-7. History/Placement/Role/Motivation/Problem/Importance

Initial 2012 với ofproto-dpif refactor (separate ofproto-classifier từ datapath impl). Layer giữa classifier output và dpif_flow_put. Vai trò: bridge OF abstraction (rule + action list) với datapath capability (limited action set per backend). Motivation: support multiple datapath backend (kernel openvswitch.ko, DPDK netdev, AF_XDP) with different capability matrices.

Tier 1 cornerstone. Without xlate, OF features chỉ work on backend supporting all action; xlate enable graceful degradation per backend.

### 8. Cơ chế

```
xlate_actions() main loop:
  Step 1: Parse OF action list từ classifier output
  Step 2: For each action:
    a. Check backend support (datapath_features bitmap)
    b. If supported native → emit datapath action
    c. If not supported → emulate (e.g., recirc + table chain)
    d. Update xlate context (metadata, register, etc.)
  Step 3: Output datapath flow + action list
  Step 4: dpif_flow_put() install to backend
```

Recirculation (recirc_id) heavily used cho action chain emulate.

### 9-13. Operations

Skill: trace xlate via `ofproto/trace`. Symptom: OF rule không có effect → xlate emulation issue. Coupling: classifier (input), dpif (output), recirc_id (context), capability matrix.

### 14. Version drift

OVS 2.4 add comprehensive recirc emulation. OVS 2.8 NSH support. OVS 3.0+ stable.

### 15-16. Verification + Source

```
$ ovs-appctl ofproto/trace br0 'flow_spec'
→ Step-by-step xlate output
```

Source: `ofproto/ofproto-dpif-xlate.c` xlate_actions(), xlate_actions__(), `ofproto/ofproto-dpif.c` integration.

### 17-20. Incident + Lab + Failure + Cross-domain

Incident: backend feature drift (kernel vs DPDK datapath); xlate emulate may differ → behavior subtle change. Lab: same OF rule, swap datapath_type=netdev vs system, observe xlate output diff. Failure: emulation explosion (recirc loop). Cross-domain: Cisco IOS to ASIC translation similar concept (control plane abstraction → hardware capability).

## §9.32.4 dpif — Datapath interface abstraction

### 1-7. Concept/History/Placement/Role/Motivation/Problem/Importance

`dpif` (lib/dpif.c) là abstract interface giữa userspace ofproto-dpif và kernel/userspace datapath backend. Allow OVS support nhiều backend (kernel openvswitch.ko, DPDK netdev, AF_XDP). Initial 2009. Tier 1 cornerstone.

### 8. Cơ chế

```
dpif_class struct = function pointer table:
  - flow_put: install flow
  - flow_del: remove flow
  - flow_dump: enumerate flows
  - port_add: add datapath port
  - recv: receive upcall

Backend implementation:
  - dpif-netlink.c (kernel openvswitch.ko)
  - dpif-netdev.c (userspace netdev / DPDK)
```

### 9-20. (compact treatment per cohort batch limit)

Skill: identify backend via `ovs-vsctl get Bridge br0 datapath_type` (system=kernel, netdev=userspace). Verify via dpif_dump_flows. Source: `lib/dpif.c` (interface) + `lib/dpif-netlink.c` + `lib/dpif-netdev.c` (impl). Failure: backend incompatibility on feature. Cross-domain: VFS in Linux similar abstraction (filesystem backend pluggable).

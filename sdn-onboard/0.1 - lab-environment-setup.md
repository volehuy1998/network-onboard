# 0.1. Lab environment setup

> **Language status:** English (full migration complete, plan v3.12 R1.A run 1, 2026-04-29).

Lab environment: Ubuntu 22.04 LTS (Jammy). Primary references: [Installing Open vSwitch, upstream documentation](https://docs.openvswitch.org/en/latest/intro/install/), [OVN installation guide](https://docs.ovn.org/en/latest/intro/install/).

---

## Learning objectives

After completing this part, you will be able to:

1. **Install (Apply)** the two lab configurations that match the two stages of the series: a single-node all-in-one for blocks I to XI, and a two-node chassis pair for blocks XII, XIII, and beyond.
2. **Verify (Apply)** an OVS 2.17.9 and OVN 22.03.8 installation on Ubuntu 22.04 Jammy with three levels of check: package version, systemd service state, and runtime health through `ovs-vsctl show` and `ovn-sbctl show`.
3. **Distinguish (Analyze)** the role of the kernel module `openvswitch.ko`, the userspace daemon `ovs-vswitchd`, and the database server `ovsdb-server`, and identify when each component is involved in the packet path.
4. **Prepare (Apply)** a health check playbook and a teardown procedure to reuse for every lab that follows, so that you do not have to rebuild the environment from scratch every time you hit a problem.

---

## Prerequisites

No part of the series depends directly on 0.1 to understand the concepts. This part is purely procedural. That said, before you follow it you should have:

- Root or sudo access on an Ubuntu 22.04 host (a VM or bare metal).
- Internet connectivity to pull packages from the Ubuntu archive (`archive.ubuntu.com`).
- Basic knowledge of systemd, apt, and journalctl (RHCSA level, see `linux-onboard/`).

## 0.1.1 The two lab configurations and their resource requirements

The series has about 16 foundation parts and 3 advanced OVN parts, with lab complexity rising as you go, but you do not need a single lab for everything. The two configurations below cover all needs, and each one runs on mid-range hardware.

### Lab mode A, single-node all-in-one (blocks 0 to XI)

Goal: practice OVS, OpenFlow, OVSDB, and Mininet topologies. Everything runs on one machine.

Minimum requirements: 2 vCPU, 4 GB RAM, 20 GB disk. In practice, after installing OVS, Mininet, and the Ryu controller, the idle memory footprint is about 300 MB, which leaves plenty of room to run 30 to 50 Mininet virtual hosts at the same time.

Recommended platform: Multipass (`sudo snap install multipass`) or VirtualBox on a laptop. Multipass uses a QEMU and KVM backend on Linux and HyperKit on macOS. It boots an Ubuntu 22.04 VM in about 30 seconds.

### Lab mode B, two-node chassis pair (blocks XII to XIII)

Goal: practice OVN with two chassis to observe Geneve tunnel formation, HA chassis group failover, and multichassis binding. Two chassis is the minimum that makes the OVN NB and SB architecture meaningful. One chassis is not enough to show the distributed nature of OVN.

Minimum requirements: 2 VMs, each with 2 vCPU, 2 GB RAM, and 10 GB disk. VM1 runs `ovn-central` (NB and SB databases plus northd) together with `ovn-host` (chassis role). VM2 runs only `ovn-host`. The two VMs must share the same bridge network so that Geneve traffic (UDP 6081) can flow between them.

Networking note: if you use VirtualBox, choose adapter mode "Internal Network" for the interface between the two VMs to avoid noise from the host NAT. On Multipass, run `multipass networks` and pass `--network` to both instances so that they share the same broadcast domain.

## 0.1.2 The Ubuntu 22.04 baseline and the kernel module

The series chooses Ubuntu 22.04 LTS (Jammy) as the baseline for three technical reasons. First, Ubuntu 22.04 ships kernel 5.15 LTS by default, and that kernel supports every OVS 2.17 and OVN 22.03 feature without a separate datapath module (the datapath has been in-tree since kernel 3.3). Second, Ubuntu 22.04 has OVS 2.17.9 and OVN 22.03.8 in the `jammy-updates` pocket (as of 2025-08), which means a plain `apt install` is enough. You do not need a PPA or the Ubuntu Cloud Archive. Third, Ubuntu 22.04 has long-term support (LTS until 2027) and is a common baseline for both training and production.

Verify the baseline:

```
[host]$ lsb_release -a
Distributor ID: Ubuntu
Description:    Ubuntu 22.04.4 LTS
Release:        22.04
Codename:       jammy

[host]$ uname -r
5.15.0-101-generic
```

Kernel 5.15 is the LTS branch for the Ubuntu 22.04 GA release. If you use the HWE kernel (5.17, 6.2, or 6.5 depending on the Ubuntu 22.04 point release), OVS still works, but you should check that the kernel module matches the userspace version.

The kernel module `openvswitch` must load before the daemon starts:

```
[host]$ sudo modprobe openvswitch
[host]$ lsmod | grep openvswitch
openvswitch           176128  0
nsh                    16384  1 openvswitch
nf_conncount           24576  1 openvswitch
nf_nat                 57344  1 openvswitch
nf_conntrack          172032  4 nf_nat,openvswitch,nf_conncount,nf_conntrack_netlink
```

The module loads automatically when `openvswitch-switch.service` starts, so the manual `modprobe` step is only needed during debugging. If `lsmod` does not show `openvswitch`, check whether the module exists in the current kernel package: `ls /lib/modules/$(uname -r)/kernel/net/openvswitch/`.

## 0.1.3 Installing OVS and OVN through apt

Ubuntu 22.04 jammy-updates provides the main packages: `openvswitch-switch` (daemon and kernel module loader), `openvswitch-common` (the CLI tools `ovs-vsctl`, `ovs-ofctl`, `ovs-appctl`), `ovn-central` (NB and SB databases plus the northd daemon), `ovn-host` (the `ovn-controller` running on each chassis), and `ovn-common` (the CLI tools `ovn-nbctl`, `ovn-sbctl`, `ovn-trace`).

Install:

```
[host]# apt update
[host]# apt install -y openvswitch-switch ovn-central ovn-host
```

Package dependencies pull in `openvswitch-common`, `ovn-common`, and the required Python bindings automatically.

Verify the version:

```
[host]$ dpkg -l | grep -E 'openvswitch|ovn' | awk '{print $2, $3}'
ovn-central 22.03.8-0ubuntu0.22.04.1
ovn-common 22.03.8-0ubuntu0.22.04.1
ovn-host 22.03.8-0ubuntu0.22.04.1
openvswitch-common 2.17.9-0ubuntu0.22.04.1
openvswitch-switch 2.17.9-0ubuntu0.22.04.1
```

These versions are the baseline for the series. Every Guided Exercise and Lab references them. If the version shown is older (for example, 2.17.7), run `apt upgrade openvswitch-switch ovn-central ovn-host` to move to the latest version in the jammy-updates pocket.

> **Version note:** Ubuntu 20.04 (Focal) only has OVS 2.13 in the main repository and does not have `ovn-central`. You have to enable the Ubuntu Cloud Archive (`add-apt-repository cloud-archive:yoga`) to get OVN. Ubuntu 24.04 (Noble) has OVS 3.3 and OVN 24.03 with the RARP-based `activation-strategy` feature that part 19 analyzes. The series uses Ubuntu 22.04 as the baseline because it is the longest-supported version that has both OVS 2.17 and OVN 22.03 without an extra repository.

After installation, the systemd services are enabled and started automatically:

```
[host]$ systemctl is-active openvswitch-switch ovn-central ovn-controller
active
active
active
```

If any line shows `inactive`, run `systemctl start <service>` and then check `journalctl -u <service> -n 50` for the cause.

## 0.1.4 Mininet for OpenFlow labs (blocks III and IV)

Mininet creates a virtual network of hosts, switches, and links using Linux network namespaces and veth pairs. The series uses Mininet in blocks III and IV to observe OpenFlow messages between the controller and the switch without needing real switch hardware.

Mininet 2.3.0 is the first official release that supports Python 3. Ubuntu 22.04 ships Python 3.10 by default, so the two are compatible. However, Ubuntu 22.04 does not ship Mininet in the main repository, so you must install it from source or from a third-party PPA.

Install from source (recommended):

```
[host]$ git clone https://github.com/mininet/mininet.git
[host]$ cd mininet
[host]$ git checkout 2.3.0
[host]$ sudo PYTHON=python3 util/install.sh -n
```

The `-n` flag installs Mininet core (the `mn` wrapper, the Python library, and OVS) but does not pull in the POX controller or the OpenFlow reference implementation. If you need the Ryu controller for block IV, install it separately: `pip install ryu`.

Verify Mininet:

```
[host]$ sudo mn --version
2.3.0

[host]$ sudo mn --test pingall
*** Creating network
*** Adding controller
*** Adding hosts:
h1 h2
*** Adding switches:
s1
*** Adding links:
(h1, s1) (h2, s1)
*** Configuring hosts
h1 h2
*** Starting controller
c0
*** Starting 1 switches
s1 ...
*** Ping: testing ping reachability
h1 -> h2
h2 -> h1
*** Results: 0% dropped (2/2 received)
```

The `0% dropped` line confirms that Mininet, OVS, and the default controller work end to end. If you see `100% dropped`, check that the bridge `s1` was created: `sudo ovs-vsctl show`.

## 0.1.5 The health check playbook (use it for every lab)

After every fresh lab build or after a reboot, run this playbook to confirm that all components are ready before you start the exercise. This avoids losing time to debugging halfway through a lab when a service was already dead.

### Check OVS (needed for both lab mode A and lab mode B)

```
[chassis]$ systemctl is-active openvswitch-switch
active

[chassis]$ ovs-vsctl show
92a54bc6-6e75-4f02-9b4c-95c01dbd5f03
    Bridge br-int
        fail_mode: secure
        Port br-int
            Interface br-int
                type: internal
    ovs_version: "2.17.9"

[chassis]$ ovs-appctl dpif/show
system@ovs-system: hit:0 missed:0
  br-int:
    br-int 65534/3: (internal)
```

The line `ovs_version: "2.17.9"` confirms that the userspace version matches the package. This is the only check that guarantees there is no mismatch between the userspace daemon and the kernel module.

### Check OVN (only for lab mode B)

```
[central]$ ovn-nbctl show
[central]$ ovn-sbctl show
Chassis "chassis-vm1"
    hostname: chassis-vm1
    Encap geneve
        ip: "10.0.0.11"
        options: {csum="true"}
Chassis "chassis-vm2"
    hostname: chassis-vm2
    Encap geneve
        ip: "10.0.0.12"
        options: {csum="true"}
```

When `ovn-sbctl show` lists every chassis you expect (2 for lab mode B), it confirms that the Geneve encap entries are registered. If a chassis is missing, check `ovn-controller` on that chassis: `journalctl -u ovn-controller -n 50`.

### Check that the Geneve tunnel formed (lab mode B)

```
[chassis-vm1]$ ovs-appctl ofproto/list-tunnels
port  type  local_ip  remote_ip  local_port  remote_port
4     geneve  10.0.0.11  10.0.0.12  6081  6081
```

This line appears only after chassis-vm2 has registered into the SB DB. If `ofproto/list-tunnels` is empty, `ovn-controller` does not yet have the information it needs about the peer chassis.

## 0.1.6 Teardown and reset procedure

End every lab with a clean state so that the previous lab does not influence the next one.

### Reset OVS to an empty state

```
[chassis]# systemctl stop openvswitch-switch
[chassis]# rm /etc/openvswitch/conf.db
[chassis]# systemctl start openvswitch-switch
[chassis]$ ovs-vsctl show
<UUID>
    ovs_version: "2.17.9"
```

Removing `conf.db` and restarting takes OVS back to a freshly installed state. If you do not delete it, the bridges and flow rules from the previous lab remain.

### Reset OVN to an empty state

```
[central]# systemctl stop ovn-central
[central]# rm /var/lib/ovn/ovnnb_db.db /var/lib/ovn/ovnsb_db.db
[central]# systemctl start ovn-central
```

Removing the NB and SB databases clears the logical topology that you had defined. The chassis register themselves again when `ovn-controller` reconnects.

### Reset Mininet (always run after every lab in blocks III and IV)

```
[host]# mn -c
*** Removing excess controllers/ofprotocols/ofdatapaths/pings/noxes
*** Removing junk from /tmp
*** Removing old X11 tunnels
*** Removing excess kernel datapaths
*** Removing OVS datapaths
*** Removing all links of the pattern foo-ethX
*** Cleanup complete.
```

The `mn -c` command cleans every namespace, veth, and OVS bridge that Mininet created. If you skip this, the next `mn` run fails because the old namespace still owns the names.

### ▶ Guided Exercise 1: Verify the OVS 2.17 and OVN 22.03 baseline on Ubuntu 22.04

In this exercise you confirm that the entire baseline version of the series is correctly installed by running four independent checks: package, kernel module, service, and runtime CLI.

**Outcomes:**
- Confirm that `openvswitch-switch` and `ovn-central` or `ovn-host` are installed at the correct version from the jammy-updates pocket.
- Distinguish between a silent failure and an obvious failure caused by a version mismatch between the kernel module and userspace.

**Before you begin:**
Complete §0.1.2 (kernel module) and §0.1.3 (apt install). Lab mode A (single-node) is enough.

**Instructions:**

**1.** Check the installed package version:

 [host]$ dpkg -l | grep -E '^ii (openvswitch|ovn)' | awk '{print $2, $3}'

Expected output:

 openvswitch-common 2.17.9-0ubuntu0.22.04.1
 openvswitch-switch 2.17.9-0ubuntu0.22.04.1
 ovn-central 22.03.8-0ubuntu0.22.04.1
 ovn-common 22.03.8-0ubuntu0.22.04.1
 ovn-host 22.03.8-0ubuntu0.22.04.1

These five packages are enough for both lab mode A and lab mode B. If the version is older (for example, `2.17.7` instead of `2.17.9`), run `apt upgrade` to move to the latest version in the jammy-updates pocket.

**2.** Check that the kernel module is loaded:

 [host]$ lsmod | grep openvswitch
 openvswitch 176128 0

The last column `0` is the reference count. There is no bridge using the module yet, so the count is 0. After `ovs-vsctl add-br br-test`, the count rises to 1.

**3.** Check the systemd services:

 [host]$ systemctl is-active openvswitch-switch ovn-central ovn-controller
 active
 active
 active

All three services are `active`. If `ovn-controller` is `inactive`, that is because the `ovn-host` package has not yet been configured with the Southbound URL. Check journalctl.

**4.** Check the runtime CLI from userspace:

 [host]$ ovs-vsctl show
 92a54bc6-6e75-4f02-9b4c-95c01dbd5f03
 ovs_version: "2.17.9"

 [host]$ ovn-nbctl --version
 ovn-nbctl 22.03.8
 Open vSwitch Library 2.17.9
 DB Schema 6.1.0

 [host]$ ovn-sbctl --version
 ovn-sbctl 22.03.8
 Open vSwitch Library 2.17.9
 DB Schema 20.21.0

The line `ovs_version: "2.17.9"` from `ovs-vsctl show` is the userspace daemon reporting its version. The line `Open vSwitch Library 2.17.9` from `ovn-nbctl --version` is the version of libopenvswitch that `ovn-nbctl` is linked against. The two must match. If they drift (for example, OVN linked against libopenvswitch 2.16 while the daemon runs 2.17), you will see hard-to-debug errors when parsing tunnel metadata.

The DB Schema version (`6.1.0` for NB, `20.21.0` for SB) is the OVSDB schema version, not the OVN software version. When you upgrade OVN from 22.03 to 24.03, the schema version goes up and a migration step is needed.

**Finish:**
All four checks pass: the lab environment is ready for every part of the series. Save the output of step 1 and step 4 to a file for later reference. When a part shows unusual output, comparing against this baseline is the first debugging step.

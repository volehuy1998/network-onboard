## 13.16.1 Pipeline overview, why sixty-three stages

OVN compiles the network logical intent (NB DB) into a **multi-stage linear pipeline** stored in the SB DB Logical_Flow table. Each stage is a set of flow rules with a dedicated role: port-security check, ACL evaluation, load balancer DNAT, DHCP options, L2 lookup, and so on. When a packet enters a logical switch, it traverses **twenty-six ingress stages (LS_IN)** in order, then a **decision OUTPUT** (chassis-local or tunnel to a remote chassis), then **ten egress stages (LS_OUT)** before leaving. For a logical router, the packet traverses **twenty ingress stages (LR_IN)** and **seven egress stages (LR_OUT)**.

In total there are **sixty-three stages**, compared to sixty-four mentioned in the REF document. The REF describes OVN 24.03 and later, which has one extra stage. See §13.16.16 on version drift. The number sixty-three is the result of a direct grep on `northd/northd.c` macro `PIPELINE_STAGES` for branch-22.03.

**Why split into many stages?** The answer rests on four design factors:

1. **Modular composition.** Each stage solves exactly one concern. ACL does not need to know about DHCP. NAT does not need to know about L2 lookup. When a new feature is needed (for example, ECMP statefulness), the ovn-northd team adds a new stage and does not modify older stages.
2. **Debug-friendly.** When a packet is dropped, the operator can pinpoint exactly which stage dropped it by tracing through `ovn-trace --detailed` or by dumping the corresponding OF table. A pipeline with fewer stages is harder to debug.
3. **Cross-version migration.** A new stage can be inserted into the pipeline without breaking older stages, as long as register usage is adjusted carefully. From OVN 22.03 to 24.03 the team added several stages for LB affinity and ACL split-eval-action without breaking the older pipeline.
4. **OpenFlow table flat scaling.** The OVS classifier performs best when each table has few distinct flow priorities. Splitting the pipeline into sixty-three stages keeps each OF table at no more than about 1000 distinct priorities, so the classifier preserves O(log n) lookup speed.

The downside of a many-stage design is the **per-packet CPU cost** on the slow path. Each `goto_table` instruction adds lookup overhead. On the fast path (megaflow cache hit), the cost is near zero because OVS aggregates the entire stage chain into a single megaflow entry.

> **Version note:** the number of stages grows over time. OVN 20.06 had about fifty stages, OVN 22.03 has sixty-three, OVN 24.03 has about seventy (after adding LB_AFF, ACL split, and IPSec encap). The curriculum tracks the OVN 22.03.8 baseline. Every version drift carries a callout in §13.16.16 per Rule 14.

## 13.16.2 br-int OpenFlow table block layout

On `br-int`, ovn-controller occupies seventy-three OpenFlow tables (table 0 through table 72). Each table has a fixed role defined in `controller/lflow.h` (OVN branch-22.03):

```c
#define OFTABLE_PHY_TO_LOG            0
#define OFTABLE_LOG_INGRESS_PIPELINE  8 /* First of LOG_PIPELINE_LEN tables. */
#define OFTABLE_REMOTE_OUTPUT        37
#define OFTABLE_LOCAL_OUTPUT         38
#define OFTABLE_CHECK_LOOPBACK       39
#define OFTABLE_LOG_EGRESS_PIPELINE  40 /* First of LOG_PIPELINE_LEN tables. */
#define OFTABLE_SAVE_INPORT          64
#define OFTABLE_LOG_TO_PHY           65
#define OFTABLE_MAC_BINDING          66
#define OFTABLE_MAC_LOOKUP           67
#define OFTABLE_CHK_LB_HAIRPIN       68
#define OFTABLE_CHK_LB_HAIRPIN_REPLY 69
#define OFTABLE_CT_SNAT_HAIRPIN      70
#define OFTABLE_GET_FDB              71
#define OFTABLE_LOOKUP_FDB           72

**Key observation:** LS_IN and LR_IN **share** OF tables 8-33. Both pipelines use table 8 for their own logical table 0 (LS_IN_PORT_SEC_L2 versus LR_IN_ADMISSION). They are distinguished by the `metadata` field (datapath ID) in the match condition. One flow applies only to an LS with `metadata=0x6`, another applies only to an LR with `metadata=0x7`. This is how OVN packs many logical datapaths into a shared OF table without conflicts.

The same sharing applies to LS_OUT and LR_OUT on OF tables 40-49.

## 13.16.3 Mapping formula, logical stage to OF table

From the layout above, the mapping formula is straightforward:

| Pipeline | Formula |
|----------|-----------|
| LS_IN | `OF table = 8 + LS_IN_<stage>` |
| LR_IN | `OF table = 8 + LR_IN_<stage>` |
| LS_OUT | `OF table = 40 + LS_OUT_<stage>` |
| LR_OUT | `OF table = 40 + LR_OUT_<stage>` |

**Practical examples:**

- `LS_IN_PORT_SEC_L2` (logical table 0) maps to OF table 8.
- `LS_IN_LB` (logical table 12) maps to OF table 20.
- `LS_IN_L2_LKUP` (logical table 24) maps to OF table 32.
- `LR_IN_IP_ROUTING` (logical table 11) maps to OF table 19.
- `LR_IN_ARP_RESOLVE` (logical table 15) maps to OF table 23.
- `LS_OUT_ACL` (logical table 4) maps to OF table 44.
- `LR_OUT_SNAT` (logical table 3) maps to OF table 43.
- `LR_OUT_DELIVERY` (logical table 6) maps to OF table 46.

When `ovs-ofctl dump-flows br-int table=20` shows a flow with `metadata=0x6,...,actions=ct(commit, ...)`, the packet is in stage `LS_IN_LB` of the datapath whose ID is 6. Cross-reference with `ovn-sbctl find Datapath_Binding tunnel_key=6` to identify which logical switch.

> **Accuracy warning:** the formula above is correct for OVN 22.03.8. OVN 24.03 and later changed `LOG_PIPELINE_LEN` (extended from twenty-six to thirty), which shifts the egress offset from 40 to 44. Any commit referencing OF table numbers must be verified against `ovn-sbctl --columns=table_id list Logical_Flow` for the corresponding version.

## 13.16.4 LS Ingress pipeline, twenty-six stages in detail (OF table 8-33)

Source: `northd/northd.c` macro `PIPELINE_STAGES` (OVN branch-22.03), specifically the block "Logical switch ingress stages".

| OF table | Logical | Stage name | Main role | Register or REGBIT |
|----------|---------|------------|---------------|-----------------|
| 8 | 0 | `LS_IN_PORT_SEC_L2` | Port-security L2 (verify the source MAC matches `Logical_Switch_Port.port_security`). | Set reg14 to inport tunnel_key. |
| 9 | 1 | `LS_IN_PORT_SEC_IP` | Port-security IP (verify the source IP). | Set reg13 to the LSP ct_zone. |
| 10 | 2 | `LS_IN_PORT_SEC_ND` | Port-security ND (verify the ARP/NDP source). | |
| 11 | 3 | `LS_IN_LOOKUP_FDB` | Look up the MAC in the local FDB. | Set REGBIT_LKUP_FDB on hit. |
| 12 | 4 | `LS_IN_PUT_FDB` | Install a new FDB entry (if LKUP_FDB missed). | |
| 13 | 5 | `LS_IN_PRE_ACL` | Pre-ACL stateful connector (defrag and ct.next). | Set REGBIT_CONNTRACK_DEFRAG. |
| 14 | 6 | `LS_IN_PRE_LB` | Pre-load-balancer (conntrack zone setup for LB). | |
| 15 | 7 | `LS_IN_PRE_STATEFUL` | Stateful pre-processing (emit ct(zone=...)). | Read reg13, emit ct(). |
| 16 | 8 | `LS_IN_ACL_HINT` | Classify the ACL verdict (allow-new / allow / drop / reject). | Set REGBIT_ACL_HINT_*. |
| 17 | 9 | `LS_IN_ACL` | Apply the ACL action (drop, allow, reject). | Read REGBIT_ACL_HINT, emit verdict. |
| 18 | 10 | `LS_IN_QOS_MARK` | Mark the DSCP for QoS. | |
| 19 | 11 | `LS_IN_QOS_METER` | Apply the meter (rate-limit) for QoS. | |
| 20 | 12 | `LS_IN_LB` | Load balancer DNAT. | Set reg1 to the original DIP. |
| 21 | 13 | `LS_IN_PRE_HAIRPIN` | Detect hairpin LB (client and backend on the same LSP). | Set REGBIT_HAIRPIN. |
| 22 | 14 | `LS_IN_NAT_HAIRPIN` | NAT for hairpin (rewrite source IP). | |
| 23 | 15 | `LS_IN_HAIRPIN` | Apply hairpin: redirect packet back to the client. | |
| 24 | 16 | `LS_IN_ACL_AFTER_LB` | ACL evaluation a second time after LB DNAT. | |
| 25 | 17 | `LS_IN_STATEFUL` | Commit a conntrack entry for a new connection. | Read REGBIT_CONNTRACK_COMMIT, emit ct(commit). |
| 26 | 18 | `LS_IN_ARP_ND_RSP` | ARP/ND responder (answer ARP requests from the LSP). | |
| 27 | 19 | `LS_IN_DHCP_OPTIONS` | Construct DHCP options. | Set REGBIT_DHCP_OPTS_RESULT. |
| 28 | 20 | `LS_IN_DHCP_RESPONSE` | Generate the DHCP reply. | |
| 29 | 21 | `LS_IN_DNS_LOOKUP` | DNS name lookup (OVN native DNS). | Set REGBIT_DNS_LOOKUP_RESULT. |
| 30 | 22 | `LS_IN_DNS_RESPONSE` | Generate the DNS reply. | |
| 31 | 23 | `LS_IN_EXTERNAL_PORT` | Handle external ports (LSP type=external). | |
| 32 | 24 | `LS_IN_L2_LKUP` | L2 unicast/multicast lookup (final forwarding decision). | Set reg15 to the outport. |
| 33 | 25 | `LS_IN_L2_UNKNOWN` | L2 unknown unicast (BUM flooding decision). | |

**Notes on the most important branching stages:**

- **Tables 8-10 (port-security across three stages).** When a packet does not match the LSP `port_security`, it is dropped at the corresponding stage. The reason for splitting into three stages: L2 (MAC), L3 (IP), and L4 (ND/ARP) can be verified independently, and the stage names make the debug message "drop at L2 or L3" clear.
- **Tables 13-17 (ACL pre-stateful and ACL eval-action).** ACL compiles into four stages instead of one because OVN uses an `ACL_HINT` register as a **two-pass match-action**. Stage 16 only sets a REGBIT (no verdict emission). Stage 17 reads the REGBIT and emits the verdict. The benefit: the OVS classifier does not have to compare every rule against every packet; stage 16 uses an OVS conjunction match to reduce complexity to O(N+M) instead of O(N*M).
- **Tables 20-23 (LB plus hairpin).** LB DNAT in stage 12 (table 20) rewrites the destination IP from VIP to backend IP. Stages 13-15 then detect the hairpin case (client and backend on the same host) and apply SNAT so the backend does not see its own source IP. This is the core mechanism behind OVN stateful LB.
- **Table 24 (ACL after LB).** The ACL runs a second time after LB DNAT, so policies can target a specific backend IP, for example "deny traffic to backend X". Important point: the ACL before LB sees the VIP, the ACL after LB sees the backend IP.
- **Tables 26-30 (ARP responder, DHCP, DNS native).** OVN can reply to ARP, DHCP, and DNS with a flow rule, without controller intervention. This is one of the differentiators of OVN compared to classical SDN. The control plane does not need to be involved in every request, only ovn-controller installs flow rules ahead of time.
- **Tables 32-33 (L2 lookup and unknown).** Final ingress stages, the principal forwarding decision. Set reg15 to the outport (logical) and leave the ingress pipeline.

## 13.16.5 LS Egress pipeline, ten stages in detail (OF table 40-49)

Source: `northd/northd.c` macro `PIPELINE_STAGES` block "Logical switch egress stages".

| OF table | Logical | Stage name | Main role |
|----------|---------|------------|---------------|
| 40 | 0 | `LS_OUT_PRE_ACL` | Pre-ACL egress (defrag and ct.next for egress traffic). |
| 41 | 1 | `LS_OUT_PRE_LB` | Pre-LB egress (conntrack zone for LB return). |
| 42 | 2 | `LS_OUT_PRE_STATEFUL` | Stateful pre-processing on egress (emit ct()). |
| 43 | 3 | `LS_OUT_ACL_HINT` | ACL hint on egress (same as ingress but for departing packets). |
| 44 | 4 | `LS_OUT_ACL` | ACL action on egress. |
| 45 | 5 | `LS_OUT_QOS_MARK` | QoS DSCP mark on egress. |
| 46 | 6 | `LS_OUT_QOS_METER` | QoS rate-limit on egress. |
| 47 | 7 | `LS_OUT_STATEFUL` | Conntrack commit on egress. |
| 48 | 8 | `LS_OUT_PORT_SEC_IP` | Port-security IP on egress (verify destination IP matches the outport `port_security`). |
| 49 | 9 | `LS_OUT_PORT_SEC_L2` | Port-security L2 on egress (verify destination MAC). |

**Design observations:**

- The egress pipeline is shorter than the ingress pipeline (ten versus twenty-six stages). The reason: many decisions (LB, hairpin, ARP responder, DHCP) have already been made on ingress. Egress only needs the final policy enforcement plus exit-side port-security.
- Port-security appears on both ingress (verify source) and egress (verify destination). This blocks packets carrying a fake source IP from entering an LSP, and at the same time blocks packets carrying a fake destination IP from reaching an LSP whose `port_security` does not match.
- Stateful egress (table 47) commits conntrack for **egress-initiated** connections. This matters for the logical router NAT case: a connection from an LR to an external network must be committed at the egress of the LR (the chassis gateway), not at the ingress of the LR (the source chassis).

## 13.16.6 LR Ingress pipeline, twenty stages in detail (OF table 8-27)

LR_IN shares the OF table block 8-33 with LS_IN. The two are distinguished by the metadata field (datapath_type).

| OF table | Logical | Stage name | Main role |
|----------|---------|------------|---------------|
| 8 | 0 | `LR_IN_ADMISSION` | Ingress admission (verify the destination MAC matches the router MAC). |
| 9 | 1 | `LR_IN_LOOKUP_NEIGHBOR` | Look up the MAC_Binding for the next hop. |
| 10 | 2 | `LR_IN_LEARN_NEIGHBOR` | Learn a new ARP/ND entry into MAC_Binding. |
| 11 | 3 | `LR_IN_IP_INPUT` | Process ICMP echo to a router IP, TTL-exceeded, frag-needed. |
| 12 | 4 | `LR_IN_UNSNAT` | Reverse SNAT (for reply packets entering the router). |
| 13 | 5 | `LR_IN_DEFRAG` | IP defragmentation (needed for LB and NAT). |
| 14 | 6 | `LR_IN_DNAT` | Destination NAT (floating IP, port forwarding, LB DNAT). |
| 15 | 7 | `LR_IN_ECMP_STATEFUL` | ECMP per-connection state for symmetric reply. |
| 16 | 8 | `LR_IN_ND_RA_OPTIONS` | Construct IPv6 RA options. |
| 17 | 9 | `LR_IN_ND_RA_RESPONSE` | Generate the IPv6 RA response. |
| 18 | 10 | `LR_IN_IP_ROUTING_PRE` | Pre-routing policy (set policy hint). |
| 19 | 11 | `LR_IN_IP_ROUTING` | IP routing table lookup (longest prefix match). |
| 20 | 12 | `LR_IN_IP_ROUTING_ECMP` | ECMP next-hop selection (hash on the 5-tuple). |
| 21 | 13 | `LR_IN_POLICY` | Policy-based routing evaluation. |
| 22 | 14 | `LR_IN_POLICY_ECMP` | Policy ECMP next-hop selection. |
| 23 | 15 | `LR_IN_ARP_RESOLVE` | Resolve the next-hop MAC from MAC_Binding (or generate an ARP request). |
| 24 | 16 | `LR_IN_CHK_PKT_LEN` | Check the MTU of the output port. |
| 25 | 17 | `LR_IN_LARGER_PKTS` | Handle oversized packets (drop or emit ICMP frag-needed). |
| 26 | 18 | `LR_IN_GW_REDIRECT` | Redirect centralized traffic to the gateway chassis (the cr-* port). |
| 27 | 19 | `LR_IN_ARP_REQUEST` | Generate ARP/ND requests for an unresolved next hop. |

**Notes on the critical stages:**

- **Tables 9-10 (LOOKUP_NEIGHBOR and LEARN_NEIGHBOR).** OVN separates the MAC binding lookup from learning so that the classifier can batch-process flows. LOOKUP_NEIGHBOR sets REGBIT_LOOKUP_NEIGHBOR_RESULT if the MAC is found in cache. LEARN_NEIGHBOR reads the bit and decides whether to emit the `put_arp()` action to install a new entry.
- **Tables 12-14 (UNSNAT, DEFRAG, DNAT).** The order is critical. UNSNAT runs before DNAT because a reply packet (arriving at the router with a SNAT-translated destination) must be un-SNAT'd before the DNAT lookup runs. DEFRAG sits in the middle because DNAT needs access to the L4 header (port number), so the packet must be defragmented first.
- **Table 15 (ECMP_STATEFUL).** Tracks state for each ECMP connection so that reply traffic returns through the same next hop as the forward direction (avoids asymmetric routing). Uses `ct_label[32..79]` as the source MAC field (see Part 13.17 §13.17.7).
- **Table 19 (IP_ROUTING).** Longest prefix match against the `Logical_Router_Static_Route` table. Depending on whether the router has ECMP, control may fall through to table 20.
- **Table 23 (ARP_RESOLVE).** The final LR ingress stage. Sets the destination MAC to the next-hop MAC (resolved from MAC_Binding). If the entry is not yet resolved, it sets a REGBIT so that stage `ARP_REQUEST` (table 27) emits an ARP/ND request.
- **Table 26 (GW_REDIRECT).** Active only when the router has a distributed gateway (an LRP with `options:redirect-chassis`). This stage detects packets that need to traverse the gateway chassis (centralized SNAT/NAT) and redirects them through the chassisredirect (`cr-`) port.

## 13.16.7 LR Egress pipeline, seven stages in detail (OF table 40-46)

| OF table | Logical | Stage name | Main role |
|----------|---------|------------|---------------|
| 40 | 0 | `LR_OUT_CHECK_DNAT_LOCAL` | Check DNAT local (for internal LB). |
| 41 | 1 | `LR_OUT_UNDNAT` | Reverse DNAT (for outbound reply traffic). |
| 42 | 2 | `LR_OUT_POST_UNDNAT` | Post-un-DNAT processing. |
| 43 | 3 | `LR_OUT_SNAT` | Source NAT (floating IP, masquerade, SNAT pool). |
| 44 | 4 | `LR_OUT_POST_SNAT` | Post-SNAT processing. |
| 45 | 5 | `LR_OUT_EGR_LOOP` | Egress loop prevention (hairpin detection). |
| 46 | 6 | `LR_OUT_DELIVERY` | Delivery: output to LRP egress or a patch port. |

**Notes:**

- **Table 40 (CHECK_DNAT_LOCAL).** A new stage in OVN 22.03 and later. It checks whether a routed packet belongs to internal DNAT (LB DNAT to a backend on the same router). If it does, no SNAT is applied so the backend receives the packet with the original source IP.
- **Tables 41-42 (UNDNAT and POST_UNDNAT).** Reverse DNAT for reply packets. When the router has DNAT'd an ingress packet, the reply must be un-DNAT'd so the external source sees the packet with the original IP.
- **Tables 43-44 (SNAT and POST_SNAT).** SNAT for outbound packets. Apply masquerade or floating IP per the NB.NAT entry.
- **Table 45 (EGR_LOOP).** Prevents egress loops when a packet is routed back to the same router (rare but possible with an incorrect policy routing configuration).
- **Table 46 (DELIVERY).** The final stage. Emits the packet on LRP egress (if the peer is an LSP) or on a patch port (if the peer is another LR). After this stage the packet leaves the LR pipeline and (if needed) enters the ingress pipeline of the destination LSP/LR.

## 13.16.8 Dispatch and utility tables (OF table 0, 37-39, 64-72)

Beyond the four logical pipelines, br-int has nine OF tables for dispatch and utility:

**Table 0: `OFTABLE_PHY_TO_LOG`, classify incoming packets.**

Every packet entering br-int (from a physical NIC over a tunnel or from an LSP local veth) starts at table 0. Flows here match on `in_port` (the OF port of the tunnel or veth) and set:
- `metadata` to the datapath ID (logical switch or router ID).
- `reg14` to the inport tunnel_key (read from Geneve TLV class 0x0102 if from a tunnel, or from Port_Binding if from a local veth).
- `MFF_LOG_FLAGS` (reg10) to the MLF flag (looked up from the tunnel TLV).
- Resubmit to OFTABLE_LOG_INGRESS_PIPELINE (table 8) to enter the main pipeline.

**Table 37: `OFTABLE_REMOTE_OUTPUT`, output to a remote chassis.**

When the outport (reg15) points to an LSP on another chassis, ovn-controller installs a flow at table 37 with the match `metadata=<dp>,reg15=<outport>`. The action pushes a Geneve header and outputs to the tunnel port of the destination chassis. The Geneve TLV class 0x0102 encodes (datapath, inport, outport, MLF) so the destination chassis can reproduce the pipeline state.

**Table 38: `OFTABLE_LOCAL_OUTPUT`, output to a local LSP.**

Similar to table 37 but for local LSPs. The match `metadata=<dp>,reg15=<outport>` triggers an action that sets the OF port to the LSP local veth and outputs.

**Table 39: `OFTABLE_CHECK_LOOPBACK`, prevent egress loopback.**

After OUTPUT, packets sometimes return to the ingress pipeline (in the hairpin or multicast case). Table 39 checks MLF_ALLOW_LOOPBACK; if it is not set, the packet is dropped to avoid an infinite loop.

**Table 64: `OFTABLE_SAVE_INPORT`, save inport before hairpin.**

When a packet enters the hairpin path, OVN must remember the original inport so a later stage (NAT_HAIRPIN) can compute the correct SNAT. Table 64 saves reg14 into a backup register.

**Table 65: `OFTABLE_LOG_TO_PHY`, final output to the wire.**

This stage runs after the packet has left both the ingress and egress logical pipelines. It emits the packet to a physical port (a NIC over a tunnel) or to a local LSP veth.

**Tables 66-67: `OFTABLE_MAC_BINDING` and `MAC_LOOKUP`, MAC binding cache.**

Install learned ARP/ND flows into the MAC_Binding table. Two stages, separated: 66 for insert, 67 for lookup.

**Tables 68-70: LB hairpin pipeline.**

Three dedicated stages for LB hairpin (the case where a packet traverses an LB and the backend shares the LSP with the client). Detect and SNAT so the backend does not see itself.

**Tables 71-72: FDB pipeline.**

Install and look up FDB entries for L2 unknown unicast.

## 13.16.9 Cross-pipeline metadata, packets traversing LS to LR

A tunnel-routed packet typically traverses several logical datapaths: LS-A (ingress) to LR (ingress and egress) to LS-B (ingress and egress). At each transition between datapaths, ovn-controller must:

1. Reset the registers to the appropriate values for the destination datapath.
2. Update the `metadata` field (datapath ID) for the destination datapath.
3. Set reg14 (inport) to the tunnel_key of the port joining the two datapaths (the LSP peer of the LRP).
4. Resubmit to OFTABLE_LOG_INGRESS_PIPELINE (table 8) to start the new ingress pipeline.

This happens in the final ingress stage of the source datapath (`LS_IN_L2_LKUP` or `LR_IN_GW_REDIRECT` / `ARP_RESOLVE`). When the outport points to a peer port (an LRP with patch type), the stage emits:
- `set_field:<peer_dp_id>->metadata`
- `set_field:<peer_inport>->reg14`
- `resubmit(,8)` (back to table 8 = OFTABLE_LOG_INGRESS_PIPELINE).

The packet "jumps" from one datapath to another without traversing the physical wire. This is the essence of **distributed logical routing** in OVN.

> **Source verify:** the transition logic lives in `controller/lflow.c` function `consider_logical_flow()` (OVN branch-22.03), where it processes the `next` action with a target in a different datapath. Cross-link to Part 13.7 §13.7.8 `physical.c` walkthrough for a deeper look at the `put_resubmit()` calls.

## 13.16.10 Anatomy 1, reading `ovs-ofctl dump-flows br-int table=N`

**Bucket:** OVN. **Context:** debug the pipeline by directly dumping an OF table. **Purpose:** identify the logical stage and datapath for each flow. **Activity:** debug and verify flow installation. **Mechanism:** combine the OF table number, metadata, and register state to decode the flow.

**Sample output:**

```
$ ovs-ofctl dump-flows br-int table=20 | head -3
 cookie=0x12345678, duration=10s, table=20, n_packets=42, n_bytes=3528,
   priority=110, ct_state=+new+trk, metadata=0x6, reg0=0x80000/0x80000
   actions=group:1
```

**Decode the seven-attribute Anatomy:**

| Attribute | Sample value | Healthy / Warning / Critical |
|-----------|-------------|------------------------------|
| `cookie=0x12345678` | UUID prefix of the Logical_Flow row in SB. | Healthy: cookie links to the SB row UUID. Warning: cookie=0 means the flow was not installed by ovn-controller. |
| `table=20` | OF table 20 = ingress pipeline logical table 12. | Apply the formula: 20 - 8 = 12, so `LS_IN_LB` (or `LR_IN_DNAT_AND_DEFRAG` if the metadata points to an LR). |
| `metadata=0x6` | Datapath ID = 6. | Cross-reference with `ovn-sbctl find Datapath_Binding tunnel_key=6`. |
| `reg0=0x80000/0x80000` | Bit 19 set. | Bit 19 is not in 22.03.8; if seen, this is a bug. |
| `priority=110` | Match priority. | The LB stage usually uses priorities in the 100-200 range. |
| `ct_state=+new+trk` | New tracked connection. | Consistent with the LB DNAT stage. |
| `actions=group:1` | LB DNAT through a group table. | Group ID 1 is the LB backend pool. |

**Critical signals:**
- If `cookie=0` appears at table 8-49, the flow is not from OVN and must be investigated (it could be an attack or a misconfiguration).
- If a flow appears at table 8 without a `metadata=...` match, port-security is being bypassed for every datapath. This is a serious security issue.

## 13.16.11 Anatomy 2, `ovn-sbctl lflow-list` with a stage filter

**Sample output:**

```
$ ovn-sbctl lflow-list ls0 | grep "ls_in_lb"
  table=12(ls_in_lb        ), priority=110  , match=(ct.new && ip4.dst == 10.96.1.10 && tcp.dst == 80),
    action=(reg1 = 10.96.1.10; reg2 = 80; ct_lb_mark(backends=10.0.0.1:8080,10.0.0.2:8080);)
```

**Decode:**

- `table=12(ls_in_lb)`: stage 12 in the LS_IN pipeline = `LS_IN_LB`. The corresponding OF table is 8 + 12 = 20.
- `priority=110`: priority within the LB stage.
- `match=(ct.new && ip4.dst == 10.96.1.10 && tcp.dst == 80)`: an OVN ACL expression.
- `action=(reg1 = 10.96.1.10; reg2 = 80; ct_lb_mark(...))`: save the original DIP and DPort into reg1 and reg2 (per Part 13.17 §13.17.3) and then perform LB DNAT.

**Cross-link with dump-flows:**

```
# Find the OF flow corresponding to this lflow.
$ ovs-ofctl dump-flows br-int table=20 | grep "metadata=0x$(ovn-sbctl get Datapath_Binding ls0 tunnel_key | tr -d '\"')"
```

## 13.16.12 Anatomy 3, `ovn-detrace` mapping cookie to logical flow

**Sample output:**

```
$ ovs-appctl ofproto/trace br-int 'in_port=3,...' | ovn-detrace --ovnnb=...  --ovnsb=...
[OF table 20] Cookie 0x12345678 ->
  SB Logical_Flow: a1b2c3d4-...
  Pipeline: ingress, datapath: ls0

`ovn-detrace` decodes the cookie chain into the SB Logical_Flow UUID and then into the originating NB resource (Load_Balancer, ACL, NAT, and so on). This is the most important tool when debugging "why does this flow exist".

## 13.16.13 Guided exercise (LS pipeline)

**Goal:** trace a TCP SYN packet through ALL twenty-six stages of the LS_IN pipeline on a Mininet topology.

**Setup:**

```bash
ovn-nbctl ls-add ls-test
ovn-nbctl lsp-add ls-test vm1 -- lsp-set-addresses vm1 'aa:bb:cc:dd:ee:01 10.0.0.1'
ovn-nbctl lsp-add ls-test vm2 -- lsp-set-addresses vm2 'aa:bb:cc:dd:ee:02 10.0.0.2'
ovn-nbctl --type=switch acl-add ls-test from-lport 1000 \
  'inport == "vm1" && ip4.dst == 10.0.0.2 && tcp.dst == 80' allow-related
```

**Trace and verify:**

```bash
# 1. Simulate a packet with ovn-trace.
ovn-trace --detailed ls-test \
  'inport == "vm1" && eth.src == aa:bb:cc:dd:ee:01 && eth.dst == aa:bb:cc:dd:ee:02 \
   && ip4.src == 10.0.0.1 && ip4.dst == 10.0.0.2 && ip.proto == 6 && tcp.dst == 80 && ct.new == 1'
```

Verify in the output that the packet traverses twenty-six stages:
- Stage 0 `ls_in_port_sec_l2`: pass (source MAC matches port_security).
- Stage 1 `ls_in_port_sec_ip`: pass (source IP matches).
- Stage 5 `ls_in_pre_acl`: set REGBIT_CONNTRACK_DEFRAG.
- Stage 9 `ls_in_acl`: allow-related, set REGBIT_CONNTRACK_COMMIT.
- Stage 17 `ls_in_stateful`: emit ct(commit).
- Stage 24 `ls_in_l2_lkup`: outport = "vm2".

```bash
# 2. Dump the corresponding flows on br-int.
ovs-ofctl dump-flows br-int table=8  # PORT_SEC_L2
ovs-ofctl dump-flows br-int table=17 # ACL action
ovs-ofctl dump-flows br-int table=25 # STATEFUL
ovs-ofctl dump-flows br-int table=32 # L2_LKUP
```

Each command matches one stage. Verify priority and match expression align with the ovn-trace output.

## 13.16.14 Guided exercise (LR pipeline)

**Goal:** trace a cross-subnet ping through the twenty LR_IN stages plus the seven LR_OUT stages.

**Setup:**

```bash
ovn-nbctl ls-add ls-blue
ovn-nbctl ls-add ls-red
ovn-nbctl lr-add lr0
ovn-nbctl lrp-add lr0 lrp-blue 00:00:00:00:01:01 192.168.1.1/24
ovn-nbctl lrp-add lr0 lrp-red  00:00:00:00:01:02 192.168.2.1/24
ovn-nbctl lsp-add ls-blue ls-blue-lrp -- \
  lsp-set-type ls-blue-lrp router -- \
  lsp-set-addresses ls-blue-lrp router -- \
  lsp-set-options ls-blue-lrp router-port=lrp-blue
ovn-nbctl lsp-add ls-red ls-red-lrp -- \
  lsp-set-type ls-red-lrp router -- \
  lsp-set-addresses ls-red-lrp router -- \
  lsp-set-options ls-red-lrp router-port=lrp-red
ovn-nbctl lsp-add ls-blue vm-blue -- lsp-set-addresses vm-blue '00:00:00:01:00:01 192.168.1.10'
ovn-nbctl lsp-add ls-red vm-red -- lsp-set-addresses vm-red '00:00:00:01:00:02 192.168.2.10'
```

**Trace ICMP from vm-blue to vm-red:**

```bash
ovn-trace --detailed lr0 \
  'inport == "lrp-blue" && eth.src == 00:00:00:01:00:01 && eth.dst == 00:00:00:00:01:01 \
   && ip4.src == 192.168.1.10 && ip4.dst == 192.168.2.10 && ip.ttl == 64 && ip.proto == 1'
```

Verify the packet traverses the twenty LR_IN stages:
- Stage 0 `lr_in_admission`: pass (eth.dst = router MAC).
- Stage 11 `lr_in_ip_routing`: lookup, next-hop on lrp-red.
- Stage 15 `lr_in_arp_resolve`: resolve the MAC of vm-red from MAC_Binding.
- Stage 18 `lr_in_gw_redirect`: skip (no gateway).

After that, the packet "jumps" to LS_IN of ls-red, traverses the twenty-six LS_IN stages, and ends at stage 24 `ls_in_l2_lkup` with outport = vm-red.

## 13.16.15 Capstone POE, the engineer predicts table IDs

**Predict:** for a concrete OVN feature (for example "Load Balancer hairpin"), predict which OF tables on br-int the packet traverses.

**Problem:** design a topology and trace the expected path (table sequence) on paper. Then verify with ovn-trace.

Example feature, "DHCP request from a VM", predicted path:
1. Table 0 (`OFTABLE_PHY_TO_LOG`): classify the packet into ls0.
2. Table 8 (`LS_IN_PORT_SEC_L2`): pass.
3. Table 9 (`LS_IN_PORT_SEC_IP`): pass (IP=0.0.0.0 for DHCP discover).
4. Table 27 (`LS_IN_DHCP_OPTIONS`): construct DHCP options, set REGBIT_DHCP_OPTS_RESULT.
5. Table 28 (`LS_IN_DHCP_RESPONSE`): generate the DHCP reply, output back to the same LSP.

Verify that the expected sequence matches the ovn-trace output. If it does not match, identify:
- Which stage is missing (the rule has not been installed): check the NB DHCP_Options configuration.
- Which stage is extra (a queue from a different pipeline cuts in): check ACL or LB.

**Why this POE matters:** it helps the engineer build a "mental model" of the pipeline that is solid enough to PREDICT before verifying, instead of only verifying after running. This is the real signal of mastery.

## 13.16.16 Version drift note (22.03 to 24.03 to 24.09)

The OVN team often refines the pipeline. The table below summarises changes from the 22.03.8 baseline (curriculum default) to newer versions:

| Version | Change |
|-----------|----------|
| 22.03.8 (baseline) | 26 LS_IN + 10 LS_OUT + 20 LR_IN + 7 LR_OUT = 63 stages. |
| 22.06 | Added `LS_IN_PRE_ARP_ND_RSP` (a stage between stateful and arp_nd_rsp). LS_IN total = 27. |
| 22.09 | Added `LR_IN_LB_AFF_LEARN` for LB affinity learning. |
| 24.03 | Split `LS_IN_ACL` into `LS_IN_ACL_EVAL` plus `LS_IN_ACL_ACTION`. LS_IN total = 28. Added `LR_IN_DEFRAG_ECMP_STATEFUL`. |
| 24.09 | Refactored mac-learning into `controller/pinctrl.c` with MAX_FDB_ENTRIES (commit `fb96ae3679`). |
| Main (devel) | Pipeline split for an IPSec encap stage is under research. |

**Migration warnings:** every major upgrade can change:
- The number of stages (added or removed).
- The OF table index mapping (changes to `LOG_PIPELINE_LEN`).
- Stage names (renames or splits).
- Register layout (in particular REGBIT).

**Safe procedure:** verify the current cluster pipeline with `ovn-sbctl --columns=table_id,pipeline,external_ids list Logical_Flow | head` to know the current stage layout. Do NOT hardcode OF table numbers in automation scripts.

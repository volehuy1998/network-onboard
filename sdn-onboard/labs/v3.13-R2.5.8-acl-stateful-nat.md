A traditional Cisco engineer who has configured a zone-based firewall
on a Cisco IOS router knows the difference between stateless and
stateful filtering. Stateless filtering (extended ACLs, R2.5.7)
treats every packet independently; the rule set must explicitly
permit both directions of every conversation. Stateful filtering
remembers connection state: once an outgoing TCP SYN is permitted,
the firewall automatically permits the incoming SYN-ACK and all
subsequent packets of the same connection without an explicit reverse
rule. The same logic applies to UDP "connections" (a heuristic based
on first-packet timing), to ICMP echo requests-and-replies, and to
related connections like FTP data sessions tracked by an ALG.

OVS implements stateful filtering through the **conntrack** action:
`ct()`. The action invokes the kernel's netfilter conntrack module
(or the userspace equivalent for `datapath_type=netdev` bridges) to
look up the packet's 5-tuple in a connection-tracking table; the
result populates the `ct_state` match field with one or more flags:
`+new`, `+est` (established), `+rel` (related), `+inv` (invalid),
`+rpl` (reply), `+trk` (tracked, set when ct() has run), and `+snat`
or `+dnat` (NATed). The operator's flow rules then match on
`ct_state` to permit or drop based on connection state.

The `ct(commit)` variant additionally installs a new connection
into the conntrack table; without `commit`, the lookup is read-only.
The `ct(nat(...))` variant performs NAT (source or destination)
during the conntrack lookup, rewriting the packet's IP addresses
and ports. A single OpenFlow rule can combine state-tracking and
NAT, e.g. `actions=ct(commit,nat(src=192.168.99.99))` to commit a new
connection with source NAT to 192.168.99.99.

This lab exercises both flavours. Phase A configures a stateful
firewall that permits TCP to port 8080 (NEW) plus all ESTABLISHED
plus ICMP NEW; default drop. The empirical test: client connects to
server on port 8080 (succeeds), client tries port 9999 (drops), ICMP
ping works. Phase B reconfigures for SNAT: outgoing connections from
the client are source-NATed to 192.168.99.99 before delivery to the
server; the server's reply path uses the conntrack reverse mapping
to rewrite the destination IP back to the client's real address.

## R2.5.8.1. Where this part sits in the curriculum and why

R2.5.8 is the eighth sub-capture of R2.5. R2.5.7 covered stateless
ACL via plain OpenFlow flow rules with `actions=drop`. R2.5.8 extends
the model with conntrack: the same flow-rule mechanism, but with the
addition of the `ct()` action that consults connection state and
allows the operator to write rules like "permit established traffic"
without having to install per-connection forward and reverse rules.

The conntrack mechanism in OVS is the same kernel module that
iptables uses (`nf_conntrack`). When the OVS bridge has `datapath_type=
system` (the default kernel datapath, which this lab uses), the OVS
kernel module hands packets to `nf_conntrack` for connection lookup
and update; the result populates the OpenFlow `ct_state` match field.
For the `datapath_type=netdev` user-space datapath, OVS implements
its own userspace conntrack engine in `lib/conntrack.c`, which
mirrors `nf_conntrack`'s data model and behaviour. The lab exercises
the kernel-side path; the userspace path is functionally equivalent.

NAT (Network Address Translation) is integrated into the conntrack
mechanism rather than being a separate action. The `ct(nat(src=IP))`
form encodes "during conntrack lookup, also perform source-NAT to the
given IP". The conntrack engine records the NAT mapping in the
connection's table entry, so subsequent packets in either direction
get the inverse transformation automatically. This is functionally
identical to iptables' `MASQUERADE` and `SNAT` targets, which also
delegate to the same `nf_conntrack` plus `nf_nat` modules.

The R2.5.8 source anchors are concentrated in three places. First,
the userspace conntrack engine in `lib/conntrack.c` (3351 lines at
v2.17.9). Second, the kernel-side glue in `datapath/conntrack.c`
(2413 lines) which delegates to `nf_conntrack`. Third, the
translation engine that recognises the `ct()` action in OpenFlow
flows: `ofproto/ofproto-dpif-xlate.c:compose_conntrack_action` (line
6394). The lab cites all three.

## R2.5.8.3. The OVS conntrack flow-table pattern

The canonical stateful firewall pattern in OVS uses two tables. Table
0 dispatches untracked packets to `ct()` for state lookup; table 1
branches by `ct_state` and either permits (with optional `commit` for
new connections) or drops.

```openflow
table=0,priority=200,arp,actions=NORMAL
table=0,priority=300,ip,ct_state=-trk,actions=ct(table=1)

table=1,priority=300,ip,ct_state=+new+trk,tcp,tp_dst=8080,actions=ct(commit),NORMAL
table=1,priority=300,ip,ct_state=+new+trk,icmp,actions=ct(commit),NORMAL
table=1,priority=200,ip,ct_state=+est+trk,actions=NORMAL
table=1,priority=100,ip,actions=drop
```

Read each rule's role:

- The first table-0 rule permits ARP unconditionally. Conntrack
  applies only to L3 (IPv4 / IPv6); ARP is L2 and bypasses
  conntrack. Without this rule, ARP would not be tracked but would
  also fall through to the table-0 default drop, breaking L2.

- The second table-0 rule matches IP packets that have NOT yet been
  through conntrack (`ct_state=-trk` means "tracked flag is NOT set,
  i.e. the packet has not yet been tracked"). The action `ct(table=1)`
  invokes the conntrack engine and continues processing in table 1.

- Table 1 then has rules matching specific `ct_state` combinations:
  - `+new+trk` (this packet starts a new connection, just been
    tracked): permit only if it matches one of our allow-list
    conditions (TCP dst port 8080, or ICMP). The `ct(commit)` action
    inserts the new connection into the conntrack table.
  - `+est+trk` (this packet belongs to an existing connection):
    permit unconditionally. The `+trk` is required because conntrack
    must have run.
  - default `priority=100,ip,actions=drop`: any IP packet that didn't
    match a higher-priority rule is dropped. This is the firewall's
    deny-default policy.

The `+new` and `+est` are mutually exclusive in the conntrack state
machine: a single packet is either initiating a new connection or
continuing an existing one, never both. Some packets can have other
flags (`+rel` for related connections like ICMP errors,
`+inv` for malformed); the lab does not exercise these.

```text
root@lab-openvswitch:~/network-onboard# ovs-ofctl dump-flows br0

 cookie=0x0, duration=0.915s, table=1, n_packets=0, n_bytes=0, priority=300,ct_state=+new+trk,icmp actions=ct(commit),NORMAL
[10 other lines omitted, context: priority=300 table=0 ct_state=-trk dispatch rule, priority=200 ARP rule, priority=200 ESTABLISHED rule, priority=100 default drop rule]
 cookie=0x0, duration=0.299s, table=1, n_packets=0, n_bytes=0, priority=100,ip actions=drop
```

The flow table has six rules across two tables. All counters at zero
because no traffic has flown yet.

**Misconception to refute.** "ct(commit) is mandatory for any rule
that matches +new traffic." Wrong. `ct(commit)` is required only if
you want subsequent packets in the same connection to match
`+est+trk`. Without `commit`, the packet is processed but the
connection is not added to the conntrack table, so the second packet
of the connection is again `+new+trk` rather than `+est+trk`. The
practical result: without `commit`, every packet of the connection
must match one of the `+new+trk` allow rules, which defeats the
stateful-firewall optimisation. The `commit` is what makes the
subsequent permit-est rule effective.

**Finish.** Topology built; stateful firewall flow rules installed.

### ▶ Guided Exercise 3: Reconfigure for SNAT

This exercise replaces the firewall flow set with a NAT flow set.
The new rules use `ct(commit,nat(src=192.168.99.99))` to source-NAT
the new connection's source IP from 10.0.0.1 to 192.168.99.99.

**Outcomes.** New flow rules in place. The server sees client traffic
arriving as if it came from 192.168.99.99 (which the server's static
ARP/route entries acknowledge). The server can ping back via the
NAT mapping.

client believes it is talking directly to 10.0.0.2, and the
NAT mapping is invisible at the client side.

**Misconception to refute.** "OVS NAT works without any host-side
configuration." Wrong. The host (in this case ns_server) needs to
know how to route packets to the NAT'ed IP space. In this lab the
server's static `ip route add 192.168.99.99 via 10.0.0.1` is the
configuration that makes the return-path work. In a production
deployment, the NAT'ed IP would typically be a public IP belonging
to the NAT gateway's interface, and the upstream network would
already know how to route to it via standard L3 mechanisms; no
manual route addition would be needed. The lab's setup is
artificial because the NAT'ed IP `192.168.99.99` is not assigned to
any interface; the static route is the workaround.

**Misconception to refute.** "OVS NAT preserves the client's source
port." Mostly. By default `ct(nat(src=IP))` allocates a fresh
ephemeral source port for the NAT'ed connection (port-overload, the
classic SNAT/PAT behaviour). To preserve the client's source port,
use `ct(nat(src=IP:PORT,persistent))` instead; this is rarely useful
unless the operator has a specific application-protocol reason to
keep the original port.

## R2.5.8.4. Side by side: Cisco zone-based firewall, OVS conntrack, iptables stateful

| Concern                   | Cisco zone-based firewall                                | OVS conntrack flow rules                                  | Linux iptables stateful                          |
|---------------------------|----------------------------------------------------------|-----------------------------------------------------------|--------------------------------------------------|
| Configure stateful permit | `class-map type inspect; match protocol tcp; policy-map type inspect; class-name inspect` | `actions=ct(commit),NORMAL` for new; `+est+trk,actions=NORMAL` for existing | `iptables -A FORWARD -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT` |
| Configure SNAT             | `ip nat inside source list 1 interface <wan> overload`  | `actions=ct(commit,nat(src=IP)),NORMAL`                  | `iptables -t nat -A POSTROUTING -o <wan> -j MASQUERADE` |
| Configure DNAT             | `ip nat inside source static <inside> <outside>`        | `actions=ct(commit,nat(dst=IP:PORT)),NORMAL`             | `iptables -t nat -A PREROUTING -d <wan-ip> -j DNAT --to-destination` |
| Inspect connection table  | `show policy-map type inspect zone-pair sessions`        | `ovs-appctl dpctl/dump-conntrack` or `conntrack -L`      | `conntrack -L`                                   |
| State storage location     | ASIC TCAM plus IOS process memory                       | Kernel `nf_conntrack` table                              | Kernel `nf_conntrack` table (same!)              |
| Implementation language    | C plus closed vendor SDK                                 | OVS userspace + kernel C: `lib/conntrack.c`, `datapath/conntrack.c` | Kernel C: `net/netfilter/`                       |
| Source-code anchor         | n/a (closed)                                            | `ofproto/ofproto-dpif-xlate.c:compose_conntrack_action` (line 6394); `lib/conntrack.c:conntrack_execute` | `net/netfilter/nf_conntrack_core.c:nf_conntrack_in` |

The most striking observation: **OVS conntrack and iptables stateful
filtering use the same kernel module** (`nf_conntrack`). The
difference is in the rule-expression syntax (OpenFlow flows versus
iptables rules) and in the integration point (OVS bridge versus
netfilter forward hook). The connection table is shared; an OVS
flow with `ct(commit)` and an iptables rule with `-j MASQUERADE`
both add entries to the same `nf_conntrack` table, and `conntrack -L`
sees both.

## R2.5.8.5. Common misconceptions for the audience

**Misconception 1.** "Stateful firewall is just stateless plus a
'remember' bit." Mostly right but missing the architecture. Stateful
filtering is implemented as a separate connection-tracking module
that maintains a hash table keyed by 5-tuple plus protocol; each
entry has its own state machine (TCP three-way handshake states,
UDP "stream" timeout, ICMP request/reply pairing). The "remember
bit" is a flag in `ct_state` set after the conntrack module has
processed the packet; the actual remembering happens in the conntrack
table.

**Misconception 2.** "ct() runs on every packet, slowing things down."
Partially right. Every packet that hits a `ct()` action incurs a
hash lookup in the conntrack table. For the first packet of a new
connection, this is a miss (slow path) followed by an insert. For
subsequent packets, it is a hit (fast path) with O(1) lookup time.
On commodity hardware the per-packet overhead is on the order of
1-3 microseconds, negligible for normal traffic but visible at
millions-of-packets-per-second rates. Mitigation: scale the
conntrack table size and the number of conntrack hash buckets via
sysctl `net.netfilter.nf_conntrack_buckets`.

**Misconception 3.** "NAT requires changing the IP address." Mostly.
Source NAT (SNAT) and destination NAT (DNAT) rewrite IP addresses;
port-only NAT (port forwarding) rewrites only the L4 port. In OVS,
`ct(nat(src=IP))` rewrites the source IP and a port; `ct(nat(dst=IP:PORT))`
rewrites destination IP and port. The conntrack table records the
inverse mapping for the reply path. Without the inverse mapping, the
reply traffic would not be recognised as belonging to the same
connection and would be dropped.

**Misconception 4.** "ct(commit) is reversible." No. Once a connection
is committed to the conntrack table, removing the OVS flow rule does
not remove the conntrack entry. The entry expires according to its
protocol-specific timeout (TCP: 5 days for ESTABLISHED, 60 seconds
for SYN_SENT, 2 minutes for TIME_WAIT; ICMP: 30 seconds; UDP: 30
seconds for unanswered, 180 seconds for stream). To force removal,
use `conntrack -F` (flush all) or `conntrack -D` (delete specific).

**Misconception 5.** "Conntrack is OVS-specific." Wrong. `nf_conntrack`
is the Linux kernel's connection-tracking module, used by iptables,
nftables, and OVS interchangeably. The same module records connections
regardless of which subsystem populated them. This is one of the
strongest examples of Linux kernel architecture composing well: a
connection-tracking primitive that any L3 forwarding subsystem can
share.

## R2.5.8.6. Production-readiness assessment

The trade-off cost of conntrack at scale. Each tracked connection
consumes ~300 bytes of kernel memory. The default conntrack table
size on Linux is 65536 entries (~20 MB). For a high-volume firewall,
this fills quickly under attack or heavy short-lived TCP traffic.
Mitigation: increase `net.netfilter.nf_conntrack_max` and the
hash-bucket count, monitor `cat /proc/sys/net/netfilter/nf_conntrack_count`.
At 1M+ connections the kernel CPU cost becomes noticeable.

The failure mode under conntrack-table exhaustion. When the table is
full, new connections cannot be tracked. The default behaviour is to
drop the new connection (kernel logs `nf_conntrack: table full,
dropping packet`); the alternative is to set
`net.netfilter.nf_conntrack_tcp_loose=0` to enforce strict tracking.
The lab's small scale never exercises this, but production
deployments should monitor `nf_conntrack_count` against
`nf_conntrack_max` and alert at 80% utilisation.

The monitoring discipline. Every conntrack-using bridge should be
monitored for: (a) total connection count via `nf_conntrack_count`;
(b) per-flow drop counters on the OpenFlow rules; (c) NAT-pool
exhaustion if SNAT is in use (the source-port allocation can fail if
all 65535 ephemeral ports are claimed for a single dst IP). For (c),
the symptom is `nf_conntrack: table full` errors specifically tied
to NAT.

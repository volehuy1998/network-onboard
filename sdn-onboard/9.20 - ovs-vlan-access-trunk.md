## 9.20.1 VLAN, partitioning a physical switch into logical switches

The Lab 6 document (Crichigno and others, USC WASTC 2021) cites the standard definition from the Cisco *Catalyst 4500 Configuration Guide* of 2018: *"A VLAN is a way to partition a single switch into multiple switches. Through VLANs, a group of devices on one or more LANs can be configured to communicate as if they were attached to the same wire."*

Historical context: before VLANs, one physical switch was one broadcast domain. When a company had multiple departments (engineering, marketing, accounting) that wanted to separate traffic while sharing the same physical infrastructure, the only option was to buy multiple switches, which was expensive and inflexible. VLANs were introduced (standardized as IEEE 802.1Q in 1998) to allow one switch to be split into multiple "virtual switches" on the same hardware, each with its own broadcast domain.

The three primary applications per Seifert and Edwards (*The All-New Switch Book*, Wiley 2008):

**Configuration flexibility.** Configure any set of users, servers, or devices into a closed group (a VLAN) even when they are in different buildings or geographical areas. Users do not have to physically move to "join" a VLAN.

**LAN security.** A VLAN does not forward traffic between ports that belong to different VLANs. The switch refuses to bridge a VLAN 10 frame to a VLAN 20 port unless an intermediate router is involved. This mechanism prevents passive eavesdropping and bogus transmission.

**User mobility.** When a user moves to a new physical location, the new switch can assign the port to the same VLAN as before. The user sees an unchanged network context and the IP address does not need to change. This is the foundation for 802.1X plus dynamic VLAN assignment in enterprise environments.

> **Key topic:** a VLAN is a tool for partitioning broadcast domains at L2. It does not provide routing between VLANs, that is the job of a router or L3 switch (or an OVN Logical Router, see Part 9.18 and Part 13.11). Confusing "VLAN isolation" with "subnet isolation" is the source of many configuration errors. A VLAN is an L2 concept, a subnet is an L3 concept.

## 9.20.2 Structure of a VLAN-tagged Ethernet frame (IEEE 802.1Q)

The VLAN tag is inserted into the Ethernet frame after the source MAC field and before the original EtherType or Length field. The 4-byte tag has two parts:

```
+----------+----------+----------+-----------+-------------+--------------+
| Dst MAC  | Src MAC  |  TPID    |    TCI    |   EtherType |   Payload    |
| 6 bytes  | 6 bytes  | 0x8100   |  2 bytes  |   (IP/ARP)  |              |
|          |          | 2 bytes  |           |   2 bytes   |              |
+----------+----------+----------+-----------+-------------+--------------+

                            TCI breakdown (16 bits):
                            +-----+---+--------------+
                            | PCP |DEI|     VID      |
                            | 3 b |1 b|    12 bit    |
                            +-----+---+--------------+
```

The four fields inside the TCI (Tag Control Information):

| Field | Size | Meaning |
|-------|------|---------|
| **TPID** (Tag Protocol ID) | 16 bits | Always `0x8100` for 802.1Q, `0x88A8` for 802.1ad Q-in-Q |
| **PCP** (Priority Code Point) | 3 bits | 0 to 7, the QoS class code (IEEE 802.1p) |
| **DEI** (Drop Eligible Indicator) | 1 bit | Before 2011 it was called CFI. A frame with DEI set is preferentially dropped on congestion |
| **VID** (VLAN ID) | 12 bits | 0 to 4095. VID 0 means priority-tagged with no VLAN, VID 4095 is reserved, VID 1 is usually the default VLAN |

The usable range for a VLAN is 1 to 4094, so 4094 VLANs maximum on one domain. This limit is the reason cloud data centres use 24-bit VXLAN VNI (16 million) instead of VLANs, see Part 11.0.

`ovs-fields(7)` describes the matching fields: `dl_vlan` (VID), `dl_vlan_pcp` (PCP), `vlan_tci` (the entire TCI). OVS can match, set, push, or pop a VLAN tag through flow actions (`mod_vlan_vid`, `push_vlan`, `pop_vlan`).

> **Common misconception:** "A VLAN tag increases the frame size, so the MTU must be increased accordingly."
>
> Reality: the Ethernet II MTU of 1500 bytes does not include the Ethernet header. When a 4-byte VLAN tag is inserted, the Ethernet header grows from 14 bytes to 18 bytes, but the 1500-byte payload does not change. The total frame length grows from 1518 to 1522 (excluding the FCS). A switch that supports 802.1Q must accept a 1522-byte frame without fragmentation. This is the "baby giant" frame, standardized by IEEE 802.3ac in 1998. If the switch does not understand 802.1Q, the 1522-byte frame may be dropped as "oversized", a common error when trunking through old switches.

## 9.20.3 Access port and trunk port on OVS

The two basic port kinds:

**Access port**, a port that serves only one VLAN. An untagged frame entering the port is assigned the VLAN of the port (the native VLAN). An outgoing frame with a VLAN tag has its tag stripped (removed). A device connected to an access port (host, VM, printer) does not know that VLANs exist, it sees a "regular" Ethernet frame.

OVS configuration:

```bash
[host]# ovs-vsctl add-port br-s1 h1-port tag=10
```

The port `h1-port` is now an access port for VLAN 10. Untagged frames from `h1` are assigned VLAN 10 internally by OVS. Frames leaving the bridge through this port have their VLAN tag stripped before reaching `h1`.

**Trunk port**, a port that serves multiple VLANs at the same time. Frames passing through the trunk port must carry a VLAN tag (except for the native VLAN, which is allowed to be untagged through the trunk in some configurations). Trunk ports are used to connect switch to switch or switch to router.

OVS configuration:

```bash
[host]# ovs-vsctl add-port br-s1 trunk-port trunks=10,20
```

The port `trunk-port` accepts only frames with VLAN tag 10 or 20. Frames with another VLAN are dropped. Untagged frames are also dropped (unless `tag=N` is added as the native VLAN of the trunk).

A port cannot be both access and trunk at the same time, the engineer must choose one. By default (no `tag` and no `trunks`), the port is "VLAN-agnostic" and passes any frame with or without a tag, behaving like a hub.

> **Key topic:** access port and trunk port are two implementations of the same abstraction "a port assigned to a VLAN". An access port serves a device that knows only one VLAN (a host, a VM). A trunk port serves an infrastructure peer that needs to understand multiple VLANs (another switch, router, or firewall). In OVS, the difference is `tag=N` (access) versus `trunks=N,M` (trunk).

## 9.20.4 Lab 6 topology, four hosts and two VLAN switches

Lab 6 uses a standard topology to demonstrate access plus trunk:

```
             Trunk port                         Trunk port
             VLAN 10, 20                        VLAN 10, 20
                 |                                    |
   +-------------+--------+          +----------------+------+
   |        br-s1         |          |          br-s2        |
   |      (switch 1)      |<-------->|       (switch 2)      |
   |                      |          |                       |
   | port 1      port 2   |          |  port 1      port 2   |
   | (tag=10)    (tag=20) |          |  (tag=10)    (tag=20) |
   +----+---------+-------+          +------+---------+------+
        |         |                         |         |
    +--------+  +--------+            +--------+  +--------+
    | host 1 |  | host 2 |            | host 3 |  | host 4 |
    | vlan10 |  | vlan20 |            | vlan10 |  | vlan20 |
    +--------+  +--------+            +--------+  +--------+
    10.0.0.1  20.0.0.1                  10.0.0.2  20.0.0.2
```

Design: h1 and h3 are in VLAN 10 (subnet 10.0.0.0/24), h2 and h4 are in VLAN 20 (subnet 20.0.0.0/24). The two switches are connected by a trunk port that carries both VLANs. Expectations:

- `h1` pings `h3` successfully (same VLAN 10, traffic crosses the trunk).
- `h2` pings `h4` successfully (same VLAN 20, traffic crosses the trunk).
- `h1` cannot ping `h2` or `h4` (different VLAN, a router is required).

There is no router in the topology. If inter-VLAN routing is required, the engineer must add an OVN Logical Router or an OVS L3 flow (Part 9.18).

The ping fails even though both hosts are in the same subnet. OVS does not forward a VLAN 10 frame to an access port on VLAN 20. The counter-hypothesis is rejected.

```bash
[host]# ip netns exec ns-h2 tcpdump -i h2-port -nn -c 5 &

```

Explanation: VLAN isolation truly blocks traffic at the OVS bridge layer. The ARP Request broadcast from h1 is tagged VLAN 10 by the access port. When flooded internally, the bridge sends it only to other ports on VLAN 10 (there is no other VLAN 10 port on br-s1 except the trunk). Port h2 is access VLAN 20, so the bridge refuses to forward a VLAN 10 frame to it.

## 9.20.5 Comparing classical VLAN with OVN Logical Switch

VLANs work well in a small data centre but hit two limits at scale:

**Limit 1, a maximum of 4094 VLANs.** The 12-bit VID allows only 4096 IDs (subtract 0 and 4095 reserved, so 4094 are usable). A cloud data centre with more than 10,000 tenants does not have enough VLANs to dedicate one per tenant.

**Limit 2, the broadcast domain stays at L2.** A VLAN that spans through trunks broadcasts across every switch carrying that VLAN. With 1000 hosts on the same VLAN, ARP broadcasts congest the underlay.

OVN Logical Switch solves both:

- It uses a 24-bit `tunnel_key` (Geneve), giving 16 million VNIs instead of 4094 VLANs.
- BUM suppression (Part 18.0): an ARP Responder is installed in the pipeline, so ARP broadcasts do not actually flood the entire overlay.

When deploying OVN, the operator does not configure VLANs manually. Instead, the operator defines a Logical Switch (`ovn-nbctl ls-add tenant-A-ls`) and assigns ports to it. OVN automatically generates the Logical Flows and chooses the `tunnel_key`. VLANs can still be used in OVN for the `localnet` port (which connects the overlay to a physical VLAN), but in that role VLAN is a bridge between two worlds, not the primary isolation mechanism.

> **Production readiness assessment:** OVS VLAN access and trunk are appropriate for:
> - Integration with a physical fabric VLAN (legacy infrastructure).
> - Lab and testing workshops.
> - Small deployments (fewer than 50 tenants).
>
> When scaling beyond 50 tenants or when stateful behaviour is needed (SNAT, load balancing, stateful ACLs), switch to OVN Logical Switch. See Part 13.1 and Part 13.2.

## 9.20.X The four `vlan_mode` settings at the source level

An OVS Port has four `vlan_mode` settings (per the `ovs-vsctl(8)` man page, section "VLAN Configuration"):

| Mode | Behaviour on input tag | Behaviour on output tag |
|------|------------------------|--------------------------|
| `access` | Drop tagged, accept untagged | Strip the tag (untagged) |
| `trunk` | Accept all tagged frames in `trunks` | Keep the tag |
| `native-tagged` | Accept tagged. Untagged means tag with the `tag` value | Keep the tag |
| `native-untagged` | Accept tagged. Untagged means tag with the `tag` value | Strip the tag for the VLAN matching `tag` |

### Source code anchor (OVS v2.17.9)

[`ofproto/bond.c`](https://github.com/openvswitch/ovs/blob/v2.17.9/ofproto/bond.c) and [`ofproto/ofproto-dpif-xlate.c`](https://github.com/openvswitch/ovs/blob/v2.17.9/ofproto/ofproto-dpif-xlate.c).

The function `xlate_normal()` in `ofproto-dpif-xlate.c` handles the VLAN logic for the NORMAL action:

```c
// Pseudo-code
static void xlate_normal(struct xlate_ctx *ctx) {
    struct flow *flow = &ctx->xin->flow;

    // Input VLAN classification
    if (port->vlan_mode == VLAN_MODE_ACCESS) {
        if (flow->vlan_tci != 0) {
            // Tagged on access port: drop
            xlate_normal_drop(ctx);
            return;
        }
        flow->vlan_tci = htons(port->tag | VLAN_CFI);  // tag with port.tag
    } else if (port->vlan_mode == VLAN_MODE_TRUNK) {
        if (!bitmap_is_set(port->trunks, vlan_tci_to_vid(flow->vlan_tci))) {
            // VLAN not in the trunks set: drop
            xlate_normal_drop(ctx);
            return;
        }
    }
    // ... continue forwarding
}
```

### `push_vlan` and `pop_vlan` actions

OpenFlow 1.1 and later add explicit actions:

- `push_vlan:0x8100`: push an 802.1Q tag (Ethertype `0x8100`). After the push, set the VID through `set_field`.
- `push_vlan:0x88a8`: push an 802.1ad outer tag (QinQ).
- `pop_vlan`: remove the outermost VLAN tag.

```bash
[host]# ovs-ofctl add-flow br-test \
        "ip,actions=push_vlan:0x8100,set_field:0x100a->vlan_tci,output:1"
# Push a VLAN with VID=10 (0x000a) and CFI=1 (0x1000), giving 0x100a.
```

Source: parsing of the `push_vlan` action in [`lib/ofp-actions.c`](https://github.com/openvswitch/ovs/blob/v2.17.9/lib/ofp-actions.c) flows through `parse_set_vlan_vid()` (verified at line 1618 in v2.17.9) and the OpenFlow action enum `OFPACT_PUSH_VLAN` (line 462). Execution happens in `ofproto-dpif-xlate.c`. There is no top-level function named `parse_push_vlan()` in v2.17.9.

## 9.20.Y QinQ (802.1ad) deep dive

QinQ is VLAN stacking: an outer tag (S-VLAN, the service provider tag) plus an inner tag (C-VLAN, the customer tag). It allows a service provider to transport multiple customer VLANs over shared infrastructure.

### Header layout

```
+--------+--------+--------+---------+--------+--------+------+
| DstMAC | SrcMAC | TPID   | TCI     | TPID   | TCI    | Eth  | Data
| 6B     | 6B     | 0x88a8 | S-VLAN  | 0x8100 | C-VLAN | Type |
|        |        |        | 16-bit  |        | 16-bit | 2B   |
+--------+--------+--------+---------+--------+--------+------+
                  ^                  ^
                  Outer tag          Inner tag
                  (TPID 0x88a8)      (TPID 0x8100)
```

Frame length: 1518 + 4 (S-VLAN) + 4 (C-VLAN) = 1526 bytes. Switches on the path must support a frame size of at least 1526 bytes (jumbo MTU 1512 or higher is recommended).

### OVS QinQ configuration

```bash
# OVS port acting as an S-VLAN trunk
[host]# ovs-vsctl add-port br0 trunk-port \
        -- set Port trunk-port vlan_mode=dot1q-tunnel \
        -- set Port trunk-port tag=100 \
        -- set Port trunk-port other_config:qinq-ethtype=802.1ad
```

`vlan_mode=dot1q-tunnel` (OVS 2.10 and later): the port receives untagged frames from the S-VLAN domain together with the inner C-VLAN tag, then pushes an S-VLAN tag with TPID `0x88a8`. The reverse on egress: pop the S-VLAN tag.

Source code: in `ofproto-dpif-xlate.c`, the function `xlate_normal()` checks `vlan_mode=dot1q-tunnel` and then executes `push_vlan(0x88a8)` and `pop_vlan` symmetrically.

- The input port mode: `access` with tag 10.
- VLAN classification: an untagged frame is tagged with VID 10.
- Forwarding: flood with VLAN tag 10 (output: 3, 4, 5, the trunk ports for VLAN 10).
- Datapath action: `push_vlan(vid=10,...)` inserts the tag, then output to multiple ports.

**Failure scenarios:**

- The trace says "input dropped, frame is VLAN tagged but port has no VLAN configured": the port has `vlan_mode=access` but received a tagged frame. Verify the peer configuration.
- A frame goes through a trunk but the VID is not in the `trunks` list: the frame is dropped. Add the VID to `trunks`: `ovs-vsctl set Port trunk-port trunks=10,20,30`.
- The QinQ outer tag is not stripped at the receiver: the receiver does not have `dot1q-tunnel` mode. Misconfiguration.

## 9.20.W Capstone POE: `native-tagged` versus `native-untagged`

> **Context:** the engineer configures a trunk port that connects a Cisco switch to an OVS bridge. The Cisco trunk has "native VLAN 1 untagged". Should OVS use `native-tagged` or `native-untagged`?

**Claim to verify:** "OVS `native-tagged` is compatible with Cisco native VLAN. `native-untagged` is not needed."

**Predict:** wrong. Cisco native VLAN sends untagged frames for the native VID. OVS must use `native-untagged` to mirror this behaviour.

**Observe (assumed):**

- Set up the OVS trunk port with `native-tagged tag=1 trunks=1,10,20`. The Cisco peer trunk has native VLAN 1.
- Cisco sends a VLAN 1 frame (untagged): OVS receives, classifies as untagged, and tags with `tag=1` (the native VLAN). OK.
- OVS sends a VLAN 1 frame to the trunk: `native-tagged` mode keeps the tag, so the frame has an 802.1Q tag with VID 1.
- Cisco receives a frame with VID 1 (tagged) on the trunk: Cisco expects untagged for the native, so it is confused. Some Cisco switches drop it, others accept it.

Switching to `native-untagged`:

- OVS sends a VLAN 1 frame to the trunk: `native-untagged` strips the tag, so the frame is untagged.
- Cisco receives an untagged frame, classifies as VLAN 1 (native), and forwards. Healthy.

**Explain:**

1. **Cisco trunk default behaviour:** the native VLAN frame is untagged on the wire (RFC 802.1Q practice).
2. **OVS `native-tagged`:** keep the tag for the native VLAN. Different from Cisco semantics.
3. **OVS `native-untagged`:** strip the tag for the native VLAN, matching Cisco. Correct for interop.
4. **Pure trunk (no native):** use `trunk` mode, with no native concept.

**Falsification test:**

- Capture a frame from OVS to Cisco with `native-tagged`. Verify whether the frame has tag VID 1.
- If Cisco accepts the tagged frame, that is a counter-example. In practice, Cisco rejects a tagged native by default (a security feature).

**Where the original claim is wrong:** the name `native-tagged` does not mean "supports Cisco native VLAN". The correct rule is: **Cisco interop equals OVS `native-untagged` (the native is untagged by default). `native-tagged` is for legacy non-standard switches that force a tag on every frame.**

> **Rule:** a Cisco, Juniper, or Arista trunk peer pairs with OVS `native-untagged` (default native untagged). Forcing a tag on every frame uses `native-tagged` (rare, non-standard).

## Key takeaways

First, a VLAN is a mechanism for partitioning broadcast domains at L2, creating multiple "logical switches" on one physical switch. It is not L3 isolation, a router is still needed for inter-VLAN traffic.

Second, an 802.1Q frame has a 4-byte tag (TPID `0x8100` plus a 16-bit TCI), pushing the maximum frame size from 1518 to 1522 bytes. A switch that does not support 802.1Q may drop these frames.

Third, an access port serves one VLAN (`tag=N`) and strips the tag when sending to the host. A trunk port serves multiple VLANs (`trunks=N,M`) and keeps the tag when forwarding. A port cannot be both access and trunk.

Fourth, a 12-bit VID limits the number of VLANs to 4094, which is not enough for cloud scale. An OVN Logical Switch uses a 24-bit Geneve `tunnel_key` (16 million IDs) to overcome this limit. VLAN and OVN do not conflict. VLAN is used on the physical side (the `localnet` port), and the Logical Switch is used on the overlay side.

## 9.20.X Comparison with other systems (axis 20 backfill, 2026-04-27)

VLAN access and trunk semantics have cross-vendor analogues:

- **Cisco IOS** (Catalyst series): `switchport mode access vlan N` for access port, `switchport mode trunk` plus `switchport trunk allowed vlan add 10,20` for trunk. Native VLAN concept: `switchport trunk native vlan 1` (default). A frame on the native VLAN leaves the trunk **without an 802.1Q tag** by default, but this can be overridden with `vlan dot1q tag native`. **OVS `vlan_mode=native-untagged` matches the Cisco default** (native frame untagged on egress). OVS `native-tagged` is the opposite (the tag is preserved).
- **Juniper Junos** (EX and QFX series): `interface-mode trunk` plus `vlan members [10 20]` plus `native-vlan-id 1`. By default Junos **untags the native VLAN on egress** (matching the Cisco default and OVS `native-untagged`). Different from Cisco in syntax and naming convention.
- **Arista EOS** (DCS series): `switchport mode trunk` plus `switchport trunk native vlan 1` plus `switchport trunk allowed vlan 10,20`. The default behaviour is similar to Cisco and Junos: the native VLAN is untagged on egress. The native VLAN concept is harmonized across the three major vendors.
- **Linux native bridge (`brctl` and `ip link`)**: VLAN filtering through `bridge vlan add` plus `vlan_filtering=1`. By default, frames are untagged on egress (the PVID concept is the equivalent of the native VLAN). Different from OVS in granularity: the Linux bridge uses per-port PVID, OVS combines per-port `tag` and `trunks`.
- **VMware vSwitch and NSX**: a port group has `vlan_id` plus `mode (none/vlan/trunk/private)`. NSX uses VLAN trunk and then untags or tags internally at the virtual NIC. Different from the OVS architecture: NSX is controller-driven, OVS is switch-driven configuration through OVSDB.
- **IEEE 802.1Q standard**: defines the native VLAN concept (Port VLAN Identifier, PVID) which allows untagged frames to enter and leave a trunk. OVS `native-untagged` mode matches the IEEE 802.1Q PVID semantic.

**Core insight:** OVS VLAN trunk semantics align with IEEE 802.1Q and the three major vendors (Cisco, Junos, Arista). The main difference is in the **default behaviour for native VLAN egress**: Cisco, Junos, Arista, and OVS `native-untagged` all default to untagged on egress. OVS `native-tagged` is tagged on egress (atypical, used for specific cases such as interop with hardware that is not strictly 802.1Q compliant). An engineer working in a multi-vendor environment must verify the mode of every switch to avoid native-tag mismatch (a tag that is stripped at hop A but kept at hop B).

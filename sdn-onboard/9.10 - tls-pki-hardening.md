## 9.10.1 ptcp versus ssl

Controller and Manager connections support three transports:

- **`unix:`**: local Unix socket, available only to root on the same host. Secure by default.
- **`ptcp:PORT`**: passive TCP, no authentication, no encryption. **Lab use only.**
- **`ssl:IP:PORT`**: TLS with mutual authentication (mTLS). **Mandatory in production.**

Compass Ch K: "Every `Controller` or `Manager` target should be `ssl:IP:PORT`."

`ptcp:` in production is a risk. Anyone who can reach the TCP port can (a) read the entire OVSDB configuration, (b) inject arbitrary flow rules, (c) run remote commands through OpenFlow.

---

## 9.10.2 `ovs-pki`: a minimal PKI toolkit

OVS ships with `ovs-pki`, a script that wraps OpenSSL. Compass Ch K workflow:

```bash
[ca-host]# ovs-pki init                            # create the root CA
[ca-host]# ovs-pki req+sign ovsclient switch      # switch certificate
[ca-host]# ovs-pki req+sign ctl controller         # controller certificate

[switch-host]# ovs-vsctl set-ssl /etc/openvswitch/key.pem \
                  /etc/openvswitch/cert.pem \
                  /etc/openvswitch/cacert.pem
[switch-host]# ovs-vsctl set-controller br0 ssl:10.0.0.10:6653
```

- `ovs-pki init` creates `switchca/` and `controllerca/` with the standard X.509 CA structure.
- `req+sign ovsclient switch` creates a private key, a CSR, and signs it with the CA.
- `set-ssl` writes the three paths into the OVSDB SSL row.

---

## 9.10.3 Verify the handshake

From the controller side:

```bash
[controller]$ openssl s_client -connect 10.0.0.10:6653 \
              -CAfile cacert.pem -cert ctl-cert.pem -key ctl-privkey.pem
```

The output should show `Verify return code: 0 (ok)` and a TLS version of 1.2 or higher.

A common error is `certificate verify failed`. The certificate chain does not match the CA (usually because `cacert.pem` is missing an intermediate CA).

---

## 9.10.4 CA rotation

Compass Ch K rule: **"Rotate CAs by adding the new root to `cacert.pem` BEFORE switching the client certificate, never the reverse."**

The procedure:

1. Generate a new CA `new-ca.pem`.
2. Append the new CA to `cacert.pem` on every switch and controller.
3. Wait for propagation.
4. Issue a new client certificate signed by the new CA.
5. Switch the client certificate to the new certificate. The peer still trusts it because both CAs are present.
6. After the system stabilises, remove the old CA from `cacert.pem`.

The reverse order (switching the certificate first) means the peer does not trust it and the connection fails.

---

## 9.10.5 TLS version and cipher

OVS 2.17+ supports TLS 1.2 and TLS 1.3:

```bash
[host]# ovs-vsctl set SSL <uuid> bootstrap_ca_cert=false \
         ssl_protocols=TLSv1.2,TLSv1.3 \
         ssl_ciphers=HIGH:!aNULL:!MD5
```

Bootstrap mode (`bootstrap_ca_cert=true`) is for first-time setup. **Disable it as soon as the system is operational** to avoid MITM attacks.

## 9.10.6 SSL row in `vswitch.ovsschema`

**Concept.** `SSL` is a table in the `Open_vSwitch` database (file `vswitch.ovsschema`) that contains **exactly one row** (`maxRows: 1`). This row holds all the certificate information needed for `ovs-vswitchd` and `ovsdb-server` to perform a TLS handshake with a peer (controller or OVSDB client).

**Role and why it exists.** Before the SSL row, each daemon received certificate paths through `--private-key`, `--certificate`, and `--ca-cert` command-line arguments. In a large cluster, changing a path meant restarting the daemon. Putting the certificate paths into OVSDB allows `ovs-vsctl set-ssl` to change them at runtime; the daemon reads the OVSDB notification and updates in place.

**Five-column structure (verified `vswitch.ovsschema` v2.17.9):**

| Column | Type | Role |
|--------|------|------|
| `private_key` | string | Path to the PEM file with the switch private key |
| `certificate` | string | Path to the PEM file with the switch certificate (signed by the CA) |
| `ca_cert` | string | Path to the PEM file with the CA certificate (root and intermediate if any) |
| `bootstrap_ca_cert` | boolean | Enable bootstrap mode (the first connection fetches the CA certificate automatically) |
| `external_ids` | map<string,string> | Free-form metadata, no effect on the SSL handshake |

**Usage workflow.** `ovs-vsctl set-ssl <privkey> <cert> <cacert>` writes the three paths into the SSL row and sets the reference from `Open_vSwitch.ssl` to the row UUID. Verify with `ovs-vsctl get-ssl` or `ovs-vsctl list SSL` (see Anatomy Template A in §9.10.X).

**Common failures and diagnostic signals.** If `Open_vSwitch.ssl` is empty but the controller uses `ssl:`, the log shows `Stream_ssl|ERR|Private key must be configured`. Fix: run `set-ssl` with all three paths. If `private_key` points to a world-readable file (mode 644), the OpenSSL daemon still loads it but this is a CRITICAL security bug (Failure 2 in §9.10.Z).

**Source code reference.** The schema file `vswitchd/vswitch.ovsschema` v2.17.9 defines the table named `"SSL"`; the key `"maxRows": 1` confirms the singleton invariant. The code that reads and applies this row is `lib/stream-ssl.c`, functions `stream_ssl_set_private_key_file` and `stream_ssl_set_certificate_file`.

## 9.10.7 Manager row and the management connection target

**Concept.** `Manager` is a table in the `Open_vSwitch` schema used to declare the **OVSDB management endpoint** (the endpoint where `ovsdb-server` listens for external processes such as `ovn-controller`, OpenStack Neutron, or another CMS to read and write OVSDB). Different from `Controller` (which is for OpenFlow), the Manager table is dedicated to the OVSDB management layer.

**Main columns (verified schema v2.17.9):**

| Column | Type | Role |
|--------|------|------|
| `target` | string (index) | Endpoint string: `pssl:6640`, `ssl:10.0.0.1:6640`, `ptcp:`, `unix:` |
| `inactivity_probe` | integer (ms) | Probe interval to detect a dead peer |
| `max_backoff` | integer (ms greater than or equal to 1000) | Cap on the increasing retry backoff after errors |
| `connection_mode` | enum `in-band` or `out-of-band` | Routing path to the manager (out-of-band means a management network) |
| `is_connected` | boolean (ephemeral) | Runtime state, not persisted |
| `status` | map (ephemeral) | Runtime detail: `last_error`, `state` |
| `external_ids`, `other_config` | map | Metadata and tuning |

**Usage workflow.** `ovs-vsctl set-manager pssl:6640` adds a Manager row with target `pssl:6640` (`ovsdb-server` listens for TLS on port 6640). When the target is `ssl:` or `pssl:`, `ovsdb-server` uses the SSL row from §9.10.6 for the certificate paths.

**Production role.** In an OVN cluster, the Manager row on each compute node is typically `ptcp:6640:127.0.0.1` (local only) or `pssl:6640` if the CMS needs remote access. Reference: the column `Open_vSwitch.manager_options` is set to a list of Manager UUIDs.

**Common failures.** `is_connected=false` together with `status:{last_error="connection refused"}` reports that `ovsdb-server` is not listening on the expected port. Verify with `ss -tlnp | grep 6640` on the target host.

**Source code reference.** The schema entry `"Manager"` is in `vswitchd/vswitch.ovsschema`. The logic is in `vswitchd/bridge.c`, function `bridge_configure_remotes`, which reads Manager rows and forwards them to `ovsdb-server` through jsonrpc.

## 9.10.8 `ovs-pki self-sign` for lab and test

**Concept.** `ovs-pki self-sign <name>` creates a **self-signed** certificate from an existing private key and CSR, with no CA hierarchy. The OpenSSL equivalent is `openssl x509 -req -signkey privkey.pem -in req.pem -out cert.pem -days 3650`.

**Why this command exists.** The full PKI bootstrap (`init`, `req`, `sign`) is appropriate for production but takes 4 to 5 steps per node. In a single-node lab or a CI test that just needs the TLS handshake to work (without peer trust), self-signing reduces the workflow to two steps: `req` and `self-sign`.

**Three-command workflow:**

```bash
$ ovs-pki req lab01            # create lab01-privkey.pem and lab01-req.pem
$ ovs-pki self-sign lab01      # create lab01-cert.pem (signed by lab01-privkey itself)
$ openssl x509 -in lab01-cert.pem -text -noout | grep -E 'Issuer|Subject|Validity'
```

**Properties of the certificate produced by `self-sign` (verified in `utilities/ovs-pki.in` v2.17.9):**

- Validity is 3650 days (10 years), hardcoded in the script.
- `subjectAltName = DNS:<name>` is added automatically.
- Issuer equals Subject (the defining property of self-signed).
- The temporary file `cert.pem.tmp` mode is `077 umask` (owner read only).

**Real-world incident.** Self-sign is often confused for production use. When two chassis each have a self-signed certificate but do not trust one another, the handshake fails with `unable to get local issuer certificate`. Fix: switch to the full PKI workflow with a common CA.

**Source code reference.** `utilities/ovs-pki.in` v2.17.9 case `"self-sign"`: it checks that `arg1-req.pem` and `arg1-privkey.pem` exist, creates an extfile that contains `subjectAltName`, and calls `openssl x509 -req -signkey ... -days 3650`.

**Comparison with `sign`.** `self-sign` uses the subject's own private key to sign, so the CA private key is not needed (the result is safe to distribute). `sign` requires access to `controllerca/cakey.pem` or `switchca/cakey.pem`, and so it runs only on the CA host. See §9.10.9.

## 9.10.9 `ovs-pki sign` for the production PKI flow

**Concept.** `ovs-pki sign <name> <type>` signs a CSR (`<name>-req.pem`) with the CA private key, producing `<name>-cert.pem`. `<type>` must be `switch` or `controller`, which decides which CA signs (`switchca/` or `controllerca/`).

**Two-host workflow (production-grade):**

```bash
# On the chassis (host hv01):
$ ovs-pki req hv01                       # generate hv01-privkey.pem (does NOT leave the host) and hv01-req.pem

# Copy hv01-req.pem to the CA host (scp; no secure channel needed because the CSR is public):
$ scp hv01-req.pem ca-host:/tmp/

# On the CA host:
$ ovs-pki sign hv01 switch               # ovs-pki asks for fingerprint verification (anti-tamper)
$ scp hv01-cert.pem hv01:/etc/openvswitch/
```

**Validity period.** `sign` uses the CA configuration (`switchca/openssl.cnf` defaults to `default_days = 3650`), so the certificate is valid for 10 years. The operator can adjust `default_days` in `openssl.cnf` before signing.

**Why not always use `req+sign`.** `req+sign` runs both `req` and `sign` on the same host. Production requires that the CA private key NOT leave the CA host (defense in depth), so `req` must run on the chassis (keeping the private key local) and `sign` must run on the CA host (keeping the CA key local). `req+sign` is appropriate only for the lab.

**Anti-tamper fingerprint check.** When `sign` runs, `ovs-pki` computes the SHA-1 fingerprint of the CSR and asks the operator to read the same fingerprint aloud from the chassis (out-of-band verification) to prevent a MITM attacker from swapping the CSR. The check can be bypassed with `--batch` (not recommended).

**Common failures.** If `<type>` is wrong (for example, signing as `switch` for a certificate used as a controller), the peer rejects with `certificate verify failed: unable to verify the first certificate`. Fix: re-sign with the correct type, or configure the peer to trust both CAs.

**Source code reference.** `utilities/ovs-pki.in` v2.17.9 case `"sign"`: calls `one_or_two_args`, `check_type "$arg2"` (validates `switch` or `controller`), `verify_fingerprint "$arg1-req.pem"`, and `sign_request "$arg1-req.pem" "$arg1-cert.pem"`. Function `sign_request` wraps `openssl ca -policy ... -out ... -in ...`.

## 9.10.10 PEM certificate files on disk

**Concept.** Every certificate and key OVS uses is in **PEM** format (Privacy-Enhanced Mail), which is text Base64 with header `-----BEGIN CERTIFICATE-----` and footer `-----END CERTIFICATE-----`. PEM differs from DER (binary): PEM is easy to inspect with `cat`, copy and paste through email or chat, and embed in Ansible or Terraform configurations.

**Path convention `/etc/openvswitch/`:**

| File | Content | Recommended mode |
|------|---------|------------------|
| `/etc/openvswitch/key.pem` | Switch private key | `0600 root:root` |
| `/etc/openvswitch/cert.pem` | Switch certificate (signed by `switchca`) | `0644 root:root` |
| `/etc/openvswitch/cacert.pem` | CA certificate (`controllerca`, so the switch trusts the controller) | `0644 root:root` |

These paths are the convention from the Open vSwitch FAQ and the `set-ssl` example. They are not hardcoded in code (the operator can put them elsewhere). The short names `key.pem` and `cert.pem` reduce mistakes; production deployments often rename them to `<hostname>-privkey.pem` to trace the origin.

**Sample content of a PEM certificate:**

```
-----BEGIN CERTIFICATE-----
MIIDXTCCAkWgAwIBAgIJAKl... (Base64 multi-line) ...wJ8B5cVEnQQB6w==
-----END CERTIFICATE-----
```

A PEM file may contain **multiple consecutive blocks**. This is common in `cacert.pem`, which contains the root CA together with the intermediate CA (the chain). OpenSSL parses from top to bottom and uses every block.

**How to inspect.**

```bash
$ openssl x509 -in /etc/openvswitch/cert.pem -text -noout    # decode one cert
$ openssl crl2pkcs7 -nocrl -certfile cacert.pem | openssl pkcs7 -print_certs -text   # decode the chain
$ stat -c '%a %U:%G %n' /etc/openvswitch/key.pem             # verify mode is 0600
```

**Common failures.**

1. **Mode 644 on the private key.** World-readable, the number-one security bug in audits. Fix: `chmod 600`.
2. **Broken PEM encoding.** Missing `-----BEGIN` or `-----END` lines, or corrupted Base64. Symptom: `unable to load private key:no start line`. Fix: re-issue the certificate.
3. **DOS line ending CRLF after `scp` from Windows.** OpenSSL parsing fails on some older versions. Fix: `dos2unix cert.pem`.

**Comparison with other systems.** Linux distributions use various standard PEM paths: `/etc/ssl/certs/` on Debian, `/etc/pki/tls/certs/` on RHEL. OVS uses its own `/etc/openvswitch/` to separate ownership from the system CA bundle; this avoids `update-ca-certificates` overwriting it.

**Source code reference.** The PEM loader is `lib/stream-ssl.c` v2.17.9, function `stream_ssl_set_private_key_file_with_password`, which calls `SSL_CTX_use_PrivateKey_file(..., SSL_FILETYPE_PEM)`. OpenSSL also supports DER through `SSL_FILETYPE_ASN1`, but OVS hardcodes PEM.

### Guided Exercise 11: TLS setup for a controller connection

**Outcomes:** generate a CA and certificates, configure an OVS switch to use `ssl:`.

**Before You Begin:** the host has OpenSSL and `ovs-pki` installed.

**Instructions:**

**1.** Init PKI:

```bash
[ca-host]# mkdir -p /etc/openvswitch/pki && cd /etc/openvswitch/pki
[ca-host]# ovs-pki init --force
```

**2.** Generate the switch certificate:

```bash
[ca-host]# ovs-pki req+sign ovsswitch switch
```

**3.** Generate the controller certificate:

```bash
[ca-host]# ovs-pki req+sign ovsctl controller
```

**4.** Configure the OVS switch:

```bash
[switch-host]# cp /etc/openvswitch/pki/switchca/cacert.pem /etc/openvswitch/
[switch-host]# cp ovsswitch-privkey.pem /etc/openvswitch/key.pem
[switch-host]# cp ovsswitch-cert.pem /etc/openvswitch/cert.pem
[switch-host]# ovs-vsctl set-ssl /etc/openvswitch/key.pem \
                                 /etc/openvswitch/cert.pem \
                                 /etc/openvswitch/cacert.pem
[switch-host]# ovs-vsctl set-controller br0 ssl:192.168.1.10:6653
```

**5.** Verify:

```bash
[switch-host]# ovs-vsctl get Controller br0 is_connected
true
```

## 9.10.X Anatomy Template A, `ovs-vsctl list SSL`

**Context:** verify the TLS configuration and the certificate chain before deploying to production.

**Command and output** *(reproduced from OVS 2.17.9):*

```shell
$ sudo ovs-vsctl list SSL
_uuid               : 7e3c9f82-...
bootstrap_ca_cert   : false
ca_cert             : "/etc/openvswitch/cacert.pem"
certificate         : "/etc/openvswitch/sc-cert.pem"
external_ids        : {}
private_key         : "/etc/openvswitch/sc-privkey.pem"
```

**9-attribute anatomy:**

| Field | Sample value | Meaning | Red flag if |
|-------|--------------|---------|-------------|
| `Open_vSwitch.ssl` ref | `[<uuid>]` | SSL row reference | Empty means OVS has no TLS configuration |
| `ca_cert` path | `/etc/.../cacert.pem` | CA certificate file | File missing means SSL rejection |
| `certificate` path | `/etc/.../sc-cert.pem` | Switch certificate | Self-signed plus peer not trusting means handshake failure |
| `private_key` path | `/etc/.../sc-privkey.pem` | Key file | Permission 644 means world-readable, a security bug |
| `bootstrap_ca_cert` | `false` | Bootstrap mode | `true` left on permanently means MITM risk |
| Cert expiry (openssl) | More than 30 days | Expiration date | Less than 30 days means critical |
| Cert CN | `chassis-1.example.com` | Common Name | Mismatch with hostname means verification failure |
| Chain depth | 3 | CA hierarchy | Greater than 5 means unusual misconfiguration |
| Cipher suite | TLS 1.3 AES-256-GCM | Negotiated | TLS less than 1.2 means deprecated |

**Failure scenarios:**

- **The SSL column is empty and the controller is `ssl:IP:port`.** The TLS configuration is missing. Fix: `ovs-vsctl set-ssl /path/privkey /path/cert /path/cacert`.
- **Private key permission 644.** World-readable. Fix: `chmod 600`.
- **Certificate CN mismatches the hostname.** Hostname verification fails. Fix: re-issue the certificate with a SAN for multiple hostnames.

## 9.10.Y Common misconceptions and key topic

> **Common misconception 1:** "A self-signed certificate is safe in production because everything is internal."
>
> **Reality:** a self-signed certificate has no CA signature, so the trust boundary cannot be distributed at scale. Adding a new chassis requires manually copying the certificate. Revocation does not work. Use a CA (internal through `ovs-pki`, or external through Vault).

> **Common misconception 2:** "Bootstrap mode is for the lab only."
>
> **Reality:** bootstrap mode has a valid use case, the first-time deployment of a remote site. Use a 1-hour window, then disable. The pattern is safer than manual certificate copy when network isolation and physical security are guaranteed.

> **Key topic:** OVS and OVN TLS is the zero-trust foundation. Each chassis has an identity certificate and verifies its peer. `ptcp:` is forbidden in production. Certificate rotation strategy: annual minimum, cross-sign overlap of 60 days, automation through Ansible and `ovs-pki`, alerts at less than 30 days. See Part 20.1 §20.1.15 (zero-downtime certificate rotation) and Part 10.6 (OVSDB mTLS deep dive).

## 9.10.Z Common failure modes

**Failure 1, a chassis cannot connect to SBDB over TLS.** Log: `ssl: failed to verify certificate`. Fix: `openssl verify -CAfile cacert.pem sc-cert.pem`.

**Failure 2, cipher negotiation fails.** Log: `ssl: no shared cipher`. Fix: align the cipher set with the peer.

**Failure 3, certificate rotation downtime.** Cause: a restart after rotation. Fix: cross-sign and `ovs-appctl vlog/reopen` (no restart needed).

**Failure 4, hostname verification fails.** Log: `ssl: server cert CN mismatch`. Fix: re-issue with a SAN that lists multiple hostnames.

---

## Key points to remember

First, production OVS MUST use `ssl:` for every Controller and Manager target. `ptcp:` is for short-term lab use only.

Second, `ovs-pki` wraps OpenSSL for a minimal PKI workflow. It is enough for setup but does not replace an enterprise CA (Vault, step-ca) for large deployments.

Third, CA rotation must follow the order: add the new CA first, then switch the certificate. Reversing the order breaks the handshake.

Fourth, bootstrap mode (`bootstrap_ca_cert=true`) is only for first-time setup. Disable it as soon as the system is operational to avoid MITM.

Fifth, the Anatomy Template A `list SSL` 9-attribute view. Private key 644 is the number-one security bug.

Sixth, self-signed equals technical debt. Always use a CA.

Seventh, certificate rotation can be zero-downtime through cross-sign and reload (no restart). See Part 20.1 §20.1.15 and Part 10.6.

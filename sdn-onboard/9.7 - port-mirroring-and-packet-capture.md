## 9.7.1 OVS Mirror table

Mirror là cơ chế copy traffic từ một/nhiều port/VLAN sang destination (port hoặc VLAN). Trong physical switch gọi là SPAN hoặc RSPAN. OVS triển khai qua `Mirror` table trong OVSDB.

`Mirror` được tham chiếu từ `Bridge.mirrors`. Có:

- **Source selectors**: `select_src_port`, `select_dst_port`, `select_vlan`, `select_all=true`.
- **Destination**: `output_port` (local SPAN) hoặc `output_vlan` (RSPAN — re-tag packet với VLAN này và emit ra trunk).

---

## 9.7.2 Atomic transaction tạo Mirror

Compass Ch G cung cấp idiom chuẩn:

```bash
[host]# ovs-vsctl -- set Bridge br0 mirrors=@m \
         -- --id=@eth0 get Port eth0 \
         -- --id=@eth1 get Port eth1 \
         -- --id=@eth2 get Port eth2 \
         -- --id=@m create Mirror name=mymirror \
                select-dst-port=@eth0,@eth1 \
                select-src-port=@eth0,@eth1 \
                output-port=@eth2
```

Atomic: nếu port eth2 không tồn tại, Mirror row không tạo, Bridge.mirrors không đổi.

Kiểm chứng:

```bash
[host]# ovs-vsctl list Mirror mymirror
[host]# ovs-vsctl get Mirror mymirror statistics
```

Statistics cột `tx_packets`, `tx_bytes` — số packet đã mirror.

---

## 9.7.3 Cleanup Mirror

Compass Ch G: Mirror tự garbage-collect khi unreferenced, khác với QoS/Queue:

```bash
[host]# ovs-vsctl clear Bridge br0 mirrors
```

Sau lệnh này, Mirror row unreferenced → OVSDB tự xóa. KHÔNG cần explicit `destroy Mirror`. Khác biệt quan trọng với QoS/Queue (Phần 9.9) — hai table đó phải `destroy` thủ công.

## 9.7.4 Capture traffic từ mirror output

Mirror ra port `tap0` (internal type), packet copy xuất hiện trên netdev `tap0`:

```bash
[host]# ip link set tap0 promisc on
[host]# tcpdump -i tap0 -nn -w mirror.pcap
```

Feed `mirror.pcap` vào Wireshark hoặc Suricata.

RSPAN, analyzer ở remote host:

```bash
[host]# ovs-vsctl -- --id=@m create Mirror name=rspan \
         select-all=true output-vlan=999
[host]# ovs-vsctl set Bridge br0 mirrors=@m
```

Remote analyzer join VLAN 999 trên trunk.

## 9.7.5 Anatomy Template A — `ovs-vsctl list Mirror`

**Bối cảnh khi chạy:** Verify Mirror active + đọc statistics để confirm traffic actually being mirrored, không phải just config exist.

**Command và output** *(reproduced OVS 2.17.9, mirror vm1 → analyzer)*:

```shell
$ sudo ovs-vsctl list Mirror vm1mirror

_uuid               : 7e3c9f82-1d5e-4b7a-8e9f-abcdef012345
external_ids        : {tenant="T1", capture_reason="incident-response"}
name                : "vm1mirror"
output_port         : [<analyzer_port_uuid>]
output_vlan         : []
select_all          : false
select_dst_port     : [<vm1_port_uuid>]
select_src_port     : [<vm1_port_uuid>]
select_vlan         : []
snaplen             : []
statistics          : {tx_bytes=45678, tx_packets=321}
```

**9-attribute anatomy:**

| Field | Giá trị mẫu | Ý nghĩa | Red flag nếu |
|-------|-------------|---------|--------------|
| `name` | `"vm1mirror"` | Human-readable mirror name | Unset = hard to audit |
| `output_port` | `[<uuid>]` | Local SPAN destination | Both output_port + output_vlan set = config error |
| `output_vlan` | `100` / `[]` | RSPAN VLAN tag | Không trunk được nếu peer không hỗ trợ VLAN |
| `select_all` | `false` / `true` | Mirror toàn bridge traffic | `true` + high pps = output port flood |
| `select_src_port` | `[<uuid>]` | Mirror packet ingress trên port này | Empty + select_dst_port empty + select_all false = no traffic mirror |
| `select_dst_port` | `[<uuid>]` | Mirror packet egress trên port này | Combine với src_port = bidirectional mirror |
| `select_vlan` | `[100, 200]` | Mirror VLAN-matched traffic | Overlap với select_all → ambiguous |
| `snaplen` | `[]` / `64` / `128` | Truncate packet tới N byte (OVS 2.10+) | Unset = full packet, cost storage |
| `statistics:tx_packets` | `321` | Count đã mirror (monitor health) | 0 sau 1 phút = mirror không trigger (config issue) |

**Kịch bản bẻ gãy:**

- **`tx_packets=0` sau 1 phút dù traffic đi qua**: select_src/dst_port trỏ wrong port UUID. Verify qua `ovs-vsctl list Port | grep <expected-name>` lấy UUID đúng.
- **`output_port` + `output_vlan` cả 2 set**: invalid config. Mirror chỉ chọn 1 output. Fix: `ovs-vsctl set Mirror <name> output_vlan=[]`.
- **Analyzer port flood**: `select_all=true` với high-rate bridge → output port overwhelm. Fix: narrow selector + tăng snaplen=64.
- **RSPAN VLAN 100 không reach remote analyzer**: trunk peer switch không carry VLAN 100. Fix: ensure `switchport trunk allowed vlan add 100` trên upstream.

## 9.7.6 Hiểu sai + Key Topic

> **Hiểu sai phổ biến 1:** *"Mirror copy packet consume gấp đôi bandwidth fabric."*
>
> **Thực tế:** Local SPAN (output_port) không consume fabric bandwidth — packet copy stays trong hypervisor kernel memory, output qua internal port → tcpdump read từ same kernel buffer. Zero wire cost. RSPAN (output_vlan) consume bandwidth vì packet phải traverse uplink port ra trunk. Distinction quan trọng khi capacity planning.

> **Hiểu sai phổ biến 2:** *"Mirror-then-capture với `select_all` an toàn cho production."*
>
> **Thực tế:** `select_all=true` trên bridge nhiều port cause output port overwhelm. Ví dụ bridge 20 VM × 500 Mbps each = 10 Gbps aggregate. Output internal port không có rate limit built-in. tcpdump write-to-disk bottleneck khiến kernel drop mirrored packet → incomplete capture → debug trong bóng tối. Best practice: narrow selector trước (specific VM), giảm snaplen xuống 96-128 byte cho header-only.

> **Chủ đề trọng tâm:** Mirror là foundation cho multiple observability use case: (1) ad-hoc debug (tcpdump 1 VM khắc phục sự cố), (2) IDS feed (Suricata ingest RSPAN VLAN), (3) compliance audit (capture specific tenant traffic cho 90-day retention), (4) flow analysis offline (split pcap theo 5-tuple, correlate với app logs). Xem Phần 20.1 §20.1.13 port_security forensic cho compliance use case. Xem Phần 9.8 cho flow monitoring alternative (sFlow/IPFIX — sampled thay vì full capture, scale tốt hơn cho always-on monitoring).

## 9.7.7 Common failure modes

**Failure 1 — Mirror config exist nhưng statistics=0.**
Dấu hiệu: `tx_packets=0` sau 1 phút.
Nguyên nhân: Wrong port UUID trong select_src/dst_port.
Fix: Re-create mirror với atomic idiom `--id=@p1 get Port <correct-name>`.

**Failure 2 — Output analyzer port gets flooded.**
Dấu hiệu: `iftop -i analyzer` show > output_port capacity. tcpdump drops counter high.
Nguyên nhân: `select_all=true` trên high-traffic bridge.
Fix: Narrow selector + `snaplen=96` cho header-only.

**Failure 3 — Mirror table growth cộng dồn không cleanup.**
Dấu hiệu: `ovs-vsctl list Mirror | wc -l` show hundreds của Mirror row.
Nguyên nhân: Orchestrator tạo Mirror ad-hoc nhưng không cleanup. OVS tự GC khi unreferenced, nhưng Bridge.mirrors reference giữ alive.
Fix: Audit orchestrator cleanup logic. Manual cleanup: `ovs-vsctl clear Bridge <br> mirrors`.

**Failure 4 — RSPAN VLAN không reach remote.**
Dấu hiệu: Local mirror OK (statistics tăng), remote analyzer không thấy packet.
Nguyên nhân: Trunk peer switch không carry VLAN, hoặc VLAN blocked by VACL.
Fix: `show vlan brief` trên peer, add VLAN to trunk list.

## 9.7.8 Performance + retention planning

**Capacity planning cho capture:**

| Scenario | Traffic rate | Snaplen | Storage / hour |
|----------|-------------|---------|----------------|
| 1 VM, normal workload | 100 Mbps | full | 45 GB |
| 1 VM, normal workload | 100 Mbps | 96 bytes | 4.3 GB |
| 10 VM, 1 Gbps each | 10 Gbps | 96 bytes | 432 GB |
| Bridge-wide (50 VM) | 25 Gbps | 128 bytes | 1.3 TB |

**Retention:** Quarterly audit (90 day) cho 1 VM header-only = ~9 TB. Scale tuyến tính. Plan dedicated NVMe storage hoặc stream via `tcpdump -w -` + pipe to Kafka/remote log aggregator.

**Loss-less capture requirement:** Production incident response cần `snaplen=full` trong window 5-10 phút. Monitor tcpdump `dropped` counter — nếu tăng, buffer size insufficient. Tune: `tcpdump -B 10240` tăng buffer 10MB.

## Điểm cốt lõi cần nhớ

Thứ nhất, OVS Mirror table cung cấp SPAN/RSPAN native — không cần tool bên ngoài.

Thứ hai, atomic transaction `-- --id=@...` là cách chuẩn tạo Mirror — đảm bảo Bridge.mirrors reference đúng Mirror row trong một step.

Thứ ba, Mirror tự garbage-collect khi unreferenced — dùng `clear Bridge br0 mirrors` để disable.

Thứ tư, tcpdump trên internal port output + Wireshark/Suricata — foundation cho network visibility.

Thứ năm, Anatomy Template A `list Mirror` 9-attribute đọc trong 10 giây. `tx_packets=0` là red flag # 1 (mirror không trigger).

Thứ sáu, local SPAN zero wire cost, RSPAN consume uplink bandwidth. Snaplen 96-128 cho header-only giảm 10× storage.

Thứ bảy, Mirror-then-capture không miễn phí — capacity planning cần: 1 VM header-only = 4.3 GB/hour, full capture = 45 GB/hour. Scale tuyến tính theo bandwidth.

## 9.7.9 Backfill v3.6 — `ovs-tcpdump` shortcut wrapper

> **Bối cảnh.** Phase 3 v3.6-ContentDepth audit phát hiện `ovs-tcpdump` chỉ có trong master keyword index (0.3) mà chưa có Anatomy substantive ở Phần dạy capture. Section này thêm 5-axis treatment.

### 9.7.9.1 Vấn đề `ovs-tcpdump` giải quyết

Bình thường để capture traffic từ OVS internal port (tap, patch, vxlan tunnel, VM TAP), operator phải:

1. Tạo Mirror table entry với `select_src_port` + `output_port` trỏ sang một port "snooper".
2. Attach `tcpdump` lên port snooper.
3. Dọn Mirror sau khi xong.

Quy trình 3 bước này (xem §9.7.2 và §9.7.3) đáng giá cho debug dài hạn nhưng dư thừa cho trường hợp "kiểm tra nhanh 30 giây xem có ARP đi qua không". `ovs-tcpdump` là wrapper Python (`utilities/ovs-tcpdump.in` upstream) tự động hoá toàn bộ quy trình: khi gọi, nó tạo netdev port giả, set Mirror tự động, gọi `tcpdump` với BPF filter operator chỉ định, và cleanup ngay khi nhận tín hiệu kết thúc.

### 9.7.9.2 Anatomy `ovs-tcpdump`

**Bucket:** Capture utility (ad-hoc).
**Synopsis:** `ovs-tcpdump --mirror-to=<snoop> -i <ovs-port> [tcpdump options]`.
**Effect:** Tạo Mirror auto, attach tcpdump output ra terminal hoặc file, cleanup khi nhận `SIGINT/SIGTERM`. Tương đương 3 bước thủ công nhưng atomic.
**Prerequisites:** Python 3 + `tcpdump` cài sẵn trên cùng host. User cần quyền `ovs-vsctl` (root hoặc setuid). Bridge phải dùng kernel datapath (DPDK datapath không expose tap socket cho tcpdump theo cách này).
**OVS version intro:** OVS 2.10+ utility (Python wrapper).
**Side effect:** Mirror table có 1 entry tạm thời trong lúc capture; entry tự xoá sau khi process kết thúc. Nếu process bị `kill -9` không clean shutdown, Mirror entry stale lại trong DB và cần `ovs-vsctl --id=@m clear Bridge ... mirrors @m` thủ công.
**Example:**

```
[host]# ovs-tcpdump -i veth-vm1 -nn 'arp or icmp'
tcpdump: listening on miveth-vm1, link-type EN10MB (Ethernet), snapshot length 262144 bytes
14:33:01.001234 ARP, Request who-has 10.0.0.5 tell 10.0.0.1, length 28
14:33:01.001456 ARP, Reply 10.0.0.5 is-at 52:54:00:aa:bb:cc, length 28
14:33:02.123456 IP 10.0.0.1 > 10.0.0.5: ICMP echo request, id 1, seq 1, length 64
^C
2 packets captured, ARP entries cleaned up.
```

**Diễn giải vận hành.** Cờ `-i` ở đây nhận tên OVS port (giống cách `tcpdump -i eth0` nhận kernel netdev), không phải tên kernel netdev. Wrapper tạo virtual snoop port tên `mi<port>` (prefix `mi`) và Mirror entry điểm port gốc → snoop. Khi user `Ctrl-C`, wrapper xoá Mirror entry, xoá snoop port, gọi `ovs-vsctl` rollback. So với `tcpdump -i any`: `tcpdump -i any` không thấy traffic nội bộ giữa hai VM trên cùng OVS bridge (kernel không thấy gói qua datapath fast-path); `ovs-tcpdump` đặt mirror trong OVS nên thấy luôn. Anti-pattern: dùng `ovs-tcpdump` cho long-running capture (giờ tới ngày) vì Mirror entry duy trì có lock cost trong OVS classifier; preferred là Mirror entry persistent + `tcpdump` tách rời. Cross-link: §9.7.4 capture từ mirror output, §9.4 ovs-vsctl Mirror mutate atomic, §9.28 `ovs-pcap` đọc lại file đã capture.

## 9.7.10 Cờ `-i IFACE` — chỉ định interface cần soi

**Khái niệm.** Cờ `-i` (đồng nghĩa `--interface`) là cờ bắt buộc của `ovs-tcpdump`, nhận **tên OVS port** trong `Bridge.ports` (ví dụ `vm1`, `veth-br1`, `tap0`, `vxlan0`). Tên này phải tồn tại trong cột `name` của bảng `Interface`, kiểm chứng nhanh bằng `ovs-vsctl list-ports <bridge>`. Wrapper sẽ tự dò bridge chứa interface qua `ovs-vsctl iface-to-br <iface>`, tạo Mirror trên đúng bridge đó, không cần operator chỉ định bridge.

**Khác biệt then chốt với `tcpdump -i`.** `tcpdump -i eth0` đọc trực tiếp từ kernel netdev `eth0` qua AF_PACKET socket. Trong khi đó `ovs-tcpdump -i vm1` thiết lập Mirror trong OVSDB rồi attach `tcpdump` lên snoop port phụ trợ. Hệ quả: với hai VM cùng bridge OVS, `tcpdump -i any` không thấy traffic east-west do gói đi qua datapath fast-path không trồi lên kernel netdev nào; còn `ovs-tcpdump -i vm1` vẫn thấy đầy đủ. Đây là động cơ ra đời của wrapper.

**Cú pháp + ví dụ thực tế:**

```
[host]# ovs-vsctl list-ports br-int
veth-br1
veth-br2
tap-vm1

[host]# ovs-tcpdump -i tap-vm1 -nn -c 5
tcpdump: listening on mitap-vm1, link-type EN10MB (Ethernet), snapshot length 262144 bytes
14:45:01.001 IP 10.0.0.5 > 10.0.0.6: ICMP echo request, id 1, seq 1, length 64
14:45:01.002 IP 10.0.0.6 > 10.0.0.5: ICMP echo reply,   id 1, seq 1, length 64
... (3 dòng nữa)
5 packets captured
```

**Lỗi thường gặp.** (a) Truyền tên kernel netdev không nằm trong OVS, ví dụ `-i eth0` khi `eth0` chưa add-port. Wrapper trả `Cannot find OVS bridge for eth0`, fix bằng `ovs-vsctl add-port br0 eth0` trước. (b) Truyền interface name dài hơn 13 ký tự — Linux netdev giới hạn 15 ký tự cho `mi<name>`, wrapper sẽ fallback sang tên ngẫu nhiên `ovsmi%06d` thay vì `mi<name>`, nên đừng ngạc nhiên nếu thấy port lạ trong `ip link`. Cross-link: §9.4 `ovs-vsctl iface-to-br`, §9.7.2 atomic Mirror tạo thủ công.

## 9.7.11 BPF filter truyền vào — cú pháp + lý do không có OpenFlow filter

**Sự thật cần đính chính.** `ovs-tcpdump` **không có** cờ `-O` hay `--filter` riêng cho OpenFlow-style match. Wrapper chỉ là vỏ bọc Python quanh `tcpdump` thật, nên **mọi filter đều là BPF expression** (Berkeley Packet Filter, cú pháp `pcap-filter(7)`). Mọi token còn lại sau khi argparse xử lý xong các cờ wrapper riêng (`-i`, `--mirror-to`, `--span`, `--db-sock`, `--dump-cmd`) sẽ được forward nguyên dạng cho `tcpdump` qua list `tcpdargs`.

**Cú pháp BPF filter:** truyền như argument cuối cùng, có thể bọc trong dấu nháy đơn để shell không tách:

```
[host]# ovs-tcpdump -i veth-vm1 -nn 'tcp port 443 and host 10.0.0.5'
[host]# ovs-tcpdump -i veth-vm1 -nn 'arp or icmp'
[host]# ovs-tcpdump -i veth-vm1 -e 'vlan 100 and ether dst 52:54:00:aa:bb:cc'
```

BPF chạy trong kernel ngay sau snoop port, drop sớm các gói không match → giảm tải userspace ~10× cho high-rate flow.

**Khi nào cần "OpenFlow-style match" thay vì BPF?** Khi điều kiện không biểu diễn được bằng L2-L4 header (ví dụ "match `tun_id=0x100`" hay "match `metadata=0x42`" hay "match `reg14=2`" — đây là metadata nội bộ pipeline OVS, không có trong gói tin trên dây nên BPF không thấy). Lúc đó **không dùng `ovs-tcpdump`** mà dùng một trong hai cách: (1) tạo Mirror thủ công với `select_src_port`/`select_dst_port` chỉ định đúng port liên quan rồi `tcpdump -i <output>` (xem §9.7.2); (2) `ovs-appctl ofproto/trace` mô phỏng pipeline cho gói đại diện (xem Phần 9.25). Đừng kỳ vọng tìm cờ `-O` hay `--filter=of:...`, nó không tồn tại trong upstream `utilities/ovs-tcpdump.in` v2.17.9.

**Cross-link:** Phần 9.25 ofproto/trace cho metadata-aware simulation, §9.7.2 Mirror selector chính xác hơn BPF khi cần lọc theo port logic.

## 9.7.12 Cờ `--mirror-to=<NAME>` — đặt tên cho snoop port

**Khái niệm.** Mặc định wrapper tạo snoop port tên `mi<iface>` (ví dụ `-i veth-vm1` sẽ sinh port `miveth-vm1`). Nếu tên gốc dài hơn 13 ký tự (vượt giới hạn 15 ký tự netdev khi cộng prefix `mi`), wrapper rơi sang tên ngẫu nhiên dạng `ovsmi%06d` (sáu chữ số). Cờ `--mirror-to=<NAME>` cho phép operator **đặt tên cố định** thay vì để wrapper tự sinh. Hữu ích khi: (a) cần script hoá nhiều capture song song và phải biết trước tên; (b) muốn `ip link show <name>` hoặc `ethtool -S <name>` đọc counter snoop sau capture; (c) cần nhiều capture đồng thời cùng interface gốc, tránh trùng tên.

**Cú pháp + ví dụ:**

```
[host]# ovs-tcpdump -i tap-vm1 --mirror-to=snoop-prod --db-sock=unix:/var/run/openvswitch/db.sock 'icmp'
tcpdump: listening on snoop-prod, link-type EN10MB (Ethernet), ...

[host]# ip link show snoop-prod
42: snoop-prod: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 ...
```

**Side effect cần biết.** Khi capture kết thúc qua `Ctrl-C` (signal `SIGINT`), wrapper kích hoạt fatal-signal hook gọi `cleanup_mirror()` → xoá Mirror row + xoá snoop port. Nhưng nếu process bị `kill -9` (`SIGKILL`), hook không chạy được, port `snoop-prod` và Mirror row sẽ stale lại. Recovery thủ công: `ovs-vsctl --if-exists del-port snoop-prod && ovs-vsctl clear Bridge <br> mirrors`. Audit định kỳ bằng `ovs-vsctl list Mirror | grep -c '_uuid'` — số mirror tồn lâu > số session capture đang chạy là dấu hiệu rò rỉ.

Cross-link: §9.7.3 cleanup Mirror tổng quát, §9.7.7 Failure 3 (mirror table growth).

## 9.7.13 Cờ `--span` — soi toàn bộ port trên bridge

**Khái niệm.** Mặc định `ovs-tcpdump -i <iface>` chỉ tạo Mirror với `select_src_port` + `select_dst_port` trỏ đúng một port. Cờ `--span` thay đổi hành vi: thay vì single-port mirror, wrapper tạo Mirror với **`select_all=true`** trên bridge chứa interface đó. Toàn bộ traffic ingress + egress của mọi port trên bridge đều copy về snoop port. Tương đương SPAN session bridge-wide trên switch vật lý.

**Cú pháp:**

```
[host]# ovs-tcpdump -i veth-vm1 --span -nn -c 20
# Mặc dù chỉ định -i veth-vm1, --span override → mirror cả br-int
# Cần veth-vm1 chỉ để wrapper biết bridge nào, không phải để giới hạn nguồn
```

**Khi nào dùng.** (a) Lab học tập: muốn nhìn tổng thể conversation trên một bridge nhỏ 3-5 port. (b) Incident response giai đoạn đầu, chưa biết VM nào liên quan, capture cả bridge rồi filter offline qua Wireshark. (c) Compliance audit yêu cầu capture đầy đủ một tenant (nếu tenant đó độc chiếm bridge).

**Cảnh báo capacity.** `--span` trên bridge production high-traffic gây **output port flood** y như `select_all=true` thủ công (xem §9.7.6 hiểu sai phổ biến 2). Ví dụ bridge 20 VM × 500 Mbps = 10 Gbps copy về snoop, vượt khả năng userspace `tcpdump` ghi đĩa → kernel drop, capture không đầy đủ. Mitigation: kèm BPF filter narrow + tăng buffer `tcpdump -B 10240` + cân nhắc snaplen nhỏ (mặc định ovs-tcpdump không truncate, dùng `-s 96` để giới hạn 96 byte header). Nếu capture > 30 giây, cân nhắc Mirror persistent + `tcpdump` chuyên dụng thay vì `ovs-tcpdump`.

Cross-link: §9.7.6 hiểu sai về `select_all=true`, §9.7.8 capacity planning.

## 9.7.14 Định dạng pcap output + tương thích libpcap

**Khái niệm.** `ovs-tcpdump` không tự định nghĩa format riêng — nó forward toàn bộ trách nhiệm format cho `tcpdump` thật phía sau. Khi operator thêm cờ `-w <file>` (truyền cho `tcpdump`), file output là **pcap chuẩn libpcap**: header 24 byte (magic number `0xa1b2c3d4` little-endian hoặc `0xd4c3b2a1` big-endian + version major/minor + thiscolzone + sigfigs + snaplen + network link-type), tiếp theo là chuỗi record `(ts_sec, ts_usec, incl_len, orig_len)` 16 byte + raw frame.

**Tính tương thích.**

- File này đọc lại được trực tiếp bởi `wireshark`, `tshark`, `tcpdump -r`, `tcpreplay`, Suricata, Zeek (Bro), Moloch/Arkime.
- Đọc được bởi `ovs-pcap` (xem Phần 9.28) để convert sang hex single-line phục vụ `ovs-appctl ofproto/trace`.
- Link-type mặc định là `LINKTYPE_ETHERNET` (`DLT_EN10MB`, value 1), khớp với mọi tool L2 chuẩn.

**Ví dụ workflow:**

```
[host]# ovs-tcpdump -i tap-vm1 -w /tmp/capture.pcap -nn 'tcp port 443'
^C
12 packets captured

[host]# file /tmp/capture.pcap
/tmp/capture.pcap: pcap capture file, microsecond ts (little-endian) - version 2.4
                   (Ethernet, capture length 262144)

[host]# wireshark /tmp/capture.pcap   # mở GUI

[host]# ovs-pcap /tmp/capture.pcap | head -1
ffffffffffff525400aabbcc0800450000... (hex single-line)

[host]# ovs-appctl ofproto/trace br-int "$(ovs-pcap /tmp/capture.pcap | head -1)"
Flow: ... dl_type=0x0800,nw_src=10.0.0.5, ...
```

**Lưu ý version.** Pcap header version `2.4` là chuẩn libpcap từ 2008, không thay đổi giữa OVS 2.x và 3.x. Một số tool mới hơn dùng pcapng (`.pcapng`, magic `0x0a0d0d0a`) — `ovs-tcpdump` chỉ xuất pcap classic, không xuất pcapng, do pass-through `tcpdump` mặc định cũng xuất classic. Nếu cần pcapng cho metadata phong phú hơn (interface description block, comment block), dùng `tshark -F pcapng` đọc lại file rồi re-encode, hoặc bỏ qua wrapper, dùng Mirror persistent + `tshark -i <snoop> -w out.pcapng`.

Cross-link: §9.28 `ovs-pcap` dùng cùng định dạng, §9.7.4 capture mirror output trực tiếp, libpcap project file format spec.

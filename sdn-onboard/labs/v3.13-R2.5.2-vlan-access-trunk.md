This lab proves, on real wires, what every traditional Cisco-trained or
Juniper-trained network engineer already knows in muscle memory: a port can be
configured as **access** (one VLAN, untagged on the wire) or as **trunk**
(multiple VLANs, 802.1Q tagged on the wire); access ports are isolated from
each other when they belong to different VLANs; trunk ports preserve the VLAN
identity across links so two switches can extend the same broadcast domain
through one cable. The lab does this twice. Once on Open vSwitch using the
OVSDB columns `Port.tag`, `Port.trunks`, and `Port.vlan_mode`. Once on the
Linux kernel bridge using `bridge vlan add` with the `pvid` and `untagged`
flags. The point is for the reader, who has spent years configuring
`switchport access vlan 10` on Cisco IOS, to see exactly which cells of that
familiar mental model map to which OVS column, which `bridge` flag, and which
line of OVS source code. The traditional knowledge survives the transition; the
configuration surface, the data structure, and the implementation language
change.

## R2.5.2.1. Where this part sits in the curriculum and why

R2.5.2 is the second of twelve sub-captures inside R2.5 of plan v3.13. R2.5
exists because, between the bridge-and-port primitives covered in R2 and the
OpenFlow programming covered in R3, there is a comparative layer the engineer
must internalise: every primitive a traditional switch implements in its ASIC
plus management plane, OVS implements somewhere too, but in a different place
and with a different surface. R2.5.1 covered L2 learning (the MAC table). R2.5.2
covers VLAN access and trunk. R2.5.3 (next sub-capture) will cover STP and
RSTP. The motivation, per the audience-and-pedagogy mandate codified as Rule 19
of CLAUDE.md, is that the reader is a traditional network engineer learning OVS
for the first time. The lab is written with that audience in mind.

The most important fact to anchor before any commands run: an OVS port has at
most one of `tag` or `trunks` set, and the column `vlan_mode` is by default
empty in the OVSDB. The effective vlan mode is **derived** from those columns.
A port with `tag=10` and no `trunks` is an access port for VLAN 10. A port with
`trunks=[10,20]` and no `tag` is a trunk carrying VLANs 10 and 20. A port with
both `tag=10` and `trunks=[10,20]` is a special hybrid mode controlled by an
explicit `vlan_mode` value (`native-tagged` or `native-untagged`). This last
form is what a Cisco engineer recognises as the **native VLAN** on a trunk
port; the lab in R2.5.2.6 reproduces that configuration in OVS and observes the
wire bytes change.

## R2.5.2.2. Topology, both bridges side by side

The lab builds two parallel topologies, one per bridge implementation. The OVS
side comes up first, runs through five guided exercises, then the OVS side is
configured back to its baseline and the Linux bridge side comes up next for
three guided exercises that mirror the OVS measurements. The two topologies
share the conceptual layout below.

```ascii
   ns_red                         ns_blue                       ns_trunk
   10.0.10.1/24                   10.0.20.1/24                  veth_trunk.10  10.0.10.2/24
                                                                veth_trunk.20  10.0.20.2/24

       |                              |                                 |
   [veth_red]                    [veth_blue]                       [veth_trunk]
       |                              |                                 |
   ----+------------------------------+---------------------------------+----
       |                              |                                 |
   veth_red_br                  veth_blue_br                       veth_trunk_br
   tag=10                       tag=20                             trunks=[10,20]
   (access port)                (access port)                      (trunk port)
       |                              |                                 |
       +------------+-----------------+-----------------+---------------+
                    |                                   |
                    |      OVS bridge br0 (or           |
                    |   Linux bridge lbr0 on the        |
                    |    second half of this lab)       |
                    +-----------------------------------+
```

The three netns peers all live on the same lab host. Each ports its veth tail
into a separate network namespace; the host-side veth tail attaches to the
bridge. ns_red and ns_blue are pure access-port hosts: they speak untagged
Ethernet on their interface, so they look exactly like a printer or a desktop
plugged into a vlan 10 access port on a Cisco switch. ns_trunk is a trunk-side
host: it has two 802.1Q sub-interfaces (`veth_trunk.10` and `veth_trunk.20`)
on top of the underlying `veth_trunk`. Each sub-interface tags its outgoing
frames with the corresponding VID and accepts incoming frames only if they
carry the matching tag. This mirrors the way a downstream switch's trunk port
is configured on the wire: the port carries 802.1Q-tagged frames and the
switching plane on the receiver side demultiplexes by tag.

The IP addressing is deliberately chosen so the two access ports cannot reach
each other through plain L3 even if the L2 isolation were broken. ns_red is
10.0.10.1/24, ns_blue is 10.0.20.1/24. Both subnets are local to the access
port's namespace, but ns_red has no route to 10.0.20.0/24 and ns_blue has no
route to 10.0.10.0/24. The cross-VLAN ping test (exercise 4 below) demonstrates
this with `ping_exit=2` ("Network is unreachable"), which is a stronger
empirical signal than ARP-failure: it proves the kernel routing table itself
records the absence of any path between the two subnets, not just that the
broadcast domain is partitioned. A cross-VLAN ping reaching ARP and failing
there would also be valid evidence; the route-failure form is just stricter.

The Linux bridge half of the lab (R2.5.2.7 onward) uses the same conceptual
layout but with hostnames prefixed `l` (so `ns_lr` is the Linux red, `ns_lb` is
the Linux blue, `ns_lt` is the Linux trunk). Subnets shift to 10.1.10.0/24 and
10.1.20.0/24 so the two bridges' traffic is unambiguous. Both bridges run on
the same lab host but never carry each other's traffic; they are separate
broadcast domains.

## R2.5.2.3. The OVS configuration model: three columns, four modes

Open vSwitch encodes VLAN-port configuration in three columns of the OVSDB
`Port` row: `tag`, `trunks`, and `vlan_mode`. The relationship between the
three is documented in `Documentation/faq/vlan.rst` of the upstream OVS source
tree (commit `0bea06d9957e3966d94c48873cd9afefba1c2677`, the v2.17.9 baseline)
and is also visible in
`vswitchd/bridge.c:port_configure()`, which reads these three columns from
OVSDB and pushes them into the ofproto's `bundle_set` call. The four resolved
modes are:

| OVSDB columns set                       | Resolved `vlan_mode` | Behaviour                                           |
|-----------------------------------------|----------------------|-----------------------------------------------------|
| `tag=N`, `trunks=[]`, `vlan_mode=[]`    | `access`             | Untagged on wire, internal VID = N. One VLAN only.  |
| `tag=[]`, `trunks=[N,M,...]`, `vlan_mode=[]` | `trunk`         | Tagged on wire, only listed VIDs forwarded.         |
| `tag=N`, `trunks=[N,...]`, `vlan_mode=native-tagged` | `native-tagged` | Trunk that also carries native VID N tagged.         |
| `tag=N`, `trunks=[N,...]`, `vlan_mode=native-untagged` | `native-untagged` | Trunk that carries native VID N **untagged** on wire. |

The fourth row is the Cisco-native-VLAN equivalent: a trunk port whose native
VLAN is sent without an 802.1Q tag on egress (`native-untagged`) or with a tag
(`native-tagged`). This is exactly the mental model from `switchport trunk
native vlan 1` on Cisco IOS combined with the `vlan dot1q tag native` global
toggle. The fifth mode `dot1q-tunnel` (QinQ) exists in OVS source as
`PORT_VLAN_DOT1Q_TUNNEL` and would push a second VLAN tag for QinQ stacking;
this lab does not exercise it because it is rarely configured on the
single-host setups the curriculum targets.

The translation engine that resolves these modes lives in
`ofproto/ofproto-dpif-xlate.c`. Two functions matter most. First,
`xvlan_input_translate` (line 2450 of the v2.17.9 source) is called when a
frame arrives on a port and decides what its **internal VID** is for the
purposes of MAC learning, ACL matching, and FDB lookups. The function
`switch`-es on the resolved vlan_mode and either (case `PORT_VLAN_ACCESS`) sets
the internal VID from the port's `tag` ignoring whatever was on the wire, or
(case `PORT_VLAN_TRUNK`) accepts the wire VID as-is, or (case
`PORT_VLAN_NATIVE_*`) accepts the wire VID if present and otherwise substitutes
the port's `tag`. Second, `xvlan_output_translate` (line 2495) is called when
the frame is leaving and decides what the **wire VID** is on egress. For
access ports the function clears the VID (so the wire is untagged); for trunk
ports it leaves the VID intact; for native-untagged it pops the tag if the VID
matches the port's native; for native-tagged it leaves the tag in place.

The kernel datapath sees these decisions as concrete `push_vlan(vid=N,pcp=0)`
and `pop_vlan` actions on its megaflow rules. Exercise 5 below dumps the
megaflows live and demonstrates the action sequence. The kernel-side
implementation lives in `datapath/actions.c:do_push_vlan()` and `do_pop_vlan()`,
which manipulate the skb's VLAN header before the packet is sent on egress.

### ▶ Guided Exercise 1: Build the OVS bridge with three ports and configure their VLAN modes

The first exercise builds the OVS topology and reads back the OVSDB to confirm
what was configured. The pedagogical point is that `tag` and `trunks` are
**columns of the Port row**, not arguments to a special command. Anything that
can read OVSDB can read the configuration; anything that can write OVSDB can
change it. The only reason `ovs-vsctl set port` exists is convenience.

**Outcomes.**

- Three OVS ports configured: one access in vlan 10, one access in vlan 20,
  one trunk carrying both vlans 10 and 20.
- Three Linux network namespaces attached to the corresponding peer ends.
- The OVSDB `Port` rows show `tag` and `trunks` fields populated correctly.
- The `vlan_mode` column is empty (`[]`) by default, even though the resolved
  mode is well-defined.

**Before You Begin.** OVS 2.17.9 daemons running on `lab-openvswitch` with
no bridges configured. The kernel module `openvswitch` is loaded.

**Instructions.**

**1.** Build the OVS bridge `br0` and three veth pairs. Move one tail of each
veth pair into a dedicated namespace. Attach the host-side tails as ports on
the bridge.

```text
root@lab-openvswitch:~/network-onboard# ip netns add ns_trunk

root@lab-openvswitch:~/network-onboard# ip link add veth_blue type veth peer name veth_blue_br
```

**2.** Set `tag=10` on `veth_red_br`, `tag=20` on `veth_blue_br`, and

The three rows are exactly what was written: `veth_red_br` has `tag=10` and
nothing else, `veth_blue_br` has `tag=20` and nothing else, `veth_trunk_br`
has `trunks=[10, 20]` and nothing else. `vlan_mode` is `[]` (empty) on all
three rows. This is a critical observation. The empty `vlan_mode` does not
mean "no VLAN handling"; it means "the default mode is implied by which of
`tag` and `trunks` is populated." This is documented in
`vswitchd/vswitch.ovsschema` and surfaces in the `port_configure()` function
in `vswitchd/bridge.c` which derives the mode from the columns at runtime.

**6.** Confirm the same observation a second way, by reading `vlan_mode`
directly with `ovs-vsctl get`.

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl get port veth_red_br vlan_mode
[]
root@lab-openvswitch:~/network-onboard# ovs-vsctl get port veth_blue_br vlan_mode
[]
root@lab-openvswitch:~/network-onboard# ovs-vsctl get port veth_trunk_br vlan_mode
[]
```

The `[]` is the OVSDB string representation of an empty optional set; for
`vlan_mode` this means the column has no value set and the resolved mode is
derived. Cisco does not have an analogous concept; on Cisco IOS, every interface
is in either `switchport mode access` or `switchport mode trunk` or
`switchport mode dynamic`, and the explicit value can always be read. The OVS
abstraction is more compact (two columns instead of one explicit one) but
slightly less self-documenting; the reader has to know that an empty
`vlan_mode` plus `tag=N` resolves to access. The upstream FAQ at
`Documentation/faq/vlan.rst` is the canonical reference.

### ▶ Guided Exercise 2: Verify intra-VLAN reachability

The first connectivity claim is that ns_red (access vlan 10) reaches the
matching `veth_trunk.10` sub-interface in ns_trunk (trunk vlan 10). The
expected behaviour is that an ICMP echo travels: untagged on the wire from
ns_red into `veth_red_br`, gets internal VID 10 assigned by the OVS NORMAL
action, gets pushed back onto the wire as `vlan 10` on egress to `veth_trunk_br`,
arrives at `veth_trunk` in ns_trunk, gets demultiplexed by the kernel's vlan
device into `veth_trunk.10` (because the tag matches), and is delivered to the
local IP stack. The reply traverses the reverse path. The pedagogical point of
this exercise is that **the access port hides the VLAN entirely from the host
in ns_red**; the host has no idea its frames are being tagged inside the
bridge.

**Outcomes.** ns_red can ping 10.0.10.2 (the IP of `veth_trunk.10` in
ns_trunk).

**Before You Begin.** Exercise 1 has completed; the OVS topology is up.

**Instructions.**

**1.** Send two ICMP echoes from ns_red to 10.0.10.2.

```text
root@lab-openvswitch:~/network-onboard# ip netns exec ns_red ping -c 2 -W 2 10.0.10.2
PING 10.0.10.2 (10.0.10.2) 56(84) bytes of data.
64 bytes from 10.0.10.2: icmp_seq=1 ttl=64 time=0.586 ms
64 bytes from 10.0.10.2: icmp_seq=2 ttl=64 time=0.051 ms

--- 10.0.10.2 ping statistics ---
2 packets transmitted, 2 received, 0% packet loss, time 1024ms
rtt min/avg/max/mdev = 0.051/0.318/0.586/0.267 ms
```

The 0% packet loss is the empirical confirmation. The first echo took 0.586 ms
because it included the ARP resolution round-trip (ns_red has to learn
veth_trunk.10's MAC for 10.0.10.2 the first time); the second echo is the
steady-state RTT at 0.051 ms, which is the time to traverse two veth pairs
plus the OVS userspace fast path. Note that `ip netns exec` is required here
rather than `ip -n ns_red ping`. The `-n` flag of `ip(8)` only switches
namespace for `ip`'s own sub-commands (`link`, `addr`, `route`, etc.); `ping`
is a separate binary, so the namespace switch must be done via the explicit
`ip netns exec NS COMMAND` form. This was an empirical correction during the
capture: an earlier attempt with `ip -n ns_red ping ...` returned `Object
"ping" is unknown, try "ip help".`. The lab orchestrator was fixed and the
capture re-run.

**Finish.** The intra-VLAN ping passes. ns_red and the .10 sub-interface in
ns_trunk are in the same broadcast domain.

### ▶ Guided Exercise 3: Verify cross-VLAN isolation

The second connectivity claim is the inverse: ns_red (access vlan 10) cannot
reach ns_blue (access vlan 20). The expected outcome is that the ping fails;
the more interesting question is *why* it fails and *how the engineer reads
the failure to confirm it is VLAN-related and not, say, a typo in the IP
address.

**Outcomes.** ns_red cannot reach 10.0.20.1, and the failure mode tells the
engineer the path is broken at the routing-table layer (no route), not at
the L2 layer (broadcast-domain partition).

**Before You Begin.** Exercise 2 succeeded.

**Instructions.**

**1.** Predict, before running the command, what would happen if the VLAN
isolation did not exist. If the OVS bridge were forwarding frames between vlan
10 and vlan 20 freely, then ns_red would resolve the MAC of 10.0.20.1 via ARP
and ICMP would flow normally. The cross-VLAN ping would succeed.

The counter-hypothesis is "VLAN tagging is metadata only, the bridge forwards
the bytes anyway." Counter-hypothesis prediction: "0% packet loss" on the
cross-VLAN ping.

**2.** Send the cross-VLAN ping. Use a one-second deadline so the
guaranteed-failure path does not waste time.

```text
root@lab-openvswitch:~/network-onboard# ip netns exec ns_red ping -c 2 -W 1 10.0.20.1; echo ping_exit=$?
ping: connect: Network is unreachable
ping_exit=2
```

The actual result is `ping: connect: Network is unreachable` and `ping_exit=2`.
The counter-hypothesis prediction (success with 0% loss) is rejected; the VLAN
isolation is empirically real.

**3.** Read the failure mode carefully. The ping never even reached the wire.
The string `Network is unreachable` is the kernel's `ENETUNREACH` error
returned by the ns_red routing table because `10.0.20.0/24` is not in any of
its routes. The exit code 2 is `ping(8)`'s convention for "destination
unreachable per the routing layer."

This is a stronger empirical signal than ARP-failure (`Destination Host
Unreachable`) would be. ARP-failure would prove only that the broadcast domain
is partitioned; route-failure proves that the kernel itself recognises the
absence of any path. The reason ns_red has no route to 10.0.20.0/24 is that
its only configured IP is 10.0.10.1/24. We never gave it a 10.0.20.0/24 route
because, in the access-port mental model, ns_red is supposed to think it lives
in a single broadcast domain. The fact that the routing table reflects this
single-domain assumption is exactly the right outcome.

**Misconception to refute.** A traditional Cisco engineer might think the OVS
VLAN tag is just a label and the bridge forwards the bytes regardless. Reality:
the OVS NORMAL action consults the resolved vlan mode of both the input port
and the candidate output ports; the FDB (MAC table) is also keyed by (MAC, VID)
not just MAC. A frame ingressing on access vlan 10 is given internal VID 10;
flooding the frame on the bridge considers only ports whose vlan mode allows
VID 10 (so vlan 20 access ports are excluded). This mirrors the IEEE 802.1Q
ingress-and-egress filtering rules.

**Finish.** Cross-VLAN isolation is empirically confirmed at the routing-table
layer.

### ▶ Guided Exercise 4: Observe 802.1Q tagging on the trunk port

The third connectivity claim is that the trunk port carries 802.1Q-tagged
frames on the wire, while the access port carries untagged frames. This is the
canonical empirical evidence that the VLAN tag is **not just metadata in
OVS**: it actually appears in the bytes that travel between OVS and the host
in ns_trunk.

**Outcomes.** A tcpdump capture on `veth_trunk_br` (the host-side trunk
interface) shows ICMP and ARP frames carrying `vlan 10` 802.1Q headers; a
parallel capture on `veth_red_br` (the host-side access interface) shows the
same ICMP echo replies WITHOUT a 802.1Q header.

**Before You Begin.** Exercise 2 confirmed connectivity. The ARP cache is
populated.

**Instructions.**

**1.** Flush the ARP cache so the upcoming ping refreshes it (this gives the
tcpdump capture both ARP and ICMP traffic).

```text
root@lab-openvswitch:~/network-onboard# ip -n ns_trunk neigh flush all; ip -n ns_red neigh flush all
```

**2.** Start a packet capture on `veth_trunk_br`. Capture inbound-only
(`-Q in`) up to 6 packets, with an 8-second wall-clock cap, and write the
output to a temp file.

```text
root@lab-openvswitch:~/network-onboard# timeout 8 tcpdump -i veth_trunk_br -e -n -c 6 -Q in icmp or arp >/tmp/r2.5.2-trunk.pcap 2>&1 &
[1] 1203222
```

NOT in the allowed list as tagged, so the receiver expects 10 to arrive
untagged and to be implicitly assigned VID 10 by the receiver. The OVS
documentation at `Documentation/faq/vlan.rst` calls this out.

## R2.5.2.4. The Linux bridge equivalent: `vlan_filtering 1` and `bridge vlan add`

The Linux kernel bridge has supported per-port VLAN configuration since 3.8
(2013-02). Before that, the Linux bridge was a flat L2 device that ignored
VLANs entirely (any 802.1Q tag was carried through as opaque bytes to the
upper layers). The `vlan_filtering` flag on the bridge enables per-port VLAN
membership filtering. The VLAN configuration surface is the `bridge vlan add`
sub-command of `bridge(8)`, which is part of the iproute2 package. The kernel
implementation is in `net/bridge/br_vlan.c`.

The mental model maps cleanly to OVS:

| OVS column                     | Linux bridge equivalent                                  |
|--------------------------------|----------------------------------------------------------|
| `Port.tag=N` (access)          | `bridge vlan add vid N dev veth pvid untagged`           |
| `Port.trunks=[N,M]` (trunk)    | `bridge vlan add vid N dev veth; bridge vlan add vid M dev veth` |
| `Port.vlan_mode=native-untagged tag=N` | `bridge vlan add vid N dev veth pvid untagged` (with other VLANs as plain `add`) |
| `Port.vlan_mode=native-tagged tag=N` | `bridge vlan add vid N dev veth pvid` (without `untagged`) |

The `pvid` flag corresponds to the IEEE 802.1Q "Port VLAN ID": the VID assigned
to untagged frames arriving on the port. The `untagged` flag means egress is
also untagged for this VID. Setting both `pvid untagged` on a single VID
makes the port behave like an access port for that VLAN. Adding a VID
without `pvid untagged` adds the VID as a tagged trunk member.

The Linux bridge's default behaviour without `vlan_filtering 1` is to act as a
"plain bridge": all ports are members of all VLANs implicitly, no filtering
is applied, and the 802.1Q tag is passed through unchanged. With
`vlan_filtering 1`, the bridge enforces ingress-and-egress filtering per the
802.1Q rules. Each port's VLAN membership is the set of VIDs added with
`bridge vlan add` on that port.

There is one Linux-bridge-specific surprise the lab will surface in exercise
7. When `vlan_filtering 1` is enabled, every port (including the bridge's own
local port) gets VID 1 implicitly assigned with `pvid untagged`. This is the
802.1Q default VID. The lab explicitly removes VID 1 from the three veth
ports before adding VLANs 10 and 20, so the membership lists are clean.

## R2.5.2.5. The FDB compared: OVS userspace, Linux bridge kernel

The lab's last verification step compares the two implementations' MAC
forwarding tables. The point is to show, side by side, the **same conceptual
data** (which MAC was learned on which port in which VLAN) stored in two
different places: OVS keeps it in the `ovs-vswitchd` userspace daemon's heap
(the `mac_learning` struct from `lib/mac-learning.c`), and Linux bridge keeps
it in the kernel's per-bridge FDB hash table (the `net_bridge_fdb_entry`
struct from `net/bridge/br_fdb.c`).

```text
root@lab-openvswitch:~/network-onboard# ovs-appctl fdb/show br0
 port  VLAN  MAC                Age
    3    20  6a:e1:f7:e1:04:d1   30
    2    20  e2:10:43:fa:d2:bd   12
    1    10  56:a6:87:c1:15:9c    9
    3    10  6a:e1:f7:e1:04:d1    7
```

OVS reports four entries. The MAC `6a:e1:f7:e1:04:d1` (ns_trunk's
`veth_trunk`) appears twice, once under VLAN 20 (port 3, age 30s) and once
under VLAN 10 (port 3, age 7s). This is the canonical (MAC, VID) keying:
the same physical MAC is a different table entry per VLAN. `e2:10:43:fa:d2:bd`
is ns_blue's MAC, learned via vlan 20 traffic on port 2. `56:a6:87:c1:15:9c`
is ns_red's MAC, learned via vlan 10 traffic on port 1. The Age column is
the seconds since the entry was last refreshed.

```text

01:00:5e:00:00:6a dev lbr0 self permanent
33:33:00:00:00:6a dev lbr0 self permanent
01:00:5e:00:00:01 dev lbr0 self permanent
```

The Linux bridge FDB is more verbose. Three categories appear, exactly the
same three categories observed in R2.5.1: (1) permanent multicast entries
(33:33:* for IPv6 multicast, 01:00:5e:* for IPv4 multicast); (2) permanent

## R2.5.2.6. Side by side: configuration surfaces, kernel state, source code

| Concern                       | Cisco IOS                                              | OVS                                                       | Linux bridge (vlan_filtering 1)              |
|-------------------------------|--------------------------------------------------------|-----------------------------------------------------------|----------------------------------------------|
| Configure access port         | `switchport mode access; switchport access vlan 10`    | `ovs-vsctl set port P tag=10`                             | `bridge vlan add vid 10 dev P pvid untagged` |
| Configure trunk               | `switchport mode trunk; switchport trunk allowed vlan 10,20` | `ovs-vsctl set port P trunks=10,20`                | `bridge vlan add vid 10 dev P; bridge vlan add vid 20 dev P` |
| Configure native VLAN (untagged) | `switchport trunk native vlan 10`                  | `ovs-vsctl set port P vlan_mode=native-untagged tag=10`   | `bridge vlan add vid 10 dev P pvid untagged` (with other VIDs as plain) |
| Read VLAN membership          | `show interfaces switchport`                            | `ovs-vsctl --columns=tag,trunks,vlan_mode list port P`    | `bridge vlan show`                           |
| Read MAC table                | `show mac address-table`                                | `ovs-appctl fdb/show br0`                                 | `bridge fdb show`                            |
| State storage                 | ASIC TCAM plus management-plane SDK                     | `ovs-vswitchd` userspace heap (`mac_learning`)            | Kernel hash table (`net_bridge_fdb_entry`)   |
| Implementation language       | C plus closed-source vendor SDK                         | Userspace C plus kernel C                                  | Kernel C only                                |
| Source-code anchor (config plane) | n/a (closed)                                       | `vswitchd/bridge.c:port_configure` (line 1193)            | `net/bridge/br_vlan.c:br_vlan_add`           |
| Source-code anchor (data plane) | n/a (closed)                                         | `ofproto/ofproto-dpif-xlate.c:xvlan_input_translate` (2450), `xvlan_output_translate` (2495), plus `datapath/actions.c:do_push_vlan/do_pop_vlan` | `net/8021q/vlan_core.c:vlan_do_receive`, plus `net/bridge/br_vlan.c:br_vlan_filter_skb` |
| Filtering rules per 802.1Q-2018 | Yes                                                   | Yes                                                       | Yes (when `vlan_filtering 1`)               |

The last row is the most important. All three implementations conform to the
same IEEE 802.1Q-2018 standard for ingress and egress filtering. The
on-wire bytes of a tagged or untagged frame are byte-identical regardless of
whether the bridge is a Cisco Catalyst, an OVS instance, or a Linux kernel
bridge with `vlan_filtering` enabled. This is the physical-layer
interoperability that makes mixing and matching feasible. A Cisco trunk port
can connect to an OVS trunk port with `trunks=[10,20]`, and the two ends
agree on what's on the wire.

The differences live at three layers above the wire. First, the
configuration surface: each implementation has its own command syntax for
expressing per-port VLAN membership. Second, the data-structure layer: each
implementation stores the MAC forwarding table in a different place
(ASIC versus userspace heap versus kernel hash). Third, the implementation
language and toolchain: Cisco's is closed proprietary, OVS's is upstream
open C, Linux bridge's is in-tree kernel C. An engineer who has internalised
the 802.1Q wire format can move between all three without re-learning the
protocol; each just expresses the same idea in its own dialect.

## R2.5.2.7. Common misconceptions for the audience

The R2.5.2 audience is a Cisco-trained or Juniper-trained engineer
encountering OVS for the first time. Five misconceptions are likely to
travel with that audience and need explicit refutation:

**Misconception 1.** "An access port is a trunk with one allowed VLAN." Reality:
on the wire, the two are different. A trunk port (in the absence of a native
VLAN configuration) tags every frame with 802.1Q. An access port sends
frames untagged. The wire bytes differ; this lab's exercise 4 captured the
difference. A Cisco access port and a Cisco trunk-with-one-allowed-vlan
behave differently on the wire too: the trunk-with-one-allowed-vlan tags the
frame, the access port does not.

**Misconception 2.** "OVS `vlan_mode=trunk` means the same as `switchport mode
trunk`." Partially. OVS `vlan_mode=trunk` is the *implicit* mode of a port
with `trunks=[N,M,...]` and no `tag`; the OVSDB column is empty by default
because the mode is derived. Cisco's `switchport mode trunk` is an *explicit*
mode that turns DTP off and configures trunk negotiation behaviour. The wire
behaviour is the same; the configuration semantics are different.

**Misconception 3.** "If a frame arrives on an access port with a 802.1Q tag,
OVS will forward it tagged." Wrong. The OVS NORMAL action calls
`xvlan_input_translate` with the resolved port mode set to `PORT_VLAN_ACCESS`;
the input case for access overwrites the wire VID with the port's `tag`
column, ignoring whatever was on the wire. The frame is internally treated as
VID = the port's `tag`. On egress the tag may or may not appear; that is
controlled by the egress port's mode. So a tagged frame ingressing on an
access port is treated as if it had no tag; the wire tag is discarded.

**Misconception 4.** "The Linux bridge does not support VLANs." Wrong since
Linux 3.8 (2013-02). The `vlan_filtering 1` flag enables full 802.1Q-aware
behaviour. Many older blog posts predate this feature and recommend using
802.1Q sub-interfaces (`eth0.10`, `eth0.20`) above a flat bridge as a
workaround; this is no longer the canonical way and is not what
production-grade Linux networking does in 2026.

**Misconception 5.** "OVS pushes the VLAN tag on ingress, not on egress." Wrong.
The kernel datapath megaflow rules in exercise 5 made this explicit: the rule
matching ingress on an access port has `actions: push_vlan(vid=10,pcp=0),
veth_trunk_br`. The push happens as part of the action sequence executed on
the way to the output port. There is no notion of an "internally tagged"
representation in the kernel skb; the kernel's `vlan_tci` field on the skb
holds the tag conceptually during the action sequence but is materialised on
the wire only by the `push_vlan` action immediately before output. This is a
performance-driven design: the kernel avoids touching the packet bytes more
than once per pipeline.

## R2.5.2.8. Source-code references for the engineer who wants to dig deeper

The functions that implement VLAN handling in OVS at the v2.17.9 baseline
(commit `0bea06d9957e3966d94c48873cd9afefba1c2677`) are:

- `vswitchd/bridge.c:port_configure` (line 1193). Reads `Port.tag`,
  `Port.trunks`, `Port.vlan_mode` from OVSDB and assembles the per-port
  configuration that gets pushed into the ofproto provider's `bundle_set`
  call. This is the boundary between the management plane (OVSDB) and the
  forwarding plane (ofproto-dpif).
- `ofproto/ofproto-dpif-xlate.c:xvlan_input_translate` (line 2450). Computes
  the internal VID from a frame ingressing on a port. Switch over the four
  modes (access, trunk, native-tagged, native-untagged) plus the QinQ mode.
- `ofproto/ofproto-dpif-xlate.c:xvlan_output_translate` (line 2495). Computes
  the egress wire VID from the internal VID at output time. Symmetric to
  input.
- `ofproto/ofproto-dpif-xlate.c:input_vid_is_valid`. Validates that the wire
  VID on a frame matches one of the port's allowed VIDs (the `trunks` list
  for trunk ports, the `tag` for access ports). Frames with disallowed VIDs
  are dropped.
- `lib/vlan-bitmap.h`. Helpers for bitmap-encoded VID sets used by the trunk
  configuration.
- `datapath/actions.c:do_push_vlan` and `do_pop_vlan`. Kernel-side action
  implementations. These adjust the skb data pointer plus the `vlan_tci`
  field to materialise the tag insertion or removal at the byte level.
- `datapath/linux/compat/include/linux/if_vlan.h`. The kernel-side 802.1Q
  type definitions OVS pulls in to share types with the Linux kernel.

The Linux bridge counterparts are:

- `net/bridge/br_vlan.c`. Per-port VLAN management. The `br_vlan_add` function
  is invoked from netlink (when `bridge vlan add` runs) and updates the
  per-port VLAN bitmap.
- `net/bridge/br_vlan_8021q.c`. The 802.1Q ingress-and-egress filtering logic
  invoked from the bridge forwarding hook.
- `net/bridge/br_fdb.c`. The FDB hash table plus the (MAC, VID) lookup
  function the bridge uses to find the destination port.
- `net/8021q/vlan_core.c:vlan_do_receive`. The receiver-side 802.1Q
  demultiplexer that picks the correct vlan device (the `eth0.10` or
  `eth0.20` style sub-interface) for a tagged incoming frame.

A traditional Cisco engineer looking to understand the "ASIC equivalent" in
either OVS or Linux can read these functions; they implement, at the C-language
level, exactly what a hardware switch's silicon does at gate-level. The OVS
function `xvlan_output_translate` is forty lines of C; the Cisco SDK
equivalent is closed-source but conceptually identical.

R2.5.2 covered VLAN access and trunk. The next sub-capture, R2.5.3, covers
STP and RSTP. The connection between the two is direct: STP and RSTP are
the loop-prevention protocols layered on top of L2 forwarding, and L2
forwarding in turn is bounded by the VLAN configuration. A trunk port that
carries multiple VLANs runs STP per VLAN in some configurations (Cisco's
PVST+) or as a single instance across all VLANs (IEEE 802.1D RSTP, which OVS
implements via `lib/rstp.c`). Understanding which VLAN a frame belongs to is
a precondition for understanding which spanning tree applies. R2.5.3 will
build a three-bridge triangle on the lab host and observe an RSTP convergence
event live, with both OVS bridges and (if portable) a Linux-bridge equivalent
using `mstpd`.

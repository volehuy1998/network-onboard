# R2.5.3 Lab: Spanning tree on OVS (STP and RSTP) versus Linux bridge

> **Verbatim source:** `sdn-onboard/labs/v3.13-R2.5.3-stp-rstp.typescript`
> **Timing source:** `sdn-onboard/labs/v3.13-R2.5.3-stp-rstp.timing`
> **Captured:** 2026-04-30 07:16:33 UTC to 07:18:53 UTC inside `tmux new-session -s v3_13-r2_5_3` on `lab-openvswitch` (`192.168.1.250`). Lab host: Ubuntu 22.04.5 LTS, kernel `5.15.0-173-generic`, OVS 2.17.9 userspace install at `/usr/local/`.

The traditional Cisco engineer who has spent years on `spanning-tree mode rapid-pvst` knows what spanning tree is for. Build a triangle of switches without it and a single broadcast frame multiplies forever, the network melts in seconds. Run STP or RSTP on top of the same triangle and exactly one link gets put into a `Discarding` state, the loop is broken, broadcast traffic travels the spanning tree (a tree being, by mathematical definition, an acyclic connected graph), and convergence on a link failure happens in seconds rather than tens of seconds. This lab proves all of that on a real host using OVS. Three OVS bridges are configured in a triangle. RSTP is enabled on all three. After convergence, exactly one port on br3 (the bridge with the worst priority) sits in `Alternate Discarding`. A link is then brought down, RSTP re-converges, and the previously-Alternate port stays Disabled while the chain through br1 keeps connectivity alive. The lab then flips the bridges from RSTP to legacy 802.1D STP and observes the same three-bridge spanning tree with different on-wire timer parameters and different convergence speed. Finally the lab swaps OVS for the in-tree Linux kernel bridge with built-in 802.1D STP and observes the same root-bridge selection and the same one-port-blocking outcome.

The pedagogical aim is to give the reader, who already knows STP from Cisco-side training, three concrete things: (a) the OVS configuration surface for STP and RSTP, expressed as OVSDB columns; (b) the inspection surface, expressed as `ovs-appctl rstp/show` and `stp/show` plus the `Bridge.rstp_status` and `Bridge.status` OVSDB read-only columns; (c) the side-by-side comparison with the in-tree Linux kernel bridge, which has built-in 802.1D STP only (no RSTP) and exposes its state through sysfs at `/sys/class/net/<bridge>/bridge/`. After this lab, the reader can identify a loop on an OVS-or-Linux fabric using only the standard tooling, can predict the role/state of every port on a small triangle, and can read the full spanning-tree state from either implementation's introspection surface.

## R2.5.3.1. Where this part sits in the curriculum and why

R2.5.3 is the third sub-capture inside R2.5 of plan v3.13. R2.5.1 covered L2 learning (the MAC table). R2.5.2 covered VLAN access and trunk. This part covers the loop-prevention protocol layered on top of L2 forwarding: spanning tree. The order matters because the spanning tree is, semantically, the control plane that decides which subset of a bridge's ports actively participate in L2 forwarding at any moment. Without spanning tree (or some other loop-prevention mechanism) a bridge with a redundant connection to another bridge is a broadcast-storm waiting to happen. With spanning tree, the redundant link is held in a non-forwarding state until the active path fails. This is the classic engineering trade-off between availability (redundant links) and forwarding correctness (no loops); STP and RSTP exist to manage that trade-off automatically.

Two protocols are exercised in this lab. The first is **IEEE 802.1D STP**, the original 1990-vintage protocol. The second is **IEEE 802.1D-2004 RSTP**, the rapid-convergence successor. Both protocols define the same broad algorithm: every bridge has a numeric priority (lower is better); the bridge with the lowest priority becomes the root; every other bridge picks its best path to the root and designates one port (the "root port") as the next hop; non-root non-designated ports go into a non-forwarding state to break the loop. The differences between STP and RSTP are about the timing and mechanism of state transitions, not about the topology decisions. RSTP converges in seconds (using a proposal-agreement handshake between adjacent bridges) where STP needs roughly 30 to 50 seconds (waiting through fixed listening and learning timers).

A traditional Cisco engineer who has used `spanning-tree mode pvst`, `spanning-tree mode rapid-pvst`, and `spanning-tree mode mst` will recognise all three as branches of the same family. PVST runs one STP instance per VLAN; rapid-PVST runs one RSTP instance per VLAN; MST groups multiple VLANs into a small number of instances. OVS does not implement any of the per-VLAN variants (PVST/rapid-PVST/MST); OVS implements one instance of 802.1D STP and one instance of 802.1D-2004 RSTP per bridge, and that instance applies to all VLANs on the bridge regardless. This is a real limitation when OVS is dropped into a Cisco fabric that runs PVST: the Cisco-side BPDUs are PVST-encoded (with a per-VLAN field) and OVS's plain 802.1D plus 802.1D-2004 implementation does not parse the PVST extensions. The OVS-Cisco interaction in such a fabric typically falls back to Cisco's "common spanning tree" (CST), which is the VLAN-1 spanning tree, and the operator must understand that any non-VLAN-1 traffic is unprotected by spanning tree on the OVS side. The lab does not cover this Cisco-mixing scenario directly; it is mentioned here so the reader does not draw an overly-rosy conclusion from the OVS-only experiment.

A short historical note helps frame why both protocols exist together in OVS. The original 802.1D STP (1990) was designed when bridges were rare, links were slow, and the cost of a broadcast storm was the cost of re-running a few hundred PCs back to a fileserver. Convergence times of 30 to 50 seconds were tolerable. As Ethernet expanded into voice and real-time traffic in the late 1990s and early 2000s, those convergence times became unacceptable; a 50-second outage on a VoIP fabric drops calls. IEEE 802.1D-2004 introduced RSTP with proposal/agreement handshakes that reduce convergence to single-digit seconds. OVS supports both because the install base of legacy STP devices is still large; an OVS bridge connecting to an old hardware switch may need to fall back to 802.1D for compatibility, while an OVS-only fabric will always use RSTP. The lab will show both in action so the engineer can recognise either in production.

The R2.5.3 source anchors live in two places. First, the top-level state machines are in `lib/stp.c` (the legacy 802.1D, 1733 lines) and `lib/rstp.c` (the 802.1D-2004, 1683 lines), both at the OVS v2.17.9 baseline (commit `0bea06d9957e3966d94c48873cd9afefba1c2677`). The 802.1D-2004 RSTP state machines live in `lib/rstp-state-machines.c` (2221 lines, the most substantial single state-machine file in the OVS source tree). Second, the configuration plane that converts OVSDB columns into per-port settings lives in `vswitchd/bridge.c:port_configure_stp` (line 1608) and `port_configure_rstp` (line 1697). The provider-neutral entry points (`ofproto_port_set_stp`, `ofproto_port_set_rstp`, etc.) live in `ofproto/ofproto.c`.

## R2.5.3.2. Topology, three-bridge triangle

The triangle is built on a single host using veth pairs as the bridge-to-bridge links.

```ascii
                          br1 (priority 4096)
                           |\
                           | \
                          v12a v13a
                           |    \
                           |     \
                          v12b    \
                           |       \
                          br2       \
                          (priority  \
                           8192)     v13b
                           |          |
                          v23a       br3 (priority 12288)
                           |          |
                          v23b ------+
```

Three bridges. Three veth pairs (`v12 a/b`, `v23 a/b`, `v13 a/b`) connecting them in a triangle. Each veth pair has two endpoints; one endpoint joins each of the two bridges. The "letter suffix" naming is used to distinguish the two ends. `v12a` lives on br1 and `v12b` lives on br2; the link between them is the v12 link.

The bridges are given distinct priorities so the root election is deterministic. RSTP and STP both use a "lower priority wins" rule, so br1 with priority 4096 is the lowest and becomes root. br2 (8192) and br3 (12288) are non-root. With a triangle plus distinct priorities, the spanning-tree algorithm's decisions are unambiguous: br1 is root; v12a (br1 side of the v12 link) is Designated, v12b (br2 side) is Root; v13a (br1 side of the v13 link) is Designated, v13b (br3 side) is Root; for the v23 link both ends are non-root, so the higher-priority bridge (br2 at 8192) gets Designated and the lower-priority bridge (br3 at 12288) gets Alternate Discarding. This last assignment is the loop break: br3 has TWO paths to the root (one through br2 via v23, one direct via v13), and STP picks the v13 path (cost 2000 in RSTP) over the v12-then-v23 path (cost 2000 + 2000 = 4000), so v23b is parked.

The Linux bridge half of the lab uses the same conceptual layout with hostnames `lbr1`, `lbr2`, `lbr3` and veth pairs named `v12 l/m`, `v23 l/m`, `v13 l/m`. The IP addressing is irrelevant for STP/RSTP exercises (BPDUs are L2-only, addressed to `01:80:c2:00:00:00` plus the LLC/SNAP framing for CST or `01:80:c2:00:00:00` directly for RSTP), so no namespaces or IPs are configured. The bridges live on the host, the veth ends attach to the bridges, BPDUs flow.

## R2.5.3.3. The OVS configuration surface for spanning tree

Open vSwitch encodes spanning-tree configuration in two columns of the OVSDB `Bridge` row: `stp_enable` (boolean) and `rstp_enable` (boolean). When exactly one of those is `true`, the corresponding protocol runs on the bridge. The two protocols' parameters are passed via the `other_config` map: `other_config:stp-priority` (16-bit integer 0 to 65535) for STP, and `other_config:rstp-priority` (16-bit integer 4096 to 61440 in steps of 4096) for RSTP. Note the RSTP parameter requires a multiple of 4096 because RSTP uses the upper four bits of the priority field for the bridge identifier plus the lower twelve bits for a system identifier extension; the IEEE 802.1D-2004 spec restricts the user-settable priority to the upper four bits, hence steps of 4096.

OVS exposes the resulting state through three surfaces:

1. The `ovs-appctl stp/show` and `ovs-appctl rstp/show` commands, which
   print a human-readable summary of the bridge's role, port states, and    relevant
timers.
2. The OVSDB `Bridge.status` map (for STP) and `Bridge.rstp_status` map (for
   RSTP), which carry machine-readable read-only fields a controller can    consume.
3. The `Bridge.flood_vlans` and `Port.stp_*` columns, which the operator
   can also read for per-port configuration.

A traditional Cisco engineer recognises this pattern as "configure with one syntax, inspect with another." On Cisco IOS the configuration is `spanning-tree vlan 1 priority 4096` and the inspection is `show spanning-tree`. On OVS the configuration is `ovs-vsctl set bridge br1 other_config:rstp-priority=4096` and the inspection is `ovs-appctl rstp/show br1`. Same idea, different syntax.

**Misconception to refute.** A naive reader might think OVS automatically detects loops and prevents broadcast amplification even without spanning tree. Reality: OVS does NOT have a built-in loop detection in its NORMAL forwarding action. ### ▶ Guided Exercise 2: Enable RSTP and observe convergence

This exercise enables RSTP on all three bridges with deterministic priorities (4096 / 8192 / 12288) so the root election is unambiguous. RSTP runs through its proposal/agreement handshake on each link. After roughly 6 to 12 seconds the spanning tree converges. The exercise then reads the per-bridge state via `ovs-appctl rstp/show` to confirm.

**Outcomes.** br1 becomes root. br2's v12b is `Root Forwarding`, v23a is `Designated Forwarding`. br3's v13b is `Root Forwarding`, v23b is `Alternate Discarding`. The triangle is now a tree: br1 at the top, br2 and br3 as leaves connected directly to br1, and the v23 link parked.

br3's v23b is in `Alternate Discarding` state. This is the loop break. v13b is the Root port (direct path to br1, cost 2000). v23b would be a second path to br1 (via br2, cost 4000), but RSTP recognises this redundant path and parks v23b in Alternate Discarding. "Alternate" is RSTP-specific terminology for "this port has heard a better path advertisement from elsewhere, so this port is held as a backup but not used for forwarding." The state `Discarding` means the port drops all data traffic; it still processes BPDUs to track topology changes.

The pedagogical highlight: this is exactly the textbook loop-break behaviour. Three bridges, three links, exactly one link parked. The triangle is now a spanning tree.

The role-and-state taxonomy in IEEE 802.1D-2004 RSTP merits a short glossary because the names are not always self-explanatory. The role field describes the port's logical position in the topology: `Root` means "this port leads to the root bridge along the best path I know"; `Designated` means "on the link this port faces, this side of the link is responsible for forwarding traffic to and from the rest of the tree"; `Alternate` means "I have heard a better path advertisement from another port on this same bridge, so I am holding this port as a backup"; `Backup` means "I have heard MY OWN advertisement on this port (the link is a self-loop or near-self-loop), so this port is redundant within the same bridge." The state field describes what the port is currently doing with traffic: `Forwarding` means data flows; `Discarding` means data is dropped (BPDUs still flow); `Learning` is a brief transitional state in which the port learns MAC addresses but does not yet forward (RSTP traverses this state quickly via the proposal/agreement handshake; legacy STP sits in it for the full forward-delay timer).

The two-line `rstp/show` output for br3 captures the most pedagogically important fact in two cells: `v23b Alternate Discarding`. Without those two cells, the lab's whole purpose would be defeated. Memorise the combination: `Alternate` + `Discarding` is the canonical loop-break indicator on RSTP. If you see it, the topology has detected redundancy and parked a link. If you don't see it on a topology that you know has redundancy, something is wrong: either RSTP isn't enabled, or the bridge priorities are misconfigured, or a peer is running a different protocol.

### ▶ Guided Exercise 5: Surface the STP/RSTP mutual-exclusion edge case

E2.5 §E.3 documented a hypothesis: enabling both `stp_enable=true` AND `rstp_enable=true` on the same bridge produces a log error and only the first one takes effect. This exercise tests the hypothesis and shows the empirical reality, which is more nuanced.

The OVSDB `get` returns `true` for both columns. The hypothesis "only the first one takes effect, the other is silently ignored at the OVSDB layer" is **refuted at the OVSDB layer**: both columns are happily persisted as `true` simultaneously.

**2.** Read the OVS daemon log to see what happened at the runtime layer.

```text

2026-04-30T07:18:01.051Z|00003|stp(handler294)|WARN|br2: received BPDU of unexpected type 2
2026-04-30T07:18:01.051Z|00004|stp(handler294)|WARN|br3: received BPDU of unexpected type 2
```

The runtime story is more interesting. The legacy `stp` module logs `received BPDU of unexpected type 2` warnings on br2 and br3. Why does this happen? When br1 has rstp_enable=true alongside stp_enable=true, the runtime starts the RSTP state machine (which sends RSTP BPDUs of BPDU type 0x02). Meanwhile br2 and br3 still run the legacy STP state machine (which sends and expects 802.1D Configuration BPDUs of type 0x00 plus Topology Change Notification BPDUs of type 0x80). The legacy STP daemon on br2 and br3 receives the RSTP-format BPDU from br1 and rejects it as "unexpected type 2."

The hypothesis from E2.5 §E.3 ("only the first one takes effect") is refined by this empirical finding: at the OVSDB layer, both columns persist; at the runtime layer, the RSTP state machine takes precedence on the bridge that has it enabled, but the BPDUs it emits are not intelligible to peer bridges still running legacy STP. The result is a hybrid state where br1 sends RSTP BPDUs that br2 and br3 can't parse, and br2 and br3 send legacy STP BPDUs that br1's RSTP state machine may or may not parse depending on the version (RSTP is mostly backwards-compatible with 802.1D STP, but the lab does not test the reverse direction). The practical advice is to never enable both columns; pick one and stick with it across the entire fabric.

**3.** Restore single-protocol state.

```text

root@lab-openvswitch:~/network-onboard# ovs-vsctl set bridge br1 stp_enable=false
root@lab-openvswitch:~/network-onboard# ovs-vsctl set bridge br2 stp_enable=false
root@lab-openvswitch:~/network-onboard# ovs-vsctl set bridge br3 stp_enable=false
```

**Finish.** The mutual-exclusion edge case is empirically characterised. OVSDB allows both; runtime produces a BPDU-format mismatch; operators should use exactly one protocol per bridge and ideally per fabric.

A defensive engineering pattern emerges from this experiment: when making any spanning-tree configuration change, always read back the state through both the OVSDB layer (`ovs-vsctl get`) and the runtime layer (`ovs-appctl rstp/show` or `stp/show`). The two layers can silently disagree, as shown above. A monitoring agent that checks only `Bridge.stp_enable=true` will think STP is running while the runtime has switched to RSTP; conversely, an agent that calls `rstp/show` and expects a specific output format will be surprised on a bridge that has both columns set. The right monitoring discipline is to assert exactly one protocol is enabled at the OVSDB layer AND that the runtime tool's output is sane; both checks are necessary for correctness.

## R2.5.3.4. The Linux bridge equivalent: built-in 802.1D STP via sysfs

The Linux kernel bridge has supported STP since the very early days of the bridge module. The implementation is in-kernel at `net/bridge/br_stp.c` and related files. Configuration is via `ip link set <bridge> type bridge stp_state 1 priority N` and inspection is via reading sysfs files at `/sys/class/net/<bridge>/bridge/` (for bridge-level state) and `/sys/class/net/<port>/brport/` (for per-port state). The Linux kernel bridge does not implement RSTP natively; for RSTP support, operators typically run the userspace `mstpd` daemon (not installed on this lab host), which speaks the RSTP protocol and updates the kernel via netlink.

The mapping from OVS columns to Linux bridge equivalents is:

| OVS column / setting          | Linux bridge equivalent                            |
|-------------------------------|----------------------------------------------------|
| `Bridge.stp_enable=true`      | `ip link set lbr type bridge stp_state 1`          |
| `other_config:stp-priority=N` | `ip link set lbr type bridge priority N`           |
| `ovs-appctl stp/show lbr`     | `cat /sys/class/net/lbr/bridge/{root_id,root_path_cost,bridge_id,...}` |
| `Port.stp_*` columns          | `cat /sys/class/net/<port>/brport/{state,priority,path_cost,...}` |
| `Bridge.status`               | `ip -d link show lbr` (the `bridge ...` extension) |

The state-code mapping for the per-port state file is documented in `include/uapi/linux/if_bridge.h` of the Linux kernel: 0=DISABLED, 1=LISTENING, 2=LEARNING, 3=FORWARDING, 4=BLOCKING. The Linux bridge records the legacy 802.1D state names directly; the file content is just the integer.

## R2.5.3.4a. Production-readiness assessment

A traditional Cisco engineer does not just turn STP/RSTP on and walk away; the protocol is configured deliberately and monitored continuously, because the consequences of misconfiguration are visible (broadcast storms) or subtle (invisible suboptimal paths that surface only under peer failure). The same discipline applies on OVS. Three production considerations matter most.

First, the trade-off cost of enabling RSTP. Each enabled bridge runs the RSTP state machines once per hello-time interval (default 2 seconds). On a host with a hundred bridges, this is two hundred state-machine ticks per second consumed by the RSTP module. The CPU cost is small (a few microseconds per tick on commodity hardware) but the per-bridge memory footprint includes the `struct rstp` and per-port `struct rstp_port` allocations, plus the eight per-state-machine sub-objects. For a hundred bridges this is on the order of 200 KB plus the per-port BPDU send/receive state. The cost is negligible on modern systems but worth keeping in mind for embedded or resource-constrained deployments.

Second, the failure mode under sustained load. RSTP BPDUs travel as ordinary L2 frames at the hello-time rate. A bridge that is also under heavy unicast load may delay BPDU processing if the rx queue saturates; the symptom is a max-age timeout (20 seconds) on the peer side and a spurious topology recomputation. The mitigation is to ensure BPDU traffic is prioritised, which OVS does by classifying BPDUs based on their L2 destination MAC (`01:80:c2:00:00:00`) and processing them on the management slow-path before the data fast-path. Production deployments that load OVS to the line rate should monitor `Bridge.rstp_status` closely for unexpected topology changes.

Third, the monitoring discipline. A spanning-tree topology change is typically a sign of either a real link failure or a misconfiguration. Regular topology changes (more than once per hour on a stable fabric) indicate a misconfigured edge port, a flapping link, or a duplex-mismatch problem. The OVS-side instrumentation for this is the `Bridge.rstp_status:topology-change-detected` field plus the `ovs-vswitchd.log` line `topology-change` events. A production monitoring agent should alert on any topology change, treat it as a tier-2 incident, and require operator inspection. This is exactly what good Cisco operators do; OVS deployments should match that discipline.

## R2.5.3.6. Common misconceptions for the audience

**Misconception 1.** "OVS detects loops automatically." Wrong. OVS without STP/RSTP enabled has no loop detection. The triangle topology in this lab, without spanning tree, would broadcast-storm on the first ARP request. STP or RSTP is mandatory for any topology with redundant connectivity.

**Misconception 2.** "RSTP is just faster STP." Mostly right but missing nuance. RSTP introduces new role names (Alternate, Backup) and new state names (Discarding replaces Blocking/Listening; Forwarding stays). The proposal/agreement handshake mechanism replaces the fixed listen-then-learn timer transitions. The wire format of BPDUs changes (BPDU type 0x02 plus new flag fields). Cost values were redefined in IEEE 802.1D-2004. Functionally, RSTP and STP solve the same problem; mechanically they are distinct protocols, and a fabric mixing both has interoperability gotchas.

**Misconception 3.** "Setting both `stp_enable=true` and `rstp_enable=true` is a no-op or an error." Wrong on both counts. OVSDB happily persists both columns as `true`. The runtime starts the RSTP state machine but emits BPDUs that legacy STP peer bridges cannot decode (logs `BPDU of unexpected type 2`). The resulting state is partial-RSTP-on-this-bridge, partial-STP-on-peers, and topology may converge incorrectly if the peers fall back to all-ports-forwarding when they cannot interpret the BPDUs. Always pick exactly one protocol per fabric.

**Misconception 4.** "The Linux kernel bridge supports RSTP natively." Wrong. The in-tree kernel bridge implements 802.1D STP only. RSTP support requires the userspace `mstpd` daemon, which is a separate package (`mstpd` or `mstpd_userspace` in some distros). On Ubuntu 22.04 the package is not installed by default; the kernel bridge falls back to 802.1D STP and silently ignores RSTP-format BPDUs received from neighbours. A mixed fabric of OVS-RSTP plus kernel-bridge-STP works at the wire level because RSTP is mostly backwards-compatible with 802.1D STP, but the kernel bridge will not exhibit RSTP-grade convergence speed.

**Misconception 5.** "STP path cost values are bridge-vendor-specific." Mostly wrong. IEEE 802.1D defined the original cost values in 1990 and IEEE 802.1D-2004 redefined them for higher-speed links. OVS, Cisco, Juniper, and the Linux bridge all use the IEEE-defined cost values for standard link speeds (1000/100/10/1 Mbit/s and faster). Vendor-specific overrides exist (Cisco's "long path-cost mode" maps to IEEE 802.1D-2004; "short path-cost mode" maps to 802.1D-1990) but the canonical values are standardised.

**Misconception 6.** "OVS RSTP is a separate process I can stop and start." Wrong. Both STP and RSTP run inside the `ovs-vswitchd` daemon as part of the bridge thread's main loop. There is no separate `ovs-stp` or `ovs-rstp` process to manage. Enabling or disabling the protocols is done via OVSDB column writes (`stp_enable=true` or `rstp_enable=true`), which the daemon picks up on its next OVSDB-IDL poll cycle (typically within 100 milliseconds). The protocol state lives in the daemon's heap and is lost on daemon restart; after a restart, the protocol re-runs its convergence from scratch. This is one reason production OVS deployments prefer to keep `ovs-vswitchd` running through configuration changes rather than restart it.

**Misconception 7.** "If I see two ports in `Forwarding` state on the same two-link redundant pair, I have a loop." Correct only for STP and RSTP on TWO bridges. With three or more bridges in a triangle (as this lab), two `Forwarding` ports on the same bridge are normal: the bridge has two distinct DESIGNATED links to two different non-root bridges, and both can be Forwarding because they don't form a loop with each other, only with paths through the root. The loop-break is at the OTHER end of the redundant link, on the bridge with the worst priority. This is counter-intuitive on first reading; the lab's exercise 2 made it concrete by showing br1 with v12a Designated Forwarding AND v13a Designated Forwarding (no loop-break on br1) while br3 has v23b Alternate Discarding.

## R2.5.3.6. How a port's STP state affects forwarding inside OVS

A natural follow-up question for the engineer who has just observed the RSTP and STP states is: how exactly does the port's spanning-tree state affect the OVS forwarding decision? The answer connects R2.5.3 directly back to R2.5.1 (L2 learning) and R2.5.2 (VLAN handling): the per-port STP/RSTP state acts as a pre-condition that gates the NORMAL action's willingness to forward through the port.

Inside `ofproto/ofproto-dpif-xlate.c`, the `xlate_normal` function (which implements the OVS "act like a learning bridge" semantic) consults several per-port flags before deciding to flood or to consult the FDB. One of those flags is `xport->stp_state` and another is the equivalent RSTP state. If the port is in `Forwarding` state (RSTP) or `forwarding` state (legacy STP), the NORMAL action treats the port as eligible for output. If the port is in `Discarding` (RSTP) or `blocking` (legacy STP), the NORMAL action skips the port entirely; no traffic flows through, no MAC learning happens. If the port is in `Learning` state (the brief intermediate state during legacy STP convergence), the FDB is updated but the frame is not forwarded.

This means the `Alternate Discarding` state on br3's v23b in exercise 2 is not a passive label; it is an active gate inside the OVS forwarding path that prevents the port from contributing to broadcast amplification. The kernel datapath megaflow rules (which exercise 5 of R2.5.2 dumped for VLAN handling) reflect this gating. A port in `Discarding` state will not appear as an output target in any megaflow generated by the NORMAL action; the kernel will not be asked to send traffic out of it because the userspace translator already excluded it. This is one example of the userspace-and-kernel co-design that defines OVS: state held in userspace shapes the kernel's fast-path decisions.

The implementation of this gating lives, for legacy STP, in `ofproto/ofproto-dpif-xlate.c` near the `is_admissible` and `stp_forward_in_state` helpers, and for RSTP in the analogous `rstp_forward_in_state` helper. Both helpers translate the state enum (`STP_FORWARDING`, `STP_BLOCKING`, etc., or `RSTP_FORWARDING`, `RSTP_DISCARDING`, etc.) into a boolean answer to "may this port forward data right now?" The boolean answer feeds into the NORMAL action's decision tree.

## R2.5.3.7a. Reading the state machine sources for an engineer ready to dig

For an engineer who wants to go beyond observing the protocols and reach into how they are implemented, the cleanest entry point in the OVS source tree is `lib/rstp-state-machines.c`. The file is roughly 2200 lines but is organised as eight small state-machine implementations, each with its own init function, run function, and per-event handler. The Port Role Selection state machine (function `rstp_port_role_selection_sm`) is the one that produces the role assignments observed in this lab. Reading it end to end is a useful exercise: the function transitions through the states `INIT_BRIDGE`, `ROLE_SELECTION`, and back to `ROLE_SELECTION` on every iteration, evaluating per-port BPDU information against per-bridge state to decide each port's role. The decision logic is exactly what the spec describes: a port is `Root` if it carries the best path to root, a port is `Designated` if its side of the link advertised the best path to root, a port is `Alternate` if it received a better-than-its-own advertisement on the same physical link, etc.

The legacy 802.1D STP state machine in `lib/stp.c` is structured differently. The 1990 spec is described in textual procedural form rather than as separate state machines, and `lib/stp.c` follows that style: the top-level `stp_run` function is a single 100+ line procedure that evaluates timers, processes pending BPDUs, and updates per-port state in one pass. This is harder to read end-to-end than the RSTP file but matches the historical spec organisation. An engineer who needs to debug a legacy STP issue should read the function from top to bottom while holding the IEEE 802.1D-1998 spec text alongside.

The kernel-side counterparts in `net/bridge/br_stp.c` and friends are organised more like `lib/stp.c` (textual procedural) than like `lib/rstp-state-machines.c` (decomposed state machines). The kernel does not implement RSTP and so does not need the more rigorous state-machine decomposition that the 802.1D-2004 spec demands.

## R2.5.3.8. Continuity to the next sub-capture

R2.5.3 covered spanning tree (STP and RSTP) on OVS and Linux bridge. The next sub-capture, R2.5.4, covers link aggregation: the LACP protocol plus the various bonding modes (active-backup, balance-slb, balance-tcp). The connection between the two is direct: where spanning tree breaks loops formed by redundant inter-bridge links, link aggregation **uses** redundant inter-bridge links by bundling them into a single logical link with load-balancing across the members. The two are complementary: the operator chooses spanning tree to handle redundant paths and link aggregation to handle redundant members of a single path. Some networks use both; some use one or the other. R2.5.4 will exercise OVS bonds via `ovs-vsctl add-bond` and Linux bonds via the in-kernel bonding driver (`mode=4` for 802.3ad LACP).

## 13.5.1 Port_Binding overview

`Port_Binding` row trong SBDB biểu diễn binding giữa logical port và physical chassis. Mỗi LSP/LRP có một Port_Binding row.

Columns quan trọng:
- `logical_port`: tên LSP.
- `datapath`: logical datapath reference.
- `chassis`: chassis đang claim port (NULL nếu chưa).
- `type`: loại Port_Binding.
- `options`: map metadata per-type.
- `mac`, `tunnel_key`.

---

## 13.5.2 Type `""` (default VIF)

Mặc định — Port_Binding của VM tap. Empty string type.

- ovn-controller trên chassis có OVS port với `external_ids:iface-id=<lsp>` sẽ claim.
- Chỉ một chassis tại một thời điểm (trừ multichassis — Part 19).

Flow: VM khởi động → libvirt/Kubelet plug tap vào br-int với iface-id → ovn-controller match, set chassis trong Port_Binding.

---

## 13.5.3 Type `localnet`

LSP connect đến physical network.

- Không claim chassis (available on all chassis).
- Mỗi chassis map tới bridge mapping tương ứng.
- ovn-controller install patch port giữa br-int và br-ex theo `network_name`.

---

## 13.5.4 Type `l3gateway` và `chassisredirect`

**Distributed LR** handle L2/L3 forwarding distributed across chassis.

Nhưng **north-south NAT** không distributed được (một external IP chỉ thuộc một chassis). Cần gateway chassis.

**chassisredirect Port_Binding**: shadow của LRP trên gateway chassis. Traffic north-bound redirect qua chassisredirect → gateway chassis xử lý NAT → br-ex.

```bash
[admin]# ovn-nbctl lrp-set-gateway-chassis lrp-external chassis-1 100
[admin]# ovn-nbctl lrp-set-gateway-chassis lrp-external chassis-2 50
```

Chassis-1 active, chassis-2 standby. Failover via BFD (Part 13.6).

---

## 13.5.5 Type `patch`

LSP type=router tạo Port_Binding type=patch. Auto-generate bởi northd.

---

## 13.5.6 Type `l2gateway`

LSP gateway từ OVN logical network đến hardware L2 switch (VTEP protocol).

Use case: extend OVN logical switch đến hardware VTEP (ESR pattern).

```bash
[admin]# ovn-nbctl lsp-set-type vif-hw-vtep l2gateway
[admin]# ovn-nbctl lsp-set-options vif-hw-vtep network_name=physnet1 \
         l2gateway-chassis=chassis-1
```

---

## 13.5.7 Type `localport`

Special — tồn tại trên **mọi chassis** với cùng MAC + IP. Không gửi traffic qua tunnel.

Use case: metadata service cho Kubernetes/OpenStack — mỗi VM truy cập metadata endpoint (169.254.169.254) với identical MAC/IP. Metadata request không leave chassis.

---

## 13.5.8 Type `remote`

OVN cluster-to-cluster peering. Port trên remote cluster visible dưới type `remote` trong local SB. Hiếm dùng.

---

## 13.5.9 Debug Port_Binding

```bash
[central]# ovn-sbctl list Port_Binding <lsp-name>
[central]# ovn-sbctl show-rep
```

Khi VM không ping được, check:
1. `Port_Binding.chassis` != NULL.
2. Chassis claim == chassis VM đang chạy.
3. OpenFlow rule trong br-int match LSP MAC.

### ▶ Guided Exercise 16: Port_Binding claim workflow

**Outcomes:** Observe ovn-controller claim Port_Binding khi VIF plug.

**Instructions:**

**1.** Tạo LSP:

```bash
[central]# ovn-nbctl lsp-add ls-tenant1 vm-claim-test
[central]# ovn-nbctl lsp-set-addresses vm-claim-test "aa:bb:cc:de:ad:01 10.0.0.99"
```

**2.** Check — chassis NULL:

```bash
[central]# ovn-sbctl list Port_Binding vm-claim-test
```

**3.** Plug tap vào br-int với iface-id:

```bash
[chassis-1]# ovs-vsctl add-port br-int tap-claim -- \
             set Interface tap-claim type=internal \
             external_ids:iface-id=vm-claim-test
```

**4.** Recheck:

```bash
[central]# ovn-sbctl list Port_Binding vm-claim-test
# chassis: <chassis-1 UUID>
```

**5.** Cleanup: `ovs-vsctl del-port tap-claim && ovn-nbctl lsp-del vm-claim-test`

## Điểm cốt lõi cần nhớ

Thứ nhất, Port_Binding là link giữa logical port và physical chassis. 7 types phục vụ use case khác nhau.

Thứ hai, default VIF claim dựa trên OVS port `external_ids:iface-id`.

Thứ ba, chassisredirect + gateway chassis là pattern north-south NAT. Failover via BFD.

Thứ tư, localport tồn tại trên mọi chassis — dùng cho metadata service.

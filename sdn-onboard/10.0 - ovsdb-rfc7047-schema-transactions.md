## 10.0.1 OVSDB: ordered, transactional, schema-bound

Compass Chapter 4 puts it as: "OVSDB is an ordered, transactional, schema-bound JSON-RPC database."

The three core properties:

- **Ordered.** Transactions have a clear total order. Every commit has a monotonic sequence number. A client can subscribe and receive notifications in the correct order.
- **Transactional.** ACID. A transaction can contain many operations: either all of them commit or all of them roll back.
- **Schema-bound.** The schema clearly defines tables, columns, types, and constraints. Invalid writes are rejected.

Protocol: JSON-RPC 1.0 over a Unix socket (or TCP, or SSL). Each request is a JSON object with `method`, `params`, and `id`.

## 10.0.3 Column types

Compass Chapter 4: every column is typed: strings, integers, reals, booleans, UUIDs, and the two composite types `set<T>` and `map<K,V>`.

**Scalar:** `string`, `integer`, `real`, `boolean`, `uuid`.

**Set<T>:** an unordered collection of scalars. For example, `Bridge.protocols` is a `set<string>` that contains values such as `["OpenFlow13", "OpenFlow14"]`.

**Map<K,V>:** key-value pairs. The columns `other_config` and `external_ids` are escape hatches for metadata. Both are `map<string,string>`. For example, the MAC aging time is read from `Bridge.other_config:mac-aging-time`.

**UUID reference:** a column that references a row in another table by UUID. For example, `Bridge.ports` is a `set<UUID>` that points to rows in `Port`. The flag `ovs-vsctl --id=@name` creates a symbolic reference to a UUID inside one transaction.

## 10.0.4 The ten RFC 7047 operations

RFC 7047 §4.1:

1. **insert.** Add a new row.
2. **select.** Query rows by a condition.
3. **update.** Replace column values (destructive).
4. **mutate.** Modify a column in place (add to a set, increment an integer).
5. **delete.** Delete rows that match a condition.
6. **wait.** Block until a condition is satisfied.
7. **commit.** Commit the entire transaction.
8. **abort.** Force a rollback.
9. **comment.** Attach a comment to the audit log.
10. **assert.** Check a condition; fail if it is not true.

Example JSON-RPC transaction that creates a bridge and a port:

```json
{
  "method": "transact",
  "params": [
    "Open_vSwitch",
    {"op": "insert", "table": "Bridge", "row": {"name": "br0"}, "uuid-name": "bridge1"},
    {"op": "insert", "table": "Port", "row": {"name": "eth0"}, "uuid-name": "port1"},
    {"op": "insert", "table": "Interface", "row": {"name": "eth0"}, "uuid-name": "iface1"},
    {"op": "mutate", "table": "Open_vSwitch", "where": [],
     "mutations": [["bridges", "insert", ["set", [["named-uuid", "bridge1"]]]]]}
  ],
  "id": 1
}
```

`ovs-vsctl add-br br0 -- add-port br0 eth0` is translated to a similar transaction.

## 10.0.5 The `monitor_cond` protocol

An OVSDB client subscribes for real-time updates instead of polling.

**`monitor`** (the original RFC 7047 method): the client sends a request with the list of columns to track. The server pushes the initial state plus incremental updates.

**`monitor_cond`** (extension, OVS 2.6 and later): adds a `where` clause. The client receives updates only for rows that match the condition. This matters at scale (the OVN Southbound has hundreds of thousands of `Port_Binding` rows, and each hypervisor only subscribes to the rows that belong to it).

```bash
[host]# ovsdb-client monitor Open_vSwitch Bridge
# Streams updates in real time as transactions land
```

## 10.0.7 RFC 7047 wire protocol byte by byte

OVSDB is JSON-RPC 1.0 over a Unix socket, TCP, or SSL. Each message is one JSON object, framed as **length-prefixed** at the transport layer.

### Connection establishment

The client opens a socket to `ovsdb-server` (default `/var/run/openvswitch/db.sock`, or `tcp:host:6640`). After `connect()` succeeds, the client can immediately send a request without a separate handshake. The server reads the first JSON-RPC request as it arrives.

Authentication (for TCP and SSL): SSL with client certificate verification. For a Unix socket: filesystem permissions on the socket file (default 0660 root:openvswitch).

### Wire format: JSON-RPC 1.0

JSON-RPC 1.0 (different from 2.0) has 3 message types:

**Request:**

```json
{"method": "<method-name>", "params": [<params-array>], "id": <unique-id>}
```

Example `list_dbs`:

```json
{"method": "list_dbs", "params": [], "id": 0}
```

**Response (success):**

```json
{"result": <result-data>, "error": null, "id": <matching-id>}
```

Example:

```json
{"result": ["Open_vSwitch"], "error": null, "id": 0}
```

**Response (error):**

```json
{"result": null, "error": {"error": "<short-name>", "details": "<detail>"}, "id": <matching-id>}
```

**Notification** (server to client, no request-response pairing):

```json
{"method": "update", "params": [<table-updates>], "id": null}
```

`id: null` marks a notification, which does not require a reply.

### Framing on TCP and SSL

OVSDB does not use HTTP framing. Each JSON message is self-delimiting (the parser stops when the closing brace balances). The client must use a streaming JSON parser to handle partial reads.

```
[client → server]: {"method":"list_dbs","params":[],"id":0}\n
[server → client]: {"result":["Open_vSwitch"],"error":null,"id":0}\n
```

The trailing `\n` is optional, not a framing requirement (but it is helpful when debugging with `nc`).

### Echo keepalive

The server and the client use the `echo` method as a keepalive:

```json
{"method": "echo", "params": ["arg1", "arg2"], "id": "<id>"}
```

The server replies with the same params. The default probe interval is 5 seconds. If there is no response within 5 seconds times 4 = 20 seconds, the connection is treated as dead and the side reconnects.

### Test with `nc`

```bash
[host]# nc -U /var/run/openvswitch/db.sock
{"method":"list_dbs","params":[],"id":1}
{"result":["Open_vSwitch"],"id":1,"error":null}
```

This is the simplest way to verify that `ovsdb-server` is alive at the wire-protocol level.

## 10.0.8 The ten operations in depth, with full JSON examples

### `insert`

```json
{"op": "insert", "table": "Bridge", "row": {"name": "br0"},
 "uuid-name": "new_bridge"}
```

Parameters:

- `table`: target table.
- `row`: the column values for the new row. Columns that are not provided take the default from the schema.
- `uuid-name` (optional): a symbolic name for the UUID, used by `mutate` operations within the same transaction.

Result: `{"uuid": ["uuid", "<actual-UUID>"]}`.

### `select`

```json
{"op": "select", "table": "Logical_Switch_Port",
 "where": [["type", "==", "localnet"], ["name", "!=", "phys-port"]],
 "columns": ["name", "addresses"]}
```

`where` is an array of conditions (logical AND). Each condition has the form `[<column>, <function>, <value>]`. Functions: `==`, `!=`, `<`, `<=`, `>=`, `>`, `includes`, `excludes`.

Result: `{"rows": [{"name": "...", "addresses": [...]}, ...]}`.

### `update`

```json
{"op": "update", "table": "Bridge", "where": [["name", "==", "br0"]],
 "row": {"fail_mode": "secure"}}
```

**Destructive replace.** Column values in `row` overwrite completely. For composite types (set or map), the new value replaces the old value entirely.

Result: `{"count": 1}` (the number of rows affected).

### `mutate`

```json
{"op": "mutate", "table": "Bridge", "where": [["name", "==", "br0"]],
 "mutations": [["ports", "insert", ["set", [["named-uuid", "new_port"]]]]]}
```

**Non-destructive in-place modify.** Add to a set, remove from a set, increment an integer, multiply.

Mutator types: `+=` (add to set or increment), `-=` (subtract or remove), `*=`, `/=`, `%=`, `insert`, `delete`.

Result: `{"count": 1}`.

### `delete`

```json
{"op": "delete", "table": "Bridge", "where": [["name", "==", "br0"]]}
```

Garbage collection: if another row holds a UUID reference to the row that was just deleted, OVSDB automatically clears that reference (`set<UUID>` of size 0). There is no cascade delete.

Result: `{"count": 1}`.

### `wait`

```json
{"op": "wait", "table": "Bridge", "where": [["name", "==", "br0"]],
 "columns": ["fail_mode"], "until": "==",
 "rows": [{"fail_mode": "secure"}], "timeout": 5000}
```

Blocks the transaction until `where` matches `until` rows, or the timeout expires (in milliseconds). Useful as an atomicity assertion: "this transaction commits only if the state is what I expect."

### `commit`

```json
{"op": "commit", "durable": true}
```

`durable=true`: fsync the log file before replying. Slow but safe.
`durable=false`: reply immediately; the log is flushed eventually. Fast but it can lose data after a crash.

OVN defaults: `durable=true` for NB, `durable=false` for SB (which is write-heavy).

### `abort`

```json
{"op": "abort"}
```

Force a rollback of the entire transaction. Useful when a precondition fails.

### `comment`

```json
{"op": "comment", "comment": "Add bridge br0 by user admin@example.com"}
```

Attaches text to the audit log. Has no effect on the data.

### `assert`

```json
{"op": "assert", "lock": "ovn_northd"}
```

Asserts that the current connection holds the named lock. Fails if the lock has been lost (for example, stolen by another client). Used for leader-election semantics.

## 10.0.9 `monitor` plus `monitor_cond` plus `monitor_cond_since`

### `monitor` (RFC 7047 §4.1.5)

```json
{"method": "monitor", "params": ["Open_vSwitch", "monitor-handle",
   {"Bridge": [{"columns": ["name", "ports"]}]}], "id": 1}
```

The server pushes:

1. **Initial state.** All existing rows.
2. **Incremental update notifications.** One for every change.

The client must handle both the same way.

### `monitor_cond` (OVS 2.6 and later, 2016)

A conditional filter:

```json
{"method": "monitor_cond", "params": ["OVN_Southbound", "handle",
   {"Port_Binding": [
     {"columns": ["chassis", "logical_port"],
      "where": [["chassis", "==", ["uuid", "<my-chassis-uuid>"]]]}
   ]}], "id": 2}
```

A chassis subscribes only to its own `Port_Binding` rows. Bandwidth drops by 50 to 100 times on a large cluster (10,000 ports across 100 chassis is 100 rows per chassis instead of 10,000).

### `monitor_cond_since` (OVS 2.12 and later, 2019)

Replays incrementally from a known state:

```json
{"method": "monitor_cond_since", "params": ["OVN_Southbound", "handle",
   {"Port_Binding": [...]}, "<last_id>"], "id": 3}
```

`last_id` is the transaction ID of the last update the client processed. The server sends every update from `last_id` to the current state, plus the initial state for new rows. The client does not have to reprocess from scratch after a reconnect.

Implementation: the server keeps a ring buffer of recent transactions (about 1,000). If `last_id` is too old (out of buffer), the server falls back to a full snapshot.

OVN 22.06 and later use `monitor_cond_since` to reduce server load when an `ovn-controller` reconnects (for example, after a brief network blip).

## 10.0.10 IDL synchronization model

An OVSDB client typically does not call JSON-RPC directly. It uses an IDL (Interface Definition Language) library. The IDL builds an in-memory replica of the tables and synchronizes it with the server through `monitor` or `monitor_cond`.

### Workflow

1. The client calls `ovsdb_idl_create()` with a schema and a database name.
2. The IDL sends a `monitor` request, pulls the initial state into an in-memory hash table.
3. The client calls `ovsdb_idl_run()` periodically to poll for updates from the server (through `recv()` on the socket).
4. The client reads state synchronously: `ovsrec_bridge_first(idl)`, traverses the linked list of rows.
5. The client modifies state: build a transaction in memory, then call `ovsdb_idl_txn_commit_block()` to flush.
6. The server replies, and the IDL updates the local replica.

### The sequence number `nb_cfg`

OVN uses `NB_Global.nb_cfg` (an incrementing integer) to measure NB-to-SB synchronization lag:

- The CMS commits a transaction to NB, and `nb_cfg` increments.
- `ovn-northd` compiles NB to SB and copies the `nb_cfg` value into `SB_Global.nb_cfg`.
- Each per-chassis `ovn-controller` copies it into `Chassis_Private.nb_cfg`.

Lag measurement:

```bash
[host]# ovn-nbctl get NB_Global . nb_cfg
142
[host]# ovn-sbctl get SB_Global . nb_cfg
138
[host]# ovn-sbctl get Chassis_Private chassis-A nb_cfg
136
```

Healthy: NB minus SB is at most 5; SB minus Chassis is at most 5. Critical: lag above 100 (compile bottleneck or cluster congestion).

### Connection failure recovery

The IDL library has 3 reconnect strategies:

- **passive_reconnect.** Wait for the application to call `run()` and re-trigger the reconnect.
- **active_reconnect** (default). The library reconnects on its own with backoff (1 second, 2, 4, 8, 16 capped).
- **bounded_recovery** (configurable). Cap the total reconnect time and throw an error past a threshold.

After a reconnect, the IDL prefers `monitor_cond_since` with the last `last_id` it had synchronized. If that fails (because the server's ring buffer is exhausted), it falls back to `monitor_cond` with the initial state.

## 10.0.11 Schema evolution and the `needs-conversion` flow

An OVS or OVN upgrade can change the schema (add a column, add a table). The process:

### Backward-compatible change

A new schema with an extra column or table (with a default value for old rows). `ovsdb-server` handles it automatically: old rows take the default and new rows take user-provided values. No migration is needed.

### Major schema change

A new schema renames a column or changes a type. The server must migrate the data. Workflow:

```bash
# Step 1: compare the current schema with the new one
[host]# ovsdb-client needs-conversion <db> <new-schema-file>
yes

# Step 2: stop the server (or signal it for live convert)
[host]# systemctl stop ovsdb-server

# Step 3: convert offline
[host]# ovsdb-tool convert <db-file> <new-schema-file>

# Step 4: restart
[host]# systemctl start ovsdb-server
```

Or live convert (no downtime, OVS 2.5 and later):

```bash
[host]# ovsdb-client convert tcp:server:6640 OVN_Northbound /usr/share/ovn/ovn-nb.ovsschema
```

OVS Raft cluster: convert is atomic across the cluster. The leader broadcasts the schema and the followers apply it inside the same transaction. A cluster-wide rolling upgrade still needs coordination (see Part 9.12 and Part 10.6).

## 10.0.12 Anatomy Template A: `ovsdb-client get-schema` JSON

> **Context for the run:** inspect the schema to understand the database structure or to generate code (a Python IDL stub).

```json
{
  "name": "Open_vSwitch",
  "version": "8.4.0",
  "cksum": "1234567890 12345",
  "tables": {
    "Open_vSwitch": {
      "columns": {
        "bridges": {"type": {"key": {"type": "uuid", "refTable": "Bridge"},
                              "min": 0, "max": "unlimited"}},
        ...
      },
      "isRoot": true,
      "maxRows": 1
    },
    "Bridge": {...},
    ...
  }
}
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| `name` | Match expected DB | Different (multi-DB server) | Empty | DB does not exist; check `list-dbs` |
| `version` | Match baseline | Mismatch (after upgrade) | Different major | Schema drift; run `needs-conversion` |
| `cksum` | Stable | Changes after upgrade | Changes unplanned | Schema patched manually |
| `tables` count | Match expected | Missing 1 or 2 tables | Many missing | Schema corrupt |
| `isRoot` flag | True for one table | True for more than one | False for every table | Schema invalid |
| Column `type.key.type` | scalar or uuid | Custom type | Empty | Schema parse error |
| Column `type.refTable` | Valid table name | Reference table does not exist | Empty for a UUID column | Schema invalid |

**Triage scenarios:**

- `cksum` changes unexpectedly: someone modified the schema by hand. Run `ovsdb-tool show-log <db-file>` to inspect the transaction history; restore from backup.
- `version` mismatch between the NB cluster and the SB cluster: northd compile fails. Run `convert` on the lagging cluster.
- A column reference table is empty: the schema is corrupt. Restore.

## 10.0.13 Capstone POE: `monitor` versus `monitor_cond` for a 100,000-row table

> **Context:** an OVN cluster whose SB DB holds 100,000 `Port_Binding` rows. 100 chassis each need to subscribe to state. An engineer proposes "use plain `monitor`; `monitor_cond` is not necessary."

**Claim to verify:** "`monitor` is enough for 100,000 rows. `monitor_cond` is just an unnecessary optimization."

**Predict:** wrong. Bandwidth and memory blow up on a large cluster.

**Observe (hypothetical):**

- One chassis using `monitor` for 100,000 `Port_Binding` rows: the initial state is about 50 MB of JSON over the socket. Each update notification is sent to every chassis (the notification is small, but N equals 100).
- 100 chassis times 50 MB equals 5 GB of initial sync at cluster startup.
- Update rate: typically 100 transactions per second. Each transaction is broadcast to all 100 chassis. The server CPU spikes when 100 clients pull at the same time.
- `ovsdb-server` memory: the server keeps a "monitor session" per client with a column subset. 100 sessions times 100,000 rows equals 10 million tracking entries.

With `monitor_cond` (each client subscribes only to relevant rows):

- One chassis subscribes to about 1,000 `Port_Binding` rows (only the bindings that mention that chassis). The initial state is about 500 KB.
- 100 chassis times 500 KB equals 50 MB. A 100-times reduction.
- Update broadcast goes only to the chassis that care (the server filters by the `where` clause). Most updates reach 1 or 2 chassis instead of 100.

**Explain:**

1. **`monitor` does not scale.** Bandwidth is O(N times M) (N clients times M rows). High M means bandwidth explodes.
2. **`monitor_cond`** is O(N times M_filtered) where `M_filtered` is typically below M divided by 100. Bandwidth, CPU, and memory drop by 50 to 100 times.
3. **OVN intentionally** uses `monitor_cond` by default for `ovn-controller`. That is why OVN scales to 1,000 chassis.
4. **When is plain `monitor` enough.** A small DB (below 1,000 rows) or 1 to 2 clients only (test or debug). A production cluster always uses `monitor_cond`.

**Falsification test:**

- Set up an OVN cluster with 10 chassis and 10,000 `Port_Binding` rows.
- Disable `monitor_cond` (force `monitor`): measure bandwidth, memory, and server CPU.
- Enable `monitor_cond`: measure again.
- If the metrics are equivalent, that is a counter-example. In practice, `monitor_cond` always wins on clusters above 5 chassis.

**Where the claim is wrong:** "an unnecessary optimization" assumes a small scale. The correct framing is: **`monitor_cond` is a design requirement for a large cluster, not an optimization. Plain `monitor` is fit only for small test or debug use.**

> **Rule:** a production OVN cluster with more than 5 chassis or a database with more than 1,000 rows must use `monitor_cond`. Plain `monitor` is for ad-hoc inspection only.

## 10.0.14 Five OVSDB observability and operations topics

This section drills into five core concepts: the RFC 7047 transaction model, the JSON-RPC `monitor` push mechanism, the conditional-subscription extension `monitor_cond`, the `ovsdb-server` role system for authorization, and the `ovsdb-client` CLI as a low-level observability surface.

### 10.0.14.1 The OVSDB transaction model (`OVSDB transaction model`)

**Concept.** The OVSDB transaction model is the set of rules that lets several operations (`insert`, `select`, `update`, `mutate`, `delete`, `wait`, `commit`, `abort`, `comment`, `assert`) be batched into a single atomic unit and sent to `ovsdb-server` through the JSON-RPC method `transact`. Either the entire group of operations succeeds and changes the database, or all of them roll back and leave the database untouched. This "all-or-nothing" property is the core of ACID atomicity, mapped to the message structure in RFC 7047 §4.1.3.

**History and background.** RFC 7047 was published in December 2013 to formalize behavior that `ovsdb-server` had implemented since OVS 1.x (2009 to 2010). Nicira originally built OVSDB as a minimal database for `ovs-vswitchd` configuration. The IETF community standardized the protocol so that third parties (NSX, OpenStack Neutron) could integrate. Before the RFC, every vendor implemented its own client based on the OVS source code. After the RFC, the standard became unambiguous.

**Position in the architecture.** The transaction model lives at the protocol layer (the same layer as JSON-RPC), above the storage layer (the OVSDB log file) and below the IDL and CLI layer. The server processes a transaction in `ovsdb_execute_compose` (file `ovsdb/execution.c`, OVS branch-2.17), which is called from `ovsdb_jsonrpc_session_got_request` when a `transact` message arrives.

**Role and the problem solved.** Role: provide an atomic API so the client changes the database consistently. Problem solved: avoid "half-configured" state (a Bridge created without a Port linked), avoid race conditions when several clients write at the same time (each transaction is sequential and the server holds a lock during execution), and allow precondition checks through `wait` and `assert`.

**How it works.** The server receives a message `{"method":"transact","params":[<db>, <op1>, <op2>, ...]}`. Each operation runs sequentially against an in-memory snapshot of the database. If any operation reports an error (constraint violation, type mismatch, condition not met), the server rolls back the entire transaction and replies `{"error": ...}`. If every operation succeeds, the server appends to the log file (durable) or to memory only (non-durable, depending on `commit.durable`), applies the change to the live database, broadcasts a notification to monitor sessions, and replies `{"result":[<r1>, <r2>, ...]}` to the client.

**Usage workflow.** Typical client: build the operation array in memory (for example, `ovs-vsctl` through the IDL), then call `ovsdb-client transact '<json>'` or `ovsdb_idl_txn_commit`. Wait for the reply. On error, parse the `error` field, log it, and either retry or abort. In a shell:

```bash
[host]# ovsdb-client transact unix:/var/run/openvswitch/db.sock \
  '["Open_vSwitch",
    {"op":"insert","table":"Bridge","row":{"name":"br-test"},"uuid-name":"b"},
    {"op":"mutate","table":"Open_vSwitch","where":[],
     "mutations":[["bridges","insert",["set",[["named-uuid","b"]]]]]}]'
```

**Reply structure.** The server replies with a `result` array of the same length as the operation array. Each element is an object: `{"uuid":[...]}` for `insert`, `{"count":N}` for `update`, `mutate`, or `delete`, `{"rows":[...]}` for `select`, or `{}` for `commit` or `abort`. If one operation fails partway through, the corresponding element is `{"error":"...","details":"..."}` and the elements that follow are `null`. The client must inspect every element instead of assuming success.

**When something goes wrong.** Common symptoms: a transaction hangs (the server is slow committing through Raft); the reply says `error: "constraint violation"` (a missing UUID reference); the reply says `error: "referential integrity violation"` (delete of a Bridge that still has Ports). Diagnostic: `ovs-appctl -t ovsdb-server coverage/show | grep txn` to inspect the `transaction/commit` counter. Read `/var/log/openvswitch/ovsdb-server.log` for the exact error.

**Importance.** Every CLI tool (`ovs-vsctl`, `ovn-nbctl`, `ovn-sbctl`) ultimately builds a transaction object and sends it through the `transact` method. Understanding the atomic boundary lets the engineer combine `ovs-vsctl -- ... -- ...` commands into one transaction instead of running them as separate transactions, which avoids half-applied state.

**Version differences.** OVS 2.5 added the `wait` operation. OVS 2.7 added cluster mode (Raft), which made `durable=true` the default for NB. Pre-2017 the server was standalone only and commit was O(disk fsync). After Raft, commit is O(majority replication) and is often faster in practice because the writes pipeline.

**Source code reference.** `ovsdb/execution.c` functions `ovsdb_execute_compose` and `ovsdb_execute`; `ovsdb/transaction.c` function `ovsdb_txn_propose_commit_block`; OVS branch-2.17.9.

**Comparison with other systems.** PostgreSQL transactions have explicit BEGIN, COMMIT, and ROLLBACK, plus nested savepoints and multi-statement scope. An OVSDB transaction is always single-shot, with no nested savepoints. In return the JSON-RPC framing is simpler than ODBC or protobuf and the latency is low when the database is small.

### 10.0.14.2 JSON-RPC monitor (`JSON-RPC monitor`)

**Concept.** `monitor` is the JSON-RPC method a client uses to register for server-pushed notifications when the contents of a table change. Unlike periodic polling, the server keeps the session open and pushes an `update` notification as soon as a transaction commits a change to a row in the subscription.

**History and background.** RFC 7047 §4.1.5 specifies `monitor` together with `monitor_cancel`. OVS 2.6 (2016) added `monitor_cond` for conditional filtering. OVS 2.12 (2019) added `monitor_cond_since` for replay. Each extension addressed a new bottleneck that emerged as OVN clusters grew to thousands of chassis.

**Position in the architecture.** The method lives at the JSON-RPC server layer. The server keeps a `struct ovsdb_jsonrpc_monitor` per session and registers a callback with the transaction system, so it gets called after every commit. The IDL client (Part 10.4) is the main consumer.

**Role and the problem solved.** Role: replace polling with push to reduce latency and server load. Problem solved: previously the client had to `select *` every few seconds to detect change (which wasted CPU and bandwidth and could miss short-lived events). With `monitor`, the client knows the exact moment and exact content of every change.

**Importance.** The foundation for every OVS and OVN daemon: `ovn-controller` watches SB, `ovn-northd` watches NB, `ovs-vswitchd` watches the `Open_vSwitch` DB, `neutron-server` watches NB through the Python IDL. Without `monitor`, the OVN event-driven architecture would not be feasible.

**How it works.** The client sends:

```json
{"method":"monitor",
 "params":["Open_vSwitch","monitor-handle-1",
           {"Bridge":[{"columns":["name","ports","fail_mode"]}]}],
 "id":1}
```

The server replies immediately with a snapshot of the existing rows:

```json
{"result":{"Bridge":{"<uuid1>":{"new":{"name":"br0","ports":[...],"fail_mode":"secure"}}}},"id":1,"error":null}
```

After that, every transaction that touches `Bridge` triggers a notification (note that `id: null` marks a notification, no reply expected):

```json
{"method":"update",
 "params":["monitor-handle-1",
           {"Bridge":{"<uuid1>":{"old":{"fail_mode":"secure"},"new":{"fail_mode":"standalone"}}}}],
 "id":null}
```

**Classification: `update`, `update2`, `update3`.** `update` is the basic RFC 7047 form (sends both `old` and `new`), `update2` (OVS 2.6 and later) sends a smaller delta, `update3` (OVS 2.12 and later) carries `last_id` for `monitor_cond_since`.

**When something goes wrong.** Symptoms: the client receives no events even though the database changed (verify the `monitor-handle` matches; check `ovs-appctl -t ovsdb-server cluster/status`); the session is reset (check the echo keepalive, default probe is 5 seconds); the server pushes duplicate events on reconnect (the fix is to use `monitor_cond_since`).

**Source code reference.** `ovsdb/jsonrpc-server.c` functions `ovsdb_jsonrpc_monitor_create` and `ovsdb_jsonrpc_monitor_compose_update`; `ovsdb/monitor.c` function `ovsdb_monitor_changes_classify`; OVS branch-2.17.9.

**Comparison with other systems.** etcd's `Watch` API streams a key prefix (gRPC). ZooKeeper's `getChildren(watch=true)` gives a one-shot watch (re-register after each event). OVSDB monitor is session-bound and continuous, no re-register after each event, which is closer to etcd than to ZooKeeper.

### 10.0.14.3 Conditional replication (`monitor_cond`)

**Concept.** The `monitor_cond` method extends the basic `monitor` method by letting a client attach a `where` predicate to its subscription. The server then sends update notifications only for rows that satisfy the predicate. Rows that fail the predicate are hidden from the client, even if they exist in the table. This filter matters for large clusters where each client only cares about a subset of the database.

**History and background.** OVS 2.6 (October 2016) added `monitor_cond` to solve a scale problem in the OVN Southbound database. Before this change, every `ovn-controller` had to subscribe to the entire `Port_Binding` table. On a large cluster the table can hold tens of thousands of rows, and the bandwidth grew on the order of N times N (N chassis multiplied by N ports), because each chassis received updates about every port on every other chassis. With `monitor_cond`, each controller subscribes only to the rows that mention its own chassis, and the bandwidth grows linearly with N.

**Position in the architecture.** Conditional replication runs at the same JSON-RPC layer as the basic `monitor`. The server evaluates the predicate inside the hot-path commit code (after the transaction has been applied to in-memory state, but before the update message is composed for the client). On the client side, the IDL library calls `ovsdb_idl_set_condition()` to set or change the condition at runtime.

**Role and the problem solved.** The role is server-side filtering, which reduces both client bandwidth and client CPU. The problem solved is the fan-out storm. Without filtering, one transaction that updates a single row forces the server to broadcast that row to every connected client, even to clients that have no interest in it. Predicate evaluation does have a cost, but the cost is much smaller than the cost of sending the row over the wire and parsing the JSON on the client.

**How it works.** A client sends a request that carries the condition along with the schema fragment to monitor:

```json
{"method":"monitor_cond",
 "params":["OVN_Southbound","sbdb-handle",
           {"Port_Binding":[
             {"columns":["chassis","logical_port","tunnel_key"],
              "where":[["chassis","==",["uuid","aaaa-bbbb-cccc-dddd"]]]}]}],
 "id":2}
```

The server compiles the predicate into an internal filter, represented by the `ovsdb_condition` structure, and attaches the filter to the monitor session. On each commit, the server walks the list of rows that the transaction touched, evaluates the predicate against each one, and pushes a notification only for the rows that pass.

**Supported predicates.** A predicate has the format `[<column>, <function>, <value>]`. The supported functions are `==`, `!=`, `<`, `<=`, `>=`, `>`, `includes`, and `excludes`. When multiple predicates appear in the same `where` array, the server treats them as a logical AND. For columns whose type is a set (notation `set<T>`), the function `includes` tests whether the given value is a member of the set.

**Runtime condition change.** The client calls `monitor_cond_change` with the old session handle plus the new condition. The server computes the difference between the old and new condition, sends an `update` notification for each row that newly passes the predicate (the initial state of those rows), and sends a delete-style update for each row that newly fails the predicate. The client does not have to re-register the session, and no state is lost.

**When something goes wrong.** If the predicate has a syntax error, the server replies with an error response on the original `monitor_cond` request. If the client appears to miss rows that it expected, the most common cause is a predicate that is too narrow. Debug by running `ovsdb-client monitor-cond <db> <table> '[]'` with an empty predicate, then compare the row set with the original subscription.

**Version differences.** `monitor_cond` first shipped in OVS 2.6 (2016). OVS 2.12 (2019) added `monitor_cond_since`, which replays updates starting from a `last_txn_id`. The replay form is useful after a short reconnect, because the client does not have to re-fetch the entire monitored set. OVN 22.06 prefers `monitor_cond_since` to reduce SBDB CPU after a network blip.

**Source code reference.** The server-side handler for `monitor_cond_change` requests is `ovsdb_jsonrpc_monitor_cond_change` in `ovsdb/jsonrpc-server.c`. The function that builds and tracks the per-table condition for a monitor is `ovsdb_monitor_table_condition_create` in `ovsdb/monitor.c`. On the client side, the IDL function that sets or changes the condition is `ovsdb_idl_set_condition` in `lib/ovsdb-idl.c`. All three names are verified at the OVS v2.17.9 baseline (commit `0bea06d9957e3966d94c48873cd9afefba1c2677`, 2024-02-08).

**Comparison with other systems.** PostgreSQL logical replication has had publication-level row filters since PostgreSQL 10, which gives a similar ability to limit what a subscriber receives. Kafka Consumer Group filtering happens on the client, not the broker. OVSDB intentionally chose server-side filtering so that cluster-wide bandwidth stays small, which matters when the database and clients live on opposite ends of a WAN link, as is common for multi-region OVN deployments.

### 10.0.14.4 ovsdb-server roles (`ovsdb-server roles`)

**Concept.** A role is a set of permissions assigned to a client. The client identity comes either from the Common Name (CN) field of the TLS certificate or from the `role` column of the `Manager` table. The role decides which tables and which columns the client can read or write. Roles are the core of OVSDB role-based access control (RBAC). The server applies the role check at the JSON-RPC layer, before the transaction reaches the executor.

**History and background.** OVS 2.7 (February 2017) added RBAC to support OVN multi-tenant deployments. The goal was to make sure that each per-chassis `ovn-controller` could only write its own records into the Southbound database, and could not impersonate another chassis. Before OVS 2.7, any client that presented a valid TLS certificate had full write access to the entire database, which was a privilege-escalation risk in multi-tenant clusters.

**Position in the architecture.** The RBAC engine lives inside `ovsdb-server`, between the JSON-RPC parser and the transaction executor. Roles are stored in two special tables: `RBAC_Role` and `RBAC_Permission`, which are part of the database that needs protection (for example, the OVN Southbound database). A session picks up its role either from the `Manager.role` column (when the server accepted an inbound connection) or from a TLS CN match (when the server made an outbound connection).

**Role and the problem solved.** The role is fine-grained authorization tied to identity. The problem solved is multi-tenant isolation. RBAC stops a rogue chassis from overwriting records that belong to another chassis. It also stops a rogue administrator from writing to tables outside their area of responsibility, and it lets the deployment satisfy compliance requirements that demand per-tenant access control.

**Standard role classification.** The OVN Southbound database ships with two main roles. The first is `ovn-controller`, used by every per-chassis daemon. It is permitted to write to `Chassis_Private` and to the rows of `Port_Binding` whose `chassis_uuid` matches the controller's identity. The second is `ovn-controller-vtep`, used by VTEP gateway controllers. Operators can define custom roles in the same tables for additional multi-tenant scenarios.

**How it works.** When a client sends a `transact` request, the server looks up the role attached to the session, either through the `Manager.role` column or through the TLS CN. For each operation inside the transaction, the server consults the row of `RBAC_Permission` that matches the role and the target table. Three columns of that row govern the decision:

- `authorization`. A regular expression or literal that checks whether some column of the new row (for example `name`) matches the client identity. A match permits the operation; a miss rejects it.
- `insert_delete`. A boolean that controls whether the role may insert or delete rows in the table.
- `update`. A list of column names that the role may modify on existing rows.

**Configuration workflow.** Create the role and permission in a single transaction:

```bash
[host]# ovn-sbctl -- --id=@perm create RBAC_Permission \
    table=Chassis_Private authorization='name' insert_delete=true \
    update='["nb_cfg","nb_cfg_timestamp","external_ids"]' \
  -- --id=@role create RBAC_Role name=ovn-controller \
    permissions:Chassis_Private=@perm
```

Then update the `Manager` row so that inbound connections pick up the new role:

```bash
[host]# ovn-sbctl set-connection role=ovn-controller pssl:6642
```

**When something goes wrong.** A typical symptom is the client reporting `error: "RBAC rules for role 'X' do not allow modification of column Y"`. The diagnostic flow is to run `ovn-sbctl list RBAC_Role`, then `ovn-sbctl list RBAC_Permission`, and finally to compare the certificate CN against the value listed in the `authorization` column of the matching row. A common pitfall is a certificate CN that contains the fully qualified hostname while the role expects the short name. The fix is to standardize the CN format when issuing the certificate.

**Version differences.** RBAC first shipped in OVS 2.7 (2017). OVS 2.13 (2020) improved its performance through rule-lookup caching. Current OVN 22.03 and later distributions include the `ovn-controller` role in the default Southbound database, so an operator does not have to create it from scratch. Releases earlier than OVS 2.7 only had authentication (TLS) without any authorization layer.

**Source code reference.** The server-side enforcement functions are `ovsdb_rbac_insert` and `ovsdb_rbac_update` in `ovsdb/rbac.c`. The session attribute that links a JSON-RPC session to its role is set inside `ovsdb_jsonrpc_session_set_options` in `ovsdb/jsonrpc-server.c`. All three names are verified at the OVS v2.17.9 baseline (commit `0bea06d9957e3966d94c48873cd9afefba1c2677`, 2024-02-08).

**Comparison with other systems.** PostgreSQL has the traditional ROLE plus GRANT and REVOKE machinery, with table-level and column-level granularity. MySQL has a user-and-host pair tied to a privilege bitmask. OVSDB RBAC is lighter than either system, but it specializes in the "self-write only" model (a client can only write rows that name its own identity), which is awkward to express in pure SQL GRANT statements.

### 10.0.14.5 ovsdb-client (`ovsdb-client`)

**Concept.** `ovsdb-client(1)` is the general-purpose CLI that interacts directly with `ovsdb-server` through JSON-RPC. Unlike `ovs-vsctl` (which is specific to `vswitch.ovsschema`) or `ovn-nbctl` and `ovn-sbctl` (which target NB and SB), `ovsdb-client` is generic and works against any schema: `list-dbs`, `get-schema`, `dump`, `monitor`, `transact`, `lock`, `wait`, `backup`, `restore`, `convert`.

**History and background.** Present since OVS 1.x alongside `ovsdb-server`. Nicira originally used `ovsdb-client` for debugging before `ovs-vsctl` was written for operator convenience. Since then `ovsdb-client` has remained the lowest-level tool, used in test frameworks, automation scripts, and troubleshooting when `ovs-vsctl` fails.

**Position in the architecture.** Lives at the tool layer and uses the OVS IDL and JSON-RPC client libraries `lib/jsonrpc.c` and `lib/ovsdb-idl.c`. Connects to `ovsdb-server` through a Unix socket, TCP, or SSL. It is an alternative to the Python or C++ IDL bindings.

**Role and the problem solved.** Role: a low-level observability and manipulation surface. Problem solved: when `ovs-vsctl` reports an unclear error, `ovsdb-client` can show the raw schema and data for debugging. When a small script needs to monitor changes against a custom schema (OVN, a custom DB), it avoids writing a full Python IDL. When a backup or restore is needed without `ovsdb-tool` offline (the server is running), `ovsdb-client` does the job.

**Main commands and classification.**

- `list-dbs`: list the databases served by the server.
- `get-schema <db>`: dump the schema as JSON.
- `list-tables <db>`: list the tables.
- `dump <db> [<table>]`: dump the contents.
- `monitor <db> <table> [<columns>]`: subscribe for push (calls the `monitor` method).
- `monitor-cond <db> <table> '<condition>' [<columns>]`: conditional subscribe.
- `transact <db> '<operations>'`: send an arbitrary transaction.
- `lock <db> <lock-name>` plus `unlock`: a leader-election primitive.
- `backup`: dump the database to stdout (point-in-time snapshot).
- `restore`: load the database from stdin (the server must be empty).
- `convert <db> <new-schema>`: live schema upgrade.
- `wait`: block until the database reaches an expected state.

**Usage workflow.**

```bash
# List the databases
[host]# ovsdb-client list-dbs unix:/var/run/openvswitch/db.sock
Open_vSwitch

# Dump a specific bridge
[host]# ovsdb-client dump unix:/var/run/openvswitch/db.sock Open_vSwitch Bridge name fail_mode

# Conditional subscribe
[host]# ovsdb-client monitor-cond Open_vSwitch Bridge \
    '["Bridge",{"where":[["name","==","br0"]]}]' name,fail_mode

# Backup
[host]# ovsdb-client backup unix:/var/run/openvswitch/db.sock > /backup/conf-$(date +%F).db
```

**When something goes wrong.** `connection refused`: the server is not running or the socket path is wrong. `database not found`: the database name is wrong (case-sensitive, for example `Open_vSwitch` not `open_vswitch`). `transaction error`: an operation does not match the schema; check `get-schema` first.

**Version differences.** OVS 2.6 added `monitor-cond`. OVS 2.12 added `monitor-cond-since`. OVS 2.13 added the `wait` subcommand. OVN 22.03 and later recommend `ovsdb-client backup` instead of `ovsdb-tool` offline when the cluster runs Raft.

**Observation and verification.** Use `--verbose=jsonrpc:dbg` to print raw JSON-RPC over stderr, useful when debugging a third-party tool. Compare the `dump` output before and after a transaction to verify the change.

**Source code reference.** `ovsdb/ovsdb-client.c` (CLI entry point); `lib/jsonrpc.c` (transport); `lib/ovsdb-idl.c` (high-level monitor binding); OVS branch-2.17.9.

**Comparison with other systems.** PostgreSQL's `psql` mixes meta-commands and SQL. etcd has `etcdctl` with `get`, `put`, and `watch`. Redis has `redis-cli` with `MONITOR`. `ovsdb-client` is closest to `etcdctl` in its watch, lock, and transact model, but it talks JSON-RPC instead of gRPC, which makes manual debugging with `nc` straightforward.

## Key points to remember

First, OVSDB is a database specialized for network configuration. ACID plus real-time subscription is enough for the SDN use case.

Second, the schema-bound design lets `ovs-vsctl` validate transactions before commit, which avoids half-configured state.

Third, `monitor_cond` is the key mechanism for large-scale deployment. The OVN Southbound uses it heavily so that each hypervisor only subscribes to the data that is relevant.

Fourth, the 10 RFC 7047 operations are enough for any OVSDB manipulation. `mutate` is thread-safe; `update` can conflict when concurrent writes happen.

## 10.0.X Real-world incident (axis 17 backfill, 2026-04-27)

### Production case: a `monitor` fanout storm in an OpenStack OVN cluster (curated 2023-Q2 pattern)

**Symptom:** after adding 50 new chassis to an existing OpenStack and OVN production cluster of 200 chassis, the `ovsdb-server` SBDB process on the leader spiked to 80 to 95 percent CPU sustained for about 30 minutes after scale-out. The `ovsdb-server` RSS climbed from 1.5 GB to 4 GB. Existing chassis intermittently logged "ovn-controller disconnected from southbound". Tenant network creation got stuck and no new VMs could be scheduled onto the new chassis.

**Initial diagnosis:**

- `top -p $(pidof ovsdb-server)` showed 90 percent CPU in the leader process.
- `ovs-appctl -t ovsdb-server memory/show` showed `monitors: 250` (up from 200) plus a total monitor-session size above 4 GB.
- `ovs-appctl -t ovsdb-server coverage/show | grep monitor` showed `monitor_compose: 50000/sec` (10 times the baseline).

**Root cause investigation.**

Each new chassis registered one `monitor` session as soon as `ovn-controller` started. The default monitor subscription covered the entire `Port_Binding` table (10,000 rows on a large cluster) instead of the relevant subset. Each transaction commit on the SBDB (for example, `neutron-server` inserting one `Port_Binding`) caused the server to run `monitor_compose` for all 250 sessions. The compose cost is roughly O(N times M) where N is the number of sessions and M is the row count each session cares about. Spike scale: 50 new sessions times 10,000 rows equals 500,000 composition events per `Port_Binding` insert. CPU bound.

**Root cause confirmation per Rule 14.**

- Verified with `ovsdb_jsonrpc_monitor_compose_update` in `ovsdb/jsonrpc-server.c` (verified through MCP, branch-2.17): a hot path called per commit per session.
- Cross-reference: §10.0.14.3 already documents `monitor_cond` as the conditional-filter mitigation.

**Resolution.**

1. Migrate `ovn-controller` configuration to use `monitor_cond` with a chassis-specific filter: subscribe only to `Port_Binding` rows that mention the chassis itself. Reduces M from 10,000 rows to about 50 rows per chassis.
2. Restart `ovn-controller` cluster-wide (rolling, per Part 9.12).
3. Post-fix: `monitor_compose` dropped to 5,000 per second, RSS stable at 800 MB, CPU at the baseline of 15 to 25 percent.

**Lessons learned.**

- Default `monitor` (cover all rows) is acceptable for a small cluster of fewer than 50 chassis, but breaks for a large cluster. Mandatory upgrade to `monitor_cond` at scale above 100 chassis.
- A pre-deployment scale test with a realistic chassis count would have caught the issue before the production rollout.
- Cross-link Part 10.0 §10.0.14.3 (`monitor_cond` protocol detail), Part 10.4 (IDL implementation), and Part 9.26 (the revalidator-storm pattern; a different hot path but a similar lesson about fanout cost).

> **Curated reproduction note:** this case is a pattern curated from public OVS and OVN mailing list discussions in 2022 and 2023, plus findings from the OpenStack `neutron-ovn-agent` ScaleTest report. The specific numbers (200 to 250 chassis, RSS 1.5 GB to 4 GB) are plausible but do not represent a single specific incident; they represent the class of "monitor fanout storm" symptoms.

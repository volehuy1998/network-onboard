## 20.2.1 Ba lớp debug OVN — tại sao `tcpdump` không phải công cụ đầu tiên

Khi một kỹ sư nhận page 2 giờ sáng "VM production không ping được gateway", phản xạ từ thời CCNA là mở `tcpdump` trên NIC. Trong hệ OVN, phản xạ này dẫn đến sai hướng trong 70% field hợp. Lý do: OVN là hệ phân tán ba tầng, và packet có thể bị drop ở ba chỗ khác nhau mà `tcpdump` chỉ nhìn thấy chỗ thứ ba (wire). Hai chỗ đầu (ý đồ cấu hình NB, Logical_Flow SB) không bao giờ xuất hiện trên wire vì chúng là abstraction.

Phần 20.0 §20.0.2 đã được đề cập mô hình 5 lớp kiểm tra (control plane health → logical topology → OpenFlow pipeline → datapath → tunnel/wire). Phần 20.2 đào sâu ba lớp giữa chuyên biệt cho OVN:

**Lớp A — NB ý đồ cấu hình (Northbound Database):** CMS (OpenStack Neutron / OVN-Kubernetes / bare-metal operator) ghi Logical_Switch, Logical_Router, Logical_Switch_Port, ACL, Load_Balancer, NAT, DHCP_Options vào NBDB. Đây là ý đồ cấu hình cấp cao, không chứa OpenFlow. Bug thường gặp lớp A: ACL syntax sai, Port_Group reference object không tồn tại, Load_Balancer có VIP trùng với address khác, DHCP_Options không gắn vào LSP.

**Lớp B — SB Logical_Flow (Southbound Database):** `ovn-northd` compile NB ý đồ cấu hình xuống Logical_Flow trong SBDB (xem Phần 13.8 §13.8.2 thuật toán compile). Mỗi object NB sinh nhiều Logical_Flow. Bug thường gặp lớp B: `ovn-northd` đang paused, mâu thuẫn trạng thái (state mismatch) giữa NB và SB, compile quá lâu (nb_cfg timestamp mâu thuẫn trạng thái (state mismatch) > 5 giây), lflow-cache stale trên ovn-controller.

**Lớp C — OpenFlow trên `br-int` (Dataplane):** `ovn-controller` trên mỗi chassis dịch Logical_Flow subset liên quan chassis cục bộ thành OpenFlow rule, install vào `br-int` qua OpenFlow protocol (xem Phần 13.7 §13.7.4). Bug thường gặp lớp C: Port_Binding chưa claim chassis, I-P engine stuck ở full recompute, OpenFlow table số lớn quá threshold HW offload, conntrack zone collision giữa LS và LR.

> **Key Topic — simulation trước capture:** Với ba lớp trên, thứ tự công cụ đúng là `ovn-trace` → `ofproto/trace` + `ovn-detrace` → `tcpdump`. `ovn-trace` simulation packet qua lớp A+B (không cần traffic thật, đọc snapshot NB+SB). `ofproto/trace` simulation lớp C (đọc OpenFlow flow table). Chỉ khi cả hai simulation báo OK mà traffic vẫn fail mới xuống `tcpdump` để kiểm tra tunnel/wire. Simulation chạy mili giây, không phụ thuộc timing; `tcpdump` có thể mất phút để catch packet hiếm xảy ra.

Nguyên tắc thứ hai là **"ba lớp phải khớp"**. Khi NB ý đồ cấu hình đúng (`ovn-nbctl show`) nhưng SB Logical_Flow thiếu entry (`ovn-sbctl lflow-list`), nguyên nhân nằm ở ovn-northd. Khi SB đủ nhưng OpenFlow thiếu (`ovs-ofctl dump-flows br-int`), nguyên nhân ở ovn-controller. Tra cứu theo hướng ngược từ chỗ missing lên để xác định daemon nào chưa làm xong việc của nó.

Phần còn lại của Phần này cung cấp công cụ chi tiết cho từng lớp: `ovn-trace` (lớp A+B, §20.2.2), `ovn-detrace` chain với `ofproto/trace` (lớp B+C junction, §20.2.3), Port_Binding forensic (lớp B+C junction theo hướng port-centric, §20.2.4), ovn-controller/northd runtime commands (§20.2.5-6), stateful table triage (§20.2.7).

## 20.2.2 `ovn-trace` — simulation packet qua Logical pipeline

`ovn-trace` là công cụ quan trọng nhất để debug lớp A+B. Nó đọc snapshot SBDB tại thời điểm chạy và mô phỏng một packet giả đi qua Logical_Flow pipeline của một datapath cụ thể. Output hiển thị từng table logical mà packet đi qua, flow nào match, action nào fire, kết quả cuối cùng (deliver / drop / reinject).

### 20.2.2.1 Grammar và môi trường chạy

Cú pháp đầy đủ từ `ovn-trace(8)` man page:

```
ovn-trace [options] [datapath] microflow
ovn-trace [options] --detach
```

`datapath` là tên logical switch hoặc logical router (string hoặc UUID). Nếu omit, `ovn-trace` tự suy từ `inport` field trong microflow: nếu inport thuộc LSP thì datapath là LS chứa LSP, nếu inport thuộc LRP thì datapath là LR. Best practice vẫn là specify explicit để tránh ambiguity.

`microflow` là expression viết bằng OVN logical khớp với language (cú pháp giống ACL expression trong NBDB), mô tả đầy đủ packet sẽ được simulate. Prerequisite fields được auto-add (ví dụ `ip4` implied khi khớp với `ip4.src`).

Môi trường chạy yêu cầu: SBDB phải reachable từ host gọi `ovn-trace`, thường qua env `OVN_SB_DB` hoặc `--db` option. Cấu hình đơn giản nhất:

```shell
$ export OVN_SB_DB=tcp:10.0.0.100:6642
$ ovn-trace ls1 'inport == "vm1" && eth.src == aa:bb:cc:dd:ee:01 && eth.dst == aa:bb:cc:dd:ee:02 && ip4 && ip4.src == 10.0.0.1 && ip4.dst == 10.0.0.2 && ip.ttl == 64 && icmp4'
```

### 20.2.2.2 Microflow syntax — 5 class ví dụ

Microflow tuân theo BNF của OVN logical expression (định nghĩa trong `ovn-sb(5)` §Logical_Flow). Prerequisites tự động (ví dụ khớp với `ip4.src` implies `ip4`, match `tcp.dst` implies `tcp` implies `ip`). Các operator logic: `==`, `!=`, `<`, `<=`, `>`, `>=`, `&&`, `||`, `!`.

**Class 1 — L2 unicast:**

```
inport == "vm1" &&
eth.src == aa:bb:cc:dd:ee:01 &&
eth.dst == aa:bb:cc:dd:ee:02
```

Case thuần L2, packet không có IP layer. Dùng để kiểm chứng `ls_in_l2_lkup` hoạt động đúng với FDB static hoặc MAC_Binding entry.

**Class 2 — L2 broadcast / multicast:**

```
inport == "vm1" &&
eth.src == aa:bb:cc:dd:ee:01 &&
eth.dst == ff:ff:ff:ff:ff:ff
```

MAC đích là broadcast. `ovn-trace` sẽ route qua Multicast_Group (MC_FLOOD) và output tới mọi port trong LS trừ inport. Dùng để kiểm chứng ARP request reach mọi VM.

**Class 3 — L3 unicast (cross-subnet qua LR):**

```
inport == "vm1" &&
eth.src == aa:bb:cc:dd:ee:01 &&
eth.dst == 00:00:00:ff:fe:01 &&  # MAC của LRP (default gateway)
ip4 &&
ip4.src == 10.0.0.1 &&
ip4.dst == 10.0.1.5 &&           # subnet khác
ip.ttl == 64 &&
icmp4
```

Packet ICMP cross-subnet. `ovn-trace` route qua LS ingress → LR ingress pipeline (lr_in_ip_routing stage) → LR egress → LS đích ingress → LSP đích. Dùng để kiểm chứng route table + ARP resolve cho next-hop.

**Class 4 — ARP request:**

```
inport == "vm1" &&
eth.src == aa:bb:cc:dd:ee:01 &&
eth.dst == ff:ff:ff:ff:ff:ff &&
arp &&
arp.op == 1 &&                   # request
arp.sha == aa:bb:cc:dd:ee:01 &&
arp.spa == 10.0.0.1 &&
arp.tha == 00:00:00:00:00:00 &&
arp.tpa == 10.0.0.254            # gateway IP
```

ARP request tìm MAC của gateway. Trong OVN, stage `ls_in_arp_rsp` (table 19 ingress LS) có thể auto-reply nếu biết MAC của IP đích (ARP Suppressor, xem Phần 18). Output `ovn-trace` sẽ thấy action `build_arp_resp(...)` + `outport = inport; output;` (reflect reply về source).

**Class 5 — DHCP Discover:**

```
inport == "vm1" &&
eth.src == aa:bb:cc:dd:ee:01 &&
eth.dst == ff:ff:ff:ff:ff:ff &&
ip4 &&
ip4.src == 0.0.0.0 &&
ip4.dst == 255.255.255.255 &&
ip.ttl == 64 &&
udp &&
udp.src == 68 &&
udp.dst == 67
```

Gói DHCP Discover broadcast. Stage `ls_in_dhcp_options` (table 20) sẽ fire `put_dhcp_opts(...)` inline-reply nếu LSP có `dhcpv4_options` attached. Dùng để kiểm chứng native DHCP cấu hình (Phần 13.10).

### 20.2.2.3 Option catalog — 11 option

Table tổng hợp 11 option từ `ovn-trace(8)`, với semantics và khi nào dùng:

| Option | Mô tả | Khi dùng |
|--------|-------|----------|
| `--detailed` | (Mặc định) Hiển thị từng table visited với priority, match, action, omit indentation tail recursion, filter flow trivial `match=1` | Default cho mọi debug thông thường. Đủ context để nhận diện flow nào fire |
| `--summary` | Format programming-language-like với indentation, cho thấy pipeline structure như pseudo-code | Khi cần nhìn tổng thể pipeline nhánh nào đi đâu, dễ đọc flow control |
| `--minimal` | Chỉ show action modify packet data (set_field, ct, dec_ttl) và output action, operand reduced to constant | Regression test, CI pipeline — output stable nhất, ít noise version lệch state |
| `--all` | Show cả ba (`--detailed` + `--summary` + `--minimal`) xếp chồng | Lần đầu debug một pipeline lạ, cần mọi angle |
| `--ovs[=remote]` | Kèm theo output OpenFlow flow tương ứng trên `br-int`. Optional `remote` specifying OVS connection (default unix socket) | Chuyển từ logical sang physical, xem actual OpenFlow sinh ra từ logical flow |
| `--ct=flags` | Set connection tracking state flags (new, est, rel, rpl, inv, trk, dnat, snat), simulation packet trong state cụ thể thay vì default `+trk+new` | Debug stateful ACL, LB reply path, NAT reverse flow. Ví dụ `--ct=trk,est,rpl` simulate reply của established connection |
| `--lb-dst=ip[:port]` | Force Load_Balancer chọn backend này thay vì hash | Debug LB affinity, isolate xem logic LB có route đúng đến một backend cụ thể không |
| `--select-id=id` | Force group bucket chọn bucket id này | Debug ECMP, multipath group — isolate một path cụ thể |
| `--friendly-names` | (Mặc định) Substitute LSP/LRP name thay vì UUID | Default, dễ đọc |
| `--no-friendly-names` | Show raw UUID | Khi cần copy-paste output vào script grep chính xác UUID |
| `--db database` | OVSDB remote thay vì `OVN_SB_DB` env | Debug cluster OVN khác, hoặc từ jumphost không có env |

Ngoài ra, `-v[spec]` set logging level (`-v dbg` rất verbose), và `--detach` chạy như daemon (hiếm dùng).

**Kịch bản bẻ gãy (troubleshooting cues):**

- **Nếu output kết thúc bằng `drop;` ở stage `ls_in_acl`** → ACL policy đang block. Kiểm chứng bằng `ovn-nbctl --bare --columns=direction,priority,match,action find ACL`. Fix: chỉnh ACL hoặc thêm `allow` rule priority cao hơn.
- **Nếu output kết thúc tại `lr_in_arp_resolve` không có `eth.dst = ...`** → ARP chưa resolve next-hop, MAC_Binding thiếu entry. Kiểm chứng bằng `ovn-sbctl find MAC_Binding ip=<next-hop>`. Fix: chờ ARP resolve (trigger bằng `ovn-appctl -t ovn-controller inject-pkt` ARP request), hoặc set `mac_addresses` static trên LRP.
- **Nếu không thấy block `ingress(dp="<next-dp>")` sau patch port output** → Logical_Flow thiếu entry cho LR/LS tiếp theo. Nguyên nhân thường là ovn-northd chưa compile xong (xem §20.2.6 `inc-engine/show-stats`) hoặc NB ý đồ cấu hình sai (LRP không gắn với LS). Kiểm chứng bằng `ovn-nbctl show`.
- **Nếu dòng `# reg14=...` parse lỗi (ví dụ "unknown field 'ipv4'")** → microflow syntax sai. OVN dùng `ip4` không phải `ipv4`, `eth.dst` không phải `eth.dst_mac`. Tra chính xác field name trong `ovn-sb(5)` §Logical_Flow.
- **Nếu output theo dõi quá dài (> 200 dòng) với nhiều loop `resubmit`** → pipeline có recursion (thường qua hairpin NAT). Switch sang `--summary` để nhìn structure tổng thể trước khi đi sâu `--detailed`.

### 20.2.2.5 Đọc hiểu output `ovn-trace --summary`

Cùng microflow như §20.2.2.4, nhưng render summary format giống pseudo-code:

```shell
$ ovn-trace --summary ls_a \
    'inport == "vm_a" && eth.src == aa:bb:cc:00:00:01 && eth.dst == 00:00:00:ff:fe:01 &&
     ip4.src == 10.0.0.1 && ip4.dst == 10.0.1.5 && ip.ttl == 64 && icmp4'

# reg14=0x1,dl_src=aa:bb:cc:00:00:01,dl_dst=00:00:00:ff:fe:01,
# nw_src=10.0.0.1,nw_dst=10.0.1.5,nw_proto=1,nw_ttl=64

ingress(dp="ls_a", inport="vm_a") {
    reg0[15] = check_in_port_sec();
    reg0[0] = 1;
    reg0[8] = 1;
    reg0[9] = 1;
    reg8[16..31] = 10;
    outport = "ls_a_rtr";
    output;
    egress(dp="ls_a", inport="vm_a", outport="ls_a_rtr") {
        output;
        /* output to "ls_a_rtr", type "patch" */
        ingress(dp="router1", inport="rtr_a") {
            xreg0[0..47] = 00:00:00:ff:fe:01;
            ip.ttl--;
            reg0 = 10.0.1.5;
            reg1 = 10.0.1.1;
            eth.src = 00:00:00:ff:fe:02;
            outport = "rtr_b";
            flags.loopback = 1;
            eth.dst = aa:bb:cc:00:00:05;
            outport = "rtr_b";
            output;
            /* output to "rtr_b", type "patch" */
            ingress(dp="ls_b", inport="ls_b_rtr") {
                outport = "vm_b";
                output;
                egress(dp="ls_b", inport="ls_b_rtr", outport="vm_b") {
                    output;
                    /* output to "vm_b", type "" */
                };
            };
        };
    };
};
```

Summary loại bỏ stage name + priority + UUID, chỉ giữ action sequence. Indent biểu diễn nesting giữa pipeline. Kỹ sư senior đọc summary nhanh hơn detailed để nhận diện pipeline structure; khi cần đào sâu từng stage thì chuyển sang detailed.

### 20.2.2.6 Đọc hiểu output `ovn-trace --minimal`

Minimal giữ lại chỉ action modify packet data và output action. Operand reduced về constant (ví dụ `eth.src = 00:00:00:ff:fe:02` thay vì `eth.src = mac_of_lrp`):

```shell
$ ovn-trace --minimal ls_a \
    'inport == "vm_a" && eth.src == aa:bb:cc:00:00:01 && eth.dst == 00:00:00:ff:fe:01 &&
     ip4.src == 10.0.0.1 && ip4.dst == 10.0.1.5 && ip.ttl == 64 && icmp4'

# reg14=0x1,dl_src=aa:bb:cc:00:00:01,dl_dst=00:00:00:ff:fe:01,
# nw_src=10.0.0.1,nw_dst=10.0.1.5,nw_proto=1,nw_ttl=64

eth.src = 00:00:00:ff:fe:02;
eth.dst = aa:bb:cc:00:00:05;
ip.ttl--;
output;
/* output to "vm_b", type "" */
```

Đây là format dùng trong regression test: output stable, ít phụ thuộc phiên bản OVN (version khác nhau có thể reorder stage nhưng final packet transform giống nhau). CI pipeline có thể diff `--minimal` output trước/sau upgrade để phát hiện regression silent.

### 20.2.2.7 Đọc hiểu output `ovn-trace --ovs`

Khi muốn bridge từ logical sang physical, `--ovs` kèm theo OpenFlow flow thực tế mà ovn-controller đã cài trên `br-int`. Output mở rộng mỗi stage với OpenFlow rule tương ứng:

```shell

       actions=load:0x1->NXM_NX_REG0[15],resubmit(,9) */
```

`--ovs` yêu cầu ovn-trace có thể kết nối OVS local DB (mặc định `unix:/var/run/openvswitch/db.sock`), thường chỉ chạy được trên chassis có OVS + ovn-controller. Đây là cách nhanh nhất để kiểm chứng ovn-controller dịch Logical_Flow thành OpenFlow đúng chính là flow expected, đối xứng với §20.2.3 ovn-detrace đi ngược chiều.

### 20.2.2.8 Option `--ct` cho stateful simulation

`ovn-trace` default set `ct_state = +trk+new` (packet là NEW connection, chưa được commit vào conntrack). Khi debug reply path của ESTABLISHED connection hoặc kiểm tra NAT reverse, phải set `--ct` thủ công:

```shell
# Simulate reply từ 10.0.1.5 về 10.0.0.1 (sau khi connection đã establish)
$ ovn-trace --ct=trk,est,rpl ls_b \
    'inport == "vm_b" && eth.src == aa:bb:cc:00:00:05 && eth.dst == 00:00:00:ff:fe:02 &&
     ip4.src == 10.0.1.5 && ip4.dst == 10.0.0.1 && ip.ttl == 64 && icmp4'
```

Khi `ct_state` có `+est+rpl`, ACL `allow-related` sẽ khớp với reply → permit. Nếu không set, reply sẽ fall through NEW rules → thường bị drop bởi default deny. Đây là lý do tại sao bug "reply không quay được" thường được reproduce bằng theo dõi NEW thì OK, trace với `+est+rpl` mới phát hiện ACL thiếu allow-related.

Tương tự `--ct=trk,new,inv` (invalid) dùng để kiểm chứng ACL reject invalid state; `--ct=trk,dnat` kiểm chứng LB DNAT reverse path.

### 20.2.2.9 Option `--lb-dst` cho load balancer debug

LB VIP có nhiều backend, OVN chọn backend theo hash 5-tuple. Khi debug một backend cụ thể, thường khó reproduce vì hash của microflow không khớp với backend muốn test. `--lb-dst` force DNAT direct tới backend IP:

```shell
$ ovn-trace --lb-dst=10.0.2.50:80 ls_a \
    'inport == "vm_a" && eth.src == aa:bb:cc:00:00:01 && eth.dst == 00:00:00:ff:fe:01 &&
     ip4.src == 10.0.0.1 && ip4.dst == 10.100.0.1 && ip.ttl == 64 && tcp.dst == 80'
```

Output sẽ force stage `ls_in_lb` DNAT vào `10.0.2.50:80` thay vì hash. Hữu dụng khi backend `10.0.2.50` đang report lỗi intermittent, cần kiểm chứng toàn bộ pipeline từ client tới backend này đi đúng.

## 20.2.3 `ovn-detrace` — từ OpenFlow cookie về Logical_Flow nguồn

`ovn-trace` simulation theo chiều **NB → OpenFlow** (logical xuống physical). `ovn-detrace` đi ngược chiều: đọc OpenFlow cookie trong `ofproto/trace` output (hoặc `dump-flows`) và tra cứu Logical_Flow trong SBDB đã sinh ra cookie đó, và khi applicable, NBDB object (ACL / Load_Balancer / NAT) gốc.

### 20.2.3.1 Vai trò trong workflow

Giả sử `ofproto/trace` báo packet bị drop tại một OpenFlow flow với cookie `0x5d2a7f91`. Hỏi: Logical_Flow nào sinh ra flow này? Object NB nào (ACL / LB / NAT) đã kích hoạt? Thiếu `ovn-detrace`, kỹ sư phải `ovn-sbctl find Logical_Flow ...` thủ công với nhiều guess. Với `ovn-detrace`, chain tự động:

```
ovs-appctl ofproto/trace -m 3 br-int <microflow>
    ↓ (output có cookie field)
ovn-detrace
    ↓ (lookup SB/NB)
annotated output: cookie 0x5d2a7f91 → Logical_Flow ls_in_acl priority 2000
                                    → NBDB ACL name="deny-ssh" in Port_Group "mgmt"
```

### 20.2.3.2 Grammar và env variable

`ovn-detrace` đọc stdin, output stdout:

```
ovn-detrace [options] < file

Common options:
    [-h | --help] [-V | --version]
```

Không có `[datapath] [microflow]` argument vì nó pure filter: input từ stdin (thường pipe từ `ofproto/trace`), output annotated.

Options quan trọng:

| Option | Mô tả | Khi dùng |
|--------|-------|----------|
| `--ovnsb=server` | SBDB remote, default env `OVN_SB_DB` | Khi chạy từ node khác central node, explicit specify SBDB |
| `--ovnnb=server` | NBDB remote, default env `OVN_NB_DB` | Cần NBDB để annotate ACL name, LB name, NAT rule |
| `--ovs=` | Kèm theo decode OVS ofport trong flow | Khi chain có flow reference OVS port cần map về iface name |
| `--ovsdb=server` | OVS DB remote nếu `--ovs` present | Tương tự, explicit khi không default socket |
| `--no-leader-only` | Connect bất kỳ Raft member, không chỉ leader | Leader đang bận, dùng follower để giảm tải |
| `-p privkey.pem` `-c cert.pem` `-C cacert.pem` | TLS certificate cho SBDB/NBDB connection | Cluster có TLS bật (recommended production) |

Thiếu `--ovnnb=...` hoặc env `OVN_NB_DB`, `ovn-detrace` chỉ annotate SB level (Logical_Flow), không mở rộng xuống NB object. Thiếu `--ovs`, không decode ofport.

| `# Logical flow: table=N (stage_name), priority=P, match=..., action=...` | ovn-detrace annotation sau mỗi flow có cookie khớp SBDB | Dòng comment `#` thêm bên dưới mỗi flow, map cookie → Logical_Flow |

| `# Logical input ports: "vm_a" (reg14=0x1)` | ovn-detrace decode `reg14` value → LSP name | Reg14 là register OVN dùng cho inport. Decode giúp đọc khớp với readable |

**Kịch bản bẻ gãy (troubleshooting cues):**

- **Nếu `ovn-detrace` output không có dòng `# Logical flow: ...` sau flow** → cookie trong OpenFlow không khớp với Logical_Flow trong SBDB. Nguyên nhân: SBDB stale (ovn-controller chưa update sau khi ovn-northd compile mới), hoặc chassis đang run OVN version khác với SBDB. Fix: `ovs-appctl -t ovn-controller recompute` hoặc kiểm chứng phiên bản match.
- **Nếu annotation chỉ có `# Logical flow: ...` không có `# Load_Balancer: ...` hoặc `# ACL: ...`** → NBDB connection chưa config. Set env `OVN_NB_DB` hoặc pass `--ovnnb=...` explicit.
- **Nếu output cho flow `priority 0` drop** → không khớp với bất kỳ rule nào cao hơn, fall through default drop. Kiểm chứng bằng `ovn-sbctl lflow-list <datapath> | grep -v priority=0` để xem có flow nào priority cao hơn expected khớp với không.
- **Nếu decode reg14 hiển thị UUID thay vì LSP name** → ovn-detrace không resolve được name, thường do LSP bị rename/delete giữa lúc OpenFlow cài đặt và lúc trace. Force reconnect SBDB.
- **Nếu `ofproto/trace -m 3` báo "Too large" với output cắt ngắn** → table count trên br-int quá nhiều. Giảm `-m 2` hoặc grep output theo cookie specific rồi pipe `ovn-detrace` riêng biệt.

### 20.2.3.4 Pattern diff chassis bằng `ovn-detrace`

Khi cluster có 500+ chassis và một chassis cụ thể (chassis-42) report packet drop, pattern debug: diff OpenFlow flow giữa chassis-42 và peer healthy, pipe qua `ovn-detrace` để nhận diện Logical_Flow nào missing.

```shell
# Trên chassis-42 (suspected)
$ ssh chassis-42 'ovs-ofctl dump-flows br-int table=8,metadata=0x1' > /tmp/ch42_t8.txt

# Trên chassis-peer (healthy)
$ ssh chassis-peer 'ovs-ofctl dump-flows br-int table=8,metadata=0x1' > /tmp/chp_t8.txt

# Diff, focus trên cookie (skip counter bytes/duration)
$ diff <(awk '{print $NF}' /tmp/chp_t8.txt | sort) \
       <(awk '{print $NF}' /tmp/ch42_t8.txt | sort)
> actions=resubmit(,9),cookie=0x7ab19c00  # present on healthy, missing on ch42

# Trace cookie back về Logical_Flow
$ echo 'cookie=0x7ab19c00, table=8, priority=110, metadata=0x1' | ovn-detrace
# Logical flow: table=14 (ls_in_lb), priority=110, ...
# Load_Balancer: name="web-lb" vips={"10.100.0.1:443"=...}
```

Pattern này đặc biệt hữu dụng khi ovn-controller trên chassis-42 bị stale cache hoặc partial sync (rare bug). Fix: `ovs-appctl -t ovn-controller recompute` trên chassis-42.

| `type` | `""` | Port type (xem 8 type dưới) | `""` (empty) là VIF mặc định |
| `up` | `true` | Port đang up (ovn-controller set khi OVS interface link up) | `false` = OVS interface chưa up hoặc link down. Check `ovs-vsctl show` |
| `virtual_parent` | `[]` | UUID virtual parent cho type `virtual` | Required nếu type=`virtual` |

**Kịch bản bẻ gãy:**

- **`chassis: []` mà VM đang chạy** → binding failure. Check `external_ids:iface-id` trên OVS interface (Phần 9.13), ovn-controller log trên chassis target, và `requested_chassis` field.
- **`chassis != requested_chassis`** → orchestrator đã schedule VM sang chassis khác nhưng OVN chưa update. Thường gặp trong live migration (Phần 19). Resolve tự động khi migration complete.
- **`up: false` mà VM đang power on** → OVS interface down. Check `ovs-vsctl list interface <tap>` → `admin_state` + `link_state`. Thường do tap config sai MTU hoặc VLAN.
- **`tunnel_key` trùng giữa 2 port trong cùng datapath** → bug ovn-northd (rất hiếm). Restart northd fix.
- **`type` khác rỗng với LSP VIF** → ovn-northd phát hiện gì đó đặc biệt (ví dụ VIF thành chassisredirect khi LSP chuyển sang gateway port). Kiểm chứng bằng `ovn-nbctl lsp-get-options <lsp>`.

### 20.2.4.2 Type `""` (VIF) — 4 failure mode

VIF là type phổ biến nhất, mặc định cho mọi LSP VM/container. Claim logic: ovn-controller trên chassis nào có OVS interface với `external_ids:iface-id=<LSP_name>` thì chassis đó claim.

**Failure mode 1 — `iface-id` chưa set.** Libvirt/Kubelet plug tap vào br-int nhưng chưa set external-ids (race, xem Phần 20.0 §20.0.7 Case 1). Symptom: `Port_Binding.chassis = []`. Diagnostic:

```shell
# Trên chassis đang chạy VM
$ ovs-vsctl --columns=name,external_ids find Interface
# Tìm tap có iface-id match LSP name. Nếu không có → orchestrator chưa set
```

Fix: `ovs-vsctl set Interface <tap> external_ids:iface-id=<lsp_name>`.

**Failure mode 2 — `iface-id` trùng giữa 2 chassis.** Hai chassis cùng có tap với same iface-id (bug orchestrator, thường khi live migration fail). Symptom: Port_Binding bounce giữa 2 chassis, log ovn-controller thấy "claim/unclaim" liên tục. Diagnostic:

```shell
$ for ch in $(ovn-sbctl --bare --columns=name list Chassis); do
    ssh $ch "ovs-vsctl --columns=name,external_ids find Interface external_ids:iface-id=vm1" | grep -q name && echo "$ch has iface-id vm1"
  done
```

Nếu > 1 chassis report → kill tap trên chassis sai.

**Failure mode 3 — `port_security` mismatch MAC/IP VM thật.** `Logical_Switch_Port.port_security` set `aa:bb:cc:00:00:01 10.0.0.1` nhưng VM thật gửi traffic với MAC khác. Stage `ls_in_check_port_sec` drop. Symptom: `ovn-trace` báo `drop;` tại table 0. Diagnostic:

```shell
$ ovn-nbctl lsp-get-port-security vm1
aa:bb:cc:00:00:01 10.0.0.1

$ ssh chassis-hv1 "ip -br link show tap-vm1"
tap-vm1  UP  aa:bb:cc:00:00:02  # MAC sai, khác port_security
```

Fix: sửa MAC VM hoặc update port_security.

**Failure mode 4 — LSP bị set `addresses=unknown` nhưng vẫn expect chuyển tiếp.** Khi addresses=unknown, OVN flood mọi packet có MAC unknown tới LSP này (mode promiscuous). Nhưng ovn-trace với specific MAC sẽ thấy LSP đích cụ thể, dẫn đến confusion. Symptom: trace OK nhưng traffic thực tế flood mọi VM. Diagnostic:

```shell
$ ovn-nbctl lsp-get-addresses vm1
unknown
```

Fix: nếu không phải promiscuous ý đồ cấu hình, set addresses explicit `ovn-nbctl lsp-set-addresses vm1 "aa:bb:cc:00:00:01 10.0.0.1"`.

### 20.2.4.3 Type `patch` — 2 failure mode

Patch port auto-generated bởi ovn-northd nối LSP type=router với LRP. Không có claim logic (mọi chassis đều có patch flow).

**Failure mode 1 — northd chưa compile patch.** Tạo LRP xong nhưng Port_Binding type=patch chưa xuất hiện. Symptom: LS và LR không kết nối. Diagnostic:

```shell
$ ovn-sbctl --bare --columns=logical_port,type find Port_Binding type=patch
# Kỳ vọng: có entry với logical_port = <LSP-router-name>

$ ovs-appctl -t ovn-northd status
# Kỳ vọng: active
```

Fix: restart northd, hoặc `ovn-appctl -t ovn-northd inc-engine/recompute`.

**Failure mode 2 — patch ports có cycle.** Hiếm, xảy ra khi NB ý đồ cấu hình có LR1 nối LR2 nối LR1 (cycle). ovn-northd phát hiện và refuse compile. Symptom: `ovn-northd.log` có error "cycle detected". Fix: sửa NB topology.

### 20.2.4.4 Type `localnet` — 3 failure mode

localnet bridge LSP tới physical network qua bridge mapping. Không claim chassis (available mọi chassis).

**Failure mode 1 — `bridge-mappings` chưa config trên chassis.** `ovn-controller` không biết map `network_name` tới bridge nào. Symptom: patch port giữa br-int và br-ex không được tạo trên chassis. Diagnostic:

```shell
$ ovs-vsctl get open_vswitch . external_ids:ovn-bridge-mappings
physnet1:br-ex,physnet2:br-provider
# Phải có mapping cho network_name của localnet port
```

Fix: `ovs-vsctl set open_vswitch . external_ids:ovn-bridge-mappings=physnet1:br-ex`.

**Failure mode 2 — VLAN tag trên localnet nhưng physical switch port là access không trunk.** Port localnet với `options:network_name=physnet1 vlan=100` nhưng switch port chỉ allow untagged. Symptom: traffic rời br-ex với VLAN tag nhưng switch drop. Diagnostic: `tcpdump -eni <br-ex-uplink> vlan 100` trên chassis. Fix: set switch port trunk, hoặc thay đổi port sang access untagged.

**Failure mode 3 — localnet xuất hiện trên LS nhưng chassis không cần traffic ra physical.** Tất cả chassis sẽ tạo patch tới br-ex, tốn resource. Symptom: `ovs-vsctl show` thấy patch port thừa trên chassis không serve external. Fix: dùng Logical_Switch có `other_config:localnet_learn_fdb=true` + giới hạn chassis qua option `requested-chassis`.

### 20.2.4.5 Type `chassisredirect` (cr-lrp) — 4 failure mode

chassisredirect là shadow LRP dùng cho distributed gateway — traffic north-bound chuyển từ distributed LR về gateway chassis qua port này. Claim bởi chassis có priority cao nhất trong HA_Chassis_Group (BFD-based election).

**Failure mode 1 — `ha_chassis_group` empty hoặc thiếu HA_Chassis entry.** Không chassis nào claim. Symptom: `Port_Binding.chassis = []` cho cr-lrp. Diagnostic:

```shell
$ ovn-sbctl list Port_Binding cr-lrp-external
# ha_chassis_group: 1234-abcd-...

$ ovn-sbctl list HA_Chassis_Group 1234-abcd-...
# Kiểm tra ha_chassis field có reference
```

Fix: `ovn-nbctl ha-chassis-group-add-chassis <group> <chassis-name> <priority>`.

**Failure mode 2 — BFD không up giữa chassis.** Election thất bại vì gateway chassis không phát hiện peer live. Symptom: `chassis` field bounce giữa nhiều chassis. Diagnostic:

```shell
$ ovs-appctl bfd/show
# Check state UP với mỗi peer

$ ssh gateway-chassis "journalctl -u ovn-controller --since '5 min ago' | grep -i bfd"
```

Fix: check underlay UDP 3784 (BFD) không bị firewall block.

**Failure mode 3 — Priority collision (2 chassis cùng priority).** OVN chọn bằng UUID tie-break, non-deterministic. Symptom: election không stable qua restart. Fix: assign priority khác nhau cho mỗi HA_Chassis member.

**Failure mode 4 — Gateway chassis không có `enable-chassis-as-gw` config.** `ovn-controller` trên gateway chassis không claim dù priority cao nhất. Diagnostic:

```shell
$ ovs-vsctl get open_vswitch . external_ids:ovn-cms-options
enable-chassis-as-gw
```

Fix: `ovs-vsctl set open_vswitch . external_ids:ovn-cms-options=enable-chassis-as-gw`.

### 20.2.4.6 Type `l3gateway` — 2 failure mode

l3gateway là LRP centralized trên một chassis, dùng cho gateway router cũ (pre-distributed). Đang được thay thế bởi chassisredirect + distributed LR. Claim logic: chassis specified trong `options:l3gateway-chassis`.

**Failure mode 1 — `l3gateway-chassis` trỏ chassis không tồn tại hoặc offline.** Port_Binding chassis rỗng. Diagnostic:

```shell
$ ovn-sbctl --columns=options list Port_Binding lrp-external
options: {l3gateway-chassis=chassis-offline-uuid}

$ ovn-sbctl list Chassis | grep chassis-offline-uuid
# Nếu không có → chassis offline
```

Fix: update `options:l3gateway-chassis` về chassis live.

**Failure mode 2 — SNAT rule không effective.** l3gateway xử lý SNAT nhưng rule compile sai thứ tự (sau LB DNAT). Symptom: traffic north-bound không được SNAT. Diagnostic: `ovn-sbctl lflow-list | grep lr_in_unsnat` và `lr_out_snat`. Fix: thường là NB ý đồ cấu hình sai (set NAT priority).

### 20.2.4.7 Type `l2gateway` — 2 failure mode

l2gateway bridge LSP tới hardware VTEP (external L2 switch chạy OVSDB hardware_vtep schema). Claim bởi `options:l2gateway-chassis`.

**Failure mode 1 — OVSDB hardware_vtep không reachable từ chassis.** Symptom: tunnel giữa chassis và hardware VTEP không form. Diagnostic:

```shell
$ ovs-vsctl get open_vswitch . external_ids:ovn-remote-vtep
tcp:10.0.0.50:6632
# Test connectivity:
$ nc -zv 10.0.0.50 6632
```

Fix: check network route + firewall.

**Failure mode 2 — Hardware VTEP schema mismatch.** VTEP spec version cũ không hỗ trợ feature OVN yêu cầu. Fix: upgrade firmware switch hoặc downgrade OVN config.

### 20.2.4.8 Type `localport` — 2 failure mode

localport tồn tại trên mọi chassis với cùng MAC + IP, dùng cho metadata service (169.254.169.254 Kubernetes/OpenStack). Traffic không leave chassis.

**Failure mode 1 — Route trên VM không point tới localport.** Symptom: curl metadata endpoint timeout. Diagnostic trong VM:

```shell
$ ip route get 169.254.169.254
# Phải thấy via eth0 (or similar)
```

Fix: CMS config wrong, route phải inject qua DHCP_Options hoặc static.

**Failure mode 2 — Multiple LSP localport trong cùng LS conflict.** Symptom: metadata request không deliver consistent. Fix: giữ duy nhất 1 localport per LS.

### 20.2.4.9 Type `virtual` — 2 failure mode

virtual port (OVN 22.03+) dùng cho pattern "floating IP" hoặc keepalived VIP. `virtual_parent` point về LSP parent, OVN route traffic virtual IP tới chassis nào đang có parent claim.

**Failure mode 1 — `virtual_parent` không khớp với LSP parent nào.** Symptom: virtual port không functional. Diagnostic:

```shell
$ ovn-sbctl --columns=logical_port,virtual_parent list Port_Binding type=virtual
$ ovn-sbctl list Port_Binding <parent-name>
# Verify parent tồn tại
```

Fix: set lại `options:virtual-parents=<correct-parent>`.

**Failure mode 2 — Parent LSP chassis flap gây virtual port flap.** Symptom: virtual IP reachable intermittent khi parent migrate. Fix: kiểm tra parent HA, giảm tần suất migrate.

### 20.2.4.10 Type `remote` — 1 failure mode

remote type dùng cho OVN interconnect (IC), port từ cluster remote visible trong local SB.

**Failure mode — IC tunnel không up.** Symptom: remote Port_Binding tồn tại nhưng traffic cross-cluster fail. Diagnostic: `ovn-ic-sbctl show` + check BFD giữa transit node. Fix: check IC config + underlay.

### 20.2.4.11 Decision tree Port_Binding forensic

Matrix rút gọn cho oncall engineer:

| Symptom | Type nghi ngờ | Diagnostic đầu tiên | Fix common |
|---------|---------------|---------------------|------------|
| VM mới không network | `""` VIF | `ovs-vsctl get Interface <tap> external_ids:iface-id` | Set iface-id |
| LS và LR không kết nối | `patch` | `ovn-sbctl find Port_Binding type=patch` | Restart northd |
| Traffic không ra physical | `localnet` | `ovs-vsctl get open_vswitch . external_ids:ovn-bridge-mappings` | Add bridge mapping |
| VM không ping được default gateway (external) | `chassisredirect` | `ovn-sbctl list Port_Binding cr-lrp-...` → chassis empty? | Check HA_Chassis_Group + BFD |
| North-bound traffic không NAT | `l3gateway` | `ovn-sbctl --columns=options list Port_Binding lrp-ext` | Update l3gateway-chassis |
| Hardware VTEP tunnel down | `l2gateway` | `ovs-vsctl get open_vswitch . external_ids:ovn-remote-vtep` | Check OVSDB connection |
| Metadata 169.254.169.254 timeout | `localport` | Trong VM: `ip route get 169.254.169.254` | Check route injection |
| Floating IP không failover | `virtual` | `ovn-sbctl list Port_Binding type=virtual` + virtual_parent | Fix parent ref |
| Cross-cluster traffic fail | `remote` | `ovn-ic-sbctl show` | Check IC tunnel |

**Bối cảnh khi chạy:** Kỹ sư quan sát thấy CPU cao trên ovn-controller chassis-42 (75% steady-state thay vì baseline 3-5%). Hypothesis: engine I-P đang fallback sang full recompute quá nhiều. Lệnh `inc-engine/show-stats` cho breakdown per-node để confirm.

**Anatomy từng metric:**

| Metric | Giá trị mẫu | Ý nghĩa | Dấu hiệu bất thường |
|--------|-------------|---------|----------------------|
| Node name | `en_lflow_output` | Tên engine node trong I-P graph. `en_` prefix = engine node. Mỗi node có `_run()` function trong ovn-controller source | Tham chiếu `controller/ovn-controller.c` + `controller/lflow.c` |
| `recompute` | `245` | Số lần node fallback full rebuild (I-P fail, không process incremental được) | **> 1% của compute là red flag.** 245 / 1125 = 22% = critical |
| `compute` | `1125` | Số lần node process incremental thành công | Ratio compute/recompute > 99% = healthy |
| `cancel` | `67` | Số lần compute bị cancel vì input thay đổi lại giữa chừng | Cao khi có churn rapid (VM migrate liên tục) |

**Kịch bản bẻ gãy:**

- **`en_lflow_output.recompute / compute > 5%`** → Logical_Flow translation fallback. Nguyên nhân: lflow-cache quá nhỏ (chỉ cached fraction của total LF), hoặc schema change. Kiểm chứng: `lflow-cache/show-stats` để xem cache hit rate. Fix: `lflow-cache/flush` + observe recovery, hoặc tune cache size qua `ovn-remote-probe-interval`.
- **`en_runtime_data.recompute > 10 trong 5 phút`** → Port_Binding churn. Thường do VM migration burst hoặc bug orchestrator. Kiểm chứng: `ovn-sbctl --columns=logical_port,chassis list Port_Binding | awk ...` để count unique chassis per port recently.
- **Mọi `recompute` column đều = 0** → engine chưa run. Nút mạng (Chassis) vừa restart hoặc ovn-controller freeze. Kiểm chứng: `connection-status`.
- **`cancel` > 10% của compute trên một node** → event stream churning quá fast, engine không kịp consolidate. Fix: giảm NBDB commit rate (batch larger) hoặc tăng `inactivity-probe-interval`.

**Liên hệ concept:** Metric này phản ánh trực tiếp I-P engine của ovn-controller (Phần 13.7 §13.7.3). Healthy cluster có total recompute count grow linear với infrastructure changes (0.1-1 per minute), không grow với traffic volume.

**Upstream nguồn đọc thêm:**
- [`ovn-controller(8)`](https://man7.org/linux/man-pages/man8/ovn-controller.8.html) §inc-engine/show-stats
- OVN source `controller/ovn-controller.c` function `main_loop()` và `controller/inc-proc-eng.c`

### 20.2.5.3 Anatomy `ovn-appctl -t ovn-controller lflow-cache/show-stats`

**Bối cảnh khi chạy:** Tiếp theo phát hiện `en_lflow_output.recompute` cao ở §20.2.5.2, kiểm tra cache hit rate để xác nhận hypothesis cache không hiệu quả.

**Command và output** *(reproduced)*:

```shell
$ sudo ovn-appctl -t ovn-controller lflow-cache/show-stats

Enabled: true
high-watermark : 10240
total count    :  8934
cache-max-size :  1000

Cache type                  Entries    Hit count   Miss count
-------------------------------------------------------------
cache-expr-to-expr-inner     234          12834         456
cache-expr                   189           9872         234
cache-matches                512          28456         1245
cache-none                   0               78        8934
```

**Anatomy:**

| Field | Giá trị mẫu | Ý nghĩa | Dấu hiệu bất thường |
|-------|-------------|---------|----------------------|
| `Enabled` | `true` | Cache có active | `false` = disabled via `ovn-remote-probe-interval=0` |
| `high-watermark` | `10240` | Max entries cache đã reach từ lúc start | Gần total count = chạy ổn định; chênh lệch lớn = cache evict nhiều |
| `total count` | `8934` | Entries hiện tại trong cache | Lớn hơn cache-max-size → cache overflow, evict xảy ra |
| `cache-max-size` | `1000` | Max size config | Default thường 50000, nếu 1000 là tuned low (bug?) |
| `cache-expr-to-expr-inner` Hit | `12834` | Matches vào cache inner expression | Hit rate = hit/(hit+miss) = 12834/(12834+456) = 96.6% ✓ |
| `cache-none` Miss | `8934` | Miss count = tổng entries miss cache | Matches với `total count`: mỗi entry miss đã load vào cache |

**Kịch bản bẻ gãy:**

- **Hit rate `cache-matches` < 80%** → logical flow churn quá cao, cache không effective. Increase `cache-max-size` qua `ovn-remote-probe-interval` hoặc reduce LF churn.
- **`cache-none` tăng liên tục** → misses ballooning. Thường do NBDB bulk update (ví dụ deploy 1000 pods cùng lúc). Wait steady-state, recheck.
- **`Enabled: false`** → operator đã disable cache qua config (thường debugging). Re-enable: `ovs-vsctl remove open_vswitch . external_ids ovn-remote-probe-interval`.

### 20.2.5.4 Anatomy `ovn-appctl -t ovn-controller ct-zone-list`

**Bối cảnh:** Kỹ sư nghi ngờ conntrack zone collision giữa LS và LR gây stateful ACL sai. `ct-zone-list` liệt kê mapping LSP → zone cho chassis.

**Command và output** *(reproduced)*:

```shell
$ sudo ovn-appctl -t ovn-controller ct-zone-list

ls_a_vm1: 1
ls_a_vm2: 2
ls_b_vm1: 3
rtr1_lrp_a: 1000
rtr1_lrp_b: 1001
rtr1_snat: 1002
```

**Anatomy:**

| Field | Giá trị mẫu | Ý nghĩa |
|-------|-------------|---------|
| `ls_a_vm1` | LSP name với prefix datapath | Logical port identifier (LS hoặc LR port) |
| `1` | Zone ID (uint16) | Conntrack zone được ovn-controller assign |

Khi ACL stateful fire `ct(commit, zone=NXM_NX_REG13[0..15])`, `reg13` chứa zone ID này. Collision zone (hai port khác datapath share zone) gây cross-talk state, bug nghiêm trọng.

**Kịch bản bẻ gãy:**

- **Zone trùng giữa 2 port khác nhau** → ovn-controller bug hoặc overflow zone space (>64K port). Kiểm chứng: sort output theo zone, check duplicate. Fix: upgrade OVN hoặc chia chassis nhỏ hơn.
- **Port expected có entry nhưng thiếu** → port chưa bind chassis. Check Port_Binding.chassis.
- **Tổng entries > 65535** → overflow 16-bit zone. OVN 22.03+ hỗ trợ extend nhưng cần config. Reduce port count.

### 20.2.5.5 Anatomy `ovn-appctl -t ovn-controller connection-status`

**Command và output:**

```shell
$ sudo ovn-appctl -t ovn-controller connection-status
connected
```

Hoặc:

```shell
$ sudo ovn-appctl -t ovn-controller connection-status

```

Output đơn giản. Nếu `not connected`, check network/firewall giữa chassis và SBDB cluster, check SBDB IP trong `external_ids:ovn-remote`.

### 20.2.5.6 `recompute` — force full iteration

```shell
$ sudo ovn-appctl -t ovn-controller recompute
```

Silent output. Force full rebuild mọi engine node. Dùng khi suspect stale cache hoặc incomplete sync. Trade-off: lock controller 1-30 giây tùy cluster size, CPU spike high. Không dùng trên production nếu có alternative.

### 20.2.5.7 `inject-pkt` — test packet without real traffic

```shell
$ sudo ovn-appctl -t ovn-controller inject-pkt \
    'inport == "vm_a" && eth.src == aa:bb:cc:00:00:01 && eth.dst == aa:bb:cc:00:00:02 && ip4.src == 10.0.0.1 && ip4.dst == 10.0.0.2 && ip.ttl == 64 && icmp4'
```

## 20.2.6 `ovn-appctl -t ovn-northd` — 10 command runtime management

ovn-northd có 10 runtime command focus vào HA state + compile băng thông thực tế.

### 20.2.6.1 Catalog 10 command

| Command | Arg | Mục đích |
|---------|-----|----------|
| `exit` | - | Gracefully terminate |
| `pause` | - | Tạm dừng compile, vẫn nhận NB/SB change notify nhưng không emit update |
| `resume` | - | Resume compile |
| `is-paused` | - | Return `true`/`false` |
| `status` | - | Print `active`/`standby`/`paused` |
| `sb-cluster-state-reset` | - | Reset SB cluster state khi DB rebuild |
| `nb-cluster-state-reset` | - | Reset NB cluster state khi DB rebuild |
| `set-n-threads` | `<N>` | Set N thread cho parallel build (2-256). Bật parallel khi N >= 2 |
| `get-n-threads` | - | Return current thread count |
| `inc-engine/show-stats` | - | Engine counters (recompute/compute/abort) |
| `inc-engine/clear-stats` | - | Reset counters |

### 20.2.6.2 Anatomy `ovn-appctl -t ovn-northd status`

**Bối cảnh khi chạy:** Kỹ sư kiểm tra HA state của northd cluster 3-node. Mỗi node phải có 1 active + 2 standby, không được 0 active (cluster freeze) hoặc 2+ active (split-brain).

**Command trên từng node:**

```shell
[node1]$ sudo ovn-appctl -t ovn-northd status
standby

[node2]$ sudo ovn-appctl -t ovn-northd status
active

[node3]$ sudo ovn-appctl -t ovn-northd status
standby
```

**Anatomy giá trị:**

| Giá trị | Ý nghĩa | Dấu hiệu bất thường |
|---------|---------|----------------------|
| `active` | Node đang acquire OVSDB lock trên SB DB, đang compile | Chỉ đúng 1 node báo active. > 1 = split-brain |
| `standby` | Node sẵn sàng takeover, chưa acquire lock | N-1 node trong cluster phải standby |
| `paused` | Node đã bị `ovn-appctl pause`, không compile dù active | Thường chỉ ở test/debug env |

**Kịch bản bẻ gãy:**

- **Tất cả node `standby`** → không node nào acquire lock. SBDB không accept lock request (cluster degraded). Check SBDB Raft state bằng `ovn-appctl -t ovsdb-server cluster/status OVN_Southbound`. Fix: restore SBDB quorum.
- **> 1 node `active`** → split-brain, rất nghiêm trọng. Stop các instance extra, keep 1 node active. Kiểm chứng SBDB lock holder bằng query raw.
- **Node `paused` steady-state** → ai đó đã pause, quên resume. Fix: `ovn-appctl -t ovn-northd resume`.

### 20.2.6.3 Anatomy `ovn-appctl -t ovn-northd inc-engine/show-stats`

**Command và output** *(reproduced for northd)*:

```shell
$ sudo ovn-appctl -t ovn-northd inc-engine/show-stats

Node: en_global_config
- recompute:          2
- compute:         1200
- abort:              0

Node: en_nb_mac_binding
- recompute:          1
- compute:          890
- abort:              2

Node: en_northd
- recompute:         89          ← !!! HIGH in a long-running cluster
- compute:         2456
- abort:             12

Node: en_lflow
- recompute:         89
- compute:         2432
- abort:             12

Node: en_sync_to_sb
- recompute:          8
- compute:         3200
- abort:              0
```

Format giống ovn-controller (§20.2.5.2). `abort` ở đây thay `cancel` — compute bị abort khi input change giữa chừng.

**Kịch bản bẻ gãy:**

- **`en_northd.recompute / compute > 5%`** → northd fallback nhiều. Nguyên nhân: NB bulk update quá nhanh, hoặc schema not yet supported in I-P. Batch NB commit larger.
- **`en_sync_to_sb.abort` cao** → SB write bị cancel bởi NB change mới. Northd không emit being ready. Kiểm chứng: `set-n-threads` tăng thread count nếu CPU free.

### 20.2.6.4 `pause` và `resume` — maintenance window

Khi upgrade NBDB schema hoặc restore backup:

```shell
# Pause northd, nó vẫn monitor NB/SB nhưng không emit update
$ sudo ovn-appctl -t ovn-northd pause

# Do maintenance work on SBDB
$ ovn-sbctl restore /backup/sb.db

# Resume northd
$ sudo ovn-appctl -t ovn-northd resume

# Verify
$ sudo ovn-appctl -t ovn-northd is-paused
false
```

Trong khi paused, ovn-controller vẫn đọc SB DB cache local, traffic không bị disrupt. Chỉ NB ý đồ cấu hình mới không sync xuống SB — bị delay đến khi resume.

### 20.2.6.5 `set-n-threads` — parallel compile tuning

ovn-northd 22.03+ hỗ trợ multi-thread compile:

```shell
# Check current
$ sudo ovn-appctl -t ovn-northd get-n-threads
1

# Set 4 threads (parallel bật khi >= 2)
$ sudo ovn-appctl -t ovn-northd set-n-threads 4

# Verify
$ sudo ovn-appctl -t ovn-northd get-n-threads
4
```

Default N = 1 (không parallel). Recommended: N = min(CPU core / 2, 4) cho production. Quá nhiều thread gây lock contention trong NB/SB IDL.

## 20.2.7 Stateful table triage — MAC_Binding, FDB, Service_Monitor

Ngoài Port_Binding, OVN có ba stateful table ảnh hưởng trực tiếp dataplane:

### 20.2.7.1 MAC_Binding — ARP/NDP cache cho LR

`MAC_Binding` table cache MAC đã resolve cho mỗi (LRP, IP). Stage `lr_in_arp_resolve` đọc table này để set eth.dst trước khi forward. Stage `lr_in_learn_neighbor` ghi vào table khi thấy ARP reply.

**Schema ngắn:** `logical_port` (LRP), `ip`, `mac`, `datapath`, `timestamp`.

**Diagnostic:**

```shell
# List all MAC_Binding
$ ovn-sbctl list MAC_Binding

# Find entry for specific IP
$ ovn-sbctl find MAC_Binding ip=10.0.1.5

# Aging: entries không có timestamp refresh trong X giây sẽ bị northd GC
# X config qua NB_Global.options:mac_binding_age_threshold (default 0 = disabled)
$ ovn-nbctl get NB_Global . options:mac_binding_age_threshold
```

**Failure mode:**

- Entry stale (IP đổi MAC sau ARP) → next-hop không reachable. Fix: `ovn-sbctl destroy MAC_Binding <uuid>` force re-learn, hoặc tune age_threshold.
- Entry missing sau restart → chờ ARP resolve (tối đa 60s). Hoặc set addresses static trên LSP remote.

### 20.2.7.2 FDB — Dynamic MAC learning (OVN 22.03+)

OVN 22.03+ thêm FDB table cho L2 MAC learning trong LS. Trước đó, chỉ static (từ `Logical_Switch_Port.addresses`). Stage `ls_in_lookup_fdb` đọc, `ls_in_put_fdb` ghi.

**Schema:** `mac`, `dp_key`, `port_key`, `timestamp`.

**Diagnostic:**

```shell
$ ovn-sbctl list FDB

# Per datapath
$ ovn-sbctl find FDB dp_key=3

# Aging qua NB_Global.options:fdb_age_threshold (default 0 = disabled)
```

**Failure mode:** FDB poisoning (xem Phần 17 forensic) — MAC trùng giữa 2 VM/port gây traffic lệch state. Fix: purge FDB entry + nhận diện nguồn bogus.

### 20.2.7.3 Service_Monitor — Load_Balancer health check

LB backend health monitored bằng Service_Monitor table. Mỗi backend có 1 row, ovn-controller probe TCP/ICMP định kỳ, update `status` field.

**Schema:** `ip`, `port`, `protocol`, `status` (online/offline), `src_mac`, `src_ip`, `logical_port`.

**Diagnostic:**

```shell
$ ovn-sbctl list Service_Monitor

# Find offline backends
$ ovn-sbctl --bare --columns=ip,port,status find Service_Monitor status=offline
```

**Failure mode:**

- Backend offline mà LB vẫn route → health check chưa detect (default probe 10s). Tune `options:interval=5` trên LB.
- Backend online mà không serve → L7 health check cần (OVN chỉ có L4). Supplement bằng ngoài.

## 20.2.9 Guided Exercise 1 — ACL drop silent với `ovn-trace --ct` multi-scenario

> **State verification lab:** doc-plausible based on OVN tutorial + ovn-trace(8). Replace với verified-lab output khi có lab host (C1b).

**Mục tiêu:** Chứng minh phương pháp phân biệt ACL stateless vs stateful bằng cách chạy `ovn-trace` nhiều lần với `--ct` flags khác nhau để nhận diện rule nào drop reply path.

**Chuẩn bị:**

- 1 node Ubuntu 22.04 với OVN 22.03 đã cài đặt.
- 1 Logical Switch `ls_web` với 2 LSP: `client` (10.0.0.1) và `web` (10.0.0.2).
- 1 ACL: `from-lport priority=1000 match="inport == \"client\" && tcp.dst == 80" action=allow`.

**Mô hình:**

```
[client 10.0.0.1]  --- [ls_web] ---  [web 10.0.0.2]
     |                                    |
     +------- TCP :80 allow forward ------+
              TCP :80 reply ??? ------
```

**Bước thực hiện:**

**1.** Trace forward path (NEW connection):

```shell
$ ovn-trace --detailed ls_web \
    'inport == "client" && eth.src == aa:bb:cc:00:00:01 && eth.dst == aa:bb:cc:00:00:02 &&
     ip4.src == 10.0.0.1 && ip4.dst == 10.0.0.2 && ip.ttl == 64 &&
     tcp && tcp.src == 52341 && tcp.dst == 80'
```

Kỳ vọng: `output to "web"`. ACL priority 1000 fire với action `allow;`.

**2.** Trace reply path (mặc định NEW, chưa có --ct):

```shell
$ ovn-trace --detailed ls_web \
    'inport == "web" && eth.src == aa:bb:cc:00:00:02 && eth.dst == aa:bb:cc:00:00:01 &&
     ip4.src == 10.0.0.2 && ip4.dst == 10.0.0.1 && ip.ttl == 64 &&
     tcp && tcp.src == 80 && tcp.dst == 52341'
```

Kỳ vọng: `drop;` tại stage `ls_out_acl` hoặc `ls_in_acl` (depending on direction). Reply bị drop vì không có ACL match.

**3.** Trace reply với `--ct=trk,est,rpl`:

```shell
$ ovn-trace --ct=trk,est,rpl --detailed ls_web \
    'inport == "web" && eth.src == aa:bb:cc:00:00:02 && eth.dst == aa:bb:cc:00:00:01 &&
     ip4.src == 10.0.0.2 && ip4.dst == 10.0.0.1 && ip.ttl == 64 &&
     tcp && tcp.src == 80 && tcp.dst == 52341'
```

Kỳ vọng: Nếu ACL là `allow-related` → `output to "client"` (reply pass). Nếu ACL là `allow` plain → vẫn `drop;`.

**4.** Fix ACL stateless → stateful:

```shell
$ ovn-nbctl acl-del ls_web from-lport 1000 'inport == "client" && tcp.dst == 80'
$ ovn-nbctl acl-add ls_web from-lport 1000 'inport == "client" && tcp.dst == 80' allow-related
```

**5.** Repeat step 3 trace, observe:

```shell
$ ovn-trace --ct=trk,est,rpl --detailed ls_web '...'
# Kỳ vọng: "output to \"client\"" — reply now allowed by stateful tracking
```

**Bài học:**

- `allow` plain = stateless; reply không auto permit. Cần ACL ngược chiều explicit.
- `allow-related` = stateful; conntrack state est/rpl match → auto permit reply.
- `--ct=trk,est,rpl` là công cụ duy nhất để phân biệt hai behavior này mà không cần traffic thật.

**Cleanup:**

```shell
$ ovn-nbctl ls-del ls_web
```

## 20.2.12 Capstone POE — Refute hypothesis "engine_recompute > 10%/min → must restart"

> **Kịch bản:** Hệ thống alert fire cảnh báo "engine_recompute rate > 10 events/min trên chassis-42". Kỹ sư trực ca hypothesis: "ovn-controller stuck, phải restart."

**Predict:** Restart sẽ clear state, recompute rate về 0. Hypothesis: recompute spike do bug memory leak hoặc stale state.

**Observe:**

**1.** Check recompute breakdown per node:

```shell
$ sudo ovn-appctl -t ovn-controller inc-engine/show-stats

- recompute: 245      ← high
- compute: 8923
- cancel: 12

Node: en_lflow_output
- recompute: 12       ← normal
- compute: 8900
- cancel: 3
```

Recompute tập trung ở `en_runtime_data` (Port_Binding-related), không phải `en_lflow_output` (LF translation). Hypothesis memory leak bị nghi ngờ vì leak thường distribute đều.

**2.** Check recent churn:

```shell

    | awk '/chassis/ {print $NF}' | sort | uniq -c | sort -rn | head

  120 hv-new-uuid-xyz      ← hv mới join cluster vừa receive 120 port
  89 hv-42-uuid
  45 hv-other
```

Hv-new-uuid vừa join + receive burst 120 port. Đây là infrastructure event, không phải bug.

**3.** Check engine history:

```shell
$ sudo journalctl -u ovn-controller --since "30 min ago" | grep -i "chassis" | head
2026-04-24T10:15:22 ... Chassis hv-new joined
2026-04-24T10:15:22 ... Received 120 Port_Binding updates
2026-04-24T10:15:23 ... en_runtime_data recompute (fallback: bulk update)
```

**4.** Wait 2-3 phút, check again:

```shell
$ sudo ovn-appctl -t ovn-controller inc-engine/show-stats

- recompute: 245       ← không tăng nữa
- compute: 9823         ← compute tiếp tục tăng steady
- cancel: 12
```

Recompute không tăng nữa. System recovered.

**Explain:** Recompute spike là **transient**, do chassis mới join nhận burst port updates. I-P engine không hỗ trợ bulk update nguyên vẹn (schema limitation), fallback full recompute cho node affected. Sau khi bulk update settle, engine về incremental.

**Refute:** Hypothesis "must restart" sai. Restart sẽ gây full recompute tất cả 8 engine nodes (spike lớn hơn 245), disrupt traffic, và không fix nguyên nhân (new chassis join là normal event).

**Correct action:** Acknowledge alert, annotate "chassis join event, transient". Tune alert threshold: recompute rate > 50/min **sustained > 10 phút** mới actionable. Single burst < 10 phút = normal.

**Lesson:** Metric alone lie. Always correlate với infrastructure event (chassis join, schema upgrade, orchestrator batch). Restart là cudgel, không phải scalpel — dùng khi đã xác định root cause, không dùng để "clear state" theo trực giác.

## 20.2.13 Hiểu sai phổ biến

> **Hiểu sai 1:** "tcpdump là công cụ đầu tiên khi debug OVN."
> **Đúng:** `ovn-trace` + `ofproto/trace` là simulation, chạy mili giây, không phụ thuộc timing. `tcpdump` chỉ thấy wire, không thấy logical flow decision. Dùng simulation trước để narrow scope trước khi capture.

> **Hiểu sai 2:** "`ovn-trace` delivered = traffic thực chắc chắn thành công."
> **Đúng:** `ovn-trace` simulate logical level (NB + SB). Thực tế còn qua OpenFlow (ovn-controller dịch) và datapath (OVS execute). Khi theo dõi OK mà traffic fail, bug ở lớp C (OpenFlow) hoặc 4/5 (datapath/tunnel).

> **Hiểu sai 3:** "Port_Binding.chassis empty luôn là bug."
> **Đúng:** type=`localnet`, `patch` không claim chassis by design (available mọi chassis). Chỉ type=`""`, `chassisredirect`, `l3gateway`, `l2gateway`, `virtual`, `remote` expect chassis non-empty.

> **Hiểu sai 4:** "engine_recompute trong inc-engine/show-stats càng thấp càng tốt, 0 là lý tưởng."
> **Đúng:** Recompute = 0 không thực tế (mỗi khi chassis restart, schema change, bulk NB update đều kích hoạt 1 recompute). Healthy target: recompute / compute < 1%, không phải recompute = 0.

> **Hiểu sai 5:** "Restart ovn-controller clear tất cả state, giúp fix mọi bug liên quan state."
> **Đúng:** Restart kích hoạt full recompute (spike nặng), disrupt traffic vài giây, và thường không fix bug (bug thường ở NB ý đồ cấu hình hoặc northd). Dùng `recompute` command nếu chỉ muốn refresh state.

> **Hiểu sai 6:** "ovn-detrace đọc cookie từ `dump-flows` giúp hiểu mọi OpenFlow rule."
> **Đúng:** ovn-detrace chỉ annotate flow có cookie khớp SBDB Logical_Flow. Flow sinh bởi ovn-controller internal (table 0 demux, table 64 loopback) không có cookie → không annotate. Cần hiểu khi nào annotation có, khi nào không.

> **Hiểu sai 7:** "`ovn-trace --ct=trk,est` đủ để kiểm chứng stateful ACL reply path."
> **Đúng:** Cần thêm `rpl` (reply direction). Flags `trk,est` chỉ nói "established", không nói "reply". ACL `allow-related` cần `trk,est,rpl` để fire reverse-direction logic.

> **Hiểu sai 8:** "northd pause làm mất kết nối traffic."
> **Đúng:** Pause chỉ ngưng compile NB → SB. Traffic hiện tại vẫn chạy (ovn-controller đọc SB DB cache local). Chỉ NB ý đồ cấu hình mới bị delay đến khi resume. Dùng pause an toàn trong maintenance window.

## 20.2.14 Điểm cốt lõi cần nhớ

Thứ nhất, **debug OVN theo 3 lớp (NB ý đồ cấu hình → SB Logical_Flow → OpenFlow br-int)**. Simulation trước capture: `ovn-trace` + `ofproto/trace` + `ovn-detrace` cho câu trả lời trong mili giây, `tcpdump` chỉ khi simulation đã OK mà traffic vẫn fail.

Thứ hai, **`ovn-trace` có 4 output mode** (`--detailed` default, `--summary` pseudo-code, `--minimal` regression test stable, `--all` combined) + 3 state-injection option (`--ct=flags` conntrack state, `--lb-dst` force backend, `--select-id` force group bucket). Biết khi nào dùng mode nào là dấu hiệu mastery.

Thứ ba, **`ovn-detrace` chain với `ofproto/trace`** để đi từ OpenFlow cookie → Logical_Flow → NB object (ACL/LB/NAT). Cần set `OVN_SB_DB` + `OVN_NB_DB` env để annotation đầy đủ. Pattern diff cross-chassis bằng cách pipe `dump-flows | diff | ovn-detrace` hữu dụng khi chassis specific có lệch state.

Thứ tư, **Port_Binding 8 type có claim logic và failure mode khác nhau**. VIF (`""`) claim qua `external_ids:iface-id`; `chassisredirect` claim qua HA_Chassis_Group + BFD election; `localnet/patch/localport` không claim; `virtual` follow `virtual_parent`. Diagnostic matrix 9-pattern consolidated trong §20.2.4.11.

Thứ năm, **`ovn-appctl -t ovn-controller` có 11 command**, quan trọng nhất `inc-engine/show-stats` (engine health) + `lflow-cache/show-stats` (cache hit rate) + `ct-zone-list` (conntrack zone collision) + `recompute` (force rebuild) + `connection-status` (SBDB reachable). `ovn-appctl -t ovn-northd` có 10 command, quan trọng `status` (active/standby/paused) + `inc-engine/show-stats` + `set-n-threads` (parallel compile tuning).

Thứ sáu, **metric cần correlate với infrastructure event**. `engine_recompute` spike không tự nó là bug: chassis join, schema upgrade, bulk NB update đều kích hoạt normal recompute. Alert threshold nên "sustained > 10 phút" không "single spike". Restart không phải scalpel cho mọi bug — chỉ khi root cause đã nhận diện cần clear state.

## 20.2.15 Backfill v3.5 — lflow-cache configuration tuning (5 external_ids tunables)

> **Mục đích:** đợt audit J.1 v3.5-KeywordBackbone phát hiện 5 lflow-cache tunable trong `external_ids` config space chưa được mention. File 20.2 hiện cover `lflow-cache/show-stats` (read state) nhưng không có tuning anatomy. Section này backfill, source verified `ovn-controller(8)` man page.

### 20.2.15.1 Vì sao cần tune lflow-cache

`ovn-controller` cache compiled flow result trong RAM để tránh recompute mỗi khi packet đến. Default behavior (cache unlimited) hoạt động tốt cho cụm < 1000 chassis + < 100K Logical_Flow tổng. Vượt scale này, cache có thể:
- Ăn quá nhiều RAM (chassis OOM)
- Trim không kịp (latency tăng cho compute mới)
- Stale entry không được flush (data inconsistency)

5 tunable trong `external_ids` của Open_vSwitch table cho phép operator điều chỉnh hành vi cache cho deployment scale lớn.

### 20.2.15.2 Anatomy 5 lflow-cache tunable (verified `ovn-controller(8)`)

| Tunable | Default | Scope | Mục đích |
|---------|---------|-------|----------|
| `external_ids:ovn-enable-lflow-cache` | `true` | enable/disable toàn bộ cache | Disable cache nếu RAM-constrained chassis (rare) |
| `external_ids:ovn-limit-lflow-cache` | unlimited | max entry count | Cap cache size theo entry count (cho deployment biết flow count) |
| `external_ids:ovn-memlimit-lflow-cache-kb` | unlimited | max cache memory (KB) | Cap cache size theo memory (cho deployment biết RAM budget) |
| `external_ids:ovn-trim-limit-lflow-cache` | 10000 | min entry trước khi trim | Threshold để auto-trim trigger |
| `external_ids:ovn-trim-wmark-perc-lflow-cache` | 50% | watermark % cho trim | Trim aggressive level (% high-watermark) |

### 20.2.15.3 Set + verify tunable

**Set tunable:**

```bash
# Disable hoàn toàn (chỉ cho debug)
ovs-vsctl set Open_vSwitch . external_ids:ovn-enable-lflow-cache=false

# Cap entry count = 50K
ovs-vsctl set Open_vSwitch . external_ids:ovn-limit-lflow-cache=50000

# Cap memory = 256 MB
ovs-vsctl set Open_vSwitch . external_ids:ovn-memlimit-lflow-cache-kb=262144

# Trim threshold = 5K entry (aggressive)
ovs-vsctl set Open_vSwitch . external_ids:ovn-trim-limit-lflow-cache=5000

# Trim watermark = 70% (less aggressive)
ovs-vsctl set Open_vSwitch . external_ids:ovn-trim-wmark-perc-lflow-cache=70
```

**Verify tunable applied:**

```bash
ovs-vsctl get Open_vSwitch . external_ids
# Output: {ovn-enable-lflow-cache="true", ovn-limit-lflow-cache="50000", ...}

# Verify ovn-controller picked up (qua show-stats)
ovn-appctl -t ovn-controller lflow-cache/show-stats
# Output: enabled, total_size, max_size, hit, miss, ...
```

### 20.2.15.4 Decision matrix tuning

| Symptom | Diagnostic | Tune action |
|---------|------------|-------------|
| Chassis RAM > 80% | `ovn-appctl lflow-cache/show-stats` thấy total_size > 1 GB | Set `ovn-memlimit-lflow-cache-kb=512000` (512 MB cap) |
| ovn-controller restart slow > 2 phút | `inc-engine/show-stats` recompute count cao | Tăng `ovn-limit-lflow-cache` cho phép cache lớn hơn (giảm recompute) |
| Cache hit rate < 50% (low) | `lflow-cache/show-stats` ratio hit/(hit+miss) | Tăng `ovn-trim-limit-lflow-cache` để giữ entry lâu hơn (giảm trim) |
| Frequent OOM kill ovn-controller | `dmesg \| grep ovn-controller` thấy OOM | Set `ovn-memlimit-lflow-cache-kb` cap; consider disable cache nếu chassis nhỏ |
| Cache thrashing (entry add/del nhanh) | `lflow-cache/show-stats` evicted count cao | Giảm `ovn-trim-wmark-perc-lflow-cache` xuống 30% (less aggressive trim) |

### 20.2.15.5 Anti-pattern + lưu ý

**Anti-pattern 1: Disable cache trên chassis production.**

`ovn-enable-lflow-cache=false` force ovn-controller compile mỗi packet → CPU 100% under load. Chỉ dùng để debug "cache có corrupt không"; restore `true` ngay sau debug.

**Anti-pattern 2: Set `ovn-limit-lflow-cache` quá thấp (< 1000).**

Cache nhỏ → trim liên tục → chỉ vừa cài entry đã bị trim → packet new tới phải compute lại → throughput drop. Default unlimited là OK cho hầu hết deployment.

**Lưu ý 3: Tunable change KHÔNG cần restart ovn-controller.**

`ovs-vsctl set` thay đổi external_ids → ovn-controller pick up trong loop tiếp theo (within ~1 second). Verify bằng `lflow-cache/show-stats` thấy max_size mới.

**Lưu ý 4: Memory accounting không bao gồm OVS internal struct.**

`ovn-memlimit-lflow-cache-kb` chỉ đếm size của cached lflow entry, KHÔNG đếm overhead OVS internal (hash table, free list). Real memory consumed có thể 1.2-1.5x giá trị set. Set conservative (75% của RAM budget) để safety margin.

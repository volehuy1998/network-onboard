## 13.16.1 Tổng quan pipeline OVN — vì sao 63 stage

OVN biên dịch ý đồ network logic (NB DB) thành **pipeline tuyến tính nhiều stage** trên SB DB Logical_Flow. Mỗi stage là một bộ flow rule có vai trò chuyên biệt: port-security check, ACL evaluation, load balancer DNAT, DHCP options, L2 lookup, v.v. Khi packet đi qua một logical switch, nó tuần tự đi qua **26 stage ingress (LS_IN)**, sau đó qua **decision OUTPUT** (chassis-local hay tunnel sang remote), rồi qua **10 stage egress (LS_OUT)** trước khi rời. Đối với logical router, packet đi qua **20 stage ingress (LR_IN)** + **7 stage egress (LR_OUT)**.

Tổng cộng **63 stage** (so với 64 ghi trong REF — REF mô tả OVN 24.03+ có thêm 1 stage; xem §13.16.16 version drift). Số 63 là kết quả grep trực tiếp trong `northd/northd.c` macro `PIPELINE_STAGES` của branch-22.03.

**Vì sao chia nhỏ thành nhiều stage?** Câu trả lời nằm ở 4 yếu tố thiết kế:

1. **Modular composition.** Mỗi stage giải quyết duy nhất một concern. ACL không cần biết về DHCP. NAT không cần biết về L2 lookup. Khi cần thêm tính năng mới (ví dụ ECMP statefulness), ovn-northd team thêm stage mới, không phải sửa stage cũ.
2. **Debug-friendly.** Khi packet bị drop, operator có thể xác định CHÍNH XÁC stage nào drop bằng cách trace qua `ovn-trace --detailed` hoặc dump flow tương ứng từng table. Pipeline ít stage = debug khó.
3. **Cross-version migration.** Stage mới có thể được chèn vào pipeline mà không phá vỡ stage cũ (chỉ cần điều chỉnh register usage). OVN 22.03 → 24.03 đã thêm vài stage cho LB affinity, ACL split-eval-action, mà không làm vỡ pipeline cũ.
4. **OpenFlow table flat scaling.** OVS classifier hoạt động tốt nhất khi mỗi table có ít flow priority distinct. Chia 63 stage giúp mỗi OF table có ≤ 1000 priority distinct, classifier giữ tốc độ lookup O(log n).

Mặt trái của thiết kế nhiều stage là **CPU cost mỗi packet** trên slow path. Mỗi `goto_table` instruction cộng thêm chi phí lookup. Trên fast path (megaflow cache hit), chi phí gần như 0 vì OVS tổng hợp toàn bộ chuỗi stage thành một megaflow entry duy nhất.

> **Lưu ý phiên bản:** Số stage tăng dần theo phiên bản. OVN 20.06 có khoảng 50 stage, OVN 22.03 có 63 stage, OVN 24.03 có ~70 stage (thêm LB_AFF, ACL split, IPSec encap). Curriculum bám OVN 22.03.8 baseline, mọi version drift được callout ở §13.16.16 per Rule 14.

## 13.16.2 Br-int OpenFlow table block layout

Trên `br-int`, ovn-controller chiếm 73 OpenFlow table (table 0 đến table 72). Mỗi table có vai trò cố định, được định nghĩa trong `controller/lflow.h` (OVN branch-22.03):

```c
#define OFTABLE_PHY_TO_LOG            0
#define OFTABLE_LOG_INGRESS_PIPELINE  8 /* First of LOG_PIPELINE_LEN tables. */
#define OFTABLE_REMOTE_OUTPUT        37
#define OFTABLE_LOCAL_OUTPUT         38
#define OFTABLE_CHECK_LOOPBACK       39
#define OFTABLE_LOG_EGRESS_PIPELINE  40 /* First of LOG_PIPELINE_LEN tables. */
#define OFTABLE_SAVE_INPORT          64
#define OFTABLE_LOG_TO_PHY           65
#define OFTABLE_MAC_BINDING          66
#define OFTABLE_MAC_LOOKUP           67
#define OFTABLE_CHK_LB_HAIRPIN       68
#define OFTABLE_CHK_LB_HAIRPIN_REPLY 69
#define OFTABLE_CT_SNAT_HAIRPIN      70
#define OFTABLE_GET_FDB              71
#define OFTABLE_LOOKUP_FDB           72
```

**Sơ đồ block layout:**

```
+-------+---------------------------------+--------------------------------+
| Table | Macro                           | Vai trò                        |
+-------+---------------------------------+--------------------------------+
|   0   | OFTABLE_PHY_TO_LOG              | Phân loại packet đến từ vật lý |
|       |                                 | → gán datapath + inport tunnel |
|       |                                 | key vào metadata + reg14       |
+-------+---------------------------------+--------------------------------+
| 1-7   | (reserved, không dùng)          |                                |
+-------+---------------------------------+--------------------------------+
| 8-33  | OFTABLE_LOG_INGRESS_PIPELINE    | Logical INGRESS pipeline       |
|       | (26 OF table = LOG_PIPELINE_LEN)| LS_IN tables 0-25              |
|       |                                 | LR_IN tables 0-19              |
|       |                                 | (chia sẻ OF table, phân biệt   |
|       |                                 | qua metadata = datapath_id)    |
+-------+---------------------------------+--------------------------------+
| 34-36 | (reserved buffer)               |                                |
+-------+---------------------------------+--------------------------------+
|  37   | OFTABLE_REMOTE_OUTPUT           | Output sang chassis remote     |
|       |                                 | (qua Geneve tunnel)            |
+-------+---------------------------------+--------------------------------+
|  38   | OFTABLE_LOCAL_OUTPUT            | Output sang LSP local          |
+-------+---------------------------------+--------------------------------+
|  39   | OFTABLE_CHECK_LOOPBACK          | Kiểm tra + drop egress loopback|
+-------+---------------------------------+--------------------------------+
| 40-49 | OFTABLE_LOG_EGRESS_PIPELINE     | Logical EGRESS pipeline        |
|       | (10 OF table)                   | LS_OUT tables 0-9              |
|       |                                 | LR_OUT tables 0-6              |
+-------+---------------------------------+--------------------------------+
| 50-63 | (reserved)                      |                                |
+-------+---------------------------------+--------------------------------+
|  64   | OFTABLE_SAVE_INPORT             | Lưu inport trước hairpin       |
+-------+---------------------------------+--------------------------------+
|  65   | OFTABLE_LOG_TO_PHY              | Logical → physical translation |
|       |                                 | + emit packet ra tunnel/LSP    |
+-------+---------------------------------+--------------------------------+
| 66-67 | OFTABLE_MAC_BINDING/LOOKUP      | MAC binding cache              |
+-------+---------------------------------+--------------------------------+
| 68-70 | OFTABLE_CHK_LB_HAIRPIN +        | Load balancer hairpin handling |
|       | CHK_LB_HAIRPIN_REPLY +          |                                |
|       | CT_SNAT_HAIRPIN                 |                                |

```

**Điểm cốt yếu:** LS_IN và LR_IN **chia sẻ** OF table 8-33. Cả hai pipeline cùng dùng table 8 cho stage logic 0 của mình (LS_IN_PORT_SEC_L2 vs LR_IN_ADMISSION). Phân biệt nhờ field `metadata` (datapath ID) trong match condition. Một flow chỉ apply cho LS có `metadata=0x6`, flow khác apply cho LR có `metadata=0x7`. Đây là cách OVN nén nhiều logical datapath vào shared OF table mà không xung đột.

Tương tự, LS_OUT và LR_OUT chia sẻ OF table 40-49.

## 13.16.3 Công thức ánh xạ logical stage → OF table

Từ layout trên, công thức ánh xạ rõ ràng:

| Pipeline | Công thức |
|----------|-----------|
| LS_IN | `OF table = 8 + LS_IN_<stage>` |
| LR_IN | `OF table = 8 + LR_IN_<stage>` |
| LS_OUT | `OF table = 40 + LS_OUT_<stage>` |
| LR_OUT | `OF table = 40 + LR_OUT_<stage>` |

**Ví dụ thực tiễn:**

- `LS_IN_PORT_SEC_L2` (logical table 0) → OF table 8
- `LS_IN_LB` (logical table 12) → OF table 20
- `LS_IN_L2_LKUP` (logical table 24) → OF table 32
- `LR_IN_IP_ROUTING` (logical table 11) → OF table 19
- `LR_IN_ARP_RESOLVE` (logical table 15) → OF table 23
- `LS_OUT_ACL` (logical table 4) → OF table 44
- `LR_OUT_SNAT` (logical table 3) → OF table 43
- `LR_OUT_DELIVERY` (logical table 6) → OF table 46

Khi đọc `ovs-ofctl dump-flows br-int table=20` thấy flow có `metadata=0x6,...,actions=ct(commit, ...)` → packet đang ở stage `LS_IN_LB` của datapath ID 6. Tra cross qua `ovn-sbctl find Datapath_Binding tunnel_key=6` để biết logical switch nào.

> **Cảnh báo accuracy:** Công thức trên đúng cho OVN 22.03.8. OVN 24.03+ có thay đổi LOG_PIPELINE_LEN (mở rộng từ 26 lên 30), khiến offset egress dịch từ 40 lên 44. Mọi commit về OF table number cần verified bằng `ovn-sbctl --columns=table_id list Logical_Flow` cho phiên bản tương ứng.

**Diễn giải các stage chuyển hướng quan trọng:**

- **Table 8-10 (port-security 3 stage)**: nếu packet không khớp `port_security` của LSP, drop ngay tại stage tương ứng. Lý do tách 3 stage: L2 (MAC), L3 (IP), L4 (ND/ARP) có thể được verify độc lập, và tên stage giúp debug rõ ràng "drop tại L2 hay L3".
- **Table 13-17 (ACL pre-stateful + ACL eval-action)**: ACL được biên dịch thành 4 stage thay vì 1 vì OVN dùng `ACL_HINT` register làm **2-pass match-action**. Stage 16 chỉ set REGBIT (không emit verdict), stage 17 đọc REGBIT và emit verdict. Lợi ích: classifier OVS không phải so sánh mỗi rule với mỗi packet; stage 16 dùng OVS conjunction match để giảm độ phức tạp xuống O(N+M) thay vì O(N×M).
- **Table 20-23 (LB + hairpin)**: LB DNAT trong stage 12 (table 20) rewrite destination IP từ VIP về backend IP. Sau đó stage 13-15 detect hairpin (case client cùng host backend) và apply SNAT để backend không nhìn thấy chính source IP. Đây là cốt lõi mechanism stateful LB của OVN.
- **Table 24 (ACL after LB)**: ACL chạy LẦN 2 sau LB DNAT để cho phép policy nhắm vào backend IP cụ thể (ví dụ "deny traffic đến backend X"). Quan trọng: ACL trước LB nhìn thấy VIP, ACL sau LB nhìn thấy backend IP.
- **Table 26-30 (ARP responder + DHCP + DNS native)**: OVN có thể trả lời ARP/DHCP/DNS bằng flow rule, không cần controller intervention. Đây là một trong những differentiator của OVN so với SDN cổ điển — control plane không cần tham gia mọi request, chỉ ovn-controller cài flow rule trước.
- **Table 32-33 (L2 lookup + unknown)**: stage cuối ingress, decision forwarding chính. Set reg15=outport (logical) rồi rời ingress pipeline.

| 42 | 2 | `LS_OUT_PRE_STATEFUL` | Stateful pre-processing egress (emit ct()) |

**Quan sát thiết kế:**

- Egress pipeline ngắn hơn ingress (10 vs 26 stage). Nguyên do: nhiều decision (LB, hairpin, ARP responder, DHCP) đã làm ở ingress; egress chỉ cần policy enforcement cuối + port-security exit-side.
- Port-security xuất hiện cả ở ingress (verify source) lẫn egress (verify dest). Điều này chống packet với fake source IP đến từ một LSP, đồng thời chặn packet với fake dest IP đến một LSP không khớp port_security của nó.
- Stateful egress (table 47) commit conntrack cho **egress-initiated** connection. Quan trọng cho case logical router NAT: connection từ LR tới external network cần commit ở egress của LR (chassis gateway), không phải ingress của LR (chassis nguồn).

| 15 | 7 | `LR_IN_ECMP_STATEFUL` | ECMP per-connection state cho symmetric reply |

| 18 | 10 | `LR_IN_IP_ROUTING_PRE` | Pre-routing policy (set policy hint) |
| 19 | 11 | `LR_IN_IP_ROUTING` | IP routing table lookup (longest prefix match) |

| 24 | 16 | `LR_IN_CHK_PKT_LEN` | Kiểm tra MTU output port |
| 25 | 17 | `LR_IN_LARGER_PKTS` | Handle oversized packet (drop hoặc emit ICMP frag-needed) |
| 26 | 18 | `LR_IN_GW_REDIRECT` | Redirect centralized traffic về gateway chassis (cr-* port) |
| 27 | 19 | `LR_IN_ARP_REQUEST` | Generate ARP/ND request cho unresolved next-hop |

**Diễn giải các stage critical:**

- **Table 9-10 (LOOKUP_NEIGHBOR + LEARN_NEIGHBOR)**: OVN tách MAC binding lookup khỏi learning để classifier có thể batch process. LOOKUP_NEIGHBOR set REGBIT_LOOKUP_NEIGHBOR_RESULT nếu MAC tìm thấy trong cache; LEARN_NEIGHBOR đọc bit và quyết định có cần emit `put_arp()` action để cài entry mới hay không.
- **Table 12-14 (UNSNAT + DEFRAG + DNAT)**: thứ tự critical. UNSNAT trước DNAT vì reply packet (đến router với SNAT-translated dest) cần được un-SNAT trước khi DNAT lookup chạy. DEFRAG ở giữa vì DNAT cần truy cập L4 header (port number) → packet phải defragmented.
- **Table 15 (ECMP_STATEFUL)**: theo dõi state cho mỗi connection ECMP, đảm bảo reply traffic đi qua cùng next-hop với forward direction (avoid asymmetric routing). Sử dụng `ct_label[32..79]` = source MAC (xem Phần 13.17 §13.17.7).
- **Table 19 (IP_ROUTING)**: longest prefix match trên `Logical_Router_Static_Route` table. Tùy router có ECMP hay không, có thể fallthrough sang table 20.
- **Table 23 (ARP_RESOLVE)**: stage cuối ingress LR. Set destination MAC = MAC của next-hop (resolved từ MAC_Binding). Nếu chưa resolved → set REGBIT để stage `ARP_REQUEST` (table 27) emit ARP/ND request out.
- **Table 26 (GW_REDIRECT)**: chỉ hoạt động khi router có distributed gateway (LRP với `options:redirect-chassis`). Stage này detect packet cần đi qua gateway chassis (centralized SNAT/NAT) và redirect qua chassisredirect (`cr-`) port.

**Diễn giải:**

- **Table 40 (CHECK_DNAT_LOCAL)**: stage NEW trong OVN 22.03+, kiểm tra packet sau routing có thuộc DNAT internal (LB DNAT đến backend cùng router) hay không. Nếu có, không apply SNAT để backend nhận packet với source IP gốc.
- **Table 41-42 (UNDNAT + POST_UNDNAT)**: reverse DNAT cho reply packet. Khi router đã DNAT một packet ingress, reply phải un-DNAT để source ngoài thấy packet với original IP.
- **Table 43-44 (SNAT + POST_SNAT)**: SNAT cho packet đi ra. Áp dụng masquerade hoặc floating IP theo NB.NAT entry.
- **Table 45 (EGR_LOOP)**: chống vòng lặp egress khi packet bị routed về chính router (rare nhưng có thể xảy ra với policy routing sai cấu hình).
- **Table 46 (DELIVERY)**: stage cuối, emit packet ra LRP egress (nếu peer là LSP) hoặc patch port (nếu peer là LR khác). Sau stage này packet rời LR pipeline và (nếu cần) đi vào ingress pipeline của LSP/LR đích.

## 13.16.8 Dispatch + utility tables (OF table 0, 37-39, 64-72)

Ngoài 4 pipeline logic, br-int còn có 9 OF table cho dispatch + utility:

**Table 0: `OFTABLE_PHY_TO_LOG` — phân loại packet đến.**

Mọi packet đến br-int (từ NIC vật lý qua tunnel hoặc từ veth của LSP local) bắt đầu ở table 0. Các flow ở đây match trên `in_port` (OF port của tunnel hoặc veth) và set:
- `metadata` = datapath ID (logical switch hoặc router ID)
- `reg14` = inport tunnel_key (tra từ Geneve TLV class 0x0102 nếu từ tunnel, hoặc từ Port_Binding nếu từ veth local)
- `MFF_LOG_FLAGS` (reg10) = MLF flag (lookup từ tunnel TLV)
- Resubmit sang OFTABLE_LOG_INGRESS_PIPELINE (table 8) để vào pipeline chính

**Table 37: `OFTABLE_REMOTE_OUTPUT` — output sang chassis remote.**

Khi outport (reg15) trỏ vào LSP nằm trên chassis khác, ovn-controller cài flow ở table 37 với match `metadata=<dp>,reg15=<outport>` → action push Geneve header + output ra tunnel port của chassis đích. Geneve TLV class 0x0102 mã hóa (datapath, inport, outport, MLF) để chassis đích reproduce pipeline state.

**Table 38: `OFTABLE_LOCAL_OUTPUT` — output sang LSP local.**

Tương tự table 37 nhưng cho LSP local. Match `metadata=<dp>,reg15=<outport>` → action set OF port = veth của LSP local + output.

**Table 39: `OFTABLE_CHECK_LOOPBACK` — chống loopback egress.**

Sau OUTPUT, packet đôi khi quay lại pipeline ingress (case hairpin, multicast). Table 39 kiểm tra MLF_ALLOW_LOOPBACK; nếu KHÔNG set, drop packet để tránh vòng vô hạn.

**Table 64: `OFTABLE_SAVE_INPORT` — lưu inport trước hairpin.**

Khi packet đi vào hairpin path, OVN cần nhớ original inport để stage sau (NAT_HAIRPIN) có thể compute SNAT đúng. Table 64 lưu reg14 vào một register backup.

**Table 65: `OFTABLE_LOG_TO_PHY` — output cuối ra dây.**

Stage sau khi packet đã ra khỏi cả ingress + egress logical pipeline. Emit packet ra physical port (NIC vật lý qua tunnel) hoặc local LSP veth.

**Table 66-67: `OFTABLE_MAC_BINDING` + `MAC_LOOKUP` — MAC binding cache.**

Cài flow ARP/ND learned vào MAC_Binding table. Hai stage tách bạch: 66 cho insert, 67 cho lookup.

**Table 68-70: LB hairpin pipeline.**

Ba stage chuyên biệt cho LB hairpin (case packet đi qua LB, backend cùng LSP với client). Detect + SNAT để backend không nhìn thấy chính nó.

**Table 71-72: FDB pipeline.**

Cài + lookup FDB cho L2 unknown unicast.

## 13.16.9 Cross-pipeline metadata — packet đi giữa LS↔LR

Một packet tunnel-routed thường đi qua nhiều logical datapath: LS-A (ingress) → LR (ingress + egress) → LS-B (ingress + egress). Mỗi transition giữa datapath, ovn-controller phải:

1. Reset register tới giá trị thích hợp cho datapath đích
2. Cập nhật `metadata` field (datapath ID) cho datapath đích
3. Set reg14 (inport) = tunnel_key của port nối hai datapath (peer LSP với LRP)
4. Resubmit sang OFTABLE_LOG_INGRESS_PIPELINE (table 8) để bắt đầu ingress pipeline mới

Điều này được thực hiện trong stage cuối ingress của datapath nguồn (`LS_IN_L2_LKUP` hoặc `LR_IN_GW_REDIRECT`/`ARP_RESOLVE`). Khi outport trỏ vào peer port (LRP với patch type), stage emit:
- `set_field:<peer_dp_id>->metadata`
- `set_field:<peer_inport>->reg14`
- `resubmit(,8)` (về table 8 = OFTABLE_LOG_INGRESS_PIPELINE)

Packet "nhảy" từ datapath này sang datapath khác mà không đi qua physical wire — đây là essence của **distributed logical routing** trong OVN.

> **Source verify:** Logic transition này nằm trong `controller/lflow.c` hàm `consider_logical_flow()` (OVN branch-22.03), khi xử lý action `next` với target khác datapath. Cross-link Phần 13.7 §13.7.8 `physical.c` walkthrough để hiểu chi tiết hơn về `put_resubmit()` calls.

## 13.16.10 Anatomy 1 — đọc `ovs-ofctl dump-flows br-int table=N`

**Bucket:** OVN | **Context:** debug pipeline qua dump trực tiếp OF table | **Purpose:** xác định stage logic + datapath cho mỗi flow | **Activity:** debug + verification flow installation | **Mechanism:** combine OF table number + metadata + reg state để decode.

**Output mẫu:**

```
$ ovs-ofctl dump-flows br-int table=20 | head -3
 cookie=0x12345678, duration=10s, table=20, n_packets=42, n_bytes=3528,
   priority=110, ct_state=+new+trk, metadata=0x6, reg0=0x80000/0x80000
   actions=group:1
```

**Decode 7-attribute Anatomy:**

| Attribute | Giá trị mẫu | Healthy / Warning / Critical |
|-----------|-------------|------------------------------|
| `cookie=0x12345678` | UUID prefix của Logical_Flow trong SB | Healthy: cookie liên kết với SB row UUID. Warning: cookie=0 = flow không phải do ovn-controller cài. |
| `table=20` | OF table 20 = ingress pipeline logical table 12 | Áp công thức: 20 - 8 = 12 → `LS_IN_LB` (hoặc `LR_IN_DNAT_AND_DEFRAG` nếu metadata trỏ LR) |
| `metadata=0x6` | Datapath ID = 6 | Tra cross qua `ovn-sbctl find Datapath_Binding tunnel_key=6` |
| `reg0=0x80000/0x80000` | bit 19 set | bit 19 không trong 22.03.8; nếu thấy = bug |
| `priority=110` | Match cao priority | Stage LB thường priority 100-200 |
| `ct_state=+new+trk` | Connection mới đã trk | Phù hợp stage LB DNAT |
| `actions=group:1` | LB DNAT qua group table | Group ID 1 = LB backend pool |

**Critical signal:**
- Nếu `cookie=0` ở table 8-49 → flow không phải OVN, cần investigate (có thể attack hoặc misconfig)
- Nếu thấy flow ở table 8 mà thiếu `metadata=...` match → port-security đang bị bypass cho mọi datapath, BIG security issue

## 13.16.11 Anatomy 2 — `ovn-sbctl lflow-list` với stage filter

**Output mẫu:**

```
$ ovn-sbctl lflow-list ls0 | grep "ls_in_lb"
  table=12(ls_in_lb        ), priority=110  , match=(ct.new && ip4.dst == 10.96.1.10 && tcp.dst == 80),
    action=(reg1 = 10.96.1.10; reg2 = 80; ct_lb_mark(backends=10.0.0.1:8080,10.0.0.2:8080);)
```

**Decode:**

- `table=12(ls_in_lb)`: stage 12 trong LS_IN pipeline = `LS_IN_LB`. OF table tương ứng = 8 + 12 = 20.
- `priority=110`: priority trong stage LB
- `match=(ct.new && ip4.dst == 10.96.1.10 && tcp.dst == 80)`: ACL expression OVN
- `action=(reg1 = 10.96.1.10; reg2 = 80; ct_lb_mark(...))`: lưu original DIP + DPort vào reg1+reg2 (per Phần 13.17 §13.17.3) rồi LB DNAT

**Cross-link với dump-flows:**

```
# Tìm OF flow tương ứng cho lflow này
$ ovs-ofctl dump-flows br-int table=20 | grep "metadata=0x$(ovn-sbctl get Datapath_Binding ls0 tunnel_key | tr -d '\"')"
```

## 13.16.14 Bài tập có hướng dẫn (LR pipeline)

**Mục tiêu:** trace cross-subnet ping qua LR_IN 20 stage + LR_OUT 7 stage.

**Setup:**

```bash
ovn-nbctl ls-add ls-blue
ovn-nbctl ls-add ls-red
ovn-nbctl lr-add lr0
ovn-nbctl lrp-add lr0 lrp-blue 00:00:00:00:01:01 192.168.1.1/24

ovn-nbctl lsp-add ls-red ls-red-lrp -- \

  lsp-set-addresses ls-red-lrp router -- \

```

**Trace ICMP từ vm-blue đến vm-red:**

```bash
ovn-trace --detailed lr0 \
  'inport == "lrp-blue" && eth.src == 00:00:00:01:00:01 && eth.dst == 00:00:00:00:01:01 \
   && ip4.src == 192.168.1.10 && ip4.dst == 192.168.2.10 && ip.ttl == 64 && ip.proto == 1'
```

Verify packet đi qua LR_IN 20 stage:
- Stage 0 `lr_in_admission`: pass (eth.dst = router MAC)
- Stage 11 `lr_in_ip_routing`: lookup → next-hop on lrp-red
- Stage 15 `lr_in_arp_resolve`: resolve MAC of vm-red từ MAC_Binding
- Stage 18 `lr_in_gw_redirect`: skip (không có gateway)

Sau đó packet "nhảy" sang LS_IN của ls-red, đi qua 26 stage LS_IN, kết thúc ở stage 24 `ls_in_l2_lkup` với outport = vm-red.

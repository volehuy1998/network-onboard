## Why R1.A exists

R1.A is the simplest possible install path. The Ubuntu archive ships an Open vSwitch package built and tested by Canonical's quality engineering process. A learner who runs `apt-get install openvswitch-switch` gets a working OVS in under a minute, with no compiler, no kernel-headers prerequisite, and no autoconf-aware shell required. R1.A captures that path end to end on `lab-openvswitch` so every later sprint of plan v3.13 can reference what a clean distro-package install looks like and what state it leaves behind.

R1.A also exposes the **trade-offs** the path makes versus paths B and C: the operator gets the version Ubuntu shipped (`2.17.9-0ubuntu0.22.04.1` on jammy), bound to the kernel module the kernel ships, in the directory layout Debian conventions dictate. R1.A never builds source; it never sees `configure` or `make`. The build-system theory file `sdn-onboard/source/v3.13-S0` walks what path A skips.

The third reason R1.A exists is to record an operationally significant finding that no documentation makes obvious: `apt-get purge openvswitch-switch` does not return the host to a clean baseline. The package's post-removal scripts leave the OVSDB database file at `/var/lib/openvswitch/conf.db` on disk and leave the `openvswitch` kernel module loaded. An operator who issues `apt-get purge` and then `apt-get install` again, expecting a fresh start, instead boots the daemon against the prior database, with whatever bridges, ports, controllers, and SSL certificates were configured before. The transcript captures the residual state and the manual cleanup commands a careful operator must run to truly reset the host.

## Pre-install state

Before any package is installed, anchor the session in absolute time with `date -u`. The transcript that follows will be cited from later sprints, and any comparison against another transcript depends on knowing exactly when this one ran.

```text
root@lab-openvswitch:~# date -u
Wed Apr 29 05:13:20 PM UTC 2026
```

The next step is to confirm the host is genuinely green-field. A learner who skips this confirmation cannot later distinguish "the install worked" from "the install was a no-op because Open vSwitch was already there from an earlier session". Three pieces of state matter. First, the on-disk layout. The Debian-packaged Open vSwitch creates three directories during install (`/etc/openvswitch/` for configuration, `/var/lib/openvswitch/` for the OVSDB database file, `/var/log/openvswitch/` for daemon logs); their absence is the cleanest signal that no install has happened yet. Second, the kernel module. If `openvswitch.ko` is already loaded, an earlier session left a footprint that this run will inherit. Third, the disk free space, which sets the baseline against which the install's footprint will be measured.

```text
root@lab-openvswitch:~# ls -la /etc/openvswitch /var/lib/openvswitch /var/log/openvswitch 2>&1
ls: cannot access '/etc/openvswitch': No such file or directory
ls: cannot access '/var/lib/openvswitch': No such file or directory
ls: cannot access '/var/log/openvswitch': No such file or directory
root@lab-openvswitch:~# lsmod | grep -E "^(openvswitch|nf_conntrack|nf_nat)" || echo "(no openvswitch module loaded)"
(no openvswitch module loaded)
root@lab-openvswitch:~# df -h / | tail -1
/dev/mapper/ubuntu--vg-root--lv  8.1G  3.8G  3.9G  50% /
```

All three signals confirm the green-field state. The three Open vSwitch directories are absent, so `ls` reports them missing instead of listing their contents. The kernel module is not loaded, so `lsmod | grep` finds no row and falls through to the explicit failure message. The root filesystem has 3.9 GiB of free space, the same value the R0 baseline recorded; no disk has been consumed in the gap between the two sprints. The host is ready for the install path A to take effect.

The install pulls `openvswitch-switch`, `openvswitch-common`, plus their dependencies (`python3-openvswitch`, `python3-sortedcontainers`, `libevent-2.1-7`, `libunbound8`).

```text
Created symlink /etc/systemd/system/multi-user.target.wants/openvswitch-switch.service → /lib/systemd/system/openvswitch-switch.service.

Processing triggers for libc-bin (2.35-0ubuntu3.13) ...
```

Four substantive details to notice:

1. **`update-alternatives` is used** to register `ovs-vswitchd` under `/usr/sbin/`. This means a future install of an out-of-tree build can install a competing binary at a different alternatives priority and the `update-alternatives` system arbitrates.
2. **Two systemd units are linked into the active target.** `openvswitch-switch.service` is the umbrella; `ovs-record-hostname.service` is a one-shot helper that writes the host's hostname into `external_ids:hostname` on the `Open_vSwitch` row at boot.
3. **The post-install scripts start the daemons.** No explicit `systemctl start` is needed; the package's `postinst` script handles it.
4. The post-install message about `Running kernel seems to be up-to-date.` is from `needrestart`, an Ubuntu helper that scans for processes using outdated libraries. The package install did not require a kernel reload.

### Post-install verification

Four binaries are checked, reporting `2.17.9` consistently. The OVSDB schema version `8.3.0` is the schema the bundled `vswitch.ovsschema` declares.

```text
root@lab-openvswitch:~# ovs-vswitchd --version
ovs-vswitchd (Open vSwitch) 2.17.9
root@lab-openvswitch:~# ovs-vsctl --version
ovs-vsctl (Open vSwitch) 2.17.9
DB Schema 8.3.0
root@lab-openvswitch:~# ovsdb-server --version
ovsdb-server (Open vSwitch) 2.17.9
root@lab-openvswitch:~# ovs-ofctl --version | head -5
ovs-ofctl (Open vSwitch) 2.17.9
OpenFlow versions 0x1:0x6
```

`OpenFlow versions 0x1:0x6` means `ovs-ofctl` speaks OpenFlow protocol numbers 1 through 6, which corresponds to OF 1.0, 1.1, 1.2, 1.3, 1.4, 1.5. R3 (OpenFlow programming sprint) exercises this range.

The two daemons are reported as active by `systemctl`, and `ps` shows their full command lines:

```text
root@lab-openvswitch:~# systemctl is-active openvswitch-switch ovs-vswitchd ovsdb-server
active
active
active
root@lab-openvswitch:~# ps -ef | grep -E "ovs-vswitchd|ovsdb-server" | grep -v grep
root        4860       1  0 17:14 ?        00:00:00 ovsdb-server /etc/openvswitch/conf.db -vconsole:emer -vsyslog:err -vfile:info --remote=punix:/var/run/openvswitch/db.sock --private-key=db:Open_vSwitch,SSL,private_key --certificate=db:Open_vSwitch,SSL,certificate --bootstrap-ca-cert=db:Open_vSwitch,SSL,ca_cert --no-chdir --log-file=/var/log/openvswitch/ovsdb-server.log --pidfile=/var/run/openvswitch/ovsdb-server.pid --detach
root        4914       1  0 17:14 ?        00:00:00 ovs-vswitchd unix:/var/run/openvswitch/db.sock -vconsole:emer -vsyslog:err -vfile:info --mlockall --no-chdir --log-file=/var/log/openvswitch/ovs-vswitchd.log --pidfile=/var/run/openvswitch/ovs-vswitchd.pid --detach
```

The `ovsdb-server` command line shows the SSL-related flags pre-wired (the daemon listens for SSL connections if the appropriate certificates are present in OVSDB; absent certificates mean the SSL listener is silent). The `--remote=punix:/var/run/openvswitch/db.sock` is the Unix-socket endpoint that `ovs-vswitchd` connects to; the `--mlockall` on `ovs-vswitchd` is the package's choice to lock all pages into memory to avoid swap-induced latency in the slow path.

## On-disk and kernel-module state after install

The package created the on-disk layout under `/etc/openvswitch/`, `/var/lib/openvswitch/`, `/var/log/openvswitch/`, plus loaded three kernel modules.

### On-disk state after install

Three directories now exist:

```text
root@lab-openvswitch:~# ls -la /etc/openvswitch/
total 20
drwxr-xr-x  2 root root  4096 Apr 29 17:14 .
drwxr-xr-x 99 root root 12288 Apr 29 17:14 ..
lrwxrwxrwx  1 root root    28 Apr 29 17:14 conf.db -> /var/lib/openvswitch/conf.db
lrwxrwxrwx  1 root root    36 Apr 29 17:14 .conf.db.~lock~ -> /var/lib/openvswitch/.conf.db.~lock~
-rw-r--r--  1 root root    37 Apr 29 17:14 system-id.conf
root@lab-openvswitch:~# ls -la /var/lib/openvswitch/
total 28
drwxr-xr-x  2 root root  4096 Apr 29 17:14 .
drwxr-xr-x 41 root root  4096 Apr 29 17:14 ..
-rw-r-----  1 root root 16564 Apr 29 17:14 conf.db
-rw-------  1 root root     0 Apr 29 17:14 .conf.db.~lock~
root@lab-openvswitch:~# ls -la /var/log/openvswitch/
total 16
drwxr-xr-x  2 root root   4096 Apr 29 17:14 .
drwxrwxr-x 11 root syslog 4096 Apr 29 17:14 ..
-rw-r-----  1 root root    355 Apr 29 17:14 ovsdb-server.log
-rw-r-----  1 root root    527 Apr 29 17:14 ovs-vswitchd.log
```

Two non-obvious details:

1. **`/etc/openvswitch/conf.db` is a symlink to `/var/lib/openvswitch/conf.db`.** The Ubuntu package separates configuration (in `/etc`) from variable state (in `/var/lib`) per the Filesystem Hierarchy Standard, then symlinks them so daemon command lines can reference `/etc/openvswitch/conf.db` while the actual file lives under `/var/lib/`.
2. **`/var/log/openvswitch/` is owned by `root` but the parent `/var/log/` is owned by `syslog`.** The asymmetric ownership is a Debian convention; the OVS-specific subdirectory is administered by the package, not by `rsyslog`.

`system-id.conf` is a 37-byte file produced at install time. It contains the host's persistent OVS system UUID, used when the daemon starts to populate the `Open_vSwitch.external_ids:system-id` column.

Notable observations:

## Uninstall and the residual-finding cleanup

The single most surprising property of path A: `apt-get purge` does not restore the host to baseline. The database file and the kernel module persist.

### Uninstall

`apt-get purge` removes the packages and the configuration:


[24 other lines omitted, context: dpkg progress percentage spam between Removing and Purging steps; full sequence preserved at sdn-onboard/labs/v3.13-R1A-apt-distro-install.typescript]

The single most important line: **`dpkg: warning: while removing openvswitch-common, directory '/var/log/openvswitch' not empty so not removed`**. The package's post-removal script found log files in `/var/log/openvswitch/` and declined to delete the directory because doing so would have destroyed user data. The behaviour is correct from a Debian-policy point of view (preserve user-generated files across package removal), but it surprises an operator who reads "purge" and expects the filesystem to look untouched. The directory survives, the database file at `/var/lib/openvswitch/conf.db` survives (next section), and the kernel module survives in memory. Three forms of residual state, none of which a casual reading of `apt-get purge` would predict.

---

### Post-uninstall state, the residual finding

After the purge completes, the host's three Open vSwitch directories are inspected one more time. The expectation is that `apt-get purge` returns the host to the pre-install layout; the actual finding is more subtle.

```text
root@lab-openvswitch:~# ls -la /etc/openvswitch /var/lib/openvswitch /var/log/openvswitch 2>&1
ls: cannot access '/etc/openvswitch': No such file or directory
/var/lib/openvswitch:
total 28
drwxr-xr-x  2 root root  4096 Apr 29 17:14 .
drwxr-xr-x 41 root root  4096 Apr 29 17:14 ..
-rw-r-----  1 root root 16564 Apr 29 17:14 conf.db
-rw-------  1 root root     0 Apr 29 17:14 .conf.db.~lock~

/var/log/openvswitch:
total 8
drwxr-xr-x  2 root root   4096 Apr 29 17:15 .
drwxrwxr-x 11 root syslog 4096 Apr 29 17:14 ..
```

`/etc/openvswitch/` is gone. `/var/lib/openvswitch/` is **not** gone: `conf.db` (16,564 bytes) and the lock file are still there. `/var/log/openvswitch/` is empty but the directory still exists.

This is the residual finding. A naive operator who expects `apt-get purge` to mean "back to baseline" and then runs `apt-get install openvswitch-switch` again will boot into the prior `conf.db`, complete with whatever bridges, ports, controllers, and ssl certificates were configured. The behaviour is reproducible and not a bug; it is Debian-package-cleanup convention. R1.A pins this convention into the historical record.

The `ovs-vsctl` binary is gone, the `openvswitch-switch.service` is inactive, but the kernel module is still loaded:

```text
root@lab-openvswitch:~# ovs-vsctl --version 2>&1 || echo "(binary removed)"
bash: /usr/bin/ovs-vsctl: No such file or directory
(binary removed)
root@lab-openvswitch:~# lsmod | grep -E "^openvswitch" || echo "(module unloaded)"
openvswitch           176128  0
root@lab-openvswitch:~# systemctl is-active openvswitch-switch 2>&1 || true
inactive
```

The kernel module is loaded (`openvswitch  176128  0`) with a refcount of 0. Module unload is operator-driven; the package does not unload modules on remove.

### Manual cleanup back to baseline

To restore the host to the green-field state of R0, the operator removes the residual and unloads the module:

```text
root@lab-openvswitch:~# ls -la /var/log/openvswitch/ 2>&1
total 8
drwxr-xr-x  2 root root   4096 Apr 29 17:15 .
drwxrwxr-x 11 root syslog 4096 Apr 29 17:14 ..
root@lab-openvswitch:~# rm -rf /var/log/openvswitch
root@lab-openvswitch:~# ls -la /var/log/openvswitch 2>&1
ls: cannot access '/var/log/openvswitch': No such file or directory
```

The transcript ends with `rm -rf /var/log/openvswitch`. The reader who wants to fully reset must additionally run `rm -rf /var/lib/openvswitch` and `rmmod openvswitch`. Both of those operations were captured outside the `script` recording (the typescript closed first because `exit` was sent before the final `rmmod`); the lab returned to baseline immediately after the typescript closed.

### What R1.A pins for later sprints

- The Ubuntu archive's candidate version on jammy is `2.17.9-0ubuntu0.22.04.1`. Every later sprint that installs from `apt` reproduces this version.
- The systemd unit topology after `apt install` is umbrella `openvswitch-switch.service` plus per-daemon `ovs-vswitchd.service` and `ovsdb-server.service` plus the one-shot `ovs-record-hostname.service`.
- The OVSDB schema version is `8.3.0`.
- `ovs-ofctl` supports OpenFlow versions 1 through 6.
- `apt-get purge` does not remove `/var/lib/openvswitch/`. The OVSDB transaction log persists across uninstall.
- `apt-get purge` does not unload the `openvswitch` kernel module. The module persists across uninstall with refcount 0.
- The system UUID for this lab host is `12015d1c-1d7c-4209-81b0-14406e1e3269`. Future sprints that examine the `Open_vSwitch` row reference this UUID.

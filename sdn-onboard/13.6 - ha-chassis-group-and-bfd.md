## 13.6.1 Vấn đề: gateway chassis SPOF

Part 13.5 phân tích `chassisredirect` Port_Binding — chỉ định một chassis làm active gateway cho north-south NAT. Nếu chassis đó chết:
- Traffic north-south fails (external traffic không reach VM, VM không reach internet).
- SNAT connection break (NAT state in kernel conntrack của chassis đó mất).
- Manual failover cần admin intervene → MTTR 10-60 phút.

**HA_Chassis_Group** giải quyết: group 2-5 chassis với priority, OVN + BFD tự động failover. Target: MTTR < 2 giây tự động, không need admin action.

## 13.6.2 HA_Chassis_Group schema

NBDB schema:

```
HA_Chassis_Group:
  name           — group identifier
  ha_chassis     — set<HA_Chassis UUID>
  external_ids   — map cho CMS annotation

HA_Chassis:
  chassis_name   — reference Chassis.name trong SBDB
  priority       — 0-32767, higher = preferred
  external_ids   — map cho CMS annotation
```

### Cấu hình

```bash
[admin]# ovn-nbctl ha-chassis-group-add hcg-external
[admin]# ovn-nbctl ha-chassis-group-add-chassis hcg-external chassis-1 100
[admin]# ovn-nbctl ha-chassis-group-add-chassis hcg-external chassis-2 50
[admin]# ovn-nbctl ha-chassis-group-add-chassis hcg-external chassis-3 30
[admin]# ovn-nbctl lrp-set-ha-chassis-group lrp-external hcg-external
```

Priority semantics:
- chassis-1 (100) = preferred active.
- chassis-2 (50) = standby 1 (activate khi chassis-1 fail).
- chassis-3 (30) = standby 2 (activate khi cả 1+2 fail).

### HA_Chassis_Group vs Gateway_Chassis (legacy)

OVN 21.03 trước dùng `Gateway_Chassis` schema — simpler nhưng không có BFD built-in. HA_Chassis_Group replace và extend:

| Feature | Gateway_Chassis (legacy) | HA_Chassis_Group (current) |
|---------|--------------------------|----------------------------|
| BFD integration | Manual (operator script) | Auto (OVN-managed) |
| Preemption control | Không có | `ha_chassis_group_options:preemption` |
| Multi-group support | 1 group per LRP | Nhiều group reference cross-LRP |
| Deprecation | OVN 22.03+ deprecated | OVN 21.03+ default |

Nếu đang dùng Gateway_Chassis, migrate sang HA_Chassis_Group.

## 13.6.3 BFD (RFC 5880) — sub-second failure detection

### Protocol fundamentals

BFD là **lightweight hello protocol** phát hiện peer failure với millisecond granularity. Chạy trên UDP port 3784 (single-hop session).

Packet format (RFC 5880 §4.1):

```
Ver (3 bit) | Diag (5 bit) | State (2 bit) | Flags (6 bit)
Detect Mult (8 bit) | Length (8 bit)
My Discriminator (32 bit)
Your Discriminator (32 bit)
Desired Min TX Interval (32 bit, microseconds)
Required Min RX Interval (32 bit, microseconds)
Required Min Echo RX Interval (32 bit)
```

Key fields:
- **State**: 2-bit, 4 values: AdminDown (0) / Down (1) / Init (2) / Up (3).
- **Detect Mult**: nhân với peer's TX interval → detection time.
- **My/Your Discriminator**: 32-bit session ID giúp demux multiple sessions.

### State machine (3 state + init)

```
  ┌─────────────┐
  │  AdminDown  │ ← admin disable
  └──────┬──────┘
         │ enable
         ▼
  ┌─────────────┐ no packets received X ms
  │    Down     │ ────────────────────────┐
  └──────┬──────┘                         │
         │ hear from peer                 │
         ▼                                │
  ┌─────────────┐ peer says Up           │
  │    Init     │ ──────────────────────┐│
  └──────┬──────┘                       ││
         │ peer says Init/Up            ││
         ▼                              ▼▼
  ┌─────────────┐ detect timeout  ┌─────────────┐
  │     Up      │ ───────────────▶│    Down     │
  └─────────────┘                 └─────────────┘
```

### Timing math

**Default OVN BFD:**
- `min_tx_interval=500000µs` (500ms)
- `min_rx_interval=500000µs` (500ms)
- `detect_multiplier=3`
- Detection time = `min_tx_interval × detect_multiplier` = 1500ms.

**Fast-timer (sub-second):**
- `min_tx_interval=100000µs` (100ms)
- `min_rx_interval=100000µs` (100ms)
- `detect_multiplier=3`
- Detection time = 300ms.

**Extremely fast (production HA trading):**
- `min_tx_interval=10000µs` (10ms)
- `detect_multiplier=3`
- Detection = 30ms.
- **CPU + network cost:** mỗi session = 100 packet/s × N session. 100 session × 100 pps = 10K pps chỉ cho BFD. Chỉ chấp nhận được trên chassis dedicated HA.

### Configure BFD trong OVN

```bash
[admin]# ovn-nbctl set NB_Global . options:bfd-min-rx=100 \
                                          options:bfd-min-tx=100 \
                                          options:bfd-mult=3
```

Applied cross all HA_Chassis_Group. Per-group override qua `HA_Chassis_Group_Options`:

```bash
[admin]# ovn-nbctl set HA_Chassis_Group hcg-external \
         options:bfd-min-rx=50 options:bfd-min-tx=50
```

## 13.6.4 Anatomy Template A — `ovs-appctl bfd/show`

**Bối cảnh khi chạy:** Sau configure HA_Chassis_Group, verify BFD session đã establish + UP giữa chassis members. Cũng là lệnh debug đầu tiên khi failover không trigger.

**Command và output** *(reproduced OVN 22.03.8 trên chassis-1, 2 BFD session tới chassis-2 + chassis-3)*:

```shell
$ sudo ovs-appctl -t ovs-vswitchd bfd/show

Interface ovn-chassis-2-0
        BFD State              : Up
        Session State          : Up
        Forwarding             : true
        Diagnostic             : No Diagnostic
        Detect Multiplier      : 3
        Concatenated Path Down : false
        Local Discriminator    : 0x8123a5f0
        Remote Discriminator   : 0x7f2b1944
        Local Min TX Interval  : 500 ms
        Local Min RX Interval  : 500 ms
        Remote Min RX Interval : 500 ms
        Poll Seq In Progress   : false
        Remote State           : Up
        Local Flag             : C
        Remote Flag            : None
        Loss Detect Timeout    : 1500 ms

Interface ovn-chassis-3-0
        BFD State              : Up
        Session State          : Up
        Forwarding             : true
        Detect Multiplier      : 3
        Local Discriminator    : 0x8124a5f1
        Remote Discriminator   : 0x9a3c4021
        Loss Detect Timeout    : 1500 ms
```

**9-attribute anatomy:**

| Field | Giá trị mẫu | Ý nghĩa | Red flag nếu |
|-------|-------------|---------|--------------|
| `Interface` | `ovn-chassis-2-0` | Tunnel port name | Missing interface = tunnel port mất, BFD không setup được |
| `BFD State` | `Up` / `Down` / `Init` | Overall BFD state | `Down` persistent = chassis peer unreachable |
| `Session State` | `Up` | Session-level state | `Init` sau 10s = one-way BFD (peer không respond) |
| `Forwarding` | `true` | BFD cho phép forward traffic | `false` = ovn-controller block traffic do BFD down |
| `Detect Multiplier` | `3` | Number of missed packets trước DOWN | <3 = aggressive detection, false positive risk |
| `Local Discriminator` | `0x8123a5f0` | Session ID local-side | Changes mỗi restart = BFD reset |
| `Remote Discriminator` | `0x7f2b1944` | Session ID peer-side | `0x00000000` = BFD chưa nhận packet nào từ peer |
| `Local Min TX Interval` | `500 ms` | Tốc độ gửi BFD packet | Mismatch với peer's min_rx_interval = negotiation |
| `Remote Min RX Interval` | `500 ms` | Peer's acceptable RX rate | Giới hạn local TX rate |
| `Loss Detect Timeout` | `1500 ms` | tx × multiplier = detection time | Peer flap nếu timeout quá ngắn |
| `Poll Seq In Progress` | `false` | Poll sequence trong progress (parameter change) | `true` stuck = parameter negotiation failed |

**Kịch bản bẻ gãy:**

- **`Session State: Down` persistent + `Remote Discriminator: 0x00000000`**: peer chassis không sending BFD. Causes: ovn-controller crashed on peer, tunnel port MTU issue blocking BFD, firewall block UDP 3784.
- **`Session State: Init` không chuyển sang Up**: one-way BFD — local sending, peer receiving, nhưng peer sending back không reach local. Asymmetric firewall rule hoặc tunnel MTU asymmetric.
- **`Loss Detect Timeout` không match expected config**: BFD negotiation fallback to slowest. Check peer's `min_rx_interval` — nếu lớn hơn local's `min_tx_interval`, BFD use the slower rate.
- **`Forwarding: false` + `BFD State: Up`**: `bfd_slow_timer_expired` flag. Manual check `ovs-appctl bfd/show -v` for detail.

## 13.6.5 Failover workflow — step-by-step

Scenario: chassis-1 active (priority 100) crash. chassis-2 (priority 50) stand-by.

```
T+0ms      chassis-1 kernel panic + reboot
T+0ms      BFD session chassis-2 ↔ chassis-1 stop receiving packets from c1
T+500ms    missed 1 BFD packet
T+1000ms   missed 2 BFD packets
T+1500ms   chassis-2 BFD state transition UP → DOWN (detect timeout 3 × 500ms)
T+1500ms   ovn-controller chassis-2 update HA_Chassis_Group:
           - ovn-sbctl update HA_Chassis_Group active=chassis-2
T+1600ms   SBDB Raft commit (3-node latency 50-100ms)
T+1650ms   ovn-northd regenerate Logical_Flow:
           - Port_Binding chassisredirect chassis field → chassis-2
T+1700ms   ovn-controller chassis-2 + chassis-3 subscribe change
T+1750ms   chassis-2 install OpenFlow rule cho NAT:
           - ARP announce 192.0.2.1 từ chassis-2's br-ex
T+1800ms   upstream switch/router update ARP cache
T+1850ms   Traffic north-bound flow through chassis-2
T+2000ms   VM ping 8.8.8.8 resumes
```

**Tổng downtime typical: 1.5-2.5s** với default BFD. Với fast-timer 100ms: 400-600ms downtime.

### Recovery + preemption

Khi chassis-1 recover (reboot xong, ovn-controller start):
1. Re-register Chassis row trong SBDB.
2. BFD resume.
3. **Nếu preemption enabled (default)**: chassis-1 priority 100 > chassis-2 priority 50 → fail-back tới chassis-1.
4. **Nếu preemption disabled**: chassis-2 giữ active, chassis-1 standby.

**Đọc config preemption:**
```bash
$ sudo ovn-nbctl get HA_Chassis_Group hcg-external options:preemption
```

Default: `true` (preemption on). Production best practice: set `false` tránh flapping nếu chassis-1 unstable.

```bash
[admin]# ovn-nbctl set HA_Chassis_Group hcg-external options:preemption=false
```

> **Key Topic:** Preemption off vs on là quyết định operational quan trọng. Preemption on = recover tự về preferred chassis (maintain priority invariant, ops visibility). Preemption off = giữ stability sau failover (tránh flapping nếu chassis restart liên tục). Rule of thumb: preemption **on** trong dev/staging (visibility), **off** trong production (stability). Backup: monitoring alert khi preemption off nhưng active != highest priority chassis — có thể là signal cần ops intervention.

## 13.6.6 Common failure modes

> **Hiểu sai phổ biến:** *"BFD phát hiện chassis fail trong 100ms luôn luôn."*
>
> **Thực tế:** BFD phát hiện trong **`min_tx × detect_multiplier` × 2 worst-case** — nếu chassis fail NGAY SAU khi vừa gửi BFD packet, peer phải đợi hết tx interval thêm 1 lần nữa. Detection time thực tế 1x-2x tính toán. Thêm nữa, SBDB Raft commit (50-100ms), ovn-northd regenerate (50-200ms), ovn-controller apply (50-100ms), ARP re-announce (100-500ms tùy upstream switch). Tổng end-to-end failover với default BFD: 1.5-3s. Đừng hứa 100ms downtime với BFD default; đó là BFD detection only, không phải end-to-end recovery.

**Failure 1 — BFD flapping do network jitter.**
Dấu hiệu: `ovs-appctl bfd/show` show BFD state toggle UP/DOWN mỗi 10-30s. ovn-sbctl log cho thấy HA_Chassis_Group active chassis đổi liên tục.
Causes: underlay network latency spike (vượt `min_rx × mult`), chassis CPU overloaded (không process BFD packet kịp).
Fix: (1) Increase `detect_multiplier` 3 → 5 (tolerance more missed packets). (2) Disable preemption. (3) Dedicate gateway chassis (không mix workload).

**Failure 2 — One-way BFD (session stuck ở Init).**
Dấu hiệu: local `BFD State: Init` không chuyển Up sau 30s.
Causes: firewall block UDP 3784 outbound/inbound asymmetric. Tunnel MTU asymmetric.
Fix: `tcpdump -i tunnel-interface udp port 3784` — nếu thấy packet one-way only, khắc phục sự cố firewall/MTU.

**Failure 3 — BFD Up nhưng Forwarding false.**
Dấu hiệu: `BFD State: Up` + `Forwarding: false`.
Causes: slow timer expired (BFD parameter change trong flight). Local flag `C` without remote flag acknowledge.
Fix: restart ovn-controller trên chassis để reset BFD session.

**Failure 4 — Failover không trigger dù chassis crash.**
Dấu hiệu: chassis-1 crash nhưng traffic vẫn route qua chassis-1 shadow (Port_Binding chassisredirect không update).
Causes: ovn-controller trên chassis-2 crashed hoặc OVSDB connection dead.
Fix: `ovn-appctl -t ovn-controller connection-status` trên chassis-2.

## 13.6.7 Best practices production

1. **3 HA_Chassis members tối thiểu**, 5 cho mission-critical. 2 member = split-brain risk nếu network partition.
2. **Disable preemption** trong production (`preemption=false`).
3. **Dedicate gateway chassis** — không mix VM workload (tránh CPU contention với BFD processing).
4. **Monitor BFD state** qua Prometheus + `ovs_exporter` (port 9475). Alert rule: session DOWN > 30s = critical.
5. **Quarterly failover drill**: kill ovn-controller chassis-1 trong maintenance window, verify migrate < 2s + fail-back đúng (hoặc không, tùy preemption).
6. **Tune BFD theo workload**: development lab = default (500ms), production = 250ms, HFT/trading = 100ms với dedicated CPU core.
7. **Log BFD transition**: `ovn-appctl bfd/show --verbose` scheduled qua cron, log 24h rolling cho post-incident review.

## 13.6.8 ▶ Capstone Block XIII — Full OVN lab 3-chassis HA + BFD failover

**Mục đích:** Deploy 3-chassis OVN với HA_Chassis_Group, test failover end-to-end với metrics.

**Mức độ:** Senior — yêu cầu 3 Ubuntu VM với OVN 22.03.8 + underlay network routable giữa 3 chassis.

**Before You Begin:**
- 3 VM Ubuntu 22.04 (`chassis-1` / `chassis-2` / `chassis-3`) + 1 VM riêng `ovn-central` (NBDB + SBDB + ovn-northd).
- SBDB cluster 3-node (optional nhưng recommended cho full HA).
- Firewall allow UDP 3784 inter-chassis.
- Underlay routable: ping chassis-to-chassis < 1ms latency ideal.

**Instructions:**

**1.** Setup OVN central:
```bash
ovn-central$ sudo apt install -y ovn-central
ovn-central$ sudo ovn-nbctl set-connection ptcp:6641
ovn-central$ sudo ovn-sbctl set-connection ptcp:6642
```

**2.** Setup 3 chassis (loop qua chassis-1/2/3):
```bash
chassis-N$ sudo apt install -y ovn-host
chassis-N$ sudo ovs-vsctl set Open_vSwitch . \
    external_ids:ovn-remote=tcp:<central-ip>:6642 \
    external_ids:ovn-encap-ip=<chassis-N-ip> \
    external_ids:ovn-encap-type=geneve \
    external_ids:ovn-bridge-mappings=physnet1:br-ex
chassis-N$ sudo ovs-vsctl add-br br-ex
chassis-N$ sudo ovs-vsctl add-port br-ex eth1
chassis-N$ sudo systemctl start ovn-controller
```

**3.** HA_Chassis_Group setup:
```bash
ovn-central$ sudo ovn-nbctl ha-chassis-group-add hcg-gw
ovn-central$ for i in 1 2 3; do
    priority=$((110 - i*20))  # 90, 70, 50
    chassis_uuid=$(ovn-sbctl --columns=_uuid list Chassis | grep -A 0 "chassis-$i" | head -1 | awk '{print $NF}')
    sudo ovn-nbctl ha-chassis-group-add-chassis hcg-gw $chassis_uuid $priority
done
```

**4.** Gateway router + provider network:
```bash
ovn-central$ sudo ovn-nbctl lr-add lr-tenant
ovn-central$ sudo ovn-nbctl lrp-add lr-tenant lrp-ext 40:44:00:00:ff:01 192.0.2.1/24
ovn-central$ sudo ovn-nbctl lrp-set-ha-chassis-group lrp-ext hcg-gw
ovn-central$ sudo ovn-nbctl ls-add ls-public
ovn-central$ sudo ovn-nbctl lsp-add ls-public provider-port \
    -- lsp-set-type provider-port localnet \
    -- lsp-set-addresses provider-port unknown \
    -- lsp-set-options provider-port network_name=physnet1
```

**5.** Verify chassis-1 active:
```bash
ovn-central$ sudo ovn-sbctl list Port_Binding cr-lrp-ext | grep chassis
# Expected: chassis=[<chassis-1-uuid>]
```

**6.** Create tenant VM + launch continuous ping:
```bash
# Setup LSP cho VM trong tenant LS connected LR
# (full LS + VM tap setup theo pattern GE 1 trong 13.4)
vm-in-tenant$ ping -i 0.1 -c 200 8.8.8.8 | tee /tmp/ping-failover.log &
```

**7.** Failover test — kill ovn-controller chassis-1:
```bash
chassis-1$ sudo systemctl stop ovn-controller
# Monitor BFD + HA_Chassis transition:
chassis-2$ watch -n 0.1 'sudo ovs-appctl bfd/show | grep -E "Interface|BFD State"'
chassis-2$ tail -f /var/log/ovn/ovn-controller.log | grep -i "ha_chassis"
```

**8.** Observe results:
- `tmp/ping-failover.log` — count packet loss window (expect 15-20 packet loss với 100ms interval = 1.5-2s).
- `ovn-sbctl list Port_Binding cr-lrp-ext` — chassis field now chassis-2 UUID.
- `/var/log/ovn/ovn-controller.log` — BFD DOWN event + HA transition log.

**9.** Recovery:
```bash
chassis-1$ sudo systemctl start ovn-controller
# Nếu preemption enabled: chassis-1 reclaim active sau ~2s.
# Nếu preemption disabled: chassis-2 giữ active.
```

**10.** POE — analyze metrics:

- **Predict**: Packet loss count trong failover window với default BFD (500ms interval, mult 3)?
- **Observe**: `cat /tmp/ping-failover.log | grep "packet loss"` — typical 15-25 packet (1.5-2.5s @ 100ms interval).
- **Explain**: Breakdown breakdown:
  - BFD detect: 500 × 3 = 1500ms
  - SBDB Raft + ovn-northd regen: 100-200ms
  - ovn-controller apply + ARP announce: 200-500ms
  - Total: ~2s matches observation
- **Falsification test**: Retry với `bfd-min-rx=100 bfd-min-tx=100`. Predict downtime 600-800ms. Observe — verify formula scale linearly.

**11.** Cleanup:
```bash
ovn-central$ sudo ovn-nbctl ls-del ls-public
ovn-central$ sudo ovn-nbctl lr-del lr-tenant
ovn-central$ sudo ovn-nbctl ha-chassis-group-del hcg-gw
chassis-N$ sudo systemctl stop ovn-controller
chassis-N$ sudo ovs-vsctl del-br br-int
chassis-N$ sudo ovs-vsctl del-br br-ex
```

## 13.6.9 Điểm cốt lõi cần nhớ

Thứ nhất, HA_Chassis_Group là OVN abstraction cho gateway failover — replace legacy Gateway_Chassis schema với BFD integration auto-managed.

Thứ hai, BFD (RFC 5880) detect sub-second với packet format fixed + state machine 3-state (INIT/UP/DOWN). Detection time = `min_tx × detect_multiplier`. Default 1500ms, fast-timer 300ms, extreme 30ms với CPU cost.

Thứ ba, end-to-end failover time = BFD detection + SBDB Raft commit + northd regenerate + ovn-controller apply + ARP announce ≈ 1.5-3s default, 400-600ms fast-timer. Đừng hứa chỉ 100ms — đó là BFD only.

Thứ tư, preemption off trong production tránh flapping. Fail-back manual hoặc auto-schedule qua maintenance window.

Thứ năm, 7 best practices: 3+ member / preemption off / dedicate gateway / Prometheus BFD monitor / quarterly drill / BFD tune theo workload / log transition.

Thứ sáu, 4 common failure modes: BFD flapping (jitter), one-way BFD (firewall/MTU asymmetric), Forwarding=false stuck (parameter change), failover không trigger (ovn-controller crash). Mỗi cái có signature + fix rõ.

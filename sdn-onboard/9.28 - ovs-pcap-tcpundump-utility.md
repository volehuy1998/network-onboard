## Learning objectives

After completing this part, the engineer can:

1. **Understand** the role of `ovs-pcap` and `ovs-tcpundump` as **two pure reformatter utilities** that do not touch the datapath. They only convert between pcap binary, tcpdump hex text, and the single-line hex format that `ofproto/trace` accepts.
2. **Apply** the workflow `tcpdump` to `ovs-pcap` to `ofproto/trace` in order to **simulate the processing of a real packet** through the OpenFlow pipeline without re-injecting the packet into the datapath.
3. **Analyze** the hex dump output to verify that the Ethernet header is preserved correctly. This requires `tcpdump -xx`, not `-x`. The verification avoids a common bug where the Ethernet header is missing and `ofproto/trace` parses the packet at the wrong layer.
4. **Evaluate** the trade-off between "simulate via trace" and "re-inject for real with `ovs-ofctl packet-out`". Trace is fast and safe. Packet-out affects OVSDB state.

Knowledge prerequisites: a basic understanding of the pcap format, `ofproto/trace` from Part 9.25, and basic `tcpdump` usage.

## 9.28.1 `ovs-pcap`, pcap binary to single-line hex

`ovs-pcap` is a pure reformatter utility. It reads a pcap file (the standard libpcap binary format) and prints each packet on its own line as a hex string. It does not touch the datapath, it does not require `ovs-vswitchd` to be running, and it does not read OVSDB.

**Synopsis:**

```
ovs-pcap file
ovs-pcap -h | --help
ovs-pcap -V | --version
```

**The two only options:**

| Option | Purpose |
|--------|---------|
| `-h`, `--help` | Print the help message and exit |
| `-V`, `--version` | Print the OVS version and exit |

**Output format:** every packet in the pcap file is serialized as one line of hex digits (no spaces, no `0x` prefix). For example, an ICMP echo request packet:

```
$ ovs-pcap capture.pcap | head -1
ffffffffffff0011223344550806000108000604000100112233445510000001000000000000c0a80164
```

This is exactly the format that `ovs-appctl ofproto/trace` accepts when simulating a packet. There is no need to parse the pcap file in reverse. The output of `ovs-pcap` can be piped directly.

**Important:** the packet must include the Ethernet header. A normal pcap file (captured by `tcpdump` or `wireshark`) already has the Ethernet header, so `ovs-pcap` works out of the box. If the pcap file was created by an L3-only capture tool (rare), the Ethernet header is missing and `ofproto/trace` will reject the input.

## 9.28.2 `ovs-tcpundump`, tcpdump hex text to single-line hex

`ovs-tcpundump` is the reverse of `ovs-pcap` in terms of input format. It reads the output of `tcpdump -xx` (hex format with the Ethernet header) from standard input. It does not read a binary pcap file.

**Synopsis:**

```
ovs-tcpundump < input.txt
ovs-tcpundump -h | --help
ovs-tcpundump -V | --version
```

**Main use case:** when only the **tcpdump capture text** is available (for example, copied from a production log) and there is no binary pcap file, `ovs-tcpundump` can convert the text to single-line hex so that the engineer can pipe it into `ofproto/trace`.

**Critical warning:** the input must come from `tcpdump -xx` (two `x` characters), not from `tcpdump -x` (one `x` character). The difference:

- `tcpdump -x` (one `x`): prints hex starting from L3 (the IP header), skipping the Ethernet header.
- `tcpdump -xx` (two `x` characters): prints hex starting from L2 (the Ethernet header), including the destination MAC, source MAC, and ethertype.

`ofproto/trace` needs the Ethernet header to know whether the packet is IPv4, IPv6, ARP, or MPLS through the `eth_type` field. Input that lacks the Ethernet header is rejected or produces an incorrect trace. This bug is fairly common, for instance in documentation copy-pasted from older tutorials that used `tcpdump -x`.

**Output:** identical to `ovs-pcap`, one line of hex per packet.

## 9.28.3 Workflow integration with `ofproto/trace`

The two utilities form a bridge between **real packet capture** and **OpenFlow pipeline simulation**. The typical workflow:

```
+-------------+          +-----------+         +----------------+
|   tcpdump   |   pcap   |  ovs-pcap |   hex   | ofproto/trace  |
| (capture)   | -------->| (convert) | ------->| (simulate)     |
+-------------+          +-----------+         +----------------+

OR

+-------------+          +-----------------+         +----------------+
| tcpdump -xx |   text   | ovs-tcpundump   |   hex   | ofproto/trace  |
| (log file)  | -------->| (convert stdin) | ------->| (simulate)     |
+-------------+          +-----------------+         +----------------+
```

**Real commands (case 1, pcap file):**

```bash
# 1. Capture on br-int for 5 seconds
sudo timeout 5 tcpdump -i br-int -w /tmp/cap.pcap

# 2. Convert and pipe into ofproto/trace
ovs-pcap /tmp/cap.pcap | head -1 | xargs -I{} \
  sudo ovs-appctl ofproto/trace br-int "{}"
```

**Real commands (case 2, from text log):**

```bash
# 1. The engineer already has a log file from production
#    (for example, pasted into a support ticket)
cat /tmp/incident-tcpdump.log
# 14:23:01.234567 IP 10.0.0.1.443 > 10.0.0.2.5678: Flags [S]
#         0x0000:  ffff ffff ffff 0011 2233 4455 0800 4500
#         0x0010:  003c 0001 0000 4006 ...

# 2. Convert and trace
cat /tmp/incident-tcpdump.log | ovs-tcpundump | \
  while read pkt; do
    sudo ovs-appctl ofproto/trace br-int "$pkt"
  done
```

## 9.28.4 Anatomy: pipe `ovs-pcap` to `ofproto/trace`

**Bucket:** OVS. **Context:** post-incident analysis on a captured packet. **Purpose:** simulate OpenFlow pipeline processing of a real captured packet without re-injecting into the datapath. **Activity:** debug and verify flow rules. **Mechanism:** `ovs-pcap` reformat, then `ofproto/trace` simulate.

**Sample output:**

```
$ ovs-pcap /tmp/incident.pcap | head -1
ffff...0800450000280001000040067...

$ echo "ffff...0800450000280001000040067..." | xargs -I{} ovs-appctl ofproto/trace br-int "{}"
Flow: in_port=LOCAL,vlan_tci=0x0000,dl_src=00:11:22:33:44:55,dl_dst=ff:ff:ff:ff:ff:ff,dl_type=0x0800,...

bridge("br-int")
----------------
 0. in_port=1, priority 100
    actions=resubmit(,8)
 8. priority 50
    actions=load:0x6->NXM_NX_REG14[],resubmit(,9)
 ...

Final flow: ...
Datapath actions: drop
```

**Decode of the five-attribute Anatomy:**

| Attribute | Sample value | Healthy / Warning / Critical |
|-----------|--------------|------------------------------|
| Hex dump length | 84 characters, equal to 42 bytes (Ethernet 14, IPv4 20, payload 8) | Healthy: at least 14 bytes (full Ethernet header) |
| Flow parsing | `dl_src=...,dl_dst=...,dl_type=0x0800,...` | Critical: if `dl_type` is missing, the Ethernet header is missing. Fix the `tcpdump` capture command. |
| Bridge name | `bridge("br-int")` | Healthy: the bridge exists. If the wrong bridge is named, `ofproto` returns an error. |
| Pipeline trace | List of actions per table | Critical: if `drop` is seen at table 8 (port security), the packet does not match port security. |
| Final flow | The final action set | Healthy: the action contains `output:N` or `output:CONTROLLER`. Critical: `drop` without a reason indicates a buggy rule. |

## 9.28.5 Guided exercise

**Goal:** capture a real ICMP packet, convert it through `ovs-pcap`, then replay it through `ofproto/trace` to verify that the pipeline simulation matches real traffic.

**Setup:**

```bash
# 1. Create a veth pair to simulate two VMs
ip netns add ns1
ip netns add ns2
ip link add veth-ns1 type veth peer name veth-br1
ip link add veth-ns2 type veth peer name veth-br2
ip link set veth-ns1 netns ns1
ip link set veth-ns2 netns ns2
ovs-vsctl add-port br-int veth-br1
ovs-vsctl add-port br-int veth-br2
ip link set veth-br1 up && ip link set veth-br2 up
ip netns exec ns1 ip addr add 10.0.0.1/24 dev veth-ns1
ip netns exec ns1 ip link set veth-ns1 up
ip netns exec ns2 ip addr add 10.0.0.2/24 dev veth-ns2
ip netns exec ns2 ip link set veth-ns2 up
```

**Capture and replay:**

```bash
# 2. Capture while pinging
sudo tcpdump -i br-int -w /tmp/icmp.pcap -c 4 &
sleep 1
ip netns exec ns1 ping -c 2 10.0.0.2
wait

# 3. Convert and simulate
ovs-pcap /tmp/icmp.pcap | head -1 | xargs -I{} ovs-appctl ofproto/trace br-int "{}"

# 4. Verify the expected outcome: the trace ends at output:<the port of veth-br2>
```

**Verify against real traffic:**

```bash
# Dump the corresponding flow to see the expected pipeline
ovs-ofctl dump-flows br-int | grep "in_port=$(ovs-vsctl get Interface veth-br1 ofport)"
```

Compare the action in the trace with the flow in the dump. If they match, the workflow is correct.

## 9.28.6 Anti-patterns and notes

**Anti-pattern 1: using `tcpdump -x` instead of `tcpdump -xx` for `ovs-tcpundump`.**

`-x` (one `x`) drops the Ethernet header. The output of `ovs-tcpundump` then lacks the MAC and ethertype fields, so `ofproto/trace` parses the packet at the wrong layer. Symptom: the trace reports "no Ethernet" or assumes a default MAC address.

**Prevention:** always use `tcpdump -xx -i <iface> -w <file>` for text output that will be fed to `ovs-tcpundump`. For a binary pcap, every `tcpdump -w` already includes the Ethernet header.

**Anti-pattern 2: truncating packets with `tcpdump -s 96`.**

By default `tcpdump` captures the full packet. If `-s 96` is set (snap length 96 bytes), the payload after byte 96 is cut. `ofproto/trace` does not error, but pipeline simulation for a rule that depends on the payload (rare, but ACL match on content does occur) becomes incorrect.

**Prevention:** use `tcpdump -s 0` (the default, full packet) when capturing for simulation purposes.

**Note 3: `ofproto/trace` does not change datapath state.**

Pipeline simulation is read-only. It does not install a flow, it does not commit conntrack, and it does not affect OVSDB. It is safe to run on production.

**Note 4: hex output from `ovs-pcap` can be very long for large packets.**

A 1500-byte MTU packet equals 3000 hex characters. Bash variables and `xargs` have an `ARG_MAX` limit. For large packets, use an intermediate file:

```bash
ovs-pcap /tmp/cap.pcap | head -1 > /tmp/pkt.hex
ovs-appctl ofproto/trace br-int "$(cat /tmp/pkt.hex)"
```

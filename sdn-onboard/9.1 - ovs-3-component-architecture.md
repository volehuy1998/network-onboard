## Context: why does OVS split into three separate components?

A common question for newcomers to OVS is: "If the Linux bridge is just one kernel module, why does OVS need three different components?" The answer lies in a critical architectural requirement that OVS had to satisfy from its earliest design days (2007): **the ability to change forwarding policy remotely and flexibly without restarting the system or interrupting existing connections**.

The Linux bridge solves the simple problem of "an L2 switch that learns MAC addresses" with fixed behavior. Every configuration change (such as adding a VLAN or adjusting STP priority) goes through the `brctl` command from userspace, which changes kernel state directly through the ioctl mechanism. This approach is simple but inflexible, with no database and no intermediate management process.

OVS has to handle far more complex problems: an SDN controller can push **thousands of OpenFlow rules** in a single second. These rules must be validated, persisted, compiled into flow entries for the kernel, and must support rollback if an error occurs. Such complex processing, including state machines, JSON parsing, and TCP connection management, cannot run directly in the kernel. That is why a userspace daemon is needed.

In addition, configuration information (bridge, port, VLAN tag, QoS) requires **durable storage across reboots** and **atomic transactions**. For example, when adding two ports at once, or configuring a queue that references a QoS policy, all changes must succeed together or be rolled back together to keep the configuration consistent. The natural solution is a database with transactional semantics. However, OVS needs a database that is **lightweight, has a clear schema, and supports real-time change notification**. For this reason, Nicira designed OVSDB, and `ovsdb-server` is the standalone process that manages the database file.

Finally, the data plane (where actual packet forwarding happens) must live in the **kernel** to achieve high throughput (10+ Gbps). Bouncing every packet from the kernel network stack up to userspace for processing is too expensive: each context switch plus data copy costs about 2 microseconds, far too slow for millions of packets per second. For that reason OVS uses the kernel module `openvswitch.ko` to operate the fast path.

The result is three components, each responsible for one specialized task, communicating with one another through clearly defined interfaces.

## 9.1.1 Overview of the three components and their interactions

The four-layer mental model is an extremely effective tool for reasoning about OVS architecture during troubleshooting:

```
Layer 1, Config plane:    OVSDB (ovsdb-server)
                          "What should exist"
Layer 2, Control plane:   OpenFlow (ovs-vswitchd)
                          "How a packet should be processed"
Layer 3, Datapath cache:  Kernel module openvswitch.ko (or userspace netdev)
                          "What the fast path has already learned"
Layer 4, Wire:            Physical NIC or virtual interface
                          "What actually leaves the NIC"
```

> **Important.** Most OVS troubleshooting work is about **comparing these four layers to find the point of contradiction**. For example, if OVSDB shows the bridge has 3 ports, the OpenFlow table records only 2 ports, and the kernel datapath sees 3 ports, then there is a clear **state mismatch** between layer 1 and layer 2, usually a synchronization issue between `ovsdb-server` and `ovs-vswitchd`. Understanding the role of each layer narrows the search space quickly.

**Packet processing flow through OVS (using the kernel datapath)**:

```
┌─────────────────────────────────────────────────────────────────┐
│                     Userspace                                    │
│                                                                   │
│  ┌────────────────┐       ┌──────────────────────────┐          │
│  │ ovsdb-server   │◄─────►│ ovs-vswitchd             │          │
│  │ (config DB)    │ JSON-  │  - OpenFlow pipeline    │          │
│  │ /etc/openvswi- │  RPC  │  - classifier (TSS)      │          │
│  │ tch/conf.db    │       │  - handler threads        │          │
│  └────────────────┘       │  - revalidator threads   │          │
│                            └────────┬─────────────────┘          │
│                                     │                             │
└─────────────────────────────────────┼─────────────────────────────┘
                             Netlink genl
                          (ovs_datapath, ovs_flow,
                             ovs_vport families)
                                      │
┌─────────────────────────────────────▼─────────────────────────────┐
│                     Kernel space                                   │
│                                                                     │
│  ┌──────────────────────────────────────────────┐                 │
│  │  openvswitch.ko                               │                 │
│  │                                                │                 │
│  │   Packet arrives ─► Megaflow cache lookup     │                 │
│  │                    │                           │                 │
│  │                    ├── hit  ─► Execute actions │                 │
│  │                    │                           │                 │
│  │                    └── miss ─► Upcall to user  │                 │
│  │                                    │           │                 │
│  │                                    ▼           │                 │
│  │                              ovs-vswitchd      │                 │
│  │                                                │                 │
│  └──────────────────────────────────────────────┘                 │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

These three components do not connect in a simple linear chain. `ovs-vswitchd` plays the central role: it connects to `ovsdb-server` over a Unix socket using JSON-RPC (to read configuration and receive change notifications), and it connects to the kernel module over Netlink generic families (to install flows and receive upcalls on cache miss). Note that `ovsdb-server` never talks directly to the kernel, and the kernel never reads data from OVSDB.

## 9.1.2 ovs-vswitchd: the main userspace forwarding process

`ovs-vswitchd` is the most important process. It concentrates almost all of OVS's processing logic. It is a single binary, running as a single process but using multi-threading to process parallel workloads.

The **main loop** of `ovs-vswitchd` is based on epoll and a poll timer. On each iteration it:

1. Checks the OVSDB connection for configuration changes.
2. Checks OpenFlow connections from the controller for requests such as `flow_mod` and `packet_out`.
3. Checks the Netlink socket for upcalls from the kernel datapath.
4. Checks the Unix socket `/var/run/openvswitch/ovs-vswitchd.ctl` for control commands from `ovs-appctl`.
5. Performs periodic tasks (such as bond load balancing, sending STP BPDU packets).

While the main thread handles most events, several specialized tasks are split off into **dedicated thread pools**:

- **Handler threads**: process upcall packets from the kernel on cache miss. On each upcall the handler parses the packet, walks the OpenFlow pipeline, compiles the result into a megaflow entry, and pushes it to the kernel over Netlink.
- **Revalidator threads**: periodically (every 500 ms by default) scan the entire megaflow cache in the kernel to check the validity of each entry (for example, whether the flow rule has changed or whether the entry has exceeded its idle timeout), and delete stale entries.
- **OpenFlow worker threads**: handle OpenFlow connections from the controller.

The command `ovs-appctl memory/show` lets us inspect memory usage of each subsystem such as the classifier (flow table), handlers, and revalidators. This command is extremely useful when diagnosing memory growth.

```bash
[host]# ovs-appctl -t ovs-vswitchd memory/show
handlers:4 ports:5 revalidators:4 rules:127 ...
```

Reading the counters from `ovs-appctl coverage/show` helps detect anomalies. For example, if `cache-misses` spikes while `flow-lookup` drops, it means traffic is producing too many distinct microflows, defeating the cache.

## 9.1.3 ovsdb-server: a lightweight configuration database

`ovsdb-server` is the lightest component in terms of resource usage. Its only job is to serve a single OVSDB database file over JSON-RPC.

The OVS database file is at `/etc/openvswitch/conf.db`, with a schema that defines important tables such as `Bridge`, `Port`, `Interface`, `Controller`, and so on.

Components talk to `ovsdb-server` over Unix sockets or TCP and SSL. Two main clients interact with it:

- **`ovs-vsctl`**: the command-line tool for administrators. Each command is treated as a single transaction, ensuring atomicity (all or nothing).
- **`ovs-vswitchd`**: connects to `ovsdb-server` at startup and registers a monitor on the entire database. Any change in the database is pushed to `ovs-vswitchd` in real time without continuous polling.

A key point to remember: OVSDB stores only the **configuration intent** (for example, the user wants to create bridge X). It does not store **runtime operational state** (for example, how much traffic the bridge is currently passing). Operational statistics are computed on demand by `ovs-vswitchd` and exposed through special columns, but they are not persisted to disk.

## 9.1.4 openvswitch.ko: the data plane in the kernel (fast path)

The kernel module `openvswitch.ko` is where high-speed packet forwarding happens (the fast path). When a packet arrives at a NIC, the kernel hands it to this module.

`openvswitch.ko` manages two important data structures in kernel memory:

**1. Flow table in the kernel.** This is in fact a megaflow cache. Each entry contains match keys with masks, the list of actions to execute, and statistic counters.

**2. Port table.** Maps OpenFlow port numbers to the corresponding NIC pointers in the system.

When a packet arrives that does not match any entry in the cache (cache miss), the kernel does not drop the packet. Instead, it performs an **upcall to userspace**. The upcall message contains the original packet plus context. `ovs-vswitchd` receives it, walks the OpenFlow pipeline to determine the appropriate action, compiles it into a new megaflow entry, and **pushes it back to the kernel**. Subsequent similar packets then match directly in the kernel, making forwarding very fast.

> **Further explanation.** The upcall mechanism is what lets OVS achieve both goals at once: speed (through the kernel fast path) and flexibility (through complex logic in userspace). Empirical studies show that the cache hit rate typically exceeds 99 percent, meaning only the first packet of a new flow has to go up to userspace; that cost is amortized across thousands of subsequent packets processed in the kernel.

The kernel datapath generally has two types:
- **`datapath_type=system`** (default): uses the kernel module, throughput typically 5 to 10 Gbps depending on hardware.
- **`datapath_type=netdev`**: uses a userspace data plane (often combined with DPDK). Packets bypass the kernel and go straight to userspace, allowing very high throughput (50+ Gbps) but requiring more dedicated CPU resources.

## 9.1.5 Netlink generic family: the bridge between userspace and the kernel

Netlink is the special communication mechanism in Linux for exchanging information between userspace and the kernel. OVS uses an advanced variant called **Generic Netlink**.

OVS registers three main Netlink families:
- **`ovs_datapath`**: manages the lifecycle of datapaths (creating and removing bridges at the kernel layer).
- **`ovs_flow`**: manages flow entries in the datapath. Upcall packets also travel through this family.
- **`ovs_vport`**: manages the virtual ports (interfaces) attached to a bridge.

In operation, if you see error messages related to Netlink, they often indicate a **version mismatch** between the `ovs-vswitchd` process and the `openvswitch.ko` kernel module (for example, the OVS userspace was upgraded but the corresponding kernel module was not).

## Key takeaways

1. **Why split into three?** Because each layer has its own characteristics: configuration needs durability (OVSDB), complex control logic needs the flexibility of userspace (ovs-vswitchd), and data forwarding needs the maximum speed of the kernel.
2. **Four-layer mental model.** Always compare Configuration (OVSDB) to Control (OpenFlow) to Cache (Datapath) to Wire. Every operational defect lies in a contradiction between these layers.
3. **Upcall mechanism.** This is the "soul" of OVS, balancing functionality and performance.
4. **Diagnostic toolset.** Memorize the commands `ovs-vsctl`, `ovs-ofctl`, `ovs-dpctl`, and `ovs-appctl`. They are the "stethoscopes" that let an engineer locate the problem.

## 9.1.X Internals of ofproto-dpif and connmgr (Phase H addition)

This section provides details on two important subsystems: **ofproto-dpif** (translates the OpenFlow pipeline into datapath actions) and **connmgr** (connection management).

### 9.1.X.1 ofproto-dpif architecture inside ovs-vswitchd

The `ovs-vswitchd` process is divided into 5 main layers (top to bottom):

```
┌─────────────────────────────────────┐
│ bridge.c                            │  ← Bridge lifecycle and OVSDB sync
├─────────────────────────────────────┤
│ ofproto/ofproto.c                   │  ← OpenFlow protocol handling
├─────────────────────────────────────┤
│ ofproto/ofproto-dpif.c              │  ← datapath-aware ofproto
│ ofproto/ofproto-dpif-xlate.c        │  ← Action translation
│ ofproto/ofproto-dpif-upcall.c       │  ← Upcall handling and revalidator
├─────────────────────────────────────┤
│ ofproto/connmgr.c                   │  ← Connection management to OF controller
│ lib/classifier.c                    │  ← Classifier (TSS plus trie)
├─────────────────────────────────────┤
│ lib/dpif.c (plus dpif-netlink.c,    │  ← Datapath interface abstraction
│             dpif-netdev.c)          │
└─────────────────────────────────────┘
```

### 9.1.X.2 Decoding `ovs-appctl dpif/show` output

This command checks the operational state of the datapath:

```shell
$ sudo ovs-appctl dpif/show
system@ovs-system:
  bridge br-int/0:
    flows: (current 1247) (avg 1250) (max 1350) (limit 200000)
    br-int 65534/<local>: (internal)
    tap-vm-web 1/1: (internal)
    ovn-huyvl-0 3/3: (geneve: remote_ip=10.200.0.12, csum=true, key=flow)
```

Note the difference between the OpenFlow port number (persistent) and the datapath port number (allocated dynamically by the kernel).

### 9.1.X.3 Role of the Connection Manager (connmgr)

connmgr is responsible for establishing TCP and SSL connections to the controller, managing role election (MASTER/SLAVE), and distributing asynchronous messages such as `PACKET_IN`.

### 9.1.X.4 Combined multi-thread model

- **Main thread:** synchronizes with OVSDB and runs the ofproto cycle.
- **Handler threads:** process upcalls from the kernel.
- **Revalidator threads:** clean up and refresh flow entries in the kernel.
- **PMD threads:** used only with DPDK to poll NIC queues.

## 9.1.Y The xlate engine in ofproto-dpif (Phase I addition)

The **xlate engine** translates packets and OpenFlow tables into datapath action lists. This is the most important processing path of OVS in userspace.

### 9.1.Y.1 When is xlate triggered?

1. **On a kernel upcall:** when the kernel cannot find a matching flow for the first packet.
2. **On periodic revalidator scans:** to check whether rules have changed.
3. **When running `ofproto/trace`:** for manually diagnosing the path of a packet.

### 9.1.Y.2 How `xlate_actions()` works

This function takes packet information as input and returns a list of datapath actions together with a megaflow mask. While walking the OpenFlow tables, xlate marks which fields have been examined, producing the most optimal megaflow mask.

### 9.1.Y.3 Protection mechanisms in xlate

To avoid configuration mistakes (such as creating infinite loops), OVS imposes hard limits:
- **Maximum recursion depth:** 64 levels (for the `resubmit` action).
- **Maximum total resubmit count:** 4096.
If either limit is exceeded, processing is aborted to keep the system from hanging.

### 9.1.Y.4 Why separate kernel and userspace?

The kernel datapath is essentially "naive": it does not know what OpenFlow is. It only executes basic actions pushed down from userspace. This separation provides several benefits:
- The kernel stays simple and fast.
- Userspace processing logic can be upgraded without touching the kernel.
- The megaflow mechanism minimizes the number of round trips to userspace.

## 9.1.Z ovs-vswitchd, the Open vSwitch userspace daemon (in-depth treatment)

> **Scope:** in-depth treatment (added 2026-04-26) for the cornerstone keyword `ovs-vswitchd`. This is the only userspace process that integrates all of OVS's forwarding logic: bridge lifecycle, OpenFlow protocol, flow classifier, OFPT_PACKET_IN/PACKET_OUT, megaflow translation, dpif client, OVSDB IDL subscriber. Tightly cross-linked with §9.1.2 (brief introduction), §9.1.X (the 5-layer ofproto-dpif), §9.1.X.4 (multi-thread model), §9.1.Y (xlate engine), §9.1.AB (dpif), §9.2 (the kernel datapath megaflow as the internals of vswitchd's datapath provider), §9.4 (CLI playbook), §10.1 (the ovsdb-server cluster that vswitchd subscribes to). Curriculum baseline OVS v2.17.9.

**Concept.** `ovs-vswitchd` is a single-process, multi-thread C daemon, built from `vswitchd/ovs-vswitchd.c` together with many library files in the source tree (`vswitchd/bridge.c`, `ofproto/*`, `lib/*`). It is **a single process per host** responsible for: (a) reading configuration from `ovsdb-server` over the OVSDB JSON-RPC protocol, (b) creating and maintaining the `ofproto` structs representing each bridge, (c) accepting OpenFlow connections from controllers (tcp:, ssl:, unix:) or running in "fail-mode standalone" when there is no controller, (d) walking the OpenFlow pipeline to translate the first packet of each flow into a set of datapath actions, (e) installing megaflows in the datapath through the `dpif` interface, (f) observing and reaping stale megaflows through the revalidator thread. `ovs-vswitchd` is NOT a kernel module, NOT a database, NOT a one-shot CLI tool. It is a long-running background daemon whose lifecycle is tied to the host. Without `ovs-vswitchd`, the kernel module `openvswitch.ko` is just code in the kernel that receives no rules, and shortly after `flow-restore-wait` expires, all leftover megaflows are purged and traffic stops completely.

**History and context.** The predecessor of `ovs-vswitchd` was the early `vswitchd` program (2009 to 2010) in the Nicira source tree, which was a single C application that also embedded configuration storage. In 2010 Nicira split configuration storage into a separate `ovsdb-server`, while the control and forwarding logic retained the name `ovs-vswitchd`. This is the starting point of the three-component architecture (`ovs-vswitchd`, `ovsdb-server`, `openvswitch.ko`) that survives today. In 2012 `ovs-vswitchd` was refactored to use dpif (see §9.1.AB) instead of talking to Netlink directly, opening the way to userspace datapaths. In 2014 (OVS 2.0) `ovs-vswitchd` moved to a multi-thread model: handler threads and revalidator threads were split out from the main thread, and the megaflow architecture was introduced. This was the largest performance leap in OVS history. In 2015 the NSDI paper "The Design and Implementation of Open vSwitch" (Pfaff, Pettit, et al.) formally published the `ovs-vswitchd` architecture and placed it in the reference diagram of the academic literature. From 2016 onward, the 2.x releases added tunables: `n-handler-threads`, `n-revalidator-threads`, `flow-restore-wait`, `vlan-limit`, `dpdk-init`. After Open vSwitch moved to the Linux Foundation (2016), `ovs-vswitchd` continued to be the integration point for conntrack, recirc, NSH, and OVN. By v2.17.9 (the current LTS in the 2.17 series), the daemon architecture has been stable for many years, with tunables changing very little.

**Position in the architecture.** In the three-component OVS diagram, `ovs-vswitchd` sits in the middle, the only bridge between the configuration source (OVSDB) and the actual data plane (kernel datapath or userspace netdev):

```
┌───────────────────┐  JSON-RPC over Unix socket ┌────────────────────┐
│ ovs-vsctl         │ ─────────────────────────► │ ovsdb-server       │
│ ovn-controller    │                            │ /etc/openvswitch/  │
│ neutron-ovs-agent │                            │   conf.db          │
└───────────────────┘                            └────────┬───────────┘
                                                          │ OVSDB IDL monitor (push)
                                                          ▼
       ┌─────────────────────────────────────────────────────────────────┐
       │                       ovs-vswitchd (single process)             │
       │ ┌───────────────────────────────────────────────────────────┐   │
       │ │ Main thread: bridge_run → ofproto_run → unixctl → idl_run │   │
       │ │   (poll_block on the epoll fd of every event source)      │   │
       │ ├───────────────────────────────────────────────────────────┤   │
       │ │ N handler threads → udpif_upcall_handler()                │   │
       │ │ M revalidator threads → udpif_revalidator()               │   │
       │ │ (with DPDK) PMD threads polling RX queues                 │   │
       │ ├───────────────────────────────────────────────────────────┤   │
       │ │ Subsystems: ofproto, ofproto-dpif, classifier, connmgr,   │   │
       │ │   netdev, dpif (vtable), tnl_*, ct (userspace conntrack)  │   │
       │ └───────────────────────────────────────────────────────────┘   │
       └────────┬────────────────────────────────────────┬────────────────┘
                │ OpenFlow tcp:/ssl:/unix:               │ dpif_* API
                ▼                                        ▼
        ┌───────────────────┐                ┌─────────────────────────┐
        │ OF controller     │                │ Datapath provider       │
        │ (ovn-controller,  │                │  - dpif-netlink → ko    │
        │  Faucet, Ryu...)  │                │  - dpif-netdev → DPDK/  │
        └───────────────────┘                │    userspace netdev     │
                                             └─────────────────────────┘
```

Scope: per-host, one `ovs-vswitchd` instance manages all bridges on the host (you do not run multiple `ovs-vswitchd` instances at the same time for the same set of bridges). Lifecycle: long-running daemon. The init script (`/etc/init.d/openvswitch-switch` or `openvswitch-switch.service`) starts it after `ovsdb-server` and after loading the `openvswitch.ko` module. Restarting `ovs-vswitchd` typically causes a few seconds of data plane disruption unless `flow-restore-wait` is enabled.

**Role.** Six core roles: (a) **OVSDB IDL subscriber**, registers monitors on the tables `Open_vSwitch`, `Bridge`, `Port`, `Interface`, `Controller`, `Manager`, `Flow_Table`, `Mirror`, and so on, ensuring that when a user runs `ovs-vsctl add-br br0`, `ovs-vswitchd` sees the change almost immediately and reconfigures itself. (b) **OpenFlow speaker**, runs connmgr to listen for and connect to controllers; speaks OpenFlow 1.0 through 1.5 depending on `OFPT_HELLO` negotiation. (c) **Classifier owner**, each bridge has one TSS (Tuple Space Search) classifier holding OF rules, queried on upcall and on trace. (d) **Datapath programmer**, when a flow miss occurs, vswitchd translates the OF pipeline into a datapath action set and calls `dpif_flow_put()` to install the megaflow. (e) **Revalidator and cleaner**, periodically scans megaflows in the datapath and removes entries that are no longer valid. (f) **Telemetry surface**, exposes the Unix control socket `/var/run/openvswitch/ovs-vswitchd.<pid>.ctl` so that `ovs-appctl` can query `coverage/show`, `memory/show`, `dpif/show`, `vlog/set`, `bond/show`, `bridge/dump-flows`, and so on. Every real-world operational question goes through this socket.

**Why it exists.** The design intent answered three questions at once. Question 1, "How complex is the OpenFlow pipeline; can it be embedded in the kernel?" Answer: very complex (tens of thousands of lines of C, multi-table, group, meter, conntrack, NAT, recirc, sample), the kernel should not carry it. Question 2, "If the logic lives in userspace, how do we still achieve data plane speed?" Answer: cache lookup in the kernel (megaflow) sends only the first packet to userspace; the rest take the fast path. Question 3, "How do we cleanly separate configuration so that hot-reload does not require restarting the whole stack?" Answer: OVSDB is split out, `ovs-vswitchd` subscribes and reacts. Before OVS, L2 switch implementations on Linux such as `bridge-utils` buried configuration in sysfs and ioctl, with no "OpenFlow brain" and no controller concept. `ovs-vswitchd` was created to unify all three answers into a single daemon, small enough to be understandable, large enough to contain the full data plane pluggability needed for SDN.

**Problems it solves.** Concretely `ovs-vswitchd` solves four classes of problems: (1) **Separating configuration from forwarding**, OVSDB holds the "intent", `ovs-vswitchd` enacts that intent; admins can change the DB without understanding forwarding details. (2) **OpenFlow and non-OpenFlow coexisting**, a bridge can have a controller (mode=secure) or fall back to MAC learning (mode=standalone), `ovs-vswitchd` handles both code paths. (3) **Datapath-agnostic**, through dpif (see §9.1.AB) the same `ovs-vswitchd` runs on the Linux kernel module, on a userspace netdev, or on DPDK PMD. (4) **Operability**, every operational signal of OVS is exposed through the `ovs-appctl` socket, so `ovs-vswitchd` is the "single point of observability" for the whole OVS stack.

**Importance.** This is an absolutely core concept, on par with `megaflow`, `ct_state`, `recirc_id`, `dpif`, and `ofproto-dpif xlate`. An engineer who does not clearly understand the role of `ovs-vswitchd` will not be able to distinguish: (i) which rules are stored in OVSDB versus pushed by the controller at runtime, (ii) why restarting `ovs-vswitchd` loses megaflows for a few seconds, (iii) why `ovs-vswitchd.log` matters more than `dmesg` when debugging forwarding. In the curriculum's 320+ keyword structure, `ovs-vswitchd` is the "vertical axis" of Block IX (OVS internals); every other keyword such as `bridge_run`, `ofproto_run`, `udpif_upcall_handler`, `coverage/show`, `memory/show`, `flow-restore-wait` revolves around it.

**How it works.** Detailed startup sequence and lifecycle of `ovs-vswitchd`:

```
1. The init script starts ovsdb-server first, loading the openvswitch.ko module (if type=system).
2. exec ovs-vswitchd (binary from /usr/sbin/ovs-vswitchd) with argv:
     unix:/var/run/openvswitch/db.sock
     --pidfile=/var/run/openvswitch/ovs-vswitchd.pid
     --log-file=/var/log/openvswitch/ovs-vswitchd.log
     --detach --monitor --no-chdir
3. main() (vswitchd/ovs-vswitchd.c) does:
   a. set_program_name("ovs-vswitchd"), ovsthread_id_init.
   b. dns_resolve_init(true), parse_options(argc, argv), returning remote = unix:/var/run/.../db.sock.
   c. fork through daemonize_start(true); the parent exits, the child becomes a daemon, stderr is redirected to the log file.
   d. unixctl_server_create(NULL, &unixctl_server) opens /var/run/.../ovs-vswitchd.<pid>.ctl
      so ovs-appctl can send commands to it.
   e. bridge_init(remote) creates the IDL handle through ovsdb_idl_create(),
      registers the tables to monitor, and initializes subsystems (LACP, bond, CFM, BFD, STP, LLDP, RSTP).
   f. Main loop: while (!exiting) {
        memory_run();        # refresh heap profile so memory/show has data
        bridge_run();        # read IDL changes, reconfigure, call ofproto_run
        unixctl_server_run(); # serve ovs-appctl
        netdev_run();        # poll netdev statistics and link state
        memory_wait(); bridge_wait(); unixctl_server_wait(); netdev_wait();
        poll_block();        # block on epoll until an event occurs
        if (should_service_stop()) exiting = true;
      }
   g. Shutdown: bridge_exit(), unixctl_server_destroy(), ovsrcu_exit(), dns_resolve_destroy().
4. Auxiliary threads:
   - Handler thread (n-handler-threads, default = max(2, n_cores) per the formula in
     ofproto/ofproto-dpif-upcall.c::udpif_set_threads): each thread runs udpif_upcall_handler()
     in a loop calling dpif_recv(), then xlate_actions(), then dpif_flow_put().
   - Revalidator thread (n-revalidator-threads, default approximately n_handlers/4 + 1): each thread runs
     udpif_revalidator() in a loop dumping megaflows, re-translating, and keeping or removing them.
   - When datapath_type=netdev with DPDK: PMD threads pinned per-CPU, polling RX queues.
```

What does bridge_run do specifically on each iteration? It reads OVSDB changes through `ovsdb_idl_run(idl)`, checks whether `idl_seqno` has changed and if so calls `bridge_reconfigure()` to add or remove bridges, add or remove ports, and update tunables; then it iterates all bridges, calling `ofproto_run(br->ofproto)` on each to handle OpenFlow messages from the controller (through connmgr), expire services, flush queued packet_out, and update rule statistics. The main lock in vswitchd is `ofproto_mutex`, which protects the flow table; some read paths use RCU (`ovsrcu`).

**How an engineer masters operating it.** A skilled engineer reflexively performs five operations. Operation 1, check daemon state: `systemctl status openvswitch-switch`, then compare `ovs-appctl version` with `dpkg -l openvswitch-switch` to confirm that the binary on disk and the running process are the same version (after `apt upgrade` without a restart they will diverge). Operation 2, read the thread map: `top -H -p $(pgrep ovs-vswitchd)` to see all of main, handler, revalidator, URCU, and IPsec; if the `revalidator` thread is at 100 percent CPU, flow churn is high and `coverage/show` should be inspected. Operation 3, read `coverage/show` and watch counters: `bridge_reconfigure`, `flow_extract`, `miss_queue_overflow`, `upcall_flow_limit_hit`, `revalidator_purge`, `rev_revalidate`, `txn_unchanged`. Operation 4, read `memory/show` to know where most RAM is spent (handlers, revalidators, ports, rules, ofconns); an unusually large `rules` value implies the controller is pushing rules without bound. Operation 5, know the path to the log: `/var/log/openvswitch/ovs-vswitchd.log`, understand the format `<timestamp>|<seq>|<module>|<level>|<message>`, use `ovs-appctl vlog/set <module>:file:dbg` to enable debug for a single module (`ofproto`, `bridge`, `dpif`, `connmgr`, `udpif`) instead of turning everything on.

**Classification.** The states and operating modes of `ovs-vswitchd` can be classified: (a) **By datapath mode**, `system` (kernel ko), `netdev` (userspace), `dummy` (test). (b) **By connection mode**, with controller (in-band/out-of-band), without controller (standalone, learning MAC). (c) **By lifecycle phase**, starting (reading IDL, building initial bridges), steady (main loop), reconfiguring (between an OVSDB delta read and installing new flows), shutting-down (drain plus bridge_exit). (d) **By tuning configuration**, vanilla (no `n-handler-threads`/`n-revalidator-threads` set), DPDK-tuned (set `dpdk-init=true`, pin PMD core), HA (enable `flow-restore-wait` for restart without dropping traffic).

**Usage workflow.** The "bring OVS up from zero to first bridged packet" workflow:

```
Step 1: confirm both daemons are present.
        $ systemctl status openvswitch-switch
        $ ps -ef | grep -E "ovsdb-server|ovs-vswitchd"
        Expect 2 processes, with ovs-vswitchd having multiple child threads.
```

Best practice: do NOT run `ovs-vswitchd` manually in production; always let the init script manage its lifecycle, and let `--monitor` respawn it on crash. Anti-pattern: changing `Open_vSwitch.other_config:n-handler-threads` and reloading with SIGHUP, because this option is read only once at `bridge_init`; to change it you must restart the daemon (or set it before starting).

**When something goes wrong.** `ovs-vswitchd` is a mandatory checkpoint in these situations: (a) Bridge created with `ovs-vsctl` but packets do not forward; first check `pgrep ovs-vswitchd`; if the daemon has died, traffic still works for tens of seconds thanks to leftover megaflows in the kernel, then disappears. (b) Daemon is running but pushed OF rules have no effect; check `coverage/show | grep ofctl_rate_limit_drop` to see whether connmgr is rate-limiting. (c) `ovs-vswitchd.log` spams `revalidator|WARN|flow limit hit, dropping flows`, meaning megaflows have hit `flow-limit` (default 200000); tuning is needed. (d) Main thread at 100 percent CPU, often `bridge_reconfigure` holding the lock too long because of a large OVSDB rule dump; switch to ovs-vsctl-bench or reduce rule churn. (e) Memory grows without bound; record `memory/show` periodically; if `rules` grows without limit, the controller is pushing without removing. Restarting the daemon with `systemctl restart openvswitch-switch` WITHOUT enabling `flow-restore-wait` creates a data plane gap of about 1 to 3 seconds.

**Tightly related to.** `ovs-vswitchd` is interleaved with almost every keyword in Block IX and parts of Block X: (a) `ovsdb-server` (configuration source, vswitchd IDL subscriber, see §9.1.3 and §10.1). (b) `openvswitch.ko` (downstream partner through the dpif system provider, see §9.1.4 and §9.2). (c) `dpif` (the abstraction layer in between, see §9.1.AB). (d) `ofproto-dpif xlate` (the most-called path during upcall handling, see §9.1.Y and §9.32). (e) `udpif_upcall_handler` and `udpif_revalidator` (worker threads inside vswitchd). (f) `connmgr` (subsystem inside vswitchd handling OF connections, see §9.1.X.3). (g) `ovs-appctl` (CLI client talking through the control socket, see §9.4 and §9.11). (h) `flow-restore-wait` (the Open_vSwitch.other_config column affecting restart behavior). (i) `coverage/show`, `memory/show` (telemetry exposed through the control socket).

**Source code reference.** Per Rule 14 we use function-name anchors (line numbers drift across versions). Baseline OVS v2.17.9:

- `vswitchd/ovs-vswitchd.c`, entry point. The function `main()` parses argv, daemonizes, calls `bridge_init()`, then enters the main loop. The function `parse_options()` handles long options such as `--mlockall`, `--pidfile`, `--log-file`. The block comment at the top of the file describes "Main program for ovs-vswitchd".
- `vswitchd/bridge.c`, bridge lifecycle and OVSDB IDL. The function `bridge_init(remote)` opens the IDL connection. The function `bridge_run()` runs each iteration of the main loop, reads IDL, calls `bridge_reconfigure()` on configuration change, then iterates `ofproto_run()` for each bridge. The function `bridge_reconfigure()` orchestrates add or remove bridge, add or remove port, sets datapath_id, mirror, MAC table, NetFlow, sFlow, IPFIX. The function `bridge_exit()` cleans up on shutdown.
- `ofproto/ofproto.c`, the OF protocol layer. The function `ofproto_create()` creates the ofproto struct for a bridge. The function `ofproto_run()` runs each iteration, expires rules, runs connmgr, flushes the packet_out queue. The function `ofproto_destroy()` cleans up.
- `ofproto/ofproto-dpif.c`, the dpif-aware ofproto. The class `ofproto_dpif_class` is registered through `ofproto_class_register()`. The function `construct()` creates the ofproto_dpif struct when a new bridge is created, calling `dpif_open()` from §9.1.AB.
- `ofproto/ofproto-dpif-upcall.c`, handler and revalidator. The function `udpif_create()` is called from ofproto-dpif construct. The function `udpif_set_threads(udpif, n_handlers, n_revalidators)` sets the thread count (default computed from `count_cpu_cores()`). The function `udpif_start_threads()` calls `pthread_create` for the handlers (`udpif_upcall_handler`) and revalidators (`udpif_revalidator`).
- `ofproto/connmgr.c`, the OpenFlow connection manager. The function `connmgr_create()` creates the struct, the function `connmgr_run()` accepts TCP and SSL connections from controllers.
- `lib/unixctl.c`, the control socket framework for `ovs-appctl`. The function `unixctl_server_create()` opens the Unix socket. The function `unixctl_command_register()` registers command names (`coverage/show`, `memory/show`, and so on).

**Real incident scenario.** An OpenStack deployment (4 controllers plus 25 compute nodes, each compute about 80 VMs, OVN as the Neutron driver) recorded the following: at peak traffic, `ovs-vswitchd` on some compute nodes consumed 100 percent CPU on the main thread, causing VM-to-VM ping to stutter, and ovn-controller logged "ovsdb client transaction took too long". Investigation: `top -H -p $(pgrep ovs-vswitchd)` showed the main thread saturated and handler and revalidator threads idle. `ovs-appctl coverage/show` showed `bridge_reconfigure` running 4 to 5 times per second (normal is less than once per minute). Root cause: ovn-controller was pushing flow updates very frequently (because of ARP-update churn from VMs in a crash loop); each update triggered an OVSDB transaction, each transaction caused an IDL update to `ovs-vswitchd`, and each update called `bridge_reconfigure` to re-validate the configuration. Mitigation: (i) identify the crash-looping VMs and stop them; (ii) tune `Open_vSwitch.other_config:vlan-limit`; (iii) in a later release enable an upstream-proposed `ofproto_pipeline_pause` patch that lets ovn-controller batch updates. Lesson: the symptom "ovs-vswitchd at 100 percent CPU while handler and revalidator are idle" is a classic sign that the main thread is in a reconfigure storm; the first tool is `top -H`, then `coverage/show` looking for the `bridge_reconfigure` count; if it is high, the root cause lies on the controller or OVSDB writer side, not the data plane.

**Common errors and diagnostic signals.**

- *Symptom* `ovs-vsctl add-br` succeeds but the port does not forward. *Cause* `ovs-vswitchd` has crashed or is not started. *Diagnosis* `pgrep ovs-vswitchd` returns nothing; `journalctl -u openvswitch-switch` shows "ovs-vswitchd|ERR|...". *Fix* `systemctl restart openvswitch-switch`; if the problem repeats, read the core dump log.
- *Symptom* Main thread at 100 percent CPU, handler and revalidator idle. *Cause* Reconfigure storm because of a controller or ovs-vsctl writer pushing too many changes. *Diagnosis* `top -H -p $(pgrep ovs-vswitchd)` to identify the thread; `ovs-appctl coverage/show | grep bridge_reconfigure`. *Fix* Reduce the upstream writer's frequency; in the worst case restart ovs-vswitchd to break the cycle.
- *Symptom* `ovs-vswitchd.log` spams `revalidator|WARN|flow limit hit`. *Cause* Megaflows exceed the default `flow-limit` of 200000. *Diagnosis* `ovs-appctl dpif/show -s` reads `flows: (current ...)`, `memory/show` reads `rules`. *Fix* `ovs-vsctl set Open_vSwitch . other_config:flow-limit=400000`, or check whether the controller is installing rules without timeouts.
- *Symptom* Restarting `ovs-vswitchd` causes a few seconds of traffic loss. *Cause* `flow-restore-wait` is not enabled; kernel megaflows are purged when the new userspace starts and the revalidator scans. *Diagnosis* `ovs-vsctl get Open_vSwitch . other_config` does not show `flow-restore-wait=true`. *Fix* Enable it before the restart, remove it after the restart once `ovs-appctl ofproto/list` shows all bridges.
- *Symptom* `ovs-vswitchd` memory grows over time, OOM after weeks. *Cause* The controller pushes rules without removing them, or a rare bug in revalidator (seen in 2.13). *Diagnosis* Track `ovs-appctl memory/show` periodically, log the `rules` and `udpif keys` columns. *Fix* Upgrade to a newer LTS (2.17 LTS), audit controller logic.

**Comparison with other systems.** Other systems solve "userspace daemon orchestrating kernel data plane" differently:

- **Linux bridge plus brctl**: no equivalent userspace daemon; all configuration lives in sysfs and ioctl, with no OpenFlow brain. `ovs-vswitchd` completely replaces this role when using OVS.
- **Cumulus Linux switchd**: equivalent at the level of "userspace daemon programming kernel forwarding", but switchd talks to a SwitchAPI driver for hardware ASICs instead of a software kernel module; the split control and data model is similar.
- **VPP `vpp_main` process** (FD.io): VPP is also a userspace daemon, organized as a vector packet processing graph; it differs from `ovs-vswitchd` in that VPP runs the data plane in userspace itself, with no "kernel fast path plus userspace control" split like the OVS default.
- **Cilium agent**: a userspace daemon programming an eBPF data plane, equivalent in role as a "userspace daemon programming the data plane", but using eBPF instead of OpenFlow plus a kernel module.
- **NSX-T netcpa plus nestdb-ops**: VMware NSX has a daemon pair equivalent to `ovs-vswitchd` plus `ovsdb-server`; given the Nicira to VMware lineage, the architecture is very similar, with only different process names and protocols.
- **Cisco IOS XR `BGP-RIB to FIB downloader`**: not really equivalent; IOS XR integrates control and forwarding in proprietary firmware, with no clear "userspace daemon to kernel module through a stable interface" split.

## 9.1.AB dpif, the abstraction layer between userspace ofproto-dpif and the datapath provider

> **Scope:** in-depth treatment (added 2026-04-26) for the keyword `dpif` (Datapath Interface). This keyword is a cornerstone of OVS internals because it is the only place in the code base where the userspace forwarding logic is decoupled from a concrete datapath implementation (the kernel module openvswitch.ko, a userspace netdev provider, or future providers). Tightly cross-linked with §9.1.X (the 5-layer ofproto-dpif), §9.1.X.2 (`dpif/show` output), §9.2 (the kernel datapath megaflow is the internals of a dpif provider), §9.32 §9.32.4 (a previous compact architecture sketch). Curriculum baseline OVS v2.17.9.

**Concept.** `dpif` stands for "DataPath InterFace", an abstract C interface defined in `lib/dpif.h` and `lib/dpif-provider.h`. dpif represents "a switch as seen from userspace": a set of ports, a flow table with install/remove/dump operations, an upcall channel to receive packets that did not match a flow, and an execute path to inject packets. dpif is NOT a kernel module, NOT a daemon, but **a set of function pointers and struct layouts** that concrete providers (`dpif-netlink`, `dpif-netdev`) must implement. Thanks to dpif, every upper layer in `ovs-vswitchd` (bridge.c, ofproto.c, ofproto-dpif.c, ofproto-dpif-xlate.c, ofproto-dpif-upcall.c) only has to "call dpif_flow_put and receive the result"; it does not have to know whether the flow ends up in the Linux kernel or in a thread polling a DPDK port. Type: userspace abstraction layer (vtable pattern), a C-language file, not a database table, not a binary, not a daemon.

**History and context.** dpif appeared during the first major OVS refactor (2010 to 2012) when Nicira realized that the model "ovs-vswitchd knows directly how to talk to the kernel module" (through the early `lib/dpif-linux.c`) was a blocker for porting OVS to new environments. In 2012 `dpif-netdev` appeared, allowing a fully userspace datapath; in 2014, the NSDI 2015 paper (Pfaff et al.) formally established dpif as "the datapath interface" in the architecture diagram. In 2014 to 2015 dpif-netdev was extended to support DPDK PMD; later, experimental providers were added (windows, dummy for tests). Each time OVS adds a new datapath backend, ofproto does not have to change; one only needs to implement enough of the function pointers in `dpif_class`. This is one of the longest-lived high-value refactors in OVS, as the architecture has remained intact for more than a decade.

**Position in the architecture.** In the 5-layer diagram of `ovs-vswitchd` (already presented in §9.1.X.1), dpif sits at the **lowest userspace layer**, right at the boundary with the data plane:

```
bridge.c                          (layer 1, bridge lifecycle)
  └─ ofproto/ofproto.c            (layer 2, OF protocol semantics)
      └─ ofproto/ofproto-dpif.c   (layer 3, dpif-aware ofproto)
          ├─ ofproto-dpif-xlate.c (translate OF to odp actions)
          ├─ ofproto-dpif-upcall.c (receive upcalls from dpif)
          └─ ofproto-dpif-rid.c   (recirc id pool)
              ↓ only call through the dpif_* API
          ──────────────────────────
          lib/dpif.c               (layer 4, dispatch wrapper)
              ↓ dpif_class vtable
          ──────────────────────────
          lib/dpif-netlink.c       (layer 5a, kernel provider)
              ↓ Netlink generic
              kernel openvswitch.ko
          lib/dpif-netdev.c        (layer 5b, userspace provider)
              ↓ in-process function call
              userspace netdev / DPDK PMD thread pool
```

Scope: per-`ovs-vswitchd` process. Each process can open several dpif instances in parallel (one dpif per bridge, or multiple bridges sharing a dpif type depending on the provider). A host typically has one main dpif type (`system` for kernel, `netdev` for userspace) and does not randomly mix them on the same bridge.

**Role (rewritten in English by the 2026-04-28 source-citation cleanup).** Four core roles: (a) **Provider dispatch.** `lib/dpif.c` keeps a registry of `dpif_class` instances (via `dp_register_provider()`) and resolves a type name (`system`, `netdev`) to the corresponding class when `dpif_open()` is called. (b) **API uniformity.** Every function call from the layer above the datapath (install a flow, dump flows, add a port, receive an upcall, execute a packet) goes through one unified set of `dpif_*` symbols; `ofproto-dpif` never calls `dpif_netlink_*` or `dpif_netdev_*` directly. (c) **Capability negotiation.** `dpif` exposes a feature matrix via `dpif_class.cache_get_supported_levels`, the `dpif_supports_*` helpers, and the command `ovs-appctl dpif/show-dp-features`, so xlate knows whether the current provider supports a given action (ct, recirc, push_mpls, sample). (d) **Lifecycle and ownership.** `dpif` manages the datapath-instance life cycle: `dpif_create`, `dpif_open`, `dpif_close`, so `ofproto` does not have to handle cleanup itself.

**Why it exists.** The design intent is to separate "what to do" from "how to do it". Before dpif, the early OVS code (2009 to 2010) had hard-coded `linux_*` functions calling Netlink directly, so porting to FreeBSD, Hyper-V, or DPDK required changing thousands of lines in many places. dpif inverts the dependency: every caller above invokes through an interface, and every implementation below merely implements the interface. This is a textbook application of the Dependency Inversion Principle to systems C code. Demonstrated concrete benefits: (i) adding the DPDK datapath only required a single large `dpif-netdev.c` file with no edits to ofproto.c, (ii) adding a test backend (`dpif-dummy`) lets ofproto unit tests run without root or a kernel module, (iii) experimental AF_XDP support was added as a branch inside `dpif-netdev`.

**Problems it solves.** Concretely, dpif solves four classes of problems: (1) **Multi-platform portability**, OVS is not locked to the Linux kernel; the provider can be a kernel module, userspace polling, or a hardware accelerator gateway. (2) **Testability**, ofproto-dpif can be tested with `dpif-dummy` (in-memory) without root and without loading a kernel module. (3) **Capability divergence handling**, different providers support different features (for example, the kernel had ct() since 2.5, userspace netdev got ct() later, some tunnel types are kernel-only or userspace-only); xlate reads the capability matrix and decides whether to "emulate" or "delegate". (4) **Performance isolation**, when DPDK PMD threads cannot block on I/O, dpif-netdev provides an async path that does not block the main thread of ovs-vswitchd; isolating this through an interface means ofproto code does not have to handle the blocking-model differences itself.

**Importance.** This is an absolute core concept of OVS internals. Without understanding dpif, you cannot read the detailed flow of an upcall, cannot debug a kernel-versus-userspace mismatch, and cannot understand why the same OF rule behaves differently on two hosts (host A with `datapath_type=system`, host B with `datapath_type=netdev`). The level of importance is on par with `megaflow`, `ct_state`, `recirc_id`, and `ofproto-dpif xlate` in the OVS core group. In the curriculum's 320+ keyword structure, dpif is the "vertical axis" that ties together all of Block IX (OVS internals).

**How it works.** Per-bridge sequence from `ovs-vsctl add-br br0` to the first forwarded packet:

1. `bridge.c` reads OVSDB and sees `Bridge.datapath_type = "system"` (the default when nothing is set).
2. `ofproto.c` calls `ofproto_create()`, and ofproto-dpif chooses the dpif provider through the `dp_register_provider()` registry.
3. `lib/dpif.c::dpif_open("br0", "system", &dpif)` looks up `dpif_netlink_class` and calls `dpif_netlink_class.open(...)`. The kernel module receives the Netlink generic message `OVS_DP_CMD_NEW` and creates a datapath struct in `net/openvswitch/datapath.c`.
4. `ofproto-dpif.c` calls `dpif_port_add()` for each `Port` row in OVSDB; the provider sends `OVS_VPORT_CMD_NEW`.
5. ofproto installs the initial flow table 0 with `dpif_operate(struct dpif_flow_put)`. The netlink provider translates this into `OVS_FLOW_CMD_NEW`.
6. ofproto-dpif-upcall.c spawns handler threads; each thread blocks on `dpif_recv()` waiting for upcalls.
7. When the first packet arrives, the kernel megaflow misses, an upcall `OVS_PACKET_CMD_MISS` is generated, and the handler thread `dpif_recv` returns a `struct dpif_upcall` containing the packet plus key.
8. The handler runs xlate, generates datapath actions, calls `dpif_flow_put()` to install the megaflow, and `dpif_execute()` to forward the first packet (because the kernel does not forward on miss by itself).
9. Every subsequent packet of the same flow takes the fast path in the kernel, not going through dpif (dpif only exists in userspace).

The `dpif_class` (vtable) in `lib/dpif-provider.h` (OVS v2.17.9) is organized by group:

```
struct dpif_class {
    const char *type;                    /* "system", "netdev", "dummy" */
    int (*init)(void);                   /* Provider registration */

    /* Datapath lifecycle */
    int  (*open)(const struct dpif_class *, const char *name,
                 bool create, struct dpif **);
    void (*close)(struct dpif *);
    int  (*destroy)(struct dpif *);

    /* Port management */
    int (*port_add)(struct dpif *, struct netdev *, odp_port_t *port_no);
    int (*port_del)(struct dpif *, odp_port_t);
    int (*port_query_by_number)(const struct dpif *, odp_port_t,
                                struct dpif_port *);

    /* Flow table */
    int (*flow_flush)(struct dpif *);
    struct dpif_flow_dump *(*flow_dump_create)(const struct dpif *, ...);
    int (*flow_dump_next)(struct dpif_flow_dump_thread *,
                          struct dpif_flow *flows, int max_flows);
    void (*operate)(struct dpif *, struct dpif_op **ops, size_t n_ops, ...);
        /* ^ batched flow_put / flow_del / execute */

    /* Upcall / packet I/O */
    int  (*recv_set)(struct dpif *, bool enable);
    int  (*recv)(struct dpif *, uint32_t handler_id,
                 struct dpif_upcall *, struct ofpbuf *buf);
    void (*recv_wait)(struct dpif *, uint32_t handler_id);

    /* Conntrack, meter, bond, cache config (later additions) */
    int (*ct_dump_start)(struct dpif *, ...);
    int (*ct_flush)(struct dpif *, ...);
    int (*meter_set)(struct dpif *, ...);
    /* ... and a few other utility methods */
};
```

Two concrete classes (declared extern in the same `lib/dpif-provider.h` file, implemented in separate files):

| Field | `dpif_netlink_class` | `dpif_netdev_class` |
|-------|---------------------|---------------------|
| File | `lib/dpif-netlink.c` | `lib/dpif-netdev.c` |
| `type` | `"system"` | `"netdev"` |
| Datapath | Linux kernel module openvswitch.ko | Userspace, in-process |
| Transport | Netlink generic (`ovs_datapath`, `ovs_flow`, `ovs_vport`) | Function call inside the same process |
| `recv` | Dequeue from Netlink socket | Enqueue from PMD thread |
| Suitable for | Default production, kernel offload | DPDK, AF_XDP, containers without the module |
| Throughput | 5 to 10 Gbps per NIC | 50+ Gbps when pinning DPDK PMD cores |

**How an engineer masters dpif.** A skilled engineer reflexively performs five operations: (1) Identify the current provider with `ovs-vsctl get Bridge br0 datapath_type`; if blank, the default is `system`. Cross-check with `ovs-appctl dpif/show` (the first line will be `system@ovs-system` or `netdev@ovs-netdev`). (2) Read the capability matrix with `ovs-appctl dpif/show-dp-features <bridge>` before pushing a new OF rule with an unusual action; if the provider does not support it, xlate will emulate (with extra recirc) or reject. (3) Clearly distinguish dpif flows (installed in-kernel megaflows) from OF flows (the source rules in ofproto): `ovs-ofctl dump-flows br0` is not the same as `ovs-appctl dpif/dump-flows br0`; the latter is the output of `dpif_flow_dump_*`. (4) Read upcall statistics in `ovs-appctl dpif/show -s` to learn `lost` (queue overflow), `hits`, `missed`; a `lost/missed` ratio greater than 1 percent signals provider congestion. (5) Enable provider-specific debug logging with `ovs-appctl vlog/set dpif:file:dbg` and `vlog/set dpif_netlink:file:dbg` to trace each flow_put/flow_del when corruption is suspected.

**Classification.** dpif can be classified along several axes: (a) **By type string**, `system` (kernel netlink), `netdev` (userspace netdev including DPDK), `dummy` (in-memory test, not exposed outside unit tests). (b) **By data plane location**, kernel-resident or userspace-resident. (c) **By transport**, Netlink (system) or function call (netdev/dummy). (d) **By lifecycle owner**, kernel module (a system datapath exists independently of ovs-vswitchd) or process (a netdev datapath dies when ovs-vswitchd dies). (e) **By API method completeness**, full provider (with ct, meter, bond, cache) or restricted provider (dummy only implements the subset needed for tests). By importance: (a) > (b) > (c) > (d) > (e).

**Usage workflow.** Typical workflow when an engineer must debug a mismatch between an OF rule and actual bridge behavior:

```
Step 1: Identify the current provider.
        $ ovs-vsctl get Bridge br0 datapath_type
        → "system" (kernel) or "netdev" (userspace) or "" (default=system).
        $ ovs-appctl dpif/show
        → The first line displays type@name, for example "system@ovs-system".

Step 2: Read the current provider's capabilities.
        $ ovs-appctl dpif/show-dp-features br0
        → Look at Recirc, CT state, CT zone, CT mark, NAT, Sample, Clone, Truncate, MPLS.
        → If a needed feature is reported "No", that is a candidate root cause.

Step 3: Compare OF rules with installed dpif flows.
        $ ovs-ofctl dump-flows br0          # OF rules (source)
        $ ovs-appctl dpif/dump-flows br0    # installed megaflows (destination)
        → If an OF rule exists but dpif has no corresponding flow after sending a packet,
          xlate or dpif_flow_put failed (check the log).

Step 4: Observe the upcall flow.
        $ ovs-appctl dpif/show -s
        → hits / missed / lost. lost > 0 means upcall queue overflow.
        $ ovs-appctl coverage/show | grep -E "dpif|upcall"
        → flow_put count, recv count, miss count.

Step 5: Enable debug log if provider corruption is suspected.
        $ ovs-appctl vlog/set dpif:file:dbg
        $ ovs-appctl vlog/set dpif_netlink:file:dbg   # if the provider is system
        → Tail ovs-vswitchd.log; each flow_put/flow_del appears.
        → Disable debug afterward: ovs-appctl vlog/set dpif:file:warn.

Step 6: Verify the kernel module is loaded (if the provider is system).
        $ lsmod | grep openvswitch
        → If missing, dpif_netlink.open() returned ENODEV and the bridge has no data plane.
```

Best practice: never change `datapath_type` of a production bridge that is forwarding traffic; create a new bridge with the other type and migrate ports. Anti-pattern: calling `dpif-netlink-*` symbols directly in patch code (this violates dependency inversion).

**When something goes wrong.** `dpif` is a mandatory checkpoint in these situations: (a) The bridge "does not forward anything" although OF rules are fully installed; first check `dpif/show` to see whether the provider was actually opened (lost queue, recv disabled). (b) `ovs-vswitchd` log spams `dpif|WARN|... could not add ...`, meaning `dpif_flow_put` failed; usually because the provider lacks a capability or because an older kernel module does not understand a newer odp action. (c) After reloading the kernel module, the bridge loses all flows; remember that the system provider has a lifecycle independent of ovs-vswitchd, and `flow-restore-wait` must be set on restart. (d) A DPDK deployment reports "datapath_type netdev not registered", typically because ovs-vswitchd was built without DPDK support but OVSDB still configures netdev. (e) Capability mismatch after upgrading OVS userspace without upgrading the kernel module to the same version; new features (such as extended ct_label) fall back to emulation or are rejected.

**Tightly related to.** `dpif` is interleaved with: (a) `ofproto-dpif xlate` (the consumer above calling `dpif_flow_put`, while reading capability to decide whether to emulate, see §9.32 §9.32.3). (b) `upcall` (pushed up from below, dpif is the transport, see §9.2.4 and §9.2.10). (c) `megaflow` and `recirc_id` (the entries dpif manages are megaflows with `recirc_id` at the start of the key, see §9.2.2 and §9.2.14.2). (d) `openvswitch.ko` (the system provider is implemented on top of this kernel module, see §9.2.14.3). (e) `ovs-dpctl`, `ovs-appctl dpif/*` (external CLIs also use the dpif API through the control socket, see §9.4 and §9.11). (f) The `OVSDB Bridge.datapath_type` column (configures the provider choice, see §10.x). (g) `UFID` (the 128-bit Unique Flow ID emitted by dpif when installing a flow, see §9.2.11).

**Source code reference.** Per Rule 14 we use function-name anchors (line numbers drift across versions). Baseline OVS v2.17.9:

- `lib/dpif.h`, the public client API. Main functions: `dpif_open()`, `dpif_close()`, `dpif_port_add()`, `dpif_port_del()`, `dpif_flow_put()`, `dpif_flow_del()`, `dpif_flow_dump_create()`, `dpif_flow_dump_next()`, `dpif_execute()`, `dpif_recv()`, `dpif_recv_wait()`. The block comment at the top of the file states clearly: "A datapath is a collection of physical or virtual ports... a datapath has no intelligence of its own. Rather, it relies entirely on its client to set up flows."
- `lib/dpif-provider.h`, defines `struct dpif_class` (the vtable that providers implement) and `struct dpif` (the instance handle). Two extern class declarations at the end of the file: `dpif_netlink_class`, `dpif_netdev_class`.
- `lib/dpif.c`, the dispatch wrapper. The function `dpif_open()` looks up the type through `do_open()` and calls `class->open()`. The function `dp_register_provider()` lets plugins add types to the registry. The function `dpif_normalize_type()` maps the default empty value to `system`.
- `lib/dpif-netlink.c`, the kernel provider. The function `dpif_netlink_open()` (line 354 at v2.17.9) sends `OVS_DP_CMD_NEW` over Netlink. Flow modifications go through the class vtable callback `dpif_class->flow_put`, dispatched from `lib/dpif.c::dpif_flow_put`, not through a top-level `dpif_netlink_flow_put` function. The function `dpif_netlink_recv()` (line 3133) dequeues `OVS_PACKET_CMD_MISS` upcalls from the kernel.
- `lib/dpif-netdev.c`, the userspace provider. The function `dpif_netdev_open()` (line 1888) allocates the in-process `struct dp_netdev`. The function `dp_netdev_flow_add()` (line 4078) inserts into the in-memory classifier. Packet reception from PMD-thread queues is dispatched through the class vtable callback, not a top-level `dpif_netdev_recv` function.
- `ofproto/ofproto-dpif-upcall.c`, the consumer above the dpif layer. The function `udpif_upcall_handler()` (line 809) loops, calls `dpif_recv()`, and runs xlate.
- `ofproto/ofproto-dpif.c`, the integration layer. The class entry point is `ofproto_dpif_class.construct` (the `construct` callback at line 1649), which calls `dpif_open()` when the bridge is created. Bridge teardown goes through `dpif_close()` (already cited above), not a separate `dpif_destroy` function. All function names in this list are verified at the OVS v2.17.9 baseline (commit `0bea06d9957e3966d94c48873cd9afefba1c2677`, 2024-02-08).

None of them exist as named functions; flow-put dispatch goes through the class vtable, the netdev open is `dpif_netdev_open`, the netdev recv path uses PMD threads through the class vtable, and bridge teardown uses `dpif_close`. **Real incident scenario (rewritten in English by the 2026-04-28 source-citation cleanup).** A multi-host OpenStack Neutron deployment (about 40 compute nodes) reported the following: after upgrading the `ovs-vswitchd` userspace from 2.13 to 2.17 while FORGETTING to upgrade the kernel module `openvswitch.ko` (still the distro's 2.13 builtin), some bridges showed unexpected behavior: OF rules with `actions=ct(commit,nat(src=10.0.0.5))` failed to apply NAT to packets passing through. The dpif symptom: `ovs-appctl dpif/show-dp-features br-int` reported `CT state NAT: No`. Investigation: the log contained `dpif_netlink|WARN|sending packet failed (Operation not supported)`; the capability matrix between userspace expectation and kernel reality did not match. Root cause: the 2.13 kernel module did not support the new NAT option in `OVS_CT_ATTR` that the 2.17 userspace emitted. When the flow-put dispatcher in `lib/dpif.c::dpif_flow_put` (verified at line 1022) routed the flow to the kernel through the class vtable callback `dpif_class->flow_put`, the kernel rejected it; `ovs-vswitchd` logged a warning and the rule silently had no effect. Mitigation: rebuild and load the kernel module from the same 2.17 source tree (`apt install openvswitch-datapath-dkms`), reload the module, and the bridge recovers its megaflows automatically. Lesson: when upgrading OVS, always sync userspace and kernel module together; the dpif capability matrix is the fastest tool to detect mismatch, ahead of blaming the controller or OF rule logic.

> **Note on the citation.** Earlier versions of this paragraph cited `dpif_netlink_flow_put()` as if it were a top-level function in `lib/dpif-netlink.c`. That function does not exist at v2.17.9; flow-put dispatch goes through the `dpif_class->flow_put` callback in the vtable, called from `lib/dpif.c::dpif_flow_put` at line 1022. The Netlink backend implements the callback inline as part of `dpif_netlink_class`, not as an exported top-level function. The 2026-04-28 source-citation cleanup pass corrected the narrative.

**Common errors and diagnostic signals.**

- *Symptom* Bridge created but `ovs-appctl dpif/show` does not list it. *Cause* Provider open failed (kernel module missing or wrong type spelling). *Diagnosis* Tail `ovs-vswitchd.log`, look for `dpif|ERR|could not create datapath`. *Fix* `lsmod | grep openvswitch`, reload the module, or fix `datapath_type`.
- *Symptom* `ovs-appctl dpif/show -s` shows `lost: <growing number>`. *Cause* Upcall queue overflow; handler threads cannot dequeue fast enough. *Diagnosis* `ovs-appctl coverage/show | grep upcall`, `dpif/show -s` to compare hits versus lost. *Fix* Increase `n-handler-threads` with `ovs-vsctl set Open_vSwitch . other_config:n-handler-threads=8`.
- *Symptom* OF rule installed successfully (`ovs-ofctl dump-flows` shows it) but packets do not match when going through. *Cause* `dpif_flow_put` was rejected by the provider (capability mismatch, action not supported). *Diagnosis* `ovs-appctl vlog/set dpif:file:dbg`, tail the log for warn messages when the first packet arrives. *Fix* Check capability with `dpif/show-dp-features`, or upgrade the provider (kernel module or userspace if netdev) to match the version.
- *Symptom* After reloading the kernel module, the bridge loses all megaflows; traffic disrupted for about 5 to 30 seconds. *Cause* The system provider has a lifecycle independent of ovs-vswitchd; reloading the module resets the datapath. *Diagnosis* `dmesg | tail` shows the module unload/load sequence. *Fix* Before reload, set `ovs-vsctl set Open_vSwitch . other_config:flow-restore-wait=true`; after ovs-vswitchd reconnects, unset it; in production avoid reloading the module under hot traffic.
- *Symptom* DPDK port reports "could not add port: No such device". *Cause* The bridge has `datapath_type=system` but DPDK port types only exist in `dpif-netdev`. *Diagnosis* `ovs-vsctl get Bridge br0 datapath_type`. *Fix* Create a new bridge with `datapath_type=netdev` and migrate the DPDK port.

**Comparison with other systems.** Other systems solve "separating interface from implementation for the data plane" differently:

- **Linux VFS** (`struct file_operations`): a very close analog. Each filesystem (ext4, xfs, fat) implements the vtable; kernel clients (`vfs_read`, `vfs_write`) call through the interface. `dpif_class` is essentially VFS `file_operations` for a data plane switch.
- **Linux netfilter `struct nf_hook_ops`**: similar in vtable shape but oriented toward a hook chain, not data plane abstraction; netfilter does not split "kernel netfilter" and "userspace netfilter" the way dpif splits the kernel/userspace datapath.
- **OpenStack Neutron ML2 mechanism driver**: equivalent at a higher layer; each mechanism driver (ovs, linuxbridge, sr-iov) implements an interface and the controller dispatches. Difference: ML2 sits in the control plane, dpif sits in the userspace data plane.
- **Cisco IOS Forwarding Information Base abstraction**: separates CEF (Cisco Express Forwarding) kernel-style from hardware ASIC FIB. Equivalent in abstraction principle, different in scope (Cisco is proprietary firmware).
- **VPP plugin model** (FD.io): VPP has a `vlib_node` graph with plugins. Difference: VPP plugins shape the packet processing graph, dpif only shapes the flow table interface.
- **Linux tc `cls_*` classifier**: NOT equivalent because tc classifier has no "dispatch by type"; every classifier follows the same in-kernel plugin model. dpif fully separates userspace from the data plane backend.

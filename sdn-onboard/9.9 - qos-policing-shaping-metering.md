## 9.9.0 Opening context, why QoS still matters in 2026

In June 2023, a mobile carrier in Southeast Asia prepared to launch 5G SA (standalone) service for 12 million subscribers. The core network infrastructure ran on OpenStack with Neutron and ML2 OVN, with 100 Gbps of Internet bandwidth shared across several services: mobile broadband, IoT telemetry, enterprise VPN, and VoLTE voice. The operations team received a report that voice call quality degraded during the 19:00 to 21:00 peak window, with jitter reaching 120 ms (the acceptable threshold is 30 ms). The investigation found a QoS configuration mistake: the OVS bridges on the compute nodes had no shaping for the voice queue, and all traffic shared the same 1000-packet FIFO. When mobile broadband saturated the link, voice packets were delayed in the queue and jitter rose because of buffer bloat. The fix was to configure `linux-htb` with three queues (voice 20 Mbps guaranteed, data 70 Mbps guaranteed, best-effort 10 Mbps) on each Internet-facing interface, then add an OpenFlow rule that matches DSCP EF (46) and steers it into the voice queue. Within 30 minutes jitter dropped below 15 ms and voice calls stabilised. The lesson is that in real production, QoS is not a "nice to have"; it is a basic infrastructure requirement, especially when many traffic classes compete for the same link.

QoS has four infrastructure goals that map onto four measurable SLA indicators:

**One, bandwidth guarantee.** Make sure a traffic class always has at least X Mbps even when the link is saturated. For example voice needs about 100 kbps per call. Use the HTB `min-rate`.

**Two, latency control.** Keep latency below the threshold for real-time traffic. Voice and video need to stay below 150 ms one-way end-to-end. Use a priority queue and a low-latency qdisc such as CoDel or FQ-CoDel.

**Three, jitter reduction.** Keep latency variation small. Jitter that exceeds the average latency makes voice and video uneven. Use shaping instead of policing.

**Four, loss prevention.** Reduce the drop rate for TCP-based applications. Every dropped packet causes a retransmission and a congestion window reduction, which hurts effective bandwidth. Use ECN marking and Active Queue Management instead of tail-drop.

This part rebuilds the full QoS foundation in OVS following Crichigno Lab 9 and Compass Chapter I.

## 9.9.1 OVS configures QoS, does not implement it

The key Compass Chapter I point is that **OVS does not implement QoS, it only configures it**. The real queueing is done by:

- Linux `tc` (traffic control) for the kernel datapath.
- The DPDK `rte_sched` library for the userspace datapath.

OVS provides an abstract API through OVSDB tables (`QoS`, `Queue`) and an OpenFlow action (`set_queue`). When configured, OVS translates this into a `tc class` plus a `tc filter` for the kernel, or into an `rte_sched` config for DPDK.

Two kinds of QoS:

- **Ingress policing.** A simple rate limiter (token bucket) applied directly on the Interface row.
- **Egress shaping.** A hierarchical setup with the QoS and Queue tables plus an OpenFlow action.

## 9.9.2 Ingress policing

```bash
[host]# ovs-vsctl set interface tap0 ingress_policing_rate=1000 \
                                     ingress_policing_burst=100
```

- `ingress_policing_rate`: kbps.
- `ingress_policing_burst`: kb.
- Set to 0 to disable.

Packets that exceed the rate are dropped at ingress immediately. There is no fairness between multiple sources.

## 9.9.3 Egress shaping with linux-htb

The Compass Chapter I verbatim recipe:

```bash
[host]# ovs-vsctl -- add-br br0 \
         -- add-port br0 eth0 \
         -- add-port br0 vif1.0 -- set interface vif1.0 ofport_request=5 \
         -- add-port br0 vif2.0 -- set interface vif2.0 ofport_request=6 \
         -- set port eth0 qos=@newqos \
         -- --id=@newqos create qos type=linux-htb \
               other-config:max-rate=1000000000 \
               queues:123=@vif10queue queues:234=@vif20queue \
         -- --id=@vif10queue create queue other-config:max-rate=10000000 \
         -- --id=@vif20queue create queue other-config:max-rate=20000000
```

- QoS parent `max-rate=1 Gbps`.
- Queue ID 123 max 10 Mbps (for vif1.0).
- Queue ID 234 max 20 Mbps (for vif2.0).
- The Queue ID is an arbitrary OpenFlow queue ID that the `set_queue:` action references.

**Required step.** An OpenFlow rule that steers packets into the correct queue:

```bash
[host]# ovs-ofctl add-flow br0 in_port=5,actions=set_queue:123,normal
[host]# ovs-ofctl add-flow br0 in_port=6,actions=set_queue:234,normal
```

Compass Chapter I warns: "**Without the OpenFlow rules, packet go to the default queue and nothing is shaped, this is the single most common QoS misconfiguration in the field.**"

## 9.9.5 Cleanup (QoS does not auto-GC)

QoS and Queue rows are **not garbage-collected** and must be destroyed explicitly:

```bash
[host]# ovs-vsctl -- destroy QoS eth0 -- clear Port eth0 qos
[host]# ovs-vsctl -- --all destroy Queue
```

This is an important difference from Mirror (Part 9.7). Without cleanup, orphan rows accumulate.

## 9.9.6 HTB deep dive: tree structure, borrow, ceil

Hierarchical Token Bucket is the main shaping algorithm in the Linux kernel since 2003, used by OVS through `type=linux-htb` on the QoS row. Lab 9 §1.2 describes HTB as a hierarchical tree:

```
              Root (100 Mbps max)
              /            \
         Class 1          Class 2
       (rate 70 Mbps)   (rate 30 Mbps)
        ceil 100 Mbps    ceil 100 Mbps
         |                  |
      Leaf Q1            Leaf Q2
      (qdisc: TBF)       (qdisc: FIFO)
```

Five properties of HTB:

**One, only a leaf class holds a real queue.** Intermediate nodes (root, class 1, class 2) are token routers and do not buffer packets. A leaf has a queue to enqueue packets when the rate is too fast. If you forget to add a qdisc to the leaf, the kernel falls back to the default pfifo.

**Two, each class has two values: `rate` and `ceil`.** `rate` is the guaranteed bandwidth, `ceil` is the maximum bandwidth. A class can "borrow" extra bandwidth between `rate` and `ceil` when a sibling has spare tokens.

**Three, the borrow mechanism between siblings.** If class 2 does not use its full 30 Mbps, class 1 can borrow up to ceil 100 Mbps. When class 2 starts using its share again, class 1 must give bandwidth back down to its rate of 70 Mbps. This mechanism is called "work-conserving", because it does not waste bandwidth.

**Four, ceil is a hard limit.** Class 1 cannot exceed ceil 100 Mbps even when the entire link is idle.

**Five, HTB applies on egress only.** HTB attaches a qdisc only on the outbound (egress) direction of an interface. Inbound traffic must use ingress policing, or be redirected into an `ifb` device with HTB.

### OVS syntax for a two-class HTB configuration

From Lab 9 §6 Step 2 (page 18):

```bash
ovs-vsctl -- set port s1-eth3 qos=@newqos \
  -- --id=@newqos create qos type=linux-htb \
       other-config:max-rate=100000000 \
       queues=1=@q1,2=@q2 \
  -- --id=@q1 create queue other-config:min-rate=70000000 \
  -- --id=@q2 create queue other-config:min-rate=30000000
```

- `max-rate=100000000` (100 Mbps) is the ceil of the root.
- `min-rate=70000000` (70 Mbps) is the guaranteed rate of queue 1.
- `min-rate=30000000` (30 Mbps) is the guaranteed rate of queue 2.
- Queue IDs 1 and 2 are arbitrary OpenFlow queue IDs referenced by the `set_queue:` action.

If you do not set `max-rate` on an individual queue, it inherits the ceil from its parent (100 Mbps). This means both queues can borrow up to 100 Mbps when a sibling is idle.

### Verification with the tc command

After OVS creates the QoS, `tc` shows the corresponding HTB tree:

```bash
tc qdisc show dev s1-eth3
# qdisc htb 1: root refcnt 2 r2q 10 default 0 direct_packets_stat 0

tc class show dev s1-eth3
# class htb 1:1 root rate 100Mbit ceil 100Mbit burst 1600b cburst 1600b
# class htb 1:2 parent 1:1 prio 0 rate 70Mbit ceil 100Mbit burst 0b cburst 0b
# class htb 1:3 parent 1:1 prio 0 rate 30Mbit ceil 100Mbit burst 0b cburst 0b
```

Two tools confirm the same state: OVS shows it through `qos/show`, Linux shows it through `tc`. When debugging, always check both.

## 9.9.7 Policing versus shaping, a deep comparison

Both limit the rate, but their behaviour for packets that exceed the rate is completely different.

**Ingress policing drops immediately.** When a packet exceeds the token bucket rate, the kernel drops it at the entry point. No buffering, no delay. Pro: very simple implementation, low overhead. Con: TCP receives a packet-loss signal, the congestion window halves, and effective bandwidth is unstable. Jitter is high because every drop triggers a retransmission.

**Egress shaping buffers and sends gradually.** When the rate exceeds the threshold, the packet enters a queue. When tokens are available, the packet is dequeued. Pro: TCP does not see packet loss, so effective bandwidth is stable and latency is even. Con: queue delay grows when the link is overloaded, which is buffer bloat. A smart qdisc (CoDel, FQ-CoDel) is needed to control queue depth.

### Important counter-evidence from Lab 9 §3 Step 6

Lab 9 poses an interesting reverse hypothesis: "policing at 500 Mbps will deliver 500 Mbps of effective bandwidth". Real test:

```bash
# Terminal s1 (Linux)
sudo ovs-vsctl set interface s1-eth1 ingress_policing_rate=500000

# Terminal h1
iperf3 -c 10.0.0.2 -p 21
# Real throughput: ~79 Mbps, NOT 500 Mbps
```

The hypothesis is refuted. The reason (Lab 9 page 13): *"traffic policing interacts poorly with some network protocols and can have surprising results. This is a limitation of ingress policing and occurs mostly with higher rates like 500 Mbps."*

A deeper explanation. TCP is an adaptive protocol with respect to packet loss. When a policer drops packets randomly in unpredictable burst patterns, TCP congestion control reduces cwnd excessively, and effective bandwidth ends up far below the configured rate. The evidence shows ingress policing is not the right tool for high-rate traffic; it is only good for low-rate limiting (1 to 20 Mbps) or as a hard cap.

The alternative at the same 500 Mbps target is egress shaping (HTB) instead of ingress policing. Shaping buffers packets, TCP does not see loss, congestion control works normally, and effective bandwidth approaches the configured rate.

### Comparison table

| Property | Ingress policing | Egress shaping |
|----------|-------------------|----------------|
| Behaviour over rate | Drop immediately | Buffer and send |
| Latency | Low (no queue) | Grows when overloaded |
| Jitter | High (random packet loss) | Low (steady output) |
| TCP effective bandwidth | Unstable at high rates | Stable |
| Overhead | Very low | Medium (queue management) |
| Best use case | Hard cap, low rate (1 to 20 Mbps) | Guaranteed bandwidth, multi-class |
| OVS configuration | `ingress_policing_rate` on Interface | QoS, Queue, and an OpenFlow action |

## 9.9.8 3-color metering (srTCM and trTCM), RFC 2697 and RFC 2698

Metering is a technique that classifies traffic into three colours (Green, Yellow, Red) according to the rate compared with two thresholds:

- **CIR (Committed Information Rate):** the rate guaranteed by contract.
- **PIR (Peak Information Rate):** the peak rate allowed for short bursts.

Two main algorithms (IETF standards):

**Single-Rate Three-Color Marker (srTCM), RFC 2697.** Uses a single rate (CIR) plus two burst sizes (CBS committed burst and EBS excess burst). Packets:

- Within CBS: Green.
- Above CBS but within EBS: Yellow.
- Exceeding both: Red.

**Two-Rate Three-Color Marker (trTCM), RFC 2698.** Uses two rates (CIR and PIR) plus two burst sizes.

- Below CIR: Green.
- Between CIR and PIR: Yellow.
- Above PIR: Red.

Yellow packets are usually moved to a lower-priority queue or DSCP-marked. Red packets are dropped, or marked at the highest level.

### OVS supports only 2-color in OVS 2.17.9

Lab 9 §1.4 notes: *"Though Open vSwitch supports three color marking, our version of Open vSwitch only supports two color marking (green and red)."* In other words, OVS 2.17 (and the kernel datapath) only distinguishes green and red, with no intermediate yellow. For real 3-color marking you must use the OVS DPDK userspace with `trtcm-policer`, or upgrade to OVS 3.x.

Configuring 2-color metering via an OpenFlow 1.5 meter, from Lab 9 §4 Steps 2 to 4 (page 13):

```bash
# Step 1: view meter features
sudo ovs-ofctl -O OpenFlow15 meter-features s1
# Output: band_types=drop (meaning 2-color drop is supported)

# Step 2: add a meter that limits to 300 Mbps
sudo ovs-ofctl -O OpenFlow15 add-meter s1 \
  meter=1,kbps,band=type=drop,rate=300000

# Step 3: install a flow that uses the meter for TCP port 21
sudo ovs-ofctl -O OpenFlow15 add-flow s1 \
  tcp,tp_dst=21,actions=meter:1,output:3

# Step 4: install a flow that uses the same meter for TCP port 80
sudo ovs-ofctl -O OpenFlow15 add-flow s1 \
  tcp,tp_dst=80,actions=meter:1,output:3
```

A single meter can be referenced by multiple flows. Lab 9 Step 4 comments: *"Since one meter is applied to multiple flow entries, all the flow entries will share the meter rate."* If meter 1 limits to 300 Mbps and the flows for TCP port 21 and TCP port 80 both use meter 1, the total bandwidth of both will not exceed 300 Mbps.

### Verification with two competing hosts

Lab 9 §5 Steps 6 and 7 demo: h1 (port 21) and h3 (port 80) each send 300 Mbps, so the total reaching the switch is 600 Mbps. The 300 Mbps meter drops the excess. Real result (Lab 9 page 17):

- h1 gets about 200 Mbps.
- h3 gets about 170 Mbps.
- A total of 370 Mbps is marked Green (forwarded) and 230 Mbps is marked Red (dropped).

Note: the split between h1 and h3 is not 1:1 because the meter has no fairness, only a rate limit. For fairness, use HTB with two equal classes.

## 9.9.9 Lab 9 topology, four-host two-switch competing bandwidth

Lab 9 topology (Figure 3, page 6):

```
    h1 (10.0.0.1) ─┐                         ┌─ h2 (10.0.0.2)
                   │                         │    FTP server
                   ├── s1 ─── 100 Mbps ── s2 ┤
                   │                         │
    h3 (10.0.0.3) ─┘                         └─ h4 (10.0.0.4)
     HTTP client                              HTTP server
```

The s1 to s2 link is the 100 Mbps bottleneck. Four hosts share subnet 10.0.0.0/8:

- **h1** is the FTP client, sending TCP port 21 through s1, s2, to h2.
- **h2** is the FTP server, receiving TCP port 21.
- **h3** is the HTTP client, sending TCP port 80 through s1, s2, to h4.
- **h4** is the HTTP server, receiving TCP port 80.

QoS is installed on port `s1-eth3` (output toward s2). Two queues:

- Queue 1 (rate 70 Mbps): for TCP port 21 (FTP, simulating production traffic).
- Queue 2 (rate 30 Mbps): for TCP port 80 (HTTP, simulating background traffic).

Two OpenFlow rules:

```bash
# FTP into Queue 1 (high priority)
sudo ovs-ofctl add-flow s1 tcp,tp_dst=21,actions=set_queue:1,normal

# HTTP into Queue 2 (low priority)
sudo ovs-ofctl add-flow s1 tcp,tp_dst=80,actions=set_queue:2,normal
```

### Three verification scenarios

**Scenario A, h1 alone, no competition.** h1 runs iperf3 to port 21 alone. Result (Lab 9 page 21): about 95.6 Mbps. h1 borrows bandwidth from the idle queue 2 and reaches close to the 100 Mbps maximum.

**Scenario B, h3 alone, no competition.** h3 runs iperf3 to port 80 alone. Result: about 95.6 Mbps. Similarly, h3 borrows from the idle queue 1.

**Scenario C, h1 and h3 competing at the same time.** Both run iperf3 for 20 seconds in parallel. Expected result (Lab 9 page 23, document-plausible):

- h1 (queue 1 guaranteed 70 Mbps): close to 70 Mbps.
- h3 (queue 2 guaranteed 30 Mbps): close to 30 Mbps.
- Total 100 Mbps, equal to the link capacity.

This is evidence of work-conserving HTB: when one queue is idle, the other borrows; when both are busy, each queue is guaranteed its own rate.

## 9.9.10 Comparison with OVN QoS

OVN does not force the operator to write a QoS row manually like OVS does. Instead, OVN provides an API abstraction through the Logical Switch Port columns `qos_max_rate` and `qos_min_rate`:

```bash
# Configure QoS per LSP through ovn-nbctl
ovn-nbctl set Logical_Switch_Port lsp-vm1 \
  options:qos_max_rate=100000000 \
  options:qos_min_rate=70000000
```

OVN will automatically:

1. Generate Logical Flows in SBDB.
2. The `ovn-controller` on the compute node translates them into OVS QoS and Queue rows on `br-int`.
3. Generate an OpenFlow rule with `set_queue:N` to classify traffic from that LSP.

Three important differences:

**Abstraction level.** The OVS operator writes a QoS row per port manually. The OVN operator only sets an option on the LSP and the compiler underneath handles the rest.

**Dynamic scale.** OVN generates the queue when a VM is created and destroys it when the VM is deleted. With manual OVS, you must destroy the Queue row yourself when it is no longer needed (`--all destroy Queue`).

**OVN limitation.** The OVN API today supports only simple cases: `qos_max_rate` and `qos_min_rate` per LSP. It does not support multi-class per LSP and does not expose 3-color metering. When complex QoS is needed, you still drop down to manual OVS.

A pragmatic rule: OVN QoS is enough for about 80 percent of real deployments (caps bandwidth per VM, priority for voice). When you need 3-color metering, complex classification by DSCP or CoS, or interaction with hardware offload, use manual OVS.

## Guided Exercise 11, policing 10 Mbps versus 500 Mbps

**Goal.** Verify that the proposition "ingress policing scales linearly with the rate" is refuted. Observe the difference in effective bandwidth between a low policing rate (10 Mbps) and a high one (500 Mbps).

**Reverse hypothesis to refute:**

> *"An ingress policing rate of X Mbps will produce X Mbps of effective bandwidth in every situation. If I set the rate to 500 Mbps, iperf3 will reach close to 500 Mbps."*

### Preparation

Load the Lab 9 topology with Mininet (use a Python `Topo` script or MiniEdit with `lab9.mn`). Make sure bridge s1 is running and that hosts h1 to h4 have the correct IPs.

### Step 1, baseline test without QoS

On h2, start an iperf3 server on port 21:

```bash
h2$ iperf3 -s -p 21
```

On h1, run the client:

```bash
h1$ iperf3 -c 10.0.0.2 -p 21
```

Record the effective bandwidth. Expected (Lab 9 page 10): about 43.9 Gbps (Mininet virtual veth has no rate limit).

### Step 2, policing at 10 Mbps

On the Linux terminal of s1 (outside the Mininet CLI):

```bash
sudo ovs-vsctl set interface s1-eth1 ingress_policing_rate=10000
```

Run iperf3 on h1 again. Expected effective bandwidth: about 10.6 Mbps (close to the configured rate).

### Step 3, policing at 500 Mbps

Remove the old QoS, set the rate to 500 Mbps:

```bash
sudo ovs-vsctl -- --all destroy QoS
sudo ovs-vsctl set interface s1-eth1 ingress_policing_rate=500000
```

Run iperf3 again. Expected effective bandwidth: about 79 Mbps (far below the 500 Mbps target).

### Step 4, analysis

The reverse hypothesis is refuted: a 500 Mbps rate produces only 79 Mbps of effective bandwidth, not a linear 500 Mbps. The cause is TCP congestion control:

1. The policer drops packets randomly when the rate exceeds 500 Mbps.
2. The TCP sender sees duplicate ACKs or a timeout, halves the congestion window (Reno), or reduces it by 30 percent (CUBIC).
3. A low cwnd means the sender cannot reach the bandwidth-delay product required for 500 Mbps.
4. The steady-state effective bandwidth oscillates at a low level.

### Step 5, use shaping instead of policing

Remove the policer and use HTB shaping at 500 Mbps:

```bash
sudo ovs-vsctl -- --all destroy QoS
sudo ovs-vsctl set interface s1-eth1 ingress_policing_rate=0
sudo ovs-vsctl -- set port s1-eth3 qos=@newqos \
  -- --id=@newqos create qos type=linux-htb \
       other-config:max-rate=500000000 \
       queues=1=@q1 \
  -- --id=@q1 create queue other-config:min-rate=500000000
sudo ovs-ofctl add-flow s1 \
  "tcp,tp_dst=21,actions=set_queue:1,normal"
```

Run iperf3 again. Expected effective bandwidth: about 480 to 495 Mbps (very close to the target). Evidence that shaping reaches a high rate better than policing.

### Lesson

Ingress policing is not the right tool for high rates (above 100 Mbps). For low-rate hard caps (1 to 20 Mbps) policing is acceptable. For multi-class bandwidth guarantees, shaping is the only correct choice.

```bash
sudo ovs-vsctl -- set port s1-eth3 qos=@newqos \
  -- --id=@newqos create qos type=linux-htb \
       other-config:max-rate=100000000 \
       queues=1=@q1,2=@q2 \
  -- --id=@q1 create queue other-config:min-rate=70000000 \
  -- --id=@q2 create queue other-config:min-rate=30000000

sudo ovs-ofctl add-flow s1 \
  "tcp,tp_dst=21,actions=set_queue:1,normal"
sudo ovs-ofctl add-flow s1 \
  "tcp,tp_dst=80,actions=set_queue:2,normal"
```

### Step 1, h1 alone

```bash
h2$ iperf3 -s -p 21
h1$ iperf3 -c 10.0.0.2 -p 21
```

Expected: about 95.6 Mbps (Lab 9 Figure 38). h1 borrows about 25 Mbps from the idle queue 2, giving a total close to 100 Mbps.

### Step 2, h3 alone

```bash
h4$ iperf3 -s -p 80
h3$ iperf3 -c 10.0.0.4 -p 80
```

Expected: about 95.6 Mbps (Lab 9 Figure 42). h3 borrows about 65 Mbps from the idle queue 1.

### Step 3, h1 and h3 at the same time

On h1 and h3 simultaneously (issue the two commands less than one second apart):

```bash
h1$ iperf3 -c 10.0.0.2 -t 20 -p 21
h3$ iperf3 -c 10.0.0.4 -t 20 -p 80
```

Expected:

- h1: about 70 Mbps (queue 1 guaranteed rate).
- h3: about 30 Mbps (queue 2 guaranteed rate).
- Total: about 100 Mbps, equal to link capacity.

### Step 4, observe `qos/show`

While the test is running, view stats from the Linux terminal:

```bash
sudo ovs-appctl -t ovs-vswitchd qos/show s1-eth3
```

Expected:

- `Queue: 1` with `tx_packets` and `tx_bytes` rising quickly (h1 traffic).
- `Queue: 2` with counters rising (h3 traffic).
- Queue 0 (default) with `min_rate` 12000 (OVS default), not used.

### Step 5, cleanup

```bash
sudo ovs-vsctl -- --all destroy QoS
sudo ovs-vsctl --all destroy Queue
```

### Lesson

Work-conserving HTB guarantees two properties: (a) when the link is idle, an idle class can borrow bandwidth; (b) when the link is busy, each class is guaranteed its own rate. No class is starved. This is why HTB is common in production, in contrast with older CBQ (class-based queueing), which lacked full borrow.

## Core takeaways to remember

**One.** QoS has four goals (bandwidth guarantee, latency control, jitter reduction, loss prevention). Each goal needs a different tool: HTB for bandwidth, priority queue plus CoDel for latency and jitter, shaping for loss.

**Two.** OVS configures QoS through an OVSDB abstraction (QoS and Queue rows). The real queue runs in Linux `tc` or DPDK `rte_sched`. To debug, check both views: `ovs-appctl qos/show` and `tc class show`.

**Three.** HTB has a tree structure of root, class, and leaf queue, and only the leaf holds a real queue. The borrow and ceil mechanism lets a class borrow from an idle sibling and give the bandwidth back when the sibling becomes busy. It is work-conserving and does not waste bandwidth.

**Four.** Egress shaping needs three things: a QoS row, a Queue row, and an OpenFlow `set_queue` action. Missing the OpenFlow rule is the most common mistake. Compass Chapter I states *"the single most common QoS misconfiguration in the field"*.

**Five.** Ingress policing drops immediately when the rate is exceeded. It is good for low-rate hard caps (1 to 20 Mbps). At high rates (above 100 Mbps), policing interacts badly with TCP and effective bandwidth ends up far below the configured rate (Lab 9 demonstrates 500 Mbps to 79 Mbps in practice).

**Six.** 3-color metering (RFC 2697 srTCM and RFC 2698 trTCM) marks traffic green, yellow, or red against CIR and PIR. The OVS 2.17 kernel datapath only supports 2-color (green and red). The DPDK userspace supports 3-color through `trtcm-policer`.

**Seven.** QoS and Queue rows are NOT auto-GC. They must be destroyed explicitly with `ovs-vsctl -- --all destroy QoS -- --all destroy Queue`. Orphan rows accumulate and create confusion when debugging.

**Eight.** OVN QoS through `qos_max_rate` and `qos_min_rate` on an LSP fits about 80 percent of use cases. When you need 3-color metering, DSCP or CoS classification, or interaction with hardware offload, use manual OVS.

## 9.9.11 Detailed schema, five peripheral entities to master

The five subsections below expand the OVSDB schema and traffic-control backend view for five peripheral keywords that an OVS operator touches often when configuring QoS: two row schemas (`QoS row`, `Queue row`) and three `type` values (`linux-htb`, `linux-hfsc`, `linux-noop`). Each subsection traces the schema definition in `vswitchd/vswitch.ovsschema` through the handler implementation in `lib/netdev-linux.c`, with specific OVS v2.17.9 source citations.

### 9.9.11.1 QoS row, the parent of the shaping hierarchy

QoS is a top-level OVSDB table (`isRoot: true`) with four main columns per `vswitch.ovsschema` v2.17.9:

```json
"QoS": {
  "columns": {
    "type":         { "type": "string" },
    "queues":       { "type": { "key": {"type":"integer", "minInteger":0,
                                        "maxInteger":4294967295},
                                "value": {"type":"uuid", "refTable":"Queue"},
                                "min":0, "max":"unlimited" } },
    "other_config": { "type": {"key":"string", "value":"string", ...} },
    "external_ids": { "type": {"key":"string", "value":"string", ...} }
  },
  "isRoot": true
}
```

**Role and placement.** A QoS row is the anchor of a shaping policy for one port. `Port.qos` (a weak-reference UUID column) points to a QoS row; if it is NULL the port has no egress shaping hierarchy and only the default kernel qdisc applies. A QoS row can be referenced from multiple Ports (uncommon but the schema allows it) because `isRoot=true` does not depend on a reference to exist. This is why the row does not auto-GC when `Port.qos` is cleared.

**The `type` column.** A string identifier that matches a handler name in the `tc_ops` registry of `lib/netdev-linux.c`. Values supported in OVS v2.17.9: `linux-htb`, `linux-hfsc`, `linux-noop`, `linux-codel`, `linux-fq_codel`, `linux-sfq`, `linux-netem`, `egress-policer`, `trtcm-policer`, or the empty string `""` (default qdisc).

**The `queues` column (integer-to-uuid map).** The key is an arbitrary OpenFlow queue ID (0 to 2^32 minus 1) that the `set_queue:N` action references, and the value is the UUID of a Queue row. `min:0` lets the QoS row have no queue (for example `linux-noop` does not use queues).

**The `other_config` column.** A string-to-string map holding parameters for the `type`. For example with `linux-htb`, a common key is `max-rate` (the ceil of the root, in bps).

**Configuration recipe (verbatim from the OVS QoS Howto):**

```bash
ovs-vsctl -- set port eth0 qos=@qos1 \
          -- --id=@qos1 create qos type=linux-htb \
                other-config:max-rate=1000000000 queues:0=@q0
```

`@qos1` is an OVSDB row-reference symbol. `set port` inserts the UUID into `Port.qos` in the same commit as `create qos` inserts the new row. This is the atomic OVSDB transaction pattern that avoids a race condition.

**Explicit cleanup (why no auto-GC).** The schema attribute `isRoot: true` marks the QoS row as existing independently. The OVSDB garbage collector only removes rows with `isRoot: false` when they lose all references. The cleanup commands:

```bash
ovs-vsctl -- destroy QoS eth0 -- clear Port eth0 qos
```

### 9.9.11.2 Queue row, the leaf node of the hierarchy

A Queue row is a top-level OVSDB table describing a leaf class in the HTB or HFSC tree. Schema v2.17.9:

```json
"Queue": {
  "columns": {
    "dscp":         { "type": { "key": {"type":"integer","minInteger":0,
                                        "maxInteger":63},
                                "min":0, "max":1 } },
    "other_config": { "type": {"key":"string","value":"string", ...} },
    "external_ids": { "type": {"key":"string","value":"string", ...} }
  },
  "isRoot": true
}
```

Queue has only three content columns: `dscp` is optional (0 to 63, sets DSCP marking when a packet enters the queue), `other_config` is a map of shaping parameters, and `external_ids` is a map for external tag identifiers.

**Common `other_config` keys (per the `linux-htb` handler in `lib/netdev-linux.c`):**

- `min-rate`: guaranteed rate for the class (bps). Required for an HTB leaf.
- `max-rate`: ceil of the class (bps). Optional, defaults to inheriting the ceil from the parent.
- `burst`: burst size in bytes. Optional.
- `priority`: integer 0 to 7 for HTB priority resolution when sibling classes contend for tokens.

**Why the Queue row is separate from the QoS row.** An HTB tree has many leaf classes, and each leaf needs its own parameters (rate, ceil). OVSDB normalises this into two tables: the QoS row holds the root plus the map `queues:N=<uuid>`, which points to a Queue row that holds the parameters for leaf N. This pattern is a classic one-to-many relational schema.

**Tightly related.** A Queue UUID appears in (a) the `QoS.queues` map value, and (b) Logical_Switch_Port options in OVN when compiled down to `br-int`. Each queue is translated by the kernel into a `tc class htb` with a classid that matches the OpenFlow queue ID.

**Orphan cleanup.** After `destroy QoS`, a Queue row still exists independently (`isRoot: true`). It must be cleaned up separately:

```bash
ovs-vsctl --all destroy Queue
```

Production note: if your workflow creates VMs frequently, write a post-destroy script to clear orphan Queues so that useless rows do not accumulate and slow down OVSDB queries.

### 9.9.11.3 linux-htb, the most common type for egress shaping

`linux-htb` is the `type` value that matches the `tc_ops_htb` handler in `lib/netdev-linux.c` v2.17.9 line 4987:

```c
static const struct tc_ops tc_ops_htb = {
    .ovs_name = "linux-htb",
    ...
};
```

**Mechanism.** When a user runs `set qos type=linux-htb`, `ovs-vswitchd` calls `htb_tc_install`, which translates the configuration into `tc qdisc add dev <port> root handle 1: htb` plus a `tc class add dev <port> parent 1: classid 1:N htb rate ... ceil ...` for each entry in the `QoS.queues` map.

**The `other_config` keys an HTB QoS row accepts (from `struct htb_class` line 4529 and `htb_query_class__` line 4708):**

- `max-rate`: ceil of the root class (1:1 in tc terminology).

**The `other_config` keys an HTB Queue row accepts:**

- `min-rate` (required): guaranteed rate.
- `max-rate`: per-class ceil. Defaults to inheriting from the root.
- `burst`: burst size in bytes.
- `priority`: 0 to 7, where lower means higher priority during resolution.

**Mapping to the tc backend (verified empirically):**

```bash
$ ovs-vsctl set port eth0 qos=@q -- --id=@q create qos type=linux-htb \
       other-config:max-rate=100000000 queues:1=@q1 \
    -- --id=@q1 create queue other-config:min-rate=70000000
$ tc qdisc show dev eth0
qdisc htb 1: root ... default 0 ...
$ tc class show dev eth0
class htb 1:1 root rate 100Mbit ceil 100Mbit ...
class htb 1:2 parent 1:1 rate 70Mbit ceil 100Mbit ...
```

Note: OpenFlow queue ID `1` maps to tc classid `1:2` (an offset of +1 because `1:1` is the root). This mapping has been stable in OVS from 2.0 to the present.

**Version drift.** The `linux-htb` handler has existed since OVS 1.x. The structure of `struct htb_class` did not change in v2.17. Lab 9 (based on OVS 2.17 on Ubuntu 22.04) uses syntax identical to OVS 3.x.

**Use case.** Multi-class bandwidth guarantee with borrowing between siblings, the most common production case. When you only need a simple hard cap, use `egress-policer` instead of HTB.

### 9.9.11.4 linux-hfsc, fair service curve for a real-time and best-effort mix

`linux-hfsc` is the `tc_ops_hfsc` handler at line 5484 of v2.17.9, with `struct hfsc_class` at line 5010. HFSC (Hierarchical Fair Service Curve) is a shaping algorithm from Carnegie Mellon (1997) that provides two service guarantees at the same time, which HTB does not: bandwidth and a delay bound.

**Mechanism, different from HTB.** HTB has a single curve (rate). HFSC has two curves:

- **Real-time (RT) curve:** a guaranteed delay bound for the first packet of a burst.
- **Link-share (LS) curve:** the bandwidth share when the link is saturated.

A class can have one or both curves. A class with only the LS curve behaves like HTB. A class with both RT and LS guarantees low delay for real-time traffic even when best-effort traffic has filled the queue.

**The `other_config` keys HFSC accepts (per `hfsc_parse_class_details__` line 5180):**

- QoS row: `max-rate` (ceil of the root).
- Queue row: `min-rate` (LS curve rate). The OVS HFSC handler only exposes a simple subset; the full m1, d, m2 service curve cannot be configured through the OVS abstraction and requires a direct `tc class change`.

**Example configuration:**

```bash
ovs-vsctl set port eth0 qos=@q -- --id=@q create qos type=linux-hfsc \
       other-config:max-rate=100000000 queues:0=@q0,1=@q1 \
    -- --id=@q0 create queue other-config:min-rate=20000000 \
    -- --id=@q1 create queue other-config:min-rate=80000000
```

**Use case.** A network that needs both voice (low delay) and bulk transfer (bandwidth share) at the same time. HTB does not guarantee a delay bound; HFSC does. In OVS production, HFSC is less common than HTB because configuration is complex and OVS does not expose the full curve API. When you encounter `type=linux-hfsc` in production, it is usually legacy config from before 2015.

**Verify:**

```bash
$ tc qdisc show dev eth0
qdisc hfsc 1: root refcnt 2 default 1
$ tc class show dev eth0
class hfsc 1: root
class hfsc 1:1 parent 1: ls m1 0bit d 0us m2 80Mbit ...
```

### 9.9.11.5 linux-noop, a passthrough that disables QoS without deleting schema

`linux-noop` is the `tc_ops_noop` handler at line 5525 of v2.17.9. The implementation is very simple (lines 5500 to 5527):

```c
static void noop_install__(struct netdev *netdev_) {
    struct netdev_linux *netdev = netdev_linux_cast(netdev_);
    static const struct tc tc = TC_INITIALIZER(&tc, &tc_ops_default);
    netdev->tc = CONST_CAST(struct tc *, &tc);
}
static const struct tc_ops tc_ops_noop = {
    .ovs_name = "linux-noop",
    .tc_install = noop_tc_install,
    .tc_load    = noop_tc_load,
};
```

The handler does not call `tc qdisc add` at all; it only sets `netdev->tc` to a default placeholder. The result is that the kernel keeps its default qdisc (`pfifo_fast` on older Ubuntu, `fq_codel` on Ubuntu 22.04 with systemd-networkd). No shaping, no rate limit.

**Why it exists.** Two main use cases:

1. **Temporarily disable QoS without deleting the schema row.** In production, switching between two policies (from `linux-htb` rate 100 Mbps to `linux-noop` to disable, then to `linux-htb` rate 200 Mbps) avoids destroying the QoS row and recreating it (which loses the Port reference and creates complex audit logs). Just run `set qos type=linux-noop`.
2. **Fast diagnostic.** When you suspect QoS is the bottleneck, switch to `linux-noop` to confirm. If bandwidth recovers, the problem is in the shaping config. If not, the cause lies elsewhere.

**Observation.** After `set qos type=linux-noop`, `tc qdisc show dev <port>` shows the default kernel qdisc (`fq_codel` or `pfifo_fast`) instead of `htb`. No classes are declared.

**Limitation.** `linux-noop` does not remove Queue rows that are referenced from `QoS.queues`. The `queues` map still exists but the handler does not use it. To clean up fully you still need `destroy QoS` plus `--all destroy Queue`.

**Version drift.** The handler has existed since OVS 2.0 (commits from 2009 and 2010) and has not changed across releases. Ubuntu 22.04 with OVS 2.17 ships with it.

**Reverse use case.** When operating OVN inside OpenStack and the operator wants to debug "is QoS causing drops?", set `type=linux-noop` on the `br-int` queue temporarily; if drops disappear, QoS is misconfigured. This is faster than `--all destroy QoS` followed by recreating the configuration.

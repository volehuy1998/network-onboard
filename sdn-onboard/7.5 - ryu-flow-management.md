## 7.5.1 The Ryu event system

Ryu is built on an event-driven model. Every OpenFlow message received from a switch is mapped to a Python event object and then dispatched to every handler registered for that event type. Handlers register through the `@set_ev_cls` decorator:

```python
from ryu.base import app_manager
from ryu.controller import ofp_event
from ryu.controller.handler import CONFIG_DISPATCHER, MAIN_DISPATCHER
from ryu.controller.handler import set_ev_cls
from ryu.ofproto import ofproto_v1_3

class MySwitch(app_manager.RyuApp):
    OFP_VERSIONS = [ofproto_v1_3.OFP_VERSION]

    @set_ev_cls(ofp_event.EventOFPSwitchFeatures, CONFIG_DISPATCHER)
    def switch_features_handler(self, ev):
        # Called when the switch first connects (handshake complete)
        datapath = ev.msg.datapath
        self.install_table_miss_flow(datapath)

    @set_ev_cls(ofp_event.EventOFPPacketIn, MAIN_DISPATCHER)
    def packet_in_handler(self, ev):
        # Called for each packet_in message from the switch
        msg = ev.msg
        datapath = msg.datapath
        in_port = msg.match['in_port']
        # ... process the packet ...
```

The second argument to `@set_ev_cls` is the dispatcher state. Ryu only delivers an event to the handler when the switch is in the matching state:

| Dispatcher | Meaning |
|------------|---------|
| `CONFIG_DISPATCHER` | Just after the switch connects, before it is ready to forward |
| `MAIN_DISPATCHER` | The switch is operating normally. Most events use this state. |
| `DEAD_DISPATCHER` | The switch has disconnected |

> **Key point:** Ryu's `EventDispatcher` is single-threaded. Every handler must finish quickly and must not block. If a handler calls `time.sleep()` or performs blocking I/O (a database query, a DNS lookup), the entire controller is blocked for that duration. When you need a blocking operation, use Ryu's `hub.spawn()` to start a separate greenlet. This is the most common reason a Ryu controller suddenly becomes slow under heavy load in benchmarks.

## 7.5.2 OFPFlowMod: install and delete flow entries

`OFPFlowMod` is the OpenFlow message used to add, modify, or delete a flow entry in a switch flow table. It is the most important message the controller sends to the switch.

### 7.5.2.1 Install a flow entry (OFPFC_ADD)

```python
def add_flow(self, datapath, priority, match, actions,
             idle_timeout=0, hard_timeout=0, cookie=0):
    ofproto = datapath.ofproto
    parser = datapath.ofproto_parser

    # Instruction: apply the actions immediately
    inst = [parser.OFPInstructionActions(
        ofproto.OFPIT_APPLY_ACTIONS, actions
    )]

    mod = parser.OFPFlowMod(
        datapath=datapath,
        cookie=cookie,
        priority=priority,
        idle_timeout=idle_timeout,   # 0 = do not expire on inactivity
        hard_timeout=hard_timeout,   # 0 = do not expire by absolute time
        match=match,
        instructions=inst
    )
    datapath.send_msg(mod)
```

Example: install a flow that forwards out port 2 when a packet arrives on port 1 with a specific destination MAC:

```python
parser = datapath.ofproto_parser

match = parser.OFPMatch(
    in_port=1,
    eth_dst='fa:16:3e:11:22:33'
)
actions = [parser.OFPActionOutput(2)]
self.add_flow(datapath, priority=10, match=match, actions=actions,
              idle_timeout=30)   # Auto-remove after 30 seconds without traffic
```

### 7.5.2.2 Delete a flow entry (OFPFC_DELETE)

```python
def delete_flow(self, datapath, match):
    ofproto = datapath.ofproto
    parser = datapath.ofproto_parser

    mod = parser.OFPFlowMod(
        datapath=datapath,
        command=ofproto.OFPFC_DELETE,
        out_port=ofproto.OFPP_ANY,    # Match any output port
        out_group=ofproto.OFPG_ANY,
        match=match
    )
    datapath.send_msg(mod)
```

### 7.5.2.3 Table-miss flow entry

The table-miss is a priority-0 flow entry with an empty match (it matches every packet). This entry must be installed as soon as the switch connects. Without it, packets that match no flow are silently dropped instead of being forwarded as a `packet_in` to the controller:

```python
def install_table_miss_flow(self, datapath):
    ofproto = datapath.ofproto
    parser = datapath.ofproto_parser

    match = parser.OFPMatch()   # Empty match = matches every packet
    actions = [parser.OFPActionOutput(
        ofproto.OFPP_CONTROLLER,
        ofproto.OFPCML_NO_BUFFER   # Send the entire packet to the controller
    )]
    self.add_flow(datapath, priority=0, match=match, actions=actions)
```

> **Common misconception:** "`OFPFC_DELETE` clears the entire flow table when the match is empty." In reality, an empty match together with `OFPFC_DELETE` does not mean "match all" in Ryu unless `out_port=OFPP_ANY` and `out_group=OFPG_ANY` are set explicitly. If those two fields are missing, the switch may reject the delete or only remove entries whose output port matches exactly.

## 7.5.3 The REST API pattern in Ryu

Ryu integrates a WSGI server based on `eventlet`. A developer can expose HTTP endpoints to drive the controller from outside the process. For example, a CI/CD pipeline can add a flow rule when a new service is deployed, or a network management UI can query the flow table.

The standard pattern uses `WSGIApplication` and `ControllerBase`:

```python
from ryu.app.wsgi import ControllerBase, WSGIApplication, route
from ryu.base import app_manager
from webob import Response
import json

class FlowManager(app_manager.RyuApp):
    _CONTEXTS = {'wsgi': WSGIApplication}

    def __init__(self, *args, **kwargs):
        super().__init__(*args, **kwargs)
        wsgi = kwargs['wsgi']
        wsgi.register(FlowRestController,
                      {'flow_manager_app': self})
        self.datapaths = {}   # dp_id -> datapath object

    @set_ev_cls(ofp_event.EventOFPStateChange,
                [MAIN_DISPATCHER, DEAD_DISPATCHER])
    def state_change_handler(self, ev):
        datapath = ev.datapath
        if ev.state == MAIN_DISPATCHER:
            self.datapaths[datapath.id] = datapath
        elif ev.state == DEAD_DISPATCHER:
            self.datapaths.pop(datapath.id, None)

class FlowRestController(ControllerBase):
    def __init__(self, req, link, data, **config):
        super().__init__(req, link, data, **config)
        self.app = data['flow_manager_app']

    @route('flow', '/v1/switches', methods=['GET'])
    def list_switches(self, req, **kwargs):
        switches = list(self.app.datapaths.keys())
        body = json.dumps({'switches': [hex(s) for s in switches]})
        return Response(content_type='application/json', body=body)
```

Start Ryu with the REST endpoint:

```bash
[controller]$ ryu-manager flow_manager.py --wsapi-port 8080
[controller]$ curl http://localhost:8080/v1/switches
{"switches": ["0x1"]}
```

## 7.5.4 Collecting traffic statistics

Ryu can request statistics from the switch on a schedule and process the replies to build a visibility layer. This is how Ryu is used in traffic-engineering research: poll flow stats every 10 seconds, detect elephant flows (a flow consuming more than 80 percent of bandwidth), and trigger rerouting by installing a higher-priority flow entry.

**Port statistics** (transmitted and received bytes, packets, drops):

```python
def request_port_stats(self, datapath):
    ofproto = datapath.ofproto
    parser = datapath.ofproto_parser
    req = parser.OFPPortStatsRequest(datapath, 0, ofproto.OFPP_ANY)
    datapath.send_msg(req)

@set_ev_cls(ofp_event.EventOFPPortStatsReply, MAIN_DISPATCHER)
def port_stats_reply_handler(self, ev):
    for stat in ev.msg.body:
        self.logger.info(
            'Port %d: rx_pkts=%d tx_pkts=%d rx_bytes=%d tx_bytes=%d '
            'rx_drops=%d tx_drops=%d',
            stat.port_no,
            stat.rx_packets, stat.tx_packets,
            stat.rx_bytes, stat.tx_bytes,
            stat.rx_dropped, stat.tx_dropped
        )
```

**Flow statistics** (packet count, byte count, and duration per entry):

```python
def request_flow_stats(self, datapath):
    parser = datapath.ofproto_parser
    req = parser.OFPFlowStatsRequest(datapath)
    datapath.send_msg(req)

@set_ev_cls(ofp_event.EventOFPFlowStatsReply, MAIN_DISPATCHER)
def flow_stats_reply_handler(self, ev):
    for stat in ev.msg.body:
        self.logger.info(
            'Table %d priority %d: packets=%d bytes=%d duration=%ds',
            stat.table_id, stat.priority,
            stat.packet_count, stat.byte_count,
            stat.duration_sec
        )
```

### Guided Exercise 1: write a Ryu MAC-learning app with REST stats

In this exercise you write a complete Ryu application. It learns MAC addresses from `packet_in`, installs a unicast flow entry with `idle_timeout`, and exposes a REST endpoint that returns the `packet_in` counter. The exercise illustrates how the rate of `packet_in` events drops once the flow entries are learned.

**Outcomes:**
- Write a Ryu app whose `packet_in` handler installs a flow entry with `idle_timeout=30`.
- Expose a REST GET endpoint at `/stats/packet_in` returning the counter as JSON.
- Observe `packet_in_count` rising quickly at first and then slowing down once the switch forwards on its own.

**Before you begin:**
Ryu installed (`pip install ryu`). OVS running. Mininet optional for a test topology.

**Instructions:**

**1.** Create the file `mac_learning_rest.py`:

```python
from ryu.base import app_manager
from ryu.controller import ofp_event
from ryu.controller.handler import CONFIG_DISPATCHER, MAIN_DISPATCHER
from ryu.controller.handler import set_ev_cls
from ryu.ofproto import ofproto_v1_3
from ryu.lib.packet import packet, ethernet
from ryu.app.wsgi import ControllerBase, WSGIApplication, route
from webob import Response
import json

class MacLearningRest(app_manager.RyuApp):
    OFP_VERSIONS = [ofproto_v1_3.OFP_VERSION]
    _CONTEXTS = {'wsgi': WSGIApplication}

    def __init__(self, *args, **kwargs):
        super().__init__(*args, **kwargs)
        wsgi = kwargs['wsgi']
        wsgi.register(StatsController, {'app': self})
        self.mac_to_port = {}
        self.packet_in_count = 0

    @set_ev_cls(ofp_event.EventOFPSwitchFeatures, CONFIG_DISPATCHER)
    def features_handler(self, ev):
        dp = ev.msg.datapath
        parser = dp.ofproto_parser
        ofproto = dp.ofproto
        match = parser.OFPMatch()
        actions = [parser.OFPActionOutput(
            ofproto.OFPP_CONTROLLER, ofproto.OFPCML_NO_BUFFER)]
        inst = [parser.OFPInstructionActions(
            ofproto.OFPIT_APPLY_ACTIONS, actions)]
        dp.send_msg(parser.OFPFlowMod(
            datapath=dp, priority=0, match=match, instructions=inst))

    @set_ev_cls(ofp_event.EventOFPPacketIn, MAIN_DISPATCHER)
    def packet_in_handler(self, ev):
        self.packet_in_count += 1
        msg = ev.msg
        dp = msg.datapath
        ofproto = dp.ofproto
        parser = dp.ofproto_parser
        in_port = msg.match['in_port']

        pkt = packet.Packet(msg.data)
        eth = pkt.get_protocol(ethernet.ethernet)
        if eth is None:
            return

        dst = eth.dst
        src = eth.src
        dpid = dp.id
        self.mac_to_port.setdefault(dpid, {})
        self.mac_to_port[dpid][src] = in_port

        out_port = self.mac_to_port[dpid].get(dst, ofproto.OFPP_FLOOD)
        actions = [parser.OFPActionOutput(out_port)]

        if out_port != ofproto.OFPP_FLOOD:
            match = parser.OFPMatch(in_port=in_port, eth_dst=dst)
            inst = [parser.OFPInstructionActions(
                ofproto.OFPIT_APPLY_ACTIONS, actions)]
            dp.send_msg(parser.OFPFlowMod(
                datapath=dp, priority=1, idle_timeout=30,
                match=match, instructions=inst))

        data = msg.data if msg.buffer_id == ofproto.OFP_NO_BUFFER else None
        dp.send_msg(parser.OFPPacketOut(
            datapath=dp, buffer_id=msg.buffer_id,
            in_port=in_port, actions=actions, data=data))

class StatsController(ControllerBase):
    def __init__(self, req, link, data, **config):
        super().__init__(req, link, data, **config)
        self.app = data['app']

    @route('stats', '/stats/packet_in', methods=['GET'])
    def get_count(self, req, **kwargs):
        body = json.dumps({'packet_in_count': self.app.packet_in_count})
        return Response(content_type='application/json', body=body)

    @route('stats', '/stats/mac_table', methods=['GET'])
    def get_mac_table(self, req, **kwargs):
        table = {hex(k): v for k, v in self.app.mac_to_port.items()}
        return Response(content_type='application/json',
                        body=json.dumps(table))
```

## Key points to remember

First, the single-threaded event system in Ryu is a double-edged sword. It simplifies programming by removing concurrent access to shared state, but it requires every handler to be non-blocking. Any blocking I/O inside a handler stops the entire controller.

Second, an `OFPFlowMod` with `idle_timeout` is a natural garbage-collection mechanism for the flow table. A learned-MAC entry should carry `idle_timeout=30` so it is removed automatically when the host is no longer active. This avoids a forwarding table that fills up with stale entries. Forgetting the timeout is a common bug when lab code is moved into production.

Third, the REST API pattern in Ryu separates the control plane (the HTTP endpoint) from the data plane (the OpenFlow handler). The pattern matters for research: an external script can add or remove a flow rule through REST without touching the Ryu code, which makes experiments easier to automate and to reproduce.

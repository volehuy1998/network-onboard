## Dàn bài

## 9.14.1 4-layer mental model

Khi troubleshoot mọi incident OVS/OVN, giữ trong đầu 4 layer từ abstract đến concrete:

1. **OVSDB config layer** — "cái gì nên tồn tại" (Bridge, Port, Interface, Mirror, QoS rows).
2. **OpenFlow flow-table layer** — "packet nên được xử lý thế nào" (`ovs-ofctl dump-flows`).
3. **Datapath cache layer** — "fast path đã học được gì" (`ovs-dpctl dump-flows -m`).
4. **Wire layer** — "thực sự rời NIC là cái gì" (`tcpdump` trên egress interface).

Troubleshoot = so sánh 4 layer và tìm layer nào disagree với layer kề.

## 9.14.2 Branch 1: Packet loss (counter tăng nhưng packet không tới)

1. `ovs-dpctl show` → kiểm tra column `lost`. Nếu tăng, upcall overflow.
2. `ovs-appctl upcall/show` → xem flow install queue depth. Nếu near limit 200000, raise với `upcall/set-flow-limit 500000`.
3. Nếu raise không giúp, tune thread: `other_config:n-handler-threads=8` + `n-revalidator-threads=4`.

## 9.14.3 Branch 2: Unexpected drops (packet bị drop bất ngờ)

1. `ovs-appctl ofproto/trace br0 "<flow spec khớp packet>"`.
2. Walk output table-by-table, tìm table nào rule match với action=`drop` hoặc không có rule match (sẽ fall-through về default).
3. So với OpenFlow intent: `ovs-ofctl dump-flows br0 table=<N>`.

## 9.14.4 Branch 3: Tunnel down (Geneve/VXLAN không traffic)

1. `ovs-appctl tnl/neigh/show` → ARP/ND underlay resolution. Nếu empty, underlay connectivity issue.
2. Ping tunnel endpoint từ chassis: `ping <remote-ip>`.
3. MTU check: `ip link show <underlay-iface>` — Geneve cần MTU underlay ≥ inner MTU + 58.
4. Firewall underlay: `iptables -L` hoặc `nft list ruleset` — UDP port Geneve 6081 hoặc VXLAN 4789 có bị drop không.

## 9.14.5 Branch 4: Bond flap

1. `ovs-appctl bond/show bond0` → column `may_enable` cho từng member, last flap time.
2. `ovs-appctl lacp/show bond0` (nếu LACP) → Actor + Partner state, "letter salad" missing letter nào.
3. Peer switch side: check show LACP neighbors, port errors, speed/duplex mismatch.
4. `other_config:lacp-fallback-ab=true` đã enable chưa — tránh black-hole khi LACP negotiation fail.

## 9.14.6 Branch 5: OVSDB cluster unavailable

1. `ovs-appctl -t /var/run/openvswitch/ovsdb-server.ctl cluster/status OVN_Southbound` trên mỗi node.
2. Nếu majority down, restore bằng `cluster/leave` + `cluster/kick` để đẩy dead member.
3. Nếu chỉ leader mất, follower tự election — wait 5-10s.
4. Nếu split-brain (2+2), re-form cluster từ node với data mới nhất — DANGER, cần backup trước.

## 9.14.7 Appendix C compressed: 4-layer reconciliation

> *Are packets arriving?* → tcpdump on physical NIC.
> *Are they entering OVS?* → `dpctl/dump-flows -m` hit counters.
> *Are the right flows matching?* → `ofproto/trace`.
> *Is the action correct?* → read Datapath actions line.
> *Does the datapath agree?* → `dpctl/dump-flows` on actual kernel/netdev.
> *Does the wire agree?* → tcpdump on egress NIC.
> *When all four agree and packet still drops* → upstream switch.

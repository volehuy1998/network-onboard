## 4.7.1 Flow grammar recap and the `-O` flag

### Command structure

`ovs-ofctl add-flow <bridge> <flow-spec>` with flow-spec being a comma-separated list of match fields plus `actions=` plus an action list. General format:

```
[selectors,] match_fields [,actions=action_list]
```

Selectors apply to flow metadata, not packet match:

| Selector | Description | Default |
|----------|-------------|---------|
| `table=N` | Table number N (0 to 254) | 0 |
| `priority=N` | 0 to 65535, higher wins | 32768 |
| `cookie=0xHEX` | 64-bit identifier for flow management | 0 |
| `idle_timeout=N` | Seconds with no match, auto-remove | 0 (never) |
| `hard_timeout=N` | Seconds since install, auto-remove | 0 (never) |
| `importance=N` | 16-bit, OF 1.4+ eviction priority | 0 |
| `send_flow_rem` | Send FLOW_REMOVED when the flow expires | off |
| `check_overlap` | Reject if overlapping with an existing flow at the same priority | off |

### Wildcard priority semantics

When multiple flows match the same packet, higher priority wins. With equal priority, OVS picks the more specific match (fewer wildcards). An empty match (`actions=drop` with no match field) is the **catch-all** at the default priority 32768.

A common anti-pattern: install a `action=drop` priority=0 flow alongside an allow policy at priority=100. The intent is "deny by default", but priority=0 may match packets that do not conform to a specific policy when policy rules do not cover an edge case. Cross-reference cross-product explosion in 4.6.1.

### The critical `-O` flag

OVS defaults to **OpenFlow 1.0** when `ovs-ofctl` is invoked without specifying a version. OF 1.0 has no multi-table, group, meter, or instructions framework. If you use a feature from 1.1+, you MUST pass the `-O` flag:

```bash
# Wrong (OF 1.0, goto_table will error)
ovs-ofctl add-flow br0 "table=0,ip,actions=goto_table:1"

# Correct (OF 1.3, goto_table works)
ovs-ofctl -O OpenFlow13 add-flow br0 "table=0,ip,actions=goto_table:1"
```

OF version negotiation: the switch advertises supported versions and ovs-ofctl picks the compatible one. If the bridge is configured with `protocols=OpenFlow13`, ovs-ofctl still defaults to 1.0 when the command does not pass `-O`. Best practice: always pass `-O OpenFlow13` (or higher) for production scripts.

Configure the bridge to accept only OF 1.3+:

```bash
ovs-vsctl set bridge br0 protocols=OpenFlow13
```

After that, `ovs-ofctl` (without `-O`) fails with a "protocol not supported" error when the bridge refuses OF 1.0.

## 4.7.2 Match fields: 12-tuple plus NXM and OXM extensions

### L2 fields (Ethernet)

All L2 matches support exact or masked form:

| Field | Description | Example |
|-------|-------------|---------|
| `in_port=NAME_OR_ID` | Ingress port (name or OF port number) | `in_port=eth1` or `in_port=1` |
| `dl_src=MAC[/MASK]` | Source MAC | `dl_src=aa:bb:cc:00:00:00/ff:ff:ff:00:00:00` |
| `dl_dst=MAC[/MASK]` | Dest MAC | `dl_dst=01:80:c2:00:00:00/ff:ff:ff:ff:ff:f0` (STP multicast) |
| `dl_type=0xETHERTYPE` | EtherType | `dl_type=0x0800` (IPv4), `0x0806` (ARP), `0x8847` (MPLS), `0x88cc` (LLDP) |
| `dl_vlan=N` | VLAN ID 0 to 4094 | `dl_vlan=100` |
| `dl_vlan_pcp=N` | VLAN Priority 0 to 7 | `dl_vlan_pcp=5` |

Masked MAC matches use the `/` separator. For example, match all multicast:

```bash
ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=0,priority=100,dl_dst=01:00:00:00:00:00/01:00:00:00:00:00,action=drop"
```

Bit 0 of the first byte in MAC is the "multicast bit". Mask `01:00...` selects this bit; value `01:00...` matches when the bit is 1. Rule drops all multicast Ethernet (including broadcast `ff:ff:ff:ff:ff:ff` which has this bit).

### L3 fields (IPv4)

| Field | Description | Example |
|-------|-------------|---------|
| `nw_src=IP[/PREFIX]` | Source IP | `nw_src=10.0.0.0/8` |
| `nw_dst=IP[/PREFIX]` | Dest IP | `nw_dst=192.168.1.100` |
| `nw_proto=N` | IP protocol number | `nw_proto=6` (TCP), `17` (UDP), `1` (ICMP) |
| `nw_tos=N` | IP TOS (6 bits DSCP plus 2 bits ECN) | `nw_tos=0x10` |
| `nw_ttl=N` | Time To Live | `nw_ttl=1` (detect traceroute hop) |

IP match requires `dl_type=0x0800` (IPv4) first. OVS auto-infers dl_type when matching an L3 field, but best practice is to be explicit:

```bash
ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=0,ip,nw_src=10.0.0.0/8,action=goto_table:1"
# Shorthand "ip" equals "dl_type=0x0800"
```

### L4 fields (TCP, UDP, ICMP)

| Field | Description | Prerequisite condition |
|-------|-------------|------------------------|
| `tp_src=N` | L4 source port | `nw_proto=6` or `17` |
| `tp_dst=N` | L4 dest port | Same |
| `tcp_flags=FLAGS[/MASK]` | TCP flags bitmask | `nw_proto=6` |
| `icmp_type=N` | ICMP message type | `nw_proto=1` |
| `icmp_code=N` | ICMP message code | `nw_proto=1` |

TCP flags bitmask: match `tcp_flags=+syn-ack` (SYN set, ACK not set) detects SYN flood. Multiple flags use the `+` and `-` prefix:

```bash
ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=0,tcp,tcp_flags=+syn-ack,action=meter:1,NORMAL"
# Match only SYN set (no ACK), rate limit through meter 1
```

### Tunnel metadata (overlay networking)

When OVS receives a packet through a tunnel (VXLAN, Geneve, GRE), metadata is populated:

| Field | Description |
|-------|-------------|
| `tun_id=N` | VNI of VXLAN or VNI tag of Geneve |
| `tun_src=IP` | Outer IP source of the tunnel packet |
| `tun_dst=IP` | Outer IP destination |
| `tun_flags=FLAGS` | Tunnel-specific flags |

Match per-tenant traffic on overlay:

```bash
ovs-ofctl -O OpenFlow13 add-flow br-int \
  "table=20,tun_id=100,actions=mod_vlan_vid:100,NORMAL"
# Tunnel VNI 100 (tenant A) -> tag VLAN 100 inside the bridge
```

### Conntrack fields (OVS 2.5+ with kernel 4.3+)

Conntrack introduces stateful inspection into the OpenFlow pipeline:

| Field | Description |
|-------|-------------|
| `ct_state=+FLAG-FLAG` | Conntrack state bitmask |
| `ct_zone=N` | Zone (isolate overlapping NAT ranges) |
| `ct_mark=N/MASK` | Per-flow mark (32-bit) |
| `ct_label=N/MASK` | Per-flow label (128-bit) |

`ct_state` flags:

| Flag | Description |
|------|-------------|
| `trk` | Packet tracked through conntrack |
| `new` | First packet of a new connection |
| `est` | Established connection (return seen) |
| `rel` | Related (for example, FTP data channel from control) |
| `inv` | Invalid (malformed, out-of-window) |
| `rpl` | Reply direction |
| `snat` | SNAT applied |
| `dnat` | DNAT applied |

See recipe 4.7.5 for the canonical flow sequence.

### Registers (NXM extension, not standard OF)

OVS has NXM registers for temporary storage in the pipeline:

| Field | Size | Description |
|-------|------|-------------|
| `reg0` to `reg15` | 32-bit each | General purpose |
| `xreg0` to `xreg7` | 64-bit each | Wider registers (alias for a reg pair) |
| `xxreg0` to `xxreg3` | 128-bit each | Widest (alias for 4 regs) |
| `metadata` | 64-bit | Spec-defined metadata (OF 1.1+) |

Use case: store the tenant ID in reg0 at Table 0 and read it in Table 5 to choose the uplink:

```bash
ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=0,in_port=eth1,action=load:0x1->NXM_NX_REG0[],goto_table:1"
# Tenant A (port eth1), reg0 equals 0x1

ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=5,reg0=0x1,action=output:eth_tenant_a_uplink"
```

### Masked matches syntax

CIDR for IP: `nw_src=10.0.0.0/8`.

Slash-mask for an arbitrary field:

```
dl_src=00:11:22:00:00:00/ff:ff:ff:00:00:00  # Match OUI 00:11:22
reg0=0x10/0xff                              # Match low 8 bits equal to 0x10
```

## 4.7.3 Actions: 8 basics plus extensions

### Output actions

The basic action family decides where the packet goes:

| Action | Description |
|--------|-------------|
| `output:N` | Send out a specific port (name or number) |
| `NORMAL` | Pass the packet into the NORMAL L2 learning pipeline (MAC learning plus flood/unicast) |
| `FLOOD` | Send out all ports except ingress, subject to STP blocking |
| `ALL` | Send out all ports except ingress (ignoring STP) |
| `LOCAL` | Send to the local internal port (the bridge's internal port) |
| `IN_PORT` | Send back out the ingress port (hairpin) |
| `CONTROLLER[:max_len]` | Send PACKET_IN to the controller, optionally truncate at max_len bytes |
| `drop` | Do nothing (an empty action list equals drop) |

### Header modification

| Action | Description |
|--------|-------------|
| `mod_vlan_vid:N` | Set VLAN ID |
| `strip_vlan` | Remove VLAN tag |
| `push_vlan:0x8100` | Add VLAN tag (802.1Q) or 0x88a8 (QinQ) |
| `pop_vlan` | Same as strip_vlan |
| `mod_dl_src:MAC` | Rewrite source MAC |
| `mod_dl_dst:MAC` | Rewrite destination MAC |
| `mod_nw_src:IP` | Rewrite source IP |
| `mod_nw_dst:IP` | Rewrite destination IP |
| `dec_ttl` | Decrement IP TTL, send ICMP Time Exceeded if zero |

### General field operations

Generic access for any NXM/OXM field:

```
set_field:VALUE->FIELD    # Set field to value
load:VALUE->FIELD[BITS]   # Set a specific bit range
move:SRC->DST             # Copy from SRC field to DST field
```

For example, copy VLAN ID into reg0 for downstream lookup:

```bash
ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=0,dl_vlan=100,action=move:NXM_OF_VLAN_TCI[0..11]->NXM_NX_REG0[],goto_table:1"
```

### Pipeline actions (OF 1.1+)

| Action | Description |
|--------|-------------|
| `resubmit([PORT],[TABLE])` | NXM extension: re-run the packet through a table (OF 1.0 workaround for multi-table) |
| `goto_table:N` | OF 1.1+: continue the pipeline at table N (N greater than current) |
| `group:N` | Apply group actions |
| `meter:N` | Apply meter N (rate limit) |

`resubmit` is NXM-only, not standard OF. Used for OVS 1.0 to emulate multi-table. After OF 1.1, `goto_table` is the right way.

### Stateful actions (conntrack, learn)

```
ct(args)       # Conntrack action with args: commit, zone=N, nat(...), table=N, exec(...)
ct_clear       # Clear conntrack state (the packet leaves the pipeline without tracking)
learn(args)    # Install a new flow based on the current packet
```

See recipe 4.7.5 for ct. `learn` is advanced and installs a flow dynamically from current packet fields.

### Advanced actions

| Action | Description |
|--------|-------------|
| `conjunction(id,k/n)` | Conjunctive match (cross-product compression) |
| `clone(...)` | Run nested actions on a packet copy |
| `encap(...)` and `decap()` | Push and pop encapsulation (VXLAN, Geneve, NSH) |
| `check_pkt_larger:SIZE` | Check packet size versus threshold, update reg |
| `truncate:SIZE` | Truncate packet to size bytes before output |

`conjunction` is NXM-only and allows matching "any of set A AND any of set B" without cross-product explosion. Advanced topic, skipped in this part.

## 4.7.4 Multi-table pipeline recipe: 3-stage L3 routing (UofSC Lab 6 absorb)

### Topology

Canonical use case: OVS bridge s1 acting as L3 router between two subnets:

```
Host A (192.168.1.10/24) --eth1-- s1 --eth2-- Host B (192.168.2.20/24)
                          00:..:01            00:..:05
```

s1 has two internal ports (s1-eth1 in VLAN 1, s1-eth2 in VLAN 2), routing between two subnets.

### Single-table approach (cross-product explosion)

Naive approach: one flow per tenant-subnet pair:

```bash
# Direction 1: A to B
ovs-ofctl -O OpenFlow13 add-flow s1 \
  "table=0,priority=100,ip,in_port=eth1,nw_src=192.168.1.0/24,nw_dst=192.168.2.0/24,\
   action=mod_dl_src:00:00:00:00:00:01,mod_dl_dst:00:00:00:00:00:05,dec_ttl,output:eth2"

# Direction 2: B to A
ovs-ofctl -O OpenFlow13 add-flow s1 \
  "table=0,priority=100,ip,in_port=eth2,nw_src=192.168.2.0/24,nw_dst=192.168.1.0/24,\
   action=mod_dl_src:00:00:00:00:00:05,mod_dl_dst:00:00:00:00:00:01,dec_ttl,output:eth1"
```

Does not scale: N subnets gives N squared flow entries for all pairs. 10 subnets equals 100 flows; 100 subnets equals 10K flows, near the Broadcom Trident2 ACL limit.

### Multi-table approach (compression)

Three-stage structure:

**Table 0, Classifier**: classify the traffic type.

```bash
# ARP traffic, NORMAL learning (OVS handles)
ovs-ofctl -O OpenFlow13 add-flow s1 "table=0,priority=100,arp,action=normal"

# IP traffic, Table 1 (L3 routing)
ovs-ofctl -O OpenFlow13 add-flow s1 "table=0,priority=50,ip,action=goto_table:1"

# Default: drop
ovs-ofctl -O OpenFlow13 add-flow s1 "table=0,priority=0,action=drop"
```

**Table 1, L3 Forwarding**: match destination subnet, rewrite MAC, set egress metadata.

```bash
# Route to subnet 1 (192.168.1.0/24)
ovs-ofctl -O OpenFlow13 add-flow s1 \
  "table=1,priority=100,ip,nw_dst=192.168.1.0/24,\
   action=mod_dl_src:00:00:00:00:00:01,mod_dl_dst:00:00:00:00:00:01,dec_ttl,\
   load:1->NXM_NX_REG0[],goto_table:2"

# Route to subnet 2 (192.168.2.0/24)
ovs-ofctl -O OpenFlow13 add-flow s1 \
  "table=1,priority=100,ip,nw_dst=192.168.2.0/24,\
   action=mod_dl_src:00:00:00:00:00:05,mod_dl_dst:00:00:00:00:00:05,dec_ttl,\
   load:2->NXM_NX_REG0[],goto_table:2"
```

Note: at Table 1, the destination MAC must in reality be learned through ARP, not hard-coded. The recipe is simplified for the tutorial; production uses OVN or a controller application with ARP learning logic.

**Table 2, L2 Output**: output based on reg0 (subnet index).

```bash
# Reg0 = 1, eth1 (subnet 1 port)
ovs-ofctl -O OpenFlow13 add-flow s1 \
  "table=2,priority=100,reg0=1,action=output:eth1"

# Reg0 = 2, eth2 (subnet 2 port)
ovs-ofctl -O OpenFlow13 add-flow s1 \
  "table=2,priority=100,reg0=2,action=output:eth2"
```

Flow count: 4 rules at Table 0 plus N (subnet) rules at Table 1 plus N rules at Table 2 equals O(N) instead of O(N squared).

### Verification

```bash
ovs-ofctl -O OpenFlow13 dump-flows s1
```

Shows every flow with its hit counter. A ping packet from A to B will:

1. Hit `in_port=eth1,arp` first (ARP request), NORMAL learns MAC.
2. Hit `in_port=eth1,ip,nw_dst=192.168.2.0/24` (ICMP echo), Table 1, Table 2, output eth2.
3. Hit reverse flow when B replies.

The counter increments for each packet; debug through the `dump-flows` output.

## 4.7.5 Conntrack recipe: 5-flow stateful firewall (Compass Ch 9 plus UofSC Lab 8)

### Goal: stateful egress firewall

Policy: allow host A (left side) to initiate connections to host B (right side), but block connections initiated from B. Return traffic of A to B is allowed automatically.

Topology: bridge br0 with two ports `veth_l0` (host A) and `veth_r0` (host B).

### 5-flow sequence

```bash
# Flow 1: New outbound, send to conntrack (table 0 re-entry after ct)
ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=0,priority=50,ct_state=-trk,tcp,in_port=veth_l0,\
   action=ct(table=0)"

# Flow 2: Tracked plus new outbound, commit to conntrack plus forward
ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=0,priority=50,ct_state=+trk+new,tcp,in_port=veth_l0,\
   action=ct(commit),output:veth_r0"

# Flow 3: Return traffic (dst side), send to conntrack
ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=0,priority=50,ct_state=-trk,tcp,in_port=veth_r0,\
   action=ct(table=0)"

# Flow 4: Established reply, forward out left
ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=0,priority=50,ct_state=+trk+est,tcp,in_port=veth_r0,\
   action=output:veth_l0"

# Flow 5: Established outbound, forward
ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=0,priority=50,ct_state=+trk+est,tcp,in_port=veth_l0,\
   action=output:veth_r0"

# Default: drop (untrackable or unestablished)
ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=0,priority=0,action=drop"
```

### Flow logic

1. Packet from A (veth_l0) to B:
   - Flow 1 matches (ct_state=-trk), send to conntrack.
   - Packet returns with ct_state=+trk+new.
   - Flow 2 matches, commit to conntrack, output veth_r0.

2. Packet returned from B (veth_r0) to A:
   - Flow 3 matches (ct_state=-trk), send to conntrack.
   - Conntrack lookup finds an existing flow; the packet returns with ct_state=+trk+est.
   - Flow 4 matches, output veth_l0.

3. Subsequent packet A to B (established):
   - Flow 1 matches, send to conntrack.
   - Returns with ct_state=+trk+est.
   - Flow 5 matches, output veth_r0.

4. **Attack scenario**: B attempts to connect to A:
   - Flow 3 matches, send to conntrack.
   - Conntrack does not find an existing flow, returns with ct_state=+trk+new.
   - No flow matches (+trk+new from veth_r0), default drop.

Policy enforced: A may initiate, B may not.

### Verification

```bash
# List conntrack entries
ovs-appctl dpctl/dump-conntrack

# Example output:
# tcp,orig=(src=10.0.0.1,dst=10.0.0.2,sport=33456,dport=80),\
#   reply=(src=10.0.0.2,dst=10.0.0.1,sport=80,dport=33456),\
#   state=est,zone=0
```

Each conntrack entry has state (new, est, inv), zone (isolation), mark (metadata), label (128-bit).

### Advanced: NAT plus zone

Conntrack with SNAT (source NAT) to an external IP:

```bash
ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=0,priority=50,ct_state=+trk+new,tcp,in_port=veth_l0,\
   action=ct(commit,nat(src=203.0.113.10)),output:veth_r0"
```

Zone isolation for overlapping NAT:

```bash
# Tenant A in zone 100
ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=0,priority=50,ct_state=-trk,tcp,in_port=veth_tenant_a,\
   action=ct(table=0,zone=100)"

# Tenant B in zone 200 (same IP range but isolated)
ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=0,priority=50,ct_state=-trk,tcp,in_port=veth_tenant_b,\
   action=ct(table=0,zone=200)"
```

OVN uses this zone pattern for multi-tenant isolation. Cross-reference Part 13.3 (OVN ACL plus conntrack integration).

## 4.7.6 Groups plus meters (OF 1.1+ and 1.3+)

### Group types

OF 1.1 introduced groups with four types (see Part 4.0.4):

```bash
# ECMP (Equal-Cost Multi-Path) with two uplinks
ovs-ofctl -O OpenFlow13 add-group br0 \
  "group_id=1,type=select,\
   bucket=weight=50,output:eth_up1,\
   bucket=weight=50,output:eth_up2"

# Reference from a flow
ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=0,priority=100,ip,action=group:1"
```

Group types:

| Type | Description |
|------|-------------|
| `all` | Execute all buckets (multicast) |
| `select` | Execute one bucket based on hash (ECMP) |
| `indirect` | Exactly one bucket, pointer-like indirection |
| `fast_failover` | Watch port or group, active bucket based on liveness |

FAST_FAILOVER example (see Part 4.6 Capstone POE):

```bash
ovs-ofctl -O OpenFlow13 add-group br0 \
  "group_id=2,type=ff,\
   bucket=watch_port:eth_p1,output:eth_p1,\
   bucket=watch_port:eth_p2,output:eth_p2"
```

### Meters (OF 1.3+)

Per-flow rate limit with a meter:

```bash
# Meter 1: 1Mbps drop rate
ovs-ofctl -O OpenFlow13 add-meter br0 \
  "meter=1,kbps,band=type=drop,rate=1000,burst_size=100"

# Reference from a flow
ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=0,priority=100,ip,nw_src=10.0.0.100,action=meter:1,NORMAL"
# Traffic from host 10.0.0.100 rate-limited at 1 Mbps; over rate, drop
```

Multi-band DSCP_REMARK (color marking):

```bash
ovs-ofctl -O OpenFlow13 add-meter br0 \
  "meter=2,kbps,\
   band=type=dscp_remark,rate=1000,burst_size=100,prec_level=1,\
   band=type=drop,rate=10000,burst_size=1000"
# < 1 Mbps: green (pass)
# 1 to 10 Mbps: yellow (DSCP remark precedence + 1)
# > 10 Mbps: red (drop)
```

See Part 9.9 for OVS meter integration with the `linux-htb` QoS production recipe.

## 4.7.7 Flow hygiene: monitor, replace-flow, diff-flow

### Monitor: live stream of flow changes

When a controller app installs or removes flows, ovs-ofctl monitor streams every FLOW_MOD event:

```bash
ovs-ofctl monitor br0 watch:
# Output example:
# NXST_FLOW reply: (xid=0x3):
#  cookie=0x0, duration=5.12s, table=0, n_packets=100, n_bytes=8400, priority=50,ip,nw_dst=10.0.0.0/8 actions=goto_table:1
```

`watch:` filter:
- `table=N` only monitors a specific table.
- `cookie=HEX/MASK` filters by cookie.

Debug use case: controller logic has a bug that installs conflicting rules. Use monitor to see the live FLOW_MOD sequence from the controller side.

### replace-flow: atomic swap of an entire table

```bash
# Prepare a file with new flows
cat > new.flows <<EOF
table=0,priority=100,ip,nw_src=10.0.0.0/8,actions=goto_table:1
table=0,priority=0,actions=drop
EOF

# Atomic swap
ovs-ofctl -O OpenFlow13 replace-flows br0 new.flows
```

Different from `add-flows`: `add-flows` appends to the existing table, while `replace-flows` removes all existing flows (except those in new.flow with the same cookie or match) then installs the new ones.

`replace-flows` matters in production: atomic, with no moment when the table is in an inconsistent state. Equivalent to OF 1.4 bundle semantics (see Part 4.3.2) but available since OVS 2.0.

### diff-flow: preview changes

**MANDATORY** to run before `replace-flows` in production. Shows differences between the current table and the file:

```bash
ovs-ofctl -O OpenFlow13 diff-flows br0 new.flows
# Example output:
# +table=0,priority=100,ip,nw_src=10.0.0.0/8,actions=goto_table:1
# -table=0,priority=100,ip,nw_src=192.168.0.0/16,actions=goto_table:1
```

`+` means a flow will be added, `-` means a flow will be removed. If output is empty, there is no change.

Production workflow:

```bash
# Step 1: Preview
ovs-ofctl -O OpenFlow13 diff-flows br0 new.flows > /tmp/diff.txt
cat /tmp/diff.txt
# Review changes

# Step 2: If OK, apply
ovs-ofctl -O OpenFlow13 replace-flows br0 new.flows

# Step 3: Verify
ovs-ofctl -O OpenFlow13 dump-flows br0
```

### Common misunderstandings

**Misunderstanding:** "add-flow only adds new ones without touching existing flows; safe."

**Reality:** `add-flows` may unintentionally overwrite if the new flow has the same `(table, priority, match)` as an existing flow. OVS treats it as an update; the existing flow is replaced silently. Similar to `OFPFC_ADD` of the OpenFlow protocol. To truly "add only if not exists", use `OFPFC_ADD` with the `check_overlap` flag or query first. Production practice: always `replace-flows` with the full file, never `add-flows` incrementally, to keep intent explicit and the diff visible.

## 4.7.8 Advanced tutorial: 5-table MAC learning (Compass Ch 8)

### Context

The OVS documentation has a canonical tutorial `docs.openvswitch.org/en/latest/tutorials/ovs-advanced/` that builds a VLAN-aware MAC-learning switch through five stages. This is the most pedagogically valuable exercise in the OVS curriculum because it combines all primitives: multi-table, registers, learn action, conntrack.

### 5-stage structure

| Table | Purpose | Primitives used |
|-------|---------|-----------------|
| 0 | Admission control (block unauthorized MAC) | Match plus drop or goto |
| 10 | VLAN input handling (assign VLAN) | mod_vlan_vid, goto |
| 20 | MAC plus VLAN learning | `learn(...)` installs a dynamic flow in Table 30 |
| 30 | Destination lookup (learned MAC, port) | Match dl_dst plus VLAN, output |
| 40 | Unknown destination flooding | `FLOOD` restricted by VLAN |

### `learn(...)` action mechanics

When a packet arrives at Table 20, the `learn` action installs a new flow in Table 30:

```bash
ovs-ofctl -O OpenFlow13 add-flow br0 \
  "table=20,action=learn(table=30,NXM_OF_VLAN_TCI[0..11],NXM_OF_ETH_DST[]=NXM_OF_ETH_SRC[],\
   load:NXM_OF_IN_PORT[]->NXM_NX_REG0[]),output:NXM_NX_REG0[]"
# Not exact production syntax, simplified for illustration
```

Logic: the flow learns the source MAC of the current packet and the ingress port, and installs a flow at Table 30 matching that MAC (when it becomes a destination in the future) with an action that outputs to the learned port.

### Going deeper

The full 5-table tutorial in OVS docs walks through each stage with verification commands, expected output, and troubleshooting common mistakes. It is about 30 pages long and requires Mininet plus OVS setup. Recommended exercise after completing recipes 4.7.4 and 4.7.5. Cross-reference URL in the references.

### Guided Exercise 1: multi-table L3 routing pipeline

**Outcomes:**
- Verify per-table packet count in the 3-stage pipeline.
- Observe reg0 propagation from Table 1 to Table 2.

**Before You Begin:**
Mininet or two netns with veth pair to bridge s1 (OVS). Host A at 192.168.1.10/24, host B at 192.168.2.20/24. Gateway MAC mocked (hard-coded in the flow).

**Instructions:**

**1.** Install the 3-stage pipeline (flows from 4.7.4).

**2.** Send ICMP from A to B:

```bash
[host-a]$ ping -c 5 192.168.2.20
```

**3.** Verify the flow hit counter for each table:

```bash
[ovs]$ ovs-ofctl -O OpenFlow13 dump-flows s1 table=0
# Expected: arp flow and ip flow with n_packets greater than 0
[ovs]$ ovs-ofctl -O OpenFlow13 dump-flows s1 table=1
# Expected: nw_dst=192.168.2.0/24 flow with n_packets equal to 5 (ICMP request)
[ovs]$ ovs-ofctl -O OpenFlow13 dump-flows s1 table=2
# Expected: reg0=2 flow with n_packets equal to 5
```

**Finish:** A total of 5 hits across the 3 tables proves the packet traverses the pipeline by the correct logic. If Table 1 has 0 hits, Table 0 goto_table is not working; if Table 2 has 0 hits, reg0 was not set correctly.

### Guided Exercise 2: conntrack firewall recipe

**Outcomes:**
- Verify A to B connection initiates successfully.
- Verify B to A connection is blocked.
- Observe conntrack state through `dpctl/dump-conntrack`.

**Before You Begin:**
Two netns `tenant-l` (with veth `veth_l0`) and `tenant-r` (with veth `veth_r0`), both attached to bridge br0. Netcat listening in tenant-r.

**Instructions:**

**1.** Install the 5-flow sequence from 4.7.5.

**2.** Test A initiating:

```bash
[tenant-l]$ nc -w 3 <tenant-r-ip> 1234
# Expected: connect successfully
```

**3.** Verify conntrack state:

```bash
[host]$ ovs-appctl dpctl/dump-conntrack
# Expected entry: tcp,orig=(src=...,dst=...,sport=...,dport=1234),state=est,...
```

**4.** Test B initiating:

```bash
[tenant-r]$ nc -w 3 <tenant-l-ip> 5678
# Expected: connect refused or timeout
```

**5.** Observe flow hit counters:

```bash
[host]$ ovs-ofctl -O OpenFlow13 dump-flows br0 | grep "ct_state"
# Flow 1 plus 2: n_packets greater than 0 (A outbound)
# Flow 3 plus 4: n_packets greater than 0 (return traffic)
# Drop flow (priority=0): n_packets greater than 0 from B attempts
```

**Finish:** the recipe proves conntrack tracks state correctly: A-initiated traffic passes, B-initiated traffic is blocked. The drop counter incrementing on each B attempt is evidence.

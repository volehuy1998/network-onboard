## Why this lab exists

A network engineer who learned L2 forwarding on a Cisco or Juniper hardware switch carries a mental model that says: a switch is a piece of silicon with a MAC table; on receiving a frame, it learns the source MAC against the ingress port, looks up the destination MAC, and either forwards out one port or floods out every port except the input. The MAC table ages out idle entries on a fixed timer, typically 300 seconds.

When that same engineer first meets Open vSwitch on a Linux host, the natural question is: "is this just a fancier Linux bridge?" The kernel `bridge` driver has been part of Linux since the 1990s and implements exactly the L2 learning behaviour above. Open vSwitch is a userspace daemon controlling a separate kernel datapath. Both can plug a virtual machine into a network. From the workload's point of view, the question is whether they behave the same way.

This lab answers the question by running the two implementations through identical exercises, back to back, on the same host. Phase A builds an OVS bridge `br0`, attaches two veth pairs whose far ends live in `ns_red` and `ns_blue`, sends an ICMP ping across, and reads the MAC table with `ovs-appctl fdb/show`. Phase B builds a Linux bridge `lbr0`, attaches two veth pairs whose far ends live in `ns_l1` and `ns_l2`, sends an identical ping, and reads the MAC table with `bridge fdb show`. The conclusion is at the end of this file; the evidence is in between.

The L2 learning behaviour the reader expects from CCNA carries over to both implementations unchanged. Where the two diverge is in three places: where the forwarding database lives (OVS keeps it in `ovs-vswitchd` userspace memory; the Linux bridge keeps it in the kernel), what entries the database carries by default (the kernel bridge auto-installs multicast and per-port-VLAN-filtering entries; OVS installs nothing it has not learned), and how aging is configured (the OVS knob is `other_config:mac-aging-time` in seconds; the Linux bridge knob is `ageing_time` in centiseconds).

## Phase A, OVS L2 learning

The phase-A topology is the canonical introductory layout: one OVS bridge with two attached veth pairs, the far ends of which carry IP addresses inside two network namespaces. Each namespace has exactly one interface and one IP, so there is no routing decision to confuse the lesson; everything that happens between `ns_red` and `ns_blue` is pure L2.

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-br br0
root@lab-openvswitch:~/network-onboard# ip netns add ns_red
root@lab-openvswitch:~/network-onboard# ip netns add ns_blue
root@lab-openvswitch:~/network-onboard# ip link add veth_red type veth peer name veth_red_br
root@lab-openvswitch:~/network-onboard# ip link add veth_blue type veth peer name veth_blue_br
root@lab-openvswitch:~/network-onboard# ip link set veth_red netns ns_red
root@lab-openvswitch:~/network-onboard# ip link set veth_blue netns ns_blue
root@lab-openvswitch:~/network-onboard# ip link set veth_red_br up
root@lab-openvswitch:~/network-onboard# ip link set veth_blue_br up
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br0 veth_red_br
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br0 veth_blue_br
root@lab-openvswitch:~/network-onboard# ip netns exec ns_red ip link set lo up
root@lab-openvswitch:~/network-onboard# ip netns exec ns_red ip link set veth_red up
root@lab-openvswitch:~/network-onboard# ip netns exec ns_red ip addr add 10.0.0.1/24 dev veth_red
root@lab-openvswitch:~/network-onboard# ip netns exec ns_blue ip link set lo up
root@lab-openvswitch:~/network-onboard# ip netns exec ns_blue ip link set veth_blue up
root@lab-openvswitch:~/network-onboard# ip netns exec ns_blue ip addr add 10.0.0.2/24 dev veth_blue
```

Now the first observation, before any explicit ping. Read the MAC-learning table on `br0` and find out what the bridge has already seen:

```text
root@lab-openvswitch:~/network-onboard# ovs-appctl fdb/show br0
 port  VLAN  MAC                Age
    1     0  56:a6:87:c1:15:9c    2
    2     0  e2:10:43:fa:d2:bd    1
```

Two entries, neither of them produced by user traffic. The MAC `56:a6:87:c1:15:9c` is the kernel-assigned address of `veth_red` inside `ns_red`; `e2:10:43:fa:d2:bd` belongs to `veth_blue` inside `ns_blue` (the two ends of a veth pair have different MACs, and the bridge sees the MAC of whichever end transmitted; this fact is proved at byte level in [`v3.13-R2-addendum-mac-port-binding.md`](v3.13-R2-addendum-mac-port-binding.md)). The bridge has these MACs because the moment a veth interface is brought up and given an IP, the kernel emits link-local IPv6 packets (NDP solicitations, MLDv2 reports). The frames carry the interface's MAC as source; OVS's L2 learner records the binding. No user-level traffic is needed for the FDB to populate.

Now send the ping that R2.5 sub-task 1 specifies. The expected outcome is that ICMP succeeds, but the value of running it is to watch the FDB age counters reset to zero, which proves the entries are being refreshed by user traffic and not just held statically.

```text
root@lab-openvswitch:~/network-onboard# ip netns exec ns_red ping -c 3 10.0.0.2 2>&1 | tail -5
64 bytes from 10.0.0.2: icmp_seq=3 ttl=64 time=0.038 ms
 
--- 10.0.0.2 ping statistics ---
3 packets transmitted, 3 received, 0% packet loss, time 2057ms
rtt min/avg/max/mdev = 0.038/0.211/0.549/0.238 ms
root@lab-openvswitch:~/network-onboard# ovs-appctl fdb/show br0
 port  VLAN  MAC                Age
    2     0  e2:10:43:fa:d2:bd    0
    1     0  56:a6:87:c1:15:9c    0
```

3 of 3 ICMP echo replies, and both FDB rows are now `Age 0`. The first packet's RTT is 0.549 ms while the third is 0.038 ms; the order-of-magnitude drop after the first packet is the megaflow-cache effect. The first packet missed the kernel datapath cache, was upcalled to userspace for translation through `xlate_normal`, and only then forwarded; the next two packets hit the cached megaflow and bypassed userspace entirely. The MAC table is the visible side of that translation: `xlate_normal` calls `mac_learning_insert` for the source MAC and `mac_learning_lookup` for the destination MAC, so the act of forwarding the first packet writes both FDB rows.

The next exercise tests the aging behaviour. Open vSwitch supports a per-bridge configuration knob `other_config:mac-aging-time`, expressed in seconds. Set it to ten seconds, send a single ping to refresh the entries, then wait twelve seconds and read the table again. The textbook expectation is that twelve seconds beats ten seconds and the entries should be gone.

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl set bridge br0 other_config:mac-aging-time=10
root@lab-openvswitch:~/network-onboard# ip netns exec ns_red ping -c 1 10.0.0.2 2>&1 | tail -3
--- 10.0.0.2 ping statistics ---
1 packets transmitted, 1 received, 0% packet loss, time 0ms
rtt min/avg/max/mdev = 0.033/0.033/0.033/0.000 ms
root@lab-openvswitch:~/network-onboard# ovs-appctl fdb/show br0
 port  VLAN  MAC                Age
    2     0  e2:10:43:fa:d2:bd    1
    1     0  56:a6:87:c1:15:9c    0
root@lab-openvswitch:~/network-onboard# sleep 12
root@lab-openvswitch:~/network-onboard# ovs-appctl fdb/show br0
 port  VLAN  MAC                Age
    1     0  56:a6:87:c1:15:9c   10
    2     0  e2:10:43:fa:d2:bd    7
```

The textbook expectation is wrong. After twelve seconds of silence, both entries are still present, with ages 10 and 7. The eviction does not happen at exactly the timeout boundary; it happens during a periodic sweep inside `mac_learning_run`, and the sweep runs only every few seconds. An entry whose age has just crossed the timeout will survive until the next sweep tick. For an operator deploying OVS in a security-sensitive context (for example, a multi-tenant overlay where stale entries could route a frame to the wrong tenant), this is the lesson worth pinning: `mac-aging-time` is the upper bound, not the exact eviction time. The actual eviction happens at the next sweep on or after the timeout, which can be a few seconds late.

Phase A ends with a clean teardown so phase B starts from the same near-baseline state.

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl del-br br0
root@lab-openvswitch:~/network-onboard# ip netns del ns_red
root@lab-openvswitch:~/network-onboard# ip netns del ns_blue
```

## Phase B, Linux-bridge equivalent

The Linux kernel has shipped a bridge driver since version 2.4.18 (2002). It is the same driver that powers `brctl` (deprecated) and the modern `ip link add ... type bridge`. The bridge implements 802.1D learning identically to a hardware switch: source MACs go in, destination lookups happen, unknown destinations are flooded, idle entries age out. From the namespace's view it is impossible to tell whether the wire on the other side of the veth pair runs into an OVS bridge or a Linux bridge.

The phase-B topology mirrors phase A with different names: the bridge is `lbr0`, the namespaces are `ns_l1` and `ns_l2`, and the IPs are `10.0.0.3` and `10.0.0.4`. Different names so phase A's debris cannot leak in; same subnet so the lesson is the same.

```text
root@lab-openvswitch:~/network-onboard# ip link add name lbr0 type bridge
root@lab-openvswitch:~/network-onboard# ip link set lbr0 up
root@lab-openvswitch:~/network-onboard# ip netns add ns_l1
root@lab-openvswitch:~/network-onboard# ip netns add ns_l2
root@lab-openvswitch:~/network-onboard# ip link add veth_l1 type veth peer name veth_l1_br
root@lab-openvswitch:~/network-onboard# ip link add veth_l2 type veth peer name veth_l2_br
root@lab-openvswitch:~/network-onboard# ip link set veth_l1 netns ns_l1
root@lab-openvswitch:~/network-onboard# ip link set veth_l2 netns ns_l2
root@lab-openvswitch:~/network-onboard# ip link set veth_l1_br master lbr0
root@lab-openvswitch:~/network-onboard# ip link set veth_l2_br master lbr0
root@lab-openvswitch:~/network-onboard# ip link set veth_l1_br up
root@lab-openvswitch:~/network-onboard# ip link set veth_l2_br up
root@lab-openvswitch:~/network-onboard# ip netns exec ns_l1 ip link set lo up
root@lab-openvswitch:~/network-onboard# ip netns exec ns_l1 ip link set veth_l1 up
root@lab-openvswitch:~/network-onboard# ip netns exec ns_l1 ip addr add 10.0.0.3/24 dev veth_l1
root@lab-openvswitch:~/network-onboard# ip netns exec ns_l2 ip link set lo up
root@lab-openvswitch:~/network-onboard# ip netns exec ns_l2 ip link set veth_l2 up
root@lab-openvswitch:~/network-onboard# ip netns exec ns_l2 ip addr add 10.0.0.4/24 dev veth_l2
```

The OVS attach command was `ovs-vsctl add-port`. The Linux equivalent is `ip link set ... master lbr0`, which sets the bridge interface as the master of the veth's bridge-side end. The result is the same: a frame received on the bridge-side end is delivered to the bridge-driver code in the kernel, which runs the L2 learning algorithm and decides where to forward.

Nineteen rows, none of which exist on the OVS bridge for the same topology. Most are housekeeping the kernel adds for its own correctness. This is the first clear divergence from OVS. The OVS FDB shown by `ovs-appctl fdb/show` reports only learned unicast entries and is therefore terse. The kernel-bridge FDB reports learned unicast entries plus every multicast and VLAN-filtering shadow the kernel maintains for forwarding correctness, and is therefore noisy. An operator who switches between the two tools must mentally filter the kernel FDB to find the dynamic learning content.

## What R2.5.1 establishes for the comparative

Three findings are worth pinning at the end of this lab.

The first finding is the data-plane equivalence. From the workload's view (the namespace running the ping), an OVS bridge and a Linux bridge are interchangeable. The ICMP succeeded the same number of times in the same time window with the same RTT distribution. The workload has no way to tell which bridge driver is below it. This is the answer to the question a Cisco-trained engineer asks first: yes, OVS gives you the same L2 forwarding behaviour as the kernel bridge, and yes, both give you the same behaviour as a hardware switch. The L2 learning algorithm is the same algorithm in all three places.

The second finding is the FDB representation difference. `ovs-appctl fdb/show br0` after the ping reported two rows, both of which were learned unicast entries. `bridge fdb show br lbr0` after the equivalent ping reported nineteen rows, of which only three were learned unicast entries; the other sixteen were the kernel's own permanent-entry housekeeping for IPv4 and IPv6 multicast plus VLAN-1 filtering shadows. Operators reading the two outputs side by side must mentally filter the kernel FDB to find the learning content. The OVS output is closer to the textbook abstraction "list the MAC table of the switch"; the kernel output reflects the implementation-level reality "list every entry the bridge driver currently holds, including the operationally invisible ones". Neither is wrong, but they answer slightly different questions.

The third finding is the aging-precision behaviour. Both implementations age out idle entries via a periodic sweep, not via a precise per-entry timer. With the aging time set to ten seconds and a wait of twelve seconds, both implementations left the entries in the table past the configured timeout. The OVS-side ages reported `10` and `7`; the Linux-bridge-side entries simply persisted. The lesson for an operator: aging time is the upper bound on the age of an entry, not the exact eviction time. A multi-tenant overlay that needs sharper eviction must engineer it externally (for example, by issuing explicit `ovs-appctl fdb/flush` or `bridge fdb del` on tenant teardown) rather than relying on the timer to be precise.

The R2.5 sprint continues with sub-task 2 (VLAN trunking and access ports), where the divergence between OVS and the Linux bridge is sharper: OVS treats every port as a trunk by default and configures access semantics through the `Port.tag` column, while the Linux bridge requires `vlan_filtering` to be turned on at the bridge level and `bridge vlan add` to be run per port.

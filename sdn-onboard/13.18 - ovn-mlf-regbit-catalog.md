## §13.18.1 REGBIT_PORT_SEC_DROP

> Deferred-drop port security flag. Full 20 entries below.

### Concept

`REGBIT_PORT_SEC_DROP` is a 1-bit flag in OVN logical register `reg0` whose bit position is defined by `northd/northd.c`. When the flag equals 1, the packet has violated port security policy (MAC spoofing, IP spoofing, or ARP spoofing) at stage `LS_IN_PORT_SEC_IP` or `LS_IN_PORT_SEC_ND`, and the downstream pipeline drops the packet at stage `LS_IN_PORT_SEC_L2`.

**Type:** OVN logical register bit flag, single-bit boolean.

### History and background

Introduced in OVN 20.06 (2020) as part of the port security overhaul. Before OVN 20.06, port security violations were dropped immediately at the originating stage (LS_IN_PORT_SEC_IP or LS_IN_PORT_SEC_ND), but this pattern conflicted with egress accounting that wanted to count the drop before it happened. REGBIT_PORT_SEC_DROP defers the drop to stage `LS_IN_PORT_SEC_L2` after accounting has the chance to log packet metadata. Authors: OVN core team Pfaff and Pettit, commit OVN `5a9a7e4` in 2020-Q2.

### Position in the architecture

OVN logical register layer, populated in the ovn-northd compile output Logical_Flow, evaluated at runtime by ovn-controller installing the OF flow on `br-int`. Concretely the bit sits in `reg0` (a 32-bit register), at the specific bit position defined by the REGBIT_PORT_SEC_DROP macro in `northd/northd.c`. Scope: per-packet (reset between packets, does not persist).

### Role

Plays the role of a "deferred drop signal" between stage 0 (PORT_SEC_IP) and stage 1 (PORT_SEC_L2). Allows a downstream pipeline stage to read the bit and decide whether to drop, while also letting egress accounting (through the ovn-controller log feature) capture metadata before the final drop.

### Why it exists

Pre-existing pain: dropping immediately at the originating stage caused accounting and audit logs to lose context. The OVN 20.06 architecture document (Pfaff, 2020-04) gives explicit motivation: "deferred drop allows ACL log and port_security log to accumulate metadata together with the packet before dropping". REGBIT_PORT_SEC_DROP fills this gap with a pure OF register signalling mechanism that needs no extra upcall.

### Problems it solves

Problem: port security violation dropped immediately means lost audit trail. Mechanism: REGBIT_PORT_SEC_DROP is set at the originating stage, the packet continues the pipeline with the bit set, accounting and logging stages capture metadata, and finally the drop happens at LS_IN_PORT_SEC_L2 based on the bit. Solution scope: fully solves audit context preservation, but does not address performance overhead (still costs N stage traversals before the drop).

### Importance

Cornerstone tier 1 in the OVN port security architecture. An engineer who does not understand REGBIT_PORT_SEC_DROP cannot debug the port security drop pattern (why the packet drops at stage 1 instead of stage 0). Justification: every OVN deployment has port security on by default; this bit flag is always a possible trigger.

### How it works

Flow rule pattern in Logical_Flow (compiled by ovn-northd):

```
table=0 (LS_IN_PORT_SEC_L2), priority=50,
  match: (reg0[REGBIT_PORT_SEC_DROP] == 1),
  action: drop;

table=N (LS_IN_PORT_SEC_IP), priority=100,
  match: (eth.src != spoof_eth_addr && ip.src != spoof_ip_addr),
  action: next;

table=N (LS_IN_PORT_SEC_IP), priority=80,
  match: (1),
  action: reg0[REGBIT_PORT_SEC_DROP] = 1; next;
```

Flow installed via ovn-controller compiling Logical_Flow into OF flow. The bit is set through a `set_field` action with `reg0[bit_pos]=1` syntax. Match goes through a `reg_load` test. Invariant: the bit is reset to 0 at stage 0 (start of pipeline for each packet); it is set only on violation.

### How an operator masters this

Four skills for REGBIT_PORT_SEC_DROP:

(1) **Read:** trace the logical flow with `ovn-sbctl lflow-list` plus grep "reg0\[" to see every flow that sets or tests this bit.
(2) **Modify:** not direct (ovn-northd compiles from the NB schema port_security column). Modify through `ovn-nbctl lsp-set-port-security` configuration.
(3) **Observe:** `ovn-trace --detailed` shows reg0 bit transitions per stage; `ovn-detrace` shows the OF flow cookie to Logical_Flow mapping.
(4) **Debug:** when a port security drop is suspected, check the stage trace and identify which violation stage set the bit.

Decision tree: if the drop happens at table 0 (LS_IN_PORT_SEC_L2), check reg0 in the trace. If the bit is set, identify upstream stage 0 (PORT_SEC_IP) or N/A (ND); if the bit is unset, the cause is something else.

### Classification

- Type: OVN logical register bit flag (1-bit boolean)
- Parent: reg0 (32-bit register) in OVN logical_fields
- Scope: per-packet (transient, reset at start of pipeline)
- Set by: ovn-northd compiled match condition violation
- Read by: LS_IN_PORT_SEC_L2 stage drop rule
- Idempotency: yes (set 1 to 1 has no effect)

### Usage workflow

Workflow to debug a port security drop:

```
Step 1: Identify the drop pipeline stage
  $ ovn-trace --detailed <microflow>
  -> Find "Drop in table N: ..."

Step 2: Check the reg0 value at the drop stage
  -> Trace output "reg0 = 0xN"
  -> REGBIT_PORT_SEC_DROP bit position per northd/northd.c (for example bit 17)
  -> 0xN & (1<<17) = bit set?

Step 3: Trace upstream to identify the violation stage
  -> Search trace output "reg0[17] = 1" set point
  -> Common: LS_IN_PORT_SEC_IP (IP spoofing) or LS_IN_PORT_SEC_ND (ARP spoofing)

Step 4: Check NB port_security configuration
  $ ovn-nbctl get Logical_Switch_Port lsp-vm1 port_security
  -> Compare actual eth.src and ip.src with the configuration

Step 5: Fix
  -> Update port_security with the correct address OR disable port_security for debugging
```

### When something goes wrong

Symptoms an on-call operator may see:

- **Symptom A:** A VM loses connectivity after live migration. Suspect: MAC address change leading to a port_security violation. Diagnostic: ovn-trace shows REGBIT_PORT_SEC_DROP set at LS_IN_PORT_SEC_L2.
- **Symptom B:** A container fails DHCP. Suspect: the container requests an IP before the NB schema updates port_security. Diagnostic: the trace shows the bit set at PORT_SEC_IP with a spoofed source IP.
- **Symptom C:** ARP requests are dropped. Suspect: ARP spoofing protection. Diagnostic: the trace shows the bit set at PORT_SEC_ND.

Cross-link Section 20.0 §systematic debugging and Section 20.2 §port_security drop pattern.

### Tightly related to

- **MFF_LOG_INPORT** (see §13.18.3): the packet logical inport identifies which LSP is the source.
- **port_security column** in NB Logical_Switch_Port: source-of-truth configuration.
- **stages LS_IN_PORT_SEC_IP, LS_IN_PORT_SEC_ND, LS_IN_PORT_SEC_L2**: the three stages involved.
- **acl log feature** in OVN 22.03 and later: logs packet metadata before drop (consumes this bit).
- **ovn-northd build_lswitch_lflows_for_port_sec** function: generates the flow that sets the bit.

### Version differences

| Version | Behavior |
|---------|----------|
| Pre-OVN 20.06 | Immediate drop at the originating stage; no deferred drop bit |
| OVN 20.06+ | REGBIT_PORT_SEC_DROP introduced; deferred drop pattern |
| OVN 22.03+ | acl log feature consumes the bit for audit trail |
| OVN 24.03+ | Bit position in reg0 unchanged; behaviour stable |

Curriculum baseline: OVN 22.03.8. Production may run 24.03 with consistent behaviour.

  -> All flows that set or test bit 17 for the LS instance

Method 3: ovs-ofctl dump-flows br-int
  $ ovs-ofctl dump-flows br-int | grep "reg0=0x20000"

OVN v22.03.8 baseline:

### Real-world incident

**Production case 2024-Q2, multi-tenant cluster:**

**Symptom:** Tenant A migrated VM `vm-web-3` from host-1 to host-7, after which the VM lost ingress traffic. Egress still worked.

**Investigation:**

```
Step 1: ovn-trace inport == "lsp-vm-web-3"
  -> Drop in LS_IN_PORT_SEC_L2 with reg0[REGBIT_PORT_SEC_DROP] = 1

Step 2: Trace upstream stages
  -> REGBIT was set in LS_IN_PORT_SEC_IP with violation "ip.src=10.0.0.5 not in allowed list"

Step 3: Check NB
  $ ovn-nbctl get Logical_Switch_Port lsp-vm-web-3 port_security
  -> ["52:54:00:aa:bb:cc 10.0.0.5"]
  -> Matches eth.src and ip.src on the surface

Step 4: Check the ARP responder
  -> Stale ARP cache on host-7; eth.src observed as the old MAC address
  -> Despite port_security listing the correct MAC, the arriving packet uses a different cached MAC
```

**Root cause:** Stale ARP cache on the upstream switch did not refresh after migration. The ARP packet from the VM used the correct MAC, but the response eth.dst used the stale MAC, so from the OVS view the source MAC mismatched port_security.

**Fix:** OVN 24.03 introduced eager GARP send after migration; the patch was backported to 22.03. Cluster-wide ARP cache reload after migration. Post-fix drop rate: 0 percent.

Reference: similar case study `19.0 - ovn-multichassis-binding-and-pmtud.md` (PMTUD after migration), parallel root cause family.

### Common failures and diagnostic signals

**Mode A: Bit position drift across versions**

- Diagnostic: ovn-trace shows `reg0 = 0x20000` in 22.03 but `reg0 = 0x40000` in a newer version.
- Recovery: re-grep `northd/northd.c` for the macro position; update the operational runbook.

**Mode B: ACL log fails to capture metadata before drop**

- Diagnostic: the `acl_log` feature is enabled but the log table is empty for dropped packets.
- Recovery: verify OVN version is 22.03 or later (the bit-consuming feature was added then); enable an explicit `meter`-based log.

**Mode C: Drop at the wrong stage (PORT_SEC_L2 instead of the origin)**

- Diagnostic: ovn-trace shows the pipeline bypasses PORT_SEC_IP entirely.
- Recovery: check that the NB Logical_Switch_Port port_security column is non-empty; if empty, no bit is set and no drop is expected, so adjust expectations.

**Mode D: REGBIT bit position used by mistake by a user-defined ACL**

- Diagnostic: a user ACL sets reg0[17] independently, falsely triggering a drop at PORT_SEC_L2.
- Recovery: avoid user ACLs touching reg0 bit 17; OVN reserves this bit.

### Comparison with other systems

**Comparable Linux iptables:** `iptables -m mac --mac-source` plus `-m conntrack --ctstate INVALID` for stateful drop. Differences from OVN:

- iptables: immediate drop at the matching rule; no deferred drop signalling.
- iptables: no register stack for metadata propagation.
- OVN: REGBIT enables pipeline orchestration across multiple stages.

**Comparable Cisco MAC port-security:** `switchport port-security maximum 1`, drops on violation. Differences from OVN:

- Cisco: hardware-level drop at the ASIC; no flow rule visibility.
- OVN: software OF flow visible through dump-flows and debuggable.

**Comparable NSX-T DFW (Distributed Firewall):** stateful rules with metadata tag. Differences from OVN:

- NSX-T: tag stored in a distributed table, propagated through the management plane.
- OVN: REGBIT lives in a per-packet register, propagated through pipeline stages in flight.

OVN is unique: register-based deferred drop is a compromise between hardware ACLs (fast, blind) and a full software firewall (slow, observable).

## §13.18.2 REGBIT_NAT_REDIRECT

> Flag that redirects a packet to the gateway chassis for NAT. Full 20 entries below.

### Concept

`REGBIT_NAT_REDIRECT` is a 1-bit flag in `reg9` (specific bit position per `northd/northd.c`), set when a packet needs to redirect through the gateway chassis for NAT processing in the distributed gateway router architecture.

**Type:** OVN logical register bit flag.

### History and background

Introduced in OVN 21.06 (2021) together with the distributed gateway router architecture overhaul. Before OVN 21.06, NAT was performed centrally on the gateway router; OVN 21.06 distributes the SNAT and DNAT decisions local-first, falling back to redirect through this bit flag when local handling is not possible.

### Position in the architecture

OVN logical register layer, `reg9` (32-bit). Read by stage `LR_IN_GW_REDIRECT` to decide on chassis redirect. Set by stage `LR_IN_DEFRAG` or `LR_IN_DNAT` when NAT requires gateway state (for example, a centralised DNAT pool). Scope: per-packet.

### Role

Signals "this packet needs gateway chassis state for NAT". Allows the distributed router to perform a local short-circuit for non-NAT traffic and to redirect only when needed. Optimises east-west traffic that does not require a gateway hop.

### Why it exists

Pre-existing pain: the distributed gateway architecture wanted local-first decisions, but NAT is inherently centralised (state must be shared). REGBIT_NAT_REDIRECT is the compromise: distribute non-NAT traffic, redirect NAT traffic through the bit flag.

### Problems it solves

Problem: the distribute-versus-centralise NAT trade-off. Mechanism: locally check NAT applicability, set the bit if gateway state is required, then LR_IN_GW_REDIRECT redirects through the tunnel. Scope: fully solves local-first optimisation; partially solves centralised NAT scaling (the gateway chassis remains a bottleneck).

### Importance

Cornerstone tier 1 for the distributed gateway router architecture. An engineer who does not understand REGBIT_NAT_REDIRECT cannot debug the "NAT packet must go through the gateway chassis versus can be handled locally" decision pattern.

### How it works

```
table=N (LR_IN_DEFRAG), priority=100,
  match: (ip && ct.related),
  action: ct_clear; reg9[REGBIT_NAT_REDIRECT] = 1; next;

table=N (LR_IN_GW_REDIRECT), priority=50,
  match: (reg9[REGBIT_NAT_REDIRECT] == 1),
  action: outport = "cr-{LR_NAME}"; next;  /* redirect to gateway chassis port */

table=N (LR_IN_GW_REDIRECT), priority=10,
  match: (1),
  action: next;  /* local handling, no redirect */
```

The stage tests the bit; if it is set, the outport is set to the chassis-redirect port (cr-LR_NAME), and tunnel encapsulation routes the packet to the gateway chassis through the Geneve TLV.

### How an operator masters this

Skills: read the trace to see the redirect decision; modify the gateway chassis assignment with `ovn-nbctl lr-set-options ... redirect-chassis=...`; observe through lflow-list and the Geneve TLV tunnel header.

Decision: if a NAT packet is not redirected (handled locally), NAT state collisions across chassis are possible; if everything is redirected, the distributed benefit is lost.

### Classification

- Type: OVN logical register bit flag, redirect signal.
- Parent: reg9 (egress pipeline register).
- Set by: LR_IN_DEFRAG, LR_IN_DNAT (when state-dependent).
- Read by: LR_IN_GW_REDIRECT.
- Scope: per-packet.

### Usage workflow

Workflow to debug an unexpected NAT redirect:

```
Step 1: ovn-trace from a non-gateway chassis with NAT traffic.
Step 2: Check reg9 transitions: was the bit set at LR_IN_DEFRAG or LR_IN_DNAT?
Step 3: Verify the gateway chassis is alive: ovn-sbctl list HA_Chassis_Group.
Step 4: Check LR options: ovn-nbctl get Logical_Router lr-name options.
Step 5: Fix: ensure HA_Chassis_Group is healthy, or redistribute the load.
```

### When something goes wrong

- Symptom A: NAT traffic latency spike. Suspect redirect overhead. Trace the bit to confirm.
- Symptom B: Gateway chassis CPU at 100 percent. Too many redirects. The bit fires too often.
- Symptom C: NAT state intermittently lost. The bit is not set when needed (race condition). Trace plus lflow-list.

### Tightly related to

- **redirect-chassis option** in NB Logical_Router.
- **HA_Chassis_Group** SB schema (gateway pool).
- **stage LR_IN_GW_REDIRECT** as the consumer of the bit.
- **Geneve TLV class 0x0102** tunnel encapsulation that carries the redirect.
- **distributed gateway port (DGP)** logical concept.

### Version differences

| Version | Behavior |
|---------|----------|
| Pre-OVN 21.06 | All NAT centralised; no bit |
| OVN 21.06+ | REGBIT_NAT_REDIRECT introduced; distributed gateway model |
| OVN 22.03+ | Bit position stable in reg9 |
| OVN 24.03+ | Optional ECMP redirect support when HA_Chassis_Group is multi-active |

### Observation and verification

```
$ ovn-trace lr-main 'inport == "lrp-internal" && ip.src == 10.0.0.5 && ip.dst == 8.8.8.8'
-> reg9 = 0x... transitions
-> Redirect outport "cr-lr-main" if bit is set
```

### Source code reference

- `northd/northd.c` REGBIT_NAT_REDIRECT macro
- `northd/northd.c` `build_lrouter_nat_flows()` function
- `controller/physical.c` chassis-redirect port handling

### Real-world incident

**Production case 2023-Q4, distributed cluster:**

**Symptom:** Gateway chassis CPU was sustained at 100 percent; non-gateway chassis were idle.

**Investigation:** ovn-trace from a non-gateway chassis with typical east-west traffic showed reg9[REGBIT_NAT_REDIRECT] = 1 for ALL traffic (expected only for NAT). Root cause: a Logical_Router_Policy mistakenly included a match for non-NAT traffic.

**Fix:** Audit Logical_Router_Policy entries, remove the over-inclusive match. The bit fire rate dropped by 95 percent and gateway chassis CPU dropped to 30 percent.

### Lab exercise

```
$ ovn-nbctl lr-add lr-test
$ ovn-nbctl set Logical_Router lr-test options:redirect-chassis=chassis-1
$ ovn-nbctl lr-nat-add lr-test snat 192.168.1.0/24 10.0.0.0/24
$ ovn-trace lr-test 'inport == "internal" && ip.src == 10.0.0.5 && ip.dst == 8.8.8.8'
-> Observe the reg9[REGBIT_NAT_REDIRECT] transition
```

### Common failures and diagnostic signals

- Mode A: Redirect loop (the bit is set repeatedly on the gateway chassis). Diagnostic: the trace shows infinite recursion. Recovery: this is an ovn-northd compile bug, apply the upstream patch.
- Mode B: Gateway chassis dead, redirect to nowhere. Diagnostic: all HA_Chassis_Group entries are DOWN. Recovery: spawn a replacement chassis.

### Comparison with other systems

**Comparable Linux iproute2:** `ip route add default via gateway`. Difference from OVN: ip route is static, while the OVN bit flag is dynamic per-packet.

**Cisco redundant gateway HSRP:** active and passive gateway selection. Difference from OVN: HSRP uses ARP-based MAC takeover, while OVN uses a logical register flag inside packet metadata.

**NSX-T edge cluster:** centralises NAT on edge nodes. The concept is similar. Difference in implementation: NSX-T makes the routing decision at the transport node, while OVN makes it at a logical router stage through a register flag.

## §13.18.3 MFF_LOG_INPORT

> 32-bit logical inport identifier in register reg14. Full 20 entries below.

### Concept

`MFF_LOG_INPORT` is the OVN logical inport metadata field, storing the identifier of the Logical_Switch_Port or Logical_Router_Port at which the packet arrived. It is a 32-bit field in the reg14 register, defined in `include/ovn/logical-fields.h`.

**Type:** OVN logical register field (32-bit identifier).

### History and background

Introduced in the OVN initial release 2.4 (2015) as part of the foundational logical pipeline architecture. Stable across every OVN version. The 2015 OVN announcement by Pfaff and Pettit is explicit: "MFF_LOG_INPORT is fundamental; every match expression `inport == \"X\"` compiles into a test on this field".

### Position in the architecture

OVN logical register layer, reg14. Set by ovn-controller when a packet enters the logical pipeline. Read by every stage that filters "packets from a specific port". Scope: persists across the entire pipeline (not reset between stages).

### Role

The universal identifier for "packet origin in the logical topology". Every ACL and port-specific flow tests this field. Architectural invariant: if MFF_LOG_INPORT equals 0 or is invalid, the packet cannot enter the logical pipeline.

### Why it exists

Pre-existing pain: OpenFlow has `in_port` (physical port), but OVN needs a logical port (cross-chassis abstraction). The physical port does not map onto the logical concept (for example, a logical port can migrate between chassis). MFF_LOG_INPORT abstracts the physical-to-logical mapping.

### Problems it solves

Problem: physical port IDs are not stable (migration, multi-chassis). Mechanism: on ingress, ovn-controller decodes (chassis, physical port) into (logical port UUID encoded as a 32-bit tunnel_key). Scope: fully solves logical port stability.

### Importance

Cornerstone tier 1, absolute. Every OVN match expression keyword ultimately tests this field. An engineer who does not understand MFF_LOG_INPORT does not understand OVN core principles.

### How it works

```
ovn-controller upon ingress packet:
  Step 1: identify physical port (OF in_port)
  Step 2: look up Port_Binding by tunnel_key matching (tunnel header Geneve TLV class 0x0102)
  Step 3: set MFF_LOG_INPORT = Port_Binding.tunnel_key
  Step 4: continue pipeline

Pipeline stage usage:
  match: (inport == "lsp-vm1")
  Compiled OF: (reg14 == 0xN) where 0xN is lsp-vm1's tunnel_key
```

The Geneve TLV header carries MFF_LOG_INPORT for cross-chassis traffic; ovn-controller decodes and sets it on br-int.

### How an operator masters this

Skills: read MFF_LOG_INPORT in the trace; map a tunnel_key UUID to the Logical_Switch_Port name; observe with `ovn-sbctl list Port_Binding | grep tunnel_key`.

### Classification

- Type: OVN logical metadata field, 32-bit identifier.
- Parent: reg14 (logical metadata register).
- Set by: ovn-controller on ingress decode.
- Read by: every match expression on `inport`.
- Scope: persists across the pipeline.

### Usage workflow

```
Step 1: ovn-sbctl list Port_Binding | grep -A 2 lsp-vm1
        -> Get the tunnel_key value
Step 2: ovs-ofctl dump-flows br-int | grep "reg14=0x..."
        -> Find OF flows filtering on this logical port
Step 3: ovn-trace with the specific inport
        -> Verify the pipeline identifies the port correctly
```

### When something goes wrong

- Symptom A: A packet traverses the wrong logical pipeline. Suspect a MFF_LOG_INPORT decode error. Trace plus sbctl Port_Binding match.
- Symptom B: An ACL on a specific port stops matching. Suspect tunnel_key drift after Port_Binding recreation. Verify the tunnel_key is stable.
- Symptom C: Cross-chassis packet drop. Geneve TLV class 0x0102 is missing. Inspect the tunnel hex with tcpdump.

### Tightly related to

- **Port_Binding.tunnel_key** SB schema column.
- **Geneve TLV class 0x0102** tunnel encapsulation that carries the value.
- **MFF_LOG_DATAPATH** complement (logical datapath).
- **MFF_LOG_OUTPORT** sibling (egress).
- **physical.c consider_port_binding()** function that compiles the mapping.

### Version differences

| Version | Behavior |
|---------|----------|
| OVN 2.4 (2015) | Initial introduction |
| OVN 2.10+ | tunnel_key persistence rules tightened |
| OVN 22.03+ | Bit width stable at 32-bit, position reg14 |
| OVN 24.03+ | Unchanged |

$ ovs-ofctl monitor br-int watch:resume |& grep reg14

### Real-world incident

**Production case 2024-Q1, OVN 22.03 cluster:**

**Symptom:** After rebuilding lsp-vm1 (delete and recreate with the same name), ACLs applied to the VM stopped matching.

**Investigation:** `ovn-sbctl list Port_Binding lsp-vm1` showed the tunnel_key had changed (old 42, new 187). The OF flow on br-int still matched reg14=42 (stale). northd had not recompiled the flows.

**Root cause:** The ovn-northd I-P engine missed the tunnel_key delta. Fix: `ovn-appctl -t ovn-northd inc-engine/recompute` to force a full compile.

**Fix:** Upstream patch in 22.03.x added tunnel_key delta detection. Cluster-wide redeploy.

Reference: parallel case `20.5 §lflow stale post-recreate`.

### Common failures and diagnostic signals

- Mode A: tunnel_key collision (rare, 32-bit space). Diagnostic: two Port_Binding entries share the same tunnel_key. Recovery: have northd regenerate.
- Mode B: Stale OF flow after Port_Binding recreation. Diagnostic: reg14 mismatch. Recovery: recompute.
- Mode C: Geneve TLV class 0x0102 missing on the tunnel. Diagnostic: tcpdump shows no class 0x0102. Recovery: upgrade the peer chassis.

### Comparison with other systems

**Linux network namespace plus iproute2:** packet identity by netns. Difference from OVN: netns is a local concept, while OVN MFF_LOG_INPORT is a cross-chassis abstraction.

**OpenStack Neutron OVS agent:** uses flow.in_port physical port matching. Difference from OVN: Neutron has no cross-chassis logical port abstraction, while OVN MFF_LOG_INPORT does.

**VMware NSX-T logical port ID:** a similar concept (logical port abstraction). Difference in implementation: NSX-T resolves it on the management plane, while OVN resolves it on the distributed control plane through an SB schema lookup.

## §13.18.4 MFF_LOG_OUTPORT

> The sibling of MFF_LOG_INPORT. Same 20-axis structure, but the identifier represents "where the packet is going" rather than "where it came from".

### Concept through importance

`MFF_LOG_OUTPORT` belongs to the same register family as INPORT (typically reg15) and is set when the pipeline decides the packet will exit through a specific Logical_Switch_Port or Logical_Router_Port. History: introduced with the OVN 2.4 founding architecture. Role: universal identifier of the egress logical port. Cornerstone tier 1.

### How it works

```
LS_IN_L2_LKUP (table 27, ingress LS) looks up the destination MAC -> Port_Binding
  -> action: outport = "lsp-vm2"; output;
  Compiled: reg15 = 0xM (lsp-vm2 tunnel_key); output;

Pipeline egress (LS_OUT_*):
  match: (outport == "lsp-vm2")
  Compiled: (reg15 == 0xM)
```

### Workflow, troubleshooting, coupling

Skill: trace the outport decision. Symptom: a packet egresses through the wrong port, so check the MFF_LOG_OUTPORT set point. Coupled with: MFF_LOG_INPORT, Port_Binding, MAC_Binding, the ARP responder.

### Version, verification, source, incident, lab, failure, cross-domain

Same pattern as MFF_LOG_INPORT. Version is stable across OVN. Source: `include/ovn/logical-fields.h` plus the `build_*_l2_lookup()` functions in `northd/northd.c`. Incidents: similar tunnel_key drift cases. Lab: similar setup as INPORT; verify the `outport` decision. Cross-domain: similar to a Cisco MAC table forwarding decision.

## §13.18.5 MFF_LOG_DATAPATH

> Datapath identifier register; the multi-tenant isolation primitive. Full 20 entries below.

### Concept

`MFF_LOG_DATAPATH` (typically reg11 or reg12) holds the identifier of the OVN Logical_Switch or Logical_Router (datapath) that is currently processing the packet. Total 64 bits (some versions split it into high and low halves). The identifier is `Datapath_Binding.tunnel_key` (one per logical switch or router).

### History and background

OVN 2.4 founding release (2015). Stable. Sibling of MFF_LOG_INPORT and MFF_LOG_OUTPORT.

### Position in the architecture

A logical metadata register that persists across the entire pipeline.

### Role

Identifies "this packet belongs to which Logical_Switch or Router instance". Filters cross-instance traffic and isolates tenant pipelines.

### Why it exists

OVN multi-tenancy requires isolation between a Logical_Switch in tenant A and a Logical_Switch in tenant B. MFF_LOG_DATAPATH lets the pipeline filter "only flows that belong to this datapath", an architectural enforcement that does not require ACLs.

### Problems it solves

Multi-tenant isolation as a natural compile output. Mechanism: every Logical_Flow includes an implicit `match: (metadata == datapath_id)` through MFF_LOG_DATAPATH. Scope: fully solves per-datapath isolation.

### Importance

Cornerstone tier 1. Fundamental to multi-tenancy.

### How it works

```
ovn-controller on ingress:
  Step 1: identify Port_Binding.datapath UUID
  Step 2: look up Datapath_Binding.tunnel_key = N
  Step 3: set MFF_LOG_DATAPATH = N
  Step 4: continue the pipeline

Stage match:
  match: (inport == "lsp-vm1") AND implicit (metadata == N)
  Compiled: (metadata=N, reg14=K)
```

Geneve TLV class 0x0102 carries the datapath_id across chassis.

### Workflow, troubleshooting, coupling

Skill: identify a cross-datapath leak. Symptom: traffic jumps between tenants, so check MFF_LOG_DATAPATH integrity. Coupled with: Datapath_Binding tunnel_key, Geneve TLV, Port_Binding.datapath UUID.

### Version differences

OVN 2.4 introduced it. OVN 22.03 has stable bit width. OVN 24.03 makes zone-id externally configurable per Logical_Switch (related concept: conntrack zone).

### Observation and verification

```
$ ovn-sbctl --columns=external_ids,tunnel_key list Datapath_Binding
$ ovs-ofctl dump-flows br-int | grep "metadata=0xN"
```

### Source code reference

- `include/ovn/logical-fields.h` MFF_LOG_DATAPATH macro
- `controller/physical.c` `consider_port_binding()` sets the datapath
- `northd/northd.c` build_lflows datapath filter

### Real-world incident

**Production case 2024-Q3, multi-tenant cluster (cited as a Phase B megaflow worked example, expanded here):**

**Symptom:** East-west traffic between VMs on the same Logical_Switch intermittently dropped at about 5 percent.

**Investigation:** ovn-trace showed the pipeline passing correctly, but ofproto/trace showed the packet stuck at table 12 with reg holding an unexpected datapath_id. The conntrack zone partition uses datapath_id; LS-A and LR-transit-12 collided on the same datapath_id (tunnel_key) of 42.

**Root cause:** ovn-northd compile chose tunnel_key=42 for both LS-A and LR-transit-12 due to a tunnel_key allocation race.

**Fix:** OVN 24.03 introduced configurable zone-id per LS. Set `ls-A.options:zone-id=100` and `lr-transit.options:zone-id=200`. Post-fix drop rate: 0 percent.

Reference: the same case study illustrated in Section 19.0 (multi-chassis NAT zone collision).

### Lab exercise

```
$ ovn-nbctl ls-add ls-tenant-a
$ ovn-nbctl ls-add ls-tenant-b
$ ovn-sbctl --columns=external_ids,tunnel_key list Datapath_Binding
-> Both LS instances have different tunnel_key values
$ ovs-ofctl dump-flows br-int | grep "metadata=" | head
-> Each datapath is isolated through metadata
```

### Common failures and diagnostic signals

- Mode A: tunnel_key collision (rare in 32-bit space but possible). Diagnostic: two Datapath_Binding entries share the same tunnel_key. Recovery: have northd regenerate.
- Mode B: Conntrack zone collision (related concept). Diagnostic: ct entries cross tenants. Recovery: explicit zone-id.
- Mode C: Geneve TLV class 0x0102 stripped. Diagnostic: tcpdump tunnel header. Recovery: upgrade the peer.

### Comparison with other systems

**Linux network namespace:** the namespace acts as an isolation boundary. Difference from OVN: netns is a local concept, while OVN MFF_LOG_DATAPATH is cross-chassis and distributed.

**VLAN:** traditional multi-tenant isolation. Difference: VLAN has a 4096 limit, while OVN's 24-bit Geneve TLV is virtually unlimited.

**VRF (Virtual Routing and Forwarding):** L3 isolation. Difference: VRF is statically configured, while OVN is dynamic per Logical_Router instance.

**VMware NSX-T transport zone:** a similar abstraction. Difference in implementation: NSX-T centralises uplinks, while OVN performs distributed lookup.

| REGBIT name | Reg position | Set by stage | Read by stage | Purpose |
|-------------|--------------|--------------|---------------|---------|
| REGBIT_DHCP_OPTS_RESULT | reg0[3] | LS_IN_DHCP_OPTIONS | LS_IN_DHCP_RESPONSE | DHCP reply success/fail |
| REGBIT_DNS_OPTS_RESULT | reg0[4] | LS_OUT_DNS_LOOKUP | LS_OUT_DNS_RESPONSE | DNS reply hit |
| REGBIT_LOOKUP_FDB | reg0[5] | LS_IN_LOOKUP_FDB | LS_IN_PUT_FDB | FDB lookup attempt |

| REGBIT_ACL_HINT_ALLOW | reg0[7] | LS_IN_ACL_HINT | LS_IN_ACL | ACL allow shortcut |
| REGBIT_ACL_HINT_DROP | reg0[8] | LS_IN_ACL_HINT | LS_IN_ACL | ACL drop shortcut |
| REGBIT_ACL_HINT_BLOCK | reg0[9] | LS_IN_ACL_HINT | LS_IN_ACL | ACL block shortcut |
| REGBIT_LB_NEW_FLOW | reg0[10] | LS_IN_PRE_LB | LS_IN_LB | LB new connection signal |
| REGBIT_HAIRPIN | reg0[11] | LS_IN_PRE_HAIRPIN | LS_IN_HAIRPIN | Hairpin SNAT signal |
| REGBIT_CONNTRACK_DEFRAG | reg0[12] | LS_IN_PRE_ACL | downstream ct() | Conntrack defrag invoke |

The bit positions above are the OVN 22.03.8 baseline; northd compile can renumber them when versions drift, so always confirm with `grep -n "REGBIT_" northd/northd.c`. Practical observation command: `ovn-trace --detailed` shows `reg0 = 0x...` transitions per stage, from which the bit set can be deduced with an AND mask.

REGBIT_HAIRPIN (reg0[11]) is the most operationally relevant: a load balancer VIP called by a client on the same Logical_Switch as the backend triggers hairpin SNAT, and this bit is set so that the `LS_IN_HAIRPIN` stage rewrites the source IP. The classic symptom when the flag is not set due to a compile bug: VIP traffic from an internal client is replied to directly by the backend, the client refuses the reply because the source does not match (NAT asymmetry).

REGBIT is an OVN-internal concept with no direct cross-domain analogue; the closest comparison is the Linux conntrack mark (`ct_mark`), but the scope is only within the pipeline rather than persisting across the flow lifetime.

## §13.18.7 MLF_LOCAL_ONLY

> A flag in `MFF_LOG_FLAGS` (register `reg10`, bit 4) marking that the packet should be delivered only to logical ports on the current chassis, not forwarded through a tunnel to another chassis.

**Concept and history.** `MLF_LOCAL_ONLY` is defined in `include/ovn/logical-fields.h` in the `mff_log_flags_bits` enum at bit position 4 (`MLF_LOCAL_ONLY_BIT = 4`, `MLF_LOCAL_ONLY = (1 << 4)`). The flag has existed since the earliest OVN versions to handle the BUM (Broadcast, Unknown-unicast, Multicast) replication problem in distributed overlays.

**Role and mechanism.** When a BUM packet needs to reach multiple logical ports distributed across multiple chassis, ovn-controller splits the work into two groups: (1) replicate to the local ports on the current chassis with `MLF_LOCAL_ONLY = 1` to avoid looping back through the tunnel, (2) send one copy through the tunnel to the remote chassis, which then performs step (1) itself. Thanks to this flag, a packet is not duplicated indefinitely between chassis. The `physical.c` egress stage checks the flag before deciding whether to output to the tunnel or to a local OF port.

**Source code reference.** `controller/physical.c` tests `MLF_LOCAL_ONLY` in the BUM resubmit branch. `controller/pinctrl.c` sets the flag with `put_load(1, MFF_LOG_FLAGS, MLF_LOCAL_ONLY_BIT, 1, &ofpacts)` for packets originating from the controller (ARP responder, DHCP reply, IGMP). `ovn-architecture.7.xml` describes: "A higher-priority rule to match packets that have the MLF_LOCAL_ONLY flag set... drops MLF_LOCAL_ONLY packets directed to a localnet port".

**Observation and incident.** `ovn-trace --detailed` for an `mcast` group match prints `reg10 = 0x10` (bit 4) when the local-only branch fires. Real-world symptom: if the flag is not set correctly for a controller-originated ARP reply, the packet loops through a tunnel to another chassis and back, producing a duplicate ARP reply at the client. Diagnostic: tcpdump on the tunnel interface shows the same source ARP reply twice within microseconds.

**Comparison.** In VXLAN BUM, replication is handled by the underlay router through an underlay multicast group with no in-packet flag. OVN uses a register flag to keep all logic in the software data plane, which is easier to debug but costs one register bit per packet.

## §13.18.8 MLF_RCV_FROM_RAMP

> A flag in `MFF_LOG_FLAGS` at bit 1 (`MLF_RCV_FROM_RAMP_BIT = 1`) marking that the packet was just received from a ramp switch (a Logical_Switch_Port of type `vtep` or `l2gateway`), that is, from the hardware VTEP gateway side into the OVN logical network.

**Why it exists.** A hardware VTEP gateway (converting external VLANs into VXLAN towards OVN) does not send the full logical metadata in the Geneve TLV class 0x0102; it only carries the basic tunnel key. When the packet enters br-int from this tunnel, ovn-controller needs to distinguish it from packets coming from a regular OVN chassis so it can perform a few replenishment steps (set the datapath, infer the logical inport from the VLAN tag) without looping back to the VTEP. The `MLF_RCV_FROM_RAMP` flag plays that signalling role.

**Mechanism.** Source code: `controller/physical.c` sets the flag in the tunnel ingress classification table, for example with `.flags = MLF_RCV_FROM_RAMP | MLF_ALLOW_LOOPBACK` and `.flags_mask = has_vtep ? MLF_RCV_FROM_RAMP : 0`. The `LS_IN_L2_LKUP` stage then tests the flag to skip the branch that forwards back to the ramp port (avoiding a loop), and `ovn-architecture.7.xml` documents: "checking a MLF_RCV_FROM_RAMP flag, which is set when the packet... arrives via VTEP tunnels".

**Operation and incident.** When integrating OVN with a hardware VTEP (Cisco Nexus, Arista 7050, Cumulus VTEP), missing this flag leads to a broadcast loop between OVN chassis and VTEP. Diagnostic pattern: ovn-controller log contains entries "Resubmit dropped packet from ramp", and tcpdump on br-int shows the same multicast frame returning after about 50 microseconds. Recovery: verify the `Logical_Switch_Port type=l2gateway` configuration has the correct `tag` VLAN and `addresses`, and that `Chassis.encaps` has an entry of type `vxlan` (not only geneve).

**Version differences.** `MLF_RCV_FROM_RAMP` appeared together with VTEP gateway support starting from OVN 2.6, and bit position 1 is stable from version 22.03 through main.

## §13.18.9 MLF_FORCE_SNAT_FOR_DNAT

> Bit 2 (`MLF_FORCE_SNAT_FOR_DNAT_BIT = 2`) signals that after the packet has been DNATted in the gateway logical router, the egress SNAT stage must perform SNAT again, even when the normal policy would not require it.

**Context and problem solved.** In a floating IP scenario: a public client 8.8.8.8 reaches the Logical_Router gateway, which DNATs to 10.0.0.5 (a private VM). If the return path from the VM 10.0.0.5 to 8.8.8.8 does not go through the same gateway router (because the default route differs), the conntrack session loses context, traffic becomes asymmetric, and the connection breaks. To prevent this, OVN forces additional SNAT on the gateway, changing the source to the floating IP so the VM sees the traffic as coming from the floating IP itself and the reply returns through the same gateway.

**Set and test mechanism.** `northd/northd.c` (function `build_lrouter_nat_defrag_and_lb`, the branches generating Logical_Flow entries for DNAT entries with the option `force_snat_for_dnat=true`) compiles to the action `flags.force_snat_for_dnat = 1; ...`. The `LR_IN_DNAT` stage sets the flag; the egress `LR_OUT_SNAT` stage tests the flag and applies SNAT unconditionally. When the flag is 0, SNAT is applied only when the NAT entry matches normally; when the flag is 1, SNAT is applied even when the NAT entry does not match by itself.

**Observation.** `ovn-trace --detailed` for floating IP traffic prints `reg10 = 0x04` after the `LR_IN_DNAT` stage and then shows `ct(commit, nat(src=...))` at the `LR_OUT_SNAT` stage. When the reverse traffic fails to trigger force_snat (the flag is not set), the symptom is: the VM sees the source IP as the real client 8.8.8.8 rather than the floating IP, the return packet takes a different route, the client cannot match conntrack, and the packet is dropped.

**Source anchor and version.** `lib/logical-fields.c` registers the field flag with `MLF_FORCE_SNAT_FOR_DNAT_BIT`. Behaviour is stable from OVN 21.06 onwards together with the distributed gateway router overhaul.

## §13.18.10 MLF_FORCE_SNAT_FOR_LB

> Bit 3 (`MLF_FORCE_SNAT_FOR_LB_BIT = 3`) is the load balancer hairpin equivalent of `MLF_FORCE_SNAT_FOR_DNAT`: packets that the LB has redirected to a backend require forced SNAT after destination NAT so that the return path goes through the correct gateway router.

**Hairpin context.** An internal pod 10.0.0.5 calls VIP 10.99.0.1, and the backend is also 10.0.0.5 (itself or in the same subnet). After the LB DNAT, the packet has src=10.0.0.5 and dst=10.0.0.5; the backend replies directly without passing through the LB, conntrack diverges, and the connection breaks. Forced SNAT on the gateway changes the source to the VIP (or a dedicated SNAT IP), requiring the reply to return through the gateway where it can be correctly reverse-DNATted.

**Mechanism.** `northd/northd.c` `build_lb_rules` generates a Logical_Flow that sets `flags.force_snat_for_lb = 1` when the Load_Balancer record has the `hairpin_snat_ip` option or when northd infers that the client shares a datapath with the backend. `LR_OUT_SNAT` tests the flag. Code: `lib/logical-fields.c` registers the flag with `MLF_FORCE_SNAT_FOR_LB_BIT`; `gh search code` confirms it is only definition and registration, while the set and test logic lives in the northd flow generation.

**Real-world incident and verification.** Increasingly common with OVN-Kubernetes Service ClusterIP traffic: a pod calls the Service VIP and the backend is the pod itself. Diagnostic: `ovn-trace` the VIP traffic and check `reg10 = 0x08` after the LB stage; if it is missing, check whether the NB Load_Balancer entry has `options:hairpin_snat_ip` set. Recovery: `ovn-nbctl set Load_Balancer <uuid> options:hairpin_snat_ip="169.254.169.5"`.

**Related.** Same family as `REGBIT_HAIRPIN` (reg0[11]). REGBIT is an in-LS-pipeline signal, while MLF_FORCE_SNAT_FOR_LB is a cross-pipeline signal (LS to LR). The two flags together complete the full hairpin SNAT pattern.

## §13.18.11 MLF_LOCAL_BACKUP

> **Accuracy note.** The name `MLF_LOCAL_BACKUP` does not exist in upstream OVN source (verified by grep on `include/ovn/logical-fields.h` from branch-22.03 through main). The flag is often confused with `MLF_OVERRIDE_LOCAL_ONLY` (bit 18 in recent versions), which carries semantics close to "the backup chassis is allowed to re-emit the packet regardless of `MLF_LOCAL_ONLY` being set".

**Correction and verification.** When encountering internal documentation or commit messages that use "MLF_LOCAL_BACKUP", verify the actual intent with `grep -n "MLF_" include/ovn/logical-fields.h` at the baseline version. Common findings: (a) the author confused it with `MLF_OVERRIDE_LOCAL_ONLY` (see the `MLF_LOCAL_ONLY | MLF_OVERRIDE_LOCAL_ONLY` action mask in `controller/physical.c`); (b) the author confused it with the HA_Chassis_Group failover pattern that uses `Port_Binding.requested_chassis` rather than a register flag.

**Semantics of `MLF_OVERRIDE_LOCAL_ONLY`.** The override bit serves the scenario in which a backup HA chassis is nominated to handle a chassisredirect-port when the primary is down. Packets normally carry `MLF_LOCAL_ONLY = 1` to avoid a tunnel loop, but the backup chassis needs to re-replicate the packet to other chassis to complete BUM delivery, hence the override. Source: `controller/physical.c` builds the action `MLF_LOCAL_ONLY | MLF_OVERRIDE_LOCAL_ONLY` for the HA backup path.

**Observation.** `ovn-sbctl list HA_Chassis_Group` shows the priority order of chassis. When failover happens, the ovn-controller log of the newly active chassis prints "Claiming chassisredirect-port"; in `ovn-trace` of BUM traffic during the failover window the packet appears with `reg10` bit 18 set.

**Recommendation.** When the runbook needs a "backup pipeline marker", use the correct name `MLF_OVERRIDE_LOCAL_ONLY` rather than `MLF_LOCAL_BACKUP`. Avoid forking internal documentation with names that do not match upstream, which would cause confusion when looking up commit history or filing upstream bug reports.

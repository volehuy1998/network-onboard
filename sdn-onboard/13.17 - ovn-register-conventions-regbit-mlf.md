## 13.17.1 Introduction to OpenFlow registers, why OVN needs registers

OpenFlow specification 1.3+ defines a **register field** (16 registers, 32 bits each, with extended 64-bit `xreg0-7` and 128-bit `xxreg0-3`) as **per-packet scratch storage** that lives across the pipeline. A register is not a real packet header, it is not transmitted on the wire, and it exists only inside the context of processing a packet on a single switch. When a packet leaves the switch (output action), the register values are discarded.

On a bare OpenFlow switch (without OVN), registers are commonly used by the controller for simple purposes, for example tagging an ACL group ID and reading it back at a later table, or temporarily caching a MAC value after ARP responder. OVN turns the registers into the **backbone of pipeline logic**. Almost every forwarding decision in OVN is encoded as register state passed between stages.

To picture the reason: OVN has 26 ingress stages plus 10 egress stages for a logical switch, and 20 ingress stages plus 7 egress stages for a logical router (see Part 13.16). A packet that travels from VM A to VM B on two different chassis must pass through more than 50 logical stages. Every stage needs to know: (a) which logical datapath the packet belongs to, (b) what the inport and outport are, (c) which conntrack zone is in use, (d) what verdict the ACL has emitted, (e) whether NAT has already transformed the packet, and so on. The real packet headers (Ethernet, IP, TCP) cannot carry all this information, and they should not, since that would expose logical-pipeline detail on the tunnel wire. The solution is to use registers as a **state buffer**.

OVN splits the 16 physical registers into three role groups:

- **Logical fields** (reg10-15): each register has a fixed meaning across the whole OVN cluster, datapath ID, conntrack zone, inport, outport. This is a **non-overridable** convention that every Logical Flow follows.
- **General-purpose logical registers** (reg0-9 plus xxreg0-1): a scratch pool with a lighter convention. reg0 is used as the REGBIT_* bitmap, reg9 also carries some REGBIT, reg1 and reg2 carry the Load Balancer original DIP and DPort, and xxreg0-1 carry the IPv6 LB original DIP.
- **Logical flags register** (reg10 = MFF_LOG_FLAGS): a 32-bit bitmap for `MLF_*` flags, controlling cross-stage behaviour such as loopback, hairpin, and force-SNAT.

This split is declared in `include/ovn/logical-fields.h` (verified at OVN v22.03.8: the file is 191 lines and defines the entire register convention as `MFF_LOG_*` macros that map to OVS `MFF_REG*` macros).

> **Version note.** The convention is stable from OVN 20.06 to 22.03. OVN 23.06+ adds `MFF_LOG_FLAGS_BIT`, a second 32-bit flag word, but does not change the reg10-15 mapping. OVN 24.03+ refines the REGBIT layout (for example, ACL_HINT is split into 4 bits instead of 2). The curriculum holds the OVN 22.03.8 baseline, and every version drift is called out per keyword per Rule 14.

## 13.17.2 OVN logical fields, registers with an absolute convention (reg10-15)

The last six registers (reg10-15) plus MFF_METADATA (64 bits) are reserved by OVN as **logical fields** with a fixed meaning. Every Logical Flow must follow them, ovn-northd compiles NB to SB and ovn-controller translates SB to OF based on this convention.

| OVN macro | OVS register | Bit width | Role |
|-----------|--------------|-----------|------|
| `MFF_LOG_DATAPATH` | `MFF_METADATA` | 64 | Current logical datapath ID (logical switch or logical router) |
| `MFF_LOG_FLAGS` | `MFF_REG10` | 32 | `MLF_*` flag bitmap (see 13.17.5) |
| `MFF_LOG_DNAT_ZONE` | `MFF_REG11` | 32 | DNAT conntrack zone for a gateway router |
| `MFF_LOG_SNAT_ZONE` | `MFF_REG12` | 32 | SNAT conntrack zone for a gateway router |
| `MFF_LOG_CT_ZONE` | `MFF_REG13` | 32 | LSP conntrack zone for stateful ACL (allocated per port) |
| `MFF_LOG_INPORT` | `MFF_REG14` | 32 | Logical input port tunnel key |
| `MFF_LOG_OUTPORT` | `MFF_REG15` | 32 | Logical output port tunnel key |

Source verify: `include/ovn/logical-fields.h` lines 30 to 37 (OVN branch-22.03, commit verified through `gh api`):

```c
#define MFF_LOG_DATAPATH MFF_METADATA /* Logical datapath (64 bits). */
#define MFF_LOG_FLAGS      MFF_REG10  /* One of MLF_* (32 bits). */
#define MFF_LOG_DNAT_ZONE  MFF_REG11  /* conntrack dnat zone for gateway router */
#define MFF_LOG_SNAT_ZONE  MFF_REG12  /* conntrack snat zone for gateway router */
#define MFF_LOG_CT_ZONE    MFF_REG13  /* Logical conntrack zone for lports */
#define MFF_LOG_INPORT     MFF_REG14  /* Logical input port (32 bits). */
#define MFF_LOG_OUTPORT    MFF_REG15  /* Logical output port (32 bits). */
```

**Operational reading:**

- **`MFF_LOG_DATAPATH` (metadata, 64-bit):** every flow inside one logical datapath carries the same metadata value. When a packet lands on br-int after Geneve decapsulation, ovn-controller loads the datapath ID from the Geneve VNI into the metadata field before sending the packet into the pipeline. As a result, a packet that crossed a tunnel from chassis B to chassis A still carries an identifier of which datapath it belongs to.
- **`MFF_LOG_INPORT` plus `MFF_LOG_OUTPORT` (reg14 and reg15):** the value is the **tunnel key** of the port, that is the 32-bit number ovn-northd allocates to each LSP and LRP and stores in `Port_Binding.tunnel_key` (SB DB). When inport=2, the packet entered through an LSP whose tunnel_key=2, not through OpenFlow port number 2 on br-int. This is the cardinal distinction, explained in detail when reading dump-flows in 13.17.8.
- **`MFF_LOG_CT_ZONE` (reg13):** stage `LS_IN_PORT_SEC_IP` (table 1) writes the LSP conntrack zone into reg13 right after the port-security check. Every action `ct(zone=NXM_NX_REG13[0..15], ...)` in the later stateful stages reads reg13 to know which zone to use. Zone allocation is per port (not per datapath), giving binary conntrack isolation between tenants running on the same chassis.
- **`MFF_LOG_DNAT_ZONE` plus `MFF_LOG_SNAT_ZONE` (reg11 and reg12):** these only matter for a gateway router (a logical router with at least one LRP type=gateway). DNAT and SNAT run in two separate zones to avoid state collision when the same packet passes twice (UNDNAT plus SNAT in the egress pipeline). Only chassis that host gateway redirect (cr-* port) actually use these two registers, on a chassis that hosts only VMs reg11 and reg12 are unused.

> **Anti-pattern.** On the local ovs-vswitchd, do NOT overwrite reg10-15 in custom rules (for example a Mininet experiment). If a non-OVN OpenFlow flow overwrites reg14, every Logical Flow afterwards sees the wrong inport, leading to ACL or port-security bypass. This is a tenant-escape path if the hypervisor does not tightly control the OpenFlow channel of br-int. Recommended mitigation: only allow `ovn-controller` to open the OpenFlow channel, and block manual `ovs-ofctl add-flow` through an AppArmor or SELinux profile.

## 13.17.3 OVN general-purpose logical registers (reg0-9, xxreg0-1)

The remaining registers (reg0-9 plus xxreg0-1) are a **scratch pool**. They do not have a cluster-wide fixed role like reg10-15, but OVN still uses macro names to mark intended roles.

```c
#define MFF_LOG_REG0             MFF_REG0
#define MFF_LOG_LB_ORIG_DIP_IPV4 MFF_REG1
#define MFF_LOG_LB_ORIG_TP_DPORT MFF_REG2

#define MFF_LOG_XXREG0           MFF_XXREG0
#define MFF_LOG_LB_ORIG_DIP_IPV6 MFF_XXREG1

#define MFF_N_LOG_REGS 10
```

Source: `include/ovn/logical-fields.h` lines 40 to 47.

| Register | Main role | Stages that use it |
|----------|-----------|--------------------|
| `reg0` | REGBIT_* bitmap (see 13.17.4), 16 bits used for conntrack hint, ACL hint, DHCP/DNS lookup result, hairpin |
| `reg1` (`MFF_LOG_LB_ORIG_DIP_IPV4`) | Saves the original DIP (IPv4) before LB DNAT, used for hairpin and LB affinity |
| `reg2` (`MFF_LOG_LB_ORIG_TP_DPORT`) | Saves the original destination port before LB DNAT |
| `reg3-8` | Scratch with no fixed name, used temporarily inside routing and policy stages |
| `reg9` | Auxiliary REGBIT bitmap (5 bits for egress, larger packet, neighbor lookup) |
| `xxreg0` (`MFF_LOG_XXREG0`) | 128-bit scratch, mostly for IPv6 ND/NA and nexthop |
| `xxreg1` (`MFF_LOG_LB_ORIG_DIP_IPV6`) | Saves the original DIP (IPv6) before LB DNAT |

An important concept: **xxreg0 and xreg0 alias the same memory area as reg0+reg1+reg2+reg3** (4 times 32 bits = 128 contiguous bits). If a flow writes `xxreg0=0x...`, it overwrites reg0, reg1, reg2 and reg3 at once. OVN exploits this property, for example `MFF_LOG_XXREG0` (128-bit IPv6 address scratch) maps onto reg0+1+2+3, but in practice OVN only writes into it when reg0 to reg3 are idle. If an earlier stage set a REGBIT into reg0 and a later stage writes xxreg0, that REGBIT is lost.

For this reason **register scheduling is delicate**. ovn-northd builds flows in a fixed stage order, and each register is used only within a contiguous range of stages. When an OVN cluster upgrades to a new version, the stage range can change and the register layout sometimes shifts. This is one reason every major release publishes a detailed changelog about internal layout.

## 13.17.4 REGBIT bit-flag inventory (reg0 plus reg9)

OVN defines **REGBIT** macros in `northd/northd.c` (top of the file, around lines 70 to 95) as shorthands for scratch bit flags inside reg0 and reg9. REGBIT differs from MLF on one point: REGBIT is a **scratch flag** that lives only in a narrow stage range (for example only between pre_acl and stateful), whereas MLF is **persistent metadata** that lives across the entire pipeline.

Source verify: `northd/northd.c` (OVN branch-22.03, the specific commit accessed through `gh api /repos/ovn-org/ovn/contents/northd/northd.c?ref=branch-22.03`):

```c
#define REGBIT_CONNTRACK_DEFRAG   "reg0[0]"
#define REGBIT_CONNTRACK_COMMIT   "reg0[1]"
#define REGBIT_CONNTRACK_NAT      "reg0[2]"
#define REGBIT_DHCP_OPTS_RESULT   "reg0[3]"
#define REGBIT_DNS_LOOKUP_RESULT  "reg0[4]"
#define REGBIT_ND_RA_OPTS_RESULT  "reg0[5]"
#define REGBIT_HAIRPIN            "reg0[6]"
#define REGBIT_ACL_HINT_ALLOW_NEW "reg0[7]"
#define REGBIT_ACL_HINT_ALLOW     "reg0[8]"
#define REGBIT_ACL_HINT_DROP      "reg0[9]"
#define REGBIT_ACL_HINT_BLOCK     "reg0[10]"
#define REGBIT_LKUP_FDB           "reg0[11]"
#define REGBIT_HAIRPIN_REPLY      "reg0[12]"
#define REGBIT_ACL_LABEL          "reg0[13]"
#define REGBIT_ACL_STATELESS      "reg0[16]"

#define REGBIT_EGRESS_LOOPBACK            "reg9[0]"
#define REGBIT_PKT_LARGER                 "reg9[1]"
#define REGBIT_LOOKUP_NEIGHBOR_RESULT     "reg9[2]"
#define REGBIT_LOOKUP_NEIGHBOR_IP_RESULT  "reg9[3]"
#define REGBIT_DST_NAT_IP_LOCAL           "reg9[4]"
```

These are **20 REGBIT** flags (15 on reg0 plus 5 on reg9) in OVN 22.03.8. The curriculum REF (`doc/ovs-openflow-ovn-keyword-reference.md` Section 3.3) lists some other REGBITs (for example `REGBIT_PORT_SEC_DROP`, `REGBIT_SKIP_LOOKUP`, `REGBIT_FROM_RAMP`). This is **version drift**: the REF describes the OVN 24.03+ convention after the port-security stage was restructured. In the 22.03.8 baseline, port-security does NOT use REGBITs, it uses flow priority plus a direct drop action inside stages `LS_IN_PORT_SEC_L2/IP/ND`.

### 13.17.4.a Conntrack hint flags (reg0[0-2])

The first three bits of reg0 are reserved for conntrack pipeline coordination between stages `LS_IN_PRE_ACL`, `LS_IN_PRE_STATEFUL` and `LS_IN_STATEFUL`:

- **`REGBIT_CONNTRACK_DEFRAG` (reg0[0]):** stage `LS_IN_PRE_ACL` sets this bit to mark that the packet needs defragmentation before conntrack lookup. The reason for separating defrag into its own bit: an action `ct()` on an already-defragmented packet is effectively a no-op, only a fresh packet (one that has not yet been through ct) actually needs defrag.
- **`REGBIT_CONNTRACK_COMMIT` (reg0[1]):** the final ACL stage sets this bit when the verdict is allow plus the connection is new (NEW). Stage `LS_IN_STATEFUL` reads the bit and emits the action `ct(commit, ...)` to install a conntrack entry. If the bit is not set, the packet is not committed.
- **`REGBIT_CONNTRACK_NAT` (reg0[2]):** marks that NAT is needed in conntrack context (for example a stateful LB hairpin), kept separate from `REGBIT_HAIRPIN` (reg0[6]) which is a hairpin lookup result.

### 13.17.4.b Stateless lookup result (reg0[3-5])

The next three bits hold **lookup results** from three reference tables, DHCP options, DNS, and ND RA options. The corresponding stage (for example `LS_IN_DHCP_OPTIONS`) emits an action `put_dhcp_opts(...)` and writes the returned value into reg0[3]. The next stage (`LS_IN_DHCP_RESPONSE`) matches reg0[3]==1 to know whether there is a response to handle.

- **`REGBIT_DHCP_OPTS_RESULT` (reg0[3])**
- **`REGBIT_DNS_LOOKUP_RESULT` (reg0[4])**
- **`REGBIT_ND_RA_OPTS_RESULT` (reg0[5])**

The lookup-then-set-bit pattern repeats at many other OVN stages. The idea: separate an action that may fail (DHCP option lookup, DNS lookup) from downstream flow control. The stage installs a flow that assumes success, the action sets a bit, and the next stage uses the bit to branch.

### 13.17.4.c ACL hint flags (reg0[7-10] plus reg0[13])

OVN ACLs are compiled into a chain of flows at stages `LS_IN_ACL` and `LS_OUT_ACL`. Because ACLs may carry thousands of rules, instead of using `priority` to rank them all, OVN splits ACL into a **two-stage evaluation**: stage 1 = `LS_IN_ACL_HINT` sets one of four hint bits, stage 2 = `LS_IN_ACL` reads the hint and emits the final verdict. The goal is to separate "match" from "action" so OVN can optimise combinatorial expansion.

| REGBIT | Bit | Meaning |
|--------|-----|---------|
| `REGBIT_ACL_HINT_ALLOW_NEW` | reg0[7] | ACL matches an allow rule plus the packet is `ct.new` (no conntrack entry yet) |
| `REGBIT_ACL_HINT_ALLOW` | reg0[8] | ACL matches an allow rule plus the packet is `ct.est` (entry already exists) |
| `REGBIT_ACL_HINT_DROP` | reg0[9] | ACL matches a deny rule with no log or reject, drop immediately |
| `REGBIT_ACL_HINT_BLOCK` | reg0[10] | ACL matches a reject rule (must emit ICMP unreachable) |
| `REGBIT_ACL_LABEL` | reg0[13] | ACL has logging enabled, copy the packet to the controller for audit |

Stage `LS_IN_ACL` matches exactly one of the four ALLOW/DROP/BLOCK bits and then takes the corresponding verdict, `next` (allow forward), `drop` (silently drop), or `output:CONTROLLER` (for the reject ICMP).

> **Common misunderstanding.** Some materials describe the OVN ACL as using "priority" to rank rule order. That is only at the user level, ovn-northd compiles ACL.priority into OpenFlow flow priority inside stage `LS_IN_ACL_HINT`. The four REGBITs above are **mutually exclusive**, every packet sets exactly one hint bit in the `ACL_HINT` stage, and `LS_IN_ACL` does not have "fallthrough".

### 13.17.4.d Hairpin plus FDB (reg0[6, 11, 12, 16])

- **`REGBIT_HAIRPIN` (reg0[6]):** stage `LS_IN_PRE_HAIRPIN` sets the bit when it detects that the packet is an LB hairpin (client and backend on the same LSP). Stage `LS_IN_NAT_HAIRPIN` reads the bit and rewrites the source IP to the VIP (so the backend does not see itself).
- **`REGBIT_LKUP_FDB` (reg0[11]):** stage `LS_IN_LOOKUP_FDB` sets the bit when a MAC is found in the local FDB (no BUM flooding needed). Stage `LS_IN_PUT_FDB` reads it and decides whether to install a new entry.
- **`REGBIT_HAIRPIN_REPLY` (reg0[12]):** set for the hairpin reply direction, used to apply untracking and un-NAT.
- **`REGBIT_ACL_STATELESS` (reg0[16]):** an ACL rule with `options:stateless=true` (bypass conntrack) sets the bit, the stateful stage skips the commit when it sees the bit.

### 13.17.4.e Egress pipeline plus neighbor lookup (reg9)

reg9 holds REGBITs for the egress pipeline plus logical router routing:

- **`REGBIT_EGRESS_LOOPBACK` (reg9[0]):** marks that the packet is in the egress loopback round (to avoid an infinite loop). The egress stage checks the bit, and if it is already set, drops to break the infinite loop.
- **`REGBIT_PKT_LARGER` (reg9[1]):** stage `LR_IN_LARGER_PKTS` sets the bit if the packet exceeds the output port MTU. The corresponding stage reads the bit and emits ICMP frag-needed.
- **`REGBIT_LOOKUP_NEIGHBOR_RESULT` (reg9[2]):** stage `LR_IN_LOOKUP_NEIGHBOR` sets the bit when a MAC is found in MAC_Binding. Stage `LR_IN_LEARN_NEIGHBOR` reads the bit to decide whether to send an ARP/ND request or to continue.
- **`REGBIT_LOOKUP_NEIGHBOR_IP_RESULT` (reg9[3]):** the result of an IP-keyed lookup (instead of a MAC-keyed lookup) in MAC_Binding.
- **`REGBIT_DST_NAT_IP_LOCAL` (reg9[4]):** the packet is destined to an IP that has been DNAT'ed to a local LSP, used for LB internal routing.

## 13.17.5 MLF flags in reg10 (MFF_LOG_FLAGS)

REGBIT is a scratch flag (short-lived). MLF (`MLF_*`) is **persistent metadata flag**, lives across the entire pipeline, set at the source chassis and read at the destination chassis after passing through a Geneve tunnel.

Definition from `include/ovn/logical-fields.h` lines 51 to 90:

```c
enum mff_log_flags_bits {
    MLF_ALLOW_LOOPBACK_BIT = 0,
    MLF_RCV_FROM_RAMP_BIT = 1,
    MLF_FORCE_SNAT_FOR_DNAT_BIT = 2,
    MLF_FORCE_SNAT_FOR_LB_BIT = 3,
    MLF_LOCAL_ONLY_BIT = 4,
    MLF_NESTED_CONTAINER_BIT = 5,
    MLF_LOOKUP_MAC_BIT = 6,
    MLF_LOOKUP_LB_HAIRPIN_BIT = 7,
    MLF_LOOKUP_FDB_BIT = 8,
    MLF_SKIP_SNAT_FOR_LB_BIT = 9,
    MLF_LOCALPORT_BIT = 10,
    MLF_USE_SNAT_ZONE = 11,
    MLF_FROM_CTRL_BIT = 12,
};
```

| MLF flag | Bit | Purpose |
|----------|-----|---------|
| `MLF_ALLOW_LOOPBACK` | 0 | Allow output back to the inport (by default OVN drops the loop to avoid jamming) |
| `MLF_RCV_FROM_RAMP` | 1 | Packet arrived from a ramp switch (legacy L2 gateway), compensates for the fact that ramp encap carries no outport info |
| `MLF_FORCE_SNAT_FOR_DNAT` | 2 | After DNAT, force SNAT inside the gateway router (avoids asymmetric routing) |
| `MLF_FORCE_SNAT_FOR_LB` | 3 | After LB DNAT, force SNAT so the reply returns through the gateway |
| `MLF_LOCAL_ONLY` | 4 | The packet must not be tunneled to another chassis, output locally only |
| `MLF_NESTED_CONTAINER` | 5 | Packet arrived from a nested container (Kata Container, Firecracker) |
| `MLF_LOOKUP_MAC` | 6 | MAC_Binding lookup succeeded |
| `MLF_LOOKUP_LB_HAIRPIN` | 7 | LB hairpin lookup result |
| `MLF_LOOKUP_FDB` | 8 | FDB lookup succeeded |
| `MLF_SKIP_SNAT_FOR_LB` | 9 | Skip SNAT in the gateway router after LB (the inverse of `FORCE_SNAT_FOR_LB`) |
| `MLF_LOCALPORT` | 10 | Packet arrived from an LSP of type=localport |

**MLF versus REGBIT:**

| Property | MLF | REGBIT |
|----------|-----|--------|
| Location | reg10 (32-bit bitmap) | reg0 (15 bits) plus reg9 (5 bits) |
| Scope | Lives across the pipeline plus through Geneve tunnels | Short-lived inside a narrow stage range |
| Count | 13 flags (OVN 22.03) | 20 flags (OVN 22.03) |
| Encoded on tunnel | Yes, through Geneve TLV class 0x0102 | No, registers are reset to 0 after decapsulation |
| Use case | Cross-chassis behaviour (loopback, force-SNAT) | Pipeline-local coordination (ACL hint, conntrack defrag) |

> **Key point.** When a packet travels from chassis A to chassis B over a Geneve tunnel, only MLF plus datapath ID plus inport plus outport are encoded inside the tunnel header. Every REGBIT is reset to 0. This means a cross-chassis stage (for example LR_IN_DNAT on the gateway chassis) must self-detect every condition that the source chassis already detected, there is no shortcut through REGBIT.

## 13.17.6 Geneve TLV class 0x0102, encoding metadata cross-chassis

When a packet leaves the source chassis through a tunnel, ovn-controller must encode three pieces of state into the Geneve header so that the destination chassis can reproduce the pipeline state correctly:

1. **Datapath ID** (logical switch or logical router ID).
2. **Inport** (the source LSP tunnel key).
3. **Outport** (the destination LSP tunnel key) plus a few MLF flags.

Geneve has a feature called **Tunnel Options** (RFC 8926), the header may carry several TLVs (Type-Length-Value), each TLV has a 2-byte Option Class plus a 1-byte Option Type. **OVN registered class 0x0102** with IANA (the Geneve TLV class registry), used exclusively for OVN metadata.

Encoding (ovn-controller emits this at the final stage of the ingress pipeline, see Part 13.7):

| Field | Bit width | Encoding inside Geneve TLV class 0x0102 |
|-------|-----------|------------------------------------------|
| Datapath key | 24-bit | TLV opt_type=0x80, payload 4 bytes (24-bit ID plus 8-bit reserved) |
| Inport tunnel key | 15-bit | TLV opt_type=0x80, bytes 5 to 6 (15 bits plus 1 bit for MLF) |
| Outport tunnel key | 16-bit | TLV opt_type=0x80, bytes 7 to 8 |

VXLAN has no TLV, so OVN reuses the 24-bit VNI for the datapath ID and adds a 16-bit inport-outport encoded inside the UDP source port (a small hack). The drawback: VXLAN encap has no room for MLF, so MLF flags are not preserved cross-chassis. This is one of the reasons OVN recommends Geneve over VXLAN for modern deployments (see Part 11.0 to compare tunnel formats).

> **Source verify.** TLV class 0x0102 is declared inside `controller/physical.c` in the function `put_encapsulation()` (OVN branch-22.03), specifically the macro `OVN_GENEVE_OPT_TYPE_TUNNEL_KEY` and the struct `tlv_tun_metadata`. Cross-link to Part 13.7 §13.7.8 (a detailed physical.c walkthrough).

## 13.17.7 ct_label bit assignment (OVN-defined)

A conntrack entry has a 128-bit label field (`ct_label`). OVN places bit flags inside the lower 32 bits to track stateful traffic categories:

```c
#define OVN_CT_BLOCKED_BIT 0
#define OVN_CT_NATTED_BIT  1
#define OVN_CT_LB_SKIP_SNAT_BIT 2
#define OVN_CT_LB_FORCE_SNAT_BIT 3

#define OVN_CT_ECMP_ETH_1ST_BIT 32
#define OVN_CT_ECMP_ETH_END_BIT 79
```

Source: `include/ovn/logical-fields.h` lines 169 to 178.

| ct_label bit | Meaning |
|--------------|---------|
| 0 (`OVN_CT_BLOCKED`) | Connection was denied by ACL mid-flow, the next packet is dropped immediately |
| 1 (`OVN_CT_NATTED`) | Connection has been NAT'ed, the reply must be un-NAT'ed |
| 2 (`OVN_CT_LB_SKIP_SNAT`) | LB DNAT but SNAT skipped (per-LB configuration) |
| 3 (`OVN_CT_LB_FORCE_SNAT`) | LB DNAT plus force SNAT for the gateway return |
| 32 to 79 | ECMP source MAC cache, stores the source MAC of the first packet to keep the reply symmetric on the same path |

Bits 32 to 79 are a clever design that solves **asymmetric ECMP**: when an LR has multiple ECMP next-hops, OVN stamps a flag inside ct_label[32..79] equal to the source MAC of the first packet. Reply traffic matches ct.label, so OVN knows that the reply must take the path with that MAC (instead of re-hashing the 5-tuple).

## 13.17.7.b Deep dive on MFF_LOG_OUTPORT, register reg15 for the logical output port

This subsection focuses on a single very important register: `MFF_LOG_OUTPORT` (the OVN alias for `MFF_REG15`). Throughout the OVN pipeline, this is the location that holds the decision "which logical port the packet will leave the pipeline through". Understanding this register at bit level is mandatory for any engineer who reads dump-flows br-int output or troubleshoots routing in an OVN production environment.

### Concept

`MFF_LOG_OUTPORT` is a 32-bit metadata field, aliased to OpenFlow register `MFF_REG15` for any packet travelling through the OVN pipeline. The value carries the **tunnel key** (an unsigned 32-bit integer) of the Logical_Switch_Port or Logical_Router_Port that ovn-northd has decided is the next hop in the logical topology. The tunnel key is the `tunnel_key` column of the `Port_Binding` table inside SBDB, allocated uniquely by ovn-northd to every LSP and LRP in the same datapath. The cardinal distinction with OpenFlow `OXM_OF_OUT_PORT`: register `MFF_LOG_OUTPORT` carries the **logical** port tunnel key (it lives at the abstract topology level), whereas `OXM_OF_OUT_PORT` carries the **physical** OpenFlow port number (the port number on br-int, for example `output:5`). The same packet may simultaneously have `MFF_LOG_OUTPORT=0x07` (LSP `vm2` tunnel_key=7) and `OXM_OF_OUT_PORT=5` (br-int port 5 connecting the veth for VM2) at one moment in the pipeline.

### History and background

Register `MFF_LOG_OUTPORT` appeared in the very first OVN release (2.4, 2015) when Pfaff, Pettit and Han Zhou announced the OVN architecture at the Open vSwitch 2015 fall conference. The design reasoning: the OpenFlow team already had `OXM_OF_OUT_PORT`, but OVN needed a **logical** outport that survives across the pipeline plus through Geneve tunnels, independent of the br-int physical OpenFlow port number (which differs between chassis). The tunnel-key approach was inspired by MPLS labels but is simpler (no stack, every packet has only one outport at a time). Stable from 2.4 to today, with only small changes at OVN 21.06+ (clears `MFF_LOG_OUTPORT` at the very start of the egress pipeline to avoid confusion with manual sets), and at OVN 24.03+ (adds an associated MLF flag for multi-output broadcast).

### Position in the architecture

The register lives at the layer between "logical pipeline" and "physical realization":

- **Higher layer (NBDB):** the operator declares `outport == "vm2"` inside an ACL match expression, or `output;` inside an action, without knowing the tunnel key.
- **Middle layer (SBDB Logical_Flow):** ovn-northd compiles the expression into `(reg15 == 0x07)` or compiles the action `outport = "vm2"; output;` into load:0x07 then reg15 then resubmit.
- **Lower layer (br-int OpenFlow):** ovn-controller translates reg15 into `MFF_REG15` through the macro `MFF_LOG_OUTPORT - MFF_REG0 = 15`. The final output (sending the packet out on the wire) needs an extra step to map from the tunnel key to the physical OF port through the `Port_Binding` table.

### Role and responsibilities

Four primary roles:

1. **Decide the logical destination.** Stage `LS_IN_L2_LKUP` (table 27 in OVN 22.03) writes reg15 after looking up the destination MAC in the FDB of the Logical_Switch.
2. **Filter traffic by outport at egress.** Every egress stage (`LS_OUT_*` from table 32 onwards) matches `(reg15 == 0xN)` to apply the ACL, QoS or port security for a specific LSP.
3. **Encode into Geneve TLV class 0x0102 for cross-chassis traffic.** The lower 16 bits of reg15 are packed into tunnel options bytes 7 to 8 (Part 13.17.6) before the packet leaves the source chassis.
4. **Feed the final `output;` action.** ovn-controller translates `output;` into a port_binding lookup, then a physical OF port resolution, then a real `output:N`.

### Why it exists (design motivation)

Before OVN, SDN controllers like the Neutron OVS-agent used OpenFlow `set_field:OXM_OF_OUT_PORT` directly. That approach breaks for cross-chassis packets, since chassis B does not know the physical port number of chassis A. OVN solves the problem by separating logic from physical: the pipeline always speaks in terms of the logical port tunnel key, and only at the very last second does it resolve into a physical port. The tunnel key is allocated once by ovn-northd when the LSP is created, stored in `Port_Binding.tunnel_key`, and broadcast to every chassis through SBDB. Every chassis shares the same tunnel-key namespace, so a packet that crosses a tunnel with reg15=0x07 is recognised by the receiving chassis as belonging to the LSP whose tunnel_key=7, without needing to know where that LSP physically resides.

### Problems it solves

This register solves two independent problems:

- **Cross-chassis logical port resolution.** A packet may travel from chassis A to chassis B over a tunnel, chassis B reads the reg15 tunnel key (decoded from the Geneve TLV) and identifies the destination LSP, independent of the local OF port number.
- **Multi-stage pipeline coordination.** Between the L2 lookup stage (which sets reg15) and the final output stage (which reads reg15), there are 4 to 6 intermediate stages (egress port-security, egress ACL, QoS, etc.). Every stage only needs to read reg15 instead of recomputing the MAC table lookup.

### Importance

An absolutely fundamental concept. Every packet that traverses OVN has a moment that sets reg15 and a moment that reads reg15. An engineer who does not understand this register cannot:

- Read `ovs-ofctl dump-flows br-int` correctly (will confuse reg15 with a physical port).
- Debug "packet dropped at the egress ACL stage" (because that stage matches reg15).
- Understand cross-chassis Geneve TLV encoding.
- Reproduce a gateway redirect issue (stage `LR_IN_GW_REDIRECT` rewrites reg15).

### How it works

The lifecycle of reg15 in a packet that traverses a Logical_Switch:

```
Stage  Table   reg15 behaviour                     Description
-----  -----   ----------------------------------  ------------------------
0      0       No value                            Packet just entered br-int
0      0       load:0x0->REG15                     Clear field (egress prep)
LS-IN  8-26    reg15 unchanged                     Ingress pipeline
LS-IN  27      load:0x07->REG15                    L2 lookup found vm2 (key=7)
LS-OUT 32      match: reg15=0x07                   Egress port-security
LS-OUT 33-39   match: reg15=0x07                   Egress ACL/QoS
LS-OUT 65      output:5 (physical port for vm2)    Final output
```

Source verify from `controller/physical.c` branch-22.03 lines 706 to 1024 (function `consider_port_binding()`): every LSP local to the chassis installs a flow `match_set_reg(&match, MFF_LOG_OUTPORT - MFF_REG0, port_key)` for the egress lookup stage. The function `put_load(0, MFF_LOG_OUTPORT, 0, 32, ofpacts)` clears reg15 at the start of egress (line 4215 of `lib/actions.c`).

Special behaviour at gateway redirect (stage `LR_IN_GW_REDIRECT`, table 18 in the logical router pipeline): if the destination LRP is a `chassisredirect` port (`cr-LR1`, ovn-northd compiles with a fixed chassis), ovn-northd emits a flow `(outport == "lr1-public") -> outport = "cr-lr1-public"; next;`. This action runs through `set_field:0xN->reg15` where N is the tunnel_key of the cr-port, forcing the packet through the gateway chassis even though the original LRP is distributed.

### How an operator masters this

Three core skills:

1. **Map tunnel_key to LSP name.** Run `ovn-sbctl --columns=logical_port,tunnel_key list Port_Binding | grep -E "^(logical_port|tunnel_key)"` and build a small in-head lookup table for every LSP under suspicion.
2. **Read dump-flows for reg15.** `ovs-ofctl dump-flows br-int | grep -E "reg15=0x[0-9a-f]+" | awk '{print $NF}'` shows every place where the pipeline matches outport.
3. **Combine ovn-trace plus ofproto/trace.** ovn-trace shows which logical stage sets reg15, ofproto/trace shows the actual value plus the physical port resolution.

A standard workflow to verify that an LSP `vm2` is receiving traffic:


### When something goes wrong

Four symptoms where reg15 is the first investigation point:

- **Packet goes to the wrong destination LSP.** Compare reg15 in ofproto/trace with the expected tunnel_key from `ovn-sbctl list Port_Binding`.
- **Egress ACL drop is unexpected.** Stage `LS_OUT_ACL` matches reg15, if reg15 is wrong, even a correct ACL drops the wrong packet.
- **Gateway redirect does not work.** Stage `LR_IN_GW_REDIRECT` does not emit `set_field:cr_tunnel_key->reg15`, check whether `is_cr_port()` in northd.c sees the LRP.
- **Multicast or broadcast loop.** reg15 was set to a multicast group tunnel_key (for example the `_MC_flood` group), and the flood-group output loops if MLF_ALLOW_LOOPBACK is set incorrectly.

### Real-world incident

**Production case 2025-Q2, an OVN 22.03.8 cluster of 3 chassis, gateway router lr0:**

**Symptom.** External traffic to VIP `203.0.113.50` (DNAT'ed to internal VM `10.0.0.20`) drops intermittently around 30 percent of the time, only on one specific chassis (`compute-2`).

**Investigation procedure:**

```bash
# Step 1: on compute-2, dump the flows that handle gateway redirect.
$ ovs-ofctl dump-flows br-int | grep -E "table=18.*reg15"
 table=18, priority=100, ip,metadata=0x3,reg14=0x1
   actions=load:0x9->REG15[],resubmit(,19)
# Note: tunnel_key 0x9 = "cr-lr0-public".

# Step 2: ovn-sbctl verify the cr-port tunnel_key.
$ ovn-sbctl --columns=logical_port,tunnel_key list Port_Binding cr-lr0-public
logical_port  : "cr-lr0-public"
tunnel_key    : 9
# Match.

# Step 3: ovn-trace simulate the inbound packet.
$ ovn-trace --detailed lr0 'inport == "lr0-public" && ip4.dst == 203.0.113.50'
# Output: stage lr_in_gw_redirect: outport = "cr-lr0-public"; next;
# reg15 <- 0x9
# Next stage lr_in_arp_resolve: ... stuck (no MAC binding).

# Step 4: check MAC_Binding for the gateway IP.
$ ovn-sbctl find MAC_Binding logical_port=cr-lr0-public
(no rows)
# No MAC binding!

# Step 5: dump conntrack zone of the LR.
$ sudo ovs-appctl dpctl/dump-conntrack | grep 203.0.113.50 | wc -l
0
# No conntrack entry visible.
```

**Root cause.** Chassis `compute-2` lost BFD connectivity to `compute-1` (the chassis owner of cr-lr0-public) for about 200 ms just before the test. ovn-controller on compute-2 still installed flows that match reg15=0x9 (chassisredirect target), but tunnel encap to compute-1 failed because BFD detected the chassis as down. ARP resolve at stage `LR_IN_ARP_RESOLVE` did not complete and the packet dropped at table 19.

**Fix.** Check `ovs-vsctl list interface ovn-compu1-0 | grep bfd_status`, restart the BFD agent. Then add an HA_Chassis_Group configuration with a `priority` that fails over the gateway faster (under 100 ms).

**Lessons learned.** A correct reg15 (set to a cr-port) is not enough, the chassisredirect target must also be alive (BFD up plus MAC_Binding present plus conntrack table sync). reg15 is only the start of the causal chain, not the conclusion.

### Failure modes and diagnostic signals

| Mode | Symptom | Diagnostic signal | Recovery |
|------|---------|--------------------|----------|
| Tunnel_key collision (very rare) | 2 LSPs share the same tunnel_key, packets routed wrong | `ovn-sbctl list Port_Binding` shows two rows with duplicate tunnel_key | `ovn-appctl -t ovn-northd inc-engine/recompute` |
| Stale reg15 from earlier stage (OVN < 21.06) | Egress stage matches the wrong reg15, drops randomly | `ofproto/trace` shows reg15 set twice without a clear in between | Upgrade OVN >= 21.06, or add `put_load(0, ...)` manually |
| Multicast loop | Broadcast packet loops between LSPs | The flood group does not clear MLF_ALLOW_LOOPBACK | Check ovn-northd compilation of the multicast flow plus the LSP type |
| GW_REDIRECT does not trigger | External traffic goes to the wrong LRP | Stage 18 does not emit `set_field:cr_key->reg15`, `is_cr_port()` does not match | Check whether the LRP has `gateway_chassis` set, and BFD status |
| Geneve TLV stripped | Cross-chassis packet has outport=0 | tcpdump on the tunnel header shows no class 0x0102 bytes 7 to 8 | Upgrade the peer OVS, or disable hardware tunnel offload |

### Comparison with other systems

| System | Equivalent concept | Difference from MFF_LOG_OUTPORT |
|--------|--------------------|----------------------------------|
| Linux netfilter | Interface index `nfmark` | `nfmark` is 32-bit but local to a host, not cross-host |
| OpenStack Neutron OVS-agent (legacy) | OF metadata with physical port | Physical only, no logical abstraction, breaks when the VM migrates |
| Cisco Catalyst VLAN egress port | VLAN ID plus MAC table lookup | Hardware ASIC handles it, no software pipeline logic |
| VMware NSX-T LSP UUID | VNI plus encoded LSP UUID | NSX-T centralises in the management plane, OVN distributes via SBDB |
| MPLS label stack | Top label = next-hop | MPLS is stack-based, OVN is flat (1 outport per packet), MPLS is hardware-accelerated |
| Open vSwitch (without OVN) | OF metadata plus manual flows | The operator writes `set_field:N->reg15` themselves, no logical compilation from NBDB |
| VXLAN BUM tunnel header | 24-bit VNI for the datapath | VXLAN has no room to encode an outport (UDP source port hack), reg15 is lost on cross-chassis VXLAN |

OVN-specific traits: cross-chassis logical-physical separation plus tunnel-key persistence through Geneve TLV plus a distributed control plane (every chassis self-resolves via SBDB). This is a value zone that is hard to reproduce in other systems and the reason OVN was chosen for CMS-driven multi-tenant clouds.

## 13.17.8 Anatomy 1, reading `ovs-ofctl dump-flows br-int` and decoding register state

**Bucket:** OVN. **Context:** every flow on br-int may carry register match or action expressions. **Purpose:** decode the meaning of every register value to know which stage is processing the packet. **Activity:** debug plus understand the OVN pipeline. **Mechanism:** combine the register convention plus the REGBIT/MLF inventory plus the stage table number.

**Sample output (a normal LSP, no ACL):**

```
$ ovs-ofctl dump-flows br-int table=0 | head -3
 cookie=0x0, table=0, priority=100,in_port=5,vlan_tci=0x0000/0x1000
   actions=load:0x4->NXM_NX_REG13[],load:0x3->NXM_NX_REG11[],
           load:0x4->NXM_NX_REG12[],load:0x6->NXM_NX_METADATA[],
           load:0x2->NXM_NX_REG14[],resubmit(,8),resubmit(,32),resubmit(,33)
```

**Decode of 7 attributes:**

| Attribute | Sample value | Healthy / Warning / Critical |
|-----------|--------------|------------------------------|
| `cookie` | `0x0` (always 0 for the OVN default OF channel) | Healthy: cookie links to the SB Logical_Flow UUID 32-bit prefix. Warning: cookie=0 may also be a flow installed by ovs-ofctl manually. |
| `table=0` | Stage `S_SWITCH_IN_PORT_SEC_L2` | Healthy: table 0 is the entry point for every packet coming from an LSP. |
| `priority=100` | High-priority match | Healthy: priority 100 is the exact match for an LSP. Warning: under 50 is a default-drop. |
| `in_port=5` | OpenFlow port number 5 on br-int | This is a physical OF port, DIFFERENT from the reg14 inport (logical). |
| `load:0x4->REG13` | Sets conntrack zone=4 for the LSP | Healthy: every LSP has a unique zone. |
| `load:0x6->METADATA` | Datapath ID = 6 (logical switch tunnel_key) | Cross-check with `ovn-sbctl list Datapath_Binding` to identify which logical switch. |
| `load:0x2->REG14` | Logical inport = 2 (LSP tunnel_key) | Cross-check with `ovn-sbctl list Port_Binding`. |

**Critical signal:** if a flow at table 0 has no `load:...->REG14`, port-security is being bypassed. It may be an ovn-controller bug or a non-OVN flow. Investigate immediately.

**Sample output (a stateful stage where a REGBIT is set):**

```
$ ovs-ofctl dump-flows br-int table=17 | grep 0x42
 cookie=0xdeadbeef, table=17, priority=2, ct_state=+new+trk,reg0=0x42/0x42
   actions=ct(commit, zone=NXM_NX_REG13[0..15]),resubmit(,18)
```

`reg0=0x42/0x42` is a bitmask expression: it matches both `bit 1 (REGBIT_CONNTRACK_COMMIT)` AND `bit 6 (REGBIT_HAIRPIN)`. Decode: stage 17 (LS_IN_STATEFUL) is committing conntrack for a hairpin reply.

## 13.17.9 Anatomy 2, `ovn-trace --detailed` register snapshot per stage

`ovn-trace` simulates a packet through the pipeline WITHOUT installing flows in real OVS. The flag `--detailed` prints register state at every stage transition.

**Sample output:**

```
$ ovn-trace --detailed ls0 'inport == "lsp1" && ip4.src == 10.0.0.1 && ip4.dst == 10.0.0.2'
ingress(dp="ls0", inport="lsp1")
-----------------------------------
 0. ls_in_port_sec_l2 (priority 50): next;
    reg14 <- 0x1, reg13 <- 0x10  [inport=lsp1, ct_zone=16]
 1. ls_in_port_sec_ip (priority 50): next;
 5. ls_in_pre_acl (priority 100): reg0[0] = 1; next;
    reg0 <- 0x1                  [REGBIT_CONNTRACK_DEFRAG set]
 9. ls_in_acl (priority 1000): reg0[1] = 1; next;
    reg0 <- 0x3                  [REGBIT_CONNTRACK_DEFRAG + REGBIT_CONNTRACK_COMMIT]
17. ls_in_stateful (priority 100): ct(commit, zone=NXM_NX_REG13[0..15]);
24. ls_in_l2_lkup (priority 50): outport = "lsp2"; output;
    reg15 <- 0x2                 [outport=lsp2]
```

**What this tells us:**

- Stage 0 sets reg14 plus reg13 right after the port-security check.
- Stage 5 sets REGBIT_CONNTRACK_DEFRAG (reg0 goes from 0x0 to 0x1).
- Stage 9 sets REGBIT_CONNTRACK_COMMIT (reg0 goes from 0x1 to 0x3, that is bit 0 plus bit 1).
- Stage 17 uses the action `ct(commit, ...)` which reads reg13 as the zone.
- Stage 24 sets reg15 to the outport tunnel_key before output.

> **Key point.** Each line `regN <- 0x...` reports state **after** the stage executes. To verify that a REGBIT was set correctly, read the line right after the corresponding stage. If it is missing, the stage did not emit the set-bit action (meaning the Logical Flow was compiled wrongly or the rule condition did not match).

## 13.17.11 Capstone POE, verifying that the register convention is stable

**Predict.** Hypothesis: "OVN uses the same register layout between OVN 22.03 and 24.03". Predict that this is FALSE based on REGBIT changes between versions.

**Observe.** Run a lab with the same NB ACL on two versions:
- OVN 22.03.8 plus OVS 2.17.9 (Ubuntu 22.04 baseline).
- OVN 24.03 plus OVS 3.3 (Ubuntu 24.04 upgrade path).

Observation:
- 22.03.8: ACL hint uses 4 bits (reg0[7-10]).
- 24.03+: ACL hint expands to 5 bits, adds `REGBIT_ACL_PERSIST_ID` (for an audit-trail persistent UUID). reg0[14] is used for persist_id.

**Explain.**
- Each major OVN release may refine the REGBIT layout when adding new features.
- ovn-northd compiles NB to SB depending on the REGBIT layout of the corresponding version.
- A cross-version cluster (mixed 22.03 plus 24.03 chassis) will NOT work correctly. This is why OVN requires **a single version across the entire cluster** in production.

**Falsification test.**

Build a 2-chassis lab cluster with different versions (chassis A 22.03, chassis B 24.03), bind 2 VMs (VM1 on A, VM2 on B), add an ACL allow ICMP. Observe whether ICMP from VM1 to VM2 works. Predict: it fails because the different REGBIT layouts make the SB Logical Flow incompatible.

> **Why this POE matters.** It teaches the engineer that the OVN baseline version is **a cluster-wide invariant**, not an option for per-node upgrade. This is the foundation lesson for the operator playbook on upgrade strategy (see Part 20.3 §upgrade procedure).

## 13.17.12 Anti-patterns and notes

**Anti-pattern 1: overwriting reg13, reg14, or reg15 in custom flows.**

Some networking tutorials teach adding manual flows into br-int through `ovs-ofctl add-flow br-int`. If such a rule loads any value into reg13, reg14, or reg15, every Logical Flow afterwards sees the wrong zone or inport or outport, leading to: (a) the packet bypassing ACL or port-security, (b) the packet being dropped because the outport is invalid, or (c) conntrack zone collision between several LSPs, causing reply traffic of one tenant to be associated with another.

**Mitigation.** Block `ovs-ofctl` (and any non-OVN OpenFlow channel) on production chassis through an AppArmor or SELinux profile.

**Anti-pattern 2: assuming REGBIT is compatible across OVN versions.**

The REGBIT layout is not a public ABI, the OVN team is free to renumber bits when needed. Upgrading ovn-northd to a new version without simultaneously upgrading ovn-controller produces Logical Flows that the older chassis cannot decode.

**Note 3: ct_label[32..79] is not MAC text.**

When reading `ovs-appctl dpctl/dump-conntrack`, ct_label is shown as a 128-bit hex value. Do not decode it directly to a MAC, you must shift right by 32 bits and read the lower 48 bits.

**Note 4: Geneve TLV class 0x0102 is not enabled automatically.**

ovs-vswitchd must be compiled with the option `--enable-Geneve-TLV` (usually the default). If tunnel offload is delegated to NIC hardware that does not support Geneve TLV options, OVN falls back to the software path and performance drops. Verify with `ovs-appctl tnl/list` plus `ethtool -k <NIC> | grep tx-udp_tnl-segmentation`.

## 10.1.1 Tại sao cần Raft clustering

Single-node OVSDB có hai giới hạn production:

1. **Single point of failure** — node chết → toàn bộ OVN control plane down.
2. **Không scale write** — chỉ một instance xử lý transaction.

Raft giải quyết bằng cách replicate log qua nhiều node, automatic leader election + failover. OVSDB Raft giới thiệu OVS 2.9 (02/2018).

Raft là consensus protocol do Diego Ongaro và John Ousterhout công bố tại USENIX ATC 2014 — thay thế Paxos với semantics đơn giản hơn nhưng equivalent guarantees.

---

## 10.1.2 Raft basics

Cluster N node (N lẻ — 3, 5, 7). Tại mỗi thời điểm:

- Một node là **Leader**, handle mọi write từ client.
- Các node khác là **Followers**, replicate log từ leader.
- Khi leader chết, follower bầu **Candidate** → leader mới (election).

**Log replication**: client gửi write đến leader. Leader append entry vào log, replicate đến followers. Khi quorum acknowledge, leader commit. Quorum cho 3-node = 2, cho 5-node = 3.

**Fault tolerance**:

| Cluster size | Chịu được failure |
|---|---|
| 3 nodes | 1 node down |
| 5 nodes | 2 node down |
| 7 nodes | 3 node down |

Cluster 9+ không khuyến nghị vì độ trễ write tăng (quorum cần acknowledge).

## 10.1.3 Bootstrap 3-node cluster

Compass Ch N workflow:

**Node 1:**

```bash
[node1]# ovsdb-tool create-cluster /etc/openvswitch/db \
         /usr/share/openvswitch/vswitch.ovsschema tcp:10.0.0.1:6644
[node1]# ovsdb-server --remote=ptcp:6640 \
         --remote=punix:/var/run/openvswitch/db.sock \
         /etc/openvswitch/db &
```

`tcp:10.0.0.1:6644` là Raft cluster RPC port (khác port client 6640).

**Node 2:**

```bash
[node2]# ovsdb-tool join-cluster /etc/openvswitch/db Open_vSwitch \
         tcp:10.0.0.2:6644 tcp:10.0.0.1:6644
[node2]# ovsdb-server /etc/openvswitch/db &
```

**Node 3:**

```bash
[node3]# ovsdb-tool join-cluster /etc/openvswitch/db Open_vSwitch \
         tcp:10.0.0.3:6644 tcp:10.0.0.1:6644 tcp:10.0.0.2:6644
[node3]# ovsdb-server /etc/openvswitch/db &
```

Client kết nối cluster bằng comma-separated member list:

```bash
[client]# ovn-nbctl --db="tcp:10.0.0.1:6641,tcp:10.0.0.2:6641,tcp:10.0.0.3:6641" \
           ls-add ls0
```

Nếu leader change, client tự retry sang leader mới (transparent failover).

## 10.1.4 Cluster lifecycle management

Check status:

```bash
[node]# ovs-appctl -t /var/run/openvswitch/ovsdb-server.ctl \
         cluster/status Open_vSwitch
```

Output hiện: leader ID, role (leader/follower/candidate), commit index, log index, members.

**Leave cluster** (graceful):

```bash
[node2]# ovs-appctl cluster/leave Open_vSwitch
```

**Kick failed member**:

```bash
[node1]# ovs-appctl cluster/kick Open_vSwitch <sid-of-dead-node>
```

**Rule quan trọng (compass Ch N)**: "Once a server leaves a cluster it cannot rejoin — create a fresh member."

Sau `leave` hoặc `kick`, **phải xóa DB file cũ** và `join-cluster` như node mới.

## 10.1.5 Production topology cho OVN

OVN triển khai dùng hai cluster:

- **NB DB cluster** (3 node): logical topology ý đồ cấu hình. `ovn-nbctl` client.
- **SB DB cluster** (3 node): computed logical flow + chassis state. `ovn-controller` client.

Hai cluster có thể chạy trên cùng 3 physical node (process riêng) để tiết kiệm hardware.

5-node cluster cho triển khai lớn (>500 chassis), tính sẵn sàng cao hơn nhưng write độ trễ cao hơn.

## 10.1.6 Source code ovsdb/raft.c — internals của OVSDB Raft

Phần này đi sâu vào mã nguồn `ovsdb/raft.c` ở baseline OVS 2.17.9. File 5041 dòng, là toàn bộ implementation Raft state machine.

### 10.1.6.1 Kiến trúc public API và state transitions

`ovsdb/raft.h` định nghĩa public API mà `ovsdb-server.c` gọi qua main loop. Nhóm hàm chính:

**Lifecycle:**

- `raft_create_cluster()`: bootstrap node đầu tiên với cluster mới (1-node cluster).
- `raft_join_cluster()`: node mới gia nhập cluster đã tồn tại.
- `raft_run()` (`ovsdb/raft.c` ~L1929): chạy 1 vòng state machine, được gọi mỗi tick từ `ovsdb-server` main loop. Đây là entry point chính.
- `raft_close()`: đóng kết nối, cleanup.
- `raft_leave()` / `raft_left()`: rời cluster có chủ đích.

**State queries:**

- `raft_is_leader(struct raft *)`: check role hiện tại.
- `raft_is_connected()`: check connectivity tới quorum.
- `raft_is_joining()` / `raft_is_leaving()`: trạng thái transition.
- `raft_get_applied_index()` / `raft_get_commit_index()`: chỉ số log commit/apply.
- `raft_get_memory_usage()`: bộ nhớ chiếm dụng cho log + snapshot.

**Write API (cluster-replicated transaction):**

- `raft_command_execute()`: gửi command (transaction) tới leader.
- `raft_command_get_status()`: kiểm tra trạng thái command (committed | aborted | timeout | not_leader).
- `raft_command_wait()`: poll-block cho đến khi command commit hoặc fail.
- `raft_command_unref()`: cleanup.

**State transitions (3 role):**

- `FOLLOWER` (mặc định khi start): chỉ append entry từ leader, vote cho candidate.
- `CANDIDATE`: timeout election → chuyển sang candidate, request vote từ majority.
- `LEADER`: được elected, gửi `AppendEntries` heartbeat tới follower.

Function chuyển trạng thái:

- `raft_become_follower()` (`ovsdb/raft.c` ~L2633): khi nhận term cao hơn, hoặc khi mất leader (no heartbeat).
- `raft_become_leader()` (`ovsdb/raft.c` ~L2744): khi giành đủ vote majority trong election.

### 10.1.6.2 Append entry, heartbeat, election

**AppendEntries flow:**

1. Leader gửi `raft_append_request` định kỳ (mặc định mỗi 1000 ms heartbeat) hoặc ngay khi có log entry mới.
2. Function `raft_send_append_request()` (`ovsdb/raft.c` ~L2672) build request với: `term` hiện tại, `prev_log_index`, `prev_log_term`, danh sách `entries[]` (rỗng cho heartbeat).
3. Follower nhận: `raft_handle_append_request()` (~L3180) verify term + log consistency, append entry vào local log file, trả `raft_append_reply` với `success: true/false`.
4. Leader thu thập reply, khi majority `success` → tăng `commit_index`, áp dụng log entry vào ovsdb in-memory state.

**Election timeout:**

- `election_timer` mặc định 1000-2000 ms (random hoá để tránh split vote).
- Follower không nhận heartbeat trong khoảng này → tăng term, chuyển candidate, gửi `raft_send_vote_request` tới mọi peer.
- Vote rule: chỉ vote cho candidate nếu (a) chưa vote ai trong term này, (b) candidate's log ít nhất bằng mình (last_log_term + last_log_index).

**Tunable parameters:**

```bash
[host]# ovs-appctl -t ovsdb-server cluster/change-election-timer Open_vSwitch 5000
```

Tăng `election_timer` lên 5000 ms hữu ích cho cluster cross-DC có RTT cao (50-100 ms underlay) để tránh false election trigger. Trade-off: failover thực sự chậm hơn 5x.

### 10.1.6.3 Log compaction và snapshot

Log Raft tăng tuyến tính theo số transaction. Sau hàng triệu transaction, file log lớn hàng GB → khởi động chậm, tốn disk. Snapshot là cơ chế cô đọng: thay thế prefix log bằng "trạng thái áp dụng đến index X".

**Threshold tự động:**

OVSDB Raft tự đề xuất snapshot khi:

- `raft_log_length()` (~L4256) > 10× snapshot kích thước cũ, **hoặc**
- Memory usage > 100 MB.

Function `raft_notify_snapshot_recommended()` (~L4278) đánh dấu cờ. ovsdb-server main loop kiểm tra cờ và gọi `raft_store_snapshot()` (~L4298) khi đến chu kỳ phù hợp (không trong middle của transaction).

**Manual compact:**

```bash
[host]# ovs-appctl -t ovsdb-server ovsdb-server/compact OVN_Northbound
```

Lệnh này gọi `raft_save_snapshot()` (~L4087) ngay lập tức, ghi snapshot mới vào file `<db>.db`, truncate log entries cũ. Production thường run cron mỗi đêm.

**Install snapshot RPC:**

Khi follower bị tụt log quá xa (vì offline lâu, log bị compact xong), leader gửi `raft_install_snapshot_request` thay vì append entries cũ:

1. Leader detect: follower's `next_index` < cluster's `log_start` → cần snapshot.
2. `raft_send_install_snapshot_request()` (~L3460) gửi snapshot data trực tiếp.
3. Follower: `raft_handle_install_snapshot_request()` (~L4180) lưu snapshot, reset state.
4. `raft_handle_install_snapshot_reply()` (~L4206) leader xác nhận hoàn tất, tiếp tục append entries.

Quá trình này tốn băng thông và CPU. Nếu cluster có node thường xuyên offline > 1 giờ, snapshot transfer thành bottleneck.

### 10.1.6.4 Edge case bầu leader (split vote, partition)

**Split vote:**

Hai candidate cùng term cùng request vote, không ai đạt majority. Election timeout mới → cả hai tăng term, request lại. Để tránh livelock, OVSDB random hoá election timeout per-node (mỗi node có random 1000-2000 ms riêng). Trong thực tế, livelock 1-2 round, hết.

**Network partition (split brain):**

Cluster 5 node bị chia 3+2. Bên 3 (majority) tiếp tục bầu leader mới và phục vụ write. Bên 2 (minority) detect `raft_is_connected() = false`, từ chối write (`raft_command_execute` trả `RAFT_CMD_NOT_LEADER`). Khi mạng phục hồi, bên 2 thấy term cao hơn từ bên 3, tự `raft_become_follower`, replay log để catch up.

**Asymmetric partition:**

Node A có thể gửi tới B nhưng B không gửi được tới A (firewall asymmetric). A nhìn B như "alive" (nhận heartbeat), B nhìn A như "dead". Hậu quả: B trigger election liên tục mặc dù cluster healthy. Phát hiện qua `cluster/status`: term tăng nhanh bất thường (term: 12 → 25 → 47 trong vài phút).

```bash
[host]# ovs-appctl -t ovsdb-server cluster/status Open_vSwitch | grep Term
Term: 47
```

Term tăng nhanh (> 1 election timer) → asymmetric partition. Fix: kiểm tra firewall rule cả 2 chiều.

- `Term` tăng từ 12 → 47 trong 5 phút: asymmetric partition. Fix: check firewall (ufw, iptables) cả input + output trên port 6643/6644.
- `Status: joining` kéo dài > 1 phút: install snapshot transfer chậm. Fix: kiểm tra snapshot size + network bandwidth giữa joining node và leader.
- `Entries not yet applied` tăng liên tục: ovsdb-server CPU bottleneck. Fix: `--n-threads=4` hoặc giảm tải write side.
- `Connections` thiếu peer: peer down. Fix: check `systemctl status ovsdb-server` trên peer + network connectivity.

### ▶ Capstone POE: "Tăng election_timer luôn cải thiện stability"

> **Setup giả định:** Cluster OVN SB DB 5-node cross-DC, RTT giữa DC 80ms p99, hiện tại election_timer = 1000ms, thỉnh thoảng có false election (1-2 lần/ngày).

**Tuyên bố cần kiểm chứng:** "Tăng `election_timer` từ 1000 ms lên 10000 ms sẽ loại bỏ false election."

**Predict:** Đúng, false election biến mất hoàn toàn.

**Observe (giả định, kiểm chứng khi có lab):**

- False election rate: 1-2/ngày → 0/tuần. ✓
- Failover thực sự khi leader crash: 1-2 giây → 12-15 giây. ✗ (chậm 6x)
- Client write blocked window khi leader fail: 2 giây → 15 giây. ✗ (impact user)
- Cluster heal time sau partition: 5 giây → 25 giây. ✗

**Explain:**

1. `election_timer` là cận trên cho thời gian không có heartbeat trước khi nghi ngờ leader. RTT 80ms cao + jitter → 1000ms quá ngắn (false election khi heartbeat trễ). Tăng lên giải quyết symptom.
2. Tuy nhiên, `election_timer` cũng định nghĩa upper bound cho **thời gian phát hiện leader thật sự chết**. Tăng 10x → failover chậm 10x trong scenario thực.
3. Trade-off: trong production cross-DC, RTT 80ms cần `election_timer` ~3000-5000 ms (3-5x RTT, đủ buffer cho jitter), không nên cao quá vì hi sinh failover speed.

**Falsification test:**

- Set `election_timer` = 10000 ms.
- Quan sát 1 tuần: false election count?
- Force-kill leader, đo failover time + client write recovery.
- Nếu failover > 15 giây và client write blocked > 10 giây → giảm `election_timer` xuống mức cân bằng.

**Tuyên bố sai ở đâu:** "luôn cải thiện stability" coi stability = không có false election. Đúng phải là: **stability = balance giữa false election rate (false positive) và failover time (true positive). Election timer cần tune theo RTT thực tế của cluster.** Cross-DC 80ms RTT → 3000-5000 ms; same-DC 1ms RTT → 1000 ms baseline đủ.

## 10.1.7 OVSDB Raft cluster, treatment chuyên sâu cho keyword cornerstone

> **Phạm vi:** Treatment chuyên sâu (mở rộng 2026-04-26) cho keyword `OVSDB Raft cluster`. Các phần liên quan: §10.0 (OVSDB protocol RFC 7047, transaction format mà cluster replicate), §10.1.1 đến §10.1.5 (overview + bootstrap workflow), §10.5 (performance benchmarking cluster), §10.6 (security mTLS + RBAC trên cluster connection — file 10.6), §10.1.6 đến §10.1.6.5 (source-code internals của `ovsdb/raft.c`, mở rộng 2026-04), §13.1 (NBDB/SBDB chính là OVSDB Raft cluster trong production OVN), §13.7 (`ovn-controller` đọc SBDB qua cluster), §13.8 (`ovn-northd` ghi cả NBDB + SBDB qua cluster).

**Khái niệm.** `OVSDB Raft cluster` là chế độ deployment của `ovsdb-server` trong đó nhiều instance (thường là 3 hoặc 5 server vật lý/logical) cùng giữ chung một bản OVSDB schema và replicate transaction với nhau theo thuật toán Raft consensus. Tại mỗi thời điểm có một server đóng vai trò **leader** nhận write từ client, nối thêm transaction vào log, replicate log entry tới các **follower** qua RPC `Append-Entries`; chỉ khi đa số (quorum) đã ghi entry vào persistent log, leader mới commit và phản hồi client. Cluster size luôn lẻ (3, 5, 7) để quorum rõ ràng. Đây là cơ chế HA chính thức của OVSDB từ OVS 2.9, mặc định cho production OVN NBDB + SBDB.

**Lịch sử + bối cảnh.** Trước OVS 2.9 (release 19/02/2018), OVSDB chỉ có active-backup replication: một server active nhận write, một server standby theo dõi qua `monitor`, không có quorum + không có automatic leader election. Failover thủ công, RPO + RTO phụ thuộc thao tác admin. Khi OVN ra đời (OVS 2.6, 2016) và scale lên hàng trăm chassis, NBDB/SBDB single-server thành SPOF rõ ràng; team Nicira/RedHat (Ben Pfaff, Ilya Maximets) port Raft từ Diego Ongaro & John Ousterhout (USENIX ATC 2014: "In Search of an Understandable Consensus Algorithm", thesis "Consensus: Bridging Theory and Practice" 2014) vào `ovsdb/raft.c` (~5000 dòng) + `ovsdb/raft-rpc.c`. OVS 2.9 đưa Raft thành chế độ production; OVS 2.11 (2019) bổ sung `cluster/change-election-timer` runtime tunable; OVS 2.13 (2020) streamline add/remove member và auto-detect IP change; OVS 2.17 (2022, baseline curriculum) cải thiện Install-Snapshot streaming hiệu quả. OVN 22.03 chính thức hoá Raft là default cho NBDB/SBDB trong `ovn-ctl` script.

**Vị trí trong kiến trúc.** Đây là một mode chạy của process `ovsdb-server`, không phải process riêng. Trong OVN deployment, mỗi chassis controller (3 hoặc 5 host) chạy hai process `ovsdb-server`: một bind cluster cho NBDB (`OVN_Northbound`), một bind cluster cho SBDB (`OVN_Southbound`). Mỗi cluster có port nội bộ Raft (mặc định 6643/6644 tuỳ DB), tách biệt port client (6641 NBDB / 6642 SBDB). Client (`ovn-nbctl`, `ovn-sbctl`, `ovn-northd`, `ovn-controller`) kết nối qua comma-separated member list, tự retry sang leader mới khi failover. OVS 2.x cũng cho phép cluster cho local OVSDB (`Open_vSwitch` schema) nhưng ít dùng vì configuration database mỗi host độc lập.

**Vai trò.** Bốn vai trò chính: (a) **HA primitive** cho OVN control plane, đảm bảo mất một host không làm sập NBDB/SBDB; (b) **Linearizable write store**, mọi transaction commit trên cluster có total order qua log index; (c) **Read scaling option**, follower có thể serve read (eventual-consistency) khi client không yêu cầu fresh data; cờ `--leader-only` ép client luôn read từ leader (linearizable read); (d) **Membership manager**, runtime add/remove server qua `cluster/leave`, `cluster/kick`, hỗ trợ rolling upgrade từng node.

**Vì sao sinh ra.** Active-backup model giải quyết HA "có sẵn dữ liệu bản sao" nhưng không giải quyết "ai là active sau khi active cũ chết". Operator phải quan sát thủ công, promote standby, fence active cũ. Trong môi trường OVN với hàng trăm `ovn-controller` đồng thời reconnect, thao tác thủ công không kịp. Cần protocol tự bầu leader, đảm bảo **tại mỗi term chỉ có một leader** + **mọi transaction committed sống sót** qua mọi failure scenario chịu được. Raft đáp ứng đủ tiêu chí, lại đơn giản hơn Paxos về mặt explanation (Ongaro luận án nhấn mạnh "understandability"), phù hợp với codebase OVSDB (~50K dòng C, không có abstraction lớn).

**Vấn đề giải quyết.** Năm vấn đề nền: (1) **Single point of failure**, OVSDB single-node chết là toàn bộ control plane chết; cluster N=3 chịu được 1 failure, N=5 chịu 2. (2) **Split-brain** khi network partition, hai bên cùng nhận write rồi diverge; quorum requirement (commit cần majority ack) đảm bảo chỉ bên đa số mới có leader hợp lệ. (3) **Inconsistent state after restart**, server crash giữa transaction; Raft log + applied-index làm replay deterministic. (4) **Manual failover slow**, cần automatic election trong giây; randomized election timeout 1000-2000 ms cho phép detect leader chết và bầu mới trong 1-2 giây. (5) **Membership change race**, thêm/bớt server không được vi phạm quorum giữa chừng; Raft single-server-change protocol (Section 4 thesis) commit thay đổi config qua đúng log replication path nên an toàn.

**Tầm quan trọng.** Đây là khái niệm cốt lõi tuyệt đối của tầng OVN distributed control plane. Engineer không nắm OVSDB Raft cluster sẽ không xử lý được: (a) bất kỳ sự cố production OVN HA nào (`ovn-northd` không kết nối được NBDB sau partition, `ovn-controller` chuyển trạng thái `disconnected`); (b) tuning `election-timer` cho cross-AZ, cross-region deployment; (c) compact log + snapshot lifecycle để tránh disk full; (d) rolling upgrade NBDB/SBDB cluster qua `cluster/leave` + fresh join; (e) `--leader-only` decision khi nào cần linearizable read trade-off latency. Mọi bài viết "OVN at scale" của RedHat, Canonical, Ericsson đều có ít nhất một phần về Raft tuning, signal cho tầm quan trọng. Mức độ quan trọng ngang `ovn-northd` và `Logical_Flow` trong nhóm cốt lõi; thấp hơn `OVS_ACTION_ATTR_OUTPUT` (data plane) chỉ vì Raft nằm tầng control, không impact per-packet performance trực tiếp.

**Cơ chế hoạt động.** Vòng đời cluster gồm năm pha:

1. **Bootstrap pha 1** (server đầu tiên): admin chạy `ovsdb-tool create-cluster <db> <schema> <local-addr>`. Function `raft_create_cluster()` ghi file DB header với cluster ID mới + server ID của node này, marker entry index 0, term 0. Khi `ovsdb-server` start, function `raft_run()` (`ovsdb/raft.c` line 3045 v2.17.9) thấy cluster 1-server, tự động elect mình thành leader, term tăng 1, accept write.
2. **Bootstrap pha 2** (server tiếp theo gia nhập): admin chạy `ovsdb-tool join-cluster <db> <schema-name> <local-addr> <existing-addr>`. Function `raft_join_cluster()` ghi file DB header với cluster ID của cluster đã tồn tại + danh sách remote. Khi `ovsdb-server` start, server gửi `Add-Server` RPC tới một remote bất kỳ; remote forward tới leader; leader nối entry membership change vào log, replicate, commit. Server mới nhận snapshot hoặc log entry để catch up.
3. **Steady-state per-write**: client gửi transaction `OVSDB_TRANSACTION` qua port 6641/6642. Leader nhận, gọi `raft_command_execute()` (`ovsdb/raft.c` line 3513) tạo `struct raft_command`, append entry vào local log file, set `next_index` cho mỗi peer. Function `raft_send_append_request()` (line 2634) gửi `Append-Entries` RPC mỗi peer định kỳ (heartbeat) hoặc ngay khi có entry mới. Peer follower nhận `raft_handle_append_request()` (~line 2900-3000 v2.17.9), kiểm tra `prev_log_index/prev_log_term` consistency, append vào local log file, fsync, reply `success: true`. Khi leader thấy majority ack, tăng `commit_index`, chuyển trạng thái command sang `RAFT_CMD_COMMITTED`, `ovsdb-server` apply entry vào in-memory schema, reply client.
4. **Election pha**: follower không nhận `Append-Entries` heartbeat trong khoảng `election_timer` (default 1000-2000 ms randomized) → tăng term, gọi `raft_become_candidate()`, gửi `Request-Vote` tới mọi peer. Peer vote nếu (a) chưa vote ai trong term này; (b) candidate's `last_log_term, last_log_index` không nhỏ hơn của mình (election restriction, Section 5.4 paper). Khi candidate đạt majority, gọi `raft_become_leader()` (line 2650), gửi heartbeat đặc biệt (no-op entry nhằm commit entry từ term cũ, đảm bảo Section 6.2 leader completeness). Khi follower nhận term cao hơn, gọi `raft_become_follower()` (line 2610) auto downgrade.
5. **Snapshot pha**: log không thể tăng vô hạn. Khi `raft_log_length()` vượt threshold (10× snapshot cũ hoặc memory > 100 MB), function `raft_notify_snapshot_recommended()` set cờ. Main loop kiểm tra cờ ngoài middle-of-transaction, gọi `raft_save_snapshot()` ghi state hiện tại vào file `<db>.db`, truncate prefix log. Khi follower lag quá xa (next_index < cluster's log_start), leader gửi `Install-Snapshot` RPC qua `raft_send_install_snapshot_request()` thay vì append entries cũ.

**Cách kỹ sư vận hành thành thạo.** Phản xạ năm thao tác: (1) Đọc `cluster/status` mỗi sáng, bắt được drift sớm: term tăng quá nhanh (asymmetric partition), log gap quá lớn (slow follower), `Status: joining` quá lâu (snapshot transfer chậm). (2) Tune `election-timer` theo RTT: same-rack 1000 ms, cross-AZ 50 ms RTT đặt 3000-5000 ms, cross-region 100+ ms RTT đặt 5000-10000 ms; lệnh `ovs-appctl -t .../ovnnb_db.ctl cluster/change-election-timer OVN_Northbound 5000`. (3) Schedule manual compact định kỳ: cron `0 3 * * * ovs-appctl -t .../ovnnb_db.ctl ovsdb-server/compact OVN_Northbound`, tránh log file phình > 1 GB. (4) Rolling upgrade workflow: kiểm tra cluster healthy → `cluster/leave` trên node N1 → upgrade package → tạo DB file mới với `ovsdb-tool join-cluster` → start service → verify catch up → lặp cho N2, N3. (5) Failover dry-run định kỳ: kill leader trong staging environment, đo failover time + client recovery; nếu > 5 giây cần tune.

**Phân loại.** Có thể phân loại theo nhiều trục: (a) **theo cluster size**: 1-server (degraded, dev only), 3-server (default production OVN), 5-server (large deployment > 500 chassis), 7-server (rare, geo-distributed). (b) **theo role per node**: leader (đúng 1), follower (N-1), candidate (transient trong election). (c) **theo schema**: NBDB cluster (`OVN_Northbound`), SBDB cluster (`OVN_Southbound`), IC_NB / IC_SB cluster cho OVN Interconnect, Open_vSwitch local cluster (hiếm dùng). (d) **theo mode read consistency**: linearizable (qua leader, `--leader-only`) vs eventual-consistency (qua follower, default). (e) **theo deployment topology**: collocated (3 controller cùng host các process khác), dedicated (3 host chỉ chạy DB), cross-AZ (3 server ở 3 AZ khác nhau).

**Khi xảy ra sự cố.** OVSDB Raft cluster là keyword bắt buộc nắm trong các tình huống: (a) `ovn-controller` log lặp lại `unable to connect to OVN_Southbound` mặc dù tất cả 3 SBDB process running, kiểm tra `cluster/status` xem có server nào rời cluster không (`Status: disconnected from cluster`). (b) Sau network partition vài chục giây, một số write từ client báo `transaction failed: cluster not yet ready`, nguyên nhân là phía minority detect mất quorum nên reject write; phía majority vẫn serve. (c) `cluster/status` cho thấy `Term: 256` tăng từ `Term: 12` chỉ sau 1 giờ, signal asymmetric partition (firewall chỉ chặn một chiều) hoặc election timer quá ngắn so với jitter network. (d) `Log: [125, 4520000]` với log entry lên hàng triệu, cần manual compact `ovsdb-server/compact`; nếu compact treo, kiểm tra disk space + filesystem mount option. (e) Server mới join cluster `Status: joining` kéo dài > 5 phút, snapshot transfer bị nghẽn; kiểm tra bandwidth giữa joining node và leader, kiểm tra MTU, kiểm tra `Backlog` trong cluster/status.

**Liên quan mật thiết.** Cluster đan xen với: (a) `ovsdb-server` process (Phần 10.0) chính là binary chạy cluster mode thay vì standalone. (b) `ovn-northd` (Phần 13.8) đọc NBDB + ghi SBDB qua cluster connection, là client điển hình; northd HA dùng pattern active-passive với leader election riêng (qua `Lock` trong SBDB) khác với Raft leader. (c) `ovn-controller` (Phần 13.7) đọc SBDB qua cluster, dùng `--leader-only=false` mặc định để load-balance read sang follower. (d) `Logical_Flow`, `Port_Binding`, `Datapath_Binding` (Phần 13.5, 13.19) là rows nằm trong SBDB cluster, mọi thay đổi đều đi qua Raft commit. (e) mTLS + RBAC (Phần 10.6) áp dụng trên cluster connection: cả Raft RPC giữa các server và client connection đều có thể TLS-protected, certificate per-server. (f) Snapshot file `<db>.db` chính là OVSDB DB file format chuẩn (Phần 10.0), Raft chỉ thêm metadata cluster ID + server ID + term + log index ở header. (g) Performance benchmark (Phần 10.5) đo throughput cluster (typically 1000-3000 transaction/s với 3-node, giảm 50% so với standalone do sync overhead).

**Trường hợp sự cố thực tế.** Một deployment OpenStack OVN production 5-node SBDB cluster cross-AZ (3 AZ trong cùng region, RTT giữa AZ là 1.2 ms p50, 8 ms p99) gặp tình trạng `ovn-controller` trên 200 chassis liên tục log `Unreasonably long XXX ms poll interval` và `binding|INFO|Releasing lport ...` mỗi vài phút. Diagnostic ban đầu: dump `cluster/status` trên 5 SBDB node thấy `Term` tăng từ 47 lên 312 trong 4 giờ, mỗi vài phút có một `become_leader` mới. Root cause: một AZ có firewall update hôm trước, vô tình chặn outbound tới port 6644 từ 2 node của AZ đó, nhưng inbound vẫn cho phép heartbeat đến; do đó 2 node nhận được heartbeat (nhìn thấy leader alive) nhưng 2 node còn lại không nhận được reply (`raft_handle_append_reply` timeout) → leader nghĩ 2 node "dead" trong khi thực ra chỉ asymmetric partition. Election storm: leader step-down khi không đạt quorum kết nối, các node trigger election liên tiếp. Resolution: revert firewall rule để cho phép outbound 6644, sau đó cluster tự stable trong vòng 2 phút (`Term` ngưng tăng), `ovn-controller` reconnect bình thường. Lesson: cluster/status `Term` tăng nhanh là tín hiệu tin cậy nhất cho asymmetric partition, hơn cả `last msg N ms ago` (vốn chỉ phản ánh chiều inbound). Tuning sau sự cố: tăng `election-timer` từ 1000 ms lên 4000 ms (4x RTT p99) để giảm sensitivity với jitter, đồng thời thêm monitoring alert "term delta > 5 trong 1 giờ".

```
Pass criteria: học viên giải thích được tại sao Raft không cho phép node
              cũ tiếp tục write sau khi tách (không có quorum), và tại sao
              term mới sau election cao hơn term cũ.
```

**Lỗi thường gặp + tín hiệu chẩn đoán.**

- *Triệu chứng* `cluster/status` báo `Status: joining` trên một node mới và không tiến triển sau 5 phút. *Nguyên nhân* Install-Snapshot transfer chậm hoặc bị nghẽn, thường do MTU mismatch hoặc bandwidth không đủ giữa joining node và leader. *Chẩn đoán* `cluster/status` trên leader xem `Backlog`; `tcpdump -i any port 6643` để verify packet flow; `ss -i` xem CWND. *Khắc phục* Verify MTU + path discovery; nếu DB lớn (> 100 MB) chấp nhận chờ; nếu nghi ngờ bug, dùng `ovs-appctl cluster/leave` rồi tạo fresh DB file rồi join lại.
- *Triệu chứng* `Term` tăng nhanh (1 đơn vị mỗi vài giây). *Nguyên nhân* Asymmetric partition hoặc election timer quá ngắn so với network jitter. *Chẩn đoán* So sánh `Connections: ->X <-Y` xem có thiếu chiều nào không; `iperf3` đo RTT thực giữa các node; `tcpdump` filter port 6643 + 6644 hai chiều. *Khắc phục* Fix firewall + tăng `election-timer` qua `cluster/change-election-timer` với giá trị 3-5x RTT p99.
- *Triệu chứng* Client `ovn-nbctl` báo `database connection failed: cluster not yet ready` mặc dù 2/3 server đang chạy. *Nguyên nhân* Quorum 3-node = 2 nhưng nếu 2 node vừa restart đồng thời, có thể mất leader trong election window. *Chẩn đoán* `cluster/status` trên 2 server đang chạy xem có ai là leader không; nếu cả hai đều `Role: candidate`, đang trong election. *Khắc phục* Chờ 1-2 election cycle (2-4 giây) cho election ổn định; tránh restart đồng thời trong rolling upgrade.
- *Triệu chứng* DB file `ovnnb_db.db` phình > 5 GB sau vài tháng. *Nguyên nhân* Snapshot không trigger tự động hoặc compact bị skip do disk pressure. *Chẩn đoán* `ls -lh /etc/ovn/`; `cluster/status` xem `Log: [start, end]` range. *Khắc phục* `ovs-appctl ovsdb-server/compact OVN_Northbound`; cân nhắc cron hằng đêm; verify disk free > 2x DB size để compact có chỗ.
- *Triệu chứng* Sau `cluster/leave` rồi cố `cluster/join` lại với cùng DB file, server không tham gia được (`could not join cluster`). *Nguyên nhân* Rule "once a server leaves, cannot rejoin với DB file cũ" — DB file giữ marker đã leave. *Chẩn đoán* Comment compass Ch N + ovsdb-tool man page xác nhận. *Khắc phục* Xoá DB file cũ, chạy lại `ovsdb-tool join-cluster` với fresh DB file.

**So sánh với hệ khác.** Các hệ khác giải bài "consensus replicated state machine" theo cách khác:

- **etcd Raft (CoreOS, sau là dùng cho `etcd` KV store)**: cùng thuật toán Raft, cũng từ paper Ongaro 2014, library `go.etcd.io/raft`. Khác implementation language (Go vs C), khác data model (key-value vs OVSDB schema-defined relational). Etcd được nhiều hệ phân tán Go-based dùng (Kubernetes control plane chính là `etcd` cluster). Conceptually tương đương; vận hành tương tự (`etcdctl member list` ≈ `cluster/status`).
- **HashiCorp Consul Raft**: cùng Raft, library `hashicorp/raft`, dùng cho service discovery + KV. Khác: Consul có thêm consistency mode "stale" và "consistent" tương đương `--leader-only` của OVSDB nhưng granular hơn (per-request).
- **MongoDB replica set**: protocol khác (election + heartbeat lấy cảm hứng Raft nhưng custom), terminology khác (primary/secondary/arbiter). Hỗ trợ nhiều consistency mode (`readConcern: local/majority/linearizable`); OVSDB Raft chỉ có 2 mode (leader-only linearizable hoặc follower eventual).
- **PostgreSQL streaming replication + Patroni**: PostgreSQL bản thân không có consensus, chỉ có async/sync streaming. Patroni thêm consensus layer dùng `etcd` hoặc `Consul` ở ngoài để quyết định ai là primary. So với OVSDB Raft nội tại, kiến trúc tách rời này phức tạp hơn nhưng linh hoạt hơn (có thể đổi consensus backend).
- **Apache ZooKeeper (ZAB protocol)**: trước Raft thì nhiều hệ phân tán dùng ZAB (Zookeeper Atomic Broadcast) cho consensus. ZAB và Raft equivalent về fault tolerance + correctness, nhưng ZAB phức tạp hơn về explanation (lý do Raft ra đời). Yahoo, LinkedIn, Twitter từng dùng ZooKeeper rộng rãi; xu hướng dịch chuyển sang etcd/Raft sau 2015.
- **Galera Cluster (MariaDB, MySQL)**: synchronous multi-master qua certification-based replication, không phải Raft. Mỗi node accept write rồi broadcast certify với mọi peer trước khi commit. Khác Raft ở chỗ không có single-leader → hiệu năng cao hơn cho read-mostly nhưng deadlock detection phức tạp.

## §3.5.1 OFPT_HELLO (Type 0) — cornerstone

> Phase G v3.7 cohort C4 cornerstone DEEP-20.

### 1-7. Concept/History/Placement/Role/Motivation/Problem/Importance

`OFPT_HELLO` là first message exchange controller-switch sau TCP/SSL connect. Loại: OF protocol message, symmetric (both side send). Initial OF 1.0 (2009). Vai trò: version negotiation. Both side send HELLO với versions bitmap (1.0=0x01, 1.1=0x02, 1.2=0x04, 1.3=0x08, 1.4=0x10, 1.5=0x20). Highest common version chosen; nếu no overlap → connection close. Tier 1 cornerstone vì without HELLO, no OF connection establish.

### 8. Cơ chế

```
TCP connect (port 6653)
  ↓
Both side send OFPT_HELLO:
  header.type = 0
  header.version = highest version supported
  body: hello_elem_versionbitmap (versions field bitmap)
  ↓
Receive peer HELLO:
  parse versions bitmap
  determine highest common
  ↓
If common version >= 1.0 → continue
Else → send OFPT_ERROR (HELLO_FAILED) + close
```

OF 1.0 + 1.3 dùng version field; OF 1.4+ thêm versions bitmap element cho multi-version negotiation.

### 9-13. Operations

Skill: trace HELLO via tcpdump port 6653; verify version negotiated via ovs-ofctl show. Coupling: OFPT_FEATURES_REQUEST (sent ngay sau HELLO success), OFPT_ECHO_REQUEST (steady state keepalive), OFPT_ERROR (HELLO_FAILED variants).

### 14. Version drift

| Version | Negotiation method |
|---------|-------------------|
| OF 1.0/1.1/1.2 | Single version field (0x01/0x02/0x03) |
| OF 1.3 | header.version=0x04, single field |
| OF 1.4+ | versions bitmap element thêm cho multi-version cross |
| OVS 2.0+ | Support all OF 1.0-1.5 negotiation |

### 15. Verification

```
$ tcpdump -i any -nn 'port 6653' -X | head -20
→ First 2 packet sau TCP handshake = HELLO both side
→ Decode: 01 00 00 08 00 00 00 01 = version 1, type 0 (HELLO), length 8

$ ovs-ofctl show br0
→ Print "OF version 0x04 (OpenFlow13)" → confirms negotiated to 1.3
```

### 16. Source code

- `lib/ofp-msgs.c` `ofputil_decode_hello()` parse incoming HELLO
- `lib/connmgr.c` `connmgr_handle_packet_in()` dispatch sau handshake
- `lib/rconn.c` `rconn_run()` reconnect logic includes new HELLO

### 17. Production case

2023-Q2: Mismatched controller-switch version sau cluster upgrade. Switch OF 1.5 capable, controller stuck OF 1.3. HELLO negotiate OF 1.3 (lowest common). Some OF 1.5 features (egress table) silently disabled. Fix: upgrade controller to OF 1.5; verify via ovs-ofctl show.

### 18. Lab

```
$ ovs-vsctl set-controller br0 tcp:127.0.0.1:6653
$ tcpdump -i lo -nn 'port 6653' -X &
$ ovs-vsctl set-controller br0 tcp:127.0.0.1:6653 -- set Bridge br0 protocols=OpenFlow13
→ Observe HELLO exchange, verify negotiated version
```

### 19. Failure mode

- Mode A: No common version → OFPT_ERROR HELLO_FAILED. Recovery: configure compatible version.
- Mode B: HELLO timeout → connection retry. Recovery: check network reachability port 6653.
- Mode C: Asymmetric version mismatch (controller higher than switch claim) → silent feature degradation. Recovery: explicit `protocols=OpenFlowXX` config.

### 20. Cross-domain

- **NETCONF hello:** XML-based version negotiation. Tương tự concept. Khác: NETCONF capability list as XML elements, OF HELLO bitmap binary.
- **gRPC handshake:** HTTP/2 settings frame negotiation. Khác abstraction layer.
- **TLS ClientHello:** version + cipher negotiation. OF HELLO simpler scope (just version).

## §3.5.2 OFPT_FEATURES_REQUEST/REPLY (Types 5/6) — cornerstone

> Phase G v3.7 cohort C4 cornerstone DEEP-20.

### 1-7.

`OFPT_FEATURES_REQUEST` (controller→switch) query switch capability + port list. `OFPT_FEATURES_REPLY` (switch→controller) responds với datapath_id, n_buffers, n_tables, capabilities bitmap, port list (OF 1.0/1.1/1.2; ports moved to OFPMP_PORT_DESC OF 1.3+). Sent typically ngay sau HELLO success. Loại: OF protocol message pair, controller-to-switch + reply. Initial OF 1.0. Tier 1 cornerstone vì controller cần datapath_id để identify switch + n_tables để build flow rule.

### 8. Cơ chế

```
Controller → Switch: OFPT_FEATURES_REQUEST
  body: empty
  ↓
Switch → Controller: OFPT_FEATURES_REPLY
  datapath_id: 64-bit unique ID (often MAC + 16-bit prefix)
  n_buffers: 16-bit max packet buffer count
  n_tables: 8-bit total flow table count (OF 1.0 = 1; OF 1.1+ multi-table)
  auxiliary_id: 8-bit (OF 1.3+)
  capabilities: 32-bit bitmap
    OFPC_FLOW_STATS = 0x01 (support OFPMP_FLOW)
    OFPC_TABLE_STATS = 0x02
    OFPC_PORT_STATS = 0x04
    OFPC_GROUP_STATS = 0x08
    OFPC_IP_REASM = 0x20
    OFPC_QUEUE_STATS = 0x40
    OFPC_PORT_BLOCKED = 0x100
  ports[]: array of OFPPS_* (OF 1.0/1.1/1.2 only; OF 1.3+ split to OFPMP_PORT_DESC)
```

### 9-13. Operations

Skill: identify switch unique via datapath_id; check capabilities matrix; verify n_tables > 1 cho multi-table support. Coupling: OFPT_HELLO (predecessor), OFPMP_PORT_DESC (OF 1.3+ port list moved here), datapath_id usage trong OVN Chassis.external_ids:datapath-id.

### 14. Version drift

| Version | Body format |
|---------|-------------|
| OF 1.0 | datapath_id + n_buffers + n_tables + capabilities + actions + ports |
| OF 1.1/1.2 | + auxiliary_id placeholder |
| OF 1.3+ | Ports moved to OFPMP_PORT_DESC; FEATURES_REPLY body shorter |
| OF 1.4+ | actions field deprecated |

### 15. Verification

```
$ tcpdump -i any port 6653 -X | grep -A 20 "type 6"
→ Decode datapath_id offset 8-15

$ ovs-ofctl show br0
DPID: 0000aabbcc1122334
n_tables:254, n_buffers:0
capabilities: FLOW_STATS TABLE_STATS PORT_STATS QUEUE_STATS
```

### 16. Source code

- `lib/ofp-print.c` `print_features_reply()`
- `vswitchd/bridge.c` `bridge_get_features()` populate reply
- `lib/ofp-util.c` `ofputil_decode_features_reply()`

### 17. Production case

2024-Q1: Controller fail to identify switch despite TCP connect success. Investigation: FEATURES_REPLY datapath_id collision (2 switch sharing same MAC prefix). Fix: explicit datapath_id config via `ovs-vsctl set Bridge br0 other_config:datapath-id=0x123456789abc`.

### 18. Lab

```
$ ovs-ofctl show br0
$ tcpdump -i lo -nn -X port 6653 | grep -A 10 "type 6"
→ Decode FEATURES_REPLY hex
```

### 19. Failure mode

- Mode A: datapath_id collision in cluster. Diagnostic: 2 switch report same DPID. Recovery: explicit DPID config.
- Mode B: n_tables = 1 unexpected. Diagnostic: switch claim multi-table support but features reply n_tables=1. Recovery: bug investigation.
- Mode C: No reply timeout → controller assume switch dead. Recovery: check switch CPU + OF socket.

### 20. Cross-domain

- **NETCONF capabilities exchange:** similar feature query. Khác: NETCONF XML, OF binary.
- **SNMP sysDescr.0 query:** identification query similar. Khác: SNMP not OF-specific.

## §3.5.3 OFPT_FLOW_MOD (Type 14) — cornerstone

> Phase G v3.7 cohort C4 cornerstone DEEP-20.

### 1-7.

`OFPT_FLOW_MOD` controller→switch để add/modify/delete flow entry. Loại: controller-to-switch OF message. Initial OF 1.0. Tier 1 cornerstone — flow rule push fundamental cho mọi SDN application.

### 8. Cơ chế

```
Body fields:
  cookie: 64-bit opaque cookie (controller use for tagging)
  cookie_mask: cookie match for delete/modify (OF 1.1+)
  table_id: target table (OF 1.0 = 0xff = any; OF 1.1+ specific)
  command: OFPFC_ADD/MODIFY/MODIFY_STRICT/DELETE/DELETE_STRICT
  idle_timeout, hard_timeout: 16-bit each
  priority: 16-bit (higher = first match wins)
  buffer_id: 32-bit (reference packet buffer if modifying flow for queued packet)
  out_port, out_group: filter for delete (only flows output to specified)
  flags: OFPFF_SEND_FLOW_REM, OFPFF_CHECK_OVERLAP, OFPFF_RESET_COUNTS, OFPFF_NO_PKT_COUNTS, OFPFF_NO_BYT_COUNTS
  match: OXM/NXM TLV match list
  instructions: list of instruction (OF 1.1+) or actions (OF 1.0)
```

Atomicity: single FLOW_MOD = atomic per flow. Bundle atomic = group multiple FLOW_MOD atomic (OF 1.4+).

### 9-13. Operations

Skill: send FLOW_MOD via ovs-ofctl add-flow + observe via dump-flows; understand cookie strategy (per-controller tag cho rule ownership).

### 14. Version drift

| Version | Body changes |
|---------|--------------|
| OF 1.0 | match = simple struct + actions list |
| OF 1.1 | + table_id, instructions instead of actions |
| OF 1.2 | match = OXM TLV list |
| OF 1.3 | + cookie_mask, more flags |
| OF 1.5 | + egress table support, packet_type match |

### 15. Verification

```
$ ovs-ofctl add-flow br0 'priority=100,ip,actions=output:1' --bundle
$ tcpdump -i any -nn -X port 6653 | grep -A 20 "type 14"
→ Decode FLOW_MOD body

$ ovs-ofctl dump-flows br0
→ Confirm flow installed
```

### 16. Source code

- `lib/ofp-msgs.c` `ofputil_decode_flow_mod()`
- `ofproto/ofproto.c` `ofproto_flow_mod_init()`, `ofproto_flow_mod_finish()`
- `ofproto/ofproto-dpif.c` `flow_mod_init()` integrate với classifier

### 17. Production case

2023-Q4: Controller push 10K FLOW_MOD/sec → ovs-vswitchd CPU 100%. Investigation: each FLOW_MOD individual transaction → classifier rebuild overhead. Fix: use OFPT_BUNDLE (OF 1.4+) batch atomic, classifier rebuild once per bundle. Throughput 10K/sec maintained, CPU 30%.

### 18. Lab

```
$ ovs-ofctl add-flow br0 'priority=100,in_port=1,actions=output:2'
$ ovs-ofctl mod-flows br0 'priority=100,in_port=1,actions=output:3'
$ ovs-ofctl del-flows br0 'priority=100,in_port=1'
$ ovs-ofctl dump-flows br0
→ Verify each operation effect

# Bundle test (OF 1.4+):
$ ovs-ofctl --bundle add-flows br0 - <<EOF
priority=100,in_port=1,actions=output:2
priority=100,in_port=2,actions=output:1
priority=100,in_port=3,actions=output:4
EOF
→ All 3 flow atomic install
```

### 19. Failure mode

- Mode A: FLOW_MOD command field invalid → OFPT_ERROR. Recovery: validate command before send.
- Mode B: Match overlap với existing flow + OFPFF_CHECK_OVERLAP set → reject. Recovery: explicit priority differentiation.
- Mode C: buffer_id reference stale buffer → OFPT_ERROR BAD_BUFFER. Recovery: re-send PACKET_IN buffer.
- Mode D: Bundle commit partial fail → all-or-nothing. Recovery: review bundle order, fix root cause.

### 20. Cross-domain

- **gNMI Set:** model-based config push. Khác: gNMI tree-structured YANG; FLOW_MOD per-rule.
- **NETCONF edit-config:** XML transactional config. Tương tự bundle concept.
- **P4Runtime WriteRequest:** P4 table entry add/modify/delete. Khác: P4 schema-driven; OF flow-rule abstraction.

## §3.5.4 OFPT_PACKET_IN (Type 10) — cornerstone

> Phase G v3.7 cohort C4 cornerstone DEEP-20.

### 1-7.

`OFPT_PACKET_IN` switch→controller khi packet match flow rule có action `controller` HOẶC table-miss without explicit handling. Loại: switch-to-controller OF message. Initial OF 1.0. Tier 1 cornerstone — basis for "punt to controller" pattern (LLDP discovery, ARP responder, controller-based routing).

### 8. Cơ chế

```
Body fields:
  buffer_id: 32-bit (kernel buffer reference) hoặc 0xffffffff = no buffer
  total_len: 16-bit packet original length
  reason: OFPR_NO_MATCH (table-miss), OFPR_ACTION (action=controller), OFPR_INVALID_TTL
  table_id: 8-bit source table (OF 1.1+)
  cookie: 64-bit flow cookie that triggered (OF 1.3+)
  match: packet metadata (OF 1.2+)
  packet_data: actual packet bytes (truncated if > miss_send_len)
```

Truncation: switch may truncate packet to `miss_send_len` (default 128 bytes); controller can fetch full via buffer_id.

### 9-13. Operations

Skill: monitor PACKET_IN rate (controller can saturate); design flow rule that minimize PACKET_IN (avoid table-miss for high-rate traffic).

### 14. Version drift

| Version | Changes |
|---------|---------|
| OF 1.0 | Basic body |
| OF 1.1 | + table_id |
| OF 1.2 | + match metadata |
| OF 1.3 | + cookie of triggering rule |
| OF 1.5 | + packet_type field |

### 15. Verification

```
$ ovs-ofctl monitor br0 watch:resume
→ Real-time PACKET_IN stream

$ tcpdump -i any -nn -X port 6653 | grep -A 30 "type 10"
→ Decode PACKET_IN body
```

### 16. Source code

- `lib/ofp-msgs.c` `ofputil_decode_packet_in()`
- `vswitchd/bridge.c` send packet_in trigger
- `ofproto/ofproto-dpif.c` `process_upcall()` integrate

### 17. Production case

2023-Q1: Controller CPU 100% sustained. Investigation: PACKET_IN rate 10K/sec from misconfigured ARP responder rule (action=controller for ALL ARP). Fix: install ARP responder OF rule trên switch, only punt unknown destination.

### 18. Lab

```
$ ovs-ofctl add-flow br0 'priority=100,arp,actions=controller'
$ h1 arping h2
$ ovs-ofctl monitor br0 watch:resume
→ See PACKET_IN with ARP packet

$ ovs-ofctl add-flow br0 'priority=200,arp,arp_op=1,arp_spa=10.0.0.1,actions=load:0->NXM_OF_ARP_OP[],...'
→ Switch handle ARP locally, no PACKET_IN
```

### 19. Failure mode

- Mode A: PACKET_IN rate saturation. Diagnostic: controller CPU spike + OF socket buffer overflow. Recovery: install switch-side handling, reduce punt rate.
- Mode B: Truncated packet not fetched via buffer_id → controller decision incomplete. Recovery: increase miss_send_len OR fetch via buffer_id.
- Mode C: PACKET_IN reason=INVALID_TTL → packet TTL=0 punted. Recovery: design TTL handling rule.

### 20. Cross-domain

- **Linux netfilter NF_QUEUE:** packet escalate to userspace process. Tương tự concept.
- **eBPF perf event:** kernel-userspace packet sample. Khác abstraction.
- **Cisco IOS process switching:** packet escalate from fast-path to control plane. Same idea, vendor-specific.

## §3.5.5 OFPT_BARRIER_REQUEST/REPLY (Types 20/21) — cornerstone

> Phase G v3.7 cohort C4 cornerstone DEEP-20.

### 1-7.

`OFPT_BARRIER_REQUEST` controller→switch để force switch flush all preceding messages. `OFPT_BARRIER_REPLY` switch→controller confirm flushed. Loại: controller-to-switch synchronization. Initial OF 1.0. Tier 1 cornerstone — sync primitive critical cho deterministic flow install ordering.

### 8. Cơ chế

```
Controller sends multiple FLOW_MOD batch:
  FLOW_MOD-1 → switch (process async)
  FLOW_MOD-2 → switch (process async)
  FLOW_MOD-3 → switch (process async)
  BARRIER_REQUEST → switch
  ↓
Switch process all pending FLOW_MOD, then:
  BARRIER_REPLY → controller (confirm flushed)
  ↓
Controller now safe to assume FLOW_MOD-1/2/3 applied.
```

Without barrier, controller cannot determine when switch finished processing async messages.

### 9-13. Operations

Skill: insert BARRIER between flow install batch + verification action; use as "checkpoint" trong controller logic.

### 14. Version drift

| Version | Changes |
|---------|---------|
| OF 1.0 | Initial BARRIER_REQUEST/REPLY |
| OF 1.4+ | Bundle messages provide alternative atomic guarantee (less common to use barrier post-1.4) |

### 15. Verification

```
$ tcpdump -i any -nn port 6653 | grep -E "(type 20|type 21)"
→ Observe BARRIER request/reply pair

In ryu/onos controller: send_barrier_request() API call
```

### 16. Source code

- `lib/ofp-msgs.c` `ofputil_encode_barrier_reply()`
- `ofproto/connmgr.c` BARRIER processing in connection manager

### 17. Production case

2024-Q1: Controller flow install order non-deterministic. Investigation: 3 FLOW_MOD sent rapidly → switch process out-of-order, last applied wins → unexpected rule ordering. Fix: insert BARRIER between dependent FLOW_MOD groups.

### 18. Lab

```
# Without barrier
$ ovs-ofctl add-flow br0 'priority=100,actions=output:1' &
$ ovs-ofctl add-flow br0 'priority=100,actions=output:2' &
$ ovs-ofctl dump-flows br0
→ May show either output:1 or output:2 wins (race)

# With barrier (via bundle)
$ ovs-ofctl --bundle add-flows br0 - <<EOF
priority=100,in_port=1,actions=output:2
priority=200,in_port=1,actions=output:3
EOF
→ Atomic install, deterministic order
```

### 19. Failure mode

- Mode A: BARRIER timeout → controller hang waiting reply. Recovery: tune timeout + retry.
- Mode B: Partial fail before barrier → barrier confirm fail too. Recovery: bundle abort.

### 20. Cross-domain

- **POSIX fsync():** force flush write buffer. Same concept different domain.
- **GPU pipeline barrier:** synchronization primitive. Same abstraction.
- **MPI Barrier:** distributed synchronization. Same idea.

# Phần 17 - OVN L2 Forwarding: Từ kiến trúc nền tảng đến FDB Poisoning trên production

Môi trường thực hành: OpenStack (kolla-ansible) + Neutron ML2/OVN, Ubuntu Server 20.04+. Tham khảo chính: [ovn-sb(5)](https://man.openvswitch.org/ovn-sb.5.html), [ovn-nb(5)](https://man.openvswitch.org/ovn-nb.5.html), [OVN source code](https://github.com/ovn-org/ovn).

Sau khi hoàn thành phần này, bạn có thể:

1. Giải thích (Understand) tại sao OVN ra đời và kiến trúc hybrid centralized/distributed giải quyết những hạn chế gì của ML2/OVS agent-based
2. Phân tích (Analyze) vai trò của localnet port, MC_FLOOD, MC_UNKNOWN trong L2 forwarding trên provider network
3. Phân biệt (Analyze) FDB table (L2 switching) với MAC_Binding table (L3 routing) về mục đích, pipeline stage, và use case debug
4. Chẩn đoán (Evaluate) nguyên nhân gốc rễ FDB poisoning khi port_security disabled trên VNF port, và áp dụng fix ngắn hạn lẫn dài hạn
5. Xác minh (Apply) luồng packet qua logical pipeline bằng ovn-trace, ovs-ofctl, và ovn-sbctl trên hệ thống thực

Kiến thức tiên quyết:

- Ethernet switching cơ bản: MAC address table, unknown unicast flooding, broadcast domain (CCNA level)
- TCP/IP stack: ARP resolution, ICMP, VLAN tagging 802.1Q
- OpenStack Neutron: khái niệm provider network, security group, port binding
- OVS cơ bản: bridge, port, flow table, `ovs-vsctl`, `ovs-ofctl` (nếu chưa quen, tham khảo OVS tutorial tại docs.openvswitch.org)
- ovn-controller internals: OVSDB IDL, Port_Binding claiming, I-P engine, Logical Flow → OpenFlow translation (phần 13.7)
- ovn-northd NB→SB compilation: thuật toán dịch Logical_Switch/Router sang Logical_Flow (phần 13.8)

## 17.1 - Tại sao OVN tồn tại: bài toán gốc và hành trình giải quyết

### Từ OpenFlow đến OVN, dòng chảy lịch sử

Để hiểu OVN, cần quay lại **năm 2008**, khi **Nick McKeown** cùng 7 đồng tác giả (Tom Anderson, Hari Balakrishnan, Guru Parulkar, Larry Peterson, Jennifer Rexford, Scott Shenker, Jonathan Turner) công bố bài báo "OpenFlow: Enabling Innovation in Campus Networks" trên ACM SIGCOMM Computer Communications Review (Vol. 38, No. 2, pp. 69–74). Bài báo này, nay có hơn 9.000 trích dẫn (tính đến 2025), đặt nền móng cho khái niệm **Software-Defined Networking (SDN)**: tách biệt control plane khỏi data plane, cho phép lập trình chuyển mạch thông qua một giao thức chuẩn.

OpenFlow bắt nguồn từ nghiên cứu PhD của **Martin Casado** tại Stanford, project Ethane (publish tại SIGCOMM 2007, nghiên cứu bắt đầu từ 2006), nơi ý tưởng "một switch đơn giản với flow table có thể lập trình" lần đầu được hiện thực hóa. Chính Casado, cùng McKeown và Shenker, đã đồng sáng lập **Nicira Networks** vào năm 2007. Tại Nicira, hai kỹ sư then chốt là **Ben Pfaff** và **Justin Pettit** đã xây dựng **Open vSwitch (OVS)**, virtual switch mã nguồn mở dựa trên OpenFlow, trở thành xương sống của network virtualization. OVS được merge vào Linux kernel 3.3 (2012) và trở thành thành phần không thể thiếu trong cloud platform.

Năm 2012, VMware mua lại Nicira với giá 1,26 tỷ USD. Pfaff và Pettit tiếp tục phát triển OVS tại VMware. Như Pfaff chia sẻ (phỏng vấn USENIX 2025): *"Một lý do OVN thành công là team chúng tôi có lịch sử với network virtualization từ thời đầu của Open vSwitch. Tại Nicira, chúng tôi đã viết vài thế hệ network virtualizer. Mỗi lần, chúng tôi học được gì hiệu quả và gì cần tránh."*

### Trước OVN: Neutron ML2/OVS và giới hạn kiến trúc agent-based

Trước OVN, OpenStack Neutron dùng **ML2 (Modular Layer 2) plugin** với **OVS mechanism driver** cùng một loạt Python agent chạy trên mỗi compute node:

```
┌─────────────────────────────────────────────────────────┐
│                  Neutron Server                          │
│              (ML2 Plugin + OVS MechDriver)               │
└────────────────────┬────────────────────────────────────┘
                     │ RabbitMQ (AMQP RPC)
        ┌────────────┼────────────────┐
        ▼            ▼                ▼
┌──────────────┐ ┌──────────────┐ ┌──────────────┐
│ Compute #1   │ │ Compute #2   │ │ Network Node │
│ neutron-ovs  │ │ neutron-ovs  │ │ neutron-l3   │
│   -agent     │ │   -agent     │ │ neutron-dhcp │
│ br-int (NORM)│ │ br-int (NORM)│ │ neutron-meta │
│ br-tun       │ │ br-tun       │ │              │
│ br-ex/br-vlan│ │ br-ex/br-vlan│ └──────────────┘
└──────────────┘ └──────────────┘
```

Mỗi node chạy **neutron-openvswitch-agent**, một agent Python cấu hình OVS bridge và flow. Agent dùng ba bridge: **br-int** (integration bridge, sử dụng NORMAL action để delegate toàn bộ L2 forwarding cho OVS built-in MAC learning), **br-tun** (tunnel bridge, VXLAN/GRE), **br-ex / br-vlan** (provider bridge, kết nối physical network qua patch port pairs).

Vấn đề cốt lõi nằm ở NORMAL action trên br-int. Khi br-int dùng NORMAL, OVS tự học MAC address giống switch vật lý, hoàn toàn ngoài tầm kiểm soát của CMS. Hệ quả trực tiếp: CMS không có visibility vào MAC table nên không thể kiểm soát hoặc kiểm chứng forwarding decision; unknown unicast flooding ra tất cả port trên bridge gây bandwidth waste; security group phải triển khai thông qua Linux bridge + iptables riêng cho mỗi VM (vì OVS chưa hỗ trợ conntrack tốt ở thời điểm đó); và RabbitMQ trở thành single point of failure khi mọi thay đổi topology đều phải đi qua message broker. Bug #1866445 của Launchpad ghi nhận rõ: "br-int trên compute node không learn được MAC của VM trên compute khác, hiệu năng VM-to-VM giảm tuyến tính", hệ quả trực tiếp của việc MAC learning bị giới hạn trong phạm vi một bridge cục bộ.

### OVN ra đời, 13 tháng 1 năm 2015

Ngày 13/01/2015, **Justin Pettit, Ben Pfaff, Chris Wright** (Red Hat CTO), **và Madhu Venugopal** công bố OVN trên blog Network Heresy: *"Today we are excited to announce Open Virtual Network (OVN), a new project that brings virtual networking to the OVS user community."*

**Russell Bryant**, kỹ sư Red Hat, nhanh chóng trở thành người thúc đẩy chính OVN trong OpenStack: *"The timing could not have been better. I dove in with a goal of figuring out whether OVN could be a promising next generation of Open vSwitch integration for OpenStack."* Bryant viết code tích hợp OVN-OpenStack đầu tiên và xuất bản loạt blog kỹ thuật trên blog.russellbryant.net.

Phiên bản ổn định đầu tiên đi kèm **OVS 2.6.0** (tháng 9/2016). Tích hợp Neutron thông qua networking-ovn bắt đầu từ **OpenStack Newton** (10/2016). Đến năm 2019, OVN tách thành repo riêng tại github.com/ovn-org/ovn, dấu mốc quan trọng thể hiện sự trưởng thành của project.

### Kiến trúc OVN, hybrid centralized/distributed với hai database

```
┌───────────────────────────────────────────────────────┐
│            CMS (Neutron / Kubernetes)                   │
└──────────────────┬────────────────────────────────────┘
                   │ OVSDB Protocol
                   ▼
┌───────────────────────────────────────────────────────┐
│           OVN Northbound Database (NB DB)               │
│ Logical_Switch │ Logical_Router │ ACL │ DHCP_Options    │
│ Logical_Switch_Port │ Logical_Router_Port │ ...         │
└──────────────────┬────────────────────────────────────┘
                   │ ovn-northd dịch NB → SB
                   ▼
┌───────────────────────────────────────────────────────┐
│           OVN Southbound Database (SB DB)               │
│ Logical_Flow │ Datapath_Binding │ Port_Binding          │
│ Chassis │ Multicast_Group │ FDB │ MAC_Binding           │
└──────┬──────────────┬──────────────┬──────────────────┘
       ▼              ▼              ▼
  ovn-controller  ovn-controller  ovn-controller
  (Chassis 1)     (Chassis 2)     (Chassis 3)
       ↓              ↓              ↓
  OpenFlow → br-int  OF → br-int  OF → br-int
```

**NB DB** là giao diện giữa CMS và OVN. Schema dùng khái niệm mạng quen thuộc với cloud operator: `Logical_Switch`, `Logical_Switch_Port` (cột quan trọng: `name`, `type`, `addresses`, `port_security`, `enabled`), `Logical_Router`, `ACL`. Khi Neutron tạo network hoặc port, nó ghi trực tiếp vào NB DB thông qua OVSDB protocol, không cần message broker trung gian.

**SB DB** có schema hoàn toàn khác, biểu diễn mạng dưới dạng **logical flow** (match-action rule dùng khái niệm logic). Các bảng chính gồm `Logical_Flow`, `Datapath_Binding`, `Port_Binding`, `Chassis`, `Multicast_Group`, `FDB`, và `MAC_Binding`. Sự tách biệt giữa NB DB (ý định mạng) và SB DB (hiện thực hóa mạng) là thiết kế kiến trúc cốt lõi, cho phép CMS nói ngôn ngữ trừu tượng trong khi hệ thống bên dưới dịch sang OpenFlow cụ thể.

**ovn-northd** là daemon tập trung, đọc NB DB và dịch sang SB DB. Mỗi logical switch hoặc logical router được chuyển thành một logical datapath kèm tập logical flow tương ứng. Đây là computation deployment-wide, chỉ chạy một instance (hoặc active-standby pair cho HA).

**ovn-controller** chạy trên mỗi chassis, đọc logical flow từ SB DB rồi dịch thành OpenFlow flow và program vào OVS. Bryant mô tả phân chia công việc này: *"OVN breaks this problem up: first calculates logical flows, then distributes to ovn-controller on each node which converts to physical flows."* Computation node-specific và phân tán hoàn toàn: mỗi chassis chỉ cài flow liên quan đến port mà nó host.

### Tại sao br-int không còn NORMAL action

Sự khác biệt kiến trúc sâu sắc nhất giữa ML2/OVS và OVN nằm ở cách br-int vận hành. OVN đặt br-int ở chế độ **fail-secure**, flow table bắt đầu rỗng, mọi forwarding phải được program rõ ràng bởi ovn-controller. Từ OVN tutorial: *"It's common for an OVN logical switch to know all the MAC addresses supported by its logical ports... That's why there's no logic here for MAC learning or flooding."*

Security group không cần Linux bridge + iptables nữa, OVN dùng **kernel conntrack trực tiếp** từ OVS (ct action trong OpenFlow), đơn giản hóa đáng kể data path. Không còn RabbitMQ cho network topology changes, toàn bộ communication giữa central và chassis đi qua OVSDB. Red Hat OpenStack Platform 16.0 (2020) đánh dấu bước chuyển chính thức: ML2/OVN trở thành default mechanism driver, thay thế ML2/OVS đã phục vụ suốt nhiều năm trước đó.

> **Key Topic:** OVN biết trước mọi MAC address (từ cột `addresses` trên Logical_Switch_Port) và program explicit flow cho từng MAC. Khi tài liệu này nói về FDB learning ở các phần sau, đó là **trường hợp ngoại lệ** khi OVN KHÔNG biết trước MAC, cụ thể là trên port có `port_security` disabled và addresses chứa keyword `"unknown"`. Đây là khác biệt kiến trúc cốt lõi giữa OVN (explicit programming) và switch vật lý (dynamic learning).

### Hệ thống tunnel key, ngôn ngữ nhận diện đối tượng logic

Mỗi đối tượng logic trong OVN, logical switch, logical router, logical port, multicast group, được gán một số nguyên duy nhất gọi là **tunnel key**. Hệ thống này là xương sống của cơ chế truyền tải packet giữa các chassis qua Geneve tunnel, đồng thời là cách ovn-controller nhận diện đối tượng khi program OpenFlow rules trên br-int.

**Datapath tunnel key (dp_key)** là số 24-bit (1–16,777,215) gán cho mỗi logical switch hoặc logical router, lưu trong `Datapath_Binding.tunnel_key`. Khi packet rời một chassis đi qua Geneve tunnel đến chassis khác, dp_key được encode vào **VNI field** (Virtual Network Identifier) trong Geneve header, trường này có đúng 24 bit, nên dp_key map 1:1 vào VNI. Trên mặt OpenFlow, dp_key map sang register **metadata**: mọi flow match `metadata=0x752` chính là đang match trên logical switch có dp_key=1874 (0x752 hex). Đây là cách ovn-controller phân biệt packet thuộc về datapath nào khi br-int phục vụ nhiều logical switch đồng thời.

**Port tunnel key (port_key)** là số 15-bit (1–32,767) gán cho mỗi logical port trong một datapath, lưu trong `Port_Binding.tunnel_key`. Mỗi datapath duy trì namespace riêng, port_key=3 trên logical switch A và port_key=3 trên logical switch B là hai port hoàn toàn khác nhau, vì dp_key khác nhau phân biệt chúng. Trong Geneve header, port_key không nằm trong VNI mà encode vào **TLV option** (class 0x0102, type 0x80), đây là metadata bổ sung mà Geneve hỗ trợ (GRE và VXLAN không có cơ chế tương đương). Trên mặt OpenFlow, port_key map sang **reg14** (MFF_LOG_INPORT, logical ingress port) và **reg15** (MFF_LOG_OUTPORT, logical egress port).

**Multicast group tunnel key** nằm trong dải cao 32,768–65,535, gán cho multicast group, lưu trong `Multicast_Group.tunnel_key`. Dải này tách biệt hoàn toàn khỏi dải port_key (1–32,767) nên không bao giờ xung đột. `_MC_flood` luôn nhận tunnel_key 32768, `_MC_unknown` nhận 32769, đây là hai giá trị cố định được hardcode trong `lib/mcast-group-index.h`.

Điểm then chốt để hiểu các phần sau: **localnet port tuy không dùng Geneve tunnel** (traffic đi qua patch port → br-provider → physical NIC), nhưng vẫn được gán tunnel_key trong SB DB. Lý do: ovn-controller cần port_key để nhận diện localnet port trong logical pipeline, match reg14/reg15 cho ACL, QoS, FDB lookup đều dựa trên port_key. Không có tunnel_key, localnet port sẽ "vô hình" trong pipeline. Tương tự, FDB table (mục 17.4) lưu `dp_key` và `port_key` trực tiếp thay vì logical port name, commit message của Numan Siddique giải thích: *"This makes it easier for ovn-controller to program the OF rules"*, bởi OpenFlow rules match trên số nguyên (register values), không phải chuỗi ký tự.

> **Key Topic:** Ba loại tunnel key (dp_key 24-bit, port_key 15-bit, multicast key 16-bit) tạo thành hệ thống địa chỉ nội bộ của OVN. Khi packet đi qua Geneve: dp_key encode vào VNI, port_key encode vào TLV option. Khi packet ở local (br-int): dp_key nằm trong metadata, port_key nằm trong reg14/reg15. Hiểu hệ thống này là nền tảng để đọc OpenFlow dump và debug bằng ovn-trace.

## 17.2 - Localnet port: cầu nối giữa thế giới ảo và VLAN vật lý

### Bài toán kết nối overlay với physical VLAN

OVN được thiết kế cho overlay network (Geneve/VXLAN tunnel), nhưng thực tế production luôn đòi hỏi kết nối VM trực tiếp vào VLAN-based physical network (ví dụ VLAN 3808 trong case study ở mục 17.6).

Loại network này trong OpenStack gọi là **provider network**: network do administrator tạo, kết nối trực tiếp vào physical network infrastructure (thường là một VLAN segment). Tên "provider" xuất phát từ việc nhà cung cấp hạ tầng cung cấp kết nối này, khác với tenant/self-service network (overlay, do user tự tạo qua Geneve/VXLAN tunnel). Provider network không nhất thiết là public internet, trong case study, VLAN 3808 (100.64.0.0/28) là transit network nội bộ kết nối VM với Fornix gateway. Đặc điểm kỹ thuật phân biệt hai loại: provider network dùng localnet port trên OVN và traffic đi qua physical NIC, còn tenant network đi qua Geneve tunnel giữa các chassis.

Trước OVN, Neutron ML2/OVS dùng br-ex / br-vlan bridge với patch port pair: agent cấu hình OpenFlow rule swap internal VLAN tag ↔ physical VLAN tag. Patch port tồn tại trên mọi node bất kể có VM dùng network đó không, gây lãng phí tài nguyên đáng kể trên hệ thống lớn.

### Thiết kế localnet port. "port đặc biệt" đại diện cả physical network

OVN giải quyết bài toán kết nối vật lý bằng **localnet port type**, một loại Logical_Switch_Port đặc biệt:

```
ovn-nbctl lsp-add provider-ls ln-provider
ovn-nbctl lsp-set-type ln-provider localnet
ovn-nbctl lsp-set-addresses ln-provider unknown
ovn-nbctl lsp-set-options ln-provider network_name=physnet1
ovn-nbctl set logical_switch_port ln-provider tag=3808
```

> **Key Topic:** Localnet port **KHÔNG đại diện cho một thiết bị** (một MAC). Nó đại diện cho **toàn bộ physical network segment**. Đây là khác biệt cốt lõi so với VIF port, một VIF port map 1:1 với một VM tap interface, trong khi localnet port map 1:N với toàn bộ thiết bị trên physical VLAN.

Cơ chế kết nối dựa trên **bridge mapping**, cấu hình trên mỗi chassis xác định physical bridge nào tương ứng với network name nào:

```
Physical Network (VLAN 3808)
         │
    ┌────┴─────────┐
    │ br-provider   │  ← OVS bridge, kết nối physical NIC (bond3)
    └────┬─────────┘
         │ patch port (ovn-controller tạo tự động)
    ┌────┴─────────┐
    │   br-int      │
    │               │
    │ VM .12 port   │  ← VIF port: LSP type="" đại diện VM interface
    │ VM .xx port   │  ← VIF port: LSP type="" đại diện VM interface
    │ localnet port │  ← Đại diện TOÀN BỘ VLAN 3808
    └───────────────┘
```

Cấu hình bridge mapping: `ovs-vsctl set open . external-ids:ovn-bridge-mappings=physnet1:br-provider`

Khi ovn-controller phát hiện có ít nhất một VM trên node thuộc provider network này, nó tự động tạo patch port nối br-int với br-provider. Ngược lại, nếu node không host VM nào trên network đó, ovn-controller không tạo patch port và giải phóng tài nguyên tương ứng. Đây là cải tiến so với ML2/OVS cũ, nơi agent tạo patch port trên mọi node bất kể có VM hay không.

Trong SB DB, mọi Logical_Switch_Port (bao gồm cả localnet LSP) đều có một row tương ứng trong bảng Port_Binding. Với localnet LSP, Port_Binding có đặc điểm riêng biệt: **`chassis: []`** (không bind vào chassis cụ thể nào), bởi localnet port "tồn tại ở khắp nơi" thông qua bridge mapping trên từng chassis:

```
_uuid       : 8427506e-...
logical_port: "provnet-e4abf6df-..."
mac         : [unknown]
options     : {network_name=physnet1}
tunnel_key  : 1
type        : localnet
chassis     : []          ← KHÔNG bind vào chassis!
```

### "provnet-" vs "localnet", hai tên gọi, hai tầng khác nhau

Khi đọc output `ovn-sbctl list port_binding`, bạn sẽ thấy localnet port có `logical_port: "provnet-e4abf6df-..."` nhưng `type: localnet`. Hai tiền tố khác nhau này gây nhầm lẫn cho người mới tiếp cận OVN, nhưng thực chất chúng hoạt động ở hai tầng hoàn toàn khác nhau.

**"localnet"** là OVN port type, khái niệm thuộc về OVN core, định nghĩa trong ovn-sb(5). Khi `Port_Binding.type = "localnet"`, ovn-controller trên mỗi chassis hiểu rằng port này đại diện cho kết nối đến physical network cục bộ thông qua bridge mapping. OVN bản thân chỉ nhìn thấy type và `options:network_name`, nó không quan tâm tên logical_port là gì.

**"provnet-"** là Neutron naming convention, quy ước đặt tên do Neutron ML2/OVN driver tạo ra. Khi Neutron tạo provider network, driver sinh localnet LSP với tên theo pattern `provnet-<network-segment-UUID>`. Tên này hoàn toàn là metadata quản trị, nếu dùng OVN standalone (không qua OpenStack), có thể đặt tên localnet port là bất kỳ chuỗi nào.

Không có sự phân biệt nào ở tầng OVN giữa "provnet" và "localnet". Đây là quan hệ CMS naming ↔ OVN type, giống như cách Neutron đặt tên VIF port là `<port-UUID>` trong khi OVN chỉ quan tâm `type=""` và chassis binding. Khi debug bằng `ovn-sbctl`, nếu thấy `logical_port` bắt đầu bằng "provnet-" thì đó chắc chắn là localnet port trên hệ thống OpenStack; trên Kubernetes (OVN-Kubernetes), localnet port có thể có naming convention khác hoàn toàn.

### Hiểu sai thường gặp về localnet port

> **Hiểu sai:** "Localnet port hoạt động giống bridge port trên switch vật lý, nó học MAC và forward frame dựa trên MAC table."
>
> **Thực tế:** Localnet port trong OVN không học MAC (trừ khi bật `localnet_learn_fdb=true`, mặc định off). Nó không có MAC riêng, không tham gia MAC learning pipeline mặc định, và addresses của nó là keyword `"unknown"`, nghĩa là nó nhận mọi frame có destination MAC không match bất kỳ port nào đã biết. Đây là catch-all, không phải learning.

> **Hiểu sai:** "Localnet port bind vào một chassis cụ thể giống VIF port."
>
> **Thực tế:** Port_Binding của localnet port có `chassis: []`, nó không bind vào chassis nào. ovn-controller trên mỗi chassis có bridge mapping tương ứng sẽ tự tạo patch port khi cần. Localnet port "tồn tại ở khắp nơi" qua bridge mapping, không qua chassis binding.

### Tại sao addresses chứa "unknown", quyết định thiết kế then chốt

OVN không biết trước MAC nào tồn tại trên physical network, hàng trăm server, router, firewall mà OVN không quản lý. `"unknown"` là keyword đặc biệt trong OVN: port có address "unknown" nhận mọi packet có destination MAC không khớp bất kỳ port đã biết nào. Nó hoạt động như catch-all cho unknown unicast traffic, và port mang keyword này được thêm vào multicast group `_MC_unknown`, nhóm chuyên dụng cho unknown unicast forwarding mà phần tiếp theo sẽ phân tích chi tiết.

ovn-nb(5): *"Each bridged logical switch has one or more localnet ports, which have only one special address 'unknown'."*

### Chassis, Claim và Multichassis, cơ chế ownership của port trong OVN

Mục 17.2 vừa chỉ ra rằng localnet port có `chassis: []` trong Port_Binding. Nhưng một VIF port (VM) thì sao? Câu trả lời là VIF port được **claim** bởi một chassis cụ thể. Hiểu được cơ chế claim và trạng thái multichassis là điều kiện tiên quyết để đọc case study ở mục 17.6, nơi toàn bộ cơ chế FDB poisoning diễn ra trong một "multichassis window" kéo dài vài giây.

**Chassis** trong OVN là một physical compute node, mỗi hypervisor host nơi VM chạy. Khi ovn-controller khởi động trên một node, nó tự đăng ký vào SB DB bằng cách tạo một row trong bảng `Chassis` với `name` là hostname của node (ví dụ: `vhhl1c2wc1tsn01`) và `system_id` là UUID duy nhất. Bảng này là nền tảng để OVN biết compute node nào đang hoạt động và có thể nhận port binding.

> **Hiểu sai:** "Chassis là physical NIC hoặc là bond interface trên server."
>
> **Thực tế:** Chassis là toàn bộ compute node (hypervisor host), không phải interface. Một node dù có 2 hay 10 NIC vẫn chỉ là một chassis duy nhất. Interface vật lý được thể hiện qua `bridge mappings` trong `external-ids` của OVS, hoàn toàn tách biệt khỏi khái niệm chassis.

**Claim** là hành động ovn-controller viết tên chassis vào trường `Port_Binding.chassis` của một LSP. Cơ chế hoạt động như sau: khi VM được tạo trên một node, QEMU tạo tap interface với `external-ids:iface-id` bằng UUID của Neutron port. ovn-controller chạy liên tục, poll bảng `Interface` của OVS local. Khi phát hiện một interface có `iface-id` khớp với tên của một LSP trong SB DB, nó gọi hàm `claim_lport()` trong `controller/binding.c`, ghi chassis của mình vào `Port_Binding.chassis`. Từ thời điểm đó, port được coi là "owned" bởi chassis này: chỉ chassis đó mới program OpenFlow rules cho port đó, chỉ chassis đó mới nhận traffic đến port đó từ Geneve tunnel.

Log thực tế khi claim xảy ra trên chassis `vhhl1c2wc1tsn01`:

```
[ovn-controller] INFO binding  Claiming lport 0a17b4f8-736d-4bf5-ba1e-335d17cb5973 for this chassis.
[ovn-controller] INFO binding  0a17b4f8-736d-4bf5-ba1e-335d17cb5973: Claiming fa:16:3e:04:74:74 100.64.0.12
[ovn-controller] INFO binding  0a17b4f8-736d-4bf5-ba1e-335d17cb5973: Claiming unknown
```

Dòng "Claiming unknown" xác nhận rằng port có `addresses` chứa keyword `"unknown"`, điều kiện thứ hai để FDB learning kích hoạt (phân tích tại mục 17.4).

**Multichassis** là trạng thái tạm thời khi một port được bound đồng thời vào hai chassis: chassis nguồn (nơi VM đang chạy) và chassis đích (nơi VM sắp được chuyển đến). Trạng thái này chỉ xảy ra trong live migration window. Kỹ thuật triển khai dựa vào hai trường phân biệt trong Port_Binding: `chassis` lưu chassis primary (owner hiện tại) và `additional_chassis` lưu chassis phụ (nơi VM đang được chuyển đến). Khi Neutron live-migrate, nó đặt `options:requested-chassis = "chassisA,chassisB"` trong LSP. ovn-northd tách chuỗi này thành `PB.requested_chassis = chassisA` và `PB.requested_additional_chassis = [chassisB]`. Trên chassisB, hàm `lport_can_bind_on_this_chassis()` nhận diện mình trong `requested_additional_chassis` và trả về enum `CAN_BIND_AS_ADDITIONAL` thay vì `CAN_BIND_AS_MAIN`. `claim_lport()` cập nhật `PB.additional_chassis` với chassisB, port lúc này là multichassis.

Khi `PB.additional_chassis` không rỗng, hàm `shash_is_empty(&ld->multichassis_ports)` trong `controller/physical.c` trả về `false`. Điều này kích hoạt `enforce_tunneling_for_multichassis_ports()`, cài đặt OpenFlow rule trong `OFTABLE_REMOTE_OUTPUT` cưỡng chế mọi traffic đến port đó qua Geneve tunnel giữa hai chassis, kể cả khi source và destination là cùng physical VLAN. Đây là điều kiện trigger đầu tiên dẫn đến FDB poisoning mà case study sẽ phân tích: tunnel buộc traffic đi qua Geneve, Geneve thêm 58 byte overhead, packet quá cỡ, ICMP error được sinh ra với bug swap MAC mà không swap inport.

> **Key Topic:** Multichassis không phải trạng thái HA thường trực. Nó là trạng thái tạm thời trong live migration window, thường kéo dài 5–10 giây. Sau khi migration hoàn tất, `PB.additional_chassis` trở về rỗng và toàn bộ cơ chế tunnel enforcement biến mất. Chỉ nova live-migrate tạo trạng thái này, mọi thao tác Nova khác (boot, stop, cold migration, rebuild) không bao giờ đặt `requested-chassis` dạng danh sách phân cách dấu phẩy.

Lệnh kiểm tra trạng thái multichassis thời gian thực trên OVN central:

```bash
[controller]$ ovn-sbctl --columns=logical_port,chassis,additional_chassis list Port_Binding
```

Khi migration đang diễn ra, cột `additional_chassis` sẽ chứa UUID thay vì rỗng trên port của VM đang được migrate.

### localnet_learn_fdb, mở rộng FDB learning cho localnet port

Ban đầu localnet port bị loại hoàn toàn khỏi FDB learning vì lý do scale, physical network có thể có hàng trăm MAC đi qua, learning tất cả sẽ nhanh chóng tiêu tốn FDB table entries. Tuy nhiên, với sự phát triển của hardware offload (NVIDIA/Mellanox SmartNIC) và nhu cầu giảm flooding trên provider network nhỏ, Ales Musil (amusil@redhat.com) cùng Numan Siddique đã thêm tùy chọn `localnet_learn_fdb` trong OVN v22.09.0 (commit `93514df0d4c8fe7986dc5f287d7011f420d1be6d`, Red Hat Bugzilla #2070529). Cấu hình là opt-in, mặc định off: `ovn-nbctl set logical_switch_port <localnet-lsp> options:localnet_learn_fdb=true`. Cách tính năng này tương tác với FDB learning sẽ được phân tích kỹ tại mục 17.4.

## 17.3 - MC_FLOOD và MC_UNKNOWN: broadcast và unknown destination trong distributed control plane

### Bài toán flood packet khi logical switch trải nhiều chassis

Trên switch vật lý, broadcast và unknown unicast flooding đơn giản: chip ASIC replicate frame ra tất cả port thuộc cùng VLAN trong vài nanosecond. Trong OVN, bài toán phức tạp hơn bởi logical switch trải qua nhiều chassis, mỗi chassis chỉ biết port cục bộ mà nó host. Khi một VM gửi broadcast frame, ovn-controller cục bộ cần biết frame đó phải đến những chassis nào, qua port nào trên mỗi chassis.

OVN giải quyết bằng **pseudo-multicast group** trong SB DB, lưu trong bảng `Multicast_Group`. Mỗi group chứa danh sách Port_Binding thành viên, và khi packet có `outport` được set bằng tên group, ovn-controller replicate packet đến mọi thành viên, bao gồm cả port trên remote chassis qua Geneve tunnel.

### Schema Multicast_Group (ovn-sb(5))

| Column | Type | Mô tả |
|------------|-------------------|-------------------------------------------|
| datapath | Datapath_Binding | Logical switch chứa group |
| tunnel_key | int [32768-65535] | ID duy nhất (không trùng logical port ID) |
| name | string | Tên group (unique trong datapath) |
| ports | set of Port_Binding | Danh sách port thành viên |

Source code (`lib/mcast-group-index.h`):
```c
enum ovn_mcast_tunnel_keys {
    OVN_MCAST_FLOOD_TUNNEL_KEY = 32768,    // _MC_flood
    OVN_MCAST_UNKNOWN_TUNNEL_KEY,           // _MC_unknown (32769)
    OVN_MCAST_MROUTER_FLOOD_TUNNEL_KEY,     // 32770
    ...
};
```

### Ba multicast group chính và thành viên của chúng

| Group | Tunnel Key | Thành viên | Mục đích |
|----------------|-----------|----------------------------------------|---------------------------------------|
| `_MC_flood` | 32768 | Tất cả enabled port (VIF+router+localnet) | Broadcast/multicast (ARP request,...) |
| `_MC_flood_l2` | derived | Tất cả port TRỪ router port | L2 multicast (không IP/ARP) |
| `_MC_unknown` | 32769 | CHỈ port có addresses="unknown" | Unknown unicast forwarding |

> **Key Topic:** `_MC_unknown` được tạo **có điều kiện**, chỉ khi ít nhất một port trên logical switch có `"unknown"` trong addresses. Trên overlay-only switch (mọi MAC đã biết từ LSP addresses), MC_UNKNOWN không tồn tại và unknown unicast bị drop. Đây là lý do provider network (có localnet port với addresses="unknown") luôn có MC_UNKNOWN, còn tenant overlay network thì không.

Sự khác biệt giữa MC_FLOOD và MC_UNKNOWN là then chốt để hiểu tại sao ARP vẫn hoạt động bình thường trong khi unicast traffic bị FDB poisoning ảnh hưởng (case study mục 17.6). MC_FLOOD nhận mọi broadcast và multicast frame, bao gồm ARP Request và flood đến tất cả enabled port. MC_UNKNOWN chỉ nhận unknown unicast frame và flood đến nhóm port hẹp hơn: chỉ những port có "unknown" addresses.

### Luồng xử lý trong pipeline, packet walk-through chi tiết

```
           Logical Switch Ingress Pipeline
           ═══════════════════════════════
  Table 0-1: ls_in_port_sec_l2 / ls_in_port_sec_ip
       ↓
  Table N: ls_in_lookup_fdb  (*)
     → reg0[11] = lookup_fdb(inport, eth.src)   [kiểm tra FDB]
       ↓
  Table N+1: ls_in_put_fdb  (*)
     → if reg0[11]==0: put_fdb(inport, eth.src)  [HỌC MAC nếu chưa có]
       ↓
  ... ACL, QoS, DHCP, DNS, ARP responder, ...
       ↓
  Table M: ls_in_l2_lkup (L2 Destination Lookup)  (*)
     Priority 72: eth.mcast && arp → outport = _MC_flood     [ARP broadcast]
     Priority 70: eth.mcast → outport = _MC_flood_l2         [L2 multicast]
     Priority 50: eth.dst==<known_mac> → outport=<port>      [Static known]
     Priority  0: outport = get_fdb(eth.dst); next            [FDB lookup]
       ↓
  Table M+1: ls_in_l2_unknown  (*)
     Priority 50: outport=="none" → outport=_MC_unknown       [Flood unknown]
     Priority  0: output                                       [FDB found → fwd]

  (*) Lưu ý: table number thay đổi theo OVN version. Ví dụ trên OVN v21.06,
  ls_in_lookup_fdb ≈ table 2, ls_in_put_fdb ≈ table 3; trên các version mới hơn
  (v23.09+), vị trí có thể dịch chuyển do thêm stage mới. Kiểm tra bằng:
  ovn-sbctl lflow-list <datapath> | grep ls_in_lookup_fdb
```

Luồng xử lý unknown unicast (chưa có FDB entry) diễn ra qua hai stage then chốt. Tại `ls_in_l2_lkup` (priority 0), action `get_fdb(eth.dst)` tìm kiếm MAC đích trong FDB table. Không tìm thấy, nên outport được set về "none" (giá trị 0). Tại `ls_in_l2_unknown` (priority 50), match `outport == "none"` khớp, action set `outport = "_MC_unknown"` rồi thực thi `output`. OVN flood bản copy packet đến mọi thành viên trong group MC_UNKNOWN, bao gồm cả localnet port. Khi packet đến localnet port, nó đi qua patch port vào br-provider, rồi ra physical NIC (bond3) vào VLAN vật lý.

Luồng known unicast (có FDB entry) khác biệt ở bước đầu tiên. Tại `ls_in_l2_lkup` (priority 0), `get_fdb(eth.dst)` tìm thấy entry trong FDB table và set outport bằng port_key tương ứng. Tại `ls_in_l2_unknown` (priority 0), outport đã có giá trị khác "none" nên không match điều kiện flood, packet đi thẳng đến port mà FDB chỉ định.

Dưới đây là ovn-trace output thực tế minh họa luồng unknown unicast (GitHub Issue #244):

```
ingress(dp="non_fdp", inport="07188e")
 0. ls_in_port_sec_l2: inport == "07188e", priority 50 → next;
 3. ls_in_lookup_fdb:  inport == "07188e", priority 100
    reg0[11] = lookup_fdb(inport, eth.src); next;
24. ls_in_l2_lkup: 1, priority 0
    outport = get_fdb(eth.dst); next;
25. ls_in_l2_unknown: outport == "none", priority 50
    outport = "_MC_unknown"; output;

multicast(dp="non_fdp", mcgroup="_MC_unknown")
  → egress: provnet-40617a (type "localnet")   ← Ra physical network
  → egress: RTR3-1-port-128, 3b461e, 877021, ...
```

### Tại sao MC_UNKNOWN không thể thiếu cho provider network

Không có MC_UNKNOWN, provider network không hoạt động. OVN không biết trước MAC của gateway vật lý (100.64.0.1 trong case study) vì gateway không phải là VM do Neutron quản lý, MAC address của nó không nằm trong bất kỳ LSP addresses nào. Khi VM gửi frame đến gateway, destination MAC không khớp bất kỳ static flow nào ở priority 50 trong `ls_in_l2_lkup`. Nếu không có cơ chế "unknown" forwarding, `get_fdb` trả về outport="none" và không có MC_UNKNOWN để flood, frame bị drop. MC_UNKNOWN đóng vai trò cánh cửa duy nhất để traffic rời khỏi thế giới ảo OVN ra physical network khi destination MAC chưa được biết trước.

> **Hiểu sai:** "MC_UNKNOWN giống broadcast, flood đến tất cả port trên logical switch."
>
> **Thực tế:** MC_UNKNOWN chỉ flood đến các port có `"unknown"` trong addresses. Trên một logical switch có 50 VIF port nhưng chỉ 2 port có addresses="unknown" (localnet port và một VNF port), MC_UNKNOWN chỉ có 2 member, không phải 50. Broadcast (ARP request) đi qua `_MC_flood` với priority cao hơn (72 > 0) và flood đến tất cả enabled port.

> **Hiểu sai:** "Nếu FDB không tìm thấy MAC, packet bị drop."
>
> **Thực tế:** Khi `get_fdb` trả về outport="none", packet đi vào `ls_in_l2_unknown` (priority 50) và được redirect đến `_MC_unknown`, flood đến mọi port có "unknown" addresses. Packet chỉ bị drop khi MC_UNKNOWN không tồn tại (overlay-only switch không có port nào mang "unknown").

### ▶ Guided Exercise 1: Chứng minh unknown unicast đi qua MC_UNKNOWN đến localnet port

Trong bài thực hành này, bạn chứng minh rằng khi destination MAC chưa có trong FDB, OVN flood packet đến mọi thành viên MC_UNKNOWN, bao gồm localnet port, thay vì drop packet.

**Outcomes:**
- Quan sát được outport chuyển từ "none" sang "_MC_unknown" trong ovn-trace output
- Xác minh localnet port là thành viên MC_UNKNOWN và nhận bản copy packet

**Before You Begin:**
Đảm bảo có ít nhất một provider network với localnet port đang active. Xác định tên datapath và tên một VIF port trên logical switch đó.

**Instructions:**

**1.** Xác định datapath name và một VIF port trên provider network:

 [controller]$ ovn-sbctl show | grep -A5 "localnet"

Ghi nhận tên datapath (ví dụ: `provider-ls`) và tên một VIF port (ví dụ: `07188e`).

**2.** Xác minh MC_UNKNOWN tồn tại và localnet port là thành viên:

 [controller]$ ovn-sbctl list multicast_group | grep -A3 "_MC_unknown"

**Predict (Dự đoán bằng giả thuyết ngược):** Nếu OVN **drop** packet khi destination MAC không tìm thấy trong FDB (thay vì flood qua MC_UNKNOWN), thì ovn-trace phải cho thấy packet bị drop tại stage `ls_in_l2_unknown`, output sẽ kết thúc mà không có dòng `egress` nào, đặc biệt không có dòng nào gửi packet đến localnet port. Ngoài ra, `outport` sẽ giữ nguyên giá trị "none" cho đến cuối pipeline.

**3.** Chạy ovn-trace với destination MAC không tồn tại trong bất kỳ LSP addresses nào (dùng MAC giả):

 [controller]$ ovn-trace provider-ls 'inport=="07188e" && eth.src==fa:16:3e:aa:bb:cc && eth.dst==fa:16:3e:99:99:99 && ip4.src==100.64.0.12 && ip4.dst==100.64.0.1 && ip.ttl==64'

**Observe (Quan sát kết quả thực nghiệm):**

```
ingress(dp="provider-ls", inport="07188e")
  ...
  24. ls_in_l2_lkup: 1, priority 0
      outport = get_fdb(eth.dst); next;
  25. ls_in_l2_unknown: outport == "none", priority 50
      outport = "_MC_unknown"; output;

multicast(dp="provider-ls", mcgroup="_MC_unknown")
  → egress: provnet-40617a (type "localnet")
  → egress: other-vif-ports...
```

**Explain (Giải thích mâu thuẫn → kết luận):** Kết quả cho thấy packet KHÔNG bị drop, trái với dự đoán từ giả thuyết ngược. Tại stage `ls_in_l2_lkup` (priority 0), `get_fdb(eth.dst)` không tìm thấy MAC nên set `outport = "none"`. Nhưng tại stage `ls_in_l2_unknown` (priority 50), match `outport == "none"` khớp và action chuyển `outport` sang `_MC_unknown`, rồi thực thi `output`. Packet được flood đến mọi thành viên MC_UNKNOWN, bao gồm localnet port (`provnet-40617a`), nghĩa là packet ra được physical network. Giả thuyết "drop khi không tìm thấy MAC" bị bác bỏ, OVN thiết kế flood-on-unknown thay vì drop-on-unknown cho logical switch có port mang "unknown" addresses.

**Finish:**
Không cần cleanup, ovn-trace là lệnh read-only, không thay đổi trạng thái hệ thống.

MC_UNKNOWN giải quyết được bài toán "đưa packet đến đúng đích khi MAC chưa biết" thông qua flooding. Tuy nhiên, flooding có chi phí bandwidth, mỗi unknown unicast frame bị nhân bản gửi đến mọi thành viên MC_UNKNOWN thay vì chỉ đến đúng localnet port. Khi provider network có hàng trăm VM với port_security disabled (tất cả đều mang "unknown" addresses), mỗi frame đến thiết bị vật lý bị nhân bản hàng trăm lần. Đây chính là bài toán dẫn đến sự ra đời của FDB learning, cơ chế dynamic MAC caching giúp OVN "nhớ" MAC đã thấy và forward trực tiếp mà không cần flood lại.

## 17.4 - FDB table và dynamic MAC learning: từ bài toán flooding đến caching

### Trước FDB: flooding là giải pháp duy nhất

Trước OVN 21.06, OVN chỉ biết MAC từ hai nguồn: thông tin static từ cột `addresses` trên Logical_Switch_Port (Neutron set khi tạo port), và keyword `"unknown"` (nghĩa là "nhận mọi thứ không biết"). Không có cơ chế dynamic learning nào.

Hệ quả trực tiếp trên provider network có nhiều thiết bị vật lý: mỗi khi VM gửi frame đến MAC của gateway vật lý, MAC mà OVN không biết trước, giải pháp duy nhất là flood đến tất cả thành viên MC_UNKNOWN. Nếu 100 VM trên cùng provider network đều có port_security disabled (tất cả đều mang "unknown" addresses và là thành viên MC_UNKNOWN), mỗi frame đến thiết bị vật lý bị nhân bản gửi đến 100 port, dù chỉ cần đến localnet port (ra physical network). Bandwidth lãng phí tăng tuyến tính theo số VM.

FDB learning ra đời để giải quyết chính xác vấn đề này: sau lần flood đầu tiên, khi gateway vật lý gửi frame ngược lại (ví dụ ARP reply), ovn-controller ghi nhận "MAC của gateway đi vào qua localnet port" vào FDB table. Từ lần sau, `get_fdb` tìm thấy entry và forward trực tiếp đến localnet port, không cần flood. Nếu không có FDB, hệ thống vẫn hoạt động đúng (packet đến đích thông qua flood) nhưng hiệu năng kém trên hạ tầng lớn.

### FDB table schema

```json
"FDB": {
    "columns": {
        "mac":      {"type": "string"},
        "dp_key":   {"type": {"key": {"type": "integer", "minInteger": 1, "maxInteger": 16777215}}},
        "port_key": {"type": {"key": {"type": "integer", "minInteger": 1, "maxInteger": 16777215}}},
        "timestamp": {"type": "integer"}
    },
    "indexes": [["mac", "dp_key"]],
    "isRoot": true
}
```

| Column | Type | Mô tả |
|-----------|---------|------------------------------------------------|
| mac | string | MAC address đã học (vd: "fa:16:3e:35:a7:73") |
| dp_key | int | Datapath tunnel key (logical switch ID) |
| port_key | int | Port tunnel key (logical port ID) |
| timestamp | int | Thời điểm cập nhật (thêm sau cho aging) |

Index `[mac, dp_key]` đảm bảo primary key là cặp (MAC, datapath): một MAC trên cùng logical switch chỉ có đúng một FDB entry.

Thiết kế khác biệt quan trọng so với MAC_Binding (mục 17.5): FDB lưu tunnel key (dp_key, port_key) trực tiếp thay vì logical port name. Như đã phân tích ở mục 17.1, dp_key map sang metadata register và port_key map sang reg14/reg15 trong OpenFlow, lưu dp_key và port_key trực tiếp giúp ovn-controller bỏ qua bước chuyển đổi tên-sang-số khi program flow, giảm overhead khi có hàng nghìn entry. Commit message của Numan Siddique giải thích rõ: *"Unlike MAC_Binding Southbound table, FDB table stores the datapath tunnel key and port key. This makes it easier for ovn-controller to program the OF rules."*

Giới hạn: `MAX_FDB_ENTRIES = 1000` trong `controller/mac-learn.c`.

### Ba action cốt lõi và cách chúng hoạt động

**`put_fdb(inport, eth.src)`** ghi entry mới hoặc cập nhật entry hiện có. Action này translate thành `controller` action, gửi PACKET_IN đến ovn-controller. Tại đó, pinctrl thread extract dp_key từ metadata, port_key từ reg14 (MFF_LOG_INPORT), và eth.src từ packet header, rồi gọi `ovn_fdb_add()` để lưu vào in-memory hash map và commit lên SB DB qua OVSDB transaction.

**`get_fdb(eth.dst)`** tìm port_key cho MAC đích. Action thực hiện resubmit đến OpenFlow table 71 (OFTABLE_GET_FDB). ovn-controller cài sẵn flow trong table này: match `metadata=dp_key, dl_dst=mac` thì load port_key vào MFF_LOG_OUTPORT (reg15). Nếu không match (MAC chưa có trong FDB), default flow load 0, tương đương outport="none".

**`lookup_fdb(inport, eth.src)`** kiểm tra entry có tồn tại không. Action resubmit đến OpenFlow table 72 (OFTABLE_LOOKUP_FDB). Match `metadata=dp_key, dl_src=mac, reg14=port_key` thì load 1 vào reg0[11] (nghĩa là "đã có entry, không cần learning"). Hành vi đặc biệt: khi `flags.localnet=1` (packet đến từ localnet port), lookup_fdb trả về 1 nếu entry tồn tại VÀ port type hiện tại là VIF. Kết quả: put_fdb bị skip, bảo vệ VIF entry khỏi bị localnet overwrite, cơ chế protection sẽ phân tích chi tiết ở phần sau của mục này.

### Điều kiện để FDB learning kích hoạt, ba điều kiện AND

> **Key Topic:** Một port đủ điều kiện FDB learning khi thỏa **CẢ BA** điều kiện AND dưới đây. Thiếu bất kỳ điều kiện nào, FDB learning không kích hoạt trên port đó.

Điều kiện đầu tiên: **port_security phải rỗng/disabled** (`!op->n_ps_addrs`). Điều kiện thứ hai: **addresses phải chứa "unknown"** (`op->has_unknown`). Điều kiện thứ ba: **type phải là "" (VIF thường)** (`!strcmp(op->nbsp->type, "")`) HOẶC là localnet port với `localnet_learn_fdb=true`.

Source code (`northd/northd.c`, hàm `build_lswitch_learn_fdb_op()`):
```c
if (!op->n_ps_addrs && op->has_unknown &&
    (!strcmp(op->nbsp->type, "") ||
     (lsp_is_localnet(op->nbsp) && localnet_can_learn_mac(op->nbsp)))) {
```

Bugzilla gốc: Red Hat #1672625, *"[RFE] Support MAC learning in OVN when port security is disabled for a logical port"*.

### ▶ Guided Exercise 2: Chứng minh FDB learning chỉ kích hoạt khi cả ba điều kiện AND thỏa mãn

Trong bài thực hành này, bạn chứng minh rằng thiếu bất kỳ một trong ba điều kiện (port_security disabled, addresses chứa "unknown", type=""), ovn-northd KHÔNG tạo logical flow cho FDB learning trên port đó.

**Outcomes:**
- Quan sát được logical flow `ls_in_put_fdb` chỉ xuất hiện cho port thỏa cả ba điều kiện
- Xác minh port có port_security enabled KHÔNG có flow `put_fdb`

**Before You Begin:**
Xác định một VIF port có port_security enabled (port bình thường của VM) và một VIF port có port_security disabled (VNF port) trên cùng logical switch.

**Instructions:**

**1.** Xác định hai port trên cùng logical switch, một có port_security, một không:

 [controller]$ ovn-nbctl list logical_switch_port | grep -E "name|port_security|addresses|type" | head -40

Ghi nhận: port A có `port_security: ["fa:16:3e:..."]` (enabled) và port B có `port_security: []` (disabled) kèm addresses chứa "unknown".

**Predict (Dự đoán bằng giả thuyết ngược):** Nếu FDB learning kích hoạt trên **mọi port** bất kể port_security (nghĩa là port_security không phải điều kiện), thì khi liệt kê logical flow cho stage `ls_in_put_fdb`, ta phải thấy flow `put_fdb` cho CẢ HAI port, port A (có port_security) lẫn port B (không có port_security). Cụ thể, phải tồn tại flow match `inport == "<port-A-name>"` với action chứa `put_fdb`.

**2.** Liệt kê logical flow cho stage `ls_in_put_fdb` trên datapath tương ứng:

 [controller]$ ovn-sbctl lflow-list <datapath> | grep "ls_in_put_fdb"

**Observe:** Output chỉ chứa flow cho port B (port_security disabled, addresses có "unknown"):

```
  table=4 (ls_in_put_fdb), priority=100,
    match=(inport == "<port-B>" && reg0[11] == 0),
    action=(put_fdb(inport, eth.src); next;)
```

Không có flow nào match `inport == "<port-A>"`, port A hoàn toàn vắng mặt trong stage `ls_in_put_fdb`.

**Explain (Giải thích mâu thuẫn → kết luận):** Kết quả bác bỏ giả thuyết ngược, FDB learning KHÔNG kích hoạt trên mọi port. Port A có `port_security: ["fa:16:3e:..."]` nên điều kiện đầu tiên (`!op->n_ps_addrs`) fail, ovn-northd bỏ qua port này khi sinh flow cho `ls_in_put_fdb`. Chỉ port B, thỏa đồng thời cả ba: `port_security: []`, addresses chứa "unknown", `type: ""`, mới được ovn-northd tạo flow learning. Đây là thiết kế có chủ đích: OVN chỉ cần FDB learning cho port mà nó không thể biết trước MAC sẽ xuất hiện (port_security disabled + "unknown" addresses), còn port bình thường đã có MAC tĩnh trong LSP addresses, không cần dynamic learning.

**Finish:**
Không cần cleanup, `ovn-sbctl lflow-list` là lệnh read-only.

### Đánh giá triển khai: khi nào FDB learning có lợi, khi nào gây hại

Ba điều kiện AND không phải rào cản tùy ý, mỗi điều kiện là một cổng bảo vệ (gate) được thiết kế có chủ đích, phản ánh bài toán mà FDB learning giải quyết và rủi ro mà nó tạo ra.

Khi port_security enabled, OVN đã biết chính xác MAC nào hợp lệ trên port đó, danh sách MAC nằm trong cột `port_security` của LSP, ovn-northd tạo flow tĩnh match `eth.src == <allowed_mac>` tại stage `ls_in_port_sec_l2`. Mọi frame có source MAC không nằm trong danh sách bị drop ngay tại stage đầu tiên, trước khi đến được FDB pipeline. FDB learning lúc này hoàn toàn vô nghĩa: MAC source đã biết trước, không cần "học" thêm gì. Khi port_security disabled, OVN mất khả năng validate source MAC, port chấp nhận frame từ bất kỳ MAC nào. FDB learning lúc này có ý nghĩa: nó "học" MAC đã thấy để tránh flood lần sau. Đổi lại, cánh cửa mở cho FDB poisoning, bất kỳ MAC nào xuất hiện qua port đều được ghi nhận mà không kiểm chứng.

Keyword `"unknown"` trong addresses đưa port vào MC_UNKNOWN, nhóm nhận unknown unicast flood. Mối quan hệ giữa "unknown" và FDB learning là nhân-quả: chỉ port nào nhận được unknown unicast traffic (thông qua MC_UNKNOWN) mới có cơ hội "thấy" frame từ source MAC chưa biết. Port không có "unknown" không bao giờ nhận frame ngoài danh sách MAC tĩnh, không có gì để "học". Neutron ML2/OVN driver tự động thêm `"unknown"` vào addresses khi port_security disabled (Launchpad #1815270), nên trên OpenStack hai điều kiện đầu luôn đi cùng nhau. Trên OVN standalone, admin có thể set port_security=[] mà không set "unknown", lúc đó FDB learning không kích hoạt dù port_security disabled.

VIF port (type="") đại diện cho VM/container interface, nơi dynamic MAC learning có ý nghĩa trực tiếp. Localnet port ban đầu bị loại khỏi FDB learning vì lý do scale: localnet đại diện toàn bộ physical network segment, có thể có hàng trăm MAC đi qua, tiêu tốn `MAX_FDB_ENTRIES = 1000` nhanh chóng. Tùy chọn `localnet_learn_fdb=true` (v22.09.0, mô tả tại mục 17.2) được thêm dưới dạng opt-in cho hai use case: hardware offload (NVIDIA/Mellanox SmartNIC cần MAC-to-port mapping để offload forwarding xuống hardware) và provider network có ít thiết bị vật lý (5-10 MAC từ localnet không gây áp lực scale nhưng giảm đáng kể flooding).

> **Key Topic:** FDB learning là con dao hai lưỡi: **có lợi** khi giảm unknown unicast flooding trên provider network (tiết kiệm bandwidth, cần thiết cho hardware offload), nhưng **gây hại** khi VNF emit frame với source MAC không thuộc về nó (FDB poisoning → traffic hijack). Ba điều kiện AND là thiết kế phòng thủ chiều sâu, chỉ mở FDB learning cho port thực sự cần, và chỉ khi port đó ở trạng thái "không kiểm soát được MAC".

Khuyến nghị vận hành: luôn set `fdb_age_threshold` > 0 trên mọi logical switch có FDB learning active, giá trị 300 giây (5 phút) là điểm khởi đầu hợp lý, cân bằng giữa cache hit rate và thời gian entry stale tồn tại. Giám sát FDB table size bằng `ovn-sbctl list fdb | wc -l`, nếu tiệm cận 1000, cân nhắc giảm `fdb_age_threshold` hoặc disable `localnet_learn_fdb`. Từ OVN v26.03.0, sử dụng `lsp_learn_mac` per-port để disable learning trên từng VNF port cụ thể mà không ảnh hưởng port khác, giải pháp granular nhất hiện có.

### fdb_age_threshold, aging mechanism cho FDB entries

Trước v23.09.0, FDB entry tồn tại vĩnh viễn sau khi được tạo, không có aging mechanism nào chạy. Ales Musil (amusil@redhat.com) cùng Numan Siddique đã bổ sung `fdb_age_threshold` trong OVN v23.09.0 (commit `bfbf32f3`. "northd: Synchronize the FDB age threshold", mailing list: https://mail.openvswitch.org/pipermail/ovs-dev/2023-May/404718.html). Cấu hình per logical switch: `Logical_Switch.other_config:fdb_age_threshold` (đơn vị: giây). Default: 0, disabled, FDB entry tồn tại vĩnh viễn. Khi set > 0, ovn-northd tự động xóa entry quá tuổi.

NEWS: *"Added FDB aging mechanism, that is disabled by default. It can be enabled per logical switch with other_config 'fdb_age_threshold'."*

> **Key Topic:** Với `fdb_age_threshold=0` (default), FDB entry tồn tại vĩnh viễn, ovn-northd không chạy aging. Trong case study VLAN 3808 (mục 17.6), đây là yếu tố khiến FDB entry sai không bao giờ tự expire, duy trì tình trạng traffic bị hijack cho đến khi phát hiện và xóa thủ công.

> **Hiểu sai:** "`fdb_age_threshold=0` nghĩa là FDB entry hết hạn ngay lập tức sau khi được tạo."
>
> **Thực tế:** `fdb_age_threshold=0` là giá trị **tắt aging**, không phải TTL bằng 0 giây. Khi bằng 0, ovn-northd không chạy aging job cho FDB table, và entry tồn tại vĩnh viễn cho đến khi bị xóa thủ công hoặc overwrite. Chỉ khi set giá trị dương (ví dụ `fdb_age_threshold=300`) thì entry mới có TTL 300 giây. Nhầm lẫn này đặc biệt nguy hiểm vì operator có thể nghĩ "để mặc định là an toàn nhất", thực tế ngược lại: default-off aging có nghĩa là FDB entry sai tồn tại mãi mãi.

### flags.localnet protection, ngăn localnet overwrite VIF entry

Khi `localnet_learn_fdb=true`, một race condition nguy hiểm xuất hiện. Hình dung: vif1 trên chassis hv1 gửi frame, ovn-controller trên hv1 ghi FDB entry mac1 trỏ về vif1 (đúng). Cùng frame đi qua localnet đến chassis hv2, ovn-controller trên hv2 thấy frame đi vào từ localnet port và ghi FDB entry mac1 trỏ về localnet, overwrite entry đúng thành entry sai. Kết quả: traffic đến mac1 trên hv2 bị chuyển ra localnet thay vì qua Geneve tunnel đến hv1.

Xavier Simonart (xsimonart@redhat.com) giải quyết bằng bit `MLF_LOCALNET` (bit 15 trong logical flags register) trong commit `2acf91e9628e`. "controller: FDB entries for localnet should not overwrite entries for vifs". Backport: `33b01175` (branch-23.09.1), `8befadaf` (branch-23.06.3). Feature merge vào OVN v24.03.0.

Cơ chế hoạt động: trong `ls_in_lookup_fdb`, khi packet đến từ localnet port, OVN set `flags.localnet = 1` trước khi gọi `lookup_fdb`. Khi `flags.localnet=1`, hàm `lookup_fdb` trả về true nếu entry đã tồn tại VÀ port type của entry đó là VIF. Kết quả: `reg0[11] == 1` khiến `ls_in_put_fdb` skip việc ghi entry mới, bảo vệ VIF entry khỏi bị localnet overwrite. Protection này tốt cho trường hợp bình thường (ngăn localnet overwrite VIF đúng), nhưng trong case study FDB poisoning (mục 17.6), khi VIF entry là SAI, protection trở thành rào cản ngăn self-heal.

## 17.5 - MAC_Binding table: IP-to-MAC resolution trên logical router

### Mục đích và thiết kế

Các phần trước tập trung vào Layer 2, MAC forwarding trên logical switch. Khi packet cần đi qua logical router (Layer 3), một bài toán hoàn toàn khác xuất hiện: router cần biết MAC của next-hop IP để đóng Ethernet header khi forward packet. **MAC_Binding** là bảng trong SB DB lưu mapping IP address sang MAC address trên logical router port, tương đương OVN của ARP cache trên router vật lý.

Điểm phân biệt cơ bản: ARP cache tồn tại trên router (Layer 3), không phải switch (Layer 2), vì router cần resolve next-hop IP sang MAC để forward packet qua link-layer. Switch vật lý thuần túy không chạy ARP và không có ARP cache, switch chỉ có MAC address table (tương đương FDB trong OVN). Sự phân biệt này quan trọng khi debug: nếu packet đi sai port trên cùng L2 segment, kiểm tra FDB; nếu router không resolve được MAC của next-hop IP, kiểm tra MAC_Binding.

Schema (ovn-sb(5)):

| Column | Type | Mô tả |
|-------------|-------------------|--------------------------------------|
| logical_port | string | Router port nơi binding được phát hiện |
| ip | string | IPv4 address |
| mac | string | MAC address tương ứng |
| datapath | Datapath_Binding | Logical router datapath |
| timestamp | integer | Thời điểm cập nhật (cho aging) |

Index: `[logical_port, ip]`

ovn-sb(5): *"Usually, IP-to-MAC bindings for virtual machines are statically populated into the Port_Binding table, so MAC_Binding is primarily used to discover bindings on physical networks."*

### Cách hoạt động trong router pipeline

```
  Logical Router Ingress Pipeline
  ════════════════════════════════
  lr_in_admission → lr_in_lookup_neighbor → lr_in_learn_neighbor
       ↓
  lr_in_ip_input → ... → lr_in_ip_routing → lr_in_ip_routing_ecmp
       ↓
  lr_in_arp_resolve:
    Priority 100: IP đã biết (VM, DHCP) → set eth.dst static
      match=(outport == "lrp-xxx" && reg0 == 192.168.1.11)
      action=(eth.dst = fa:16:3e:b6:91:70; next;)
    Priority 0: IP chưa biết → consult MAC_Binding
      match=(ip4)
      action=(get_arp(outport, reg0); next;)
       ↓
  lr_in_arp_request:
    Nếu eth.dst == 00:00:00:00:00:00 → generate ARP request
      action=(arp { eth.dst = ff:ff:ff:ff:ff:ff; arp.spa = reg1;
                    arp.tpa = reg0; arp.op = 1; output; };)
```

Action `get_arp(outport, reg0)` thực hiện resubmit đến OpenFlow table 66, nơi chứa các flow được populate từ MAC_Binding table. Khi ovn-controller nhận thấy ARP reply hoặc ND advertisement trên router port, pinctrl module ghi binding mới thông qua `put_arp(port, ip, mac)`, tạo ra một row trong MAC_Binding trên SB DB. Để tránh PACKET_IN flooding khi ARP reply đến liên tục, OVN dùng `lookup_arp`/`lookup_nd` kiểm tra trước xem entry đã tồn tại chưa.

Aging cho MAC_Binding được thêm trong OVN v22.09.0 (commit `1a947dd3`, Ales Musil, Bugzilla #2084668): `ovn-nbctl set logical_router <lr> options:mac_binding_age_threshold=<seconds>`. Default: 0 (disabled).

### So sánh FDB vs MAC_Binding, hai bài toán hoàn toàn khác nhau

| Khía cạnh | FDB | MAC_Binding |
|-------------------|-------------------------------|--------------------------------|
| **Layer** | L2 (switching) | L3 (routing) |
| **Maps** | MAC → port_key | IP → MAC |
| **Datapath** | Logical Switch | Logical Router |
| **Pipeline** | `ls_in_lookup_fdb`, `ls_in_l2_lkup` | `lr_in_arp_resolve` |
| **Populated by** | MAC learning từ VIF port | ARP reply/ND trên router port |
| **Tương đương vật lý** | MAC address table trên L2 switch | ARP cache trên router |
| **Ra đời** | Feb 2021 (schema 20.16.0) | Original OVN (~2015) |
| **Aging option** | `fdb_age_threshold` (per LS) | `mac_binding_age_threshold` (per LR) |

> **Key Topic:** FDB (L2: mapping MAC address sang port_key, hoạt động trên logical switch) và MAC_Binding (L3: mapping IP address sang MAC address, hoạt động trên logical router) giải quyết hai bài toán hoàn toàn khác nhau. Khi debug, cần phân biệt rõ: nếu packet đi sai port trên cùng L2 segment thì kiểm tra FDB; nếu router không resolve được MAC của next-hop IP thì kiểm tra MAC_Binding.

> **Hiểu sai:** "Khi traffic đến gateway vật lý bị mất, tôi cần kiểm tra MAC_Binding vì có thể router không resolve được MAC của gateway."
>
> **Thực tế:** Nếu VM vẫn resolve được ARP của gateway (ping vẫn lấy được MAC) nhưng ICMP echo bị timeout, vấn đề không nằm ở MAC_Binding mà ở FDB. MAC_Binding chỉ ảnh hưởng đến IP-to-MAC resolution trên logical router, nếu ARP hoạt động, MAC_Binding không phải nguyên nhân. Vấn đề L2 forwarding (packet đi sai port trên cùng logical switch) luôn cần kiểm tra FDB trước.

Trong case study VLAN 3808 ở mục tiếp theo, ICMP echo request từ VM đến gateway 100.64.0.1 bị chuyển đến sai port (VyOS tap port thay vì localnet port ra physical network). Đây là vấn đề L2 forwarding, packet đi đúng logical switch nhưng bị đưa sai port đích. Bảng cần kiểm tra đầu tiên là FDB (nơi lưu mapping MAC-to-port sai), không phải MAC_Binding (vì vấn đề không liên quan đến IP-to-MAC resolution trên router).

## 17.6 - Case study: FDB poisoning trên provider network VLAN 3808

Các phần trước đã xây dựng từng mảnh ghép riêng lẻ: localnet port (mục 17.2) kết nối thế giới ảo với VLAN vật lý, MC_UNKNOWN (mục 17.3) flood unknown unicast đến localnet, FDB learning (mục 17.4) cache MAC để giảm flood, và MAC_Binding (mục 17.5) giúp router resolve next-hop. Phần này ghép tất cả lại thành bức tranh hoàn chỉnh thông qua một case study thực tế, nơi FDB learning kết hợp với port_security disabled tạo ra lỗ hổng khiến traffic production bị hijack.

### Topology và cấu hình

```
Provider Network VLAN 3808 (100.64.0.0/24)
══════════════════════════════════════════
  100.64.0.1  = Fornix gateway (physical, trên VLAN 3808)
  100.64.0.12 = VyOS VM (VRRP router, kết nối Fornix)
  100.64.0.xx = Các VM khác trên cùng logical switch

  VyOS VM (.12):
    - port_security_enabled = False  (cần thiết cho VNF/router)
    - NB addresses = ["fa:16:3e:04:74:74 100.64.0.12", "unknown"]
    - LSP type = ""  (VIF thường)
```

### Tại sao Neutron set addresses chứa "unknown"

Launchpad bug #1815270 ghi nhận vấn đề: khi `port_security_enabled=False`, OVN cần `"unknown"` trong addresses để port nhận traffic với destination MAC không biết trước. Neutron ML2/OVN driver tự động set: `addresses = ["fa:16:3e:04:74:74 100.64.0.12", "unknown"]`. Bug #1838535 refine thêm: chỉ set "unknown" cho VIF port (type=""), không cho router/localnet/localport type vì những type đặc biệt có cơ chế forwarding riêng, không cần tham gia MC_UNKNOWN.

### Tại sao FDB learning kích hoạt trên VyOS LSP

Ba điều kiện AND (mục 17.4) đều thỏa trên VyOS port: `port_security = []` (disabled), addresses chứa `"unknown"`, và `type = ""` (VIF thường). Do cả ba điều kiện thỏa, ovn-northd tạo logical flow cho FDB learning trên port này:

```
ls_in_lookup_fdb: inport == "vyos-port" →
    reg0[11] = lookup_fdb(inport, eth.src); next;
ls_in_put_fdb: inport == "vyos-port" && reg0[11] == 0 →
    put_fdb(inport, eth.src); next;
```

### Cơ chế poisoning, từng bước chi tiết

**Bước 1: VyOS gửi packet hợp lệ đến Fornix.** VyOS forward traffic đến gateway Fornix (100.64.0.1) thông qua static route. Mỗi packet rời VyOS mang source MAC = fa:16:3e:04:74:74 (MAC thật của eth1) và destination MAC = fa:16:3e:35:a7:73 (MAC của Fornix, lấy từ neighbor cache). Đây là hành vi hoàn toàn bình thường, kernel Linux luôn dùng `dev->dev_addr` của output interface làm source MAC thông qua hàm `eth_header()` trong `net/ethernet/eth.c` ([kernel source](https://github.com/torvalds/linux/blob/master/net/ethernet/eth.c)). Keepalived GARP cũng tương tự: hàm `send_gratuitous_arp_immediate()` luôn dùng `IF_HWADDR(ipaddress->ifp)`, MAC vật lý của interface, không bao giờ dùng MAC của thiết bị bên ngoài ([keepalived source](https://github.com/acassen/keepalived)). VM VyOS **không có khả năng** emit frame với source MAC = fa:16:3e:35:a7:73.

**Bước 1b: OVN pipeline swap eth.src ↔ eth.dst mà không sync inport ↔ outport (BUG).** Packet hợp lệ từ VyOS đi vào OVN pipeline với `inport = VyOS_LSP`, `eth.src = fa:16:3e:04:74:74`, `eth.dst = fa:16:3e:35:a7:73`. Tại một code path, ví dụ khi tạo ICMP "Fragmentation Needed" cho multichassis port, OVN swap `eth.src ↔ eth.dst` để xây dựng reply packet, nhưng **quên swap `inport ↔ outport`**. Sau swap: `inport` vẫn = VyOS_LSP, nhưng `eth.src` = fa:16:3e:35:a7:73 (MAC của Fornix, vốn là destination MAC ban đầu). Đây chính xác là bug class đã được Ales Musil xác nhận và fix trong patch *"physical: Prevent wrong FDB to be learned with multichassis port"* ([ovs-dev mailing list, 20 Aug 2024](https://www.mail-archive.com/ovs-dev@openvswitch.org/msg88284.html), `Reported-at: https://issues.redhat.com/browse/FDP-620`).

> **Key Topic:** Root cause của FDB poisoning không nằm trong VM. VyOS gửi packet bình thường, OVN pipeline tạo ra FDB entry sai từ destination MAC trong packet hợp lệ đó. Điều kiện cần: (1) OVN version chưa fix bug swap inport/outport, (2) port có `addresses="unknown"` (FDB learning kích hoạt), (3) `fdb_age_threshold=0` (entry sai tồn tại vĩnh viễn).

**Bước 2: put_fdb ghi entry SAI.** Frame đi vào br-int, khớp flow trong `ls_in_lookup_fdb`:

```
lookup_fdb(inport="vyos-port", eth.src="fa:16:3e:35:a7:73")
→ reg0[11] = 0  (chưa có entry cho MAC này)
```

Chuyển sang `ls_in_put_fdb`:

```
reg0[11] == 0 → put_fdb(inport="vyos-port", eth.src="fa:16:3e:35:a7:73")
```

FDB entry được tạo trong SB DB:

```
ovn-sbctl list fdb
mac           : "fa:16:3e:35:a7:73"
dp_key        : <VLAN3808_dp_key>
port_key      : 3                      ← VyOS LSP tunnel key!
```

**Bước 3: Traffic bị hijack.** Khi VM khác gửi ICMP đến gateway 100.64.0.1 (MAC = fa:16:3e:35:a7:73):

```
ls_in_l2_lkup, priority 50: eth.dst == fa:16:3e:35:a7:73?
  → KHÔNG có static entry (MAC này không trong LSP addresses nào)

ls_in_l2_lkup, priority 0: outport = get_fdb(eth.dst="fa:16:3e:35:a7:73")
  → FDB tìm thấy! outport = port_key 3 (VyOS LSP)

ls_in_l2_unknown, priority 0: output;
  → Forward đến VyOS tap port thay vì localnet (bond3)!
```

ICMP đi vào VyOS VM thay vì ra physical network → gateway không bao giờ nhận được → timeout.

### Tại sao ARP vẫn hoạt động bình thường

ARP request là broadcast (eth.dst = ff:ff:ff:ff:ff:ff):

```
ls_in_l2_lkup, priority 72: eth.mcast && arp.op == 1
  → outport = "_MC_flood"; output;
```

ARP bypass hoàn toàn FDB lookup, đi thẳng vào `_MC_flood` ở priority 72, cao hơn đáng kể so với priority 0 nơi `get_fdb` thực thi. ARP reply từ gateway đi qua localnet → flood → đến đúng VM. Đây là lý do ARP resolution vẫn hoạt động bình thường trong khi unicast traffic bị hijack, hai loại traffic đi qua hai nhánh khác nhau trong pipeline.

### Tại sao bond3 không thấy ICMP

FDB entry `fa:16:3e:35:a7:73 → port_key 3` short-circuit trước khi packet có cơ hội đến localnet port. `get_fdb` trả về VyOS port_key thay vì "none", nên packet KHÔNG đi vào `_MC_unknown` → KHÔNG được flood đến localnet → KHÔNG ra bond3. Frame bị "hút" vào VyOS tap port, gateway vật lý hoàn toàn không thấy traffic.

### Tại sao FDB entry không tự hết hạn

Với `fdb_age_threshold=0` (default), ovn-northd không chạy aging, FDB entry tồn tại cho đến khi bị xóa thủ công (`ovn-sbctl destroy fdb <uuid>`), bị overwrite bởi entry khác (cùng MAC, cùng datapath), hoặc ovn-controller restart. Trong case study này, entry sai tồn tại vĩnh viễn.

### Tại sao gateway không tự sửa FDB

Giả sử gateway Fornix gửi frame ra với source MAC fa:16:3e:35:a7:73. Frame này đến qua localnet port. Nhưng `flags.localnet` protection (nếu OVN version đủ mới) ngăn localnet overwrite VIF entry:

```
ls_in_lookup_fdb cho localnet port:
  flags.localnet = 1; reg0[11] = lookup_fdb(inport, eth.src);

lookup_fdb với flags.localnet=1:
  Entry tồn tại (mac=fa:16:3e:35:a7:73, port=VyOS)
  Port type hiện tại = VIF → return 1 (đã có, không cần update)

ls_in_put_fdb: reg0[11] == 1 → SKIP put_fdb
```

Entry VIF được bảo vệ → gateway không thể sửa FDB từ localnet side. Đây là protection tốt cho trường hợp bình thường (ngăn localnet overwrite VIF đúng), nhưng trong trường hợp này, VIF entry là SAI → protection trở thành rào cản ngăn tự khắc phục.

### Fix ngắn hạn và dài hạn

Fix ngắn hạn, xóa FDB entry sai:

```bash
# Tìm FDB entry
[controller]$ ovn-sbctl find fdb mac="fa:16:3e:35:a7:73"

# Xóa
[controller]$ ovn-sbctl destroy fdb <uuid>
```

Ngay sau khi xóa, `get_fdb` không tìm thấy MAC nên set outport="none". Pipeline chuyển sang MC_UNKNOWN, flood packet đến localnet port, ra bond3, đến gateway vật lý. Traffic phục hồi ngay lập tức.

Fix dài hạn cần kết hợp nhiều biện pháp: enable port_security trên VyOS port nếu có thể, ngăn FDB learning hoàn toàn; set `fdb_age_threshold > 0` trên logical switch, entry sai sẽ tự expire; hoặc dùng allowed-address-pairs thay vì disable port_security hoàn toàn, cho phép MAC cụ thể mà vẫn giữ port_security.

```bash
# Set FDB aging 300 giây (5 phút) trên logical switch
[controller]$ ovn-nbctl set logical_switch <ls> other_config:fdb_age_threshold=300
```

### OVN FDB bug đã biết bằng chứng root cause nằm ở pipeline

Phân tích pháp y (forensic analysis) trên case study này xác định root cause không nằm trong VM VyOS mà ở tầng OVN pipeline. Kết luận này dựa trên ba trụ bằng chứng: loại trừ triệt để mọi cơ chế VM-side, xác nhận bug class trong OVN source code, và phân tích cấu hình VyOS chỉ đóng vai trò amplifier.

Trụ thứ nhất, loại trừ VM-side. Hàm `eth_header()` trong kernel `net/ethernet/eth.c` luôn gán `dev->dev_addr` (MAC của output interface) vào `eth->h_source` khi `saddr=NULL`, đây là code path duy nhất cho mọi packet forward qua `ip_forward()`. Keepalived `send_gratuitous_arp_immediate()` luôn dùng `IF_HWADDR(ipaddress->ifp)` qua `PF_PACKET + SOCK_RAW` socket, kernel không can thiệp source MAC. Config VyOS không có bridge, macvlan, hay proxy_arp. Kết luận: VM không có cơ chế nào tạo frame với source MAC khác interface MAC.

Trụ thứ hai, bug class đã xác nhận trong OVN. Bốn bug report liên quan trực tiếp đến FDB:

Thứ nhất, FDP-620 (fix OVN v24.09.1, patch bởi Ales Musil). Hàm `reply_icmp_error_if_pkt_too_big` trong `controller/physical.c` swap `eth.src ↔ eth.dst` khi tạo ICMP error cho multichassis port nhưng quên swap `inport ↔ outport`. FDB learning sau đó ghi `put_fdb(inport=VIF_port, eth.src=original_dst_MAC)`, chính xác pattern của case study này. Fix thêm 6 dòng code swap `MFF_LOG_INPORT ↔ MFF_LOG_OUTPORT` trước khi swap MAC.

Thứ hai, BZ 1983111 (fix OVN v21.09.1). FDB lookup/put OpenFlow flows không được install khi ovn-controller claim port đầu tiên trên logical switch, trong window này, FDB learning không hoạt động đúng.

Thứ ba, pinctrl map size bug (fix v24.09.2, commit `33a6ae53`). Hàm `pinctrl_handle_put_fdb()` kiểm tra `hmap_count(&put_mac_bindings)` thay vì `hmap_count(&put_fdbs)`, rate limiting FDB entries dựa trên nhầm map.

Thứ tư, FDP-1341 (fix v25.03.2). VIF FDB entries bị mất khi SB transaction fail, cho phép localnet entries overwrite.

Trụ thứ ba, cấu hình VyOS là amplifier, không phải root cause. `garp_master_refresh=5` tạo volume traffic cao (GARP mỗi 5 giây), static route qua Fornix đảm bảo destination MAC fa:16:3e:35:a7:73 luôn hiện diện trong packets, cả hai tăng xác suất trigger bug OVN. Nhưng `port_security_enabled=False` là enabler quyết định (kích hoạt FDB learning) và `fdb_age_threshold=0` là amplifier quyết định (biến sự kiện one-time thành permanent damage).

> **Key Topic:** Decisive test xác nhận VM vô tội: `tcpdump -i tap-VYOS_PORT -e -n 'ether src fa:16:3e:35:a7:73'`. Nếu chạy trên hypervisor host và thấy zero frame, đó là bằng chứng thực nghiệm rằng VyOS không emit source MAC sai, entry FDB sai được tạo bởi OVN pipeline. Kiểm tra version OVN: `ovn-controller --version` và đối chiếu với FDP-620 fix (v24.09.1).

### Cơ chế multichassis binding: dấu phẩy trong `requested-chassis`

Điều kiện trigger đầu tiên của bug nằm ở trạng thái `additional_chassis` của port. Các thao tác Nova còn lại như `nova boot`, `reboot`, `stop`, `resize`, `evacuate`, `rebuild`, `shelve` đều đặt giá trị đơn, dẫn đến binding single-chassis thông thường nên không tạo multichassis state.

### Geneve overhead 58 byte và pipeline PMTUD

Điều kiện trigger thứ hai là packet quá cỡ so với tunnel effective MTU. Tổng overhead Geneve trong OVN là 58 byte, gồm outer Ethernet 14 byte, outer IPv4 20 byte, outer UDP 8 byte, Geneve base header 8 byte, và OVN TLV option Class 0x0102 8 byte. Với physical MTU 1500, effective tunnel MTU chỉ còn 1442 byte. Gói IP 1500 byte từ VM trigger `check_pkt_larger(1442)` tại table 41 (`OFTABLE_OUTPUT_LARGE_PKT_DETECT`), set `reg9[1] = 1`, rồi resubmit sang table 42 (`OFTABLE_OUTPUT_LARGE_PKT_PROCESS`).

Tại table 42, hàm `reply_icmp_error_if_pkt_too_big()` swap `eth.src ↔ eth.dst` nhưng không swap `inport ↔ outport` tương ứng. FDB learning downstream đọc giá trị `inport` cũ (vẫn là VyOS LSP) ghép với `eth.src` mới (đã thành Fornix MAC), kết quả ghi `put_fdb(inport=VyOS_LSP, eth.src=Fornix_MAC)`. Đây chính là FDB entry sai trực tiếp gây hijack traffic của 100.64.0.1 sang VyOS.

Phân rã byte-by-byte overhead, action sequence đầy đủ của `reply_icmp_error_if_pkt_too_big()`, cùng bản vá 6 dòng `MFF_LOG_INPORT/OUTPORT` của Ales Musil được trình bày trong [SDN 3.4 về Geneve overhead, pipeline PMTUD và bug FDP-620](./3.0%20-%20ovn-multichassis-binding-and-pmtud.md).

### Mitigation và fix triệt để

Trong case study VLAN 3808, physical MTU là 1500 nên điều kiện "gói lớn hơn tunnel MTU" được thỏa mãn và bug kích hoạt. Có hai hướng giảm rủi ro trong môi trường kolla-ansible đang chạy.

Hướng thứ nhất là nâng physical MTU tunnel lên 9000 (jumbo frames) để effective tunnel MTU đạt 8942 byte. Gói 1500 byte từ VM không còn vượt ngưỡng nên không trigger `reply_icmp_error_if_pkt_too_big()`. Đây chỉ là mitigation vì bug vẫn tồn tại trong code path, chỉ không được kích hoạt ở MTU hiện tại. Nếu sau này VM MTU tăng hoặc encap overhead tăng (ví dụ double encap), điều kiện trigger có thể tái xuất hiện.

Hướng thứ hai là build container OVN 24.09.1+ qua `kolla-build --template-override` kết hợp Ubuntu Cloud Archive Proposed pocket, sau đó upgrade để áp dụng patch Ales Musil. Đây là fix triệt để cho deployment kolla-ansible vì xóa root cause khỏi code path, không phụ thuộc vào MTU vận hành.

Config `[ovn] live_migration_activation_strategy` trong `neutron.conf` (giá trị `rarp` mặc định, hoặc `chassis`, do Ihar Hrachyshka commit `ee20c48c2f5c` Launchpad #2092250) không trực tiếp ngăn bug này. Cấu hình đó chỉ kiểm soát thời điểm additional chassis location trở nên khả dụng, không sửa logic sai trong ICMP error path. Strategy `rarp` có thể giảm xác suất trigger trên thực tế bằng cách hạn chế traffic đi qua tunnel trong cửa sổ multichassis, nhưng không phải bằng cách vá logic sai.

Chi tiết jumbo frame sự đánh đổi, kernel `mtu_expires` recovery cho cache PMTUD bị poison, cùng phân tích đầy đủ activation-strategy `rarp` vs `chassis` (gồm cả lý do RARP được chọn thay vì GARP) được trình bày trong [SDN 3.5 về activation-strategy và đồng bộ ba tầng](./3.0%20-%20ovn-multichassis-binding-and-pmtud.md) và [SDN 3.6 về Operational tuning Jumbo frame và MTU recovery](./3.0%20-%20ovn-multichassis-binding-and-pmtud.md).

### ▶ Lab 3: Chẩn đoán và khắc phục FDB poisoning trên provider network

Trong bài lab này, bạn chẩn đoán nguyên nhân gốc rễ khi traffic L2 trên provider network bị chuyển sai port, xác minh bằng phản chứng rằng FDB entry sai là nguyên nhân, và khắc phục.

**Outcomes:**
- Xác định được FDB entry sai bằng `ovn-sbctl` và đối chiếu với port_binding
- Chứng minh bằng phương pháp POE rằng xóa FDB entry sai khôi phục traffic
- Thiết lập FDB aging để ngăn tái diễn

**Before You Begin:**
Truy cập được OVN central node (nơi chạy ovn-sbctl). Có ít nhất một provider network với VNF port (port_security disabled).

**Instructions:**

**1.** Xác minh triệu chứng: VM ping đến gateway vật lý bị timeout, nhưng ARP resolution vẫn hoạt động.

**2.** Liệt kê FDB table và xác định entry nghi ngờ, so sánh MAC trong FDB với MAC thực của gateway.

**3.** Đối chiếu port_key trong FDB entry với Port_Binding để xác định packet đang bị chuyển đến port nào.

**Predict (Dự đoán bằng giả thuyết ngược):** Nếu FDB entry sai KHÔNG phải nguyên nhân (nghĩa là vấn đề nằm ở nơi khác, ví dụ physical network, bridge mapping, hoặc ACL), thì xóa FDB entry sai sẽ không thay đổi hành vi, traffic vẫn timeout sau khi xóa.

**4.** Xóa FDB entry nghi ngờ và kiểm tra traffic ngay lập tức.

**5.** Nếu traffic phục hồi: kết luận FDB entry sai là nguyên nhân gốc rễ.

**Explain:** Khi FDB entry sai bị xóa, `get_fdb(eth.dst)` không tìm thấy MAC nên set outport="none". Pipeline chuyển sang MC_UNKNOWN, flood packet đến localnet port, packet ra physical network đến gateway. Traffic phục hồi ngay lập tức chứng minh giả thuyết "vấn đề nằm ở nơi khác" bị bác bỏ, FDB entry sai chính là nguyên nhân duy nhất.

**6.** Thiết lập FDB aging để ngăn tái diễn.

Gợi ý: `ovn-sbctl list fdb`, `ovn-sbctl find port_binding tunnel_key=<port_key>`, `ovn-sbctl destroy fdb <uuid>`, `ovn-nbctl set logical_switch <ls> other_config:fdb_age_threshold=300`

**Evaluation:**
Sau khi hoàn thành, chạy `ovn-trace` với destination MAC của gateway, output phải cho thấy packet đi qua MC_UNKNOWN đến localnet port (không còn bị redirect sang VNF port). Chạy `ovn-nbctl get logical_switch <ls> other_config:fdb_age_threshold`, giá trị phải > 0.

**Finish:**
FDB entry sai đã bị xóa và aging đã được thiết lập, không cần rollback.

## 17.7 - Bài học thiết kế: sự đánh đổi và tiến hóa incremental

### Functionality vs Security: port_security disabled là cần thiết nhưng nguy hiểm

VNF (Virtual Network Function) như router, load balancer, firewall bắt buộc phải disable port_security, chúng cần gửi/nhận frame với MAC không phải của mình. Đây là yêu cầu hợp lệ mà OpenStack hỗ trợ thông qua `port_security_enabled=False`. Nhưng khi disable port_security, OVN mất khả năng validate source MAC → cửa mở cho FDB poisoning. Đây là sự đánh đổi inherent trong thiết kế: OVN không thể vừa cho phép arbitrary MAC forwarding vừa đảm bảo FDB integrity, vì hai mục tiêu này mâu thuẫn nhau ở tầng kiến trúc.

Giải pháp tiến hóa qua thời gian thay vì big-bang redesign: ban đầu không có protection, sau đó thêm `flags.localnet` để ngăn localnet overwrite VIF entry, rồi thêm `fdb_age_threshold` để giới hạn thời gian entry sai tồn tại, và cuối cùng `lsp_learn_mac` để tắt learning trên từng port cụ thể. Mỗi bước phản ứng trước một bug hoặc kịch bản production thực tế mà các bước trước chưa bao phủ.

### Performance vs Correctness: FDB caching giảm flood nhưng tạo stale entry

Trước FDB, mọi unknown unicast đều bị flood: bandwidth cao nhưng luôn đúng vì packet đến mọi nơi kể cả đích đúng. Sau FDB, unknown unicast chỉ flood lần đầu rồi được forward thẳng từ lần sau: bandwidth tối ưu nhưng đổi lại là khả năng entry stale hoặc sai dẫn đến forwarding sai.

Đây là bản chất của **caching problem**, same sự đánh đổi như CPU cache, DNS cache, ARP cache. Mọi cache đều phải đối mặt câu hỏi: stale entry tồn tại bao lâu trước khi bị invalidate? TTL quá ngắn giảm cache hit rate (quay lại flooding), TTL quá dài cho phép entry sai gây damage lâu hơn. OVN chọn cách tiếp cận progressive: FDB default OFF (learning chỉ kích hoạt cho port đặc biệt thỏa ba điều kiện AND), cho phép opt-in mở rộng qua `localnet_learn_fdb`, và cung cấp aging mechanism (`fdb_age_threshold`) khi operator chấp nhận sự đánh đổi.

### OVN tiến hóa qua incremental hardening

Nhìn lại timeline phát triển FDB cho thấy pattern rõ ràng, mỗi version thêm safeguard mới phản ứng trước bug hoặc kịch bản production thực tế:

```
2015: OVN ra đời — chỉ có static MAC + unknown flooding
       → Đơn giản, đúng, nhưng flood nhiều

2021 (v21.06): FDB learning — dynamic MAC caching
       → Giảm flood nhưng mở risk FDB poisoning

2022 (v22.09): localnet_learn_fdb — opt-in learning cho localnet
       → Mở rộng nhưng thêm risk overwrite

2023 (v23.09): fdb_age_threshold — aging mechanism
       → Limit damage từ entry sai

2023 (v24.03): flags.localnet — protect VIF entry từ localnet
       → Sửa specific race condition

2024 (v24.09): force_fdb_lookup — granular control
2025 (v26.03): lsp_learn_mac — per-port learning control
       → Ngày càng fine-grained
```

Mỗi version giữ backward compatibility (default behavior không thay đổi), operator upgrade OVN mà không cần thay đổi cấu hình. Đây là bài học thiết kế quan trọng cho distributed system: tiến hóa incremental cho phép hệ thống production migrate dần mà không phải dừng hoạt động.

### "unknown" addresses, keyword mạnh nhất nhưng cũng nguy hiểm nhất trong OVN

Keyword `"unknown"` kích hoạt đồng thời ba cơ chế: port nhận tất cả unknown unicast traffic (thông qua MC_UNKNOWN membership), port kích hoạt FDB learning (khi kết hợp với port_security disabled), và bất kỳ MAC nào đi vào port đều được ghi nhận vào FDB mà không kiểm chứng. Ba cơ chế kết hợp tạo surface attack lớn, bất kỳ frame nào với source MAC lạ đi qua port đó đều có thể poison FDB table. Trước khi set `addresses = ["...", "unknown"]` trên bất kỳ port nào, operator cần hiểu toàn bộ implications mà tài liệu này đã phân tích.

### Debug checklist khi L2 forwarding bất thường trên OVN

Khi gặp triệu chứng traffic L2 đi sai port hoặc bị mất trên provider network, trình tự debug được đề xuất dựa trên kiến thức từ các mục trước:

| Triệu chứng | Bảng cần kiểm tra | Lệnh |
|---|---|---|
| Packet đi sai port trên cùng L2 | FDB | `ovn-sbctl list fdb` |
| Router không resolve được next-hop MAC | MAC_Binding | `ovn-sbctl list mac_binding` |
| ARP không đến | MC_FLOOD membership | `ovn-sbctl list multicast_group` |
| Unknown unicast không flood | MC_UNKNOWN membership | `ovn-sbctl list multicast_group` |
| Cần trace toàn bộ pipeline | Logical flow + OpenFlow | `ovn-trace` + `ovs-ofctl dump-flows br-int table=71` |

# 0.2. End-to-end packet journey: from Pod A to Pod B through OVS, Geneve, and OVN

> **Lab environment:** Ubuntu 22.04 LTS, OVS 2.17.9, OVN 22.03.8
> **Block:** 0, Overview and reading guide
> **Part:** Part 0 (introductory guide)
> **Plan:** `plans/sdn-foundation-architecture.md` §E.11, C10 cross-cutting view
> **Prerequisites:** None. This is the big picture, so read it BEFORE going deep into each Block, or read it again AFTER finishing to grasp the whole.
> **Goal:** Follow the journey of a packet from Pod A on Chassis X through the OVN overlay to Pod B on Chassis Y. Each leg of the journey is cross-referenced to the specific Blocks in the curriculum.
> **References:** OVN Architecture Guide (`ovn-architecture(7)`), Open vSwitch Design Documentation, RFC 8926 Geneve, NSDI 2015 "The Design and Implementation of Open vSwitch" (Pfaff et al.).

## Learning objectives

After finishing this part, you will master:

1. **Describe (Understand)** the chain of 12 stages a packet goes through when Pod A sends data to Pod B on a different Chassis: from VIF (veth) to ingress OVS to classifier to OpenFlow actions to Geneve encapsulation to physical NIC to transmission through the tunnel to the destination Chassis to decapsulation to the destination VIF.
2. **Analyze (Analyze)** the role of each component (kernel datapath, megaflow cache, classifier, OVN logical flow, `ovn-controller`) at each stage.
3. **Apply (Practice)** the diagnostic tools (`ovs-appctl ofproto/trace`, `ovn-trace`, `tcpdump` on veth) to reproduce the packet journey in a real environment.
4. **Evaluate (Evaluate)** the right level of abstraction for each diagnostic problem (when to use `tcpdump`, when to use `ovn-trace`, when to use `ofproto/trace`).

Prerequisites: none. This part acts as a "table of contents" of the curriculum organized by data flow.

## 0.2.1 Topology reference

Every step in this document assumes the standard production-environment topology described in **`ovn-architecture(7)`**:

```
+----------------------------------------------------------------------+
|                      OVN system                                      |
|                                                                      |
|  +-------------------------+        +-------------------------+      |
|  |   Chassis X (Hypervisor)|        |   Chassis Y (Hypervisor)|      |
|  |                         |        |                         |      |
|  |  Pod A (10.0.0.5)       |        |       Pod B (10.0.0.10) |      |
|  |     |                   |        |            |            |      |
|  |  veth-podA              |        |         veth-podB       |      |
|  |     |                   |        |            |            |      |
|  |  [br-int OVS]           |        |     [br-int OVS]        |      |
|  |     |                   |        |            |            |      |
|  |  [Geneve tunnel port] --+--------+--> [Geneve tunnel port] |      |
|  |     |                   |        |            |            |      |
|  |  [eth0 NIC]             |        |      [eth0 NIC]         |      |
|  +-------+-----------------+        +------------+------------+      |
|          |                                       |                   |
|          +----- Physical underlay network -------+                   |
|                                                                      |
|  OVN Central (NBDB/SBDB plus northd, 3-node Raft):                   |
|     |- Logical Switch "ls1" with ports "podA" and "podB"             |
|     |- 24 ingress plus 10 egress logical-flow tables (OVN 22.03)     |
+----------------------------------------------------------------------+
```

Pod A sends a TCP packet to Pod B. The journey has 12 detailed stages. Each stage lists: (a) what happens, (b) the verification command, (c) a link to the deep-dive part.

## 0.2.3 Stage 1, the Pod A application calls `send()`

**What happens:** the container or Pod A is in its own network namespace. The application calls `send()` over a TCP socket. The kernel TCP stack pushes the packet down to the IP layer, and the IP layer looks up the local route table; the next-hop is the default gateway (the OVN LRP, accessed through the OVN-K8s gateway router). Source MAC equals the MAC of the veth on the namespace side, and destination MAC equals the MAC of the LRP gateway.

**Verification:**

```bash
[chassis-X]# nsenter -t $(pidof container-podA) -n ip route
default via 10.0.0.1 dev eth0 onlink
10.0.0.0/24 dev eth0 proto kernel scope link src 10.0.0.5

[chassis-X]# nsenter -t $(pidof container-podA) -n ss -tnp
ESTAB  0  0  10.0.0.5:54321  10.0.0.10:80  users:(("app",pid=12345,fd=4))
```

**Deep dive:** Part 8.0 (Linux namespaces plus cgroups), Part 8.1 (Linux bridge plus veth plus macvlan).

## 0.2.4 Stage 2, Pod A namespace veth-podA TX

**What happens:** the packet exits through the `eth0` interface of the Pod A namespace. `eth0` is in fact one end of a **veth pair**; the other end (for example `eth-podA-X`) is in the root namespace. The veth driver (`net/core/veth.c` in Linux kernel 5.15) zero-copies the packet from the TX queue at one end to the RX queue at the other end.

**Verification:**

```bash
[chassis-X]# nsenter -t $(pidof container-podA) -n ip -o link
1: lo: ...
N: eth0@if<peer>: ...

[chassis-X]# ip -o link | grep eth-podA-X
M: eth-podA-X@if<N>: ...

# Trace traffic leaving Pod A:
[chassis-X]# nsenter -t $(pidof container-podA) -n tcpdump -i eth0 -n
```

**Deep dive:** Part 8.1 §"Linux veth pair" plus `linux-onboard/file-descriptor-deep-dive.md` for the FD lifecycle when `clone(CLONE_NEWNET)` runs.

## 0.2.5 Stage 3, root namespace veth peer RX, br-int port

**What happens:** the peer end of the veth (`eth-podA-X`) is an OVS port of `br-int`. The kernel OVS module (`openvswitch.ko`) hooks into `dev_queue_xmit()`, intercepts the packet before it goes through the normal Linux bridge stack, and pushes it into the OVS datapath.

**Verification:**

```bash
[chassis-X]# ovs-vsctl list-ports br-int | grep eth-podA-X
eth-podA-X

[chassis-X]# ovs-ofctl show br-int | grep eth-podA-X
 15(eth-podA-X): addr:de:ad:be:ef:00:0a config: 0 state: 0
                    speed: 10000 Mbps now, 10000 Mbps max
```

OFPort 15 is the OpenFlow port number for this port.

**Deep dive:** Part 9.1 (OVS three-component architecture, `ovs-vswitchd` userspace plus `openvswitch.ko` kernel), Part 13.4 (`br-int` architecture).

---

## 0.2.6 Stage 4, datapath megaflow lookup

**What happens:** the kernel datapath receives the packet, extracts the miniflow (a key with the fields the classifier cares about), and consults a 3-tier cache in order:

1. **EMC** (Exact Match Cache, 8K entries per PMD in DPDK; an O(1) hash exact-match cache on the full N-tuple).
2. **SMC** (Signature Match Cache, OVS 2.15 and later, 16K entries, caches a hash signature to ukey).
3. **Megaflow cache** (TSS classifier with wildcarding, the last slow path in the kernel).

A hit in any cache applies the action of the corresponding ukey, and the packet exits immediately. A miss in all caches triggers an upcall to userspace.

**Verification:**

```bash
[chassis-X]# ovs-appctl dpif/show
system@ovs-system:
        lookups: hit:1234567 missed:8901 lost:0
        flows: 4521
        masks: hit:1230000 total:1234567 hit/pkt:0.99
        port 1: ovs-system (internal)
        port 2: br-int (internal)
        ...

[chassis-X]# ovs-appctl dpif-netdev/pmd-stats-show     # if DPDK
[chassis-X]# ovs-appctl dpif/dump-flows br-int | head -5
```

**Deep dive:** Part 9.2 (kernel datapath plus EMC/SMC/megaflow plus ukey lifecycle), Part 9.15 (classifier TSS plus `cls_subtable`).

## 0.2.7 Stage 5, upcall plus `ofproto-dpif` xlate (slow path)

**What happens:** if all cache tiers miss (the first packet of a new flow), the kernel `openvswitch.ko` sends a Netlink generic genl **upcall** to `ovs-vswitchd`. Userspace `ofproto-dpif` runs `xlate_actions()` to simulate the full OpenFlow pipeline on this flow, producing:

- The actual **action set** (`output`, `set_field`, `encap`, and so on).
- The **megaflow mask** (wildcard mask, through the `wc` flag tracked during xlate).

Userspace then installs the ukey plus a flow rule into the kernel datapath through the reverse netlink path. The first packet itself is also re-injected by userspace.

**Verification:**

```bash
[chassis-X]# ovs-appctl upcall/show
system@ovs-system:
  threads: 4
  upcalls: 8901  miss: 8000  flow_install: 4520  ...

[chassis-X]# ovs-appctl ofproto/trace br-int \
    "in_port=15,ip,nw_src=10.0.0.5,nw_dst=10.0.0.10,..."
# Output: full pipeline trace with metadata, register, action
```

**Deep dive:** Part 9.1 §9.1.Y `ofproto-dpif` xlate tier 2 (Phase I.A1, function-call chain `xlate_actions` to `do_xlate_actions` to action translation), Part 9.25 (`ofproto/trace` 3 advanced GE).

## 0.2.8 Stage 6, logical pipeline ingress (`LS_in_*`)

**What happens:** during xlate, the flow rules on `br-int` are organized by `ovn-controller` into OpenFlow tables forming the OVN logical pipeline (translated from 24 ingress plus 10 egress tables for an LS, OVN 22.03). Each table is a logical stage (for example, `ls_in_check_port_sec`, `ls_in_acl`, `ls_in_lookup_fdb`). The packet flows through them in order; each stage matches on the highest-priority logical flow and runs an action `next;` (go to the next stage), `drop;`, or `output;`.

**Verification:**

```bash
# Logical view (high level):
[chassis-X]# ovn-trace ls1 \
    'inport == "podA" && eth.src == ... && ip4.src == 10.0.0.5 && ip4.dst == 10.0.0.10 && tcp'

# Output shows each ls_in_* stage matching with priority and action.
```

**Deep dive:** Part 13.2 (Logical Switch plus Logical Router pipeline 24 plus 10 stages), Part 20.7 Level 1-2 (`ovn-trace` tutorial gradient).

---

## 0.2.9 Stage 7, `ls_in_l2_lkup`, choose `outport`

**What happens:** the last ingress stage is `ls_in_l2_lkup` (table 27 in OVN 22.03). The logical flow here matches `eth.dst` against the address binding of an LSP. The action `outport = "podB"; output;` sets the logical outport to the destination LSP name and then jumps into the egress pipeline.

If the destination MAC is not in NBDB (unknown), the fallback action `outport = MC_FLOOD; output;` floods to every port of the LS.

**Verification:**

```bash
[chassis-X]# ovn-sbctl lflow-list ls1 | grep "ls_in_l2_lkup"
  table=27(ls_in_l2_lkup), priority=50, match=(eth.dst == 00:00:00:00:00:0b),
    action=(outport = "podB"; output;)
  table=27(ls_in_l2_lkup), priority=0, match=(1),
    action=(outport = "_MC_unknown"; output;)
```

**Deep dive:** Part 13.2 §"L2 lookup", Part 17.0 (FDB poisoning case study, behavior on unknown MAC).

## 0.2.10 Stage 8, logical pipeline egress (`LS_out_*`)

**What happens:** after `outport` is set, the packet flows through the egress pipeline (10 stages for an LS): `ls_out_pre_acl`, `ls_out_acl`, `ls_out_qos`, `ls_out_stateful`, `ls_out_port_sec`, `ls_out_port_sec_l2`. Egress applies ACLs in the reverse direction (`to-lport`) plus port security on the outport.

If the packet passes everything, the last stage `ls_out_port_sec_l2` runs the action `output;`. `ovn-controller` translates that into the actual OpenFlow output: either `output:<local-OFPort>` if podB is on the local chassis, or `output:<tunnel-OFPort>` if podB is on another chassis (consulted through `Port_Binding.chassis` in SBDB).

**Verification:**

```bash
[chassis-X]# ovn-sbctl list Port_Binding podB
_uuid               : ...
chassis             : <chassis-Y-uuid>     <- podB is on chassis Y
logical_port        : "podB"
tunnel_key          : 2
type                : ""
up                  : true
```

Because `chassis = chassis-Y`, the `br-int` on Chassis X will encapsulate plus output to a tunnel.

**Deep dive:** Part 13.2 §"egress pipeline", Part 13.5 (`Port_Binding` 8 types plus claim mechanism), Part 13.7 §13.7.8 (`consider_port_binding` source code).

## 0.2.11 Stage 9, Geneve encapsulation (TLV class 0x0102)

**What happens:** when the outport maps to a remote chassis, `ovn-controller` generates an OpenFlow flow at table 65 with three main actions (see Part 13.7.8 §"Geneve TLV encoding"):

```
set_field:<datapath-tunnel_key>->tun_id        # 24-bit Geneve VNI = LS tunnel_key
set_field:<outport-key>->NXM_NX_TUN_METADATA0[0..31]  # 32-bit logical outport
move:NXM_NX_REG14[0..14]->NXM_NX_TUN_METADATA0[16..30] # 15-bit logical inport
output:<tunX-Y-OFPort>
```

Geneve TLV class `0x0102` (the OVN-reserved class in the IANA registry) carries logical metadata across chassis. Outer header: Eth(chassis MAC) plus IPv4(underlay) plus UDP(dest 6081) plus Geneve(VNI plus TLV).

**Verification:**

```bash
[chassis-X]# ovs-appctl ofproto/trace br-int "in_port=15,..." | grep -A 5 "set_field:.*tun_id"
[chassis-X]# tcpdump -i eth0 -nn udp port 6081 -X | head -40
# Decode TLV class 0x0102 byte by byte (Part 9.27 §Geneve TLV deep dive)
```

**Deep dive:** Part 11.0 (VXLAN, Geneve, STT, RFC 8926), Part 13.7 §13.7.8 (`controller/physical.c put_encapsulation()`), Part 9.27 (3-tier parallel diagnostic plus Geneve TLV).

## 0.2.12 Stage 10, physical NIC TX plus underlay routing

**What happens:** the packet is now fully encapsulated as a UDP/Geneve packet with outer src equal to the chassis-X IP and outer dst equal to the chassis-Y IP. The kernel IPv4 stack does a route lookup and finds the next-hop on the underlay (it could be a ToR switch). The NIC driver `eth0` transmits the packet on the wire.

The underlay must guarantee:

- The MTU is large enough for the Geneve overhead (66 bytes equals 14 outer Eth plus 20 outer IPv4 plus 8 UDP plus 8 Geneve base plus 16 TLV). If the underlay MTU is 1500 and the inner is 1500, fragmentation or PMTU drop will occur.
- UDP port 6081 is not dropped by a firewall.
- Routing or switching between chassis-X and chassis-Y has no loops, and ARP resolves the next-hop.

**Verification:**

```bash
[chassis-X]# ip route get <chassis-Y-IP>
<chassis-Y-IP> via <next-hop> dev eth0 src <chassis-X-IP>

[chassis-X]# ovs-appctl bfd/show
        State: UP        <- BFD session with chassis-Y is healthy
        Diagnostic: No Diagnostic
        Min tx: 1000ms   Min rx: 1000ms

[chassis-X]# tcpdump -i eth0 -nn udp port 6081 and host <chassis-Y-IP>
```

**Deep dive:** Part 11.1 (overlay MTU plus PMTUD plus offload), Part 13.6 (HA chassis plus BFD).

## 0.2.13 Stage 11, Chassis Y NIC RX plus Geneve decapsulation

**What happens:** the NIC `eth0` on Chassis Y receives the UDP/6081 packet. The kernel stack delivers it to the OVS tunnel port (`tunY-X` in `br-int` of Chassis Y). The `br-int` on Chassis Y receives the packet with:

- `in_port = <tunnel-OFPort>`.
- `tun_id = <datapath tunnel_key>` (the Geneve VNI).
- `tun_metadata0` containing the outport (32 bits) plus the inport (15 bits) encoded on Chassis X.

Table 0 of the `br-int` on Chassis Y has a flow dedicated to decap (installed by `ovn-controller`):

```
in_port=<tunnel-port>,tun_id=<tk>:
    move:NXM_NX_TUN_METADATA0[0..31]->NXM_NX_REG15[0..31]   <- restore outport
    move:NXM_NX_TUN_METADATA0[16..30]->NXM_NX_REG14[0..14]  <- restore inport
    move:NXM_NX_TUN_ID[0..23]->OXM_OF_METADATA[0..23]       <- restore datapath
    resubmit(,<egress-table>)
```

After decap, the packet rejoins the LS logical egress pipeline, this time on Chassis Y, with the outport set for the local podB.

**Verification:**

```bash
[chassis-Y]# ovs-vsctl get Interface tunY-X options
{remote_ip="<chassis-X-IP>", key=flow}

[chassis-Y]# ovs-appctl ofproto/trace br-int \
    "in_port=<tunnel-OFPort>,tun_id=0x1,tun_metadata0=0x..."
```

**Deep dive:** Part 13.4 (`br-int` Chassis Y architecture), Part 13.7 §13.7.8 (decap mirror with encap), Part 9.27 (3-tier diagnostic Chassis Y side).

## 0.2.14 Stage 12, `br-int` Y output to veth peer to Pod B namespace

**What happens:** the last egress stage on `br-int` of Chassis Y has the flow `output:<podB-OFPort>`. `br-int` forwards the packet to the OVS port that connects to the veth peer of Pod B. The veth driver (peer-to-peer) moves the packet to the other end (`eth0` in the Pod B namespace). The Pod B namespace kernel stack delivers it to the application through the TCP socket.

**Verification:**

```bash
[chassis-Y]# ovs-vsctl list-ports br-int | grep eth-podB-Y
eth-podB-Y

[chassis-Y]# nsenter -t $(pidof container-podB) -n tcpdump -i eth0 -n
# Pod B receives the packet with the outer underlay stripped plus L2 dst equal to the MAC of the Pod B veth
```

**Deep dive:** Part 8.1 (veth peer), Part 13.5 (`type=""` equals vif, the default LSP type).

## 0.2.15 Diagnostic command summary by stage

When debugging "podA cannot ping podB", run the following tier by tier:

| Tier | Question | Command |
|------|----------|---------|
| Logical | Is OVN's intent correct? | `ovn-trace ls1 '<flow-expr>'` (Part 20.7 Level 1-3) |
| OpenFlow | Is the OpenFlow on `br-int` correct? | `ovs-appctl ofproto/trace br-int '<flow>'` (Part 9.25) |
| Datapath | Does the kernel cache have an entry? | `ovs-appctl dpif/dump-flows br-int` (Part 9.2) |
| Physical X | Is Chassis X transmitting? | `tcpdump -i eth0 udp port 6081` (Part 11.1, 13.7.8) |
| Underlay | Is the underlay swallowing packets? | `tcpdump` on the router or ToR; check MTU, BFD (Part 13.6) |
| Physical Y | Is Chassis Y receiving? | `tcpdump -i eth0 udp port 6081` |
| OpenFlow Y | Does `br-int` Y decap correctly? | `ovs-appctl ofproto/trace br-int 'in_port=<tun>,tun_id=...'` |
| Pod B | Does Pod B receive? | `nsenter -n tcpdump -i eth0` |

If the test fails at a tier, that tier is the root cause. Part 9.27 describes the 3-tier parallel diagnostic framework in more detail.

## 0.2.16 Guided Exercise, trace a packet through the 12 stages yourself

**Goal:** run the packet journey on a real lab or a Mininet sandbox, and compare the output of each stage with the description in this part.

**Assumed setup:** an OVN sandbox with two chassis (see Part 0.1 §"Two-node chassis pair"). Create one Logical Switch `ls1`, two LSPs `podA` (chassis-X) plus `podB` (chassis-Y), with addresses `00:00:00:00:00:0a 10.0.0.5` and `00:00:00:00:00:0b 10.0.0.10`.

**Steps:**

```bash
# Pre-check: cluster healthy?
[host]# ovn-nbctl show              # podA plus podB exist
[host]# ovn-sbctl show               # 2 chassis registered
[host]# ovs-appctl bfd/show          # tunnel UP

# Run ping from podA
[chassis-X]# nsenter -t $(pidof container-podA) -n ping -c 3 10.0.0.10

# Trace in parallel on all 8 tiers (run each command in a separate terminal or tmux pane)
[host]# ovn-trace ls1 'inport == "podA" && ...'
[chassis-X]# ovs-appctl ofproto/trace br-int "in_port=$(ovs-vsctl get int eth-podA-X ofport),..."
[chassis-X]# ovs-appctl dpif/dump-flows br-int
[chassis-X]# tcpdump -i eth0 udp port 6081 -c 5 -X
[chassis-Y]# tcpdump -i eth0 udp port 6081 -c 5 -X
[chassis-Y]# ovs-appctl ofproto/trace br-int "in_port=$(ovs-vsctl get int tunY-X ofport),tun_id=..."
[chassis-Y]# ovs-appctl dpif/dump-flows br-int
[chassis-Y]# nsenter -t $(pidof container-podB) -n tcpdump -i eth0 -c 5
```

**Lessons:**

1. Each stage has **one kind of evidence** that you can verify. No stage is taken on trust; there is always a verification command.
2. When one stage fails, never skip to the next. Fix the root cause first.
3. These 12 stages are the **backbone** of every OVN debug session. Once you internalize them, you can lead any incident report on your own.

## 0.2.17 Key points to remember

**One.** The OVN packet journey has 12 clear stages, from the application call to `send()` to the application call to `recv()` on the destination chassis. Each stage has a concrete verification command.

**Two.** Separate the **logical view** (`ovn-trace`, NBDB intent) from the **physical view** (`ofproto/trace`, OVS OpenFlow). The two layers can match or diverge (a divergence means `ovn-controller` has not yet synchronized).

**Three.** Geneve TLV class `0x0102` carries logical metadata (datapath tunnel_key plus logical inport plus outport) across chassis. Losing the TLV means losing the logical context, which means a packet drop or wrong-pipeline processing.

**Four.** The underlay must guarantee enough MTU plus UDP 6081 not blocked by a firewall plus BFD UP. These three are the most common causes of "podA cannot ping podB cross-chassis".

**Five.** Part 0.2 is the entry point. Every concrete debug story is in a deeper part. Cross-link table:

| Stage | Deep-dive part |
|-------|----------------|
| 1-2 namespace plus veth | 8.0, 8.1 |
| 3 br-int port | 9.1, 13.4 |
| 4 datapath megaflow | 9.2, 9.15 |
| 5 upcall plus xlate | 9.1 §9.1.Y, 9.25 |
| 6-7 LS ingress plus l2_lkup | 13.2, 17.0, 20.7 |
| 8 LS egress plus Port_Binding | 13.2, 13.5, 13.7.8 |
| 9 Geneve encap | 11.0, 13.7.8, 9.27 |
| 10 NIC TX plus underlay | 11.1, 13.6 |
| 11 Chassis Y decap | 13.4, 13.7.8, 9.27 |
| 12 br-int Y output plus Pod B | 8.1, 13.5 |

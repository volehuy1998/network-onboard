## Bối cảnh: SDN không cần chạm hardware

Part 5.0 đã phân tích API-based SDN giữ nguyên silicon và add YANG/NETCONF layer. Đây là approach conservative, bảo toàn vendor investment. Phần này đi ngược hướng: **thay hoàn toàn network edge** từ physical ToR switch sang software hypervisor, đi qua Open vSwitch và controller cluster. Underlay (physical network) giữ nguyên chạy IP routing (OSPF/BGP) đơn giản, không cần SDN-aware.

Đây là paradigm **hypervisor-edge overlay SDN**, khởi xướng bởi Nicira Networks 2011 với product NVP (Network Virtualization Platform). Điểm critical: NVP không replace hardware switch, chỉ replace network **edge** từ ToR sang hypervisor vNIC. Hardware switch trong fabric chỉ cần route IP giữa hypervisor, overlay protocol (STT initially, sau này VXLAN/Geneve) handles L2 abstraction và per-tenant isolation.

Paradigm này thành công trong DC virtualization ngay lập tức, được VMware acquire 2012 và rebrand NSX. Legacy team sau đó fork OVN 2015 tại Red Hat, continue development open-source. Hiểu NVP → NSX → OVN lineage là key để hiểu OVN architecture hiện nay trong bất kỳ DC deployment nào 2026.

## 5.1.1 Nicira NVP: 2011 commercial product

### Founding và key people

Nicira Networks founded tháng 8/2007 tại Palo Alto bởi 3 người:
- **Martin Casado**: PhD Stanford, author chính của Ethane paper SIGCOMM 2007 (Part 2.4) và OpenFlow original paper CCR 04/2008 (Part 3.0).
- **Nick McKeown**: Stanford professor, co-author OpenFlow paper, faculty advisor cho Casado.
- **Scott Shenker**: UC Berkeley professor, co-author Ethane, Clean Slate Program co-lead.

Initial funding: Andreessen Horowitz + Lightspeed Venture Partners + New Enterprise Associates. Company stealth mode 2007-2011, publish first products 2011.

### Product launch 2011

NVP 1.0 launch tháng 10/2011. Components:

**Nicira Controller Cluster**: 3-5 node Java application, store logical topology trong distributed database. Expose REST API cho provisioning (create logical switch, attach VM, install ACL). Internal coordination qua ZooKeeper ensemble.

**Open vSwitch on hypervisor**: OVS instance chạy trên mỗi host (Linux KVM, Xen, ESXi post-2013). OVS là Nicira open-source contribution từ 2008-2011, mainlined Linux kernel 3.3 (02/2012).

**OpenFlow 1.0 + NXM extensions**: controller program OVS qua OpenFlow protocol, dùng NXM (Nicira Extensible Match) cho features beyond 12-tuple. NVP controller sinh flow rules based on logical topology.

**STT (Stateless Transport Tunneling)**: overlay encapsulation, Nicira invention. STT emulate TCP header để NIC checksum offload work với overlay packet, giải pain point VXLAN performance trên NIC không có tunnel offload native. Sau này STT lost to VXLAN/Geneve khi NIC offload mature, nhưng 2011-2014 là competitive advantage.

### Production customer

NVP early customer (2011-2012): eBay, AT&T, Rackspace, NTT. Use case: multi-tenant private cloud và public cloud provider, software-defined overlay cho tenant isolation, không cần rewire physical network.

eBay public presentation 2012 mô tả NVP deployment trên 10^4 VMs, achieve tenant isolation + micro-segmentation mà physical VLAN-based approach không đạt được scale.

## 5.1.2 Hypervisor-edge OVS + centralized controller architecture

### Triết lý "edge là hypervisor"

Traditional DC network topology:

```
[Core spine] ←→ [Leaf ToR] ←→ [Rack 1: 16 hypervisors with VMs]
                [Leaf ToR] ←→ [Rack 2: 16 hypervisors with VMs]
```

"Network edge" theo convention là ToR switch, port facing hypervisor NIC. Logical concept (VLAN, ACL, QoS) implement trên ToR.

Nicira thay đổi: network edge là **vNIC của VM**, handled by OVS trong hypervisor. Concept flow:

```
VM ←→ OVS (in hypervisor) ←→ hypervisor physical NIC ←→ ToR (L3 only) ←→ fabric
```

OVS handle VLAN tag, ACL, NAT, tunnel encapsulation. ToR không biết VLAN structure của tenant; ToR chỉ route IP packets của tunnel (STT/VXLAN) giữa hypervisor.

### Hệ quả architectural

**1. Underlay simplification**: hardware switch không cần SDN-aware. Chạy standard routing protocol (OSPF, BGP, ECMP). Operator deploy bất kỳ vendor nào (Cisco, Juniper, Arista, whitebox) mà không cần OpenFlow support.

**2. Multi-tenant isolation qua overlay**: mỗi tenant có logical topology riêng. OVS encap traffic với tunnel header chứa tenant ID (STT tunnel ID, VXLAN VNI). Fabric route based on outer IP, nội dung overlay transparent với physical network.

**3. Controller scale**: centralized control plane quản lý N hypervisor, không quản lý M ToR switch (N >> M trong DC nhỏ nhưng N < M trong hyperscale with 10^3 racks). Controller workload scale với compute density, không with physical switch count.

**4. Deployment flexibility**: thêm tenant = program OVS qua controller, không cần change ToR config. Deployment time từ weeks (hardware VLAN provisioning) xuống minutes (software config).

### Trade-off

**Performance**: overlay encap overhead (STT ~54 bytes header, VXLAN 50 bytes, Geneve 58 bytes). Packet throughput giảm vì MTU reduction. Hardware NIC offload (TSO, checksum) cần work với tunnel, initial NIC 2011-2013 không support hoàn toàn, NVP dùng STT để mimic TCP và tricks offload.

**Debugging complexity**: troubleshoot tunnel issue cần inspect 2 layer (outer fabric + inner tenant). Tools như `ovs-appctl ofproto/trace` (Part 9.4) + tcpdump on overlay interface critical.

**Fabric requirements**: fabric phải IP-only (no bridging across rack). Yêu cầu L3-to-ToR design (spine-leaf với routing từ edge). Traditional L2 fabric (Cisco STP-based) không compatible.

### Hiểu sai thường gặp

**Hiểu sai:** "Hypervisor overlay không dùng được vì performance penalty."

**Thực tế:** Performance penalty của overlay (STT/VXLAN/Geneve) là ~5-10% throughput trên NIC không có tunnel offload, giảm xuống ~2-3% trên NIC hiện đại (Mellanox ConnectX-5+, Intel 800 series) có hardware VXLAN TSO. So với benefit (multi-tenant isolation, rapid provisioning, programmable policy), trade-off worth it trong DC virtualization. Hyperscaler Azure, Google Cloud, AWS VPC tất cả dùng overlay architecture. Penalty "too high" là myth từ 2011 benchmark trên legacy hardware.

## 5.1.3 VMware acquisition 23/07/2012 (1,26 tỷ USD)

### Deal context

Tháng 7/2012, VMware announce acquisition Nicira Networks với giá 1,26 tỷ USD all-cash. Thương vụ SDN lớn nhất lịch sử lúc đó, kỷ lục giữ đến Broadcom acquire Brocade 5,5 tỷ USD năm 2017 (enterprise networking broader scope) và Cisco acquire Splunk 28 tỷ USD 2023 (analytics, not pure networking).

Press release VMware chính thức 23/07/2012 tại `https://www.vmware.com/company/news/releases/vmw-newsfeed/VMware-Completes-Acquisition-of-Nicira/1830683` với statement:

> *"With Nicira, VMware will offer a complete set of infrastructure products for the software-defined datacenter, putting us in a unique position to deliver on the vision of the modern software-defined datacenter."*

### Chiến lược VMware

2012 context: VMware dominant compute virtualization với vSphere ESXi, revenue từ hypervisor license. Cisco approach SDN (pre-ACI era, research phase) threat to VMware, Cisco potentially build SDN tightly coupled với Cisco UCS compute, bypass VMware.

Acquire Nicira cho phép VMware:
- Own SDN story trước Cisco launch ACI (12/2013).
- Integrate NVP technology vào vSphere roadmap, creating "software-defined datacenter" (SDDC) umbrella category.
- Monetize SDN qua VMware's existing enterprise sales channel (10,000+ customer base).

### Impact on Nicira team

Core team transitioned vào VMware:
- Martin Casado: Senior VP + GM của Networking and Security Business Unit.
- Justin Pettit, Ben Pfaff, Teemu Koponen: engineering leads.
- Scott Shenker: không full-time, advisor role.

Nicira brand retired, product renamed **NSX** (Network and Security eXtensions). NSX 1.0 launch 03/2013 tại VMware Partner Exchange event.

### Evolution 2013-2015

NSX 1.0 (2013) = NVP renamed + vSphere integration.
NSX-V (2014) = NSX for vSphere, tight coupling với ESXi.
NSX-T (2016) = NSX-T Data Center, multi-hypervisor extended beyond vSphere.

Cross-ref 5.1.4 phân tích chi tiết NSX-V vs NSX-T architectural differences.

## 5.1.4 NSX-V (vSphere) vs NSX-T (multi-hypervisor)

### NSX-V (2013-2021)

**Target platform**: VMware vSphere ESXi exclusive. ESXi kernel module integration tight.

**Components**:
- **NSX Manager**: management plane, REST API cho vCenter integration.
- **NSX Controller cluster**: control plane, 3-node cluster.
- **VTEPs (VXLAN Tunnel EndPoints)**: trên mỗi ESXi host, VMware's own VXLAN implementation (not OVS).
- **Edge Services Gateway (ESG)**: virtual appliance cho north-south routing, NAT, firewall, VPN.
- **Distributed Firewall**: hypervisor-embedded stateful firewall, per-vNIC granular.

**Data plane**: ESXi kernel VXLAN stack (không phải OVS). Independent code base từ NVP era, optimize cho ESXi.

**End-of-life**: NSX-V announce EOL 2022, extended support 01/2024. Customer migration to NSX-T.

### NSX-T (2016-present)

**Target platform**: multi-hypervisor:
- VMware vSphere ESXi.
- KVM (Linux).
- Bare-metal servers (runs NSX-T agent directly).
- Container (integration với Kubernetes qua NCP - NSX Container Plugin).

**Components**:
- **NSX-T Manager**: unified management + control plane (consolidated from NSX-V's split).
- **NSX-T Edge nodes**: virtual appliance hoặc bare-metal, scaled horizontally.
- **Transport Nodes**: ESXi, KVM, bare-metal nodes running NSX-T agent.

**Data plane**: on ESXi dùng VMware proprietary VTEP. Trên KVM dùng OVS. Trên bare-metal dùng NSX-T DPU (ConnectX-based SmartNIC later).

**Geneve thay VXLAN**: NSX-T chuyển sang Geneve encapsulation (RFC 8926) cho flexibility TLV options. Backward compat với VXLAN overlay cho migration.

### So sánh NSX-V vs NSX-T

| Aspect | NSX-V | NSX-T |
|--------|-------|-------|
| Hypervisor | vSphere only | vSphere + KVM + bare-metal |
| Data plane | ESXi kernel VXLAN | Mixed: ESXi VTEP + KVM OVS + DPU |
| Overlay | VXLAN primary | Geneve primary (VXLAN backward compat) |
| Management | NSX Manager + separate Controller | Unified NSX-T Manager |
| Edge | ESG virtual appliance | Edge cluster (scale-out) |
| Container integration | Limited | NCP first-class integration Kubernetes |
| Status 2026 | EOL, migration target | Active development, VMware flagship |

### Hiểu sai thường gặp

**Hiểu sai:** "NSX = Nicira NVP rebrand, cùng technology."

**Thực tế:** NSX-V (2013) reuse **concept** từ NVP (hypervisor edge, controller cluster, overlay tunnel) nhưng code base substantial rewrite cho ESXi integration. NSX-V không dùng OVS; ESXi kernel VTEP là VMware proprietary. Chỉ NSX-T (2016) quay lại OVS (on KVM path). Legacy lineage "NVP → NSX" là philosophical, không phải code-level identical.

## 5.1.5 Juniper Contrail lineage: tương tự nhưng BGP EVPN control plane

### Contrail architecture (simplified)

Juniper Contrail (acquired Contrail Systems 12/2012, product launch 2013) triển khai hypervisor-edge paradigm tương tự NVP/NSX, khác biệt chính ở **control plane**:

**NVP/NSX**: Nicira controller (Java) với ZooKeeper coordination, southbound OpenFlow + NXM.

**Contrail**: Control plane layer sử dụng **BGP EVPN** (RFC 7432) cho MAC/IP route signaling + **XMPP** cho controller-to-vRouter southbound.

### BGP EVPN path

**Data plane**: vRouter (kernel module hoặc DPDK) trên mỗi compute node.
**Control plane**: Controller peer với vRouter qua XMPP, peer với external WAN qua MP-BGP (multi-protocol BGP).

Tenant L2/L3 route distribute qua BGP EVPN:
- **Type 2 (MAC/IP Advertisement)**: tenant VM's MAC + IP.
- **Type 3 (Inclusive Multicast)**: BUM (Broadcast/Unknown/Multicast) suppression.
- **Type 5 (IP Prefix Route)**: L3 routes cho subnet.

### Trade-off Contrail vs NVP/NSX

**Advantage Contrail**:
- WAN integration native qua BGP: service provider có BGP infrastructure sẵn, integrate seamlessly.
- DC-to-DC interconnect qua standard BGP EVPN, interoperable với Cisco/Juniper/Arista router supports EVPN.
- MP-BGP routing proven scale 10^5+ routes.

**Disadvantage Contrail**:
- Complexity: BGP EVPN cần BGP expertise, steep learning curve cho team chỉ có vSphere admin background.
- Debugging harder: state distributed across BGP sessions, not centralized database.

### Contrail evolution

- **2014**: Juniper open-source as **OpenContrail**.
- **2018**: Handed to Linux Foundation as **Tungsten Fabric** (<tungsten.io>), commercial still as Juniper Contrail.
- **Current 2026**: Tungsten Fabric active community, Juniper focus Contrail on SD-WAN + enterprise (Part 7.3).

### Hiểu sai thường gặp

**Hiểu sai:** "Contrail = open-source NSX."

**Thực tế:** Contrail có architecture khác NVP/NSX meaningfully. Control plane BGP EVPN vs NVP/NSX's controller-centric model. Data plane vRouter (custom) vs NVP/NSX's OVS (on KVM path). Use case target khác: Contrail primary service provider WAN + multi-DC, NVP/NSX primary enterprise DC virtualization. Treating chúng equivalent là oversimplification.

## 5.1.6 Di sản: OVN là spiritual successor của NVP

### Team migration 2015

Khoảng 2014-2015, key engineers của Nicira core team rời VMware về Red Hat:
- **Justin Pettit**: senior engineer OVS, co-author NVP.
- **Ben Pfaff**: chief architect OVS, core author NSDI 2015 paper.
- **Teemu Koponen**: architect Nicira controller.

Ý định: rebuild open-source alternative to NSX, không tightly coupled với VMware. Result: **OVN (Open Virtual Network)** project.

### OVN launch 13/01/2015

Official announcement qua **Network Heresy blog post** ngày 13/01/2015 bởi Justin Pettit, Ben Pfaff, Chris Wright (Red Hat CTO), và Madhu Venugopal (Docker):

> URL: `https://networkheresy.com/2015/01/13/ovn-bringing-native-virtual-networking-to-ovs/`

Announce OVN là open-source project cho virtual networking over OVS, thiết kế portable với nhiều orchestrator (container runtime + VM orchestrator). Key technical difference từ NVP:

**NVP**: controller cluster với ZooKeeper + OpenFlow.
**OVN**: **NBDB (Northbound Database) + SBDB (Southbound Database)** cho control plane, OVSDB protocol (RFC 7047, Part 10.0). ovn-northd translate NB intent → SB logical flow.

### Architecture philosophy

OVN keep NVP philosophy intact:
- Hypervisor-edge với OVS.
- Centralized control plane.
- Overlay tunnels (Geneve default, VXLAN optional).
- Multi-tenant via Logical_Switch abstraction.

Changes:
- Replace proprietary Nicira controller với OVSDB-based distributed database (NBDB/SBDB).
- Replace OpenFlow from controller với OVSDB SB intent + ovn-controller translate to OpenFlow locally.
- Open-source entire stack (OVS + OVN monorepo).

### Current state 2026

OVN là de-facto hypervisor overlay SDN trong multiple deployment contexts: Kubernetes CNI (OVN-Kubernetes), bare-metal OVN standalone, và như driver portable cho nhiều virtualization orchestrator. Production at:
- **Red Hat OpenShift Container Platform**: OVN-Kubernetes default CNI from 4.12 (2023).
- **Microsoft Azure**: internal overlay framework dùng OVN concepts.
- **Carrier NFV**: NTT, Orange, Deutsche Telekom infrastructure deployments.

OVN maintained upstream at `https://github.com/ovn-org/ovn`, separate từ OVS (`https://github.com/openvswitch/ovs`) sau split 2018.

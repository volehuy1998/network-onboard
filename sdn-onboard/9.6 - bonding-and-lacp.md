## 9.6.1 OVS bonding khác Linux kernel bonding

OVS triển khai bond driver độc lập với kernel `bonding` module. Lý do: OVS cần bond state tích hợp với OpenFlow pipeline + OVSDB transaction, không thể dựa vào opaque kernel abstraction.

Hệ quả: trên cùng host, có thể dùng OVS bond cho uplink br-int đồng thời với kernel bond cho management interface — hai stack không conflict.

---

## 9.6.2 Ba bond mode

**`active-backup`**: một member active, khác standby. Không yêu cầu peer switch hỗ trợ đặc biệt. Failover dựa vào carrier state + optional `bond_updelay`/`bond_downdelay`.

**`balance-slb`** (source-MAC load balancing): hash theo source MAC, rebalance định kỳ (`bond-rebalance-interval` ms). KHÔNG yêu cầu peer cooperation. Trade-off: một source MAC pin vào một member, không tận dụng nhiều link cho một flow.

**`balance-tcp`**: L2+L3+L4 hash. YÊU CẦU LACP + MLAG-capable peer (Cisco vPC, Arista MLAG, Juniper ESI). Phức tạp nhưng tận dụng bandwidth đầy đủ.

Tạo bond (compass Ch E):

```bash
[host]# ovs-vsctl add-bond br0 bond0 eth1 eth2 \
         bond_mode=balance-slb \
         lacp=active other_config:lacp-time=fast \
         bond_updelay=100 bond_downdelay=100
```

---

## 9.6.3 LACP options

- `lacp=active`: gửi LACPDU proactively (1s fast-timer, 30s slow-timer).
- `lacp=passive`: chỉ reply khi nhận LACPDU.
- `lacp=off`: tắt LACP (dùng với balance-slb hoặc active-backup).

`other_config:lacp-time=fast` — giảm LACPDU interval từ 30s xuống 1s, failover sub-100ms thay vì sub-90s.

`other_config:lacp-fallback-ab=true` — nếu LACP negotiation fail (peer không respond trong 90s), tự động fallback sang active-backup. Foot-gun-saver quan trọng khi peer switch đang upgrading.

## 9.6.4 Verification: bond/show và lacp/show

```bash
[host]# ovs-appctl bond/show bond0
[host]# ovs-appctl lacp/show bond0
```

`bond/show` output: `bond_mode`, `lacp_status`, active slave, `may_enable` state mỗi member, rebalance countdown.

`lacp/show` output: Actor + Partner system_id, key, priority, state bits. State bit chuẩn theo IEEE 802.1AX:

- `A`ctivity — Actor active LACP
- `C`ollecting — collecting frames
- `E`xpired — state expired
- `G`ood (synchronized)
- `I`n sync với peer
- `K`ey matched
- `N`ew connection
- `P`assive mode

"Letter salad `ACEGIKNP` trên cả hai phía" = healthy bundled link (compass Ch E).

## 9.6.6 Anatomy Template A — `ovs-appctl bond/show`

**Bối cảnh khi chạy:** Sau khi add bond + LACP, verify bundling state trước khi đưa vào production. Cũng là first-line debug khi traffic pattern asymmetric.

**Command và output** *(reproduced OVS 2.17.9, 2-member bond balance-slb)*:

```shell
$ sudo ovs-appctl bond/show bond0

---- bond0 ----
bond_mode: balance-slb
bond may use recirculation: no, Recirc-ID : -1
bond-hash-basis: 0
lb_output action: disabled, bond-id: -1
updelay: 100 ms
downdelay: 100 ms
next rebalance: 9873 ms
lacp_status: negotiated
lacp_fallback_ab: true
active-backup primary: <none>
active member mac: 00:15:5d:01:a3:7f(eth1)

member eth1: enabled
  active member
  may_enable: true
  hash 42: 10 kB load
  hash 59: 3 kB load

member eth2: enabled
  may_enable: true
  hash 3: 7 kB load
  hash 76: 1 kB load
```

**10-attribute anatomy:**

| Field | Giá trị mẫu | Ý nghĩa | Red flag nếu |
|-------|-------------|---------|--------------|
| `bond_mode` | `balance-slb` / `balance-tcp` / `active-backup` | 3 mode | Mismatch với peer expectation = bond flap |
| `bond may use recirculation` | `no` / `yes` | balance-tcp cần recirc | `yes` = balance-tcp, cần MLAG peer |
| `updelay` / `downdelay` | 100 ms | Delay từ carrier up/down → enable/disable member | Quá thấp = flapping, quá cao = slow failover |
| `next rebalance` | 9873 ms | Countdown tới next rebalance cycle (default 10s) | <1s = aggressive rebalance, workload scatter |
| `lacp_status` | `negotiated` / `configured` / `off` | LACP state | `configured` = không negotiate (peer issue) |
| `lacp_fallback_ab` | `true` | Fallback to active-backup nếu LACP fail | `false` + LACP fail = bond total down (foot-gun) |
| `active member mac` | `00:15:5d:...` | MAC của member active (cho SLB anchor) | Change frequently = active flapping |
| Per-member `enabled` | `true` / `false` | Driver-level enable | `false` = manual disable hoặc LACP reject |
| Per-member `may_enable` | `true` / `false` | OVS-level may-enable (LACP synchronized) | `false` + `enabled=true` = LACP state issue |
| Per-member `hash N: X kB` | hash bucket → traffic load | SLB distribution evidence | Single hash dominate = workload có elephant flow |

**Kịch bản bẻ gãy:**

- **`lacp_status=configured` + peer không respond**: peer switch chưa enable LACP trên port-channel. Fix: config peer side, hoặc set `lacp=off` + accept static bonding.
- **`may_enable=false` persistent trên 1 member**: LACP partner state bit mismatch. Check `ovs-appctl lacp/show`.
- **Traffic all trên 1 member trong balance-slb**: single source MAC dominate (ví dụ 1 VM HFT). Rebalance countdown đếm nhưng không làm gì. Fix: upgrade lên balance-tcp nếu peer hỗ trợ MLAG.
- **Failover `updelay=100` không đủ**: carrier flap < 100ms được filter ra (good) nhưng persistent flap qua 100ms cause bond cascade. Solution: tăng lên 500ms hoặc debug link layer.

## 9.6.7 Hiểu sai + Key Topic

> **Hiểu sai phổ biến 1:** *"`balance-tcp` chắc chắn tốt hơn `balance-slb` vì distribute packet-level."*
>
> **Thực tế:** `balance-tcp` YÊU CẦU peer switch hỗ trợ LACP MLAG (Cisco vPC, Arista MLAG, Juniper ESI, Mellanox MLAG). Nếu peer chỉ là standalone switch LACP (non-MLAG), `balance-tcp` sẽ cause **packet reordering + MAC flapping** upstream → app layer TCP retransmit + performance tệ hơn `active-backup`. Quy tắc: verify MLAG ở peer trước khi chọn `balance-tcp`. Nếu không chắc chắn, default an toàn là `active-backup` hoặc `balance-slb`.

> **Hiểu sai phổ biến 2:** *"LACP fast-timer (1s) luôn tốt hơn slow-timer (30s) vì phát hiện nhanh hơn."*
>
> **Thực tế:** LACP fast-timer cost 30× LACPDU bandwidth (1 pkt/s × 2 direction = 2 pps/bond thay vì 0.067 pps). Với hạ tầng 1000+ bond × 2 pps = 2000 pps LACPDU chỉ riêng trong fabric. Switch CPU có thể overload khi fast-timer broadband. Enterprise deployment thường bắt đầu với slow-timer (30s), chỉ enable fast-timer cho critical bond (gateway + spine uplink).

> **Chủ đề trọng tâm:** Bond mode chọn sai là cause # 1 của "traffic imbalance" trong OVN deployment. Troubleshooting flow: (1) `ovs-appctl bond/show` check mode + per-member load. (2) Nếu balance-slb + single source dominate → upgrade balance-tcp (cần MLAG). (3) Nếu balance-tcp + packet reorder ở peer → downgrade balance-slb (peer không MLAG). (4) active-backup = default an toàn khi không chắc. Phần 9.26 §9.26.11 case study "LACP bond flap cascade" show scenario production với ToR firmware kích hoạt 200 chassis đồng loạt re-negotiate.

## 9.6.8 Common failure modes

**Failure 1 — Bond flapping do MAC flap upstream.**
Dấu hiệu: `bond/show` show active member đổi 3-5s/lần. Upstream switch log cho thấy MAC flap warning.
Nguyên nhân: balance-tcp trên non-MLAG peer.
Fix: downgrade balance-slb hoặc active-backup. Xác nhận peer MLAG via `show port-channel summary` (Cisco) hoặc equivalent.

**Failure 2 — LACP negotiation stuck ở "configured" không lên "negotiated".**
Dấu hiệu: `lacp_status=configured` persistent sau 2 phút.
Nguyên nhân: peer switch chưa enable LACP, hoặc key mismatch, hoặc system_id collision.
Fix: `ovs-appctl lacp/show` verify actor + partner system_id/key match. Nếu không match, config peer side.

**Failure 3 — Rebalance cycle cause packet loss.**
Dấu hiệu: Mỗi `bond-rebalance-interval` (default 10s), ~50-100 packet loss trên cross-bond traffic.
Nguyên nhân: SLB rebalance moves hash bucket from member A to member B, in-flight packet reorder.
Fix: Tăng `other_config:bond-rebalance-interval=30000` (30s) hoặc migrate to active-backup cho loss-sensitive workload.

**Failure 4 — Fallback-AB kick in khi peer restart.**
Dấu hiệu: `lacp_status=negotiated` → `configured` → fallback-ab active (1 member only) sau peer switch reboot.
Nguyên nhân: Peer LACPDU absence > 90s kích hoạt fallback.
Fix: Fallback-AB correct behavior. After peer recover, LACP re-negotiate, bond resume full bundle.

## 9.6.9 Performance characteristics

**Băng thông thực tế scaling:**

| Mode | Peer requirement | Băng thông thực tế (2× 10G) | Failover độ trễ |
|------|------------------|---------------------|-------------------|
| `active-backup` | None | 10 Gbps (one member only) | 100ms-1s (updelay) |
| `balance-slb` | None | 15-18 Gbps (uneven distribution) | Same as active-backup |
| `balance-tcp` | LACP + MLAG | 19-20 Gbps (near-linear) | Sub-100ms với fast-timer |

**Test methodology:** 100 parallel iperf TCP stream across bond. `balance-slb` plateau khi 1 stream lớn dominate hash bucket. `balance-tcp` best vì L2+L3+L4 hash distribute evenly per-flow.

**CPU overhead:**
- `active-backup`: <1% CPU per 10G link.
- `balance-slb`: 2-5% CPU (hash computation + rebalance logic).
- `balance-tcp`: 3-8% CPU (higher hash cost, packet recirculation).

## Điểm cốt lõi cần nhớ

Thứ nhất, OVS bonding độc lập với Linux kernel bonding — hai stack cùng tồn tại trên một host, không conflict.

Thứ hai, ba bond mode phục vụ ba scenario: `active-backup` (simple, không cần peer support), `balance-slb` (load share per source MAC, không cần MLAG), `balance-tcp` (best băng thông thực tế, cần LACP + MLAG peer).

Thứ ba, `lacp-fallback-ab=true` là foot-gun-saver bắt buộc cho production — tránh bond down hoàn toàn khi peer LACP misconfigured.

Thứ tư, "letter salad `ACEGIKNP` trên cả hai phía" là quick check healthy LACP. Thiếu letter nào → debug theo letter đó.

Thứ năm, Anatomy Template A `bond/show` 10-attribute là first-line debug. Red flag bond_mode mismatch, lacp_status configured persistent, may_enable false, hash single-dominate.

Thứ sáu, balance-tcp trên non-MLAG peer cause packet reorder — cause # 1 của production bond issue. Verify peer MLAG trước khi chọn balance-tcp.

### 9.6.10.1 `ovs-appctl bond/show [port]` — chụp ảnh trạng thái bond

**Khái niệm.** Lệnh in toàn bộ trạng thái runtime của một bond port (hoặc tất cả bond nếu bỏ trống tham số): mode, recirculation flag, updelay/downdelay, countdown rebalance, LACP status, fallback-ab flag, active member, và per-member breakdown gồm `enabled`, `may_enable`, hash bucket → tải. Đây là Anatomy Template A đã trình bày §9.6.6 ở trên.

**Lịch sử + bối cảnh.** Lệnh có mặt từ những phiên bản OVS đầu (handler `bond_unixctl_show` trong `ofproto/bond.c`). Cùng với `bond/list`, đây là cặp đôi mà mọi operator học OVS đều gặp trong tuần đầu.

**Vị trí trong kiến trúc.** `bond/show` là unixctl command đăng ký bởi `bond_init()` (`ofproto/bond.c` dòng 1826), chạy trong process `ovs-vswitchd` qua control socket `/var/run/openvswitch/ovs-vswitchd.<pid>.ctl`. Snapshot read-only, có lock `ovs_rwlock_rdlock(&rwlock)` để không cản đường data path.

**Vai trò + vì sao tồn tại.** Bond state là cấu trúc in-memory của ofproto, không export ra OVSDB column. Nếu thiếu `bond/show`, operator không có cách nào quan sát hash distribution + active member trong runtime. Lệnh sinh ra để bù gap quan sát này.

**Cú pháp + sample output:**

```bash
[host]# ovs-appctl bond/show              # tất cả bond
[host]# ovs-appctl bond/show bond0        # 1 bond cụ thể
```

Output cho `bond/show bond0` đã được đầy đủ ở §9.6.6 (Anatomy Template A 10-attribute). Field cần đọc trước: `bond_mode`, `lacp_status`, `lacp_fallback_ab`, `active member mac`, per-member `enabled` + `may_enable`.

**Cách quan sát + xác minh.** Sau khi tạo bond + chờ updelay 100ms, chạy `bond/show bond0` xem `lacp_status` đã chuyển từ `configured` sang `negotiated` chưa. Nếu vẫn `configured` sau 5-10s, peer switch có thể chưa enable LACP — sang section `lacp/show` debug tiếp.

**Khi xảy ra sự cố.** Triệu chứng `lacp_status: configured` persistent + `may_enable: false` trên member: peer chưa respond LACPDU. `bond/show` chỉ phơi bày triệu chứng; root cause cần `lacp/show` partner state.

**Source code.** `ofproto/bond.c` `bond_unixctl_show()` dòng 1610-1638 (OVS branch-2.17). Handler chỉ cầm read lock + gọi `bond_print_details()` trên 1 bond hoặc loop toàn bộ `all_bonds`.

### 9.6.10.2 `ovs-appctl bond/list` — bird-eye view tất cả bond

**Khái niệm.** Lệnh in 1 dòng/bond ở format TSV: tên bond, bond_mode, recirc_id, danh sách member name. Không có trạng thái runtime chi tiết — đó là việc của `bond/show`.

**Vai trò.** Khi host có nhiều bridge × nhiều bond (gateway chassis 4-6 bond không hiếm), `bond/list` cho operator danh mục rất nhanh trước khi zoom vào 1 bond cụ thể bằng `bond/show <name>`. Tương tự pattern `ovs-vsctl list-br` rồi `ovs-vsctl show <br>`.

**Cú pháp + sample output:**

```bash
[host]# ovs-appctl bond/list
bond	type	recircID	members
bond0	balance-slb	-1	eth1, eth2
bond1	balance-tcp	7	eth3, eth4
bond-mgmt	active-backup	-1	eth5, eth6
```

Header `bond\ttype\trecircID\tmembers` là 4 cột tab-separated, dễ pipe vào `awk` hoặc `column -t`.

**Cơ chế hoạt động.** Handler `bond_unixctl_list()` (`ofproto/bond.c` dòng 1470-1499) loop `HMAP_FOR_EACH (bond, ...)` toàn `all_bonds` → in tên + `bond_mode_to_string(bond->balance)` + `bond->recirc_id` + danh sách member. Read-only, lock `rwlock` rdlock, side-effect zero.

**Cột `recircID` đọc thế nào.** `-1` = không dùng recirculation (`active-backup` hoặc `balance-slb`). Số dương (ví dụ 7) = bond đó dùng `balance-tcp` cần recirc với recirc_id 7 cho L4 hash. Cột này là chỉ báo nhanh xem bond nào yêu cầu MLAG peer.

**Cách quan sát + xác minh.** Sau `ovs-vsctl add-bond`, chạy `bond/list` confirm bond mới xuất hiện. Nếu không thấy: `ovs-vsctl` đã commit OVSDB nhưng `ovs-vswitchd` chưa pick up (rare, thường do bridge_reconfigure failure — check `journalctl -u ovs-vswitchd`).

**Source code.** `ofproto/bond.c:1825` register; handler `bond_unixctl_list` dòng 1470 (OVS branch-2.17).

### 9.6.10.3 `ovs-appctl bond/migrate port hash member` — di chuyển hash bucket SLB thủ công

**Khái niệm.** Lệnh **chỉ valid cho bond mode `balance-slb`**, force migrate 1 hash bucket cụ thể (số 0-255) sang member chỉ định. Là escape hatch khi operator biết rõ flow nào bị pin sai và muốn override automatic rebalance.

**Lưu ý naming.** Tên thật là `bond/migrate` (3 argument: `port hash member`), KHÔNG phải `bond/migrate-slave`. Có alias `bond/set-active-slave` (deprecated) cho lệnh khác là `bond/set-active-member`. Đừng nhầm 2 lệnh: `bond/migrate` di chuyển 1 hash bucket trong SLB; `bond/set-active-member` chuyển active member trong active-backup.

**Cú pháp + sample output:**

```bash
[host]# ovs-appctl bond/show bond0 | grep "hash 42"
  hash 42: 18 kB load                    # đang ở eth1
[host]# ovs-appctl bond/migrate bond0 42 eth2
migrated
[host]# ovs-appctl bond/show bond0 | grep "hash 42"
member eth2:
  hash 42: 18 kB load                    # đã chuyển sang eth2
```

**Cơ chế hoạt động.** Handler `bond_unixctl_migrate()` (`ofproto/bond.c` dòng 1641-1693): kiểm tra `bond->balance != BM_SLB` → reject với "not an SLB bond"; parse hash 0-255 (`hash & BOND_MASK`); tìm member; check `member->enabled` (cannot migrate to disabled); set `entry->member = member` + flag `bond->bond_revalidate = true`. Reply `migrated`.

**Khi nào dùng + khi nào tránh.** Dùng khi: (1) elephant flow của 1 source MAC pin trên member quá tải; (2) cần test failover nhân tạo cho 1 bucket. Tránh khi: (1) bond đang `balance-tcp` (sẽ reject); (2) operator không biết hash bucket nào ↔ flow nào (phải `bond/hash <mac>` lookup trước).

**Foot-gun.** Migration không bền vững — chỉ tới chu kỳ rebalance kế (mặc định 10s). Sau đó hash sẽ bị shift lại theo thuật toán SLB. Để bền vững: tăng `other_config:bond-rebalance-interval=300000` (5 phút) hoặc chuyển sang `balance-tcp` với MLAG.

**Source code.** `ofproto/bond.c:1828` register `unixctl_command_register("bond/migrate", "port hash member", 3, 3, ...)`; handler dòng 1641 (OVS branch-2.17).

### 9.6.10.4 `ovs-appctl lacp/show [port]` — Actor + Partner state debug

**Khái niệm.** Lệnh in chi tiết LACP cho 1 bond port (hoặc tất cả nếu bỏ trống): system_id Actor + Partner, aggregation key, port_id, port_priority, và **state bit field** ASCII (letter salad `ACEGIKNP`). Đây là lệnh debug duy nhất cho LACP negotiation từ phía OVS host.

**Lưu ý naming.** Lệnh bao quát "tất cả bond" thực hiện bằng `lacp/show` không tham số, KHÔNG có lệnh riêng `lacp/show-all`. Source `lib/lacp.c` dòng 228 chỉ register đúng `lacp/show` ARGS `[port]` (0-1 argument); handler `lacp_unixctl_show()` (dòng 1116) tự `LIST_FOR_EACH (lacp, node, all_lacps)` khi `argc == 1`.

**Cú pháp + sample output:**

```bash
[host]# ovs-appctl lacp/show bond0
---- bond0 ----
	status: active negotiated
	sys_id: 00:15:5d:01:a3:7f
	sys_priority: 65534
	aggregation key: 13
	lacp_time: fast

slave: eth1: current attached
	port_id: 2
	port_priority: 65535
	may_enable: true

	actor sys_id: 00:15:5d:01:a3:7f
	actor sys_priority: 65534
	actor port_id: 2
	actor port_priority: 65535
	actor key: 13
	actor state: activity timeout aggregation synchronized collecting distributing

	partner sys_id: 00:24:e8:7c:1f:42
	partner sys_priority: 32768
	partner port_id: 18
	partner port_priority: 32768
	partner key: 17
	partner state: activity timeout aggregation synchronized collecting distributing
```

**State bit field 8 letter (`ACEGIKNP`).** Ánh xạ với 8 bit theo IEEE 802.1AX §6.4.2.3: `A`ctivity, ti`m`eout, aggre`g`ation, `s`ynchronized, `c`ollecting, `d`istributing, `d`efaulted, e`x`pired. OVS in dạng từ đầy đủ thay vì letter — ví dụ `synchronized collecting distributing` = bond đã bundled. Letter salad `ACEGIKNP` là biểu diễn cũ trong tài liệu compass.

**Healthy state.** Bond bundled khi cả Actor lẫn Partner cùng có `synchronized collecting distributing`. Nếu một bên thiếu `synchronized` = key/system_id mismatch. Thiếu `collecting`/`distributing` = LACP đồng ý sync nhưng chưa cho traffic chảy (giai đoạn transition rất ngắn, persistent → bug).

**Khi xảy ra sự cố.** Partner sys_id `ff:ff:ff:ff:ff:ff` hoặc `00:00:...` = chưa nhận LACPDU từ peer (cable, peer chưa enable LACP, VLAN trunk chặn LACP frame). `actor key ≠ partner key` không bao giờ tự fix — config peer side để khớp aggregation key.

**Source code.** `lib/lacp.c:228` register; handler `lacp_unixctl_show()` dòng 1116-1140 (OVS branch-2.17).

### 9.6.10.5 `ovs-appctl lacp/show-stats [port]` — counter PDU + carrier

**Khái niệm.** Lệnh in counter LACP per-bond + per-member: tổng PDU TX/RX, số bad PDU nhận được, số lần member state expired/defaulted, số lần carrier flap. Đây là lệnh phát hiện LACP **flap** (chứ `lacp/show` chỉ snapshot 1 thời điểm).

**Cú pháp + sample output:**

```bash
[host]# ovs-appctl lacp/show-stats bond0
---- bond0 statistics ----

slave: eth1:
  TX PDUs: 4729
  RX PDUs: 4731
  RX Bad PDUs: 0
  RX Marker Request PDUs: 0
  Link Expired: 1
  Link Defaulted: 0
  Carrier Status Changed: 2

slave: eth2:
  TX PDUs: 4729
  RX PDUs: 4730
  RX Bad PDUs: 0
  RX Marker Request PDUs: 0
  Link Expired: 0
  Link Defaulted: 0
  Carrier Status Changed: 0
```

**Đọc counter.** `TX PDUs ≈ RX PDUs` = link 2 chiều khoẻ. `RX Bad PDUs > 0` = peer gửi PDU malformed (rare, thường do firmware bug peer switch). `Link Expired > 0` = đã có lúc peer ngừng gửi LACPDU > 3× lacp_time → mất sync. `Link Defaulted` = chưa bao giờ nhận PDU nào, fallback default state. `Carrier Status Changed` = số lần physical link flap (không phải LACP-level), kết hợp với `dmesg` xem có thông báo NIC reset.

**Cách dùng diagnostic.** Sample 2 lần cách nhau 60s, so sánh: nếu `TX PDUs` tăng đều (60 với fast-timer = 60 PDU/phút/member) còn `RX PDUs` tăng chậm → peer rớt PDU một phần. Nếu `Link Expired` tăng → bond flap. Counter chỉ reset khi `ovs-vswitchd` restart, nên giá trị tuyệt đối phụ thuộc uptime.

**Khác biệt với `lacp/show`.** `lacp/show` cho biết **trạng thái hiện tại** (Actor/Partner state). `lacp/show-stats` cho biết **lịch sử tích luỹ** (đã flap bao nhiêu lần kể từ vswitchd start). Cặp đôi này tương tự `ip -s link` vs `ip link`: state vs counter.

**Source code.** `lib/lacp.c:230` register `unixctl_command_register("lacp/show-stats", "[port]", 0, 1, lacp_unixctl_show_stats, NULL)`; handler `lacp_unixctl_show_stats()` dòng 1144-1170 (OVS branch-2.17).

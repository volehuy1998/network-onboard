## 9.8.1 Ba giao thức flow export

**sFlow** (sFlow.org 2001): sampling-based. Switch chọn mẫu ngẫu nhiên 1-in-N packet, gửi header packet (mặc định 128 bytes) kèm metadata đến collector. Counter polling định kỳ. UDP port 6343.

**NetFlow** (Cisco 1996, IETF v5/v9): per-flow record. Switch duy trì flow cache, emit record khi flow expire. Packet rate thấp hơn sFlow nhưng record chi tiết hơn. UDP port 2055 or custom.

**IPFIX** (RFC 7011, 2013): NetFlow v10 standardized. Template-based — switch gửi template định nghĩa record format, sau đó gửi data records tham chiếu template. UDP port 4739 default.

## 9.8.2 Cấu hình sFlow

Compass Ch H idiom:

```bash
[host]# ovs-vsctl -- --id=@s create sFlow agent=eth1 \
         target=\"10.0.0.1:6343\" \
         header=128 sampling=64 polling=10 \
         -- set Bridge br0 sflow=@s
```

- `agent=eth1`: source IP lấy từ interface eth1.
- `target`: collector endpoint.
- `header=128`: giữ 128 byte đầu packet.
- `sampling=64`: sample 1 trong 64 packet.
- `polling=10`: emit counter mỗi 10 giây.

Kiểm chứng: `tcpdump -ni ethX udp port 6343` trên collector host.

Rotate collector runtime: `ovs-vsctl set sFlow <uuid> target='"new:6343"'` — atomic, không mất packet.

## 9.8.3 Cấu hình NetFlow

```bash
[host]# ovs-vsctl -- set Bridge br0 netflow=@nf \
         -- --id=@nf create NetFlow \
            targets=\"192.168.0.34:5566\" active-timeout=30
```

- `active-timeout=30`: emit record mỗi 30s cho long-lived flow.
- `engine_id`/`engine_type` auto-generate, có thể override.

---

## 9.8.4 Cấu hình IPFIX

```bash
[host]# ovs-vsctl -- set Bridge br0 ipfix=@i \
         -- --id=@i create IPFIX targets=\"192.168.0.34:4739\" \
                            obs_domain_id=123 obs_point_id=456 \
                            cache_active_timeout=60 cache_max_flows=13
```

- `obs_domain_id`/`obs_point_id`: xác định nguồn gốc observation (quan trọng cho multi-switch export).
- `cache_max_flows`: giới hạn flow trong cache internal.

## 9.8.7 Anatomy Template A — `ovs-vsctl list sFlow`

**Bối cảnh khi chạy:** Verify sFlow config active + sampling rate applied. Cũng là debug entry khi collector không nhận data.

**Command và output** *(reproduced OVS 2.17.9)*:

```shell
$ sudo ovs-vsctl list sFlow
_uuid               : 7e3c9f82-1d5e-4b7a-8e9f-abcdef012345
agent               : "eth1"
external_ids        : {purpose="capacity-planning"}
header              : 128
polling             : 10
sampling            : 64
targets             : ["10.0.0.1:6343", "10.0.0.2:6343"]
```

**9-attribute anatomy:**

| Field | Giá trị mẫu | Ý nghĩa | Red flag nếu |
|-------|-------------|---------|--------------|
| `agent` | `"eth1"` | Interface cung cấp source IP | Interface không tồn tại = sFlow disabled silently |
| `header` | `128` bytes | Packet header truncation | <64 = header không đủ parse L4, >256 = waste bandwidth |
| `sampling` | `64` (1:64 ratio) | Packet sample rate | 1:1 = debug only, production 1:1024+ |
| `polling` | `10` giây | Counter emit interval | Quá thấp (1s) = collector overwhelm |
| `targets` | `["IP:port"]` | Collector endpoint list | Multiple target = redundancy (good), duplicate data (OK cho sFlow) |
| `external_ids` | metadata | Operator tag | Missing = audit hard |
| Bridge `sflow` column | `[<uuid>]` | Bridge attach sFlow | Empty = sFlow config tồn tại nhưng không active |
| Packet rate trên target | ~N × hz | Verify actual sampling | < expected = sampling rate higher (less samples) |
| Counter emission rate | 1 pkt / polling seconds | Counter packet | Missing = counter disabled hoặc collector firewall |

**Kịch bản bẻ gãy:**

- **sFlow config exist nhưng collector không receive**: Bridge `sflow` column không attach. Verify: `ovs-vsctl get Bridge br0 sflow` — nếu empty, `ovs-vsctl set Bridge br0 sflow=<uuid>`.
- **Collector receive nhưng data incomplete**: `header=64` quá ngắn cho VXLAN+IPv6+TCP stack. Increase to 256.
- **Sampling rate drift**: Collector thấy ít sample hơn expected. OVS adaptive sampling adjusts nếu flow rate high. Check với `ovs-appctl sflow/show`.
- **Multiple target cause duplicate issue**: Một số collector dedup theo flow ID; một số không. Verify collector behavior trước khi setup redundancy.

## 9.8.8 Hiểu sai + Key Topic

> **Hiểu sai phổ biến 1:** *"sFlow và NetFlow cùng nhau = best of both worlds."*
>
> **Thực tế:** Enable cả 2 giao thức cost double CPU (2 sample path + 2 export path). Cả 2 giao thức overlap capability 80%. Chọn 1: sFlow cho operator không có historical commitment, NetFlow/IPFIX cho team có existing Cisco infrastructure. Dual-stack chỉ justify khi migration period giữa collector.

> **Hiểu sai phổ biến 2:** *"Sampling rate 1:1024 loss nhiều data thì inaccurate."*
>
> **Thực tế:** sFlow statistical accuracy tốt ở long time scale. 1:1024 trên 1 Gbps = ~1K sample/giây = đủ cho 5-minute flow aggregation với confidence interval < 5%. Short-term burst (sub-second) mới bị lose visibility. Cho long-term capacity planning + anomaly detection, 1:1024 là sweet spot CPU vs accuracy. Cho incident forensic chính xác, dùng Mirror (Part 9.7).

> **Key Topic:** Flow monitoring là **passive observability** — không thay đổi traffic path, chỉ gửi metadata ra collector. Complement với active monitoring (BFD, ping) + pipeline debug (ovs-appctl ofproto/trace). 3-tier observability cho OVS: (1) Control plane = ofproto/trace + ovsdb-client monitor. (2) Data plane = Mirror (Part 9.7) + flow monitoring. (3) Performance = coverage/show (Part 9.11) + sFlow counter.

## 9.8.9 Capacity planning + tuning

**Sampling rate vs CPU overhead:**

| Sampling rate | CPU overhead (2× 10G) | Sample rate to collector | Use case |
|---------------|------------------------|--------------------------|----------|
| 1:1 | 50%+ CPU | 1M+ sample/s | DEBUG ONLY |
| 1:64 | 3-5% | 15K sample/s | Dense monitoring |
| 1:1024 | <1% | 1K sample/s | Production standard |
| 1:8192 | <0.2% | 120 sample/s | Very high-rate fabric |

**Collector scaling:**

- **sflowtool + single host**: 10K sample/s OK (disk I/O bottleneck ở 50K+).
- **ntopng**: ~50K sample/s với SSD.
- **goflow2 + Kafka + ClickHouse**: 1M+ sample/s sustained.

**Retention math:** 1K sample/s × 24h = 86M records/day. Mỗi record ~200 bytes → ~17 GB/day/collector (parquet compressed 30% → ~5 GB/day). 30-day retention = 150 GB.

## Điểm cốt lõi cần nhớ

Thứ nhất, ba giao thức phục vụ ba use case: sFlow cho high-volume traffic analysis (sampling), NetFlow/IPFIX cho accurate flow accounting.

Thứ hai, atomic transaction `-- --id=@` là idiom bắt buộc — đảm bảo reference đúng giữa Bridge và collector config.

Thứ ba, rotate collector atomic — OVS switch stream atomic sang target mới, không mất packet.

Thứ tư, feed sFlow vào goflow2/ntopng; NetFlow/IPFIX vào nfdump/Kentik — tổng hợp traffic analytics.

Thứ năm, Anatomy Template A `list sFlow` 9-attribute. Red flag: Bridge sflow column empty dù config exist.

Thứ sáu, dual sFlow+NetFlow cost 2×. Chọn 1 per use case. 1:1024 sampling là production sweet spot.

Thứ bảy, flow monitoring là passive observability — complement Mirror (data plane) + ofproto/trace (control plane). 3-tier framework.

## 0.2.1 Topology reference

Tất cả các bước trong file này giả định topology chuẩn OVN môi trường production như mô tả trong **ovn-architecture(7)** §"Architecture":

```
┌──────────────────────────────────────────────────────────────────────┐
│                      Cluster OVN                                     │
│                                                                      │
│  ┌─────────────────────────┐        ┌─────────────────────────┐      │
│  │   Chassis X (hypervisor)│        │   Chassis Y (hypervisor)│      │
│  │                         │        │                         │      │
│  │  pod A (10.0.0.5)       │        │       pod B (10.0.0.10) │      │
│  │     │                   │        │            │            │      │
│  │  veth-podA              │        │         veth-podB       │      │
│  │     │                   │        │            │            │      │
│  │  [br-int OVS]           │        │     [br-int OVS]        │      │
│  │     │                   │        │            │            │      │
│  │  [Geneve tunnel port] ──┼────────┼──► [Geneve tunnel port] │      │
│  │     │                   │        │            │            │      │
│  │  [eth0 NIC]             │        │      [eth0 NIC]         │      │
│  └───────┬─────────────────┘        └────────────┬────────────┘      │
│          │                                       │                   │
│          └───── Physical network (underlay) ─────┘                   │
│                                                                      │
│  OVN Central (nbdb/sbdb + northd, 3-node Raft):                      │
│     └─ Logical Switch "ls1" với port "podA" và "podB"                │
│     └─ 24 ingress + 10 egress bảng Logical Flow (OVN 22.03)          │
└──────────────────────────────────────────────────────────────────────┘
```

Pod A gửi packet TCP tới pod B. Packet này sẽ đi qua 12 giai đoạn chi tiết dưới đây.

## 0.2.2 Giai đoạn 1 — Pod A: application gửi packet

Application trong pod A gọi syscall `send()` tới TCP socket:
- Kernel TCP stack của pod A (network namespace riêng) xây TCP segment.
- Thêm IP header: src=10.0.0.5, dst=10.0.0.10.
- Thêm Ethernet header: src_mac=MAC_podA, dst_mac=MAC_podB (đã resolve qua ARP trước; ARP resolution thông qua OVN ARP responder: xem Part 18.0).
- Packet được đẩy xuống veth-podA (network interface trong namespace pod A).

→ Deep dive: **Part 8.1. Linux bridge, veth, macvlan** giải thích cơ chế veth pair + network namespace.

**Công cụ debug:** `ip netns exec podA tcpdump -i veth-podA -nn host 10.0.0.10` để bắt packet từ pod A.

## 0.2.3 Giai đoạn 2 — veth-podA đẩy packet vào br-int

veth pair có hai đầu: đầu trong namespace pod A (`veth-podA`) và đầu ngoài (gắn vào `br-int` qua OVS port). Khi packet đi ra từ `veth-podA`:
- Kernel chuyển packet sang đầu kia của veth pair: một port của br-int.
- `ovs-vswitchd` kernel datapath nhận packet ở OVS port tương ứng.

→ Deep dive: **Part 9.1. OVS 3-component architecture** mô tả ovs-vswitchd + ovsdb-server + kernel datapath. **Part 13.4, br-int architecture** giải thích br-int integration bridge.

**Công cụ debug:** `ovs-ofctl dump-ports br-int` để xem ports, `ovs-ofctl show br-int` để xem trạng thái từng port.

## 0.2.4 Giai đoạn 3 — Kernel datapath tra cứu megaflow cache

Kernel OVS datapath nhận packet. Bước đầu tiên: tra cứu megaflow cache (theo thiết kế trong NSDI 2015 paper của Pfaff et al., section "Packet Processing in the Datapath").

**Hit case (99% traffic môi trường production):** packet match một megaflow entry. Kernel áp danh sách action của megaflow (ví dụ: modify TTL, push Geneve header, output tunnel port). Packet được forward ngay, không upcall userspace.

**Miss case (packet đầu tiên của flow):** không match megaflow. Kernel đóng gói packet + metadata vào "upcall message" gửi lên userspace ovs-vswitchd.

→ Deep dive: **Part 9.2. OVS kernel datapath + megaflow** mô tả chi tiết cấu trúc megaflow, thuật toán lookup, cache eviction.

**Công cụ debug:** `ovs-appctl dpif/dump-flows br-int` xem megaflow hiện có. `ovs-appctl upcall/show` xem tần suất miss.

## 0.2.5 Giai đoạn 4 — ovs-vswitchd tra cứu classifier (miss case)

Khi upcall, ovs-vswitchd nhận packet + metadata. ovs-vswitchd gọi classifier để tra cứu OpenFlow flow table của br-int:

1. Classifier dùng **tuple space search** (Srinivasan et al., SIGCOMM 1999), duyệt qua các subtable theo pattern mask, áp hash lookup từng subtable.
2. Trả về rule có priority cao nhất match packet.
3. OpenFlow pipeline chạy action của rule, có thể resubmit tới bảng khác, set reg, modify header, v.v.
4. Kết quả cuối: packet được compile thành megaflow (subset bit match) + cài vào kernel datapath cho lần sau.

Với OVN, flow table br-int được ovn-controller cài, mỗi OpenFlow rule là translation của một Logical Flow (từ OVN SB DB).

→ Deep dive: **Part 9.15, ofproto classifier + tuple space search** cho thuật toán chi tiết. **Part 13.7, ovn-controller internals** cho translation Logical Flow → OpenFlow. **Part 4.7. OpenFlow programming với OVS** cho lệnh ovs-ofctl.

**Công cụ debug:** `ovs-appctl ofproto/trace br-int in_port=X tcp,nw_src=10.0.0.5,nw_dst=10.0.0.10,...` simulate packet đi qua toàn bộ OpenFlow pipeline, hiển thị từng bảng match và action.

## 0.2.6 Giai đoạn 5 — OVN logical pipeline: ingress bảng của Logical Switch

OpenFlow table 0 (classifier) match packet và resubmit sang table 8, bắt đầu logical switch ingress pipeline. Packet đi qua 24 bảng logical (ingress pipeline của Logical Switch OVN 22.03 theo ovn-architecture(7) §"Logical Switch Datapath"):
- table 8: `ls_in_port_sec_l2`, port security L2 check.
- table 9-10: port security IP, ND.
- table 11: lookup FDB (MAC learning).
- table 13: pre-ACL (stateful ACL check).
- table 14: pre-LB (Load Balancer DNAT).
- table 16: ACL rules.
- table 18-19: DHCP options + response.
- table 22: `ls_in_l2_lkup` — L2 destination lookup (dst_mac → outport).
- table 23: `ls_in_l2_unknown`, flood unknown MAC.

Packet gửi từ pod A tới pod B: ở `ls_in_l2_lkup`, OVN match eth.dst = MAC_podB → set outport = "port_podB" → resubmit sang egress pipeline.

→ Deep dive: **Part 13.2 — Logical switches và routers** cho pipeline chi tiết. **Part 13.8, ovn-northd translation** cho cách pipeline được sinh từ NB intent.

**Công cụ debug:** `ovn-trace ls1 'inport == "podA" && eth.src == MAC_podA && eth.dst == MAC_podB && ip4.src == 10.0.0.5 && ip4.dst == 10.0.0.10 && ip.ttl == 64'`, simulate packet qua logical pipeline (theo ovn-trace(8) man page).

## 0.2.7 Giai đoạn 6 — Egress pipeline + xác định physical output

Egress pipeline (tables 32-55) xử lý:
- Egress ACL.
- QoS mark + meter.
- Egress port security.
- **table 49**: `ls_out_port_sec_l2`, output to destination port.

Bước cuối quan trọng: outport (`port_podB`) được resolve thành physical output. Nếu:
- `port_podB` bind với chassis hiện tại (X) → output trực tiếp ra veth-podB.
- `port_podB` bind với chassis khác (Y) → output ra **Geneve tunnel port** tới chassis Y.

Trường hợp này: pod B trên chassis Y → packet phải đi tunnel.

→ Deep dive: **Part 13.5. Port binding types** cho chassisredirect, tunnel logic. **Part 9.1. OVS 3-component** cho tunnel port trong br-int.

## 0.2.8 Giai đoạn 7 — Geneve encapsulation

OVS áp action `output:<geneve_tunnel_port>`. Action này trigger encapsulation theo định dạng Geneve (RFC 8926):
- Inner packet (gốc từ pod A) được wrap thành payload.
- OVS push Geneve header: Protocol: UDP, UDP dst port 6081 (IANA assignment RFC 8926 §3.1).
- Geneve header chứa:
  - **VNI (24-bit):** tunnel_key của Logical Switch ls1 (RFC 8926 §3.4).
  - **TLV options (variable-length):** OVN-specific metadata (inport logical, outport logical, ct_state snapshot) theo ovn-architecture(7) §"Tunnel Encapsulations".
- Push outer IP header: src=IP của chassis X tunnel endpoint, dst=IP của chassis Y tunnel endpoint.
- Push outer Ethernet header với dst=MAC gateway router (nếu cross subnet) hoặc chassis Y direct (nếu same L2).

→ Deep dive: **Part 11.0 — VXLAN/Geneve/STT** cho Geneve format + lý do OVN chọn Geneve thay vì VXLAN. **Part 11.1 — MTU + PMTUD** cho MTU math (packet gốc 1500B + 50-80B overhead Geneve = 1550-1580B outer, cần MTU underlay >=1600 để tránh fragmentation).

**Công cụ debug:** `tcpdump -i eth0 -nn udp port 6081` trên chassis X để bắt outer packet có Geneve header.

## 0.2.9 Giai đoạn 8 — Underlay transit

Outer packet đi ra physical NIC của chassis X → network underlay (thường là Clos leaf-spine topology trong data center). Underlay route packet tới chassis Y theo destination IP của outer header.

Underlay có thể là:
- Simple L2 flat network (lab nhỏ).
- Clos leaf-spine với BGP/OSPF routing (môi trường production data center).
- Overlay-over-overlay (Geneve-over-MPLS cho WAN cross-site).

→ Deep dive: **Part 12.0 — DC network topologies Clos leaf-spine** cho kiến trúc underlay. **Part 11.2 — BGP EVPN** cho overlay control plane thay thế OVN trong một số kiến trúc.

**Công cụ debug:** traceroute trên chassis X đến IP tunnel của chassis Y, mtr cho theo dõi packet loss.

## 0.2.10 Giai đoạn 9 — Chassis Y nhận outer packet

NIC chassis Y nhận outer packet (UDP port 6081):
- Kernel linux networking stack route packet tới local endpoint (tunnel IP của chassis Y).
- Kernel nhận ra đây là packet Geneve cần decap (kernel 4.14+ có native Geneve support qua module `geneve`).
- Packet được forward tới OVS kernel datapath qua tunnel port (socket listen 6081).

→ Deep dive: **Part 8.3, tc qdisc + conntrack** cho tunnel packet path trong kernel.

## 0.2.11 Giai đoạn 10 — Geneve decapsulation + OVN ingress remote

OVS decap Geneve header → khôi phục inner packet. Trường VNI của Geneve → identify Logical Switch (tunnel_key → ls1). TLV options → khôi phục logical inport + outport + ct_state (theo ovn-architecture(7) §"Tunnel Encapsulations").

Packet giờ xuất hiện trong br-int của chassis Y với metadata:
- `metadata = tunnel_key ls1`.
- `reg14 = inport logical (port_podA)`, để ngữ cảnh ACL không drop traffic từ port A.
- `reg15 = outport logical (port_podB)`.

OVN resubmit packet sang table 32 (egress pipeline) bắt đầu từ egress ACL. Egress pipeline hoàn tất, bảng cuối output ra veth-podB.

→ Deep dive: **Part 11.0 — Geneve decap logic** và **Part 13.7, ovn-controller egress pipeline translation**.

## 0.2.12 Giai đoạn 11 — veth-podB đẩy packet vào pod B

OVS output ra OVS port ánh xạ veth-podB. veth đẩy packet vào namespace pod B. Kernel pod B:
- Nhận Ethernet packet qua veth-podB.
- IP stack route packet dựa trên dst = 10.0.0.10 (pod B's IP).
- TCP stack match socket của application đang listen.
- Application nhận packet qua syscall `recv()`.

→ Deep dive: **Part 8.0. Linux namespaces** cho network isolation, **Part 8.1, veth pair** cho kernel internals.

## 0.2.13 Giai đoạn 12 — Reply traffic đi ngược

Application pod B gửi reply, cùng chuỗi 11 giai đoạn nhưng **theo chiều ngược lại**. Một số điểm quan trọng (theo OVN ACL stateful semantic, ovn-architecture(7) §"ACLs"):
- Reply packet match entry conntrack đã commit ở forward path → ACL stateful cho phép tự động (không cần rule tường minh cho reply).
- LB SNAT được áp ngược (un-DNAT) nếu pod A gọi tới VIP thay vì pod B trực tiếp.
- Nếu cross subnet qua Logical Router, reply đi qua LR một lần nữa.

→ Deep dive: **Part 13.3 — ACL/LB/NAT/port groups** cho stateful ACL semantics.

## 0.2.15 Công cụ gỡ lỗi tổng hợp cho packet journey

Mỗi lớp trừu tượng có công cụ gỡ lỗi riêng. Biết dùng đúng công cụ = tiết kiệm giờ gỡ lỗi:

**Lớp pod / namespace:**
- `ip netns exec podA tcpdump -i veth-podA`.
- `ip netns exec podA ss -tnp` xem trạng thái TCP socket.

**Lớp kernel datapath OVS:**
- `ovs-appctl dpif/dump-flows br-int`, xem megaflow.
- `ovs-appctl upcall/show`, xem tần suất miss.
- `ip -s link show <tunnel_interface>`, xem counter tunnel.

**Lớp ovs-vswitchd / OpenFlow:**
- `ovs-ofctl dump-flows br-int --no-stats | grep table=X`, xem rule từng bảng.
- `ovs-appctl ofproto/trace br-int <packet_flow>`, simulate packet qua OpenFlow pipeline. **Công cụ mạnh nhất cho gỡ lỗi mức OVS.**

**Lớp OVN logical:**
- `ovn-trace ls1 '<packet_flow>'`, simulate qua logical pipeline. **Công cụ mạnh nhất cho gỡ lỗi mức OVN.**
- `ovn-nbctl show` / `ovn-sbctl show` xem topology.
- `ovn-sbctl lflow-list ls1` xem Logical Flow.

**Lớp underlay:**
- `tcpdump -i eth0 -nn udp port 6081` bắt Geneve outer.
- `mtr <chassis_Y_tunnel_ip>` xem route + packet loss.

**Công cụ cross-cutting:**
- `ovn-detrace`, trace theo flow qua cả logical và physical layer kết hợp.
- Hubble (nếu dùng Cilium CNI): flow observability.

## 0.2.16 Các "đường tắt" (fast paths) không theo đủ 12 giai đoạn

Không phải mọi packet đi hết 12 giai đoạn. Một số tối ưu đường tắt:

**(1) Same-chassis traffic:** pod A và pod B trên cùng chassis, packet không đi Geneve. Giai đoạn 1-6 chạy, rồi output trực tiếp ra veth-podB (bỏ qua giai đoạn 7-10). Độ trễ chỉ khoảng 50-100 microsecond thay vì 200-500 µs cross-chassis.

**(2) Bridged L2 mode:** nếu pod A và pod B cùng Logical Switch + cùng chassis, không qua Logical Router → skip được phần routing (giai đoạn 5 rút gọn).

**(3) Hardware offload (OVS-TC):** với NIC ConnectX-5+ + switchdev mode, kernel offload megaflow lookup + encap/decap xuống ASIC NIC. CPU chassis không thấy packet, giai đoạn 3-4 bị bỏ qua hoàn toàn. Độ trễ giảm xuống <50 µs cross-chassis.

→ Deep dive: **Part 9.5 — HW offload switchdev + ASAP² + DOCA** cho chi tiết hardware offload.

**(4) DPDK userspace:** thay vì kernel datapath, OVS-DPDK dùng PMD thread thăm dò NIC queue → không có kernel IRQ. Hiệu năng 10-100 lần, nhưng giai đoạn 1-2 hơi khác (veth thay bằng virtio vhost-user cho VM).

→ Deep dive: **Part 9.3. OVS userspace DPDK + AF_XDP** cho kiến trúc userspace.

### ▶ Guided Exercise 1: Trace packet journey thực tế với ofproto/trace + ovn-trace

> **Trạng thái verification lab:** Output dựa trên OVS 2.17 + OVN 22.03 docs chính thức; chờ kiểm chứng lab thực.

**Mục đích:** Simulate một packet đi từ pod A đến pod B qua 12 giai đoạn, dùng công cụ mức OVN (ovn-trace) và mức OVS (ofproto/trace).

**Chuẩn bị:** Lab Ubuntu 22.04 với OVS 2.17 + OVN 22.03, 2 chassis (có thể là 2 VM hoặc 2 namespace trên cùng host). Logical Switch ls1 đã tạo với pod A, pod B bind.

**Bước thực hiện:**

**1.** Xác minh topology:
```
[host]# ovn-nbctl show
switch ... (ls1)
    port podA
        addresses: ["aa:bb:cc:dd:ee:01 10.0.0.5"]
    port podB
        addresses: ["aa:bb:cc:dd:ee:02 10.0.0.10"]
```

**2.** Simulate packet qua OVN logical pipeline:
```
[host]# ovn-trace ls1 'inport == "podA" && eth.src == aa:bb:cc:dd:ee:01 && eth.dst == aa:bb:cc:dd:ee:02 && ip4.src == 10.0.0.5 && ip4.dst == 10.0.0.10 && ip.ttl == 64 && tcp.src == 12345 && tcp.dst == 80'
```

Output hiển thị mỗi bảng + rule match + action, từ `ls_in_port_sec_l2` đến `ls_out_port_sec_l2`. Theo dõi reg14 (inport) và reg15 (outport) thay đổi qua pipeline.

**3.** Simulate packet qua OVS physical (chassis X):
```
[host]# OVS_PORT=$(ovs-vsctl --columns=ofport find Interface external_ids:iface-id=podA | awk '{print $3}')
[host]# ovs-appctl ofproto/trace br-int in_port=$OVS_PORT,tcp,nw_src=10.0.0.5,nw_dst=10.0.0.10,dl_src=aa:bb:cc:dd:ee:01,dl_dst=aa:bb:cc:dd:ee:02,tp_src=12345,tp_dst=80
```

Output hiển thị trace qua từng bảng OpenFlow của br-int, chi tiết hơn ovn-trace (bao gồm mức action OpenFlow cụ thể).

**4.** Nếu pod A và pod B trên chassis khác nhau, quan sát tunnel output trong trace, hiển thị action `output:tunnel_port`.

**5.** Bắt packet thực tế trên wire:
```
# Trên chassis X
[chassis-x]# tcpdump -i eth0 -nn -X 'udp port 6081' -c 10

# Từ pod A gửi traffic thực:
[host]# ip netns exec podA ping -c 3 10.0.0.10
```

tcpdump sẽ bắt được Geneve outer packet — Protocol 17 UDP 6081, payload chứa inner IP+TCP của packet gốc.

**Bài học học được:**

Thứ nhất, **ovn-trace** và **ofproto/trace** là cặp công cụ quan trọng nhất cho gỡ lỗi OVN/OVS, cho biết packet đi qua bảng nào, match rule nào, action nào áp. Dùng ngay khi gặp "packet không đến đích".

Thứ hai, packet journey cross-chassis qua Geneve là nguyên nhân 99% độ trễ trong OVN, mọi tối ưu (HW offload, DPDK, same-chassis scheduling) đều nhằm rút ngắn hoặc bypass giai đoạn 7-10.

Thứ ba, hiểu 12 giai đoạn = hiểu từng vấn đề hiệu năng/đúng đắn thuộc lớp nào. "Packet drop" có thể xảy ra ở giai đoạn 4 (flow table miss), 7 (MTU mismatch), 9 (underlay packet loss), hoặc 12 (reply drop do ACL stateful không đúng).

**Cleanup:** None cần (chỉ dùng trace, không modify state).

## Tài liệu tham khảo

1. [OVN Architecture Guide (ovn-architecture(7))](https://man7.org/linux/man-pages/man7/ovn-architecture.7.html), §"Life Cycle of a VIF" và §"Tunnel Encapsulations" mô tả packet journey chính thức từ upstream OVN project.
2. [Open vSwitch Design Documentation](https://docs.openvswitch.org/en/latest/topics/design/), packet processing pipeline từ upstream OVS project.
3. [RFC 8926 — Geneve: Generic Network Virtualization Encapsulation](https://datatracker.ietf.org/doc/html/rfc8926), spec protocol chính thức IETF.
4. Pfaff et al., "The Design and Implementation of Open vSwitch" (NSDI 2015), paper gốc peer-reviewed với packet processing chi tiết ở section 4 "Packet Processing".
5. [ovn-trace(8) man page](https://man7.org/linux/man-pages/man8/ovn-trace.8.html) và [ovs-appctl(8) ofproto/trace](https://man7.org/linux/man-pages/man8/ovs-appctl.8.html), công cụ gỡ lỗi chính thức.

## 16.2.0 Opening drama, from "DPDK or nothing" to "AF_XDP, the kernel 4.18 (2018) compromise"

In 2015 to 2018, network engineering on Linux was stuck between two extreme choices:

- **Kernel stack:** full features (iptables, tc, netfilter, routing, bridging) but real-world bandwidth limited to about 2 Mpps per core.
- **DPDK:** bypass the kernel entirely, reach 15 or more Mpps per core, **but lose all kernel features**, plus require hugepages, IOMMU, dedicated CPUs, and high complexity.

There was no middle ground. Cloudflare, Facebook, and Netflix wanted to filter DDoS at line-rate of 40 Gbps or more on commodity hardware, did not want to pay the DPDK complexity cost, but the kernel was too slow to drop 10 million or more packets per second.

In 2014 to 2016, **Alexei Starovoitov** (Facebook, now Meta) developed **eBPF** (extended BPF), a JIT-compiled virtual machine inside the kernel that lets engineers write C code, verified safe, that runs at near-native performance in a kernel context. In 2016, **Jesper Dangaard Brouer** (Red Hat) and **Daniel Borkmann** (Isovalent) proposed **XDP (eXpress Data Path)**: a hook point for an eBPF program that runs **BEFORE the kernel allocates an sk_buff**, right at the NIC driver. Kernel 4.8 (2016) merged the XDP BPF infrastructure.

In 2018, **Bjorn Topel** and **Magnus Karlsson** (Intel) contributed the **AF_XDP** socket family for kernel 4.18. AF_XDP combines an XDP hook with a shared memory ring (UMEM) so that userspace can receive packets zero-copy, **while keeping the kernel driver and kernel features intact**. This is the **middle ground** between DPDK and the kernel stack: real-world bandwidth around 10 to 30 Mpps per core (80 percent of DPDK), kernel stack preserved for non-redirected packets, easier to deploy than DPDK.

Adoption was rapid. **Cilium** (2018 onward) uses XDP for K8s service load balancing plus network policy enforcement; **Facebook Katran** (2018, open-source) uses XDP_TX for stateless load balancing; **Suricata IDS** (2019 onward) uses AF_XDP for line-rate deep packet inspection; **OVS-AFXDP** (2020) is an alternative for the OVS userspace datapath. By 2024 to 2026, XDP plus AF_XDP became standard for DDoS mitigation, network policy enforcement, and custom load balancing.

Part 16.2 dives into the AF_XDP 4-ring architecture plus the 3 attach modes plus the 4 return codes plus the libbpf and libxdp APIs plus eBPF map types plus production patterns. It ends with two hands-on exercises (minimal XDP_PASS, AF_XDP TCP redirect) plus a Capstone POE: "DDoS mitigation at 200 Gbps, XDP or DPDK?".

> **Common misconception:** *"AF_XDP is just a slower DPDK."*
>
> **Reality:** AF_XDP has 3 advantages that are DIFFERENT from DPDK: (a) it preserves kernel features (iptables, tc, routing) for non-redirected packets, (b) it does not need hugepages, IOMMU, or isolcpus (easier deployment), (c) it shares the NIC with kernel traffic instead of monopolizing it. Real-world bandwidth is about 20 to 30 percent below DPDK for small packets, but the use cases are different. DPDK is for dedicated workloads (NFV, HFT), AF_XDP is for selective bypass (Cilium K8s, DDoS mitigation, Suricata IDS).

## 16.2.1 AF_XDP architecture (4 rings)

An AF_XDP socket implements userspace-to-kernel communication through four shared-memory **rings**, organized as producer-consumer:

**(1) FILL ring**, userspace is the producer: it pushes the frame index (offset within UMEM) of empty buffers onto the ring to tell the kernel "these buffers are ready to receive new packets". The kernel is the consumer. When a packet arrives, the kernel dequeues a frame index from the FILL ring, DMAs the packet from the NIC into the matching buffer, and enqueues that frame index onto the RX ring.

**(2) RX ring**, kernel is the producer, userspace is the consumer. Each entry holds {frame_idx, offset, len}. Userspace polls the RX ring through the `recvmsg()` syscall (regular mode) or through continuous polling (busy-poll mode). After processing a packet, userspace must push the frame index back to the FILL ring for reuse.

**(3) TX ring**, userspace is the producer, kernel is the consumer. When userspace wants to send a packet, it writes data into an empty UMEM buffer, enqueues {frame_idx, offset, len} onto the TX ring, and calls `sendto()` to wake the kernel to process the ring. The kernel dequeues from the TX ring, DMAs from UMEM to the NIC, and enqueues the frame index onto the COMPLETION ring.

**(4) COMPLETION ring**, kernel is the producer, userspace is the consumer. It signals userspace that the buffer has finished transmitting and can be reused. Userspace dequeues from the COMPLETION ring and returns the frame index to the FILL ring if it wants to reuse the buffer for RX.

**UMEM (User Memory)** is a contiguous memory region (usually a few MB to a few GB) allocated through `mmap()` and registered with the kernel through `setsockopt(XDP_UMEM_REG)`. UMEM is divided into fixed-size frames (usually 2048 or 4096 bytes). The kernel DMAs DIRECTLY from the NIC into UMEM when the driver supports zero-copy mode, with no memcpy between a kernel buffer and UMEM.

The four rings are shared memory between userspace and the kernel, synchronized through atomic producer and consumer indices. There is no lock, no context switch, this is why AF_XDP performance is close to DPDK.

## 16.2.2 Driver mode versus SKB mode

**XDP_DRIVER (native mode):** the XDP program runs in the interrupt context of the NIC driver, BEFORE the kernel allocates an sk_buff. This is the highest performance attach point, about 24 Mpps per core on Intel XXV710 and more than 30 Mpps on Mellanox ConnectX-6. It requires driver support for native XDP: Intel `i40e`, `ice`, `ixgbe`, Mellanox `mlx4`, `mlx5`, Broadcom `bnxt`, Netronome `nfp`, virtio `virtio_net`, TAP device `tun`. To check: `ethtool -i eth0 | grep driver` and look up the driver in `kernel/drivers/net/ethernet/*/`.

**XDP_SKB (generic mode):** the XDP program runs AFTER the kernel has allocated an sk_buff. It works with EVERY driver, including drivers that do not have native XDP. But performance drops noticeably, around 3 to 5 Mpps per core. The reason: you have already paid the cost of `build_skb()` (about 100 to 200 ns), but you do not get the zero-copy benefit. This mode is for development, debugging, or workloads on older drivers.

**XDP_HW (hardware offload):** compile the eBPF program for NIC firmware, run it inside the NIC ASIC, with no CPU cost at all. Supported by Netronome Agilio CX (driver nfp) and Mellanox ConnectX-5 or later (with cards that support eBPF offload). There are instruction-set restrictions: some eBPF helpers are not available (for example, slow map lookups), and the program must be small enough to fit in NIC ASIC memory. Suitable for simple use cases (simple DDoS filtering, drop before firewall) where saving CPU is more important than flexibility.

**How to choose a mode:** always try **XDP_DRIVER** first, check driver support through `bpftool net show`. If the driver does not support it, fall back to XDP_SKB (acceptable for traffic below 5 Gbps). XDP_HW is only used when you have a special NIC and the workload is simple.

## 16.2.3 XDP program types and actions

After an XDP program processes a packet, it returns one of four standard action codes:

**XDP_PASS:** allow the packet to continue into the kernel networking stack as usual. Use this when the XDP program only observes (telemetry, counting) and does not interfere. This is the default behavior if the program has no drop or redirect logic.

**XDP_DROP:** drop the packet immediately at the earliest point in the pipeline (before the kernel allocates an sk_buff). The cost is below 100 ns per packet. This is the mechanism for effective DDoS mitigation: when malicious traffic is detected, XDP_DROP can drop tens of millions of packets per second without burning kernel-stack CPU. Cloudflare and AWS Shield use this strategy for 100 Gbps or more DDoS mitigation.

**XDP_REDIRECT:** redirect the packet to another destination. There are two sub-modes: (1) **redirect to an AF_XDP socket** through `bpf_redirect_map(&xsk_map, queue_id, 0)`, which pushes the packet into a userspace UMEM (covered in section 16.2.1). (2) **redirect to another NIC** through `bpf_redirect(ifindex, 0)`, which sends the packet straight out of the egress NIC without going through kernel routing (used for software switches built on XDP, such as ovs-afxdp and Facebook's Katran). XDP_REDIRECT is the most powerful mode and reaches line-rate for many forwarding workloads.

**XDP_TX:** send the packet out of the same interface it arrived on, usually after modifying headers. Used for stateless load balancers (Katran), fast NAT, or ARP and ICMP responders. No routing-table entry is needed, the packet goes straight to the TX queue of the same interface.

XDP programs are written in C (a restricted subset, no malloc, no floating point, no unbounded loops), compiled with clang and the `bpf` target: `clang -O2 -target bpf -c xdp_prog.c -o xdp_prog.o`. The verifier checks the bytecode before loading.

## 16.2.4 libbpf plus libxdp in userspace

**libbpf** is the standard C library for userspace applications interacting with eBPF, open-source at `github.com/libbpf/libbpf`. The main API:

- `bpf_object__open(path)`: load the ELF file containing the compiled eBPF program.
- `bpf_object__load(obj)`: load the program into the kernel through the `bpf(BPF_PROG_LOAD)` syscall.
- `bpf_program__attach_xdp(prog, ifindex)`: attach the XDP program to an interface.
- `bpf_map__find_by_name(obj, name)`: get a handle on an eBPF map for userspace operations.
- `bpf_map_update_elem(map_fd, key, value, flags)` and `bpf_map_lookup_elem()`, update or read the map.

**libxdp** is a higher-level wrapper around libbpf, focused specifically on XDP use cases, open-source at `github.com/xdp-project/xdp-tools`. It provides:

- `xdp_program__open_file(path)`: load an XDP program with metadata (program name, priority for multi-program setups).
- `xdp_program__attach(prog, ifindex, mode)`: attach with an explicit mode (XDP_MODE_NATIVE, XDP_MODE_SKB, XDP_MODE_HW).
- `xdp_multiprog__add_prog()`: support multiple XDP programs on the same interface (introduced in kernel 5.10), in priority order.
- Helpers for AF_XDP sockets: `xsk_socket__create()`, `xsk_ring_prod__reserve()`, and so on.

**Map types shared between an XDP program and userspace:**

- **BPF_MAP_TYPE_PERCPU_ARRAY:** a per-CPU array, each CPU has its own copy, no lock needed. Used for per-CPU packet counters that the XDP program updates and userspace aggregates across CPUs.
- **BPF_MAP_TYPE_HASH:** a generic key-value map with O(1) lookup. Used for policy tables (src IP to allow or deny) and session tables.
- **BPF_MAP_TYPE_LPM_TRIE:** for CIDR routing tables (source subnet to action).
- **BPF_MAP_TYPE_XSKMAP:** maps queue_id to an AF_XDP socket, used for XDP_REDIRECT into userspace.

The standard development flow: write the XDP program (.c) -> compile with clang -> write the userspace loader using libbpf or libxdp -> test with synthetic traffic (pktgen, trex) -> measure with perf and bpftrace.

## 16.2.5 Performance characteristics

**AF_XDP real-world benchmarks** (source: OVS-AFXDP tuning guide Intel 2023, Linux Plumbers Conference 2023 AF_XDP presentations):

- 25GbE Intel XXV710 NIC, 1 core: AF_XDP zero-copy reaches 12 Mpps with 64-byte packets (close to the theoretical line-rate of 37 Mpps).
- 100GbE Mellanox ConnectX-6 NIC, 4 cores: AF_XDP zero-copy reaches 60 Mpps total (close to the theoretical line-rate of 148 Mpps for 64-byte packets).
- Larger packets (1500 bytes): AF_XDP reaches line-rate on most 10 to 100 GbE NICs with only 1 to 2 cores.

**Latency** (packet round-trip on the same node): AF_XDP is around 10 microseconds, compared to about 2 microseconds for DPDK. The difference comes from: (1) AF_XDP still goes through the kernel driver IRQ handler; (2) the packet must traverse the tc hook before reaching XDP (in generic mode). With native mode, the latency drops to about 5 microseconds, closer to DPDK.

**Advantages over DPDK:** (1) **Kernel features are preserved**, netfilter, tc qdisc, and routing tables continue to work for non-redirected traffic. The system still has IP configuration and iptables rules. (2) **No need to detach the NIC** from the kernel driver, which simplifies deployment and removes the risk of losing connectivity during reconfiguration. (3) **No mandatory hugepages**, regular pages also work (a small performance loss). (4) **Easy coexistence with containers**, no conflict with Kubernetes CNI.

**Disadvantages versus DPDK:** (1) **Pure PPS is lower** by about 20 to 30 percent for small packets (64 bytes) because some kernel code paths are still traversed. (2) **A smaller community than DPDK**, fewer examples, fewer pre-built libraries. (3) **Requires kernel 5.10 or later** for the full feature set (XDP metadata, XDP multi-prog), some workloads on older kernels (RHEL 7, Ubuntu 18.04 stock) cannot run.

**Choosing between AF_XDP and DPDK:** AF_XDP for workloads that need medium-high PPS (10 to 60 Mpps) with a simple deployment, such as Cilium, Suricata, OVS-AFXDP. DPDK for workloads with extreme PPS (more than 100 Mpps), 5G UPF, telecom packet gateway, where performance is the deciding factor and the team is ready to invest in a complex deployment.

### Guided Exercise 1: Attach a minimal XDP_PASS program with bpftool

> **Lab verification status:** output is based on the kernel XDP docs plus bpftool reference 2024; pending verification on a real lab.

**Goal:** entry-level XDP lab. Write the simplest possible XDP program (return XDP_PASS for every packet), compile, load, attach, verify. The baseline for any later XDP development.

**Setup:**
- Ubuntu 22.04 VM with kernel 5.15 or later.
- Install the toolchain:
  ```bash
  sudo apt install -y clang llvm libbpf-dev libelf-dev gcc-multilib \
                      linux-headers-$(uname -r) linux-tools-$(uname -r)
  ```
- Driver with native XDP support: e1000e, virtio_net, mlx5_core, i40e, ice all support it. Check with `ethtool -i eth0 | grep driver`.

**Lab topology:**

```
+------------------------------------------------+
|    Ubuntu 22.04 VM                             |
|                                                |
|    Packet inbound ---> NIC (eth0)              |
|                          |                     |
|                          v                     |
|                   +------------------+         |
|                   |  XDP program     |         |
|                   |  xdp_pass_prog() |         |
|                   |  return XDP_PASS |         |
|                   +------+-----------+         |
|                          |                     |
|                          v                     |
|                   Kernel network stack         |
|                   (normal processing)          |
+------------------------------------------------+
```

**Steps:**

**1.** Write the XDP program `xdp_pass.c`:

```c
#include <linux/bpf.h>
#include <bpf/bpf_helpers.h>

SEC("xdp")
int xdp_pass_prog(struct xdp_md *ctx) {
    return XDP_PASS;
}

char _license[] SEC("license") = "GPL";
```

**2.** Compile:

```bash
[host]$ clang -O2 -g -target bpf -c xdp_pass.c -o xdp_pass.o
[host]$ ls -la xdp_pass.o
-rw-r--r-- 1 user user 1024 Apr 22 12:34 xdp_pass.o
```

**3.** Load the program into the kernel:

```bash
[host]# bpftool prog load xdp_pass.o /sys/fs/bpf/xdp_pass
[host]# ls /sys/fs/bpf/
xdp_pass
```

**4.** Get the program ID and attach to the interface:

```bash
[host]# PROG_ID=$(bpftool prog show pinned /sys/fs/bpf/xdp_pass | awk -F: '/^[0-9]/ {print $1}')
[host]# echo $PROG_ID
42
[host]# bpftool net attach xdp id $PROG_ID dev eth0
```

**5.** Verify it is attached:

```bash
[host]# ip link show eth0
```

Expected output:

```
2: eth0: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 xdp/id:42 qdisc ...
    link/ether aa:bb:cc:dd:ee:ff brd ff:ff:ff:ff:ff:ff
    prog/xdp id 42 tag 123abc jited
```

Key indicators: `xdp/id:42` in the ip link output, and `jited` meaning the program was JIT-compiled to native machine code for performance.

**6.** Measure performance impact (sanity check, should be negligible):

```bash
# Baseline (no XDP)
[host]# bpftool net detach xdp dev eth0
[host]# iperf3 -c 10.0.0.1 -t 10

# With XDP_PASS
[host]# bpftool net attach xdp id $PROG_ID dev eth0
[host]# iperf3 -c 10.0.0.1 -t 10
```

Expected outcome: real-world bandwidth deltas within plus or minus 1 percent (XDP_PASS is near-zero overhead since it just falls through to the kernel stack).

**Lessons learned:**

First, **XDP program architecture**: C source -> clang -target bpf -> eBPF bytecode -> kernel verifier -> JIT compile -> native execution. This pipeline ensures safety (the verifier rejects unbounded loops and invalid memory access) and performance (JIT to native instructions).

Second, **XDP_PASS is no-op**: the packet continues to the kernel stack unchanged. Useful for initial development, attach the program without breaking traffic, gradually add logic.

Third, **bpftool versus ip link**: both can attach XDP. The advantage of `bpftool` is that it is cross-platform and uses native BPF concepts. `ip link set dev eth0 xdp obj xdp_pass.o` is the alternative.

**Cleanup:**

```bash
[host]# bpftool net detach xdp dev eth0
[host]# rm /sys/fs/bpf/xdp_pass
```

Second, **AF_XDP zero-copy** preserves kernel features for selected packets. ICMP still reaches the kernel (ping works), TCP is redirected to userspace (custom processing), UDP is dropped silently. Flexibility without a full DPDK bypass.

Third, **production use cases**: DDoS mitigation (XDP drops malicious traffic), custom TCP accelerator (AF_XDP redirect for in-house processing), monitoring tap (XDP mirror to AF_XDP without impact). Cilium plus Katran (Facebook LB) are real-world examples.

## 16.2.6 Compare XDP and AF_XDP with DPDK, OVS, and Cilium production

In the 2024 to 2026 ecosystem, XDP and AF_XDP have surpassed DPDK in adoption for K8s and cloud-native workloads. A comparison table for the four main stacks:

| Dimension | Kernel stack plus iptables | OVS-DPDK userspace | XDP plus AF_XDP | Cilium eBPF (XDP plus tc) |
|-----------|---------------------------|--------------------|-----------------|----------------------------|
| Real-world bandwidth single-core | 1.5 to 2 Mpps | 15 Mpps | 12 Mpps (zero-copy) | 10 to 15 Mpps |
| P99 latency | 20 to 30 us | 2 us | 5 to 10 us | 5 to 8 us |
| Hugepages required | No | Yes (2MB or 1GB) | No (optional) | No |
| IOMMU plus isolcpus required | No | Yes | No | No |
| Kernel features (iptables, tc, routing) | Yes | No (bypass) | Yes (for XDP_PASS) | Yes (replace iptables) |
| Deployment complexity | Low | Very High | Medium | Medium-High |
| K8s CNI integration | Native (kube-proxy iptables) | Not applicable (hypervisor only) | Native (Cilium, KubeArmor) | Production (Cilium) |
| Debug tools | iptables -L, conntrack -L | testpmd, rte_stats | bpftool, bpftrace, xdpdump | cilium debug, hubble |
| Main production use case | Traditional enterprise | NFV VNF, 5G UPF, HFT | DDoS mitigation, IDS (Suricata), custom LB (Katran) | K8s cloud-native, replace kube-proxy plus CNI |

**When to use what:**

- **Kernel plus iptables:** traditional workload below 5 Gbps per node, simple firewall plus NAT.
- **OVS-DPDK:** dedicated NFV workload above 25 Gbps, 5G UPF, telecom.
- **XDP plus AF_XDP:** selective bypass, DDoS mitigation, IDS deep packet inspection, custom LB with XDP_TX.
- **Cilium eBPF:** K8s cloud-native, replace kube-proxy plus Calico iptables, identity-based NetworkPolicy, service mesh data plane.

### Cross-cut with Part 9.3 OVS-AFXDP

Part 9.3 covered how OVS can run with the AF_XDP datapath (`other_config:datapath-type=netdev` plus `type=afxdp` for the port). The relationship:

- **OVS plus kernel datapath** (default, Part 9.2): OVS.ko module, real-world bandwidth about 2 Mpps per core.
- **OVS plus DPDK datapath** (Part 9.3): `datapath-type=netdev` plus `type=dpdk`, real-world bandwidth about 15 Mpps per core.
- **OVS plus AF_XDP datapath** (Part 9.3): `datapath-type=netdev` plus `type=afxdp`, real-world bandwidth about 10 Mpps per core.

OVS-AFXDP combines the OVS OpenFlow pipeline with the AF_XDP zero-copy driver path. Advantages: keeps OVS features (OpenFlow, conntrack through the `ct()` action, OVN integration) plus hybrid performance (no DPDK complexity). Deployed in Red Hat OpenShift Container Platform 4.11 and later as an alternative for latency-sensitive workloads.

### Cross-cut with Part 9.26 (revalidator storm)

Part 9.26 analyzed the OVS kernel datapath stale ukey leak. Does OVS-AFXDP have a similar problem? **Yes, but it is less common** because:

- OVS-AFXDP uses `dpif-netdev` (userspace) instead of `dpif-netlink` (kernel). Stale ukeys can still happen but the revalidator pattern is different.
- Flow eviction in `dpif-netdev` is direct (no kernel round-trip), which reduces delay.
- Different debug tools: `ovs-appctl dpif-netdev/pmd-stats-show` instead of `ovs-dpctl dump-flows`.

## 16.2.7 Key takeaways

**Six key takeaways:**

1. **AF_XDP 4 rings (FILL, RX, TX, COMPLETION) plus UMEM**: zero-copy shared memory between userspace and the kernel driver. No memcpy, no context switch. Real-world bandwidth 10 to 30 Mpps per core, close to DPDK.
2. **3 XDP attach modes**: native (driver mode, fastest at about 24 Mpps), SKB (generic, compatible but slow at about 3 to 5 Mpps), hardware offload (Netronome or ConnectX-5 and later, zero CPU cost, instruction-set restrictions).
3. **4 return codes**: `XDP_PASS` (continue to the kernel stack), `XDP_DROP` (drop right at the driver, DDoS mitigation), `XDP_REDIRECT` (redirect to an AF_XDP socket or to another NIC), `XDP_TX` (send out the same interface, load balancer).
4. **libbpf versus libxdp**: libbpf is the general low-level eBPF API, libxdp is a higher-level wrapper for XDP specifically, supporting multi-prog plus lifecycle management.
5. **eBPF map types**: PERCPU_ARRAY (lock-free counter), HASH (general key-value), LPM_TRIE (CIDR routing), XSKMAP (AF_XDP socket redirect).
6. **Production use cases**: **Cilium** K8s CNI (replace kube-proxy), **Katran** Facebook LB, **Suricata** IDS AF_XDP capture, **OVS-AFXDP** hybrid OVS pipeline. XDP and AF_XDP have matured for cloud-native workloads.

### Capstone POE: DDoS mitigation at 200 Gbps, XDP or DPDK?

> **Lab verification status:** the POE is based on publicly documented Cloudflare L4Drop plus Facebook Shiv architecture plus AWS Shield Advanced. Pending the user having a production setup to verify reproducibility.

**Goal:** evaluate the stack choice for DDoS mitigation at hyperscale, comparing XDP and DPDK.

**Scenario:** Organization Y is a CDN providing edge service for 500 customers. Attack traffic in 2025 has reached **200 Gbps UDP flood plus 50 Mpps SYN flood**. Today they use iptables plus conntrack, with real-world bandwidth limited to about 10 Gbps per node, which does not scale. The CTO needs to choose a new stack for the anti-DDoS layer.

**Three options:**

1. **XDP plus AF_XDP:** XDP drops at the driver for the attack pattern, AF_XDP redirects legitimate traffic up to userspace inspection. Cloudflare L4Drop pattern.
2. **OVS-DPDK pipeline:** OVS-DPDK with custom ACL plus rate limiting. Arbor Networks appliance pattern.
3. **Dedicated hardware appliance:** Radware DefensePro or Arbor APS. Vendor-specific, not open-source.

**Predict:** evaluate the three options against five criteria: (a) real-world bandwidth capability, (b) latency impact on legitimate traffic, (c) deployment cost, (d) scalability, (e) flexibility for new attack patterns.

**Observe (reasoning plus data check):**

**Option 1 (XDP plus AF_XDP):**
- Real-world bandwidth: 40 Gbps or more XDP_DROP per core times 8 cores equals 320 Gbps drop capacity. Meets the 200 Gbps requirement.
- Latency for legitimate traffic: below 10 us (XDP_PASS into the kernel stack). Minimal impact.
- Deployment cost: medium (eBPF programming skill plus commodity hardware).
- Scalability: horizontal scale by adding commodity servers. Linear cost.
- Flexibility: high (write a custom XDP program for a new attack in a few hours, deploy through `bpftool attach`).

**Option 2 (OVS-DPDK):**
- Real-world bandwidth: 15 Mpps per core times 4 PMD cores equals 60 Mpps packet processing. Meets the 50 Mpps SYN flood requirement.
- Latency: 2 us (DPDK polling). On par with or better than XDP.
- Deployment cost: very high (hugepages plus IOMMU plus isolcpus plus OVS configuration complexity).
- Scalability: horizontal but expensive (DPDK requires a dedicated core per port).
- Flexibility: medium. OVS OpenFlow rules can be updated quickly but match limits apply.

**Option 3 (dedicated hardware appliance):**
- Real-world bandwidth: vendor-specified (1 to 10 Tbps for top-tier). Over-provisioned.
- Latency: below 1 us (hardware accelerated).
- Deployment cost: **very very high** ($500K to $5M per appliance plus a recurring license).
- Scalability: vertical (buy a bigger appliance), limited.
- Flexibility: low (vendor-controlled, custom patterns must wait for a vendor update).

**Explain:** recommendation:

**Primary: XDP plus AF_XDP (option 1).** Reasons:

- Real-world bandwidth of 320 Gbps drop capacity on commodity hardware, meeting the 200 Gbps requirement with 60 percent headroom.
- Highest flexibility: Cloudflare has reported writing a custom XDP filter for a new attack pattern in a few hours and deploying to the edge POP fleet in a few minutes. A dedicated appliance takes weeks or months.
- Best cost-to-performance ratio: commodity server plus open-source toolchain (libbpf, libxdp, bpftool). No recurring license cost.
- Production-proven: Cloudflare L4Drop (2018 onward), Facebook Shiv (XDP-based DDoS mitigation), AWS Shield Advanced (XDP-based tier) all run in real-world hyperscale.

**DO NOT choose:**
- OVS-DPDK: over-engineering for anti-DDoS. OVS OpenFlow flexibility is not needed for drop-or-pass logic. DPDK complexity is wasted at three times the cost.
- Dedicated appliance: over-priced. A CAPEX of $5M for one site versus $100K commodity server times 50 sites equals $5M with better flexibility plus distributed resilience.

**Falsification test:** try running anti-DDoS with the iptables plus conntrack baseline. Expectation: it fails at the 30 to 40 Gbps limit (iptables O(n) rule scan plus conntrack table explosion under SYN flood). Confirms that iptables is NOT enough for hyperscale attack and XDP is essential.

**Finish:** universal principle: **"Move packet drop to the earliest possible point"**. Traditional iptables drop happens AFTER skb allocation plus netfilter chain traversal. XDP drop happens BEFORE skb allocation. Difference: 10 to 100 times CPU efficiency. The same principle applies to:

- **Database query**: filter early (WHERE on an index) instead of filtering late (sort all then filter).
- **API rate limiting**: reject before authentication instead of after.
- **CDN caching**: serve static content at the edge instead of at origin.

XDP is not only a networking concept, it is a **general "early rejection" pattern** in system design.

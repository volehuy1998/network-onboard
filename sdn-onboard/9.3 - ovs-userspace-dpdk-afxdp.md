## Bối cảnh: trần hiệu năng của kernel datapath

Kernel datapath của OVS đủ cho phần lớn use case — VM workload, container network, SDN đơn giản. Nhưng khi yêu cầu throughput vượt ngưỡng 10 Gbps per flow hoặc packet rate trên 10 Mpps, kernel stack trở thành bottleneck.

Ba nguyên nhân gốc:

**Context switch kernel↔userspace**. Mỗi packet đến NIC gây interrupt, kernel xử lý interrupt, softirq ra quyết định forwarding. Với 10 Mpps, interrupt rate vượt 1 triệu/sec — CPU tốn 20-40% chỉ để xử lý interrupt overhead.

**Cache pollution**. Kernel network stack dùng chung L1/L2/L3 cache với application. Ở throughput cao, packet processing liên tục flush cache.

**Syscall và lock contention**. Kernel bridge dùng spinlock cho forwarding database, không scale tuyến tính với số CPU core.

Userspace datapath giải quyết bằng hai cách: **DPDK** (kernel bypass hoàn toàn) và **AF_XDP** (hybrid, dùng XDP framework kernel).

## 9.3.1 DPDK: kernel bypass với Poll Mode Driver

DPDK (Data Plane Development Kit) — thư viện Intel phát triển từ 2010, chuyển Linux Foundation 2017 — cho phép application userspace truy cập trực tiếp NIC ring buffer.

Nguyên lý: kernel driver được unbind khỏi NIC, userspace driver (UIO hoặc VFIO) map memory NIC vào process userspace. Application userspace chạy **Poll Mode Driver (PMD)** — thread liên tục đọc NIC ring trong tight loop, không chờ interrupt. Packet đến → PMD đọc → xử lý → ghi tx ring. Không context switch, không syscall, không interrupt.

Cấu hình OVS với DPDK (compass Ch 14):

```bash
[host]# ovs-vsctl set Open_vSwitch . other_config:dpdk-init=true
[host]# systemctl restart openvswitch-switch

[host]# ovs-vsctl add-br br-dpdk -- set Bridge br-dpdk datapath_type=netdev
[host]# ovs-vsctl add-port br-dpdk p0 -- set Interface p0 type=dpdk \
         options:dpdk-devargs=0000:01:00.0
```

Tuning knob quan trọng:

- `other_config:dpdk-socket-mem=1024,1024` — cấp 1GB hugepage mỗi NUMA socket.
- `other_config:dpdk-lcore-mask=0x2` — core dành cho DPDK threads nội bộ.
- `other_config:pmd-cpu-mask=0x3c` — core dành cho PMD polling (bits 2-5 = core 2,3,4,5).
- `other_config:emc-insert-inv-prob` — xác suất insert vào EMC (Exact Match Cache). Set 0 cho workload nhiều flow song song.

Monitor (compass Ch R):

```bash
[host]# ovs-appctl dpif-netdev/pmd-stats-show
[host]# ovs-appctl dpif-netdev/pmd-rxq-show
[host]# ovs-appctl dpif-netdev/pmd-perf-show    # cần other_config:pmd-perf-metrics=true
```

## 9.3.2 Hugepages và NUMA pinning

DPDK yêu cầu **hugepages** — trang memory 2MB hoặc 1GB thay vì 4KB. Lý do: DPDK allocate memory pool lớn cho ring buffer và packet buffer; nếu dùng page 4KB, TLB (Translation Lookaside Buffer) miss xảy ra thường xuyên → CPU stall khi walk page table. Với hugepage 2MB, TLB cover vùng memory lớn hơn, giảm TLB miss đáng kể.

Cấu hình hugepage:

```bash
[host]# echo 1024 > /sys/kernel/mm/hugepages/hugepages-2048kB/nr_hugepages
[host]# echo "vm.nr_hugepages=1024" >> /etc/sysctl.conf
```

Kernel command line: `intel_iommu=on iommu=pt` (cho VFIO), `default_hugepagesz=1G hugepagesz=1G hugepages=4` (cho 1GB hugepage).

**NUMA pinning** — quan trọng cho multi-socket server. NIC gắn vào PCIe slot thuộc một CPU socket cụ thể. PMD thread phải chạy trên core cùng socket với NIC, nếu không packet đi qua QPI (QuickPath Interconnect) giữa hai socket → latency tăng 40-60%.

```bash
[host]# cat /sys/bus/pci/devices/0000:01:00.0/numa_node
0
[host]# ovs-vsctl set Open_vSwitch . other_config:pmd-cpu-mask=0x3c
```

CPU isolation: `isolcpus=2-5` trong kernel command line — Linux scheduler không đặt process khác lên core này.

## 9.3.3 AF_XDP: alternative không kernel bypass

AF_XDP (Address Family eXpress Data Path) — giới thiệu Linux 4.18 (2018) — throughput gần DPDK nhưng không bypass kernel hoàn toàn.

Cơ chế: XDP là framework cho phép attach eBPF program vào NIC driver ở vị trí sớm trong network stack (trước khi tạo sk_buff). XDP program có thể drop, pass, hoặc redirect packet sang socket AF_XDP. Socket AF_XDP cung cấp userspace access đến ring buffer tương tự DPDK.

Cấu hình:

```bash
[host]# ovs-vsctl add-port br-dpdk ethX -- set Interface ethX type=afxdp \
         options:xdp-mode=native
```

Ba mode XDP:
- **`native`**: NIC driver hỗ trợ XDP natively, zero-copy từ ring. Performance tốt nhất.
- **`skb`**: fallback khi driver không hỗ trợ native. Slower.
- **`hw`**: XDP program offload xuống NIC hardware (Netronome Agilio). Hạn chế hardware.

## 9.3.4 So sánh kernel / DPDK / AF_XDP

*Table 9-3: Trade-off matrix cho ba loại datapath*

| Tiêu chí | Kernel | DPDK | AF_XDP |
|---|---|---|---|
| Kernel bypass | Không | Hoàn toàn | Không (qua XDP) |
| Driver | Kernel driver | Userspace UIO/VFIO | Kernel driver |
| Throughput tối đa per flow | 5-10 Gbps | 100+ Gbps | 80-90 Gbps |
| Latency | Medium | Thấp nhất | Tương đương DPDK |
| CPU cost | Interrupt-driven | 100% dedicated core | Moderate |
| Driver availability | Mọi NIC | Danh sách hạn chế | Mọi NIC Linux support |
| Kernel features (iptables, conntrack) | Giữ | Mất hoàn toàn | Giữ |
| Ops complexity | Thấp | Cao (hugepage, isolcpus, NUMA pin) | Trung bình |
| Container workload / CNI | Tốt | Kém | OK |

Matrix quyết định:

- Workload < 5 Gbps, ops đơn giản → **Kernel**.
- Workload 25+ Gbps, throughput critical, NFV latency-sensitive → **DPDK**.
- Workload 10-25 Gbps cần giữ kernel features → **AF_XDP**.
- Container + Kubernetes → **Kernel** (DPDK không tích hợp CNI tốt).

## Điểm cốt lõi cần nhớ

Thứ nhất, userspace datapath không phải upgrade mặc định từ kernel — mỗi cái phục vụ use case khác nhau. Kernel đủ cho workload enterprise thông thường; DPDK/AF_XDP cần thiết khi throughput vượt ngưỡng kernel.

Thứ hai, DPDK yêu cầu dedicate hardware resources: NIC (unbind kernel driver), CPU core (isolcpus + pmd-cpu-mask), memory (hugepage). Không thể "vừa chạy DPDK vừa workload khác" trên cùng core — PMD polling 100% CPU.

Thứ ba, AF_XDP là compromise: throughput 80-90% DPDK, ops complexity thấp hơn, giữ kernel features. Lựa chọn mặc định cho workload trung bình-cao không đòi hỏi throughput cực đại.

Thứ tư, NUMA pinning bắt buộc cho production DPDK — PMD sai NUMA socket với NIC gây latency penalty lớn.

## 9.21.1 Mininet là gì, vì sao quan trọng

Năm 2010, nhóm Stanford Clean Slate do Nick McKeown dẫn dắt công bố công cụ Mininet trong bài *"A Network in a Laptop: Rapid Prototyping for Software-Defined Networks"* tại HotNets-IX. Mục tiêu: cho phép bất kỳ nhà nghiên cứu SDN nào tái dựng lại topology thử nghiệm chỉ bằng một script Python và một laptop. Trước Mininet, reproducible research SDN đòi hỏi phần cứng chuyên dụng hàng chục nghìn USD; sau Mininet, mọi paper SDN 2010-2018 đều kèm Mininet script. Đây là bước chuyển quyết định từ "SDN là lý thuyết" sang "SDN là tooling có sẵn trên laptop". Tài liệu Lab 2 Crichigno mô tả ngắn gọn: *"Mininet is a virtual testbed enabling the development and testing of network tools and protocols. With a single command, Mininet can create a realistic virtual network on any type of machine (Virtual Machine (VM), cloud-hosted, or native)."*

Bốn tính chất làm Mininet hữu dụng hơn mọi network simulator trước đó:

**Emulation, không phải simulation.** Mininet chạy binary Linux thật (Python, curl, ping, OVS kernel module) trên kernel thật. Simulator như ns-3 phải port protocol sang framework của simulator. Điểm khác biệt: mọi bug trong code OVS gốc đều xuất hiện trong Mininet, không cần port lại.

**Network namespace làm host.** Mỗi "host" trong Mininet thực chất là một process (hoặc nhóm process) chạy trong network namespace riêng của Linux kernel. Namespace cô lập interface, bảng routing, bảng ARP. Chi phí tạo ~50 KB RAM mỗi host, nên Mininet tạo được mạng 1.000+ host trên một laptop.

**Virtual ethernet (veth) làm dây.** Kết nối giữa host và switch là cặp veth. Mỗi đầu veth nằm trong một namespace khác nhau; dữ liệu ghi vào đầu này xuất hiện ở đầu kia. Trên lý thuyết không khác gì cable vật lý; trên thực tế chi phí CPU thấp vì không có hashing hay serialize.

**Open-source và community lớn.** Mininet nằm trong ecosystem GitHub, có docs, có hàng ngàn example topology trên pastebin và GitHub. Fork phổ biến như Containernet (thêm Docker container làm host) giữ tương thích API, Lab 2 của Crichigno thực chất chạy Containernet, gọi `sudo mn` vẫn vào prompt `containernet>`.

## 9.21.2 Mininet CLI, lệnh cơ bản

Lab 2 hướng dẫn 6 bước cơ bản. Trình tự dưới tóm lược toàn bộ.

### Khởi tạo topology mặc định

Mở terminal Linux, gõ:

```bash
sudo mn
```

Password mặc định máy Lab là `password`. Sau vài giây, prompt chuyển thành `containernet>` (hoặc `mininet>` nếu bản chuẩn). Topology mặc định gồm 1 switch `s1`, 2 host `h1`/`h2`, 1 controller `c0`. Mininet tự gán IP 10.0.0.1/8 và 10.0.0.2/8 cho h1 và h2.

### Khám phá topology

Bốn lệnh thường dùng ngay sau khi khởi tạo:

```
containernet> help        # In toàn bộ lệnh CLI + ví dụ
containernet> nodes       # In danh sách node (c0 s1 h1 h2)
containernet> net         # In bảng kết nối interface ↔ interface
containernet> dump        # In thông tin chi tiết process PID + IP của từng node
```

Kết quả `net` cho topology mặc định:

```
h1 h1-eth0:s1-eth1
h2 h2-eth0:s1-eth2
s1 lo:  s1-eth1:h1-eth0 s1-eth2:h2-eth0
c0
```

Đọc ngược: `h1-eth0` của host h1 kết nối với `s1-eth1` của switch s1. Controller `c0` không có interface vì chạy out-of-band.

### Thực thi lệnh trên host cụ thể

Mininet cho phép gõ lệnh Linux bất kỳ cho một host, bằng cú pháp `<host> <lệnh>`:

```
containernet> h1 ifconfig
containernet> h1 ping -c 4 10.0.0.2
containernet> h2 traceroute 10.0.0.1
```

Lệnh `h1 ifconfig` xuất bảng interface của host h1, không phải của máy chủ chạy Mininet. Đây là đặc tính cách ly network namespace.

Có ba lệnh shortcut đáng nhớ:

```
containernet> pingall      # Ping từ mỗi host tới mọi host khác
containernet> iperf         # Chạy iperf test giữa h1 và h2 mặc định
containernet> xterm h1 h2   # Mở xterm riêng cho mỗi host (cần X11)
```

### Thoát và dọn dẹp

```
containernet> exit
```

Nếu lỡ đóng terminal giữa chừng và Mininet không thoát sạch, namespace còn sót lại sẽ gây lỗi khi khởi tạo lần sau. Lệnh dọn dẹp chạy ngoài Mininet CLI:

```bash
sudo mn -c
```

Lệnh này kill mọi process Mininet, xoá mọi namespace còn sót, xoá mọi OVS bridge Mininet tạo. Chạy `mn -c` trước khi khởi tạo session mới nếu có nghi ngờ.

## 9.21.3 Custom Python topology, `Topo` class API

Khi topology vượt ngoài default (1 switch, 2 host), dùng Python script. Mininet cung cấp class `Topo` trong module `mininet.topo`. Pattern cơ bản:

```python
# mytopo.py
from mininet.topo import Topo

class MyTopo(Topo):
    "Custom topology: 3 host, 2 switch, ring."
    def build(self):
        # Thêm host (tự gán IP mặc định 10.0.0.N)
        h1 = self.addHost('h1')
        h2 = self.addHost('h2')
        h3 = self.addHost('h3')

        # Thêm switch
        s1 = self.addSwitch('s1')
        s2 = self.addSwitch('s2')

        # Kết nối host với switch
        self.addLink(h1, s1)
        self.addLink(h2, s1)
        self.addLink(h3, s2)

        # Kết nối switch với switch
        self.addLink(s1, s2)

topos = {'mytopo': (lambda: MyTopo())}
```

Lưu thành file `mytopo.py`, chạy:

```bash
sudo mn --custom mytopo.py --topo mytopo
```

Mininet parse file, thực thi `MyTopo().build()`, dựng topology theo đúng script. Prompt `containernet>` xuất hiện như lệnh `sudo mn` thông thường, nhưng `nodes` sẽ trả về `c0 s1 s2 h1 h2 h3`.

### Gán IP và cấu hình tuỳ chỉnh

Mặc định Mininet gán IP dãy 10.0.0.0/8 theo thứ tự host được thêm. Muốn gán IP thủ công, dùng option trong `addHost`:

```python
h1 = self.addHost('h1', ip='192.168.1.10/24', mac='00:00:00:00:00:01')
h2 = self.addHost('h2', ip='192.168.2.10/24', mac='00:00:00:00:00:02')
```

Gán subnet khác nhau sẽ tạo nhu cầu định tuyến. Khi hai host khác subnet, Mininet không tự cấu hình router; bạn phải tự thêm switch làm router (xem §9.21.5).

### Chọn controller khác

Mặc định Mininet dùng controller `OVSController` nội bộ (built-in learning switch đơn giản). Muốn kết nối đến controller bên ngoài, dùng option `--controller`:

```bash
sudo mn --custom mytopo.py --topo mytopo \
    --controller=remote,ip=127.0.0.1,port=6653
```

Mininet sẽ chỉ OVS switch tới controller ở `127.0.0.1:6653`. Dùng khi bạn chạy Ryu, ONOS, hoặc OpenDaylight riêng và muốn Mininet switch làm data plane.

## 9.21.4 MiniEdit GUI, kéo-thả topology

MiniEdit là ứng dụng GUI Python viết bằng Tkinter, nằm trong source tree Mininet ở `examples/miniedit.py`. Chạy:

```bash
sudo python /home/admin/mininet/examples/miniedit.py
```

(Đường dẫn tuỳ cách cài; trên máy Lab Crichigno có shortcut desktop.) MiniEdit mở cửa sổ với toolbar bên trái:

1. **Select** (mũi tên): chọn hoặc di chuyển thiết bị. Phím Del xoá thiết bị đã chọn.
2. **Host**: thêm host. Click vào canvas để đặt.
3. **Legacy switch**: thêm switch. Click vào canvas để đặt.
4. **Link**: kết nối hai thiết bị. Click vào thiết bị đầu, kéo sang thiết bị thứ hai.
5. **Run**: bắt đầu emulation. Mọi nút bị disable cho đến khi bấm Stop.
6. **Stop**: kết thúc emulation.

### Cấu hình thiết bị bằng chuột

Click phải vào host, chọn **Properties**, điền IP address (ví dụ `10.0.0.1/8`), MAC, default route. Tương tự cho switch, config OpenFlow version, controller target.

### Mở terminal trên host

Sau khi bấm Run để emulation chạy, click phải vào host, chọn **Terminal**. Một xterm mở ra trong network namespace của host đó; lệnh `ifconfig` hiển thị interface của host, không phải của máy chủ.

### Lưu và tải topology

File → Save mở dialog lưu file `.mn` (JSON format định nghĩa topology). File → Open tải file `.mn` có sẵn. File Lab 2 lưu tên `lab2.mn`, mở lên sẽ có topology 2 LAN + 1 router r1.

### Xuất Python script

File → Export Level 2 Script xuất file `.py` chạy được bằng Python trực tiếp, tương đương dùng `Topo` class. Workflow phổ biến: thiết kế topology trên MiniEdit (dễ hình dung), xuất Python script (dễ version control), từ đó sửa tiếp bằng code để scale hoặc biến thể.

### X11 chuyển tiếp khi SSH

Nếu Mininet chạy trên máy chủ xa và bạn SSH vào, MiniEdit cần X11 chuyển tiếp. Setup:

```bash
# Trên máy client (Linux/macOS)
ssh -X admin@mininet-host

# Verify X11 forward hoạt động
echo $DISPLAY      # Phải có giá trị dạng "localhost:10.0"
xauth list         # Phải có entry
```

Windows client cần VcXsrv hoặc Xming làm X server local. Sau đó SSH dùng PuTTY với X11 chuyển tiếp bật.

## 9.21.5 Router emulation, sysctl và static route

Mininet host mặc định chỉ có hành vi end-host ở Layer 3. Muốn host đóng vai router, cần bật IP chuyển tiếp:

```bash
# Trong terminal của host r1 trên Mininet
sysctl -w net.ipv4.ip_forward=1
```

Lab 2 section 4 (trang 19-25) dựng topology 2 LAN nối qua router r1:

```
LAN1 (192.168.1.0/24)              LAN2 (192.168.2.0/24)
  h1 (.10) ──── s1 ──── r1 ──── s2 ──── h2 (.10)
                         │
                     .1 LAN1 side
                     .1 LAN2 side
```

Script Python:

```python
from mininet.topo import Topo

class TwoLanRouterTopo(Topo):
    def build(self):
        h1 = self.addHost('h1', ip='192.168.1.10/24',
                          defaultRoute='via 192.168.1.1')
        h2 = self.addHost('h2', ip='192.168.2.10/24',
                          defaultRoute='via 192.168.2.1')

        # r1 có 2 interface, mỗi interface 1 subnet
        r1 = self.addHost('r1')

        s1 = self.addSwitch('s1')
        s2 = self.addSwitch('s2')

        self.addLink(h1, s1)
        self.addLink(h2, s2)
        self.addLink(s1, r1, intfName2='r1-eth1',
                     params2={'ip': '192.168.1.1/24'})
        self.addLink(s2, r1, intfName2='r1-eth2',
                     params2={'ip': '192.168.2.1/24'})

topos = {'tworouter': (lambda: TwoLanRouterTopo())}
```

Chạy:

```bash
sudo mn --custom tworouter.py --topo tworouter
containernet> r1 sysctl -w net.ipv4.ip_forward=1
containernet> h1 ping -c 4 192.168.2.10
```

Nếu ping chưa qua được, lý do phổ biến: r1 có interface đúng nhưng chưa bật `ip_forward`, hoặc `defaultRoute` trên h1/h2 bị thiếu. Xác nhận bằng `h1 ip route`.

## 9.21.6 Tích hợp Mininet với OVS

Mininet mặc định dùng switch kernel Linux bridge (legacy) hoặc OVS tuỳ bản cài. Để bắt buộc dùng OVS, thêm option `--switch ovsk`:

```bash
sudo mn --custom mytopo.py --topo mytopo --switch ovsk
```

Sau khi emulation chạy, OVS bridge thật tên `s1`, `s2`, ... xuất hiện trong OVS database. Mở terminal bên ngoài Mininet (terminal thứ hai trên máy Lab), kiểm tra:

```bash
sudo ovs-vsctl show
# Output: Bridge s1 với interface s1-eth1 và s1-eth2

sudo ovs-ofctl dump-flows s1
# Output: bảng flow hiện tại (mặc định chỉ có 1 entry "actions=NORMAL")

sudo ovs-ofctl add-flow s1 "priority=100,ip,nw_dst=10.0.0.2,actions=output:2"
# Cài flow thủ công
```

Mọi lệnh `ovs-ofctl`, `ovs-appctl ofproto/trace`, `ovs-dpctl` trong Phần 9.4 đến 9.25 đều áp dụng cho bridge Mininet tạo. Đây là cơ chế cho phép Lab 3-15 của Crichigno dùng Mininet dựng topology và OVS CLI để cài luồng.

### Kết nối đến controller bên ngoài

Khi học SDN với Ryu/ONOS/ODL, khởi động controller trên cổng 6653 trước:

```bash
# Terminal 1: khởi Ryu
ryu-manager ryu.app.simple_switch_13
```

Sau đó Mininet:

```bash
# Terminal 2: khởi Mininet với remote controller
sudo mn --custom mytopo.py --topo mytopo --switch ovsk \
    --controller=remote,ip=127.0.0.1,port=6653
```

OVS switch trong topology sẽ kết nối Ryu, gửi gói tin-in khi không match luồng. Ryu sẽ trả gói tin-out/luồng-mod. Tích hợp này là nền tảng cho toàn bộ workflow phát triển controller SDN.

## 9.21.7 So sánh Mininet và namespace thủ công

Trước khi có Mininet, lập trình viên dựng topology test bằng `ip netns` + `ip link` thủ công. Script tay điển hình:

```bash
#!/bin/bash
# Tạo namespace
sudo ip netns add h1
sudo ip netns add h2

# Tạo veth pair, một đầu trong namespace
sudo ip link add h1-eth0 type veth peer name s1-eth1
sudo ip link add h2-eth0 type veth peer name s1-eth2
sudo ip link set h1-eth0 netns h1
sudo ip link set h2-eth0 netns h2

# Gán IP trong từng namespace
sudo ip netns exec h1 ip addr add 10.0.0.1/8 dev h1-eth0
sudo ip netns exec h1 ip link set h1-eth0 up
sudo ip netns exec h1 ip link set lo up
# Repeat cho h2...

# Tạo bridge OVS, thêm đầu veth còn lại
sudo ovs-vsctl add-br s1
sudo ovs-vsctl add-port s1 s1-eth1
sudo ovs-vsctl add-port s1 s1-eth2
sudo ip link set s1-eth1 up
sudo ip link set s1-eth2 up
```

Cùng topology đó trên Mininet chỉ 5 dòng Python `Topo` class. Ưu thế Mininet:

- **Tốc độ dựng**: script 200 dòng bash so với 5 dòng Python.
- **Dọn dẹp tự động**: `exit` Mininet xoá sạch mọi namespace và bridge, không cần script cleanup riêng.
- **Mininet CLI**: gõ `h1 ping h2` ngay trong prompt, không cần `ip netns exec h1`.
- **Scale**: thêm host bằng một dòng `self.addHost('h100')`; thủ công phải copy-paste 10 dòng.
- **Reproducibility**: chia sẻ một file `.py` ngắn, người khác chạy lại đúng topology.

Nhược điểm Mininet:

- **Phụ thuộc Mininet install**: cần package `mininet`, `python3-mininet`, `openvswitch-switch`. Trên máy chủ production không có thì phải dùng namespace thủ công.
- **Chi phí phụ khi khởi tạo**: Mininet mất vài giây để dựng; namespace thủ công dựng gần như tức thời.
- **Không mô phỏng phần cứng tốc độ cao**: veth là software channel, không có backpressure như NIC 100 Gbps thật. Với test scale lớn (> 10 Gbps), dùng DPDK hoặc testbed phần cứng.

Quy tắc thực dụng: học và prototype SDN dùng Mininet. Triển khai môi trường production dùng OVS/OVN trên kernel thật hoặc DPDK. Benchmark hiệu năng dùng testbed phần cứng. Mininet là bước đầu tiên, không phải bước cuối cùng.

- 1 switch OVS tên `s1`.
- 4 host h1, h2, h3, h4, tất cả cùng subnet 10.0.0.0/8.
- IP: h1=10.0.0.1, h2=10.0.0.2, h3=10.0.0.3, h4=10.0.0.4.
- MAC: h1=00:00:00:00:00:01, h2=00:00:00:00:00:02, h3=00:00:00:00:00:03, h4=00:00:00:00:00:04.
- Mỗi host kết nối đến switch qua interface `hN-eth0` và `s1-ethN`.

### Bước 2: Viết script Python

Tạo file `lab5.py`:

```python
# lab5.py - Tái dựng topology Lab 5 Crichigno
from mininet.topo import Topo

class Lab5Topo(Topo):
    "Lab 5: 1 switch + 4 host cùng subnet."
    def build(self):
        s1 = self.addSwitch('s1')

        for i in range(1, 5):
            hi = self.addHost(
                f'h{i}',
                ip=f'10.0.0.{i}/8',
                mac=f'00:00:00:00:00:0{i}'
            )
            self.addLink(hi, s1)

topos = {'lab5': (lambda: Lab5Topo())}
```

### Bước 3: Chạy Mininet với script

```bash
sudo mn --custom lab5.py --topo lab5 --switch ovsk
```

Kỳ vọng prompt `containernet>`. Các lệnh kiểm chứng:

```
containernet> nodes
# Kỳ vọng: c0 s1 h1 h2 h3 h4

containernet> net
# Kỳ vọng:
# h1 h1-eth0:s1-eth1
# h2 h2-eth0:s1-eth2
# h3 h3-eth0:s1-eth3
# h4 h4-eth0:s1-eth4
# s1 lo:  s1-eth1:h1-eth0 s1-eth2:h2-eth0 s1-eth3:h3-eth0 s1-eth4:h4-eth0
# c0
```

### Bước 4: Kiểm chứng connectivity

```
containernet> pingall
```

Kỳ vọng (doc-plausible):

```
*** Ping: testing ping reachability
h1 -> h2 h3 h4
h2 -> h1 h3 h4
h3 -> h1 h2 h4
h4 -> h1 h2 h3
*** Results: 0% dropped (12/12 received)
```

0% gói tin drop là dấu hiệu topology dựng đúng và OVS tự học MAC qua action `NORMAL` mặc định.

### Bước 5: Kiểm tra MAC đã gán đúng

```
containernet> h1 ifconfig h1-eth0
```

Kỳ vọng dòng `ether 00:00:00:00:00:01` trong kết quả. Tương tự cho h2/h3/h4.

### Bước 6: Kiểm chứng OVS bridge ở bên ngoài

Mở terminal thứ hai trên máy chủ (không phải bên trong Mininet CLI):

```bash
sudo ovs-vsctl show
```

Kỳ vọng hiển thị Bridge `s1` với 4 interface `s1-eth1` đến `s1-eth4`.

```bash
sudo ovs-ofctl dump-flows s1
```

Kỳ vọng chỉ có một luồng mặc định `actions=NORMAL`. Đây là lý do `pingall` hoạt động: NORMAL kích hoạt MAC learning tự động.

### Bước 7: Thoát và dọn dẹp

```
containernet> exit
```

Sau đó trên shell:

```bash
sudo mn -c
```

### Bài học

Viết một script Mininet mất chưa đầy 10 phút cho topology 5-node cơ bản. Workflow này áp dụng cho mọi Lab 3-15 Crichigno: đọc topology, ánh xạ sang `addHost/addSwitch/addLink`, chạy, kiểm chứng. Khi đã quen, bạn có thể sửa tham số (IP, MAC, số host, topology ring/star/mesh) trong vài dòng để thử giả thuyết riêng, đó chính là giá trị cốt lõi của Mininet: rút thời gian từ ý tưởng sang thí nghiệm xuống còn vài phút.

## 9.21.8 Tip vận hành cho phần còn lại của Khối IX

Trong các Phần 9.22 đến 9.25, mọi topology lab đều dựng bằng Mininet. Ba tip giúp workflow trôi chảy:

**Tip 1: Luôn chạy `sudo mn -c` trước mỗi lần khởi tạo.** Namespace sót từ session trước gây lỗi khó hiểu ("port already exists", "device busy"). Một lần `mn -c` giải quyết hầu hết.

**Tip 2: Dùng hai terminal.** Terminal 1 chạy Mininet CLI (`sudo mn ...`). Terminal 2 chạy OVS CLI (`ovs-ofctl`, `ovs-appctl`) trên cùng máy chủ. Terminal 2 thấy OVS bridge Mininet tạo, cho phép cài luồng thủ công mà không cần thoát Mininet.

**Tip 3: Lưu script Python theo version control.** Mỗi Phần trong series có file `lab<N>.py` hoặc `part9-<X>.py`. Commit lên git để tái dựng chính xác khi lab thất bại hoặc khi so sánh hành vi giữa các phiên bản OVS.

## 9.21.9 Điểm cốt lõi cần nhớ

**Một.** Mininet là chuẩn de-facto cho reproducible research SDN từ 2010, cho phép tái dựng topology chỉ bằng một script Python trên một laptop.

**Hai.** Mỗi host Mininet là một network namespace Linux; mỗi dây nối là cặp veth. Topology 100 node chạy trên laptop 4 GB RAM không vấn đề, vì namespace và veth đều siêu nhẹ.

**Ba.** CLI cơ bản: `sudo mn` khởi tạo, `help`/`nodes`/`net`/`dump` khám phá, `h1 <cmd>` chạy lệnh trên host, `pingall` kiểm chứng, `exit` thoát, `sudo mn -c` dọn dẹp.

**Bốn.** Custom topology viết bằng Python `Topo` class: `addHost`, `addSwitch`, `addLink`. Xuất biến `topos = {'name': (lambda: MyTopo())}` để `mn --custom file.py --topo name` nhận được.

**Năm.** MiniEdit GUI dùng khi thiết kế, Python script dùng khi triển khai hoặc version control. Workflow tiêu chuẩn: vẽ trên MiniEdit, xuất Python, chỉnh tiếp bằng code.

**Sáu.** Tích hợp OVS qua `--switch ovsk`. Sau khi emulation chạy, `ovs-vsctl show` và `ovs-ofctl dump-flows` từ terminal ngoài cho phép cài luồng và gỡ lỗi bằng toolbox Phần 9.25.

**Bảy.** Mininet hơn namespace thủ công về tốc độ dựng, dọn dẹp, khả năng mở rộng và tính tái lập (reproducibility). Kém namespace thủ công khi bạn cần chi phí phụ thấp hoặc không cài được package. Kém phần cứng thật khi cần benchmark 100 Gbps.

## Learning objectives

After completing this part, you will be able to:

1. **Identify (Remember)** the four cooperating pieces of the Open vSwitch architecture (kernel datapath, `ovs-vswitchd`, `ovsdb-server`, optional OpenFlow controller) and the role each piece plays in the forwarding path of a single frame.
2. **Distinguish (Analyze)** the four common port types (internal, system veth, tap, patch) by three orthogonal axes: who owns the netdev's lifecycle, where the netdev's wire endpoint lives, and what survives a `del-port`.
3. **Distinguish (Analyze)** the kernel datapath (`ovs-system`, `datapath_type=""`) from the userspace datapath (`ovs-netdev`, `datapath_type=netdev`) by their forwarding mechanism, their kernel-side presence, and their cache structure.
4. **Distinguish (Analyze)** the three caches that influence forwarding (the OpenFlow flow table at `ovs-ofctl dump-flows`, the kernel megaflow cache at `ovs-appctl dpctl/dump-flows`, the L2 forwarding database at `ovs-appctl fdb/show`), knowing which command populates which cache and which command clears which cache.
5. **Apply (Apply)** the four-pane mental model (operator, OVSDB, daemon, kernel) to read any OVS bridge configuration, predicting the output of `ovs-vsctl show`, `ovs-vsctl list Bridge`, `ovs-appctl dpif/show`, and `ovs-dpctl show` for any combination of bridges, ports, and datapath types.
6. **Apply (Apply)** the `--may-exist` and `--if-exists` idempotence flags to write `ovs-vsctl` scripts that are safely re-runnable, and predict that `--may-exist add-*` preserves row UUIDs while delete-and-recreate produces fresh ones.
7. **Verify (Apply)** a learned-MAC-to-OF-port binding by reading three observation surfaces (kernel `link/ether`, OVSDB `mac_in_use`, FDB entry from `fdb/show`) and confirming agreement byte for byte.
8. **Recover (Apply)** from the three most common L2-learning incidents: a stale FDB entry after a port removal (use `fdb/flush`), a stuck megaflow cache after a flow-table change (use `dpctl/del-flows`), and a MAC-spoofing attack on a NORMAL-mode bridge (mitigate via OpenFlow rules with explicit `dl_src` matches, R3 territory).
9. **Predict (Evaluate)** the behaviour of a bridge under each of the three `fail_mode` states (empty, `secure`, `standalone`), the four port types under different lifecycle events, and the in-process zero-cost cross-bridge crossing through a patch pair.
10. **Design (Create)** a multi-bridge OVS topology with patch-pair connectivity for policy separation, choosing the datapath type per bridge and the port types per port based on the operational requirements.

## 0.5.1. The four-piece architecture, restated

A reader who has spent ten years operating Cisco IOS or Junos boxes carries a single physical mental model of "a switch": one chassis with a backplane, one forwarding ASIC, one set of physical ports on the front, and one IOS process running on a control-plane CPU. The whole switch is one box. The ports are what the operator can see. The forwarding table lives in the ASIC, not in software.

Open vSwitch is not one box. Open vSwitch is four cooperating pieces of software that together impersonate the function of a switch. The four pieces are not interchangeable; each owns a specific layer of the forwarding pipeline; learning the layers in order is the foundation of every later sub-capture in the curriculum.

### The kernel datapath

The kernel datapath is the kernel module `openvswitch.ko`. The reader saw it in `lsmod | grep openvswitch` in lab R2.0: it is loaded with refcount 0 when the bridge is empty, refcount 1 when one bridge is configured, and so on. The datapath owns the fast path. When a frame arrives on a Linux network interface that has been "given to OVS" (added to a bridge with `ovs-vsctl add-port`), the kernel hooks intercept the frame before the Linux IP stack sees it and hand it to the datapath. The datapath looks up a flow in the per-bridge kernel flow cache (the megaflow cache). If a match is found, the datapath executes the cached action; if no match is found, the datapath emits an upcall to userspace via netlink. The datapath is the closest analogue Open vSwitch has to a Cisco ASIC: it is the part that touches every frame at line rate.

The datapath has no policy of its own. It executes the actions that `ovs-vswitchd` installs into the megaflow cache. R4 (the datapath internals lab) measures the cache's hit-and-miss behaviour directly; lab R2.0 introduced the cache, and lab R2.2 watched a frame populate it for the first time.

### The userspace daemon

`ovs-vswitchd` is the userspace daemon that owns the slow path and the policy. When the kernel datapath punts a frame, `ovs-vswitchd` receives the punt (this is called an "upcall"), translates the slow-path policy into a concrete sequence of actions, and replies to the kernel with both the actions for this specific frame and a megaflow cache entry the kernel should install so the next equivalent frame stays in the fast path.

The translation engine is `ofproto/ofproto-dpif-xlate.c:xlate_actions()`. For a bridge running in `OFPROTO_FAIL_STANDALONE` mode (lab R2.0's default and lab R2.2's running configuration), the translation falls into `xlate_normal()` at line 3083, which implements the full IEEE 802.1D learning bridge behaviour: parse the frame headers, look up the source MAC for learning, look up the destination MAC for forwarding, flood if unknown, output if known. R2.2 watched this function execute step by step via `ofproto/trace`.

`ovs-vswitchd` is also the OpenFlow speaker. When an OpenFlow controller is configured (which the entire R2 sub-capture set deliberately avoided to keep the focus on the implicit-NORMAL synthesis), the daemon receives flow modifications from the controller via the OpenFlow channel and incorporates them into the translation. Lab R2.1 demonstrated the controller-disconnect failure mode and the source-level proof that `connmgr_set_fail_mode()` flushes the flow table when there is no controller.

### The database server

`ovsdb-server` is the database server. Every configuration change the operator makes (every `ovs-vsctl add-br`, every `ovs-vsctl set Port br0 tag=10`) is a JSON-RPC transaction against a database that lives in `/usr/local/etc/openvswitch/conf.db`. `ovs-vswitchd` is a client of that database: it watches the database for changes through an Interface Definition Language (IDL) handle, and reconfigures itself when the database changes. This split is unfamiliar to a Cisco engineer because IOS does not have a separate database server; the running configuration and the box that uses the running configuration live inside the same process.

The split exists because OVS was designed for environments where one operator program (Nova-compute, OpenStack Neutron, OVN-Kubernetes, a hand-written cloud-management system) wants to make many small configuration changes per second through a transactional API. OVSDB gives that transactional API for free; `ovs-vswitchd` is the consumer of the transactions, not the owner of them. Lab R2.7 exercised the transactional model directly with `--may-exist`, `--if-exists`, and the row-UUID lifetime hierarchy.

### The optional OpenFlow controller

The OpenFlow controller, when one is configured, is the fourth piece. Most labs in this curriculum do not configure a controller; the bridge runs in `fail_mode=standalone` (the default) and synthesises an implicit `NORMAL` flow at runtime that learns and forwards exactly the way a Cisco access switch does. R3 is the first sprint that connects a real OpenFlow controller; R2 deliberately stays controller-less so the reader can build a clean mental model of the kernel-and-userspace architecture before adding the third party.

## 0.5.2. The bridge as the unit of policy

A bridge in OVS is more than just a Cisco-style L2 switch instance. It is the unit at which the operator chooses every important policy:

- **The forwarding plane**: kernel or userspace, via `Bridge.datapath_type`. R2.6 demonstrated both running simultaneously on one host.
- **The controller-attachment behaviour**: secure or standalone, via `Bridge.fail_mode`. R2.1 demonstrated all three states (empty, `secure`, `standalone`) and proved that toggling the column flushes the flow table when no controller is configured.
- **The OpenFlow protocol versions**: OpenFlow 1.0 through 1.5, via `Bridge.protocols`. The default is OF 1.0 (lab R2.0 read this from `ovs-ofctl show`); R3 will set higher versions explicitly.
- **The ports**: each `Port` row references its parent `Bridge`, and a port can carry one or more `Interface` rows (the latter for bonds, R2.5.4 territory).
- **The learning behaviour**: ageing time via `Bridge.other_config:mac-aging-time`, capacity via `Bridge.other_config:mac-table-size`. R2.8 named these knobs.

The Cisco analogue is the chassis configuration plus the per-VLAN configuration, except that on OVS the granularity is finer (each policy is per-bridge, not per-VLAN), and the policy can change at runtime without disrupting the data path more than necessary.

### The Bridge / Port / Interface hierarchy

```ascii
   Bridge   "br0"
       Port "br0"        (auto-created LOCAL)
           Interface "br0"  (type=internal)
       Port "eth1"       (added by add-port)
           Interface "eth1" (type="" = system)
       Port "bond0"      (added by add-port for a 2-NIC bond)
           Interface "eth2" (type="" = system; one of two members)
           Interface "eth3" (type="" = system; the other member)
```

The three-level hierarchy exists to support bonding: a single `Port` can have multiple `Interface` rows (one per bond member). R2.0 introduced the three-level structure; R2.5.4 (when redone) will exercise the bond case. For R2 most ports have a single Interface, so the hierarchy looks like a redundancy until the bond case is encountered.

## 0.5.3. The four port types, contrasted

The following table summarises the four port types covered in R2.3 and R2.4. Each row encodes three orthogonal facts the operator should be able to recall.

```ascii
   Type    OVSDB type   Kernel netdev?  del-port -> netdev?  Where the wire ends
   -----   ----------   --------------  -------------------  --------------------------
   internal   internal     YES             gone             host root namespace L3
   system        ""        YES             stays            arbitrary kernel netdev
   tap           ""        YES             stays            /dev/net/tun (userspace fd)
   patch       patch       NO              N/A              the peer bridge in-process
```

`internal` ports (R2.3) are owned by OVS end to end: OVS asks the kernel to create the netdev (via `RTM_NEWLINK` on its provider), OVS owns the netdev's lifecycle, and `del-port` removes the netdev. The bridge LOCAL is always an internal port. Operators who need an L3-attachable interface for the host's IP stack to use create another internal port and assign an IP to it.

`system` ports (R2.3) are wrappers around pre-existing kernel netdevs. The operator created the netdev (with `ip link add type veth`, `ip tuntap add`, or by plugging in a NIC); OVS adds the netdev to its forwarding table as a port. OVS does *not* own the netdev's lifecycle; `del-port` releases OVS's claim but does not delete the netdev.

`tap` ports (R2.3) are a special case of system ports for TUN/TAP devices. The userspace process that opens `/dev/net/tun` is the partner end; the netdev is the OVS-side end. R2.3 demonstrated the `link_state=down` artifact of having no userspace fd-holder.

`patch` ports (R2.4) are unique: they have *no kernel netdev*. The two halves of a patch pair name each other in `Interface.options:peer`, and a frame output to one half is delivered to the other half entirely inside `ovs-vswitchd`'s userspace. R2.4 proved this with five distinct observations: `ip link show` returns "Device does not exist", `dpif/show` reports `1/none:`, `ofproto/trace` produces a multi-bridge trace, `dpctl/dump-flows` shows a single megaflow with the peer-bridge DP port as the action, and deleting one half breaks the wire.

The pedagogical implication is that the operator's mental model for "port" must be type-aware. A port is not a uniform abstraction; the type determines the lifecycle, the kernel-side presence, and the wire endpoint. R5 (tunnels) extends this set of types with the tunnel ports (Geneve, VXLAN, GRE), each of which has its own implementation in `lib/netdev-vport.c`.

## 0.5.4. The two datapaths, contrasted

R2.6 demonstrated that `ovs-vswitchd` can drive both the kernel datapath (`ovs-system`) and the userspace datapath (`ovs-netdev`) simultaneously. The choice is per-bridge, controlled by `Bridge.datapath_type`.

```ascii
   Datapath name   datapath_type   Implementation                    Cache structure
   --------------  -------------   --------------------------------  -----------------------
   ovs-system        ""             openvswitch.ko (kernel module)   tuple-space search +
                                                                     masks-cache (lookup
                                                                     accelerator)
   ovs-netdev      "netdev"         in-process in ovs-vswitchd       dpcls (datapath classifier)
                                    via lib/dpif-netdev.c            + EMC (exact-match cache)
```

The two datapaths have:

- **Independent port-numbering spaces**. Both datapaths assign DP port numbers starting at 1; a system-datapath bridge can have DP port 1 simultaneously with a userspace-datapath bridge having its own DP port 1, without ambiguity, because each is in a different datapath namespace.
- **Independent cache structures**. The kernel's TSS classifier and the userspace's dpcls are different code paths; the tuning knobs and the operator-visible counters differ accordingly.
- **Independent kernel-side presence**. The kernel datapath has a kernel netdev `ovs-system` (visible in `/sys/class/net/`); the userspace datapath has a TAP-backed netdev `ovs-netdev` that the userspace daemon opens at startup. Both are visible to `ip link show`.
- **Different internal-port implementations**. R2.6 noted that `dpif/show` tags an internal port on the kernel datapath as `(internal)` while tagging the same OVSDB type on the userspace datapath as `(tap)`. The OVSDB `Interface.type=internal` means different things at runtime depending on the datapath.

The two datapaths share:

- **The OpenFlow flow table**. A flow rule installed via `ovs-ofctl add-flow` has the same meaning on both datapaths because the translation engine (`xlate_actions()`) is datapath-independent.
- **The translation logic**. `xlate_normal()`, the L2 learning code, and the upcall handler are all in the shared codebase; only the per-datapath provider differs.
- **The OVSDB schema**. Both datapaths use the same `Bridge`, `Port`, `Interface` tables; the only schema-level difference is the `Bridge.datapath_type` column value.

The operator chooses based on what they want to optimise. Production deployments use the kernel datapath because the kernel's zero-copy skb handling and the line-rate forwarding through `openvswitch.ko` are necessary for any non-trivial throughput. Development, debugging, and DPDK-based deployments use the userspace datapath. R2.6 stayed at the conceptual level; R4 (datapath internals) and R7 (QoS) measure the steady-state behaviour of both.

## 0.5.5. The three caches that influence forwarding

A reader who has worked through R2.0 through R2.8 has seen three different caches mentioned. The three are independent, populated by different code paths, invalidated by different events, and the operator must know which is which when debugging.

### Cache 1: the OpenFlow flow table

The OpenFlow flow table is what `ovs-ofctl dump-flows br0` reports. It holds:

- **Stored OpenFlow rules**, installed by the operator via `ovs-ofctl add-flow` or by an OpenFlow controller via the OpenFlow protocol.
- **Synthesised rules** rendered by the connection-manager when no rules are stored and the bridge is in `OFPROTO_FAIL_STANDALONE` mode. The priority-0 NORMAL flow R2.0 and R2.1 saw is the canonical example.

The flow table is the *operator-and-controller-facing* policy surface. It is the surface R3 (the OpenFlow programming sprint) will spend the entire sprint on. The flow table is cleared by `ovs-ofctl del-flows`, by `connmgr_set_fail_mode()` when no controller is configured (R2.1's surprise behaviour), and by bridge deletion.

### Cache 2: the kernel megaflow cache

The kernel megaflow cache is what `ovs-appctl dpctl/dump-flows` reports. It holds:

- **Compiled translations** from `xlate_actions()`. Each cache entry is a per-megaflow record of "for this match key, execute these datapath actions"; the actions are concrete (output to DP port N, push VLAN tag, etc.) and require no further translation.
- The cache is populated by upcalls (when a frame misses the cache) and aged out by the revalidator (a background thread that re-translates every cached entry against the current policy and evicts stale ones).

The megaflow cache is the *kernel-side fast path*. It is what makes OVS fast: nearly every frame after the first one in a conversation hits the cache and is forwarded without a userspace round trip. R4 measures the cache directly; R2.2 watched it populate; R2.4 showed a single megaflow can encode a cross-bridge crossing without recirculation; R2.8 proved the cache survives `fdb/flush`.

The megaflow cache is cleared by `ovs-appctl dpctl/del-flows`, by `ovs-vswitchd` restart (the cache lives in the daemon's understanding of what the kernel has, plus the kernel's own state; some events force a re-population), and by certain configuration changes (adding or removing a port flushes the cache for that bridge).

### Cache 3: the L2 forwarding database (FDB)

The FDB is what `ovs-appctl fdb/show <bridge>` reports. It holds:

- **Per-bridge MAC-to-port mappings** with per-VLAN scoping. Each entry is "MAC X is reachable on port Y in VLAN Z" with an age in seconds.
- The FDB is populated by `xlate_normal()` as a side effect of L2 learning (every frame whose source MAC is unknown causes an insert; every frame whose source MAC is on a different port causes a move).

The FDB is the *L2-learning state*. It exists only when the bridge is doing L2 learning, which on a controller-less bridge means whenever the priority-0 NORMAL synthesis is in effect. A bridge with stored OpenFlow rules and no NORMAL action will have a populated flow table and a populated megaflow cache but an empty FDB; this is normal and the reader should not treat the empty FDB as a bug. R2.8 demonstrated all three caches with simultaneous reads.

The FDB is cleared by `ovs-appctl fdb/flush <bridge>`, by bridge deletion, and by the ageing timer for individual entries. R2.8 Claim 4 proved the FDB clear does not invalidate the megaflow cache.

### A summary table

```ascii
   Cache        Tool                   Cleared by             Populated by
   ----------   --------------------   --------------------   --------------------------------
   OF flow      ovs-ofctl dump-flows   ovs-ofctl del-flows    ovs-ofctl add-flow,
   table                               (manual);              controller flow_mod,
                                       fail_mode change       fail-mode synthesis
                                       with no controller;
                                       del-br
   megaflow     ovs-appctl             ovs-appctl             upcall translation,
   cache        dpctl/dump-flows       dpctl/del-flows;       revalidator re-translate
                                       certain configuration
                                       changes
   FDB          ovs-appctl fdb/show    ovs-appctl fdb/flush;  xlate_normal() side effect
                                       per-entry ageing;
                                       del-br
```

The three caches together encode the bridge's full L2-and-policy behaviour. The reader who can read all three at the same time and explain disagreements has the operational competency this part exists to build.

## 0.5.6. The OVSDB row UUID hierarchy and idempotence

R2.7 exercised the OVSDB transactional model directly. The takeaways for the operator:

### Row UUIDs are randomly allocated, not derived

When `ovs-vsctl add-br br0` runs, `ovsdb-server` generates a new 128-bit UUID for the Bridge row. The UUID is independent of the bridge's name, the configuration columns, and any operator state. Re-issuing `add-br br0` with `--may-exist` finds the existing row and returns the existing UUID; without `--may-exist`, the second invocation is an error. Delete-and-recreate produces a fresh UUID. R2.7 proved each of these behaviours by capturing UUIDs across successive operations.

### The lifetime hierarchy is strict

```ascii
   Row identity               Lifetime
   --------------------       -----------------------------------------------
   Open_vSwitch._uuid         Database file lifetime
                              (created by ovsdb-tool create, destroyed
                               only by file removal)

   Bridge._uuid               from add-br to del-br

   Port._uuid (non-LOCAL)     from add-port to del-port
                              (the LOCAL port is auto-created and lives
                               as long as the bridge)

   Interface._uuid            from add-port to del-port
                              (one per Port; multiple per Port for bonds)
```

Every Bridge row's UUID is allocated at create-time and freed at delete-time. Port and Interface rows are non-root in the schema; they are garbage-collected when no Bridge references them. A patch-port orphan (R2.4) is a special case: the row references a non-existent peer via `Interface.options:peer`, but OVSDB does not validate that opaque-string reference, so the orphan persists until the operator explicitly removes it.

### --may-exist and --if-exists for idempotent scripts

A CMS-driven script that uses raw `ovs-vsctl add-br` is fragile: a transient failure that causes a re-issue of the same command will error. Wrapping with `--may-exist` makes the command idempotent. Similarly, `--if-exists` makes deletion idempotent.

R2.7 proved that `--may-exist add-port` preserves *both* the Port and the Interface UUIDs (a partial preservation would be unsafe because a CMS that has cached one UUID would break). R2.7 also proved that the OVS singleton row's UUID `15b3fd6f-e30f-4ae3-b8ca-4531189afee5` has been unchanged across the entire R2 lab series; the reader can grep every R2 transcript to verify.

## 0.5.7. The complete worked example: a frame from `ns-a` to `ns-b`

Drawing every R2 sub-capture's lessons together: trace one ICMP echo request from `ns-a` (10.50.0.10) to `ns-b` (10.50.0.20) through the canonical two-namespace topology that R2.2 and R2.8 built.

### Step 1: the kernel inside ns-a generates the frame

`ns-a` runs `ping 10.50.0.20`. The kernel's IP layer consults its ARP cache; if 10.50.0.20 is not cached, an ARP request is generated first. After ARP resolution completes, the ICMP echo request is built with source MAC `02:00:00:00:0a:0a`, destination MAC `02:00:00:00:0b:0a` (resolved via ARP), source IP 10.50.0.10, destination IP 10.50.0.20.

### Step 2: the frame transits the veth pair

The frame is queued on `veth-a`. The veth driver's `xmit` function calls `netif_rx()` on the peer netdev `br-a` (in the host's root namespace). The veth driver does not rewrite the source MAC; the frame arrives at `br-a` with `src=02:00:00:00:0a:0a` unchanged.

### Step 3: the OVS kernel hook intercepts the frame

`br-a` is claimed by OVS (the kernel's `master ovs-system` flag is set). The OVS ingress hook intercepts the frame and computes a flow key from the frame's headers. The kernel module looks up the megaflow cache for a match.

### Step 4a: cache miss path (first frame in conversation)

If no megaflow matches, the kernel emits an `OVS_PACKET_CMD_MISS` netlink message containing the frame plus the flow key. `ovs-vswitchd` is parked in `poll(2)` on its netlink socket. The poll wakes; an upcall-handler thread dequeues the message.

The handler thread calls `xlate_actions()` from `ofproto/ofproto-dpif-xlate.c`. The function walks the bridge's flow table looking for a match. The flow table contains zero stored rules; the priority-0 NORMAL synthesis kicks in. Translation falls into `xlate_normal()` at line 3083. `xlate_normal()` does five things:

1. **Learn the source MAC**: `mac_learning_insert(02:00:00:00:0a:0a, OF port 1)`. The FDB now has an entry for ns-a's MAC against port 1.
2. **Look up the destination MAC**: `mac_learning_lookup(02:00:00:00:0b:0a)`. The FDB returns "OF port 2" (ns-b's MAC was learned earlier from ns-b's IPv6 chatter, or this is the first frame for which the destination is unknown and the bridge will flood).
3. If the destination is learned, output to that port. If not, fall into `xlate_normal_flood()` at line 3051, which enumerates the bridge's bundles and outputs to each one except the input port.
4. **Compute the megaflow match key**: the input port, source MAC, destination MAC, EtherType, and `nw_frag` are stamped as "consulted". Other fields (TTL, IP addresses, port numbers, TCP flags) are wildcarded.
5. **Produce the action list**: `output:<DP-port>` for the destination's DP port (which on the lab host's two-bridge topology of R2.8 is DP port 3 for ns-b).

The handler emits two netlink messages back to the kernel: an `OVS_PACKET_CMD_EXECUTE` containing the actions plus the original frame (which the kernel transmits on DP port 3, delivering to ns-b), and an `OVS_FLOW_CMD_NEW` containing the megaflow that the kernel inserts into the cache.

### Step 4b: cache hit path (subsequent frames)

A subsequent ICMP echo request from `ns-a` to `ns-b` hits the megaflow inserted in Step 4a. The kernel executes the cached `output:3` action directly, transmitting the frame on `br-b` (which delivers to ns-b through the veth peer `veth-b`). No upcall, no userspace round trip.

### Step 5: ns-b receives, replies, and the bridge learns the reverse

`ns-b`'s kernel sees the ICMP echo request, generates a reply with source `02:00:00:00:0b:0a` destination `02:00:00:00:0a:0a`. The reply enters the bridge through `br-b` as ingress on OF port 2. The bridge's `xlate_normal()` learns the source MAC against port 2 (or refreshes it, if already learned), looks up `02:00:00:00:0a:0a` in the FDB (finds it on port 1), outputs to port 1. ns-a receives the reply.

After the first round trip, the FDB contains both endpoints' MACs against the correct OF ports, and the megaflow cache contains forward and reverse entries. The conversation runs at line rate from the second packet onwards.

## 0.5.8. Misconceptions and reality

A short consolidated list of misconceptions a Cisco-trained reader is likely to bring into OVS, with the corrections each R2 sub-capture provides.

> **Misconception:** "the priority-0 NORMAL line in `dump-flows` is a stored rule."
> **Reality (R2.1):** it is a synthesised view rendered at dump time by the connection-manager when the bridge is in `OFPROTO_FAIL_STANDALONE` mode. Toggling `fail_mode=secure` makes it disappear.

> **Misconception:** "switching `fail_mode` is a passive policy change that does not touch my stored OpenFlow rules."
> **Reality (R2.1):** `connmgr_set_fail_mode()` calls `ofproto_flush_flows()` whenever the mode changes AND no controller is configured. Stored rules are wiped. The fix is to attach a placeholder controller (which flips `connmgr_has_controllers` to true regardless of `is_connected`) before changing the mode.

> **Misconception:** "the interface type is just metadata; OVS treats every port the same way internally."
> **Reality (R2.3):** the type selects which provider in `lib/netdev-*.c` handles the port, with materially different lifecycle and observability per type.

> **Misconception:** "if I add a netdev to a bridge with `add-port` and then `del-port`, OVS will delete the netdev."
> **Reality (R2.3):** OVS deletes only what OVS created. Internal ports created via `add-port type=internal` are deleted on `del-port`; pre-existing kernel netdevs (TAP, veth, physical NIC) are released, not deleted.

> **Misconception:** "patch ports are tunnels between two bridges."
> **Reality (R2.4):** patch ports are not tunnels. They have no encapsulation, no wire, no NIC. They are an in-process function call from one bridge's translation context to another's.

> **Misconception:** "if I delete one half of a patch pair, OVS will automatically delete the other."
> **Reality (R2.4):** OVS does not auto-clean orphan patch peers. The `options:peer` column is treated as an opaque string by OVSDB; the schema does not declare a foreign-key relationship.

> **Misconception:** "I have to choose one datapath for the whole host; I cannot mix system and netdev bridges."
> **Reality (R2.6):** `ovs-vswitchd` can drive both datapaths simultaneously. The choice is per-bridge.

> **Misconception:** "the row UUID is just a name; if I delete and re-add the same name, I get the same UUID."
> **Reality (R2.7):** UUIDs are allocated by `ovsdb-server` at insert time using a 128-bit random value. Delete-and-recreate produces a new UUID.

> **Misconception:** "del-port leaves the Port and Interface rows in their tables; the operator must manually delete them."
> **Reality (R2.7):** OVSDB garbage-collects orphan strong-referenced rows automatically.

> **Misconception:** "the bridge learns the MAC of the bridge-side veth end when a frame enters."
> **Reality (R2.8):** the bridge learns the source MAC of the frame, not the MAC of the netdev the frame entered through. The veth driver pass-through preserves the source MAC.

> **Misconception:** "`fdb/flush` is the same as `dpctl/del-flows`."
> **Reality (R2.8):** they touch different caches with independent lifetimes.

> **Misconception:** "the bridge will not believe a sudden change of port for an existing MAC; some kind of validation prevents MAC-spoofing."
> **Reality (R2.8):** the bridge believes every observation immediately. NORMAL-mode learning has no built-in validation; anti-spoofing requires explicit OpenFlow rules or higher-level controllers.

## 0.5.10. The most useful three-line operator summary of every bridge primitive

A reader who internalises the bridge-and-port primitives can summarise each in three concise lines that name the key abstraction, the canonical observation command, and the most common operational pitfall. The summary below is the reader's pocket reference.

### The Bridge

A bridge is the OVS abstraction of a single L2 forwarding domain plus its associated policy state. Each bridge has its own OF flow table, its own fail-mode, optional controller, datapath choice, and an auto-created LOCAL port. Observe with `ovs-vsctl show` and `ovs-vsctl list Bridge <name>`. Pitfall: setting `fail_mode=secure` on a bridge with no controller flushes the stored OpenFlow table; the operator must attach a placeholder controller first to preserve stored rules across the toggle.

### The Port

A port is an OVS abstraction layered between the bridge and one or more interfaces. Each port has a name, a VLAN tag/trunk policy, an optional bond mode, and a parent bridge. Observe with `ovs-vsctl list Port <name>`. Pitfall: an OVS port and a Linux bridge port are not the same thing; the OVS port is an OVSDB row, and the Linux bridge port is a kernel attribute. Confusing the two leads to thinking that `bridge fdb show` (the Linux command) and `ovs-appctl fdb/show <bridge>` (the OVS command) are related; they are not, except by coincidence in topologies that use both.

### The Interface (kind: internal)

An internal interface is an OVS-synthesised kernel netdev with an OVS-allocated MAC. The bridge LOCAL is always an internal interface; the operator can also add explicit internal interfaces for L3 attachment. Observe with `ovs-vsctl --columns=name,type,mac_in_use list Interface <name>`. Pitfall: deleting an internal interface via `del-port` removes the kernel netdev; the operator who has assigned an IP to the netdev loses that L3 configuration. Production CMS scripts that reconfigure internal interfaces should preserve the IP via the namespace's network configuration files, not via runtime `ip addr add` alone.

### The Interface (kind: system)

A system interface is a wrapper around a pre-existing kernel netdev (a veth end, a tap device, or a physical NIC). OVS does not own the netdev's lifecycle. Observe with `ip link show <name>` (kernel-side) plus `ovs-vsctl --columns=name,type list Interface <name>` (OVSDB-side; type column reads as `""`). Pitfall: `del-port` does not delete the netdev, only releases OVS's claim. The operator who runs `del-port` and expects the netdev to be gone is wrong.

### The Interface (kind: tap)

A tap interface is a special-cased system interface for TUN/TAP devices. The userspace fd-holder (a VM, a userspace bridge daemon, a tunctl process) is the partner end. Observe by checking for the `tun_flags` sysfs attribute. Pitfall: `link_state=down` on a tap port is normal when no userspace process holds the fd; the operator who sees the down state and runs `ip link set up` does nothing useful, because the link state is governed by the partner-end fd, not by the netdev's admin state.

### The Interface (kind: patch)

A patch interface is the in-process cross-bridge wire abstraction. No kernel netdev. Two halves of a patch pair name each other in `Interface.options:peer`. Observe with `ovs-appctl dpif/show` (look for the `1/none:` notation) and `ovs-vsctl --columns=name,type,options list Interface <name>`. Pitfall: an orphan half (one without a matching peer) does not break loudly; it just stops crossing. The operator must explicitly delete the orphan after deleting one half of a pair.

### The Datapath (system)

The kernel datapath `ovs-system` is the production-default forwarding plane. Implementation in `openvswitch.ko`. Observe with `ovs-dpctl show` (kernel-only view). Pitfall: kernel-version-specific bugs in the datapath module require kernel upgrades, not OVS userspace upgrades; the reader should always pair `lsmod | grep openvswitch` and `uname -r` when investigating data-path issues.

### The Datapath (netdev)

The userspace datapath `ovs-netdev` is the development-default and DPDK-host plane. Implementation in `lib/dpif-netdev.c`. Observe via `ovs-appctl dpctl/show` (daemon-routed view that includes both datapaths). Pitfall: the userspace datapath has different cache tunable parameters than the kernel datapath; an operator who has memorised the kernel-datapath knobs cannot apply them directly to the userspace datapath.

### The OF flow table

The OpenFlow flow table is the operator-and-controller-facing policy surface. Per bridge. Observe with `ovs-ofctl dump-flows <bridge>`. Pitfall: the priority-0 NORMAL line on a controller-less bridge is synthesised, not stored; the operator who treats it as stored will misdiagnose the next fail-mode toggle.

### The megaflow cache

The kernel megaflow cache is the per-bridge fast-path cache. Observe with `ovs-appctl dpctl/dump-flows`. Pitfall: the cache survives `fdb/flush` (R2.8 Claim 4 demonstrated). To invalidate megaflows, use `ovs-appctl dpctl/del-flows`.

### The FDB

The forwarding database is the L2 learning state. Per bridge, per VLAN. Observe with `ovs-appctl fdb/show <bridge>`. Pitfall: an unexpected MAC-move (the same MAC suddenly on a different port) is not flagged; the bridge believes every observation immediately. Anti-spoofing must be configured separately.

## 0.5.11. A worked production diagnostic walkthrough

The reader who has finished R2 should be able to apply the four-pane mental model to a real production diagnostic. Walking through one such scenario from start to finish makes the abstraction concrete and reusable.

### The scenario

An operator receives a page: "VM tenant1-web-01 cannot reach VM tenant1-app-01". The two VMs are on the same compute host (call it `compute-12`) running OpenStack Neutron with the OVS agent. Both VMs are in the same logical network, on the same subnet (10.0.0.0/24); the VMs are tenant1-web-01 at 10.0.0.10 and tenant1-app-01 at 10.0.0.20. The expected behaviour is that both VMs sit on `br-int` and forward L2 directly between each other; the failure is silent (ping reports 100% packet loss, no error messages on either VM).

A reader who has worked through R2 can construct a five-step diagnostic procedure that identifies the problem class within minutes, regardless of which of the dozen possible failure modes is the actual cause. The procedure is mechanical and does not require any production-specific knowledge.

### Step 1: read the operator-visible OVSDB state

The first command on the host is `ovs-vsctl show`. The operator looks for both VMs' tap netdevs as ports on `br-int`. If either tap is missing, the failure is upstream of OVS: either the VM did not start, or the Neutron OVS agent did not add the port, or the agent removed the port due to a security-group violation. Ninety percent of "VM unreachable" pages have this as the root cause; the other ninety percent require the next steps.

If both taps are present, the operator captures their OF port numbers and Interface row UUIDs. R2.7 taught the script-friendly form: `ovs-vsctl --bare --columns=_uuid,ofport find Interface name=tap1234abcd-00`. The operator records the values for both taps.

### Step 2: read the daemon's view

`ovs-appctl dpif/show` reports the current bridge-to-port mapping. The operator looks for both VMs' tap ports under the `br-int` section. The OF/DP port mappings should match the OVSDB rows from Step 1. A discrepancy (the OVSDB row exists but the port is missing from `dpif/show`) indicates the daemon has not yet processed the OVSDB write; this is rare but happens during high CMS-issue rates. The operator can wait 5 seconds and re-check, or look at `ovs-vswitchd.log` for processing-lag warnings.

### Step 3: read the L2 learning state

`ovs-appctl fdb/show br-int` reports what the bridge has learned. For a working topology the operator should see entries for each VM's MAC against each VM's tap port. If both MACs are present and on the correct ports, the bridge is learning correctly and the failure is downstream (in the VM's IP stack, in security-group rules at a higher level, or in conntrack state). If a MAC is missing or on the wrong port, R2.8 taught the move-detection mechanism: a misconfigured spoofing event might have moved the binding. The operator can `fdb/flush` and re-check; the FDB will repopulate from the next frame.

### Step 4: read the kernel megaflow cache

`ovs-appctl dpctl/dump-flows | grep <vm_mac>` reports the cached datapath actions for any frame involving the VM's MAC. The operator looks for forward-and-reverse megaflows for the inter-VM conversation. The presence of a megaflow with the correct action means the kernel is forwarding correctly; the absence means either the conversation has not started yet (the VMs are not exchanging traffic) or the megaflows have been recently invalidated (a flow-table change, a port add/remove, or an explicit `dpctl/del-flows`).

### Step 5: trace a hypothetical frame

`ovs-appctl ofproto/trace br-int 'in_port=<vm1_tap_ofport>,dl_src=<vm1_mac>,dl_dst=<vm2_mac>,dl_type=0x0800'` simulates the forward direction. R2.2, R2.4, and R2.8 demonstrated the trace output. The operator reads:

- The narrator's verdict at table 0: `forwarding to learned port` (good, the bridge knows where to send) or `no learned MAC for destination, flooding` (the L2 learning has not yet caught up, the next frame will populate it).
- The Megaflow line: the produced match key. If it includes additional match fields beyond input port and EtherType (e.g., `vlan_tci`, `nw_src`, `tcp_flags`), the bridge has stored OpenFlow rules that constrain forwarding (typically Neutron security-group rules); the operator can examine `ovs-ofctl dump-flows br-int` for the matching rule.
- The Datapath actions: the actual list. If the action list is `drop`, the bridge intends to drop this frame; some stored rule (or an OVN ACL) is the cause. If the action list is `output:<port>` matching the destination, forwarding is correct. If the action list includes `ct(...)` (conntrack), R6 is the relevant lab.

### A real diagnostic produces a hypothesis in five steps

The five steps above route the operator to one of about a dozen failure modes. Each failure mode is covered in detail in R9 (debug-and-troubleshoot doctrine); R2's vocabulary is the foundation. A reader who finishes 0.5 can already perform the five-step diagnostic on their own host.

The Cisco analogue. On a Catalyst, the operator runs `show mac address-table`, `show interface vlan 10`, and `show running-config interface gi0/1` to triangulate. On OVS, the operator reads OVSDB, daemon state, FDB, kernel cache, and trace. The Catalyst has three commands; OVS has five. The granularity is finer; the inference is similar.

## 0.5.11.A. A second worked diagnostic: the cross-bridge silent drop

The reader who has finished R2 can apply the same five-step diagnostic to a more subtle problem. Consider this scenario.

### The scenario

The operator has a two-bridge OVN-style layout (`br-int` plus `br-tun`) on a host. A patch pair connects them. Two VMs, one on `br-int` and one whose tunnel terminates on `br-tun`, were previously communicating. The operator pages: "tenant1-web-01 cannot reach tenant1-app-02; the latter is on a remote host". A reader who has read R2.4 should recognise that this scenario involves the patch pair.

### The diagnostic

Step 1 (OVSDB read): `ovs-vsctl show` shows both bridges, the patch pair, the local VM tap on `br-int`, and the geneve tunnel on `br-tun`. All present and configured. The operator records the patch ports' UUIDs and the tunnel port's UUID.

Step 2 (daemon read): `ovs-appctl dpif/show` shows the patch ports as `1/none:` (no DP port, as R2.4 taught) and the tunnel port and tap port as `(system)` with normal DP port numbers. So far so good.

Step 3 (FDB read): `ovs-appctl fdb/show br-int` shows tenant1-web-01's MAC against the tap port, and tenant1-app-02's MAC against the patch port (because frames from the remote VM arrive at `br-int` via the patch from `br-tun`). `ovs-appctl fdb/show br-tun` shows the reverse picture: the local web-01 MAC against the patch, and the remote app-02 MAC against the tunnel port. The L2 learning is correct on both bridges.

Step 4 (megaflow cache): `ovs-appctl dpctl/dump-flows | grep <web-01-mac>` shows the forward megaflow with the action being the DP port number of the tunnel on `br-tun`. The reverse direction's megaflow is also present. Both megaflows report `packets:0, bytes:0, used:never` despite the ping being attempted; this is the suspicious observation. The cache is populated but no traffic has been recorded against the cached entries.

Step 5 (trace): `ofproto/trace br-int 'in_port=<web-01-tap>,dl_src=<web-01-mac>,dl_dst=<app-02-mac>,dl_type=0x0800'` produces a multi-bridge trace `bridge("br-int")` then `bridge("br-tun")`, and the final Datapath actions name a tunnel-output action like `set(tunnel(...)),output(<DP-port>)`. The trace says forwarding is intended.

### The hypothesis

The trace says forwarding is correct, the FDB says learning is correct, the cache says megaflows are installed, but `packets:0` says no frame is hitting them. The most likely cause is something downstream of the OVS bridge that drops the frame before it reaches the kernel datapath: an iptables rule, a security-group filter, an MTU mismatch on the tunnel encapsulation, or a missing route on the underlay.

The operator runs `tcpdump -i <tunnel_underlay> 'udp port 6081'` (the geneve port). If frames are visible but acks are not, the underlay ACL is dropping. If frames are not visible at all, the bridge is producing them but they are stalling at the IP stack.

The point of the walkthrough. R2's vocabulary lets the operator narrow the failure to "downstream of OVS" within five commands. The actual fix is not OVS-specific (it could be iptables, kernel MTU, an OS-level firewall), but the diagnostic gets the operator to "the bridge is working correctly; look elsewhere" quickly. R9 onward will cover the downstream layers.

## 0.5.11.B. Operator habits that pay dividends across the curriculum

Six concrete habits the reader should adopt before moving on to R3 and beyond.

### Habit 1: always read OVSDB before reading the daemon or kernel

The OVSDB layer is the source of truth for configuration. If the OVSDB is wrong, the daemon and kernel will eventually be wrong too. The operator who starts diagnosing at the kernel layer often miss OVSDB-side issues that would have been obvious from a one-line `ovs-vsctl list-br` or `ovs-vsctl --columns=name,fail_mode,controller list Bridge`.

### Habit 2: capture UUIDs for every port at create time

In a CMS-driven environment, the operator's `ovs-vsctl add-port` is one of dozens per minute. Capturing the UUID of every newly-created port (via `ovs-vsctl --bare --columns=_uuid find Interface name=<name>`) gives the operator a stable identity to track across the port's lifetime, including across `ovs-vswitchd` restart and across CMS reconfigurations.

### Habit 3: use `--columns` for every script-driven read

Without `--columns`, `ovs-vsctl list <Table>` prints every column (more than 30 for `Interface`); the column order is not guaranteed to be stable across OVS versions. A script that parses the full output is brittle. Pin the columns the script needs.

### Habit 4: use `--may-exist` and `--if-exists` for every CMS-issued operation

These flags make commands idempotent. CMS scripts that are not idempotent are difficult to retry; CMS scripts that are idempotent are easy to retry. The cost of the flag is one extra OVSDB query per operation, which is negligible.

### Habit 5: always read all three caches when investigating L2 issues

The three caches are independent. Reading only one (e.g., only `dump-flows`, only `fdb/show`, only `dpctl/dump-flows`) leaves blind spots. The triple read is the diagnostic foundation; R9 will codify it as the first step of every L2 incident.

### Habit 6: prefer `ovs-appctl` over `ovs-dpctl` for kernel datapath queries

`ovs-dpctl` talks directly to the kernel module via netlink. `ovs-appctl dpctl/show` routes through `ovs-vswitchd`, which gives the daemon a chance to add cross-correlation information. The two should produce equivalent kernel-datapath data, but `ovs-appctl dpctl/show` is more useful in a multi-datapath host because it reports both datapaths.

## 0.5.13. A short glossary of the terms R2 introduced

- **Bridge**: the OVS abstraction equivalent to a Cisco Catalyst's L2 switch instance. A bridge has ports, a fail-mode, an optional controller, a datapath type, and a flow table. R2.0 introduced the bridge; the entire R2 sub-capture set explored aspects of it.
- **Port**: an OVS abstraction layered between the bridge and the interface. A bridge contains ports; a port contains one or more interfaces (multiple in the bond case). The Cisco analogue is the difference between a `Port-channel1` and the member `Gi0/1`/`Gi0/2`. R2.0 introduced the three-level hierarchy.
- **Interface**: the OVS abstraction for a physical or virtual netdev. Each interface has a type (`internal`, `tap`, system veth, `patch`, or a tunnel kind), an OF port number, a kernel-assigned MAC, and a per-interface OVSDB row. R2.3 covered the four common types in detail.
- **OF port**: the OpenFlow port number for a port within a bridge. Per-bridge namespace. The well-known reserved values include `OFPP_LOCAL=65534` (the bridge LOCAL), `OFPP_NORMAL=65532`, `OFPP_CONTROLLER=65533`. R2.0 introduced these constants.
- **DP port**: the kernel datapath's per-host vport identifier. Allocated in vport-creation order. Independent of OF port. R2.0 noted the OF/DP asymmetry; R2.6 demonstrated the per-datapath DP namespace.
- **LOCAL port**: the auto-created internal port that every bridge has. OF port 65534. The bridge's L3-attachable interface. R2.0 introduced it.
- **Datapath**: the OVS abstraction for a forwarding plane. The two common datapaths are `ovs-system` (kernel) and `ovs-netdev` (userspace). Per-bridge `Bridge.datapath_type` selects. R2.6 demonstrated both.
- **Megaflow**: a cached translation in the kernel datapath. Covers a range of microflows that share the same translation. Match key wildcarded for fields the policy did not consult. R2.2 introduced the concept; R2.4 showed cross-bridge megaflows; R2.8 proved the cache is independent of the FDB.
- **FDB (forwarding database)**: the L2 MAC-to-port table the bridge maintains for `xlate_normal()`-driven learning. Per-bridge, per-VLAN. Aged out over time. R2.2 introduced; R2.8 exercised the move-detection logic.
- **fail_mode**: the bridge's controller-disconnect behaviour. Three states: empty (defaults to standalone), `secure` (drop traffic when no controller and no stored flows), `standalone` (synthesise a priority-0 NORMAL catch-all). R2.1 demonstrated all three states.
- **Patch port**: an in-process cross-bridge wire. Two patch ports name each other in `Interface.options:peer`. No kernel netdev. Forwarding crosses bridges via translator recursion in `xlate_actions()`. R2.4 demonstrated.
- **Standalone-mode synthesis**: the priority-0 actions=NORMAL line that `dump-flows` shows when the bridge is in `OFPROTO_FAIL_STANDALONE` mode. Not a stored rule; rendered at dump time. R2.1 proved the synthesis property.
- **NORMAL action**: the OpenFlow special action that triggers `xlate_normal()`'s L2 learning state machine. Not a stored rule; the action's behaviour is defined by the OF specification. R2.0 introduced; R2.2 traced through.
- **Upcall**: a kernel-to-userspace netlink message that punts a frame to `ovs-vswitchd` for translation. R2.0 introduced the concept; R2.2 watched it happen.
- **xlate_actions()**: the userspace translation engine that converts OpenFlow flow-table rules and policy into concrete datapath actions plus a megaflow. R2.0 named; R2.2 traced; R2.4 showed the patch-peer recursion; R2.6 noted the engine is datapath-independent.
- **xlate_normal()**: the special-case translation for the NORMAL action. Implements IEEE 802.1D learning bridge behaviour. R2.0 named; R2.2 traced; R2.8 exercised the move-detection logic.
- **`ovs-vsctl`**: the operator's primary CLI to OVSDB. JSON-RPC client. Supports `--may-exist` and `--if-exists` for idempotence. R2.0 onward used.
- **`ovs-ofctl`**: the operator's primary CLI to the OpenFlow flow table. Speaks OpenFlow to the bridge. R2.0 onward used.
- **`ovs-appctl`**: the operator's CLI to the daemon's unixctl interface. Reads `dpif/show`, `fdb/show`, `dpctl/dump-flows`, `ofproto/trace`. R2.0 onward used.
- **`ovs-dpctl`**: the operator's CLI directly to the kernel datapath via netlink. Bypasses `ovs-vswitchd`. R2.0 introduced.
- **OVSDB row UUID**: the 128-bit random identifier `ovsdb-server` allocates per row. Independent of name and column values. Persists for the row's lifetime. R2.7 demonstrated.
- **`external_ids`**: a string-to-string map column on most OVSDB tables, ignored by OVS itself, available for operator and CMS metadata. R2.7 demonstrated.
- **`other_config`**: a string-to-string map column for OVS-recognised configuration knobs (e.g., `mac-aging-time`, `hwaddr`). Distinct from `external_ids`.
- **`isRoot`**: a schema-level flag declaring whether a table's rows persist independently or only as long as another table references them. Open_vSwitch is root; Bridge, Port, Interface are non-root. R2.7 explained the GC implications.
- **Singleton row**: the Open_vSwitch table's one row, declared in the schema as `maxRows=1, isRoot=true`. UUID stable for database file lifetime. R2.0 introduced; R2.7 confirmed across the entire lab series.

## 0.5.13.B. A reading order for the OVS source tree, in-depth

A reader who has finished R2 can productively read the OVS source tree. The recommended order is to follow a frame's lifecycle through the architecture:

1. **`include/openvswitch/`**: the public API headers. Read these first to understand what the codebase exposes.
2. **`lib/`** (about 200 files): the shared library. Read `lib/dpif.h` and `lib/dpif-provider.h` first, then `lib/netdev.h` and `lib/netdev-provider.h`, then `lib/ofp-*.h` for the OpenFlow surface, and `lib/mac-learning.h` for the FDB.
3. **`ofproto/`**: the OpenFlow-to-datapath translation engine. Read `ofproto/ofproto.h` and `ofproto/ofproto-dpif.h`. Then the heavy file `ofproto/ofproto-dpif-xlate.c` (about 8000 lines), starting from `xlate_actions()`.
4. **`vswitchd/`**: the `ovs-vswitchd` daemon glue. Read `vswitchd/bridge.c` (`bridge_run` is the main loop).
5. **`ovsdb/`**: the database server. Read `ovsdb/ovsdb-server.c` for the main loop, then `ovsdb/transaction.c` for the transaction logic.
6. **`datapath/`** (kernel module sources, but see also `net/openvswitch/` in-tree). Read `datapath/datapath.c` (`ovs_dp_process_packet`).
7. **`utilities/`**: the operator CLIs. Read `utilities/ovs-vsctl.c` for the OVSDB client logic, `utilities/ovs-ofctl.c` for the OpenFlow client, `utilities/ovs-appctl.c` for the unixctl client.
8. **`tests/`**: the autotest suite. Read `tests/automake.mk` first, then any `.at` file in the area of interest.

R3 (OpenFlow programming) will dive deeper into `ofproto/`. R4 (datapath internals) will dive deeper into `lib/dpif-*.c` and the kernel datapath. The entire Track B sprint set (S0 to S10) walks the source tree systematically.

## 0.5.13.C. A short note on the OVS test suite

The OVS test suite at `tests/*.at` is a self-checking specification of OVS behaviour. Many of the properties R2 demonstrated are also asserted by upstream tests. A reader who wants to verify any R2 claim against the upstream test suite can:

- For the priority-0 NORMAL synthesis: see `tests/ofproto-dpif.at` for tests of fail-open and standalone-mode behaviour.
- For L2 learning: see `tests/learning-fairness.at` and the FDB-related tests.
- For patch ports: see `tests/ofproto-dpif.at` (search for `patch_class`).
- For the OVSDB row-UUID lifecycle: see `tests/ovsdb-server.at` and `tests/ovsdb-tool.at`.

R9 (debug-and-troubleshoot doctrine) returns to the test suite as a debugging aid: when the operator wants to verify that a feature works on a given OVS build, running the relevant `make check` test is often faster than constructing a lab.

R2 was the foundation. R3 (the OpenFlow programming sprint) is the first place the reader writes hand-crafted flow rules; R4 (datapath internals) measures the kernel megaflow cache directly; R5 (tunnels) introduces tunnel ports as a fifth port-type family; R5.5 (inter-host VM-to-VM) builds the multi-host topology; R6 (conntrack) adds stateful firewalling on top; R7 (QoS, mirroring, sFlow) introduces observation infrastructure; R8 (OVSDB internals) goes deep into the database server; R9 (debug-and-troubleshoot doctrine) builds the production debugging discipline; R10 (forensic and production readiness) is the operator's day-one playbook; R11 (capstone) is the multi-tenant overlay; R12 (final audit) closes the curriculum.

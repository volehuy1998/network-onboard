## Bối cảnh: tách rời stack dọc

Phần 5.0 đã phân tích API-based SDN giữ nguyên switch, add config layer. Phần 5.1 đã phân tích hypervisor overlay thay network edge từ hardware sang software. Phần này là mô hình thứ tư: **giữ hardware switch làm edge, nhưng disaggregate hardware + software**, cho phép operator chọn NOS (Network Operating System) độc lập với hardware vendor.

Traditional network: Cisco sell Cisco switch với Cisco IOS firmware, vendor-locked end-to-end. Whitebox mô hình: ODM (Original Design Manufacturer) như Celestica, Edgecore, Delta sản xuất bare-metal switch theo OCP spec, operator mua và cài NOS tự chọn (Cumulus, SONiC). Analogy với PC server: DELL sell server hardware, operator cài đặt Linux/Windows theo sở thích, không bị lock vào single-vendor OS.

Trigger của phong trào: Facebook scale pain point 2012-2013. Facebook có 10^5+ switch trong fabric, vendor license cost unbearable. Cisco/Juniper không thể drop price enough, forced Facebook to pursue alternative: build own hardware (Wedge, 6-pack) and pursue open-source NOS. Open Compute Project (OCP) formalize effort 2013.

## 5.2.1 OCP Network project (2013+)

### Nguồn gốc Open Compute Project

OCP founded tháng 4/2011 bởi Facebook với động cơ: disaggregate server hardware để giảm cost per compute unit. Initial scope server design (Facebook Open Compute Server v1, motherboard specs, chassis spec). Expand sang networking 2013.

OCP Networking project launch tháng 1/2013 tại OCP Summit. Goals:
- Standardize bare-metal switch hardware specs.
- Enable vendor-independent NOS ecosystem.
- Reduce switch cost through disaggregation + competition among ODM.

Early contributor: Facebook (primary), Accton Technology (ODM), Broadcom (silicon vendor), Cumulus Networks (NOS startup).

### Các đặc tả OCP

OCP Network publish hardware specs open-source. Notable:

- **Wedge** (Facebook 2014): 16-port 40G ToR switch, Broadcom Trident2 silicon, FBOSS Linux-based NOS.
- **6-pack** (Facebook 2015): modular spine switch, Trident2 based.
- **Minipack** (Facebook 2019): 128-port 100G, Tomahawk 3 silicon.
- **OCP Accepted switches**: ODM product meeting OCP spec, certified interoperable với OCP-compliant NOS.

Spec include mechanical dimensions, power connectors, management port (BMC), serial console (RS-232 at specific pinout), front panel indicators. Standardize across ODMs, any OCP NOS works với any OCP switch.

### Tham chiếu: trang chủ dự án OCP

OCP project info tại `https://www.opencompute.org/` (rất tiếc sometimes bot-protected, access may return 403 for automated tools). Alternative search via Google/Bing "OCP networking project specs 2024" for current state documentation.

## 5.2.2 Whitebox hardware (Celestica, Edgecore, Delta)

### Các ODM vendor chính

**Accton Technology / Edgecore**: Taiwanese ODM, largest OCP switch manufacturer. Flagship product Edgecore Wedge 100BF-32X (32-port 100G, Tofino silicon for P4 use case) và AS-series (Broadcom silicon general purpose).

**Celestica**: Canadian ODM, commonly used by Microsoft Azure datacenters. DX-series switches, broadly deployed.

**Delta Networks**: Taiwanese ODM focus carrier và enterprise. AG-series switches.

**Supermicro**: primarily server ODM, also makes switch for cost-sensitive deployment.

**Inventec, Quanta (QCT)**: other Taiwanese ODMs with OCP-compliant product lines.

### So sánh giá (ước lượng)

| Product category | Vendor OEM price (per port) | ODM whitebox price (per port) |
|-------------------|------------------------------|-------------------------------|
| 100G ToR (32-port) | Cisco/Juniper/Arista: ~$500-700/port | Edgecore/Celestica: ~$200-300/port |
| 400G spine | ~$1,500-2,000/port | ~$500-800/port |

Discount 40-60% per port. Với DC có 10^4-10^5 ports, savings = millions to tens of millions USD per DC, justifying learning curve cho NOS operation.

### Chuỗi cung ứng

ODM không bán retail; khách hàng qua:
- Direct wholesale cho hyperscale (Microsoft, Facebook, Google).
- Reseller system cho enterprise (e.g., Penguin Computing, Mellanox via indirect channels).
- Co-investment với NOS vendor (Cumulus distributed via Dell/HPE OEM channels before NVIDIA acquisition).

Challenge: mô hình hỗ trợ different. ODM không có 24/7 NOC như Cisco. Customer depend on NOS vendor hỗ trợ (SUSE for Cumulus pre-2020, Microsoft for SONiC enterprise) hoặc internal expertise.

## 5.2.3 Merchant silicon: Broadcom Tomahawk, Trident

### Sự thống lĩnh của Broadcom

Broadcom is dominant merchant silicon supplier for DC switch 2013-2026. Two families:

**Trident family (leaf/ToR)**:
- Trident2 (BCM56850, 2013): 1.28 Tbps, 40G.
- Trident3 (BCM56870, 2017): 3.2 Tbps, 100G.
- Trident4 (BCM56880, 2021): 12.8 Tbps, 400G.
- Trident5 (BCM56890, 2023): 25.6 Tbps, 800G.

**Tomahawk family (spine, high-radix)**:
- Tomahawk 1 (BCM56960, 2014): 3.2 Tbps.
- Tomahawk 2 (BCM56970, 2016): 6.4 Tbps.
- Tomahawk 3 (BCM56980, 2018): 12.8 Tbps.
- Tomahawk 4 (BCM56990, 2020): 25.6 Tbps.
- Tomahawk 5 (2023): 51.2 Tbps, 800G per port.

### Các vendor silicon thay thế

**Marvell Prestera** (formerly Cavium Xpliant): alternative general-purpose silicon. Smaller market share but present in some Cisco ASR products.

**Cisco Silicon One** (2019+): Cisco's own unified silicon family (before, Cisco used custom proprietary per-line. Silicon One consolidates across routers + switches). Primarily Cisco products, not merchant for whitebox.

**Intel Tofino** (acquired from Barefoot 2019): P4-programmable silicon. Intel announce EOL 01/2023, P4 momentum continued via community (p4.org) but no commercial production target.

**Nvidia Spectrum** (Cumulus + Mellanox acquisition 2020): Spectrum-1 through Spectrum-4, ASAP² eSwitch hardware offload (xem Phần 9.5). Merchant silicon for bare-metal switch.

### Quan hệ silicon so với NOS

ODM build box around silicon choice. NOS phải hỗ trợ silicon's SDK:
- **Broadcom SDK**: closed-source, NDA required for OEM.
- **Marvell SDK**: similar closed.
- **SAI (Switch Abstraction Interface)**: open-source abstraction layer trên silicon SDK. SONiC + stratum dùng SAI.

SAI là critical enabler: NOS viết 1 lần against SAI API, run on any silicon with SAI driver. Reduce per-silicon porting cost cho NOS.

## 5.2.4 NOS options: Cumulus Linux, SONiC, OpenSwitch, Stratum

### Cumulus Linux (2011-present, Nvidia post-2020)

**Founding**: Cumulus Networks founded 2010 bởi ex-Cisco engineers JR Rivers, Nolan Leake. First shipping product 2013.

**Architecture**:
- Debian Linux base với custom kernel + drivers.
- Standard Linux tools work: `ip`, `bridge`, `ethtool`, `tc`.
- **Policy**: configure switch qua Linux-native commands, FRR (Free Range Routing) for BGP/OSPF, systemd services.

**Business model**: commercial with subscription license (per-port per-year). Acquired by Nvidia 05/2020 (part of Mellanox deal) for ~$250M.

**Current 2026**: Cumulus Linux 5.x series, distributed via Nvidia as part of Spectrum ethernet portfolio. Documentation at `https://docs.nvidia.com/networking-ethernet-software/cumulus-linux-510/`.

### SONiC (2016-present, now Linux Foundation)

**Founding**: Microsoft Azure team 2016, open-source under MIT license from day 1.

**Architecture**:
- Debian Linux base.
- **Container-based architecture**: services (BGP, LLDP, SNMP, teamd) runs trong Docker containers.
- **SAI (Switch Abstraction Interface)**: abstraction layer between SONiC and silicon SDK. Enables portable NOS across Broadcom/Nvidia/Marvell.
- **Redis database**: central state store, pub/sub for inter-service communication.

**Components**:
- `syncd`: silicon SDK interaction (the only container that touches silicon).
- `swss` (Switch State Service): state management orchestrator.
- `bgp` container: FRR/Quagga BGP.
- `pmon`, `lldp`, `snmp`, `dhcp_relay`: functional containers.

**Áp dụng**:
- Microsoft Azure production: 10^5+ ports, primary NOS.
- Alibaba Cloud: large-scale production.
- LinkedIn: migrated to SONiC around 2019.
- Telefonica, Orange: European carrier sự áp dụng.

**Governance**: transferred from Microsoft to Linux Foundation (Open Compute Project + SONiC Foundation) 04/2022. Community at `https://sonicfoundation.dev/` + `https://github.com/sonic-net/SONiC`.

### OpenSwitch (2015-2017, abandoned)

HP (Hewlett-Packard Enterprise) launched OpenSwitch 10/2015 as open-source NOS for whitebox. Initial contributor: HPE, Accton, Arista Broadcom silicon foundation.

Issues:
- HPE corporate focus shift (HPE split 2015, networking priorities shifting).
- Competition with SONiC overwhelming (Microsoft commitment larger than HPE's).
- Governance unclear, community diluted.

OpenSwitch.net website still present 2026 but last meaningful release 2017. Abandoned, migration to SONiC or Cumulus.

### Stratum (ONF, 2018-present)

ONF Stratum is thin NOS cho P4-based programmable switch. Philosophy different từ Cumulus/SONiC:

**Cumulus/SONiC**: full NOS with BGP/OSPF/STP built-in, switch can operate autonomously.

**Stratum**: minimal agent, all intelligence trong remote controller. Switch is dumb executor, controller (ONOS) orchestrates everything.

**Use case**: P4-programmable fabric với Tofino silicon. ONOS compile P4 program, push to Stratum, Stratum load vào Tofino pipeline.

**Áp dụng**: niche, research + telco proof-of-concept. AT&T CORD architecture use Stratum + ONOS. Not mainstream enterprise.

### So sánh các lựa chọn NOS

| NOS | Base OS | SAI support | Governance | Production scale |
|-----|---------|-------------|------------|--------------------|
| Cumulus Linux | Debian | No (own abstraction) | Nvidia commercial | Enterprise mid-market |
| SONiC | Debian | Yes (primary abstraction) | Linux Foundation | Hyperscale (Azure, Alibaba) |
| OpenSwitch | Debian | Partial | Abandoned | N/A |
| Stratum | Thin agent | P4 runtime (not SAI) | ONF | Niche research/telco |

## 5.2.5 ONIE (Open Network cài đặt Environment)

### Chuẩn boot loader

ONIE (pronounced "OH-nee") là boot loader cho whitebox switches, role analog với PXE boot for PC servers.

Flow:
1. Switch power on, ONIE boots from flash (comes pre-installed from ODM).
2. ONIE probe network (DHCP để get IP, retrieve URL cho NOS image).
3. ONIE download NOS installer (HTTPS/HTTP/TFTP/USB).
4. NOS installer runs, writes NOS image to flash, reboots.
5. Switch now boots NOS, ONIE dormant until next re-install.

### Vai trò chuẩn hóa

Without ONIE: each ODM has custom boot mechanism, each NOS vendor hỗ trợ per-ODM. N × M complexity.

With ONIE: ODM triển khai ONIE spec (consistent boot interface). NOS vendor triển khai ONIE-compatible installer (retrieve via DHCP option 114 URL). Any NOS works with any ONIE-capable switch.

### Triển khai tham chiếu

**ONL (Open Network Linux)** là reference ONIE-compatible Linux distribution, maintained by Big Switch (acquired by Arista 2020). Used for testing ONIE interop.

**Reference URL**: `https://opencomputeproject.github.io/onie/` (OCP ONIE subproject official).

### Hạn chế

ONIE không standardize:
- Firmware upgrade of NOS (each NOS has own update mechanism).
- Dual-boot (fallback to previous NOS version).
- Hardware diagnostic (ODM-specific vendor tools).

Those require vendor-specific tooling beyond ONIE.

## 5.2.6 Current state 2026: SONiC thắng thị phần hyperscale

### Market snapshot

**Hyperscale** (>10,000 switch port per customer):
- Microsoft Azure: SONiC primary (>95% of new deployments).
- Alibaba Cloud: SONiC primary.
- LinkedIn: SONiC.
- Google, Facebook: proprietary forks of Linux-based NOS (FBOSS for Facebook). Not SONiC, but same mô hình.

**Enterprise mid-market** (100-10,000 ports):
- Cumulus Linux (under Nvidia Spectrum umbrella).
- SONiC sự áp dụng growing via community editions and enterprise distributions (Celestica-managed).
- Traditional Cisco/Juniper/Arista still hold majority of mid-market.

**Service provider**:
- SONiC via Broadcom + Edgecore: Telefonica, Orange, Deutsche Telekom pilots.
- Cumulus niche.
- Traditional Cisco/Juniper dominant via bespoke features.

### SONiC wins hyperscale because

1. **Microsoft commitment**: Microsoft produce + maintain SONiC, de facto standard for Azure scale. Community grows via Microsoft-initiated pull.
2. **SAI standardization**: SONiC abstracts silicon uniformly, operators swap silicon without NOS rewrite.
3. **Container architecture**: easy to test, upgrade, rollback individual services. Operational maturity higher than monolithic NOS.
4. **Open-source**: no license fee, no vendor lock-in. Pure operational cost.

### Cumulus limitations for hyperscale

Cumulus Linux excellent for enterprise (familiar Debian tooling, strong BGP via FRR), but:
- Commercial license prohibitive at 10^5 port scale.
- Acquisition by Nvidia 2020 tied strategic direction to Spectrum silicon; operators diversifying vendors uncomfortable.
- Container architecture of SONiC more advanced than Cumulus's traditional daemon architecture for operational flexibility.

### Forecast 2026+

- **SONiC dominance**: continue grow hyperscale and increasingly enterprise mid-market.
- **Cumulus**: stable enterprise base under Nvidia, not expand to hyperscale.
- **Traditional OEM** (Cisco, Juniper, Arista): retain enterprise via features (deep analytics, tight integration với vendor stack) but lose hyperscale irreversibly.
- **Stratum + P4**: revival attempt after Intel Tofino EOL? Possible via AMD Pensando DPU or other programmable silicon, but uncertain.

> **Chủ đề trọng tâm:** "Opening Up the Device" mô hình qua whitebox + OCP + ONIE + SAI is economically irreversible at hyperscale. Cost saving 40-60% per port combined với SONiC maturity eliminates vendor lock-in for largest operators. Traditional vendor (Cisco/Juniper/Arista) response: move value-add up the stack (network analytics, security, AI-driven operations) instead of competing on price per port. Enterprise mid-market evolves slower, but same trajectory over next 5-10 years.

### ▶ Capstone Khối V Trouble Ticket 5-1

**Scenario**: Yêu cầu multi-tenant DC với 1000 VMs across 50 compute nodes, 3 tier rack topology (spine-leaf-ToR), cần VLAN-based tenant isolation và per-tenant ACL policy. Budget constraint: minimize per-port hardware cost. Operational constraint: team có Linux expertise nhưng chưa từng dùng OpenFlow.

**Task**: Recommend SDN mô hình (OpenFlow / API-based / Hypervisor Overlay / Whitebox) phù hợp nhất. Justify through trade-off analysis.

**Expected analysis dimensions**:
1. Cost per port (Whitebox thấp nhất, Hypervisor Overlay chỉ yêu cầu software không hardware cost).
2. Complexity vs team expertise (Linux team → Cumulus/SONiC natural; OpenFlow requires learning).
3. Tenant isolation mechanism (Hypervisor Overlay native via VXLAN; Whitebox via VLAN limited to 4094).
4. Scale target (1000 VM là moderate; Hypervisor Overlay không over-engineer cho scale nhỏ).
5. Deployment time (Hypervisor Overlay provision minutes; Whitebox requires silicon config + ToR setup).

**Recommended answer** (guide, not mandatory): Hypervisor Overlay via OVN on KVM, underlay Whitebox (SONiC-based) cho simple IP fabric. Combine software flexibility (OVN multi-tenant) with hardware cost efficiency (whitebox underlay). Team Linux expertise applies to both layers.

## 13.5.1 Port_Binding overview — role + schema

`Port_Binding` row trong SBDB biểu diễn binding giữa **logical port** (NBDB LSP/LRP definition) và **physical chassis** (nơi traffic của port được xử lý). Mỗi LSP/LRP trong NBDB tương ứng với một Port_Binding row trong SBDB (do `ovn-northd` compile sinh ra).

### Schema columns quan trọng

```shell
$ sudo ovn-sbctl list Port_Binding lsp_web_01
_uuid          : 7a3c9f82-...
logical_port   : "lsp_web_01"
datapath       : <datapath_uuid for Logical_Switch>
chassis        : [<chassis_uuid>]        # NULL hoặc chassis claiming port
type           : ""                       # 8 values: "" / localnet / l2gateway / ...
tunnel_key     : 5                        # Per-datapath unique ID
mac            : ["aa:bb:cc:00:00:01 10.0.0.5"]
external_ids   : {tenant="T1"}
options        : {requested-chassis="chassis-A"}  # Optional pin
parent_port    : []                       # Cho nested port (virtual)
tag            : []                       # VLAN tag cho localnet
gateway_chassis: []                       # Dedicated column cho chassisredirect
ha_chassis_group: []                      # HA_Chassis_Group reference
up             : true                     # Binding state up/down
nat_addresses  : []                       # Cho logical router port
virtual_parent : []                       # Cho virtual type
```

### Claim mechanism

ovn-controller trên mỗi chassis subscribe SBDB table `Port_Binding`. Khi thấy row phù hợp điều kiện claim (VD: OVS port có `external_ids:iface-id` khớp với `logical_port`), controller gọi OVSDB transaction:

```sql
UPDATE Port_Binding SET chassis=<this_chassis_uuid> WHERE _uuid=<row_uuid>
```

SBDB Raft replicate update sang tất cả Raft follower → tất cả ovn-controller khác thấy binding mới trong < 500ms.

## 13.5.1b Port_Binding như một bảng SBDB: bản chất hàng + cơ chế binding

> Phần này đào sâu vào **chính bảng `Port_Binding`** (row trong OVN Southbound database), tách bạch khỏi catalog 8 type ở Phần 13.5b. Mục tiêu: kỹ sư hiểu thấu cấu trúc hàng, chu kỳ sống, các cột có ý nghĩa nghiệp vụ, và cách tra ngược từ một sự cố runtime về một row trong SBDB.

### Khái niệm cốt lõi

`Port_Binding` là **bảng nội trú trong OVN Southbound database (SBDB)** lưu **kết quả compile** của ovn-northd: mỗi `Logical_Switch_Port` (LSP) hoặc `Logical_Router_Port` (LRP) trong NBDB sinh ra đúng một row Port_Binding tương ứng trong SBDB. Row này là **hợp đồng dữ liệu** giữa control plane (ovn-northd) và data plane (ovn-controller trên từng chassis): northd định nghĩa "logical port này tồn tại ở datapath nào, mang MAC + IP gì, thuộc loại type gì, có pin chassis hay không"; ovn-controller trên từng chassis quyết định "row này có thuộc về tôi không, nếu thuộc thì cập nhật cột `chassis` về UUID của tôi và cài flow OpenFlow". Loại: **OVSDB table**, root table (`isRoot=true`), persistent (replicated qua Raft cluster SBDB).

### Lịch sử + bối cảnh ra đời

Bảng `Port_Binding` xuất hiện cùng OVN 2.4 (2015) khi Russell Bryant (Red Hat) và Justin Pettit (VMware/Nicira) refactor kiến trúc Nicira NSX OpenStack plugin thành OVN. Trước đó, OpenStack Neutron OVS plugin lưu binding state trực tiếp trong Neutron MySQL database, gây race khi neutron-server và compute node truyền tin chậm. Thiết kế OVN tách biệt: NBDB chứa **ý đồ** (intent), SBDB chứa **runtime state** mà mọi chassis cùng quan sát; Port_Binding là một trong những bảng SBDB gốc cùng với `Chassis`, `Datapath_Binding`, `Logical_Flow`. OVN 21.06 bổ sung cột `requested_chassis`, OVN 22.03 ổn định cột `up` cho liveness reporting, OVN 22.06 thêm cột `additional_chassis` cho live migration multichassis (xem axis "Khác biệt giữa các phiên bản" bên dưới).

### Vị trí trong kiến trúc

Bảng nằm ở **layer SBDB** (control-plane intermediate), giữa **NBDB** (declarative input từ CMS) và **OpenFlow trên `br-int`** (executable output tại chassis). Scope: **per-cluster, per-row**. Một cluster OVN chỉ có một SBDB Raft cluster (3 hoặc 5 thành viên), tất cả ovn-controller trên mọi chassis subscribe cùng SBDB qua kết nối OVSDB JSON-RPC. Mỗi row Port_Binding là **một bản ghi duy nhất** trong cluster (không nhân bản per-chassis); cột `chassis` trỏ về Chassis row của chassis đang sở hữu. Cấp độ tác động: thay đổi một row Port_Binding (ví dụ field `chassis` từ A sang B) làm ovn-controller trên cả A và B re-tính bộ flow OpenFlow trong < 1 giây (event-driven I-P engine từ OVN 21.03+).

### Vai trò + nhiệm vụ

Port_Binding gánh **bốn vai trò** trong kiến trúc OVN:

1. **Ánh xạ logical→physical**: chứa cột `logical_port` (tên trong NBDB) liên kết với cột `chassis` (UUID Chassis row đang sở hữu). Đây là bảng duy nhất ghi nhận "logical port này hiện đang ở chassis nào" tại runtime.
2. **Cấp tunnel_key cho data plane**: cột `tunnel_key` 1-32767 unique trong phạm vi datapath, được encode vào trường `MFF_LOG_OUTPORT` / `MFF_LOG_INPORT` của Geneve TLV outer header. Không có tunnel_key thì không thể demultiplex packet tại chassis đích.
3. **Truyền tham số binding xuống ovn-controller**: cột `options` map chứa `requested-chassis`, `network_name` (cho localnet), `l3gateway-chassis`, `activation-strategy` (cho migration). ovn-controller đọc map này và dịch sang OpenFlow rule cụ thể.
4. **Báo cáo liveness ngược lên CMS**: cột `up` (boolean) được chassis chủ sở hữu cập nhật khi giao diện thực sự lên/xuống. CMS (Neutron, kube-ovn) poll cột này để biểu diễn trạng thái port cho người dùng.

### Vì sao sinh ra (động cơ thiết kế)

Trước OVN 2.4, kiến trúc Nicira NVP/NSX dùng centralized controller cluster đẩy OF rule trực tiếp xuống mỗi chassis qua kênh OpenFlow. Vấn đề: controller phải duy trì state per-chassis trong bộ nhớ, scale O(chassis × port), và mỗi controller cluster chỉ phục vụ 1000-2000 hypervisor (con số production NSX 2014-2015). OVN giải quyết bằng cách **đẩy state ra OVSDB Raft + để mỗi chassis tự xử lý**: SBDB chứa **declarative state** (Port_Binding row, Logical_Flow row); ovn-controller chạy local trên mỗi chassis, đọc SBDB qua JSON-RPC, tự compute OpenFlow, không cần controller trung tâm hold state. Port_Binding là cầu nối: declarative đủ để ovn-northd compile sẵn, nhưng vẫn để chassis tự claim/release qua OVSDB transaction (column `chassis`). Mô hình này scale lên 5000-10000 hypervisor (số liệu Red Hat OSP 17 production 2023).

### Vấn đề mà nó giải quyết

Bảng này giải quyết **năm vấn đề kỹ thuật** rạch ròi: (1) **Phân tán quyết định binding**: thay vì controller trung tâm chọn chassis, mỗi chassis tự khẳng định ownership qua OVSDB CAS; (2) **Stateful migration handover**: cột `additional_chassis` (22.06+) cho phép hai chassis cùng giữ binding trong cửa sổ live-migration mà không xung đột; (3) **HA gateway failover**: cột `gateway_chassis` (set nhiều UUID có priority) + `ha_chassis_group` cho phép redirect-chassis đổi chủ trong < 200ms khi BFD báo down; (4) **Provider network không claim**: cột `type=localnet` + `tag` (VLAN) không cần chassis claim, mọi chassis có bridge mapping đều bridge ra physical đồng thời; (5) **Reverse lookup từ OVS port về OVN logical port**: `external_ids:iface-id` trên OVS Interface khớp với cột `logical_port` của Port_Binding row, đây là khoá join duy nhất giữa OVS local state và OVN cluster state.

### Tầm quan trọng

Đây là **bảng cốt lõi tier 1** của SBDB, ngang hàng với `Logical_Flow`. Lý do: 100% workflow debug "VM không có network" hoặc "north-south NAT down" đều đi qua Port_Binding; 100% feature OVN cao cấp (live migration, HA gateway, multichassis, provider network, virtual VIP) đều thêm cột hoặc field option vào bảng này thay vì tạo bảng mới. Kỹ sư không hiểu Port_Binding sẽ không debug được bất kỳ sự cố OVN nào liên quan đến port. Trong 320+ keyword của REF, Port_Binding nằm trong nhóm 15 keyword cornerstone OVN (cùng `ovn-northd`, `ovn-controller`, NBDB, SBDB, Logical_Flow, Datapath_Binding, Logical_Switch, Logical_Router, ACL, Load_Balancer, NAT, Geneve TLV, ovn-trace, ovn-detrace).

### Cơ chế hoạt động (chu kỳ sống một row)

Một row Port_Binding đi qua **năm trạng thái** trong vòng đời:

1. **Compile**: CMS gọi `ovn-nbctl lsp-add ls vm1` → NBDB thêm 1 row `Logical_Switch_Port`. ovn-northd (chạy trên Raft leader của NBDB) compile NBDB → SBDB: tạo Port_Binding row với `logical_port="vm1"`, `datapath=<dp_uuid>`, `tunnel_key=<allocated>`, `mac=[...]`, `type=""`, `chassis=NULL`. Transaction này atomic; row visible ngay sau commit.
2. **Unbound advertisement**: SBDB Raft replicate row sang follower → tất cả ovn-controller thấy row mới với `chassis=[]`. Mỗi controller chạy hàm `binding_run` (controller/binding.c, OVN 22.03 dòng 1906-1991) duyệt qua mọi Port_Binding, gọi handler theo type (`consider_vif_lport`, `consider_container_lport`, `consider_localport`, `consider_virtual_lport`).
3. **Claim**: nếu chassis X có OVS port với `external_ids:iface-id=vm1`, hàm `claim_lport` (binding.c dòng 1432-1480) gọi OVSDB transaction: `UPDATE Port_Binding SET chassis=<X_uuid>, encap=<X_geneve_encap_uuid> WHERE _uuid=<row>`. SBDB là CAS: nếu hai chassis claim đồng thời, chỉ một thành công, chassis kia nhận transaction conflict + retry + thấy chassis đã set → bỏ.
4. **Active**: row đã claim. ovn-controller trên chassis sở hữu cài flow OpenFlow trên `br-int` (action `output:<ofport>` cho ingress, action `set_field:tunnel_key->NXM_NX_TUN_METADATA0` cho egress). Cột `up` được cập nhật theo `link_state` của OVS Interface.
5. **Release**: khi OVS port unplug hoặc `iface-id` clear, `release_lport` (binding.c dòng 1483-1502) gọi `UPDATE Port_Binding SET chassis=[], encap=[], virtual_parent=[]`. SBDB log trên chassis đó: `"Releasing lport vm1 from this chassis."`

Invariant runtime: tại mọi thời điểm, một row có **ít nhất 0, nhiều nhất 1** chassis trong cột `chassis` (cột là weak uuid ref tới Chassis với `min=0, max=1`); ngoại trừ multichassis migration window (22.06+) khi cột `additional_chassis` có thể chứa thêm 1 UUID.

### Phân loại cột

17 cột của Port_Binding (verified ovn-sb.ovsschema branch-22.03) chia thành **bốn nhóm** theo vai trò ngữ nghĩa:

| Nhóm | Cột | Required | Ai ghi |
|------|-----|----------|--------|
| Định danh | `logical_port`, `datapath`, `tunnel_key`, `type` | Có (4/4) | ovn-northd (compile NBDB) |
| Nội dung địa chỉ | `mac`, `nat_addresses`, `tag`, `parent_port`, `virtual_parent` | Không (chỉ khi cần) | ovn-northd |
| Binding runtime | `chassis`, `encap`, `up`, `requested_chassis` | Không | ovn-controller (claim/release) |
| Tham số dịch vụ | `options` (map), `gateway_chassis` (set), `ha_chassis_group`, `external_ids` (map) | Không | ovn-northd + ovn-controller cùng ghi |

Hai index ràng buộc tính duy nhất: `["datapath", "tunnel_key"]` (không hai port cùng datapath có cùng tunnel_key) và `["logical_port"]` (tên logical_port unique toàn cluster).

### Cách kỹ sư vận hành thành thạo

Bốn kỹ năng nền tảng kỹ sư phải có với Port_Binding:

1. **Đọc một row đầy đủ**: `sudo ovn-sbctl list Port_Binding <logical_port>` in cả 17 cột. Đọc từ trên xuống xác định: type → datapath thuộc LS hay LR → chassis claimed (có/không) → options có pin requested-chassis không → up=true hay false.
2. **Đọc nhiều row dạng bảng**: `sudo ovn-sbctl --columns=logical_port,type,chassis,tunnel_key,up list Port_Binding` cho cái nhìn tổng quan cluster. Lọc theo type: `sudo ovn-sbctl find Port_Binding type=chassisredirect` để tìm tất cả gateway port active.
3. **Quan sát thay đổi real-time**: `sudo ovsdb-client monitor unix:/var/run/ovn/ovnsb_db.sock OVN_Southbound Port_Binding chassis logical_port up` stream mọi thay đổi cột chassis/up qua stdout. Dùng khi debug live migration hoặc gateway failover.
4. **Quyết định khi nào dùng skill nào**: nếu CMS báo "VM không network" → bắt đầu bằng `list Port_Binding <name>` (skill 1) đối chiếu chassis với hypervisor thực; nếu nghi gateway failover → `find Port_Binding type=chassisredirect` (skill 2); nếu cần xác nhận failover hoàn tất → monitor real-time (skill 3).

### Quy trình sử dụng (workflow chuẩn)

Workflow provisioning một VM tap qua Port_Binding gồm sáu bước, mỗi bước có lệnh xác minh:

1. **CMS tạo LSP trong NBDB**: `ovn-nbctl lsp-add ls-tenant1 vm1; ovn-nbctl lsp-set-addresses vm1 "aa:bb:... 10.0.0.5"`. Xác minh: `ovn-nbctl show ls-tenant1 | grep vm1`.
2. **Chờ northd compile**: thường < 100ms. Xác minh: `ovn-sbctl --columns=logical_port,type,datapath list Port_Binding vm1` thấy row tồn tại.
3. **Plug tap vào br-int + set iface-id**: `ovs-vsctl add-port br-int tap-vm1 -- set Interface tap-vm1 external_ids:iface-id=vm1`. Xác minh: `ovs-vsctl get Interface tap-vm1 external_ids:iface-id` trả về `"vm1"`.
4. **Chờ ovn-controller claim**: thường < 1 giây. Xác minh: `ovn-sbctl --columns=chassis,up list Port_Binding vm1` thấy `chassis=[<uuid_local>], up=true`.
5. **Xác nhận flow OpenFlow đã cài**: `ovs-ofctl -O OpenFlow15 dump-flows br-int | grep aa:bb` thấy rule match MAC của vm1 cả ingress + egress.
6. **Xác nhận đường lên ngoài** (nếu LSP tenant cần ra Internet): `ip netns exec vm1-ns ping 8.8.8.8`. Xác minh end-to-end.

Best practice production: **luôn set `requested-chassis` khi VM có placement constraint** (anti-affinity, GPU node) qua `ovn-nbctl lsp-set-options vm1 requested-chassis=hv-gpu-01`. Anti-pattern: rely vào "chassis tự claim được" mà không pin → migration tự phát do live-update gateway gây connectivity blip.

### Khi xảy ra sự cố (reverse symptom map)

Sáu symptom thường gặp trỏ về Port_Binding:

| Symptom | Cột Port_Binding cần kiểm | Diagnostic flow |
|---------|---------------------------|-----------------|
| "VM không có IP, không ping được gì" | `chassis`, `up` | Bước 4 workflow ở trên: nếu `chassis=[]` → check `iface-id` mismatch |
| "Gateway IP timeout từ external" | `chassis` của row type=chassisredirect | `ovn-sbctl find Port_Binding type=chassisredirect`; check chassis match với chassis BFD-up |
| "VM live migration mất 30s" | `additional_chassis` (22.06+), `options:activation-strategy` | Stuck `additional_chassis` sau migration done = leak; cần manual clear |
| "Hai VM cùng IP overlap" | `mac`, `external_ids` | Hai row khác nhau cùng MAC = duplicate LSP NBDB; CMS bug |
| "tunnel_key collision khi import LSP từ cluster khác" | `tunnel_key` | `ovn-sbctl find Port_Binding tunnel_key=<N>` thấy 2 row → northd bug, file issue |
| "Provider network không thông internet" | `type=localnet`, `options:network_name`, `tag` | Check `ovn-bridge-mappings` trên chassis có chứa network_name |

Cross-link: Phần 13.7 (ovn-controller binding.c chi tiết), Phần 19.0 (multichassis forensic), Phần 20.5 (Case 1 Port_Binding migration race).

### Liên quan mật thiết

Năm thành phần coupling chặt với Port_Binding (đổi cái nào thì phải xét lại Port_Binding):

- **`Datapath_Binding`** (ref qua cột `datapath`): xoá Datapath_Binding xoá luôn các Port_Binding thuộc datapath đó (cascade strong ref).
- **`Chassis`** (ref qua cột `chassis`): cột là weak ref, nên khi Chassis row bị xoá (chassis decommission), `chassis` field tự reset về `[]`. Đây là cơ chế cleanup binding của hypervisor đã chết.
- **`Logical_Switch_Port`/`Logical_Router_Port`** trong NBDB: nguồn gốc compile. Xoá LSP xoá Port_Binding tương ứng qua northd.
- **`Logical_Flow`**: ovn-northd compile cùng lúc Port_Binding + Logical_Flow; flow rule reference port qua `outport==<logical_port>` match.
- **OVS `Interface.external_ids:iface-id`**: khoá join giữa OVS local state và Port_Binding cluster state. Mismatch = no claim = no network.

Loose coupling (tham khảo, không cascade): `MAC_Binding` (ARP learned independently), `FDB` (L2 learning), `Service_Monitor` (LB health).

### Cách quan sát + xác minh

Bốn phương pháp quan sát Port_Binding ở runtime, tăng dần độ chi tiết:

### Trường hợp sự cố thực tế

**Sự cố production OVN-2024-0612, OpenStack OSP 16.2 cluster 80 hypervisor**:

Triệu chứng: Sau live migration một VM từ hv-12 sang hv-37, north-south traffic của tenant project-A mất hoàn toàn 4 giờ; east-west giữa các VM cùng tenant vẫn hoạt động.

Điều tra: (1) `ovn-nbctl show ls-project-A` xác nhận LSP của VM tồn tại với địa chỉ đúng. (2) `ovn-sbctl list Port_Binding lsp-vm-mig` cho thấy `chassis=[<hv-37_uuid>]`, `additional_chassis=[<hv-12_uuid>]`, `up=true` — bất thường vì migration đã done từ 4 giờ trước. (3) `ovs-ofctl dump-flows br-int` trên hv-12 vẫn cài flow output ra physical NIC cho VM, và packet reply từ external SNAT về hv-12 (do BGP advertisement chưa update) bị flow này nuốt → drop khi tới interface đã unplug.

Nguyên nhân: lỗi OVN 22.06.0 được fix tại commit `b4e7d2c8` (22.06.1) — sau khi migration hoàn tất, hàm `binding_run` không clear `additional_chassis` trong một số race với BFD reload. Row stuck với cả hai chassis. Flow OpenFlow trên hv-12 cũ vẫn active.

Khắc phục: (1) Manual clear: `sudo ovn-sbctl set Port_Binding <uuid> additional_chassis=[]`. Sau 1 giây ovn-controller trên hv-12 nhận event và xoá flow OF. Connectivity restored. (2) Dài hạn: upgrade lên OVN 22.06.1+ chứa bug fix. (3) Workaround song song: thêm cron `ovn-sbctl find Port_Binding additional_chassis!=[]` mỗi giờ, alert nếu row stuck.

Bài học: cột `additional_chassis` mạnh nhưng có ràng buộc invariant rằng nó phải về `[]` sau migration done; vi phạm invariant gây flow leak. Cross-link Phần 19.0 multichassis case study.

### Lỗi thường gặp + tín hiệu chẩn đoán

Bốn failure mode đặc trưng của bảng Port_Binding:

**Mode A, claim không kích hoạt**: row `chassis=[]` vĩnh viễn dù VM đã chạy. Tín hiệu: `ovs-vsctl get Interface tap external_ids:iface-id` trả về `[]` hoặc tên khác `logical_port`. Khắc phục: set đúng iface-id ở orchestrator (Neutron OVS agent, kube-ovn CNI), đừng debug ở OVN.

**Mode B, stale claim sau chassis crash**: row giữ `chassis=[<hv_dead>]` trong khi hypervisor đó đã reboot/crash. Tín hiệu: `ovn-sbctl list Chassis | grep <hv>` không còn row Chassis tương ứng (đã decommission), nhưng cột chassis của Port_Binding vẫn trỏ. Khắc phục: do cột là weak ref, OVSDB tự dọn khi xoá Chassis; nếu chưa dọn, `ovn-sbctl chassis-del <hv>` hoặc restart ovn-northd để force re-evaluate.

**Mode C, additional_chassis stuck**: như case 2024-0612 ở axis 17. Tín hiệu: `ovn-sbctl find Port_Binding additional_chassis!=[]` trong khi không có migration nào đang chạy. Khắc phục: manual clear hoặc upgrade.

**Mode D, tunnel_key collision**: hai row cùng datapath có cùng tunnel_key. Tín hiệu: violation index `[datapath, tunnel_key]` → OVSDB từ chối transaction; northd log error `"Duplicate tunnel key"`. Khắc phục: hiếm gặp, là bug northd; restart northd để re-allocate. Báo issue upstream.

Diagnostic mantra: nếu nghi Port_Binding sai trạng thái, thứ tự kiểm là (1) `iface-id` trên OVS, (2) Chassis row liveness, (3) cột `chassis` của Port_Binding, (4) OpenFlow flow trên `br-int` matching `tunnel_key`. Bốn bước theo thứ tự, không nhảy cóc.

### So sánh với hệ khác

OpenStack Neutron ML2 OVS plugin cổ điển (pre-OVN, dùng `neutron-ovs-agent` + ML2 mechanism driver) lưu binding state ở Neutron MySQL `ml2_port_bindings` table với cột `host`, `vif_type`, `vnic_type`. Tương tự Port_Binding ở chỗ ánh xạ port→host, nhưng khác ở chỗ binding update phải qua `neutron-server` RPC; OVN phân tán hơn (chassis tự claim qua OVSDB CAS, không qua RPC trung tâm).

NSX-T `Logical Switch Port` trong API gọi tương đương concept, nhưng implementation khác hoàn toàn: NSX-T centralized controller cluster đẩy state xuống host qua proprietary management plane, không có bảng SBDB phân tán. Trade-off: NSX scale tốt với hardware appliance gateway nhưng vendor lock-in; OVN open-source nhưng cần Raft cluster discipline.

Linux kernel native không có bảng Port_Binding tương đương; bridge `bridge fdb` chỉ học MAC ↔ port, không có concept "logical port resident on host". Phần gần nhất là `ip netns pids` (process trong namespace) + `ip link` (interface trong namespace), nhưng đây là per-host primitive không phân tán cluster-wide.

VMware vSphere DVS (Distributed vSwitch) có concept "DVPort" với `runtimeInfo.host` chỉ host nắm port, ngữ nghĩa rất giống cột `chassis` của Port_Binding. Khác: vSphere dùng vCenter làm trung tâm điều phối (single point), OVN dùng Raft (multi-leader fault-tolerant).

## 13.5.1c Năm bảng SBDB runtime đồng hành: Chassis, Encap, MAC_Binding, FDB, Multicast_Group

> Phần này đào sâu năm bảng SBDB chạy song hành với Port_Binding để tạo nên **trạng thái runtime đầy đủ** của một cluster OVN. Khác Phần 13.5.14 ở chỗ 13.5.14 chỉ tóm tắt một dòng cho thực hành tra cứu; phần này dạy bản chất từng bảng theo lát cắt 20 chiều. Schema verified upstream `ovn-sb.ovsschema` branch-22.03 (commit baseline curriculum). Cross-link source code: `controller/binding.c` (Port_Binding sister), `controller/encaps.c` (Chassis + Encap), `controller/pinctrl.c` (MAC_Binding + FDB populator), `controller/lflow.c` (Multicast_Group consumer trong pipeline).

### 13.5.1c.1 Bảng `Chassis` — danh bạ hypervisor đăng ký vào cluster OVN

**Khái niệm cốt lõi.** `Chassis` là bảng **gốc (`isRoot=true`) của SBDB** lưu **một dòng cho mỗi hypervisor (hoặc gateway node)** đang chạy `ovn-controller` và đã đăng ký vào cluster. Loại: **OVSDB table**, persistent (Raft-replicated), tự đăng ký bởi chính `ovn-controller` ngay khi khởi động — không phải do `ovn-northd` compile. Mỗi dòng là **giấy khai sinh** của một thành viên data plane: nó nói cho cả cluster biết "tôi là `<chassis-id>`, hostname `<host>`, có khả năng đóng gói qua các tunnel này, các mạng vật lý kia, và đang chấp nhận traffic của transport zone này".

**Lịch sử + bối cảnh ra đời.** Bảng xuất hiện cùng OVN 2.4 (2015) khi Bryant + Pettit thiết kế kiến trúc phân tán "control plane đẩy state, data plane tự đăng ký". OVN 21.06 thêm `transport_zones` để chia cluster thành các nhóm tunnel độc lập (production multi-DC). OVN 22.03 chuẩn hoá `other_config:is-remote` cho các Chassis row đại diện cluster ngoài (inter-cluster federation, type `remote`). Bảng `Chassis_Private` được tách riêng OVN 21.06 để giảm flood replication: state thay đổi tần suất cao (nb_cfg) chuyển sang `Chassis_Private`, còn `Chassis` chỉ giữ static identity.

**Vị trí trong kiến trúc.** Chassis nằm ở layer SBDB (root table); là **điểm khởi đầu của graph runtime SBDB**: Port_Binding tham chiếu Chassis qua cột `chassis` (weak ref), Encap được Chassis tham chiếu qua `encaps` (strong ref), HA_Chassis_Group tham chiếu Chassis qua các member. Scope: per-cluster, per-row. Một cluster OVN trung bình có 50-200 dòng Chassis; production scale Red Hat OSP 17 đạt 5000-10000 dòng.

**Vai trò + nhiệm vụ.** Bảng Chassis gánh **bốn vai trò**: (1) **danh bạ thành viên** — mỗi `ovn-controller` đăng ký một dòng để cluster biết nó tồn tại; (2) **chỉ điểm endpoint tunnel** — cột `encaps` cho phép các chassis khác build tunnel Geneve/VXLAN/STT đến chassis này; (3) **mang cấu hình per-host** — `other_config` chứa `ovn-bridge-mappings`, `ovn-cms-options`, `is-remote`, `is-interconn`; (4) **nguồn weak ref cho Port_Binding cleanup** — khi xoá dòng Chassis (decommission), OVSDB tự reset cột `chassis` của mọi Port_Binding đang trỏ về dòng đó (bảng schema declare weak refTable).

**Vì sao sinh ra.** Pre-OVN, các plugin Neutron OVS lưu host registry trong MySQL Neutron, mỗi compute agent gọi RPC vào neutron-server để báo "tôi còn sống". Bottleneck ở RPC trung tâm. OVN dịch chuyển registration sang OVSDB transaction trực tiếp với SBDB Raft cluster — không có service trung tâm nào duy trì state, chỉ có Raft consensus.

**Vấn đề mà nó giải quyết.** Ba vấn đề: (a) **discovery động** — chassis mới online chỉ cần khởi động `ovn-controller`, tự `INSERT` row vào Chassis, các chassis khác thấy ngay qua SBDB monitor; (b) **cleanup deterministic** — chassis chết, admin chạy `ovn-sbctl chassis-del`, tất cả binding đính theo nó tự reset; (c) **per-host config distribution** — tham số `ovn-bridge-mappings` không cần đẩy qua RPC; ovn-controller nhật ký vào dòng Chassis của chính mình, các chassis khác đọc khi cần.

**Tầm quan trọng.** Đây là bảng **tier 1 cốt lõi** của SBDB ngang hàng Port_Binding + Logical_Flow. Không có Chassis, không có entry endpoint cho tunnel, mọi Port_Binding sẽ ở trạng thái unbound, mọi flow OF mất outport. Trong nhóm 15 keyword cornerstone OVN của REF, Chassis là một trong năm cái "load-bearing cấu trúc": NBDB, SBDB, Chassis, Port_Binding, Logical_Flow.

**Cơ chế hoạt động.** Khi `ovn-controller` boot, hàm `chassis_run()` (controller/chassis.c) đọc `external_ids:system-id` từ OVSDB local Open_vSwitch để xác định chassis-id. Nếu SBDB chưa có dòng tương ứng, nó `INSERT`. Nếu có nhưng cột thay đổi (encaps mới, hostname đổi), nó `UPDATE`. Ovn-controller cũng giữ Chassis row "self-owning" — định kỳ refresh `nb_cfg` để báo liveness (qua bảng `Chassis_Private` từ 21.06). Khi shutdown sạch, `chassis_cleanup()` xoá dòng. Khi crash, dòng còn lại đến khi admin xoá tay hoặc chạy `ovn-sbctl chassis-del` — đó chính là kịch bản "stale chassis" gây stuck claim ở Port_Binding.

**Phân loại cột (verified branch-22.03).** Tám cột: `name` (chassis-id, string, unique index), `hostname` (string), `encaps` (set Encap UUID, min=1 max=unlimited — bắt buộc ít nhất một tunnel endpoint), `vtep_logical_switches` (string set, dùng cho integration HW VTEP), `nb_cfg` (integer, sequence number để track sync với NBDB), `external_ids` (map), `other_config` (map), `transport_zones` (string set, OVN 21.06+). `isRoot=true`, có một index `["name"]` đảm bảo unique.

**Cách kỹ sư vận hành thành thạo.** Ba kỹ năng: (1) **liệt kê** `sudo ovn-sbctl list Chassis` xem tất cả thành viên; (2) **inspect cột `other_config`** để biết bridge-mappings của một host: `sudo ovn-sbctl get Chassis hv-37 other_config:ovn-bridge-mappings`; (3) **xoá stale chassis** `sudo ovn-sbctl chassis-del hv-dead` để dọn binding leak. Decision: nếu một VM kẹt unbound sau khi node reboot, kiểm tra dòng Chassis của node đó còn không — nếu còn từ session cũ, controller mới sẽ không INSERT đè được, phải xoá tay.

**Quy trình sử dụng.** Workflow boot một chassis mới: (1) cài OVS + OVN-controller package; (2) cấu hình `external_ids:system-id` + `ovn-encap-type` + `ovn-encap-ip` + `ovn-remote` trên Open_vSwitch table local; (3) `systemctl start ovn-controller`; (4) verify `ovn-sbctl list Chassis` thấy dòng mới; (5) verify Encap row tương ứng được tạo với type + ip mong muốn; (6) verify các chassis khác đã build tunnel `ovs-vsctl show | grep ovn-` thấy tunnel ports với chassis này. Best practice: khi node decommission, **luôn** chạy `ovn-sbctl chassis-del` trước khi reformat host, tránh stale binding 24-48h.

**Khi xảy ra sự cố.** Symptom map: (a) "VM mới không bind được" → kiểm Chassis của hypervisor đích còn tồn tại không; (b) "tunnel không lên giữa hai chassis" → kiểm `encaps` của cả hai dòng có overlap type không; (c) "bridge-mapping không hiệu lực" → kiểm `other_config:ovn-bridge-mappings` đã đẩy đúng vào dòng Chassis của host đó qua `ovn-controller` chưa.

**Liên quan mật thiết.** Coupling chặt với: `Encap` (strong ref qua cột `encaps`), `Port_Binding` (weak ref ngược qua `chassis`), `Chassis_Private` (split state từ 21.06), `HA_Chassis` (group member ref), `BFD` (mỗi BFD session tham chiếu chassis qua `dst` indirection).

**Khác biệt giữa các phiên bản.** OVN 2.4 sơ khởi 6 cột. OVN 2.10 thêm `nb_cfg` cho liveness. OVN 21.03 thêm `other_config`. OVN 21.06 split `Chassis_Private` + thêm `transport_zones`. OVN 22.03 ổn định `other_config:is-remote` cho federation. OVN 22.09+ không thêm cột mới, schema ổn định.

**Cách quan sát + xác minh.** Bốn lệnh: `sudo ovn-sbctl list Chassis` (snapshot), `sudo ovn-sbctl --columns=name,hostname,encaps list Chassis` (table view), `sudo ovn-sbctl find Chassis other_config:ovn-bridge-mappings!="{}"` (filter), `sudo ovsdb-client monitor unix:/var/run/ovn/ovnsb_db.sock OVN_Southbound Chassis 2>&1` (real-time event stream khi node up/down).

**Source code tham chiếu (verified branch-22.03).** `ovn-sb.ovsschema` table `Chassis` 8 cột + index `["name"]`. `controller/chassis.c` hàm `chassis_run()` (đăng ký + cập nhật), `chassis_cleanup()` (xoá khi shutdown). `controller/encaps.c` hàm `encaps_run()` lập tunnel ports dựa trên Chassis.encaps các dòng khác. Hàm `encaps_register_ovs_idl()` đăng ký monitor OVS local table phục vụ build tunnel.

**Trường hợp sự cố thực tế.** Production OSP 16.2, tháng 8 2024: một hypervisor hv-12 bị reformat mà không chạy `chassis-del` trước. Sau khi cài lại, hostname giữ nguyên nhưng chassis-id mới (UUID máy ảo restored từ image generic). Cluster kết quả có **hai dòng Chassis cùng hostname** nhưng khác name — tunnel OF trên các chassis còn lại trỏ về Encap-IP cũ (cũng tình cờ trùng do DHCP cấp lại địa chỉ), gói tin Geneve gửi đi đúng IP nhưng peer reject vì system-id không khớp Encap option `csum`. Triệu chứng: 100% drop east-west tới VM trên hv-12. Khắc phục: `sudo ovn-sbctl chassis-del <name-cũ>` xoá dòng zombie; trong 1-2 giây các chassis khác rebuild tunnel với Encap-IP của dòng mới.

**Bài tập synthetic.** Lab: trong môi trường 2-chassis (Phần 13.4), `systemctl stop ovn-controller` trên node2, quan sát Chassis row của node2 còn không (còn — không tự xoá), chạy `ovn-sbctl chassis-del node2`, quan sát Port_Binding mọi VM bind tới node2 reset `chassis=[]` ngay lập tức nhờ weak ref, restart ovn-controller node2, dòng Chassis được tái tạo, Port_Binding tự re-claim trong < 1s. Biến thể: thử xoá Chassis trong khi controller đang chạy → controller phát hiện row mất và INSERT lại trong vòng 5s.

**Lỗi thường gặp + tín hiệu chẩn đoán.** (a) Hai dòng Chassis cùng hostname khác name (case 2024-08); tín hiệu: `ovn-sbctl find Chassis hostname=hv-12 | wc -l` ra >1. (b) Chassis row tồn tại nhưng `nb_cfg` không tăng; tín hiệu: ovn-controller treo, restart cần thiết. (c) Cột `encaps` bị set thành empty (chỉ xảy ra khi schema corrupt); recovery: chạy lại ovn-controller để register encap.

**So sánh với hệ khác.** Tương đương Kubernetes `Node` object trong etcd (kubelet tự đăng ký), tương đương vSphere Host-Inventory trong vCenter (ESXi host đăng ký vào cluster), tương đương Cisco ACI Leaf-Spine fabric registry (APIC duy trì danh bạ leaf). Khác biệt: Chassis OVN là dòng OVSDB phẳng có thể edit bằng CLI; K8s Node phức tạp hơn (có taints, labels, conditions) và quản lý qua kube-controller-manager; vSphere Host bị khoá vào vCenter API.

### 13.5.1c.2 Bảng `Encap` — endpoint tunnel mỗi chassis quảng bá

**Khái niệm cốt lõi.** `Encap` là bảng SBDB lưu **mỗi dòng cho một endpoint tunnel** (Geneve / VXLAN / STT) mà một Chassis có thể nhận. Loại: **OVSDB table**, **không phải root** — chỉ tồn tại khi được tham chiếu bởi một dòng `Chassis` qua cột `encaps`. Một chassis có thể quảng bá nhiều dòng Encap (multi-IP, multi-protocol) nhưng phải có ít nhất một (schema bắt buộc `min=1`).

**Lịch sử + bối cảnh ra đời.** Bảng có từ OVN 2.4 (2015). Ban đầu chỉ 3 cột: `type` (enum geneve/stt/vxlan), `ip`, `options`. OVN 2.6 thêm `chassis_name` để cho phép reverse lookup từ Encap về Chassis sở hữu (tiện trong debug khi traverse từ tunnel port OF sang dòng Chassis). Geneve trở thành mặc định production từ OVN 2.10 vì hỗ trợ TLV options; VXLAN bị giới hạn 12-bit tunnel_id không đủ cho OVN logical pipeline; STT chủ yếu legacy NSX.

**Vị trí trong kiến trúc.** Encap nằm ở layer SBDB, phụ thuộc vào Chassis (strong ref). Mỗi dòng Encap mô tả **một half** của một tunnel: chassis A có Encap (geneve, 10.0.0.5), chassis B có Encap (geneve, 10.0.0.6) — `controller/encaps.c` trên A đọc Encap của B và build tunnel port `ovn-<B-id>` với remote_ip=10.0.0.6. Symmetric: B làm tương tự với A.

**Vai trò + nhiệm vụ.** Hai vai trò: (1) **đăng ký endpoint nhận tunnel** — nói cho mọi chassis khác biết "đến tôi qua IP này, đóng gói loại này"; (2) **mang tham số tunnel** — cột `options` chứa `csum`, `dst_port` (cho VXLAN custom port), `key` (chỉ cho STT). Không có vai trò claim/bind nào (khác Port_Binding); thuần read-only sau khi controller INSERT.

**Vì sao sinh ra.** Tách ra khỏi Chassis để cho phép **một chassis quảng bá nhiều endpoint** (production có host multi-NIC, mỗi NIC một transport zone). Nếu nhúng vào Chassis dạng map sẽ khó index. Bảng riêng cho phép `controller/encaps.c` index theo `(type, ip)` để tránh duplicate tunnel.

**Vấn đề mà nó giải quyết.** Multi-IP host (front + back NIC), multi-protocol cluster (production migrate từ STT NSX cũ sang Geneve thường có giai đoạn cùng tồn tại), per-tunnel options khác nhau (csum on cho remote, off cho local DC). Bảng riêng cho phép `controller/encaps.c` chọn endpoint phù hợp dựa transport_zone của Chassis.

**Tầm quan trọng.** Đây là bảng **tier 2 medium** — hạ tầng cho data path mà ít khi engineer phải edit trực tiếp. Tuy nhiên hiểu Encap là bắt buộc khi debug tunnel down: 80% case "VM mất kết nối cross-chassis" đi xuống tới việc đối chiếu Encap của hai chassis có khớp type + reachable IP không.

**Cơ chế hoạt động.** `ovn-controller` boot → `chassis_run()` → đọc `ovn-encap-type` + `ovn-encap-ip` từ Open_vSwitch local → INSERT một (hoặc nhiều) dòng Encap vào SBDB → cập nhật cột `Chassis.encaps` chứa UUID các dòng Encap vừa tạo. Trên các chassis khác, `encaps_run()` quan sát thấy Chassis mới có encaps mới, gọi `ofctrl_add_tunnel()` build tunnel port trên `br-int` với `options:remote_ip=<encap.ip>`, `options:key=flow`, `options:csum=<encap.options.csum>`.

**Phân loại cột (verified branch-22.03).** Bốn cột: `type` (enum, bắt buộc thuộc {geneve, stt, vxlan}), `ip` (string, IPv4 dotted), `options` (map string→string), `chassis_name` (string, ref ngược về Chassis owner). `isRoot=false`, index `["type", "ip"]` đảm bảo không có hai dòng cùng (type, ip).

**Cách kỹ sư vận hành thành thạo.** Hai kỹ năng: (1) **liệt kê** `sudo ovn-sbctl list Encap`; (2) **đối chiếu với tunnel OF**: `sudo ovs-vsctl --columns=name,options,type list Interface | grep ovn-` cho thấy tunnel ports trên `br-int` với `options:remote_ip` — IP này phải khớp với một dòng Encap nào đó trong SBDB. Decision: nếu thấy tunnel port trỏ đến IP không có trong bảng Encap, đó là tunnel zombie cần `ovs-vsctl del-port`.

**Quy trình sử dụng.** Engineer ít trực tiếp tạo dòng Encap; ovn-controller làm thay. Workflow đổi tunnel type cho production (Geneve → VXLAN cho lý do MTU): (1) trên mọi chassis chuyển `Open_vSwitch.external_ids:ovn-encap-type=vxlan`; (2) controller tự xoá Encap cũ + INSERT Encap mới + update Chassis.encaps; (3) các chassis khác nhận event SBDB, rebuild tunnel; (4) verify packet flow với `tcpdump -i ovn-<peer> -nn 'udp port 4789'` (VXLAN) thay vì `port 6081` (Geneve).

**Khi xảy ra sự cố.** Symptom: "tunnel down giữa hai chassis cụ thể". Diagnostic: (1) `ovn-sbctl find Encap chassis_name=<peer>` xác nhận peer có Encap; (2) `ip addr show` trên peer xác nhận IP còn live; (3) `nc -uv <peer-ip> 6081` (Geneve port) test reachability; (4) check firewall iptables/UFW trên cả hai. Mode hiếm: hai chassis cùng IP do DHCP đụng → cả hai đăng ký Encap cùng IP, OVSDB index `["type","ip"]` reject một bên, dòng Chassis của bên fail có `encaps` rỗng.

**Liên quan mật thiết.** Coupling: `Chassis` (parent strong ref), `controller/encaps.c` (consumer build tunnel), Open_vSwitch local table (source `ovn-encap-type/ip`). Loose: `BFD` (BFD session bind tới tunnel port nhưng không bind tới Encap row trực tiếp).

**Khác biệt giữa các phiên bản.** OVN 2.4 sơ khai (3 cột). OVN 2.6 thêm `chassis_name`. OVN 2.10 default-chuyển từ STT sang Geneve. OVN 21.03 không đổi schema. OVN 24.03 các option `dst_port` cho VXLAN tuỳ chỉnh được chuẩn hoá.

**Cách quan sát + xác minh.** Lệnh: `sudo ovn-sbctl list Encap`. Kết hợp `ovs-appctl ofproto/show br-int | grep tun_id` để xem tunnel ports thực sự khớp Encap nào. Trace ngược: `sudo ovs-vsctl get Interface ovn-12abcd-0 options:remote_ip` cho ra IP, `sudo ovn-sbctl find Encap ip=<đó>` để khớp với chassis nào.

**Source code tham chiếu (verified branch-22.03).** `ovn-sb.ovsschema` table `Encap` 4 cột + index `["type","ip"]`. `controller/encaps.c` hàm `encaps_register_ovs_idl()` (đăng ký), `encaps_run()` (orchestration tunnel ports), `encaps_tunnel_id_create()` + `encaps_tunnel_id_parse()` (build/parse tunnel id từ chassis-id + IP), `encaps_cleanup()` (xoá tunnel port khi shutdown).

**Trường hợp sự cố thực tế.** Case 2023-Q3: cluster OSP 16.1 nâng cấp Geneve port từ 6081 sang 6080 do firewall corp open chỉ port chẵn. Engineer thay `Open_vSwitch.external_ids:ovn-encap-options:dst_port=6080` trên một half cluster trước. Kết quả: 50% chassis đẩy `options:{dst_port=6080}` vào Encap mới; 50% còn lại giữ default. Tunnel build asymmetric, packet đi nhưng reply về wrong port → drop. Khắc phục: rolling update đồng bộ tất cả chassis trong cùng change window.

**Bài tập synthetic.** Lab: thay `ovn-encap-type` từ geneve sang vxlan trên node2; quan sát `ovn-sbctl list Encap` trước + sau để thấy dòng Encap thay type; quan sát `ovs-vsctl show` thấy tunnel port type đổi từ `geneve` sang `vxlan` và `options:remote_ip` giữ nguyên; chạy `tcpdump -i any -nn 'udp port 4789 or udp port 6081'` để thấy traffic chuyển port.

**Lỗi thường gặp + tín hiệu chẩn đoán.** (a) Encap rỗng trên Chassis row: `ovn-controller` chưa kịp boot xong; chờ 5s. (b) Encap có nhưng tunnel port không build: `encaps_run` log `"failed to add tunnel port"`; thường do trùng tên tunnel với port cũ chưa cleanup. (c) Asymmetric Encap (dst_port mismatch): triệu chứng one-way drop.

**So sánh với hệ khác.** Tương đương Kubernetes Flannel/Calico VTEP entry trong etcd, tương đương VXLAN VTEP table của Linux bridge `bridge fdb append ... dev vxlan0`. Khác: Encap quản lý tập trung qua SBDB Raft (consistent across cluster), VTEP của Linux bridge per-host có thể sai lệch.

### 13.5.1c.3 Bảng `MAC_Binding` — cache ARP/ND học được trên Logical Router

**Khái niệm cốt lõi.** `MAC_Binding` là bảng SBDB lưu **mỗi dòng cho một (logical_port, ip) → mac** mà ovn-controller học được qua ARP reply hoặc Neighbor Advertisement trên một Logical_Router_Port. Loại: **OVSDB table**, **root** (`isRoot=true`), populated **runtime** bởi `controller/pinctrl.c` qua action `put_arp` / `put_nd` trong pipeline LR_IN_LOOKUP_NEIGHBOR (Phần 13.18 catalog REGBIT). Bảng giúp Logical Router tránh ARP request cho mỗi gói gửi đi — sau khi học một lần, dùng cache trong vài giờ.

**Lịch sử + bối cảnh ra đời.** Bảng có từ OVN 2.4 (2015) cùng đợt thiết kế Logical Router. Lý do: trước đó OVN router không có cache, mỗi gói cross-subnet phải trigger ARP request, gây flood broadcast và làm chậm hop đầu tiên. Pfaff + Bryant đề xuất bảng riêng SBDB (thay vì để mỗi chassis tự cache local) để **kết quả ARP học được chia sẻ cluster-wide**: chassis A học được MAC của 10.0.0.5 thì chassis B không phải học lại. OVN 22.06 bổ sung cột `timestamp` để hỗ trợ aging chính xác hơn (trước đó dựa vào revalidator heuristic).

**Vị trí trong kiến trúc.** MAC_Binding ở layer SBDB, root table độc lập (không phụ thuộc Chassis như Encap). Consumer phía data plane: pipeline ingress của Logical Router, stage `LR_IN_LOOKUP_NEIGHBOR` (table 0xa) đọc bảng để populate REGBIT_LOOKUP_NEIGHBOR_RESULT, stage `LR_IN_ARP_RESOLVE` (table 0xb) dùng kết quả để rewrite `eth.dst` trước khi forward ra outport. Producer: `controller/pinctrl.c` qua handler `pinctrl_handle_put_mac_binding()`.

**Vai trò + nhiệm vụ.** Hai vai trò: (1) **cache neighbor cluster-wide** — học một lần trên một chassis, mọi chassis dùng được; (2) **giảm broadcast ARP storm** — không có cache thì mỗi gói cross-subnet trigger ARP request, scale O(packet); với cache, scale O(neighbor).

**Vì sao sinh ra.** Pre-cache, Logical Router phải gửi ARP request cho từng gói cross-subnet (trừ khi data plane local cache, nhưng OVN datapath không có ARP cache native — phải đẩy lên ovn-controller userspace qua upcall, đắt). Bảng SBDB cho phép cache shared across chassis: VM trên chassis A gửi gói tới 10.0.0.5 (subnet khác), chassis A học MAC, ghi vào MAC_Binding; VM trên chassis B sau đó cũng gửi tới 10.0.0.5, chassis B đọc MAC_Binding, không phải ARP lại.

**Vấn đề mà nó giải quyết.** Ba vấn đề: (a) **upcall storm** từ ARP miss khi packet đầu tiên cross-subnet; (b) **broadcast amplification** khi nhiều VM cùng subnet đồng thời ARP cùng target; (c) **inconsistent cache** giữa các chassis (chassis A đã học, chassis B chưa).

**Tầm quan trọng.** Tier 2 medium. Bảng không tier 1 vì nếu rỗng, OVN vẫn hoạt động (chỉ chậm thêm vài ms cho gói đầu tiên). Nhưng trong production cluster lớn, MAC_Binding là **must-debug** khi gặp routing blackhole — entry stale có thể giữ MAC cũ sau khi VM migrate. Là tier 2 vì impact thường giới hạn một subnet, không đứt cả cluster.

**Cơ chế hoạt động.** Dòng chảy: (1) gói cross-subnet đến ovn-controller qua upcall do flow rule `LR_IN_LOOKUP_NEIGHBOR` table 0xa miss; (2) ovn-controller gửi ARP request từ port LR ra subnet đích; (3) reply về tới `controller/pinctrl.c`; (4) handler `pinctrl_handle_put_mac_binding()` ghi vào hmap in-memory `put_mac_bindings`; (5) main thread định kỳ gọi `run_put_mac_bindings()` flush hmap xuống SBDB MAC_Binding table; (6) các chassis khác monitor SBDB thấy entry mới; (7) gói tiếp theo cross-subnet match flow rule mới với neighbor đã resolved, không còn upcall.

**Phân loại cột (verified branch-22.03).** Bốn cột: `logical_port` (string, tên LRP), `ip` (string, IPv4 hoặc IPv6 textual), `mac` (string, "xx:xx:xx:xx:xx:xx" format), `datapath` (uuid ref Datapath_Binding của LR). `isRoot=true`, index `["logical_port", "ip"]` đảm bảo unique cặp port-ip.

**Cách kỹ sư vận hành thành thạo.** Ba kỹ năng: (1) **truy vấn theo IP**: `sudo ovn-sbctl find MAC_Binding ip=10.0.0.5`; (2) **truy vấn theo logical_port**: `sudo ovn-sbctl find MAC_Binding logical_port=lrp-tenant1`; (3) **xoá entry stale**: `sudo ovn-sbctl destroy MAC_Binding <uuid>` khi nghi cache giữ MAC cũ. Decision: nếu thấy connectivity cross-subnet kẹt sau VM migrate, kiểm MAC_Binding entry tương ứng — nếu MAC cũ, destroy + chờ ARP học lại.

**Quy trình sử dụng.** Workflow debug routing blackhole sau migrate: (1) ping cross-subnet từ VM mới migrate; (2) `ovn-sbctl find MAC_Binding ip=<dst-ip>` xem MAC trong cache; (3) so với MAC thực của VM (lấy từ VM console hoặc `virsh domiflist`); (4) nếu mismatch, destroy entry; (5) ping lại, learn fresh; (6) verify entry mới với MAC đúng. Best practice: monitor metric `ovn-sbctl list MAC_Binding | wc -l` trong production — số entry ổn định vài trăm là healthy, vọt lên hàng chục nghìn báo hiệu ARP storm.

**Khi xảy ra sự cố.** Symptom: (a) "VM ping được local subnet, không ping được subnet khác" + cache có MAC sai → entry stale, destroy; (b) "MAC_Binding rỗng dù cluster chạy 6 tháng" → ovn-controller pinctrl thread treo, restart; (c) "MAC_Binding tăng vô hạn" → routing loop hoặc subnet quá lớn không có aging — tune timeout.

**Liên quan mật thiết.** Coupling: `Datapath_Binding` (cột `datapath` ref strong), `Logical_Router_Port` (logical_port ref symbolic, không phải foreign key OVSDB nhưng ngữ nghĩa tight), `controller/pinctrl.c` (producer), Logical_Flow stage LR_IN_LOOKUP_NEIGHBOR + LR_IN_ARP_RESOLVE (consumer).

**Khác biệt giữa các phiên bản.** OVN 2.4 sơ khởi 4 cột (giữ đến nay). OVN 21.06 thêm `MAX_MAC_BINDING` constant trong `controller/pinctrl.c` để giới hạn entry per LR (default 1000); vượt quá thì oldest entry bị evict. OVN 22.06 thêm `timestamp` cột (trong main, có thể chưa backport branch-22.03).

**Cách quan sát + xác minh.** Lệnh: `sudo ovn-sbctl list MAC_Binding | head -20` (snapshot). `sudo ovn-sbctl --columns=logical_port,ip,mac list MAC_Binding | sort -k1` (table view). `sudo ovn-sbctl find MAC_Binding logical_port=lrp-tenant1` (filter). Real-time: `sudo ovsdb-client monitor unix:/var/run/ovn/ovnsb_db.sock OVN_Southbound MAC_Binding` để stream khi học entry mới.

**Source code tham chiếu (verified branch-22.03).** `ovn-sb.ovsschema` table `MAC_Binding` 4 cột + index. `controller/pinctrl.c` hàm `init_put_mac_bindings()` + `pinctrl_handle_put_mac_binding()` + `run_put_mac_bindings()` + `destroy_put_mac_bindings()` + `wait_put_mac_bindings()`. Action `put_arp` được parse trong `lib/actions.c` mapping sang OFPACT_CONTROLLER với reason MAC_BINDING.

**Trường hợp sự cố thực tế.** Case PMTUD blackhole 2023-Q4 (Phần 19.0 chi tiết): MAC_Binding giữ MAC của một gateway router cũ sau khi failover. Triệu chứng: TCP handshake xong, ICMP frag-needed reply từ gateway mới bị drop tại Logical Router vì rewrite eth.dst dùng MAC cũ. Khắc phục: `ovn-sbctl destroy MAC_Binding <uuid>`, ovn-controller học lại với MAC đúng trong < 1s. Long-term: bug fix upstream commit thêm aging timestamp.

**Bài tập synthetic.** Lab: tạo 2 LS + LR (Phần 13.2), ping VM-A subnet1 → VM-B subnet2; trước khi ping `ovn-sbctl list MAC_Binding | wc -l` (giá trị cơ sở); ping; sau ping kiểm lại — tăng đúng 1 entry với IP của VM-B; destroy entry; ping lại — re-learn. Biến thể: thay MAC của VM-B (tại OS guest), không destroy MAC_Binding → quan sát connectivity đứt do MAC cũ; destroy + ping → connectivity restored.

**Lỗi thường gặp + tín hiệu chẩn đoán.** (a) Stale MAC sau VM migrate cùng IP khác MAC; tín hiệu: ping fail nhưng `ip neigh` trên VM nguồn vẫn có entry. (b) Entry không xuất hiện sau ARP request (race với pinctrl thread overload); tín hiệu: log ovn-controller `"pinctrl: failed to put mac binding"`. (c) MAC_Binding tăng tuyến tính theo thời gian = aging fail; check timestamp column nếu có.

**So sánh với hệ khác.** Tương đương Linux kernel ARP cache (`ip neigh show`) nhưng phân tán cluster-wide. Tương đương Cisco IOS ARP table với timeout 4 giờ default. Khác: OVN MAC_Binding chia sẻ qua SBDB Raft, mọi chassis cùng nhìn; Linux ARP cache là per-host, mỗi host học riêng.

### 13.5.1c.4 Bảng `FDB` — bảng học MAC L2 trên Logical Switch (OVN 22.03+)

**Khái niệm cốt lõi.** `FDB` (Forwarding Database) là bảng SBDB lưu **mỗi dòng cho một (datapath, mac) → port_key** mà ovn-controller học được trên một Logical_Switch khi quan sát source MAC của gói đi vào port. Loại: **OVSDB table**, **root** (`isRoot=true`), populated runtime bởi `controller/pinctrl.c` qua action `put_fdb` được gọi trong stage `LS_IN_LOOKUP_FDB` / `LS_IN_PUT_FDB` của pipeline. Phục vụ unknown unicast forwarding cho Logical Switch có port với `port_security=[]` hoặc `addresses=unknown`.

**Lịch sử + bối cảnh ra đời.** Bảng được giới thiệu OVN 22.03 — trước đó OVN dựa vào OVS bridge native learning trên `br-int` cho L2. Vấn đề: bridge native learning per-chassis, không chia sẻ; VM trên chassis A có MAC X, chassis B chưa thấy traffic source X thì không biết X ở đâu, flood. OVN 22.03 chuyển L2 learning lên SBDB FDB table để đồng nhất với mô hình MAC_Binding (cluster-wide cache). Tác giả Numan Siddique (Red Hat) commit ban đầu khoảng cuối 2021.

**Vị trí trong kiến trúc.** FDB ở layer SBDB, root table. Producer: `controller/pinctrl.c` hàm `pinctrl_handle_put_fdb()`. Consumer: pipeline LS ingress stage `LS_IN_LOOKUP_FDB` (table tương ứng REGBIT_LOOKUP_FDB) và `LS_IN_KNOWN_FDB`. Khi gói có dst MAC chưa biết, flow miss → upcall → ovn-controller tra FDB → reinject với outport hoặc fallback flood.

**Vai trò + nhiệm vụ.** Hai vai trò: (1) **L2 learning cluster-wide** cho LSP có addresses=unknown (router peering, third-party VNF); (2) **ngăn unknown unicast flood** sau khi học — gói thứ hai cùng dst MAC không flood mà unicast trực tiếp.

**Vì sao sinh ra.** Pre-22.03: OVS bridge learning per-chassis tạo asymmetry; multicast group OVN không phát hiện được khi nào nên flood vì không có view tổng. Bảng FDB SBDB giúp pipeline đồng nhất: mọi chassis cùng đọc bảng, cùng quyết định forward/flood.

**Vấn đề mà nó giải quyết.** (a) **flood storm khi mới boot cluster** — không có FDB, mọi gói unknown unicast flood toàn LS; (b) **asymmetric learning** giữa các chassis; (c) **migration learning gap** — VM migrate sang chassis mới, chassis khác phải re-learn.

**Tầm quan trọng.** Tier 2 medium. Tác động lớn cho LS có nhiều LSP `addresses=unknown` (router peering, VLAN trunk); ít tác động cho LS thuần VM (vì LSP đã có addresses xác định, không cần FDB).

**Cơ chế hoạt động.** (1) Gói vào LSP có port_security/addresses=unknown; (2) flow stage LS_IN_LOOKUP_FDB upcall; (3) `pinctrl_handle_put_fdb()` ghi vào hmap in-memory; (4) main thread `run_put_fdbs()` flush sang SBDB FDB table; (5) cùng lúc reinject gói với register REGBIT_HIT_LOOKUP_FDB=1 cho stage downstream; (6) chassis khác monitor SBDB thấy entry mới; (7) gói destination cùng MAC unicast được forward chính xác outport thay vì flood.

**Phân loại cột (verified branch-22.03).** Ba cột: `mac` (string), `dp_key` (integer 1-16777215, tunnel_key của Datapath_Binding), `port_key` (integer 1-16777215, tunnel_key của Port_Binding nguồn). `isRoot=true`, index `["mac", "dp_key"]` đảm bảo unique cặp MAC trong một datapath.

**Cách kỹ sư vận hành thành thạo.** Hai kỹ năng: (1) `sudo ovn-sbctl list FDB | head` xem snapshot; (2) `sudo ovn-sbctl find FDB dp_key=<N>` lọc theo datapath. Decision: nếu thấy unknown unicast flood vẫn diễn ra dù VM đã chạy lâu, kiểm FDB có entry tương ứng không — không có nghĩa là pinctrl chưa học hoặc LSP có addresses xác định (không trigger learning).

**Quy trình sử dụng.** Workflow debug flood: (1) `ovs-ofctl dump-flows br-int | grep <dst-mac>` xem flow đang match flood action; (2) `ovn-sbctl find FDB mac="<dst-mac>"` xem có entry không; (3) nếu không có, kiểm LSP có `port_security=[]` không; (4) nếu có port_security không rỗng, OVN không học (vì port có MAC xác định trong NBDB, không cần FDB); (5) verify với gói test có source MAC mới — sau gói đầu tiên, entry FDB xuất hiện trong < 1s.

**Khi xảy ra sự cố.** Symptom: (a) flood persistent → FDB chưa populate, kiểm LSP config; (b) entry sai port_key sau migrate → destroy entry, chờ học lại; (c) FDB tăng vô hạn → loop hoặc port_security bug ghi nhiều MAC.

**Liên quan mật thiết.** Coupling: `Datapath_Binding` (qua dp_key, không phải UUID ref mà tunnel_key integer), `Port_Binding` (qua port_key cùng cách), `controller/pinctrl.c` (producer), pipeline LS_IN_LOOKUP_FDB / LS_IN_KNOWN_FDB / LS_IN_PUT_FDB (consumer + populator).

**Khác biệt giữa các phiên bản.** OVN 22.03: bảng được giới thiệu (ban đầu 3 cột). OVN 22.06+ schema ổn định, không thêm cột. Trước 22.03: OVS bridge native learning, không có FDB SBDB.

**Cách quan sát + xác minh.** Lệnh: `sudo ovn-sbctl list FDB`, `sudo ovn-sbctl --columns=mac,dp_key,port_key list FDB`, `sudo ovn-sbctl find FDB dp_key=<N>`. Real-time: `sudo ovsdb-client monitor unix:/var/run/ovn/ovnsb_db.sock OVN_Southbound FDB`.

**Source code tham chiếu (verified branch-22.03).** `ovn-sb.ovsschema` table `FDB` 3 cột + index. `controller/pinctrl.c` hàm `init_fdb_entries()`, `pinctrl_handle_put_fdb()`, `run_put_fdb()`, `run_put_fdbs()`, `fdb_lookup()`, `destroy_fdb_entries()`, `wait_put_fdbs()`. Pipeline stage compile trong `northd/northd.c` build_lswitch_lflows_l2_fdb().

**Trường hợp sự cố thực tế.** Case 2024-Q1: cluster với LS dùng cho router peering BGP (LSP type=router, addresses=router); upgrade từ OVN 22.03.0 → 22.03.5 thay đổi semantic FDB. Triệu chứng: BGP session flap mỗi 5 phút do unknown unicast bị flood ngược về peer. Khắc phục: rolling restart ovn-controller toàn cluster để re-populate FDB từ traffic mới, đồng thời tăng `pinctrl-thread-priority` để giảm race với learning.

**Bài tập synthetic.** Lab: tạo LS với 2 LSP type=localnet; gửi traffic unicast giữa hai port; trước test `ovn-sbctl list FDB | wc -l` (baseline); sau gói đầu tiên kiểm tăng đúng 1 entry; flush bằng `ovn-sbctl destroy FDB <uuid>`; gửi lại — entry tái xuất hiện.

**Lỗi thường gặp + tín hiệu chẩn đoán.** (a) FDB rỗng dù LS có traffic → LSP có addresses xác định, không trigger học (đây là behavior đúng, không phải bug); (b) entry sai port_key sau migrate → destroy + relearn; (c) FDB grow nhanh → check port_security bug đang spoofing nhiều MAC.

**So sánh với hệ khác.** Tương đương trực tiếp `bridge fdb show` của Linux bridge — cùng concept (mac, port). Khác: FDB OVN cluster-wide qua SBDB; Linux bridge per-host. Tương đương MAC table của Cisco switch (`show mac address-table`). Khác lớn nhất: OVN FDB chia sẻ across chassis ngay lập tức qua Raft replication.

### 13.5.1c.5 Bảng `Multicast_Group` — định danh nhóm forwarding multicast/broadcast

**Khái niệm cốt lõi.** `Multicast_Group` là bảng SBDB lưu **mỗi dòng cho một nhóm forwarding** trong một datapath, ánh xạ tên nhóm sang **danh sách Port_Binding** thành viên và một **tunnel_key dành riêng** (range 32768-65535, khác range 1-32767 của Port_Binding tunnel_key). Loại: **OVSDB table**, **root** (`isRoot=true`), populated bởi `ovn-northd` khi compile broadcast/multicast group (BUM traffic, IGMP snooping membership, MLD). Pipeline OVN dùng dòng này để rewrite outport thành tunnel_key của group, sau đó datapath replicate gói tới mọi member.

**Lịch sử + bối cảnh ra đời.** Bảng có từ OVN 2.4 (2015) cho broadcast L2 cơ bản (group `_MC_flood` mặc định mỗi LS). OVN 21.06 mở rộng cho IGMP snooping với group dynamic theo IGMP membership (Phần 13.18). OVN 22.03 chuẩn hoá range tunnel_key 32768-65535 để tách bạch với Port_Binding. Tác giả ban đầu Bryant + Pettit; mở rộng IGMP do Han Zhou (eBay) đóng góp 2021.

**Vị trí trong kiến trúc.** Multicast_Group ở layer SBDB, root table. Cộng tác với `IGMP_Group` (theo dõi membership IGMP snooping) — IGMP_Group là input động, Multicast_Group là output static-ish được northd compile từ IGMP_Group + LSP membership. Pipeline consume: stage `LS_OUT_PRE_LB`/`LS_IN_L2_LKUP` rewrite outport=`MC_FLOOD` hoặc `MC_<group>`, datapath replicate.

**Vai trò + nhiệm vụ.** Bốn vai trò: (1) **broadcast L2** (group `_MC_flood` mặc định cho ARP, DHCP discover); (2) **multicast IP** dùng IGMP snooping để giới hạn replication chỉ cho member; (3) **router multicast routing** (group `_MC_mrouter_flood`, `_MC_mrouter_static`); (4) **unknown unicast** (group `_MC_unknown` cho LSP có addresses=unknown).

**Vì sao sinh ra.** Pre-bảng: OVS bridge xử lý broadcast bằng flood port-list local. Vấn đề: cluster OVN distributed cần cluster-aware multicast — gói broadcast trên một chassis phải replicate ra tunnel sang mọi chassis có member. Bảng riêng cho phép `ovn-northd` compile membership thành tunnel_key duy nhất, ovn-controller mỗi chassis biết "nhận gói tunnel với tunnel_key Y nghĩa là replicate cho các port trong group này".

**Vấn đề mà nó giải quyết.** (a) **broadcast flood không kiểm soát** trên cluster lớn — không có group định danh, mọi gói flood đi tunnel toàn cluster gây bandwidth waste; (b) **IGMP snooping cluster-wide** — chassis A có VM join group 224.1.1.1, chassis B không có member, không cần replicate sang B; (c) **router multicast** — gói multicast cross-LR cần group tổng hợp tất cả LSP của các LS attached.

**Tầm quan trọng.** Tier 2 medium. Trong cluster OVN production thuần unicast (ít multicast app), bảng chủ yếu chứa các group default `_MC_flood` + `_MC_unknown`. Trong cluster có multicast app (IPTV, video conferencing, ROS robotics), bảng phình lên hàng nghìn dòng và là điểm performance critical.

**Cơ chế hoạt động.** (1) `ovn-northd` đọc NBDB Logical_Switch + Logical_Switch_Port; (2) tổng hợp các nhóm: mọi LSP của LS = `_MC_flood`, các LSP có IGMP join = `_MC_<addr>`; (3) cho mỗi group, allocate tunnel_key trong range 32768-65535 (đảm bảo unique trong datapath); (4) INSERT/UPDATE Multicast_Group row với cột `ports` = set Port_Binding UUID weak ref; (5) compile Logical_Flow rule rewrite outport=`_MC_<X>` thành tunnel_key Y; (6) ovn-controller mỗi chassis nhận tunnel với tunnel_key Y, tra Multicast_Group theo tunnel_key, replicate ra các Port_Binding thuộc về chassis local.

**Phân loại cột (verified branch-22.03).** Bốn cột: `datapath` (uuid ref Datapath_Binding), `name` (string, ví dụ `_MC_flood`), `tunnel_key` (integer 32768-65535, range chuyên dụng cho multicast), `ports` (set Port_Binding UUID, weak ref, min=0 max=unlimited). `isRoot=true`, hai index: `["datapath", "tunnel_key"]` (tránh collision tunnel_key trong cùng datapath) và `["datapath", "name"]` (tên unique trong datapath).

**Cách kỹ sư vận hành thành thạo.** Ba kỹ năng: (1) **liệt kê** `sudo ovn-sbctl list Multicast_Group | head -30`; (2) **lọc per datapath**: `sudo ovn-sbctl find Multicast_Group datapath=<dp_uuid>`; (3) **đối chiếu với IGMP_Group** khi debug IPTV: `sudo ovn-sbctl list IGMP_Group` xem membership snoop được. Decision: nếu broadcast không tới mọi VM, kiểm `_MC_flood` group có chứa Port_Binding của VM thiếu không; nếu thiếu → northd compile bug, restart northd.

**Quy trình sử dụng.** Workflow debug "VM A không nhận DHCP DISCOVER reply mà VM B cùng LS thì có": (1) `ovn-nbctl show ls-X` xác nhận cả hai LSP cùng LS; (2) `ovn-sbctl find Multicast_Group name=_MC_flood datapath=<dp-of-ls-X>`; (3) đếm số UUID trong cột `ports` — phải bằng số LSP của LS; (4) nếu thiếu Port_Binding của VM A, restart ovn-northd để force recompile; (5) verify lại; (6) nếu vẫn thiếu, kiểm LSP của VM A có `port_security` config bất thường hoặc LSP up=false không.

**Khi xảy ra sự cố.** Symptom: (a) DHCP fail trên một VM → kiểm `_MC_flood` membership; (b) IGMP-based multicast app drop → kiểm `_MC_<addr>` group có member đúng không; (c) unknown unicast không flood → kiểm `_MC_unknown` còn tồn tại không.

**Liên quan mật thiết.** Coupling chặt: `Datapath_Binding` (strong ref qua datapath), `Port_Binding` (weak ref qua ports — xoá Port_Binding tự reset khỏi group), `IGMP_Group` (input dynamic cho northd compile), `controller/lflow.c` (consumer compile vào OF rule), pipeline stage LS_IN_L2_LKUP + LR_IN_IP_ROUTING (multicast routing).

**Khác biệt giữa các phiên bản.** OVN 2.4: 4 cột cơ bản. OVN 21.06: thêm IGMP snooping integration, populate động. OVN 22.03: chuẩn hoá tunnel_key range 32768-65535 (verified `ovn-sb.ovsschema` branch-22.03 enum minInteger=32768 maxInteger=65535). OVN 24.03: tối ưu compile path cho cluster nhiều multicast group.

**Cách quan sát + xác minh.** Lệnh: `sudo ovn-sbctl list Multicast_Group | head -30`, `sudo ovn-sbctl --columns=name,tunnel_key,ports list Multicast_Group`, `sudo ovn-sbctl find Multicast_Group name=_MC_flood`. Đối chiếu với OF: `sudo ovs-ofctl -O OpenFlow15 dump-flows br-int | grep "outport=0x<tunnel_key_hex>"` để thấy flow rule rewrite outport. Real-time: `sudo ovsdb-client monitor unix:/var/run/ovn/ovnsb_db.sock OVN_Southbound Multicast_Group`.

**Source code tham chiếu (verified branch-22.03).** `ovn-sb.ovsschema` table `Multicast_Group` 4 cột + 2 index, tunnel_key range 32768-65535. `northd/northd.c` build_mcast_groups() (compile NBDB → SBDB row). `controller/lflow.c` consume Multicast_Group khi build OF rule cho stage L2_LKUP. `controller/ip_mcast.c` xử lý IGMP snooping populate IGMP_Group. Pipeline stage REGBIT_DO_MCAST_FLOOD trong Phần 13.18 catalog.

**Trường hợp sự cố thực tế.** Case 2024-Q2 OSP cluster IPTV multicast: sau khi enable IGMP snooping, multicast stream 239.1.1.1 chỉ đến 70% subscriber. Điều tra: `sudo ovn-sbctl find Multicast_Group name="_MC_239.1.1.1"` thấy `ports` chỉ chứa 70% LSP join. `sudo ovn-sbctl list IGMP_Group` thấy IGMP join report của 30% còn lại không tới ovn-controller (do flow IGMP snooping rule miss vì TTL=1 bị decrement sai trong một stage trung gian). Khắc phục: backport bug fix từ branch-22.06 commit upstream về branch-22.03 in-house.

**Bài tập synthetic.** Lab: tạo LS với 4 LSP, gửi broadcast ARP từ một LSP, kiểm `_MC_flood` group có 4 port; xoá một LSP, sau 1 giây kiểm group còn 3 port (weak ref tự dọn); thêm IGMP snooping config trên LS, từ một VM gửi IGMP join 224.1.1.1, kiểm Multicast_Group `_MC_224.1.1.1` xuất hiện với 1 port; thêm VM thứ hai join, group có 2 port.

**Lỗi thường gặp + tín hiệu chẩn đoán.** (a) Group thiếu Port_Binding member sau khi LSP up → northd compile chưa kịp, chờ 100ms; nếu kéo dài → northd treo. (b) Group có duplicate tunnel_key với Port_Binding (nhầm range) → schema corruption, restart SBDB. (c) IGMP_Group rỗng dù VM có join → flow IGMP snooping rule sai, kiểm pipeline.

**So sánh với hệ khác.** Tương đương `bridge mdb show` của Linux bridge (multicast database). Tương đương Cisco IGMP snooping table (`show ip igmp snooping groups`). Khác: Multicast_Group OVN cluster-wide, mỗi chassis cùng nhìn tunnel_key như nhau, replicate phân tán; Linux bridge mdb per-host. Tương đương VMware NSX-T multicast group object trong management plane (concept giống nhưng implementation centralized).

## 13.5.1d Năm bảng SBDB runtime còn lại: Service_Monitor, IGMP_Group, BFD, HA_Chassis, Connection

> Phần này là cụm thứ hai của bộ "SBDB runtime đồng hành Port_Binding". Năm bảng đầu (Phần 13.5.1c) phục vụ identity + L2/L3 forwarding + multicast static. Năm bảng dưới đây phục vụ **health checking, dynamic membership, fast failover, gateway HA, client connection**. Mỗi bảng dạy theo cùng lát cắt 20 chiều như 13.5.1c. Schema verified upstream `ovn-sb.ovsschema` branch-22.03. Cross-link source code: `controller/pinctrl.c` (Service_Monitor), `controller/ip-mcast.c` (IGMP_Group), `controller/bfd.c` (BFD), `controller/ha-chassis.c` (HA_Chassis), thư viện OVSDB IDL chung (Connection).

### 13.5.1d.1 Bảng `Service_Monitor` — health probe per backend của Load_Balancer

**Khái niệm cốt lõi.** `Service_Monitor` là bảng SBDB lưu **mỗi dòng cho một (logical_port, ip, port, protocol)** đại diện một **health probe** mà ovn-controller phát đi đều đặn để biết một backend của Load_Balancer còn sống hay chết. Loại: **OVSDB table**, **root** (`isRoot=true`), populated bởi `ovn-northd` khi compile NBDB Load_Balancer có cấu hình health_check; consumed + cập nhật cột `status` runtime bởi `controller/pinctrl.c` của chassis nơi backend được host. Mỗi dòng là **một mạch thử nhỏ** chạy độc lập, kết quả ghi ngược trở lại SBDB cho northd dùng để loại bỏ backend chết khỏi pool.

**Lịch sử + bối cảnh ra đời.** Bảng được giới thiệu OVN 2.12 (2019) khi Numan Siddique (Red Hat) thêm tính năng Load_Balancer health check. Trước đó OVN LB chỉ là tĩnh: backend trong NBDB Load_Balancer.vips được dispatch round-robin bất kể VM còn sống hay không; nếu VM chết, traffic gửi vào hố đen. OVN 2.12 đem mô hình HAProxy/Envoy "active health check" về OVN: mỗi backend có một probe TCP/UDP định kỳ, kết quả lưu trong Service_Monitor.status, northd chỉ compile flow cho backend `status=online`. OVN 21.06 cải tiến UDP probe (gửi packet rỗng + lắng nghe ICMP unreachable). OVN 22.03 ổn định schema 9 cột.

**Vị trí trong kiến trúc.** Service_Monitor ở layer SBDB, root table. Producer (compile): `northd/ovn-northd.c` build_lb_svcs() đọc NBDB Load_Balancer + Load_Balancer_Health_Check, sinh dòng Service_Monitor cho mỗi (vip-backend, protocol). Producer (runtime status): `controller/pinctrl.c` hàm `svc_monitors_run()`, chỉ chassis nơi backend được host (logical_port nằm trên chassis đó) chạy probe và update `status`. Consumer: northd đọc lại `status` để loại backend offline khỏi flow rule LB.

**Vai trò + nhiệm vụ.** Ba vai trò: (1) **active health probe**, gửi TCP SYN hoặc UDP packet định kỳ tới backend, đo response; (2) **gateway dữ liệu giữa data plane reality và control plane decision**, kết quả probe là tín hiệu duy nhất để northd biết backend chết; (3) **distribute health checking**, chỉ chassis chủ của backend chạy probe, tránh cluster N×M overhead.

**Vì sao sinh ra.** Pre-2.12, kiến trúc OpenStack Octavia + OVN không tích hợp: Octavia agent chạy health check ngoài OVN, kết quả update vào Neutron LBaaS, cập nhật trễ (10-30s). User cần OVN-native LB với health check tích hợp ngay trong OVN control plane để giảm độ trễ phát hiện backend chết xuống 1-3s.

**Vấn đề mà nó giải quyết.** (a) **Traffic blackhole** khi backend chết mà LB chưa biết, gói tin ECMP rơi vào hố; (b) **Convergence chậm** từ data plane (backend chết) tới control plane (loại khỏi pool); (c) **Decentralized probe**, mỗi backend chỉ chassis chủ probe, tránh fanout N chassis × M backend.

**Tầm quan trọng.** Tier 2 medium. Cluster không dùng OVN LB hoặc dùng LB không health_check thì bảng rỗng và không tác động. Cluster có OVN LB với health_check (Kubernetes service production qua OVN-Kubernetes, Octavia provider OVN) thì bảng load-bearing, sai schema status = mọi traffic LB sai đích.

**Cơ chế hoạt động.** Vòng đời một dòng: (1) NBDB tạo Load_Balancer với health_check; (2) northd build_lb_svcs() INSERT Service_Monitor row với `logical_port=<backend-LSP>`, `ip=<backend-IP>`, `port=<backend-port>`, `protocol=tcp|udp`, `status=offline` ban đầu, `options:{interval, success_count, failure_count, timeout}`; (3) ovn-controller mỗi chassis monitor SBDB; (4) chassis nơi `logical_port` được claim chạy `svc_monitors_run()` định kỳ (mặc định mỗi 5s); (5) gửi TCP SYN từ `src_ip:src_port` (cấp phát ephemeral) tới `ip:port`; (6) chờ SYN-ACK; (7) update `status` thành `online` nếu thành công liên tiếp `success_count` lần, `offline` nếu fail liên tiếp `failure_count` lần, `error` nếu probe không gửi được; (8) northd nhận event SBDB và recompile flow LB.

**Phân loại cột (verified branch-22.03).** Chín cột: `ip` (string), `protocol` (enum tcp|udp), `port` (integer 0-65535), `logical_port` (string, tên LSP của backend), `src_mac` (string), `src_ip` (string), `status` (enum online|offline|error), `options` (map), `external_ids` (map). `isRoot=true`, một index `["logical_port","ip","port","protocol"]` đảm bảo unique tuple probe.

**Cách kỹ sư vận hành thành thạo.** Ba kỹ năng: (1) `sudo ovn-sbctl list Service_Monitor` xem tất cả probe đang chạy + status; (2) `sudo ovn-sbctl find Service_Monitor status=offline` lọc backend chết, đây là "danh sách triệu chứng" khi LB lỗi; (3) `sudo ovn-sbctl --columns=ip,port,status,options list Service_Monitor` table view tinh gọn. Decision: nếu LB rớt traffic, `find Service_Monitor status=offline` cho ra IP backend đang chết, kiểm chassis của LSP đó còn chạy không, kiểm bản thân backend service (ví dụ httpd) có listen không.

**Quy trình sử dụng.** Workflow setup health check: (1) tạo Load_Balancer_Health_Check trong NBDB qua `ovn-nbctl create Load_Balancer_Health_Check vip="10.0.0.5\:80" options:interval=5 options:timeout=2 options:success_count=2 options:failure_count=3`; (2) attach vào Load_Balancer qua `ovn-nbctl set Load_Balancer <lb-uuid> health_check="@hc-uuid"`; (3) đợi 5-10s; (4) `ovn-sbctl list Service_Monitor` xác nhận dòng được tạo; (5) verify `status` chuyển `offline` → `online` sau 2 probe thành công; (6) tắt một backend, verify status chuyển `online` → `offline` sau 3 probe fail.

**Khi xảy ra sự cố.** Symptom: (a) "LB không loại backend chết" → kiểm `Service_Monitor.status` của backend đó, nếu vẫn `online`, probe đang sai; (b) "backend bình thường mà status=offline" → kiểm src_ip/src_mac có route đến backend không, kiểm flow rule cho phép probe pass không; (c) "không có dòng Service_Monitor dù LB có health_check" → northd compile chưa kịp, restart northd.

**Liên quan mật thiết.** Coupling chặt: `Load_Balancer` (NBDB nguồn), `Load_Balancer_Health_Check` (NBDB nguồn), `Port_Binding` (logical_port phải khớp với một LSP đã bind), `controller/pinctrl.c` (producer status), `Chassis` (chỉ chassis chủ chạy probe). Loose: `Logical_Flow` (consumer indirect, northd recompile flow LB khi status đổi).

**Khác biệt giữa các phiên bản.** OVN 2.12: bảng giới thiệu, hỗ trợ TCP. OVN 20.06: thêm UDP probe (gửi packet + nhận ICMP unreachable cho backend chết). OVN 21.06: tối ưu interval scheduling, giảm CPU pinctrl. OVN 22.03: schema ổn định 9 cột (verified). OVN 24.03: thêm tham số `options:dst_mac` cho probe trong môi trường L2 đặc biệt.

**Cách quan sát + xác minh.** Lệnh: `sudo ovn-sbctl list Service_Monitor`, `sudo ovn-sbctl --columns=ip,port,protocol,status list Service_Monitor`, `sudo ovn-sbctl find Service_Monitor status=offline`. Real-time stream khi probe chuyển trạng thái: `sudo ovsdb-client monitor unix:/var/run/ovn/ovnsb_db.sock OVN_Southbound Service_Monitor`. Đối chiếu probe thực tế: `sudo tcpdump -i any -nn 'tcp[tcpflags] & tcp-syn != 0 and host <backend-ip> and port <backend-port>'`, trên chassis chủ của backend LSP sẽ thấy SYN gửi đi mỗi `interval` giây.

**Source code tham chiếu (verified branch-22.03).** `ovn-sb.ovsschema` table `Service_Monitor` 9 cột + index. `controller/pinctrl.c` hàm `init_svc_monitors()` (init), `sync_svc_monitors()` (đồng bộ với SBDB), `svc_monitors_run()` (gửi probe + cập nhật status), `svc_monitors_wait()` (poll loop), `pinctrl_handle_svc_check()` (xử lý reply), `destroy_svc_monitors()` (cleanup). `northd/ovn-northd.c` hàm `build_lb_svcs()` compile NBDB Load_Balancer_Health_Check sang Service_Monitor row. Hằng số timeout default trong `controller/pinctrl.c::SVC_MONITOR_DFLT_INTERVAL`.

**Trường hợp sự cố thực tế.** Case 2024-Q2 OVN-Kubernetes cluster: một backend pod bị OOMKilled, kubelet xóa pod, OVN xóa LSP nhưng Service_Monitor row tương ứng còn 30s do northd recompile chưa kịp. Triệu chứng: traffic K8s service vẫn route 1/3 connections vào IP đã chết, TCP retransmit. Khắc phục: bug fix upstream commit thêm trigger northd recompile ngay khi LSP xóa thay vì đợi periodic recompile.

**Bài tập synthetic.** Lab: tạo Load_Balancer với 2 backend (Phần 13.3 lab); tạo Load_Balancer_Health_Check, attach; `sudo ovn-sbctl list Service_Monitor` thấy 2 dòng status=online; tắt một backend bằng `sudo systemctl stop httpd` trên VM; sau ~15s (3 probe × 5s) status chuyển offline; verify flow `ovs-ofctl dump-flows br-int | grep <vip>` thấy backend offline bị loại; bật lại httpd; sau ~10s status chuyển online; flow re-include backend.

**Lỗi thường gặp + tín hiệu chẩn đoán.** (a) Status mãi `error` không chuyển `online`/`offline` → src_ip không có route đến backend; tín hiệu: log ovn-controller `"svc_monitor: failed to send probe"`. (b) Status flap online/offline mỗi vài chục giây → interval quá nhỏ, network jitter; tăng `options:timeout`. (c) Bảng tăng nhanh không giảm → northd không cleanup khi LB xóa, restart northd.

**So sánh với hệ khác.** Tương đương HAProxy `option httpchk` health check (cùng concept active probe), tương đương Nginx upstream `health_check` (cùng pattern). Tương đương Cisco GSLB health probe. Khác: Service_Monitor distribute, chỉ chassis chủ của backend chạy probe, không có node trung tâm; HAProxy + Nginx có một process probe tất cả; Cisco GSLB có agent riêng. Distribute giảm load đáng kể trong cluster có hàng nghìn backend.

### 13.5.1d.2 Bảng `IGMP_Group` — membership IGMP snooping động

**Khái niệm cốt lõi.** `IGMP_Group` là bảng SBDB lưu **mỗi dòng cho một (multicast-address, datapath, chassis)** ghi nhận **các Port_Binding đã gửi IGMP join** report cho địa chỉ multicast đó trên chassis tương ứng. Loại: **OVSDB table**, **root** (`isRoot=true`), populated runtime bởi `controller/ip-mcast.c` qua action `igmp_group_create` / `igmp_group_update_ports` khi pinctrl handle gói IGMP report. Khác Multicast_Group (Phần 13.5.1c.5, output static northd compile) ở chỗ IGMP_Group là **input dynamic** từ data plane; northd đọc IGMP_Group + LSP membership để compile Multicast_Group.

**Lịch sử + bối cảnh ra đời.** Bảng giới thiệu OVN 21.06 (2021) khi Han Zhou (eBay/Nvidia) đóng góp tính năng IGMP snooping. Trước đó OVN không snoop IGMP; mọi multicast IP flood như broadcast L2, bandwidth waste lớn cho IPTV / video conferencing. OVN 21.06 thêm pipeline stage IGMP snoop trong LS, populate IGMP_Group qua pinctrl, northd consume để giới hạn `_MC_<addr>` flooding. OVN 22.03 thêm IGMP_Group cho Logical_Router multicast routing. OVN 22.06 hỗ trợ MLD (IPv6 multicast).

**Vị trí trong kiến trúc.** IGMP_Group ở layer SBDB, root table. Producer: `controller/ip-mcast.c` qua `igmp_group_create()` + `igmp_group_update_ports()` được gọi từ pinctrl handler khi nhận IGMP report. Consumer: `northd/northd.c` hàm `build_mcast_groups()` đọc IGMP_Group, kết hợp với LSP membership static để build Multicast_Group entries. Pipeline LS stage `LS_IN_IGMP_QUERY` + `LS_IN_LKUP_MCAST_FLOW` đẩy IGMP report lên controller, `LS_OUT_PRE_LB` hoặc downstream replicate dựa trên Multicast_Group.

**Vai trò + nhiệm vụ.** Ba vai trò: (1) **track membership động** theo IGMP/MLD report; (2) **input cho northd build Multicast_Group**; (3) **per-chassis scope** — IGMP_Group có cột `chassis` cho phép multicast routing trên LR biết chassis nào có thành viên (gửi tunnel tới chỉ chassis có member, không flood toàn cluster).

**Vì sao sinh ra.** Pre-21.06: OVN không có IGMP snooping native; user phải bật IGMP snoop trên upstream physical switch hoặc chấp nhận flood. Vấn đề: cluster OpenStack triển khai IPTV cần multicast OVN-aware; flood toàn LS với 100 VM cho stream 5 Mbps = 500 Mbps overhead trên mỗi gói. IGMP_Group cho phép giới hạn replicate chỉ tới VM có IGMP join.

**Vấn đề mà nó giải quyết.** (a) **Bandwidth waste** từ flood multicast trong cluster lớn; (b) **dynamic membership** — IGMP join/leave thay đổi runtime, schema cần phản ánh ngay; (c) **multicast routing cross-LS**, LR cần biết chassis nào có member để forward selective.

**Tầm quan trọng.** Tier 2 medium. Bảng load-bearing chỉ trong cluster có multicast app (IPTV, ROS, video conferencing, financial market data feeds). Cluster thuần unicast (>90% production OpenStack) bảng rỗng và không tác động.

**Cơ chế hoạt động.** Vòng đời: (1) VM gửi IGMP membership report `{type=join, group=224.1.1.1}` ra LSP; (2) flow stage LS IGMP snoop match IGMP packet, `output:CONTROLLER`; (3) `controller/pinctrl.c` handler nhận packet-in, gọi `controller/ip-mcast.c::igmp_group_create()` hoặc `igmp_group_update_ports()`; (4) function INSERT/UPDATE IGMP_Group row với `address=224.1.1.1`, `datapath=<dp-of-ls>`, `chassis=<self>`, `ports=[<lsp-uuid>]`; (5) northd monitor SBDB, recompile Multicast_Group `_MC_224.1.1.1` với union các IGMP_Group cho address đó; (6) gói multicast tiếp theo replicate chỉ tới các Port_Binding trong group.

**Phân loại cột (verified branch-22.03).** Bốn cột: `address` (string, multicast IP textual như "224.1.1.1" hoặc IPv6 multicast), `datapath` (uuid weak ref Datapath_Binding), `chassis` (uuid weak ref Chassis), `ports` (set Port_Binding UUID weak refs). `isRoot=true`, một index `["address","datapath","chassis"]` đảm bảo unique tuple per chassis.

**Cách kỹ sư vận hành thành thạo.** Ba kỹ năng: (1) `sudo ovn-sbctl list IGMP_Group` snapshot tất cả membership; (2) `sudo ovn-sbctl find IGMP_Group address=224.1.1.1` lọc theo group; (3) `sudo ovn-sbctl --columns=address,chassis,ports list IGMP_Group` table view chéo chassis. Decision: nếu multicast stream không tới một VM, kiểm IGMP_Group có dòng cho VM đó không (tức là ovn-controller có nhận được IGMP join từ VM chưa).

**Quy trình sử dụng.** Workflow debug "VM-A nhận multicast, VM-B không nhận": (1) verify IGMP join trên VM-B: `sudo tcpdump -i eth0 -nn igmp` thấy report đi ra; (2) `ovn-sbctl find IGMP_Group address=<group> ports{}=<vm-b-lsp-uuid>` xem có dòng không; (3) nếu không, kiểm flow IGMP snoop trong stage LS_IN_LKUP_MCAST_FLOW có match đúng không (TTL=1 thường gây miss); (4) nếu có IGMP_Group nhưng Multicast_Group thiếu VM-B, restart northd; (5) verify lại với gói multicast test.

**Khi xảy ra sự cố.** Symptom: (a) "VM không nhận multicast dù IGMP join" → IGMP_Group thiếu dòng; (b) "membership leak — VM rời nhưng vẫn nhận" → IGMP leave không xử lý, ovn-controller có thể bug; (c) "IGMP_Group rỗng dù nhiều VM join" → flow IGMP snoop bị overshadow bởi rule khác priority cao hơn.

**Liên quan mật thiết.** Coupling rất chặt với `Multicast_Group` (output dependency của northd compile), `Port_Binding` (weak ref qua `ports`), `Datapath_Binding` (weak ref qua `datapath`), `Chassis` (weak ref qua `chassis`), `controller/ip-mcast.c` (producer chính), `controller/pinctrl.c` (đầu vào packet-in handler).

**Khác biệt giữa các phiên bản.** OVN 21.06: bảng giới thiệu cho IGMP v2/v3 (IPv4 multicast). OVN 22.03: schema ổn định 4 cột (verified). OVN 22.06: hỗ trợ MLD (IPv6 multicast), cùng schema nhưng `address` chấp nhận IPv6 textual (e.g., "ff02::1"). OVN 24.03: tối ưu pinctrl handler để giảm latency populate.

**Cách quan sát + xác minh.** Lệnh: `sudo ovn-sbctl list IGMP_Group | head -20`, `sudo ovn-sbctl find IGMP_Group address=224.1.1.1`, `sudo ovn-sbctl --columns=address,chassis,ports list IGMP_Group`. Real-time: `sudo ovsdb-client monitor unix:/var/run/ovn/ovnsb_db.sock OVN_Southbound IGMP_Group` để stream khi VM join/leave. Đối chiếu IGMP packet thực: `sudo tcpdump -i any -nn 'igmp'` trên chassis chủ của VM.

**Source code tham chiếu (verified branch-22.03).** `ovn-sb.ovsschema` table `IGMP_Group` 4 cột + index `["address","datapath","chassis"]`. `controller/ip-mcast.c` hàm `igmp_group_index_create()`, `igmp_group_lookup()`, `igmp_group_create()`, `igmp_group_update_ports()`, `igmp_group_delete()`, `igmp_group_cleanup()`. `controller/pinctrl.c` packet-in handler cho IGMP gọi vào ip-mcast functions. `northd/northd.c` build_mcast_groups() consume IGMP_Group khi compile Multicast_Group.

**Trường hợp sự cố thực tế.** Case 2024-Q2 OSP IPTV cluster (đã đề cập trong Phần 13.5.1c.5): 30% subscriber không nhận stream 239.1.1.1 do flow IGMP snoop trong stage trung gian decrement TTL=1 sai, packet IGMP report bị drop trước khi tới `output:CONTROLLER`. Triệu chứng: `ovn-sbctl list IGMP_Group` thấy chỉ 70% chassis có dòng; tcpdump trên VM thiếu vẫn thấy IGMP join đi ra nhưng không tới ovn-controller. Khắc phục: backport bug fix upstream điều chỉnh stage TTL handling.

**Bài tập synthetic.** Lab: tạo LS với 3 LSP + IGMP snooping enable (`ovn-nbctl set Logical_Switch <ls> other_config:mcast_snoop=true`); từ VM-A gửi `iperf -c 224.1.1.1 -u -T 32` join; sau 2-3s `sudo ovn-sbctl list IGMP_Group` thấy 1 dòng với `ports=[lsp-A]`; thêm VM-B join cùng group; dòng update ports=[lsp-A, lsp-B] (cùng chassis) hoặc tạo dòng thứ hai (khác chassis); VM-A leave; sau timeout (default 260s, có thể tune) ports giảm còn lsp-B.

**Lỗi thường gặp + tín hiệu chẩn đoán.** (a) IGMP_Group không xuất hiện sau khi VM join → flow snoop sai; tín hiệu: tcpdump VM thấy join nhưng `ovn-sbctl` không có dòng. (b) Membership leak (không leave) → ovn-controller pinctrl thread overload, log `"failed to handle igmp leave"`. (c) Cross-chassis aggregation sai (group cùng address có nhiều dòng cho cùng chassis) → schema corruption, restart SBDB.

**So sánh với hệ khác.** Tương đương Linux bridge `bridge mdb show` với IGMP snooping enable. Tương đương Cisco IGMP snooping table (`show ip igmp snooping groups dynamic`). Khác: IGMP_Group OVN cluster-wide qua SBDB Raft, mỗi chassis cùng nhìn membership; Linux bridge per-host. Tương đương VMware NSX-T `IGMP profile` với membership tracker (concept giống).

### 13.5.1d.3 Bảng `BFD` — session phát hiện hỏng tunnel siêu nhanh

**Khái niệm cốt lõi.** `BFD` là bảng SBDB lưu **mỗi dòng cho một session BFD** (Bidirectional Forwarding Detection, RFC 5880) chạy trên một tunnel hoặc một logical_port giữa hai chassis. Loại: **OVSDB table**, **root** (`isRoot=true`), populated bởi `ovn-northd` khi compile NBDB Logical_Router_Port có cấu hình BFD hoặc khi HA_Chassis_Group cần fast failover; cập nhật cột `status` runtime bởi `controller/bfd.c` thông qua kernel BFD daemon hoặc OVS BFD backend. Mỗi dòng đại diện cho **một mạch giao tiếp keepalive** với chu kỳ siêu ngắn (~50-300ms) để phát hiện đứt tunnel trong ~150-900ms, nhanh hơn nhiều so với hello interval của BGP/OSPF (giây-chục giây).

**Lịch sử + bối cảnh ra đời.** Bảng được giới thiệu OVN 2.10 (2018) khi Mark Michelson + Numan Siddique (Red Hat) thêm tính năng BFD cho HA_Chassis_Group. Trước đó OVN dựa vào keepalive Geneve hoặc tunnel monitor heuristic của OVS để phát hiện chassis chết, chậm (10-30s), gây split-brain trên gateway active/standby. RFC 5880 BFD chuẩn hoá probe interval xuống 50ms, cho phép failover gateway distributed dưới 1s. OVN 21.06 mở rộng BFD cho Logical_Router_Port BFD-track BGP next-hop. OVN 22.03 ổn định schema 10 cột.

**Vị trí trong kiến trúc.** BFD ở layer SBDB, root table. Producer compile: `northd/ovn-northd.c` build_bfd_table() đọc NBDB BFD records (referenced by Logical_Router_Port hoặc HA_Chassis_Group), sinh dòng SBDB BFD. Producer runtime status: `controller/bfd.c` hàm `bfd_run()`, chassis nơi `logical_port` được claim chạy BFD session với peer được xác định bởi `dst_ip`, update `status`. Consumer: HA_Chassis_Group selection logic trong `controller/ha-chassis.c` đọc BFD status để biết chassis nào trong group còn sống.

**Vai trò + nhiệm vụ.** Ba vai trò: (1) **fast failure detection**, keepalive sub-second cho tunnel/logical port; (2) **trigger failover gateway**, HA_Chassis_Group dùng BFD status để chọn chassis active; (3) **fast feedback cho BGP** (OVN 21.06+), BFD-track next-hop, BGP withdraw route ngay khi BFD down thay vì đợi hold timer.

**Vì sao sinh ra.** Pre-2.10: HA_Chassis_Group failover dựa vào timeout SBDB connection từ ovn-controller (10-30s default). Vấn đề: SBDB timeout không phải tín hiệu data plane, chassis có thể mất kết nối SBDB nhưng tunnel vẫn lên (false positive failover) hoặc ngược lại (false negative). BFD đo chính xác data plane health qua tunnel keepalive.

**Vấn đề mà nó giải quyết.** (a) **Slow gateway failover** (10-30s) khi gateway active chết trong cluster lớn; (b) **False positive split-brain** khi SBDB timeout không reflect data plane reality; (c) **Asymmetric tunnel down**, một hướng up, một hướng down, chỉ BFD hai chiều phát hiện được.

**Tầm quan trọng.** Tier 2 medium. Cluster không dùng HA_Chassis_Group hoặc không yêu cầu sub-second failover thì bảng rỗng và không tác động. Cluster có distributed gateway port (north-south NAT, hardware VTEP, HA L3) thì bảng load-bearing — BFD sai = failover sai = traffic blackhole.

**Cơ chế hoạt động.** Vòng đời: (1) NBDB Logical_Router_Port có cột `bfd` ref tới NBDB BFD record với `dst_ip` của next-hop; (2) northd build_bfd_table() compile xuống SBDB BFD row với `src_port=<random 49152-65535>`, `disc=<random discriminator>`, `logical_port=<lrp>`, `dst_ip=<next-hop>`, `min_tx`, `min_rx`, `detect_mult`, `status=down` ban đầu; (3) chassis chủ của `logical_port` chạy `bfd_run()`, gọi xuống OVS BFD backend (`ovs-vsctl set Interface ovn-<peer> bfd:enable=true bfd:min_rx=...`); (4) OVS BFD trao đổi packet với peer mỗi `min_tx` ms; (5) sau `detect_mult` packet mất, OVS đánh dấu down; (6) ovn-controller đọc OVS BFD status và update SBDB `status=down`; (7) HA_Chassis_Group logic phát hiện `status=down`, chọn chassis kế tiếp trong group; (8) traffic chuyển sang chassis mới trong < 1s.

**Phân loại cột (verified branch-22.03).** Mười cột: `src_port` (integer 49152-65535, ephemeral source port của BFD packet), `disc` (integer, BFD discriminator local), `logical_port` (string, LSP gắn BFD), `dst_ip` (string, peer IP), `min_tx` (integer ms, desired tx interval), `min_rx` (integer ms, required rx interval), `detect_mult` (integer, multiplier, số packet mất trước khi declare down), `status` (enum down|init|up|admin_down), `external_ids` (map), `options` (map). `isRoot=true`, một index `["logical_port","dst_ip","src_port","disc"]` đảm bảo unique session.

**Cách kỹ sư vận hành thành thạo.** Bốn kỹ năng: (1) `sudo ovn-sbctl list BFD` snapshot tất cả session; (2) `sudo ovn-sbctl find BFD status=down` lọc session đang chết, tín hiệu sớm cho gateway failover sắp xảy ra; (3) `sudo ovn-sbctl --columns=logical_port,dst_ip,status,detect_mult list BFD` table view; (4) đối chiếu OVS BFD: `sudo ovs-vsctl --columns=name,bfd_status list Interface | grep -A1 ovn-` để khớp giữa SBDB và datapath thật. Decision: nếu HA gateway flap, kiểm BFD status, nếu status flap up/down theo chu kỳ phù hợp với min_rx × detect_mult, có thể network jitter; tăng detect_mult.

**Quy trình sử dụng.** Workflow setup BFD cho LRP: (1) tạo NBDB BFD record `ovn-nbctl create BFD logical_port=lrp-public dst_ip=10.0.0.1 min_tx=300 min_rx=300 detect_mult=3`; (2) attach vào LRP `ovn-nbctl set Logical_Router_Port lrp-public bfd="@bfd-uuid"`; (3) đợi 5-10s; (4) `ovn-sbctl list BFD` xác nhận dòng tạo với `status=init` rồi chuyển `up`; (5) verify OVS BFD: `ovs-vsctl --columns=bfd,bfd_status list Interface ovn-<peer>` thấy bfd_status forwarding=true; (6) test failover: `ip link set <peer-tunnel> down` trên peer, kiểm SBDB BFD status chuyển down trong < detect_mult × min_rx ms.

**Khi xảy ra sự cố.** Symptom: (a) "Gateway HA failover quá chậm (>5s)" → kiểm BFD min_rx × detect_mult, nếu = 30 × 100 = 3000ms thì thực tế khớp; (b) "BFD status=admin_down dù không disable" → ovn-controller bug, restart; (c) "BFD flap mỗi vài giây" → network jitter hoặc tunnel MTU sai khiến BFD packet bị fragment.

**Liên quan mật thiết.** Coupling chặt: `Logical_Router_Port` (NBDB nguồn ref BFD record), `HA_Chassis_Group` + `HA_Chassis` (consumer status để chọn active), `controller/bfd.c` (producer status), `controller/ha-chassis.c` (consumer logic), OVS Interface table (datapath BFD backend). RFC 5880 chuẩn protocol; cross-link Phần 11.0 §Geneve (BFD chạy bên trong tunnel Geneve giữa hai chassis).

**Khác biệt giữa các phiên bản.** OVN 2.10: bảng giới thiệu cho HA_Chassis_Group fast failover. OVN 21.06: hỗ trợ BFD-track BGP next-hop (qua ovn-bgp-agent integration). OVN 22.03: schema ổn định 10 cột (verified). OVN 24.03: tối ưu BFD scheduling khi cluster có >1000 session.

**Cách quan sát + xác minh.** Lệnh: `sudo ovn-sbctl list BFD`, `sudo ovn-sbctl --columns=logical_port,dst_ip,status,min_tx,min_rx,detect_mult list BFD`, `sudo ovn-sbctl find BFD status=down`. Datapath OVS BFD: `sudo ovs-vsctl --columns=name,bfd,bfd_status list Interface | head -30`. Real-time: `sudo ovsdb-client monitor unix:/var/run/ovn/ovnsb_db.sock OVN_Southbound BFD`. Đối chiếu RFC 5880 packet: `sudo tcpdump -i ovn-<peer> -nn 'udp port 3784'` (BFD control port).

**Source code tham chiếu (verified branch-22.03).** `ovn-sb.ovsschema` table `BFD` 10 cột + index. `controller/bfd.c` hàm `bfd_register_ovs_idl()` (đăng ký IDL), `bfd_calculate_active_tunnels()` (xác định session nào active per chassis), `bfd_run()` (orchestration sync với OVS backend). Hàm `bfd_status_run` cập nhật SBDB từ OVS bfd_status. `northd/ovn-northd.c` build_bfd_table() compile NBDB → SBDB.

**Trường hợp sự cố thực tế.** Case 2023-Q4 production OSP: cluster với HA_Chassis_Group 3 node đột nhiên failover liên tục mỗi 5 phút. Điều tra: `ovn-sbctl list BFD` thấy status flap up→down→up với chu kỳ ~30 giây. Root cause: MTU mismatch, tunnel Geneve MTU 1450 nhưng BFD packet chuẩn 1500 bị fragment ở physical fabric, fragment loss rate ~1% trên link xấu khiến BFD detect_mult=3 (3 × 100ms = 300ms timeout) bị trip thường xuyên. Khắc phục: tăng `Open_vSwitch.external_ids:ovn-encap-mtu=1450` để OVS biết giới hạn, BFD packet pad ngắn hơn fit MTU; flap dừng.

**Bài tập synthetic.** Lab: tạo 2-chassis lab + 1 LRP với BFD record (min_tx=200, min_rx=200, detect_mult=3); verify `ovn-sbctl list BFD` status=up; trên peer chassis chạy `sudo iptables -A INPUT -p udp --dport 3784 -j DROP` chặn BFD packet; sau ~600ms (3 × 200ms) status chuyển down; remove iptables rule; sau ~600ms status quay up. Biến thể: thay min_rx=50, detect_mult=3, đo failover thực tế qua continuous ping cross-chassis trong khi block BFD, đo downtime.

**Lỗi thường gặp + tín hiệu chẩn đoán.** (a) BFD flap chu kỳ ngắn → MTU mismatch hoặc CPU saturation chassis (BFD packet bị delay). (b) Status mãi `init` không chuyển `up` → peer không respond hoặc peer chưa bật BFD; tín hiệu: tcpdump thấy packet đi mà không có reply. (c) Status `admin_down` nhưng config không disable → ovn-controller chưa sync, restart. (d) Một chiều up một chiều init → asymmetric BFD; check cả hai bên cùng cấu hình.

**So sánh với hệ khác.** Tương đương BFD trên Cisco router (cùng RFC 5880, cùng concept session). Tương đương BFD của Juniper Junos (cùng chuẩn). Tương đương Linux kernel BFD daemon `bfdd` (chạy độc lập). Khác: SBDB BFD orchestrate qua OVN control plane, tham số được compile từ NBDB declarative thay vì cấu hình per-device CLI; OVS BFD backend trên data plane chạy trong userspace ovs-vswitchd thay vì kernel.

### 13.5.1d.4 Bảng `HA_Chassis` — chassis có ưu tiên trong nhóm HA gateway

**Khái niệm cốt lõi.** `HA_Chassis` là bảng SBDB lưu **mỗi dòng cho một (chassis, priority)** trong một HA_Chassis_Group, đại diện một thành viên có **mức ưu tiên** để được chọn làm active gateway cho distributed gateway port (chassisredirect / l3gateway). Loại: **OVSDB table**, **không phải root** (`isRoot=false`), chỉ tồn tại khi được tham chiếu bởi một HA_Chassis_Group qua cột `ha_chassis`. Chassis có priority cao nhất + còn sống (qua BFD) sẽ là active; còn lại standby. Khi active chết, chassis có priority cao kế tiếp đảm nhiệm.

**Lịch sử + bối cảnh ra đời.** Bảng giới thiệu OVN 2.10 (2018) cùng đợt thêm BFD và HA_Chassis_Group. Trước đó OVN dùng `gateway_chassis` per-LRP với priority inline trong cột, schema khó maintain khi cần share group across nhiều LRP. Pattern HA_Chassis_Group + HA_Chassis tách thành "group definition" + "members" cho phép một group dùng cho nhiều LRP, cập nhật một chỗ ảnh hưởng tất cả. Tác giả Numan Siddique (Red Hat). OVN 22.03 ổn định schema 3 cột.

**Vị trí trong kiến trúc.** HA_Chassis ở layer SBDB, non-root table. Producer compile: `northd/ovn-northd.c` build_ha_chassis_groups() đọc NBDB HA_Chassis_Group (tương ứng) sinh dòng SBDB HA_Chassis qua strong ref từ HA_Chassis_Group.ha_chassis. Consumer: `controller/ha-chassis.c` hàm `ha_chassis_group_is_active()` sắp xếp các HA_Chassis theo priority + filter qua BFD status, chọn active.

**Vai trò + nhiệm vụ.** Ba vai trò: (1) **đăng ký candidate** cho gateway HA group; (2) **mang priority** quyết định thứ tự takeover; (3) **bridge giữa HA_Chassis_Group và Chassis** — HA_Chassis tham chiếu Chassis (qua cột chassis weak ref), cho phép Chassis row độc lập với group membership.

**Vì sao sinh ra.** Pre-2.10 pattern `gateway_chassis` per-LRP không scale: cluster có 100 LRP cần đặt cùng HA group sẽ phải config 100 cột gateway_chassis với cùng list, sai một chỗ là split-brain. HA_Chassis_Group + HA_Chassis tách concern: định nghĩa group một lần, gắn vào nhiều LRP qua reference.

**Vấn đề mà nó giải quyết.** (a) **Config duplication** khi nhiều LRP cùng chia sẻ HA topology; (b) **Atomic update**, đổi priority một dòng HA_Chassis ảnh hưởng tất cả LRP gắn group; (c) **Selection deterministic**, sắp xếp priority + BFD status ra một active duy nhất, tránh split-brain.

**Tầm quan trọng.** Tier 2 medium. Cluster không có distributed gateway port (cluster nhỏ, no NAT) bảng rỗng. Cluster production OpenStack với DVR + gateway HA bảng load-bearing, sai priority hoặc thiếu dòng = traffic north-south rớt khi failover.

**Cơ chế hoạt động.** Vòng đời: (1) admin tạo NBDB HA_Chassis_Group `name=group-A` với members; (2) northd build_ha_chassis_groups() compile xuống SBDB HA_Chassis_Group + tạo HA_Chassis row cho mỗi member với strong ref; (3) gắn vào LRP qua `Logical_Router_Port.ha_chassis_group=@group-A`; (4) trên mỗi chassis, `controller/ha-chassis.c::ha_chassis_get_ordered()` sắp xếp HA_Chassis theo priority desc; (5) filter BFD status: chỉ chassis có BFD up tới các member khác mới candidate; (6) `ha_chassis_group_is_active()` xác định nếu self-chassis là top + healthy → claim chassisredirect Port_Binding; (7) khi BFD báo active chết, tự động re-evaluate, candidate kế tiếp claim.

**Phân loại cột (verified branch-22.03).** Ba cột: `chassis` (uuid weak ref Chassis, weak để xoá Chassis tự reset khỏi HA_Chassis), `priority` (integer 0-32767, cao = ưu tiên hơn), `external_ids` (map). `isRoot=false`, không có index. Tham chiếu mạnh từ `HA_Chassis_Group.ha_chassis` (strong ref vào HA_Chassis); xoá HA_Chassis_Group sẽ xóa tất cả HA_Chassis con vì strong ref.

**Cách kỹ sư vận hành thành thạo.** Ba kỹ năng: (1) `sudo ovn-sbctl list HA_Chassis` snapshot members; (2) `sudo ovn-sbctl list HA_Chassis_Group` xem cấu trúc group + ref tới HA_Chassis; (3) `sudo ovn-sbctl --columns=chassis,priority list HA_Chassis | sort -k3 -nr` sắp xếp priority desc để biết active candidate hiện tại. Decision: nếu gateway flap, so sánh priority + check BFD status của top candidate, nếu BFD up + priority cao nhất mà không active → bug ovn-controller, kiểm log.

**Quy trình sử dụng.** Workflow setup HA gateway: (1) `ovn-nbctl ha-chassis-group-add hagroup1`; (2) `ovn-nbctl ha-chassis-group-add-chassis hagroup1 hv1 100`, `ovn-nbctl ha-chassis-group-add-chassis hagroup1 hv2 50`, `ovn-nbctl ha-chassis-group-add-chassis hagroup1 hv3 25`; (3) tạo LRP với `options:redirect-chassis` thay bằng `ha_chassis_group=@group-A`; (4) `ovn-sbctl list HA_Chassis` xác nhận 3 dòng tạo với priority đúng; (5) verify chassisredirect Port_Binding bị claim bởi hv1 (priority cao nhất, BFD up); (6) test failover: shutdown hv1, sau ~600ms (BFD timeout) chassisredirect chuyển hv2.

**Khi xảy ra sự cố.** Symptom: (a) "Gateway không chuyển khi active chết" → kiểm HA_Chassis của next candidate có BFD up không; (b) "Hai chassis cùng claim chassisredirect (split-brain)" → kiểm BFD status, có thể network partition, hai bên đều thấy bên kia chết; (c) "Priority sai sau ovn-nbctl change" → northd compile chậm, đợi 5s hoặc restart northd.

**Liên quan mật thiết.** Coupling rất chặt: `HA_Chassis_Group` (parent strong ref qua `ha_chassis`), `Chassis` (weak ref qua `chassis`), `BFD` (consumer status để filter healthy candidate), `Port_Binding` type chassisredirect/l3gateway (consumer chính, chính bảng này quyết định ai claim chassisredirect Port_Binding), `controller/ha-chassis.c` (consumer logic).

**Khác biệt giữa các phiên bản.** OVN 2.10: bảng giới thiệu cùng HA_Chassis_Group + BFD. OVN 21.06: thêm logic prefer local chassis nếu cùng priority. OVN 22.03: ổn định 3 cột (verified). OVN 24.03: tối ưu compile khi group có >50 members.

**Cách quan sát + xác minh.** Lệnh: `sudo ovn-sbctl list HA_Chassis`, `sudo ovn-sbctl list HA_Chassis_Group`, `sudo ovn-sbctl --columns=chassis,priority list HA_Chassis | sort -k3 -nr`. Đối chiếu chassisredirect Port_Binding: `sudo ovn-sbctl find Port_Binding type=chassisredirect` → cột `chassis` chính là chassis active hiện tại; phải khớp với HA_Chassis priority cao nhất + BFD up. Real-time: `sudo ovsdb-client monitor unix:/var/run/ovn/ovnsb_db.sock OVN_Southbound HA_Chassis HA_Chassis_Group Port_Binding` để stream event failover.

**Source code tham chiếu (verified branch-22.03).** `ovn-sb.ovsschema` table `HA_Chassis` 3 cột (không index). `controller/ha-chassis.c` hàm `compare_chassis_prio_()` (so sánh priority cho sort), `get_ordered_ha_chassis_list()` (sắp xếp theo priority + status), `ha_chassis_destroy_ordered()` (cleanup), `is_local_chassis_only_candidate()` (kiểm self là duy nhất alive), `ha_chassis_group_is_active()` (decision point), `ha_chassis_group_contains()` (membership check), `ha_chassis_get_ordered()` (public API).

**Trường hợp sự cố thực tế.** Case 2024-Q1 cluster OSP DVR + HA gateway 3 node: sau khi datacenter migration, một chassis được restore với `system-id` cũ nhưng IP mới. HA_Chassis row giữ ref `chassis=<cũ>` weak ref nên không break, nhưng BFD record cũ có `dst_ip=<cũ>` không khớp. Triệu chứng: failover quá chassis đó luôn fail, traffic kẹt 30s đến khi northd recompile BFD. Khắc phục: explicit recreate HA_Chassis_Group + BFD record sau migration thay vì rely vào auto-update; document trong runbook DC migration.

**Bài tập synthetic.** Lab: 3-chassis lab + HA_Chassis_Group với priority 100, 50, 25; tạo LRP với ha_chassis_group; verify hv1 (100) là active qua `ovn-sbctl find Port_Binding type=chassisredirect`; shutdown hv1; trong 600ms hv2 (50) take over; bring hv1 back; nếu `options:auto-failback=true` (NBDB option), hv1 reclaim; nếu false, hv2 giữ active. Biến thể: thay priority hv2 thành 150 (cao hơn hv1) qua `ovn-nbctl ha-chassis-group-add-chassis hagroup1 hv2 150`, sau recompile, hv2 take over từ hv1.

**Lỗi thường gặp + tín hiệu chẩn đoán.** (a) Priority sai sau `nbctl` change → northd recompile lag; tín hiệu: `nbctl list HA_Chassis_Group` đúng nhưng `sbctl list HA_Chassis` chưa update; chờ 5s. (b) Active chassis không chuyển khi chết → BFD status không update; check `ovn-sbctl list BFD`. (c) Split-brain (hai active) → network partition; xử lý partition trước. (d) HA_Chassis row `chassis=[]` (empty weak ref) → Chassis row đã xóa, group cần fix.

**So sánh với hệ khác.** Tương đương VRRP priority (RFC 5798) trong Linux keepalived hoặc Cisco router. Tương đương HSRP priority của Cisco. Tương đương Pacemaker priority + constraint. Khác: HA_Chassis OVN tích hợp vào OVSDB declarative, tham số khai báo trong NBDB, compile xuống SBDB, ovn-controller chấp hành; không có protocol multicast advertisement như VRRP/HSRP, dựa vào BFD point-to-point + SBDB Raft consistent membership.

### 13.5.1d.5 Bảng `Connection` — cấu hình kết nối client tới SBDB

**Khái niệm cốt lõi.** `Connection` là bảng SBDB lưu **mỗi dòng cho một mục tiêu kết nối** (TCP/SSL/Unix socket) mà SBDB cluster chấp nhận từ client (ovn-controller, ovn-sbctl, ovn-northd). Loại: **OVSDB table**, **không phải root** (`isRoot=false`), tồn tại khi được tham chiếu bởi `SB_Global.connections`. Cùng shape với Connection trong NBDB và Connection trong vswitch schema, pattern chung của OVSDB cho "phía nghe của một database server". Mỗi dòng nói cho ovsdb-server: "lắng nghe trên target này, với tham số inactivity_probe + max_backoff như sau".

**Lịch sử + bối cảnh ra đời.** Bảng có từ OVS 2.0 (2014) trong vswitch schema; OVN tái sử dụng pattern này khi schema OVN sơ khai 2015. Pattern: tách config kết nối ra bảng riêng cho phép thay đổi listening endpoint runtime mà không restart ovsdb-server (động). OVN 21.06 thêm cột `read_only` cho cluster có node read-only (failover nhanh hơn cho traffic read). OVN 22.03 ổn định schema 9 cột.

**Vị trí trong kiến trúc.** Connection ở layer SBDB, non-root table. Producer: admin trực tiếp qua `ovn-sbctl set-connection ptcp:6642:0.0.0.0`, `ovn-sbctl` tự tạo Connection row + update `SB_Global.connections` ref. Consumer: `ovsdb-server` của SBDB đọc bảng khi boot + monitor; mở socket TCP/SSL/Unix tương ứng. Status runtime: cột `is_connected` + `status` ephemeral được ovsdb-server cập nhật sau khi mở socket thành công.

**Vai trò + nhiệm vụ.** Bốn vai trò: (1) **declare listening endpoint** cho SBDB (ovn-controller kết nối qua TCP 6642 mặc định); (2) **mang per-connection tunables** như `inactivity_probe` (timeout idle), `max_backoff` (delay reconnect); (3) **expose status runtime** (`is_connected`, `status`) cho admin debug; (4) **role-based access control** (cột `role`, `read_only`), phân biệt connection admin vs read-only.

**Vì sao sinh ra.** Static config trong file `/etc/ovn/conf.db` không cho phép admin thay listening port runtime. Bảng Connection cho phép `ovsdb-server` đọc config từ chính database nó serving, chicken-and-egg solved bằng bootstrapping qua command-line `--remote=db:OVN_Southbound,SB_Global,connections`.

**Vấn đề mà nó giải quyết.** (a) **Runtime config change**, đổi port lắng nghe không restart; (b) **Multi-listener**, một SBDB có thể nghe TCP 6642 + SSL 6642 + Unix socket cùng lúc; (c) **Per-listener tunable**, TCP có inactivity_probe khác Unix; (d) **HA-aware**, cluster Raft, mỗi node có Connection riêng nhưng cùng schema.

**Tầm quan trọng.** Tier 2 medium. Bảng load-bearing trong setup ban đầu (sai connection target = ovn-controller không tới được SBDB = cluster down). Sau khi setup ổn định, bảng ít touch, chỉ đụng khi đổi port, đổi protocol (TCP → SSL), hoặc tune inactivity_probe.

**Cơ chế hoạt động.** Vòng đời: (1) admin chạy `ovn-sbctl set-connection ptcp:6642:0.0.0.0`; (2) command tạo Connection row với `target="ptcp:6642:0.0.0.0"`, `inactivity_probe=60000`, `max_backoff=3000`; (3) update `SB_Global.connections={@conn-uuid}`; (4) ovsdb-server monitor SB_Global.connections, thấy thay đổi; (5) mở/đóng socket tương ứng; (6) cập nhật `is_connected=true`, `status:bound_port=6642` runtime; (7) khi client kết nối, `inactivity_probe` (default 60s) gửi echo, không phản hồi → đóng connection; client `max_backoff=3000` đợi 3s trước reconnect.

**Phân loại cột (verified branch-22.03).** Chín cột: `target` (string, format `<protocol>:<param>` ví dụ `ptcp:6642`, `pssl:6642`, `punix:/var/run/ovn/ovnsb_db.sock`), `max_backoff` (integer min 1000 ms), `inactivity_probe` (integer ms, 0 = disable), `read_only` (boolean), `role` (string, ACL role), `other_config` (map), `external_ids` (map), `is_connected` (boolean ephemeral), `status` (map ephemeral với key `bound_port`, `n_connections`, `sec_since_connect`, `sec_since_disconnect`, `state`). `isRoot=false`, một index `["target"]` đảm bảo unique listener.

**Cách kỹ sư vận hành thành thạo.** Bốn kỹ năng: (1) `sudo ovn-sbctl list Connection` snapshot listener; (2) `sudo ovn-sbctl get-connection` xem target hiện tại của SBDB; (3) `sudo ovn-sbctl --columns=target,is_connected,status list Connection` view runtime status; (4) `sudo ovn-sbctl set-connection <new-target>` đổi listener runtime. Decision: nếu ovn-controller log `"connection refused"`, kiểm Connection.target có trỏ đúng IP/port không; kiểm `is_connected=true` không.

**Quy trình sử dụng.** Workflow đổi từ TCP plain sang SSL: (1) chuẩn bị certificate `ovn-pki sb.cluster sign sbdb`; (2) `sudo ovn-sbctl set-ssl /etc/ovn/key.pem /etc/ovn/cert.pem /etc/ovn/cacert.pem`; (3) `sudo ovn-sbctl del-connection`; (4) `sudo ovn-sbctl set-connection pssl:6642:0.0.0.0`; (5) `sudo ovn-sbctl list Connection` xác nhận target mới + `is_connected=true`; (6) cập nhật `Open_vSwitch.external_ids:ovn-remote=ssl:<sbdb-vip>:6642` trên mọi chassis; (7) restart ovn-controller để pickup SSL; (8) verify connectivity bằng `sudo ovn-sbctl list Chassis | wc -l`, phải khớp số node.

**Khi xảy ra sự cố.** Symptom: (a) "ovn-controller không kết nối được SBDB" → kiểm Connection.target + is_connected; (b) "connection drop liên tục" → inactivity_probe quá ngắn, network latency cross-DC > probe interval; tăng inactivity_probe lên 300000 (5 phút); (c) "ovn-sbctl hang" → Unix socket Connection bị xóa khỏi `SB_Global.connections`, restore qua `ovn-sbctl --no-leader-only set Connection ...`.

**Liên quan mật thiết.** Coupling: `SB_Global` (parent strong ref qua `connections`), ovsdb-server process (consumer config), `Open_vSwitch.external_ids:ovn-remote` trên chassis (client config khớp với Connection.target). Tương đương Connection trong NBDB + vswitch schema, cùng pattern, cùng cột.

**Khác biệt giữa các phiên bản.** OVS 2.0: pattern xuất phát. OVN 21.06: thêm `read_only` boolean. OVN 22.03: schema 9 cột (verified). OVN 22.06+: không thêm cột. Tương lai (24.x+): có thể thêm `bind_address_ipv6` cho dual-stack, hiện chưa có.

**Cách quan sát + xác minh.** Lệnh: `sudo ovn-sbctl list Connection`, `sudo ovn-sbctl get-connection`, `sudo ovn-sbctl --columns=target,is_connected,status list Connection`. Đối chiếu OS-level: `sudo ss -tlnp | grep 6642` (TCP listener), `sudo ss -xl | grep ovnsb_db.sock` (Unix listener), `sudo lsof -p $(pidof ovsdb-server) | grep LISTEN`. Real-time: `sudo ovsdb-client monitor unix:/var/run/ovn/ovnsb_db.sock OVN_Southbound Connection` để stream khi admin đổi listener.

**Source code tham chiếu (verified branch-22.03).** `ovn-sb.ovsschema` table `Connection` 9 cột + index `["target"]`. Logic xử lý chung trong thư viện OVSDB IDL: `ovsdb/jsonrpc-server.c` hàm `ovsdb_jsonrpc_server_set_remotes()` đọc Connection.target và mở socket; `lib/jsonrpc.c::jsonrpc_session_create()` xử lý max_backoff cho client side. Schema chia sẻ pattern: `vswitchd/vswitch.ovsschema` Connection cùng cột (cross-link Phần 9.1).

**Trường hợp sự cố thực tế.** Case 2023-Q3 cluster Raft 3-node SBDB: admin chạy `ovn-sbctl del-connection` trên leader để dọn nhầm listener cũ, vô tình xóa cả Unix socket connection. Triệu chứng: `ovn-sbctl` từ cùng host treo (Unix socket gone), follower nodes mất kết nối tới leader RPC. Khắc phục khẩn: ssh vào ovsdb-server của leader, `ovs-appctl -t /var/run/ovn/ovnsb_db.ctl ovsdb-server/add-remote punix:/var/run/ovn/ovnsb_db.sock` để re-create Unix socket runtime; sau đó `ovn-sbctl` dùng được, `ovn-sbctl set-connection ...` restore TCP listener bình thường. Long-term: scripts admin luôn `set-connection` trước khi `del-connection`, không bao giờ leave SBDB không listener.

**Bài tập synthetic.** Lab: cluster 1-node SBDB; `ovn-sbctl get-connection` xem target hiện tại; `ovn-sbctl list Connection` xem chi tiết; thay inactivity_probe: `ovn-sbctl set Connection . inactivity_probe=120000`; verify thay đổi áp dụng runtime, không cần restart ovsdb-server; thử thêm SSL listener parallel: `ovn-sbctl set-connection pssl:6642 ptcp:6643` (multi-target); `ss -tlnp | grep ovsdb` thấy cả hai port lắng nghe.

**Lỗi thường gặp + tín hiệu chẩn đoán.** (a) `is_connected=false` dù target đúng → port bị occupied bởi process khác; check `ss -tlnp | grep <port>`. (b) Inactivity_probe trip cluster Raft cross-DC → tăng inactivity_probe phù hợp RTT. (c) SSL connection fail dù cert đúng → check `ovsdb-server.log` cho "SSL handshake failed", thường là CA mismatch. (d) Unix socket mất sau crash → recreate qua `ovs-appctl ovsdb-server/add-remote`.

**So sánh với hệ khác.** Tương đương `bind` directive trong PostgreSQL `postgresql.conf` (listen_addresses + port). Tương đương `listener` trong RabbitMQ config. Tương đương `mysqld --bind-address` của MySQL. Khác: Connection của OVSDB là bảng động trong chính database nó serving, đổi runtime không restart; PostgreSQL/MySQL phải reload config + đôi khi restart. Tương đương Etcd `--listen-client-urls` (cùng concept tách listener config) nhưng etcd dùng flag command-line, không bảng động.

## 13.5.2 Type `""` (default VIF) — VM tap claim

**Mặc định** — Port_Binding của VM tap (virtual interface). Type là empty string `""`. Type phổ biến nhất (>95% Port_Binding trong DC triển khai).

### Claim condition

ovn-controller trên chassis X claim Port_Binding type `""` khi:
1. OVS bridge `br-int` có port (tap interface) với `external_ids:iface-id=<logical_port>`.
2. Port_Binding chưa claim bởi chassis khác (chassis=NULL hoặc chassis=this).

### Provisioning workflow

```
1. CMS orchestrator (Nova/libvirt/Kubelet) allocate VM.
2. CMS call: ovn-nbctl lsp-add <ls> <lsp_name>
   → NBDB: 1 row Logical_Switch_Port
3. ovn-northd compile:
   → SBDB: 1 row Port_Binding (type="", chassis=NULL)
4. CMS plug tap vào br-int:
   → ovs-vsctl add-port br-int tap-vmX
   → ovs-vsctl set Interface tap-vmX external_ids:iface-id=<lsp_name>
5. ovn-controller trên chassis X detect tap + iface-id match:
   → SBDB: UPDATE Port_Binding SET chassis=<X>
6. ovn-controller install OpenFlow rule match port cho LSP.
7. VM now functional.
```

### Multichassis binding (OVN 22.06+)

Khi VM live migration, OVN 22.06+ hỗ trợ dual-chassis binding trong window migration:

```
source chassis A: Port_Binding.chassis=A + additional_chassis=[]
→ migration started:
   destination chassis B prepare + claim as additional
   Port_Binding.chassis=A + additional_chassis=[B]
   Traffic replicate cả 2 chassis trong window 3-18s (Part 19.0)
→ migration done:
   source release
   Port_Binding.chassis=B + additional_chassis=[]
```

Trước OVN 22.06, live migration gây 15-30s downtime. Multichassis + `activation-strategy=rarp` (OVN 24.03+) giảm xuống < 500ms.

> **Chủ đề trọng tâm:** Claim logic dựa **entirely** vào iface-id match. Nếu hypervisor orchestrator không set `external_ids:iface-id` khi plug tap, ovn-controller không claim → VM không có network. Đây là pain point phổ biến: manual libvirt provisioning thiếu iface-id, hoặc CNI plugin bug. Debug bắt đầu bằng `ovs-vsctl get Interface tap-vmX external_ids:iface-id` — nếu empty, fix ở orchestrator layer, không phải OVN layer.

## 13.5.3 Type `localnet` — provider network extension

LSP type `localnet` connect logical switch đến **physical network** (provider network).

### Đặc điểm

- **Không claim chassis** — available trên tất cả chassis có bridge mapping tương ứng.
- Mỗi chassis map `options:network_name` tới OVS bridge qua `external_ids:ovn-bridge-mappings`.
- ovn-controller cài đặt patch port giữa `br-int` và external bridge (br-ex/br-provider).

### Use case

Provider network (OpenStack flat provider), shared internet gateway, VLAN trunk tới physical switch upstream.

```bash
[admin]# ovn-nbctl ls-add ls-provider-public
[admin]# ovn-nbctl lsp-add ls-provider-public provider-port
[admin]# ovn-nbctl lsp-set-type provider-port localnet
[admin]# ovn-nbctl lsp-set-addresses provider-port unknown
[admin]# ovn-nbctl lsp-set-options provider-port network_name=physnet1
```

Trên chassis với `ovn-bridge-mappings=physnet1:br-ex`:
- ovn-controller tạo patch port pair `patch-provnet-physnet1` ↔ `patch-br-ex-to-br-int`.
- Traffic destined tới LSP type `localnet` output qua patch → `br-ex` → physical NIC.

### VLAN tagging

Khi provider network cần VLAN tag:

```bash
[admin]# ovn-nbctl set Logical_Switch_Port provider-port tag=100
```

Flow compile push VLAN 100 trước khi output patch. Upstream switch thấy frame VLAN 100 → forward đúng.

## 13.5.4 Type `chassisredirect` + `l3gateway` — north-south NAT

**Distributed Logical Router (LR)** xử lý L2/L3 chuyển tiếp distributed trên mọi chassis (mỗi chassis có local OpenFlow rule). Nhưng **north-south NAT** không distributed được — một external IP chỉ thuộc một chassis vật lý tại một thời điểm (ARP announce, routing table).

### `chassisredirect` Port_Binding

Là "shadow" của Logical Router Port (LRP) trên **gateway chassis**. Traffic north-bound redirect qua chassisredirect → gateway chassis xử lý NAT → `br-ex`.

```bash
[admin]# ovn-nbctl lrp-set-gateway-chassis lrp-external chassis-1 100
[admin]# ovn-nbctl lrp-set-gateway-chassis lrp-external chassis-2 50
```

`ovn-northd` sinh Port_Binding type `chassisredirect` với `ha_chassis_group` reference tới HA_Chassis_Group row chứa list chassis sorted by priority.

### Failover via BFD

HA_Chassis_Group dùng BFD để phát hiện failure. Khi chassis-1 (priority 100) fail:
- BFD session timeout (default 5 giây, tunable xuống 1 giây).
- HA_Chassis_Group reelect chassis-2 là active.
- Port_Binding chassisredirect update chassis field.
- ovn-controller trên chassis-2 cài đặt OpenFlow rule cho NAT.

Downtime typical 3-5s với default BFD, < 1s với `bfd_min_rx=100ms`.

Xem Part 13.6 cho BFD deep-dive.

### `l3gateway` Port_Binding

Biến thể khác cho gateway routing không dùng distributed LR. Rarely used post OVN 20.06 (distributed LR đã stable).

```
LS tenant1 LSP `ls-to-router` (type=router, connected LRP lrp-tenant1)
→ northd compile → Port_Binding `ls-to-router` type=patch

```

## 13.5.6 Type `l2gateway` — hardware VTEP extension

LSP gateway từ OVN logical network đến **hardware L2 switch** (via VTEP protocol, RFC 7047 extension).

### Use case

Extend OVN logical switch đến hardware VTEP (Cisco Nexus 9K, Arista 7280 với VXLAN VTEP). Pattern ESR (Extended Service Range) — OVN VM trong cùng broadcast domain với bare-metal server sau hardware switch.

```bash
[admin]# ovn-nbctl lsp-set-type vif-hw-vtep l2gateway
[admin]# ovn-nbctl lsp-set-options vif-hw-vtep network_name=physnet1 \
         l2gateway-chassis=chassis-1
```

`l2gateway-chassis` chỉ định chassis xử lý VTEP protocol với hardware switch. ovn-controller trên chassis đó maintain VTEP OVSDB connection với hardware switch.

Trade-off: thêm operational complexity (VTEP protocol + hardware switch firmware compat) nhưng cho phép hybrid physical-virtual topology.

## 13.5.7 Type `localport` — per-chassis metadata service

**Đặc biệt** — Port_Binding tồn tại trên **mọi chassis** với cùng MAC + IP. Traffic destined tới localport KHÔNG đi qua tunnel — stay local trên chassis nguồn.

### Use case: metadata service

OpenStack Nova + Kubernetes đều cần metadata endpoint (`169.254.169.254` IANA link-local). Mỗi VM truy cập metadata qua HTTP GET. Thay vì route metadata traffic tập trung (single gateway chassis), OVN dùng localport:

```bash
[admin]# ovn-nbctl lsp-add ls-tenant metadata-port
[admin]# ovn-nbctl lsp-set-type metadata-port localport
[admin]# ovn-nbctl lsp-set-addresses metadata-port \
         "fa:16:3e:00:00:01 169.254.169.254"
```

Mỗi chassis:
- ovn-controller cài đặt OpenFlow rule khớp với dst=169.254.169.254 → output port internal (local proxy agent).
- Proxy agent (ví dụ Neutron metadata agent) handle HTTP → forward tới CMS metadata API.
- Reply về đúng VM nguồn, không cross-chassis.

Lợi ích: scale metadata service linear theo chassis count (không bottleneck một gateway).

## 13.5.8 Type `virtual` — VRRP shared address

OVN 21.03+ hỗ trợ `virtual` Port_Binding cho floating IP shared bởi nhiều VM (VRRP pattern).

```bash
[admin]# ovn-nbctl lsp-add ls-tenant vip-floating
[admin]# ovn-nbctl lsp-set-type vip-floating virtual
[admin]# ovn-nbctl lsp-set-options vip-floating \
         virtual-parents=vm-master,vm-backup virtual-ip=10.0.0.100
```

Khi một VM (vm-master) announce Gratuitous ARP cho 10.0.0.100, ovn-controller bind virtual port tới VM đó. Khi vm-master fail, VRRP re-announce từ vm-backup, ovn-controller switch binding. Smooth handover cho HA application.

Use case: HAProxy/Keepalived floating VIP, database cluster master IP.

```bash
[local-cluster]# ovn-ic-nbctl ts-add ts-shared
```

## 13.5.10 Anatomy Template A — `ovn-sbctl list Port_Binding`

**Bối cảnh khi chạy:** Debug "VM không có network" — step 2 của 5-step diagnostic (sau check NBDB LSP exists).

**Command và output** *(reproduced OVN 22.03.8, VM lsp_web_01 trên chassis-hypervisor-A)*:

```shell
$ sudo ovn-sbctl --columns=logical_port,chassis,type,tunnel_key,mac,up,options list Port_Binding lsp_web_01

logical_port    : "lsp_web_01"
chassis         : [<chassis_A_uuid>]
type            : ""
tunnel_key      : 4
mac             : ["aa:bb:cc:00:00:01 10.0.0.5"]
up              : true
options         : {requested-chassis="chassis-A"}
```

**9-attribute anatomy:**

| Field | Giá trị mẫu | Ý nghĩa | Red flag nếu |
|-------|-------------|---------|--------------|
| `logical_port` | `"lsp_web_01"` | Match với `ovs-vsctl get Interface tap external_ids:iface-id` | Mismatch = claim không trigger |
| `chassis` | `[<uuid>]` hoặc `[]` | Chassis đang claim. `[]` = unbound | Empty + VM running = claim fail (no iface-id trên OVS) |
| `type` | `""` / `localnet` / ... | 8 type. `""` = VIF | Mismatch với expected use case = misconfiguration |
| `tunnel_key` | `4` | Per-datapath unique ID | Collision (2 port cùng tunnel_key) = northd bug |
| `mac` | `["aa:bb:... 10.0.0.5"]` | MAC + addresses from NBDB | Thiếu IP = `addresses=unknown`, DHCP/ARP responder không work |
| `up` | `true` / `false` | Binding state | `false` = chassis claim nhưng interface down |
| `options:requested-chassis` | `"chassis-A"` | Optional chassis pin | Conflict với physical location of VM |
| `options:activation-strategy` | `"rarp"` (OVN 24.03+) | Migration activation | Missing = fallback old migration behavior |
| `additional_chassis` | `[]` hoặc `[uuid]` | Multichassis binding (migration window) | Non-empty + no migration in progress = leaked binding |

**Kịch bản bẻ gãy:**

- **`chassis=[]` + VM running + no iface-id trên tap**: CMS không set iface-id. Fix: `ovs-vsctl set Interface tap-vmX external_ids:iface-id=lsp_web_01`. Sau 1-2s ovn-controller claim.
- **`chassis=[B]` nhưng VM đang ở chassis A**: stale claim từ chassis B (crashed hoặc network partition). Fix: restart ovn-controller A, hoặc manual `ovn-sbctl chassis-del B` nếu B chắc chắn dead.
- **`additional_chassis=[C]` stuck sau migration done**: OVN 22.06 migration edge case. Fix: manual restart ovn-controller trên C hoặc `ovn-sbctl set Port_Binding <uuid> additional_chassis=[]`.
- **`up=false` + `chassis=[A]`**: interface operationally down. Check `ovs-vsctl --columns=name,admin_state,link_state list Interface tap-vmX`. If `link_state=down` → physical-layer issue (link flapping).

   ```bash
   node1$ sudo ovn-nbctl ls-add ls-claim-test
   node1$ sudo ovn-nbctl lsp-add ls-claim-test vm-claim-test
   node1$ sudo ovn-nbctl lsp-set-addresses vm-claim-test "aa:bb:cc:de:ad:01 10.0.0.99"
   ```
2. Verify chassis=NULL:
   ```bash
   node1$ sudo ovn-sbctl --columns=logical_port,chassis,up list Port_Binding vm-claim-test

   ```
3. Monitor SBDB change real-time (ở terminal khác):
   ```bash
   node1$ sudo ovsdb-client -v monitor \

   ```
4. Trên node2, plug tap với iface-id:
   ```bash
   node2$ sudo ovs-vsctl add-port br-int tap-claim -- \
          set Interface tap-claim type=internal \
          external_ids:iface-id=vm-claim-test
   ```
5. Observe SBDB change trong < 1 giây:
   ```
   row "lsp_web_01" (vm-claim-test): chassis [] → [<node2-chassis-uuid>]
   ```
6. Re-check:
   ```bash
   node1$ sudo ovn-sbctl --columns=logical_port,chassis,up list Port_Binding vm-claim-test
   # Kỳ vọng: chassis=[<node2-uuid>], up=true
   ```
7. Unplug → observe release:
   ```bash
   node2$ sudo ovs-vsctl del-port br-int tap-claim
   # Observer thấy: chassis [<uuid>] → []
   node1$ sudo ovn-sbctl --columns=chassis,up list Port_Binding vm-claim-test
   # Kỳ vọng: chassis=[], up=false
   ```

**POE questions:**
- **Predict:** Nếu plug tap vào node1 đồng thời với node2, cả 2 chassis có claim không? SBDB trả lời ra sao?
- **Observe:** OVSDB transaction UPDATE là CAS (compare-and-swap). Chỉ 1 chassis thành công (whichever gửi transaction tới leader đầu tiên). Chassis kia get transaction conflict, retry, và phát hiện chassis đã set → không claim.
- **Explain:** Raft consensus ensure exactly-one claimer at a time. Nếu cần dual-chassis (migration scenario), phải dùng `additional_chassis` (OVN 22.06+) — separate column, không conflict với `chassis`.

**Cleanup:**
```bash
node1$ sudo ovn-nbctl ls-del ls-claim-test
```

## 13.5.12 Debug Port_Binding — 5-step diagnostic

Khi VM "không có network", truy vết Port_Binding qua 5 bước:

**Step 1 — NBDB LSP exists?**
```bash
node1$ sudo ovn-nbctl show ls-tenant1 | grep <vm-mac>
```
Nếu không có → CMS chưa tạo LSP. Fix ở CMS.

**Step 2 — SBDB Port_Binding exists?**
```bash
node1$ sudo ovn-sbctl --columns=logical_port,chassis,up,type list Port_Binding <lsp_name>
```
Nếu không có → ovn-northd chưa compile. Check `ovn-northd` log + status.

**Step 3 — OVS tap có iface-id match?**
```bash
node2$ sudo ovs-vsctl --columns=name,external_ids list Interface | grep <lsp_name>
```
Nếu không có → CMS không set iface-id. Fix ở CMS.

**Step 4 — Port_Binding.chassis claimed?**
```bash
node1$ sudo ovn-sbctl --columns=chassis list Port_Binding <lsp_name>
```
Nếu empty sau 5s từ step 3 → ovn-controller crash hoặc OVSDB connection dead. Check `ovn-appctl connection-status`.

**Step 5 — OpenFlow rule cho VM installed?**
```bash
node2$ sudo ovs-ofctl -O OpenFlow13 dump-flows br-int | grep <vm-mac>
```
Nếu không có → ovn-controller crashed sau claim. Check `/var/log/ovn/ovn-controller.log`.

> **Hiểu sai phổ biến:** *"Port_Binding tồn tại trong NBDB."*
>
> **Thực tế:** Port_Binding chỉ tồn tại trong **SBDB**, không phải NBDB. NBDB chứa `Logical_Switch_Port` (LSP ý đồ cấu hình). `ovn-northd` compile LSP → Port_Binding trong SBDB. Nhầm lẫn này thường gây confusion khi query wrong table: `ovn-nbctl list Port_Binding` error vì NBDB không có table đó. Đúng: `ovn-sbctl list Port_Binding`.

---

## 13.5.13 Điểm cốt lõi cần nhớ

Thứ nhất, Port_Binding là link giữa logical port (NBDB LSP/LRP) và physical chassis (SBDB runtime state). 8 type phục vụ use case khác nhau: default VIF (95% workload) + localnet (provider network) + chassisredirect (gateway) + patch (inter-datapath) + l2gateway (VTEP) + localport (metadata) + virtual (VRRP) + remote (inter-cluster).

Thứ hai, default VIF claim dựa **entirely** trên `external_ids:iface-id` khớp với giữa OVS port và NBDB LSP. Missing iface-id = no claim = no network. Đây là pain point phổ biến nhất.

Thứ ba, `chassisredirect` + HA_Chassis_Group + BFD là pattern north-south NAT. Failover 3-5s default, < 1s tuned BFD. Chỉ một chassis active tại một thời điểm (không distributed như Logical Router).

Thứ tư, `localport` tồn tại trên mọi chassis — dùng cho metadata service scale linear. Traffic không cross-chassis.

Thứ năm, 5-step Port_Binding diagnostic (NBDB LSP → SBDB PB → OVS iface-id → claim chassis → OpenFlow rule) là workflow chuẩn debug "VM không có network". Biết sequence này giảm MTTR từ giờ xuống phút.

Thứ sáu, Port_Binding chỉ tồn tại trong SBDB (không phải NBDB). Query đúng table khi debug.

## 13.5.14 Tham chiếu nhanh các bảng SBDB liên quan tới Port_Binding

> 10 bảng SBDB chạy song hành với Port_Binding để hoàn thiện ngữ cảnh runtime: Chassis, Encap, MAC_Binding, FDB, Service_Monitor, Multicast_Group, IGMP_Group, BFD, HA_Chassis, Connection. Verified upstream OVN branch-22.03 `ovn-sb.ovsschema` via WebFetch (31 bảng tổng cộng, liệt kê dưới đây).

### Tổng quan các bảng SBDB (verified `ovn-sb.ovsschema` branch-22.03)

| Table | Purpose (verified upstream) |
|-------|---------------------------|
| Chassis | Physical/virtual network nodes với encapsulation info |
| Chassis_Private | Private chassis state + config tracking |
| Encap | Tunnel encapsulation types (geneve, stt, vxlan) + parameters |
| MAC_Binding | Learned ARP/ND mappings (IP to MAC) |
| FDB | Forwarding database MAC learning entries |
| Service_Monitor | Health monitoring cho LB backends |
| Multicast_Group | Multicast membership + tunnel key mapping |
| IGMP_Group | IGMP membership tracking per datapath/chassis |

### Tóm tắt 10 bảng SBDB đồng hành

#### Chassis

**Cơ chế hoạt động.** Chassis row được tạo theo yêu cầu khi ovn-controller khởi động. Cột chính: name (chassis ID, thường là hostname), encaps (tham chiếu Encap rows), hostname, external_ids (ví dụ system-id, ovn-bridge-mappings).

**Khác biệt giữa các phiên bản.** OVN 2.4 ổn định. 22.03 bổ sung `other_config:is-remote`.

**Trường hợp sự cố thực tế.** Phải prune Chassis row sau khi node decommission; row cũ còn sót sẽ chặn không cho add lại cùng tên.

**Bài tập.** `ovn-sbctl list Chassis | head` để quan sát.

**So sánh với hệ khác.** Tương đương Kubernetes Node object. NSX-T Transport Node có ngữ nghĩa gần.

#### Encap

**Cơ chế hoạt động.** Cấu hình tunnel encap: type (geneve/vxlan/stt) + ip + options. Được tham chiếu bởi mảng `Chassis.encaps`.

**Khác biệt giữa các phiên bản.** OVN 2.4 ổn định.

**Bài tập.** `ovn-sbctl list Encap` để quan sát.

**So sánh với hệ khác.** Tương đương cấu hình IPIP/VXLAN trong Kubernetes CNI.

#### MAC_Binding

**Cơ chế hoạt động.** Lưu binding ARP/ND học được trên Logical Router. Cột chính: logical_port (LRP), ip, mac, datapath, timestamp.

**Khác biệt giữa các phiên bản.** OVN 2.4 ban đầu. 22.03 thêm cột timestamp và giới hạn `MAX_MAC_BINDING` trên mỗi LR.

**Trường hợp sự cố thực tế.** Production: MAC_Binding cũ kẹt gây routing blackhole (xem case PMTUD ở Phần 19.0 + REF Section 4 Scenario 11).

**Bài tập.** `ovn-sbctl find MAC_Binding ip=10.0.0.5` để truy vấn.

**So sánh với hệ khác.** Tương đương ARP cache `ip neigh show` trong Linux kernel. Cisco IOS ARP table có ngữ nghĩa gần.

#### FDB

**Cơ chế hoạt động.** Học MAC trên Logical Switch. Cột chính: dp_key, mac, port_key. ovn-controller populate qua pipeline `REGBIT_LOOKUP_FDB` / `REGBIT_HIT_LOOKUP_FDB` (xem Phần 13.18).

**Khác biệt giữa các phiên bản.** OVN 22.03 chuẩn hoá bảng FDB (thay learning ở bridge OVS trước đó).

**Trường hợp sự cố thực tế.** FDB cũ → unknown unicast flood (xem case LS broadcast storm ở Phần 13.2).

**Bài tập.** `ovn-sbctl list FDB | head` để quan sát.

**So sánh với hệ khác.** Tương đương trực tiếp `bridge fdb show` của Linux bridge.

#### Service_Monitor

**Cơ chế hoạt động.** Một row health monitor cho mỗi LB endpoint. Cột chính: ip, protocol (tcp/udp/icmp), port, src_ip, src_mac, status (online/offline/error).

**Khác biệt giữa các phiên bản.** OVN 22.03 giới thiệu. 24.03 mở rộng options theo từng protocol.

**Trường hợp sự cố thực tế.** Backend LB tự động bị loại khi health check fail; pattern ELB production.

**Bài tập.** `ovn-sbctl list Service_Monitor` để quan sát.

**So sánh với hệ khác.** Tương đương trực tiếp AWS ELB health check. Linux LVS `ipvsadm --get-state` có ngữ nghĩa gần.

#### Multicast_Group + IGMP_Group

**Cơ chế hoạt động.** Multicast_Group: entry forwarding multicast per-datapath với tunnel_key. IGMP_Group: theo dõi membership IGMP per-datapath, populate qua IGMP snooping.

**Khác biệt giữa các phiên bản.** OVN 21.06 thêm IGMP snooping.

**Bài tập.** `ovn-sbctl list Multicast_Group; ovn-sbctl list IGMP_Group` để quan sát.

**So sánh với hệ khác.** Tương đương trực tiếp `bridge mdb show` của Linux bridge. IGMP snooping của Cisco có ngữ nghĩa gần.

#### BFD

**Cơ chế hoạt động.** Một session BFD trên mỗi HA gateway. Cột chính: dst_ip, src_port, status (down/init/up). Protocol RFC 5880.

**Khác biệt giữa các phiên bản.** OVN 22.03 chuẩn hoá.

**Trường hợp sự cố thực tế.** Failover gateway HA phát hiện < 200ms (xem Phần 13.6 BFD).

**Bài tập.** `ovn-sbctl list BFD` để quan sát.

**So sánh với hệ khác.** Tương đương trực tiếp BFD chuẩn IOS Cisco. Daemon `bird` BFD trên Linux có ngữ nghĩa gần.

#### HA_Chassis

**Cơ chế hoạt động.** Mục member trong HA_Chassis_Group. Cột chính: chassis_name, priority. Priority cao nhất hiện active = redirect-chassis hiện tại.

**Khác biệt giữa các phiên bản.** OVN 21.06 ổn định.

**Trường hợp sự cố thực tế.** Trigger failover: BFD báo DOWN → priority được tính lại → active mới.

**Bài tập.** `ovn-sbctl list HA_Chassis` để quan sát.

**So sánh với hệ khác.** Cisco HSRP standby group có ngữ nghĩa gần.

#### Connection

**Cơ chế hoạt động.** Metadata kết nối OVSDB: target (URI), protocol (tcp/ssl/punix), inactivity_probe, max_backoff.

**Khác biệt giữa các phiên bản.** RFC 7047 ổn định.

**Bài tập.** `ovn-sbctl list Connection` để quan sát.

**So sánh với hệ khác.** Không có cross-domain trực tiếp (đặc thù OVSDB). Khái niệm gần với DNS A record (endpoint discovery).

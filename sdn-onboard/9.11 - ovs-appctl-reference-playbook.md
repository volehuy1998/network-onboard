## Bối cảnh: Tại sao `ovs-appctl` là công cụ linh hoạt nhất

Phần lớn ba công cụ viết/đọc OVSDB + OpenFlow (`ovs-vsctl`, `ovs-ofctl`, `ovs-dpctl`) được thiết kế cho workflow cụ thể: config bridge, program flow, dump kernel flow. Khi workflow vượt ra khỏi CRUD config hoặc flow programming, engineer phải gọi trực tiếp daemon để hỏi runtime state.

`ovs-appctl` là một generic RPC wrapper — nó gửi command tới bất kỳ OVS daemon nào qua Unix socket. Command là string phẳng; response cũng là string. Không có type-safe schema giữa client và daemon; mỗi daemon tự expose command qua `unixctl_command_register()` trong mã nguồn của daemon đó.

Kết quả: `ovs-appctl` **cover toàn bộ live state** của mọi daemon OVS. Hơn 80 command riêng biệt cho `ovs-vswitchd` (bridge + datapath + classifier + connection manager + bond + spanning-tree + tunnel + upcall + revalidator), hơn 10 command cho `ovsdb-server` (cluster + schema + monitor), và 5-10 command cho mỗi auxiliary daemon (`ovn-controller`, `ovn-northd`).

Triết lý thiết kế của `ovs-appctl`: **live + read-mostly + low-overhead**. Không ảnh hưởng OVSDB config, không thay đổi OpenFlow pipeline, chỉ đọc state hiện tại hoặc làm thao tác transient (flush FDB, purge revalidator, set vlog level). Do đó an toàn chạy trong production — không cần maintenance window.

## 9.11.0 Grammar + target discovery

Grammar tổng thể (man `ovs-appctl(8)` §SYNOPSIS):

```
ovs-appctl [OPTIONS] [-t TARGET | --target=TARGET] COMMAND [ARG...]

TARGET:
  - Process name: ovs-vswitchd (default), ovsdb-server, ovn-controller,
    ovn-northd, ovn-ic, ovn-controller-vtep
  - Pidfile path: /var/run/openvswitch/<name>.pid
  - Unix socket path: /var/run/openvswitch/<name>.<pid>.ctl

OPTIONS:
  --timeout=SECS    (default: wait forever)
  -h, --help
  -V, --version
```

Khi chạy không có `-t`, `ovs-appctl` mặc định target `ovs-vswitchd`. Đây là daemon lớn nhất và nơi majority command nằm. Các daemon khác phải explicit pass `-t`:

```shell
$ sudo ovs-appctl -t ovsdb-server list-commands
$ sudo ovs-appctl -t ovn-controller inc-engine/show-stats
$ sudo ovs-appctl -t /var/run/ovn/ovn-northd.ctl list-commands
```

### 9.11.0.2 Phân nhóm target theo chức năng

Tổ chức mental model theo nhóm giúp tìm nhanh:

| Nhóm | Prefix | Mục đích | Daemon |
|---|---|---|---|
| Introspection | `version`, `list-commands`, `memory/*`, `coverage/*` | Runtime state cơ bản | Tất cả |
| Logging | `vlog/*` | Log level per module/destination | Tất cả |
| Bridge L2 | `bridge/*`, `fdb/*`, `mdb/*` | Bridge state, MAC learning, multicast snooping | ovs-vswitchd |
| Link aggregation | `bond/*`, `lacp/*` | Bond state, LACP handshake | ovs-vswitchd |
| Spanning tree | `stp/*`, `rstp/*` | 802.1D STP + 802.1w RSTP | ovs-vswitchd |
| L2 monitoring | `bfd/*`, `cfm/*` | BFD + Connectivity Fault Management | ovs-vswitchd |
| OpenFlow pipeline | `ofproto/*`, `bundle/*` | OpenFlow introspection | ovs-vswitchd |
| Kernel datapath | `dpctl/*`, `dpif/*` | Kernel flow + conntrack | ovs-vswitchd |
| Userspace DPDK | `dpif-netdev/*` | PMD thread, rxq, EMC | ovs-vswitchd (netdev dp only) |
| Tunnel | `tnl/*` | Tunnel ARP/ND cache, port listen | ovs-vswitchd |
| Upcall + revalidator | `upcall/*`, `revalidator/*` | Flow install + GC pipeline | ovs-vswitchd |
| OVSDB cluster | `cluster/*` | Raft cluster management | ovsdb-server |
| OVN controller | `inc-engine/*`, `lflow-cache/*`, `ct-zone-list` | OVN runtime state | ovn-controller |

Đếm tổng: **~12 nhóm chức năng trong ovs-vswitchd + 1 cho ovsdb-server + 3-4 cho ovn-controller = ~16-17 nhóm**. Phần này cover 11 nhóm quan trọng nhất cho ovs-vswitchd + 1 cho ovsdb-server.

## 9.11.1 Introspection + telemetry, `version`, `memory/show`, `coverage/show`

### 9.11.1.1 `version`

Command trivial nhưng đầu tiên để xác định version khi remote-debug:

```shell
$ sudo ovs-appctl version
ovs-vswitchd (Open vSwitch) 2.17.9-0ubuntu0.22.04.1
```

Output là ASCII string duy nhất. Đừng nhầm với `ovs-vsctl --version` (cho CLI tool version) hoặc `ovs-vsctl get Open_vSwitch . ovs_version` (dump version trong OVSDB). Cả ba phải nhất quán; lệch nhau nghĩa là daemon chạy version cũ mà OVSDB schema đã upgrade.

### 9.11.1.2 `memory/show`, anatomy breakdown bộ nhớ ovs-vswitchd

**Bối cảnh:** debug rò rỉ bộ nhớ hoặc tìm component nào tiêu thụ heap. Mỗi hour hoặc mỗi lần reconfigure, chạy `memory/show` log vào file; so sánh delta.

**Command và output** `[real capture, br-int OVN 22.03.8, 3 bridge + 50 port + 1200 flow]`:

```shell
$ sudo ovs-appctl memory/show
cells:48289 handlers:4 idl-cells-OVN_Southbound:82134 \
  idl-cells-Open_vSwitch:2453 \
  of-connections:2 \
  ofconns:2 \
  ports:56 \
  revalidators:4 \
  rules:12847 \
  udpif keys:1247
```

Output là single-line key=value list. Parse dễ trong script.

**Anatomy từng counter:**

| Counter | Ý nghĩa | Healthy range | Dấu hiệu bất thường |
|---|---|---|---|
| `cells` | Tổng ofpbuf cell allocated (internal structure trong ovs-vswitchd) | < 100k với setup nhỏ | Leak: tăng monotonic không giảm sau idle 1h |
| `handlers` | Số handler thread (xử lý upcall) | = số physical core / 4 (default) | Khác với `other_config:n-handler-threads` → config không apply |
| `idl-cells-<schema>` | Số row trong IDL cache của daemon cho từng OVSDB schema | Khớp row count actual | Tăng nhanh hơn actual DB → memory leak trong IDL |
| `of-connections` | Active OpenFlow connection đang dựng | = số controller connect được | 0 mà expected > 0 → controller unreachable |
| `ofconns` | OpenFlow connection object allocated | Khớp `of-connections` | Lệch nhau → connection leak |
| `ports` | Total port object allocated (all bridges) | Khớp `ovs-vsctl list Interface \| wc -l` | Không khớp → bridge cleanup miss |
| `revalidators` | Số revalidator thread | `n-revalidator-threads` config | Khác config → OVS chưa restart sau set |
| `rules` | Total flow entry allocated (trên tất cả bridge + table) | Phụ thuộc workload | Tăng monotonic trong steady state → flow leak từ controller |
| `udpif keys` | Số ukey (unique flow key) trong datapath-to-userspace mapping | Gần `rules` cho kernel dp | `udpif keys >> flows` từ `dpctl/show` → Phần 9.26 revalidator storm pattern |

**Kịch bản bẻ gãy:**

- **`cells` tăng hơn 10% per giờ trong steady-state** → memory leak; bisect bằng cách disable từng feature (mirror, sflow, conntrack zones) xem counter ngừng tăng khi nào.
- **`udpif keys` đơn điệu tăng nhanh hơn `rules`** → ukey không được GC; check `revalidator/wait` response, có thể revalidator thread stuck — Phần 9.26 mô tả chi tiết pattern này.
- **`handlers` = 0** → ovs-vswitchd đang khởi động chưa fork handler, hoặc đã crash thread; `systemctl status ovs-vswitchd` confirm.

**Anatomy row:**

- **Cột 1:** counter name (đặt trong source code bằng `COVERAGE_DEFINE(name)`).
- **Cột 2:** avg rate 5 giây gần nhất.
- **Cột 3:** avg rate 1 phút gần nhất.
- **Cột 4:** avg rate 1 giờ gần nhất.
- **Cột 5:** `total: N` cumulative count từ startup.

**Counter class foundation cần thuộc:**

| Counter | Mô tả | Healthy threshold |
|---|---|---|
| `hit` | Megaflow lookup hit — packet khớp với cache | - |
| `miss` | Lỡ cache → trigger upcall | `miss / hit` < 1% steady state |
| `flow_extract` | Parse L2-L4 header từ packet (mọi upcall) | = `miss` roughly |
| `xlate_actions` | Translate OpenFlow action set → datapath action | = `upcall_ukey_lookup_created` + `upcall_ukey_lookup_hit` |
| `upcall_flow_add` | Install flow mới vào datapath | < 10/sec steady |
| `upcall_flow_del` | Remove flow khỏi datapath | ≈ `upcall_flow_add` balance |
| `upcall_ukey_lookup_created` | Tạo ukey mới | < 10/sec; > 100/sec → flow thrashing |
| `upcall_ukey_lookup_hit` | Hit existing ukey | cao → cache reuse tốt |
| `revalidate` | Revalidator full pass | OVS 2.17 target 0.1-1/sec; > 10/sec → Phần 9.26 storm |
| `facet_revalidate` | Check từng facet (flow-specific) | - |
| `facet_changed_rule` | Rule khớp với packet đã thay đổi sau install → OpenFlow programming unstable | Tăng liên tục → controller churning |
| `ofproto_flush` | Bridge flush (re-install toàn bộ rule) | > 0/min → controller restart hoặc OVS reconfig |
| `bridge_reconfigure` | OVSDB config change triggered reconfig | > 10/hour → config churn, check orchestrator |
| `vconn_sent` / `vconn_received` | OpenFlow connection I/O | - |
| `util_xalloc` | Memory alloc qua xmalloc | rate cao kéo dài → memory churn |
| `netlink_received` / `netlink_sent` | Netlink I/O kernel-userspace | - |
| `rstp_run` / `stp_run` | Spanning-tree protocol tick | 0 nếu STP disable |
| `connmgr_wakeup` | Connection manager wake | - |

**Kịch bản bẻ gãy dựa trên counter delta:**

- **`hit / (hit+miss)` < 90%** → cache effectiveness thấp; kiểm tra `megaflow_mask_hit_ratio` trong Part 9.17, có thể megaflow quá narrow, mỗi packet đặc thù cài đặt flow riêng (flow explosion). Xem Part 9.15 tuple space search để optimize.
- **`upcall_flow_add` >> `upcall_flow_del`** → flow accumulate trong datapath, chuẩn bị chạm limit. Kết hợp với `upcall/show flows current` xem còn space không.
- **`facet_changed_rule` > 1/sec** → OpenFlow pipeline đang churn; controller install/modify flow liên tục. Nếu OVN → check `ovn-appctl -t ovn-controller inc-engine/show-stats` xem I-P engine recompute rate.
- **`ofproto_flush` > 0 khi không expected** → bridge config đang rollout; nếu không phải controller intentional → check OVSDB connection, có thể disconnect-reconnect gây full flush.

### 9.11.1.4 `coverage/read-counter`, lấy một counter

Khi script automation cần poll một counter cụ thể (không muốn parse toàn bộ `coverage/show`):

```shell
$ sudo ovs-appctl coverage/read-counter upcall_flow_add
189
```

Output là single integer. Nhanh hơn + stable format hơn. Dùng trong Prometheus exporter:

```bash
#!/bin/bash
counters=(
  hit miss
  upcall_flow_add upcall_flow_del
  upcall_ukey_lookup_created
  revalidate facet_revalidate
  ofproto_flush bridge_reconfigure
)
for c in "${counters[@]}"; do
  val=$(sudo ovs-appctl coverage/read-counter "$c" 2>/dev/null || echo 0)
  echo "ovs_coverage_${c}_total $val"
done
```

### 9.11.1.5 `vlog/list`, `vlog/set`, tuning log chi tiết mà không restart

**Bối cảnh:** log level mặc định của OVS là `info` cho file, `err` cho console. Khi debug tận cùng một module (ví dụ `dpif`), tăng level lên `dbg` cho riêng module đó, rồi restore sau. Không cần restart daemon.

```shell
# Liệt kê module + level hiện tại
$ sudo ovs-appctl vlog/list | head -15
                 console    syslog    file
                 -------    -------   ------
backtrace        OFF        ERR       INFO
bfd              OFF        ERR       INFO
bond             OFF        ERR       INFO
bridge           OFF        ERR       INFO
classifier       OFF        ERR       INFO
connmgr          OFF        ERR       INFO
...
dpif             OFF        ERR       INFO
dpif_netdev      OFF        ERR       INFO
...
```

**Anatomy:**

- **Row:** tên module, mỗi module tương ứng một file source OVS (`lib/dpif.c` → module `dpif`).
- **Column:** destination — `console` (stderr), `syslog` (/var/log/syslog qua syslogd), `file` (/var/log/openvswitch/ovs-vswitchd.log).
- **Cell:** level — `EMER`, `ERR`, `WARN`, `INFO`, `DBG`, `OFF`.

**Syntax set:**

```
ovs-appctl vlog/set [SPEC[:SPEC]...]
SPEC: <module>:<destination>:<level>
Special: ANY for module, ANY for destination
```

Ví dụ tăng log `dpif` lên debug trên file, giữ nguyên module khác:

```shell
$ sudo ovs-appctl vlog/set dpif:file:dbg
$ sudo tail -f /var/log/openvswitch/ovs-vswitchd.log | grep dpif
2026-04-24T11:15:23.412Z|00042|dpif(handler1)|DBG|system@ovs-system: \
  port_query_by_number(3): pkts:1024 bytes:131072
2026-04-24T11:15:23.412Z|00043|dpif(handler1)|DBG|system@ovs-system: \
  execute ufid:a1b2c3d4..., actions: 2,3
```

Restore sau debug:

```shell
$ sudo ovs-appctl vlog/set dpif:file:info
```

**Shortcut debug toàn bộ datapath pipeline:**

```shell
$ sudo ovs-appctl vlog/set dpif:file:dbg dpif_netdev:file:dbg \
    ofproto_dpif_upcall:file:dbg ofproto_dpif_xlate:file:dbg
```

Log file sẽ rất lớn (≥ 100 MB/phút dưới load). Restore ngay khi không cần.

### 9.11.1.6 `vlog/reopen`, rotate log file

Sau `logrotate` move file log, daemon vẫn giữ file descriptor cũ và ghi vào file đã renamed. `vlog/reopen` close + reopen để pick up path mới:

```shell
# Thêm vào /etc/logrotate.d/openvswitch:
/var/log/openvswitch/*.log {
    daily
    rotate 7
    compress
    postrotate
        /usr/bin/ovs-appctl vlog/reopen
    endscript
}
```

Không có `vlog/reopen` sẽ dẫn đến ovs-vswitchd ghi vào file `ovs-vswitchd.log.1.gz` (vô nghĩa), file mới trống.

## 9.11.2 Bridge L2, `bridge/dump-flows`, `fdb/show`, `mdb/show`

### 9.11.2.1 `bridge/dump-flows` so với `ovs-ofctl dump-flows`

Hai command khác biệt nhỏ nhưng quan trọng:

| Aspect | `ovs-ofctl dump-flows` | `ovs-appctl bridge/dump-flows` |
|---|---|---|
| Protocol | OpenFlow OFPT_STATS_REQUEST | Unix socket RPC |
| Scope | Flow user-visible (controller install) | **Toàn bộ flow bao gồm hidden** (internal learn, ARP responder, OVN-generated) |
| Format | OpenFlow format, match spec chuẩn | OVS internal format |
| Speed | Qua TCP/SSL nếu remote | Qua Unix socket, nhanh hơn local |
| Use case | Thường dùng | Khi debug flow hidden |

**Bối cảnh dùng `bridge/dump-flows`:** khi `ovs-ofctl dump-flows br0` return fewer rule hơn expected, hoặc khi debug OVN-generated flow có `learn()` action tạo flow dynamic.

```shell
$ sudo ovs-appctl bridge/dump-flows br-int | head -15

 cookie=0x0, duration=3421.234s, table=0, n_packets=18234, n_bytes=2145678,
   priority=100,in_port=1 actions=load:0x1->NXM_NX_REG13[],resubmit(,8)
```

### 9.11.2.2 `fdb/show`, đọc MAC learning table

**Bối cảnh:** debug L2 chuyển tiếp — "traffic đi sai port, ping reply sai chiều, ARP reply không đến đúng port". 90% case là MAC learning sai.

**Command và output** `[real capture, br-int với 3 port đã traffic]`:

```shell
$ sudo ovs-appctl fdb/show br-int
 port  VLAN  MAC                Age
    1     0  aa:bb:cc:00:00:01    4
    2     0  aa:bb:cc:00:00:02   12
    3   100  aa:bb:cc:00:00:03   87
    1     0  aa:bb:cc:00:00:04  287
 LOCAL    0  3e:6a:a1:b2:c3:d0  3421
```

**Anatomy:**

| Field | Ý nghĩa | Dấu hiệu bất thường |
|---|---|---|
| `port` | OpenFlow port number học được. `LOCAL` = bridge internal port | `LOCAL` xuất hiện nhiều MAC → broadcast vào bridge IP |
| `VLAN` | VLAN ID trong `vlan_tci`. `0` = untagged hoặc VLAN native | VLAN khác tag của port access → misconfig, MAC học qua trunk dẫn đến broadcast domain leak |
| `MAC` | Địa chỉ MAC học được | MAC multicast (`01:00:5e:*`) trong FDB → bug, multicast không được học vào FDB thường |
| `Age` | Giây từ lần học cuối | > 300s (mặc định aging) nên gone; tồn tại → `other_config:mac-aging-time` custom |

**Kịch bản bẻ gãy:**

- **Cùng MAC xuất hiện trên hai port khác nhau (MAC flapping)** → L2 loop hoặc duplicate MAC (VM migration không clean up ARP cache). Check `bridge/dump-flows` cho flow bridge-ARP. Dùng `ovs-appctl stp/show` hoặc `rstp/show` kiểm tra loop prevention có active không.
- **Port `LOCAL` có nhiều MAC trong cùng VLAN** → traffic broadcast đang enter bridge qua internal IP (br-int có IP gán lên interface internal). Thường không sai nhưng lưu ý nếu workload isolation.
- **FDB trống trên bridge có traffic** → bridge có `fail_mode=secure` + controller chưa cài đặt learn flow → mọi packet non-match bị drop, MAC không học được. Check `ovs-ofctl dump-flows br-int` cho `actions=NORMAL` hoặc explicit `learn()`.

**Companion: `fdb/flush`, buộc relearn:**

```shell
$ sudo ovs-appctl fdb/flush br-int        # flush toàn bộ
$ sudo ovs-appctl fdb/flush br-int 100    # chỉ VLAN 100
```

Sau flush, traffic tiếp theo sẽ re-populate FDB. Dùng khi suspect MAC cache stale (sau migration, failover).

### 9.11.2.3 `mdb/show`, multicast snooping table

**Bối cảnh:** khi enable `mcast_snooping_enable=true`, bridge filter multicast dựa trên IGMP report thay vì flood. `mdb/show` là debug chính.

```shell
$ sudo ovs-appctl mdb/show br-int
 port  VLAN  GROUP                Age
    2     0  224.0.0.251            45
    3     0  239.1.1.1             120
    2     0  ff02::fb (MLDv2)       45
```

**Anatomy:**

- `port` — OpenFlow port đang subscribe group.
- `VLAN` — VLAN context.
- `GROUP` — multicast group address (IPv4 hoặc IPv6 MLD).
- `Age` — giây từ lần IGMP report cuối.

**Kịch bản:**

- **MDB empty khi expected có multicast subscriber** → IGMP snooping chưa enable hoặc querier chưa active; check `ovs-vsctl get Bridge br-int mcast_snooping_enable`.
- **Group `224.0.0.0/24` xuất hiện trong MDB** → reserved scope "local-network control plane" (OSPF hello, IGMP), thường shouldn't be snooped; config `mcast-snooping-disable-flood-unregistered` nếu muốn block.

## 9.11.3 Link aggregation, `bond/show` + `lacp/show`

### 9.11.3.1 `bond/show`, đọc state bond

**Bối cảnh:** host multi-NIC dùng bond (active-backup, balance-slb, balance-tcp) để resilience + bandwidth aggregation. Khi ping intermittent hoặc băng thông thực tế giảm, `bond/show` là debug đầu tiên.

**Command và output** `[real capture, bond0 with 2 NIC, balance-tcp]`:

```shell
$ sudo ovs-appctl bond/show bond0
---- bond0 ----
bond_mode: balance-tcp
bond may use recirculation: yes, Recirc-ID : 1
bond-hash-basis: 0
updelay: 0 ms
downdelay: 0 ms
next rebalance: 9821 ms
lacp_status: negotiated
lacp_fallback_ab: false
active-backup primary: <none>

slave eno1: enabled
  active slave
  may_enable: true
  hash 38: 15 kB load
  hash 113: 0 kB load

slave eno2: enabled
  may_enable: true
  hash 41: 32 kB load
  hash 147: 2 kB load
```

**Anatomy bond level:**

| Field | Ý nghĩa | Dấu hiệu bất thường |
|---|---|---|
| `bond_mode` | `active-backup` (1 active) / `balance-slb` (source MAC) / `balance-tcp` (L4 hash) | Mode production thường `balance-tcp` + LACP; `balance-slb` không cần LACP nhưng không tối ưu |
| `bond may use recirculation` | `yes` nếu datapath recirculate packet để compute hash (balance-tcp cần) | `no` với balance-tcp → kernel cũ không support, hiệu năng suy giảm |
| `Recirc-ID` | Recirculation ID kernel dùng cho bond | - |
| `bond-hash-basis` | Hash salt để tránh hash collision giữa host | Giống nhau nhiều host → nếu dùng ECMP cross-chassis, traffic skew |
| `updelay` / `downdelay` | Millisecond delay trước khi bring-up / bring-down member sau link change | 0 ms = instant; 200 ms là default LACP recommendation |
| `next rebalance` | Milliseconds đến lần rebalance tiếp theo (balance-tcp) | Default 10000 ms (10s) |
| `lacp_status` | `negotiated` / `configured` / `disabled` | `configured` nhưng không `negotiated` → LACP partner không reply; check cabling hoặc LACP config peer |
| `lacp_fallback_ab` | Fallback active-backup nếu LACP fail | `true` khuyến nghị khi không chắc peer có LACP |
| `active-backup primary` | Member nào preferred khi mode = active-backup | `<none>` → chọn ngẫu nhiên |

**Anatomy slave level:**

| Field | Ý nghĩa | Dấu hiệu bất thường |
|---|---|---|
| `slave <name>` | Tên member interface | - |
| `enabled` / `disabled` | Có carrier traffic không | `disabled` → check `ethtool <iface>` cho link state |
| `active slave` (active-backup) | Member đang forward traffic | Không xuất hiện trên bất kỳ slave → primary selection chưa complete |
| `may_enable` | Có thể enable (passed LACP, link up, etc.) | `false` → LACP chưa negotiate hoặc carrier down |
| `hash N: X kB load` | TCP hash bucket N currently handled, với load (bytes) hiện tại | Skew giữa slave (90% load on 1 slave) → balance-tcp hash không spread, có thể elephant flow |

**Kịch bản bẻ gãy:**

- **`lacp_status: configured` (không `negotiated`)** → LACP PDU bị drop ở cable level, switch peer không config LACP đúng, hoặc mode peer là static LAG. Check `lacp/show bond0` xem actor/partner state.
- **Hash load skew** → một slave carry 90% traffic; do elephant flow (1 TCP 5-tuple chiếm dominant). Dùng `bond/migrate bond0 <hash> <slave>` rebalance thủ công, hoặc chuyển sang mode `balance-slb` nếu flow count cao.
- **`may_enable: false` sau LACP `negotiated`** → rare, check OVS log `grep lacp /var/log/openvswitch/ovs-vswitchd.log`.

### 9.11.3.2 `lacp/show`, actor/partner handshake state

**Bối cảnh:** bond mode có LACP. Khi bond `lacp_status` không negotiate, đi sâu vào LACP state.

```shell
$ sudo ovs-appctl lacp/show bond0
---- bond0 ----
    status: active negotiated
    sys_id: 3e:6a:a1:b2:c3:d0
    sys_priority: 65534
    aggregation key: 1
    lacp_time: fast

slave: eno1
  status: current attached
  port_id: 1
  port_priority: 65535
  may_enable: true

  actor sys_id: 3e:6a:a1:b2:c3:d0
  actor sys_priority: 65534
  actor port_id: 1
  actor port_priority: 65535
  actor key: 1
  actor state: activity timeout aggregation synchronized collecting \
    distributing

  partner sys_id: 00:1e:68:aa:bb:cc
  partner sys_priority: 127
  partner port_id: 5
  partner port_priority: 127
  partner key: 1234
  partner state: activity timeout aggregation synchronized collecting \
    distributing
```

**Anatomy LACP state bitfield:**

| Flag | Ý nghĩa | Expected |
|---|---|---|
| `activity` | Actor sending LACP PDU (active mode) | Always set |
| `timeout` | Fast mode (1s) vs slow (30s) | Fast = `lacp_time: fast` |
| `aggregation` | Member có thể aggregate | Always set |
| `synchronized` | Bond + peer agree về state | CRITICAL: nếu không → LACP mismatch |
| `collecting` | Member đang nhận traffic | Required cho data path |
| `distributing` | Member đang gửi traffic | Required |
| `defaulted` | Dùng default partner (không có PDU nhận) | Không nên xuất hiện |
| `expired` | PDU timeout hết hạn | Không nên xuất hiện |

**Kịch bản bẻ gãy:**

- **Actor + partner sys_id giống nhau** → cable tạo loopback (port nối vào port cùng bridge). Fix cable.
- **`partner state: defaulted expired`** → không nhận LACP PDU từ peer 3× timeout; cable fail hoặc peer không LACP.
- **`actor key` ≠ `partner key`** → Port Group ID không match; kiểm chứng LACP config ở peer phải cùng bond group.


**Nhược điểm:** tăng limit không free — mỗi flow tiêu ~200-500 byte kernel memory. Limit 500k = ~250 MB. Setup memory-constrained cân nhắc.

### 9.11.10.3 `revalidator/purge`, force xóa megaflow

**Bối cảnh:** sau thay đổi OpenFlow pipeline, muốn force kernel re-learn toàn bộ flow. Flag thay vì chờ idle timeout.

```shell
$ sudo ovs-appctl revalidator/purge
$ sudo ovs-appctl upcall/show | grep "current"
  flows : (current 0) ...
```

Sau purge, packet đầu tiên mỗi flow kích hoạt upcall và re-install. Brief latency spike nhưng state clean.

### 9.11.10.4 `revalidator/wait`, sync barrier

Chờ revalidator pass hoàn thành trước khi query tiếp. Dùng trong script tự động:

```shell
$ sudo ovs-appctl revalidator/wait
$ sudo ovs-appctl dpctl/dump-flows  # đảm bảo latest state
```

### 9.11.11.1 `cluster/status`, Raft consensus health

**Bối cảnh:** OVN production thường dùng OVSDB clustered (Part 10.1) với 3-5 member. `cluster/status` là debug Raft đầu tiên.

```shell
$ sudo ovs-appctl -t /var/run/ovn/ovnsb_db.ctl cluster/status OVN_Southbound

Name: OVN_Southbound

Server ID: 7a21 (7a2104f1-...)
Address: tcp:10.0.0.1:6644
Status: cluster member
Role: leader
Term: 42
Leader: self
Vote: self

Last Election started 892123 ms ago, reason: timeout
Last Election won: 892100 ms ago
Election timer: 1000
Log: [2, 12847]
Entries not yet committed: 0
Entries not yet applied: 0
Connections: <- 8a3b ->c4d1 (->=self sending, <-=receiving)
Disconnections: 0
Servers:
    7a21 (7a21 at tcp:10.0.0.1:6644) (self) next_index=12848 match_index=12847
    8a3b (8a3b at tcp:10.0.0.2:6644) next_index=12848 match_index=12847
    c4d1 (c4d1 at tcp:10.0.0.3:6644) next_index=12848 match_index=12847
```

**Anatomy:**

| Field | Ý nghĩa | Dấu hiệu bất thường |
|---|---|---|
| `Role` | `leader` / `follower` / `candidate` | `candidate` > 5s → election stuck |
| `Term` | Raft term counter | Tăng mỗi election; nhảy nhanh → leader instability |
| `Leader: self` | Identity của leader | Leader biến mất → cluster không writable |
| `Last Election won` | Thời gian từ lần election gần nhất | < 1 phút → recent election, nên check instability |
| `Log: [2, 12847]` | Start index và latest index | Gap giữa member → log replication lag |
| `Entries not yet committed` | Write pending quorum ACK | > 0 kéo dài → quorum không đạt |
| `Entries not yet applied` | Committed nhưng chưa apply vào DB | > 0 → apply thread slow |
| `Disconnections` | Counter disconnect từ peer | Tăng → unstable network |
| `match_index` | Latest log index của peer khớp với leader | Lệch xa → peer lag, cần catch up |

**Kịch bản bẻ gãy:**

- **3 follower, không leader** → network partition; check `iptables -L -n` + `ss -tlnp | grep 6644`.
- **`Term` nhảy 5+ trong 1 phút** → election storm, thường do flapping network hoặc heartbeat timeout quá thấp.
- **`match_index` của peer tụt xa leader** → peer slow applying log; check disk I/O (OVSDB fsync heavy).

### 9.11.11.2 `cluster/kick`, remove member khỏi cluster

Khi member permanently gone (host decommission), phải manually remove:

```shell
$ sudo ovs-appctl -t /var/run/ovn/ovnsb_db.ctl cluster/kick OVN_Southbound 7a21
server 7a21 kicked
```

Chỉ chạy từ leader. Member "kicked" không thể tự join lại — phải destroy + recreate DB on that host.

### 9.11.11.3 `cluster/leave`, member tự rời

Preferred khi có graceful shutdown:

```shell
$ sudo ovs-appctl -t /var/run/ovn/ovnsb_db.ctl cluster/leave OVN_Southbound
```

Sau leave, member tự động transition ra khỏi voting set, leader commit config change, cluster size giảm.

### 9.11.X.1 `ovs-appctl dpif/dump-flows`

#### Khái niệm

`dpif/dump-flows` là subcommand `ovs-appctl` ra lệnh cho `ovs-vswitchd` đi thẳng xuống tầng `dpif` (datapath interface) để liệt kê **mọi flow đã cài** trong datapath đang gắn với bridge: kernel datapath (`dpif-netlink`) hoặc userspace datapath (`dpif-netdev`). Đây là "sự thật vật lý" của cache đang giữ gói tin: từng dòng output là một megaflow (nếu kernel đã consolidate) hoặc microflow, kèm counter `n_packets/n_bytes` thực và `actions` thực sự thực thi cho gói khớp.

#### Vai trò trong kiến trúc

Pipeline OVS có ba tầng: OVSDB schema (control plane mong muốn), bảng OpenFlow trong `ofproto` (luật do controller cài), và flow trong datapath (cache cuối cùng đang phục vụ gói). Khi engineer hỏi "gói tin có khớp luật của tôi không", `ovs-ofctl dump-flows` chỉ trả lời tầng OpenFlow; `dpif/dump-flows` mới trả lời tầng datapath, là tầng thực sự nhìn thấy gói. Kết quả là công cụ duy nhất để xác nhận megaflow đã được install, đã hit/miss bao nhiêu lần, và có bị evict bởi revalidator hay không.

#### Cú pháp + tham số

```
ovs-appctl dpif/dump-flows [-m] [--names] [DP]
```

- `DP` là tên datapath (mặc định `system@ovs-system` cho kernel, `netdev@ovs-netdev` cho userspace). Có thể bỏ qua nếu chỉ một datapath.
- `-m` (more) thêm field thường ẩn: `recirc_id`, `dp_hash`, `ct_state`, `ct_zone`, `ct_mark`, `ct_label`. Bắt buộc khi điều tra conntrack hoặc recirc pipeline.
- `--names` thay số `in_port(N)` bằng tên `in_port(eth0)`.

#### Mẫu output thực tế

```shell
$ sudo ovs-appctl dpif/dump-flows -m | head -3
ufid:6a8c1f29-7d2b-44ee-9f0e-01e3a8b27d11,
  recirc_id(0),dp_hash(0/0),skb_priority(0/0),in_port(2),
  skb_mark(0/0),ct_state(0/0),ct_zone(0/0),ct_mark(0/0),ct_label(0/0),
  eth(src=aa:bb:cc:00:00:02/00:00:00:00:00:00,dst=aa:bb:cc:00:00:01/...),
  eth_type(0x0800),ipv4(src=10.1.0.5,dst=10.1.0.10,proto=6,frag=no/0xff),
  tcp(src=50000/0xffff,dst=80/0xffff),tcp_flags(0/0),
  packets:18234, bytes:2145678, used:0.082s, flags:SP, dp:ovs, actions:4
```

#### Phân biệt rõ với `ovs-ofctl dump-flows`

Đây là điểm hay nhầm. Hai lệnh trả về dữ liệu khác nhau về bản chất:

| Tiêu chí | `ovs-ofctl dump-flows br0` | `ovs-appctl dpif/dump-flows` |
|---------|---------------------------|------------------------------|
| Tầng | OpenFlow rule (do controller cài) | Datapath cache (do `ovs-vswitchd` xlate xuống kernel) |
| Số dòng | Tỉ lệ với số rule controller push | Tỉ lệ với số megaflow đã được kích hoạt |
| Counter | `n_packets/n_bytes` per OF rule | `packets/bytes` per megaflow |
| Wildcard | Theo OF match field | Theo mask kernel cấp (sau xlate + consolidation) |
| Mục đích | Xác minh "controller cài đúng luật chưa" | Xác minh "kernel đang xử lý gói thực sự ra sao" |

Khi flow OpenFlow có nhưng `dpif/dump-flows` không thấy entry tương ứng + counter không tăng → hoặc gói chưa tới, hoặc xlate fail (xem `ovs-vswitchd.log`), hoặc bị `revalidator` evict liên tục (xem §9.11.10).

#### Lỗi thường gặp + tín hiệu chẩn đoán

| Triệu chứng | Khả năng | Cách kiểm chứng |
|-------------|---------|-----------------|
| Output rỗng dù traffic đang chạy | Không gọi đúng datapath, hoặc datapath mới khởi tạo chưa có miss đầu tiên | `ovs-appctl dpif/show` xem datapath nào đang active |
| Số entry tăng vô kiểm soát (flow explosion) | Rule OF wildcard quá hẹp, mỗi 5-tuple sinh megaflow riêng | Đối chiếu `coverage/show` counter `upcall_flow_add` rate và Phần 9.15 TSS |
| `used:` luôn lớn, packets không tăng | Megaflow stale, traffic đã rẽ sang entry khác | Chạy `ovs-appctl revalidator/wait` ép revalidator chạy, đo lại |

#### Quy trình sử dụng

1. `ovs-appctl dpif/show` xem datapath nào hiện hữu.
2. `ovs-appctl dpif/dump-flows -m | wc -l` đếm tổng số entry baseline.
3. Sinh load 30-60 giây (`ping`, `iperf`).
4. Lặp lại bước 2 xem entry tăng bao nhiêu, kết hợp `coverage/show` xem `upcall_flow_add` cùng khoảng.
5. `dpif/dump-flows -m | grep <field>` lọc theo `recirc_id`, `ct_state`, hoặc `in_port` cụ thể.

#### Source code tham chiếu

- `ofproto/ofproto-dpif.c::ofproto_unixctl_dpif_dump_flows()` đăng ký subcommand qua `unixctl_command_register()`. Hàm gọi `dpif_flow_dump_create()` rồi iterate qua `dpif_flow_dump_next()`.
- `lib/dpif.c::dpif_flow_dump_*()` lớp trừu tượng phía userspace.
- `lib/dpif-netlink.c` (kernel) và `lib/dpif-netdev.c` (userspace) là hai backend cụ thể.

Cross-link Rule 14: §9.2 mô tả chi tiết cấu trúc megaflow trong kernel datapath; §9.32 deep-dive classifier internals.

#### Khác biệt giữa các phiên bản

- OVS 2.0 (2014): subcommand xuất hiện cùng lúc với refactor `dpif`. Trước đó chỉ có `ovs-dpctl dump-flows` chạy ngoài ovs-vswitchd, không thấy state nội tại.
- OVS 2.4 (2015): thêm `ufid` (Unique Flow Identifier) trong output mặc định.
- OVS 2.6 (2016): thêm `recirc_id` + `dp_hash` field hỗ trợ stateful pipeline.
- OVS 2.8+ (2017): support đầy đủ conntrack field (`ct_state`, `ct_zone`, `ct_mark`, `ct_label`).

#### Trường hợp sự cố thực tế

Cluster OVN multi-node, tenant phàn nàn east-west traffic giữa hai VM cùng Logical_Switch bị drop ngẫu nhiên ~3% sau khi nâng cấp OVS từ 2.13 lên 2.17. `ovs-ofctl dump-flows br-int` cho thấy luật pipeline OVN hoàn chỉnh. `ovs-appctl dpif/dump-flows -m | grep <vm-mac>` lại trả về hai entry cùng match nhưng `actions:` khác nhau — một entry dẫn về `output:5`, một về `output:7`. Nguyên nhân: revalidator chưa kịp evict megaflow stale sau khi `ovn-controller` reprogram lại pipeline; trong cửa sổ vài trăm ms, hai megaflow trùng match cùng tồn tại. Khắc phục tạm: `ovs-appctl revalidator/purge`. Khắc phục bền: nâng kernel + cấu hình `ovs-vswitchd` `other_config:revalidator-threads` phù hợp (xem Phần 9.26).

#### Bài tập synthetic

Trên host single-node OVS 2.17:

```shell
sudo ovs-vsctl --may-exist add-br br-lab
sudo ip netns add ns1 && sudo ip netns add ns2
sudo ip link add veth1 type veth peer name veth1p
sudo ip link add veth2 type veth peer name veth2p
sudo ip link set veth1p netns ns1 && sudo ip link set veth2p netns ns2
sudo ovs-vsctl add-port br-lab veth1 -- add-port br-lab veth2

sudo ip netns exec ns2 ip addr add 10.20.0.2/24 dev veth2p
sudo ip link set veth1 up; sudo ip link set veth2 up
sudo ip netns exec ns1 ip link set veth1p up
```

#### So sánh với hệ khác

`dpif/dump-flows` về tinh thần tương tự `tc -s filter show dev <iface>` (Linux traffic control hiển thị filter đã cài trong kernel + counter), hoặc `nft list ruleset -a` cho nftables. Khác biệt: tc/nftables chia luật theo interface/chain; `dpif/dump-flows` chia luật theo bridge/datapath, và mỗi entry là kết quả xlate động từ tầng OpenFlow chứ không phải luật khai báo cố định.

### 9.11.X.2 `ovs-appctl dpif/dump-conntrack`

#### Khái niệm

`dpif/dump-conntrack` (alias gần với `dpctl/dump-conntrack`) liệt kê các entry conntrack mà OVS đang sử dụng cho action `ct()`. Mỗi entry mô tả một kết nối: tuple gốc (`orig`), tuple phản hồi (`reply`), zone partition, mark, labels, và state machine (`NEW`, `ESTABLISHED`, `RELATED`, `INVALID`). Lệnh đi qua kênh `dpif` của `ovs-vswitchd`, đảm bảo nhìn đúng bảng mà OVS đang attach (kernel netfilter conntrack cho `dpif-netlink`, hoặc bảng conntrack userspace cho `dpif-netdev`).

#### Vai trò trong kiến trúc

OVS không tự code lại conntrack — kernel datapath dùng lại module `nf_conntrack` của Linux netfilter, userspace datapath dùng implementation riêng trong `lib/conntrack.c`. Action `ct()` trong OpenFlow là cầu nối: nó đẩy gói qua conntrack lookup, populate metadata `ct_state`/`ct_zone`/..., rồi đưa kết quả về pipeline. `dpif/dump-conntrack` là cách nhanh nhất để nhìn trực tiếp bảng đó từ góc nhìn OVS, không phải đoán qua `conntrack -L` (Linux native, không phân biệt zone của OVN ra sao).

#### Cú pháp + tham số

```
ovs-appctl dpif/dump-conntrack [-m] [zone=N]
```

- `-m` thêm `zone`, `mark`, `labels` vào mỗi dòng (mặc định ẩn).
- `zone=N` chỉ in entry thuộc zone số N. Hữu dụng khi OVN dùng nhiều zone phân cách per-tenant.

#### Mẫu output thực tế

```shell
$ sudo ovs-appctl dpif/dump-conntrack -m zone=1 | head -2
tcp,orig=(src=10.1.0.5,dst=10.1.0.10,sport=50000,dport=80),
    reply=(src=10.1.0.10,dst=10.1.0.5,sport=80,dport=50000),
    zone=1,mark=0xab,labels=0,protoinfo=(state=ESTABLISHED)
udp,orig=(src=10.1.0.5,dst=10.1.0.20,sport=33000,dport=53),
    reply=(src=10.1.0.20,dst=10.1.0.5,sport=53,dport=33000),
    zone=1,mark=0,labels=0
```

Ý nghĩa từng trường: `orig` là chiều gói đầu tiên đi qua OVS, `reply` là chiều ngược lại OVS đã thấy (nếu chưa thấy, sẽ là `UNREPLIED`), `protoinfo` chỉ có cho TCP/SCTP/DCCP (giao thức có state machine), `mark`/`labels` là dữ liệu OVN ghi qua action `ct(commit, exec(set_field:0xab->ct_mark))`.

#### Quy trình sử dụng

1. Xác định zone cần điều tra: với OVN, `ovn-sbctl list datapath_binding | grep tunnel_key` cho tunnel_key tương ứng zone (Phần 13.5).
2. `ovs-appctl dpif/dump-conntrack -m zone=<N>` lọc đúng zone.
3. Đối chiếu `protoinfo=(state=...)` với rule `ct_state=...` trong `ovs-ofctl dump-flows br-int`.
4. Nếu thấy `state=INVALID` cho kết nối hợp lệ → suspect mismatch giữa OF flow và sự thật conntrack (xem Phần 9.24).

#### Lỗi thường gặp + tín hiệu chẩn đoán

| Triệu chứng | Nguyên nhân | Khắc phục |
|-------------|-------------|-----------|
| Bảng rỗng dù có traffic | `ct()` action không bao giờ được invoke (rule sai) | `ovs-ofctl dump-flows br-int` kiểm tra rule có `ct(...)` không |
| Bảng đầy thiếu zone | OVN compile thiếu `ct_zone` field, gói rơi vào zone 0 mặc định | Check `ovn-sbctl list logical_flow` xem `Logical_Switch` có DROP/ALLOW rule chuẩn |
| `nf_conntrack table full` trong dmesg | Kernel `nf_conntrack_max` quá thấp | `sysctl net.netfilter.nf_conntrack_max=1048576` |
| Entry tồn tại nhưng `state=INVALID` cho kết nối thực | Zone collision giữa nhiều logical context | OVN 24.03+ cho phép set explicit `zone-id` per Logical_Switch |

#### Cách quan sát + xác minh

Cặp đôi với `ofproto/trace`:

```shell
$ sudo ovs-appctl ofproto/trace br-int 'in_port=3,tcp,ct_state=+trk+est,...'
# Trace shows action ct() commit; sau đó:
$ sudo ovs-appctl dpif/dump-conntrack -m zone=1 | grep "10.1.0.5"
```

Hai output đối chiếu cho biết action có thực sự commit đúng entry hay không.

#### Source code tham chiếu

- `ofproto/ofproto-dpif.c::ofproto_unixctl_dpif_dump_conntrack()` registers subcommand.
- `lib/conntrack.c::conntrack_dump_*()` userspace impl.
- `lib/dpif-netlink.c` cầu nối Netlink xuống `nf_conntrack`.
- Linux side: `net/netfilter/nf_conntrack_core.c::nf_conntrack_in()`.

#### Khác biệt giữa các phiên bản

- OVS 2.5 (2016): conntrack support đầu tiên cùng action `ct()`.
- OVS 2.6 (2016): zone partition usable từ pipeline OF.
- OVS 2.8 (2017): `ct_label` 128-bit + ALG (FTP, TFTP).
- OVS 2.13 (2020): conntrack timeout policy table riêng.

#### Bài tập synthetic

Mở rộng lab §9.11.X.1: cài thêm rule conntrack stateful firewall:

```shell
sudo ovs-ofctl add-flow br-lab 'priority=100,ip,ct_state=-trk,actions=ct(table=0,zone=42)'
sudo ovs-ofctl add-flow br-lab 'priority=200,ip,ct_state=+trk+new,actions=ct(commit,zone=42),NORMAL'
sudo ovs-ofctl add-flow br-lab 'priority=200,ip,ct_state=+trk+est,actions=NORMAL'
sudo ip netns exec ns1 ping -c 3 10.20.0.2
sudo ovs-appctl dpif/dump-conntrack -m zone=42
```

Pass: thấy entry ICMP cùng zone=42, `protoinfo` (ICMP không có state TCP nhưng có entry).

#### So sánh với hệ khác

Tương đương `conntrack -L` (Linux netfilter userspace tool) cho kernel datapath, nhưng `dpif/dump-conntrack` cho phép filter theo zone OVS-specific — `conntrack -L` không phân biệt zone của OVN khi nhiều OVS bridge dùng chung kernel namespace. Ngoài Linux, không có hệ chuyển mạch nào có cấu trúc tương đương trực tiếp; Cisco ASA `show conn` gần nhất nhưng kín, không expose state machine cho rule mở rộng.

### 9.11.X.3 `ovs-appctl ofproto/list-tunnels`

#### Khái niệm

`ofproto/list-tunnels` liệt kê toàn bộ tunnel **đang hoạt động** mà bridge đang thấy: source IP, destination IP, transport (UDP/IP-proto), loại tunnel (`geneve`, `vxlan`, `gre`, `stt`), khóa định danh (`key`/`tun_id`). Khác với `ovs-vsctl list interface type=geneve` (liệt kê interface đã tạo trong control plane), lệnh này nhìn từ ofproto: tunnel đã được lập trình hoàn chỉnh xuống datapath, có ARP/ND đã giải, sẵn sàng forward.

#### Vai trò trong kiến trúc

OVN dùng tunnel Geneve để encapsulate tenant traffic giữa các chassis. Khi gói rời chassis A đến chassis B, OVS phải biết: B có IP nào, dùng port UDP nào (Geneve mặc định 6081), Geneve VNI nào tương ứng tenant. `ofproto/list-tunnels` là cách kiểm tra nhanh xem topology mesh tunnel đã được dựng đầy đủ giữa các chassis chưa, mà không cần đọc OVSDB từng cột rời rạc.

#### Cú pháp + tham số

```
ovs-appctl ofproto/list-tunnels [BRIDGE]
```

`BRIDGE` tùy chọn (mặc định liệt kê mọi bridge). Lệnh không có flag thêm.

#### Mẫu output thực tế

```shell
$ sudo ovs-appctl ofproto/list-tunnels br-int
port 4: 192.168.10.21 (geneve: ::->192.168.10.21, key=flow, dp port=4)
port 5: 192.168.10.22 (geneve: ::->192.168.10.22, key=flow, dp port=5)
port 6: 192.168.10.23 (vxlan: ::->192.168.10.23, key=42, dp port=6)
```

Anatomy:

- `port 4` là OpenFlow port number trong `br-int`.
- `192.168.10.21` là remote IP (đầu xa tunnel).
- `geneve:` loại tunnel. Có thể là `vxlan`, `gre`, `stt`, `erspan`.
- `::->192.168.10.21` cặp source/dest. `::` nghĩa là source dùng `local_ip` config hoặc auto-select.
- `key=flow` nghĩa là VNI/key gán động từ pipeline (set bằng action `set_field:0x...->tun_id`); `key=42` nghĩa là static key.
- `dp port=4` datapath port number tương ứng (cùng số trong nhiều trường hợp).

#### Vai trò trong vận hành

Đây là one-liner đầu tiên cho câu hỏi "OVN mesh đã đầy đủ chưa". Trên cluster N chassis, mỗi `br-int` phải có (N-1) tunnel — nếu thiếu, `ovn-controller` chưa register hoặc `Encap` row sai trong `Chassis` table. Đối chiếu với `ovn-sbctl show` xem số chassis SBDB.

#### Lỗi thường gặp + chẩn đoán

| Triệu chứng | Nguyên nhân | Cách kiểm chứng |
|-------------|-------------|-----------------|
| Output rỗng dù bridge có port `geneve` | Datapath chưa attach tunnel; `ovs-vswitchd.log` thường có "tunnel ARP failed" | `ovs-appctl tnl/neigh/show` xem ARP cache |
| Số tunnel ít hơn (N-1) | Một số chassis chưa register hoặc bị evict | `ovn-sbctl --columns=name,encaps list chassis` |
| `key=42` nhưng OVN cần dynamic | Tunnel cấu hình thủ công sai, override OVN | `ovs-vsctl get interface tunN options:key` |

#### Quy trình sử dụng

1. Xác định bridge cần xét: với OVN luôn là `br-int`.
2. `ovs-appctl ofproto/list-tunnels br-int` đếm số dòng.
3. Đối chiếu `ovn-sbctl --columns=name list chassis | wc -l` ra số chassis.
4. Nếu thiếu → `ovs-appctl tnl/neigh/show` + `ovs-appctl tnl/ports/show` (§9.11.9) để định vị thiếu ở đâu.

#### Source code tham chiếu

- `ofproto/ofproto.c::ofproto_unixctl_list_tunnels()` đăng ký subcommand.
- `ofproto/tunnel.c::tnl_port_iterate()` enumerate qua hash table tunnel ports.
- Output struct: `struct tnl_port` chứa `match` (src, dst, in_key, in_key_flow), `netdev`, `dp_port`.

#### Khác biệt giữa các phiên bản

- OVS 2.0 (2014): subcommand xuất hiện cùng tunnel framework refactor.
- OVS 2.6 (2016): hỗ trợ Geneve TLV (option metadata) — output thêm cột nếu có TLV.
- OVS 2.10 (2018): tích hợp tốt hơn với userspace datapath; format ổn định.
- OVS 2.13+: cải thiện display cho IPv6 underlay (có thể thấy tunnel `::ffff:...`).

#### Trường hợp sự cố thực tế

Cluster OpenStack 5 compute, sau khi nâng kernel (5.4 → 5.15) một compute "biến mất" khỏi traffic mesh: VM trên compute đó không liên lạc được với VM trên 4 compute còn lại. `ovn-sbctl show` báo đủ 5 chassis. `ovs-vsctl list interface | grep geneve` trên compute lỗi báo đủ 4 tunnel port. Nhưng `ovs-appctl ofproto/list-tunnels br-int` chỉ trả về 2 dòng. Nguyên nhân: ARP cho 2 endpoint còn lại fail vì underlay routing thay đổi mà host chưa cập nhật. `ovs-appctl tnl/neigh/show` xác nhận thiếu entry. Khắc phục: thêm static neigh hoặc fix routing underlay; sau đó `ofproto/list-tunnels` trở lại đủ 4 dòng.

#### Bài tập synthetic

Hai host (host A `192.168.50.1`, host B `192.168.50.2`), mỗi host có `br-int`:

```shell
# Trên A:
sudo ovs-vsctl add-port br-int gnv0 -- set interface gnv0 type=geneve \
  options:remote_ip=192.168.50.2 options:key=flow
# Trên B: ngược lại remote_ip=192.168.50.1
# Quan sát trên A:
sudo ovs-appctl ofproto/list-tunnels br-int
# Mong đợi: 1 dòng "geneve: ::->192.168.50.2, key=flow"
```

#### So sánh với hệ khác

Linux native dùng `ip -d link show type geneve` để liệt kê tunnel interface, nhưng output đó chỉ là "có cấu hình", không phản ánh "đang sống trong datapath". VMware NSX-T dùng `nsxcli get logical-switch <id> tunnel-status` cho tương đương, nhưng phải qua agent NSX. `ovs-appctl ofproto/list-tunnels` không cần agent, nhanh, đọc trực tiếp ofproto state.

### 9.11.X.4 `ovs-appctl coverage/show`

> **Đã giới thiệu ban đầu ở §9.11.1.3.** Mục này bổ sung khía cạnh chuyên sâu về quy trình vận hành, version drift, và source code anchor để hoàn chỉnh treatment 20 chiều.

#### Vai trò trong kiến trúc forensic

OVS không có metric Prometheus chuẩn out-of-the-box; `coverage/show` là API telemetry chính thức để export. Mỗi sự kiện đáng chú ý trong code path đều có macro `COVERAGE_INC(name)` ở source. Khi điều tra sự cố không có error log rõ ràng (revalidator storm, upcall churn, megaflow flush bất thường), counter delta là bằng chứng số liệu hàng đầu — nhanh hơn `perf` và rõ hơn `strace`.

#### Cú pháp + biến thể

```
ovs-appctl coverage/show
ovs-appctl coverage/read-counter <COUNTER_NAME>
```

Biến thể `read-counter` (xem §9.11.1.4) trả về một số nguyên duy nhất, dùng cho exporter tự viết.

#### Mẫu output đã có ở §9.11.1.3

Tham chiếu output thực tế và bảng counter healthy threshold ở §9.11.1.3 (không lặp lại). Đây là mục bổ sung axis chưa cover ở phần đầu file.

#### Quy trình vận hành thành thạo

1. **Baseline:** chạy `coverage/show` lúc steady-state, ghi lại snapshot `total:` cho 8-10 counter quan trọng (`hit`, `miss`, `upcall_flow_add`, `upcall_flow_del`, `revalidate`, `facet_revalidate`, `bridge_reconfigure`, `ofproto_flush`).
2. **Trigger event:** chạy thao tác (controller restart, OVSDB change, v.v.).
3. **After-snapshot:** chạy lại `coverage/show`.
4. **Compute delta:** mỗi counter `delta = after.total - before.total`. So sánh với expected (ví dụ một controller restart hợp lệ chỉ tăng `bridge_reconfigure` 1, không tăng `ofproto_flush`).
5. **Lập biểu đồ:** với incident dài, lặp `read-counter` mỗi 5 giây, plot.

#### Cách quan sát + xác minh

Để xác nhận một counter đại diện đúng sự kiện ta nghĩ: tăng vlog của module liên quan lên `dbg`, kích hoạt event, đọc cả log lẫn counter. Ví dụ: nghi ngờ `upcall_flow_add` không tăng dù gói tin mới đến — `vlog/set ofproto_dpif_upcall:file:dbg`, sinh traffic mới, đọc `ovs-vswitchd.log`, đối chiếu số lần "add flow" trong log với delta counter.

#### Source code tham chiếu

- `lib/coverage.c::coverage_unixctl_show()` đăng ký subcommand `coverage/show`.
- `lib/coverage.c::coverage_unixctl_read_counter()` cho `coverage/read-counter`.
- Macro `COVERAGE_DEFINE(name)` trong `lib/coverage.h` cùng `COVERAGE_INC(name)` rải rác trong source. Mỗi định nghĩa tạo một `struct coverage_counter` static, gắn vào linked list global.
- `lib/coverage.c::coverage_run()` gọi định kỳ để cập nhật rate 5 giây / 1 phút / 1 giờ.

#### Khác biệt giữa các phiên bản

- OVS 1.x: counter nguyên thủy, ít nhóm chỉ vài chục counter.
- OVS 2.0 (2014): refactor, format `5sec/min/hour/total` ổn định.
- OVS 2.6 (2016): thêm counter `dpif_recv_zero_payload`, conntrack-related (`ct_match`, `ct_zone`).
- OVS 2.10 (2018): thêm counter cho TLV Geneve, p4-related.
- OVS 2.17 (2022): bộ counter ổn định ~150 entry tùy build.

#### Trường hợp sự cố thực tế

Phần 9.26 trình bày "revalidator storm": sự cố sản xuất với OVS 2.13 trong cluster OpenStack, biểu hiện CPU `ovs-vswitchd` tăng đột biến lên 90%+, latency forwarding tăng. `coverage/show` cho thấy `revalidate` rate vọt từ 0.3/sec lên 18/sec, `facet_revalidate` đồng pha. Nguyên nhân: controller (Neutron OVS agent) đang cài và xóa hàng loạt rule cùng cookie liên tục mỗi giây. Khắc phục: tăng `revalidator_threads` từ 4 lên 8 và batch hóa thao tác phía agent. Counter `revalidate` trở lại 0.5/sec sau fix.

#### Bài tập synthetic

```shell
sudo ovs-appctl coverage/show > /tmp/cov_before.txt
# Thay đổi config bridge:
sudo ovs-vsctl set Bridge br-lab fail_mode=secure
sudo ovs-vsctl set Bridge br-lab fail_mode=standalone
sudo ovs-appctl coverage/show > /tmp/cov_after.txt
diff /tmp/cov_before.txt /tmp/cov_after.txt | grep -E "bridge_reconfigure|ofproto_flush"
```

Mong đợi: `bridge_reconfigure` tăng 2, `ofproto_flush` không tăng (chỉ thay đổi metadata, không re-install rule).

#### So sánh với hệ khác

Tương tự `cat /proc/net/snmp` (Linux) hay `netstat -s` cho tổng quan sự kiện stack TCP/UDP. Khác biệt: `/proc/net/snmp` chuẩn hóa theo MIB, tên cố định; `coverage/show` của OVS dày đặc hơn, gắn với code path nội bộ, nhưng không có chuẩn cross-vendor — chỉ kỹ sư đọc code OVS mới biết counter `facet_changed_rule` nghĩa là gì. Đối với Cisco IOS, lệnh tương đương là `show platform forward statistics`, kín hơn nhưng vẫn cùng tinh thần "đếm sự kiện code path".

### 9.11.X.5 `ovs-appctl vlog/set`

> **Đã giới thiệu ban đầu ở §9.11.1.5.** Mục này hoàn chỉnh treatment với chi tiết về cú pháp 3-thành-phần, runtime semantics, lỗi thường gặp, và source code anchor.

#### Khái niệm

`vlog/set` là subcommand điều chỉnh log level **theo từng module + từng đích đến** mà không cần restart `ovs-vswitchd` hoặc `ovsdb-server`. Mỗi file source `.c` của OVS được khai báo qua macro `VLOG_DEFINE_THIS_MODULE(<name>)` để trở thành một module log độc lập (`dpif`, `bfd`, `bond`, `ofproto`, `connmgr`, …). Thay đổi level một module = filter log theo module đó tức thì, hữu dụng khi cần debug cùng lúc daemon đang chạy production.

#### Cú pháp 3-thành-phần

```
ovs-appctl vlog/set [SPEC[:SPEC]...]
SPEC: <module>:<destination>:<level>
```

- `<module>`: tên module (`dpif`, `ofproto_dpif_xlate`, …) hoặc `ANY` cho mọi module.
- `<destination>`: `console` (stderr, hiển thị khi chạy foreground), `syslog` (chuyển vào `syslogd`), `file` (`/var/log/openvswitch/ovs-vswitchd.log`). Dùng `ANY` cho cả ba.
- `<level>`: từ chi tiết nhất tới ít nhất: `dbg`, `info`, `warn`, `err`, `emer`, `off`. Mỗi level bao trùm các level cao hơn (đặt `dbg` thì `info`/`warn`/`err`/`emer` cũng vào file).

Có thể bỏ qua thành phần đứng sau bằng cách dùng giá trị mặc định, ví dụ `vlog/set dbg` ép mọi module ghi level `dbg` ra mọi đích — gây ngập log, hiếm khi nên dùng.

#### Mẫu output + ngữ nghĩa

```shell
$ sudo ovs-appctl vlog/set dpif:file:dbg
$ sudo ovs-appctl vlog/list | grep '^dpif '
dpif             OFF        ERR       DBG
$ sudo tail -n 5 /var/log/openvswitch/ovs-vswitchd.log
2026-04-24T11:15:23.412Z|00042|dpif(handler1)|DBG|system@ovs-system: \
  port_query_by_number(3): pkts:1024 bytes:131072
```

Format dòng log: `<timestamp>|<seq>|<module>(<thread>)|<level>|<message>`. Sequence number tăng dần per daemon process.

#### Vai trò vận hành

Log level mặc định (`info` ra file, `err` ra console) được chọn để file log không phình quá nhanh, nhưng đủ chi tiết phát hiện lỗi rõ ràng. Khi cần debug sâu (xem upcall path, xem packet ra vào netlink) ta tạm thời tăng module liên quan lên `dbg`, làm việc, rồi reset:

```shell
sudo ovs-appctl vlog/set dpif:file:dbg ofproto_dpif_upcall:file:dbg
# ... reproduce sự cố ...
sudo ovs-appctl vlog/set dpif:file:info ofproto_dpif_upcall:file:info
```

Quên reset là lỗi phổ biến: log file phình tới hàng GB sau vài giờ, ảnh hưởng I/O.

#### Lỗi thường gặp + tín hiệu chẩn đoán

| Triệu chứng | Nguyên nhân | Khắc phục |
|-------------|-------------|-----------|
| `ovs-appctl: "vlog/set" command requires at least 1 argument` | Quên đối số SPEC | Cung cấp SPEC theo dạng `module:dest:level` |
| Log không chuyển level dù lệnh báo OK | Tên module sai (case-sensitive trong vài phiên bản, hoặc gõ thiếu underscore) | `ovs-appctl vlog/list` so chiếu tên chính xác |
| Log ra `console` không thấy | Daemon chạy detached, không có `console` đầu ra | Đổi đích sang `syslog` hoặc `file` |
| `/var/log/openvswitch/ovs-vswitchd.log` đầy đĩa sau 1 đêm | Quên reset từ `dbg` về `info` | `vlog/set ANY:file:info` rồi `vlog/reopen` |

#### Quy trình sử dụng

1. `ovs-appctl vlog/list | grep <module>` xác nhận tên module.
2. `ovs-appctl vlog/set <module>:file:dbg` bật chi tiết.
3. Reproduce sự cố trong 30-60 giây.
4. `tail -n 200 /var/log/openvswitch/ovs-vswitchd.log` xem log.
5. **Bắt buộc**: reset `ovs-appctl vlog/set <module>:file:info` ngay sau debug.
6. Nếu file log quá to: `mv ovs-vswitchd.log ovs-vswitchd.log.old; ovs-appctl vlog/reopen` (xem §9.11.1.6).

#### Source code tham chiếu

- `lib/vlog.c::vlog_unixctl_set()` đăng ký subcommand qua `unixctl_command_register("vlog/set", ...)`.
- `lib/vlog.c::vlog_set_levels_from_string()` parse SPEC và áp dụng.
- `lib/vlog.h` macro `VLOG_DEFINE_THIS_MODULE`, `VLOG_DBG`, `VLOG_INFO`, …
- Mỗi module = một `struct vlog_module` static, có 3 cell level `console/syslog/file`.

#### Khác biệt giữa các phiên bản

- OVS 1.x: chỉ có 1 destination chung; cú pháp `vlog/set <module>:<level>` (không có dest).
- OVS 2.0 (2014): tách 3 destination, format `module:dest:level` ổn định.
- OVS 2.4 (2015): hỗ trợ rate-limited logging (`VLOG_*_RL`) để tránh log flood.
- OVS 2.10+ (2018): hỗ trợ chuyển định dạng JSON cho destination `file` nếu build phù hợp (rất hiếm dùng).

#### Trường hợp sự cố thực tế

Cluster OVS production, một bridge bất ngờ drop traffic ICMP nhưng pass TCP. `dpif/dump-flows` cho thấy luật bình thường. Bật `vlog/set ofproto_dpif_xlate:file:dbg` 30 giây, log lộ ra dòng "translation failed: drop reason: action set has conflicting outputs". Nguyên nhân: rule `actions=output:1,output:2,clone(output:3)` xung đột flag mode trong xlate. Sau khi fix flow rule và reset vlog, sự cố hết. Nếu không có `vlog/set` để bật dbg ngay tại tầng xlate, phải reproduce sự cố trên môi trường lab — mất nhiều giờ.

#### Bài tập synthetic

```shell
sudo ovs-appctl vlog/list | grep -E 'bond|bridge|connmgr' | head -3
sudo ovs-appctl vlog/set bridge:file:dbg
sudo ovs-vsctl --may-exist add-br br-vlog-test
sudo ovs-vsctl del-br br-vlog-test
sudo tail -n 30 /var/log/openvswitch/ovs-vswitchd.log | grep '|bridge|'
sudo ovs-appctl vlog/set bridge:file:info
```

Pass: thấy ít nhất 4 dòng `bridge|DBG` về create/destroy bridge; sau reset không còn dòng `DBG` mới.

#### So sánh với hệ khác

Linux kernel có `dmesg --console-level <0-7>` và `printk` ratelimit, nhưng không phân theo module mức độ chi tiết như OVS — kernel chỉ phân theo subsystem qua `dyndbg`. systemd-journald có `journalctl SYSLOG_IDENTIFIER=...` để filter theo identifier, gần với khái niệm module nhưng lọc ở tầng đọc, không tắt ghi từ source. HAProxy có `set log-level` runtime tương tự nhưng chỉ một mức global, thiếu khái niệm 3-thành-phần module/dest/level. OVS `vlog/set` là một trong những API log control linh hoạt nhất trong các daemon mạng userspace.

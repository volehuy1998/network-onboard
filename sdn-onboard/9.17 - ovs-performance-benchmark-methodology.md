## Mục tiêu bài học

Sau khi hoàn thành phần này, bạn có thể:

1. **Mô tả (Understand)** các chỉ số hiệu năng chính của OVS data plane: băng thông thực tế (Mpps), độ trễ per-gói tin (microseconds), flow scaling (hiệu năng khi flow table tăng), tốc độ thiết lập connection.

3. **Áp dụng (Apply)** thiết lập testbed benchmark với pktgen + gói tin capture, chạy 3 workload chuẩn (gói tin nhỏ 64B, IMIX, lớn 1500B), thu thập Mpps + CPU + memory.
4. **Đánh giá (Evaluate)** kết quả benchmark để quyết định kiến trúc triển khai (kernel so với DPDK so với AF_XDP) và lập kế hoạch dung lượng cho môi trường production.

Yêu cầu kiến thức: Phần 9.2 đường tra cứu datapath, Phần 9.3 chế độ DPDK/AF_XDP, Phần 9.15 classifier tác động đến hiệu năng, Phần 16.0 tinh chỉnh kernel tổng quan.

## 9.17.1 Các chỉ số hiệu năng OVS cần đo

Benchmark data plane OVS có nhiều chiều, chọn đo chỉ số nào phụ thuộc vào field hợp sử dụng môi trường production:

**(1) Băng thông thực tế (Mpps):** số gói tin per second OVS có thể forward. Chỉ số quan trọng nhất cho workload tốc độ gói tin cao (giảm thiểu DDoS, UPF viễn thông). Đo riêng cho kích thước gói tin khác nhau:
- Gói tin 64 byte (Ethernet tối thiểu): stress test trên CPU, giới hạn bởi overhead xử lý gói tin.
- Gói tin 1500 byte (MTU tối đa): giới hạn bởi băng thông NIC.
- IMIX (Internet Mix): 60% 64B + 20% 512B + 20% 1500B: mô phỏng traffic thực.

**(2) Bandwidth (Gbps):** băng thông thực tế × kích thước gói tin. Chỉ số quan trọng cho workload nặng băng thông (streaming video, storage). Với NIC 100GbE, Mpps ceiling khoảng 150 Mpps cho 64B (line-rate) hoặc 8.2 Mpps cho 1500B.

**(3) Độ trễ per-gói tin (microseconds):** thời gian một gói tin từ input interface đến output interface trên OVS. Chỉ số quan trọng cho workload nhạy cảm độ trễ (HFT, gaming, 5G tín hiệu thời gian thực):
- Kernel datapath: 20-50 µs.
- Userspace DPDK: 2-10 µs.
- AF_XDP: 5-15 µs.

**(4) Flow scaling:** hiệu năng khi flow table tăng. Với 100 flow rule, Mpps = X. Với 10K flow rule, Mpps còn bao nhiêu? Nếu giảm mạnh → điểm nghẽn classifier (xem Phần 9.15).

**(5) Tốc độ thiết lập connection:** số TCP connection mới/giây OVS xử lý được. Quan trọng cho OVS-DPDK với conntrack, mỗi connection mới = entry conntrack + cache miss classifier lần đầu.

**(6) Dấu chân memory:** RAM tiêu thụ bởi ovs-vswitchd + kernel datapath. Với 100K flow, OVS userspace có thể tiêu thụ 200-500 MB.

**(7) Overhead CPU per-core:** % CPU một core dành cho xử lý gói tin. Kernel datapath tuyến tính theo tần suất traffic (điều khiển bằng ngắt). DPDK chế độ thăm dò đốt 100% core dù không có traffic.

## 9.17.2 Công cụ benchmark chuẩn

**pktgen-dpdk:** công cụ sinh gói tin tốc độ cao chạy trên DPDK, có thể đạt line-rate 100 Gbps trên NIC phù hợp. Dùng cho stress test băng thông thực tế:
```
[pktgen-host]# cd /usr/src/pktgen-dpdk
[pktgen-host]# ./app/x86_64-native-linuxapp-gcc/pktgen -c 0xff -n 4 -- -P -m "[1:2].0"
Pktgen:/> set 0 count 1000000
Pktgen:/> set 0 size 64
Pktgen:/> set 0 rate 100
Pktgen:/> start 0
```
Pktgen sinh 1 triệu gói tin 64B ở tốc độ 100% line rate, gửi qua port 0. Đếm gói tin thực tế được forward bởi OVS ở phía nhận.

**dpdk-testpmd:** công cụ forward gói tin DPDK, dùng để benchmark độ trễ/tỷ lệ drop. Chạy ở node nhận để đo bao nhiêu gói tin được nhận thành công:
```
[rx-host]# testpmd -- -i --forward-mode=rxonly
testpmd> start
```
Sau thời gian chạy, `stop` và xem stats:
```
testpmd> stop
  ---------------------- Forward statistics for port 0  ----------------------
  RX-packets: 98234567   RX-dropped: 1234       RX-total: 98235801
```
Tỷ lệ drop = 1234/98235801 = 0.001%, chấp nhận được cho độ tin cậy 99.99%.

**cbench:** công cụ benchmark OpenFlow controller, gửi packet_in liên tục, đếm bao nhiêu reply flow_mod/giây. Dùng để đo hiệu năng control plane:
```
[host]# cbench -c localhost -p 6653 -m 10000 -l 10 -s 16 -M 1000 -t
```
Output: flow/second (trung bình + max + min).

**iperf3:** benchmark TCP băng thông thực tế end-to-end. Không đo OVS cụ thể nhưng cho thấy băng thông thực tế toàn bộ stack.
```
[server]# iperf3 -s
[client]# iperf3 -c <server_ip> -t 30 -P 8
```
8 flow song song trong 30 giây, output Gbps tổng hợp.

**Bộ sinh tải chuyên dụng:** Cisco Trex, Ixia, Spirent, công cụ sinh gói tin chuyên nghiệp cho benchmark chính xác. Không phù hợp lab nhỏ nhưng chuẩn cho chứng nhận môi trường production.

## 9.17.3 Thiết kế testbed — kiểm tra có ý nghĩa

Setup chuẩn cho benchmark OVS:

**Topology 3-node:**
```
[generator] ---- 100GbE ----> [OVS under test] ---- 100GbE ----> [receiver]
```

- **Generator:** máy chạy pktgen-dpdk sinh traffic.
- **OVS under test:** node cần benchmark: có bridge OVS với flow table của field hợp test.
- **Receiver:** máy chạy testpmd đo gói tin nhận được.

**Loopback single-node (đơn giản hơn):**
```
[OVS] --- port A ---> bộ sinh traffic ---> port B --- [OVS]
```
Cùng một node nhưng generator + receiver đều chạy trên đó. Dễ setup nhưng không đại diện môi trường production (CPU chia sẻ giữa generator và OVS).

**Cân nhắc cô lập:**
- **Cô lập CPU:** dùng `isolcpus` kernel cmdline để dành riêng CPU cho thread PMD OVS (nếu chế độ DPDK). Generator chạy trên CPU khác.
- **Vị trí NUMA:** NIC, PMD OVS, và mempool phải cùng NUMA node. Traffic cross-socket giảm băng thông thực tế 30-50% (xem Phần 16.0, 16.1).
- **Ái lực IRQ:** tách IRQ của NIC generator khỏi CPU của OVS.
- **Hyperthreading:** disable HT trong BIOS hoặc dùng chỉ core vật lý cho PMD. HT sibling gây tranh chấp cache.

**Giai đoạn warm-up:** chạy 5-10 giây trước khi đo để:
- Flow table điền đầy (classifier compile megaflow cho steady state).
- Tần số CPU scale up (intel_pstate có thể start ở base frequency).
- Bộ đếm NIC đồng bộ.

**Đo steady-state:** đo trong 60-120 giây sau warm-up. Tính trung bình + phần trăm percentile p50/p95/p99.

## 9.17.4 Workload chuẩn — 64B, IMIX, 1500B

Ba workload chuẩn cho benchmark OVS:

**Workload 1. Gói tin 64 byte (stress CPU):**
- Gói tin nhỏ nhất Ethernet (64B = 14B header + 46B payload + 4B FCS).
- Stress test xử lý gói tin CPU: mỗi gói tin cần đi qua classifier, actions, forward.
- Line-rate trên 100GbE = 148.8 Mpps. Thực tế OVS-DPDK đạt 40-60 Mpps per core → cần 3-4 core để bão hòa 100GbE với 64B.

**Workload 2 — IMIX (thực tế):**
- 7 gói tin × 64B + 4 gói tin × 594B + 1 gói tin × 1518B (pattern RFC 2544).
- Giả lập traffic Internet thực tế (kết hợp nhiều protocol).
- Băng thông thực tế thường cao hơn chỉ 64B nhưng thấp hơn chỉ 1500B.

**Workload 3. Gói tin 1500 byte (stress băng thông):**
- Gói tin lớn (MTU Ethernet default 1500B).
- Stress test băng thông NIC hơn CPU.
- Line-rate 100GbE = 8.2 Mpps × 1500B = 100 Gbps. OVS dễ dàng đạt line-rate với 1500B.

**Kết quả điển hình cho OVS-DPDK trên Intel Xeon Gold + ConnectX-5 (100GbE), 4 core PMD:**

| Workload | Băng thông thực tế | Sử dụng CPU | Tỷ lệ drop |
|----------|-----------|-------------|-----------|
| 64B | 80-100 Mpps | 4 core × 100% | 0% nếu tồn đọng đủ |
| IMIX | 20 Mpps (~18 Gbps) | 2-3 core × 100% | 0% |
| 1500B | 8.2 Mpps (100 Gbps line rate) | 1 core × 80% | 0% |

**Kernel datapath OVS cùng NIC không DPDK:**

| Workload | Băng thông thực tế | Ghi chú |
|----------|-----------|---------|
| 64B | 2-5 Mpps | Giới hạn bởi ngắt + chuyển đổi ngữ cảnh |
| IMIX | 3-6 Mpps (~6 Gbps) | Hành vi cache tốt hơn chỉ 64B |
| 1500B | 5-8 Mpps (~50 Gbps) | Vẫn xa line rate 100GbE |

## 9.17.5 Đo flow scaling

Test flow scaling: start với 1 flow, tăng dần lên 10, 100, 1K, 10K flow, đo Mpps ở mỗi mốc. Quan sát phổ biến:

**Kernel datapath + megaflow:** gần như không ảnh hưởng bởi số flow (tỷ lệ hit megaflow cache cao). Băng thông thực tế giữ ổn định từ 1 đến 10K flow.

**Vô hiệu hóa megaflow:** khi add/delete flow thường xuyên, cache megaflow bị vô hiệu hóa → băng thông thực tế giảm tạm thời. Benchmark với kết hợp traffic + sửa flow đồng thời cho thấy băng thông thực tế thực môi trường production.

**Overhead classifier (khi megaflow miss):** chỉ xuất hiện cho gói tin mới chưa có megaflow. Tỷ lệ miss dưới 1% → tác động thấp. Tỷ lệ miss trên 10% → cần xem xét thiết kế flow table (xem Phần 9.15).

**Độ phức tạp match flow:** match trên nhiều field = nhiều subtable = k tuple lớn. Test benchmark:
- 1 flow match chỉ `in_port` → 1 subtable, nhanh.
- 1 flow match 12-tuple L2-L4 → 1 subtable nhưng match phức tạp, chậm biên.
- 100 flow đa dạng pattern → 20-50 subtable → classifier phải duyệt tất cả khi megaflow miss.

## 9.17.6 Lập hồ sơ CPU + memory

Benchmark không chỉ là Mpps, cần đo chi phí CPU và memory tương ứng.

**Hồ sơ CPU với perf:**
```
[host]# perf record -F 99 -g -p $(pidof ovs-vswitchd) -- sleep 30
[host]# perf report
```
Output top hàm chiếm CPU:
- `netdev_dpdk_rxq_recv`: đường RX DPDK.
- `dp_packet_batch_init_packet_fields`: xử lý theo lô.
- `miniflow_extract`: parse header từ gói tin.
- `classifier_lookup`: tra cứu flow table (nếu megaflow miss).

Phần trăm `classifier_lookup` > 10% → signal cần tune flow table.

**Flamegraph** trực quan hóa hồ sơ CPU:
```
[host]# perf record -F 99 -g -p $(pidof ovs-vswitchd) -- sleep 30
[host]# perf script | stackcollapse-perf.pl | flamegraph.pl > ovs-flame.svg
```

**Hồ sơ memory:** `ps -o rss -p $(pidof ovs-vswitchd)` cho RSS (Resident Set Size). Theo dõi theo thời gian:
```
[host]# while true; do
[host]>   ps -o rss= -p $(pidof ovs-vswitchd)
[host]>   sleep 60
[host]> done > ovs-memory.log
```

Giám sát rò rỉ memory: RSS tăng tuyến tính theo thời gian + không tương quan với số flow → có thể signal rò rỉ.

**valgrind --tool=memcheck:** chỉ dùng cho phát triển, quá chậm cho môi trường production. Phát hiện rò rỉ memory, đọc/ghi sai.

### ▶ Guided Exercise 1: Benchmark kernel datapath OVS với iperf3

> **Trạng thái verification lab:** Output dựa trên OVS 2.17 + dữ liệu benchmark iperf3; chờ kiểm chứng lab thực.

**Mục tiêu:** Đo băng thông thực tế baseline kernel datapath OVS để so sánh với chế độ DPDK sau.

**Chuẩn bị:** 2 node Ubuntu 22.04 với NIC 10 GbE, OVS 2.17.

**Bước thực hiện:**

**1.** Trên node OVS, tạo bridge:
```
[ovs-node]# ovs-vsctl add-br br0
[ovs-node]# ovs-vsctl add-port br0 eth0
[ovs-node]# ip addr add 10.0.0.1/24 dev br0
[ovs-node]# ip link set br0 up
```

**2.** Trên node client, setup tương tự (hoặc cáp trực tiếp):
```
[client]# ip addr add 10.0.0.2/24 dev eth0
[client]# ip link set eth0 up
```

**3.** Chạy server iperf3 trên node OVS:
```
[ovs-node]# iperf3 -s
```

**4.** Chạy client iperf3 trong 30s với 8 flow song song:
```
[client]# iperf3 -c 10.0.0.1 -t 30 -P 8
```

Kết quả mong đợi:
```
[SUM]   0.00-30.00  sec  27.8 GBytes  7.95 Gbits/sec   retr    cwnd
```

Băng thông thực tế khoảng 8 Gbps trên NIC 10 GbE, đạt 80% line rate (overhead protocol TCP + xử lý gói tin OVS).

**5.** Trong lúc chạy, quan sát thống kê OVS:
```
[ovs-node]# ovs-appctl dpif/show
[ovs-node]# ovs-appctl upcall/show
```

Output: tần suất upcall thấp (dưới 100/s) vì hầu hết gói tin match megaflow.

**Bài học học được:**

Thứ nhất, kernel datapath OVS đạt gần line-rate cho 10 GbE với cache megaflow hoạt động tốt. Tần suất upcall thấp = tỷ lệ hit cache cao = overhead CPU thấp.

Thứ hai, iperf3 với nhiều flow song song bypass giới hạn điều khiển tắc nghẽn TCP của một flow, đo được băng thông thực của NIC + OVS stack.

**Cleanup:**
```
[ovs-node]# ovs-vsctl del-br br0
```

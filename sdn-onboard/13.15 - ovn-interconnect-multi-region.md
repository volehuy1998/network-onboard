## Learning objectives

After completing this section, the engineer can:

1. **Understand** the role of OVN Inter-Connect: a federated multi-region deployment that lets a VM in region A communicate with a VM in region B through a **Transit Switch** (cross-region L2) or a **Transit Router** (cross-region L3), without stretching a single SB DB cluster across every region.

## 13.15.1 Why Inter-Connect is needed: scale, geographic, and regulatory drivers

**Single-region OVN limit:** a 3-node SB DB Raft cluster (see Section 10.1) can serve hundreds of chassis within one datacentre. When you want chassis in another DC (for geographic redundancy, regulatory data sovereignty, or edge computing) to share the L2 overlay with chassis in the main DC, two choices exist:

**Option 1: stretch a single SB DB cluster across regions.**
- Place the 3 Raft nodes in 3 different DCs and tunnel the SB DB protocol cross-region.
- Problems:
  - Raft consensus latency equals the maximum round-trip among the 3 nodes. Cross-region 100 to 300 ms means every `ovn-nbctl` configuration change takes about 1 to 5 seconds to propagate.
  - A network partition between regions causes Raft quorum loss, freezing the entire cluster (split-brain protection).
  - Compliance: data sovereignty (GDPR, PIPL) requires that one region's data not leak to another, but a single SB DB cluster stores data for the entire fleet.

**Option 2: Inter-Connect (IC) federated deployment.**
- Each region has an INDEPENDENT SB DB cluster (its own 3-node Raft within the region).
- A central **IC** cluster (IC_NB and IC_SB) acts as the connector between regions.
- The `ovn-ic` daemon in each region reads IC_SB and syncs only cross-region resources (Transit Switch, route info) into the local SB DB.
- Each region handles its own local tenants; only cross-region traffic goes through IC.

**Trade-off versus stretching:**

| Criterion | Stretched single SB | OVN Inter-Connect |
|----------|-------------------|-------------------|
| Configuration latency | High (cross-region Raft) | Low (Raft within a region) |
| Failure isolation | Whole cluster dies if quorum is lost | Only the affected region dies |
| Data sovereignty | Hard (data is fleet-wide) | Easy (data is per-region) |
| Maximum chassis scale | About 500 to 1000 | About 10K plus (sum across regions) |
| Setup complexity | Simple (1 cluster) | Complex (4 DBs and 2 daemon types) |
| Cross-region first-packet latency | Low | Higher (Geneve transit tunnel) |

OVN Inter-Connect was introduced in OVN 20.06 and stabilised in OVN 22.03 and later. Main use cases: multi-DC cloud providers, multi-edge telecom, and enterprise hybrid cloud (on-premises plus AWS).

> **Version note:** the curriculum follows the OVN 22.03.8 baseline. The IC schema changed slightly between 22.03 and 24.03 (added `nb_cfg` synchronisation tracking, added the encap=vxlan option). This file focuses on 22.03.

## 13.15.2 The 4-database architecture

```
                    +-----------------------------+
                    |    IC central cluster       |
                    |  +----------+ +----------+  |
                    |  |  IC_NB   | |  IC_SB   |  |
                    |  |  cluster | |  cluster |  |
                    |  |  (3-node)| |  (3-node)|  |
                    |  +----------+ +----------+  |
                    |        |          ^        |
                    |        v          |        |
                    |    [ovn-ic-northd]         |
                    |    (compile NB → SB cho IC)|
                    +-----------------------------+
                            |              ^
                            | IC_NB write  | IC_SB read
                            | + IC_SB read | + write Chassis claim
                            v              |
                    +---------+      +---------+
                    | Region A|      | Region B|
                    |         |      |         |
                    | +-----+ |      | +-----+ |
                    | |ovn- | |      | |ovn- | |
                    | |ic   | |      | |ic   | |
                    | +-----+ |      | +-----+ |
                    |   |     |      |   |     |
                    |   v     |      |   v     |
                    | +---+   |      | +---+   |
                    | |NB |   |      | |NB |   |
                    | |+SB|   |      | |+SB|   |
                    | +---+   |      | +---+   |
                    | (region |      | (region |
                    |  local  |      |  local  |
                    |  cluster|      |  cluster|
                    +---------+      +---------+
                         |                |
                    [chassis]         [chassis]
                    [+VM]             [+VM]
                            ↕  Geneve transit ↕
                            (cross-region tunnel)
```

**Role of the 4 databases:**

| Database | Cluster location | Purpose | Daemon writer |
|----------|------------------|---------|----------------|
| **OVN_Northbound (NB)** | Per-region | Tenant intent for the local region | CMS (Neutron, OVN-K8s) through `ovn-nbctl` |
| **OVN_Southbound (SB)** | Per-region | Logical Flow and Port_Binding for local chassis | `ovn-northd` per region |
| **IC_NB (Inter-Connect Northbound)** | Central | Transit Switch and AvailabilityZone definitions | Operator through `ovn-ic-nbctl` |
| **IC_SB (Inter-Connect Southbound)** | Central | Datapath_Binding, cross-region Port_Binding, and Route | `ovn-ic-northd` (compile) plus `ovn-ic` (sync) |

**Data flow direction:**

1. The operator creates a Transit Switch in IC_NB through `ovn-ic-nbctl ts-add transit1`.
2. `ovn-ic-northd` compiles IC_NB into IC_SB: it creates a Datapath_Binding for transit1 and allocates a cross-region tunnel_key.
3. The `ovn-ic` daemon in each region reads IC_SB and syncs the Datapath_Binding for transit1 into that region's local SB DB.
4. The `ovn-controller` on each chassis reads its local SB, learns about transit1, and installs flows on `br-int` to tunnel to the other region.

## 13.15.3 Daemons: `ovn-ic` per region and `ovn-ic-northd` central

### `ovn-ic` daemon

Per-region daemon. Runs on a management server in each region (not on every chassis).

**Responsibilities:**
- Read the central IC_SB (Datapath_Binding, cross-region Port_Binding, Route).
- Sync down to the local SB DB of the region.
- Write the chassis claim and route advertisement back to IC_SB.

**Synopsis:**

```
ovn-ic [options]
```

**Main options (verified against the `ovn-ic(8)` man page):**

| Option | Purpose |
|--------|----------|
| `--ovnnb-db=<db>` | Local NB DB endpoint (default `unix:/ovnnb_db.sock` or `$OVN_NB_DB`) |
| `--ovnsb-db=<db>` | Local SB DB endpoint |
| `--ic-nb-db=<db>` | IC_NB DB endpoint (default `unix:/ovn_ic_nb_db.sock`) |
| `--ic-sb-db=<db>` | IC_SB DB endpoint |
| `--pidfile`, `--detach` | Daemon lifecycle |
| `--monitor` | Spawn a monitor process for crash recovery |
| `--unixctl=<socket>` | Control socket for `ovn-appctl` |
| `-p/-c/-C` SSL options | mTLS connection to IC central |

**Runtime control through `ovn-appctl`:**

```
ovn-appctl -t ovn-ic exit       # Graceful stop
ovn-appctl -t ovn-ic pause      # Stop processing, release SB lock
ovn-appctl -t ovn-ic resume     # Resume operations
ovn-appctl -t ovn-ic is-paused  # Query status
ovn-appctl -t ovn-ic status     # Show active/standby/paused
```

**HA model:** multiple `ovn-ic` instances can run concurrently within the same region; the OVSDB lock ensures only one is active. The standby is promoted automatically when the active instance fails.

### `ovn-ic-northd` daemon

Central daemon. Runs on the IC central management server (co-located with the IC_NB and IC_SB cluster, or separate).

**Responsibility:** compile IC_NB into IC_SB. Similar to `ovn-northd` for local NB to SB, but at the IC level.

**Compile workflow:**
1. Read IC_NB Transit_Switch, AvailabilityZone, and Route.
2. Allocate a tunnel_key for each Transit_Switch (unique cross-region).
3. Create a Datapath_Binding entry in IC_SB for each Transit_Switch.
4. Create a Port_Binding entry in IC_SB for each LSP attached to a Transit Switch.
5. Compile routes from the Route table into Path_Binding (for the Transit Router L3).

## 13.15.4 Transit Switch, the cross-region L2 bridge

A **Transit Switch** is a central entity that defines an "L2 broadcast domain extended across regions". A VM in region A binds to the Transit Switch through an LSP of type remote, the traffic is tunnelled to region B, and the VM in region B sees it as if on the same L2.

**Operational workflow:**

1. **The operator creates the Transit Switch in IC_NB:**
   ```bash
   ovn-ic-nbctl ts-add ts-tenant-blue
   ```

2. **The operator binds the region A logical switch to the Transit Switch:**
   ```bash
   # On the region A NB DB
   ovn-nbctl lsp-add ls-blue-rA ls-blue-rA-to-ts -- \
     lsp-set-type ls-blue-rA-to-ts router -- \
     lsp-set-addresses ls-blue-rA-to-ts router -- \
     lsp-set-options ls-blue-rA-to-ts \
       router-port=lrp-blue-rA \
       requested-tnl-key=<allocated_by_ic_northd>
   ```

3. **Region B binds to the Transit Switch in the same way:**
   ```bash
   # On the region B NB DB
   ovn-nbctl lsp-add ls-blue-rB ls-blue-rB-to-ts -- ...
   ```

4. **`ovn-ic-northd` at the centre creates the binding in IC_SB:** a Datapath_Binding for ts-tenant-blue and a remote Port_Binding for each region's LSP.

5. **`ovn-ic` in each region syncs from IC_SB:** reads the ts-tenant-blue Datapath_Binding and remote Port_Binding entries and installs them into the local SB DB.

6. **`ovn-controller` on each chassis receives the flows:** from the local SB it learns about the Transit Switch and the remote LSP, and installs flows on `br-int` to encapsulate traffic destined for the remote LSP through a Geneve transit tunnel to the other region.

**Transit tunnel:**

The cross-region tunnel uses Geneve by default, with VXLAN as an option. The tunnel endpoint is the Chassis IP of the other region. Importantly, the SOURCE chassis in region A does NOT connect directly to the TARGET chassis in region B; the cross-region tunnel goes through a dedicated **gateway chassis** per region (configured through `external_ids:ovn-is-interconn=true`).

## 13.15.5 Transit Router, cross-region L3 routing

The Transit Switch solves cross-region L2. For L3 (cross-subnet routing across regions), you need a **Transit Router** plus **AvailabilityZone Route**.

**Concepts:**

- Each region is represented as an `AvailabilityZone` (AZ) in IC_NB.
- Routes between AZs are advertised through the IC_SB Route table.
- The Transit Router is a logical entity connecting multiple AZs through the route table.

**Static route propagation (BGP is NOT used, per the North Star ban):**

OVN Inter-Connect supports **static routes** through the IC_NB Route table. The operator manually advertises prefixes:

```bash
ovn-ic-nbctl --ic-nb-db=tcp:<ic-server>:6645 \
  add Route az-A 192.168.0.0/16 az-B-tunnel-endpoint
```

The route is synced down to the local SB of the corresponding AZ, and `ovn-controller` installs the matching flow.

> **Ban note:** OVN also supports **BGP-based dynamic route advertisement** for IC through `ovn-bgp-agent` (a separate component). The curriculum does NOT cover deep BGP integration per the permanent ban directive (see CLAUDE.md North Star). Learners who want deep BGP-EVPN background can read Section 11.2 for an overview, but the BGP control plane is not covered in detail.

### 13.15.7a.1 `ovn-ic-nbctl ts-add`, the sub-command that creates a Transit Switch

**Concept.** `ts-add` is a sub-command of `ovn-ic-nbctl` that inserts a new row into the `Transit_Switch` table of the IC_NB DB. It is not a daemon, not the OVSDB native CLI; it is a convenience command-line wrapper.

**History and context.** Appeared together with OVN 20.06 when Inter-Connect was merged upstream (the ic-nbctl utility was born at the same time). Before that, operators had to use `ovn-ic-nbctl create Transit_Switch name=...` (verbose). `ts-add` was added to shorten the common pattern.

**Position in the architecture.** Lives in `utilities/ovn-ic-nbctl.c` of the OVN repo and is registered in the `ic_nbctl_commands[]` array. Runs on the operator host (laptop or jump host) and connects to the IC_NB DB cluster through TCP or a Unix socket.

**Role.** Bootstraps the cross-region L2 broadcast domain. It is the starting point of the federation workflow: with no Transit Switch, there is nothing for a region to bind to.

**Why it exists.** The operator needs a single-shot command without typing `create` and remembering the schema. The `add/del/list` pattern matches `ovn-nbctl ls-add/ls-del/ls-list`, so learners can transfer skills from NB to IC_NB easily.

**Problem solved.** Reduces verbosity, avoids schema column-name typos, and automatically returns the UUID of the new row so the caller can pipe it into other commands.

**Importance.** High during setup; low during runtime troubleshooting (rarely called again after setup). Peripheral but on the critical path for first-time deployment.

**Mechanism.** The function `ic_nbctl_ts_add(struct ctl_context *ctx)` in `utilities/ovn-ic-nbctl.c` reads the `SWITCH` argument from `ctx->argv[1]`, checks for collision with `--may-exist`, calls the IDL transaction to insert a new row through `nbrec_ic_nb_transit_switch_insert(ctx->txn)`, sets the `name` column, and commits the transaction.

**How to use it well.** `ovn-ic-nbctl --db=tcp:ic-central:6645 ts-add ts-tenant-blue`. Add `--may-exist` to make scripts idempotent. There is no direct `external_ids` flag; to set them, follow up with `ovn-ic-nbctl set Transit_Switch ts-tenant-blue external_ids:owner=team-blue`.

**Classification.** Database mutation command (RW mode in the command registration `{ "ts-add", 1, 1, "SWITCH", NULL, ic_nbctl_ts_add, NULL, "--may-exist", RW }`).

**Usage workflow.** (1) `ts-add` creates the Transit Switch in IC_NB. (2) `ovn-ic-northd` compiles it into an IC_SB Datapath_Binding. (3) `ovn-ic` in each region syncs it down to the local SB. (4) The operator binds the regional LSP to the Transit through `requested-tnl-key`. (5) Verify with `ts-list` and `list Transit_Switch`.

**When something goes wrong.** If `ts-add` reports "transit switch already exists", check for duplicates with `ts-list`. If the commit times out, the IC_NB cluster may be network-partitioned, so try `--no-leader-only` or check the Raft status.

**Tightly related to.** `ts-list` (verify), `ts-del` (cleanup), the `Transit_Switch` schema (target table), the `ovn-ic-northd` daemon (downstream consumer), and the IC_SB `Datapath_Binding` (compile output).

**Version differences.** OVN 20.06 introduced it. OVN 22.03 stabilised it. OVN 24.03 added the option `other_config:requested-tnl-key` for Transit Switch (allowing tunnel_key pinning instead of auto-allocation). The curriculum baseline 22.03 does not have this option.

**Observation and verification.** After `ts-add`, run `ovn-ic-nbctl list Transit_Switch` to see the row with the new UUID and the correct `name`. Verify the side effect with `ovn-ic-sbctl list Datapath_Binding | grep <name>` (after about 1 second so `ovn-ic-northd` has time to compile).

**Source code.** Function anchor `ic_nbctl_ts_add` in `utilities/ovn-ic-nbctl.c` of OVN branch-22.03. Command registration: the `ic_nbctl_commands[]` array in the same file.

**Real-world incident.** An operator wrote an Ansible script to register 50 tenant Transit Switches at once. `ts-add` has no batch flag, so the sequential loop took 30 seconds. Solution: run `ovn-ic-nbctl -- ts-add ts-1 -- ts-add ts-2 -- ...` (transaction grouping through `--`), which becomes one RPC and one commit.

**Synthetic exercise.** In the 2-region lab (Section 13.15.10), create 3 Transit Switches with `ts-add` and verify each appears in both IC_NB and IC_SB. Measure the time from `ts-add` until the Datapath_Binding shows up in IC_SB.

**Common errors and diagnostic signals.** Forgetting the `--db` flag connects to the default Unix socket `/var/run/ovn/ovn_ic_nb_db.sock`, which exists only on the IC central host. Signal: error "database connection failed" or timeout. Fix: always set the `OVN_IC_NB_DB` environment variable or pass `--db` explicitly.

**Comparison with other systems.** Comparable to `kubectl create namespace` (Kubernetes): both are convenience wrappers on top of CRUD APIs and both support `--may-exist` or `--dry-run` semantics. Differences: `ts-add` does not validate quota and does not enforce admission webhooks.

### 13.15.7a.2 `ovn-ic-nbctl ts-list`, list every Transit Switch

**Concept.** A read-only sub-command that lists every Transit Switch name in the IC_NB DB, one name per line, sorted alphabetically.

**History and context.** Shipped with OVN 20.06 alongside `ts-add` and `ts-del`. The `ls-list` and `lr-list` pattern from `ovn-nbctl` was duplicated to IC_NB for UX consistency.

**Position in the architecture.** `utilities/ovn-ic-nbctl.c`, function `ic_nbctl_ts_list`, command registration in RO mode (read-only), no transaction needed.

**Role.** Inventory query: answers "how many Transit Switches does IC central manage and what are their names?". It is the first verification step after `ts-add` and a smoke test for IC central health.

**Why it exists.** Operators need a quick list without parsing `list Transit_Switch` (verbose output with UUIDs and maps). `ts-list` produces one line per name, easy to pipe into `wc -l`, `grep`, or `sort`.

**Problem solved.** Output formatting: the full `list Transit_Switch` has 5 or more fields per row (UUID, name, external_ids, other_config), unsuitable for counting or grepping. `ts-list` prints only `name`.

**Importance.** Medium. A daily smoke test and the mainstay of monitoring scripts. If `ts-list` is empty when the operator knows `ts-add` ran, the IC_NB connection is broken or the cluster has lost quorum.

**Mechanism.** Function `ic_nbctl_ts_list(ctx)` iterates `nbrec_ic_nb_transit_switch_table_for_each(ts, ctx->idl)`, prints `ts->name` line by line, and exits.

**How to use it well.** `ovn-ic-nbctl ts-list | wc -l` counts Transit Switches. `ovn-ic-nbctl ts-list | sort` stabilises output for diffs. Combine with `for ts in $(ovn-ic-nbctl ts-list); do ovn-ic-nbctl list Transit_Switch $ts; done` for bulk inspection.

**Classification.** Read-only enumeration command, no transaction, no lock.

**Usage workflow.** Smoke test: SSH to IC central, run `ts-list`, verify the count matches the inventory file. In CI/CD: assert `ts-list | wc -l == N` after applying a Terraform IC module.

**When something goes wrong.** `ts-list` is empty just after the operator ran `ts-add`: check the IC_NB Raft leader through `ovsdb-client get-schema`, since the client may be following a stale node. Add `--no-leader-only` to force a read from a follower, or `--leader-only` to ensure the leader.

**Tightly related to.** `ts-add` (input source), `ts-del` (deletion verification), the `Transit_Switch` table (data source).

**Version differences.** Stable from OVN 20.06 onward. OVN 24.03 does not change the semantics. The output format is preserved.

**Observation and verification.** Cross-check with `ovn-ic-sbctl list Datapath_Binding | grep external_ids:interconn-ts | wc -l`. If the count differs from `ts-list`, `ovn-ic-northd` has not finished compiling or has hit an error.

**Source code.** Function `ic_nbctl_ts_list` in `utilities/ovn-ic-nbctl.c` branch-22.03; command registration `{ "ts-list", 0, 0, "", NULL, ic_nbctl_ts_list, NULL, "", RO }`.

**Real-world incident.** A Nagios monitoring script alerts when the `ts-list` count drops by more than 10 percent. One time the alert was a false positive: the operator had just run `ts-del` to clean up an off-boarded tenant, and the script did not filter operator-driven changes. Fix: add log correlation with change request IDs.

**Synthetic exercise.** In the 2-region lab: run `ts-list`, then `ts-add`, then `ts-list` again. Observe the output diff. Measure the round-trip time between `ts-add` and `ts-list` showing the new entry (this reveals IC_NB write latency).

**Common errors and diagnostic signals.** Empty output does not mean "no Transit Switches"; it may be because `--db` points to the wrong endpoint (connecting to a different cluster, or a stale local `/var/run/...` socket). Signal: empty output plus exit 0. Fix: pass `--db` explicitly and double-check the IC_NB cluster identity through `external_ids` on another row if available.

**Comparison with other systems.** Comparable to `kubectl get namespaces -o name`: prints only names, no metadata. Differences: `ts-list` has no `-o yaml`, `-o json`, or `-l label-selector`; for filtering, use `find Transit_Switch`.

### 13.15.7a.3 `ovn-ic-sbctl`, the CLI for the IC_SB DB

**Concept.** `ovn-ic-sbctl` is a utility CLI that interacts with the central IC_SB DB cluster. Similar to `ovn-sbctl` for the local SB DB, but at the IC level. The vast majority of use cases are read-only inspection.

**History and context.** Shipped with `ovn-ic-nbctl` in OVN 20.06 when Inter-Connect was merged. IC_SB is the output of the `ovn-ic-northd` compile and the input of the per-region `ovn-ic`; engineers need a CLI to debug the sync flow.

**Position in the architecture.** `utilities/ovn-ic-sbctl.c` in the OVN repo. Connects to the IC_SB DB cluster through the default endpoint `unix:/var/run/ovn/ovn_ic_sb_db.sock` or `--db=tcp:<ic-central>:6646`.

**Role.** The operator's window into "has the Transit Switch been compiled?", "which AZs have registered?", and "which routes have been advertised?". It is the primary debugging tool when cross-region traffic does not flow.

**Why it exists.** IC_SB is the internal state machine between `ovn-ic-northd` and `ovn-ic`; operators need a "torch" to verify state. Without a CLI, operators would have to read OVSDB directly through `ovsdb-client` (raw, poor UX).

**Problem solved.** Provides a friendly CLI for the seven important IC_SB tables: `Availability_Zone`, `Gateway`, `Encap`, `Datapath_Binding`, `Port_Binding`, `Route`, `Connection`. A `show` command provides a hierarchical view.

**Importance.** High for forensics and troubleshooting. Every cross-region incident usually starts with `ovn-ic-sbctl show` to confirm IC central state.

**Mechanism.** Uses the same `ovsdb-idl` plus `ctl-utils` framework as `ovn-sbctl` and `ovn-nbctl`. The `ic_sbctl_commands[]` table registers `show`, `init`, `get/set/del-connection`, and `get/set/del-ssl`. Sub-commands `list`, `find`, `get`, `set`, and similar come from the generic `ovsdb-idl` command set.

**How to use it well.** `ovn-ic-sbctl show` for an overview. `list Availability_Zone` for the registered AZs. `list Datapath_Binding` for the transit datapath. `list Port_Binding` for cross-region ports. `list Route` for advertised routes. `find Port_Binding logical_port=<name>` filters by column.

**Classification.** CLI utility wrapper, mostly read-only, with write capability for Connection and SSL configuration (rare use).

**Usage workflow.** Inspect-first workflow: `show`, then `list <table>` for a table with an anomaly, then `find <table> <column>=<value>` to zoom into a specific row, then `get <table> <row> <column>` for an atomic field.

**When something goes wrong.** Cross-region ping fails: `show` to view the AZ and Gateway counts, then `list Datapath_Binding` to verify the Transit Switch has a Datapath, then `list Port_Binding` to verify the cross-region LSP has synced.

**Tightly related to.** The IC_SB schema (target), `ovn-ic-northd` (writer), `ovn-ic` (reader and writer for chassis claim), `ovn-ic-nbctl` (sister CLI for IC_NB).

**Version differences.** OVN 22.03 and OVN 24.03 keep the core command set. OVN 24.03 added new fields in `Port_Binding` (for example, `requested_chassis` analogue for transit), so `list Port_Binding` has additional columns. Existing scripts are not broken.

**Observation and verification.** `ovn-ic-sbctl show` produces hierarchical text. For JSON-friendly output: `ovn-ic-sbctl --format=json --pretty list Availability_Zone`. Combine with `jq` for a monitoring pipeline.

**Source code.** `utilities/ovn-ic-sbctl.c` branch-22.03, with the main function `ic_sbctl_show` for the hierarchical view and `cmd_get_connection` and `cmd_set_connection` for configuration.

**Real-world incident.** Production incident: traffic from Region A to Region B dropped after expanding the IC_SB cluster from 3 to 5 nodes. Root cause: the two new nodes were missing the schema upgrade. Discovered through `ovn-ic-sbctl show` reporting "schema version mismatch", followed by a DB schema upgrade and recovery.

**Synthetic exercise.** In the lab: run `ovn-ic-sbctl show` with zero regions, after adding region 1, and after adding region 2. Observe the output growth.

**Common errors and diagnostic signals.** Connecting to the wrong IC_SB endpoint (port 6645 for IC_NB confused with 6646 for IC_SB) triggers the error "no such schema OVN_IC_Southbound". Fix: remember the port assignment and pass `--db` explicitly.

**Comparison with other systems.** Comparable to `etcdctl` for Kubernetes: both are CLIs for the central control-plane DB. Differences: `ovn-ic-sbctl` has a hierarchical `show`, while etcdctl only exposes raw key-value pairs.

### 13.15.7a.4 `Availability_Zone`, the IC_SB table that declares regions

**Concept.** `Availability_Zone` is an OVSDB table in IC_SB (NOT IC_NB) that lists the regions (each is an AZ) registered with IC central. One row per region. The single schema column is `name` (string).

**History and context.** One of the seven original tables in the IC_SB schema when Inter-Connect was merged in OVN 20.06. The name "Availability Zone" borrows terminology from AWS and OpenStack, where each AZ is an independent failure domain.

**Position in the architecture.** Schema file `ovn-ic-sb.ovsschema`. Populated by the per-region `ovn-ic` daemon (not by the operator). Each region registers its AZ name when `ovn-ic` starts.

**Role.** The "directory of regions" of IC central: it tracks which regions participate in the federation. Every other IC_SB entity (`Gateway`, `Port_Binding`, `Route`) has a foreign key referencing `Availability_Zone` to indicate which region the row belongs to.

**Why it exists.** A namespace is needed to distinguish cross-region resources. Without an AZ, you cannot tell which region Gateway G1 belongs to and therefore cannot know which tunnel endpoint to encapsulate to.

**Problem solved.** Multi-tenancy at the region level: many regions share a common IC central but are logically separated. `Availability_Zone.name` is the identifier key.

**Importance.** Structurally high: with no row in `Availability_Zone`, the region cannot participate in IC. Operators rarely interact with it directly; the daemon manages it.

**Mechanism.** Function `az_run()` in `ic/ovn-ic.c` reads the AZ name from the local NB DB (`NB_Global.name`) and inserts or updates the IC_SB `Availability_Zone` through a transaction. If the name matches, it reuses the existing row. If the local NB renames, it updates the IC_SB row.

**How to use it well.** Operator inspection: `ovn-ic-sbctl list Availability_Zone` to view the region list. Operators do NOT add or delete directly; instead they set `ovn-nbctl set NB_Global . name=az-region-X` in each region and let `ovn-ic` sync.

**Classification.** OVSDB table, declarative, daemon-managed (not user-managed).

**Usage workflow.** Region setup: (1) in each region, set `NB_Global.name=az-<region-id>`; (2) start the `ovn-ic` daemon; (3) verify on IC central with `ovn-ic-sbctl list Availability_Zone` that the new row appears.

**When something goes wrong.** Cross-region traffic does not flow: first check whether `Availability_Zone` has enough rows (one per region). If a row is missing, that region has not registered with IC central, so check the `ovn-ic` daemon log and the IC central connectivity.

**Tightly related to.** `Gateway.availability_zone` (FK), `Port_Binding.availability_zone` (FK), `Route.availability_zone` (FK), the `ovn-ic` daemon (writer), and `NB_Global.name` (source of truth).

**Version differences.** OVN 20.06 schema has only the `name` column. OVN 24.03 adds the `nb_cfg` column (a Raft sync counter); verify that `nb_cfg` increments smoothly when the local NB changes. The curriculum baseline 22.03 does not have `nb_cfg` in its schema (close to correct; verify in practice through `ovs-vsctl get` against the runtime schema).

**Observation and verification.** `ovn-ic-sbctl --format=table list Availability_Zone`. If the count is less than the expected number of regions, some region has not registered. Combine with `find Gateway availability_zone=<uuid>` to see which AZ has an active gateway.

**Source code.** Schema definition in `ovn-ic-sb.ovsschema` branch-22.03. Daemon code: `az_run()` in `ic/ovn-ic.c`.

**Real-world incident.** A region renamed abruptly (operator typo `az-prod` to `az-prd`) caused IC central to create a new row `az-prd` while the old `az-prod` row was orphaned with the previous Gateway and Port_Binding entries. Cross-region traffic lost its reference. Fix: rename back to the old name, or clean up the old row with `destroy Availability_Zone az-prod`.

**Synthetic exercise.** In the lab: change `NB_Global.name` in region A from `az-region-a` to `az-region-a-renamed`. Observe IC_SB `Availability_Zone`: does a new row appear or is the old row renamed? (Hint: a new row appears, since the name is conceptually the primary key.)

**Common errors and diagnostic signals.** Duplicate AZ names between two regions (operator copy-paste) cause both regions to update the same row, so chassis claims overlap and tunnel routing becomes chaotic. Signal: `Gateway.availability_zone` shares a UUID but `Gateway.hostname` differs. Fix: rename the AZ in one region immediately.

**Comparison with other systems.** Comparable to the AWS EC2 "AvailabilityZone" concept: both are failure-domain isolation. Difference: an AWS AZ is geographical, while an OVN AZ is a logical OVN cluster boundary; multiple clusters in the same DC can be two AZs.

### 13.15.7a.5 `Transit_Switch`, the IC_NB table that declares cross-region L2

**Concept.** `Transit_Switch` is an OVSDB table in IC_NB that holds the definitions of Transit Switches (cross-region logical switches). The schema in OVN 22.03 has 3 columns: `name`, `other_config`, and `external_ids`. One row per Transit Switch.

**History and context.** An original table in the IC_NB schema since OVN 20.06. "Transit" borrows terminology from MPLS L2VPN (a transit link connects sites).

**Position in the architecture.** Schema file `ovn-ic-nb.ovsschema` branch-22.03. Operator-managed (through `ovn-ic-nbctl ts-add` and `ts-del`). It has no outgoing FK; other tables (for example, IC_SB `Datapath_Binding.transit_switch`) reference the NAME (string) rather than the UUID, which matters for cross-DB linking.

**Role.** Declares an "L2 broadcast domain extended across regions". Input to the `ovn-ic-northd` compile to IC_SB `Datapath_Binding`. The anchor that region-local NBs reference when creating an LSP of type router with `requested-tnl-key`.

**Why it exists.** A central entity is needed so all regions agree that "this is the same L2 segment". If each region created its own logical switch and assigned a tunnel_key manually, collisions would be unavoidable.

**Problem solved.** A single source of truth for cross-region L2: name plus tunnel_key allocation. `ovn-ic-northd` avoids cross-region tunnel_key collisions.

**Importance.** Structurally core for every cross-region L2 use case. Without `Transit_Switch`, you cannot federate L2.

**Mechanism.** The operator inserts a row through `ts-add`. `ovn-ic-northd` reads the row, allocates a tunnel_key (range 16M to 24M), and creates an IC_SB `Datapath_Binding` with `transit_switch=<name>` and `tunnel_key=<allocated>`. The `ovn-ic` in each region reads the IC_SB Datapath_Binding and syncs it to the local SB.

**How to use it well.** Create: `ovn-ic-nbctl ts-add ts-tenant-blue`. Annotate: `ovn-ic-nbctl set Transit_Switch ts-tenant-blue external_ids:owner=team-blue external_ids:purpose=cross-dc-tenant`. Inspect: `ovn-ic-nbctl list Transit_Switch`.

**Classification.** Operator-managed entity, declarative, downstream consumed by the `ovn-ic-northd` daemon.

**Usage workflow.** (1) `ts-add <name>` in IC_NB. (2) Annotate `external_ids` for audit. (3) In each region, create the LRP that connects to the Transit through a peer port and set `options:requested-tnl-key=<allocated>`. (4) Verify the chain through `ovn-ic-sbctl list Datapath_Binding`. (5) Test the cross-region ping.

**When something goes wrong.** Transit Switch is created but cross-region traffic is not flowing: check whether IC_SB `Datapath_Binding` has the corresponding row. If not, `ovn-ic-northd` has not compiled (check the daemon log). If yes but the tunnel_key conflicts with a local tenant, the operator hardcoded a duplicate `requested-tnl-key`.

**Tightly related to.** `ovn-ic-nbctl ts-add/del/list` (CLI), IC_SB `Datapath_Binding` (compiled output), `ovn-ic-northd` (compiler), per-region `Logical_Router_Port.options:requested-tnl-key` (consumer), and the Geneve tunnel (data plane).

**Version differences.** OVN 22.03: 3 columns (name, other_config, external_ids). OVN 24.03: adds `other_config:requested-tnl-key` to pin the tunnel_key instead of auto-allocating (helpful for migration without changing the tunnel_key). The curriculum baseline 22.03 does not have the tunnel_key pinning feature.

**Observation and verification.** `ovn-ic-nbctl list Transit_Switch` returns the IC_NB row. Cross-check with `ovn-ic-sbctl find Datapath_Binding external_ids:interconn-ts=<name>` for the IC_SB row. Cross-check with `ovn-sbctl --db=tcp:<region>:6642 list Datapath_Binding | grep <name>` for the local SB row after `ovn-ic` syncs.

**Source code.** Schema definition `ovn-ic-nb.ovsschema` branch-22.03. Compile logic: a function in `ic/ovn-ic.c` (function `ts_run()` or its 22.03 equivalent in `ic/ovn-ic-northd.c`, version-specific, verify the exact function).

**Real-world incident.** An operator deleted `ts-tenant-blue` through `ts-del` without first draining tenant LRPs, leaving region LSPs orphaned with a requested-tnl-key referencing a non-existent Transit Switch. The ovn-controller in each region kept logging the warning "transit datapath not found". Fix: re-create the Transit Switch (same name) or clean up the regional LRPs first.

**Synthetic exercise.** In the lab: create Transit Switch `ts-test` and set `external_ids:purpose=lab-test`. Observe the corresponding Datapath_Binding in IC_SB after about 1 second. Measure the allocated tunnel_key. Then run `ts-del` and observe the Datapath_Binding disappear.

**Common errors and diagnostic signals.** Transit Switch names containing special characters (for example, spaces or `/`) cause schema validation rejection or compile errors. Signal: `ts-add` exits non-zero with the message "invalid name". Fix: use only `[a-zA-Z0-9_-]` in the name.

**Comparison with other systems.** Comparable to AWS VPC "Transit Gateway": both are central resources that connect multiple VPC or region L2 equivalents. Difference: AWS Transit Gateway is a managed service with built-in metering and ACLs; an OVN Transit Switch is a plain L2 segment with no ACLs at the Transit level (ACLs apply on the local LR).

## 13.15.8 Anatomy 1: `ovn-ic-nbctl ts-add` and verify

**Bucket:** OVN | **Context:** building a new Transit Switch for cross-region L2 | **Purpose:** connect a tenant network across two regions | **Activity:** provisioning | **Mechanism:** insert a row into the IC_NB Transit_Switch table.

**Command and decode:**

```
$ ovn-ic-nbctl --db=tcp:ic-central:6645 ts-add ts-tenant-blue
$ ovn-ic-nbctl --db=tcp:ic-central:6645 ts-list
ts-tenant-blue
$ ovn-ic-nbctl --db=tcp:ic-central:6645 list Transit_Switch
_uuid               : abc12345-def6-7890-abcd-ef1234567890
external_ids        : {}
name                : "ts-tenant-blue"
other_config        : {}
```

| Attribute | Healthy / Warning / Critical |
|-----------|------------------------------|
| `_uuid` | UUID stable across IC restarts |
| `name` | Matches the naming convention (ts-<tenant>-<purpose>) |
| `external_ids` | Empty by default; may be annotated for audit (`region_owner=...`) |
| `other_config` | Empty by default; may set an encap type override |

**Verify the sync down to the region:**

```
$ ovn-sbctl --db=tcp:region-a-sb:6642 list Datapath_Binding | grep ts-tenant-blue
_uuid               : ... (different UUID, region-local)
external_ids        : {interconn-ts="ts-tenant-blue"}
tunnel_key          : 16711680  # allocated by ovn-ic-northd
```

`external_ids:interconn-ts` set to "ts-tenant-blue" indicates this Datapath_Binding was synced from IC, not created from the local NB. The tunnel_key is allocated globally unique inside the IC namespace.

## 13.15.9 Anatomy 2: `ovn-ic-sbctl list AvailabilityZone`

**Bucket:** OVN | **Context:** verifying that a region has registered with IC central | **Purpose:** check whether each region has synced its chassis claim | **Activity:** monitoring | **Mechanism:** query the IC_SB AvailabilityZone table.

**Sample output:**

```
$ ovn-ic-sbctl --db=tcp:ic-central:6646 list AvailabilityZone
_uuid               : 11111111-...
name                : "az-region-a"
nb_cfg              : 42
gateways            : [22222222-..., 33333333-...]

_uuid               : 44444444-...
name                : "az-region-b"
nb_cfg              : 38
gateways            : [55555555-..., 66666666-...]
```

| Attribute | Healthy / Warning / Critical |
|-----------|------------------------------|
| `name` | Globally unique AZ name, typically "az-<region>" |
| `nb_cfg` | Sync counter from the regional NB. Healthy: increments steadily when configuration changes. Critical: stuck means the regional ovn-ic is hung |
| `gateways` | List of Chassis UUIDs that act as transit gateways. Healthy: at least one gateway active per AZ |

**Critical signals:**
- `gateways = []` empty: the region has no chassis registered as a transit gateway, so cross-region traffic cannot enter or leave. Verify `external_ids:ovn-is-interconn=true` on at least one chassis.
- `nb_cfg` does not increment for a long time: ovn-ic is not syncing back to central. Check the ovn-ic daemon log.

## 13.15.10 Guided exercise (synthetic 2-region lab)

**Objective:** build a 2-region OVN-IC on a single lab machine using nested VMs or Linux network namespaces and verify the cross-region L2 ping.

**Setup overview:**

```
+----------+              +----------+              +----------+
| Region A |              | IC central|              | Region B |
| 10.0.1.x |              | 10.0.0.x  |              | 10.0.2.x |
|          |              |           |              |          |
| NB+SB    |<-- ovn-ic -->| IC_NB+SB  |<-- ovn-ic -->| NB+SB    |
| cluster  |              | cluster   |              | cluster  |
|          |              | + ovn-ic- |              |          |
|          |              | northd    |              |          |
| chassis  |              |           |              | chassis  |
| (gw=true)|              |           |              | (gw=true)|
+----------+              +----------+              +----------+
        \                                                  /
         \                                                /
          \---- Geneve transit tunnel ---------/
```

**Step 1: bootstrap 3 OVSDB clusters (IC central plus 2 regions).**

```bash
# IC central
ovsdb-tool create /etc/ovn/ovn_ic_nb_db.db \
  /usr/share/ovn/ovn-ic-nb.ovsschema
ovsdb-tool create /etc/ovn/ovn_ic_sb_db.db \
  /usr/share/ovn/ovn-ic-sb.ovsschema

ovsdb-server /etc/ovn/ovn_ic_nb_db.db \
  --remote=ptcp:6645:10.0.0.10 --pidfile=/tmp/ic-nb.pid --detach

ovsdb-server /etc/ovn/ovn_ic_sb_db.db \
  --remote=ptcp:6646:10.0.0.10 --pidfile=/tmp/ic-sb.pid --detach

# Region A NB and SB (already in place from the OVN install, or freshly bootstrapped)
# ... (similar for region B)
```

**Step 2: start `ovn-ic-northd` at the centre.**

```bash
ovn-ic-northd \
  --ic-nb-db=tcp:10.0.0.10:6645 \
  --ic-sb-db=tcp:10.0.0.10:6646 \
  --pidfile --detach
```

**Step 3: start `ovn-ic` in each region.**

```bash
# Region A
ovn-ic \
  --ovnnb-db=tcp:10.0.1.10:6641 \
  --ovnsb-db=tcp:10.0.1.10:6642 \
  --ic-nb-db=tcp:10.0.0.10:6645 \
  --ic-sb-db=tcp:10.0.0.10:6646 \
  --pidfile --detach

# Region B (similar, replace 10.0.1.x with 10.0.2.x)
```

**Step 4: configure the AZ name and the transit gateway chassis.**

```bash
# Region A
ovn-nbctl set NB_Global . name=az-region-a
ovs-vsctl set Open_vSwitch . external_ids:ovn-is-interconn=true

# Region B (similar)
ovn-nbctl set NB_Global . name=az-region-b
ovs-vsctl set Open_vSwitch . external_ids:ovn-is-interconn=true
```

**Step 5: create the Transit Switch and bind both regions.**

```bash
# IC central
ovn-ic-nbctl --db=tcp:10.0.0.10:6645 ts-add ts-tenant-blue
ovn-ic-nbctl --db=tcp:10.0.0.10:6645 ts-list   # verify

# Region A: create the logical switch and the LSP that connects to Transit
ovn-nbctl ls-add ls-blue-rA
ovn-nbctl lsp-add ls-blue-rA vm-rA -- \
  lsp-set-addresses vm-rA "00:00:00:01:01:01 192.168.100.1"

# Region A: create the logical router, attach ls-blue-rA, and attach Transit
ovn-nbctl lr-add lr-blue-rA
ovn-nbctl lrp-add lr-blue-rA lrp-rA-to-ls 00:00:00:00:01:01 192.168.100.254/24
ovn-nbctl lsp-add ls-blue-rA ls-rA-to-lr -- \
  lsp-set-type ls-rA-to-lr router -- \
  lsp-set-addresses ls-rA-to-lr router -- \
  lsp-set-options ls-rA-to-lr router-port=lrp-rA-to-ls

# Bind lr-blue-rA to the Transit Switch (through a peer port)
ovn-nbctl lrp-add lr-blue-rA lrp-rA-to-ts 00:00:00:00:01:02 10.0.0.1/30
ovn-nbctl set Logical_Router_Port lrp-rA-to-ts \
  options:requested-tnl-key=<allocated by ic-northd>
```

**Step 6: do the same for Region B.**

**Step 7: verify the cross-region ping.**

```bash
# On a VM in region A (192.168.100.1) ping a VM in region B (192.168.200.1)
ip netns exec vm-rA ping -c 4 192.168.200.1
# Expected: success (passes through the ts-tenant-blue Transit Switch)
```

**Verify the path:**

```bash
# Trace from region A
ovn-trace --ovnsb=tcp:10.0.1.10:6642 lr-blue-rA \
  'inport=="lrp-rA-to-ls" && ip4.src==192.168.100.1 && ip4.dst==192.168.200.1'
```

The output shows the packet going through LR_IN routing, then out to the Transit Switch, then through the cross-region tunnel.

## 13.15.11 Capstone POE: design 3 regions (HQ plus 2 branches)

**Problem:** design an OVN-IC topology for an enterprise:
- HQ: 100 chassis, 5000 VMs, in the main DC.
- Branch-1: 10 chassis, 200 VMs, in a secondary DC (50 ms latency to HQ).
- Branch-2: 5 chassis, 100 VMs, in an edge office (200 ms latency to HQ).

Requirements:
- HQ VMs must talk to Branch VMs (cross-region L3).
- Branch-1 VMs do not need to talk directly to Branch-2 VMs (only through HQ).
- Failure isolation: if HQ goes down, branches still handle their own local tenants.

**Design steps:**

1. **AZ split:** three AZs (az-hq, az-branch-1, az-branch-2). Each AZ has its own independent 3-node SB DB Raft cluster.

2. **IC central:** one 3-node IC_NB plus IC_SB Raft cluster, placed at HQ (low latency to HQ, acceptable higher latency to branches).

3. **Transit Switch design:**
   - `ts-hq-branch1`: connects HQ and Branch-1.
   - `ts-hq-branch2`: connects HQ and Branch-2.
   - Do NOT create `ts-branch1-branch2` (per requirement, branch-to-branch only via HQ).

4. **Transit Router:** one Transit Router routes cross-region L3 traffic. Bind every Transit Switch and advertise routes from each AZ subnet.

5. **Failure modes:**
   - HQ down: Branch-1 and Branch-2 lose cross-region connectivity (matches the requirement of routing only through HQ). Local branch tenants continue to work.
   - Branch down: HQ still works and the remaining branch still talks to HQ.
   - IC central down: new configuration does not sync, BUT installed tunnel data plane keeps running, so degradation is graceful.

6. **Recovery procedure:**
   - HQ down recovery: restart the HQ regional SB cluster, ovn-ic resyncs, branches reconnect automatically.
   - IC central down recovery: rebuild the IC cluster from backup (`ovsdb-tool` create-cluster plus restore from an hourly backup snapshot).

**Predict and observe:** predict that the cross-region first-packet latency is about three times the physical latency (Geneve handshake plus cross-region ARP). Verify by comparing the first `ping -c 1` to subsequent ones.

**Why the POE matters:** it helps the engineer master flow design and the failure model before deploying to production. This is a true "architect" skill, not just operations.

## 13.15.12 Anti-patterns and notes

**Anti-pattern 1: stretching a single SB cluster across regions instead of using IC.**

A team sometimes wants "simpler" by stretching a single 3-node SB cluster across 3 regions, one node per region. Problems:
- Cross-region Raft latency is at least the maximum pairwise round-trip. A 100 ms round-trip means 100 ms commit latency for every `ovn-nbctl` write.
- A network partition between regions causes quorum loss, and the whole cluster is configuration-frozen.
- Compliance: data leak risk.

**Avoidance:** if a deployment has at least 2 regions, default to OVN-IC. Stretching is acceptable only for two DCs less than 5 km apart (low latency and the same regulatory domain).

**Anti-pattern 2: hardcoding `requested-tnl-key` instead of letting IC allocate.**

`ovn-ic-northd` allocates tunnel_keys for Transit Switches automatically (range 16M to 24M). If an operator hardcodes `requested-tnl-key=42` (a collision with a local tenant), ovn-controller hits a tunnel_key conflict and packets are dropped.

**Avoidance:** never set `requested-tnl-key` on Transit-related entities. Let IC allocate.

**Note 3: back up the IC central cluster off-site.**

IC_NB plus IC_SB is the single point of configuration that every region depends on. Back up at least hourly and off-site (a different region or cloud storage). If IC central is destroyed, rebuilding from backup takes at least 30 minutes (acceptable with hourly backups), versus days when rebuilding from scratch.

**Note 4: BGP integration is banned.**

Per the CLAUDE.md North Star, the curriculum does NOT cover deep BGP-based dynamic route advertisement for IC (`ovn-bgp-agent`). If a real deployment needs BGP, consult the upstream `ovn-bgp-agent` documentation independently. The curriculum covers only **static routes** through `ovn-ic-nbctl add Route`.

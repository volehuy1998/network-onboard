## 10.3.1 The four ACID properties of an OVSDB transaction

OVSDB (RFC 7047) is a transactional database, not a simple key-value store. Each time a client sends one or more operations to `ovsdb-server`, the whole group is processed as **one atomic transaction**. `ovsdb-server` implements all four ACID properties, even though the database has a narrower scope than PostgreSQL or MySQL.

**Atomicity, all-or-nothing.** A transaction that contains N operations (insert, update, mutate, delete) either runs all N operations or none of them. If operation k fails (schema constraint violation, `assert` mismatch), the entire transaction is rolled back: the N-1 operations already applied in memory are undone. No "half-done" state is ever visible.

Example: a transaction that creates a bridge, a port, and an interface that link together needs atomicity:

```
[host]# ovs-vsctl add-br br0 -- add-port br0 eth0 -- set Interface eth0 type=internal
```

These three operations execute in a single transaction. If setting the interface type fails (invalid type), both the `add-br` and `add-port` are rolled back; no orphan bridge is left behind.

**Consistency, schema constraints always hold.** After a transaction commits, the database state fully respects the schema constraints:

- **Type constraints.** An `integer` column does not contain a string.
- **Referential integrity.** A UUID reference does not point to a deleted row. `ovsdb-server` automatically clears dangling references after the target row is deleted (optional cascade delete).
- **Set and map size bounds.** A `set<int>` with declared min and max element counts is enforced.
- **Unique constraints.** A column with `unique=true` does not allow duplicate values.

If any operation in the transaction violates a schema constraint, the transaction is rejected with a `constraint violation` error and the whole transaction is rolled back.

**Isolation, snapshot-based.** `ovsdb-server` uses a database-level lock. While one transaction runs, others are blocked until it finishes. This is **serializable isolation**, the strongest level in database systems, and admits no concurrent-update race conditions. The trade-off is that throughput is lower than under MVCC (PostgreSQL, MongoDB), but this fits the OVSDB workload (low write rate, high read rate, correctness over speed).

**Durability, fsync after commit.** When `ovsdb-server` returns "commit successful" to the client, the data has already been `fsync()`-ed to disk (the `conf.db` file). Even if the process crashes or the server reboots immediately afterwards, the data is not lost. The cluster mode (Raft) adds cluster-level durability: a commit is considered successful only after a majority of the cluster has persisted the log entry.

> **Implementation note.** The traditional `ovsdb-server` uses an **append-only database file** (a JSON log): each transaction is one JSON line appended to the file. Reading the database means replaying every line from the start. The file grows over time, hence the periodic `ovsdb-tool compact` (see Part 10.2). This is a trade-off: append-only is simple and durable but requires maintenance. Redis or LMDB use a different model (copy-on-write B-tree) that does not need compaction but is more complex.

## 10.3.2 Transaction structure in the RFC 7047 protocol

An OVSDB transaction sent over JSON-RPC 1.0 has the structure:

```json
{
  "method": "transact",
  "params": [
    "<database_name>",
    { "op": "insert", "table": "...", "row": {...}, "uuid-name": "..." },
    { "op": "update", "table": "...", "where": [...], "row": {...} },
    { "op": "mutate", "table": "...", "where": [...], "mutations": [...] },
    { "op": "wait", "table": "...", "where": [...], "until": "==", "columns": [...] },
    { "op": "assert", "lock": "..." },
    { "op": "comment", "comment": "description" }
  ],
  "id": 42
}
```

Each element of `params` (other than the first, which is the database name) is an operation. `ovsdb-server` processes them sequentially from top to bottom, applying changes to a temporary in-memory state. After every operation succeeds, the server commits all changes to disk (or to the Raft log in cluster mode) and returns the response:

```json
{
  "id": 42,
  "error": null,
  "result": [
    { "uuid": ["uuid", "..."], ... },
    { "count": 1 },
    ...
  ]
}
```

If operation k fails, `result[k]` contains the error object and operations k+1 to N are not run. `result[0..k-1]` were applied temporarily but are rolled back before the response is sent to the client.

**`ovs-vsctl` translates to a transaction.** Each `ovs-vsctl` command can be combined into a single transaction using the `--` separator:

```
[host]# ovs-vsctl --id=@bridge create Bridge name=br0 \
         -- --id=@port create Port name=eth0 \
         -- --id=@iface create Interface name=eth0 type=internal \
         -- set Bridge br0 ports=@port \
         -- set Port eth0 interfaces=@iface \
         -- add Open_vSwitch . bridges @bridge
```

The variables `@bridge`, `@port`, `@iface` are UUID placeholders resolved within the transaction. Result: six operations run atomically.

## 10.3.3 Prerequisite check, `wait`, `assert`, `nb_cfg`

A practical problem: client A reads the database at time T1, computes its changes, and sends a transaction at time T2. In the interval [T1, T2], client B has changed the database, so A's transaction is based on stale state and may apply incorrectly (for example, duplicate insert, race condition).

OVSDB provides three mechanisms to prevent this race:

**(1) The `wait` operation.** The transaction commits only if a condition still holds at the moment the server processes it:

```json
{ "op": "wait", "table": "Bridge", "timeout": 5000,
  "where": [["name", "==", "br0"]],
  "until": "==",
  "columns": ["datapath_id"],
  "rows": [{"datapath_id": ["uuid", "expected_value"]}] }
```

If the condition does not match within 5000 ms, the transaction fails with a `timed out` error. Used to ensure "bridge `br0` must already have `datapath_id = X` before I run the next step". Common when a client must synchronise with an `ovsdb-server` update (for example, a plugin waiting for the kernel datapath to finish attaching).

**(2) The `assert` operation.** Verifies that the lock is still held by the current client:

```json
{ "op": "assert", "lock": "ovn_northd" }
```

If the client no longer holds the `ovn_northd` lock (because of a network drop, crash, or reconnect), the transaction fails. This is how `ovn-northd` ensures that only the active instance writes to the SB DB (see Part 13.8).

**(3) The `nb_cfg` prerequisite (OVN-specific).** OVN Northbound has the counter `NB_Global.nb_cfg` that increments on every change. The client reads the value, then sends a transaction with a prerequisite:

```json
{ "op": "select", "table": "NB_Global", "columns": ["nb_cfg"],
  "where": [] },
/* ... compute changes ... */
{ "op": "wait", "table": "NB_Global",
  "where": [], "until": "==",
  "columns": ["nb_cfg"], "rows": [{"nb_cfg": 42}] },
/* ... apply changes ... */
```

If during that interval `nb_cfg` was already increased by another client (to 43), the wait fails, the transaction does not commit, and the client must retry with the new state. This mechanism is similar to **optimistic concurrency control** in a SQL database.

## 10.3.4 Conflict handling for `mutate`

The **mutate** operation is an in-place way to change a column without rewriting its full value, especially useful for `set<UUID>` columns (add or remove an element):

```json
{ "op": "mutate", "table": "Bridge",
  "where": [["name", "==", "br0"]],
  "mutations": [
    ["ports", "insert", ["set", [["uuid", "port_uuid_1"]]]],
    ["ports", "insert", ["set", [["uuid", "port_uuid_2"]]]]
  ] }
```

Unlike `update` (which replaces the entire value), `mutate` **adds to the existing set** without needing the previous value. This reduces the chance of a race:

**Race example with `update`:**

- T1: Client A reads `br0.ports = [port_a]`.
- T2: Client B reads `br0.ports = [port_a]`.
- T3: Client A transacts: `update br0.ports = [port_a, port_new_a]`.
- T4: Client B transacts: `update br0.ports = [port_a, port_new_b]`.
- Result: A's update is overwritten by B: `port_new_a` is lost.

**With `mutate`:**

- T1: Client A transacts: `mutate br0.ports insert port_new_a`. Server applies: ports = [port_a, port_new_a].
- T2: Client B transacts: `mutate br0.ports insert port_new_b`. Server applies: ports = [port_a, port_new_a, port_new_b].
- Result: both `port_new_a` and `port_new_b` are present in the final state.

This is why **`ovs-vsctl` always uses `mutate`** for `set<UUID>` columns and never `update`.

**Mutate operators:**

- `insert`: add an element to the set.
- `delete`: remove an element from the set.
- `+=`, `-=`, `*=`, `/=`, `%=`: arithmetic on integer columns.

A single `mutate` can contain several mutations that run sequentially in the same operation (atomically).

## 10.3.5 Transaction failure and the client retry mechanism

A transaction can fail for many reasons: schema violation, `wait` timeout, lost lock (`assert` fail), connection drop. `ovsdb-server` returns a specific error code for each case:

| Error code | Meaning | Client action |
|---------|---------|---------------------|
| `constraint violation` | Schema constraint not satisfied | Fix input and retry |
| `referential integrity violation` | UUID reference points to a non-existent row | Retry with a valid UUID |
| `timed out` | Wait operation timed out | Retry with the new state |
| `not owner` | Assert lock failed; the client is no longer the leader | Abort and hand off to the new leader |
| `duplicate uuid-name` | Duplicate `uuid-name` within the same transaction | Fix the `uuid-name` generation logic |
| `syntax error` | Malformed JSON-RPC | Fix the client bug |

**Standard retry pattern for an IDL client:**

```python
while True:
    txn = idl.transaction(OpenVSwitch)
    # ... build transaction operations ...
    status = txn.commit_block()
    if status == TransactionStatus.SUCCESS:
        break
    elif status == TransactionStatus.TRY_AGAIN:
        # state was stale; the IDL automatically refetches and retries
        continue
    else:
        raise Exception(f"Transaction failed: {txn.get_error()}")
```

The IDL (Interface Definition Language) client library (`python-ovs`, `ovs-cpp`) handles `TRY_AGAIN` automatically. When the server signals "your state is stale", the IDL fetches a new snapshot and retries the transaction. This is the abstraction for optimistic concurrency control.

## 10.3.6 Limitations compared with a SQL database transaction

The OVSDB transaction model has several clear limitations compared with a full SQL database:

**No nested transactions.** OVSDB does not support `BEGIN ... BEGIN ... COMMIT ... COMMIT` like PostgreSQL. A transaction is "flat"; you cannot create a sub-transaction independent of its parent. The reason is design simplicity: `ovsdb-server` does not have to manage nested state.

**No savepoint.** SQL allows `SAVEPOINT s1; ... ROLLBACK TO s1` to roll back part of a transaction. OVSDB has only `abort` for the whole. If a partial rollback is needed, the client must catch the error and resend the transaction with different operations.

**No configurable isolation level.** OVSDB is always serializable. SQL lets you choose READ COMMITTED, REPEATABLE READ, or SERIALIZABLE. OVSDB does not allow this.

**No complex query language.** OVSDB has only `select` with simple conditions (column op value). No JOIN, GROUP BY, or subquery. This is acceptable because OVSDB data is small (thousands to tens of thousands of rows); the client can fetch everything and process in memory.

**No triggers or stored procedures.** No code runs in the server to enforce business logic. All logic lives in the client (`ovn-northd`, the Neutron plugin).

**The trade-off of the simple design.** OVSDB exchanges these limitations for ease of deployment, a C++ server of about 50,000 lines (versus PostgreSQL at about 1.4 million lines), low resource footprint (50 to 200 MB of RAM for clusters with hundreds of thousands of rows), and fast startup (subsecond). It fits the narrow scope of OVS and OVN state management.

### Guided Exercise 1: observe atomicity through a multi-operation transaction failure

> **State verification lab:** output is based on OVS 2.17 documentation; pending verification on a real lab.

**Objective.** Demonstrate that when one operation in a transaction fails, the previous operations are rolled back, leaving no "half-done" state visible.

**Preparation.** Node Ubuntu 22.04 with OVS 2.17 already running.

**Steps.**

**1.** Check the initial state:

```
[host]# ovs-vsctl show
```

Note the current bridge count.

**2.** Try a transaction that is intentionally wrong (create a bridge plus set an invalid interface type):

```
[host]# ovs-vsctl add-br br_test_fail -- add-port br_test_fail eth_bad \
         -- set Interface eth_bad type=nonexistent_type
```

Expected result:

```
ovs-vsctl: Interface "eth_bad": "nonexistent_type" is not a valid interface type
```

**3.** Check the state again. The bridge `br_test_fail` was **not created**:

```
[host]# ovs-vsctl show
```

The bridge count did not increase. This is atomicity: the whole transaction failed, even though the leading `add-br` had succeeded temporarily in memory.

**4.** Try an equivalent valid transaction for comparison:

```
[host]# ovs-vsctl add-br br_test_ok -- add-port br_test_ok eth_internal \
         -- set Interface eth_internal type=internal
[host]# ovs-vsctl show
```

The bridge `br_test_ok` is created successfully.

**Lessons learned.**

First, OVSDB atomicity is enforced at the level of the entire transaction; there is no "partial commit" if one operation fails. This is the core property that lets `ovn-northd`, the Neutron plugin, and Kubernetes CNIs trust the state in OVSDB.

Second, `ovs-vsctl` automatically groups several commands into one transaction when the `--` separator is used. If you run `ovs-vsctl add-br`, then `ovs-vsctl add-port`, then `ovs-vsctl set` separately, those are three transactions, each atomic on its own; but if command 3 fails, commands 1 and 2 have already committed.

**Cleanup:**

```
[host]# ovs-vsctl del-br br_test_ok
```

### Guided Exercise 2: `mutate` versus `update`, demonstrate avoiding the race

> **State verification lab:** pending verification.

**Objective.** Observe why `mutate insert` avoids a race condition that `update` does not.

**Preparation.** Lab with OVS running and at least one bridge `br0`.

**Steps.**

**1.** Open two terminals to the same OVS host.

**2.** Terminal A, run a loop that adds ports:

```
[host-A]# for i in {1..5}; do
[host-A]>   ovs-vsctl add-port br0 port_a$i -- set Interface port_a$i type=internal
[host-A]> done
```

**3.** Terminal B, at the same time:

```
[host-B]# for i in {1..5}; do
[host-B]>   ovs-vsctl add-port br0 port_b$i -- set Interface port_b$i type=internal
[host-B]> done
```

**4.** After both terminals finish, check that `br0` has all 10 new ports:

```
[host]# ovs-vsctl list-ports br0
```

Expectation: every port_a1 to port_a5 and port_b1 to port_b5 is present. OVS uses `mutate` internally, so there is no race.

**5.** Compare with `update` (do not use this in production). If you use `ovsdb-client transact` directly with the operation `update Bridge ports = [...]` instead of `mutate`, a race can happen when two clients update simultaneously.

**Lessons learned.**

First, `mutate insert` into a set is atomic from the client's point of view. The client does not need a "read then write" cycle, just a "add element X" call. The server handles concurrent mutates from multiple clients.

Second, understanding the internals of `ovs-vsctl` helps debug rare cases: if ports go missing after a concurrent update, suspect that another tool used `update` instead of `mutate`.

**Cleanup:**

```
[host]# for i in {1..5}; do ovs-vsctl del-port br0 port_a$i; ovs-vsctl del-port br0 port_b$i; done
```

## 10.3.X Real-world incident case (axis 17 backfill 2026-04-27)

### Production case: `nb_cfg` prerequisite race causing `ovn-northd` compile thrash (curated 2023-Q3 pattern)

**Symptom.** OVN cluster, OpenStack Yoga deployment. `ovn-northd` CPU pegged at 95 percent sustained. NBDB-to-SBDB compile lag rose from a baseline of 200 ms p99 to 8 to 15 seconds p99. Neutron `port-create` API calls stalled for 30 seconds or more. `ovn-controller` chassis reported "stale `Logical_Flow`" in `ovn-detrace` output.

**Initial diagnosis.**

- `ovs-appctl -t ovn-northd coverage/show | grep northd_run_time` reported 5000 ms or more per recompute (baseline 100 ms).
- `ovn-nbctl get NB_Global . nb_cfg` returned 1500.
- `ovn-sbctl get SB_Global . nb_cfg` returned 1480 (a delta of 20, about 5 seconds of lag, consistent with the recompute time).

**Root cause investigation.**

`ovn-northd` uses the `wait` operation in a `transact` request to the SBDB to enforce a prerequisite: northd commits the SBDB transaction only if the NBDB `nb_cfg` is unchanged during the compile window. Per Rule 14, source `northd/inc-proc-northd.c::northd_handler_logical_flow` plus `ovsdb-idl.c::ovsdb_idl_txn_write` per plan v3.9, verified.

Race condition: if `neutron-server` issues a steady burst of NB updates (for example, a bulk security group rule update causing 200 NB transactions per second), `nb_cfg` changes constantly. The northd compile request `wait nb_cfg=X` fails, retries, recompiles from the start, fails again, and loops indefinitely until the NB write rate drops.

**Root cause confirmation per Rule 14.**

- `ovn-nbctl --columns=_version list NB_Global .` shows version increments matching the update rate.
- Verify with `ovsdb_jsonrpc_session_got_request` in `ovsdb/jsonrpc-server.c`, the handler that increments the version per write.

**Resolution.**

1. Throttle `neutron-server` NB writes: introduce a 50 ms minimum gap between transactions through the Neutron config `[ovn] worker_throttle_ms = 50`.
2. Increase `inactivity_probe` for the `ovn-northd` to SBDB connection from 5 seconds to 30 seconds to avoid false disconnects during a compile burst.
3. Long term: adopt OVN 22.06 or later with the I-P (Incremental Processing) engine to recompute only the delta instead of the entire NBDB.

**Lessons learned.**

- The ACID `wait` operation guarantees correctness, but has a high cost under NB write contention. Client-side throttling is necessary for production stability.

## 8.1.1 brctl legacy vs ip link modern

Package `bridge-utils` (command `brctl`) được deprecate từ Ubuntu 18.04+. Modern syntax dùng iproute2:

```bash
# Legacy (avoid):
[host]# brctl addbr br0
[host]# brctl addif br0 eth0

# Modern:
[host]# ip link add name br0 type bridge
[host]# ip link set eth0 master br0
[host]# ip link set br0 up
```

Kernel implementation tại `net/bridge/` với FDB (chuyển tiếp database) aging time mặc định 5 phút. iproute2 unified cả bridge + VLAN + tunnel command thành một interface.

VLAN filtering trên Linux bridge (enable qua `vlan_filtering=1`):

```bash
[host]# ip link add name br0 type bridge vlan_filtering 1
[host]# bridge vlan add dev eth0 vid 100 pvid untagged
[host]# bridge vlan add dev eth1 vid 100
```

Tuy nhiên, VLAN filtering trên Linux bridge không đầy đủ như OVS — không có per-port VLAN translation, không có QinQ native, không có programmable matching.

## 8.1.2 Bridge datapath trong kernel

`net/bridge/br_forward.c` triển khai logic chuyển tiếp:

1. Packet đến bridge port → `br_handle_frame` hook được gọi.
2. Lookup FDB theo (dst MAC, VLAN ID).
3. Match → forward tới port đích.
4. Miss → flood ra mọi port (trừ ingress port).
5. Update FDB với (src MAC, VLAN ID, ingress port) cho learning.

STP optional (disable by default). netfilter bridge-hook (`ebtables`) cho filtering L2. Băng thông thực tế kernel bridge ~5-10 Gbps trên modern hardware — đủ cho hầu hết container use case không cần SDN.

Giới hạn quan trọng:

- **Không programmable** — rule chuyển tiếp cố định (MAC learning + flood on miss).
- **Không per-flow statistics** — chỉ có counter per-port.
- **Không có action phức tạp** — không set_queue, mod_nw_src, push_vlan per-rule.

## 8.1.3 veth pair: net/core/veth.c

veth tạo cặp đối xứng device, packet đưa vào một đầu xuất ra đầu còn lại — giống ethernet cable ảo.

```bash
[host]# ip link add veth-a type veth peer name veth-b
[host]# ip link set veth-a netns ns1
[host]# ip link set veth-b master br0    # hoặc netns ns2
```

Use case chính: nối namespace với bridge (một đầu trong namespace, một đầu trên bridge). Implementation: virtual Ethernet driver với synthetic tx/rx queue, không có hardware offload → pure CPU work, overhead ~1-2 μs per packet hop.

USC Lab 1 dùng pattern này:

```
namespace h1 ─ veth1 ── veth1-br ─ bridge s1 ─ veth2-br ── veth2 ─ namespace h2
```

veth kết nối namespace vào bridge, bridge làm L2 switch giữa các namespace.

## 8.1.4 macvlan 4 modes

macvlan tạo virtual interface với MAC address riêng trên cùng physical port — mỗi macvlan sibling có thể có IP riêng, xuất hiện như device độc lập với external switch.

**`private`**: sibling không forward cho nhau. Traffic giữa hai container trên cùng host đi ra external switch (nếu switch routing về) hoặc drop. Ít dùng vì performance kém.

**`vepa`** (Virtual Ethernet Port Aggregator, IEEE 802.1Qbg): traffic giữa sibling đi ra external switch rồi switch hairpin về. Yêu cầu switch hỗ trợ reflective relay.

**`bridge`** (default): container giao tiếp trực tiếp trong kernel bridge domain. Nhanh nhất, most common.

**`passthru`**: expose 1-to-1 physical port cho một container (SR-IOV-like). Container thấy port physical trực tiếp.

```bash
[host]# ip link add mac1 link eth0 type macvlan mode bridge
[host]# ip link set mac1 netns ns1
[host]# ip netns exec ns1 ip addr add 10.0.0.100/24 dev mac1
```

Nhược điểm macvlan: không giao tiếp được với host namespace (do thiết kế) — Docker không dùng macvlan làm default CNI vì host không ping được container.

## 8.1.5 ipvlan L2 vs L3

ipvlan share **một MAC address duy nhất** nhưng nhiều IP address. Khác macvlan ở chỗ không tạo MAC mới.

**L2 mode**: switching trong kernel, giống macvlan bridge mode nhưng shared MAC. External switch chỉ thấy một MAC, không đếm MAC table cho từng container.

**L3 mode**: kernel làm L3 chuyển tiếp giữa ipvlan children, không broadcast, không ARP qua IPVLAN boundary. Container phải có default route ra ipvlan parent.

Use case ipvlan: container overlay khi cần scale >1000 container per host — MAC table của physical switch upstream có giới hạn (32K-128K entries), quá nhiều container với MAC riêng (macvlan) gây overflow.

```bash
[host]# ip link add ipvl1 link eth0 type ipvlan mode l2
[host]# ip link set ipvl1 netns ns1
```

## 8.1.6 Pattern br-int/br-ex/br-tun (compass Ch F)

Compass Ch F mô tả canonical multi-bridge pattern cho OVS — foundation concept áp dụng cả cho Linux bridge:

```
      VMs                           Physical
       │                               │
   ┌───▼──────┐   patch   ┌────────────▼─┐
   │ br-int   ├──────────►│ br-ex         │
   └─┬────────┘           └───────────────┘
     │ patch
   ┌─▼────────┐
   │ br-tun   ├── VXLAN/Geneve đến remote hosts
   └──────────┘
```

- **`br-int`**: carry VM/container-facing traffic. OpenFlow policy live here.
- **`br-ex`**: peer với physical uplink, VLAN translation, NAT, provider network.
- **`br-tun`**: terminate overlay tunnel.

OVS patch port (Phần 9.1) nối 3 bridge với zero-copy (compass: "zero-copy tail-call, not a real queue"). Linux bridge không có patch port native — phải dùng veth pair, chậm hơn vì có rx/tx queue thật.

Đây là một trong những lý do OVS vượt trội cho SDN triển khai.

## 8.1.7 Tại sao OVS thay thế Linux bridge cho SDN

Linux bridge limitations:

- Không programmable — không có OpenFlow API.
- Không per-flow statistics.
- Không per-flow action (set_queue, modify headers).
- VLAN handling đơn giản, không QinQ native.
- Không có patch port zero-copy giữa bridge.

OVS giải quyết qua:

- OpenFlow pipeline (Phần 3.1, 4.0-4.6) — programmable classifier.
- Megaflow cache (Phần 9.2) — per-flow statistics tự động.
- Extensible action set (Phần 4.7) — set_queue, mod_*, ct(), set_tunnel.
- Native VXLAN/Geneve/STT (Phần 11.0).
- Patch port zero-copy.

Trade-off: Linux bridge đơn giản hơn vận hành, đủ cho workload không cần SDN. OVS phức tạp hơn nhưng cần thiết khi có overlay, multi-tenant, hay policy routing.

## 8.1.8 Source code veth driver tier 2 (`drivers/net/veth.c` v5.15)

veth driver là virtual Ethernet driver kernel mode. Khi packet vào một đầu, driver chuyển ngay sang đầu peer qua `netif_rx()` không có hardware queue. Lookup hàm chính:

### `veth_xmit()` — packet transmission

```c
// drivers/net/veth.c, function veth_xmit (gọi từ ndo_start_xmit)
static netdev_tx_t veth_xmit(struct sk_buff *skb, struct net_device *dev)
{
    struct veth_priv *priv = netdev_priv(dev);
    struct net_device *rcv;
    int length = skb->len;
    
    rcu_read_lock();
    rcv = rcu_dereference(priv->peer);
    if (unlikely(!rcv)) {
        kfree_skb(skb);
        goto drop;
    }
    
    if (likely(veth_forward_skb(rcv, skb, ...) == NET_RX_SUCCESS)) {
        // packet đã đẩy vào peer's RX queue
    }
    rcu_read_unlock();
    return NETDEV_TX_OK;
}
```

Key points:

- `priv->peer` là `struct net_device *` của đầu kia, gán lúc create veth pair.
- `rcu_read_lock()` đảm bảo peer không bị free trong khi đang access (ví dụ peer ns đang teardown).
- `veth_forward_skb()` gọi `__dev_forward_skb()` rồi `netif_rx()` để inject packet vào RX path của peer.

Không có DMA, không có hardware queue, không có IRQ. Mọi processing pure CPU.

### `veth_xdp_xmit()` — XDP fast path (kernel 4.18+)

Veth modern hỗ trợ XDP (eXpress Data Plane). Khi peer attach XDP program, packet đi qua XDP hook trước khi vào TCP/IP stack:

```c
static int veth_xdp_xmit(struct net_device *dev, ...)
{
    // Bulk forward XDP frames
    for (i = 0; i < n; i++) {
        struct xdp_frame *frame = frames[i];
        // Inject vào peer's XDP RX path
    }
}
```

Performance: XDP path bypass skb allocation + protocol stack → 5-10x throughput so với regular veth_xmit (15-30 Gbps thay vì 3-5 Gbps).

### Memory accounting

Veth không tốn memory đặc biệt: chỉ `struct net_device` (~2KB) + skb được pass-through. Khác container runtime tag đặc biệt — pure netdev.

### Limitation

- **No hardware offload** (TSO/GSO/GRO chỉ software). Throughput inter-namespace ~5-10 Gbps tối đa cho regular path.
- **No checksum offload**: kernel phải compute checksum cho mỗi packet.
- **CPU bound**: 1 CPU core per veth pair direction. Không phân tải qua RSS.

### So sánh với OVS internal port

OVS internal port (`type=internal`) có overhead lớn hơn (qua OVS userspace upcall path nếu cache miss), nhưng có advantage:

- Programmable OpenFlow (action set, priority, conntrack).
- Statistics per-port + per-flow.
- Hardware offload qua TC offload framework (Phần 9.5).

Pattern OVN-K8s dùng OVS internal thay veth vì cần programmability. Pattern legacy Docker dùng veth + bridge vì đơn giản hơn.

## 8.1.9 Anatomy Template A: `bridge fdb show` + `ethtool -S veth*`

> **Bối cảnh khi chạy:** Kiểm tra FDB của Linux bridge + statistics veth pair.

```bash
[host]# bridge fdb show br br0
01:00:5e:00:00:01 dev veth1-br self permanent       # multicast group join
33:33:00:00:00:01 dev veth1-br self permanent
aa:bb:cc:00:00:01 dev veth1-br vlan 1 master br0    # learned host MAC
aa:bb:cc:00:00:02 dev veth2-br vlan 1 master br0
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| `<MAC>` | Unique per host | Duplicate (different port) | Same MAC across multiple bridge | MAC flap; check loop |
| `dev` | Match expected egress port | `master br0 self` cho permanent (multicast) | Wrong port | Bridge loop hoặc misconfig |
| `vlan N` | Match VLAN config | (n/a nếu vlan_filtering off) | Empty cho VLAN bridge | VLAN filtering disabled |
| `master` | bridge name | (n/a) | Empty | Port không attach bridge |
| Aging | Auto-aged sau 300s default | Not aging (permanent stuck) | Aging too aggressive | `ip link set br0 type bridge ageing_time` |

```bash
[host]# ethtool -S veth1
NIC statistics:
     peer_ifindex: 23
     rx_queue_0_xdp_packets: 0
     rx_queue_0_xdp_bytes: 0
     ...
```

Veth ít stat (so với physical NIC), chủ yếu peer_ifindex + XDP counters nếu enabled.

**Kịch bản bẻ gãy:**

- Bridge fdb empty cho port có traffic: `vlan_filtering=1` nhưng port không cấu hình VLAN. Add `bridge vlan add dev <port> vid 1 pvid untagged`.
- MAC flap (cùng MAC trên 2 port): bridge loop. Run `mstpd` hoặc disable 1 port.
- Veth throughput 5 Gbps cap dù có 25G NIC: kernel veth path bottleneck. Switch sang OVS internal port (DPDK) hoặc XDP path.

## 8.1.10 ▶ Capstone POE: veth vs OVS internal port cho 1000 container per host

> **Bối cảnh:** Hyperscale node 1000 container, mỗi container 1 interface. So sánh veth+Linux bridge vs OVS internal port.

**Tuyên bố cần kiểm chứng:** "veth + Linux bridge đơn giản hơn nên scale tốt hơn cho 1000 container."

**Predict:** Sai. Linux bridge có scale wall ở MAC table + FDB lookup time + không có programmable policy.

**Observe (giả định):**

- 1000 veth pair + 1 Linux bridge: FDB 2000+ entries (mỗi veth có 1 MAC + multicast group). Bridge MAC lookup O(log N) qua hash, ~1-2 μs. CPU usage ovs daemon = 0 (không dùng), CPU usage bridge kernel = 5-10% steady.
- Mỗi container ping cross-container: 1.5-3 μs latency (kernel bridge L2 lookup).
- Throughput per pair: 5-8 Gbps (CPU bound).
- **Vấn đề scale:** không có per-flow policy. Mọi container cùng broadcast domain → broadcast storm risk. Không có ACL per-container. Không có overlay tự nhiên.

OVS internal port + br-int:

- 1000 OVS port (internal hoặc tap-bridged): OVS classifier 3-tier (EMC + SMC + megaflow), per-flow lookup ~100-500 ns sau khi cache warm.
- OpenFlow policy enforce per-port (ACL, QoS, conntrack).
- Tunnel native (Geneve/VXLAN) cho cross-host.
- CPU usage ovs-vswitchd: 30-50% on 1 core, có thể scale lên multi-core qua multi-threading.
- Throughput: 25+ Gbps với hardware offload.

**Explain:**

1. **Linux bridge hợp cho < 100 container** không cần policy. Đơn giản, dễ debug, kernel built-in.
2. **OVS bắt buộc cho > 100 container** hoặc cần ACL/QoS/overlay. Programmability + hardware offload + tunnel native.
3. "Đơn giản" của Linux bridge thực ra là **giới hạn**: không scale được policy → kỹ sư phải workaround qua iptables/nftables (slow, không per-container).

**Falsification test:**

- Setup 1000 container với Linux bridge → measure throughput + latency. Throughput bottleneck CPU bound 5-8 Gbps total per pair.
- Setup 1000 container với OVS internal port → throughput per pair 20+ Gbps với offload. Latency tương đương ~2 μs cho cache hit.
- Nếu Linux bridge thắng OVS về throughput → counter-example. Trong thực tế OVS thắng cho hyperscale.

**Tuyên bố sai ở đâu:** "Đơn giản hơn nên scale tốt hơn" là logical fallacy (đơn giản về interface ≠ scale về workload). Đúng phải: **Linux bridge cho lab/dev hoặc < 100 container không cần policy. OVS bắt buộc cho production hyperscale + multi-tenant.**

> **Quy tắc:** < 100 container, no policy = Linux bridge OK. > 100 container hoặc policy/overlay = OVS bắt buộc.

### ▶ Guided Exercise 2: Linux bridge + 2 namespace + veth

**Outcomes:** Tạo Linux bridge + 2 namespace + veth pair, kiểm chứng L2 connectivity.

**Instructions:**

**1.** Setup:

```bash
[host]# ip netns add h1 && ip netns add h2
[host]# ip link add name br0 type bridge
[host]# ip link set br0 up

[host]# ip link add veth1 type veth peer name veth1-br
[host]# ip link add veth2 type veth peer name veth2-br

[host]# ip link set veth1 netns h1
[host]# ip link set veth2 netns h2
[host]# ip link set veth1-br master br0
[host]# ip link set veth2-br master br0
[host]# ip link set veth1-br up
[host]# ip link set veth2-br up

[host]# ip -n h1 addr add 10.0.0.1/24 dev veth1
[host]# ip -n h2 addr add 10.0.0.2/24 dev veth2
[host]# ip -n h1 link set veth1 up
[host]# ip -n h2 link set veth2 up
```

**2.** Kiểm chứng:

```bash
[host]# ip netns exec h1 ping -c 3 10.0.0.2
[host]# bridge fdb show br br0
```

FDB entries sẽ hiện MAC của veth1/veth2-br sau ping đầu tiên (learned).

**3.** Cleanup:

```bash
[host]# ip netns delete h1 h2
[host]# ip link delete br0
```

## Điểm cốt lõi cần nhớ

Thứ nhất, Linux bridge là L2 learning switch trong kernel, đủ cho container workload đơn giản. iproute2 (`ip link`, `bridge`) thay thế brctl legacy.

Thứ hai, veth pair là primitive duy nhất cho SDN lab — nối namespace với bridge, zero overhead trong kernel context.

Thứ ba, macvlan (4 mode) và ipvlan (L2/L3) là alternatives cho scenario nhiều container — scale tốt hơn veth khi >100 container/host. Trade-off: không host-container communication với macvlan.

Thứ tư, OVS thay thế Linux bridge khi cần SDN feature (OpenFlow, overlay, flow statistics). Linux bridge đủ cho non-SDN container workload.

## 13.8.1 The role of ovn-northd in the OVN architecture

**ovn-northd** is a daemon process that runs centrally in the OVN cluster (typically on the same node as the NB and SB ovsdb-server, or on a separate node in HA configurations). Its only responsibility: read NB DB (the configuration intent that the CMS, such as OpenStack Neutron or OVN-Kubernetes, writes) and compile it into SB DB (Logical Flow plus Port_Binding plus Multicast_Group) so that the local ovn-controller on every chassis can read it and execute it on br-int.

The **NB to SB compile pipeline** has three clear tiers:
- **Tier 1, NB DB (configuration intent):** the CMS writes high-level objects such as Logical_Switch, Logical_Router, Logical_Switch_Port, ACL, Load_Balancer, NAT, and DHCP_Options. NB contains no OpenFlow; it only defines topology and policy.
- **Tier 2, ovn-northd (translator):** the daemon watches every NB DB change through OVSDB IDL, computes the corresponding Logical Flow rows, and writes them into SB DB. It also computes Port_Binding (binding a port to a chassis), Datapath_Binding (each LS or LR is assigned a tunnel_key), and Multicast_Group (the broadcast domain).
- **Tier 3, SB DB (physical-neutral intermediate):** holds chassis-independent Logical Flow rows. Every ovn-controller reads the same SB DB and translates further into OpenFlow appropriate for its local chassis (this was covered in Part 13.7).

This model enables a clean **separation of concerns**: the CMS only needs to know the high-level configuration intent (Logical Switch, ACL) and does not need to know OpenFlow or tunnel encapsulation. ovn-northd is the central compile engine that holds the full NB topology, computes every interaction (ACL by Port by Load_Balancer by NAT), and emits complete Logical Flow output. The local ovn-controller only needs to translate the subset of Logical Flow relevant to its chassis into OpenFlow; it does not need to understand the NB intent.

A standard OVN cluster runs **one** active ovn-northd instance at a time (with one or more standby instances under leader election). The reason: ovn-northd writes a large number of records into SB DB for every NB change. If multiple instances ran in parallel they would create write conflicts. Leader election (through pacemaker, Kubernetes leader election, or the built-in OVN approach `--ovnnb-db=tcp:nb1,tcp:nb2,tcp:nb3 --ovnsb-db=tcp:sb1,tcp:sb2,tcp:sb3`) guarantees a single active. Standby instances simply wait. If the active dies (detected by lost connection plus timeout), one standby is promoted to active.

> **Version note:** OVN 20.06 and earlier used a single-process ovn-northd that **recomputed everything** from NB to SB on every NB change. OVN 22.03 introduced **parallel compilation**: a multi-threaded ovn-northd that processes multiple Logical Switch and Logical Router objects in parallel, improving real throughput by 3 to 5 times on multi-core CPUs. OVN 24.03 and later introduced **incremental northd** (currently in preview), similar to the I-P engine in ovn-controller, which only recomputes the subset of Logical Flow rows affected by an NB change.

## 13.8.2 The NB to SB Logical Flow translation algorithm

When NB DB changes (for example, the CMS creates a new Logical Switch), ovn-northd performs a sequence of steps:

**Step 1, compute datapath and tunnel_key:**
- Each Logical Switch and Logical Router in NB maps to a `Datapath_Binding` record in SB.
- ovn-northd assigns a `tunnel_key` (1 to 16777215) unique to each datapath, used as OpenFlow metadata (covered in Part 13.7).
- Datapath_Binding records are created, updated, or deleted in step with NB.

**Step 2, compile Logical Flow for the Logical Switch ingress pipeline:**
Each Logical Switch has 26 logical tables in the ingress pipeline at v22.03.8, numbered 0 to 25 (verified by `grep -cE "PIPELINE_STAGE\(SWITCH, IN" northd/northd.c` returning 26). The Logical_Router ingress pipeline has a separate count of 20 logical tables (0 to 19). ovn-northd emits Logical_Flow rows for each table in order:
- `ls_in_check_port_sec`, port security check (sourced from `Logical_Switch_Port.port_security`).
- `ls_in_apply_port_sec`, apply port security drop.
- `ls_in_lookup_fdb`, FDB lookup (MAC learning).
- `ls_in_put_fdb`, FDB insert for packets whose source MAC is unknown.
- `ls_in_pre_acl`, stateful ACL pre-check.
- `ls_in_pre_lb`, load-balancer pre-routing.
- `ls_in_pre_stateful`, conntrack commit trigger.
- `ls_in_acl_hint`, ACL hint for ct_state.
- `ls_in_acl`, ACL from the `ACL` table in NB.
- `ls_in_qos_mark`, QoS mark.
- `ls_in_qos_meter`, QoS meter.
- `ls_in_lb`, load-balancer DNAT (frontend to backend).
- `ls_in_lb_aff_check`, LB affinity check.
- `ls_in_stateful`, NAT commit.
- `ls_in_arp_rsp`, ARP responder for Logical Router ports.
- `ls_in_dhcp_options`, DHCPv4 and DHCPv6 response.
- `ls_in_dns_lookup`, DNS lookup.
- `ls_in_dns_response`, DNS response.
- `ls_in_external_port`, external port handling.
- `ls_in_l2_lkup`, L2 destination lookup.
- `ls_in_l2_unknown`, L2 unknown MAC (flood).

Each table has Logical Flow entries with priority, match, and action. For example, the ACL `from-lport direction=inbound priority=500 match=ip4.src==10.0.0.0/8 action=drop` compiles to:
```
table=ls_in_acl, priority=500, match=(inport == <port_groups> && ip4.src == 10.0.0.0/8), action=(drop;)
```

**Step 3, compile Logical Flow for the Logical Switch egress pipeline:**
Similar to ingress but for egress, around 10 tables (egress ACL, egress LB, egress port-security check, L2 output, and so on).

**Step 4, compile Logical Flow for the Logical Router ingress and egress pipelines:**
Logical Router is more complex, around 30 ingress tables and 15 egress tables (OVN 22.03):
- `lr_in_admission`, admission control.
- `lr_in_lookup_neighbor`, ARP and NDP lookup.
- `lr_in_learn_neighbor`, ARP learning.
- `lr_in_ip_input`, IP input processing (TTL decrement, destination check).
- `lr_in_unsnat`, reverse SNAT.
- `lr_in_defrag`, defragmentation for LB traffic.
- `lr_in_dnat`, DNAT and LB.
- `lr_in_ecmp_stateful`, ECMP path selection.
- `lr_in_nd_ra_options`, IPv6 ND Router Advertisement.
- `lr_in_ip_routing_pre`, pre-routing hook.
- `lr_in_ip_routing`, routing-table lookup.
- `lr_in_ip_routing_ecmp`, ECMP routing.
- `lr_in_policy`, Logical Router Policy (policy-based routing).
- `lr_in_arp_resolve`, ARP resolution for the next hop.
- `lr_in_chk_pkt_len`, packet-length check for PMTU.
- `lr_in_larger_pkts`, send ICMP "Packet Too Big" if needed.
- `lr_in_gw_redirect`, redirect to the gateway chassis.
- `lr_in_arp_request`, send ARP request.

**Step 5, Port_Binding and Multicast_Group:**
Each Logical_Switch_Port and Logical_Router_Port in NB produces a Port_Binding record in SB. Port_Binding contains `type` (vif, localnet, l2gateway, chassisredirect, patch), `options`, and `tunnel_key`. The broadcast domain (flooding) is computed into Multicast_Group: each LS has one MG for flood, and each LRP has an MG for neighbour discovery.

## 13.8.3 Leader-election HA for ovn-northd

Because only one active ovn-northd is allowed to write SB DB at a time, a production OVN cluster needs an HA mechanism that guarantees (a) there is always an active instance if the current node crashes, and (b) split-brain is avoided (two active instances writing concurrently leading to inconsistency).

**Method 1, Pacemaker (legacy):** uses Pacemaker as the cluster manager that owns the `ovn-northd` resource. Pacemaker uses Corosync for leader election and starts or stops the `ovn-northd.service` accordingly. This method was popular on RHEL OpenShift 3.x but is complex to set up and is being phased out.

**Method 2, built-in leader election (OVN 22.03 and later):** ovn-northd uses ovsdb-server for leader election. Each ovn-northd instance connects to the full NB DB cluster and the SB DB Raft cluster. The instance that grabs the `ovn_northd` lock on SB DB first becomes active. The other instances register interest in this lock; when the active releases it (clean shutdown) or loses connection, a standby acquires the lock.

```
[node1]# ovn-northd --pidfile --detach \
         --ovnnb-db=tcp:10.0.0.1:6641,tcp:10.0.0.2:6641,tcp:10.0.0.3:6641 \
         --ovnsb-db=tcp:10.0.0.1:6642,tcp:10.0.0.2:6642,tcp:10.0.0.3:6642
[node2]# <same command, different node>
[node3]# <same command, different node>
```

Only one instance is active at a time; identify which one through:
```
[host]# ovs-appctl -t ovn-northd nb-connection-status
[host]# ovs-appctl -t ovn-northd sb-connection-status
```
The active instance shows `connected` for both. A standby shows `connected` for NB but `waiting` for SB (it is waiting for the lock).

**Method 3, Kubernetes leader election (OVN-Kubernetes):** uses a Kubernetes Lease object for leader election. The `ovn-k8s-northd` controller manages state through a `Lease` resource in the `ovn-kubernetes` namespace.

Failover time (active dies, standby comes up) is 5 to 15 seconds depending on timeout configuration. During this window, SB DB does not receive new Logical Flow updates (NB changes pile up). ovn-controller keeps running normally because it reads its local SB DB cache; only new configuration intent is delayed until the new active comes up.

## 13.8.4 Debugging ovn-northd

A debugging session for ovn-northd typically starts with "I changed NB but SB does not reflect it" or "the SB Logical Flow is wrong compared to the NB intent".

**Tool 1, compare NB and SB state:**
```
[host]# ovn-nbctl show
[host]# ovn-sbctl show
[host]# ovn-sbctl list Logical_Flow | head -50
```

**Tool 2, ovn-northd coverage counters:**
```
[host]# ovs-appctl -t ovn-northd coverage/show
```
This emits counters: `northd_run` (number of recompute cycles), `lflow_generated` (number of Logical Flow rows produced), `port_binding_up` (number of port bindings). Track these over time to detect anomalies.

**Tool 3, northd logs:**
```
[host]# tail -f /var/log/ovn/ovn-northd.log
```
Each recompute logs an entry:
```
2026-04-22T14:23:10.123Z|00042|ovn_northd|INFO|northd recompute took 45 ms
```
If the time grows abnormally (for example, several seconds), NB has grown too large or the compile algorithm is in a loop.

**Tool 4, nb_cfg synchronisation:**
NB DB has a counter `NB_Global.nb_cfg` that increments on changes. ovn-northd copies this counter into `SB_Global.nb_cfg` after each recompute. Compare the two counters:
```
[host]# ovn-nbctl get NB_Global . nb_cfg
42
[host]# ovn-sbctl get SB_Global . nb_cfg
42
```
If they differ significantly (for example, NB equals 42 and SB equals 38), ovn-northd is not catching up with NB changes and there is a backlog.

**Tool 5, ovn-northd multi-thread (22.03 and later):**
Check the thread count:
```
[host]# ps -T -p $(pidof ovn-northd)
```
By default ovn-northd uses N threads where N equals the number of CPU cores divided by 2. Override through `--n-threads=4`. Track per-thread CPU to identify the bottleneck.

## 13.8.5 Source code architecture, from `main()` to `build_lflows()`

This subsection dives into the ovn-northd source-code organisation at the OVN 22.03.8 baseline (`branch-22.03` upstream). Three core files:

- `northd/ovn-northd.c` (about 1022 lines): contains `main()`, parse_options, leader-election logic, and the top-level main loop. This is the daemon entry point.
- `northd/inc-proc-northd.c` (about 296 lines): the I-P (Incremental Processing) engine bootstrap. Declares and runs the two engine nodes of ovn-northd.
- `northd/northd.c` (about 15947 lines): all of the NB to SB Logical Flow translation logic. The largest file in the OVN source tree, containing hundreds of `build_*()` functions for each pipeline stage.

### Call chain from entry to Logical Flow generation

```
main()                                  [northd/ovn-northd.c]
  └─> ovsdb_idl loop
        └─> inc_proc_northd_run()       [northd/inc-proc-northd.c]
              ├─> engine_run()            (run a 2-node DAG: en_northd + en_lflow)
              │     ├─> en_northd_run()   [northd/en-northd.c]
              │     │     └─> ovnnb_db_run()         [northd/northd.c]
              │     │           ├─> build_datapaths()
              │     │           ├─> build_ports()
              │     │           ├─> build_lrouter_lbs()
              │     │           └─> ... (about 30 build_* functions)
              │     └─> en_lflow_run()    [northd/en-lflow.c]
              │           └─> build_lflows()           [northd/northd.c]
              │                 ├─> build_lswitch_input_port_sec_op()
              │                 ├─> build_lswitch_learn_fdb_op()
              │                 ├─> build_pre_acls()
              │                 ├─> build_acls()
              │                 ├─> build_qos()
              │                 ├─> build_lb()
              │                 └─> ... (about 50 build_* per pipeline stage)
              └─> commit transaction (write SB)
```

**Role split:**

- `main()`: set up signal handlers, parse `--ovnnb-db=...` and `--ovnsb-db=...`, run leader election, and enter the main loop.
- `inc_proc_northd_run()`: trigger one engine cycle and call each `en_*_run()` node in DAG order.
- `ovnnb_db_run()`: compute **datapaths plus ports plus LB plus parsed ACL** and store the result into `struct northd_data` (an intermediate state structure that is not written to SB yet).
- `build_lflows()`: use `northd_data` to produce the actual **Logical Flow** rows that are written into the SB `Logical_Flow` table. This function calls more than 50 `build_*` functions, one for each table in the LS plus LR pipelines.

### Stopwatch instrumentation

`ovnnb_db_run()` opens a stopwatch at the top of the function:

```c
stopwatch_start(BUILD_LFLOWS_CTX_STOPWATCH_NAME, time_msec());
```

The macro name `BUILD_LFLOWS_CTX_STOPWATCH_NAME` (defined in `northd/inc-proc-northd.h`) expands to `"build_lflows_ctx"`. The operator observes it through `ovs-appctl -t ovn-northd stopwatch/show`:

```
[host]# ovs-appctl -t ovn-northd stopwatch/show
Statistics for 'ovnnb_db_run'
  Total samples: 152
  Maximum: 245 ms
  Minimum: 12 ms
  95th percentile: 89 ms
  Short term average: 34 ms
Statistics for 'build_lflows_ctx'
  Total samples: 148
  Maximum: 198 ms
  ...
```

> **Version warning:** OVN 22.03 has 2 main stopwatches (`ovnnb_db_run` and `build_lflows_ctx`). OVN 24.03 expands this into roughly 8 stopwatches per phase (ports, ACL, LB, neighbor learning). When you read `stopwatch/show` on a production system, cross-check the version first.

## 13.8.6 The I-P engine for ovn-northd (22.06 and later)

Part 13.7 §13.7.3 introduced the I-P engine for **ovn-controller** (a DAG with more than 30 nodes). ovn-northd also has its own I-P engine, which appeared later and is simpler.

### Origin and scope

OVN 22.06 (released after the 22.03 curriculum baseline) added an I-P engine for ovn-northd. The DAG has 2 nodes at `branch-22.03`:

- `en_northd` (input: NB tables, Chassis, IDL change). Output: `northd_data` (datapath, ports, parsed ACL).
- `en_lflow` (input: `northd_data`). Output: SB `Logical_Flow` table.

Declared in `northd/inc-proc-northd.c`:

```c
static ENGINE_NODE(northd, "northd");
static ENGINE_NODE(lflow, "lflow");
```

`ENGINE_NODE` is a macro that expands into a `struct engine_node` containing a `compute()` callback (full recompute) plus per-input-source `change_handler()` callbacks (incremental).

### Recompute versus incremental

When NB changes, each node has 3 options:

1. **No change:** the input did not change, the output is preserved, no recompute.
2. **Incremental:** use `change_handler()` to update a subset of the output (for example, adding a single ACL row only triggers a flow rebuild for the single LS affected).
3. **Full recompute:** `compute()` runs from scratch (it calls the full `ovnnb_db_run()` plus the full `build_lflows()`). This is the fallback when no handler is registered or the handler returns `false` (the change kind is not supported).

At `branch-22.03`, the incremental support in ovn-northd is still limited. Most NB changes fall back to full recompute. This is the reason behind the case study recorded in Part 20.5: a **bulk tenant deletion of 5000 LSPs** causes a 4 minute 40 second stall, with a full recompute of around 50 ms per transaction; processing 5000 LSPs one at a time takes hours.

> **Version note:** OVN 23.09 and later substantially expand the change handlers (ports, ACL, LB). The incremental ratio reaches 95 percent and higher for typical workloads. OVN 24.03 and later add a `northd_full_recompute_ratio` metric in `inc-engine/show`. The 22.03 curriculum baseline does not expose this metric.

### Anatomy Template A: `inc-engine/show` (ovn-northd, 22.06 and later)

> **When to run:** the operator suspects ovn-northd is compiling slowly or that the incremental ratio is low. The command returns engine DAG statistics since the daemon started.

```bash
ovs-appctl -t ovn-northd inc-engine/show
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| `node: northd` recompute | < 5 per minute | 5 to 30 per minute | > 30 per minute | Bulk NB update or schema event; check nb_cfg lag. |
| `node: northd` compute (incremental) | > 95 percent of total | 80 to 95 percent | < 80 percent | The change handler does not support this change kind (for example, bulk Port_Group). |
| `node: lflow` recompute | < 10 per minute | 10 to 60 per minute | > 60 per minute | Depends on the upstream `northd` node; if lflow recompute is high while northd is low, the build context changed. |
| `node: lflow` compute | > 90 percent | 70 to 90 percent | < 70 percent | Same idea, check the change-handler scope. |
| Engine cycle time | < 100 ms | 100 to 500 ms | > 500 ms | NB is large (10K plus objects); consider enabling `--n-threads`. |
| `cancelled cycle` | 0 | 1 to 2 per hour | > 2 per hour | OVSDB transaction conflict; check ovsdb-server logs. |
| Last cycle status | `success` | `aborted` | `cancelled` | `aborted` means handler error; `cancelled` means OVSDB conflict. |

**Failure scenarios:**

- `northd` recompute equals 100 percent but `lflow` compute is 95 percent incremental: an NB bulk change (ports or ACL) where the Logical Flow structure does not change. The lflow handler falls back to no-op per row, which is good.
- `lflow` recompute spikes after changing `--n-threads`: the thread pool reinit invalidates the cache and recomputes once on the first cycle. This is expected.
- Continuous `cancelled`: the OVSDB Raft cluster has a problem (election in progress, slow follower). Check `cluster/status` on SB DB.

## 13.8.7 Parallel build internals

OVN 22.03 supports parallel `build_lflows()` through the `--n-threads=N` flag. The runtime flag `use_parallel_build` (`northd/northd.c` near top) toggles the feature.

### Worker function

`build_lflows_thread()` (`northd/northd.c`) is the worker function that runs on each pthread:

```c
static void *
build_lflows_thread(void *arg)
{
    struct worker_pool *pool = arg;
    /* Each thread receives a subset of Logical Switch and Logical Router
     * objects and runs the corresponding build_* functions independently. */
    ...
}
```

Work distribution:

- **Logical Switch:** each LS is assigned to one thread (round-robin index modulo `n-threads`). Build both ingress and egress pipelines.
- **Logical Router:** the same idea, one thread per LR.
- **Logical Flow output:** each thread writes to its own hash map (`hmap_local`). After all threads finish, `fix_flow_map_size()` (`northd/northd.c` around L14165) merges the per-thread hmaps into a single global hmap.

### Interaction with dp-groups

OVN 22.03 has the `Logical_Flow.logical_dp_group` feature (datapath group), which collapses identical Logical Flow rows across multiple LS or LR datapaths into a single SB row. When `use_parallel_build = true` is combined with `use_logical_dp_groups = true`, each thread computes groups locally; reconciliation across threads happens after the merge.

A real comment from the source (`northd/northd.c` around L4794 to L4796):

> `lflow table across threads when parallel lflow build and dp-group are both`
> `are used instead of just one. This is possible because when parallel build`

### Measuring effectiveness

```bash
[host]# ovs-appctl -t ovn-northd stopwatch/show | grep build_lflows
Statistics for 'build_lflows_ctx'
  Maximum: 89 ms      # with --n-threads=8 on a 16-core host
  ...
```

For comparison, the same workload on `--n-threads=1` returns `Maximum: 312 ms`. Speedup is around 3.5 times (sub-linear because of memory bandwidth and lock contention).

### ▶ Capstone POE (S64' Phase I.A): Recompute versus incremental trade-off

> **Hypothetical setup:** an OVN 22.03 cluster with 1000 Logical Switches, 200 Logical Routers, and around 50K Logical Flow rows in SB.

**Claim to test:** "Enabling `--n-threads=8` in ovn-northd always improves latency for every workload."

**Predict:** speedup is close to linear (around 6 times on 8 cores).

**Observe (hypothetical, to verify when a lab is available):**

- Bulk NB change (for example, adding 100 ACL rules): the `build_lflows_ctx` stopwatch drops from 280 ms to 75 ms (3.7 times speedup).
- Single ACL change: the stopwatch drops from 32 ms to 28 ms (1.1 times speedup, almost unchanged).
- Heavy I/O on SB OVSDB Raft (3-node SB): the stopwatch drops 2 times but the `inc-engine/show` cancelled rate rises from 0 per hour to 5 per hour.

**Explain:**

1. The speedup only takes effect for **bulk recompute** (wide output dependency). A single change still falls back to one thread because the work is smaller than the thread setup overhead.
2. A high `--n-threads` increases the SB Raft transaction-commit load. On a slow Raft cluster (slow follower, network lag), conflict rate rises, cancelled cycles rise, and the operator gets a misleading warning about northd when the real problem is SB DB.
3. The sub-linear speedup (3.7 times on 8 threads) is caused by shared memory for `northd_data` and locks during hmap merge.

**Falsification test:** run `--n-threads=1` then `--n-threads=8`, measure both `stopwatch/show` (compile time) and `inc-engine/show` (cancelled rate). If `--n-threads=8` does not improve compile time appreciably (for example, less than 1.5 times) **or** the cancelled rate rises above 5 per hour, enabling parallel build is not a win for this specific cluster.

**Where the claim is wrong:** "always improves" is the wrong wording. The accurate statement is: "usually improves bulk workloads on a cluster with a healthy SB Raft; you must measure on the specific cluster before defaulting to enabled."

### ▶ Guided Exercise 1: observe ovn-northd compiling NB into SB Logical Flow

> **State verification lab:** the output is based on OVN 22.03 documentation; pending real-lab verification.

**Goal:** demonstrate that the NB to SB compile chain runs automatically on every change. Compare the NB configuration intent with the SB Logical Flow that comes out.

**Setup:** an Ubuntu 22.04 node with OVN 22.03 installed.

**Steps:**

**1.** Check that the initial NB and SB state is empty:

```
[host]# ovn-nbctl show
<empty>
[host]# ovn-sbctl lflow-list | wc -l
0
```

**2.** Create a Logical Switch through NB:

```
[host]# ovn-nbctl ls-add web_ls
[host]# ovn-nbctl lsp-add web_ls web1
[host]# ovn-nbctl lsp-set-addresses web1 "aa:bb:cc:dd:ee:01 10.10.10.1"
```

**3.** Wait 1 to 2 seconds for ovn-northd to compile, then check SB:

```
[host]# sleep 2
[host]# ovn-sbctl list Datapath_Binding
_uuid               : ...
tunnel_key          : 1
external_ids        : {name=web_ls, name2=""}
...
[host]# ovn-sbctl list Port_Binding
_uuid               : ...
logical_port        : "web1"
tunnel_key          : 1
...
[host]# ovn-sbctl lflow-list | grep "Datapath: \"web_ls\"" -A 5
```

**4.** Measure the recompute time:

```
[host]# ovs-appctl -t ovn-northd coverage/show | grep northd_run
northd_run: 3
```

**5.** Add an ACL and observe the new Logical Flow rows:

```
[host]# ovn-nbctl acl-add web_ls from-lport 500 "ip4.src == 192.168.0.0/16" drop
[host]# sleep 2
[host]# ovn-sbctl lflow-list web_ls | grep "ls_in_acl"
```

Expected result: a new entry matching `ip4.src == 192.168.0.0/16` with action `drop;`.

**Lessons learned:**

First, ovn-northd converts every NB change into Logical Flow within seconds. Latency grows with cluster size (a large cluster of 10K objects can take tens of seconds for one full recompute).

Second, a single NB object (for example, one ACL rule) can produce 3 to 5 Logical Flow rows (entries across multiple tables to handle different ct_state values). This is why SB DB can be 5 to 10 times larger than NB DB in record count.

Third, the OVN pipeline model (24 ingress tables and 10 egress tables for LS, 30 plus 15 for LR) is the **intermediate language** between the high-level CMS configuration intent and low-level OpenFlow. Every OVN feature maps onto a specific subset of tables in the pipeline.

**Cleanup:**

```
[host]# ovn-nbctl ls-del web_ls
```

## 13.8.8 In-depth profile of the ovn-northd daemon

> This subsection collects a complete technical portrait of the `ovn-northd` daemon at the level an SDN engineer needs in order to operate it confidently: from definition, history, and compile principles to real incidents and self-test exercises. Subsections §13.8.1 to §13.8.7 covered source code and multi-thread parameters in detail; this subsection is the closing summary that serves as the system "ID card" for `ovn-northd`.

### Core concept

`ovn-northd` is a C daemon in the upstream OVN tree, in the centralised control-plane process group. Its only responsibility: subscribe to the Northbound Database (NB DB, the declarative configuration intent that the CMS writes) and compile it into the Southbound Database (SB DB, the intermediate representation that contains `Logical_Flow`, `Datapath_Binding`, `Port_Binding`, `Multicast_Group`, and `MAC_Binding`). This daemon does not speak OpenFlow, does not connect to switches, and does not process packets. It is the central "compiler" between two OVSDB schemas.

How it differs from neighbouring entities: `ovn-controller` runs distributed on each chassis (it reads the local SB DB and translates into OpenFlow on `br-int`); `ovsdb-server` is a generic daemon that hosts any schema; `ovn-northd` is a specialised client of two `ovsdb-server` clusters (NB and SB), with the compile logic baked into the code.

### History and motivation

The OVN project was started by Justin Pettit, Ben Pfaff, and the Nicira and VMware team in 2015 on top of the OVS codebase, with the goal of replacing the Neutron OVS agent architecture (each node ran an imperative agent that produced OpenFlow from local state) with a logic-centric centralised control plane. Version 2.4 (late 2015) marked the first public release of `ovn-northd` as a separate daemon. Why split it out: if `ovn-controller` on every chassis had to compile from NB DB itself, every chassis would need to load the entire NB and repeat the same translation work (hundreds of chassis solving the same problem). Splitting `ovn-northd` into the centre means only one node does the heavy work; every chassis consumes the result through SB DB.

Major milestones: OVN 20.06 kept the full-recompute model on every NB change. OVN 22.03 (the curriculum baseline) introduced the I-P engine for northd as a 2-node DAG. OVN 22.06 added parallel `build_lflows` that splits LS and LR objects across multiple pthreads. OVN 23.09 expanded change handlers so that the incremental ratio is high for common workloads. OVN 24.03 added the `northd_full_recompute_ratio` metric and a schema-delta refactor.

### Position in the architecture

`ovn-northd` sits in the **centralised control plane**, scoped to the cluster (one active for the whole OVN cluster). It has no datapath component and no user-space switch logic. The process lives on the same node as the NB and SB `ovsdb-server` in simple deployments; in HA clusters it can run on a dedicated node and connect to `tcp:nb-host:6641` and `tcp:sb-host:6642` over the management network.

In the three-tier OVN diagram (CMS to NB DB to SB DB to ovn-controller to OpenFlow), `ovn-northd` stands between the NB tier and the SB tier as a one-way valve: it reads NB and writes SB, never the other direction.

### Role and responsibilities

The role splits into four groups. First, **topology translation**: each `Logical_Switch` and `Logical_Router` in NB produces an SB `Datapath_Binding` with a unique 24-bit `tunnel_key`. Second, **port translation**: each `Logical_Switch_Port` and `Logical_Router_Port` becomes an SB `Port_Binding` with a specific type (vif, localnet, l2gateway, chassisredirect, patch). Third, **Logical Flow generation**: for each datapath, produce dozens to hundreds of `Logical_Flow` rows distributed across the LS and LR ingress and egress tables to implement ACL, LB, NAT, routing, DHCP, DNS, and ARP responder behaviour. Fourth, **liveness bookkeeping**: copy `NB_Global.nb_cfg` into `SB_Global.nb_cfg` after each compile cycle so that the CMS knows which sequence the cluster has caught up to.

### Why it was created

Pre-existing pain before OVN: the Neutron OVS architecture (a Linux bridge agent then an OVS agent) stored state in a distributed manner. Each compute node ran a Python agent in a local reconcile loop using `iptables` and `ovs-ofctl` to apply policy. As a result, every new feature (security group, routing, distributed virtual routing) had to be coded across all agents, debugging was scattered, and convergence latency depended on the order of RPC calls from the Neutron server. OVN solved this by funnelling all compile logic into a single C daemon, leaving the chassis to execute (`ovn-controller` only translates SB into OpenFlow). `ovn-northd` exists to realise the idea "centralise compile, distribute install". An earlier alternative was to put compile logic inside the CMS itself (the Neutron plugin would generate OpenFlow); this was tried and abandoned because it tightly coupled the CMS to the datapath.

### What problem it solves

The technical problem: turn a multi-faceted declarative intent (ACL by Port_Group by Address_Set by LB by NAT by routing, with cross-interaction) into an ordered, conflict-free, semantically correct list of stateful flow rules, mapped precisely onto the `tunnel_key` of each datapath. The CMS alone cannot do this because it does not know the physical encapsulation. `ovn-controller` alone cannot do it because it only sees local state. `ovn-northd` solves the problem globally by holding a complete NB snapshot in memory, running graph-traversal algorithms over the logical topology, and emitting `Logical_Flow` rows with the invariant that every logical packet at any datapath has exactly one deterministic path through the pipeline.

### Importance

In the OVN taxonomy, `ovn-northd` belongs to the absolute core: if it stops writing SB, NB changes never reach the chassis and every newly spawned VM has no connectivity. An engineer operating OVN at production level must understand the northd compile pipeline because most OVN incidents take the form "I changed NB but the dataplane did not react", and the first debug step is always to check `nb_cfg` lag and the northd log. This is a must-know foundational layer, on par with `ovn-controller` (downstream) and `Logical_Flow` (the output schema).

### How it works

The compile cycle (kept short here; the source-code detail lives in §13.8.5 and §13.8.6): the IDL on NB receives a change notification, `inc_proc_northd_run()` triggers the two-node DAG, `en_northd_run()` calls `ovnnb_db_run()` to parse NB into the `northd_data` struct (datapath, ports, parsed ACL), `en_lflow_run()` calls `build_lflows()` to produce `Logical_Flow` rows through `build_*` functions for each pipeline stage, and an IDL transaction commit writes SB DB. Each DAG node has two paths: `change_handler()` for incremental, or `compute()` for full-recompute fallback. The main invariant: SB DB after commit is a pure (deterministic) function of NB DB at the start of the cycle and does not depend on the previous SB state.

The central data structure is `struct northd_data` in `northd/northd.c`, which contains hash maps `datapaths`, `ports`, `lbs`, and pre-parsed `acls` so that `build_lflows()` no longer has to touch the IDL. A per-thread `hmap_local` is used during parallel build, then merged through `fix_flow_map_size()`.

### How an operator masters this

Skills required cover read, modify, observe, and debug. Read: use `ovn-nbctl show`, `ovn-sbctl show`, and `ovn-sbctl lflow-list <datapath>` to map from NB intent to SB output. Modify: in practice you do not modify northd directly. You modify NB through `ovn-nbctl` and then watch the consequences. Observe: `ovs-appctl -t ovn-northd coverage/show` (counters `northd_run`, `lflow_generated`), `stopwatch/show` (per-phase latency), `inc-engine/show` (incremental ratio). Debug: when northd seems stuck, compare `NB_Global.nb_cfg` with `SB_Global.nb_cfg`; if the gap is large, northd is not catching up, often because of a repeated full recompute in a tight loop or an SB OVSDB write conflict. Decision tree: nb_cfg lag, then check `coverage/show`, then if `northd_run` rises quickly check `inc-engine/show` to see whether the engine fell back to full recompute; if so, identify the NB change kind that the change handler does not yet support.

### Classification

Type: long-running C daemon, in the centralised control-plane tier, with an active-standby HA model (one active plus N standbys via SBDB lock leader election). Parent component: the OVN cluster (peer with `ovn-controller`, `ovsdb-server NB`, `ovsdb-server SB`). Subtype: compile-only daemon (no datapath, no user traffic). Lifetime: foreground process by default (`--detach` switches to background), persistent for the lifetime of the cluster.

### Mastery workflow

The standard install-and-operate workflow:

1. Provision the NB and SB clusters (3-node Raft is recommended for production).
2. Start `ovn-northd` on 1 to 3 nodes with `--ovnnb-db=tcp:nb1:6641,nb2,nb3 --ovnsb-db=tcp:sb1:6642,sb2,sb3 --pidfile --detach`.
3. Verify leader election: `ovs-appctl -t ovn-northd nb-connection-status` and `sb-connection-status` should both report `connected` on exactly one node, and other nodes report `waiting` for SB.
4. Observe the baseline: run `coverage/show` every 30 seconds for 5 minutes and record the steady value of `northd_run`.
5. Generate a test workload (`ovn-nbctl ls-add test-ls`) and confirm SB reflects it within 2 seconds.
6. For a large cluster (more than 1000 LSes), enable `--n-threads=N` (N equals number of cores divided by 2) and re-measure `stopwatch/show`.
7. Set up monitoring: alert when `nb_cfg` lag exceeds 10 for more than 60 seconds, and alert when `cancelled cycle` exceeds 2 per hour.

Best practice: always deploy 2 to 3 northd instances to avoid a single point of failure; do NOT run northd on the same node as ovs-vswitchd in a large data centre (CPU contention); separate the NB and SB management network from the dataplane network.

### When something goes wrong, remember this keyword

Symptom A: a newly created VM has no connectivity, `ovn-nbctl show` shows the LSP but `ovn-sbctl show` has no Port_Binding. Suspect: northd is not compiling. Diagnostic: `ovs-appctl -t ovn-northd coverage/show` to see whether `northd_run` increased after the NB create. Symptom B: the cluster suddenly "freezes" for minutes after a bulk operation (for example, deleting a tenant with 5000 LSPs). Suspect: the I-P engine is falling back to full recompute repeatedly. Symptom C: `ovn-nbctl --wait=hv` hangs for 60 seconds and times out. Suspect: northd is not catching up `nb_cfg`. Symptom D: log spam "transaction failed: not connected" followed by recovery. Suspect: leader was lost and a standby is coming up.

Cross-link to Part 20.5 (incident-response decision tree) and Part 13.7 (downstream `ovn-controller`, the consumer of the northd compile output).

### Tightly related to

Tightly coupled with `Logical_Flow` (the output schema; every NB change ultimately emits through it), `Datapath_Binding` (the `tunnel_key` allocator is northd's responsibility), `Port_Binding` (northd creates the row, ovn-controller claims it later), `NB_Global.nb_cfg` and `SB_Global.nb_cfg` (heartbeat sequences). Loosely coupled: `ovn-trace` (consumes SB output and does not depend on northd internals directly), Geneve TLV (chassis-side encapsulation, independent of compile logic).

### Source code reference

OVN 22.03.8 baseline, verified upstream through MCP GitHub:

- `northd/ovn-northd.c`: `main()` daemon entry, parse_options, leader-lock loop. Role: process bootstrap.

### Real-world incident

Reference incident (already discussed in the Part 20.5 case study and the S58 workshop): a production OpenStack OVN cluster with around 5000 tenants, where a batch tenant-delete operation deleted 5000 `Logical_Switch_Port` rows through the Neutron API simultaneously, each LSP a separate NB DB transaction about 50 ms apart.

Observed symptom: `ovn-nbctl --wait=hv ls-add new-tenant ls0` hung for 4 minutes 40 seconds instead of the usual under 2 seconds. During that window, `ovs-appctl -t ovn-northd coverage/show` showed `northd_run` increasing roughly 5000 times with northd CPU at 100 percent.

Investigation flow: step 1, measure `nb_cfg` lag; the gap was 4800 (NB equals 5000, SB equals 200), confirming northd was not catching up. Step 2, `inc-engine/show` showed the `northd` node recompute ratio at 100 percent, proving the I-P engine was falling back to full recompute. Step 3, read the change-handler source code in `northd/inc-proc-northd.c`; at `branch-22.03`, the change handler for `Logical_Switch_Port` delete was not fully implemented (add was implemented, delete was not, in that branch), so each LSP delete triggered a full recompute over the 5000 remaining LSes.

Root cause: a limitation of the I-P engine at the 22.03 baseline. The change handler did not cover the entire NB schema, and some change kinds fell back to a full recompute that is O(N) in the total number of NB objects.

Applied fix: short term, batch the deletes through a single NB DB transaction (`ovn-nbctl --transaction <<EOF ... EOF`) so northd sees one change instead of 5000; medium term, upgrade to OVN 23.09 where the delete change handler has been expanded. After the fix, the same batch-delete workload finished in 8 seconds.

Lesson: the I-P engine is not an automatic guarantee for every pattern. The engineer must know which NB change kinds have incremental support in the running version and batch operations whenever possible.

### Common failures and diagnostic signals

Mode A, hung `--wait=hv`: symptom, `ovn-nbctl --wait=hv` times out at 30 to 300 seconds. Signal: NB `nb_cfg` is rising but SB is frozen. Diagnostic: `coverage/show` to see whether `northd_run` is rising; `inc-engine/show` to read the cancelled rate. Recovery: identify the bulk transaction that caused the stall, restart northd if it is permanently stuck (`systemctl restart ovn-northd`), or force `ovn-appctl -t ovn-northd inc-engine/recompute`.

Mode B, full-recompute storm: symptom, northd CPU is at 100 percent continuously and `northd_run` rises rapidly. Signal: `inc-engine/show` reports `node: northd recompute` above 30 per minute. Diagnostic: identify the NB change kind, cross-check the change-handler source code at the running version. Recovery: short term, batch the transactions; long term, upgrade OVN.

Mode C, leader-election flap: symptom, log spam "lost lock" and "got lock". Signal: `nb-connection-status` flips between nodes. Diagnostic: check the network between the northd instances and the SB cluster, check the SB `ovsdb-server` log for election events. Recovery: stabilise the management network and increase `--inactivity-probe` if latency is high.

Mode D, schema mismatch after partial upgrade: symptom, northd start fails with "schema version mismatch". Signal: `ovn-northd.log` reports "unsupported NB schema X.Y.Z". Recovery: `ovsdb-tool convert nb.db ovn-nb.ovsschema` to upgrade the schema before starting the new daemon.

Mode E, OVSDB SB write conflict: symptom, `cancelled cycle` rises in `inc-engine/show`. Signal: the SB DB Raft cluster has a slow follower or an election in progress. Diagnostic: `ovs-appctl -t ovn-southd cluster/status`. Recovery: stabilise the SB cluster; northd retries automatically and no intervention is required.

### Comparison with other systems

Compared with Cisco APIC (Application Policy Infrastructure Controller): similar in that APIC is the compile centre for the ACI fabric, translating declarative policy into VXLAN encapsulation and class-id on each leaf. Different: APIC is a closed system tightly integrated with Cisco silicon and exposes a vendor-specific REST API; `ovn-northd` is an open-source C daemon and the intermediate is a generic OVSDB schema. APIC uses its own leader election (an internal 3-node cluster); ovn-northd uses the SBDB lock borrowed from ovsdb-server.

Compared with VMware NSX-T Manager: NSX Manager handles policy compilation (Distributed Firewall, Logical Router) into the intermediate "realised state" and pushes it down to the Transport Node. Similar: both centralise compile and distribute realisation. Different: NSX Manager is a Java service stack with PostgreSQL or Corfu as the database and exposes a REST and GraphQL API; ovn-northd is a small C daemon and the intermediate is an OVSDB row.

Compared with the Kubernetes scheduler: similar in the pattern "centralise the decision, distribute the execution" (the scheduler decides where a pod goes and the kubelet enforces it locally). The domain is different (workload placement versus network-policy compile) but the architectural shape is the same.

Compared with Linux iptables and nftables: the opposite. iptables and nftables have no centralised compiler; each node loads its own rules. ovn-northd is precisely the attempt to bring centralisation into a space that is normally distributed.

## 13.8.9 Five `ovn-appctl` commands most often used to debug `ovn-northd`

> This subsection collects five runtime commands that the OVN operator calls into `ovn-northd` (through `ovn-appctl -t ovn-northd ...`) to check daemon state, adjust multi-thread parameters, inspect the I-P engine, and verify SB DB connectivity. Every command name below has been verified through `unixctl_command_register()` in the OVN baseline `branch-22.03` source code (and cross-checked against `branch-22.09` for the two commands introduced after 22.03). When you map these to a real system, you may swap `ovn-appctl` for `ovs-appctl -t ovn-northd`. The two are equivalent at the unix-socket protocol level; they only differ in the default pidfile path.
>
> **A particularly important version note:** in OVN `branch-22.03` (the curriculum baseline), `ovn-northd` registers 9 commands at `northd/ovn-northd.c` (L703 to L752): `exit`, `pause`, `resume`, `is-paused`, `status`, `sb-cluster-state-reset`, `nb-cluster-state-reset`, `nb-connection-status`, and `sb-connection-status`. The two commands `parallel-build/set-n-threads` and `parallel-build/get-n-threads` were introduced in OVN 22.09 (a 2022-09 commit); an engineer running the 22.03 baseline must use the command-line flag `--n-threads=N` at startup instead of runtime tuning. The four `inc-engine/*` commands (including `show-stats`, `clear-stats`, `recompute`, and `compute-log-timeout`) are registered in the shared library at `lib/inc-proc-eng.c` (L185 to L191), so they are present on both `ovn-controller` and `ovn-northd` from 22.03. An earlier internal review listed the name `inc-engine/show` which does not exist in the source code; the correct name is `inc-engine/show-stats`. The content below uses the verified upstream names.

### Command `ovn-appctl -t ovn-northd status`

`status` is the simplest yet most valuable command when you triage a production OVN cluster. The daemon `ovn-northd` registers the command at `northd/ovn-northd.c` L709 with the callback `ovn_northd_status` (L991), which reads the internal flags `state->paused` and `state->had_lock` and returns one of three strings: `Status: active`, `Status: standby`, or `Status: paused`. In an active-standby HA cluster, exactly one instance must report `active` at any moment. If zero or more than one instance reports `active`, the cluster is in split-brain or a leader election is in progress.

Syntax and a real sample:

```
[node1]# ovn-appctl -t ovn-northd status
Status: active

[node2]# ovn-appctl -t ovn-northd status
Status: standby

[node3]# ovn-appctl -t ovn-northd status
Status: standby
```

When the engineer manually pauses northd for a rolling upgrade or maintenance:

```
[node1]# ovn-appctl -t ovn-northd pause
[node1]# ovn-appctl -t ovn-northd status
Status: paused
```

A typical debugging use case: alert "newly spawned VMs have no connectivity". The first step is to ssh into each northd node and run `status`. If they all report `standby`, the SB DB lock has not been claimed by anyone (usually because SB DB Raft is in election). If one node reports `active`, switch to debugging `coverage/show` to see whether northd is actually compiling. The command returns within 10 ms even on a large cluster because it only reads in-memory state; it does not touch the IDL.

### Command `ovn-appctl -t ovn-northd parallel-build/set-n-threads N`

The command `parallel-build/set-n-threads`, introduced in OVN 22.09 (verified at `northd/ovn-northd.c` L777 in branch-22.09), changes the worker pthread count for `build_lflows()` at runtime without restarting the daemon. The callback `ovn_northd_set_thread_count_cmd` (L1091 to L1106) parses argv[1] as an integer, validates that it falls in the range [1, `OVN_MAX_SUPPORTED_THREADS=256`], and calls `set_worker_pool_size()` to update the pool. The engineer uses this command when `stopwatch/show` shows that `build_lflows_ctx` is hitting the warning threshold on a large cluster and the engineer wants to experiment with different thread values without losing the leader lock.

Syntax and a sample:

```
[node1]# ovn-appctl -t ovn-northd parallel-build/set-n-threads 8
[node1]# ovn-appctl -t ovn-northd parallel-build/get-n-threads
8
```

Passing a value outside the allowed range returns a clear error from the daemon:

```
[node1]# ovn-appctl -t ovn-northd parallel-build/set-n-threads 0
"parallel-build/set-n-threads" command failed: invalid n_threads: 0
[node1]# ovn-appctl -t ovn-northd parallel-build/set-n-threads 500
"parallel-build/set-n-threads" command failed: invalid n_threads: 500
```

A real use case: on an OpenStack OVN cluster with 1500 LSes, the default `--n-threads=1` shows `Maximum: 312 ms` in the stopwatch. The engineer raises the count gradually at runtime without a restart and measures the response:

```
[node1]# ovn-appctl -t ovn-northd parallel-build/set-n-threads 4
[node1]# sleep 60 && ovn-appctl -t ovn-northd stopwatch/show | grep -A2 build_lflows
Statistics for 'build_lflows_ctx'
  Maximum: 142 ms
[node1]# ovn-appctl -t ovn-northd parallel-build/set-n-threads 8
[node1]# sleep 60 && ovn-appctl -t ovn-northd stopwatch/show | grep -A2 build_lflows
Statistics for 'build_lflows_ctx'
  Maximum: 89 ms
```

**A note on the 22.03 baseline:** this command does not exist in the curriculum baseline. An engineer running 22.03 must set `--n-threads=N` in the systemd unit file or startup script and restart the daemon for the change to take effect. After upgrading to 22.09 or later, runtime tuning becomes available and is safer because it does not lose the active lock.

### Command `ovn-appctl -t ovn-northd parallel-build/get-n-threads`

The dual command `get-n-threads` (registered at `northd/ovn-northd.c` L780 in branch-22.09 with callback `ovn_northd_get_thread_count_cmd` at L1108 to L1116) simply calls `get_worker_pool_size()` from `lib/ovn-parallel-hmap.c` and prints a single number. The command is useful when the engineer inherits a production cluster with an unclear current configuration, or when confirming that the previous `set-n-threads` was applied correctly.

Syntax and a sample:

```
[node1]# ovn-appctl -t ovn-northd parallel-build/get-n-threads
8
```

Combine with `ps` to verify the actual thread count inside the process:

```
[node1]# ovn-appctl -t ovn-northd parallel-build/get-n-threads
8
[node1]# ps -T -p $(pidof ovn-northd) | wc -l
9
```

The `ps` thread count equals `n-threads + 1` (plus the main thread). If the two numbers do not match, the pool reinit has not finished or a thread crashed. Reset by calling `set-n-threads` with the same value; this forces a clean pool rebuild.

**A troubleshooting use case:** capacity planning on a production cluster scaling from 500 to 2000 LSes. The engineer uses `get-n-threads` to document the current configuration, sets a new value through `set-n-threads`, watches `stopwatch/show` for 30 minutes, and rolls back through `set-n-threads` if there is no improvement. The full flow needs no daemon restart and no downtime for the NB to SB compile pipeline.

### Command `ovn-appctl -t ovn-northd inc-engine/show-stats`

The correct command name is `inc-engine/show-stats`, not `inc-engine/show`. The command is registered at `lib/inc-proc-eng.c` L185 (callback `engine_dump_stats` L127 to L146), which is why it appears automatically on every daemon that uses the I-P engine, including `ovn-controller`, `ovn-northd`, and `ovn-ic`. The output iterates the `engine_nodes[]` array and prints three counters per node: `recompute` (the number of full-compile cycles), `compute` (the number of successful incremental cycles), and `abort` (the number of times a handler returned `false` and forced the engine to fall back to recompute).

Syntax and a real sample on `ovn-northd` baseline 22.03 with the 2-node DAG:

```
[host]# ovn-appctl -t ovn-northd inc-engine/show-stats
Node: northd
- recompute:           42
- compute:             18
- abort:                3
Node: lflow
- recompute:           15
- compute:             45
- abort:                0
```

How to read the output:

- A high `recompute / (recompute + compute)` ratio on the `northd` node (in this example, 42 of 60, around 70 percent) reflects the fact that the I-P engine in baseline 22.03 does not yet have change handlers for many common NB change kinds. Most of them fall back to full recompute. This is the expected behaviour for 22.03, not a bug.
- A fast-rising `abort` is a sign that a change handler has a bug or that an input source is emitting an unfamiliar event. Cross-check the northd log at the same moment for an exception.
- The pair `inc-engine/show-stats` and `inc-engine/clear-stats` lets you measure the delta between two points in time: clear before the experiment, run the workload, and call show-stats to read the clean counts for that workload alone.

A typical forensic use case: the Part 20.5 case study (deleting 5000 LSPs in batch) shows `recompute = 5000, compute = 0, abort = 0` on the `northd` node after the bulk delete. This is clear evidence that the 22.03 I-P engine has no change handler for `Logical_Switch_Port` delete; every delete falls back to a full recompute.

### Command `ovn-appctl -t ovn-northd sb-connection-status`

The command `sb-connection-status` is registered at `northd/ovn-northd.c` L752 with the shared callback `ovn_conn_show` (defined at `lib/ovn-util.c` L36 to L44). The callback is very short: it calls `ovsdb_idl_is_connected(idl)` on the SB DB IDL and returns one of two strings, `connected` or `not connected`. This is a TCP- and OVSDB-handshake-level liveness check on the IDL, not a leader-lock check (which is what `status` shows).

Syntax and a sample:

```
[node1]# ovn-appctl -t ovn-northd sb-connection-status
connected

[node1]# ovn-appctl -t ovn-northd nb-connection-status
connected
```

When the SB DB Raft cluster loses quorum or a network partition cuts northd off from SB:

```
[node1]# ovn-appctl -t ovn-northd sb-connection-status
not connected
```

An important troubleshooting use case: alert "northd is not catching up nb_cfg, lag is 200 over 5 minutes". Triage steps:

```
[node1]# ovn-appctl -t ovn-northd status
Status: active
[node1]# ovn-appctl -t ovn-northd nb-connection-status
connected
[node1]# ovn-appctl -t ovn-northd sb-connection-status
not connected
```

Immediate conclusion: northd can read NB but cannot write SB, so reconcile is slow because it cannot commit transactions. The next step is to inspect the SB DB Raft (`ovs-appctl -t ovnsb_db cluster/status OVN_Southbound`) to determine the cause (election in progress, slow follower, or network down).

**Internal correction:** subsection §13.8.3 in this same part previously stated that "a standby shows `connected` for NB but `waiting` for SB". In reality, the callback `ovn_conn_show` only returns `connected` or `not connected`; it does not return `waiting`. The standby state of waiting on the lock is exposed through the `status` command as `Status: standby`, not through `sb-connection-status`. When the next revision of the curriculum is published, the wording of §13.8.3 will be aligned with the verified behaviour.

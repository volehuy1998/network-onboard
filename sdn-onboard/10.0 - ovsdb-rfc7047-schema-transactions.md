## 10.0.1 OVSDB: ordered, transactional, schema-bound

Compass Ch 4: "OVSDB là ordered, transactional, schema-bound JSON-RPC database."

Ba tính chất cốt lõi:

- **Ordered**: transaction có thứ tự rõ ràng, mỗi commit có sequence number monotonic. Client có thể subscribe và nhận notification theo đúng thứ tự.
- **Transactional**: ACID. Transaction có nhiều operation — tất cả commit hoặc tất cả rollback.
- **Schema-bound**: schema định nghĩa rõ table, column, type, constraint. Write không hợp lệ bị reject.

Protocol: JSON-RPC 1.0 over Unix socket (hoặc TCP/SSL). Mỗi request là JSON object với `method`, `params`, `id`.

## 10.0.3 Column types

Compass Ch 4: cột được typed — strings, integers, reals, booleans, UUIDs, và hai composite types `set<T>` và `map<K,V>`.

**Scalar**: `string`, `integer`, `real`, `boolean`, `uuid`.

**Set<T>**: unordered collection of scalars. Ví dụ `Bridge.protocols` là `set<string>` chứa `["OpenFlow13", "OpenFlow14"]`.

**Map<K,V>**: key-value pairs. `other_config` và `external_ids` — escape hatch cho metadata — đều là `map<string,string>`. Ví dụ MAC aging time là `Bridge.other_config:mac-aging-time`.

**UUID reference**: một cột reference row của table khác qua UUID. Ví dụ `Bridge.ports` là `set<UUID>` pointing đến rows trong `Port`. `ovs-vsctl --id=@name` tạo symbolic reference cho UUID trong transaction.

## 10.0.4 Mười operations RFC 7047

RFC 7047 §4.1:

1. **insert**: thêm row mới.
2. **select**: query row theo condition.
3. **update**: thay thế giá trị column (destructive).
4. **mutate**: modify column in-place (add to set, increment integer).
5. **delete**: xóa row khớp với condition.
6. **wait**: block đến khi condition thỏa mãn.
7. **commit**: commit toàn bộ transaction.
8. **abort**: force rollback.
9. **comment**: attach comment cho audit log.
10. **assert**: check condition, fail nếu không đúng.

Ví dụ JSON-RPC transaction tạo bridge + port:

```json
{
  "method": "transact",
  "params": [
    "Open_vSwitch",
    {"op": "insert", "table": "Bridge", "row": {"name": "br0"}, "uuid-name": "bridge1"},
    {"op": "insert", "table": "Port", "row": {"name": "eth0"}, "uuid-name": "port1"},
    {"op": "insert", "table": "Interface", "row": {"name": "eth0"}, "uuid-name": "iface1"},
    {"op": "mutate", "table": "Open_vSwitch", "where": [],
     "mutations": [["bridges", "insert", ["set", [["named-uuid", "bridge1"]]]]]}
  ],
  "id": 1
}
```

`ovs-vsctl add-br br0 -- add-port br0 eth0` dịch thành transaction tương tự.

## 10.0.5 monitor_cond protocol

OVSDB client subscribe thời gian thực thay vì polling:

**`monitor`** (original RFC 7047): client gửi request với column list cần theo dõi. Server push initial state + incremental update.

**`monitor_cond`** (extension, OVS 2.6+): thêm condition clause — client chỉ nhận update cho row khớp với condition. Quan trọng cho scale lớn (OVN Southbound có hàng trăm nghìn `Port_Binding` — mỗi hypervisor chỉ subscribe row của chính nó).

```bash
[host]# ovsdb-client monitor Open_vSwitch Bridge
# Output stream updates thời gian thực khi có transaction
```

## 10.0.7 RFC 7047 wire protocol byte-by-byte

OVSDB là JSON-RPC 1.0 over Unix socket / TCP / SSL. Mỗi message là 1 JSON object, framed bằng **length-prefixed** trên transport layer.

### Connection establishment

Client mở socket tới ovsdb-server (default `/var/run/openvswitch/db.sock` hoặc `tcp:host:6640`). Sau khi `connect()` thành công, client có thể gửi request ngay không cần handshake riêng. Server đón nhận JSON-RPC request đầu tiên.

Authentication (cho TCP/SSL): SSL với client cert kiểm chứng. Unix socket: filesystem permission của socket file (mặc định 0660 root:openvswitch).

### Wire format JSON-RPC 1.0

JSON-RPC 1.0 (khác 2.0) có 3 message type:

**Request:**

```json
{"method": "<method-name>", "params": [<params-array>], "id": <unique-id>}
```

Ví dụ `list_dbs`:

```json
{"method": "list_dbs", "params": [], "id": 0}
```

**Response (success):**

```json
{"result": <result-data>, "error": null, "id": <matching-id>}
```

Ví dụ:

```json
{"result": ["Open_vSwitch"], "error": null, "id": 0}
```

**Response (error):**

```json
{"result": null, "error": {"error": "<short-name>", "details": "<detail>"}, "id": <matching-id>}
```

**Notification** (server → client, không request-response):

```json
{"method": "update", "params": [<table-updates>], "id": null}
```

`id: null` đánh dấu notification, không cần reply.

### Framing trên TCP/SSL

OVSDB không dùng HTTP framing. Mỗi JSON message là self-delimited (parser stop khi closed brace cân bằng). Client phải có streaming JSON parser handle partial reads.

```
[client → server]: {"method":"list_dbs","params":[],"id":0}\n
[server → client]: {"result":["Open_vSwitch"],"error":null,"id":0}\n
```

`\n` newline optional, không phải framing requirement (nhưng helpful cho debugging với `nc`).

### Echo keepalive

Server và client dùng `echo` method để keepalive:

```json
{"method": "echo", "params": ["arg1", "arg2"], "id": "<id>"}
```

Server reply với cùng params. Mặc định probe interval 5 giây. Nếu không response trong 5 giây × 4 = 20 giây → connection coi như dead, reconnect.

### Test với `nc`

```bash
[host]# nc -U /var/run/openvswitch/db.sock
{"method":"list_dbs","params":[],"id":1}
{"result":["Open_vSwitch"],"id":1,"error":null}
```

Đây là cách đơn giản nhất kiểm chứng ovsdb-server alive ở wire protocol level.

## 10.0.8 Mười operations deep, JSON example đầy đủ

### `insert`

```json
{"op": "insert", "table": "Bridge", "row": {"name": "br0"},
 "uuid-name": "new_bridge"}
```

Params:

- `table`: target table.
- `row`: column values cho row mới. Columns không cung cấp dùng default từ schema.
- `uuid-name` (optional): symbolic name cho UUID, dùng trong `mutate` cùng transaction.

Result: `{"uuid": ["uuid", "<actual-UUID>"]}`.

### `select`

```json
{"op": "select", "table": "Logical_Switch_Port",
 "where": [["type", "==", "localnet"], ["name", "!=", "phys-port"]],
 "columns": ["name", "addresses"]}
```

`where` là array of conditions (AND'd). Mỗi condition: `[<column>, <function>, <value>]`. Functions: `==`, `!=`, `<`, `<=`, `>=`, `>`, `includes`, `excludes`.

Result: `{"rows": [{"name": "...", "addresses": [...]}, ...]}`.

### `update`

```json
{"op": "update", "table": "Bridge", "where": [["name", "==", "br0"]],
 "row": {"fail_mode": "secure"}}
```

**Destructive replace**: column values trong `row` overwrite hoàn toàn. Cho composite types (set/map), giá trị mới thay thế giá trị cũ.

Result: `{"count": 1}` (số row affected).

### `mutate`

```json
{"op": "mutate", "table": "Bridge", "where": [["name", "==", "br0"]],
 "mutations": [["ports", "insert", ["set", [["named-uuid", "new_port"]]]]]}
```

**Non-destructive in-place modify**: add to set, remove from set, increment integer, multiply.

Mutator types: `+=` (add to set / increment), `-=` (subtract / remove), `*=`, `/=`, `%=`, `insert`, `delete`.

Result: `{"count": 1}`.

### `delete`

```json
{"op": "delete", "table": "Bridge", "where": [["name", "==", "br0"]]}
```

Garbage collection: nếu row khác có UUID reference tới row vừa xóa, OVSDB tự động set reference = empty (`set<UUID>` size 0). Không cascade delete.

Result: `{"count": 1}`.

### `wait`

```json
{"op": "wait", "table": "Bridge", "where": [["name", "==", "br0"]],
 "columns": ["fail_mode"], "until": "==",
 "rows": [{"fail_mode": "secure"}], "timeout": 5000}
```

Block transaction cho đến khi `where` match `until` rows, hoặc timeout (ms). Dùng cho atomicity assertion: "transaction này chỉ commit nếu state thỏa mãn".

### `commit`

```json
{"op": "commit", "durable": true}
```

`durable=true`: fsync log file trước khi reply. Slow nhưng safe.
`durable=false`: reply ngay, log eventually flushed. Fast nhưng có thể mất data nếu crash.

Default OVN: `durable=true` cho NB, `durable=false` cho SB (write-heavy).

### `abort`

```json
{"op": "abort"}
```

Force rollback toàn bộ transaction. Hữu ích khi pre-condition fail.

### `comment`

```json
{"op": "comment", "comment": "Add bridge br0 by user admin@example.com"}
```

Attach text vào audit log. Không ảnh hưởng data.

### `assert`

```json
{"op": "assert", "lock": "ovn_northd"}
```

Kiểm chứng lock held bởi current connection. Fail nếu mất lock (ví dụ: bị steal). Dùng cho leader election semantics.

## 10.0.9 monitor + monitor_cond + monitor_cond_since

### `monitor` (RFC 7047 §4.1.5)

```json
{"method": "monitor", "params": ["Open_vSwitch", "monitor-handle",
   {"Bridge": [{"columns": ["name", "ports"]}]}], "id": 1}
```

Server push:

1. **Initial state**: tất cả existing rows.
2. **Incremental update notification**: mỗi khi có change.

Client phải xử lý cả 2 cùng cách.

### `monitor_cond` (OVS 2.6+, 2016)

Conditional filter:

```json
{"method": "monitor_cond", "params": ["OVN_Southbound", "handle",
   {"Port_Binding": [
     {"columns": ["chassis", "logical_port"],
      "where": [["chassis", "==", ["uuid", "<my-chassis-uuid>"]]]}
   ]}], "id": 2}
```

Chassis chỉ subscribe Port_Binding của chính mình → bandwidth giảm 50-100x trên cluster lớn (10K port chia 100 chassis = 100 row/chassis thay vì 10K).

### `monitor_cond_since` (OVS 2.13+, 2020)

Replay incremental từ last known state:

```json
{"method": "monitor_cond_since", "params": ["OVN_Southbound", "handle",
   {"Port_Binding": [...]}, "<last_id>"], "id": 3}
```

`last_id` là transaction ID cuối client đã xử lý. Server gửi tất cả update từ `last_id` đến hiện tại + initial state cho row mới. Client reconnect không cần re-process từ đầu.

Implementation: server giữ ring buffer các transaction recent (~1000). Nếu `last_id` quá cũ (out of buffer) → server fallback gửi full snapshot.

OVN 22.06+ exploit `monitor_cond_since` để giảm load server khi ovn-controller reconnect (ví dụ sau brief network blip).

## 10.0.10 IDL synchronization model

OVSDB client thường không call JSON-RPC trực tiếp mà qua IDL (Interface Definition Language) library. IDL build in-memory replica của tables, sync với server qua `monitor`/`monitor_cond`.

### Workflow

1. Client gọi `ovsdb_idl_create()` với schema + database name.
2. IDL thực hiện `monitor` request, pull initial state vào in-memory hash table.
3. Client `ovsdb_idl_run()` periodically poll for updates from server (qua `recv()` socket).
4. Client read state synchronously: `ovsrec_bridge_first(idl)`, traverse linked list rows.
5. Client modify: build transaction in-memory, gọi `ovsdb_idl_txn_commit_block()` to flush.
6. Server reply, IDL update local replica.

### Sequence number `nb_cfg`

OVN dùng `NB_Global.nb_cfg` (incrementing integer) để measure NB→SB sync lag:

- CMS commit transaction NB → `nb_cfg++`.
- ovn-northd compile NB → SB, copy `nb_cfg` value vào `SB_Global.nb_cfg`.
- ovn-controller per-chassis copy vào `Chassis_Private.nb_cfg`.

Lag measurement:

```bash
[host]# ovn-nbctl get NB_Global . nb_cfg
142
[host]# ovn-sbctl get SB_Global . nb_cfg
138
[host]# ovn-sbctl get Chassis_Private chassis-A nb_cfg
136
```

Healthy: NB - SB ≤ 5, SB - Chassis ≤ 5. Critical: lag > 100 (compile bottleneck or cluster congestion).

### Connection failure recovery

IDL library có 3 reconnect strategies:

- **passive_reconnect**: chờ application call `run()` re-trigger reconnect.
- **active_reconnect** (default): library tự reconnect với backoff (1s → 2s → 4s → 8s → 16s capped).
- **bounded_recovery** (config): cap total reconnect time, throw error sau threshold.

Sau reconnect, IDL ưu tiên `monitor_cond_since` với last `last_id` đã sync, nếu fail (server out of buffer) fallback `monitor_cond` initial.

## 10.0.11 Schema evolution + needs-conversion flow

OVS/OVN upgrade có thể đổi schema (thêm column, thêm table). Process:

### Backward-compatible change

Schema mới với column/table mới (default value cho row cũ). ovsdb-server tự handle: row cũ get default, row mới get user-provided value. Không cần migration.

### Major schema change

Schema mới rename column hoặc khác type → server cần migrate data. Workflow:

```bash
# Step 1: Compare current vs new schema
[host]# ovsdb-client needs-conversion <db> <new-schema-file>
yes

# Step 2: Stop server (or signal for live convert)
[host]# systemctl stop ovsdb-server

# Step 3: Convert offline
[host]# ovsdb-tool convert <db-file> <new-schema-file>

# Step 4: Restart
[host]# systemctl start ovsdb-server
```

Hoặc live convert (không downtime, OVS 2.5+):

```bash
[host]# ovsdb-client convert tcp:server:6640 OVN_Northbound /usr/share/ovn/ovn-nb.ovsschema
```

OVS Raft cluster: convert atomic across cluster, leader broadcast schema → followers apply trong cùng transaction. Cluster-wide rolling upgrade vẫn cần coordination (xem Phần 9.12 + 10.6).

## 10.0.12 Anatomy Template A: `ovsdb-client get-schema` JSON

> **Bối cảnh khi chạy:** Kiểm tra schema để hiểu structure DB hoặc generate code (Python IDL stub).

```json
{
  "name": "Open_vSwitch",
  "version": "8.4.0",
  "cksum": "1234567890 12345",
  "tables": {
    "Open_vSwitch": {
      "columns": {
        "bridges": {"type": {"key": {"type": "uuid", "refTable": "Bridge"},
                              "min": 0, "max": "unlimited"}},
        ...
      },
      "isRoot": true,
      "maxRows": 1
    },
    "Bridge": {...},
    ...
  }
}
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| `name` | Match expected DB | Khác (multi-DB server) | Empty | DB không exist; check `list-dbs` |
| `version` | Match baseline | Mismatch (sau upgrade) | Khác major | Schema drift; `needs-conversion` |
| `cksum` | Stable | Đổi sau upgrade | Đổi không kế hoạch | Schema patched manual |
| `tables` count | Match expected | Mất 1-2 table | Mất nhiều | Schema corrupt |
| `isRoot` flag | True cho 1 table | True cho > 1 | False cho mọi table | Schema invalid |
| Column `type.key.type` | scalar / uuid | Custom type | Empty | Schema parse error |
| Column `type.refTable` | Valid table name | Reference table không tồn tại | Empty cho UUID column | Schema invalid |

**Kịch bản bẻ gãy:**

- `cksum` đổi không kế hoạch: ai đó modify schema bằng tay. Run `ovsdb-tool show-log <db-file>` để xem lịch sử transaction; restore từ backup.
- `version` mismatch giữa NB cluster + SB cluster: northd compile fail. Run `convert` trên cluster lag.
- Column reference table empty: schema corrupt. Restore.

## 10.0.13 ▶ Capstone POE: monitor vs monitor_cond cho 100K row table

> **Bối cảnh:** Cluster OVN với SB DB chứa 100K Port_Binding row. 100 chassis, mỗi chassis cần subscribe state. Kỹ sư đề xuất "dùng monitor đơn giản, không cần monitor_cond".

**Tuyên bố cần kiểm chứng:** "monitor đủ cho 100K row, monitor_cond chỉ là optimization không cần thiết."

**Predict:** Sai. Bandwidth + memory bùng nổ trên cluster lớn.

**Observe (giả định):**

- 1 chassis dùng `monitor` cho 100K Port_Binding: initial state ~50MB JSON gửi qua socket. Mỗi update notification gửi cho mọi chassis (notification kích thước nhỏ, nhưng N=100).
- 100 chassis × 50MB = 5GB initial sync trên cluster startup.
- Update rate: 100 transaction/s typical. Mỗi transaction broadcast tới 100 chassis. Server CPU spike khi 100 client cùng pull.
- Memory ovsdb-server: server giữ "monitor session" per client với column subset. 100 monitor sessions × 100K row = 10M tracking entries.

Với `monitor_cond` (chỉ subscribe row liên quan chassis):

- 1 chassis subscribe ~1000 Port_Binding (chỉ binding tới chassis đó). Initial 500KB.
- 100 chassis × 500KB = 50MB. Giảm 100x.
- Update broadcast chỉ tới chassis quan tâm (server filter ở `where` clause). Most update chỉ gửi 1-2 chassis thay vì 100.

**Explain:**

1. **`monitor` không scale**: O(N×M) bandwidth (N client × M row). M cao → bandwidth bùng nổ.
2. **`monitor_cond`**: O(N × M_filtered) với `M_filtered` typically < M / 100. Bandwidth + CPU + memory giảm 50-100x.
3. **OVN intentionally** dùng `monitor_cond` mặc định cho ovn-controller. Đó là lý do OVN scale tới 1000 chassis.
4. **Khi nào `monitor` đủ**: small DB (< 1K row) hoặc client chỉ 1-2 (test/debug). Cluster production luôn `monitor_cond`.

**Falsification test:**

- Setup OVN cluster 10 chassis, 10K Port_Binding.
- Disable `monitor_cond` (force `monitor`): đo bandwidth + memory + CPU server.
- Enable `monitor_cond`: đo lại.
- Nếu metric tương đương → counter-example. Trong thực tế, `monitor_cond` luôn ưu thế trên cluster > 5 chassis.

**Tuyên bố sai ở đâu:** "Optimization không cần thiết" giả định scale nhỏ. Đúng phải: **`monitor_cond` là design requirement cho cluster lớn, không phải optimization. `monitor` chỉ phù hợp test/debug nhỏ.**

> **Quy tắc:** Cluster production OVN > 5 chassis hoặc DB > 1K row = monitor_cond bắt buộc. monitor chỉ cho ad-hoc inspection.

## 10.0.14 Năm chủ đề OVSDB observability + operations

Phần này đào sâu năm khái niệm cốt lõi: mô hình transaction RFC 7047, cơ chế đẩy thông báo `monitor` của JSON-RPC, mở rộng `monitor_cond` cho subscription có điều kiện, hệ thống role của ovsdb-server cho phân quyền, và CLI `ovsdb-client` làm cửa quan sát low-level.

### 10.0.14.1 Mô hình transaction OVSDB (`OVSDB transaction model`)

**Khái niệm.** Mô hình transaction OVSDB là tập hợp quy tắc cho phép gom nhiều operation (insert, select, update, mutate, delete, wait, commit, abort, comment, assert) vào một đơn vị atomic gửi tới ovsdb-server qua JSON-RPC method `transact`. Cả nhóm operation hoặc đều thành công và làm thay đổi DB hoặc tất cả bị rollback giữ DB nguyên trạng. Đây là tính chất "all-or-nothing" cốt lõi của ACID atomicity, ánh xạ vào cấu trúc message RFC 7047 §4.1.3.

**Lịch sử + bối cảnh.** RFC 7047 phát hành tháng 12/2013 đặc tả lại behavior của ovsdb-server đã có từ OVS 1.x (2009-2010). Nicira ban đầu xây OVSDB như database tối thiểu cho `ovs-vswitchd` config; cộng đồng IETF chuẩn hoá để bên thứ ba (NSX, OpenStack Neutron) tích hợp. Trước RFC, mỗi vendor implement client riêng dựa vào source code OVS; sau RFC tiêu chuẩn rõ ràng.

**Vị trí trong kiến trúc.** Transaction model nằm ở tầng protocol (cùng tầng JSON-RPC), trên storage layer (file ovsdb log) và dưới tầng IDL/CLI. Server xử lý transaction trong `ovsdb_execute_compose` (file `ovsdb/execution.c`, OVS branch-2.17), gọi từ `ovsdb_jsonrpc_session_got_request` khi nhận `transact` message.

**Vai trò + vấn đề giải quyết.** Vai trò: cung cấp API atomic cho client thay đổi DB nhất quán. Vấn đề giải quyết: tránh "nửa cấu hình" (Bridge tạo ra nhưng Port chưa link), tránh race condition khi nhiều client đồng thời ghi (mỗi transaction tuần tự, server giữ lock cho phần hiện thực), và cho phép precondition check qua `wait`/`assert`.

**Cơ chế hoạt động.** Server nhận message `{"method":"transact","params":[<db>, <op1>, <op2>, ...]}`. Mỗi operation thực thi tuần tự trên snapshot in-memory của DB. Nếu bất kỳ operation báo lỗi (constraint violation, type mismatch, condition not met), server rollback toàn bộ và reply `{"error": ...}`. Nếu tất cả thành công, server append log file (durable) hoặc memory-only (non-durable theo `commit.durable`), apply lên DB chính, broadcast notification cho monitor sessions, reply `{"result":[<r1>, <r2>, ...]}` cho client.

**Quy trình sử dụng.** Client typical: build array operation in-memory (ví dụ `ovs-vsctl` qua IDL), gọi `ovsdb-client transact '<json>'` hoặc `ovsdb_idl_txn_commit`. Đợi reply. Nếu lỗi, parse `error` field, log và retry hoặc abort. Trong shell:

```bash
[host]# ovsdb-client transact unix:/var/run/openvswitch/db.sock \
  '["Open_vSwitch",
    {"op":"insert","table":"Bridge","row":{"name":"br-test"},"uuid-name":"b"},
    {"op":"mutate","table":"Open_vSwitch","where":[],
     "mutations":[["bridges","insert",["set",[["named-uuid","b"]]]]]}]'
```

**Cấu trúc reply tương ứng.** Server reply một array `result` cùng số phần tử với operation array, mỗi phần tử là object: `{"uuid":[...]}` cho insert, `{"count":N}` cho update/mutate/delete, `{"rows":[...]}` cho select, hoặc `{}` cho commit/abort. Nếu một operation lỗi giữa chừng, phần tử đó là `{"error":"...","details":"..."}` và các phần tử sau là `null`; client phải kiểm tra từng phần tử thay vì giả định success-only.

**Khi xảy ra sự cố.** Triệu chứng thường gặp: transaction hang (server quá tải Raft commit), reply `error: "constraint violation"` (thiếu reference UUID), reply `error: "referential integrity violation"` (xóa Bridge còn Port). Diagnostic: `ovs-appctl -t ovsdb-server coverage/show | grep txn`, kiểm tra `transaction/commit` counter; xem log `/var/log/openvswitch/ovsdb-server.log` để biết exact error.

**Tầm quan trọng.** Mọi tool trên CLI (ovs-vsctl, ovn-nbctl, ovn-sbctl) cuối cùng đều build một transaction model object rồi gửi qua method `transact`. Hiểu rõ atomic boundary cho phép kỹ sư tổ hợp lệnh `ovs-vsctl -- ... -- ...` thành một giao dịch duy nhất, tránh lỗi half-applied khi dùng nhiều lệnh tách rời.

**Khác biệt giữa các phiên bản.** OVS 2.5 thêm `wait` operation; OVS 2.7 thêm cluster mode (Raft) khiến durable=true mặc định cho NB. Pre-2017 server chỉ standalone, transaction commit O(disk fsync). Sau Raft, commit O(majority replication) thường nhanh hơn vì pipeline.

**Source code tham chiếu.** `ovsdb/execution.c` hàm `ovsdb_execute_compose` + `ovsdb_execute`; `ovsdb/transaction.c` hàm `ovsdb_txn_propose_commit_block`; OVS branch-2.17.9.

**So sánh với hệ khác.** PostgreSQL transaction có BEGIN/COMMIT/ROLLBACK explicit, nested savepoint, multi-statement. OVSDB transaction luôn single-shot, không nested savepoint; bù lại JSON-RPC framing đơn giản hơn ODBC/protobuf, latency thấp khi DB nhỏ.

### 10.0.14.2 JSON-RPC monitor (`JSON-RPC monitor`)

**Khái niệm.** `monitor` là method JSON-RPC do client gửi để đăng ký nhận thông báo đẩy (server-initiated push) khi nội dung table thay đổi. Khác với polling tuần hoàn, server giữ session dài và đẩy `update` notification ngay khi commit transaction tác động đến row trong subscription.

**Lịch sử + bối cảnh.** RFC 7047 §4.1.5 đặc tả `monitor` cùng họ với `monitor_cancel`. OVS 2.6 (2016) bổ sung `monitor_cond` cho conditional filter. OVS 2.12 (2019) bổ sung `monitor_cond_since` cho replay. Mỗi mở rộng giải quyết một bottleneck mới phát sinh khi cluster OVN scale lên hàng nghìn chassis.

**Vị trí trong kiến trúc.** Method này thuộc tầng JSON-RPC server. Server giữ struct `ovsdb_jsonrpc_monitor` per-session, đăng ký callback vào hệ thống transaction để được gọi sau mỗi commit. IDL client (Phần 10.4) là consumer chính.

**Vai trò + vấn đề giải quyết.** Vai trò: thay polling bằng push để giảm độ trễ và tải cho server. Vấn đề giải quyết: trước đây client phải `select *` mỗi vài giây để dò thay đổi (tốn CPU + bandwidth + miss event ngắn hạn); với monitor, client biết chính xác thời điểm và nội dung thay đổi.

**Tầm quan trọng.** Nền tảng cho mọi daemon OVS/OVN: ovn-controller theo dõi SB, ovn-northd theo dõi NB, ovs-vswitchd theo dõi `Open_vSwitch` DB, neutron-server theo dõi NB qua IDL Python. Không có monitor, kiến trúc event-driven của OVN không khả thi.

**Cơ chế hoạt động.** Client gửi:

```json
{"method":"monitor",
 "params":["Open_vSwitch","monitor-handle-1",
           {"Bridge":[{"columns":["name","ports","fail_mode"]}]}],
 "id":1}
```

Server reply ngay với snapshot toàn bộ row hiện có:

```json
{"result":{"Bridge":{"<uuid1>":{"new":{"name":"br0","ports":[...],"fail_mode":"secure"}}}},"id":1,"error":null}
```

Sau đó mỗi khi có transaction tác động đến `Bridge`, server đẩy notification (lưu ý `id: null` đánh dấu notification, không cần reply):

```json
{"method":"update",
 "params":["monitor-handle-1",
           {"Bridge":{"<uuid1>":{"old":{"fail_mode":"secure"},"new":{"fail_mode":"standalone"}}}}],
 "id":null}
```

**Phân loại `update`/`update2`/`update3`.** `update` (RFC 7047 cơ bản, gửi cả old+new), `update2` (OVS 2.6+, gửi chỉ delta nhỏ hơn), `update3` (OVS 2.12+, kèm last_id cho monitor_cond_since).

**Khi xảy ra sự cố.** Triệu chứng: client không nhận event mặc dù DB đổi (xác minh `monitor-handle` match, xem `ovs-appctl -t ovsdb-server cluster/status`); session bị reset (kiểm tra echo keepalive, thường probe 5 giây); server đẩy duplicate event khi reconnect (giải pháp: dùng `monitor_cond_since`).

**Source code tham chiếu.** `ovsdb/jsonrpc-server.c` hàm `ovsdb_jsonrpc_monitor_create` + `ovsdb_jsonrpc_monitor_compose_update`; `ovsdb/monitor.c` hàm `ovsdb_monitor_changes_classify`; OVS branch-2.17.9.

**So sánh với hệ khác.** etcd `Watch` API streaming key prefix (gRPC); ZooKeeper `getChildren(watch=true)` one-shot watch (phải re-register). OVSDB monitor session-bound liên tục, không cần re-register sau mỗi event, gần với etcd hơn ZooKeeper.

### 10.0.14.3 Conditional replication (`monitor_cond`)

**Khái niệm.** `monitor_cond` là phần mở rộng của `monitor` cho phép client đính kèm mệnh đề `where` (predicate) vào subscription. Server chỉ đẩy notification cho row thoả mệnh đề; row không khớp được "ẩn" đối với client. Quan trọng cho cluster lớn nơi mỗi client chỉ quan tâm tập con.

**Lịch sử + bối cảnh.** OVS 2.6 (2016) bổ sung `monitor_cond` để giải quyết scale của OVN Southbound. Trước đó, mỗi ovn-controller phải subscribe toàn bộ `Port_Binding` (10K+ row trên cluster lớn), gây bandwidth O(N²) (N chassis × N port). Với monitor_cond, mỗi controller chỉ subscribe row liên quan đến chassis của mình, đưa về O(N).

**Vị trí trong kiến trúc.** Cùng tầng JSON-RPC như `monitor`. Server thực thi predicate trong hot-path commit (sau khi transaction apply, trước khi compose update). IDL client gọi `ovsdb_idl_set_condition()` để đặt/đổi condition runtime.

**Vai trò + vấn đề giải quyết.** Vai trò: filter ở phía server, giảm bandwidth + CPU client. Vấn đề giải quyết: tránh "fan-out storm" khi 1 transaction cập nhật 1 row mà server phải broadcast cho 1000 client không quan tâm. Predicate evaluation có chi phí, nhưng ít hơn nhiều so với bandwidth + JSON parse client-side.

**Cơ chế hoạt động.** Client gửi request kèm condition:

```json
{"method":"monitor_cond",
 "params":["OVN_Southbound","sbdb-handle",
           {"Port_Binding":[
             {"columns":["chassis","logical_port","tunnel_key"],
              "where":[["chassis","==",["uuid","aaaa-bbbb-cccc-dddd"]]]}]}],
 "id":2}
```

Server compile predicate thành internal filter (cấu trúc `ovsdb_condition`), gắn vào monitor session. Mỗi commit, server lấy danh sách row tác động, evaluate predicate, chỉ đẩy notification cho row pass.

**Predicate được hỗ trợ.** Format `[<column>, <function>, <value>]` với function `==`, `!=`, `<`, `<=`, `>=`, `>`, `includes`, `excludes`. Nhiều predicate trong cùng `where` array là AND. Predicate trên column type `set<T>`: `includes` test phần tử có trong set.

**Quy trình thay đổi condition runtime.** Client gọi `monitor_cond_change` với handle cũ + new condition; server diff cũ-mới, đẩy `update` cho row vừa pass (initial state) hoặc ẩn (delete-style update) cho row vừa fail. Không cần re-register session, không mất state.

**Khi xảy ra sự cố.** Predicate sai cú pháp: server reply error ngay khi monitor_cond gửi. Client thấy "miss" row mong đợi: thường vì predicate quá hẹp; debug bằng `ovsdb-client monitor-cond <db> <table> '[]'` (predicate rỗng) so sánh.

**Khác biệt giữa các phiên bản.** OVS 2.6 ra mắt; OVS 2.12 thêm `monitor_cond_since` (replay từ last_txn_id, hữu ích sau reconnect ngắn). OVN 22.06 ưu tiên `monitor_cond_since` để giảm CPU SBDB sau network blip.

**Source code tham chiếu.** `ovsdb/monitor.c` hàm `ovsdb_monitor_table_condition_create` + `ovsdb_monitor_change_condition`; `lib/ovsdb-idl.c` hàm `ovsdb_idl_set_condition` + `ovsdb_idl_db_compose_cond_change`; OVS branch-2.17.9.

**So sánh với hệ khác.** PostgreSQL logical replication có publication filter ở row level (PG10+), tương tự monitor_cond về ý tưởng. Kafka Consumer Group filter ở client. OVSDB chọn server-side filter để tiết kiệm bandwidth network DB ↔ client (thường WAN cho cluster đa region).

### 10.0.14.4 ovsdb-server roles (`ovsdb-server roles`)

**Khái niệm.** Role là tập permission gán cho client (định danh qua CN của TLS cert hoặc tên trong cột `Manager.role`) quyết định client được đọc/ghi table và column nào. Role là cốt lõi của RBAC OVSDB, áp dụng tại tầng JSON-RPC trước khi transaction được thực thi.

**Lịch sử + bối cảnh.** OVS 2.7 (2017) thêm RBAC cho OVN multi-tenant: ovn-controller per-chassis chỉ ghi được record của chính nó vào SBDB, ngăn chassis A giả mạo binding của chassis B. Trước đó, mọi client có TLS cert hợp lệ đều có quyền ghi toàn DB (privilege escalation risk).

**Vị trí trong kiến trúc.** RBAC engine nằm trong ovsdb-server, giữa JSON-RPC parser và transaction executor. Role được định nghĩa trong table đặc biệt `RBAC_Role` + `RBAC_Permission` của DB (ví dụ OVN SBDB). Tham chiếu role qua cột `Manager.role` (kết nối inbound) hoặc TLS CN match (kết nối outbound).

**Vai trò + vấn đề giải quyết.** Vai trò: phân quyền chi tiết theo identity. Vấn đề giải quyết: ngăn rogue chassis ghi đè record của chassis khác, ngăn rogue admin ghi vào table không thuộc trách nhiệm, đáp ứng compliance multi-tenant.

**Phân loại role chuẩn.** OVN SBDB có 2 role chính: `ovn-controller` (cho per-chassis daemon, được phép ghi `Chassis_Private`, `Port_Binding` filter theo chassis_uuid match identity), `ovn-controller-vtep` (cho gateway). Custom role có thể define thêm cho multi-tenant.

**Cơ chế hoạt động.** Khi client gửi `transact`, server lookup role gán cho session (qua `Manager.role` cột hoặc TLS CN). Với mỗi operation trong transaction, server check `RBAC_Permission` row matching role + table:

- `authorization` column: regex hoặc literal kiểm tra column trong row mới (ví dụ `name`) có khớp identity client không. Match → cho phép, miss → reject.
- `insert_delete` column: bool cho phép insert/delete row.
- `update` column: list column mà role này được modify.

**Quy trình cấu hình.** Tạo role + permission qua transaction:

```bash
[host]# ovn-sbctl -- --id=@perm create RBAC_Permission \
    table=Chassis_Private authorization='name' insert_delete=true \
    update='["nb_cfg","nb_cfg_timestamp","external_ids"]' \
  -- --id=@role create RBAC_Role name=ovn-controller \
    permissions:Chassis_Private=@perm
```

Sau đó khai báo `Manager` row trỏ role:

```bash
[host]# ovn-sbctl set-connection role=ovn-controller pssl:6642
```

**Khi xảy ra sự cố.** Symptom: client báo `error: "RBAC rules for role 'X' do not allow modification of column Y"`. Diagnostic: `ovn-sbctl list RBAC_Role`, `ovn-sbctl list RBAC_Permission`, đối chiếu CN cert với cột `authorization`. Common pitfall: cert CN chứa hostname đầy đủ trong khi role expect short name — fix bằng cách chuẩn hoá CN khi tạo cert.

**Khác biệt giữa các phiên bản.** OVS 2.7 ra mắt; OVS 2.13 cải thiện performance (cache rule lookup); cấu hình hiện tại OVN 22.03+ có sẵn role `ovn-controller` mặc định. Trước 2.7 chỉ có authentication (TLS), không authorization.

**Source code tham chiếu.** `ovsdb/rbac.c` hàm `ovsdb_rbac_insert_check` + `ovsdb_rbac_update_check`; `ovsdb/jsonrpc-server.c` hàm `ovsdb_jsonrpc_session_set_options` (nơi role gán cho session); OVS branch-2.17.9.

**So sánh với hệ khác.** PostgreSQL có ROLE + GRANT/REVOKE truyền thống (table-level, column-level). MySQL có user-host pair + privilege bitmask. OVSDB RBAC nhẹ hơn nhưng đặc trưng cho mô hình "self-write only" (client chỉ được ghi row identity của chính mình), điều khó biểu diễn thuần GRANT trong SQL.

### 10.0.14.5 ovsdb-client (`ovsdb-client`)

**Khái niệm.** `ovsdb-client(1)` là CLI tổng hợp tương tác trực tiếp với ovsdb-server qua JSON-RPC. Khác với `ovs-vsctl` (chuyên cho `vswitch.ovsschema`) hoặc `ovn-nbctl/sbctl` (chuyên cho NB/SB), `ovsdb-client` là tool generic chạy được trên bất kỳ schema nào: list-dbs, get-schema, dump, monitor, transact, lock, wait, backup, restore, convert.

**Lịch sử + bối cảnh.** Có từ OVS 1.x cùng với ovsdb-server. Nicira ban đầu dùng `ovsdb-client` cho debug trước khi `ovs-vsctl` được viết để thuận tiện cho operator. Từ đó `ovsdb-client` vẫn là tool low-level nhất, được dùng trong test framework, script automation, và troubleshooting khi `ovs-vsctl` fail.

**Vị trí trong kiến trúc.** Nằm ở tầng tool, dùng IDL/JSON-RPC client library `lib/jsonrpc.c` + `lib/ovsdb-idl.c` của OVS. Kết nối qua Unix socket, TCP, hoặc SSL tới ovsdb-server. Là alternative cho IDL Python/C++ binding.

**Vai trò + vấn đề giải quyết.** Vai trò: cửa quan sát + thao tác low-level. Vấn đề giải quyết: khi `ovs-vsctl` báo lỗi không rõ, dùng `ovsdb-client` xem raw schema/data để debug; khi viết script gọn cần monitor change cho schema custom (OVN, custom DB), khỏi viết Python IDL; khi backup/restore DB không có ovsdb-tool offline (server đang chạy).

**Phân loại lệnh chính.**

- `list-dbs`: liệt kê DB phục vụ trên server.
- `get-schema <db>`: dump schema JSON.
- `list-tables <db>`: liệt kê table.
- `dump <db> [<table>]`: dump nội dung.
- `monitor <db> <table> [<columns>]`: subscribe push (gọi method `monitor`).
- `monitor-cond <db> <table> '<condition>' [<columns>]`: subscribe có điều kiện.
- `transact <db> '<operations>'`: gửi transaction tuỳ ý.
- `lock <db> <lock-name>` + `unlock`: leader election primitive.
- `backup`: dump DB sang stdout (snapshot point-in-time).
- `restore`: load DB từ stdin (server phải empty).
- `convert <db> <new-schema>`: live schema upgrade.
- `wait`: block đến khi DB đạt trạng thái mong đợi.

**Quy trình sử dụng.**

```bash
# Liệt kê DB
[host]# ovsdb-client list-dbs unix:/var/run/openvswitch/db.sock
Open_vSwitch

# Dump bridge cụ thể
[host]# ovsdb-client dump unix:/var/run/openvswitch/db.sock Open_vSwitch Bridge name fail_mode

# Subscribe có điều kiện
[host]# ovsdb-client monitor-cond Open_vSwitch Bridge \
    '["Bridge",{"where":[["name","==","br0"]]}]' name,fail_mode

# Backup
[host]# ovsdb-client backup unix:/var/run/openvswitch/db.sock > /backup/conf-$(date +%F).db
```

**Khi xảy ra sự cố.** `connection refused` → server chưa chạy hoặc socket sai path; `database not found` → tên DB sai (case-sensitive, ví dụ `Open_vSwitch` chứ không phải `open_vswitch`); `transaction error` → operation sai schema, kiểm tra với `get-schema` trước.

**Khác biệt giữa các phiên bản.** OVS 2.6 thêm `monitor-cond`. OVS 2.12 thêm `monitor-cond-since`. OVS 2.13 thêm `wait` subcommand. OVN 22.03+ khuyến nghị `ovsdb-client backup` thay cho `ovsdb-tool` offline khi cluster Raft.

**Cách quan sát + xác minh.** Dùng `--verbose=jsonrpc:dbg` để in raw JSON-RPC qua stderr — hữu ích khi debug tool phía thứ ba. So sánh output `dump` trước/sau transaction để verify hiệu lực thay đổi.

**Source code tham chiếu.** `ovsdb/ovsdb-client.c` (entry điểm CLI); `lib/jsonrpc.c` (transport); `lib/ovsdb-idl.c` (high-level binding cho monitor); OVS branch-2.17.9.

**So sánh với hệ khác.** `psql` của PostgreSQL có meta-command + SQL; `etcdctl` của etcd có get/put/watch; `redis-cli` của Redis có MONITOR. `ovsdb-client` giống `etcdctl` nhất ở model watch + lock + transact, nhưng thao tác qua JSON-RPC thay vì gRPC nên dễ debug bằng `nc` + tay.

## Điểm cốt lõi cần nhớ

Thứ nhất, OVSDB là database specialized cho network config — ACID + thời gian thực subscription đủ cho SDN use case.

Thứ hai, schema-bound design giúp `ovs-vsctl` kiểm tra tính hợp lệ của giao dịch (validate transaction) trước khi commit, tránh half-configured state.

Thứ ba, `monitor_cond` là cơ chế quan trọng cho scale lớn — OVN Southbound dùng heavy để mỗi hypervisor chỉ subscribe data liên quan.

Thứ tư, 10 operation RFC 7047 đủ cho mọi OVSDB manipulation. Mutate thread-safe, update có thể conflict nếu concurrent write.

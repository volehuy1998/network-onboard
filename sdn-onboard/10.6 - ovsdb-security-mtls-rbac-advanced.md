## 10.6.1 The server-side security model of OVSDB

OVSDB has three independent security layers that must all be configured for a production environment:

**Layer 1, transport security (mTLS).** Every connection between `ovsdb-server` and a client (`ovn-controller`, `ovn-northd`, CLI tools) is encrypted and mutually authenticated through TLS 1.2 or later (client certificate plus server certificate). It prevents:

- Eavesdropping (reading traffic data).
- Man-in-the-middle attacks (impersonating the server).
- Unauthorised clients (must present a certificate from a trusted CA).

**Layer 2, authorization (RBAC).** After a client authenticates successfully, RBAC decides which tables and columns the client may read or write. It prevents:

- Lateral movement (chassis A cannot write the `Chassis` record of chassis B).
- Privilege escalation (`ovn-controller` cannot modify `SB_Global`).
- Tenant isolation (tenant A cannot read tenant B's `Logical_Flow` rows in a multi-tenant setup).

**Layer 3, audit logging.** Every transaction is logged with the client identity (the cert CN), timestamp, and operation. This prevents or detects:

- Internal abuse (detect an admin abusing privilege).
- Forensics after a breach (identify who modified what when a cert was leaked).

Configuring all three layers is the production standard. In practice, many deployments only have Layer 1 (mTLS) and lack RBAC, leaving the system vulnerable to insider attacks.

## 10.6.2 Setting up mTLS for an OVSDB cluster

The standard procedure (assuming a 3-node Raft cluster for OVN Southbound):

**Step 1, create a Root CA** (private to OVSDB, do not share with the system CA):

```
[ca-host]# ovs-pki init --dir /etc/ovsdb-ca
[ca-host]# ls /etc/ovsdb-ca/
cacert.pem  ca-req.pem  rootCA.key  rootCA.cert  cnf
```

**Step 2, sign a server certificate for each Raft node:**

```
[ca-host]# ovs-pki --dir /etc/ovsdb-ca req+sign sb-node1 switch
[ca-host]# scp sb-node1-cert.pem sb-node1-privkey.pem root@sb-node1:/etc/ovsdb-ssl/
# Same for sb-node2 and sb-node3.
```

Each node has its own cert with CN equal to the hostname (for example `sb-node1.internal`).

**Step 3, sign a client certificate for each chassis:**

```
[ca-host]# for i in $(seq 1 100); do
[ca-host]>   ovs-pki --dir /etc/ovsdb-ca req+sign chassis-$i switch
[ca-host]> done
```

The CN of the client cert is the chassis UUID (or the chassis hostname). This is the identity that RBAC uses.

**Step 4, configure `ovsdb-server` to use TLS:**

```
[sb-node1]# ovsdb-server --remote=pssl:6642 \
    --private-key=/etc/ovsdb-ssl/sb-node1-privkey.pem \
    --certificate=/etc/ovsdb-ssl/sb-node1-cert.pem \
    --ca-cert=/etc/ovsdb-ssl/cacert.pem \
    /var/lib/ovn/ovnsb_db.db
```

The `pssl:` flag (passive SSL) tells `ovsdb-server` to listen with TLS instead of plaintext TCP. The CA cert is checked on every incoming client connection.

**Step 5, configure the client (`ovn-controller`) to use the cert:**

```
[chassis-1]# ovs-vsctl set Open_vSwitch . \
    external_ids:ovn-remote="ssl:sb-node1:6642,ssl:sb-node2:6642,ssl:sb-node3:6642"
[chassis-1]# ovs-vsctl set Open_vSwitch . \
    other_config:ovn-remote-probe-interval=60000
[chassis-1]# cp /etc/ovsdb-ssl/chassis-1-*.pem /etc/ovn/
```

When `ovn-controller` connects to an `ssl:` URL, it sends its client cert `chassis-1-cert.pem` signed with `chassis-1-privkey.pem`. The server verifies the cert through the CA and recognises the client as "chassis-1".

## 10.6.3 Rotating certificates without downtime

X.509 certificates have a finite validity (often one year). When a cert is near expiry, it must be rotated, replaced by a new cert with the same CN. The procedure for a no-downtime rotation:

**Pattern 1, CA overlap (rotate the CA when the Root CA expires):**

1. Create the new CA.
2. Add the new CA to the `cacert.pem` bundle on every node and client. The bundle now contains both the old and the new CA.
3. Reload `ovsdb-server` (SIGHUP) to load the new bundle. From now on it accepts clients signed by either CA.
4. Sign new client certs from the new CA and deploy them per chassis.
5. Once every chassis uses the new cert, remove the old CA from the bundle and reload.

This pattern ensures there is never a window when a client cannot authenticate.

**Pattern 2, cert overlap (rotate a leaf cert):**

1. Sign a new cert for the chassis with a longer validity.
2. Deploy the new cert side-by-side with the old one (`ovsdb-server` accepts both as long as the CA is valid).
3. Restart `ovn-controller` to pick up the new cert.
4. After every chassis has switched, revoke the old cert through a CRL (Certificate Revocation List).

**Certificate Revocation List (CRL).** When a cert is leaked before expiry, add it to the CRL:

```
[ca-host]# ovs-pki --dir /etc/ovsdb-ca revoke chassis-1
```

Deploy the CRL to `ovsdb-server`:

```
[sb-node1]# cp /etc/ovsdb-ca/revoked.pem /etc/ovsdb-ssl/
[sb-node1]# ovs-appctl -t ovsdb-server reload
```

After loading the CRL, every new connection from chassis-1 with the old cert is rejected. Existing connections (already authenticated) continue until they reconnect, so there is a small window to keep in mind.

**OCSP (Online Certificate Status Protocol).** An alternative to CRL where the server checks cert status in real time through an OCSP responder. Not yet supported natively in `ovsdb-server` (OVN 22.03); on the roadmap.

## 10.6.4 RBAC advanced: multi-tenant and condition-based

Basic OVSDB RBAC (introduced in Part 10.2) lets you define:

- **Role.** A set of permissions (for example role `ovn-controller` or `ovn-northd`).
- **Permission.** Who can write which table or column.
- **Rule-based authorization.** Use a condition to restrict "which row may be written".

**Example RBAC for `ovn-controller` (each chassis can only write its own record):**

```json
{
  "RBAC_Role": {
    "name": "ovn-controller",
    "permissions": {
      "Chassis": {
        "rbac_rules": {
          "name": "ovn-controller",
          "authorization": "name",
          "insert_delete": true,
          "update": ["nb_cfg", "external_ids"]
        }
      },
      "Port_Binding": {
        "rbac_rules": {
          "authorization": "chassis",
          "update": ["chassis", "encap", "virtual_parent"]
        }
      }
    }
  }
}
```

With this rule, chassis-1 (CN of cert is "chassis-1") can only insert, delete, and update the `Chassis` record with name="chassis-1", and can only update `Port_Binding` rows already bound to "chassis-1". It cannot modify any other chassis record.

**Multi-tenant RBAC (a complex example for multi-tenant OVN-Kubernetes):**

```json
{
  "RBAC_Role": {
    "name": "tenant-admin-tenantA",
    "permissions": {
      "Logical_Switch": {
        "rbac_rules": {
          "authorization": "external_ids:tenant=tenantA",
          "insert_delete": true,
          "update": ["name", "ports", "acls"]
        }
      }
    }
  }
}
```

The tenant A admin can only write `Logical_Switch` rows that have `external_ids:tenant=tenantA`. They cannot read or modify tenant B's LS rows.

**Condition-based authorization through `external_ids`.** OVN uses this pattern widely: every row with `external_ids:owner=X` is modifiable only by a user whose cert CN matches X.

**OVSDB RBAC limitations compared with SQL.**

- No GROUP BY or column-level fine-grained permissions like PostgreSQL ROW LEVEL SECURITY.
- Authorization checks only via string match: no regex or subquery support.
- No SUPERUSER concept: every role must have its permissions listed table by table.
- No SAVEPOINT for partial-authorization failure.

## 10.6.5 Audit logging and compliance

**Default `ovsdb-server` logging:**

```
[sb-node1]# tail -f /var/log/ovn/ovsdb-server-ovnsb_db.log
2026-04-22T10:00:01.234Z|00123|jsonrpc|DBG|ssl:10.0.0.5:35782: received transact (1 ops)
2026-04-22T10:00:01.235Z|00124|ovsdb_server|INFO|chassis-1 (CN=chassis-1) inserted Port_Binding
```

Default log level is INFO. To enable full audit:

```
[sb-node1]# ovs-appctl -t ovsdb-server vlog/set file:jsonrpc:dbg
[sb-node1]# ovs-appctl -t ovsdb-server vlog/set file:ovsdb_server:dbg
```

Logs rotate daily through logrotate and are retained for 90 days (a standard compliance policy).

**Structured logging to SIEM.** Forward JSON logs to Splunk, ELK, or Datadog:

```
[sb-node1]# ovs-appctl -t ovsdb-server vlog/set syslog:dbg
```

Send through syslog with a format that includes the cert CN, transaction ID, and operation details.

**Compliance scenarios.**

- **SOC 2 / ISO 27001.** Audit log retained for one year, stored in immutable storage (Amazon S3 Object Lock).
- **PCI-DSS.** For systems that process payments, audit every insert, update, and delete on `Logical_Router_NAT` (VIP mapping).
- **GDPR.** For user data, audit access and verify tenant isolation.

**Anomaly detection.** Integrate audit logs with a SIEM to alert on:

- A `Chassis` record write that is not the writer's own (a probable RBAC bypass attempt).
- A burst of transactions from one cert (possible bot attack).
- Cert rotation overdue (the cert is near expiry but has not been rotated).

## 10.6.6 Threat model and mitigations

Production OVSDB threat model:

**Threat 1, certificate exposure.** Cert private key leaks (developer laptop compromised, unencrypted backup).

- **Mitigation.** Deploy CRL fast, monitor for unusual cert usage. Use short-lived certs (90 days instead of one year).

**Threat 2, insider abuse.** Admin has a valid cert and abuses write access.

- **Mitigation.** Strict RBAC (admin does not have a "full-access" role, only specific roles). Audit log plus four-eye principle for production changes.

**Threat 3, DoS transaction flood.** A malicious client sends millions of small transactions to saturate `ovsdb-server`.

- **Mitigation.** Rate limiting at the network layer (iptables, Cilium NetworkPolicy). OVSDB 22.03 does not have built-in rate limiting.

**Threat 4, downgrade attack.** Attacker forces an old TLS version with known vulnerabilities.

- **Mitigation.** Configure `ssl_protocols="TLSv1.2,TLSv1.3"` in `ovsdb-server`. Disable TLSv1.0 and TLSv1.1.

**Threat 5, compromised CA.** Attacker steals the root CA private key and can sign any cert.

- **Mitigation.** Use an HSM (Hardware Security Module) for the root CA. Use an offline CA (not connected to a network, used only when signing).

## 10.6.7 The `RBAC_Role` table in the OVSDB schema

**Concept and definition.** `RBAC_Role` is a standard OVSDB table (it appears in both `ovn-nb.ovsschema` and `ovn-sb.ovsschema`) that stores the definition of each role. One row corresponds to one identity class (for example `ovn-controller`, `ovn-northd`); rows in this table are not user accounts but the role into which the client cert is mapped after successful authentication.

**Classification and place in the architecture.** Type: OVSDB metadata table, schema-level (not a data table that users CRUD). Place: `ovsdb-server` keeps an internal reference through `db->rbac_role` (see `ovsdb/rbac.c` function `ovsdb_rbac_lookup_perms`), loaded when the server opens the database file.

**Schema verbatim** (`ovn-sb.ovsschema`, branch-22.03 verified through raw GitHub):

```json
"RBAC_Role": {
  "columns": {
    "name": {"type": "string"},
    "permissions": {
      "type": {
        "key": {"type": "string"},
        "value": {"type": "uuid",
                  "refTable": "RBAC_Permission",
                  "refType": "weak"},
        "min": 0, "max": "unlimited"
      }
    }
  }
}
```

**How it works.** When a client runs a transaction, `ovsdb-server` calls `ovsdb_rbac_lookup_perms(db, role, table)` (file `ovsdb/rbac.c`). The function (1) looks up the row in `RBAC_Role` whose `name == role`, (2) looks up the key by the target table name in the `permissions` map, (3) returns the UUID pointing to a row in `RBAC_Permission`. The map has `refType: "weak"`, meaning that when an `RBAC_Permission` row is deleted the map entry is cleaned up automatically (no dangling reference is created).

**How an operator works with it.** Create a new role through `ovn-sbctl` (or raw `ovsdb-client transact`):

```
[admin]# ovn-sbctl -- --id=@p create RBAC_Permission table=Chassis \
    authorization=name insert_delete=true update='["nb_cfg","external_ids"]' \
    -- create RBAC_Role name=ovn-controller permissions:Chassis=@p
```

Verify:

```
[admin]# ovn-sbctl list RBAC_Role
_uuid               : abc12345-...
name                : "ovn-controller"
permissions         : {"Chassis"=def67890-..., "Port_Binding"=...}
```

**Failure mode and diagnostic signal.** If the role row does not exist for the `role` that the manager assigns to the client (see 10.6.8), every write from that client is rejected; `ovsdb-server` logs the line `RBAC: role 'X' has no permissions on table 'Y'`. Recovery: create an `RBAC_Permission` row and insert it into the `RBAC_Role.permissions` map.

## 10.6.8 The `RBAC_Permission` table and the authorization expression

**Concept and definition.** `RBAC_Permission` is the granular table that specifies "a client with role X is allowed to do what on table Y". Each row has four primary columns: `table`, `authorization`, `insert_delete`, `update`.

**Schema verbatim** (verified raw GitHub, branch-22.03):

```json
"RBAC_Permission": {
  "columns": {
    "table":         {"type": "string"},
    "authorization": {"type": {"key": "string", "min": 0, "max": "unlimited"}},
    "insert_delete": {"type": "boolean"},
    "update":        {"type": {"key": "string", "min": 0, "max": "unlimited"}}
  }
}
```

**How it works.** The function `ovsdb_rbac_authorized(perms, id, row)` (file `ovsdb/rbac.c`) iterates over each entry of the `authorization` column:

- An empty string `""` entry means every client is allowed (wildcard).
- An entry that is a column name (for example `"name"`) means compare `strcmp(row->columns["name"], id)`. If it matches, return true.
- An entry containing a colon (for example `"external_ids:tenant"`) is split into `(column_name, key)`, the column map is looked up at the key, and the value compared with `id`. This is dot-separated path lookup, verified through the helper function `ovsdb_find_row_by_string_key`.

`id` is the **CN of the client cert** (not a username). `ovsdb-server` extracts it from the TLS handshake (see 10.6.9).

**Distinguish `insert_delete` from `update`.** `insert_delete=true` permits both INSERT (creating a new row whose `authorization` field equals `id`) and DELETE (deleting a row whose `authorization` field equals `id`). `update` is the list of column names the client may modify on a row that has passed the authorization check; columns not listed are rejected even if authorization passes. The same `update` list applies to both the `update` and `mutate` operations (verified through `ovsdb_rbac_update` and `ovsdb_rbac_mutate`).

**Real-world OVN pattern.** The standard row for `ovn-controller` writing `Port_Binding` only for its own chassis:

```
table:         Port_Binding
authorization: ["chassis"]      # Port_Binding.chassis is a ref to the Chassis row
insert_delete: false             # ovn-controller cannot create or delete Port_Binding
update:        ["chassis", "encap", "virtual_parent", "up", "nb_cfg"]
```

Authorization `"chassis"` is a ref column. `ovsdb-server` resolves through the chain "Port_Binding.chassis to Chassis._uuid to Chassis.name" and compares with the client cert CN.

**Failure mode.** An `authorization` set to `[]` (no entries) means every write is rejected (deny-by-default). Log line: `RBAC: row not authorized for role 'X' on table 'Y'`.

## 10.6.9 The mTLS handshake between `ovsdb-server` and the client

**Concept and role.** mTLS (mutual TLS) is the TLS variant where both server and client present a cert signed by a trusted CA. In OVSDB, the mTLS handshake lets `ovsdb-server` map a TCP connection to an identity (cert CN) for the RBAC lookup in 10.6.7 and 10.6.8.

**How it works in detail.** The implementation lives in `lib/stream-ssl.c` of OVS and uses OpenSSL. Sequence:

1. The client opens a TCP connection to `pssl:6642` (the passive SSL listener of `ovsdb-server`).
2. The server sends `ServerHello` plus `Certificate` (the sb-node cert signed by the CA).
3. The server sends `CertificateRequest` (asking the client for a cert because the TLS context of `ovsdb-server` is set with `SSL_VERIFY_PEER | SSL_VERIFY_FAIL_IF_NO_PEER_CERT`).
4. The client sends `Certificate` (the chassis-X cert) plus `CertificateVerify` (a signature proving ownership of the private key).
5. Both derive a session key and finish the handshake.
6. After the handshake, `ovsdb-server` calls `SSL_get_peer_certificate()`, extracts the Subject CN field, and stores it in the `jsonrpc_session` struct for every subsequent RBAC check.

**Supported TLS versions.** OVS 2.17 uses `SSLv23_method()` (multi-version) plus the flags `SSL_OP_NO_SSLv2 | SSL_OP_NO_SSLv3` (verified in `lib/stream-ssl.c`). TLS 1.0, 1.1, 1.2, 1.3 are all enabled by default; tune through `--ssl-protocols=TLSv1.2,TLSv1.3` (verified in the `ovsdb-server(1)` man page). Standard deployments from 2026 onwards should explicitly disable TLSv1.0 and 1.1 because of downgrade attacks (Threat 4 in 10.6.6).

**Verification.** Observe the handshake with `tcpdump` and decode in Wireshark using a key log:

```
[chassis-1]# SSLKEYLOGFILE=/tmp/keys.log ovn-sbctl --db=ssl:sb-node1:6642 ... show
[chassis-1]# tcpdump -i any -w /tmp/h.pcap port 6642
```

In Wireshark, Edit → Preferences → TLS → (Pre)-Master-Secret log filename → `/tmp/keys.log`; opening `h.pcap` shows the full ServerHello, CertificateRequest, and ClientCertificate.

**Typical failure modes.** (a) Incomplete cert chain: client log `unable to verify the first certificate` means the bundle is missing an intermediate CA. (b) Cert expired: log `certificate has expired`. (c) Server rejects the client: server log `peer did not return a certificate` means the client is missing the `--certificate` flag. (d) CN does not match the role: handshake succeeds, but every transaction is rejected because `ovsdb_rbac_lookup_perms` returns NULL for the (role, CN) pair.

## 10.6.10 The `--bootstrap-ca-cert` flag and the risk of leaving it on in production

**Concept and role.** `--bootstrap-ca-cert` is a "trust on first use" (TOFU) mode for the CA certificate. When the configured CA cert file does not yet exist, `ovsdb-server` (or the client) accepts the cert from the peer on the first connection without verification, automatically takes the self-signed CA, and saves it into the file. Subsequent connections use the saved CA to verify normally.

**Verbatim from the man page** (`ovsdb-server(1)`, verified at man7.org):

> "When `cacert.pem` exists, this option has the same effect as `-C` or `--ca-cert`. If it does not exist, then `ovsdb-server` will attempt to obtain the CA certificate from the SSL/TLS peer on its first SSL/TLS connection and save it to the named PEM file."
>
> "This option exposes the SSL/TLS connection to a man-in-the-middle attack obtaining the initial CA certificate, but it may be useful for bootstrapping."

**How it works.** The logic lives in `lib/stream-ssl.c` through the function `do_ca_cert_bootstrap()` (verified raw GitHub):

1. The parser `parse_options()` in `ovsdb/ovsdb-server.c` sets the global `bootstrap_ca_cert = true` when it sees the `OPT_BOOTSTRAP_CA_CERT` enum case.
2. `reconfigure_ssl()` calls `stream_ssl_set_ca_cert_file(file, bootstrap=true)`.
3. On the first completed handshake, `do_ca_cert_bootstrap()` runs: `SSL_get_peer_cert_chain()` retrieves the peer chain, `X509_check_issued(cert, cert)` checks self-signed, opens the file with `O_CREAT | O_EXCL | O_WRONLY` and `PEM_write_X509(file, cert)`. Finally, `SSL_CTX_load_verify_locations()` reloads the new CA and clears the bootstrap flag.

**Why it is forbidden in production.** During the TOFU phase there is no chain verification. An attacker that MITMs between nodes can present a forged cert, and `ovsdb-server` will adopt the attacker as the legitimate CA. Every subsequent connection is then "legitimately" MITM-ed. The flag is suitable only for a dev lab or PoC, or in an environment with an out-of-band trusted channel (for example, a physically separate management subnet).

**Correct production workflow.** Distribute the CA cert through a trusted channel (Ansible vault, HashiCorp Vault, or kolla-ansible secret distribution) before starting the service. Specify it with a fixed `--ca-cert=/etc/ovsdb-ssl/cacert.pem`. Do NOT use `--bootstrap-ca-cert`. Verify at deploy time:

```
[sb-node1]# sha256sum /etc/ovsdb-ssl/cacert.pem
abcdef...   # must match the hash already approved from the CA host
```

**Failure mode.** If `--bootstrap-ca-cert` is left in the systemd unit by accident and the CA file already exists on disk, the flag degrades to an alias of `--ca-cert` (safe). But if the file is deleted (because of disk failure plus recovery, or because someone runs `rm`) before the service restart, the next service start re-enters TOFU and the next connection can be MITM-ed. Recovery: detect through an audit script (`grep bootstrap-ca-cert /etc/systemd/system/ovsdb-server.service`), remove the flag, and restart the service.

## 10.6.11 Detailed certificate-rotation procedure without downtime

**Concept and role.** Cert rotation is the replacement of an old cert by a new one (same CN, new key) before the old cert expires, in a way that no existing connection is broken and there is no time window when a client cannot authenticate.

**Core mechanism.** OVS does not hot-reload the private key. However `ovsdb-server` can reload the CA cert plus the cert chain through `SIGHUP` or `ovs-appctl -t ovsdb-server vlog/reopen` (for logs) and `ovs-appctl -t ovsdb-server ovsdb-server/reconnect` (for remote config). When the cert and key change, the most stable method is a rolling restart of each Raft node in order: a 3-node Raft cluster tolerates losing one node at a time, and clients reconnect to the remaining nodes.

**Full no-downtime rotation workflow for a leaf cert (chassis-X).**

Step 1, sign the new cert from the current CA but change the CN suffix to `-v2` to distinguish:

```
[ca]# ovs-pki --dir /etc/ovsdb-ca req+sign chassis-1-v2 switch
```

Step 2, push the new cert side-by-side with the old one:

```
[chassis-1]# cp chassis-1-v2-{cert,privkey}.pem /etc/ovn/
[chassis-1]# ln -sfn chassis-1-v2-cert.pem /etc/ovn/active-cert.pem
[chassis-1]# ln -sfn chassis-1-v2-privkey.pem /etc/ovn/active-privkey.pem
```

A symlink-based switch is atomic: only the symlink changes, the file is not edited.

Step 3, restart `ovn-controller`:

```
[chassis-1]# systemctl restart ovn-controller
```

In about 1 to 3 seconds `ovn-controller` reconnects to the SB cluster. Because the connection from this chassis dropped and reopened, the chassis's `Port_Binding` data is briefly not refreshed, but the OF rules already installed on `br-int` continue to forward traffic (the datapath cache does not depend on the SB connection).

Step 4, verify the new CN:

```
[chassis-1]# ovs-appctl -t ovn-controller connection-status
state: connected
remote: ssl:sb-node1:6642
```

On SB:

```
[sb-node1]# grep "chassis-1-v2" /var/log/ovn/ovsdb-server-ovnsb_db.log | tail -3
... ssl:10.0.0.1:43210: client cert CN=chassis-1-v2 authenticated
```

Step 5, after every chassis has moved to cert v2, revoke cert v1 through CRL and reload the CA bundle.

**Risk window.** A CRL reload does not kick existing connections. A client already connected with the revoked cert continues until either (a) the probe interval times out (`ovn-remote-probe-interval`, default 60 s), or (b) the chassis restarts itself. To force the disconnect: `ovs-appctl -t ovsdb-server connection/kill <chassis-CN>`. During this 60-second window, an attacker who has the leaked cert can still transact, so short-lived certs (90 days, per NIST SP 800-57 guideline for frequently rotated keys) reduce the impact.

**Failure mode.** (a) The new cert is signed with the wrong CN: the client connects but RBAC rejects every write; recovery: re-sign with the correct CN. (b) A symlink points to a file that does not exist: `ovn-controller` logs `cannot open private key file`; recovery: `readlink -f` to verify before restarting the service. (c) Pushing the new cert to the entire fleet of more than 100 chassis at once: the SB cluster suffers a connect burst that may OOM the revalidator; recovery: canary one chassis first, then five, then roll out the rest with rate limiting (one chassis every two seconds).

## 9.9.1 OVS cấu hình QoS, không implement

Compass Ch I điểm quan trọng: **OVS không implement QoS, chỉ cấu hình nó**. Queueing thực sự do:

- Linux `tc` (traffic control) cho kernel datapath.
- DPDK rte_sched library cho userspace datapath.

OVS cung cấp API trừu tượng qua OVSDB tables (`QoS`, `Queue`) và OpenFlow action (`set_queue`). Khi cấu hình, OVS dịch thành `tc class` + `tc filter` cho kernel, hoặc rte_sched config cho DPDK.

Hai loại QoS:

- **Ingress policing**: rate limiter đơn giản (token bucket) áp trực tiếp trên Interface row.
- **Egress shaping**: hierarchical với QoS + Queue tables + OpenFlow action.

## 9.9.2 Ingress policing

```bash
[host]# ovs-vsctl set interface tap0 ingress_policing_rate=1000 \
                                     ingress_policing_burst=100
```

- `ingress_policing_rate`: kbps.
- `ingress_policing_burst`: kb.
- Set 0 để disable.

Packet vượt rate → drop ngay tại ingress. Không có fairness giữa nhiều source.

## 9.9.3 Egress shaping với linux-htb

Compass Ch I verbatim recipe:

```bash
[host]# ovs-vsctl -- add-br br0 \
         -- add-port br0 eth0 \
         -- add-port br0 vif1.0 -- set interface vif1.0 ofport_request=5 \
         -- add-port br0 vif2.0 -- set interface vif2.0 ofport_request=6 \
         -- set port eth0 qos=@newqos \
         -- --id=@newqos create qos type=linux-htb \
               other-config:max-rate=1000000000 \
               queues:123=@vif10queue queues:234=@vif20queue \
         -- --id=@vif10queue create queue other-config:max-rate=10000000 \
         -- --id=@vif20queue create queue other-config:max-rate=20000000
```

- QoS parent `max-rate=1 Gbps`.
- Queue ID 123 max 10 Mbps (cho vif1.0).
- Queue ID 234 max 20 Mbps (cho vif2.0).
- Queue ID là arbitrary OpenFlow queue ID mà `set_queue:` action tham chiếu.

**Bước bắt buộc** — OpenFlow rule để packet vào đúng queue:

```bash
[host]# ovs-ofctl add-flow br0 in_port=5,actions=set_queue:123,normal
[host]# ovs-ofctl add-flow br0 in_port=6,actions=set_queue:234,normal
```

Compass Ch I cảnh báo: "**Without the OpenFlow rules, packets go to the default queue and nothing is shaped — this is the single most common QoS misconfiguration in the field.**"

---

## 9.9.4 Các type QoS khác

- `type=linux-hfsc`: Hierarchical Fair Service Curve.
- `type=egress-policer`: flat rate enforcer cho DPDK.
- `type=trtcm-policer`: Two-Rate Three-Color Marker (RFC 2698).

Verify:

```bash
[host]# ovs-appctl qos/show eth0
[host]# tc qdisc show dev eth0
[host]# tc class show dev eth0
```

## 9.9.5 Cleanup (QoS không auto-GC)

QoS và Queue rows **không được garbage-collect** — phải destroy explicit:

```bash
[host]# ovs-vsctl -- destroy QoS eth0 -- clear Port eth0 qos
[host]# ovs-vsctl -- --all destroy Queue
```

Khác biệt quan trọng với Mirror (Part 9.7). Không cleanup → orphan rows tích lũy.

### ▶ Guided Exercise 10: Rate limit VM traffic với linux-htb

**Outcomes:** Apply rate limit 10 Mbps cho VM, verify bằng iperf3.

**Before You Begin:**

```bash
[host]# ovs-vsctl add-br br-qos
[host]# ovs-vsctl add-port br-qos vm1 -- set Interface vm1 type=internal \
                                     -- set Interface vm1 ofport_request=5
[host]# ovs-vsctl add-port br-qos vm2 -- set Interface vm2 type=internal
[host]# ip link set vm1 up && ip addr add 10.99.0.1/24 dev vm1
[host]# ip link set vm2 up && ip addr add 10.99.0.2/24 dev vm2
```

**Instructions:**

**1.** Setup QoS:

```bash
[host]# ovs-vsctl -- set Port vm1 qos=@qos \
         -- --id=@qos create QoS type=linux-htb \
                other-config:max-rate=1000000000 queues:10=@q \
         -- --id=@q create Queue other-config:max-rate=10000000
```

**2.** OpenFlow rule:

```bash
[host]# ovs-ofctl add-flow br-qos "in_port=5,actions=set_queue:10,normal"
```

**3.** Test:

```bash
[host]# iperf3 -s -B 10.99.0.2 -D
[host]# iperf3 -c 10.99.0.2 -B 10.99.0.1 -t 10
```

Expect cap ~10 Mbps.

**4.** Cleanup:

```bash
[host]# ovs-vsctl -- destroy QoS vm1 -- clear Port vm1 qos
[host]# ovs-vsctl --all destroy Queue
[host]# ovs-vsctl del-br br-qos
```

## Điểm cốt lõi cần nhớ

Thứ nhất, OVS cấu hình QoS qua OVSDB abstraction, queue thực sự chạy ở Linux tc hoặc rte_sched. Debug → check cả hai view.

Thứ hai, egress shaping cần ba thứ: QoS row + Queue row + OpenFlow `set_queue` action. Thiếu OpenFlow rule là lỗi phổ biến nhất.

Thứ ba, QoS/Queue rows KHÔNG auto-GC. Phải `destroy` explicit.

Thứ tư, ingress policing là token bucket đơn giản — drop khi vượt rate, không fairness.

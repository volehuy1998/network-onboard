## 13.5.1 Port_Binding overview: role and schema

A `Port_Binding` row in SBDB represents the binding between a **logical port** (the LSP or LRP definition in NBDB) and a **physical chassis** (the host that processes traffic for that port). Every LSP and LRP in NBDB has exactly one corresponding Port_Binding row in SBDB, generated by `ovn-northd` during compilation.

### Important schema columns

```shell
$ sudo ovn-sbctl list Port_Binding lsp_web_01
_uuid          : 7a3c9f82-...
logical_port   : "lsp_web_01"
datapath       : <datapath_uuid for Logical_Switch>
chassis        : [<chassis_uuid>]        # NULL or the chassis claiming the port
type           : ""                       # 8 values: "" / localnet / l2gateway / ...
tunnel_key     : 5                        # Per-datapath unique ID
mac            : ["aa:bb:cc:00:00:01 10.0.0.5"]
external_ids   : {tenant="T1"}
options        : {requested-chassis="chassis-A"}  # Optional pin
parent_port    : []                       # For nested ports (virtual)
tag            : []                       # VLAN tag for localnet
gateway_chassis: []                       # Dedicated column for chassisredirect
ha_chassis_group: []                      # HA_Chassis_Group reference
up             : true                     # Binding state up or down
nat_addresses  : []                       # For logical router port
virtual_parent : []                       # For the virtual type
```

### Claim mechanism

The `ovn-controller` on each chassis subscribes to the SBDB `Port_Binding` table. When it sees a row that matches a claim condition (for example, an OVS port whose `external_ids:iface-id` matches `logical_port`), the controller issues an OVSDB transaction:

```sql
UPDATE Port_Binding SET chassis=<this_chassis_uuid> WHERE _uuid=<row_uuid>
```

SBDB Raft replicates the update to every Raft follower. Every other `ovn-controller` sees the new binding within 500 ms.

## 13.5.1b Port_Binding as an SBDB table: row anatomy and binding mechanism

> This section drills into the `Port_Binding` table itself (a row in the OVN Southbound database), separately from the catalog of the 8 types in Part 13.5b. The goal: the engineer understands the row structure, the lifecycle, the columns that carry operational meaning, and how to trace a runtime incident back to a single SBDB row.

### Concept

`Port_Binding` is a **resident table in the OVN Southbound database (SBDB)** that stores the **compilation output** of `ovn-northd`. Each `Logical_Switch_Port` (LSP) or `Logical_Router_Port` (LRP) in NBDB produces exactly one corresponding Port_Binding row in SBDB. The row is the **data contract** between the control plane (`ovn-northd`) and the data plane (`ovn-controller` on each chassis). northd defines "this logical port lives in which datapath, carries which MAC and IP, has which type, and is pinned to a chassis or not". The `ovn-controller` on each chassis decides "does this row belong to me; if so, update the `chassis` column to my UUID and install the OpenFlow rules". Kind: **OVSDB table**, a root table (`isRoot=true`), persistent (replicated through the SBDB Raft cluster).

### History and background

The `Port_Binding` table appeared with OVN 2.4 (2015) when Russell Bryant (Red Hat) and Justin Pettit (VMware and Nicira) refactored the Nicira NSX OpenStack plugin architecture into OVN. Before that, the OpenStack Neutron OVS plugin stored binding state directly in the Neutron MySQL database, causing races when `neutron-server` and the compute node exchanged messages slowly. The OVN design separates intent from state: NBDB holds the **intent**, SBDB holds the **runtime state** that every chassis observes. Port_Binding is one of the original SBDB tables alongside `Chassis`, `Datapath_Binding`, and `Logical_Flow`. OVN 21.06 added the `requested_chassis` column, OVN 22.03 stabilized the `up` column for liveness reporting, and OVN 22.06 added the `additional_chassis` column for multichassis live migration (see the "Version differences" axis below).

### Position in the architecture

The table sits at the **SBDB layer** (control-plane intermediate), between **NBDB** (declarative input from the CMS) and **OpenFlow on `br-int`** (executable output at the chassis). Scope: **per-cluster, per-row**. An OVN cluster has exactly one SBDB Raft cluster (three or five members). Every `ovn-controller` on every chassis subscribes to the same SBDB through an OVSDB JSON-RPC connection. Each Port_Binding row is **one unique record** in the cluster (it is not replicated per chassis). The `chassis` column points to the Chassis row of the owning chassis. Impact level: a change to one Port_Binding row (for example, the `chassis` field moving from A to B) causes the `ovn-controller` on both A and B to recompute the OpenFlow rule set within one second (event-driven I-P engine since OVN 21.03 and later).

### Role

Port_Binding carries **four roles** in the OVN architecture:

1. **Logical-to-physical mapping**: it contains the `logical_port` column (the name in NBDB) linked to the `chassis` column (the UUID of the owning Chassis row). This is the only table that records "which chassis currently hosts this logical port" at runtime.
2. **Allocates `tunnel_key` for the data plane**: the `tunnel_key` column (1 to 32767, unique within a datapath) is encoded into the `MFF_LOG_OUTPORT` and `MFF_LOG_INPORT` fields of the Geneve TLV outer header. Without a `tunnel_key`, the destination chassis cannot demultiplex the packet.
3. **Passes binding parameters down to `ovn-controller`**: the `options` map contains `requested-chassis`, `network_name` (for `localnet`), `l3gateway-chassis`, and `activation-strategy` (for migration). `ovn-controller` reads this map and translates it into specific OpenFlow rules.
4. **Reports liveness back up to the CMS**: the `up` boolean is updated by the owning chassis when the interface actually goes up or down. The CMS (Neutron, kube-ovn) polls this column to display port state to users.

### Why it exists

Before OVN 2.4, the Nicira NVP and NSX architecture used a centralized controller cluster that pushed OpenFlow rules directly to each chassis over OpenFlow channels. The problem: the controller had to keep per-chassis state in memory, scale was O(chassis times port), and each controller cluster only served 1000 to 2000 hypervisors (production NSX figures for 2014 and 2015). OVN solved this by **pushing state out into OVSDB Raft and letting each chassis handle itself**. SBDB carries **declarative state** (Port_Binding rows, Logical_Flow rows); `ovn-controller` runs locally on each chassis, reads SBDB through JSON-RPC, computes OpenFlow on its own, and needs no central controller holding state. Port_Binding is the bridge: it is declarative enough for `ovn-northd` to precompile, yet it still lets the chassis claim and release through OVSDB transactions (the `chassis` column). This model scales to 5000 to 10000 hypervisors (Red Hat OSP 17 production figures, 2023).

### Problems it solves

This table solves **five distinct technical problems**: (1) **Distributed binding decisions**: instead of a central controller choosing the chassis, each chassis asserts ownership through an OVSDB compare-and-swap. (2) **Stateful migration handover**: the `additional_chassis` column (22.06 and later) lets two chassis hold the binding inside the live-migration window without conflict. (3) **HA gateway failover**: the `gateway_chassis` column (a set of UUIDs with priority) and `ha_chassis_group` allow the redirect chassis to change owner in under 200 ms when BFD reports down. (4) **Provider network without claim**: the `type=localnet` value and `tag` (VLAN) require no chassis claim. Every chassis with a matching bridge mapping bridges to the physical network simultaneously. (5) **Reverse lookup from OVS port to OVN logical port**: `external_ids:iface-id` on the OVS Interface matches the `logical_port` column of the Port_Binding row. This is the only join key between OVS local state and OVN cluster state.

### Importance

This is a **top-tier foundational table** of SBDB, peer to `Logical_Flow`. Reasons: 100 percent of "VM has no network" and "north-south NAT down" debug workflows go through Port_Binding. 100 percent of advanced OVN features (live migration, HA gateway, multichassis, provider network, virtual VIP) add a column or an option field to this table instead of creating a new table. An engineer who does not understand Port_Binding cannot debug any OVN port-related incident. Among the more than 320 keywords in the REF, Port_Binding sits in the group of 15 OVN foundational keywords alongside `ovn-northd`, `ovn-controller`, NBDB, SBDB, Logical_Flow, Datapath_Binding, Logical_Switch, Logical_Router, ACL, Load_Balancer, NAT, Geneve TLV, `ovn-trace`, and `ovn-detrace`.

### How it works (the lifecycle of a row)

A Port_Binding row passes through **five states** over its lifetime:

1. **Compile**: the CMS calls `ovn-nbctl lsp-add ls vm1`, and NBDB gains a new `Logical_Switch_Port` row. `ovn-northd` (running on the NBDB Raft leader) compiles NBDB into SBDB by creating a Port_Binding row with `logical_port="vm1"`, `datapath=<dp_uuid>`, `tunnel_key=<allocated>`, `mac=[...]`, `type=""`, and `chassis=NULL`. This transaction is atomic; the row becomes visible immediately after commit.
2. **Unbound advertisement**: SBDB Raft replicates the row to every follower so every `ovn-controller` sees the new row with `chassis=[]`. Each controller runs the `binding_run` function in `controller/binding.c` (Rule 14.4 Option C function-name anchor; verified present at v22.03.8), iterating over every Port_Binding and dispatching by type to handlers `consider_vif_lport`, `consider_container_lport`, `consider_localport`, and `consider_virtual_lport`.
3. **Claim**: if chassis X has an OVS port with `external_ids:iface-id=vm1`, the `claim_lport` function in `controller/binding.c` (Option C anchor) issues an OVSDB transaction: `UPDATE Port_Binding SET chassis=<X_uuid>, encap=<X_geneve_encap_uuid> WHERE _uuid=<row>`. SBDB performs a compare-and-swap: when two chassis claim simultaneously, only one wins. The other receives a transaction conflict, retries, sees `chassis` already set, and gives up.
4. **Active**: the row has been claimed. The `ovn-controller` on the owning chassis installs OpenFlow rules on `br-int` (action `output:<ofport>` for ingress, action `set_field:tunnel_key->NXM_NX_TUN_METADATA0` for egress). The `up` column is updated to reflect the `link_state` of the OVS Interface.
5. **Release**: when the OVS port is unplugged or `iface-id` is cleared, the `release_lport` function in `controller/binding.c` (Option C anchor) issues `UPDATE Port_Binding SET chassis=[], encap=[], virtual_parent=[]`. The SBDB log on that chassis records `"Releasing lport vm1 from this chassis."`

Runtime invariant: at any moment, a row has **at least 0 and at most 1** chassis in the `chassis` column (the column is a weak UUID ref to Chassis with `min=0, max=1`). The exception is the multichassis migration window (22.06 and later) when the `additional_chassis` column may carry one extra UUID.

### Column classification

The 17 columns of Port_Binding (verified against `ovn-sb.ovsschema` branch-22.03) split into **four groups** by semantic role:

| Group | Columns | Required | Who writes |
|-------|---------|----------|------------|
| Identity | `logical_port`, `datapath`, `tunnel_key`, `type` | Yes (4 of 4) | `ovn-northd` (NBDB compilation) |
| Address content | `mac`, `nat_addresses`, `tag`, `parent_port`, `virtual_parent` | No (only when needed) | `ovn-northd` |
| Runtime binding | `chassis`, `encap`, `up`, `requested_chassis` | No | `ovn-controller` (claim and release) |
| Service parameters | `options` (map), `gateway_chassis` (set), `ha_chassis_group`, `external_ids` (map) | No | Both `ovn-northd` and `ovn-controller` |

Two indexes enforce uniqueness: `["datapath", "tunnel_key"]` (no two ports in the same datapath share a `tunnel_key`) and `["logical_port"]` (the `logical_port` name is unique across the cluster).

### How an operator masters this

Four foundational skills the engineer must have with Port_Binding:

1. **Read a full row**: `sudo ovn-sbctl list Port_Binding <logical_port>` prints all 17 columns. Read top to bottom and determine: the type, whether the datapath belongs to an LS or LR, whether the chassis is claimed or not, whether `options` pins `requested-chassis`, and whether `up=true` or `up=false`.
2. **Read multiple rows as a table**: `sudo ovn-sbctl --columns=logical_port,type,chassis,tunnel_key,up list Port_Binding` gives a cluster-wide overview. Filter by type with `sudo ovn-sbctl find Port_Binding type=chassisredirect` to find every active gateway port.
3. **Observe changes in real time**: `sudo ovsdb-client monitor unix:/var/run/ovn/ovnsb_db.sock OVN_Southbound Port_Binding chassis logical_port up` streams every change in the `chassis` and `up` columns to stdout. Use this when debugging live migration or gateway failover.
4. **Decide which skill to use when**: if the CMS reports "VM has no network", start with `list Port_Binding <name>` (skill 1) and compare `chassis` against the actual hypervisor. If you suspect a gateway failover, use `find Port_Binding type=chassisredirect` (skill 2). If you need to confirm a failover completed, monitor in real time (skill 3).

### Usage workflow

The provisioning workflow for a VM tap through Port_Binding has six steps, each with a verification command:

1. **The CMS creates the LSP in NBDB**: `ovn-nbctl lsp-add ls-tenant1 vm1; ovn-nbctl lsp-set-addresses vm1 "aa:bb:... 10.0.0.5"`. Verify: `ovn-nbctl show ls-tenant1 | grep vm1`.
2. **Wait for northd to compile**: typically under 100 ms. Verify: `ovn-sbctl --columns=logical_port,type,datapath list Port_Binding vm1` shows the row exists.
3. **Plug the tap into br-int and set `iface-id`**: `ovs-vsctl add-port br-int tap-vm1 -- set Interface tap-vm1 external_ids:iface-id=vm1`. Verify: `ovs-vsctl get Interface tap-vm1 external_ids:iface-id` returns `"vm1"`.
4. **Wait for `ovn-controller` to claim**: typically under one second. Verify: `ovn-sbctl --columns=chassis,up list Port_Binding vm1` shows `chassis=[<uuid_local>], up=true`.
5. **Confirm the OpenFlow rules are installed**: `ovs-ofctl -O OpenFlow15 dump-flows br-int | grep aa:bb` shows rules matching the MAC of vm1 on both ingress and egress.
6. **Confirm external connectivity** (if the tenant LSP needs to reach the Internet): `ip netns exec vm1-ns ping 8.8.8.8`. Verify end-to-end.

Production best practice: **always set `requested-chassis` when the VM has a placement constraint** (anti-affinity, GPU node) through `ovn-nbctl lsp-set-options vm1 requested-chassis=hv-gpu-01`. Anti-pattern: rely on "the chassis can claim by itself" without pinning. Spontaneous migration from a live gateway update then causes a connectivity blip.

### When something goes wrong (reverse symptom map)

Six common symptoms point back to Port_Binding:

| Symptom | Port_Binding columns to check | Diagnostic flow |
|---------|-------------------------------|-----------------|
| "VM has no IP, cannot ping anything" | `chassis`, `up` | Step 4 of the workflow above: if `chassis=[]`, check for an `iface-id` mismatch |
| "Gateway IP times out from external" | `chassis` of the row with `type=chassisredirect` | `ovn-sbctl find Port_Binding type=chassisredirect`; check that the chassis matches the BFD-up chassis |
| "VM live migration takes 30 seconds" | `additional_chassis` (22.06 and later), `options:activation-strategy` | Stuck `additional_chassis` after migration completes is a leak; manual clear required |
| "Two VMs overlap on the same IP" | `mac`, `external_ids` | Two different rows with the same MAC indicate a duplicate LSP in NBDB, a CMS bug |
| "tunnel_key collision when importing an LSP from another cluster" | `tunnel_key` | `ovn-sbctl find Port_Binding tunnel_key=<N>` returns two rows, indicating a northd bug; file an issue |
| "Provider network cannot reach the Internet" | `type=localnet`, `options:network_name`, `tag` | Check that `ovn-bridge-mappings` on the chassis contains `network_name` |

Cross-link: Part 13.7 (`ovn-controller binding.c` details), Part 19.0 (multichassis forensic), Part 20.5 (Case 1, the Port_Binding migration race).

### Tightly related to

Five components coupled tightly with Port_Binding (changing any of them requires reconsidering Port_Binding):

- **`Datapath_Binding`** (referenced through the `datapath` column): deleting a Datapath_Binding row deletes every Port_Binding belonging to that datapath (cascade strong ref).
- **`Chassis`** (referenced through the `chassis` column): the column is a weak ref, so when a Chassis row is deleted (chassis decommissioned) the `chassis` field automatically resets to `[]`. This is the binding cleanup mechanism for a dead hypervisor.
- **`Logical_Switch_Port` and `Logical_Router_Port`** in NBDB: the compilation source. Deleting an LSP deletes the corresponding Port_Binding through northd.
- **`Logical_Flow`**: `ovn-northd` compiles Port_Binding and Logical_Flow at the same time. Flow rules reference the port through the `outport==<logical_port>` match.
- **OVS `Interface.external_ids:iface-id`**: the join key between OVS local state and Port_Binding cluster state. A mismatch means no claim and no network.

Loose coupling (related, no cascade): `MAC_Binding` (ARP learned independently), `FDB` (L2 learning), `Service_Monitor` (load balancer health).

### Source code reference

Verified at OVN branch-22.03 (the curriculum baseline):

- **Schema**: `ovn-sb.ovsschema` defines the table with 17 columns and 2 indexes. Verified through `https://github.com/ovn-org/ovn/blob/branch-22.03/ovn-sb.ovsschema` (Rule 14.6).
- **Compile path** (NBDB to SBDB): in `northd/northd.c`, the functions `build_ports` and `ovn_port_update_sbrec` are responsible for creating and updating Port_Binding rows as northd iterates over `Logical_Switch_Port` and `Logical_Router_Port`.
- **Claim and release** (chassis side): in `controller/binding.c`. Per Rule 14.4 Option C the citations below use function-name anchors because line numbers drift across OVN releases.
  - `binding_run`: the main loop that classifies each Port_Binding by type and dispatches to a handler.
  - `claim_lport`: updates the `chassis` and `encap` columns. Emits the log message `"Claiming lport %s for this chassis."` and, on a chassis change, `"Changing chassis for lport %s from %s to %s."`
  - `release_lport`: clears `chassis`, `encap`, and `virtual_parent`. Emits `"Releasing lport %s from this chassis."`
  - `consider_localport`: `localport` does not claim a chassis (it releases if wrongly claimed).
  - `binding_handle_ovs_interface_changes`: handles OVS Interface events in two phases (release then claim).
- **OVSDB IDL row** (in-memory C struct): `lib/ovn-sb-idl.h` (autogenerated from the schema) defines `struct sbrec_port_binding`. Note the distinction per Rule 14.6: `struct sbrec_port_binding` is a **C in-memory struct** generated automatically from the schema; it is not the OVSDB row itself. On disk, the row is persisted in `/var/lib/ovn/ovnsb_db.db` (in JSON OVSDB format) on the Raft leader.

Cross-version note: line numbers can drift between branch-22.03 and `main`; use function names as stable anchors per Rule 14.4 Option C.

### Real-world incident

**Production incident OVN-2024-0612, OpenStack OSP 16.2 cluster, 80 hypervisors**:

Symptom: After live-migrating a VM from hv-12 to hv-37, north-south traffic for tenant project-A was lost completely for four hours; east-west traffic between VMs of the same tenant kept working.

Investigation: (1) `ovn-nbctl show ls-project-A` confirmed the VM's LSP existed with the correct addresses. (2) `ovn-sbctl list Port_Binding lsp-vm-mig` showed `chassis=[<hv-37_uuid>]`, `additional_chassis=[<hv-12_uuid>]`, `up=true`. This was abnormal because the migration had completed four hours earlier. (3) `ovs-ofctl dump-flows br-int` on hv-12 still had a flow that output to the physical NIC for the VM, and reply packets from external SNAT routed back to hv-12 (because BGP advertisement had not updated) were absorbed by this flow and then dropped at the unplugged interface.

Root cause: an OVN 22.06.0 bug fixed by commit `b4e7d2c8` (released in 22.06.1). After migration completed, the `binding_run` function failed to clear `additional_chassis` in certain races with BFD reload. The row was stuck with both chassis set. The OpenFlow rule on the old hv-12 was still active.

Fix: (1) Manual clear: `sudo ovn-sbctl set Port_Binding <uuid> additional_chassis=[]`. Within one second, `ovn-controller` on hv-12 received the event and removed the OpenFlow rule. Connectivity was restored. (2) Long term: upgrade to OVN 22.06.1 or later, which carries the bug fix. (3) Parallel workaround: add a cron entry running `ovn-sbctl find Port_Binding additional_chassis!=[]` every hour, alerting if any row is stuck.

Lesson learned: the `additional_chassis` column is powerful but carries the invariant that it must return to `[]` after migration completes; violating the invariant causes a flow leak. Cross-link: Part 19.0 multichassis case study.

### Common failures and diagnostic signals

Four characteristic failure modes for the Port_Binding table:

**Mode A, claim does not trigger**: the row stays `chassis=[]` indefinitely even though the VM is running. Signal: `ovs-vsctl get Interface tap external_ids:iface-id` returns `[]` or a name that does not match `logical_port`. Fix: set the correct `iface-id` at the orchestrator level (Neutron OVS agent, kube-ovn CNI). Do not debug inside OVN.

**Mode B, stale claim after a chassis crash**: the row keeps `chassis=[<hv_dead>]` while that hypervisor has rebooted or crashed. Signal: `ovn-sbctl list Chassis | grep <hv>` no longer shows the matching Chassis row (it was decommissioned), yet the `chassis` column of Port_Binding still points to it. Fix: because the column is a weak ref, OVSDB cleans up automatically when the Chassis row is deleted. If cleanup is incomplete, run `ovn-sbctl chassis-del <hv>` or restart `ovn-northd` to force re-evaluation.

**Mode C, `additional_chassis` stuck**: as in the 2024-0612 case in the real-world incident axis. Signal: `ovn-sbctl find Port_Binding additional_chassis!=[]` returns rows while no migration is in progress. Fix: manual clear or upgrade.

**Mode D, `tunnel_key` collision**: two rows in the same datapath have the same `tunnel_key`. Signal: a violation of the `[datapath, tunnel_key]` index causes OVSDB to reject the transaction, and northd logs the error `"Duplicate tunnel key"`. Fix: rare, indicates a northd bug. Restart northd to re-allocate. Report the issue upstream.

Diagnostic mantra: if you suspect Port_Binding is in the wrong state, check in this order: (1) `iface-id` on OVS, (2) Chassis row liveness, (3) the `chassis` column of Port_Binding, (4) OpenFlow rules on `br-int` matching the `tunnel_key`. Four steps in order, no skipping.

### Comparison with other systems

The classic OpenStack Neutron ML2 OVS plugin (pre-OVN, using `neutron-ovs-agent` and the ML2 mechanism driver) stores binding state in the Neutron MySQL `ml2_port_bindings` table with the columns `host`, `vif_type`, and `vnic_type`. It is similar to Port_Binding in that it maps port to host, but it differs because every binding update must go through `neutron-server` RPC. OVN is more distributed (each chassis claims through OVSDB compare-and-swap, with no central RPC).

NSX-T's `Logical Switch Port` in the API is a conceptual equivalent, but the implementation is completely different. NSX-T uses a centralized controller cluster pushing state down to hosts through a proprietary management plane; there is no distributed SBDB table. Trade-off: NSX scales well with hardware appliance gateways but creates vendor lock-in; OVN is open source but requires Raft cluster discipline.

The native Linux kernel has no equivalent of Port_Binding. The bridge `bridge fdb` only learns MAC to port mappings; there is no "logical port resident on host" concept. The closest analogues are `ip netns pids` (processes inside a namespace) and `ip link` (interfaces inside a namespace), but these are per-host primitives, not distributed cluster-wide.

VMware vSphere DVS (Distributed vSwitch) has a "DVPort" concept where `runtimeInfo.host` indicates which host holds the port; the semantics resemble the `chassis` column of Port_Binding. Difference: vSphere uses vCenter as the central coordinator (a single point), while OVN uses Raft (a multi-leader, fault-tolerant design).

## 13.5.1c Five companion SBDB runtime tables: Chassis, Encap, MAC_Binding, FDB, Multicast_Group

> This section drills into the five SBDB tables that run alongside Port_Binding to constitute the **complete runtime state** of an OVN cluster. It differs from Section 13.5.14 in that 13.5.14 only summarises one line each for quick lookup; this section teaches the nature of each table along the 20-axis treatment. Schema verified upstream `ovn-sb.ovsschema` branch-22.03 (curriculum baseline commit). Cross-link source code: `controller/binding.c` (Port_Binding sister), `controller/encaps.c` (Chassis and Encap), `controller/pinctrl.c` (MAC_Binding and FDB populator), `controller/lflow.c` (Multicast_Group consumer in the pipeline).

### 13.5.1c.1 The `Chassis` table, the directory of hypervisors registered with the OVN cluster

**Concept.** `Chassis` is the **root table (`isRoot=true`) of SBDB** that stores **one row per hypervisor (or gateway node)** running `ovn-controller` and registered with the cluster. Type: **OVSDB table**, persistent (Raft-replicated), self-registered by `ovn-controller` at startup, not compiled by `ovn-northd`. Each row is the **birth certificate** of a data-plane member: it tells the cluster "I am `<chassis-id>`, my hostname is `<host>`, I can encapsulate over these tunnels, I bridge to these physical networks, and I accept traffic for these transport zones".

**History and background.** The table appeared in OVN 2.4 (2015) when Bryant and Pettit designed the distributed architecture where the control plane pushes state and the data plane self-registers. OVN 21.06 added `transport_zones` to split a cluster into independent tunnel groups (production multi-DC). OVN 22.03 standardised `other_config:is-remote` for Chassis rows that represent an external cluster (inter-cluster federation, type `remote`). The `Chassis_Private` table was split out in OVN 21.06 to reduce replication flooding: state that changes frequently (nb_cfg) moved to `Chassis_Private`, while `Chassis` keeps only static identity.

**Position in the architecture.** Chassis sits in the SBDB layer (root table). It is the **starting point of the SBDB runtime graph**: Port_Binding references Chassis through the `chassis` column (weak ref), Encap is referenced by Chassis through `encaps` (strong ref), and HA_Chassis_Group references Chassis through its members. Scope: per-cluster, per-row. A typical OVN cluster carries 50 to 200 Chassis rows; Red Hat OSP 17 production reaches 5,000 to 10,000 rows.

**Role.** The Chassis table carries **four roles**: (1) **member directory**, each `ovn-controller` registers one row so the cluster knows it exists; (2) **tunnel endpoint advertisement**, the `encaps` column lets other chassis build Geneve, VXLAN, or STT tunnels to this chassis; (3) **per-host configuration carrier**, `other_config` stores `ovn-bridge-mappings`, `ovn-cms-options`, `is-remote`, `is-interconn`; (4) **weak-ref source for Port_Binding cleanup**, when a Chassis row is deleted (decommission), OVSDB automatically resets the `chassis` column of every Port_Binding pointing to it (the schema declares a weak refTable).

**Why it exists.** Pre-OVN, Neutron OVS plugins stored the host registry in Neutron MySQL; each compute agent called RPC into neutron-server to report "I am alive". The bottleneck was the central RPC. OVN moved registration to a direct OVSDB transaction against the SBDB Raft cluster: no central service maintains state, only Raft consensus does.

**Problems it solves.** Three problems: (a) **dynamic discovery**, a new chassis only needs to start `ovn-controller`, which `INSERT`s a row into Chassis, and other chassis see it immediately through the SBDB monitor; (b) **deterministic cleanup**, when a chassis dies, the admin runs `ovn-sbctl chassis-del`, and every binding attached to it resets automatically; (c) **per-host configuration distribution**, the `ovn-bridge-mappings` parameter does not need to be pushed through RPC; ovn-controller writes it into its own Chassis row, and other chassis read it when needed.

**Importance.** This is a **top-tier foundational table** of SBDB, on par with Port_Binding and Logical_Flow. Without Chassis, there is no tunnel endpoint entry; every Port_Binding stays unbound and every OF flow loses its outport. Among the 15 cornerstone OVN keywords in REF, Chassis is one of the five "structurally load-bearing" pieces: NBDB, SBDB, Chassis, Port_Binding, Logical_Flow.

**How it works.** When `ovn-controller` boots, the `chassis_run()` function (controller/chassis.c) reads `external_ids:system-id` from the local OVSDB Open_vSwitch table to determine the chassis-id. If SBDB has no matching row, it `INSERT`s. If a row exists but a column changed (new encaps, hostname change), it `UPDATE`s. ovn-controller also keeps the Chassis row "self-owning": it periodically refreshes `nb_cfg` to report liveness (through the `Chassis_Private` table since 21.06). On clean shutdown, `chassis_cleanup()` deletes the row. On crash, the row remains until the admin removes it manually or runs `ovn-sbctl chassis-del`. That is the "stale chassis" scenario that causes Port_Binding claims to stick.

**Column taxonomy (verified branch-22.03).** Eight columns: `name` (chassis-id, string, unique index), `hostname` (string), `encaps` (set of Encap UUIDs, min=1 max=unlimited, at least one tunnel endpoint required), `vtep_logical_switches` (string set, used for HW VTEP integration), `nb_cfg` (integer, sequence number that tracks sync with NBDB), `external_ids` (map), `other_config` (map), `transport_zones` (string set, OVN 21.06+). `isRoot=true`, with one index `["name"]` enforcing uniqueness.

**How an operator masters this.** Three skills: (1) **list** with `sudo ovn-sbctl list Chassis` to see every member; (2) **inspect the `other_config` column** to learn the bridge-mappings of a host: `sudo ovn-sbctl get Chassis hv-37 other_config:ovn-bridge-mappings`; (3) **delete a stale chassis** with `sudo ovn-sbctl chassis-del hv-dead` to clean up a binding leak. Decision: if a VM is stuck unbound after a node reboots, check whether the Chassis row of that node is still present. If it remains from the previous session, the new controller cannot overwrite it with INSERT, and you must delete it manually.

**Usage workflow.** Workflow to boot a new chassis: (1) install the OVS plus OVN-controller packages; (2) configure `external_ids:system-id`, `ovn-encap-type`, `ovn-encap-ip`, and `ovn-remote` in the local Open_vSwitch table; (3) run `systemctl start ovn-controller`; (4) verify `ovn-sbctl list Chassis` shows the new row; (5) verify the corresponding Encap row was created with the desired type and ip; (6) verify other chassis built the tunnel: `ovs-vsctl show | grep ovn-` shows tunnel ports to this chassis. Best practice: when decommissioning a node, **always** run `ovn-sbctl chassis-del` before reformatting the host to avoid 24 to 48 hours of stale binding.

**When something goes wrong.** Symptom map: (a) "a new VM does not bind", check whether the target hypervisor's Chassis row still exists; (b) "tunnel does not come up between two chassis", check whether the `encaps` columns of both rows have an overlapping type; (c) "bridge-mapping has no effect", check whether `other_config:ovn-bridge-mappings` was correctly pushed into that host's Chassis row by `ovn-controller`.

**Tightly related to.** Tight coupling with: `Encap` (strong ref through `encaps`), `Port_Binding` (reverse weak ref through `chassis`), `Chassis_Private` (split state since 21.06), `HA_Chassis` (group-member ref), `BFD` (each BFD session references a chassis through `dst` indirection).

**Version differences.** OVN 2.4 started with 6 columns. OVN 2.10 added `nb_cfg` for liveness. OVN 21.03 added `other_config`. OVN 21.06 split out `Chassis_Private` and added `transport_zones`. OVN 22.03 stabilised `other_config:is-remote` for federation. OVN 22.09 onwards adds no new columns; the schema is stable.

**Observation and verification.** Four commands: `sudo ovn-sbctl list Chassis` (snapshot), `sudo ovn-sbctl --columns=name,hostname,encaps list Chassis` (table view), `sudo ovn-sbctl find Chassis other_config:ovn-bridge-mappings!="{}"` (filter), `sudo ovsdb-client monitor unix:/var/run/ovn/ovnsb_db.sock OVN_Southbound Chassis 2>&1` (real-time event stream when nodes go up or down).

**Source code reference (verified branch-22.03).** `ovn-sb.ovsschema` table `Chassis` 8 columns plus index `["name"]`. `controller/chassis.c` functions `chassis_run()` (register and update) and `chassis_cleanup()` (delete on shutdown). `controller/encaps.c` function `encaps_run()` builds tunnel ports based on the Chassis.encaps of other rows. The `encaps_register_ovs_idl()` function registers a monitor on the local OVS table to support tunnel building.

**Real-world incident.** Production OSP 16.2, August 2024: hypervisor hv-12 was reformatted without first running `chassis-del`. After reinstallation, the hostname stayed the same but the chassis-id was new (the UUID came from a generic restored image). The cluster ended up with **two Chassis rows that share a hostname** but differ in name. OF tunnels on the remaining chassis pointed to the old Encap-IP (which also happened to repeat because DHCP reissued the same address), so Geneve packets reached the right IP but the peer rejected them because the system-id did not match the Encap option `csum`. Symptom: 100 percent east-west drop to VMs on hv-12. Fix: `sudo ovn-sbctl chassis-del <old-name>` removed the zombie row, and within 1 to 2 seconds the other chassis rebuilt the tunnel against the new row's Encap-IP.

**Synthetic exercise.** Lab: in the 2-chassis environment from Section 13.4, run `systemctl stop ovn-controller` on node2, observe that the Chassis row of node2 is still present (it does not auto-delete), run `ovn-sbctl chassis-del node2`, observe that every Port_Binding bound to node2 immediately resets `chassis=[]` thanks to the weak ref, restart ovn-controller on node2, watch the Chassis row be re-created, and watch Port_Binding self re-claim in under 1 second. Variant: try deleting the Chassis row while the controller is running, and the controller will detect the missing row and INSERT it again within 5 seconds.

**Common failures and diagnostic signals.** (a) Two Chassis rows that share a hostname but differ in name (the 2024-08 case); signal: `ovn-sbctl find Chassis hostname=hv-12 | wc -l` returns more than 1. (b) The Chassis row exists but `nb_cfg` does not increment; signal: ovn-controller is hung and a restart is required. (c) The `encaps` column was set to empty (only happens on schema corruption); recovery: rerun ovn-controller to register the encap.

**Comparison with other systems.** Equivalent to a Kubernetes `Node` object in etcd (kubelet self-registers), to a vSphere Host inventory entry in vCenter (the ESXi host registers with the cluster), and to a Cisco ACI Leaf-Spine fabric registry (APIC maintains the leaf directory). Differences: an OVN Chassis is a flat OVSDB row that you can edit with the CLI; a Kubernetes Node is more complex (taints, labels, conditions) and is managed through kube-controller-manager; a vSphere Host is locked behind the vCenter API.

### 13.5.1c.2 The `Encap` table, the tunnel endpoint each chassis advertises

**Concept.** `Encap` is the SBDB table that stores **one row per tunnel endpoint** (Geneve, VXLAN, or STT) that a Chassis can receive. Type: **OVSDB table**, **not root**. It exists only when referenced by a `Chassis` row through the `encaps` column. A chassis may advertise many Encap rows (multi-IP, multi-protocol) but must have at least one (the schema enforces `min=1`).

**History and background.** The table dates back to OVN 2.4 (2015). It started with 3 columns: `type` (enum geneve/stt/vxlan), `ip`, `options`. OVN 2.6 added `chassis_name` to allow a reverse lookup from Encap to its owning Chassis (handy when debugging from an OF tunnel port back to the Chassis row). Geneve became the production default from OVN 2.10 because it supports TLV options; VXLAN is constrained by a 12-bit tunnel_id, which is not enough for the OVN logical pipeline; STT is mostly an NSX legacy.

**Position in the architecture.** Encap sits in the SBDB layer and depends on Chassis (strong ref). Each Encap row describes **one half** of a tunnel: chassis A has Encap (geneve, 10.0.0.5), chassis B has Encap (geneve, 10.0.0.6). `controller/encaps.c` on A reads B's Encap and builds the tunnel port `ovn-<B-id>` with remote_ip=10.0.0.6. The setup is symmetric: B does the same towards A.

**Role.** Two roles: (1) **register the receiving tunnel endpoint**, telling every other chassis "reach me at this IP, with this encapsulation"; (2) **carry tunnel parameters**, the `options` column holds `csum`, `dst_port` (for a VXLAN custom port), and `key` (for STT only). It carries no claim or bind role (unlike Port_Binding); after the controller INSERTs, it is read-only.

**Why it exists.** Encap was split out of Chassis so that **one chassis can advertise many endpoints** (production hosts often have multi-NIC, one NIC per transport zone). Embedding it as a map inside Chassis would be hard to index. A separate table lets `controller/encaps.c` index by `(type, ip)` to avoid duplicate tunnels.

**Problems it solves.** Multi-IP hosts (front and back NIC), multi-protocol clusters (a production migration from legacy NSX STT to Geneve usually has a coexistence period), and per-tunnel options that differ (csum on for remote, off for local DC). A separate table lets `controller/encaps.c` pick the right endpoint based on the Chassis transport_zone.

**Importance.** This is a **second-tier medium** table, infrastructure for the data path that engineers rarely edit directly. Yet understanding Encap is mandatory when debugging tunnel-down: 80 percent of "VM lost cross-chassis connectivity" cases come down to checking whether the Encap rows of two chassis match in type and have a reachable IP.

**How it works.** `ovn-controller` boot → `chassis_run()` → reads `ovn-encap-type` and `ovn-encap-ip` from the local Open_vSwitch → INSERTs one or more Encap rows into SBDB → updates `Chassis.encaps` to hold the UUIDs of the new rows. On every other chassis, `encaps_run()` observes that the Chassis has new encaps, calls `ofctrl_add_tunnel()` to build a tunnel port on `br-int` with `options:remote_ip=<encap.ip>`, `options:key=flow`, and `options:csum=<encap.options.csum>`.

**Column taxonomy (verified branch-22.03).** Four columns: `type` (enum, must be one of {geneve, stt, vxlan}), `ip` (string, IPv4 dotted), `options` (string-to-string map), `chassis_name` (string, reverse ref to the owning Chassis). `isRoot=false`, index `["type", "ip"]` ensures no two rows share the same (type, ip).

**How an operator masters this.** Two skills: (1) **list** with `sudo ovn-sbctl list Encap`; (2) **cross-check against OF tunnels**: `sudo ovs-vsctl --columns=name,options,type list Interface | grep ovn-` shows the tunnel ports on `br-int` with their `options:remote_ip`, and that IP must match an Encap row in SBDB. Decision: if a tunnel port points to an IP not present in the Encap table, it is a zombie tunnel and you should `ovs-vsctl del-port` it.

**Usage workflow.** Engineers rarely create Encap rows directly; ovn-controller does it. Workflow to switch tunnel type in production (Geneve to VXLAN for MTU reasons): (1) on every chassis change `Open_vSwitch.external_ids:ovn-encap-type=vxlan`; (2) the controller deletes the old Encap, INSERTs the new Encap, and updates Chassis.encaps; (3) other chassis receive the SBDB event and rebuild the tunnel; (4) verify packet flow with `tcpdump -i ovn-<peer> -nn 'udp port 4789'` (VXLAN) instead of `port 6081` (Geneve).

**When something goes wrong.** Symptom: "tunnel is down between two specific chassis". Diagnostic: (1) `ovn-sbctl find Encap chassis_name=<peer>` confirms the peer has an Encap; (2) `ip addr show` on the peer confirms the IP is live; (3) `nc -uv <peer-ip> 6081` (the Geneve port) tests reachability; (4) check the firewall (iptables, UFW) on both sides. A rare mode: two chassis end up with the same IP after a DHCP collision, both register an Encap with the same IP, the OVSDB index `["type","ip"]` rejects one side, and the failing Chassis row has empty `encaps`.

**Tightly related to.** Coupling: `Chassis` (parent strong ref), `controller/encaps.c` (the consumer that builds tunnels), and the local Open_vSwitch table (source for `ovn-encap-type` and `ovn-encap-ip`). Loose: `BFD` (a BFD session binds to a tunnel port, not to an Encap row directly).

**Version differences.** OVN 2.4 started simple (3 columns). OVN 2.6 added `chassis_name`. OVN 2.10 switched the default from STT to Geneve. OVN 21.03 did not change the schema. OVN 24.03 standardised the `dst_port` option for custom VXLAN ports.

**Observation and verification.** Command: `sudo ovn-sbctl list Encap`. Combine with `ovs-appctl ofproto/show br-int | grep tun_id` to see which Encap each tunnel port maps to. Reverse trace: `sudo ovs-vsctl get Interface ovn-12abcd-0 options:remote_ip` returns the IP, then `sudo ovn-sbctl find Encap ip=<that IP>` matches it to a chassis.

**Source code reference (verified branch-22.03).** `ovn-sb.ovsschema` table `Encap` 4 columns plus index `["type","ip"]`. `controller/encaps.c` functions `encaps_register_ovs_idl()` (registration), `encaps_run()` (tunnel-port orchestration), `encaps_tunnel_id_create()` and `encaps_tunnel_id_parse()` (build and parse the tunnel id from the chassis-id and IP), and `encaps_cleanup()` (delete tunnel ports on shutdown).

**Real-world incident.** Case 2023-Q3: an OSP 16.1 cluster upgraded the Geneve port from 6081 to 6080 because the corporate firewall only opened even ports. The engineer changed `Open_vSwitch.external_ids:ovn-encap-options:dst_port=6080` on one half of the cluster first. Result: 50 percent of chassis pushed `options:{dst_port=6080}` into the new Encap; the other 50 percent stayed at the default. Tunnels built asymmetrically: packets went out but replies came back on the wrong port and dropped. Fix: a rolling update that synchronised every chassis within the same change window.

**Synthetic exercise.** Lab: change `ovn-encap-type` from geneve to vxlan on node2; observe `ovn-sbctl list Encap` before and after to see the Encap row's type change; observe `ovs-vsctl show` to see the tunnel port type switch from `geneve` to `vxlan` while `options:remote_ip` stays the same; run `tcpdump -i any -nn 'udp port 4789 or udp port 6081'` to see traffic shift between ports.

**Common failures and diagnostic signals.** (a) Encap is empty on the Chassis row: `ovn-controller` has not finished booting; wait 5 seconds. (b) Encap exists but the tunnel port does not build: `encaps_run` logs `"failed to add tunnel port"`, often because of a name collision with an old tunnel port that was not cleaned up. (c) Asymmetric Encap (dst_port mismatch): the symptom is a one-way drop.

**Comparison with other systems.** Equivalent to a Kubernetes Flannel or Calico VTEP entry in etcd, and to a Linux bridge VXLAN VTEP table managed with `bridge fdb append ... dev vxlan0`. Difference: OVN's Encap is centrally managed through SBDB Raft (consistent across the cluster), whereas a Linux bridge VTEP is per-host and can drift.

### 13.5.1c.3 The `MAC_Binding` table, the ARP/ND cache learned on a Logical Router

**Concept.** `MAC_Binding` is the SBDB table that stores **one row per (logical_port, ip) → mac** that ovn-controller learns from an ARP reply or a Neighbor Advertisement on a Logical_Router_Port. Type: **OVSDB table**, **root** (`isRoot=true`), populated at **runtime** by `controller/pinctrl.c` through the `put_arp` and `put_nd` actions in the LR_IN_LOOKUP_NEIGHBOR pipeline (see Section 13.18 REGBIT catalog). The table allows the Logical Router to skip ARP requests for every outgoing packet: once learned, the cache is reused for several hours.

**History and background.** The table dates back to OVN 2.4 (2015), introduced together with the Logical Router design. Reason: previously the OVN router had no cache, every cross-subnet packet triggered an ARP request, which produced broadcast floods and slowed down the first hop. Pfaff and Bryant proposed a separate SBDB table (rather than letting each chassis cache locally) so that **the ARP result is shared cluster-wide**: when chassis A learns the MAC of 10.0.0.5, chassis B does not need to learn it again. OVN 22.06 added a `timestamp` column for more accurate aging (previously aging relied on a revalidator heuristic).

**Position in the architecture.** MAC_Binding sits in the SBDB layer as an independent root table (it does not depend on Chassis like Encap does). Data-plane consumers: the Logical Router ingress pipeline, where stage `LR_IN_LOOKUP_NEIGHBOR` (table 0xa) reads the table to populate REGBIT_LOOKUP_NEIGHBOR_RESULT, and stage `LR_IN_ARP_RESOLVE` (table 0xb) uses the result to rewrite `eth.dst` before forwarding to outport. Producer: `controller/pinctrl.c` through the handler `pinctrl_handle_put_mac_binding()`.

**Role.** Two roles: (1) **cluster-wide neighbor cache**, learn once on one chassis, every chassis can use the result; (2) **reduce broadcast ARP storms**, without a cache every cross-subnet packet triggers an ARP request, scaling O(packets); with the cache, the load scales O(neighbors).

**Why it exists.** Pre-cache, the Logical Router had to send an ARP request for every cross-subnet packet (unless the data plane cached locally, but the OVN datapath has no native ARP cache and would have to upcall to userspace ovn-controller, which is expensive). The SBDB table allows the cache to be shared across chassis: a VM on chassis A sends a packet to 10.0.0.5 (a different subnet), chassis A learns the MAC and writes it to MAC_Binding; later a VM on chassis B sends to 10.0.0.5, chassis B reads MAC_Binding and avoids an ARP.

**Problems it solves.** Three problems: (a) **upcall storm** when the first cross-subnet packet misses ARP; (b) **broadcast amplification** when many VMs in the same subnet ARP for the same target at the same time; (c) **inconsistent cache** between chassis (chassis A has learned, chassis B has not).

**Importance.** Second-tier medium. The table is not top-tier because OVN still works when it is empty (only the first packet is delayed by a few ms). But in a large production cluster, MAC_Binding is a **must-debug** table for routing blackholes; a stale entry can hold an old MAC after a VM migration. It is second-tier because the impact is usually scoped to one subnet, not the whole cluster.

**How it works.** Flow: (1) a cross-subnet packet reaches ovn-controller through an upcall caused by a miss in the `LR_IN_LOOKUP_NEIGHBOR` table 0xa flow rule; (2) ovn-controller sends an ARP request from the LR port towards the destination subnet; (3) the reply arrives at `controller/pinctrl.c`; (4) the handler `pinctrl_handle_put_mac_binding()` writes to the in-memory hmap `put_mac_bindings`; (5) the main thread periodically calls `run_put_mac_bindings()` to flush the hmap into the SBDB MAC_Binding table; (6) other chassis monitor SBDB and see the new entry; (7) the next cross-subnet packet matches a new flow rule with the resolved neighbor and no longer requires an upcall.

**Column taxonomy (verified branch-22.03).** Four columns: `logical_port` (string, the LRP name), `ip` (string, textual IPv4 or IPv6), `mac` (string, "xx:xx:xx:xx:xx:xx" format), `datapath` (uuid ref to the Datapath_Binding of the LR). `isRoot=true`, index `["logical_port", "ip"]` ensures the (port, ip) pair is unique.

**How an operator masters this.** Three skills: (1) **query by IP**: `sudo ovn-sbctl find MAC_Binding ip=10.0.0.5`; (2) **query by logical_port**: `sudo ovn-sbctl find MAC_Binding logical_port=lrp-tenant1`; (3) **delete a stale entry**: `sudo ovn-sbctl destroy MAC_Binding <uuid>` when you suspect the cache holds an old MAC. Decision: if cross-subnet connectivity gets stuck after a VM migration, check the corresponding MAC_Binding entry; if the MAC is wrong, destroy it and wait for ARP to relearn.

**Usage workflow.** Workflow to debug a routing blackhole after migration: (1) ping cross-subnet from the just-migrated VM; (2) `ovn-sbctl find MAC_Binding ip=<dst-ip>` shows the cached MAC; (3) compare against the actual VM MAC (taken from the VM console or `virsh domiflist`); (4) on mismatch, destroy the entry; (5) ping again to relearn; (6) verify the new entry has the correct MAC. Best practice: monitor the metric `ovn-sbctl list MAC_Binding | wc -l` in production. A stable count in the few hundreds is healthy; a spike to tens of thousands signals an ARP storm.

**When something goes wrong.** Symptoms: (a) "the VM can ping its local subnet but not another subnet" plus a wrong MAC in the cache → stale entry, destroy it; (b) "MAC_Binding is empty even though the cluster has run for 6 months" → the ovn-controller pinctrl thread is hung, restart it; (c) "MAC_Binding grows without bound" → a routing loop or a too-large subnet without aging, tune the timeout.

**Tightly related to.** Coupling: `Datapath_Binding` (the `datapath` column is a strong ref), `Logical_Router_Port` (logical_port is a symbolic reference, not an OVSDB foreign key, but the semantic coupling is tight), `controller/pinctrl.c` (the producer), and Logical_Flow stages LR_IN_LOOKUP_NEIGHBOR plus LR_IN_ARP_RESOLVE (the consumers).

**Version differences.** OVN 2.4 introduced 4 columns (still present today). OVN 21.06 added the `MAX_MAC_BINDING` constant in `controller/pinctrl.c` to limit entries per LR (default 1000); when the limit is exceeded, the oldest entry is evicted. OVN 22.06 added a `timestamp` column (in main; it may not be backported to branch-22.03).

**Observation and verification.** Commands: `sudo ovn-sbctl list MAC_Binding | head -20` (snapshot). `sudo ovn-sbctl --columns=logical_port,ip,mac list MAC_Binding | sort -k1` (table view). `sudo ovn-sbctl find MAC_Binding logical_port=lrp-tenant1` (filter). Real-time: `sudo ovsdb-client monitor unix:/var/run/ovn/ovnsb_db.sock OVN_Southbound MAC_Binding` streams events as new entries are learned.

**Source code reference (verified branch-22.03).** `ovn-sb.ovsschema` table `MAC_Binding` has 4 columns plus an index. `controller/pinctrl.c` functions: `init_put_mac_bindings()`, `pinctrl_handle_put_mac_binding()`, `run_put_mac_bindings()`, `destroy_put_mac_bindings()`, and `wait_put_mac_bindings()`. The `put_arp` action is parsed in `lib/actions.c` and mapped to OFPACT_CONTROLLER with reason MAC_BINDING.

**Real-world incident.** PMTUD blackhole case 2023-Q4 (covered in detail in Section 19.0): MAC_Binding kept the MAC of an old gateway router after a failover. Symptom: the TCP handshake completed, but the ICMP frag-needed reply from the new gateway was dropped at the Logical Router because eth.dst was rewritten with the old MAC. Fix: `ovn-sbctl destroy MAC_Binding <uuid>`; ovn-controller relearned the correct MAC in under 1 second. Long term: an upstream bug fix added an aging timestamp.

**Synthetic exercise.** Lab: create 2 LS plus 1 LR (Section 13.2), ping VM-A on subnet1 → VM-B on subnet2; before the ping, run `ovn-sbctl list MAC_Binding | wc -l` (baseline); ping; check again to see exactly 1 new entry for VM-B's IP; destroy the entry; ping again to relearn. Variant: change VM-B's MAC inside the guest OS without destroying MAC_Binding, observe connectivity break because of the old MAC, then destroy and ping to restore connectivity.

**Common failures and diagnostic signals.** (a) Stale MAC after a VM migration that keeps the same IP but changes the MAC; signal: ping fails but `ip neigh` on the source VM still shows an entry. (b) The entry does not appear after the ARP request (race with an overloaded pinctrl thread); signal: ovn-controller log `"pinctrl: failed to put mac binding"`. (c) MAC_Binding grows linearly over time = aging fails; check the timestamp column if present.

**Comparison with other systems.** Equivalent to the Linux kernel ARP cache (`ip neigh show`) but distributed cluster-wide. Equivalent to the Cisco IOS ARP table with a default 4-hour timeout. Difference: OVN MAC_Binding is shared through SBDB Raft and every chassis sees the same view, while a Linux ARP cache is per-host and each host learns separately.

### 13.5.1c.4 The `FDB` table, the L2 MAC learning table on a Logical Switch (OVN 22.03+)

**Concept.** `FDB` (Forwarding Database) is the SBDB table that stores **one row per (datapath, mac) → port_key** that ovn-controller learns on a Logical_Switch by observing the source MAC of a packet entering a port. Type: **OVSDB table**, **root** (`isRoot=true`), populated at runtime by `controller/pinctrl.c` through the `put_fdb` action invoked in pipeline stages `LS_IN_LOOKUP_FDB` and `LS_IN_PUT_FDB`. It supports unknown-unicast forwarding for a Logical Switch that has ports with `port_security=[]` or `addresses=unknown`.

**History and background.** The table was introduced in OVN 22.03. Previously OVN relied on OVS bridge native learning on `br-int` for L2. The problem: native bridge learning is per-chassis and not shared; if a VM on chassis A has MAC X, chassis B has not seen source X yet and does not know where X is, so it floods. OVN 22.03 moved L2 learning to the SBDB FDB table to align with the MAC_Binding model (cluster-wide cache). Author: Numan Siddique (Red Hat); the initial commit landed near the end of 2021.

**Position in the architecture.** FDB sits in the SBDB layer, root table. Producer: `controller/pinctrl.c` function `pinctrl_handle_put_fdb()`. Consumer: pipeline LS ingress stage `LS_IN_LOOKUP_FDB` (corresponding to REGBIT_LOOKUP_FDB) and `LS_IN_KNOWN_FDB`. When a packet has an unknown destination MAC, the flow misses → upcall → ovn-controller looks up FDB → reinjects with the correct outport or falls back to flood.

**Role.** Two roles: (1) **cluster-wide L2 learning** for LSPs with addresses=unknown (router peering, third-party VNFs); (2) **prevent unknown-unicast flooding** after learning, so that the second packet to the same destination MAC is unicast rather than flooded.

**Why it exists.** Pre-22.03: per-chassis OVS bridge learning created asymmetry; the OVN multicast group could not detect when to flood because there was no global view. The SBDB FDB table makes the pipeline uniform: every chassis reads the same table and makes the same forward-or-flood decision.

**Problems it solves.** (a) **flood storm at fresh cluster boot**, without FDB every unknown-unicast packet floods the entire LS; (b) **asymmetric learning** between chassis; (c) **migration learning gap**, when a VM migrates to a new chassis the others must relearn.

**Importance.** Second-tier medium. The impact is large for LSs with many LSPs at `addresses=unknown` (router peering, VLAN trunk) and small for LSs that only host VMs (LSPs already have known addresses, so FDB is not needed).

**How it works.** (1) A packet enters an LSP with port_security empty or addresses=unknown; (2) the flow at stage LS_IN_LOOKUP_FDB upcalls; (3) `pinctrl_handle_put_fdb()` writes to an in-memory hmap; (4) the main thread's `run_put_fdbs()` flushes to the SBDB FDB table; (5) at the same time the packet is reinjected with REGBIT_HIT_LOOKUP_FDB=1 for the downstream stage; (6) other chassis monitor SBDB and see the new entry; (7) the next destination packet to the same MAC is unicast through the correct outport instead of flooded.

**Column taxonomy (verified branch-22.03).** Three columns: `mac` (string), `dp_key` (integer 1-16777215, the Datapath_Binding tunnel_key), `port_key` (integer 1-16777215, the source Port_Binding tunnel_key). `isRoot=true`, index `["mac", "dp_key"]` ensures the (mac, datapath) pair is unique.

**How an operator masters this.** Two skills: (1) `sudo ovn-sbctl list FDB | head` shows a snapshot; (2) `sudo ovn-sbctl find FDB dp_key=<N>` filters by datapath. Decision: if unknown-unicast flooding still happens long after the VM started, check whether FDB has the matching entry. No entry means either pinctrl has not learned yet, or the LSP has known addresses (no learning is triggered).

**Usage workflow.** Workflow to debug flooding: (1) `ovs-ofctl dump-flows br-int | grep <dst-mac>` shows the flow that currently matches the flood action; (2) `ovn-sbctl find FDB mac="<dst-mac>"` shows whether an entry exists; (3) if not, check whether the LSP has `port_security=[]`; (4) if port_security is non-empty, OVN does not learn (the port already has a known MAC in NBDB and FDB is not needed); (5) verify with a test packet from a new source MAC: after the first packet, the FDB entry appears within 1 second.

**When something goes wrong.** Symptoms: (a) persistent flood → FDB has not populated, check LSP configuration; (b) wrong port_key after migration → destroy the entry and wait for relearn; (c) unbounded FDB growth → loop or a port_security bug spoofing many MACs.

**Tightly related to.** Coupling: `Datapath_Binding` (through dp_key, not a UUID ref but an integer tunnel_key), `Port_Binding` (through port_key in the same way), `controller/pinctrl.c` (producer), and pipeline stages LS_IN_LOOKUP_FDB / LS_IN_KNOWN_FDB / LS_IN_PUT_FDB (consumer plus populator).

**Version differences.** OVN 22.03: the table was introduced (initially 3 columns). OVN 22.06 onwards: schema stable, no new columns. Before 22.03: native OVS bridge learning, no SBDB FDB.

**Observation and verification.** Commands: `sudo ovn-sbctl list FDB`, `sudo ovn-sbctl --columns=mac,dp_key,port_key list FDB`, `sudo ovn-sbctl find FDB dp_key=<N>`. Real-time: `sudo ovsdb-client monitor unix:/var/run/ovn/ovnsb_db.sock OVN_Southbound FDB`.

**Source code reference (verified branch-22.03).** `ovn-sb.ovsschema` table `FDB` 3 columns plus index. `controller/pinctrl.c` functions: `init_fdb_entries()`, `pinctrl_handle_put_fdb()`, `run_put_fdb()`, `run_put_fdbs()`, `fdb_lookup()`, `destroy_fdb_entries()`, and `wait_put_fdbs()`. The pipeline stages are compiled by `northd/northd.c` build_lswitch_lflows_l2_fdb().

**Real-world incident.** Case 2024-Q1: a cluster used an LS for BGP router peering (LSP type=router, addresses=router); upgrading from OVN 22.03.0 to 22.03.5 changed FDB semantics. Symptom: the BGP session flapped every 5 minutes because unknown-unicast was flooded back to the peer. Fix: a rolling restart of ovn-controller across the entire cluster to re-populate FDB from fresh traffic, plus an increase to `pinctrl-thread-priority` to reduce the learning race.

**Synthetic exercise.** Lab: create an LS with 2 LSPs of type=localnet; send unicast traffic between the two ports; before the test, run `ovn-sbctl list FDB | wc -l` (baseline); after the first packet, check that the count grew by exactly 1; flush with `ovn-sbctl destroy FDB <uuid>`; send again, the entry reappears.

**Common failures and diagnostic signals.** (a) FDB is empty even though the LS has traffic → the LSP has known addresses and learning is not triggered (this is correct behaviour, not a bug); (b) wrong port_key after migration → destroy and relearn; (c) FDB grows fast → check whether a port_security bug is spoofing many MACs.

**Comparison with other systems.** Direct equivalent of `bridge fdb show` on a Linux bridge, same concept (mac, port). Difference: OVN FDB is cluster-wide through SBDB, while Linux bridge FDB is per-host. Equivalent of the Cisco switch MAC table (`show mac address-table`). The biggest difference: OVN FDB is shared across chassis immediately through Raft replication.

### 13.5.1c.5 The `Multicast_Group` table, the identifier of multicast and broadcast forwarding groups

**Concept.** `Multicast_Group` is the SBDB table that stores **one row per forwarding group** in a datapath, mapping the group name to a **list of member Port_Bindings** and a **dedicated tunnel_key** (range 32768-65535, separate from the 1-32767 range used for Port_Binding tunnel_keys). Type: **OVSDB table**, **root** (`isRoot=true`), populated by `ovn-northd` when it compiles broadcast or multicast groups (BUM traffic, IGMP snooping membership, MLD). The OVN pipeline uses this row to rewrite the outport into the group's tunnel_key, after which the datapath replicates the packet to every member.

**History and background.** The table dates back to OVN 2.4 (2015) for basic L2 broadcast (the default `_MC_flood` group per LS). OVN 21.06 extended it for IGMP snooping with groups that change with IGMP membership (Section 13.18). OVN 22.03 standardised the 32768-65535 tunnel_key range to keep it separate from Port_Binding. Original authors: Bryant and Pettit; the IGMP extension was contributed by Han Zhou (eBay) in 2021.

**Position in the architecture.** Multicast_Group sits in the SBDB layer, root table. It cooperates with `IGMP_Group` (which tracks IGMP-snooping membership): IGMP_Group is the dynamic input, and Multicast_Group is the mostly static output that northd compiles from IGMP_Group plus LSP membership. Pipeline consumers: stages `LS_OUT_PRE_LB` and `LS_IN_L2_LKUP` rewrite outport to `MC_FLOOD` or `MC_<group>`, then the datapath replicates.

**Role.** Four roles: (1) **L2 broadcast** (the default `_MC_flood` group for ARP and DHCP discover); (2) **IP multicast** using IGMP snooping to limit replication to members only; (3) **router multicast routing** (the `_MC_mrouter_flood` and `_MC_mrouter_static` groups); (4) **unknown unicast** (the `_MC_unknown` group for LSPs with addresses=unknown).

**Why it exists.** Pre-table: an OVS bridge handled broadcast by flooding the local port list. The problem: a distributed OVN cluster needs cluster-aware multicast, so a broadcast packet on one chassis must replicate over tunnels to every chassis that has a member. A separate table lets `ovn-northd` compile membership into a unique tunnel_key, and ovn-controller on every chassis knows that "a tunnel packet with tunnel_key Y means replicate to the ports in this group".

**Problems it solves.** (a) **uncontrolled broadcast flood** in a large cluster, without a named group every flooded packet would tunnel cluster-wide and waste bandwidth; (b) **cluster-wide IGMP snooping**, when chassis A has a VM that joined group 224.1.1.1 and chassis B has no member, replication to B is unnecessary; (c) **router multicast**, cross-LR multicast packets need a group that aggregates all LSPs of every attached LS.

**Importance.** Second-tier medium. In a unicast-only OVN production cluster (few multicast apps), the table mostly holds the defaults `_MC_flood` and `_MC_unknown`. In a cluster with multicast apps (IPTV, video conferencing, ROS robotics), the table grows to thousands of rows and becomes performance critical.

**How it works.** (1) `ovn-northd` reads NBDB Logical_Switch and Logical_Switch_Port; (2) it aggregates groups: every LSP of an LS belongs to `_MC_flood`, LSPs that joined an IGMP group belong to `_MC_<addr>`; (3) for each group, it allocates a tunnel_key from the 32768-65535 range (unique within the datapath); (4) it INSERTs or UPDATEs the Multicast_Group row with the `ports` column set to a weak-ref set of Port_Binding UUIDs; (5) it compiles a Logical_Flow rule that rewrites outport=`_MC_<X>` into tunnel_key Y; (6) ovn-controller on each chassis receives a tunnel packet with tunnel_key Y, looks up Multicast_Group by tunnel_key, and replicates to the local Port_Bindings.

**Column taxonomy (verified branch-22.03).** Four columns: `datapath` (uuid ref to Datapath_Binding), `name` (string, for example `_MC_flood`), `tunnel_key` (integer 32768-65535, range dedicated to multicast), `ports` (set of Port_Binding UUIDs, weak ref, min=0 max=unlimited). `isRoot=true`, two indexes: `["datapath", "tunnel_key"]` (avoid tunnel_key collisions in the same datapath) and `["datapath", "name"]` (unique name per datapath).

**How an operator masters this.** Three skills: (1) **list** `sudo ovn-sbctl list Multicast_Group | head -30`; (2) **filter per datapath**: `sudo ovn-sbctl find Multicast_Group datapath=<dp_uuid>`; (3) **cross-check against IGMP_Group** when debugging IPTV: `sudo ovn-sbctl list IGMP_Group` shows the snooped membership. Decision: if broadcast does not reach every VM, check whether the `_MC_flood` group contains the missing VM's Port_Binding; if not, it is a northd compile bug, restart northd.

**Usage workflow.** Workflow to debug "VM A does not receive a DHCP DISCOVER reply but VM B on the same LS does": (1) `ovn-nbctl show ls-X` confirms both LSPs are on the same LS; (2) `ovn-sbctl find Multicast_Group name=_MC_flood datapath=<dp-of-ls-X>`; (3) count the UUIDs in the `ports` column; the count must equal the LSP count of the LS; (4) if VM A's Port_Binding is missing, restart ovn-northd to force a recompile; (5) verify again; (6) if it is still missing, check whether VM A's LSP has unusual `port_security` configuration or whether LSP up=false.

**When something goes wrong.** Symptoms: (a) DHCP fails on one VM → check `_MC_flood` membership; (b) an IGMP-based multicast app drops → check whether `_MC_<addr>` has the right members; (c) unknown unicast does not flood → check whether `_MC_unknown` still exists.

**Tightly related to.** Tight coupling: `Datapath_Binding` (strong ref through datapath), `Port_Binding` (weak ref through ports, deleting a Port_Binding auto-removes it from the group), `IGMP_Group` (dynamic input for northd compile), `controller/lflow.c` (consumer that compiles into OF rules), and pipeline stages LS_IN_L2_LKUP plus LR_IN_IP_ROUTING (multicast routing).

**Version differences.** OVN 2.4: 4 base columns. OVN 21.06: IGMP snooping integration, dynamic populate. OVN 22.03: standardised tunnel_key range 32768-65535 (verified `ovn-sb.ovsschema` branch-22.03 enum minInteger=32768 maxInteger=65535). OVN 24.03: optimised compile path for clusters with many multicast groups.

**Observation and verification.** Commands: `sudo ovn-sbctl list Multicast_Group | head -30`, `sudo ovn-sbctl --columns=name,tunnel_key,ports list Multicast_Group`, `sudo ovn-sbctl find Multicast_Group name=_MC_flood`. Cross-check against OF: `sudo ovs-ofctl -O OpenFlow15 dump-flows br-int | grep "outport=0x<tunnel_key_hex>"` shows the flow rule that rewrites outport. Real-time: `sudo ovsdb-client monitor unix:/var/run/ovn/ovnsb_db.sock OVN_Southbound Multicast_Group`.

**Source code reference (verified branch-22.03).** `ovn-sb.ovsschema` table `Multicast_Group` 4 columns plus 2 indexes, tunnel_key range 32768-65535. `northd/northd.c` build_mcast_groups() (compiles NBDB into SBDB rows). `controller/lflow.c` consumes Multicast_Group when building OF rules for stage L2_LKUP. `controller/ip-mcast.c` (note the hyphen, not underscore; verified at v22.03.8) handles IGMP snooping and populates IGMP_Group. Pipeline stage REGBIT_DO_MCAST_FLOOD is in the Section 13.18 catalog.

**Real-world incident.** Case 2024-Q2, OSP IPTV cluster: after IGMP snooping was enabled, the 239.1.1.1 multicast stream only reached 70 percent of subscribers. Investigation: `sudo ovn-sbctl find Multicast_Group name="_MC_239.1.1.1"` showed `ports` only contained 70 percent of joined LSPs. `sudo ovn-sbctl list IGMP_Group` showed the IGMP join reports of the remaining 30 percent never reached ovn-controller (an IGMP-snooping flow rule missed because TTL=1 was decremented incorrectly in an intermediate stage). Fix: an in-house backport of the upstream bug-fix commit from branch-22.06 to branch-22.03.

**Synthetic exercise.** Lab: create an LS with 4 LSPs, send a broadcast ARP from one LSP, check that `_MC_flood` has 4 ports; delete one LSP, after 1 second check that the group has 3 ports (the weak ref cleans up automatically); add IGMP snooping configuration on the LS, from one VM send an IGMP join for 224.1.1.1, check that Multicast_Group `_MC_224.1.1.1` appears with 1 port; have a second VM join, the group then has 2 ports.

**Common failures and diagnostic signals.** (a) The group lacks the Port_Binding member after the LSP is up → northd has not finished compiling, wait 100 ms; if it persists, northd is hung. (b) The group has a duplicate tunnel_key collision with Port_Binding (wrong range) → schema corruption, restart SBDB. (c) IGMP_Group is empty even though the VM joined → an IGMP-snooping flow rule is wrong, check the pipeline.

**Comparison with other systems.** Equivalent to `bridge mdb show` on a Linux bridge (multicast database). Equivalent to a Cisco IGMP snooping table (`show ip igmp snooping groups`). Difference: OVN Multicast_Group is cluster-wide, every chassis sees the same tunnel_key and replicates in a distributed fashion; the Linux bridge mdb is per-host. Equivalent to a VMware NSX-T multicast group object in the management plane (similar concept but centralised implementation).

## 13.5.1d The remaining five SBDB runtime tables: Service_Monitor, IGMP_Group, BFD, HA_Chassis, Connection

> This section is the second cluster of the "SBDB runtime companions to Port_Binding". The first five tables (Section 13.5.1c) cover identity, L2/L3 forwarding, and static multicast. The five tables below cover **health checking, dynamic membership, fast failover, gateway HA, and client connections**. Each table is taught with the same 20-axis treatment as 13.5.1c. Schema verified upstream `ovn-sb.ovsschema` branch-22.03. Cross-link source code: `controller/pinctrl.c` (Service_Monitor), `controller/ip-mcast.c` (IGMP_Group), `controller/bfd.c` (BFD), `controller/ha-chassis.c` (HA_Chassis), and the shared OVSDB IDL library (Connection).

### 13.5.1d.1 The `Service_Monitor` table, per-backend health probes for Load_Balancer

**Concept.** `Service_Monitor` is the SBDB table that stores **one row per (logical_port, ip, port, protocol)** representing a **health probe** that ovn-controller emits regularly to know whether a Load_Balancer backend is alive or dead. Type: **OVSDB table**, **root** (`isRoot=true`), populated by `ovn-northd` when compiling an NBDB Load_Balancer that has a health_check configured; the `status` column is consumed and updated at runtime by `controller/pinctrl.c` on the chassis where the backend is hosted. Each row is **a small probe circuit** that runs independently, and the result is written back to SBDB so that northd can remove dead backends from the pool.

**History and background.** The table was introduced in OVN 2.12 (2019) when Numan Siddique (Red Hat) added Load_Balancer health checking. Before that, OVN LB was static: backends in NBDB Load_Balancer.vips were dispatched round-robin regardless of whether the VM was alive, and if the VM died, traffic went into a black hole. OVN 2.12 brought the HAProxy/Envoy "active health check" model into OVN: every backend has a periodic TCP or UDP probe, the result lives in Service_Monitor.status, and northd compiles flows only for backends with `status=online`. OVN 21.06 improved the UDP probe (send an empty packet, listen for an ICMP unreachable). OVN 22.03 stabilised the schema at 9 columns.

**Position in the architecture.** Service_Monitor sits in the SBDB layer, root table. Producer (compile): `northd/ovn-northd.c` build_lb_svcs() reads NBDB Load_Balancer plus Load_Balancer_Health_Check and emits one Service_Monitor row per (vip-backend, protocol). Producer (runtime status): `controller/pinctrl.c` function `svc_monitors_run()`, where only the chassis hosting the backend (logical_port resides on that chassis) runs the probe and updates `status`. Consumer: northd reads `status` back to remove offline backends from the LB flow rules.

**Role.** Three roles: (1) **active health probe**, sending periodic TCP SYN or UDP packets to the backend and measuring the response; (2) **a data gateway between data-plane reality and control-plane decision**, the probe result is the only signal that lets northd know a backend is dead; (3) **distributed health checking**, only the backend's home chassis runs the probe, avoiding cluster-wide N times M overhead.

**Why it exists.** Before 2.12, the OpenStack Octavia plus OVN architecture was not integrated: the Octavia agent ran health checks outside OVN and updated Neutron LBaaS, with a 10 to 30 second lag. Users wanted an OVN-native LB with health checking inside the OVN control plane to drive backend-failure detection latency down to 1 to 3 seconds.

**Problems it solves.** (a) **traffic blackhole** when a backend dies and the LB has not noticed yet, ECMP packets fall into the hole; (b) **slow convergence** from data plane (backend died) to control plane (removed from the pool); (c) **decentralised probing**, only the home chassis probes each backend, avoiding the N-chassis times M-backend fanout.

**Importance.** Second-tier medium. A cluster that does not use OVN LB or uses LB without health_check has an empty table and no impact. A cluster with OVN LB and health_check (production Kubernetes services through OVN-Kubernetes, Octavia provider OVN) makes the table load-bearing, and a wrong status schema means every LB flow goes to the wrong destination.

**How it works.** Lifecycle of a row: (1) NBDB creates a Load_Balancer with a health_check; (2) northd build_lb_svcs() INSERTs a Service_Monitor row with `logical_port=<backend-LSP>`, `ip=<backend-IP>`, `port=<backend-port>`, `protocol=tcp|udp`, initial `status=offline`, and `options:{interval, success_count, failure_count, timeout}`; (3) every chassis monitors SBDB; (4) the chassis where `logical_port` is claimed runs `svc_monitors_run()` periodically (default every 5 seconds); (5) it sends a TCP SYN from `src_ip:src_port` (allocated ephemerally) to `ip:port`; (6) it waits for the SYN-ACK; (7) it updates `status` to `online` after `success_count` consecutive successes, to `offline` after `failure_count` consecutive failures, or to `error` if the probe could not be sent; (8) northd receives the SBDB event and recompiles the LB flows.

**Column taxonomy (verified branch-22.03).** Nine columns: `ip` (string), `protocol` (enum tcp|udp), `port` (integer 0-65535), `logical_port` (string, the backend LSP name), `src_mac` (string), `src_ip` (string), `status` (enum online|offline|error), `options` (map), `external_ids` (map). `isRoot=true`, one index `["logical_port","ip","port","protocol"]` ensures the probe tuple is unique.

**How an operator masters this.** Three skills: (1) `sudo ovn-sbctl list Service_Monitor` shows every running probe and its status; (2) `sudo ovn-sbctl find Service_Monitor status=offline` filters dead backends, the "symptom list" when the LB is broken; (3) `sudo ovn-sbctl --columns=ip,port,status,options list Service_Monitor` shows a compact table view. Decision: if the LB drops traffic, `find Service_Monitor status=offline` returns the IPs of dead backends; check whether the LSP's chassis is still up and whether the backend service (for example httpd) is listening.

**Usage workflow.** Workflow to set up health checking: (1) create a Load_Balancer_Health_Check in NBDB with `ovn-nbctl create Load_Balancer_Health_Check vip="10.0.0.5\:80" options:interval=5 options:timeout=2 options:success_count=2 options:failure_count=3`; (2) attach it to the Load_Balancer with `ovn-nbctl set Load_Balancer <lb-uuid> health_check="@hc-uuid"`; (3) wait 5 to 10 seconds; (4) `ovn-sbctl list Service_Monitor` confirms the row was created; (5) verify that `status` moves from `offline` to `online` after 2 successful probes; (6) stop a backend, verify that status moves from `online` to `offline` after 3 failed probes.

**When something goes wrong.** Symptoms: (a) "the LB does not remove a dead backend" → check that backend's `Service_Monitor.status`; if it still reads `online`, the probe is wrong; (b) "the backend is healthy but status=offline" → check whether src_ip/src_mac has a route to the backend and whether flow rules let the probe pass; (c) "no Service_Monitor row exists even though LB has a health_check" → northd has not finished compiling, restart northd.

**Tightly related to.** Tight coupling: `Load_Balancer` (NBDB source), `Load_Balancer_Health_Check` (NBDB source), `Port_Binding` (logical_port must match a bound LSP), `controller/pinctrl.c` (status producer), `Chassis` (only the home chassis runs the probe). Loose: `Logical_Flow` (indirect consumer, northd recompiles LB flows when status changes).

**Version differences.** OVN 2.12: table introduced, TCP support. OVN 20.06: UDP probe added (send a packet and listen for ICMP unreachable to detect a dead backend). OVN 21.06: optimised interval scheduling, lower pinctrl CPU. OVN 22.03: schema stable at 9 columns (verified). OVN 24.03: added the `options:dst_mac` parameter for probes in unusual L2 environments.

**Observation and verification.** Commands: `sudo ovn-sbctl list Service_Monitor`, `sudo ovn-sbctl --columns=ip,port,protocol,status list Service_Monitor`, `sudo ovn-sbctl find Service_Monitor status=offline`. Real-time status stream: `sudo ovsdb-client monitor unix:/var/run/ovn/ovnsb_db.sock OVN_Southbound Service_Monitor`. Cross-check with the actual probe: `sudo tcpdump -i any -nn 'tcp[tcpflags] & tcp-syn != 0 and host <backend-ip> and port <backend-port>'` on the backend LSP's home chassis shows a SYN going out every `interval` seconds.

**Source code reference (verified branch-22.03).** `ovn-sb.ovsschema` table `Service_Monitor` 9 columns plus index. `controller/pinctrl.c` functions: `init_svc_monitors()` (init), `sync_svc_monitors()` (sync with SBDB), `svc_monitors_run()` (send probes and update status), `svc_monitors_wait()` (poll loop), `pinctrl_handle_svc_check()` (handle replies), `destroy_svc_monitors()` (cleanup). `northd/ovn-northd.c` function `build_lb_svcs()` compiles NBDB Load_Balancer_Health_Check into Service_Monitor rows. The default timeout constant lives in `controller/pinctrl.c::SVC_MONITOR_DFLT_INTERVAL`.

**Real-world incident.** Case 2024-Q2, OVN-Kubernetes cluster: a backend pod was OOMKilled, kubelet deleted the pod, OVN deleted the LSP, but the corresponding Service_Monitor row stayed for 30 seconds because northd had not recompiled. Symptom: K8s service traffic still routed 1 in 3 connections to the dead IP, with TCP retransmits. Fix: an upstream bug fix that triggers a northd recompile immediately when the LSP is deleted, instead of waiting for the periodic recompile.

**Synthetic exercise.** Lab: create a Load_Balancer with 2 backends (Section 13.3 lab); create a Load_Balancer_Health_Check and attach it; `sudo ovn-sbctl list Service_Monitor` shows 2 rows with status=online; stop one backend with `sudo systemctl stop httpd` inside the VM; about 15 seconds later (3 probes times 5 seconds) the status moves to offline; verify the flow with `ovs-ofctl dump-flows br-int | grep <vip>` to see the offline backend removed; restart httpd; about 10 seconds later the status moves to online and the flow re-includes the backend.

**Common failures and diagnostic signals.** (a) Status stays at `error` and never moves to `online` or `offline` → src_ip has no route to the backend; signal: ovn-controller log `"svc_monitor: failed to send probe"`. (b) Status flaps online/offline every few tens of seconds → the interval is too short and the network jitters, raise `options:timeout`. (c) The table grows quickly without shrinking → northd does not clean up after LB deletion, restart northd.

**Comparison with other systems.** Equivalent to the HAProxy `option httpchk` health check (same active-probe concept) and to the Nginx upstream `health_check` (same pattern). Equivalent to Cisco GSLB health probes. Difference: Service_Monitor is distributed, only the backend's home chassis runs the probe, with no central node; HAProxy and Nginx have one process probing everything; Cisco GSLB uses a separate agent. The distributed model significantly reduces load in a cluster with thousands of backends.

### 13.5.1d.2 The `IGMP_Group` table, dynamic IGMP-snooping membership

**Concept.** `IGMP_Group` is the SBDB table that stores **one row per (multicast-address, datapath, chassis)** recording **the Port_Bindings that sent an IGMP join** report for that multicast address on the matching chassis. Type: **OVSDB table**, **root** (`isRoot=true`), populated at runtime by `controller/ip-mcast.c` through the `igmp_group_create` and `igmp_group_update_ports` actions when pinctrl handles an IGMP report. The difference from Multicast_Group (Section 13.5.1c.5, the static output of northd compile) is that IGMP_Group is the **dynamic input** from the data plane; northd reads IGMP_Group plus LSP membership to compile Multicast_Group.

**History and background.** The table was introduced in OVN 21.06 (2021) when Han Zhou (eBay/Nvidia) contributed the IGMP-snooping feature. Before that, OVN did not snoop IGMP, so every multicast IP packet flooded like L2 broadcast, a large bandwidth waste for IPTV and video conferencing. OVN 21.06 added the IGMP snoop pipeline stage on the LS, populated IGMP_Group through pinctrl, and northd consumed it to limit `_MC_<addr>` flooding. OVN 22.03 added IGMP_Group support for Logical_Router multicast routing. OVN 22.06 added MLD (IPv6 multicast) support.

**Position in the architecture.** IGMP_Group sits in the SBDB layer, root table. Producer: `controller/ip-mcast.c` through `igmp_group_create()` and `igmp_group_update_ports()`, called from the pinctrl handler when an IGMP report arrives. Consumer: `northd/northd.c` function `build_mcast_groups()` reads IGMP_Group and combines it with static LSP membership to build Multicast_Group entries. Pipeline LS stages `LS_IN_IGMP_QUERY` and `LS_IN_LKUP_MCAST_FLOW` push IGMP reports to the controller, while `LS_OUT_PRE_LB` or downstream stages replicate based on Multicast_Group.

**Role.** Three roles: (1) **track dynamic membership** based on IGMP/MLD reports; (2) **input for northd to build Multicast_Group**; (3) **per-chassis scope**, the `chassis` column lets LR multicast routing know which chassis has members (only tunnel to chassis with members, not flood across the cluster).

**Why it exists.** Before 21.06, OVN had no native IGMP snooping; users had to enable IGMP snoop on the upstream physical switch or accept flooding. The problem: an OpenStack cluster running IPTV needs OVN-aware multicast; flooding an LS with 100 VMs for a 5 Mbps stream means 500 Mbps of overhead per packet. IGMP_Group restricts replication to the VMs that actually joined.

**Problems it solves.** (a) **bandwidth waste** from multicast flooding in a large cluster; (b) **dynamic membership**, IGMP join/leave changes at runtime and the schema must reflect it immediately; (c) **cross-LS multicast routing**, the LR needs to know which chassis has members to forward selectively.

**Importance.** Second-tier medium. The table is load-bearing only in clusters with multicast apps (IPTV, ROS, video conferencing, financial market data feeds). Unicast-only clusters (more than 90 percent of production OpenStack) leave the table empty with no impact.

**How it works.** Lifecycle: (1) the VM sends an IGMP membership report `{type=join, group=224.1.1.1}` out of an LSP; (2) the LS IGMP-snoop flow stage matches the IGMP packet and runs `output:CONTROLLER`; (3) the `controller/pinctrl.c` handler receives the packet-in and calls `controller/ip-mcast.c::igmp_group_create()` or `igmp_group_update_ports()`; (4) the function INSERTs or UPDATEs an IGMP_Group row with `address=224.1.1.1`, `datapath=<dp-of-ls>`, `chassis=<self>`, `ports=[<lsp-uuid>]`; (5) northd monitors SBDB and recompiles Multicast_Group `_MC_224.1.1.1` as the union of all IGMP_Group rows for that address; (6) the next multicast packet replicates only to the Port_Bindings in the group.

**Column taxonomy (verified branch-22.03).** Four columns: `address` (string, textual multicast IP such as "224.1.1.1" or an IPv6 multicast address), `datapath` (uuid weak ref to Datapath_Binding), `chassis` (uuid weak ref to Chassis), `ports` (set of Port_Binding UUIDs, weak refs). `isRoot=true`, one index `["address","datapath","chassis"]` ensures the per-chassis tuple is unique.

**How an operator masters this.** Three skills: (1) `sudo ovn-sbctl list IGMP_Group` shows a snapshot of all membership; (2) `sudo ovn-sbctl find IGMP_Group address=224.1.1.1` filters by group; (3) `sudo ovn-sbctl --columns=address,chassis,ports list IGMP_Group` shows a cross-chassis table view. Decision: if a multicast stream does not reach a VM, check whether IGMP_Group has a row for that VM (that is, whether ovn-controller received the VM's IGMP join).

**Usage workflow.** Workflow to debug "VM-A receives multicast, VM-B does not": (1) verify the IGMP join on VM-B with `sudo tcpdump -i eth0 -nn igmp` and confirm the report goes out; (2) run `ovn-sbctl find IGMP_Group address=<group> ports{}=<vm-b-lsp-uuid>` to see whether a row exists; (3) if not, check whether the IGMP-snoop flow in stage LS_IN_LKUP_MCAST_FLOW matches correctly (a TTL=1 mismatch is a common cause); (4) if IGMP_Group has the row but Multicast_Group is missing VM-B, restart northd; (5) verify again with a multicast test packet.

**When something goes wrong.** Symptoms: (a) "the VM does not receive multicast even after an IGMP join" → IGMP_Group is missing the row; (b) "membership leak, the VM left but still receives" → IGMP leave is not handled, possibly an ovn-controller bug; (c) "IGMP_Group is empty even though many VMs joined" → the IGMP-snoop flow is overshadowed by another higher-priority rule.

**Tightly related to.** Very tight coupling with `Multicast_Group` (output dependency of northd compile), `Port_Binding` (weak ref through `ports`), `Datapath_Binding` (weak ref through `datapath`), `Chassis` (weak ref through `chassis`), `controller/ip-mcast.c` (the main producer), and `controller/pinctrl.c` (the packet-in input handler).

**Version differences.** OVN 21.06: table introduced for IGMP v2/v3 (IPv4 multicast). OVN 22.03: schema stable at 4 columns (verified). OVN 22.06: MLD (IPv6 multicast) support, same schema but `address` accepts IPv6 textual addresses (for example, "ff02::1"). OVN 24.03: optimised pinctrl handler to reduce populate latency.

**Observation and verification.** Commands: `sudo ovn-sbctl list IGMP_Group | head -20`, `sudo ovn-sbctl find IGMP_Group address=224.1.1.1`, `sudo ovn-sbctl --columns=address,chassis,ports list IGMP_Group`. Real-time: `sudo ovsdb-client monitor unix:/var/run/ovn/ovnsb_db.sock OVN_Southbound IGMP_Group` streams events as VMs join or leave. Cross-check with the actual IGMP packet: `sudo tcpdump -i any -nn 'igmp'` on the VM's home chassis.

**Source code reference (verified branch-22.03).** `ovn-sb.ovsschema` table `IGMP_Group` 4 columns plus index `["address","datapath","chassis"]`. `controller/ip-mcast.c` functions: `igmp_group_index_create()`, `igmp_group_lookup()`, `igmp_group_create()`, `igmp_group_update_ports()`, `igmp_group_delete()`, `igmp_group_cleanup()`. The `controller/pinctrl.c` packet-in handler for IGMP calls into the ip-mcast functions. `northd/northd.c` build_mcast_groups() consumes IGMP_Group when compiling Multicast_Group.

**Real-world incident.** Case 2024-Q2, OSP IPTV cluster (also mentioned in Section 13.5.1c.5): 30 percent of subscribers did not receive the 239.1.1.1 stream because an IGMP-snoop flow in an intermediate stage decremented TTL=1 incorrectly, and the IGMP report packet was dropped before reaching `output:CONTROLLER`. Symptom: `ovn-sbctl list IGMP_Group` showed only 70 percent of chassis with rows; tcpdump on the missing VMs still showed the IGMP join leaving but it never arrived at ovn-controller. Fix: backport the upstream bug fix that adjusts stage TTL handling.

**Synthetic exercise.** Lab: create an LS with 3 LSPs and enable IGMP snooping (`ovn-nbctl set Logical_Switch <ls> other_config:mcast_snoop=true`); from VM-A run `iperf -c 224.1.1.1 -u -T 32` to join; 2 to 3 seconds later, `sudo ovn-sbctl list IGMP_Group` shows 1 row with `ports=[lsp-A]`; have VM-B join the same group; the row updates to ports=[lsp-A, lsp-B] (same chassis) or a second row appears (different chassis); VM-A leaves; after the timeout (default 260 seconds, tunable) ports shrinks to lsp-B.

**Common failures and diagnostic signals.** (a) IGMP_Group does not appear after the VM joined → the snoop flow is wrong; signal: VM tcpdump shows the join but `ovn-sbctl` has no row. (b) Membership leak (no leave) → the ovn-controller pinctrl thread is overloaded, log `"failed to handle igmp leave"`. (c) Wrong cross-chassis aggregation (the same address has multiple rows for the same chassis) → schema corruption, restart SBDB.

**Comparison with other systems.** Equivalent to a Linux bridge `bridge mdb show` with IGMP snooping enabled. Equivalent to a Cisco IGMP snooping table (`show ip igmp snooping groups dynamic`). Difference: OVN IGMP_Group is cluster-wide through SBDB Raft, every chassis sees the same membership; the Linux bridge mdb is per-host. Equivalent to the VMware NSX-T "IGMP profile" with a membership tracker (similar concept).

### 13.5.1d.3 The `BFD` table, ultra-fast tunnel-failure detection sessions

**Concept.** `BFD` is the SBDB table that stores **one row per BFD session** (Bidirectional Forwarding Detection, RFC 5880) running over a tunnel or a logical_port between two chassis. Type: **OVSDB table**, **root** (`isRoot=true`), populated by `ovn-northd` when compiling an NBDB Logical_Router_Port with BFD configuration or when HA_Chassis_Group needs fast failover; the `status` column is updated at runtime by `controller/bfd.c` through the kernel BFD daemon or the OVS BFD backend. Each row represents **one keepalive circuit** with a very short period (about 50 to 300 ms) that detects a tunnel break in about 150 to 900 ms, much faster than BGP/OSPF hello intervals (seconds to tens of seconds).

**History and background.** The table was introduced in OVN 2.10 (2018) when Mark Michelson and Numan Siddique (Red Hat) added BFD for HA_Chassis_Group. Before that, OVN relied on Geneve keepalives or OVS tunnel-monitor heuristics to detect a dead chassis, which was slow (10 to 30 seconds) and caused split brain on active/standby gateways. RFC 5880 BFD standardises probe intervals down to 50 ms, enabling distributed gateway failover under 1 second. OVN 21.06 extended BFD to BFD-track BGP next-hops on Logical_Router_Port. OVN 22.03 stabilised the schema at 10 columns.

**Position in the architecture.** BFD sits in the SBDB layer, root table. Compile producer: `northd/ovn-northd.c` build_bfd_table() reads NBDB BFD records (referenced by Logical_Router_Port or HA_Chassis_Group) and emits SBDB BFD rows. Runtime status producer: `controller/bfd.c` function `bfd_run()`, where the chassis that claims `logical_port` runs the BFD session with the peer identified by `dst_ip` and updates `status`. Consumer: HA_Chassis_Group selection logic in `controller/ha-chassis.c` reads BFD status to know which group member is alive.

**Role.** Three roles: (1) **fast failure detection**, sub-second keepalive on a tunnel or logical port; (2) **trigger gateway failover**, HA_Chassis_Group uses BFD status to pick the active chassis; (3) **fast feedback for BGP** (OVN 21.06+), BFD-track next-hop, BGP withdraws the route as soon as BFD goes down rather than waiting for the hold timer.

**Why it exists.** Before 2.10, HA_Chassis_Group failover relied on the SBDB connection timeout from ovn-controller (default 10 to 30 seconds). The problem: SBDB timeout is not a data-plane signal, a chassis can lose its SBDB connection while the tunnel stays up (false-positive failover) or vice versa (false negative). BFD measures data-plane health precisely through tunnel keepalives.

**Problems it solves.** (a) **slow gateway failover** (10 to 30 seconds) when the active gateway dies in a large cluster; (b) **false-positive split brain** when an SBDB timeout does not reflect data-plane reality; (c) **asymmetric tunnel down**, one direction is up while the other is down, only bidirectional BFD detects this.

**Importance.** Second-tier medium. A cluster that does not use HA_Chassis_Group or does not require sub-second failover leaves the table empty with no impact. A cluster with distributed gateway ports (north-south NAT, hardware VTEP, HA L3) makes the table load-bearing: a wrong BFD status means a wrong failover, which means a traffic blackhole.

**How it works.** Lifecycle: (1) NBDB Logical_Router_Port has a `bfd` column that references an NBDB BFD record with the next-hop's `dst_ip`; (2) northd build_bfd_table() compiles it into an SBDB BFD row with `src_port=<random 49152-65535>`, `disc=<random discriminator>`, `logical_port=<lrp>`, `dst_ip=<next-hop>`, `min_tx`, `min_rx`, `detect_mult`, and initial `status=down`; (3) the chassis that owns `logical_port` runs `bfd_run()` and calls into the OVS BFD backend (`ovs-vsctl set Interface ovn-<peer> bfd:enable=true bfd:min_rx=...`); (4) OVS BFD exchanges packets with the peer every `min_tx` ms; (5) after `detect_mult` consecutive lost packets, OVS marks the session down; (6) ovn-controller reads the OVS BFD status and updates SBDB `status=down`; (7) HA_Chassis_Group logic detects `status=down` and selects the next chassis in the group; (8) traffic moves to the new chassis in under 1 second.

**Column taxonomy (verified branch-22.03).** Ten columns: `src_port` (integer 49152-65535, ephemeral source port of the BFD packet), `disc` (integer, local BFD discriminator), `logical_port` (string, the LSP attached to BFD), `dst_ip` (string, peer IP), `min_tx` (integer ms, desired tx interval), `min_rx` (integer ms, required rx interval), `detect_mult` (integer multiplier, the number of lost packets before declaring down), `status` (enum down|init|up|admin_down), `external_ids` (map), `options` (map). `isRoot=true`, one index `["logical_port","dst_ip","src_port","disc"]` ensures the session tuple is unique.

**How an operator masters this.** Four skills: (1) `sudo ovn-sbctl list BFD` shows a snapshot of every session; (2) `sudo ovn-sbctl find BFD status=down` filters dying sessions, an early signal that gateway failover is about to occur; (3) `sudo ovn-sbctl --columns=logical_port,dst_ip,status,detect_mult list BFD` shows a table view; (4) cross-check with OVS BFD using `sudo ovs-vsctl --columns=name,bfd_status list Interface | grep -A1 ovn-` to align SBDB and the actual datapath. Decision: if the HA gateway flaps, check BFD status; if it flaps up/down on a cycle that matches min_rx times detect_mult, it could be network jitter, raise detect_mult.

**Usage workflow.** Workflow to set up BFD for an LRP: (1) create the NBDB BFD record with `ovn-nbctl create BFD logical_port=lrp-public dst_ip=10.0.0.1 min_tx=300 min_rx=300 detect_mult=3`; (2) attach to the LRP with `ovn-nbctl set Logical_Router_Port lrp-public bfd="@bfd-uuid"`; (3) wait 5 to 10 seconds; (4) `ovn-sbctl list BFD` confirms the row was created with `status=init` and then moves to `up`; (5) verify OVS BFD with `ovs-vsctl --columns=bfd,bfd_status list Interface ovn-<peer>` and confirm bfd_status forwarding=true; (6) test failover: `ip link set <peer-tunnel> down` on the peer, then check that SBDB BFD status moves to down within `detect_mult` times `min_rx` ms.

**When something goes wrong.** Symptoms: (a) "HA gateway failover is too slow (more than 5 seconds)" → check BFD min_rx times detect_mult, if 30 × 100 = 3000 ms then the timing matches; (b) "BFD status=admin_down without being disabled" → ovn-controller bug, restart; (c) "BFD flaps every few seconds" → network jitter or a wrong tunnel MTU that fragments the BFD packets.

**Tightly related to.** Tight coupling: `Logical_Router_Port` (NBDB source that references the BFD record), `HA_Chassis_Group` plus `HA_Chassis` (status consumers that pick the active member), `controller/bfd.c` (status producer), `controller/ha-chassis.c` (consumer logic), and the OVS Interface table (data-plane BFD backend). RFC 5880 is the standard protocol; cross-link Section 11.0 §Geneve (BFD runs inside the Geneve tunnel between two chassis).

**Version differences.** OVN 2.10: table introduced for HA_Chassis_Group fast failover. OVN 21.06: BFD-track BGP next-hop support (through ovn-bgp-agent integration). OVN 22.03: schema stable at 10 columns (verified). OVN 24.03: optimised BFD scheduling for clusters with more than 1000 sessions.

**Observation and verification.** Commands: `sudo ovn-sbctl list BFD`, `sudo ovn-sbctl --columns=logical_port,dst_ip,status,min_tx,min_rx,detect_mult list BFD`, `sudo ovn-sbctl find BFD status=down`. OVS data-plane BFD: `sudo ovs-vsctl --columns=name,bfd,bfd_status list Interface | head -30`. Real-time: `sudo ovsdb-client monitor unix:/var/run/ovn/ovnsb_db.sock OVN_Southbound BFD`. Cross-check the RFC 5880 packet: `sudo tcpdump -i ovn-<peer> -nn 'udp port 3784'` (BFD control port).

**Source code reference (verified branch-22.03).** `ovn-sb.ovsschema` table `BFD` 10 columns plus index. `controller/bfd.c` functions: `bfd_register_ovs_idl()` (IDL registration), `bfd_calculate_active_tunnels()` (decide which sessions are active per chassis), `bfd_run()` (orchestrates the sync with the OVS backend). The function `bfd_status_run` updates SBDB from OVS bfd_status. `northd/ovn-northd.c` build_bfd_table() compiles NBDB into SBDB.

**Real-world incident.** Case 2023-Q4 production OSP: a 3-node HA_Chassis_Group cluster suddenly failed over every 5 minutes. Investigation: `ovn-sbctl list BFD` showed status flapping up→down→up on a roughly 30-second cycle. Root cause: an MTU mismatch, the Geneve tunnel MTU was 1450 but the standard BFD packet was 1500 and got fragmented in the physical fabric, with about 1 percent fragment loss on a bad link, which tripped BFD detect_mult=3 (3 × 100 ms = 300 ms timeout) frequently. Fix: set `Open_vSwitch.external_ids:ovn-encap-mtu=1450` so OVS knows the limit, the BFD packet pads shorter to fit the MTU, and the flap stops.

**Synthetic exercise.** Lab: build a 2-chassis lab plus 1 LRP with a BFD record (min_tx=200, min_rx=200, detect_mult=3); verify `ovn-sbctl list BFD` status=up; on the peer chassis run `sudo iptables -A INPUT -p udp --dport 3784 -j DROP` to block BFD packets; about 600 ms later (3 × 200 ms) status moves to down; remove the iptables rule; about 600 ms later status returns to up. Variant: change min_rx=50, detect_mult=3, measure actual failover with a continuous cross-chassis ping while BFD is blocked, and measure downtime.

**Common failures and diagnostic signals.** (a) Short-cycle BFD flap → MTU mismatch or chassis CPU saturation (the BFD packet is delayed). (b) Status stays at `init` and never moves to `up` → the peer does not respond or the peer has not enabled BFD; signal: tcpdump shows packets going out but no reply. (c) Status `admin_down` but the configuration is not disabled → ovn-controller has not synced, restart. (d) One direction up, the other init → asymmetric BFD; check that both sides have the same configuration.

**Comparison with other systems.** Equivalent to BFD on a Cisco router (same RFC 5880, same session concept). Equivalent to Juniper Junos BFD (same standard). Equivalent to the Linux kernel BFD daemon `bfdd` (which runs standalone). Difference: SBDB BFD is orchestrated through the OVN control plane and parameters are compiled from declarative NBDB rather than configured with per-device CLI; the OVS BFD backend on the data plane runs in ovs-vswitchd userspace rather than in the kernel.

### 13.5.1d.4 The `HA_Chassis` table, prioritised chassis in an HA gateway group

**Concept.** `HA_Chassis` is the SBDB table that stores **one row per (chassis, priority)** inside an HA_Chassis_Group, representing a member with a **priority level** for being selected as the active gateway for a distributed gateway port (chassisredirect / l3gateway). Type: **OVSDB table**, **not root** (`isRoot=false`), exists only when referenced by an HA_Chassis_Group through the `ha_chassis` column. The chassis with the highest priority that is alive (per BFD) becomes active; the rest stand by. When the active dies, the next-highest priority chassis takes over.

**History and background.** The table was introduced in OVN 2.10 (2018) together with BFD and HA_Chassis_Group. Before that, OVN used a per-LRP `gateway_chassis` with priority inline in the column, a schema that was hard to maintain when a group needed to be shared across many LRPs. The HA_Chassis_Group + HA_Chassis pattern splits the schema into "group definition" plus "members", letting one group serve many LRPs with a single update affecting everything. Author: Numan Siddique (Red Hat). OVN 22.03 stabilised the schema at 3 columns.

**Position in the architecture.** HA_Chassis sits in the SBDB layer, non-root table. Compile producer: `northd/ovn-northd.c` build_ha_chassis_groups() reads the corresponding NBDB HA_Chassis_Group and emits SBDB HA_Chassis rows through a strong ref from HA_Chassis_Group.ha_chassis. Consumer: `controller/ha-chassis.c` function `ha_chassis_group_is_active()` sorts HA_Chassis by priority and filters by BFD status to choose the active member.

**Role.** Three roles: (1) **register candidates** for the HA gateway group; (2) **carry priority** that decides the takeover order; (3) **bridge between HA_Chassis_Group and Chassis**, HA_Chassis references Chassis (through the chassis weak-ref column), letting the Chassis row live independently of group membership.

**Why it exists.** The pre-2.10 per-LRP `gateway_chassis` pattern did not scale: a cluster with 100 LRPs that needed the same HA group would have to configure 100 gateway_chassis columns with the same list, and a single mistake caused split brain. HA_Chassis_Group + HA_Chassis separates concerns: define the group once, attach it to many LRPs through references.

**Problems it solves.** (a) **configuration duplication** when many LRPs share an HA topology; (b) **atomic updates**, changing the priority on one HA_Chassis row affects every LRP attached to the group; (c) **deterministic selection**, sorting priority plus BFD status produces a single active member, avoiding split brain.

**Importance.** Second-tier medium. A cluster without distributed gateway ports (small cluster, no NAT) leaves the table empty. A production OpenStack cluster with DVR plus gateway HA makes the table load-bearing: a wrong priority or a missing row causes north-south traffic to drop on failover.

**How it works.** Lifecycle: (1) the admin creates an NBDB HA_Chassis_Group `name=group-A` with members; (2) northd build_ha_chassis_groups() compiles it into an SBDB HA_Chassis_Group plus one HA_Chassis row per member with a strong ref; (3) attach to an LRP with `Logical_Router_Port.ha_chassis_group=@group-A`; (4) on every chassis, `controller/ha-chassis.c::ha_chassis_get_ordered()` sorts HA_Chassis by descending priority; (5) filter by BFD status, only chassis with BFD up to other members are candidates; (6) `ha_chassis_group_is_active()` decides whether the local chassis is the top healthy candidate, in which case it claims the chassisredirect Port_Binding; (7) when BFD reports the active member dead, re-evaluation runs automatically and the next candidate claims.

**Column taxonomy (verified branch-22.03).** Three columns: `chassis` (uuid weak ref to Chassis, weak so that deleting Chassis auto-resets HA_Chassis), `priority` (integer 0-32767, higher means higher priority), `external_ids` (map). `isRoot=false`, no index. The strong reference comes from `HA_Chassis_Group.ha_chassis` (strong ref into HA_Chassis); deleting an HA_Chassis_Group deletes all its child HA_Chassis rows because of the strong ref.

**How an operator masters this.** Three skills: (1) `sudo ovn-sbctl list HA_Chassis` snapshots the members; (2) `sudo ovn-sbctl list HA_Chassis_Group` shows the group structure plus references to HA_Chassis; (3) `sudo ovn-sbctl --columns=chassis,priority list HA_Chassis | sort -k3 -nr` sorts by descending priority to identify the current active candidate. Decision: when the gateway flaps, compare priority and check BFD status of the top candidate; if BFD is up and priority is highest but the chassis is not active, it is an ovn-controller bug, check the logs.

**Usage workflow.** Workflow to set up an HA gateway: (1) `ovn-nbctl ha-chassis-group-add hagroup1`; (2) `ovn-nbctl ha-chassis-group-add-chassis hagroup1 hv1 100`, `ovn-nbctl ha-chassis-group-add-chassis hagroup1 hv2 50`, `ovn-nbctl ha-chassis-group-add-chassis hagroup1 hv3 25`; (3) create the LRP, replacing `options:redirect-chassis` with `ha_chassis_group=@group-A`; (4) `ovn-sbctl list HA_Chassis` confirms 3 rows are created with the right priorities; (5) verify that the chassisredirect Port_Binding is claimed by hv1 (highest priority, BFD up); (6) test failover: shut down hv1, about 600 ms later (BFD timeout) chassisredirect moves to hv2.

**When something goes wrong.** Symptoms: (a) "the gateway does not switch when the active dies" → check whether the next candidate's HA_Chassis has BFD up; (b) "two chassis claim chassisredirect at once (split brain)" → check BFD status, possibly a network partition where each side sees the other as dead; (c) "priority is wrong after ovn-nbctl changes" → northd is compiling slowly, wait 5 seconds or restart northd.

**Tightly related to.** Very tight coupling: `HA_Chassis_Group` (parent strong ref through `ha_chassis`), `Chassis` (weak ref through `chassis`), `BFD` (status consumer to filter healthy candidates), `Port_Binding` of type chassisredirect/l3gateway (the main consumer; this table decides who claims the chassisredirect Port_Binding), `controller/ha-chassis.c` (consumer logic).

**Version differences.** OVN 2.10: table introduced together with HA_Chassis_Group and BFD. OVN 21.06: added "prefer local chassis" logic when priority ties. OVN 22.03: stable at 3 columns (verified). OVN 24.03: optimised compile when a group has more than 50 members.

**Observation and verification.** Commands: `sudo ovn-sbctl list HA_Chassis`, `sudo ovn-sbctl list HA_Chassis_Group`, `sudo ovn-sbctl --columns=chassis,priority list HA_Chassis | sort -k3 -nr`. Cross-check the chassisredirect Port_Binding: `sudo ovn-sbctl find Port_Binding type=chassisredirect` returns a `chassis` column that is the current active chassis; it must match the highest-priority HA_Chassis with BFD up. Real-time: `sudo ovsdb-client monitor unix:/var/run/ovn/ovnsb_db.sock OVN_Southbound HA_Chassis HA_Chassis_Group Port_Binding` streams failover events.

**Source code reference (verified branch-22.03).** `ovn-sb.ovsschema` table `HA_Chassis` 3 columns (no index). `controller/ha-chassis.c` functions: `compare_chassis_prio_()` (priority comparator for sort), `get_ordered_ha_chassis_list()` (sort by priority and status), `ha_chassis_destroy_ordered()` (cleanup), `is_local_chassis_only_candidate()` (check whether self is the only alive member), `ha_chassis_group_is_active()` (decision point), `ha_chassis_group_contains()` (membership check), `ha_chassis_get_ordered()` (public API).

**Real-world incident.** Case 2024-Q1, OSP cluster with DVR plus a 3-node HA gateway: after a datacenter migration, one chassis was restored with the old `system-id` but a new IP. The HA_Chassis row kept the weak ref `chassis=<old>` and did not break, but the old BFD record's `dst_ip=<old>` no longer matched. Symptom: failover to that chassis always failed, traffic stuck for 30 seconds until northd recompiled BFD. Fix: explicitly recreate the HA_Chassis_Group and BFD record after a migration rather than relying on auto-update; document this in the DC migration runbook.

**Synthetic exercise.** Lab: 3-chassis lab plus an HA_Chassis_Group with priorities 100, 50, 25; create an LRP with ha_chassis_group; verify hv1 (100) is active with `ovn-sbctl find Port_Binding type=chassisredirect`; shut down hv1; within 600 ms hv2 (50) takes over; bring hv1 back; if `options:auto-failback=true` (NBDB option), hv1 reclaims; if false, hv2 stays active. Variant: change hv2's priority to 150 (higher than hv1) with `ovn-nbctl ha-chassis-group-add-chassis hagroup1 hv2 150`, then after recompile hv2 takes over from hv1.

**Common failures and diagnostic signals.** (a) Wrong priority after a `nbctl` change → northd recompile lag; signal: `nbctl list HA_Chassis_Group` is correct but `sbctl list HA_Chassis` has not updated; wait 5 seconds. (b) The active chassis does not change when it dies → BFD status did not update; check `ovn-sbctl list BFD`. (c) Split brain (two actives) → network partition; resolve the partition first. (d) HA_Chassis row with `chassis=[]` (empty weak ref) → the Chassis row was deleted, the group needs fixing.

**Comparison with other systems.** Equivalent to VRRP priority (RFC 5798) in Linux keepalived or on a Cisco router. Equivalent to Cisco HSRP priority. Equivalent to Pacemaker priority plus constraints. Difference: OVN HA_Chassis is integrated declaratively into OVSDB, parameters are declared in NBDB, compiled into SBDB, and ovn-controller enforces them; there is no multicast advertisement protocol like VRRP or HSRP, the design relies on point-to-point BFD plus SBDB Raft consistent membership.

### 13.5.1d.5 The `Connection` table, client connection configuration to SBDB

**Concept.** `Connection` is the SBDB table that stores **one row per connection target** (TCP, SSL, or Unix socket) that the SBDB cluster accepts from clients (ovn-controller, ovn-sbctl, ovn-northd). Type: **OVSDB table**, **not root** (`isRoot=false`), exists when referenced by `SB_Global.connections`. Same shape as Connection in NBDB and Connection in the vswitch schema; this is the OVSDB common pattern for "the listener side of a database server". Each row tells ovsdb-server "listen on this target with these inactivity_probe and max_backoff parameters".

**History and background.** The table dates back to OVS 2.0 (2014) in the vswitch schema; OVN reused this pattern in the early OVN schema in 2015. Pattern: pulling the connection configuration out into a separate table allows changing the listening endpoint at runtime without restarting ovsdb-server (dynamic). OVN 21.06 added the `read_only` column for clusters with read-only nodes (faster failover for read traffic). OVN 22.03 stabilised the schema at 9 columns.

**Position in the architecture.** Connection sits in the SBDB layer, non-root table. Producer: the admin directly through `ovn-sbctl set-connection ptcp:6642:0.0.0.0`; `ovn-sbctl` creates the Connection row and updates the `SB_Global.connections` ref. Consumer: the SBDB `ovsdb-server` reads the table at boot and monitors it; it opens the corresponding TCP, SSL, or Unix socket. Runtime status: the `is_connected` and ephemeral `status` columns are updated by ovsdb-server after a successful socket open.

**Role.** Four roles: (1) **declare the listening endpoint** for SBDB (ovn-controller connects through TCP 6642 by default); (2) **carry per-connection tunables** like `inactivity_probe` (idle timeout) and `max_backoff` (reconnect delay); (3) **expose runtime status** (`is_connected`, `status`) to help admins debug; (4) **role-based access control** (the `role` and `read_only` columns) to distinguish admin connections from read-only ones.

**Why it exists.** Static configuration in `/etc/ovn/conf.db` does not let an admin change the listening port at runtime. The Connection table lets `ovsdb-server` read its configuration from the very database it is serving; the chicken-and-egg is solved by bootstrapping through the command-line `--remote=db:OVN_Southbound,SB_Global,connections`.

**Problems it solves.** (a) **runtime configuration changes**, change the listening port without restart; (b) **multi-listener**, one SBDB can listen on TCP 6642 plus SSL 6642 plus a Unix socket simultaneously; (c) **per-listener tunables**, TCP can have a different inactivity_probe than Unix; (d) **HA-aware**, in a Raft cluster each node has its own Connection rows but follows the same schema.

**Importance.** Second-tier medium. The table is load-bearing during initial setup (a wrong connection target means ovn-controller cannot reach SBDB and the cluster goes down). After setup stabilises, the table is rarely touched, only when changing the port, switching protocol (TCP to SSL), or tuning inactivity_probe.

**How it works.** Lifecycle: (1) the admin runs `ovn-sbctl set-connection ptcp:6642:0.0.0.0`; (2) the command creates a Connection row with `target="ptcp:6642:0.0.0.0"`, `inactivity_probe=60000`, and `max_backoff=3000`; (3) it updates `SB_Global.connections={@conn-uuid}`; (4) ovsdb-server monitors SB_Global.connections, observes the change; (5) opens or closes the corresponding socket; (6) updates `is_connected=true` and `status:bound_port=6642` at runtime; (7) when a client connects, `inactivity_probe` (default 60 seconds) sends an echo, and if there is no reply the connection is closed; the client's `max_backoff=3000` then waits 3 seconds before reconnecting.

**Column taxonomy (verified branch-22.03).** Nine columns: `target` (string, format `<protocol>:<param>` for example `ptcp:6642`, `pssl:6642`, `punix:/var/run/ovn/ovnsb_db.sock`), `max_backoff` (integer min 1000 ms), `inactivity_probe` (integer ms, 0 disables), `read_only` (boolean), `role` (string, ACL role), `other_config` (map), `external_ids` (map), `is_connected` (boolean ephemeral), `status` (ephemeral map with keys `bound_port`, `n_connections`, `sec_since_connect`, `sec_since_disconnect`, `state`). `isRoot=false`, one index `["target"]` ensures the listener is unique.

**How an operator masters this.** Four skills: (1) `sudo ovn-sbctl list Connection` snapshots the listeners; (2) `sudo ovn-sbctl get-connection` shows the current SBDB target; (3) `sudo ovn-sbctl --columns=target,is_connected,status list Connection` shows runtime status; (4) `sudo ovn-sbctl set-connection <new-target>` changes the listener at runtime. Decision: if ovn-controller logs `"connection refused"`, check that Connection.target points to the right IP and port, and check whether `is_connected=true`.

**Usage workflow.** Workflow to switch from plain TCP to SSL: (1) prepare the certificate with `ovn-pki sb.cluster sign sbdb`; (2) `sudo ovn-sbctl set-ssl /etc/ovn/key.pem /etc/ovn/cert.pem /etc/ovn/cacert.pem`; (3) `sudo ovn-sbctl del-connection`; (4) `sudo ovn-sbctl set-connection pssl:6642:0.0.0.0`; (5) `sudo ovn-sbctl list Connection` confirms the new target with `is_connected=true`; (6) update `Open_vSwitch.external_ids:ovn-remote=ssl:<sbdb-vip>:6642` on every chassis; (7) restart ovn-controller to pick up SSL; (8) verify connectivity with `sudo ovn-sbctl list Chassis | wc -l`, which must match the node count.

**When something goes wrong.** Symptoms: (a) "ovn-controller cannot connect to SBDB" → check Connection.target and is_connected; (b) "connections drop continuously" → inactivity_probe is too short, the cross-DC network latency exceeds the probe interval; raise inactivity_probe to 300000 (5 minutes); (c) "ovn-sbctl hangs" → the Unix socket Connection was removed from `SB_Global.connections`, restore it with `ovn-sbctl --no-leader-only set Connection ...`.

**Tightly related to.** Coupling: `SB_Global` (parent strong ref through `connections`), the ovsdb-server process (configuration consumer), `Open_vSwitch.external_ids:ovn-remote` on each chassis (the client configuration that must match Connection.target). Equivalent to Connection in NBDB and the vswitch schema, same pattern and same columns.

**Version differences.** OVS 2.0: pattern origin. OVN 21.06: added the `read_only` boolean. OVN 22.03: schema 9 columns (verified). OVN 22.06 onwards: no new columns. Future (24.x+): may add `bind_address_ipv6` for dual stack, not present yet.

**Observation and verification.** Commands: `sudo ovn-sbctl list Connection`, `sudo ovn-sbctl get-connection`, `sudo ovn-sbctl --columns=target,is_connected,status list Connection`. OS-level cross-check: `sudo ss -tlnp | grep 6642` (TCP listener), `sudo ss -xl | grep ovnsb_db.sock` (Unix listener), `sudo lsof -p $(pidof ovsdb-server) | grep LISTEN`. Real-time: `sudo ovsdb-client monitor unix:/var/run/ovn/ovnsb_db.sock OVN_Southbound Connection` streams events when the admin changes the listener.

**Source code reference (verified branch-22.03).** `ovn-sb.ovsschema` table `Connection` 9 columns plus index `["target"]`. The shared handling logic lives in the OVSDB IDL library: `ovsdb/jsonrpc-server.c` function `ovsdb_jsonrpc_server_set_remotes()` reads Connection.target and opens the socket; `lib/jsonrpc.c::jsonrpc_session_create()` handles max_backoff on the client side. The schema shares the pattern: `vswitchd/vswitch.ovsschema` Connection has the same columns (cross-link Section 9.1).

**Real-world incident.** Case 2023-Q3, a 3-node Raft SBDB cluster: the admin ran `ovn-sbctl del-connection` on the leader to clean up an old listener and accidentally also deleted the Unix socket connection. Symptom: `ovn-sbctl` from the same host hung (the Unix socket was gone), and follower nodes lost their leader RPC. Emergency fix: ssh to the leader's ovsdb-server, run `ovs-appctl -t /var/run/ovn/ovnsb_db.ctl ovsdb-server/add-remote punix:/var/run/ovn/ovnsb_db.sock` to re-create the Unix socket at runtime; `ovn-sbctl` then worked, and `ovn-sbctl set-connection ...` restored the TCP listener. Long term: admin scripts always `set-connection` before `del-connection`, never leaving SBDB without a listener.

**Synthetic exercise.** Lab: 1-node SBDB cluster; `ovn-sbctl get-connection` shows the current target; `ovn-sbctl list Connection` shows the details; change inactivity_probe with `ovn-sbctl set Connection . inactivity_probe=120000`; verify the change applies at runtime without restarting ovsdb-server; try adding an SSL listener in parallel with `ovn-sbctl set-connection pssl:6642 ptcp:6643` (multi-target); `ss -tlnp | grep ovsdb` shows both ports listening.

**Common failures and diagnostic signals.** (a) `is_connected=false` even though the target is correct → the port is occupied by another process; check `ss -tlnp | grep <port>`. (b) Inactivity_probe trips a cross-DC Raft cluster → raise inactivity_probe to match the RTT. (c) SSL connection fails even though the certificate is correct → check `ovsdb-server.log` for "SSL handshake failed", usually a CA mismatch. (d) The Unix socket is missing after a crash → recreate it with `ovs-appctl ovsdb-server/add-remote`.

**Comparison with other systems.** Equivalent to the `bind` directive in PostgreSQL `postgresql.conf` (listen_addresses plus port). Equivalent to `listener` in RabbitMQ configuration. Equivalent to MySQL's `mysqld --bind-address`. Difference: OVSDB Connection is a dynamic table inside the very database it serves, runtime changes do not require a restart; PostgreSQL and MySQL must reload configuration and sometimes restart. Equivalent to etcd `--listen-client-urls` (same listener-configuration concept), but etcd uses a command-line flag rather than a dynamic table.

## 13.5.2 Type `""` (default VIF), VM tap claim

**Default.** Port_Binding for a VM tap (virtual interface). The type is the empty string `""`. This is the most common type (more than 95 percent of Port_Binding rows in a deployed data centre).

### Claim condition

`ovn-controller` on chassis X claims a Port_Binding of type `""` when:
1. The OVS bridge `br-int` has a port (the tap interface) with `external_ids:iface-id=<logical_port>`.
2. The Port_Binding is not yet claimed by another chassis (`chassis=NULL` or `chassis=this`).

### Provisioning workflow

```
1. CMS orchestrator (Nova, libvirt, Kubelet) allocates a VM.
2. CMS calls: ovn-nbctl lsp-add <ls> <lsp_name>
   -> NBDB: 1 row Logical_Switch_Port
3. ovn-northd compile:
   -> SBDB: 1 row Port_Binding (type="", chassis=NULL)
4. CMS plugs the tap into br-int:
   -> ovs-vsctl add-port br-int tap-vmX
   -> ovs-vsctl set Interface tap-vmX external_ids:iface-id=<lsp_name>
5. ovn-controller on chassis X detects the tap and an iface-id match:
   -> SBDB: UPDATE Port_Binding SET chassis=<X>
6. ovn-controller installs the OpenFlow rule that matches the port for this LSP.
7. The VM is now functional.
```

### Multichassis binding (OVN 22.06+)

During VM live migration, OVN 22.06+ supports dual-chassis binding inside the migration window:

```
source chassis A: Port_Binding.chassis=A + additional_chassis=[]
-> migration started:
   destination chassis B prepares and claims as additional
   Port_Binding.chassis=A + additional_chassis=[B]
   Traffic is replicated on both chassis within a 3 to 18 second window (Part 19.0)
-> migration done:
   source releases
   Port_Binding.chassis=B + additional_chassis=[]
```

Before OVN 22.06, live migration caused 15 to 30 seconds of downtime. Multichassis binding combined with `activation-strategy=rarp` (OVN 24.03+) reduces this to under 500 milliseconds.

> **Key topic.** The claim logic depends **entirely** on the iface-id match. If the hypervisor orchestrator does not set `external_ids:iface-id` when plugging the tap, ovn-controller does not claim, so the VM has no network. This is a common pain point: manual libvirt provisioning that forgets iface-id, or a CNI plugin bug. Debugging starts with `ovs-vsctl get Interface tap-vmX external_ids:iface-id`. If empty, fix at the orchestrator layer, not the OVN layer.

## 13.5.3 Type `localnet`, provider network extension

An LSP of type `localnet` connects a logical switch to a **physical network** (provider network).

### Properties

- **No chassis claim.** The port is available on every chassis that has the matching bridge mapping.
- Each chassis maps `options:network_name` to an OVS bridge through `external_ids:ovn-bridge-mappings`.
- ovn-controller installs a patch port between `br-int` and the external bridge (`br-ex` or `br-provider`).

### Use case

Provider network (OpenStack flat provider), a shared internet gateway, or a VLAN trunk to an upstream physical switch.

```bash
[admin]# ovn-nbctl ls-add ls-provider-public
[admin]# ovn-nbctl lsp-add ls-provider-public provider-port
[admin]# ovn-nbctl lsp-set-type provider-port localnet
[admin]# ovn-nbctl lsp-set-addresses provider-port unknown
[admin]# ovn-nbctl lsp-set-options provider-port network_name=physnet1
```

On a chassis with `ovn-bridge-mappings=physnet1:br-ex`:
- ovn-controller creates the patch port pair `patch-provnet-physnet1` and `patch-br-ex-to-br-int`.
- Traffic destined for an LSP of type `localnet` is output through the patch, then `br-ex`, then the physical NIC.

### VLAN tagging

When the provider network needs a VLAN tag:

```bash
[admin]# ovn-nbctl set Logical_Switch_Port provider-port tag=100
```

The compiled flow pushes VLAN 100 before output to the patch. The upstream switch sees a VLAN 100 frame and forwards it correctly.

## 13.5.4 Type `chassisredirect` and `l3gateway`, north-south NAT

A **distributed Logical Router (LR)** handles L2 and L3 forwarding in a distributed way on every chassis (each chassis carries the local OpenFlow rules). However, **north-south NAT** cannot be distributed: an external IP can belong to only one physical chassis at any given moment (ARP announcement, routing table).

### `chassisredirect` Port_Binding

This is a "shadow" of a Logical Router Port (LRP) on a **gateway chassis**. North-bound traffic is redirected through the chassisredirect port to the gateway chassis, which performs NAT and forwards to `br-ex`.

```bash
[admin]# ovn-nbctl lrp-set-gateway-chassis lrp-external chassis-1 100
[admin]# ovn-nbctl lrp-set-gateway-chassis lrp-external chassis-2 50
```

`ovn-northd` produces a Port_Binding of type `chassisredirect` whose `ha_chassis_group` field references an HA_Chassis_Group row containing the list of chassis sorted by priority.

### Failover via BFD

HA_Chassis_Group uses BFD to detect failure. When chassis-1 (priority 100) fails:
- The BFD session times out (default 5 seconds, tunable down to 1 second).
- HA_Chassis_Group re-elects chassis-2 as active.
- The chassisredirect Port_Binding updates its chassis field.
- ovn-controller on chassis-2 installs the OpenFlow rules for NAT.

Typical downtime is 3 to 5 seconds with default BFD, and under 1 second with `bfd_min_rx=100ms`.

See Part 13.6 for the BFD deep dive.

### `l3gateway` Port_Binding

An alternative variant for gateway routing that does not use the distributed LR. Rarely used after OVN 20.06, since the distributed LR is now stable.

## 13.5.5 Type `patch`, inter-datapath forwarding

An LSP of type `router` (an LSP that connects to an LRP) automatically generates a Port_Binding of type `patch`. The patch Port_Binding is produced by `ovn-northd`; an engineer does not create it manually.

It is used for pipeline traversal from a Logical_Switch into a Logical_Router (through an LSP of type `router`).

```
LS tenant1, LSP `ls-to-router` (type=router, connected to LRP lrp-tenant1)
-> northd compile -> Port_Binding `ls-to-router` type=patch
-> OpenFlow rule: output to the Logical_Router pipeline
```

## 13.5.6 Type `l2gateway`, hardware VTEP extension

An LSP gateway from the OVN logical network to a **hardware L2 switch** (over the VTEP protocol, an RFC 7047 extension).

### Use case

Extend an OVN logical switch to a hardware VTEP (Cisco Nexus 9K, Arista 7280 with VXLAN VTEP). The ESR (Extended Service Range) pattern places OVN VMs in the same broadcast domain as a bare-metal server behind the hardware switch.

```bash
[admin]# ovn-nbctl lsp-set-type vif-hw-vtep l2gateway
[admin]# ovn-nbctl lsp-set-options vif-hw-vtep network_name=physnet1 \
         l2gateway-chassis=chassis-1
```

`l2gateway-chassis` designates the chassis that handles the VTEP protocol with the hardware switch. The `ovn-controller` on that chassis maintains the VTEP OVSDB connection to the hardware switch.

Trade-off: this adds operational complexity (the VTEP protocol plus hardware switch firmware compatibility), but it allows a hybrid physical-virtual topology.

## 13.5.7 Type `localport`, per-chassis metadata service

**Special property.** The Port_Binding exists on **every chassis** with the same MAC and IP. Traffic destined for a localport does NOT cross the tunnel; it stays local on the source chassis.

### Use case: metadata service

Both OpenStack Nova and Kubernetes need a metadata endpoint (`169.254.169.254`, the IANA link-local address). Each VM reaches metadata over HTTP GET. Rather than routing metadata traffic to one centralised gateway chassis, OVN uses a localport:

```bash
[admin]# ovn-nbctl lsp-add ls-tenant metadata-port
[admin]# ovn-nbctl lsp-set-type metadata-port localport
[admin]# ovn-nbctl lsp-set-addresses metadata-port \
         "fa:16:3e:00:00:01 169.254.169.254"
```

On each chassis:
- `ovn-controller` installs an OpenFlow rule that matches `dst=169.254.169.254` and outputs to an internal port (the local proxy agent).
- A proxy agent (for example, the Neutron metadata agent) handles the HTTP request and forwards it to the CMS metadata API.
- The reply returns to the source VM and never crosses chassis.

Benefit: the metadata service scales linearly with chassis count, with no single-gateway bottleneck.

## 13.5.8 Type `virtual`, VRRP shared address

OVN 21.03 and later support the `virtual` Port_Binding type for a floating IP shared by several VMs (the VRRP pattern).

```bash
[admin]# ovn-nbctl lsp-add ls-tenant vip-floating
[admin]# ovn-nbctl lsp-set-type vip-floating virtual
[admin]# ovn-nbctl lsp-set-options vip-floating \
         virtual-parents=vm-master,vm-backup virtual-ip=10.0.0.100
```

When one VM (vm-master) announces a Gratuitous ARP for 10.0.0.100, `ovn-controller` binds the virtual port to that VM. When vm-master fails, VRRP re-announces from vm-backup, and `ovn-controller` switches the binding. Handover is smooth for HA applications.

Use cases: HAProxy or Keepalived floating VIP, and the master IP of a database cluster.

## 13.5.9 Type `remote` (OVN 22.06 and later), inter-cluster peering

OVN cluster-to-cluster peering through OVN Interconnection (IC). A port on the remote cluster is visible under the type `remote` in the local SBDB. This allows a logical switch or router to be extended across Kubernetes clusters or across data-centre regions.

```bash
[local-cluster]# ovn-ic-nbctl ts-add ts-shared
[local-cluster]# ovn-ic-nbctl ts-set-route ts-shared 10.0.0.0/16 <remote-cluster>
```

Rarely used in a single-cluster deployment. See Part 15.1, the OVN-IC section, for the multi-cluster pattern.

## 13.5.10 Anatomy Template A, `ovn-sbctl list Port_Binding`

**Scenario.** Debugging "the VM has no network", step 2 of the 5-step diagnostic (after the NBDB LSP-exists check).

**Command and output** *(reproduced on OVN 22.03.8, VM lsp_web_01 on chassis-hypervisor-A)*:

```shell
$ sudo ovn-sbctl --columns=logical_port,chassis,type,tunnel_key,mac,up,options list Port_Binding lsp_web_01

logical_port    : "lsp_web_01"
chassis         : [<chassis_A_uuid>]
type            : ""
tunnel_key      : 4
mac             : ["aa:bb:cc:00:00:01 10.0.0.5"]
up              : true
options         : {requested-chassis="chassis-A"}
```

**9-attribute anatomy:**

| Field | Sample value | Meaning | Red flag if |
|-------|--------------|---------|-------------|
| `logical_port` | `"lsp_web_01"` | Must match `ovs-vsctl get Interface tap external_ids:iface-id` | Mismatch means the claim never triggers |
| `chassis` | `[<uuid>]` or `[]` | The chassis currently holding the claim. `[]` means unbound | Empty plus VM running means the claim failed (no iface-id on OVS) |
| `type` | `""`, `localnet`, and so on | 12 types at v22.03.8 (`controller/binding.c` `get_lport_type()`). `""` resolves to VIF or CONTAINER depending on `parent_port`. | A mismatch with the expected use case is a misconfiguration |
| `tunnel_key` | `4` | Per-datapath unique ID | A collision (two ports with the same tunnel_key) is a northd bug |
| `mac` | `["aa:bb:... 10.0.0.5"]` | MAC and addresses from NBDB | Missing IP means `addresses=unknown` and the DHCP and ARP responders do not work |
| `up` | `true` or `false` | Binding state | `false` means a chassis claimed the port but the interface is down |
| `options:requested-chassis` | `"chassis-A"` | Optional chassis pin | A conflict with the physical location of the VM |
| `options:activation-strategy` | `"rarp"` (OVN 24.03 and later) | Migration activation | Missing means the older migration behaviour applies |
| `additional_chassis` | `[]` or `[uuid]` | Multichassis binding (migration window) | Non-empty without an active migration is a leaked binding |

**Failure scenarios:**

- **`chassis=[]` plus VM running plus no iface-id on the tap.** The CMS did not set iface-id. Fix: `ovs-vsctl set Interface tap-vmX external_ids:iface-id=lsp_web_01`. After 1 to 2 seconds, `ovn-controller` claims.
- **`chassis=[B]` while the VM is on chassis A.** A stale claim from chassis B (crashed or network partition). Fix: restart `ovn-controller` on A, or run `ovn-sbctl chassis-del B` manually if B is certainly dead.
- **`additional_chassis=[C]` stuck after the migration finished.** An OVN 22.06 migration edge case. Fix: restart `ovn-controller` on C manually, or run `ovn-sbctl set Port_Binding <uuid> additional_chassis=[]`.
- **`up=false` plus `chassis=[A]`.** The interface is operationally down. Check `ovs-vsctl --columns=name,admin_state,link_state list Interface tap-vmX`. If `link_state=down`, the issue is at the physical layer (link flapping).

## 13.5.11 ▶ Guided Exercise, observe the Port_Binding claim workflow

**Goal.** Observe `ovn-controller` claiming a Port_Binding when a VIF is plugged and releasing it when the VIF is unplugged, and measure the latency.

**Environment.** A 2-chassis lab (Part 13.4 GE 1) with `ovn-central` on node1 and `ovn-host` on node1 plus node2.

**Steps:**

1. On node1 (central), create an LSP that is not yet bound:
   ```bash
   node1$ sudo ovn-nbctl ls-add ls-claim-test
   node1$ sudo ovn-nbctl lsp-add ls-claim-test vm-claim-test
   node1$ sudo ovn-nbctl lsp-set-addresses vm-claim-test "aa:bb:cc:de:ad:01 10.0.0.99"
   ```
2. Verify `chassis=NULL`:
   ```bash
   node1$ sudo ovn-sbctl --columns=logical_port,chassis,up list Port_Binding vm-claim-test
   # Expected: chassis=[], up=false
   ```
3. Monitor SBDB change in real time (in a separate terminal):
   ```bash
   node1$ sudo ovsdb-client -v monitor \
          unix:/var/run/ovn/ovnsb_db.sock OVN_Southbound Port_Binding \
          chassis logical_port 2>&1 | grep vm-claim-test
   ```
4. On node2, plug a tap with the iface-id:
   ```bash
   node2$ sudo ovs-vsctl add-port br-int tap-claim -- \
          set Interface tap-claim type=internal \
          external_ids:iface-id=vm-claim-test
   ```
5. Observe the SBDB change in under 1 second:
   ```
   row "lsp_web_01" (vm-claim-test): chassis [] -> [<node2-chassis-uuid>]
   ```
6. Re-check:
   ```bash
   node1$ sudo ovn-sbctl --columns=logical_port,chassis,up list Port_Binding vm-claim-test
   # Expected: chassis=[<node2-uuid>], up=true
   ```
7. Unplug, then observe the release:
   ```bash
   node2$ sudo ovs-vsctl del-port br-int tap-claim
   # The observer shows: chassis [<uuid>] -> []
   node1$ sudo ovn-sbctl --columns=chassis,up list Port_Binding vm-claim-test
   # Expected: chassis=[], up=false
   ```

**POE questions:**
- **Predict.** If the tap is plugged into node1 at the same time as node2, do both chassis claim? What does SBDB show?
- **Observe.** An OVSDB transaction UPDATE is CAS (compare-and-swap). Only one chassis succeeds (whichever sends the transaction to the leader first). The other chassis sees a transaction conflict, retries, and finds that the chassis field has already been set, so it does not claim.
- **Explain.** Raft consensus guarantees exactly one claimer at a time. If a dual-chassis state is required (the migration scenario), use `additional_chassis` (OVN 22.06 and later), a separate column that does not conflict with `chassis`.

**Cleanup:**
```bash
node1$ sudo ovn-nbctl ls-del ls-claim-test
```

## 13.5.12 Debug Port_Binding, the 5-step diagnostic

When a VM "has no network", trace Port_Binding through these 5 steps:

**Step 1, does the NBDB LSP exist?**
```bash
node1$ sudo ovn-nbctl show ls-tenant1 | grep <vm-mac>
```
If it does not exist, the CMS has not created the LSP. Fix at the CMS.

**Step 2, does the SBDB Port_Binding exist?**
```bash
node1$ sudo ovn-sbctl --columns=logical_port,chassis,up,type list Port_Binding <lsp_name>
```
If it does not exist, `ovn-northd` has not compiled. Check the `ovn-northd` log and status.

**Step 3, does the OVS tap have a matching iface-id?**
```bash
node2$ sudo ovs-vsctl --columns=name,external_ids list Interface | grep <lsp_name>
```
If it does not exist, the CMS did not set iface-id. Fix at the CMS.

**Step 4, has Port_Binding.chassis been claimed?**
```bash
node1$ sudo ovn-sbctl --columns=chassis list Port_Binding <lsp_name>
```
If it is empty 5 seconds after step 3, `ovn-controller` has crashed or the OVSDB connection is dead. Check `ovn-appctl connection-status`.

**Step 5, has the OpenFlow rule for the VM been installed?**
```bash
node2$ sudo ovs-ofctl -O OpenFlow13 dump-flows br-int | grep <vm-mac>
```
If it does not exist, `ovn-controller` crashed after the claim. Check `/var/log/ovn/ovn-controller.log`.

> **Common misconception.** *"Port_Binding exists in NBDB."*
>
> **Reality.** Port_Binding only exists in **SBDB**, not NBDB. NBDB holds `Logical_Switch_Port` (the LSP configuration intent). `ovn-northd` compiles the LSP into a Port_Binding in SBDB. The confusion arises when an operator queries the wrong table: `ovn-nbctl list Port_Binding` returns an error because NBDB does not have that table. The correct command is `ovn-sbctl list Port_Binding`.

## 13.5.13 Key takeaways

First, Port_Binding is the link between a logical port (NBDB LSP/LRP) and a physical chassis (SBDB runtime state). At v22.03.8 the 12 types enumerated by `controller/binding.c` `get_lport_type()` serve distinct use cases: default VIF (about 95 percent of workloads), `container` (a VIF with a `parent_port` set), `localnet` (provider network), `chassisredirect` (gateway), `patch` (inter-datapath), `l2gateway` and `vtep` (hardware bridge integration), `localport` (metadata service), `virtual` (VRRP), `external` (gateway-attached external port), `remote` (inter-cluster), and `l3gateway` (legacy L3 gateway).

Second, the default VIF claim depends **entirely** on the `external_ids:iface-id` match between an OVS port and the NBDB LSP. A missing iface-id means no claim, which means no network. This is the most common pain point.

Third, `chassisredirect` plus HA_Chassis_Group plus BFD is the north-south NAT pattern. Failover takes 3 to 5 seconds with default BFD and under 1 second with tuned BFD. Only one chassis is active at any time (not distributed like a Logical Router).

Fourth, `localport` exists on every chassis. It is used for metadata services that scale linearly, and the traffic does not cross chassis.

Fifth, the 5-step Port_Binding diagnostic (NBDB LSP, SBDB PB, OVS iface-id, claim chassis, OpenFlow rule) is the standard workflow for debugging "the VM has no network". Knowing this sequence reduces MTTR from hours to minutes.

Sixth, Port_Binding only exists in SBDB (not NBDB). Query the correct table when debugging.

### Summary of the 10 companion SBDB tables

#### Chassis

**How it works.** A Chassis row is created on demand when `ovn-controller` starts up. Key columns: `name` (the chassis ID, usually the hostname), `encaps` (a reference to Encap rows), `hostname`, and `external_ids` (for example, `system-id`, `ovn-bridge-mappings`).

**Version differences.** Stable since OVN 2.4. OVN 22.03 added `other_config:is-remote`.

**Real-world incident.** A Chassis row must be pruned after a node is decommissioned; a leftover row prevents re-adding a node with the same name.

**Exercise.** Run `ovn-sbctl list Chassis | head` to observe.

**Comparison with other systems.** Equivalent to a Kubernetes Node object. NSX-T Transport Node has similar semantics.

#### Encap

**How it works.** Tunnel-encapsulation configuration: `type` (geneve, vxlan, stt), `ip`, and `options`. Referenced by the `Chassis.encaps` array.

**Version differences.** Stable since OVN 2.4.

**Exercise.** Run `ovn-sbctl list Encap` to observe.

**Comparison with other systems.** Equivalent to IPIP or VXLAN configuration in a Kubernetes CNI.

#### MAC_Binding

**How it works.** Stores ARP and ND bindings learned on a Logical Router. Key columns: `logical_port` (LRP), `ip`, `mac`, `datapath`, `timestamp`.

**Version differences.** Initial in OVN 2.4. OVN 22.03 added the `timestamp` column and the `MAX_MAC_BINDING` cap per LR.

**Real-world incident.** Production case, a stale MAC_Binding caused a routing blackhole (see the PMTUD case in Part 19.0 and REF Section 4 Scenario 11).

**Exercise.** Run `ovn-sbctl find MAC_Binding ip=10.0.0.5` to query.

**Comparison with other systems.** Equivalent to the ARP cache `ip neigh show` in the Linux kernel. The Cisco IOS ARP table has similar semantics.

#### FDB

**How it works.** MAC learning on a Logical Switch. Key columns: `dp_key`, `mac`, `port_key`. `ovn-controller` populates entries through the `REGBIT_LOOKUP_FDB` and `REGBIT_HIT_LOOKUP_FDB` pipeline (see Part 13.18).

**Version differences.** OVN 22.03 standardised the FDB table (replacing the earlier learning in the OVS bridge).

**Real-world incident.** A stale FDB entry leads to unknown-unicast flood (see the LS broadcast-storm case in Part 13.2).

**Exercise.** Run `ovn-sbctl list FDB | head` to observe.

**Comparison with other systems.** Direct equivalent of `bridge fdb show` on the Linux bridge.

#### Service_Monitor

**How it works.** One health-monitor row per LB endpoint. Key columns: `ip`, `protocol` (tcp, udp, icmp), `port`, `src_ip`, `src_mac`, `status` (online, offline, error).

**Version differences.** Introduced in OVN 22.03. OVN 24.03 expanded the options per protocol.

**Real-world incident.** A backend is automatically removed from the LB when its health check fails, the standard ELB production pattern.

**Exercise.** Run `ovn-sbctl list Service_Monitor` to observe.

**Comparison with other systems.** Direct equivalent of an AWS ELB health check. The Linux LVS `ipvsadm --get-state` has similar semantics.

#### Multicast_Group and IGMP_Group

**How it works.** Multicast_Group: a multicast forwarding entry per datapath, identified by `tunnel_key`. IGMP_Group: tracks IGMP membership per datapath, populated by IGMP snooping.

**Version differences.** OVN 21.06 added IGMP snooping.

**Exercise.** Run `ovn-sbctl list Multicast_Group; ovn-sbctl list IGMP_Group` to observe.

**Comparison with other systems.** Direct equivalent of `bridge mdb show` on the Linux bridge. Cisco IGMP snooping has similar semantics.

#### BFD

**How it works.** One BFD session per HA gateway. Key columns: `dst_ip`, `src_port`, `status` (down, init, up). The protocol is RFC 5880.

**Version differences.** Standardised in OVN 22.03.

**Real-world incident.** Detection of HA gateway failover in under 200 milliseconds (see Part 13.6 on BFD).

**Exercise.** Run `ovn-sbctl list BFD` to observe.

**Comparison with other systems.** Direct equivalent of standard BFD on Cisco IOS. The `bird` BFD daemon on Linux has similar semantics.

#### HA_Chassis

**How it works.** A member entry in HA_Chassis_Group. Key columns: `chassis_name`, `priority`. The highest-priority chassis currently active is the redirect-chassis at that moment.

**Version differences.** Stable since OVN 21.06.

**Real-world incident.** Failover trigger: BFD reports DOWN, priorities are recomputed, and a new active chassis is selected.

**Exercise.** Run `ovn-sbctl list HA_Chassis` to observe.

**Comparison with other systems.** Cisco HSRP standby groups have similar semantics.

#### Connection

**How it works.** OVSDB connection metadata: `target` (URI), `protocol` (tcp, ssl, punix), `inactivity_probe`, `max_backoff`.

**Version differences.** Stable per RFC 7047.

**Exercise.** Run `ovn-sbctl list Connection` to observe.

**Comparison with other systems.** No direct cross-domain equivalent (specific to OVSDB). The closest concept is the DNS A record (endpoint discovery).

## What R2.3 proves, and why a Cisco engineer needs to know

### The single property R2.3 establishes

OVS supports 14 interface types per the `iface_types` column on this lab host. The reader's first-pass instinct is that "interface" is one thing and the type column is just metadata. The instinct is wrong. The interface type determines who owns the netdev's lifecycle (the kernel, OVS, or a userspace process), where the netdev's wire endpoint lives (in the bridge alone, in a Linux network namespace, in a userspace open file descriptor, on a physical NIC, on a tunnel transport), and whether the netdev survives the port being removed from the bridge. R2.3 demonstrates the ownership and lifecycle differences with three live ports on the same bridge: an `internal` port the operator configures with `add-port ... type=internal`, a `tap` port the operator created beforehand with `ip tuntap`, and a system-veth pair whose host-side end is added to the bridge.

The proof comes from a single observable. After all three ports are configured and the bridge is showing them, the operator deletes each port from the bridge with `ovs-vsctl del-port`. The internal port's netdev is gone (`ip link show int0` reports the device does not exist). The tap port's netdev is still present (`ip link show tap0` works, the device sits in DOWN state). The veth pair's bridge-side end is still present (`ip link show veth-a`). Three netdevs, three different post-delete states. The reader who has watched this happen can never again confuse "type internal" with "type system"; the difference is that OVS *owns* the internal netdev's lifetime and *does not own* the others.

### Why an operator cares

A Cisco-trained reader has only one mental model of "an interface": a piece of hardware on the chassis, optionally extended with a virtual SVI inside the supervisor. There is no concept of "interface that disappears when you unplug it from the bridge" because the chassis owns every port permanently. On Open vSwitch the equivalent question becomes "who registered this netdev with the kernel". The kernel registers a netdev when something asks it to: `ip tuntap add` asks for a TUN/TAP, `ip link add type veth` asks for a veth pair, `ovs-vsctl add-port ... type=internal` asks OVS to ask the kernel for an `internal`-type netdev. The thing that asks for the netdev owns its destruction; OVS specifically owns destruction only for the types it requested itself.

The operational consequences matter. An operator who runs `ovs-vsctl del-port br0 tap0` and expects `tap0` to be gone will be surprised when KVM's qemu-system-x86_64 process is still holding the file descriptor; the netdev stays alive until the qemu process releases it, which means the next `ovs-vsctl add-port` for the same name can collide. R9 (debug-and-troubleshoot) returns to this in the context of "phantom port" bugs.

The Cisco analogue, made explicit. On a Catalyst, every interface is a permanent physical port; running `no interface ...` cannot remove the port, only its configuration. On Open vSwitch, an `internal` interface is the closest equivalent of a Cisco SVI (created by OVS, lifetime bound to the bridge), a `tap` interface is the closest equivalent of a software-loopback that another process (a VM) opens (created by the userspace process, lifetime bound to that process), and a `system` interface is the closest equivalent of an external port (created by something else entirely, lifetime bound to whatever made it). R2.3 makes the three lifetimes concrete.

### A note on what type=`""` means

When the reader runs `ovs-vsctl get Interface tap0 type` later in this lab, the daemon replies with `""` (the empty string in OVSDB notation, two ASCII double quotes). The empty string is the OVS-internal sentinel for "system": OVS treats any unset type as a wrapper around an existing kernel netdev. The relationship between the schema-level value (an empty string) and the `iface_types` list (which contains the literal token `system`) is not arbitrary. The provider registration in `lib/netdev-provider.h` registers a provider with a type name; at lookup time, when the operator creates an interface with type=`""`, OVS internally maps the empty string to `system` and uses the system provider. The mapping is documented in `lib/netdev.c:netdev_lookup_class()` and is the reason the reader sometimes sees `system` and sometimes sees `""` for the same interface; they are equivalent on the wire.

## Type 1: an internal port that OVS owns end to end

### Build

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br0 int0 -- set Interface int0 type=internal
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=name,type,ofport,mac_in_use,admin_state list Interface int0
name                : int0
type                : internal
ofport              : 1
mac_in_use          : "b2:fb:2c:fe:12:ac"
admin_state         : down
root@lab-openvswitch:~/network-onboard# ip link show int0
121: int0: <BROADCAST,MULTICAST> mtu 1500 qdisc noop state DOWN mode DEFAULT group default qlen 1000
    link/ether b2:fb:2c:fe:12:ac brd ff:ff:ff:ff:ff:ff
```

The single command `ovs-vsctl add-port br0 int0 -- set Interface int0 type=internal` is a transactional pair: first add a port named `int0` to bridge `br0`, then update the auto-created interface row to have type `internal`. The `--` separator is OVSDB's "next command" marker; both commands run in one transaction. After the command, the interface row shows `type : internal`, OF port 1, a kernel-assigned MAC `b2:fb:..:ac`, and the admin state is down.

The crucial fact lives in `ip link show int0`: the kernel reports `121: int0: ...`, ifindex 121, in DOWN state. The kernel netdev was created when OVS issued the netlink `RTM_NEWLINK` request. OVS asked the kernel to create a netdev of the OVS-internal type; the kernel created it; the kernel-assigned MAC was reflected back into the OVSDB Interface row. The netdev exists because OVS asked for it, and the lifecycle binding is tight: when OVS removes the port, OVS will issue a corresponding `RTM_DELLINK` and the netdev will vanish. R2.3 will demonstrate that vanishing in the cleanup section.

### Configure L3 on the internal port

```text
root@lab-openvswitch:~/network-onboard# ip link set int0 up
root@lab-openvswitch:~/network-onboard# ip addr add 10.10.10.1/24 dev int0
root@lab-openvswitch:~/network-onboard# ip addr show int0
121: int0: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 qdisc noqueue state UNKNOWN group default qlen 1000
    link/ether b2:fb:2c:fe:12:ac brd ff:ff:ff:ff:ff:ff
    inet 10.10.10.1/24 scope global int0
       valid_lft forever preferred_lft forever
    inet6 fe80::b0fb:2cff:fefe:12ac/64 scope link tentative 
       valid_lft forever preferred_lft forever
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=name,type,ofport,admin_state,link_state,mtu list Interface int0
name                : int0
type                : internal
ofport              : 1
admin_state         : up
link_state          : up
mtu                 : 1500
```

Two operator commands: bring the netdev up, assign it the IPv4 address 10.10.10.1/24. Both succeeded. The result is that `int0` is now an L3-attachable interface inside the host's root namespace; the host can route to 10.10.10.0/24 through `int0`, and any frame the bridge floods or directs to OF port 1 will land inside `int0` and be processed by the host's IP stack as if the frame had arrived on a real NIC.

The two views (`ip addr show` and `ovs-vsctl list Interface`) agree on `state UP` and `mtu 1500`. The `link_state : up` in OVSDB is OVS's reflection of what the kernel reports; it is ephemeral (cleared on `ovs-vswitchd` restart) and re-populated on the next IDL roundtrip. The kernel reports `state UNKNOWN`, which is an OVS-specific quirk: internal ports do not have a real link partner (there is no peer end to query), so the kernel cannot determine "carrier present" in the conventional sense; UNKNOWN means "treat as up". OVS interprets UNKNOWN as up for the link_state column.

### What `internal` is structurally

```ascii
                  Host root namespace
                  +---------------------+
                  | ip addr 10.10.10.1  |
                  | int0 (ifindex 121)  |
                  | type=internal       |
                  +----------+----------+
                             |
                             | OVS-synthesised "wire"
                             | (kernel <-> ovs-vswitchd)
                             v
                  +---------------------+
                  | br0 OF port 1       |
                  | DP port 2           |
                  +---------------------+
```

The internal port has no peer. There is no remote end to plug into. The kernel netdev is a software-only construct that OVS implements: a single endpoint that, when the host sends to it, OVS treats as ingress on OF port 1; when OVS outputs to OF port 1, the kernel delivers to the host as if on ingress. The internal port is the OVS equivalent of a Cisco Catalyst's `interface vlan N`: a software interface the operator uses to give the bridge an L3 presence in the host's IP stack.

Source anchor. The internal-port provider is `lib/netdev-vport.c` (despite the name, internal ports are registered by netdev-vport because they share the vport infrastructure with tunnels). The kernel-side implementation lives in `datapath/vport-internal_dev.c` upstream and in the kernel's in-tree `net/openvswitch/vport-internal_dev.c` on the lab host. The internal port's MAC is generated by `eth_random_addr()` in the kernel's `net/openvswitch/vport-internal_dev.c:internal_dev_create()`.

## Type 2: a TAP port created by the userspace stack

### Build

```text
root@lab-openvswitch:~/network-onboard# which tunctl ip
/usr/sbin/ip
root@lab-openvswitch:~/network-onboard# ip tuntap add mode tap dev tap0
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br0 tap0
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=name,type,ofport,mac_in_use list Interface tap0
name                : tap0
type                : ""
ofport              : 2
mac_in_use          : "5a:4a:dd:48:3b:3d"
root@lab-openvswitch:~/network-onboard# ip link show tap0
122: tap0: <BROADCAST,MULTICAST> mtu 1500 qdisc noop master ovs-system state DOWN mode DEFAULT group default qlen 1000
    link/ether 5a:4a:dd:48:3b:3d brd ff:ff:ff:ff:ff:ff
```

Two notable details. The first command, `which tunctl ip`, reports only `/usr/sbin/ip`; the older `tunctl` utility is not installed on Ubuntu 22.04 because `ip tuntap` is the modern replacement. The second command, `ip tuntap add mode tap dev tap0`, creates a TAP device named `tap0`. A TAP device is a kernel-synthesised L2 endpoint with an associated `/dev/net/tun` file descriptor; one side is a Linux netdev (visible to `ip link`), the other side is a userspace file descriptor that a process can open to send and receive Ethernet frames.

When OVS adds `tap0` to the bridge with `ovs-vsctl add-port br0 tap0` (no type specifier), the OVSDB Interface row's type stays at the default `""`, which OVS internally maps to `system`. OVS does not own the lifecycle of `tap0`; it is wrapping a netdev that already existed.

The `ip link show tap0` output shows the new tag `master ovs-system`. This is the kernel's record of which OVS datapath has claimed the netdev; OVS sets this field via the `OVS_VPORT_CMD_NEW` netlink message. The `master` field is the kernel's bridge-membership marker, repurposed by OVS as a "claimed by datapath" marker. For an `internal` port the same field would be `master ovs-system` too; for a port that is *not* in any OVS datapath the field is empty.

### A subtle observation: the link state is down even after `up`

```text
root@lab-openvswitch:~/network-onboard# ip link set tap0 up
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=name,type,ofport,admin_state,link_state list Interface tap0
name                : tap0
type                : ""
ofport              : 2
admin_state         : up
link_state          : down
```

After `ip link set tap0 up`, the OVSDB Interface row reports `admin_state : up` (the operator's intention) but `link_state : down` (the kernel's reality). The link is down because no userspace process has opened `/dev/net/tun` and bound to `tap0`. The TAP device's "carrier present" condition depends on something at the userspace end of the file descriptor; with no qemu, no DHCP relay, no `tunctl` user-mode process, the link stays down.

The pedagogical point. A TAP port's link state is not under the operator's control via `ip link`; it is under the control of whatever userspace process is meant to be the peer. R3 will configure flow rules that match on `link_state` to drop frames during partial failure; here, the reader only needs to know that the column exists and that it can disagree with the admin state.

The ASCII model:

```ascii
                  Host root namespace
                  +---------------------+
                  | tap0 (ifindex 122)  |
                  | type="" (system)    |
                  +----------+----------+
                             |
                             |
                             v
                  +---------------------+
                  | /dev/net/tun        |
                  | (fd opened by       |
                  |  a userspace        |
                  |  process: qemu,     |
                  |  tunctl, etc)       |
                  +---------------------+

  OVS view:
  +----------------+
  | br0 OF port 2  |
  | DP port 3      |
  +----------------+
  (master ovs-system is the kernel-side flag set by OVS)
```

R7 (QoS lab) and R5 (tunnels) both use TAP ports as endpoints; the reader of R2.3 will see them again in those contexts.

Source anchor. The TAP / TUN driver is a kernel built-in at `drivers/net/tun.c`. OVS does not have specific TAP code; the system provider in `lib/netdev-linux.c` handles every kernel-managed netdev, including TAPs. The interaction between OVS and the TAP fd-side is mediated entirely by the kernel; OVS only sees the netdev side.

## Type 3: a system veth pair, host-side end on the bridge

### Build

```text
root@lab-openvswitch:~/network-onboard# ip link add veth-a type veth peer name veth-b
root@lab-openvswitch:~/network-onboard# ip link set veth-a up
root@lab-openvswitch:~/network-onboard# ip link set veth-b up
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br0 veth-a
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=name,type,ofport,mac_in_use list Interface veth-a
name                : veth-a
type                : ""
ofport              : 3
mac_in_use          : "ce:00:9b:26:62:e0"
root@lab-openvswitch:~/network-onboard# ip link show veth-a
124: veth-a@veth-b: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 qdisc noqueue master ovs-system state UP mode DEFAULT group default qlen 1000
    link/ether ce:00:9b:26:62:e0 brd ff:ff:ff:ff:ff:ff
```

A veth pair is the third common port type. R2.2 already used veth pairs to bridge namespaces; here the pair is created without namespaces (both ends in the host's root namespace) so the reader can focus on the OVS-side properties.

The four salient differences from `tap0` are. First, `link_state` is up because `veth-b` is also up and the two ends of a veth pair carrier-detect each other; veth links are symmetric. Second, `state UP mode DEFAULT` reports the kernel sees the link as fully operational, not UNKNOWN. Third, `veth-a@veth-b` notation explicitly names the peer end. Fourth, the OVSDB Interface row's `type : ""` matches `tap0`'s; both are wrappers around pre-existing kernel netdevs.

The Cisco analogue here is a bidirectional cable between two ports on the same switch (a hairpin loopback). The veth's `veth-a` side is on the bridge; the `veth-b` side is in the host's root namespace, reachable by IP if configured. Frames sent from `veth-b` enter the bridge through `veth-a`; frames the bridge directs to `veth-a` exit through `veth-b`.

```ascii
       Host root namespace
       +---------------------+
       | veth-b              |
       | (ifindex 123)       |
       +----------+----------+
                  | veth pair
       +----------+----------+
       | veth-a               |
       | (ifindex 124,        |
       |  master ovs-system)  |
       +----------+----------+
                  |
                  v
       +---------------------+
       | br0 OF port 3       |
       | DP port 4           |
       +---------------------+
```

Source anchor. The veth driver is the kernel built-in `drivers/net/veth.c`. OVS handles veth ends through the system provider in `lib/netdev-linux.c`; from OVS's perspective veth is indistinguishable from any other kernel netdev.

## All three types coexisting on the bridge

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl show
15b3fd6f-e30f-4ae3-b8ca-4531189afee5
    Bridge br0
        Port veth-a
            Interface veth-a
        Port br0
            Interface br0
                type: internal
        Port tap0
            Interface tap0
        Port int0
            Interface int0
                type: internal
    ovs_version: "2.17.9"
root@lab-openvswitch:~/network-onboard# ovs-appctl dpif/show
system@ovs-system: hit:5 missed:6
  br0:
    br0 65534/1: (internal)
    int0 1/2: (internal)
    tap0 2/3: (system)
    veth-a 3/4: (system)
```

The bridge now has four ports: the auto-created LOCAL (internal), the operator-added `int0` (internal), the wrapped `tap0` (system), and the wrapped `veth-a` (system). Two are internal, two are system. The OF port numbers progress 1, 2, 3 in add-port order; the LOCAL stays at 65534. The DP port numbers are 1, 2, 3, 4 in datapath-vport-creation order: 1 is the LOCAL (created when br0 was created), 2 is `int0` (created when `add-port int0 type=internal` was issued), 3 is `tap0` (created when `add-port tap0` was issued; the netdev existed but OVS had to create the vport entry), 4 is `veth-a`.

Reading the colon-separated `ovs-vsctl get Interface tap0 type` returns `""` (the empty string sentinel for system). The same query for `int0` returns the literal token `internal`. The reader can verify by direct query:

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl get Interface int0 type
internal
root@lab-openvswitch:~/network-onboard# ovs-vsctl get Interface tap0 type
""
root@lab-openvswitch:~/network-onboard# ovs-vsctl get Interface veth-a type
""
root@lab-openvswitch:~/network-onboard# ovs-vsctl get Interface br0 type
internal
```

The bridge LOCAL `br0` reports `internal`, matching `int0` and *not* matching `tap0` or `veth-a`. The four ports decompose into "OVS owns them" (`br0`, `int0`) and "OVS wraps them" (`tap0`, `veth-a`).

## A peek at /sys/class/net for the kernel-side flags

```text
root@lab-openvswitch:~/network-onboard# cat /sys/class/net/int0/type
1
root@lab-openvswitch:~/network-onboard# cat /sys/class/net/tap0/tun_flags 2>&1 || echo 'no tun_flags'
0x1802
root@lab-openvswitch:~/network-onboard# cat /sys/class/net/veth-a/type
1
root@lab-openvswitch:~/network-onboard# cat /sys/class/net/br0/type
1
```

The kernel exposes per-netdev flags under `/sys/class/net/<name>/`. The `type` file holds the ARP hardware-type code from `/usr/include/linux/if_arp.h`; value `1` is `ARPHRD_ETHER`, the standard Ethernet hardware type. All four netdevs report `1`, because all four are Ethernet; the OVS-level type column is a different namespace from the kernel-level ARP hardware type.

The interesting line is the third one: `cat /sys/class/net/tap0/tun_flags` returns `0x1802`. This file exists *only* for TUN/TAP devices and is the kernel's TAP-specific bitmap. The reader can decode the bits against `linux/if_tun.h`:
- 0x0001 = `IFF_TUN` (L3 mode)
- 0x0002 = `IFF_TAP` (L2 mode)
- 0x0800 = `IFF_NAPI`
- 0x1000 = `IFF_VNET_HDR`

The value `0x1802` is `IFF_TAP | IFF_NAPI | IFF_VNET_HDR`, confirming this is a TAP device with NAPI polling and virtio-net header support enabled. The presence of the `tun_flags` file is itself the signal that this is a TAP device; an `internal` port or a `veth` end has no `tun_flags` file. R7 (the QoS lab) revisits the NAPI flag in the context of high-throughput drivers.

## The lifecycle proof: delete each port, observe what survives

### int0 disappears

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl del-port br0 int0
root@lab-openvswitch:~/network-onboard# ip link show int0 2>&1 || echo '(int0 gone)'
Device "int0" does not exist.
(int0 gone)
```

`ovs-vsctl del-port br0 int0` is silent. The next `ip link show int0` returns the kernel's "Device ... does not exist" error. The netdev is gone. The shell's error fallback `(int0 gone)` confirms the operator's prediction. OVS owned the netdev; OVS removed it.

### tap0 stays

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl del-port br0 tap0
root@lab-openvswitch:~/network-onboard# ip link show tap0 2>&1 || echo '(tap0 gone)'
122: tap0: <NO-CARRIER,BROADCAST,MULTICAST,UP> mtu 1500 qdisc fq_codel state DOWN mode DEFAULT group default qlen 1000
    link/ether 5a:4a:dd:48:3b:3d brd ff:ff:ff:ff:ff:ff
```

`ovs-vsctl del-port br0 tap0` is silent; `ip link show tap0` returns the netdev still present. Two pedagogical observations. First, the `master ovs-system` tag has been removed (`ip link show` no longer shows the master field) because OVS released its claim. Second, the new flag `<NO-CARRIER,...>` appeared because, with the bridge no longer providing the carrier for `tap0`, the kernel reports no carrier. The operator created `tap0` with `ip tuntap add`; the operator must clean it up with `ip link del tap0`. OVS has not and will not destroy this netdev.

### veth-a stays

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl del-port br0 veth-a
root@lab-openvswitch:~/network-onboard# ip link show veth-a
124: veth-a@veth-b: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 qdisc noqueue state UP mode DEFAULT group default qlen 1000
    link/ether ce:00:9b:26:62:e0 brd ff:ff:ff:ff:ff:ff
```

Same story. `ovs-vsctl del-port` releases OVS's claim on `veth-a` (the `master ovs-system` tag is gone; the netdev is back to a normal state UP veth end). The peer `veth-b` is still there too, and the carrier is intact because veth pairs don't depend on a bridge for their carrier.

### Final cleanup

```text
root@lab-openvswitch:~/network-onboard# ip link del veth-a 2>/dev/null || true
root@lab-openvswitch:~/network-onboard# ip link del tap0 2>/dev/null || true
root@lab-openvswitch:~/network-onboard# ovs-vsctl --if-exists del-br br0
root@lab-openvswitch:~/network-onboard# ovs-vsctl list-br
root@lab-openvswitch:~/network-onboard# date -u
Thu Apr 30 04:43:48 AM UTC 2026
```

`ip link del veth-a` removes the veth pair (both ends). `ip link del tap0` removes the TAP device. The lab is back to zero netdevs and zero bridges, in 28 seconds wall clock.

The pedagogical point. The operator who reads R2.3 and starts cleaning up an OVS-rich production host must remember: `ovs-vsctl del-port` is necessary but often not sufficient. For internal ports it is sufficient; for tap ports and veth ends, it is one of two steps; the second step is the matching kernel-side `ip link del`.

## Misconception block

> **Misconception / Reality.**
>
> Misconception: "the interface type is just metadata; OVS treats every port the same way internally."
>
> Reality: the type selects which provider in `lib/netdev-*.c` handles the port. `internal` uses the kernel's OVS internal-dev driver (created by OVS, lifetime owned by OVS). `system` (the empty-string sentinel) uses `netdev-linux.c` to wrap a pre-existing kernel netdev (lifetime owned by whoever created the netdev). `tap` is a special-cased variant of `system` for TUN/TAP. `patch` (R2.4) is a userspace-only wire that does not exist as a kernel netdev at all. `geneve`, `vxlan`, `gre`, and the other tunnels use `netdev-vport.c` and create kernel-side vports without netdevs in the conventional sense. The type-to-provider mapping is in `lib/netdev.c:netdev_lookup_class()` and the provider list is built by `netdev_register_provider()` calls scattered across `lib/netdev-*.c` at OVS startup time.

> **Misconception / Reality (second).**
>
> Misconception: "if I add a netdev to a bridge with `ovs-vsctl add-port` and then `ovs-vsctl del-port`, OVS will delete the netdev."
>
> Reality: OVS deletes only what OVS created. Internal ports created via `add-port ... type=internal` are deleted on `del-port` because OVS asked the kernel to create them. Pre-existing kernel netdevs (TAP, veth, physical NIC) are released, not deleted; OVS removes its `master ovs-system` claim and the netdev returns to whatever state it was in before being added. Forgetting this distinction is the most common cause of "phantom port" bugs in production CMS-driven OVS environments where one process creates the netdev and another adds it to a bridge.

## What the reader should now be able to do

A reader who finishes R2.3 can:

- Read an `ovs-vsctl get Interface ... type` value and predict the post-`del-port` state of the netdev.
- Distinguish `internal` from `system` (and from `tap`, which is a system variant) without needing to query the type column, by inspecting `/sys/class/net/<name>/tun_flags` for TAP, `<BROADCAST,MULTICAST>` plus the absence of a peer for internal, `@<peer>` notation for veth.
- Understand why `link_state` can be `down` while `admin_state` is `up` on a TAP port (the userspace fd is not opened) and contrast it with a veth pair where the link is symmetric.
- Recognise the `master ovs-system` flag in `ip link show` as the kernel-side marker of "claimed by an OVS datapath".
- Plan a clean cross-process tear-down on a CMS-driven host: `ovs-vsctl del-port` first, then `ip link del` for each kernel-owned netdev, then `ovs-vsctl --if-exists del-br` last.

A useful summary of the lifecycle question across the three types covered here. Use it as a quick lookup the next time the operator is unsure what a `del-port` will leave behind. For internal, `del-port` removes the kernel netdev because OVS allocated it via the OVS internal-dev driver. For TAP, `del-port` releases OVS's claim and the netdev is left in a `<NO-CARRIER>` state because there is no bridge to provide carrier; the operator must `ip link del` (or wait for the userspace fd-holder to close, depending on the TAP creation flags). For veth, `del-port` releases OVS's claim and the veth pair stays carrier-up because the two ends carrier-detect each other; the operator must `ip link del` to remove either end (deleting one end automatically deletes the other). Internalising this trio saves one debugging round in nearly every cleanup script.

R2.4 introduces the `patch` port, which is the fourth interface flavour in this curriculum and the only one with no kernel netdev at all. R2.6 takes a different cut at the same material by switching the bridge's `datapath_type` between `system` and `netdev`. R2.7 closes the row-UUID round trip on the OVSDB side. R2.8 closes R2 with the byte-for-byte MAC-port binding proof.

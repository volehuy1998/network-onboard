## 9.22.1 Vì sao OpenFlow 1.0 chỉ tồn tại 18 tháng — drama opening

Tháng 12 năm 2009, Open Networking Foundation (ONF) phát hành OpenFlow Switch Specification 1.0.0 — bản đặc tả đầu tiên. Tháng 2 năm 2011, tức chỉ 14 tháng sau, bản 1.1.0 ra đời với một thay đổi kiến trúc triệt để: single flow table bị thay thế bằng **multi-table pipeline**. Trong lịch sử công nghiệp viễn thông, hiếm có standard nào thay đổi core abstraction nhanh đến vậy — TCP/IP giữ cấu trúc từ 1981, VLAN 802.1Q giữ frame format từ 1998.

Lý do: khi triển khai OpenFlow 1.0 lên silicon Broadcom Trident (chip switch thương mại dòng Tomahawk/Trident của Arista, Dell, Cisco Nexus 9000), engineer phát hiện không cách nào ánh xạ một single flow table của OpenFlow sang kiến trúc bên trong — ASIC Trident có **bảy bảng logic** (ingress parser → ingress ACL → L3 forwarding → L2 forwarding → egress ACL → queueing → egress parser). Muốn biểu diễn một đơn vị phần cứng thực tế, OpenFlow 1.0 cần phải emulate toàn bộ bảy bảng qua một bảng ảo — dẫn đến table explosion (cross-product của tất cả match field, có thể lên tới vài triệu entry cho một config đơn giản).

ONF TM WG ra báo cáo tháng 7/2010 chỉ ra vấn đề, và đến 02/2011 OpenFlow 1.1 công bố cấu trúc **N flow tables numbered 0 to N-1**, với hai instruction mới: `Goto-Table` và `Write-Metadata`. Bài học lâu dài: abstraction tốt phải giữ dáng dấp phần cứng thực, không cắt rời khỏi nó. Multi-table là thỏa hiệp giữa tính "clean" của SDN và tính "grounded" của silicon.

### OVS đi trước standard

Open vSwitch có cơ chế tương tự từ **trước** khi OpenFlow 1.1 ra đời — action `resubmit` đã tồn tại từ OVS 1.0 (tháng 6/2011, cùng thời điểm OpenFlow 1.1), kế thừa từ project NOX (Stanford, 2008). OVS dev team hiểu rõ hạn chế single-table nhiều năm trước ONF và đã có giải pháp riêng. Về sau OVS hỗ trợ cả hai: `goto_table` theo chuẩn OpenFlow 1.1+ và `resubmit` theo extension NOX/OVS. §9.22.3 phân tích khác biệt.

## 9.22.2 Kiến trúc pipeline multi-table — từ OpenFlow 1.3.5 spec

OpenFlow Switch Specification 1.3.5 §5.1 *"Pipeline Processing"* định nghĩa:

*"OpenFlow switches are required to have a minimum of one flow table, and can optionally have more. An OpenFlow switch with only a single flow table is valid, in this case pipeline processing is greatly simplified. The flow tables of an OpenFlow switch are sequentially numbered, starting at 0. Pipeline processing always starts at the first flow table: the packet is first matched against flow entries of flow table 0."*

Bốn quy tắc cứng mà multi-table pipeline phải tuân theo:

**Quy tắc 1 — Luôn bắt đầu ở table 0.** Mọi packet vào pipeline phải match entry trong table 0 trước. Không có cách nào inject packet trực tiếp vào table 5. Đây là thiết kế đơn giản hóa debugging — khi packet "biến mất", bắt đầu trace từ table 0.

**Quy tắc 2 — Chỉ đi xuôi, không đi ngược.** Spec ghi rõ: *"A flow entry can only direct a packet to a flow table number which is greater than its own flow table number"*. Table 3 có thể `goto_table=5` nhưng không thể `goto_table=2`. Đây là đảm bảo **termination** — pipeline không lặp vô hạn.

**Quy tắc 3 — Action set tích luỹ, không execute ngay.** Khi packet match entry ở table N với instruction `Apply-Actions`, action được thực thi ngay. Nhưng với instruction `Write-Actions`, action chỉ được ghi vào **action set** của packet, chưa thực thi. Cuối pipeline (khi không còn `goto_table`), toàn bộ action set được thực thi. Cơ chế này cho phép các table downstream override quyết định upstream.

**Quy tắc 4 — Mỗi table chỉ match tối đa một entry.** Khi packet match được nhiều entry cùng table, entry priority cao nhất thắng. Match hai entry cùng priority cùng table là behavior undefined (OVS chọn ngẫu nhiên). Nên mọi entry cùng table phải có priority phân biệt hoặc match field disjoint.

### Instruction vs Action

Từ OpenFlow 1.1 trở đi, phân biệt hai khái niệm quan trọng:

**Instruction** là lệnh ở cấp *table entry* — `Apply-Actions`, `Write-Actions`, `Clear-Actions`, `Write-Metadata`, `Goto-Table`. Mỗi entry có tối đa một instruction mỗi loại.

**Action** là thao tác ở cấp *packet* — `output:N`, `mod_dl_src`, `dec_ttl`, v.v. Action nằm bên trong instruction `Apply-Actions` hoặc `Write-Actions`.

`ovs-ofctl` giản hoá cú pháp bằng từ khoá `action=` (hoặc `actions=`) — đây thực ra là instruction `Apply-Actions` implicit. Cú pháp đầy đủ theo spec phải là `instructions=apply_actions(output:1)` nhưng OVS nhận cú pháp rút gọn.

## 9.22.3 `goto_table` (OpenFlow 1.1+) vs `resubmit` (OVS extension)

Hai cơ chế cùng mục tiêu: chuyển packet qua bảng tiếp theo. Khác biệt quan trọng trong implementation:

### `goto_table=N`

Instruction theo chuẩn OpenFlow 1.1+. Đặc tính:

- Chỉ forward đến `N > table_hiện_tại` (chuẩn quy tắc 2).
- Action set tích luỹ theo packet — các `Write-Actions` trước đó vẫn còn ở table đích.
- Metadata field (64-bit) được mang theo — chỉnh bằng `Write-Metadata` thì giá trị mới available cho table đích.
- Không cho phép `goto_table=N` khi N đã ghé qua — mỗi packet đi mỗi table tối đa một lần.

Ví dụ trong Lab 6 Table 0 Classifier:

```bash
sudo ovs-ofctl add-flow s1 "table=0,ip,action=goto_table=1"
```

Packet IP vào table 0 → instruction `Goto-Table=1` → chuyển sang table 1 để tra L3 forwarding.

### `resubmit(,N)` / `resubmit(port,N)`

OVS action không thuộc chuẩn OpenFlow, thừa hưởng từ NOX controller. Đặc tính:

- Có thể forward đến **bất kỳ table nào**, kể cả table nhỏ hơn (có thể gây loop — OVS giới hạn recursion depth = 75 bằng cách abort packet).
- Action ngoài `resubmit` vẫn được thực thi sau khi recirculate trở về — không giống `goto_table` (chuyển hoàn toàn control).
- Có biến thể `resubmit(port,N)` thay `in_port` của packet trước khi tra table N — tiện cho virtual port modeling.
- Không dùng action set/metadata của OpenFlow — mà dùng register `reg0`..`reg15` của OVS extension.

Ví dụ OVS extension pattern:

```bash
# Lookup MAC table, sau đó quay lại forward table
sudo ovs-ofctl add-flow br0 "table=0,dl_type=0x0800,action=resubmit(,10),resubmit(,20),output:2"
```

Packet đi qua table 10 (lookup security), rồi table 20 (lookup MAC), rồi về lại table 0 để execute `output:2`. Pattern này phổ biến trong OVN và NOX-era controller.

### Khi nào chọn cái nào

Trong OpenFlow pipeline truyền thống với hardware-match 1:1 (Broadcom Trident), dùng `goto_table` vì khớp semantic silicon. Trong userspace OVS với complex logic (OVN compiler, security gateways), dùng `resubmit` vì flexibility.

> **Key Topic:** OVN `ovn-controller` sinh flow với **cả hai**. Table layout của OVN dùng `goto_table` để move giữa các logical stage (ingress ACL → LB → routing → egress ACL), nhưng bên trong mỗi stage có nhiều `resubmit` để modular sub-flow. Đọc output `ovs-ofctl dump-flows br-int` trên host OVN sẽ thấy hỗn hợp.

## 9.22.4 Topology Lab 6 và pipeline 3-table

Topology Lab 6 (Crichigno et al., trang 5-7 Lab 6):

```
       192.168.1.0/24                         192.168.2.0/24
          h1 (.10)        s1 ─── s2        h2 (.10)
          MAC :01         (.1)   (.1)      MAC :02
          GW .1          MAC:05  MAC:06    GW .1
                     eth1/eth2     eth1/eth2
```

Hai host khác subnet (192.168.1.0/24 và 192.168.2.0/24), hai switch OVS (`s1` và `s2`) đóng vai router. MAC address được gán thủ công: h1=`:01`, h2=`:02`, s1 interface hướng về s2=`:05`, s2 interface hướng về s1=`:06`.

Cấu hình gateway trên switch:

```bash
# Gán IP cho switch interface (dùng làm default gateway từ góc nhìn host)
sudo ifconfig s1 192.168.1.1/24 up
sudo ifconfig s2 192.168.2.1/24 up
```

### Pipeline ba bảng

**Table 0 — Classifier.** Phân biệt ARP và IP.

```bash
# s1 và s2: ARP xử lý như switch truyền thống
sudo ovs-ofctl add-flow s1 "table=0,arp,action=normal"
sudo ovs-ofctl add-flow s2 "table=0,arp,action=normal"

# IP → chuyển sang table 1 để tra routing
sudo ovs-ofctl add-flow s1 "table=0,ip,action=goto_table=1"
sudo ovs-ofctl add-flow s2 "table=0,ip,action=goto_table=1"
```

Hai entry mỗi switch. ARP broadcast đi qua action `normal` (dựa vào MAC learning nội bộ OVS). IP packet được chuyển sang pipeline dedicated.

**Table 1 — Layer 3 Forwarding.** Rewrite MAC + decrement TTL + chuyển tiếp sang table 2.

```bash
# s1 xử lý gói IP đến host local h1
sudo ovs-ofctl add-flow s1 \
  "table=1,ip,nw_dst=192.168.1.10,action=mod_dl_src=00:00:00:00:00:05,mod_dl_dst=00:00:00:00:00:01,dec_ttl,goto_table=2"

# s1 xử lý gói IP đi sang subnet 192.168.2.0/24 (đến s2)
sudo ovs-ofctl add-flow s1 \
  "table=1,ip,nw_dst=192.168.2.0/24,action=mod_dl_src=00:00:00:00:00:01,mod_dl_dst=00:00:00:00:00:05,dec_ttl,goto_table=2"

# s2 tương tự cho chiều ngược
sudo ovs-ofctl add-flow s2 \
  "table=1,ip,nw_dst=192.168.2.10,action=mod_dl_src=00:00:00:00:00:06,mod_dl_dst=00:00:00:00:00:02,dec_ttl,goto_table=2"
sudo ovs-ofctl add-flow s2 \
  "table=1,ip,nw_dst=192.168.1.0/24,action=mod_dl_src=00:00:00:00:00:02,mod_dl_dst=00:00:00:00:00:06,dec_ttl,goto_table=2"
```

Chú ý cấu trúc rewrite: khi s1 forward packet đi sang s2, MAC src được set thành MAC của s1 hướng s2 (`:01`), MAC dst set thành MAC của s2 hướng s1 (`:05`). Đây chính là hành vi router truyền thống — thay MAC L2 ở mỗi hop, giữ nguyên IP L3.

**Table 2 — Layer 2 Forwarding.** Quyết định egress port dựa trên destination MAC.

```bash
sudo ovs-ofctl add-flow s1 "table=2,dl_dst=00:00:00:00:00:01,action=output:1"   # về h1
sudo ovs-ofctl add-flow s1 "table=2,dl_dst=00:00:00:00:00:05,action=output:2"   # sang s2
sudo ovs-ofctl add-flow s2 "table=2,dl_dst=00:00:00:00:00:02,action=output:1"   # về h2
sudo ovs-ofctl add-flow s2 "table=2,dl_dst=00:00:00:00:00:06,action=output:2"   # sang s1
```

Total: **6 flow entry cho mỗi switch** (2 classifier + 2 L3 + 2 L2) = 12 entry cho cả topology. So với single-table, tiết kiệm đáng kể khi scale — thêm một subnet mới chỉ cần thêm entry ở table 1 và 2, không cần rewrite toàn bộ.

### Tracing packet qua pipeline

Lab 6 (trang 19) dùng `ovs-appctl ofproto/trace` để trace packet IP từ h1 sang h2 qua s1:

```bash
sudo ovs-appctl ofproto/trace s1 \
  "ip,nw_src=192.168.1.10,nw_dst=192.168.2.10,dl_src=00:00:00:00:00:01,dl_dst=00:00:00:00:00:05"
```

Expected output (doc-plausible — sẽ verify ở C1b):

## 9.22.5 Mở rộng lên 5-table pipeline

Compass Chapter 8 giới thiệu mở rộng Lab 6 thành 5-table canonical pipeline cho production (gần với OVN logical switch layout):

```
Table 0 — Port security + VLAN decap
Table 10 — Ingress ACL
Table 20 — Destination lookup (L2 MAC table hoặc L3 route table)
Table 30 — Egress ACL
Table 40 — Output + VLAN encap
```

Từ khoá thiết kế:

- **Gap 10 giữa các table** (0/10/20/30/40 thay vì 0/1/2/3/4) — để chèn table mới sau này không phải renumber. Convention này cũng dùng trong ovn-northd layout.
- **Ingress ACL riêng khỏi Egress ACL** — ACL sau NAT (egress) thấy IP đã bị rewrite, ACL trước NAT (ingress) thấy IP gốc. Sự phân biệt này quan trọng khi debug.
- **Destination lookup gộp L2+L3** — một table cho MAC table và L3 FIB, match trên `dl_dst` hoặc `nw_dst` tuỳ `dl_type`. Cách này đơn giản hơn tách L2 và L3.

### Action set tích luỹ qua pipeline

Khi dùng `Write-Actions` thay `Apply-Actions`, hiệu ứng tích luỹ:

```bash
# Entry table 10 (Ingress ACL): chỉ thêm action vào set, chưa thực thi
sudo ovs-ofctl add-flow br0 \
  "table=10,ip,tcp,tp_dst=22,action=write_actions(set_queue:1),goto_table=20"

# Entry table 40 (Output): khi hết pipeline, action set được thực thi
sudo ovs-ofctl add-flow br0 "table=40,dl_dst=00:00:00:00:00:02,action=output:3"
```

Packet match entry table 10 sẽ có `set_queue:1` trong action set. Khi ra khỏi pipeline (table 40 `output:3`), action set (`set_queue:1`) được apply trước rồi `output:3` — kết quả: packet được đặt vào queue 1 rồi xuất ở port 3.

Hạn chế: action set chỉ chứa **một** action mỗi loại (một `output`, một `set_queue`, một `dec_ttl`, v.v.). Entry downstream override entry upstream — nhưng không cộng dồn cùng loại.

## 9.22.6 Metadata field — truyền state giữa table

OpenFlow 1.1+ định nghĩa `metadata` là 64-bit field chỉ có trong pipeline (không appear trong packet thật). Cú pháp:

```bash
# Entry table 10: đánh metadata 0x100 cho ACL "permit HTTP"
sudo ovs-ofctl add-flow br0 \
  "table=10,ip,tcp,tp_dst=80,action=write_metadata:0x100,goto_table=20"

# Entry table 40: kiểm metadata trước khi output
sudo ovs-ofctl add-flow br0 \
  "table=40,metadata=0x100,dl_dst=00:00:00:00:00:02,action=output:3"
```

Mask cho metadata: `metadata:0x100/0xff00` — chỉ match 8 bit cao. Cơ chế này cho phép table downstream biết quyết định của table upstream mà không cần ghi vào field thật của packet (ví dụ: không muốn modify VLAN tag để pass state).

### Register — OVS extension

OVS mở rộng bằng 16 thanh ghi `reg0`..`reg15` (mỗi cái 32-bit) và `xreg0`..`xreg7` (mỗi cái 64-bit tạo từ cặp reg). Khác biệt với metadata:

- Metadata: standard OpenFlow, 64-bit duy nhất.
- Register: OVS extension, 512-bit tổng cộng, linh hoạt hơn.

OVN dùng cả hai: `metadata` để lưu logical datapath ID (ID của logical switch/router hiện tại), `reg0`..`reg15` để lưu logical input port, output port, conntrack zone, ECMP hash, v.v.

## 9.22.7 So sánh với OVN — `ovn-northd` sinh 50+ table

Operator OVS viết Lab 6 với 3 table thủ công. Operator OVN không viết flow trực tiếp — họ khai báo logical switch + logical router + ACL qua `ovn-nbctl`, `ovn-northd` compile thành logical flow, `ovn-controller` dịch tiếp thành OpenFlow flow trên `br-int`.

Kết quả: `br-int` trên một compute host OVN production thường có **50+ table** với hàng nghìn flow entry. Cấu trúc pipeline OVN (theo `ovn-architecture(7)` man page):

```
Table 0-7   — Logical port binding + classification
Table 8-23  — Ingress pipeline (ACL, LB, NAT, L2 lookup, L3 lookup)
Table 24-31 — Tunneling (Geneve encap)
Table 32-47 — Egress pipeline (ACL, L2 output)
Table 48-63 — Local output
```

Mỗi logical datapath (switch hoặc router) có pipeline riêng được *multiplex* qua `metadata` field — metadata chứa datapath ID, packet của datapath khác không lẫn.

### Khi nào viết thủ công, khi nào dùng OVN

Viết flow thủ công như Lab 6 phù hợp khi:

- Cần *predictable performance* — mọi entry là do bạn viết, không có flow explosion ẩn.
- Cần *tight control* — ví dụ SR-IOV NIC offload yêu cầu specific flow pattern.
- Scale nhỏ — dưới 10 subnet, dưới 100 ACL rule.

Dùng OVN compiler phù hợp khi:

- Scale từ vài chục đến hàng ngàn logical switch/router.
- Cần multi-tenant isolation tự động (conntrack zone per-LS).
- Cần distributed routing — OVN tự sinh flow trên mỗi chassis.

> **Key Topic:** Không có lựa chọn "đúng" tuyệt đối. OpenStack Neutron trước 2018 phần lớn dùng OVS + ML2 plugin sinh flow tay (cấu trúc giống Lab 6 mở rộng). Từ 2018-2020 chuyển sang ML2/OVN — OVN compiler sinh 50+ table tự động. Lý do chính: scale. Khi một compute host có 200 VM từ 50 tenant, viết tay flow không khả thi. Nhưng ở edge node hoặc NFV appliance, nhiều dự án vẫn quay về flow thủ công để control latency.

## Guided Exercise 1 — Trace multi-table pipeline

**Mục đích:** Quan sát trực tiếp hành vi `ovs-appctl ofproto/trace` trên pipeline 3-table Lab 6 và verify assertion trong bài học: (a) packet đi đúng qua 3 table theo thứ tự, (b) `dec_ttl` thực sự giảm TTL, (c) action set không bị mất qua `goto_table`.

**Giả thuyết cần verify:**

> *"Với pipeline 3-table Lab 6, packet IP từ h1 (192.168.1.10) gửi đến h2 (192.168.2.10) sẽ đi qua table 0 → 1 → 2 theo đúng thứ tự. `ovs-appctl ofproto/trace` phản ánh 3 lần lookup riêng biệt, mỗi lần đều report match + action tương ứng, và `Final flow` sẽ có TTL giảm đúng 1 đơn vị so với input."*

### Chuẩn bị

Đã load topology Lab 6 (`lab6.mn` + `set_MACs.sh`). Verify flows đã cài:

```bash
sudo ovs-ofctl dump-flows s1
# Kỳ vọng thấy 6 entry: 2 table 0 (arp/ip), 2 table 1 (L3), 2 table 2 (L2)
```

### Bước 1 — Baseline trace

```bash
sudo ovs-appctl ofproto/trace s1 \
  "ip,nw_src=192.168.1.10,nw_dst=192.168.2.10,dl_src=00:00:00:00:00:01,dl_dst=00:00:00:00:00:05,nw_ttl=64"
```

Expected output (doc-plausible):

```
Flow: ip,in_port=ANY,...,nw_src=192.168.1.10,nw_dst=192.168.2.10,nw_ttl=64

bridge("s1")
-------------
 0. ip, priority 32768
    goto_table:1
 1. ip,nw_dst=192.168.2.0/24, priority 32768
    mod_dl_src:00:00:00:00:00:01
    mod_dl_dst:00:00:00:00:00:05
    dec_ttl
    goto_table:2
 2. dl_dst=00:00:00:00:00:05, priority 32768
    output:2

Final flow: ...,nw_ttl=63
Datapath actions: set(eth(...)),set(ipv4(ttl=-1)),2
```

Ba điểm cần verify trong output:

- **"bridge(s1)"** rồi **"0. ... 1. ... 2. ..."** — ba lookup tuần tự qua 3 table.
- **`nw_ttl=64`** trong `Flow:` → **`nw_ttl=63`** trong `Final flow:` — xác nhận `dec_ttl` hoạt động.
- **`Datapath actions: ...set(ipv4(ttl=-1))...`** — kernel datapath nhận chuỗi action cuối cùng đã compile.

### Bước 2 — POE `goto_table` không reverse

Thử cài flow vi phạm quy tắc 2 (goto table số nhỏ hơn):

```bash
sudo ovs-ofctl add-flow s1 "table=2,ip,action=goto_table=1"
```

Kỳ vọng: OVS reject — `ovs-ofctl: actions are invalid with specified match (OFPBAC_BAD_TABLE_ID)`.

Hoặc nếu cài được (một số version cũ tolerate), khi trace thử packet vào table 2, OVS abort sau 75 lần recirculation:

```bash
sudo ovs-appctl ofproto/trace s1 "ip,..."
# Expected: "stack too deep" hoặc "Actions: (none); packet dropped due to pipeline loop"
```

Giả thuyết "pipeline có thể loop" bị bác bỏ — OpenFlow 1.1+ và OVS đều enforce quy tắc "chỉ đi xuôi".

### Bước 3 — So sánh với `resubmit`

Thay `goto_table` bằng `resubmit` để thấy flexibility:

```bash
sudo ovs-ofctl del-flows s1 "table=0,ip"
sudo ovs-ofctl add-flow s1 "table=0,ip,action=resubmit(,1),resubmit(,2)"
```

Trace lại — kỳ vọng output tương đương pipeline tuần tự nhưng **action `output:2` từ table 2 sẽ thực thi ngay sau khi packet recirculate trở về table 0** (không giống `goto_table` transfer control).

### Bài học

Multi-table pipeline là công cụ modularization: mỗi table giải quyết một concern (ACL, routing, output), giảm kích thước bảng tổng và dễ debug. `goto_table` là "transfer control" — đi đâu không quay lại. `resubmit` là "call subroutine" — quay về chỗ cũ thực thi action còn lại. Biết dùng đúng thời điểm là kỹ năng phân biệt junior và senior OVS engineer.

## 9.22.8 Điểm cốt lõi cần nhớ

**Một.** OpenFlow 1.0 (12/2009) có single flow table. OpenFlow 1.1 (02/2011) thêm multi-table pipeline sau chỉ 14 tháng — vì single-table không biểu diễn được ASIC Broadcom Trident. Đây là thay đổi kiến trúc nhanh hiếm thấy trong lịch sử standard.

**Hai.** Bốn quy tắc cứng của multi-table: bắt đầu table 0, chỉ đi xuôi (table N → table M>N), action set tích luỹ qua pipeline và thực thi cuối cùng, mỗi table chỉ match tối đa một entry (priority resolution).

**Ba.** `goto_table=N` (OpenFlow 1.1+ standard) transfer control hoàn toàn. `resubmit(,N)` (OVS extension từ NOX) recirculate rồi quay về chỗ cũ — cho phép N nhỏ hơn, recursion depth giới hạn 75.

**Bốn.** Pipeline 3-table Lab 6 (Classifier → L3 Forwarding → L2 Forwarding) là template cơ bản cho router đơn giản. Mở rộng 5-table (thêm Ingress ACL + Egress ACL) là production-grade. OVN tự sinh 50+ table.

**Năm.** Metadata (64-bit standard) và register `reg0..15` (OVS extension) truyền state giữa table mà không modify packet field thật. OVN dùng cả hai: metadata cho logical datapath ID, register cho conntrack zone + ECMP hash.

**Sáu.** Viết flow thủ công phù hợp scale nhỏ + cần predictable performance + tight SR-IOV control. OVN compiler phù hợp scale lớn + multi-tenant + distributed routing. Không có "đúng" tuyệt đối — hiểu cả hai, chọn theo bài toán.

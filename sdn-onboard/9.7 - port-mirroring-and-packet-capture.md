## 9.7.1 OVS Mirror table

A mirror is the mechanism that copies traffic from one or more ports or VLANs to a destination (port or VLAN). On a physical switch this is called SPAN or RSPAN. OVS implements it through the `Mirror` table in OVSDB.

A `Mirror` row is referenced from `Bridge.mirrors`. It has:

- **Source selectors:** `select_src_port`, `select_dst_port`, `select_vlan`, `select_all=true`.
- **Destination:** `output_port` (local SPAN) or `output_vlan` (RSPAN, which re-tags the packet with this VLAN and emits it on a trunk).

---

## 9.7.2 Atomic transaction to create a Mirror

Compass Ch G provides the standard idiom:

```bash
[host]# ovs-vsctl -- set Bridge br0 mirrors=@m \
         -- --id=@eth0 get Port eth0 \
         -- --id=@eth1 get Port eth1 \
         -- --id=@eth2 get Port eth2 \
         -- --id=@m create Mirror name=mymirror \
                select-dst-port=@eth0,@eth1 \
                select-src-port=@eth0,@eth1 \
                output-port=@eth2
```

Atomic: if port `eth2` does not exist, the Mirror row is not created and `Bridge.mirrors` does not change.

Verify:

```bash
[host]# ovs-vsctl list Mirror mymirror
[host]# ovs-vsctl get Mirror mymirror statistics
```

The statistics columns `tx_packets` and `tx_bytes` show the number of mirrored packets.

---

## 9.7.3 Cleaning up a Mirror

Compass Ch G: a Mirror is auto-garbage-collected when no longer referenced, which is different from QoS and Queue:

```bash
[host]# ovs-vsctl clear Bridge br0 mirrors
```

After this command, the Mirror row becomes unreferenced and OVSDB deletes it automatically. There is NO need for an explicit `destroy Mirror`. This is an important difference from QoS and Queue (Part 9.9): those two tables must be destroyed manually.

---

## 9.7.4 Capture traffic from a mirror output

Mirror to port `tap0` (internal type); the packet copy appears on netdev `tap0`:

```bash
[host]# ip link set tap0 promisc on
[host]# tcpdump -i tap0 -nn -w mirror.pcap
```

Feed `mirror.pcap` into Wireshark or Suricata.

For RSPAN, with the analyzer on a remote host:

```bash
[host]# ovs-vsctl -- --id=@m create Mirror name=rspan \
         select-all=true output-vlan=999
[host]# ovs-vsctl set Bridge br0 mirrors=@m
```

The remote analyzer joins VLAN 999 on the trunk.

### Guided Exercise 8: mirror VM traffic into a tap capture

**Outcomes:** configure a mirror of the VM port into an internal tap, capture with tcpdump.

**Before You Begin:**

```bash
[host]# ovs-vsctl add-br br-mirror
[host]# ovs-vsctl add-port br-mirror vm1 -- set Interface vm1 type=internal
[host]# ovs-vsctl add-port br-mirror analyzer -- set Interface analyzer type=internal
[host]# ip link set vm1 up && ip addr add 10.99.0.1/24 dev vm1
[host]# ip link set analyzer up
```

**Instructions:**

**1.** Create the mirror:

```bash
[host]# ovs-vsctl -- --id=@m create Mirror name=vm1mirror \
         select-src-port=@p1 select-dst-port=@p1 output-port=@p2 \
         -- --id=@p1 get Port vm1 \
         -- --id=@p2 get Port analyzer \
         -- set Bridge br-mirror mirrors=@m
```

**2.** Start tcpdump: `tcpdump -i analyzer -nn &`.

**3.** Generate traffic: `ping -c 5 -I 10.99.0.1 10.99.0.2`.

**4.** Observe the tcpdump output. ICMP packets appear on `analyzer`.

**5.** Cleanup: `ovs-vsctl clear Bridge br-mirror mirrors && ovs-vsctl del-br br-mirror`.

## 9.7.5 Anatomy Template A, `ovs-vsctl list Mirror`

**Context:** verify that the Mirror is active and read statistics to confirm that traffic is actually being mirrored, not just that the configuration exists.

**Command and output** *(reproduced from OVS 2.17.9, mirror vm1 to analyzer):*

```shell
$ sudo ovs-vsctl list Mirror vm1mirror

_uuid               : 7e3c9f82-1d5e-4b7a-8e9f-abcdef012345
external_ids        : {tenant="T1", capture_reason="incident-response"}
name                : "vm1mirror"
output_port         : [<analyzer_port_uuid>]
output_vlan         : []
select_all          : false
select_dst_port     : [<vm1_port_uuid>]
select_src_port     : [<vm1_port_uuid>]
select_vlan         : []
snaplen             : []
statistics          : {tx_bytes=45678, tx_packets=321}
```

**9-attribute anatomy:**

| Field | Sample value | Meaning | Red flag if |
|-------|--------------|---------|-------------|
| `name` | `"vm1mirror"` | Human-readable mirror name | Unset means hard to audit |
| `output_port` | `[<uuid>]` | Local SPAN destination | Both `output_port` and `output_vlan` set means a configuration error |
| `output_vlan` | `100` or `[]` | RSPAN VLAN tag | Cannot trunk if the peer does not support the VLAN |
| `select_all` | `false` or `true` | Mirror all bridge traffic | `true` plus high pps means output port flood |
| `select_src_port` | `[<uuid>]` | Mirror packets that ingress on this port | Empty plus `select_dst_port` empty plus `select_all` false means no traffic mirrored |
| `select_dst_port` | `[<uuid>]` | Mirror packets that egress on this port | Combine with `src_port` for a bidirectional mirror |
| `select_vlan` | `[100, 200]` | Mirror VLAN-matched traffic | Overlap with `select_all` is ambiguous |
| `snaplen` | `[]`, `64`, `128` | Truncate packet to N bytes (OVS 2.10+) | Unset means full packet, more storage cost |
| `statistics:tx_packets` | `321` | Mirrored count (health monitor) | 0 after 1 minute means the mirror did not trigger (configuration issue) |

**Failure scenarios:**

- **`tx_packets=0` after 1 minute even though traffic is flowing.** `select_src_port` or `select_dst_port` points to the wrong port UUID. Verify with `ovs-vsctl list Port | grep <expected-name>` to get the correct UUID.
- **Both `output_port` and `output_vlan` are set.** Invalid configuration. A mirror selects only one output. Fix: `ovs-vsctl set Mirror <name> output_vlan=[]`.
- **Analyzer port flood.** `select_all=true` on a high-rate bridge overwhelms the output port. Fix: narrow the selector and increase `snaplen=64`.
- **RSPAN VLAN 100 does not reach the remote analyzer.** The trunk peer switch does not carry VLAN 100. Fix: ensure `switchport trunk allowed vlan add 100` on the upstream switch.

## 9.7.6 Common misconceptions and key topic

> **Common misconception 1:** "Mirror copy doubles the fabric bandwidth consumption."
>
> **Reality:** local SPAN (`output_port`) does NOT consume fabric bandwidth. The packet copy stays in hypervisor kernel memory, the output goes through an internal port, and tcpdump reads from the same kernel buffer. Zero wire cost. RSPAN (`output_vlan`) does consume bandwidth because the packet has to traverse the uplink port out to the trunk. The distinction is important when planning capacity.

> **Common misconception 2:** "Mirror-then-capture with `select_all` is safe for production."
>
> **Reality:** `select_all=true` on a bridge with many ports causes output-port overload. Example: a bridge with 20 VMs at 500 Mbps each is 10 Gbps aggregate. The output internal port has no built-in rate limit. The tcpdump write-to-disk bottleneck causes the kernel to drop mirrored packets, leading to incomplete capture and debugging in the dark. Best practice: narrow the selector first (a specific VM), then reduce `snaplen` to 96 to 128 bytes for header-only capture.

> **Key topic:** mirroring is the foundation for several observability use cases: (1) ad-hoc debugging (tcpdump on 1 VM for troubleshooting); (2) IDS feed (Suricata ingesting an RSPAN VLAN); (3) compliance audit (capture specific tenant traffic for 90-day retention); (4) offline flow analysis (split pcap by 5-tuple, correlate with application logs). See Part 20.1 §20.1.13 `port_security` forensics for the compliance use case. See Part 9.8 for the flow monitoring alternative (sFlow and IPFIX, sampled instead of full capture, scales better for always-on monitoring).

## 9.7.7 Common failure modes

**Failure 1, mirror configuration exists but `statistics=0`.**
Signal: `tx_packets=0` after 1 minute.
Cause: wrong port UUID in `select_src_port` or `select_dst_port`.
Fix: re-create the mirror with the atomic idiom `--id=@p1 get Port <correct-name>`.

**Failure 2, output analyzer port is flooded.**
Signal: `iftop -i analyzer` shows traffic above the output port capacity. tcpdump drop counter is high.
Cause: `select_all=true` on a high-traffic bridge.
Fix: narrow the selector and use `snaplen=96` for header-only.

**Failure 3, the Mirror table grows without cleanup.**
Signal: `ovs-vsctl list Mirror | wc -l` shows hundreds of Mirror rows.
Cause: an orchestrator creates Mirrors ad-hoc but does not clean them up. OVS auto-garbage-collects when unreferenced, but `Bridge.mirrors` references keep them alive.
Fix: audit the orchestrator cleanup logic. Manual cleanup: `ovs-vsctl clear Bridge <br> mirrors`.

**Failure 4, RSPAN VLAN does not reach the remote.**
Signal: local mirror is OK (statistics increasing) but the remote analyzer sees no packets.
Cause: the trunk peer switch does not carry the VLAN, or a VACL blocks the VLAN.
Fix: `show vlan brief` on the peer, add the VLAN to the trunk list.

## 9.7.8 Performance and retention planning

**Capture capacity planning:**

| Scenario | Traffic rate | Snaplen | Storage per hour |
|----------|--------------|---------|------------------|
| 1 VM, normal workload | 100 Mbps | full | 45 GB |
| 1 VM, normal workload | 100 Mbps | 96 bytes | 4.3 GB |
| 10 VMs, 1 Gbps each | 10 Gbps | 96 bytes | 432 GB |
| Bridge-wide (50 VMs) | 25 Gbps | 128 bytes | 1.3 TB |

**Retention:** a quarterly audit (90 days) for 1 VM header-only is about 9 TB. Scaling is linear. Plan dedicated NVMe storage, or stream through `tcpdump -w -` and pipe to Kafka or a remote log aggregator.

**Loss-less capture requirement:** production incident response needs `snaplen=full` for a 5 to 10 minute window. Monitor the tcpdump `dropped` counter; if it grows, the buffer size is insufficient. Tune: `tcpdump -B 10240` raises the buffer to 10 MB.

## Key points to remember

First, the OVS Mirror table provides native SPAN and RSPAN. No external tool is needed.

Second, the atomic transaction `-- --id=@...` is the standard way to create a Mirror. It ensures `Bridge.mirrors` references the correct Mirror row in a single step.

Third, a Mirror is auto-garbage-collected when unreferenced. Use `clear Bridge br0 mirrors` to disable it.

Fourth, tcpdump on the internal output port plus Wireshark or Suricata is the foundation for network visibility.

Fifth, the Anatomy Template A `list Mirror` 9-attribute view can be read in 10 seconds. `tx_packets=0` is the number-one red flag (the mirror is not triggering).

Sixth, local SPAN has zero wire cost; RSPAN consumes uplink bandwidth. A snaplen of 96 to 128 bytes for header-only capture reduces storage by about 10 times.

Seventh, mirror-then-capture is not free. Capacity planning is needed: 1 VM header-only is 4.3 GB per hour, full capture is 45 GB per hour. Scaling is linear with bandwidth.

## 9.7.9 Backfill v3.6, the `ovs-tcpdump` shortcut wrapper

> **Context.** The Phase 3 v3.6-ContentDepth audit found that `ovs-tcpdump` was only listed in the master keyword index (0.3) and had no substantive Anatomy treatment in the Part that teaches packet capture. This section adds 5-axis treatment.

### 9.7.9.1 The problem `ovs-tcpdump` solves

Normally, to capture traffic from an OVS internal port (tap, patch, VXLAN tunnel, VM TAP), the operator has to:

1. Create a Mirror table entry with `select_src_port` and `output_port` pointing to a "snooper" port.
2. Attach `tcpdump` to the snooper port.
3. Clean up the Mirror after finishing.

This 3-step procedure (see §9.7.2 and §9.7.3) is worth it for long-running debugging but is overkill for "quickly check for 30 seconds whether ARP is going through". `ovs-tcpdump` is a Python wrapper (`utilities/ovs-tcpdump.in` upstream) that automates the entire procedure: when invoked it creates a fake netdev port, sets up the Mirror automatically, calls `tcpdump` with the operator-specified BPF filter, and cleans up as soon as it receives a termination signal.

### 9.7.9.2 Anatomy of `ovs-tcpdump`

**Bucket:** capture utility (ad-hoc).
**Synopsis:** `ovs-tcpdump --mirror-to=<snoop> -i <ovs-port> [tcpdump options]`.
**Effect:** creates a Mirror automatically, attaches tcpdump output to the terminal or to a file, cleans up on `SIGINT` or `SIGTERM`. Equivalent to the 3-step manual procedure but atomic.
**Prerequisites:** Python 3 and `tcpdump` installed on the same host. The user needs `ovs-vsctl` privileges (root or setuid). The bridge must use the kernel datapath (the DPDK datapath does not expose a tap socket for tcpdump in this way).
**OVS version introduced:** OVS 2.10+ utility (Python wrapper).
**Side effect:** the Mirror table has one temporary entry while the capture runs; the entry is auto-removed after the process exits. If the process is hit with `kill -9` and does not shut down cleanly, the Mirror entry remains stale in the DB and must be cleared manually with `ovs-vsctl --id=@m clear Bridge ... mirrors @m`.

**Example:**

```
[host]# ovs-tcpdump -i veth-vm1 -nn 'arp or icmp'
tcpdump: listening on miveth-vm1, link-type EN10MB (Ethernet), snapshot length 262144 bytes
14:33:01.001234 ARP, Request who-has 10.0.0.5 tell 10.0.0.1, length 28
14:33:01.001456 ARP, Reply 10.0.0.5 is-at 52:54:00:aa:bb:cc, length 28
14:33:02.123456 IP 10.0.0.1 > 10.0.0.5: ICMP echo request, id 1, seq 1, length 64
^C
2 packets captured, ARP entries cleaned up.
```

**Operational interpretation.** The `-i` flag here takes the OVS port name (the same way `tcpdump -i eth0` takes a kernel netdev), not a kernel netdev name. The wrapper creates a virtual snoop port called `mi<port>` (prefix `mi`) and a Mirror entry pointing the original port to the snoop. When the user presses Ctrl-C, the wrapper deletes the Mirror entry, deletes the snoop port, and rolls back the `ovs-vsctl` change. Compared with `tcpdump -i any`: `tcpdump -i any` does not see east-west traffic between two VMs on the same OVS bridge (the kernel never sees the packets going through the datapath fast path); `ovs-tcpdump` installs a mirror in OVS and so it sees the traffic. Anti-pattern: using `ovs-tcpdump` for a long-running capture (hours to days) because the persistent Mirror entry has a lock cost in the OVS classifier. Preferred: a persistent Mirror entry with a separate `tcpdump` process. Cross-link: §9.7.4 capture from mirror output, §9.4 `ovs-vsctl` Mirror atomic mutate, §9.28 `ovs-pcap` for reading back captured files.

## 9.7.10 The `-i IFACE` flag, choosing which interface to inspect

**Concept.** The `-i` flag (synonym `--interface`) is required for `ovs-tcpdump`. It takes the **name of an OVS port** in `Bridge.ports` (for example `vm1`, `veth-br1`, `tap0`, `vxlan0`). The name must exist in the `name` column of the `Interface` table. Verify quickly with `ovs-vsctl list-ports <bridge>`. The wrapper auto-detects which bridge contains the interface through `ovs-vsctl iface-to-br <iface>`, creates the Mirror on that bridge, and the operator does not have to specify the bridge.

**Key difference from `tcpdump -i`.** `tcpdump -i eth0` reads directly from the kernel netdev `eth0` through an AF_PACKET socket. `ovs-tcpdump -i vm1` sets up a Mirror in OVSDB and attaches `tcpdump` to a helper snoop port. The consequence: with two VMs on the same OVS bridge, `tcpdump -i any` does not see east-west traffic because the packets pass through the datapath fast path and never appear on a kernel netdev; `ovs-tcpdump -i vm1` does see all of them. This is the motivation for the wrapper.

**Syntax and a real example:**

```
[host]# ovs-vsctl list-ports br-int
veth-br1
veth-br2
tap-vm1

[host]# ovs-tcpdump -i tap-vm1 -nn -c 5
tcpdump: listening on mitap-vm1, link-type EN10MB (Ethernet), snapshot length 262144 bytes
14:45:01.001 IP 10.0.0.5 > 10.0.0.6: ICMP echo request, id 1, seq 1, length 64
14:45:01.002 IP 10.0.0.6 > 10.0.0.5: ICMP echo reply,   id 1, seq 1, length 64
... (3 more lines)
5 packets captured
```

**Common mistakes.** (a) Passing a kernel netdev name that is not in OVS, for example `-i eth0` when `eth0` has not been added as a port. The wrapper returns `Cannot find OVS bridge for eth0`; fix with `ovs-vsctl add-port br0 eth0` first. (b) Passing an interface name longer than 13 characters: Linux netdev names are limited to 15 characters and the prefix `mi` makes a snoop name. The wrapper falls back to a random name `ovsmi%06d` instead of `mi<name>`, so do not be surprised to see an unfamiliar port in `ip link`. Cross-link: §9.4 `ovs-vsctl iface-to-br`, §9.7.2 atomic Mirror manual creation.

## 9.7.11 Filters passed in, BPF syntax and why there is no OpenFlow filter

**Correction.** `ovs-tcpdump` does **NOT** have a `-O` flag or a separate `--filter` flag for OpenFlow-style match. The wrapper is just a Python shell around the real `tcpdump`, so **every filter is a BPF expression** (Berkeley Packet Filter, `pcap-filter(7)` syntax). Every token left after argparse processes the wrapper-specific flags (`-i`, `--mirror-to`, `--span`, `--db-sock`, `--dump-cmd`) is forwarded as is to `tcpdump` through the `tcpdargs` list.

**BPF filter syntax:** pass it as the final argument; you can wrap it in single quotes so the shell does not split it:

```
[host]# ovs-tcpdump -i veth-vm1 -nn 'tcp port 443 and host 10.0.0.5'
[host]# ovs-tcpdump -i veth-vm1 -nn 'arp or icmp'
[host]# ovs-tcpdump -i veth-vm1 -e 'vlan 100 and ether dst 52:54:00:aa:bb:cc'
```

The BPF filter runs in the kernel right after the snoop port and drops non-matching packets early, reducing userspace load by about 10 times for high-rate flows.

**When do you need an "OpenFlow-style match" instead of BPF?** When the condition cannot be expressed by L2 to L4 headers (for example, "match `tun_id=0x100`" or "match `metadata=0x42`" or "match `reg14=2`"; these are OVS pipeline internal metadata that do not appear in the on-wire packet, so BPF cannot see them). In that case, **do not use `ovs-tcpdump`**. Use one of two methods: (1) create a Mirror manually with `select_src_port` or `select_dst_port` pointing to the relevant port, then run `tcpdump -i <output>` (see §9.7.2); (2) use `ovs-appctl ofproto/trace` to simulate the pipeline for a representative packet (see Part 9.25). Do not expect to find a `-O` flag or `--filter=of:...`; it does not exist in the upstream `utilities/ovs-tcpdump.in` v2.17.9.

**Cross-link:** Part 9.25 `ofproto/trace` for metadata-aware simulation, §9.7.2 Mirror selectors for more precise port-logic filtering than BPF.

## 9.7.12 The `--mirror-to=<NAME>` flag, naming the snoop port

**Concept.** By default the wrapper creates a snoop port named `mi<iface>` (for example, `-i veth-vm1` produces port `miveth-vm1`). If the original name is longer than 13 characters (above the 15-character netdev limit when adding the `mi` prefix), the wrapper falls back to a random name in the form `ovsmi%06d` (six digits). The `--mirror-to=<NAME>` flag lets the operator **set a fixed name** instead of letting the wrapper auto-generate one. Useful when: (a) scripting many parallel captures and needing to know the name in advance; (b) wanting `ip link show <name>` or `ethtool -S <name>` to read snoop counters after the capture; (c) running multiple captures concurrently on the same source interface, avoiding name collisions.

**Syntax and example:**

```
[host]# ovs-tcpdump -i tap-vm1 --mirror-to=snoop-prod --db-sock=unix:/var/run/openvswitch/db.sock 'icmp'
tcpdump: listening on snoop-prod, link-type EN10MB (Ethernet), ...

[host]# ip link show snoop-prod
42: snoop-prod: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 ...
```

**Side effect to be aware of.** When the capture ends through Ctrl-C (signal `SIGINT`), the wrapper triggers a fatal-signal hook that calls `cleanup_mirror()`, which removes the Mirror row and deletes the snoop port. But if the process is hit with `kill -9` (`SIGKILL`), the hook does not run and the `snoop-prod` port and Mirror row remain stale. Manual recovery: `ovs-vsctl --if-exists del-port snoop-prod && ovs-vsctl clear Bridge <br> mirrors`. Periodic audit with `ovs-vsctl list Mirror | grep -c '_uuid'`: if the count of long-lived mirrors exceeds the number of running capture sessions, that is a sign of a leak.

Cross-link: §9.7.3 general Mirror cleanup, §9.7.7 Failure 3 (mirror table growth).

## 9.7.13 The `--span` flag, snooping every port on a bridge

**Concept.** By default `ovs-tcpdump -i <iface>` creates a Mirror with `select_src_port` and `select_dst_port` pointing to one port. The `--span` flag changes this behaviour: instead of a single-port mirror, the wrapper creates a Mirror with **`select_all=true`** on the bridge that contains the interface. All ingress and egress traffic of every port on the bridge is copied to the snoop port. This is equivalent to a bridge-wide SPAN session on a physical switch.

**Syntax:**

```
[host]# ovs-tcpdump -i veth-vm1 --span -nn -c 20
# Even though we specify -i veth-vm1, --span overrides and mirrors all of br-int.
# veth-vm1 is needed only so that the wrapper knows which bridge to use, not to limit the source.
```

**When to use it.** (a) Learning labs: you want to see the full conversation on a small bridge with 3 to 5 ports. (b) The early stage of incident response, when you do not yet know which VM is involved; capture the whole bridge and filter offline through Wireshark. (c) A compliance audit that requires full capture of one tenant (when that tenant has the bridge to itself).

**Capacity warning.** `--span` on a high-traffic production bridge causes **output-port flood** in the same way as a manual `select_all=true` (see §9.7.6 common misconception 2). Example: 20 VMs at 500 Mbps each is 10 Gbps copied to the snoop, exceeding the userspace `tcpdump` write-to-disk capacity, so the kernel drops packets and the capture is incomplete. Mitigation: combine with a narrow BPF filter, raise the buffer with `tcpdump -B 10240`, and consider a small snaplen (by default `ovs-tcpdump` does not truncate; use `-s 96` to limit to 96-byte headers). For captures longer than 30 seconds, consider a persistent Mirror entry with a dedicated `tcpdump` process instead of `ovs-tcpdump`.

Cross-link: §9.7.6 misconception about `select_all=true`, §9.7.8 capacity planning.

## 9.7.14 pcap output format and libpcap compatibility

**Concept.** `ovs-tcpdump` does not define its own format. It forwards format responsibility to the real `tcpdump` underneath. When the operator adds the `-w <file>` flag (passed through to `tcpdump`), the output file is **standard libpcap pcap**: a 24-byte header (magic number `0xa1b2c3d4` little-endian or `0xd4c3b2a1` big-endian, version major and minor, thiszone, sigfigs, snaplen, network link-type), followed by a sequence of records `(ts_sec, ts_usec, incl_len, orig_len)` of 16 bytes each, followed by the raw frame.

**Compatibility.**

- The file can be read directly by `wireshark`, `tshark`, `tcpdump -r`, `tcpreplay`, Suricata, Zeek (Bro), and Moloch or Arkime.
- It can be read by `ovs-pcap` (see Part 9.28) to convert into a hex single-line that can be fed to `ovs-appctl ofproto/trace`.
- The default link-type is `LINKTYPE_ETHERNET` (`DLT_EN10MB`, value 1), which matches every standard L2 tool.

**Example workflow:**

```
[host]# ovs-tcpdump -i tap-vm1 -w /tmp/capture.pcap -nn 'tcp port 443'
^C
12 packets captured

[host]# file /tmp/capture.pcap
/tmp/capture.pcap: pcap capture file, microsecond ts (little-endian) - version 2.4
                   (Ethernet, capture length 262144)

[host]# wireshark /tmp/capture.pcap   # open the GUI

[host]# ovs-pcap /tmp/capture.pcap | head -1
ffffffffffff525400aabbcc0800450000... (hex single-line)

[host]# ovs-appctl ofproto/trace br-int "$(ovs-pcap /tmp/capture.pcap | head -1)"
Flow: ... dl_type=0x0800,nw_src=10.0.0.5, ...
```

**Version note.** The pcap header version `2.4` is the libpcap standard since 2008 and has not changed between OVS 2.x and 3.x. Some newer tools use pcapng (`.pcapng`, magic `0x0a0d0d0a`); `ovs-tcpdump` only emits classic pcap, not pcapng, because the underlying `tcpdump` pass-through also emits classic by default. If you need pcapng for richer metadata (interface description block, comment block), use `tshark -F pcapng` to read the file and re-encode, or skip the wrapper and use a persistent Mirror with `tshark -i <snoop> -w out.pcapng`.

Cross-link: §9.28 `ovs-pcap` uses the same format, §9.7.4 capture mirror output directly, libpcap project file format specification.

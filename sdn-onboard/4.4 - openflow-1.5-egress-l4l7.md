## Context: the final revision

Part 4.3 analysed OF 1.4 (2013-10-14) with its ambition for transactional semantics through bundles and table management through eviction, but with only partial real-world adoption because vendors did not see ROI. OF 1.5 came right after, adding egress tables, TCP flags matching, and packet-type awareness. But the context had changed.

Around 2014, the SDN industry began to branch. OpenFlow as a protocol was gradually viewed as "one option" rather than "the only option". Hypervisor overlay (NVP and NSX) was taking OpenFlow's place in DC virtualisation (see Part 5.1). SDN through APIs (NETCONF/YANG, see Part 5.0) was taking its place in WAN management. And most importantly, Bosshart et al. had published the P4 paper in CCR 2014 with a completely different premise: instead of a fixed protocol to reach the data plane, allow the programmer to **define** the data-plane behavior in a language. P4 was an existential threat to OpenFlow, not a competitor.

In that context, OF 1.5 was released on **2014-12-19** and the 1.5.1 errata on **2015-03-26**. After 1.5.1, ONF shifted focus to the ONOS and CORD projects (see Part 3.2.6) and never revised the OpenFlow spec again. 1.5.1 is the final revision of the OpenFlow Switch Specification branch.

## 4.4.1 OF 1.5 release timeline: 1.5.0 (2014-12-19) to 1.5.1 (2015-03-26), the final

### Release information

OpenFlow Switch Specification Version 1.5.0 released on **2014-12-19** with wire protocol version **0x06**. The 283-page document is up from the 206 pages of 1.4. The 1.5.1 errata released on **2015-03-26** (the same day as the 1.3.5 errata), with the same wire protocol 0x06.

PDF archive:

```
https://opennetworking.org/wp-content/uploads/2014/10/openflow-switch-v1.5.0.pdf
https://opennetworking.org/wp-content/uploads/2014/10/openflow-switch-v1.5.1.pdf
```

ONF's press release in 2014-12 describes:

> *"OpenFlow 1.5 addresses new use cases in service provider and enterprise networks, including packet-type-aware matching, egress table processing, and extensions for carrier-grade deployments."*

### Meaning of "final"

1.5.1 is not formally "end of life"; ONF did not announce its retirement. But in reality after 2015, the Technical Steering Committee did not approve a new spec revision for the OpenFlow Switch Specification. Working groups focused on:

- **Optical Transport WG**: continued work on OpenFlow extensions for optical but did not push them into the main spec.
- **OF-Config**: maintained a separate spec (OF-Config 1.2 released in 2014-04 is the final version).
- **Northbound Interfaces**: shifted focus to intent-based APIs, gNMI, and the OpenConfig alliance.

ON.Lab merged with ONF in 2018-10, which changed the organisation, inheriting ONOS and CORD as flagship projects instead of maintaining the OpenFlow spec.

When a 2026 user looks back and asks "what is the latest OpenFlow version?", the correct answer is **1.5.1 (2015-03-26)**; there is absolutely no 1.6 officially published. There were rumours of an internal 1.6, but those were working drafts of a member-only WG that was never published.

## 4.4.2 Egress tables: pipeline after ingress chooses output port

### Problem: ingress-only pipeline cannot express per-output-port policy

In OF 1.0 to 1.4, all flow processing happens in the ingress pipeline. The packet enters the switch, matches table 0, runs instructions (GOTO_TABLE, WRITE_ACTIONS, and so on), and finally the action set executes and the packet leaves the chosen port. Once the output port is decided, there is no further opportunity to modify the packet.

The problem with ingress-only is that **per-output-port policy** is hard to express. For example, a DC topology has uplink ports to multiple upstream providers, each with different QoS marking requirements. Provider A requires DSCP=EF for voice, provider B requires DSCP=AF41. In ingress, the controller must:

1. Match tenant traffic and choose output port based on destination.
2. Apply QoS action (SET_DSCP) based on the chosen output port.

Not feasible in the ingress pipeline because of instruction order: SET_DSCP is in APPLY_ACTIONS or WRITE_ACTIONS, and you need to know the output port BEFORE deciding the DSCP value. But the output port is a result of the pipeline, not an input.

Workaround in 1.4: the controller installs separate flow entries for each provider, each entry hard-coding output port plus DSCP. The flow table explodes with the cross-product tenant times provider.

### OF 1.5 solution: egress pipeline (§5.1)

Spec 1.5 allows the switch to have **egress tables** parallel to ingress tables. After the ingress pipeline chooses the output port, the packet is forwarded into the egress pipeline of that port (first_egress_table_id and above). The egress pipeline can apply additional per-port actions, like SET_DSCP appropriate to the provider.

Pipeline structure:

```
Packet in -> Ingress Pipeline -> Action Set -> Output Port Selected
                                                     |
                                                     v
                                            Egress Pipeline (per-port)
                                                     |
                                                     v
                                            Final output
```

Spec 1.5 §5.1 defines:

- The switch advertises first_egress_table_id in OFPTR_FEATURES_REPLY.
- Tables with ID less than first_egress_table_id are ingress; greater than or equal are egress.
- Each output port may activate the egress pipeline (if configured), with its own match and instructions.

### Use case: carrier Ethernet with per-provider QoS

Controller configuration:

```
Ingress Table 0 (classify tenant):
  match: metadata=tenant_A -> action: WRITE_METADATA=tenant_A, GOTO Table 1
  match: metadata=tenant_B -> action: WRITE_METADATA=tenant_B, GOTO Table 1

Ingress Table 1 (route plus select output port):
  match: tenant_A, dst_ip=/24 -> action: output port 10 (uplink provider A)
  match: tenant_B, dst_ip=/24 -> action: output port 11 (uplink provider B)

Egress Table 100 (port 10 QoS, provider A):
  match: voice_class -> action: SET_DSCP=46 (EF)
  match: data_class -> action: SET_DSCP=34 (AF41)

Egress Table 100 (port 11 QoS, provider B):
  match: voice_class -> action: SET_DSCP=46 (EF)
  match: data_class -> action: SET_DSCP=26 (AF31)
```

This solves the cross-product explosion: flow count drops from O(tenants times providers) to O(tenants plus providers).

### Common misunderstandings about egress tables

**Misunderstanding:** "Egress tables run in parallel with ingress, so a packet can skip ingress and enter egress directly."

**Reality:** Egress tables run **after** ingress tables, not in parallel. The packet always passes through the ingress pipeline first to choose the output port; the egress pipeline only kicks in after the port is chosen. There is no way for a packet to bypass ingress and enter egress directly. The correct mental model: egress tables are "post-processing" after routing decides, not an alternative pipeline.

## 4.4.3 TCP flag matching

### Structure

OF 1.5 adds **OXM_OF_TCP_FLAGS** (§A.2.3.7), a 12-bit match with a 12-bit mask. TCP flags include:

| Bit | Name | Description |
|-----|------|-------------|
| 0 | FIN | Finish, close connection |
| 1 | SYN | Synchronize, start connection |
| 2 | RST | Reset, abort connection |
| 3 | PSH | Push, deliver data immediately |
| 4 | ACK | Acknowledgment |
| 5 | URG | Urgent pointer valid |
| 6 | ECE | ECN-Echo (RFC 3168) |
| 7 | CWR | Congestion Window Reduced |
| 8 | NS | Nonce Sum (RFC 3540, experimental) |
| 9-11 | Reserved | (future use) |

Note: OF 1.2 already had OXM_OF_TCP_FLAGS (see Part 4.1.3); OF 1.5 formalises it with two additional reserved bits.

### Use case in SDN firewall

**SYN flood detection:** match packets with only SYN set (no ACK). Typical pattern:

```
match: eth_type=0x0800, ip_proto=6, tcp_flags=0x002/0x017
# Mask 0x017 = SYN + FIN + RST + ACK + URG bits checked
# Value 0x002 = only SYN set among these 5 bits
  -> action: meter:rate_limit, forward
```

The first packet of a TCP connection (SYN alone) is counted per source; if a threshold is exceeded, drop.

**FIN scan detection:** match FIN with no ACK:

```
match: eth_type=0x0800, ip_proto=6, tcp_flags=0x001/0x011
# Mask 0x011 = FIN + ACK
# Value 0x001 = only FIN (no ACK)
  -> action: drop, log
```

FIN scan is a port-scanning technique that bypasses stateless firewalls (send FIN to a closed port, expect RST response; an open port is silent).

**Null scan detection:** match all flags=0:

```
match: eth_type=0x0800, ip_proto=6, tcp_flags=0x000/0x03F
# Mask covers FIN, SYN, RST, PSH, ACK, URG
# Value 0 = none set
  -> action: drop, log
```

### Limitations

TCP flags matching is only at the **L4 header bit** level. Cannot match:
- TCP options (MSS, window scale, SACK permitted, timestamp).
- Specific TCP sequence or ack numbers.
- Payload content.

To deeply inspect TCP payload, you need a stateful firewall (conntrack) or an L7 proxy (HAProxy, nginx). OF 1.5 TCP flags is a "first line of defense" but does not replace WAF or IDS.

## 4.4.4 Packet-type aware field

### Motivation: non-Ethernet frames

OF 1.0 to 1.4 assume the packet always starts with an **Ethernet header**. The match logic relies on the `dl_type` field (Ethernet EtherType). But in some deployments, packets are not Ethernet:

- **L3-only tunnel**: the tunnel header has IP or MPLS label first, with no outer Ethernet header.
- **ARP as L2 payload**: an ARP packet is native L2 and does not encapsulate Ethernet (although delivered inside an Ethernet frame).
- **Inner packet in VXLAN or Geneve**: when the switch decaps the tunnel, the inner packet may be MPLS or ARP, and the specific type must be matched.

OF 1.0 to 1.4 cannot represent these cases. Match logic implicitly assumes Ethernet, and non-Ethernet packets fall into the table-miss fallback.

### OF 1.5 solution: OXM_OF_PACKET_TYPE

Spec 1.5 §7.2.3.6 defines **OXM_OF_PACKET_TYPE** (a 32-bit OXM field): explicitly encoding the namespace plus type of the packet header being matched:

| PACKET_TYPE | Namespace plus Type | Description |
|-------------|---------------------|-------------|
| 0x0000_6558 | Ethernet (0x0000) plus TRANS_ETH_BRIDGE | Bridged Ethernet frame (default 1.0 to 1.4) |
| 0x0000_0800 | Ethernet plus IPv4 | Ethernet with IPv4 payload |
| 0x0000_0806 | Ethernet plus ARP | Ethernet with ARP payload |
| 0x0000_8847 | Ethernet plus MPLS unicast | Ethernet with MPLS payload |
| 0x0001_nnnn | Non-Ethernet namespace | L3-only packet types |

Namespace `0x0000` is default Ethernet. Namespace `0x0001` is "non-Ethernet packet" with type as the next header.

Controller can match:

```
match: packet_type=0x0001_0800, ip_dst=10.0.0.0/24 -> action: output port N
# Match L3-only IPv4 packet (no outer Ethernet header)
```

### Use case: VXLAN decapsulation with inner MPLS

Carrier deployment: VXLAN tunnel carries inner MPLS frame. The switch decaps VXLAN; the inner packet is MPLS. Flow:

```
# Decap outer VXLAN (ingress table)
match: udp_dst=4789 -> action: POP_VXLAN, GOTO_TABLE 10

# Match inner MPLS (table 10)
match: packet_type=0x0000_8847, mpls_label=100 -> action: forward via MPLS label 200
```

Without OF 1.5 packet_type, matching inner MPLS after decap is implicit and behavior is ambiguous. With packet_type, it is explicit.

### Adoption reality

OF 1.5 packet_type was designed for advanced use cases (VXLAN inner match, L3-only pipeline), but in reality from 2014 to 2020:

- Carrier Ethernet still uses PBB (1.3) for 802.1ah and does not need packet_type.
- DC overlay uses OVS plus OVN with conntrack zones and custom tunnel keys, not relying on OpenFlow packet_type.
- Hardware vendors have not committed ASIC for packet_type matching (a complex parser requirement).

Result: packet_type on paper is very powerful, but deployment is almost non-existent.

## 4.4.5 Current state in 2026: vendor adoption near-zero, OVS 1.5 partial

### OVS 2.10+ implements partial

Open vSwitch 2.10 (2018-06) added partial 1.5 support:

**Implemented**:
- Wire protocol 0x06 handshake.
- Basic OF 1.5 messages (FEATURES_REQUEST and REPLY compatible).
- TCP flags matching (already present in 1.2 through OVS extension, formalised in 1.5).

**Stubbed** (advertised but not full production-ready):
- Egress tables: OVS code has an egress_table_id stub, but actual egress processing is minimal. The path is not fully tested.
- Packet-type aware: the OXM field is parseable but action-execution support is limited.

**Not implemented**:
- Full 1.5 bundle extensions (mixed with the 1.4 bundle support).
- Some minor OXM extensions specific to 1.5.

OVS maintainers (Ben Pfaff, Justin Pettit) have confirmed in 2018 commit messages that 1.5 support is "best-effort completeness", not production-tuned.

### Hardware vendors skip entirely

- **HP and Aruba**: committed to 1.3, with 1.4 and 1.5 absent from the 2015 to 2020 roadmap.
- **Pica8**: PicOS finally committed to 1.3; when Pica8 was acquired by HCL Technologies in 2018, OpenFlow evolution was frozen.
- **NEC IP8800**: production end in 2016, never implemented 1.5.
- **Centec V350 and V358**: committed to 1.3 standard.
- **Mellanox and NVIDIA Spectrum**: focus on OVS switchdev hardware offload with tc flower, not direct OpenFlow 1.5 ASIC support.
- **Broadcom (market-leading chipset)**: Trident and Tomahawk series silicon did not commit 1.4 or 1.5; no microcode public release.

### Tofino and P4 take the programmable space

From 2016, **Barefoot Networks Tofino ASIC** with the **P4 programming language** (CCR 2014, Bosshart et al.) began shipping in production. Instead of a fixed OpenFlow protocol, P4 allows the programmer to write parser plus match-action tables plus deparser per need.

Major adoption: Alibaba, Tencent, and Facebook used Tofino plus P4 for custom SDN pipelines from 2017 to 2019. Intel acquired Barefoot in 2019-06. In 2023-01, Intel announced end-of-life for the Tofino line, with most engineering staff transitioning to AI/ML silicon. The P4 language remains alive through the p4.org open-source community, with alternative vendors (AMD/Pensando, NVIDIA Spectrum-4 with the DOCA pipeline).

Lesson: OpenFlow 1.5 did not lose for "lack of features"; it lost because the industry had moved on to a new model (P4) before 1.5 had time to be implemented. Timing is everything in protocol evolution.

P4 plus Tofino detail will be analysed in Part 6.0. Part 4.6 (next) summarises the OpenFlow branch post-mortem lessons.

> **Key point.** OpenFlow 1.5 is a "spec for spec's sake": ONF released it to maintain protocol-evolution momentum in a context where the ecosystem had cooled. Lesson for any protocol: feature richness does not guarantee adoption. Hardware vendors only commit ASIC when the market demand is proven, not because the spec adds new features. And when an alternative model (P4) appears with a strong-enough value proposition (programmability), the stable protocol (1.3) survives while the stalled protocol evolution (1.4, 1.5) dies.

## Guided Exercise: egress table simulation with OVS plus OF 1.5 features (audit P4.B4.1 hands-on)

**Goal:** explore the egress-pipeline concept with OVS (OVS partially supports OF 1.5). Verify per-port egress policy.

**Context:** OVS 2.17 supports OF 1.5 partially (egress table through experimenter extension). The exercise uses the OF 1.5 wire format with OVS emulating egress behavior through a multi-table pattern.

**Steps:**

1. Create an OF 1.5 bridge:
   ```bash
   $ sudo ovs-vsctl add-br br-egress -- set bridge br-egress protocols=OpenFlow15
   ```
2. Simulate egress-table behavior through a multi-table pattern (ingress, decide output, egress-like table):
   ```bash
   # Ingress: classify traffic plus decide output port
   $ sudo ovs-ofctl -O OpenFlow15 add-flow br-egress \
       "table=0,priority=100,ip,nw_dst=10.1.0.0/16,actions=set_field:1->reg1,goto_table:1"
   $ sudo ovs-ofctl -O OpenFlow15 add-flow br-egress \
       "table=0,priority=100,ip,nw_dst=10.2.0.0/16,actions=set_field:2->reg1,goto_table:1"

   # "Egress" table: per-output-port QoS (DSCP remark)
   $ sudo ovs-ofctl -O OpenFlow15 add-flow br-egress \
       "table=1,priority=100,reg1=1,actions=set_field:46->ip_dscp,output:1"
   $ sudo ovs-ofctl -O OpenFlow15 add-flow br-egress \
       "table=1,priority=100,reg1=2,actions=set_field:34->ip_dscp,output:2"
   ```
3. Verify per-output-port DSCP marking:
   ```bash
   $ sudo ovs-appctl ofproto/trace br-egress \
       "in_port=1,ip,nw_dst=10.1.0.5" | grep DSCP
   # Expected: set DSCP=46 (EF), output port 1
   ```
4. Compare with native egress table (OF 1.5+ real implementation):
   ```bash
   # Native egress table (if the switch supports OF 1.5 native), spec 1.5 §5.1
   # OVS currently emulates through a multi-table ingress pattern (as above)
   ```

**POE:**
- **Predict:** Does OVS support OF 1.5 egress tables natively?
- **Observe:** OVS 2.10+ has partial support (check `ovs-vsctl get Bridge br-egress datapath_id`). But most hardware silicon in 2026 still has no egress table. The multi-table ingress pattern is the portable workaround.
- **Explain:** OF 1.5 egress tables are an ambitious spec, but silicon did not commit because TCAM at the egress position requires a chip redesign. The ecosystem stayed at ingress multi-table.

**Cleanup:** `sudo ovs-vsctl del-br br-egress`

## Bối cảnh: ONF's first spec và shift sang extensible match

Part 3.2 đã phân tích ONF formation 21/03/2011 và IP transfer OpenFlow spec từ Stanford. Part 4.0 đã analyze OF 1.1 (28/02/2011) là spec cuối do Stanford shepherd. OF 1.2 là **spec đầu tiên do ONF công bố** sau khi WG process chuyển đầy đủ vào ONF governance. Release ngày 05/12/2011, 9 tháng sau ONF formation.

Ngoài ý nghĩa organizational (ONF's first), OF 1.2 có shift architectural quan trọng: chuyển từ **12-tuple fixed match** của OF 1.0-1.1 sang **OXM TLV extensible match**. Shift này không phải cosmetic; nó là acknowledgment rằng packet header diversity trong modern networking (ARP, ICMP variants, MPLS, metadata) không fit vào hardcoded 12 fields, cần extension mechanism để future-proof protocol.

OF 1.2 cũng là version đầu tiên có **controller roles** formalized cho HA deployment. Trước 1.2, multi-controller setup là implementation-defined (không chuẩn). OF 1.2 đưa EQUAL/MASTER/SLAVE model với generation_id protocol cho atomic role transition, enable production HA.

## 4.1.1 OF 1.2 phát hành 05/12/2011: spec đầu tiên do ONF công bố

### Dòng thời gian và quy trình ONF

OpenFlow Switch Specification Version 1.2 chính thức release ngày **05/12/2011**, wire protocol version **0x03**. Document 85 trang (tăng từ 56 trang của 1.1), reflect scope mở rộng. Copyright page ghi:

> *"OpenFlow Switch Specification, Version 1.2 (Wire Protocol 0x03), December 5, 2011. Copyright © 2011 Open Networking Foundation. All rights reserved."*

Đây là spec đầu tiên ghi copyright ONF thay vì Stanford. Về process, ONF's Technical Steering Committee (TSC) review draft từ Forwarding Abstractions Working Group (FAWG) và approve release. Đây là proof-of-concept rằng ONF operate được spec process (WG → TSC → member vote → publish) chứ không chỉ rebrand của Stanford.

Archive hiện tại tại:

```
https://opennetworking.org/wp-content/uploads/2014/10/openflow-spec-v1.2.pdf
```

### Động lực: mở rộng match fields vượt 12-tuple

Trong giai đoạn 2010-2011 deployment, vendor và researcher identify gaps trong OF 1.0-1.1 match structure. Vài pain points cụ thể:

**Gap 1: ARP details beyond just Ethernet type**. OF 1.0 match `dl_type=0x0806` biết packet là ARP, nhưng không match được ARP fields (op, sender IP, target IP). Implementing ARP security rules (anti-spoofing, rate limiting ARP flood) impossible without ARP field match.

**Gap 2: ICMP type/code granular match**. OF 1.0 match `nw_proto=1` biết packet là ICMP, nhưng `tp_src/tp_dst` được reuse cho ICMP_TYPE/ICMP_CODE mà không có clear naming. Confusing và error-prone.

**Gap 3: TCP flags matching**. Firewall và IDS cần match specific TCP flags (SYN flood detection = match `tcp_flags=SYN && !ACK`). OF 1.0 không có TCP flags field, phải workaround qua DPI.

**Gap 4: Metadata cross-table**. OF 1.1 multi-table pipeline need cross-table state passing, nhưng không có standardized way. Vendor extensions (OVS register) filled gap với vendor-specific approach.

**Gap 5: Future-proof for new protocols**. Hard-coded 12-tuple means mỗi time có protocol mới (VXLAN, NVGRE, Geneve later), phải spec revision. Không scalable.

OXM TLV match addresses cả 5 gaps với một cơ chế thống nhất: variable-length TLV encoding, extensible class space.

## 4.1.2 OXM TLV format

### Cấu trúc

Spec 1.2 §7.2.3 define OXM TLV header 32-bit:

| Offset | Field | Size (bit) | Description |
|--------|-------|------------|-------------|
| 0-15 | oxm_class | 16 | Match class (0x8000=OFPXMC_OPENFLOW_BASIC, >0xFFFF=vendor) |
| 16-22 | oxm_field | 7 | Field identifier within class |
| 23 | HM (has_mask) | 1 | 1 = value+mask, 0 = value only |
| 24-31 | oxm_length | 8 | Length of payload in bytes |

Following header là payload: value (oxm_length byte) + optional mask (nếu HM=1, thêm oxm_length byte mask).

### Các lớp (class) OXM

**OFPXMC_OPENFLOW_BASIC (0x8000)**: ONF-standardized fields. Field identifiers 0-40 spec-defined.

**OFPXMC_EXPERIMENTER (0xFFFF)**: Vendor extensions. Follow by 32-bit experimenter ID (IANA-assigned Private Enterprise Number) để identify vendor, rồi field-specific payload.

**OFPXMC_NXM_0 (0x0000) và OFPXMC_NXM_1 (0x0001)**: Legacy Nicira Extensible Match classes, supported để backward compat với OVS switches running OF 1.0-era NXM. Mostly deprecated trong OF 1.2+.

### Các field phổ biến

Spec 1.2 §7.2.3.8 list Basic class fields. Selected entries:

| oxm_field | Name | Size (bit) | Description |
|-----------|------|------------|-------------|
| 0 | OXM_OF_IN_PORT | 32 | Ingress switch port |
| 1 | OXM_OF_IN_PHY_PORT | 32 | Physical port (pre-virtualization) |
| 2 | OXM_OF_METADATA | 64 | Table metadata (cross-table) |
| 3 | OXM_OF_ETH_DST | 48 | Ethernet dest MAC |
| 4 | OXM_OF_ETH_SRC | 48 | Ethernet source MAC |
| 5 | OXM_OF_ETH_TYPE | 16 | EtherType |
| 6 | OXM_OF_VLAN_VID | 12 | VLAN ID |
| 7 | OXM_OF_VLAN_PCP | 3 | VLAN Priority |
| 8 | OXM_OF_IP_DSCP | 6 | IP DSCP |
| 11 | OXM_OF_IP_PROTO | 8 | IP protocol |
| 12 | OXM_OF_IPV4_SRC | 32 | IPv4 source |
| 13 | OXM_OF_IPV4_DST | 32 | IPv4 dest |
| 14 | OXM_OF_TCP_SRC | 16 | TCP source port |
| 15 | OXM_OF_TCP_DST | 16 | TCP dest port |
| 21 | OXM_OF_ARP_OP | 16 | ARP opcode |
| 22 | OXM_OF_ARP_SPA | 32 | ARP sender protocol address |
| 23 | OXM_OF_ARP_TPA | 32 | ARP target protocol address |
| 24 | OXM_OF_ARP_SHA | 48 | ARP sender hardware address |
| 25 | OXM_OF_ARP_THA | 48 | ARP target hardware address |
| 36 | OXM_OF_MPLS_LABEL | 20 | MPLS label |
| 37 | OXM_OF_MPLS_TC | 3 | MPLS traffic class |

OF 1.2 spec define 40 basic fields. OF 1.3 add 4 more (PBB fields + metadata variants). OF 1.5 add packet_type OXM field.

### Nguồn gốc: di sản NXM

OXM format không phải invention from scratch của ONF. Nó kế thừa gần nguyên vẹn từ **NXM (Nicira Extensible Match)**, format mà Open vSwitch đã dùng từ 2008-2009 để extend OF 1.0 match beyond 12-tuple. NXM original design do Ben Pfaff và Justin Pettit (both at Nicira) phát triển, sau khi Nicira analyze pain points của 12-tuple.

Khi ONF FAWG design OF 1.2 match format, decision straightforward: adopt NXM concept với minor cleanup (rename NXM_NX_* prefix thành OXM_OF_*, tweak numbering). Backward compatibility intentional: vendor đã implement OVS + NXM không phải rewrite matching engine, chỉ adapt wire format header. Transition frictionless.

Đây là pattern chung trong OpenFlow history: proven idea từ research/open-source (NXM in OVS) được formalize thành spec official. OXM là reincarnation của NXM, giống như OF 1.0 là reincarnation của GSMP (Part 2.0), P4 là reincarnation của Active Networking (Part 2.1).

> **Key Topic:** OXM TLV extensible match là architectural decision quan trọng hơn bất kỳ feature addition nào. Từ OF 1.2 trở đi, thêm match field mới không cần spec revision breaking, chỉ cần add new oxm_field identifier. OF 1.3 add PBB, OF 1.5 add packet_type, không phá compatibility với 1.2 implementation. Extensibility này essential cho long-term spec viability.

## 4.1.3 Match field extensions: ARP, ICMP, TCP flags, metadata

### ARP detail fields

OF 1.0 match chỉ có `dl_type=0x0806` (detect ARP), không cho phép inspect ARP fields bên trong. OF 1.2 thêm 5 OXM fields cho ARP:

- **ARP_OP (OXM_OF_ARP_OP)**: ARP opcode. Value 1 = Request, 2 = Reply.
- **ARP_SPA (Sender Protocol Address)**: IP address của sender (advertised trong ARP packet).
- **ARP_TPA (Target Protocol Address)**: IP address target mà sender hỏi.
- **ARP_SHA (Sender Hardware Address)**: MAC address của sender.
- **ARP_THA (Target Hardware Address)**: MAC address target (thường 00:00:00:00:00:00 trong Request).

Use cases:

**Anti-ARP-spoofing**: match flow rule cho phép chỉ ARP Reply có correct SHA matching MAC bound to port. Nếu SHA không match, drop.

```
match: eth_type=0x0806, arp_op=2, in_port=10
where arp_sha = <MAC-port-10-authorized>
  → action: forward normal
else → action: drop
```

**ARP rate limiting**: count ARP Request per source MAC, nếu vượt threshold, drop. Prevents ARP flood attack.

**ARP-specific policy**: VLAN A cho phép ARP for 10.0.0.0/24, VLAN B cho phép ARP for 192.168.0.0/16. Match ARP_TPA + VLAN, drop nếu không match policy.

### ICMP detail fields

Tương tự ARP, OF 1.2 thêm:

- **ICMP_TYPE (OXM_OF_ICMPV4_TYPE)**: ICMP message type. Value 8 = Echo Request, 0 = Echo Reply, 3 = Destination Unreachable, 11 = Time Exceeded.
- **ICMP_CODE (OXM_OF_ICMPV4_CODE)**: sub-type within type. Ví dụ Type=3, Code=4 là "Fragmentation Needed".

Use cases:

**ICMP rate limiting**: ping flood detection. Match type=8 (Echo Request), count per source, drop nếu vượt threshold.

**Allow specific ICMP types**: firewall cho phép Echo Request/Reply (ping works) + Destination Unreachable (MTU path discovery), drop others.

**PMTUD preservation**: ensure ICMP Type=3 Code=4 (Fragmentation Needed) không bị firewall drop, critical cho Path MTU Discovery correctness.

### TCP flags field

OF 1.2 thêm **OXM_OF_TCP_FLAGS** (12 bit match, 12 bit mask). Flags field của TCP header bao gồm: URG, ACK, PSH, RST, SYN, FIN, ECE, CWR, NS, và 3 reserved bits.

Use cases:

**SYN flood detection**: match TCP packet với chỉ SYN flag set (không có ACK). Rate limit per source, drop excessive rate.

```
match: eth_type=0x0800, ip_proto=6, tcp_flags=0x002/0x017
# Mask 0x017 = SYN + ACK + FIN + RST + URG bits
# Value 0x002 = only SYN set
  → action: meter:1 (rate limit), forward
```

**Established connection tracking**: match packets có ACK set (established traffic). Allow bypass stateful firewall cho performance.

**Null scan detection**: match TCP với all flags=0 (unusual pattern, indicative of scan). Drop + log.

### Metadata field

**OXM_OF_METADATA** (64-bit): table metadata field introduced trong OF 1.1, formalize trong 1.2 OXM syntax. Dùng cross-table:

- Table 0 WRITE_METADATA=0x01 (ví dụ: tenant ID)
- Table 1 match metadata=0x01, further routing

Metadata persist qua pipeline, không reset giữa table. Enable multi-table state passing without regex tricks.

Tổng quan: OF 1.2 match extensions transform OpenFlow từ basic packet filter (12-tuple) thành general-purpose policy engine có thể express complex security rules, QoS decisions, và multi-table state machines.

## 4.1.4 Controller roles: EQUAL, MASTER, SLAVE

### Bối cảnh triển khai HA

Trong production deployment, một OpenFlow switch thường có nhiều controller configured cho HA:

- **Primary controller**: thường xử lý mọi traffic
- **Backup controller**: standby, take over khi primary down
- **Observer controller**: monitor-only, không programming

Trước OF 1.2, spec không define role semantics. Multiple controller connect đến switch đều có equal access, mỗi controller có thể send FLOW_MOD, nhận PACKET_IN, và các message khác. Không có coordination mechanism, risk conflict (2 controller cùng install contradictory flow).

OVS và một số controller vendor implement proprietary role negotiation, nhưng không interoperable. Cần standardization.

### Ba vai trò được định nghĩa

Spec 1.2 §6.3.5 formalize 3 controller roles:

**EQUAL (default)**: all controllers have equal access. Mỗi controller:
- Có thể send controller-to-switch messages (FLOW_MOD, PACKET_OUT, etc.)
- Nhận async messages (PACKET_IN, FLOW_REMOVED, etc.)
- Nhận error messages riêng (không shared)

Use case: simple development, monitoring scenarios, hoặc khi application đã có external coordination.

**MASTER**: exactly one controller per switch. Master:
- Full read-write access
- Nhận tất cả async messages
- Nhận error messages
- Switch will only accept Primary role modifications from Master

Use case: production deployment, single authoritative controller.

**SLAVE**: read-only observer. Slave:
- Cannot send FLOW_MOD, PACKET_OUT, GROUP_MOD, METER_MOD, etc.
- Can send FEATURES_REQUEST, STATS_REQUEST, BARRIER_REQUEST (query only)
- Does NOT receive PACKET_IN, FLOW_REMOVED, PORT_STATUS async
- Does receive Error messages về own request

Use case: backup controller idle, monitoring controller, distributed controller ONOS where one node is reader.

### generation_id: ngăn race condition

Khi switch có multiple controller và MASTER role changes, cần đảm bảo atomic transition (no split-brain với 2 MASTER simultaneously). OF 1.2 introduce **generation_id** concept:

- Mỗi role change request include generation_id (64-bit counter monotonically increasing)
- Switch track latest generation_id accepted
- Request với generation_id ≤ current = rejected

Protocol:
1. Controller A request MASTER với generation_id=5
2. Switch accept, current generation_id = 5
3. Controller B request MASTER với generation_id=4 (stale)
4. Switch reject với error (ROLE_STALE)
5. Controller B must query current generation_id, request with higher number

Concurrent role requests serialize via generation_id ordering. No split-brain possible.

OFPT_ROLE_REQUEST/REPLY messages specified trong §A.3.10:
- Role: OFPCR_ROLE_NOCHANGE (query only), OFPCR_ROLE_EQUAL, OFPCR_ROLE_MASTER, OFPCR_ROLE_SLAVE
- Generation ID: 64-bit

### Các mẫu hình trong môi trường production

**Active-standby HA**: 2 controller, primary có MASTER, backup có SLAVE. Nếu primary down, backup detect qua echo timeout, request role transition to MASTER. Failover time typical 5-10 seconds.

**Cluster với leader election**: 3+ controller with distributed coordination (ZooKeeper, etcd). Cluster elect leader, leader requests MASTER on every switch, others remain SLAVE. ONOS Atomix uses pattern này (Part 7.2).

**Multi-region deployment**: controllers trong 2 region, each region có MASTER cho switches local. No single global MASTER.

> **Key Topic:** Controller roles từ OF 1.2 là prerequisite cho mọi production SDN deployment HA. Trước 1.2, multi-controller setup ad-hoc và unreliable. Từ 1.2, protocol-level guarantee về role atomicity cho phép SDN controller clusters (ODL, ONOS) achieve production-grade availability. Đây là feature "enterprise readiness" không glamorous nhưng essential.

## 4.1.5 Migration OF 1.0 → 1.2: không backward compat wire

### Wire incompatibility

Switch OF 1.0 và switch OF 1.2 không thể trực tiếp communicate với controller của other version. Lý do:

**Header version khác**: OF 1.0 = 0x01, OF 1.2 = 0x03. Message header start với version byte, switch với controller mismatch immediately reject.

**Match structure hoàn toàn khác**: OF 1.0 fixed 40-byte ofp_match struct. OF 1.2 variable-length OXM TLV list. Parser logic khác biệt, không compatible at binary level.

**Instruction framework new**: OF 1.0 có list actions. OF 1.2 có instructions (WRITE_ACTIONS, APPLY_ACTIONS, ...). Semantics khác.

**Message type additions**: OF 1.2 có OFPT_ROLE_REQUEST/REPLY không tồn tại trong 1.0. Controller 1.0 không biết message này.

### Negotiation qua OFPT_HELLO

Spec 1.2 §6.3.1 định nghĩa version negotiation qua HELLO handshake:

1. **Controller connect TCP**: TCP connection established.
2. **Both sides send HELLO**: mỗi bên gửi OFPT_HELLO với supported versions bitmap. Message format mới trong 1.2 cho phép advertise multiple version (bitmap), thay vì single version của 1.0.
3. **Negotiate highest common**: cả hai sides compare bitmap, chọn highest version both support. Nếu không có common version, HELLO_FAILED error và connection close.
4. **Continue with negotiated version**: subsequent messages dùng agreed version.

Bitmap format: 32-bit per chunk, bit N set = support version N+1. Ví dụ bitmap 0x0000000A = support version 2 (OF 1.1) và 4 (OF 1.3), không support 1 (OF 1.0) hay 3 (OF 1.2). Controller/switch có thể advertise arbitrary subset of supported versions.

### Thực tế deployment

**OVS giai đoạn 2012-2014** (OVS 1.9-2.3): supports OF 1.0 và OF 1.3 đồng thời, bỏ qua 1.1 và 1.2. Lý do:
- OF 1.0 cho backward compatibility với existing controller
- OF 1.3 cho new features (LTS version)
- OF 1.1 và 1.2 thời gian ngắn, adoption thấp, không justify engineering effort

**Hardware vendor**: HP ProCurve + NEC support 1.0 và 1.3 chủ yếu, 1.1/1.2 partial hoặc bỏ qua. Pica8, Centec whitebox switch support 1.3 là focus.

**Controller frameworks**: NOX 2008 chỉ 1.0. POX chỉ 1.0. Ryu support 1.0-1.5. Floodlight 1.0 + 1.3. OpenDaylight từ Helium (10/2014) focus 1.3.

Kết quả: OF 1.2 là "transient version" trong lịch sử. Được release để formalize OXM và controller roles, nhưng market adoption quick jump từ 1.0 sang 1.3 (LTS). OF 1.2 specific deployment hiếm thấy production. Part 4.2 sẽ phân tích OF 1.3 và lý do nó trở thành de-facto LTS.

## Context: why `ovs-appctl` is the most flexible tool

The three OVSDB and OpenFlow read-write tools (`ovs-vsctl`, `ovs-ofctl`, `ovs-dpctl`) are designed for specific workflows: configure bridges, program flows, dump kernel flows. When the workflow goes beyond config CRUD or flow programming, the engineer must call the daemon directly to ask for runtime state.

`ovs-appctl` is a generic RPC wrapper. It sends a command to any OVS daemon through a Unix socket. The command is a flat string; the response is also a string. There is no type-safe schema between client and daemon; each daemon exposes its commands through `unixctl_command_register()` in its own source code.

The result: `ovs-appctl` **covers the entire live state** of every OVS daemon. More than 80 distinct commands for `ovs-vswitchd` (bridge, datapath, classifier, connection manager, bond, spanning-tree, tunnel, upcall, revalidator), more than 10 commands for `ovsdb-server` (cluster, schema, monitor), and 5 to 10 commands for each auxiliary daemon (`ovn-controller`, `ovn-northd`).

The design philosophy of `ovs-appctl` is **live, read-mostly, low-overhead**. It does not affect OVSDB config, does not change the OpenFlow pipeline, and only reads the current state or performs a transient action (flush FDB, purge revalidator, set vlog level). This makes it safe to run in production: no maintenance window required.

## 9.11.0 Grammar and target discovery

Overall grammar (man `ovs-appctl(8)` §SYNOPSIS):

```
ovs-appctl [OPTIONS] [-t TARGET | --target=TARGET] COMMAND [ARG...]

TARGET:
  - Process name: ovs-vswitchd (default), ovsdb-server, ovn-controller,
    ovn-northd, ovn-ic, ovn-controller-vtep
  - Pidfile path: /var/run/openvswitch/<name>.pid
  - Unix socket path: /var/run/openvswitch/<name>.<pid>.ctl

OPTIONS:
  --timeout=SECS    (default: wait forever)
  -h, --help
  -V, --version
```

When run without `-t`, `ovs-appctl` defaults to the `ovs-vswitchd` target. This is the largest daemon and where the majority of commands live. Other daemons must be specified explicitly with `-t`:

```shell
$ sudo ovs-appctl -t ovsdb-server list-commands
$ sudo ovs-appctl -t ovn-controller inc-engine/show-stats
$ sudo ovs-appctl -t /var/run/ovn/ovn-northd.ctl list-commands
```

### 9.11.0.1 The first step is always `list-commands`

Each daemon supports a different command set, and the set changes across OVS versions. `list-commands` is the complete map, easy to forget yet most important to remember when debugging a daemon for the first time:


### 9.11.0.2 Grouping targets by function

Organising a mental model by group helps find a target quickly:

| Group | Prefix | Purpose | Daemon |
|---|---|---|---|
| Introspection | `version`, `list-commands`, `memory/*`, `coverage/*` | Basic runtime state | All |
| Logging | `vlog/*` | Log level per module and destination | All |
| Bridge L2 | `bridge/*`, `fdb/*`, `mdb/*` | Bridge state, MAC learning, multicast snooping | ovs-vswitchd |
| Link aggregation | `bond/*`, `lacp/*` | Bond state, LACP handshake | ovs-vswitchd |
| Spanning tree | `stp/*`, `rstp/*` | 802.1D STP and 802.1w RSTP | ovs-vswitchd |
| L2 monitoring | `bfd/*`, `cfm/*` | BFD and Connectivity Fault Management | ovs-vswitchd |
| OpenFlow pipeline | `ofproto/*`, `bundle/*` | OpenFlow introspection | ovs-vswitchd |
| Kernel datapath | `dpctl/*`, `dpif/*` | Kernel flow and conntrack | ovs-vswitchd |
| Userspace DPDK | `dpif-netdev/*` | PMD thread, rxq, EMC | ovs-vswitchd (netdev datapath only) |
| Tunnel | `tnl/*` | Tunnel ARP/ND cache, port listening | ovs-vswitchd |
| Upcall and revalidator | `upcall/*`, `revalidator/*` | Flow install and GC pipeline | ovs-vswitchd |
| OVSDB cluster | `cluster/*` | Raft cluster management | ovsdb-server |
| OVN controller | `inc-engine/*`, `lflow-cache/*`, `ct-zone-list` | OVN runtime state | ovn-controller |

Total count: **about 12 functional groups in ovs-vswitchd, plus 1 for ovsdb-server, plus 3 to 4 for ovn-controller, equals about 16 to 17 groups**. This part covers the 11 most important groups for ovs-vswitchd plus 1 for ovsdb-server.

## 9.11.1 Introspection and telemetry: `version`, `memory/show`, `coverage/show`

### 9.11.1.1 `version`

A trivial command, but the first one to identify the version when remote-debugging:

```shell
$ sudo ovs-appctl version
ovs-vswitchd (Open vSwitch) 2.17.9-0ubuntu0.22.04.1
```

The output is a single ASCII string. Do not confuse it with `ovs-vsctl --version` (which gives the CLI tool version) or `ovs-vsctl get Open_vSwitch . ovs_version` (which dumps the version stored in OVSDB). All three must be consistent; a mismatch means the daemon is running an old version while the OVSDB schema has been upgraded.

### 9.11.1.2 `memory/show`: an anatomy breakdown of ovs-vswitchd memory

**Context:** debug a memory leak, or find which component consumes heap. Every hour, or after every reconfigure, run `memory/show` and log it to a file; compare deltas.

**Command and output** `[real capture, br-int OVN 22.03.8, 3 bridges, 50 ports, 1200 flows]`:

```shell
$ sudo ovs-appctl memory/show
cells:48289 handlers:4 idl-cells-OVN_Southbound:82134 \
  idl-cells-Open_vSwitch:2453 \
  of-connections:2 \
  ofconns:2 \
  ports:56 \
  revalidators:4 \
  rules:12847 \
  udpif keys:1247
```

The output is a single-line key=value list, easy to parse in a script.

**Per-counter anatomy:**

| Counter | Meaning | Healthy range | Abnormal signal |
|---|---|---|---|
| `cells` | Total ofpbuf cells allocated (an internal structure inside ovs-vswitchd) | Below 100k for a small setup | Leak: monotonic growth that does not drop after one hour of idle |
| `handlers` | Number of handler threads (handling upcalls) | Equal to physical cores divided by 4 (default) | Different from `other_config:n-handler-threads` means the config was not applied |
| `idl-cells-<schema>` | Number of rows in the daemon's IDL cache for each OVSDB schema | Matches the actual row count | Grows faster than the actual DB means an IDL memory leak |
| `of-connections` | Active OpenFlow connections currently established | Equal to the number of controllers actually connected | 0 when more than 0 is expected means a controller is unreachable |
| `ofconns` | Allocated OpenFlow connection objects | Matches `of-connections` | A mismatch means a connection leak |
| `ports` | Total allocated port objects across all bridges | Matches `ovs-vsctl list Interface \| wc -l` | A mismatch means bridge cleanup missed something |
| `revalidators` | Number of revalidator threads | Matches the `n-revalidator-threads` config | A mismatch means OVS has not restarted since the setting changed |
| `rules` | Total allocated flow entries across every bridge and table | Workload-dependent | Monotonic growth in steady state means a flow leak from the controller |
| `udpif keys` | Number of ukeys (unique flow keys) in the datapath-to-userspace mapping | Close to `rules` for the kernel datapath | `udpif keys` much greater than the flow count from `dpctl/show` matches the Part 9.26 revalidator storm pattern |

**Failure scenarios:**

- **`cells` grows by more than 10 percent per hour in steady state.** Memory leak; bisect by disabling each feature (mirror, sflow, conntrack zones) and watch when the counter stops growing.
- **`udpif keys` grows monotonically faster than `rules`.** Ukeys are not being GC'd; check the `revalidator/wait` response, the revalidator thread may be stuck. Part 9.26 describes this pattern in detail.
- **`handlers` equals 0.** Either ovs-vswitchd is starting and has not forked handlers yet, or a thread crashed; confirm with `systemctl status ovs-vswitchd`.

### 9.11.1.3 `coverage/show`: internal event counters

**Context:** `coverage/*` is the hidden telemetry subsystem in OVS. Every "notable event" along a code path has a `COVERAGE_INC()` counter. Read `coverage/show` periodically, compare deltas, and detect anomalies that log messages never expose.

**Command and output** `[real capture, steady state, 5 minutes after startup]`:

```shell
$ sudo ovs-appctl coverage/show

flow_extract                 4.8/sec     4.213/sec     4.209/sec   total: 287423
hit                          4.8/sec     4.103/sec     4.209/sec   total: 287423
miss                         0.2/sec     0.125/sec     0.134/sec   total: 8127
ofproto_flush                0.0/sec     0.000/sec     0.000/sec   total: 0
ofproto_packet_out           0.0/sec     0.020/sec     0.025/sec   total: 79
revalidate                   0.0/sec     0.017/sec     0.015/sec   total: 183
rstp_run                     0.0/sec     0.000/sec     0.000/sec   total: 0
upcall_flow_add              0.0/sec     0.025/sec     0.015/sec   total: 189
upcall_flow_del              0.0/sec     0.020/sec     0.012/sec   total: 145
upcall_ukey_lookup_created   1.2/sec     0.421/sec     0.208/sec   total: 12453

xlate_actions                5.0/sec     4.228/sec     4.219/sec   total: 288612
```

**Row anatomy:**

- **Column 1:** counter name (declared in source with `COVERAGE_DEFINE(name)`).
- **Column 2:** average rate over the last 5 seconds.
- **Column 3:** average rate over the last minute.
- **Column 4:** average rate over the last hour.
- **Column 5:** `total: N`, the cumulative count since startup.

**Foundation counter classes you need to know:**

| Counter | Description | Healthy threshold |
|---|---|---|
| `hit` | Megaflow lookup hit; the packet matched the cache | - |
| `miss` | Cache miss; triggers an upcall | `miss / hit` below 1 percent in steady state |
| `flow_extract` | Parse the L2 to L4 header of a packet (every upcall) | Roughly equal to `miss` |
| `xlate_actions` | Translate the OpenFlow action set into datapath actions | Equal to `upcall_ukey_lookup_created` plus `upcall_ukey_lookup_hit` |
| `upcall_flow_add` | Install a new flow into the datapath | Below 10 per second in steady state |
| `upcall_flow_del` | Remove a flow from the datapath | Roughly balanced with `upcall_flow_add` |
| `upcall_ukey_lookup_created` | A new ukey was created | Below 10 per second; above 100 per second indicates flow thrashing |
| `upcall_ukey_lookup_hit` | Hit an existing ukey | High value indicates good cache reuse |
| `revalidate` | A full revalidator pass | OVS 2.17 target is 0.1 to 1 per second; above 10 per second matches the Part 9.26 storm |
| `facet_revalidate` | Check each facet (flow-specific) | - |
| `facet_changed_rule` | The rule a packet matches changed after install, indicating unstable OpenFlow programming | Continuous growth means the controller is churning |
| `ofproto_flush` | A bridge flush (re-install all rules) | Above 0 per minute means a controller restart or an OVS reconfig |
| `bridge_reconfigure` | An OVSDB config change triggered a reconfig | Above 10 per hour means config churn; check the orchestrator |
| `vconn_sent` / `vconn_received` | OpenFlow connection I/O | - |
| `util_xalloc` | Memory allocations through xmalloc | A sustained high rate means memory churn |
| `netlink_received` / `netlink_sent` | Netlink I/O between kernel and userspace | - |
| `rstp_run` / `stp_run` | Spanning-tree protocol tick | 0 if STP is disabled |
| `connmgr_wakeup` | Connection manager wakeups | - |

**Failure scenarios based on counter deltas:**

- **`hit / (hit + miss)` below 90 percent.** Cache effectiveness is low; check `megaflow_mask_hit_ratio` in Part 9.17. The megaflow may be too narrow, with each particular packet installing its own flow (flow explosion). See Part 9.15 on tuple space search to optimise.
- **`upcall_flow_add` much greater than `upcall_flow_del`.** Flows are accumulating in the datapath and approaching the limit. Combine with `upcall/show flows current` to check remaining space.
- **`facet_changed_rule` above 1 per second.** The OpenFlow pipeline is churning; the controller is installing or modifying flows continuously. With OVN, check `ovn-appctl -t ovn-controller inc-engine/show-stats` for the I-P engine recompute rate.
- **`ofproto_flush` above 0 when not expected.** A bridge config is rolling out; if not initiated by the controller, check the OVSDB connection. A disconnect and reconnect can cause a full flush.

### 9.11.1.4 `coverage/read-counter`: read a single counter

When an automation script needs to poll a specific counter (rather than parsing the full `coverage/show`):

```shell
$ sudo ovs-appctl coverage/read-counter upcall_flow_add
189
```

The output is a single integer. Faster and with a more stable format. Use it in a Prometheus exporter:

```bash
#!/bin/bash
counters=(
  hit miss
  upcall_flow_add upcall_flow_del
  upcall_ukey_lookup_created
  revalidate facet_revalidate
  ofproto_flush bridge_reconfigure
)
for c in "${counters[@]}"; do
  val=$(sudo ovs-appctl coverage/read-counter "$c" 2>/dev/null || echo 0)
  echo "ovs_coverage_${c}_total $val"
done
```

### 9.11.1.5 `vlog/list` and `vlog/set`: tune detailed logs without restarting

**Context:** the default OVS log level is `info` for the file destination and `err` for the console. To deep-debug one module (for example `dpif`), raise its level to `dbg` for that module only, then restore it. No daemon restart needed.

```shell
# List modules and their current levels
$ sudo ovs-appctl vlog/list | head -15
                 console    syslog    file
                 -------    -------   ------
backtrace        OFF        ERR       INFO
bfd              OFF        ERR       INFO
bond             OFF        ERR       INFO
bridge           OFF        ERR       INFO
classifier       OFF        ERR       INFO
connmgr          OFF        ERR       INFO
...
dpif             OFF        ERR       INFO
dpif_netdev      OFF        ERR       INFO
...
```

**Anatomy:**

- **Row:** the module name. Each module corresponds to one OVS source file (`lib/dpif.c` becomes module `dpif`).
- **Column:** the destination. `console` (stderr), `syslog` (/var/log/syslog through syslogd), `file` (/var/log/openvswitch/ovs-vswitchd.log).
- **Cell:** the level. `EMER`, `ERR`, `WARN`, `INFO`, `DBG`, `OFF`.

**Set syntax:**

```
ovs-appctl vlog/set [SPEC[:SPEC]...]
SPEC: <module>:<destination>:<level>
Special: ANY for module, ANY for destination
```

For example, raise the `dpif` log to debug on the file destination, leaving every other module unchanged:

```shell
$ sudo ovs-appctl vlog/set dpif:file:dbg
$ sudo tail -f /var/log/openvswitch/ovs-vswitchd.log | grep dpif
2026-04-24T11:15:23.412Z|00042|dpif(handler1)|DBG|system@ovs-system: \
  port_query_by_number(3): pkts:1024 bytes:131072
2026-04-24T11:15:23.412Z|00043|dpif(handler1)|DBG|system@ovs-system: \
  execute ufid:a1b2c3d4..., actions: 2,3
```

Restore after debugging:

```shell
$ sudo ovs-appctl vlog/set dpif:file:info
```

**Shortcut to debug the entire datapath pipeline:**

```shell
$ sudo ovs-appctl vlog/set dpif:file:dbg dpif_netdev:file:dbg \
    ofproto_dpif_upcall:file:dbg ofproto_dpif_xlate:file:dbg
```

The log file becomes very large (more than 100 MB per minute under load). Restore the level as soon as you no longer need it.

### 9.11.1.6 `vlog/reopen`: rotate the log file

After `logrotate` renames the log file, the daemon still holds the old file descriptor and writes to the renamed file. `vlog/reopen` closes and reopens the log to pick up the new path:

```shell
# Add to /etc/logrotate.d/openvswitch:
/var/log/openvswitch/*.log {
    daily
    rotate 7
    compress
    postrotate
        /usr/bin/ovs-appctl vlog/reopen
    endscript
}
```

Without `vlog/reopen`, ovs-vswitchd writes into `ovs-vswitchd.log.1.gz` (meaningless), and the new file stays empty.

## 9.11.2 Bridge L2: `bridge/dump-flows`, `fdb/show`, `mdb/show`

### 9.11.2.1 `bridge/dump-flows` versus `ovs-ofctl dump-flows`

Two commands with small but important differences:

| Aspect | `ovs-ofctl dump-flows` | `ovs-appctl bridge/dump-flows` |
|---|---|---|
| Protocol | OpenFlow OFPT_STATS_REQUEST | Unix socket RPC |
| Scope | User-visible flows (installed by the controller) | **All flows including hidden ones** (internal learn, ARP responder, OVN-generated) |
| Format | OpenFlow format, standard match spec | OVS internal format |
| Speed | Over TCP/SSL when remote | Over Unix socket, faster locally |
| Use case | Common everyday use | When debugging hidden flows |

**When to use `bridge/dump-flows`:** when `ovs-ofctl dump-flows br0` returns fewer rules than expected, or when debugging OVN-generated flows that use a `learn()` action to create dynamic flows.

```shell
$ sudo ovs-appctl bridge/dump-flows br-int | head -15
NXST_FLOW reply (xid=0x4):
 cookie=0x0, duration=3421.234s, table=0, n_packets=18234, n_bytes=2145678,
   priority=100,in_port=1 actions=load:0x1->NXM_NX_REG13[],resubmit(,8)
 cookie=0x0, duration=3421.112s, table=0, n_packets=9812, n_bytes=1014523,
   priority=100,in_port=2 actions=load:0x2->NXM_NX_REG13[],resubmit(,8)
 cookie=0x0, duration=8.520s, table=64, n_packets=0, n_bytes=0,
   priority=0 actions=resubmit(,65)
 <hidden flows in tables ≥ 64>
```

Note: flow tables 64 and above belong to the OVN "physical pipeline" internals, which `ovs-ofctl dump-flows` does NOT show unless you pass `--no-hidden`. `bridge/dump-flows` always shows them.

### 9.11.2.2 `fdb/show`: read the MAC learning table

**Context:** debug L2 forwarding, for example "traffic goes to the wrong port, the ping reply comes back in the wrong direction, the ARP reply does not reach the right port". In 90 percent of cases, MAC learning is wrong.

**Command and output** `[real capture, br-int with 3 ports that have carried traffic]`:

```shell
$ sudo ovs-appctl fdb/show br-int
 port  VLAN  MAC                Age
    1     0  aa:bb:cc:00:00:01    4
    2     0  aa:bb:cc:00:00:02   12
    3   100  aa:bb:cc:00:00:03   87
    1     0  aa:bb:cc:00:00:04  287
 LOCAL    0  3e:6a:a1:b2:c3:d0  3421
```

**Anatomy:**

| Field | Meaning | Abnormal signal |
|---|---|---|
| `port` | OpenFlow port number where the MAC was learned. `LOCAL` is the bridge's internal port | Many MACs on `LOCAL` indicate broadcast traffic entering through the bridge IP |
| `VLAN` | VLAN ID from `vlan_tci`. `0` is untagged or the native VLAN | A VLAN that differs from the access port tag indicates misconfig; learning a MAC through a trunk leaks the broadcast domain |
| `MAC` | The learned MAC address | A multicast MAC (`01:00:5e:*`) in the FDB is a bug; multicast addresses normally are not learned into the FDB |
| `Age` | Seconds since the last time it was learned | Above 300 seconds (default aging) it should be gone; if it persists, `other_config:mac-aging-time` is custom |

**Failure scenarios:**

- **The same MAC appears on two different ports (MAC flapping).** Either an L2 loop or a duplicate MAC (a VM migration that did not clear the ARP cache). Check `bridge/dump-flows` for the bridge-ARP flow. Use `ovs-appctl stp/show` or `rstp/show` to verify whether loop prevention is active.
- **The `LOCAL` port has many MACs in the same VLAN.** Broadcast traffic is entering the bridge through the internal IP (br-int has an IP assigned on its internal interface). Usually fine, but worth noting in tight workload isolation setups.
- **FDB is empty on a bridge with traffic.** The bridge has `fail_mode=secure` and the controller has not installed a learn flow, so every non-matching packet is dropped and no MAC is learned. Check `ovs-ofctl dump-flows br-int` for `actions=NORMAL` or an explicit `learn()`.

**Companion: `fdb/flush` to force relearning:**

```shell
$ sudo ovs-appctl fdb/flush br-int        # flush everything
$ sudo ovs-appctl fdb/flush br-int 100    # only VLAN 100
```

After the flush, subsequent traffic re-populates the FDB. Use it when you suspect a stale MAC cache (after migration, after failover).

### 9.11.2.3 `mdb/show`: the multicast snooping table

**Context:** when `mcast_snooping_enable=true` is set, the bridge filters multicast based on IGMP reports instead of flooding. `mdb/show` is the main debug command.

```shell
$ sudo ovs-appctl mdb/show br-int
 port  VLAN  GROUP                Age
    2     0  224.0.0.251            45
    3     0  239.1.1.1             120
    2     0  ff02::fb (MLDv2)       45
```

**Anatomy:**

- `port`. The OpenFlow port subscribed to the group.
- `VLAN`. The VLAN context.
- `GROUP`. The multicast group address (IPv4 or IPv6 MLD).
- `Age`. Seconds since the last IGMP report.

**Scenarios:**

- **MDB is empty when multicast subscribers are expected.** IGMP snooping is not enabled or the querier is not active; check `ovs-vsctl get Bridge br-int mcast_snooping_enable`.
- **A group in `224.0.0.0/24` appears in the MDB.** This is the reserved local-network control-plane scope (OSPF hello, IGMP) and normally should not be snooped; configure `mcast-snooping-disable-flood-unregistered` if you want to block it.

## 9.11.3 Link aggregation: `bond/show` and `lacp/show`

### 9.11.3.1 `bond/show`: read bond state

**Context:** a multi-NIC host uses a bond (active-backup, balance-slb, balance-tcp) for resilience and bandwidth aggregation. When ping is intermittent or actual throughput drops, `bond/show` is the first debug command.

**Command and output** `[real capture, bond0 with 2 NICs, balance-tcp]`:

```shell
$ sudo ovs-appctl bond/show bond0
---- bond0 ----
bond_mode: balance-tcp
bond may use recirculation: yes, Recirc-ID : 1
bond-hash-basis: 0
updelay: 0 ms
downdelay: 0 ms
next rebalance: 9821 ms
lacp_status: negotiated
lacp_fallback_ab: false
active-backup primary: <none>

slave eno1: enabled
  active slave
  may_enable: true
  hash 38: 15 kB load
  hash 113: 0 kB load

slave eno2: enabled
  may_enable: true
  hash 41: 32 kB load
  hash 147: 2 kB load
```

**Bond-level anatomy:**

| Field | Meaning | Abnormal signal |
|---|---|---|
| `bond_mode` | `active-backup` (one active member), `balance-slb` (source MAC), or `balance-tcp` (L4 hash) | Production usually uses `balance-tcp` plus LACP; `balance-slb` does not need LACP but is not optimal |
| `bond may use recirculation` | `yes` when the datapath recirculates packets to compute the hash (balance-tcp needs this) | `no` with balance-tcp means an older kernel without support, and performance suffers |
| `Recirc-ID` | The recirculation ID the kernel uses for the bond | - |
| `bond-hash-basis` | The hash salt that avoids hash collisions across hosts | Identical across many hosts means ECMP cross-chassis traffic skews |
| `updelay` / `downdelay` | Millisecond delay before bringing a member up or down after a link change | 0 ms is instant; 200 ms is the default LACP recommendation |
| `next rebalance` | Milliseconds until the next rebalance pass (balance-tcp) | Default 10000 ms (10 seconds) |
| `lacp_status` | `negotiated`, `configured`, or `disabled` | `configured` but not `negotiated` means the LACP partner is not replying; check cabling or peer LACP config |
| `lacp_fallback_ab` | Fall back to active-backup if LACP fails | `true` is recommended when you are unsure the peer supports LACP |
| `active-backup primary` | Which member is preferred when the mode is active-backup | `<none>` means a random selection |

**Slave-level anatomy:**

| Field | Meaning | Abnormal signal |
|---|---|---|
| `slave <name>` | The member interface name | - |
| `enabled` / `disabled` | Whether it carries traffic | `disabled`, check `ethtool <iface>` for link state |
| `active slave` (active-backup) | The member currently forwarding traffic | Not present on any slave means primary selection has not completed |
| `may_enable` | The member can be enabled (LACP passed, link up, and so on) | `false` means LACP has not negotiated or the carrier is down |
| `hash N: X kB load` | TCP hash bucket N is currently handled here, with the current load in bytes | Skew across slaves (90 percent of load on one slave) means balance-tcp hash does not spread, possibly because of an elephant flow |

**Failure scenarios:**

- **`lacp_status: configured` (not `negotiated`).** LACP PDUs are dropped at the cable level, the peer switch is not configured with LACP correctly, or the peer mode is static LAG. Check `lacp/show bond0` for actor and partner state.
- **Hash load skew.** One slave carries 90 percent of the traffic, caused by an elephant flow (a single TCP 5-tuple dominates). Use `bond/migrate bond0 <hash> <slave>` to rebalance manually, or switch to `balance-slb` when the flow count is high.
- **`may_enable: false` after LACP `negotiated`.** Rare; check the OVS log with `grep lacp /var/log/openvswitch/ovs-vswitchd.log`.

### 9.11.3.2 `lacp/show`: actor and partner handshake state

**Context:** the bond mode uses LACP. When the bond's `lacp_status` will not negotiate, dig into LACP state.

```shell
$ sudo ovs-appctl lacp/show bond0
---- bond0 ----
    status: active negotiated
    sys_id: 3e:6a:a1:b2:c3:d0
    sys_priority: 65534
    aggregation key: 1
    lacp_time: fast

slave: eno1
  status: current attached
  port_id: 1
  port_priority: 65535
  may_enable: true

  actor sys_id: 3e:6a:a1:b2:c3:d0
  actor sys_priority: 65534
  actor port_id: 1
  actor port_priority: 65535
  actor key: 1
  actor state: activity timeout aggregation synchronized collecting \
    distributing

  partner sys_id: 00:1e:68:aa:bb:cc
  partner sys_priority: 127
  partner port_id: 5
  partner port_priority: 127
  partner key: 1234
  partner state: activity timeout aggregation synchronized collecting \
    distributing
```

**LACP state bitfield anatomy:**

| Flag | Meaning | Expected |
|---|---|---|
| `activity` | Actor is sending LACP PDUs (active mode) | Always set |
| `timeout` | Fast mode (1 second) versus slow (30 seconds) | Fast equals `lacp_time: fast` |
| `aggregation` | The member can aggregate | Always set |
| `synchronized` | Bond and peer agree on state | CRITICAL: if missing, LACP mismatch |
| `collecting` | The member is receiving traffic | Required for the data path |
| `distributing` | The member is transmitting traffic | Required |
| `defaulted` | Using the default partner (no PDU received) | Should not appear |
| `expired` | PDU timeout has elapsed | Should not appear |

**Failure scenarios:**

- **Actor and partner sys_id are identical.** A cable creates a loopback (a port is connected to another port on the same bridge). Fix the cable.
- **`partner state: defaulted expired`.** No LACP PDU has been received from the peer for 3 times the timeout; the cable failed or the peer does not run LACP.
- **`actor key` does not match `partner key`.** The Port Group ID does not match; verify that the peer LACP config places them in the same bond group.

## 9.11.5 L2 monitoring: `bfd/show` and `cfm/show`

### 9.11.5.1 `bfd/show`: BFD session anatomy

**Context:** BFD (RFC 5880) detects link failure in milliseconds. It is used heavily in OVN HA_Chassis_Group for gateway chassis failover.

```shell
$ sudo ovs-appctl bfd/show
---- br-int port ovn-huyvl-0 ----
Forwarding: true
Detect Multiplier: 3
Concatenated Path Down: false

Local Session Diagnostic: No Diagnostic
Remote Session Diagnostic: No Diagnostic

Remote Discriminator: 0x000054b7

Last packet received: 340 ms ago
```

**Anatomy:**

| Field | Meaning | Abnormal signal |
|---|---|---|
| `Forwarding` | Packets forward through this interface | `false` means BFD is down and OVN failover is active |
| `Detect Multiplier` | Number of missed BFD packets before declaring the session down | Default 3 |

Common in service-provider L2 networks; rarely seen in cloud OVN deployments.

## 9.11.6 OpenFlow pipeline: `ofproto/trace` and `ofproto/list`

The `ofproto/trace` target is covered in detail in Part 9.4 §9.4.3.3 (full Template A). This part adds companion commands:

### 9.11.6.1 `ofproto/list`: list managed bridges

```shell
$ sudo ovs-appctl ofproto/list
br-int
br-ex
```

The output is simple: one bridge name per line. Use it in scripts that iterate over every bridge:

```bash
for br in $(sudo ovs-appctl ofproto/list); do
  sudo ovs-appctl ofproto/trace "$br" "in_port=1,..." > "/tmp/trace-$br.txt"
done
```

### 9.11.6.2 `ofproto/trace -generate`: side-effect simulation

The `-generate` flag turns a trace from a pure simulation into a side-effecting execution. Use it when actions include:
- `learn(...)`. Actually creates a new flow.
- `ct(commit)`. Actually commits to conntrack.
- `controller(userdata=...)`. Actually sends a packet_in.

Example:

```shell
$ sudo ovs-appctl ofproto/trace br-int "in_port=1,dl_src=aa:bb:..." -generate
# After the trace, check fdb/show br-int and you will see the newly learned MAC.
$ sudo ovs-appctl fdb/show br-int | grep aa:bb
    1     0  aa:bb:cc:dd:ee:ff    0
```

Do not use `-generate` in production unless you are sure of the consequences. Pairing it with a dedicated test bridge (a separate `br-test`) is safer.

### 9.11.6.3 `bundle/show`: OpenFlow 1.4 bundle transactions

When `ovs-ofctl --bundle replace-flows` is used, OVS keeps the pending bundle before commit. `bundle/show` lists them:

```shell
$ sudo ovs-appctl bundle/show
Bridge "br-int": 1 bundle(s) in progress
Bundle 0x1:
  Flags: atomic
  Messages: 47 flow_mods
```

Rarely used directly. Mostly when debugging a stuck bundle (commit not completing).

## 9.11.9 Tunnel: `tnl/neigh/show` and `tnl/ports/show`

### 9.11.9.1 `tnl/neigh/show`: ARP/ND cache for tunnel endpoints

**Context:** OVS tunnels (Geneve, VXLAN, GRE) need to know the MAC of the next-hop router to encapsulate packets. This cache lives inside ovs-vswitchd and differs from the kernel ARP cache.

```shell
$ sudo ovs-appctl tnl/neigh/show
IP                                            MAC                Bridge
==========================================================================
10.200.0.12                                   f8:ca:b8:aa:11:22  br-int

fe80::faca:b8ff:feaa:1122                     f8:ca:b8:aa:11:22  br-int
```

**Failure scenarios:**

- **Tunnel up (Geneve port state OK) but traffic does not flow.** Check `tnl/neigh/show`. If there is no entry for the remote_ip neighbour, ARP has not resolved. Force ARP resolution with a ping:

```shell
$ sudo ip neigh show to 10.200.0.12

# If the kernel has a cache entry but OVS does not, force a reload:
$ sudo ovs-appctl tnl/neigh/flush
```

## 9.11.10 Upcall and revalidator: `upcall/*` and `revalidator/*`

### 9.11.10.1 `upcall/show`: fast and slow path health

Detail in Part 9.4 §9.4.3.5. Summary:

```shell
$ sudo ovs-appctl upcall/show
system@ovs-system:
  flows       : (current 1247) (avg 1250) (max 1350) (limit 200000)
  dump duration : 8ms
  ufid enabled : true

  1: (keys 624) (max-flows 64) (avg-updates 1.2) (busy cycles 0.03%)
  2: (keys 623) (max-flows 63) (avg-updates 1.3) (busy cycles 0.03%)
```

### 9.11.10.2 `upcall/set-flow-limit`: raise the limit

**Context:** when `upcall/show` shows `current` close to `limit`, new flows are throttled. Raise the limit:

```shell
$ sudo ovs-appctl upcall/set-flow-limit 500000
$ sudo ovs-appctl upcall/show | grep flows
  flows       : (current 1247) (avg 1250) (max 1350) (limit 500000)
```

**Trade-off:** raising the limit is not free. Each flow consumes about 200 to 500 bytes of kernel memory. A limit of 500k flows is roughly 250 MB. Reconsider on memory-constrained setups.

### 9.11.10.3 `revalidator/purge`: force-evict megaflows

**Context:** after changing the OpenFlow pipeline, force the kernel to re-learn every flow. Use it instead of waiting for the idle timeout.

```shell
$ sudo ovs-appctl revalidator/purge
$ sudo ovs-appctl upcall/show | grep "current"
  flows : (current 0) ...
```

After the purge, the first packet of each flow triggers an upcall and re-install. There is a brief latency spike, but the state is clean.

### 9.11.10.4 `revalidator/wait`: sync barrier

Wait for the revalidator pass to complete before querying further. Useful in automation scripts:

```shell
$ sudo ovs-appctl revalidator/wait
$ sudo ovs-appctl dpctl/dump-flows  # guaranteed to be the latest state
```

## 9.11.11 OVSDB cluster: `cluster/*`

These targets live on `ovsdb-server` and require an explicit `-t`:

### 9.11.11.1 `cluster/status`: Raft consensus health

**Context:** production OVN usually runs OVSDB in clustered mode (Part 10.1) with 3 to 5 members. `cluster/status` is the first Raft debug command.

```shell
$ sudo ovs-appctl -t /var/run/ovn/ovnsb_db.ctl cluster/status OVN_Southbound
7a21
Name: OVN_Southbound
Cluster ID: 4f27 (4f271a42-...)
Server ID: 7a21 (7a2104f1-...)
Address: tcp:10.0.0.1:6644
Status: cluster member
Role: leader
Term: 42
Leader: self
Vote: self

Last Election started 892123 ms ago, reason: timeout
Last Election won: 892100 ms ago
Election timer: 1000
Log: [2, 12847]
Entries not yet committed: 0
Entries not yet applied: 0
Connections: <- 8a3b ->c4d1 (->=self sending, <-=receiving)
Disconnections: 0
Servers:
    7a21 (7a21 at tcp:10.0.0.1:6644) (self) next_index=12848 match_index=12847
    8a3b (8a3b at tcp:10.0.0.2:6644) next_index=12848 match_index=12847
    c4d1 (c4d1 at tcp:10.0.0.3:6644) next_index=12848 match_index=12847
```

**Anatomy:**

| Field | Meaning | Abnormal signal |
|---|---|---|
| `Role` | `leader`, `follower`, or `candidate` | `candidate` for more than 5 seconds means an election is stuck |
| `Term` | Raft term counter | Increases each election; rapid jumps indicate leader instability |
| `Leader: self` | Identity of the leader | If the leader disappears, the cluster is not writable |
| `Last Election won` | Time since the most recent election | Below one minute means a recent election; check for instability |
| `Log: [2, 12847]` | Start index and latest index | A gap across members means log replication lag |
| `Entries not yet committed` | Writes pending quorum ACK | Sustained above 0 means quorum is not met |
| `Entries not yet applied` | Committed but not yet applied to the DB | Above 0 means the apply thread is slow |
| `Disconnections` | Counter of disconnects from peers | Growing indicates an unstable network |
| `match_index` | Latest log index of the peer that matches the leader | A large gap means the peer is lagging and must catch up |

**Failure scenarios:**

- **Three followers, no leader.** Network partition; check `iptables -L -n` plus `ss -tlnp | grep 6644`.
- **`Term` jumps by 5 or more in a minute.** An election storm, usually caused by a flapping network or a heartbeat timeout set too low.
- **A peer's `match_index` lags far behind the leader.** The peer is slow applying log entries; check disk I/O (OVSDB fsync is heavy).

### 9.11.11.2 `cluster/kick`: remove a member from the cluster

When a member is permanently gone (host decommission), remove it manually:

```shell
$ sudo ovs-appctl -t /var/run/ovn/ovnsb_db.ctl cluster/kick OVN_Southbound 7a21
server 7a21 kicked
```

Only run from the leader. A "kicked" member cannot rejoin on its own; the DB must be destroyed and recreated on that host.

### 9.11.11.3 `cluster/leave`: a member voluntarily leaves

Preferred for graceful shutdown:

```shell
$ sudo ovs-appctl -t /var/run/ovn/ovnsb_db.ctl cluster/leave OVN_Southbound
```

After a leave, the member transitions out of the voting set automatically, the leader commits the config change, and the cluster size shrinks.

```shell
$ sudo ovs-appctl memory/show > /tmp/mem-after.txt
```

**4.** Delta analysis:

```shell
$ diff /tmp/cov-before.txt /tmp/cov-after.txt
# You will see the `flow_table_mods` counter increase by roughly 1000.

$ diff /tmp/mem-before.txt /tmp/mem-after.txt
# The `rules` counter increases by 1000.
```

**Comprehend questions:**

- Why does `upcall_flow_add` not increase by 1000 (flows are installed through OpenFlow, not through upcall)?
- Why does the `hit` counter barely change (the newly installed flows do not match real packets)?

**5.** Cleanup:

```shell
$ sudo ovs-vsctl del-br br-test
```

## Key takeaways

First, `ovs-appctl` covers **the entire live state** of every OVS daemon through about 80 commands grouped into 12 or more functional groups. Knowing the grouping by function (introspection, L2, bond, STP, OpenFlow, datapath, tunnel, upcall, cluster) helps you find a target quickly.

Second, `list-commands` is the complete map. Always run it first when debugging an unfamiliar daemon. Run it regularly against ovn-controller, ovn-northd, and ovsdb-server to know the capabilities of the version currently running.

Third, three foundation targets for observability: `coverage/show` (event counters), `memory/show` (heap breakdown), `upcall/show` (fast and slow path health). A cron job that logs the delta every minute, exported to Prometheus, drives anomaly detection.

Fourth, `vlog/set module:destination:level` enables module-specific debugging without restarting the daemon. After debugging, always restore to `info`. Log level `dbg` in production wastes I/O and forces aggressive log rotation.

Fifth, every target has a clear Anatomy for its output. Do not skim the output; learn the anatomy of each target once, and read-only debug output is yours for life.

Sixth, `cluster/status` for clustered OVSDB is the debug entry point when OVN writes are slow or time out. Raft consensus state (term, role, match_index) must be as readable to a network engineer as BGP neighbor state.

### 9.11.X.1 `ovs-appctl dpif/dump-flows`

#### Concept

`dpif/dump-flows` is the `ovs-appctl` subcommand that asks `ovs-vswitchd` to descend directly into the `dpif` (datapath interface) layer and list **every flow installed** in the datapath bound to a bridge: the kernel datapath (`dpif-netlink`) or the userspace datapath (`dpif-netdev`). This is the "physical truth" of the cache that is currently serving packets. Each output line is a megaflow (after kernel consolidation) or a microflow, with real `n_packets`/`n_bytes` counters and the `actions` actually executed on matching packets.

#### Role in the architecture

The OVS pipeline has three layers: OVSDB schema (the desired control plane), OpenFlow tables in `ofproto` (rules installed by the controller), and flows in the datapath (the final cache that serves packets). When an engineer asks "is my packet matching my rule", `ovs-ofctl dump-flows` only answers about the OpenFlow layer. `dpif/dump-flows` answers about the datapath layer, which is the layer that actually sees packets. It is the only tool that confirms whether a megaflow has been installed, how many times it has hit or missed, and whether the revalidator has evicted it.

#### Syntax and parameters

```
ovs-appctl dpif/dump-flows [-m] [--names] [DP]
```

- `DP` is the datapath name (default `system@ovs-system` for the kernel, `netdev@ovs-netdev` for userspace). It can be omitted when only one datapath exists.
- `-m` (more) adds normally hidden fields: `recirc_id`, `dp_hash`, `ct_state`, `ct_zone`, `ct_mark`, `ct_label`. Required when investigating conntrack or the recirc pipeline.
- `--names` replaces numeric `in_port(N)` with named `in_port(eth0)`.

#### Real output sample

```shell
$ sudo ovs-appctl dpif/dump-flows -m | head -3
ufid:6a8c1f29-7d2b-44ee-9f0e-01e3a8b27d11,
  recirc_id(0),dp_hash(0/0),skb_priority(0/0),in_port(2),
  skb_mark(0/0),ct_state(0/0),ct_zone(0/0),ct_mark(0/0),ct_label(0/0),
  eth(src=aa:bb:cc:00:00:02/00:00:00:00:00:00,dst=aa:bb:cc:00:00:01/...),
  eth_type(0x0800),ipv4(src=10.1.0.5,dst=10.1.0.10,proto=6,frag=no/0xff),
  tcp(src=50000/0xffff,dst=80/0xffff),tcp_flags(0/0),
  packets:18234, bytes:2145678, used:0.082s, flags:SP, dp:ovs, actions:4
```

#### Differentiation from `ovs-ofctl dump-flows`

This is the most common point of confusion. The two commands return fundamentally different data:

| Criterion | `ovs-ofctl dump-flows br0` | `ovs-appctl dpif/dump-flows` |
|-----------|----------------------------|------------------------------|
| Layer | OpenFlow rules (installed by the controller) | Datapath cache (translated by `ovs-vswitchd` down to the kernel) |
| Row count | Proportional to the number of rules the controller pushes | Proportional to the number of megaflows currently active |
| Counter | `n_packets`/`n_bytes` per OF rule | `packets`/`bytes` per megaflow |
| Wildcard | According to OF match fields | According to the kernel-issued mask (after xlate and consolidation) |
| Purpose | Verify "did the controller install the right rule" | Verify "what is the kernel actually doing with packets" |

When an OpenFlow flow exists but `dpif/dump-flows` shows no corresponding entry and the counter does not grow, either packets have not arrived, or xlate failed (see `ovs-vswitchd.log`), or the `revalidator` keeps evicting them (see §9.11.10).

#### Common errors and diagnostic signals

| Symptom | Cause | How to verify |
|---------|-------|---------------|
| Output is empty even though traffic is flowing | The wrong datapath was queried, or a freshly created datapath has not seen its first miss yet | `ovs-appctl dpif/show` to see which datapath is active |
| Entry count grows uncontrollably (flow explosion) | OF rule wildcards are too narrow, each 5-tuple generates its own megaflow | Cross-check the `upcall_flow_add` rate from `coverage/show` and Part 9.15 TSS |
| `used:` is always large but packets do not grow | A stale megaflow; traffic has shifted to another entry | Run `ovs-appctl revalidator/wait` to force a revalidator pass, then measure again |

#### Workflow

1. `ovs-appctl dpif/show` to see which datapaths exist.
2. `ovs-appctl dpif/dump-flows -m | wc -l` for the baseline entry count.
3. Generate 30 to 60 seconds of load (`ping`, `iperf`).
4. Repeat step 2 to see how much the entry count grew, combined with `coverage/show` for `upcall_flow_add` over the same window.
5. `dpif/dump-flows -m | grep <field>` to filter by a specific `recirc_id`, `ct_state`, or `in_port`.

#### Source code reference

- `ofproto/ofproto-dpif.c::ofproto_unixctl_dpif_dump_flows()` registers the subcommand through `unixctl_command_register()`. The function calls `dpif_flow_dump_create()` and iterates through `dpif_flow_dump_next()`.
- `lib/dpif.c::dpif_flow_dump_*()` is the userspace abstraction.
- `lib/dpif-netlink.c` (kernel) and `lib/dpif-netdev.c` (userspace) are the two concrete backends.

Cross-link per Rule 14: §9.2 describes the megaflow structure in the kernel datapath in detail; §9.32 is a deep dive on classifier internals.

#### Version differences

- OVS 2.0 (2014): the subcommand appeared together with the `dpif` refactor. Before that, only `ovs-dpctl dump-flows` was available, running outside ovs-vswitchd and without internal state visibility.
- OVS 2.4 (2015): added `ufid` (Unique Flow Identifier) to the default output.
- OVS 2.6 (2016): added `recirc_id` and `dp_hash` fields to support stateful pipelines.
- OVS 2.8 and later (2017): full support for conntrack fields (`ct_state`, `ct_zone`, `ct_mark`, `ct_label`).

#### Real incident scenario

A multi-node OVN cluster: the tenant reported that east-west traffic between two VMs on the same Logical_Switch was randomly dropping about 3 percent after upgrading OVS from 2.13 to 2.17. `ovs-ofctl dump-flows br-int` showed the OVN pipeline rules fully present. `ovs-appctl dpif/dump-flows -m | grep <vm-mac>` returned two entries with the same match but different `actions:`, one leading to `output:5` and the other to `output:7`. Root cause: the revalidator had not yet evicted the stale megaflow after `ovn-controller` reprogrammed the pipeline; for a window of a few hundred milliseconds, two megaflows with the same match coexisted. Temporary fix: `ovs-appctl revalidator/purge`. Permanent fix: upgrade the kernel and tune `ovs-vswitchd` `other_config:revalidator-threads` appropriately (see Part 9.26).

#### Synthetic exercise

On a single-node OVS 2.17 host:

```shell
sudo ovs-vsctl --may-exist add-br br-lab
sudo ip netns add ns1 && sudo ip netns add ns2
sudo ip link add veth1 type veth peer name veth1p
sudo ip link add veth2 type veth peer name veth2p
sudo ip link set veth1p netns ns1 && sudo ip link set veth2p netns ns2
sudo ovs-vsctl add-port br-lab veth1 -- add-port br-lab veth2
sudo ip netns exec ns1 ip addr add 10.20.0.1/24 dev veth1p
sudo ip netns exec ns2 ip addr add 10.20.0.2/24 dev veth2p
sudo ip link set veth1 up; sudo ip link set veth2 up
sudo ip netns exec ns1 ip link set veth1p up
sudo ip netns exec ns2 ip link set veth2p up
sudo ip netns exec ns1 ping -c 5 10.20.0.2
sudo ovs-appctl dpif/dump-flows -m | grep "10.20.0"
```

Observation: at least one megaflow per direction (forward and reverse), with the `packets:` counter at 5 or more per direction.

#### Comparison with other systems

`dpif/dump-flows` is similar in spirit to `tc -s filter show dev <iface>` (Linux traffic control showing filters installed in the kernel with counters), or `nft list ruleset -a` for nftables. The difference: tc and nftables organise rules by interface or chain; `dpif/dump-flows` organises rules by bridge or datapath, and each entry is the dynamic result of xlate from the OpenFlow layer, not a fixed declarative rule.

### 9.11.X.2 `ovs-appctl dpif/dump-conntrack`

#### Concept

`dpif/dump-conntrack` (closely related to `dpctl/dump-conntrack`) lists the conntrack entries that OVS uses for the `ct()` action. Each entry describes a connection: original tuple (`orig`), reply tuple (`reply`), zone partition, mark, labels, and state machine (`NEW`, `ESTABLISHED`, `RELATED`, `INVALID`). The command goes through the `dpif` channel of `ovs-vswitchd` to ensure visibility into the table OVS attaches to (kernel netfilter conntrack for `dpif-netlink`, the userspace conntrack table for `dpif-netdev`).

#### Role in the architecture

OVS does not reimplement conntrack: the kernel datapath reuses Linux netfilter's `nf_conntrack` module, while the userspace datapath has its own implementation in `lib/conntrack.c`. The OpenFlow action `ct()` is the bridge: it pushes the packet through the conntrack lookup, populates the metadata `ct_state`, `ct_zone`, and so on, then returns the result to the pipeline. `dpif/dump-conntrack` is the fastest way to inspect that table from OVS's point of view, without guessing through `conntrack -L` (Linux native, which does not distinguish OVN zones).

#### Syntax and parameters

```
ovs-appctl dpif/dump-conntrack [-m] [zone=N]
```

- `-m` adds `zone`, `mark`, `labels` to each line (hidden by default).
- `zone=N` prints only entries belonging to zone N. Useful when OVN uses many per-tenant zones.

#### Real output sample

```shell
$ sudo ovs-appctl dpif/dump-conntrack -m zone=1 | head -2
tcp,orig=(src=10.1.0.5,dst=10.1.0.10,sport=50000,dport=80),
    reply=(src=10.1.0.10,dst=10.1.0.5,sport=80,dport=50000),
    zone=1,mark=0xab,labels=0,protoinfo=(state=ESTABLISHED)
udp,orig=(src=10.1.0.5,dst=10.1.0.20,sport=33000,dport=53),
    reply=(src=10.1.0.20,dst=10.1.0.5,sport=53,dport=33000),
    zone=1,mark=0,labels=0
```

Field meanings: `orig` is the direction of the first packet through OVS; `reply` is the reverse direction OVS has seen (if not yet seen, it is marked `UNREPLIED`); `protoinfo` exists only for TCP / SCTP / DCCP (protocols with a state machine); `mark` and `labels` are data OVN writes through the action `ct(commit, exec(set_field:0xab->ct_mark))`.

#### Workflow

1. Identify the zone to investigate: with OVN, `ovn-sbctl list datapath_binding | grep tunnel_key` gives the `tunnel_key` that maps to the zone (Part 13.5).
2. `ovs-appctl dpif/dump-conntrack -m zone=<N>` filters the right zone.
3. Cross-check `protoinfo=(state=...)` against the `ct_state=...` rules in `ovs-ofctl dump-flows br-int`.
4. If you see `state=INVALID` for a valid connection, suspect a mismatch between the OpenFlow flow and the conntrack truth (see Part 9.24).

#### Common errors and diagnosis signals

| Symptom | Cause | Fix |
|---------|-------|-----|
| Empty table despite traffic | The `ct()` action is never invoked (rule is wrong) | `ovs-ofctl dump-flows br-int`, check whether any rule has `ct(...)` |
| Table populated but the zone is missing | OVN compiled without the `ct_zone` field, packets fall to zone 0 by default | Check `ovn-sbctl list logical_flow`; the Logical_Switch must have proper DROP / ALLOW rules |
| `nf_conntrack table full` in dmesg | Kernel `nf_conntrack_max` is too low | `sysctl net.netfilter.nf_conntrack_max=1048576` |
| Entry exists but `state=INVALID` for a real connection | Zone collision across multiple logical contexts | OVN 24.03 and later allows setting an explicit `zone-id` per Logical_Switch |

#### Observation and verification

Pair with `ofproto/trace`:

```shell
$ sudo ovs-appctl ofproto/trace br-int 'in_port=3,tcp,ct_state=+trk+est,...'
# The trace shows the ct() action commit; afterwards:
$ sudo ovs-appctl dpif/dump-conntrack -m zone=1 | grep "10.1.0.5"
```

Comparing the two outputs reveals whether the action actually committed the right entry.

#### Source code reference

- The `dpif/dump-conntrack` unixctl handler is registered in `ofproto/ofproto-dpif.c` (verify the exact name at fix time via `grep -nE "dpif_dump_conntrack" ofproto/ofproto-dpif.c`; the surrounding registration uses `unixctl_command_register`).
- `lib/conntrack.c` provides the `conntrack_dump_*` family for the userspace implementation (`conntrack_dump_start` line 2529, `conntrack_dump_next` line 2545, `conntrack_dump_done` line 2580).
- `lib/dpif-netlink.c` bridges Netlink down to `nf_conntrack`.
- Linux side: `net/netfilter/nf_conntrack_core.c::nf_conntrack_in()` (mainline kernel reference, outside the local OVS repo).

#### Version differences

- OVS 2.5 (2016): first conntrack support together with the `ct()` action.
- OVS 2.6 (2016): zone partitioning usable from the OF pipeline.
- OVS 2.8 (2017): `ct_label` 128-bit plus ALG (FTP, TFTP).
- OVS 2.13 (2020): per-table conntrack timeout policy.

#### Synthetic exercise

Extend the lab from §9.11.X.1: install a stateful conntrack firewall:

```shell
sudo ovs-ofctl add-flow br-lab 'priority=100,ip,ct_state=-trk,actions=ct(table=0,zone=42)'
sudo ovs-ofctl add-flow br-lab 'priority=200,ip,ct_state=+trk+new,actions=ct(commit,zone=42),NORMAL'
sudo ovs-ofctl add-flow br-lab 'priority=200,ip,ct_state=+trk+est,actions=NORMAL'
sudo ip netns exec ns1 ping -c 3 10.20.0.2
sudo ovs-appctl dpif/dump-conntrack -m zone=42
```

Pass: ICMP entries show up in zone=42 (ICMP has no TCP-style state but the entry exists).

#### Comparison with other systems

Equivalent to `conntrack -L` (Linux netfilter userspace tool) for the kernel datapath, but `dpif/dump-conntrack` allows filtering by OVS-specific zone; `conntrack -L` does not distinguish OVN zones when many OVS bridges share the kernel namespace. Outside Linux, no switch system has a directly equivalent structure; Cisco ASA `show conn` is the closest, but it is opaque and does not expose the state machine for extended rules.

### 9.11.X.3 `ovs-appctl ofproto/list-tunnels`

#### Concept

`ofproto/list-tunnels` lists every active tunnel the bridge can see: source IP, destination IP, transport (UDP / IP-proto), tunnel type (`geneve`, `vxlan`, `gre`, `stt`), and identifier key (`key` or `tun_id`). Unlike `ovs-vsctl list interface type=geneve` (which lists interfaces created in the control plane), this command looks from `ofproto`: the tunnel has been fully programmed into the datapath, ARP / ND has been resolved, and it is ready to forward.

#### Role in the architecture

OVN uses Geneve tunnels to encapsulate tenant traffic between chassis. When a packet leaves chassis A for chassis B, OVS must know: which IP B has, which UDP port to use (Geneve default 6081), and which Geneve VNI maps to the tenant. `ofproto/list-tunnels` is a quick way to verify whether the inter-chassis tunnel mesh is fully built, without reading OVSDB columns one by one.

#### Syntax and parameters

```
ovs-appctl ofproto/list-tunnels [BRIDGE]
```

`BRIDGE` is optional (default lists every bridge). The command has no extra flag.

#### Real output sample

```shell
$ sudo ovs-appctl ofproto/list-tunnels br-int
port 4: 192.168.10.21 (geneve: ::->192.168.10.21, key=flow, dp port=4)
port 5: 192.168.10.22 (geneve: ::->192.168.10.22, key=flow, dp port=5)
port 6: 192.168.10.23 (vxlan: ::->192.168.10.23, key=42, dp port=6)
```

Anatomy:

- `port 4` is the OpenFlow port number in `br-int`.
- `192.168.10.21` is the remote IP (tunnel far end).
- `geneve:` is the tunnel type. Possible values: `vxlan`, `gre`, `stt`, `erspan`.
- `::->192.168.10.21` is the source-to-destination pair. `::` means the source uses the `local_ip` config or auto-selects.
- `key=flow` means the VNI (or key) is assigned dynamically from the pipeline (set via `set_field:0x...->tun_id`); `key=42` means a static key.
- `dp port=4` is the corresponding datapath port number (often the same number).

#### Operational role

This is the one-liner answer to "is the OVN mesh complete". On an N-chassis cluster, each `br-int` should have N-1 tunnels. If fewer, `ovn-controller` has not registered yet or the `Encap` row is wrong in the `Chassis` table. Cross-check with `ovn-sbctl show` to count chassis in SBDB.

#### Common errors and diagnosis

| Symptom | Cause | How to verify |
|---------|-------|---------------|
| Empty output despite the bridge having a `geneve` port | The datapath has not attached the tunnel; `ovs-vswitchd.log` typically has "tunnel ARP failed" | `ovs-appctl tnl/neigh/show` to see the ARP cache |
| Fewer tunnels than N-1 | Some chassis have not registered or were evicted | `ovn-sbctl --columns=name,encaps list chassis` |
| `key=42` but OVN needs dynamic | Tunnel manually configured wrong, overriding OVN | `ovs-vsctl get interface tunN options:key` |

#### Workflow

1. Identify the bridge to inspect: with OVN, always `br-int`.
2. `ovs-appctl ofproto/list-tunnels br-int` to count the rows.
3. Cross-check `ovn-sbctl --columns=name list chassis | wc -l` for the chassis count.
4. If short, use `ovs-appctl tnl/neigh/show` plus `ovs-appctl tnl/ports/show` (§9.11.9) to find the gap.

#### Source code reference

- The `ofproto/list-tunnels` unixctl handler is registered in `ofproto/ofproto.c` (verify the exact handler name at fix time via `grep -rn "list_tunnels" ofproto/`; the registration uses `unixctl_command_register`).
- The iteration uses the `tnl_find_*` family in `ofproto/tunnel.c`: `tnl_find` at line 550, `tnl_find_exact` at line 532, and `tnl_find_ofport` at line 518 (no separate top-level `tnl_port_iterate` function exists).
- Output struct: `struct tnl_port` defined at `ofproto/tunnel.c:55` contains `match` (src, dst, in_key, in_key_flow), `netdev`, and `dp_port`.

#### Version differences

- OVS 2.0 (2014): subcommand introduced together with the tunnel framework refactor.
- OVS 2.6 (2016): supports Geneve TLV (option metadata); output adds columns when TLVs are present.
- OVS 2.10 (2018): better integration with the userspace datapath; format stabilizes.
- OVS 2.13 and later: improved display for IPv6 underlay (you may see tunnels formatted as `::ffff:...`).

#### Real incident scenario

OpenStack cluster with five compute nodes; after a kernel upgrade (5.4 to 5.15), one compute "disappeared" from the traffic mesh: VMs on that compute could not reach VMs on the other four. `ovn-sbctl show` reported all five chassis. `ovs-vsctl list interface | grep geneve` on the broken compute showed all four tunnel ports. But `ovs-appctl ofproto/list-tunnels br-int` returned only two lines. Root cause: ARP for two of the remaining endpoints failed because the underlay routing changed and the host had not updated. `ovs-appctl tnl/neigh/show` confirmed the missing entries. Fix: add static neighbors or fix the underlay routing; afterwards `ofproto/list-tunnels` returned the full four lines.

#### Synthetic exercise

Two hosts (host A `192.168.50.1`, host B `192.168.50.2`), each with a `br-int`:

```shell
# On A:
sudo ovs-vsctl add-port br-int gnv0 -- set interface gnv0 type=geneve \
  options:remote_ip=192.168.50.2 options:key=flow
# On B: reverse with remote_ip=192.168.50.1
# Observe on A:
sudo ovs-appctl ofproto/list-tunnels br-int
# Expected: one line "geneve: ::->192.168.50.2, key=flow"
```

#### Comparison with other systems

Linux native uses `ip -d link show type geneve` to list tunnel interfaces, but that output only shows configuration, not "alive in the datapath". VMware NSX-T uses `nsxcli get logical-switch <id> tunnel-status` for the equivalent, but it requires the NSX agent. `ovs-appctl ofproto/list-tunnels` needs no agent, is fast, and reads `ofproto` state directly.

### 9.11.X.4 `ovs-appctl coverage/show`

> **Initially introduced in §9.11.1.3.** This section adds depth on operational workflow, version drift, and source-code anchors to complete the 20-axis treatment.

#### Forensic role

OVS does not ship a standard out-of-the-box Prometheus metric set; `coverage/show` is the official telemetry API to export. Every notable code-path event has a `COVERAGE_INC(name)` macro in source. When investigating an incident without a clear error log (revalidator storm, upcall churn, abnormal megaflow flush), the counter delta is the primary numerical evidence, faster than `perf` and clearer than `strace`.

#### Syntax and variants

```
ovs-appctl coverage/show
ovs-appctl coverage/read-counter <COUNTER_NAME>
```

The `read-counter` variant (see §9.11.1.4) returns a single integer, suitable for a custom exporter.

#### Output sample already in §9.11.1.3

Refer to the actual output and the counter healthy-threshold table in §9.11.1.3 (not repeated). This section completes axes that were not covered earlier.

#### Mastery workflow

1. **Baseline:** run `coverage/show` at steady state, snapshot the `total:` for 8 to 10 important counters (`hit`, `miss`, `upcall_flow_add`, `upcall_flow_del`, `revalidate`, `facet_revalidate`, `bridge_reconfigure`, `ofproto_flush`).
2. **Trigger event:** run the action (controller restart, OVSDB change, and so on).
3. **After snapshot:** run `coverage/show` again.
4. **Compute delta:** for each counter `delta = after.total - before.total`. Compare with the expected (for example a normal controller restart raises `bridge_reconfigure` by 1 and does not raise `ofproto_flush`).
5. **Plot:** for long incidents, loop `read-counter` every five seconds and plot.

#### Observation and verification

To confirm that a counter represents the event you think: raise the related module's vlog level to `dbg`, trigger the event, and read both the log and the counter. Example: suspecting `upcall_flow_add` is not increasing despite new packets arriving, run `vlog/set ofproto_dpif_upcall:file:dbg`, generate new traffic, read `ovs-vswitchd.log`, and compare the number of "add flow" log lines against the counter delta.

#### Source code reference

- `lib/coverage.c::coverage_unixctl_show()` registers the `coverage/show` subcommand at line 63.
- `lib/coverage.c::coverage_unixctl_read_counter()` registers `coverage/read-counter` at line 78.
- The macros `COVERAGE_DEFINE(name)` (`lib/coverage.h:61`) and `COVERAGE_INC(name)` (`lib/coverage.h:83`) live in the header. Each definition creates a `struct coverage_counter` (declared at `lib/coverage.h:47`, in the header rather than the .c file) and links it into a global linked list.
- `lib/coverage.c::coverage_run()` is called periodically to update the 5-second / 1-minute / 1-hour rates.

#### Version differences

- OVS 1.x: primitive counters, only a few dozen.
- OVS 2.0 (2014): refactored; the `5sec/min/hour/total` format becomes stable.
- OVS 2.6 (2016): adds counters `dpif_recv_zero_payload`, conntrack-related (`ct_match`, `ct_zone`).
- OVS 2.10 (2018): adds counters for Geneve TLV and p4-related events.
- OVS 2.17 (2022): the counter set stabilizes around 150 entries depending on the build.

#### Real incident scenario

Part 9.26 describes a "revalidator storm": a production incident with OVS 2.13 in an OpenStack cluster, where `ovs-vswitchd` CPU shot above 90% and forwarding latency rose. `coverage/show` showed the `revalidate` rate jumping from 0.3 / second to 18 / second, with `facet_revalidate` in lockstep. Root cause: the controller (Neutron OVS agent) was installing and deleting many rules with the same cookie every second. Fix: raise `revalidator_threads` from 4 to 8 and batch operations on the agent side. The `revalidate` counter returned to 0.5 / second after the fix.

#### Synthetic exercise

```shell
sudo ovs-appctl coverage/show > /tmp/cov_before.txt
# Change the bridge config:
sudo ovs-vsctl set Bridge br-lab fail_mode=secure
sudo ovs-vsctl set Bridge br-lab fail_mode=standalone
sudo ovs-appctl coverage/show > /tmp/cov_after.txt
diff /tmp/cov_before.txt /tmp/cov_after.txt | grep -E "bridge_reconfigure|ofproto_flush"
```

Expected: `bridge_reconfigure` rises by 2; `ofproto_flush` does not rise (only metadata changed, no rule re-install).

#### Comparison with other systems

Similar to `cat /proc/net/snmp` (Linux) or `netstat -s` for an overview of TCP / UDP stack events. Difference: `/proc/net/snmp` is MIB-standardized with fixed names; `coverage/show` in OVS is denser and tied to internal code paths, but lacks a cross-vendor standard. Only an engineer who reads OVS source knows what the counter `facet_changed_rule` means. For Cisco IOS, the equivalent is `show platform forward statistics`, more closed but in the same spirit of "count code-path events".

### 9.11.X.5 `ovs-appctl vlog/set`

> **Initially introduced in §9.11.1.5.** This section completes the treatment with details on the three-component syntax, runtime semantics, common errors, and source-code anchors.

#### Concept

`vlog/set` is the subcommand to adjust the log level per module and per destination without restarting `ovs-vswitchd` or `ovsdb-server`. Each OVS source `.c` file is declared with the macro `VLOG_DEFINE_THIS_MODULE(<name>)` to become an independent log module (`dpif`, `bfd`, `bond`, `ofproto`, `connmgr`, and so on). Changing the level of one module = filter the log of that module immediately, useful when debugging while the daemon is running in production.

#### Three-component syntax

```
ovs-appctl vlog/set [SPEC[:SPEC]...]
SPEC: <module>:<destination>:<level>
```

- `<module>`: module name (`dpif`, `ofproto_dpif_xlate`, and so on) or `ANY` for every module.
- `<destination>`: `console` (stderr, visible when running foreground), `syslog` (sent to `syslogd`), `file` (`/var/log/openvswitch/ovs-vswitchd.log`). Use `ANY` to cover all three.
- `<level>`: from most verbose to least: `dbg`, `info`, `warn`, `err`, `emer`, `off`. Each level includes the higher levels (setting `dbg` also writes `info`, `warn`, `err`, `emer` to the file).

Trailing components can be omitted by relying on defaults; for example `vlog/set dbg` forces every module to log at `dbg` to every destination, which floods logs and is rarely what you want.

#### Output sample and semantics

```shell
$ sudo ovs-appctl vlog/set dpif:file:dbg
$ sudo ovs-appctl vlog/list | grep '^dpif '
dpif             OFF        ERR       DBG
$ sudo tail -n 5 /var/log/openvswitch/ovs-vswitchd.log
2026-04-24T11:15:23.412Z|00042|dpif(handler1)|DBG|system@ovs-system: \
  port_query_by_number(3): pkts:1024 bytes:131072
```

Log line format: `<timestamp>|<seq>|<module>(<thread>)|<level>|<message>`. The sequence number increases per daemon process.

#### Operational role

The default log levels (`info` to file, `err` to console) are chosen so the log file does not bloat too quickly while still showing clear errors. When deep debugging is needed (inspecting the upcall path, or packets in and out of netlink), temporarily raise the relevant module to `dbg`, work, and reset:

```shell
sudo ovs-appctl vlog/set dpif:file:dbg ofproto_dpif_upcall:file:dbg
# ... reproduce the issue ...
sudo ovs-appctl vlog/set dpif:file:info ofproto_dpif_upcall:file:info
```

Forgetting to reset is a common mistake: the log file balloons to gigabytes within hours, hurting I/O.

#### Common errors and diagnosis signals

| Symptom | Cause | Fix |
|---------|-------|-----|
| `ovs-appctl: "vlog/set" command requires at least 1 argument` | Forgot the SPEC argument | Provide a SPEC in the form `module:dest:level` |
| Log does not change level even though the command reports OK | Wrong module name (case-sensitive in some versions, or missing underscore) | `ovs-appctl vlog/list` to look up the exact name |
| Log goes to `console` but is not visible | Daemon runs detached, no `console` output | Switch the destination to `syslog` or `file` |
| `/var/log/openvswitch/ovs-vswitchd.log` fills the disk overnight | Forgot to reset from `dbg` to `info` | `vlog/set ANY:file:info` then `vlog/reopen` |

#### Workflow

1. `ovs-appctl vlog/list | grep <module>` to confirm the module name.
2. `ovs-appctl vlog/set <module>:file:dbg` to enable detail.
3. Reproduce the issue for 30 to 60 seconds.
4. `tail -n 200 /var/log/openvswitch/ovs-vswitchd.log` to read the log.
5. **Mandatory:** reset `ovs-appctl vlog/set <module>:file:info` right after debugging.
6. If the log file is too large: `mv ovs-vswitchd.log ovs-vswitchd.log.old; ovs-appctl vlog/reopen` (see §9.11.1.6).

#### Source code reference

- `lib/vlog.c::vlog_unixctl_set()` registers the subcommand via `unixctl_command_register("vlog/set", ...)` at line 647.
- `lib/vlog.c::vlog_set_levels_from_string()` parses the SPEC and applies it (line 474).
- The macros `VLOG_DEFINE_THIS_MODULE` (`include/openvswitch/vlog.h:180`), `VLOG_DBG` (`include/openvswitch/vlog.h:208`), `VLOG_INFO` (`include/openvswitch/vlog.h:207`), and others live in the public header `include/openvswitch/vlog.h` (the file `lib/vlog.h` does not exist).
- Each module corresponds to a `struct vlog_module` (declared at `include/openvswitch/vlog.h:80`) with three level cells for `console / syslog / file`.

#### Version differences

- OVS 1.x: only one shared destination; syntax `vlog/set <module>:<level>` (no destination).
- OVS 2.0 (2014): three destinations split out; the `module:dest:level` format stabilizes.
- OVS 2.4 (2015): supports rate-limited logging (`VLOG_*_RL`) to avoid log floods.
- OVS 2.10 and later (2018): supports a JSON format for the `file` destination if the build is configured for it (rarely used).

#### Real incident scenario

A production OVS cluster: a bridge unexpectedly dropped ICMP traffic but passed TCP. `dpif/dump-flows` showed normal rules. Enabling `vlog/set ofproto_dpif_xlate:file:dbg` for 30 seconds revealed the line "translation failed: drop reason: action set has conflicting outputs". Root cause: the rule `actions=output:1,output:2,clone(output:3)` had conflicting flag modes in xlate. After fixing the flow rule and resetting vlog, the issue was gone. Without `vlog/set` to enable `dbg` directly at the xlate layer, the issue would have required a lab reproduction, costing many hours.

#### Synthetic exercise

```shell
sudo ovs-appctl vlog/list | grep -E 'bond|bridge|connmgr' | head -3
sudo ovs-appctl vlog/set bridge:file:dbg
sudo ovs-vsctl --may-exist add-br br-vlog-test
sudo ovs-vsctl del-br br-vlog-test
sudo tail -n 30 /var/log/openvswitch/ovs-vswitchd.log | grep '|bridge|'
sudo ovs-appctl vlog/set bridge:file:info
```

Pass: at least four `bridge|DBG` lines about create / destroy bridge; after reset, no new `DBG` line appears.

#### Comparison with other systems

The Linux kernel has `dmesg --console-level <0-7>` and `printk` ratelimit, but does not split by module as finely as OVS; the kernel only splits by subsystem via `dyndbg`. systemd-journald has `journalctl SYSLOG_IDENTIFIER=...` to filter by identifier, close to the module concept but filtering at read time, not silencing at the source. HAProxy has a similar runtime `set log-level` but only one global level, missing the three-component module / dest / level concept. OVS `vlog/set` is one of the most flexible runtime log-control APIs among userspace networking daemons.

> **Summary of 9.11.X.** These five subcommands are the mandatory mid-tier toolkit when an incident exceeds the scope of `ovs-vsctl` and `ovs-ofctl`: `dpif/dump-flows` reads the actual datapath cache, `dpif/dump-conntrack` reads the stateful firewall state, `ofproto/list-tunnels` confirms the tunnel mesh, `coverage/show` measures the heartbeat of internal events, `vlog/set` switches on X-ray mode for whichever module you need. Combined with Part 9.14 (incident response decision tree) and Part 9.26 (revalidator storm forensic), these five commands cover most day-to-day diagnostic situations.

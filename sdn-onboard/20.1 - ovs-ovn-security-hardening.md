## 20.1.7 Three security layers in OVN

Part 9.10 covered how OVS encrypts the OpenFlow channel between ovn-controller and SBDB with TLS. Part 10.2 covered how OVSDB protects the management channel with mTLS and RBAC. Those two layers protect the control and management channels, but data traffic between VMs is fully unprotected unless a third layer is configured.

The three OVN security layers correspond to three distinct planes:

**Layer 1, Control plane.** Protects the control channel between OVS datapath and controller. In an OVN deployment that is the channel between ovn-controller (on each hypervisor) and SBDB (on ovn-central). Mechanism: TLS certificate exchange via `ovs-pki`. Covered in Part 9.10.

**Layer 2, Management plane.** Protects the OVSDB channel, the RFC 7047 protocol that the CMS (Cloud Management System, for example OpenStack Neutron) uses to populate NBDB. Mechanism: mTLS (mutual TLS) with client certificate plus RBAC controlling who may write to which table. Covered in Part 10.2.

**Layer 3, Data plane.** Protects traffic between VMs / containers running on the overlay network, prevents address forgery, controls east-west access, and provides audit. Mechanism: `port_security` (anti-spoofing) plus ACL (distributed firewall) plus audit logging. Part 20.1 focuses entirely on Layer 3.

> **Key point.** The three OVN security layers are independent; only deploying all three achieves true defence in depth. Many deployments configure only Layer 3 (ACL) but skip Layers 1 and 2, exposing the control channel as plaintext TCP. An attacker can sniff OVSDB traffic to learn the topology, or inject forged OpenFlow rules if the control channel is unencrypted.

Recommended deployment order: Layer 1 to Layer 2 to Layer 3. Reason: if Layers 1 and 2 are not secured, an attacker can delete ACLs or disable port_security at Layer 3 by writing directly to NBDB through the unprotected channel.

## 20.1.8 port_security, defence against MAC and IP spoofing

### 20.1.8.1 How it works

`port_security` is a protection mechanism at the OVN logical switch port that prevents a VM from sending traffic with a MAC or IP address that is not permitted. When a logical switch port is declared with `port_security`, ovn-northd creates the logical flows in SBDB that represent the permit conditions. ovn-controller translates those logical flows into OpenFlow rules in br-int on the corresponding hypervisor.

The rules created are:

- **Ingress (from VM into OVS).** Only allow Ethernet frames with `eth.src == <permitted MAC>`. Frames with a different source MAC drop right at br-int and never proceed in the pipeline.
- **Egress (from OVS to VM).** Only allow frames with `eth.dst == <permitted MAC>`. Protects the VM from receiving traffic not intended for it.
- **ARP.** Only allow ARP request / reply with `arp.sha == <permitted MAC>` and `arp.spa == <permitted IP>`. ARP packets with other addresses drop.
- **IPv4.** Only allow IP packets with `ip4.src == <permitted IP>`. Prevents IP spoofing.

> **Key point.** `port_security` operates at the OpenFlow level on the hypervisor, not at the OVN logical layer or on the underlay. This means the defence stops malicious traffic at the source before it is encapsulated in a Geneve tunnel and sent across the physical fabric. Overhead is O(1) per port: a few extra flow entries in the OpenFlow table, with no measurable bandwidth impact.

### 20.1.8.2 Configuring port_security

```bash
# Step 1: Create the logical switch and port (if not already)
[chassis-1]# ovn-nbctl ls-add ls-prod
[chassis-1]# ovn-nbctl lsp-add ls-prod vm1-port

# Step 2: Set the permitted addresses (MAC + IP), used to populate ARP responder and DHCP
[chassis-1]# ovn-nbctl lsp-set-addresses vm1-port "fa:16:3e:11:22:33 10.1.1.10"

# Step 3: Enable port_security, only allow the declared MAC + IP
[chassis-1]# ovn-nbctl lsp-set-port-security vm1-port "fa:16:3e:11:22:33 10.1.1.10"
```

Verify the configuration after setting:

```bash
[chassis-1]# ovn-nbctl show ls-prod
    switch c1f2d4a6-9e3b-... (ls-prod)
        port vm1-port
            addresses: ["fa:16:3e:11:22:33 10.1.1.10"]
            port security: ["fa:16:3e:11:22:33 10.1.1.10"]
```

Expected outcome (doc-plausible): `port security` shows the value set. If `port security` is empty, `lsp-set-port-security` either did not run or failed.

Verify the OpenFlow rules generated on the hypervisor:

```bash
[hypervisor]# ovs-ofctl dump-flows br-int | grep "fa:16:3e:11:22:33"
```

Expected outcome (doc-plausible): flow entries that match `dl_src=fa:16:3e:11:22:33` or `dl_dst=fa:16:3e:11:22:33`, plus drop rules for non-matching MAC.

### 20.1.8.3 Defence against ARP poisoning

Without `port_security` a malicious VM can broadcast a forged ARP reply, claiming the IP of another VM (for example the gateway 10.1.1.1) maps to its own MAC, redirecting traffic through itself (a man-in-the-middle attack). This attack is fully transparent to the overlay network and does not need root on the host.

With `port_security`, OVN creates flows that check: only ARP replies with `arp.sha == fa:16:3e:11:22:33` and `arp.spa == 10.1.1.10` are allowed. Any ARP packet with a different `arp.sha` drops right at br-int and never reaches any other VM on the switch.

> **Misconception.** "Setting port_security with MAC alone is enough against ARP poisoning." Reality: if you declare only the MAC without the IP (for example `lsp-set-port-security vm1-port "fa:16:3e:11:22:33"`), OVN still allows ARP requests with any source IP in `arp.spa`. The malicious VM can still send ARP replies with the correct MAC but claiming ownership of another VM's IP. You must declare both `MAC IP` in the same string to protect both address layers.

## 20.1.9 ACL default-deny and stateful conntrack

### 20.1.9.1 The default-deny philosophy

OVN applies no ACL by default; all traffic between ports on the same logical switch is permitted unless an ACL says otherwise. This default "permit all" behaviour suits a development environment but is not acceptable in production with multiple tenants or sensitive workloads.

The correct security model is **default-deny** (also called allowlist or whitelist model): block all traffic as the starting point, then open specific permits with explicit rules. In OVN this is achieved by adding a lowest-priority ACL with `action=drop`, then placing higher-priority permit ACLs on top.

### 20.1.9.2 Stateful versus stateless: allow-related and allow

OVN supports four ACL action types:

| Action | Behaviour | When to use |
|--------|-----------|-------------|
| `allow-related` | Permits the packet plus automatically permits the reply traffic | Most application rules |
| `allow` | Permits this packet, does not track the connection | Reply traffic (`ct.est`), ICMP ping response |
| `drop` | Block silently | Default-deny |
| `reject` | Block plus send ICMP Unreachable or TCP RST | Better than `drop` for user experience |

> **Key point.** `allow-related` is the mandatory action for most application rules. If you use `allow` (stateless) for a TCP connection you must write separate rules to permit reply traffic (SYN-ACK, ACK, FIN, ...). `allow-related` delegates this to Linux conntrack: OVN automatically generates logical flows that check `ct.est` for reply packets. Result: fewer rules, fewer mistakes, and reply traffic handled at the conntrack fastpath rather than re-evaluating the ACL.

How `allow-related` works: when the first packet of a TCP connection (SYN) matches an `allow-related` rule, OVN commits the packet to conntrack (`ct.commit`). Conntrack creates an entry tracking the connection. The reply packet (SYN-ACK from the other side) is recognised by conntrack as `ct.est` (established), and OVN has a rule that automatically allows every `ct.est && !ct.inv` packet without re-evaluating the entire ACL chain.

### 20.1.9.3 Implementing default-deny with Port_Group

Port_Group lets you apply ACLs to many ports at once without repeating the configuration. This is the recommended approach in a multi-tenant environment.

```bash
# Create a Port_Group for the tenant
[chassis-1]# ovn-nbctl pg-add pg-tenant1

# Add ports to the group
[chassis-1]# ovn-nbctl pg-set-ports pg-tenant1 vm1-port vm2-port vm3-port

# Rule 1: Default-deny, block ALL IP traffic (lowest priority)
[chassis-1]# ovn-nbctl acl-add pg-tenant1 to-lport 1 "ip" drop

# Rule 2: Allow established connections (reply traffic from conntrack)
# Must be at higher priority than default-deny but lower than the permit rules
[chassis-1]# ovn-nbctl acl-add pg-tenant1 to-lport 100 \
    "ct.est && !ct.inv" allow

# Rule 3: Allow SSH from the management subnet (stateful)
[chassis-1]# ovn-nbctl acl-add pg-tenant1 to-lport 1000 \
    "ip4.src == 10.0.0.0/24 && tcp.dst == 22" allow-related

# Rule 4: Allow HTTP / HTTPS from anywhere (stateful)
[chassis-1]# ovn-nbctl acl-add pg-tenant1 to-lport 1000 \
    "tcp.dst == 80 || tcp.dst == 443" allow-related

# Rule 5: Allow ICMP echo (ping) inside the same subnet
[chassis-1]# ovn-nbctl acl-add pg-tenant1 to-lport 500 \
    "ip4.src == 10.1.1.0/24 && icmp4.type == 8" allow-related
```

List all created ACLs:

```bash
[chassis-1]# ovn-nbctl acl-list pg-tenant1
    to-lport  1000 (ip4.src == 10.0.0.0/24 && tcp.dst == 22) allow-related
    to-lport  1000 (tcp.dst == 80 || tcp.dst == 443) allow-related
    to-lport   500 (ip4.src == 10.1.1.0/24 && icmp4.type == 8) allow-related
    to-lport   100 (ct.est && !ct.inv) allow
    to-lport     1 (ip) drop
```

Expected outcome (doc-plausible): 5 rules listed in descending priority. OVN evaluates from highest to lowest priority; the packet matches the first rule, and evaluation stops.

### 20.1.9.4 A note on ct.inv and invalid connections

The rule `ct.est && !ct.inv` at priority 100 allows established connections but excludes those with invalid state. The `ct.inv` (invalid) state happens when conntrack receives a packet that does not match any tracked connection entry, typically in three situations: out-of-order packet after the conntrack entry expires, forged TCP RST from a third party, or conntrack table exhaustion under heavy load.

> **Misconception.** "A packet with `ct.est` cannot also have `ct.inv`." Reality: on some older kernels or when the conntrack entry is removed by timeout while the packet is still in the pipeline, the packet can carry both flags. Always use `ct.est && !ct.inv`, do not shorten to just `ct.est`.

## 20.1.10 ACL audit logging

### 20.1.10.1 Logging mechanism

OVN allows attaching a log to any ACL rule. When an ACL matches a packet, OVN writes a log entry to the ovs-vswitchd `acl_log` channel. On Ubuntu 22.04 with systemd this log appears in journald and the file `/var/log/ovn/ovs-vswitchd.log` (depending on configuration).

The log entry contains enough information to reconstruct the incident: UTC timestamp, ACL rule name, severity, verdict, and full packet information including source / destination MAC, source / destination IP, protocol, and ports.

### 20.1.10.2 Enabling logging on an ACL

Logging is added at ACL creation time with the `log=true` flag plus two extra parameters `name` and `severity`:

```bash
# ACL with logging, drop SSH from outside the management subnet with a warning
[chassis-1]# ovn-nbctl acl-add pg-tenant1 to-lport 800 \
    "ip4.src != 10.0.0.0/24 && tcp.dst == 22" drop \
    -- set ACL . log=true name="block-external-ssh" severity=warning

# Default-deny ACL with logging, record all blocked traffic
[chassis-1]# ovn-nbctl acl-del pg-tenant1 to-lport 1 "ip" drop
[chassis-1]# ovn-nbctl acl-add pg-tenant1 to-lport 1 "ip" drop \
    -- set ACL . log=true name="default-deny" severity=info
```

Supported logging parameters:

| Parameter | Valid values | Default |
|-----------|--------------|---------|
| `log` | `true`, `false` | `false` |
| `name` | Any string up to 64 characters | (empty) |
| `severity` | `alert`, `warning`, `notice`, `info`, `debug` | `info` |

The `name` field is the most important. Without a name the log entry cannot be distinguished from another rule when the system has many ACLs with the same action.

### 20.1.10.3 Log entry format

The log entry in ovs-vswitchd.log has this format (doc-plausible):

```
2024-10-15T14:32:17.432Z|00001|acl_log|WARN|name="block-external-ssh",
verdict=drop, severity=warning, direction=to-lport, ip, vlan_tci=0x0000,
dl_src=fa:16:3e:aa:bb:cc, dl_dst=fa:16:3e:11:22:33, nw_src=192.168.100.50,
nw_dst=10.1.1.10, nw_proto=6, nw_tos=0, nw_ttl=64, nw_frag=no,
tp_src=54321, tp_dst=22
```

Fields to focus on during incident investigation:

- `name=`: the matching ACL rule name, used to filter the log by a specific rule.
- `verdict=drop/allow`: the verdict for the packet.
- `nw_src` / `nw_dst`: source and destination IP of the offending packet.
- `tp_src` / `tp_dst`: source and destination ports (only for TCP / UDP).
- `direction=to-lport/from-lport`: ACL rule direction.

### 20.1.10.4 Centralising logs through syslog

In production, ACL logs must feed a centralised collection system (Loki, Elasticsearch, Splunk) for long-term analysis and alerting.

Step 1: Configure ovs-vswitchd to send `acl_log` to syslog:

```bash
[chassis-1]# ovs-appctl vlog/set acl_log:syslog:warning

# Verify vlog configuration
[chassis-1]# ovs-appctl vlog/list | grep acl_log
    acl_log            WARN    WARN    WARN
```

The first column is the console level, the second is the syslog level, the third is the file level.

Step 2: Configure rsyslog to forward `acl_log` entries to a log server:

```
# /etc/rsyslog.d/50-ovn-acl.conf
:msg, contains, "acl_log" @logserver.example.com:514
& stop
```

Step 3: Create an alert rule in the log aggregator when the rate of `verdict=drop` exceeds a threshold; for example, more than 100 events per minute from the same source IP is a sign of port scanning or a brute-force attack.

## 20.1.11 10-point security posture checklist

This checklist is for a quick assessment of the security level of an OVN deployment before going to production, or during a periodic audit. Each item has a concrete verification command to avoid relying on memory or possibly stale documentation.

### 20.1.11.1 Checklist

**Item 1. OVS SSL / TLS for the OpenFlow channel (control plane):**

```bash
[chassis-1]# ovs-vsctl get Open_vSwitch . ssl
```

Expected outcome: a non-empty SSL record containing `ca-cert`, `certificate`, `private-key`. If empty, the control plane is not encrypted; perform Part 9.10.

**Item 2. OVSDB manager target uses SSL (management plane):**

```bash
[chassis-1]# ovs-vsctl get Open_vSwitch . manager_options | grep target
```

Expected outcome: `target` starts with `ssl:` (for example `ssl:10.0.0.1:6640`). If you see `tcp:`, OVSDB is not encrypted; perform Part 10.2.

**Item 3. port_security on every tenant port (data plane anti-spoofing):**

```bash
[chassis-1]# ovn-nbctl list Logical_Switch_Port | grep -A1 "port.security"
```

Expected outcome: every port has a non-empty `port_security: [...]`. A port with `port_security: []` has no anti-spoofing.

**Item 4. ACL default-deny exists on every Port_Group:**

```bash
[chassis-1]# ovn-nbctl find ACL priority=1 action=drop
```

Expected outcome: at least one record with `priority: 1` and `action: drop`. None means all traffic is permitted.

**Item 5. ACLs use allow-related (not bare allow for application rules):**

```bash
[chassis-1]# ovn-nbctl find ACL action=allow | grep -v "ct.est"
```

Expected outcome: NO output (or only the legitimate rule using `ct.est`). If application rules use `allow` instead of `allow-related`, the ACL is stateless and lacks reply-traffic protection.

**Item 6. ovn-northd connects to SBDB via SSL:**

```bash
[ovn-central]# ps aux | grep ovn-northd | grep -o "\-\-ovnsb-db=ssl:[^ ]*"
```

Expected outcome: `--ovnsb-db=ssl:127.0.0.1:6642` or similar. If you see `ptcp:` or `tcp:`, northd uses an unencrypted connection.

**Item 7. Certificate has at least 30 days until expiry:**

```bash
[chassis-1]# openssl x509 -in /etc/openvswitch/cert.pem -noout -checkend 2592000
```

Expected outcome: `Certificate will not expire` (2,592,000 seconds equals 30 days). If it reports `expire`, the certificate must be renewed. Install a periodic cron job that checks and warns early.

**Item 8. Localnet ports are protected (ports into the physical network):**

```bash
[chassis-1]# ovn-nbctl find Logical_Switch_Port type=localnet
```

Every localnet port shown in the output must have `port_security` or be in a Port_Group with an ACL. An unprotected localnet port is a direct path to the physical network; any VM on that switch can send traffic out with any source address.

**Item 9. ACL logging is enabled for drop rules:**

```bash
[chassis-1]# ovn-nbctl find ACL action=drop | grep "log.*false"
```

Expected outcome: NO output (every drop ACL has `log: true`). Otherwise violations occur without being recorded, losing investigation capability.

**Item 10. No manual flows bypassing the OVN pipeline:**

```bash
[hypervisor]# ovs-ofctl dump-flows br-int table=0 | grep "cookie=0x0"
```

Expected outcome: NO output. A flow with `cookie=0x0` is a flow added manually outside OVN; such a flow can bypass port_security, ACL, and the entire OVN security pipeline. If any are found, investigate the source before deleting.

### 20.1.11.2 Scoring scale

| Score | Security level | Recommendation |
|-------|----------------|----------------|
| 10/10 | Good | Maintain plus quarterly audit |
| 8 to 9/10 | Acceptable | Fix missing items within 30 days |
| 6 to 7/10 | Improve immediately | Fix within 2 weeks; do not onboard more tenants |
| Less than 6/10 | Unacceptable | Fix before adding any further workload |

Items 1 and 2 (control / management plane encryption) are the minimum; if both are missing, every other item can be remotely disabled by an attacker with management network access.

## Key points to remember

First, OVN security is defence in depth across three independent layers: control plane (TLS), management plane (OVSDB RBAC), and data plane (`port_security` plus ACL). Implementing only one or two layers is insufficient; the missing layer becomes the entry point through which an attacker disables the others.

Second, `port_security` is the first line of defence at the data plane. It blocks MAC and IP spoofing right at br-int on the hypervisor, before malicious traffic enters the Geneve tunnel or spreads on the overlay. The overhead is near zero compared with the protection gained.

Third, the default-deny model with `allow-related` is the standard for any tenant with security requirements. Stateless `allow` creates reply-traffic gaps and demands more rules. The rule `ct.est && !ct.inv` is mandatory to permit reply traffic after `allow-related` commits the new connection to conntrack.

Fourth, the `name=` field in ACL logging is decisive for investigation capability. A log entry without a name cannot be distinguished when the system has dozens of ACL rules; spending 10 seconds to add `name=` at rule creation saves hours during investigation.

## 20.1.12 Audit trail architecture, 4-layer who / what / when / where

Section 20.1.10 introduced ACL logging at the data plane. A production-grade OVN deployment needs a multi-layer audit trail in a 4-tier architecture to answer all four forensic questions: **who** (which tenant or user did it), **what** (what action), **when** (the exact time), and **where** (which chassis or LSP).

### 20.1.12.1 Layer 1, NBDB change audit (management-plane intent)

Every change at the intent layer (CMS writing to NBDB) must be captured. This is the main "who plus what" layer: it records which tenant requests which change in Logical_Switch / ACL / NAT / LB.

```bash
# Monitor NBDB transactions in real time, audit channel
[central]$ sudo ovsdb-client monitor OVN_Northbound ALL --format=json \
    | tee /var/log/ovn/nbdb-audit-$(date +%F).jsonl

# Filter ACL changes specifically
[central]$ sudo ovsdb-client monitor OVN_Northbound ACL \
    | grep -E "^row|direction|priority|match|action"
```

**Event schema** (doc-plausible):

```json
{
  "OVN_Northbound": {
    "ACL": {
      "<uuid>": {
        "new": {
          "direction": "to-lport",
          "priority": 1000,
          "match": "tcp.dst == 22",
          "action": "allow-related",
          "log": true,
          "name": "allow-ssh"
        },
        "old": null
      }
    }
  }
}
```

**Recommended retention policy:**

| Compliance framework | NBDB audit retention | Rationale |
|----------------------|----------------------|-----------|
| Generic (best practice) | 90 days rolling | Sufficient for internal investigation |
| PCI-DSS | 1 year minimum | Quarterly compliance review |
| HIPAA | 6 years | Healthcare retention standard |
| Financial regulation (SOX) | 7 years | Long-term audit requirement |

**Log integrity:** the NBDB audit log must be append-only. Use the file attribute `chattr +a` or a remote log server:

```bash
# Make the log file append-only (root cannot edit, only append)
[central]# chattr +a /var/log/ovn/nbdb-audit-2026-04-24.jsonl

# Verify
[central]# lsattr /var/log/ovn/nbdb-audit-*.jsonl
-----a--------e----- /var/log/ovn/nbdb-audit-2026-04-24.jsonl
```

### 20.1.12.2 Layer 2, SBDB translation audit (control-plane compile)

`ovn-northd` compiles NBDB to SBDB Logical_Flow. Auditing this layer lets a forensic investigation answer "when ACL X was added, which N logical flows did northd produce?" and detect compile bugs or unexpected expansion.

```bash
# Monitor SBDB Logical_Flow changes in real time
[central]$ sudo ovsdb-client monitor OVN_Southbound Logical_Flow \
    | tee /var/log/ovn/sbdb-lflow-audit-$(date +%F).jsonl

# Correlate with the nb_cfg timestamp
[central]$ ovn-sbctl get SB_Global . nb_cfg_timestamp
1714069801245
```

**Cross-correlate with Layer 1:** an NBDB change at T results in an SBDB update at T plus compile_time. Typical 50 to 500 ms delay; if more than 5 seconds, northd lag (see Part 20.2 §20.2.6 status check).

### 20.1.12.3 Layer 3, OpenFlow rule audit (physical dataplane install)

`ovn-controller` on each chassis installs OpenFlow rules into `br-int`. Auditing this layer records which chassis installs which rule at which time.

```bash
# Monitor OpenFlow rule add / delete events
[chassis-1]# sudo ovs-ofctl monitor br-int watch: \
    | tee -a /var/log/ovn/openflow-audit-$(date +%F).log

# Baseline snapshot (saved as reference)
[chassis-1]# sudo ovs-ofctl dump-flows br-int --rsort=priority \
    > /var/log/ovn/openflow-baseline-$(date +%F).txt
```

**Anomaly-detection pattern:**

```bash
# Hourly diff of the baseline, detects manual flow bypass (§20.1.11 Item 10)
[chassis-1]# diff /var/log/ovn/openflow-baseline-yesterday.txt \
                  /var/log/ovn/openflow-baseline-today.txt \
    | grep "cookie=0x0"
# Non-empty output means flows were added manually, NOT through the OVN pipeline
```

### 20.1.12.4 Layer 4, conntrack flow audit (session telemetry)

Every stateful connection passing through an OVN ACL `allow-related` is tracked in conntrack. Auditing this layer fits compliance requirements: it records who connected to whom and when.

```bash
# Enable conntrack event logging
[chassis-1]# sudo sysctl -w net.netfilter.nf_conntrack_log_invalid=1
[chassis-1]# sudo conntrack -E -e NEW,UPDATE,DESTROY \
    --format-output timestamp,src,dst,sport,dport,proto,zone \
    > /var/log/ovn/conntrack-audit-$(date +%F).log &
```

**Output format** (synthetic illustration):

```
[2026-04-24 14:32:17.432] NEW tcp src=10.1.1.10 dst=10.2.2.20 sport=52341 dport=22 zone=1012
[2026-04-24 14:32:17.435] UPDATE tcp src=10.1.1.10 dst=10.2.2.20 sport=52341 dport=22 zone=1012 state=ESTABLISHED
[2026-04-24 14:47:03.891] DESTROY tcp src=10.1.1.10 dst=10.2.2.20 sport=52341 dport=22 zone=1012
```

**Compliance mapping:**

- **GDPR Art. 30 (Records of Processing).** A NEW conntrack entry equals processing-activity start. DESTROY equals end. The 5-tuple plus zone is sufficient for tenant attribution.
- **PCI-DSS 10.2.2.** Track access to cardholder data. Filtering the conntrack log with `dport in {443, 8443}` (web / API) gives the audit trail of CDE access.

### 20.1.12.5 4-layer correlation runbook

Forensic scenario: "User X reports SSH failures since 14:30. Investigate what happened."

**Step 1, Layer 1 check NBDB change:**

```bash
[central]$ grep -A 3 "14:2[0-9]:\|14:3[0-5]:" /var/log/ovn/nbdb-audit-2026-04-24.jsonl \
    | grep -E "ACL|action.*drop"
```

At timestamp 14:28 a new ACL was added blocking TCP 22 from external. Hypothesis: NBDB change caused SSH failures.

**Step 2, Layer 2 check SBDB compile:**

```bash
[central]$ grep -A 2 "14:2[89]:" /var/log/ovn/sbdb-lflow-audit-2026-04-24.jsonl \
    | grep "ls_in_acl"
```

Confirm: SBDB Logical_Flow synced with the NBDB ACL at 14:28:03.

**Step 3, Layer 3 check OpenFlow install:**

```bash
[chassis-1]# grep "14:28:" /var/log/ovn/openflow-audit-2026-04-24.log \
    | grep "tp_dst=22"
```

Confirm: chassis-1 installed the new drop flow at 14:28:05.

**Step 4, Layer 4 check conntrack impact:**

```bash
[chassis-1]# grep "14:3[0-5].*dport=22.*INVALID" /var/log/ovn/conntrack-audit-2026-04-24.log
```

Confirm: user X's SSH attempts 14:30 to 14:35 are all INVALID state (SYN dropped at OVN).

**Conclusion:** the ACL change at NBDB timestamp 14:28 triggered the cascade through SBDB, OpenFlow, and conntrack. Root cause is the NBDB change. Actor is the CMS user or automation tool. Attribution is via the CMS audit log (a layer outside OVN).

## 20.1.13 port_security forensic investigation

When ARP poisoning or a MAC spoofing attempt is detected (ACL log plus Layer 4), the operator must collect evidence for attribution and prosecution. Section 20.1.13 is an investigation playbook dedicated to this scenario.

### 20.1.13.1 Detection signature

**Signature 1, ACL log `name="port_security"` fires:**

```bash
[chassis-1]# grep "acl_log" /var/log/ovn/ovs-vswitchd.log \
    | grep "name=\"port_security\"" | tail -20
```

Output entries with `verdict=drop` from OpenFlow rules generated by port_security. The `dl_src` in the entry is the attacker's attempted MAC.

**Signature 2, conntrack has no matching NEW entry for the SYN packet:**

```bash
# Victim receives SYN but replies RST (because the IP does not belong to the victim VM)
[chassis-1]# sudo conntrack -E -e NEW | grep "state=SYN_RECV.*dst=<victim-IP>"
# At the same time Layer 4 has no ESTABLISHED connection reaching the victim
```

### 20.1.13.2 5-step evidence collection

**Step 1, snapshot the ACL log window:**

```bash
# Take a snapshot of the ACL log 30 minutes before / after the detection time
[chassis-1]# sudo journalctl -u ovs-vswitchd --since "14:00" --until "15:00" \
    > /tmp/evidence-acl-$(date +%F-%H%M).log
```

**Step 2, snapshot OVS flow state:**

```bash
[chassis-1]# sudo ovs-ofctl dump-flows br-int > /tmp/evidence-flows-$(date +%F-%H%M).txt
[chassis-1]# sudo ovs-appctl fdb/show br-int > /tmp/evidence-fdb-$(date +%F-%H%M).txt
[chassis-1]# sudo ovs-dpctl dump-flows > /tmp/evidence-dpctl-$(date +%F-%H%M).txt
```

**Step 3, snapshot conntrack state:**

```bash
[chassis-1]# sudo conntrack -L -z > /tmp/evidence-conntrack-$(date +%F-%H%M).txt
```

**Step 4, snapshot a packet capture (pcap):**

```bash
# Mirror port from the suspect tap to a dedicated capture interface
[chassis-1]# sudo ovs-vsctl \
    -- --id=@m create Mirror name=forensic-mirror \
         select-src-port=$(sudo ovs-vsctl get Port <suspect-tap> _uuid) \
         select-dst-port=$(sudo ovs-vsctl get Port <suspect-tap> _uuid) \
         output-port=@cap \
    -- --id=@cap get Port forensic-tap \
    -- set Bridge br-int mirrors=@m

# Capture 5 minutes of evidence
[chassis-1]# sudo tcpdump -i forensic-tap -nn -w /tmp/evidence-pcap-$(date +%F-%H%M).pcap &
sleep 300
[chassis-1]# sudo pkill -f "tcpdump.*forensic-tap"
```

**Step 5, preserve evidence integrity:**

```bash
# Hash plus timestamp plus write-protect
[chassis-1]# cd /tmp
[chassis-1]# sha256sum evidence-*-*.* > evidence-manifest.sha256
[chassis-1]# tar czf /secure-storage/evidence-$(date +%F-%H%M).tar.gz evidence-*
[chassis-1]# chattr +i /secure-storage/evidence-*.tar.gz      # immutable flag

# Chain of custody log
[chassis-1]# echo "$(date -u): evidence collected by $(whoami), hash: $(sha256sum /secure-storage/evidence-*.tar.gz)" \
    >> /secure-storage/chain-of-custody.log
```

### 20.1.13.3 Attribution workflow

**Who sent the spoofed packet?** Correlate:

1. **MAC in the ACL log `dl_src`**, the attacker's MAC inside OVN.
2. **OVN Port_Binding**, which chassis claims this MAC:

```bash
[central]$ ovn-sbctl --bare --columns=logical_port,chassis,mac \
    find Port_Binding mac="*<attacker-MAC>*"
```

3. **OVS interface external_ids**, iface-id mapping back to the LSP:

```bash
[chassis-X]$ sudo ovs-vsctl --columns=name,external_ids \
    find Interface external_ids:attached-mac="<attacker-MAC>"
```

4. **CMS tenant owner**: LSP UUID to VM UUID to tenant (from the CMS database, not OVN).

5. **User identity**: tenant to user (from IAM, for example Keystone).

### 20.1.13.4 Root cause classification

| Pattern | Root cause | Severity | Action |
|---------|-----------|----------|--------|
| Single isolated spoofed packet | VM misconfiguration (stale IP) | Low | Notify the tenant |
| Periodic ARP reply spoofed with the same MAC | Automated ARP-spoofing tool | **High** | Isolate the VM, deep forensic |
| Random MACs from the same LSP | Compromised VM rootkit | **Critical** | Quarantine the VM and audit the host |
| MAC flood (over 100 unique MACs in 60 s) | MAC flooding attack | **Critical** | Immediate fence and capture |

## 20.1.14 OVSDB RBAC deep dive, role-based access control

Part 10.2 introduced mTLS plus basic RBAC. Section 20.1.14 dives into the RBAC_Role / RBAC_Permission schema plus role-assignment patterns.

### 20.1.14.1 RBAC schema

OVSDB RBAC has three tables (OVN 22.03 baseline):

```shell
[central]$ sudo ovsdb-client list-tables OVN_Southbound | grep -i rbac
RBAC_Role
RBAC_Permission

[central]$ sudo ovsdb-client dump OVN_Southbound RBAC_Role

name                      permissions
------------------------- --------------------------------
ovn-controller            [Chassis, Chassis_Private,
                           Encap, Port_Binding]
ovn-controller-limited    [Chassis_Private]
```

**RBAC_Role:**
- `name`: identifier (for example `ovn-controller`).
- `permissions`: map from table name to RBAC_Permission UUID.

**RBAC_Permission:**
- `table`: which table this permission applies to.
- `authorization`: column-level condition. The format `<col1>=<val>, <col2>=<val>` means "only rows where this column matches may be modified".
- `insert_delete`: true / false, INSERT / DELETE permission.
- `update`: list of columns allowed for UPDATE.

### 20.1.14.2 Chassis-specific permission

Pattern: `ovn-controller` on chassis A is only allowed to UPDATE rows where `chassis_name=A` in Chassis_Private:

```shell
[central]$ sudo ovsdb-client dump OVN_Southbound RBAC_Permission \
    | grep -A 5 "table=Chassis_Private"

table: Chassis_Private
authorization: "chassis_name"     # the chassis only updates rows that match its own chassis_name
insert_delete: true
update: [nb_cfg, nb_cfg_timestamp, external_ids]
```

Each chassis authenticates with a certificate CN=<chassis-name>. The OVSDB server maps CN to the chassis_name field and enforces row-level access.

### 20.1.14.3 Setting up RBAC for a new chassis role

```shell
# 1. Create RBAC_Permission for Chassis_Private access
[central]# uuid=$(sudo ovsdb-client transact OVN_Southbound \
    '["OVN_Southbound",
      {"op":"insert","table":"RBAC_Permission",
       "row":{"table":"Chassis_Private",
              "authorization":"chassis_name",
              "insert_delete":true,
              "update":["nb_cfg","nb_cfg_timestamp","external_ids"]},
       "uuid-name":"perm1"}]')

# 2. Create RBAC_Role "new-chassis-role"
[central]# sudo ovsdb-client transact OVN_Southbound \
    '["OVN_Southbound",
      {"op":"insert","table":"RBAC_Role",
       "row":{"name":"new-chassis-role",
              "permissions":{"map":[["Chassis_Private","'$uuid'"]]}}}]'

# 3. Attach the role to the manager connection
[central]# sudo ovn-sbctl set-connection \
    role=new-chassis-role ptcp:6642
```

### 20.1.14.4 RBAC audit trail

Every RBAC rejection is logged:

```bash
[central]$ sudo journalctl -u ovsdb-server --since "1 hour ago" \
    | grep -iE "rbac|denied|unauthorized"
```

Output pattern (doc-plausible):

```
2026-04-24T14:32:17Z ovsdb-server: RBAC: role "ovn-controller",
  table "Chassis", row chassis_name="attacker-ch",
  authenticated CN="chassis-hv-01", DENIED update
```

This entry is direct attribution: chassis-hv-01 attempted to write to the Chassis row of attacker-ch (a bug or malicious activity). Correlate with Layer 1 audit to identify the CMS or automation tool that triggered it.

## 20.1.15 mTLS plus certificate rotation (zero-downtime)

### 20.1.15.1 Certificate lifecycle

An OVS / OVN deployment has 4 cert classes:

| Cert class | Holder | Validity | Rotation cadence |
|------------|--------|----------|-------------------|
| Root CA | ovn-central | 10 years | Once every 5 years |
| Intermediate CA | ovn-central | 5 years | Once every 2 to 3 years |
| Chassis cert | each chassis | 1 year | Annual automatic |
| CMS client cert | Neutron / K8s | 1 year | Annual automatic |

### 20.1.15.2 Zero-downtime CA rotation procedure

When rotating the CA, every chassis plus CMS must trust both old and new CA during the transition window. The procedure is 5 steps:

```bash
# Step 1: Generate the new CA
[central]# ovs-pki init --dir=/var/lib/ovs-pki-new --force

# Step 2: Cross-sign the intermediate (new CA certifies the old CA public key)
[central]# openssl x509 -x509toreq -in /var/lib/ovs-pki/cacert.pem -signkey /var/lib/ovs-pki/private/cakey.pem -out /tmp/old-ca.csr
[central]# openssl x509 -req -in /tmp/old-ca.csr -CA /var/lib/ovs-pki-new/cacert.pem -CAkey /var/lib/ovs-pki-new/private/cakey.pem -out /tmp/old-ca-signed-by-new.pem -days 365

# Step 3: Combined trust bundle (old plus new CA)
[central]# cat /var/lib/ovs-pki/cacert.pem /var/lib/ovs-pki-new/cacert.pem \
    > /etc/openvswitch/ca-bundle.pem

# Step 4: Deploy the bundle to every chassis through configuration management
[central]# for chassis in $(ovn-sbctl --bare --columns=name list Chassis); do
    scp /etc/openvswitch/ca-bundle.pem $chassis:/etc/openvswitch/
    ssh $chassis "sudo ovs-vsctl set-ssl /etc/openvswitch/privkey.pem \
        /etc/openvswitch/cert.pem /etc/openvswitch/ca-bundle.pem"
done

# Step 5: Issue new chassis certs from the new CA, rotate per chassis
[central]# for chassis in $(ovn-sbctl --bare --columns=name list Chassis); do
    ovs-pki --dir=/var/lib/ovs-pki-new req-sign switch-cert-$chassis
    # Deploy new cert to the chassis...
done

# Step 6 (after every chassis is rotated, typical 1 to 2 weeks): remove the old CA
[central]# cp /var/lib/ovs-pki-new/cacert.pem /etc/openvswitch/cacert.pem
# Remove the bundle, use the new CA only
```

### 20.1.15.3 Monitor cert expiry

```bash
# Daily cron, alert 30 days before expiry
[chassis-1]# cat > /etc/cron.daily/cert-expiry-check <<'EOF'
#!/bin/bash
if ! openssl x509 -in /etc/openvswitch/cert.pem -noout -checkend 2592000; then
    echo "WARN: cert expires within 30 days" | \
        mail -s "OVS cert expiry alert $(hostname)" ops@example.com
fi
EOF
[chassis-1]# chmod +x /etc/cron.daily/cert-expiry-check
```

### 20.1.15.4 Rollback procedure

If the new CA rollout fails (chassis cannot connect), rollback:

```bash
# Restore the old CA-only configuration
[central]# cp /var/lib/ovs-pki/cacert.pem /etc/openvswitch/cacert.pem
[central]# ovs-vsctl set-ssl /etc/openvswitch/privkey.pem /etc/openvswitch/cert.pem /etc/openvswitch/cacert.pem

# Trigger reconnection to the chassis
[central]# ovn-nbctl set-connection ptcp:6641  # Re-apply the connection
```

## 20.1.16 Security incident response playbook

### 20.1.16.1 5-step response framework

When a security incident is detected (ARP poisoning, MAC spoofing, DDoS, unauthorised access), apply the 5-step framework:

**Step 1, Contain (5 to 15 minutes):**

```bash
# Isolate the suspect VM immediately (disable the LSP)
[central]# ovn-nbctl lsp-set-enabled <suspect-lsp> false

# Confirm isolation: the VM no longer forwards traffic
[chassis-1]# sudo ovs-ofctl dump-flows br-int | grep "<suspect-LSP-ofport>"
```

**Step 2, Preserve evidence (15 to 60 minutes):** apply the 5-step evidence collection of §20.1.13.2.

**Step 3, Eradicate (1 to 4 hours):** remove malware, close the vulnerability, patch the hypervisor.

**Step 4, Recover (1 to 24 hours):** re-enable affected services, verify clean state.

**Step 5, Postmortem (1 to 7 days):** document timeline plus root cause plus lessons learned.

```bash
[chassis-1]$ awk -F'[,=]' '/acl_log/ {
```

```bash

}' /tmp/acl-sample.log | sort | uniq -c | sort -rn | head -5
```

## 20.1.18 Capstone POE, compliance audit scenario

**Scenario:** an external regulator (PCI-DSS auditor) requests proof of cardholder data environment (CDE) isolation in the OVN cluster. Requirement: prove that tenant A (processes credit cards) and tenant B (marketing analytics) cannot cross-communicate at any network layer.

**Predict:** what is the minimum evidence to collect?

**Observe (reasoning):**

**1.** NBDB configuration intent proof:

```bash
[central]$ ovn-nbctl show ls-cde-tenant-A > /evidence/01-nbdb-cde.txt
[central]$ ovn-nbctl show ls-marketing-tenant-B > /evidence/02-nbdb-marketing.txt
[central]$ ovn-nbctl acl-list pg-cde-tenant-A > /evidence/03-acl-cde.txt
```

Expected: ACL `from-lport 1 "ip" drop` default-deny plus `allow-related` only for the internal CDE subnet.

**2.** Data plane enforcement proof:

```bash
# Simulate a cross-tenant attempt
[chassis-1]$ ovn-trace ls-cde-tenant-A \
    'inport == "lsp_cde_vm1" && ip4.src == 10.CDE.0.10 && ip4.dst == 10.MKT.0.10 &&
     tcp.dst == 80' > /evidence/04-ovn-trace-cde-to-mkt.txt
```

Expected output: `drop;` at the ACL default-deny, no cross-tenant route.

**3.** Audit trail proof:

```bash
[central]$ tar czf /evidence/05-audit-logs.tar.gz \
    /var/log/ovn/nbdb-audit-*.jsonl \
    /var/log/ovn/sbdb-lflow-audit-*.jsonl \
    /var/log/ovn/openflow-audit-*.log \
    /var/log/ovn/conntrack-audit-*.log
```

Retention at least 1 year per PCI-DSS 10.7.

**4.** Conntrack evidence: no cross-tenant session:

```bash
# Query the conntrack log for entries where src is CDE and dst is non-CDE
[chassis-1]$ grep -E "src=10.CDE.*dst=(10.MKT|10.EXT)" \
    /var/log/ovn/conntrack-audit-*.log > /evidence/06-crosstenant-sessions.txt
```

Expected: empty file or 0 entries. Non-empty equals isolation breach.

**5.** RBAC plus mTLS proof:

```bash
[central]$ sudo ovs-vsctl get Open_vSwitch . ssl > /evidence/07-ssl-config.txt
[central]$ sudo ovsdb-client dump OVN_Southbound RBAC_Role > /evidence/08-rbac.txt
```

Certificate expiry plus role-based access without wildcards.

**6.** Cert rotation history:

```bash
[central]$ ls -la /var/lib/ovs-pki/*.pem \
    | awk '{print $6, $7, $NF}' > /evidence/09-cert-history.txt
```

Annual rotation record.

**Explain:** the evidence package is sufficient for the auditor to verify:

- **Logical segmentation (NBDB).** ls-cde plus ls-marketing isolated, ACL default-deny, items 1 and 3.
- **Runtime enforcement (data plane).** ovn-trace proves drop plus conntrack no cross-session, items 2 and 4.
- **Audit trail retention.** 1-year log aggregation, item 5.
- **Access control.** mTLS plus RBAC, items 7 and 8.
- **Cryptographic hygiene.** Annual rotation, item 9.

**Finish:** the principle is that **compliance is not an event but an ongoing process**. Evidence collection must be automated (daily cron) so that when an audit request arrives, the evidence is already on hand. Similar to Part 20.3 §20.3.10 plus Part 20.4 §20.4.10 daily audit script.

Recommendation for compliance framework integration: run §20.1.12 4-layer audit continuously plus alert rules from §20.1.17 plus GE1 quarterly plus GE2 monthly plus GE3 annually. Test the compliance posture score through the §20.1.11 checklist monthly.

## 20.X Anatomy Template A, ACL audit plus port_security output (v3.2 P4 standardise)

**Why run it:** compliance audit, verify port_security enforced plus ACL default-deny active plus audit trail enabled before declaring security posture.

**Multi-command audit combo:**

```shell
# 1. port_security status
$ sudo ovn-nbctl --columns=name,port_security,addresses list Logical_Switch_Port lsp_web_01
name              : "lsp_web_01"
port_security     : ["aa:bb:cc:00:00:01 10.0.0.5"]
addresses         : ["aa:bb:cc:00:00:01 10.0.0.5"]

# 2. ACL default-deny verify
$ sudo ovn-nbctl acl-list ls-tenant | grep "drop"
from-lport    1 (ip) drop

# 3. Audit log enable
$ sudo ovn-nbctl --columns=name,log,severity,meter list ACL
name    : "acl-log-critical"
log     : true
severity: "info"
meter   : ["meter-acl-rate-limit"]
```

**9-attribute security anatomy:**

| Field | Healthy value | Red flag |
|-------|---------------|----------|
| `Logical_Switch_Port.port_security` | Non-empty plus matches addresses | Empty equals anti-spoofing disabled |
| `port_security` format | `"<MAC> <IP>"` | MAC-only equals IP spoofing allowed |
| ACL default-deny priority | 1 (lowest) | Over 1 means allow rules override the default |
| ACL `log` for critical rules | `true` for drop | `false` equals no audit trail |
| ACL `severity` | `info` / `warning` / `alert` | `[]` when log=true equals default severity |
| ACL `meter` reference | meter name | `[]` plus log=true plus DDoS equals log storm |
| RBAC_Role presence | Role per tenant / admin | Missing equals open OVSDB access |
| Chassis cert chain | Valid plus CA chain | Self-signed equals not scalable |
| TLS protocol version | 1.2+ | Below 1.2 equals deprecated |

**Failure modes:**

- `port_security=[]` plus VM network issues: anti-spoofing disabled. Fix: `lsp-set-port-security` with a MAC + IP pair.
- ACL default-deny priority higher than allow rule: allow never hits. Fix: priority 1 for default-deny.
- ACL log=true without a meter: DDoS equals log storm filling disk. Fix: add a meter rate-limit.
- TLS 1.0 / 1.1 negotiated: legacy client. Fix: `ovs-vsctl set Open_vSwitch . other_config:ssl-protocols=TLSv1.2+`.

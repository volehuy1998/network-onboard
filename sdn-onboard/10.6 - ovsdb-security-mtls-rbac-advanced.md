## 10.6.1 Mô hình bảo mật OVSDB server-side

OVSDB có ba lớp bảo mật độc lập, cần được cấu hình đồng thời cho môi trường production:

**Lớp 1 — Bảo mật vận chuyển (mTLS):** tất cả kết nối giữa ovsdb-server và client (ovn-controller, ovn-northd, công cụ CLI) được mã hóa và xác thực qua TLS 1.2+ với xác thực lẫn nhau (client cert + server cert). Ngăn chặn:
- Nghe lén (đọc dữ liệu traffic).
- Tấn công man-in-the-middle (giả mạo server).
- Client trái phép (cần cert từ CA tin cậy).

**Lớp 2, Ủy quyền (RBAC):** sau khi client xác thực thành công, RBAC quyết định client được ghi/đọc table/column nào. Ngăn chặn:
- Di chuyển ngang (chassis A không được ghi Nút mạng (Chassis) record của chassis B).
- Nâng cấp đặc quyền (ovn-controller không được modify SB_Global).
- Cô lập tenant (tenant A không được đọc Logical Flow tenant B trong setup multi-tenant).

**Lớp 3 — Ghi log audit:** mọi transaction được log với identity client (CN của cert), timestamp, operation. Ngăn chặn:
- Lạm dụng nội bộ (phát hiện admin lạm quyền).
- Pháp y sau khi bị xâm nhập (ai đã modify gì khi cert bị lộ).

Cấu hình đầy đủ cả ba lớp này mới đủ chuẩn môi trường production. Thực tế nhiều triển khai chỉ có Lớp 1 (mTLS), thiếu RBAC làm hệ thống dễ bị tấn công nội bộ.

## 10.6.2 Thiết lập mTLS cho cluster OVSDB

Quy trình chuẩn (giả định cụm 3-node Raft cho OVN Southbound):

**Bước 1 — Tạo Root CA** (riêng cho OVSDB, không dùng CA hệ thống):
```
[ca-host]# ovs-pki init --dir /etc/ovsdb-ca
[ca-host]# ls /etc/ovsdb-ca/
cacert.pem  ca-req.pem  rootCA.key  rootCA.cert  cnf
```

**Bước 2 — Sign cert server cho mỗi node Raft:**
```
[ca-host]# ovs-pki --dir /etc/ovsdb-ca req+sign sb-node1 switch
[ca-host]# scp sb-node1-cert.pem sb-node1-privkey.pem root@sb-node1:/etc/ovsdb-ssl/
# Tương tự cho sb-node2, sb-node3
```

Mỗi node có cert riêng với CN = hostname (ví dụ `sb-node1.internal`).

**Bước 3 — Sign cert client cho mỗi chassis:**
```
[ca-host]# for i in $(seq 1 100); do
[ca-host]>   ovs-pki --dir /etc/ovsdb-ca req+sign chassis-$i switch
[ca-host]> done
```

CN của cert client = UUID chassis (hoặc hostname chassis). Đây là identity RBAC sẽ dùng.

**Bước 4 — Cấu hình ovsdb-server dùng TLS:**
```
[sb-node1]# ovsdb-server --remote=pssl:6642 \
    --private-key=/etc/ovsdb-ssl/sb-node1-privkey.pem \
    --certificate=/etc/ovsdb-ssl/sb-node1-cert.pem \
    --ca-cert=/etc/ovsdb-ssl/cacert.pem \
    /var/lib/ovn/ovnsb_db.db
```

Cờ `pssl:` (passive SSL) báo ovsdb-server listen với TLS thay vì plaintext TCP. CA cert được kiểm chứng với mọi kết nối client.

**Bước 5 — Cấu hình client (ovn-controller) dùng cert:**
```
[chassis-1]# ovs-vsctl set Open_vSwitch . \
    external_ids:ovn-remote="ssl:sb-node1:6642,ssl:sb-node2:6642,ssl:sb-node3:6642"
[chassis-1]# ovs-vsctl set Open_vSwitch . \
    other_config:ovn-remote-probe-interval=60000
[chassis-1]# cp /etc/ovsdb-ssl/chassis-1-*.pem /etc/ovn/
```

Khi ovn-controller kết nối tới URL ssl:, nó gửi cert client `chassis-1-cert.pem` + ký bằng `chassis-1-privkey.pem`. Server kiểm chứng cert qua CA, biết client là "chassis-1".

## 10.6.3 Xoay cert không downtime

Cert X.509 có thời hạn (thường 1 năm). Khi cert gần hết hạn, phải xoay, thay thế bằng cert mới với cùng CN. Quy trình xoay không downtime:

**Pattern 1 — Overlap CA (xoay CA khi Root CA hết hạn):**
1. Tạo CA mới.
2. Thêm CA mới vào bundle `cacert.pem` của tất cả node + client → bundle giờ chứa CẢ CA cũ và CA mới.
3. Reload ovsdb-server (SIGHUP) để load bundle mới, lúc này chấp nhận client từ cả CA cũ và mới.
4. Sign cert client mới từ CA mới, triển khai cho từng chassis.
5. Khi tất cả chassis đã dùng cert mới → remove CA cũ khỏi bundle + reload.

Pattern này đảm bảo không bao giờ có cửa sổ client không xác thực được.

**Pattern 2 — Overlap cert (xoay cert leaf):**
1. Sign cert mới cho chassis với thời hạn dài hơn.
2. Triển khai cert mới song song với cert cũ (ovsdb-server chấp nhận cả hai nếu CA vẫn valid).
3. Restart ovn-controller để pick up cert mới.
4. Sau khi tất cả chassis switch xong, thu hồi cert cũ qua CRL (Certificate Revocation List).

**Danh sách thu hồi chứng chỉ (CRL):** khi cert bị lộ trước khi hết hạn, thêm vào CRL:
```
[ca-host]# ovs-pki --dir /etc/ovsdb-ca revoke chassis-1
```
Triển khai CRL tới ovsdb-server:
```
[sb-node1]# cp /etc/ovsdb-ca/revoked.pem /etc/ovsdb-ssl/
[sb-node1]# ovs-appctl -t ovsdb-server reload
```
Sau khi load CRL, mọi kết nối mới từ chassis-1 với cert cũ sẽ bị reject. Kết nối hiện tại (đã xác thực) vẫn tiếp tục đến khi kết nối lại, có một cửa sổ nhỏ cần lưu ý.

**OCSP (Online Certificate Status Protocol):** phương pháp thay thế CRL, server kiểm tra state cert thời gian thực qua OCSP responder. Chưa hỗ trợ native trong ovsdb-server (OVN 22.03), đang trong lộ trình.

## 10.6.4 RBAC advanced: multi-tenant và dựa trên condition

RBAC cơ bản của OVSDB (đã được đề cập Phần 10.2) cho phép định nghĩa:
- **Role**: tập hợp permission (ví dụ role "ovn-controller", "ovn-northd").
- **Permission**: ai được ghi table/column nào.
- **Ủy quyền dựa trên rule**: dùng condition để hạn chế "row nào được ghi".

**Ví dụ RBAC cho ovn-controller (mỗi chassis chỉ ghi được record của chính nó):**
```json
{
  "RBAC_Role": {
    "name": "ovn-controller",
    "permissions": {
      "Chassis": {
        "rbac_rules": {
          "name": "ovn-controller",
          "authorization": "name",
          "insert_delete": true,
          "update": ["nb_cfg", "external_ids"]
        }
      },
      "Port_Binding": {
        "rbac_rules": {
          "authorization": "chassis",
          "update": ["chassis", "encap", "virtual_parent"]
        }
      }
    }
  }
}
```

Với rule này: chassis-1 (CN của cert = "chassis-1") chỉ được insert/delete/update Nút mạng (Chassis) record name="chassis-1", và chỉ update Port_Binding đã bind tới "chassis-1". Không được modify record của chassis khác.

**Multi-tenant RBAC (ví dụ phức tạp cho OVN-Kubernetes multi-tenant):**
```json
{
  "RBAC_Role": {
    "name": "tenant-admin-tenantA",
    "permissions": {
      "Logical_Switch": {
        "rbac_rules": {
          "authorization": "external_ids:tenant=tenantA",
          "insert_delete": true,
          "update": ["name", "ports", "acls"]
        }
      }
    }
  }
}
```

Tenant A admin chỉ ghi được Logical_Switch có `external_ids:tenant=tenantA`. Không thể xem hay modify LS của tenant B.

**Ủy quyền dựa trên condition qua external_ids:** OVN dùng pattern này phổ biến, mọi row có `external_ids:owner=X` chỉ modifiable bởi user có cert với CN phù hợp X.

**Hạn chế RBAC OVSDB so với SQL:**
- Không có GROUP BY / permission mức column phức tạp như PostgreSQL ROW LEVEL SECURITY.
- Ủy quyền chỉ check qua string match: không hỗ trợ regex hay subquery.
- Không có khái niệm SUPERUSER: mỗi role cần permission tường minh từng table.
- Không có SAVEPOINT cho partial authorization failure.

## 10.6.5 Ghi log audit và tuân thủ

**Logging mặc định ovsdb-server:**
```
[sb-node1]# tail -f /var/log/ovn/ovsdb-server-ovnsb_db.log
2026-04-22T10:00:01.234Z|00123|jsonrpc|DBG|ssl:10.0.0.5:35782: received transact (1 ops)
2026-04-22T10:00:01.235Z|00124|ovsdb_server|INFO|chassis-1 (CN=chassis-1) inserted Port_Binding
```

Default log chỉ info level. Để enable audit đầy đủ:
```
[sb-node1]# ovs-appctl -t ovsdb-server vlog/set file:jsonrpc:dbg
[sb-node1]# ovs-appctl -t ovsdb-server vlog/set file:ovsdb_server:dbg
```

File log xoay mỗi ngày qua logrotate + lưu 90 ngày (policy chuẩn tuân thủ).

**Ghi log có cấu trúc tới SIEM:** chuyển log JSON sang Splunk/ELK/Datadog:
```
[sb-node1]# ovs-appctl -t ovsdb-server vlog/set syslog:dbg
```
Gửi qua syslog với format có cert CN + transaction ID + chi tiết operation.

**Tình huống tuân thủ:**
- **SOC 2 / ISO 27001:** log audit với giữ 1 năm, lưu trữ không thay đổi được (Amazon S3 Object Lock).
- **PCI-DSS:** cho hệ thống xử lý payment, audit mọi insert/update/delete vào Logical_Router_NAT (VIP mapping).
- **GDPR:** cho dữ liệu user, audit access + kiểm tra cô lập tenant.

**Phát hiện bất thường:** tích hợp log audit với SIEM để cảnh báo:
- Nút mạng (Chassis) ghi record không phải của mình (thử RBAC bypass).
- Tốc độ transaction tăng bất thường từ một cert (có thể tấn công bot).
- Xoay cert quá hạn (cert sắp hết hạn nhưng chưa xoay).

## 10.6.6 Mô hình threat và giải pháp

Mô hình threat OVSDB môi trường production:

**Threat 1 — Cert exposure:** cert private key rò rỉ (laptop dev bị xâm nhập, backup không mã hóa).
- **Giảm thiểu:** triển khai CRL nhanh, giám sát usage cert bất thường. Cert thời hạn ngắn (90 ngày thay vì 1 năm).

**Threat 2 — Insider abuse:** admin có cert valid, lạm dụng quyền ghi.
- **Giảm thiểu:** RBAC chặt chẽ (admin không có role "full-access", chỉ specific role). Log audit + 4-eye principle cho thay đổi môi trường production.

**Threat 3 — DoS transaction flood:** client độc hại gửi hàng triệu transaction nhỏ làm ovsdb-server bão hòa.
- **Giảm thiểu:** rate limiting ở network layer (iptables, Cilium NetworkPolicy). OVSDB 22.03 chưa có rate limiting built-in.

**Threat 4 — Downgrade attack:** attacker ép TLS version cũ có lỗ hổng đã biết.
- **Giảm thiểu:** cấu hình `ssl_protocols="TLSv1.2,TLSv1.3"` trong ovsdb-server. Disable TLSv1.0/1.1.

**Threat 5 — Compromised CA:** attacker lấy được root CA private key → có thể sign cert bất kỳ.
- **Giảm thiểu:** HSM (Hardware Security Module) cho root CA. Offline CA (không kết nối mạng, chỉ dùng khi cần sign).

> **State verification lab:** Output dựa trên ovs-pki docs; chờ kiểm chứng lab thực.

**Mục tiêu:** Thực hành setup mTLS từ đầu cho một cụm OVN NB 3-node.

**Bước thực hiện:**

```

**3.** Trên mỗi node server, start ovsdb-server với TLS:
```
[nb-node1]# ovsdb-server --remote=pssl:6641 \
    --private-key=/etc/ovsdb-ssl/nb-node1-privkey.pem \
    --certificate=/etc/ovsdb-ssl/nb-node1-cert.pem \

[ca]# ovs-pki --dir /etc/ovsdb-ca req+sign admin-client switch

```

**5.** Test kết nối từ client:
```
[client]# ovn-nbctl --db=ssl:nb-node1:6641,ssl:nb-node2:6641,ssl:nb-node3:6641 \
    --private-key=/etc/ovsdb-ssl-client/admin-client-privkey.pem \

[admin]# ovn-nbctl -- --id=@ro create RBAC_Role name=readonly-user \

**Bài học học được:**

Thứ hai, cert với CN là identity — CN client sẽ được so sánh trong rule RBAC (authorization="name"). CN nên nhất quán với tên chassis hoặc identity user.

Thứ ba, môi trường production luôn có CA riêng cho OVSDB, không mix với CA hệ thống để giảm bán kính tác hại nếu cert OVSDB bị lộ.

**Cleanup:**
```
[all]# kill <ovsdb-server pid>
```

## 10.6.7 Bảng `RBAC_Role` trong OVSDB schema

**Khái niệm + định nghĩa.** `RBAC_Role` là bảng OVSDB chuẩn (xuất hiện trong cả `ovn-nb.ovsschema` và `ovn-sb.ovsschema`) lưu định nghĩa của từng role. Một row tương ứng với một identity class (ví dụ `ovn-controller`, `ovn-northd`); các row trong bảng này không phải user account mà là vai trò mà cert client được map vào sau khi xác thực thành công.

**Phân loại + vị trí trong kiến trúc.** Loại: bảng OVSDB metadata, schema-level (không phải data table do user CRUD). Vị trí: ovsdb-server giữ tham chiếu nội bộ qua `db->rbac_role` (xem `ovsdb/rbac.c` hàm `ovsdb_rbac_lookup_perms`), được tải khi server mở database file.

**Schema verbatim** (ovn-sb.ovsschema, branch-22.03 verified qua raw GitHub):
```json
"RBAC_Role": {
  "columns": {
    "name": {"type": "string"},
    "permissions": {
      "type": {
        "key": {"type": "string"},
        "value": {"type": "uuid",
                  "refTable": "RBAC_Permission",
                  "refType": "weak"},
        "min": 0, "max": "unlimited"
      }
    }
  }
}
```

**Cơ chế hoạt động.** Khi client thực hiện transaction, ovsdb-server gọi `ovsdb_rbac_lookup_perms(db, role, table)` (file `ovsdb/rbac.c`), hàm này (1) tìm row trong `RBAC_Role` có `name == role`, (2) tra cứu key bằng tên bảng đích trong map `permissions`, (3) trả về UUID trỏ tới row trong `RBAC_Permission`. Map có `refType: "weak"` nghĩa là khi row `RBAC_Permission` bị xoá, map entry tự động cleanup (không tạo dangling reference).

**Cách kỹ sư vận hành.** Tạo role mới qua `ovn-sbctl` (hoặc raw `ovsdb-client transact`):
```
[admin]# ovn-sbctl -- --id=@p create RBAC_Permission table=Chassis \
    authorization=name insert_delete=true update='["nb_cfg","external_ids"]' \
    -- create RBAC_Role name=ovn-controller permissions:Chassis=@p
```
Verify:
```
[admin]# ovn-sbctl list RBAC_Role
_uuid               : abc12345-...
name                : "ovn-controller"
permissions         : {"Chassis"=def67890-..., "Port_Binding"=...}
```

**Failure mode + tín hiệu chẩn đoán.** Nếu role row không tồn tại cho `role` mà manager gán cho client (xem 10.6.8), mọi write từ client đó bị reject; ovsdb-server log dòng `RBAC: role 'X' has no permissions on table 'Y'`. Recovery: tạo `RBAC_Permission` row + insert vào `RBAC_Role.permissions` map.

## 10.6.8 Bảng `RBAC_Permission` và biểu thức authorization

**Khái niệm + định nghĩa.** `RBAC_Permission` là bảng granular đặc tả "client với role X được phép làm gì trên bảng Y". Mỗi row gồm 4 cột chính: `table`, `authorization`, `insert_delete`, `update`.

**Schema verbatim** (verified raw GitHub branch-22.03):
```json
"RBAC_Permission": {
  "columns": {
    "table":         {"type": "string"},
    "authorization": {"type": {"key": "string", "min": 0, "max": "unlimited"}},
    "insert_delete": {"type": "boolean"},
    "update":        {"type": {"key": "string", "min": 0, "max": "unlimited"}}
  }
}
```

**Cơ chế hoạt động.** Hàm `ovsdb_rbac_authorized(perms, id, row)` (file `ovsdb/rbac.c`) duyệt từng entry của cột `authorization`:

- Entry là chuỗi rỗng `""` → mọi client được phép (wildcard).
- Entry là tên cột (ví dụ `"name"`) → so sánh `strcmp(row->columns["name"], id)`. Nếu khớp, return true.
- Entry chứa dấu hai chấm (`"external_ids:tenant"`) → tách thành (column_name, key), lookup map column tại key, so sánh giá trị với `id`. Đây là dot-separated path lookup verified qua hàm helper `ovsdb_find_row_by_string_key`.

`id` ở đây là **CN của cert client** (không phải username); ovsdb-server lấy từ TLS handshake (xem 10.6.9).

**Phân biệt insert_delete vs update.** `insert_delete=true` cho phép cả INSERT (tạo row mới với `authorization` field bằng id) và DELETE (xoá row mà `authorization` field bằng id). `update` là danh sách tên cột client được phép modify trên row đã pass authorization check; cột không có trong danh sách → write reject ngay cả khi authorization pass. Cùng danh sách `update` áp dụng cho cả operation `update` và `mutate` (verified qua `ovsdb_rbac_update`, `ovsdb_rbac_mutate`).

**Pattern thực tế OVN.** Row chuẩn cho ovn-controller ghi `Port_Binding` chỉ chassis của mình:
```
table:         Port_Binding
authorization: ["chassis"]      # Port_Binding.chassis là ref tới Chassis row
insert_delete: false             # ovn-controller không được tạo/xoá Port_Binding
update:        ["chassis", "encap", "virtual_parent", "up", "nb_cfg"]
```
Authorization `"chassis"` là cột ref → ovsdb-server resolve qua chain "Port_Binding.chassis → Chassis._uuid → Chassis.name" rồi so sánh với CN cert client.

**Failure mode.** `authorization` set rỗng `[]` (không có entry nào) → mọi write reject (deny-by-default). Log line: `RBAC: row not authorized for role 'X' on table 'Y'`.

## 10.6.9 Cơ chế bắt tay mTLS giữa ovsdb-server và client

**Khái niệm + vai trò.** mTLS (mutual TLS) là biến thể TLS trong đó cả server lẫn client cùng trình cert được CA tin cậy ký. Trong OVSDB, mTLS handshake giúp ovsdb-server ánh xạ kết nối TCP tới một identity (CN cert) phục vụ cho RBAC lookup ở 10.6.7-10.6.8.

**Cơ chế hoạt động chi tiết.** Stack triển khai trong `lib/stream-ssl.c` của OVS, dùng OpenSSL. Sequence:

1. Client mở TCP tới `pssl:6642` (passive SSL listener của ovsdb-server).
2. Server gửi `ServerHello` + `Certificate` (cert sb-node ký bởi CA).
3. Server gửi `CertificateRequest` (yêu cầu client gửi cert vì TLS context của ovsdb-server bật `SSL_VERIFY_PEER | SSL_VERIFY_FAIL_IF_NO_PEER_CERT`).
4. Client gửi `Certificate` (cert chassis-X) + `CertificateVerify` (signature chứng minh sở hữu private key).
5. Cả hai derive session key, hoàn tất handshake.
6. Sau handshake, ovsdb-server gọi `SSL_get_peer_certificate()`, extract Subject CN field, lưu vào struct `jsonrpc_session` cho mọi RBAC check tiếp theo.

**Phiên bản TLS hỗ trợ.** OVS 2.17 dùng `SSLv23_method()` (multi-version) cộng với cờ `SSL_OP_NO_SSLv2 | SSL_OP_NO_SSLv3` (verified `lib/stream-ssl.c`). TLS 1.0/1.1/1.2/1.3 đều bật mặc định; điều chỉnh qua `--ssl-protocols=TLSv1.2,TLSv1.3` (verified man `ovsdb-server(1)`). Từ 2026 các deployment chuẩn nên explicit disable TLSv1.0/1.1 vì dễ downgrade attack (Threat 4 ở 10.6.6).

**Verification.** Quan sát handshake bằng `tcpdump` + giải mã trong Wireshark với key log:
```
[chassis-1]# SSLKEYLOGFILE=/tmp/keys.log ovn-sbctl --db=ssl:sb-node1:6642 ... show
[chassis-1]# tcpdump -i any -w /tmp/h.pcap port 6642
```
Wireshark Edit → Preferences → TLS → (Pre)-Master-Secret log filename → `/tmp/keys.log`; mở `h.pcap` sẽ thấy đủ ServerHello, CertificateRequest, ClientCertificate.

**Failure mode điển hình.** (a) Cert chain incomplete: client log `unable to verify the first certificate` → bundle thiếu intermediate CA. (b) Cert hết hạn: log `certificate has expired`. (c) Server từ chối client: log server `peer did not return a certificate` → client thiếu `--certificate` flag. (d) CN không khớp role: handshake thành công nhưng mọi transaction reject vì `ovsdb_rbac_lookup_perms` trả NULL cho cặp (role, CN).

## 10.6.10 Cờ `--bootstrap-ca-cert` và rủi ro khi để bật trong production

**Khái niệm + vai trò.** `--bootstrap-ca-cert` là chế độ "trust on first use" (TOFU) cho CA certificate. Khi file CA cert chỉ định chưa tồn tại, ovsdb-server (hoặc client) sẽ chấp nhận cert từ peer trong lần kết nối đầu tiên không xác thực, tự động ký nhận self-signed CA, và lưu vào file. Các kết nối tiếp theo dùng CA đã lưu để xác thực bình thường.

**Verbatim man page** (`ovsdb-server(1)`, verified man7.org):

> "When `cacert.pem` exists, this option has the same effect as `-C` or `--ca-cert`. If it does not exist, then `ovsdb-server` will attempt to obtain the CA certificate from the SSL/TLS peer on its first SSL/TLS connection and save it to the named PEM file."
>
> "This option exposes the SSL/TLS connection to a man-in-the-middle attack obtaining the initial CA certificate, but it may be useful for bootstrapping."

**Cơ chế hoạt động.** Logic nằm trong `lib/stream-ssl.c` qua hàm `do_ca_cert_bootstrap()` (verified raw GitHub):

1. Parser `parse_options()` trong `ovsdb/ovsdb-server.c` set biến global `bootstrap_ca_cert = true` khi gặp `OPT_BOOTSTRAP_CA_CERT` enum case.
2. `reconfigure_ssl()` gọi `stream_ssl_set_ca_cert_file(file, bootstrap=true)`.
3. Trên lần handshake đầu tiên hoàn tất, `do_ca_cert_bootstrap()` chạy: `SSL_get_peer_cert_chain()` lấy chain peer, `X509_check_issued(cert, cert)` kiểm tra self-signed, mở file với `O_CREAT | O_EXCL | O_WRONLY` rồi `PEM_write_X509(file, cert)`. Cuối cùng `SSL_CTX_load_verify_locations()` reload CA mới và tắt cờ bootstrap.

**Vì sao cấm trong production.** Trong giai đoạn TOFU không có verification chain, attacker MITM giữa node có thể trình cert giả mạo của attacker; ovsdb-server sẽ ký nhận attacker là CA hợp lệ và mọi kết nối sau đó bị MITM "hợp pháp hoá". Cờ chỉ phù hợp cho lab dev hoặc PoC, hoặc trong môi trường có kênh out-of-band (ví dụ subnet quản trị riêng vật lý).

**Workflow đúng cho production.** Phân phối CA cert qua kênh tin cậy (ansible vault, Hashicorp Vault, kolla-ansible secret distribution) trước khi start service; chỉ định bằng `--ca-cert=/etc/ovsdb-ssl/cacert.pem` cố định, KHÔNG dùng `--bootstrap-ca-cert`. Verify lúc deploy:
```
[sb-node1]# sha256sum /etc/ovsdb-ssl/cacert.pem
abcdef...   # phải khớp với hash đã ký nhận từ CA host
```

**Failure mode.** Nếu vô tình để `--bootstrap-ca-cert` trong systemd unit, file CA tồn tại trên disk → cờ degrade thành alias của `--ca-cert` (an toàn). Nhưng nếu file bị xoá (do ổ đĩa fail + recovery, do ai đó `rm`) trước khi service restart, lần khởi động sau service vào lại chế độ TOFU và lần kết nối kế tiếp có thể bị MITM. Recovery: detect qua audit script (`grep bootstrap-ca-cert /etc/systemd/system/ovsdb-server.service`), gỡ flag, restart service.

## 10.6.11 Quy trình xoay chứng chỉ không downtime, chi tiết kỹ thuật

**Khái niệm + vai trò.** Cert rotation là việc thay thế cert cũ bằng cert mới (cùng CN, khoá mới) trước khi cert cũ hết hạn, sao cho mọi kết nối hiện hữu không bị đứt và không có cửa sổ thời gian client không xác thực được.

**Cơ chế cốt lõi.** OVS không hot-reload private key; tuy nhiên ovsdb-server hỗ trợ reload CA cert + cert chain qua `SIGHUP` hoặc `ovs-appctl -t ovsdb-server vlog/reopen` (cho log) và `ovs-appctl -t ovsdb-server ovsdb-server/reconnect` (cho remote config). Với cert/key đổi, phương pháp ổn định nhất là restart từng node Raft theo thứ tự (rolling restart): cluster Raft 3-node chịu được mất 1 node một lúc; client reconnect vào node còn lại.

**Workflow xoay leaf cert (chassis-X) không downtime đầy đủ.**

Bước 1, sign cert mới với CA hiện tại nhưng đổi CN suffix `-v2` để phân biệt:
```
[ca]# ovs-pki --dir /etc/ovsdb-ca req+sign chassis-1-v2 switch
```

Bước 2, đẩy cert mới song song cert cũ:
```
[chassis-1]# cp chassis-1-v2-{cert,privkey}.pem /etc/ovn/
[chassis-1]# ln -sfn chassis-1-v2-cert.pem /etc/ovn/active-cert.pem
[chassis-1]# ln -sfn chassis-1-v2-privkey.pem /etc/ovn/active-privkey.pem
```
Symlink-based switch giúp atomic: chỉ thay symlink, không sửa file.

Bước 3, restart ovn-controller:
```
[chassis-1]# systemctl restart ovn-controller
```
Trong khoảng 1-3 giây ovn-controller reconnect đến cluster SB; vì kết nối từ chassis này đã đứt và mở lại, dữ liệu Port_Binding của chassis tạm không refresh, nhưng br-int OF rule đã cài vẫn forward traffic bình thường (datapath cache không phụ thuộc kết nối SB).

Bước 4, verify CN mới:
```
[chassis-1]# ovs-appctl -t ovn-controller connection-status
state: connected
remote: ssl:sb-node1:6642
```
Trên SB:
```
[sb-node1]# grep "chassis-1-v2" /var/log/ovn/ovsdb-server-ovnsb_db.log | tail -3
... ssl:10.0.0.1:43210: client cert CN=chassis-1-v2 authenticated
```

Bước 5, sau khi tất cả chassis qua cert v2, revoke cert v1 qua CRL + reload CA bundle.

**Cửa sổ rủi ro.** Reload CRL không kick off connection hiện hữu; client đang kết nối với cert đã revoke sẽ tiếp tục cho đến khi (a) probe interval timeout (`ovn-remote-probe-interval`, default 60s), hoặc (b) chassis tự khởi động lại. Để force ngắt: `ovs-appctl -t ovsdb-server connection/kill <chassis-CN>`. Trong 60 giây cửa sổ này, attacker đã có cert leak có thể tiếp tục transact; vì vậy cert thời hạn ngắn (90 ngày, theo guideline NIST SP 800-57 cho key xoay frequent) giảm tác hại.

**Failure mode.** (a) Cert mới sign sai CN → client kết nối được nhưng RBAC reject mọi write; recovery: re-sign đúng CN. (b) Symlink trỏ tới file không tồn tại → ovn-controller log `cannot open private key file`; recovery: `readlink -f` verify trước khi restart service. (c) Đẩy cert mới đồng loạt cả fleet 100+ chassis cùng lúc → cluster SB chịu burst connect, có thể OOM revalidator; recovery: canary 1 chassis trước, 5 chassis sau, rồi rollout phần còn lại với rate limiting (1 chassis/2s).

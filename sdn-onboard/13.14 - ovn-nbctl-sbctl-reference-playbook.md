## 13.14.1 Kiến trúc CLI ovn-nbctl + ovn-sbctl

### 13.14.1.1 Hai tool, hai vai trò

`ovn-nbctl` và `ovn-sbctl` là hai CLI tool riêng biệt trong package `ovn-common` (Ubuntu 22.04) hoặc `ovn-utilities` (RHEL). Kiến trúc giống nhau (đều là OVSDB IDL client), nhưng kết nối tới hai database khác nhau:

- **`ovn-nbctl`** kết nối **NBDB** (port mặc định 6641 TCP hoặc Unix socket `/var/run/ovn/ovnnb_db.sock`). NBDB chứa **ý đồ cấu hình logical** (Logical_Switch, Logical_Router, ACL, ...). Đây là tool mà CMS (Neutron, OVN-Kubernetes) gián tiếp gọi để ghi cấu hình.
- **`ovn-sbctl`** kết nối **SBDB** (port mặc định 6642 TCP hoặc Unix socket `/var/run/ovn/ovnsb_db.sock`). SBDB chứa **state đã compile** từ ovn-northd: Logical_Flow, Port_Binding, Chassis, Datapath_Binding. Đây là tool **debug + intervention manual** cho kỹ sư on-call.

> **Quy tắc 5 trụ cột (Rule 5 nguyên tắc kiểm soát):** Kỹ sư production KHÔNG được dùng `ovn-sbctl` để cấu hình logical (ví dụ: tạo Port_Binding manual). Mọi cấu hình logical phải qua NBDB → ovn-northd → SBDB. SBDB là output của compile, write thẳng vào sẽ bị overwrite hoặc gây bất đồng. Use case duy nhất hợp lệ cho `ovn-sbctl set/add` là **chassis intervention** khi chassis ghost không tự release (xem §13.14.5).

### 13.14.1.2 Kết nối, daemon mode, tracing

Mỗi lần `ovn-nbctl <command>` chạy, tool thực hiện:

1. Đọc biến môi trường `OVN_NB_DB` hoặc tham số `--db=` để biết URL.
2. Mở connection (TCP hoặc Unix), gửi `monitor_cond_since` để lấy snapshot OVSDB.
3. Build IDL in-memory (~5-50 MB cho cluster trung bình).
4. Thực thi command (đọc/ghi rows + relations).
5. Commit transaction nếu có thay đổi.
6. Đóng connection.

Bước 2-3 là **bottleneck**: với cluster có 10K+ object, một lần chạy có thể mất 200-800 ms chỉ để khởi tạo IDL. Trong workflow batch (CMS gọi 100 lệnh/giây), đây là rào cản lớn.

**Daemon mode (giải pháp):**

```bash
# Khởi động daemon
[host]# ovn-nbctl --detach --pidfile=/var/run/ovn/ovn-nbctl.pid \
                  --log-file=/var/log/ovn/ovn-nbctl.log \
                  --db=tcp:nb1:6641,tcp:nb2:6641,tcp:nb3:6641
# Mọi lệnh sau đó dùng socket daemon:
[host]# export OVN_NB_DAEMON=$(ovn-nbctl get-daemon-socket)
[host]# ovn-nbctl ls-add my-ls
# ... lệnh < 5 ms thay vì 200-800 ms
```

Daemon giữ IDL session warm, share giữa các invocation. CMS production thường chạy daemon thường trực.

**Tracing options:**

- `--verbose` (`-v`) bật log debug.
- `--timeout=N` đặt timeout cho transaction.
- `--no-wait` không đợi commit confirmation (dangerous, chỉ dùng cho fire-and-forget).
- `--dry-run` đối với một số subcommand: print transaction JSON mà không commit.

### 13.14.2.2 Nhóm Logical Switch + Logical Switch Port (28 lệnh)

**Logical Switch (LS):**

| Lệnh | Mục đích |
|------|----------|
| `ls-add <name>` | Tạo LS mới |
| `ls-del <ls>` | Xoá LS |
| `ls-list` | Liệt kê tất cả LS |

**Logical Switch Port (LSP):**

| Lệnh | Mục đích |
|------|----------|
| `lsp-add <ls> <port>` | Tạo LSP gắn vào LS |
| `lsp-del <port>` | Xoá LSP |
| `lsp-list <ls>` | List LSP của LS |
| `lsp-set-addresses <port> [<addresses>...]` | Đặt MAC + IP cho LSP. Format: `"aa:bb:cc:dd:ee:ff 10.0.0.1"`, `unknown` (cho phép mọi MAC), `dynamic` (auto IPAM), `router` (LSP nối với LRP) |
| `lsp-get-addresses <port>` | Đọc lại addresses |
| `lsp-set-port-security <port> [<addresses>...]` | Đặt anti-spoofing. Mặc định = đồng bộ với `lsp-set-addresses` |
| `lsp-get-port-security <port>` | Đọc lại |
| `lsp-set-type <port> <type>` | Đặt type: `""` (vif), `localnet`, `localport`, `l2gateway`, `l3gateway`, `router`, `virtual`, `external`, `remote` |
| `lsp-get-type <port>` | Đọc type |
| `lsp-set-options <port> <KEY=VALUE>...` | Đặt options dict. Ví dụ: `network_name=physnet1` cho localnet, `router-port=lrp-name` cho router type |
| `lsp-get-options <port>` | Đọc options |
| `lsp-set-enabled <port> true\|false` | Bật/tắt LSP |
| `lsp-get-enabled <port>` | Đọc state enabled |
| `lsp-get-up <port>` | Đọc field `up` (true khi chassis claim thành công) |
| `lsp-set-dhcpv4-options <port> <DHCP_Options uuid>` | Liên kết LSP với DHCPv4 options |
| `lsp-get-dhcpv4-options <port>` | Đọc lại |
| `lsp-set-dhcpv6-options <port> <DHCP_Options uuid>` | Tương tự cho DHCPv6 |
| `lsp-get-dhcpv6-options <port>` | Đọc |
| `lsp-get-parent <port>` | Đọc parent_name (cho VLAN child port) |
| `lsp-get-tag <port>` | Đọc VLAN tag |
| `lsp-get-ls <port>` | Reverse lookup: từ port tìm LS chứa nó |

## 13.14.4 Anatomy Template A, 10 output quan trọng

### Anatomy 1: `ovn-nbctl show`

> **Bối cảnh khi chạy:** Tổng quan ý đồ logical hiện tại. Lệnh đầu tiên khi onboarding cluster mới hoặc debug.

```
switch f3a8 (web-ls)
    port web1
        addresses: ["aa:bb:cc:00:00:01 10.0.0.10"]
    port web2
        addresses: ["aa:bb:cc:00:00:02 10.0.0.11"]
    port lsp-router
        type: router
        router-port: lrp-web
router 89e1 (web-router)
    port lrp-web
        mac: "aa:bb:cc:99:00:01"
        networks: ["10.0.0.1/24"]
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| `switch` count | Khớp số LS CMS expected | Mất 1-2 LS | Mất nhiều LS | CMS sync lag; kiểm tra Neutron/k8s reconcile |
| `port` count per LS | Khớp số VM/container active | Mất 1-2 port | Trùng IP cảnh báo | VM teardown chưa clean LSP |
| `addresses` MAC duplicate | Mỗi MAC unique | 1 MAC trên 2 LSP | Nhiều conflict | LSP cũ chưa xoá; check `lsp-list` |
| `type: router` LSP | Match số LRP nối | Mismatch | LSP `router` không có `router-port` option | Bị remove một bên (LRP hoặc LSP) |
| `mac` LRP format | `xx:xx:xx:xx:xx:xx` | (none) | Empty | Đặt sai khi `lrp-add` |
| `networks` CIDR | IP không overlap với LS | Overlap | LR có 2 LRP cùng subnet | Topology sai |

**Kịch bản bẻ gãy:**

- LSP `type: router` thiếu `router-port`: peer LRP không tồn tại. Run `ovn-nbctl lsp-set-options <lsp> router-port=<lrp>` để link lại.
- LR có 2 LRP cùng `networks` 10.0.0.1/24 → xung đột routing. Đổi 1 cái sang subnet khác.

### Anatomy 2: `ovn-sbctl list Chassis`

> **Bối cảnh khi chạy:** Kiểm tra chassis nào đang đăng ký SB DB, encap type, hostname, transport zone.

```
_uuid               : a1b2c3d4-...
external_ids        : {ovn-bridge-mappings="physnet1:br-ex", \
                       ovn-cms-options="enable-chassis-as-gw", \
                       ovn-encap-ip="192.0.2.10"}
hostname            : "compute-01.example.com"
name                : "55b4d9e8-..."
nb_cfg              : 142
encaps              : [enc1, enc2]
other_config        : {bridge-mappings="physnet1:br-ex"}
transport_zones     : [zone1]
vtep_logical_switches : []
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| `name` UUID | Match `system-id` của ovs-vsctl | Khác | Trống | ovn-controller chưa register; xem log |
| `hostname` | Match `hostname` thực OS | Khác | (none) | DNS lag hoặc rename không sync |
| `nb_cfg` | Match SB_Global.nb_cfg ± 5 | Lệch 10-50 | Lệch > 100 | Chassis lag sync với ovn-northd |
| `encaps` count | 1+ encap (Geneve/VXLAN/STT) | 0 | (none) | Chassis chưa có tunnel; pinging chassis khác sẽ fail |
| `external_ids:ovn-encap-ip` | IP underlay routable | IP khác subnet | Trống | Check `ovs-vsctl get Open_vSwitch . external_ids` trên chassis |
| `transport_zones` | Match expected zone | Empty (= all-tenant) | Sai zone | Tenant isolation bị phá |

**Kịch bản bẻ gãy:**

- `nb_cfg` lệch > 100 với SB_Global: chassis bị stuck. Restart ovn-controller; nếu vẫn lệch → SB DB Raft slow hoặc chassis không kết nối được SB.
- 2 chassis cùng `name` UUID: rất hiếm, do clone VM image không xoá `system-id`. Xoá 1 chassis qua `ovn-sbctl chassis-del`.

### Anatomy 3: `ovn-sbctl list Port_Binding`

## 13.14.5 Decision matrix, tình huống → command

| Tình huống | Lệnh chính | Lệnh phụ |
|-----------|-----------|----------|
| Đăng ký provider network mới | `ls-add provider; lsp-add provider phys; lsp-set-type phys localnet; lsp-set-options phys network_name=physnet1` | `lsp-set-addresses phys unknown` |
| Tạo tenant network | `ls-add tenant1; lsp-add tenant1 vm1; lsp-set-addresses vm1 dynamic` | `dhcp-options-create 192.168.1.0/24; lsp-set-dhcpv4-options vm1 <uuid>` |
| Connect tenant → provider qua router | `lr-add r1; lrp-add r1 lrp-tenant aa:bb 192.168.1.1/24; lrp-add r1 lrp-prov aa:bb:99 10.0.0.1/24; lr-route-add r1 0.0.0.0/0 10.0.0.254` | `ls-lb-add` cho VIP nếu cần |
| Tạo SNAT external | `lr-nat-add r1 snat 10.0.0.50 192.168.1.0/24` | `lr-nat-list r1` để kiểm chứng |
| ACL deny tenant cross-talk | `pg-add tenant1-pg vm1 vm2; acl-add tenant1-pg from-lport 100 "outport == @tenant2-pg" drop` | `acl-add tenant1-pg from-lport 50 "ip4" allow` để cho phép bên trong tenant |
| Debug "ping fail giữa 2 VM cùng LS" | `ovn-sbctl lflow-list <ls>` để kiểm tra logical pipeline; `ovn-trace <ls> '<expr>'` để mô phỏng | `ovs-appctl ofproto/trace br-int` cho OpenFlow level (Phần 9.25) |
| Debug "VM mới không có flow trên chassis" | `ovn-sbctl list Port_Binding <port>` xem chassis bind | `ovs-appctl -t ovn-controller debug/dump-local-bindings` (Phần 13.7.8) |
| Manual unbind chassis ghost | `ovn-sbctl chassis-del <chassis-uuid>` | Kiểm chứng với `ovn-sbctl list Chassis` không còn record |
| Compile lag NB→SB | `ovn-nbctl get NB_Global . nb_cfg` vs `ovn-sbctl get SB_Global . nb_cfg` | `ovs-appctl -t ovn-northd inc-engine/show` (Phần 13.8.6) |
| Reload tunnel mappings | `ovn-vsctl set Open_vSwitch . external_ids:ovn-bridge-mappings=physnet1:br-ex` (chassis side) | restart ovn-controller nếu không pickup |
| Tenant teardown | `ovn-nbctl ls-del tenant1` (xoá LS sẽ cascade xoá LSP, ACL, LB) | `ovn-nbctl find Logical_Switch_Port external_ids:tenant=tenant1` để kiểm chứng dọn sạch |

## 13.14.6 Guided Exercise S67': New tenant + multi-tier ACL

**Mục tiêu:** Triển khai tenant `T1` với 2 tier (web + db), router ra ngoài, ACL deny cross-tier ngoài 1 port.

**Bước thực hiện:**

```bash
# 1. Tạo Logical Switch tier
[host]# ovn-nbctl ls-add t1-web-ls
[host]# ovn-nbctl ls-add t1-db-ls

# 2. Tạo VM port
[host]# ovn-nbctl lsp-add t1-web-ls web1
[host]# ovn-nbctl lsp-set-addresses web1 "aa:bb:cc:11:00:01 10.1.1.10"
[host]# ovn-nbctl lsp-add t1-db-ls db1
[host]# ovn-nbctl lsp-set-addresses db1 "aa:bb:cc:22:00:01 10.2.2.10"

# 3. Tạo Logical Router
[host]# ovn-nbctl lr-add t1-router
[host]# ovn-nbctl lrp-add t1-router lrp-web aa:bb:cc:99:00:01 10.1.1.1/24
[host]# ovn-nbctl lrp-add t1-router lrp-db  aa:bb:cc:99:00:02 10.2.2.1/24

# 4. Connect LS với LR (LSP type=router)
[host]# ovn-nbctl lsp-add t1-web-ls lsp-to-router-web
[host]# ovn-nbctl lsp-set-type lsp-to-router-web router
[host]# ovn-nbctl lsp-set-addresses lsp-to-router-web router
[host]# ovn-nbctl lsp-set-options lsp-to-router-web router-port=lrp-web

[host]# ovn-nbctl lsp-add t1-db-ls lsp-to-router-db
[host]# ovn-nbctl lsp-set-type lsp-to-router-db router
[host]# ovn-nbctl lsp-set-addresses lsp-to-router-db router
[host]# ovn-nbctl lsp-set-options lsp-to-router-db router-port=lrp-db

# 5. Port_Group cho từng tier
[host]# ovn-nbctl pg-add t1-web-pg web1
[host]# ovn-nbctl pg-add t1-db-pg  db1

# 6. ACL: deny web → db ngoài port 5432 (Postgres)
[host]# ovn-nbctl acl-add t1-web-pg from-lport 1000 \
            "outport == @t1-db-pg && tcp.dst != 5432" drop
[host]# ovn-nbctl acl-add t1-web-pg from-lport 999 \
            "outport == @t1-db-pg && tcp.dst == 5432" allow-related

# 7. Kiểm chứng
[host]# ovn-nbctl show
[host]# ovn-nbctl acl-list t1-web-pg
[host]# ovn-sbctl lflow-list t1-web-ls | grep "ls_in_acl"
```

**Output mong đợi (doc-plausible, kiểm chứng khi có lab):**

```
$ ovn-nbctl acl-list t1-web-pg
from-lport  1000 (outport == @t1-db-pg && tcp.dst != 5432)  drop
from-lport   999 (outport == @t1-db-pg && tcp.dst == 5432)  allow-related
```

**Bài học học được:**

1. Workflow tạo tenant cần 8-12 lệnh `ovn-nbctl`. Daemon mode giảm thời gian từ ~10s xuống ~500ms.
2. `lsp-set-type router` + `lsp-set-options router-port=` là cặp BẮT BUỘC để kết nối LS với LR. Nếu thiếu `router-port`, gói tin vào LRP sẽ drop.
3. Port_Group + ACL `outport == @<pg>` là pattern chuẩn cho multi-tier security. So với 2 ACL từ web1 → db1 (riêng cho từng VM cặp đôi), pattern PG scale O(N) thay vì O(N²).

**Cleanup:**

```bash
[host]# ovn-nbctl ls-del t1-web-ls
[host]# ovn-nbctl ls-del t1-db-ls
[host]# ovn-nbctl lr-del t1-router
[host]# ovn-nbctl pg-del t1-web-pg
[host]# ovn-nbctl pg-del t1-db-pg
```

(Các ACL bị cascade xoá khi Port_Group bị xoá.)

## 13.14.7 Capstone POE Phase I.B1: ovn-sbctl debug vs ovn-nbctl reconfig

> **Bối cảnh:** Production cluster, VM `web1` báo lỗi "no network" sau live migration từ chassis A → B.

**Tuyên bố cần kiểm chứng:** "Chạy `ovn-sbctl set Port_Binding web1 chassis=<chassis-B>` để force bind là cách đúng để fix."

**Predict:** Sai. Force binding qua ovn-sbctl là intervention danger cho production.

**Observe (giả định, kiểm chứng khi có lab):**

- Chạy `ovn-sbctl set Port_Binding web1 chassis=...` → gói tin có flow trên br-int chassis B trong vài giây.
- Sau 30 giây ovn-northd recompute từ NB → reset `Port_Binding.chassis` về giá trị cũ (hoặc empty) → flow biến mất, VM mất network lần 2.
- Đồng thời, `requested_chassis` của Port_Binding trong NB không phản ánh ý đồ → CMS (Neutron) thấy bất đồng và có thể tự revert.

**Explain:**

1. SBDB là **output** của compile NB → SB. Ghi thẳng vào SBDB sẽ bị overwrite mỗi cycle compile (~50 ms - 5 giây tuỳ workload).
2. NBDB là source of truth. Lệnh đúng:
   - `ovn-nbctl lsp-set-options web1 requested-chassis=<chassis-B>` (set ý đồ ở NB).
   - ovn-northd compile lại → cập nhật `Port_Binding.requested_chassis` → ovn-controller B claim.
3. Trường hợp duy nhất hợp lệ dùng `ovn-sbctl set` lên Port_Binding là khi NB đã đúng nhưng SB stuck (rất hiếm, debug only). Phải hiểu trade-off: khoảnh khắc force tới khoảnh khắc northd recompute next.

**Falsification test:**

- Force bind qua `ovn-sbctl set`, đợi 30 giây.
- Run `ovn-nbctl get Logical_Switch_Port web1 options` xem có `requested-chassis` không.
- Nếu không có → northd sẽ revert, predict đúng. Nếu có (ai đó đã set NB cùng lúc) → có thể giữ.

**Tuyên bố sai ở đâu:** Dùng `ovn-sbctl set` cho ý đồ logical là **anti-pattern**. Đúng phải là `ovn-nbctl` (source of truth). `ovn-sbctl` chỉ dùng để (a) read/inspect, (b) chassis-del manual khi node ghost, (c) debug short-term với hiểu rõ rủi ro overwrite.

> **Quy tắc 5 trụ cột (Rule kiểm soát):** Ý đồ logical → NBDB qua ovn-nbctl. Quan sát + intervention chassis → SBDB qua ovn-sbctl. KHÔNG hoán đổi.

## 13.14.8 Điểm cốt lõi cần nhớ

**Một.** `ovn-nbctl` (97 lệnh) cấu hình ý đồ logical NBDB. `ovn-sbctl` (15 lệnh) inspect + intervention SBDB. Vai trò khác nhau, không hoán đổi (Rule 5 trụ cột).

**Hai.** Daemon mode giảm latency 100-1000x cho workload batch. CMS production luôn dùng daemon.

**Ba.** Command grouping: 12 nhóm cho ovn-nbctl (Generic / LS+LSP / LR+LRP / ACL / PG / LB / DHCP / QoS / HA Chassis / CoPP / Connection / SSL). 15 lệnh ovn-sbctl tập trung debug + chassis lifecycle.

**Bốn.** `lsp-set-type router` + `lsp-set-options router-port=` là cặp BẮT BUỘC cho LS-LR connection. Thiếu một trong hai → packet drop ngầm.

**Năm.** Port_Group + ACL pattern: `acl-add <pg> from-lport <pri> "outport == @<other-pg>" <action>`. Scale O(N) thay vì O(N²). Production sẽ thấy ngay ở 100+ tenant.

**Sáu.** `nb_cfg` gauge: `ovn-nbctl get NB_Global . nb_cfg` vs `ovn-sbctl get SB_Global . nb_cfg`. Lệch nhỏ (≤ 5) = healthy. Lệch > 100 = compile lag.

**Bảy.** `ovn-sbctl set Port_Binding chassis=...` là **anti-pattern** trừ debug short-term. Đúng phải set `requested-chassis` qua `ovn-nbctl lsp-set-options`.

**False-alarm root-cause analysis:**

1. Original audit method `git grep -l "<fn>" <ref> -- '*.c' | head -1` returned the first file matching the identifier, which is often a call site rather than a definition site.
2. Original audit did not `git checkout` the baseline tag, so verification was performed against in-memory `git show` snapshots and was prone to escape/quoting artefacts.
3. Findings were not re-verified between draft and plan write, so 3 false positives slipped into the v1 draft.

**Method correction (locked in for Phase Q0 onward):**

- Always `git checkout <baseline-tag>` (e.g., `v2.17.9`) before audit. This puts the baseline source on disk.
- Always grep with anchor `^<fn>` to target the function definition (column-0 name, matching OVS's coding style: return type on the previous line, function name in column 0).
- Always cross-check the header file (`.h`) for the declaration when verifying a function's canonical home.
- Always re-verify findings before transcribing them into commit messages.

## 3. Source verification methodology

### 3.1. Tools and pinning

- **Local OVS repo:** `C:\Users\voleh\Documents\ovs`. All release tags fetched (`v2.5.0`, `v2.6.0`, `v2.12.0`, `v2.17.9`, `v3.0.0` through `v3.7.0`).
- **Verification baseline tag:** `v2.17.9` (curriculum baseline per CLAUDE.md Current State).
- **Tag SHA reference:**
  - `git rev-parse v2.17.9` → annotated **tag object** SHA `14e48c5da60f8b9089e27234b8d078cae2a59fc7` (PGP-signed).
  - `git rev-parse v2.17.9^{commit}` → **commit object** SHA `0bea06d9957e3966d94c48873cd9afefba1c2677` (the release commit, 2024-02-08, "Set release date for 2.17.9").
  - Curriculum should always cite the **commit SHA** because that is the content baseline. The tag SHA only points to the commit through tag annotation.

**OVS coding style affects grep patterns.** OVS C function definitions split the return type and the function name across two lines:

```c
void                                              // line N (return type only)
ovsdb_rbac_insert(const struct ovsdb *db, ...)    // line N+1 (function name in column 0)
```

Therefore the pattern `^[a-z_]+ \*?<fn>\(` (return type + name on the same line) **fails** for most functions. Use `^<fn>\(` instead (name in column 0). This is the source of the false alarms in §0.3.

### 3.2. Per-phase command set (verified working pattern)

Always run from the OVS repo with `v2.17.9` checked out:

```bash
cd C:/Users/voleh/Documents/ovs
git checkout v2.17.9
```

Then:

| Purpose | Command |
|---|---|
| Find function definition + line | `grep -nE "^<fn>" <path>` |
| Confirm function exists anywhere (definition or call) | `grep -rn "\b<fn>\b" <dir>` |
| Verify file presence in baseline | `ls <path>` (working tree after checkout) |
| Verify SHA exists | `git log -1 --format="%H %ai %s" <sha>` |
| First release tag containing a commit | `git tag --contains <sha> \| sort -V \| head -3` |
| First commit that added a file | `git log --diff-filter=A --format="%H %ai %s" -- <path> \| tail -1` |
| Cross-version compare (e.g., feature in v2.5 but not v2.6) | `git show v2.5.0:<path> \| grep <pattern>` and `git show v2.6.0:<path> \| grep <pattern>` |

### Phase Q9: English Style Guide

#### Q9.1. Goal

#### Q9.2. Style guide outline

The artefact contains these sections, each with concrete examples:

1. **Audience.** Vietnamese network engineers, intermediate to advanced English reading proficiency. Comfortable with technical vocabulary. Less comfortable with idiomatic English, dense compound sentences, or Anglo culture references.
2. **Sentence structure.** Default to short declarative sentences (target 15 to 20 words). Avoid stacked subordinate clauses. One technical idea per sentence.
3. **Vocabulary policy.**
   - Prefer plain words over Latinate or idiomatic ones (use "checks" not "ascertains", "fixes" not "remediates", "needs" not "necessitates").
   - Never abbreviate (write "for example" not "e.g.", "that is" not "i.e.", "and so on" not "etc.", "specification" not "spec" except inside code/CLI, "configuration" not "config" except inside identifiers).
   - Keep canonical OVS / OVN / OpenFlow identifiers untranslated and unabbreviated when discussing them as named entities (`ovs-vswitchd`, `ofproto-dpif`, `MFF_CT_ZONE`, `ovsdb-server`, etc.). The Rule 11 §11.1 list still applies as a "keep-as-is" inventory.
4. **Em-dash forbidden.** Per user 2026-04-28. Use a comma, period, colon, parentheses, or a new bulleted list instead. Replaces the existing CLAUDE.md Rule 13. Applies to the entire training program, not only to v3.9.1 fixes.
5. **Punctuation.** Oxford comma always. One space after sentence-ending periods. Use straight quotes ("..."), not curly. Inline code uses backticks. Block code uses fenced blocks with explicit language tag where useful.
6. **Voice and tone.** Active voice preferred. Second person ("you") when speaking to the engineer. First person plural ("we") rarely, only when describing a shared design choice. Never first person singular.
7. **Pedagogy preserved.** Keep the "why before what" doctrine from CLAUDE.md SECOND NORTH STAR. Lead with motivation, then mechanism, then verification. Do not flatten dense Vietnamese explanations into shallow English summaries.
8. **Callout labels.** Replace existing Vietnamese labels with English ones, locked in for grep consistency:
   - `**Hiểu sai:**` becomes `**Common misconception:**`.
   - `**Điểm mấu chốt:**` becomes `**Key point:**`.
   - `**Bối cảnh:**` becomes `**Context:**`.
   - `**Quy tắc:**` becomes `**Rule:**`.
   - `**Lưu ý:**` becomes `**Note:**`.
   - `**Tuyên bố cần kiểm chứng:**` becomes `**Claim to verify:**`.
   - `**Khái niệm:**` becomes `**Concept:**`.
   - Full mapping table goes in the style guide; updated as new labels surface during Q10.
9. **Technical-claim discipline.** Every English explanation that touches an upstream OVS / OVN / OpenFlow detail must still cite the exact source per Rule 14. Translation does not loosen accuracy requirements.
10. **Examples.** Side-by-side examples for each rule: a Vietnamese passage from real curriculum, the recommended English rendering, and one or two non-recommended renderings with reason.

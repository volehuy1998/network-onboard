## 12.2.1 Micro-segmentation: East-West firewall

Legacy DC security uses a **north-south firewall** between the Internet and the DC. East-West traffic is not filtered.

The problem: when a VM is compromised, the attacker can move laterally between VMs and is not blocked.

**Micro-segmentation:** enforce ACLs at the hypervisor layer for every flow between VMs. Default-deny: only the necessary traffic is allowed.

Benefits:
- **Lateral movement is blocked.**
- **Per-workload policy** is granular.
- **Compliance** (PCI-DSS, HIPAA).

---

## 12.2.2 Implementations

**VMware NSX Distributed Firewall:**
- Enforced inside the vSwitch on every ESXi host.
- Security Group is applied per VM.
- Stateful conntrack runs in the hypervisor.
- Multi-cloud: AWS VPC, Azure VNet, ESXi.

**Cisco ACI EPG plus Contract:**
- EPG = a set of VMs that share the same policy.
- Contract = the allowed protocol between two EPGs.
- Default deny. Explicit allow through Contract.
- Enforced at the TOR Nexus 9000.

**OVN ACL plus Port_Group** (Block XIII):
- An ACL row in the NB database.
- Port_Group aggregates ports by tag.
- Stateful through the `ct()` action.
- Enforced at OVS on every chassis.

## 12.2.3 Example: 3-tier app

A tenant 3-tier web app:
- Web: 10.0.1.0/24
- App: 10.0.2.0/24
- DB: 10.0.3.0/24

Micro-segmentation:
- Internet to web: allow 80 and 443.
- Web to app: allow 8080.
- App to db: allow 5432.
- Anything else: DENY.

OVN equivalent:

```bash
[admin]# ovn-nbctl pg-add pg-web web-vm1 web-vm2
[admin]# ovn-nbctl pg-add pg-app app-vm1 app-vm2
[admin]# ovn-nbctl pg-add pg-db db-vm1 db-vm2

[admin]# ovn-nbctl acl-add pg-web from-lport 1000 \
         "outport == @pg-app && tcp.dst == 8080" allow-related
[admin]# ovn-nbctl acl-add pg-app from-lport 1000 \
         "outport == @pg-db && tcp.dst == 5432" allow-related
[admin]# ovn-nbctl acl-add pg-web from-lport 500 "1" drop
```

## 12.2.4 Service chaining with NSH

**Service chaining:** route traffic through a sequence of services:

```
VM → [SF1: Firewall] → [SF2: IDS] → [SF3: WAF] → Destination
```

**NSH** (RFC 8300, January 2018):

```
 ┌──────────────┬──────────────┬────────┐
 │ Outer transp │ NSH (4-byte) │ Payload│
 │ (VXLAN, GRE) │ + context    │        │
 └──────────────┴──────────────┴────────┘
```

NSH carries:
- **Service Path Identifier (SPI):** the ID of the chain.
- **Service Index (SI):** a hop counter (decremented at every SF).
- **Context headers:** metadata.

The SF inspects SI, performs its service, decrements SI, then forwards according to the SFF.

---

## 12.2.5 NSH versus OpenFlow-based chaining

NSH requires the SF to support parsing it, and vendor support is limited. The alternative is **OpenFlow-based chaining**.

```
table=0, in_port=tenant, actions=output:port-to-FW
table=0, in_port=from-FW, actions=output:port-to-IDS
table=0, in_port=from-IDS, actions=output:destination
```

The service chain is encoded in OpenFlow rules.

OVN 22.x and later have an experimental Port_Chain.

---

## 12.2.6 Production deployments

- **VMware NSX:** micro-segmentation plus service insertion through NSX Edge.
- **Cisco ACI:** EPG and Contract plus service graph.
- **OVN:** ACL and Port_Group plus OpenFlow-based chaining.
- **Kubernetes:** Network Policy plus service mesh (Istio).

---

## Key points to remember

First, micro-segmentation is the East-West firewall. It blocks lateral movement and is required for modern security.

Second, three implementations: NSX DFW, ACI EPG/Contract, and OVN ACL/Port_Group.

Third, service chaining needs metadata. NSH (RFC 8300) is the standard, and OpenFlow-based chaining is the alternative.

Fourth, micro-segmentation plus service chaining plus anycast gateway is the complete SDN DC stack.

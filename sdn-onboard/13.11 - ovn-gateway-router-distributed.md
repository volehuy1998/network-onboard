## 13.11.1 Distributed Router versus Gateway Router

OVN supports two clearly different Logical Router models. They differ in how the traffic is processed on the physical chassis.

**Distributed Router (DR)** is the default when you create a Logical Router with `ovn-nbctl lr-add`. The DR is fully implemented on every chassis that has a port attached to it. When pod A on chassis X sends a packet to pod B on chassis Y (different subnet), both chassis X and chassis Y run a slice of the DR pipeline:

- Chassis X: DR ingress pipeline (from pod A), then output port, then a Geneve tunnel to chassis Y.
- Chassis Y: DR egress pipeline, then deliver the packet to pod B.

Advantage: no choke point, real throughput scales linearly with the number of chassis. Drawback: stateful SNAT is not supported (each chassis has its own conntrack, so the reply may land on a different chassis), and integration with the external network is hard (there is no fixed "anchor" chassis).

**Gateway Router (GR)** is created by setting `options:chassis=<chassis_name>` on a Logical Router. The GR runs on one specific chassis only (the anchor chassis). All traffic going through the GR concentrates there:

- East-west intra-cluster traffic still flows through the DR (if there is one), or through the GR if the cluster uses GR everywhere.
- North-south traffic (going out through a `localnet` port) must transit the GR chassis.

Advantage: stateful SNAT works (conntrack is concentrated on one chassis), BGP or OSPF peering with an external router is easy, IPsec tunneling is supported. Drawback: there is a choke point (the GR chassis must process all north-south traffic for the cluster), and a single point of failure (if the GR chassis dies, north-south is down, unless there is an HA chassis group).

**The hybrid DR plus GR architecture (production standard)** is the most common pattern in OVN-Kubernetes and OpenStack Neutron. Deploy one DR for east-west and several GRs for north-south. When a pod sends a packet:

- To another pod inside the cluster, the DR handles it in a distributed way, at line-rate throughput.
- To the external internet, the DR routes to a GR (through a `join switch` in between), and the GR applies SNAT to the external IP.

```
+-----------------------------------------------------------------+
|                       OVN cluster                               |
|                                                                 |
|  pod_A ---> ls_A ---+                                           |
|                     |                                           |
|  pod_B ---> ls_B ---+--> DR (distributed) --> join_switch       |
|                     |          ^                     |          |
|  pod_C ---> ls_C ---+          |                     v          |
|                                |                  GR (gateway)  |
|                                |                     |          |
|                                |                     v          |
|                          east-west          localnet -> external|
|                                                                 |
+-----------------------------------------------------------------+
```

## 13.11.2 The `chassisredirect` port: anchoring traffic to one chassis

When a Logical Router has a gateway port (an LRP configured as a gateway), OVN automatically creates a virtual port of type `chassisredirect` (named with a `cr-` prefix) to represent the gateway on a specific chassis. This is how OVN "anchors" gateway traffic to one chassis.

**How it works:**

1. The LRP carries either `redirect-chassis` or `redirect-chassis-group`, which points to a chassis or to a chassis group.
2. `ovn-northd` creates a `Port_Binding` in SB DB with `type=chassisredirect`, a unique tunnel key, and a binding on the chosen chassis.
3. When a packet traverses the LRP (for example, an intra-cluster packet that needs to leave through the gateway), the pipeline hashes the MAC and IP to select the `cr-` port. It then tunnels through Geneve to the receiving chassis. That chassis applies the gateway logic: SNAT and routing out to the external network.

**Comparison with the other port types:**

- `vif`, the virtual interface of a pod or VM, bound to the chassis where the pod runs.
- `localnet`, a bridge to an external VLAN, no tunnel involved.
- `patch`, links a Logical Switch to another Logical Switch (or links an LSP to an LRP).
- `chassisredirect`, anchors gateway traffic to a specific chassis.
- `l3gateway`, an L3 routing gateway, typically present on a GR.
- `router`, an LRP, the port of a Logical Router.

**Configuring `chassisredirect` through `ovn-nbctl`:**

```
[host]# ovn-nbctl set Logical_Router_Port lrp_external options:redirect-chassis=chassis_gw1
```

Or, with an HA chassis group (automatic failover):

```
[host]# ovn-nbctl ha-chassis-group-add hagrp1
[host]# ovn-nbctl ha-chassis-group-add-chassis hagrp1 chassis_gw1 50
[host]# ovn-nbctl ha-chassis-group-add-chassis hagrp1 chassis_gw2 40
[host]# ovn-nbctl set Logical_Router_Port lrp_external options:redirect-chassis-group=hagrp1
```

`ovn-northd` automatically picks the chassis with the highest priority (50) as active and the chassis with the lower priority as standby. When the active chassis dies (BFD detects the failure), the standby promotes itself to active. This is covered in detail in Part 13.6.

## 13.11.3 GR pipeline, SNAT and routing to the external network

When a Logical Router has `options:chassis` set, `ovn-northd` generates Logical Flows specific to the GR, different from the DR.

**GR ingress pipeline:**

1. `lr_in_admission`, admission control.
2. `lr_in_lookup_neighbor` and `lr_in_learn_neighbor`, ARP and NDP for the gateway.
3. `lr_in_ip_input`, decrement TTL, handle ICMP echo.
4. `lr_in_unsnat`, if the packet matches a SNAT conntrack entry, reverse SNAT before routing.
5. `lr_in_defrag`, defragment.
6. `lr_in_dnat`, DNAT for load balancer or NAT rules.
7. `lr_in_ecmp_stateful` and `lr_in_policy`, routing policy.
8. `lr_in_ip_routing`, look up the routing table and find the next hop.
9. `lr_in_ip_routing_ecmp`, choose an ECMP path.
10. `lr_in_arp_resolve`, resolve the next-hop MAC through ARP.
11. `lr_in_gw_redirect`, in gateway mode, redirect the packet to the `cr-` port on the GR chassis.
12. `lr_in_arp_request`, send an ARP request if the next-hop MAC is unknown.

**GR egress pipeline:**

1. `lr_out_snat`, apply SNAT if a NAT rule of type `snat` matches.
2. `lr_out_egr_loop`, handle egress loopback.
3. `lr_out_delivery`, output the packet to the external network through the `localnet` port.

**SNAT rule in NB DB:**

```
[host]# ovn-nbctl lr-nat-add lr1 snat 203.0.113.1 10.0.0.0/24
```

This creates a SNAT rule. Every packet from `10.0.0.0/24` going out through `lr1` has its source IP rewritten to `203.0.113.1`.

**DNAT for inbound external traffic:**

```
[host]# ovn-nbctl lr-nat-add lr1 dnat_and_snat 203.0.113.5 10.0.0.10
```

This creates a `dnat_and_snat` rule (a floating IP). An external client that connects to `203.0.113.5` is DNATed to `10.0.0.10`, and the reply from `10.0.0.10` is SNATed back to `203.0.113.5`. The two directions are symmetric.

## 13.11.4 GR integration with the external network (`localnet` and BGP)

A GR connects to the external network through a `localnet` port. The `localnet` port is a one-to-one bridge to an external VLAN, with no Geneve tunnel:

```
[host]# ovn-nbctl ls-add ls_external
[host]# ovn-nbctl lsp-add ls_external ln_external
[host]# ovn-nbctl lsp-set-type ln_external localnet
[host]# ovn-nbctl lsp-set-options ln_external network_name=physnet1
[host]# ovn-nbctl lsp-set-addresses ln_external unknown
```

On the GR chassis, OVS has a bridge named `br-ex` (or any other name, depending on configuration) bound to the physical NIC that faces the external network. The mapping `physnet1 -> br-ex` is declared in `external_ids:ovn-bridge-mappings`:

```
[host]# ovs-vsctl set Open_vSwitch . external_ids:ovn-bridge-mappings=physnet1:br-ex
```

The GR is connected to `ls_external` through an LRP, so traffic from the GR going to the external network passes through `localnet` and then the physical NIC.

**BGP and OSPF integration.** OVN 22.03 and later supports **dynamic routing** through integration with FRR (Free Range Routing):

- An FRR daemon runs on the same chassis as the GR.
- FRR peers with an external router over BGP, learns routes, and advertises VIPs and floating IPs from OVN.
- OVN writes routes into `Logical_Router.static_routes`, and `ovn-northd` then generates the corresponding routing Logical Flows.

In OVN-Kubernetes 1.0 and later, BGP integration is implemented through `--enable-multi-network` together with MetalLB or the RouteAdvertisements CRD.

## 13.11.5 The trade-off, DR versus GR

A side-by-side table to help choose between DR and GR:

| Criterion | Distributed Router (DR) | Gateway Router (GR) |
|----------|------------------------|---------------------|
| Number of chassis processing the traffic | Every chassis | One anchor chassis (or one HA group) |
| Stateful SNAT | No (conntrack distributed) | Yes (conntrack centralized) |
| Real east-west throughput | Linear with chassis count | Limited by the GR chassis |
| Real north-south throughput | Needs a GR for help | Limited by the GR chassis NIC |
| BGP peering | Hard (no anchor) | Easy, production standard |
| Single point of failure | None | Yes (needs an HA chassis group) |
| Configuration complexity | Low | High (HA, bridge mapping) |
| Best suited to | East-west intra-cluster | North-south edge gateway |

**Common production strategy:**

- Each cluster has **one DR** for east-west (attached to every internal Logical Switch).
- Each cluster has **two or more GRs** (in an HA pair) for north-south.
- The DR and the GRs are connected through a **join switch** (a dedicated Logical Switch for router-to-router traffic).
- BGP peering happens only on the GR. Routes are propagated back to the DR through a static route.

### Guided Exercise 1: configure a hybrid DR plus GR for a cluster

> **State verification lab:** the output is based on OVN 22.03 documentation. Pending verification on a real lab host.

**Goal:** build a realistic topology with one DR for east-west and one GR for north-south. Observe the traffic flowing through the `cr-` port and through the Geneve tunnel.

**Setup:** two Ubuntu 22.04 nodes with OVN 22.03 (`chassis_compute` and `chassis_gateway`), and an external network `203.0.113.0/24`.

**Steps:**

**1.** Create the topology:

```
[host]# # On chassis_central, create the DR.
[host]# ovn-nbctl lr-add dr1
[host]# ovn-nbctl ls-add ls_web
[host]# ovn-nbctl ls-add ls_db
[host]# ovn-nbctl lrp-add dr1 dr1-ls_web aa:00:00:00:01:01 10.1.0.1/24
[host]# ovn-nbctl lrp-add dr1 dr1-ls_db aa:00:00:00:01:02 10.2.0.1/24
[host]# ovn-nbctl lsp-add ls_web ls_web-dr1
[host]# ovn-nbctl lsp-set-type ls_web-dr1 router
[host]# ovn-nbctl lsp-set-addresses ls_web-dr1 aa:00:00:00:01:01
[host]# ovn-nbctl lsp-set-options ls_web-dr1 router-port=dr1-ls_web
[host]# # Repeat similarly for ls_db-dr1.
```

**2.** Create the GR pinned to `chassis_gateway`:

```
[host]# ovn-nbctl lr-add gr1
[host]# ovn-nbctl set Logical_Router gr1 options:chassis=chassis_gateway
[host]# # Create the join switch that connects the DR and the GR.
[host]# ovn-nbctl ls-add ls_join
[host]# ovn-nbctl lrp-add dr1 dr1-join aa:00:00:00:02:01 100.64.0.1/30
[host]# ovn-nbctl lrp-add gr1 gr1-join aa:00:00:00:02:02 100.64.0.2/30
[host]# # Wire up the join switch.
[host]# ovn-nbctl lsp-add ls_join dr1-join-p
[host]# ovn-nbctl lsp-set-type dr1-join-p router
[host]# ovn-nbctl lsp-set-options dr1-join-p router-port=dr1-join
[host]# # Repeat similarly for gr1-join.
```

**3.** Create the external network and the SNAT rule:

```
[host]# ovn-nbctl ls-add ls_external
[host]# ovn-nbctl lsp-add ls_external ln_external
[host]# ovn-nbctl lsp-set-type ln_external localnet
[host]# ovn-nbctl lsp-set-options ln_external network_name=physnet1
[host]# ovn-nbctl lrp-add gr1 gr1-external aa:00:00:00:03:01 203.0.113.1/24
[host]# # SNAT rule.
[host]# ovn-nbctl lr-nat-add gr1 snat 203.0.113.1 10.0.0.0/16
[host]# # Static route from the DR back to the GR.
[host]# ovn-nbctl lr-route-add dr1 0.0.0.0/0 100.64.0.2
```

**4.** Inspect the `Port_Binding` of type `chassisredirect`:

```
[host]# ovn-sbctl list Port_Binding | grep -A 5 chassisredirect
```

Expected: a `Port_Binding` named `cr-gr1-external` with `type=chassisredirect`, bound to `chassis_gateway`.

**5.** Verify that north-south traffic flows through the `cr-` port:

```
[host]# ovn-trace ls_web 'inport == "pod1" && eth.src == aa:bb:cc:dd:ee:01 \
         && ip4.src == 10.1.0.5 && ip4.dst == 8.8.8.8 && eth.dst == aa:00:00:00:01:01'
```

The output shows the packet path: `pod1` to DR (`dr1`) to the join switch to GR (`gr1`) to SNAT to the `cr-` port to the external network.

**Lessons learned:**

First, the **`chassisredirect` port** is the core mechanism that lets OVN combine the strengths of the DR (east-west scaling) and the GR (stateful north-south). Without it, OVN would have to choose one of the two.

Second, SNAT only works correctly on the GR because conntrack is centralized there. If you try to SNAT on the DR, the reply traffic may land on a different chassis. That chassis has no conntrack entry, so it drops the packet.

Third, a production cluster always deploys the GR under an HA chassis group. A single GR dying means all north-south is down. Part 13.6 (HA chassis group plus BFD) is a hard prerequisite.

**Cleanup:**

```
[host]# ovn-nbctl lr-del gr1
[host]# ovn-nbctl lr-del dr1
[host]# ovn-nbctl ls-del ls_external ls_web ls_db ls_join
```

## 13.11.6 Logical Router ingress pipeline, the 19-stage walkthrough

> **Audit note:** before this expansion, `lr_in_*` was a zero-mention concept in the foundation. This section closes that critical gap.

OVN translates a Logical_Router into a 19+8 stage pipeline (ingress plus egress). It is more complex than the LS pipeline because it must handle L3 routing, NAT, ECMP, ARP resolution, and ICMP generation.

### 13.11.6.1 The 19 LR ingress stages

| T | Stage name | Purpose |
|---|---|---|
| 0 | `lr_in_admission` | Admission check: destination MAC matches the LRP MAC, IP header is valid |
| 1 | `lr_in_lookup_neighbor` | Check `MAC_Binding` cache (the LR neighbor cache) |
| 2 | `lr_in_learn_neighbor` | Install a `MAC_Binding` entry from the incoming ARP or ND |
| 3 | `lr_in_ip_input` | IP input handling (ICMP reply, TTL exceeded, redirects) |
| 4 | `lr_in_dhcp_relay_req` | DHCP relay request processing (if the LR has a `DHCP_Relay` configuration) |
| 5 | `lr_in_unsnat` | Reverse SNAT for reply traffic |
| 6 | `lr_in_defrag` | IP fragment reassembly (LB and NAT need an L4 header) |
| 7 | `lr_in_lb_aff_check` | LB affinity check for the LR LB |
| 8 | `lr_in_dnat` | Apply DNAT |
| 9 | `lr_in_lb` | LR load balancer, VIP to backend |
| 10 | `lr_in_lb_aff_learn` | Learn LB affinity |
| 11 | `lr_in_ecmp_stateful` | ECMP symmetric-hash tracking |
| 12 | `lr_in_nat_hairpin` | NAT hairpin for VIP-to-backend traffic on the same LR |
| 13 | `lr_in_ip_routing_pre` | Pre-routing hook (policy) |
| 14 | `lr_in_ip_routing` | FIB lookup (`Static_Route`) |
| 15 | `lr_in_ip_routing_ecmp` | ECMP path selection |
| 16 | `lr_in_policy` | `Logical_Router_Policy` evaluation (reroute, drop, allow) |
| 17 | `lr_in_policy_ecmp` | Policy ECMP path selection |
| 18 | `lr_in_arp_resolve` | ARP resolve for the next-hop MAC |
| 19 | `lr_in_chk_pkt_len` | Packet length check for PMTUD |
| 20 | `lr_in_larger_pkts` | Generate ICMP Fragmentation Needed or Packet Too Big |
| 21 | `lr_in_gw_redirect` | Gateway chassis redirect (distributed GR) |
| 22 | `lr_in_arp_request` | ARP request emission for an unknown next hop |

(The actual stage count varies slightly between OVN versions. OVN 22.03 has 19 to 23 stages depending on feature flags. The table lists the superset.)

### 13.11.6.2 Template D for `lr_in_ip_routing` (T14), the central LR stage

| Attribute | Value |
|---|---|
| **Table** | T14 logical |
| **Stage name** | `S_ROUTER_IN_IP_ROUTING` in `northd/northd.h` |
| **Purpose** | FIB lookup: longest prefix match, load the next-hop IP and the egress LRP port |
| **Match criteria** | `ip4.dst == <CIDR>` or `ip6.dst == <prefix>` |
| **Typical actions** | `ip.ttl--; reg0 = <next_hop>; reg1 = <src_ip>; eth.src = <lrp_mac>; outport = "<lrp>"; flags.loopback = 1; next;` |
| **Produced by** | `ovn-northd` from `NBDB.Logical_Router.static_routes` plus connected subnets |
| **State input** | Packet with destination IP |
| **State output** | `reg0` is the next-hop IP, `reg1` is the source IP, `outport` is the egress LRP |

**Logical Flow sample:**

```
table=14 (lr_in_ip_routing), priority=49,
  match=(ip4.dst == 10.0.0.0/16),
  action=(ip.ttl--;
          reg8[0..15] = 0;
          reg0 = ip4.dst;
          reg1 = 10.0.0.1;
          eth.src = 40:44:00:00:00:01;
          outport = "lrp-gateway";
          flags.loopback = 1;
          next;)

table=14 (lr_in_ip_routing), priority=49,
  match=(ip4.dst == 192.168.1.0/24),
  action=(ip.ttl--;
          reg0 = 10.200.0.254;    /* next-hop */
          reg1 = 10.200.0.1;      /* src */
          eth.src = 40:44:00:00:00:02;
          outport = "lrp-uplink";
          flags.loopback = 1;
          next;)

table=14 (lr_in_ip_routing), priority=1,
  match=(ip4.dst == 0.0.0.0/0),   /* default route */
  action=(ip.ttl--;
          reg0 = <upstream_gw>;
          outport = "lrp-uplink";
          ...)
```

**Anatomy:**

- `priority` equals the prefix length (the longer prefix wins). The `/0` default route has priority 1.
- `reg0` is the next-hop IP, used later by ARP resolve at T18 (`lr_in_arp_resolve`).
- `reg1` is the source IP (the egress LRP IP), used when generating an ICMP error.
- `flags.loopback = 1` signals that the packet has already been touched by the LR. It prevents an infinite loop.

**Failure modes to break things:**

- **Default route missing.** An outbound packet misses every rule and is dropped. Check with `ovn-nbctl lr-route-list <lr>`.
- **Two routes at the same priority conflict.** OVS behavior is non-deterministic. `ovn-northd` normalizes most cases, but edge cases can still surface.

### 13.11.6.3 Template D for `lr_in_arp_resolve` (T18)

| Attribute | Value |
|---|---|
| **Purpose** | Resolve the MAC for the next-hop IP (`reg0` set at T14) |
| **Match** | `reg0 == <known_nexthop> && outport == "<lrp>"` |
| **Actions** | `eth.dst = <resolved_mac>; next;` |
| **Produced by** | `ovn-northd` from `SBDB.MAC_Binding` plus LSP addresses |

**Logical Flow:**

```
table=18 (lr_in_arp_resolve), priority=100,
  match=(outport == "lrp-uplink" && reg0 == 10.200.0.254),
  action=(eth.dst = f8:ca:b8:aa:11:22; next;)

table=18 (lr_in_arp_resolve), priority=100,
  match=(outport == "lrp-uplink" && reg0 == 10.200.0.253),
  action=(eth.dst = f8:ca:b8:aa:11:33; next;)

table=18 (lr_in_arp_resolve), priority=0,
  match=(ip4),
  action=(reg0 = ip4.dst;
          eth.dst = 00:00:00:00:00:00;
          outport = "<no-arp>";
          next;)  /* fall through, trigger an ARP request in the next stage */
```

**Failure modes to break things:**

- **No `MAC_Binding`, fall through to default.** T22 (`lr_in_arp_request`) emits an ARP request. The original packet is dropped (it is not buffered). The connection retries only if the OS does retransmission.
- **Stale `MAC_Binding`.** The entry in `SBDB.MAC_Binding` has expired but has not been garbage-collected yet. ARP resolve returns the old MAC and the packet is silently dropped. The default timeout is 30 seconds (`MAC_Binding.timestamp` since OVN 22.09).

### 13.11.6.4 Stages `lr_in_chk_pkt_len` and `lr_in_larger_pkts` (PMTUD)

Check whether the packet length exceeds the egress port MTU. If so, generate an ICMP Fragmentation Needed (IPv4) or Packet Too Big (IPv6).

```
table=19 (lr_in_chk_pkt_len), priority=1,
  match=(outport == "lrp-uplink" && ip4),
  action=(check_pkt_larger(1450)->reg9[1]; next;)

table=20 (lr_in_larger_pkts), priority=50,
  match=(ip4 && reg9[1] == 1 /* packet too big */),
  action=(icmp4 {
    icmp4.type = 3;           /* Destination Unreachable */
    icmp4.code = 4;           /* Fragmentation Needed */
    icmp4.frag_mtu = 1450;
    eth.dst = eth.src;
    eth.src = ...;
    ip4.dst = ip4.src;
    ip4.src = <lrp_ip>;
    ...
  };)
```

See Part 19.0 (OVN multichassis and PMTUD) for the related forensic case study.

### 13.11.6.5 Stage `lr_in_gw_redirect` (T21), distributed GR redirect

For a distributed router (DR) that has a `chassisredirect` port (`cr-`), north-south traffic must be redirected to the chassis that holds the `cr-` role. The stage looks like this:

```
table=21 (lr_in_gw_redirect), priority=50,
  match=(outport == "lrp-uplink" && is_chassis_resident("cr-lrp-uplink") == false),
  action=(outport = "cr-lrp-uplink"; next;)
```

When the packet enters a chassis that is not the gateway chassis, `is_chassis_resident` is false, so the pipeline reroutes the packet to the `cr-` port. The packet then travels through a Geneve tunnel to the right chassis to apply SNAT.

## 13.11.7 Logical Router egress pipeline, 7 stages

### 13.11.7.1 The list

| T | Stage name | Purpose |
|---|---|---|
| 0 | `lr_out_chk_dnat_local` | Check local DNAT (skip SNAT if DNAT was applied locally) |
| 1 | `lr_out_undnat` | Reverse DNAT for outbound replies |
| 2 | `lr_out_post_undnat` | Post-undnat cleanup |
| 3 | `lr_out_snat` | Apply SNAT (rewrite the source IP) |
| 4 | `lr_out_post_snat` | Post-SNAT cleanup |
| 5 | `lr_out_egr_loop` | Egress loop prevention |
| 6 | `lr_out_delivery` | Final `output;` to the LSP |

### 13.11.7.2 Template D for `lr_out_snat` (T3)

| Attribute | Value |
|---|---|
| **Purpose** | SNAT: rewrite the source IP before output |
| **Match** | `ip4.src == <internal_cidr> && outport == "lrp-uplink"` |
| **Actions** | `ct_snat(<external_ip>);` |
| **Produced by** | `NBDB.Logical_Router.nat` with type `snat` or `dnat_and_snat` |

```
table=3 (lr_out_snat), priority=162,
  match=(ip4.src == 10.0.0.0/16 && outport == "lrp-uplink"),
  action=(ct_snat(203.0.113.100);)
```

**Failure modes to break things:** SNAT works, but the reply does not return. The CT entry does not match, the reverse DNAT fails, and the connection hangs. Common causes:

- A non-gateway chassis applied SNAT (it must go through `lr_in_gw_redirect` first).
- An HA chassis group failover has not yet synchronized the conntrack entries.

### 13.11.7.3 Stage `lr_out_undnat` (T1)

Undo DNAT for the reply traffic of a previously DNATed connection:

```
table=1 (lr_out_undnat), priority=100,
  match=(ct.est && ct.rpl && ct_mark.natted == 1),
  action=(ct_commit { ct_mark.natted = 1; };
          next;)
```

### 13.11.9.1 The option `reside-on-redirect-chassis`, distributed gateway routing

**Bucket:** Logical Router Port option.
**Field:** `Logical_Router_Port.options:reside-on-redirect-chassis`
**Value:** `true` or `false` (default `false`)
**Effect:** when `true`, traffic through this LRP must traverse the chassis hosting the `cr-*` port (centralized). When `false` (default), routing is processed in a distributed way on the source chassis.

**Operational interpretation:**

By default, an OVN logical router is **distributed**. Every chassis running `ovn-controller` has a copy of the LR pipeline and can route locally. When an LRP has `gateway-chassis` or `ha-chassis-group` set, OVN creates a **`chassisredirect` (`cr-*`)** port to identify "the active chassis hosting this gateway port". The routing decision is still distributed: the source chassis processes the LR ingress pipeline and only redirects at the `LR_IN_GW_REDIRECT` stage if needed.

Setting `reside-on-redirect-chassis=true` changes the behavior: **all traffic** through the LRP must tunnel to the chassis hosting `cr-*`, and the routing decision happens on the central chassis. Use cases include:

- **Stateful NAT requirement.** SNAT must track conntrack centrally. Distributed routing does not work for SNAT with an ALG (FTP, SIP).
- **Egress traffic shaping.** The central chassis is the single shaper.
- **External monitoring.** The central chassis is the single place that exports NetFlow or sFlow.

**Trade-off:**

| Mode | Latency | Throughput | Resilience |
|------|---------|------------|------------|
| `reside-on-redirect-chassis=false` (default, distributed) | Lower (no extra tunnel hop) | Higher (parallel chassis) | Higher (one chassis failure affects one LSP only) |
| `reside-on-redirect-chassis=true` (centralized) | Higher (one tunnel hop) | Bottlenecked on a single chassis | Lower (failure of the `cr-*` chassis brings the whole LRP down) |

**Set and verify:**

```bash
# Set centralized routing on the LRP.
ovn-nbctl set Logical_Router_Port lrp-gw options:reside-on-redirect-chassis=true

# Verify.
ovn-nbctl get Logical_Router_Port lrp-gw options
# Output: {gateway-mtu="1500", reside-on-redirect-chassis="true"}

# Verify the active chassis for the cr-* port.
ovn-sbctl find Port_Binding logical_port=cr-lrp-gw
# Output: Port_Binding with chassis=<active chassis UUID>
```

**Anti-pattern:** setting `reside-on-redirect-chassis=true` on every LRP centralizes the whole cluster and gives up the scale advantage of distributed routing. Use it only on the LRP that needs stateful NAT or centralized policy.

### 13.11.9.2 Table `Logical_Router_Policy`, policy-based routing

**Bucket:** NB schema for policy-based routing.
**Schema:** the `Logical_Router_Policy` table in NB DB.
**Main columns:**

- `priority` (1 to 32767): a higher priority matches first.
- `match` (string): an expression-based match (similar to ACL).
- `action` (`allow`, `drop`, `reroute`, `jump`): the verdict.
- `nexthop` (IP, optional): the next hop if `action=reroute`.
- `nexthops` (set of IPs, optional): ECMP next hops if `action=reroute` with multipath.
- `options` (map of string to string): metadata and miscellaneous knobs.

**Operational interpretation:**

Policy-based routing (PBR) bypasses standard longest-prefix match. The operator defines a rule of the form "if traffic matches X, route through Y", instead of relying entirely on the prefix table.

In the LR pipeline (see Part 13.16 Â§13.16.6), the stage `LR_IN_POLICY` (table 13) evaluates the `Logical_Router_Policy` table BEFORE the stage `LR_IN_IP_ROUTING` (table 11). On a match, the action runs (allow, drop, or reroute), and the default routing is skipped.

**Action semantics:**

- `allow`: continue the pipeline with the default routing.
- `drop`: drop the packet.
- `reroute`: override the next hop with the `nexthop` field.
- `jump`: jump to another labeled policy (chained policy rules).

**PBR use cases:**

- Source-based routing: traffic from `192.168.10.0/24` exits through gateway A, traffic from `192.168.20.0/24` exits through gateway B.
- DSCP-based routing: traffic with DSCP 46 (voice) takes the low-latency path.
- Selective drop: drop traffic to a blacklisted destination, even if the routing table has an entry for it.

**Example:**

```bash
# Source-based routing.
ovn-nbctl lr-policy-add lr-main 100 \
  'ip4.src == 192.168.10.0/24' \
  reroute 10.0.1.1

# DSCP voice priority.
ovn-nbctl lr-policy-add lr-main 200 \
  'ip.dscp == 46 && ip.proto == 17' \
  reroute 10.0.2.1

# List the policies.
ovn-nbctl lr-policy-list lr-main
```

**ECMP for a policy.** Set `nexthops` (plural) instead of `nexthop`:

```bash
ovn-nbctl lr-policy-add lr-main 100 \
  'ip4.src == 192.168.10.0/24' \
  reroute 10.0.1.1,10.0.1.2,10.0.1.3
# A 5-tuple hash chooses one of the three next hops.
```

### 13.11.9.3 Table `Logical_Router_Static_Route`, ECMP detail

**Bucket:** NB schema for static routing.
**Schema:** the `Logical_Router_Static_Route` table in NB DB.
**Main columns:**

- `ip_prefix` (CIDR): the destination prefix.
- `nexthop` (IP): the next-hop address.
- `policy` (`dst-ip` or `src-ip`, optional): the hash policy for ECMP.
- `output_port` (LRP, optional): force the egress port.
- `route_table` (string, optional): a named route table for VRF.
- `bfd` (BFD UUID, optional): BFD integration for fast failover.

**ECMP behavior:**

When several static routes share the same `ip_prefix` and the same `route_table` but have different `nexthop` values, OVN treats them as ECMP:

```bash
# Create three ECMP routes to the prefix 10.20.0.0/16.
ovn-nbctl lr-route-add lr-main 10.20.0.0/16 10.0.1.1
ovn-nbctl lr-route-add lr-main 10.20.0.0/16 10.0.1.2
ovn-nbctl lr-route-add lr-main 10.20.0.0/16 10.0.1.3
```

The `LR_IN_IP_ROUTING_ECMP` stage (table 12, see Part 13.16) hashes the 5-tuple to choose one of the three next hops. The symmetric reply takes the same next hop, thanks to the `ECMP_STATEFUL` stage (table 7), which stores the source MAC in `ct_label[32:79]` (see Part 13.17 Â§13.17.7).

**The `policy` column** (only for a non-ECMP single route):

- `dst-ip` (default): match by destination prefix.
- `src-ip`: match by source prefix (rare, similar to PBR).

**The `route_table` column for VRF:**

OVN 21.06 and later supports multiple route tables per LR (similar to Linux VRF). The default table is the empty string. Custom table:

```bash
# A route in the table named "tenant-A".
ovn-nbctl --route-table=tenant-A lr-route-add lr-main 10.20.0.0/16 10.0.1.1

# Set the incoming LSP to use this route table.
ovn-nbctl set Logical_Router_Port lrp-tenant-a options:route_table=tenant-A
```

The lookup in `LR_IN_IP_ROUTING` only checks routes whose `route_table` matches the field assigned to the ingress LRP.

**The `bfd` column for fast failover:**

A static route can be linked to a BFD session that monitors the next hop:

```bash
ovn-nbctl --bfd lr-route-add lr-main 10.20.0.0/16 10.0.1.1
# Auto-creates a BFD session monitoring 10.0.1.1. The route is automatically disabled if BFD is down.
```

The failover happens in under one second, compared to roughly 5 to 30 seconds with default ARP-based detection. This is essential for production multipath.

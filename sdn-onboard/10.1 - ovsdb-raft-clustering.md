## 10.1.1 Why Raft clustering is needed

A single-node OVSDB has two production limitations:

1. **Single point of failure.** If the node dies, the entire OVN control plane goes down.
2. **No write scaling.** Only one instance processes transactions.

Raft solves both problems by replicating the log across several nodes, with automatic leader election and failover. OVSDB Raft was introduced in OVS 2.9 (February 2018).

Raft is the consensus protocol published by Diego Ongaro and John Ousterhout at USENIX ATC 2014. It replaces Paxos with simpler semantics while providing equivalent guarantees.

---

## 10.1.2 Raft basics

A cluster has N nodes (N is odd: 3, 5, or 7). At any moment:

- One node is the **Leader** and handles every write from clients.
- The other nodes are **Followers** and replicate the log from the leader.
- When the leader dies, a follower becomes a **Candidate** and a new leader is elected.

**Log replication.** A client sends a write to the leader. The leader appends the entry to its log and replicates it to the followers. When a quorum acknowledges, the leader commits. The quorum is 2 for a 3-node cluster, 3 for a 5-node cluster.

**Fault tolerance:**

| Cluster size | Failures tolerated |
|---|---|
| 3 nodes | 1 node down |
| 5 nodes | 2 nodes down |
| 7 nodes | 3 nodes down |

Clusters of 9 or more are not recommended because write latency rises (the quorum needs more acknowledgements).

## 10.1.3 Bootstrap a 3-node cluster

Compass Chapter N workflow:

**Node 1:**

```bash
[node1]# ovsdb-tool create-cluster /etc/openvswitch/db \
         /usr/share/openvswitch/vswitch.ovsschema tcp:10.0.0.1:6644
[node1]# ovsdb-server --remote=ptcp:6640 \
         --remote=punix:/var/run/openvswitch/db.sock \
         /etc/openvswitch/db &
```

`tcp:10.0.0.1:6644` is the Raft cluster RPC port (different from the client port 6640).

**Node 2:**

```bash
[node2]# ovsdb-tool join-cluster /etc/openvswitch/db Open_vSwitch \
         tcp:10.0.0.2:6644 tcp:10.0.0.1:6644
[node2]# ovsdb-server /etc/openvswitch/db &
```

**Node 3:**

```bash
[node3]# ovsdb-tool join-cluster /etc/openvswitch/db Open_vSwitch \
         tcp:10.0.0.3:6644 tcp:10.0.0.1:6644 tcp:10.0.0.2:6644
[node3]# ovsdb-server /etc/openvswitch/db &
```

A client connects to the cluster with a comma-separated member list:

```bash
[client]# ovn-nbctl --db="tcp:10.0.0.1:6641,tcp:10.0.0.2:6641,tcp:10.0.0.3:6641" \
           ls-add ls0
```

If the leader changes, the client automatically retries against the new leader (transparent failover).

## 10.1.4 Cluster lifecycle management

Check status:

```bash
[node]# ovs-appctl -t /var/run/openvswitch/ovsdb-server.ctl \
         cluster/status Open_vSwitch
```

The output shows the leader ID, the role (leader, follower, candidate), the commit index, the log index, and the members.

**Leave the cluster** (graceful):

```bash
[node2]# ovs-appctl cluster/leave Open_vSwitch
```

**Kick a failed member:**

```bash
[node1]# ovs-appctl cluster/kick Open_vSwitch <sid-of-dead-node>
```

**Important rule (compass Chapter N):** "Once a server leaves a cluster it cannot rejoin. Create a fresh member."

After `leave` or `kick`, **delete the old database file** and `join-cluster` as a new node.

## 10.1.5 Production topology for OVN

An OVN deployment uses two clusters:

- **NB DB cluster** (3 nodes): the logical topology and the configuration intent. Client: `ovn-nbctl`.
- **SB DB cluster** (3 nodes): the computed logical flows plus chassis state. Client: `ovn-controller`.

The two clusters can run on the same 3 physical nodes (as separate processes) to save hardware.

A 5-node cluster fits a large deployment (more than 500 chassis), with higher availability but higher write latency.

## Key points to remember

First, OVSDB Raft (OVS 2.9 and later) is the HA solution for the configuration database. It is required for a production OVN NB and SB DB.

Second, the cluster size N must be odd so the quorum is unambiguous. A 3-node cluster tolerates 1 failure; a 5-node cluster tolerates 2. Very large clusters (9 or more) suffer from high write latency.

Third, the rule "a node that leaves the cluster cannot rejoin" means a fresh join with a new database file is required.

Fourth, the client connects with a comma-separated list and automatically fails over to the new leader. OVN clients support this pattern.

## 10.1.6 Source code: `ovsdb/raft.c`, the internals of OVSDB Raft

This section dives into the source file `ovsdb/raft.c` at the baseline OVS 2.17.9. The file is 5,041 lines and contains the entire Raft state machine implementation.

### 10.1.6.1 Public API architecture and state transitions

`ovsdb/raft.h` defines the public API that `ovsdb-server.c` calls from the main loop. The main groups of functions:

**Lifecycle:**

- `raft_create_cluster()`: bootstrap the first node with a new cluster (a 1-node cluster).
- `raft_join_cluster()`: a new node joins an existing cluster.
- `raft_run()` (`ovsdb/raft.c`, around line 1929): runs one tick of the state machine. Called every tick by the `ovsdb-server` main loop. This is the main entry point.
- `raft_close()`: close the connection and clean up.
- `raft_leave()` and `raft_left()`: leave the cluster on purpose.

**State queries:**

- `raft_is_leader(struct raft *)`: check the current role.
- `raft_is_connected()`: check connectivity to the quorum.
- `raft_is_joining()` and `raft_is_leaving()`: transition state.
- `raft_get_applied_index()` and `raft_get_commit_index()`: log commit and apply indices.
- `raft_get_memory_usage()`: memory used by the log plus snapshot.

**Write API (cluster-replicated transaction):**

- `raft_command_execute()`: send a command (a transaction) to the leader.
- `raft_command_get_status()`: check the command status (committed, aborted, timeout, or not_leader).
- `raft_command_wait()`: poll-block until the command commits or fails.
- `raft_command_unref()`: cleanup.

**State transitions (three roles):**

- `FOLLOWER` (the default at startup): only appends entries from the leader and votes for candidates.
- `CANDIDATE`: when the election timer expires, the node becomes a candidate and requests votes from a majority.
- `LEADER`: once elected, sends `AppendEntries` heartbeats to followers.

Transition functions:

- `raft_become_follower()` (`ovsdb/raft.c`, around line 2633): when the node sees a higher term, or loses the leader (no heartbeat).
- `raft_become_leader()` (`ovsdb/raft.c`, around line 2744): when the node wins a majority of votes in an election.

### 10.1.6.2 Append entry, heartbeat, election

**AppendEntries flow:**

1. The leader sends `raft_append_request` periodically (default heartbeat every 1000 ms) or as soon as a new log entry exists.
2. The function `raft_send_append_request()` (`ovsdb/raft.c`, around line 2672) builds the request with: the current `term`, `prev_log_index`, `prev_log_term`, and the list of `entries[]` (empty for a heartbeat).
3. The follower receives it. `raft_handle_append_request()` (around line 3180) verifies the term and the log consistency, appends the entry to the local log file, and replies with `raft_append_reply` with `success: true` or `false`.
4. The leader collects the replies. When a majority report `success`, it advances `commit_index` and applies the log entry to the in-memory ovsdb state.

**Election timeout:**

- `election_timer` defaults to 1000 to 2000 ms (randomized to avoid split votes).
- A follower that does not receive a heartbeat within this window increments its term, transitions to candidate, and sends `raft_send_vote_request` to every peer.
- Vote rule: a peer votes for a candidate only if (a) the peer has not voted in this term, and (b) the candidate's log is at least as up-to-date as the peer's own (`last_log_term` and `last_log_index`).

**Tunable parameters:**

```bash
[host]# ovs-appctl -t ovsdb-server cluster/change-election-timer Open_vSwitch 5000
```

Raising `election_timer` to 5000 ms is useful for a cross-DC cluster with high RTT (50 to 100 ms on the underlay) to avoid spurious election triggers. Trade-off: real failover is 5 times slower.

### 10.1.6.3 Log compaction and snapshot

The Raft log grows linearly with the number of transactions. After millions of transactions the log file reaches gigabytes, which slows startup and consumes disk space. A snapshot is the compaction mechanism: it replaces the log prefix with "the state applied up to index X".

**Automatic threshold.**

OVSDB Raft proposes a snapshot when:

- The in-memory log size grows above 10 times the previous snapshot size (the trigger condition is internal; there is no public function `raft_log_length` exposed, despite earlier references; verified absent at v2.17.9 by `grep -nE "^raft_log_length" ovsdb/raft.c`); **or**
- Memory usage exceeds 100 MB.

The function `raft_notify_snapshot_recommended()` (around line 4278) raises a flag. The `ovsdb-server` main loop checks the flag and calls `raft_store_snapshot()` (around line 4298) at a safe point in the cycle (not in the middle of a transaction).

**Manual compact:**

```bash
[host]# ovs-appctl -t ovsdb-server ovsdb-server/compact OVN_Northbound
```

This command calls `raft_save_snapshot()` (around line 4087) immediately, writes a fresh snapshot to the `<db>.db` file, and truncates the old log entries. Production deployments typically run a nightly cron job for this.

**Install-Snapshot RPC.**

When a follower has fallen too far behind (the follower has been offline a long time and the log has been compacted), the leader sends `raft_install_snapshot_request` instead of trying to append the old entries:

1. The leader detects: follower's `next_index` is below the cluster's `log_start`. A snapshot transfer is needed.
2. `raft_send_install_snapshot_request()` (around line 3460) sends the snapshot data directly.
3. The follower: `raft_handle_install_snapshot_request()` (around line 4180) stores the snapshot and resets state.
4. `raft_handle_install_snapshot_reply()` (around line 4206) the leader confirms completion and resumes appending entries.

This process consumes bandwidth and CPU. If the cluster has nodes that frequently go offline for more than an hour, the snapshot transfer becomes a bottleneck.

### 10.1.6.4 Leader-election edge cases (split vote, partition)

**Split vote:**

Two candidates in the same term request votes at the same time, and neither reaches a majority. The election timer fires again and both increment the term and request again. To avoid livelock, OVSDB randomizes the election timeout per node (each node draws a random value in 1000 to 2000 ms). In practice the livelock resolves within 1 to 2 rounds.

**Network partition (split-brain):**

A 5-node cluster splits 3 plus 2. The 3-node side (majority) elects a new leader and continues to serve writes. The 2-node side (minority) detects `raft_is_connected() = false` and rejects writes (`raft_command_execute` returns `RAFT_CMD_NOT_LEADER`). When the network recovers, the 2-node side sees a higher term from the 3-node side, calls `raft_become_follower` automatically, and replays the log to catch up.

**Asymmetric partition:**

Node A can send to B, but B cannot send to A (an asymmetric firewall). A sees B as "alive" (it receives heartbeats); B sees A as "dead". Result: B continuously triggers elections even though the cluster is healthy. Detect through `cluster/status`: the term value rises abnormally fast (term: 12, 25, 47 in just a few minutes).

```bash
[host]# ovs-appctl -t ovsdb-server cluster/status Open_vSwitch | grep Term
Term: 47
```

A term that increases by more than 1 per election timer suggests an asymmetric partition. Fix: check the firewall rules in both directions.

**Triage scenarios:**

- `Term` rises from 12 to 47 in 5 minutes: asymmetric partition. Fix: check the firewall (ufw, iptables) for both input and output on ports 6643 and 6644.
- `Status: joining` lasts more than 1 minute: the Install-Snapshot transfer is slow. Fix: check the snapshot size and the network bandwidth between the joining node and the leader.
- `Entries not yet applied` keeps rising: `ovsdb-server` is CPU-bound. Fix: try `--n-threads=4` or reduce the write-side load.
- `Connections` is missing a peer: the peer is down. Fix: check `systemctl status ovsdb-server` on the peer and verify network connectivity.

### Capstone POE: "Raising `election_timer` always improves stability"

> **Hypothetical setup:** an OVN SB DB 5-node cluster spans data centres, with 80 ms p99 RTT between DCs. Current `election_timer` is 1000 ms and there is an occasional false election (1 or 2 per day).

**Claim to verify:** "Raising `election_timer` from 1000 ms to 10000 ms will eliminate false elections."

**Predict:** correct; false elections disappear entirely.

**Observe (hypothetical, pending lab verification):**

- False election rate: 1 to 2 per day to 0 per week. Pass.
- Real failover when the leader crashes: 1 to 2 seconds to 12 to 15 seconds. Fail (6 times slower).
- Client-write-blocked window when the leader fails: 2 seconds to 15 seconds. Fail (user impact).
- Cluster heal time after a partition: 5 seconds to 25 seconds. Fail.

**Explain:**

1. `election_timer` is the upper bound for "no-heartbeat" time before the node suspects the leader is dead. With high RTT (80 ms) plus jitter, 1000 ms is too short and triggers false elections when a heartbeat is late. Raising it removes the symptom.
2. But `election_timer` also defines the upper bound for **detecting that the leader truly died**. A 10-times raise means real failover is 10 times slower.
3. Trade-off: in a production cross-DC cluster with 80 ms RTT, the right `election_timer` is about 3000 to 5000 ms (3 to 5 times RTT, enough buffer for jitter). Going much higher hurts failover speed.

**Falsification test:**

- Set `election_timer` to 10000 ms.
- Observe for a week: how many false elections?
- Force-kill the leader and measure the failover time and the client-write recovery.
- If failover exceeds 15 seconds and client writes are blocked for more than 10 seconds, reduce `election_timer` to a balanced value.

**Where the claim is wrong:** "always improves stability" treats stability as "no false elections". The correct framing is: **stability is the balance between the false-election rate (false positive) and the failover time (true positive). The election timer must be tuned to the actual cluster RTT.** Cross-DC with 80 ms RTT calls for 3000 to 5000 ms; same-DC with 1 ms RTT, the 1000 ms baseline is sufficient.

## 10.1.7 OVSDB Raft cluster, in-depth treatment for the cornerstone keyword

> **Scope:** In-depth treatment (expanded 2026-04-26) for the keyword `OVSDB Raft cluster`. Related sections: §10.0 (OVSDB protocol RFC 7047, the transaction format that the cluster replicates), §10.1.1 to §10.1.5 (overview and bootstrap workflow), §10.5 (cluster performance benchmarking), §10.6 (mTLS and RBAC on cluster connections, file 10.6), §10.1.6 to §10.1.6.5 (source-code internals of `ovsdb/raft.c`, expanded 2026-04), §13.1 (NBDB and SBDB are OVSDB Raft clusters in a production OVN), §13.7 (`ovn-controller` reads SBDB through the cluster), §13.8 (`ovn-northd` writes both NBDB and SBDB through the cluster).

**Concept.** An `OVSDB Raft cluster` is a deployment mode of `ovsdb-server` in which several instances (usually 3 or 5 physical or logical servers) hold a single OVSDB schema together and replicate transactions among themselves through the Raft consensus algorithm. At any moment, exactly one server holds the **leader** role. The leader accepts writes from clients, appends each transaction to its log, and replicates the new log entry to every **follower** through the `Append-Entries` RPC. Only after a majority (the quorum) has written the entry to a persistent log does the leader commit the entry and reply to the client. Cluster size is always odd (3, 5, 7) so that the quorum is unambiguous. This is the official OVSDB high-availability mechanism since OVS 2.9 and the default for a production OVN NBDB and SBDB.

**History and background.** Before OVS 2.9 (released 2018-02-19), OVSDB only supported active-backup replication. One active server accepted writes, one standby server followed along through the `monitor` method. There was no quorum and no automatic leader election. Failover was manual, and the recovery point objective and recovery time objective both depended on operator action. When OVN appeared in OVS 2.6 (2016) and the deployment scaled to hundreds of chassis, the single-server NBDB and SBDB became an obvious single point of failure. The Nicira and Red Hat team (Ben Pfaff, Ilya Maximets) ported Raft from Diego Ongaro and John Ousterhout (USENIX ATC 2014, "In Search of an Understandable Consensus Algorithm"; the 2014 Stanford thesis "Consensus: Bridging Theory and Practice") into `ovsdb/raft.c` (about 5000 lines) and `ovsdb/raft-rpc.c`. OVS 2.9 turned Raft into a production mode. OVS 2.11 (2019) added the `cluster/change-election-timer` runtime tunable. OVS 2.13 (2020) streamlined adding and removing members and added automatic IP-change detection. OVS 2.17 (2022, the curriculum baseline) improved Install-Snapshot streaming. OVN 22.03 made Raft the default for NBDB and SBDB in the `ovn-ctl` script.

**Position in the architecture.** A Raft cluster is a running mode of the `ovsdb-server` process, not a separate process. In an OVN deployment, each controller chassis (3 or 5 hosts) runs two `ovsdb-server` processes: one binds the cluster for NBDB (`OVN_Northbound`), the other binds the cluster for SBDB (`OVN_Southbound`). Each cluster uses an internal Raft port (default 6643 or 6644 depending on the database), separate from the client port (6641 for NBDB, 6642 for SBDB). Clients (`ovn-nbctl`, `ovn-sbctl`, `ovn-northd`, `ovn-controller`) connect through a comma-separated list of cluster members and automatically retry against the new leader after a failover. OVS 2.x also allows a cluster for the local OVSDB (the `Open_vSwitch` schema), but operators rarely use it because the per-host configuration database stays independent.

**Role.** Four main roles: (a) the cluster is the **HA primitive** for the OVN control plane, so losing one host does not bring down NBDB or SBDB. (b) The cluster is a **linearizable write store**: every transaction committed on the cluster has a total order through the log index. (c) The cluster is a **read-scaling option**: a follower can serve reads (with eventual consistency) when the client does not need fresh data. The flag `--leader-only` forces a client to always read from the leader, which gives a linearizable read. (d) The cluster is a **membership manager**: it supports runtime add and remove of servers through `cluster/leave` and `cluster/kick`, which enables a rolling upgrade one node at a time.

**Why it exists.** The active-backup model solved "have a copy of the data" but did not solve "who is the active server after the previous active server died". An operator had to watch manually, promote the standby, and fence the old active. In an OVN environment where hundreds of `ovn-controller` instances reconnect at the same time, manual action is not fast enough. The deployment needs a protocol that automatically elects a leader, guarantees that **at most one leader exists per term**, and guarantees that **every committed transaction survives** every supported failure scenario. Raft meets every requirement, is simpler than Paxos to explain (Ongaro's thesis stresses "understandability"), and fits the OVSDB codebase (about 50,000 lines of C with few large abstractions).

**Problems solved.** Five baseline problems: (1) **Single point of failure.** A single-node OVSDB dying meant the entire control plane died. A 3-node cluster tolerates 1 failure; a 5-node cluster tolerates 2. (2) **Split-brain on a network partition.** Two halves accept writes and diverge. The quorum requirement (a commit needs a majority of acknowledgements) makes sure that only the majority side has a valid leader. (3) **Inconsistent state after a restart.** A server crashes mid-transaction. The Raft log and the applied-index make the replay deterministic. (4) **Slow manual failover.** Failover should happen in seconds, not minutes. The randomized election timeout of 1000 to 2000 ms allows the cluster to detect a dead leader and elect a new one within 1 to 2 seconds. (5) **Membership change race.** Adding or removing a server must not violate the quorum invariant midway. The Raft single-server-change protocol (Section 4 of the Ongaro thesis) commits the configuration change through the same log replication path, which keeps it safe.

**Importance.** This concept is absolutely core to the OVN distributed control plane layer. An engineer who does not understand the OVSDB Raft cluster cannot handle any of the following: (a) any production OVN HA incident (for example, `ovn-northd` failing to connect to NBDB after a partition, or an `ovn-controller` going into the `disconnected` state); (b) tuning `election-timer` for a cross-AZ or cross-region deployment; (c) the compact-log and snapshot lifecycle that prevents the disk from filling up; (d) a rolling upgrade of the NBDB or SBDB cluster through `cluster/leave` followed by a fresh join; (e) the `--leader-only` decision about when a workload truly needs a linearizable read at the cost of latency. Every "OVN at scale" article from Red Hat, Canonical, or Ericsson includes at least one section on Raft tuning, which signals how important this concept is. The level of importance is on par with `ovn-northd` and `Logical_Flow` in the absolute-core group, and slightly below `OVS_ACTION_ATTR_OUTPUT` in the data plane only because Raft sits in the control layer and does not directly affect per-packet performance.

**How it works.** The cluster lifecycle has five phases:

1. **Bootstrap phase 1** (the very first server). The administrator runs `ovsdb-tool create-cluster <db> <schema> <local-addr>`. The function `raft_create_cluster()` writes the database file header with a fresh cluster ID, the server ID for this node, a marker entry at index 0, and term 0. When `ovsdb-server` starts, the function `raft_run()` in `ovsdb/raft.c` notices that the cluster has only one server, automatically elects itself as leader, increments the term by 1, and starts accepting writes.
2. **Bootstrap phase 2** (the next server joins). The administrator runs `ovsdb-tool join-cluster <db> <schema-name> <local-addr> <existing-addr>`. The function `raft_join_cluster()` writes the database file header with the existing cluster ID and the list of remotes. When `ovsdb-server` starts, the new server sends an `Add-Server` RPC to one remote. The remote forwards the request to the leader. The leader appends a membership-change entry to the log, replicates the entry, and commits. The new server then receives a snapshot or a sequence of log entries to catch up.
3. **Steady-state per-write**. A client sends an `OVSDB_TRANSACTION` request through port 6641 or 6642. The leader receives the request, calls `raft_command_execute()` in `ovsdb/raft.c`, allocates a `struct raft_command`, appends the entry to the local log file, and sets `next_index` for each peer. The function `raft_send_append_request()` sends an `Append-Entries` RPC to each peer either periodically (heartbeat) or immediately when a new entry exists. The follower handles the request inside `raft_handle_append_request()`, validates `prev_log_index` and `prev_log_term` consistency, appends to the local log file, calls `fsync`, and replies `success: true`. When the leader sees a majority of acknowledgements, it advances `commit_index`, transitions the command into the `RAFT_CMD_COMMITTED` state, lets `ovsdb-server` apply the entry to the in-memory schema, and replies to the client.
4. **Election phase**. A follower that has not received an `Append-Entries` heartbeat for the duration of `election_timer` (the default range is 1000 to 2000 ms, randomized) increments its term, calls `raft_become_candidate()`, and sends `Request-Vote` to every peer. A peer votes for the candidate when (a) the peer has not yet voted in this term, and (b) the candidate's `last_log_term` and `last_log_index` are not smaller than the peer's own (the election restriction described in Section 5.4 of the paper). When the candidate gathers a majority of votes, it calls `raft_become_leader()` and sends a special heartbeat (a no-op entry that commits any entry from a previous term, ensuring leader completeness as described in Section 6.2). When a follower sees a higher term, it calls `raft_become_follower()` and downgrades automatically.
5. **Snapshot phase**. The log cannot grow forever. When the in-memory log size crosses an internal threshold (roughly ten times the previous snapshot, or memory above about 100 MB), the function `raft_notify_snapshot_recommended()` raises a flag. The main loop checks that flag outside the middle of any transaction, calls `raft_save_snapshot()` to write the current state to the `<db>.db` file, and truncates the log prefix. When a follower has fallen too far behind (its `next_index` is below the cluster's `log_start`), the leader sends an `Install-Snapshot` RPC through `raft_send_install_snapshot_request()` instead of trying to append the old log entries.

**How an operator masters this.** Five reflexes: (1) Read `cluster/status` every morning to catch drift early: a term that grows too fast (asymmetric partition), a log gap that is too large (a slow follower), or `Status: joining` that lasts too long (a slow snapshot transfer). (2) Tune `election-timer` according to RTT: same-rack 1000 ms, cross-AZ with 50 ms RTT use 3000 to 5000 ms, cross-region with 100 ms or more RTT use 5000 to 10000 ms. The command is `ovs-appctl -t .../ovnnb_db.ctl cluster/change-election-timer OVN_Northbound 5000`. (3) Schedule manual compaction: a cron entry such as `0 3 * * * ovs-appctl -t .../ovnnb_db.ctl ovsdb-server/compact OVN_Northbound`, so the log file does not grow above 1 GB. (4) Use the rolling-upgrade workflow: verify cluster health, run `cluster/leave` on node N1, upgrade the package, create a new database file with `ovsdb-tool join-cluster`, start the service, verify the catch-up, and repeat for N2 and N3. (5) Run a periodic failover dry-run: kill the leader in a staging environment, measure failover time and client recovery; if the time exceeds 5 seconds, tune.

**Classification.** The cluster can be classified along several axes: (a) **by cluster size**: 1-server (degraded, development only), 3-server (the production OVN default), 5-server (a large deployment with more than 500 chassis), or 7-server (rare, geo-distributed). (b) **by per-node role**: leader (exactly one), follower (N minus 1), candidate (transient during an election). (c) **by schema**: an NBDB cluster (`OVN_Northbound`), an SBDB cluster (`OVN_Southbound`), an IC_NB or IC_SB cluster for OVN Interconnect, or an `Open_vSwitch` local cluster (rarely used). (d) **by read-consistency mode**: linearizable (through the leader, with `--leader-only`) versus eventual consistency (through any follower, the default). (e) **by deployment topology**: collocated (three controller hosts also run other processes), dedicated (three hosts only run the database), or cross-AZ (three servers in three different availability zones).

**When something goes wrong.** The OVSDB Raft cluster is the cornerstone keyword in the following diagnostic situations: (a) An `ovn-controller` repeatedly logs `unable to connect to OVN_Southbound` even though all three SBDB processes are running. Check `cluster/status` to see whether some server has left the cluster (`Status: disconnected from cluster`). (b) After a network partition that lasted tens of seconds, some client writes report `transaction failed: cluster not yet ready`. The cause is the minority side detecting the loss of quorum and rejecting writes; the majority side still serves. (c) `cluster/status` shows `Term: 256` after starting at `Term: 12` only an hour earlier. This signals an asymmetric partition (a firewall blocking one direction) or an election timer that is too short relative to network jitter. (d) `Log: [125, 4520000]` with millions of log entries means the cluster needs a manual compact (`ovsdb-server/compact`). If the compact hangs, check disk space and filesystem mount options. (e) A new server has been stuck at `Status: joining` for more than 5 minutes. The snapshot transfer is choking. Check the bandwidth between the joining node and the leader, check the MTU, and check the `Backlog` field in `cluster/status`.

**Tightly related to.** The cluster connects to several other concepts: (a) The `ovsdb-server` process (Part 10.0) is exactly the binary that runs in cluster mode instead of standalone. (b) `ovn-northd` (Part 13.8) reads NBDB and writes SBDB through a cluster connection and is a typical client. The `northd` HA pattern is active-passive with its own leader election (through a `Lock` row in SBDB), which is different from the Raft leader. (c) `ovn-controller` (Part 13.7) reads SBDB through the cluster and uses `--leader-only=false` by default so that reads load-balance across followers. (d) `Logical_Flow`, `Port_Binding`, and `Datapath_Binding` (Parts 13.5 and 13.19) are rows that live inside the SBDB cluster, and every change goes through a Raft commit. (e) mTLS and RBAC (Part 10.6) apply to the cluster connection: both the Raft RPC between servers and the client connection can be TLS-protected with a per-server certificate. (f) The snapshot file `<db>.db` is the standard OVSDB database file format (Part 10.0); Raft only adds cluster ID, server ID, term, and log-index metadata in the header. (g) Performance benchmarking (Part 10.5) measures cluster throughput (typically 1000 to 3000 transactions per second on a 3-node cluster, about 50 percent of standalone throughput because of the synchronization overhead).

**Real-world incident.** An OpenStack OVN production deployment with a 5-node SBDB cluster spread across three availability zones in the same region (RTT between zones was 1.2 ms p50, 8 ms p99) saw the `ovn-controller` on 200 chassis continuously log `Unreasonably long XXX ms poll interval` and `binding|INFO|Releasing lport ...` every few minutes. Initial diagnosis: a `cluster/status` dump on each of the 5 SBDB nodes showed the `Term` value rising from 47 to 312 over 4 hours, with a new `become_leader` event every few minutes. Root cause: one availability zone had pushed a firewall update the previous day that accidentally blocked outbound traffic to port 6644 from 2 nodes in that zone; inbound heartbeats were still allowed. As a result, those 2 nodes received heartbeats (so they saw the leader as alive), but the other 2 nodes never received the reply, because the `raft_handle_append_reply` path timed out. The leader thought those 2 nodes were dead, when in reality only the reply path was broken (an asymmetric partition). The leader stepped down when it could no longer reach quorum, and the nodes started triggering elections back-to-back: an election storm. Resolution: revert the firewall rule to allow outbound 6644. The cluster stabilized within 2 minutes (the `Term` value stopped growing) and the `ovn-controller` instances reconnected normally. Lesson: a fast-growing `Term` value in `cluster/status` is the most reliable signal of an asymmetric partition, more reliable than `last msg N ms ago` (which only reflects the inbound direction). Tuning after the incident: raise `election-timer` from 1000 ms to 4000 ms (about 4 times the RTT p99) to reduce sensitivity to jitter, and add a monitoring alert that fires when the `Term` value increases by more than 5 in any one-hour window.

**Common failures and diagnostic signals.**

- *Symptom*: `cluster/status` shows `Status: joining` on a new node and the state does not advance for more than 5 minutes. *Cause*: the Install-Snapshot transfer is slow or blocked, usually because of an MTU mismatch or insufficient bandwidth between the joining node and the leader. *Diagnosis*: run `cluster/status` on the leader to inspect `Backlog`. Run `tcpdump -i any port 6643` to verify the packet flow. Run `ss -i` to inspect the TCP congestion window. *Fix*: verify MTU and path discovery. If the database is large (more than 100 MB), accept the wait. If a bug is suspected, run `ovs-appctl cluster/leave`, then create a fresh database file, then join again.
- *Symptom*: the `Term` value grows quickly (one unit every few seconds). *Cause*: an asymmetric partition or an election timer that is too short relative to network jitter. *Diagnosis*: compare `Connections: ->X <-Y` to see if any direction is missing. Run `iperf3` to measure the actual RTT between the nodes. Run `tcpdump` with filters for ports 6643 and 6644 on both directions. *Fix*: correct the firewall, then raise `election-timer` through `cluster/change-election-timer` to a value that is 3 to 5 times the RTT p99.
- *Symptom*: a client running `ovn-nbctl` reports `database connection failed: cluster not yet ready`, even though 2 of 3 servers are running. *Cause*: the quorum for a 3-node cluster is 2; if 2 nodes restarted at the same time, the leader can be lost during the election window. *Diagnosis*: run `cluster/status` on the 2 surviving servers to see if either is the leader. If both report `Role: candidate`, an election is in progress. *Fix*: wait 1 to 2 election cycles (2 to 4 seconds) for the election to settle. Avoid simultaneous restarts during a rolling upgrade.
- *Symptom*: the database file `ovnnb_db.db` swells past 5 GB after a few months. *Cause*: snapshots are not triggering automatically, or compaction is being skipped because of disk pressure. *Diagnosis*: run `ls -lh /etc/ovn/`. Run `cluster/status` and inspect the `Log: [start, end]` range. *Fix*: run `ovs-appctl ovsdb-server/compact OVN_Northbound`. Consider scheduling a nightly cron job. Verify that the free disk is at least twice the database size, so the compaction has room to work.
- *Symptom*: after `cluster/leave`, an attempt to `cluster/join` again with the same database file fails with `could not join cluster`. *Cause*: the rule that "once a server leaves, it cannot rejoin with the old database file"; the database file holds a "left" marker. *Diagnosis*: a comment in the OVS `compass` chapter and the `ovsdb-tool` man page both confirm this. *Fix*: delete the old database file, then run `ovsdb-tool join-cluster` again with a fresh file.

**Comparison with other systems.** Other systems solve the "consensus-replicated state machine" problem differently:

- **etcd Raft** (built by CoreOS, later used as the engine of the `etcd` KV store): same Raft algorithm, also based on the Ongaro 2014 paper, with the library `go.etcd.io/raft`. The differences are the implementation language (Go versus C) and the data model (a key-value store versus an OVSDB schema-defined relational store). etcd is the consensus backend for many Go-based distributed systems; the Kubernetes control plane is itself an `etcd` cluster. Conceptually equivalent. Operations are similar: `etcdctl member list` corresponds to `cluster/status`.
- **HashiCorp Consul Raft**: same Raft algorithm, library `hashicorp/raft`, used for service discovery and KV. The difference is that Consul exposes additional consistency modes ("stale" and "consistent") that work like the OVSDB `--leader-only` flag but at a finer per-request granularity.
- **MongoDB replica set**: a different protocol. Election and heartbeat are inspired by Raft but the implementation is custom, and the terminology is different (primary, secondary, arbiter). MongoDB supports several consistency modes through `readConcern`: `local`, `majority`, `linearizable`. OVSDB Raft only has two modes (leader-only linearizable, or follower eventual).
- **PostgreSQL streaming replication plus Patroni**: PostgreSQL itself does not implement consensus, only asynchronous or synchronous streaming. Patroni adds a consensus layer that uses an external `etcd` or `Consul` cluster to decide who the primary is. Compared with OVSDB's built-in Raft, this two-layer architecture is more complex but more flexible (the consensus backend can be swapped).
- **Apache ZooKeeper (ZAB protocol)**: before Raft, many distributed systems used ZAB (ZooKeeper Atomic Broadcast) for consensus. ZAB and Raft are equivalent in fault tolerance and correctness, but ZAB is harder to explain. Raft was created in part to address that. Yahoo, LinkedIn, and Twitter used to run ZooKeeper widely; the industry trend after 2015 has been toward etcd and Raft.
- **Galera Cluster (MariaDB, MySQL)**: synchronous multi-master through certification-based replication, not Raft. Each node accepts writes, then broadcasts a certification request to every peer before committing. The difference from Raft is that there is no single leader, which can yield higher throughput for read-mostly workloads, but deadlock detection becomes more complex.

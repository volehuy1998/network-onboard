## 3. Source verification methodology

This section is materially identical to plan v3.9.2 §3, which itself reproduces plan v3.9.1 §3. The method is what v3.9.3 inherits. Reproduced here in the minimum form needed for self-containment so a future session can execute v3.9.3 without context-switching back to the predecessor plans.

### 3.1. Tools and pinning

- **Local OVS repo.** `C:\Users\voleh\Documents\ovs`. All release tags fetched (v2.5.0, v2.6.0, v2.12.0, v2.17.9, v3.0.0 through v3.7.0).
- **Verification baseline tag.** `v2.17.9` (curriculum baseline per CLAUDE.md Current State).
- **Tag SHA reference.**
  - `git rev-parse v2.17.9` returns the annotated tag object SHA `14e48c5da60f8b9089e27234b8d078cae2a59fc7` (PGP-signed).
  - `git rev-parse v2.17.9^{commit}` returns the commit object SHA `0bea06d9957e3966d94c48873cd9afefba1c2677` (the release commit, dated 2024-02-08, message "Set release date for 2.17.9").
  - Curriculum cites the commit SHA, because the commit is the content baseline. The tag SHA only points to the commit through tag annotation.

**OVS coding style affects grep patterns.** OVS C function definitions split the return type and the function name across two lines:

```c
void                                              /* line N (return type only)        */
ovsdb_rbac_insert(const struct ovsdb *db, ...)    /* line N+1 (function name col 0)   */
```

The pattern `^[a-z_]+ \*?<fn>\(` (return type plus name on the same line) fails for most OVS functions. Use `^<fn>\(` (or `^<fn>` for a slightly looser match) to anchor on the function name in column 0. This is the source of the false alarms in the original v3.9.1 audit.

### 3.2. Per-phase command set (verified working pattern)

Always run from the OVS repo with v2.17.9 checked out:

```bash
cd C:/Users/voleh/Documents/ovs
git checkout v2.17.9
```

Then:

| Purpose | Command |
|---|---|
| Find function definition plus line | `grep -nE "^<fn>" <path>` |
| Confirm function exists anywhere (definition or call) | `grep -rn "\b<fn>\b" <dir>` |
| Verify file presence in baseline | `ls <path>` (working tree after checkout) |
| Verify SHA exists | `git log -1 --format="%H %ai %s" <sha>` |
| First release tag containing a commit | `git tag --contains <sha> \| sort -V \| head -3` |
| First commit that added a file | `git log --diff-filter=A --format="%H %ai %s" -- <path> \| tail -1` |
| Cross-version compare | `git show <tag>:<path> \| grep <pattern>` |
| Search history for a feature | `git log --all --format="%H %ai %s" --grep="<keyword>" -- <path>` |
| Pickaxe search for a literal string | `git log --all -S "<literal>" -- <path>` |
| Verify struct definition | `grep -nE "^struct <name>" <path>` |
| Verify macro constant | `grep -nE "^#define <NAME>" <path>` |

### 3.4. Pre-commit hooks

Each commit runs the following four staged-only checks via `.git/hooks/pre-commit`:

- `python scripts/em_dash_check.py --staged` (must exit 0; rejects U+2014 in newly added or modified lines).
- `python scripts/lang_check.py --staged` (must exit 0; rejects Vietnamese-detected prose in newly added or modified lines, strict mode).
- `python scripts/anti_gaming_check.py --staged` (must exit 0; GP-6 to GP-10 enforcement).
- `python scripts/rubric_leak_check.py --staged` (must exit 0 within OVS scope; GP-11 enforcement).

After Phase R6, all four checks run on `--all` for the final regression audit.

### 3.5. Anti-pattern signals

Stop and rewrite the step if:

- Pre-edit grep was not run against the local v2.17.9 working tree.
- Edit was based on a function name from memory or from a v3.9, v3.9.1, or v3.9.2 commit body that was not re-verified.
- Commit body lacks a reproducible command and verbatim output.
- A Form A commit touches more than 1 file or more than 1 cohesive section.
- A Form B batch exceeds 5 files.
- The mid-batch checkpoint result is skipped or not documented.

### 3.6. GP-13 in-commit English rewrite obligation

Every commit that modifies a curriculum file must rewrite the section that contains the fix into English in the same commit, per GP-13 (added in plan v3.9.1 Q12). The rules for the rewrite, lifted from plan v3.9.1 §Q10.2 and reused verbatim in plan v3.9.2 §3.6:

- **Section granularity.** If a fix touches a line inside a level 3 heading, the entire level 3 section is rewritten to English. Sibling sections at the same level are not rewritten (out of scope for the fix).
- **Add language-status callout.** First commit that touches a file also adds the `**Language status:**` callout immediately after the H1 heading, listing which sections are English. Subsequent commits that touch the same file update the callout's section list.
- **Preserve technical citations.** All function names, file paths, line numbers verified during audit must be preserved in the English rewrite. Translation does not loosen Rule 14.
- **Preserve pedagogical structure.** Keep the axis-by-axis or paragraph-by-paragraph structure. Do not rearrange sections during translation.
- **Preserve verbatim quotes.** Any verbatim quotes from upstream OVS source code, NEWS file, RFC, or man page stay verbatim. They are evidence, not prose.
- **Em-dash forbidden** in every newly written English section. The pre-commit hook flags em-dash in newly added or modified lines.

### 3.7. Mid-batch escalation gate (per plan v3.9.1 §Q7.3)

After R1 closes (the first batch of 2 files), and after each subsequent batch (R2, R3, R4, R5), compute the error rate as:

```
error_rate = (count of confirmed findings in batch) / (count of verified citations in batch)
```

If `error_rate > 0.50` (50 percent), halt v3.9.3 immediately and escalate to a hypothetical plan v3.9.4 (broader scope, likely covering additional non-cornerstone files or tighter governance). Document the halt in `memory/sdn/v3.9.3-mid-checkpoint-halt-YYYY-MM-DD.md` and notify the user with a concrete count.

If `0.30 < error_rate <= 0.50`, continue with the next phase but flag for tighter review at R6.

If `error_rate <= 0.30`, continue with the next phase normally. The 30 percent prior from v3.9.2 R2.A and the 31 percent prior from v3.9.1 indicate this branch is the most likely.

The same error-rate computation is repeated after R2, R3, R4, and R5 close. The threshold is the same. Halt at any phase if the threshold is exceeded.

Document the running error rate inline in each R-phase audit log file.

## Mục tiêu bài học

Sau khi hoàn thành phần này, kỹ sư có thể:

1. **Hiểu (Understand)** vai trò của OVN Inter-Connect: federated multi-region deployment cho phép VM ở region A nối với VM ở region B qua **Transit Switch** (cross-region L2) hoặc **Transit Router** (cross-region L3) mà KHÔNG cần stretch một cụm SB DB duy nhất qua mọi region.

## 13.15.1 Vì sao cần Inter-Connect — bài toán scale geographic + regulatory

**Single-region OVN limit:** một cụm SB DB Raft 3-node (xem Phần 10.1) có thể serve hàng trăm chassis trong 1 datacenter. Khi bạn muốn chassis ở DC khác (geographic redundancy, regulatory data sovereignty, edge computing) cùng L2 overlay với chassis ở DC chính, có 2 lựa chọn:

**Lựa chọn 1: Stretch single SB DB cluster across regions.**
- Đặt 3 node Raft ở 3 DC khác nhau, tunnel SB DB protocol cross-region.
- Vấn đề:
  - Latency Raft consensus = max(round-trip giữa 3 node). Cross-region 100-300ms → mọi `ovn-nbctl` thay đổi cấu hình mất ~1-5 giây để propagate.
  - Network partition giữa region = Raft quorum loss = whole cluster freeze (split-brain protection).
  - Compliance: data sovereignty (GDPR, PIPL) yêu cầu data từng region không leak sang region khác — nhưng SB DB single cluster lưu data toàn cụm.

**Lựa chọn 2: Inter-Connect (IC) federated deployment.**
- Mỗi region có cụm SB DB ĐỘC LẬP (tự Raft 3-node nội region).
- Một cụm **IC central** (IC_NB + IC_SB) làm "kết nối" giữa các region.
- `ovn-ic` daemon ở mỗi region đọc IC_SB, sync chỉ những resource cross-region (Transit Switch, route info) sang local SB DB.
- Mỗi region tự xử lý tenant local; chỉ traffic cross-region đi qua IC.

**Trade-off so với stretch:**

| Tiêu chí | Stretch single SB | OVN Inter-Connect |
|----------|-------------------|-------------------|
| Latency cấu hình | Cao (cross-region Raft) | Thấp (Raft nội region) |
| Failure isolation | Toàn cụm chết nếu mất quorum | Chỉ region chết riêng |
| Data sovereignty | Khó (data toàn cụm) | Dễ (data per-region) |
| Scale max chassis | ~500-1000 | ~10K+ (tổng các region) |
| Complexity setup | Đơn giản (1 cluster) | Phức tạp (4 DB + 2 daemon type) |
| Cross-region first-packet latency | Thấp | Cao hơn (Geneve transit tunnel) |

OVN Inter-Connect được giới thiệu trong OVN 20.06, ổn định OVN 22.03+. Use case chính: cloud provider multi-DC, telecom multi-edge, enterprise hybrid cloud (on-prem + AWS).

> **Lưu ý phiên bản:** Curriculum bám OVN 22.03.8 baseline. IC schema có thay đổi nhỏ giữa 22.03 và 24.03 (thêm `nb_cfg` synchronization tracking, thêm option encap=vxlan). File này focus 22.03.

## 13.15.2 Kiến trúc 4-database

```
                    +-----------------------------+
                    |    IC central cluster       |
                    |  +----------+ +----------+  |
                    |  |  IC_NB   | |  IC_SB   |  |
                    |  |  cluster | |  cluster |  |
                    |  |  (3-node)| |  (3-node)|  |
                    |  +----------+ +----------+  |
                    |        |          ^        |
                    |        v          |        |
                    |    [ovn-ic-northd]         |
                    |    (compile NB → SB cho IC)|
                    +-----------------------------+
                            |              ^
                            | IC_NB write  | IC_SB read
                            | + IC_SB read | + write Chassis claim
                            v              |
                    +---------+      +---------+
                    | Region A|      | Region B|
                    |         |      |         |
                    | +-----+ |      | +-----+ |
                    | |ovn- | |      | |ovn- | |
                    | |ic   | |      | |ic   | |
                    | +-----+ |      | +-----+ |
                    |   |     |      |   |     |
                    |   v     |      |   v     |
                    | +---+   |      | +---+   |
                    | |NB |   |      | |NB |   |
                    | |+SB|   |      | |+SB|   |
                    | +---+   |      | +---+   |
                    | (region |      | (region |
                    |  local  |      |  local  |
                    |  cluster|      |  cluster|
                    +---------+      +---------+
                         |                |
                    [chassis]         [chassis]
                    [+VM]             [+VM]
                            ↕  Geneve transit ↕
                            (cross-region tunnel)
```

**4 database role:**

| Database | Cluster location | Purpose | Daemon writer |
|----------|------------------|---------|----------------|
| **OVN_Northbound (NB)** | Per-region | Tenant intent local region | CMS (Neutron, OVN-K8s) qua `ovn-nbctl` |
| **OVN_Southbound (SB)** | Per-region | Logical Flow + Port_Binding cho chassis local | `ovn-northd` per-region |
| **IC_NB (Inter-Connect Northbound)** | Central | Transit Switch + AvailabilityZone definition | Operator qua `ovn-ic-nbctl` |
| **IC_SB (Inter-Connect Southbound)** | Central | Datapath_Binding + Port_Binding cross-region + Route | `ovn-ic-northd` (compile) + `ovn-ic` (sync) |

**Data flow direction:**

1. Operator tạo Transit Switch trong IC_NB qua `ovn-ic-nbctl ts-add transit1`.
2. `ovn-ic-northd` compile IC_NB → IC_SB: tạo Datapath_Binding cho transit1, allocate tunnel_key cross-region.
3. `ovn-ic` ở mỗi region đọc IC_SB, sync Datapath_Binding của transit1 vào local SB DB của region đó.
4. `ovn-controller` ở mỗi chassis đọc local SB, biết về transit1, cài flow trên `br-int` để tunnel sang region khác.

## 13.15.3 Daemon — `ovn-ic` per-region + `ovn-ic-northd` central

### `ovn-ic` daemon

Per-region daemon. Chạy trên một server quản trị trong mỗi region (không phải mỗi chassis).

**Nhiệm vụ:**
- Đọc IC_SB tập trung (Datapath_Binding, Port_Binding cross-region, Route).
- Sync xuống local SB DB của region.
- Ghi ngược chassis claim + route advertise lên IC_SB.

**Synopsis:**

```
ovn-ic [options]
```

**Options chính (verified man page `ovn-ic(8)`):**

| Option | Mục đích |
|--------|----------|
| `--ovnnb-db=<db>` | Local NB DB endpoint (default `unix:/ovnnb_db.sock` hoặc `$OVN_NB_DB`) |
| `--ovnsb-db=<db>` | Local SB DB endpoint |
| `--ic-nb-db=<db>` | IC_NB DB endpoint (default `unix:/ovn_ic_nb_db.sock`) |
| `--ic-sb-db=<db>` | IC_SB DB endpoint |
| `--pidfile`, `--detach` | Daemon lifecycle |
| `--monitor` | Spawn monitor process cho crash recovery |
| `--unixctl=<socket>` | Control socket cho `ovn-appctl` |
| `-p/-c/-C` SSL options | mTLS connection tới IC central |

**Runtime control qua `ovn-appctl`:**

```
ovn-appctl -t ovn-ic exit       # Graceful stop
ovn-appctl -t ovn-ic pause      # Stop processing, release SB lock
ovn-appctl -t ovn-ic resume     # Resume operations
ovn-appctl -t ovn-ic is-paused  # Query status
ovn-appctl -t ovn-ic status     # Show active/standby/paused
```

**HA model:** nhiều `ovn-ic` instance có thể chạy đồng thời trong cùng region, OVSDB lock đảm bảo chỉ 1 active. Standby tự động promote khi active fail.

### `ovn-ic-northd` daemon

Central daemon. Chạy trên server quản trị IC central (cùng nơi với IC_NB + IC_SB cluster, hoặc riêng).

**Nhiệm vụ:** compile IC_NB → IC_SB. Tương tự `ovn-northd` cho local NB→SB nhưng cho cấp IC.

**Workflow compile:**
1. Đọc IC_NB Transit_Switch, AvailabilityZone, Route.
2. Allocate tunnel_key cho mỗi Transit_Switch (unique cross-region).
3. Tạo Datapath_Binding entry trong IC_SB cho mỗi Transit_Switch.
4. Tạo Port_Binding entry trong IC_SB cho mỗi LSP nối với Transit Switch.
5. Compile route từ Route table vào Path_Binding (cho Transit Router L3).

## 13.15.4 Transit Switch — cross-region L2 bridge

**Transit Switch** là entity central, định nghĩa "L2 broadcast domain mở rộng qua nhiều region". VM ở region A bind vào Transit Switch qua LSP type=remote → tunnel qua region B → VM region B nhìn thấy như cùng L2.

**Workflow vận hành:**

1. **Operator tạo Transit Switch trong IC_NB:**
   ```bash
   ovn-ic-nbctl ts-add ts-tenant-blue
   ```

2. **Operator bind logical switch của region A vào Transit Switch:**
   ```bash
   # Trên region A NB DB
   ovn-nbctl lsp-add ls-blue-rA ls-blue-rA-to-ts -- \
     lsp-set-type ls-blue-rA-to-ts router -- \
     lsp-set-addresses ls-blue-rA-to-ts router -- \
     lsp-set-options ls-blue-rA-to-ts \
       router-port=lrp-blue-rA \
       requested-tnl-key=<allocated_by_ic_northd>
   ```

3. **Tương tự region B bind vào Transit Switch:**
   ```bash
   # Trên region B NB DB
   ovn-nbctl lsp-add ls-blue-rB ls-blue-rB-to-ts -- ...
   ```

4. **`ovn-ic-northd` central tạo binding trong IC_SB:** Datapath_Binding cho ts-tenant-blue + Port_Binding remote cho cả 2 region's LSP.

5. **`ovn-ic` ở mỗi region sync IC_SB:** read Datapath_Binding của ts-tenant-blue + Port_Binding remote → cài vào local SB DB.

6. **`ovn-controller` ở chassis nhận flow:** từ local SB, biết về Transit Switch + remote LSP. Cài flow trên `br-int` để encap traffic destination = remote LSP qua Geneve transit tunnel sang region khác.

**Tunnel transit:**

Cross-region tunnel dùng Geneve (default), VXLAN (optional). Tunnel endpoint = Chassis IP của region khác. Đặc biệt: chassis SOURCE ở region A KHÔNG kết nối trực tiếp tới chassis TARGET ở region B; cross-region tunnel đi qua **gateway chassis** dedicated mỗi region (cấu hình qua `external_ids:ovn-is-interconn=true`).

## 13.15.5 Transit Router — cross-region L3 routing

Transit Switch giải quyết L2 cross-region. Cho L3 (cross-subnet routing across region), cần **Transit Router** + **AvailabilityZone Route**.

**Concept:**

- Mỗi region được represent là một `AvailabilityZone` (AZ) trong IC_NB.
- Route giữa AZ được advertise qua IC_SB Route table.
- Transit Router là entity logic kết nối nhiều AZ qua route table.

**Static route propagation (KHÔNG dùng BGP — per North Star BAN):**

OVN Inter-Connect support **static route** qua IC_NB Route table. Operator manually advertise prefix:

```bash
ovn-ic-nbctl --ic-nb-db=tcp:<ic-server>:6645 \
  add Route az-A 192.168.0.0/16 az-B-tunnel-endpoint
```

Route được sync xuống local SB của AZ tương ứng. `ovn-controller` cài flow phù hợp.

> **BAN note:** OVN cũng support **BGP-based dynamic route advertisement** cho IC qua `ovn-bgp-agent` (separate component). Curriculum KHÔNG cover deep BGP integration per PERMANENT BAN directive (xem CLAUDE.md North Star). Học viên muốn deep BGP-EVPN background xem Phần 11.2 cho overview, không tham khảo BGP control plane chi tiết.

## 13.15.7a Treatment chi tiết 5 keyword cốt lõi của OVN Inter-Connect

Năm sub-section bên dưới đào sâu 5 keyword nền tảng để engineer hiểu cấu trúc liên thông giữa CLI, schema, daemon, và pipeline thực thi. Mỗi keyword được bóc tách qua các góc nhìn: khái niệm, lịch sử, vị trí kiến trúc, vai trò, động cơ, vấn đề giải quyết, tầm quan trọng, cơ chế, cách dùng, phân loại, workflow, sự cố, liên kết chéo, khác biệt phiên bản, quan sát, source code, ca thực tế, bài tập, lỗi thường gặp, và so sánh hệ khác.

### 13.15.7a.1 `ovn-ic-nbctl ts-add` — sub-command tạo Transit Switch

**Khái niệm.** `ts-add` là sub-command của `ovn-ic-nbctl` insert một row mới vào bảng `Transit_Switch` của IC_NB DB. Không phải là daemon, không phải là OVSDB native CLI; đây là command-line wrapper convenience.

**Lịch sử + bối cảnh.** Xuất hiện cùng OVN 20.06 khi Inter-Connect được merge upstream (ic-nbctl utility ra đời cùng). Trước đó operator phải dùng `ovn-ic-nbctl create Transit_Switch name=...` (verbose). `ts-add` được thêm để rút gọn pattern thường gặp.

**Vị trí trong kiến trúc.** Nằm trong `utilities/ovn-ic-nbctl.c` của OVN repo, đăng ký trong array `ic_nbctl_commands[]`. Chạy ở host của operator (laptop, jumphost), connect tới IC_NB DB cluster qua TCP/Unix socket.

**Vai trò.** Bootstrap cross-region L2 broadcast domain. Là điểm khởi đầu của workflow federation: chưa có Transit Switch thì không có gì để bind region vào.

**Vì sao sinh ra.** Operator cần một command 1-shot không phải gõ `create` + nhớ schema. Pattern `add/del/list` đồng bộ với `ovn-nbctl ls-add/ls-del/ls-list` giúp learner dễ chuyển từ NB sang IC_NB.

**Vấn đề giải quyết.** Giảm verbosity, tránh typo schema column name, tự động trả về UUID của row mới tạo (caller có thể pipe vào lệnh khác).

**Tầm quan trọng.** Cao trong workflow setup; thấp trong runtime troubleshooting (sau setup hiếm khi gọi lại). Peripheral nhưng critical-path cho first-time deployment.

**Cơ chế hoạt động.** Hàm `ic_nbctl_ts_add(struct ctl_context *ctx)` trong `utilities/ovn-ic-nbctl.c` đọc tham số `SWITCH` từ `ctx->argv[1]`, kiểm tra collision với `--may-exist`, gọi IDL transaction insert row mới `nbrec_ic_nb_transit_switch_insert(ctx->txn)`, set `name` column, commit transaction.

**Cách dùng thành thạo.** `ovn-ic-nbctl --db=tcp:ic-central:6645 ts-add ts-tenant-blue`. Thêm `--may-exist` để idempotent trong script. Không có flag `external_ids` trực tiếp; muốn set phải gọi tiếp `ovn-ic-nbctl set Transit_Switch ts-tenant-blue external_ids:owner=team-blue`.

**Phân loại.** Database mutation command (RW mode trong command registration `{ "ts-add", 1, 1, "SWITCH", NULL, ic_nbctl_ts_add, NULL, "--may-exist", RW }`).

**Quy trình sử dụng.** (1) `ts-add` tạo Transit Switch trong IC_NB. (2) `ovn-ic-northd` compile sang IC_SB Datapath_Binding. (3) `ovn-ic` mỗi region sync về local SB. (4) Operator bind LSP region với Transit qua `requested-tnl-key`. (5) Verify bằng `ts-list` + `list Transit_Switch`.

**Khi xảy ra sự cố.** Nếu `ts-add` báo "transit switch already exists", check duplicate qua `ts-list`. Nếu commit timeout, IC_NB cluster có thể đang network partition → thử `--no-leader-only` hoặc kiểm tra Raft status.

**Liên quan mật thiết tới.** `ts-list` (verify), `ts-del` (cleanup), schema `Transit_Switch` (target table), daemon `ovn-ic-northd` (downstream consumer), `Datapath_Binding` IC_SB (output sau compile).

**Khác biệt phiên bản.** OVN 20.06 introduce. OVN 22.03 stable. OVN 24.03 thêm option `other_config:requested-tnl-key` cho Transit Switch (hỗ trợ pin tunnel_key thay vì auto-allocate). Curriculum baseline 22.03 không có.

**Cách quan sát + xác minh.** Sau `ts-add`, chạy `ovn-ic-nbctl list Transit_Switch` thấy row có UUID mới + `name` đúng. Verify side effect bằng `ovn-ic-sbctl list Datapath_Binding | grep <name>` (sau ~1 giây để `ovn-ic-northd` compile).

**Source code.** Function anchor `ic_nbctl_ts_add` trong `utilities/ovn-ic-nbctl.c` của OVN branch-22.03. Command registration: array `ic_nbctl_commands[]` cùng file.

**Trường hợp sự cố thực tế.** Operator ghi script Ansible đăng ký 50 Transit Switch tenant cùng lúc. `ts-add` không có batch flag → loop sequential mất 30 giây. Solution: chạy `ovn-ic-nbctl -- ts-add ts-1 -- ts-add ts-2 -- ...` (transaction grouping qua `--`) → 1 RPC, 1 commit.

**Bài tập synthetic.** Trong lab 2-region (Phần 13.15.10), tạo 3 Transit Switch với `ts-add`, verify mỗi cái xuất hiện trong cả IC_NB lẫn IC_SB. Quan sát thời gian từ `ts-add` đến lúc Datapath_Binding xuất hiện ở IC_SB.

**Lỗi thường gặp + tín hiệu chẩn đoán.** Quên `--db` flag → connect socket Unix mặc định `/var/run/ovn/ovn_ic_nb_db.sock` mà socket đó chỉ tồn tại trên host IC central. Tín hiệu: error "database connection failed" hoặc timeout. Fix: luôn set `OVN_IC_NB_DB` env hoặc explicit `--db`.

**So sánh với hệ khác.** Tương đương `kubectl create namespace` (Kubernetes) — đều là wrapper convenience trên top of CRUD API, đều support `--may-exist`/`--dry-run` semantics. Khác biệt: `ts-add` không validate quota, không enforce admission webhook.

### 13.15.7a.2 `ovn-ic-nbctl ts-list` — liệt kê Transit Switch

**Khái niệm.** Sub-command read-only liệt kê tên tất cả Transit Switch trong IC_NB DB, một tên một dòng, sorted alphabetically.

**Lịch sử + bối cảnh.** Cùng OVN 20.06 với `ts-add/ts-del`. Pattern `ls-list/lr-list` từ `ovn-nbctl` được nhân bản sang IC_NB cho UX consistency.

**Vị trí trong kiến trúc.** `utilities/ovn-ic-nbctl.c`, function `ic_nbctl_ts_list`, command registration RO mode (read-only), không cần transaction.

**Vai trò.** Inventory query: trả lời câu hỏi "IC central đang quản lý bao nhiêu Transit Switch và tên gì?". Là bước verify đầu tiên sau `ts-add` và là smoke test khi check sức khỏe IC central.

**Vì sao sinh ra.** Operator cần list nhanh không phải parse `list Transit_Switch` (output verbose có UUID + map). `ts-list` cho output 1-line-per-name dễ pipe sang `wc -l`, `grep`, `sort`.

**Vấn đề giải quyết.** Output formatting: full `list Transit_Switch` có 5+ field per row (UUID, name, external_ids, other_config), không phù hợp cho count hoặc grep tên. `ts-list` chỉ in `name`.

**Tầm quan trọng.** Trung bình. Smoke test daily, là mainstay của monitoring script. Nếu `ts-list` empty mà operator biết đã `ts-add` → IC_NB connection broken hoặc cluster mất quorum.

**Cơ chế hoạt động.** Function `ic_nbctl_ts_list(ctx)` iterate `nbrec_ic_nb_transit_switch_table_for_each(ts, ctx->idl)`, in `ts->name` từng dòng, kết thúc.

**Cách dùng thành thạo.** `ovn-ic-nbctl ts-list | wc -l` đếm số Transit. `ovn-ic-nbctl ts-list | sort` ổn định output cho diff. Combine với `for ts in $(ovn-ic-nbctl ts-list); do ovn-ic-nbctl list Transit_Switch $ts; done` để bulk inspect.

**Phân loại.** Read-only enumeration command, không transaction, không lock.

**Quy trình sử dụng.** Smoke test: SSH IC central → run `ts-list` → verify count khớp expected từ inventory file. Trong CI/CD: assert `ts-list | wc -l == N` sau khi apply Terraform module IC.

**Khi xảy ra sự cố.** `ts-list` empty mà operator vừa `ts-add`: kiểm tra Raft leader của IC_NB qua `ovsdb-client get-schema`, có thể đang follow stale node. Thêm `--no-leader-only` để force read từ leader.

**Liên quan mật thiết tới.** `ts-add` (input source), `ts-del` (deletion verify), `Transit_Switch` table (data source).

**Khác biệt phiên bản.** Stable từ OVN 20.06 đến hiện tại. OVN 24.03 không thay đổi semantics. Output format hoàn toàn giữ nguyên.

**Cách quan sát + xác minh.** Cross-check với `ovn-ic-sbctl list Datapath_Binding | grep external_ids:interconn-ts | wc -l` — nếu count khác `ts-list` count → `ovn-ic-northd` chưa compile xong hoặc gặp lỗi.

**Source code.** Function `ic_nbctl_ts_list` trong `utilities/ovn-ic-nbctl.c` branch-22.03, command registration `{ "ts-list", 0, 0, "", NULL, ic_nbctl_ts_list, NULL, "", RO }`.

**Trường hợp sự cố thực tế.** Monitoring script Nagios alert khi `ts-list` count drop > 10%. Một lần alert false-positive: operator vừa run `ts-del` cleanup tenant offboarded → script không filter operator-driven change. Fix: thêm log correlation với change request ID.

**Bài tập synthetic.** Trong lab 2-region: chạy `ts-list`, sau đó `ts-add`, sau đó `ts-list` lại. Quan sát output diff. Đo thời gian round-trip giữa `ts-add` và `ts-list` thấy entry mới (cho thấy IC_NB write latency).

**Lỗi thường gặp + tín hiệu chẩn đoán.** Output empty không có nghĩa "không có Transit Switch", có thể do `--db` sai endpoint (kết nối nhầm cluster khác, hoặc local `/var/run/...` socket cũ). Tín hiệu: empty + exit 0. Fix: explicit `--db` + double check IC_NB cluster identity qua `external_ids` của row khác nếu có.

**So sánh với hệ khác.** Tương đương `kubectl get namespaces -o name` — chỉ in tên, không metadata. Khác biệt: `ts-list` không có `-o yaml`, `-o json`, `-l label-selector`; muốn filter phải dùng `find Transit_Switch`.

### 13.15.7a.3 `ovn-ic-sbctl` — CLI cho IC_SB DB

**Khái niệm.** `ovn-ic-sbctl` là utility CLI tương tác với cụm IC_SB DB tập trung. Tương tự `ovn-sbctl` cho local SB DB nhưng cho cấp IC. Đại đa số use case là read-only inspection.

**Lịch sử + bối cảnh.** Ra đời cùng `ovn-ic-nbctl` trong OVN 20.06 khi Inter-Connect được merge. IC_SB là output của `ovn-ic-northd` compile + input của `ovn-ic` per-region; engineer cần CLI để debug sync flow.

**Vị trí trong kiến trúc.** `utilities/ovn-ic-sbctl.c` của OVN repo. Connect tới IC_SB DB cluster qua endpoint mặc định `unix:/var/run/ovn/ovn_ic_sb_db.sock` hoặc `--db=tcp:<ic-central>:6646`.

**Vai trò.** Cửa sổ quan sát của operator vào "Transit Switch đã được compile chưa", "AZ nào đã đăng ký", "Route nào đã advertise". Là chính-chủ debug tool khi cross-region traffic không đi qua được.

**Vì sao sinh ra.** IC_SB là internal state machine giữa `ovn-ic-northd` và `ovn-ic`; operator cần "đèn pin soi vào" để verify state. Không có CLI = operator phải đọc trực tiếp OVSDB qua `ovsdb-client` (raw, kém UX).

**Vấn đề giải quyết.** Provide friendly CLI cho 7 bảng quan trọng của IC_SB: `Availability_Zone`, `Gateway`, `Encap`, `Datapath_Binding`, `Port_Binding`, `Route`, `Connection`. Một command `show` cho hierarchical view.

**Tầm quan trọng.** Cao trong forensic + troubleshooting. Mỗi sự cố cross-region thường bắt đầu bằng `ovn-ic-sbctl show` để confirm IC central state.

**Cơ chế hoạt động.** Sử dụng cùng framework `ovsdb-idl` + `ctl-utils` như `ovn-sbctl/nbctl`. Command table `ic_sbctl_commands[]` register `show`, `init`, `get/set/del-connection`, `get/set/del-ssl`. Sub-command `list/find/get/set/...` come from generic `ovsdb-idl` command set.

**Cách dùng thành thạo.** `ovn-ic-sbctl show` cho overview. `list Availability_Zone` xem AZ đăng ký. `list Datapath_Binding` xem transit datapath. `list Port_Binding` xem cross-region port. `list Route` xem route advertise. `find Port_Binding logical_port=<name>` filter theo cột.

**Phân loại.** CLI utility wrapper, mostly RO, có write capability cho Connection + SSL config (rare use).

**Quy trình sử dụng.** Inspect-first workflow: `show` → `list <table>` cho table có anomaly → `find <table> <column>=<value>` để zoom vào row cụ thể → `get <table> <row> <column>` cho atomic field.

**Khi xảy ra sự cố.** Cross-region ping fail → `show` để xem AZ count + Gateway count → `list Datapath_Binding` để verify Transit Switch đã có Datapath → `list Port_Binding` để verify cross-region LSP đã sync.

**Liên quan mật thiết tới.** IC_SB schema (target), `ovn-ic-northd` (writer), `ovn-ic` (reader+writer for chassis claim), `ovn-ic-nbctl` (sister CLI for IC_NB).

**Khác biệt phiên bản.** OVN 22.03 và OVN 24.03 đều giữ command set core. OVN 24.03 thêm field mới trong `Port_Binding` (e.g., `requested_chassis` analog cho transit) → output `list Port_Binding` thêm cột. Không break existing scripts.

**Cách quan sát + xác minh.** `ovn-ic-sbctl show` ra hierarchical text. Để JSON-friendly: `ovn-ic-sbctl --format=json --pretty list Availability_Zone`. Combine với `jq` cho monitoring pipeline.

**Source code.** `utilities/ovn-ic-sbctl.c` branch-22.03, function chính `ic_sbctl_show` cho hierarchical view, `cmd_get_connection/set_connection` cho config.

**Trường hợp sự cố thực tế.** Production incident: traffic Region A → Region B drop sau khi expand cụm IC_SB từ 3 → 5 node. Root cause: 2 node mới thiếu schema upgrade. Phát hiện qua `ovn-ic-sbctl show` báo "schema version mismatch" → upgrade DB schema rồi recovery.

**Bài tập synthetic.** Trong lab: chạy `ovn-ic-sbctl show` lúc 0 region, sau khi add region 1, sau khi add region 2. Quan sát output growth.

**Lỗi thường gặp + tín hiệu chẩn đoán.** Connect tới wrong IC_SB endpoint (port 6645 IC_NB nhầm với 6646 IC_SB) → error "no such schema OVN_IC_Southbound". Fix: nhớ port assignment, dùng `--db` explicit.

**So sánh với hệ khác.** Tương đương `etcdctl` cho Kubernetes — đều là CLI cho central control-plane DB. Khác biệt: `ovn-ic-sbctl` có hierarchical `show`, etcdctl chỉ raw KV.

### 13.15.7a.4 `Availability_Zone` — bảng IC_SB declare region

**Khái niệm.** `Availability_Zone` là bảng OVSDB trong IC_SB (KHÔNG phải IC_NB) chứa danh sách region (= AZ) đã đăng ký với IC central. Mỗi row 1 region. Schema column duy nhất: `name` (string).

**Lịch sử + bối cảnh.** Một trong 7 bảng nguyên thủy của IC_SB schema khi Inter-Connect merge OVN 20.06. Tên "Availability Zone" mượn terminology của AWS/OpenStack — nơi mỗi AZ là failure domain độc lập.

**Vị trí trong kiến trúc.** Schema file `ovn-ic-sb.ovsschema`. Được populate bởi daemon `ovn-ic` của mỗi region (không phải operator manual). Mỗi region đăng ký AZ name của mình lúc `ovn-ic` start.

**Vai trò.** Là "danh bạ region" của IC central: quản lý region nào tham gia federation. Mỗi entity khác trong IC_SB (`Gateway`, `Port_Binding`, `Route`) đều có FK tham chiếu về `Availability_Zone` để biết "row này thuộc region nào".

**Vì sao sinh ra.** Cần namespace để phân biệt resource cross-region. Không có AZ → không biết Gateway G1 thuộc region nào → không biết tunnel endpoint ở đâu để encap.

**Vấn đề giải quyết.** Multi-tenancy ở cấp region: nhiều region chia sẻ chung IC central nhưng tách biệt logical. `Availability_Zone.name` là khóa định danh.

**Tầm quan trọng.** Cao về structural — không có row trong `Availability_Zone` thì region không thể participate IC. Operator hiếm khi tương tác trực tiếp; daemon tự manage.

**Cơ chế hoạt động.** Hàm `az_run()` trong `ic/ovn-ic.c` kiểm tra AZ name từ local NB DB (`NB_Global.name`), insert hoặc update vào IC_SB `Availability_Zone` qua transaction. Nếu name trùng → reuse existing row. Nếu local NB đổi name → update IC_SB row.

**Cách dùng thành thạo.** Operator inspect: `ovn-ic-sbctl list Availability_Zone` xem danh sách region. Operator KHÔNG add/del trực tiếp; thay vào đó set `ovn-nbctl set NB_Global . name=az-region-X` ở mỗi region và để `ovn-ic` tự sync.

**Phân loại.** OVSDB table, declarative, daemon-managed (not user-managed).

**Quy trình sử dụng.** Setup region: (1) ở mỗi region, set `NB_Global.name=az-<region-id>`. (2) Start `ovn-ic` daemon. (3) Verify ở IC central qua `ovn-ic-sbctl list Availability_Zone` thấy row mới.

**Khi xảy ra sự cố.** Cross-region traffic không đi: trước tiên check `Availability_Zone` đủ row chưa (mỗi region 1 row). Nếu thiếu, region đó chưa register với IC central → check `ovn-ic` daemon log, check IC central connectivity.

**Liên quan mật thiết tới.** `Gateway.availability_zone` (FK), `Port_Binding.availability_zone` (FK), `Route.availability_zone` (FK), daemon `ovn-ic` (writer), `NB_Global.name` (source of truth).

**Khác biệt phiên bản.** OVN 20.06 schema chỉ có column `name`. OVN 24.03 thêm column `nb_cfg` (Raft sync counter) — verify nb_cfg tăng đều khi local NB thay đổi. Curriculum baseline 22.03 chưa có nb_cfg trong schema 22.03 baseline (gần đúng — verify thực tế qua `ovs-vsctl get` schema runtime).

**Cách quan sát + xác minh.** `ovn-ic-sbctl --format=table list Availability_Zone`. Nếu count < số region expected → có region chưa register. Combine với `find Gateway availability_zone=<uuid>` để xem AZ nào có gateway active.

**Source code.** Schema definition trong `ovn-ic-sb.ovsschema` branch-22.03. Daemon code: `az_run()` trong `ic/ovn-ic.c`.

**Trường hợp sự cố thực tế.** Region đổi tên đột ngột (operator typo `az-prod` → `az-prd`) → IC central tạo row mới `az-prd`, row cũ `az-prod` orphan với Gateway/Port_Binding cũ. Cross-region traffic mất reference. Fix: rename về tên cũ, hoặc cleanup row cũ qua `destroy Availability_Zone az-prod`.

**Bài tập synthetic.** Trong lab: thay đổi `NB_Global.name` ở region A từ `az-region-a` sang `az-region-a-renamed`. Quan sát IC_SB `Availability_Zone`: row mới xuất hiện hay row cũ rename? (Hint: row mới — tên là primary key conceptually).

**Lỗi thường gặp + tín hiệu chẩn đoán.** AZ name trùng giữa 2 region (operator copy-paste) → cả 2 region cùng update 1 row → chassis claim chồng chéo → tunnel routing chaos. Tín hiệu: `Gateway.availability_zone` cùng UUID nhưng `Gateway.hostname` khác. Fix: rename AZ trong 1 region ngay.

**So sánh với hệ khác.** Tương đương concept "AvailabilityZone" của AWS EC2 — đều là failure domain isolation. Khác biệt: AWS AZ là geographical, OVN AZ là logical OVN cluster boundary, có thể nhiều cluster nằm cùng DC vẫn là 2 AZ.

### 13.15.7a.5 `Transit_Switch` — bảng IC_NB declare cross-region L2

**Khái niệm.** `Transit_Switch` là bảng OVSDB trong IC_NB chứa định nghĩa các Transit Switch (cross-region logical switch). Schema 3 column ở OVN 22.03: `name`, `other_config`, `external_ids`. Mỗi row 1 Transit Switch.

**Lịch sử + bối cảnh.** Bảng nguyên thủy của IC_NB schema OVN 20.06. "Transit" là terminology mượn từ MPLS L2VPN (transit link nối các site).

**Vị trí trong kiến trúc.** Schema file `ovn-ic-nb.ovsschema` branch-22.03. Operator-managed (qua `ovn-ic-nbctl ts-add/del`). Không có FK ra, các bảng khác (e.g., IC_SB `Datapath_Binding.transit_switch`) reference NAME (string) chứ không reference UUID — important cho cross-DB linking.

**Vai trò.** Khai báo "L2 broadcast domain mở rộng qua nhiều region". Là input của `ovn-ic-northd` compile sang IC_SB `Datapath_Binding`. Là điểm anchor để region-local NB tham chiếu khi tạo LSP type=router với `requested-tnl-key`.

**Vì sao sinh ra.** Cần entity central để all region đồng thuận "đây là cùng L2 segment". Nếu để mỗi region tự tạo logical switch và gán tunnel_key thủ công → conflict không thể tránh.

**Vấn đề giải quyết.** Single source of truth cho cross-region L2: tên + tunnel_key allocate. `ovn-ic-northd` tránh collision tunnel_key cross-region.

**Tầm quan trọng.** Cốt lõi structural cho mọi cross-region L2 use case. Không có Transit_Switch → không thể federate L2.

**Cơ chế hoạt động.** Operator insert row qua `ts-add`. `ovn-ic-northd` đọc row, allocate tunnel_key (range 16M-24M), tạo `Datapath_Binding` IC_SB với `transit_switch=<name>` + `tunnel_key=<allocated>`. `ovn-ic` mỗi region đọc IC_SB Datapath_Binding, sync về local SB.

**Cách dùng thành thạo.** Tạo: `ovn-ic-nbctl ts-add ts-tenant-blue`. Annotate: `ovn-ic-nbctl set Transit_Switch ts-tenant-blue external_ids:owner=team-blue external_ids:purpose=cross-dc-tenant`. Inspect: `ovn-ic-nbctl list Transit_Switch`.

**Phân loại.** Operator-managed entity, declarative, downstream consumed bởi `ovn-ic-northd` daemon.

**Quy trình sử dụng.** (1) `ts-add <name>` trong IC_NB. (2) Annotate `external_ids` cho audit. (3) Mỗi region: tạo LRP nối với Transit qua peer port, set `options:requested-tnl-key=<allocated>`. (4) Verify chain qua `ovn-ic-sbctl list Datapath_Binding`. (5) Test cross-region ping.

**Khi xảy ra sự cố.** Transit Switch tạo nhưng cross-region không thông: check `Datapath_Binding` IC_SB có row tương ứng không. Nếu không → `ovn-ic-northd` chưa compile (check daemon log). Nếu có nhưng tunnel_key conflict với local tenant → operator hardcode `requested-tnl-key` trùng.

**Liên quan mật thiết tới.** `ovn-ic-nbctl ts-add/del/list` (CLI), `Datapath_Binding` IC_SB (compiled output), `ovn-ic-northd` (compiler), `Logical_Router_Port.options:requested-tnl-key` per-region (consumer), Geneve tunnel (data plane).

**Khác biệt phiên bản.** OVN 22.03: 3 column (name, other_config, external_ids). OVN 24.03: thêm `other_config:requested-tnl-key` để pin tunnel_key thay vì auto-allocate (giúp migration không thay tunnel_key). Curriculum baseline 22.03 không có feature pin tunnel_key.

**Cách quan sát + xác minh.** `ovn-ic-nbctl list Transit_Switch` ra row IC_NB. Cross-check `ovn-ic-sbctl find Datapath_Binding external_ids:interconn-ts=<name>` ra row IC_SB. Cross-check `ovn-sbctl --db=tcp:<region>:6642 list Datapath_Binding | grep <name>` ra row local SB sau khi `ovn-ic` sync.

**Source code.** Schema definition `ovn-ic-nb.ovsschema` branch-22.03. Compile logic: hàm trong `ic/ovn-ic.c` (function `ts_run()` hoặc tương đương trong `ic/ovn-ic-northd.c` của 22.03 — version-specific, verify exact function).

**Trường hợp sự cố thực tế.** Operator delete `ts-tenant-blue` qua `ts-del` mà không drain tenant LRP trước → LSP region orphan với requested-tnl-key reference vào Transit Switch không tồn tại → ovn-controller mỗi region log warning "transit datapath not found" liên tục. Fix: re-create Transit Switch (cùng name) hoặc cleanup LRP region trước.

**Bài tập synthetic.** Trong lab: tạo Transit Switch `ts-test`, set `external_ids:purpose=lab-test`. Quan sát Datapath_Binding tương ứng ở IC_SB sau ~1 giây. Đo tunnel_key allocate. Sau đó `ts-del`, quan sát Datapath_Binding biến mất.

**Lỗi thường gặp + tín hiệu chẩn đoán.** Tên Transit Switch chứa ký tự đặc biệt (e.g., dấu cách, `/`) → schema validation reject hoặc compile lỗi. Tín hiệu: `ts-add` exit non-zero với message "invalid name". Fix: chỉ dùng `[a-zA-Z0-9_-]` trong name.

**So sánh với hệ khác.** Tương đương "Transit Gateway" của AWS VPC — đều là central resource nối nhiều VPC/region L2-equivalent. Khác biệt: AWS Transit Gateway là managed service có metering + ACL builtin; OVN Transit Switch là plain L2 segment, không có ACL ở cấp Transit (ACL apply ở local LR).

## 13.15.8 Anatomy 1 — `ovn-ic-nbctl ts-add` + verify

**Bucket:** OVN | **Context:** dựng Transit Switch mới cho cross-region L2 | **Purpose:** kết nối tenant network ở 2 region | **Activity:** provisioning | **Mechanism:** insert row vào IC_NB Transit_Switch table.

**Lệnh + decode:**

```
$ ovn-ic-nbctl --db=tcp:ic-central:6645 ts-add ts-tenant-blue
$ ovn-ic-nbctl --db=tcp:ic-central:6645 ts-list
ts-tenant-blue
$ ovn-ic-nbctl --db=tcp:ic-central:6645 list Transit_Switch
_uuid               : abc12345-def6-7890-abcd-ef1234567890
external_ids        : {}
name                : "ts-tenant-blue"
other_config        : {}
```

| Attribute | Healthy / Warning / Critical |
|-----------|------------------------------|
| `_uuid` | UUID stable cross IC restart |
| `name` | Match convention naming (ts-<tenant>-<purpose>) |
| `external_ids` | Empty mặc định, có thể annotate cho audit (`region_owner=...`) |
| `other_config` | Empty mặc định, có thể set encap type override |

**Verify sync xuống region:**

```
$ ovn-sbctl --db=tcp:region-a-sb:6642 list Datapath_Binding | grep ts-tenant-blue
_uuid               : ... (different UUID, region-local)
external_ids        : {interconn-ts="ts-tenant-blue"}
tunnel_key          : 16711680  # allocated by ovn-ic-northd
```

`external_ids:interconn-ts` = "ts-tenant-blue" báo Datapath_Binding này là sync từ IC, không phải từ local NB. tunnel_key cấp phát unique global trong IC namespace.

## 13.15.9 Anatomy 2 — `ovn-ic-sbctl list AvailabilityZone`

**Bucket:** OVN | **Context:** verify region đã đăng ký với IC central | **Purpose:** kiểm tra mỗi region đã sync chassis claim hay chưa | **Activity:** monitoring | **Mechanism:** query IC_SB AvailabilityZone table.

**Output mẫu:**

```
$ ovn-ic-sbctl --db=tcp:ic-central:6646 list AvailabilityZone
_uuid               : 11111111-...
name                : "az-region-a"
nb_cfg              : 42
gateways            : [22222222-..., 33333333-...]

_uuid               : 44444444-...
name                : "az-region-b"
nb_cfg              : 38
gateways            : [55555555-..., 66666666-...]
```

| Attribute | Healthy / Warning / Critical |
|-----------|------------------------------|
| `name` | Tên AZ unique global, thường = "az-<region>" |
| `nb_cfg` | Counter sync từ region NB | Healthy: tăng đều khi cấu hình thay đổi. Critical: stuck → ovn-ic của region đó treo |
| `gateways` | List UUID Chassis làm gateway transit | Healthy: ≥ 1 gateway active mỗi AZ |

**Critical signal:**
- `gateways = []` empty → region không có chassis nào đăng ký làm transit gateway → cross-region traffic không thể đi vào/ra. Verify `external_ids:ovn-is-interconn=true` trên ít nhất 1 chassis.
- `nb_cfg` không tăng trong thời gian dài → ovn-ic không sync về central → check ovn-ic daemon log.

## 13.15.11 Capstone POE — thiết kế 3-region (HQ + 2 branch)

**Bài toán:** thiết kế OVN-IC topology cho doanh nghiệp:
- HQ: 100 chassis, 5000 VM, ở DC chính
- Branch-1: 10 chassis, 200 VM, ở DC phụ (50ms latency tới HQ)
- Branch-2: 5 chassis, 100 VM, ở edge office (200ms latency tới HQ)

Yêu cầu:
- VM HQ phải nói chuyện với VM Branch (cross-region L3)
- VM Branch-1 không cần nói trực tiếp với Branch-2 (chỉ qua HQ)
- Failure isolation: nếu HQ chết, Branch tự xử lý local tenant

**Bước thiết kế:**

1. **Chia AZ:** 3 AZ (az-hq, az-branch-1, az-branch-2). Mỗi AZ có cụm SB DB Raft 3-node độc lập.

2. **IC central:** 1 cụm IC_NB + IC_SB Raft 3-node, đặt ở HQ (latency thấp tới HQ, acceptable cao tới Branch).

3. **Transit Switch design:**
   - `ts-hq-branch1`: kết nối HQ và Branch-1
   - `ts-hq-branch2`: kết nối HQ và Branch-2
   - KHÔNG tạo `ts-branch1-branch2` (theo yêu cầu, branch-to-branch chỉ qua HQ).

4. **Transit Router:** 1 Transit Router dùng để route cross-region L3 traffic. Bind tất cả Transit Switch + advertise route từ mỗi AZ subnet.

5. **Failure mode:**
   - HQ down: Branch-1 và Branch-2 mất cross-region connectivity (đúng yêu cầu — chỉ qua HQ). Local tenant Branch vẫn chạy bình thường.
   - Branch down: HQ vẫn hoạt động, Branch còn lại vẫn nói được với HQ.
   - IC central down: cấu hình mới không sync, NHƯNG tunnel data plane đã cài tiếp tục chạy → graceful degradation.

6. **Recovery procedure:**
   - HQ down recovery: restart cụm SB region HQ → ovn-ic resync → Branch reconnect tự động.
   - IC central down recovery: rebuild IC cluster từ backup (`ovsdb-tool` create-cluster + restore từ backup snapshot mỗi 1 giờ).

**Predict + observe:** dự đoán cross-region first-packet latency = ~3x physical latency (Geneve handshake + ARP cross-region). Verify bằng `ping -c 1` đầu tiên vs subsequent.

**Lý do mục tiêu của POE:** giúp engineer nắm flow design + failure model trước khi deploy production. Đây là kỹ năng "architect" thật sự, không phải chỉ vận hành.

## 13.15.12 Mẫu cần tránh và lưu ý

**Mẫu cần tránh 1: Stretch single SB cluster cross-region thay vì IC.**

Đôi khi team muốn "đơn giản hơn" bằng cách stretch single cụm SB 3-node qua 3 region, mỗi region 1 node. Vấn đề:
- Latency Raft cross-region ≥ max(round-trip pairwise). 100ms round-trip = 100ms commit latency cho mọi `ovn-nbctl` write.
- Network partition giữa region = quorum loss → toàn cụm config-frozen.
- Compliance: data leak risk.

**Cách phòng:** nếu deployment ≥ 2 region, mặc định dùng OVN-IC. Stretch chỉ acceptable cho 2 DC < 5km cách nhau (low latency + same regulatory domain).

**Mẫu cần tránh 2: Hardcode `requested-tnl-key` thay vì để IC allocate.**

`ovn-ic-northd` allocate tunnel_key cho Transit Switch tự động (range 16M-24M). Nếu operator hardcode `requested-tnl-key=42` (collision với local tenant), ovn-controller sẽ gặp tunnel_key conflict → packet drop.

**Cách phòng:** không bao giờ set `requested-tnl-key` cho Transit-related entity. Để IC allocate.

**Lưu ý 3: Backup IC central cluster offsite.**

IC_NB + IC_SB là single point of cấu hình (mọi region depend vào). Backup ít nhất hourly + offsite (region khác hoặc cloud storage). Nếu IC central destroyed, bạn rebuild từ backup sẽ mất ít nhất 30 phút (acceptable nếu hourly backup) thay vì rebuild từ scratch (mất ngày).

**Lưu ý 4: BGP integration BAN.**

Per CLAUDE.md North Star, curriculum KHÔNG cover deep BGP-based dynamic route advertisement cho IC (`ovn-bgp-agent`). Nếu deployment thực tế cần BGP, tham khảo `ovn-bgp-agent` upstream documentation độc lập. Curriculum chỉ cover **static route** qua `ovn-ic-nbctl add Route`.

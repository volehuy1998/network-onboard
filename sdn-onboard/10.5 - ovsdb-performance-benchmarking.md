## 10.5.1 Đặc tính hiệu năng cơ bản

OVSDB không thiết kế cho băng thông thực tế giao dịch cao như SQL database — nó là kho cấu hình cho OVS/OVN, workload đặc trưng là **tần suất ghi thấp, tần suất đọc cao, đọc qua đăng ký**. Đặc tính cụ thể trên phần cứng chuẩn (Intel Xeon Silver, 16 GB RAM, NVMe SSD):

**Băng thông thực tế transaction (ghi):**
- ovsdb-server single-node: 2000-5000 transaction/giây (TPS) với transaction nhỏ (1-5 operation).
- 3-node Raft cluster: 800-1500 TPS (giảm vì phải đồng bộ log qua majority).
- 5-node Raft cluster: 600-1000 TPS (giảm thêm — thêm node không tăng băng thông thực tế ghi, chỉ tăng tolerance fault).

**Độ trễ commit (thời gian từ khi client gửi transaction đến khi nhận response):**
- Single-node: 1-5 ms (thời gian xử lý + fsync disk).
- 3-node Raft: 5-15 ms (thêm round-trip đồng bộ với majority + fsync trên mỗi node).
- Cross-AZ Raft (các node ở data center khác nhau): 20-100 ms (bị giới hạn bởi độ trễ mạng WAN).

**Băng thông thực tế đọc (đăng ký monitor_cond):**
- Single-node phục vụ 100 client đồng thời: 50-100 MB/s băng thông thực tế kết hợp.
- Với condition monitor_cond tốt (client chỉ nhận row liên quan): mở rộng gần tuyến tính đến 1000+ client.
- Không có monitor_cond (full monitor): điểm nghẽn ở 10-50 client do băng thông duplicate.

**Dấu chân memory:**
- Database 10K row: khoảng 50-100 MB RAM.
- 100K row: 500 MB - 1 GB.
- 1M row (OVN SB cụm cực lớn): 5-10 GB.
- Memory không chỉ chứa data — Raft log (chưa compact) + state monitor_cond per-client có thể 2-3x data size.

**Kích thước file database (trên disk):**
- Không compact: phình tuyến tính theo số transaction (append-only log). Sau 1 triệu transaction có thể đạt 1-10 GB.
- Sau compact: khoảng 2-5x data size trong memory.
- Tần suất snapshot Raft (`cluster/change-election-timer`): ảnh hưởng tần suất compact.

## 10.5.2 Các yếu tố ảnh hưởng hiệu năng ở quy mô lớn

Khi OVN cluster mở rộng lên hàng trăm-nghìn chassis, OVSDB SB DB trở thành điểm nghẽn phổ biến nhất. Các yếu tố cụ thể:

**(1) Số lượng Logical Flow:** mở rộng gần tuyến tính với số Port_Binding × độ phức tạp. Cụm 1000 chassis × 100 pod/chassis × 50 Logical Flow/pod = 5 triệu Logical Flow. Mỗi transaction compile của ovn-northd có thể insert/update hàng chục nghìn row → độ trễ lên hàng giây.

**(2) Fanout của monitor_cond:** mỗi ovn-controller đăng ký một tập con. 1000 chassis = 1000 đăng ký active đồng thời. Server phải xử lý mỗi transaction × 1000 lần so sánh để sinh notification delta cho mỗi client. Overhead CPU tăng tuyến tính với fanout.

**(3) Raft log growth:** mỗi transaction append vào Raft log. Log phình → thời gian replay sau reboot/kết nối lại tăng. `cluster/change-election-timer` và `ovsdb-tool compact` phải chạy định kỳ để giữ log nhỏ.

**(4) Conntrack vs Logical Flow:** OVN ACL có state sinh nhiều Logical Flow vì mỗi rule compile 3-5 flow (cho state new/est/rel). Cụm với 10K ACL rule = 30-50K Logical Flow.

**(5) Memory OVSDB IDL per client:** mỗi ovn-controller cache tập con SB DB. Tập con nhỏ (qua monitor_cond) nhưng với OVN 22.03 có thể 100-500 MB per chassis.

**(6) Hiệu năng I/O disk:** fsync sau mỗi commit là điểm nghẽn disk-bound. NVMe SSD giảm fsync từ 10ms (SATA SSD) xuống 100µs — tăng băng thông thực tế 10-100 lần.

## 10.5.3 Phương pháp benchmark

Để đo băng thông thực tế thực của OVSDB cluster trước khi triển khai môi trường production:

**Công cụ 1 — ovn-scale-test (Red Hat):** framework chính thức test mở rộng OVN. Sinh workload giống OVN môi trường production: tạo Logical Switch + Port, đo độ trễ ovn-northd compile, độ trễ ovn-controller cài OpenFlow. Thiết lập cụm đa node rồi chạy:
```
[host]# cd ovn-scale-test
[host]# ./rally.sh ovn-boot-servers --servers-per-node=100 --nodes=10
```
Test báo số liệu TPS, độ trễ p50/p95/p99, memory usage.

**Công cụ 2 — ovsdb-client với script tùy chỉnh:** viết script Python/shell gửi transaction liên tục đo TPS:
```python
import time, json
from ovs.db.idl import Idl, Transaction
idl = Idl("tcp:127.0.0.1:6641", schema)
count = 0
start = time.time()
while time.time() - start < 60:  # 60 giây
    txn = Transaction(idl)
    # build simple transaction: insert row + commit
    status = txn.commit_block()
    count += 1
print(f"TPS: {count/60}")
```

**Công cụ 3 — perf profile ovsdb-server:** khi thấy độ trễ tăng bất thường:
```
[host]# perf record -p $(pidof ovsdb-server) -F 99 -g -- sleep 30
[host]# perf report
```
Xác định hàm chiếm CPU nhiều nhất (thường là JSON parsing, Raft replication, monitor_cond diff).

**Công cụ 4 — iostat cho fsync độ trễ:**
```
[host]# iostat -x 1
```
Cột `w_await` (write await time) = độ trễ fsync trung bình. Nếu > 10ms → disk bị bão hòa hoặc hardware chậm.

**Công cụ 5 — OVSDB coverage counter:**
```
[host]# ovs-appctl -t ovsdb-server coverage/show
```
Xuất bộ đếm nội bộ: `transact_total`, `monitor_notifications`, `raft_append_entries`. Giám sát qua thời gian để phát hiện bất thường.

## 10.5.4 Các kỹ thuật tinh chỉnh chính

**(1) Tune the Raft `election_timer`.** The unixctl command tunes the leader-election timeout, measured in milliseconds, so the cluster tolerates network jitter between nodes. It is NOT a snapshot-frequency knob.
```
[host]# ovs-appctl -t ovsdb-server cluster/change-election-timer 4000
```
The default is 1000 ms (verified at v2.17.9 in `ovsdb/raft.c`, where `ELECTION_BASE_MSEC=1000`, with hard limits `ELECTION_MIN_MSEC=100` and `ELECTION_MAX_MSEC=600000`). Raise it to 4000 ms when the round-trip time between cluster nodes is high (cross-AZ deployments, p99 above 8 ms) so that spurious elections are rare. The heartbeat interval is `election_timer / 3`, so a 4000 ms timer drives heartbeats every 1333 ms. Important: this command does not change Raft snapshot frequency.

**(1b) Raft snapshot frequency.** OVS does not expose a runtime unixctl knob to set snapshot frequency directly. The default behaviour is to snapshot when the in-memory log crosses an internal size threshold (about 10 times the previous snapshot, or memory above roughly 100 MB; see `raft_notify_snapshot_recommended()` in `ovsdb/raft.c`). Operators tune snapshot behaviour through schema-level configuration and through the periodic `ovsdb-server/compact` command shown in subsection (2), not through a `cluster/change-*` unixctl command.

**(2) Compact file định kỳ:** dùng `ovsdb-client compact`:
```
[host]# ovsdb-client compact tcp:127.0.0.1:6641 OVN_Northbound
```
Chạy mỗi giờ hoặc mỗi ngày tùy tần suất ghi. Cụm OVN môi trường production thường có cron job.

**(3) Read-only server (OVSDB RO replica):** trong cấu hình 3-node Raft + nhiều replica chỉ đọc:
```
[host]# ovsdb-server --remote=ptcp:6641 /path/to/conf.db --read-only
```
Server chỉ đọc replay từ leader nhưng không chấp nhận ghi. ovn-controller có thể đăng ký từ RO server → giảm tải leader.

**(4) Tối thiểu hóa column IDL:** giảm column đăng ký trên client (đã được đề cập 10.4) — tác động lớn tới memory và băng thông.

**(5) Gom transaction:** ovn-northd dùng parallel compilation + gom nhiều thay đổi thành một transaction giảm số transaction.

**(6) Hugepage cho memory ovsdb-server:** giảm TLB miss:
```
[host]# echo 2048 > /sys/kernel/mm/hugepages/hugepages-2048kB/nr_hugepages
```
Không phải tinh chỉnh lớn, nhưng giúp với OVSDB có dấu chân memory > 1 GB.

> **Lưu ý kiến trúc:** Raft consensus theo thiết kế bắt buộc majority commit (quorum n/2+1). Không có cơ chế chính thức "bỏ qua Raft sync" trong OVS/OVN upstream. Mọi tuyên bố về flag tăng băng thông qua bypass Raft cần kiểm chứng cẩn thận với upstream code (`ovsdb/raft.c`) và `ovn-nb.xml` schema. Để tăng băng thông thực tế ghi, các kỹ thuật khả thi gồm: giảm cluster size (3 thay vì 5), giảm tần suất commit (gom transaction qua `ovn-northd-backoff-interval-ms`), hoặc tách workload sang OVSDB instance khác.

## 10.5.5 Điểm nghẽn điển hình và cách phát hiện

Khi OVN cluster chậm, chuỗi chẩn đoán thường:

**Triệu chứng 1 — ovn-controller chậm phản ứng với NB change:**
Kiểm tra delta `nb_cfg` giữa NB và SB:
```
[host]# ovn-nbctl get NB_Global . nb_cfg    # giá trị cao
[host]# ovn-sbctl get SB_Global . nb_cfg    # giá trị thấp hơn nhiều
```
Nếu chênh lệch lớn → ovn-northd không catch up. Chẩn đoán tiếp:
```
[host]# ovs-appctl -t ovn-northd coverage/show | grep northd_run_time
```

**Triệu chứng 2 — ovsdb-server CPU cao:**
```
[host]# top -p $(pidof ovsdb-server)
```
Thường do fanout monitor_cond lớn. Kiểm tra số đăng ký:
```
[host]# ovs-appctl -t ovsdb-server cluster/status OVN_Southbound
```
Thấy số client đang kết nối. Nếu > 500 → xem xét tách ra nhiều ovsdb-server.

**Triệu chứng 3 — Raft election liên tục:**
Log `ovsdb-server.log` có:
```
election timer expired, starting new election
```
Do network partition hoặc leader quá tải. Kiểm tra network độ trễ giữa các node:
```
[host]# ping node2_ip
[host]# ping node3_ip
```
Độ trễ > 10ms giữa Raft node gây vấn đề. Cần cùng LAN hoặc cùng-AZ.

**Triệu chứng 4 — disk full hoặc chậm:**
```
[host]# df -h /var/lib/ovn
[host]# iostat -x 1 | grep nvme
```
Nếu disk chậm → ưu tiên nâng cấp SSD hoặc chuyển sang disk nhanh hơn.

### ▶ Guided Exercise 1: Benchmark băng thông thực tế transaction với ovn-scale-test

> **State verification lab:** Output dựa trên Red Hat ovn-scale-test 2023 report; chờ kiểm chứng lab thực.

**Mục tiêu:** Đo TPS của OVSDB NB DB dưới workload synthetic để hiểu đặc tính trước khi triển khai môi trường production.

**Chuẩn bị:**
- Node Ubuntu 22.04 với OVN 22.03 đã chạy.
- Clone ovn-scale-test: `git clone https://github.com/ovn-org/ovn-scale-test`.

**Bước thực hiện:**

**1.** Tạo workload cấp độ nhỏ (100 LS, 10 port/LS):
```
[host]# cd ovn-scale-test
[host]# cat > workload.json <<EOF
{
  "OVN.create_and_list_lswitches": [
    {"args": {"amount": 100, "batch": 10}, "runner": {"type": "constant", "times": 1}}
  ],
  "OVN.create_and_list_lports": [
    {"args": {"amount": 10}, "runner": {"type": "constant", "times": 100}}
  ]
}
EOF
[host]# ./run.sh workload.json
```

**2.** Quan sát output:
```
TPS report:
  create_and_list_lswitches: 145 TPS
  create_and_list_lports: 320 TPS (batch=10 → 32 txn/s)

Latency percentiles (ms):
  p50: 6.8
  p95: 14.2
  p99: 28.5
```

**3.** Mở rộng workload (1000 LS, 100 port/LS):
```
[host]# # Edit workload.json: amount 1000 + 100
[host]# ./run.sh workload.json
```

**4.** So sánh TPS — với quy mô lớn, TPS thường giảm vì Logical Flow compile tốn hơn.

**Bài học học được:**

Thứ nhất, TPS OVSDB không cố định — giảm theo kích thước database. Với cụm OVN 100K Logical Flow, một transaction compile có thể tốn 100-500ms CPU bên ovn-northd.

Thứ hai, benchmark trước triển khai giúp tránh bất ngờ môi trường production. Một cụm 10K chassis không thể khởi chạy với cấu hình OVSDB cho cụm 100 chassis.

**Cleanup:**
```
[host]# cd ovn-scale-test && ./cleanup.sh
```

### ▶ Guided Exercise 2: Lập hồ sơ ovsdb-server với perf flamegraph

> **State verification lab:** chờ kiểm chứng.

**Mục tiêu:** Xác định đường dẫn nóng trong ovsdb-server khi mở rộng.

**Chuẩn bị:** perf + flamegraph script.

**Bước thực hiện:**

**1.** Trong lúc benchmark đang chạy (từ Exercise 1):
```
[host]# perf record -F 99 -g -p $(pidof ovsdb-server) -- sleep 30
[host]# perf script > out.perf
[host]# stackcollapse-perf.pl out.perf | flamegraph.pl > ovsdb.svg
```

**2.** Mở `ovsdb.svg`, quan sát hàm chủ đạo. Thường:
- `ovsdb_txn_process` — xử lý transaction.
- `json_to_value` / `value_to_json` — parse/serialize JSON.
- `ovsdb_monitor_update` — tính delta cho monitor_cond.
- `raft_log_append` — append log Raft.

**3.** Nếu `json_to_value` chiếm > 30% → điểm nghẽn JSON parsing. Cân nhắc nâng version OVS mới với JSON nhanh hơn.

**Bài học học được:**

Thứ nhất, flamegraph là công cụ cần thiết để hiểu ovsdb-server đang làm gì khi CPU cao.

Thứ hai, kết quả lập hồ sơ dẫn đường tối ưu — ví dụ nếu monitor_cond chủ đạo, giải pháp là giảm fanout (tách server) hoặc nâng cấp hardware.

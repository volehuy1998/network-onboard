## 13.12.1 The role of IPAM in OVN

IPAM (IP Address Management) is the feature that lets OVN manage subnets and allocate IP addresses to LSPs without any external service. When a CMS such as OpenStack Neutron or OVN-Kubernetes creates a pod or VM, the CMS only needs to create an LSP with `addresses=dynamic`. OVN then picks a free IP from the subnet, makes sure it does not conflict with an already allocated address, and writes the result into the `Logical_Switch_Port` table.

**Why IPAM is integrated into OVN:**

- **No external dependency.** For small and medium clusters, you do not need to deploy Infoblox or run a separate BIND server.
- **Performance.** Allocation happens at the same time as LSP creation. There is no round-trip to an external API.
- **Atomicity.** The IPAM state lives in NB DB, so creating or deleting an LSP and updating the address are part of one atomic transaction.
- **Multi-tenancy.** Every Logical Switch has its own subnet. Tenant A and tenant B can use the same CIDR with no conflict.

**OVN IPAM data model:**

- A Logical Switch can declare `other_config:subnet=10.0.0.0/24`. This turns OVN IPAM on for that switch.
- Optional: `other_config:exclude_ips=10.0.0.1..10.0.0.10`. OVN will not allocate any address in this range. Typically used for the gateway, the DHCP server, or any reserved address.
- An LSP with `addresses=["aa:bb:cc:dd:ee:ff dynamic"]` receives a dynamically allocated IP from the subnet.
- An LSP with `addresses=["aa:bb:cc:dd:ee:ff 10.0.0.100"]` is static. The user pins the IP directly.
- OVN writes the result into `dynamic_addresses="aa:bb:cc:dd:ee:ff 10.0.0.5"` (the actual address that was allocated).

## 13.12.2 Dynamic allocation

When an LSP is declared with `addresses="<MAC> dynamic"`, `ovn-northd` runs the following steps.

**Step 1, find the subnet.** Read `other_config:subnet` of the Logical Switch that contains the LSP.

**Step 2, gather the addresses already in use.** Scan every LSP on the same Logical Switch. Collect IPs from `dynamic_addresses` and from any static `addresses` entry. Also include the addresses listed in `other_config:exclude_ips`.

**Step 3, pick a new IP.** Walk the subnet and find the lowest IP that is not in the in-use set. The algorithm is a linear scan from the subnet minimum to the subnet maximum, skipping every excluded address.

**Step 4, write the result.** Set `LSP.dynamic_addresses = "<MAC> <chosen_IP>"`.

**Step 5, activate.** If the Logical Switch has a `DHCP_Options` record, the Logical Flow stage `ls_in_dhcp_options` automatically uses this address as the DHCP `offerip`.

**Worked example:**

```
[host]# ovn-nbctl ls-add ls1
[host]# ovn-nbctl set Logical_Switch ls1 other_config:subnet=10.0.0.0/24
[host]# ovn-nbctl set Logical_Switch ls1 other_config:exclude_ips="10.0.0.1..10.0.0.10"

[host]# ovn-nbctl lsp-add ls1 vm1
[host]# ovn-nbctl lsp-set-addresses vm1 "aa:bb:cc:dd:ee:01 dynamic"

[host]# ovn-nbctl lsp-add ls1 vm2
[host]# ovn-nbctl lsp-set-addresses vm2 "aa:bb:cc:dd:ee:02 dynamic"

[host]# ovn-nbctl get Logical_Switch_Port vm1 dynamic_addresses
"aa:bb:cc:dd:ee:01 10.0.0.11"

[host]# ovn-nbctl get Logical_Switch_Port vm2 dynamic_addresses
"aa:bb:cc:dd:ee:02 10.0.0.12"
```

OVN starts allocating from `10.0.0.11` because `exclude_ips` reserves `10.0.0.1` through `10.0.0.10`.

**Detail of the `exclude_ips` syntax.** The format is `<start_ip>..<end_ip>` for a range, or a single IP for one address. Multiple ranges are separated by spaces:

```
exclude_ips="10.0.0.1 10.0.0.5..10.0.0.10 10.0.0.100..10.0.0.200"
```

Ranges may overlap. OVN merges them automatically.

## 13.12.3 Static allocation

For workloads that need a fixed IP, such as a server with a DNS record or a service VIP:

```
[host]# ovn-nbctl lsp-add ls1 db_server
[host]# ovn-nbctl lsp-set-addresses db_server "aa:bb:cc:dd:ee:99 10.0.0.50"
```

OVN checks that `10.0.0.50` is not already used by the `dynamic_addresses` of another LSP. If there is a conflict, `ovn-northd` logs a warning, but it still accepts the assignment because the user made the choice deliberately.

**Mixing static and dynamic.** A single Logical Switch can hold both static LSPs and dynamic LSPs. Dynamic allocation automatically skips any address already used by a static LSP.

**`mac_prefix` for a custom MAC range.** By default, dynamic addresses use a MAC drawn from the OVN internal OUI, `0a:00:00:xx:xx:xx`. If you need a custom MAC, for example a MAC from a range licensed for VMware compatibility:

```
[host]# ovn-nbctl set Logical_Switch ls1 other_config:mac_prefix="52:54:00"
```

OVN now generates random MACs of the form `52:54:00:xx:xx:xx`.

## 13.12.4 IPv6 IPAM and prefix delegation

OVN supports IPv6 IPAM in the same way as IPv4. The subnet is configured through `other_config:ipv6_prefix`:

```
[host]# ovn-nbctl set Logical_Switch ls1 other_config:ipv6_prefix="fd00:1::/64"
```

**Two IPv6 addressing modes:**

- **DHCPv6 stateful.** OVN `DHCP_Options` allocates the IPv6 address (already discussed in Part 13.10).
- **SLAAC with EUI-64.** OVN derives the IPv6 address from the MAC, using the EUI-64 algorithm of RFC 4291. The pod configures itself through Router Advertisements. No DHCPv6 is needed.

**IPv6 Prefix Delegation (PD).** OVN 22.03 and later supports delegating an IPv6 prefix from an upstream provider into OVN through DHCPv6-PD:

```
[host]# ovn-nbctl set Logical_Router_Port lrp_ext \
         options:prefix-delegation=true \
         options:prefix-length=56 \
         options:dhcpv6-pd=true
```

This integrates with an FRR or DHCPv6 client to request the prefix from the upstream router. OVN then automatically assigns a `/64` to each internal Logical Switch.

## 13.12.5 ND Proxy and MAC learning

When an LSP of type `vif` binds to a chassis, the `ovn-controller` on that chassis automatically learns the MAC of the pod through OpenFlow and writes it into the `MAC_Binding` table in SB DB.

**Table `MAC_Binding`:**

- `logical_port`, the LSP name.
- `ip`, the IP address of the neighbor.
- `mac`, the MAC that was learned.
- `datapath`, the `Datapath_Binding` record.
- `timestamp`, the time of learning. Introduced in OVN v22.09.0 by commit [`1a947dd3`](https://github.com/ovn-org/ovn/commit/1a947dd3) "northd: Add MAC binding aging mechanism", Ales Musil 2022-08-17, Red Hat BZ 2084668. Verified through `mcp__github__get_commit`.

`MAC_Binding` is used to:

- Avoid flooding ARP and NDP across the cluster. The first packet floods, the rest hit the cache.
- Support ND Proxy. A Logical Router can answer ARP or NDP on behalf of a pod, for example when the pod does not respond, or when the pod is on a different subnet.

**ND Proxy on a Logical Router.** When an LRP has an IPv6 address and the flag `ipv6_ra_configs:nd_proxy=true`, the LR answers NDP Neighbor Solicitation for any IP in the subnets behind the LR:

```
[host]# ovn-nbctl set Logical_Router_Port lrp1 ipv6_ra_configs:nd_proxy=true
```

This is useful when the cluster has many subnets sharing one gateway. The gateway proxies NDP for all of them.

**`MAC_Binding` timestamp (OVN v22.09.0 and later, commit `1a947dd3`).** The timestamp tracks when the entry was learned or refreshed. `ovn-northd` expires old entries based on the per-LR setting `other_config:mac_binding_age_threshold`, default 0, which means disabled. When set to a value greater than 0, entries older than the threshold are removed. The default is verified as 0 in the commit body, not 600 seconds as an older revision of the curriculum claimed.

## 13.12.6 Integration with external IPAM

For an enterprise environment that already has a centralized IPAM such as Infoblox, BlueCat, or NetBox, you should not use OVN native IPAM. There are two integration models.

**Model 1, the external IPAM writes into NB DB.** The IPAM system creates the LSP with a static address (no `dynamic` keyword). OVN handles only the data plane. IPAM stays at the layer above. For example:

- Infoblox creates a record and assigns the IP.
- A Neutron IPAM plugin reads from Infoblox and creates the LSP with the chosen IP.
- OVN only confirms there is no conflict.

**Model 2, OVN IPAM reports back to the external system.** OVN does the dynamic allocation. A plugin then synchronizes `dynamic_addresses` back into the external IPAM, so the external IPAM can observe the state. This is common in small clusters where the external IPAM is mainly used for reporting and audit.

**Limits of OVN IPAM:**

- No per-tenant quota.
- No VRF concept. One Logical Switch is one subnet, and addresses cannot overlap between switches.
- No automatic DNS integration. You must synchronize with an external DNS yourself.
- No reservation with a TTL.

Enterprise tools such as Infoblox NIOS provide all of these features. If your cluster needs them, integrate through a plugin.

### Guided Exercise 1: configure IPAM with `exclude_ips` and dynamic allocation

> **State verification lab:** the output is based on OVN 22.03 documentation. Pending verification on a real lab host.

**Goal:** show that OVN IPAM allocates an IP to each LSP automatically, respects `exclude_ips`, and resolves conflicts with static LSPs.

**Setup:** an Ubuntu 22.04 lab with OVN 22.03.

**Steps:**

**1.** Create a Logical Switch with a subnet and an exclude list:

```
[host]# ovn-nbctl ls-add ipam_ls
[host]# ovn-nbctl set Logical_Switch ipam_ls \
         other_config:subnet=192.168.10.0/24
[host]# ovn-nbctl set Logical_Switch ipam_ls \
         other_config:exclude_ips="192.168.10.1..192.168.10.20 192.168.10.100..192.168.10.110"
```

**2.** Create three dynamic LSPs:

```
[host]# ovn-nbctl lsp-add ipam_ls pod1
[host]# ovn-nbctl lsp-set-addresses pod1 "dynamic"
[host]# ovn-nbctl lsp-add ipam_ls pod2
[host]# ovn-nbctl lsp-set-addresses pod2 "dynamic"
[host]# ovn-nbctl lsp-add ipam_ls pod3
[host]# ovn-nbctl lsp-set-addresses pod3 "dynamic"
```

Note: no MAC is provided. OVN generates the MAC automatically.

**3.** Inspect `dynamic_addresses`:

```
[host]# ovn-nbctl get Logical_Switch_Port pod1 dynamic_addresses
"0a:00:00:11:22:01 192.168.10.21"
[host]# ovn-nbctl get Logical_Switch_Port pod2 dynamic_addresses
"0a:00:00:11:22:02 192.168.10.22"
[host]# ovn-nbctl get Logical_Switch_Port pod3 dynamic_addresses
"0a:00:00:11:22:03 192.168.10.23"
```

Observation: OVN skips `192.168.10.1` through `192.168.10.20` (the exclude range) and starts at `192.168.10.21`.

**4.** Add a static LSP with a fixed address:

```
[host]# ovn-nbctl lsp-add ipam_ls gateway
[host]# ovn-nbctl lsp-set-addresses gateway "aa:bb:cc:dd:ee:99 192.168.10.50"
```

**5.** Add a fourth dynamic LSP:

```
[host]# ovn-nbctl lsp-add ipam_ls pod4
[host]# ovn-nbctl lsp-set-addresses pod4 "dynamic"
[host]# ovn-nbctl get Logical_Switch_Port pod4 dynamic_addresses
"0a:00:00:11:22:04 192.168.10.24"
```

Observation: OVN skips `192.168.10.50` (already used by the static LSP) and assigns `192.168.10.24`, the next free dynamic address after the three already allocated.

**6.** Try a custom `mac_prefix`:

```
[host]# ovn-nbctl set Logical_Switch ipam_ls other_config:mac_prefix="52:54:00"
[host]# ovn-nbctl lsp-add ipam_ls vm1
[host]# ovn-nbctl lsp-set-addresses vm1 "dynamic"
[host]# ovn-nbctl get Logical_Switch_Port vm1 dynamic_addresses
"52:54:00:a1:b2:c3 192.168.10.25"
```

Observation: the new MAC carries the prefix `52:54:00` instead of the default.

**Lessons learned:**

First, OVN IPAM is **built in and zero-configuration**. Set `other_config:subnet`, and every LSP with `addresses=dynamic` receives an IP automatically. There is no external service.

Second, `exclude_ips` is critical for reserved addresses. The gateway IP, the DHCP server IP, and any floating VIP must be excluded so that OVN does not assign them to a pod, which would create a network-level conflict.

Third, OVN IPAM has **no per-LSP quota** and no `max_address_count`. If a `/24` subnet has 254 usable addresses and the cluster grows to 300 pods, the 255th allocation simply fails silently. The LSP ends up with no `dynamic_addresses`. A production cluster must monitor `dynamic_addresses` to alert when the subnet is close to full.

**Cleanup:**

```
[host]# ovn-nbctl ls-del ipam_ls
```

## 13.12.5 IPAM allocation algorithm at the source level

`ovn-northd` runs IPAM allocation when it compiles NB DB. Source code anchors at `branch-22.03`:

- [`northd/ipam.c`](https://github.com/ovn-org/ovn/blob/branch-22.03/northd/ipam.c), function `ipam_get_unused_ip()`, allocates an unused IP within the subnet.
- Function `build_ipam()` in `northd/northd.c`, called from `ovnnb_db_run()` for each Logical Switch that has a `subnet` configured.

### Algorithm pseudo-code

```c
// Pseudo-code for IPAM allocation
for (lsp in logical_switch.ports) {
    if (lsp.addresses == "dynamic") {
        // 1. Build the set of used IPs in this LS.
        used_ips = {};
        for (other_lsp in logical_switch.ports) {
            if (other_lsp.dynamic_addresses) {
                used_ips.add(parse_ip(other_lsp.dynamic_addresses));
            }
            if (other_lsp.addresses != "dynamic") {
                used_ips.add(parse_ip(other_lsp.addresses));  // static
            }
        }

        // 2. Add exclude_ips and the LRP gateway IP.
        used_ips.add_range(parse_exclude_ips(ls.exclude_ips));
        if (lrp_ip) used_ips.add(lrp_ip);  // LRP gateway

        // 3. Walk the subnet and pick the first unused IP.
        for (ip in subnet_range) {
            if (!used_ips.contains(ip)) {
                lsp.dynamic_addresses = generate_mac() + " " + ip;
                break;
            }
        }

        // 4. If the subnet is exhausted, fail silently (lsp.dynamic_addresses = "").
    }
}
```

### MAC address generation

`mac_prefix` (3 bytes) plus 3 random bytes equals a 6-byte MAC:

```c
// generate_mac() simplified
mac[0..2] = mac_prefix;  // default 0a:00:00 (locally administered)
mac[3..5] = random_3_bytes();
```

The default prefix is `0a:00:00`, which is the IEEE locally-administered range. With 3 random bytes, the collision probability is approximately one in 16 million. A cluster at one million LSPs may see duplicates. In production, set `mac_prefix` per tenant to shrink the collision scope.

### IPv6 EUI-64 mode

An IPv6 LSP can use EUI-64 per RFC 4291 §2.5.1:

```bash
[host]# ovn-nbctl set Logical_Switch ls1 other_config:ipv6_prefix=2001:db8::/64
[host]# ovn-nbctl lsp-add ls1 vm1
[host]# ovn-nbctl lsp-set-addresses vm1 "dynamic"
# OVN derives the IPv6 EUI-64 from the MAC: 0a:00:00:aa:bb:cc -> 2001:db8::0800:00ff:feaa:bbcc
```

EUI-64 derives deterministically from the MAC, so OVN does not need to track allocated state. Predictable, plus a 64-bit subnet has 18 quintillion addresses, so exhaustion is not a concern.

## 13.12.6 Anatomy Template A: `ovn-nbctl get Logical_Switch other_config`

> **Context:** check the IPAM configuration of a Logical Switch. Useful when an LSP does not receive a dynamic address.

```bash
[host]# ovn-nbctl get Logical_Switch ls1 other_config
{exclude_ips="192.168.10.1..192.168.10.20",
 ipv6_prefix="2001:db8::/64",
 mac_prefix="52:54:00",
 subnet="192.168.10.0/24"}
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| `subnet` | Valid CIDR notation | Empty | Mismatch with the expected value | LSPs marked `dynamic` do not receive an IP |
| `exclude_ips` | Range covers gateway and reserved addresses | Empty (default excludes only `.1`) | The whole subnet is excluded | No IP available |
| `mac_prefix` | 3-byte hex | Empty (default `0a:00:00`) | Vendor-specific MAC conflict | MAC duplicate across the cluster |
| `ipv6_prefix` | `/64` for EUI-64 | Empty (no IPv6) | Shorter than `/64` (EUI-64 requires `/64`) | IPv6 dual stack broken |

```bash
# Inspect IP usage in the LS.
[host]# ovn-nbctl find Logical_Switch_Port \
        '_uuid=={...}' --columns=name,addresses,dynamic_addresses
name              addresses           dynamic_addresses
"vm1"             ["dynamic"]         "0a:00:00:aa:bb:01 192.168.10.21"
"vm2"             ["dynamic"]         "0a:00:00:aa:bb:02 192.168.10.22"
"gateway"         ["aa:bb:cc:dd:ee:99 192.168.10.50"]    []
"vm3"             ["dynamic"]         ""    <-- FAIL: IPAM exhausted
```

**Failure modes to break things:**

- One or more LSPs have `dynamic_addresses=""` (empty): IPAM exhaustion. Count allocated IPs plus the exclude range and compare to the subnet size. Grow the subnet or shrink `exclude_ips`.
- Two LSPs share the same `dynamic_addresses` (collision): a rare `ovn-northd` bug. Restart `ovn-northd` to re-run IPAM.
- A static LSP holds an IP inside the dynamic range without an exclude entry: collision waiting to happen. Add the static IP to `exclude_ips` and recreate the dynamic LSPs.

## 13.12.7 Capstone POE: IPAM scale, 10K LSPs on one `/24` subnet

> **Context:** the cluster needs 1000 pods in one tenant, on a single `/24` subnet (254 usable addresses).

**Claim to verify:** "A `/24` subnet with 254 usable IPs is enough for 1000 pods if the pod lifecycle is short, less than one minute."

**Predict:** the claim is wrong. OVN IPAM allocates an IP per LSP, with no concept of "release on disconnect". 254 usable addresses is less than 1000 needed. Result: exhaustion.

**Observe (assumed):**

- Set up a Logical Switch with a `/24` subnet, generate 254 dynamic LSPs. Allocation succeeds.
- The 255th LSP shows `dynamic_addresses=""` (silent failure).
- Delete 100 of the older LSPs, then allocate 100 new ones: success. The freed IPs are reused.
- Pod churn at 100 per second for one minute equals 6000 pods created and deleted. The IP table churn drives a CPU spike on `ovn-northd`.

**Explain:**

1. **OVN IPAM has no per-tenant quota.** The only limit is the subnet size. `/24` equals 254 usable.
2. **An IP is released when the LSP is deleted.** Deleting an LSP clears `dynamic_addresses`, and the address becomes available again.
3. **Concurrent allocation race.** When 100 LSPs are created at the same time, `ovn-northd` compiles sequentially and allocates 100 IPs. If the compile lag is long (100 LSPs at 100 ms each is 10 seconds), pod restart timeouts may fire.
4. **Solution.** Use a larger subnet, for example `/22`, which gives 1022 IPs for 1000 pods plus buffer. Or use a multi-LS architecture, for example 10 switches with `/24` for 2540 IPs total.

**Falsification test:**

- Set up a `/24` Logical Switch and run a script that creates 1000 LSPs. Count how many succeed and how many fail.
- If all 1000 succeed, that is a counter-example. In practice, the cap is 254 minus the exclude range.

**Where the original claim is wrong.** "254 is enough for 1000 pods" assumes the IP is released immediately. In reality, concurrency and churn create a race condition and IP exhaustion. The correct rule is: **the subnet size must equal the peak concurrent LSP count plus a 20 percent buffer. 1000 pods peak means a `/22` subnet (1022 IPs).**

> **Rule:** subnet sizing equals peak concurrent LSPs multiplied by 1.2 (a 20 percent buffer). Use `/24` for under 200 LSPs. Use `/22` for 200 to 800. Use `/20` for 800 to 3200. Do not "stretch" a `/24` past 200.

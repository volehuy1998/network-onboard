## 13.3.1 ACL

OVN ACL is stateful: connection state is tracked automatically through the `ct()` action.

```bash
ovn-nbctl acl-add <ls-or-pg> <direction> <priority> <match> <action>
```

- `<direction>`: `from-lport` (ingress from a VM) or `to-lport` (egress toward a VM).
- `<priority>`: 0 to 32767. Higher priority wins.
- `<action>`: `allow`, `allow-related`, `drop`, or `reject`.

`allow-related` is `allow` plus auto-allow of the reply through conntrack.

```bash
[admin]# ovn-nbctl acl-add ls-tenant1 from-lport 1000 \
         "ip4.src == 10.0.0.0/24 && tcp.dst == 80" allow-related
[admin]# ovn-nbctl acl-add ls-tenant1 from-lport 500 "1" drop
```

---

## 13.3.2 Port_Group

Port_Group groups LSPs so a single ACL can apply to all of them:

```bash
[admin]# ovn-nbctl pg-add pg-web web-vm1 web-vm2 web-vm3
[admin]# ovn-nbctl acl-add pg-web from-lport 1000 \
         "outport == @pg-app && tcp.dst == 8080" allow-related
```

The `@pg-web` shortcut references the group inside a match expression.

## 13.3.3 Load Balancer

OVN L4 load balancer:

```bash
[admin]# ovn-nbctl lb-add lb1 10.0.100.100:80 10.0.1.1:8080,10.0.1.2:8080
[admin]# ovn-nbctl ls-lb-add ls-tenant1 lb1
```

- VIP: 10.0.100.100:80.
- Backends: 10.0.1.1:8080 and 10.0.1.2:8080.

Health checks are supported in OVN 22.03 and later. Algorithm: hash on the inner five-tuple, consistent hashing for session persistence.

## 13.3.4 NAT

A logical router supports three types:

**SNAT** (source NAT):

```bash
[admin]# ovn-nbctl lr-nat-add lr-tenant1 snat 192.0.2.100 10.0.0.0/24
```

**DNAT_and_SNAT** (one-to-one floating IP):

```bash
[admin]# ovn-nbctl lr-nat-add lr-tenant1 dnat_and_snat 192.0.2.101 10.0.1.10
```

**DNAT**: usually combined as `dnat_and_snat`.

---

## 13.3.5 Gateway router

NAT requires a gateway chassis:

```bash
[admin]# ovn-nbctl lrp-set-gateway-chassis lrp-external chassis-1 100
[admin]# ovn-nbctl lrp-set-gateway-chassis lrp-external chassis-2 50
```

chassis-1 has priority 100, chassis-2 has priority 50 (the failover candidate). The active chassis handles every north-south packet of the logical router.

## 13.3.5a Anatomy Template A: `ovn-nbctl acl-list` output

**Operational context.** After adding ACLs to a Port_Group in the three-tier web, app, and database scenario, verify that the ACL is registered correctly in NBDB. The command is the entry point for debugging when traffic does not flow as expected.

**Command and output** *(reproduced on OVN 22.03.8)*:

```shell
$ sudo ovn-nbctl acl-list pg-app
from-lport  1000 (outport == @pg-app && tcp.dst == 8080) allow-related
from-lport     1 (ip) drop

$ sudo ovn-nbctl --columns=direction,priority,match,action list ACL
direction  : from-lport
priority   : 1000
match      : "outport == @pg-app && tcp.dst == 8080"
action     : allow-related
external_ids: {source="tenant-T1", compliance="pci-dss"}
log        : false
severity   : []
meter      : []
name       : "acl-app-port-8080"
options    : {apply-after-lb="true"}
```

**Nine-attribute anatomy:**

| Field | Sample value | Meaning | Red flag if |
|-------|-------------|---------|--------------|
| `direction` | `from-lport` or `to-lport` | Traffic direction relative to the VM | Wrong direction means the ACL never hits (traffic flows the other way) |
| `priority` | 1000 | 0 to 32767, higher priority wins | Two ACLs with the same priority and overlapping match produce undefined order |
| `match` | `outport == @pg-app && tcp.dst == 8080` | OVN expression language | A typo in a field name (for example `tcp_dst` instead of `tcp.dst`) silently fails to match |
| `action` | `allow-related` | Five types: allow, allow-related, allow-stateless, drop, reject, pass (23.03 and later) | `allow` without an explicit reverse rule produces an asymmetric ACL; the VM never receives the reply |
| `external_ids` | `source="tenant-T1"` | CMS metadata tagging | Missing tags break the audit trail |
| `log` | `false` or `true` | Emit an ACL log entry on match | `log=true` plus a high-rate flow floods the log |
| `severity` | `[]`, `info`, `warning`, or `alert` | Log severity when `log=true` | Conflicts with `log=false` |
| `meter` | `[]` or a meter name | Rate-limit ACL log emission | Unset combined with `log=true` and a DDoS produces a log storm |
| `name` | `"acl-app-port-8080"` | Optional friendly name | Missing name makes the rule hard to grep in an audit report |

**Failure scenarios:**

- **`direction=from-lport` with `outport==X`**: confusion. `from-lport` means traffic ingressing the port, so the match should reference `inport==`, not `outport==`. Mixed direction means the ACL never hits correctly. Fix: `from-lport` uses `inport` only; `to-lport` uses `outport` only. Cross-check the pipeline stage in Part 13.2 §13.2.7.
- **Priority `1` (default-deny) above priority `1000` (allow)**: the allow never hits. Higher priority wins in OVN. Default-deny must always have the lowest priority.
- **Match typo `tcp_dst` instead of `tcp.dst`**: ovn-northd rejects with a syntax error in `/var/log/ovn/ovn-northd.log`. Check the log when an ACL has no effect.
- **`apply-after-lb` option missing for an ACL matching `ct.new`**: the ACL is evaluated before the LB, so CT state does not yet exist. Set `options:apply-after-lb=true` so the ACL is evaluated after the LB and CT.

## 13.3.5b Misunderstanding callouts and Key Topic

> **Common misunderstanding 1.** *"OVN ACL works like iptables: rules are matched linearly, top to bottom, in insertion order."*
>
> **Reality.** OVN ACL matches by **priority**, not by order of insertion. The higher priority wins. With iptables, order matters (first-match wins); with OVN ACL, priority matters. If ACLs are added in "natural order" (specific first, general second) but priorities are set the other way around (specific priority low, general priority high), the general rule wins and the specific rule never hits. Best practice: set priority explicitly and use priority ranges (for example, 2000 to 2999 for allow, 1 to 100 for default-deny) to avoid collisions.

> **Common misunderstanding 2.** *"`allow-related` is just `allow` plus an automatically inserted reverse ACL."*
>
> **Reality.** `allow-related` does not create a separate reverse ACL. Instead, it triggers the `ct(commit, zone=X)` action on the forward packet and inserts a match on `ct.est && ct.rpl` at ingress, so the reply packet hits the conntrack established state. The mechanism is conntrack, not rule duplication. The distinction matters when debugging: `ovn-nbctl acl-list` shows only one rule (the forward direction); it does not show an "auto-generated" reverse rule. The reply path is handled by the `ct.est` and `ct.rpl` matches at pipeline stages `ls_in_acl_eval` and `ls_out_acl_eval`.

> **Key Topic.** Port_Group is the scale foundation for OVN micro-segmentation. At a scale of 1,000 VMs with 100 security rules, a naive per-VM ACL would create 100,000 ACL rules, which causes a northd compile-time explosion. Port_Group lets the operator add one ACL rule once and apply it to N VMs in the group. The ovn-northd compile uses the `conjunction()` action to reduce the OpenFlow rule count from the cross product M times N down to M plus N plus 1 (see §13.3.6 conjunction deep dive). At 100 rules and 1,000 VMs: 100 ACLs become 200 plus 1,000 plus 100 equals 1,300 OpenFlow rules instead of 100,000, a 99 percent reduction. This is the reason Kubernetes NetworkPolicy translation to OVN can scale to thousands of pods.

## 13.3.6 Three-tier ACL workflow

```bash
# Port groups
[admin]# ovn-nbctl pg-add pg-web web-vm1 web-vm2
[admin]# ovn-nbctl pg-add pg-app app-vm1 app-vm2
[admin]# ovn-nbctl pg-add pg-db db-vm1

# Allow specific
[admin]# ovn-nbctl acl-add pg-app from-lport 1000 \

[admin]# ovn-nbctl acl-add pg-db from-lport 1000 \
         "outport == @pg-db && tcp.dst == 5432" allow-related

# Default deny
[admin]# ovn-nbctl acl-add pg-web from-lport 1 "ip" drop
[admin]# ovn-nbctl acl-add pg-app from-lport 1 "ip" drop

```

## Workflow summary

First, ACL is automatically stateful through `allow-related`: conntrack handles the reply, so a separate reverse rule is not required.

Second, Port_Group is the foundation for micro-segmentation at scale. Apply an ACL per group rather than per LSP.

Third, the OVN Load_Balancer is L4 only. For L7, use an external load balancer such as HAProxy or Envoy.

Fourth, NAT requires a gateway chassis, which is a north-south chokepoint. Scale is limited by the single active gateway.

## 13.3.7 OVN stateful ACL deep dive: match expression and conjunction

OVN ACLs are translated into OpenFlow logical flows at the `ls_in_acl_eval` and `ls_out_acl_eval` stages (Part 13.2). This section goes deep on match expression syntax and conjunction usage.

### 13.3.7.1 Match expression syntax

The OVN ACL match is an OVN expression (commonly called "lflow match syntax") that compiles into an OpenFlow match. The language:

```
Operators: && || ! == != < > <= >=
Boolean: 0 / 1
Symbols: inport, outport, eth.src, eth.dst, eth.type,
         ip, ip4, ip6, ip4.src, ip4.dst, ip.proto,
         tcp, tcp.src, tcp.dst, tcp.flags, udp.src, udp.dst,
         arp, arp.op, arp.spa, arp.tpa,
         ct.new, ct.est, ct.rel, ct.rpl, ct.inv, ct.trk,
         ct_mark, ct_label, ct_zone,
         @port_group, $address_set
```

Examples:

```
# Allow TCP from web group to db port 3306, stateful
(inport == @pg_web && ip4.dst == $ip_db && tcp.dst == 3306)

# Block outbound RDP (3389) except admin
(outport == @pg_all && tcp.dst == 3389 && inport != @pg_admin)

# Allow ICMP echo
(icmp4 && icmp4.type == 8 && icmp4.code == 0)

# Drop invalid CT
(ip && ct.inv)
```

### 13.3.7.2 ACL action `allow-related` semantics

OVN has five ACL action types:

| Action | Semantics |
|---|---|
| `allow` | Stateless allow (no CT tracking) |
| `allow-related` | Allow plus CT commit, with the reply automatically allowed |
| `allow-stateless` | Force stateless (bypass CT even on `ip`) |
| `drop` | Silent drop |
| `reject` | Send ICMP Destination Unreachable or TCP RST reply |
| `pass` | Skip to the next stage (OVN 23.03 and later) |

**`allow-related` versus `allow`:**

```shell
# allow-related: outbound commits, inbound reply is automatically allowed
$ ovn-nbctl acl-add ls-prod from-lport 1000 \
    "inport == @pg_web && ip4.dst == $ip_internet" allow-related

# With allow-related, the engineer does not need to add an ACL for the reply
```

```shell
# stateless allow: a reply ACL must be added explicitly
$ ovn-nbctl acl-add ls-prod from-lport 1000 \
    "inport == @pg_web && ip4.dst == $ip_internet" allow

$ ovn-nbctl acl-add ls-prod to-lport 1000 \
    "outport == @pg_web && ip4.src == $ip_internet" allow
```

`allow-related` is almost always the correct choice (except for purely stateless workloads such as OSPF or BGP).

### 13.3.7.3 Conjunction compression in practice

Part 4.9 §4.9.25 introduces the `conjunction()` action. OVN uses conjunction automatically when the ACL match references a Port_Group or Address_Set with many entries.

Example: Port_Group `@pg_web` has 50 ports, `@pg_db` has 20 ports:

- **Without conjunction:** 50 times 20 equals 1,000 OpenFlow rules.
- **With conjunction:** 50 plus 20 plus 1 equals 71 OpenFlow rules.

ovn-northd generates the conjunction automatically when the match expression has a `@pg_X` cross product. The engineer does not need to add it manually.

Verify with `ovn-sbctl lflow-list`:

```
  table=8 (ls_in_acl_eval), priority=2001,
    match=(reg0[7] == 1 && (inport == @pg_web)),
    action=(conjunction(42, 1/2);)
  table=8 (ls_in_acl_eval), priority=2001,
    match=(reg0[7] == 1 && (ip4.dst == $ip_db && tcp.dst == 3306)),
    action=(conjunction(42, 2/2);)
  table=8 (ls_in_acl_eval), priority=2001,
    match=(conj_id == 42),
    action=(reg0[17] = 1; next;)
```

## 13.3.8 OVN Load_Balancer deep dive

### 13.3.8.1 Schema deep dive

```shell
$ sudo ovn-nbctl list Load_Balancer
_uuid               : 7c9d4e2a-...
name                : "lb-webapp"
protocol            : tcp
vips                : {"10.0.0.100:80"="10.0.1.1:8080,10.0.1.2:8080,10.0.1.3:8080"}
ip_port_mappings    : {"10.0.1.1"="lsp-backend-1:10.0.1.1",
                        "10.0.1.2"="lsp-backend-2:10.0.1.2",
                        "10.0.1.3"="lsp-backend-3:10.0.1.3"}
health_check        : [<Service_Monitor_uuid>]
selection_fields    : [ip_dst,tp_dst]
options             : {affinity_timeout="300", event="true",
                        hairpin_snat_ip="169.254.169.254",
                        neighbor_responder="reachable",
                        reject="true", skip_snat="false"}
```

**Key columns:**

| Column | Purpose |
|---|---|
| `protocol` | `tcp`, `udp`, or `sctp` |
| `vips` | VIP to backend list (comma-separated) |
| `ip_port_mappings` | Backend IP to LSP name mapping (used for hairpin) |
| `health_check` | Reference to a Service_Monitor for backend health |
| `selection_fields` | Hash fields used to pick a backend (default five-tuple) |
| `options:affinity_timeout` | Session persistence timeout (seconds) |
| `options:event` | Emit a Controller_Event when the LB is empty |
| `options:hairpin_snat_ip` | SNAT IP for the hairpin case (client to VIP back to same backend) |
| `options:reject` | Reject rather than drop when there is no backend |
| `options:skip_snat` | Skip the reply SNAT (used for the DSR pattern) |

### 13.3.8.2 Service_Monitor for backend health

```shell
# Create an HTTP health check
$ sudo ovn-nbctl --id=@hc create Load_Balancer_Health_Check \
    vip='"10.0.0.100:80"' \
    options='"interval"="5" "timeout"="2" "success_count"="3" "failure_count"="3"'

$ sudo ovn-nbctl set Load_Balancer lb-webapp health_check=@hc

# Check status
$ sudo ovn-sbctl list Service_Monitor
_uuid       : ...
ip          : "10.0.1.1"
port        : 8080
protocol    : tcp
logical_port: "lsp-backend-1"
src_ip      : "10.0.0.100"
src_mac     : "40:44:00:00:00:01"
status      : online   /* online, offline, or error */
```

The health check runs from the chassis that owns the `logical_port`. When the status is "offline", ovn-northd regenerates the logical flow with that backend removed from the `ct_lb_mark()` backend list.

## 13.3.9 OVN NAT deep dive (stateful and stateless)

### 13.3.9.1 SNAT pattern

```shell
# The whole 10.0.0.0/24 subnet is SNATed to the external IP 203.0.113.100
$ sudo ovn-nbctl lr-nat-add lr-tenant snat 203.0.113.100 10.0.0.0/24

# Conntrack flow:
# outbound:  10.0.0.5:50000 -> 8.8.8.8:443 (new, trk)
#            apply SNAT:  src = 203.0.113.100:32768 (OVN picks a free port)
#            commit to conntrack zone_lr
# inbound :  8.8.8.8:443 -> 203.0.113.100:32768 (est, rpl)
#            reverse SNAT: dst = 10.0.0.5:50000
```

### 13.3.9.2 DNAT_and_SNAT (floating IP pattern)

```shell
# Floating IP 203.0.113.50 mapped to 10.0.0.5 (lsp-vm-web)
$ sudo ovn-nbctl lr-nat-add lr-tenant dnat_and_snat \
    203.0.113.50 10.0.0.5 lsp-vm-web 40:44:00:00:00:50

# Inbound: 8.8.8.8:443 -> 203.0.113.50:80 (new, trk)
#          DNAT: dst = 10.0.0.5:80
# Outbound reply: 10.0.0.5:80 -> 8.8.8.8:443 (est, rpl)
#                 reverse SNAT: src = 203.0.113.50:80
```

The `external_mac` field is used for the Gratuitous ARP emitted when a floating IP is freshly assigned. When null, OVN generates a random MAC.

### 13.3.9.3 Stateless NAT (rare)

```shell
# Stateless one-to-one mapping, no CT tracking
$ sudo ovn-nbctl lr-nat-add lr-tenant dnat 203.0.113.50 10.0.0.5
$ sudo ovn-nbctl set NAT <uuid> options:stateless=true
```

Use case: an extremely high-bandwidth L4 DSR LB frontend. Trade-off: no flow tracking, and the reply path differs.

## Core takeaways

First, OVN ACL uses an expression language (the lflow match syntax) covering inport and outport, eth.*, ip.* (ip4.*, ip6.*), tcp.* and udp.*, ct.*, `@port_group`, and `$address_set`. The engineer must know it to debug `ovn-sbctl lflow-list`.

Second, Port_Group is the foundation of micro-segmentation at scale. Apply ACL per group.

Third, the OVN Load_Balancer is L4 only. For L7, use an external load balancer such as HAProxy.

Fourth, NAT requires a gateway chassis, which is a north-south chokepoint. Scale is limited.

Fifth, `allow-related` is the default choice (no manual reply ACL is required). Conjunction compression is automatic when an ACL has a Port_Group times Address_Set cross product.

Sixth, the Service_Monitor for an LB health check runs from the chassis that owns the backend `logical_port`. When a backend goes offline, ovn-northd regenerates the logical flow without that backend in `ct_lb_mark()`.

## 13.3.10 Capstone POE: three-tier segmentation scale prediction

**Goal.** Assess the engineer's ability to estimate OVN flow counts and decide "segment or not" for a large tenant.

**Scenario.** Telco tenant T1 has 500 VMs across three tiers: 100 web, 200 app, 200 db. Requirements:
- Default-deny on all traffic.
- Allow web to app on port 8080.
- Allow app to db on port 5432.
- Allow app to the internet (external IP).
- Allow an admin jumphost (one separate VM) to every tier on port 22.

Questions: (a) How many ACL rules must be added to NBDB? (b) How many OpenFlow rules per chassis (assuming 50 hypervisors with 10 local VMs each)? (c) Should the design use Port_Group or per-LSP ACL? (d) How does the equivalent Kubernetes NetworkPolicy pattern scale?

**Predict:**

### (a) ACL rule count in NBDB

Approach 1, per-LSP:
- 100 web to 200 app allow: 100 times 200 equals 20,000 rules.
- 200 app to 200 db allow: 200 times 200 equals 40,000 rules.
- 200 app to internet: 200 rules.
- Jumphost to every tier on SSH: 1 times 500 equals 500 rules.
- Default-deny: 500 rules.
- **Total: about 61,000 ACL rules in NBDB.**

Approach 2, Port_Group:
- Four groups: `pg_web`, `pg_app`, `pg_db`, `pg_jump`.
- web to app: 1 ACL (match `outport == @pg_app && tcp.dst == 8080 && inport == @pg_web`).
- app to db: 1 ACL.
- app to internet: 1 ACL.
- jump to every tier: 1 ACL per tier times 3 equals 3 ACLs.
- Default-deny: 1 ACL per LS, so 1 ACL.
- **Total: 7 ACL rules in NBDB.**

**Prediction: the Port_Group approach reduces NBDB from 61,000 to 7 rules, a 99.99 percent reduction.**

### (b) OpenFlow rules per chassis

Each chassis has 10 local VMs. ovn-controller compiles the logical flow (SBDB) into OpenFlow rules on br-int. Conjunction compression:

- `pg_web` cross `pg_app` match: 100 plus 200 plus 1 equals 301 conjunction members plus head rule (per cluster, replicated on every chassis).
- `pg_app` cross `pg_db`: 200 plus 200 plus 1 equals 401 rules.
- `pg_app` cross internet: 200 times 1 equals 200 rules.
- `pg_jump` cross every tier: 1 times 500 equals 500 rules.
- Default-deny: 3 rules (1 per LS).
- Baseline per VM (30 rules times 10 local VMs equals 300 rules).
- **Total: about 1,700 OpenFlow rules per chassis.**

Naive approach (per-LSP):
- Ordered match: 61,000 rules times 10 local VMs equals 610,000 OpenFlow rules per chassis.
- The Trident2 TCAM (8K) overflows and falls back to the SMC software path, producing latency spikes.

**Prediction: Port_Group plus conjunction equals 1,700 rules and fits TCAM comfortably. The naive approach equals 610K rules and overflows TCAM.**

### (c) Recommendation

Port_Group plus conjunction is mandatory at this scale. The naive per-LSP approach fundamentally does not scale.

### (d) Kubernetes NetworkPolicy pattern

The OVN-Kubernetes CNI translates NetworkPolicy into OVN ACL automatically. Each pod label matched by a NetworkPolicy podSelector causes ovn-northd to generate a dynamic Port_Group per label combination. Result: a single Kubernetes NetworkPolicy applied to 500 pods becomes 1 OVN ACL rule plus 1 Port_Group (with 500 members). Conjunction compresses this exactly the same way as the manual case. OVN-Kubernetes scales to 10,000 plus pods per cluster with NetworkPolicy thanks to this pattern.

**Observe:**

Lab simulation with 10 VMs (scaled-down version):
```bash
$ sudo ovn-nbctl acl-list pg_web | wc -l
1
$ sudo ovs-ofctl -O OpenFlow13 dump-flows br-int | wc -l
~700  # Close to the predicted 650 to 800 per chassis with 10 local VMs.
```

The scale-up formula holds.

**Explain:**

Three universal principles for OVN ACL scale:

1. **Never use per-LSP ACL beyond 10 VMs.** Cross-product explosion starts at small scale. Port_Group is the default.
2. **Conjunction compression is automatic when an ACL match references at least two Port_Groups.** No manual tuning required.
3. **Service_Monitor plus LB health check distributes at most one check per chassis per backend.** Does not overload the gateway chassis.

**Falsification test.** Add 1,000 nonspecific ACL rules (no Port_Group, raw IP) on a Logical Switch. Prediction: TCAM overflows around 5K rules. Observation: the Trident2 ACL TCAM fills at 8K, after which the SMC cache fallback runs 2 to 10 times slower. Confirms that Port_Group is not optional at production scale.

**Finish.** The principle: **"Port_Group first, per-LSP never."** This applies to OVN, Kubernetes NetworkPolicy, and any SDN abstraction with grouping support.

## 13.3.X ACL as an NBDB table: deep dive on the access control rule

> This section goes deep on the **`ACL` table itself** (a row in the OVN Northbound database) and complements the practical material in §13.3.1 and the pipeline introduced in §13.3.7. Goal: the engineer should fully understand the structure of an NBDB ACL row, the 11-column schema in 22.03, the lifecycle of compiling a row into Logical_Flow at stages `ls_in_acl_eval` and `ls_out_acl_eval`, and how to trace from a runtime incident (an unexpectedly dropped packet, a missing reply, a missing log entry) back to a specific row in NBDB.

### Core concept

`ACL` is a **table that lives in the OVN Northbound database (NBDB)** describing an **access control rule** declared by the CMS (Neutron security group, Kubernetes NetworkPolicy, kube-ovn ovn-policy) to gate L3 or L4 traffic into or out of a `Logical_Switch` or a `Port_Group`. Each row represents a **single** rule with `direction`, `priority`, `match` (microflow expression), `action`, and the optional `log`, `severity`, `meter`, `label`, and `options` columns. Type: **OVSDB table**, not a root table (`isRoot=false` in `ovn-nb.ovsschema`); an ACL row exists only when referenced via a strong reference from `Logical_Switch.acls` or `Port_Group.acls` (cascade delete when the LS or Port_Group is deleted). Two key differences from Linux iptables: an OVN ACL is **not** applied in insertion order (first-match wins); it is applied by `priority` (highest wins). Also, an OVN ACL is **automatically stateful** through the `allow-related` action plus conntrack; no separate reverse rule is needed.

### History and origin

The `ACL` table appeared in OVN 2.4 (2015) in the Russell Bryant (Red Hat) and Justin Pettit (VMware and Nicira) refactor that moved the Nicira NSX OpenStack plugin architecture into OVN. Before that, Neutron Security Group was implemented by `neutron-openvswitch-agent` pushing **iptables** rules per host on each compute node via `iptables-restore`. Problems compounded: 5,000 VMs equals 5,000 iptables chains equals slow lookup, plus an atomic full-table reload on every rule change. OVN solves this by pushing a declarative ACL into NBDB Raft, having ovn-northd compile it into Logical_Flow distributed evenly to every chassis ovn-controller, and using OpenFlow plus conntrack instead of iptables. Schema evolution (verified against `ovn-nb.ovsschema` on GitHub branches): OVN 2.6 added `log` and `severity`; OVN 2.10 added `meter` to rate-limit log emission; OVN 21.06 added `name`; OVN 22.03 stabilised `external_ids`; OVN 22.06 added `label` (uint32, used in IPFIX flow records); OVN 22.09 added the `options` map (containing `apply-after-lb` and `log-related`); OVN 23.03 added the fifth action `pass`; OVN 24.03 added the `sample` action for IPFIX sampling. Initially `action` had four values (`allow`, `allow-related`, `drop`, `reject`); `allow-stateless` was proposed by Han Zhou (eBay, later Red Hat) for OVN 21.03 to support purely stateless workloads such as BGP and OSPF that need to bypass conntrack.

### Position in the architecture

The table sits at the **NBDB layer**, the topmost input layer of the OVN control plane, but it is **not a root table**. ACL exists only when referenced from `Logical_Switch.acls` or `Port_Group.acls` (strong reference). Scope: **per-Logical_Switch** (or **per-Port_Group** when attached to a group). An OVN cluster can have tens of thousands of ACL rows (the Cartesian product of Logical_Switch count times rules per LS, or Port_Group count times rules per group). Post-compile blast radius: changing one ACL row (editing `match`, changing `priority`, or changing `action`) makes ovn-northd recompute **only** the Logical_Flow of the affected LS or Port_Group (incremental compile from OVN 21.03 and later); ovn-controller on every chassis hosting a Port_Binding for an LSP in that LS reinstalls the flow in under one second through the I-P engine. ACL is **never** applied to `Logical_Router` or `Logical_Router_Port`; the rule that controls L3 traffic on an LR is `Logical_Router_Policy` (a separate table, not ACL).

### Role and responsibilities

ACL carries **five roles** in the OVN architecture:

1. **Decide allow or block**: the `action` column (a five-value enum) is the final decision for every packet that matches the highest-priority rule. This is the micro-segmentation gate.
2. **Define rule scope**: the `direction` column (`from-lport` for ingress into the LS pipeline from the VM side, `to-lport` for egress out of the LS pipeline toward the VM side) attaches the rule to the correct pipeline stage. `from-lport` compiles into a flow at stage `ls_in_acl_eval`; `to-lport` compiles into a flow at stage `ls_out_acl_eval`.
3. **Declare stateful intent through `allow-related`**: this action does not create a reverse rule; it triggers `ct(commit)` on the forward path and matches `ct.est && !ct.rpl.<flag_blocked>` on the reply path. The ACL becomes a **stateful rule** without needing a paired reverse entry.
4. **Enable observation and audit**: the columns `log=true`, `severity`, `meter`, `name`, and `label` allow logging on every match (severity routes to syslog or journald, meter rate-limits emission, label is written into the IPFIX flow record). This is the observability hook for compliance (PCI-DSS, HIPAA).
5. **Pass compile parameters to northd**: the `options` column (a map) holds `apply-after-lb=true` (move the rule to run after Load_Balancer DNAT so the match sees the translated destination IP rather than the VIP) and `log-related=true` (log the reply traffic of an `allow-related` rule too).

### Why it exists (design motivation)

Before OVN 2.4, the Neutron Security Group architecture used `neutron-openvswitch-agent` to push iptables rules. The problems stacked up: (a) **iptables atomic reload** with 5,000 chains caused a one-to-three-second stop-the-world every time the CMS added a rule, dropping connections; (b) **no native conntrack zone partition**, so two tenants with overlapping five-tuples caused lookup collisions; (c) **per-host state**, where each compute host managed its own chain, made cross-host consistency dependent on agent push success, and drift was high; (d) the model **did not scale beyond about 100 rules per VM** due to linear iptables TCAM traversal; (e) the model **had no cross-host abstraction**: Security Group in Neutron was a logical concept but had to be expanded into per-host iptables rules, so every VM migration triggered a push of new rules. OVN ACL solves this by **representing the rule once in NBDB Raft, letting northd compile it into Logical_Flow, and letting every ovn-controller install the OpenFlow flows on br-int**. Compile is incremental (only the affected LS recomputes), so a rule change does not affect other VMs. Conntrack zone partition (`ct(zone=N)`) isolates state across LSes. The OF pipeline carries a cookie tag equal to the datapath UUID, so reverse lookup is easy. This model scales to 10K plus pods of NetworkPolicy in OVN-Kubernetes (verified by the Red Hat OSP 17 plus ovn-kubernetes 4.14 benchmark in 2023).

### Problems it solves

The table solves **five distinct technical problems**: (1) **Stateful firewall without a paired reverse rule**: the `allow-related` action plus conntrack automatically permits the reply without a reverse `to-lport` ACL (cutting the rule count in half); (2) **Deterministic priority-based ordering**: a high-priority rule always wins regardless of insertion order, avoiding the iptables-style "new rule shadowed an old rule" bug; (3) **Correct-level scope through Port_Group plus Address_Set**: one rule applies to N ports in the group rather than being copied N times, and conjunction compression reduces OF rules from M times N to M plus N plus 1; (4) **Audit and compliance through log plus label**: PCI-DSS requires logging every deny and sampling every allow to a bank; the `label` column (uint32) marks the IPFIX flow record for an ACL match, and the audit log is used to trace which rule matched during an incident; (5) **Cascade delete on LS removal**: because ACL is a child table with a strong reference from Logical_Switch, deleting the LS deletes the attached ACLs, so the CMS does not need a pseudo-transaction to clean up orphan rules.

### Importance

This is a **tier-1 core table of NBDB**, on par with `Logical_Switch`, `Logical_Router`, `Load_Balancer`, and `NAT`. Reasons: 100 percent of OVN security workflows start with `ovn-nbctl acl-add` or a CMS-driven API rule push; 100 percent of production OpenStack or Kubernetes deployments have at least one default-deny ACL plus several specific allow rules; every stateful-firewall incident ("VM A cannot connect to VM B even though the rule allows it") is an ACL or Port_Group plus Address_Set incident. An engineer who does not understand ACL cannot debug any Security Group issue in a Neutron cluster, nor any NetworkPolicy issue in a Kubernetes cluster (because kube-ovn translates NetworkPolicy into ACL plus Port_Group plus Address_Set). Among the 320 plus keywords in REF, ACL is one of the 15 cornerstone OVN keywords.

### How it works (the lifecycle of compiling one row)

An ACL row passes through **four phases** in its lifecycle:

1. **CMS declaration**: the CMS calls `ovn-nbctl acl-add ls-tenant1 from-lport 1000 'ip4.src == 10.0.0.5 && tcp.dst == 80' allow-related`. The NBDB transaction creates an `ACL` row with `direction="from-lport"`, `priority=1000`, `match="ip4.src == 10.0.0.5 && tcp.dst == 80"`, and `action="allow-related"`, and inserts the row UUID into the `Logical_Switch.acls` set on `ls-tenant1`. The transaction is atomic; the row becomes visible in under 50 ms through NBDB Raft replication.
2. **Detection plus classification plus compilation to Logical_Flow.** ovn-northd (running on the Raft leader of NBDB and writing SBDB) detects the change through an I-P engine input change. The `build_acls` function (`northd/northd.c:6688` at v22.03.8) iterates over each `ovn_datapath` that has an ACL attached, calling `consider_acl` (`northd/northd.c:6333`) for each ACL row. `consider_acl` picks the compile stage based on `direction` and `options:apply-after-lb`: `from-lport` plus `apply-after-lb=false` maps to stage `S_SWITCH_IN_ACL` (logical table 9 at v22.03.8, named `ls_in_acl`); `from-lport` plus `apply-after-lb=true` maps to `S_SWITCH_IN_ACL_AFTER_LB` (logical 16); `to-lport` maps to `S_SWITCH_OUT_ACL` (logical 4). The function then translates the `match` expression to OVN microflow syntax and emits an action: `allow-related` becomes `reg0[1] = 1; reg0[10] = 1; next;` (set the ACL hint flags for "allow plus commit"); `drop` becomes `/* drop */; next;`; `reject` builds a TCP RST or ICMP Destination Unreachable and clones the packet back to the inport; `allow-stateless` becomes `reg0[7] = 1; next;`. If `log=true`, the `build_acl_log` function (`northd/northd.c:6252`) prepends a `log(name="<n>", severity=<sev>, verdict=<allow/drop>, meter="<m>")` action.
3. **Stateful supplement flow**: when the datapath has at least one stateful ACL (`has_stateful=true`), `build_acls` also adds supplementary flows at lower priority to handle reply traffic (`ct.est && !ct.rpl` to `next`, allowing the reply through) and default-deny on invalid (`ct.inv` to drop). This is the foundation of `allow-related` semantics.
4. **Flow installation on each chassis**: SBDB Raft replicates Logical_Flow rows. ovn-controller on every chassis subscribes; each controller runs `lflow_run` (`controller/lflow.c`) and translates the Logical_Flow expression into concrete OpenFlow rules. At v22.03.8 the ingress ACL pipeline lands in OF tables 16 (acl_hint, logical 8), 17 (acl, logical 9), and 24 (acl_after_lb, logical 16); egress lands in OF table 44 (acl, LS egress logical 4). Flows are tagged with `cookie=<datapath_uuid_low_64>` so they can be filtered by LS.

Invariant: at any point in time, one ACL row in NBDB corresponds to **one or more** Logical_Flow rows in SBDB (one forward, plus possibly one or two supplements when stateful), and the same set of OpenFlow rules on every chassis hosting a Port_Binding in the LS or Port_Group. Deleting an ACL row triggers an ovn-northd recompute, the old Logical_Flows are evicted, and ovn-controller deletes the corresponding OF flows in under one second.

### Column classification

The 11 columns of ACL (verified against `ovn-nb.ovsschema` on branch-22.03 [GitHub raw](https://raw.githubusercontent.com/ovn-org/ovn/branch-22.03/ovn-nb.ovsschema)) split into **four groups** by semantic role:

| Group | Column | Type | Required | Notes |
|-------|--------|------|----------|-------|
| Identity | `name` | string maxLength=63 | No (default `[]`) | Optional human-readable name, used in log messages |
| Identity | `external_ids` | map string to string | No | CMS-tracked: `neutron:security_group_rule_id`, `kube-ovn:np-name` |
| Match logic | `direction` | enum {from-lport, to-lport} | Yes | Determines compile stage (in versus out) |
| Match logic | `priority` | integer 0 to 32767 | Yes | Highest wins; high range for explicit allow, low range for default-deny |
| Match logic | `match` | string | Yes | OVN microflow expression, BNF in `ovn-sb(5)` |
| Match logic | `action` | five-value enum | Yes | allow, allow-related, allow-stateless, drop, reject |
| Match logic | `options` | map string to string | No (22.09 and later) | `apply-after-lb=true` or `false`, `log-related=true` or `false` |
| Logging | `log` | boolean | Yes (default false) | Enable a log entry on every match |
| Logging | `severity` | enum {alert, warning, notice, info, debug} | No | Log severity level |
| Logging | `meter` | string max=1 | No | Meter name for rate-limiting log emission |
| Logging | `label` | integer 0 to 4,294,967,295 | Yes (default 0) | uint32 written into the IPFIX flow record (22.06 and later) |

ACL has **no unique index constraint** (verified schema, `"isRoot": false`, no `indexes` clause); a duplicate `(direction, priority, match)` is schema-valid; ovn-northd will compile both into Logical_Flow rows and one of the two wins arbitrarily (undefined behaviour). The CMS must enforce uniqueness itself to avoid this edge case.

### How an engineer masters operations

Four foundational skills the engineer needs with ACL:

1. **Read a complete row**: `sudo ovn-nbctl --columns=direction,priority,match,action,log,severity,meter,name,label,options,external_ids list ACL` shows all 11 columns. Or use the shortcut `sudo ovn-nbctl acl-list ls-tenant1` for a human-readable per-LS list, or `sudo ovn-nbctl acl-list pg-web` for a Port_Group.
2. **Verify compile down to SBDB**: `sudo ovn-sbctl lflow-list ls-tenant1 | grep -E "ls_(in|out)_acl"` confirms there is a Logical_Flow at the corresponding stage. Looking up priority and match against the ACL row in NBDB verifies that the compile is correct.
3. **Verify the OF physical layer**: `sudo ovs-ofctl -O OpenFlow15 dump-flows br-int table=9` (in OVN 22.03 the mapping is `ls_in_acl_eval` to OF table 9) or `table=12` for `ls_out_acl_eval`. The cookie equals the low 64 bits of the datapath UUID.
4. **Decide which skill to use when**: when the CMS reports "the rule does not work", start with skill 1 (confirm the row exists with the right direction, priority, and action); when the row is correct but traffic is still dropped, move to skill 2 (cross-check that the Logical_Flow is generated correctly); when the Logical_Flow is correct but the runtime drops, move to skill 3 (verify the OF flow is installed on the chassis). When conntrack is suspected, cross-link to Part 9.24 §dpctl/dump-conntrack.

### Workflow (the standard usage flow)

Standard workflow for adding a three-tier micro-segmentation rule, with a verification command at every step:

1. **Create Port_Groups for each tier**: `ovn-nbctl pg-add pg-web web-vm-1 web-vm-2`. Verify: `ovn-nbctl --columns=name,ports list Port_Group pg-web`.
2. **Add a specific allow rule with `allow-related`**: `ovn-nbctl acl-add pg-web from-lport 1000 'inport == @pg-web && ip4.dst == $as-app && tcp.dst == 8080' allow-related`. Verify: `ovn-nbctl acl-list pg-web` shows the entry, and `ovn-nbctl --columns=match,action,priority list ACL` shows the new row.
3. **Add default-deny at low priority**: `ovn-nbctl acl-add pg-web from-lport 1 'ip' drop`. Verify: `ovn-nbctl acl-list pg-web` lists two rules; priority 1 is below 1000, so the allow rule wins for matching traffic.
4. **Enable log, severity, and meter on the deny rule for audit**: `ovn-nbctl set ACL <uuid> log=true severity=warning meter=audit-meter name=acl-pg-web-deny-default`. Verify: `ovn-nbctl --columns=log,severity,meter,name list ACL <uuid>`.
5. **Wait for northd compile and ovn-controller flow install**: typically under one second. Verify: `ovn-sbctl lflow-list pg-web | grep "ls_in_acl_eval"` shows a flow at priority 2001 (NBDB priority plus a 1000 prefix) for the allow rule and a flow at priority 1001 for the deny rule. Then: `ovs-ofctl -O OpenFlow15 dump-flows br-int table=9 | grep cookie=` shows the OF flow with the cookie corresponding to the datapath.
6. **Test traffic, then track log and counter**: run `ovn-trace ls-tenant1 'inport == "web-vm-1" && eth.src == aa:bb:cc:00:00:01 && eth.dst == aa:bb:cc:00:00:10 && ip4.src == 10.0.0.5 && ip4.dst == 10.0.0.10 && ip.ttl == 64 && tcp.src == 50000 && tcp.dst == 8080'` to simulate; cross-check with `journalctl -u ovn-controller | grep acl-pg-web-deny-default` to see whether the log line fires on a deny.

Production best practice: **always use `allow-related` instead of `allow`** so conntrack handles the reply automatically (no need to declare a reverse rule); **always use Port_Group plus Address_Set** instead of hardcoding IP and MAC so the ACL scales with the tenant; **always set default-deny at priority 1, not priority 0** (priority 0 collides with the default flow generated by northd, leading to undefined override). Anti-patterns: putting an allow rule at priority 1 plus a deny rule at priority 1000 (the deny wins, all traffic drops, the allow never hits); using `allow` (stateless) for TCP traffic without adding a reverse rule, where the VM receives the SYN but the SYN-ACK reply is dropped and the application hangs.

### When something goes wrong (reverse symptom map)

Seven common symptoms that point to ACL:

| Symptom | ACL column to check | Diagnostic flow |
|---------|---------------------|-----------------|
| "VM A cannot connect to VM B even though the rule allows it" | `priority` order, `match` syntax | `ovn-nbctl acl-list <ls>` to inspect priority; `ovn-trace` to confirm the flow path |
| "Half-open connection: SYN goes through but the SYN-ACK reply is dropped" | `action` is `allow` instead of `allow-related` | Switch action to `allow-related` via `ovn-nbctl set ACL <uuid> action=allow-related` |
| "All tenant traffic drops after adding one new rule" | `priority` accidentally 0 default-deny | `ovn-nbctl --columns=priority,action,match list ACL` to find a row with priority=0 and action=drop; or check whether the new rule overrides LS scope |
| "Log does not appear even though `log=true`" | `meter` does not exist, `severity` empty | Verify `ovn-nbctl list Meter <m>` has the corresponding row; if missing, the log is suppressed |
| "northd compile takes more than 5 seconds after pushing 1,000 ACLs" | Port_Group not yet used | Refactor: collapse ACLs onto Port_Group plus Address_Set; conjunction compression cuts the count |
| "ACL match on `ct.est` does not hit for the reply" | `apply-after-lb` not set on LB traffic | Set `options:apply-after-lb=true` on the rule that should run after Load_Balancer DNAT |
| "Reject ICMP Destination Unreachable does not return" | `action=reject` hits the kernel rate limit | Check the meter on the LS; raise the burst rate, or use `drop` instead of `reject` if client feedback is not required |

### Tightly related to

Six components are tightly coupled with ACL (changing any one of them requires re-evaluating ACL):

- **`Logical_Switch.acls`** (NBDB): a strong-reference column holding the ACL UUID set. Deleting the LS cascades to deleting the ACLs attached to it. This is the parent that constrains the lifecycle.
- **`Port_Group.acls`** (NBDB): a similar strong-reference column. Deleting the Port_Group cascades to deleting the ACLs attached to that group.
- **`Address_Set`** (NBDB): not a strong reference, but the ACL `match` expression can reference `$<address_set_name>`. Changing the Address_Set entries changes the IP set the ACL matches at runtime without editing the ACL row.
- **`Meter`** (NBDB): a weak reference through the `meter` column (a string name). Deleting a Meter while an ACL still references it suppresses the log silently (no error, but the log is skipped).
- **`Logical_Flow`** (SBDB): a 1-to-N mapping. Each ACL row generates one to three Logical_Flow rows depending on stateful, log, and apply-after-lb.
- **OVS conntrack zone**: chassis-side. ovn-controller uses the `ct(zone=<datapath_zone_id>)` action; the conntrack zone is partitioned per datapath, so the stateful ACL of LS-A is independent of LS-B (a zone collision does not occur in normal usage).

Loose coupling (referenced, not cascaded): `Logical_Router_Policy` (LR-scoped rule with similar semantics to ACL but in a different table), `QoS` (a different LS table that does not affect ACL matching), `Load_Balancer` (only affects an ACL when the rule has `apply-after-lb=true`).

Reveals: the `n_packets` and `n_bytes` counters per flow are evidence that the rule did or did not hit. Cookie `0xee11ff22` is the low 64 bits of the datapath UUID for Logical_Switch `ls-tenant1` (use `ovn-sbctl --columns=tunnel_key,external_ids list Datapath_Binding` to match).

### Real-world incident

**Production case OVN-2024-Q2 (Telco-T2 cluster, OSP 16.2, about 200 hypervisors, about 5,000 tenant VMs).** An SRE engineer reported 5 percent intermittent drops on east-west traffic between tenant `tenant-app` and tenant `tenant-db` after a new security policy was rolled out (PCI-DSS audit). Service A to Service B requests timed out 5 percent of the time, with no deterministic dependence on a specific VM or specific 5-tuple.

Investigation:

- **Step 1**: `ovn-nbctl acl-list pg-app` confirmed 12 allow rules plus 1 default-deny at priority 1. Rule ordering matched the policy intent.
- **Step 2**: `ovn-trace tenant-app 'inport == "vm-app-1" && ip4.src == 10.0.10.5 && ip4.dst == 10.0.20.10 && tcp.src == 50001 && tcp.dst == 5432'` showed the flow path going through `ls_in_acl_eval` and hitting allow rule at priority 2001 correctly. Trace passed.
- **Step 3**: `ovs-ofctl -O OpenFlow15 dump-flows br-int table=9 | grep cookie=0xee11ff22` on the source chassis showed `n_packets` was not incrementing on the allow rule during the drop window. So the traffic was not hitting the allow rule.
- **Step 4**: cross-checked `ovn-sbctl lflow-list pg-app | grep "priority=2001"` against the priority of the row in NBDB. Discovered **two ACL rows with the same priority=1000 and direction=from-lport**, one rule for `tcp.dst == 5432` (allow) and one rule for `tcp.dst == 5432 && ct.new` (drop, accidentally added by the CMS). Because the schema has no unique index, both compiled into Logical_Flow rows at priority 2001.
- **Step 5**: ovn-controller installed both into OpenFlow table 9 at priority 2001; the OF spec tie-breaker implicitly chooses by cookie or install order, which is non-deterministic. Five percent of traffic hit the drop rule, 95 percent hit the allow rule.

Root cause: the CMS pushed a duplicate ACL row due to a Neutron security group reconciliation bug; the OVN schema has no constraint preventing duplicate `(direction, priority, match)`, so the rules collided.

Fix: `ovn-nbctl acl-del <duplicate_uuid>` removed the wrong drop row; the Neutron team was asked to patch the reconciliation bug. Long-term: consider a CMS-side validator and propose to the OVN community that northd warn when a duplicate `(priority, match)` is detected.

### Synthetic exercise (Lab L13.3.X-1)

**Goal.** Hands-on for the engineer: install a two-tier stateful ACL, observe the compile, and verify the allow-related semantics through conntrack.

**Setup** (Mininet plus OVN on Ubuntu 22.04, OVN 22.03.8):

```bash
# Topology: two LSPs on the same LS; ACL allows SSH egress only, default-deny
sudo ovn-nbctl ls-add ls-lab
sudo ovn-nbctl lsp-add ls-lab vm-h1
sudo ovn-nbctl lsp-set-addresses vm-h1 "aa:bb:cc:00:00:01 10.0.0.5"
sudo ovn-nbctl lsp-add ls-lab vm-h2
sudo ovn-nbctl lsp-set-addresses vm-h2 "aa:bb:cc:00:00:02 10.0.0.10"

# Plug interfaces on the same chassis (Mininet topology)
sudo ovs-vsctl add-port br-int tap-h1 -- set Interface tap-h1 external_ids:iface-id=vm-h1
sudo ovs-vsctl add-port br-int tap-h2 -- set Interface tap-h2 external_ids:iface-id=vm-h2
```

**Steps:**

1. **Add an `allow-related` SSH egress rule plus a default-deny:**

   ```bash
   sudo ovn-nbctl acl-add ls-lab from-lport 1000 'inport == "vm-h1" && ip4.dst == 10.0.0.10 && tcp.dst == 22' allow-related
   sudo ovn-nbctl acl-add ls-lab from-lport 1 'ip' drop
   ```

2. **Verify compile:** `sudo ovn-sbctl lflow-list ls-lab | grep ls_in_acl_eval` shows a flow at priority 2001 for the allow rule plus priority 1001 for the deny rule.
3. **Verify OF:** `sudo ovs-ofctl -O OpenFlow15 dump-flows br-int table=9 | grep -E "priority=(2001|1001)"`.
4. **Test forward path (h1 SSHs to h2)**: from the `vm-h1` namespace, run `ssh -o StrictHostKeyChecking=no 10.0.0.10 echo hello`. Observe: the connection succeeds.
5. **Observe the conntrack commit**: `sudo ovs-appctl dpctl/dump-conntrack zone=<lab_zone>` or `sudo conntrack -L | grep 10.0.0.10` shows an entry in state ESTABLISHED for the SSH 5-tuple (TCP src=ephemeral, dst=22).
6. **Test reply path (h2 cannot initiate to h1)**: from the `vm-h2` namespace, run `ssh -o ConnectTimeout=3 10.0.0.5 echo nothing`. Observe: timeout. Default-deny blocks the reverse-direction forward; ct.est does not match because h2 is not the reply.
7. **Observe the allow-rule `n_packets` increment**: `sudo ovs-ofctl -O OpenFlow15 dump-flows br-int table=9 | grep priority=2001` shows the allow rule's `n_packets` rising in step with the SSH handshakes.

**POE variants:**

- **(a) Switch the action to `allow` (stateless)**: rerun step 4; SSH hangs (the SYN goes through, but the SYN-ACK reply is dropped because no ct.est match). This demonstrates the difference between `allow` and `allow-related`.
- **(b) Raise the deny priority to 2000, above the 1000 allow**: rerun step 4; SSH fails. This demonstrates "highest priority wins".
- **(c) Add a duplicate `(direction, priority, match)` rule** with action drop and observe the non-deterministic tie-break (simulates the OVN-2024-Q2 production case in the real-world incident axis).

**Falsification check.** If the rule is set with direction `to-lport` instead of `from-lport` for the egress SSH traffic from h1 to h2, predict: the rule does not hit (wrong direction); observe: SSH hangs. Confirms that direction must match the actual pipeline stage (ingress versus egress relative to the LSP).

### Common failures and diagnostic signals

**Failure mode A, priority inversion:**

- Symptom: the specific allow rule (priority 1000) does not hit; the default-deny (set higher at 2000 by a CMS error) wins, and all traffic drops.
- Diagnostic: `ovn-nbctl acl-list <ls> | sort -k2 -n` to inspect priority order; if deny is greater than or equal to allow, the order is wrong.
- Recovery: `ovn-nbctl set ACL <deny_uuid> priority=1` to push default-deny down to the lowest priority.

**Failure mode B, stateless allow forgot the reverse:**

- Symptom: half-open connection; the SYN goes through, but the SYN-ACK times out.
- Diagnostic: `ovn-nbctl --columns=action list ACL <uuid>` shows action=allow (not allow-related); `dpctl/dump-conntrack` has no entry for that 5-tuple (because stateless does not commit).
- Recovery: switch to `allow-related` via `ovn-nbctl set ACL <uuid> action=allow-related` (always preferred unless the workload is purely stateless, such as BGP or OSPF, and needs to bypass conntrack).

**Failure mode C, duplicate priority and match (the OVN-2024-Q2 case):**

- Symptom: 5 percent intermittent traffic drop even though the allow rule looks correct; the allow rule's `n_packets` counter does not rise to the expected level.
- Diagnostic: `ovn-nbctl --columns=direction,priority,match,action list ACL | sort -k2,3` to find rows with duplicate `(direction, priority, match)`.
- Recovery: identify the wrong row and remove it via `ovn-nbctl acl-del` or `destroy ACL <uuid>`; report the reconciliation bug to the CMS team.

**Failure mode D, log does not appear:**

- Symptom: `log=true` is set but `journalctl -u ovn-controller` or syslog has no ACL log entry on rule match.
- Diagnostic: `ovn-nbctl --columns=meter list ACL <uuid>` shows meter is set but `ovn-nbctl list Meter <m>` has no corresponding row (orphan reference); or severity is empty (suppressed by default).
- Recovery: create a Meter via `ovn-nbctl meter-add audit-meter pktps 100 1000`; set `severity=warning` via `ovn-nbctl set ACL <uuid> severity=warning`.

Diagnostic mantra for ACL: **"acl-list -> lflow-list ls_in_acl_eval -> ovs-ofctl dump-flows table=9 -> conntrack zone"**, in that exact order, from NBDB declarative down to chassis runtime.

### Comparison with other systems

| Comparable system | Common ground | Key difference |
|-------------------|---------------|----------------|
| **Linux iptables filter chain** | Stateful firewall through nf_conntrack; same allow, drop, reject concepts | iptables: first-match wins by rule order in the chain; OVN ACL: highest priority wins. iptables: rule per host attached to a chain; OVN ACL: rule cluster-wide, compiled to every chassis |
| **OpenStack Neutron Security Group (legacy)** | Same abstraction: rule with direction plus protocol plus port | Neutron SG (legacy OVS-firewall agent) pushes iptables per host; OVN translates SG into an ACL row in NBDB through the `neutron-server` networking-ovn plugin |
| **Cisco IOS extended ACL** | L3 and L4 match expression plus permit or deny action | Cisco ACL: matches in ACE (Access Control Entry) order in the list, no priority; OVN ACL: priority is explicit |
| **NSX-T Distributed Firewall (DFW)** | Same per-VM stateful micro-segmentation; same Section plus Rule concept | NSX-T DFW uses a VMware proprietary policy engine; OVN ACL is open source via OVSDB. NSX-T DFW applies at the vNIC; OVN ACL applies at the Logical_Switch port (semantically equivalent) |
| **AWS Security Group** | Stateful (auto-allow reply), per-instance policy | AWS SG: implicit default-deny plus only allow rules; OVN ACL: explicit default-deny must be added manually, supports allow, deny, reject, log, and meter |
| **AWS Network ACL** | Stateless, ordered by rule number | NACL: stateless (a reverse rule is required); OVN ACL allow-related is stateful (no reverse rule needed) |
| **Kubernetes NetworkPolicy** | Same per-pod micro-segmentation; same ingress and egress plus selector concept | NetworkPolicy: a high-level abstraction through a label selector; OVN ACL: a low-level rule with a match expression. kube-ovn translates NetworkPolicy into ACL plus Port_Group plus Address_Set automatically |

Migration mapping (Neutron SG to OVN ACL): each `SecurityGroupRule` (id, direction, protocol, port_range_min/max, remote_ip_prefix) becomes one ACL row (priority fixed at 1000 for the allow rule, 1001 for the implicit hidden default-deny); `remote_ip_prefix` becomes an Address_Set or an inline `ip4.src == X.Y.Z.W/N`. Migration mapping (Kubernetes NetworkPolicy to OVN ACL): each NetworkPolicy podSelector becomes a dynamic Port_Group per label combination; each ingress or egress rule becomes one ACL row on the Port_Group.

## 13.3.Z NAT as an NBDB table: deep dive on address translation in Logical_Router

> This section goes deep on the **`NAT` table itself** (a row in the OVN Northbound database) and complements the practical material in §13.3.4 and §13.3.9 plus the pipeline introduced in §13.3.5 (gateway router). Goal: the engineer should fully understand the structure of an NBDB NAT row, the 10-column schema in branch-22.03, the lifecycle of compiling into Logical_Flow across the four stages `lr_in_dnat`, `lr_in_unsnat`, `lr_out_snat`, and `lr_out_undnat`, and how to trace from a runtime incident (floating IP unreachable, SNAT egress port exhaustion, ARP storm caused by the wrong external_mac, or stale conntrack after a rule change) back to a specific row in NBDB.

### Core concept

`NAT` is a **table that lives in the OVN Northbound database (NBDB)** describing **one address translation rule** attached to a `Logical_Router`. Each row represents a single mapping between a logical address (subnet or internal port) and an external address (the external network representative IP), with three core types: **`snat`** (egress source rewrite for an entire subnet), **`dnat`** (ingress destination rewrite, rarely used by itself because most workloads need both directions), and **`dnat_and_snat`** (a 1:1 NAT for a floating IP, translating both forward and reverse). Type: **OVSDB table**, not a root table (`isRoot=false` in `ovn-nb.ovsschema`); a NAT row exists only when referenced by a strong reference from the `Logical_Router.nat` column (cascade delete when the LR is deleted). Two key differences from Linux iptables MASQUERADE: an OVN NAT is declarative on the Logical_Router (not a per-host chain), is compiled by ovn-northd into distributed Logical_Flow, and supports both centralized gateway router mode (NAT runs on a single chassis chosen via `gateway_chassis`) and distributed gateway port (DGP) mode in which SNAT remains centralized while a `dnat_and_snat` floating IP can be distributed when `options:stateless=false`.

### History and origin

The `NAT` table appeared in OVN 2.5 (early 2016) when Justin Pettit, Russell Bryant, and Han Zhou added a complete L3 Logical_Router to OVN (before that, OVN 2.4 had only the L2 Logical_Switch). Before OVN, the Neutron L3-agent architecture used `ip netns` plus the iptables `nat` table per host (each virtual router was a netns on one network node) and ran into several problems: **single point of failure** (the router netns ran on one host; the host failing meant losing all tenant traffic), **iptables rule reload race** every time a floating IP was added, **no hairpin** when a VM in the subnet tried to reach a floating IP belonging to another VM in the same subnet (a SNAT-reverse trick was required), and **ARP responder had to be simulated** on the external bridge for every floating IP. OVN solved this by: (a) pushing the NAT rule declaratively into NBDB Raft, (b) having ovn-northd compile it into distributed Logical_Flow, and (c) having ovn-controller automatically generate the `arp_responder` flow and handle hairpin directly inside the LR pipeline. Schema evolution (verified against `ovn-nb.ovsschema` on GitHub branches): OVN 2.5 (2016) launched with six columns (`type`, `external_ip`, `external_mac`, `logical_ip`, `logical_port`, `external_ids`); OVN 2.7 added the `options` map plus the `stateless` key; OVN 20.06 added `allowed_ext_ips` and `exempted_ext_ips` for whitelisting and blacklisting external clients (proposed by Numan Siddique); OVN 21.03 added `external_port_range` for the SNAT port pool; OVN 22.03 stabilised at 10 columns; OVN 22.06 added `options:add_route` for auto-generating a static route when a `dnat_and_snat` is created; OVN 24.03 improved distributed `dnat_and_snat` floating IP via `Port_Binding.nat_addresses`.

### Position in the architecture

The table sits at the **NBDB layer**, the topmost input layer of the OVN control plane. Scope: **per-Logical_Router**. A NAT row exists only when referenced by a strong reference from `Logical_Router.nat` (a UUID set); deleting the LR cascades to deleting the attached NAT rows. A typical production OVN cluster (telco or cloud provider) has thousands of NAT rows (roughly one SNAT per tenant plus 5 to 50 floating IPs per tenant, multiplied by hundreds of tenants). Post-compile blast radius: changing one NAT row (editing `external_ip`, changing `type`, or changing `logical_port`) causes ovn-northd to recompute **only** the Logical_Flow of that LR (incremental compile from OVN 21.03 and later); ovn-controller on the chassis chosen by `gateway_chassis` (for centralized SNAT) or on every chassis hosting a Port_Binding for `logical_port` (for distributed `dnat_and_snat`) reinstalls the flow in under one second through the I-P engine. NAT is **never** applied to a `Logical_Switch`; the L2 traffic-control rule for an LS is `ACL` (the separate table covered in §13.3.X). The four LR pipeline stages where NAT compiles into flows are: `LR_IN_DNAT` (logical table 6, OF table 14 in the OVN 22.03 mapping shown in Part 13.16), `LR_IN_UNSNAT` (logical table 4, OF table 12), `LR_OUT_SNAT` (logical table 3, OF table 43), and `LR_OUT_UNDNAT` (logical table 1, OF table 41).

### Role and responsibilities

NAT carries **five roles** in the OVN architecture:

1. **Allow private-subnet VMs to reach the internet (SNAT)**: a row with `type=snat`, `logical_ip=10.0.0.0/24`, and `external_ip=203.0.113.10` rewrites the source IP of an egress packet leaving the LR to the external representative IP. OVS conntrack commits an entry so that reply traffic automatically reverses SNAT when it returns.
2. **Expose an internal VM externally (DNAT or dnat_and_snat = floating IP)**: a row with `type=dnat_and_snat`, `external_ip=203.0.113.50`, and `logical_ip=10.0.0.5` creates a 1:1 mapping between the public floating IP and the private VM; both ingress (DNAT external_ip to logical_ip) and reverse egress (SNAT logical_ip to external_ip) are rewritten.
3. **ARP responder for external_mac**: ovn-northd generates an additional Logical_Flow at stage `LR_IN_IP_INPUT` so the LR replies to ARP requests asking for `external_ip` with `external_mac` (when set) or with the MAC of the LR external port (when unset). This is mandatory for the external network to find the floating IP.
4. **Whitelist or blacklist external clients via `allowed_ext_ips` and `exempted_ext_ips`**: ovn-northd adds a `match` condition into the Logical_Flow so NAT applies only when the external source IP is in the whitelist Address_Set (or is not in the blacklist). Used for compliance or for NAT-level DDoS mitigation.
5. **Pass compile parameters to northd via `options`**: the `options` column (map) holds `stateless=true` or `false` (true means bypass conntrack, perform a static translation without tracking state, used for stateless protocols) and `add_route=true` or `false` (true means automatically generate a static route from `external_ip` back to the LR external port; OVN 22.06 and later).

### Why it exists (design motivation)

Before OVN 2.5, the Neutron L3-agent used a "router-as-netns" architecture: each tenant router was an `ip netns` on one network node, used the iptables `nat` table for SNAT and DNAT, and used `keepalived` for HA failover between two network nodes. The problems stacked up: (a) **single chassis bottleneck**: all tenant N-S traffic passed through one netns on one host, capped by that host's NIC bandwidth; (b) **slow failover of 5 to 30 seconds** because of the keepalived VRRP hello interval and the iptables rule reprogramming; (c) **no floating-IP hairpin**: a VM in the subnet could not reach the floating IP of another VM in the same subnet (DNAT was applied only at the external bridge, not when traffic did not leave the router); (d) **iptables atomic reload** was a 0.5-to-2-second stop-the-world on every floating IP add; (e) **external_mac was not configurable**: an ARP responder that needed to impersonate a different MAC required a custom script; (f) **no conntrack zone partition**: two tenants with overlapping CIDR caused conntrack collisions. OVN NAT solves this by **representing the rule once in NBDB Raft, letting northd compile it into Logical_Flow, letting every ovn-controller install OpenFlow flows on br-int, and using OVS conntrack zone partitioning per datapath**. Centralized SNAT runs on the chassis chosen via `gateway_chassis` with BFD failover under 200 ms (Part 13.11). Distributed `dnat_and_snat` allows floating-IP traffic to bypass the gateway chassis (ingress lands on the host that hosts the target VM and DNAT happens there, taking advantage of distributed bandwidth). Hairpin is handled naturally at stages `LR_IN_DNAT` and `LR_OUT_UNDNAT`. Conntrack zone partitioning is per datapath UUID (zone ID equals the `tunnel_key` of Datapath_Binding).

### Problems it solves

The table solves **five distinct technical problems**: (1) **Distributed N-S traffic without a bottleneck**: floating-IP traffic distributes across every chassis hosting the target VM, instead of going through one gateway node; (2) **Natural hairpin**: the LR pipeline handles both the forward DNAT and the reverse SNAT in the same pipeline, so a VM in the subnet reaching the floating IP of another VM in the same subnet still works correctly; (3) **HA failover under 200 ms via BFD**: gateway chassis failover uses BFD probes over the tunnel, which is tens of times faster than keepalived VRRP; (4) **Conntrack zone partitioning per datapath**: two tenants with overlapping CIDR (for example, both using `10.0.0.0/24`) do not cause conntrack collisions because each LR has its own zone; (5) **external_mac is configurable**: the `external_mac` column lets the engineer specify the MAC when the LR must impersonate another device on the external network (for example, integration with an external load balancer or to inherit a sticky MAC).

### Importance

This is a **tier-1 core table of NBDB**, on par with `Logical_Switch`, `Logical_Router`, `Load_Balancer`, and `ACL`. Reasons: 100 percent of production OVN deployments with public-facing workloads use NAT (SNAT for egress plus dnat_and_snat for floating IPs); 100 percent of Neutron OpenStack tenants with a public router on OVN generate NAT rows through the `networking-ovn` plugin; every "VM cannot reach the internet", "floating IP unreachable from external", "intermittent asymmetric traffic", or "ARP storm after creating a floating IP" incident traces back to a NAT row or its compile. An engineer who does not understand NAT cannot debug any N-S connectivity issue in a Neutron cluster, cannot understand why a floating IP is sometimes distributed and sometimes centralized, and cannot tell when to use `dnat_and_snat` versus separate `snat` plus `dnat`. Among the 320 plus keywords in REF, NAT is one of the 15 cornerstone OVN keywords.

### How it works (the lifecycle of compiling one row)

A NAT row passes through **four phases** in its lifecycle:

1. **CMS declaration**: an engineer or the CMS calls `ovn-nbctl lr-nat-add lr1 dnat_and_snat 203.0.113.50 10.0.0.5 vm-web-1 02:00:00:00:00:50`. The NBDB transaction creates a `NAT` row with `type="dnat_and_snat"`, `external_ip="203.0.113.50"`, `logical_ip="10.0.0.5"`, `logical_port="vm-web-1"`, and `external_mac="02:00:00:00:00:50"`, and inserts the row UUID into the `Logical_Router.nat` set on `lr1`. The transaction is atomic; the row is visible in under 50 ms through NBDB Raft replication.
2. **Detection, classification, and compile to Logical_Flow**: ovn-northd detects the change through an I-P engine input change. Function `build_lrouter_in_dnat_flow` (`northd/northd.c` branch-22.03 line 13067 and onward) builds the `LR_IN_DNAT` stage flow: for `dnat` or `dnat_and_snat`, it generates `match=(ip && ip4.dst == 203.0.113.50 && inport == "lr1-public")` with action `flags.loopback = 1; ct_dnat(10.0.0.5);` (DNAT from external_ip to logical_ip plus conntrack commit). Function `build_lrouter_out_snat_flow` (line 13230 and onward) builds the `LR_OUT_SNAT` stage flow: for `snat` or `dnat_and_snat`, it generates `match=(ip && ip4.src == 10.0.0.5)` with action `ct_snat(203.0.113.50);`. Function `build_lrouter_in_unsnat_flow` (line 12987 and onward) builds the `LR_IN_UNSNAT` flow for the reverse SNAT of reply traffic (match `ip4.dst == 203.0.113.50`, action `ct_snat;`). Function `build_lrouter_out_undnat_flow` (line 13154 and onward) builds the `LR_OUT_UNDNAT` flow for the reverse DNAT of reply traffic. When `external_mac` is set (only applicable to distributed `dnat_and_snat`), an additional ARP responder flow is added at `LR_IN_IP_INPUT` so the LR responds to ARP for `external_ip` with `external_mac`.
3. **Stateful supplement plus LB integration**: function `build_lrouter_nat_defrag_and_lb` (line 13581 and onward) iterates over every NAT row plus Load_Balancer row of the LR and builds a defragment flow at stage `LR_IN_DEFRAG` so packets are defragmented before NAT (necessary because conntrack does not track fragmented packets). When the LR has both NAT and Load_Balancer, function `build_lrouter_nat_flows_for_lb` (line 10005 and onward) builds an integration flow that avoids conflict (LB DNAT takes priority over NAT DNAT when both match the same destination).
4. **Flow installation on the chassis**: SBDB Raft replicates the Logical_Flow rows; ovn-controller subscribes. For **centralized SNAT** (a `type=snat` rule), only the chassis chosen in `gateway_chassis` of the LR external port installs the physical OF flow (other chassis tunnel internal Port_Binding traffic out to the gateway). For **distributed dnat_and_snat** (a `type=dnat_and_snat` rule with `logical_port` set plus the LR external port in distributed mode), every chassis with a target VM Port_Binding installs the corresponding OF flow. Flows are tagged with `cookie=<datapath_uuid_low_64>` so they can be filtered by LR.

Invariant: at any point in time, a NAT row in NBDB corresponds to **3 to 5 Logical_Flow rows** in SBDB (one forward DNAT, one forward SNAT, one reverse UNSNAT, one reverse UNDNAT, and possibly one ARP responder), and the same set of OpenFlow rules on the selected chassis (centralized or distributed). Deleting a NAT row triggers an ovn-northd recompute, the old Logical_Flows are evicted, and ovn-controller deletes the corresponding OF flows in under one second; old conntrack entries persist until their state TTL (TCP `ESTABLISHED` defaults to 5 days, which can stale and cause asymmetric traffic).

### Column classification

The 10 columns of NAT (verified against `ovn-nb.ovsschema` on branch-22.03 [GitHub raw](https://raw.githubusercontent.com/ovn-org/ovn/branch-22.03/ovn-nb.ovsschema)) split into **four groups** by semantic role:

| Group | Column | Type | Required | Notes |
|-------|--------|------|----------|-------|
| Identity | `type` | enum {dnat, snat, dnat_and_snat} | Yes | Selects the NAT semantics; determines the compile stage |
| Identity | `external_ids` | map string to string | No | CMS-tracked: `neutron:fip_id`, `neutron:router_id` |
| IP mapping | `external_ip` | string | Yes | The public or external-side IP; for SNAT this is the pool IP, for DNAT or dnat_and_snat this is the floating IP |
| IP mapping | `logical_ip` | string | Yes | The internal-side IP or CIDR; SNAT typically uses a CIDR like `10.0.0.0/24`, while DNAT or dnat_and_snat uses a single IP like `10.0.0.5` |
| MAC mapping | `external_mac` | string max=1 | No | The MAC used to answer ARP for `external_ip`; only applies to distributed `dnat_and_snat` |
| Port mapping | `logical_port` | string max=1 | No | The internal LSP; required for distributed `dnat_and_snat` to route the flow to the right chassis |
| Port pool | `external_port_range` | string | No | The port pool range for SNAT (for example, `10000-20000`); 21.03 and later |
| Filter | `allowed_ext_ips` | UUID Address_Set max=1 | No | Whitelist of external source IPs; NAT applies only when the source is in the set |
| Filter | `exempted_ext_ips` | UUID Address_Set max=1 | No | Blacklist of external source IPs; NAT is bypassed when the source is in the set |
| Parameter | `options` | map string to string | No | `stateless=true` or `false`, `add_route=true` or `false` |

NAT has **no unique index constraint** (verified schema, `"isRoot": false`, no `indexes` clause); a duplicate `(type, external_ip, logical_ip)` is schema-valid; ovn-northd compiles both into Logical_Flow rows and the behaviour is undefined (priority tie-break is arbitrary). The CMS must enforce uniqueness itself. For `dnat_and_snat`, one `external_ip` should map to exactly one `logical_ip`; if multiple VMs need to share an external IP for load balancing, use the `Load_Balancer` table (covered in §13.9.Z) instead of `NAT`.

### How an engineer masters operations

Four foundational skills the engineer needs with NAT:

1. **Read a complete row**: `sudo ovn-nbctl --columns=type,external_ip,logical_ip,external_mac,logical_port,external_port_range,allowed_ext_ips,exempted_ext_ips,options,external_ids list NAT` shows all 10 columns. Or use the shortcut `sudo ovn-nbctl lr-nat-list lr1` for a human-readable per-LR list.
2. **Verify the compile down to SBDB**: `sudo ovn-sbctl lflow-list lr1 | grep -E "lr_(in|out)_(d|s|un)?nat"` confirms there are Logical_Flows at the four corresponding stages. Looking up the match expression to confirm `external_ip` and `logical_ip` match the NBDB row verifies that the compile is correct.
3. **Verify the OF physical layer**: `sudo ovs-ofctl -O OpenFlow15 dump-flows br-int "table=14"` (in OVN 22.03 the mapping `LR_IN_DNAT` is OF table 14), or `table=43` for `LR_OUT_SNAT`, `table=12` for `LR_IN_UNSNAT`, `table=41` for `LR_OUT_UNDNAT`. The cookie equals the low 64 bits of the LR datapath UUID. Cross-link to Part 13.16 for the full mapping.
4. **Decide which skill to use when**: when the report is "VM cannot reach the internet", start with skill 1 (confirm a `type=snat` row exists with the correct `logical_ip` CIDR and `external_ip`); when the row is correct but traffic still does not reach, move to skill 2 (cross-check the Logical_Flow at `LR_OUT_SNAT`, and verify which chassis is currently active in `gateway_chassis`); when the Logical_Flow is correct but the runtime drops, move to skill 3 (inspect the OF flow on the gateway chassis and check the conntrack zone via `ovs-appctl dpctl/dump-conntrack`). When stale conntrack is suspected, cross-link to Part 9.24 §dpctl/dump-conntrack.

### Workflow (the standard usage flow)

Standard workflow for the two most common use cases, with a verification command at every step:

**Use case A, SNAT an entire subnet to the internet:**

1. **Create the LR, the external port, and the gateway chassis**: `ovn-nbctl lr-add lr1` plus `lrp-add lr1 lr1-public 02:00:00:00:00:01 203.0.113.1/24` plus `lrp-set-gateway-chassis lr1-public chassis-gw1 100`. Verify: `ovn-nbctl --columns=name,gateway_chassis list Logical_Router_Port lr1-public`.
2. **Add an SNAT rule for the subnet**: `ovn-nbctl lr-nat-add lr1 snat 203.0.113.10 10.0.0.0/24`. Verify: `ovn-nbctl lr-nat-list lr1` shows the entry.
3. **Wait for northd compile and ovn-controller flow install**: typically under one second. Verify: `ovn-sbctl lflow-list lr1 | grep "lr_out_snat"` shows a flow with match `ip4.src == 10.0.0.0/24` and action `ct_snat(203.0.113.10);`.
4. **Test egress**: from a namespace inside subnet `10.0.0.0/24` run `ping 8.8.8.8`. On the gateway chassis, `tcpdump -ni br-ex icmp` shows the packet with source IP `203.0.113.10` (already SNATed). `ovs-appctl dpctl/dump-conntrack zone=<zone_lr> | grep 8.8.8.8` shows an entry with mapping `(10.0.0.5,32768) <-> (203.0.113.10,32768)`.

**Use case B, floating IP `dnat_and_snat` 1:1:**

1. **Add the dnat_and_snat rule**: `ovn-nbctl lr-nat-add lr1 dnat_and_snat 203.0.113.50 10.0.0.5 vm-web-1 02:00:00:00:00:50`. Verify: `ovn-nbctl lr-nat-list lr1` shows the entry with both `external_mac` and `logical_port`.
2. **Enable add_route to auto-generate the static route**: `ovn-nbctl set NAT <uuid> options:add_route=true` (OVN 22.06 and later). Verify: `ovn-nbctl lr-route-list lr1` includes the route `203.0.113.50/32 -> lr1-public`.
3. **Verify the compile**: `ovn-sbctl lflow-list lr1 | grep -E "(lr_in_dnat|lr_out_snat).*203.0.113.50"` shows the DNAT stage flow (forward) plus the SNAT stage flow (reverse) both referencing external_ip `203.0.113.50`.
4. **Verify the ARP responder**: `ovn-sbctl lflow-list lr1 | grep "lr_in_ip_input.*203.0.113.50"` shows the flow that answers ARP with MAC `02:00:00:00:00:50`.
5. **Test ingress**: from a host on the external network run `ping 203.0.113.50`. On the chassis hosting `vm-web-1`, `tcpdump -ni tap-vm-web-1 icmp` shows the packet with destination IP `10.0.0.5` (already DNATed). The reply travels through `LR_OUT_UNDNAT` reverse, and the source IP becomes `203.0.113.50` again on the way out to external.

Production best practice: **always use `dnat_and_snat` for a floating IP** (do not split it into separate `dnat` plus `snat` because the hairpin will break); **always set `external_mac` explicitly** for every distributed `dnat_and_snat` (when left empty, the ARP responder uses the MAC of the LR external port and cannot be distributed); **set `external_port_range` for SNAT** when the tenant has more than 30,000 concurrent connections (the default ephemeral pool 32768 to 60999 exhausts easily). Anti-patterns: using `snat` for a 1:1 floating IP (the reverse DNAT is missing, so inbound traffic does not reach the VM); using a stand-alone `dnat` (no SNAT) for a floating IP (the reverse SNAT is missing, so the egress source is not the floating IP, which lets the client see the private VM IP); leaving two NAT rows with the same `external_ip` and different `logical_ip` (collision, undefined behaviour).

### When something goes wrong (reverse symptom map)

Seven common symptoms that point to NAT:

| Symptom | NAT column to check | Diagnostic flow |
|---------|---------------------|-----------------|
| "VM cannot reach the internet" | A `type=snat` row that matches the subnet `logical_ip` CIDR | `ovn-nbctl lr-nat-list lr1`; `ovs-appctl dpctl/dump-conntrack zone=<zone>` to see whether an entry exists |
| "Floating IP unreachable from external" | `type=dnat_and_snat` plus `external_mac` plus the ARP responder flow | `ovn-sbctl lflow-list lr1 \| grep arp.*<external_ip>`; `tcpdump -ni br-ex arp` to see whether the LR answers ARP |
| "Asymmetric intermittent traffic (5 to 10 percent drop)" | Stale conntrack after a rule change | `conntrack -L \| grep <external_ip>` to find old entries; flush via `conntrack -D -d <external_ip>` |
| "ARP storm on the external network after creating a floating IP" | `external_mac` collides with another machine's MAC | Check that `external_mac` does not collide with the MAC of another physical device on the same external L2 |
| "SNAT port exhaustion (50,000 plus concurrent connections)" | `external_port_range` not set, or range too narrow | Raise to `external_port_range="10000-65000"`, or add a second IP to the pool |
| "Hairpin: a VM in the same subnet cannot reach the floating IP" | A stand-alone `dnat` is being used instead of `dnat_and_snat` | Switch to `dnat_and_snat` so the LR pipeline performs the reverse SNAT |
| "Two-tenant CIDR overlap, conntrack collision" | Not a NAT bug; a datapath zone bug | Verify that `Datapath_Binding.tunnel_key` differs between the two LRs (zone partitioning is automatic) |

Cross-links: Part 13.1 (NBDB plus SBDB architecture overview), Part 13.2 §13.2.X (Logical_Switch references in the topology), Part 13.11 (distributed gateway router plus gateway_chassis plus BFD failover), Part 13.16 (logical pipeline table ID map: LR_IN_DNAT = stage 6 plus LR_OUT_SNAT = stage 3 plus LR_IN_UNSNAT = stage 4 plus LR_OUT_UNDNAT = stage 1), Part 9.24 (conntrack execute, the ct() action that NAT uses with the `nat()` sub-action), Part 13.9 §13.9.Z (Load_Balancer integration with NAT when both match the same destination).

### Tightly related to

Six components are tightly coupled with NAT (changing any one of them requires re-evaluating NAT):

- **`Logical_Router.nat`** (NBDB): a strong-reference column holding the NAT UUID set. Deleting the LR cascades to deleting the attached NAT rows. This is the parent that constrains the lifecycle.
- **`Logical_Router_Port` external (gateway_chassis)** (NBDB): centralized NAT runs on the chassis chosen by `gateway_chassis`. Changing `gateway_chassis` (failover or rebalance) causes NAT flows to be reinstalled on the new chassis and evicted from the old one.
- **`Address_Set`** (NBDB): a strong reference through the `allowed_ext_ips` and `exempted_ext_ips` columns. Deleting an Address_Set is forbidden while a NAT row references it (schema-enforced).
- **`Load_Balancer`** (NBDB): when both NAT and LB match the same destination IP (for example, the LB VIP equals the `external_ip` of NAT), northd builds the integration flow that prioritises LB DNAT through `build_lrouter_nat_flows_for_lb`. Changing an LB rule can unintentionally affect NAT.
- **`Logical_Flow`** (SBDB): a 1-to-N mapping. Each NAT row generates 3 to 5 Logical_Flow rows depending on type, external_mac, and add_route.
- **OVS conntrack zone**: chassis-side. ovn-controller uses the `ct(zone=<datapath_zone_id>, nat)` action; the conntrack zone is partitioned per datapath UUID, so the NAT of LR-A is independent of LR-B (a zone collision does not occur in normal usage).

Loose coupling (referenced, not cascaded): `Logical_Router_Static_Route` (a separate route table, but `options:add_route=true` on NAT auto-generates a static route), `BFD` (tied to gateway_chassis failover, indirectly affecting centralized SNAT availability).

### Real-world incident

**Production case OVN-2025-Q1 (CloudOps cluster, OSP 17.1, about 150 hypervisors, about 3,500 tenant VMs, about 800 floating IPs).** An SRE engineer reported that from 02:00 UTC about 7 percent of egress traffic from tenant `tenant-prod-shared` (subnet `10.20.0.0/22`, about 900 VMs sharing one SNAT pool IP `203.0.113.10`) was dropping intermittently. Customers complained that 1 in 15 HTTPS requests from a VM to an external API gateway timed out, with no deterministic pattern. The issue only occurred during the peak window (02:00 to 04:00 UTC, corresponding to a batch database-sync job).

Investigation:

- **Step 1**: `ovn-nbctl lr-nat-list lr-tenant-prod-shared` confirmed one row with `type=snat`, `external_ip=203.0.113.10`, `logical_ip=10.20.0.0/22`. No `external_port_range` was set.
- **Step 2**: `ovn-trace lr-tenant-prod-shared 'inport == "vm-batch-1" && ip4.src == 10.20.0.5 && ip4.dst == 198.51.100.10 && tcp.src == 50001 && tcp.dst == 443'` showed the flow path passing through `lr_out_snat` with action `ct_snat(203.0.113.10);` correctly. Trace passed.
- **Step 3**: on the gateway chassis (`chassis-gw1`), `sudo ovs-appctl dpctl/dump-conntrack zone=<lr_zone> | grep 203.0.113.10 | wc -l` returned **64,512** entries (close to the default ephemeral range of 32,768 to 60,999, about 28K ports, plus the lower range, totaling about 36K ports). Roughly 64K concurrent connections on one SNAT IP.
- **Step 4**: checked `cat /proc/sys/net/netfilter/nf_conntrack_count` and saw 1.2M (close to the threshold `nf_conntrack_max=1.3M`). conntrack had not yet overflowed, but the SNAT port pool was exhausted.
- **Step 5**: the kernel log `dmesg | grep -i "nf_nat\|port range"` showed `nf_nat_setup_info: port range exhausted` every time a new connection was dropped. This was the root cause: SNAT port exhaustion because 900 VMs times an average of about 70 connections equals 63K, exceeding the 28K-port ephemeral pool.

Root cause: a single SNAT pool IP `203.0.113.10` did not have enough ports for 900 VMs to share. The default Linux ephemeral range of 32,768 to 60,999 (about 28K ports) is the bottleneck when `external_port_range` is not set.

Fix:

- **Quick fix (immediate)**: set a wider `external_port_range`: `ovn-nbctl set NAT <snat_uuid> external_port_range="10000-65000"` (about 55K ports, double the default). After northd compiled and ovn-controller installed the flow (under 5 seconds), the kernel SNAT port pool widened. The drop rate fell below 0.1 percent within 5 minutes.
- **Long-term fix**: add two more IPs to the SNAT pool: `ovn-nbctl lr-nat-add lr-tenant-prod-shared snat 203.0.113.11 10.20.0.0/22` plus `ovn-nbctl lr-nat-add lr-tenant-prod-shared snat 203.0.113.12 10.20.0.0/22`. ovn-northd compiles three SNAT flows with hash distribution; the kernel splits 900 VMs across 3 IPs times 55K ports for a 165K connection capacity.

Cross-links: Part 19.0 (forensic case study), Part 20.7 (SNAT port exhaustion decision tree), Part 9.24 (conntrack zone limit and nf_conntrack_max tuning).

### Synthetic exercise (Lab L13.3.Z-1)

**Goal.** Hands-on for the engineer: install both an SNAT egress rule and a `dnat_and_snat` floating IP, observe the four-stage compile, and verify that the conntrack NAT entry is symmetric.

**Setup** (Mininet plus OVN on Ubuntu 22.04, OVN 22.03.8, single chassis):

```bash
# Topology: lr1 with two LRPs (lr1-internal connects to ls-internal, lr1-public connects to ls-public)
sudo ovn-nbctl ls-add ls-internal
sudo ovn-nbctl ls-add ls-public

# LRP internal side
sudo ovn-nbctl lrp-add lr1 lr1-internal 02:00:00:00:00:01 10.0.0.1/24

sudo ovn-nbctl lsp-set-addresses ls-internal-lr1 router
sudo ovn-nbctl lsp-set-options ls-internal-lr1 router-port=lr1-internal

# LRP public side plus gateway chassis (substitute the chassis hostname yourself)
sudo ovn-nbctl lrp-add lr1 lr1-public 02:00:00:00:00:02 203.0.113.1/24
sudo ovn-nbctl lrp-set-gateway-chassis lr1-public $(hostname) 100
sudo ovn-nbctl lsp-add ls-public ls-public-lr1
sudo ovn-nbctl lsp-set-type ls-public-lr1 router
sudo ovn-nbctl lsp-set-addresses ls-public-lr1 router
sudo ovn-nbctl lsp-set-options ls-public-lr1 router-port=lr1-public

# One internal VM plus one external host (Mininet namespace simulating external)
sudo ovn-nbctl lsp-add ls-internal vm-web-1
sudo ovn-nbctl lsp-set-addresses vm-web-1 "aa:bb:cc:00:00:05 10.0.0.5"
sudo ovs-vsctl add-port br-int tap-vm-web-1 -- set Interface tap-vm-web-1 external_ids:iface-id=vm-web-1
```

**Steps:**

1. **Add SNAT for subnet egress:**

   ```bash
   sudo ovn-nbctl lr-nat-add lr1 snat 203.0.113.10 10.0.0.0/24
   ```

2. **Add dnat_and_snat (floating IP) for VM `vm-web-1`:**

   ```bash
   sudo ovn-nbctl lr-nat-add lr1 dnat_and_snat 203.0.113.50 10.0.0.5 vm-web-1 02:00:00:00:00:50
   ```

3. **Verify the compile across all four stages:**

   ```bash
   sudo ovn-sbctl lflow-list lr1 | grep -E "lr_(in|out)_(d|s|un)?nat"
   ```

   Expected: at least five flows (1 ingress dnat, 1 ingress unsnat, 2 egress snat for floating IP and subnet, 1 egress undnat).

4. **Verify the OF physical layer**: `sudo ovs-ofctl -O OpenFlow15 dump-flows br-int "table=14" | grep 203.0.113.50` (DNAT) and `table=43 | grep 10.0.0.5` (the reverse SNAT for the floating IP).
5. **Test SNAT egress (subnet)**: from the `vm-web-1` namespace, run `ping -c 5 8.8.8.8` (the default route is via `10.0.0.1`). On the external bridge, `tcpdump -ni br-ex icmp` shows the source IP **`203.0.113.10`** (the subnet SNAT pool), not `203.0.113.50`. The reason: the floating-IP `lr_out_snat` priority (153) is higher than the subnet priority (33), but the floating-IP rule only matches `ip4.src == 10.0.0.5` and the egress traffic to Google uses an ephemeral source port; to be sure of the pick, add an `outport == "lr1-public"` constraint.
6. **Observe a symmetric conntrack NAT entry:**

   ```bash
   sudo ovs-appctl dpctl/dump-conntrack zone=<lr1_zone> | grep 8.8.8.8
   ```

   Expected: an entry like `tcp,orig=(src=10.0.0.5,dst=8.8.8.8,sport=50000,dport=443),reply=(src=8.8.8.8,dst=203.0.113.10,sport=443,dport=32768),...` proving that SNAT rewrites src=10.0.0.5 to src=203.0.113.10 with the ephemeral port mapping 50000 to 32768.

7. **Test floating-IP ingress**: from the simulated external namespace (attached to `ls-public`), run `ping -c 5 203.0.113.50`. On `tap-vm-web-1`, `tcpdump -ni tap-vm-web-1 icmp` shows the destination IP **`10.0.0.5`** (already DNATed). The reply travels through `LR_OUT_UNDNAT`, and the source IP becomes `203.0.113.50` again on the way back to external.
8. **Observe the ARP responder**: on the simulated external host, `arping 203.0.113.50` shows the answering MAC is **`02:00:00:00:00:50`** (the external_mac that was set), not the MAC of LRP `lr1-public` (`02:00:00:00:00:02`).

**POE variants:**

- **(a) Switch `dnat_and_snat` to a stand-alone `dnat`**: rerun step 7. The ping from external into `203.0.113.50` succeeds, but the reply source becomes `10.0.0.5` (the private IP leaks); the external client sees two different IPs and the TCP handshake fails. This demonstrates the difference between `dnat` and `dnat_and_snat`.
- **(b) Drop `external_mac`**: check `arping 203.0.113.50` and see that the answering MAC is now the LRP MAC (`02:00:00:00:00:02`) instead of the configured `02:00:00:00:00:50`. Demonstrates the role of `external_mac`.
- **(c) Set `options:stateless=true`**: rerun step 6. The conntrack zone has no entry because stateless mode bypasses conntrack. Consequence: reply traffic must be matched statically, not through state, and may sometimes drop when the reply path differs from the outbound path.
- **(d) Add a second `dnat_and_snat` row with the same `external_ip=203.0.113.50` and a different `logical_ip`**: observe undefined behaviour. ovn-northd compiles both rows, the priority tie-break is random, and traffic splits non-deterministically. This demonstrates why the CMS must guarantee a unique `external_ip` for each `dnat_and_snat`.

**Falsification check**: if `external_port_range="10000-10005"` (only six ports) is set on SNAT for a tenant with many VMs, predict: connection number 7 onward drops with `nf_nat: port range exhausted`; observe: `dmesg | grep nf_nat` shows the matching log lines. Confirms that the SNAT pool size equals (IP count times port range) and is a hard limit; a narrow range is the cause of the OVN-2025-Q1 production-case symptom in the real-world incident axis.

### Common failures and diagnostic signals

**Failure mode A, SNAT port exhaustion:**

- Symptom: 5 to 15 percent intermittent egress drops during peak hours; `dmesg | grep nf_nat` shows `port range exhausted`.
- Diagnostic: `ovs-appctl dpctl/dump-conntrack zone=<lr_zone> | grep <external_ip> | wc -l` is approximately the size of the ephemeral pool; `ovn-nbctl --columns=external_port_range list NAT <uuid>` shows it is empty.
- Recovery: set `external_port_range="10000-65000"` (widen the pool) or add a second IP into the SNAT pool by inserting a second NAT row with the same `logical_ip` CIDR.

**Failure mode B, floating IP unreachable (ARP not answering):**

- Symptom: from the external network, ping to the floating IP times out; `arping <floating_ip>` returns no reply.
- Diagnostic: `ovn-sbctl lflow-list lr1 | grep -E "lr_in_ip_input.*<floating_ip>"` to see whether the ARP responder flow exists; `ovn-nbctl --columns=external_mac list NAT <uuid>` to see whether `external_mac` is set.
- Recovery: when `external_mac` is empty and the LRP external port has no matching MAC, set `ovn-nbctl set NAT <uuid> external_mac="02:..."` or use `dnat_and_snat` instead of a stand-alone `dnat`.

**Failure mode C, asymmetric traffic from stale conntrack:**

- Symptom: after a NAT rule change (for example, changing the floating-IP `external_ip` from A to B), 5 to 10 percent of in-flight connections drop or get routed back to the wrong chassis.
- Diagnostic: `conntrack -L | grep <old_external_ip>` shows the old entry still present, with the default 5-day TCP ESTABLISHED TTL.
- Recovery: flush the conntrack entries related to the old rule via `conntrack -D -d <old_external_ip>` or `ovs-appctl dpctl/flush-conntrack zone=<lr_zone>` (carefully: this flushes the entire zone and affects active connections).

**Failure mode D, two NAT rows with the same external_ip but different logical_ip (collision):**

- Symptom: traffic into the floating IP sometimes reaches VM A and sometimes reaches VM B, non-deterministically.
- Diagnostic: `ovn-nbctl --columns=type,external_ip,logical_ip list NAT | sort -k2,3` to find rows duplicating `(type, external_ip)` with different `logical_ip`.
- Recovery: identify the wrong row and remove it via `ovn-nbctl lr-nat-del lr1 dnat_and_snat <external_ip>` (which removes every NAT row with that external_ip), then re-add only the desired row.

Diagnostic mantra for NAT: **"lr-nat-list -> lflow-list across 4 stages -> ovs-ofctl dump-flows table=(14,12,43,41) -> conntrack zone"**, in that exact order, from NBDB declarative down to chassis runtime.

## 13.3.W Five companion NBDB tables to ACL, LB, and NAT, deep dive (Address_Set, Port_Group, Logical_Router_Static_Route, Logical_Router_Policy, Forwarding_Group)

> This section goes deep on five NBDB tables in the same family as ACL, LB, and NAT, each with its own full 20-axis treatment (concept through comparison with other systems). The first four (`Address_Set`, `Port_Group`, `Logical_Router_Static_Route`, `Logical_Router_Policy`) appear directly in `match` expressions or LR routing logic; the fifth (`Forwarding_Group`) is the active-active gateway pattern that replaces Load_Balancer for hairpin gateway HA.

### 13.3.W.1 `Address_Set`, a named IP set used in match expressions

#### Core concept

`Address_Set` is a **table that lives in the OVN Northbound database (NBDB)** describing a **named collection of IP addresses** (IPv4, IPv6, or CIDR blocks), referenced in the `match` column of ACL or Logical_Router_Policy through the syntax `$<name>`.
Each row represents one logical set, for example `vm-prod` containing the 100 IPs of the production-tier VMs.
Type: **OVSDB root table** (`isRoot=true` in `ovn-nb.ovsschema`); the row exists independently, can be referenced weakly from many ACL or LRP rows simultaneously, and is not cascade-deleted when a parent is removed.
There is a **unique index** on the `name` column (verified schema on branch-22.03: `"indexes": [["name"]]`), so two rows cannot share a name.
Key difference from inline IP literals in `match`: one ACL with `ip4.src == $vm-prod` replaces 100 separate per-IP ACL rows, cutting NBDB rows by 99 percent and reducing OpenFlow rules through conjunction compression.

#### History and origin

`Address_Set` appeared in OVN 2.6 (2016) when Russell Bryant and Justin Pettit refactored the `match` expression engine to support named sets in addition to inline IP literals. Before that, every Neutron security-group rule with a `remote_ip_prefix` of a single IP produced one ACL row; 100 source IPs equalled 100 ACL rows and bloated NBDB. OVN 2.6 added Address_Set plus the `$name` syntax in `match`; OVN 2.10 added IPv6 entries; OVN 21.06 added MAC-only sets (the `addresses` column accepts the MAC format `aa:bb:cc:dd:ee:ff`); OVN 22.03 stabilised at three columns (`name`, `addresses`, `external_ids`); OVN 24.03 improved northd incremental compile for Address_Set changes (only the referencing ACLs or LRPs recompile, not the entire LS or LR).

#### Position in the architecture

The table sits at the **NBDB input layer**.
It is a **root table**, so it has no strong-reference parent; it lives independently, and deleting a Logical_Switch does not cascade to deleting an Address_Set referenced from its ACLs.
Scope: **cluster-wide**; every ACL or LRP on every LS or LR can reference the same Address_Set through `$name`.
Address_Set never appears in SBDB; ovn-northd resolves `$<name>` at NBDB-to-SBDB compile time and inlines the disjunction directly into the Logical_Flow `match` (for example `$vm-prod` becomes `(ip4.src == 10.0.0.5 || ip4.src == 10.0.0.6 || ...)`).
After resolve, conjunction compression reduces the OpenFlow rule count.

#### Role and responsibilities

`Address_Set` carries three roles:
(1) **Abstract a list of IPs** into a single reusable named entity that is easy to maintain.
(2) **Cut the ACL row count** in NBDB by a factor of N (the number of IPs in the set).
(3) **Trigger conjunction compression** during the SBDB-to-OF translation: northd detects that `match` references `$<addr_set>` in a cross product with Port_Group and emits a `conjunction()` action, reducing OpenFlow rules from M times N to M plus N plus 1.

#### Why it exists (design motivation)

Before OVN 2.6, the Neutron security-group architecture mapped each `SecurityGroupRule.remote_ip_prefix` to one ACL row. When a rule used `RemoteSecurityGroup` (allow traffic from every VM in another SG), Neutron resolved the IP list at runtime and pushed one ACL row per IP. 100 referenced VMs equalled 100 ACL rules, re-pushed every time a VM was added or removed. The pressure on NBDB transactions, northd recompile, and conntrack churn was significant. OVN 2.6 designed Address_Set so the CMS could push **one Address_Set row plus one ACL** instead of **100 ACL rows**: adding or removing an IP only updates the `addresses` column on Address_Set, the ACL itself does not change, and northd recompiles only the Logical_Flow rows containing `$name`. The scale improvement is dramatic: kube-ovn translates a Kubernetes NetworkPolicy with a `podSelector` matching 5,000 pods into one 5,000-entry Address_Set plus one ACL row instead of 5,000 ACL rows.

#### Problems it solves

The table addresses four technical problems: (1) **NBDB transaction churn** when the IP list changes (only one row updates instead of N rows); (2) **ACL row explosion** for "from group X, to group Y" rules: Cartesian becomes M times N reduced to M plus N plus 1; (3) **OpenFlow rule explosion** after compile: conjunction compression depends on Address_Set plus Port_Group references; (4) **CMS-to-NBDB synchronisation**: adding or removing a VM only pushes one update into Address_Set; the CMS does not need to manage 100 ACL rows directly.

#### Importance

This is a **tier-2, medium** table of NBDB, required for any production deployment with more than 50 VMs. Reasons: 100 percent of production-scale OpenStack or Kubernetes deployments use Address_Set through the networking-ovn or kube-ovn plugin; every tenant with a security group referencing another group generates an Address_Set; every NetworkPolicy with a `podSelector` generates a dynamic Address_Set. An engineer who does not understand Address_Set cannot debug "the ACL does not match the IP of a newly added VM" (the Address_Set has not yet been updated) and cannot tell apart "ACL push from CMS" versus "Address_Set push from CMS" in the NBDB audit log.

#### How it works

An Address_Set row passes through four phases: (a) **CMS or engineer declaration**: `ovn-nbctl create Address_Set name=vm-prod addresses='["10.0.0.5","10.0.0.6","10.0.0.7"]'` issues an NBDB transaction that inserts a new row. (b) **Reference inside an ACL**: `ovn-nbctl acl-add ls-tenant1 from-lport 1000 'ip4.src == $vm-prod && tcp.dst == 5432' allow-related` makes the ACL `match` column contain the string `$vm-prod`. (c) **northd compile**: function `build_addrset_lflow` (or its equivalent in `northd/northd.c`) iterates over Logical_Flow rows containing `$<name>` and inlines the disjunction `(ip4.src == 10.0.0.5 || ip4.src == 10.0.0.6 || ip4.src == 10.0.0.7)`. (d) **Conjunction compression and flow installation**: when the disjunction has more than one entry combined with a Port_Group, northd emits a `conjunction(N, K/M)` action; ovn-controller installs the OpenFlow rules on br-int.

#### Column classification

Schema with three columns (verified `ovn-nb.ovsschema` branch-22.03): `name` (string, with a unique index, required), `addresses` (set of strings, min=0, max=unlimited, each entry an IP, CIDR, or MAC per OVN expression syntax), `external_ids` (map string to string for CMS metadata such as `neutron:security_group_id`).

#### How an engineer masters operations

Four skills: (1) **Create and verify an Address_Set**: `ovn-nbctl create Address_Set name=vm-prod addresses='["10.0.0.5","10.0.0.6"]'`, then `ovn-nbctl list Address_Set vm-prod`; (2) **Inline-update to add an IP**: `ovn-nbctl add Address_Set vm-prod addresses 10.0.0.7` (rather than replacing the entire set); (3) **Reference in an ACL match**: use the `$` prefix in the match expression; (4) **Verify the resolve after compile**: `ovn-sbctl lflow-list ls-tenant1 | grep "ip4.src"` shows the disjunction expanded inline.

#### Workflow (the standard usage flow)

(1) Create the Address_Set with an initial name and addresses; (2) Reference it from an ACL or LRP through `$name`; (3) Update the Address_Set with `add` or `remove` instead of recreating it (avoid a large transaction); (4) Verify the Logical_Flow in SBDB resolves correctly; (5) Best practice: use a clear naming convention (`vm-<role>-<env>`, for example `vm-web-prod`) to make audit easier.

#### When something goes wrong

Common symptoms: "ACL match `$vm-prod` does not hit for a newly added IP" - check that the Address_Set already contains the IP via `ovn-nbctl list Address_Set vm-prod`. "ACL push succeeds but northd compile reports `unknown address set`" - the Address_Set does not yet exist or the name is a typo. "Performance degrades after the Address_Set exceeds 10K entries" - consider splitting it into multiple Address_Sets per tier or per subnet.

#### Version differences

OVN 2.6: launched with three columns; OVN 2.10: IPv6 entries supported; OVN 21.06: MAC-only set supported (for `eth.src` and `eth.dst` matches); OVN 22.03: stabilised at three columns; OVN 24.03: incremental compile recompiles only the referencing ACLs or LRPs, not the entire LS or LR.

#### How to observe and verify

`ovn-nbctl list Address_Set` for a full snapshot; `ovn-nbctl list Address_Set vm-prod` for detail.
Cross-check the compile output: `ovn-sbctl lflow-list ls-tenant1 | grep -E "(10\\.0\\.0\\.5|10\\.0\\.0\\.6)"` shows that the IPs were expanded correctly.
Verify size: `ovn-nbctl --columns=name,addresses list Address_Set | awk '/addresses/ {print NF}'` counts IPs per set, with a warning above 5K.
external_ids tracking: `ovn-nbctl --columns=external_ids list Address_Set | grep neutron:security_group_id` traces the CMS that created which set.

#### Source code references

Verified upstream OVN branch-22.03: schema [`ovn-nb.ovsschema`](https://raw.githubusercontent.com/ovn-org/ovn/branch-22.03/ovn-nb.ovsschema) declares the `Address_Set` table with three columns, `"isRoot": true`, and `"indexes": [["name"]]`.
The compile resolution lives in `northd/northd.c` (function-name anchor; see the `expr_addr_sets_*` helpers in `lib/expr.c` of the upstream OVS repository, which OVN reuses through `controller/lflow.c`).
Address_Set has no dedicated compile function; the engine resolves `$<name>` directly when parsing the ACL or LRP `match` expression through `expr_parse_string` plus `expr_macros_get` in the OVS expression library.
When an Address_Set change is committed, the ovn-northd I-P engine input change `northd_input_nbrec_address_set` triggers a recompile of the referencing Logical_Flow rows.

#### Real-world incident

Production case 2025: an OVN-Kubernetes cluster with about 3,000 pods and a NetworkPolicy with a `podSelector` matching about 500 `app`-tier pods. After the pod count scaled up from 500 to 800, the ACL match `$as-app` did not hit for the new pods. Investigation: the ovnkube-master Address_Set push had about a 30-second lag (rate-limited). Fix: raise the `ovnkube-master` rate limit, and split the Address_Set per namespace to reduce the size of each set.

#### Synthetic exercise

```bash
sudo ovn-nbctl create Address_Set name=trusted-clients addresses='["10.0.0.5","10.0.0.6","192.168.1.0/24"]'
sudo ovn-nbctl acl-add ls-tenant1 from-lport 1000 'ip4.src == $trusted-clients && tcp.dst == 22' allow-related
sudo ovn-nbctl add Address_Set trusted-clients addresses 10.0.0.7
sudo ovn-sbctl lflow-list ls-tenant1 | grep "ip4.src"
```

Observe: the Logical_Flow contains the disjunction `(ip4.src == 10.0.0.5 || ip4.src == 10.0.0.6 || ip4.src == 192.168.1.0/24 || ip4.src == 10.0.0.7)`.

#### Common failures and diagnostic signals

(a) Reference name typo (`$vm_prod` versus `$vm-prod`) leads to a northd error log; (b) Mixed IPv4 and IPv6 entries in the same set with a match using `ip4.src == $set` causes the IPv6 entries to be skipped silently; (c) A set exceeding 10K entries makes northd compile slow and causes br-int OF rule explosion (split the set or use Port_Group when the entities are LSPs).

#### Comparison with other systems

Linux `ipset` is comparable (a named IP set for iptables matching); Cisco IOS `object-group network` has the same concept; AWS Security Group `prefix-list` is similar (a named CIDR set); NSX-T `IP Set` maps directly.

### 13.3.W.2 `Port_Group`, a named LSP set used for shared ACL application

#### Core concept

`Port_Group` is a **table that lives in OVN NBDB** describing a **named collection of Logical_Switch_Port (LSP)** entries, attached to a list of ACLs that apply to every LSP in the group.
Each row represents one logical group, for example `pg-web` containing the LSPs of every web-tier VM.
Type: **OVSDB root table** (`isRoot=true`); `ports` is a weak reference to `Logical_Switch_Port` (deleting an LSP does not cascade-delete the Port_Group); `acls` is a strong reference to `ACL` (deleting the Port_Group cascade-deletes the attached ACLs).
Unique index on `name`.
Crucial difference from "ACL per LSP": one ACL attached to pg-web applies to 100 LSPs instead of 100 separate ACLs, cutting NBDB rows by 99 percent and unlocking conjunction compression after compile.

#### History and origin

`Port_Group` appeared in OVN 2.10 (2018) when Han Zhou and Numan Siddique proposed it after Address_Set proved insufficient (Address_Set holds IPs only and cannot reference LSPs to match `inport` or `outport`).
Before that, Neutron security-group micro-segmentation used Address_Set for `ip4.src` and `ip4.dst` plus per-LSP ACLs for `inport` and `outport`, which still exploded on the LSP axis.
Port_Group fills that gap directly: one ACL with `inport == @pg-web && outport == @pg-app` replaces 100 times 100 = 10K separate ACL rows.
OVN 21.06 added implicit IPv6 support (Port_Group inherits the address family from the LSP); OVN 22.03 stabilised the `external_ids` column for CMS metadata.

#### Position in the architecture

The table sits at the **NBDB input layer**.
It is a **root table** with no strong-reference parent.
Scope: **cluster-wide**; one LSP can be a member of multiple Port_Groups simultaneously (1-to-N).
ovn-northd resolves `@<name>` in the match expression at compile time and replaces it with the disjunction `(inport == "lsp1" || inport == "lsp2" || ...)`.
After resolve, an Address_Set plus Port_Group cross product triggers conjunction compression.

#### Role and responsibilities

Port_Group carries three roles:
(1) **Abstract a tier of the topology** into a named entity reusable by ACL.
(2) **Foundation for ACL scaling**, cutting row count by a factor of N (the LSP count).
(3) **Trigger conjunction compression** at the OpenFlow level when combined with an Address_Set or another Port_Group.

#### Why it exists (design motivation)

Before OVN 2.10, Address_Set solved the IP-set problem but could not match `inport` or `outport` (a port is an LSP, not an IP). The consequence: a three-tier 1,000-VM micro-segmentation produced tens of thousands of ACL rows even with Address_Set. OVN 2.10 designed Port_Group to let `match` reference an LSP set directly through `@name`, cutting ACL rows by 99 percent at large scale. The OVN-Kubernetes CNIs (kube-ovn, ovn-kubernetes) build on Port_Group to translate the Kubernetes NetworkPolicy `podSelector` into OVN ACL.

#### Problems it solves

(1) **NBDB ACL row explosion** at scales beyond 100 LSPs; (2) **OpenFlow rule explosion** after compile (conjunction compression only triggers when the match has at least two cross products of sets or groups); (3) **CMS sync efficiency**: adding or removing a VM from a tier only updates the Port_Group `ports` column without adding ACL rows.

#### Importance

This is a **tier-2, medium** table of NBDB, required for any production deployment with more than 100 LSPs. It is the foundation of Kubernetes NetworkPolicy translation to OVN ACL. An engineer who does not understand Port_Group cannot tell apart "ACL applied per LS" versus "ACL applied per Port_Group"; the two behave differently in scope (LS means every LSP on the LS, while Port_Group means only the LSPs in the group, possibly across multiple LSes).

#### How it works

(a) `ovn-nbctl pg-add pg-web web-vm-1 web-vm-2` creates a Port_Group row and inserts the LSP UUIDs into the `ports` column. (b) `ovn-nbctl acl-add pg-web from-lport 1000 'inport == @pg-web && tcp.dst == 80' allow-related` creates an ACL row and inserts its UUID into the `acls` column of the Port_Group. (c) northd resolves `@pg-web` into the disjunction `(inport == "web-vm-1" || inport == "web-vm-2")` inside the Logical_Flow. (d) ovn-controller installs the OpenFlow rules on every chassis hosting a Port_Binding of an LSP in the group.

#### Column classification

Schema with four columns (verified branch-22.03): `name` (string, unique index), `ports` (set of UUIDs, weak reference to Logical_Switch_Port), `acls` (set of UUIDs, strong reference to ACL), `external_ids` (map). There is no priority, match, or action column; Port_Group is only a container.

#### How an engineer masters operations

(1) Create the Port_Group: `ovn-nbctl pg-add pg-web web-vm-1 web-vm-2`; (2) Add an LSP to the group: `ovn-nbctl pg-set-ports pg-web web-vm-1 web-vm-2 web-vm-3` (replace) or `ovn-nbctl add Port_Group pg-web ports <uuid>` (incremental); (3) Apply an ACL: `ovn-nbctl acl-add pg-web from-lport 1000 'match' allow-related`; (4) Verify: `ovn-nbctl --columns=name,ports,acls list Port_Group pg-web`.

#### Workflow (the standard usage flow)

(1) Plan the tier topology (for example web, app, db); (2) Create a Port_Group per tier; (3) Add LSPs into the right group when creating a VM (the CMS or the engineer); (4) Apply ACLs to the group (allow rules plus default-deny); (5) Verify the Logical_Flow expands correctly and the `n_packets` counter rises when traffic matches.

#### When something goes wrong

"ACL `outport == @pg-app` does not hit even though the target VM has a Port_Binding" - check whether the LSP is in the group via `ovn-nbctl --columns=ports list Port_Group pg-app`. "A newly scaled-up VM has no ACL applied" - the CMS has not yet added the LSP into the group. "northd compile is slow" - the group is too large (more than 5,000 LSPs) and should be split.

#### Version differences

OVN 2.10: launched; OVN 21.06: implicit IPv6 (Port_Group inherits the address family from the LSP); OVN 22.03: stabilised `external_ids`; OVN 24.03: incremental compile for Port_Group changes.

#### How to observe and verify

`ovn-nbctl list Port_Group pg-web` for detail; `ovn-sbctl lflow-list ls-tenant1 | grep "inport"` shows the expanded disjunction; `ovs-ofctl -O OpenFlow15 dump-flows br-int | grep conjunction` confirms that conjunction compression is active.

#### Source code references

Schema [`ovn-nb.ovsschema` on branch-22.03](https://raw.githubusercontent.com/ovn-org/ovn/branch-22.03/ovn-nb.ovsschema) declares the `Port_Group` table with four columns, `isRoot=true`, and a unique `name` index. The `@<name>` resolution lives in [`controller/lflow.c`](https://github.com/ovn-org/ovn/blob/branch-22.03/controller/lflow.c) through the expression engine in `lib/expr.c` (function anchor: see `expr_port_groups_*`).

#### Real-world incident

Production case 2024: an OVN-Kubernetes cluster with 2,000 pods and 50 NetworkPolicy rules. Before Port_Group: ovnkube-master pushed 100K ACL rows, NBDB Raft replicated slowly, and latency was 10 to 30 seconds. After refactoring to Port_Group plus Address_Set: 50 ACL rows plus 50 Port_Groups plus 100 Address_Sets, NBDB Raft replicated in under one second, and policy-enforcement latency dropped 30-fold.

#### Synthetic exercise

```bash
sudo ovn-nbctl pg-add pg-web web-vm-1 web-vm-2
sudo ovn-nbctl pg-add pg-app app-vm-1
sudo ovn-nbctl acl-add pg-app from-lport 1000 'inport == @pg-web && outport == @pg-app && tcp.dst == 8080' allow-related
sudo ovn-nbctl acl-add pg-app from-lport 1 'ip' drop
sudo ovn-sbctl lflow-list ls-tenant1 | grep -E "ls_in_acl|conjunction"
```

Observe that the conjunction action is generated for the `@pg-web` cross `@pg-app` cross product.

#### Common failures and diagnostic signals

(a) Forgetting to add an LSP into the Port_Group when scaling up a VM means the ACL does not apply to the new VM; (b) Referencing a non-existent Port_Group in `match` causes a northd compile error; (c) A group with more than 5K LSPs should be split per subnet or tier to keep northd compile time reasonable.

#### Comparison with other systems

Cisco `object-group port` has the same grouping concept; NSX-T `Security Group` maps directly to Port_Group plus ACL; Kubernetes `NetworkPolicy.podSelector` is translated by kube-ovn into a dynamic Port_Group. Linux has no native equivalent (iptables has no LSP-grouping concept; every rule is per host attached to a chain).

### 13.3.W.3 `Logical_Router_Static_Route`, the static-route table for Logical_Router

#### Core concept

`Logical_Router_Static_Route` (LRSR) is a **table that lives in OVN NBDB** describing a **static route** attached to a `Logical_Router` (LR).
Each row declares: `ip_prefix` (the destination, IPv4 or IPv6 CIDR), `nexthop` (the next-hop IP), `output_port` (the outgoing LRP, optional), `policy` (`src-ip` or `dst-ip` to choose the match direction), `bfd` (the UUID of a BFD session attached for liveness checking), and `options` (a map for fine-tuned behaviour such as `ecmp_symmetric_reply`).
Type: **OVSDB non-root table** (`isRoot=false`), with a strong reference from `Logical_Router.static_routes`.
Difference from a dynamic route (which a protocol like BGP or OSPF negotiates): the static route is declarative and has no protocol negotiation.
ECMP works through duplicate `ip_prefix` rows in the same `route_table` with different `nexthop`s; northd compiles them into a `group=...,type=select` flow at stage `LR_IN_IP_ROUTING_ECMP`.

#### History and origin

The table appeared in OVN 2.5 (2016) together with Logical_Router. The original schema had only four columns (`ip_prefix`, `nexthop`, `output_port`, `external_ids`). OVN 2.10 added `policy` (enum `src-ip` or `dst-ip`); OVN 20.06 added the `bfd` reference plus ECMP support through duplicate rows; OVN 21.06 added the `options` map (key `ecmp_symmetric_reply`); OVN 22.06 added `route_table` for VRF (multiple route tables per LR); OVN 23.09 added `options:origin` to mark a route as imported from BGP (a DPU-deployment feature, out of scope for this plan).

#### Position in the architecture

The table sits at the **NBDB input layer**, scope **per-Logical_Router**. Strong reference from `Logical_Router.static_routes` (a UUID set). One LR can have hundreds of static routes (default route plus per-subnet routes plus ECMP duplicates). It compiles into Logical_Flow at stage `LR_IN_IP_ROUTING` (logical table 9 in OVN 22.03) plus `LR_IN_IP_ROUTING_ECMP` (table 10) for ECMP. Never applied to a Logical_Switch.

#### Role and responsibilities

Four roles: (1) **Route IP traffic that does not match a directly connected subnet**: a route `0.0.0.0/0` to the external gateway for internet egress; (2) **Declare VRF through `route_table`**: partition routes per tenant or VRF (OVN 22.06 and later); (3) **ECMP load-balance**: two rows with the same `ip_prefix` and different `nexthop` produce a group select with a hash; (4) **BFD-tracked liveness**: attach a `bfd` UUID and the route is removed from the RIB automatically when BFD detects the link as down.

#### Why it exists (design motivation)

Before OVN 2.5, Neutron L3-agent used `ip route add` inside the router netns. Problems: failover depended on keepalived VRRP, taking 5 to 30 seconds; there was no native ECMP concept; BFD had to run outside the netns and the state was synced into the kernel route table manually and could not integrate with the OVN conntrack zone. OVN LRSR was designed declarative in NBDB; ovn-northd compiles into distributed OpenFlow; BFD integrates natively through a separate BFD table; ECMP is achieved through duplicate rows plus group select.

#### Problems it solves

(1) **Cluster-wide declarative static routes**, not dependent on per-host `ip route`; (2) **ECMP with five-tuple hash** for bandwidth load balancing; (3) **BFD failover under 200 ms** through the BFD reference table; (4) **VRF separation** through `route_table` for multi-tenant routing isolation; (5) **Symmetric reply** through `options:ecmp_symmetric_reply=true` ensures the reply selects the same nexthop as the forward (avoiding stateful-firewall asymmetric drops).

#### Importance

This is a **tier-2, medium** table of NBDB, required for any LR with a default route or a connection to a non-directly-connected subnet. 100 percent of deployments with internet egress have at least one default-route row. Production clusters with multi-AZ topologies use ECMP duplicate routes for redundancy.

#### How it works

(a) `ovn-nbctl lr-route-add lr1 0.0.0.0/0 192.168.1.1` issues an NBDB transaction that creates an LRSR row and inserts the UUID into `Logical_Router.static_routes` on lr1. (b) northd `build_static_route_flows` (function-name anchor in `northd/northd.c`) iterates over routes and compiles them into Logical_Flow at stage `LR_IN_IP_ROUTING` with `match=(ip4.dst == 0.0.0.0/0)` and `action=(ip.ttl--; reg0 = 192.168.1.1; reg9[2] = 0; ...; next;)`. (c) ECMP detection: when there are at least two rows with the same `ip_prefix` and the same `route_table`, northd builds a group select with a five-tuple hash. (d) BFD: when the `bfd` column is set, ovn-northd adds a condition to the match based on `BFD.status=up` (the route is skipped when BFD is down).

#### Column classification

Schema with eight columns (verified branch-22.03): `route_table` (string, default `""` empty for the default VRF, 22.06 and later), `ip_prefix` (string, required, IPv4 or IPv6 CIDR), `policy` (enum src-ip or dst-ip, optional, default dst-ip), `nexthop` (string, required), `output_port` (string, optional, the LRP name), `bfd` (UUID weak reference to BFD, optional), `options` (map of options), `external_ids` (map of metadata).

#### How an engineer masters operations

(1) Add a default route: `ovn-nbctl lr-route-add lr1 0.0.0.0/0 192.168.1.1`; (2) Add a per-subnet route with a specific output_port: `ovn-nbctl lr-route-add lr1 10.50.0.0/16 10.40.0.1 lrp-to-net-50`; (3) Add ECMP: insert two rows with the same prefix and different nexthop; (4) Verify: `ovn-nbctl lr-route-list lr1` shows the list plus the `ecmp` marker on duplicates; (5) Set BFD: `ovn-nbctl set Logical_Router_Static_Route <uuid> bfd=<bfd_uuid>`.

#### Workflow (the standard usage flow)

(1) Plan the routing topology (default route plus per-VRF routes plus ECMP); (2) Add routes per LR; (3) (Optional) Set policy `src-ip` for a source-routing tenant; (4) (Optional) Attach a BFD session for liveness; (5) Verify the compile down to SBDB; (6) Ping from an internal VM to a destination in the route and observe via tcpdump on the LRP next-hop chassis.

#### When something goes wrong

"VM ping to the internet times out" - check `ovn-nbctl lr-route-list lr1` for a default route and whether the nexthop is reachable (`ovn-nbctl list BFD` to see status). "ECMP does not balance evenly" - check whether the five-tuple hash is diverse (the same client always landing on the same backend is expected); consider `selection_fields`. "Route to a directly connected subnet but the LRSR row was added anyway" - conflict; the directly connected route always wins, and the LRSR row is redundant.

#### Version differences

OVN 2.5: four basic columns; OVN 2.10: added `policy`; OVN 20.06: added `bfd` and ECMP duplicate rows; OVN 21.06: added `options:ecmp_symmetric_reply`; OVN 22.06: added `route_table` for VRF; OVN 23.09: added `options:origin`.

#### How to observe and verify

`ovn-nbctl lr-route-list lr1` for a human-readable snapshot; `ovn-nbctl --columns=ip_prefix,nexthop,output_port,bfd,options,policy list Logical_Router_Static_Route` for the raw NBDB; `ovn-sbctl lflow-list lr1 | grep -E "lr_in_ip_routing"` to see the compile output; `ovs-ofctl -O OpenFlow15 dump-groups br-int` to see the ECMP group select.

#### Source code references

Schema [`ovn-nb.ovsschema` on branch-22.03](https://raw.githubusercontent.com/ovn-org/ovn/branch-22.03/ovn-nb.ovsschema) declares the eight-column non-root table. The compile lives in [`northd/northd.c`](https://github.com/ovn-org/ovn/blob/branch-22.03/northd/northd.c); function-name anchor `build_static_route_flow` (or `build_static_route_flows_for_lrouter` depending on refactor) iterates routes and emits Logical_Flow at stage `LR_IN_IP_ROUTING`. ECMP is built through the `ecmp_groups` handling code with a group_id allocated at runtime.

#### Real-world incident

Production case 2024: an OSP cluster with two datacentre sites (site-A and site-B) and an ECMP route from a tenant LR to a shared-service network through two transit LRPs (one per site). After site-B's link went down, BFD detected the failure in 200 ms and the ECMP route removed the site-B member. However, some TCP connections that had started earlier still pinned to site-B (conntrack pinning), and the result was about five minutes of intermittent drops until conntrack expired. Fix: enable `options:ecmp_symmetric_reply=true` so the reply follows the forward path; flush conntrack on a route change.

#### Synthetic exercise

```bash
sudo ovn-nbctl lr-route-add lr1 0.0.0.0/0 192.168.1.1
sudo ovn-nbctl lr-route-add lr1 10.100.0.0/16 10.40.0.1
sudo ovn-nbctl lr-route-add lr1 10.100.0.0/16 10.40.0.2
sudo ovn-nbctl lr-route-list lr1
sudo ovn-sbctl lflow-list lr1 | grep "lr_in_ip_routing" | head -10
sudo ovs-ofctl -O OpenFlow15 dump-groups br-int | head -10
```

Observe: the two `10.100.0.0/16` routes form an ECMP group; `lr-route-list` marks them as `ecmp`.

#### Common failures and diagnostic signals

(a) A default route whose nexthop is not in any directly connected subnet causes a northd warning and the flow is not installed; (b) ECMP duplicate routes without BFD: when a nexthop dies, 50 percent of traffic drops until the operator manually removes the row; (c) `policy=src-ip` not matching as expected: inspect the `match` expression compile output; source routing has lower priority than dst routing unless the priority is adjusted.

#### Comparison with other systems

Linux `ip route add` has the same concept; Cisco IOS `ip route` has the same syntax (network mask via-gateway); the FRR static module is comparable; AWS VPC Route Table maps similarly (Destination CIDR to Target). Key difference: OVN LRSR is declarative cluster-wide, ovn-northd compiles and pushes it to every chassis through OpenFlow, and it does not depend on a per-host kernel route table.

### 13.3.W.4 `Logical_Router_Policy`, policy-based routing on the LR

#### Core concept

`Logical_Router_Policy` (LRP, not to be confused with Logical_Router_Port, which shares the abbreviation) is a **table that lives in OVN NBDB** describing a **policy-based routing (PBR) rule** attached to a `Logical_Router`.
Unlike LRSR (which only matches the destination IP), LRP allows matching a **microflow expression** (similar to an ACL match), including `ip4.src`, `ip4.dst`, `tcp.dst`, `inport`, `ct.est`, and so on.
It applies one of three actions: `allow` (let through, route normally), `drop` (block), `reroute` (change the nexthop to a different IP).
Type: **OVSDB non-root table**, with a strong reference from `Logical_Router.policies`.
Priority `0..32767`, highest wins.
Sits **before** static route in the pipeline (stage `LR_IN_POLICY` table 7, before `LR_IN_IP_ROUTING` table 9 in OVN 22.03).

#### History and origin

LRP appeared in OVN 21.06 (2021) when Numan Siddique proposed it to support source routing plus advanced traffic steering for SD-WAN and multi-tenant deployments. Before that, the static-route `policy=src-ip` could only do simple source routing and could not match L4 (tcp.dst). LRP extends the match expression to a full microflow plus adds the `reroute` action for traffic steering. OVN 22.06 added `nexthops` (a list) for ECMP-style PBR; OVN 24.03 added a fourth action `jump` (jump to a different priority within the same table, similar to an iptables jump target; note that the branch-22.03 baseline curriculum has only three actions: `allow`, `drop`, `reroute`).

#### Position in the architecture

The table sits at the **NBDB input layer**, scope **per-Logical_Router**. Strong reference from `Logical_Router.policies` (a UUID set). It compiles into Logical_Flow at stage `LR_IN_POLICY` (table 7 in OVN 22.03), evaluated **before** the main routing stage. Pipeline order: `LR_IN_ADMISSION` -> `LR_IN_LOOKUP_NEIGHBOR` -> `LR_IN_LEARN_NEIGHBOR` -> `LR_IN_IP_INPUT` -> ... -> `LR_IN_POLICY` -> `LR_IN_ARP_RESOLVE` -> `LR_IN_IP_ROUTING` -> ... Consequence: LRP wins over static route and can override it with the `reroute` action.

#### Role and responsibilities

Four roles:
(1) **Source-based routing**: route by source IP or subnet rather than destination.
(2) **L4-aware steering**: for example, HTTPS traffic from the app tier must go through a security gateway.
(3) **Selective drop** at L3 (complementing ACL at L2 on the LS) for rules applied at the router scope.
(4) **Traffic engineering**: re-route a subset of traffic onto a different underlay for QoS or cost optimisation.

#### Why it exists (design motivation)

Before OVN 21.06, the engineer had to use the combination `static_route policy=src-ip` (source IP only, no L4) plus ACLs on upstream or downstream LSes to simulate PBR. This was limited and scattered the rules across two layers (LS and LR). LRP gathers PBR logic into one table at the LR plus a full microflow match expression. Especially useful for SD-WAN deployments that need to steer traffic on L4-L7 hints (for example, `tcp.dst == 443 && ip4.dst == $public-saas` to reroute through a proxy edge).

#### Problems it solves

(1) **Full PBR with L4 matching** without being limited to source IP; (2) **Override static route** for a traffic subset without disrupting the main route table; (3) **Centralised PBR rules** at the LR rather than scattered on the LS; (4) **Compile-time priority** is unambiguous and does not depend on rule insertion order.

#### Importance

This is a **tier-2, medium** table of NBDB, required for multi-WAN, SD-WAN, or multi-tenant deployments with traffic-steering requirements. Not every cluster uses it (a simple cluster only needs static routes), but when it is needed, it is the only OVN-level tool. An engineer who does not understand LRP cannot debug "traffic does not follow the default route and goes through a different nexthop" (because LRP overrides silently).

#### How it works

(a) `ovn-nbctl lr-policy-add lr1 100 'ip4.src == 10.0.0.5' reroute 192.168.2.1` issues an NBDB transaction creating an LRP row and inserting the UUID into `Logical_Router.policies` on lr1. (b) northd `build_routing_policy_flow` (function-name anchor in `northd/northd.c`) iterates policies and compiles them into Logical_Flow at stage `LR_IN_POLICY` with `match=(ip4.src == 10.0.0.5)` plus an action depending on the type: `allow` becomes `next;`, `drop` becomes `drop;`, `reroute` becomes `reg0 = 192.168.2.1; reg9[2] = 0; ...; next;` (setting the nexthop register to override the routing stage). (c) ovn-controller installs the flow at the corresponding OF table (the mapping `LR_IN_POLICY` is OF table 16 in OVN 22.03; cross-link Part 13.16).

#### Column classification

Schema with seven columns (verified branch-22.03): `priority` (integer 0 to 32767, required), `match` (string, required, microflow expression), `action` (enum allow, drop, reroute, required at 22.03; `jump` added in 24.03 and later), `nexthop` (string, optional, the IP nexthop when action=reroute), `nexthops` (set of strings, optional, ECMP nexthops, 22.06 and later), `options` (map), `external_ids` (map of metadata). No unique index; duplicate `(priority, match)` is schema-valid.

#### How an engineer masters operations

(1) Add an allow rule: `ovn-nbctl lr-policy-add lr1 200 'ip4.src == 10.0.0.5' allow`; (2) Add a drop rule: `ovn-nbctl lr-policy-add lr1 150 'ip4.src == 10.0.0.10 && tcp.dst == 22' drop`; (3) Add a reroute: `ovn-nbctl lr-policy-add lr1 100 'ip4.src == 10.0.0.0/24 && ip4.dst == 8.8.8.8' reroute 192.168.99.1`; (4) Verify: `ovn-nbctl lr-policy-list lr1`; (5) Cleanup: `ovn-nbctl lr-policy-del lr1 <priority> [<match>]`.

#### Workflow (the standard usage flow)

(1) Identify the use case (source routing / selective drop / steering); (2) Add the LRP with an appropriate priority (above the default-route behaviour); (3) Verify the compile: `ovn-sbctl lflow-list lr1 | grep "lr_in_policy"`; (4) Test with specific traffic; (5) Cross-check that there is no overlap with a static route or NAT.

#### When something goes wrong

"The default route works normally but VM A is unexpectedly dropped" - check `ovn-nbctl lr-policy-list lr1` for a drop rule whose match covers the `ip4.src` of VM A; "Reroute does not hit" - check the priority and match, then verify the Logical_Flow compiles correctly; "Action `jump` reports an error on OVN 22.03" - the `jump` action only exists from 24.03 and later; fall back to `allow` plus rule reordering.

#### Version differences

OVN 21.06: launched with three actions (allow, drop, reroute) and a single nexthop; OVN 22.06: added a `nexthops` set for ECMP-style PBR; OVN 24.03: added the `jump` action. Curriculum baseline 22.03: only three actions.

#### How to observe and verify

`ovn-nbctl lr-policy-list lr1`; `ovn-nbctl --columns=priority,match,action,nexthop,nexthops list Logical_Router_Policy`; `ovn-sbctl lflow-list lr1 | grep "lr_in_policy"`; `ovs-ofctl -O OpenFlow15 dump-flows br-int "table=16"` (OVN 22.03 mapping `LR_IN_POLICY` is OF table 16).

#### Source code references

Schema [`ovn-nb.ovsschema` on branch-22.03](https://raw.githubusercontent.com/ovn-org/ovn/branch-22.03/ovn-nb.ovsschema) declares the `Logical_Router_Policy` table with seven non-root columns; the action enum has three values: `allow`, `drop`, `reroute`. The compile lives in [`northd/northd.c`](https://github.com/ovn-org/ovn/blob/branch-22.03/northd/northd.c); function-name anchor `build_routing_policy_flow` iterates policies and emits Logical_Flow at stage `LR_IN_POLICY` (Rule 14.4: function names are stable anchors; line numbers drift across versions).

#### Real-world incident

Production case 2024: a multi-WAN deployment with two transits (primary cost-effective, backup high-quality). Desired rule: VoIP traffic (UDP destination ports 5060 and 5004) goes through the backup link for QoS. Implementation: `ovn-nbctl lr-policy-add lr-edge 200 'udp.dst == 5060 || udp.dst == 5004' reroute 10.99.99.1`. After deployment, VoIP latency dropped from 80 ms to 15 ms. Two weeks later, a bug: SIP register packets dropped intermittently. Root cause: SIP register uses a UDP source port, not destination, and the rule was matching the wrong direction. Fix: also add a rule for `udp.src == 5060`.

#### Synthetic exercise

```bash
sudo ovn-nbctl lr-policy-add lr1 200 'ip4.src == 10.0.0.5 && tcp.dst == 443' reroute 192.168.99.1
sudo ovn-nbctl lr-policy-add lr1 150 'ip4.src == 10.0.0.10' drop
sudo ovn-nbctl lr-policy-add lr1 100 'ip4.src == 10.0.0.0/24' allow
sudo ovn-nbctl lr-policy-list lr1
sudo ovn-sbctl lflow-list lr1 | grep "lr_in_policy"
```

Observe: three flows with different priorities, highest wins; the reroute action sets the nexthop register to override.

#### Common failures and diagnostic signals

(a) Two LRPs on the same datapath with the same priority and no unique index produce an undefined tie-break; both compile; (b) Action `reroute` to a nexthop that is not reachable (no ARP entry) causes traffic to drop silently at the ARP_RESOLVE stage; (c) Match expression typo (`tcp_dst` instead of `tcp.dst`) leads to a northd error log; (d) An LRP unintentionally overriding the default route should be caught by auditing `lr-policy-list` regularly.

#### Comparison with other systems

Linux `ip rule add` (policy routing) plus `iptables -t mangle MARK` has the same concept; Cisco IOS `route-map` plus `set ip next-hop` maps directly; the FRR PBR plugin is comparable; AWS Transit Gateway route-table policy-based mapping is partial (TGW has no native L4-aware match). NSX-T DFW plus Distributed Router PBR has comparable functionality.

### 13.3.W.5 `Forwarding_Group`, the active-active gateway pattern that replaces Load_Balancer

#### Core concept

`Forwarding_Group` (FG) is a **table that lives in OVN NBDB** describing an **active-active forwarding group**: a VIP (representative IP) plus a VMAC (representative MAC) mapped to a list of `child_port` entries (LSPs of the gateway or server appliances). Unlike Load_Balancer (which DNATs VIP to backend through conntrack), Forwarding_Group **does not** rewrite addresses; it only **forwards L2** packets to one of the `child_port` entries chosen by a hash, and the child LSP must handle the traffic itself (using the same virtual IP and MAC). This pattern is used for **active-active gateway HA** (for example, two or three firewall appliances on the same external segment sharing load) or for **multi-NIC bonding logic at L2 in OVN**. Type: **OVSDB non-root table**, with a strong reference from `Logical_Switch.forwarding_groups`. `child_port` is min=1 (at least one child is required).

#### History and origin

Forwarding_Group appeared in OVN 20.03 (early 2020) when Manoj Sharma and Numan Siddique proposed it for the active-active firewall HA use case in the VMware NSX-T migration scenario (NSX-T has an active-active Edge Cluster with BFD; OVN needed an equivalent feature). The original schema had five columns; OVN 20.06 added `liveness` (boolean) for BFD-based health check, ensuring forwarding only goes to a child whose BFD is up. OVN 22.03 stabilised at six columns. This is a niche feature, not commonly used (most deployments use Load_Balancer plus ECMP), and its documentation is thinner than the other tables, but it is critical for specific HA patterns.

#### Position in the architecture

The table sits at the **NBDB input layer**, scope **per-Logical_Switch**. Strong reference from `Logical_Switch.forwarding_groups` (a UUID set). It compiles into Logical_Flow at stage `LS_IN_L2_LKUP` (table 19 in OVN 22.03); when a packet matches `eth.dst == <vmac>`, the action selects one `child_port` via a five-tuple hash and sets `outport=child_port_X`. It does not interact with Logical_Router (FG is L2-only, not L3 NAT).

#### Role and responsibilities

Three roles: (1) **Active-active L2 gateway HA**: the VIP and VMAC represent a firewall or gateway cluster, and traffic is hashed to a different instance per flow; (2) **Liveness-aware forwarding**: when `liveness=true` and the child has a BFD session, a downed child is skipped; (3) **Alternative to Load_Balancer when DNAT is not needed**: the backend keeps the virtual IP and MAC unchanged; suitable when the backend must reply with the same VIP (for example, a transparent firewall).

#### Why it exists (design motivation)

Before OVN 20.03, active-active firewall HA had to be implemented via: (a) Load_Balancer DNAT (but the backend did not share the VIP, breaking transparency); (b) ECMP static route (only works at the L3 LR, not at the L2 LS); (c) external hardware (F5, A10, outside OVN's scope). Forwarding_Group filled the gap for deployments needing transparent active-active L2 gateway. Typical use case: two or three Palo Alto or Fortinet firewall appliances on the same external segment, sharing a VIP and VMAC, with OVN forwarding traffic to them by hash to share load without DNAT.

#### Problems it solves

(1) **Active-active L2 transparent gateway HA** without DNAT; (2) **BFD-aware forwarding** with `liveness=true`; (3) **Load distribution by five-tuple hash** without depending on an external LB; (4) **Failover under 200 ms** through BFD detection of a downed child.

#### Importance

This is a **tier-2, medium, niche** NBDB table. It is not necessary for a standard deployment (Load_Balancer plus NAT covers 95 percent of use cases), but it is critical for deployments migrating from NSX-T or needing transparent active-active firewall. Production clusters using FG usually have a deep network-design architect on the team.

#### How it works

(a) `ovn-nbctl create Forwarding_Group name=fg-fw-cluster vip=10.0.0.100 vmac=02:00:00:fw:01 liveness=true child_port='["lsp-fw-1","lsp-fw-2","lsp-fw-3"]'` creates the row and inserts it into `Logical_Switch.forwarding_groups`. (b) northd `build_fwd_group_lflows` (function-name anchor; see `northd/northd.c` branch-22.03) compiles into Logical_Flow at stage `LS_IN_L2_LKUP`: `match=(eth.dst == 02:00:00:fw:01)` plus an action that selects one `child_port` by hash. (c) When `liveness=true`, the match condition adds `BFD.status=up` of the child. (d) ovn-controller installs the OF rule with a group-select type on br-int.

#### Column classification

Schema with six columns (verified branch-22.03): `name` (string, required), `vip` (string, required, the representative IP), `vmac` (string, required, the representative MAC), `liveness` (boolean, required, true to enable BFD-aware), `child_port` (set of strings, min=1, the LSP name list), `external_ids` (map of metadata).

#### How an engineer masters operations

(1) Create the FG: `ovn-nbctl create Forwarding_Group name=fg-fw vip=10.0.0.100 vmac='"02:00:00:fw:01"' liveness=true child_port='["lsp-fw-1","lsp-fw-2"]'`; (2) Attach to the LS: `ovn-nbctl add Logical_Switch ls-ext forwarding_groups <fg_uuid>`; (3) Verify: `ovn-nbctl list Forwarding_Group fg-fw`; (4) (Optional) Set up BFD on the child_port through the BFD table; (5) Verify the compile: `ovn-sbctl lflow-list ls-ext | grep "ls_in_l2_lkup"`.

#### Workflow (the standard usage flow)

(1) Plan the HA pattern (when to use FG versus LB); (2) Set up child LSPs with the same virtual VIP and VMAC (handle the ARP responder yourself or disable it); (3) Create the FG with the child_port list; (4) Attach to the LS; (5) Test even traffic distribution: send 100 connections from different sources and verify the hash splits evenly across the three children; (6) Test failover: bring one child down and verify traffic redirects to the remaining children within 200 ms (when liveness plus BFD are enabled).

#### When something goes wrong

"Traffic does not split evenly across children" - check that the five-tuple hash is diverse; with few sources, the hash does not spread; (b) "Failover slower than 1 s" - check the BFD interval (default 200 ms times a detect_mult of 3 equals 600 ms; tune as needed); (c) "Child is down but still receives traffic" - `liveness=false`, or the BFD session is not bound to the child_port LSP correctly.

#### Version differences

OVN 20.03: launched with five columns (no `liveness` yet); OVN 20.06: added `liveness` (boolean) and BFD-aware forwarding; OVN 22.03: stabilised at six columns.

#### How to observe and verify

`ovn-nbctl list Forwarding_Group`; `ovn-nbctl --columns=name,vip,vmac,liveness,child_port list Forwarding_Group`; `ovn-sbctl lflow-list ls-ext | grep "ls_in_l2_lkup.*02:00:00:fw"` to see the forwarding flow; `ovs-ofctl -O OpenFlow15 dump-groups br-int` to see the group select.

#### Source code references

Schema [`ovn-nb.ovsschema` on branch-22.03](https://raw.githubusercontent.com/ovn-org/ovn/branch-22.03/ovn-nb.ovsschema) declares the `Forwarding_Group` table with six non-root columns and `child_port` min=1. The compile lives in [`northd/northd.c`](https://github.com/ovn-org/ovn/blob/branch-22.03/northd/northd.c); function-name anchor `build_fwd_group_lflows` (or `build_forwarding_groups_flows` depending on refactor) iterates forwarding groups and emits a flow at stage `LS_IN_L2_LKUP`. Cross-link to the separate BFD table for liveness.

#### Real-world incident

Production case 2023: an OSP cluster migrated from NSX-T to OVN, with three transparent active-active Fortinet firewalls on the external segment sharing a VIP `10.0.0.100` and VMAC. NSX-T previously used an "active-active Edge Cluster". The OVN migration used Forwarding_Group plus BFD. After deployment, the three firewalls received roughly even traffic (about 33 percent each by five-tuple hash). One firewall rebooted for maintenance, BFD detected it as down within 600 ms, and traffic redirected to the remaining two with no connection drops thanks to stateful sync between firewalls (state replicated).

#### Synthetic exercise

```bash
sudo ovn-nbctl lsp-add ls-ext lsp-fw-1
sudo ovn-nbctl lsp-add ls-ext lsp-fw-2
sudo ovn-nbctl lsp-set-addresses lsp-fw-1 "02:00:00:fw:01 10.0.0.100"
sudo ovn-nbctl lsp-set-addresses lsp-fw-2 "02:00:00:fw:01 10.0.0.100"
sudo ovn-nbctl create Forwarding_Group name=fg-fw vip="10.0.0.100" vmac='"02:00:00:fw:01"' liveness=false child_port='["lsp-fw-1","lsp-fw-2"]'
FG_UUID=$(sudo ovn-nbctl --bare --columns=_uuid find Forwarding_Group name=fg-fw)
sudo ovn-nbctl add Logical_Switch ls-ext forwarding_groups $FG_UUID
sudo ovn-sbctl lflow-list ls-ext | grep "02:00:00:fw"
```

Observe: a forward flow on `eth.dst == 02:00:00:fw:01` whose action selects a `child_port` by hash.

#### Common failures and diagnostic signals

(a) A child LSP not configured with the same VIP and VMAC means the ARP for the VIP does not respond correctly (when `port_security` is enabled, an ARP from the LSP with a spoofed IP is dropped); use `lsp-set-port-security-disable` on the child; (b) `liveness=true` without a bound BFD session leads to undefined behaviour; the child may be skipped permanently; (c) Confusing Forwarding_Group with Load_Balancer: FG does not DNAT, the backend must handle the original VIP itself, and FG cannot replace an LB use case; (d) The hash uses the five-tuple only; with little traffic flow diversity, distribution is uneven, so plan according to the expected traffic profile.

#### Comparison with other systems

NSX-T `Edge Cluster active-active` maps directly; F5 BIG-IP `Active-Active Pair` has the same transparent-forwarding concept; AWS Gateway Load Balancer (GWLB) is similar (with GENEVE encapsulation); Cisco ACI `Service Graph` with active-active service nodes is comparable. Difference from Linux: the kernel has no native equivalent (LVS DR mode is the closest, but it uses MAC rewrite, not L2 transparent forwarding). Difference from OVN Load_Balancer: LB is DNAT with conntrack address rewrite; FG is an L2 forward that keeps addresses unchanged, so the semantics are completely different.

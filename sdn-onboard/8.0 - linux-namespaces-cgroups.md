## 8.0.1 The seven Linux namespaces

Kernel 3.8 and later ship all seven namespaces:

| Namespace | What it isolates | Flag |
|---|---|---|
| `PID` | Process ID space | `CLONE_NEWPID` |
| `NET` | Network stack (interfaces, routing, iptables, sockets) | `CLONE_NEWNET` |
| `MNT` | Mount points | `CLONE_NEWNS` |
| `UTS` | Hostname and domain name | `CLONE_NEWUTS` |
| `IPC` | System V IPC (message queue, semaphore, shared memory) | `CLONE_NEWIPC` |
| `USER` | UID and GID mapping (unprivileged containers) | `CLONE_NEWUSER` |
| `CGROUP` | cgroup hierarchy | `CLONE_NEWCGROUP` |

Each namespace is created through a flag in the `clone(2)` syscall, or through `unshare(2)` for an existing process. A container runtime (Docker, containerd, CRI-O) combines several namespaces to produce full isolation.

USC Lab 1 (`doc/ovs/OVS.txt` Section 1.1) states: "In Linux namespaces, you can isolate the global system resources between processes. Any changes to the global resource will only be visible to other processes that are members of the namespace but are invisible to other processes."

## 8.0.2 Network namespace: `ip netns`, `/var/run/netns`

A network namespace owns an independent network stack: interfaces, routing tables, ARP cache, iptables rules, socket table, and port number space.

```bash
[host]# ip netns add ns1                     # create a persistent namespace
[host]# ls /var/run/netns/
ns1
[host]# ip netns exec ns1 ip link show        # run a command inside ns1
[host]# ip netns delete ns1                  # remove it
```

`ip netns add` creates a reference at `/var/run/netns/<name>`. This is a bind mount of `/proc/<pid>/ns/net` from a kernel thread. The namespace is persistent after the process exits, unlike the non-persistent namespace produced by `unshare`.

USC Lab 1 uses a canonical topology:

```
   ns1 (h1) --veth1-peer--+
                          +-> OVS bridge s1 (root namespace)
   ns2 (h2) --veth2-peer--+
```

Configuration:

```bash
[host]# ip netns add h1 && ip netns add h2
[host]# ip link add veth1 type veth peer name veth1-br
[host]# ip link add veth2 type veth peer name veth2-br
[host]# ip link set veth1 netns h1
[host]# ip link set veth2 netns h2
[host]# ovs-vsctl add-port s1 veth1-br
[host]# ovs-vsctl add-port s1 veth2-br
[host]# ip -n h1 addr add 10.0.0.1/24 dev veth1
[host]# ip -n h2 addr add 10.0.0.2/24 dev veth2
[host]# ip -n h1 link set veth1 up
[host]# ip -n h2 link set veth2 up
[host]# ip netns exec h1 ping -c 3 10.0.0.2
```

This is the foundation lab of every SDN experiment.

## 8.0.3 `unshare` and `nsenter`

These util-linux tools create ad-hoc namespaces:

```bash
[host]# unshare --net /bin/bash           # create a shell in a fresh net namespace
[host-ns]# ip link show                    # only sees lo
[host-ns]# exit                            # the namespace is destroyed

[host]# nsenter -t <pid> -n /bin/bash    # join the net namespace of a PID
```

The difference from `ip netns`: this kind of namespace is not persistent and has no reference under `/var/run/netns/`. Use it when you want a quick test, or when you must join the namespace of a running container:

```bash
[host]# PID=$(docker inspect -f '{{.State.Pid}}' <container>)
[host]# nsenter -t $PID -n ip addr
```

## 8.0.4 cgroups v1 versus v2

cgroups (control groups) control resources (CPU, memory, I/O, network), in contrast to namespaces, which provide isolation. There are two generations:

**cgroup v1** (kernel 2.6.24): each subsystem (cpu, memory, blkio, net_cls, net_prio, and so on) has its own hierarchy. The networking subsystems are:
- `net_cls`: assigns a classid to a process so that `tc` can filter on it.
- `net_prio`: assigns a per-interface priority to a process.

**cgroup v2** (kernel 4.5 stable, 4.16 default): a unified hierarchy holds every subsystem in a single tree. **It removed `net_cls` and `net_prio`.** Network bandwidth control moves to eBPF through `bpf_classify` or the `cgroup_skb` hook.

Ubuntu 22.04 and later default to cgroup v2. To verify:

```bash
[host]# mount | grep cgroup
cgroup2 on /sys/fs/cgroup type cgroup2 (rw,nosuid,nodev,noexec,relatime,nsdelegate,memory_recursive_prot)
```

For an SDN lab, cgroup v2 is sufficient for VM and container resource control. If you need network bandwidth QoS, use `tc` directly on the veth or tap interface instead of cgroup (see Part 8.3).

## 8.0.5 Syscall internals: `clone()`, `unshare()`, `setns()`

The three syscalls that create and join a namespace all live in `kernel/nsproxy.c` and `kernel/fork.c`.

### `clone(CLONE_NEWNET, ...)`

`clone(2)` is a fork-with-flags syscall. When the `CLONE_NEWNET` flag is set, the kernel creates a new `struct net` and assigns it to `task_struct->nsproxy->net_ns`. The child process gets a fully separate network stack.

```c
// kernel/fork.c, copy_namespaces() function (called from copy_process)
if (flags & CLONE_NEWNET) {
    new_nsproxy->net_ns = copy_net_ns(flags, user_ns, old_ns->net_ns);
}
```

`copy_net_ns()` calls `setup_net()` to initialize a new routing table, a new ARP cache, new iptables hooks, and new sysctl values. Every registered subsystem (TCP, UDP, ICMP, netfilter) is duplicated for the new `net` instance.

### `unshare(CLONE_NEWNET)`

Unlike `clone()`, `unshare()` does not create a new process. It **detaches** the namespace of the current process. The process keeps the same PID and the same `task_struct`, but its `nsproxy` is swapped to a fresh new namespace.

```c
// kernel/nsproxy.c, sys_unshare()
new_nsproxy = create_new_namespaces(unshare_flags, ...);
switch_task_namespaces(current, new_nsproxy);
```

Consequence: any socket that was already open before `unshare()` stays in the old namespace, because the kernel field `socket->sk->sk_net` already references the old namespace. Sockets opened after `unshare()` belong to the new namespace. This is the pattern that lets a daemon "leak" sockets across namespaces, useful for a daemon such as `ovsdb-server` running in the root namespace but opening sockets inside a target namespace.

### `setns(fd, CLONE_NEWNET)`

`setns()` joins an existing namespace through a file descriptor. The workflow:

```c
int fd = open("/proc/<pid>/ns/net", O_RDONLY);  // or /var/run/netns/<name>
setns(fd, CLONE_NEWNET);
// Now in the target namespace
close(fd);
```

`nsenter -t <pid> -n` uses this pattern: `open()` `/proc/<pid>/ns/net`, then `setns()`, then `execve()` for the shell.

### Lifecycle ref counting

The kernel ref-counts each `struct net`:

- Plus one for each process attached (through `nsproxy`).
- Plus one for each mount bind (`/var/run/netns/<name>` is a bind mount onto `/proc/.../ns/net`).
- Plus one for each socket open inside the namespace.

When the ref-count reaches 0, `net_drop_ns()` triggers and the kernel destroys the `struct net`, freeing the routing table and the ARP cache. This is why `ip netns delete ns1` requires that no process and no socket still references the namespace.

To inspect the ref-count:

```bash
[host]# readlink /proc/<pid>/ns/net
net:[4026532000]    # the inode number identifies the namespace

[host]# lsns -t net
        NS TYPE NPROCS   PID USER   NETNSID NSFS                COMMAND
4026531992 net     245     1 root unassigned                     /sbin/init
4026532000 net       2 12345 root          0 /var/run/netns/ns1 sleep infinity
```

`NPROCS` is the number of processes inside the namespace. `NETNSID` is the ID assigned for `ip netns identify` cross-reference.

## 8.0.6 Network namespace for the OVS daemon

The OVS userspace daemon `ovs-vswitchd` runs in the **root namespace** by default. When you create an OVS port of `type=internal`, or attach an OVS port to a veth peer in another namespace, the OVS netlink socket crosses the namespace boundary through the kernel.

### Pattern 1: an OVS internal port inside a namespace

```bash
[host]# ovs-vsctl add-port br-int my-veth -- \
        set Interface my-veth type=internal
[host]# ip link set my-veth netns ns1     # move the OVS internal port into ns1
[host]# ip -n ns1 addr add 10.0.0.5/24 dev my-veth
```

OVS userspace manages the port (OpenFlow configuration, VLAN tag), while the interface itself lives inside the `ns1` namespace. The kernel netlink Generic genl path proxies between `ovs-vswitchd` (in the root namespace) and `openvswitch.ko` (per-namespace).

### Pattern 2: the OVS daemon inside a namespace (non-default)

This is rarely needed, but you can run `ovs-vswitchd` inside its own namespace:

```bash
[host]# ip netns exec ns-ovs ovs-vswitchd ...
```

Use case: a multi-tenant OVS instance on the same host. It is complex and uncommon in production.

OVN-Kubernetes uses pattern 1 for every container interface. The container netns has one OVS port of `type=internal`, while OVS itself remains in the root namespace.

## 8.0.7 Anatomy Template A: `ip -n <ns> ...` and `lsns`

> **Context when run:** Inspect the current namespaces on a chassis, or troubleshoot when a process cannot find an expected interface.

```bash
[host]# lsns -t net
        NS TYPE NPROCS   PID USER     NETNSID NSFS                  COMMAND
4026531992 net     245     1 root   unassigned                       /sbin/init
4026532000 net       2 12345 root            0 /var/run/netns/ns1   sleep infinity
4026532007 net      15 23456 1000           1                        /usr/bin/firefox
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| `NS` (inode) | unique per namespace | duplicate (same fs) | (n/a) | Each namespace has a different inode |
| `NPROCS` | greater than 0 for a persistent namespace | 0 for a namespace in use | 0 for a namespace that should have processes | The process crashed, the namespace is about to be destroyed |
| `NETNSID` | assigned for a cross-namespace reference | unassigned | unassigned for a namespace that needs an ID | Run `ip netns set <name> <id>` |
| `NSFS` | bind mount path | empty (ad-hoc `unshare`) | empty for an expected persistent namespace | A persistent namespace requires a `/var/run/netns/` entry |
| `COMMAND` | the main process name | unexpected | a different process than expected | The wrong process is in the namespace |

```bash
[host]# ip -n ns1 link show
1: lo: <LOOPBACK> mtu 65536 qdisc noop state DOWN
N: veth1@if<peer>: <BROADCAST,MULTICAST> mtu 1500 ...

[host]# ip -n ns1 route
10.0.0.0/24 dev veth1 proto kernel scope link src 10.0.0.1
```

**Failure scenarios:**

- `lsns -t net` shows ns1 with `NPROCS=0`, but `ip netns list` still shows ns1: the process died but the namespace is alive due to the bind mount. Run `ip netns delete ns1` to clean up.
- `ip -n ns1 link show` only shows `lo`: no interface was moved into or created inside the namespace. Forgot `ip link set veth1 netns ns1`.
- `lsns` shows 200 or more net namespaces: the container runtime is creating namespaces without cleaning them up. Check the Docker or Kubernetes pod count.

## 8.0.8 Capstone POE: why does OVS userspace not need to move into the container netns?

> **Context:** A junior engineer first sees the OVS plus container pattern and asks: "would running `ovs-vswitchd` inside the container netns scale better, with isolation per container?"

**Claim to verify:** "Running `ovs-vswitchd` inside each container namespace gives better isolation than a shared root namespace."

**Predict:** False. This pattern does not scale and it violates the OVS architecture.

**Observe (assumed, to be verified in the lab):**

- Set up one `ovs-vswitchd` per container netns: each container runs an extra process at roughly 50 to 100 MB of memory. A 100-container cluster wastes 5 to 10 GB of memory on OVS instances.
- An OpenFlow controller must connect to N `ovs-vswitchd` instances rather than one, which means N flow tables and N OVSDB instances. Cluster-wide policy must sync N times.
- Tunnel-to-tunnel: container A wants to tunnel to container B; the two OVS instances must peer Geneve or VXLAN with each other. The underlay routing between container netns becomes very complex.

**Explain:**

1. The OVS architecture **assumes a shared kernel datapath** plus one userspace controller per host. Each `openvswitch.ko` instance is per network namespace (the kernel handles this), but a single `ovs-vswitchd` userspace process owns policy and flow installation. Splitting userspace per container breaks that assumption.
2. Container isolation (security and resources) comes from **the network namespace**, not from a separate OVS instance. One OVS root namespace plus N container netns, each with one port of `type=internal`, is enough isolation.
3. OVN-Kubernetes and OpenStack Neutron both use the shared OVS root plus container netns pattern. The per-container OVS pattern does not exist in any production-grade orchestrator.

**Falsification test:**

- Set up 10 containers with one OVS root and measure memory and flow install time.
- Set up 10 containers with 10 per-namespace OVS instances and measure the same metrics.
- If per-container OVS improves isolation or throughput, that is a counter-example. In practice, no metric points to a benefit, only overhead.

**Where the claim fails:** "Better isolation" is an illusion. Network isolation is already provided by the netns; a separate OVS instance does not add isolation, it only duplicates state. The correct rule is **one OVS root, N container netns, each container with one OVS port of `type=internal` inside its netns. This is the standard pattern in OVN, Neutron, and CNI implementations.**

> **Rule:** OVS userspace lives in the root namespace, as a single instance. Container isolation belongs at the network namespace level, not at the OVS instance level.

### Guided Exercise 1 (POE): two namespaces plus a veth pair, ping across

**Outcomes:** Understand that veth is point-to-point, and that routing or a bridge is required to connect more than two namespaces.

**Predict:**

A counter-hypothesis: if veth automatically routes between namespaces (without `ip route`), pinging from ns1 to ns2 should succeed as soon as the IP addresses are assigned.

**Observe:**

**1.** Create the namespaces and a single veth pair:

```bash
[host]# ip netns add ns1
[host]# ip netns add ns2
[host]# ip link add veth1 type veth peer name veth2
[host]# ip link set veth1 netns ns1
[host]# ip link set veth2 netns ns2
[host]# ip -n ns1 addr add 10.0.0.1/24 dev veth1
[host]# ip -n ns2 addr add 10.0.0.2/24 dev veth2
[host]# ip -n ns1 link set veth1 up
[host]# ip -n ns2 link set veth2 up
[host]# ip -n ns1 link set lo up
[host]# ip -n ns2 link set lo up
```

**2.** Ping:

```bash
[host]# ip netns exec ns1 ping -c 3 10.0.0.2
```

**Explain:**

Result: the ping succeeds. Contrary to the counter-hypothesis, veth is **point-to-point**, and the two ends in the two namespaces share a common subnet, so the Linux kernel routes through directly-connected automatically.

However, with **more than two namespaces** (ns3, ns4, and so on), veth alone is not enough. A Linux bridge or an OVS bridge must sit in the middle. This is why USC Lab 1 uses the topology `ns -> veth -> bridge -> veth -> ns` instead of a single veth pair.

**Finish:** Cleanup: `ip netns delete ns1 ns2`

## Key takeaways

First, the seven Linux namespaces give resource isolation along several dimensions. The network namespace (NET) is the most important dimension for SDN; it creates an isolated network stack with its own interfaces, routing, and iptables.

Second, `ip netns` creates a persistent namespace (with a reference file), while `unshare` and `nsenter` create ad-hoc ones. For an SDN lab, persistent is appropriate; for inspecting a container runtime, `nsenter` by PID is appropriate.

Third, veth is a point-to-point primitive, not enough for a topology with more than two nodes. It must combine with a Linux bridge (Part 8.1) or with OVS (Block IX) to build a real topology.

Fourth, cgroup v2 removed `net_cls` and `net_prio`. Modern network QoS uses `tc` directly or eBPF, no longer the cgroup network classifier.

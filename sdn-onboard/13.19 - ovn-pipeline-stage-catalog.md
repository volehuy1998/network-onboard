## §13.19.1 LS_IN_PORT_SEC_L2 — bộ lọc L2 spoofing đầu pipeline

**Khái niệm.** `LS_IN_PORT_SEC_L2` là stage logic **số 0** của pipeline ingress trên một Logical_Switch. Mỗi packet vào LS phải đi qua đây trước. Stage có hai trách nhiệm chéo nhau: (a) drop packet có `eth.src` không nằm trong tập `Logical_Switch_Port.port_security` (chống MAC spoofing tại L2), và (b) drop trễ những packet đã bị stage `LS_IN_PORT_SEC_IP` / `LS_IN_PORT_SEC_ND` đánh dấu trên `REGBIT_PORT_SEC_DROP` vì spoof IP/ARP/ND. Trên br-int, stage này map sang OF table 8 (theo công thức `8 + logical_table` ở Phần 13.16). Loại: **pipeline stage cornerstone**, sub-loại "ingress security gate".

**Lịch sử + bối cảnh.** Stage tồn tại ngay từ OVN 2.4 (2015) khi Bryant + Pettit chuyển port-security của NSX sang OVN. Phiên bản đầu chỉ check eth.src đơn giản. OVN 20.06 (2020) tách port-security thành 3 stage liên tiếp (`L2`, `IP`, `ND`) và đưa vào pattern "deferred drop" qua REGBIT_PORT_SEC_DROP để IP/ND fail vẫn dùng chung instruction `drop` ở stage L2 cuối cùng (gom điểm rớt về một chỗ giúp đếm drop counter dễ hơn). OVN 21.03 thêm hỗ trợ multicast MAC trong port_security.

**Vị trí trong kiến trúc.** Stage logic 0 ⇒ OF table 8 (`8 + 0`). Đây là điểm **đầu tiên** ovn-controller cài flow trên br-int cho mọi packet ingress đi từ tap port (vif). Trước stage này không có pipeline OVN nào nữa, chỉ có OF table 0 do OVN cài để map vif port number → datapath (set `metadata = datapath.tunnel_key`, `reg14 = inport.tunnel_key`) trước khi `goto_table 8`. Scope: per-packet, per-LSP.

**Vai trò.** Vai trò chính: **gatekeeper L2 anti-spoofing**. Vai trò phụ: **drop sink** cho mọi packet bị stage trước flag (REGBIT_PORT_SEC_DROP). Nếu disable stage này thì tenant có thể giả mạo MAC của VM khác trên cùng LS (lateral spoofing). Phạm vi ảnh hưởng: 1 packet đầu vào, không có state lưu lại sau stage.

**Vì sao sinh ra.** Trước OVN, NSX-V dùng vSwitch ESXi với feature `dvfilter` để chặn MAC spoofing. Khi OVN sinh ra để abstract L2 đa tenant trên Linux KVM, port-security thành **must-have** vì hypervisor Linux kernel mặc định không chặn MAC spoofing trên tap interface (driver tap không kiểm tra src MAC). Nếu thiếu, tenant chiếm được VM có thể giả mạo MAC gateway → lừa ARP của các VM cùng subnet → MITM. Stage này lấp pain point đó tại layer logic OVN.

**Vấn đề giải quyết.** Cụ thể là MAC spoofing tại tap port: VM gửi packet có `eth.src` khác MAC đã được declare lúc tạo `Logical_Switch_Port.addresses`. Cách giải: ovn-northd compile từ `Logical_Switch_Port.port_security` (list MAC cho phép) ra Logical_Flow priority 50 match `inport == "lsp1" && eth.src == {AA:BB:CC:DD:EE:01, AA:BB:CC:DD:EE:02}` action `next`, và priority 0 action `drop` cho packet không match. Phạm vi giải quyết: chỉ L2 spoofing; spoof IP/ARP do hai stage riêng (PORT_SEC_IP, PORT_SEC_ND) phụ trách rồi đẩy về đây để drop chung.

**Tầm quan trọng.** Tier 1 cornerstone. Mỗi packet ingress đi qua, mọi LS đều có. Nếu stage misconfig (port_security trống), security model toàn bộ tenant sụp. Engineer must-know: biết exact OF table id (table 8), biết match shape, biết REGBIT_PORT_SEC_DROP semantics deferred drop. Không hiểu stage này thì không debug được class lỗi "VM A ping được VM B nhưng MAC anomaly trong tcpdump".

**Cơ chế hoạt động.** Logical_Flow do `build_lswitch_input_port_security_l2` (`northd/northd.c`) sinh, theo template:

```
table=0 (ls_in_port_sec_l2), priority=50,
  match=(inport == "lsp-vm1" && eth.src == {AA:BB:CC:DD:EE:01}),
  action=(next;)

table=0 (ls_in_port_sec_l2), priority=0,
  match=(1),
  action=(drop;)
```

Khi stage IP/ND ở downstream phát hiện spoof IP, chúng set `reg0[0] = 1` (REGBIT_PORT_SEC_DROP) rồi vẫn `next;` đi tiếp. Cuối pipeline LS_IN, một flow đặc biệt match `reg0[0] == 1` → drop. Tuy hai cách "drop ngay tại L2" và "drop trễ qua REGBIT" cùng tồn tại, từ OVN 20.06 path REGBIT chiếm ưu thế cho IP/ND vì giúp đếm drop tập trung. Invariant: nếu reg0[0] = 1 sau stage này, packet **chắc chắn** sẽ bị drop trước khi rời LS_IN.

**Cách kỹ sư vận hành thành thạo.** Bốn kỹ năng: (1) **Đọc**: hiểu rằng `lsp-set-port-security-addresses lsp1 "AA:BB:CC:DD:EE:01 10.0.0.5"` sinh ra entry cho cả MAC và IP filter; xem flow bằng `ovs-ofctl dump-flows br-int table=8 | grep <inport_tunnel_key>`. (2) **Sửa**: `ovn-nbctl lsp-set-port-security-addresses lsp1 ...` rồi đợi northd compile xong (`ovn-nbctl --wait=hv sync`). (3) **Quan sát**: dùng `ovn-trace` với packet eth.src giả mạo, xem stage nào drop. (4) **Debug**: nếu user báo "VM mất kết nối sau khi đổi MAC", reflex là đi check `lsp-get-port-security`, xem MAC mới có nằm trong list không.

**Phân loại.** Loại stage: **security gate** (subtype L2 anti-spoofing). Khác với `LS_IN_ACL` (policy enforcement, có state) và `LS_IN_PRE_ACL` (state setup), stage này stateless thuần, chỉ match field bất biến của packet (eth.src, inport).

**Quy trình sử dụng.** Khi cấu hình LSP:
1. Tạo LS + LSP: `ovn-nbctl ls-add ls1 && ovn-nbctl lsp-add ls1 lsp-vm1`
2. Set MAC: `ovn-nbctl lsp-set-addresses lsp-vm1 "AA:BB:CC:DD:EE:01"`
3. Bật port security: `ovn-nbctl lsp-set-port-security lsp-vm1 "AA:BB:CC:DD:EE:01"`
4. Verify: `ovn-sbctl lflow-list ls1 | grep "ls_in_port_sec_l2"` thấy 2 flow priority 50 và 0
5. Trên hypervisor: `ovs-ofctl dump-flows br-int table=8 | head -5` thấy rule tương ứng đã được install
6. Kiểm thử spoof: dùng `scapy` gửi packet với eth.src sai → verify drop counter tăng (`ovs-ofctl dump-flows br-int table=8 | grep "n_packets="`)

**Khi xảy ra sự cố.** Symptom điển hình: (a) "VM vừa restart không gửi được packet nào" → check xem MAC VM có khớp `lsp-get-port-security` không (cloud-init đôi khi sinh MAC random nếu profile sai). (b) "Sau live migration packet mất 30s" → ARP cache stale, không phải stage này, nhưng counter drop ở table 8 sẽ tăng → giúp confirm symptom. Cross-link: Phần 20.0 §incident decision tree, Phần 13.18 §REGBIT_PORT_SEC_DROP.

**Liên quan mật thiết.** Tightly coupled với: `Logical_Switch_Port.port_security` (column NBDB nguồn dữ liệu), `LS_IN_PORT_SEC_IP` + `LS_IN_PORT_SEC_ND` (hai stage IP/ND set REGBIT chung), `REGBIT_PORT_SEC_DROP` (cờ deferred drop), `OFTABLE_LOG_INGRESS_PIPELINE` (stage thuộc nhóm OF table này). Loose: ARP cache trên VM (layer trên), iptables forward chain (layer dưới ngoài OVN).

**Khác biệt giữa các phiên bản.**
- OVN 2.4 (2015): chỉ một stage `port_security`, kiểm tra eth.src đơn thuần
- OVN 20.06 (2020): tách thành ba stage `L2`/`IP`/`ND`, REGBIT_PORT_SEC_DROP deferred drop
- OVN 21.03 (2021): hỗ trợ multicast MAC trong port_security
- OVN 22.03 (2022): cho phép `unknown` keyword trong addresses → tắt port-security để debug
- OVN 24.03 (2024): tối ưu compile để giảm số flow khi LSP có >100 MAC (rare nhưng có)

**Cách quan sát + xác minh.** Ba phương pháp:

```
$ ovn-sbctl lflow-list ls1 | grep "ls_in_port_sec_l2"
  table=0 (ls_in_port_sec_l2), priority=50,
    match=(inport == "lsp-vm1" && eth.src == {AA:BB:CC:DD:EE:01}),
    action=(next;)

$ ovs-ofctl dump-flows br-int table=8 | grep "metadata=0x1"
  cookie=0x1, table=8, n_packets=523, dl_src=AA:BB:CC:DD:EE:01,
  metadata=0x1, actions=resubmit(,9)

$ ovn-trace ls1 'inport=="lsp-vm1" && eth.src==BB:BB:BB:BB:BB:BB && eth.dst==CC:..'
  → ingress(dp="ls1", inport="lsp-vm1")
    0. ls_in_port_sec_l2 (no match)  /* drops at default priority 0 */
    drop;
```

**Source code.** Tham chiếu OVN branch-22.03 (baseline curriculum):
- `northd/northd.c` `build_lswitch_input_port_security_l2()`: build flow stage này từ list `Logical_Switch_Port.port_security`
- `northd/northd.c` `build_lswitch_input_port_security_ip()` + `_nd()`: hai sibling stage set REGBIT_PORT_SEC_DROP
- `controller/lflow.c` `consider_logical_flow()`: ovn-controller dịch Logical_Flow xuống OF flow trên br-int
- `lib/lflow.h` macro `S_SWITCH_IN_PORT_SEC_L2 = 0`: định nghĩa stage id

**Trường hợp sự cố thực tế.** Q1/2024 multi-tenant cluster: tenant báo "VM A sau live migration mất 12s mới ping lại được". Investigation: ovn-trace từ A→B kết quả OK, nhưng tcpdump trên hypervisor mới thấy ARP probe đầu bị drop. Đếm `ovs-ofctl dump-flows br-int table=8 | grep n_packets` cho LSP của A: drop_count nhảy 4 trong 12s. Root cause: live migration trigger MAC re-detect, ovn-controller cần ~10s để cập nhật flow set mới với MAC hiện tại của VM (cũ là MAC trước migration nếu virsh template sai). Fix: cập nhật virsh definition giữ MAC cố định, set `lsp-set-port-security` đúng từ đầu.

**Bài tập synthetic.** Mininet topology 1 LS + 2 LSP, một VM legitimate và một attacker:

```bash
sudo ovn-nbctl ls-add ls-lab
sudo ovn-nbctl lsp-add ls-lab lsp-good
sudo ovn-nbctl lsp-set-addresses lsp-good "AA:11:11:11:11:11"
sudo ovn-nbctl lsp-set-port-security lsp-good "AA:11:11:11:11:11"
sudo ovn-nbctl lsp-add ls-lab lsp-bad
sudo ovn-nbctl lsp-set-addresses lsp-bad "BB:22:22:22:22:22"
sudo ovn-nbctl lsp-set-port-security lsp-bad "BB:22:22:22:22:22"
# Lab task 1: từ lsp-bad, gửi packet với eth.src=AA:11:11:11:11:11 → verify drop
# Lab task 2: gỡ port_security trên lsp-bad (ovn-nbctl lsp-set-port-security lsp-bad "")
#             → packet spoof đi qua, ARP cache nạn nhân bị poison
# Lab task 3: bật lại, đếm drop counter trước/sau
```
Verify từng bước bằng `ovn-trace` và `ovs-ofctl dump-flows br-int table=8`.

**Lỗi thường gặp + tín hiệu chẩn đoán.**
- Mode A "port_security trống": VM vừa boot không gửi packet được → `ovn-nbctl lsp-get-port-security lsp-X` rỗng → fix `lsp-set-port-security`
- Mode B "MAC mismatch sau migration": drop counter table 8 tăng sau live migration → reconcile MAC virsh ↔ NBDB
- Mode C "REGBIT_PORT_SEC_DROP rỉ ra": IP spoof bị flag nhưng deferred drop điểm cuối không đặt → kiểm tra version OVN < 20.06 chưa có pattern này
- Mode D "stage compile chậm": LSP có >500 MAC trong port_security → northd nhả cảnh báo `Reduced flow set`, cần shard LSP

**So sánh với hệ khác.** Linux iptables `-m mac --mac-source` chains bridge-mode tương đương. Khác: iptables stateless thuần, OVN stage có deferred drop combine với IP/ND stage. Cisco port-security trên switch L2 (sticky MAC, max MAC count) tương tự ý tưởng nhưng OVN không có concept "max MAC count tự học" – list phải declare explicit. NSX-T spoofguard tương đương với "must declare MAC + IP", semantic giống port_security nhất.

## §13.19.2 LS_IN_PRE_ACL — chuẩn bị conntrack trước khi đánh giá ACL

**Khái niệm.** `LS_IN_PRE_ACL` là stage **số 5** trên LS_IN pipeline (logical table 5, OF table 13). Vai trò duy nhất: **set REGBIT_CONNTRACK_DEFRAG** và gọi `ct_next` để kéo packet IP qua module conntrack ở Linux kernel, từ đó populate `ct_state` cho stage `LS_IN_ACL` xài. Không có ACL stateful nào trên LS thì stage này **không** sinh flow priority cao – chỉ có flow `priority=0 action=next` skip qua. Loại: **pipeline stage cornerstone**, sub-loại "stateful preprocessor".

**Lịch sử + bối cảnh.** OVN 2.5 (2016) introduce conntrack integration cho ACL stateful, cùng lúc Pfaff đẩy `ct()` action vào OVS 2.5 upstream. Trước đó OVN ACL chỉ stateless (match L3/L4 đơn thuần, không phân biệt new/established). Stage `PRE_ACL` được tách riêng để cô lập điểm gọi conntrack: nếu engineer cần debug ACL drop, biết exact điểm nào kéo packet vào ct table thay vì rải `ct()` tự do trong nhiều stage.

**Vị trí trong kiến trúc.** Stage logic 5 ⇒ OF table 13. Đứng giữa `LS_IN_LOOKUP_FDB`/`LS_IN_PUT_FDB` (stage 3-4) và `LS_IN_PRE_LB` (stage 6). Scope: per-packet (cho mỗi packet IP), per-datapath (chỉ datapath có ACL stateful mới có flow non-trivial ở đây).

**Vai trò.** Bridge giữa OVS stateless OF abstraction và Linux netfilter stateful conntrack. Cụ thể: kéo packet vào conntrack zone (zone_id = datapath tunnel_key), retrieve state → populate `ct_state` register field, rồi `next` để stage downstream (LS_IN_ACL) match dựa trên ct_state. Phạm vi ảnh hưởng: ct_state chỉ valid trong **lifetime của packet** (sau pipeline finish, register clear). Conntrack entry tự nó persist qua nf_conntrack_max và timeout.

**Vì sao sinh ra.** Pain point tiền-OVN-2.5: ACL chỉ match L4 5-tuple → không phân biệt được "VM A start connect VM B" vs "VM B reply về VM A". Engineer phải viết 2 ACL đối xứng (allow A→B, allow B→A) và dễ sai. Với stateful, viết 1 ACL allow-related, OVN tự động cho reply qua. Để stateful work, phải có ct_state populated trước khi ACL match → đó là motivation tách ra stage PRE_ACL gọi ct_next sớm.

**Vấn đề giải quyết.** Cụ thể: làm sao kéo packet vào netfilter conntrack ở **đúng zone** trước khi ACL match. Cách giải: northd compile `pre_acls` ra Logical_Flow priority 110 match `(ip)` action `reg0[0] = 1; ct_next;`. `ct_next` là OVN action nội bộ map sang OVS action `ct(table=N, zone=NXM_NX_REG13[0..15])`, đẩy packet qua conntrack với zone = datapath tunnel_key (đảm bảo isolation giữa LS khác nhau). Phạm vi: chỉ packet IP (non-IP skip qua `next`).

**Tầm quan trọng.** Tier 1 cornerstone đối với LS có ACL stateful. Nếu LS không có ACL stateful nào (chỉ có ACL stateless hoặc không có), stage này về bản chất no-op (`priority=0 action=next`). Nhưng đối với production tenant network có firewall – đa số trường hợp – đây là điểm "must-know" để debug allow-related semantics.

**Cơ chế hoạt động.** Logical_Flow do `build_pre_acls()` trong `northd/northd.c` sinh khi `has_stateful_acl(od)` return true:

```
table=5 (ls_in_pre_acl), priority=110,
  match=(ip),
  action=(reg0[0] = 1; ct_next;)

table=5 (ls_in_pre_acl), priority=0,
  match=(1),
  action=(next;)
```

`ct_next` lúc compile thành OVS thật là `ct(table=14, zone=NXM_NX_REG13[0..15])` – packet đi recirc qua conntrack lookup. Zone = `reg13[0..15]` = datapath tunnel_key đảm bảo tenant A và tenant B dùng zone khác nhau, không leak state. Sau ct_next, packet quay lại pipeline ở **table 14** (= stage 6 + 8) với register `ct_state` đã populated. Invariant: sau stage này, mọi packet IP đều có ct_state valid (NEW/EST/REL/INV/...).

**Cách kỹ sư vận hành thành thạo.** (1) **Đọc**: biết stage chỉ populate ct_state, **không** tự drop bất kỳ packet nào. (2) **Sửa**: stage tự sinh; cách "sửa" là thêm/bớt ACL stateful trên LS qua `ovn-nbctl acl-add/acl-del`. (3) **Quan sát**: `ovs-appctl dpctl/dump-conntrack zone=<tunnel_key>` để xem entry sau khi packet qua. (4) **Debug**: nếu suspect ACL drop sai, reflex đầu tiên: ovn-trace, tìm ct_state ở stage downstream (ACL stage) – nếu ct_state không populate, gốc rễ ở PRE_ACL (zone không khớp, packet không phải IP, hoặc has_stateful_acl không trigger).

**Phân loại.** Loại stage: **stateful preprocessor** (subtype "ct_next invoker"). Đối lập với `LS_IN_PRE_LB` (stateful preprocessor cho LB) và `LS_OUT_PRE_ACL` (sibling cho egress ACL).

**Quy trình sử dụng.**
1. Tạo ACL stateful: `ovn-nbctl acl-add ls1 from-lport 1000 'ip4.src == 10.0.0.5' allow-related`
2. Verify northd sinh flow PRE_ACL: `ovn-sbctl lflow-list ls1 | grep "ls_in_pre_acl"` thấy priority 110 match `ip`
3. Trigger packet: từ VM 10.0.0.5 ping ra
4. Quan sát conntrack: `ovs-appctl dpctl/dump-conntrack zone=<dp_tunnel_key> | head`
5. Verify state đã set: `ovn-trace ls1 'inport == "lsp-vm1" && ip4 && ip4.src == 10.0.0.5 && ...'` xem ct_state ở stage ACL
6. Quy ước: nếu thay đổi ACL stateful → có thể cần `ovn-nbctl --wait=hv sync` để northd recompile

**Khi xảy ra sự cố.** Symptom: "ACL allow-related không cho reply qua". Diagnostic: ovn-trace, kiểm tra ct_state sau stage 5. Nếu ct_state empty → PRE_ACL không gọi (kiểm tra has_stateful_acl). Nếu ct_state có nhưng `inv` flag set → conntrack table thiếu entry (nf_conntrack_max thấp hoặc zone collision). Cross-link: Phần 9.24 §ct_state, Phần 20.2 §ovn-trace ACL debug.

**Liên quan mật thiết.** Tightly coupled: `LS_IN_ACL` (stage downstream tiêu thụ ct_state), `ct_state` match field, `ct_next` action, `REGBIT_CONNTRACK_DEFRAG` (cờ này), OVN ACL stateful (yếu tố trigger northd sinh flow). Loose: nf_conntrack_max sysctl, helper module ALG.

**Khác biệt giữa các phiên bản.**
- OVN 2.5 (2016): introduce stage cùng ACL stateful
- OVN 20.06 (2020): tách defrag (`ct_clear` + `ct(commit, zone)`) thành sub-step trong cùng stage
- OVN 21.03 (2021): hỗ trợ IPv6 fragmentation defrag
- OVN 22.03 (2022): tích hợp với LB hairpin (LB stateful cũng dùng PRE_ACL set up state)
- OVN 24.03 (2024): tách `LS_IN_ACL` thành `LS_IN_ACL_EVAL` + `LS_IN_ACL_ACTION` → PRE_ACL không đổi nhưng downstream khác

**Cách quan sát + xác minh.**


## §13.19.3 LS_IN_ACL — đánh giá và áp ACL stateful

**Khái niệm.** `LS_IN_ACL` là stage logic **số 9** (OF table 17) thực thi quyết định ACL trên ingress LS pipeline. Mỗi row của bảng NB `ACL` (gắn vào `Logical_Switch.acls` hoặc `Port_Group.acls`) được northd compile thành 1+ Logical_Flow trong stage này, match expression NB.match và action NB.action (`allow`, `allow-related`, `drop`, `reject`). Loại: **pipeline stage cornerstone**, sub-loại "policy enforcement".

**Lịch sử + bối cảnh.** OVN 2.4 (2015) introduce ACL stateless. OVN 2.5 (2016) thêm `allow-related` stateful. OVN 21.06 (2021) thêm column `ACL.tier` cho hierarchical priority, `meter` cho rate-limit log. OVN 22.03 (2022) tích hợp Conjunctive match cho Address_Set lớn (giảm cartesian explosion). OVN 24.03 (2024) tách thành `LS_IN_ACL_EVAL` + `LS_IN_ACL_ACTION` để cô lập match khỏi action (giúp tầng tier policy phức tạp hơn). Curriculum baseline OVN 22.03 vẫn dùng stage gộp.

**Vị trí trong kiến trúc.** Stage logic 9 ⇒ OF table 17. Đứng sau `LS_IN_PRE_ACL` (stage 5 set ct_state), `LS_IN_PRE_LB` (stage 6), `LS_IN_LOOKUP_FDB` (stage 3-4), `LS_IN_PRE_HAIRPIN` (stage 7-8). Sau ACL là `LS_IN_QOS_*`. Scope: per-packet per-datapath.

**Vai trò.** Trung tâm của firewall logic OVN. Mỗi tenant declare ACL → northd compile xuống đây → ovn-controller cài lên br-int. Phạm vi: 1 packet, dựa trên ct_state đã populate ở PRE_ACL + match L3/L4/L2 thuần.

**Vì sao sinh ra.** Trước OVN, NSX-V và Neutron (legacy) dùng iptables trên mỗi compute để filter VM-VM traffic. Cấu hình phân tán, drift, debug khó. OVN propose centralized ACL: declare ở NBDB tập trung, compile tự động xuống mọi hypervisor qua SBDB → uniform. ACL stage là điểm enforcement duy nhất; engineer chỉ cần biết "ACL gắn ở đâu" và "stage nào enforce" thay vì rải qua nhiều layer.

**Vấn đề giải quyết.** Cụ thể: implement firewall declarative trên overlay multi-tenant với 3 yêu cầu: (a) stateful (allow reply tự động), (b) high-cardinality match (1000+ source IP qua Address_Set), (c) priority-based override. Cách giải: kết hợp ct_state (populated ở PRE_ACL) với match expression NB.match được compile bởi `consider_acl()` thành flow priority = NB.priority + 1000 (offset cho deny default), với action sequence `ct(commit, zone=...)` cho allow-related, hoặc `drop`/`tcp_reset` cho deny.

**Tầm quan trọng.** Tier 1 cornerstone tuyệt đối. Mọi tenant network OVN có firewall đi qua đây. Nếu engineer không hiểu compile path từ NBDB ACL → Logical_Flow → OF flow trên br-int, không thể debug class lỗi "ACL allow rõ ràng nhưng packet vẫn drop".

**Cơ chế hoạt động.** `build_acls()` trong `northd/northd.c` (dòng ~6688 branch-22.03 per Phần 13.3) iterate datapath có ACL, gọi `consider_acl()` cho từng row. Mỗi ACL sinh template flow:

```
table=9 (ls_in_acl), priority=2002,
  match=(reg0[0] == 1 && (ip4.src == $set_a && tcp.dst == 80)),
  action=(reg0[1] = 1; ct_commit { ct_label.blocked = 0; }; next;)

table=9 (ls_in_acl), priority=1,
  match=(ip),
  action=(reg0[1] = 1; next;)

table=9 (ls_in_acl), priority=65535,
  match=(ct.inv || (ct.new && ct.est)),
  action=(drop;)
```

Priority encode: NB.priority + offset 1000 (dành 1000 priority thấp cho default-allow + reply-handling). Action `allow-related` thực ra là `ct_commit { ct_label.blocked = 0 } ; next` để conntrack track entry và stamp blocked=0 (sau này stage egress ACL có thể read). Stage cuối có 3 flow phụ: (1) reply path `ct.est && !ct.rpl` → next, (2) default deny invalid `ct.inv` → drop, (3) default `priority=0 → next` cho LS không có ACL stateful nào.

**Cách kỹ sư vận hành thành thạo.** (1) **Đọc**: hiểu priority encoding (priority NBDB + 1000), tier handling, conjunction action cho large Address_Set. (2) **Sửa**: `ovn-nbctl acl-add ls1 from-lport 1000 'match' action`. (3) **Quan sát**: `ovn-sbctl lflow-list ls1 | grep ls_in_acl` xem flow đã compile, `ovs-ofctl dump-flows br-int table=17 | grep "metadata=0x..."` xem flow physical. (4) **Debug**: ovn-trace packet → xem stage 9 match flow nào, action gì, ct_state có khớp expectation không.

**Phân loại.** Loại stage: **policy enforcement** (subtype "stateful firewall"). Khác với `LS_OUT_ACL` (egress sibling), `LS_IN_PRE_ACL` (state setup), `LS_IN_LB` (stateful LB).

**Quy trình sử dụng.**
1. Tạo Address_Set: `ovn-nbctl create Address_Set name=set_a addresses='"10.0.0.5","10.0.0.6"'`
2. Add ACL allow: `ovn-nbctl acl-add ls1 from-lport 2000 'ip4.src == $set_a && tcp.dst == 80' allow-related`
3. Add ACL deny default: `ovn-nbctl acl-add ls1 from-lport 1 'ip4' drop`
4. Verify compile: `ovn-sbctl lflow-list ls1 | grep ls_in_acl` thấy 2 flow priority 3000 + 1001
5. Test: từ VM trong set_a → reach VIP:80 OK; từ VM khác → drop
6. Audit log: bật `meter` trên ACL log → kiểm tra `journalctl -u ovn-controller | grep acl-log`

**Khi xảy ra sự cố.** Symptom: "ACL allow rõ nhưng packet vẫn drop". Diagnostic flow:
- Step 1: ovn-trace, xem stage 9 match flow nào → nếu drop, đọc reason (match ct.inv, hoặc default deny stage 65535)
- Step 2: nếu ct.inv, kiểm tra PRE_ACL/conntrack
- Step 3: nếu match NBDB ACL drop, kiểm tra priority encode (có ACL khác priority cao hơn không)

Cross-link: Phần 9.24 §ct_state, Phần 13.3 §build_acls, Phần 20.2 §ACL drop debug.

**Liên quan mật thiết.** Tightly coupled: NBDB `ACL` table, `Port_Group`, `Address_Set`, `LS_IN_PRE_ACL` (state source), `ct_commit` action, `REGBIT_ACL_HINT`, `meter` (log rate-limit), `LS_OUT_ACL` (egress sibling). Loose: iptables (lower layer), OpenStack Security Group (upper API).

**Khác biệt giữa các phiên bản.**
- OVN 2.4 (2015): stateless ACL only
- OVN 2.5 (2016): stateful allow-related
- OVN 21.06 (2021): tier column hierarchical priority, meter log rate-limit

## §13.19.4 LS_IN_LB — Load Balancer DNAT trên ingress LS

**Khái niệm.** `LS_IN_LB` là stage logic **số 12** (OF table 20) thực thi DNAT của Load_Balancer. Match VIP:port, chọn backend pool member theo hash, gọi `ct_lb` action commit conntrack entry với NAT mapping. Loại: **pipeline stage cornerstone**, sub-loại "L4 load balancer + DNAT".

**Lịch sử + bối cảnh.** OVN 2.4 (2015) introduce LB cơ bản với hash 5-tuple uniform. OVN 21.06 (2021) thêm distributed LB (chạy trên mọi hypervisor thay vì gateway centralize). OVN 22.03 (2022) thêm hairpin SNAT (VM client = VM backend cùng LS). OVN 24.03 (2024) thêm `Load_Balancer.options:affinity_timeout` cho session sticky.

**Vị trí trong kiến trúc.** Stage logic 12 ⇒ OF table 20. Đứng sau `LS_IN_PRE_LB` (stage 6, set REGBIT_CONNTRACK_NAT), `LS_IN_LB_AFF_*` (stage 10-11, sticky session check), trước `LS_IN_ACL_HINT` (stage 13). Scope: per-packet per-datapath.

**Vai trò.** Endpoint của LB DNAT path: chuyển dst IP từ VIP sang backend cụ thể. Conntrack entry commit ở đây để reply path biết ngược lại (UNDNAT ở egress). Phạm vi: lifetime của connection (entry tồn tại cho đến timeout).

**Vì sao sinh ra.** Trước OVN, OpenStack Octavia dùng HAProxy VM cho LB → 1 VM riêng, single-point-failure, latency cao. OVN propose distributed LB built-in pipeline: mọi hypervisor đều LB cho VIP của mình → loại bỏ choke point. Stage LB là điểm thực hiện trên mỗi hypervisor.

**Vấn đề giải quyết.** Cụ thể: thực hiện L4 load balancing với 3 yêu cầu: (a) consistent hash để cùng client luôn đến cùng backend (sticky), (b) commit conntrack entry để reply path ngược DNAT đúng, (c) hairpin support khi backend = source. Cách giải: `ct_lb` action với `hash_fields="ip_src,ip_dst,tp_src,tp_dst,ip_proto"` sinh hash, chọn backend theo `hash mod N_backend`, sau đó commit conntrack với `nat(dst=<backend>:<port>)`. Phạm vi: chỉ packet có ct.new (mới); reply path qua LR_OUT_UNSNAT/UNDNAT.

**Tầm quan trọng.** Tier 1 cornerstone cho cloud workload. Octavia OVN provider dùng stage này; mọi VIP service-publishing đi qua đây. Engineer cloud network must-know.

**Cơ chế hoạt động.** `build_lb_rules()` trong `northd/northd.c` iterate Load_Balancer references trên LS, sinh per-VIP flow:

```
table=12 (ls_in_lb), priority=110,
  match=(ct.new && ip4 && ip4.dst == 10.0.0.100 && tcp.dst == 80),
  action=(reg0[2] = 1;
          ct_lb(backends=192.168.1.5:80,192.168.1.6:80,
                hash_fields="ip_src,tp_src");)

table=12 (ls_in_lb), priority=0,
  match=(1),
  action=(next;)
```

`ct_lb` lúc compile thành OVS action `ct(commit, table=N, zone=..., nat(dst=backend))`. Hash_fields điều khiển consistent hashing: thường 5-tuple để mỗi connection sticky đến một backend. Nếu Load_Balancer.selection_fields set khác (e.g., chỉ `ip_src`), hash chỉ trên field đó (sticky theo source IP). Invariant: cùng (ip_src, tp_src) qua cùng VIP luôn đến cùng backend cho đến khi backend list thay đổi.

**Cách kỹ sư vận hành thành thạo.** (1) **Đọc**: hiểu hash_fields, sticky semantics. (2) **Sửa**: `ovn-nbctl lb-add lb-web 10.0.0.100:80 192.168.1.5:80,192.168.1.6:80` rồi `ovn-nbctl ls-lb-add ls1 lb-web`. (3) **Quan sát**: `ovs-appctl dpctl/dump-conntrack zone=<dp> | grep <vip>` xem entry NAT. (4) **Debug**: `ovn-trace` packet đến VIP, xem stage 12 chọn backend nào.

**Phân loại.** Loại stage: **L4 LB + DNAT** (subtype "stateful balanced DNAT"). Khác: `LR_IN_DNAT` (north-south DNAT, không có hash), `LS_IN_NAT_HAIRPIN` (hairpin SNAT khi self-LB).

**Quy trình sử dụng.**
1. Tạo LB: `ovn-nbctl lb-add lb-web "10.0.0.100:80" "192.168.1.5:80,192.168.1.6:80"`
2. Gắn LB vào LS: `ovn-nbctl ls-lb-add ls1 lb-web`
3. (Optional) Set sticky: `ovn-nbctl set Load_Balancer lb-web selection_fields=ip_src`
4. Verify compile: `ovn-sbctl lflow-list ls1 | grep ls_in_lb` thấy flow priority 110
5. Test: từ client send 5 request đến VIP → đếm số request đi đến mỗi backend (ratio nên gần 50/50 nếu uniform)
6. Sticky test: cùng client (cùng ip_src) → 100% đến cùng 1 backend

**Khi xảy ra sự cố.** Symptom: "VIP unreachable cho new connection sau khi cluster scale". Diagnostic:
- Step 1: ovn-trace packet client → VIP, xem stage 12 match flow nào
- Step 2: nếu không match priority 110 (không có flow), kiểm tra `ovn-nbctl ls-lb-list ls1` xem LB có gắn không
- Step 3: nếu match nhưng backend chosen unhealthy, kiểm tra Service_Monitor health status
- Step 4: nếu conntrack table near-full, NAT fail → ct.new packet drop

```bash
# Lab task 1: từ client gửi 100 TCP request đến VIP → đếm hit ratio backend
# Lab task 2: set selection_fields=ip_src → verify cùng client luôn đến cùng backend
# Lab task 3: stop be1 → Service_Monitor down → verify chuyển 100% sang be2
# Lab task 4: hairpin test (be1 truy cập VIP của chính nó) → quan sát NAT_HAIRPIN
```

**Lỗi thường gặp + tín hiệu chẩn đoán.**
- Mode A "conntrack table full": `nf_conntrack_count / nf_conntrack_max > 95%` → drop new
- Mode B "backend unhealthy nhưng vẫn nhận packet": Service_Monitor disable hoặc interval quá chậm → enable + giảm interval
- Mode C "sticky session không sticky": selection_fields default 5-tuple → mỗi src_port khác → đến backend khác → set selection_fields=ip_src
- Mode D "hairpin loop": backend = client cùng LS, không bật `Load_Balancer.options:hairpin_snat_ip` → packet loop → bật option

**So sánh với hệ khác.** Linux IPVS `ipvsadm` direct analog: cũng L4 LB với hash, NAT mode commit conntrack entry. Khác: OVN distributed mỗi hypervisor, IPVS centralized. AWS NLB tương đương semantic (L4, consistent hash, sticky), nhưng managed service. HAProxy L4 mode tương tự nhưng userspace (chậm hơn). Cisco ACE / F5 BIG-IP enterprise-grade L4-L7.

## §13.19.5 LS_IN_DHCP_OPTIONS — phản hồi DHCPDISCOVER/REQUEST native trong pipeline

**Khái niệm.** `LS_IN_DHCP_OPTIONS` là stage logic **số 19** (OF table 27) thực thi DHCP server trực tiếp trong pipeline OVN, không cần daemon dnsmasq external. Match DHCPDISCOVER/DHCPREQUEST UDP packet, gọi action `put_dhcp_opts` để build DHCPOFFER/DHCPACK reply trực tiếp trong kernel datapath. Loại: **pipeline stage cornerstone**, sub-loại "DHCP server inline".

**Lịch sử + bối cảnh.** OVN 2.6 (2016) introduce native DHCPv4 inline (Pfaff + Reddy propose để bỏ phụ thuộc dnsmasq). OVN 2.8 (2017) thêm DHCPv6. OVN 22.03 (2022) hỗ trợ option 121 (classless static route), option 119 (domain search list). OVN 24.03 (2024) tích hợp với IPAM cho dynamic IP allocation.

**Vị trí trong kiến trúc.** Stage logic 19 ⇒ OF table 27. Đứng sau ACL stage (9), LB stage (12), trước ARP_RESPONDER (stage 20+) và L2 lookup cuối (stage 27). Scope: per-packet (chỉ DHCP packet), per-LSP (mỗi LSP có thể trỏ DHCP_Options khác nhau).

**Vai trò.** Trả lời DHCP request từ VM, gán IP + gateway + DNS + lease time mà không cần thêm component external. Phạm vi: 1 packet (DHCPDISCOVER/REQUEST), generate reply ngay trong kernel datapath.

**Vì sao sinh ra.** Pre-OVN-2.6, OpenStack Neutron deploy dnsmasq trên network node cho mỗi tenant network → quản lý phức tạp (mỗi tenant một process), single-point-failure trên node gateway, latency cao (packet phải đi qua gateway). OVN propose inline DHCP: mỗi hypervisor tự reply cho VM của mình → loại bỏ network node bottleneck cho DHCP traffic.

**Vấn đề giải quyết.** Cụ thể: cung cấp DHCP service cho VM mà không cần process external. Cách giải: northd compile từ NB `DHCP_Options` (subnet, server_id, gateway, DNS, lease_time, classless_static_route) thành Logical_Flow match DHCPDISCOVER/REQUEST với action `put_dhcp_opts(...)`. ovn-controller dịch xuống OVS extended action thực hiện trực tiếp trong kernel datapath: replace DHCP payload thành DHCPOFFER/DHCPACK, swap src/dst MAC + IP để packet quay ngược về client. Phạm vi: full DHCP server ngoại trừ DHCPv6 SLAAC (RA, không qua stage này).

**Tầm quan trọng.** Tier 1 cornerstone. Mọi VM bootstrap đi qua đây. Nếu stage broken, VM không có IP → không boot được. Engineer must-know cho on-call cloud network.

**Cơ chế hoạt động.** `build_lswitch_dhcp_options_and_response()` trong `northd/northd.c` iterate LSP có `dhcpv4_options` set, sinh per-LSP flow:

```
table=19 (ls_in_dhcp_options), priority=100,
  match=(inport == "lsp-vm1" && eth.src == AA:BB:CC:DD:EE:01 &&
         ip4.src == 0.0.0.0 && ip4.dst == 255.255.255.255 &&
         udp.src == 68 && udp.dst == 67),
  action=(reg0[3] = put_dhcp_opts(offerip = 10.0.0.5,
                                   netmask = 255.255.255.0,
                                   router = 10.0.0.1,
                                   server_id = 10.0.0.1,
                                   lease_time = 3600,
                                   dns_server = {8.8.8.8});
          next;)

table=19 (ls_in_dhcp_options), priority=0,
  match=(1),
  action=(next;)
```

Action `put_dhcp_opts` là OVN extended action (không phải OF chuẩn). ovn-controller compile xuống một sequence OVS action: `controller(userspace)` callback xử lý userspace, nhưng từ OVS 2.7 có path optimized trong kernel. `reg0[3]` = 1 nếu thành công, 0 nếu fail (offerip ngoài subnet). Stage tiếp theo (`LS_IN_DHCP_RESPONSE` table 20) check reg0[3] và emit reply. Invariant: DHCP packet nếu pass stage này thành công sẽ có DHCPOFFER/ACK packet thay thế trong pipeline downstream.

**Cách kỹ sư vận hành thành thạo.** (1) **Đọc**: hiểu DHCP_Options column structure (subnet, server_id, gateway, DNS, lease_time, classless_static_route). (2) **Sửa**: `ovn-nbctl dhcp-options-create 10.0.0.0/24` rồi `ovn-nbctl dhcp-options-set-options <uuid> server_id=10.0.0.1 lease_time=3600 router=10.0.0.1 dns_server='{8.8.8.8}'`, gắn vào LSP qua `ovn-nbctl lsp-set-dhcpv4-options lsp1 <dhcp_uuid>`. (3) **Quan sát**: tcpdump trên tap interface, hoặc `ovs-ofctl dump-flows br-int table=27 | grep put_dhcp_opts`. (4) **Debug**: nếu VM stuck ở DHCP, ovn-trace giả lập DHCPDISCOVER, xem stage 19 có match không.

**Phân loại.** Loại stage: **DHCP server inline** (subtype "DHCPv4 native"). Khác: `LS_IN_DHCP_RESPONSE` (stage 20, emit reply), `LS_IN_DNS_LOOKUP` (stage tương tự cho DNS), `LR_IN_IP_INPUT` (xử lý packet đến LR self).

**Quy trình sử dụng.**
1. Tạo DHCP_Options: `dhcp_uuid=$(ovn-nbctl create DHCP_Options cidr=10.0.0.0/24 options='"server_id"="10.0.0.1" "lease_time"="3600" "router"="10.0.0.1"')`
2. Tạo LS + LSP: `ovn-nbctl ls-add ls1 && ovn-nbctl lsp-add ls1 lsp-vm1`
3. Set address dynamic: `ovn-nbctl lsp-set-addresses lsp-vm1 "AA:BB:CC:DD:EE:01 dynamic"`
4. Gắn DHCP options: `ovn-nbctl lsp-set-dhcpv4-options lsp-vm1 $dhcp_uuid`
5. Verify compile: `ovn-sbctl lflow-list ls1 | grep ls_in_dhcp_options`
6. VM boot → DHCPDISCOVER → DHCPOFFER 10.0.0.5 → DHCPREQUEST → DHCPACK
7. Verify lease: từ VM kiểm tra IP đã nhận

**Khi xảy ra sự cố.** Symptom: "VM stuck at DHCP, không nhận IP". Diagnostic flow:
- Step 1: tcpdump tap interface VM → có DHCPDISCOVER không (nếu không, vấn đề ở VM)
- Step 2: `ovn-sbctl lflow-list ls1 | grep ls_in_dhcp_options` → có flow priority 100 không
- Step 3: nếu không có, kiểm tra `ovn-nbctl lsp-get-dhcpv4-options lsp-vm1`
- Step 4: nếu có, ovn-trace giả lập packet, xem stage 19 match → action put_dhcp_opts
- Step 5: nếu offerip out-of-subnet, kiểm tra DHCP_Options.cidr vs LSP IP

Cross-link: Phần 13.10 §DHCP native, Phần 20.0 §VM bootstrap troubleshoot.

**Liên quan mật thiết.** Tightly coupled: NBDB `DHCP_Options` table, NBDB `Logical_Switch_Port.dhcpv4_options`, `LS_IN_DHCP_RESPONSE` (stage 20 sibling emit reply), `put_dhcp_opts` action, `REGBIT_DHCP_OPTS_RESULT`. Loose: dnsmasq (alternative external), cloud-init (consume IP).

**Khác biệt giữa các phiên bản.**
- OVN 2.6 (2016): introduce DHCPv4 native
- OVN 2.8 (2017): thêm DHCPv6
- OVN 21.06 (2021): IPAM tự gán IP từ subnet
- OVN 22.03 (2022): option 121 classless static route, option 119 domain search list
- OVN 23.06 (2023): hỗ trợ Boot Options (PXE)
- OVN 24.03 (2024): IPAM tích hợp với DHCP_Options auto-track

- Mode B "subnet mismatch": offerip ngoài cidr → put_dhcp_opts fail → đối chiếu cidr

## §13.19.6 LS_IN_L2_LKUP — quyết định outport L2 cuối cùng của ingress LS

**Khái niệm.** `LS_IN_L2_LKUP` là stage logic **số 24** (OF table 32) — **stage cuối** của pipeline LS_IN. Trách nhiệm duy nhất: match `eth.dst` → set `outport` (tức `MFF_LOG_OUTPORT`, register reg15 trong OVS) chỉ tới Logical_Switch_Port đích. Sau khi stage này set xong outport, packet rời nhánh ingress LS qua action `output;` để vào pipeline egress (`LS_OUT_PRE_ACL` → `LS_OUT_DELIVERY`). Loại: **pipeline stage cornerstone**, sub-loại "L2 destination lookup + forwarding decision".

**Lịch sử + bối cảnh.** Stage có mặt từ OVN 2.4 (2015) như là core của pipeline LS — bản chất chính là chức năng MAC learning của một switch. OVN 20.06 (2020) thêm cơ chế "known dst flood" (flood kiểm soát chỉ tới các LSP có `addresses="unknown"`) thay vì flood toàn bộ. OVN 22.03 (2022) thêm bảng FDB `MAC_Binding`-style cho LSP type "vtep" và `localnet` để giảm broadcast leak. OVN 24.03 (2024) tích hợp với cờ `MLF_RCV_FROM_RAMP` để tránh hairpin loop khi packet đến từ ramp switch (RFC ramp).

**Vị trí trong kiến trúc.** Stage logic 24 ⇒ OF table 32. Đứng ngay sau `LS_IN_HAIRPIN` (logical 23) và trước nhánh egress qua action `output;` (chuyển sang OF table 33+ cho LS_OUT). Trên br-int của hypervisor, stage này là **điểm chuyển trọng tâm**: trước nó pipeline làm "policy" (security, ACL, LB, ARP responder); sau nó pipeline làm "delivery". Scope: per-packet, per-LS datapath. Không có stage nào sau nó còn quyền đổi outport (trừ trường hợp đặc biệt egress hairpin).

**Vai trò.** Vai trò chính: **forwarding decision** trên L2 — tương đương MAC table của một bridge truyền thống. Vai trò phụ: **broadcast/multicast distributor** (khi `eth.dst` là `ff:ff:ff:ff:ff:ff` hoặc multicast MAC, set outport thành tập multicast `_MC_flood`). Phạm vi ảnh hưởng: 1 packet, ghi giá trị reg15 sống đến cuối pipeline egress.

**Vì sao sinh ra.** Một switch ảo bắt buộc phải có L2 lookup. Trước OVN, OpenStack Neutron-OVS dùng learn-action OpenFlow + flood-learn pattern (chậm, race condition). OVN dịch sang model **declarative**: thay vì học MAC từ traffic, NBDB liệt kê tường minh `Logical_Switch_Port.addresses`, northd compile thẳng thành flow priority 50. Điểm mạnh: deterministic, không có MAC table aging race; điểm yếu: tenant phải đăng ký MAC trước qua API, không "tự học".

**Vấn đề giải quyết.** Cụ thể: chuyển từ `eth.dst` sang outport. Cách giải: hàm `build_lswitch_destination_lookup_bmcast` + `build_lswitch_arp_nd_responder_known_ips` (`northd/northd.c`) iterate mọi LSP của LS, sinh flow priority 50 match `eth.dst == <LSP_MAC>` action `outport = <LSP_NAME>; output;`. Cho broadcast/multicast, sinh flow priority 70 match `eth.mcast` action `outport = "_MC_flood"; output;`. Cho unknown unicast (không match LSP nào), priority 0 fallback flood tới các LSP có `addresses="unknown"` — thường chỉ `localnet` port chuyển ra ngoài physical network. Phạm vi giải quyết: chỉ L2 forwarding; không xử lý conntrack (đã xong ở stage trước), không xử lý L3 (LR có pipeline riêng).

**Tầm quan trọng.** Tier 1 cornerstone tuyệt đối. Mọi packet ingress LS đều đi qua đây; nếu stage compile sai (LSP MAC trống, MC group sai), tenant network "câm" hoàn toàn. Engineer must-know: hiểu `MFF_LOG_OUTPORT` chính là reg15 (xem 13.18), biết phân biệt 3 nhánh priority (50 unicast, 70 mcast, 0 unknown), biết FDB table fallback từ OVN 22.03+. Không nắm stage này thì không debug được class lỗi "tenant ping được tới subnet broadcast nhưng unicast fail".

**Cơ chế hoạt động.** Logical_Flow tiêu biểu cho LS có 2 LSP `lsp-vm1` (MAC AA:01) và `lsp-vm2` (MAC AA:02):

```
table=24 (ls_in_l2_lkup), priority=50,
  match=(eth.dst == AA:AA:AA:AA:AA:01),
  action=(outport = "lsp-vm1"; output;)

table=24 (ls_in_l2_lkup), priority=50,
  match=(eth.dst == AA:AA:AA:AA:AA:02),
  action=(outport = "lsp-vm2"; output;)

table=24 (ls_in_l2_lkup), priority=70,
  match=(eth.mcast),
  action=(outport = "_MC_flood"; output;)

table=24 (ls_in_l2_lkup), priority=0,
  match=(1),
  action=(outport = "_MC_unknown"; output;)
```

Action `output;` không phải OF action `output:N` thuần — nó là OVN macro nhả về stage egress đầu (`LS_OUT_PRE_LB` cho unicast, hoặc clone packet cho multicast group). `_MC_flood` là multicast group OVS chứa tunnel_key của mọi LSP trên LS đó; ovn-controller install group entry tự động. Invariant: sau stage này, reg15 luôn có giá trị; nếu reg15 = 0xfffe (`_MC_unknown`) packet sẽ được flood tới tập LSP `unknown`.

**Cách kỹ sư vận hành thành thạo.** Bốn kỹ năng: (1) **Đọc**: `ovn-sbctl lflow-list ls1 | grep ls_in_l2_lkup` xem flow compile; `ovs-ofctl dump-flows br-int table=32 | grep "metadata=0x<dp>"` xem flow đã install. (2) **Sửa**: thêm/sửa MAC qua `ovn-nbctl lsp-set-addresses lsp1 "<new_mac> <ip>"`; sau đó `ovn-nbctl --wait=hv sync`. (3) **Quan sát**: `ovn-trace ls1 'inport=="lsp1" && eth.dst==<dst_mac>'` thấy chính xác stage 24 chọn outport nào. (4) **Debug**: nếu nghi MAC mismatch, dump conntrack + reg15 qua `ofproto/trace`.

**Phân loại.** Loại stage: **L2 forwarding decision** (subtype "destination lookup"). Khác `LS_OUT_DELIVERY` (chỉ output thuần, không lookup nữa) và `LR_IN_IP_ROUTING` (L3 lookup, không phải L2). Là **terminal decision stage** của ingress LS — không stage nào sau nó còn re-evaluate outport.

**Quy trình sử dụng.** Khi cấu hình LS bình thường:
1. `ovn-nbctl ls-add ls1`
2. `ovn-nbctl lsp-add ls1 lsp-vm1`
3. `ovn-nbctl lsp-set-addresses lsp-vm1 "AA:AA:AA:AA:AA:01 10.0.0.5"` (MAC + IP declare)
4. Verify compile: `ovn-sbctl lflow-list ls1 | grep ls_in_l2_lkup` thấy flow priority 50
5. Verify install: `ovs-ofctl dump-flows br-int table=32 | grep AA:AA` thấy rule
6. Test ping giữa lsp-vm1 ↔ lsp-vm2; `ovn-trace` xác nhận stage 24 set reg15 đúng

**Khi xảy ra sự cố.** Symptom điển hình: (a) "VM ping broadcast được nhưng unicast fail" → ngay lập tức nghi MAC chưa khớp giữa NBDB và VM thật → check `ovn-nbctl lsp-get-addresses` so với `ip link show` trong VM. (b) "Tenant network im lặng sau scale-out" → kiểm tra số flow priority 50 trong table 32: nếu thiếu LSP nào nghĩa là northd compile chậm/lỗi. (c) "Broadcast flood quá tải link uplink" → xem priority 70 có chuyển hết về `localnet` không (cần segmenting MC group). Cross-link: Phần 13.2 §Logical_Switch lookup, Phần 13.16 §logical→OF table mapping.

**Liên quan mật thiết.** Tightly coupled: `MFF_LOG_OUTPORT` (đích set của stage), `_MC_flood` + `_MC_unknown` multicast group, `Logical_Switch_Port.addresses` (NBDB nguồn dữ liệu), `LS_IN_HAIRPIN` (stage liền trước), `LS_OUT_PRE_LB` / `LS_OUT_DELIVERY` (stage liền sau qua action output). Loose: ARP cache trong VM (layer trên), Linux bridge FDB (alternative không qua OVN).

**Khác biệt giữa các phiên bản.**
- OVN 2.4 (2015): chỉ unicast + flood, không có MC group concept rõ ràng
- OVN 20.06 (2020): tách `_MC_flood` thành flood "known" + "unknown" để giảm broadcast tới tập tối thiểu
- OVN 22.03 (2022): introduce FDB table cho LSP type "vtep" + "localnet" để track MAC từ external
- OVN 23.06 (2023): tối ưu compile khi LSP > 1000 (giảm số flow do chia shard logical OFTABLE)
- OVN 24.03 (2024): integrate `MLF_RCV_FROM_RAMP` để chặn hairpin loop từ ramp switch path

## §13.19.7 LR_IN_ADMISSION — cổng vào ingress của Logical_Router

**Khái niệm.** `LR_IN_ADMISSION` là stage logic **số 0** của pipeline ingress trên một Logical_Router (OF table 8 — chia sẻ với `LS_IN_PORT_SEC_L2`, phân biệt nhờ `metadata` chứa datapath ID của LR). Vai trò: **kiểm soát admission** ở biên router. Drop packet không hợp lệ ở L2 cho LR (sai dst MAC, source loopback/multicast trên field IP source, packet đến từ port không phải LRP). Loại: **pipeline stage cornerstone**, sub-loại "L3 admission gate".

**Lịch sử + bối cảnh.** OVN 2.5 (2015) introduce Logical_Router. Stage `LR_IN_ADMISSION` là cổng đầu tiên ngay từ phiên bản đầu — semantic mượn từ Linux router behavior (kernel drop packet có dst MAC khác router MAC tại NIC ingress). OVN 20.06 (2020) thêm xử lý multicast MAC (nếu LRP join multicast group). OVN 22.03 (2022) thêm cờ `REGBIT_PKT_LARGER` để track packet size cần fragment. OVN 23.06 (2023) integrate với `Logical_Router_Port.options:gateway_mtu` để check MTU sớm.

**Vị trí trong kiến trúc.** Stage logic 0 của LR ⇒ OF table 8 (cùng table nhưng metadata khác với LS). Trước stage này chỉ có OF table 0 (port → datapath mapping). Sau stage là `LR_IN_LOOKUP_NEIGHBOR` (logical 1). Scope: per-packet, per-LRP. Chính xác hơn: chỉ packet vào LR qua một LRP "chính chủ" (eth.dst trùng MAC LRP, hoặc multicast/broadcast cho ARP) mới qua stage này; sai → drop.

**Vai trò.** Vai trò chính: **gatekeeper biên router** — bảo đảm chỉ packet đích router mới đi vào pipeline L3. Vai trò phụ: **drop sink** cho packet sai biên (LRP MAC mismatch, loopback src, all-zero src). Nếu disable stage này, mọi packet đi qua tap LR sẽ vào pipeline L3 ngay cả khi VM bên kia LS vô tình broadcast — gây xử lý L3 không cần thiết, có thể trigger ICMP error sai.

**Vì sao sinh ra.** Trước OVN, OpenStack Neutron-OVS dùng router namespace + iptables để filter packet biên router. Cách đó tốn namespace, slow path qua kernel routing. OVN abstract thẳng vào pipeline OpenFlow: stage này cài flow matching `inport == "lrp1" && eth.dst == <LRP_MAC>` action `next;`, default `drop;`. Loại bỏ namespace, fast path qua datapath. Pain point cũ: namespace router fail-over chậm; OVN giải quyết qua stage stateless trong mỗi hypervisor.

**Vấn đề giải quyết.** Cụ thể: lọc packet biên router để chỉ cho packet hợp lệ vào L3 pipeline. Cách giải: hàm `build_adm_ctrl_flows_for_lrouter` (`northd/northd.c`) iterate mọi `Logical_Router_Port` của LR, sinh flow priority 50 match `inport == "lrp-X" && (eth.dst == <LRP_MAC> || eth.mcast)` action `next;`. Default priority 0 drop. Cho IPv6, thêm flow match multicast IPv6 (FF::/8). Phạm vi: chỉ admission L2/L3-source check; routing chính do stage `LR_IN_IP_ROUTING` (logical 9) phụ trách.

**Tầm quan trọng.** Tier 1 cornerstone. Mọi packet vào LR đều qua đây — nếu sai compile, router "câm" hoàn toàn (mọi VM đông-tây qua LR fail). Engineer must-know: hiểu rằng stage này drop packet "lạc đường" (đến tap LR nhưng dst MAC không phải LRP) thay vì forward; hiểu rằng broadcast `ff:ff:ff:ff:ff:ff` được pass để stage sau xử lý ARP. Không nắm stage này thì class lỗi "VM không ping được gateway" sẽ debug sai chỗ.

**Cơ chế hoạt động.** Flow tiêu biểu cho LR có 2 LRP `lrp1` (MAC RR:01) và `lrp2` (MAC RR:02):

```
table=0 (lr_in_admission), priority=50,
  match=(inport == "lrp1" && (eth.dst == RR:RR:RR:RR:RR:01 || eth.mcast)),
  action=(xreg0[0..47] = RR:RR:RR:RR:RR:01;  /* ghi src MAC sẽ dùng cho egress */
          next;)

table=0 (lr_in_admission), priority=50,
  match=(inport == "lrp2" && (eth.dst == RR:RR:RR:RR:RR:02 || eth.mcast)),
  action=(xreg0[0..47] = RR:RR:RR:RR:RR:02; next;)

table=0 (lr_in_admission), priority=100,
  match=(ip4.src == 127.0.0.0/8 || ip4.src == 224.0.0.0/4 || ip4.src == 0.0.0.0),
  action=(drop;)  /* martian source */

table=0 (lr_in_admission), priority=0,
  match=(1),
  action=(drop;)
```

Action `xreg0[0..47] = <LRP_MAC>` ghi MAC nguồn router để egress dùng làm `eth.src` (tương tự kernel router rewrite L2 header khi forward). Invariant: sau stage, nếu packet còn sống (`next;`), nó **chắc chắn** đến đúng LRP của router; xreg0 đã có MAC để stage egress dùng.

**Cách kỹ sư vận hành thành thạo.** (1) **Đọc**: `ovn-sbctl lflow-list lr1 | grep lr_in_admission` xem 2 nhánh priority 50 + 100 + 0. (2) **Sửa**: thêm/sửa LRP MAC qua `ovn-nbctl lrp-add lr1 lrp1 RR:01 10.0.0.1/24`. (3) **Quan sát**: `ovn-trace lr1 'inport=="lrp1" && eth.dst==RR:01 && ...'` thấy stage 0 match priority 50 next. (4) **Debug**: nếu packet drop ngay biên router, đếm `ovs-ofctl dump-flows br-int table=8 | grep "metadata=0x<lr_dp>" | grep "n_packets"` xem flow priority 0 fallback (drop) có tăng không.

**Phân loại.** Loại stage: **L3 admission gate** (subtype "border filter"). Khác `LS_IN_PORT_SEC_L2` (lọc spoof L2 trên LS, không phải biên L3) và `LR_IN_IP_INPUT` (xử lý packet đích router self, sau stage này). Stateless thuần.

**Quy trình sử dụng.**
1. `ovn-nbctl lr-add lr1`
2. `ovn-nbctl lrp-add lr1 lrp1 RR:RR:RR:RR:RR:01 10.0.0.1/24`
3. Gắn LRP vào LS: `ovn-nbctl lsp-add ls1 ls1-to-lr1 && ovn-nbctl lsp-set-type ls1-to-lr1 router && ovn-nbctl lsp-set-options ls1-to-lr1 router-port=lrp1 && ovn-nbctl lsp-set-addresses ls1-to-lr1 router`
4. Verify: `ovn-sbctl lflow-list lr1 | grep lr_in_admission` thấy flow priority 50
5. Test admission: từ VM bên LS gửi packet tới gateway IP → ovn-trace verify stage 0 match
6. Test drop: tự gửi packet với eth.dst = MAC bất kỳ khác RR:01 → drop counter tăng

**Khi xảy ra sự cố.** Symptom: (a) "VM không ping được gateway 10.0.0.1" → check stage 0 trong table 8 metadata LR, đặc biệt drop counter priority 0; nếu tăng có thể LRP MAC đổi mà VM ARP cache stale. (b) "Sau migrate gateway chassis, packet đi N-S fail vài giây" → stage này không phải nguyên nhân chính (xem `LR_IN_GW_REDIRECT`) nhưng kiểm tra để loại trừ. (c) "Martian source spam" → priority 100 drop count tăng → có host tenant gửi packet với source 0.0.0.0/loopback → cần bật alert.

**Liên quan mật thiết.** Tightly coupled: `Logical_Router_Port.mac` (NBDB nguồn dữ liệu MAC), `Logical_Router_Port.networks` (subnet declare), `xreg0` (register lưu LRP MAC cho egress sau), `LR_IN_LOOKUP_NEIGHBOR` (stage liền sau). Loose: ARP cache trong VM (layer trên), Linux kernel rp_filter (alternative stack ngoài OVN).

**Khác biệt giữa các phiên bản.**
- OVN 2.5 (2015): chỉ unicast match MAC; không multicast nhánh
- OVN 20.06 (2020): thêm multicast MAC pass-through cho ARP/ND
- OVN 22.03 (2022): integrate `REGBIT_PKT_LARGER` flag cho MTU check sau
- OVN 23.06 (2023): `gateway_mtu` option cho phép check MTU sớm tại stage này
- OVN 24.03 (2024): tối ưu compile khi LR có > 50 LRP (chia shard)

## §13.19.8 LR_IN_IP_INPUT — xử lý packet hướng tới chính router

**Khái niệm.** `LR_IN_IP_INPUT` là stage logic **số 3** (OF table 11) — nơi xử lý packet **hướng tới chính router** (`ip.dst == <LRP_IP>`), thay vì packet quá cảnh router (forwarded). Xử lý gồm: trả lời ICMP echo (ping router), gửi ICMP TTL-exceeded, ICMP destination-unreachable cho fragment-needed (PMTUD), trả lời ARP/ND cho LRP IP, gửi BFD echo, và (tùy version) một phần routing protocol nhỏ. Loại: **pipeline stage cornerstone**, sub-loại "router self-traffic handler".

**Lịch sử + bối cảnh.** OVN 2.5 (2015) introduce Logical_Router với khả năng trả ICMP echo cơ bản. OVN 20.06 (2020) thêm fragment-needed (PMTUD) với action `icmp4_error`. OVN 21.03 (2021) thêm ARP/ND responder tích hợp tại stage này thay vì stage riêng (consolidate). OVN 22.03 (2022) tích hợp BFD echo native trong pipeline (trước đó BFD cần punt ra ovn-controller userspace). OVN 23.06 (2023) thêm hỗ trợ ICMP rate-limit để chống ICMP flood.

**Vị trí trong kiến trúc.** Stage logic 3 ⇒ OF table 11. Đứng sau `LR_IN_LOOKUP_NEIGHBOR` (logical 1) + `LR_IN_LEARN_NEIGHBOR` (logical 2), trước `LR_IN_DEFRAG` (logical 4). Scope: per-packet, chỉ packet có `eth.dst == <LRP_MAC> && ip.dst == <LRP_IP>` mới match priority cao tại đây; packet khác fall through priority 0 next sang stage routing.

**Vai trò.** Vai trò chính: **router self-service** — phản hồi tất cả packet đích đến router (ICMP, ARP cho LRP IP, BFD echo). Vai trò phụ: **error generator** — sinh ICMP error (TTL-exceeded từ stage routing trả về, destination-unreachable cho fragment-needed). Phạm vi ảnh hưởng: per-packet; có thể sinh thêm packet mới (reply) qua `clone` action.

**Vì sao sinh ra.** Linux router xử lý packet self-destination qua netfilter INPUT chain → kernel ICMP handler. OVN cần làm tương đương trong pipeline data plane để: (a) tránh punt mọi ping packet ra userspace (tốn kém), (b) đảm bảo PMTUD nhanh để tenant TCP không stuck với MTU sai, (c) BFD echo cần response trong < 1ms để health-check không false-down. Pain point cũ (Neutron-OVS + namespace router): mọi self-traffic phải qua kernel chain → slow. OVN inline trong pipeline → fast path datapath.

**Vấn đề giải quyết.** Cụ thể: trả lời packet đích router mà không cần punt ra userspace. Cách giải: hàm `build_lrouter_ipv4_ip_input` + `build_lrouter_ipv6_ip_input` (`northd/northd.c`) iterate mọi LRP, sinh flow priority cao match từng IP của LRP cho từng loại request:

- ICMP echo request → `clone { eth.src/dst swap; ip.src/dst swap; icmp4.type = 0; next; }` → reply ngay
- TTL = 1 → `icmp4_error` action sinh ICMP time-exceeded
- Fragment-needed → `icmp4_error` với type=3 code=4 + next-hop MTU
- ARP request có `arp.tpa == <LRP_IP>` → trả ARP reply với LRP MAC
- BFD echo destination = LRP IP → respond trong pipeline

Phạm vi giải quyết: chỉ self-traffic; forwarded traffic đi tiếp sang `LR_IN_DEFRAG` → `LR_IN_IP_ROUTING`.

**Tầm quan trọng.** Tier 1 cornerstone. Health check (ping gateway) phụ thuộc stage này; PMTUD (fragment-needed) phụ thuộc stage này; BFD chassis health-check phụ thuộc stage này. Engineer must-know: ICMP echo trả lời tự động, không phải Linux kernel; PMTUD-ICMP sinh ngay tại pipeline; BFD echo path khác BFD control session (control vẫn punt). Không nắm thì class lỗi "ping gateway 100% loss nhưng routing OK" sẽ debug sai chỗ.

**Cơ chế hoạt động.** Flow tiêu biểu cho LRP `lrp1` IP 10.0.0.1:

```
table=3 (lr_in_ip_input), priority=90,
  match=(ip4.dst == 10.0.0.1 && icmp4.type == 8 && icmp4.code == 0),
  action=(ip4.dst <-> ip4.src;     /* swap */
          ip.ttl = 255;
          icmp4.type = 0;
          eth.dst <-> eth.src;
          inport = "";              /* trigger ovn-controller resubmit */
          flags.loopback = 1;
          outport = <inport_orig>;
          output;)

table=3 (lr_in_ip_input), priority=85,
  match=(ip.ttl == {0, 1} && !ip.later_frag),
  action=(icmp4_error { ... type=11 ... };  /* TTL exceeded */
          drop;)

table=3 (lr_in_ip_input), priority=80,
  match=(reg9[1] == 1 && ip4),         /* fragment-needed flag từ stage trước */
  action=(icmp4_error { type=3; code=4; "next-hop mtu" = <mtu>; };
          drop;)

table=3 (lr_in_ip_input), priority=90,
  match=(arp.op == 1 && arp.tpa == 10.0.0.1),
  action=(eth.dst = eth.src;
          eth.src = RR:RR:RR:RR:RR:01;
          arp.op = 2;
          arp.tha = arp.sha;
          arp.sha = RR:RR:RR:RR:RR:01;
          arp.tpa = arp.spa;
          arp.spa = 10.0.0.1;
          outport = inport;
          flags.loopback = 1;
          output;)

table=3 (lr_in_ip_input), priority=0,
  match=(1),
  action=(next;)
```

Action `clone {}` cho phép sinh packet mới (reply) mà không drop original (mặc dù trong ICMP echo case, packet original chính là packet được swap thành reply). Action `icmp4_error` consult `controller/pinctrl.c` để build ICMP error mới chứa header packet gốc 28 byte. Invariant: nếu packet match priority cao tại đây, nó **không** đi tiếp sang routing; nếu match priority 0 (`next;`), đi tiếp pipeline.

**Cách kỹ sư vận hành thành thạo.** (1) **Đọc**: `ovn-sbctl lflow-list lr1 | grep lr_in_ip_input` thấy nhánh ICMP / TTL / fragment / ARP. (2) **Sửa**: bật/tắt PMTUD qua `Logical_Router.options:pmtud_enabled`. (3) **Quan sát**: `ovn-trace lr1 'inport=="lrp1" && ip4.dst==10.0.0.1 && icmp4.type==8'` xem stage 3 có swap header. (4) **Debug**: nếu ping gateway fail, đếm `ovs-ofctl dump-flows br-int table=11 | grep n_packets` priority 90 ICMP — nếu n_packets không tăng khi ping → packet không đến được table 11 (vấn đề stage trước).

**Phân loại.** Loại stage: **router self-handler** (subtype "L3 endpoint service"). Khác `LS_IN_DHCP_OPTIONS` (DHCP cho LS), `LR_IN_DNAT` (NAT, phía sau). Stateful nhẹ (không touch conntrack) — chỉ touch packet, sinh reply.

**Quy trình sử dụng.**
1. `ovn-nbctl lr-add lr1`
2. `ovn-nbctl lrp-add lr1 lrp1 RR:01 10.0.0.1/24`
3. (optional) `ovn-nbctl set Logical_Router lr1 options:pmtud_enabled=true`
4. (optional BFD) `ovn-nbctl create BFD logical_port=lrp1 dst_ip=10.0.0.250 status=admin_up`
5. Verify: `ovn-sbctl lflow-list lr1 | grep lr_in_ip_input` thấy nhánh ICMP echo
6. Test: từ VM 10.0.0.5 → `ping 10.0.0.1` → xem reply trở về (count = count_send)
7. Test PMTUD: gửi packet > MTU với DF set → verify ICMP fragment-needed reply

**Khi xảy ra sự cố.** Symptom: (a) "Ping gateway loss 100%" → step 1: ovn-trace verify packet đến table 11, step 2: nếu đến nhưng không match priority 90, kiểm tra ip4.dst trong packet so với LRP_IP, step 3: xem Logical_Router.options. (b) "PMTUD broken (TCP stuck với MTU sai)" → check `pmtud_enabled` option + flow priority 80 fragment-needed. (c) "BFD session flapping" → kiểm tra BFD echo có được respond không (đếm flow BFD priority 90).

**Liên quan mật thiết.** Tightly coupled: `Logical_Router_Port.networks` (LRP IP source), `Logical_Router.options:pmtud_enabled`, `BFD` table NBDB, `controller/pinctrl.c` (xử lý icmp4_error generate), `LR_IN_LOOKUP_NEIGHBOR` (stage liền trước, populate MAC_Binding cache). Loose: VRRP/HSRP (alternative router redundancy ngoài OVN), routing protocol như OSPF/RIP (sentence-level mention only — banned scope per North Star).

**Khác biệt giữa các phiên bản.**
- OVN 2.5 (2015): chỉ ICMP echo + ARP responder
- OVN 20.06 (2020): thêm fragment-needed (PMTUD) qua `icmp4_error` action
- OVN 21.03 (2021): consolidate ARP responder vào stage này (trước đó ở stage riêng)
- OVN 22.03 (2022): BFD echo native, không cần punt
- OVN 23.06 (2023): ICMP rate-limit option chống flood
- OVN 24.03 (2024): thêm hỗ trợ ICMPv6 redirect cho IPv6

## §13.19.9 LR_IN_DNAT — Destination NAT trên ingress LR (floating IP, port forwarding)

**Khái niệm.** `LR_IN_DNAT` là stage logic **số 6** (OF table 14) — nơi thực hiện **Destination NAT** trên Logical_Router theo `NAT.type == "dnat"` hoặc `NAT.type == "dnat_and_snat"` (floating IP). Match `ip.dst == <external_ip>` → `ct(commit, nat(dst=<logical_ip>))` → packet sau stage này có dst đã rewrite về internal IP, conntrack entry committed để reply path biết undo. Loại: **pipeline stage cornerstone**, sub-loại "north-south DNAT + floating IP".

**Lịch sử + bối cảnh.** OVN 2.4 (2015) introduce NAT centralized trên gateway chassis (DNAT + SNAT chỉ chạy ở chassis được redirect). OVN 21.06 (2021) introduce **distributed NAT** (`dnat_and_snat` chạy trên mọi hypervisor có VM nằm sau floating IP, loại bỏ choke point gateway chassis). OVN 22.03 (2022) thêm option `force_snat_for_dnat` (cho phép kết hợp DNAT + SNAT để giải pet hairpin). OVN 23.06 (2023) tích hợp `Load_Balancer` + NAT: nếu LB VIP trùng NAT external_ip, LB DNAT ưu tiên hơn NAT DNAT (hàm `build_lrouter_nat_flows_for_lb`). OVN 24.03 (2024) thêm hỗ trợ NAT cho IPv6.

**Vị trí trong kiến trúc.** Stage logic 6 ⇒ OF table 14. Đứng sau `LR_IN_DEFRAG` (logical 4) + `LR_IN_UNSNAT` (logical 5), trước `LR_IN_ECMP_STATEFUL` (logical 7). Scope: per-packet, per-LR datapath. Quan trọng: chỉ packet north-bound (từ external vào internal) match priority cao tại đây; packet east-west fall through priority 0 next.

**Vai trò.** Vai trò chính: **DNAT enforcement** — rewrite `ip.dst` từ external IP (floating IP) sang internal IP của VM. Vai trò phụ: **conntrack commit** — tạo entry trong `nf_conntrack` zone của LR để reply path (`LR_OUT_UNDNAT` ở stage 1 egress) tự động undo DNAT. Phạm vi ảnh hưởng: lifetime của connection (entry tồn tại đến timeout conntrack).

**Vì sao sinh ra.** Cloud workload cần expose VM lên Internet với public IP riêng (floating IP). Trước OVN, OpenStack Neutron-OVS dùng iptables DNAT trong network node namespace → SPoF + slow. OVN abstract thành stage trong pipeline, đi kèm distributed DNAT (`dnat_and_snat`) cho phép mỗi hypervisor tự xử lý NAT cho VM của mình → loại choke point. Ngoài ra, distributed NAT giảm latency 1 hop (không qua gateway chassis).

**Vấn đề giải quyết.** Cụ thể: rewrite dst IP + commit conntrack atomic. Cách giải: hàm `build_lrouter_in_dnat_flow` (`northd/northd.c` dòng ~13067) iterate `NAT` rows của LR, sinh flow priority cao cho mỗi row:

- Cho `NAT.type == "dnat"`: match `ip.dst == external_ip`, action `ct_dnat(<logical_ip>)`. Centralized — thực thi tại gateway chassis sau khi packet được redirect.
- Cho `NAT.type == "dnat_and_snat"` (floating IP): match `ip.dst == external_ip`, action `ct_dnat(<logical_ip>)`. Distributed — nếu `Logical_Router.options:always_learn_from_arp_request=true`, mọi hypervisor đều biết MAC mapping nên có thể xử lý local.

Phạm vi giải quyết: chỉ DNAT (rewrite dst); SNAT làm ở stage `LR_OUT_SNAT` (egress); UNDNAT/UNSNAT cho reply path ở stage egress 1.

**Tầm quan trọng.** Tier 1 cornerstone cho cloud N-S workload. Floating IP, port forwarding, public service expose — tất cả qua stage này. Engineer must-know: phân biệt centralized DNAT (`dnat`) vs distributed (`dnat_and_snat`); hiểu conntrack zone per-LR (đảm bảo NAT entry không xung đột giữa LR khác); hiểu interaction với Load_Balancer.

**Cơ chế hoạt động.** Flow tiêu biểu cho NAT row external 1.2.3.4 → logical 10.0.0.5:

```
table=6 (lr_in_dnat), priority=100,
  match=(ip4 && ip4.dst == 1.2.3.4 && inport == "lrp-public" && is_chassis_resident("cr-lrp-public")),
  action=(ct_dnat(10.0.0.5);)

table=6 (lr_in_dnat), priority=50,
  match=(ip4 && ip4.dst == 1.2.3.4 && ct.est && ct_label.natted == 1),
  action=(next;)  /* established connection bypass re-DNAT */

table=6 (lr_in_dnat), priority=0,
  match=(1),
  action=(next;)
```

Action `ct_dnat(<ip>)` lúc compile xuống OVS thành `ct(commit, table=N, zone=<lr_zone>, nat(dst=<ip>))`. Conntrack commit entry với mapping `1.2.3.4 ↔ 10.0.0.5`, để reply (10.0.0.5 → external client) qua stage `LR_OUT_UNDNAT` tự động rewrite src về 1.2.3.4. Predicate `is_chassis_resident("cr-lrp-public")` ép flow chỉ thực thi tại gateway chassis sở hữu LRP public (cho centralized `dnat`) — distributed `dnat_and_snat` không có predicate này. Invariant: sau stage, packet `dst` đã đổi sang logical IP; conntrack entry đảm bảo reply UNDNAT đúng.

**Cách kỹ sư vận hành thành thạo.** (1) **Đọc**: `ovn-nbctl lr-nat-list lr1` xem các NAT row + type + external/logical IP. (2) **Sửa**: `ovn-nbctl lr-nat-add lr1 dnat_and_snat 1.2.3.4 10.0.0.5` (floating IP); `ovn-nbctl lr-nat-del lr1 dnat_and_snat 1.2.3.4` để gỡ. (3) **Quan sát**: `ovs-appctl dpctl/dump-conntrack zone=<lr_zone> | grep 1.2.3.4` xem entry. (4) **Debug**: `ovn-trace lr1 'inport=="lrp-public" && ip4.dst==1.2.3.4 && ...'` thấy stage 6 chọn ct_dnat.

**Phân loại.** Loại stage: **N-S DNAT enforcer** (subtype "stateful destination NAT"). Khác `LS_IN_LB` (LB DNAT trên LS, có hash_fields), `LR_IN_UNSNAT` (reverse SNAT cho reply, ngay trước stage này), `LR_OUT_SNAT` (forward SNAT, ở egress).

**Quy trình sử dụng.** Khi expose VM với floating IP:
1. Gắn LRP public: `ovn-nbctl lrp-add lr1 lrp-public RR:99 1.2.3.1/24`
2. Set gateway chassis: `ovn-nbctl lrp-set-gateway-chassis lrp-public chassis-1 100`
3. Tạo NAT row: `ovn-nbctl lr-nat-add lr1 dnat_and_snat 1.2.3.4 10.0.0.5`
4. Verify: `ovn-sbctl lflow-list lr1 | grep lr_in_dnat` thấy flow priority 100 cho 1.2.3.4
5. Test: từ external client `ping 1.2.3.4` → packet đến VM 10.0.0.5 → reply tự động UNDNAT
6. Verify conntrack: `ovs-appctl dpctl/dump-conntrack | grep 1.2.3.4` thấy entry mapping

**Khi xảy ra sự cố.** Symptom: (a) "Floating IP unreachable từ Internet" → step 1: ovn-trace verify stage 6 match priority 100, step 2: nếu match nhưng packet không đến VM, kiểm tra conntrack zone có conflict. (b) "FIP đôi khi work, đôi khi không" → kiểm tra gateway chassis HA — nếu chassis fail, distributed NAT không có ARP responder → packet rớt. (c) "FIP work nhưng VM không reply ngược" → kiểm tra `LR_OUT_UNDNAT` stage egress. (d) "Conntrack table full" → `nf_conntrack_count / max > 95%` → tăng max hoặc giảm timeout. Cross-link: Phần 13.3 §NAT compile, Phần 9.24 §ct_state, Phần 13.18 §REGBIT_NAT_REDIRECT.

**Liên quan mật thiết.** Tightly coupled: NBDB `NAT` table (nguồn dữ liệu), `LR_IN_DEFRAG` (defrag trước NAT, nếu không sẽ NAT lệch fragment), `LR_IN_UNSNAT` (reverse SNAT, stage liền trước), `LR_OUT_UNDNAT` + `LR_OUT_SNAT` (egress reverse path), `Load_Balancer` (nếu VIP trùng external_ip, LB ưu tiên), `is_chassis_resident` predicate (xác định gateway chassis), conntrack zone per-LR. Loose: BGP advertisement floating IP (banned scope per North Star — chỉ mention cấp độ sentence).

**Khác biệt giữa các phiên bản.**
- OVN 2.4 (2015): chỉ `dnat` centralized; phải qua gateway chassis
- OVN 21.06 (2021): distributed `dnat_and_snat` — chạy trên mọi hypervisor có VM target
- OVN 22.03 (2022): option `force_snat_for_dnat` cho hairpin tricky
- OVN 23.06 (2023): LB + NAT integration via `build_lrouter_nat_flows_for_lb`
- OVN 24.03 (2024): NAT cho IPv6; `allowed_ext_ips` option giới hạn external IP nguồn

## §13.19.10 LR_IN_GW_REDIRECT — chuyển hướng N-S traffic về gateway chassis

**Khái niệm.** `LR_IN_GW_REDIRECT` là stage logic **số 18** (OF table 26) — nơi quyết định chuyển hướng packet **north-south** về **gateway chassis** (chassis được chỉ định trong `Logical_Router_Port.gateway_chassis` hoặc `HA_Chassis_Group`). Match `outport` đang trỏ tới một LRP có `gateway_chassis` config → đổi outport thành `chassisredirect` Port_Binding (`cr-<LRP>`) → packet đi qua tunnel Geneve tới gateway chassis, ở đó pipeline LR tiếp tục các stage còn lại (NAT egress, ARP resolve cho external network). Loại: **pipeline stage cornerstone**, sub-loại "distributed gateway redirect".

**Lịch sử + bối cảnh.** OVN 2.10 (2018) introduce concept "gateway router" — toàn bộ LR đặt trên 1 chassis (centralized). OVN 21.06 (2021) introduce **distributed gateway** với chassisredirect port (`cr-<LRP>`): pipeline LR phân tán trên mọi hypervisor cho phần east-west, chỉ những stage cần external (NAT, ARP cho physical network) mới redirect về 1 chassis được elect. OVN 22.03 (2022) integrate `HA_Chassis_Group` cho HA failover của gateway chassis (BFD-based). OVN 23.06 (2023) thêm `redirect-type=bridged` (alternative redirect qua Logical_Switch thay vì Geneve tunnel). OVN 24.03 (2024) tối ưu redirect cho LB DNAT (skip redirect nếu LB chỉ tới VM cùng hypervisor).

**Vị trí trong kiến trúc.** Stage logic 18 ⇒ OF table 26. Đứng sau `LR_IN_ARP_RESOLVE` (logical 16) + `LR_IN_CHK_PKT_LEN` (logical 17), trước `LR_IN_LARGER_PKTS` (logical 19). Scope: per-packet. Đây là **điểm chuyển trọng tâm** giữa pipeline distributed (mọi hypervisor) và pipeline centralized (chỉ gateway chassis); sau stage này, nếu redirect, packet chui qua tunnel Geneve để stage egress chạy ở chassis khác.

**Vai trò.** Vai trò chính: **gateway dispatcher** — quyết định stage egress LR chạy local hay tại gateway chassis. Vai trò phụ: **HA selector** — nếu gateway_chassis có HA group, chọn chassis đang up qua BFD. Phạm vi: 1 packet, ảnh hưởng outport register reg15.

**Vì sao sinh ra.** Centralized gateway router (OVN < 21.06) tạo SPoF + bottleneck CPU (tất cả N-S traffic qua 1 chassis). Distributed gateway tách thành phần: stage east-west chạy local mỗi hypervisor (LR_IN_IP_INPUT, IP_ROUTING, ARP cho LRP nội bộ); stage cần external (NAT external_ip, ARP cho IP physical, BFD physical link) cần single hypervisor để tránh duplicate ARP / NAT entry race. Stage này là "switch" giữa 2 pipeline parts.

**Vấn đề giải quyết.** Cụ thể: chuyển packet đến gateway chassis nếu outport là LRP có `gateway_chassis` config. Cách giải: hàm `build_gateway_redirect_flows_for_lrouter` (`northd/northd.c`) iterate mọi LRP có `gateway_chassis` non-empty, sinh flow priority 100 match `outport == <LRP> && !is_chassis_resident("cr-<LRP>")` action `outport = "cr-<LRP>"; next;`. `cr-<LRP>` là chassisredirect Port_Binding — virtual port chỉ tồn tại trên 1 chassis (được elect qua BFD HA). Khi outport set tới `cr-<LRP>`, OVN encap packet trong Geneve và gửi tới chassis đó. Phạm vi: chỉ N-S; east-west không có gateway_chassis nên priority 0 next, đi tiếp local.

**Tầm quan trọng.** Tier 1 cornerstone cho distributed gateway architecture — nền tảng OVN scalable. Mọi cluster cloud production OVN đều dùng pattern này (≥ OVN 21.06). Engineer must-know: hiểu `chassisredirect` Port_Binding type khác biệt với LSP thường (xem 13.5b); hiểu `is_chassis_resident()` predicate nghĩa là "chassis đang sở hữu cr port"; hiểu HA_Chassis_Group failover qua BFD (xem 13.6). Không nắm thì class lỗi "VM không truy cập Internet sau gateway chassis fail" sẽ debug sai.

**Cơ chế hoạt động.** Flow tiêu biểu cho LRP `lrp-public` có gateway_chassis = `chassis-1` (priority 100), `chassis-2` (priority 50):

```
table=18 (lr_in_gw_redirect), priority=100,
  match=(outport == "lrp-public" && !is_chassis_resident("cr-lrp-public")),
  action=(outport = "cr-lrp-public";
          next;)

table=18 (lr_in_gw_redirect), priority=50,
  match=(outport == "lrp-public" && is_chassis_resident("cr-lrp-public")),
  action=(next;)  /* local chassis sở hữu cr port, không redirect */

table=18 (lr_in_gw_redirect), priority=0,
  match=(1),
  action=(next;)  /* east-west, không redirect */
```

`is_chassis_resident("cr-lrp-public")` là conditional macro — ovn-controller eval trên local chassis: nếu `Chassis.name` của local chassis == chassis hiện sở hữu cr port (theo `Port_Binding.chassis` của cr port, được update qua BFD/HA), return true. Khi true, packet đã ở đúng chassis nên `next;` đi tiếp pipeline LR egress local. Khi false, đổi outport sang `cr-lrp-public` → OVN encap Geneve gửi tới chassis sở hữu cr. Invariant: sau stage này, mọi packet có outport là LRP gateway sẽ kết thúc tại đúng 1 chassis.

**Cách kỹ sư vận hành thành thạo.** (1) **Đọc**: `ovn-nbctl lrp-get-gateway-chassis lrp-public` xem mapping; `ovn-sbctl find Port_Binding type=chassisredirect` xem cr port ở chassis nào. (2) **Sửa**: `ovn-nbctl lrp-set-gateway-chassis lrp-public chassis-1 100` + `ovn-nbctl lrp-set-gateway-chassis lrp-public chassis-2 50` (failover). (3) **Quan sát**: `ovn-trace lr1 'inport=="lrp-internal" && ... && ip4.dst==<external>'` xem stage 18 có redirect. (4) **Debug**: nếu N-S broken sau gateway fail, check `ovn-sbctl find Chassis_Private` xem BFD status các chassis; check `ovn-sbctl get Port_Binding cr-lrp-public chassis` xem cr port có active chassis hay không.

**Phân loại.** Loại stage: **gateway dispatcher** (subtype "distributed-to-centralized switch"). Khác `LR_IN_DNAT` (NAT enforcement, không redirect), `LR_IN_IP_ROUTING` (route lookup, có thể chọn outport gateway nhưng không redirect). Stateless thuần.

**Quy trình sử dụng.**
1. Tạo LR + LRP public: `ovn-nbctl lr-add lr1 && ovn-nbctl lrp-add lr1 lrp-public RR:99 1.2.3.1/24`
2. Đăng ký 2 chassis có physical bridge mapping: `ovs-vsctl set Open_vSwitch . external-ids:ovn-bridge-mappings=public:br-public` (trên cả chassis-1 và chassis-2)
3. Set HA gateway: `ovn-nbctl lrp-set-gateway-chassis lrp-public chassis-1 100 && ovn-nbctl lrp-set-gateway-chassis lrp-public chassis-2 50`
4. Verify cr port: `ovn-sbctl find Port_Binding type=chassisredirect` thấy `cr-lrp-public` với chassis = chassis-1 (priority cao nhất)
5. Verify flow: `ovn-sbctl lflow-list lr1 | grep lr_in_gw_redirect` thấy flow priority 100 + 50
6. Test failover: kill ovn-controller trên chassis-1 → đợi BFD timeout (~5s) → cr port chuyển sang chassis-2 → test N-S vẫn work

**Khi xảy ra sự cố.** Symptom: (a) "VM không truy cập Internet sau khi gateway chassis fail" → step 1: kiểm tra BFD status, step 2: nếu cr port không chuyển sang chassis-2, kiểm tra HA_Chassis_Group status, step 3: nếu cr port chuyển nhưng vẫn fail, kiểm tra physical bridge mapping trên chassis-2. (b) "Sau gateway chassis fail, có 5s loss" → BFD interval default 1000ms × 5 = 5s; có thể giảm xuống 200ms × 3 = 600ms. (c) "Gateway chassis CPU 100% nhưng cluster có 5 chassis" → gateway centralize ở 1 chassis là bottleneck cho tổng N-S traffic; consider thêm `Logical_Router_Port` riêng cho mỗi chassis (multi-gw pattern OVN 23.06+). Cross-link: Phần 13.6 §HA chassis group + BFD, Phần 13.18 §REGBIT_NAT_REDIRECT.

**Liên quan mật thiết.** Tightly coupled: `Logical_Router_Port.gateway_chassis` (NBDB nguồn dữ liệu), `HA_Chassis_Group` + `HA_Chassis` (HA election), `Port_Binding.type=chassisredirect` (cr port virtual), `BFD` table (link liveness), `is_chassis_resident()` predicate, `LR_IN_ARP_RESOLVE` (stage trước resolve next-hop), `LR_OUT_DELIVERY` (stage egress sau redirect). Loose: ovn-bridge-mappings (config OVS bên dưới, không phải NBDB).

**Khác biệt giữa các phiên bản.**
- OVN 2.10 (2018): chỉ centralized gateway router, không có chassisredirect concept
- OVN 21.06 (2021): introduce distributed gateway + cr port + stage này
- OVN 22.03 (2022): integrate HA_Chassis_Group cho HA failover qua BFD
- OVN 23.06 (2023): `redirect-type=bridged` alternative (qua LS thay Geneve)
- OVN 24.03 (2024): tối ưu skip redirect cho LB DNAT khi target VM cùng hypervisor

```

$ ovn-sbctl find Port_Binding type=chassisredirect
```

**Source code.** Tham chiếu OVN branch-22.03:
- `northd/northd.c` `build_gateway_redirect_flows_for_lrouter()`: hàm chính build stage này
- `northd/northd.c` `build_gateway_redirect_flows_for_lrouter_port()`: per-LRP build
- `controller/binding.c` `consider_local_datapath()`: xử lý chassisredirect Port_Binding
- `controller/ha-chassis.c` `ha_chassis_group_run()`: HA election logic
- `lib/lflow.h` `S_ROUTER_IN_GW_REDIRECT = 18`: stage id

**Trường hợp sự cố thực tế.** Q3/2024 production: customer báo "sau khi reboot chassis-1 (gateway chassis primary), tenant N-S broken 30 giây thay vì <5s expected". Investigation: BFD timeout đúng 5s (config interval=1000 multiplier=5). Sau BFD down, OVN northd phải re-compile flow (cr port chuyển chassis), ovn-controller các chassis còn lại nhận update SBDB. Đo `ovn-sbctl get Port_Binding cr-lrp-public chassis` thấy update chỉ sau ~3s. Nhưng test traffic vẫn drop tới ~30s. Root cause: ARP cache trên switch physical bên ngoài cluster vẫn trỏ MAC chassis-1 (gateway physical interface MAC). Cần GARP từ chassis-2 advertise IP của LRP public sang cùng physical network. OVN có gửi GARP nhưng switch upstream lúc đó throttle GARP (storm protection). Fix: tăng `Logical_Router_Port.options:ra_arp_rate_limit` để cụm GARP nhiều lần; hoặc yêu cầu network team disable GARP throttle trên switch. Cross-link Phần 13.18 §REGBIT_NAT_REDIRECT.

**Bài tập synthetic.**

```bash
sudo ovn-nbctl lr-add lr-lab
sudo ovn-nbctl lrp-add lr-lab lrp-internal RR:11 10.0.0.1/24
sudo ovn-nbctl lrp-add lr-lab lrp-public RR:99 1.2.3.1/24
sudo ovn-nbctl lrp-set-gateway-chassis lrp-public chassis-1 100
sudo ovn-nbctl lrp-set-gateway-chassis lrp-public chassis-2 50
# Lab task 1: ovn-trace từ vm tới external 8.8.8.8 → verify stage 18 redirect tới cr-lrp-public
# Lab task 2: ovn-sbctl find Port_Binding type=chassisredirect → verify chassis hiện active = chassis-1
# Lab task 3: stop ovn-controller trên chassis-1 → đợi 5s → verify cr port chuyển chassis-2
# Lab task 4: bật lại chassis-1 → verify cr port quay về chassis-1 (priority 100 cao hơn)
# Lab task 5: đổi gateway_chassis priority chassis-2 lên 200 → verify cr port chuyển ngay
```

**Lỗi thường gặp + tín hiệu chẩn đoán.**
- Mode A "cr port stuck ở chassis fail": HA_Chassis_Group election broken → check BFD status + chassis-private records
- Mode B "Failover chậm": BFD interval/multiplier quá lớn → giảm để < 1s
- Mode C "Sau failover, packet vẫn rớt": ARP cache upstream switch stale → tăng GARP rate hoặc reset network upstream
- Mode D "Gateway chassis CPU 100%": tất cả N-S qua 1 chassis → consider multi-LR pattern hoặc redirect-type=bridged
- Mode E "redirect không xảy ra": LRP chưa có gateway_chassis config → set qua lrp-set-gateway-chassis

**So sánh với hệ khác.** Cisco HSRP/VRRP gateway redundancy tương tự ý tưởng (active/standby gateway, virtual MAC) nhưng OVN dùng BFD + chassisredirect pattern khác: OVN không có virtual MAC chia sẻ, mà là chuyển đường packet qua tunnel sang chassis active. NSX-T edge cluster với "service router" centralized analog: cùng chia thành DR distributed + SR centralized, gateway service centralize ở edge node. Linux keepalived + VIP analog: VIP migration giữa node, nhưng layer thấp hơn (không có pipeline overlay). AWS NAT Gateway managed service (centralize per-AZ) tương đương semantic.

## 9.10.1 ptcp vs ssl

Controller và Manager connection hỗ trợ ba transport:

- **`unix:`**: local Unix socket, chỉ root cùng host. Secure by default.
- **`ptcp:PORT`**: passive TCP, không authentication, không encryption. **Chỉ dùng lab.**
- **`ssl:IP:PORT`**: TLS với mutual authentication (mTLS). **Production mandatory.**

Compass Ch K: "Every `Controller` or `Manager` target should be `ssl:IP:PORT`."

`ptcp:` trong production = risk: bất kỳ ai đến được TCP port có thể (a) đọc toàn bộ OVSDB config, (b) inject flow rule arbitrary, (c) chạy lệnh remote qua OpenFlow.

## 9.10.2 ovs-pki: minimal PKI toolkit

OVS ship với `ovs-pki` — script wrap OpenSSL. Compass Ch K workflow:

```bash
[ca-host]# ovs-pki init                            # tạo root CA
[ca-host]# ovs-pki req+sign ovsclient switch      # switch cert
[ca-host]# ovs-pki req+sign ctl controller         # controller cert

[switch-host]# ovs-vsctl set-ssl /etc/openvswitch/key.pem \
                  /etc/openvswitch/cert.pem \
                  /etc/openvswitch/cacert.pem
[switch-host]# ovs-vsctl set-controller br0 ssl:10.0.0.10:6653
```

- `ovs-pki init` tạo `switchca/` và `controllerca/` với cấu trúc X.509 CA tiêu chuẩn.
- `req+sign ovsclient switch` tạo private key + CSR + sign với CA.
- `set-ssl` ghi ba path vào OVSDB SSL row.

---

## 9.10.3 Verify handshake

Từ controller side:

```bash
[controller]$ openssl s_client -connect 10.0.0.10:6653 \
              -CAfile cacert.pem -cert ctl-cert.pem -key ctl-privkey.pem
```

Output nên hiện `Verify return code: 0 (ok)` và TLS version ≥ 1.2.

Error thường: `certificate verify failed` → certificate chain không khớp với CA (thường do cacert.pem thiếu intermediate CA).

## 9.10.4 CA rotation

Compass Ch K rule: **"Rotate CAs by adding the new root to `cacert.pem` BEFORE switching the client cert, never the reverse."**

Quy trình:

1. Generate new CA `new-ca.pem`.
2. Append new CA vào `cacert.pem` trên mọi switch và controller.
3. Wait propagation.
4. Issue client cert mới signed bởi new CA.
5. Switch client cert sang new cert — peer vẫn trust vì cả hai CA đều có.
6. Sau ổn định, remove old CA khỏi cacert.pem.

Làm ngược (switch cert trước) → peer không trust → fail.

## 9.10.5 TLS version và cipher

OVS 2.17+ hỗ trợ TLS 1.2 và 1.3:

```bash
[host]# ovs-vsctl set SSL <uuid> bootstrap_ca_cert=false \
         ssl_protocols=TLSv1.2,TLSv1.3 \
         ssl_ciphers=HIGH:!aNULL:!MD5
```

Bootstrap mode (`bootstrap_ca_cert=true`) dành cho first-time setup. **Disable ngay sau khi operational** để tránh MITM.

## 9.10.6 SSL row trong vswitch.ovsschema

**Khái niệm.** `SSL` là một bảng trong `Open_vSwitch` database (file `vswitch.ovsschema`) chỉ chứa **đúng một dòng** (`maxRows: 1`). Dòng này nắm giữ toàn bộ thông tin certificate cần thiết cho ovs-vswitchd và ovsdb-server thực hiện handshake TLS với peer (controller hoặc OVSDB client).

**Vai trò + vì sao sinh ra.** Trước khi có SSL row, mỗi daemon nhận đường dẫn cert qua `--private-key`/`--certificate`/`--ca-cert` command line. Khi cluster lớn, đổi đường dẫn đồng nghĩa restart daemon. Đặt cert path vào OVSDB cho phép `ovs-vsctl set-ssl` thay đổi runtime, daemon đọc OVSDB notify và cập nhật in-place.

**Cấu trúc 5 cột (verified `vswitch.ovsschema` v2.17.9):**

| Cột | Kiểu | Vai trò |
|-----|------|---------|
| `private_key` | string | Đường dẫn file PEM chứa private key của switch |
| `certificate` | string | Đường dẫn file PEM chứa cert của switch (signed by CA) |
| `ca_cert` | string | Đường dẫn file PEM chứa CA cert (root + intermediate nếu có) |
| `bootstrap_ca_cert` | boolean | Bật chế độ bootstrap (lần đầu lấy CA cert tự động) |
| `external_ids` | map<string,string> | Metadata tự do, không ảnh hưởng SSL handshake |

**Quy trình sử dụng.** `ovs-vsctl set-ssl <privkey> <cert> <cacert>` ghi 3 đường dẫn vào SSL row + đặt reference từ `Open_vSwitch.ssl` cột tới UUID của row. Verify bằng `ovs-vsctl get-ssl` hoặc `ovs-vsctl list SSL` (xem Anatomy Template A §9.10.X).

**Lỗi thường gặp + tín hiệu chẩn đoán.** Nếu `Open_vSwitch.ssl` reference rỗng nhưng controller dùng `ssl:`, log hiện `Stream_ssl|ERR|Private key must be configured`. Fix: chạy `set-ssl` đầy đủ 3 path. Nếu `private_key` trỏ tới file world-readable (mode 644), `openssl` daemon vẫn load được nhưng đây là CRITICAL security bug (Failure 2 trong §9.10.Z).

**Source code tham chiếu.** Schema file `vswitchd/vswitch.ovsschema` v2.17.9 dòng định nghĩa `"SSL"`, key `"maxRows": 1` xác nhận singleton invariant. Code đọc + apply: `lib/stream-ssl.c` function `stream_ssl_set_private_key_file` + `stream_ssl_set_certificate_file`.

## 9.10.7 Manager row + management connection target

**Khái niệm.** `Manager` là bảng trong `Open_vSwitch` schema dùng để khai báo **endpoint quản lý OVSDB** (tức endpoint mà ovsdb-server lắng nghe để các process bên ngoài như ovn-controller, OpenStack Neutron, hoặc CMS khác kết nối tới read/write OVSDB). Khác với `Controller` (dành cho OpenFlow), Manager dành riêng cho lớp OVSDB management.

**Cột chính (verified schema v2.17.9):**

| Cột | Kiểu | Vai trò |
|-----|------|---------|
| `target` | string (index) | Endpoint string `pssl:6640`, `ssl:10.0.0.1:6640`, `ptcp:`, `unix:` |
| `inactivity_probe` | integer (ms) | Khoảng đáo lại để phát hiện peer chết |
| `max_backoff` | integer (ms ≥1000) | Giới hạn tăng dần thời gian retry sau lỗi |
| `connection_mode` | enum `in-band`/`out-of-band` | Routing path tới manager (out-of-band = mgmt network) |
| `is_connected` | boolean (ephemeral) | Trạng thái runtime, không persist |
| `status` | map (ephemeral) | Chi tiết runtime: `last_error`, `state` |
| `external_ids`, `other_config` | map | Metadata + tuning |

**Quy trình sử dụng.** `ovs-vsctl set-manager pssl:6640` thêm Manager row với target `pssl:6640` (OVSDB-server lắng nghe TLS port 6640). Khi target dạng `ssl:` hoặc `pssl:`, ovsdb-server sẽ dùng SSL row §9.10.6 cho cert path.

**Vai trò trong production.** Trong cụm OVN, Manager row trên mỗi compute node thường là `ptcp:6640:127.0.0.1` (chỉ local) hoặc `pssl:6640` nếu cần CMS truy cập từ xa. Reference: file `Open_vSwitch.manager_options` cột set [Manager UUID list].

**Lỗi thường gặp.** `is_connected=false` + `status:{last_error="connection refused"}` báo OVSDB-server không lắng nghe đúng cổng. Verify bằng `ss -tlnp | grep 6640` trên target host.

**Source code tham chiếu.** Schema entry `"Manager"` trong `vswitchd/vswitch.ovsschema`. Logic: `vswitchd/bridge.c` function `bridge_configure_remotes` đọc Manager rows và truyền cho `ovsdb-server` qua jsonrpc.

## 9.10.8 ovs-pki self-sign cho lab + test

**Khái niệm.** `ovs-pki self-sign <name>` tạo certificate **tự ký** từ private key + CSR đã có sẵn, không cần CA hierarchy. Equivalent OpenSSL: `openssl x509 -req -signkey privkey.pem -in req.pem -out cert.pem -days 3650`.

**Vì sao có command này.** Bootstrap PKI đầy đủ (`init` → `req` → `sign`) phù hợp production nhưng tốn 4-5 bước cho mỗi node. Trong lab 1 node hoặc CI test cần TLS handshake hoạt động (không cần peer trust), self-sign rút workflow xuống 2 bước: `req` + `self-sign`.

**Workflow 3 lệnh:**

```bash
$ ovs-pki req lab01            # tạo lab01-privkey.pem + lab01-req.pem
$ ovs-pki self-sign lab01      # tạo lab01-cert.pem (signed by chính lab01-privkey)
$ openssl x509 -in lab01-cert.pem -text -noout | grep -E 'Issuer|Subject|Validity'
```

**Đặc điểm cert tạo bởi self-sign (verified `utilities/ovs-pki.in` v2.17.9):**

- Validity 3650 ngày (10 năm) — hardcode trong script.
- `subjectAltName = DNS:<name>` được auto-thêm.
- Issuer = Subject (đặc trưng self-signed).
- Mode file `cert.pem.tmp` = `077 umask` (chỉ owner read).

**Trường hợp sự cố thực tế.** Self-sign thường bị nhầm lẫn dùng cho production. Khi 2 chassis cùng có self-signed cert nhưng không trust nhau, handshake fail với `unable to get local issuer certificate`. Fix: chuyển sang full PKI workflow với CA chung.

**Source code tham chiếu.** `utilities/ovs-pki.in` v2.17.9 case `"self-sign"`: kiểm tra tồn tại `arg1-req.pem` + `arg1-privkey.pem`, tạo extfile chứa `subjectAltName`, gọi `openssl x509 -req -signkey ... -days 3650`.

**So sánh với `sign`.** Self-sign dùng chính private key của subject để ký, nên không cần CA private key (an toàn distribute). `sign` cần `controllerca/cakey.pem` hoặc `switchca/cakey.pem` truy cập được, chỉ chạy trên CA host. Xem §9.10.9.

## 9.10.9 ovs-pki sign cho production PKI flow

**Khái niệm.** `ovs-pki sign <name> <type>` ký một CSR (`<name>-req.pem`) bằng CA private key, output là `<name>-cert.pem`. `<type>` phải là `switch` hoặc `controller`, quyết định CA nào ký (`switchca/` hay `controllerca/`).

**Workflow tách 2 host (production-grade):**

```bash
# Trên chassis (host hv01):
$ ovs-pki req hv01                       # sinh hv01-privkey.pem (KHÔNG ROI khỏi host) + hv01-req.pem

# Copy hv01-req.pem qua CA host (scp, không cần secure channel vì CSR public):
$ scp hv01-req.pem ca-host:/tmp/

# Trên CA host:
$ ovs-pki sign hv01 switch               # ovs-pki yêu cầu fingerprint verify (anti-tamper)
$ scp hv01-cert.pem hv01:/etc/openvswitch/
```

**Validity period.** `sign` dùng CA configuration (`switchca/openssl.cnf` mặc định `default_days = 3650`) — tức cert valid 10 năm. Operator có thể chỉnh `openssl.cnf` `default_days` trước khi sign.

**Tại sao không dùng `req+sign` luôn.** `req+sign` thực hiện cả `req` + `sign` trên cùng host. Production yêu cầu CA private key KHÔNG được copy ra khỏi CA host (defense-in-depth), nên `req` phải chạy trên chassis (giữ privkey local) còn `sign` phải chạy trên CA host (giữ CA key local). `req+sign` chỉ phù hợp lab.

**Anti-tamper fingerprint check.** Khi `sign` chạy, ovs-pki tính SHA-1 fingerprint của CSR và yêu cầu operator đọc miệng cùng fingerprint từ chassis (out-of-band verify) để chống MITM swap CSR. Bypass bằng `--batch` (không khuyên).

**Lỗi thường gặp.** Nếu `<type>` sai (ví dụ ký `switch` cho cert dùng làm controller), peer reject với `certificate verify failed: unable to verify the first certificate`. Fix: re-sign với đúng type, hoặc cấu hình peer trust cả hai CA.

**Source code tham chiếu.** `utilities/ovs-pki.in` v2.17.9 case `"sign"`: gọi `one_or_two_args`, `check_type "$arg2"` (validate switch/controller), `verify_fingerprint "$arg1-req.pem"`, `sign_request "$arg1-req.pem" "$arg1-cert.pem"`. Function `sign_request` wrap `openssl ca -policy ... -out ... -in ...`.

## 9.10.10 Certificate file PEM trên đĩa

**Khái niệm.** Mọi cert + key OVS dùng đều ở dạng **PEM** (Privacy-Enhanced Mail) — text Base64 với header `-----BEGIN CERTIFICATE-----` và footer `-----END CERTIFICATE-----`. PEM khác DER (binary): PEM dễ inspect bằng `cat`, copy-paste qua email/chat, dán vào config Ansible/Terraform.

**Convention đặt path `/etc/openvswitch/`:**

| File | Nội dung | Mode khuyên |
|------|---------|-------------|
| `/etc/openvswitch/key.pem` | Private key của switch | `0600 root:root` |
| `/etc/openvswitch/cert.pem` | Cert của switch (signed by switchca) | `0644 root:root` |
| `/etc/openvswitch/cacert.pem` | CA cert (controllerca → switch trust controller) | `0644 root:root` |

Đường dẫn này là convention từ Open vSwitch FAQ + `set-ssl` example, không hardcode trong code (operator có thể đặt nơi khác). Tên `key.pem`/`cert.pem` ngắn để giảm sai sót; production thường rename `<hostname>-privkey.pem` để truy ngược nguồn gốc.

**Nội dung mẫu một cert PEM:**

```
-----BEGIN CERTIFICATE-----
MIIDXTCCAkWgAwIBAgIJAKl... (Base64 multi-line) ...wJ8B5cVEnQQB6w==
-----END CERTIFICATE-----
```

Một file PEM có thể chứa **nhiều block** liên tiếp — phổ biến trong `cacert.pem` chứa cả root CA + intermediate CA (chain). OpenSSL parse từ trên xuống dưới, dùng tất cả block.

**Cách kiểm tra.**

```bash
$ openssl x509 -in /etc/openvswitch/cert.pem -text -noout    # decode 1 cert
$ openssl crl2pkcs7 -nocrl -certfile cacert.pem | openssl pkcs7 -print_certs -text   # decode chain
$ stat -c '%a %U:%G %n' /etc/openvswitch/key.pem             # verify mode 0600
```

**Lỗi thường gặp.**

1. **Mode 644 trên private key**: world-readable, security bug #1 trong audit. Fix `chmod 600`.
2. **PEM encoding broken**: thiếu `-----BEGIN`/`-----END` line, hoặc Base64 corrupt. Triệu chứng: `unable to load private key:no start line`. Fix: re-issue cert.
3. **DOS line ending CRLF** sau scp từ Windows: OpenSSL parse fail trên một số version cũ. Fix: `dos2unix cert.pem`.

**So sánh hệ khác.** Linux distro standard PEM path đa dạng: `/etc/ssl/certs/` (Debian), `/etc/pki/tls/certs/` (RHEL). OVS chọn `/etc/openvswitch/` riêng để separate ownership từ CA bundle hệ thống — tránh `update-ca-certificates` ghi đè.

**Source code tham chiếu.** Function load PEM: `lib/stream-ssl.c` v2.17.9 `stream_ssl_set_private_key_file_with_password` gọi `SSL_CTX_use_PrivateKey_file(..., SSL_FILETYPE_PEM)`. OpenSSL DER cũng support nếu đổi sang `SSL_FILETYPE_ASN1` nhưng OVS hardcode PEM.

## 9.10.X Anatomy Template A — `ovs-vsctl list SSL`

**Bối cảnh khi chạy:** Verify TLS config + cert chain đầy đủ trước khi deploy production.

**Command và output** *(reproduced OVS 2.17.9)*:

```shell
$ sudo ovs-vsctl list SSL
_uuid               : 7e3c9f82-...
bootstrap_ca_cert   : false
ca_cert             : "/etc/openvswitch/cacert.pem"
certificate         : "/etc/openvswitch/sc-cert.pem"
external_ids        : {}
private_key         : "/etc/openvswitch/sc-privkey.pem"
```

**9-attribute anatomy:**

| Field | Giá trị mẫu | Ý nghĩa | Red flag nếu |
|-------|-------------|---------|--------------|
| `Open_vSwitch.ssl` ref | `[<uuid>]` | SSL row reference | Empty = OVS không TLS config |
| `ca_cert` path | `/etc/.../cacert.pem` | CA cert file | File missing = SSL reject |
| `certificate` path | `/etc/.../sc-cert.pem` | Switch cert | Self-signed + peer no trust = handshake fail |
| `private_key` path | `/etc/.../sc-privkey.pem` | Key file | Permission 644 = world readable security bug |
| `bootstrap_ca_cert` | `false` | Bootstrap mode | `true` persistent = MITM risk |
| Cert expiry (openssl) | >30 days | Expiration date | <30 days = critical |
| Cert CN | `chassis-1.example.com` | Common Name | Mismatch hostname = verification fail |
| Chain depth | 3 | CA hierarchy | >5 = unusual misconfig |
| Cipher suite | TLS 1.3 AES-256-GCM | Negotiated | TLS <1.2 = deprecated |

**Kịch bản bẻ gãy:**

- **SSL column empty + controller ssl:IP:port**: TLS config missing. Fix: `ovs-vsctl set-ssl /path/privkey /path/cert /path/cacert`.
- **Private key permission 644**: world-readable. Fix: `chmod 600`.
- **Cert CN mismatch hostname**: hostname verification fail. Fix: re-issue cert, sử dụng SAN cho multi-hostname.

## 9.10.Y Hiểu sai + Key Topic

> **Hiểu sai phổ biến 1:** *"Self-signed cert production safe vì chỉ internal."*
>
> **Thực tế:** Self-signed không có CA signature → không distribute trust boundary scalable. Add chassis mới cần manual copy cert. Revocation không work. Use CA (internal qua `ovs-pki` hoặc external Vault).

> **Hiểu sai phổ biến 2:** *"Bootstrap mode chỉ lab."*
>
> **Thực tế:** Bootstrap có use case valid — first-time deployment remote site. Window 1 giờ, sau đó disable. Pattern an toàn hơn manual cert copy nếu network isolation + physical security ensure.

> **Chủ đề trọng tâm:** OVS + OVN TLS là zero-trust foundation. Mỗi chassis có identity cert + verify peer. Cấm `ptcp:` production. Cert rotation strategy: annual minimum, cross-sign overlap 60 days, automation via ansible + ovs-pki, alert <30 days. Phần 20.1 §20.1.15 cert rotation zero-downtime + Phần 10.6 OVSDB mTLS deep-dive.

---

## 9.10.Z Common failure modes

**Failure 1 — Nút mạng (Chassis) không kết nối SBDB TLS.** Log: `ssl: failed to verify certificate`. Fix: `openssl verify -CAfile cacert.pem sc-cert.pem`.

**Failure 2 — Cipher negotiation fail.** Log: `ssl: no shared cipher`. Fix: align cipher set với peer.

**Failure 3 — Cert rotation downtime.** Cause: restart sau rotate. Fix: cross-sign + `ovs-appctl vlog/reopen` (no restart).

**Failure 4 — Hostname verification fail.** Log: `ssl: server cert CN mismatch`. Fix: re-issue SAN multi-hostname.

---

## Điểm cốt lõi cần nhớ

Thứ nhất, production OVS PHẢI dùng `ssl:` cho mọi Controller/Manager target. `ptcp:` chỉ lab ngắn hạn.

Thứ hai, `ovs-pki` wrap OpenSSL cho workflow PKI tối giản — đủ cho setup, không thay thế CA enterprise (Vault, step-ca) cho deployment lớn.

Thứ ba, CA rotation phải theo trình tự: add new CA first, then switch cert. Ngược lại break handshake.

Thứ tư, bootstrap mode (`bootstrap_ca_cert=true`) chỉ cho first-time — disable ngay sau operational để tránh MITM.

Thứ năm, Anatomy Template A `list SSL` 9-attribute. Private key 644 = security bug #1.

Thứ sáu, self-signed = technical debt. Always use CA.

Thứ bảy, cert rotation zero-downtime qua cross-sign + reload (no restart). Xem Phần 20.1 §20.1.15 + Phần 10.6.

## 13.10.1 Tại sao DHCP phải được hiện thực trong data plane?

Trong kiến trúc truyền thống, DHCP server là một tiến trình userspace (ISC DHCPD, `dnsmasq`, Kea) chạy trên một host, lắng nghe broadcast DHCPDISCOVER trên một network interface. Mô hình này không phù hợp với OVN vì:

**Vấn đề 1 — Broadcast không đi qua được chassis:** trong overlay network, pod trên chassis A broadcast DHCPDISCOVER; nếu DHCP server chạy trên chassis B, broadcast phải được flood qua tunnel Geneve tới mọi chassis. Flooding này tốn băng thông và chỉ dùng cho DHCP.

**Vấn đề 2, Điểm fail duy nhất:** DHCP server userspace duy nhất → nếu crash, pod mới không lấy được IP. Phải triển khai cụm HA DHCP, phức tạp.

**Vấn đề 3 — Mở rộng:** cụm Kubernetes 10K pod với pod thay đổi cao (scale up/down liên tục) có thể sinh hàng nghìn DHCPDISCOVER per giây, một DHCP server userspace không đáp ứng nổi.

**Giải pháp OVN — DHCP native:** mỗi ovn-controller trên mỗi chassis tự trả lời DHCP request cho pod cục bộ. Không có broadcast đi qua chassis, không có server đơn lẻ, mở rộng tuyến tính theo số chassis. Cơ chế:
- CMS định nghĩa `DHCP_Options` trong NB DB (subnet, gateway, DNS servers, lease_time, v.v.).
- Mỗi LSP được gán `dhcpv4_options` trỏ tới DHCP_Options record.
- ovn-northd sinh Logical Flow trong table `ls_in_dhcp_options` (match DHCPDISCOVER + DHCPREQUEST) và `ls_in_dhcp_response` (sinh DHCPOFFER + DHCPACK).
- ovn-controller dịch sang OpenFlow với action `put_dhcp_opts`, action này build DHCPOFFER/DHCPACK packet trực tiếp trong kernel datapath, không cần userspace.

## 13.10.2 Cấu trúc table DHCP_Options

Table `DHCP_Options` trong NB DB có các field chính:
- `cidr`, subnet mà DHCP phục vụ (ví dụ `10.0.0.0/24`).
- `options`, map từ mã DHCP option → giá trị:
  - `server_id` (option 54), IP của DHCP server (thường là gateway IP).
  - `server_mac` (option đặc biệt), MAC của server, dùng làm MAC nguồn của DHCP response.
  - `lease_time` (option 51), thời gian lease (giây).
  - `router` (option 3), default gateway cho client.
  - `dns_server` (option 6), DNS server list.
  - `netmask` (option 1), subnet mask.
  - `domain_name` (option 15), domain search.
  - `mtu` (option 26), MTU cho interface.
  - `classless_static_route` (option 121), định tuyến static ngoài default.
- `external_ids`, map tự do cho CMS label.

Ví dụ cấu hình:
```
[host]# ovn-nbctl dhcp-options-create 10.0.0.0/24
<uuid_dhcp>
[host]# ovn-nbctl dhcp-options-set-options <uuid_dhcp> \
         server_id=10.0.0.1 \
         server_mac="ca:fe:ba:be:00:01" \
         lease_time=3600 \
         router=10.0.0.1 \
         dns_server="8.8.8.8,8.8.4.4" \
         netmask=255.255.255.0 \
         mtu=1500
```

**Gán DHCP options vào LSP:**
```
[host]# ovn-nbctl lsp-set-dhcpv4-options vm1 <uuid_dhcp>
```

Khi LSP được gán dhcpv4_options, ovn-northd sinh Logical Flow xử lý DHCP request từ port đó.

## 13.10.3 Logical Flow xử lý DHCP

ovn-northd sinh 3 nhóm Logical Flow cho DHCP:

**Nhóm 1, ls_in_dhcp_options (table 16):**
Match DHCPDISCOVER + DHCPREQUEST từ client, apply action `put_dhcp_opts` để build response:
```
table=16 (ls_in_dhcp_options), priority=100,
  match=(inport == "vm1" && eth.src == aa:bb:cc:dd:ee:01 && ip4.src == 0.0.0.0
         && ip4.dst == 255.255.255.255 && udp.src == 68 && udp.dst == 67),
  action=(reg0[3] = put_dhcp_opts(offerip = 10.0.0.5,
                                   server_id = 10.0.0.1,
                                   router = 10.0.0.1,
                                   dns_server = {8.8.8.8, 8.8.4.4},
                                   lease_time = 3600,
                                   netmask = 255.255.255.0);
          next;)
```
Action `put_dhcp_opts` là action tùy chỉnh của OVN, nó thay payload DHCP của packet hiện tại thành DHCPOFFER hoặc DHCPACK, đồng thời tráo src/dst MAC và IP để packet đi ngược lại về client. `reg0[3]` = 1 nếu thành công, 0 nếu fail (ví dụ offerip không thuộc subnet).

**Nhóm 2, ls_in_dhcp_response (table 17):**
Sau khi packet đã được biến đổi thành DHCPOFFER/DHCPACK, table 17 resubmit để packet đi qua pipeline ngược lại về client:
```
table=17 (ls_in_dhcp_response), priority=100,
  match=(reg0[3] == 1),
  action=(eth.dst = eth.src; eth.src = aa:aa:aa:aa:aa:aa; /* server_mac */
          ip4.src = 10.0.0.1; ip4.dst = 10.0.0.5; udp.src = 67; udp.dst = 68;
          outport = inport; flags.loopback = 1; output;)
```

**Nhóm 3 — DHCP drop (table 16 fallback):**
Nếu DHCP request không khớp với DHCP_Options của port, drop:
```
table=16, priority=0, match=(udp.dst == 67), action=(drop;)
```

Logic này đảm bảo chỉ port có cấu hình DHCP mới nhận được response, port không có dhcpv4_options → request bị drop (ngăn chặn DHCP snooping).

## 13.10.4 DHCPv6 và stateless configuration

OVN hỗ trợ DHCPv6 qua table `DHCPv6_Options` tương tự DHCP_Options nhưng với mã option IPv6. Ngoài ra OVN hỗ trợ **SLAAC (StateLess Address AutoConfiguration)** qua Router Advertisement (RA), không cần DHCPv6 server:
- ovn-northd sinh Logical Flow gửi RA định kỳ từ Logical Router Port (LRP) có IPv6 address.
- Pod IPv6 nhận RA, tự cấu hình IPv6 address theo prefix (global unicast từ LRP + EUI-64 hoặc random).

Cấu hình SLAAC:
```
[host]# ovn-nbctl set Logical_Router_Port lrp1 ipv6_ra_configs:address_mode=slaac
[host]# ovn-nbctl set Logical_Router_Port lrp1 ipv6_ra_configs:send_periodic=true
[host]# ovn-nbctl set Logical_Router_Port lrp1 ipv6_ra_configs:max_interval=600
```

**Kết hợp DHCPv6 stateful + SLAAC stateless:** một số network cần IPv6 tự cấu hình cho phần lớn client (SLAAC) nhưng DHCPv6 cho DNS server + domain name (DHCPv6 stateless mode, chỉ options, không address). OVN hỗ trợ qua `ra_configs:address_mode=dhcpv6_stateless`.

## 13.10.5 DNS native trong OVN

OVN cũng hỗ trợ DNS resolver native trong data plane, qua table `DNS` trong NB DB. Không cần triển khai DNS server (CoreDNS, `dnsmasq`) riêng cho cluster.

**Cấu trúc table DNS:**
- `records`, map từ "fqdn" → "ipv4_list,ipv6_list" (ví dụ `"pod1.cluster.local" → "10.0.0.5"`).
- `external_ids`, label tự do.

**Gắn DNS vào Logical Switch:**
```
[host]# ovn-nbctl create DNS records={"\"vm1.ovn.local\""="10.0.0.10"}
<uuid_dns>
[host]# ovn-nbctl add Logical_Switch ls1 dns_records <uuid_dns>
```

**Logical Flow xử lý DNS query:**
Table `ls_in_dns_lookup` (table 18) match DNS query packet (UDP dst port 53), apply action `dns_lookup()` để tra cứu table DNS:
```
table=18 (ls_in_dns_lookup), priority=100,
  match=(udp.dst == 53),
  action=(reg0[4] = dns_lookup(); next;)
```
Action `dns_lookup` tìm fqdn từ DNS query trong table DNS, nếu tìm thấy build DNS response trong packet hiện tại.

Table `ls_in_dns_response` (table 19) gửi response ngược lại client:
```
table=19 (ls_in_dns_response), priority=100,
  match=(reg0[4] == 1),
  action=(eth.dst <-> eth.src; ip4.src <-> ip4.dst; udp.src <-> udp.dst;
          outport = inport; flags.loopback = 1; output;)
```

**Hạn chế DNS native:** chỉ hỗ trợ A, AAAA, PTR, SRV record. Không hỗ trợ zone transfer, DNSSEC, recursive resolution (nếu query fqdn không trong table DNS → drop, không forward lên DNS upstream). Môi trường production cần recursive DNS thường triển khai CoreDNS làm upstream cho OVN native DNS.

## 13.10.6 Tích hợp với external DHCP/DNS

Trong môi trường doanh nghiệp, DHCP/DNS thường có IPAM trung tâm (Infoblox, BIND), không thể thay thế bằng OVN native. OVN hỗ trợ hai mô hình tích hợp:

**Mô hình 1 — DHCP Relay:** OVN hiện thực `localnet` port type cho phép bridge Logical Switch với external VLAN. DHCP request từ pod broadcast trong Logical Switch, OVN flood ra localnet port tới external DHCP server. External server reply, OVN forward về pod. Mô hình này giữ DHCP server tập trung nhưng hy sinh tính phi tập trung của OVN.

**Mô hình 2 — External DNS upstream:** OVN native DNS có thể forward query không phân giải được qua upstream. OVN 22.03+ hỗ trợ `DNS.options:upstream_dns_servers`, nếu DNS native không match, forward query tới upstream. Kết hợp cache cục bộ cho tra cứu nhanh + external upstream cho full DNS namespace.

## 13.10.Y `put_dhcp_opts` + `dns_lookup` action source-level

DHCP và DNS native trong OVN không phải userspace daemon (như dnsmasq). Implement qua 2 OpenFlow action: `put_dhcp_opts` (DHCP) và `dns_lookup` (DNS).

### `put_dhcp_opts` action

OVS action implement trong `ofproto/ofproto-dpif-xlate.c` + kernel datapath path. Khi packet match DHCP request (UDP 67/68), action build DHCP reply ngay trong datapath:

Source anchor (`branch-22.03`):

- [`controller/pinctrl.c`](https://github.com/ovn-org/ovn/blob/branch-22.03/controller/pinctrl.c), function `pinctrl_handle_put_dhcp_opts()`. ovn-controller chạy thread riêng (`pinctrl`) handle các action cần userspace (DHCP, ARP responder, ND).
- Function flow: receive controller-bound packet → parse DHCP options field → look up `DHCP_Options` row from NBDB → build DHCP reply → inject về OVS datapath.

Pseudo-code workflow:

```c
// pinctrl_handle_put_dhcp_opts() simplified
dhcp_request = parse_packet(skb);
dhcp_opts = lookup_dhcp_options_for_lsp(lsp_uuid);
dhcp_reply = build_reply(dhcp_request, dhcp_opts);
inject_packet(dhcp_reply, lsp_inport);
```

Latency: < 1ms p50, < 5ms p99 cho cluster healthy. So với dnsmasq userspace daemon (10-50ms), OVN native nhanh hơn 10x.

### `dns_lookup` action

Tương tự cho DNS, nhưng dữ liệu lookup từ NBDB `DNS` table:

```bash
[host]# ovn-nbctl create DNS records='{"web.app=10.0.0.10","db.app=10.0.0.20"}'
[host]# ovn-nbctl set Logical_Switch ls1 dns_records=@dns
```

Action `dns_lookup` parse DNS query packet, match `qname` với `records` field, build DNS response. Function: `pinctrl_handle_dns_lookup()` trong `controller/pinctrl.c`.

Limitation: chỉ support A/AAAA record lookup từ static map. Không recursive resolver, không CNAME chain. Cho service discovery internal (Pod-to-Service) đủ; không thay được public DNS.

### Pipeline NBDB → SBDB compile

ovn-northd compile NBDB `DHCP_Options` thành Logical_Flow:

```
Stage ls_in_dhcp_options:
  priority=100, match=(inport == "vm1" && eth.src == aa:00:00:00:00:01 &&
                       ip4.src == 0.0.0.0 && ip4.dst == 255.255.255.255 &&
                       udp.src == 68 && udp.dst == 67),
  action=(reg0[3] = put_dhcp_opts(...);
          next;)

Stage ls_in_dhcp_response:
  priority=100, match=(inport == "vm1" && reg0[3] == 1),
  action=(eth.dst = eth.src;
          eth.src = 40:44:00:00:00:01;
          ip4.dst = ip4.src;
          ip4.src = 10.0.0.1;
          udp.src = 67;
          udp.dst = 68;
          outport = inport;
          flags.loopback = 1;
          output;)
```

2 stage: `ls_in_dhcp_options` build reply payload, `ls_in_dhcp_response` swap header (src ↔ dst) + output back qua inport (loopback flag).

## 13.10.Z Anatomy Template A: `ovn-nbctl get Logical_Switch_Port dhcpv4_options`

> **Bối cảnh khi chạy:** Kiểm tra DHCP options binding cho LSP cụ thể, debug khi VM không nhận IP.

```bash
[host]# ovn-nbctl get Logical_Switch_Port vm1 dhcpv4_options
abc12345-abcd-1234-...

[host]# ovn-nbctl list DHCP_Options abc12345-...
_uuid               : abc12345-abcd-1234-...
cidr                : "10.0.0.0/24"
external_ids        : {neutron:subnet_id="..."}
options             : {dns_server="{8.8.8.8,8.8.4.4}",
                       domain_name="lab.local",
                       lease_time="3600",
                       mtu="1442",
                       netmask="255.255.255.0",
                       router="10.0.0.1",
                       server_id="10.0.0.1",
                       server_mac="40:44:00:00:00:01"}
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| `cidr` | Match LS subnet | Mismatch | Empty | LSP-LS subnet conflict |
| `options:server_id` | IP gateway của LS | Empty | Mismatch với router | DHCP reply src IP confusion |
| `options:server_mac` | MAC unique cho gateway | Empty | Conflict | DHCP reply src MAC duplicate |
| `options:lease_time` | 3600 (1h) baseline | < 60s (frequent renew) | > 86400 (cache stale) | Tune theo VM lifecycle |
| `options:router` | Default gateway IP | Empty | Khác `server_id` | Routing broken |
| `options:dns_server` | 1+ DNS server | Empty | Unreachable IP | DNS resolution fail |
| `options:netmask` | Match `cidr` mask | Mismatch | Empty | VM thinks subnet sai |
| `options:mtu` | Match underlay - overhead (1442 cho Geneve) | 1500 default | Underlay mismatch | PMTUD black hole; xem 11.1 |

**Kịch bản bẻ gãy:**

- `dhcpv4_options` UUID empty: LSP không có DHCP config. VM phải dùng static IP. Add: `ovn-nbctl --id=@dhcp create DHCP_Options ... -- set Logical_Switch_Port vm1 dhcpv4_options=@dhcp`.
- `cidr` mismatch với LS: VM nhận IP wrong subnet. Recreate `DHCP_Options` với `cidr` đúng.
- `mtu=1500` cho Geneve underlay 1500 → fragmentation. Set `mtu=1442` hoặc tăng underlay (Phần 11.1).

## 13.10.W Capstone POE: OVN native DHCP vs dnsmasq

> **Bối cảnh:** Cluster cần serve DHCP cho 1000 VM/container đang start cùng lúc (cold cluster boot). Kỹ sư chọn dnsmasq vì dễ debug + có log file rõ ràng.

**Tuyên bố cần kiểm chứng:** "dnsmasq đơn giản hơn nên ổn định hơn cho mass DHCP burst (1000 VM cùng start)."

**Predict:** Sai. dnsmasq single-threaded userspace daemon → bottleneck với burst, latency p99 cao.

**Observe (giả định):**

- **dnsmasq setup:** 1 process trên 1 chassis (gateway). 1000 VM cùng broadcast DHCPDISCOVER → dnsmasq queue.
  - Latency p50: 50ms (process queue + parse).
  - Latency p99: 2-5 giây (queue full + retransmit).
  - Drop: 5-10% (UDP socket queue overflow).
  - VM affected: 50-100 VM cần retry.

- **OVN native DHCP:** mỗi chassis có ovn-controller, mỗi chassis tự handle DHCP cho LSP local. Phân tán parallel.
  - Latency p50: 0.5ms.
  - Latency p99: < 5ms.
  - Drop: 0%.
  - All 1000 VM bootstrap < 2 giây total.

**Explain:**

1. **dnsmasq centralized**: 1 process queue 1000 request → serial processing → queue overflow.
2. **OVN native distributed**: N chassis × M VM per chassis = parallel processing. Mỗi chassis chỉ handle subset → no bottleneck.
3. **Latency**: dnsmasq userspace context-switch + UDP socket; OVN datapath inline reply trong kernel/userspace OVN-controller (single fast hop).
4. **Debug**: dnsmasq có log file rõ; OVN dùng `ovn-trace ls1 'inport=...&&udp.dst=67'` + `ovs-ofctl dump-flows br-int | grep put_dhcp_opts`. Khác phương pháp nhưng đều có visibility.

**Falsification test:**

- Setup 100-VM mass boot lab. Đo DHCP latency p50/p99 với dnsmasq vs OVN native.
- Nếu dnsmasq p99 < OVN p99 → counter-example. Trong thực tế, OVN luôn thắng cho burst > 50 concurrent.

**Tuyên bố sai ở đâu:** "Đơn giản hơn nên ổn định hơn" giả định scale nhỏ. Cho mass DHCP burst, distributed beat centralized. OVN native = production-grade cho cluster lớn.

> **Quy tắc:** OVN native DHCP cho production cluster (10+ chassis, 100+ VM). dnsmasq cho dev/test environment hoặc bare-metal không có OVN.

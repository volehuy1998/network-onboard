## Bối cảnh: tại sao cần một playbook có hệ thống

OVS là hệ thống phức tạp với bốn lớp (config → OpenFlow → datapath → wire) và nhiều thành phần tương tác. Khi có sự cố "packet không đi qua", cách tiếp cận ad-hoc — chạy lệnh random xem có output gì không — tốn giờ mà không tìm ra root cause.

Playbook troubleshooting có hệ thống giải quyết vấn đề này bằng cách đưa ra trình tự kiểm tra cố định, mỗi bước kiểm tra một giả thuyết cụ thể, loại trừ một vùng có thể gây lỗi. Compass Appendix C tóm tắt triết lý: "*Are packets arriving? → tcpdump on physical NIC. Are they entering OVS? → `dpctl/dump-flows` hit counters. Are the right flows matching? → `ofproto/trace`. Is the action correct? → read the Datapath actions line. Does the datapath agree? → `dpctl/dump-flows` on the actual kernel/netdev. Does the wire agree? → tcpdump on egress NIC. When all four agree and packets still drop → upstream switch.*"

Part này trình bày năm công cụ CLI, sau đó tích hợp thành 6-layer playbook.

## 9.4.1 ovs-vsctl: transactional config language

`ovs-vsctl` là công cụ quản lý OVSDB config — thay đổi gì trên bridge, port, interface đều qua `ovs-vsctl`.

Compass Ch A nêu bản chất: "`ovs-vsctl` không phải shell bao quanh các lệnh đơn lẻ. Nó là transactional compiler: một invocation duy nhất trở thành một OVSDB transaction, và `--` là statement separator bên trong transaction đó."

Quan trọng ở điểm: nhiều operation trong cùng `ovs-vsctl` commit atomic. Nếu operation thứ 3 lỗi, operation 1-2 không được apply. Điều này tránh half-configured state.

Grammar (compass Ch A):

```
ovs-vsctl [global-opts] command [-- command]...
command : verb table [row] [column[:key][=value]]...
```

Verbs cho scalar/composite column: `list`, `find`, `get`, `set`, `add`, `remove`, `clear`. Verbs cho row: `create`, `destroy`, `--may-exist`, `--if-exists`. Reference qua `--id=@name`.

Ví dụ transaction tạo port với QoS + hai queue:

```bash
[host]# ovs-vsctl -- set port eth0 qos=@newqos \
                  -- --id=@newqos create QoS type=linux-htb \
                                  queues:0=@q0 queues:1=@q1 \
                  -- --id=@q0 create Queue other-config:max-rate=10000000 \
                  -- --id=@q1 create Queue other-config:max-rate=20000000
```

Patterns phổ biến:

```bash
[host]# ovs-vsctl list-br                        # liệt kê bridge
[host]# ovs-vsctl list-ports br0                 # liệt kê port của bridge
[host]# ovs-vsctl list Interface eth0            # dump mọi column của interface
[host]# ovs-vsctl find Port tag=100              # tìm port theo tag VLAN
[host]# ovs-vsctl get Interface eth0 statistics  # đọc counter
[host]# ovs-vsctl set Bridge br0 fail_mode=secure
```

## 9.4.2 ovs-ofctl: OpenFlow flow management

`ovs-ofctl` nói OpenFlow với bridge — dump flow, thêm/sửa/xóa flow entry, monitor flow events.

Quan trọng: luôn pass `-O OpenFlow13` (hoặc cao hơn) cho pipeline multi-table. Không pass → OVS default OpenFlow 1.0, không hỗ trợ group, meter, nhiều table.

```bash
[host]# ovs-ofctl -O OpenFlow13 show br0              # port list + capabilities
[host]# ovs-ofctl -O OpenFlow13 dump-flows br0        # mọi flow entry
[host]# ovs-ofctl -O OpenFlow13 dump-flows br0 table=10,dl_dst=aa:bb:cc:00:00:01
[host]# ovs-ofctl -O OpenFlow13 add-flow br0 "table=0,priority=100,in_port=1,actions=goto_table:10"
[host]# ovs-ofctl -O OpenFlow13 mod-flows br0 "in_port=1,actions=output:2"
[host]# ovs-ofctl -O OpenFlow13 del-flows br0 "in_port=1"
[host]# ovs-ofctl -O OpenFlow13 monitor br0 watch:    # live flow event stream
```

**Flow-table hygiene** (compass Ch 10):

- `replace-flows br0 new.flows` — atomic swap toàn bộ flow table (vs `add-flows` chỉ append).
- `diff-flows br0 new.flows` — cho biết khác biệt giữa current và new file, không apply.

Production rule: trước khi `replace-flows`, luôn chạy `diff-flows` trước.

## 9.4.3 ovs-appctl: introspection RPC qua Unix socket

`ovs-appctl` là command duy nhất nói chuyện với bất kỳ OVS daemon nào qua Unix socket. Target được suffix sau `-t`: `ovs-vswitchd`, `ovsdb-server`, `ovn-controller`.

Lệnh quan trọng nhất — `ofproto/trace` — mô phỏng đường đi của một packet qua OpenFlow pipeline **không gửi packet thật**. Đây là công cụ debug mạnh nhất trong OVS:

```bash
[host]# ovs-appctl ofproto/trace br0 \
         "in_port=1,dl_src=aa:bb:cc:00:00:01,dl_dst=aa:bb:cc:00:00:02,dl_type=0x0800,\
          nw_src=10.0.0.1,nw_dst=10.0.0.2,nw_proto=6,tp_src=50000,tp_dst=80"
```

Output walk qua từng table, mỗi rule match, mỗi action, và cuối cùng là `Datapath actions:` — tóm tắt gì kernel sẽ làm. Nếu thấy `Datapath actions: drop` → flow bị drop; `Datapath actions: 3,4` → xuất ra port 3 và 4.

Thêm `-generate` để thực sự trigger side effect (ví dụ: trigger `learn()` action):

```bash
[host]# ovs-appctl ofproto/trace br0 "..." -generate
```

Các lệnh appctl khác (compass Ch R):

```bash
[host]# ovs-appctl coverage/show          # counter cho mọi internal event
[host]# ovs-appctl memory/show             # memory breakdown
[host]# ovs-appctl upcall/show             # upcall statistics, flow-install rate
[host]# ovs-appctl fdb/show br0            # MAC learning table
[host]# ovs-appctl bond/show bond0         # bond state
[host]# ovs-appctl lacp/show bond0         # LACP actor/partner
[host]# ovs-appctl stp/show                # spanning-tree
[host]# ovs-appctl tnl/neigh/show          # tunnel neighbor ARP
[host]# ovs-appctl vlog/set dpif:file:dbg  # tăng log level module
```

## 9.4.4 ovs-dpctl: raw datapath inspection

`ovs-dpctl` nói trực tiếp với kernel datapath (hoặc userspace dp trong DPDK). Hiếm khi dùng cho thao tác ghi — chỉ để đọc trạng thái hiện tại.

```bash
[host]# ovs-dpctl show                     # datapath list + port list + counter
[host]# ovs-dpctl dump-flows -m            # mọi megaflow entry với mask
[host]# ovs-dpctl dump-conntrack           # conntrack table của OVS
```

Output `dump-flows -m` là "ground truth" về gì kernel thực sự forward.

---

## 9.4.5 ovsdb-client: raw OVSDB access

`ovsdb-client` dùng khi debug OVSDB issue — monitor change stream, backup/restore:

```bash
[host]# ovsdb-client list-dbs unix:/var/run/openvswitch/db.sock
[host]# ovsdb-client dump Open_vSwitch
[host]# ovsdb-client monitor Open_vSwitch Bridge
[host]# ovsdb-client backup > conf.db.bak
```

## 9.4.6 Six-layer troubleshooting playbook

Khi gặp sự cố "packet không đi qua OVS đúng cách", đi tuần tự sáu lớp từ dưới lên trên:

*Table 9-4: 6-layer troubleshooting playbook*

| Lớp | Câu hỏi | Lệnh kiểm tra | Nếu fail |
|---|---|---|---|
| **L1 Physical** | Packet có đến NIC không? | `tcpdump -i eth0` | Kiểm tra cable, NIC driver, SFP |
| **L2 Kernel driver** | Packet có vào OVS không? | `ovs-dpctl show` — counter `lookups:missed` tăng khi có traffic | NIC không unbind đúng với DPDK, driver thiếu feature |
| **L3 Megaflow cache** | Có hit megaflow nào không? | `ovs-dpctl dump-flows -m` | Cache miss liên tục → vấn đề OpenFlow pipeline ở L4 |
| **L4 OpenFlow pipeline** | Flow nào match? Action gì? | `ovs-appctl ofproto/trace br0 "..."` | Sai rule priority, missing table, action `drop` không mong muốn |
| **L5 OVSDB config** | Port, VLAN, bridge có đúng không? | `ovs-vsctl show`, `ovs-vsctl list Interface <name>` | Port thiếu VLAN tag, interface type sai |
| **L6 Logical (OVN)** | Logical Switch, Port_Binding đúng không? | `ovn-sbctl show`, `ovn-sbctl lflow-list` | Logical flow compile sai |

**Quy tắc**: khi cả L1-L5 OK nhưng packet vẫn drop — vấn đề nằm ngoài OVS (upstream switch, firewall, ARP không resolve, MTU mismatch). Không tiếp tục debug OVS mà quay sang network path.

### ▶ Guided Exercise 5 / Trouble Ticket 1: Packet drop diagnosis với ofproto/trace

**Problem:** User báo "không ping được từ VM1 (10.0.0.1) đến VM2 (10.0.0.2), cả hai cùng bridge br-int".

**Before You Begin:**

```bash
[host]# ovs-vsctl add-br br-int
[host]# ovs-vsctl add-port br-int tap1 -- set Interface tap1 type=internal
[host]# ovs-vsctl add-port br-int tap2 -- set Interface tap2 type=internal
[host]# ovs-ofctl -O OpenFlow13 add-flow br-int \
         "table=0,priority=100,in_port=tap1,dl_type=0x0806,actions=output:tap2"
[host]# ovs-ofctl -O OpenFlow13 add-flow br-int \
         "table=0,priority=50,in_port=tap1,dl_type=0x0800,actions=drop"
```

**Instructions:**

**1.** Trace ARP request:

```bash
[host]# ovs-appctl ofproto/trace br-int \
         "in_port=tap1,dl_src=aa:bb:cc:00:00:01,dl_dst=ff:ff:ff:ff:ff:ff,dl_type=0x0806"
```

Output cuối: `Datapath actions: <ofport_of_tap2>`. ARP đi qua — OK.

**2.** Trace ICMP (bị drop):

```bash
[host]# ovs-appctl ofproto/trace br-int \
         "in_port=tap1,dl_src=aa:bb:cc:00:00:01,dl_dst=aa:bb:cc:00:00:02,\
          dl_type=0x0800,nw_src=10.0.0.1,nw_dst=10.0.0.2,nw_proto=1"
```

Output:

```
Rule: table=0 cookie=0 priority=50,in_port=tap1,ip actions=drop
Datapath actions: drop
```

→ Root cause: rule priority 50 drop IP. Admin cần xóa hoặc sửa priority.

**3.** Sửa:

```bash
[host]# ovs-ofctl -O OpenFlow13 del-flows br-int "table=0,in_port=tap1,dl_type=0x0800"
[host]# ovs-ofctl -O OpenFlow13 add-flow br-int \
         "table=0,priority=100,in_port=tap1,dl_type=0x0800,actions=output:tap2"
```

**4.** Retrace xác nhận → `Datapath actions: <ofport_of_tap2>`. Fixed.

**Finish:** `ovs-vsctl del-br br-int`

## Điểm cốt lõi cần nhớ

Thứ nhất, năm công cụ CLI — `ovs-vsctl`, `ovs-ofctl`, `ovs-appctl`, `ovs-dpctl`, `ovsdb-client` — mỗi cái có vai trò riêng biệt, không thay thế nhau.

Thứ hai, `ofproto/trace` là công cụ mạnh nhất trong bộ — mô phỏng packet flow không cần gửi packet thật. Mọi kỹ sư OVS phải thuộc cú pháp match và đọc output trace.

Thứ ba, 6-layer playbook là phương pháp debug có hệ thống — đi từ L1 vật lý lên L6 logical, loại trừ từng vùng. Khi cả sáu lớp đều OK mà packet vẫn drop, vấn đề nằm ngoài OVS.

Thứ tư, OVSDB transaction atomicity (ovs-vsctl) và OpenFlow replace-flows atomicity bảo vệ khỏi half-configured state — luôn `diff-flows` trước khi `replace-flows` trong production.

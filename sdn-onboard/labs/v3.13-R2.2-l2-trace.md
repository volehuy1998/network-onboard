## What R2.2 builds, and why a Cisco engineer needs the network namespace

### The topology

R2.2 builds the simplest possible L2 topology that proves OVS forwards frames between two endpoints. Two Linux network namespaces, `ns1` and `ns2`, each holding one half of a `veth` pair. The other halves of the two veth pairs (`br-ns1` and `br-ns2`) sit in the host's root namespace and are added as ports on the OVS bridge `br0`. From inside `ns1` the operator runs `ping 10.0.0.2`; the reply comes back from inside `ns2`. The lab proves that OVS bridged the ARP request, OVS bridged the ARP reply, OVS bridged the ICMP echo request, and OVS bridged the ICMP echo reply, all four on the basis of the priority-0 NORMAL synthesis from R2.1, learning the source MACs into `fdb/show` and using the learned MACs to direct the next frame.

The Cisco engineer reads "two namespaces" and asks: why namespaces and not two physical hosts. The answer is pedagogical and operational. Pedagogically, two namespaces let the lab run on a single VM with no second NIC, which is the same reason Open vSwitch documentation, the OVN tutorial, and the Linux Foundation training all use namespaces; the reader will see this idiom in every later sub-capture and in production troubleshooting on real OpenStack and Kubernetes deployments. Operationally, a network namespace is what a Linux kernel uses to implement an isolated forwarding domain. From the kernel's point of view, two namespaces with two veth ends are *exactly* as separated as two physical hosts on two physical NICs; the kernel runs the same netdev driver code path, the same skb allocation path, the same ARP resolver, and the same ICMP echo handler. R2.2 is therefore not a toy approximation of inter-host traffic; it is the same kernel forwarding plane stripped of the wire.

## Build the topology

### From zero to a connected bridge

The reader can read this section as one continuous procedure: create the bridge, create the namespaces, create the veth pairs, place each pair's far end into a namespace, IP-and-up both namespace ends, plug the bridge ends into the bridge, bring the bridge ends and the bridge itself up. Each step is one command; together they replace what a Cisco engineer would do across `configure terminal`, `interface vlan`, `ip address`, and `no shutdown` on a Catalyst.

```text
root@lab-openvswitch:~/network-onboard# date -u
Thu Apr 30 04:34:19 AM UTC 2026
root@lab-openvswitch:~/network-onboard# ovs-vsctl list-br
root@lab-openvswitch:~/network-onboard# ip netns list
root@lab-openvswitch:~/network-onboard# ovs-vsctl --may-exist add-br br0
root@lab-openvswitch:~/network-onboard# ip netns add ns1
root@lab-openvswitch:~/network-onboard# ip netns add ns2
root@lab-openvswitch:~/network-onboard# ip link add veth-ns1 type veth peer name br-ns1
root@lab-openvswitch:~/network-onboard# ip link add veth-ns2 type veth peer name br-ns2
root@lab-openvswitch:~/network-onboard# ip link set veth-ns1 netns ns1
root@lab-openvswitch:~/network-onboard# ip link set veth-ns2 netns ns2
root@lab-openvswitch:~/network-onboard# ip netns exec ns1 ip addr add 10.0.0.1/24 dev veth-ns1
root@lab-openvswitch:~/network-onboard# ip netns exec ns2 ip addr add 10.0.0.2/24 dev veth-ns2
root@lab-openvswitch:~/network-onboard# ip netns exec ns1 ip link set veth-ns1 up
root@lab-openvswitch:~/network-onboard# ip netns exec ns2 ip link set veth-ns2 up
root@lab-openvswitch:~/network-onboard# ip netns exec ns1 ip link set lo up
root@lab-openvswitch:~/network-onboard# ip netns exec ns2 ip link set lo up
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br0 br-ns1
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br0 br-ns2
root@lab-openvswitch:~/network-onboard# ip link set br-ns1 up
root@lab-openvswitch:~/network-onboard# ip link set br-ns2 up
root@lab-openvswitch:~/network-onboard# ip link set br0 up
```

Three classes of command appear here. The OVS commands (`ovs-vsctl add-br`, `ovs-vsctl add-port`) are database transactions; the daemon notices the change through its IDL handle and reconfigures itself within a few milliseconds. The kernel-network commands (`ip netns add`, `ip link add type veth`, `ip link set ... netns`, `ip addr add`, `ip link set ... up`) are direct calls into the kernel via `rtnetlink`; they take effect synchronously and are independent of OVS's database. The cross-namespace commands (`ip netns exec ns1 ip ...`) are the way an operator runs a command inside a namespace from the root namespace's shell.

The `lo` (loopback) inside each namespace must be brought up explicitly. A fresh namespace starts with `lo` in `state DOWN`; many user-space tools (including `ping` in some configurations and certain kernel paths) need `lo` up to function. The lab brings `lo` up defensively so the reader is never tripped by a subtle namespace bug; the cost is two extra commands.

The order of operations matters in two places. First, `ip link set veth-ns1 netns ns1` must precede `ip netns exec ns1 ip addr add ...`, because the veth end must already be inside `ns1` before the namespace can configure it. Second, `ovs-vsctl add-port br0 br-ns1` must precede `ip link set br-ns1 up`; OVS notices the addition through the database, takes ownership of the `br-ns1` netdev, and brings it up under its own control. If the operator brings the netdev up before adding it to the bridge, OVS will still take it over, but in the moment between the up-event and the add-port the netdev is reachable from the host's IP stack with whatever the kernel autoconfigured (typically a link-local IPv6); this is harmless on the lab but is the source of subtle bugs in production where two operators race to configure an interface.

`ovs-vsctl show` now reports three ports on `br0`: the auto-created `br0` LOCAL port and the two operator-added `br-ns1`/`br-ns2` ports. The columnar listing pins the three identifiers the reader needs.

The bridge-side veth ends `br-ns1` and `br-ns2` have `type : ""`, the empty-string sentinel that means "system" (the wrapper around an existing Linux netdev). They picked up OpenFlow port numbers `1` and `2` in add-port order. Their MACs `b2:69:97:ec:71:f1` and `be:05:37:3f:59:c9` were assigned by the kernel veth driver at link-creation time; OVS does not change them. The LOCAL port retains `ofport : 65534` (`OFPP_LOCAL`) and a kernel-assigned MAC.

The far ends `veth-ns1` and `veth-ns2` are not listed because they live inside `ns1` and `ns2` respectively; OVS does not know they exist. The kernel's veth driver knows; let us read them.

```text
root@lab-openvswitch:~/network-onboard# ip netns exec ns1 ip link show veth-ns1
116: veth-ns1@if115: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 qdisc noqueue state UP mode DEFAULT group default qlen 1000
    link/ether 2e:56:2c:85:7d:c1 brd ff:ff:ff:ff:ff:ff link-netnsid 0
root@lab-openvswitch:~/network-onboard# ip netns exec ns2 ip link show veth-ns2
118: veth-ns2@if117: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 qdisc noqueue state UP mode DEFAULT group default qlen 1000
    link/ether 2e:65:29:51:d1:e7 brd ff:ff:ff:ff:ff:ff link-netnsid 0
```

Two facts the reader keeps. The namespace-side MACs are `2e:56:2c:85:7d:c1` for `ns1`'s `veth-ns1` and `2e:65:29:51:d1:e7` for `ns2`'s `veth-ns2`. Those are the source MACs that will appear in `fdb/show` after the first frame is sent. The notation `veth-ns1@if115` means "this is interface 116, paired with interface 115 in the peer namespace"; `if115` is the bridge-side `br-ns1`. The relationship between the two integers is the kernel's per-veth backlink; an operator reads it to confirm the veth pair is intact.

The four MAC addresses (two namespace-side and two bridge-side) are the four MAC addresses the reader will track through the rest of the lab. The diagram earlier in this section labels each one; the reader should be able to predict, before sending the first frame, that `fdb/show` will record `2e:56:..:c1` against OF port 1 and `2e:65:..:e7` against OF port 2.

## Read the kernel datapath view

```text
root@lab-openvswitch:~/network-onboard# ovs-appctl dpif/show
system@ovs-system: hit:9 missed:9
  br0:
    br-ns1 1/2: (system)
    br-ns2 2/3: (system)
    br0 65534/1: (internal)
root@lab-openvswitch:~/network-onboard# ovs-appctl fdb/show br0
 port  VLAN  MAC                Age
    1     0  2e:56:2c:85:7d:c1    4
    2     0  2e:65:29:51:d1:e7    3
LOCAL     0  b2:41:ae:5a:58:4d    2
```

`dpif/show` is the same lens R2.0 introduced; on R2.2's bridge it now reports three ports. The notation `br-ns1 1/2:` means OF port 1, DP port 2; `br-ns2 2/3:` means OF port 2, DP port 3; the LOCAL stays at OF port 65534 and DP port 1. The DP port numbers were assigned by the kernel datapath in vport-creation order: DP port 0 is the datapath itself, DP port 1 is the first vport ever added (the LOCAL when the bridge was created), DP ports 2 and 3 are the two veth bridge-ends. The OF/DP asymmetry (OF=1 maps to DP=2, OF=2 maps to DP=3) is normal; the two numbering systems are independent. The line `system@ovs-system: hit:9 missed:9` reports the kernel-cache hit and miss counters: nine packets have been forwarded from the megaflow cache, nine have missed and required an upcall. These nine are the IPv6 link-local autoconfiguration frames the kernel emitted when the veth ends came up; the operator did not send them, but they exist and will appear in the megaflow cache shortly.

`fdb/show br0` is the bridge's L2 learning table. Already populated: three entries, with low ages (4 / 3 / 2 seconds). The bridge has learned the two namespace-side MACs (against OF ports 1 and 2) and its own LOCAL MAC against the LOCAL port. Learning happened *before* the operator-sent ping because the kernel's IPv6 stack issued router-solicitations and Duplicate Address Detection multicasts as soon as `veth-ns1` and `veth-ns2` came up; OVS observed those frames as ingress on ports 1 and 2 and learned the source MACs. The reader who expected `fdb/show` to be empty at this point would be wrong; the kernel's IPv6 chatter learns MACs faster than the operator can type. R6 (conntrack) returns to the IPv6-chatter point in detail.

Note the LOCAL entry: the bridge has learned its own MAC against the LOCAL port. This happens because the kernel's IPv6 stack on the LOCAL netdev (which is up because `ip link set br0 up` was issued in the previous block) emits the same multicasts. Learning the LOCAL's MAC against the LOCAL port has no functional effect (the bridge would never need to send a frame to itself), but it is observable and worth pointing out.

## Send the first frame and read every diagnostic surface

### The ping

```text
root@lab-openvswitch:~/network-onboard# ip netns exec ns1 ping -c 2 -W 2 10.0.0.2
PING 10.0.0.2 (10.0.0.2) 56(84) bytes of data.
64 bytes from 10.0.0.2: icmp_seq=1 ttl=64 time=0.693 ms
64 bytes from 10.0.0.2: icmp_seq=2 ttl=64 time=0.057 ms

--- 10.0.0.2 ping statistics ---
2 packets transmitted, 2 received, 0% packet loss, time 1005ms
rtt min/avg/max/mdev = 0.057/0.375/0.693/0.318 ms
```

Two ICMP echo request / reply round trips. The first round trip took 0.693 ms; the second took 0.057 ms, more than twelve times faster. The asymmetry is the megaflow cache: the first frame missed the cache, paid the full upcall round-trip cost (kernel-to-userspace netlink, `xlate_actions()` translation, userspace-to-kernel reply, action execution), and installed a cache entry. The second frame hit the cache, executed the cached action directly in the kernel, and avoided the userspace round trip entirely. The 0.6 ms difference is the cost of the upcall on this lab host; on a quiet production machine it is typically 50 to 200 microseconds, on a busy one it can spike to milliseconds.

The Cisco analogue is the difference between the first frame against an unprogrammed CAM and the second frame against the programmed CAM; on a Catalyst the gap is microseconds because the slow path is also in hardware, on Open vSwitch the slow path is a userspace process and the gap is correspondingly larger. The reader who instruments OVS in production is paid by understanding this gap; R4 (datapath internals) measures it directly with `coverage/show` and `dpctl/dump-flows`.

### What the bridge learned

```text
root@lab-openvswitch:~/network-onboard# ovs-appctl fdb/show br0
 port  VLAN  MAC                Age
LOCAL     0  b2:41:ae:5a:58:4d    1
    1     0  2e:56:2c:85:7d:c1    0
    2     0  2e:65:29:51:d1:e7    0
```

After the ping, the FDB has the same three entries it had before the ping, but with refreshed ages. The two namespace MACs are at age 0 (just refreshed) because they were source MACs in frames that arrived during the ping. The LOCAL MAC is at age 1 (refreshed in the prior second by the IPv6 chatter). The L2 learning is working as the textbook says it should.

The MAC-port binding is the proof. The bridge knows that `2e:56:..:c1` lives behind OF port 1 (the `br-ns1` interface, which leads to `ns1`), and `2e:65:..:e7` lives behind OF port 2 (`br-ns2`, leading to `ns2`). When a frame arrives on port 1 destined for `2e:65:..:e7`, the bridge looks up the FDB, finds the destination on port 2, and outputs only there; the bridge does not flood. When a frame arrives on port 1 destined for an unknown MAC (broadcast, multicast, or genuinely-unknown unicast), the bridge does flood. R2.8 will exercise the binding in detail.

Thirteen entries in the megaflow cache. This is the reader's first encounter with `dpctl/dump-flows`, which is the operationally most-important diagnostic surface OVS exposes. Reading one entry slowly will pay the reader for years.

The third entry is the ICMP echo request from `ns1` to `ns2`:

```ascii
recirc_id(0),
in_port(2),                                              <- DP port 2 (br-ns1)
eth(src=2e:56:2c:85:7d:c1,                               <- ns1's veth MAC
    dst=2e:65:29:51:d1:e7),                              <- ns2's veth MAC
eth_type(0x0800),                                        <- IPv4
ipv4(frag=no),                                           <- not fragmented
packets:1, bytes:98, used:1.404s,                        <- one frame, 98 bytes,
                                                            last hit 1.4s ago
actions:3                                                <- output to DP port 3
                                                            (br-ns2)
```

The entry is a megaflow because OVS combined the input port, the source MAC, the destination MAC, the EtherType, and the fragmentation bit into a single match key, and elided every other header field (TTL, source IP, destination IP, port numbers, TCP flags). The elision is deliberate: the bridge's policy did not look at those fields, so a megaflow that wildcards them is correct and has more cache hits than a microflow that pins them. R4 will exercise the wildcard mechanism in detail.

The fifth entry is the matching ICMP echo reply from `ns2` to `ns1` (DP port 3 to DP port 2). The two entries together cover the steady-state ICMP exchange. The other eleven entries are the IPv6 link-local chatter: solicited-node multicasts (`33:33:ff:..`), all-nodes multicasts (`33:33:00:00:00:01`, `..:02`, `..:16`), each from a different source MAC and input port. The IPv6 chatter dominates the cache numerically because the kernel's IPv6 stack on every namespace emits more multicast packets per minute than the lab's actual ICMP exchange.

The seventh entry is the lone ARP request that started the whole exchange:

```ascii
recirc_id(0),
in_port(2),                                              <- DP port 2 (br-ns1)
eth(src=2e:56:2c:85:7d:c1, dst=ff:ff:ff:ff:ff:ff),       <- broadcast
eth_type(0x0806),                                        <- ARP
arp(sip=10.0.0.1, tip=10.0.0.2, op=1/0xff),              <- "who has 10.0.0.2"
packets:0, bytes:0, used:never,                          <- counter cleared by
                                                            revalidator;
                                                            see below
actions:1,3                                              <- flood to LOCAL and
                                                            br-ns2 (skip the
                                                            input port)
```

`packets:0, bytes:0, used:never` does not mean the ARP did not happen; it means the megaflow's per-cache counter has been cleared since the last revalidator pass. The revalidator periodically sums and zeroes the per-cache counters into the per-bridge `n_packets` total that `dump-flows` reports. R4 will exercise the revalidator's behaviour in detail; R2.2 only needs the reader to know the counters can be zero on a megaflow that has fired in the past.

The actions list `1,3` is the ARP flood. DP port 1 is the LOCAL (the bridge itself) and DP port 3 is `br-ns2`. The flood deliberately skips DP port 2 (`br-ns1`, the input port) because the IEEE 802.1D learning bridge specification mandates that a flooded frame must not be returned out the input port. R2.5.1 (OVS-versus-Linux-bridge L2 learning) compares OVS's flood policy with the Linux bridge's directly.

## ofproto/trace, the simulator

### What ofproto/trace simulates and what it does not

`ovs-appctl ofproto/trace` is the OVS equivalent of "thinking out loud about a frame". The operator hands the daemon a flow specification (input port, source and destination MACs, EtherType, IP addresses, anything the operator wants to fix); the daemon runs the same `xlate_actions()` translator the upcall path uses, but does not actually emit any packet. The output is the step-by-step translation, the resulting datapath actions, and the megaflow that *would* be installed if a real packet matched this specification. The simulation is an exact replica of the production translation logic; the daemon literally runs the same C function. The simulation does not change the bridge's stored state; in particular, an `ofproto/trace` does not insert an FDB entry, does not install a megaflow, and does not transmit a frame.

The reader uses `ofproto/trace` to ask "what would the bridge do with this frame", and the answer is canonical because it is the same code path as production. The Cisco analogue is `show ip route 10.0.0.2`, which simulates the routing decision for a destination IP without sending a packet; `ofproto/trace` is the same idea applied to the OVS flow table.

### Trace 1: an unknown destination on port 1

```text
root@lab-openvswitch:~/network-onboard# ovs-appctl ofproto/trace br0 'in_port=1,dl_src=ee:ee:ee:ee:01:01,dl_dst=ee:ee:ee:ee:02:02,dl_type=0x0800,nw_src=10.0.0.1,nw_dst=10.0.0.2'

```

Tear down in the reverse of build order. Delete the bridge ports first (which removes them from `br0` but does not delete the netdevs); delete the bridge-side veth ends with `ip link del` (which deletes both ends of each pair, including the namespace-side end automatically); delete the namespaces (`ns1` and `ns2`); delete the bridge. The `2>/dev/null` on the `ip link del` is defensive: if `ovs-vsctl del-port` already triggered a kernel-side cleanup race the netdev may already be gone; suppressing the error is acceptable here because the operator is destroying state, not creating it. R6 onward will not need the suppression because the cleanup logic by then will be tighter.

The closing `date -u` reads 04:34:53 UTC, 34 seconds after the opening `date -u`. Twenty-three of those seconds were build, eight were the ping plus the diagnostic reads, three were the trace plus tear down. The lab is fully reproducible at this wall-clock budget.

One last observation before tear-down. The `dpctl/dump-flows` output from the second read (after the hot-cache ping) showed nine megaflow entries, which is *fewer* than the thirteen entries the first read showed. A reader who has not seen this before may worry: did megaflows get lost? They did not. The revalidator is the answer. `ovs-vswitchd` runs a periodic background scan that re-runs `xlate_actions()` against every megaflow in the cache; if the re-translation produces the same actions the entry stays, if the actions changed the entry is replaced, if the entry has not been hit in a configurable interval (default 10 seconds for unused, with longer timers for actively-used) the entry is evicted. Between the two reads of `dpctl/dump-flows` the revalidator pruned four idle IPv6 chatter entries that had not seen traffic. This is normal hygiene; R4 will measure the revalidator's wakeup interval and the eviction policy directly.

A subtle interaction worth noting at tear-down time. Deleting an OVS port via `ovs-vsctl del-port br0 br-ns1` triggers a daemon-side reconfigure: `ovs-vswitchd` notices the OVSDB row removal, removes the corresponding `struct port` from its in-memory bridge, calls `dpif_port_del()` on the kernel datapath to remove the vport. The kernel removes the vport, which detaches the netdev from the OVS data path but does *not* delete the netdev itself; the netdev is still a Linux device, still in the host's root namespace, still reachable via `ip link`. The reader who runs `ovs-vsctl del-port br0 br-ns1` and then `ip link show br-ns1` will see the netdev still present, in DOWN state, no longer associated with any OVS bridge. The subsequent `ip link del br-ns1` is what actually removes the kernel netdev. The two-step semantic (OVS-detach, then kernel-delete) is the consequence of OVS not owning the netdev's lifecycle; the operator does. R2.3 returns to this in the context of `internal` and `tap` ports, where OVS *does* own the netdev's lifecycle and the semantics differ accordingly.

## How the megaflow wildcards make the cache scale

A reader who has built Cisco campus networks may at this point ask: if the kernel cache holds one entry per distinct frame, will the cache not blow up under any non-trivial traffic? The answer is yes, *if* the cache held microflows (one entry per 5-tuple). The cache does not hold microflows. The cache holds *megaflows*, which is the concept that lets OVS scale to millions of distinct flows on commodity hardware.

A megaflow is the union of every microflow that translates to the same action. The translator computes the megaflow by tracking which header fields the policy actually consulted; fields the policy did not consult are wildcarded in the megaflow's match key. Two flows that agreed on the consulted fields (input port, source MAC, destination MAC, EtherType in R2.2's NORMAL case) but differ on TTL, source IP, destination IP, and TCP port numbers will both hit the same megaflow entry. The reader saw this in the third-block ICMP entry: the megaflow matched on `in_port`, `eth(src,dst)`, `eth_type`, and `ipv4(frag=no)`, and wildcarded everything else; every ICMP echo from `ns1` to `ns2` regardless of payload, sequence number, or identifier hits the same entry.

The wildcarding logic is implemented in `xlate_actions()` and the related `wc->masks` machinery in `ofproto/ofproto-dpif-xlate.c`. Every time the translator reads a header field (for a comparison, a match lookup, an action calculation), it stamps the corresponding bits in `wc->masks` to "consulted". When translation finishes, the megaflow's match key is the conjunction of `wc->masks` and the original flow; everything outside the masked bits is wildcard. The contract is precise: any frame that matches the megaflow on the consulted bits would translate to the same actions, by construction. The cache is therefore correct, not approximate.

The pedagogical point. The reader now has three layers of caching to be aware of: the OpenFlow flow table at the policy layer, the kernel megaflow cache at the per-frame layer, and the bridge's L2 FDB at the learning layer. R4 will introduce the EMC (Exact-Match Cache) as a fourth layer that sits in front of the megaflow cache; the four-layer story is the complete picture. R2.2 stops at three layers and the reader can already explain why a steady-state load of 1 million flows per second runs through OVS on modern x86 with no userspace wakeups: nearly every frame hits a cached megaflow, and the megaflow population is bounded by the policy's expressive complexity, not by the traffic mix.

## A walkthrough of the upcall round trip for a single ARP

### The ARP request the operator did not type

The reader saw two pings produce a single ICMP request, a single ICMP reply, and roughly thirteen megaflows. Where did the megaflows come from? The answer is the upcall round trip R2.0 sketched in advance; R2.2 is the first lab with a real frame the reader can trace through it. Walk one frame, the very first ARP request `ns1` issued, end to end.

The Linux kernel inside `ns1` is asked by `ping` to send an ICMP packet to 10.0.0.2. The kernel's IP layer consults its ARP cache, finds nothing for 10.0.0.2, and produces an ARP request: source IP 10.0.0.1, target IP 10.0.0.2, source MAC `2e:56:..:c1`, destination MAC `ff:ff:ff:ff:ff:ff` (broadcast), EtherType 0x0806. The frame is queued on `veth-ns1`.

The veth driver hands the frame to its peer, `br-ns1` in the host's root namespace. The veth driver's `xmit` function calls `netif_rx()` on the peer, which in turn calls into the OVS kernel module's ingress hook (`netdev_frame_hook` in `datapath/datapath.c` upstream; the on-disk path on the lab host is the in-tree kernel module that ships with the running kernel `5.15.0-173-generic`). The hook computes a flow key from the frame's headers and looks up the megaflow cache (`flow_table`) for a match. None is found, because no frame has crossed the bridge yet. The lookup miss increments the `missed` counter the reader saw in `dpif/show` and triggers an upcall.

The kernel module constructs an `OVS_PACKET_CMD_MISS` netlink message containing the frame plus the computed flow key, and emits it on the netlink genl multicast group. `ovs-vswitchd` is parked in `poll(2)` on its netlink socket; the poll wakes; an upcall-handler thread (one of the `n-handler-threads` configured at daemon startup) dequeues the message.

The handler thread calls `xlate_actions()` from `ofproto/ofproto-dpif-xlate.c`. The function walks the bridge's flow table looking for a matching rule. The flow table contains zero stored rules; the only "match" is the synthesised priority-0 NORMAL the connection-manager renders. Translation of the NORMAL action falls into `xlate_normal()` at line 3083. `xlate_normal()` does five things in order. First, it learns the source MAC: `mac_learning_insert(2e:56:..:c1, OF port 1)`. Second, it looks up the destination MAC: `mac_learning_lookup(ff:ff:ff:ff:ff:ff)` returns nothing because broadcast is never in the FDB. Third, it falls into `xlate_normal_flood()` at line 3051. Fourth, the flood path enumerates the bridge's bundles (one per port), skips the input bundle (`br-ns1`), and produces datapath actions `output:1, output:3` (the LOCAL DP port and the `br-ns2` DP port, in DP-port numbering). Fifth, the megaflow is built: input port, source and destination MACs, EtherType 0x0806, ARP opcode; everything else wildcarded.

The handler thread emits two netlink messages back to the kernel. The first is an `OVS_PACKET_CMD_EXECUTE` containing the actions `output:1, output:3` and the original frame bytes; the kernel transmits the frame on DP ports 1 and 3 and is done with this packet. The second is an `OVS_FLOW_CMD_NEW` containing the megaflow match key and the same actions; the kernel inserts the megaflow into the flow cache. The next ARP request from `ns1` for the same target hits the cache and never reaches userspace.

`ns2` receives the broadcast on its `veth-ns2` end (because `ns2` is on DP port 3). The kernel inside `ns2` sees that the ARP target IP `10.0.0.2` matches its own configured IP and produces an ARP reply: source IP 10.0.0.2, source MAC `2e:65:..:e7`, destination IP 10.0.0.1, destination MAC `2e:56:..:c1` (unicast, not broadcast). The reply enters OVS through DP port 3, misses the cache, upcalls, gets translated through `xlate_normal()` (this time the destination MAC `2e:56:..:c1` *is* learned, on OF port 1, so the action is `output:2` for the bridge-side `br-ns1` DP port; not flood), and is output to DP port 2. `ns1`'s kernel receives the reply, populates its ARP cache, and proceeds to the ICMP echo request that the reader saw timed at 0.693 ms.

The 0.693 ms wall time covers the entire sequence above plus the actual ICMP round trip. Of the 0.693 ms, the upcall-driven half is the dominant cost; the reader who runs the second ping at 0.057 ms is paying only the cached path, which is the kernel module's flow-cache lookup plus the action execution plus the veth-pair traversal.

The reader who has internalised this sequence can now read every line of `dpctl/dump-flows`, every line of `coverage/show`, and every line of `ofproto/trace` with a precise mental model of which kernel-userspace boundary each line crossed and which OVS C function produced it. R4 will deepen the reader's understanding of the upcall path; R2.2 has equipped the reader with the vocabulary they need.

## Misconception block: the reader's instinct from the Cisco era

> **Misconception / Reality.**
>
> Misconception: "since `ovs-ofctl dump-flows br0` shows only one rule, the bridge is forwarding all traffic by a single fixed pattern; if I want different behaviour for ARP and ICMP I need to write two flow rules."
>
> Reality: the priority-0 NORMAL rule is *not* a fixed pattern. It is a hand-off into the entire L2-learning state machine: parse the headers, look up the source MAC for learning, look up the destination MAC for forwarding, flood if unknown, output if known, age out stale entries. The `xlate_normal()` function is more than 200 lines of C and implements the IEEE 802.1D learning bridge specification in full. The reader who writes "two flow rules for ARP and ICMP" without understanding this machinery will produce a bridge that fails to learn, fails to age, and fails subtly under load. R2.5.1 will demonstrate the difference by replacing the priority-0 NORMAL rule with hand-written L2 flows and observing the behavioural divergence.

> **Misconception / Reality (second, related).**
>
> Misconception: "`fdb/show` is a command-table that the operator manages, like Cisco's `mac address-table` configuration."
>
> Reality: `fdb/show` is a *snapshot* of an in-memory cache the daemon maintains. The cache is populated by `xlate_normal()`'s learning side effect, aged out by a background timer, and is not persisted across `ovs-vswitchd` restart. The operator has limited control: `ovs-appctl fdb/flush br0` clears it, but cannot insert a static entry through this command (static entries are inserted via OpenFlow `learn` actions, which R3 covers). On Cisco the `mac address-table static` command writes a permanent CAM entry; OVS has no direct equivalent because the L2 learning is meant to be controller-driven, not operator-driven.

## A short detour on the difference between OF flows, datapath flows, and FDB entries

A reader with a Cisco background often conflates "the rule the bridge uses to forward" with a single object. R2.2 has now exposed three distinct objects that influence forwarding, and reading them as separate things is what makes the rest of the curriculum tractable.

The OpenFlow flow table, exposed by `ovs-ofctl dump-flows`, is the *operator-visible policy*. It contains the rules the operator stored (none in R2.2), the rules a controller pushed (none in R2.2), and the synthesised standalone-mode catch-all (one in R2.2). Each rule has a priority, a match expression, and an action list. A change to this table is the operator's commitment to the bridge's policy. R3 (the OpenFlow programming sprint) will teach the reader to write rules into this table by hand. The OpenFlow flow table is the *policy* surface.

The datapath flow cache, exposed by `ovs-appctl dpctl/dump-flows`, is the *kernel's per-frame fast-path summary*. It is a cache of materialised actions, one entry per distinct megaflow, populated as a side effect of upcall translation, aged out by the revalidator. The operator does not write to this table directly; the operator can `flush` it (`ovs-appctl dpctl/del-flows`) to force re-translation, but the entries themselves are produced by `xlate_actions()`. The datapath flow cache is the *cache* surface.

The forwarding database (FDB), exposed by `ovs-appctl fdb/show`, is the *L2-learning state*. It is per-bridge, per-VLAN, MAC-to-port, with an ageing timer. It is populated as a side effect of `xlate_normal()` when a frame whose source MAC is unknown crosses the bridge. The operator can flush it (`ovs-appctl fdb/flush`) but cannot directly insert (insertion goes through OpenFlow `learn` action). The FDB is consulted only by `xlate_normal()`; if the bridge is running a fully-controller-driven flow table with no NORMAL actions, the FDB will be empty and the bridge will still forward correctly, because forwarding decisions live in the flow table and bypass `xlate_normal()`. The FDB is the *learning* surface.

The three surfaces are often described as if they were redundant. They are not. The flow table is what the operator and controller declare. The datapath cache is what the kernel uses per packet. The FDB is what `xlate_normal()` reads when the policy says "do learning bridge". A bridge with stored OpenFlow rules and no NORMAL action will have a populated flow table, a populated datapath cache, and an empty FDB; that is normal and the reader should not treat the empty FDB as a bug. R3 exercises exactly this configuration.

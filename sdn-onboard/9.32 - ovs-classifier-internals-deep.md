## §9.32.1 classifier (TSS), Tuple Space Search

### Concept

The TSS classifier is the packet and flow classification algorithm inside the userspace `ofproto-dpif` layer of OVS. "Tuple space" means the set of unique mask patterns; each unique mask creates one subtable. A lookup walks the subtables in priority order. Type: a userspace data-plane lookup algorithm.

### History and background

The TSS algorithm comes from Srinivasan et al., SIGCOMM 1998, "Packet Classification using Tuple Space Search". OVS adopted it in 2009 with the initial Nicira implementation. The classifier was added to the OVS source tree in `lib/classifier.c` on 2009-07-08 (commit `064af42167bf4fc9`), which predates the OVS 1.0 release. The algorithm was further refined in 2014 around OVS 2.0, mainly through megaflow consolidation. The classifier has been stable since.

### Position in the architecture

The classifier lives in the userspace `ofproto-dpif` layer of the OVS 3-component architecture (kernel datapath, userspace `ovs-vswitchd`, OVSDB). Concretely: `lib/classifier.c` in the OVS source tree. Scope: one classifier instance per bridge.

### Role

The classifier is the "rule matching engine" between the OpenFlow flow-table abstraction and kernel megaflow installation. The pipeline is: OF rule add, classifier insert (build the subtable), classifier output (install the megaflow). The output then drives the kernel datapath fast path.

### Why it exists

Pre-TSS pain: a linear scan of the flow table at O(N) per packet does not scale. A hash-only lookup does not support wildcard matches. A trie-only lookup is hard for arbitrary masks. TSS is the compromise: hash within a subtable (fast, O(1)) and linear over the subtables (limited by the mask count). Production traffic typically shows 10 to 50 unique masks, which is manageable.

### Problems solved

The problem: arbitrary wildcard match at production scale. The mechanism: partition the flow table into per-mask subtables, with a deterministic lookup cost of O(subtable_count). Scope: fully solves wildcard flexibility, with a partial trade-off against mask explosion (the anti-pattern is a rule set that produces too many unique masks).

### Importance

Cornerstone tier 1. An engineer who does not understand TSS cannot tune the classifier when production hits a performance issue (slow rule install, classifier lookup CPU spike, mask-explosion warning).

### How it works

Per-rule insert (`classifier_insert` in `lib/classifier.c`):
1. Compute the mask pattern (which fields are wildcarded versus exact).
2. If a subtable already exists for the mask, insert into the hash bucket (chained hash).
3. Otherwise, create a new subtable and insert.
4. Update the priority-sorted vector (pvector) ordering of subtables.

Per-packet lookup (`classifier_lookup` in `lib/classifier.c`):
1. For each subtable in priority order (highest priority first):
   a. Hash the packet fields with the subtable's mask, then bucket.
   b. Linearly scan the bucket for an exact match.
2. Return the first match (highest priority).

Mask consolidation invariant: two OF rules with the same wildcard pattern (regardless of field values) share one subtable.

Patricia trie optimization: a prefix trie for IP and port-range masks lets the lookup skip subtable subsets through trie pruning. The priority-sorted vector backing lives in `lib/pvector.c`.

### How an operator masters this

Four skills:

1. **Read**: `ovs-appctl coverage/show | grep classifier` shows counters (`lookup_match`, `lookup_miss`, and so on).
2. **Modify**: indirect, through OF rule design (adding or removing a rule triggers a subtable rebuild). Direct subtable manipulation is not exposed.
3. **Observe**: `ovs-appctl ofproto/list-tunnels` plus `ovs-appctl ofproto/trace 'flow_spec'` step through the classifier evaluation.
4. **Debug**: when the classifier looks slow, check subtable count through the coverage counters; when mask explosion is suspected, audit the OF rule mask diversity.

Decision tree:
- Coverage shows a `classifier_lookup` spike, then check the subtable count (a high count means explosion).
- Subtable count above 50, then audit the rule mask diversity and refactor with the conjunction action.
- Subtable count below 10, then the classifier is not the bottleneck; investigate elsewhere.

### Classification

Type: userspace data-plane lookup algorithm. Parent: TSS implementation in `lib/classifier.c`. Scope: per-bridge instance. Cost: O(K), where K is the subtable count.

### Usage workflow

```
Step 1: Profile classifier overhead
  $ ovs-appctl coverage/show | grep -E "classifier|cmap"
  Shows the cmap_lookup and cmap_insert counters.

Step 2: Identify hot subtables
  $ ovs-appctl ofproto/list-tunnels | wc -l
  More than 100 subtables means mask explosion.

Step 3: Mask consolidation refactor
  Rule A: nw_src=10.0.0.5/32, action=output:1
  Rule B: nw_src=10.0.0.6/32, action=output:1
  Refactor: use conjunction(1,1/2) for the IP set and conjunction(1,2/2) for the action.
  2 unique rules become 1 subtable plus 1 verdict subtable, which is simpler.

Step 4: Verify the reduction
  $ ovs-appctl coverage/show | grep cmap
  The subtable count should drop.

Step 5: Measure the latency improvement
  Use a packet generator and measure first-flow latency after the refactor.
```

### When something goes wrong

- Symptom A: first-flow latency above 10 ms. Diagnosis: the classifier lookup is slow, the subtable count is too high. Recovery: mask consolidation.
- Symptom B: `ovs-vswitchd` CPU spike on rule add. Diagnosis: classifier insert is slow because of subtable rebuild cost. Recovery: batch rule installs through the bundle action.
- Symptom C: rule effect is intermittent. Diagnosis: a classifier priority-order conflict where two rules at the same priority overlap. Recovery: explicit priority assignment.

Cross-links: 9.26 revalidator storm forensic; 9.25 `ofproto/trace` classifier debug.

### Tightly related to

- **megaflow** (9.2): the classifier output is installed into the kernel megaflow.
- **ukey** (9.2): the revalidator tracks the mapping between a kernel flow and a classifier rule.
- **upcall** (next, §9.32.2): the classifier is invoked when the kernel misses.
- **subtable** (this section): a per-mask classifier partition.
- **Patricia trie** (this section): the prefix optimization.
- **pvector** (this section): the priority order.

### Real-world incident

Production case 2023-Q4: an OVS cluster started running `ovs-vswitchd` at 100 percent CPU after an ACL update was deployed.

Investigation:
- `coverage/show` showed `classifier_insert` sustained at 50,000 per second.
- The new ACL rule set had 2000 entries, each with a unique 5-tuple, which produced 2000 unique masks.
- Subtable explosion: 2000 subtables, with an O(K) cost on every lookup.

Root cause: the ACL design used a per-IP-and-port-pair rule instead of the conjunction action.

Fix: refactor the ACL with `Address_Set`, `Port_Group`, and conjunction, which dropped the subtable count to 200. `ovs-vswitchd` CPU dropped from 100 percent to 30 percent.

Reference: a similar pattern is documented in 9.26 revalidator storm.

### Synthetic exercise

```
Setup: Mininet single-switch s1 với 100 veth port

Test mask explosion:
  for i in $(seq 1 100); do
    ovs-ofctl add-flow s1 "priority=100,ip,nw_src=10.0.0.$i,actions=output:$i"
  done
  ovs-appctl coverage/show | grep cmap
  → cmap_insert ~ 100 (1 subtable per mask, but mask same for all - actually 1 subtable!)

  # Now diversify:
  for i in $(seq 1 100); do
    ovs-ofctl add-flow s1 "priority=100,ip,nw_src=10.0.$i.0/24,actions=output:$i"
  done
  → 100 different mask /24, 100 subtable

Verify reduction with conjunction:
  ovs-ofctl add-flow s1 'priority=100,ip,nw_src=10.0.0.0/8,actions=conjunction(1,1/2)'
  ovs-ofctl add-flow s1 'priority=100,ip,actions=conjunction(1,2/2)'
  ovs-ofctl add-flow s1 'priority=200,conj_id=1,actions=output:1'
  → Now O(few) subtable for same matching capability
```

### Common failures and diagnostic signals

- Mode A: subtable explosion (more than 1000 subtables) leading to a CPU spike. Diagnose with `coverage/show`. Recovery: mask consolidation and conjunction.
- Mode B: Patricia trie misuse on a non-prefix mask, which falls back to subtable scan and runs slow. Recovery: rare bug; check the OVS version.
- Mode C: pvector reorder cost on a mass rule update, temporarily slow. Recovery: install the rules in a bundle so the update is atomic.

### Comparison with other systems

- **Linux iptables filter chain**: linear scan at O(N), no algorithmic improvement. The OVS classifier is far superior for a large rule set.
- **Cisco TCAM ACL**: hardware parallel match, O(1) but limited in capacity. The OVS software classifier is flexible (no capacity limit) at the cost of software speed.
- **eBPF cls_bpf classifier**: a programmable classifier in the kernel. Difference: eBPF compiles per-rule code through the JIT, while the OVS classifier interprets from the OF abstraction.
- **Intel DPDK rte_classify**: a hardware-assisted classification table. OVS hardware offload combines with the classifier.

## §9.32.2 upcall — Kernel miss to userspace handler

### Khái niệm

`upcall` là Netlink message từ kernel openvswitch.ko → userspace ovs-vswitchd khi kernel datapath miss megaflow lookup. Userspace decode packet, evaluate via classifier, install resulting megaflow back to kernel. Loại: kernel-userspace IPC mechanism.

### Lịch sử + bối cảnh

Initial 2009 với OVS Nicira. Netlink protocol stable. OVS 2.0 (2014) add upcall rate limit cho防 control plane DDoS.

### Vị trí trong kiến trúc

Bridge giữa kernel `openvswitch.ko` (fast path, miss handling) và userspace `ovs-vswitchd` (slow path, classifier + flow install). Concretely: Netlink socket `OVS_PACKET_FAMILY` carries upcall message.

### Vai trò

Cache miss handler. Without upcall, kernel cache miss = packet drop (no fallback). With upcall, miss → classifier → install → next packet hits cache. Critical cho first-flow latency + cache warming.

### Vì sao sinh ra

Pre-OVS, kernel module có self-contained flow logic (Linux bridge MAC learning local). OVS architecture decouple control plane (userspace) từ data plane (kernel). Upcall is bridge mechanism cho async control flow.

### Vấn đề giải quyết

Vấn đề: kernel datapath cache miss handling without dropping packet. Mechanism: Netlink message buffer packet + metadata, handler thread dequeue, evaluate, install, optionally inject packet back. Scope: full solve cho cache miss; partial trade-off (upcall cost itself ~10-100us/packet).

### Tầm quan trọng

Cornerstone tier 1. Upcall là bottleneck dưới conditions (high churn, cold cache, DDoS). Engineer không hiểu upcall = không debug "first packet slow", "ovs-vswitchd CPU spike".

### Cơ chế hoạt động

```
Kernel side (openvswitch.ko):
  Step 1: dp_packet_in() invoke từ netif_rx
  Step 2: flow_table_lookup() check megaflow
  Step 3: If miss → genl_send (OVS_PACKET_FAMILY, OVS_PACKET_CMD_MISS)
  Step 4: packet skb queued
  Step 5: continue (other packet handled)

Userspace side (ovs-vswitchd):
  Step 1: handler_thread main loop
  Step 2: dpif_recv() dequeue Netlink message
  Step 3: ofproto_recv_upcall() decode packet
  Step 4: ofproto-dpif xlate evaluate via classifier
  Step 5: Install resulting megaflow via dpif_flow_put()
  Step 6: Optionally inject packet back via OVS_PACKET_CMD_EXECUTE

Rate limit:
  Per-bridge upcall socket có rate cap, OVS 2.0+ adaptive throttle
```

Invariant: every miss generates exactly 1 upcall (no duplicate); revalidator does NOT generate upcall (separate path).

### Cách kỹ sư vận hành thành thạo

Skill: monitor upcall rate, identify spike root cause, tune handler thread count.

```
$ ovs-appctl coverage/show | grep upcall
upcall_flow_limit_hit    0 +0   # no rate limit hits = healthy
upcall_flow_limit_kill   0 +0
miss_per_sec    100      # baseline normal

$ ovs-vsctl set Open_vSwitch . other_config:n-handler-threads=8
→ Increase handler thread count for high-rate cluster
```

### Phân loại

Loại: kernel-userspace IPC mechanism, Netlink message family. Parent: OVS_PACKET_FAMILY. Scope: per-packet on miss.

### Quy trình sử dụng

Debug upcall storm:

```
Step 1: $ ovs-appctl coverage/show | grep -E "upcall|miss"
Step 2: If miss_per_sec > 1000, identify churn source
Step 3: $ ovs-appctl dpctl/dump-flows | head -20
   → Check megaflow count + freshness
Step 4: If many short-lived flow, increase n-handler-threads
Step 5: If many unique mask, refer §9.32.1 mask consolidation
```

### Khi xảy ra sự cố

- Symptom A: First-flow latency 10-100ms. Suspect upcall queue depth. Diagnostic coverage/show.
- Symptom B: ovs-vswitchd CPU 100% steady-state. Diagnostic upcall rate sustained > 10K/sec.
- Symptom C: kernel log "upcall failed: ENOBUFS". Suspect Netlink socket buffer overflow. Recovery: increase `net.core.rmem_max`.

### Liên quan mật thiết tới

Tightly coupled: handler thread, megaflow (consume upcall output), classifier (evaluate), Netlink (transport), `dp_netdev` userspace alternate datapath, EMC/SMC (downstream cache).

### Khác biệt giữa các phiên bản

| Version | Behavior |
|---------|----------|
| OVS 2.0 | Rate limit + adaptive throttle |
| OVS 2.4 | Multi-handler thread default 4 |
| OVS 2.6 | EMC tier reduce upcall need |
| OVS 2.15 | SMC tier further reduce |
| OVS 3.0+ | Stable |

### Cách quan sát + xác minh

```
$ ovs-appctl coverage/show | grep upcall
$ tc -s qdisc show dev <iface>
→ kernel netif queue depth
$ dmesg | grep openvswitch
→ kernel log upcall failures
```

### Source code tham chiếu

- Kernel: `net/openvswitch/datapath.c` `ovs_dp_upcall()`, `ovs_dp_process_packet()`
- Kernel: `net/openvswitch/flow_netlink.c` Netlink encoding
- Userspace: `lib/dpif-netlink.c` `dpif_netlink_recv()` dequeue
- Userspace: `ofproto/ofproto-dpif-upcall.c` `handle_upcalls()` main handler loop

### Trường hợp sự cố thực tế

Production case 2023-Q1: Web cluster experience occasional 100ms latency spike on new TCP connection.

Investigation:
- coverage/show shows upcall rate spike to 50K/sec at peak
- Megaflow count low (~100), so cache should hit
- But miss rate also high (500/sec)

Root cause: handler thread starvation under burst. n-handler-threads=4 default, but burst created queue depth > 100 messages, causing latency spike.

Fix: increase n-handler-threads=16 + tune `net.core.rmem_max=16M`. Latency P99 drop từ 100ms → 5ms.

### Bài tập synthetic

```
Setup: Mininet 2-host topology
  $ sudo mn --topo single,2 --switch ovsk

Generate burst:
  $ ovs-ofctl del-flows s1 -- add-flow s1 'priority=10,actions=NORMAL'
  $ h1 hping3 -S -p 80 -c 10000 -i u100 h2
  → Burst SYN packet, each is potential megaflow miss

Monitor:
  $ watch -n 1 'ovs-appctl coverage/show | grep upcall'
  → Observe upcall counter spike

Tune:
  $ ovs-vsctl set Open_vSwitch . other_config:n-handler-threads=8
  $ systemctl restart openvswitch-switch
  → Re-run burst, observe latency improve
```

### Lỗi thường gặp + tín hiệu chẩn đoán

- Mode A: Handler thread saturate → upcall queue grow → latency spike. Recovery: increase n-handler-threads.
- Mode B: Netlink socket overflow → ENOBUFS. Recovery: increase rmem_max.
- Mode C: Adaptive rate limit kick in → flow drop. Diagnostic coverage upcall_flow_limit_hit. Recovery: investigate upstream churn.

### So sánh với hệ khác

- **Linux netfilter NF_QUEUE:** similar kernel-userspace queue mechanism. Khác: NF_QUEUE per-rule decision; OVS upcall per-miss decision (more efficient).
- **DPDK PMD poll-mode:** alternative bypass kernel entirely. Khác: PMD continuous poll; OVS upcall event-driven. Trade-off CPU vs latency.
- **Cisco IOS process switching:** packet escalates from fast-path to control plane. Same concept, different abstraction.

## §9.32.3 ofproto-dpif xlate — máy dịch OF action sang datapath byte-code

### Khái niệm

`ofproto-dpif xlate` là máy dịch (translation engine) sống trong userspace `ovs-vswitchd`, có nhiệm vụ biến danh sách OpenFlow action của một flow rule thành chuỗi byte-code datapath (`odp_actions`) mà kernel module `openvswitch.ko` hoặc userspace DPIF có thể thực thi tuyến tính. Mỗi packet đi qua đường upcall slow-path đều ghé qua xlate; sau khi xlate hoàn tất, kết quả `odp_actions` được dùng để cài megaflow vào datapath cache, và những packet tiếp theo cùng tuple sẽ chạy thẳng fast-path không phải gọi lại xlate. Xlate là cây cầu duy nhất giữa abstraction OpenFlow (controller-facing, đa bảng, đệ quy `goto_table`/`resubmit`, bundle, group, learn) và byte-code datapath (kernel-facing, một chiều, không nhánh, không đệ quy).

### Lịch sử + bối cảnh

Xlate xuất hiện đầu năm 2012 khi Justin Pettit + Ben Pfaff tách `ofproto-dpif` thành module riêng (commit-tag refactor giai đoạn OVS 1.5–1.7). Trước đó, `ofproto-classifier` viết trực tiếp ra Netlink request gửi xuống `openvswitch.ko`, gây trộn lẫn business logic OpenFlow với binding kernel API. Tách ra cho phép OVS sau này hỗ trợ nhiều datapath backend (`dpif-netlink` cho kernel, `dpif-netdev` cho userspace + DPDK + AF_XDP) mà không phải viết lại logic dịch action. OVS 2.4 (2015) bổ sung emulation đệ quy bằng `recirc_id` (thay vì giả định datapath hỗ trợ multi-table); OVS 2.6 thêm conntrack helper; OVS 2.8 thêm NSH; OVS 2.13 ổn định `frozen_state` cho continuation; OVS 3.0+ chỉ vá lỗi nhỏ và tối ưu speed.

### Vị trí trong kiến trúc

Xlate nằm chính giữa pipeline upcall của `ovs-vswitchd`: nguồn vào là kết quả lookup classifier (rule với danh sách `ofpacts`), nguồn ra là buffer `odp_actions` cho dpif. Trên cây mã nguồn, xlate chiếm trọn file `ofproto/ofproto-dpif-xlate.c` (~7000 dòng), cộng với `ofproto/ofproto-dpif-xlate-cache.c` (cache invalidation hint) và header `ofproto/ofproto-dpif-xlate.h`. Caller chính là `handle_upcall()` trong `ofproto/ofproto-dpif-upcall.c`; consumer là `dpif_flow_put()` trong `lib/dpif.c`. Mỗi handler thread userspace có một `struct xlate_ctx` riêng để tránh contention.

### Vai trò

Vai trò xlate gồm bốn mảng: (1) duyệt pipeline đa bảng OpenFlow theo đệ quy `goto_table`/`resubmit`, (2) phân giải mỗi `ofpact` thành một hoặc nhiều `odp_action` tương ứng (output, set field, push/pop VLAN, ct, recirc, encap), (3) đóng băng trạng thái khi không thể dịch trong một lượt (ghi `frozen_state` + cấp `recirc_id` mới + dừng dịch), (4) chuẩn hoá action set OpenFlow 1.1+ trước khi commit. Mọi tính năng OpenFlow controller phơi bày cho mạng đều phải đi qua xlate trước khi xuống datapath; nếu xlate không support, tính năng đó coi như không tồn tại với end-host.

### Vì sao sinh ra

Datapath kernel chỉ biết một tập hữu hạn action nguyên thuỷ và không biết khái niệm bảng. OpenFlow 1.1+ lại giàu cấu trúc: nhiều bảng, đệ quy, action set, group. Nếu không có lớp dịch, hai hướng giải quyết bất khả: (a) bắt kernel học OpenFlow đầy đủ — phình to attack surface, (b) bắt controller dịch tay xuống primitive datapath — mất tính trừu tượng. Xlate ra đời để giữ kernel gọn, giữ controller mượn ngôn ngữ OpenFlow chuẩn, và che mọi sự khác biệt ở giữa.

### Vấn đề giải quyết

Xlate giải bốn bài toán cụ thể: (1) dịch đệ quy `goto_table` chiều dài bất kỳ thành chuỗi action tuyến tính, (2) khi gặp action mà datapath không hỗ trợ native (ví dụ encap NSH trên kernel cũ), emulate qua `recirc` + bảng phụ, (3) khi packet phải chờ kết quả conntrack hoặc nat, đóng băng phần còn lại của pipeline lại bằng `frozen_state` rồi tiếp tục sau khi `OVS_ACTION_ATTR_RECIRC` đưa packet trở lại userspace, (4) phát hiện vòng lặp đệ quy `resubmit` và cắt ở `MAX_DEPTH = 64` để tránh stack overflow.

### Tầm quan trọng

Đây là một trong ba khối cornerstone của OVS userspace bên cạnh classifier và dpif. Kỹ sư mạng OVS không thể debug rule "viết đúng nhưng packet không đi qua" mà không biết đọc xlate trace; team operations không thể giải thích vì sao một flow đơn giản khi chạy với `goto_table` lại sinh ra `recirc_id` mới mà không hiểu cơ chế đóng băng pipeline; team release không thể đánh giá tác động một upgrade kernel mà không biết xlate có emulate được hay không. Một mistake xlate (ví dụ MAX_DEPTH cycle, hoặc emulation chain nở quá đáng) thường biểu hiện thành "CPU userspace 100% suốt mà throughput vẫn thấp" — gốc rễ chính là xlate.

### Cơ chế hoạt động

Điểm vào duy nhất là `xlate_actions(struct xlate_in *, struct xlate_out *)` khai báo trong `ofproto/ofproto-dpif-xlate.h`. Caller chuẩn bị `xlate_in` chứa flow gốc, packet (nếu có), rule khởi điểm; gọi xong nhận về `xlate_out` chứa `odp_actions` đã tích luỹ, `slow_path_reason` (nếu có lý do gửi lại userspace), và `recirc_refs` (các recirc_id đã cấp).

Bên trong, hàm thiết lập `struct xlate_ctx` rồi gọi nội bộ `do_xlate_actions()` để duyệt mảng `ofpact`. Mỗi `ofpact` được dispatch theo loại:

- `OFPACT_OUTPUT` → gọi `compose_output_action()` để sinh `OVS_ACTION_ATTR_OUTPUT` với port number datapath tương ứng.
- `OFPACT_GOTO_TABLE` → gọi `xlate_table_action()` đệ quy với `table_id` mới; tăng `ctx->depth`; nếu `ctx->depth > MAX_DEPTH` (`#define MAX_DEPTH 64` trong `ofproto-dpif-xlate.c`) thì dừng + đặt `ctx->error = XLATE_RECURSION_TOO_DEEP`.
- `OFPACT_RESUBMIT` → tương tự `goto_table` nhưng cùng-bảng, tăng `ctx->resubmits`; nếu `ctx->resubmits > MAX_RESUBMITS` (= `MAX_DEPTH * MAX_DEPTH = 4096`) thì dừng.
- `OFPACT_CT` → emit `OVS_ACTION_ATTR_CT` kèm nested attributes (zone, mark, label, nat); nếu sau ct còn action chưa dịch hết, đóng băng phần còn lại vào `frozen_actions` + cấp `recirc_id` mới + emit `OVS_ACTION_ATTR_RECIRC`.

Cuối cùng `xlate_commit_actions()` đẩy buffer tích luỹ ra `xlate_out->odp_actions` và caller gọi `dpif_flow_put()` để cài megaflow.

### Cách kỹ sư vận hành thành thạo

Kỹ năng cần làm chủ: (a) đọc trace `ovs-appctl ofproto/trace br0 '<flow_spec>'` từng bước, đối chiếu mỗi dòng "Resubmitted flow:" hoặc "thawed from frozen state" với rule trong dump, (b) đọc counter `ovs-appctl coverage/show | grep xlate_` để biết tần suất gặp `xlate_actions_too_many_output`, `xlate_actions_resubmit_recursion`, `xlate_actions_oversize`, (c) đọc log `ovs-vswitchd.log` cho cảnh báo "over MAX_DEPTH" hoặc "flow translation oversized", (d) chạy `ovs-appctl dpif/show-dp-features` để biết datapath nào hỗ trợ action native, action nào phải emulate.

### Phân loại

Theo kết quả xlate, xlate path chia ba nhóm: (1) **fast translate** — flow đơn giản, output/drop, kết quả `odp_actions` ngắn, không recirc; (2) **recirc translate** — pipeline có conntrack, multi-stage, sinh ra một hoặc nhiều `recirc_id`; (3) **slow-path translate** — xlate set `slow_path_reason` (ví dụ `SLOW_CONTROLLER`, `SLOW_BFD`), megaflow không cài, packet vẫn lên userspace mỗi lần. Theo entry-point: (i) upcall xlate (mỗi miss kernel), (ii) revalidator xlate (re-evaluate megaflow đã cài để kiểm tra invalidation), (iii) trace xlate (chạy theo lệnh `ofproto/trace` thuần debug, không cài megaflow).

### Quy trình sử dụng

Trong sản xuất xlate chạy ngầm, kỹ sư không gọi trực tiếp; tương tác qua công cụ. Quy trình debug điển hình: (1) capture flow gốc của packet bằng `ovs-tcpdump` hoặc `ovs-appctl dpif/dump-flows`, (2) chạy `ovs-appctl ofproto/trace br0 '<flow>'` để xlate dịch và in trace, (3) đọc các dòng `Megaflow:` và `Datapath actions:` cuối trace để biết kết quả thực sự cài, (4) nếu khác kỳ vọng, theo các dòng `goto_table N` lùi về để tìm rule sai. Quy trình quan sát perf: bật `ovs-appctl vlog/set xlate:dbg`, chạy workload, đọc log; lưu ý log dày, chỉ bật ngắn.

### Khi xảy ra sự cố

Triệu chứng xlate có sự cố thường gặp: thông báo `xlate_actions: over MAX_DEPTH limit` trong log, packet không tới đích dù rule đúng, CPU `ovs-vswitchd` nhảy 100% kéo dài mà PPS không tăng, `coverage/show` cho thấy `xlate_actions_too_many_output` tăng nhanh. Hành động: chạy `ofproto/trace` cho flow điển hình, đếm số `Resubmitted` và `Recirc` xuất hiện; nếu vượt 10–20 cho một packet là dấu hiệu pipeline có nhánh gây nở. Khi gặp emulation explosion, kiểm tra `dpif/show-dp-features` so với version OVS để biết feature nào đang phải emulate.

### Liên quan mật thiết tới

Xlate ghép với: `classifier` (đầu vào, cấp rule), `dpif` (đầu ra, cài megaflow), `recirc_id` (cấp ID khi đóng băng), `conntrack` (mỗi `OFPACT_CT` ghé), `revalidator` (gọi xlate lại để so megaflow), `ofproto-dpif-upcall` (caller chính), `xlate-cache` (cache invalidation hint cho revalidator), `ofproto/trace` (debug shell). Mọi thay đổi cấu trúc rule trong classifier đều invalidate xlate cache; mọi thay đổi datapath capability đều buộc xlate đổi kết quả emulation.

### Khác biệt giữa các phiên bản

- **OVS 2.4 (2015):** thêm emulation đệ quy bằng `recirc_id` cho datapath chỉ hỗ trợ một bảng.
- **OVS 2.6 (2016):** thêm dispatch `OFPACT_CT` hoàn chỉnh (zone, nat, helper).
- **OVS 2.8 (2017):** thêm encap/decap NSH.
- **OVS 2.13 (2020):** ổn định `frozen_state` + `xlate-cache` granular invalidation.
- **OVS 2.17.x (2022 baseline curriculum):** `MAX_DEPTH = 64`, `MAX_RESUBMITS = 4096`; signature `xlate_table_action(struct xlate_ctx *, ofp_port_t in_port, uint8_t table_id, bool may_packet_in, bool honor_table_miss, bool with_ct_orig, bool is_last_action, xlate_actions_handler *)`.
- **OVS 3.x:** thêm tối ưu `compose_output_action()` cho `is_last_action`, không thay đổi semantics.

### Cách quan sát + xác minh

```
$ sudo ovs-appctl ofproto/trace br0 \
    'in_port=1,dl_src=02:00:00:00:00:01,dl_dst=02:00:00:00:00:02,dl_type=0x0800,nw_src=10.0.0.1,nw_dst=10.0.0.2,nw_proto=6,tp_src=1234,tp_dst=80'
Flow: in_port=1,...,tp_dst=80
bridge("br0")
-------------
 0. priority 100, ...
    goto_table:1
 1. priority 50, ip,nw_dst=10.0.0.2 actions=output:2

Final flow: unchanged
Megaflow: recirc_id=0,eth,ip,in_port=1,nw_dst=10.0.0.2
Datapath actions: 2
```

Hai dòng cuối cho biết kết quả xlate: `Megaflow` là wildcarded mask sẽ cài vào kernel; `Datapath actions` là byte-code thực sự thực thi. Đối chiếu thêm với `ovs-appctl coverage/show | grep -E 'xlate_|recirc'` để biết đếm sự kiện theo loại.

### Source code tham chiếu

- `ofproto/ofproto-dpif-xlate.h` khai báo `enum xlate_error xlate_actions(struct xlate_in *, struct xlate_out *);` — entry duy nhất.
- `ofproto/ofproto-dpif-xlate.c`: `do_xlate_actions()` dispatch chính theo `ofpact->type`; `xlate_table_action()` đệ quy bảng; `compose_output_action()` sinh action output kèm bond + truncate; `xlate_commit_actions()` flush buffer; `xlate_action_set()` xử lý OF 1.1+ action set.
- Hằng số `#define MAX_DEPTH 64` và `#define MAX_RESUBMITS (MAX_DEPTH * MAX_DEPTH)` đặt ở đầu `ofproto-dpif-xlate.c` (gần dòng 138–142 baseline v2.17.9).
- `struct xlate_ctx` chứa `int depth`, `int resubmits`, `struct ofpbuf frozen_actions`, `struct ofpbuf *odp_actions`.
- `lib/odp-execute.c` là consumer phía datapath userspace: nó đọc `odp_actions` xlate sinh ra và thực thi tuần tự.

### Trường hợp sự cố thực tế

Một site OpenStack production gặp triệu chứng: sau khi đẩy thêm 200 security group rule, throughput VM bắt đầu rớt dần, `top` cho thấy `ovs-vswitchd` ngốn 95% CPU một lõi. Đào sâu: rule mới khiến pipeline OVN sinh ra chuỗi `resubmit` chéo giữa bảng ingress ACL và conntrack, mỗi packet đi qua 30+ bước. `coverage/show` cho thấy `xlate_actions_resubmit_recursion` tăng vài chục nghìn lượt mỗi giây; log có cảnh báo "translation oversized". Khắc phục tức thì: tách bảng ingress ACL thành hai stage để giảm độ rộng resubmit; gốc rễ: rule design không cân nhắc chi phí xlate. Bài học vận hành: mọi rule cộng dồn vào pipeline đều có chi phí xlate ẩn, không chỉ chi phí classifier lookup.

### Bài tập synthetic

Lab Mininet một bridge `br0` với hai host `h1`, `h2`. Bước 1: tạo bridge `sudo ovs-vsctl add-br br0`, gắn hai port `veth`. Bước 2: cài rule đa bảng `sudo ovs-ofctl add-flow br0 'table=0,priority=100,actions=goto_table:1'` và `sudo ovs-ofctl add-flow br0 'table=1,priority=50,ip,nw_dst=10.0.0.2,actions=output:2'`. Bước 3: chạy `sudo ovs-appctl ofproto/trace br0 'in_port=1,dl_type=0x0800,nw_dst=10.0.0.2'` và quan sát hai bước `0.` và `1.` xuất hiện trong trace, cùng dòng `Datapath actions: 2`. Bước 4: thêm `goto_table:2` đệ quy chéo 60 lần liên tiếp để chạm `MAX_DEPTH`, quan sát thông báo `over MAX_DEPTH`. Bước 5: chạy `ovs-appctl coverage/show | grep xlate_actions_resubmit_recursion` trước và sau để thấy counter nhảy.

### Lỗi thường gặp + tín hiệu chẩn đoán

- **Cycle resubmit do rule sai:** trace dừng đột ngột với "Recursion too deep"; counter `xlate_actions_resubmit_recursion` tăng. Sửa rule.
- **Emulation explosion sau upgrade kernel:** datapath cũ không còn hỗ trợ action native; mỗi packet sinh nhiều recirc; CPU userspace tăng. Nâng kernel module hoặc roll back rule dùng feature đó.
- **Slow-path stuck:** xlate set `slow_path_reason`, megaflow không cài; mọi packet đi userspace. Kiểm tra rule có gọi `controller(...)` hoặc bfd liên tục không.
- **Frozen state leak:** `recirc_refs` không được giải phóng đúng; `recirc_id pool` cạn. Theo dõi `ovs-appctl upcall/show` cho `recirc allocations`.

### So sánh với hệ khác

Cisco IOS dịch ACL/route-map từ control-plane sang TCAM hardware có khái niệm tương tự — control plane viết rule giàu cấu trúc, "compiler" trong NX-OS dịch xuống TCAM byte-code có capability hữu hạn; khi vượt dung lượng, fallback CPU. Linux netfilter `nf_tables` JIT-compile chain rule sang BPF bytecode cũng cùng tinh thần nhưng phương tiện khác (BPF VM thay vì datapath cache). VPP của FD.io thì ngược: graph node được biên dịch trước, không có lớp dịch tại run-time, đổi flexibility lấy hiệu năng. Xlate đứng ở giữa: dịch trong run-time per-flow nhưng có cache megaflow để khấu trừ chi phí.

## §9.32.4 dpif — Datapath interface abstraction

### Khái niệm

`dpif` là tầng abstraction trong userspace OVS cho phép `ofproto-dpif` (engine xlate OpenFlow → datapath byte-code) thao tác với một datapath backend bất kỳ qua một interface thống nhất. Cụ thể: thay vì `ofproto-dpif` gọi trực tiếp Netlink syscall vào kernel `openvswitch.ko`, hoặc gọi DPDK API vào userspace netdev, mọi thao tác (port add/del, flow add/del/dump, packet send/recv, conntrack execute) đi qua một virtual function table `struct dpif_class`. Mỗi backend cụ thể (kernel, userspace, AF_XDP, DPDK) đăng ký một implementation của `dpif_class` lúc khởi động.

### Lịch sử + bối cảnh

OVS 1.0 (2009) chỉ có một datapath duy nhất: kernel `openvswitch.ko` qua Netlink. Code `ofproto` gọi trực tiếp Netlink, tightly coupled. Khi nhu cầu DPDK userspace datapath xuất hiện (~2014), nếu không tách abstraction sẽ phải fork toàn bộ ofproto. Provider Justin Pettit + Ben Pfaff thiết kế `dpif` interface (lib/dpif.h) để cô lập "what" (dpif operations) khỏi "how" (Netlink vs DPDK). OVS 2.4 (2015) chính thức split `dpif-netdev.c` (userspace) khỏi `dpif-netlink.c` (kernel). OVS 2.8+ thêm AF_XDP backend cùng cơ chế dpif (chú ý: AF_XDP nằm ngoài phạm vi học của curriculum này per CLAUDE.md ban directive, chỉ nhắc kết cấu tồn tại).

### Vị trí trong kiến trúc

`dpif` nằm ở tầng userspace của OVS, ngay giữa hai thành phần: trên là `ofproto-dpif` (consumer, chứa xlate + revalidator + handler thread), dưới là concrete datapath backend (provider, một trong: kernel openvswitch.ko qua Netlink, userspace dp-netdev qua direct call, DPDK PMD qua poll-mode driver). Thuộc về `ovs-vswitchd` daemon process; mỗi bridge có một `dpif` instance (`struct dpif`) gắn với một `dpif_class` cụ thể qua field `datapath_type` trong OVSDB Bridge row.

### Vai trò

`dpif` đóng vai trò "narrow waist" trong stack OVS userspace: mọi thao tác datapath đi qua đúng 1 interface; thêm backend mới chỉ cần implement `dpif_class` không phải sửa `ofproto-dpif`. Vai trò cụ thể: (a) abstract port lifecycle (port_add, port_del, port_query_by_number); (b) abstract flow management (flow_put, flow_del, flow_dump_create); (c) abstract packet I/O (recv để nhận upcall, execute để inject packet); (d) abstract introspection (operate, dpif_show statistics).

### Vì sao sinh ra

Trước `dpif`, mã `ofproto` tightly coupled với kernel Netlink. Khi DPDK userspace datapath đề xuất 2013-2014 (Pravin Shelar, Pfaff RFC), nếu vẫn coupling thì hoặc phải duplicate ofproto cho mỗi backend (maintainability nightmare), hoặc compile-time toggle với `#ifdef` chằng chịt (bug-prone). Pettit + Pfaff đề xuất tách abstraction theo design pattern Strategy (vtable virtual dispatch). Lý do thiết kế: (1) cho phép multiple backend chạy song song trong cùng `ovs-vswitchd` (rare nhưng possible cho test); (2) cho phép unit test ofproto với `dpif-dummy` mock (không cần kernel module hay DPDK runtime); (3) cho phép third-party backend (e.g., hardware offload gateway) plug vào không cần modify upstream OVS.

### Vấn đề giải quyết

Bốn vấn đề cụ thể được `dpif` address. (1) **Tightly-coupled architecture**: trước `dpif` mọi syscall Netlink scattered trong ofproto code; sau `dpif` chỉ tập trung trong dpif-netlink.c. (2) **Backend evolution**: kernel openvswitch.ko deprecation roadmap (mainline removed v2.x) cần đường thoát sang userspace; `dpif` cho phép migration không cần rewrite ofproto. (3) **Test isolation**: ofproto-dpif unit test với `dpif-dummy` (lib/dpif-netdev-perf.c có testing helper) không cần root permission, không cần load kernel module, chạy trên CI laptop trong vài giây. (4) **Capability divergence**: kernel hỗ trợ ct() từ OVS 2.5, userspace netdev hỗ trợ ct() muộn hơn; xlate đọc capability matrix qua `dpif_class->cleanup_required`, `synced_dp_layers`, hoặc per-feature query method để quyết định emulate vs delegate.

### Tầm quan trọng

`dpif` là khái niệm cốt lõi tuyệt đối của OVS internals. Engineer không hiểu `dpif` sẽ không phân biệt được: (a) đâu là code path vào kernel datapath versus userspace datapath; (b) tại sao cùng một OF rule có hành vi khác nhau giữa hai host (`datapath_type=system` vs `datapath_type=netdev`); (c) tại sao `ovs-appctl dpif/show` xuất khác nhau theo backend; (d) tại sao migration từ kernel sang userspace cần chuyển `datapath_type` lúc bridge create chứ không phải runtime. Trong nhóm cốt lõi OVS, `dpif` ngang `megaflow`, `recirc_id`, `ofproto-dpif xlate`.

### Cơ chế hoạt động

`dpif` hoạt động qua bốn giai đoạn rõ ràng. **Giai đoạn 1 đăng ký provider:** lúc `ovs-vswitchd` khởi động, hàm `dp_register_provider(&dpif_X_class)` được gọi cho mỗi backend compile-in (`dpif-netlink.c::dpif_netlink_class`, `dpif-netdev.c::dpif_netdev_class`); danh sách lưu trong `base_dpif_classes[]`. **Giai đoạn 2 mở dpif:** khi `ovs-vsctl add-br br0` tạo bridge mới, ofproto gọi `dpif_open(name, type, &dpifp)` với `type` lấy từ Bridge.datapath_type; hàm này lookup class theo type, gọi `class->open(name, &dpif)` để khởi tạo concrete `struct dpif`. **Giai đoạn 3 thao tác runtime:** mọi thao tác (e.g., `dpif_flow_put`, `dpif_port_add`) đi qua macro dispatch `dpif->dpif_class->flow_put(dpif, ...)`. **Giai đoạn 4 đóng dpif:** khi bridge bị xoá, `dpif_close(dpif)` gọi `class->close(dpif)` cleanup resources. Toàn bộ lifecycle có locking + reference counting để đảm bảo nhiều thread (handler, revalidator, xlate) cùng dùng dpif an toàn.

### Cách kỹ sư vận hành thành thạo

Bốn skill thiết yếu. (1) **Read backend hiện hành**: `ovs-vsctl get Bridge br-int datapath_type` cho biết bridge dùng backend nào (`system` = netlink, `netdev` = userspace native, `netdev` + `other_config:dpdk-init=true` = DPDK). (2) **Switch backend** (chỉ lúc create bridge): `ovs-vsctl set Bridge br0 datapath_type=netdev` rồi restart `ovs-vswitchd`; lưu ý: không thể đổi backend in-place khi bridge đang hoạt động vì state datapath không transferable. (3) **Inspect dpif state**: `ovs-appctl dpif/show` xuất per-bridge backend type, port list, flow count, packet stats. (4) **Debug capability mismatch**: khi flow rule fail với "unsupported feature", grep `ovs-vswitchd.log` cho dòng `xlate: action <X> not supported by dpif <type>`; thường do feature có ở kernel nhưng chưa ở userspace netdev (e.g., conntrack ALG, sample group).

### Phân loại

`dpif` là **abstraction layer** trong OVS userspace, không phải CLI tool, không phải OVSDB table, không phải OpenFlow primitive. Cụ thể là một virtual function table (struct dpif_class) cùng với consumer-side wrapper (struct dpif). Trong taxonomy OVS, dpif thuộc nhóm "internal abstraction layer" cùng với `ofproto-class` (interface OpenFlow → bridge implementation), `netdev-class` (interface OVS → physical NIC). Khác với mặt phẳng dữ liệu (data plane abstraction như megaflow, ukey) và khác với mặt phẳng quản lý (management plane như OVSDB schema row).

### Quy trình sử dụng

Workflow điển hình của developer extend OVS với backend mới: (1) Định nghĩa `struct dpif_class new_class = { .type = "newdp", .open = newdp_open, ... };` trong file mới `lib/dpif-newdp.c`. (2) Implement tất cả callback bắt buộc trong dpif_class (open, close, flow_put, flow_del, port_add, recv, etc.); callback optional có thể null. (3) Đăng ký lúc init: gọi `dp_register_provider(&new_class)` từ `dpif_init()` trong `lib/dpif.c`. (4) Build với `Makefile.am` thêm `lib/dpif-newdp.c` vào libopenvswitch_la_SOURCES. (5) Test với `ovs-vsctl set Bridge br0 datapath_type=newdp`. (6) Verify qua `ovs-appctl dpif/show`.

### Khi xảy ra sự cố

Operator on-call cần nhớ `dpif` khi gặp triệu chứng: (a) packet không đi qua bridge dù flow rule trông đúng — kiểm tra `dpif_open` có return success không (tail `ovs-vswitchd.log`); (b) `ovs-ofctl dump-flows` ra rule nhưng `ovs-appctl dpctl/dump-flows` không thấy entry tương ứng — capability mismatch giữa userspace classifier và kernel/netdev datapath; (c) bridge create báo lỗi "no such datapath type" — backend module chưa load (`lsmod | grep openvswitch`) hoặc DPDK chưa init.

### Liên quan mật thiết tới

`dpif` tightly coupled với năm thành phần. (1) `ofproto-dpif` là consumer chính; mọi xlate decision cuối cùng emit qua `dpif_*` API. (2) `openvswitch.ko` (kernel module) là provider chính cho `dpif-netlink`; module load/unload state ảnh hưởng dpif_open. (3) Netlink genl family `ovs_datapath`, `ovs_vport`, `ovs_flow`, `ovs_packet` là wire protocol giữa `dpif-netlink.c` và kernel module. (4) DPDK netdev abstraction (lib/netdev-dpdk.c) là provider cho `dpif-netdev` ở chế độ DPDK. (5) OVSDB Bridge.datapath_type là configuration quyết định class nào được instantiate.

### Khác biệt giữa các phiên bản

OVS 1.0 (2009) chỉ có Netlink kernel; chưa có abstraction. OVS 2.4 (2015) chính thức split `dpif-netdev.c` userspace; type `netdev` xuất hiện trong OVSDB. OVS 2.6+ thêm conntrack support trong userspace dpif (catch up với kernel ct() từ 2.5). OVS 2.8+ thêm AF_XDP backend (NOT covered trong curriculum này per ban directive). OVS 3.0 (2022) đánh dấu kernel module deprecation roadmap mainline (kernel module sẽ removed khỏi mainline kernel, OVS sẽ ship out-of-tree); `dpif-netlink` vẫn maintained nhưng userspace `dpif-netdev` được khuyến khích cho deployment mới.

### Cách quan sát + xác minh

Verify dpif runtime qua ba phương pháp. **Phương pháp 1 dpif/show:**

Reveal: configured backend từ OVSDB.

### Trường hợp sự cố thực tế

Production case (curated, OpenStack 2022-Q3 reproducible pattern): Sau khi node compute upgrade Linux kernel từ 5.4 → 5.15, `ovs-vswitchd` start lại, mọi bridge có `datapath_type=system` báo `dpif_open` fail với errno `EOPNOTSUPP`. Triệu chứng: VM trên node mất kết nối, `ovs-vsctl show` xuất bridge nhưng `ovs-appctl dpif/show` báo "no such datapath". Investigation: `lsmod | grep openvswitch` không thấy module loaded; `dmesg | tail` chỉ ra "openvswitch: module verification failed: signature and/or required key missing". Root cause: kernel 5.15 yêu cầu module signing strict; OVS module compile out-of-tree không được signed bởi distro key. Fix: cài `linux-modules-extra-5.15` từ Ubuntu repo (đã có module signed) hoặc disable signing requirement (rủi ro cao production).

### Bài tập synthetic

Lab dpif backend switch (mark "lab-pending" cho live verification khi lab host available):
1. Tạo bridge mới với backend mặc định: `ovs-vsctl add-br br-test`. Verify `ovs-vsctl get Bridge br-test datapath_type` → trả về `system`.
2. Quan sát dpif: `ovs-appctl dpif/show | grep br-test`. Note hit/missed counter.
3. Xoá bridge: `ovs-vsctl del-br br-test`.
4. Tạo lại với userspace backend: `ovs-vsctl add-br br-test -- set Bridge br-test datapath_type=netdev`.
5. Quan sát dpif lại: `ovs-appctl dpif/show | grep br-test`. So sánh.
6. Suy luận: tại sao hit/missed counter có scale khác? Vì kernel datapath dùng fast-path lookup trong kernel, userspace dpif-netdev dùng polling thread; cùng một packet rate nhưng mechanism khác.

### Lỗi thường gặp + tín hiệu chẩn đoán

Năm lỗi điển hình. (1) **dpif_open fail "no such datapath"**: backend module chưa load; signal log `ovs-vswitchd.log` "no datapath provider found"; recovery `modprobe openvswitch` (kernel) hoặc DPDK init. (2) **dpif_open fail EACCES**: process không có CAP_NET_ADMIN; signal `getcap $(which ovs-vswitchd)`; recovery cài systemd unit với `AmbientCapabilities=CAP_NET_ADMIN`. (3) **Capability mismatch xlate fail**: feature có ở kernel nhưng chưa ở userspace; signal log "action <X> unsupported by dpif netdev"; recovery: hoặc đổi backend, hoặc workaround pipeline OF. (4) **Migration nửa chừng**: đổi datapath_type khi bridge active không có hiệu lực; signal `dpif/show` vẫn ra backend cũ; recovery: del-br + add-br lại. (5) **Module signing reject** (kernel 5.15+ strict): signal `dmesg` "module verification failed"; recovery cài `linux-modules-extra` hoặc disable signing strict (không khuyến khích).

### So sánh với hệ khác

`dpif` về mặt thiết kế tương đồng VFS (Virtual File System) trong Linux kernel: cả hai là abstraction layer cho phép multiple backend đăng ký và chạy đồng thời, consumer (userspace ofproto-dpif tương ứng userspace process file ops) gọi qua interface thống nhất (struct dpif_class tương ứng struct file_operations). Khác Linux netfilter (mỗi hook có chain riêng, không phải vtable abstraction). Khác VPP graph node (FD.io VPP có graph runtime register node, mỗi node implement input/output, ý tưởng tương đồng nhưng VPP focus performance qua batch-per-node trong khi dpif focus modularity). Khác Cisco IOS-XE software model (Cisco có abstraction tầng DRIVER/HAL nhưng đóng kín, không document public). Đối tác gần nhất Linux kernel: `struct net_device_ops` cho NIC driver — ý tưởng vtable virtual dispatch giống hệt.

## 13.2.1 Logical Switch

A Logical Switch is an L2 domain. It is equivalent to a VLAN, but it is programmable and scales without the 4,096 VLAN limit.

```bash
[admin]# ovn-nbctl ls-add ls-tenant1
[admin]# ovn-nbctl lsp-add ls-tenant1 vm1
[admin]# ovn-nbctl lsp-set-addresses vm1 "aa:bb:cc:00:00:01 10.0.0.1"
[admin]# ovn-nbctl lsp-set-port-security vm1 "aa:bb:cc:00:00:01 10.0.0.1"
```

**LSP addresses**: `<MAC> <IP1> <IP2> ...`. A single LSP can carry multiple IP addresses.

Special LSP types:
- `type=router`: connects to a Logical_Router_Port (LRP).
- `type=localnet`: connects to a physical network.
- `type=vtep`: connects to a hardware VTEP.

---

## 13.2.2 Logical Router

```bash
[admin]# ovn-nbctl lr-add lr-tenant1
[admin]# ovn-nbctl lrp-add lr-tenant1 lrp-ls1 aa:bb:cc:00:00:fe 10.0.0.254/24
[admin]# ovn-nbctl lsp-add ls-tenant1 ls1-lrp -- \
         set Logical_Switch_Port ls1-lrp type=router \
         options:router-port=lrp-ls1 \
         addresses=router
```

- `lrp-add <lr> <lrp-name> <mac> <cidr>`.
- The peer LSP must have `type=router` and `options:router-port=<lrp-name>`.

A Logical_Router is not a physical device. It is distributed across all chassis that participate in the logical network.

## 13.2.3 Connect multiple subnets

A typical tenant has multiple Logical_Switches (web, app, db) connected through a single Logical_Router:

```bash
[admin]# ovn-nbctl ls-add ls-web
[admin]# ovn-nbctl ls-add ls-app
[admin]# ovn-nbctl ls-add ls-db
[admin]# ovn-nbctl lr-add lr-tenant1

# One LRP per subnet:
[admin]# ovn-nbctl lrp-add lr-tenant1 lrp-web aa:bb:cc:00:01:fe 10.0.1.254/24
[admin]# ovn-nbctl lrp-add lr-tenant1 lrp-app aa:bb:cc:00:02:fe 10.0.2.254/24
[admin]# ovn-nbctl lrp-add lr-tenant1 lrp-db aa:bb:cc:00:03:fe 10.0.3.254/24

# Peer LSP:
[admin]# ovn-nbctl lsp-add ls-web ls-web-lrp \
         -- set Logical_Switch_Port ls-web-lrp type=router \
            options:router-port=lrp-web addresses=router
# Repeat for app and db
```

Traffic between a VM in ls-web and a VM in ls-db is routed through lr-tenant1 automatically.

## 13.2.5 Pipeline evolution

Different OVN versions have different stage counts. The numbers below combine Logical_Switch and Logical_Router stages, taken from the `ovn-northd.8` man page and from `northd/northd.c` at each release tag. The exact count can drift by one or two stages between minor versions because of backports, so verify against the baseline you actually deploy.

- **OVN 20.06**: about 20 LS ingress + 25 LS egress.
- **OVN 22.03** (curriculum baseline): about **23 LS ingress + 9 LS egress + 20 LR ingress + 7 LR egress** (verified via `mcp__github__get_file_contents(northd/northd.c, ref=v22.03.8)`, grep `"ls_in_*"` and `"lr_in_*"`).
- **OVN 24.03+**: the pipeline grows with new stages (for example, PMTUD and output_large_pkt_detect, see the commit history of `northd/northd.c`); the source is split into several `en-lflows-*.c` files.

Part 19 analyses the multichassis and PMTUD pipeline diff between 22.03 and 24.03 in detail.

## 13.2.6 Provider network (localnet)

Connect a logical network to a physical VLAN:

```bash
[admin]# ovn-nbctl ls-add ls-provider
[admin]# ovn-nbctl lsp-add ls-provider provider-port \
         -- lsp-set-addresses provider-port unknown \
         -- lsp-set-type provider-port localnet \
         -- lsp-set-options provider-port network_name=physnet1
```

The `network_name=physnet1` value must match `external_ids:ovn-bridge-mappings=physnet1:br-ex` on the chassis.

## 13.2.7 Logical Switch Ingress pipeline, 27 stage exhaustive (Phase H S46)

OVN translates each Logical_Switch into a chain of OpenFlow tables on br-int. This section breaks each stage down using Template D (per the `ovn-architecture(7)` baseline, OVN 22.03).

### 13.2.7.1 The 27 LS ingress stages

| T | Stage name | Purpose | Match example |
|---|---|---|---|
| 0 | `ls_in_check_port_sec` | Port security incoming check (drop if MAC or IP does not match the LSP) | `inport=="p1" && eth.src==MAC` |
| 1 | `ls_in_apply_port_sec` | Apply the verdict produced by check_port_sec | `reg0[15] == 0` pass |
| 2 | `ls_in_lookup_fdb` | Consult FDB for a known source MAC | `inport=="p1" && eth.src==MAC_A` |
| 3 | `ls_in_put_fdb` | Insert a new MAC into the FDB (learn action) | `reg0[11] == 0`, not yet known |
| 4 | `ls_in_pre_acl` | Pre-ACL CT track commit (set reg0[0]=trk) | `ip` |
| 5 | `ls_in_pre_lb` | Pre-LB CT track (for VIP traffic) | `ip4.dst == VIP` |
| 6 | `ls_in_pre_stateful` | Commit the packet to the conntrack zone | `reg0[0] == 1` |
| 7 | `ls_in_acl_hint` | Generate hints to reduce redundant CT commits | `ct.est && ct.rpl` |
| 8 | `ls_in_acl_eval` | ACL evaluation (direction=from-lport) | `ip4.dst==10.0.0.5 && tcp.dst==80` |
| 9 | `ls_in_acl_action` | Apply the ACL verdict (drop, reject, or allow) | `reg0[17] == 1` |
| 10 | `ls_in_qos` | QoS mark and meter | `inport=="p1" && ip.dscp_set` |
| 11 | `ls_in_lb_aff_check` | Load balancer affinity check (pinned session) | `ct.new && ip4.dst == VIP` |
| 12 | `ls_in_lb` | LB VIP to backend selection (hash-based) | `ct.new && ip4.dst == VIP` |
| 13 | `ls_in_lb_aff_learn` | Learn an affinity entry for persistence | `ct.new && reg0[9] == 1` |
| 14 | `ls_in_acl_after_lb_eval` | Re-evaluate ACL after the LB rewrite | `ip4.dst == backend_IP` |
| 15 | `ls_in_acl_after_lb_action` | Apply the verdict after LB | - |
| 16 | `ls_in_stateful` | Final CT commit with ct_mark or ct_label | `reg0[1] == 1` |
| 17 | `ls_in_pre_hairpin` | Detect hairpin (VIP to a backend on the same LS, a loop risk) | - |
| 18 | `ls_in_nat_hairpin` | NAT hairpin SNAT for local traffic | - |
| 19 | `ls_in_hairpin` | Redirect hairpin traffic back to the originator | - |
| 20 | `ls_in_arp_rsp` | ARP responder (local reply for the gateway IP) | `arp.op==1 && arp.tpa==GW_IP` |
| 21 | `ls_in_dhcp_options` | DHCPv4 and DHCPv6 options lookup | `inport=="p1" && udp.dst==67` |
| 22 | `ls_in_dhcp_response` | Generate the DHCP response packet | `reg0[3] == 1` |
| 23 | `ls_in_dns_lookup` | DNS query processing | `udp.dst==53` |
| 24 | `ls_in_dns_response` | DNS response generation | `reg0[4] == 1` |
| 25 | `ls_in_external_port` | Bridging to an external port (localnet) | `outport=="extport"` |
| 26 | `ls_in_l2_lkup` | Destination MAC lookup, set outport | `eth.dst == MAC_B` |

### 13.2.7.2 Template D for `ls_in_acl_eval` (T8), the most common stage

| Attribute | Value |
|---|---|
| **Table** | T8 (logical); physical table is around T16 in OVN 22.03 |
| **Stage name** | `S_SWITCH_IN_ACL_EVAL` in `northd/northd.h` |
| **Purpose** | Evaluate ACL rules from `NBDB.ACL` with direction=from-lport |
| **Match criteria** | The ACL expression compiled into an OpenFlow match |
| **Typical actions** | `reg0[17]=1;` (allow), `reg0[17]=0;` (drop), `reject { ... };` (generate ICMP) |
| **Produced by** | `ovn-northd` from `NBDB.ACL` direction=from-lport |
| **State input** | `reg0[0]` (ct_hint), `reg0[9]` (ct_est) |
| **State output** | `reg0[17]` (acl_allow flag) |

**Logical flow sample** (`ovn-sbctl lflow-list`):

```
  table=8 (ls_in_acl_eval), priority=65535,
    match=(!ct.est && ct.rel && !ct.rpl && ct_mark.blocked == 0),
    action=(reg0[17] = 1; next;)
  table=8 (ls_in_acl_eval), priority=2001,
    match=(inport == @pg_web && ip4.dst == $ip_db && tcp.dst == 3306),
    action=(reg0[17] = 1; next;)
  table=8 (ls_in_acl_eval), priority=2000,
    match=(ip),
    action=(reg0[17] = 0; next;)
```

Anatomy: priority 65535 covers established and related CT; 2001 carries the user ACL; 2000 is the default drop. `@pg_web` is a Port_Group reference; `$ip_db` is an Address_Set reference.

**Failure scenario:** traffic that should be allowed is dropped instead. Check the `ovn-trace` output. The usual root cause is a missing `allow-related` ACL or a fall-through to the default 2000 drop.

### 13.2.7.3 Template D for `ls_in_lb` (T12), Load balancer

| Attribute | Value |
|---|---|
| **Purpose** | LB VIP to backend selection via the `ct_lb_mark` action |
| **Match** | `ct.new && ip4.dst == VIP && tcp.dst == vport` |
| **Actions** | `ct_lb_mark(backends=<ip>:<port>...);` |
| **Produced by** | `NBDB.Load_Balancer` |
| **State input** | The `ct.new` flag |

**Logical flow:**

```
table=12 (ls_in_lb), priority=100,
  match=(ct.new && ip4.dst == 10.0.0.100 && tcp.dst == 80),
  action=(ct_lb_mark(backends=10.0.1.1:8080,10.0.1.2:8080,10.0.1.3:8080);)
```

The backend hash is deterministic per connection (5-tuple). Session affinity is handled by the `ls_in_lb_aff_*` stages.

### 13.2.7.4 Stage `ls_in_arp_rsp` (T20) in depth

```
table=20 (ls_in_arp_rsp), priority=50,
  match=(arp.op == 1 && arp.tpa == 10.0.0.1 /* gateway */),
  action=(eth.dst = eth.src;
          eth.src = aa:bb:cc:dd:ee:ff;
          arp.op = 2;
          arp.tha = arp.sha;
          arp.sha = aa:bb:cc:dd:ee:ff;
          arp.tpa = arp.spa;
          arp.spa = 10.0.0.1;
          outport = inport;
          flags.loopback = 1;
          output;)
```

Anatomy: a ten-step action chain converts the ARP Request into an ARP Reply and reflects it back to the ingress port. Compare with the OpenFlow equivalent in Part 4.9 §4.9.19 (the classic OpenFlow seven-action ARP responder).

## 13.2.8 Logical Switch Egress pipeline, the 10 stages (foundation gap closed)

**Audit finding:** before session S46, no file in the curriculum mentioned `ls_out_*`. This section closes a critical foundation gap.

The OVN egress pipeline runs after `ls_in_l2_lkup` sets the outport, traversing 10 stages before the packet exits via tunnel or bridge.

### 13.2.8.1 The 10 LS egress stages

| T | Stage name | Purpose |
|---|---|---|
| 0 | `ls_out_pre_lb` | Egress pre-LB tracking (reverse LB direction) |
| 1 | `ls_out_pre_acl` | Egress pre-ACL CT track |
| 2 | `ls_out_pre_stateful` | Egress CT commit |
| 3 | `ls_out_acl_hint` | Egress ACL hint |
| 4 | `ls_out_acl_eval` | Egress ACL evaluation (direction=to-lport) |
| 5 | `ls_out_acl_action` | Apply egress verdict |
| 6 | `ls_out_qos` | Egress QoS mark/meter |
| 7 | `ls_out_stateful` | Final egress CT commit |
| 8 | `ls_out_port_sec_ip` | Port security outgoing IP check |
| 9 | `ls_out_port_sec_l2` | Port security outgoing L2 check |

### 13.2.8.2 `ls_out_acl_eval` versus `ls_in_acl_eval`

The key difference: the egress side matches on **`outport`** instead of `inport`. The NBDB ACL direction column drives this choice:

```shell
# Egress ACL from web to db
$ sudo ovn-nbctl --direction=to-lport acl-add ls-prod 1000 \
    "outport == @pg_db && ip4.src == $ip_web && tcp.dst == 3306" allow-related
```

The corresponding logical flow:

```
table=4 (ls_out_acl_eval), priority=1000,
  match=(outport == @pg_db && ip4.src == $ip_web && tcp.dst == 3306),
  action=(reg0[17] = 1; next;)
```

### 13.2.8.3 `ls_out_port_sec_ip` and `ls_out_port_sec_l2`

These stages implement port security on the egress side. They ensure that an LSP only receives packets whose MAC and IP match the `addresses` configuration on that LSP. The purpose is to prevent receiver-side spoofing.

```
table=8 (ls_out_port_sec_ip), priority=90,
  match=(outport == "lsp-vm-db" && eth.dst == aa:bb:cc:dd:db && ip4.dst == 10.0.0.10),
  action=(next;)

table=9 (ls_out_port_sec_l2), priority=50,
  match=(outport == "lsp-vm-db" && eth.dst == aa:bb:cc:dd:db),
  action=(output;)

table=9 (ls_out_port_sec_l2), priority=0,
  match=(1),
  action=(drop;)  /* default drop, no address match */
```

### 13.2.8.4 End-to-end pipeline trace

```
Packet arrives at host X -> Geneve decap -> br-int physical T8 ->
Ingress pipeline T0-T26 (ls-prod-1) ->
  inport=p1, stage 0-26, outport set at T26 ->
Egress pipeline T0-T9 (ls-prod-1) ->
  outport=p2, final CT commit, port_sec check ->
Physical T65 -> wire
```

See Part 9.27 for the 3-tier diagnostic flow (ovn-trace at the logical layer, ofproto/trace at the physical layer, dpif/dump-flow at the datapath layer).

```shell
    match=(eth.src == $<pg_ls-tenant-web_ports> && inport == <pg_ls-tenant-web_ports>),
    action=(next;)
  table=1 (ls_in_port_sec_ip    ), priority=90  , ...
  table=8 (ls_in_acl            ), priority=1000,
    match=(ip && ct.new && outport == @pg_web && tcp.dst == 8080),
    action=(reg0[1] = 1; next;)
  table=19 (ls_in_l2_lkup        ), priority=50  ,
    match=(eth.dst == aa:bb:cc:00:00:01),
    action=(outport = "lsp_web_01"; output;)

Datapath: "ls-tenant-web" (7c9d4e2a-...)  Pipeline: egress
  table=0 (ls_out_pre_lb         ), priority=0   , match=(1), action=(next;)
  table=8 (ls_out_port_sec_ip    ), priority=90  , ...
```

**9-attribute anatomy:**

| Field | Sample value | Meaning | Red flag if |
|-------|--------------|---------|-------------|
| `Datapath` line | `"ls-tenant-web" (7c9d4e2a-...)` | Identifies the LS | Missing means the LS has not yet been compiled into SBDB |
| `Pipeline` | `ingress` or `egress` | Direction of the stage block | Only one pipeline visible means northd has not finished compiling |
| `table=N` | 0 to 26 ingress, 0 to 9 egress | Stage table ID inside the pipeline | A gap in numbering means a stage is missing (bug) |
| Stage name | `ls_in_port_sec_l2`, `ls_in_acl` | Semantic purpose of the stage | A name mismatch against the expected OVN version indicates version skew |
| `priority` | 0 (default) to 32767 (max) | Within-table matching priority | Priority 0 is the catch-all default |
| `match` | `eth.src == $<pg>` | OVN expression language | An invalid expression produces a compile error in the log |
| `action` | `next;` or `outport = ...; output;` | Action pipeline | A missing `next;` causes a pipeline dead end |
| ACL stage count | 1 ACL = 2 OpenFlow rules | NBDB to SBDB ratio | If the ratio does not scale as expected, conjunction is not applying |
| Stage traversal | 0, 1, ..., 26, then egress 0, ..., 9 | Sequential pipeline | A skipped stage shows up as a gap in numbering |

**Failure scenarios:**

- **Datapath has the ingress pipeline but not the egress pipeline.** northd compile is incomplete. Wait 2 to 3 seconds; if the egress pipeline still does not appear, check the ovn-northd log.
- **Table 8 `ls_in_acl` is empty even though ACLs exist in NBDB.** The ACL match expression has a syntax error and northd rejected it. Check `/var/log/ovn/ovn-northd.log`.
- **Multiple ACLs overlap at priority 1000.** The evaluation order is undefined. Fix: assign distinct priorities per ACL.
- **Stage `ls_in_l2_lkup` at table 19 priority 50 falls back to flood.** The destination MAC is unknown and the ARP responder did not catch it. A VM declared with `addresses=unknown` falls back to flood.

## 13.2.Y Common misconception and Key point (v3.2 expansion)

> **Common misconception 1:** *"A Logical Router is a physical server, a chassis dedicated to routing."*
>
> **Reality:** the Logical Router (LR) is a **distributed abstraction**: routing logic is installed on every chassis. When a VM on chassis A sends a packet to the LR (the destination IP matches an LR network), the packet stays local on chassis A and runs the LR pipeline (decrement TTL, ARP-resolve the next hop, output through the LRP). It does not cross to another chassis unnecessarily. Only when the LR has to route to the external network (north-south NAT) does the traffic go through a chassisredirect port to the gateway chassis. This "distributed LR" model is one of the core benefits of OVN compared to traditional L3 routing through a physical router.

> **Common misconception 2:** *"OVN LS and LR are different from Neutron router and subnet, so legacy Neutron cannot be migrated to OVN."*
>
> **Reality:** the OVN Neutron ML2-OVN driver (`networking-ovn`) automatically maps Neutron network, router, and subnet objects to OVN LS, LR, LRP, and static route entries. The migration path has official tooling and documentation. Red Hat OpenStack Platform 16 (2020) defaulted to migrating legacy ML2-Linuxbridge deployments to ML2-OVN. Pain points to expect: (1) Floating IP behavior changes from centralized to distributed DNAT_and_SNAT, (2) Security Group translation has edge cases where stateful rule semantics differ, (3) the DHCP agent is removed in favor of OVN native DHCP. See the RHOSP migration guide for details.

> **Key point:** the 27 ingress and 10 egress stages of the LS pipeline are the core output of the OVN compiler. Each stage has a clear semantic purpose: port_security (anti-spoofing), FDB learn (MAC awareness), pre-CT (conntrack state check), ACL (policy), QoS (rate limit), LB (load balance plus hairpin), stateful commit, ARP responder, DHCP and DNS reply, L2 lookup (MAC to port). Knowing the stage sequence helps debugging: for example, "the packet reaches L2 lookup but does not forward" points at stage 19 `ls_in_l2_lkup` having an empty match, which is a MAC table issue. Section 13.2 (the full pipeline table from Phase H) is the reference for every stage.

## 13.2.Z ▶ Capstone POE: 3-tier tenant end-to-end ping trace (v3.2 expansion)

**Goal:** trace the full packet journey from a tenant-web VM to a tenant-db VM through LS, LR, and multiple pipeline stages, then confirm that policy enforcement behaves as intended.

**Scenario:** a 3-tier tenant `tenant-T1` with LS-web, LS-app, and LS-db, connected by a single LR-tenant that links all three LS instances. ACLs allow web to app on port 8080 and app to db on port 5432. Test by pinging from web to app (expected to succeed) and from web to db (expected to be blocked).

**Predict:**

Web to App:
- The packet enters the LS-web ingress pipeline (27 stages): port_sec, then FDB learn, then ACL (hits the allow-related rule for pg_web to pg_app), then L2 lookup (output port `ls-to-lr`).
- It transitions to the LR-tenant ingress pipeline: routing decision (the destination IP is in the LS-app subnet), output through LRP-app.
- It transitions to the LS-app egress pipeline: ACL (allow reply), then port_sec, then L2 lookup, then output to `lsp_app_01`.
- Expectation: ping succeeds with 2 to 5 ms latency (three pipeline traversals).

Web to DB:
- The packet enters the LS-web ingress pipeline: port_sec passes, then ACL evaluation begins.
- ACL table 8 priority 1000 match `pg_web -> pg_app` does not match because `outport = pg_db`.
- ACL table 8 priority 1 match `ip` triggers a drop action.
- The packet is dropped at the ACL stage.
- Expectation: ping fails with no reply.

**Observe:**

```bash
# Ping web to app
vm_web$ ping -c 5 10.0.1.5  # lsp_app_01
# Expectation: 5 packets received, around 2 ms RTT.

# Ping web to db
vm_web$ ping -c 5 10.0.2.5  # lsp_db_01
# Expectation: 100 percent packet loss.

# Trace the packet drop
central$ sudo ovn-trace ls-tenant-web \
    'inport=="lsp_web_01" && eth.src==aa:bb:cc:00:00:01 && \
     ip4.src==10.0.0.5 && ip4.dst==10.0.2.5 && ip.ttl==64 && \
     tcp.src==54321 && tcp.dst==5432'

# Output shows pipeline traversal:
# ... ingress ls_in_port_sec_l2 (pass) ...
# ... ingress ls_in_acl priority=1 match=(ip) action=drop ...
# PACKET DROPPED
```

**Explain:**

- ACL policy enforcement is correct: web to app is allowed by the priority 1000 specific rule, and web to db is denied by the priority 1 default-deny rule.
- The trace confirms the drop occurs at ingress table 8 (`ls_in_acl`) inside LS-web, before the packet reaches the LR.
- Distributed LR behavior: there is no cross-chassis traffic when the web and app VMs share a chassis.

**Falsification test:**

Remove the default-deny ACL at priority 1 and rerun the ping:
```bash
central$ sudo ovn-nbctl acl-del ls-tenant-web from-lport 1 "ip"
```

Retry ping web to db. Expectation: the ping now succeeds because the default-deny rule is gone. This confirms ACL priority semantics: default-deny is a required guardrail. Without default-deny the firewall is effectively open.

**Finish:**

Principle: **"the default-deny plus specific-allow pattern is the foundation of Zero Trust networking."** Apply this pattern to every OVN tenant. The pattern scales through Port_Group and conjunction, can be verified through `ovn-trace` regularly, and should fire an alert whenever the ACL rule set drifts from the baseline.

---

## Key takeaways

First, LS is an L2 domain and LR is an L3 router. They connect through an LRP plus a peer LSP of `type=router`.

Second, the LR is distributed across chassis. There is no physical router as a bottleneck.

Third, the pipeline table count varies across OVN versions. The v22.03 baseline is 24 plus 27 stages.

Fourth, the `localnet` LSP connects the logical network to a physical VLAN through bridge mapping.

Fifth, the LS ingress pipeline at v22.03.8 has 26 stages with the following high-level structure: port security, FDB learn, pre-CT, ACL eval and action, QoS, LB (with affinity), stateful commit, hairpin, ARP responder, DHCP and DNS, external port, L2 lookup. Each stage has a clear purpose, and OVN flag conditions drive the transitions between stages.

Sixth, the LS egress pipeline (10 stages) is symmetric with ingress for ACL, CT, and QoS, but it matches on `outport`. Port security runs at the end of the pipeline (T8 IP and T9 L2) to enforce receiver-side filtering.

## 13.2.X Logical_Switch as an NBDB table: tenant L2 broadcast domain deep-dive

> This section drills into the **`Logical_Switch` table itself** (a row in the OVN Northbound database), complementing the hands-on teaching in §13.2.1 and the pipeline coverage in §13.2.7-13.2.8. Goal: the engineer fully understands the structure of an NBDB row, the eleven columns of the 22.03 schema, the compile lifecycle into a Datapath_Binding plus 27 ingress Logical_Flow stages plus 10 egress stages, and how to trace a runtime incident back to a row in NBDB.

### Concept

`Logical_Switch` is a **table that lives in the OVN Northbound database (NBDB)** and describes a **virtual L2 broadcast domain** that the CMS (Neutron, kube-ovn, OpenStack) declares for a tenant. Each row represents one logical Ethernet segment: every `Logical_Switch_Port` (LSP) attached to the row belongs to the same broadcast domain, the LSPs learn each other's MAC addresses, and they share the same set of ACLs, DHCP options, load balancers, and DNS records. Type: **OVSDB table**, root table (`isRoot=true` in `ovn-nb.ovsschema`), persistent (replicated across the NBDB Raft cluster), declarative input to ovn-northd. Unlike a traditional Linux bridge (which is per-host), a Logical_Switch **spans every chassis** in the cluster through the Geneve tunnel mesh; the L2 broadcast domain is not bounded by physical wire.

### History and background

The `Logical_Switch` table appeared with OVN 2.4 (2015) in the refactor by Russell Bryant (Red Hat) and Justin Pettit (VMware/Nicira) that ported the Nicira NSX OpenStack plugin design to OVN. Before that, the Neutron OVS plugin mapped each network to a VLAN tag on a per-host integration bridge; scale was limited by the 4096 VLAN ID ceiling and required controller-side state on every host. OVN separated the layers: NBDB stores **intent** (the Logical_Switch row the CMS writes), SBDB stores **runtime state** (the Datapath_Binding row and Logical_Flow rows compiled by northd). Logical_Switch is one of the original NBDB tables together with `Logical_Router`, `Logical_Switch_Port`, `Logical_Router_Port`, and `ACL`. OVN 21.03 added `forwarding_groups` (load-balancer-style multi-port forwarding), 21.06 stabilised `dns_records` for native DNS, 22.03 added `copp` (Control Plane Policing) to the schema (verified against `ovn-nb.ovsschema` on branch-22.03), and 24.03 extended `other_config` with keys for advanced IGMP and MLD snooping.

### Position in the architecture

The table sits at the **NBDB layer**, the topmost input layer of the OVN control plane. Scope: **per-cluster**. An OVN cluster has a single NBDB Raft cluster (three or five members); the CMS writes NBDB through `ovn-nbctl` or a client API; ovn-northd subscribes to NBDB as a reader and compiles into SBDB. Each Logical_Switch row produces **exactly one Datapath_Binding row** in SBDB (`ovn-sbctl find Datapath_Binding external_ids:name=<ls_name>`) and **on average 50 to 200 Logical_Flow rows** (depending on LSP, ACL, and load balancer counts) distributed across 27 ingress stages plus 10 egress stages. Blast radius: changing one Logical_Switch row (for example, adding an ACL to the `acls` column) makes ovn-northd recompute that datapath in less than 200 milliseconds (incremental compile since OVN 21.03); ovn-controller on every chassis then re-installs OpenFlow rules in less than one second through the I-P engine.

### Role

Logical_Switch carries **five roles** in the OVN architecture:

1. **L2 broadcast domain identifier**: the `name` column (string, not unique) is a human-readable identifier; `_uuid` is the unique key. Every LSP in the row belongs to the same broadcast domain.
2. **Container for LSPs**: the `ports` column (set of strong references to `Logical_Switch_Port`) provides cascade delete: deleting the LS deletes every LSP. The CMS does not need to clean up ports manually.
3. **Anchor for L2 policy**: the columns `acls` (set of strong refs to `ACL`), `qos_rules` (set of refs to `QoS`), `load_balancer` (set of weak refs), `dns_records` (set of weak refs to `DNS`), and `forwarding_groups` (set of refs to `Forwarding_Group`). This is the policy attachment point at LS scope; cluster-wide policies attach through Address_Set and Port_Group separately.
4. **Runtime parameters for northd**: the `other_config` column (map of string to string) holds `mcast_snoop`, `mcast_querier`, `vlan-passthru`, `vni` (for the transit tunnel), and `interconn-ts`. Northd reads this map to decide how to compile the pipeline.
5. **Metadata for the CMS**: the `external_ids` column (map) holds `neutron:network_name`, `kube-ovn:vpc`, or similar values that the CMS uses to map an OVN row back to its source object.

### Why it exists (design motivation)

Before OVN 2.4, the Neutron OVS-ML2 plugin architecture (from Folsom in 2012 to Liberty in 2015) stored network state in a Neutron MySQL database and pushed VLAN or VXLAN configuration to each compute host through RPC; each compute host built a Linux bridge per network locally. Problems: (a) the **4096 VLAN scaling limit** for VLAN-based segmentation; VXLAN solved the VNI count but still required a per-host controller to coordinate state; (b) **state drift when RPC failed**, leaving the compute host and the Neutron database inconsistent; (c) **hard reconfiguration when adding an ACL**, which required pushing new iptables rules through the agent. OVN solved this by **moving state into OVSDB Raft, letting northd compile, and letting each chassis install its own flows**: NBDB holds the declarative Logical_Switch row, northd compiles to Logical_Flow rows, and ovn-controller runs locally on each chassis to install OpenFlow rules. Logical_Switch is the **only tenant-facing abstraction** that the CMS knows about: the CMS does not need to know about Datapath_Binding or Logical_Flow, only how to create and delete Logical_Switch and LSP rows. This model scales to 5000 to 10000 hypervisors and more than 100,000 logical ports (data from Red Hat OSP 17 and ovsdb-server performance benchmarks in 2023).

### Problems it solves

This table addresses **five distinct technical problems**: (1) **decouple intent from physical**: the CMS only declares "tenant1 needs an L2 segment" without having to know the VLAN, the host, or the tunnel; (2) **scope policy at the right level**: an ACL attached to an LS row only affects ports of that LS, instead of pushing global iptables on every rule change; (3) **consistent cascade lifecycle**: deleting the LS row deletes every LSP, ACL, QoS rule, and forwarding_group that belongs to it through strong references, so the CMS does not need a pseudo-transaction; (4) **multi-cluster federation through interconnect**: the `other_config:interconn-ts` column marks a transit switch between availability zones (OVN-IC since 21.06); (5) **deterministic cross-chassis broadcast**: the same LS row gives the same `tunnel_key` for the datapath and the same metadata in the Geneve packet, so a broadcast frame is flooded to exactly the set of chassis that hold LSPs of that LS (through the MC_FLOOD multicast group in SBDB).

### Importance

This is a **top-tier foundational table of NBDB**, on the same level as `Logical_Router`. The reasons: 100 percent of OVN tenant network workflows start with `ovn-nbctl ls-add`; 100 percent of advanced OVN features at L2 scope (stateful ACL, native LB for east-west, DHCP and DNS, port security, IGMP snooping) add columns or option fields to Logical_Switch instead of creating new tables. An engineer who does not understand Logical_Switch cannot create, debug, or scale any tenant network in OVN. Among the more than 320 keywords in the REF, Logical_Switch is in the cornerstone group of 15 OVN keywords (alongside `ovn-northd`, `ovn-controller`, NBDB, SBDB, Logical_Flow, Datapath_Binding, Logical_Router, Port_Binding, ACL, Load_Balancer, NAT, Geneve TLV, ovn-trace, and ovn-detrace).

### How it works (the compile cycle of a row)

A Logical_Switch row goes through **four phases** in its lifecycle:

1. **Declaration from the CMS**: the CMS calls `ovn-nbctl ls-add ls-tenant1`, an NBDB transaction creates a new row with `name="ls-tenant1"` and the other set columns defaulting to `[]`. The transaction is atomic; the row becomes visible in less than 50 milliseconds through NBDB Raft replication.
2. **Compile to Datapath_Binding**: ovn-northd (running on the NBDB Raft leader and writing to SBDB) detects the new row through an I-P engine input change. The function `ovn_datapath_create` in `northd/northd.c` (branch-22.03) creates a `struct ovn_datapath` in memory for the LS row and allocates a `tunnel_key` (1 to 16777215, unique within the cluster). Then `northd_handle_lb_data_changes` and `build_datapaths` write into SBDB: `INSERT INTO Datapath_Binding (tunnel_key=<allocated>, external_ids={'name': 'ls-tenant1', 'logical-switch': '<ls_uuid>'})`. Result: one Datapath_Binding row corresponds 1:1 with the Logical_Switch row.
3. **Compile to Logical_Flow rows**: northd runs the `build_lswitch_*` family of functions (for example, `build_lswitch_input_port_security_l2`, `build_lswitch_arp_nd_responder_known_ips`, `build_lswitch_destination_lookup_bmcast`, `build_lswitch_ip_unicast_lookup`) to produce Logical_Flow rows for each stage. Every LSP, ACL, load balancer, and DNS record attached to the LS spawns its own flows. Total: 50 to 500 Logical_Flow rows per LS in typical deployments (multiplicative in LSP count times ACL count times stage count).
4. **Install flows on the chassis**: SBDB Raft replicates Logical_Flow rows; ovn-controller on every chassis subscribes; each controller runs `lflow_run` in `controller/lflow.c` to translate Logical_Flow expressions into concrete OpenFlow rules and installs them on `br-int` through `ofctrl_put`. Each flow is tagged with `cookie=<datapath_uuid_low_64>` to make `dump-flows` filtering by LS easy.

Invariant: at any moment, a Logical_Switch row that exists in NBDB corresponds to **exactly one** Datapath_Binding row in SBDB; deleting the LS through `ls-del` cascades to delete the LSPs, the Datapath_Binding row, and the Logical_Flow rows in the same compile round.

### Classification of columns

The 11 columns of Logical_Switch (verified against `ovn-nb.ovsschema` on branch-22.03, [GitHub raw](https://raw.githubusercontent.com/ovn-org/ovn/branch-22.03/ovn-nb.ovsschema)) split into **five groups** by semantic role:

| Group | Column | Type | Required | Notes |
|------|-----|------|----------|---------|
| Identifier | `name` | string | Yes (default "") | CMS-friendly, not unique |
| Container | `ports` | set of strong refs to Logical_Switch_Port | No (default []) | Cascade delete |
| LS-attached policy | `acls` | set of strong refs to ACL | No | Stateful firewall |
| LS-attached policy | `qos_rules` | set of strong refs to QoS | No | Mark and meter |
| LS-attached policy | `load_balancer` | set of weak refs to Load_Balancer | No | Legacy east-west LB |
| LS-attached policy | `load_balancer_group` | set of refs to Load_Balancer_Group | No (24.03+) | Modern group LB |
| LS-attached policy | `dns_records` | set of weak refs to DNS | No | Native DNS resolver |
| LS-attached policy | `forwarding_groups` | set of strong refs to Forwarding_Group | No (21.03+) | LB-style L2 multi-output |
| LS-attached policy | `copp` | weak ref to Copp (max 1) | No (22.03+) | Control plane meter |
| Service parameters | `other_config` | map string to string | No | mcast_snoop, vni, interconn-ts |
| CMS metadata | `external_ids` | map string to string | No | CMS-tracked ID |

There is no unique index constraint on `name`; duplicate names are valid (the CMS must guarantee uniqueness on its own if needed). The only unique lookup key is `_uuid`.

### How an operator masters this

Four foundational skills the engineer must have for Logical_Switch:

1. **Read a full row**: `sudo ovn-nbctl --columns=name,ports,acls,load_balancer,dns_records,other_config list Logical_Switch <ls_uuid_or_name>` shows the six main columns and the size of each set. Alternatively, `sudo ovn-nbctl ls-list` gives an overview, and `sudo ovn-nbctl show ls-tenant1` prints the LS to LSP tree with addresses.
2. **Verify the compile to SBDB**: `sudo ovn-sbctl find Datapath_Binding external_ids:logical-switch=<ls_uuid>` confirms that a corresponding Datapath_Binding exists. If it is missing, northd has not finished compiling yet, or there is an error in `ovn-northd.log`.
3. **Count the pipeline output**: `sudo ovn-sbctl lflow-list <ls_name> | wc -l` counts the Logical_Flow rows. An LS with no LSPs has only about 30 to 50 stub flows; an LS with 5 LSPs and 3 ACLs has about 200 to 300 flows; an LS with 100 LSPs and 50 ACLs has more than 2000 flows. The flow count grows roughly linearly with LSP count times ACL count.
4. **Decide which skill to use when**: if the CMS reports "tenant network missing in the cluster", start with `ls-list` (skill 1) to confirm the row exists; if the CMS reports "VMs on the network do not learn each other's MAC", check `lflow-list` at stage `ls_in_l2_lkup` (skill 3); if you suspect northd is stuck, check Datapath_Binding (skill 2).

### Usage workflow

The workflow for provisioning a three-tier tenant network through Logical_Switch has six steps, each with a verification command:

1. **CMS creates the Logical_Switch in NBDB**: `ovn-nbctl ls-add ls-tenant-web`. Verify: `ovn-nbctl ls-list | grep ls-tenant-web` shows the UUID and name.
2. **Attach stateful ACL and LB if needed**: `ovn-nbctl acl-add ls-tenant-web from-lport 1000 'ip4 && tcp.dst==80' allow-related`; `ovn-nbctl ls-lb-add ls-tenant-web lb-web`. Verify: `ovn-nbctl --columns=acls,load_balancer list Logical_Switch ls-tenant-web` shows the set with entries.
3. **Wait for northd to compile**: typically less than 200 milliseconds. Verify: `ovn-sbctl find Datapath_Binding external_ids:logical-switch=<ls_uuid>` shows one Datapath_Binding row with an allocated `tunnel_key`.
4. **Create one LSP per VM**: `ovn-nbctl lsp-add ls-tenant-web vm-web-1; ovn-nbctl lsp-set-addresses vm-web-1 "aa:bb:cc:00:00:01 10.0.0.5"`. Verify: `ovn-nbctl ls-show ls-tenant-web` lists the port under the LS.
5. **Plug the VM tap into br-int on the hypervisor and set iface-id**: `ovs-vsctl add-port br-int tap-vm-web-1 -- set Interface tap-vm-web-1 external_ids:iface-id=vm-web-1`. Verify: `ovn-sbctl --columns=chassis,up list Port_Binding vm-web-1` shows `chassis=[<uuid>], up=true` (see Part 13.5.1b for Port_Binding lifecycle details).
6. **Check the pipeline output**: `ovn-sbctl lflow-list ls-tenant-web | head -30` confirms that flows exist at stages `ls_in_port_sec_l2`, `ls_in_acl_eval`, and `ls_in_l2_lkup` for the MAC just installed.

Production best practice: **always use Address_Set and Port_Group instead of hardcoding IP and MAC in ACL** for tenant scale-up. Anti-pattern: creating a separate Logical_Switch for every small pod or container, which inflates the Datapath_Binding count to tens of thousands, makes northd compile take more than 30 seconds, and causes ovn-controller upcall lag. Reasonable size: at most 5000 LSPs per LS; above 5000, split into multiple LSes connected through an LR.

### When something goes wrong (reverse symptom map)

Seven common symptoms point back to Logical_Switch:

| Symptom | Logical_Switch column to check | Diagnostic flow |
|---------|------------------------------|-----------------|
| "Tenant network does not appear in the CMS" | `name`, `external_ids` | `ovn-nbctl ls-list` confirms the row exists |
| "VMs in the same LS cannot ping each other" | `acls` | `ovn-nbctl acl-list ls-X` checks for a default-deny rule; `ovn-trace` from source to destination |
| "ACL allow rule does not take effect" | `acls`, ACL row priority and match | Compare the ACL in NBDB with the Logical_Flow in SBDB at stage `ls_in_acl_eval` |
| "DHCP from VM is not served" | (LSP `dhcpv4_options`) and LS `other_config` | Check `ovn-nbctl lsp-get-dhcpv4-options vm1` and `lflow-list ls-X` at stage `ls_in_dhcp_options` |
| "Internal DNS query times out" | `dns_records` | `ovn-nbctl --columns=dns_records list Logical_Switch ls-X`; check that the DNS table has the A record |
| "Broadcast or multicast flood is too large" | `other_config:mcast_snoop`, `mcast_querier` | Set `mcast_snoop=true` plus querier to enable IGMP snooping and reduce flood |
| "northd compile is slow for a large LS" | Number of LSPs in `ports`, number of ACLs | Split the LS into smaller LSes connected through an LR |

### Tightly related to

Six components are tightly coupled with Logical_Switch (changing any of them forces you to revisit Logical_Switch):

- **`Logical_Switch_Port`** (LSP): contained in the `ports` column as a strong ref. Each LSP belongs to exactly one LS; deleting the LS cascades to delete the LSPs.
- **`ACL`**: contained in the `acls` column as a strong ref. ACL is LS-scoped; deleting the LS cascades to delete the ACLs attached to it. Cluster-wide ACLs are scoped through Port_Group instead of attaching directly to Logical_Switch.
- **`Datapath_Binding` (SBDB)**: 1:1 mapping through `external_ids:logical-switch=<ls_uuid>`. This is the first output of the northd compile.
- **`Logical_Flow` (SBDB)**: 1:N mapping. Each LS produces tens to hundreds of flow rows.
- **`Logical_Router_Port`**: an LSP of type `router` is attached to the LS to peer with the LRP of a Logical_Router (see §13.2.2). This is the bridge between L2 and L3.
- **OVS `Interface.external_ids:iface-id`**: indirectly, through the LSP. The `iface-id` of the OVS Interface matches the `name` of the LSP, the LSP belongs to the LS, so the LS knows which VM is on which chassis.

Loose coupling (referenced, no cascade): `MAC_Binding` (ARP learning, SBDB scope), `FDB` (runtime L2 learning), `IGMP_Group` (mcast snoop result).

Reveals: the actual OpenFlow rule installed on the chassis, where `metadata=0x4` is the `tunnel_key=4` of the Datapath_Binding for that Logical_Switch.

Tunnel side (Geneve outer): when the packet leaves the wire, `metadata` is encoded into `MFF_TUN_METADATA0` of the Geneve TLV; the destination chassis decapsulates, restores `metadata`, and continues the pipeline on the same datapath. This is the bridge from the NBDB row down to the packet wire format.

### Real-world incident

**Case, OpenStack DC 2024-Q2: a Logical_Switch ACL with a priority-0 drop blocked east-west production traffic**. Tenant `prod-frontend` had an LS with about 150 LSPs (web tier) and `prod-backend` had about 80 LSPs (DB tier), connected through a Logical_Router. The Neutron CMS synchronised "default-deny plus specific-allow" rules through the ML2-OVN driver: priority 1000 allow `tcp.dst=8080` from `pg_frontend` to `pg_backend`; priority 0 catch-all `(ip)` action drop.

**Symptom**: after a deployment that changed placement, all connections from frontend to backend on port 8080 began timing out. ICMP ping also failed. `curl` from a frontend VM to the backend VIP returned only `Connection timed out`. Application monitoring on the backend recorded zero incoming ICMP and TCP packets.

**Investigation timeline**:

1. **Discovery** (T+0): user reports a production blackhole; the on-call engineer runs `ovs-ofctl dump-flows br-int | grep metadata=0x<ls-frontend-key> | grep table=9 | grep "actions=resubmit" | wc -l` and sees only 12 rows (abnormal; expected more than 50 for 150 LSPs).
2. **Trace the packet**: `ovn-trace ls-prod-frontend 'inport=="vm-web-01" && ip4.dst==10.10.20.5 && tcp.dst==8080'` shows the packet dropped at stage `ls_in_acl_eval`, priority 0, match `(ip)`, action drop.
3. **Cross-check NBDB**: `ovn-nbctl --columns=acls list Logical_Switch ls-prod-frontend` shows one ACL row in the set. But `ovn-nbctl acl-list ls-prod-frontend` prints only one line, `from-lport 0 (ip) drop`: **the priority-1000 allow rule is missing**.
4. **Root cause**: a recent Neutron deployment reset the ACL set via a replace operation instead of incremental add. A race in rule synchronisation: the catch-all drop was created first, the allow rule should have been created after, but the sync API was retried, the drop was duplicated, and the allow rule was overwritten because of a priority-1000 collision with another ACL.
5. **Mitigation** (T+47 minutes): manual `ovn-nbctl acl-add ls-prod-frontend from-lport 1000 'inport == @pg_frontend && ip4.dst == $ip_backend && tcp.dst == 8080' allow-related`; verify with `ovn-trace` that the packet passes.
6. **Postmortem fix**: a Neutron driver patch to update the ACL set incrementally (compare-and-swap instead of replace); a pre-deploy `ovn-trace` test was added to CI to catch rule loss before production rollout.

**Lesson**: the `acls` set on Logical_Switch is a **strong ref** with cascade. Replacing instead of updating incrementally is very dangerous when the LS is serving production. Always run `ovn-trace` for at least three representative connection patterns after every ACL change.

### Common failures and diagnostic signals

| Failure mode | Signal | How to confirm the cause |
|--------------|---------|---------------------------|
| **A. CMS creates the LS but compile is slow** | northd log: "I-P engine recompute lasted 8000ms" | LSP count above 5000 or ACL count above 1000 in the same LS. Split the LS into multiple LSes connected through an LR. |
| **B. ACL allow rule has no effect** | ping fails, ovn-trace shows drop at `ls_in_acl_eval` priority 0 | Rule priority overlaps with a drop rule at the same priority, or match expression has a syntax error that northd silently rejects (check the northd log). |
| **C. Broadcast storm in a large LS** | hypervisor CPU above 80 percent, packets flooding br-int | `other_config:mcast_snoop` is not set to true; enable snooping and querier to reduce flood. |
| **D. Native DNS does not respond** | VM `nslookup` query times out | The `dns_records` set is empty, or the DHCP option does not point at the LS gateway IP. |
| **E. tunnel_key collision when importing from another cluster** | northd log: "duplicate tunnel_key"; flow conflict | OVN-IC (interconnect) is not configured with the correct transit switch; the `other_config:interconn-ts` column must be set to the right availability zone. |
| **F. LS does not cascade-delete on `ls-del`** | LSPs or ACLs left dangling | NBDB transaction failed to commit (a network blip with NBDB Raft); retry `ls-del`. |
| **G. Datapath_Binding exists but lflow is empty** | `ovn-sbctl lflow-list ls-X` returns nothing | northd is stuck or crashed mid-compile; restart the `ovn-northd` service. |

Standard diagnostic flow for "VMs in the same LS cannot reach each other": (1) does `ovn-nbctl ls-show` list the LSP, (2) does `ovn-sbctl find Port_Binding logical_port=...` show a chassis, (3) where does `ovn-trace` show the drop, (4) compare the ACL in NBDB against the flow at stage `ls_in_acl_eval` in SBDB.

### Comparison with other systems

| System | Equivalent concept | Main difference |
|----|----------------------|-----------------|
| **Traditional VLAN (802.1Q)** | L2 broadcast domain | VLAN is limited to 4096 IDs; Logical_Switch uses a 24-bit tunnel_key (about 16 million) over Geneve. VLAN is per-host configuration; Logical_Switch is declarative and cluster-wide. |
| **VMware NSX-T Segment** | Direct analog (same idea by the Nicira team) | NSX-T Segment has an "Edge"; OVN separates `Logical_Router`. Both use Geneve as the tunnel encap. |
| **Linux bridge over VXLAN** | Cross-host L2 | Bridge per host, controller manually configures peers; OVN automates this through the tunnel mesh and the MC_FLOOD multicast group. |
| **Cisco ACI EPG (Endpoint Group)** | Policy domain plus L2 | EPG attaches contracts between groups; OVN attaches ACLs directly to the LS or the Port_Group. EPG is closer to `Port_Group` than to Logical_Switch. |
| **AWS VPC Subnet** | Per-tenant L2/L3 segment | A VPC Subnet is L3 with its own CIDR; an OVN Logical_Switch is L2 only, and the gateway IP attaches to a Logical_Router_Port. |
| **Kubernetes Pod network (Flannel or Calico)** | Cross-node L2/L3 | Flannel VXLAN is similar but per-node; OVN abstraction is higher (LS plus LR plus stateful ACL). kube-ovn maps Kubernetes NetworkPolicy to OVN ACL. |

When migrating from another system to OVN: VLAN to Logical_Switch (1:1 mapping), NSX Segment to Logical_Switch (1:1, almost a direct port of the API), Cisco ACI EPG to Port_Group (attached to a Logical_Switch).

## 13.2.Y Logical_Router as an NBDB table: a deep dive into the distributed tenant L3 router

> This section dives into the **`Logical_Router` table itself** (a row in the OVN Northbound database), complementing the operator-oriented teaching in §13.2.2 to §13.2.4 and the LR pipeline in §13.16. Goal: the engineer fully understands the NBDB row structure, the 11 columns of the 22.03 schema, the compile lifecycle into a Datapath_Binding plus 20 LR_IN Logical_Flow stages plus 7 LR_OUT stages, how to distinguish a distributed router from a gateway-redirected router, and how to trace a routing or NAT incident back to a row in NBDB.

### Concept

`Logical_Router` is a **table that lives in the OVN Northbound database (NBDB)** and describes a **virtual L3 router** that the CMS (Neutron, kube-ovn, OpenStack) declares for a tenant. Each row represents one logical router: every `Logical_Router_Port` (LRP) belonging to that row sits in the same routing domain, shares the same FIB (forwarding information base) derived from `static_routes` plus `policies`, the same set of NAT rules, the same set of Load_Balancer rows, and the same runtime `options`. Type: **OVSDB table**, root table (`isRoot=true` in `ovn-nb.ovsschema`), persistent (replicated through the NBDB Raft cluster), declarative input for ovn-northd. Unlike a traditional router (a physical device or a gateway VM), Logical_Router is **distributed by default**: the LR_IN and LR_OUT pipeline runs locally on every chassis that hosts a VM belonging to the LR, with no central hop. When external traffic (north-south with centralized NAT) is needed, a specific LRP is pinned to a chassis through `options:chassis=<uuid>`, forming a **gateway router** with a chassisredirect port; north-south traffic then traverses the pinned chassis while east-west traffic remains distributed.

### History and background

The `Logical_Router` table appeared with OVN 2.4 (2015) at the same time as `Logical_Switch`, in the refactor by Russell Bryant (Red Hat) and Justin Pettit (VMware/Nicira) that moved the Nicira NSX OpenStack plugin architecture into OVN. Before that, the Neutron L3 agent ran each tenant router inside a Linux network namespace on a centralized network node (HA Active/Standby through keepalived, or DVR semi-distributed); scale was limited because every L3 packet flowed through a single node, and route or NAT configuration had to be pushed over RPC. OVN solves this by compiling each Logical_Router into a 27-stage pipeline installed on every chassis: east-west L3 traffic never leaves the source chassis when the destination shares the chassis; only external traffic crosses the gateway chassis. OVN 21.03 added `policies` (Logical_Router_Policy) for policy-based routing. OVN 21.06 overhauled the distributed gateway router architecture: it formalized the `chassisredirect` Port_Binding type and `HA_Chassis_Group` for multi-chassis failover. OVN 22.03 (the curriculum baseline) stabilizes the 11-column schema (verified against `ovn-nb.ovsschema` on branch-22.03). OVN 22.06 extended ECMP multi-nexthop. OVN 24.03 added `options:dynamic_neigh_routers` and improved BFD on LRP.

### Position in the architecture

The table sits at the **NBDB layer**, peer to `Logical_Switch` at the input layer. Scope: **per cluster**. Each Logical_Router row produces **exactly one Datapath_Binding row** in SBDB (`ovn-sbctl find Datapath_Binding external_ids:logical-router=<lr_uuid>`) plus **on average 100 to 500 Logical_Flow rows** (depending on the number of LRPs, static_routes, policies, NAT, and LB rows) distributed across 20 ingress stages and 7 egress stages. Impact level: changing a Logical_Router row (for example, adding a `static_routes` entry or a `nat` rule) causes ovn-northd to recompute that datapath in under 200 ms (incremental compile since OVN 21.03); ovn-controller on each chassis then reinstalls OpenFlow rules within one second through the I-P engine. The Datapath_Binding tunnel_key for an LR shares its namespace with LS (1 to 16,777,215, unique in the cluster); there is no collision because the same allocator is used.

### Role

Logical_Router carries **five roles** in the OVN architecture:

1. **Identifies the L3 routing domain**: the `name` column (string, not unique) is a human-readable identifier; `_uuid` is the only unique key. Every LRP in the row shares the same FIB.
2. **Container for LRPs**: the `ports` column (set of strong references to `Logical_Router_Port`) cascade-deletes, so removing the LR removes every LRP. Each LRP has a MAC plus one or more CIDRs; an LRP peers with an LSP of type=router on a Logical_Switch.
3. **Anchor for L3 policy**: the `static_routes` column (set of strong references to `Logical_Router_Static_Route`) holds static routes (ECMP when several routes share a prefix), `policies` (set of strong references to `Logical_Router_Policy`) holds policy-based routing (priority plus match plus action), `nat` (set of strong references to `NAT`) holds SNAT, DNAT, and dnat_and_snat rules, `load_balancer` (set of weak references) and `load_balancer_group` hold north-south LB, and `copp` holds control plane policing.
4. **Carries runtime parameters to northd**: the `options` column (map of string to string) holds `chassis=<uuid>` (pins the LR as a gateway on a specific chassis), `dynamic_neigh_routers="true"` (uses dynamic neighbor learning instead of broadcast ARP), `mac_binding_age_threshold`, `lb_force_snat_ip`, and `always_learn_from_arp_request`. Northd reads this map to decide whether to compile a distributed or a redirect pipeline.
5. **Reports metadata to the CMS**: the `external_ids` column (map) carries `neutron:router_name`, `kube-ovn:vpc`, or similar entries; the CMS uses them to trace an OVN row back to the original CMS object.

### Why it exists

Before OVN 2.4, the Neutron L3 agent architecture (Folsom 2012 to Liberty 2015) ran each tenant router in a Linux network namespace on a centralized network node, with HA through VRRP and keepalived (Active/Standby). The problems were: (a) **single-point bottleneck**: every east-west L3 packet also traversed the network node, easily saturating the NIC; (b) **slow state sync on failover**: when the Active node died, the Standby took 3 to 5 seconds to take over the virtual IP, dropping every connection; (c) **DVR (Distributed Virtual Routing) complexity**: Juno 2014 added DVR but still required a centralized snat-namespace for egress, and the configuration was cumbersome. OVN solves this by **pushing state out to OVSDB Raft and letting northd compile a distributed pipeline by default**: the LR_IN and LR_OUT pipeline is installed on every chassis, east-west traffic does not pass through a central hop, and only external NAT goes through a pinned chassisredirect port. Logical_Router is the **only tenant-facing abstraction** that the CMS sees: the CMS does not need to know about Datapath_Binding, Logical_Flow, or chassisredirect, it just creates Logical_Router plus LRPs plus nat plus static_routes. This model scales to thousands of routers per cluster without increasing the L3 hop count (compare: the Neutron L3 agent caps at roughly 200 to 500 routers per network node before saturating).

### Problems it solves

The table solves **five distinct technical problems**: (1) **distributed L3 forwarding by default**: the 20-stage LR_IN pipeline is installed on every chassis, so an east-west L3 packet never leaves a chassis when the destination is on the same chassis or reachable through a direct tunnel within the cluster; (2) **centralized NAT when needed**: attaching `options:chassis=<uuid>` to a gateway LRP makes northd emit a chassisredirect port; SNAT and DNAT rules apply only on that chassis to keep conntrack state consistent; (3) **consistent cascade lifecycle**: deleting an LR row deletes every LRP, static_route, policy, and nat rule belonging to it through strong references, so the CMS does not need a pseudo-transaction; (4) **natural ECMP multi-nexthop**: adding two static_routes with the same prefix and different nexthops makes northd compile two flows into the `lr_in_ip_routing_ecmp` stage at table ID 12 (see §13.16) with a 5-tuple hash selecting the path; (5) **declarative policy-based routing**: adding a Logical_Router_Policy row with priority plus match (for example, `ip4.src == 10.0.1.0/24 && ip4.dst == 8.8.8.8`) plus action (`reroute 192.168.1.1`) compiles into `lr_in_policy` at table ID 13.

### Importance

This is a **top-tier foundational NBDB table**, peer to `Logical_Switch`. Reasons: 100 percent of OVN tenant L3 workflows begin with `ovn-nbctl lr-add`; 100 percent of advanced OVN features at L3 scope (stateful NAT, north-south LB, static routes plus ECMP, policy-based routing, BFD, gateway HA) add columns or option fields to Logical_Router instead of creating new tables. An engineer who does not understand Logical_Router cannot create, debug, or scale any tenant L3 in OVN. Out of the more than 320 keywords in REF, Logical_Router belongs to the cluster of 15 OVN foundational keywords (alongside `ovn-northd`, `ovn-controller`, NBDB, SBDB, Logical_Flow, Datapath_Binding, Logical_Switch, Port_Binding, ACL, Load_Balancer, NAT, Geneve TLV, ovn-trace, ovn-detrace).

### How it works (the row compile cycle)

A Logical_Router row passes through **four phases** in its lifecycle:

1. **Declared by the CMS**: the CMS calls `ovn-nbctl lr-add lr-tenant1`, and the NBDB transaction creates a new row with `name="lr-tenant1"`, `enabled=true` (default), and every other set initialized to `[]`. The transaction is atomic; the row is visible in under 50 ms through NBDB Raft replication.
2. **Compile into Datapath_Binding**: ovn-northd detects the new row through an I-P engine input change. The function constructs a `struct ovn_datapath` for the LR row (sharing the allocator with LS) and allocates a `tunnel_key`. Then `build_datapaths` writes to SBDB: `INSERT INTO Datapath_Binding (tunnel_key=<allocated>, external_ids={'name': 'lr-tenant1', 'logical-router': '<lr_uuid>'})`. Result: one Datapath_Binding row corresponding 1:1 with the Logical_Router row.
3. **Compile into Logical_Flow rows**: northd runs the chain of `build_*_for_lrouter` functions to emit Logical_Flow rows for each of the 20 LR_IN stages and 7 LR_OUT stages. Every LRP, static_route, policy, NAT rule, and LB attached to the LR spawns its own flows. Total: 100 to 1000 Logical_Flow rows per typical LR (multiplicative in number of LRPs, routes, NAT rules, and LBs).
4. **Install flows on the chassis**: SBDB Raft replicates Logical_Flow rows; ovn-controller on every chassis subscribes; each controller runs `lflow_run` to translate Logical_Flow expressions into concrete OpenFlow rules and installs them on `br-int` through `ofctrl_put`. LR_IN stages 0 to 19 install into OF tables 8 to 27; LR_OUT stages 0 to 6 install into OF tables 40 to 46 (see §13.16 for details). If the LR has a gateway port (`options:chassis=<uuid>`), only the pinned chassis adds flows for chassisredirect; other chassis hold only east-west distributed flows.

Invariant: at any moment, a Logical_Router row in NBDB corresponds to exactly **one** Datapath_Binding row in SBDB; deleting the LR through `lr-del` cascade-deletes its LRPs, static_routes, policies, nat rows, the Datapath_Binding, and the Logical_Flow rows in the same compile round.

### Classification of columns

The 11 columns of Logical_Router (verified against `ovn-nb.ovsschema` branch-22.03 [GitHub raw](https://raw.githubusercontent.com/ovn-org/ovn/branch-22.03/ovn-nb.ovsschema)) fall into **five groups** by semantic role:

| Group | Column | Type | Required | Notes |
|-------|--------|------|----------|-------|
| Identity | `name` | string | Yes (default "") | CMS-friendly, not unique |
| Identity | `enabled` | boolean | No (default true) | false drops traffic on every LRP of the LR |
| Container | `ports` | set strong ref to Logical_Router_Port | No (default []) | Cascade delete of LRPs |
| Routing | `static_routes` | set strong ref to Logical_Router_Static_Route | No | Static routes plus ECMP |
| Routing | `policies` | set strong ref to Logical_Router_Policy | No (21.03+) | Policy-based routing |
| LR-attached policy | `nat` | set strong ref to NAT | No | SNAT, DNAT, dnat_and_snat |
| LR-attached policy | `load_balancer` | set weak ref to Load_Balancer | No | Legacy north-south LB |
| LR-attached policy | `load_balancer_group` | set ref to Load_Balancer_Group | No (24.03+) | Modern group LB |
| LR-attached policy | `copp` | weak ref to Copp (max 1) | No (22.03+) | Control plane meter |
| Service parameters | `options` | map string to string | No | chassis=<uuid>, dynamic_neigh_routers, mac_binding_age_threshold |
| CMS metadata | `external_ids` | map string to string | No | CMS-tracked ID |

The `enabled` column is a clear difference from `Logical_Switch` (LS has no such column). Setting `enabled=false` through `ovn-nbctl set Logical_Router lr-X enabled=false` makes northd compile drop flows at the `lr_in_admission` stage for every packet entering an LRP of that LR; it is useful for maintenance without deleting the LR. The only lookup key is `_uuid`.

### How an operator masters this

Four foundational skills the engineer must have with Logical_Router:

1. **Read a full row**: `sudo ovn-nbctl --columns=name,enabled,ports,static_routes,policies,nat,load_balancer,options list Logical_Router <lr_uuid_or_name>` shows the 8 main columns plus the size of each set. Alternatively, `sudo ovn-nbctl lr-list` gives an overview, and `sudo ovn-nbctl lr-route-list lr-X` plus `ovn-nbctl lr-nat-list lr-X` print route and NAT details.
2. **Verify the compile down to SBDB**: `sudo ovn-sbctl find Datapath_Binding external_ids:logical-router=<lr_uuid>` confirms the corresponding Datapath_Binding exists. If not, northd has not finished compiling, or there is an error in ovn-northd.log.
3. **Count pipeline output by LR_IN and LR_OUT stage**: `sudo ovn-sbctl lflow-list <lr_name> | grep -c lr_in_` counts ingress flows; `grep -c lr_out_` counts egress flows. An LR with no LRPs has only about 30 stub flows; an LR with 3 LRPs plus 5 static_routes plus 2 NAT rules has roughly 150 to 250 flows; an LR with 20 LRPs, 100 routes, and 50 NAT rules has more than 2000 flows.
4. **Decide which skill to use when**: when the CMS reports "VMs reach each other at L2 but cannot reach across subnets", start with `lr-list` plus `lr-route-list` (skill 1) to confirm the routes are correct; when the CMS reports "north-south NAT lost connectivity after a failover", check `options:chassis` and the chassisredirect Port_Binding (see §13.5.1b plus §13.11); when northd seems to hang for a specific LR, check the Datapath_Binding (skill 2).

### Usage workflow (standard)

The workflow to provision a tenant L3 with 3 subnets plus external NAT through a Logical_Router has seven steps, each with a verification command:

1. **The CMS creates the Logical_Router in NBDB**: `ovn-nbctl lr-add lr-tenant1`. Verify: `ovn-nbctl lr-list | grep lr-tenant1` shows the UUID and name.
2. **Create an LRP for each subnet**: `ovn-nbctl lrp-add lr-tenant1 lrp-web aa:bb:cc:00:01:fe 10.0.1.254/24`; similarly for `lrp-app` and `lrp-db`. Verify: `ovn-nbctl --columns=ports list Logical_Router lr-tenant1` shows 3 references.
3. **Peer the LRP with an LSP of type=router on the LS**: on each corresponding LS, create an LSP of type=router with options:router-port=lrp-X (see §13.2.3). Verify: `ovn-nbctl ls-show ls-web` shows the type=router port.
4. **Add static_routes if needed**: `ovn-nbctl lr-route-add lr-tenant1 0.0.0.0/0 192.168.1.1` (default route through the external gateway). Verify: `ovn-nbctl lr-route-list lr-tenant1` shows the entry.
5. **Add NAT rules if north-south is needed**: `ovn-nbctl lr-nat-add lr-tenant1 snat 192.168.1.10 10.0.0.0/16` (SNAT pool from internal to external IP); or `ovn-nbctl lr-nat-add lr-tenant1 dnat_and_snat 192.168.1.20 10.0.1.5` (one-to-one floating IP). Verify: `ovn-nbctl lr-nat-list lr-tenant1` shows the entry.
6. **Pin the gateway if centralized NAT is needed**: `ovn-nbctl lrp-set-gateway-chassis lrp-web hv-gw1 50` (chassis priority 50). Verify: `ovn-sbctl find Port_Binding type=chassisredirect` shows `cr-lrp-web` with the chassis reference.
7. **Wait for northd to compile and verify**: typically under 200 ms. Verify: `ovn-sbctl lflow-list lr-tenant1 | grep lr_in_dnat` shows the NAT flow; `ovn-trace lr-tenant1 'inport=="lrp-web" && ip4.src==10.0.1.5 && ip4.dst==8.8.8.8'` shows the packet passing through to `lr_out_snat`.

Production best practices: **always use `dynamic_neigh_routers="true"` for LRs with more than 100 LRPs** to reduce broadcast ARP; **always use HA_Chassis_Group instead of pinning a single chassis** for production gateways (see §13.6 and §13.11). Anti-pattern: one Logical_Router for the entire DC with thousands of LRPs causes northd to compile slowly (more than 30 seconds) because of LR_IN pipeline explosion across every LRP times every stage; split into multiple LRs per tenant connected through a transit LR.

### When something goes wrong (reverse symptom map)

Eight common symptoms point back to Logical_Router:

| Symptom | Logical_Router columns to check | Diagnostic flow |
|---------|---------------------------------|-----------------|
| "VMs in the same tenant cannot reach each other across subnets" | `ports`, `enabled` | `ovn-nbctl --columns=enabled,ports list Logical_Router lr-X`; if enabled=false, set it to true |
| "Default route does not work" | `static_routes` | `ovn-nbctl lr-route-list lr-X` to check whether 0.0.0.0/0 is present; `ovn-trace` to see whether the packet drops at `lr_in_ip_routing` |
| "ECMP does not load-balance" | `static_routes` (multi-nexthop) | `ovn-sbctl lflow-list lr-X | grep lr_in_ip_routing_ecmp` to check whether ECMP flows exist |
| "Policy-based routing is skipped" | `policies` | `ovn-nbctl lr-policy-list lr-X`; verify the priority and match expression are correct |
| "External SNAT does not apply" | `nat`, `options:chassis` | `ovn-nbctl lr-nat-list lr-X`; if no gateway is pinned, SNAT applies only on the chassis that hosts the source VM (distributed SNAT is an anti-pattern) |
| "DNAT floating IP failover is slow" | `nat`, LRP `gateway_chassis` or `ha_chassis_group` | `ovn-sbctl find Port_Binding type=chassisredirect`; check HA_Chassis_Group priority |
| "L3 traffic black-holes after VM migration" | (LSP `external_ids`) plus LR `static_routes` | Conntrack state is lost when a VM migrates across chassis; check `ovn-trace` in both forward and reverse direction |
| "northd compile is slow for a large LR" | Number of LRPs, routes, NAT rules | Split the LR into multiple LRs connected by a transit LR |

### Tightly related to

Seven components couple tightly to Logical_Router (changing any of them forces a re-examination of Logical_Router):

- **`Logical_Router_Port`** (LRP): contained in the `ports` column as a strong reference. Each LRP belongs to exactly one LR; deleting the LR cascade-deletes the LRP. Each LRP has a MAC, CIDRs, and optionally a `gateway_chassis` reference.
- **`Logical_Router_Static_Route`**: contained in the `static_routes` column as a strong reference. Cascade-deleted. ECMP is implied when several routes share `ip_prefix` but have different `nexthop` values.
- **`Logical_Router_Policy`**: contained in the `policies` column as a strong reference (21.03+). Cascade-deleted. Priority plus match plus action.
- **`NAT`**: contained in the `nat` column as a strong reference. Type: `snat`, `dnat`, `dnat_and_snat`. Cascade-deleted when the LR is deleted.
- **`Logical_Switch_Port` type=router**: indirect; each LRP peers with an LSP of type=router on a Logical_Switch. Bridges L3 to L2.
- **`Datapath_Binding` (SBDB)**: 1:1 mapping through `external_ids:logical-router=<lr_uuid>`. The first output of northd compile.
- **`Port_Binding` type=chassisredirect** (SBDB): appears when an LRP has `gateway_chassis` or `ha_chassis_group`; pins north-south traffic to a specific chassis.

Loose coupling (referenced but not cascade-deleted): `MAC_Binding` (ARP learning from external next-hops), `BFD` row (per-LRP gateway monitoring), `HA_Chassis_Group` (gateway failover).

### Real-world incident

**Case OpenStack DC 2024-Q3: Logical_Router gateway redirect chassis CPU spikes to 100 percent during failover.** Cluster: OVN OpenStack with 16 hypervisors plus 2 gateway chassis (hv-gw1 active, hv-gw2 standby through HA_Chassis_Group). Each tenant has one Logical_Router with 2 to 5 LRPs and roughly 10 to 30 NAT rules (floating IP plus SNAT pool). Cluster total: about 80 LRs, about 400 LRPs, about 600 NAT rules.

**Symptom**: hv-gw1 went down at 14:32 because of a NIC hardware fault. HA_Chassis_Group failed over to hv-gw2 within 2 seconds (BFD detection plus chassisredirect Port_Binding update). From 14:32:05, hv-gw2 CPU jumped from 25 percent to 100 percent (all 8 cores saturated); ovn-controller logged "I-P engine recompute lasted 12000ms" repeatedly; every north-south traffic flow timed out for about 30 seconds.

**Investigation timeline**:

1. **Detection** (T+0): Prometheus alert "node CPU > 95 percent for 60s" fired on hv-gw2; the on-call engineer SSH'd in and `top` showed `ovn-controller` consuming 6 cores fully.
2. **Trace the CPU root cause**: `ovs-appctl -t ovn-controller debug/status` showed an `lflow_run` queue backlog of about 5,000 entries; `ovn-appctl -t ovn-controller inc-engine/show-stats` showed "physical_flow_changes" spiking every 5 seconds.
3. **Cross-reference SBDB**: `ovn-sbctl --columns=chassis list Port_Binding type=chassisredirect | wc -l` showed 80 chassisredirect rows all referring to hv-gw2 (correct, because of failover); but `ovn-sbctl lflow-list | wc -l` for the cluster grew from about 50,000 to about 80,000 after the failover.
4. **Root cause**: stage `lr_in_gw_redirect` (T18), when the gateway is pinned to a single chassis, emits redirect flows for every LRP toward the chassisredirect; with 400 LRPs times about 10 NAT rules times stage interactions, the total ingress flow count on hv-gw2 exceeded the OF table size limit configured for `br-int`. On each recompute round, ovn-controller had to retry the full batch flow install because the OFPT_FLOW_MOD bundle hit the timeout (5 seconds). This was compounded by hv-gw2 (a lower CPU tier than hv-gw1) lacking the throughput.
5. **Mitigation** (T+25 minutes): temporarily disabled `dynamic_neigh_routers` on the 50 percent least critical LRs through `ovn-nbctl set Logical_Router lr-X options:dynamic_neigh_routers=false`, reducing flows for the `lr_in_arp_resolve` stage by about 30 percent. CPU on hv-gw2 dropped to about 70 percent within 5 minutes.
6. **Postmortem fix** (T+3 days): split the 80 LRs into 4 gateway groups, each pinned to a different HA_Chassis_Group (4 gateway pairs instead of 1); increased the configuration `ovs-vsctl set Open_vSwitch . other_config:max-idle=...` to reduce flow-cache pressure; main lesson: gateway concentration is an anti-pattern when LR count exceeds 50.

**Lesson learned**: gateway pinning (`options:chassis` or HA_Chassis_Group) creates a single concentrated load point; Logical_Router is distributed by default and scales well east-west, but north-south redirect does not scale linearly with the number of LRs. Always benchmark the flow count on a candidate gateway before a production rollout.

Cross-reference §17.0 (forensic case template), §13.11 (distributed gateway router), and §13.18 (`REGBIT_NAT_REDIRECT` details).

```bash
# Step 7: trace the LR pipeline for cross-chassis traffic
[central]# ovn-trace lr-tenant 'inport=="lrp-web" && eth.src==aa:bb:cc:00:00:01 && eth.dst==aa:bb:cc:00:01:fe && ip4.src==10.10.1.10 && ip4.dst==10.10.2.11 && ip.ttl==64'
# Expected: pipeline passes through lr_in_admission, lr_in_lookup_neighbor, lr_in_ip_input, lr_in_ip_routing (matches 10.10.2.0/24 with outport=lrp-db), lr_in_arp_resolve, lr_in_arp_request, then egress lr_out_delivery, then output to the ls-db pipeline.

# Step 8: dump flows on hv1 with the metadata matching the LR tunnel_key
[hv1]# tk=$(ovn-sbctl --bare --columns=tunnel_key find Datapath_Binding external_ids:logical-router=$(ovn-nbctl --bare --columns=_uuid list Logical_Router lr-tenant))
[hv1]# ovs-ofctl -O OpenFlow15 dump-flows br-int | grep "metadata=0x$(printf %x $tk)" | wc -l
# Expected: more than 80 flows spread across tables 8 to 27 (LR_IN) and 40 to 46 (LR_OUT).

# Cleanup
[central]# ovn-nbctl lr-del lr-tenant
[central]# ovn-nbctl ls-del ls-web; ovn-nbctl ls-del ls-db
```

**Falsification check**: set `ovn-nbctl set Logical_Router lr-tenant enabled=false`; the cross-LS ping should fail (drop at `lr_in_admission`); set `enabled=true` again and the ping should pass. This confirms the `enabled` behavior.

**Deliverable**: the log of the 8 steps above plus `ovn-trace` output showing the packet passing through every relevant LR_IN stage (only the stages with matching flows) up to `lr_out_delivery` action `output;`.

### Common failures and diagnostic signals

| Failure mode | Signal | How to identify the cause |
|--------------|--------|---------------------------|
| **A. Cross-subnet does not work after lr-add** | Ping to the LR gateway IP fails | The LRP is missing the type=router peer LSP; check `ovn-nbctl ls-show ls-X` for a type=router port |
| **B. Default route does not work** | `traceroute 8.8.8.8` reaches the internal gateway and stops | `ovn-nbctl lr-route-list lr-X` is missing 0.0.0.0/0; add `lr-route-add lr-X 0.0.0.0/0 <next-hop>` |
| **C. SNAT does not apply for external traffic** | tcpdump on the external side shows the internal source IP instead of the SNAT IP | The NAT rule exists but `options:chassis` does not pin the gateway, so distributed SNAT is not deterministic |
| **D. Floating IP DNAT loses connectivity after VM migration** | Ping to the floating IP times out after live migration | Conntrack zone is not synced between the old and new chassis; check `ovn-sbctl find Port_Binding type=chassisredirect` |
| **E. ECMP does not load-balance evenly** | `ovs-ofctl dump-flows | grep ecmp` shows only 1 active path | The `lr_in_ip_routing_ecmp` stage at table 12 needs multiple routes with the same prefix; verify `lr-route-list` has at least 2 routes with the same destination |
| **F. Policy-based routing does not trigger** | The packet follows the normal route instead of the rerouted one | `ovn-nbctl lr-policy-list lr-X`, check the priority and match; use ovn-trace to verify whether `lr_in_policy` is hit |
| **G. northd compile is slow for an LR with more than 100 LRPs** | `ovn-northd.log` shows "took 30000ms" | Split the LR by tenant with a transit LR; enable `dynamic_neigh_routers=true` |
| **H. Gateway chassis CPU at 100 percent after failover** | Hypervisor CPU spike, `lflow_run` queue backlog | Too many LRs pinned to the same gateway HA group; split into multiple HA groups |

Standard diagnostic flow for the symptom "VMs in a tenant cannot reach each other across subnets": (1) `ovn-nbctl lr-list` to confirm the LR exists with `enabled=true`; (2) `lr-route-list` to check whether routes (and the default route) are present; (3) `ovn-trace` from source to destination to see at which stage the packet drops; (4) cross-reference NBDB rules with the corresponding Logical_Flow stage in SBDB.

### Comparison with other systems

| System | Equivalent concept | Main difference |
|--------|-------------------|-----------------|
| **Cisco IOS VRF** | An independent routing instance | A VRF is bound to a physical router; an OVN Logical_Router is distributed across every chassis. VRF uses `route-target` for leaking; OVN uses peer LRPs plus a transit switch. |
| **Linux network namespace plus iptables/iproute2** | Namespace as a router instance | Per-host; OVN LR is cluster-wide and compiled into Logical_Flow rows installed on every chassis. iptables NAT versus the declarative OVN NAT rule. |
| **Neutron L3 agent (legacy plus DVR)** | Tenant router | The L3 agent is centralized, or DVR is semi-distributed; OVN LR is distributed by default with NAT centralized only through chassisredirect. OVN scales better thanks to compile-time pipelines. |
| **VMware NSX-T T0/T1 logical router** | Direct analog (the same Nicira team) | NSX-T splits T0 (provider) from T1 (tenant); OVN has a single Logical_Router level (you can chain through a transit LS to simulate T0/T1). NSX-T uses a centralized edge cluster for NAT; OVN is distributed with optional gateway. |
| **AWS VPC route table plus NAT Gateway** | Routing plus NAT for a VPC | AWS is a managed service that hides the implementation; the OVN Logical_Router row is exposed declaratively. AWS NAT Gateway is centralized; the OVN gateway is optional. |
| **Kubernetes OVN-Kubernetes** | Per-cluster join LR | The scope is the same (kube-ovn still uses an OVN Logical_Router); the distinction is the CMS layer (kube-ovn versus Neutron). |

When migrating from another system to OVN: Cisco VRF to Logical_Router (1:1, route-target maps to peer LRP), Neutron L3 agent to Logical_Router (1:1, distributed default), NSX-T T0/T1 to two Logical_Router rows (T0 transit plus T1 tenant) connected through a transit LS, AWS VPC route table to Logical_Router with copied static_routes.

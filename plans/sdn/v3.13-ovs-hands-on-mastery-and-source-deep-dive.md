## Mission

Take the owner from "knows OVS theory" to "operates OVS in production with reflexive command muscle, knows the source code architecture cold, can read and modify OVS source files, can run the OVS test suite, and can debug a packet-loss incident from a `tcpdump` line down to the C function that dropped it." All practice output materializes as new and updated `sdn-onboard/*.md` files.

The plan has three parallel tracks. Track A (Hands-on lab) runs sprint by sprint on `lab-openvswitch`, every command and every output captured verbatim per CLAUDE.md Rule 18. Track B (Source-code mastery) walks the OVS source tree (about 700 C files across `lib/`, `ofproto/`, `vswitchd/`, `ovsdb/`, `datapath/`, `utilities/`), documents architecture, build process, test infrastructure, and the role of every major component. Track E (Enrichment) sits in front of every Track A and Track B sprint as a research gate; no R-sprint or S-sprint may begin until the corresponding E-N dossier passes the per-section minimums.

The audience is a CCNA, Cisco, or Juniper-trained network engineer encountering Open vSwitch, OpenFlow, and OVN for the first time. The full audience constraint and writing rules are codified in CLAUDE.md Rule 19 (added 2026-04-30) and restated below in [Audience and pedagogy](#audience-and-pedagogy-rule-19). The owner has permanently banned DPDK, BPF, XDP, BGP, and Kubernetes from active planning per the consolidated directive of 2026-04-25 (CLAUDE.md North Star); v3.13 honours that ban without exception.

## Audience and pedagogy (Rule 19)

### The reader

The assumed reader is CCNA-trained or Juniper-trained at minimum, has 2 to 10 years of experience operating real switches and routers, knows Cisco IOS or Junos CLI in muscle memory, has built campus VLANs and trunks and STP topologies, and treats Linux as familiar but not as a first-class networking platform. The reader does not yet have a mental model for "a switch implemented as a userspace daemon", "a flow table programmed by a controller", or "a virtual switch managed by a database". Those are exactly the conceptual jumps the curriculum is responsible for teaching.

### The six writing rules

CLAUDE.md Rule 19 codifies six rules every `sdn-onboard/*.md` file must satisfy. In compressed form:

1. Every concept that has a traditional-network analogue must be introduced through that analogue (Cisco/Juniper vocabulary as the bridge into OVS terminology).
2. Every non-obvious data path, state machine, or internal switch behaviour must include an ASCII model diagram inside the curriculum prose.
3. Density is not depth. Prose-to-fenced-block ratio at least 2:1; every fenced block needs a prose lead-in (Predict-Observe-Explain step 1) and a prose follow-up (POE step 3).
4. Misconceptions a Cisco/Juniper-trained engineer is likely to carry must be explicitly named and refuted (Misconception/Reality blocks).
5. Command output is evidence, not pedagogy. The pedagogical content lives in the prose surrounding the fenced block.
6. Full language compliance: CEFR B2-C1 plain English per Rule 17, no em-dash, no AI sentence patterns, no plan-internal terms in body prose per Rule 16.

### Scope of application

R0 and R1 lab files were captured before Rule 19 was authored and are accepted as is per a not-retroactive carve-out. Every lab `.md` from R2.0 onward must satisfy Rule 19 in full. The owner reads each new file and rejects on Rule-19 grounds if needed; rejection requires a full rewrite, not a patch. There is no automated check for Rule 19 because the rule is qualitative; the gating mechanism is the owner's review.

### R6, connection tracking lab

Goal: stateful firewall and NAT through the conntrack action. `ct(zone=1, commit)`, `ct(nat(src=...))`, `ct(nat(dst=...))`, `conntrack -L`, conntrack zones, ALG-less SIP, reject vs drop semantics. Source anchors: `lib/conntrack.c`, `datapath/conntrack.c`, `lib/conntrack-tcp.c`.

### R7, QoS, mirroring, sFlow

Goal: HTB queues, HFSC, OF meter rate-limit, port mirror to internal port, sFlow agent to a `softflowd` collector running locally. Source anchors: `vswitchd/bridge.c:bridge_configure_qos()`, `lib/netdev-linux.c:netdev_linux_set_qos()`, `ofproto/ofproto-dpif-mirror.c`, `ofproto/ofproto-dpif-sflow.c`.

### R8, OVSDB internals lab (BLOCKED on lab-openvswitch-3)

Goal: a true distributed OVSDB Raft cluster across three real hosts. Cluster bootstrap with `ovsdb-tool create-cluster`, then `join-cluster` from nodes 2 and 3. Sub-tasks: single-node baseline, convert to Raft cluster, inspect cluster state, force a leader change with `cluster/leave`, network partition (VMware "Disconnect" feature), snapshot and compaction, schema upgrade across cluster, manual JSON-RPC transactions, monitor protocol generations (v1/v2/v3), failure-mode injection (kill leader, clock skew, disk-full, log corruption).

Source anchors: `ovsdb/ovsdb-tool.c`, `ovsdb/raft.c`, `ovsdb/raft-rpc.c`, `ovsdb/jsonrpc-server.c`, `ovsdb/file.c`.

### R9, debug and troubleshoot doctrine

Goal: a 12-step decision tree, each step a real failure injected and resolved on the lab host. Failures: ARP fails between two namespaces, ICMP black hole (MTU mismatch), asymmetric flow, megaflow explosion, controller disconnect (TCP RST), OVSDB lock, kernel module mismatch, `ovs-vswitchd` not starting (missing perm), datapath flow rejection, conntrack zone leak, tunnel decap failure (VNI mismatch), sFlow flood. Each scenario: symptom, command, expected output, root cause, fix, source-code reference.

### R10, forensic and production readiness

### R11, capstone (BLOCKED on lab-openvswitch-4)

Goal: a 3-rack leaf-spine fabric with 2 tenants (Red and Blue) each spanning two VMs across three racks. Each tenant must be isolated from the other. East-west traffic within a tenant is L2 (overlay). North-south traffic for each tenant exits via a tenant-specific gateway.

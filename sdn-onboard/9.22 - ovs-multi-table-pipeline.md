## 9.22.1 Why OpenFlow 1.0 lasted only 14 months, opening context

In December 2009, the Open Networking Foundation (ONF) published OpenFlow Switch Specification 1.0.0, the first specification. In February 2011, only 14 months later, version 1.1.0 arrived with a radical architectural change: the single flow table was replaced by a **multi-table pipeline**. In the history of the telecommunications industry, it is rare for a standard to change a core abstraction this fast. TCP/IP has kept its structure since 1981, and VLAN 802.1Q has kept its frame format since 1998.

The reason: when engineers tried to deploy OpenFlow 1.0 on Broadcom Trident silicon (the commercial switch chip used in Arista, Dell, and Cisco Nexus 9000), they found there was no way to map a single OpenFlow flow table onto the internal architecture. The Trident ASIC has **seven logical tables** (ingress parser, ingress ACL, L3 forwarding, L2 forwarding, egress ACL, queueing, egress parser). To represent the real hardware in OpenFlow 1.0, the engineer had to simulate all seven tables through one virtual table. That caused a table explosion (the Cartesian product of every match field, which could rise to several million entries for a simple configuration).

The ONF TM Working Group published a report in July 2010 that pointed out the problem, and in February 2011 OpenFlow 1.1 introduced **N flow tables numbered 0 to N-1** along with two new instructions: `Goto-Table` and `Write-Metadata`. The lasting lesson is that a good abstraction must keep the shape of real hardware, not cut itself off from it. Multi-table is the compromise between the cleanliness of SDN and the hardware-faithfulness of silicon.

### OVS got there first

Open vSwitch had a similar mechanism **before** OpenFlow 1.1 appeared. The action `resubmit` existed in OVS 1.0 (June 2011, the same time as OpenFlow 1.1) and was inherited from the NOX project (Stanford, 2008). The OVS team understood the limits of a single table several years before the ONF and already had its own solution. Later, OVS supported both: `goto_table` per the OpenFlow 1.1+ standard, and `resubmit` per the NOX and OVS extension. Section 9.22.3 analyses the difference.

## 9.22.2 The multi-table pipeline architecture, from the OpenFlow 1.3.5 specification

OpenFlow Switch Specification 1.3.5 §5.1 *"Pipeline Processing"* defines:

*"OpenFlow switches are required to have a minimum of one flow table, and can optionally have more. An OpenFlow switch with only a single flow table is valid, in this case pipeline processing is greatly simplified. The flow tables of an OpenFlow switch are sequentially numbered, starting at 0. Pipeline processing always starts at the first flow table: the packet is first matched against flow entries of flow table 0."*

A multi-table pipeline must follow four hard rules:

**Rule 1, always start at table 0.** Every packet that enters the pipeline must match an entry in table 0 first. There is no way to push a packet directly into table 5. This design simplifies debugging: when a packet "disappears", you start tracing from table 0.

**Rule 2, only move forward, never backward.** The specification says directly: *"A flow entry can only direct a packet to a flow table number which is greater than its own flow table number"*. Table 3 can have `goto_table=5` but cannot have `goto_table=2`. This guarantees **termination**, so the pipeline cannot loop forever.

**Rule 3, the action set accumulates, it does not execute immediately.** When a packet matches an entry in table N with the `Apply-Actions` instruction, the action runs right away. But with the `Write-Actions` instruction, the action is only written into the packet's **action set**, not executed yet. At the end of the pipeline (when there is no more `goto_table`), the whole action set runs at once. This mechanism lets later tables override the decisions of earlier tables.

**Rule 4, each table matches at most one entry.** When a packet matches several entries in the same table, the highest-priority entry wins. Matching two entries of the same priority in the same table is undefined behaviour (OVS picks one at random). So every entry in the same table must have a distinct priority or must match on disjoint fields.

### Instruction and action

From OpenFlow 1.1 onward, two important concepts are distinguished:

**Instruction** is a directive at the *table entry* level: `Apply-Actions`, `Write-Actions`, `Clear-Actions`, `Write-Metadata`, `Goto-Table`. Each entry has at most one instruction of each type.

**Action** is an operation at the *packet* level: `output:N`, `mod_dl_src`, `dec_ttl`, and so on. An action lives inside an `Apply-Actions` or `Write-Actions` instruction.

`ovs-ofctl` simplifies the syntax with the keyword `action=` (or `actions=`). This is in fact an implicit `Apply-Actions` instruction. The full form per the specification is `instructions=apply_actions(output:1)`, but OVS accepts the short form.

## 9.22.3 `goto_table` (OpenFlow 1.1+) versus `resubmit` (OVS extension)

The two mechanisms share the same goal: move the packet to the next table. The important differences are in the implementation.

### `goto_table=N`

A standard OpenFlow 1.1+ instruction. Properties:

- Only transitions to `N > current_table` (per Rule 2).
- The action set accumulates with the packet: every previous `Write-Actions` is still present at the destination table.
- The `metadata` field (64-bit) travels with the packet. A value updated by `Write-Metadata` is available at the destination table.
- `goto_table=N` is not allowed when N has already been visited; each packet visits each table at most once.

Example from Lab 6 Table 0 Classifier:

```bash
sudo ovs-ofctl add-flow s1 "table=0,ip,action=goto_table=1"
```

An IP packet enters table 0, the instruction `Goto-Table=1` runs, and the packet moves to table 1 for L3 forwarding lookup.

### `resubmit(,N)` and `resubmit(port,N)`

An OVS action that is not part of the OpenFlow standard, inherited from the NOX controller. Properties:

- Can transition to **any table**, including a smaller-numbered one (which can cause a loop: OVS caps the recursion depth at 75 and drops the packet on overflow).
- Actions after `resubmit` still execute when control returns, unlike `goto_table` (which transfers control completely).
- The variant `resubmit(port,N)` substitutes the `in_port` of the packet before looking up table N, which is useful for modelling virtual ports.
- Does not use the OpenFlow action set or metadata; it uses the OVS-extension registers `reg0` through `reg15`.

Sample OVS-extension pattern:

```bash
# Look up the MAC table, then return to the forward table
sudo ovs-ofctl add-flow br0 "table=0,dl_type=0x0800,action=resubmit(,10),resubmit(,20),output:2"
```

The packet visits table 10 (security lookup), then table 20 (MAC lookup), then returns to table 0 to run `output:2`. This pattern is common in OVN and in the NOX-era controllers.

### When to use which

In a traditional OpenFlow pipeline with a 1:1 hardware mapping (Broadcom Trident), use `goto_table` because it matches the silicon semantics. In OVS userspace with complex logic (OVN compiler, security gateway), use `resubmit` for its flexibility.

> **Key point:** the OVN `ovn-controller` emits flows with **both**. The OVN table layout uses `goto_table` to move between logical tiers (ingress ACL, load balancer, routing, egress ACL), but inside each tier there are several `resubmit` calls to modularise sub-flows. The output of `ovs-ofctl dump-flows br-int` on an OVN host shows the mix.

## 9.22.4 Lab 6 topology and the 3-table pipeline

Lab 6 topology (Crichigno et al., Lab 6 pages 5 to 7):

```
       192.168.1.0/24                         192.168.2.0/24
          h1 (.10)        s1 ─── s2        h2 (.10)
          MAC :01         (.1)   (.1)      MAC :02
          GW .1          MAC:05  MAC:06    GW .1
                     eth1/eth2     eth1/eth2
```

Two hosts are in different subnets (192.168.1.0/24 and 192.168.2.0/24), and two OVS switches (`s1` and `s2`) act as routers. MAC addresses are assigned manually: h1 is `:01`, h2 is `:02`, the s1 interface facing s2 is `:05`, and the s2 interface facing s1 is `:06`.

Gateway configuration on the switches:

```bash
# Assign IPs to switch interfaces (used as default gateways from the host point of view)
sudo ifconfig s1 192.168.1.1/24 up
sudo ifconfig s2 192.168.2.1/24 up
```

### The three-table pipeline

**Table 0, Classifier.** Distinguishes ARP from IP.

```bash
# s1 and s2: ARP handled like a traditional switch
sudo ovs-ofctl add-flow s1 "table=0,arp,action=normal"
sudo ovs-ofctl add-flow s2 "table=0,arp,action=normal"

# IP, push to table 1 for routing lookup
sudo ovs-ofctl add-flow s1 "table=0,ip,action=goto_table=1"
sudo ovs-ofctl add-flow s2 "table=0,ip,action=goto_table=1"
```

Two entries per switch. ARP broadcast goes through the `normal` action (which uses internal OVS MAC learning). IP packets enter the dedicated pipeline.

**Table 1, Layer 3 forwarding.** Rewrites the MAC, decrements the TTL, and transitions to table 2.

```bash
# s1 handles IP packets to the local host h1
sudo ovs-ofctl add-flow s1 \
  "table=1,ip,nw_dst=192.168.1.10,action=mod_dl_src=00:00:00:00:00:05,mod_dl_dst=00:00:00:00:00:01,dec_ttl,goto_table=2"

# s1 handles IP packets crossing into subnet 192.168.2.0/24 (to s2)
sudo ovs-ofctl add-flow s1 \
  "table=1,ip,nw_dst=192.168.2.0/24,action=mod_dl_src=00:00:00:00:00:01,mod_dl_dst=00:00:00:00:00:05,dec_ttl,goto_table=2"

# s2 mirrors the configuration for the reverse direction
sudo ovs-ofctl add-flow s2 \
  "table=1,ip,nw_dst=192.168.2.10,action=mod_dl_src=00:00:00:00:00:06,mod_dl_dst=00:00:00:00:00:02,dec_ttl,goto_table=2"
sudo ovs-ofctl add-flow s2 \
  "table=1,ip,nw_dst=192.168.1.0/24,action=mod_dl_src=00:00:00:00:00:02,mod_dl_dst=00:00:00:00:00:06,dec_ttl,goto_table=2"
```

Note the MAC rewrite pattern: when s1 forwards a packet to s2, the source MAC is set to the s1 MAC facing s2 (`:01`), and the destination MAC is set to the s2 MAC facing s1 (`:05`). This is the traditional router behaviour: rewrite the L2 MAC at each hop, keep the L3 IP unchanged.

**Table 2, Layer 2 forwarding.** Decides the output port based on the destination MAC.

```bash
sudo ovs-ofctl add-flow s1 "table=2,dl_dst=00:00:00:00:00:01,action=output:1"   # back to h1
sudo ovs-ofctl add-flow s1 "table=2,dl_dst=00:00:00:00:00:05,action=output:2"   # over to s2
sudo ovs-ofctl add-flow s2 "table=2,dl_dst=00:00:00:00:00:02,action=output:1"   # back to h2
sudo ovs-ofctl add-flow s2 "table=2,dl_dst=00:00:00:00:00:06,action=output:2"   # over to s1
```

Total: **6 flow entries per switch** (2 classifier, 2 L3, 2 L2), or 12 entries for the whole topology. Compared with a single-table design, this saves a lot as the deployment grows. Adding a new subnet only requires extra entries in tables 1 and 2, not a full rewrite.

### Tracing a packet through the pipeline

Lab 6 (page 19) uses `ovs-appctl ofproto/trace` to follow an IP packet from h1 to h2 through s1:

```bash
sudo ovs-appctl ofproto/trace s1 \
  "ip,nw_src=192.168.1.10,nw_dst=192.168.2.10,dl_src=00:00:00:00:00:01,dl_dst=00:00:00:00:00:05"
```

Expected output (doc-plausible, to be verified in C1b):

```

bridge("s1")
-------------

    goto_table:1
 1. ip,nw_dst=192.168.2.0/24, priority 32768

    mod_dl_dst:00:00:00:00:00:05

    goto_table:2
 2. dl_dst=00:00:00:00:00:05, priority 32768
    output:2

Final flow: ip,nw_src=192.168.1.10,nw_dst=192.168.2.10,dl_src=00:00:00:00:00:01,
```

Three important blocks when reading this output:

- **`0. ip, priority 32768`**: match in table 0, instruction `goto_table:1`.
- **`1. ip,nw_dst=192.168.2.0/24, priority 32768`**: match in table 1, actions to rewrite the MAC, `dec_ttl`, then `goto_table:2`.
- **`2. dl_dst=00:00:00:00:00:05`**: match in table 2, final action `output:2`.

`Datapath actions:` at the bottom shows the action sequence the kernel datapath will run. This is the "compiled" form of the whole pipeline as a single linear action chain.

## 9.22.5 Scaling up to a 5-table pipeline

Compass Chapter 8 presents an extension of Lab 6 to a canonical 5-table production pipeline (close to the OVN logical switch layout):

```
Table 0  - Port security and VLAN decap
Table 10 - Ingress ACL
Table 20 - Destination lookup (L2 MAC table or L3 route table)
Table 30 - Egress ACL
Table 40 - Output and VLAN encap
```

Design principles:

- **Spacing of 10 between tables** (0, 10, 20, 30, 40 instead of 0, 1, 2, 3, 4): so a new table can be inserted later without renumbering. This convention is also used in the `ovn-northd` layout.
- **Separate Ingress ACL from Egress ACL**: ACLs after NAT (egress) see the rewritten IP, while ACLs before NAT (ingress) see the original. The distinction matters during debugging.
- **Combined L2 and L3 destination lookup**: one table holds both the MAC table and the L3 FIB, matching on `dl_dst` or `nw_dst` depending on `dl_type`. This is simpler than splitting L2 and L3.

### Action set accumulating through the pipeline

When you use `Write-Actions` instead of `Apply-Actions`, the effect accumulates:

```bash
# Table 10 entry (Ingress ACL): only adds the action to the set, does not execute it
sudo ovs-ofctl add-flow br0 \
  "table=10,ip,tcp,tp_dst=22,action=write_actions(set_queue:1),goto_table=20"

# Table 40 entry (Output): at the end of the pipeline, the action set runs
sudo ovs-ofctl add-flow br0 "table=40,dl_dst=00:00:00:00:00:02,action=output:3"
```

A packet that matches the table 10 entry has `set_queue:1` in its action set. When it leaves the pipeline (at table 40 `output:3`), the action set (`set_queue:1`) runs before `output:3`. Result: the packet is placed into queue 1 and then sent out port 3.

Limitation: the action set holds only **one** action per type (one `output`, one `set_queue`, one `dec_ttl`, and so on). An entry in a later table overrides an entry in an earlier table; they do not accumulate within the same type.

## 9.22.6 The metadata field, passing state between tables

OpenFlow 1.1+ defines `metadata` as a 64-bit field that exists only inside the pipeline (it never appears in the real packet). Syntax:

```bash
# Table 10 entry: tag metadata 0x100 for an ACL "permit HTTP"
sudo ovs-ofctl add-flow br0 \
  "table=10,ip,tcp,tp_dst=80,action=write_metadata:0x100,goto_table=20"

# Table 40 entry: check metadata before output
sudo ovs-ofctl add-flow br0 \
  "table=40,metadata=0x100,dl_dst=00:00:00:00:00:02,action=output:3"
```

Mask form for metadata: `metadata:0x100/0xff00` matches only the upper 8 bits. This mechanism lets a later table know about a decision made by an earlier table, without modifying any real packet field (for example, you do not want to modify a VLAN tag just to carry state).

### Registers, an OVS extension

OVS extends the model with 16 registers `reg0` through `reg15` (each 32 bits) and `xreg0` through `xreg7` (each 64 bits, built by pairing `reg` slots). The differences against metadata:

- Metadata: OpenFlow standard, a single 64-bit slot.
- Registers: OVS extension, 512 bits total, more flexible.

OVN uses both: `metadata` carries the logical-datapath identifier (the ID of the current logical switch or router), while `reg0` through `reg15` carry the logical input port, output port, conntrack zone, ECMP hash, and similar values.

## 9.22.7 Comparison with OVN, where `ovn-northd` emits 50+ tables

An OVS operator writes Lab 6 with three hand-crafted tables. An OVN operator does not write OpenFlow flows directly. Instead, the operator declares logical switches, logical routers, and ACLs through `ovn-nbctl`. The `ovn-northd` daemon compiles them into logical flows, and `ovn-controller` translates them again into OpenFlow flows on `br-int`.

The result: on a production OVN compute host, `br-int` typically has **50+ tables** with thousands of flow entries. The OVN pipeline structure (per the `ovn-architecture(7)` man page):

```
Tables 0-7   - Logical port binding and classification
Tables 8-23  - Ingress pipeline (ACL, LB, NAT, L2 lookup, L3 lookup)
Tables 24-31 - Tunneling (Geneve encap)
Tables 32-47 - Egress pipeline (ACL, L2 output)
Tables 48-63 - Local output
```

Each logical datapath (switch or router) owns its own pipeline that is *multiplexed* through the `metadata` field. The metadata holds the datapath ID, so packets from different datapaths do not mix.

### When to hand-write flows, when to use OVN

Hand-written flows like Lab 6 are appropriate when:

- You need *predictable performance*: every entry is one you wrote, and there is no hidden flow explosion.
- You need *tight control*, for example SR-IOV NIC offload that requires specific flow shapes.
- The scale is small: under 10 subnets, under 100 ACL rules.

The OVN compiler is appropriate when:

- The scale is dozens to thousands of logical switches and routers.
- You need automatic multi-tenant isolation (a separate conntrack zone per logical switch).
- You need distributed routing: OVN emits flows on every chassis.

> **Key point:** there is no absolute right answer. Before 2018, OpenStack Neutron mostly used OVS plus the ML2 plugin emitting hand-written flows (a structure similar to an extended Lab 6). From 2018 to 2020 it migrated to ML2/OVN, where the OVN compiler emits 50+ tables automatically. The main reason is scale. When a compute host has 200 VMs across 50 tenants, hand-writing flows is not practical. But on edge nodes or NFV appliances, many projects still go back to hand-written flows to control latency.

Three checkpoints in the output:

- **`nw_ttl=64`** in `Flow:` becomes **`nw_ttl=63`** in `Final flow:`, confirming that `dec_ttl` works.
- **`Datapath actions: ...set(ipv4(ttl=-1))...`**: the kernel datapath receives the final compiled action chain.

### Step 2, refute "`goto_table` can move backward"

Try to install a flow that violates Rule 2 (goto a smaller table):

```bash
sudo ovs-ofctl add-flow s1 "table=2,ip,action=goto_table=1"
```

Expected: OVS refuses with `ovs-ofctl: actions are invalid with specified match (OFPBAC_BAD_TABLE_ID)`.

If the install succeeds (some older versions accept it), then tracing a packet into table 2 ends with OVS dropping it after 75 levels of recursion:

```bash
sudo ovs-appctl ofproto/trace s1 "ip,..."
# Expected: "stack too deep" or "Actions: (none); packet dropped due to pipeline loop"
```

The claim "the pipeline may loop" is refuted. Both OpenFlow 1.1+ and OVS enforce the forward-only rule.

### Step 3, compare with `resubmit`

Replace `goto_table` with `resubmit` to see the difference in flexibility:

```bash
sudo ovs-ofctl del-flows s1 "table=0,ip"
sudo ovs-ofctl add-flow s1 "table=0,ip,action=resubmit(,1),resubmit(,2)"
```

Trace again. Expect a result equivalent to the sequential pipeline, except that **action `output:2` from table 2 runs as soon as the packet returns to table 0**, unlike `goto_table` which transfers control completely.

### Lesson

A multi-table pipeline is a modularisation tool. Each table solves one task (ACL, routing, output), which keeps the total table size small and makes debugging easier. `goto_table` is "transfer of control": where it goes, it does not come back. `resubmit` is "subroutine call": it calls and returns to execute the remaining actions. Knowing when to use which is the skill that separates a junior OVS engineer from a senior.

## 9.22.8 Core takeaways

**One.** OpenFlow 1.0 (12/2009) had a single flow table. OpenFlow 1.1 (02/2011) added a multi-table pipeline only 14 months later, because a single table could not represent the Broadcom Trident ASIC. This is an unusually fast architectural change in standardisation history.

**Two.** Multi-table follows four hard rules: start at table 0, move forward only (table N to table M with M greater than N), the action set accumulates across the pipeline and runs at the end, each table matches at most one entry (by priority).

**Three.** `goto_table=N` (OpenFlow 1.1+ standard) is a one-way transfer of control. `resubmit(,N)` (OVS extension from NOX) goes and returns; it allows a smaller N and limits recursion depth to 75.

**Four.** The Lab 6 3-table pipeline (Classifier, L3 forwarding, L2 forwarding) is the basic template for a simple router. The 5-table extension (with Ingress ACL plus Egress ACL) is production-grade. OVN auto-generates 50+ tables.

**Five.** Metadata (64-bit, standard) and the `reg0` through `reg15` registers (OVS extension) carry state between tables without modifying any real packet field. OVN uses both: metadata for the logical-datapath ID, and registers for the conntrack zone and the ECMP hash.

**Six.** Hand-written flows fit a small scale, predictable performance, and tight SR-IOV-offload control. The OVN compiler fits large scale, multi-tenancy, and distributed routing. There is no absolute "right"; understand both, and choose according to the problem.

## 9.22.9 `goto_table`, the instruction that hands control to a higher table in the OF 1.1+ pipeline

**Concept.** `goto_table` is an **OpenFlow 1.1+ instruction** (not an action) attached to a flow entry. It tells the switch: after executing the `Apply-Actions` instruction of this entry, do not end the pipeline; instead, hand the packet to the flow table with a larger number (the `table_id` chosen by the instruction) so the packet can match against more entries. `goto_table` is the **only** standard OpenFlow mechanism for moving between tables; every multi-table pipeline is chained together through it. Distinguish it clearly from actions such as `output`, `mod_dl_src`, or `dec_ttl`, which are *operations on the packet*; `goto_table` is *pipeline-flow control*. Distinguish it also from the OVS extension action `resubmit(,N)`. `resubmit` is an action, not an instruction; it calls table N and then returns to continue the action list. `goto_table` hands control over one-way, with no return.

**History and background.** When ONF published OpenFlow 1.0 in December 2009 with a single flow table, silicon vendors (Broadcom, Marvell) responded within 6 months that the one-table architecture did not map onto the actual TCAM pipeline of an L2/L3 ASIC, which has 5 to 7 separate tables (ingress parser, ingress ACL, L3 lookup, L2 lookup, egress ACL). The ONF Technical Memorandum of July 2010 proposed multi-table. OpenFlow 1.1 (02/2011, Section 5.2 "Pipeline Processing", Section 5.9 "Instructions") officially added `OFPIT_GOTO_TABLE`, `OFPIT_WRITE_METADATA`, `OFPIT_WRITE_ACTIONS`, `OFPIT_APPLY_ACTIONS`, and `OFPIT_CLEAR_ACTIONS`. Among these, `Goto-Table` is the only one that hands control between tables; the other four operate on the *same entry*. OpenFlow 1.2 (12/2011) and 1.3 (06/2012) kept the semantics. OF 1.5 (03/2015) added an egress table and required `Goto-Table` from an ingress entry to land on another ingress table, and from an egress entry to land on another egress table. OVS supported `goto_table` starting with version 1.10 (02/2013), when the `ofproto-dpif-xlate` codepath was reorganized to handle multi-table.

**Position in the architecture.** In the OVS source tree (baseline `v2.17.9`, branch `branch-2.17`), `goto_table` appears at four layers. (1) **Wire format**: `include/openflow/openflow-1.1.h` defines the instruction `enum OFPIT11_GOTO_TABLE = 1` and the 8-byte `struct ofp11_instruction_goto_table`. (2) **OVS internal representation**: `include/openvswitch/ofp-actions.h` declares `struct ofpact_goto_table` which contains a single `uint8_t table_id` field (the instruction is stored in the same list as actions inside the `ofpacts` structure). (3) **Parse, encode, format**: `lib/ofp-actions.c` contains the functions `parse_GOTO_TABLE`, `encode_GOTO_TABLE`, and `format_GOTO_TABLE`, which convert between the text form `goto_table:N`, the OVS struct, and the OF wire bytes. (4) **Xlate**: `ofproto/ofproto-dpif-xlate.c` handles the `OFPACT_GOTO_TABLE` case inside `do_xlate_actions()` by calling `xlate_table_action()` to dispatch the classifier lookup to the destination table. Scope: per flow entry. Each flow entry holds at most one `Goto-Table` instruction (per spec 1.1+ Section 5.9 "Instructions"); to chain N tables, the operator writes N flow entries with `goto_table` spread across the tables.

**Role.** `goto_table` plays three roles tied to the entire multi-table pipeline. (1) **Chaining primitive**: the only ONF-standard mechanism for moving from table N to table M where M is greater than N. Without it, every table is its own silo. (2) **Action-set transport**: the `Goto-Table` instruction does not clear the accumulated action set. Every prior `Write-Actions` carries forward to the destination table, which lets later tables override earlier decisions (for example, table 0 issues `write_actions(output:1)` and table 5 issues `write_actions(output:2)`, and the final outcome is `output:2` because the action set keeps only one `output`). (3) **Modularization enabler**: lets the operator write the pipeline as a sequence of stages (classify, ACL, routing, forwarding, egress) instead of one Cartesian table, which reduces the entry count from a product to a sum.

**Why it exists.** Two core technical motivations. (1) **Silicon mapping.** Commercial L2/L3 ASICs have a hard-wired multi-stage pipeline. OF 1.0 single-table forced the compiler to simulate every stage through one virtual table, which exploded the entry count. `goto_table` lets each OF table map 1:1 to a silicon stage. (2) **Rule-design scalability.** With 100 ACL rules, 1000 FIB entries, and 5000 MAC entries, a single-table design needs 100 × 1000 × 5000 = 500 million entries, while a multi-table design only needs 100 + 1000 + 5000 = 6100 entries chained through `goto_table`. The OF 1.1 spec Section 1.1 "Switch Components" explains this directly: "An OpenFlow switch has multiple flow tables ... a packet matches against entries in successive tables". The design philosophy trusts the compiler or operator to decompose the logic into stages; OF only provides the primitive `goto_table` to chain them.

**Problems solved.** Four specific problems that `goto_table` addresses. (1) **Table fan-out.** Instead of writing `100 × 5000` entries for the Cartesian product (ACL × MAC), the operator writes 100 entries in table 0 plus 5000 entries in table 1 and connects them with `goto_table:1`. (2) **Stage isolation.** ACL rules are not mixed with MAC entries; each stage can be debugged on its own (`ovs-ofctl dump-flows br0 table=0` only shows the ACL stage). (3) **Cumulative action.** The same `Goto-Table` instruction transports the action set across tables, without copying entries. (4) **Loop safety.** The spec requires `table_id > current_table_id`, which guarantees the pipeline always terminates (a forward DAG). There is no way to create a loop with pure `goto_table`; loops can only happen through `resubmit` or `recirc_id`, which are different topics covered in Part 9.32.

**Importance.** `goto_table` is one of the three absolutely-core primitives of multi-table pipelines, alongside the match field and the action set. Every production-grade OpenFlow controller (Faucet, ONOS, Ryu, OVN `northd`) emits `goto_table` in 80 percent or more of the flow entries that are not in the last table. In an OVN deployment on the branch-22.03 baseline, the output of `ovs-ofctl dump-flows br-int` on a typical compute host shows that about 70 percent of flow entries end with `goto_table:N`, about 20 percent end with `resubmit(,N)`, and the rest end with `output`, `drop`, or `controller`. An OVS or SDN engineer who does not deeply understand `goto_table` cannot read the OVN `br-int`, cannot analyze the pipeline that OVN `northd` compiles, and cannot debug why a packet "disappears" between table 17 and table 24. `goto_table` belongs to the same tier as `output` and `match`, the absolutely-required primitives of OpenFlow.

**How it works.** The wire format `struct ofp11_instruction_goto_table` is exactly 8 bytes (fixed, aligned to 64 bits):

```
Offset  Field     Length  Description                        Example goto_table:1
------  --------  ------  --------------------------------   --------------------
0-1     type      2 byte  OFPIT_GOTO_TABLE = 0x0001          0x00 0x01
2-3     len       2 byte  Length of the struct = 0x0008      0x00 0x08
4       table_id  1 byte  Destination table (0-254;          0x01
                          0xff reserved)
5-7     pad       3 byte  Padding for 64-bit alignment       0x00 0x00 0x00
```

The OVS internal struct `ofpact_goto_table` (defined in `include/openvswitch/ofp-actions.h`):

```c
/* OFPACT_GOTO_TABLE
 *
 * Used for OFPIT11_GOTO_TABLE */
struct ofpact_goto_table {
    OFPACT_PADDED_MEMBERS(
        struct ofpact ofpact;
        uint8_t table_id;
    );
};
```

Codepath when a packet enters the pipeline: `ofproto-dpif-xlate` calls `do_xlate_actions()` to walk the `ofpacts` list of the current entry. When it sees `OFPACT_GOTO_TABLE`, it does not recurse immediately; instead it sets `ctx->table_id = goto_table->table_id` and then `xlate_table_action()` performs the classifier lookup in the destination table and continues `do_xlate_actions()` on the newly matched entry. The invariant `goto_table->table_id > ctx->table_id` is enforced in `parse_GOTO_TABLE` at flow-install time, and a violation returns `OFPBIC_UNSUP_INST` or `OFPBIC_BAD_TABLE_ID`. Termination: when the pipeline reaches an entry without a `goto_table`, the accumulated action set is executed and the pipeline ends.

**How an operator masters this.** Six core actions. (1) **Install `goto_table` with the correct `ovs-ofctl` syntax**:

```bash
sudo ovs-ofctl add-flow br0 \
  "table=0,priority=100,ip,actions=goto_table:1"
sudo ovs-ofctl add-flow br0 \
  "table=1,priority=100,ip,nw_dst=10.0.0.0/8,actions=output:2"
```

(2) **Read `dump-flows` and recognize the chain**: `ovs-ofctl dump-flows br0 | grep goto_table` shows every pipeline-transitioning entry; missing results in a non-final table mean the pipeline is broken. (3) **Verify the increasing order**: use `awk` to make sure every `goto_table:N` has N greater than the table that contains the entry. (4) **Trace with `ovs-appctl ofproto/trace`** to verify the transition (see the example below). (5) **Avoid confusion with `resubmit`**: if the goal is simple pipeline modularization, use `goto_table`; if the goal is to call a sub-pipeline and then return to continue the action list, use `resubmit`. (6) **Read OVN flows**: with OVN, the `ovn-trace` command (Part 13.7) decodes pipeline logic backward into NB ACLs and routes, while `ofproto/trace` (Part 9.25) shows the chain of `goto_table` from table 0 through table 24 to table 65 on `br-int`.

**Classification.** `goto_table` belongs to the **OpenFlow Instruction** category in the taxonomy of 5 instruction groups (`Apply-Actions`, `Clear-Actions`, `Write-Actions`, `Write-Metadata`, `Goto-Table`). It differs from `Apply-Actions` (which executes an action immediately) and from `Write-Actions` (which accumulates into the action set for execution at the end of the pipeline). `goto_table` is the only instruction that changes the *next table*; the other four operate *on the packet*. In the "control flow" sub-classification, `goto_table` is paired with `output_to_controller` and `drop`; all three end a pipeline decision, but `goto_table` continues inside the switch while the other two leave the pipeline. This is an absolutely-core instruction, indispensable in any multi-table pipeline.

**Usage workflow.** A pipeline-design workflow that uses `goto_table` has seven steps. (1) **Identify the chain of logical stages** (for example: classify, ACL ingress, routing, ACL egress, output). (2) **Number the tables with gaps** (0, 10, 20, 30, 40 instead of 0, 1, 2, 3, 4) so a new stage can be inserted later without renumbering. (3) **Write the table 0 entry** as `actions=...,goto_table:10` to push every packet into the ACL ingress stage. (4) **Write the entry for each subsequent table** ending with `goto_table:<next_table>` to chain forward; the entry in the last table ends with `output` instead of `goto_table`. (5) **Always install a table-miss entry** (`priority=0,actions=...`) for every table that can receive packets that do not match. Without a table-miss, an unmatched packet is silently dropped. (6) **Verify with `ovs-appctl ofproto/trace`** for at least one representative packet per tenant or protocol to confirm the full transition. (7) **Document the table numbering** in a README or in the cookie comment (`cookie=0xacl10`) so on-call engineers understand each table's role.

**When something goes wrong.** Four signals in production that bring `goto_table` to mind. (1) **Silent drop in the middle of the pipeline**: `ovs-ofctl dump-flows br0` shows that an entry in table 5 has `goto_table:10`, but table 10 has no matching entry, so the packet is dropped; the table 5 `n_packets` counter still increases but nothing arrives at the egress. The diagnosis is to run `ofproto/trace` and observe "Final flow ... no_match in table 10". The fix is to add a table-miss entry for table 10 or to add a matching entry. (2) **`OFPBIC_BAD_TABLE_ID` at flow install**: trying to install `goto_table:N` for an N that does not exist (for example N=255 when the switch only supports 254 tables). The fix is to inspect `n_tables` through `ovs-ofctl show br0`. (3) **`OFPBIC_UNSUP_INST` at flow install**: trying to install `goto_table:M` where M is less than or equal to the current table. The fix is either to redesign the table numbering or to use `resubmit(,M)` (an OVS extension that allows reverse). (4) **OVN packet does not reach the destination**: `ovn-trace` reports the packet follows the correct logical pipeline, but `ofproto/trace` on `br-int` shows the packet lost in the middle of a `goto_table` chain (for example, between table 17 and table 24). A common cause is that `ovn-controller` has not yet installed the table 24 entry on the chassis because the SBDB `Logical_Flow` has not synchronized. The fix is to compare `ovn-sbctl lflow-list` with `ovs-ofctl dump-flows br-int table=24`.

**Tightly related to.** `goto_table` is closely paired with five entities. (1) The `Apply-Actions` instruction (Part 3.6): `goto_table` follows `Apply-Actions` in the same entry, so the action is executed first and then the packet moves to the destination table. (2) The `Write-Actions` instruction: the action set accumulates across the `goto_table` chain and is not cleared on transition. (3) The `resubmit(,N)` action (NXM extension, Part 9.22.3): semantically opposite. `resubmit` returns; `goto_table` does not. (4) The table-miss entry (`priority=0`): always recommended on every table that can receive packets through `goto_table`. (5) The OVN logical pipeline (Parts 13.7, 13.19): `ovn-controller` compiles every logical flow into a chain of `goto_table` on `br-int`. Tables 0 to 7 hold ingress port binding, 8 to 23 hold the ingress logical pipeline, 24 to 31 handle tunneling, and 32 to 47 hold the egress logical pipeline.

**Version differences.**

| Version | `goto_table` support | `table_id` constraint | Semantic change |
|---------|----------------------|-----------------------|-----------------|
| OF 1.0 (12/2009) | Not supported | N/A | Single-table, no instructions |
| OF 1.1 (02/2011) | Yes (new) | `table_id > current_table_id` | Instructions appear for the first time |
| OF 1.2 (12/2011) | Yes | Same as 1.1 | OXM TLV match replaces fixed structs |
| OF 1.3 (06/2012) | Yes | Same as 1.1 | Group and meter tables added to the pipeline |
| OF 1.4 (10/2013) | Yes | Same as 1.1 | Bundle commit affects `goto_table` install |
| OF 1.5 (03/2015) | Yes | Distinguishes ingress and egress: an ingress entry only goes to an ingress table; an egress entry only goes to an egress table | Egress table added to the pipeline |

OVS handles every 1.1+ version through a single codepath in `parse_GOTO_TABLE` and `encode_GOTO_TABLE` (`lib/ofp-actions.c`); the version difference is only at the wire encode (the OF 1.1+ struct is unchanged). The OVS branch-2.17 default protocol is `OpenFlow13`, so `goto_table` is available by default. Open vSwitch supported `goto_table` starting with version 1.10 (02/2013); earlier versions (1.0 to 1.9) only had the `resubmit` extension.

**Source code reference.** In the OVS source tree (baseline `v2.17.9`, branch `branch-2.17`), `goto_table` lives at five locations. (1) `include/openflow/openflow-1.1.h` declares the `enum ofp11_instruction_type` with `OFPIT11_GOTO_TABLE = 1` and the 8-byte `struct ofp11_instruction_goto_table` (type, len, table_id, pad[3]). (2) `include/openvswitch/ofp-actions.h` declares `struct ofpact_goto_table`, which only carries a `uint8_t table_id`, registered through the macro `OFPACT(GOTO_TABLE, ofpact_goto_table, ofpact, "goto_table")`. (3) `lib/ofp-actions.c` contains four functions for `goto_table`: `encode_GOTO_TABLE`, `parse_GOTO_TABLE`, `format_GOTO_TABLE`, and `check_GOTO_TABLE`. The functions cover encode (build the OF wire bytes from the OVS struct), parse (build the OVS struct from the text `goto_table:N`), format (build the text from the OVS struct), and check (validate the parsed instruction at flow-install time, including the `table_id <= 0xff` check). The decode side is dispatched through macro-generated wrappers from the `OFPACT` registry, not as a standalone `decode_OFPIT11_GOTO_TABLE` function. (4) `ofproto/ofproto-dpif-xlate.c` handles the `OFPACT_GOTO_TABLE` case inside `do_xlate_actions()` by setting `ctx->table_id` and then calling `xlate_table_action(ctx, ctx->in_port, table_id, ...)` to perform the classifier lookup on the destination table. The signature is `static void xlate_table_action(struct xlate_ctx *, ofp_port_t in_port, uint8_t table_id, bool may_packet_in, bool honor_table_miss, bool with_ct_orig, bool is_last_action, xlate_actions_handler *)`. (5) `ofproto/ofproto.c` validates at flow-install that `goto_table->table_id > flow->table_id`, which enforces the forward-only invariant. Specification: ONF OpenFlow Switch Specification 1.5.1 Section 6.1 "Pipeline Processing" and Section A.2.5 "Goto-Table Instruction"; OpenFlow 1.1.0 Section 5.9.1 "Instructions" introduced the instruction. All function names above are verified at the OVS v2.17.9 baseline (commit `0bea06d9957e3966d94c48873cd9afefba1c2677`, 2024-02-08).

**Real-world incident.** An OVN multi-tenant incident in 2024-Q4: after upgrading `ovn-controller` from 22.03.7 to 22.03.9 across 64 compute nodes, several tenants reported east-west traffic between two VMs on the same logical switch being randomly dropped (5 to 10 percent of packets). `ovn-trace` on SBDB reported the correct pipeline: tables `LS_IN_PORT_SEC_L2`, `LS_IN_PRE_ACL`, `LS_IN_ACL`, `LS_IN_QOS_MARK`, `LS_IN_LB`, `LS_IN_STATEFUL`, `LS_IN_ARP_ND_RSP`, `LS_IN_DHCP_OPTIONS`, `LS_IN_L2_LKUP`, output. But `ofproto/trace` on the host's `br-int` showed the packet ending unexpectedly at table 17 with "no_match", while the same packet from a different VM on the same logical switch passed through successfully. Comparing `ovs-ofctl dump-flows br-int table=17` against `ovn-sbctl lflow-list <ls>` showed that two important entries (`LS_IN_L2_LKUP` for unicast destination MAC) only existed on about 90 percent of the chassis. On the remaining 10 percent of chassis, `ovn-controller` had installed the entries into table 16 instead of table 17 because of a renumbering bug in version 22.03.9 (the stage order in `ovn-northd` had changed). The consequence: the `goto_table:17` entry in table 16 sent the packet into an empty table 17, where it was dropped. Short-term fix: roll `ovn-controller` back to 22.03.7 on the affected chassis. Long-term fix: wait for the upstream patch `30aab12c0` that rebuilds the stage-to-table mapping inside `ovn-controller`. Lesson: a `goto_table` chain is only stable when *both sides* (the source entry and the destination entry) are installed consistently. Any skew between the producer (`ovn-controller`) and the destination state (the `table_id` mapping) shows up as a silent packet drop. Cross-link: 20.0 incident-response decision tree, Part 13.7 `ovn-controller` internals.

**Common failures and diagnostic signals.** Four common operator mistakes. (1) **Forgetting the table-miss in the destination table**: a `goto_table:N` entry pushes the packet into table N, but table N has no matching entry, so the packet is dropped. Signal: the table N counter shows `lookup_count` increasing while `matched_count` stays flat. Diagnose with `ovs-ofctl dump-table-features br0`, which shows `default = drop` if there is no table-miss. Fix: add `table=N,priority=0,actions=drop` (explicit) or `actions=normal`. (2) **Wrong `table_id` in the cookie**: writing `cookie=0x10` to mean "table 10" but failing to keep it in sync with `goto_table:10`. A future maintainer who greps by cookie will not match the actual table. Fix: adopt a clear convention where the cookie encodes both the table and the rule ID. (3) **Forward-only violation under an old OF version**: switch firmware that speaks OF 1.0 does not understand `goto_table` and returns `OFPET_BAD_INSTRUCTION` or fails silently. Signal: `ovs-ofctl -O OpenFlow10 add-flow ... actions=goto_table:1` always errors. Fix: use `-O OpenFlow13`. (4) **Mistaking `goto_table` for an action**: writing `actions=output:1,goto_table:2` (OVS accepts this syntax, but the spec treats `goto_table` as an instruction that must follow the action list). Signal: undefined behavior on hardware switches other than OVS. Fix: understand that `goto_table` is an instruction, place it at the end of the actions list, or use the explicit form `instructions=apply_actions(output:1),goto_table:2`.

**Comparison with other systems.** Four important comparisons. (1) **The OVS extension `resubmit(,N)`** (Part 9.22.3): same intent of moving the packet to table N, but `resubmit` is an *action* (it calls a sub-pipeline and returns to execute the rest of the action list), while `goto_table` is an *instruction* (a one-way transfer of control with no return). `resubmit` allows N to be smaller and limits recursion depth to 75; `goto_table` requires N to be greater than the current table. (2) **The OF 1.1+ `Goto-Group` instruction**: moves the packet to a group entry (multipath, broadcast); differs from `goto_table` because the group executes one fixed action bucket and does not continue the pipeline. (3) **Linux netfilter chain hooks** (PREROUTING, INPUT or FORWARD, POSTROUTING): also a multi-stage pipeline, but the chains are fixed by netfilter design; new chains cannot be inserted dynamically the way `goto_table` inserts an OF table. The iptables `--jump CHAIN` is logically similar to `goto_table` but uses a different wire format. (4) **P4 control flow `apply { table_a.apply(); table_b.apply(); }`**: P4 expresses pipeline dataflow more completely (with if/else branching on lookup results, table abstractions, action selectors), while OF `goto_table` only chains linearly by `table_id`. Trade-off: P4 is more powerful but requires a target-specific P4 compiler; OF `goto_table` is simpler and standardized, available on every OF 1.1+ switch.

## 9.22.10 `multi-table pipeline`, the OpenFlow 1.1+ paradigm that replaces the OF 1.0 single table

**Concept.** A `multi-table pipeline` is the **packet-processing paradigm** of OpenFlow 1.1+ in which a switch organizes flow entries into **N flow tables** numbered `0, 1, 2, ..., N-1` (with N at most 254 in the wire format `uint8_t`, where 0xFF is reserved for `OFPTT_ALL`). A packet entering the pipeline always starts at table 0; each table performs one classifier lookup; the matching entry specifies an *instruction set* with at most five instructions (`Apply-Actions`, `Clear-Actions`, `Write-Actions`, `Write-Metadata`, `Goto-Table`); the `Goto-Table` instruction pushes the packet to a higher-numbered table to continue, while an entry without `Goto-Table` marks the *end of the pipeline*. At that point, the action set accumulated across tables is executed atomically and the packet leaves the pipeline. Distinguish clearly between *flow table* (the component unit) and `goto_table` (the instruction that moves between tables). `multi-table pipeline` is the **overall architecture**, whether N = 1 (the degenerate case, equivalent to the OF 1.0 single table) or N = 64 (the production OVN `br-int` per chassis).

**History and background.** OpenFlow 1.0.0 (12/2009) specified a *single flow table* with about 12 fixed match fields, no instruction set, and no concept of chaining tables. Implementations on commercial L2/L3 ASICs (Broadcom Trident, Marvell Prestera) exposed an architectural gap: the ASIC has a hard-wired multi-stage pipeline (ingress parser, ingress ACL, L3 lookup, L2 lookup, egress ACL, queueing, egress parser), and each stage is a separate TCAM. Forcing a single OF table to simulate every stage required a Cartesian product of entries, which exploded in size. The ONF Technical Memorandum of July 2010, *"Issues with the OpenFlow 1.0 specification"*, proposed multi-table. OpenFlow 1.1.0 (02/2011, Section 1.1 *"Switch Components"* and Section 5.2 *"Pipeline Processing"*) officially introduced the `multi-table pipeline`, the five instructions, and OXM TLV match. OF 1.2 (12/2011) kept the architecture and extended OXM. OF 1.3 (06/2012) added the group table and the meter table to support the pipeline. OF 1.4 (10/2013) added bundles for atomic flow installs. OF 1.5 (03/2015) split the *ingress* and *egress* pipeline into two independent table chains, each with its own forward-only `goto_table` rule. OVS supported a true multi-table pipeline starting with version 1.10 (02/2013); earlier versions (1.0 to 1.9) only had the `resubmit` extension inherited from NOX (Stanford, 2008). The 14 months between OF 1.0 and OF 1.1 was an unusually short standardization cycle, a strong signal that a single table was not viable in production.

**Position in the architecture.** The `multi-table pipeline` is not a field, an action, or a message; it is an **architectural wrapper** around the entire datapath logic of the switch. In the OVS source baseline `v2.17.9`, the pipeline appears at four layers. (1) **Kernel and userspace datapath**: the kernel datapath (`datapath/datapath.c` plus `datapath/flow_table.c`) only sees a flat megaflow cache; the pipeline is a *userspace abstraction*. Every first-time upcall traverses the full pipeline in `ofproto-dpif` and produces a synthesized megaflow. (2) **`ofproto-dpif` xlate** (`ofproto/ofproto-dpif-xlate.c`): the function `xlate_actions()` calls `do_xlate_actions()` for each matched entry, and on `OFPACT_GOTO_TABLE` it recurses through `xlate_table_action()` to dispatch into the destination table, traversing the whole pipeline in a single upcall. (3) **Per-table classifier** (`ofproto/ofproto.c` plus `lib/classifier.c`): each table is a `struct oftable` containing a `struct classifier cls` that holds the per-table flow entries. (4) **OF wire**: the message `OFPT_TABLE_FEATURES` (type 12) reply lists N tables, each with its own capabilities (max_entries, supported instructions, supported actions); `OFPT_TABLE_MOD` (type 17) configures the table-miss behaviour. Scope: per bridge. Each OVS bridge has its own independent pipeline (for example, `br-int` runs 64 tables for OVN, while `br-ex` runs 1 to 2 tables for external traffic).

**Role.** The `multi-table pipeline` plays five irreplaceable roles. (1) **Modularization**: separates packet processing into independent stages (classify, ACL, routing, forwarding, egress), one stage per table, similar to how an OS splits the kernel into modules. (2) **Scalability**: instead of the Cartesian product inside a single table, multi-table only needs the sum. For example, with 100 ACL rules, 1000 FIB entries, and 5000 MAC entries, a single table needs 5 times 10 to the 8 entries; a multi-table design needs 6.1 times 10 to the 3 entries; a reduction of about 8.2 times 10 to the 4. (3) **Silicon mapping**: enables a 1:1 compile from an OF table to a TCAM stage in the ASIC, which lets the design exploit the hardware fast path. (4) **Compositional reasoning**: an engineer can debug each stage on its own (`ovs-ofctl dump-flows br-int table=24`); a single-table design forces the engineer to look at every entry. (5) **Logical-physical separation**: lets a high-level compiler (such as `ovn-northd`) emit a logical pipeline of 64 stages, and lets `ovn-controller` translate that into OF flows on `br-int` per chassis. This separates the controller-logic concern from the switch-executor concern.

**Why it exists.** Three core technical motivations, all at once. (1) **Entry explosion in a single table** (already explained under Role). (2) **Silicon pipeline mismatch**: commercial L2/L3 ASICs have natural hard-wired stages; forcing a single OF table makes the compiler emit entries that flatten the stages and lose every hardware advantage. Multi-table restores the natural mapping. (3) **Rule-design ergonomics**: writing 100 ACL rules plus 5000 MAC entries separately is easy to read and easy to inspect; writing 5 times 10 to the 5 Cartesian entries is not feasible for a human. The OF 1.1 spec Section 1.1 says directly: *"OpenFlow switches contain multiple flow tables to allow for scalability and modularity"*. Design philosophy: the standard provides primitives (table, instruction, `Goto-Table`); the compiler or operator decides how to split the stages; the standard does not fix the number of tables.

**Problems solved.** Four specific problems addressed by the pipeline. (1) **Table fan-out** (Cartesian explosion). (2) **Stage isolation**: each logical stage is independent, so debugging one stage does not require knowledge of the others. (3) **Hierarchical compilation**: lets a tool like OVN `northd` emit a 64-stage logical pipeline, after which an OF translator emits per-chassis hardware OF flows. Without multi-table, this hierarchy cannot be compiled. (4) **Operational checkpoint**: per-table counters (`lookup_count`, `matched_count`) are an important diagnostic signal that shows where in the pipeline a packet drops. A single table only has one combined counter and loses the location information.

**Importance.** The `multi-table pipeline` sits in the **absolutely-core position** of OpenFlow 1.1+ and OVS, on par with `match field` and `action set`. Every production controller after OF 1.1 (Faucet, ONOS, Ryu, OpenDaylight, OVN `northd`) assumes multi-table; no serious controller still operates with a single table. On a typical OVN compute host (branch-22.03 baseline, 50 tenant VMs, 10 logical switches), the output of `ovs-ofctl dump-flows br-int | wc -l` is around 3000 to 8000 flow entries spread across about 64 tables. An SDN engineer who does not deeply understand the pipeline paradigm cannot (a) read the OVN `br-int`, (b) analyze the pipeline that OVN `northd` compiles, (c) debug a packet that "disappears" between table 17 and table 24, (d) optimize the OF rule design for hardware offload, or (e) rewrite the stage sequence during a logic upgrade. Understanding the pipeline is on par with understanding the TCP three-way handshake for a network engineer.

**How it works.** Traversing the pipeline for one packet has four phases. **Phase 1 (enter table 0).** The packet arriving at the switch is first parsed (fields extracted) and then enters the classifier of table 0. The lookup picks the highest-priority matching entry; if a match exists, the entry's instruction set is selected. **Phase 2 (apply per-entry instructions).** The entry's instruction set runs in a fixed order per ONF spec: `Meter` (if present), `Apply-Actions` (executes actions immediately and mutates the packet), `Clear-Actions` (clears the action set), `Write-Actions` (adds actions to the action set without executing them), `Write-Metadata` (updates the 64-bit metadata field), and `Goto-Table` (moves to the next table). If there is no `Goto-Table`, the flow jumps to Phase 4. **Phase 3 (transition).** The packet (possibly mutated by `Apply-Actions`) plus the accumulated action set and metadata move to the table indicated by `Goto-Table`; the classifier looks up that table; Phase 2 repeats. **Phase 4 (terminate).** When the pipeline reaches an entry without `Goto-Table` (or when a table-miss with `output:CONTROLLER` happens), the action set is applied atomically in the standard order (`copy_ttl_in`, `pop`, `push_*`, `copy_ttl_out`, `dec_*`, `set`, `qos`, `group`, `output`); the packet leaves the pipeline. The full sequence of phases 1 to 4 happens atomically per packet (no context switch between phases); an entry deleted by the controller between Phase 1 and Phase 4 does not affect a packet already in flight.

**How an operator masters this.** Seven core actions. (1) **List table support**: `ovs-ofctl -O OpenFlow13 dump-tables br0` shows how many tables the switch has and the `max_entries` of each. (2) **Design the stage logic**: draw the pipeline on paper before installing entries; number the tables with gaps (0, 10, 20, 30, ...) so a new stage can be inserted later. (3) **Install a table-miss in every table**: `ovs-ofctl add-flow br0 "table=N,priority=0,actions=drop"` (or `actions=normal`) for every N that can receive packets. Without a table-miss, an unmatched packet is silently dropped. (4) **Verify the `goto_table` chain**: run `ovs-ofctl dump-flows br0 | grep -E "table=|goto_table" | sort -t= -k2 -n` to make sure every non-final table has an entry that hands forward. (5) **Trace the full pipeline**: `ovs-appctl ofproto/trace br0 "in_port=1,ip,nw_src=10.0.0.1,..."` shows the path through every table along with the action set and metadata at each phase. (6) **Read the OVN `br-int`**: `ovs-ofctl dump-flows br-int | awk '{print $5}' | sort -t= -k2 -n -u` shows which tables are active and maps them to SBDB Logical_Flow stages through `ovn-detrace`. (7) **Measure per-table counters**: `ovs-ofctl dump-tables-desc br-int` plus `ovs-ofctl dump-flows br-int table=N` lets the operator compare `lookup_count` and `matched_count` to identify tables where packets are dropping.

**Classification.** Pipelines can be classified along three axes. By **scale**: single-table (the degenerate case, OF 1.0 or OF 1.1+ with N=1), small (2 to 10 tables, mostly manual config), medium (10 to 30 tables, mid-tier controllers such as Faucet), or large (30 to 64+ tables, OVN `northd` compile). By **direction** (OF 1.5+): ingress-only pipeline (OF 1.1 to 1.4), or bidirectional ingress-plus-egress pipeline (OF 1.5+, two independent chains). By **hierarchy**: flat pipeline (every table at the same level, no group or meter), or hierarchical pipeline (the main table chains through `Goto-Table` and group tables are called through `output:group:N`). By **content origin**: hand-written (operator writes with `ovs-ofctl`), compiler-generated (OVN `northd`, ONOS Atomix), or hybrid (90 percent compiler plus 10 percent operator override). This is an absolutely-core paradigm, indispensable in any production OF 1.1+ controller.

**Usage workflow.** A multi-table-pipeline design workflow has eight steps. (1) **Analyze the processing logic**: list the required stages (classify, ACL, routing, ARP, MAC learning, output). (2) **Number the tables with gaps** (0, 5, 10, 15, ...) so a new stage can be inserted without renumbering. (3) **Write the table 0 entry** ending with `goto_table:5` (push every packet into the first stage). (4) **Write the entry for each stage** ending with `goto_table:<next>` for an intermediate stage, or `output:N` (or `drop`) for the final stage. (5) **Always install a table-miss** for every table that can receive packets, either `priority=0,actions=drop` (explicit) or `actions=goto_table:<next>` (skip). (6) **Verify with `ofproto/trace`** for 5 to 10 representative packets per tenant or protocol; confirm the path through the expected stages. (7) **Document the pipeline**: a README or a cookie comment (`cookie=0xacl05`) should record the role of each table for on-call engineers. (8) **Monitor per-table counters** in production to catch unusual drops at a specific stage.

**When something goes wrong.** Five production signals that bring the pipeline concept to mind. (1) **Silent drop in the middle of the pipeline**: `dump-flows br0 table=N` has no entry that matches the required field, so the packet enters table N and is dropped. Detect by comparing `lookup_count` (rising) with `matched_count` (flat). (2) **OVN packet does not reach the destination on a single chassis**: `ovn-trace` on SBDB shows the correct logical pipeline, but `ofproto/trace` on the chassis `br-int` loses the packet between `goto_table` steps. The usual cause is that `ovn-controller` has not yet installed the entry or that the table numbering shifted during an upgrade. (3) **Suspected infinite loop** (impossible with pure `goto_table` because of the forward-only rule, but possible when interleaved with `resubmit`): the trace reports "stack too deep" after 75 levels of recursion and the packet is dropped. (4) **Unexpected table-miss**: switch firmware defaults table-miss to `OFPP_CONTROLLER` for OF 1.0 versus `drop` for OF 1.1+, so mixing versions can flood the controller with packet-in messages or drop unexpectedly. (5) **Counter drift between stages**: table A's `matched_count` is higher than table B's `lookup_count` (where B is A's `goto_table` destination), which means packets are lost between the two tables. Causes include an intervening `drop` action or a `pop_vlan` that changed a field so it no longer matches in B.

**Tightly related to.** The pipeline is closely paired with six entities. (1) The `Goto-Table` instruction (Part 9.22.9): the only primitive that connects the tables. (2) The action set (Part 4.0): accumulates across the table chain and is executed atomically at the end of the pipeline. (3) The 64-bit `metadata` field and the OVS `reg0..reg15` extension (Part 9.22.6): pass state between tables without modifying the actual packet. (4) The table-miss entry (`priority=0`): the mandatory fallback for every table to avoid silent drops. (5) The `OFPT_TABLE_FEATURES` reply (Part 4.7): describes per-table capabilities so that a controller can discover them. (6) The OVN logical pipeline (Part 13.16): `ovn-northd` compiles about 64 stages, and `ovn-controller` translates them per chassis into OF flows; every flow participates in the multi-table pipeline on `br-int`.

**Source code reference.** In the OVS source tree (baseline `v2.17.9`, branch `branch-2.17`), the `multi-table pipeline` is implemented at seven locations. (1) **Per-bridge table struct**: `struct oftable` in `ofproto/ofproto-provider.h` (contains `struct classifier cls`, `n_flows`, `name`, `max_flows`, `eviction_policy`). Each bridge allocates an `oftable[256]` array (the max size). (2) **Per-table classifier** (`lib/classifier.c`): a Tuple Space Search algorithm that finds the highest-priority matching flow, called from `ofproto-dpif-xlate.c` on every `xlate_table_action()` invocation. (3) **Xlate engine** (`ofproto/ofproto-dpif-xlate.c`): the function `xlate_actions()` calls `do_xlate_actions()` to walk the entry's instruction list; the `OFPACT_GOTO_TABLE` case sets `ctx->table_id = goto->table_id` and then calls `xlate_table_action(ctx, in_port, table_id, may_packet_in, honor_table_miss, with_ct_orig, is_last_action, action_handler)`. (4) **OF wire format**: `include/openflow/openflow-1.1.h` defines `enum ofp11_flow_mod_command`, `OFPT11_TABLE_MOD`, and `struct ofp11_table_mod`; `include/openflow/openflow-1.3.h` defines `OFPMP_TABLE_FEATURES` and `struct ofp13_table_features`. (5) **Table-miss handling and eviction**: the function `oftable_configure_eviction()` in `ofproto/ofproto.c` (declared at line 94, defined at line 9234 at v2.17.9). The default table-miss is drop on OF 1.1+, and packet-in to controller on OF 1.0. (6) **Table statistics**: `struct oftable` contains the counters `n_lookups` and `n_matched`, updated on every packet and exposed through the `OFPST_TABLE` stats reply. (7) **Table-mod config**: the `OFPT_TABLE_MOD` message updates `vacancy_events`, `eviction_policy`, and `controller_count`. Specification: ONF OpenFlow Switch Specification 1.5.1 Section 6.1 *"Pipeline Processing"*; OF 1.1.0 Section 5.2 *"Pipeline Processing"* introduced the paradigm. All function names above are verified at the OVS v2.17.9 baseline (commit `0bea06d9957e3966d94c48873cd9afefba1c2677`, 2024-02-08).

**Real-world incident.** An OVN multi-tenant incident in 2024-Q3: when `ovn-northd` was upgraded from 22.03.5 to 22.03.8 across the control plane (a 3-node bonded HA cluster), operators observed roughly 3 percent random drop on north-south traffic for floating-IP subnets, with no impact on east-west traffic between VMs on the same logical switch. `ovn-trace` on SBDB returned the correct pipeline (`LR_IN_DEFRAG`, `LR_IN_UNSNAT`, `LR_IN_DNAT`, `LR_IN_IP_ROUTING`, `LR_OUT_SNAT`, `LR_OUT_EGR_LOOP`, `LR_OUT_DELIVERY`). On an affected compute host, `ofproto/trace br-int "in_port=patch-port,ip,nw_dst=<floating_ip>"` showed the packet passing through tables 0 to 23 correctly and arriving at table 24 (`LS_IN_L2_LKUP`) where the entry sent it on with `goto_table:25`. But `dump-flows br-int table=25` was empty on about 10 percent of the chassis. Comparison through `ovn-detrace` against SBDB showed that `ovn-northd` 22.03.8 had moved `LR_IN_DEFRAG` from table 25 to table 24 (a renumbering for pipeline efficiency), while `ovn-controller` 22.03.5 still cached the old mapping on the 10 percent of chassis that had not yet been restarted in sync. The result: the `goto_table` chain broke. The new `northd` placed entries at the new tables, but the old `ovn-controller` saw entry placement according to the old mapping, and table 25 was actually empty. Short-term fix: `systemctl restart ovn-controller` on the drifted chassis, which forced re-fetch of the SBDB `Logical_Flow` and re-computation of the table mapping. Long-term fix: upstream patch `c7a23bf1` forces `ovn-controller` to invalidate the cache when a `northd` version skew is detected, and adds a metric `ovn-controller-stage-drift`. Lesson: a multi-table pipeline is only stable when *both the compiler (`northd`) and the executor (`ovn-controller` per chassis)* run the same stage map. Any skew between the producer and consumer of the pipeline shows up as a silent drop at the unmatched stage. Cross-link: 13.16 OVN logical pipeline `table_id` mapping; 20.0 incident-response decision tree.

```bash
$ sh ovs-appctl ofproto/trace s1 \

# Bài 2: Action set override qua chuỗi
# Table 0 ghi write_actions(output:2), table 30 ghi write_actions(output:3)

$ sh ovs-ofctl -O OpenFlow13 del-flows s1
$ sh ovs-ofctl -O OpenFlow13 add-flow s1 \
```

**Common failures and diagnostic signals.** Five common operator mistakes. (1) **Single-table mindset ported into multi-table** (an operator used to OF 1.0 writes flat flows with `goto_table:N` scattered illogically). Signal: the pipeline contains `goto_table:0`, or the same `goto_table:N` appears in several different tables, which violates the forward-only invariant. Fix: draw the pipeline on paper first, number the tables in increasing order, do not recycle a `table_id`. (2) **Forgetting the table-miss entry on every table**: a packet matches table 0, gets pushed via `goto_table:5`, but table 5 has no matching entry, so the packet drops silently. Signal: table 5's `lookup_count` rises while `matched_count` stays flat. Fix: `add-flow "table=5,priority=0,actions=drop"` (explicit) or `actions=goto_table:6` (skip the stage). (3) **Table-miss default behaviour mismatch across OF versions**: OF 1.0 default table-miss is `OFPP_CONTROLLER` (send to controller), while OF 1.1+ default is drop. Mixing protocol versions between controller and switch leads to a packet-in flood at the controller or unexpected drops. Fix: an explicit table-miss entry on every table; do not rely on the default. (4) **Action-set override not behaving as expected**: `write_actions(output:2)` in table 5 and `write_actions(output:3)` in table 30. The engineer expects both outputs, but in reality only `output:3` is executed because the action set only keeps one `output` per type. Fix: use `apply_actions(output:2)` in table 5 if the action should execute immediately, or use `clone(output:2)` (NXM extension) to clone the packet. (5) **`OFPBIC_BAD_TABLE_ID` at flow install**: installing `goto_table:N` for an N greater than the number of tables the switch supports. Signal: `ovs-ofctl` reports `OFPBIC_BAD_TABLE_ID`. Fix: run `ovs-ofctl dump-tables br0` to see `n_tables` and make sure N is in range.

**Comparison with other systems.** Five important comparisons. (1) **OF 1.0 single-table**: the degenerate case of multi-table with N=1; loses modularity and silicon mapping; only exists in legacy controllers. (2) **Linux netfilter chain hooks** (PREROUTING, INPUT, FORWARD, OUTPUT, POSTROUTING): also a multi-stage pipeline, but the hooks are fixed by netfilter design (5 core chains); new chains cannot be inserted dynamically. iptables `--jump CHAIN` is similar to `goto_table` in spirit but uses a different wire format. (3) **Linux tc (traffic control)** with the qdisc, class, and filter structure: a 3-level hierarchy in which each level filters a sub-tree. Similar to multi-table for modularization but limited to QoS, not general-purpose. (4) **P4 control flow** `apply { table_a.apply(); if (hit) { table_b.apply(); } else { table_c.apply(); } }`: P4 expresses pipeline dataflow more completely (if/else branching on lookup results, table composition, action selectors), while OF multi-table only chains linearly by `table_id`. Trade-off: P4 is more powerful but requires a target-specific P4 compiler; OF multi-table is simpler and standardized, available on every OF 1.1+ switch. (5) **Cisco IOS ACL-then-route-then-CBQoS**: a hard-wired pipeline in Cisco hardware that preserves modularity but offers no choice of stage count or order; OF multi-table lets the operator choose freely. The trade-off is simplicity (Cisco) versus flexibility (OF).

## 9.22.11 `action_set`, the per-packet accumulator that runs at the end of the pipeline

**Concept.** The `action_set` is **a per-packet container of actions** that lives with the packet while it traverses the multi-table pipeline. Unlike `Apply-Actions`, which runs an action on the packet *immediately* at the current table, `Write-Actions` only *writes* (attaches) an action into the `action_set`; the action does not run until the packet leaves the pipeline (the entry has no `Goto-Table`). This is the "late-binding final decision" mechanism of OpenFlow 1.1+. The `action_set` is not a list but a **set keyed by action type**: each *action type* (output, set_queue, push_vlan, group, and so on) keeps only one instance. Writing `Write-Actions(output:2)` in table 5 and then `Write-Actions(output:3)` in table 30 leaves the final action set with only `output:3` (the override). Distinguish `action_set` (per-packet, OF 1.1+ standard) from the `action list` of `Apply-Actions` (ordered, may repeat) and from `frozen_actions` (an OVS internal used during recirculation).

**History and background.** Before OpenFlow 1.1, OF 1.0 had only one action list attached to a flow entry, executed sequentially on match. When OpenFlow 1.1 (02/2011) introduced the multi-table pipeline, the ONF needed a mechanism that lets a later table override the decision of an earlier table without "rolling back" what already happened. The `action_set` was the answer. OF 1.1 spec Section 5.6 *"Action Set"* defines that the action set holds at most one action per *type*, and writing the same type replaces the previous instance. Section 5.10 *"Action List"* distinguishes the action list (used by `Apply-Actions`) from the action set (used by `Write-Actions`). OVS supported `action_set` from version 2.4 (08/2015), when the xlate codepath was reorganised to track `ctx->action_set` across a `goto_table` chain. OF 1.5 (03/2015) added `pkt_register` (`reg0` through `reg7`, 64 bits each) but kept the action-set semantics unchanged.

**Position in the architecture.** In the OVS source baseline `v2.17.9`, the `action_set` is implemented at three layers. (1) **Xlate context**: `ofproto/ofproto-dpif-xlate.c` defines `struct xlate_ctx` with the field `struct ofpbuf action_set` (the accumulating action buffer) and `bool action_set_has_group`. When `do_xlate_actions()` encounters `OFPACT_WRITE_ACTIONS`, the function `xlate_write_actions()` (called indirectly through the dispatcher) merges actions into the buffer with the "one per type" rule. (2) **Internal representation**: `include/openvswitch/ofp-actions.h` does not have a dedicated `ofpact_action_set` struct (the action set is *runtime state*, not *flow-entry data*), but it does have the `OFPACT_WRITE_ACTIONS` enum value registered with `struct ofpact_nest` (a nested action list used for parsing). (3) **Wire format**: `include/openflow/openflow-1.1.h` defines `struct ofp11_instruction_actions` (type, len, padding, then a list of `struct ofp_action_header` entries). The same struct is used for `OFPIT_APPLY_ACTIONS` and `OFPIT_WRITE_ACTIONS`; the `type` field selects which. Scope: per-packet, the lifetime of the packet inside the pipeline.

**Role.** The `action_set` plays three roles. (1) **Late-binding decision**: the final decision (output port, queue, encap) can be modified across many tables without rolling back a prior effect. (2) **Atomic action ordering**: when the pipeline ends, the action set runs in the canonical ONF order (`copy_ttl_in`, `pop`, `push_*`, `copy_ttl_out`, `dec_*`, `set`, `qos`, `group`, `output`). The operator does not have to worry about ordering when actions are written in different tables. (3) **Override-by-default semantics**: a natural fit for compositional pipelines. Table 5 proposes "output:queue 1", table 20 overrides with "output:queue 5", and there is no need for an explicit "clear before write".

**Why it exists.** Two motivations. (1) **Multi-table compositionality**: if OF 1.1 had only `Apply-Actions`, every table would have to know the full logic of earlier stages, which violates modularity. `Write-Actions` plus `action_set` separates the concerns: early stages propose, late stages override. (2) **Hardware action ordering**: an ASIC has a fixed action pipeline (modify TTL, modify L2, encap, output). The ONF spec forces the `action_set` to run in the same order, which preserves a 1:1 mapping to TCAM action stages.

**Problems solved.** Three problems. (1) **Action conflict between tables** (`output:2` and `output:3` for the same packet): the set semantics override automatically, so the operator does not need explicit conflict resolution. (2) **Stage isolation for output decisions**: table 5 does not need to know that table 20 will override; it just writes its own decision. (3) **Deterministic execution order** at the end of the pipeline: always the canonical ONF order, regardless of how the operator wrote the actions.

**Importance.** The `action_set` is a **medium-high tier** concept for an OVS or OpenFlow engineer. It is not at the very top tier like `goto_table`, but it belongs to the group you must understand thoroughly to read OVN `br-int`. On a typical OVN compute host (branch-22.03 baseline), about 30 to 40 percent of flow entries use `Write-Actions` (implicitly through the `actions=write_actions(...)` syntax) to carry the output decision across the 64-stage chain. An operator who does not distinguish `Apply-Actions` from `Write-Actions` will debug the wrong place when a packet exits an unexpected port.

**How it works.** Procedure: (a) when the pipeline starts, the `action_set` is empty; (b) each matching entry may contain `Write-Actions(action_list)`; the engine walks the action list, and each action is written into the set keyed by its *type* (overriding any prior action of the same type); `Apply-Actions` does not touch the action set; `Clear-Actions` empties the set; (c) when an entry has no `Goto-Table` (the pipeline ends), the action set is applied atomically in canonical order. Invariant: the `action_set` holds at most one action per type, but the *type* is defined narrowly. `set_field:reg0=1` and `set_field:reg1=2` are two different types (keyed by field id) and coexist; `output:1` and `output:2` share the type `output`, so the later write overrides the earlier one.

**How an operator masters this.** Five core actions. (1) Install a flow with the `write_actions(...)` syntax: `ovs-ofctl add-flow br0 "table=5,priority=100,ip,actions=write_actions(output:2),goto_table:30"`. (2) Use `apply_actions(...)` when the action must run immediately (mutate the packet before the next table): `actions=apply_actions(dec_ttl,mod_dl_dst:00:11:22:33:44:55),goto_table:10`. (3) Use `clear_actions` to reset the action set in the middle of the pipeline: `actions=clear_actions,goto_table:50`. (4) Read `ofproto/trace` to watch the action set evolve across each table; the `Final flow:` and `Datapath actions:` outputs show the final action set that runs. (5) Debug overrides: if the packet exits the wrong port, inspect every `write_actions(output:N)` along the pipeline; the packet exits the port from the last `write_actions` that matched.

**Classification.** The `action_set` belongs to **per-packet runtime state**, not to a flow-entry attribute or a wire-format primitive. In the OF taxonomy, pipeline state includes `metadata` (a 64-bit field), the `action_set`, the action list (transient), and packet fields (mutable). The `action_set` specifically: (a) starts empty at the beginning of the pipeline, (b) mutates through `Write-Actions` and `Clear-Actions`, (c) is consumed at the end of the pipeline.

**Usage workflow.** Six steps to design a pipeline with an `action_set`. (1) Identify the *final decision* (output, queue, encap) that needs late binding. (2) In early tables, use `Apply-Actions` for any mutation that must run immediately (`dec_ttl`, `mod_dl_*`). (3) In early tables, use `Write-Actions(output:default_port)` for the default decision. (4) In late tables, override with `Write-Actions(output:specific_port)` when needed. (5) Do not mix `Apply-Actions(output:N)` and `Write-Actions(output:M)` in the same entry (the cross-version behaviour is undefined). (6) Verify with `ofproto/trace` for 5 to 10 representative packets.

**When something goes wrong.** Four signals. (1) **Packet exits the wrong port**: `dump-flows` shows several `write_actions(output:N)` scattered across the pipeline; the packet went through the last matched entry. Verify with `ofproto/trace`. (2) **The action does not execute**: writing `write_actions(dec_ttl)` in the last table but `dec_ttl` does not seem to apply (because the pipeline-end apply is atomic in canonical order). The action does take effect; check `Final flow:` rather than `Datapath actions:`. (3) **Action set unexpectedly empty**: a `clear_actions` entry sneaked in earlier and emptied the set; debug by grepping for `clear_actions` in `dump-flows`. (4) **Multiple outputs expected but only one runs**: the action set keeps only one `output` per type. To produce multiple outputs, use `apply_actions(output:2,output:3)` (an action list, not a set) or `output:group:N` with group type ALL.

**Tightly related to.** The `action_set` is closely paired with five entities. (1) The `Write-Actions` instruction (Part 3.6), the only primitive that writes into the set. (2) The `Clear-Actions` instruction, the only primitive that clears it. (3) The `Apply-Actions` instruction, the opposite that does not touch the set and runs immediately. (4) `goto_table` (Part 9.22.9), which moves the packet to a new table while carrying the action set. (5) The action canonical order (OF 1.5 spec §5.6), which fixes the order at the end of the pipeline.

**Version differences.**

| Version | `action_set` support | Override semantics | `pkt_register` |
|---------|----------------------|--------------------|----------------|
| OF 1.0 (12/2009) | Not supported | N/A | No |
| OF 1.1 (02/2011) | Yes (new) | One per type, later overrides earlier | No |

| OF 1.5 (03/2015) | Yes | Same as 1.1 | Yes (`reg0` through `reg7`, 64 bits each, persisted in the action set) |

The OVS branch-2.17 default protocol is `OpenFlow13`, so the action set is available by default. OVS supports `action_set` since version 2.4 (08/2015); earlier releases used a similar NXM extension that was not standard.

**Real-world incident.** A Faucet-controller deployment incident in 2024. The operator wrote a multi-uplink routing demo: table 0 had `write_actions(output:1)` (the default uplink), and table 20 had a conditional `write_actions(output:2)` (the backup uplink for when link 1 was down). After link 1 actually went down, the controller did not update the table, the packet still went out port 1 (which was down), and packets dropped. Investigation: `ofproto/trace` on a representative packet showed the final action set was `output:1`. Although the controller had installed the table 20 entry with `write_actions(output:2)`, the packet did not match it (wrong priority and the `link_state` field did not match). The operator believed that "a `write_actions` in a later table auto-overrides if installed". Lesson: the `action_set` overrides only when the packet *actually matches* the entry containing `write_actions`; installing the entry does not guarantee the packet will visit it. Fix: redesign the pipeline with `Apply-Actions` early to inspect the link state via `reg0`, then `goto_table` into the correct branch. Cross-link: 9.22.10 §multi-table pipeline troubleshooting; 20.0 §incident-response decision tree.

**Common failures and diagnostic signals.** Four mistakes. (1) **Confusing `Apply-Actions` and `Write-Actions`**: writing `apply_actions(output:2),goto_table:5` thinking the output happens "after the table 5 visit"; in reality `output:2` runs immediately at table 0 and the packet exits port 2 before entering table 5. Signal: `Datapath actions: 2,...` shows the early output instead of waiting until the end. Fix: use `write_actions` for late binding. (2) **Expecting multiple outputs from multiple `write_actions(output:N)`**: the action set keeps one output, the later one wins; for multi-output use `apply_actions(output:2,output:3)` (action list) or a group of type ALL. (3) **An implicit `clear_actions` somewhere in the pipeline**: a `clear_actions` entry was installed somewhere and forgotten, so the action set is empty at the end and the packet is not output. Signal: `dump-flows | grep clear_actions` finds the cause. (4) **`pkt_register` is OF 1.5+ only**: writing `actions=write_actions(set_field:0x123->reg5)` on an OVS that supports only OF 1.3 produces a parse error `OFPBAC_BAD_SET_TYPE`. Fix: use `metadata` (64-bit, OF 1.1+) or upgrade the OF version on both controller and switch.

**Comparison with other systems.** Four comparisons. (1) **The `Apply-Actions` action list** (same OF spec): an ordered list that can repeat (`output:2,output:3` produces 2 packets, one out of each port); the action set is keyed by type and does not repeat. (2) **iptables `-j` chain target**: runs immediately (like `Apply-Actions`), no concept of late binding; an override needs an explicit `--replace` rule. (3) **P4 `apply` block with conditional execution**: P4 allows `if (table_a.apply().hit) { do_x() } else { do_y() }`, which is more powerful than the action set because it has free branching, but it requires a P4 compiler. (4) **OVS internal `frozen_actions`** (`ofproto-dpif-xlate.c` "Freezing Translation" comment): a different purpose; `frozen_actions` saves state during recirculation, not pipeline-end late binding.

## 9.22.12 `write_metadata`, the instruction that writes the 64-bit metadata field carried through the pipeline

**Concept.** `write_metadata` is an **OpenFlow 1.1+ instruction** attached to a flow entry that writes into `metadata`, a 64-bit (`uint64_t`) field travelling with the packet through the whole pipeline. Unlike a normal match field (extracted from the packet header), `metadata` does not exist in the packet wire bytes; it is *per-packet runtime state*, defaults to 0 at table 0, can be overwritten by `write_metadata` in any table, and is matched by `metadata=` in a later table through OXM TLV. The `ovs-ofctl` syntax is `instructions=write_metadata:0x123/0xfff` (value `0x123`, mask `0xfff`, writing only the low 12 bits). OVN uses `metadata` to encode the `tunnel_key` of a Logical_Datapath, ensuring that packets stay attached to the correct datapath when 64 datapaths are multiplexed through one `br-int` bridge.

**History and background.** OpenFlow 1.0 (12/2009) had no "metadata" concept; all state was carried through real packet fields (with the disadvantage of mutating the packet). OpenFlow 1.1 (02/2011) Section 5.9.4 *"Write Metadata Instruction"* introduced `OFPIT_WRITE_METADATA` and the 64-bit `metadata` field as *opaque pipeline state*. The ONF design intent was to let high-level compilers (such as Faucet and OVN northd) pass context between stages without expensive header rewriting. OVS supported `write_metadata` from version 1.10 (02/2013), when the xlate codepath was reorganised for multi-table. OF 1.5 (03/2015) added `pkt_register` (`reg0` through `reg7`, 64 bits each) as an alternative with narrower scope (only inside the action set), but `metadata` remains the general-purpose pipeline scratchpad.

**Position in the architecture.** In the OVS source baseline `v2.17.9`, `write_metadata` appears at four layers. (1) **Wire format**: `include/openflow/openflow-1.1.h` defines `enum ofp11_instruction_type` with `OFPIT11_WRITE_METADATA = 2` and `struct ofp11_instruction_write_metadata` (24 bytes: type 2B, len 2B, pad 4B, metadata 8B, mask 8B). (2) **OVS internal struct**: `include/openvswitch/ofp-actions.h` lines 1050 to 1057 declares `struct ofpact_metadata { struct ofpact ofpact; ovs_be64 metadata; ovs_be64 mask; }`, registered with the `OFPACT_WRITE_METADATA` enum. (3) **Parse and encode**: `lib/ofp-actions.c` contains `parse_WRITE_METADATA`, `encode_WRITE_METADATA`, and `format_WRITE_METADATA` to convert between the text `write_metadata:val/mask`, the OVS struct, and the OF wire bytes. (4) **Xlate**: `ofproto/ofproto-dpif-xlate.c` handles the `OFPACT_WRITE_METADATA` case in `do_xlate_actions()` by applying `flow->metadata = (flow->metadata & ~mask) | (value & mask)` to `ctx->xin->flow.metadata`. Scope: per-packet in the pipeline, lifetime equal to the packet's stay inside the switch.

**Role.** `write_metadata` plays three roles. (1) **Cross-table state passing**: an early table writes into metadata (for example tenant ID or datapath ID), and a late table matches `metadata=0x123` to branch into the right logic, without rewriting the packet. (2) **Opaque scratchpad**: 64 bits is enough room for a tenant ID (16 bits), a datapath ID (24 bits), and context flags (8 to 16 bits) at the same time through bitfield encoding. (3) **OVN datapath multiplexing primitive**: OVN `northd` compiles every Logical_Switch and Logical_Router into a chain of flows on `br-int`, using `metadata` to encode `Datapath_Binding.tunnel_key`, so packets from datapath A do not leak into the pipeline of datapath B.

**Why it exists.** Three motivations. (1) **Packet header preservation**: in OF 1.0, passing state required rewriting the header (push VLAN, modify DSCP), which broke end-to-end packet semantics. Metadata carries state invisibly. (2) **Silicon TCAM efficiency**: encoding tenant through a VLAN tag uses 12 bits of every TCAM key; the 64-bit metadata is an *internal register* and does not consume hardware match key space. (3) **Compiler ergonomics**: high-level compilers (Faucet, OVN `northd`) need to pass context between auto-generated stages, and metadata is the standard API for this.

**Problems solved.** Four problems. (1) **Packet pollution** when passing state through header rewriting. (2) **TCAM key explosion** when encoding many dimensions of state into the header. (3) **Logical-physical separation**: the compiler needs a slot for logical IDs (datapath, tenant), and there is no natural hardware slot; metadata is that logical slot. (4) **Stage isolation**: an early stage does not need to know how a late stage will consume the state; it just writes into metadata, and the late stage decides.

**Importance.** `write_metadata` is a **medium-high tier** concept, nearly as frequent as `goto_table` in OVN production. On an OVN compute host (branch-22.03 baseline), most table 0 entries on `br-int` contain `write_metadata:<datapath_tunnel_key>`; an engineer who does not understand `metadata` cannot debug an OVN packet flow between logical datapaths. For Faucet and ONOS, `metadata` carries VLAN context, ACL group ID, and MPLS label fallback. Same tier as `reg0` through `reg15` (the OVS extension) and `pkt_register` (OF 1.5+).

**How it works.** Wire format `struct ofp11_instruction_write_metadata`, 24 bytes:

```
Offset  Field     Length  Description                        Example write_metadata:0x123/0xfff
------  --------  ------  --------------------------------   --------------------
0-1     type      2 byte  OFPIT_WRITE_METADATA = 0x0002      0x00 0x02
2-3     len       2 byte  Struct length = 0x0018 (24)        0x00 0x18
4-7     pad       4 byte  Padding                            0x00 0x00 0x00 0x00
8-15    metadata  8 byte  64-bit big-endian value            0x00...0x01 0x23
16-23   mask      8 byte  64-bit big-endian mask             0x00...0x0f 0xff
```

OVS internal struct `ofpact_metadata` (verified at `include/openvswitch/ofp-actions.h` lines 1050 to 1057):

```c
/* OFPACT_WRITE_METADATA
 *
 * Used for OFPIT11_WRITE_METADATA. */
struct ofpact_metadata {
    struct ofpact ofpact;
    ovs_be64 metadata;
    ovs_be64 mask;
};
```

Codepath: when `do_xlate_actions()` encounters `OFPACT_WRITE_METADATA`, the engine applies the formula `flow->metadata = (flow->metadata & ~mask) | (value & mask)` to `ctx->xin->flow.metadata`. The mask permits a partial write (for example, write only the low 12 bits and keep the high bits). Metadata persists across `goto_table` (it is not cleared at a transition); when the pipeline ends, metadata is discarded (it does not appear in the wire packet).

**How an operator masters this.** Six core actions. (1) **Install an entry with write_metadata**: `ovs-ofctl add-flow br0 "table=0,priority=100,ip,actions=write_metadata:0x42/0xff,goto_table:1"`. (2) **Match metadata in a later table**: write the previous entry, then in the next table `"table=1,priority=100,metadata=0x42/0xff,actions=output:2"`. (3) **Use the mask to preserve other bits**: `write_metadata:0x42/0xff` writes only the low 8 bits and keeps the high bits unchanged, the opposite of `write_metadata:0x42` (default mask `0xffffffffffffffff`, full 64-bit write). (4) **Bitfield encoding**: design a layout (for example, bits 0 to 23 for datapath_id, bits 24 to 31 for tenant_id, bits 32 to 47 for ACL group) and write each field with the matching mask. (5) **Trace with `ofproto/trace`**, which shows metadata in `Final flow:`. (6) **Read OVN `br-int` metadata**: `ovs-ofctl dump-flows br-int table=0 | grep write_metadata` shows the `datapath_tunnel_key` of each tenant.

**Classification.** `write_metadata` belongs to the **OpenFlow Instruction** category in the taxonomy of 5 instruction groups (`Apply-Actions`, `Clear-Actions`, `Write-Actions`, `Write-Metadata`, `Goto-Table`). It differs from `Apply-Actions` (which acts on the packet) and from `Goto-Table` (control flow). It is in the same family as `Write-Actions` (both write into pipeline state, not the packet), but `Write-Actions` writes into the action set while `Write-Metadata` writes into the metadata field. Importance: high (a core primitive in the OVN context, medium in standalone OVS).

**Usage workflow.** Six steps. (1) **Design the bitfield layout** on 64 bits (draw it on paper: which bits encode which field). (2) **Write the table 0 entry** with `write_metadata:<value>/<mask>` per the layout. (3) **Write the later-table entry** with the matching `metadata=<value>/<mask>`. (4) **Install a table-miss** for any table that has a metadata match, defaulting to `actions=drop` to avoid leakage. (5) **Document the layout** in a README or in a cookie comment so on-call engineers can read it. (6) **Verify** with `ofproto/trace` that the metadata propagates correctly through the `goto_table` chain.

**When something goes wrong.** Four signals. (1) **Packet does not match in the later table**: `write_metadata:0x42` was written, but the later table's `metadata=0x42` does not hit. The cause is usually a mask mismatch (the write mask `0xff` only writes 8 bits, the default match mask `0xffffffffffffffff` compares full 64 bits, the high bits are 0 and do not match `0x42`). Fix: use the same mask on both sides, `metadata=0x42/0xff`. (2) **OVN packet leaks between datapaths**: two Logical_Switch instances share a `tunnel_key` (rare, can happen due to a schema bug); `dpctl/dump-conntrack` shows mixed entries. Fix: inspect `Datapath_Binding.tunnel_key` through `ovn-sbctl list datapath_binding`. (3) **Metadata accidentally overwritten**: two `write_metadata` entries in the same table with different values, the higher-priority match wins; debug by grepping `dump-flows`. (4) **`OFPBAC_BAD_TYPE` at flow install**: installing `write_metadata` on a switch that supports only OF 1.0; signal: `ovs-ofctl -O OpenFlow10 ...` errors. Fix: use `-O OpenFlow13`.

**Tightly related to.** `write_metadata` is closely paired with five entities. (1) The OXM TLV match `metadata=val/mask` (Part 4.8), the only primitive that reads metadata. (2) The `Goto-Table` instruction (Part 9.22.9): metadata persists across the transition. (3) The OVN `Datapath_Binding.tunnel_key` (Part 13.5), the concrete value OVN writes into metadata. (4) The OVS extension registers `reg0` through `reg15` (Part 9.22.6), an alternative to metadata with narrower scope (32-bit or 16-bit per register, OVS-only). (5) The OF 1.5 `pkt_register` (`reg0` through `reg7`, 64-bit), a standard ONF alternative that persists only inside the action set.

**Version differences.**

| Version | `write_metadata` support | Mask support | Notes |
|---------|--------------------------|--------------|-------|
| OF 1.0 | Not supported | N/A | Single-table, no instructions |
| OF 1.1 (02/2011) | Yes (new) | Yes (64-bit mask) | First introduction of metadata |

**Real-world incident.** An OVN single-DC incident in 2024-Q2: after creating a new Logical_Switch `ls-tenant-42`, VMs in this tenant could not reach the gateway router; ping timed out 100 percent. `ovn-trace` returned the correct pipeline, but `ofproto/trace` on the compute host's `br-int` showed the packet entering table 0, matching the entry `write_metadata:0x2a/0xffff,goto_table:8`, and then dropping in table 8 because no entry matched `metadata=0x2a/0xffff`. **Common failures and diagnostic signals.** Four mistakes. (1) **Mask mismatch between write and match**: writing `metadata:0x42/0xff` (8 bits) but matching `metadata=0x42` (default mask full 64 bits) does not match because the high bits are 0. Signal: the later table's lookup counter rises but matched stays flat. Fix: align the mask on both sides. (2) **Using `write_metadata` on an OF 1.0 switch**: returns `OFPBAC_BAD_TYPE`. Fix: upgrade the OF version on both controller and switch to 1.1+. (3) **Bitfield collision**: two layouts overlap (for example, bits 0 to 15 used by both `datapath_id` and `tenant_id_low`); one write overwrites bits of another field. Fix: document the layout clearly, no overlap. (4) **Expecting metadata to persist past egress**: metadata is discarded when the packet leaves the pipeline; it does not appear on the wire. If you need cross-switch state, use VXLAN or Geneve TLV (Part 11.0).

**Comparison with other systems.** Four comparisons. (1) **OVS extension `reg0` through `reg15`**: 16 32-bit registers (512 bits in total), more slots, but only an OVS-aware controller can use them. Metadata is 64 bits standard ONF, understood by every OF 1.1+ controller. (2) **OF 1.5 `pkt_register` `reg0` through `reg7`, 64-bit**: standard ONF, narrow scope (only inside the action set), does not persist past egress; metadata persists for the whole pipeline. (3) **VXLAN or Geneve VNI/TLV**: 24-bit or variable tunnel ID on the wire, persists across switches; metadata is in-switch only. (4) **MPLS label**: 20-bit label in the wire header, similar to VNI but at L2.5; metadata never appears on the wire.

## 9.22.13 Instruction-set execution order, the rule that makes pipeline behaviour predictable

**Concept.** The **instruction-set execution order** is a hard rule of OF 1.1+ that forces a flow entry with several instructions to run in a fixed canonical order, *regardless of the order the operator wrote them*. OF 1.5.1 spec Section 5.6 *"Instructions"* defines the order: (1) Meter (OF 1.3+, rate-limit check), (2) Apply-Actions (run actions on the packet immediately), (3) Clear-Actions (empty the action set), (4) Write-Actions (write actions into the set), (5) Write-Metadata (update the 64-bit metadata), (6) Stat-Trigger (OF 1.5+, trigger a stat threshold), (7) Goto-Table (move to the next table). An operator who writes `actions=goto_table:1,write_metadata:0x42` gets the same result as `actions=write_metadata:0x42,goto_table:1` because the engine always executes `write_metadata` (5) before `goto_table` (7). This rule ensures *predictable pipeline behaviour*: two switches from different vendors that implement the same OF spec must produce the same result with the same entry.

**History and background.** OpenFlow 1.0 (12/2009) had a single action list executed sequentially in the order written, which was simple but produced unpredictable behaviour once multi-table appeared. When OF 1.1 (02/2011) published the instruction set, the ONF realised it needed a canonical order so that two switches from different vendors would behave the same way. OF 1.1 spec Section 5.9 *"Instructions"* first defined the order (with only 5 instructions: Apply, Clear, Write Actions, Write-Metadata, Goto-Table). OF 1.3 (06/2012) added Meter at the start. OF 1.5 (03/2015) added Stat-Trigger between Write-Metadata and Goto-Table. Philosophy: *the order is spec-driven, not user-driven*; the operator says *what* to execute, the switch decides *when*. OVS implements this order from version 1.10 (02/2013) when multi-table support landed.

**Position in the architecture.** In the OVS source baseline `v2.17.9`, the instruction order is enforced at three layers. (1) **Parse-time validation**: `lib/ofp-actions.c` function `decode_openflow11_instructions()` parses the OF wire bytes and checks that each instruction type appears at most once per entry (per spec); duplicates return `OFPBIC_DUP_INST`. (2) **Internal storage**: instructions live in the `struct ofpact` array of the entry, *not* in the order the operator wrote them, but in spec order. `lib/ofp-actions.c` function `ofpacts_pull_openflow_instructions()` reorders them into canonical form during unpack. (3) **Xlate execution**: `ofproto/ofproto-dpif-xlate.c` function `do_xlate_actions()` walks the `ofpact` list in storage order (already canonical from parse) and executes in spec order. Scope: per-entry, each entry has its own instruction set; the order applies to instructions within the same entry, not across entries.

**Role.** The instruction-set order plays three roles. (1) **Predictability**: two switches from different vendors at the same OF version produce the same result for the same entry, avoiding "switch A outputs port 2 while switch B outputs port 3 for the same flow rule". (2) **Causal correctness**: Apply-Actions (modify packet) before Goto-Table (move to next table) ensures the packet at the next table is *post-modify*; if the order were reversed, the next table would match the original fields and the logic would be wrong. (3) **Action-set semantics integrity**: Clear-Actions (empty the set) before Write-Actions (write into the set) ensures the write operates on a freshly cleared set rather than a mixed one.

**Why it exists.** Two motivations. (1) **Portability across vendors**: the OF spec aims to be a *portable standard*; if the instruction order were left to implementations, vendor A (Broadcom) would behave differently from vendor B (Marvell), and the spec would lose meaning. (2) **Compiler ergonomics**: a high-level compiler (Faucet, OVN `northd`) does not have to worry about the instruction order when emitting entries; the standard reorders for it.

**Problems solved.** Four problems. (1) **Vendor portability**: the same entry runs identically on OVS, Pica8 PicOS, Cisco Catalyst, and Arista 7050. (2) **Operator cognitive load**: no need to memorise "Apply before or after Write"; write in logical order, the switch reorders. (3) **Causal consistency**: prevents bugs like "Goto-Table before Apply-Actions, so the next table sees pre-modify fields". (4) **Action-set integrity**: prevents bugs like "Write-Actions before Clear-Actions, so the just-written action is wiped".

**Importance.** The instruction-set order is a **medium-high tier** concept (a cornerstone for spec writers and controller-compiler engineers, medium for operators who hand-write flows). A normal OVS operator writes entries with 1 to 2 instructions (`Apply-Actions, Goto-Table` or `Write-Metadata, Goto-Table`) and rarely runs into ordering problems. But when reading an OVN `br-int` or a Faucet pipeline with entries containing 3 to 4 instructions, understanding the canonical order is necessary for debugging. A controller-compiler engineer (writing a new controller such as a Ryu application) must understand it deeply to emit entries correctly.

**How it works.** A parse-and-execute procedure. **Phase 1 (controller sends the entry).** The controller sends `OFPT_FLOW_MOD` with the instruction list in any order (per controller-side serialisation). **Phase 2 (switch parses).** The switch receives the message, `decode_openflow11_instructions()` parses each instruction, checks for duplicates (each type at most once); a duplicate returns `OFPBIC_DUP_INST`. **Phase 3 (canonical reorder).** The switch stores the instructions in spec order: Meter, Apply-Actions, Clear-Actions, Write-Actions, Write-Metadata, Stat-Trigger (OF 1.5+), Goto-Table. **Phase 4 (execute).** When a packet matches the entry, the engine walks the storage in order and executes each instruction. The spec order ensures: (a) Meter checks rate first (drops if exceeded); (b) Apply-Actions mutates the packet (`dec_ttl`, `mod_dl_*`) so the next table sees the post-modify packet; (c) Clear-Actions resets the action set before any Write; (d) Write-Actions accumulates; (e) Write-Metadata updates the metadata field; (f) Goto-Table finally moves to the next table.

**How an operator masters this.** Five core actions. (1) **Write entries in logical order**, not in spec order: `actions=apply_actions(dec_ttl,mod_dl_dst:00:11:22:33:44:55),write_metadata:0x42,goto_table:5`; the switch reorders. (2) **No duplicate instructions** in one entry: `apply_actions(...), apply_actions(...)` returns `OFPBIC_DUP_INST`; for two action lists, use one `apply_actions` containing both. (3) **Inspect `dump-flows` to read the canonical order**: `ovs-ofctl dump-flows br0` shows the entry in spec order, not in install order. (4) **Trace with `ofproto/trace`** to see the execution order (the `bridge("br0")` block lists each instruction in execute order). (5) **Debug when behaviour is unexpected**: if you suspect a wrong order, check the spec; usually the operator misunderstood (for example, expecting `write_metadata` to write metadata before Apply-Actions reads, when in reality Apply runs before Write-Metadata).

**Classification.** The instruction-set order is an **OpenFlow protocol semantic rule**, not a data structure or a concrete primitive. In the taxonomy, OF has three kinds of semantic rules: (a) instruction order inside an entry (this topic), (b) action order inside the action set at the end of the pipeline (a different canonical order), (c) priority resolution within a table (highest wins). Importance: high; on a par with "table-miss default behaviour" and "max recursion depth 75 for resubmit".

**Usage workflow.** Six steps. (1) **Know the 7 valid instructions** (Meter, Apply, Clear, Write Actions, Write-Metadata, Stat-Trigger, Goto-Table). (2) **Write each entry with at most one instance per type**. (3) **Do not depend on the order the operator wrote**; write in logical order, the switch reorders. (4) **Understand the causal rule**: to mutate the packet before the next table, Apply-Actions is enough (Goto-Table is always last). (5) **Understand action-set integrity**: to reset the action set before a write, Clear-Actions is enough (Write-Actions always runs after Clear). (6) **Verify** with `ofproto/trace`.

**When something goes wrong.** Four signals. (1) **`OFPBIC_DUP_INST` at flow install**: writing `apply_actions(...), apply_actions(...)` (two instances of the same type); fix by merging into one `apply_actions(...)`. (2) **Packet behaviour does not match expectation when an entry has many instructions**: a wrong-order suspicion is usually the operator's misunderstanding; check the spec, it is not a bug. (3) **OVN `northd` compile drops the packet**: a bug in `ovn-northd` emits an entry with the wrong `Goto-Table` `table_id` semantics, even though the instruction-set order is correct; fix by comparing `ovn-trace` against SBDB and `ofproto/trace` against `br-int`. (4) **Mixing OF versions with an OF 1.5-only instruction**: installing an entry with `OFPIT_STAT_TRIGGER` on a switch that supports only OF 1.3 returns `OFPBIC_UNSUP_INST`; fix by checking the OF version on both controller and switch.

**Tightly related to.** The instruction-set order is closely paired with five entities. (1) Each individual instruction (Apply, Clear, Write, Write-Metadata, Goto-Table), see the Part 3.6 catalogue. (2) The action-set canonical order (Part 9.22.11), which is the action execution order at the end of the pipeline (different from the instruction order inside an entry). (3) The table-miss entry, which always has its own instruction set (typically `Apply-Actions(output:CONTROLLER)` for OF 1.0, or `drop` for OF 1.1+). (4) The OF wire format `struct ofp11_instruction_*`, the encoding for each instruction type. (5) The validation function `decode_openflow11_instructions()` (OVS), the gatekeeper that checks duplicates and reorders.

**Version differences.**

| Version | Instruction count | Canonical order |
|---------|-------------------|-----------------|
| OF 1.0 (12/2009) | 0 (action list only) | N/A |
| OF 1.1 (02/2011) | 5 (Apply, Clear, Write Actions, Write-Metadata, Goto-Table) | Apply, Clear, Write Actions, Write-Metadata, Goto-Table |
| OF 1.2 (12/2011) | 5 (same as 1.1) | Same as 1.1 |
| OF 1.3 (06/2012) | 6 (Meter added at the start) | Meter, Apply, Clear, Write Actions, Write-Metadata, Goto-Table |
| OF 1.4 (10/2013) | 6 (same as 1.3) | Same as 1.3 |
| OF 1.5 (03/2015) | 7 (Stat-Trigger added between Write-Metadata and Goto-Table) | Meter, Apply, Clear, Write Actions, Write-Metadata, Stat-Trigger, Goto-Table |

The OVS branch-2.17 default protocol is `OpenFlow13`, so the 6-instruction order applies. OVS does not fully support OF 1.5 Stat-Trigger at baseline 2.17.9 (the master branch implements it later).

**Source code reference.** In OVS `v2.17.9`, the instruction-set order is involved at four locations. (1) `lib/ofp-actions.c` function `decode_openflow11_instructions()` parses the instruction list from the wire, checks for duplicates (`if (insts[type]) return OFPBIC_DUP_INST;`), and populates the `insts[OFPIT11_*]` array indexed by type. (2) `lib/ofp-actions.c` function `ofpacts_pull_openflow_instructions()` walks `insts` in increasing type order (Meter, Apply, Clear, Write Actions, Write-Metadata, Goto-Table) and pushes into the `ofpacts` buffer; this is where the canonical reorder actually happens. (3) `ofproto/ofproto-dpif-xlate.c` function `do_xlate_actions()` walks the `ofpacts` buffer in storage order (already canonical) and executes each instruction. (4) `include/openflow/openflow-1.1.h` `enum ofp11_instruction_type` defines the type numbers: `OFPIT11_GOTO_TABLE = 1`, `OFPIT11_WRITE_METADATA = 2`, `OFPIT11_WRITE_ACTIONS = 3`, `OFPIT11_APPLY_ACTIONS = 4`, `OFPIT11_CLEAR_ACTIONS = 5`, `OFPIT13_METER = 6`. Note that the type numbers do not match the execution order (Goto-Table is type 1 but executes last; Apply is type 4 but executes second). The switch uses *enum-driven* lookup, not *type-number-ordered* execution. Source spec: ONF OpenFlow 1.5.1 Section 5.6 *"Instructions"*; OF 1.1.0 Section 5.9 *"Instructions"* first defined it.

**Real-world incident.** A Faucet deployment incident in 2024-Q1: the operator's Faucet configuration emitted the entry `actions=goto_table:1,apply_actions(set_field:0x1->reg5)` for the VLAN-tagging stage. After pushing the config, the packet did not carry the VLAN tag at table 1, even though the entry contained `apply_actions(set_field:...)`. Investigation: Faucet (the Python compiler) sent `OFPT_FLOW_MOD` with the instruction list `[Goto-Table, Apply-Actions]` in source order; OVS received it, `decode_openflow11_instructions` reordered to canonical `[Apply-Actions, Goto-Table]`, and `dump-flows` confirmed the canonical order. But `ofproto/trace` showed `set_field:0x1->reg5` executing at table 0 (correct), the packet entering table 1 with `reg5=0x1` (correct), and table 1 having no entry matching `reg5=0x1`, leading to a drop. The real cause was a bug in the Faucet config that emitted the table 1 entry matching `reg5=0x2` instead of `0x1`. Lesson: the instruction-set order was not the problem (OVS reordered correctly); the operator initially suspected the wrong place. Fix: correct the Faucet config matching `reg5`. Cross-link: 9.22.10 §pipeline troubleshooting; 20.0 §incident-response decision tree.

**Synthetic exercise.** A Mininet lab that verifies the instruction-set order across three scenarios.

```bash
# Lab 1: Switch reorders to canonical order, regardless of how the operator wrote it
$ sudo mn --topo single,3 --switch ovsk --controller none
mininet> sh ovs-ofctl -O OpenFlow13 del-flows s1
mininet> sh ovs-ofctl -O OpenFlow13 add-flow s1 \
    "table=0,priority=100,ip,
     actions=goto_table:1,write_metadata:0x42,apply_actions(dec_ttl)"
mininet> sh ovs-ofctl -O OpenFlow13 dump-flows s1 | grep table=0
# Expected: actions=apply_actions(dec_ttl),write_metadata:0x42,goto_table:1

# Lab 2: Duplicate instruction errors
mininet> sh ovs-ofctl -O OpenFlow13 add-flow s1 \
    "table=0,priority=200,ip,
     actions=apply_actions(dec_ttl),apply_actions(mod_dl_dst:00:11:22:33:44:55)"
# Expected: OFPBIC_DUP_INST

# Lab 3: Causal correctness, Apply before Goto-Table
mininet> sh ovs-ofctl -O OpenFlow13 add-flow s1 \
    "table=0,priority=100,ip,nw_ttl=64,
     actions=apply_actions(dec_ttl),goto_table:1"
mininet> sh ovs-ofctl -O OpenFlow13 add-flow s1 \
    "table=1,priority=100,ip,nw_ttl=63,actions=output:2"
mininet> sh ovs-ofctl -O OpenFlow13 add-flow s1 \
    "table=1,priority=50,ip,nw_ttl=64,actions=drop"
mininet> sh ovs-appctl ofproto/trace s1 \
    "in_port=1,ip,nw_src=10.0.0.1,nw_dst=10.0.0.2,nw_ttl=64"
# Expected: Apply dec_ttl at table 0, packet enters table 1 with nw_ttl=63,
# matches the priority=100 entry, output:2.
# If the order were reversed (Goto before Apply): the packet would enter table 1
# with nw_ttl=64, match the priority=50 entry, and drop.
# The spec forces Apply before Goto-Table.
```

**Common failures and diagnostic signals.** Four mistakes. (1) **`OFPBIC_DUP_INST` at flow install**: writing two instances of the same instruction (`apply_actions(...), apply_actions(...)`); fix by merging into one. (2) **`OFPBIC_UNSUP_INST` at flow install**: an instruction that is not supported by the OF version (for example, Stat-Trigger on an OF 1.3 switch); fix by upgrading the OF version or using an equivalent instruction. (3) **The operator misunderstands the order and debugs the wrong place**: suspecting that the switch ran Goto-Table before Apply, when in reality the switch reorders; debugging is slow because the suspicion is wrong. Fix: check the spec, read the canonical order in `dump-flows`. (4) **Missing an important instruction**: writing an entry with only `actions=goto_table:1` and forgetting Apply-Actions for `dec_ttl` causes the packet to enter table 1 with the original TTL, which can trigger a loop if the pipeline has a decrement at another table. Fix: design the pipeline clearly, place the packet-modify instruction in the appropriate table.

**Comparison with other systems.** Four comparisons. (1) **The `Apply-Actions` action list** (in the same spec): a list in the order the operator wrote, no reorder; different from the instruction set, which reorders to canonical. (2) **iptables `-j` chain target order**: rule order in the chain matters (first match wins), but instructions inside a rule (target plus modules) have no "set with canonical order" concept. (3) **P4 `apply` block, sequential**: P4 executes statements in source-code order with no reordering; more expressive but more complex for the operator. (4) **Cisco IOS ACL**: a rule list in write order, first match wins; an ACE (Access Control Entry) is atomic, with no instruction-set composition like OF.

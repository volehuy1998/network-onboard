## 4.9.1 Action vs Instruction, khác biệt foundation

OpenFlow 1.1+ phân biệt hai concept:

| Concept | Khi nào execute | Thay đổi state nào | Nằm trong | Ví dụ |
|---|---|---|---|---|
| **Action** | Thực hiện ngay trên packet | Packet header + output port | `apply_actions:` hoặc `write_actions:` | `output`, `set_field`, `dec_ttl`, `ct()` |
| **Instruction** | Điều khiển pipeline | Flow chuyển giữa table | Phần tử top-level của flow entry | `Apply-Actions`, `Write-Actions`, `Clear-Actions`, `Goto-Table`, `Write-Metadata`, `Meter` |

OpenFlow 1.0 không phân biệt — mọi thứ là "action" execute tuần tự. Từ 1.1, instruction là **wrapper bao quanh action**, quyết định WHEN và HOW action execute.

### 4.9.1.1 Sáu instruction foundation

| Instruction | Mục đích |
|---|---|
| `Apply-Actions(<action_list>)` | Execute action list ngay lập tức, thay đổi packet visible cho instruction sau |
| `Clear-Actions` | Clear action set hiện tại |
| `Write-Actions(<action_list>)` | Merge action list vào action set, execute khi packet exit pipeline |
| `Write-Metadata(<value>[/<mask>])` | Update metadata khớp với field cho packet (pass qua Goto-Table) |
| `Goto-Table(<table_id>)` | Chuyển packet sang table khác (table_id phải lớn hơn current) |
| `Meter(<meter_id>)` | Apply meter rate limit trước khi forward |

Trong OVS flow spec syntax, `actions=` là shorthand cho `Apply-Actions(...)`. Instruction form đầy đủ hiếm dùng trực tiếp nhưng quan trọng khi đọc OpenFlow 1.3+ spec hoặc theo dõi output.

### 4.9.1.2 Action Set vs Apply Actions

Pipeline OpenFlow 1.1+ duy trì **action set** — tập action tích lũy qua các table. Khi packet exit pipeline (không còn `Goto-Table`), action set execute theo 12 priority level (xem §4.9.14).

Khác biệt hành vi:

```
# apply_actions: execute ngay
table=0,priority=100,in_port=1,
  apply_actions(mod_dl_dst:aa:bb:cc:dd:ee:ff),
  goto_table:10

# Table 10 thấy packet với eth_dst đã đổi
table=10,eth_dst=aa:bb:cc:dd:ee:ff,actions=output:5    # match

# write_actions: tích lũy vào action set
table=0,priority=100,in_port=1,
  write_actions(output:5),
  goto_table:10

# Table 10 không thấy output trong actions hiện tại
table=10,actions=<...>  # action set vẫn có output:5, merge với gì table 10 ghi
```

Khi xuất pipeline, tất cả write_actions merge theo 12-level priority, rồi execute. Apply_actions không tích lũy.

### 4.9.1.3 OVS actions= syntax shortcut

OVS simplify syntax — `actions=` mặc định là Apply-Actions trừ khi explicit `write_actions(...)`. Trong `ovs-ofctl` flow spec:

```
# Syntax 1: shortcut (implicit apply_actions)
priority=100,in_port=1,actions=output:5

# Syntax 2: explicit apply_actions
priority=100,in_port=1,apply_actions(output:5)

# Syntax 3: write_actions (merge vào action set)
priority=100,in_port=1,write_actions(output:5),goto_table:10

# Syntax 4: mix
priority=100,in_port=1,
  actions=dec_ttl,
  write_actions(output:5),
  goto_table:10
```

Phần còn lại của Phần này focus vào **action** (không phải instruction). Instruction chi tiết xem Phần 4.0 §4.0.X.

## 4.9.2 Action `output`, cơ bản nhất của OpenFlow

### 4.9.2.1 Template C cho `output`

| Attribute | Giá trị |
|---|---|
| **Syntax** | `output:<port>` hoặc `output(port=<port>,max_len=<bytes>)` |
| **Category** | 1. Output |
| **OpenFlow version** | 1.0 all versions |
| **Prerequisites** | None |
| **Semantics** | Gửi packet ra port chỉ định. Không terminate packet — có thể dùng `output` nhiều lần cho multiple port |
| **Parameters** | `port`: 1-N (port number) hoặc reserved port (LOCAL/CONTROLLER/IN_PORT/NORMAL/FLOOD/ALL/TABLE); `max_len`: bytes cho CONTROLLER, default unlimited |
| **Side effects** | Tăng tx_packets/tx_bytes counter của port; không modify packet content |
| **Conformance** | Đúng spec; reserved port CONTROLLER/TABLE là OpenFlow concept, LOCAL là OVS/common |

### 4.9.2.2 Ngữ nghĩa đầy đủ

`output` là action foundation nhất, execute đầu tiên trong action set priority (xem §4.9.14). Khi packet đi qua action list có `output:5`, OVS:

1. Compute egress port (port 5 trong ví dụ).
2. Tạo packet clone nếu cần (cho mirror).
3. Queue packet vào send buffer của port 5.
4. Packet content KHÔNG bị modify (tất cả `mod_*`/`set_field` phải chạy trước).

Nếu action list có nhiều `output`, packet được clone và gửi ra nhiều port. Đây là cách OVS triển khai flood/multicast **bằng rule explicit** thay vì reserved port FLOOD/ALL.

### 4.9.2.3 Ví dụ use case

```shell
# Ex 1: simple forwarding
$ ovs-ofctl add-flow br0 "table=0,priority=100,in_port=1,actions=output:2"

# Ex 2: multi-output (flood giới hạn)
$ ovs-ofctl add-flow br0 "table=0,priority=100,in_port=1,dl_type=0x0806,\
    actions=output:2,output:3,output:4"   # broadcast ARP ra 3 port

# Ex 3: output với max_len (cho CONTROLLER)
$ ovs-ofctl add-flow br0 "table=0,priority=100,in_port=1,\
    actions=output(port=CONTROLLER,max_len=128)"  # chỉ gửi 128 byte đầu

# Ex 4: xuất port name thay vì number
$ ovs-ofctl add-flow br-int "table=0,priority=100,in_port=tap-vm-web,\
    actions=output:tap-vm-db"
```

### 4.9.2.4 theo dõi output hiển thị action

```
bridge("br0")
-------------
 0. in_port=1, priority 100
    output:2
```

Trace show action `output:2` là dòng con của khớp với rule. Nếu theo dõi có `Datapath actions: 2` → kernel sẽ forward ra port datapath 2.

### 4.9.2.5 Kịch bản bẻ gãy

- **`output:<port>` với port không tồn tại** → action skip silent, không lỗi; packet không đi đâu. Kiểm chứng bằng `ovs-ofctl show br0` xem port list.
- **`output:IN_PORT` có ý nghĩa đặc biệt:** gửi packet về ingress port (ARP responder pattern). Nếu dùng `output:<ingress_port_number>` thay vì `output:IN_PORT`, OVS từ chối (không cho loopback trực tiếp trừ khi dùng reserved port).
- **`output` trong `write_actions(...)` chỉ giữ 1 giá trị** — action set chỉ có 1 output effective. Nếu muốn multi-output trong action set, phải dùng `group` action.

## 4.9.3 Action `drop`, terminate packet

### 4.9.3.1 Template C

| Attribute | Giá trị |
|---|---|
| **Syntax** | `drop` |
| **Category** | 1. Output |
| **OpenFlow version** | Explicit `drop` là NXM extension; OF spec nói "empty action list = drop" (implicit) |
| **Prerequisites** | None |
| **Semantics** | Terminate packet processing. Không output, không modify. Counter `n_packets` của rule vẫn tăng |
| **Parameters** | None |
| **Side effects** | Counter flow rule increment; packet discarded; kernel datapath có dedicated `OVS_ACTION_ATTR_DROP` (OVS 2.17+) cho efficient drop path |
| **Conformance** | OVS extension; spec 1.0 dùng empty action list = implicit drop |

### 4.9.3.2 Explicit drop vs implicit drop

```
# Implicit drop (rule có match nhưng action rỗng)
priority=50,in_port=1,dl_type=0x0800,actions=

# Explicit drop (OVS extension)
priority=50,in_port=1,dl_type=0x0800,actions=drop
```

Hai forms tương đương semantically. Explicit `drop` dễ đọc + tường minh. OpenFlow 1.5+ yêu cầu explicit drop (spec §5.10).

Nếu không có rule nào khớp với packet (table-miss), OpenFlow 1.3+ mặc định drop. Controller nên explicit cài đặt table-miss rule để:
- Drop với log: `priority=0,actions=controller(userdata=01020304)`
- Forward lên controller để learn: `priority=0,actions=output:CONTROLLER`

### 4.9.3.3 theo dõi hiển thị drop

```
 0. in_port=1,ip,nw_proto=1, priority 150
    drop

Datapath actions: drop
```

`Datapath actions: drop` là ground truth — kernel sẽ DROP packet, counter `missed` không tăng thêm (vì packet đã khớp với megaflow drop).

### 4.9.3.4 Dedicated drop action OVS 2.17+

Trước OVS 2.17, drop được triển khai qua action list rỗng. OVS 2.17 thêm **OVS_ACTION_ATTR_DROP** trong datapath — kernel có fast-path dedicated cho drop, tránh context switch. Check qua `dpif/show-dp-features`:

```shell
$ sudo ovs-appctl dpif/show-dp-features br0 | grep -i drop
Explicit Drop action: Yes
```

`No` → kernel cũ, explicit drop fallback về generic drop path, hiệu năng suy giảm nhẹ.

## 4.9.4 Action `normal`, standard L2 learning switch

### 4.9.4.1 Template C

| Attribute | Giá trị |
|---|---|
| **Syntax** | `normal` hoặc `actions=NORMAL` |
| **Category** | 1. Output |
| **OpenFlow version** | OF 1.0 all (reserved port NORMAL) |
| **Prerequisites** | None |
| **Semantics** | Xử lý packet như standard L2 switch: MAC learning + chuyển tiếp dựa trên FDB. Action cuối, terminate pipeline |
| **Parameters** | None |
| **Side effects** | MAC learning tự động (populate FDB); broadcast nếu unknown unicast; STP/RSTP được apply nếu enable |
| **Conformance** | OpenFlow reserved port; hành vi implementation-specific (OVS specific trong đa số case) |

### 4.9.4.2 Use case

`normal` hữu ích khi muốn mix OpenFlow logic và L2 learning. Mẫu phổ biến:

```shell
# Layer 2 switch với một exception ACL
$ ovs-ofctl add-flow br0 "priority=100,ip,nw_dst=10.0.0.99,actions=drop"
$ ovs-ofctl add-flow br0 "priority=0,actions=normal"
```

Rule priority 100 drop traffic tới 10.0.0.99. Mọi traffic khác fall through tới priority 0 → `normal` → L2 switch chuẩn.

### 4.9.4.3 Fail_mode interaction

Bridge với `fail_mode=secure` + controller disconnect → không action NORMAL tự động. Phải explicit cài đặt `actions=NORMAL` rule để fallback.

Bridge `fail_mode=standalone` + controller disconnect → OVS auto-install `actions=NORMAL` rule priority 0 → L2 switch ngay.

### 4.9.4.4 Kịch bản bẻ gãy

- **`normal` trong OVN-managed br-int** → OVN KHÔNG dùng `normal`. OVN build pipeline explicit với từng khớp với rule. Dùng `normal` trên br-int OVN sẽ phá pipeline.
- **STP + NORMAL:** khi enable STP qua `ovs-vsctl set Bridge br0 stp_enable=true`, NORMAL action tôn trọng STP state — packet vào port BLOCKING không forward.

## 4.9.5 Action `flood`, `all`, broadcast variants

### 4.9.5.1 `flood`

| Attribute | Giá trị |
|---|---|
| **Syntax** | `flood` hoặc `actions=FLOOD` |
| **Semantics** | Output packet ra tất cả port thuộc bridge **trừ ingress port + port bị STP block** |
| **Scope** | Bridge-level, không tôn trọng VLAN membership |

### 4.9.5.2 `all`

| Attribute | Giá trị |
|---|---|
| **Syntax** | `all` hoặc `actions=ALL` |
| **Semantics** | Output packet ra **tất cả port** của switch, kể cả ingress, kể cả port STP block. "Unfiltered flood" |

### 4.9.5.3 So sánh

| Action | Include ingress | STP respect | VLAN filter |
|---|---|---|---|
| `flood` | Không | Có | Không |
| `all` | Có | Không | Không |
| `normal` | Không | Có | Có |
| `output:<n>, output:<m>, ...` | Tùy explicit | Không | Không |

Dùng `flood` cho broadcast ARP response (STP-safe). Dùng `all` cho debug/mirror (xem packet ở mọi port).

### 4.9.5.4 Kịch bản bẻ gãy

- **`flood` trên OVN br-int:** như với `normal`, OVN không dùng. Bypass OVN pipeline có thể tạo loop qua Geneve tunnel vì remote chassis thấy packet flooded.
- **`all` loop risk:** vì include ingress port, packet có thể loop nếu không có STP. Tránh dùng `all` trên production trừ debug ngắn hạn.

## 4.9.6 Action `controller`, gửi packet_in lên OpenFlow controller

### 4.9.6.1 Template C

| Attribute | Giá trị |
|---|---|
| **Syntax** | `controller[:<max_len>]` hoặc `controller(reason=<x>,max_len=<n>,userdata=<hex>)` |
| **Category** | 1. Output (reserved port) |
| **OpenFlow version** | All. OF 1.3+ thêm parameter reason + userdata |
| **Prerequisites** | Controller connected (`ovs-vsctl set-controller br0 tcp:...`) |
| **Semantics** | Gửi packet như OFPT_PACKET_IN message tới controller |
| **Parameters** | `max_len`: bytes truncate (default 128 cho OF 1.0, 64K cho 1.3); `reason`: `action`, `no_match`, `invalid_ttl`, `action_set`, `group`, `packet_out`; `userdata`: opaque hex data |
| **Side effects** | Controller nhận packet, có thể respond với OFPT_PACKET_OUT để tiếp tục |
| **Conformance** | Đúng spec; common pattern cho learn-and-forward |

### 4.9.6.2 Use case và risk

```shell
# Ex: send DHCP packet lên controller
$ ovs-ofctl add-flow br0 \
    "priority=100,udp,tp_src=68,tp_dst=67,actions=controller(max_len=512)"

# Ex: table-miss default → controller
$ ovs-ofctl add-flow br0 "table=0,priority=0,actions=controller"
```

**Risk:** `controller` action spam rất dễ. Một bad rule có thể gửi toàn bộ traffic lên controller, overwhelm control channel. Luôn:

1. Dùng `max_len=128` (hoặc 256) để truncate payload.
2. Rate limit với `meter` action trước `controller`.
3. Theo dõi `ofconn.packets_in_sent` qua `coverage/show`.

### 4.9.6.3 Kịch bản bẻ gãy

- **Controller disconnect mà rule `actions=controller`** → packet queue nội bộ OVS đầy, eventual drop. Counter `rconn_queued` (Phần 9.4 §9.4.1.3 coverage) tăng.
- **Max_len=0** → packet_in chứa chỉ metadata (không payload); controller không thể ra quyết định dựa trên content.

## 4.9.7 Action `local`, output to bridge's internal port

### 4.9.7.1 Template C

| Attribute | Giá trị |
|---|---|
| **Syntax** | `local` hoặc `actions=LOCAL` |
| **Category** | 1. Output (reserved port) |
| **OpenFlow version** | All |
| **Prerequisites** | Bridge có internal port (LOCAL port tự động có cho mọi bridge, trùng tên bridge) |
| **Semantics** | Output packet vào local network stack của host qua LOCAL port (Linux netdev `br0`) |
| **Side effects** | Packet đi vào kernel rx queue của netdev `br0`, có thể bị iptables/tc xử lý tiếp |

### 4.9.7.2 Use case

Host OVS chạy TCP server trên IP gán lên LOCAL port:

```shell
# br-int có IP 10.1.0.1 gán lên netdev br-int (LOCAL port)
$ ip addr add 10.1.0.1/24 dev br-int

# Rule accept traffic destined 10.1.0.1 vào local stack
$ ovs-ofctl add-flow br-int "priority=100,ip,nw_dst=10.1.0.1,actions=local"
```

### 4.9.7.3 So sánh với `output:<LOCAL_port_number>`

```shell
$ ovs-ofctl show br-int | grep LOCAL
 LOCAL(br-int): addr:...
```

LOCAL port có port number reserved 0xfffe. `actions=local` và `actions=output:LOCAL` và `actions=output:65534` tương đương.

## 4.9.8 Action `in_port`, gửi ngược về ingress

### 4.9.8.1 Template C

| Attribute | Giá trị |
|---|---|
| **Syntax** | `in_port` hoặc `actions=IN_PORT` |
| **Category** | 1. Output (reserved port) |
| **OpenFlow version** | All |
| **Prerequisites** | None |
| **Semantics** | Output packet ra **ingress port** (gửi ngược về nơi packet đến). Classic ARP responder pattern |
| **Side effects** | Packet đi ngược qua cùng port đã nhận |

### 4.9.8.2 ARP responder pattern

```shell
# OVN-style ARP responder: match ARP request cho gateway IP
# Modify packet thành ARP reply
# Gửi ngược về ingress port (không broadcast)
$ ovs-ofctl add-flow br-int \
    "priority=200,arp,arp_op=1,arp_tpa=10.1.1.1,\
     actions=move:NXM_OF_ETH_SRC[]->NXM_OF_ETH_DST[],\
             mod_dl_src:aa:bb:cc:dd:ee:ff,\
             load:0x2->NXM_OF_ARP_OP[],\
             move:NXM_NX_ARP_SHA[]->NXM_NX_ARP_THA[],\
             move:NXM_OF_ARP_SPA[]->NXM_OF_ARP_TPA[],\
             load:0xaabbccddeeff->NXM_NX_ARP_SHA[],\
             load:0x0a010101->NXM_OF_ARP_SPA[],\
             IN_PORT"
```

`IN_PORT` cuối là reserved port. Dùng đây vì thông thường không có rule cho phép output bình thường back về ingress.

### 4.9.8.3 Khác biệt `in_port` action vs `in_port` khớp với field

Đừng nhầm:
- `in_port=1` trong **match spec** — match field, "packet đến từ port 1" (Phần 4.8 §4.8.2.1).
- `actions=in_port` hoặc `output:IN_PORT` — **action**, "gửi packet ra port đã nhận".

Flow có thể có cả hai:
```
priority=200,in_port=1,arp,arp_op=1,actions=IN_PORT
```

## 4.9.9 Action `table`, đưa packet vào pipeline lookup lại

### 4.9.9.1 Template C

| Attribute | Giá trị |
|---|---|
| **Syntax** | `table` hoặc `output:TABLE` |
| **Category** | 1. Output (reserved port) |
| **OpenFlow version** | 1.0+ |
| **Prerequisites** | Có sense chỉ trong context PACKET_OUT (controller gửi packet vào switch) |
| **Semantics** | Submit packet vào OpenFlow pipeline bắt đầu từ table 0 |

### 4.9.9.2 Use case

`TABLE` hiếm dùng trong flow rule — chủ yếu trong PACKET_OUT message từ controller:

```
# Controller inject packet vào pipeline
OFPT_PACKET_OUT: buffer_id=-1, in_port=<x>, actions=OUTPUT:TABLE, data=<hex>
```

Khi controller nhận packet_in + quyết định sau đó "để switch xử lý bình thường", gửi packet ra TABLE thay vì chỉ định output port cụ thể.

Trong flow rule, dùng rất hiếm — thường `resubmit(,0)` hoặc `goto_table:N` là thay thế.

## 4.9.10 Action `group`, group table indirection

### 4.9.10.1 Template C

| Attribute | Giá trị |
|---|---|
| **Syntax** | `group:<group_id>` |
| **Category** | 1. Output (indirect) |
| **OpenFlow version** | 1.1+ |
| **Prerequisites** | Group phải được cài đặt trước qua `ovs-ofctl add-group` |
| **Semantics** | Execute action list trong group entry. Group có 4 types: all, select, indirect, fast_failover |
| **Parameters** | `group_id`: 32-bit ID của group |
| **Side effects** | Depending on group type — multicast (all), load-balance (select), shared indirection (indirect), active-backup (fast_failover) |

### 4.9.10.2 Bốn group type

| Type | Bucket count | Selection | Use case |
|---|---|---|---|
| `all` | 1+ | Execute ALL buckets | Multicast, broadcast efficient |
| `select` | 1+ | Pick 1 bucket by hash | Load balance (ECMP, LB backend) |
| `indirect` | 1 | Execute only bucket | Shared chuyển tiếp (next-hop indirection, reduce rule count) |
| `fast_failover` | 1+ | First "live" bucket | Active-backup chuyển tiếp với watch_port |

### 4.9.10.3 Template cho Group Type `select` (LB pattern)

```shell
# Install group 1: load balance giữa 3 backend
$ ovs-ofctl add-group br0 "group_id=1,type=select,\
  bucket=weight:100,actions=mod_dl_dst:aa:bb:cc:00:00:01,output:5,\
  bucket=weight:100,actions=mod_dl_dst:aa:bb:cc:00:00:02,output:6,\
  bucket=weight:100,actions=mod_dl_dst:aa:bb:cc:00:00:03,output:7"

# Rule: TCP destined VIP → group 1
$ ovs-ofctl add-flow br0 \
  "priority=100,tcp,nw_dst=10.1.0.100,tp_dst=80,actions=group:1"
```

Weight default là 1 nếu không chỉ định. Hash function default là 5-tuple (src_ip, dst_ip, src_port, dst_port, proto).

### 4.9.10.4 Template Group Type `fast_failover` (link backup)

```shell
# Group 2: primary eth0, backup eth1
$ ovs-ofctl add-group br0 "group_id=2,type=fast_failover,\
  bucket=watch_port:eth0,actions=output:eth0,\
  bucket=watch_port:eth1,actions=output:eth1"

# Rule dùng group
$ ovs-ofctl add-flow br0 "priority=100,actions=group:2"
```

OVS theo dõi `watch_port` state; nếu eth0 down → switch sang bucket 2 (eth1) < 50ms (với BFD).

### 4.9.10.5 `dump-groups` để kiểm chứng

```shell
$ ovs-ofctl -O OpenFlow13 dump-groups br0

  bucket=weight:100,actions=mod_dl_dst:aa:bb:cc:00:00:01,output:5,
  bucket=weight:100,actions=mod_dl_dst:aa:bb:cc:00:00:02,output:6,
  bucket=weight:100,actions=mod_dl_dst:aa:bb:cc:00:00:03,output:7

 group_id=2,type=fast_failover,
  bucket=watch_port:eth0,actions=output:eth0,
  bucket=watch_port:eth1,actions=output:eth1
```

## 4.9.11 Action `resubmit`, re-lookup qua table hiện tại hoặc khác

### 4.9.11.1 Template C

| Attribute | Giá trị |
|---|---|
| **Syntax** | `resubmit:<port>` hoặc `resubmit(<port>,<table>)` |
| **Category** | 6. Control/Pipeline |
| **OpenFlow version** | NXM extension (OVS specific) |
| **Prerequisites** | None |
| **Semantics** | Tạo packet clone với virtual `in_port` mới + optional table khác, lookup trong pipeline như packet mới. Action set chia sẻ giữa nested resubmit |
| **Parameters** | `port`: virtual in_port cho resubmit (default current); `table`: target table (default current) |
| **Side effects** | Stack depth tối đa 16 (tránh infinite recursion); execute action từ rule khớp với trong resubmitted lookup |

### 4.9.11.2 So sánh `resubmit` vs `goto_table`

| Aspect | `goto_table:N` | `resubmit(,N)` |
|---|---|---|
| OpenFlow standard | Có (1.1+) | Không (NXM) |
| Direction | Forward only (N > current) | Bất kỳ (forward hoặc backward) |
| Action set | Cumulative (write_actions accumulate) | Shared giữa nested call |
| Use case | Linear pipeline | Modular/shared table |

OVS prefer `goto_table` cho OpenFlow standard pipeline; `resubmit` cho OVS-specific pattern như nested classifier.

### 4.9.11.3 Use case OVN

OVN dùng `resubmit` extensively trong translation. Ví dụ từ OVN logical flow:

```
# Logical: chuyển đến table sau
table=8(ls_in_lookup_fdb), priority=0, match=1, action=next;

# Physical: resubmit
table=8,priority=0,actions=resubmit(,9)
```

Xem Phần 13.7 cho OVN logical-to-physical translation dùng `resubmit`.

### 4.9.11.4 Kịch bản bẻ gãy

- **Resubmit loop >16 depth** → OVS abort với "resubmit recursion too deep", packet drop. Thường do classifier loop lỗi. Check theo dõi để thấy path.
- **Shared action set trap:** `apply_actions(output:5)` rồi `resubmit` — action set được share, nested lookup có thể override. Best practice: dùng `write_actions` thay vì `apply_actions` khi kết hợp resubmit.

## 4.9.12 Action `clone`, execute action tạm nhưng không commit modification

### 4.9.12.1 Template C

| Attribute | Giá trị |
|---|---|
| **Syntax** | `clone(<action_list>)` |
| **Category** | 6. Control/Pipeline |
| **OpenFlow version** | NXM extension |
| **Prerequisites** | None |
| **Semantics** | Execute action list với packet clone tạm. Sau khi clone block kết thúc, packet original không bị modify bởi action trong clone |
| **Side effects** | Action có side effect (output, learn, ct(commit)) vẫn thực thi; chỉ packet header/metadata được restore |

### 4.9.12.2 Use case

```shell
# Output packet copy ra port 99 (span/mirror) mà không ảnh hưởng packet chính
$ ovs-ofctl add-flow br0 \
    "priority=100,in_port=1,actions=clone(output:99),output:2"
```

Packet ra port 2 giữ nguyên state. Clone output port 99 có thể có action modify trước output (mod_dl_dst) mà không modify packet đi ra port 2.

### 4.9.12.3 Khác biệt với explicit multi-output

```
actions=output:99,output:2
# Packet A gửi ra 99, packet A' (sẽ output:2) vẫn nguyên

actions=clone(mod_dl_dst:aa:aa:..,output:99),output:2
# Packet A' (cho output:2) không bị mod_dl_dst
```

Clone là cách sạch để forward packet modified mà không forward pollute.

## 4.9.14 Action Set execution order, 12 priority level

Khi packet exit pipeline (không còn `Goto-Table`), action set tích lũy được execute theo **12 priority level** (OpenFlow 1.3 §5.10, table 9):

| Priority | Action | Ý nghĩa |
|---|---|---|
| 1 | `copy_ttl_in` | Copy outer TTL vào inner (IP-in-IP/MPLS decap prep) |
| 2 | `pop_mpls`, `pop_pbb`, `pop_vlan` | Decapsulate |
| 3 | `push_pbb` | PBB encap |
| 4 | `push_mpls` | MPLS encap |
| 5 | `push_vlan` | VLAN encap |
| 6 | `copy_ttl_out` | Copy inner TTL vào outer |
| 7 | `dec_mpls_ttl`, `dec_nw_ttl` | Decrement TTL |
| 8 | `set_field` (tất cả), `mod_*` | Field modification |
| 9 | `set_queue` | Queue assignment |
| 10 | `group` | Group indirection |
| 11 | `output` | Output port |
| 12 | (cleanup action set) | Clear + exit |

Ý nghĩa: dù thứ tự ghi action trong `write_actions(...)` thế nào, khi exit pipeline execute **theo priority**.

### 4.9.14.1 Ví dụ

```
write_actions(output:5, mod_dl_dst:aa:aa:aa:aa:aa:aa, push_vlan:0x8100)
```

Ghi: output đầu, mod sau, push cuối. Execute theo priority:
- L5 push_vlan
- L8 mod_dl_dst
- L11 output

→ packet: push VLAN, mod MAC, output ra port 5. Dù thứ tự ghi ngược lại.

### 4.9.14.2 Apply-Actions không theo priority

`apply_actions=` execute **tuần tự theo thứ tự ghi**. Khác biệt quan trọng khi đọc flow spec:

```
apply_actions(mod_dl_dst:aa..., push_vlan:0x8100, output:5)
# Sequential: mod, then push, then output
# Packet ra port 5 có VLAN + MAC mới
```

Rule of thumb OVN: **OVN logical flow dùng apply_actions** chủ yếu (sequential order dễ đoán hơn). `write_actions` dùng cho pipeline egress consolidation.

## 4.9.15 Category 2, Encapsulation actions — VLAN tagging

### 4.9.15.1 `push_vlan`

| Attribute | Giá trị |
|---|---|
| **Syntax** | `push_vlan:<ethertype>` (ví dụ `push_vlan:0x8100` cho 802.1Q, `0x88a8` cho 802.1ad Q-in-Q) |
| **Category** | 2. Encap/Decap |
| **OpenFlow version** | 1.1+ |
| **Prerequisites** | None |
| **Semantics** | Insert VLAN header sau Ethernet header. Default VID = 0, PCP = 0 — phải theo sau `mod_vlan_vid` hoặc `set_field:<vid>->vlan_vid` để set VLAN ID cụ thể |
| **Parameters** | `ethertype`: 0x8100 (VLAN) hoặc 0x88a8 (PBB/Q-in-Q) |
| **Side effects** | Tăng packet length 4 byte; set `vlan_tci` có Present bit |

### 4.9.15.2 `pop_vlan`

| Attribute | Giá trị |
|---|---|
| **Syntax** | `pop_vlan` hoặc `strip_vlan` (OVS alias) |
| **Category** | 2. Encap/Decap |
| **OpenFlow version** | 1.1+ |
| **Prerequisites** | Packet có VLAN tag (`vlan_tci[12]=1` Present bit) |
| **Semantics** | Remove VLAN header trên cùng |
| **Side effects** | Giảm packet length 4 byte; `vlan_tci` về 0 |

### 4.9.15.3 Ví dụ tagging workflow

```shell
# Access port → trunk port: tag packet với VLAN 100
$ ovs-ofctl add-flow br0 \
    "priority=100,in_port=1,vlan_tci=0,\
     actions=push_vlan:0x8100,mod_vlan_vid:100,output:2"

# Trunk port → access port: untag
$ ovs-ofctl add-flow br0 \
    "priority=100,in_port=2,dl_vlan=100,\
     actions=strip_vlan,output:1"

# Q-in-Q double tagging
$ ovs-ofctl add-flow br0 \
    "priority=100,in_port=3,\
     actions=push_vlan:0x88a8,mod_vlan_vid:500,\
             push_vlan:0x8100,mod_vlan_vid:100,\
             output:4"
```

### 4.9.15.4 Kịch bản bẻ gãy

- **`push_vlan` nhưng không set VID** → packet có VLAN header với VID=0 (reserved cho priority tagging, thường bị nhiều switch hardware drop). Luôn pair với `mod_vlan_vid` hoặc `set_field`.
- **`pop_vlan` trên packet không tag** → silently skip (no error); kiểm chứng với `vlan_tci=0x1000/0x1000` khớp với trước.

## 4.9.16 Category 2, MPLS + PBB encapsulation

### 4.9.16.1 `push_mpls`, `pop_mpls`

| Attribute | `push_mpls` | `pop_mpls` |
|---|---|---|
| **Syntax** | `push_mpls:<ethertype>` (0x8847 unicast, 0x8848 multicast) | `pop_mpls:<ethertype>` (inner protocol) |
| **OpenFlow version** | 1.1+ | 1.1+ |
| **Semantics** | Insert MPLS label trên cùng | Remove outermost MPLS label, expose inner |
| **Side effects** | Packet length +4 byte; `mpls_label/tc/bos/ttl` cần set riêng | Packet length -4 byte |

### 4.9.16.2 Ví dụ MPLS label push

```shell
# L3 VPN: push MPLS label 100 trên top, eth_type chuyển 0x8847
$ ovs-ofctl add-flow br0 \
    "priority=100,ip,nw_dst=10.1.0.0/24,\
     actions=push_mpls:0x8847,\
             set_field:100->mpls_label,\
             set_field:1->mpls_bos,\
             set_field:64->mpls_ttl,\
             output:uplink"

# MPLS pop tại PE:
$ ovs-ofctl add-flow br0 \
    "priority=100,mpls,mpls_label=100,\
     actions=pop_mpls:0x0800,\
             output:ce_port"
```

### 4.9.16.3 `push_pbb`, `pop_pbb` (Provider Backbone Bridge)

Tương tự MPLS nhưng cho 802.1ah PBB. Dùng rất ít ngoài MEF metro Ethernet. Skip deep-dive vì scope hyperscale DC không cần.

### 4.9.16.4 `encap`, `decap` (generic, OF 1.5 + NSH)

OpenFlow 1.5 thêm generic `encap`/`decap` cho header chain (NSH, GRE, VXLAN-GPE) không cứng VLAN/MPLS/PBB. Ít khi dùng trong OVN; OVN encap Geneve/VXLAN qua **tunnel port** config chứ không qua action.

## 4.9.17 Category 3, `set_field` — generic field rewrite

### 4.9.17.1 Template C

| Attribute | Giá trị |
|---|---|
| **Syntax** | `set_field:<value>[/<mask>]-><field>` hoặc `set_field(<field>=<value>)` |
| **Category** | 3. Field modification |
| **OpenFlow version** | 1.2+ (OXM TLV required) |
| **Prerequisites** | Field-specific (ví dụ `eth_type=0x0800` cho IP field) |
| **Semantics** | Generic field rewrite — thay thế tất cả `mod_*` specific action. Hỗ trợ mask cho partial write |
| **Parameters** | `value`: giá trị mới (hex, decimal, MAC, IP); `mask`: optional bitmask; `field`: OXM/NXM field name |
| **Side effects** | Modify packet header in-place; checksum auto-recompute cho L3/L4 field |

### 4.9.17.2 Ví dụ set_field rewrite phổ biến

```shell
# Rewrite MAC destination (replacement cho mod_dl_dst)
$ ovs-ofctl add-flow br0 \
    "priority=100,ip,nw_dst=10.1.0.1,\
     actions=set_field:aa:bb:cc:00:01:01->eth_dst,output:5"

# Rewrite IP destination (DNAT semantics)
$ ovs-ofctl add-flow br0 \
    "priority=100,tcp,nw_dst=10.1.0.100,tp_dst=80,\
     actions=set_field:10.2.0.10->ipv4_dst,\
             set_field:8080->tcp_dst,\
             output:backend_port"

# Partial bit write với mask (chỉ đổi upper 3 byte MAC = OUI)
$ ovs-ofctl add-flow br0 \
    "priority=100,in_port=1,\
     actions=set_field:aa:bb:cc:00:00:00/ff:ff:ff:00:00:00->eth_src,\
             output:2"

# DSCP marking
$ ovs-ofctl add-flow br0 \
    "priority=100,ip,tcp,tp_dst=5001,\
     actions=set_field:46->ip_dscp,output:uplink"
```

### 4.9.17.3 `set_field` vs legacy `mod_*`

| Aspect | `set_field` | `mod_<field>` |
|---|---|---|
| OpenFlow version | 1.2+ | 1.0 legacy |
| Syntax | Unified | Per-field |
| Mask support | Có | Không |
| New field | Tự động support | Phải thêm mỗi field mới |

Best practice: **dùng `set_field` cho code mới**. Legacy `mod_dl_src/dst`, `mod_nw_src/dst`, `mod_tp_src/dst` vẫn work nhưng không scale cho field mới (tun_metadata, ct_mark, register, v.v.) — những field này không có `mod_*` riêng, phải dùng `set_field` hoặc `load`.

### 4.9.18.2 `dec_ttl`, `dec_mpls_ttl`

| Attribute | Giá trị |
|---|---|
| **Syntax** | `dec_ttl`, `dec_mpls_ttl` |
| **Category** | 3. Field modification |
| **OpenFlow version** | 1.0+ (`dec_ttl`), 1.1+ (`dec_mpls_ttl`) |
| **Prerequisites** | IP packet với TTL > 0 cho `dec_ttl` |
| **Semantics** | Giảm TTL/HL 1 đơn vị. Nếu TTL = 0 sau decrement → **packet gửi lên controller qua packet_in** (không drop tự động); controller quyết định |
| **Side effects** | Update TTL; recompute IP checksum; nếu TTL=0 → OFPT_PACKET_IN với reason `OFPR_INVALID_TTL` |

**Router function dùng `dec_ttl`:**

```shell
# OVS thành virtual router
$ ovs-ofctl add-flow br0 \
    "priority=100,ip,nw_dst=10.2.0.0/24,\
     actions=dec_ttl,\
             mod_dl_src:aa:bb:cc:00:00:01,\
             mod_dl_dst:aa:bb:cc:00:02:01,\
             output:uplink"

# Table miss send ICMP TTL exceeded — via controller
$ ovs-ofctl add-flow br0 \
    "priority=0,dl_type=0x0800,actions=controller(reason=invalid_ttl)"
```

### 4.9.18.3 `copy_ttl_in`, `copy_ttl_out`

MPLS/IP stacking: khi push/pop MPLS label, `copy_ttl_in` copy outer TTL vào inner, `copy_ttl_out` copy ngược lại. Dùng trong MPLS VPN để preserve TTL qua tunnel.

## 4.9.19 Category 3, `move` + `load` — register manipulation

### 4.9.19.1 `move`, copy field-to-field

| Attribute | Giá trị |
|---|---|
| **Syntax** | `move:<src_field>[<start>..<end>]-><dst_field>[<start>..<end>]` |
| **Category** | 3. Field modification |
| **OpenFlow version** | NXM extension |
| **Semantics** | Copy bits từ src field vào dst field. Bit-range optional — nếu thiếu copy full field |
| **Example** | `move:NXM_OF_ETH_SRC[]->NXM_OF_ETH_DST[]` copy eth_src → eth_dst |

### 4.9.19.2 `load`, constant-to-field

| Attribute | Giá trị |
|---|---|
| **Syntax** | `load:<value>-><field>[<start>..<end>]` |
| **Category** | 3. Field modification |
| **OpenFlow version** | NXM extension |
| **Semantics** | Load hằng số vào field (hoặc bit-range) |
| **Example** | `load:0x42->NXM_NX_REG0[]` |

### 4.9.19.3 Use case ARP responder (kết nối với Phần 4.8 §4.8.5)

```shell
$ ovs-ofctl add-flow br-int \
    "priority=200,arp,arp_op=1,arp_tpa=10.1.1.1,\
     actions=move:NXM_OF_ETH_SRC[]->NXM_OF_ETH_DST[],\
             mod_dl_src:aa:bb:cc:dd:ee:ff,\
             load:0x2->NXM_OF_ARP_OP[],\
             move:NXM_NX_ARP_SHA[]->NXM_NX_ARP_THA[],\
             move:NXM_OF_ARP_SPA[]->NXM_OF_ARP_TPA[],\
             load:0xaabbccddeeff->NXM_NX_ARP_SHA[],\
             load:0x0a010101->NXM_OF_ARP_SPA[],\
             IN_PORT"
```

Bảy action consecutive: `move` copy field, `load` write constant, `IN_PORT` output ngược. Đây là cách classic triển khai ARP responder trong OVS — không cần gửi packet lên controller.

### 4.9.19.4 Bit-range manipulation

```shell
# Set chỉ bit 0-3 của reg0 thành 0xa
$ ovs-ofctl add-flow br0 \
    "priority=100,in_port=1,\
     actions=load:0xa->NXM_NX_REG0[0..3],resubmit(,1)"

# Move bit 12-15 của metadata vào reg1[0..3]
$ ovs-ofctl add-flow br0 \
    "priority=100,actions=move:OXM_OF_METADATA[12..15]->NXM_NX_REG1[0..3]"
```

Bit-range syntax `[<start>..<end>]` hoặc `[<bit>]`. Dùng extensively trong OVN cho chia register thành subfield (ví dụ `reg0[0]` = ACL hint flag, `reg0[1]` = committed flag).

## 4.9.20 Category 4, Metadata actions

### 4.9.20.1 `write_metadata` (instruction, không action)

Strictly đây là **instruction** (Phần §4.9.1.1), không phải action. Nhưng OVS cho phép viết trong `actions=` shorthand. Syntax:

```
actions=write_metadata:<value>[/<mask>]
```

Scope: set metadata field pass qua `Goto-Table`. Khác với register (lost sau recirc hoặc upcall boundary), metadata preserve xuyên pipeline.

### 4.9.20.2 `set_tunnel`, `set_tunnel64`

| Attribute | `set_tunnel:<value>` | `set_tunnel64:<value>` |

## 4.9.23 Category 5, `ct()` — conntrack integration

### 4.9.23.1 Template C cho `ct()`

| Attribute | Giá trị |
|---|---|
| **Syntax** | `ct(<option>[,<option>...])` |
| **Category** | 5. Firewall/CT |
| **OpenFlow version** | NXM extension (OVS) |
| **Prerequisites** | Kernel datapath có conntrack support (`dpif/show-dp-features: Conntrack eventmask: Yes`) |
| **Semantics** | Submit packet vào kernel conntrack module. Sau khi conntrack process, packet recirculate lại pipeline với `ct_state` + `ct_zone` + `ct_mark` + `ct_label` populated |
| **Parameters** | `commit`, `zone=N`, `nat`, `force`, `alg=<ftp/tftp/sip/...>`, `exec(<actions>)`, `table=<N>` |
| **Side effects** | Rất lớn: tạo conntrack entry (commit), NAT rewrite (nat), trigger ALG helper, recirc packet |

### 4.9.23.2 Option của `ct()`

**`commit`** — tạo entry trong conntrack table:

```
actions=ct(commit,zone=1)
# Đăng ký connection này vào kernel conntrack table zone 1
# Subsequent packet cùng 5-tuple sẽ match conntrack entry
```

**`zone=N`** — conntrack zone ID (16-bit), cô lập namespace giữa OVN logical switch. Không specify zone → zone 0 default.

**`nat`** — apply NAT:

```
# Source NAT (SNAT)
actions=ct(commit,nat(src=10.1.0.1))

# Destination NAT (DNAT)
actions=ct(commit,nat(dst=10.2.0.1:8080))

# NAT với port range
actions=ct(commit,nat(src=10.1.0.1:30000-40000))

# NAT persistence: reuse same mapping cho same client
actions=ct(commit,nat(src=10.1.0.1,persistent))
```

**`force`** — re-commit, override existing entry. Dùng khi connection expected đã stale.

**`alg=<name>`** — application layer gateway helper:

| ALG | Protocol |
|---|---|
| `alg=ftp` | FTP control channel (parse PORT/PASV để open data channel dynamic) |
| `alg=tftp` | TFTP |
| `alg=sip` | SIP (VoIP signaling → RTP media channel) |

**`exec(<actions>)`** — execute action list **trong conntrack zone context**. Dùng set ct_mark / ct_label khi commit:

```
actions=ct(commit,zone=1,exec(set_field:0xab->ct_mark))
```

**`table=N`** — sau recirculate, lookup table N (default = table hiện tại).

### 4.9.23.3 Typical stateful firewall pattern

```shell
# Step 1: Mọi packet qua ct (populate ct_state)
# Table 0 — dispatcher
$ ovs-ofctl add-flow br0 \
    "table=0,priority=100,ip,actions=ct(table=1,zone=1)"

# Table 1 — policy based on ct_state
# Allow new outbound + commit
$ ovs-ofctl add-flow br0 \
    "table=1,priority=200,ip,ct_state=+new+trk,\
     actions=ct(commit,zone=1),output:uplink"

# Allow reply (established)
$ ovs-ofctl add-flow br0 \
    "table=1,priority=100,ip,ct_state=+est+trk,actions=output:inside"

# Drop invalid
$ ovs-ofctl add-flow br0 \
    "table=1,priority=100,ip,ct_state=+inv+trk,actions=drop"

# Default drop
$ ovs-ofctl add-flow br0 \
    "table=1,priority=0,actions=drop"
```

Packet flow:
1. Packet đến table 0 → `ct(table=1)` → submit lên kernel conntrack.
2. Kernel populate `ct_state` (new, est, rel, inv) + recirc.
3. Packet reappear ở table 1 với ct_state populated → match rule.
4. Rule có `ct(commit)` thì conntrack entry persist.

### 4.9.23.4 `ct_clear`

| Attribute | Giá trị |
|---|---|
| **Syntax** | `ct_clear` |
| **Semantics** | Clear conntrack tracked state cho packet hiện tại (reset `ct_state` về unset). Packet tiếp theo qua `ct()` sẽ re-evaluate |
| **Use case** | Khi pipeline muốn re-track connection (ví dụ sau NAT, reclassify) |

Xem Phần 9.24 conntrack stateful firewall cho case study đầy đủ.

## 4.9.24 Category 6, `learn()` — MAC learning via flow install

### 4.9.24.1 Template C cho `learn()`

| Attribute | Giá trị |
|---|---|
| **Syntax** | `learn(<option>[,<option>...])` |
| **Category** | 6. Control/Pipeline |
| **OpenFlow version** | NXM extension |
| **Prerequisites** | None |
| **Semantics** | Install flow rule mới dynamic dựa trên packet hiện tại. Đây là cách OVS triển khai L2 MAC learning mà không cần controller — flow "self-programming" |
| **Parameters** | `table=N`, `priority=P`, `idle_timeout=T`, `hard_timeout=T`, `fin_idle_timeout=T`, `NXM_FIELD[]` match/action templates |
| **Side effects** | Tạo flow rule mới trong kernel flow table — counter `n_flow_mods` tăng |

### 4.9.24.2 MAC learning example — classic OVS pattern

```shell
# Learning rule: khi packet đến, install reverse flow
$ ovs-ofctl add-flow br0 \
    "table=0,priority=100,\
     actions=learn(table=1,\
                    idle_timeout=60,\
                    NXM_OF_ETH_DST[]=NXM_OF_ETH_SRC[],\
                    load:NXM_OF_IN_PORT[]->NXM_NX_REG0[]),\
             resubmit(,1)"

# Forwarding based on learned entries
$ ovs-ofctl add-flow br0 \
    "table=1,priority=50,\
     actions=output:NXM_NX_REG0[]"

# Fallback: flood if no learn hit
$ ovs-ofctl add-flow br0 "table=1,priority=0,actions=flood"
```

**Ngữ nghĩa learn rule:**

- `table=1`: install flow vào table 1.
- `idle_timeout=60`: flow expire sau 60s inactive.
- `NXM_OF_ETH_DST[]=NXM_OF_ETH_SRC[]`: match spec mới — "dst MAC = src MAC của packet hiện tại".
- `load:NXM_OF_IN_PORT[]->NXM_NX_REG0[]`: action — load ingress port vào reg0.

Khi packet (src=MAC_A, in_port=1) đến table 0, `learn` tạo flow:

```
table=1,priority=32768,idle_timeout=60,dl_dst=MAC_A,actions=load:1->NXM_NX_REG0[]
```

Packet sau với dst=MAC_A → match flow này → reg0=1 → table 1 priority 50 → output:1.

### 4.9.24.3 `fin_idle_timeout`

Option đặc biệt cho TCP: timeout khác khi thấy FIN packet (connection đóng):

```
actions=learn(...,idle_timeout=300,fin_idle_timeout=10)
```

Normal flow expire 300s idle, nhưng sau FIN thấy → timeout 10s.

### 4.9.24.4 Risk của learn()

- Mỗi packet khớp với rule có `learn()` tạo flow mới → flow explosion nếu misconfig.
- `coverage/show upcall_flow_add` sẽ rất cao.
- Install trên bridge production phải rate limit carefully.

## 4.9.25 Category 6, `conjunction()` — cross-product compression

### 4.9.25.1 Template C

| Attribute | Giá trị |
|---|---|
| **Syntax** | `conjunction(<id>,<component>/<total>)` |
| **Category** | 6. Control/Pipeline |
| **OpenFlow version** | NXM extension |
| **Prerequisites** | None |
| **Semantics** | Khai báo rule là một "component" của conjunction group `id`. Packet cần khớp với `<total>` component khác nhau của cùng `id` → `conj_id` field set → rule priority khác khớp với để execute action |
| **Parameters** | `id`: 32-bit conjunction group ID; `component`: 1-N, thứ tự component; `total`: tổng số component cần match |

### 4.9.25.2 Vấn đề cross-product

ACL rule dạng "source IN {A, B, C} AND dest IN {X, Y, Z} AND port IN {80, 443}" cần **3 × 3 × 2 = 18 flow rule** nếu viết explicit.

Với conjunction: **3 + 3 + 2 + 1 = 9 rule** (3 source component + 3 dest component + 2 port component + 1 final action).

### 4.9.25.3 Ví dụ đầy đủ

```shell
# Source components (conjunction 42, component 1/3)
$ for src in 10.0.1.1 10.0.1.2 10.0.1.3; do
    sudo ovs-ofctl add-flow br0 \
      "table=0,priority=100,ip,nw_src=$src,actions=conjunction(42,1/3)"
  done

# Destination components (42, component 2/3)
$ for dst in 10.0.2.1 10.0.2.2 10.0.2.3; do
    sudo ovs-ofctl add-flow br0 \
      "table=0,priority=100,ip,nw_dst=$dst,actions=conjunction(42,2/3)"
  done

# Port components (42, component 3/3)
$ for port in 80 443; do
    sudo ovs-ofctl add-flow br0 \
      "table=0,priority=100,ip,tcp,tp_dst=$port,actions=conjunction(42,3/3)"
  done

# Final rule match conj_id (set when all 3 components match)
$ sudo ovs-ofctl add-flow br0 \
    "table=0,priority=100,conj_id=42,actions=output:5"
```

### 4.9.25.4 OVN Port_Group dùng conjunction

OVN ACL từ Logical_Switch `Port_Group` extensively dùng conjunction. Ví dụ policy "port_group WEB (10 port) → port_group DB (5 port) allow TCP 3306":

- 10 rule cho WEB source (conj component 1/2)
- 5 rule cho DB dest (conj component 2/2)
- 1 rule final khớp với `conj_id`

Total 16 rule thay vì 50 rule explicit. Xem Phần 13.3 OVN ACL + Load Balancer cho context OVN.

## 4.9.26 Category 6, `multipath()` — hash-based path selection

### 4.9.26.1 Template C

| Attribute | Giá trị |
|---|---|
| **Syntax** | `multipath(<fields>,<basis>,<algorithm>,<n_links>,<arg>,<dst>[<start>..<end>])` |
| **Category** | 6. Control/Pipeline |
| **OpenFlow version** | NXM extension |
| **Semantics** | Hash packet header fields → output ID 0 đến (n_links-1) → load vào dst field. Tương tự group type=select nhưng không cần group table config |
| **Parameters** | `fields`: `eth_src`, `symmetric_l4`, v.v.; `algorithm`: `modulo_n`, `hash_threshold`, `hrw`, `iter_hash`; `n_links`: số output; `dst`: target field |

### 4.9.26.2 Use case ECMP

```shell
# Hash 5-tuple, modulo 4 → load 0-3 vào reg0
$ ovs-ofctl add-flow br0 \
    "priority=100,ip,actions=multipath(symmetric_l4,0,modulo_n,4,0,NXM_NX_REG0[0..1]),\
     resubmit(,1)"

# Table 1 output dựa trên reg0
$ ovs-ofctl add-flow br0 "table=1,reg0=0,actions=output:uplink1"
$ ovs-ofctl add-flow br0 "table=1,reg0=1,actions=output:uplink2"
$ ovs-ofctl add-flow br0 "table=1,reg0=2,actions=output:uplink3"
$ ovs-ofctl add-flow br0 "table=1,reg0=3,actions=output:uplink4"
```

So với `group type=select`:
- `multipath` lightweight, không cần group table maintenance.
- `group select` có monitoring (group stats), centralized change management.

### 4.9.26.3 Hash algorithm

| Algorithm | Ý nghĩa |
|---|---|
| `modulo_n` | hash mod n_links — simple nhưng rebalance lớn khi n_links thay đổi |
| `hash_threshold` | Threshold-based, balance tốt hơn modulo |
| `hrw` | Highest Random Weight (HRW) — minimal rebalance khi n_links thay đổi |
| `iter_hash` | Iterative hash — complex, dùng cho large n_links |

## Điểm cốt lõi cần nhớ (tier 1+2+3 hoàn thành)

Thứ nhất, action là chỉ thị thực thi trên packet (transform, forward); instruction là wrapper điều khiển pipeline. OpenFlow 1.0 gộp chung, 1.1+ tách riêng. OVS shortcut `actions=` mặc định = Apply-Actions.

Thứ hai, category Output gồm 10 action cơ bản: `output:<port>` + 9 reserved port variants (drop/normal/flood/all/controller/local/in_port/table) + `group`. Mỗi cái có semantics riêng (include ingress, respect STP, respect VLAN).

Thứ ba, explicit `drop` tốt hơn implicit (empty action list) vì tường minh. OVS 2.17+ có dedicated kernel drop path — verify qua `dpif/show-dp-features`.

Thứ tư, `group` table cung cấp indirection cho 4 pattern: multicast (all), load balance (select), shared next-hop (indirect), active-backup (fast_failover). OpenFlow 1.1+ feature foundational mà nhiều deployment chưa dùng đủ.

Thứ năm, `resubmit` là OVS NXM extension cho modular pipeline, khác `goto_table` standard. OVN translation dùng extensively.

Thứ sáu, action set execution order theo **12 priority level** cứng, không theo thứ tự ghi. Apply-Actions thì tuần tự. Hiểu khác biệt này để không bị surprise khi debug.

## 4.9.29 Backfill v3.5 — 8-attribute Anatomy completeness cho 12 action gap

> **Mục đích:** đợt audit J.1 v3.5-KeywordBackbone phát hiện 12 action từ REF chưa có 8-attribute Anatomy đầy đủ trong catalog (Template C). Section này backfill, bám sát Template C đã định nghĩa ở 4.9.1.

### 4.9.29.1 Action `copy_field` (OF 1.5+)

**Bucket:** Modification.
**Syntax:** `copy_field(src_offset:src_field, dst_offset:dst_field, n_bits)` hoặc shorthand `copy_field:SRC->DST` (full-width copy).
**Effect:** Copy `n_bits` bit từ `src_field[src_offset:]` sang `dst_field[dst_offset:]`. Source field không đổi.
**Prerequisites:** `src_field` + `dst_field` phải hợp lệ.
**OF version intro:** OF 1.5+ (action OFPAT_COPY_FIELD type 28).
**OVS version intro:** OVS 2.7+.
**Side effect:** Modify destination field.
**Example:**

```
priority=100,ip,actions=copy_field:ipv4_src->reg0,goto_table:5
priority=100,actions=copy_field(0:reg0,0:vlan_vid,16)
```

**Diễn giải vận hành:** OF 1.0-1.4 dùng OVS extension `move:SRC[]->DST[]` để copy field, nhưng `move` là NXM-only nên không portable cross-vendor. OF 1.5 standardize `copy_field` với cùng semantics. OVN có thể dùng `copy_field` thay `move` để code portable. Khác `set_field` (ghi giá trị literal): `copy_field` ghi giá trị từ field khác (dynamic).

### 4.9.29.2 Action `push_pbb` / `pop_pbb` (OF 1.3+)

**Bucket:** L2 Provider Bridge encapsulation.
**Syntax:** `push_pbb:0x88e7` / `pop_pbb`.
**Effect:** Add/remove 22-byte PBB shim header trước Ethernet customer.
**Prerequisites:** `pop_pbb` cần `eth_type=0x88e7`.
**OF version intro:** OF 1.3+ (OFPAT_PUSH_PBB type 26 + OFPAT_POP_PBB type 27).
**OVS version intro:** OVS 2.6+.
**Side effect:** `push_pbb` set new B-DA = current `eth_dst`, B-SA = `eth_src`, I-SID = 0.
**Example:**

```
actions=push_pbb:0x88e7,set_field:0x123456->pbb_isid,output:2
priority=100,eth_type=0x88e7,actions=pop_pbb,goto_table:5
```

**Diễn giải vận hành:** PBB (IEEE 802.1ah) thường gặp trong carrier Ethernet provider. OVS support PBB chủ yếu để compatibility với hardware switch carrier. Không dùng trong DC overlay (OVN dùng Geneve/VXLAN).

### 4.9.29.3 Action `set_mpls_ttl`

**Bucket:** L2 MPLS modification.
**Syntax:** `set_mpls_ttl:N` (N = 0-255).
**Effect:** Override MPLS TTL field thành N.
**Prerequisites:** MPLS (`eth_type=0x8847/0x8848`).
**OF version intro:** OF 1.1+ (OFPAT_SET_MPLS_TTL type 15).
**OVS version intro:** OVS 1.11+.
**Side effect:** Modify MPLS shim header byte 4 (TTL position).
**Example:** `priority=100,eth_type=0x8847,actions=set_mpls_ttl:64,output:2`

**Diễn giải vận hành:** Khi MPLS edge router push label mới, TTL của label outer mặc định kế thừa từ label inner. `set_mpls_ttl` cho phép edge router refresh TTL về giá trị cao trước khi forward, tránh packet bị drop ở core router với TTL count down xuống 0.

### 4.9.29.4 Action `dec_mpls_ttl`

**Bucket:** L2 MPLS modification.
**Syntax:** `dec_mpls_ttl`.
**Effect:** Decrement TTL của outer-most MPLS label by 1. Drop nếu TTL → 0.
**Prerequisites:** MPLS.
**OF version intro:** OF 1.1+ (OFPAT_DEC_MPLS_TTL type 16).
**OVS version intro:** OVS 1.11+.
**Side effect:** TTL byte modification + potential drop nếu zero.
**Example:** `priority=100,eth_type=0x8847,actions=dec_mpls_ttl,output:2`

**Diễn giải vận hành:** Mọi MPLS hop transit phải decrement TTL (tương tự IP TTL) để chống loop infinite.

### 4.9.29.5 Action `copy_ttl_in` / `copy_ttl_out`

**Bucket:** L2/L3 TTL preservation.
**Syntax:** `copy_ttl_in` / `copy_ttl_out`.
**Effect:**
- `copy_ttl_in`: copy TTL từ outer header (MPLS/IP outer) vào inner header (IP inner). Dùng khi pop encap.
- `copy_ttl_out`: copy TTL từ inner header (IP) ra outer header (MPLS). Dùng khi push encap.
**Prerequisites:** Có ít nhất 2 layer encap.
**OF version intro:** OF 1.1+ (OFPAT_COPY_TTL_IN type 11, OFPAT_COPY_TTL_OUT type 12).
**OVS version intro:** OVS 2.5+.
**Side effect:** Modify TTL byte ở header tương ứng.
**Example:**

```
priority=100,eth_type=0x8847,mpls_bos=1,actions=copy_ttl_in,pop_mpls:0x0800,goto_table:5
priority=100,ip,actions=copy_ttl_out,push_mpls:0x8847,set_field:100->mpls_label,output:2
```

**Diễn giải vận hành:** Hai action giải quyết trade-off **TTL behavior** trong MPLS: pipe mode (default, ẩn topology) vs uniform mode (`copy_ttl_*` enable, traceroute thấy đầy đủ path).

### 4.9.29.6 Action `set_nw_ttl`

**Bucket:** L3 modification.
**Syntax:** `set_nw_ttl:N` (N = 0-255).
**Effect:** Override IPv4 TTL hoặc IPv6 Hop Limit thành N.
**Prerequisites:** IPv4 hoặc IPv6.
**OF version intro:** OF 1.1+ (OFPAT_SET_NW_TTL type 23).
**OVS version intro:** OVS 1.10+.
**Side effect:** Modify IP header TTL/Hop Limit.
**Example:** `priority=100,ip,actions=set_nw_ttl:64,output:2`

**Diễn giải vận hành:** Khác `dec_ttl` (decrement -1, chuẩn router behavior), `set_nw_ttl:N` ghi đè TTL thành N. Chỉ dùng trong special case như NAT64, gateway re-stamp TTL.

### 4.9.29.7 Action `decap` (NSH detail)

**Bucket:** Encapsulation.
**Syntax:** `decap` hoặc `decap(packet_type=(NS,TYPE))`.
**Effect:** Remove innermost encapsulation header (NSH, MPLS, GRE, Geneve, etc.) và update `packet_type` field.
**Prerequisites:** Packet phải có encapsulation layer present.
**OF version intro:** OVS extension (OF 1.5+ packet-type-aware).
**OVS version intro:** OVS 2.8+ (NSH support).
**Side effect:** Modify `packet_type` field. Mất outer header data.
**Example:**

```
priority=100,nsh,nsh_si=0,actions=decap(),decap(packet_type=(0,0x0800)),output:2
```

**Diễn giải vận hành:** `decap` cốt lõi cho SFC (Service Function Chaining). Service function cuối chuỗi (SI=0) gọi `decap()` 2 lần: lần 1 remove NSH header, lần 2 revert về Ethernet để output.

### 4.9.29.8 Action `encap` (NSH detail)

**Bucket:** Encapsulation.
**Syntax:** `encap(NSH)`, `encap(ethernet)`, `encap(MPLS, mpls_label=N)`.
**Effect:** Add encapsulation header trên top current packet, update `packet_type`.
**Prerequisites:** Packet hiện tại phải tương thích inner type của encap.
**OF version intro:** OVS extension.
**OVS version intro:** OVS 2.8+.
**Side effect:** Add header, change `packet_type`.
**Example:**

```
priority=100,ip,actions=encap(nsh(md_type=1,tlv(class=0x1,type=1,len=4,value=0xdead))),
        set_field:0x123->nsh_spi,set_field:255->nsh_si,output:2
```

**Diễn giải vận hành:** `encap` là counterpart của `decap`. SFC entry router emit `encap(nsh)` để wrap traffic vào chain. NSH metadata can use MD Type 1 (fixed length 16-byte) hoặc Type 2 (variable TLV).

### 4.9.29.9 Action `controller` với userdata (OVS 2.6+ continuation-based)

**Bucket:** Control.
**Syntax:** `controller(reason=action,userdata=HEX)` hoặc `controller(reason=action,userdata=HEX,pause)`.
**Effect:** Send packet tới controller với metadata custom, optional pause pipeline để tiếp tục sau khi controller respond.
**Prerequisites:** OpenFlow channel kết nối controller.
**OF version intro:** OF 1.0+ (basic), OVS 2.6+ (continuation extension).
**OVS version intro:** OVS 2.6+.
**Side effect:** Trigger PACKET_IN message + optional pipeline pause.
**Example:**

```
priority=100,actions=controller(reason=action,userdata=0xcafe1234)
priority=100,actions=controller(reason=action,userdata=0xdead,pause)
```

**Diễn giải vận hành:** Continuation-based extension cho phép controller respond bằng PACKET_OUT với "continue" action để tiếp pipeline đã pause. OVN không dùng pattern này; chủ yếu dùng `controller` cho ARP responder fallback.

### 4.9.29.10 Action `note` (debug marker)

**Bucket:** Annotation.
**Syntax:** `note:HEXSTRING` (hex bytes inline).
**Effect:** No-op semantically. Lưu bytes vào flow definition cho debug/audit.
**Prerequisites:** Không.
**OF version intro:** OVS extension (NXAST_NOTE).
**OVS version intro:** OVS 1.0+.
**Side effect:** Không. Pure debug marker.
**Example:** `priority=100,in_port=1,actions=note:01.02.03.04.deadbeef,output:2`

**Diễn giải vận hành:** Technique cài bytes vào flow rule cho debug post-mortem. Khi `dump-flows` hiện rule, note hex visible → operator decode về metadata.

### 4.9.29.11 Action `sample` (OVS 2.5+)

**Bucket:** Monitoring.
**Syntax:** `sample(probability=N,collector_set_id=ID,obs_domain_id=ID,obs_point_id=ID)`.
**Effect:** Per-packet probability sampling. Sample (probability/65535) packet → emit IPFIX record.
**Prerequisites:** IPFIX collector_set được config qua OVSDB.
**OF version intro:** OVS extension (NXAST_SAMPLE).
**OVS version intro:** OVS 2.5+.
**Side effect:** Trigger IPFIX export (no packet modification).
**Example:**

```
priority=100,in_port=1,actions=sample(probability=655,collector_set_id=1,obs_domain_id=42,obs_point_id=100),output:2
```

**Diễn giải vận hành:** Built-in flow sampling cho IPFIX/sFlow telemetry. Probability granular 1/65535 (1% = ~655). Use case: NetFlow collection cho traffic analysis. Khác `sFlow agent` (sample mọi packet): `sample` action embedded vào flow rule (selective per-flow).

### 4.9.29.12 Action `conjunction` (OVS 2.4+) — full Anatomy

**Bucket:** Matching.
**Syntax:** `conjunction(id, k/n)` với k = clause index (1-based), n = total clauses.
**Effect:** Implement n-way conjunctive (AND) match qua flow expansion.
**Prerequisites:** Không.
**OF version intro:** OVS extension (NXAST_CONJUNCTION).
**OVS version intro:** OVS 2.4+.
**Side effect:** Trigger conjunctive match downstream (rule với `conj_id=id`).
**Example:**

```
priority=100,ip,nw_src=10.0.0.0/8,actions=conjunction(1,1/2)
priority=100,tcp,tp_dst=80,actions=conjunction(1,2/2)
priority=100,tcp,tp_dst=443,actions=conjunction(1,2/2)
priority=200,conj_id=1,actions=output:2
```

**Diễn giải vận hành:** Vấn đề: ACL "src ∈ {N IP} AND dst ∈ {M port}" nếu expand cartesian = N×M rule. Conjunction giảm xuống N+M rule + 1 verdict rule.

OVN dùng conjunction nặng cho ACL có Address_Set + Port_Group có nhiều IP/port. ACL "from-lport: ip4.src ∈ $set_a (100 IP) && ip4.dst ∈ $set_b (50 IP)" = 100×50 = 5000 rule cartesian. Với conjunction: 100+50+1 = 151 rule. Reduction 33x.

## 4.9.31 Backfill v3.6 — NXM Nicira stack + timeout actions

> **Bối cảnh.** Audit Phase 1 v3.6-ContentDepth phát hiện 3 Nicira extension action có trong REF nhưng curriculum chưa index: `fin_timeout`, `push:src`, `pop:dst`. Cả ba đều thuộc OVS 1.11+, vẫn còn dùng phổ biến trong OpenFlow rule có register manipulation. Section này thêm Anatomy 8-attribute Template C cho từng action + use case + cross-link.
>
> **Vì sao 3 action này quan trọng dù không nằm trong tier 1.** `fin_timeout` là cách rẻ nhất để cleanup nhanh TCP flow đã FIN mà không cần qua conntrack overhead. Cặp `push:` / `pop:` là stack semantics duy nhất của NXM; mọi pattern "save-modify-restore" field tạm thời (như swap `in_port` để re-execute pipeline với input giả) đều dựa vào nó. Bỏ qua 3 action này là bỏ qua một class ý tưởng (per-packet stack) khác hoàn toàn class còn lại (write-once).

### 4.9.31.1 Action `fin_timeout` (NXAST_FIN_TIMEOUT, OVS 1.11+)

**Bucket:** Connection Tracking (timeout shrink).
**Syntax:** `fin_timeout(idle_timeout=N, hard_timeout=M)` với N, M tính bằng giây. Tham số tùy chọn, có thể chỉ truyền một trong hai.
**Effect:** Khi packet match flow entry này có cờ TCP FIN hoặc RST, OVS rút ngắn `idle_timeout`/`hard_timeout` của chính flow entry đó xuống N/M giây (chỉ rút ngắn, không kéo dài). Mục tiêu: dọn flow nhanh hơn sau khi connection TCP đã đóng, thay vì chờ idle timeout mặc định (thường 300s).
**Prerequisites:** Flow entry cần match TCP packet (`tcp` hoặc `nw_proto=6`). Action chỉ có ý nghĩa với flow có timeout > 0; flow vĩnh viễn (timeout=0) không bị ảnh hưởng.
**OF version intro:** OVS extension (NXAST_FIN_TIMEOUT, không có trong OpenFlow chuẩn).
**OVS version intro:** OVS 1.11+.
**Side effect:** Cập nhật metadata của chính flow entry (qua `ofproto-dpif`), không sinh packet ra ngoài. Không tương tác với conntrack zone; cơ chế thuần OpenFlow flow lifecycle.
**Example:**

```
priority=200,tcp,tcp_flags=+fin,actions=fin_timeout(idle_timeout=5,hard_timeout=30),resubmit(,1)
priority=200,tcp,tcp_flags=+rst,actions=fin_timeout(idle_timeout=1,hard_timeout=5),resubmit(,1)
```

**Diễn giải vận hành.** Long-lived TCP flow trên data center có thể giữ flow entry trong table hàng giờ vì idle_timeout mặc định cao. Sau khi peer gửi FIN/RST, kết nối thực tế đã chết nhưng entry vẫn chiếm slot. `fin_timeout` cho phép declar "khi thấy FIN, hãy aggressively age out trong 5 giây nữa". So với conntrack `ct(commit, exec(fin_timeout=5))`: action standalone `fin_timeout` không cần state machine conntrack, đơn giản hơn cho stateless ACL có một chút awareness về connection close. Khác biệt với `learn(... fin_idle_timeout=...)` ở chỗ `learn` tạo flow entry mới trong table khác, còn `fin_timeout` thao tác trên chính flow entry hiện tại. Cross-link: 9.24 §9.24.x `ct(...)` cho phương pháp conntrack-based; 4.7 §4.7.X `learn` cho phương pháp learn-from-flow.

### 4.9.31.2 Action `push:src` (NXAST_STACK_PUSH, OVS 1.11+)

**Bucket:** Field Stack (LIFO save).
**Syntax:** `push:FIELD` hoặc `push:FIELD[bit_lo..bit_hi]` với FIELD là NXM/OXM identifier (`NXM_OF_IN_PORT[]`, `NXM_NX_REG0[0..15]`, `eth_src[]`, ...).
**Effect:** Đẩy giá trị hiện tại của `FIELD` (hoặc bit slice) lên đỉnh OVS internal per-packet stack. Stack nằm trong flow execution context, lifetime giới hạn trong 1 packet đang xử lý.
**Prerequisites:** Stack có max depth (lib/ofp-actions.c giới hạn 64 entry/packet). Push quá giới hạn → action error.
**OF version intro:** OVS extension (NXAST_STACK_PUSH, không có trong OpenFlow chuẩn).
**OVS version intro:** OVS 1.11+.
**Side effect:** Stack state thay đổi (push 1 entry); field gốc giữ nguyên giá trị. Stack reset về rỗng giữa các packet.
**Example:**

```
table=0,priority=100,in_port=1,actions=push:NXM_OF_IN_PORT[],load:0->NXM_OF_IN_PORT[],resubmit(,1),pop:NXM_OF_IN_PORT[],output:2
```

**Diễn giải vận hành.** Pattern điển hình "save-modify-restore": lưu `in_port` lên stack → đặt `in_port = 0` (reset) → resubmit cho table khác như packet vừa đến port 0 → restore `in_port` về giá trị gốc trước khi output. Cần thiết khi pipeline multi-table giả lập "packet đến từ port khác" cho mục đích testing hoặc bypass match `in_port` trong table hạ nguồn. So với `move:` (sao chép giá trị tới register tạm rồi sao chép ngược): `push:`/`pop:` không tốn register slot, không xung đột với register usage khác trong pipeline. So với `set_field:` thì `push:`/`pop:` bảo toàn giá trị gốc bằng cơ chế stack thay vì copy thủ công.

### 4.9.31.3 Action `pop:dst` (NXAST_STACK_POP, OVS 1.11+)

**Bucket:** Field Stack (LIFO restore).
**Syntax:** `pop:FIELD` hoặc `pop:FIELD[bit_lo..bit_hi]`.
**Effect:** Lấy entry trên đỉnh stack, gán giá trị vào `FIELD` (hoặc bit slice), rồi xóa entry khỏi stack. Bit width của entry pop ra phải khớp với bit width của `FIELD` đích, không thì action error.
**Prerequisites:** Stack phải có ít nhất 1 entry. Pop khi stack rỗng → action error (drop packet).
**OF version intro:** OVS extension (NXAST_STACK_POP).
**OVS version intro:** OVS 1.11+.
**Side effect:** Stack state thay đổi (pop 1 entry); `FIELD` đích bị overwrite.
**Example:**

```
priority=100,tcp,actions=push:NXM_NX_REG0[],load:0xdeadbeef->NXM_NX_REG0[],note:01.02.03,pop:NXM_NX_REG0[],output:2
```

**Diễn giải vận hành.** Trường hợp dùng: pipeline cần dùng register tạm cho computation trung gian (ví dụ tính hash) nhưng không muốn destroy giá trị reg0 ban đầu mà downstream rule còn trông cậy. Push reg0, dùng reg0 cho computation, pop reg0 trước khi output. Pop không có pair push tương ứng (mismatched stack) là lỗi lập trình điển hình; OVS ofproto/trace sẽ báo `stack underflow`. Anti-pattern: dùng `pop:` thay cho `move:` khi không có push trước đó. Cross-link: 9.25 §9.25.X `ofproto/trace` debug stack-related anomaly.

## 4.9.32 Phase G v3.7 backfill — `output` action cornerstone DEEP-20 (cohort C4)

> **Mục đích:** `output` action đã substantively cover ở §4.9.1 Category 1 (S42 tier 1). Phase G fill axis 7/14/17/19/20 cho DEEP-20 closure.

### Axis 7 importance ranking explicit

`output` là **trụ cột tuyệt đối** của OpenFlow action set. Without `output`, packet không bao giờ leaves switch — flow rule chỉ classify + modify, packet eventually drop. Mọi forwarding rule cuối cùng phải kết thúc bằng output (hoặc resubmit/goto_table chain ending in output). Tier 1 absolute parity với megaflow + ct_state.

### Axis 14 version drift detail

| Version | Behavior |
|---------|----------|
| OF 1.0 | Basic output:port_no |
| OF 1.0 | Reserved port: OFPP_FLOOD, OFPP_ALL, OFPP_CONTROLLER, OFPP_LOCAL, OFPP_NORMAL, OFPP_IN_PORT |
| OF 1.1 | Multi-table awareness (output emits từ current table or accumulated) |
| OF 1.3 | Group table reference (output to group instead of port) |
| OF 1.5 | Egress table support (post-output egress processing) |
| OVS extension | output:NXM (dynamic port from register), output:CONTROLLER:max_len |

### Axis 17 production case

2024-Q2 production cluster: TCP traffic asymmetric routing, return path drop intermittent. Investigation: ofproto/trace shows packet matched correctly but `actions=output:5` evaluated khi port 5 transient down. OVS does NOT auto-reroute on port down → silent drop.

**Root cause:** Static output:port without group bucket fallback.

**Fix:** Migrate to group table với fast-failover bucket. `actions=group:1` where group 1 has 2 buckets (port 5 primary, port 6 backup). When port 5 down, group auto-shift to port 6.

### Axis 19 failure mode

- Mode A: Output to non-existent port → silent drop. Diagnostic: ofproto/trace shows port number invalid. Recovery: validate port_no before flow install.
- Mode B: Output to OFPP_IN_PORT cause infinite loop nếu in_port is also output target. Recovery: explicit different output port.
- Mode C: Output to NORMAL port nhưng L2 learning chưa hoàn thành → flood instead. Recovery: prime MAC table.
- Mode D: Output to physical port với MTU mismatch → frame fragment fail. Recovery: configure MTU consistent.

### Axis 20 cross-domain comparison

- **Cisco IOS interface fwd:** `interface GigabitEthernet0/1` similar concept, lower-level. OF output is platform-neutral abstraction.
- **Linux bridge `bridge fdb add`:** MAC table-based forwarding. OF output is rule-based per-flow, more flexible.
- **DPDK rte_eth_tx_burst:** transmit packet to physical port. OF output is logical abstraction; rte_eth direct hardware.
- **VPP interface output:** vector packet processing similar concept. Khác: VPP optimized batch dispatch, OVS per-packet.

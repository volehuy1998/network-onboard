## Bối cảnh: từ 1.0 tới 1.1, hai năm để giải hạn chế table đơn

Phần 3.1 đã phân tích kỹ limitation của OpenFlow 1.0: single flow table với cross-product explosion khi policy có nhiều dimension orthogonal. Spec 1.0.0 công bố 31/12/2009, cộng đồng implement và deploy trong 2010 nhanh chóng gặp vấn đề scale. Google B4 WAN (deployed 2011, paper SIGCOMM 2013), các Stanford research deployment trên GENI, và hyperscaler internal testing đều báo cáo cùng conclusion: single-table không scale cho policy DC-level phức tạp.

ONF chưa thành lập (sẽ thành lập 21/03/2011, Phần 3.2), nên Stanford Clean Slate Lab tiếp tục shepherd spec evolution. Quyết định của Stanford team: thay vì chờ ONF ready, release spec 1.1 càng sớm càng tốt để industry có gì mới để implement. Kết quả: OpenFlow 1.1.0 release ngày **28/02/2011**, đúng ba tuần trước khi ONF chính thức thành lập (21/03/2011).

OF 1.1 là spec cuối cùng do Stanford shepherd. Sau chuyển IP ownership sang ONF, mọi version 1.2+ đều do ONF publish với copyright ONF. Nhưng spec 1.1 đã đặt nền móng kiến trúc multi-table pipeline + group table mà 1.2 đến 1.5.1 chỉ mở rộng, không thay đổi căn bản.

## 4.0.1 OF 1.1 phát hành 28/02/2011: spec cuối cùng do Stanford chỉ dẫn (trước ONF)

### Dòng thời gian và bối cảnh tổ chức

Spec **OpenFlow Switch Specification Version 1.1.0** release ngày 28/02/2011, 14 tháng sau 1.0.0. Wire protocol version **0x02** (vs 0x01 của 1.0). Document 56 trang (tăng từ 42 trang của 1.0), ghi nhận scope mở rộng đáng kể.

Copyright page của spec ghi:

> *"OpenFlow Switch Specification, Version 1.1.0 Implemented (Wire Protocol 0x02), February 28, 2011. Copyright © 2008, 2009, 2010, 2011 Stanford University. Permission is granted..."*

Chưa thấy ONF trong copyright. Chuyển IP ownership xảy ra ngày 21/03/2011 khi ONF formation, nhưng spec 1.1 đã hoàn thành trước đó. Đây là lý do 1.1 là "last Stanford-shepherded spec" trong lịch sử OpenFlow.

### Động lực chính: giải quyết bùng nổ cross-product

Phần 3.1 §3.1.7 đã phân tích cross-product problem của single-table 1.0: với N VLAN × M IP subnet × K ACL rule, cần N·M·K flow entry. Với realistic numbers (20 tenant × 100 subnet × 10 ACL = 20,000 entry), vượt capacity của TCAM hardware commodity (1500-4000 entry).

Multi-table pipeline giải by **nested lookup** thay vì flat lookup. Table 0 match dimension 1 (ví dụ VLAN), Table 1 match dimension 2 (ví dụ IP subnet), Table 2 match dimension 3 (ví dụ ACL). Tổng entry count = N + M + K thay vì N·M·K, linear thay vì multiplicative.

Với cùng numbers above: 20 + 100 + 10 = 130 entry, giảm 150× so với cách tiếp cận flat. Easily fit trong hardware table 4K entry, còn dư chỗ cho expansion.

### Những mở rộng khác của 1.1

Ngoài multi-table pipeline (đóng góp chính), 1.1 thêm:

- **Group table** (mới): abstraction cho multicast, load balancing, failover không cần controller intervention.
- **MPLS support native**: actions PUSH_MPLS, POP_MPLS, match fields MPLS label/TC/BoS.
- **Virtual ports**: khái niệm logical port có thể map sang multiple physical port hoặc function.
- **Flow entry matching extensions**: match on MPLS fields, tunnel ID.
- **Controller roles initial** (chỉ preliminary, fully developed ở 1.2): master/slave model cho HA controller deployment.

## 4.0.2 Ngữ nghĩa pipeline nhiều table

### Flow của gói tin qua pipeline

OF 1.1 spec §5.2 định nghĩa flow processing qua multi-table pipeline:

1. **Ingress**: gói tin đến switch port, được parse thành internal representation (extract header fields).
2. **Table 0 lookup**: gói tin matched với flow entries trong Table 0. Nếu match:
   - Instructions được execute (actions hoặc goto_table).
   - Nếu goto_table = N (N > 0), gói tin forward sang Table N.
   - Nếu không có goto_table, pipeline stop và action set được execute.
3. **Subsequent tables**: nếu goto_table, repeat lookup ở table đích.
4. **Action set execution**: sau khi pipeline stop, execute tất cả actions trong action set (tích lũy qua pipeline).
5. **Egress**: gói tin forward ra port theo OUTPUT action trong action set.

### Constraint quan trọng: monotonic progression

Spec §5.2 specify: **goto_table target phải có số cao hơn current table**. Không được goto backward hoặc same-table. Lý do: chống loop. Nếu cho goto backward, gói tin có thể bị loop vô hạn giữa hai table (A → B → A → B → ...).

Constraint monotonic đảm bảo pipeline là acyclic DAG (directed acyclic graph). Mỗi gói tin traverse pipeline trong finite number of table (≤ số table), không thể loop.

### Action set vs action list: khác biệt chính

**Action list** (từ OF 1.0, vẫn valid trong 1.1): list actions execute theo thứ tự xuất hiện, như sequence statement trong programming language. Instruction APPLY_ACTIONS trong 1.1 wrap action list, execute ngay sau khi match.

**Action set** (mới trong 1.1): không-ordered set actions, mỗi action type chỉ có một instance (last-write-wins). Execute một lần ở cuối pipeline, theo fixed order quy định bởi spec §5.10 (copy_ttl_in, pop, push-MPLS/PBB/VLAN, copy_ttl_out, decrement_ttl, set, qos, group, output).

Instruction WRITE_ACTIONS add action vào action set; action set accumulate qua pipeline. Instruction CLEAR_ACTIONS empty action set.

Distinction quan trọng vì: action list cho phép intermediate effects (set field rồi resubmit), action set cho phép accumulate final ý đồ cấu hình từ nhiều table rồi execute một lần.

### Ví dụ concrete: multi-table với action set

Scenario: Table 0 classify theo VLAN (tenant), Table 1 apply QoS, Table 2 apply routing, Table 3 forward.

```
Table 0:
  match: dl_vlan=100
  instructions: APPLY_ACTIONS(set_metadata=0x01), GOTO_TABLE=1

Table 1:
  match: metadata=0x01
  instructions: WRITE_ACTIONS(set_queue=5), GOTO_TABLE=2

Table 2:
  match: nw_dst=10.0.0.0/8
  instructions: WRITE_ACTIONS(set_dl_dst=00:11:22:33:44:55), GOTO_TABLE=3

Table 3:
  match: * (default)
  instructions: WRITE_ACTIONS(output=port 5)
```

Gói tin flow:
- Table 0 match VLAN 100, set metadata, goto Table 1
- Table 1 match metadata, write set_queue action (accumulate in action set), goto Table 2
- Table 2 match destination, write set_dl_dst action, goto Table 3
- Table 3 match default, write output action
- Pipeline end, execute action set: set_queue → set_dl_dst → output

Tổng 4 entry thay vì 1 flat entry chứa tất cả. Mỗi table match độc lập, composable.

> **Chủ đề trọng tâm:** Multi-table pipeline là architectural shift, không chỉ feature addition. Spec từ 1.1 trở đi đều dùng pipeline + action set model. OVS internally model pipeline như acyclic DAG, optimize execution qua megaflow cache. Developer writing OpenFlow controller cần hiểu pipeline semantics để avoid common pitfall (WRITE vs APPLY confusion, action set ordering).

## 4.0.3 Instructions so với actions: điểm khác biệt then chốt từ 1.1

### Instruction framework

Spec 1.1 §5.9 định nghĩa 5 instruction types:

**APPLY_ACTIONS**: execute action list ngay, before going to next table. Giống hành vi OF 1.0 (nơi mọi action execute ngay). Use case: intermediate effect cần visible tại subsequent table match.

**WRITE_ACTIONS**: add action vào action set (tích lũy). Không execute ngay. Execute tại cuối pipeline sau khi mọi table processed. Use case: accumulate final ý đồ cấu hình từ nhiều table.

**CLEAR_ACTIONS**: empty action set hiện tại. Use case: rare, khi cần "reset" decision từ table trước.

**WRITE_METADATA**: set metadata field (64-bit register cross-table). Metadata persist qua pipeline, dùng để pass context giữa table.

**GOTO_TABLE**: direct gói tin sang table kế tiếp. Target table phải có ID cao hơn current.

### Fixed execution order

Spec §5.9.1 specify: khi một flow entry có multiple instructions, execute theo fixed order:

1. METER (OF 1.3+)
2. APPLY_ACTIONS
3. CLEAR_ACTIONS
4. WRITE_ACTIONS
5. WRITE_METADATA
6. GOTO_TABLE

Order này fixed, controller cannot override. Controller chỉ chọn instructions nào include, order execute deterministic.

### Distinction quan trọng

| Aspect | APPLY_ACTIONS | WRITE_ACTIONS |
|--------|---------------|---------------|
| Timing | Execute ngay, inline | Accumulate, execute cuối pipeline |
| Ordering | List order (như 1.0) | Set order (spec fixed) |
| Visibility | Intermediate effect visible | Only final result visible |
| Use case | Table-local transformation | Accumulate cross-table ý đồ cấu hình |

Common pitfall: developer dùng APPLY_ACTIONS với set_field khi cần WRITE_ACTIONS, resulting inconsistent behavior khi multiple table set same field. Luôn check ý đồ cấu hình: nếu cần final effect duy nhất, dùng WRITE.

## 4.0.4 Group table: bốn loại cho multicast, cân bằng tải, chuyển đổi dự phòng

### Group abstraction

Spec 1.1 §5.6 introduce **group table**, abstraction cho scenarios mà flow entry đơn không đủ express:

- Multicast (replicate gói tin ra nhiều port)
- Load balancing (choose một trong nhiều next-hop theo hash)
- Failover (switch sang backup nếu primary down)
- Shared action set (nhiều flow entry cùng dùng một action sequence)

Group table có group entries, mỗi entry có group_id + type + bucket list. Bucket là tập actions + optional weight/watch_port/watch_group.

Action `GROUP` reference group_id trong flow entry. Khi gói tin match flow entry có action GROUP, gói tin được submit đến group, bucket(s) execute theo type.

### Bốn group types

**Type ALL** (multicast, flooding):
- Execute TẤT CẢ buckets cho mỗi gói tin
- Each bucket produces a clone của gói tin với different actions
- Use case: multicast group, mirror traffic ra multiple port, BUM replication trong overlay
- Example: bucket 1 = output:port 2, bucket 2 = output:port 3, bucket 3 = output:port 4 → gói tin được clone ra 3 port

**Type SELECT** (load balancing):
- Execute MỘT bucket duy nhất, chosen by switch implementation
- Algorithm implementation-defined, thường là hash (e.g., hash trên 5-tuple modulo bucket count)
- Bucket có optional weight cho weighted load balancing
- Use case: ECMP multi-path routing, load balancing backend servers
- Example: bucket 1 weight=50 output:port 2, bucket 2 weight=50 output:port 3 → gói tin route one path hoặc other 50/50

**Type INDIRECT** (single bucket, shared):
- Chỉ có MỘT bucket duy nhất, execute cho mỗi gói tin
- Conceptually giống action list direct, nhưng lợi ích: nhiều flow entry reference cùng group
- Shared action set: update action một lần ở group, thay vì update N flow entries
- Use case: common next-hop shared bởi nhiều destination; centralize policy change
- Example: group indirect với bucket = set_dl_dst + output:uplink; 1000 flow entry cho 1000 destination reference group này; khi next-hop MAC change, chỉ update group bucket

**Type FAST_FAILOVER** (sub-10ms reroute):
- Execute bucket đầu tiên có `watch_port` hoặc `watch_group` status UP (alive)
- Switch monitor port/group status, auto-switch bucket khi primary down
- Failover time limited by switch's ability to detect down status (typically < 10 ms với carrier lost detection on port)
- Use case: fast reroute trong backbone link protection, MPLS FRR equivalent without controller involvement
- Example: bucket 1 watch_port=primary_port output:primary_nexthop, bucket 2 watch_port=backup_port output:backup_nexthop → khi primary port down, switch tự switch sang bucket 2

### Fast_failover: sub-10ms reroute không qua controller

FAST_FAILOVER đặc biệt quan trọng. Trong legacy routing, failover phụ thuộc routing protocol convergence (OSPF LSA flooding, SPF recomputation): typical 200ms-2s. Trong OpenFlow controller-based, failover depends on controller RTT + reinstall flow: typical 50-500ms.

FAST_FAILOVER cho phép switch tự reroute trong microseconds: chỉ cần detect port down (typically done by hardware port status interrupt < 1ms), switch the active bucket, packets start flowing through backup path. Không cần controller notification, không cần new flow install.

Comparison:
- OSPF failover: 200-2000 ms
- BGP failover: 10-180 seconds (without fast feature)
- OpenFlow controller-based failover: 50-500 ms
- OpenFlow FAST_FAILOVER: < 10 ms typical

Use case compelling: ISP backbone với link protection requirement, hyperscale DC với high-availability SLA, WAN SDN (Google B4) dùng FAST_FAILOVER cho link failover.

> **Chủ đề trọng tâm:** Group table, đặc biệt FAST_FAILOVER, là feature OpenFlow cạnh tranh trực tiếp với MPLS Fast Reroute (FRR) cho WAN/service provider use case. Group SELECT thay ECMP routing. Group ALL thay IGMP multicast snooping. Mỗi group type address một pain point cụ thể của legacy protocols, justify OpenFlow value proposition cho DC/WAN scenarios phức tạp.

## 4.0.5 Hỗ trợ MPLS nguyên bản

### MPLS actions và match fields

Trước 1.1, OpenFlow không hỗ trợ MPLS; muốn implement MPLS with OF 1.0 cần dùng vendor extension hoặc custom action. 1.1 chính thức add MPLS vào core spec:

**MPLS actions (5 types)**:

- **PUSH_MPLS**: add MPLS label lên đầu gói tin (outermost). Ethertype bên ngoài set thành 0x8847 (MPLS unicast) hoặc 0x8848 (MPLS multicast).
- **POP_MPLS**: remove outermost MPLS label. Nếu no more label, Ethertype revert.
- **SET_MPLS_TTL**: set TTL của outermost MPLS label.
- **DEC_MPLS_TTL**: decrement TTL bởi 1. Nếu TTL = 0, gói tin drop.
- **COPY_TTL_IN / COPY_TTL_OUT**: copy TTL giữa IP và MPLS headers for proper TTL propagation.

**MPLS match fields (3 types)**:

- **mpls_label**: 20-bit label value
- **mpls_tc** (Traffic Class): 3-bit, tương ứng EXP bits trong MPLS header
- **mpls_bos** (Bottom of Stack): 1-bit, indicate whether this is last label in stack

### Ý nghĩa: OpenFlow vào WAN / carrier network

MPLS (RFC 3031, Rosen et al., January 2001) là de-facto transport cho service provider backbone. Network operator build MPLS core cho VPN (L2VPN, L3VPN, MPLS Traffic Engineering), và mọi vendor core router (Cisco, Juniper, Nokia Alcatel-Lucent) support MPLS native.

Trước 1.1, OpenFlow không được xem là viable cho WAN use case vì thiếu MPLS. 1.1 open cửa: operator có thể deploy OpenFlow switch trong MPLS backbone, với controller drive label distribution thay vì LDP/RSVP-TE distributed protocol.

Google B4 WAN (paper SIGCOMM 2013, Jain et al.) là deployment production đầu tiên sử dụng cách tiếp cận OpenFlow-like (not strictly OpenFlow protocol, but conceptually similar) cho WAN traffic engineering. B4 implement MPLS fast reroute (FRR) via OpenFlow group FAST_FAILOVER equivalent.

### Use case: ingress classify → MPLS encapsulation

Pattern phổ biến trong WAN deployment:

Table 0 classify traffic theo (source, destination):
```
match: in_port=X, nw_dst=10.1.0.0/16
instructions: APPLY_ACTIONS(push_mpls:0x8847, set_mpls_label=100, set_mpls_ttl=255),
              GOTO_TABLE=1
```

Table 1 forward theo MPLS label:
```
match: mpls_label=100
instructions: APPLY_ACTIONS(output:port_to_next_hop)
```

Gói tin từ local tenant qua ingress port X với destination 10.1.0.0/16 được push label 100, sau đó forward theo label. Đây là simplified MPLS VPN pattern, controller cấu hình thay vì LDP protocol.

Forward reference đến Phần 4.7 (OpenFlow programming với ovs-ofctl cho MPLS push/pop trong practice) để thấy exact syntax `ovs-ofctl add-flow` cho MPLS operations.

## 4.0.6 Use case: multi-tenant classifier → tunnel

### Pattern triển khai DC multi-tenant

Multi-tenant DC với 20 tenant, mỗi tenant có traffic VLAN riêng, cần forward qua tunnel (VXLAN equivalent nhưng express bằng OpenFlow). Use case minh họa power của multi-table pipeline.

**Cách tiếp cận flat với OF 1.0 (single table)**:

```
# Tenant 1 traffic đi tunnel 1
match: dl_vlan=100, nw_dst=10.1.0.0/16 → action: set_tunnel=1, output:port_tun_1
match: dl_vlan=100, nw_dst=10.1.1.0/24 → action: set_tunnel=1, output:port_tun_1
... (M destination per tenant) ...
match: dl_vlan=100, nw_dst=10.1.M.0/24 → action: set_tunnel=1, output:port_tun_1
# Tenant 2 traffic đi tunnel 2
match: dl_vlan=200, nw_dst=10.2.0.0/16 → action: set_tunnel=2, output:port_tun_2
... ... ...
# Tenant 20 traffic đi tunnel 20
match: dl_vlan=4000, nw_dst=10.20.M.0/24 → action: set_tunnel=20, output:port_tun_20
```

Total entries: 20 tenant × M destination = 20M entries. Với M = 100, = 2000 entries.

**Cách tiếp cận multi-table với OF 1.1**:

```
Table 0 (classify tenant):
  match: dl_vlan=100 → WRITE_METADATA=0x01, GOTO_TABLE=1
  match: dl_vlan=200 → WRITE_METADATA=0x02, GOTO_TABLE=1
  ...
  match: dl_vlan=4000 → WRITE_METADATA=0x14, GOTO_TABLE=1

Table 1 (route destination):
  match: nw_dst=10.1.0.0/16 → WRITE_ACTIONS(set_tunnel=1, output:port_tun_1)
  match: nw_dst=10.1.1.0/24 → WRITE_ACTIONS(set_tunnel=1, output:port_tun_1)
  ... ...
```

Wait, multi-table not sufficient yet. Table 1 cần check metadata too để không conflict giữa tenant. Correction:

```
Table 0 (classify tenant):
  match: dl_vlan=100 → WRITE_METADATA=0x01, GOTO_TABLE=1

Table 1 (tenant-aware routing):
  match: metadata=0x01, nw_dst=10.1.0.0/16 → WRITE_ACTIONS(set_tunnel=1, output:port_tun_1)
  match: metadata=0x02, nw_dst=10.2.0.0/16 → WRITE_ACTIONS(set_tunnel=2, output:port_tun_2)
```

Vẫn multiplicative. Tốt hơn là:

```
Table 0 (classify + tag):
  match: dl_vlan=100 → WRITE_ACTIONS(set_tunnel=1), GOTO_TABLE=1

Table 1 (route by IP):
  match: nw_dst=10.1.0.0/16 → WRITE_ACTIONS(output:port_tun_1)
```

Đây là cách tiếp cận additive: N tenant + M destination = 20 + 100 entries = 120. So với 2000 entry flat, saving 94%.

### Comparison concrete

| Metric | Flat (1.0) | Multi-table (1.1) |
|--------|-----------|-------------------|
| Total entries | N × M = 2000 | N + M = 120 |
| Scale formula | O(N·M) | O(N+M) |
| Hardware table fit (4K) | ~2 tenant max | >30 tenant |
| Policy change: add 1 tenant | M entries | 1 entry table 0 |
| Policy change: add 1 subnet | N entries | 1 entry table 1 |

Cách tiếp cận multi-table scale independently per dimension, policy change locally, dễ maintain.

Đây là minh chứng rõ rệt why multi-table pipeline là architectural feature, không chỉ nice-to-have. Với DC multi-tenant scale, single-table 1.0 không viable; multi-table 1.1+ là requirement.

## ▶ Guided Exercise — Implement 3-stage multi-table pipeline với OVS (audit P4.B4.1 hands-on)

**Mục tiêu:** Hands-on thực hiện kiến trúc multi-table OpenFlow 1.1 trên OVS, verify giảm cross-product từ O(N·M) sang O(N+M).

**Môi trường:** Ubuntu 22.04 + OVS 2.17.9 + 3 veth pair mô phỏng tenant tap.

**Bước:**

1. Tạo OVS bridge với OF 1.3 (OVS implement OF 1.1+ features qua OF 1.3 protocol):
   ```bash
   $ sudo ovs-vsctl add-br br-mt -- set bridge br-mt protocols=OpenFlow13
   $ for i in 1 2 3; do
       sudo ip link add veth-t$i type veth peer name peer-t$i
       sudo ip link set veth-t$i up; sudo ip link set peer-t$i up
       sudo ovs-vsctl add-port br-mt veth-t$i
     done
   ```
2. Install multi-table pipeline — Table 0 classify tenant, Table 1 route, Table 2 output:
   ```bash
   # Table 0: classify tenant theo VLAN → set metadata
   $ sudo ovs-ofctl -O OpenFlow13 add-flow br-mt \
       "table=0,priority=100,dl_vlan=100,actions=write_metadata:0x01,goto_table:1"
   $ sudo ovs-ofctl -O OpenFlow13 add-flow br-mt \
       "table=0,priority=100,dl_vlan=200,actions=write_metadata:0x02,goto_table:1"

   # Table 1: route theo destination → set tunnel
   $ sudo ovs-ofctl -O OpenFlow13 add-flow br-mt \
       "table=1,priority=100,metadata=0x01,nw_dst=10.1.0.0/16,\
        actions=set_field:1->tun_id,goto_table:2"

   # Table 2: output
   $ sudo ovs-ofctl -O OpenFlow13 add-flow br-mt \
       "table=2,priority=100,tun_id=1,actions=output:2"
   ```
3. Verify pipeline entry count (formula O(N+M) vs O(N·M)):
   ```bash
   $ sudo ovs-ofctl -O OpenFlow13 dump-flows br-mt | wc -l
   # Kỳ vọng: ~5-6 rules (2 tenants + 1 route + 1 output)
   ```
4. Trace gói tin để verify flow:
   ```bash
   $ sudo ovs-appctl ofproto/trace br-mt \
       "in_port=veth-t1,dl_vlan=100,ip,nw_dst=10.1.0.5"
   # Observe: Table 0 match → Table 1 → Table 2 → output
   ```

**POE:**
- **Predict:** Với 10 tenant × 100 destination, single-table 1.0 cần bao nhiêu rule? Multi-table 1.1 cần bao nhiêu?
- **Observe:** Single-table: 10 × 100 = 1000 rule. Multi-table: 10 + 100 = 110 rule. Giảm 9×.
- **Explain:** Multi-table additive thay vì multiplicative — scale linear theo tổng dimension, không phải product.

**Cleanup:**
```bash
$ sudo ovs-vsctl del-br br-mt
$ for i in 1 2 3; do sudo ip link del veth-t$i; done
```

## Mạch nối với phần sau

Phần 4.0 đã analyze OpenFlow 1.1 với ba feature chính: multi-table pipeline, group table, và MPLS support. Phần 4.1 tiếp theo analyze OF 1.2 (release 05/12/2011, first ONF-published spec) với OXM extensible match (kế thừa NXM của OVS) và controller roles EQUAL/MASTER/SLAVE cho HA controller deployment.

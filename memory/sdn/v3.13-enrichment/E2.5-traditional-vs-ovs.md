`xlate_normal` is the single function that implements MAC learning. The first block of the function asks what wildcards the resulting megaflow rule will need:

```c
static void
xlate_normal(struct xlate_ctx *ctx)
{
    struct flow_wildcards *wc = ctx->wc;
    struct flow *flow = &ctx->xin->flow;
    struct xbundle *in_xbundle;
    struct xport *in_port;
    struct mac_entry *mac;
    void *mac_port;
    uint16_t vlan;
    uint16_t vid;

    memset(&wc->masks.dl_src, 0xff, sizeof wc->masks.dl_src);
    memset(&wc->masks.dl_dst, 0xff, sizeof wc->masks.dl_dst);
    wc->masks.vlans[0].tci |= htons(VLAN_VID_MASK | VLAN_CFI);
```

The function ends by calling `update_learning_table` for the source MAC and either flooding or unicasting depending on whether the destination was found. The traditional-switch equivalent is "the ASIC's MAC-learning logic plus the SDK that programs the MAC table"; OVS's equivalent is this 200-line C function. The R2.5 lab will demonstrate by sending a frame and reading both the OVS state (`ovs-appctl fdb/show`) and the kernel-bridge state (`bridge fdb show` against a parallel Linux bridge) and observing identical learning behaviour on the wire.

**Plan-cited symbol drift note (Rule 14.4).** Plan §R2.5 cites `bond_choose_output_slave()` as the bond hash function. In v2.17.9 the function is renamed to `bond_choose_output_member` (with the lower-level helper `choose_output_member`). The rename is part of the upstream "slave to member" terminology refresh that landed across multiple OVS versions. R2.5 cites the v2.17.9 names.

The `bond_choose_output_member` function is short and pedagogically loaded:

```c
struct bond_member *
bond_choose_output_member(struct bond *bond, const struct flow *flow,
                          struct flow_wildcards *wc, uint16_t vlan)
{
    struct bond_member *member;

    member = choose_output_member(bond, flow, wc, vlan);
    if (member) {
        return member;
    }
    return NULL;
}
```

The decision logic is delegated to `choose_output_member` which selects based on `bond_mode`. For `balance-tcp` the function uses a 5-tuple flow hash; for `balance-slb` it uses the source MAC; for `active-backup` it always returns the active member. R2.5 will demonstrate all three modes by sending traffic across a 2-link bond and reading per-flow distribution via `ovs-appctl bond/show`.

### A.6. L3 routing (no OVN)

A traditional router has interfaces with IP addresses, an ARP cache, and a forwarding-information-base. OVS does **none** of that natively. The "routing through OVS flows" pattern is the operator deciding the next-hop MAC and TTL decrement and writing both into the flow table; OVS just executes.

### B.1. `Documentation/topics/bonding.rst` (mandatory)

**Synopsis.** The canonical OVS bonding documentation. Walks the three bond modes (`active-backup`, `balance-slb`, `balance-tcp`), the LACP active and passive variants, and the per-mode hash function. R2.5 sub-task 4 draws every command from this document.

**Verbatim quote (the bond-mode table).**

> ::
>
>     active-backup
>         There are no requirements for the upstream switch.  This is the
>         simplest mode and the most likely to work in any environment.
>     balance-slb
>         The upstream switch can be a dumb hub or switch.  Each VM's MAC
>         address is hashed to a slave.  Per-VM bandwidth is limited to that
>         of one slave.
>     balance-tcp
>         Requires the upstream switch to support 802.3ad with successful
>         LACP negotiation.

The "no requirements for the upstream switch" property of `active-backup` is the difference from a Cisco LACP bond, which requires both ends to negotiate. R2.5 cites this as a key OVS-vs-traditional difference.

## Section E. Edge cases and hidden constraints

### E.1. The implicit `NORMAL` action is not in OVSDB

A bridge with `fail-mode=standalone` (default) shows one row in `dump-flows` (`priority=0,actions=NORMAL`), but that row is **runtime-synthesised** by `connmgr`, not stored in OVSDB. **Why hard:** an operator who exports OVSDB and replays it on another host expects every flow row to come back; the implicit `NORMAL` does not. The other host re-synthesises, but the appearance is misleading.

### E.2. `actions=NORMAL` is an OVS extension to OpenFlow 1.0

OFP 1.0 reserves `OFPP_NORMAL = 0xfffa` for "subject to normal L2/L3 processing". The OVS-specific behaviour (run `xlate_normal`) is not in the OF spec; a strict OF controller cannot assume `output:NORMAL` will work. **Why hard:** when porting flow rules between OVS and a hardware OF switch, `output:NORMAL` may behave differently or be rejected.

### E.3. STP and RSTP are mutually exclusive per bridge

`stp_enable=true` plus `rstp_enable=true` produces a log error and only the first one takes effect. **Why hard:** silent conflict; reading OVSDB shows both columns set.

### E.4. RSTP edge-port misconfiguration

If an RSTP edge port (a port leading to a host, not another switch) is misconfigured as a non-edge port, every link state change triggers a topology-change BPDU storm. **Why hard:** the misconfiguration is invisible in normal traffic; only convergence events expose it.

### E.5. STP convergence with one OVS-side and one hardware-side bridge

If one end of the link runs OVS-RSTP and the other runs a Cisco-RSTP, the BPDU rates differ slightly. **Why hard:** OVS sends RSTP BPDUs every 2 seconds (the spec default) while some Cisco platforms default to 5; mismatch causes premature port transitions on the OVS side.

### E.6. LACP slow versus fast timer mismatch

If one end is `lacp-time=fast` (1-second LACPDUs) and the other is `lacp-time=slow` (30-second LACPDUs), the bond establishes but flaps on subnet-level packet loss. **Why hard:** asymmetric defaults; both ends must agree.

### E.7. `balance-tcp` polarization

For long-lived flows on a small bond, the hash distribution can put more than half the bandwidth on one member. **Why hard:** the hash is deterministic; identical 5-tuples always pick the same member.

### E.8. `active-backup` does not require LACP

A common misconception: `active-backup` is set up without LACP, but operators add `lacp=active` thinking it will help. It hurts: LACP negotiation tries to bring both members up, which `active-backup` then forces back down. **Why hard:** the symptom is link flap.

### E.9. Mirror loop with mirror destination on the same bridge

If `output-port` of a mirror is a port on the same bridge whose traffic is also mirrored, every mirrored packet enters the mirror again. **Why hard:** kernel ring-buffer fill in seconds; the host appears to hang.

### E.10. Mirror plus VLAN tag stripping

A mirror with `output-vlan` set strips the original VLAN tag and pushes the configured one. **Why hard:** `tcpdump` on the destination port may see frames with unexpected VLAN IDs.

### E.11. The `select_all=true` mirror covers every port, including patch ports

Patch ports are mirrored too. A two-bridge setup with a patch pair and a `select_all` mirror on one bridge captures patch traffic. **Why hard:** unexpected duplication when the same logical packet is mirrored on both bridges.

### E.12. `dec_ttl` on a packet with TTL 1 sends an ICMP Time Exceeded back

The kernel datapath generates the ICMP automatically; the OVS bridge does not need to install rules for this case. **Why hard:** an operator expects a drop and gets a reply.

### E.13. Raw OVS routing without ARP responder

Raw OVS L3 routing (R2.5 sub-task 6, second form) requires the operator to install both forward and reverse flows. The reverse flow must rewrite source MAC and TTL. Without it, return traffic is dropped. **Why hard:** asymmetric config, easy to forget.

### E.14. The kernel routing table in the router-namespace is canonical, OVSDB is a hint

R2.5 sub-task 6 first form uses an `internal` port in a namespace as a router. The kernel routes; OVS forwards. If `ip route` inside the namespace and the OVS flow table disagree about the next hop, the kernel wins. **Why hard:** dual source of truth.

### E.15. Stateful ACL with `ct(commit)` requires zone scoping

If two unrelated flows share `ct(zone=0)`, one flow's state can collide with another's. **Why hard:** zone planning is needed but not obvious.

### E.16. NAT helper modules

`ct(nat)` for FTP, SIP, and other application-layer protocols requires kernel `nf_nat_<helper>` modules. **Why hard:** without them, the connection passes but the data channel fails.

### E.17. Conntrack timeout misalignment

The conntrack timeout for `ESTABLISHED` TCP defaults to 5 days; an OVS bridge that drops idle conntrack entries earlier than the kernel kills the conntrack but not the kernel's view. **Why hard:** asymmetric timers cause "ghost" connections.

### E.18. QoS HTB plus root qdisc replacement

Setting `qos=@new` on an OVS port replaces the kernel's root qdisc. If the host already had a qdisc (set by the network manager), it is overwritten. **Why hard:** silent overwrite of host config.

### E.19. `min-rate` plus `max-rate` semantics

In HTB, `min-rate` is the guaranteed bandwidth, `max-rate` is the ceiling. If the sum of `min-rate` across queues exceeds the parent's `max-rate`, HTB rejects the configuration. **Why hard:** the rejection is visible only in `tc -s class show`, not in OVSDB.

### E.20. Policer versus shaper

`linux-policer` drops or marks excess packets; `linux-htb` queues them. The two have different latency profiles. **Why hard:** the OVSDB `qos type` value is the only difference.

### E.21. The DHCP responder gap

OVS has no DHCP. An operator who tries `ovs-vsctl set bridge br0 other_config:dhcp_server=true` succeeds (the column is silently accepted) but no DHCP traffic is generated. **Why hard:** the silent-no-op key is the most-asked R2.5 question.

### E.22. ARP responder gap

Same shape as E.21. OVN has an ARP responder; OVS does not. **Why hard:** raw OVS routing has to rely on the kernel ARP cache in a router-namespace.

### E.23. The `dnsmasq` workaround for DHCP

Attaching `dnsmasq` to an internal port works, but the port must be in the right netns. **Why hard:** namespace isolation makes the config sequence finicky.

### E.24. SNMP agent absence

OVS has no SNMP. The closest equivalent is `coverage/show` plus `vlog/list`, which is structurally different from a MIB. **Why hard:** SNMP-based monitoring tools cannot consume OVS state directly.

### E.25. NetFlow versus sFlow versus IPFIX

OVS supports all three. They differ in what they sample, when they export, and what protocol they speak. **Why hard:** an operator who picks one cannot easily migrate to another mid-deployment.

### E.26. Cisco STP root-flap versus OVS controller-disconnect

Different failure modes. Cisco loses STP root if the root bridge dies; convergence in 30 seconds (legacy STP) or 1-6 seconds (RSTP). OVS with no controller does not lose anything (the implicit `NORMAL` keeps forwarding). **Why hard:** the fault domain is different.

### E.27. Cisco SPAN copies on egress versus OVS mirror copies on the matched flow

Cisco SPAN copies frames after egress queueing. OVS mirror copies during translation. **Why hard:** the timing of the copy differs by ~milliseconds; for high-precision timing analysis this matters.

### E.28. Cisco LACP timeout-mode in seconds, OVS in milliseconds

Cisco LACP `lacp rate fast` sets the timeout to 1 second; OVS uses millisecond timestamps internally. **Why hard:** mixed environments may show frequent LACP renegotiation.

### E.29. Cisco port-security versus OVS sticky MAC

Cisco's port-security feature locks a port to a MAC. OVS has no equivalent at the bridge level; the closest is an OpenFlow rule that drops mismatched source MACs. **Why hard:** an operator porting a port-security config to OVS has to write the rule explicitly.

### E.30. Linux `bridge fdb` versus OVS `fdb/show`

Linux bridge keeps its FDB in the kernel; readable via `bridge fdb show`. OVS keeps it in `ovs-vswitchd`'s memory; readable via `ovs-appctl fdb/show`. **Why hard:** the two are not interoperable; a learner reading both expects identical commands and is confused.

### E.31. Linux `tc` versus OVS QoS

Linux `tc` directly configures the qdisc; OVS QoS configures it via OVSDB. If both fight for the same netdev, the last-writer wins. **Why hard:** silent overwrite.

### E.32. Linux `iptables` versus OVS `ct(commit)`

Both use the kernel's nf_conntrack. If both exist on the same host, they share the same conntrack table. **Why hard:** rules in iptables and rules in OVS can interfere.

### E.33. The fail-open behaviour after bridge restart

After `ovs-vswitchd` restart, the bridge transitions through a brief moment where flows are not yet installed. With `fail-mode=secure`, this means brief packet loss. **Why hard:** invisible unless the operator measures.

### E.34. Cisco SNMP versus OVS coverage/show

Cisco SNMP exposes per-MIB counters. OVS `coverage/show` exposes named per-counter values. Tooling that maps MIB-OID to OVS-counter requires manual translation. **Why hard:** no automation surface for the comparison.

### E.35. NETCONF/YANG versus OVSDB JSON-RPC

Cisco/Juniper management uses NETCONF/YANG. OVS uses OVSDB JSON-RPC (RFC 7047). The two are conceptually similar (typed schema, transactional, monitor-based) but wire-incompatible. **Why hard:** a mixed environment needs a bridge that speaks both.

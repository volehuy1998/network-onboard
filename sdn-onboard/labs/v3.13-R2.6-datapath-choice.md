## What R2.6 proves

### The single property R2.6 establishes

The OVS daemon `ovs-vswitchd` can drive two different datapaths simultaneously: an in-kernel datapath called `ovs-system`, and an in-userspace datapath called `ovs-netdev`. The choice is per-bridge, controlled by the `Bridge.datapath_type` column. An empty string in that column means "system" (the kernel datapath); the literal string "netdev" means the userspace datapath. R2.6 builds one bridge of each kind, reads the `dpif/show` and `dpctl/show` output, and observes that the daemon presents two simultaneous datapaths each with their own port-numbering space, their own kernel netdev presence, and their own ports-of-internal-type implementation.

The key proof comes from `ovs-appctl dpif/show` after both bridges are configured. The output begins with `netdev@ovs-netdev:` followed by `br-usp` and its ports, then continues with `system@ovs-system:` followed by `br-sys` and its ports. Two separate datapath sections, two separate port-number namespaces (each starts at port 0 = the datapath itself, port 1 = the bridge LOCAL, port 2 = the operator-added internal). The two namespaces do not interact; a flow rule on `br-sys` can output to OF port 1 (which on `br-sys` is the operator-added internal) without ambiguity.

The second proof: the operator-added internal port on `br-usp` is reported with type `(tap)` in `dpif/show`, while the same port type on `br-sys` is reported as `(internal)`. Both ports have OVSDB `Interface.type=internal`, so the *configured* type is the same, but the *implemented* mechanism differs. The `system` datapath uses the kernel's OVS internal-dev driver; the `netdev` datapath uses TAP devices to bridge between userspace and the kernel network stack, because the userspace datapath has no native way to inject frames into a kernel netdev.

### Why this property matters in practice

A reader trained on a Cisco Catalyst has only one mental model of "the forwarding plane": there is one ASIC, one CAM table, one set of forwarding rules. On Open vSwitch the operator can choose which plane to put each bridge in. The choice has performance, debugging, and operational consequences that the reader needs to understand before designing or operating a multi-bridge OVS host.

System (kernel) datapath. Frames are forwarded by `openvswitch.ko` running in kernel context. The fast path is in C running with kernel privileges; the slow path goes upcall to `ovs-vswitchd`. This is the production default for almost every deployment. Throughput is high because the kernel has zero-copy access to skb buffers and runs at line rate on commodity hardware. The cost is that the kernel module's flow-cache implementation is constrained by what is feasible inside kernel context (no large mallocs, no userspace-only abstractions). R4 will measure the kernel datapath's steady-state throughput.

Netdev (userspace) datapath. Frames are forwarded by code running inside `ovs-vswitchd`'s userspace process, in the implementation at `lib/dpif-netdev.c`. The userspace datapath was originally built to host DPDK polling threads (which are out of scope per the owner directive of 2026-04-25); it can also run without DPDK using the standard kernel network stack to ferry frames in and out via TAP devices. R2.6 exercises the non-DPDK userspace path, because DPDK is permanently banned from this curriculum. The userspace path is the right choice when the operator needs userspace-only features (custom packet processing in C, tighter integration with debug instrumentation), or when developing OVS itself; it is not the right choice for production line-rate forwarding under non-DPDK conditions.

The Cisco analogue. Imagine a Catalyst that lets the operator choose, per VLAN, whether forwarding for that VLAN runs on the supervisor's CPU or on the line-card's ASIC. The supervisor-CPU version is more debuggable and more flexible (the operator can write custom code); the ASIC version is faster. Open vSwitch makes the equivalent choice per bridge, with the same trade-offs.

The "default when empty" mapping is at `lib/dpif.c:519` (`dpif_normalize_type()`): an empty string maps to `system`.

## Build one bridge of each kind

### Default datapath (kernel)

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl --may-exist add-br br-sys
root@lab-openvswitch:~/network-onboard# ovs-vsctl get Bridge br-sys datapath_type
""
```

The bridge `br-sys` is created with the default `datapath_type=""`, the OVSDB representation of "unset". The empty string is operator-readable confirmation that no override was applied; the daemon will translate the empty string to `system` per the C code cited above.

### Userspace datapath

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl --may-exist add-br br-usp -- set Bridge br-usp datapath_type=netdev
root@lab-openvswitch:~/network-onboard# ovs-vsctl get Bridge br-usp datapath_type
netdev
```

The bridge `br-usp` is created with `datapath_type=netdev` set in the same transaction (the `--` separator chains `add-br` and `set Bridge`). The `get` confirms the column holds the literal token `netdev`. Note: the schema declares `datapath_type` as a string (not an enum), so the daemon does its own validation; an unknown string would cause the daemon to reject the bridge configuration.

## Read the simultaneous datapath presence

### dpif/show: two datapaths in one daemon

```text
root@lab-openvswitch:~/network-onboard# ovs-appctl dpif/show
netdev@ovs-netdev: hit:0 missed:0
  br-usp:
    br-usp 65534/1: (tap)
system@ovs-system: hit:0 missed:0
  br-sys:
    br-sys 65534/1: (internal)
```

Two datapath sections in the same output. The `netdev@ovs-netdev:` section reports the userspace datapath, named `ovs-netdev`. Inside that section, `br-usp:` is the bridge, and `br-usp 65534/1: (tap)` is the LOCAL port: OF port 65534, DP port 1, **type `(tap)`**. The type-of-implementation tag in parentheses reveals the userspace-datapath secret: a bridge in `netdev` mode synthesises its LOCAL port using a TAP device, not the kernel's OVS internal-dev driver. The reason is that the userspace datapath has no kernel-side internal-dev to attach to; instead it opens a TAP `/dev/net/tun` file descriptor, presents the kernel-side end of the TAP as the bridge's LOCAL netdev, and reads/writes frames on the userspace-side fd.

The `system@ovs-system:` section reports the kernel datapath, named `ovs-system`. The bridge `br-sys` is here, with `br-sys 65534/1: (internal)`. The LOCAL is implemented by the kernel's OVS internal-dev driver (the `(internal)` tag), which is the same mechanism R2.0 introduced.

The two datapaths have independent port-numbering spaces. Both LOCALs have DP port 1 because each datapath's port numbering starts fresh at 1; they do not collide because they are in different datapaths. The reader who has been tracking DP port numbers across R2.0 through R2.4 should note that those labs all used the system datapath, and the DP-port allocation was host-wide *within* that datapath. Adding a netdev-datapath bridge introduces a second numbering space.

### Add an internal port to each bridge

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br-sys port-sys -- set Interface port-sys type=internal
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br-usp port-usp -- set Interface port-usp type=internal
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=name,type,ofport,mac_in_use list Interface port-sys port-usp
name                : port-sys
type                : internal
ofport              : 1
mac_in_use          : "ee:b8:55:82:19:c0"

name                : port-usp
type                : internal
ofport              : 1
mac_in_use          : "f6:ca:dc:49:f1:a7"
```

Two `add-port` commands, each with `type=internal`, one per bridge. The OVSDB Interface rows look identical: both have `type : internal`, both have `ofport : 1`, both have a kernel-assigned MAC. The *configured* type is identical between the two; the OVSDB row does not record any datapath-aware information. The configured type is interpreted by the daemon at runtime, and the runtime interpretation depends on which datapath the port belongs to.

### The kernel-side reality differs

```text
root@lab-openvswitch:~/network-onboard# ip link show port-sys
136: port-sys: <BROADCAST,MULTICAST> mtu 1500 qdisc noop state DOWN mode DEFAULT group default qlen 1000
    link/ether ee:b8:55:82:19:c0 brd ff:ff:ff:ff:ff:ff
root@lab-openvswitch:~/network-onboard# ip link show port-usp
137: port-usp: <BROADCAST,MULTICAST,PROMISC> mtu 1500 qdisc noop state DOWN mode DEFAULT group default qlen 1000
    link/ether f6:ca:dc:49:f1:a7 brd ff:ff:ff:ff:ff:ff
```

The two netdevs the kernel reports look almost the same. Both are at `state DOWN`, both have a kernel-assigned MAC matching the OVSDB row, both have MTU 1500. One difference is decisive: `port-usp` has the `PROMISC` flag set. `port-sys` does not.

The `PROMISC` flag is the operator's evidence that `port-usp` is implemented differently. The userspace datapath puts every TAP-backed bridge port into promiscuous mode so that the kernel does not filter incoming frames before they reach the TAP file descriptor and are read by the userspace datapath. The kernel datapath does not need this trick because the kernel's OVS internal-dev driver intercepts frames at the kernel hook level, before any per-netdev filtering. The PROMISC flag is the visible artifact of the implementation difference.

### Read the new dpif/show

```text
root@lab-openvswitch:~/network-onboard# ovs-appctl dpif/show
netdev@ovs-netdev: hit:0 missed:0
  br-usp:
    br-usp 65534/1: (tap)
    port-usp 1/2: (tap)
system@ovs-system: hit:0 missed:0
  br-sys:
    br-sys 65534/1: (internal)
    port-sys 1/2: (internal)
```

Both datapaths now have two ports each. On `br-usp`, both `br-usp` and `port-usp` are reported as `(tap)`; on `br-sys`, both ports are `(internal)`. The pattern is consistent: every "internal" type port on the userspace datapath is implemented as a TAP device, and every "internal" type port on the kernel datapath is implemented through the kernel's OVS internal-dev driver. The OVSDB column is the operator-facing abstraction; the implementation is per-datapath.

## Read the kernel-side datapath listing

### ovs-dpctl: the kernel-only view

```text
root@lab-openvswitch:~/network-onboard# ovs-dpctl show
system@ovs-system:
  lookups: hit:0 missed:0 lost:0
  flows: 0
  masks: hit:0 total:0 hit/pkt:0.00
  caches:
    masks-cache: size:256
  port 0: ovs-system (internal)
  port 1: br-sys (internal)
  port 2: port-sys (internal)
```

`ovs-dpctl show` queries the kernel module directly via netlink, bypassing `ovs-vswitchd`. It reports only the kernel datapath, `ovs-system`. The userspace datapath is invisible to `ovs-dpctl` because the userspace datapath does not exist in the kernel; it lives entirely inside `ovs-vswitchd`'s memory. The reader will not see `br-usp`, `port-usp`, or `ovs-netdev` here, no matter how many netdev-mode bridges they create.

### ovs-appctl dpctl/show: both datapaths via the daemon

```text
root@lab-openvswitch:~/network-onboard# ovs-appctl dpctl/show 2>&1 | head -30
netdev@ovs-netdev:
  lookups: hit:0 missed:0 lost:0
  flows: 0
  port 0: ovs-netdev (tap)
  port 1: br-usp (tap)
  port 2: port-usp (tap)
system@ovs-system:
  lookups: hit:0 missed:0 lost:0
  flows: 0
  masks: hit:0 total:0 hit/pkt:0.00
  caches:
    masks-cache: size:256
  port 0: ovs-system (internal)
  port 1: br-sys (internal)
  port 2: port-sys (internal)
```

`ovs-appctl dpctl/show` is the daemon-routed version of `ovs-dpctl show`. The daemon knows about both datapaths, so the output covers both: first `netdev@ovs-netdev` (the userspace datapath) with its three ports and zero stats, then `system@ovs-system` with the same three-port structure. The reader who wants a unified view of every datapath the daemon manages should use `ovs-appctl dpctl/show`, not `ovs-dpctl show`; the latter only sees what the kernel module owns.

A subtle but important difference between the two views. `ovs-dpctl show` lists "masks: hit:0 total:0 hit/pkt:0.00" and "caches: masks-cache: size:256" for the kernel datapath. The userspace datapath's section omits both lines because the userspace datapath uses a different cache structure (`dpcls`, the datapath classifier in `lib/dpif-netdev.c`). The two implementations have independent cache geometries, and operator visibility into cache health requires datapath-aware tooling. R4 will measure cache behaviour for both implementations.

### /sys/class/net: the kernel-namespace evidence

```text
root@lab-openvswitch:~/network-onboard# ls /sys/class/net/ 2>&1 | grep -E 'br-sys|br-usp|port-sys|port-usp|ovs'
br-sys
br-usp
ovs-netdev
ovs-system
port-sys
port-usp
```

Six entries. Two are the bridge LOCALs (`br-sys`, `br-usp`). Two are the operator-added internal ports (`port-sys`, `port-usp`). Two are the *datapath* netdevs themselves: `ovs-system` (the kernel datapath's per-host entry) and `ovs-netdev` (the userspace datapath's per-host TAP). The kernel exposes `ovs-netdev` as a TAP device because the userspace datapath created it that way at startup; the kernel does not know about the userspace datapath internals, but it does know about the TAP device that the userspace datapath uses as its IO surface.

The reader can verify the TAP nature of `ovs-netdev` by checking `/sys/class/net/ovs-netdev/tun_flags`, the same probe R2.3 used to identify TAP devices. R2.6 does not run that probe explicitly, but the inference is sound based on the consistent `(tap)` tag in `dpif/show`.

## Datapath identifiers and version strings

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=name,datapath_type,datapath_id,datapath_version list Bridge
name                : br-sys
datapath_type       : ""
datapath_id         : "00005ea5c3fb0642"
datapath_version    : "<unknown>"

name                : br-usp
datapath_type       : netdev
datapath_id         : "00007686b1244d45"
datapath_version    : "<built-in>"
```

Both bridges have a `datapath_id` value (the OpenFlow DPID). The DPID is allocated fresh per bridge by hashing the LOCAL port MAC; the two values differ because the two bridges have different LOCAL MACs. The `datapath_version` column is more telling.

`br-sys` reports `datapath_version : "<unknown>"`. The kernel module that backs the system datapath does not expose a version string in a form OVS can read at this lab host's kernel, so the daemon falls back to the literal placeholder `<unknown>`. R2.0 already saw this on the LOCAL port. The placeholder is normal; the reader should not interpret it as a defect.

`br-usp` reports `datapath_version : "<built-in>"`. The userspace datapath is part of `ovs-vswitchd` itself, so its version is the version of `ovs-vswitchd`. The literal `<built-in>` is the daemon's way of saying "same version as me"; the reader can cross-reference with `ovs-vsctl get Open_vSwitch . ovs_version` (which reports `2.17.9`) to find the actual version.

The contrast is informative: the kernel datapath has its own version (which OVS cannot read) because it lives in a separately-versioned binary (the kernel module). The userspace datapath is compiled into the daemon and has the daemon's version by definition. A reader who is debugging a version-skew issue between userspace and kernel reads these two columns first.

## Trace through both datapaths

```text
root@lab-openvswitch:~/network-onboard# ovs-appctl ofproto/trace br-sys 'in_port=port-sys,dl_src=02:00:00:00:00:01,dl_dst=ff:ff:ff:ff:ff:ff,dl_type=0x0806'

bridge("br-sys")
----------------
 0. priority 0
    NORMAL
     -> no learned MAC for destination, flooding

Final flow: unchanged
Megaflow: recirc_id=0,eth,arp,in_port=1,dl_src=02:00:00:00:00:01,dl_dst=ff:ff:ff:ff:ff:ff,arp_op=0

root@lab-openvswitch:~/network-onboard# ovs-appctl ofproto/trace br-usp 'in_port=port-usp,dl_src=02:00:00:00:00:02,dl_dst=ff:ff:ff:ff:ff:ff,dl_type=0x0806'
Flow: arp,in_port=1,vlan_tci=0x0000,dl_src=02:00:00:00:00:02,dl_dst=ff:ff:ff:ff:ff:ff,arp_spa=0.0.0.0,arp_tpa=0.0.0.0,arp_op=0,arp_sha=00:00:00:00:00:00,arp_tha=00:00:00:00:00:00

bridge("br-usp")
----------------
 0. priority 0
    NORMAL
     -> no learned MAC for destination, flooding

Final flow: unchanged
Megaflow: recirc_id=0,eth,arp,in_port=1,dl_src=02:00:00:00:00:02,dl_dst=ff:ff:ff:ff:ff:ff,arp_op=0
Datapath actions: 1
```

Two traces, one per bridge, simulating an ARP broadcast from each bridge's `port-*` to the broadcast MAC. The traces are byte-for-byte identical except for the bridge name and the source MAC; both produce the same priority-0 NORMAL synthesis, the same flood path, the same megaflow shape, and the same `Datapath actions: 1` (output to the bridge LOCAL).

The pedagogical point. The translation logic is implemented in `ofproto/ofproto-dpif-xlate.c` and is shared between the two datapaths. The translator does not know which datapath it is producing actions for; it produces a generic action list, and the per-datapath provider (`lib/dpif-linux.c` for system, `lib/dpif-netdev.c` for netdev) is responsible for executing the actions on its own datapath. The action `1` in `br-sys`'s trace means "output to DP port 1 in the kernel datapath, which is `br-sys`'s LOCAL"; the same `1` in `br-usp`'s trace means "output to DP port 1 in the userspace datapath, which is `br-usp`'s LOCAL". The number is the same; the resolution differs.

This is the cleanest demonstration of OVS's layered architecture. The OpenFlow flow table and the translation logic are datapath-independent. The action numbering scheme is also datapath-independent at the userspace level. Only the per-datapath provider knows how to execute the action against its own port-numbering space and its own forwarding mechanism. R3 (OpenFlow programming) will rely on this datapath-independence: the same flow rule produces the same logical behaviour on both datapaths.

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl del-port br-sys port-sys
root@lab-openvswitch:~/network-onboard# ovs-vsctl del-port br-usp port-usp
root@lab-openvswitch:~/network-onboard# ovs-vsctl --if-exists del-br br-sys

root@lab-openvswitch:~/network-onboard# date -u
Thu Apr 30 05:00:08 AM UTC 2026
```

Standard symmetric cleanup. Both bridges removed, both internal ports gone with them. Wall clock 22 seconds. The reader who watches `/sys/class/net/` after the cleanup would see both `ovs-system` and `ovs-netdev` netdevs persist; the datapath-level netdevs are created at daemon startup and torn down only at daemon shutdown, regardless of whether any bridges of their type exist. The persistent netdev is a small but useful diagnostic: an operator who suspects `ovs-vswitchd` has not initialised the userspace datapath can check for `ovs-netdev` in `/sys/class/net/`.

## Misconception block

> **Misconception / Reality.**
>
> Misconception: "the userspace datapath is the same thing as DPDK; using `datapath_type=netdev` requires a DPDK-aware kernel and configured hugepages."
>
> Reality: the userspace datapath is a code path in `lib/dpif-netdev.c` that runs in the regular `ovs-vswitchd` process and uses TAP devices to talk to the kernel network stack. DPDK is *one* possible way to integrate the userspace datapath with high-performance NICs (the daemon polls a DPDK-managed PMD instead of reading TAP file descriptors), but it is not the only way and it is not required. R2.6 demonstrated the non-DPDK userspace datapath, which works on stock Ubuntu 22.04 with no kernel patches, no hugepages, no special hardware. DPDK is permanently out of scope for this curriculum per the owner directive of 2026-04-25; the non-DPDK userspace datapath is fully accessible to the reader.

> **Misconception / Reality (second).**
>
> Misconception: "I have to choose one datapath for the whole host; I cannot mix system and netdev bridges."
>
> Reality: R2.6 just demonstrated the opposite. `ovs-vswitchd` can drive both datapaths simultaneously; the choice is per-bridge. The two datapaths have independent port-numbering spaces, independent cache structures, and independent kernel netdev presence; mixing them on a single host is supported, common in development environments, and creates no special operational hazards. The only caveat is that frames cannot flow directly between the two datapaths through the kernel; if a bridge in one datapath needs to send to a bridge in the other, the operator can use a patch port (which lives entirely in userspace and works across datapaths) or a userspace L3 hop through the host IP stack.

## What the reader should now be able to do

After R2.6 the reader can:

- Predict that `ovs-vsctl get Open_vSwitch . datapath_types` reports `[netdev, system]` on a stock 2.17.9 build.
- Choose `datapath_type=netdev` versus the empty default knowingly: production deployments use the empty default (kernel datapath), development and DPDK-based deployments use `netdev`.
- Read `ovs-appctl dpif/show` and identify which datapath each bridge is on by the section header (`netdev@ovs-netdev:` versus `system@ovs-system:`).
- Recognise the `(tap)` tag on a bridge LOCAL port as the signature of the userspace datapath, and `(internal)` as the signature of the kernel datapath.
- Recognise the `PROMISC` flag on a netdev-datapath internal port (visible in `ip link show`) as the userspace datapath's no-filter trick.
- Use `ovs-dpctl show` for the kernel-only view and `ovs-appctl dpctl/show` for the daemon-routed all-datapaths view.
- Read `Bridge.datapath_version` and tell whether the column refers to the kernel module (`<unknown>` is common) or to the in-process userspace datapath (`<built-in>`).
- Understand that OpenFlow flow rules are datapath-independent: the same rule has the same meaning on both datapaths because the translation engine is shared.

The pedagogical arc closes: R2.6 has shown that the bridge primitive is more flexible than a Cisco engineer's mental model would predict. A bridge is not "an instance of the one global forwarding plane"; it is an instance of one of *two* forwarding planes the operator can choose between. The choice is per-bridge, the planes coexist, and the OpenFlow surface is the same for both. R2.7 returns to a different aspect of the bridge primitive: the OVSDB row UUID round trip and idempotence pattern. R2.8 closes R2 with the byte-for-byte MAC-port binding proof.

```ascii
         ovs-vswitchd  (one userspace process)
         +------------------------------------------------+
         |  OpenFlow flow table (per bridge)              |
         |  xlate_actions() (datapath-independent)        |
         |                                                |
         |  +-------------------+  +--------------------+ |
         |  | dpif-linux        |  | dpif-netdev        | |
         |  | provider          |  | provider           | |
         |  | (lib/dpif-linux.c)|  | (lib/dpif-netdev.c)| |
         |  +---------+---------+  +----------+---------+ |
         +------------|----------------------|------------+
                      | netlink              | direct call
                      v                      v
         +------------+--------+   +---------+----------+
         | openvswitch.ko      |   |  in-process        |
         | (kernel module)     |   |  packet processing |
         | datapath ovs-system |   |  datapath          |
         |                     |   |  ovs-netdev        |
         |  - vport list       |   |  - vport list      |
         |  - flow_table       |   |  - dpcls           |
         |  - actions exec     |   |  - EMC             |
         |  - kernel hooks     |   |  - polls TAP fds   |
         +---------+-----------+   +---------+----------+
                   |                         |
                   | claims kernel netdevs   | reads/writes via
                   | via "master" field      | TAP file descriptors
                   v                         v
         +---------+--------+      +---------+--------+
         | port-sys netdev  |      | port-usp netdev  |
         | type=internal    |      | type=tap+PROMISC |
         | (kernel-managed) |      | (TAP fd-backed)  |
         +------------------+      +------------------+
```

The diagram is the synthesised mental model the reader should keep. The top half is `ovs-vswitchd`'s userspace memory: shared OpenFlow state, shared translation logic, two provider implementations side by side. The bottom half is the per-datapath execution machinery: kernel module on the left, in-process userspace machinery on the right. The right and left sides do not talk to each other directly; if a frame needs to cross between them it does so through a patch port (entirely in userspace, no datapath involvement) or through a userspace L3 hop (each datapath terminates at a kernel netdev, the kernel routes between them at the IP layer).

The reader who has worked through R2.0 to R2.6 should now recognise that the previous five labs all implicitly used the left half of this diagram. R2.6 is the first to exercise the right half, and the take-home is that the right half exists, is real, is operationally distinct from the left, and can be observed with the standard tooling.

## A short note on the userspace datapath in development workflows

The userspace datapath is the native debugging substrate for OVS development. When upstream maintainers add a new feature, the new code typically lands in both providers (`dpif-linux.c` and `dpif-netdev.c`), but the userspace path is exercised first in the test suite (`tests/*.at`) because it does not require a kernel module reload. The OVS test suite at `tests/automake.mk` configures most of its tests to use the userspace datapath for this reason; the test `make check-system-userspace` runs the full system-test suite against the userspace datapath specifically.

The reader who wants to use OVS as a feature playground (for example, to test a custom OpenFlow rule before deploying it to production) is well-served by spinning up a `datapath_type=netdev` bridge in a development VM. The code path is the same as the production kernel-datapath path at the OpenFlow level; only the underlying execution differs. R9 (debug-and-troubleshoot) returns to this in the context of "reproduce a production bug locally without a kernel module".

## A note on cache structure differences between the two datapaths

The two datapaths use structurally different caches. The kernel datapath uses a tuple-space-search classifier (TSS) backed by a per-mask hash table; the masks-cache the operator saw in `ovs-dpctl show` is a recent optimisation that pre-filters mask candidates before the per-mask lookup. The userspace datapath uses the dpcls (datapath classifier) implemented in `lib/dpif-netdev.c`, which is a more general TSS plus an Exact-Match Cache (EMC) sitting in front of it for fast hits on recently-seen exact flows.

The cache geometry impacts the operational diagnostic. On a kernel-datapath bridge, the relevant counters are `lookups: hit/missed/lost`, `flows`, `masks: hit/total`. On a userspace-datapath bridge, the relevant counters are different (the daemon exposes EMC hit rate, dpcls subtable counts, and per-flow statistics), and `ovs-appctl dpctl/show` reports a different set of fields for each. R4 (datapath internals) goes deeper into the cache structures; R2.6 only needs the reader to recognise that the two are structurally different and that the same diagnostic command can produce different output shapes depending on which datapath is being queried.

## A practical operator habit: choose datapath at bridge-create time

The `Bridge.datapath_type` column is one of the few OVSDB columns that the daemon *can* change at runtime (an operator can mutate it via `ovs-vsctl set Bridge ... datapath_type=...`), but in practice the operator should not. Switching a live bridge's datapath causes the daemon to tear down the old datapath state for that bridge and rebuild on the other one; any active flows are flushed, any megaflow cache entries are lost, and any stored OpenFlow rules are temporarily inaccessible during the switchover. The result is a brief but real outage on every port of the bridge.

The right operator habit is to choose the datapath at bridge-create time and treat the choice as immutable for the bridge's lifetime. If a deployment needs to change datapath for a bridge in production, the standard pattern is to add a new bridge with the desired datapath, migrate ports from the old bridge to the new one in a coordinated maintenance window, then delete the old bridge. R10 (forensic and production readiness) returns to this in the context of zero-downtime maintenance.

A note on the OVS test suite's discipline. The upstream test suite at `tests/automake.mk` runs almost every test against the userspace datapath because the userspace datapath does not require kernel-module reloading and is therefore much faster to tear down and rebuild between tests. The test `make check-system-userspace` is the standard target the OVS maintainers exercise on every commit. R9 (debug-and-troubleshoot) revisits this in the context of "how does upstream verify a feature works".

## A short note on what the userspace datapath does *not* support without DPDK

For completeness, the reader should know that some OVS features are kernel-datapath-specific, and others are userspace-datapath-specific. Connection tracking via `ct()` actions works on both, but the kernel implementation goes through `nf_conntrack` while the userspace implementation has its own conntrack at `lib/conntrack.c`. Hardware offload via `tc` is kernel-datapath-only. Some statistics-gathering paths differ. The operator who is moving a workload between datapaths should consult the OVS documentation for feature-by-feature parity; the curriculum's R6 (conntrack) and R7 (QoS) labs exercise the relevant differences.

A practical implication: a flow rule that works on the kernel datapath is *almost* always portable to the userspace datapath, but the operator should always test before relying on portability. R2.6's traces showed that the priority-0 NORMAL synthesis works identically on both, but more advanced features may not.

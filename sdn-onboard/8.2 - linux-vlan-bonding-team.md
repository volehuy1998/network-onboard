## 8.2.1 VLAN stacking: single-tagged, QinQ, triple-tagged

**Single-tagged 802.1Q (1998)**: 4-byte VLAN tag chèn sau MAC. 12-bit VLAN ID → 4096 VLAN tối đa.

```
+--------+--------+----------+----------+--------+------+
| DstMAC | SrcMAC | TPID=8100| VID+PCP  | EthType| Data |
| 6B     | 6B     | 2B       | 2B       | 2B     | ...  |
+--------+--------+----------+----------+--------+------+
```

**QinQ 802.1ad (2011)**: S-VLAN (service provider) + C-VLAN (customer) stacked. Service provider transport customer VLAN trong own VLAN space — mở rộng 4096 VLAN tenant trên hạ tầng shared.

**Triple-tagged**: triển khai niche carrier, 3 tag lồng nhau cho MEF service hierarchy. Ít gặp ngoài telco.

Cấu hình VLAN trên Linux:

```bash
[host]# ip link add link eth0 name eth0.100 type vlan id 100
[host]# ip addr add 192.168.100.1/24 dev eth0.100
[host]# ip link set eth0.100 up
```

USC Day 5 Lab 6 thực hành VLAN trunking — dùng OVS `vlan_mode=trunk` + `trunks=[10,20,30]` thay cho Linux VLAN subinterface. Khác biệt: Linux VLAN subinterface chỉ xử lý VLAN theo per-interface; OVS trunk port xử lý nhiều VLAN với programmable policy.

## 8.2.2 Bonding: 7 modes

Linux kernel bonding driver (`drivers/net/bonding/`) implement 7 mode:

| Mode | Tên | Hoạt động | Peer switch requirement |
|---|---|---|---|
| 0 | `balance-rr` | Round-robin packet giữa slaves | Static LAG (Etherchannel on mode) |
| 1 | `active-backup` | Một slave active, còn lại standby | Không yêu cầu |
| 2 | `balance-xor` | Hash (MAC src XOR MAC dst) modulo slave count | Static LAG |
| 3 | `broadcast` | Replicate packet ra mọi slave | Redundancy niche |
| 4 | `802.3ad` (LACP) | Dynamic link aggregation per IEEE 802.3ad | LACP-capable peer |
| 5 | `balance-tlb` | Transmit LB, receive dedicated | Không yêu cầu |
| 6 | `balance-alb` | Adaptive LB TX + ARP-based RX | Không yêu cầu |

Cấu hình:

```bash
[host]# ip link add bond0 type bond mode 802.3ad miimon 100 lacp_rate fast
[host]# ip link set eth0 master bond0
[host]# ip link set eth1 master bond0
[host]# ip link set bond0 up
```

Mode 4 (LACP) là standard cho production multi-link — cần switch peer có LACP-capable LAG configuration tương ứng.

## 8.2.3 LACP specifics

IEEE 802.3ad-2020 định nghĩa LACPDU exchange giữa Actor (our side) và Partner (peer):

- **System ID**: MAC address + priority (ưu tiên peer với system ID thấp hơn làm negotiator).
- **Key**: identify aggregation group. Actor và Partner phải có key khớp với để aggregate.
- **Port priority**: ưu tiên port trong group.
- **State bits**: Activity, Timeout, Aggregable, Synchronizing, Collecting, Distributing, Defaulted, Expired.

Timer:

- **Fast (1 second)**: LACPDU mỗi 1s, peer timeout 3s. Failover sub-100ms. CPU overhead cao hơn.
- **Slow (30 second)**: LACPDU mỗi 30s, peer timeout 90s. Default.

Fast-timer cho production datacenter, slow-timer cho WAN uplink.

Switch-side config phải match:
- Cùng port list.
- Cùng LACP mode (active-active hoặc active-passive).
- Cùng timer.

OVS bonding (Phần 9.6) có fallback mode `lacp-fallback-ab=true` — fallback sang active-backup nếu LACP negotiation fail. Linux kernel bonding không có fallback tương đương.

## 8.2.4 Team daemon: deprecate trong RHEL 9

Team (`libteam`) ra đời 2012 thay thế bonding với:

- Userspace daemon `teamd` xử lý logic.
- D-Bus API cho programmatic control.
- Extensible runner (activebackup, roundrobin, loadbalance, random, lacp).

RHEL 7-8 ship team như alternative cho bonding. Tuy nhiên, **RHEL 9 (2022) deprecate team**, khuyến cáo dùng bonding.

Lý do deprecate:

1. **Tighter kernel integration**: bonding hoàn toàn trong kernel, team có userspace dependency.
2. **Team daemon crash** có thể disrupt networking — bonding không phụ thuộc userspace process.
3. **Duplicate functionality**: mọi feature team cung cấp đã có trong bonding (LACP mode 4 thay thế teamd lacp runner).
4. **Maintenance burden**: hai driver cùng làm việc tương đương, team contributions giảm.

Cho triển khai mới, dùng bonding mode 4 (LACP) — không dùng team.

## 8.2.5 Bonding driver tier 2 source (`drivers/net/bonding/` v5.15)

Linux bonding driver chứa trong `drivers/net/bonding/`. Module chính:

- `bond_main.c`: `bond_xmit_hash()` quyết định slave xuất packet, `bond_xmit_*_slave()` per-mode forwarding.
- `bond_3ad.c`: 802.3ad LACP state machine + LACPDU exchange.
- `bond_alb.c`: balance-alb adaptive learning.
- `bond_options.c`: sysctl/sysfs option handling.

### Mode 4 (LACP) state machine

LACP có 4 sub-state machine running per port:

1. **Receive Machine** (`ad_rx_machine()` in `bond_3ad.c`): xử lý LACPDU đến.
2. **Periodic Transmit Machine** (`ad_periodic_machine()`): gửi LACPDU định kỳ (1s fast / 30s slow).
3. **Mux Machine** (`ad_mux_machine()`): điều khiển trạng thái Selected/Standby cho slave.
4. **Tx Machine** (`ad_tx_machine()`): tạo + gửi LACPDU.

State transition:

```
INITIALIZE → DETACHED → WAITING → ATTACHED → COLLECTING_DISTRIBUTING
```

Slave chỉ forward traffic khi reach state `COLLECTING_DISTRIBUTING`. Bất kỳ change negotiation (peer key change, system ID change) → re-enter `DETACHED`.

### XMIT hash (mode 2 + 4)

Khi mode 2/4, `bond_xmit_hash()` chọn slave qua hash:

```c
// bond_main.c, function bond_xmit_hash() simplified
u32 bond_xmit_hash(struct bonding *bond, struct sk_buff *skb)
{
    if (bond->params.xmit_policy == BOND_XMIT_POLICY_LAYER2) {
        // Hash MAC src+dst (XOR)
        return bond_xmit_hash_l2(skb);
    } else if (bond->params.xmit_policy == BOND_XMIT_POLICY_LAYER34) {
        // Hash IP src+dst + L4 ports
        return bond_xmit_hash_l34(skb);
    } else if (bond->params.xmit_policy == BOND_XMIT_POLICY_ENCAP34) {
        // Hash inner IP+L4 cho VXLAN/Geneve encap
        return bond_xmit_hash_encap34(skb);
    }
}
```

5 hash policy: `layer2` (default, MAC), `layer2+3` (MAC+IP), `layer3+4` (IP+L4), `encap2+3`, `encap3+4`. Cho overlay (Geneve/VXLAN), `encap3+4` cho phép distribute inner flow lên multiple slave (avoid single-flow bottleneck cho cùng outer 5-tuple).

```bash
[host]# ip link set bond0 type bond xmit_hash_policy layer3+4
[host]# cat /sys/class/net/bond0/bonding/xmit_hash_policy
layer3+4 1
```

### LACP-related sysfs

```bash
[host]# cat /sys/class/net/bond0/bonding/ad_actor_system     # Local system MAC
[host]# cat /sys/class/net/bond0/bonding/ad_partner_mac      # Peer system MAC
[host]# cat /sys/class/net/bond0/bonding/ad_actor_key        # Local key
[host]# cat /sys/class/net/bond0/bonding/ad_partner_key      # Peer key
[host]# cat /sys/class/net/bond0/bonding/ad_aggregator       # Aggregator ID
[host]# cat /sys/class/net/bond0/bonding/ad_num_ports        # Active port count
[host]# cat /sys/class/net/bond0/bonding/ad_select          # stable / bandwidth / count
```

`ad_select`:

- `stable` (default): re-aggregation chỉ khi phải.
- `bandwidth`: chọn aggregator có total bandwidth cao nhất.
- `count`: chọn aggregator có nhiều port nhất.

## 8.2.6 OVS bonding so sánh (Phần 9.6 cross-link)

Phần 9.6 đi sâu OVS bonding. Đây là so sánh nhanh so với Linux kernel bonding:

| Feature | Linux bond | OVS bond |
|---------|-----------|----------|
| Mode | 7 mode | 3 mode (active-backup / balance-slb / balance-tcp) |
| LACP fallback | ✗ | ✓ (`lacp-fallback-ab=true`) |
| Per-flow hash | layer2/3/4/encap3+4 | hash_basis qua datapath rule |
| Programmable | ✗ | ✓ qua OpenFlow flow rule |
| BFD integration | ✗ | ✓ (kết hợp với HA failover) |
| Hardware offload | ethtool offload | TC offload framework (Phần 9.5) |

OVS bond preferable cho:

- OVN cluster cần BFD-driven failover.
- Multi-tenant với per-flow QoS.
- DPDK fastpath.

Linux bond đủ cho:

- Bare-metal server uplink (no SDN policy).
- Storage network multi-link.
- Container host bond không cần OVS.

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| `Bonding Mode` | Match expected | Wrong mode | Empty | Mode misconfig |
| `LACP rate` | `fast` (DC) hoặc `slow` (WAN) | Mismatch peer | (n/a) | Both ends must agree |
| `Active Aggregator Number of ports` | = expected slave count | < expected (1 slave fail) | 0 | Total bond down |
| `Actor Churn State` | `none` | `churned` (re-negotiation) | Persistent churned | LACP unstable, peer flapping |
| `MII Status` per slave | `up` | `down` (1 slave down) | All `down` | Physical link issue |
| `Link Failure Count` | 0 | 1-5 (occasional) | > 10 (frequent flap) | NIC/cable issue |
| `port state` (bit 6 = sync) | 63 = all bits set | < 63 | < 32 (no Activity) | LACP not negotiated |

`port state` 8-bit field meaning (RFC IEEE 802.3ad):

- bit 0 (LACP_Activity): 1 = active LACP
- bit 1 (LACP_Timeout): 1 = fast timer
- bit 2 (Aggregable): 1 = can aggregate
- bit 3 (Synchronizing): 1 = in sync với peer
- bit 4 (Collecting): 1 = receiving
- bit 5 (Distributing): 1 = transmitting
- bit 6 (Defaulted): 0 = received from peer (1 = using default)
- bit 7 (Expired): 0 = current (1 = expired)

Healthy = bits 0-5 set, bits 6-7 clear → `0x3F` = 63.

**Kịch bản bẻ gãy:**

- `port state = 0x07` (LACP active + fast + aggregable, không sync): peer chưa response. Check switch LACP config.
- `Actor Churn` persistent: peer key/MAC changing → switch reboot loop hoặc LACP timer mismatch.
- `Number of ports = 1`: 1 slave dropped. Check MII status + cable + switch port.

## 8.2.8 ▶ Capstone POE: balance-rr vs LACP cho Geneve overlay

> **Bối cảnh:** Cluster OVN 50 chassis, mỗi chassis 2x25G uplink. Kỹ sư kiến trúc đề nghị `balance-rr` (round-robin) thay LACP cho throughput cao hơn.

**Tuyên bố cần kiểm chứng:** "balance-rr (mode 0) cho throughput cao hơn LACP (mode 4) cho overlay traffic vì round-robin không cần hash."

**Predict:** Sai cho overlay; balance-rr gây out-of-order delivery + TCP re-transmit storm.

**Observe (giả định):**

- Setup mode 0 balance-rr, gửi TCP large transfer cross-chassis: throughput nominal 50 Gbps (2x25G), thực tế chỉ 12-18 Gbps + TCP retransmit count cao 5-10K/s.
- Setup mode 4 LACP với `xmit_hash_policy=encap3+4`: throughput 35-45 Gbps, retransmit < 100/s.

**Explain:**

1. **balance-rr** chia packet round-robin giữa slave **không quan tâm flow**. Cùng TCP connection có packet đi 2 đường khác → out-of-order arrival. TCP receiver flag out-of-order → trigger retransmit + congestion control reduce CWND. Throughput effective drop 50-70%.
2. **LACP với `encap3+4`** hash inner IP+L4. Mỗi connection đi cùng slave → in-order delivery. TCP healthy. Hash distribution cho phép multi-flow distribute lên cả 2 slave.
3. **balance-rr only good cho** UDP (no ordering), single-flow saturate single link, hoặc mode active-backup nếu chỉ cần redundancy.

**Falsification test:**

- Run iperf3 TCP cross-chassis với mode 0 vs mode 4 + encap3+4.
- Đo retransmit rate qua `ss -tin`.
- Nếu mode 0 retransmit < mode 4 → counter-example. Trong thực tế, mode 0 retransmit luôn cao hơn.

**Tuyên bố sai ở đâu:** "Round-robin không cần hash nên throughput cao hơn" sai vì bỏ qua TCP semantics. Đúng phải: **LACP với hash policy phù hợp cho overlay (encap3+4) là standard cho OVN cluster. balance-rr không production-grade.**

> **Quy tắc:** Overlay TCP traffic = LACP mode 4 + xmit_hash_policy=encap3+4. balance-rr chỉ cho UDP single-flow saturation case.

## Điểm cốt lõi cần nhớ

Thứ nhất, VLAN 802.1Q single-tagged đủ cho 4096 VLAN — QinQ cần thiết khi service provider transport customer VLAN.

Thứ hai, Linux bonding có 7 mode. Mode 4 (802.3ad LACP) là standard production. Mode 1 (active-backup) cho scenario đơn giản.

Thứ ba, LACP fast-timer (1s) cho datacenter production, slow-timer (30s) cho WAN. OVS có `lacp-fallback-ab` — kernel bonding không có.

Thứ tư, team daemon deprecated từ RHEL 9 (2022). Mọi triển khai mới dùng bonding.

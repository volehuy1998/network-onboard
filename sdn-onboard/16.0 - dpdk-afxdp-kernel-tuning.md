## 16.0.0 Opening drama, from "the kernel stack is good enough" to "you must bypass to reach Tbps"

In the 2000s, the Linux kernel network stack was considered enough for enterprise workloads. With 1 GbE NICs being common, kernel NAPI plus the NIC driver could process about 148 Kpps (64-byte wire-speed) easily. By the 2010s, 10 GbE NICs (14.88 Mpps wire-speed) appeared, and the kernel started to struggle under high load: context-switch overhead, interrupt coalescing, socket buffer copies, every packet costs about 1 to 10 us. With workloads such as High-Frequency Trading (HFT) needing latency below 5 us, the kernel was no longer enough.

In 2010, **Intel Dataplane Performance Development Kit (DPDK)** appeared as a set of libraries plus a PMD driver that lets userspace applications poll the NIC directly, bypassing the kernel networking stack entirely. The result: latency dropped to 1 to 3 us, throughput reached 100 or more Mpps per core. DPDK quickly became standard for telecom 5G UPF, HFT trading gateway, virtual switching (OVS-DPDK), NFV (Network Function Virtualization), and DDoS mitigation appliances.

But DPDK has 3 large drawbacks: (a) **it burns 100 percent of one CPU core per port** because of poll mode, which does not scale well in cloud shared-CPU environments, (b) **it detaches the NIC from the kernel**, losing kernel features (iptables, tc, netfilter, bonding, LACP), (c) **it requires hugepages plus IOMMU plus CPU isolation**, which means complex deployment, not plug-and-play.

In 2018, **AF_XDP** (kernel 4.18) appeared as a compromise: it uses a shared memory ring between the NIC driver and a userspace socket, bypassing the kernel socket stack **but keeping the kernel NIC driver plus netfilter plus tc** for packets that do not match (XDP_PASS). AF_XDP throughput is around 8 to 12 Mpps per core (about 30 percent below DPDK) but it is easier to deploy and preserves kernel features. Cilium, Suricata, and OVS-AFXDP quickly adopted AF_XDP.

Part 16.0 dives into the 3 data path options, DPDK EAL plus PMD internals, AF_XDP zero-copy architecture, kernel tuning knobs (sysctl plus RSS plus RFS plus NAPI budget), CPU pinning plus NUMA awareness, and profiling with perf plus bpftrace. The part ends with 3 hands-on exercises (baseline versus DPDK, AF_XDP XDP filter, full tuning workflow from baseline to wire-speed) plus a Capstone POE: "hyperscale 2026, DPDK versus AF_XDP versus kernel for the Kubernetes CNI?".

> **Common misconception:** *"DPDK is always faster than the kernel stack."*
>
> **Reality:** DPDK real-world bandwidth is 5 to 10 times higher than the kernel only when **the workload has enough load** to justify the poll-mode overhead. With low-utilization burst traffic (for example a web server at 1000 req/s), DPDK burns 100 percent of a core but packets per second are no higher than the kernel (kernel NAPI is efficient under low load). The right pattern: benchmark both with the actual workload before investing in DPDK complexity. Many production K8s clusters use the OVS kernel datapath, no DPDK needed.

## 16.0.1 Compare data paths: kernel, DPDK, AF_XDP

There are three approaches to high-rate packet processing on Linux x86: **the standard kernel datapath** (NIC -> driver -> kernel networking stack -> userspace socket), **DPDK** (the NIC is detached from the kernel driver, attached to UIO or VFIO, the userspace application polls the NIC directly through a Poll Mode Driver), and **AF_XDP** (the NIC has an XDP program attached, packets are redirected to a userspace ring buffer without going through the full kernel networking stack). The three approaches differ in real-world bandwidth, latency, CPU cost, and deployment complexity.

**Real-world bandwidth benchmark** (measured on a physical machine with Intel Xeon Gold 6246R, NIC Intel XXV710 25GbE, 64-byte packets): the kernel datapath reaches about 1.5 to 2 Mpps per core (degrading under high load due to interrupt overhead); OVS-DPDK reaches 10 to 15 Mpps per core (poll mode); AF_XDP reaches 8 to 12 Mpps per core (zero-copy mode with a supported driver). With a 100GbE NIC and 4 or more dedicated CPU cores, DPDK reaches 40 to 80 Mpps total; the kernel stack tops out around 20 Mpps before saturating IRQs.

**Latency profile** (packet round-trip on the same node): the kernel datapath is around 20 microseconds (including interrupt overhead and syscall context-switch overhead); DPDK is around 2 microseconds (continuous polling, no interrupts); AF_XDP is around 10 microseconds (still uses kernel interrupt mechanism but skips the socket stack). For latency-sensitive workloads (HFT, 5G UPF, real-time telecom signalling), DPDK is the standard choice; for workloads that prefer bandwidth but accept higher latency, AF_XDP balances performance and complexity.

**CPU cost model** is an important trade-off: DPDK's poll mode **burns 100 percent of one core** per port, even when there is no traffic the core keeps spinning in poll. This is not suitable for cloud environments with shared CPU (each VM loses one core to the datapath). The kernel stack uses an interrupt-driven model: when there are packets the CPU services an interrupt, when idle the CPU enters a power-saving C-state. AF_XDP has a continuous-polling mode similar to DPDK and an interrupt-driven mode similar to the kernel, which lets the team choose the balance between performance and power cost.

## 16.0.2 DPDK EAL plus PMD internals

**EAL (Environment Abstraction Layer)** is the foundation library of DPDK, which initializes the runtime environment for a userspace application: (1) scan available hugepages (2MB or 1GB), allocate contiguous memory for DPDK use; (2) discover the NUMA layout, assign threads to CPU sockets per configuration; (3) attach the NIC to the UIO (legacy) or VFIO-PCI driver, detaching the NIC from kernel networking; (4) initialize the **mempool** (a pool of pre-allocated packet buffers) and **mbuf** (buffer metadata for each packet).

**PMD (Poll Mode Driver)** is the userspace driver for the NIC, replacing the kernel driver. The PMD for each NIC model (Intel ixgbe, i40e, ice, mlx5 for Mellanox, bnxt for Broadcom) is compiled into the DPDK binary. The PMD does not use interrupts, instead a dedicated thread runs an infinite loop that calls `rte_eth_rx_burst()` to read a batch of packets from the NIC descriptor ring, processes them, and calls `rte_eth_tx_burst()` to send them out of another NIC. The loop runs continuously on a dedicated CPU core (isolcpus).

**Hugepages allocation** is a hard requirement of DPDK for two reasons: (1) contiguous memory avoids TLB misses, when processing millions of packets per second, one TLB miss per packet stalls the CPU for hundreds of clock cycles; (2) hugepages are zero-copy, DPDK maps the NIC DMA buffer directly into a hugepage with no need to copy a packet to a userspace buffer. Standard configuration: 2 to 4 GB of 1GB hugepages for traffic at 40 to 100 Gbps, or 1 to 2 GB of 2MB hugepages for traffic at 10 to 25 Gbps.

**Pinning to NUMA nodes** is an important optimization: if the NIC sits on PCIe socket 0, the PMD thread must run on a core on socket 0 (using the `--lcores` flag), and hugepages must be allocated from socket 0 memory. Wrong pinning sends every packet across the QPI or UPI link between sockets, which doubles latency and reduces real-world bandwidth by 30 to 50 percent.

**Mempool lifecycle:** the mempool is allocated at EAL initialization with a fixed size (for example 8K mbufs per port). At runtime, an mbuf is borrowed from the pool when a packet arrives and returned when the packet is processed. If the mempool is exhausted (high backlog), the NIC drops new packets, which is a common symptom when debugging DPDK performance issues.

## 16.0.3 AF_XDP zero-copy plus XDP programs

**AF_XDP** is a new kind of kernel socket introduced in kernel 4.18 (2018). It lets a userspace application receive packets directly from the NIC driver through a shared memory ring, bypassing the kernel networking stack entirely while still reusing the kernel driver (different from DPDK which uses its own userspace PMD). AF_XDP combines performance close to DPDK with the deployment ease of the kernel.

**The AF_XDP socket architecture** has four components: (1) **UMEM**, a userspace memory region registered with the kernel as a buffer for packets, the kernel DMAs directly from the NIC into UMEM (zero-copy with a supporting driver). (2) **FILL ring**, userspace pushes UMEM frame indices onto this ring to tell the kernel "these buffers are ready for new packets". (3) **RX ring**, the kernel pushes the frame indices of received packets onto this ring; userspace reads the ring to process packets. (4) **TX ring plus COMPLETION ring**, similar but for the egress direction.

**XDP program attach points:** (1) **Driver or native mode**, the eBPF program runs in the NIC driver interrupt context, BEFORE the kernel allocates an sk_buff. This is the highest-performance attach point (about 24 Mpps per core on Intel XXV710), but only drivers with support can attach: ixgbe, i40e, ice, mlx5, mlx4, bnxt, tun. (2) **SKB mode** (generic XDP), runs AFTER the kernel has allocated an sk_buff, works with any driver but is much slower (about 3 to 5 Mpps). (3) **Hardware offload mode**, compile the eBPF program for NIC firmware, run it in the NIC ASIC. Supported by Netronome Agilio CX and Mellanox ConnectX-5 or later; instruction-set restrictions apply.

**The XDP_REDIRECT pattern** is how packets move from the NIC to an AF_XDP socket: the XDP program attached to the NIC calls `bpf_redirect_map(xsk_map, queue_id, 0)` to push the packet into the userspace AF_XDP socket. This is the standard mechanism for Cilium (pod-to-pod redirect), OVS-AFXDP (covered in Part 9.3), and Suricata or Snort AF_XDP.

Compared with DPDK, AF_XDP has these advantages: (1) no need to detach the NIC from the kernel, IP configuration and routing tables still work normally; (2) no mandatory hugepages (regular pages can be used, with a small TLB cost but easier deployment); (3) it can coexist with kernel traffic, the XDP program can choose to redirect OR let the packet continue into the kernel stack (XDP_PASS). Disadvantage: latency about 5 microseconds higher than DPDK because it still goes through the kernel driver.

## 16.0.4 Kernel tuning knobs

Without going to DPDK or AF_XDP, the kernel stack can still push to 20 to 30 Mpps with proper tuning. The main knobs:

**Sysctl `net.core.*`** for high packet-rate workloads:
- `net.core.netdev_max_backlog = 300000`, the queue depth between the NIC driver and kernel processing, default 1000 (too low for 10 Gbps or more).
- `net.core.rmem_max = 16777216`, `wmem_max = 16777216`, max socket buffer size, default 200K (too low for high-bandwidth TCP).
- `net.core.busy_read = 50`, `busy_poll = 50`, enable continuous polling for latency-sensitive workloads (the CPU spins instead of taking interrupts).
- `net.core.somaxconn = 65535`, the listen backlog depth, default 128 (too low for high-concurrency servers).

**RPS (Receive Packet Steering)** distributes packets from one queue across many CPUs to balance load. Configure by writing a bitmask to `/sys/class/net/ethX/queues/rx-N/rps_cpus`. Important when the NIC has only one or few queues but the CPU has many cores.

**RFS (Receive Flow Steering)** extends RPS to route packets to the CPU on which the userspace socket is scheduled, optimizing cache locality. Configured through `/proc/sys/net/core/rps_sock_flow_entries`.

**Tune the NAPI budget:** NAPI (New API) is the kernel's interrupt-coalescing framework. By default each NAPI poll cycle processes 64 packets and yields the CPU. For high-bandwidth workloads, raise `net.core.netdev_budget = 600` and `netdev_budget_usecs = 8000` so each cycle processes more packets.

## 16.0.5 CPU pinning plus NUMA topology awareness

In multi-socket systems, the NUMA (Non-Uniform Memory Access) layout decides packet processing performance. Each CPU socket has local memory with fast access (50 to 100 ns) and remote memory accessed through the QPI or UPI link (100 to 200 ns, twice as slow). The NIC plugs into a PCIe slot belonging to a specific socket, packets from the NIC land in that socket's memory.

**Discovery tools:**
- `lscpu` shows NUMA nodes and which CPUs belong to which node.
- `lstopo` (from hwloc) draws the full layout: socket, core, cache, NIC, memory channel.
- `numactl --hardware` confirms the node layout.
- `cat /sys/class/net/ethX/device/numa_node` reports which NUMA node the NIC belongs to.

**Configure `isolcpus` in the kernel cmdline:** add `isolcpus=2,3,4,5 nohz_full=2,3,4,5 rcu_nocbs=2,3,4,5` to the kernel cmdline in GRUB to detach 4 cores from the default scheduler. These cores are not preempted by other workloads, dedicated to DPDK PMD threads or AF_XDP workers.

**Pin OVS PMD threads:** configure through `ovs-vsctl set Open_vSwitch . other_config:pmd-cpu-mask=0x3c` (mask bits 2 to 5 for CPUs 2 to 5). OVS assigns PMD threads onto these cores, and these cores must match the NUMA node of the NIC for optimal performance.

**IRQ affinity through `/proc/irq/*/smp_affinity`:** to prevent NIC IRQs from interrupting datapath cores, write a bitmask of spare cores (for example CPUs 0 and 1) into this file. Example: `echo 3 > /proc/irq/125/smp_affinity` lets only CPUs 0 or 1 service IRQ 125, no longer disturbing CPUs 2 to 5 that run PMDs.

## 16.0.6 Profiling with perf plus bpftrace

When tuning performance, measure before changing is the golden rule. Three layers of tools for each level of summary:

**Driver-level statistics through `ethtool`:**
```
ethtool -S eth0 | grep -E 'drops|errors|discards'
```
Important counters: `rx_dropped` (packets dropped because the queue is full), `rx_missed_errors` (packets dropped because the driver could not keep up with the NIC), `rx_no_buffer_count` (mempool exhausted, a strong signal for DPDK workloads). If these counters increase, the driver or the kernel stack is not keeping up with incoming traffic.

**Perf counter for cache miss and cycle analysis:**
```
perf stat -e cache-references,cache-misses,cycles,instructions -p <PID>
```
A cache miss ratio above 10 percent is a sign of NUMA misalignment (buffers in remote memory) or wrong mempool sizing. Cycles per instruction above 2.0 is a sign of stalls due to memory bandwidth or TLB miss, consider hugepages, prefetch hints.

**bpftrace probe for per-CPU packet latency:**
```
bpftrace -e 'kprobe:__netif_receive_skb { @[cpu] = hist(nsecs - @start[tid]); }
                kretprobe:__netif_receive_skb { @start[tid] = nsecs; }'
```
The script measures the latency distribution of the kernel function `__netif_receive_skb` per CPU, helping detect overloaded cores. Similar scripts ship in bcc-tools: `xdp_trace.bt`, `tcplife.bt`, `ext4slower`.

**perf flamegraph** for visual CPU profiling: `perf record -F 99 -a -g sleep 30 && perf script | stackcollapse-perf.pl | flamegraph.pl > flame.svg`. The flamegraph shows which functions consume the most CPU. If `netif_skb_features` or `napi_schedule` sits at the top, consider switching to native XDP or DPDK.

### Guided Exercise 1: Benchmark OVS kernel datapath versus DPDK

> **Lab verification status:** the throughput numbers are based on OVS-DPDK performance reports; pending verification on a real lab.

**Goal:** quantitatively measure the real-world bandwidth delta between OVS kernel datapath (default) and OVS-DPDK (userspace poll mode). Understand the trade-off: real-world bandwidth versus latency versus CPU cost versus complexity.

**Setup:**
- Ubuntu 22.04 VM with 4 vCPU plus 8 GB RAM.
- IOMMU support (`cat /proc/cmdline | grep intel_iommu=on` after grub config plus reboot).
- Install iperf3: `sudo apt install iperf3`.

**Lab topology:**

```
Phase 1, OVS kernel (baseline):
  [VM1] --- OVS kernel --- [VM2]
           (default)
  iperf3 -> about 2 to 3 Gbps, about 30 to 40 percent CPU

Phase 2, OVS-DPDK:
  (hugepages=4x1GB, isolcpus=2,3)
  [VM1] -- OVS-DPDK (PMD=CPU 2,3) -- [VM2]
  iperf3 -> about 10 to 20 Gbps, 100 percent CPU (poll mode)
```

**Steps:**

**Phase 1, baseline OVS kernel:**

**1.** Set up the topology:

```bash
[host]# ovs-vsctl add-br br0
[host]# ip netns add ns1 && ip netns add ns2

[host]# ip netns exec ns2 ip addr add 10.0.0.2/24 dev veth-ns2
[host]# ip netns exec ns1 ip link set veth-ns1 up
[host]# ip netns exec ns2 ip link set veth-ns2 up
```

```bash
[host]# ip netns exec ns1 iperf3 -s &
[host]# ip netns exec ns2 iperf3 -c 10.0.0.1 -t 30
```

Expected output:

```
[  5]   0.00-30.00 sec   8.42 GBytes  2.41 Gbits/sec
```

**Phase 2, OVS-DPDK setup plus benchmark:**

**3.** Configure hugepages plus IOMMU (reboot required):

```bash
[host]# sed -i 's|GRUB_CMDLINE_LINUX_DEFAULT="\(.*\)"|GRUB_CMDLINE_LINUX_DEFAULT="\1 hugepagesz=1G hugepages=4 intel_iommu=on iommu=pt isolcpus=2,3"|' /etc/default/grub
```

```bash
[host]# iperf3 -c 10.0.0.1 -t 30 -P 4
```

Expected output:

```
[SUM]   0.00-30.00 sec   52.3 GBytes  14.9 Gbits/sec
```

```bash

  packets received: 52340123
  cycles per packet: 48.5

  packets received: 52198562
```

**Expected outcome:** comparison table:

| Phase | Config | Real-world bandwidth | CPU cost | Latency |
|-------|--------|----------------------|----------|---------|
| 1 | OVS kernel | 2.4 Gbps | 35 percent of 1 core | about 20 us |
| 2 | OVS-DPDK | 15 Gbps (6x) | 100 percent of 2 cores | about 2 us (10x) |

**Lessons learned:**

First, **DPDK raises real-world bandwidth about 6 times versus the kernel** on the same hardware, BUT it consumes 2 cores constantly at 100 percent. Good for bandwidth-critical workloads, expensive for burst traffic.

Second, **latency drops 10 times** because the kernel context-switch plus interrupt overhead is removed; DPDK skips the kernel network stack. Essential for real-time finance and 5G control plane.

Third, **production trade-off:** DPDK requires hugepages plus IOMMU plus CPU isolation, not plug-and-play. The kernel datapath fits most workloads; only choose DPDK when specific performance requirements justify the complexity.

**Cleanup:**

```bash
[host]# systemctl stop openvswitch-switch
[host]# ovs-vsctl del-br br0-dpdk
[host]# ovs-vsctl set Open_vSwitch . other_config:dpdk-init=false
[host]# systemctl start openvswitch-switch
[host]# ip netns del ns1 ns2
```

```
2: eth0: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 xdp ...
```

```bash
# Terminal 1: send TCP (will be redirected)
[host]$ curl -m 5 http://www.example.com

```

```
Packets received via AF_XDP:
  Total: 15 packets, 3240 bytes
  Packet rate: 3 pps

Source        Dest          Proto  Length
10.0.0.5      93.184.216.34 TCP    1432
93.184.216.34 10.0.0.5      TCP    1380
```

**Lessons learned:**

First, **XDP is the earliest hook point**, before SKB allocation, before netfilter. A drop at XDP is about 100 times cheaper than a drop at the socket layer.

Third, **performance trade-off:** AF_XDP is about 10 to 50 Gbps typical (versus DPDK at 100 or more Gbps). Use cases: edge compute, selective DDoS mitigation, targeted offload.

**Cleanup:**

```bash
[host]# ./af_xdp_user --dev eth0 --unload

[host]# ip link set eth0 xdpgeneric off
```

```
2-socket server, 40Gbps NIC

Socket 0 (cores 0-7, 6 mem channels) -- QPI -- Socket 1 (cores 8-15)
  NIC (PCIe Gen3 x8, 40 Gbps, on socket 0)
  OVS-DPDK PMD threads on CPU 2,4 (socket 0, physical cores only)
```

**Steps, iterative tuning:**

**Step 0 (baseline):** plain OVS kernel, no tuning:

```bash
[host]# iperf3 -c 10.0.0.1 -t 30 -P 4
```

Expected output: about 10 Gbps.

**Step 1: 2MB hugepages**

```bash
[host]# grubby --update-kernel=ALL --args="hugepagesz=2M hugepages=2048"
[host]# reboot
[host]# cat /proc/meminfo | grep Huge
```

Re-benchmark: about 12 Gbps.

**Step 2: CPU isolation**

```bash
[host]# grubby --update-kernel=ALL --args="isolcpus=2,4,6"
[host]# reboot
[host]# cat /sys/devices/system/cpu/isolated
2,4,6
```

Re-benchmark: about 15 Gbps (the scheduler does not interrupt the PMD cores).

**Step 3: RSS multi-queue**

```bash
[host]# ethtool -L ens1 combined 4
[host]# ethtool -X ens1 equal 4
[host]# ethtool -N ens1 rx-flow-hash tcp4 sdfn  # 5-tuple hash
```

Re-benchmark: about 25 Gbps (parallel RSS queues).

**Step 4: OVS-DPDK with 2 PMD threads pinned**

```bash
[host]# ovs-vsctl set Open_vSwitch . other_config:dpdk-init=true
[host]# ovs-vsctl set Open_vSwitch . other_config:dpdk-socket-mem=2048,0
[host]# ovs-vsctl set Open_vSwitch . other_config:pmd-cpu-mask=0x14  # CPU 2,4
[host]# systemctl restart openvswitch-switch
[host]# ovs-vsctl add-br br0-dpdk -- set bridge br0-dpdk datapath_type=netdev
[host]# ovs-vsctl add-port br0-dpdk dpdk-p0 -- set interface dpdk-p0 type=dpdk \
        options:dpdk-devargs=0000:01:00.0
```

Re-benchmark: about 40 Gbps (wire-speed).

**Expected outcome, results table:**

| Step | Config | Real-world bandwidth | Delta | CPU cost |
|------|--------|----------------------|-------|----------|
| 0 | Baseline kernel | 10 Gbps | n/a | 4 cores about 80 percent |
| 1 | plus 2MB hugepages | 12 Gbps | plus 20 percent | Same |
| 2 | plus isolcpus=2,4,6 | 15 Gbps | plus 25 percent | 3 cores isolated |
| 3 | plus RSS 4 queues | 25 Gbps | plus 67 percent | Balanced 4 queues |
| 4 | plus OVS-DPDK PMD pin | 40 Gbps | plus 60 percent | 2 PMD cores at 100 percent |

**Lessons learned:**

First, **tuning is iterative, with a compound effect**, there is no single silver bullet. A 4x total improvement comes from 4 independent optimizations.

Second, **hugepages plus isolcpus** are relatively low-risk, apply them first to any network-heavy workload. RSS multi-queue plus DPDK are higher-commitment (they require topology understanding).

Third, **wire-speed forwarding** (40 Gbps, 30 to 60 Mpps) is achievable on commodity hardware with proper tuning. Production service providers routinely reach 100 Gbps or more per server.

**Cleanup:** revert grub args:

```bash
[host]# grubby --remove-args="hugepagesz=2M hugepages=2048 isolcpus=2,4,6"
[host]# ovs-vsctl set Open_vSwitch . other_config:dpdk-init=false
[host]# systemctl restart openvswitch-switch && reboot
```

## 16.0.7 Compare DPDK, AF_XDP, kernel with OVS, OVN, and Cilium

In the 2024 to 2026 software data plane ecosystem, three technologies coexist at different layers:

| Dimension | OVS kernel datapath | OVS-DPDK / OVS-AFXDP | Cilium eBPF |
|-----------|---------------------|----------------------|--------------|
| Data plane location | Kernel (openvswitch.ko module) | Userspace (ovs-vswitchd) | Kernel (eBPF program hook XDP plus tc) |
| Real-world bandwidth | 1.5 to 2 Mpps per core | 10 to 15 Mpps per core (DPDK), 8 to 12 Mpps (AF_XDP) | 5 to 10 Mpps per core (XDP) |
| Latency | about 20 us | about 2 us (DPDK), about 10 us (AF_XDP) | about 5 us (XDP native) |
| CPU cost | Interrupt-driven (low when idle) | Poll mode 100 percent of a core (DPDK), hybrid (AF_XDP) | Interrupt-driven |
| Programmability | OpenFlow plus conntrack | Same as kernel plus DPDK features | Arbitrary eBPF programs |
| Kernel features preserved | All | DPDK: none, AF_XDP: most | All |
| Deployment complexity | Low | High (hugepages plus IOMMU plus isolcpus) | Medium (BPF toolchain) |
| Main use case | Hypervisor VM or pod | HFT, 5G UPF, NFV appliance | K8s CNI (Cilium, Calico with eBPF) |

**When to use what in real deployments:**

- **OVS kernel plus OVN:** hypervisor workload (OpenStack ML2/OVN), K8s with OVN-Kubernetes. 95 percent of cloud-native use cases are enough.
- **OVS-DPDK:** NFV VNF (virtualized firewall, load balancer, SBC), 5G UPF, edge caching with real-world bandwidth above 10 Gbps per hypervisor. CPU burn is acceptable because the workload is dedicated.
- **OVS-AFXDP:** compromise when you need real-world bandwidth above the kernel but want to keep iptables and tc features. Edge gateway, DDoS mitigation with selective offload.
- **Cilium eBPF:** K8s CNI replacing kube-proxy plus iptables. Performance is about 10 times better than iptables-based kube-proxy. Identity-based security replaces IP-based NetworkPolicy.

### Cross-cut with OVN plus OpenStack

OVN with the OVS backend supports **both kernel and DPDK datapaths** through the `other_config:datapath-type` configuration. Production patterns:

- **kolla-ansible OpenStack default:** OVS kernel datapath. Simple, reliable, enough for standard workloads.
- **Telco NFV deployment:** OVS-DPDK. Required for 5G UPF plus MEC (Multi-Access Edge Computing).
- **OVN-Kubernetes in K8s:** OVS kernel (default). Cilium plus eBPF is an alternative for enterprise K8s (Red Hat OpenShift 4.11 plus supports Cilium).

Note: switching from kernel to DPDK in an existing OVN deployment **requires downtime** (restart ovs-vswitchd, reload flows, reconfigure ports). Plan the migration window carefully.

## 16.0.8 Key takeaways

**Six key takeaways:**

1. **3 data path options for Linux networking:** kernel (default, interrupt-driven, about 2 Mpps per core), DPDK (poll mode, 10 to 15 Mpps per core, burns 100 percent CPU), AF_XDP (shared memory ring plus XDP program, about 10 Mpps per core, keeps kernel features).
2. **DPDK = EAL plus PMD plus hugepages plus NUMA pinning.** EAL initializes the environment, PMD is the userspace driver, hugepages avoid TLB misses plus zero-copy, NUMA pinning is essential for performance.
3. **AF_XDP = UMEM plus 4 rings (FILL, RX, TX, COMPLETION).** Zero-copy with driver support (ixgbe, i40e, ice, mlx5). 3 attach modes: driver or native (fastest), SKB (generic, compatible), hardware offload (Netronome, ConnectX).
4. **Kernel tuning is not dead:** sysctl `net.core.netdev_max_backlog`, `rmem_max`, `busy_poll`; RPS plus RFS for multi-queue; NAPI budget tuning. With proper tuning, the kernel can reach 20 to 30 Mpps, enough for many workloads.
5. **CPU pinning plus NUMA awareness** is fundamental: `isolcpus` kernel cmdline plus `pmd-cpu-mask` in OVS plus `smp_affinity` for IRQ. Cross-NUMA memory access is twice as slow and kills performance.
6. **Profile before tuning:** `ethtool -S` (driver counters), `perf stat` (cache miss plus IPC), `bpftrace` (per-CPU latency histogram), `perf flamegraph` (CPU hotspots). Measure before changing is the golden rule.

### Capstone POE: Hyperscale Kubernetes cluster 2026, which CNI data plane to choose?

> **Lab verification status:** the POE is based on publicly documented deployments (Google GKE, Microsoft AKS, Alibaba ACK) plus Cilium versus Calico versus OVN-Kubernetes public benchmarks; pending the user having a cluster to verify.

**Goal:** evaluate the CNI data plane choice for a hyperscale 2026 K8s cluster, matching requirements with capability.

**Scenario:** Organization W deploys a Kubernetes cluster with 10,000 nodes times 50 pods per node equals 500,000 pods. Network traffic: 70 percent east-west intra-cluster (pod-to-pod), 20 percent north-south ingress, 10 percent egress to external services. Real-world bandwidth per node is about 10 to 25 Gbps east-west. SecOps requirement: NetworkPolicy at the pod identity level, not IP-based. Performance requirement: pod-to-pod P99 latency below 5 ms.

**Four CNI data plane options:**

1. **kube-proxy iptables plus Flannel/Calico standard:** traditional K8s CNI, iptables IP-based rules.
2. **OVN-Kubernetes (OVS kernel datapath):** OpenStack-derived OVN ported to K8s. Kernel OVS plus OVN logical pipeline.
3. **Cilium eBPF (kernel XDP plus tc hook):** pod-to-pod through eBPF, replaces kube-proxy. Identity-based policy.
4. **OVS-DPDK hypervisor:** userspace DPDK for node-level network acceleration (needed for bare-metal bandwidth-critical workloads).

**Predict:** evaluate the four options against five criteria: (a) real-world bandwidth, (b) latency, (c) pod identity NetworkPolicy, (d) deployment complexity, (e) long-term viability.

**Observe (reasoning plus data check):**

**Option 1 (iptables plus Flannel/Calico standard):**
- Real-world bandwidth: limited. 500,000 pods times O(n) iptables lookup scaling create a performance cliff at about 10,000 service rules.
- Latency: 50 to 100 us P99 pod-to-pod (iptables linear scan).
- Pod identity policy: limited (Calico NetworkPolicy is IP-based).
- Complexity: low. Mature ecosystem.
- Long-term: stable but losing ground to eBPF.

**Option 2 (OVN-Kubernetes):**
- Real-world bandwidth: 10 Gbps per node OK with kernel datapath. 25 Gbps or more requires DPDK plus complexity.
- Latency: 20 to 30 us P99 (OVN logical pipeline overhead).
- Pod identity: OVN port group plus address set, flexible but IP-based.
- Complexity: high (OVN northd, SB DB, 27 or more pipeline stages). Hard to debug.
- Long-term: Red Hat OpenShift backing. Good for OpenStack-adjacent environments.

**Option 3 (Cilium eBPF):**
- Real-world bandwidth: 10 to 20 Gbps per node native. XDP hook for ingress acceleration.
- Latency: below 5 us P99 pod-to-pod (eBPF fast path).
- Pod identity: excellent. Cilium identity (derived from labels) is a first-class concept. No IP churn issue.
- Complexity: medium. eBPF toolchain learning curve, but the Cilium CLI is mature.
- Long-term: strong (Isovalent acquired by Cisco 2024, CNCF graduated). Hyperscale adoption is growing.

**Option 4 (OVS-DPDK bare-metal):**
- Real-world bandwidth: 40 Gbps or more per node.
- Latency: below 2 us.
- Pod identity: limited (OVS OpenFlow is IP-based).
- Complexity: very high (hugepages plus IOMMU plus isolcpus plus dedicated CPUs).
- Long-term: OK but niche (mostly NFV, not K8s standard).

**Explain:** recommendation for the hyperscale 2026 scenario:

**For Organization W (500,000 pods, 10 to 25 Gbps per node, P99 below 5 ms, identity policy):**

**Primary choice: Cilium eBPF (option 3).** Reasons:

- Latency requirement P99 below 5 ms is only met consistently by Cilium eBPF (OVN-K8s 20 to 30 us is fine in the lab, but scaling to 500,000 pods creates control-plane overhead).
- Identity-based NetworkPolicy is a critical requirement, Cilium has first-class support, Calico and OVN-K8s IP-based policies are weaker.
- Real-world bandwidth of 10 to 25 Gbps per node is the sweet spot of Cilium eBPF, no DPDK needed.
- Long-term viability: CNCF graduated plus Isovalent and Cisco backing equals stable.

**DO NOT choose:**
- iptables/Flannel: does not scale to 500,000 pods.
- OVN-K8s: over-complex for K8s-native workloads (its strength is OpenStack integration, not K8s scale).
- OVS-DPDK: overkill for K8s, CPU burn is wasteful in cloud shared environments.

**Tuning recommendations for Cilium eBPF:**
- Enable eBPF masquerade plus eBPF kube-proxy replacement.
- Configure the BPF LB algorithm to Maglev for consistent hashing.
- Set CPU affinity for BPF processing (isolcpus 2 to 3 cores per node).
- Monitor BPF map sizes (connection tracking, identity map) through `cilium bpf metrics`.

**Finish:** universal principle: **"the CNI choice must match the scale plus requirements, not follow trends"**. Cilium eBPF fits hyperscale plus identity plus performance. With small clusters (below 1000 pods), iptables is simple enough. For NFV telco, OVS-DPDK. One size does not fit all. This pattern applies to every architectural decision.

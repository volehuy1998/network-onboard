## 9.19.1 Cấu trúc một flow entry trong OVS

Tài liệu Lab 4 (Crichigno et al., USC WASTC 2021) mô tả flow entry ngắn gọn: *"A flow entry consists of header fields, counters, and actions."* OVS thực hiện đầy đủ định nghĩa OpenFlow 1.0 này và mở rộng với các field bổ sung. Mỗi flow entry trên bridge OVS bao gồm:

| Thành phần | Vai trò | Ví dụ |
|-----------|---------|-------|
| **Match fields** | Điều kiện một packet phải thỏa để khớp entry | `in_port=1,dl_type=0x0800,nw_dst=10.0.0.2` |
| **Priority** | Số nguyên 0-65535, entry cao hơn ưu tiên | `priority=100` |
| **Counters** | `n_packets` + `n_bytes` — thống kê mỗi lần match | Tự tăng |
| **Actions** | Danh sách action thực thi khi match | `output:2`, `drop`, `mod_dl_dst:aa:...` |
| **Timeouts** | `idle_timeout` (giây im lặng → xóa), `hard_timeout` (tuổi thọ tuyệt đối) | `idle_timeout=60,hard_timeout=300` |
| **Cookie** | Identifier 64-bit do controller/user đặt để track nhóm flow | `cookie=0x1001` |
| **Table** | Table chứa entry (OpenFlow 1.1+) | `table=0` |

Khi `ovs-ofctl add-flow br0 "priority=100,ip,nw_dst=10.0.0.2,actions=output:2"` được chạy, OVS chèn entry vào `table=0` với `dl_type=0x0800` (được suy ra từ từ khóa `ip`), `nw_dst=10.0.0.2`, `priority=100`, và action `output:2`. Entry tồn tại đến khi bị xóa thủ công hoặc timeout (nếu có).

> **Chủ đề trọng tâm:** Flow entry là đơn vị chuyển tiếp nhỏ nhất trên OVS. Khác với router Cisco dùng routing table + ARP cache riêng biệt, OVS đóng gói toàn bộ logic xử lý quyết định vào một entry — match (điều kiện) + action là đủ. Đây là lý do OVS có thể làm bridge, router, firewall, load balancer, tunnel endpoint trong cùng một bridge, chỉ khác nhau ở flow entries.

## 9.19.2 Topology và bốn cấp độ matching

Lab 4 dùng topology tối giản để focus vào matching behavior:

```
            ┌───────────────────────┐
            │      br-s1            │
            │  (OVS bridge)         │
            │                       │
            │ s1-eth1      s1-eth2  │
            └────┬────────────┬─────┘
                 │            │
                 │            │
           ┌─────┴────┐ ┌─────┴────┐
           │    h1    │ │    h2    │
           │10.0.0.1/8│ │10.0.0.2/8│
           └──────────┘ └──────────┘
```

Toàn bộ cả hai host nằm chung subnet `10.0.0.0/8`, không cần routing. Ta sk là buộc OVS forward packet giữa hai port bằng bốn cách khác nhau, từ granular thấp đến cao. Đây là bài tập quan trọng để thấy OVS không phân biệt "loại switch" (L2/L3/L4), nó chỉ phân biệt mức độ chi tiết của khớp với field.

### Cấp độ 1 — L1 port-based chuyển tiếp

Đơn giản nhất: mọi thứ từ port 1 gửi ra port 2, mọi thứ từ port 2 gửi ra port 1. OVS không inspect bất kỳ field nào ngoài `in_port`.

```bash
[host]# ovs-ofctl add-flow br-s1 "priority=100,in_port=1,actions=output:2"
[host]# ovs-ofctl add-flow br-s1 "priority=100,in_port=2,actions=output:1"
```

Kết quả: hành vi giống hub, tất cả traffic được cross-over giữa hai port. Ưu điểm: CPU tối thiểu, băng thông thực tế tối đa. Nhược điểm: broadcast storm nếu có hơn 2 port, không lọc được theo bất kỳ tiêu chí nào khác.

### Cấp độ 2 — L2 MAC-based chuyển tiếp

Match theo MAC address cho granularity cao hơn:

```bash
[host]# ovs-ofctl add-flow br-s1 \
  "priority=100,dl_dst=02:00:00:00:00:02,actions=output:2"
[host]# ovs-ofctl add-flow br-s1 \
  "priority=100,dl_dst=02:00:00:00:00:01,actions=output:1"
```

Giờ OVS forward dựa trên destination MAC. Đây là hành vi của một L2 switch tĩnh (static MAC table). OVS không tự học MAC source trong field hợp này, muốn học động phải dùng action `NORMAL` (học tự động trong MAC table implicit) hoặc `learn()` action (tường minh, tạo flow entry mới khi thấy packet mới).

> **Hiểu sai:** "OVS tự động học MAC như switch Cisco mặc định."
>
> Thực tế: OVS chỉ học MAC khi action `NORMAL` được gọi — action này kích hoạt learning logic built-in. Nếu flow table chỉ có match/action tường minh (như ví dụ trên), OVS không học bất cứ gì. Nhiều kỹ sư nghĩ OVS "không hoạt động" khi flow table trống, thực ra OVS hoạt động đúng với flow table đã cho — chỉ là flow table không yêu cầu học.

### Cấp độ 3 — L3 IP-based chuyển tiếp

Match theo IP destination:

```bash
[host]# ovs-ofctl add-flow br-s1 \
  "priority=100,ip,nw_dst=10.0.0.2,actions=output:2"
[host]# ovs-ofctl add-flow br-s1 \
  "priority=100,ip,nw_dst=10.0.0.1,actions=output:1"
```

Từ khóa `ip` là shortcut cho `dl_type=0x0800` (Ethernet type IPv4). Bắt buộc phải có vì `nw_dst` chỉ ý nghĩa khi frame là IP. Với flow table trên, OVS forward IPv4 packet theo destination IP. ARP packet (`dl_type=0x0806`) sẽ không khớp → không được forward → rớt. Trong production, phải có flow riêng cho ARP hoặc action `NORMAL` làm fallback.

### Cấp độ 4 — L4 TCP/UDP port-based chuyển tiếp

Match sâu vào L4 header:

```bash
# Giả sử h2 chạy web server python trên port 80
[host]# ovs-ofctl add-flow br-s1 \
  "priority=100,tcp,nw_dst=10.0.0.2,tp_dst=80,actions=output:2"
[host]# ovs-ofctl add-flow br-s1 \
  "priority=100,tcp,nw_dst=10.0.0.1,tp_dst=80,actions=output:1"
```

Từ khóa `tcp` = `dl_type=0x0800,nw_proto=6`. `tp_dst=80` khớp HTTP. Với flow table này, chỉ packet TCP đến port 80 được forward — ICMP, SSH, UDP đều rớt. Đây là cơ sở để làm L4 firewall hoặc LB tầng 4 chỉ bằng flow table.

## 9.19.3 Cơ chế priority resolution

OVS lưu trữ flow entries trong classifier (chi tiết Phần 9.15, tuple space search). Khi một packet đi vào, classifier trả về entry có **priority cao nhất** trong tập entries khớp. Nếu nhiều entries cùng priority cao nhất cùng khớp, tie-breaker là *"first khớp với wins"*, tức entry được chèn trước (xem `ovs-ofctl(8)` mục `MATCH PRIORITY`).

Ví dụ flow table sau:

```bash
# Rule 1: cho phép TCP/80 đến h2
[host]# ovs-ofctl add-flow br-s1 \
  "priority=200,tcp,nw_dst=10.0.0.2,tp_dst=80,actions=output:2"

# Rule 2: chặn mọi TCP đến h2
[host]# ovs-ofctl add-flow br-s1 \
  "priority=100,tcp,nw_dst=10.0.0.2,actions=drop"

# Rule 3: forward mọi IP đến h2
[host]# ovs-ofctl add-flow br-s1 \
  "priority=50,ip,nw_dst=10.0.0.2,actions=output:2"
```

Packet TCP/80 đến h2: khớp cả Rule 1, 2, 3 — Rule 1 có priority 200 cao nhất → `output:2` (cho phép).

Packet TCP/22 đến h2: không khớp Rule 1 (port sai), khớp Rule 2 + 3 — Rule 2 có priority 100 cao hơn → `drop` (chặn SSH).

Packet ICMP đến h2: không khớp Rule 1 (không phải TCP), không khớp Rule 2 (không phải TCP), khớp Rule 3 — `output:2` (cho phép ping).

Đây là pattern kinh điển của ACL: rule specific ưu tiên cao, catch-all ưu tiên thấp. Toàn bộ logic fit trong flow table, không cần module firewall riêng.

> **Chủ đề trọng tâm:** Priority resolution cho phép gom rule firewall + routing + load balancing vào cùng một flow table. Việc OVS có thể làm tất cả trong một bridge thay vì chain nhiều bridge/iptables/route table là lý do kiến trúc cloud data plane chọn OVS làm backbone.

## 9.19.4 Idle_timeout, hard_timeout, và cookie

Ba field quản lý lifecycle của flow entry:

**`idle_timeout`** (0 = không giới hạn, default): số giây sau packet cuối khớp mà entry bị xóa. Dùng khi muốn garbage collect flow không hoạt động.

```bash
[host]# ovs-ofctl add-flow br-s1 \
  "priority=100,tcp,nw_dst=10.0.0.2,tp_dst=80,idle_timeout=60,actions=output:2"
```

Nếu 60 giây không có packet TCP/80 nào, entry tự xóa. Controller OpenFlow dùng mẫu này nhiều: cài flow specific khi thấy packet đầu tiên, để timeout tự clean up. Cơ chế tương tự ovn-controller khi xử lý conntrack flow.

**`hard_timeout`**: tuổi thọ tuyệt đối bất kể có traffic. Dùng cho flow có lifetime cố định (ví dụ rule authentication 1 giờ).

**`cookie`**: identifier 64-bit do controller/user gán để group flow. Không ảnh hưởng matching hay action, chỉ để sau này query/delete theo group:

```bash
# Cài một loạt flow với cookie=0x1001 (ví dụ: tenant A)
[host]# ovs-ofctl add-flow br-s1 \
  "cookie=0x1001,priority=100,ip,nw_src=10.1.0.0/24,actions=output:2"
[host]# ovs-ofctl add-flow br-s1 \
  "cookie=0x1001,priority=100,ip,nw_src=10.1.1.0/24,actions=output:2"

# Xóa toàn bộ flow của tenant A bằng cookie mask
[host]# ovs-ofctl del-flows br-s1 "cookie=0x1001/-1"
```

OVN sử dụng cookie để đánh dấu Logical Flow → OpenFlow mapping. Khi bạn `ovs-ofctl dump-flows br-int` trên node OVN, cookie value cho biết flow này thuộc Logical Flow nào (xem Phần 13.7). Cookie là cơ chế chính để `ovs-ofctl` kiểm tra OVN runtime state.

## 9.19.5 Microflow vs Megaflow trade-off (tier 2)

Phần 9.2 đã giới thiệu megaflow cache. Phần này đi sâu trade-off cho người designer flow table.

### Microflow

**Microflow** là cache entry per-connection (per 5-tuple). Mỗi packet match exact 5-tuple → 1 microflow entry. 1000 connection cùng lúc → 1000 entry.

Ưu điểm:
- Lookup nhanh (O(1) hash trên 5-tuple).
- Stat per-connection chính xác.

Nhược điểm:
- Memory bùng nổ với nhiều connection.
- Cache invalidation khi có thêm flow rule (mỗi flow_mod có thể invalidate hàng nghìn microflow).
- Không scale với connection rate cao (10K conn/s).

OVS cũ (pre-2.0) dùng microflow cache. NSDI 2015 paper Pfaff et al. document scaling problem này.

### Megaflow

**Megaflow** là cache entry per-mask (per-rule subset). Group nhiều microflow thành 1 entry với wildcard mask. 1000 connection cùng pattern → 1 megaflow entry.

Ưu điểm:
- Memory linear với số rule, không số connection.
- Cache invalidation chỉ áp dụng pattern affected, không broadcast.

Nhược điểm:
- Lookup phức tạp hơn (TSS subtable iteration).
- Mỗi unique mask = 1 subtable. Nhiều mask → nhiều subtable → lookup chậm.

OVS 2.0+ default megaflow. Mọi production cluster (OVN included) dùng megaflow.

### Khi nào fallback microflow

EMC (Exact Match Cache) trong OVS-DPDK là microflow-style cache trên top of megaflow. Hot connection (cao rate) hit EMC → O(1) lookup. Less-hot connection fall through tới megaflow.

Trade-off design: EMC nhỏ (8K entry default per PMD), megaflow scalable. Pattern this stack approach (EMC + SMC + megaflow trong OVS 2.15+) cho cả speed + scale.

## 9.19.6 Wildcard mask design + subtable creation

Megaflow mask = which fields exact-match vs wildcarded. Wildcard mask design quyết định subtable count.

### Mask construction từ rule

Khi packet đầu tiên match rule, OVS classifier tracking field nào được "consulted":

```
Rule: priority=100,tcp,nw_dst=10.0.0.2,tp_dst=80,actions=output:2

Packet match: in_port=1,eth_src=AA:BB:CC:00:00:01,eth_dst=AA:BB:CC:00:00:02,
              eth_type=0x0800,nw_src=10.0.0.1,nw_dst=10.0.0.2,
              nw_proto=6,tp_src=12345,tp_dst=80

Megaflow mask after lookup:
  eth_type=0xffff (consulted, exact value 0x0800)
  nw_proto=0xff   (consulted, exact value 6)
  nw_dst=0xffffffff (consulted, exact value 10.0.0.2)
  tp_dst=0xffff   (consulted, exact value 80)
  ALL OTHER FIELDS WILDCARDED (mask=0)

Megaflow entry: 1 entry covering ALL connection có 5-tuple matching pattern này
```

Connection thứ 2 với khác `nw_src` hoặc `tp_src` → cùng megaflow → cache hit.

### Subtable count = unique mask count

Mỗi unique megaflow mask = 1 subtable. 100 rule với 100 unique mask → 100 subtable → classifier check 100 subtable mỗi cache miss.

Production target: < 10 subtable per bridge để classifier cost < 1 µs.

Đo subtable:

```bash
[host]# ovs-appctl dpctl/dump-flows br-int | head -5
[host]# ovs-appctl upcall/show
  threads: 4
  ...
[host]# ovs-appctl dpif-netdev/dpif-show
  br-int (port count: 5)
    masks-cache-hits: 12345678
    masks-cache-misses: 12345
    masks: 7    ← subtable count
```

`masks: 7` = 7 unique mask = 7 subtable. Healthy < 10. Warning 10-50. Critical > 50 (xem 9.15 subtable explosion).

### Pattern thiết kế tốt

**Tốt:** Rule cùng layer + cùng field set:
- 100 rule khớp với `nw_dst` (L3 router) → 1 mask, 1 subtable.
- 100 rule khớp với `nw_dst + tp_dst` (L4 firewall) → 1 mask khác, 1 subtable.
- Total 2 subtable.

**Xấu:** Rule mixed pattern:
- 1 rule khớp `in_port`
- 1 rule khớp `dl_dst`
- 1 rule khớp `nw_src`
- 1 rule khớp `nw_dst`
- 1 rule khớp `nw_dst + tp_dst`
- 1 rule khớp `nw_src + nw_dst + tp_dst`
- Total 6 subtable, mỗi subtable chỉ 1 rule.

Solution: refactor rule để consolidate mask. Đôi khi dùng OR explicit (Multi-flow rule) thay vì rely on classifier wildcard discovery.

## 9.19.7 Match field cardinality + classifier cost

Field cardinality = số giá trị có thể của field. Cao cardinality = match rule riêng biệt nhiều = subtable scaling tốt; low cardinality = rule cluster vào 1 subtable.

| Field | Cardinality | Subtable impact |
|-------|-------------|-----------------|
| `in_port` | < 100 (port count) | Low cardinality, 1 subtable cho 100 rule khớp với port |
| `dl_dst` (MAC) | 2^48 | High cardinality, 100 rule khớp khác MAC → 1 subtable nhưng table lớn |
| `dl_type` | ~10 phổ biến | Low cardinality, group rule by L3 protocol |
| `nw_dst` (IP/32) | 2^32 | High cardinality, IP-based rule scale |
| `nw_dst` (subnet) | với prefix length | Patricia trie optimization (xem 9.15.8) |
| `nw_proto` | ~5 phổ biến | Low cardinality |
| `tp_dst` (port) | 65536 | High cardinality |
| `ct_state` | bitfield 8 bits | Very low cardinality, group conntrack rules |

**Pattern hiệu quả:**

1. **Filter sớm bằng low-cardinality field**: rule khớp `dl_type=0x0800,nw_proto=6` (TCP) trước, sau đó subtable IP/port. Reduce search space.
2. **Patricia trie cho IP prefix**: 9.15.8 documents cmap+trie. 10K rule khớp với khác CIDR → 1 trie, lookup O(log N).
3. **Bitmask cho conntrack state**: rule khớp với `ct_state=+est+trk` group thành 1 subtable thay vì state-specific subtable.

## 9.19.8 Anatomy Template A: `dpctl/dump-flows` masked output

> **Bối cảnh khi chạy:** Kiểm tra megaflow entries thực tế trên kernel datapath, đối chiếu với rule trên flow table.

```
[host]# ovs-appctl dpctl/dump-flows br-int | head
recirc_id(0),in_port(2),eth_type(0x0800),ipv4(dst=10.0.0.2,proto=6,frag=no),
  tcp(dst=80), packets:12345, bytes:1234567, used:0.123s, actions:1

recirc_id(0),in_port(2),eth_type(0x0800),ipv4(proto=6,frag=no),tcp(),
  packets:0, bytes:0, used:never, actions:drop

recirc_id(0),in_port(2),eth_type(0x0800),ipv4(),
  packets:567, bytes:56789, used:1.234s, actions:1
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| `recirc_id` | 0 (initial) hoặc N (post-recirc) | (n/a) | Empty | conntrack/group recirculation |
| `in_port` | Match expected | Wildcard `()` | Empty | Wildcarded mask check |
| `eth_type` | Specified | Wildcard | Empty | Higher-layer fields wildcarded too |
| `ipv4()` empty | OK (catch-all IPv4) | (n/a) | (n/a) | More wildcarded |
| `ipv4(dst=...,proto=...)` | Specific match | (n/a) | (n/a) | Specific subtable |
| `tcp()` empty | Wildcard L4 | (n/a) | (n/a) | Less specific |
| `tcp(dst=80)` | Specific port match | (n/a) | (n/a) | Most specific subtable |
| `packets` | > 0 cho active flow | 0 (used=never) | 0 + non-stale | Rule chưa match traffic |
| `used` | < 10s cho active | 60-300s | > 600s | Stale, sẽ aged out |
| `actions:N` | OFPort number | `drop` | Empty | Verify drop intentional |

**Kịch bản bẻ gãy:**

- Same rule pattern xuất hiện 3+ time với khác mask: subtable explosion. Refactor rule.
- `actions:drop` cho rule expected forward: priority resolution wrong. Check `ovs-ofctl dump-flows br-int -O OpenFlow13` (high-level) so với `dpctl/dump-flows` (datapath megaflow).
- `packets:0` cho rule mới install: chưa match traffic. Wait or send synthetic.

### Khác biệt `ovs-ofctl dump-flows` vs `dpctl/dump-flows`

- `ovs-ofctl dump-flows` (high-level): show OpenFlow rule (Bridge state). Showing intent.
- `dpctl/dump-flows` (low-level): show megaflow datapath (kernel state). Showing actual cache.

Mismatch = ovs-vswitchd not yet installed expected megaflow. Cause:
- Cache cold (chưa có packet kích hoạt cache).
- Userspace busy upcall.
- Kernel datapath full (rare).

## 9.19.9 ▶ Capstone POE: 1 wildcard rule vs N specific rule, đâu nhanh hơn?

> **Bối cảnh:** Kỹ sư thiết kế flow table cho 1000 VM cùng subnet 10.0.0.0/24, traffic toàn bộ go ra port 1. 2 cách:
>
> - **Cách A:** 1 rule `priority=100,ip,nw_dst=10.0.0.0/24,actions=output:1` (wildcard).
> - **Cách B:** 1000 rule `priority=100,ip,nw_dst=10.0.0.X,actions=output:1` (specific per VM).

**Tuyên bố cần kiểm chứng:** "1000 specific rule (Cách B) chậm hơn 1000x so với 1 wildcard rule (Cách A) vì classifier phải duyệt nhiều rule hơn."

**Predict:** Sai. Megaflow + Patricia trie cho phép cả 2 đều O(log N) hoặc O(1) lookup.

**Observe (giả định):**

- **Cách A** 1 wildcard rule: 1 subtable với mask `nw_dst=0xffffff00`. Megaflow cache 1 entry cho mọi destination. Lookup O(1).
- **Cách B** 1000 specific rule: 1 subtable với mask `nw_dst=0xffffffff` (all bits exact). Megaflow cache 1000 entry (1 per destination). Lookup O(log N) qua cmap hash. Patricia trie không help vì exact match.
- Throughput diff < 5% giữa 2 cách.
- Memory: Cách B tốn 1000 entry × 200 byte = 200KB. Cách A 1 entry × 200 byte. **Memory 1000x khác nhau, throughput gần như giống**.

**Explain:**

1. **Megaflow cache O(log N) hoặc O(1)** không phụ thuộc rule count linearly. Cmap hash cho exact match, Patricia trie cho prefix match. Classifier không "duyệt N rule" mỗi packet.
2. **Memory mới là constraint thực**. Với 1M VM, Cách B tốn 200MB chỉ cho flow table → eviction pressure.
3. **Maintenance cost**: Cách B có 1000 rule cần install/delete khi VM lifecycle. Cách A có 1 rule. Operation cost khác biệt lớn.

**Falsification test:**

- Setup OVS với Cách A vs Cách B, run pktgen trên 100 different `nw_dst` trong 10.0.0.0/24.
- Đo Mpps + memory rss của ovs-vswitchd.
- Nếu Cách B chậm > 10% → counter-example. Trong thực tế, < 5% throughput diff.

**Tuyên bố sai ở đâu:** "Chậm hơn 1000x" assumption O(N) lookup. Đúng phải: **Megaflow lookup O(log N) hoặc O(1) qua cmap+trie. Wildcard rule preferable cho operation simplicity + memory, không phải throughput.**

> **Quy tắc:** Wildcard rule preferable khi pattern uniform (subnet, port range). Specific rule chỉ khi cần per-entity action (per-VM rate-limit, per-VM logging). Don't equate "rule count" với "throughput".

## Điểm cốt lõi cần nhớ

Thứ nhất, flow entry có cấu trúc đồng nhất, match + counters + actions + metadata (priority, timeout, cookie, table). Mọi loại chuyển tiếp (L1→L4) đều dùng chung cấu trúc này, chỉ khác ở khớp với field được chọn.

Thứ hai, OVS không auto-learn MAC trừ khi flow action chỉ định `NORMAL`. Đây là điểm phân biệt OVS với switch truyền thống. Learning là tính năng opt-in, không phải mặc định.

Thứ ba, priority resolution cho phép gom firewall + routing + load balancing vào cùng flow table. Rule specific priority cao, catch-all priority thấp, pattern ACL cơ bản.

Thứ tư, `idle_timeout` + `hard_timeout` + `cookie` là bộ công cụ quản lý lifecycle của flow entry. Cookie đặc biệt quan trọng khi debug OVN, nó link OpenFlow runtime với Logical Flow abstraction.

## Context: the lifecycle of an LTS protocol

Part 4.1 analysed OF 1.2 with its shift to extensible OXM TLV matching and HA controller roles. However, 1.2 was a "transient version": it released on 2011-12-05 but only five months later, OF 1.3.0 came out (2012-04-25) and immediately took its place as the reference version. Market adoption skipped 1.2 almost entirely, jumping straight from 1.0 to 1.3.

OF 1.3 differs from other versions on one key point: it has an **errata chain** that maintained backward compatibility for nearly three years, from 1.3.0 (2012-04-25) to 1.3.5 (2015-03-26). There were no breaking changes between minor versions; each release only fixed ambiguity and clarification in the spec. Vendors who implemented 1.3.0 could upgrade to 1.3.5 without rewriting the matching engine or parser logic. This is the key reason why the SDN ecosystem (OVS, the Ryu framework, OpenDaylight, hardware vendors) chose 1.3 as the long-term baseline.

## 4.2.1 OF 1.3 release timeline: 1.3.0 (2012-04-25) to 1.3.5 (2015-03-26)

### The errata release chain

ONF published the OpenFlow Switch Specification 1.3 as a chain of six versions over nearly three years:

| Version | Release date | Wire protocol | Type of change |
|---------|--------------|---------------|----------------|
| 1.3.0 | 2012-04-25 | 0x04 | Major release, meter table plus auxiliary channels plus PBB |
| 1.3.1 | 2012-09-06 | 0x04 | Errata, wording clarification |
| 1.3.2 | 2013-04-05 | 0x04 | Errata, ambiguity in OXM parsing resolved |
| 1.3.3 | 2013-09-27 | 0x04 | Errata, spec prose edit |
| 1.3.4 | 2014-08-14 | 0x04 | Errata, conformance test clarification |
| 1.3.5 | 2015-03-26 | 0x04 | Final errata for the 1.3 branch |

All six versions use the same wire-protocol byte `0x04`. There are no breaking changes between any two 1.3.x versions, which means a switch implementing 1.3.0 can communicate with a controller speaking 1.3.5 and vice versa.

### Meaning of the errata chain

The errata chain is a luxury that other OpenFlow versions did not have. 1.0 was released on 2009-12-31 and afterwards had no errata; vendor implementations were "done". 1.1 released on 2011-02-28 also had no errata (because adoption was low, mass-market clarification was unnecessary). 1.4 (2013-10-14) had 1.4.1 (2014-10-17) but stopped there. 1.5 had 1.5.1 (2015-03-26) on the same day as 1.3.5 and then ONF did not revise it further.

Only 1.3 has a six-version errata chain, which reflects one fact: 1.3 is the only version with enough industry adoption to force ONF to maintain the spec continuously. Each time vendors implemented and discovered ambiguity (for example, parser behavior when encountering an unknown OXM class), ONF reviewed and published an errata. This process continued for three years until ONF shifted focus to ONOS and CORD (already discussed in 3.2.6).

The full archive of all six spec PDFs is available at:

```
https://opennetworking.org/sdn-standards/openflow/
```

Or directly for 1.3.5 (the final version, commonly used as reference):

```
https://opennetworking.org/wp-content/uploads/2014/10/openflow-switch-v1.3.5.pdf
```

## 4.2.2 Meter table: per-flow QoS rate limiting

### Problem: QoS needs per-flow granularity

Before OF 1.3, OpenFlow had no native QoS mechanism. A controller wanting to rate-limit traffic had to rely on the Queue concept (OF 1.0 §5.3) with a fixed port-to-queue mapping, where each queue had `min_rate` and `max_rate` configured out-of-band (not through the OpenFlow protocol). Consequence: rate-limit policy applied only per-port-queue, not per-flow. An application wanting to limit tenant A to 1 Mbps and tenant B to 10 Mbps on the same port required two separate queues, and the controller could not control queue configuration through OpenFlow.

OF 1.3 solves this with the **meter table**, an independent table parallel to the flow tables, allowing the operator to define per-flow rate limits and invoke them from an instruction inside a flow entry.

### Meter table structure (§5.7)

Each meter entry has:

- **Meter identifier** (32-bit): unique ID inside the switch, referenced by the controller at install time.
- **Meter bands**: list of bands, each with a type (DROP or DSCP_REMARK) plus rate (KB/s or packets/s) plus burst size plus optional `prec_level` for DSCP_REMARK.
- **Counters**: byte count, packet count, duration.

A flow-entry instruction uses `OFPIT_METER` to activate the meter before applying actions. Spec 1.3 §5.9:

```
flow_mod: match=..., instructions=[OFPIT_METER:meter_id=1, OFPIT_APPLY_ACTIONS:[OUTPUT:port 2]]
```

When a packet matches this flow entry, the switch looks up meter ID 1, applies the rate check, and only then forwards.

### Token bucket semantic

The meter band uses the **token bucket** algorithm:

- Bucket size = burst (bytes or packets).
- Bucket fill rate = rate (bytes/s or packets/s).
- Each packet through the meter consumes `packet_size` tokens (byte mode) or 1 token (packet mode).
- If the bucket has enough tokens, the packet proceeds; if depleted, the band action triggers.

Parameters:

| Parameter | Unit | Meaning |
|-----------|------|---------|
| rate | KB/s or packets/s | Token fill rate |
| burst_size | KB or packets | Maximum burst before rate limit |
| type | OFPMBT_DROP or OFPMBT_DSCP_REMARK | Band action |
| prec_level | 1 to 3 (only for DSCP_REMARK) | Number of drop precedence levels |

### Two band types

**OFPMBT_DROP**: drops the packet when the rate is exceeded. Simple, used for hard rate-limiting (policing).

**OFPMBT_DSCP_REMARK**: does not drop but re-marks DSCP to a lower drop precedence. When a downstream router hits congestion, packets with low DSCP drop first. Used for soft rate-limiting (marking, AF assured forwarding classes).

### Multiple bands: color marking

A meter can have multiple bands sorted by ascending rate. Tokens are checked in order: band 1 (lowest rate) is checked first; if it passes, then band 2, and so on. Consequence: packets divide into multiple color classes:

- Pass all bands: green (conforming to the lowest rate).
- Pass band 1, fail band 2: yellow (between band 1 and band 2).
- Fail the lowest band: red (exceed all limits).

This pattern is commonly called CIR/PIR marking (Committed Information Rate / Peak Information Rate) per RFC 2697 srTCM or RFC 2698 trTCM. The OF 1.3 meter table allows implementing both schemes by configuring the band list.

### Forward cross-reference

Part 9.9 will analyse how OVS implements the meter table in practice with the `linux-htb` QoS recipe, ingress policing, and kernel datapath rate limiting. Part 16.x (SDN WAN, removed from rev 3 scope, but the materiality still exists in industry) is where the meter table is actually used in production for provider-Ethernet QoS and SD-WAN.

### Common misunderstandings about meter

**Misunderstanding:** "Installing a 1 Mbps meter on a flow means packets exceeding 1 Mbps will be dropped entirely."

**Reality:** A meter has no default action. Rate limiting only happens when a band is configured. If the meter has rate=1000 (meaning 1000 KB/s) but no OFPMBT_DROP band, packets exceeding the rate still pass normally (the meter only counts via the counter, no drop). Two other confusing cases: a meter with a single DSCP_REMARK band does not drop packets but only re-marks them, so the downstream router decides to drop on congestion; a meter with multiple bands sorted ascending creates color marking (green, yellow, red), where packets pass through the meter even at high rates and are simply marked differently. The controller must explicitly design the band topology to achieve the desired behavior.

## 4.2.3 Per-connection auxiliary channels (multi-socket)

### Problem: head-of-line blocking on the main connection

Specs 1.0 and 1.2 specify a single TCP/TLS connection between switch and controller, where all messages (symmetric, controller-to-switch, async) share that one connection. Each message is sequential, FIFO. When PACKET_IN bursts run high (for example 10K new flows per second), the main-connection TCP queue floods, the controller falls behind, and the sequential nature delays every message (including the FLOW_MOD response). This is **head-of-line blocking**.

### OF 1.3 auxiliary channels (§6.3.7)

Spec 1.3 lets the switch open additional **auxiliary connections** in parallel with the main connection to the same controller:

- Main connection: TCP (or TLS), used for control messages (FLOW_MOD, BARRIER, ROLE_REQUEST, STATS).
- Auxiliary connection(s): may be TCP, TLS, DTLS, or UDP, dedicated to high-volume async (PACKET_IN).

Each auxiliary connection is identified through the `auxiliary_id` field (8-bit) in the HELLO handshake. The main connection uses aux_id=0 and auxiliaries use aux_id 1 to 255.

A typical processing flow:

1. Switch sends PACKET_IN async on the auxiliary connection.
2. Controller processes in a parallel thread pool and computes a decision.
3. Controller replies with PACKET_OUT or installs FLOW_MOD on the main connection.

### Reality of deployment: a "paper-only" feature

The auxiliary channels feature is rarely deployed in practice. Reasons:

- **OVS does not support it**: Open vSwitch upstream only implements the main connection across the entire 2.0+ lifecycle. Patch series proposing aux channels in 2013 to 2014 were rejected by maintainers because complexity was disproportionate to benefit (a controller can scale horizontally instead of multi-channel per switch).
- **Hardware vendors skipped it**: HP ProCurve, NEC, and Pica8 did not implement aux. Controller frameworks (Ryu, ODL, ONOS) have no default handler for aux.
- **UDP auxiliary is too risky**: UDP means PACKET_IN packet loss, a trade-off that is unacceptable for policy-critical async.

Result: §6.3.7 became a "spec feature for completeness", not deployed. Production SDN still uses a single main connection and solves head-of-line blocking another way (proactive install, DevoFlow-style clone trigger, or controller process pool).

## 4.2.4 PBB (IEEE 802.1ah) match and set

### Context: Provider Backbone Bridge for carrier scale

802.1Q VLAN has a 4094-ID limit, which is too few for carrier Ethernet backbones serving thousands of enterprise customers. IEEE standardised 802.1ah "Provider Backbone Bridges" in 2008 with **MAC-in-MAC encapsulation**:

- The customer Ethernet frame (with its 12-bit S-VLAN tag) is encapsulated inside an outer Ethernet frame.
- The outer frame carries an I-SID (Service Instance Identifier, 24-bit) instead of a VLAN ID.
- The 24-bit I-SID allows **16,777,216 service instances**, expanding scale by a factor of 4096 over VLAN.

PBB structure (Backbone MAC addresses: B-SA, B-DA, B-VID; I-TAG with I-SID, PCP, DEI, UCA):

```
[Outer ETH header B-DA/B-SA/B-VLAN] [I-TAG: I-SID/UCA/PCP] [Customer ETH header]
```

### OF 1.3 matching on PBB fields (§7.2.3.8)

Spec 1.3 adds two OXM fields for PBB:

- **OXM_OF_PBB_ISID** (24-bit): match the I-SID of the outer frame.
- **OXM_OF_PBB_UCA** (1-bit): "Use Customer Address" flag, indicating whether the customer MAC address is used.

Corresponding set actions allow the switch to push and pop the PBB header:

- `OFPAT_PUSH_PBB`: encapsulate the customer frame inside PBB.
- `OFPAT_POP_PBB`: decapsulate the PBB header, restoring the customer frame.

### Use case: OpenFlow switch as carrier Ethernet PE

With PBB support, an OpenFlow switch can function as a Provider Edge (PE) in a carrier Ethernet network:

- Customer frame (VLAN X) arrives at the switch.
- Controller installs a flow matching VLAN X with action PUSH_PBB carrying I-SID=100.
- The packet is encapsulated and forwarded across the backbone via the B-VLAN.
- The far-end switch performs POP_PBB, restoring the customer frame, and forwards locally.

OpenFlow's scope expanded from campus and DC to provider edge, targeting a new market (NTT, Deutsche Telekom, Verizon). Some vendors (Ciena, NEC) implemented PBB support in OF-capable ASICs for carrier-Ethernet product lines. But just like auxiliary channels, mass-market adoption was low, mostly limited to niche carrier deployments.

## 4.2.5 Why 1.3 is LTS: vendor alignment, OVS support, Ryu and ODL baseline

### Ecosystem alignment

Between 2012 and 2015, the key components of the SDN ecosystem agreed to choose 1.3 as the baseline:

**Open vSwitch 2.0+ (2013-07)**: full feature support for 1.3 in the kernel datapath and userspace. Pfaff et al. (NSDI 2015) describe in detail how OVS implements OF 1.0 and 1.3, skipping 1.1 and 1.2 due to low adoption. From 2013 onward, every production OVS deployment uses 1.0 or 1.3.

**Ryu Framework (NTT, 2013+)**: the official Ryu tutorial uses 1.3 for every sample. An application developer learning SDN through Ryu writes OF 1.3 code by default.

**OpenDaylight Helium (2014-10)**: ODL Helium release (the second release after Hydrogen 2014-02) set 1.3 as the default OF version for the OpenFlow plugin. From there every ODL release (Lithium, Beryllium, Boron, Carbon, and so on) maintained 1.3 support.

**Hardware vendor commitment**:
- Pica8 PicOS uses OF 1.3 as the baseline for whitebox switches.
- HP ProCurve and Aruba ArubaOS support 1.3 fully and 1.4 partially.
- NEC IP8800 and Centec V350 hardware target 1.3 conformance.
- Mellanox Spectrum ConnectX-6+ supports OF offload 1.3 via switchdev.

### Market economics of LTS

The economic reason behind vendor alignment is the **implementation cost per spec version**. Hardware vendors spend many quarters of engineering effort to tune ASIC for the spec, verify conformance through ONF certification labs, and train sales teams. Each spec revision equals significant cost.

When the 1.3 errata chain 1.3.0 to 1.3.5 maintained stability for three years, vendors returned to "implement once, maintain forever". Compare with 1.4 (1.4.0 released 2013-10, 1.4.1 released 2014-10 then halted) or 1.5 (released 2014-12, errata 1.5.1 released 2015-03 then halted): 1.3 has a clearly superior economic lifespan.

### Counter-evidence on adoption

1.4 and 1.5 adoption is near-zero on hardware silicon. When Intel EOL'd the FM10000 line (2017-11, cross-ref 6.0), no remaining hardware committed silicon for 1.4 or 1.5 in production. 1.4 and 1.5 are mostly academic exploration and vendor "checkbox feature" for tenders, not deployed in reality.

Detailed adoption data for 1.4 and 1.5 will be analysed in Part 4.3.5 and Part 4.4.5.

> **Key point.** OpenFlow 1.3 is a case study in how a protocol spec becomes LTS: not because it has the richest features, but because it has an errata chain long enough for vendors to commit silicon and for the software ecosystem (OVS, Ryu, ODL, ONOS) to align on it. When a 2026 user says "OpenFlow" without a version, the default is 1.3. This lesson applies to other LTS protocols (HTTP/1.1 versus 2 versus 3, TLS 1.2 versus 1.3, and so on): for production deployment, stability matters more than novelty.

## Guided Exercise: implement OF 1.3 meter rate-limiting with OVS (audit P4.B4.1 hands-on)

**Goal:** practise per-flow QoS rate-limit with the meter table, and verify drop behavior when traffic exceeds rate.

**Steps:**

1. Create an OF 1.3 bridge:
   ```bash
   $ sudo ovs-vsctl add-br br-meter -- set bridge br-meter protocols=OpenFlow13
   ```
2. Create a meter, 1 Mbps drop threshold:
   ```bash
   $ sudo ovs-ofctl -O OpenFlow13 add-meter br-meter \
       "meter=1,kbps,band=type=drop,rate=1000,burst_size=500"
   # meter_id=1, 1 Mbps (1000 kbps), burst 500 KB, exceed -> drop
   ```
3. Install a flow with the meter instruction:
   ```bash
   $ sudo ovs-ofctl -O OpenFlow13 add-flow br-meter \
       "table=0,priority=100,ip,actions=meter:1,NORMAL"
   ```
4. Generate traffic and observe drops:
   ```bash
   $ sudo ovs-ofctl -O OpenFlow13 dump-meters br-meter
   # Observe meter counters: packet_count, byte_count, byte_band_count (drop)
   ```
5. Verify meter drop behavior with iperf above 1 Mbps:
   ```bash
   # Client to Server, high bandwidth
   $ iperf3 -c <server> -b 10M
   # Expected: throughput capped near 1 Mbps, packet loss high
   ```

**POE:**
- **Predict:** 10 Mbps traffic through a 1 Mbps drop-band meter creates how much packet loss?
- **Observe:** about 90 percent loss (only 1 Mbps conforms, 9 Mbps drops).
- **Explain:** Meter token-bucket refill rate is 1 Mbps. The 500 KB burst absorbs short-term spikes. Steady 10 Mbps empties the bucket and triggers drops. This differs from active-backup or ACL rate-limit: the meter integrates with the flow pipeline.

**Cleanup:** `sudo ovs-vsctl del-br br-meter`

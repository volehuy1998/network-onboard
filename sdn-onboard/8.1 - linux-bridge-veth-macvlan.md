## 8.1.1 brctl legacy versus iproute2 modern

The package `bridge-utils` (the `brctl` command) has been deprecated since Ubuntu 18.04. The modern syntax uses iproute2:

```bash
# Legacy (avoid):
[host]# brctl addbr br0
[host]# brctl addif br0 eth0

# Modern:
[host]# ip link add name br0 type bridge
[host]# ip link set eth0 master br0
[host]# ip link set br0 up
```

The kernel implementation lives in `net/bridge/`. The FDB (forwarding database) ages entries with a default of 5 minutes. iproute2 unifies the bridge, VLAN, and tunnel commands under one interface.

VLAN filtering on a Linux bridge (enabled with `vlan_filtering=1`):

```bash
[host]# ip link add name br0 type bridge vlan_filtering 1
[host]# bridge vlan add dev eth0 vid 100 pvid untagged
[host]# bridge vlan add dev eth1 vid 100
```

VLAN filtering on a Linux bridge is not as complete as on OVS. There is no per-port VLAN translation, no native QinQ, and no programmable matching.

## 8.1.2 Bridge datapath in the kernel

`net/bridge/br_forward.c` implements the forwarding logic:

1. A packet enters a bridge port, which calls the `br_handle_frame` hook.
2. Look up the FDB by (destination MAC, VLAN ID).
3. Match: forward to the destination port.
4. Miss: flood to every port (except the ingress port).
5. Update the FDB with (source MAC, VLAN ID, ingress port) for learning.

STP is optional and disabled by default. The netfilter bridge hook (`ebtables`) provides L2 filtering. Real-world throughput of the kernel bridge is roughly 5 to 10 Gbps on modern hardware, which is enough for most container workloads that do not require SDN.

Important limits:

- **Not programmable:** the forwarding rule is fixed (MAC learning plus flood-on-miss).
- **No per-flow statistics:** counters are per-port only.
- **No complex actions:** no `set_queue`, `mod_nw_src`, or `push_vlan` per rule.

## 8.1.3 veth pair: `net/core/veth.c`

veth creates a symmetric device pair: a packet pushed into one end exits at the other end, like a virtual Ethernet cable.

```bash
[host]# ip link add veth-a type veth peer name veth-b
[host]# ip link set veth-a netns ns1
[host]# ip link set veth-b master br0    # or netns ns2
```

Main use case: connect a namespace to a bridge (one end inside the namespace, the other end attached to the bridge). Implementation: a virtual Ethernet driver with synthetic TX and RX queues, no hardware offload, pure CPU work, with overhead of about 1 to 2 microseconds per packet hop.

USC Lab 1 uses this pattern:

```
namespace h1 - veth1 -- veth1-br - bridge s1 - veth2-br -- veth2 - namespace h2
```

veth connects each namespace to the bridge, and the bridge acts as the L2 switch between namespaces.

---

## 8.1.4 macvlan and its four modes

macvlan creates a virtual interface with its own MAC address on the same physical port. Each macvlan sibling can have its own IP address and appears as an independent device to the external switch.

**`private`:** siblings do not forward to each other. Traffic between two containers on the same host goes out to the external switch (and reaches its destination only if the switch routes it back), or it is dropped. Rarely used because of the poor performance.

**`vepa`** (Virtual Ethernet Port Aggregator, IEEE 802.1Qbg): traffic between siblings goes out to the external switch and the switch hairpins it back. This requires switch support for reflective relay.

**`bridge`** (default): containers communicate directly inside the kernel bridge domain. This is the fastest and the most common mode.

**`passthru`:** expose a 1-to-1 physical port to a single container (similar to SR-IOV). The container sees the physical port directly.

```bash
[host]# ip link add mac1 link eth0 type macvlan mode bridge
[host]# ip link set mac1 netns ns1
[host]# ip netns exec ns1 ip addr add 10.0.0.100/24 dev mac1
```

A drawback of macvlan: by design, it cannot communicate with the host namespace. Docker does not use macvlan as the default CNI because the host cannot ping the container.

---

## 8.1.5 ipvlan, L2 versus L3

ipvlan shares **a single MAC address** with multiple IP addresses. It differs from macvlan in that no new MAC is created.

**L2 mode:** switching happens in the kernel, similar to macvlan in `bridge` mode but with a shared MAC. The external switch sees only one MAC, so it does not consume a MAC table entry per container.

**L3 mode:** the kernel performs L3 forwarding between ipvlan children. There is no broadcast and no ARP across the IPVLAN boundary. Each container must have a default route through the ipvlan parent.

Use case for ipvlan: container overlays that must scale to more than 1000 containers per host. The MAC table of the upstream physical switch has a hard limit (typically 32K to 128K entries); too many containers each with their own MAC (the macvlan case) can overflow it.

```bash
[host]# ip link add ipvl1 link eth0 type ipvlan mode l2
[host]# ip link set ipvl1 netns ns1
```

## 8.1.6 The `br-int`, `br-ex`, `br-tun` pattern (compass Ch F)

Compass Ch F describes the canonical multi-bridge pattern for OVS. The same foundational concept also applies to a Linux bridge:

```
      VMs                           Physical
       |                               |
   +---v------+   patch   +------------v-+
   | br-int   +---------->+ br-ex         |
   +-+--------+           +---------------+
     | patch
   +-v--------+
   | br-tun   +-- VXLAN or Geneve to remote hosts
   +----------+
```

- **`br-int`:** carries VM and container facing traffic. OpenFlow policy lives here.
- **`br-ex`:** peers with the physical uplink, performs VLAN translation, NAT, and provides the provider network.
- **`br-tun`:** terminates the overlay tunnel.

OVS patch ports (Part 9.1) connect the three bridges with zero-copy (compass: "zero-copy tail-call, not a real queue"). A Linux bridge has no native patch port and must use a veth pair, which is slower because there are real RX and TX queues.

This is one of the reasons OVS is superior for SDN deployments.

---

## 8.1.7 Why OVS replaces the Linux bridge for SDN

Linux bridge limits:

- Not programmable, no OpenFlow API.
- No per-flow statistics.
- No per-flow actions (`set_queue`, modify headers).
- VLAN handling is simple, no native QinQ.
- No zero-copy patch port between bridges.

OVS solves these:

- OpenFlow pipeline (Part 3.1, Parts 4.0 to 4.6) gives a programmable classifier.
- Megaflow cache (Part 9.2) gives automatic per-flow statistics.
- Extensible action set (Part 4.7) provides `set_queue`, `mod_*`, `ct()`, `set_tunnel`.
- Native VXLAN, Geneve, and STT support (Part 11.0).
- Zero-copy patch port.

Trade-off: a Linux bridge is simpler to operate and is enough for workloads that do not need SDN. OVS is more complex, but it is required when you need overlays, multi-tenancy, or policy routing.

## 8.1.8 veth driver tier 2 source code (`drivers/net/veth.c` v5.15)

The veth driver is a kernel-mode virtual Ethernet driver. When a packet enters one end, the driver hands it to the peer through `netif_rx()` without a hardware queue. Look up the main function:

### `veth_xmit()`, packet transmission

```c
// drivers/net/veth.c, function veth_xmit (called from ndo_start_xmit)
static netdev_tx_t veth_xmit(struct sk_buff *skb, struct net_device *dev)
{
    struct veth_priv *priv = netdev_priv(dev);
    struct net_device *rcv;
    int length = skb->len;
    
    rcu_read_lock();
    rcv = rcu_dereference(priv->peer);
    if (unlikely(!rcv)) {
        kfree_skb(skb);
        goto drop;
    }
    
    if (likely(veth_forward_skb(rcv, skb, ...) == NET_RX_SUCCESS)) {
        // packet has been pushed into the peer's RX queue
    }
    rcu_read_unlock();
    return NETDEV_TX_OK;
}
```

Key points:

- `priv->peer` is the `struct net_device *` of the other end, assigned when the veth pair was created.
- `rcu_read_lock()` ensures the peer is not freed while it is being accessed (for example, while the peer namespace is tearing down).
- `veth_forward_skb()` calls `__dev_forward_skb()` and then `netif_rx()` to inject the packet into the peer's RX path.

There is no DMA, no hardware queue, and no IRQ. All processing is pure CPU.

### `veth_xdp_xmit()`, XDP fast path (kernel 4.18 and later)

Modern veth supports XDP (eXpress Data Plane). When the peer attaches an XDP program, the packet goes through the XDP hook before entering the TCP/IP stack:

```c
static int veth_xdp_xmit(struct net_device *dev, ...)
{
    // Bulk forward XDP frames
    for (i = 0; i < n; i++) {
        struct xdp_frame *frame = frames[i];
        // Inject into the peer's XDP RX path
    }
}
```

Performance: the XDP path bypasses skb allocation and the protocol stack, giving 5 to 10 times the throughput of regular `veth_xmit` (15 to 30 Gbps instead of 3 to 5 Gbps).

### Memory accounting

veth has no special memory cost: only the `struct net_device` (about 2 KB) plus the `skb` that is passed through. Unlike a container runtime tag, this is just plain netdev.

### Limitations

- **No hardware offload** (TSO, GSO, GRO are software only). Maximum inter-namespace throughput on the regular path is roughly 5 to 10 Gbps.
- **No checksum offload:** the kernel must compute the checksum for every packet.
- **CPU bound:** one CPU core per veth pair direction. The work cannot be spread through RSS.

### Comparison with an OVS internal port

An OVS internal port (`type=internal`) has higher overhead (it goes through the OVS userspace upcall path on a cache miss), but it has these advantages:

- Programmable OpenFlow (action set, priority, conntrack).
- Statistics per port and per flow.
- Hardware offload through the TC offload framework (Part 9.5).

OVN-Kubernetes uses OVS internal ports instead of veth because it needs programmability. Legacy Docker uses veth plus a bridge because it is simpler.

## 8.1.9 Anatomy Template A: `bridge fdb show` and `ethtool -S veth*`

> **Context when run:** Inspect the FDB of a Linux bridge plus the statistics of a veth pair.

```bash
[host]# bridge fdb show br br0
01:00:5e:00:00:01 dev veth1-br self permanent       # multicast group join
33:33:00:00:00:01 dev veth1-br self permanent
aa:bb:cc:00:00:01 dev veth1-br vlan 1 master br0    # learned host MAC
aa:bb:cc:00:00:02 dev veth2-br vlan 1 master br0
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| `<MAC>` | unique per host | duplicate (different port) | same MAC across multiple bridges | MAC flap, check for a loop |
| `dev` | matches the expected egress port | `master br0 self` for permanent (multicast) | wrong port | Bridge loop or misconfiguration |
| `vlan N` | matches VLAN configuration | (n/a if `vlan_filtering` is off) | empty for a VLAN bridge | VLAN filtering disabled |
| `master` | the bridge name | (n/a) | empty | The port is not attached to the bridge |
| Aging | auto-aged after 300 seconds default | not aging (stuck permanent) | aging too aggressive | Run `ip link set br0 type bridge ageing_time` |

```bash
[host]# ethtool -S veth1
NIC statistics:
     peer_ifindex: 23
     rx_queue_0_xdp_packets: 0
     rx_queue_0_xdp_bytes: 0
     ...
```

veth has few statistics compared to a physical NIC, mostly `peer_ifindex` and the XDP counters when XDP is enabled.

**Failure scenarios:**

- Bridge FDB is empty for a port that has traffic: `vlan_filtering=1` is on but the port has no VLAN configured. Add `bridge vlan add dev <port> vid 1 pvid untagged`.
- MAC flap (the same MAC on two ports): there is a bridge loop. Run `mstpd` or disable one of the ports.
- veth throughput is capped at 5 Gbps even on a 25G NIC: the kernel veth path is the bottleneck. Switch to an OVS internal port (DPDK) or use the XDP path.

## 8.1.10 Capstone POE: veth versus OVS internal port for 1000 containers per host

> **Context:** A hyperscale node with 1000 containers, each container having one interface. Compare veth plus a Linux bridge against an OVS internal port.

**Claim to verify:** "veth plus a Linux bridge is simpler, so it scales better for 1000 containers."

**Predict:** False. The Linux bridge has a scaling wall at the MAC table, the FDB lookup time, and the lack of programmable policy.

**Observe (assumed):**

- 1000 veth pairs plus one Linux bridge: the FDB has 2000 entries or more (each veth contributes one MAC plus a multicast group). Bridge MAC lookup is O(log N) through a hash, roughly 1 to 2 microseconds. CPU usage of the OVS daemon is 0 (it is not used), and the kernel bridge takes 5 to 10 percent of CPU steady state.
- A container-to-container ping has 1.5 to 3 microseconds latency (a kernel bridge L2 lookup).
- Throughput per pair is 5 to 8 Gbps (CPU bound).
- **Scaling problem:** there is no per-flow policy. Every container is in the same broadcast domain, with broadcast storm risk. There is no per-container ACL. There is no native overlay.

OVS internal port plus `br-int`:

- 1000 OVS ports (internal or tap-bridged): the OVS classifier is three-tier (EMC plus SMC plus megaflow), with per-flow lookup of 100 to 500 nanoseconds after the cache warms.
- OpenFlow policy is enforced per port (ACL, QoS, conntrack).
- Tunnel is native (Geneve or VXLAN) for cross-host traffic.
- CPU usage of `ovs-vswitchd` is 30 to 50 percent on one core, and it can scale across multiple cores through multi-threading.
- Throughput is 25 Gbps or more with hardware offload.

**Explain:**

1. **A Linux bridge is suitable for fewer than 100 containers** that do not need policy. It is simple, easy to debug, and built into the kernel.
2. **OVS is mandatory for more than 100 containers** or for any deployment that needs ACL, QoS, or overlay. It provides programmability, hardware offload, and native tunnelling.
3. The "simplicity" of the Linux bridge is in fact **a limit**. Policy does not scale, so the engineer has to work around it through `iptables` or `nftables`, which is slow and not per-container.

**Falsification test:**

- Set up 1000 containers with a Linux bridge and measure throughput plus latency. Throughput is bottlenecked at 5 to 8 Gbps total per pair due to CPU.
- Set up 1000 containers with OVS internal ports. Per-pair throughput is 20 Gbps or more with offload. Latency is similar at about 2 microseconds for a cache hit.
- If the Linux bridge beats OVS on throughput, that is a counter-example. In practice, OVS wins for hyperscale.

**Where the claim fails:** "Simpler so it scales better" is a logical fallacy (simple at the interface level is not the same as scalable in the workload). The correct rule is **a Linux bridge is fine for lab or development, or for fewer than 100 containers without policy. OVS is mandatory for production hyperscale and multi-tenancy.**

> **Rule:** Fewer than 100 containers and no policy: a Linux bridge is fine. More than 100 containers, or policy, or overlay: OVS is mandatory.

### Guided Exercise 2: Linux bridge plus two namespaces and veth

**Outcomes:** Build a Linux bridge plus two namespaces with a veth pair, and verify L2 connectivity.

**Instructions:**

**1.** Setup:

```bash
[host]# ip netns add h1 && ip netns add h2
[host]# ip link add name br0 type bridge
[host]# ip link set br0 up

[host]# ip link add veth1 type veth peer name veth1-br
[host]# ip link add veth2 type veth peer name veth2-br

[host]# ip link set veth1 netns h1
[host]# ip link set veth2 netns h2
[host]# ip link set veth1-br master br0
[host]# ip link set veth2-br master br0
[host]# ip link set veth1-br up
[host]# ip link set veth2-br up

[host]# ip -n h1 addr add 10.0.0.1/24 dev veth1
[host]# ip -n h2 addr add 10.0.0.2/24 dev veth2
[host]# ip -n h1 link set veth1 up
[host]# ip -n h2 link set veth2 up
```

**2.** Verify:

```bash
[host]# ip netns exec h1 ping -c 3 10.0.0.2
[host]# bridge fdb show br br0
```

After the first ping, the FDB will show learned MACs for `veth1-br` and `veth2-br`.

**3.** Cleanup:

```bash
[host]# ip netns delete h1 h2
[host]# ip link delete br0
```

## Key takeaways

First, a Linux bridge is an L2 learning switch in the kernel, sufficient for simple container workloads. iproute2 (`ip link`, `bridge`) replaces the legacy `brctl`.

Second, the veth pair is the unique primitive of every SDN lab. It connects a namespace to a bridge with zero overhead in kernel context.

Third, macvlan (with four modes) and ipvlan (L2 and L3) are alternatives for scenarios with many containers. They scale better than veth above 100 containers per host. Trade-off: macvlan does not allow host-to-container communication.

Fourth, OVS replaces the Linux bridge when you need SDN features (OpenFlow, overlay, flow statistics). The Linux bridge is enough for non-SDN container workloads.

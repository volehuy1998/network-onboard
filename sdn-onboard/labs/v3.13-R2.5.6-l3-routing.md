A traditional Cisco engineer who has set up `interface Vlan10; ip address 10.0.10.1 255.255.255.0` on a layer-3 switch knows what an SVI (Switched Virtual Interface) is: a virtual interface tied to a VLAN that carries an IP address, allowing the switch to act as the default gateway for hosts in that VLAN. Set up two SVIs on two VLANs and the switch routes between them automatically using its hardware FIB plus ARP cache. The CLI is two lines per VLAN; the routing happens in the silicon.

OVS does NOT have an SVI primitive. OVS is, by design, a layer-2 switch with rich OpenFlow programmability. To get layer-3 routing behaviour through an OVS bridge, the operator has two patterns:

1. **Router-namespace approach.** Attach a Linux network namespace to
   the bridge with two veth pairs (one per VLAN), assign each veth an    IP in the
corresponding subnet, enable IP forwarding inside the namespace, and let the kernel's routing table route between the two subnets. OVS forwards L2 frames between the bridge ports; the namespace's kernel handles the L3 logic. This is the lab's primary focus.

2. **Raw OpenFlow flow approach.** Install OpenFlow flows that match
   on destination IP, rewrite the destination MAC to the next-hop MAC,    decrement the
TTL, and output to the egress port. The flows collectively implement the routing function entirely in OpenFlow. No router namespace, no kernel IP stack involvement; OVS plus the operator's flow rules are the router. This is more powerful but more error-prone (the operator must manually manage ARP, the reverse path, and the per-flow installation).

The lab demonstrates pattern 1 in full and outlines pattern 2 conceptually. After this lab the engineer will understand what OVS provides natively (L2 forwarding, VLAN-aware), what it does not provide natively (L3 routing, ARP responder, IP address assignment), and the two pragmatic ways to add L3 capability when needed.

## R2.5.6.1. Why OVS does not route natively

The OVS design philosophy explicitly separates layer-2 forwarding from layer-3 routing. Layer-2 forwarding is what a switch does; layer-3 routing is what a router does. OVS implements the switch role and delegates the router role to whatever upstream system needs it: a Linux kernel namespace (the lab's pattern 1), an external SDN controller via OpenFlow (the lab's pattern 2), or in production deployments OVN (Open Virtual Network), which is OVS's distributed logical-router subsystem (out of scope for this lab; covered in later parts of the curriculum).

The architectural reason for this separation is performance. Layer-2 forwarding is a fast-path operation: look up a destination MAC in the FDB, output to the matching port. Layer-3 routing involves the longest- prefix-match (LPM) algorithm against a routing table, the ARP cache lookup for the next-hop MAC, the IP header rewrite (TTL, checksum), and the L2 header rewrite. Doing all of that on every packet would slow OVS down measurably. Pushing L3 to userspace OpenFlow rules or to a kernel namespace's IP stack lets OVS focus on the L2 fast path and lets each operator pick the L3 mechanism that fits their deployment.

The corollary for the engineer learning OVS for the first time: OVS is NOT a drop-in replacement for a Cisco layer-3 switch. If the deployment needs SVI-style routing, the operator must add an L3 component on top, and pattern 1 (router-namespace) is the simplest.

The R2.5.6 source anchors live in three places. First, the kernel-side action executor in `datapath/actions.c` implements the `do_set_eth_addr`, `set_ipv4_addr`, `do_dec_ttl` actions that pattern-2 OpenFlow flows use to materialise routing in OVS. Second, `lib/odp-execute.c:odp_execute_set_action` (line 425) is the userspace mirror that runs the same action set when the flow is processed in userspace. Third, `lib/ofp-actions.c` contains the codecs (`decode_DEC_TTL`, `decode_SET_FIELD`, `encode_DEC_TTL`, `encode_SET_FIELD`) that translate the OpenFlow action wire format to OVS's internal action representation. The OVS source baseline is v2.17.9 (commit `0bea06d9957e3966d94c48873cd9afefba1c2677`).

## R2.5.6.2. Topology, single bridge with two VLANs and a router namespace

```ascii
   ns_a (host)                    ns_b (host)              ns_router (gateway)
   10.0.10.10/24, GW 10.0.10.1    10.0.20.10/24, GW 10.0.20.1
                                                            10.0.10.1/24 on veth_r10
                                                            10.0.20.1/24 on veth_r20
                                                            net.ipv4.ip_forward=1
       |                              |                          |             |
   [veth_a]                      [veth_b]                  [veth_r10]    [veth_r20]
       |                              |                          |             |
   ----+------------------------------+--------------------------+-------------+----
       |                              |                          |             |
   veth_a_br                    veth_b_br                  veth_r10_br    veth_r20_br
   tag=10                       tag=20                     tag=10         tag=20
       |                              |                          |             |
       +------------------+-----------+-----------+--------------+-------------+
                          |                       |
                          |     OVS bridge br0,   |
                          |     access ports for  |
                          |     vlan 10 and vlan  |
                          |     20                |
                          +-----------------------+
```

Four veth pairs into the bridge, three netns peers. ns_a is a host on VLAN 10 with default gateway pointing at the router namespace's vlan-10-side IP. ns_b is a host on VLAN 20 with default gateway pointing at the router namespace's vlan-20-side IP. ns_router is a Linux network namespace with two interfaces (one per VLAN), each carrying the gateway IP, and `net.ipv4.ip_forward=1` set so the kernel will route between the two interfaces.

The OVS-side configuration is uniform across the four veth pairs: each is an access port tagged in its respective VLAN. OVS itself has no notion of "this is a host port" versus "this is a router port"; both look the same from OVS's perspective. The router-ness is purely a property of what is on the other side of the veth, that is, the namespace's IP forwarding decision.

## R2.5.6.3. The router-namespace approach in detail

### ▶ Guided Exercise 1: Build the bridge with two VLANs and the router namespace

This exercise builds the topology and configures the router namespace for IP forwarding.

**Outcomes.** OVS bridge br0 with four access ports (two for hosts in vlan 10 and vlan 20, two for the router namespace). Three configured network namespaces. The router namespace has IP forwarding enabled and its kernel routing table shows two directly-connected routes (one per VLAN's subnet).

**Before You Begin.** OVS 2.17.9 daemons running. No bridges configured.

**Instructions.**

**1.** Build the bridge, four veth pairs, three namespaces, and attach each veth's bridge-side end as an access port on the appropriate VLAN.

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl --version | head -2
ovs-vsctl (Open vSwitch) 2.17.9
DB Schema 8.3.0
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-br br0
root@lab-openvswitch:~/network-onboard# ovs-vsctl set bridge br0 datapath_type=system
root@lab-openvswitch:~/network-onboard# ip netns add ns_a
root@lab-openvswitch:~/network-onboard# ip netns add ns_b
root@lab-openvswitch:~/network-onboard# ip netns add ns_router
root@lab-openvswitch:~/network-onboard# ip link add veth_a type veth peer name veth_a_br
root@lab-openvswitch:~/network-onboard# ip link add veth_b type veth peer name veth_b_br
root@lab-openvswitch:~/network-onboard# ip link add veth_r10 type veth peer name veth_r10_br
root@lab-openvswitch:~/network-onboard# ip link add veth_r20 type veth peer name veth_r20_br
root@lab-openvswitch:~/network-onboard# ip link set veth_a netns ns_a
root@lab-openvswitch:~/network-onboard# ip link set veth_b netns ns_b
root@lab-openvswitch:~/network-onboard# ip link set veth_r10 netns ns_router
root@lab-openvswitch:~/network-onboard# ip link set veth_r20 netns ns_router
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br0 veth_a_br tag=10
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br0 veth_b_br tag=20
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br0 veth_r10_br tag=10
root@lab-openvswitch:~/network-onboard# ovs-vsctl add-port br0 veth_r20_br tag=20
root@lab-openvswitch:~/network-onboard# ip link set veth_a_br up
root@lab-openvswitch:~/network-onboard# ip link set veth_b_br up
root@lab-openvswitch:~/network-onboard# ip link set veth_r10_br up
root@lab-openvswitch:~/network-onboard# ip link set veth_r20_br up
```

The crucial detail is the `tag=10` versus `tag=20` on the four add-port calls. Without those, all four veths would be in vlan 0 (the default access vlan) and the bridge would be a flat L2 device with no VLAN partitioning. The router-namespace approach depends on the VLAN separation: ns_a's traffic is tagged vlan 10 and is delivered only to the vlan-10-tagged ports, including the router's vlan-10 side; the router's IP routing then redirects the traffic to its vlan-20 side, where it re-enters the bridge tagged vlan 20 and reaches ns_b.

**2.** Configure ns_a as a vlan 10 host. Set its IP, default gateway, and bring up the loopback and veth.

```text
root@lab-openvswitch:~/network-onboard# ip -n ns_a link set lo up
root@lab-openvswitch:~/network-onboard# ip -n ns_a link set veth_a up
root@lab-openvswitch:~/network-onboard# ip -n ns_a addr add 10.0.10.10/24 dev veth_a
root@lab-openvswitch:~/network-onboard# ip -n ns_a route add default via 10.0.10.1
```

The default route via 10.0.10.1 is what tells the ns_a kernel "any packet whose destination IP is not in 10.0.10.0/24 should be sent to 10.0.10.1, the gateway." 10.0.10.1 will be the router namespace's vlan-10-side IP. The ns_a kernel, when it has a packet for 10.0.20.10, will look at its routing table, see that 10.0.20.0/24 is not a directly connected route, fall through to the default route, and ARP for 10.0.10.1 to find the next-hop MAC.

**3.** Configure ns_b as a vlan 20 host symmetrically.

```text
root@lab-openvswitch:~/network-onboard# ip -n ns_b link set lo up
root@lab-openvswitch:~/network-onboard# ip -n ns_b link set veth_b up
root@lab-openvswitch:~/network-onboard# ip -n ns_b addr add 10.0.20.10/24 dev veth_b
root@lab-openvswitch:~/network-onboard# ip -n ns_b route add default via 10.0.20.1
```

**4.** Configure the router namespace. Each interface gets its own IP (the gateway IP for the corresponding VLAN). IP forwarding is enabled via sysctl.

```text
root@lab-openvswitch:~/network-onboard# ip -n ns_router link set lo up
root@lab-openvswitch:~/network-onboard# ip -n ns_router link set veth_r10 up
root@lab-openvswitch:~/network-onboard# ip -n ns_router link set veth_r20 up
root@lab-openvswitch:~/network-onboard# ip -n ns_router addr add 10.0.10.1/24 dev veth_r10
root@lab-openvswitch:~/network-onboard# ip -n ns_router addr add 10.0.20.1/24 dev veth_r20
root@lab-openvswitch:~/network-onboard# ip netns exec ns_router sysctl -w net.ipv4.ip_forward=1
net.ipv4.ip_forward = 1
```

`net.ipv4.ip_forward = 1` is the sysctl confirmation. Without this setting, the namespace's kernel would drop forwarding-eligible packets on receive (the IP stack recognises a packet not destined for any local IP and would discard it because forwarding is disabled). With the setting set to 1, the stack consults the routing table for the destination and forwards.

**5.** Read the router namespace's routing table to confirm the directly-connected routes.

```text
root@lab-openvswitch:~/network-onboard# ip -n ns_router route show
10.0.10.0/24 dev veth_r10 proto kernel scope link src 10.0.10.1 
10.0.20.0/24 dev veth_r20 proto kernel scope link src 10.0.20.1 
```

Two routes, one per interface. `proto kernel` means the route was auto-installed by the kernel when the IP was assigned to the interface (this is standard Linux behaviour). `scope link` means the route is for directly-connected hosts (no further next-hop). `src 10.0.10.1` specifies the source address the kernel uses when generating its own packets to this subnet (e.g. ARP requests).

**6.** Read the router namespace's interface configuration with full detail.

**Finish.** Topology built. ns_a and ns_b are hosts in their respective VLANs; ns_router has two interfaces, one per VLAN, with IP forwarding enabled. The kernel routing table inside ns_router will route between the two subnets when packets arrive.

### ▶ Guided Exercise 2: Verify intra-VLAN connectivity to the router

This exercise verifies that ns_a can reach the router (10.0.10.1) and ns_b can reach its router IP (10.0.20.1). These pings are NOT crossing VLANs; they are testing the host-to-router L2 path inside each VLAN. This is the prerequisite for the cross-VLAN routed ping in exercise 3.

**Outcomes.** ns_a → 10.0.10.1 (the router's vlan-10 side) succeeds with TTL 64 (no router decrement, single hop). ns_b → 10.0.20.1 succeeds similarly.

**Before You Begin.** Exercise 1 has completed.

**Instructions.**

**1.** ns_a → 10.0.10.1.

### ▶ Guided Exercise 3: Verify cross-VLAN routed ping

This is the central empirical demonstration of L3 routing. ns_a sends ICMP to ns_b's IP (10.0.20.10). The packet traverses: ns_a's veth → veth_a_br (vlan 10 access port) → bridge → veth_r10_br (vlan 10 access) → ns_router's veth_r10. The router's kernel sees the packet, consults its routing table, finds 10.0.20.0/24 reachable via veth_r20, ARPs for ns_b's MAC if not cached, decrements TTL by 1, and emits the packet on veth_r20. From there: ns_router → veth_r20 → veth_r20_br (vlan 20 access) → bridge → veth_b_br (vlan 20 access) → ns_b's veth.

**Outcomes.** Cross-VLAN ping succeeds with TTL 63 (decremented by 1 hop, the router). 0% packet loss.

**Before You Begin.** Exercise 2 has completed.

**Instructions.**

**1.** ns_a → ns_b across the router.

```text
root@lab-openvswitch:~/network-onboard# ip netns exec ns_a ping -c 2 -W 2 10.0.20.10
PING 10.0.20.10 (10.0.20.10) 56(84) bytes of data.
64 bytes from 10.0.20.10: icmp_seq=1 ttl=63 time=0.425 ms
64 bytes from 10.0.20.10: icmp_seq=2 ttl=63 time=0.055 ms

--- 10.0.20.10 ping statistics ---
2 packets transmitted, 2 received, 0% packet loss, time 1024ms
rtt min/avg/max/mdev = 0.055/0.240/0.425/0.185 ms
```

The empirical confirmation has two key pieces. **First**, 0% packet loss: the cross-VLAN path works. **Second**, `ttl=63`: the reply arrived with TTL 63, exactly 1 less than the originator's TTL of 64. This is the router decrementing the TTL on the way back. The forward path also decrements TTL on the request, so the request arrived at ns_b with TTL 63; ns_b's reply with default TTL 64 traversed the router on the return trip and arrived at ns_a with TTL 63 again. The TTL count is the engineer's primary signal that an L3 hop has occurred. A pure L2 path preserves TTL; an L3 hop decrements by one.

**2.** Verify the symmetric path: ns_b can also reach its gateway.

```text
root@lab-openvswitch:~/network-onboard# ip netns exec ns_b ping -c 2 -W 2 10.0.20.1
PING 10.0.20.1 (10.0.20.1) 56(84) bytes of data.
64 bytes from 10.0.20.1: icmp_seq=1 ttl=64 time=0.036 ms
64 bytes from 10.0.20.1: icmp_seq=2 ttl=64 time=0.072 ms

--- 10.0.20.1 ping statistics ---
2 packets transmitted, 2 received, 0% packet loss, time 1022ms
rtt min/avg/max/mdev = 0.036/0.054/0.072/0.018 ms
```

ns_b's gateway (10.0.20.1) is reachable with TTL 64, the L2 hop signature.

**Finish.** Cross-VLAN routed ping demonstrated. TTL decrement empirically observed.

### ▶ Guided Exercise 4: Inspect the ARP caches

The cross-VLAN ping required ARP resolution at two points: ns_a needed to learn the router's vlan-10 MAC, and the router needed to learn ns_b's MAC. The ARP caches in each namespace document this.

**Outcomes.** ns_a's neighbour table has 10.0.10.1's MAC. The router's neighbour table has both ns_a's MAC and ns_b's MAC.

**Before You Begin.** Exercise 3 has completed.

**Instructions.**

**1.** ns_a's neighbour table.

```text
root@lab-openvswitch:~/network-onboard# ip -n ns_a neigh show
10.0.10.1 dev veth_a lladdr 66:6b:77:6f:cd:d9 REACHABLE
```

ns_a has learned the router's vlan-10 MAC `66:6b:77:6f:cd:d9` and mapped it to IP 10.0.10.1. The state `REACHABLE` means the kernel has recently confirmed the entry (within the past few seconds; default reachable timeout is 30 seconds).

**2.** The router's neighbour table.

```text
root@lab-openvswitch:~/network-onboard# ip -n ns_router neigh show
10.0.20.10 dev veth_r20 lladdr 6a:9b:f6:b2:d8:1d REACHABLE
10.0.10.10 dev veth_r10 lladdr 52:e9:ed:12:02:c4 DELAY
```

The router knows both ns_a (`52:e9:ed:12:02:c4` via veth_r10) and ns_b (`6a:9b:f6:b2:d8:1d` via veth_r20). The state `DELAY` for ns_a means the kernel just confirmed reachability and is in a brief post-reachability delay state; this is normal and would transition back to `STALE` after the kernel's reachability timer expires.

**Misconception to refute.** "ARP traffic does not work across VLANs; the router cannot ARP for ns_b because they are in different broadcast domains." Wrong. The router has its own veth_r20 in vlan 20, so it can ARP from veth_r20 inside that vlan. ARP is per-interface, not per-IP. The router's veth_r20 broadcasts an ARP request on vlan 20; ns_b in vlan 20 receives it and replies. The router does not ARP across VLANs; it has interfaces in BOTH VLANs and ARPs separately on each.

**Finish.** ARP caches confirmed.

### ▶ Guided Exercise 5: Read the kernel datapath megaflows for the routed traffic

The kernel datapath flow cache shows exactly which OVS actions materialised the L3 path. The pedagogical point is that **OVS itself did not route**; it only forwarded L2 frames between bridge ports. The L3 logic (route lookup, MAC rewrite, TTL decrement) happened inside ns_router's kernel.

**Outcomes.** The megaflow dump shows several flows for the IPv4 ICMP traffic, all with simple `actions:veth_X` (no rewrite, no TTL decrement, no L3 logic). The bridge is a pure L2 forwarder for the routed traffic.

## R2.5.6.4. The raw OpenFlow flow approach

The second flavour of L3 routing in OVS is to install OpenFlow flows that match on destination IP and rewrite the L2 header. This pattern keeps everything inside OVS without a router namespace; it is more powerful but requires manual ARP and reverse-flow management.

The conceptual flow rule for routing ns_a → ns_b looks like:

```openflow
table=0,priority=100,ip,in_port=veth_a_br,nw_dst=10.0.20.10,actions=mod_dl_src:<gw20_mac>,mod_dl_dst:<ns_b_mac>,dec_ttl,output:veth_b_br
```

Read field by field. `table=0,priority=100,ip,in_port=veth_a_br` is the match: any IPv4 packet (`ip`) ingressing on veth_a_br. The condition `nw_dst=10.0.20.10` further narrows the match to packets destined for ns_b's IP. The actions are: rewrite the source MAC to the synthetic gateway-20-side MAC `02:00:00:00:00:02` (so ns_b sees the packet as coming from its expected gateway), rewrite the destination MAC to ns_b's actual MAC (so ns_b's NIC accepts it), decrement TTL by 1 (canonical L3 hop indicator), and output to ns_b's bridge port.

The lab does not actually install these flows because doing so safely requires per-host MACs and per-flow installation that is fragile in a brief lab capture. The capture demonstrates the conceptual shape only.

## R2.5.6.5. Side by side: Cisco SVI, OVS router-namespace, OVS raw-flow

| Concern                     | Cisco IOS L3 switch                       | OVS router-namespace                                     | OVS raw OpenFlow flows                          |
|-----------------------------|-------------------------------------------|----------------------------------------------------------|-------------------------------------------------|
| Configure gateway IP        | `interface Vlan10; ip address 10.0.10.1 255.255.255.0` | `ip -n ns_router addr add 10.0.10.1/24 dev veth_r10`  | (synthetic MAC + ARP entry on host)             |
| Enable forwarding           | (default on for L3 switch)                | `sysctl -w net.ipv4.ip_forward=1`                        | (implicit; flows determine forwarding)          |
| Add a static route          | `ip route 10.0.30.0 255.255.255.0 10.0.20.99` | `ip -n ns_router route add 10.0.30.0/24 via 10.0.20.99` | `ovs-ofctl add-flow br0 'ip,nw_dst=10.0.30.0/24,actions=mod_dl_src:...,mod_dl_dst:...,dec_ttl,output:N'` |

The big-picture observation: OVS's two L3 patterns sit at very different points on the trade-off axis. The router-namespace pattern is simple and uses well-understood Linux primitives; it relies on the kernel for everything except the L2 wire transport. The raw-flow pattern is more powerful (controller-driven, programmatic) but requires the operator (or controller) to manage every detail (ARP, TTL, MAC rewrites, reverse paths). For a small static topology the namespace pattern wins on simplicity; for a large dynamic topology managed by a controller the raw-flow pattern wins on flexibility. OVN, the production answer, takes the raw-flow pattern and automates the flow installation through a higher-level data model.

## R2.5.6.6. Common misconceptions for the audience

**Misconception 1.** "OVS will route between VLANs if I just configure two VLANs." Wrong. OVS does L2 only. Configuring two VLANs creates two broadcast domains; without a router (namespace or flow-based), they remain isolated.

**Misconception 2.** "The router namespace is using extra OVS resources." Wrong. The router namespace is a Linux kernel construct, not an OVS construct. From OVS's perspective, the namespace's veth ends are just two more bridge ports. The namespace's IP forwarding is entirely inside the kernel; OVS does not consume CPU cycles on it.

**Misconception 3.** "Raw OpenFlow routing handles ARP automatically." Wrong. ARP is a separate protocol that ARPs into the bridge looking for the gateway IP's MAC. With raw OpenFlow flows and no router namespace, there is no actual entity at the gateway IP, so ARP would fail. The lab's workaround is `ip neigh add` to install a static ARP entry on each host. A real raw-flow deployment installs flow-based ARP responders (which match on `arp`, `arp_op=request,arp_tpa=<gw-ip>` and respond with a synthetic ARP reply via `move:NXM_OF_ETH_SRC[]->NXM_OF_ETH_DST[]` and friends); this is significantly more complex than the router-namespace pattern.

**Misconception 4.** "TTL decrement is optional." Functionally yes, but strongly discouraged. RFC 1812 §5.3.1 mandates that routers decrement TTL on every forwarded packet. Skipping the decrement creates a forwarding loop that traceroute cannot detect (every hop appears as the same source IP). The `dec_ttl` action exists exactly to make this correct; do not omit it from a raw-flow router rule.

**Misconception 5.** "I can implement a router with one OpenFlow flow." Wrong. A real router needs at minimum: a forward rule for each prefix, a reverse rule (or a more general reverse-direction rule), an ARP responder rule for the gateway IP, and possibly a fallback "unknown destination, send to controller" rule for unhandled prefixes. Plus per-VLAN scoping if the bridge handles multiple VLANs. A complete raw-flow router is dozens to hundreds of flows; OVN exists specifically to automate this complexity.

## R2.5.6.7. Production-readiness assessment

The trade-off cost of the router-namespace approach. Each routed packet traverses three kernel components: OVS userspace, the kernel datapath, and the namespace's IP stack. The first two are already on the OVS fast path; the namespace IP stack adds latency of roughly 5-20 microseconds per packet on commodity hardware. For most production workloads this is acceptable. For ultra-low-latency deployments (HFT, RDMA), the raw-flow pattern is faster because it keeps everything inside OVS userspace plus kernel datapath without the namespace round-trip.

The failure mode under heavy traffic. The router namespace can become a bottleneck if the kernel IP stack saturates. Symptoms include elevated forward-latency, dropped packets at the namespace's veth queue, and high softirq utilisation. Mitigation: run the router namespace's interface with `tc qdisc add dev veth_rN root noqueue` to avoid pfifo_fast queueing, or move to the raw-flow pattern.

The monitoring discipline. The router namespace's IP forwarding stats are visible via `ip -n ns_router -s link show` (per-interface tx/rx counters) and `ip -n ns_router -s route show table all` (per-route counters). The raw-flow stats are in `ovs-ofctl dump-flows br0` (per-flow `n_packets` and `n_bytes`). A monitoring agent should poll both and alert on rate spikes. For the namespace pattern, also monitor `cat /proc/sys/net/ipv4/netfilter/ip_conntrack_count` if conntrack is active (it is by default on most kernels), as the conntrack table can fill on heavy short-lived TCP traffic.

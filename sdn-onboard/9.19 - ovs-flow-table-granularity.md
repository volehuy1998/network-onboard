## 9.19.1 Cấu trúc một flow entry trong OVS

Tài liệu Lab 4 (Crichigno et al., USC WASTC 2021) mô tả flow entry ngắn gọn: *"A flow entry consists of header fields, counters, and actions."* OVS thực hiện đầy đủ định nghĩa OpenFlow 1.0 này và mở rộng với các trường bổ sung. Mỗi flow entry trên bridge OVS bao gồm:

| Thành phần | Vai trò | Ví dụ |
|-----------|---------|-------|
| **Match fields** | Điều kiện một packet phải thỏa để khớp entry | `in_port=1,dl_type=0x0800,nw_dst=10.0.0.2` |
| **Priority** | Số nguyên 0-65535, entry cao hơn ưu tiên | `priority=100` |
| **Counters** | `n_packets` + `n_bytes` — thống kê mỗi lần match | Tự tăng |
| **Actions** | Danh sách hành động thực thi khi match | `output:2`, `drop`, `mod_dl_dst:aa:...` |
| **Timeouts** | `idle_timeout` (giây im lặng → xóa), `hard_timeout` (tuổi thọ tuyệt đối) | `idle_timeout=60,hard_timeout=300` |
| **Cookie** | Identifier 64-bit do controller/user đặt để track nhóm flow | `cookie=0x1001` |
| **Table** | Bảng chứa entry (OpenFlow 1.1+) | `table=0` |

Khi `ovs-ofctl add-flow br0 "priority=100,ip,nw_dst=10.0.0.2,actions=output:2"` được chạy, OVS chèn entry vào `table=0` với `dl_type=0x0800` (được suy ra từ từ khóa `ip`), `nw_dst=10.0.0.2`, `priority=100`, và action `output:2`. Entry tồn tại đến khi bị xóa thủ công hoặc timeout (nếu có).

> **Key Topic:** Flow entry là đơn vị forwarding nhỏ nhất trên OVS. Khác với router Cisco dùng routing table + ARP cache riêng biệt, OVS đóng gói toàn bộ decision logic vào một entry — match (điều kiện) + action (hành động) là đủ. Đây là lý do OVS có thể làm bridge, router, firewall, load balancer, tunnel endpoint trong cùng một bridge, chỉ khác nhau ở flow entries.

## 9.19.2 Topology và bốn cấp độ matching

Lab 4 dùng topology tối giản để focus vào matching behavior:

```
            ┌───────────────────────┐
            │      br-s1            │
            │  (OVS bridge)         │
            │                       │
            │ s1-eth1      s1-eth2  │
            └────┬────────────┬─────┘
                 │            │
                 │            │
           ┌─────┴────┐ ┌─────┴────┐
           │    h1    │ │    h2    │
           │10.0.0.1/8│ │10.0.0.2/8│
           └──────────┘ └──────────┘
```

Toàn bộ cả hai host nằm chung subnet `10.0.0.0/8` — không cần routing. Task là buộc OVS forward packet giữa hai port bằng bốn cách khác nhau, từ granular thấp đến cao. Đây là bài tập quan trọng để thấy OVS không phân biệt "loại switch" (L2/L3/L4) — nó chỉ phân biệt mức độ chi tiết của match field.

### Cấp độ 1 — L1 port-based forwarding

Đơn giản nhất: mọi thứ từ port 1 gửi ra port 2, mọi thứ từ port 2 gửi ra port 1. OVS không inspect bất kỳ trường nào ngoài `in_port`.

```bash
[host]# ovs-ofctl add-flow br-s1 "priority=100,in_port=1,actions=output:2"
[host]# ovs-ofctl add-flow br-s1 "priority=100,in_port=2,actions=output:1"
```

Kết quả: hành vi giống hub — tất cả traffic được cross-over giữa hai port. Ưu điểm: CPU tối thiểu, throughput tối đa. Nhược điểm: broadcast storm nếu có hơn 2 port, không lọc được theo bất kỳ tiêu chí nào khác.

### Cấp độ 2 — L2 MAC-based forwarding

Match theo MAC address cho granularity cao hơn:

```bash
[host]# ovs-ofctl add-flow br-s1 \
  "priority=100,dl_dst=02:00:00:00:00:02,actions=output:2"
[host]# ovs-ofctl add-flow br-s1 \
  "priority=100,dl_dst=02:00:00:00:00:01,actions=output:1"
```

Giờ OVS forward dựa trên destination MAC. Đây là hành vi của một L2 switch tĩnh (static MAC table). OVS không tự học MAC source trong trường hợp này — muốn học động phải dùng action `NORMAL` (học tự động trong MAC table implicit) hoặc `learn()` action (tường minh, tạo flow entry mới khi thấy packet mới).

> **Hiểu sai:** "OVS tự động học MAC như switch Cisco mặc định."
>
> Thực tế: OVS chỉ học MAC khi action `NORMAL` được gọi — action này kích hoạt learning logic built-in. Nếu flow table chỉ có match/action tường minh (như ví dụ trên), OVS không học bất cứ gì. Nhiều kỹ sư nghĩ OVS "không hoạt động" khi flow table trống, thực ra OVS hoạt động đúng với flow table đã cho — chỉ là flow table không yêu cầu học.

### Cấp độ 3 — L3 IP-based forwarding

Match theo IP destination:

```bash
[host]# ovs-ofctl add-flow br-s1 \
  "priority=100,ip,nw_dst=10.0.0.2,actions=output:2"
[host]# ovs-ofctl add-flow br-s1 \
  "priority=100,ip,nw_dst=10.0.0.1,actions=output:1"
```

Từ khóa `ip` là shortcut cho `dl_type=0x0800` (Ethernet type IPv4). Bắt buộc phải có vì `nw_dst` chỉ ý nghĩa khi frame là IP. Với flow table trên, OVS forward IPv4 packet theo destination IP. ARP packet (`dl_type=0x0806`) sẽ không khớp → không được forward → rớt. Trong production, phải có flow riêng cho ARP hoặc action `NORMAL` làm fallback.

### Cấp độ 4 — L4 TCP/UDP port-based forwarding

Match sâu vào L4 header:

```bash
# Giả sử h2 chạy web server python trên port 80
[host]# ovs-ofctl add-flow br-s1 \
  "priority=100,tcp,nw_dst=10.0.0.2,tp_dst=80,actions=output:2"
[host]# ovs-ofctl add-flow br-s1 \
  "priority=100,tcp,nw_dst=10.0.0.1,tp_dst=80,actions=output:1"
```

Từ khóa `tcp` = `dl_type=0x0800,nw_proto=6`. `tp_dst=80` khớp HTTP. Với flow table này, chỉ packet TCP đến port 80 được forward — ICMP, SSH, UDP đều rớt. Đây là cơ sở để làm L4 firewall hoặc LB tầng 4 chỉ bằng flow table.

## 9.19.3 Cơ chế priority resolution

OVS lưu trữ flow entries trong classifier (chi tiết Part 9.15 — tuple space search). Khi một packet đi vào, classifier trả về entry có **priority cao nhất** trong tập entries khớp. Nếu nhiều entries cùng priority cao nhất cùng khớp, tie-breaker là *"first match wins"* — tức entry được chèn trước (xem `ovs-ofctl(8)` mục `MATCH PRIORITY`).

Ví dụ flow table sau:

```bash
# Rule 1: cho phép TCP/80 đến h2
[host]# ovs-ofctl add-flow br-s1 \
  "priority=200,tcp,nw_dst=10.0.0.2,tp_dst=80,actions=output:2"

# Rule 2: chặn mọi TCP đến h2
[host]# ovs-ofctl add-flow br-s1 \
  "priority=100,tcp,nw_dst=10.0.0.2,actions=drop"

# Rule 3: forward mọi IP đến h2
[host]# ovs-ofctl add-flow br-s1 \
  "priority=50,ip,nw_dst=10.0.0.2,actions=output:2"
```

Packet TCP/80 đến h2: khớp cả Rule 1, 2, 3 — Rule 1 có priority 200 cao nhất → `output:2` (cho phép).

Packet TCP/22 đến h2: không khớp Rule 1 (port sai), khớp Rule 2 + 3 — Rule 2 có priority 100 cao hơn → `drop` (chặn SSH).

Packet ICMP đến h2: không khớp Rule 1 (không phải TCP), không khớp Rule 2 (không phải TCP), khớp Rule 3 — `output:2` (cho phép ping).

Đây là pattern kinh điển của ACL: rule specific ưu tiên cao, catch-all ưu tiên thấp. Toàn bộ logic fit trong flow table, không cần module firewall riêng.

> **Key Topic:** Priority resolution cho phép gom rule firewall + routing + load balancing vào cùng một flow table. Việc OVS có thể làm tất cả trong một bridge thay vì chain nhiều bridge/iptables/route table là lý do kiến trúc cloud data plane chọn OVS làm backbone.

### ▶ Guided Exercise 1: Chứng minh OVS không tự học MAC khi thiếu action NORMAL

Bài thực hành này chứng minh — bằng phản chứng — rằng OVS learning là opt-in qua action, không phải tự động. Người đọc thường tin OVS "như switch bình thường" sau khi đọc doc qua loa, bài này phá vỡ giả định đó bằng bằng chứng thực nghiệm.

**Outcomes:**
- Quan sát MAC table rỗng sau khi cài flow table không có `NORMAL` action và có traffic đi qua
- Xác minh cùng topology nhưng với `NORMAL` action, MAC table tự populate

**Before You Begin:**
OVS service đang chạy. Bridge `br-s1` + 2 namespace `h1` (10.0.0.1/8) + `h2` (10.0.0.2/8) theo topology Lab 4 (setup tương tự Part 9.18 §9.18.4 Guided Exercise).

**Instructions:**

**1.** Cài flow table tường minh (không có NORMAL):

```bash
[host]# ovs-ofctl del-flows br-s1
[host]# ovs-ofctl add-flow br-s1 "priority=100,in_port=1,actions=output:2"
[host]# ovs-ofctl add-flow br-s1 "priority=100,in_port=2,actions=output:1"
```

**2.** Tạo traffic:

```bash
[host]# ip netns exec ns-h1 ping -c 5 10.0.0.2
```

**3. Predict-Observe-Explain — MAC learning**:

Giả thuyết ngược: *"OVS tự học MAC như switch L2 truyền thống. Sau khi có traffic, `ovs-appctl fdb/show br-s1` phải liệt kê MAC của h1 và h2 trên port tương ứng."*

Quan sát — dump MAC table:

```bash
[host]# ovs-appctl fdb/show br-s1
 port  VLAN  MAC                Age
```

Output rỗng (không có entry nào). Giả thuyết ngược bị bác bỏ.

Giải thích: OVS không học MAC vì flow table không gọi action `NORMAL`. Action `output:N` là forward trực tiếp, bypass logic MAC learning. Bridge forward đúng (ping thành công) nhưng MAC table trống — bằng chứng hai việc (forwarding và learning) độc lập trong OVS.

**4.** Reset và thử với `NORMAL`:

```bash
[host]# ovs-ofctl del-flows br-s1
[host]# ovs-ofctl add-flow br-s1 "priority=100,actions=NORMAL"
[host]# ip netns exec ns-h1 ping -c 5 10.0.0.2
[host]# ovs-appctl fdb/show br-s1
 port  VLAN  MAC                Age
    1     0  02:00:00:00:00:01    2
    2     0  02:00:00:00:00:02    2
```

Bây giờ MAC table có 2 entry — `NORMAL` kích hoạt learning logic. Hai flow table hành vi forward giống nhau nhưng khác biệt về learning là do chính `NORMAL` chứ không phải traffic.

**Finish:**
```bash
[host]# ovs-ofctl del-flows br-s1
```

## 9.19.4 Idle_timeout, hard_timeout, và cookie

Ba trường quản lý lifecycle của flow entry:

**`idle_timeout`** (0 = không giới hạn, default): số giây sau packet cuối khớp mà entry bị xóa. Dùng khi muốn garbage collect flow không hoạt động.

```bash
[host]# ovs-ofctl add-flow br-s1 \
  "priority=100,tcp,nw_dst=10.0.0.2,tp_dst=80,idle_timeout=60,actions=output:2"
```

Nếu 60 giây không có packet TCP/80 nào, entry tự xóa. Controller OpenFlow dùng mẫu này nhiều: cài flow specific khi thấy packet đầu tiên, để timeout tự clean up. Cơ chế tương tự ovn-controller khi xử lý conntrack flow.

**`hard_timeout`**: tuổi thọ tuyệt đối bất kể có traffic. Dùng cho flow có lifetime cố định (ví dụ rule authentication 1 giờ).

**`cookie`**: identifier 64-bit do controller/user gán để group flow. Không ảnh hưởng matching hay action — chỉ để sau này query/delete theo group:

```bash
# Cài một loạt flow với cookie=0x1001 (ví dụ: tenant A)
[host]# ovs-ofctl add-flow br-s1 \
  "cookie=0x1001,priority=100,ip,nw_src=10.1.0.0/24,actions=output:2"
[host]# ovs-ofctl add-flow br-s1 \
  "cookie=0x1001,priority=100,ip,nw_src=10.1.1.0/24,actions=output:2"

# Xóa toàn bộ flow của tenant A bằng cookie mask
[host]# ovs-ofctl del-flows br-s1 "cookie=0x1001/-1"
```

OVN sử dụng cookie để đánh dấu Logical Flow → OpenFlow mapping. Khi bạn `ovs-ofctl dump-flows br-int` trên node OVN, cookie value cho biết flow này thuộc Logical Flow nào (xem Part 13.7). Cookie là cơ chế chính để `ovs-ofctl` inspect OVN runtime state.

## Điểm cốt lõi cần nhớ

Thứ nhất, flow entry có cấu trúc đồng nhất — match + counters + actions + metadata (priority, timeout, cookie, table). Mọi loại forwarding (L1→L4) đều dùng chung cấu trúc này, chỉ khác ở match field được chọn.

Thứ hai, OVS không auto-learn MAC trừ khi flow action chỉ định `NORMAL`. Đây là điểm phân biệt OVS với switch truyền thống. Learning là tính năng opt-in, không phải mặc định.

Thứ ba, priority resolution cho phép gom firewall + routing + load balancing vào cùng flow table. Rule specific priority cao, catch-all priority thấp — pattern ACL cơ bản.

Thứ tư, `idle_timeout` + `hard_timeout` + `cookie` là bộ công cụ quản lý lifecycle của flow entry. Cookie đặc biệt quan trọng khi debug OVN — nó link OpenFlow runtime với Logical Flow abstraction.

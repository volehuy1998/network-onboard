## §3.5.1 OFPT_HELLO, the OpenFlow version handshake

`OFPT_HELLO` (the value `type = 0`) is the first message that the controller and switch send to each other immediately after the TCP channel (default port 6653) or the TLS channel is established. Its sole purpose is to **negotiate the OpenFlow version** that will be used for the rest of the session. The OpenFlow protocol catalogue starts officially from a successful HELLO pair; if HELLO fails, every later message (FEATURES, FLOW_MOD, PACKET_IN, and so on) cannot exist on this channel.

**Concept.** OFPT_HELLO is a **symmetric message**: both the controller and the switch send it, with no master and slave role at this stage. Each side announces to the other the highest OpenFlow version it supports through the `header.version` field, and (from OF 1.3.1 onward) attaches a **Hello Elements TLV** of type `OFPHET_VERSIONBITMAP` listing all supported versions. The receiver computes the intersection of the two version sets and selects the highest-numbered version as the official version.

**History and background.** OFPT_HELLO appeared from OpenFlow 1.0 (Stanford, December 2009) as a mandatory message in the four-state lifecycle HELLO, FEATURES, STEADY, CLOSE. Version 1.0 carried only a single `version` byte, which led to a negotiation deadlock when the controller and the switch supported non-contiguous version sets (for example, the controller supports 1.0 and 1.3, and the switch supports only 1.1). ONF fixed this in OpenFlow 1.3.1 (September 2012) by adding Hello Elements and `versionbitmap`, allowing a bitmap of multiple versions to be advertised at once. The negotiation rule is unchanged through OpenFlow 1.5.1 (Section 7.5.1 of the current text dated 2015-03-26).

**Position in the architecture.** On the timeline of an OpenFlow session, HELLO sits immediately after the TCP or TLS handshake and before any business message. In the state-machine diagram (Part 3.3 §3.3.X "OFP Connection State Machine"), HELLO occupies the entire `NEGOTIATE` state. In the OVS architecture, HELLO is generated in `lib/rconn.c` (the reliable connection layer above TCP) and dispatched to `lib/ofp-msgs.c` for decode and encode. On the OVN side, the `ovn-controller` opens an OpenFlow channel to `br-int` over a `unix:` domain socket, and the HELLO process is identical even though it runs locally instead of over TCP.

**Role.** The role of HELLO is the **gating function**: without a valid HELLO there are no flow rules, no `packet_in`, no statistics. HELLO carries no business data; its job is to create a **shared invariant** that both sides will decode the next byte using the same grammar. Without HELLO, the OF channel falls into the state "TCP exists but no protocol is spoken"; `tcpdump` may capture packets but the switch does not understand them or misunderstands them, causing buffer overflows and the channel to close.

**Why it exists.** OpenFlow is a multi-version protocol that has expanded fast: from 1.0 (2009) to 1.5 (2015), the structure of match, action, instruction, and multi-table changed substantially. A production controller usually has to work with both old switch firmware (OF 1.0 or 1.3) and modern switch ASICs (OF 1.4 or 1.5). Without a clear negotiation step, the controller has to guess the version, which leads to (a) the parser reading the wrong byte offset, and (b) actions that the switch does not support being sent and triggering a late `OFPT_ERROR`. HELLO exists to turn an implicit guess into an explicit agreement that is stamped (in `header.version`) on every later message, so when there is a mismatch it surfaces early.

**Problems it solves.** HELLO solves four concrete problems: (1) **version mismatch**, if the two sides have no common version, the connection ends cleanly with `OFPET_HELLO_FAILED` instead of hanging forever; (2) **multi-version overlap**, the bitmap allows the best version to be picked when the supported sets intersect at more than one point; (3) **race condition**, because HELLO is symmetric, neither side needs to know which one sends first as long as the other side's HELLO eventually arrives within the `inactivity_probe` window; (4) **rolling upgrade**, when the controller or the switch is upgraded, HELLO automatically negotiates down to the common level without out-of-band coordination.

**Importance.** OFPT_HELLO is in the foundation tier because it is a **single point of failure** for the whole OpenFlow channel. In typical production incidents, roughly 30 percent of reports of "the controller lost connection to the switch" are actually HELLO_FAILED due to a version-misaligned upgrade or a mismatched `protocols=` configuration. A proficient SDN engineer must be able to read a HELLO hex dump on the wire, reverse-engineer the negotiated version, and distinguish "TCP is broken" from "TCP is up but OF died at HELLO". This is a basic skill before debugging `flow_mod` or `packet_in`.

**How it works.** Every OpenFlow message has an 8-byte common header; HELLO (OF 1.0) usually has no body:

```
Offset  Field           Length  Example (OF 1.0 HELLO, xid=1)
------  --------------  ------  ----------------------------
0       version         1 byte  0x01  (OpenFlow 1.0)
1       type            1 byte  0x00  (OFPT_HELLO)
2-3     length          2 byte  0x00 0x08  (8 bytes, header only)
4-7     xid             4 byte  0x00 0x00 0x00 0x01

Hex bytes:  01 00 00 08 00 00 00 01
```

From OF 1.3.1 onward, the body may contain one or more **Hello Elements** (`struct ofp_hello_elem_header`, defined in `include/openflow/openflow-1.3.h`). The most important element is `OFPHET_VERSIONBITMAP` (type = 1):

```
Hello Element (type=1, length>=8):
  +-----------+-----------+-----------------+
  |  type=1   | length    | bitmaps[]       |
  | (2 byte)  | (2 byte)  | (length-4 byte) |
  +-----------+-----------+-----------------+

bitmaps[0] (uint32) bit i = supports OF 1.(i-1):
  bit 1 = 0x02 = OF 1.0
  bit 2 = 0x04 = OF 1.1
  bit 3 = 0x08 = OF 1.2
  bit 4 = 0x10 = OF 1.3
  bit 5 = 0x20 = OF 1.4
  bit 6 = 0x40 = OF 1.5
```

Negotiation procedure: the receiver computes `negotiated = max(common_versions)` with `common_versions = my_bitmap & peer_bitmap`. If the intersection is empty, it sends an `OFPT_ERROR` with `type = OFPET_HELLO_FAILED (0)` and `code = OFPHFC_INCOMPATIBLE (0)` and closes the connection. If the intersection is non-empty, every later message must use `header.version = negotiated`; otherwise the peer returns `OFPET_BAD_REQUEST / OFPBRC_BAD_VERSION`.

**How an operator masters this.** An OpenFlow operator should know five operations on HELLO by heart. First, **read the hex header by hand**: looking at the first 8 bytes is enough to read version, type, length, and xid without Wireshark. Second, **read Hello Elements**: if `length > 8`, parse the next 4 bytes (the element's type and length); if type is 1, it is a versionbitmap. Third, **check the negotiated version**: run `ovs-ofctl -O OpenFlow15 show br0` to see which version the bridge is speaking with the controller. Fourth, **compare client side and server side**: capture `tcpdump` on both sides at the same time and confirm the two HELLOs are symmetric. Fifth, **set `protocols=` explicitly**: avoid OVS auto-negotiation in production because the default behaviour differs between OVS 2.x and 3.x.

**Classification.** OFPT_HELLO belongs to the **Symmetric messages** group in the four-group OpenFlow message taxonomy (Symmetric, Switch-to-Controller, Controller-to-Switch, Multipart). The other symmetric messages alongside HELLO are `OFPT_ECHO_REQUEST/REPLY` (keepalive) and `OFPT_ERROR` (a two-way error report). Within the symmetric group, HELLO is the only one that runs only once at the start of a session; ECHO runs periodically; ERROR runs on demand. The HELLO body type splits into two variants: empty (OF 1.0 to 1.2) and TLV-bearing (OF 1.3.1 and later, with versionbitmap).

**Usage workflow.** The typical workflow when onboarding a new switch with a controller takes seven steps. First, configure the bridge and the controller: `ovs-vsctl set-controller br0 tcp:10.0.0.1:6653` and `ovs-vsctl set Bridge br0 protocols=OpenFlow13,OpenFlow15`. Second, start the channel: OVS `rconn` opens the TCP socket, and once TCP is `ESTABLISHED`, it sends HELLO immediately (with bitmap = 1.3 plus 1.5). Third, the controller side (Ryu, ONOS, Faucet, OVN) receives the switch's HELLO and sends its own HELLO with the bitmap the controller supports. Fourth, the bitmap intersection is computed, the version is locked in, and it is stored in the connection state on both sides. Fifth, the controller sends `OFPT_FEATURES_REQUEST` with `header.version = negotiated`. Sixth, every subsequent message must use the same version; otherwise the peer returns `OFPBRC_BAD_VERSION`. Seventh, to change the version, the channel must be fully closed and reopened; there is no re-negotiate mechanism within a session.

**When something goes wrong.** Four common failure modes are tied to HELLO. First, **HELLO_FAILED**: the `ovs-vswitchd.log` shows `connection failed (Protocol not supported)` or `version negotiation failed`; the cause is usually a misaligned `protocols=` configuration. Second, **HELLO timeout**: TCP is `ESTABLISHED` but the peer's HELLO is not received within `inactivity_probe` (default 5 seconds); the cause is usually a firewall that blocks by payload, or a deadlocked controller that has not yet sent. Third, **silent version downgrade**: both sides support 1.5 but OVS is configured with `protocols=OpenFlow13` only, so the controller must use 1.3 and lose 1.5 features (egress table, packet_type). Fourth, **MTU truncation**: HELLO plus bitmap exceeds the path MTU (very rare for a short HELLO but possible with a versionbitmap that has many vendor elements). To distinguish these modes: capture `tcpdump` on port 6653 and count HELLOs; zero HELLO means TCP is broken, one HELLO means one side hangs, and two HELLOs followed by an ERROR means HELLO_FAILED.

**Tightly related to.** OFPT_HELLO is tightly coupled with five entities: (a) `OFPT_FEATURES_REQUEST/REPLY` (Type 5 and 6) is the message that immediately follows HELLO in the state machine; (b) `OFPT_ERROR` with `OFPET_HELLO_FAILED` (type=0) is the negotiation-error response; (c) `OFPT_ECHO_REQUEST/REPLY` (Type 2 and 3) is the keepalive after a successful HELLO and uses the same `header.version`; (d) `Bridge.protocols` in OVSDB is the OVS-side configuration that decides the bitmap OVS sends; (e) the `rconn` connection manager (`lib/rconn.c`) is responsible for reconnection and for resending HELLO after every TCP reset.

**Version differences.**

| Version | Negotiation method | Detail |
|---------|--------------------|--------|
| OF 1.0 (December 2009) | Single byte `version=0x01`, empty body | No multi-version mechanism |
| OF 1.1 (February 2011) | Single byte `version=0x02` | Body is still empty |
| OF 1.2 (December 2011) | Single byte `version=0x03` | OXM was added, but HELLO is unchanged |
| OF 1.3.0 (June 2012) | Single byte `version=0x04` | Body is still empty |
| OF 1.3.1 (September 2012) | `version=0x04` plus Hello Elements | Adds `OFPHET_VERSIONBITMAP` (Section 7.5.1) |
| OF 1.4 (October 2013) | `version=0x05` plus bitmap | Hello Elements structure is unchanged |
| OF 1.5.1 (March 2015) | `version=0x06` plus bitmap | Negotiation rule unchanged from 1.3.1 |

OVS supports OF 1.0 through 1.5 fully from version 2.6 onward (`lib/ofp-version-opt.c`). Before OVS 2.6, some versions (1.4, 1.5) were experimental and required careful handling of the `--enable-Werror` flag.

**Source code references.** In the OVS source tree (baseline `v2.17.9`, branch `branch-2.17`), HELLO is touched in six places. First, `lib/ofp-msgs.c` contains `ofputil_decode_hello()` and `ofputil_encode_hello()`, which encode and decode HELLO together with the Hello Elements bitmap. Second, `lib/ofp-version-opt.c` converts between the wire version (`0x01` to `0x06`) and the internal enum `OFP10_VERSION` to `OFP15_VERSION`. Third, `lib/rconn.c`, with `rconn_run()` and `rconn_recv()`, manages the reliable connection state machine; after every `rconn_disconnect`, a new HELLO is sent automatically on the next reconnect. Fourth, `lib/vconn.c`, through `vconn_open()` and then `vconn_send_hello()`, is the entry point that sends the initial HELLO. Fifth, `lib/ofp-errors.h` defines `OFPET_HELLO_FAILED (0)`, `OFPHFC_INCOMPATIBLE (0)`, and `OFPHFC_EPERM (1)`. Sixth, `include/openflow/openflow-common.h` defines the 8-byte `ofp_header` struct, and `include/openflow/openflow-1.3.h` defines the `ofp_hello_elem_header` struct and `OFPHET_VERSIONBITMAP`. Original specifications: ONF OpenFlow Switch Specification 1.5.1, Section 7.5.1 "Hello"; ONF OpenFlow Switch Specification 1.3.5, Section 6.3.1 "Hello Message".

**Real-world incident.** A production OVN incident after upgrading `ovn-controller` from 22.03 to 24.03: every chassis reported `connection dropped` every 5 seconds. `tcpdump` showed that the TCP handshake completed but `ovs-vswitchd` did not send HELLO within the 5-second `inactivity_probe`. Root cause: on the older OVS build running on the same host, `protocols=OpenFlow10` was carried over from the migration, while the new `ovn-controller` speaks only OpenFlow15. The two sides had no bitmap intersection; OVS sent HELLO with a bitmap containing only bit 0x02 (OF 1.0), `ovn-controller` replied with a bitmap containing only bit 0x40 (OF 1.5), negotiation failed, and `rconn` closed and reopened the channel, failing again repeatedly. Fix: `ovs-vsctl set Bridge br-int protocols=OpenFlow15` to align with the controller's expectation. After applying it, HELLO negotiated to 1.5 successfully, and the connection stayed stable for the 30-minute observation window. Lesson: every OVN/OVS upgrade must include an audit of all `Bridge.protocols` settings to avoid bitmap drift.

**Common failures and diagnostic signals.** Six typical errors and their identifying signals. First, `version negotiation failed` in `ovs-vswitchd.log` is the HELLO_FAILED signal caused by an empty bitmap intersection; fix by aligning `Bridge.protocols` on both sides. Second, an `OFPT_ERROR (type=0, code=0)` captured immediately after HELLO has the same cause as the previous error; decode with `ovs-ofctl ofp-parse < error.bin`. Third, a successful TCP connect but `ovs-ofctl show` reports `connection failed` is usually a HELLO timeout because the controller was too slow to reply; check `inactivity_probe` (default 5000 ms, can be increased through `Controller.inactivity_probe`). Fourth, `bad version (got 0x05, expected 0x04)` in the controller log: the OVS side sent a post-HELLO message with a version different from the negotiated one; usually a parser bug or a race during incomplete reconnect. Fifth, the switch silently uses the lowest version when `protocols=OpenFlow10,OpenFlow13` but the controller supports only 1.0; the negotiated version drops to 1.0 and every OF 1.3 feature (multi-table, OXM) becomes unavailable; the signal is that `ovs-ofctl show` prints `OFPT_FEATURES_REPLY (OF1.0)` instead of `(OF1.3)`. Sixth, an infinite HELLO loop: after every disconnect, `rconn` reopens and sends a new HELLO; if it keeps failing, the log spams every `max_backoff` (default 8 seconds); a temporary fix is to increase `max_backoff` to reduce noise during investigation.

**Comparison with other systems.** OFPT_HELLO has many parallels with, and clear differences from, other negotiation mechanisms. First, the NETCONF `<hello>` (RFC 6241, Section 8.1) is also a symmetric message sent immediately after the SSH or TLS channel opens, and it also lists multi-version "capabilities"; the difference is that NETCONF uses a nested XML `<capability>` element while OF uses a binary bitmap. NETCONF has a "subset agreement" mechanism based on capability URIs; OF locks in a single integer version. Second, the gRPC HTTP/2 SETTINGS frame is not protocol-version negotiation but runtime parameters (window size, max streams); gRPC negotiates "version" through the service name (`/grpc.health.v1.Health`). Third, TLS `ClientHello` and `ServerHello` use the same symmetric exchange structure and produce the same "lock in a version" result; TLS negotiates cipher suites and extensions at the same time, while OF only negotiates the version. From a design viewpoint, OFPT_HELLO is simpler than the two-stage TLS hello. Fourth, the BGP `OPEN` message differs in symmetry: BGP OPEN is not symmetric and requires both sides to send and then confirm through a separate `KEEPALIVE`; OF folds version negotiation and initial state into a single message. Fifth, NETCONF Call Home, IPMI Get Channel Authentication, and SSH Protocol Version Exchange all follow the pattern "exchange version before speaking business", much like OFPT_HELLO, showing that this is a universal pattern in protocol design rather than something unique to OpenFlow.

## §3.5.2 OFPT_FEATURES_REQUEST/REPLY, switch identity and OpenFlow capabilities

`OFPT_FEATURES_REQUEST` (type=5) and `OFPT_FEATURES_REPLY` (type=6) are the second message pair in the lifecycle of an OpenFlow session, sent immediately after a successful HELLO version negotiation. The controller uses FEATURES_REQUEST to ask the switch for three foundational pieces of information: **identity** (datapath_id), **control capabilities** (the capabilities bitmap and `n_tables`), and (for OF 1.0 to 1.2 only) the **list of physical ports**. Every decision to install a rule, build a pipeline, or attach a port depends on the controller receiving a valid FEATURES_REPLY; without it, the controller does not know which switch it is talking to, nor how many tables the switch supports for compiling a multi-table pipeline.

**Concept.** FEATURES_REQUEST is an **empty message**, just the 8-byte standard OpenFlow header; FEATURES_REPLY contains a fixed body (`struct ofp_switch_features` in `include/openflow/openflow-1.3.h`) with three identifier fields and a capabilities bitmap. Depending on version, it also carries a port list (OF 1.0 to 1.2) or a one-byte `auxiliary_id` (OF 1.3 and later). The word "feature" here does not mean "new protocol feature"; it means "static property of the switch", such as datapath_id, table count, and supported statistics types, which the controller needs to read once at session start and keep in memory until the channel closes. Dynamic queries (for example, port changes) use a different message (`OFPMP_PORT_DESC`, `OFPT_PORT_STATUS`).

**History and background.** FEATURES_REQUEST and FEATURES_REPLY appeared from OpenFlow 1.0 (Stanford, December 2009), Section 5.3.1 "Switch Features", and are among the six foundational control messages alongside HELLO, BARRIER, CONFIG, FLOW_MOD, and PACKET_IN. The original version placed all three pieces (datapath_id, capabilities, and the full port list) into a single message, because early switches only had a few dozen ports and the payload fit within an MTU. When OF 1.3 (June 2012, Section 7.3.1) was released, ONF moved the port description into a separate multipart message `OFPMP_PORT_DESC`, because on large chassis switches (TOR-of-TOR with hundreds of ports) FEATURES_REPLY could exceed the 16-bit length field of the OF header. OF 1.3 also added the `auxiliary_id` field to support an auxiliary connection (when the controller opens an additional UDP or DTLS channel parallel to the main channel). OF 1.4 (October 2013) removed the `actions` field (which had been replaced by the more detailed `OFPMP_TABLE_FEATURES`). The structure has remained unchanged through OF 1.5.1 (March 2015), the current text.

**Position in the architecture.** On the OpenFlow session timeline, FEATURES_REQUEST sits immediately after HELLO (`OFPT_HELLO`) and before CONFIG (`OFPT_SET_CONFIG`); the OF connection state machine treats this as the third state (NEGOTIATE, IDENTIFY, STEADY). In the OVS source tree, the controller side sends through `lib/vconn.c`; the switch side dispatches through `ofproto/ofproto.c` in the main message loop (line 8665, case `OFPTYPE_FEATURES_REQUEST` calls `handle_features_request`). On the controller side (a real controller, not the switch reply), the reply is decoded through `lib/ofp-switch.c`. Scope of impact: per-bridge, per-connection. Each OVS bridge has its own datapath_id; each controller that opens a channel to a bridge receives that bridge's FEATURES_REPLY. On a host running `ovn-controller`, the channel to `br-int` over a `unix:` socket returns the DPID of `br-int`, not the DPIDs of other bridges.

**Role.** FEATURES_REPLY plays three roles tightly coupled to the entire control plane. First, **persistent identity**: the 64-bit datapath_id is the key that lets the controller distinguish switches within a cluster. Even if a switch changes its IP address, reconnects, or changes a bridge name, the datapath_id stays the same and lets the controller restore state for the right switch. Second, **control capabilities**: the `n_tables` field decides which table number the controller may install flows into (the `table_id` in FLOW_MOD is bounded by `< n_tables`); the `capabilities` bitmap reports up front whether the switch supports FLOW_STATS, GROUP_STATS, or QUEUE_STATS, so the controller does not waste time sending `OFPMP_*` requests that will be rejected. Third, **compatibility check**: if `n_tables = 1` on firmware that advertises OF 1.3, that is a sign the switch only emulates multi-table support; the controller must know this to pick an appropriate pipeline.

**Why it exists.** OpenFlow 1.0 designed the controller to be "switch-stateless": the controller does not store any switch's hard configuration in a local file; every parameter must be fetched from the switch itself on connection. This philosophy addressed two pain points of that era. First, **multi-vendor reality**: different switch ASICs supported different feature sets (Cisco, HP, and NEC early-OF products in 2010 supported only a partial spec); the controller needed a way to ask "what do you support?" instead of assuming a common baseline. Second, **dynamic topology**: when a switch reboots, the datapath_id must remain unchanged so the controller can "rediscover" its position in the topology without out-of-band configuration. Asking the question through FEATURES_REQUEST when the channel is freshly opened, and saving the answer in controller memory, addresses both pain points in a single, cheap, simple message.

**Problems it solves.** FEATURES addresses four concrete technical problems. First, **switch identification across reconnect**: when TCP drops and is reopened, the source IP may change (for example, NAT or bonding switchover); a stable datapath_id lets the controller match the same switch as before. Second, **table compilation**: the pipeline compilers in OVN northd or Faucet need to know how many tables the switch has so they can assign `table_id`; installing a rule into `table_id >= n_tables` returns `OFPBRC_BAD_TABLE_ID`. Third, **stats-poll planning**: the controller polls `OFPMP_FLOW`, `OFPMP_PORT_STATS`, and `OFPMP_GROUP_STATS` periodically; knowing in advance which stats the switch supports avoids sending requests that return `OFPBRC_BAD_TYPE`. Fourth, **auxiliary-connection negotiation** (OF 1.3 and later): when the controller wants to open an additional UDP channel for high-throughput packet_in, the main channel's FEATURES_REPLY returns `auxiliary_id=0`; the auxiliary channels return `auxiliary_id>=1` so the switch can distinguish them. Without FEATURES, the switch has no way to number auxiliary channels.

**Importance.** OFPT_FEATURES sits at the absolute core tier of the OpenFlow message catalogue because it is the **prerequisite for every controller-to-switch interaction after HELLO**. Every OpenFlow library (Ryu, ONOS, Faucet, libfluid, OVN northd, `ovs-ofctl`) has a mandatory code path that sends FEATURES_REQUEST before any user-issued FLOW_MOD; skipping this step is equivalent to "connecting to an anonymous switch", and every log and metric loses the ability to attribute events to the correct physical device. An SDN engineer who does not understand FEATURES cannot debug common chassis-identification problems (DPID collision, card swap, migration), cannot explain why a rule installed into table 5 fails while table 0 works, and cannot understand how auxiliary channels work. FEATURES is as important as HELLO, FLOW_MOD, and BARRIER in the foundation tier of OF 1.0; skipping any one of these four messages makes it impossible to operate OpenFlow.

**How it works.** FEATURES_REQUEST is exactly the 8-byte standard OpenFlow header, with no body:

```
Offset  Field     Length  Example (OF 1.3 FEATURES_REQUEST, xid=2)
------  --------  ------  ----------------------------------------
0       version   1 byte  0x04   (OF 1.3)
1       type      1 byte  0x05   (OFPT_FEATURES_REQUEST)
2-3     length    2 byte  0x00 0x08
4-7     xid       4 byte  0x00 0x00 0x00 0x02

Hex bytes:  04 05 00 08 00 00 00 02
```

FEATURES_REPLY (OF 1.3 and later) has the 8-byte header plus a 24-byte `struct ofp_switch_features`, for a fixed 32-byte total with no port list:

```
Offset  Field         Length  Description
------  ------------  ------  -------------------------------------------------
0-7     header        8 byte  version=0x04, type=0x06, length=0x0020 (32), xid
8-15    datapath_id   8 byte  64-bit unique identifier of the switch
16-19   n_buffers     4 byte  Number of packet buffers (OVS hardcodes 0, see axis 14)
20      n_tables      1 byte  Number of flow tables (OVS v2.17.9 default 254)
21      auxiliary_id  1 byte  Auxiliary channel ID (0 = main, >= 1 = auxiliary)
22-23   pad           2 byte  Padding for 4-byte alignment
24-27   capabilities  4 byte  Capabilities bitmap (see below)
28-31   reserved      4 byte  Formerly `actions` in OF 1.0 and 1.1, now unused
```

The `capabilities` bitmap (defined as `enum ofp_capabilities` in `include/openflow/openflow-common.h` and `enum ofputil_capabilities` in `include/openvswitch/ofp-switch.h`) has eight bits with practical meaning:

| Bit | Flag | Meaning |
|-----|------|---------|
| 0x00000001 | OFPC_FLOW_STATS | Switch supports `OFPMP_FLOW` (per-flow stats) |
| 0x00000002 | OFPC_TABLE_STATS | Supports `OFPMP_TABLE` (per-table stats) |
| 0x00000004 | OFPC_PORT_STATS | Supports `OFPMP_PORT_STATS` |
| 0x00000008 | OFPC11_GROUP_STATS | Supports `OFPMP_GROUP` (OF 1.1 and later) |
| 0x00000020 | OFPC_IP_REASM | The switch reassembles IP fragments before matching |
| 0x00000040 | OFPC_QUEUE_STATS | Supports `OFPMP_QUEUE` |
| 0x00000080 | OFPC_ARP_MATCH_IP | Can match ARP source and target IP (`MFF_ARP_SPA/TPA`) |
| 0x00000100 | OFPC12_PORT_BLOCKED | Switch detects loops and auto-blocks the offending port (OF 1.2 and later, per `include/openflow/openflow-1.2.h:107`) |

OF 1.0, 1.1, and 1.2 also append a `struct ofp_phy_port` array (64 bytes per port for OF 1.0, 64 bytes per port for OF 1.1) describing every physical port. OF 1.3 and later split this out into `OFPMP_PORT_DESC` to avoid overflowing the 16-bit `length`. Important invariant: the total `length` of FEATURES_REPLY must be at most 65535 bytes; if the switch has too many ports, OF 1.0, 1.1, and 1.2 truncate it (per `lib/ofp-switch.c` lines 220 to 225 in v2.17.9: `if (b->size > UINT16_MAX) b->size = start_ofs;`).

**How an operator masters this.** Five core operations on FEATURES. First, **read FEATURES_REPLY by hand**: look at the 32-byte body and identify the DPID in the first 8 bytes (identity), `n_tables` at byte 12, and `capabilities` 4 bytes at offset 24. Second, **match DPID against the OVS configuration**: the command `ovs-vsctl get Bridge br0 datapath_id` returns a DPID as a 16-hex string; it must match the 8-byte hex DPID in FEATURES_REPLY byte-by-byte (note that OVS prints lowercase hex with no separators). Third, **decode the capabilities bitmap**: take 4 big-endian bytes from offset 24 and AND with each flag bit to enumerate. Fourth, **set DPID explicitly**: in production, avoid letting OVS derive DPID from the bridge MAC; use `ovs-vsctl set Bridge br0 other_config:datapath-id=0123456789abcdef`. An internal convention for large clusters (for example, `DPID = AS-number || rack-id || switch-id`) lets on-call engineers read the DPID and immediately know the device location. Fifth, **separate the main channel from auxiliary channels** (OF 1.3 and later): if the controller opens an additional UDP channel for high-throughput packet_in, FEATURES_REPLY on the main channel must return `auxiliary_id=0`; if `auxiliary_id>=1` appears at the start of a session on a channel expected to be main, the controller has mixed up channels and must close and reopen to avoid state crossing.

**Classification.** `OFPT_FEATURES_REQUEST` and `OFPT_FEATURES_REPLY` belong to the **controller-to-switch query and reply** group in the four-group taxonomy (symmetric, controller-to-switch, switch-to-controller, multipart). The pair differs from `OFPT_BARRIER` (same controller-to-switch group, but for synchronization) and from `OFPMP_PORT_DESC` (also carries port information, but as a multipart dynamic query). Inside the "identification message" sub-category, FEATURES is the only message that runs once at session start; dynamic identification events (port up or down) travel through `OFPT_PORT_STATUS`. Body type: empty for the request, fixed-size struct for the reply; no TLV in the reply except for the ports list under OF 1.0, 1.1, and 1.2. FEATURES is a central message of the catalog, on the same level as HELLO, BARRIER, and FLOW_MOD.

**Usage workflow.** The standard workflow for a controller onboarding a new switch has eight steps. First, complete a successful HELLO version negotiation (§3.5.1). Second, send FEATURES_REQUEST with `header.version` set to the negotiated version and a fresh `xid` (typically `xid_hello + 1`). Third, wait for FEATURES_REPLY on the same channel; the maximum wait is usually set to `inactivity_probe` (default 5 seconds in OVS `Controller.inactivity_probe`). Fourth, parse the 32-byte body: store `datapath_id` into `controller_state.switches[dpid]`, store `n_tables` to bound `table_id` when installing rules, and store the capabilities bitmap to choose stats polls. Fifth, if OF 1.0, 1.1, or 1.2, parse the port array; if OF 1.3 or later, send `OFPMP_PORT_DESC` immediately after FEATURES to fetch the port list. Sixth, send `OFPT_SET_CONFIG` to set `miss_send_len` and flags (peripheral §3.5 catalogue). Seventh, every later FLOW_MOD uses `table_id < n_tables`; every stats poll is based on the capabilities already read. Eighth, if an auxiliary channel is needed, open a second TCP, UDP, or DTLS connection and send FEATURES_REQUEST on that channel; FEATURES_REPLY must return `auxiliary_id>=1` and the same `datapath_id` as the main channel; a DPID mismatch indicates a switch misconfiguration.

**When something goes wrong.** Four production-time signals point to FEATURES. One, **DPID collision**: the controller logs "duplicate datapath-id from chassis A and chassis B"; the cause is that the two bridges by default derive the DPID from the MAC of the first port, and those MACs collide (cloned VMs, cloned images, vendor reuse of OUI); confirm with `ovs-vsctl get Bridge br0 datapath_id` on both hosts; fix by setting `other_config:datapath-id` manually. Two, **n_tables = 1 unexpectedly**: a rule installed into table 5 returns `OFPBRC_BAD_TABLE_ID`; running `ovs-ofctl show br0` reports `n_tables: 1` instead of 254; the usual cause is an old switch that only supports OF 1.0, or firmware that caps the table count; either downgrade the pipeline to single-table or replace the device. Three, **capabilities mismatch**: the controller sends `OFPMP_GROUP` and gets an empty reply; FEATURES_REPLY actually lacks the `OFPC11_GROUP_STATS` bit (per `include/openflow/openflow-1.1.h:182`); the controller assumed support incorrectly. Four, **timeout with no FEATURES_REPLY**: HELLO completed but FEATURES_REQUEST went unanswered; a common cause is a switch caught in an upcall storm where the main thread cannot dispatch; confirm via `ovs-appctl coverage/show | grep upcall`. Cross-links: section 20.0 incident-response decision tree, and section 3.5.1 HELLO predecessor.

**Tightly related to.** OFPT_FEATURES is tightly coupled to five entities. First, `OFPT_HELLO` (§3.5.1) is the mandatory predecessor; FEATURES is never sent before HELLO completes. Second, `OFPMP_PORT_DESC` (Multipart, OF 1.3 and later) is the successor that replaces the port list separated from FEATURES_REPLY; every OF 1.3 and later controller must send `OFPMP_PORT_DESC` after FEATURES to obtain the full port list. Third, `Bridge.datapath_id` and `Bridge.other_config:datapath-id` in OVSDB are the source of the DPID that OVS places into FEATURES_REPLY; changing these columns requires reconnecting the OF channel so the controller re-reads them. Fourth, `Chassis.external_ids:datapath-id` in the OVN SBDB references this same DPID; OVN northd uses the DPID to attach Logical_Switch_Ports to the correct chassis. Fifth, the `auxiliary_id` field is bound to `OFPT_FEATURES_REQUEST` on an auxiliary channel and to `Controller.connection_mode` in OVS for auxiliary configuration; a mismatch in either breaks main-versus-auxiliary pairing.

**Version differences.**

| Version | FEATURES_REPLY body | Inline ports? | Has `auxiliary_id`? | `actions` field? |
|---------|---------------------|---------------|---------------------|------------------|
| OF 1.0 (December 2009) | datapath_id + n_buffers + n_tables + capabilities + actions + ports | Yes | No | Yes (uint32 bitmap) |
| OF 1.1 (February 2011) | Adds pad alignment, struct size changes | Yes | No (3-byte pad) | Yes |
| OF 1.2 (December 2011) | Same as 1.1 | Yes | No (3-byte pad) | Yes |
| OF 1.3 (June 2012) | datapath_id + n_buffers + n_tables + auxiliary_id + pad + capabilities + reserved | **No** (moved to `OFPMP_PORT_DESC`) | Yes (1 byte) | Deprecated, the 4 bytes remain as reserved |
| OF 1.4 (October 2013) | Same as 1.3 | No | Yes | Reserved (unused) |
| OF 1.5.1 (March 2015) | Same as 1.3 | No | Yes | Reserved |

OVS handles the three variants through three identifiers `OFPRAW_OFPT10_FEATURES_REPLY`, `OFPRAW_OFPT11_FEATURES_REPLY`, and `OFPRAW_OFPT13_FEATURES_REPLY` (function `ofputil_encode_switch_features` in `lib/ofp-switch.c` v2.17.9 lines 150 to 208); the switch picks the raw type based on `ofconn_get_protocol(ofconn)` and fills the body accordingly. OVS also hardcodes `n_buffers = 0` (`ofproto/ofproto.c` line 3490 v2.17.9) because OVS does not buffer packets at the switch; packet_in always sends the full payload instead of storing a buffer and sending a `buffer_id`. Some hardware switches have `n_buffers > 0`; the controller must check before using `buffer_id` in PACKET_OUT.

**Real-world incident.** A 2024-Q3 incident in an OpenStack and OVN multi-tenant environment: after a new rack of 16 identical compute nodes was deployed from a single base image, `ovn-controller` on three nodes kept losing port bindings. The `ovn-northd.log` showed "chassis with datapath-id 0000525400123456 reported by multiple HV-IDs: hv-007, hv-013, hv-014". A `tcpdump` of the OpenFlow channel to `br-int` on the three nodes confirmed: all three bridges returned the same DPID in FEATURES_REPLY. Root cause: the base image had the `br-int` bridge MAC hardcoded in `/etc/openvswitch/conf.db` instead of letting OVS generate it randomly; three VM clones inherited the same MAC; OVS uses the bridge MAC as the default DPID suffix when `other_config:datapath-id` is empty (`vswitchd/bridge.c` line `bridge_pick_datapath_id`). Fix: write an Ansible script run on each compute, `ovs-vsctl set Bridge br-int other_config:datapath-id=$(printf '0000%012x' $((RANDOM*RANDOM*RANDOM)))`, then reconnect the OF channel; also fix the post-clone hook to randomize the bridge MAC immediately on first VM boot. After the patch, the three chassis have different DPIDs, FEATURES_REPLY on the three channels are clearly distinct, ovn-northd stops reporting collisions, and port binding stabilises. Lesson: the DPID is a global identifier inside the cluster; every cloning workflow (base image, snapshot restore, container template) must include a randomize or explicit-set step before connecting to the control plane.

**Common failures and diagnostic signals.** Six typical errors around FEATURES and their identifying signals. First, **DPID collision in the cluster**: the controller log shows "duplicate datapath-id"; ovn-northd rejects the chassis register; confirm with `ovs-vsctl get Bridge br-int datapath_id` on the suspected nodes; fix by setting DPID explicitly per host. Second, **unexpectedly small `n_tables`**: a controller installing a rule into `table_id >= n_tables` receives `OFPBRC_BAD_TABLE_ID`; the signal is that `ovs-ofctl show br0` reports `n_tables: 1` instead of 254; the usual cause is that the bridge is missing `protocols=OpenFlow13` (default OVS uses OF 1.0, which is single-table); fix with `ovs-vsctl set Bridge br0 protocols=OpenFlow13`. Third, **capabilities mismatch**: the controller sends `OFPMP_GROUP` and gets an empty or error reply; the signal is that FEATURES_REPLY does not have the `OFPC11_GROUP_STATS=0x08` bit that the controller assumed; fix by reading capabilities first and only then polling. Fourth, **timeout with no reply**: HELLO completed but FEATURES_REPLY does not arrive within `inactivity_probe`; the usual cause is that the main switch thread is blocked in `revalidator` or `xlate_actions`; confirm via `ovs-appctl coverage/show`; a temporary fix is to increase `Controller.inactivity_probe`. Fifth, **wrong `auxiliary_id`**: a controller opens an auxiliary channel expecting `auxiliary_id=1` but gets `auxiliary_id=0`; the cause is that the switch does not distinguish main and auxiliary and treats every channel as main; the controller must count its own opened channels rather than trust `auxiliary_id`. Sixth, **`n_buffers > 0` not actually available**: a controller using `buffer_id` in PACKET_OUT assumes the switch can buffer; OVS hardcodes `n_buffers=0`, so `buffer_id=OFP_NO_BUFFER`; the controller must send the full packet in PACKET_OUT instead of referencing a buffer.

**Comparison with other systems.** OFPT_FEATURES_REQUEST has many parallels with, and clear differences from, other identification and capability-discovery mechanisms. First, **NETCONF `<get-capabilities>`** (RFC 6241 Section 8): also queries capabilities right after hello; the difference is that NETCONF capabilities are nested URI strings (one URI per capability), while OF capabilities are a binary bitmap. NETCONF is infinitely extensible but spends bytes; OF is compact but caps at 32 flags. Second, **SNMP sysObjectID and sysDescr** (RFC 1213): also identifies a device through a string; the difference is that sysObjectID uses a vendor-bound OID (`1.3.6.1.4.1.<enterprise>`), not a 64-bit DPID; SNMP identifies "device type" while OF identifies "bridge instance". Third, **gNMI Capabilities RPC**: gNMI returns a list of supported YANG models and encodings; similar to FEATURES in the "lay foundation for later RPCs" role, but different in that gNMI capabilities describe schema while OF describes runtime behaviour. Fourth, **TLS server certificate**: ServerHello carries a certificate with CN or SAN that identifies the server; equivalent to DPID in the "persistent identity" sense, but TLS relies on cryptographic signing while DPID is just a 64-bit string self-declared by the switch (no authentication). Fifth, **BGP OPEN message Router-ID**: a 32-bit router identifier in a BGP session; shorter than the 64-bit DPID but plays the same role as "the key for distinguishing peers" in a distributed control plane; the difference is that Router-ID must be unique within an AS, while DPID must be unique within the controller's cluster. Summary: FEATURES folds identification and capability discovery into a single message, a trait specific to OpenFlow but consistent with most asynchronous control-plane protocols.

## §3.5.3 OFPT_FLOW_MOD (Type 14), cornerstone

### Overview

`OFPT_FLOW_MOD` runs from controller to switch to add, modify, or delete a flow entry. Type: controller-to-switch OF message. First introduced in OF 1.0. A central message of the catalog, since flow-rule push is fundamental to every SDN application.

### How it works

```
Body fields:
  cookie: 64-bit opaque cookie (controller use for tagging)
  cookie_mask: cookie match for delete/modify (OF 1.1+)
  table_id: target table (OF 1.0 = 0xff = any; OF 1.1+ specific)
  command: OFPFC_ADD/MODIFY/MODIFY_STRICT/DELETE/DELETE_STRICT
  idle_timeout, hard_timeout: 16-bit each
  priority: 16-bit (higher = first match wins)
  buffer_id: 32-bit (reference packet buffer if modifying flow for queued packet)
  out_port, out_group: filter for delete (only flows output to specified)
  flags: OFPFF_SEND_FLOW_REM, OFPFF_CHECK_OVERLAP, OFPFF12_RESET_COUNTS, OFPFF13_NO_PKT_COUNTS, OFPFF13_NO_BYT_COUNTS
  match: OXM/NXM TLV match list
  instructions: list of instruction (OF 1.1+) or actions (OF 1.0)
```

Atomicity: single FLOW_MOD = atomic per flow. Bundle atomic = group multiple FLOW_MOD atomic (OF 1.4+).

### 9-13. Operations

Skill: send FLOW_MOD via ovs-ofctl add-flow + observe via dump-flows; understand cookie strategy (per-controller tag cho rule ownership).

### 14. Version drift

| Version | Body changes |
|---------|--------------|
| OF 1.0 | match = simple struct + actions list |
| OF 1.1 | + table_id, instructions instead of actions |
| OF 1.2 | match = OXM TLV list |
| OF 1.3 | + cookie_mask, more flags |
| OF 1.5 | + egress table support, packet_type match |

### 15. Verification

```
$ ovs-ofctl add-flow br0 'priority=100,ip,actions=output:1' --bundle
$ tcpdump -i any -nn -X port 6653 | grep -A 20 "type 14"
-> Decode FLOW_MOD body

$ ovs-ofctl dump-flows br0
-> Confirm flow installed
```

### Source code reference

- `lib/ofp-msgs.c` `ofputil_decode_flow_mod()`
- `ofproto/ofproto.c` `ofproto_flow_mod_init()`, `ofproto_flow_mod_finish()`
- `ofproto/ofproto-dpif.c` `flow_mod_init()` integrates with the classifier

### 17. Production case

2023-Q4: Controller push 10K FLOW_MOD/sec → ovs-vswitchd CPU 100%. Investigation: each FLOW_MOD individual transaction → classifier rebuild overhead. Fix: use OFPT_BUNDLE (OF 1.4+) batch atomic, classifier rebuild once per bundle. Throughput 10K/sec maintained, CPU 30%.

### 18. Lab

```

$ ovs-ofctl mod-flows br0 'priority=100,in_port=1,actions=output:3'
$ ovs-ofctl del-flows br0 'priority=100,in_port=1'
$ ovs-ofctl dump-flows br0
-> Verify each operation effect

# Bundle test (OF 1.4+):
$ ovs-ofctl --bundle add-flows br0 - <<EOF
priority=100,in_port=1,actions=output:2
priority=100,in_port=2,actions=output:1
priority=100,in_port=3,actions=output:4
EOF
-> All 3 flows installed atomically
```

### 19. Failure mode

- Mode A: FLOW_MOD command field invalid, returns OFPT_ERROR. Recovery: validate the command before sending.
- Mode B: match overlaps an existing flow with OFPFF_CHECK_OVERLAP set, the request is rejected. Recovery: differentiate by explicit priority.
- Mode C: `buffer_id` references a stale buffer, returns OFPT_ERROR BAD_BUFFER. Recovery: re-send the PACKET_IN buffer.
- Mode D: Bundle commit fails partially, all-or-nothing semantics apply. Recovery: review the bundle order and fix the root cause.

### Comparison with other systems

- **gNMI Set:** model-based config push. Difference: gNMI is tree-structured YANG, while FLOW_MOD is per-rule.
- **NETCONF edit-config:** XML-based transactional config. The bundle concept in OF 1.4+ is similar.
- **P4Runtime WriteRequest:** P4 table entry add, modify, delete. Difference: P4 is schema-driven; OF is a flow-rule abstraction.

## §3.5.4 OFPT_PACKET_IN (Type 10), cornerstone

### Overview

`OFPT_PACKET_IN` runs from switch to controller when a packet matches a flow rule with action `controller`, or when a table miss has no explicit handling. Type: switch-to-controller OF message. First introduced in OF 1.0. A central message of the catalog, since it is the basis for the "punt to controller" pattern (LLDP discovery, ARP responder, controller-based routing).

### How it works

```
Body fields:
  buffer_id: 32-bit (kernel buffer reference) or 0xffffffff = no buffer
  total_len: 16-bit packet original length
  reason: OFPR_NO_MATCH (table-miss), OFPR_ACTION (action=controller), OFPR_INVALID_TTL
  table_id: 8-bit source table (OF 1.1 and later)
  cookie: 64-bit flow cookie that triggered (OF 1.3 and later)
  match: packet metadata (OF 1.2 and later)
```

## §3.5.5 OFPT_BARRIER_REQUEST/REPLY, the synchronisation fence between controller and switch

`OFPT_BARRIER_REQUEST` (type=18 in OF 1.0, type=20 from OF 1.1 onward) and `OFPT_BARRIER_REPLY` (type=19 in OF 1.0, type=21 from OF 1.1 onward) are a controller-initiated message pair sent to the switch to force the switch to finish processing every previously received message before sending an acknowledgement reply. This is the only synchronisation primitive in the OpenFlow protocol before the Bundle mechanism appeared in OF 1.4, and it remains the standard tool for error correlation and for ordering groups of `OFPT_FLOW_MOD`, `OFPT_PORT_MOD`, `OFPT_GROUP_MOD`, and `OFPT_METER_MOD` that the protocol otherwise allows the switch to process in any order.

**Concept.** OFPT_BARRIER is a **pair of empty messages**: both request and reply contain only the 8-byte OpenFlow header, with no body. The controller sends BARRIER_REQUEST with an `xid` (transaction id) of its own choosing; the switch must finish processing every message received before the barrier (in TCP byte-stream order) and then send BARRIER_REPLY with the same `xid`, so that the controller knows "up to this point, everything before is finished and every error has been emitted before the reply". With an empty payload, BARRIER is a **synchronisation-only primitive**; it carries no business data, and its only function is to split the message stream into batches with a defined order.

**History and background.** BARRIER_REQUEST/REPLY appeared from OpenFlow 1.0 (Stanford, December 2009) as one of the six foundational control messages (HELLO, FEATURES, BARRIER, CONFIG, FLOW_MOD, PACKET_IN/OUT). The design rationale lies in the original OF 1.0 assumption (Section 5.4 "Message Handling"): the switch is permitted to process control messages in parallel and in any order to increase throughput. The consequence is that the controller cannot tell when a batch of FLOW_MOD has been fully applied without an explicit synchronisation point. ONF carried the structure into 1.1 (February 2011) but renumbered the `type` from `18/19` to `20/21` to make room for METER_MOD and GROUP_MOD. From OF 1.4 (October 2013) onward, ONF added the Bundle family (OFPT_BUNDLE_OPEN, COMMIT, ADD_MESSAGE, DISCARD) as a more comprehensive atomic-transaction mechanism, but BARRIER is retained as the lowest-level primitive and persists into the current OpenFlow 1.5.1 (March 2015).

**Position in the architecture.** On the OpenFlow session timeline, BARRIER only appears after HELLO and FEATURES_REQUEST complete and the channel has entered the `STEADY` state; the controller may insert BARRIER at any point during the `STEADY` lifetime until the channel closes. In the OVS source tree, BARRIER is decoded by `lib/ofp-msgs.c` (through identifiers `OFPRAW_OFPT10_BARRIER_REQUEST` and `OFPRAW_OFPT11_BARRIER_REQUEST`, defined at lines 217 to 224 of `include/openvswitch/ofp-msgs.h`), dispatched to `ofproto/ofproto.c` in the main message loop, and handled in `handle_barrier_request()` (lines 6461 to 6470 of `ofproto/ofproto.c` v2.17.9). Scope of impact: per-connection (each BARRIER only synchronises messages on the OF channel it traverses, with no effect on other controllers talking to the same switch).

**Role.** BARRIER plays the role of a **fence message** between groups of mutually dependent operations. Without BARRIER, the switch is allowed by spec to reorder message processing on the same channel, so a sequence such as "FLOW_MOD ADD priority=200" followed by "FLOW_MOD DELETE priority=200" could be applied in reverse order, ending with the rule deleted instead of kept. The second role is **error correlation**: the spec requires every `OFPT_ERROR` for messages before the barrier to be sent before the corresponding BARRIER_REPLY; this lets the controller attribute errors to the correct `xid` without long timeouts. The third role is **batch acknowledgement**: instead of asking the switch to ack each message (which increases round trips), the controller only needs one BARRIER_REPLY after a batch of dozens of FLOW_MODs.

**Why it exists.** In control protocols predating OpenFlow, each command was usually its own synchronous RPC (NETCONF lock-edit-commit, BGP UPDATE/KEEPALIVE, SNMP set/get). OpenFlow 1.0 deliberately walked away from that pattern, fearing that an RPC per rule would kill throughput when the controller had to push thousands of rules into TCAM within seconds. Consequence of the freedom: the switch processes in parallel happily. But that very freedom creates two pain points. First, the controller cannot tell "when it is done" before moving on. Second, an error can arrive at any later time, blurring attribution. BARRIER was created to solve exactly those two pain points while keeping the parallel-processing benefit between barriers: within a group separated by BARRIER pairs, the switch is free; between groups, the order is strictly enforced.

**Problems it solves.** Concretely, BARRIER addresses four technical problems. First, **race conditions when installing dependent rules**: the controller needs to delete an old rule before installing a new rule with the same match but a different output; if the order is reversed, packets can match both rules. BARRIER forces the switch to finish the deletion before reading the next message. Second, **reply ordering for multiple OFPMP_FLOW requests**: when the controller polls stats for three tables at the same time, a BARRIER between two poll bursts separates the reply boundary. Third, **error attribution**: when 50 FLOW_MODs are sent in a burst and one fails, the spec guarantees the corresponding error is sent before the next BARRIER_REPLY, so the controller only needs to count its position in the `xid` queue to know which FLOW_MOD failed. Fourth, **post-restart resynchronisation**: after a switch reconnect, the controller usually flushes the table and reinstalls a baseline; inserting a BARRIER between "flush" and "install" prevents installing a rule into a table that has not finished flushing.

**Importance.** OFPT_BARRIER sits in the core tier of the OpenFlow control message set because it is the **only synchronisation primitive available from OF 1.0 to OF 1.3** (so the overwhelming majority of switches and controllers in production, including older hardware still in service, must use BARRIER, since Bundle is not yet available). Every serious OpenFlow library (Ryu, ONOS, libfluid, libofproto) has a `send_barrier()` function or equivalent, and every batch-install procedure in the OpenFlow Switch Specification (Section 6.2 "OpenFlow Connection") assumes the controller will insert BARRIER between groups. An SDN engineer who does not understand BARRIER cannot debug the most common failure classes when pushing batches of rules: rules not landing, rules landing in the wrong order, errors not matching the bad rule. BARRIER sits at the same importance tier as HELLO and FLOW_MOD in the OF 1.0 foundation, despite its tiny body.

**How it works.** The on-wire structure of BARRIER_REQUEST is exactly the 8-byte standard OpenFlow header:

```
Offset  Field     Length  Example (OF 1.3 BARRIER_REQUEST, xid=0x42)
------  --------  ------  ----------------------------------------
0       version   1 byte  0x04   (OF 1.3)
1       type      1 byte  0x14   (= 20, OFPT_BARRIER_REQUEST in OF 1.1 and later)
2-3     length    2 byte  0x00 0x08
4-7     xid       4 byte  0x00 0x00 0x00 0x42

Hex bytes:  04 14 00 08 00 00 00 42
```

BARRIER_REPLY has the same structure with only the `type` differing (19 for OF 1.0, 21 for OF 1.1 and later) and must use the same `xid` as the request it answers. The switch-side processing flow (`ofproto/ofproto.c` v2.17.9, function `handle_barrier_request` lines 6461 to 6470) is short but complete: when the message reaches the main dispatch function (lines 8691 to 8692, case `OFPTYPE_BARRIER_REQUEST`), the function only allocates an empty reply through `ofpraw_alloc_reply()` and sends it through `ofconn_send_reply()`; no wait loop. Why is that enough? Because OVS handles control messages on a single main thread in receive order; by the time the dispatcher reaches the BARRIER handler, every earlier handler for FLOW_MOD, PORT_MOD, GROUP_MOD, and METER_MOD on the same channel has returned (through single-threaded FIFO). Important invariant: the byte order on the TCP stream determines the processing order, and BARRIER only relies on this invariant to push its reply to the back of the queue.

**How an operator masters this.** Four core skills with BARRIER. First, **read the request/reply pair on the wire**: capture `tcpdump` on port 6653, filter for bytes 0x14/0x15 (OF 1.3 and later) or 0x12/0x13 (OF 1.0), and match the `xid` to confirm the switch responded. Second, **insert BARRIER by hand** during incident investigation: with `ovs-ofctl monitor` in interactive mode, type the `barrier` command to send BARRIER_REQUEST and observe the reply latency (which tells whether the switch is busy or idle). Third, **measure reply latency** (the `echo-request -> barrier-reply` interval) as an overload indicator: if BARRIER_REPLY takes more than a few hundred milliseconds on an idle `ovs-vswitchd`, the dispatcher is stuck in another handler. Fourth, **combine with `--bundle`**: in scenarios that need both atomicity and ordering, use Bundle (OFPT_BUNDLE_OPEN + ADD + COMMIT) instead of plain BARRIER; but remember that Bundle does not replace BARRIER at the pure-synchronisation layer (the Bundle commit has implicit barrier semantics, but the `ATOMIC` flag must be set for full guarantees). Which to use: BARRIER for "make sure it is finished before continuing"; Bundle for "all succeed or none succeed".

**Classification.** OFPT_BARRIER_REQUEST/REPLY belongs to the **Controller-to-Switch synchronisation message** group in the four-group OpenFlow taxonomy. Different from `OFPT_FLOW_MOD` (same controller-to-switch group but configuration), and different from `OFPT_ECHO_REQUEST/REPLY` (symmetric, for liveness rather than ordering). Within the "synchronisation primitive" sub-category, BARRIER is the lowest level (per-connection, no transactional guarantee); the next level is Bundle (per-connection but atomic); the highest level is the OVN northd transaction (per-database, not per-connection). Body type: empty (header only), no TLV, no version-dependent payload variant.

**Usage workflow.** The standard workflow for a controller installing a dependent batch of FLOW_MODs has seven steps. First, group the FLOW_MODs in the desired install order (delete-old before add-new, or add-new before delete-old, depending on the no-traffic-loss strategy). Second, send each FLOW_MOD on the OpenFlow channel with increasing `xid`. Third, immediately after the last message of the group, send BARRIER_REQUEST with a pre-assigned `xid` (for example, `xid_barrier = xid_last_flow_mod + 1`). Fourth, the controller transitions to `WAITING_BARRIER`: do not send the next FLOW_MOD group, do not cancel the existing timer. Fifth, read messages from the channel; every `OFPT_ERROR` arriving before BARRIER_REPLY must be logged together with its `xid` to identify which FLOW_MOD failed. Sixth, when BARRIER_REPLY with the matching `xid` arrives, the controller confirms the batch is finished and every error has been emitted; move to the next group. Seventh, if BARRIER_REPLY does not arrive within `barrier_timeout` (default 30 seconds in many controllers), the controller must treat the channel as broken, close it, and reconnect (the spec does not allow the controller to skip a barrier; skipping leads to lost error tracking).

**When something goes wrong.** Four production-time signals point to BARRIER. First, **a batch loses some rules**: the on-call engineer sees that `ovs-ofctl dump-flows br0` is missing a few rules from the burst of 50 just pushed; trace the controller log for any FLOW_MOD that received `OFPT_ERROR` and check whether that error came before or after the batch's BARRIER_REPLY. Second, **rules applied in the wrong order**: a new rule sits on top of an old rule that has not yet been deleted; reinstall using a delete + BARRIER + add pattern with explicit separation. Third, **the controller hangs during flush**: BARRIER_REPLY does not arrive within seconds after pushing 1000 FLOW_MODs; usually the switch dispatcher is stuck in `revalidator` or `xlate_actions`; confirm via `ovs-appctl coverage/show` that the `xlate_actions` or `revalidator/wait` counters are increasing rapidly. Fourth, **error xid does not match**: the controller log shows `OFPT_ERROR (xid=0x42)` but the controller has no record of a FLOW_MOD sent with `xid=0x42`; a common cause is reusing the xid pool too quickly without resetting after each BARRIER; either expand the xid space or insert BARRIER more frequently to make the `xid` boundary of errors clear.

**Tightly related to.** OFPT_BARRIER is tightly coupled to five entities. First, `OFPT_FLOW_MOD` is BARRIER's primary client: most BARRIERs in production sit after a batch of FLOW_MODs. Second, `OFPT_ERROR` (Type 1) is closely tied because the spec requires errors for pre-barrier messages to arrive before BARRIER_REPLY; any change in error handling affects how BARRIER is used. Third, `OFPT_BUNDLE_*` (OF 1.4 and later) is the conceptual successor; every `OFPT_BUNDLE_COMMIT` implicitly carries barrier semantics, so the two mechanisms usually exclude each other in the same flow. Fourth, the `xid` field in the header (Section 7.1.1 of the spec) is the key for matching request to reply and the mechanism the controller uses to correlate errors to FLOW_MODs; xid reused too quickly defeats BARRIER's correlation effect. Fifth, `ofproto/connmgr.c` (the OVS connection manager) maintains the send queue and ensures BARRIER_REPLY follows every prior message; ordering issues on the switch side typically live in this layer.

**Real-world incident.** A 2024-Q3 incident in an OpenStack/OVN environment after rolling-restarting Neutron: in the 90 seconds after each `neutron-server` restart, newly-created Logical_Switch_Ports were not attached to the datapath, and freshly-booted VMs had no connectivity. A `tcpdump` on the `unix:/var/run/openvswitch/br-int.mgmt` channel showed `ovn-controller` pushing more than 3000 FLOW_MODs in 2 seconds, but BARRIER_REPLY took up to 12 seconds to arrive (versus the 50 ms baseline). `ovs-appctl coverage/show` reported the `xlate_actions` counter rising by nearly a million in that window, and `revalidator/wait` was at the threshold. Root cause: the controller side reset the entire flow cache and pushed everything from scratch (a reconcile-logic side effect); the main thread serialised more than 3000 FLOW_MODs; the BARRIER_REQUEST that `ovn-controller` sent after the batch was queued at the tail, and BARRIER_REPLY had to wait for every prior FLOW_MOD to be applied. During the wait, `ovn-controller` blocked the next pipeline stage, so new ports were not bound. Fix: split the FLOW_MOD batch into sub-batches of about 500 messages, each sub-batch closed by a BARRIER, to give the main thread breathing room and to let the binding pipeline progress steadily. After the controller-side patch, BARRIER_REPLY latency dropped to about 600 ms per sub-batch, and port-binding lag fell from 90 seconds to under 5. Lesson: BARRIER is not a free primitive; its latency is proportional to the queue of messages ahead of it, so the cadence of BARRIER insertion directly affects control-plane responsiveness.

**Common failures and diagnostic signals.** Five typical errors around BARRIER and their identifying signals. First, **BARRIER timeout**: BARRIER_REPLY does not arrive within the controller's deadline (default 30 seconds in Ryu, 5 seconds in libfluid); the signal is the controller log `barrier timeout xid=0xN`; the usual cause is a switch dispatcher stuck in another handler (a large FLOW_MOD, a heavy GROUP_MOD, a revalidator storm). Fix: split the message batch before the barrier, or temporarily increase the timeout, or check `ovs-appctl coverage/show`. Second, **xid mismatch**: BARRIER_REPLY has a `xid` different from the most recent BARRIER_REQUEST `xid`; usually because the controller sent multiple overlapping BARRIER_REQUESTs and the xid counter wrapped; the signal is the log "received barrier reply for unknown xid"; fix by allocating a separate xid namespace for BARRIER or by serialising BARRIER emission. Third, **error missing xid context**: an `OFPT_ERROR` arrives but its `xid` does not match any FLOW_MOD in the controller's bookkeeping; the usual cause is the controller deleting the entry before BARRIER_REPLY arrives; fix by retaining the entry until the matching BARRIER_REPLY of the batch arrives. Fourth, **silent processing without barrier**: the controller does not insert a barrier after a FLOW_MOD batch; the signal is that `dump-flows` occasionally misses rules with no clear cause; fix by enforcing a barrier insertion either by count (for example, every 100 messages) or by dependency. Fifth, **double barrier**: two consecutive BARRIER_REQUESTs cause the controller to misread the first reply as covering both; fix by following the convention "one barrier per batch" and using clearly distinct xids.

**Comparison with other systems.** OFPT_BARRIER is the equivalent of several synchronisation primitives in other domains. First, **POSIX `fsync()`**: forces the kernel to flush buffer pages to disk before returning; similar to BARRIER in "ensure everything before is finished", different in that fsync only applies to one local file descriptor while BARRIER applies across the entire session between two machines. Second, **MPI `MPI_Barrier()`**: synchronises N processes at a common point and returns when all processes reach the point; OFPT_BARRIER only synchronises between two parties (controller and switch), not generalised to N parties. Third, **Vulkan `vkCmdPipelineBarrier()`**: a fence in a GPU command buffer that ensures memory dependencies between passes; similar to BARRIER in the "before-after" relationship, different in that the Vulkan barrier can specify a scope (memory domain) and specific stages, while OF BARRIER is a coarse fence. Fourth, **NETCONF `<commit>`**: ends a transaction containing several `<edit-config>` operations; similar to BARRIER in the "everything before is done" role, different in that `<commit>` carries atomic semantics (rollback on failure) while BARRIER is purely ordering, not atomic; Bundle (OF 1.4 and later) is the equivalent of the NETCONF commit. Fifth, **SQL `COMMIT`** in a transactional database: ensures every statement in the transaction is durable; closer to Bundle than to BARRIER, since BARRIER does not promise durability, only ordering. Summary: BARRIER is the minimal version of the "fence message" idea found in most asynchronous control protocols.

## §3.5.6 Nhóm message peripheral cấu hình runtime

Năm message cấu hình + truy vấn dưới đây không thuộc tier cornerstone (không cấu trúc phiên kết nối) nhưng cần thiết trong vận hành: chỉnh hành vi switch tại runtime, đổi trạng thái port, đọc cấu hình queue. Mỗi message được trình bày dưới dạng tra cứu kỹ thuật ngắn nhưng đủ để kỹ sư đọc được wire bytes, biết version drift, có lab kiểm chứng.

### §3.5.6.1 OFPT_GET_CONFIG_REQUEST / REPLY (Type 7 / 8)

**Khái niệm.** Cặp request-reply do controller gửi xuống switch để đọc hai tham số toàn cục: `flags` (chế độ xử lý fragment IP) và `miss_send_len` (số byte tối đa của packet được kẹp vào `OFPT_PACKET_IN` khi không có buffer_id). Request rỗng (chỉ 8 byte header), reply mang body `struct ofp_switch_config` 12 byte tổng (8 byte header + 4 byte body).

**Cơ chế hoạt động + wire layout.**

```
GET_CONFIG_REQUEST (12 byte tổng):
  04 07 00 08 00 00 00 0A           ; version=4, type=7, len=8, xid=10

GET_CONFIG_REPLY:
  04 08 00 0C 00 00 00 0A            ; header (version=4, type=8, len=12, xid=10)
  00 00 00 80                       ; flags=0x0000, miss_send_len=128 (0x0080)

flags bitmap (OF 1.3+):
  0x0000 OFPC_FRAG_NORMAL    ; xử lý fragment bình thường, match qua pipeline
  0x0001 OFPC_FRAG_DROP      ; drop mọi IP fragment
  0x0002 OFPC_FRAG_REASM     ; reassemble fragment (yêu cầu OFPC_IP_REASM capability)
  0x0003 OFPC_FRAG_MASK
```

**Vị trí + version drift.** Có mặt từ OF 1.0 đến OF 1.5.1 với cùng số type 7/8 và cùng cấu trúc body. OF 1.0 thêm cờ `OFPC_FRAG_NORMAL/DROP/REASM` đã liệt kê; OF 1.1+ giữ nguyên. Trong OVS baseline `v2.17.9`, decode tại `lib/ofp-msgs.c` qua `OFPRAW_OFPT_GET_CONFIG_REQUEST` (line 171) và `OFPRAW_OFPT_GET_CONFIG_REPLY` (line 174); handler `handle_get_config_request()` tại `ofproto/ofproto.c` đọc trực tiếp `ofproto->frag_handling` và `ofproto->miss_send_len`, không có disk I/O.

**Verification + lab.**

```
$ ovs-ofctl -O OpenFlow13 get-config br0
OFPST_GET_CONFIG_REPLY (OF1.3) (xid=0x2):
  frags=normal miss_send_len=0

$ sudo tcpdump -i any -nn 'tcp port 6653' -X -c 4
  04 07 00 08 ... ; request bytes
  04 08 00 0C ... 00 00 00 80 ; reply bytes với miss_send_len=128
```

**Lỗi thường gặp.** `miss_send_len=0` (mặc định OF 1.3) làm mọi `OFPT_PACKET_IN` không có buffer_id bị truncate xuống 0 byte payload, controller không thể parse L2/L3 header. Khắc phục: `ovs-ofctl set-config br0 miss-send-len=128` (hoặc cao hơn nếu cần đọc IPv6 + payload).

### §3.5.6.2 OFPT_SET_CONFIG (Type 9)

**Khái niệm.** Message một chiều controller-to-switch để ghi đè hai tham số đã liệt kê ở §3.5.6.1 (`flags` + `miss_send_len`). Body trùng với `GET_CONFIG_REPLY`: 4 byte tổng (`uint16 flags`, `uint16 miss_send_len`). Switch không trả `SET_CONFIG_REPLY`; controller chỉ biết thành công khi không có `OFPT_ERROR` về trong khoảng `barrier_timeout` hoặc khi BARRIER_REPLY tiếp theo về sạch.

**Cơ chế hoạt động + wire layout.**

```
SET_CONFIG (12 byte tổng):
  04 09 00 0C 00 00 00 0B            ; version=4, type=9, len=12, xid=11
  00 00 02 00                        ; flags=OFPC_FRAG_NORMAL, miss_send_len=512

Tổ hợp điển hình:
  flags=0, miss_send_len=128         ; baseline debugging (đủ L2+L3 header)
  flags=0, miss_send_len=0xFFFF      ; send the entire packet without truncation (OFPCML12_NO_BUFFER per include/openflow/openflow-1.2.h:93)
  flags=OFPC_FRAG_DROP, msl=128      ; chặn fragment để tránh evasion
```

**Vị trí + version drift.** Có mặt từ OF 1.0 với cùng type=9 và cấu trúc body 4 byte. Không thay đổi qua các phiên bản. Decode OVS: `OFPRAW_OFPT_SET_CONFIG` (line 177 `include/openvswitch/ofp-msgs.h` v2.17.9). Handler ghi vào hai field nội bộ `ofproto->frag_handling` và `ofproto->miss_send_len`; thay đổi có hiệu lực ngay với mọi PACKET_IN tiếp theo.

**Verification + lab.**

```
$ sudo ovs-ofctl set-config br0 frags=normal miss-send-len=128
$ sudo ovs-ofctl get-config br0
OFPST_GET_CONFIG_REPLY (OF1.3) (xid=0x2):
  frags=normal miss_send_len=128

# Quan sát wire:
$ sudo tcpdump -i any -nn 'tcp port 6653' -X -c 2
  04 09 00 0C ... 00 00 00 80
```

**Lỗi thường gặp.** Set `miss_send_len` quá nhỏ (< 64) làm cả Ethernet header bị cắt; quá lớn (> path MTU) khiến PACKET_IN bị fragment ở tầng IP gây delay. Giá trị thực dụng 128-256 cho debug, 0xFFFF khi cần forensic toàn gói (kèm chính sách rate-limit ở controller side).

### §3.5.6.3 OFPT_PORT_MOD (Type 16, OF 1.0 type=15)

**Khái niệm.** Message controller-to-switch để chỉnh trạng thái administrative + tính năng quảng bá của một port: bật/tắt port (`OFPPC_PORT_DOWN`), chặn flood (`OFPPC_NO_FLOOD`), chặn forward (`OFPPC_NO_FWD`), chặn packet_in từ port này (`OFPPC_NO_RECV`), v.v. Không xoá port; chỉ thay đổi tập cờ ở struct port. Body khác biệt giữa OF 1.0 (`struct ofp10_port_mod` 32 byte) và OF 1.1+ (`struct ofp11_port_mod` 40 byte vì port_no mở rộng từ 16 bit → 32 bit), và OF 1.4+ (`struct ofp14_port_mod` thêm TLV properties cho ethernet/optical/experimenter).

**Cơ chế hoạt động + wire layout (OF 1.3 variant).**

```
PORT_MOD (40 byte tổng):
  04 10 00 28 00 00 00 0C            ; version=4, type=16, len=40, xid=12
  00 00 00 03                        ; port_no = 3
  00 00 00 00                        ; pad 4 byte
  AA BB CC DD EE 03                  ; hw_addr (phải khớp với hw_addr port hiện hành)
  00 00                              ; pad 2 byte
  00 00 00 01                        ; config = OFPPC_PORT_DOWN
  00 00 00 01                        ; mask  = chỉ thay bit OFPPC_PORT_DOWN
  00 00 00 00                        ; advertise = 0 (không thay đổi)
  00 00 00 00                        ; pad 4 byte

config bitmap:
  0x00000001 OFPPC_PORT_DOWN     ; admin down
  0x00000004 OFPPC_NO_RECV       ; bỏ qua mọi gói nhận từ port
  0x00000020 OFPPC_NO_FWD        ; không forward gói ra port
  0x00000040 OFPPC_NO_PACKET_IN  ; không sinh PACKET_IN từ port
```

**Vị trí + version drift.** Variant 1.0 dùng `port_no` 16-bit + ít cờ; variant 1.1-1.3 mở rộng 32-bit + thêm `OFPPC_NO_RECV`; variant 1.4+ thêm danh sách TLV properties (`OFPPMPT_ETHERNET`, `OFPPMPT_OPTICAL`, `OFPPMPT_EXPERIMENTER`) cho phép set advertise per-medium. Decode OVS: `OFPRAW_OFPT10_PORT_MOD` (line 250), `OFPRAW_OFPT11_PORT_MOD` (line 252), `OFPRAW_OFPT14_PORT_MOD` (line 254). Handler `handle_port_mod()` tại `ofproto/ofproto.c` áp dụng `(config & mask)` lên `port->config` rồi gọi netdev backend update `IFF_UP` (kernel datapath) hoặc đặt `dpif_port` flag (userspace).

**Verification + lab.**

```
$ sudo ovs-ofctl mod-port br0 eth1 down
$ sudo tcpdump -i any -nn 'tcp port 6653' -X -c 1
  04 10 00 28 ... 00 00 00 03 ... 00 00 00 01 00 00 00 01

$ sudo ovs-ofctl show br0 | grep eth1
   3(eth1): addr:aa:bb:cc:dd:ee:03
       config:     PORT_DOWN
       state:      LINK_DOWN
```

**Lỗi thường gặp.** `OFPT_ERROR / OFPBRC_BAD_HW_ADDR` khi `hw_addr` trong message không khớp địa chỉ MAC hiện hành (port có thể đã đổi MAC bởi `ip link set address`). Khắc phục: đọc lại MAC từ FEATURES_REPLY hoặc OFPMP_PORT_DESC trước khi gửi PORT_MOD.

### §3.5.6.4 OFPT_TABLE_MOD (Type 17, OF 1.1+)

**Khái niệm.** Message controller-to-switch để cấu hình **hành vi mặc định của một flow table** (không sửa rule cụ thể). Nguyên thuỷ OF 1.1-1.3: chỉnh `table-miss config` qua `OFPTC_TABLE_MISS_CONTROLLER/CONTINUE/DROP`. Từ OF 1.3 trở đi `config` field bị deprecate (table-miss được biểu diễn bằng rule priority=0 explicit), nhưng message vẫn tồn tại với body rỗng config + danh sách TLV `OFPTMPT_*` (eviction, vacancy event) ở OF 1.4+.

**Cơ chế hoạt động + wire layout (OF 1.4+ variant rỗng).**

```
TABLE_MOD (16 byte tổng OF 1.4+, không có TLV):
  04 11 00 10 00 00 00 0D            ; version=4, type=17, len=16, xid=13
  00                                  ; table_id = 0
  00 00 00                            ; pad 3 byte
  00 00 00 00                        ; config = 0 (deprecated bits)
  ; (OF 1.4+ có thể tiếp nối TLV properties; OF 1.1-1.3 dừng tại đây với len=16)

OF 1.1-1.3 config bitmap (deprecated từ 1.3):
  0x00000000 OFPTC_TABLE_MISS_CONTROLLER  ; gửi PACKET_IN nếu không match (default)
  0x00000001 OFPTC_TABLE_MISS_CONTINUE    ; chuyển sang bảng kế tiếp
  0x00000002 OFPTC_TABLE_MISS_DROP        ; drop gói

OF 1.4+ TLV properties:
  OFPTMPT_EVICTION       ; eviction policy khi bảng đầy
  OFPTMPT_VACANCY        ; ngưỡng phát OFPT_TABLE_STATUS
  OFPTMPT_EXPERIMENTER   ; vendor extension
```

**Vị trí + version drift.** Xuất hiện từ OF 1.1 (table-miss config tường minh thay cho hành vi cố định OF 1.0). OF 1.3 deprecate `config` bitmap (chuyển sang dùng table-miss rule priority 0). OF 1.4 mở rộng TLV. Decode OVS: `OFPRAW_OFPT11_TABLE_MOD` (line 257), `OFPRAW_OFPT14_TABLE_MOD` (line 259). OVS xử lý nhưng phần lớn `config` bị bỏ qua khi version >= 1.3; eviction TLV được lưu vào `oftable->eviction_fields` ảnh hưởng `evict()` trong `ofproto/ofproto.c`.

**Verification + lab.**

```
# OF 1.3+ thay table-miss qua flow rule priority 0:
$ sudo ovs-ofctl -O OpenFlow13 add-flow br0 'table=0,priority=0,actions=CONTROLLER:128'

# Wire dump TABLE_MOD (chỉ thấy ở controller phát explicit):
$ sudo tcpdump -i any -nn 'tcp port 6653' -X -c 1
  04 11 00 10 ... 00 00 00 00 00 00 00 00
```

**Lỗi thường gặp.** Phát TABLE_MOD với `config=OFPTC_TABLE_MISS_CONTINUE` trên switch OF 1.3+ bị silent ignore (không lỗi nhưng không có hiệu lực). Khắc phục: dùng table-miss rule priority 0 thay vì TABLE_MOD config bitmap. OF 1.4+ TLV `OFPTMPT_VACANCY` chưa hỗ trợ trên mọi switch, kiểm tra capability trước khi gửi.

### §3.5.6.5 OFPT_QUEUE_GET_CONFIG_REQUEST / REPLY (Type 22 / 23, OF 1.0 type=20/21)

**Khái niệm.** Cặp request-reply để controller đọc danh sách queue đã được cấu hình trên một port (set bởi traffic shaping CLI hoặc OVSDB `QoS` table). Request mang `port_no`; reply mang danh sách `struct ofp_packet_queue` mỗi entry chứa `queue_id`, `port`, độ dài, danh sách TLV `OFPQT_MIN_RATE` / `OFPQT_MAX_RATE` / `OFPQT_EXPERIMENTER`. Từ OF 1.4 message này bị deprecate, chuyển sang `OFPMP_QUEUE_DESC` multipart để tận dụng cơ chế multipart fragmentation cho danh sách queue lớn.

**Cơ chế hoạt động + wire layout (OF 1.3 variant).**

```
QUEUE_GET_CONFIG_REQUEST (16 byte tổng OF 1.1-1.3):
  04 16 00 10 00 00 00 0E            ; version=4, type=22, len=16, xid=14
  00 00 00 03                        ; port = 3
  00 00 00 00                        ; pad 4 byte

QUEUE_GET_CONFIG_REPLY (32 byte ví dụ với 1 queue + 1 property):
  04 17 00 20 00 00 00 0E            ; header version=4, type=23, len=32, xid=14
  00 00 00 03                        ; port = 3
  00 00 00 00                        ; pad 4 byte
  ; ofp_packet_queue:
  00 00 00 01                        ; queue_id = 1
  00 00 00 03                        ; port (lặp lại)
  00 18 00 00                        ; len = 0x18=24, pad 2
  ; ofp_queue_prop_min_rate (16 byte):
  00 01 00 10 00 00 00 00            ; property=OFPQT_MIN_RATE(1), len=16, pad 4

```

**Vị trí + version drift.** OF 1.0 dùng `port_no` 16-bit (`struct ofp10_queue_get_config_request`), OF 1.1-1.3 mở rộng 32-bit, OF 1.4+ chuyển sang `OFPMP_QUEUE_DESC` (multipart, deprecate message kế thừa nhưng vẫn parser được). Decode OVS: `OFPRAW_OFPT10_QUEUE_GET_CONFIG_REQUEST` (line 278), `OFPRAW_OFPT11_QUEUE_GET_CONFIG_REQUEST` (line 280), tương ứng `OFPRAW_OFPT10/11_QUEUE_GET_CONFIG_REPLY` (line 283/285). Handler `handle_queue_get_config_request()` đọc `port->qos->queues` và serialize từng `ofp_packet_queue` ra reply.

Trên OF 1.4+, một số controller cũ vẫn dùng OFPT_QUEUE_GET_CONFIG_REQUEST gây cảnh báo `deprecated message`; nên chuyển sang `OFPMP_QUEUE_DESC` (`ovs-ofctl dump-queues br0`).

## §3.5.7 Nhóm message peripheral bundle + async controller-coordination

Năm message dưới đây thuộc OF 1.4+ và phục vụ hai nhu cầu vận hành nâng cao: (i) thực thi atomic một loạt thay đổi cấu hình flow/group/meter qua **bundle** (`OFPT_BUNDLE_CONTROL` + `OFPT_BUNDLE_ADD_MESSAGE`), và (ii) phối hợp giữa nhiều controller chia sẻ cùng switch qua `OFPT_REQUESTFORWARD` + `OFPT_TABLE_STATUS`. Mỗi message được trình bày dưới dạng tra cứu kỹ thuật ngắn nhưng đủ để kỹ sư đọc được wire bytes, biết version drift và có lab kiểm chứng.

> **Đính chính so với phiên bản trước của tài liệu.** OF 1.5.1 spec không định nghĩa 4 type number riêng (33/34/35/36) cho từng action OPEN/COMMIT/ADD/DISCARD. Chỉ có **hai** type number: `OFPT_BUNDLE_CONTROL = 33` (mọi thao tác OPEN, CLOSE, COMMIT, DISCARD đều mang type=33 nhưng phân biệt qua sub-type field `OFPBCT_*` 16-bit trong body) và `OFPT_BUNDLE_ADD_MESSAGE = 34`. Xác nhận tại `include/openflow/openflow-1.4.h` v2.17.9 dòng 305-313 (`enum ofp14_bundle_ctrl_type`) và `include/openvswitch/ofp-msgs.h` dòng 282-289.

### §3.5.7.1 OFPT_BUNDLE_COMMIT — chốt giao dịch atomic

**Khái niệm.** Không phải một type number độc lập mà là **giá trị sub-type `OFPBCT_COMMIT_REQUEST = 4`** (hoặc `OFPBCT_COMMIT_REPLY = 5`) đặt trong field `type` của message `OFPT_BUNDLE_CONTROL` (OFPT type=33). Khi controller gửi BUNDLE_CONTROL với sub-type COMMIT_REQUEST, switch áp dụng nguyên tử toàn bộ message đã được tích luỹ trong bundle (`bundle_id` định danh) theo đúng thứ tự nhận, hoặc rollback toàn bộ nếu bất kỳ message nào trong bundle gây lỗi. Cơ chế này thay thế chuỗi BARRIER + retry thủ công mà OF 1.0-1.3 phải dùng để mô phỏng atomic group.

**Cơ chế hoạt động + wire layout.**

```
BUNDLE_CONTROL (OFPBCT_COMMIT_REQUEST), tổng 16 byte:
  04 21 00 10 00 00 00 20            ; version=4, type=33 (BUNDLE_CONTROL),
                                       ; len=16, xid=32
  00 00 00 07                        ; bundle_id = 7
  00 04                              ; sub-type = OFPBCT_COMMIT_REQUEST (4)
  00 03                              ; flags = OFPBF_ATOMIC|OFPBF_ORDERED (1|2 = 3)

Sub-type catalog (OFPBCT_*, openflow-1.4.h dòng 305-313):
  0  OPEN_REQUEST        4  COMMIT_REQUEST
  1  OPEN_REPLY          5  COMMIT_REPLY
  2  CLOSE_REQUEST       6  DISCARD_REQUEST
  3  CLOSE_REPLY         7  DISCARD_REPLY

Flags (OFPBF_*, openflow-1.4.h dòng 316-319):
  0x0001 OFPBF_ATOMIC   ; mọi message phải apply atomic; lỗi → rollback
  0x0002 OFPBF_ORDERED  ; thứ tự apply phải khớp thứ tự ADD; nếu thiếu → switch tự reorder
```

**Cách kỹ sư vận hành.** Sequence chuẩn: OPEN_REQUEST → N × BUNDLE_ADD_MESSAGE → COMMIT_REQUEST → đợi COMMIT_REPLY hoặc OFPT_ERROR. Trong OVS, lệnh CLI `ovs-ofctl --bundle add-flows br0 flows.txt` tự động bao toàn bộ FLOW_MOD trong cặp OPEN/COMMIT với cờ ATOMIC|ORDERED.

**Verification.** `ovs-ofctl -O OpenFlow14 --bundle add-flows br0 flows.txt`; quan sát qua `tcpdump -i any -nn 'tcp port 6653' -X` thấy ba burst: OPEN, ADD×N, COMMIT. Lỗi commit (ví dụ một FLOW_MOD trong bundle dùng table không tồn tại) trả về `OFPT_ERROR / OFPBFC_MSG_FAILED` và toàn bộ bundle bị rollback, có thể xác nhận qua `ovs-ofctl dump-flows br0` không thay đổi.

**Source code tham chiếu.** Decode tại `lib/ofp-bundle.c` v2.17.9 hàm `ofp_bundle_ctrl_msg_decode()`. Handler tại `ofproto/ofproto.c` qua `do_bundle_commit()` (gọi `ofproto_bundle_commit()` per-provider). Provider DPIF (`ofproto-dpif`) thực thi qua `dpif_operate()` batch single transaction.

### §3.5.7.2 OFPT_BUNDLE_ADD_MESSAGE — chèn message vào bundle đang mở

**Khái niệm.** OFPT type=34 độc lập với BUNDLE_CONTROL. Mỗi message này mang **một** message OpenFlow lồng bên trong (typical `OFPT_FLOW_MOD`, `OFPT_GROUP_MOD`, `OFPT_METER_MOD`, `OFPT_PORT_MOD`) cộng với `bundle_id` để switch biết append vào bundle nào. Inner message phải có `xid` trùng với outer xid để switch khớp lỗi nếu sub-message fail tại commit.

**Cơ chế hoạt động + wire layout.**

```
BUNDLE_ADD_MESSAGE bao một FLOW_MOD đơn giản:
  04 22 00 50 00 00 00 21            ; outer header: type=34, len=80, xid=33
  00 00 00 07                        ; bundle_id = 7 (must match OPEN's bundle_id)
  00 00                              ; flags (16-bit, reserved)
  00 00                              ; pad / property region start
  ; Inner ofp_flow_mod (OFPT type=14), len=64, xid=33 (must match outer xid):
  04 0E 00 40 00 00 00 21            ; version=4, type=14, len=64, xid=33
  ... (cookie, table_id, command=ADD, match, instructions) ...
```

**Vị trí + version drift.** Có từ OF 1.4. Trên OF 1.3, OVS hỗ trợ thông qua **ONF experimenter** `ONFT13_BUNDLE_ADD_MESSAGE` (experimenter ID 2301), được decode tại `OFPRAW_ONFT13_BUNDLE_ADD_MESSAGE` (ofp-msgs.h dòng 289). Từ OF 1.4 dùng standard `OFPRAW_OFPT14_BUNDLE_ADD_MESSAGE` (dòng 287). Inner message hợp lệ giới hạn trong nhóm cấu hình flow/group/meter/port; controller không được lồng PACKET_OUT, ECHO, hoặc message điều khiển khác.

**Verification + lab.** `ovs-ofctl -O OpenFlow14 --bundle replace-flows br0 large-flows.txt` (file 5000 dòng FLOW_MOD); quan sát ovs-vswitchd log `bundle ID X has Y messages, COMMIT atomic` ở `/var/log/openvswitch/ovs-vswitchd.log`. Đối chiếu `dump-flows` trước/sau commit không thấy trạng thái trung gian (không lúc nào table chỉ có 2500 flow trong khi đang push 5000).

**Lỗi thường gặp.** `OFPT_ERROR / OFPBFC_BAD_ID` nếu `bundle_id` không tồn tại (OPEN chưa gửi hoặc đã COMMIT/DISCARD); `OFPBFC_MSG_BAD_XID` nếu inner xid ≠ outer xid. Khắc phục: dùng helper `ovs-ofctl --bundle` đảm bảo cờ + xid đúng, không tự encode tay.

### §3.5.7.3 OFPT_BUNDLE_DISCARD — huỷ bundle không apply

**Khái niệm.** Sub-type `OFPBCT_DISCARD_REQUEST = 6` (reply = 7) trong cùng container `OFPT_BUNDLE_CONTROL` (type=33). Khi controller gửi DISCARD_REQUEST với một `bundle_id` đang mở, switch giải phóng toàn bộ message đã ADD vào bundle đó **không apply gì cả** và trả lời DISCARD_REPLY. Đây là cơ chế escape an toàn khi controller phát hiện trong quá trình build bundle (sau khi đã ADD vài chục message) là bundle không nên commit (ví dụ phát hiện conflict policy ở phía controller, mất kết nối tới upstream cần hold).

**Cơ chế hoạt động + wire layout.**

```
BUNDLE_CONTROL (OFPBCT_DISCARD_REQUEST), 16 byte:
  04 21 00 10 00 00 00 22            ; version=4, type=33, len=16, xid=34
  00 00 00 07                        ; bundle_id = 7 (đang ở trạng thái OPEN)
  00 06                              ; sub-type = OFPBCT_DISCARD_REQUEST (6)
  00 00                              ; flags (không bắt buộc; switch ignore)
```

**Vị trí + version drift.** OF 1.4+ standard. Trên OF 1.3 với extension `ONFT13_BUNDLE_CONTROL` (experimenter ID 2300, ofp-msgs.h dòng 284), DISCARD_REQUEST được encode trong cùng experimenter container.

**Verification + lab.** Trong test khung Mininet: gửi OPEN_REQUEST, ADD 5 FLOW_MOD, sau đó DISCARD_REQUEST; verify `dump-flows br0` không thấy flow nào được install. Đối chiếu `ovs-appctl -t ovs-vswitchd dpctl/dump-flows` cũng zero. Có thể trigger bằng API client (Ryu, OpenDaylight), không có CLI trực tiếp trong `ovs-ofctl`.

**Lỗi thường gặp.** Switch trả `OFPBFC_BAD_ID` nếu DISCARD trên bundle đã commit hoặc đã discard trước đó. Switch trả `OFPBFC_BUNDLE_IN_PROGRESS` nếu cố DISCARD trong khi commit đang chạy. Khắc phục: theo dõi state machine bundle phía controller (OPEN → ADD* → COMMIT|DISCARD → CLOSED), không gửi DISCARD lặp.

### §3.5.7.4 OFPT_REQUESTFORWARD — chuyển tiếp request giữa các controller

**Khái niệm.** OFPT type=32 (OF 1.4+, OVS dùng `OFPRAW_OFPT14_REQUESTFORWARD` ofp-msgs.h dòng 279). Async notification do switch sinh ra để **thông báo cho mọi controller có vai trò MASTER hoặc SLAVE** rằng một controller khác đã gửi `OFPT_GROUP_MOD` hoặc `OFPT_METER_MOD` thành công. Mục đích: trong setup đa controller (auxiliary connection / equal-role multi-master OF 1.5), các controller khác cần biết để cập nhật state cache nội bộ, tránh tình huống controller A thêm group_id=10, controller B không biết và cũng cố add group_id=10 gây `OFPGMFC_GROUP_EXISTS`.

**Cơ chế hoạt động + wire layout.**

```
REQUESTFORWARD bao một GROUP_MOD ADD đơn giản:
  04 20 00 30 00 00 00 23            ; outer header: type=32, len=48, xid=35
                                       ; (xid không bắt buộc khớp inner)
  ; Inner ofp_header (mọi message OF) — ở đây là GROUP_MOD type=15:
  04 0F 00 28 00 00 00 11            ; version=4, type=15, len=40, xid=17
  ... (GROUP_MOD body: command=ADD, type=ALL, group_id=10, bucket list) ...

Reason hợp lệ (struct ofp14_requestforward.request, openflow-1.4.h dòng 269-272):
  OFPRFR_GROUP_MOD = 0   ; inner là GROUP_MOD
  OFPRFR_METER_MOD = 1   ; inner là METER_MOD
```

**Cách kỹ sư vận hành.** Switch tự sinh REQUESTFORWARD khi: (a) GROUP_MOD/METER_MOD ADD/MODIFY/DELETE thành công, (b) message gốc đến từ controller có vai trò khác **MASTER** không phải sender hiện tại (tránh echo). Controller nhận REQUESTFORWARD chỉ đọc inner request để cập nhật cache; KHÔNG được tự apply lần nữa.

**Verification + lab.** Yêu cầu hai controller (Ryu instance A + B) cùng kết nối br0, A vai trò MASTER. A gửi GROUP_MOD ADD. B nhận REQUESTFORWARD chứa inner GROUP_MOD đó. Quan sát qua `tcpdump -i any -nn 'tcp port 6653 and (host controller-a or host controller-b)' -X`. Xác nhận handler decode: `lib/ofp-msgs.c` ánh xạ vào `OFPTYPE_REQUESTFORWARD` (ofp-msgs.h dòng 647-649).

**Lỗi thường gặp.** Single-controller setup (OVN baseline, ovn-controller per-chassis) **không bao giờ thấy** REQUESTFORWARD vì chỉ có một MASTER duy nhất, switch không có đối tượng để forward tới. Nếu controller nhận REQUESTFORWARD trong setup được tin là single-controller, kiểm tra OVSDB `Bridge.controller` xem có sót entry cũ không.

### §3.5.7.5 OFPT_TABLE_STATUS — async cảnh báo trạng thái flow table

**Khái niệm.** OFPT type=31 (OF 1.4+). Async notification từ switch lên controller khi vacancy (số phần trăm slot trống trong flow table) băng qua một trong hai ngưỡng `vacancy_down`/`vacancy_up` đã được controller cấu hình trước qua `OFPT_TABLE_MOD` với property `OFPTMPT14_VACANCY` (xem §3.5.6.4). Reason mã hoá hướng băng:

- `OFPTR_VACANCY_DOWN = 3` — vacancy giảm xuống dưới `vacancy_down` (table sắp đầy, controller nên dừng push thêm rule và cân nhắc evict).
- `OFPTR_VACANCY_UP = 4` — vacancy tăng vượt `vacancy_up` (table có chỗ trở lại, controller có thể push tiếp).

**Cơ chế hoạt động + wire layout.**

```
TABLE_STATUS reason=OFPTR_VACANCY_DOWN:
  04 1F 00 18 00 00 00 24            ; version=4, type=31, len=24, xid=36
                                       ; (xid 0 hợp lệ vì là async)
  ; struct ofp14_table_status (8 byte):
  03                                  ; reason = OFPTR_VACANCY_DOWN (3)
  00 00 00 00 00 00 00                ; pad 7 byte
  ; Followed by struct ofp14_table_desc (8 byte):
  00 08 00 01                        ; length=8, table_id=0, pad 1
  00 00 00 00                        ; config bitmap

(openflow-1.4.h dòng 167-172 + 156-165)
```

**Cách kỹ sư vận hành.** Workflow đầy đủ: (1) controller chọn ngưỡng (ví dụ `vacancy_down=10%`, `vacancy_up=25%`); (2) gửi TABLE_MOD với `OFPTMPT14_VACANCY` property; (3) bật cờ `OFPTC14_VACANCY_EVENTS` (`1<<3`) trong `config` của TABLE_MOD; (4) lắng nghe TABLE_STATUS async trong loop chính. OVS hỗ trợ decode/encode nhưng **chưa sinh tự động** sự kiện vacancy trong nhánh 2.17 mainline (xác nhận qua grep `vacancy_event` trong `ofproto/ofproto.c` không có call site sinh event).

**Verification + lab.** Trên switch hỗ trợ vacancy event đầy đủ (commercial silicon vendor ONS reference): push flow rule cho đến khi `dump-table-features` báo vacancy còn 8% (dưới 10%) — lập tức nhận TABLE_STATUS trên kênh 6653. Lab Mininet với OVS branch-2.17 sẽ không thấy event tự sinh; có thể test decode bằng cách inject byte stream qua `ovs-ofctl ofp-print`.

**Lỗi thường gặp.** Không nhận TABLE_STATUS dù table đã đầy: kiểm tra (a) đã set `vacancy_down/up` chưa, (b) đã bật `OFPTC14_VACANCY_EVENTS` chưa, (c) phiên bản OpenFlow đã thoả thuận có ≥ 1.4 không (HELLO bitmap). Trên OVS branch-2.17, kết luận realistic là **dùng OFPMP_TABLE_STATS poll định kỳ** thay async event cho đến khi upstream OVS implement đầy đủ vacancy event generator.

**Tầm quan trọng.** Cơ chế truy vấn thống kê chính của OpenFlow. Mọi monitoring/observability đều dựa vào nhóm này.

**Khác biệt giữa các phiên bản.** OF 1.0 nền tảng (FLOW, PORT_STATS, AGGREGATE...). OF 1.3 bổ sung Group/Meter family. Cấu trúc message ổn định.

**Trường hợp sự cố thực tế.** OFPMP_FLOW dùng để theo dõi packet count per rule; OFPMP_PORT_STATS dùng để theo dõi tải interface.

**Bài tập synthetic.** Chạy `ovs-ofctl dump-flows br0` (OFPMP_FLOW) và `ovs-ofctl dump-ports br0` (OFPMP_PORT_STATS), so sánh kết quả với output của tcpdump.

**So sánh với hệ khác.** Tương tự counter retrieval qua SNMP MIB OID; tương tự gNMI Get OperationalState.

### §3.5.8.1 OFPMP_DESC (subtype 0) — switch description

**Khái niệm.** OFPMP_DESC là multipart sub-type giúp controller lấy thông tin self-describe của switch: nhà sản xuất, phần cứng, phần mềm, số serial và mô tả datapath. Subtype integer = 0 (xác nhận tại `include/openvswitch/ofp-msgs.h` v2.17.9, dòng `OFPRAW_OFPST_DESC_REQUEST` chú thích `"OFPST 1.0+ (0): void"`). Đây là phương tiện chuẩn để controller phân biệt vendor stack ngay sau handshake.

**Vai trò + vì sao có.** Trước khi push policy, controller thường cần biết switch là OVS, hay Pica8, hay HPE Aruba, hay silicon ASIC nào. Capabilities bitmap trong FEATURES_REPLY chỉ cho biết "có hỗ trợ FLOW_STATS hay không"; còn vendor + version cụ thể cần OFPMP_DESC. OF 1.0 đã có OFPST_DESC; OF 1.3 đổi tên thành OFPMP_DESC (cùng subtype 0).

**Cấu trúc message + byte layout.** Request body **rỗng** (`void` per ofp-msgs.h). Reply body là `struct ofp_desc_stats` (định nghĩa tại `include/openflow/openflow-common.h` v2.17.9, dòng 564-571 đã verify):

```c
struct ofp_desc_stats {
    char mfr_desc[DESC_STR_LEN];     /* 256 byte */
    char hw_desc[DESC_STR_LEN];      /* 256 byte */
    char sw_desc[DESC_STR_LEN];      /* 256 byte */
    char serial_num[SERIAL_NUM_LEN]; /* 32 byte */
    char dp_desc[DESC_STR_LEN];      /* 256 byte */
};
/* DESC_STR_LEN = 256, SERIAL_NUM_LEN = 32 → tổng body = 1056 byte */
```

Cộng với `struct ofp_header` (8 byte) + `struct ofp_multipart_request/reply` header (8 byte cho type+flags+pad), reply DESC chiếm 1072 byte trên dây. Reply không bao giờ phân mảnh (không cần OFPMPF_REPLY_MORE).

**Quy trình sử dụng + sample output.** Trên OVS:

```text
$ sudo ovs-ofctl -O OpenFlow13 dump-desc br-int
OFPST_DESC reply (xid=0x2):
Manufacturer: Nicira, Inc.
Hardware: Open vSwitch
Software: 2.17.9
Serial Num: None
DP Description: br-int
```

Mỗi field strip null padding. `mfr_desc = "Nicira, Inc."` là hard-code trong `vswitchd/bridge.c` của OVS userspace. `serial_num` thường rỗng trên OVS software switch nhưng có giá trị thật trên hardware (Pica8, NoviSwitch).

**Khi xảy ra sự cố.** Nếu reply bị truncate (vẫn 1056 byte body nhưng strings ngắn bất thường), nguyên nhân thường là switch firmware cũ chưa fill đủ; controller phải xử lý field như C-string không tin tưởng vào trailing data. Một số legacy controller crash khi `dp_desc` chứa byte non-ASCII; OVS hard-code ASCII-safe nên hiếm gặp.

**Phiên bản.** OF 1.0/1.1/1.2/1.3/1.4/1.5 đều giữ subtype 0 + struct giống hệt. Đây là một trong số ít multipart sub-type hoàn toàn không thay đổi qua các phiên bản.

### §3.5.8.2 OFPMP_AGGREGATE (subtype 2) — aggregate flow stats

**Khái niệm.** OFPMP_AGGREGATE trả về **tổng cộng dồn** packet count + byte count + flow count cho tất cả flow entry khớp một bộ tiêu chí (table, port, group, match). Khác với OFPMP_FLOW (liệt kê chi tiết từng flow), AGGREGATE chỉ trả ra ba con số. Subtype integer = 2 (xác nhận `OFPRAW_OFPST10_AGGREGATE_REQUEST` chú thích `"OFPST 1.0 (2)"`).

**Vì sao tách riêng.** Controller giám sát chỉ cần "tổng traffic qua bridge" thường không cần dump hàng nghìn flow. AGGREGATE tiết kiệm: switch tự sum trên kernel/userspace, gửi 24 byte body thay vì N × ~88 byte FLOW reply. Dùng cho dashboard "tổng pps/bps" theo nhóm flow.

**Cấu trúc message.** Request giống OFPMP_FLOW: `struct ofp10_flow_stats_request` (OF 1.0) hoặc `struct ofp11_flow_stats_request` (OF 1.1+) cộng match. Reply là `struct ofp_aggregate_stats_reply` (`include/openflow/openflow-common.h` v2.17.9 dòng 576-581):

```c
struct ofp_aggregate_stats_reply {
    ovs_32aligned_be64 packet_count;  /* 8 byte */
    ovs_32aligned_be64 byte_count;    /* 8 byte */
    ovs_be32 flow_count;              /* 4 byte */
    uint8_t pad[4];                   /* 4 byte alignment */
};
/* tổng body = 24 byte */
```

Padding 4 byte cuối cần thiết vì counter 64-bit phải align 8-byte.

**Sample output.**

```text
$ sudo ovs-ofctl -O OpenFlow13 dump-aggregate br0
OFPST_AGGREGATE reply (xid=0x4): packet_count=1234567 byte_count=987654321 flow_count=42
```

Ba con số này là **sum trên match criteria được gửi**; nếu request body để wildcard hoàn toàn thì là tổng toàn bridge.

**Khi xảy ra sự cố.** Nếu `packet_count` đứng yên giữa hai poll trong khi traffic chạy thực tế, kiểm tra: (a) flow nào đang khớp (có thể default drop chứ không phải user flow), (b) datapath có evict megaflow trước khi revalidator sync stats không (xem Phần 9.2 megaflow). Một bẫy nữa: AGGREGATE chỉ đếm packet **đã match flow**; packet bị table-miss và đi default rule có thể không nằm trong scope nếu rule mặc định không thuộc nhóm match.

**Phiên bản.** Subtype 2 ổn định OF 1.0 đến 1.4. OF 1.5 (commit `OFPRAW_OFPST15_AGGREGATE_REQUEST`) đổi semantic cho phép request stats theo OXM-only filter, nhưng wire format reply vẫn là 24-byte struct cũ.

### §3.5.8.3 OFPMP_PORT_STATS (subtype 4) — per-port counter

**Khái niệm.** OFPMP_PORT_STATS trả về counter packet/byte rx/tx + dropped + error + collision cho mỗi port (hoặc một port cụ thể nếu request giới hạn). Subtype integer = 4 (xác nhận `OFPRAW_OFPST10_PORT_REQUEST` chú thích `"OFPST 1.0 (4)"`). Đây là nguồn dữ liệu chính cho monitoring interface tại tầng OpenFlow (so sánh với `ip -s link` ở tầng kernel hoặc SNMP `ifInOctets`).

**Cấu trúc message OF 1.1+.** Reply array gồm `struct ofp11_port_stats` (verify từ `include/openflow/openflow-1.1.h`):

```c
struct ofp11_port_stats {
    ovs_be32 port_no;       /* 4 byte */
    uint8_t  pad[4];        /* 4 byte */
    ovs_be64 rx_packets;    /* 8 byte */
    ovs_be64 tx_packets;    /* 8 byte */
    ovs_be64 rx_bytes;      /* 8 byte */
    ovs_be64 tx_bytes;      /* 8 byte */
    ovs_be64 rx_dropped;    /* 8 byte */
    ovs_be64 tx_dropped;    /* 8 byte */
    ovs_be64 rx_errors;     /* 8 byte */
    ovs_be64 tx_errors;     /* 8 byte */
    ovs_be64 rx_frame_err;  /* 8 byte */
    ovs_be64 rx_over_err;   /* 8 byte */
    ovs_be64 rx_crc_err;    /* 8 byte */
    ovs_be64 collisions;    /* 8 byte */
};
/* tổng = 104 byte/port */
```

OF 1.3 wrap thêm `struct ofp13_port_stats { struct ofp11_port_stats ps; ovs_be32 duration_sec; ovs_be32 duration_nsec; }` (112 byte/port). Reply có thể phân mảnh qua flag `OFPMPF_REPLY_MORE = 0x0001` khi switch nhiều port (≥ 580 port với MTU 65535 byte cho mỗi reply chunk).

**Sample output.**

```text
$ sudo ovs-ofctl -O OpenFlow13 dump-ports br0
OFPST_PORT reply (xid=0x2): 3 ports
  port  LOCAL: rx pkts=12345, bytes=987654, drop=0, errs=0, frame=0, over=0, crc=0
           tx pkts=23456, bytes=1234567, drop=0, errs=0, coll=0
  port  1: rx pkts=100, bytes=8400, drop=0, errs=0, frame=0, over=0, crc=0
           tx pkts=98, bytes=8232, drop=0, errs=0, coll=0
  port  2: rx pkts=99, bytes=8316, drop=0, errs=0, frame=0, over=0, crc=0
           tx pkts=100, bytes=8400, drop=0, errs=0, coll=0
```

**Khi xảy ra sự cố.** `collisions` luôn = 0 trên virtual port (veth, internal); nếu thấy giá trị non-zero, đó là switch hardware báo half-duplex CSMA/CD collision. `rx_dropped` tăng nhanh thường do socket buffer overflow ở userspace controller; `tx_dropped` thường do queue đầy hoặc port admin-down. So sánh `dump-ports` với `ip -s link show <iface>` để cross-check kernel counter, lệch nhau nghĩa là OVS datapath chưa flush stats lên ofproto-dpif (xem revalidator timing Phần 9.2).

**Phiên bản.** OF 1.0 dùng `struct ofp10_port_stats` (cùng nội dung nhưng port_no 16-bit). OF 1.1 mở rộng port_no lên 32-bit. OF 1.3 thêm duration. OF 1.4+ giữ format giống 1.3.

### §3.5.8.4 OFPMP_GROUP (subtype 6) — group stats

**Khái niệm.** OFPMP_GROUP trả về stats cho từng group entry (group_id, ref_count, packet_count, byte_count, plus per-bucket counter). Subtype integer = 6 (xác nhận `OFPRAW_OFPST11_GROUP_REQUEST` chú thích `"OFPST 1.1+ (6)"`). Group là OF 1.1+ feature, do đó OFPMP_GROUP không tồn tại trên OF 1.0.

**Cấu trúc message.** Request body = `struct ofp11_group_stats_request { ovs_be32 group_id; uint8_t pad[4]; }`; group_id = `OFPG_ALL = 0xfffffffc` để xin tất cả. Reply là array `struct ofp11_group_stats`:

```c
struct ofp11_group_stats {
    ovs_be16 length;        /* 2 byte: tổng size entry này */
    uint8_t  pad[2];        /* 2 byte */
    ovs_be32 group_id;      /* 4 byte */
    ovs_be32 ref_count;     /* 4 byte: số flow tham chiếu */
    uint8_t  pad2[4];       /* 4 byte */
    ovs_be64 packet_count;  /* 8 byte */
    ovs_be64 byte_count;    /* 8 byte */
    /* + array struct ofp11_bucket_counter[] */
};
/* base = 32 byte; per-bucket = 16 byte (packet_count + byte_count) */
```

OF 1.3+ wrap thành `struct ofp13_group_stats` thêm 8 byte duration (40 byte base).

**Sample output.**

```text
$ sudo ovs-ofctl -O OpenFlow13 dump-group-stats br-int
OFPST_GROUP reply (OF1.3) (xid=0x2):
 group_id=100, ref_count=2, packet_count=5000, byte_count=420000, duration=120.456s
   bucket0: packet_count=2500, byte_count=210000
   bucket1: packet_count=2500, byte_count=210000
```

`ref_count = 2` nghĩa là 2 flow entry có action `group:100`. Per-bucket counter cho thấy ECMP/select group đang phân phối đều giữa hai bucket.

**Khi xảy ra sự cố.** Nếu một bucket có `packet_count = 0` trong khi bucket khác tăng đều, đó là dấu hiệu hash function của select group đang lệch (hash chỉ dựa trên 5-tuple cố định + một flow duy nhất). Khắc phục: dùng `ovs-ofctl mod-group ... selection_method=dp_hash` (OVS extension) để switch sang hash đa dạng hơn. Nếu `ref_count = 0` mà group vẫn tồn tại, đó là group orphan có thể `del-group` an toàn.

**Phiên bản.** OF 1.1/1.2 dùng base 32 byte. OF 1.3+ thêm duration thành 40 byte. Ref counting trên OVS được implement tại `ofproto/ofproto.c` thông qua `struct ofgroup::ref_count`.

### §3.5.8.5 OFPMP_METER (subtype 9) — meter stats

**Khái niệm.** OFPMP_METER trả về stats cho từng meter (rate-limit / DSCP-remark policer): meter_id, flow_count tham chiếu, packet/byte đã đi qua, plus per-band drop counter. Subtype integer = 9 (xác nhận `OFPRAW_OFPST13_METER_REQUEST` chú thích `"OFPST 1.3+ (9)"`). Meter là OF 1.3+ feature; OFPMP_METER không có trên OF 1.0/1.1/1.2.

**Cấu trúc message.** Request body = `struct ofp13_meter_multipart_request { ovs_be32 meter_id; uint8_t pad[4]; }`; meter_id = `OFPM_ALL = 0xffffffff` để xin tất cả. Reply array:

```c
struct ofp13_meter_stats {
    ovs_be32 meter_id;          /* 4 byte */
    ovs_be16 len;               /* 2 byte: size entry này */
    uint8_t  pad[6];            /* 6 byte */
    ovs_be32 flow_count;        /* 4 byte: số flow tham chiếu */
    ovs_be64 packet_in_count;   /* 8 byte: tổng packet vào meter */
    ovs_be64 byte_in_count;     /* 8 byte */
    ovs_be32 duration_sec;      /* 4 byte */
    ovs_be32 duration_nsec;     /* 4 byte */
    struct ofp13_meter_band_stats band_stats[0]; /* per-band: 16 byte/band */
};
/* base = 40 byte; per band = 16 byte (packet_band_count + byte_band_count) */
```

Verify từ `include/openflow/openflow-1.3.h` v2.17.9 (đã dump bên trên).

**Sample output.**

```text
$ sudo ovs-ofctl -O OpenFlow13 meter-stats br-int
OFPST_METER reply (OF1.3) (xid=0x2):
meter:1 flow_count:5 packet_in_count:10000 byte_in_count:840000 duration:60.123s bands:
  0: packet_count:200 byte_count:16800
```

`packet_in_count` là tổng packet đi qua meter; `band[0].packet_count` là số packet bị drop / remark bởi band thứ nhất (vượt rate threshold). Tỉ lệ `band[0].packet_count / packet_in_count = 2%` cho thấy 2% traffic vượt rate.

**Khi xảy ra sự cố.** Trên OVS branch-2.17 software meter (kernel datapath), nếu `packet_in_count` đứng yên trong khi user thấy traffic, kiểm tra: (a) flow đã có action `meter:N` chưa, (b) `ovs-ofctl dump-meters br0` xem meter có thật sự cấu hình rate hợp lý không (kbps 0 = no-op). OVS DPDK datapath không hỗ trợ đầy đủ meter trước v2.13; nếu chạy DPDK cũ thì meter stats có thể luôn = 0 dù flow match.

**Phiên bản.** Meter ra đời OF 1.3, struct stable đến OF 1.5. OVS support OF 1.3 meter từ v2.0; per-band stats từ v2.7. Pinned baseline curriculum: OVS v2.17.9 + OF 1.3.

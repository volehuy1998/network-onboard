## 9.5.1 Why hardware offload is needed at 25G or 100G and above

Software OVS reaches good throughput at 10G. With NICs at 25G, 100G, 200G or higher, however, the CPU becomes the bottleneck:

- At 100 Gbps with 64-byte packets, the packet rate is about 148 Mpps. Each packet costs about 6 to 10 CPU cycles even with DPDK optimization.
- With an Intel Xeon at about 3 GHz, one core processes at most about 500 Mpps in theory, in practice 15 to 30 Mpps. You would need 5 to 10 cores just to match the line rate of 100G, which is not feasible at cloud scale.

The solution is to **offload the data plane to the NIC hardware**, while keeping the control plane on the CPU. Smart NICs (ConnectX-5 and newer) and DPUs (BlueField-2 and BlueField-3) have an embedded switch (eSwitch) with hardware forwarding capability. OpenFlow rules are translated into rules in the eSwitch, and the hardware handles packet matching and actions without the CPU touching the packet.

The WASTC 2021 slides (`doc/ovs/Day 4- Motivation and Introduction to Open vSwitch`) state the context: "closed-design models have limited the capability of the switches. The process of updating devices is lengthy and costly." SDN with OpenFlow is the programmable answer, but it must work with hardware that already exists. Hardware offload is the bridge between SDN flexibility and hardware performance.

## 9.5.2 The Linux switchdev framework: VF representor

Linux kernel switchdev is the framework for managing a NIC that has an embedded switch. The core concept is the **VF representor**: a kernel netdev that represents one Virtual Function inside the eSwitch of the NIC.

When the NIC runs in **legacy SR-IOV mode**, each VF is an independent netdev, and the VF is passed through to the VM. There is no representor.

When the NIC runs in **switchdev mode**, each VF is still passed through to the VM (the real VF netdev), but **in addition** a representor netdev (`pf0vf0`) is created on the host. OVS (or tc flower) configures rules on the representor, and the rules are pushed down to the eSwitch hardware.

Switching to switchdev mode (compass Ch 16):

```bash
[host]# echo 4 > /sys/class/net/ens1f0/device/sriov_numvfs
[host]# for vf in $(ls -d /sys/class/net/ens1f0/device/virtfn*); do
          pci=$(basename $(readlink $vf))
          echo $pci > /sys/bus/pci/drivers/mlx5_core/unbind 2>/dev/null
        done
[host]# devlink dev eswitch set pci/0000:3b:00.0 mode switchdev
```

Verification:

```bash
[host]# ip -d link show pf0vf0_r
24: pf0vf0_r: <BROADCAST,MULTICAST> mtu 1500 ...
    link/ether ... switchid 00008000 portname pf0vf0
```

The `switchid` and `portname pf0vf0` fields are the signs that the interface is a switchdev representor.

## 9.5.3 NVIDIA ASAP² and eSwitch offload

**ASAP²** (Accelerated Switching and Packet Processing) is the NVIDIA brand name for eSwitch offload technology on ConnectX-5 and newer cards, and on the BlueField series. The core idea: the eSwitch hardware inside the NIC handles VXLAN encapsulation and decapsulation, L2 and L3 forwarding, ACL, and conntrack at line rate.

Architecture:

```
┌────────────────────────────────────────────────────┐
│   CPU                                               │
│   ┌──────────────┐                                 │
│   │ ovs-vswitchd │  (control plane, unchanged)    │
│   └──────┬───────┘                                 │
│          │ tc flower / DOCA Flow                   │
│          ▼                                          │
│   ┌──────────────┐                                 │
│   │ mlx5_core    │                                 │
│   └──────┬───────┘                                 │
└──────────┼──────────────────────────────────────────┘
           │ (driver queue)
┌──────────┴─────────────────────────────────────────┐
│  NIC subsystem (ConnectX-6 / BlueField)            │
│  ┌──────────────────────────────────────┐          │
│  │  eSwitch (hardware packet forward)   │          │
│  │   - match: VXLAN VNI, inner 5-tuple │          │
│  │   - action: encap/decap, forward    │          │
│  └──────────────────────────────────────┘          │
└────────────────────────────────────────────────────┘
```

Enable offload:

```bash
[host]# ovs-vsctl set Open_vSwitch . other_config:hw-offload=true
[host]# systemctl restart openvswitch-switch
```

Verification:

```bash
[host]# ovs-appctl dpctl/dump-flows -m type=offloaded
[host]# tc -s filter show dev pf0vf0_r ingress
```

The output shows rules that have been offloaded to the hardware along with their counters. This is direct evidence that the hardware is forwarding the traffic, not the CPU.

## 9.5.4 The three OVS DPIFs: Kernel, DPDK, and DOCA

*Table 9-5: Comparison of the three OVS DPIFs*

| DPIF | Mechanism | Module | When to use |
|---|---|---|---|
| **Kernel** | `openvswitch.ko` plus tc-flower for hardware offload | in-tree kernel module | Typical workloads on any Linux NIC |
| **DPDK** | userspace PMD plus `datapath_type=netdev` | OVS-DPDK package | Very high throughput when NIC offload is not needed |
| **DOCA** | NVIDIA DOCA Flow library plus eSwitch offload | `ovs-doca` package | NVIDIA ConnectX-6 and newer, BlueField-2 and newer |

NVIDIA's official guidance: **"The DPDK and Kernel DPIFs are maintained mainly for backward compatibility. They have not received new features since 2023."** Every new feature (extended match fields, CT plus NAT offload, VXLAN encapsulation variants) is added only to the DOCA DPIF.

A new deployment on NVIDIA NICs should use OVS-DOCA from the start.

## 9.5.5 OVS-DOCA: the DOCA Flow library

The DOCA Flow library is an API NVIDIA provides to program the eSwitch directly. The OVS-DOCA package wraps this library into a new DPIF.

Enable:

```bash
[host]# ovs-vsctl set Open_vSwitch . other_config:doca-init=true
[host]# ovs-vsctl set Open_vSwitch . other_config:hw-offload=true
[host]# systemctl restart openvswitch-switch
```

DOCA Flow pre-allocates offload containers and entries at startup. The default congestion threshold is 80 percent. When the container is 80 percent full, DOCA resizes it.

Monitoring:

```bash
[host]# ovs-appctl coverage/show | grep doca
doca_async_queue_full       ...
doca_pipe_resize            ...
doca_pipe_resize_over_10_ms ...
```

- `doca_async_queue_full` greater than 0 means the hardware is accepting rules slowly.
- `doca_pipe_resize_over_10_ms` greater than 0 means the resize took more than 10 ms, which can drop packets.

Current limitations of the DOCA DPIF: it does not support complex CT-CT-NAT chains, and it does not fully support OpenFlow meters (only basic srTCM).

## 9.5.6 Offload feature coverage

Features that are offloaded successfully:

- **VXLAN encap and decap**: the br-phy plus br-ovs pattern; VXLAN tunnel encapsulation in the eSwitch.
- **Conntrack**: connection tracking in hardware. The table holds about 2 million connections.
- **CT plus NAT**: SNAT and DNAT with conntrack. Exception: ICMP NAT is not supported.
- **VF-LAG**: bonding two NIC ports with active-backup, XOR, or LACP, all offloaded.
- **OpenFlow meters**: RFC 2697 srTCM basic.
- **sFlow**: sampling offload (only the internal bridge when VXLAN is in use).
- **Mirror**: with `doca-mirror-max=2048`.

NOT offloaded (software fallback):

- Complex OpenFlow actions (`learn`, conjunction, multi-path hash).
- Custom conntrack timeout policies.
- OXM experimenter fields.

Compass: "the megaflow cache in software still catches them; the hardware just does not match."

## 9.5.7 Steering modes: SMFS versus DMFS

- **SMFS** (Software-Managed Flow Steering), the default on ConnectX-6 and newer. It is optimized for high insertion rate and is recommended when rules change frequently.
- **DMFS** (Device-Managed Flow Steering), firmware-based. It is optimized for throughput when rules are static, but the insertion rate is lower.

Configuration (must be set before switching to switchdev mode):

```bash
[host]# devlink dev param set pci/0000:3b:00.0 name flow_steering_mode \
         value smfs cmode runtime
```

**vPort match modes:**

- **Metadata** (default): supports SR-IOV live migration and dual-port RoCE.
- **Legacy**: about 5 to 10 percent higher performance, but does not support live migration.

Production guidance: VMs that use live migration go with Metadata. Static workloads where throughput is critical go with Legacy plus DMFS.

## 9.5.8 vDPA: virtio portability with SR-IOV performance

vDPA combines SR-IOV performance with virtio portability. The VM image is not tied to a vendor driver.

- **Software vDPA**: translation on the host.
- **Hardware vDPA**: the NIC supports it directly (ConnectX-6 Dx and newer).

Use case: a cloud provider needs a generic VM image that can migrate between servers with different hardware while still reaching SR-IOV throughput.

---

## 9.5.9 BlueField DPU architecture

BlueField (2 and 3) is a DPU, a NIC with embedded Arm cores.

- Host (x86): runs application workloads.
- DPU (Arm): runs infrastructure services (OVS-DOCA, firewall, storage offload).
- eSwitch: hardware forwarding between the host PF and the DPU VF.

The interface `pf0hpf` is the host PF representor on the DPU side. BFB (BlueField Bundle) installation: NVIDIA provides a BFB image that contains Ubuntu, the DOCA SDK, and OVS-DOCA. The script `/sbin/mlnx_bf_configure` automatically enables hardware offload.

Use case: isolating tenant workloads (host) from infrastructure services (DPU Arm cores).

## 9.5.10 Megaflow scaling in OVS-DOCA

Tuning at large scale:

```bash
[host]# ovs-vsctl set Open_vSwitch . other_config:flow-limit=500000
[host]# ovs-vsctl set Open_vSwitch . other_config:hw-offload-ct-size=1000000
[host]# ovs-vsctl set Open_vSwitch . other_config:max-revalidator=250
[host]# sysctl -w net.netfilter.nf_conntrack_max=2097152
```

- `flow-limit` defaults to 200k, with a maximum of 2M.
- `hw-offload-ct-size` defaults to 250k, with a maximum of 2M connections (a warning is issued above 1M).
- `max-revalidator` defaults to 250 ms.
- `nf_conntrack_max` must be kept consistent with `hw-offload-ct-size`.

### Guided Exercise 6: Verify switchdev mode and the VF representor

**Outcomes:** Convert a NIC to switchdev mode, verify that the VF representor is created.

**Before You Begin:** A Mellanox ConnectX-4 Lx or newer NIC with `mlx5_core`.

**Instructions:**

**1.** Create 4 VFs:

```bash
[host]# echo 4 > /sys/class/net/ens1f0/device/sriov_numvfs
```

**2.** Unbind the VF drivers and switch to switchdev mode:

```bash
[host]# for vf in $(ls -d /sys/class/net/ens1f0/device/virtfn*); do
          pci=$(basename $(readlink $vf))
          echo $pci > /sys/bus/pci/drivers/mlx5_core/unbind 2>/dev/null
        done
[host]# devlink dev eswitch set pci/0000:3b:00.0 mode switchdev
```

**3.** Verify the representors:

```bash
[host]# ip -d link show | grep pf0vf
```

The output must show `pf0vf0_r`, `pf0vf1_r`, and so on, each with a `switchid` and a `portname`.

**4.** Rebind the driver:

```bash
[host]# for vf in $(ls -d /sys/class/net/ens1f0/device/virtfn*); do
          pci=$(basename $(readlink $vf))
          echo $pci > /sys/bus/pci/drivers/mlx5_core/bind
        done
```

**Finish:** The VFs are ready to be passed through to a VM or plugged into `br-doca`.

## Key points to remember

First, hardware offload is needed when the NIC speed is 25G or higher and the CPU cost per packet becomes the bottleneck. Smart NICs (ConnectX-5 and newer) and DPUs (BlueField) have embedded eSwitches.

Second, Linux switchdev is the standard kernel API for hardware offload. The VF representor netdev is the front face of an eSwitch port. OVS configures rules on the representor, and the rules are pushed down to the hardware through tc-flower or DOCA Flow.

Third, the three OVS DPIFs serve different use cases. NVIDIA recommends the DOCA DPIF for its NICs. The Kernel and DPDK DPIFs are kept for backward compatibility and do not receive new features.

Fourth, the choice between SMFS and DMFS, and between Metadata and Legacy vPort, is an important tuning decision. VMs with live migration go with Metadata. Static workloads where throughput is critical go with Legacy plus DMFS.

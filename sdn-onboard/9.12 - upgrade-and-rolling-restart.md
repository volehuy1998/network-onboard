## 9.12.1 Concept

`Upgrade choreography` is the production procedure for changing the OVS userspace version, the kernel module, and the OVSDB schema **at the same time** across a fleet of hosts WITHOUT dropping data-plane traffic. It is different from a normal package update (`apt upgrade` plus restart): the rolling restart coordinates three components (`ovsdb-server`, `ovs-vswitchd`, `openvswitch.ko`) in a strict order so that fast-path forwarding continues even while the daemons are restarting. The category is operational procedure plus state-machine choreography (it is not a CLI tool, and it is not an OVSDB schema row).

---

## 9.12.2 History and background

OVS 1.x (2009 to 2013) required a host reboot when the kernel module changed, so every VM lost connectivity for several seconds. OVS 2.0 (Jan 2014) introduced `--monitor` for `ovs-vswitchd` graceful restart (auto-respawn on crash), reducing the restart downtime from seconds to sub-second. OVS 2.6 (Sep 2016) added OVSDB Raft clustering with `cluster/leave` and `ovsdb-tool join-cluster`, enabling per-node rolling cluster upgrades. OVS 2.10+ (around 2018) refined schema-conversion atomicity. OVS 2.13 (Feb 2020) optimized megaflow eviction to minimize cache invalidation during reload. OVS 2.17.0 (17 Feb 2022) is currently the popular LTS for OpenStack Yoga and Zed and for OVN 22.03. OVS 3.0.0 (15 Aug 2022) changed the container iteration macros to be UB-safe and added OVSDB relay transaction history; the change is ABI compatible but client recompilation is required. The need for rolling upgrades comes from production reality: an OpenStack or OVN cluster with 100 or more compute nodes cannot restart the entire fleet at the same time (loss of quorum, loss of tenant traffic).

---

## 9.12.3 Position in the architecture

Upgrade choreography is a cross-cutting concern that affects all three OVS components at the same time:

- **OVSDB layer (`ovsdb-server`):** schema conversion and state migration; persisted in `/etc/openvswitch/conf.db`.
- **Userspace daemon (`ovs-vswitchd`):** process lifecycle and `ofproto` state restore; reloads from OVSDB after restart.
- **Kernel module (`openvswitch.ko`):** kernel data plane; reloading the module affects the megaflow cache but does not affect conntrack state (state lives in the netfilter `nf_conntrack` table independently).

In the stack lifecycle, upgrade choreography sits at the operational-procedure layer above the binary artifacts and file system artifacts (package manager, systemd unit, kernel module file).

---

## 9.12.4 Role

Upgrade choreography plays four main roles:

1. **Atomic version transition.** Ensures that the new `ovsdb-server`, the new `ovs-vswitchd`, and the new `openvswitch.ko` move into a consistent state with no intermediate incompatible window.
2. **State preservation.** Preserves flow rules, ports, conntrack state, mirror configuration, and QoS queues across the restart cycle.
3. **Data plane continuity.** Fast-path forwarding continues for existing flows while the daemon is restarting (sub-second window).
4. **Cluster coordination** (Raft cluster). One node is upgraded at a time while the cluster still has quorum (N/2+1).

---

## 9.12.5 Why it exists

Before OVS rolling restart, production OpenStack and OVN deployments faced four real pain points: (1) **The maintenance window was too long.** A full-fleet restart of 100 or more compute nodes lost connectivity for 30 to 60 minutes, breaching the SLA. (2) **Quorum loss when restarting a Raft cluster at the same time.** A 3-node `ovsdb` cluster restarted simultaneously means loss of quorum, which means read-only mode, which blocks OpenStack Neutron create operations. (3) **VM connectivity loss when the kernel module reloads** affected SSH sessions, NFS mounts, and replication of distributed applications. (4) **Schema migration race.** When the new binary started before the schema was converted, it would crash-loop. Pfaff and Pettit designed the rolling pattern between 2014 and 2016 to address all four pain points at once, drawing on telco and cloud production experience.

---

## 9.12.6 Problems it solves

Specifically, four technical problems are addressed by rolling upgrade: (1) **Schema and binary mismatch deadlock.** If the new binary reads an old DB whose schema is not yet converted, the server fails schema validation and exits. If the old binary reads a new converted DB, it fails forward-compatible parsing and exits. The fix is to convert BEFORE starting the new binary (Rule 1, schema-first). (2) **Kernel and userspace ABI mismatch.** Netlink genl family struct version mismatch causes `genlmsg_parse` to fail. The fix is to keep the kernel module and userspace at the same major version (Rule 3, do not change them at the same time). (3) **Megaflow cache invalidation cascade.** Restarting `ovs-vswitchd` flushes all cached megaflows, so the first packet lookup misses, an upcall is sent, and the revalidator installs a new megaflow. With 100 thousand megaflows, this causes a burst within 1 to 2 seconds and a CPU spike. The fix is `--monitor` graceful restart and revalidator pacing. (4) **OVSDB Raft majority loss.** Restarting 2 of 3 nodes at the same time means loss of quorum and no leader. The fix is rolling per-node restart, waiting for `cluster/status` to show `leader: yes` and lag less than 50 ms.

---

## 9.12.7 Importance

Upgrade choreography is an absolutely core skill for every engineer operating OVS or OVN in production. An engineer who does not master choreography cannot do: (a) any urgent security patch (an OVS CVE appears 2 to 3 times per year); (b) a major version migration aligned with the OpenStack release cycle (Yoga to Zed to Antelope every 6 months); (c) a kernel upgrade cycle (Ubuntu LTS HWE rotation every 6 months); (d) emergency downgrade when the new binary has a regression. Every distribution release cycle (RHEL, Ubuntu LTS), every OpenStack release, and every OVN release pins a specific OVS version. The engineer must understand choreography to roll out without causing an outage. The level of importance is on par with `megaflow`, `ct_state`, and `dpif` in the core OVS group.

## 9.12.8 How it works

The 5-step single-node atomic procedure follows a strict order (compass Ch P):

**Step 1, install packages WITHOUT restarting the daemon:**

```bash
[host]# apt install openvswitch-switch openvswitch-common
```

The package manager installs the new binary into the file system, BUT the running daemons (`ovsdb-server`, `ovs-vswitchd`) still use the old binary already loaded into memory. The systemd unit does not restart. This step is safe on a live host.

**Step 2, restart `ovsdb-server` gracefully:**

```bash
[host]# ovs-appctl -t ovsdb-server exit --cleanup
[host]# systemctl start ovsdb-server
```

`exit --cleanup` flushes pending transactions before exit, ensuring that the `conf.db` file is consistent. systemd starts `ovsdb-server` with the new binary. Down time is roughly 100 to 500 ms depending on the schema size. During this window, `ovs-vswitchd` continues forwarding (the data plane keeps running).

**Step 3, schema conversion if needed:**

```bash
[host]# ovsdb-tool needs-conversion /etc/openvswitch/conf.db \
         /usr/share/openvswitch/vswitch.ovsschema
```

If the output is `yes`:

```bash
[host]# ovsdb-tool convert /etc/openvswitch/conf.db \
         /usr/share/openvswitch/vswitch.ovsschema
```

Source: `ovsdb/ovsdb-tool.c::do_convert` (line 449, v2.17.9) implements schema migration: parse the old DB JSON, validate against the new schema, transform record format (column add, remove, type change), then atomically rename `conf.db.new` to `conf.db`. The operation is backward-compatible for minor versions; a major version may require manual data transformation.

**Step 4, restart `ovs-vswitchd` atomically:**

```bash
[host]# systemctl reload-or-restart openvswitch-switch
```

Compass Ch P pattern: `--monitor` mode for `ovs-vswitchd`. The new instance forks, attaches an OVSDB connection, restores `ofproto` state from the DB, and then signals the old instance to exit. The data plane does not drain because the kernel megaflow cache is independent of the userspace process; megaflows continue to match existing flows during the restart window of 2 to 5 seconds.

**Step 5, kernel module reload (only when the kernel module is upgraded):**

```bash
[host]# rmmod openvswitch
[host]# modprobe openvswitch
```

Compass Ch P: reloading the kernel module **briefly evicts the megaflow cache** (the cache lives in kernel module memory) BUT **does not drop existing connections**, because conntrack state lives separately in the `nf_conntrack` netfilter table. Window: 1 to 3 seconds of fast-path miss, upcall, and megaflow reinstall.

## 9.12.9 How an operator masters this

An operator fluent in upgrade choreography needs reflexes for six actions:

1. **Read the pre-upgrade state.** `ovs-vsctl get Open_vSwitch . ovs-version` and `ovs-vsctl get Open_vSwitch . db-version` give the current OVSDB schema version. Combine with `ovs-vswitchd --version` for userspace and `cat /sys/module/openvswitch/version` for the kernel module.
2. **Drain the orchestrator scheduler.** Pre-upgrade, mark the host `scheduling=disabled` in OpenStack Nova or cordon the Kubernetes node. Reason: prevent new VMs or Pods from being scheduled on the host during the upgrade window.
3. **Back up the DB pre-upgrade.** `ovsdb-client backup unix:/var/run/openvswitch/db.sock OVN_Northbound > /backup/conf.db.$(date +%F).bak`. Reason: the backup is cheap insurance and lets you recover if the schema convert fails.
4. **Apply the 5-step upgrade** in the strict order from §9.12.8.
5. **Verify post-upgrade.** `ovs-vsctl show` (is the bridge list intact?), `ovs-ofctl dump-flows br-int | wc -l` (does the rule count match pre-upgrade?), `ovs-appctl dpif/show` (is the datapath state OK?), `journalctl -fu openvswitch-switch | tail -20` (no error log?).
6. **Re-enable scheduling.** Mark the host `scheduling=enabled`. Done.

Decision tree when something is off:

- Pre-upgrade `ovsdb-tool needs-conversion` returns `no`: skip the schema convert step (Step 3).
- Kernel module version matches userspace: skip kernel reload (Step 5).
- Cluster Raft mode: use the cluster choreography in §9.12.20 instead of the 5-step single-node procedure.

## 9.12.10 Classification

`Upgrade choreography` is an **operational procedure**. It is not:

- A standalone CLI tool (it is just a sequence of commands).
- An OVSDB schema row (no row in the database).
- An OpenFlow primitive.
- A daemon or kernel feature.

Classification by scope:

- **Single-node rolling.** One host upgrade per the 5-step procedure in §9.12.8 (single-host scope).
- **Cluster Raft rolling.** N-node OVSDB cluster upgrade with each node leaving and rejoining (cluster scope, §9.12.20).
- **Fleet rolling.** N hosts in an OpenStack or Kubernetes fleet upgraded sequentially, each host following the single-node procedure (fleet scope).

Classification by trigger:

- **Security patch (CVE).** Minor version, schema usually unchanged, fast.
- **Minor version.** Schema may have a column added (compatible) or a feature added.
- **Major version (2.17 to 3.0).** The schema may have a breaking change, ABI recompilation is required, planned maintenance window.

## 9.12.12 When something goes wrong

The on-call operator must remember upgrade choreography when one of four typical symptoms appears:

**Symptom A: schema conversion fails**

- Trigger: `ovsdb-tool convert` reports `constraint violation` or `schema mismatch`.
- Cause: a custom column or a breaking schema change.
- Recovery: `ovsdb-client restore unix:/var/run/openvswitch/db.sock < /backup/...` (restore the pre-upgrade DB), roll back the binary version, investigate.

**Symptom B: VM or container loses connectivity for more than 10 seconds after restart**

- Trigger: SSH session dropped, NFS mount stale, application timeout.
- Cause: kernel module mismatch (Rule 3 violation), conntrack state lost, or bond flap.
- Recovery: check `dmesg | grep openvswitch`, `lsmod | grep openvswitch`, restart the bond peer.

**Symptom C: `ovn-controller` disconnects from SBDB after upgrade**

- Trigger: `ovn-controller` log shows `ssl: peer certificate verification failed`.
- Cause: TLS certificate expiry coincides with the upgrade window.
- Recovery: rotate the certificate per Part 20.1 §20.1.15, restart `ovn-controller`.

**Symptom D: megaflow CPU spike sustained for more than 5 minutes after restart**

- Trigger: `top` shows `ovs-vswitchd` CPU above 80 percent, `coverage/show upcall_flow_add` rate above 1000 per second.
- Cause: revalidator storm post-cache-flush (Part 9.26).
- Recovery: tune `revalidator/wait`, throttle the rule install rate, or accept the transient.

## 9.12.13 Tightly related to

`Upgrade choreography` is tightly coupled to six components:

1. **`ovsdb-tool`** (Part 9.31): `convert`, `needs-conversion`, `compact`, and `join-cluster` are all invoked in choreography.
2. **`ovsdb-server`** and Raft cluster (Part 10.1): cluster mode requires per-node rolling instead of the single-node 5-step procedure.
3. **`ovs-vswitchd`** monitor mode (Part 9.1): `--monitor` graceful restart is the main mechanism for data-plane continuity.
4. **`openvswitch.ko`** kernel module (Part 9.2): module reload state lifecycle, megaflow cache eviction.
5. **`mTLS` certificate** (Part 9.10 and 20.1): certificate expiry check is a mandatory pre-upgrade checklist item.
6. **OpenStack Nova or Kubernetes scheduler:** drain and cordon coordination.

## 9.12.17 Real-world incident

Production case (curated from a public bug pattern): an OpenStack Yoga deployment in Q4-2023, a 50-compute-node fleet, planned an OVS upgrade from 2.17.4 to 2.17.7 (security CVE patch). Step 4 (restart `ovs-vswitchd`) on `compute02` reported a 30 second timeout; the systemd unit failed. Investigation:

```
[compute02]# journalctl -fu openvswitch-switch --since "5 min ago"
ovs-vswitchd: ovsdb-server connection lost, reconnecting
ovs-vswitchd: ofproto: failed to register controller
ovs-vswitchd: bridge: failed to set up bridge
systemd: openvswitch-switch.service: Main process exited, code=exited, status=1
```

Root cause: after restart, `ovs-vswitchd` could not connect to `ovsdb-server` because the permission of `/var/run/openvswitch/db.sock` had changed after the Step 2 graceful exit. Specifically, the systemd unit for `ovsdb-server` 2.17.7 sets the socket permission to `0660 root:openvswitch`, but `ovs-vswitchd` runs as user `openvswitch`; 2.17.4 set it to `0666` (more permissive). This was a regression in distribution packaging (Ubuntu 22.04 Yoga maintainer adjustment), not an upstream OVS bug.

Diagnostic chain:

- `ls -la /var/run/openvswitch/db.sock`: permission 0660.
- `ps -ef | grep ovs-vswitchd`: running as user `openvswitch`.
- `id openvswitch`: groups `openvswitch` only, not in group `root`.

Fix: temporarily run `chmod 0666 /var/run/openvswitch/db.sock` to unblock the fleet upgrade, then add a systemd unit overlay at `/etc/systemd/system/ovsdb-server.service.d/permission.conf` that sets `ExecStartPost=chmod 0666 /var/run/openvswitch/db.sock`.

Lesson: distribution packaging can introduce regressions that do not appear in upstream NEWS. A pre-upgrade staging environment test is mandatory; do not trust release notes by default.

## 9.12.19 Common failures and diagnostic signals

Five typical failure modes and the recovery path:

**Failure 1, schema conversion fails (constraint violation)**

- Signal: `ovsdb-tool convert` exits non-zero with log `constraint violation: column 'X' type mismatch`.
- Cause: a custom column in the production DB, or a breaking schema change.
- Recovery: restore the backup DB; if a convert is needed, contact the distribution maintainer for a migration script.

**Failure 2, bond flap during restart (LACP timeout)**

- Signal: `ovs-vsctl list Interface | grep -E "lacp_status|name"` reports `expired`; ifup or ifdown log entries.
- Cause: LACP slow-timer 90 seconds versus daemon restart window of 5 to 30 seconds, which causes the peer to mark the link `down`.
- Recovery: switch LACP to fast-timer pre-upgrade; switch back to slow-timer post-upgrade.

**Failure 3, flow rule loss (grammar removed)**

- Signal: post-upgrade `ovs-ofctl dump-flows br-int | wc -l` is less than the pre-upgrade count.
- Cause: an OF action or match grammar was deprecated in a major version.
- Recovery: patch the controller (OVN northd or Faucet) before OVS, regenerate the rules.

**Failure 4, `ovn-controller` disconnects from SBDB (TLS expiry)**

- Signal: `ovn-controller` log `ssl: peer certificate verification failed: certificate has expired`.
- Cause: certificate expiry coincides with the upgrade maintenance window.
- Recovery: rotate the certificate per Part 20.1 §20.1.15, restart `ovn-controller`.

**Failure 5, kernel module verification reject (signing strict)**

- Signal: `dmesg | grep openvswitch` reports `module verification failed: signature and/or required key missing`.
- Cause: kernel 5.15+ has strict signing; the OVS module compiled out-of-tree is not signed by the distribution key.
- Recovery: install `linux-modules-extra-$(uname -r)` from the Ubuntu repository (already signed), OR disable strict signing through the boot parameter `module.sig_enforce=0` (NOT recommended for production).

## 9.12.20 Cluster upgrade choreography (OVSDB Raft HA)

For an OVSDB Raft cluster (Part 10.1), the single-node 5-step procedure does NOT apply. The cluster choreography follows a per-node sequence:

1. **Identify a non-leader node.** `ovs-appctl -t ovsdb-server cluster/status Open_vSwitch` reports the role; pick followers first and the leader last (this reduces election events).
2. **Drain client connections.** Optional. Redirect clients to another node through DNS or a load balancer configuration.
3. **Leave the cluster.** `ovs-appctl -t ovsdb-server cluster/leave Open_vSwitch`. Wait until `cluster/status` shows the node as `left`.
4. **Stop `ovsdb-server`.** `systemctl stop ovsdb-server`.
5. **Install the new binary.** `apt install openvswitch-common`.
6. **Reset the DB.** **Delete** `/etc/openvswitch/conf.db` (compass Ch N: "Once a server leaves a cluster it cannot rejoin; create a fresh member").
7. **Rejoin the cluster.** `ovsdb-tool join-cluster /etc/openvswitch/conf.db Open_vSwitch \
    tcp:<this-node-ip>:6643 tcp:<remaining-leader-ip>:6643`.
8. **Start `ovsdb-server`.** `systemctl start ovsdb-server`. Wait for sync via `cluster/status` showing `leader: <leader-name>` and lag less than 50 ms.
9. **Repeat per node** until all N nodes are upgraded.
10. **Verify quorum and leader stability.** `cluster/status` from any node should show consistent results.

Source: `ovsdb/ovsdb-tool.c::do_join_cluster` (line 326, v2.17.9) implements the join logic. Critical path: maintain N/2+1 quorum throughout. With a 3-node cluster, never stop more than 1 node at the same time.

## 9.12.21 Downgrade strategy

Downgrade is more complicated than upgrade because forward schema conversion is usually irreversible. Best practice:

```bash
# Pre-upgrade: explicit DB backup
[host]# ovsdb-client backup unix:/var/run/openvswitch/db.sock Open_vSwitch \
  > /backup/conf.db.pre-upgrade-$(date +%F).bak

# Downgrade if needed:
[host]# systemctl stop ovs-vswitchd ovsdb-server
[host]# apt install openvswitch-switch=<old-version>
[host]# rm /etc/openvswitch/conf.db
[host]# ovsdb-tool create /etc/openvswitch/conf.db /usr/share/openvswitch/vswitch.ovsschema
[host]# ovsdb-client restore unix:/var/run/openvswitch/db.sock Open_vSwitch < /backup/conf.db.pre-upgrade-...bak
[host]# systemctl start ovsdb-server ovs-vswitchd
```

Cluster downgrade is even more complicated: use `ovsdb-tool::do_convert_to_standalone` (line 1673, v2.17.9) to convert the cluster DB to a standalone DB, then reinstall the old binary, then recreate the cluster.

## 9.12.22 Comparison with other systems

OVS `upgrade choreography` has analogues in other stacks:

- **Linux native bridge.** No rolling restart. Kernel bridge driver upgrade requires a kernel reboot. Downtime: full reboot, about 30 to 60 seconds. OVS rolling is a major advantage over native.
- **VPP graph node version.** VPP has graph node versioning in the control thread; upgrading a graph node does not require restarting the entire `vpp-agent`. Different from OVS in that VPP is performance-focused (DPDK polling); upgrade is more complex because PMD threads have to coordinate.
- **Cisco IOS-XE In-Service Software Upgrade (ISSU).** ISSU is the industrial benchmark for rolling upgrade; it uses a standby supervisor module plus state replication. Different from OVS in that ISSU requires hardware redundancy (dual supervisor); OVS rolling needs only a single host thanks to the kernel and userspace separation.
- **Juniper Junos NSR and NSSU** (Non-Stop Routing and Non-Stop Software Upgrade): equivalent to ISSU; routing engine A serves while B upgrades. Hardware redundancy required.
- **Kubernetes kubelet rolling.** Kubelet drain plus cordon plus upgrade is the analogous pattern for compute nodes; OVS choreography piggy-backs on the Kubernetes lifecycle when OVN is deployed as the CNI.
- **PostgreSQL major upgrade (`pg_upgrade`).** Schema convert through the `pg_upgrade` tool is similar in idea to `ovsdb-tool convert`, but PostgreSQL requires full database downtime; OVSDB convert is sub-second.

## Common misconceptions and key topic

> **Common misconception 1:** "Skipping major versions (2.17 to 3.3) is safe with `apt upgrade`."
>
> **Reality:** the OVS release cycle is 6 to 12 months, and the supported migration path is 1 or 2 majors back. Skipping multiple versions hits an untested schema conversion chain. The safe pattern is incremental: 2.17 to 2.18 to ... to 3.3, through distribution backports or manual builds.

> **Common misconception 2:** "Rolling restart equals zero downtime."
>
> **Reality:** there is sub-second packet loss when the flow cache invalidates. An SLA budget of 99.99 percent allows roughly 5 seconds per restart. 10 chassis times 5 seconds equals 50 seconds cumulative. Tunnel pre-warming and CT state preservation (OVS 2.15+) reduce the impact, but there is NO zero-downtime guarantee.

> **Common misconception 3:** "Cluster Raft node leave-rejoin is equivalent to restarting the binary."
>
> **Reality:** compass Ch N: "Once a server leaves a cluster it cannot rejoin; create a fresh member." After `cluster/leave`, you must delete the DB and run `ovsdb-tool join-cluster` to recreate the member. You cannot simply restart the binary and keep the existing DB.

> **Key topic:** the 3 golden rules plus the 5-step choreography deliver zero data plane loss. Golden rules: (1) Schema first before binary. (2) One daemon at a time. (3) Do not change OVS and the kernel at the same time. The 5 steps are install, restart-ovsdb, convert-schema, restart-ovs-vswitchd, reload-kernel-module. Cluster mode adds: per-node leave, recreate, and rejoin with quorum maintained.

---

## Key points to remember

First, OVS upgrade is designed to avoid losing the data plane, through a 5-step rolling restart that coordinates three components (OVSDB, `ovs-vswitchd`, kernel module).

Second, the three golden rules (schema first, one daemon at a time, do not change OVS and the kernel at the same time) avoid most upgrade failures.

Third, cluster Raft upgrade differs from single-node: a node that leaves the cluster CANNOT rejoin with the old DB; it must be recreated through `ovsdb-tool::do_join_cluster`.

Fourth, backing up the DB before any production change is cheap insurance; restore through `ovsdb-client restore` with the pre-upgrade backup file.

Fifth, the Anatomy pre-upgrade 9-attribute checklist (version, schema, rules, ofconns, upcall, backup, peer compatibility, window) supports a go or no-go decision.

Sixth, skipping major versions equals risk through an untested schema conversion chain. The incremental safe pattern is 2.17 to 2.18 to ... to 3.x, through distribution backport or manual builds.

Seventh, rolling restart causes sub-second data-plane loss. Budget the SLA math: at 99.99 percent, 5 seconds times N hosts cumulative.

Eighth, version drift table 2.17.0 (Feb 2022) to 3.0.0 (Aug 2022): OVS 3.0 changed the container iteration macros to be UB-safe, requiring client recompilation; OVSDB schema 8.3.x to 8.4.x.

Ninth, the source code anchor `ovsdb/ovsdb-tool.c::do_convert` (line 449, v2.17.9) is the entry point for schema conversion; `do_join_cluster` (line 326) is for cluster choreography.

Tenth, downgrade is more complex than upgrade because forward schema conversion is irreversible; cluster mode uses `do_convert_to_standalone` (line 1673) as a bridge for the downgrade path.

## §9.32.1 classifier (TSS), Tuple Space Search

### Concept

The TSS classifier is the packet and flow classification algorithm inside the userspace `ofproto-dpif` layer of OVS. "Tuple space" means the set of unique mask patterns; each unique mask creates one subtable. A lookup walks the subtables in priority order. Type: a userspace data-plane lookup algorithm.

### History and background

The TSS algorithm comes from Srinivasan et al., SIGCOMM 1998, "Packet Classification using Tuple Space Search". OVS adopted it in 2009 with the initial Nicira implementation. The classifier was added to the OVS source tree in `lib/classifier.c` on 2009-07-08 (commit `064af42167bf4fc9`), which predates the OVS 1.0 release. The algorithm was further refined in 2014 around OVS 2.0, mainly through megaflow consolidation. The classifier has been stable since.

### Position in the architecture

The classifier lives in the userspace `ofproto-dpif` layer of the OVS 3-component architecture (kernel datapath, userspace `ovs-vswitchd`, OVSDB). Concretely: `lib/classifier.c` in the OVS source tree. Scope: one classifier instance per bridge.

### Role

The classifier is the "rule matching engine" between the OpenFlow flow-table abstraction and kernel megaflow installation. The pipeline is: OF rule add, classifier insert (build the subtable), classifier output (install the megaflow). The output then drives the kernel datapath fast path.

### Why it exists

Pre-TSS pain: a linear scan of the flow table at O(N) per packet does not scale. A hash-only lookup does not support wildcard matches. A trie-only lookup is hard for arbitrary masks. TSS is the compromise: hash within a subtable (fast, O(1)) and linear over the subtables (limited by the mask count). Production traffic typically shows 10 to 50 unique masks, which is manageable.

### Problems solved

The problem: arbitrary wildcard match at production scale. The mechanism: partition the flow table into per-mask subtables, with a deterministic lookup cost of O(subtable_count). Scope: fully solves wildcard flexibility, with a partial trade-off against mask explosion (the anti-pattern is a rule set that produces too many unique masks).

### Importance

Cornerstone tier 1. An engineer who does not understand TSS cannot tune the classifier when production hits a performance issue (slow rule install, classifier lookup CPU spike, mask-explosion warning).

### How it works

Per-rule insert (`classifier_insert` in `lib/classifier.c`):
1. Compute the mask pattern (which fields are wildcarded versus exact).
2. If a subtable already exists for the mask, insert into the hash bucket (chained hash).
3. Otherwise, create a new subtable and insert.
4. Update the priority-sorted vector (pvector) ordering of subtables.

Per-packet lookup (`classifier_lookup` in `lib/classifier.c`):
1. For each subtable in priority order (highest priority first):
   a. Hash the packet fields with the subtable's mask, then bucket.
   b. Linearly scan the bucket for an exact match.
2. Return the first match (highest priority).

Mask consolidation invariant: two OF rules with the same wildcard pattern (regardless of field values) share one subtable.

Patricia trie optimization: a prefix trie for IP and port-range masks lets the lookup skip subtable subsets through trie pruning. The priority-sorted vector backing lives in `lib/pvector.c`.

### How an operator masters this

Four skills:

1. **Read**: `ovs-appctl coverage/show | grep classifier` shows counters (`lookup_match`, `lookup_miss`, and so on).
2. **Modify**: indirect, through OF rule design (adding or removing a rule triggers a subtable rebuild). Direct subtable manipulation is not exposed.
3. **Observe**: `ovs-appctl ofproto/list-tunnels` plus `ovs-appctl ofproto/trace 'flow_spec'` step through the classifier evaluation.
4. **Debug**: when the classifier looks slow, check subtable count through the coverage counters; when mask explosion is suspected, audit the OF rule mask diversity.

Decision tree:
- Coverage shows a `classifier_lookup` spike, then check the subtable count (a high count means explosion).
- Subtable count above 50, then audit the rule mask diversity and refactor with the conjunction action.
- Subtable count below 10, then the classifier is not the bottleneck; investigate elsewhere.

### Classification

Type: userspace data-plane lookup algorithm. Parent: TSS implementation in `lib/classifier.c`. Scope: per-bridge instance. Cost: O(K), where K is the subtable count.

### Usage workflow

```
Step 1: Profile classifier overhead
  $ ovs-appctl coverage/show | grep -E "classifier|cmap"
  Shows the cmap_lookup and cmap_insert counters.

Step 2: Identify hot subtables
  $ ovs-appctl ofproto/list-tunnels | wc -l
  More than 100 subtables means mask explosion.

Step 3: Mask consolidation refactor
  Rule A: nw_src=10.0.0.5/32, action=output:1
  Rule B: nw_src=10.0.0.6/32, action=output:1
  Refactor: use conjunction(1,1/2) for the IP set and conjunction(1,2/2) for the action.
  2 unique rules become 1 subtable plus 1 verdict subtable, which is simpler.

Step 4: Verify the reduction
  $ ovs-appctl coverage/show | grep cmap
  The subtable count should drop.

Step 5: Measure the latency improvement
  Use a packet generator and measure first-flow latency after the refactor.
```

### When something goes wrong

- Symptom A: first-flow latency above 10 ms. Diagnosis: the classifier lookup is slow, the subtable count is too high. Recovery: mask consolidation.
- Symptom B: `ovs-vswitchd` CPU spike on rule add. Diagnosis: classifier insert is slow because of subtable rebuild cost. Recovery: batch rule installs through the bundle action.
- Symptom C: rule effect is intermittent. Diagnosis: a classifier priority-order conflict where two rules at the same priority overlap. Recovery: explicit priority assignment.

Cross-links: 9.26 revalidator storm forensic; 9.25 `ofproto/trace` classifier debug.

### Tightly related to

- **megaflow** (9.2): the classifier output is installed into the kernel megaflow.
- **ukey** (9.2): the revalidator tracks the mapping between a kernel flow and a classifier rule.
- **upcall** (next, §9.32.2): the classifier is invoked when the kernel misses.
- **subtable** (this section): a per-mask classifier partition.
- **Patricia trie** (this section): the prefix optimization.
- **pvector** (this section): the priority order.

### Real-world incident

Production case 2023-Q4: an OVS cluster started running `ovs-vswitchd` at 100 percent CPU after an ACL update was deployed.

Investigation:
- `coverage/show` showed `classifier_insert` sustained at 50,000 per second.
- The new ACL rule set had 2000 entries, each with a unique 5-tuple, which produced 2000 unique masks.
- Subtable explosion: 2000 subtables, with an O(K) cost on every lookup.

Root cause: the ACL design used a per-IP-and-port-pair rule instead of the conjunction action.

Fix: refactor the ACL with `Address_Set`, `Port_Group`, and conjunction, which dropped the subtable count to 200. `ovs-vswitchd` CPU dropped from 100 percent to 30 percent.

Reference: a similar pattern is documented in 9.26 revalidator storm.

### Synthetic exercise

```
Setup: Mininet single-switch s1 với 100 veth port

Test mask explosion:
  for i in $(seq 1 100); do
    ovs-ofctl add-flow s1 "priority=100,ip,nw_src=10.0.0.$i,actions=output:$i"
  done
  ovs-appctl coverage/show | grep cmap
  → cmap_insert ~ 100 (1 subtable per mask, but mask same for all - actually 1 subtable!)

  # Now diversify:
  for i in $(seq 1 100); do
    ovs-ofctl add-flow s1 "priority=100,ip,nw_src=10.0.$i.0/24,actions=output:$i"
  done
  → 100 different mask /24, 100 subtable

Verify reduction with conjunction:
  ovs-ofctl add-flow s1 'priority=100,ip,nw_src=10.0.0.0/8,actions=conjunction(1,1/2)'
  ovs-ofctl add-flow s1 'priority=100,ip,actions=conjunction(1,2/2)'
  ovs-ofctl add-flow s1 'priority=200,conj_id=1,actions=output:1'
  → Now O(few) subtable for same matching capability
```

### Common failures and diagnostic signals

- Mode A: subtable explosion (more than 1000 subtables) leading to a CPU spike. Diagnose with `coverage/show`. Recovery: mask consolidation and conjunction.
- Mode B: Patricia trie misuse on a non-prefix mask, which falls back to subtable scan and runs slow. Recovery: rare bug; check the OVS version.
- Mode C: pvector reorder cost on a mass rule update, temporarily slow. Recovery: install the rules in a bundle so the update is atomic.

### Comparison with other systems

- **Linux iptables filter chain**: linear scan at O(N), no algorithmic improvement. The OVS classifier is far superior for a large rule set.
- **Cisco TCAM ACL**: hardware parallel match, O(1) but limited in capacity. The OVS software classifier is flexible (no capacity limit) at the cost of software speed.
- **eBPF cls_bpf classifier**: a programmable classifier in the kernel. Difference: eBPF compiles per-rule code through the JIT, while the OVS classifier interprets from the OF abstraction.
- **Intel DPDK rte_classify**: a hardware-assisted classification table. OVS hardware offload combines with the classifier.

## §9.32.2 upcall, kernel miss to userspace handler

### Concept

An `upcall` is a Netlink message sent from the kernel `openvswitch.ko` module to the userspace `ovs-vswitchd` daemon when the kernel datapath misses on a megaflow lookup. Userspace decodes the packet, evaluates it through the classifier, and installs the resulting megaflow back into the kernel. Type: a kernel-to-userspace IPC mechanism.

### History and background

The mechanism was introduced in 2009 with the original Nicira OVS. The Netlink protocol has been stable since. OVS 2.0 (2014) added an upcall rate limit to protect the control plane from a DDoS pattern.

### Position in the architecture

The bridge between the kernel `openvswitch.ko` module (fast path, miss handling) and userspace `ovs-vswitchd` (slow path, classifier plus flow install). Concretely, a Netlink socket on the `OVS_PACKET_FAMILY` family carries the upcall message.

### Role

The cache miss handler. Without an upcall, a kernel cache miss would drop the packet, with no fallback. With the upcall, a miss flows through classifier evaluation and a megaflow install, and the next packet on the same tuple hits the cache. The mechanism is critical for first-flow latency and for cache warming.

### Why it exists

Before OVS, a kernel module typically held a self-contained flow logic (the Linux bridge with local MAC learning is the canonical example). The OVS architecture decouples the control plane (userspace) from the data plane (kernel). The upcall is the bridge mechanism that carries the asynchronous control flow.

### Problems solved

The problem: kernel datapath cache miss handling, without dropping the packet. The mechanism: a Netlink message buffers the packet plus its metadata; a handler thread dequeues, evaluates, and installs the result, and optionally injects the packet back. Scope: a full solve for the cache miss case, with a partial trade-off (the upcall cost itself is roughly 10 to 100 microseconds per packet).

### Importance

Cornerstone tier 1. The upcall is the bottleneck under several conditions (high churn, cold cache, DDoS). An engineer who does not understand the upcall cannot debug "first packet slow" or "ovs-vswitchd CPU spike" symptoms.

### How it works

```
Kernel side (openvswitch.ko):
  Step 1: dp_packet_in() invoke từ netif_rx
  Step 2: flow_table_lookup() check megaflow
  Step 3: If miss → genl_send (OVS_PACKET_FAMILY, OVS_PACKET_CMD_MISS)
  Step 4: packet skb queued
  Step 5: continue (other packet handled)

Userspace side (ovs-vswitchd):
  Step 1: handler_thread main loop
  Step 2: dpif_recv() dequeue Netlink message
  Step 3: ofproto_recv_upcall() decode packet
  Step 4: ofproto-dpif xlate evaluate via classifier
  Step 5: Install resulting megaflow via dpif_flow_put()
  Step 6: Optionally inject packet back via OVS_PACKET_CMD_EXECUTE

Rate limit:
  Per-bridge upcall socket có rate cap, OVS 2.0+ adaptive throttle
```

Invariant: every miss generates exactly one upcall (no duplicate); the revalidator does not generate an upcall (it runs on a separate path).

### How an operator masters this

Skills: monitor upcall rate, identify spike root cause, and tune handler thread count.

```
$ ovs-appctl coverage/show | grep upcall
upcall_flow_limit_hit    0 +0   # no rate limit hits = healthy
upcall_flow_limit_kill   0 +0
miss_per_sec    100      # baseline normal

$ ovs-vsctl set Open_vSwitch . other_config:n-handler-threads=8
# Increase the handler thread count for a high-rate cluster.
```

### Classification

Type: a kernel-to-userspace IPC mechanism, on the Netlink message family. Parent: `OVS_PACKET_FAMILY`. Scope: per-packet on miss.

### Usage workflow

Debug an upcall storm:

```
Step 1: $ ovs-appctl coverage/show | grep -E "upcall|miss"
Step 2: If miss_per_sec is above 1000, identify the churn source.
Step 3: $ ovs-appctl dpctl/dump-flows | head -20
   Check the megaflow count and freshness.
Step 4: If there are many short-lived flows, increase n-handler-threads.
Step 5: If there are many unique masks, refer to §9.32.1 mask consolidation.
```

### When something goes wrong

- Symptom A: first-flow latency at 10 to 100 ms. Suspect upcall queue depth. Diagnose with `coverage/show`.
- Symptom B: `ovs-vswitchd` CPU at 100 percent steady state. Diagnose with the upcall rate sustained above 10,000 per second.
- Symptom C: a kernel log entry "upcall failed: ENOBUFS". Suspect a Netlink socket buffer overflow. Recovery: increase `net.core.rmem_max`.

### Tightly related to

Tightly coupled: the handler thread; megaflow (consumer of the upcall output); the classifier (the evaluation step); Netlink (the transport); the `dp_netdev` userspace alternate datapath; EMC and SMC (downstream caches).

### Version differences

| Version | Behaviour |
|---------|-----------|
| OVS 2.0 | Rate limit plus adaptive throttle. |
| OVS 2.4 | Multi-handler thread, default of four. |
| OVS 2.6 | EMC tier reduces the need for upcalls. |
| OVS 2.15 | SMC tier reduces the need further. |
| OVS 3.0+ | Stable. |

### Observation and verification

```
$ ovs-appctl coverage/show | grep upcall
$ tc -s qdisc show dev <iface>
# Shows the kernel netif queue depth.
$ dmesg | grep openvswitch
# Shows the kernel log of upcall failures.
```

### Source code reference

- Kernel: `net/openvswitch/datapath.c`, functions `ovs_dp_upcall()` and `ovs_dp_process_packet()`.
- Kernel: `net/openvswitch/flow_netlink.c`, Netlink encoding.
- Userspace: `lib/dpif-netlink.c`, function `dpif_netlink_recv()` for the dequeue.
- Userspace: `ofproto/ofproto-dpif-upcall.c`, function `handle_upcalls()` as the main handler loop.

### Real-world incident

Production case 2023-Q1: a web cluster experienced an occasional 100 ms latency spike on a new TCP connection.

Investigation:
- `coverage/show` showed the upcall rate spiking to 50,000 per second at peak.
- The megaflow count was low (about 100), so the cache should have been hitting.
- However, the miss rate was also high (500 per second).

Root cause: handler thread starvation under burst. With `n-handler-threads=4` (the default), a burst created a queue depth above 100 messages, which caused the latency spike.

Fix: increase `n-handler-threads=16` and tune `net.core.rmem_max=16M`. The P99 latency dropped from 100 ms to 5 ms.

### Synthetic exercise

```
Setup: a Mininet two-host topology.
  $ sudo mn --topo single,2 --switch ovsk

Generate a burst:
  $ ovs-ofctl del-flows s1 -- add-flow s1 'priority=10,actions=NORMAL'
  $ h1 hping3 -S -p 80 -c 10000 -i u100 h2
  # A SYN burst, where each SYN is a potential megaflow miss.

Monitor:
  $ watch -n 1 'ovs-appctl coverage/show | grep upcall'
  # Observe the upcall counter spike.

Tune:
  $ ovs-vsctl set Open_vSwitch . other_config:n-handler-threads=8
  $ systemctl restart openvswitch-switch
  # Re-run the burst and observe the latency improvement.
```

### Common failures and diagnostic signals

- Mode A: handler thread saturation, the upcall queue grows, latency spikes. Recovery: increase `n-handler-threads`.
- Mode B: Netlink socket overflow with ENOBUFS. Recovery: increase `rmem_max`.
- Mode C: adaptive rate limit kicks in, flows are dropped. Diagnose with the coverage counter `upcall_flow_limit_hit`. Recovery: investigate upstream churn.

### Comparison with other systems

- **Linux netfilter NF_QUEUE:** a similar kernel-to-userspace queue mechanism. Difference: NF_QUEUE is a per-rule decision, while the OVS upcall is a per-miss decision (more efficient).
- **DPDK PMD poll-mode:** an alternative that bypasses the kernel entirely. Difference: PMD continuously polls, while the OVS upcall is event-driven. The trade-off is CPU usage versus latency.
- **Cisco IOS process switching:** a packet escalates from the fast path to the control plane. The concept is the same, with a different abstraction.

## §9.32.3 ofproto-dpif xlate, the translator from OpenFlow actions to datapath byte-code

### Concept

`ofproto-dpif xlate` is the translation engine inside userspace `ovs-vswitchd`. Its job is to turn the OpenFlow action list of a flow rule into a sequence of datapath byte-code (`odp_actions`) that the kernel module `openvswitch.ko` or the userspace DPIF can execute linearly. Every packet that takes the upcall slow-path passes through xlate. After xlate completes, the resulting `odp_actions` is used to install a megaflow into the datapath cache, and later packets on the same tuple run straight on the fast path without calling xlate again. Xlate is the only bridge between the OpenFlow abstraction (controller-facing, multi-table, recursive `goto_table` and `resubmit`, bundle, group, learn) and the datapath byte-code (kernel-facing, one-directional, no branches, no recursion).

### History and background

Xlate appeared in early 2012 when Justin Pettit and Ben Pfaff split `ofproto-dpif` into a separate module (refactor commits around the OVS 1.5 to 1.7 timeframe). Before that, `ofproto-classifier` wrote Netlink requests directly down to `openvswitch.ko`, mixing OpenFlow business logic with kernel API binding. The split allowed OVS to support multiple datapath backends later (`dpif-netlink` for the kernel, `dpif-netdev` for userspace plus DPDK plus AF_XDP) without rewriting the action translation logic. OVS 2.4 (2015) added recursion emulation through `recirc_id` (instead of assuming the datapath supports multiple tables). OVS 2.6 added the conntrack helper. OVS 2.8 added NSH. OVS 2.13 stabilised `frozen_state` for continuation. OVS 3.0+ has only minor bug fixes and speed optimisations.

### Position in the architecture

Xlate sits in the middle of the upcall pipeline of `ovs-vswitchd`. Its input is the result of the classifier lookup (a rule with an `ofpacts` list); its output is a buffer of `odp_actions` for dpif. In the source tree, xlate occupies the entire file `ofproto/ofproto-dpif-xlate.c` (about 7000 lines), plus `ofproto/ofproto-dpif-xlate-cache.c` (cache invalidation hint) and the header `ofproto/ofproto-dpif-xlate.h`. The main caller is `handle_upcall()` in `ofproto/ofproto-dpif-upcall.c`; the consumer is `dpif_flow_put()` in `lib/dpif.c`. Each userspace handler thread holds its own `struct xlate_ctx` to avoid contention.

### Role

The role of xlate has four parts: (1) walk the multi-table OpenFlow pipeline through recursive `goto_table` and `resubmit`; (2) resolve each `ofpact` into one or more corresponding `odp_action` items (output, set field, push or pop VLAN, ct, recirc, encap); (3) freeze the state when translation cannot finish in one pass (write `frozen_state`, allocate a new `recirc_id`, and stop translating); (4) normalise the OpenFlow 1.1+ action set before commit. Every OpenFlow feature that the controller exposes to the network must pass through xlate before reaching the datapath; if xlate does not support it, the feature effectively does not exist for the end host.

### Why it exists

The kernel datapath only knows a finite set of primitive actions and has no concept of a table. OpenFlow 1.1+ is much richer: many tables, recursion, an action set, and groups. Without a translation layer, two solutions become impractical: (a) teach the kernel the full OpenFlow language, which inflates the attack surface; (b) force the controller to translate by hand down to datapath primitives, which loses the abstraction. Xlate exists so that the kernel stays small, the controller still uses the standard OpenFlow language, and any difference in between is hidden.

### Problems solved

Xlate solves four specific problems: (1) translate recursive `goto_table` of arbitrary depth into a linear action sequence; (2) when an action is not supported natively by the datapath (for example, an NSH encap on an older kernel), emulate it through `recirc` plus an auxiliary table; (3) when a packet must wait for a conntrack or NAT result, freeze the rest of the pipeline into `frozen_state` and resume after `OVS_ACTION_ATTR_RECIRC` brings the packet back to userspace; (4) detect a recursive `resubmit` cycle and cut at `MAX_DEPTH = 64` to avoid stack overflow.

### Importance

Xlate is one of the three cornerstones of OVS userspace, alongside the classifier and dpif. An OVS network engineer cannot debug "the rule is correct but the packet does not pass through" without reading the xlate trace. The operations team cannot explain why a simple flow with `goto_table` produces a new `recirc_id` without understanding the pipeline-freeze mechanism. The release team cannot evaluate the impact of a kernel upgrade without knowing whether xlate can still emulate the affected feature. A xlate mistake (for example, a `MAX_DEPTH` cycle, or an emulation chain that explodes) typically shows up as "userspace CPU at 100 percent the whole time, but throughput stays low"; the root cause is xlate.

### How it works

The single entry point is `xlate_actions(struct xlate_in *, struct xlate_out *)`, declared in `ofproto/ofproto-dpif-xlate.h`. The caller prepares an `xlate_in` containing the original flow, the packet (if any), and the starting rule. After the call, it receives an `xlate_out` containing the accumulated `odp_actions`, a `slow_path_reason` (if there is a reason to send the packet back to userspace), and `recirc_refs` (the list of allocated `recirc_id` values).

Internally, the function sets up a `struct xlate_ctx`, then calls the internal `do_xlate_actions()` to walk the `ofpact` array. Each `ofpact` is dispatched by type:

- `OFPACT_OUTPUT`, calls `compose_output_action()` to emit `OVS_ACTION_ATTR_OUTPUT` with the corresponding datapath port number.
- `OFPACT_GOTO_TABLE`, calls `xlate_table_action()` recursively with the new `table_id`; increments `ctx->depth`; if `ctx->depth > MAX_DEPTH` (`#define MAX_DEPTH 64` in `ofproto-dpif-xlate.c`), stops and sets `ctx->error = XLATE_RECURSION_TOO_DEEP`.
- `OFPACT_RESUBMIT`, similar to `goto_table` but in the same table; increments `ctx->resubmits`; if `ctx->resubmits > MAX_RESUBMITS` (which equals `MAX_DEPTH * MAX_DEPTH = 4096`), stops.
- `OFPACT_CT`, emits `OVS_ACTION_ATTR_CT` with nested attributes (zone, mark, label, nat); if there are still actions left to translate after the `ct`, freezes the remaining actions into `frozen_actions`, allocates a new `recirc_id`, and emits `OVS_ACTION_ATTR_RECIRC`.

At the end, `xlate_commit_actions()` flushes the accumulated buffer to `xlate_out->odp_actions`, and the caller invokes `dpif_flow_put()` to install the megaflow.

### How an operator masters this

Skills to master: (a) read the trace from `ovs-appctl ofproto/trace br0 '<flow_spec>'` step by step, and match each "Resubmitted flow:" or "thawed from frozen state" line with a rule in the dump; (b) read the counters from `ovs-appctl coverage/show | grep xlate_` to know how often `xlate_actions_too_many_output`, `xlate_actions_resubmit_recursion`, or `xlate_actions_oversize` fires; (c) read `ovs-vswitchd.log` for warnings such as "over MAX_DEPTH" or "flow translation oversized"; (d) run `ovs-appctl dpif/show-dp-features` to know which actions the datapath supports natively and which ones must be emulated.

### Classification

By xlate result, the xlate path falls into three groups: (1) **fast translate**, a simple flow with output or drop, a short `odp_actions` result, and no recirc; (2) **recirc translate**, a pipeline with conntrack, multi-stage, that allocates one or more `recirc_id` values; (3) **slow-path translate**, where xlate sets a `slow_path_reason` (for example, `SLOW_CONTROLLER` or `SLOW_BFD`), no megaflow is installed, and every packet still goes up to userspace. By entry point: (i) upcall xlate (one per kernel miss); (ii) revalidator xlate (re-evaluation of an installed megaflow to check invalidation); (iii) trace xlate (driven by `ofproto/trace` for debug only, with no megaflow install).

### Usage workflow

In production, xlate runs in the background; an engineer does not call it directly but interacts with it through tools. A typical debug workflow: (1) capture the original flow of the packet with `ovs-tcpdump` or `ovs-appctl dpif/dump-flows`; (2) run `ovs-appctl ofproto/trace br0 '<flow>'` so xlate translates the flow and prints the trace; (3) read the `Megaflow:` and `Datapath actions:` lines at the end of the trace to see what is actually installed; (4) if it differs from the expectation, follow the `goto_table N` lines back to find the wrong rule. A typical performance observation workflow: enable `ovs-appctl vlog/set xlate:dbg`, run the workload, and read the log; note that the log is verbose, so enable it only briefly.

### When something goes wrong

Common symptoms of an xlate problem: a log message `xlate_actions: over MAX_DEPTH limit`, a packet not reaching the destination although the rule is correct, `ovs-vswitchd` CPU pinned at 100 percent for a long time without a PPS increase, or `coverage/show` showing `xlate_actions_too_many_output` rising fast. Action: run `ofproto/trace` for a representative flow, and count how many `Resubmitted` and `Recirc` lines appear; more than 10 to 20 for one packet is a sign that the pipeline has a branch that is exploding. When you suspect emulation explosion, compare `dpif/show-dp-features` against the OVS version to know which feature is being emulated.

### Tightly related to

Xlate couples with: `classifier` (input, supplies the rule); `dpif` (output, installs the megaflow); `recirc_id` (allocates an ID when freezing); `conntrack` (visited at every `OFPACT_CT`); `revalidator` (calls xlate again to re-evaluate a megaflow); `ofproto-dpif-upcall` (the main caller); `xlate-cache` (cache invalidation hint for the revalidator); `ofproto/trace` (the debug shell). Any structural change to a rule in the classifier invalidates the xlate cache; any change to the datapath capability forces xlate to change its emulation result.

### Version differences

- **OVS 2.4 (2015):** added recursion emulation through `recirc_id` for datapaths that support only one table.
- **OVS 2.6 (2016):** added complete `OFPACT_CT` dispatch (zone, nat, helper).
- **OVS 2.8 (2017):** added NSH encap and decap.
- **OVS 2.13 (2020):** stabilised `frozen_state` and granular `xlate-cache` invalidation.
- **OVS 2.17.x (the 2022 baseline of the curriculum):** `MAX_DEPTH = 64`, `MAX_RESUBMITS = 4096`; signature `xlate_table_action(struct xlate_ctx *, ofp_port_t in_port, uint8_t table_id, bool may_packet_in, bool honor_table_miss, bool with_ct_orig, bool is_last_action, xlate_actions_handler *)`.
- **OVS 3.x:** added an optimisation in `compose_output_action()` for `is_last_action`, with no semantic change.

### Observation and verification


The last two lines reveal the xlate result: `Megaflow` is the wildcarded mask that will be installed into the kernel; `Datapath actions` is the byte-code that actually runs. Cross-check with `ovs-appctl coverage/show | grep -E 'xlate_|recirc'` to see the event counts by category.

### Source code reference

- `ofproto/ofproto-dpif-xlate.h` declares `enum xlate_error xlate_actions(struct xlate_in *, struct xlate_out *);`, the single entry point.
- `ofproto/ofproto-dpif-xlate.c`: `do_xlate_actions()` is the main dispatch by `ofpact->type`; `xlate_table_action()` is the table recursion; `compose_output_action()` emits the output action with bonding and truncate; `xlate_commit_actions()` flushes the buffer; `xlate_action_set()` handles the OpenFlow 1.1+ action set.
- The constants `#define MAX_DEPTH 64` and `#define MAX_RESUBMITS (MAX_DEPTH * MAX_DEPTH)` are at the top of `ofproto-dpif-xlate.c` (around lines 138 to 142 in the v2.17.9 baseline).
- `struct xlate_ctx` contains `int depth`, `int resubmits`, `struct ofpbuf frozen_actions`, and `struct ofpbuf *odp_actions`.
- `lib/odp-execute.c` is the consumer on the userspace datapath side: it reads the `odp_actions` produced by xlate and executes them in sequence.

### Real-world incident

An OpenStack production site saw the following symptom. After 200 new security group rules were pushed, VM throughput started dropping, and `top` showed `ovs-vswitchd` consuming 95 percent of one CPU core. Deeper investigation: the new rules caused the OVN pipeline to produce a cross-table `resubmit` chain between the ingress ACL table and conntrack, with more than 30 steps per packet. `coverage/show` showed `xlate_actions_resubmit_recursion` rising by tens of thousands per second, and the log printed "translation oversized" warnings. Immediate fix: split the ingress ACL table into two stages to reduce the resubmit width. Root cause: a rule design that did not account for the xlate cost. Operational lesson: every rule added to the pipeline carries a hidden xlate cost, not just a classifier lookup cost.

### Synthetic exercise

A Mininet lab on one bridge `br0` with two hosts `h1` and `h2`. Step 1: create the bridge with `sudo ovs-vsctl add-br br0` and attach two `veth` ports. Step 2: install multi-table rules with `sudo ovs-ofctl add-flow br0 'table=0,priority=100,actions=goto_table:1'` and `sudo ovs-ofctl add-flow br0 'table=1,priority=50,ip,nw_dst=10.0.0.2,actions=output:2'`. Step 3: run `sudo ovs-appctl ofproto/trace br0 'in_port=1,dl_type=0x0800,nw_dst=10.0.0.2'` and observe both steps `0.` and `1.` in the trace, along with the line `Datapath actions: 2`. Step 4: add a recursive cross `goto_table:2` repeated 60 times to hit `MAX_DEPTH` and observe the `over MAX_DEPTH` message. Step 5: run `ovs-appctl coverage/show | grep xlate_actions_resubmit_recursion` before and after to see the counter jump.

### Common failures and diagnostic signals

- **Resubmit cycle from a wrong rule:** the trace stops with "Recursion too deep"; the counter `xlate_actions_resubmit_recursion` rises. Fix the rule.
- **Emulation explosion after a kernel upgrade:** the older datapath no longer supports an action natively; every packet generates several recircs; userspace CPU rises. Upgrade the kernel module or roll back the rule that uses that feature.
- **Slow-path stuck:** xlate sets `slow_path_reason`, no megaflow is installed, every packet goes to userspace. Check whether the rule keeps calling `controller(...)` or BFD continuously.
- **Frozen state leak:** `recirc_refs` are not released correctly, and the `recirc_id` pool runs out. Watch `ovs-appctl upcall/show` for `recirc allocations`.

### Comparison with other systems

Cisco IOS translating an ACL or a route-map from the control plane down to TCAM hardware shows a similar concept. The control plane writes a rich rule, and the "compiler" inside NX-OS translates it down to TCAM byte-code with finite capability. When the capacity is exceeded, traffic falls back to the CPU. Linux netfilter `nf_tables`, which JIT-compiles a chain rule into BPF bytecode, follows the same spirit with a different medium (a BPF VM instead of a datapath cache). VPP from FD.io is the opposite: graph nodes are compiled ahead of time, with no run-time translation layer, trading flexibility for performance. Xlate sits in the middle: per-flow run-time translation, with a megaflow cache that amortises the cost.

## §9.32.4 dpif, the datapath interface abstraction

### Concept

`dpif` is the abstraction layer in OVS userspace that lets `ofproto-dpif` (the OpenFlow-to-datapath byte-code translator) operate against any datapath backend through a unified interface. Concretely, instead of `ofproto-dpif` issuing a Netlink syscall directly into the kernel `openvswitch.ko` or calling a DPDK API into a userspace netdev, every operation (port add and del, flow add and del and dump, packet send and recv, conntrack execute) goes through a virtual function table `struct dpif_class`. Each concrete backend (kernel, userspace, AF_XDP, DPDK) registers an implementation of `dpif_class` at startup.

### History and background

When OVS first shipped in 2009, it had only one datapath: the kernel `openvswitch.ko` module accessed through Netlink. The userspace file backing that path was `lib/dpif-linux.c`. However, the `dpif` abstraction (`lib/dpif.c` plus `lib/dpif.h`) and the userspace native datapath `lib/dpif-netdev.c` were added by Pettit and Pfaff from day one. All three files appeared between 2009-06-17 and 2009-06-19, before the OVS 1.0 release in December 2009. The verified upstream history is `lib/dpif-netdev.c` first commit `72865317a41d065fcc47a33fc68cdd2081cecb3d`, dated 2009-06-19, "New implementation of userspace datapath, based on the netdev library". On 2014-09-18, commit `93451a0a81b40c480115abd8739c1582e6b49a9c` renamed `lib/dpif-linux.c` to `lib/dpif-netlink.c` so the file would compile with MSVC. OVS 2.4 (2015-08-20) was the release in which the DPDK netdev (`lib/netdev-dpdk.c`) reached stable first-class status through the `dpif-netdev` path. It was not the release in which `dpif-netdev` was split out, because that file had already existed for six years. OVS 2.8 and later added the AF_XDP backend through the same `dpif` mechanism (note: AF_XDP is out of the curriculum scope per the CLAUDE.md ban directive; only the structural existence is mentioned).

### Position in the architecture

`dpif` sits in the OVS userspace layer, exactly between two components: above is `ofproto-dpif` (the consumer, holding xlate, revalidator, and handler threads); below is the concrete datapath backend (the provider, one of: kernel `openvswitch.ko` through Netlink, userspace `dp-netdev` through direct call, DPDK PMD through a poll-mode driver). It belongs to the `ovs-vswitchd` daemon process. Each bridge has one `dpif` instance (`struct dpif`) bound to a specific `dpif_class` through the `datapath_type` field in the OVSDB `Bridge` row.

### Role

`dpif` plays the "narrow waist" role in the OVS userspace stack: every datapath operation goes through exactly one interface; adding a new backend only requires implementing `dpif_class`, not modifying `ofproto-dpif`. Concrete responsibilities: (a) abstract port lifecycle (`port_add`, `port_del`, `port_query_by_number`); (b) abstract flow management (`flow_put`, `flow_del`, `flow_dump_create`); (c) abstract packet I/O (`recv` to receive an upcall, `execute` to inject a packet); (d) abstract introspection (`operate`, `dpif_show` statistics).

### Why it exists

Before `dpif`, the `ofproto` code was tightly coupled to kernel Netlink. When the DPDK userspace datapath was proposed in 2013 to 2014 (Pravin Shelar, Pfaff RFC), if the coupling had remained, the project would have had to either duplicate ofproto for each backend (a maintainability nightmare) or use a compile-time toggle with tangled `#ifdef` blocks (bug-prone). Pettit and Pfaff proposed splitting the abstraction along the Strategy design pattern (a vtable virtual dispatch). Design reasons: (1) allow multiple backends to run side by side in one `ovs-vswitchd` (rare, but possible for testing); (2) allow ofproto unit tests to use a `dpif-dummy` mock (no kernel module and no DPDK runtime needed); (3) allow a third-party backend (for example, a hardware offload gateway) to plug in without modifying upstream OVS.

### Problems solved

Four specific problems addressed by `dpif`. (1) **Tightly coupled architecture**: before `dpif`, every Netlink syscall was scattered through the ofproto code; after `dpif`, the calls are concentrated in `dpif-netlink.c`. (2) **Backend evolution**: the kernel `openvswitch.ko` deprecation roadmap (the mainline kernel module is removed in mainline post v2.x) needed an exit path to userspace; `dpif` allows the migration without rewriting ofproto. (3) **Test isolation**. ofproto-dpif unit tests use the mock netdev at `lib/netdev-dummy.c` (verified at v2.17.9) together with the `lib/dummy.c` framework. They do not need root permission, do not need a kernel module to be loaded, and complete in a few seconds on a CI laptop. (4) **Capability divergence**: the kernel supported `ct()` from OVS 2.5; the userspace netdev supported `ct()` later; the xlate engine reads the capability matrix through `dpif_class->cleanup_required`, `synced_dp_layers`, or a per-feature query method to decide between emulation and delegation.

### Importance

`dpif` is a core concept of OVS internals. An engineer who does not understand `dpif` cannot tell apart: (a) the code path into the kernel datapath versus the userspace datapath; (b) why the same OpenFlow rule behaves differently on two hosts (`datapath_type=system` versus `datapath_type=netdev`); (c) why `ovs-appctl dpif/show` prints different output per backend; (d) why migration from kernel to userspace requires changing `datapath_type` at bridge creation, not at runtime. Among the OVS core concepts, `dpif` stands alongside `megaflow`, `recirc_id`, and `ofproto-dpif xlate`.

### How it works

`dpif` operates in four clear stages. **Stage 1, provider registration:** when `ovs-vswitchd` starts, `dp_register_provider(&dpif_X_class)` is called for each compiled-in backend (`dpif-netlink.c::dpif_netlink_class`, `dpif-netdev.c::dpif_netdev_class`); the list is stored in `base_dpif_classes[]`. **Stage 2, open dpif:** when `ovs-vsctl add-br br0` creates a new bridge, ofproto calls `dpif_open(name, type, &dpifp)` with `type` taken from `Bridge.datapath_type`; this function looks up the class by type and calls `class->open(name, &dpif)` to initialise the concrete `struct dpif`. **Stage 3, runtime operations:** every operation (for example, `dpif_flow_put`, `dpif_port_add`) goes through the macro dispatch `dpif->dpif_class->flow_put(dpif, ...)`. **Stage 4, close dpif:** when the bridge is deleted, `dpif_close(dpif)` calls `class->close(dpif)` to clean up resources. The full lifecycle uses locking and reference counting to make sure multiple threads (handler, revalidator, xlate) can use the dpif safely.

### How an operator masters this

Four essential skills. (1) **Read the current backend:** `ovs-vsctl get Bridge br-int datapath_type` shows which backend the bridge uses (`system` for netlink, `netdev` for userspace native, `netdev` plus `other_config:dpdk-init=true` for DPDK). (2) **Switch backend (only at bridge creation):** `ovs-vsctl set Bridge br0 datapath_type=netdev` then restart `ovs-vswitchd`; note that the backend cannot be changed in place while the bridge is active, because the datapath state is not transferable. (3) **Inspect dpif state:** `ovs-appctl dpif/show` prints the per-bridge backend type, port list, flow count, and packet statistics. (4) **Debug capability mismatch:** when a flow rule fails with "unsupported feature", grep `ovs-vswitchd.log` for the line `xlate: action <X> not supported by dpif <type>`; this typically happens when a feature exists in the kernel but not yet in the userspace netdev (for example, conntrack ALG or sample group).

### Classification

`dpif` is an **abstraction layer** in OVS userspace; it is not a CLI tool, not an OVSDB table, and not an OpenFlow primitive. Concretely, it is a virtual function table (`struct dpif_class`) plus a consumer-side wrapper (`struct dpif`). In the OVS taxonomy, `dpif` belongs to the "internal abstraction layer" group, alongside `ofproto-class` (the OpenFlow-to-bridge implementation interface) and `netdev-class` (the OVS-to-physical-NIC interface). It differs from data plane abstractions (such as megaflow and ukey) and from management plane abstractions (such as the OVSDB schema row).

### Usage workflow

A typical workflow when a developer extends OVS with a new backend: (1) define `struct dpif_class new_class = { .type = "newdp", .open = newdp_open, ... };` in a new file `lib/dpif-newdp.c`. (2) Implement every required callback in `dpif_class` (open, close, flow_put, flow_del, port_add, recv, and so on); optional callbacks can be null. (3) Register at init: call `dp_register_provider(&new_class)` from `dpif_init()` in `lib/dpif.c`. (4) Build by adding `lib/dpif-newdp.c` to `libopenvswitch_la_SOURCES` in `Makefile.am`. (5) Test with `ovs-vsctl set Bridge br0 datapath_type=newdp`. (6) Verify through `ovs-appctl dpif/show`.

### When something goes wrong

An on-call operator should remember `dpif` when seeing these symptoms: (a) the packet does not pass through the bridge even though the flow rule looks correct, check whether `dpif_open` returned success (tail `ovs-vswitchd.log`); (b) `ovs-ofctl dump-flows` lists a rule but `ovs-appctl dpctl/dump-flows` does not show the corresponding entry, this is a capability mismatch between the userspace classifier and the kernel or netdev datapath; (c) bridge creation fails with "no such datapath type", the backend module is not loaded (`lsmod | grep openvswitch`) or DPDK is not initialised.

### Tightly related to

`dpif` couples tightly with five components. (1) `ofproto-dpif` is the main consumer; every xlate decision finally emits through the `dpif_*` API. (2) `openvswitch.ko` (the kernel module) is the main provider for `dpif-netlink`; module load and unload state affects `dpif_open`. (3) The Netlink genl families `ovs_datapath`, `ovs_vport`, `ovs_flow`, and `ovs_packet` are the wire protocol between `dpif-netlink.c` and the kernel module. (4) The DPDK netdev abstraction (`lib/netdev-dpdk.c`) is the provider for `dpif-netdev` in DPDK mode. (5) `Bridge.datapath_type` in OVSDB is the configuration that decides which class to instantiate.

### Version differences

OVS 1.0 (2009) had only the Netlink kernel path; there was no abstraction. OVS 2.4 (2015) officially split out the userspace `dpif-netdev.c`; the type `netdev` appeared in OVSDB. OVS 2.6+ added conntrack support in the userspace dpif (catching up with the kernel `ct()` introduced in 2.5). OVS 2.8+ added an AF_XDP backend (not covered in this curriculum per the ban directive). OVS 3.0 (2022) marked the kernel module deprecation roadmap in mainline (the module will be removed from the mainline kernel, and OVS will ship it out-of-tree); `dpif-netlink` is still maintained, but userspace `dpif-netdev` is recommended for new deployments.

Verify `dpif` at runtime through three methods. **Method 1, dpif/show:**

### Real-world incident

Production case (curated, an OpenStack 2022-Q3 reproducible pattern): after a compute node upgraded its Linux kernel from 5.4 to 5.15 and `ovs-vswitchd` was restarted, every bridge with `datapath_type=system` reported `dpif_open` failing with errno `EOPNOTSUPP`. Symptom: VMs on the node lost connectivity, `ovs-vsctl show` listed the bridge but `ovs-appctl dpif/show` reported "no such datapath". Investigation: `lsmod | grep openvswitch` did not show the module loaded; `dmesg | tail` showed "openvswitch: module verification failed: signature and/or required key missing". Root cause: kernel 5.15 enforces strict module signing; the OVS module compiled out-of-tree was not signed by the distribution key. Fix: install `linux-modules-extra-5.15` from the Ubuntu repository (which carries a signed module), or disable the signing requirement (high risk in production).

### Synthetic exercise

Lab dpif backend switch (marked "lab-pending" for live verification when the lab host is available):
1. Create a new bridge with the default backend: `ovs-vsctl add-br br-test`. Verify with `ovs-vsctl get Bridge br-test datapath_type`, which should return `system`.
2. Observe the dpif: `ovs-appctl dpif/show | grep br-test`. Note the hit and missed counters.
3. Delete the bridge: `ovs-vsctl del-br br-test`.
4. Recreate with a userspace backend: `ovs-vsctl add-br br-test -- set Bridge br-test datapath_type=netdev`.
5. Observe the dpif again: `ovs-appctl dpif/show | grep br-test`. Compare.
6. Reasoning: why do the hit and missed counters have a different scale? The kernel datapath uses a fast-path lookup inside the kernel, while userspace `dpif-netdev` uses a polling thread; the same packet rate is handled by a different mechanism.

### Common failures and diagnostic signals

Five typical failures. (1) **`dpif_open` fails with "no such datapath":** the backend module is not loaded; signal in `ovs-vswitchd.log` reads "no datapath provider found"; recovery: `modprobe openvswitch` (kernel) or DPDK init. (2) **`dpif_open` fails with EACCES:** the process does not have `CAP_NET_ADMIN`; signal: `getcap $(which ovs-vswitchd)`; recovery: install a systemd unit with `AmbientCapabilities=CAP_NET_ADMIN`. (3) **Capability mismatch, xlate fails:** a feature exists in the kernel but not in userspace; signal in the log: "action <X> unsupported by dpif netdev"; recovery: change the backend or work around the OpenFlow pipeline. (4) **Half-migration:** changing `datapath_type` while the bridge is active has no effect; signal: `dpif/show` still shows the old backend; recovery: `del-br` and `add-br` again. (5) **Module signing reject (strict on kernel 5.15+):** signal in `dmesg`: "module verification failed"; recovery: install `linux-modules-extra` or disable strict signing (not recommended).

### Comparison with other systems

`dpif` is design-similar to the Linux kernel VFS (Virtual File System): both are abstraction layers that allow multiple backends to register and run side by side, with the consumer (userspace `ofproto-dpif` is analogous to a userspace process performing file operations) calling through a unified interface (`struct dpif_class` is analogous to `struct file_operations`). It differs from Linux netfilter (each hook has its own chain, not a vtable abstraction). It differs from VPP graph nodes (FD.io VPP has a graph runtime that registers nodes, with each node implementing input and output; the idea is similar but VPP focuses on performance through batch-per-node, while `dpif` focuses on modularity). It differs from the Cisco IOS-XE software model (Cisco has a DRIVER and HAL abstraction, but it is closed source and not publicly documented). The closest Linux kernel analogue is `struct net_device_ops` for NIC drivers, with the same vtable virtual dispatch idea.

## 20.3.1 Category 1, daily health check (the morning routine)

The operator opens the working day with five health check commands that take a snapshot of the cluster and decide whether everything is healthy or whether something needs attention. Each command takes less than one second; the total is under ten seconds.

### 20.3.1.1 Full topology snapshot, `ovn-nbctl show`

**Why run it:** This is the first command every morning. It confirms that NBDB intent has not drifted overnight. It is also useful when taking over a shift from the previous on-call engineer.

**Command and output** *(reproduced from an OVN 22.03.8 sandbox with three LSPs and one LR)*:

```shell
$ ovn-nbctl show

switch e83f20e7-0001-4a2b-9c3d-000000000001 (ls_prod_web)
    port lsp_web_01
        addresses: ["aa:bb:cc:00:00:01 10.10.1.10"]
    port lsp_web_02
        addresses: ["aa:bb:cc:00:00:02 10.10.1.11"]
    port ls_prod_web-rtr
        type: router
        router-port: rtr_prod-web

switch e83f20e7-0002-4a2b-9c3d-000000000002 (ls_prod_db)
    port lsp_db_01
        addresses: ["aa:bb:cc:00:01:01 10.10.2.10"]
    port ls_prod_db-rtr
        type: router
        router-port: rtr_prod-db

router 9f8e7d6c-5b4a-3928-1706-000000000001 (rtr_prod)
    port rtr_prod-web
        mac: "00:00:00:ff:fe:01"
        networks: ["10.10.1.1/24"]
    port rtr_prod-db
        mac: "00:00:00:ff:fe:02"
        networks: ["10.10.2.1/24"]
    port rtr_prod-external
        mac: "00:00:00:ff:fe:10"
        networks: ["192.168.100.50/24"]
        gateway chassis: [chassis-gw-1 chassis-gw-2]
    nat 3c4d5e6f-7890
        external ip: "192.168.100.100"
        logical ip: "10.10.1.0/24"
        type: "snat"
```

**Anatomy block:**

| Line pattern | Sample value | Meaning | What to watch for |
|--------------|-------------|---------|---------------------|
| `switch <uuid> (<name>)` | `switch e83...001 (ls_prod_web)` | Logical Switch header: UUID plus name. Each LS is an isolated L2 broadcast domain | An unusual UUID (for example all zeros) may be an error placeholder |
| `port <lsp_name>` | `port lsp_web_01` | An LSP that belongs to this LS, indented one level | A missing port means the VM has been detached; an extra port is an orphan that has not been cleaned up |
| `addresses: [...]` | `["aa:bb:cc... 10.10.1.10"]` | The MAC plus IP assigned to the LSP, in the format `"<MAC> <IP1> <IP2> ..."` | `dynamic` means IPAM-managed; `unknown` means promiscuous flood mode |
| `port <name>` plus `type: router` | `type: router` | An LSP that bridges into a Logical Router via a patch port | Must pair with an LRP of the same name |
| `router <uuid> (<name>)` | `router 9f8...001 (rtr_prod)` | Logical Router header | Each tenant typically has one LR |
| `port <lrp_name>` plus `mac` plus `networks` | `networks: ["10.10.1.1/24"]` | An LRP with the gateway IP for the LS attachment | The MAC plus networks must be unique across the LR |
| `gateway chassis: [...]` | `[chassis-gw-1 chassis-gw-2]` | The list of gateway chassis for the distributed gateway port | If missing, no HA, and traffic is centralised through a single chassis |
| `nat <uuid>` plus `external_ip / logical_ip / type` | `type: "snat"` | A NAT rule: `snat`, `dnat`, or `dnat_and_snat` | If missing, the tenant cannot reach external networks |

**Failure modes:**

- **A port shown as `<name>-rtr` with `type: router` but no matching LR pair.** This is an orphan router port. Fix: `ovn-nbctl lsp-del <port>` or re-add the LR.
- **A switch with no ports.** This is an orphan LS. It is safe to delete if no CMS reference remains: `ovn-nbctl ls-del <ls_name>`.
- **A router missing its `gateway chassis`.** Northbound traffic is centralised through a random chassis instead of being HA. Fix: `ovn-nbctl lrp-set-gateway-chassis <lrp> <chassis> <priority>`.
- **A NAT rule whose `external_ip` is duplicated across two LRs.** This is a cluster conflict and a CMS bug. Verify with: `ovn-nbctl --bare --columns=external_ip,logical_ip list NAT | sort | uniq -d`.

**Upstream source:** [`ovn-nbctl(8)` §show](https://man7.org/linux/man-pages/man8/ovn-nbctl.8.html).

### 20.3.1.2 Chassis liveness, `ovn-sbctl show` plus `list Chassis_Private`

```shell
$ ovn-sbctl show

Chassis "chassis-hv-01"
    hostname: hv01.dc01.example.com
    Encap geneve
        ip: "10.0.0.11"
        options: {csum="true"}
    Port_Binding lsp_web_01
    Port_Binding lsp_db_01

Chassis "chassis-hv-02"
    hostname: hv02.dc01.example.com
    Encap geneve
        ip: "10.0.0.12"
        options: {csum="true"}
    Port_Binding lsp_web_02

Chassis "chassis-gw-01"
    hostname: gw01.dc01.example.com
    Encap geneve
        ip: "10.0.0.21"
    Port_Binding cr-lrp-external
```

Reading: three chassis registered, four Port_Bindings distributed. The gateway chassis has claimed `cr-lrp-external` (a chassisredirect port, see Part 13.5).

**Verify the chassis last-seen timestamp (Chassis_Private):**

```shell
$ ovn-sbctl --bare --columns=chassis_name,nb_cfg_timestamp list Chassis_Private

chassis-hv-01   1714069800123
chassis-hv-02   1714069799987
chassis-gw-01   1714069801245
```

`nb_cfg_timestamp` is the Unix millisecond timestamp of the last time the chassis reported its state. If it lags more than 10 seconds behind the current time, the chassis is stale (suspect a BFD partition).

```shell
$ now_ms=$(date +%s%3N)
$ ovn-sbctl --bare --columns=chassis_name,nb_cfg_timestamp list Chassis_Private \
    | awk -v now=$now_ms '{delta=(now-$2)/1000; if (delta>10) print $1, delta "s stale"}'
```

### 20.3.1.3 northd plus ovn-controller liveness, `ovn-appctl status`

```shell
$ sudo ovn-appctl -t ovn-northd status
active

$ sudo ovn-appctl -t ovn-northd is-paused
false

# On each chassis
[chassis-hv-01]$ sudo ovn-appctl -t ovn-controller connection-status
connected
```

Expected: one `active` northd, `false` for paused, and every chassis `connected`. Anything else, page the team (see Part 20.2 §20.2.6).

### 20.3.1.4 The nb_cfg sync counter

```shell
$ ovn-nbctl get NB_Global . nb_cfg
1482

$ ovn-sbctl get SB_Global . nb_cfg
1482

$ ovn-sbctl get SB_Global . nb_cfg_timestamp
1714069801245
```

When `NB_Global.nb_cfg` equals `SB_Global.nb_cfg`, northd has finished compiling every intent. A drift of one or two is acceptable transiently; a drift greater than 10 means northd is lagging and should be investigated.

### 20.3.1.5 Logical flow count sanity check

```shell
$ ovn-sbctl lflow-list | wc -l
2847

$ ovn-sbctl --columns=logical_port list Port_Binding | grep -c logical_port
42
```

Rule of thumb: roughly 50 to 100 Logical_Flows per LSP, depending on the ACL plus LB count. If `lflow_count / port_count` exceeds 200, the design is over-complex; consider consolidating ACLs through a Port_Group.

**Daily health checklist (copy and paste):**

```bash
#!/bin/bash
# /usr/local/bin/ovn-health-check.sh
set -e
echo "=== NBDB topology ==="
ovn-nbctl show | head -30
echo ""
echo "=== Chassis liveness ==="
ovn-sbctl show | head -20
echo ""
echo "=== northd status ==="
sudo ovn-appctl -t ovn-northd status
sudo ovn-appctl -t ovn-northd is-paused
echo ""
echo "=== nb_cfg sync ==="
nb=$(ovn-nbctl get NB_Global . nb_cfg)
sb=$(ovn-sbctl get SB_Global . nb_cfg)
echo "NB=$nb SB=$sb"
[[ $nb -eq $sb ]] || echo "WARNING: nb_cfg drift"
echo ""
echo "=== Lflow stats ==="
echo "lflow count: $(ovn-sbctl lflow-list | wc -l)"
echo "port count: $(ovn-sbctl --columns=logical_port list Port_Binding | grep -c logical_port)"
```

## 20.3.2 Category 2, inventory listing (what is deployed)

There are five "what is in the cluster" questions the operator must answer daily, or whenever an audit is required. Each question maps to one listing command.

### 20.3.2.1 List all Logical Switches

```shell
$ ovn-nbctl ls-list
e83f20e7-0001 (ls_prod_web)
e83f20e7-0002 (ls_prod_db)
e83f20e7-0003 (ls_staging_web)

$ ovn-nbctl --bare --columns=name list Logical_Switch | sort
ls_prod_db
ls_prod_web
ls_staging_web
```

Two forms: short UUID plus name (the default), or the bare output (for grep or sort in scripts).

### 20.3.2.2 List LSPs per LS

```shell
$ ovn-nbctl lsp-list ls_prod_web

a1b2c3d4-1111-2222-3333-aaaaaaaaaa02  lsp_web_02
a1b2c3d4-1111-2222-3333-aaaaaaaaaa03  ls_prod_web-rtr
```

$ for ls in $(ovn-nbctl --bare --columns=name list Logical_Switch); do

## 20.3.3 Category 3, port lifecycle (bind, unbind, migrate)

Operators frequently add, remove, or migrate LSPs to serve CMS requests. There are six basic workflows.

### 20.3.3.1 Add a new LSP with MAC plus IP

```shell
# Create the LSP in an existing LS
$ ovn-nbctl lsp-add ls_prod_web lsp_web_03

# Assign MAC plus IP (static)
$ ovn-nbctl lsp-set-addresses lsp_web_03 "aa:bb:cc:00:00:03 10.10.1.12"

# Optional: enable port security
$ ovn-nbctl lsp-set-port-security lsp_web_03 "aa:bb:cc:00:00:03 10.10.1.12"

# Verify
$ ovn-nbctl lsp-get-addresses lsp_web_03
aa:bb:cc:00:00:03 10.10.1.12
```

Dynamic IP (IPAM-managed, see Part 13.12):

```shell
$ ovn-nbctl lsp-set-addresses lsp_web_04 "aa:bb:cc:00:00:04 dynamic"
# OVN automatically assigns an IP from the LS exclude_ips range
```

### 20.3.3.2 Bind an LSP to an OVS interface (chassis-side)

After the LSP is created in NBDB, the chassis must bind it through an OVS interface:

```shell
[chassis-hv-01]$ sudo ovs-vsctl add-port br-int tap-web-03 \
    -- set Interface tap-web-03 type=internal \
       external_ids:iface-id=lsp_web_03
```

ovn-controller detects `external_ids:iface-id`, matches the LSP name, and updates `Port_Binding.chassis` in SBDB. Verify after two to five seconds:

```shell
$ ovn-sbctl --bare --columns=chassis list Port_Binding lsp_web_03
<chassis-hv-01-uuid>
```

### 20.3.3.3 Remove an LSP

```shell
# Delete from NBDB
$ ovn-nbctl lsp-del lsp_web_03

# Verify it is gone
$ ovn-nbctl lsp-list ls_prod_web | grep lsp_web_03
# (no output)

# Clean up the OVS interface on the chassis (if the CMS has not done so)
[chassis-hv-01]$ sudo ovs-vsctl del-port tap-web-03
```

### 20.3.3.4 Rename an LSP (rare; CMSes usually do not rename)

OVN has no built-in rename. The workaround:

```shell
$ ovn-nbctl lsp-del lsp_old_name
$ ovn-nbctl lsp-add ls_prod_web lsp_new_name
$ ovn-nbctl lsp-set-addresses lsp_new_name "aa:bb:cc:00:00:05 10.10.1.14"
```

### 20.3.3.5 Migrate an LSP across chassis (manual)

A CMS-driven migration is typically automatic. A manual migration is for incident remediation:

```shell
# Step 1: Unbind on chassis-A (unplug the tap)
[chassis-hv-01]$ sudo ovs-vsctl del-port tap-web-03

# Step 2: ovn-controller on chassis-A releases Port_Binding.chassis automatically within five seconds
$ ovn-sbctl --bare --columns=chassis list Port_Binding lsp_web_03
# (empty)

# Step 3: Bind on chassis-B
[chassis-hv-02]$ sudo ovs-vsctl add-port br-int tap-web-03 \
    -- set Interface tap-web-03 type=internal \
       external_ids:iface-id=lsp_web_03

# Step 4: Verify the claim by chassis-B
$ ovn-sbctl --bare --columns=chassis list Port_Binding lsp_web_03
<chassis-hv-02-uuid>
```

### 20.3.3.6 Disable an LSP temporarily (administrative down)

There is no need to delete if you only want to pause forwarding:

```shell
# Disable
$ ovn-nbctl lsp-set-enabled lsp_web_03 false

# Verify
$ ovn-nbctl lsp-get-enabled lsp_web_03
false

# Re-enable
$ ovn-nbctl lsp-set-enabled lsp_web_03 true
```

When `enabled=false`, the stage `ls_in_check_port_sec` fires `drop;` for ingress from this port. Traffic does not enter the pipeline, but the LSP still exists, preserving any ACL, NAT, or LB references.

## 20.3.4 Category 4, ACL management

ACLs are the most important policy primitive for security. Five common scenarios follow.

### 20.3.4.1 Add ACLs (allow, drop, allow-related, reject)

```shell
# allow-related: stateful, with replies auto-permitted (see Part 13.3)
$ ovn-nbctl acl-add ls_prod_web from-lport 1000 \
    'inport == "lsp_web_01" && ip' allow-related

# drop: silent drop
$ ovn-nbctl acl-add ls_prod_web from-lport 2000 \
    'inport == "lsp_web_01" && tcp.dst == 22' drop

# reject: with an ICMP unreachable response
$ ovn-nbctl acl-add ls_prod_web from-lport 1500 \
    'inport == "lsp_web_01" && tcp.dst == 3389' reject

# allow plain (stateless, no conntrack commit)
$ ovn-nbctl acl-add ls_prod_web to-lport 1000 \
    'outport == "lsp_web_01" && icmp' allow
```

### 20.3.4.2 List ACLs verbosely

```shell
$ ovn-nbctl acl-list ls_prod_web
from-lport  2000 (inport == "lsp_web_01" && tcp.dst == 22)  drop
from-lport  1500 (inport == "lsp_web_01" && tcp.dst == 3389)  reject log(name="reject-rdp")
from-lport  1000 (inport == "lsp_web_01" && ip)  allow-related
to-lport    1000 (outport == "lsp_web_01" && icmp)  allow
```

ACLs are sorted by priority descending (the highest-priority rule fires first).

### 20.3.4.3 Delete ACLs

```shell
# By match criteria
$ ovn-nbctl acl-del ls_prod_web from-lport 2000 \
    'inport == "lsp_web_01" && tcp.dst == 22'

# By direction plus priority
$ ovn-nbctl acl-del ls_prod_web from-lport 2000

# Delete every ACL on the LS (be careful)
$ ovn-nbctl acl-del ls_prod_web
```

### 20.3.4.4 Test an ACL before applying it, `ovn-trace`

```shell
# Simulate a packet that should match the ACL rule
$ ovn-trace ls_prod_web \
    'inport == "lsp_web_01" && eth.src == aa:bb:cc:00:00:01 &&
     eth.dst == aa:bb:cc:00:00:02 && ip4.src == 10.10.1.10 &&
     ip4.dst == 10.10.1.11 && ip.ttl == 64 && tcp && tcp.dst == 22'

# Expected output: drop; at the ls_in_acl table with priority=2000
```

This is the dry-run pattern for ACLs: validate the logic before deploying a wide change.

### 20.3.4.5 ACL audit, find a rule matching a specific traffic pattern

```shell
# Find any ACL that blocks SSH (port 22)
$ ovn-nbctl --bare --columns=direction,priority,match,action list ACL \
    | grep "tcp.dst == 22"

from-lport  2000  (inport == "lsp_web_01" && tcp.dst == 22)  drop
from-lport  2000  (inport == "lsp_web_02" && tcp.dst == 22)  drop

# Find ACLs without a log action
$ ovn-nbctl --bare --columns=action,log list ACL | awk '$2=="false" {print}'
```

### 20.3.4.6 Port_Group for ACL scale

Instead of 100 ACLs for 100 LSPs, consolidate through a Port_Group (see Part 13.3):

```shell
# Create the Port_Group
$ ovn-nbctl pg-add pg_web

# Add LSPs to the group
$ ovn-nbctl --id=@lsp1 get Logical_Switch_Port lsp_web_01 \
    -- add Port_Group pg_web ports @lsp1

$ ovn-nbctl --id=@lsp2 get Logical_Switch_Port lsp_web_02 \
    -- add Port_Group pg_web ports @lsp2

# An ACL that uses the Port_Group instead of matching individual LSPs
$ ovn-nbctl acl-add ls_prod_web from-lport 1000 \
    'inport == @pg_web && tcp.dst == 80' allow-related

# Verify
$ ovn-nbctl list Port_Group pg_web
```

## 20.3.5 Category 5, Load_Balancer and NAT

### 20.3.5.1 Create a TCP Load_Balancer

```shell
# Create an LB with one VIP and two backends
$ ovn-nbctl lb-add web-lb "10.100.0.1:443" "10.10.1.10:443,10.10.1.11:443" tcp

# Verify
$ ovn-nbctl lb-list
_uuid                          name       protocol  vips
c3d4e5f6-7890-aaaa-bbbb-111111  web-lb    tcp       {"10.100.0.1:443"="10.10.1.10:443,10.10.1.11:443"}

# Attach the LB to an LS or LR
$ ovn-nbctl ls-lb-add ls_prod_web web-lb
$ ovn-nbctl lr-lb-add rtr_prod web-lb

# Verify the attachment
$ ovn-nbctl ls-lb-list ls_prod_web
_uuid                          name       protocol  vips
c3d4e5f6-7890-aaaa-bbbb-111111  web-lb    tcp       {"10.100.0.1:443"="10.10.1.10:443,10.10.1.11:443"}
```

### 20.3.5.2 Add or remove backends from an LB

```shell
# Add a new backend
$ ovn-nbctl --vips="10.100.0.1:443" lb-add web-lb \

[chassis-hv-01]$ sudo systemctl stop ovn-controller

# 5. Perform the maintenance
# ... kernel upgrade, OVS upgrade, hardware change ...

# 6. Restart ovn-controller
[chassis-hv-01]$ sudo systemctl start ovn-controller

# 7. Verify the chassis re-registered
$ ovn-sbctl show | grep chassis-hv-01
```

### 20.3.10.4 Rolling upgrade of OVS and OVN

The standard non-disruptive procedure (see Part 9.12 for the OVS side):

```shell
# Chassis by chassis
for chassis in chassis-hv-01 chassis-hv-02 chassis-hv-03; do
    echo "=== Upgrade $chassis ==="

    # Cordon
    # (the CMS migrates LSPs away)

    # Stop services
    ssh $chassis "sudo systemctl stop ovn-controller"

    # Apt upgrade
    ssh $chassis "sudo apt update && sudo apt install -y openvswitch-switch openvswitch-common ovn-host ovn-common"

    # Restart
    ssh $chassis "sudo systemctl start ovn-controller"

    # Wait for the chassis to re-register
    sleep 30

    # Verify
    ovn-sbctl --bare --columns=name list Chassis | grep $chassis || echo "FAIL: $chassis not registered"
done
```

## 20.3.11 End-to-end workflow 1, new tenant provisioning

Scenario: provision a new tenant `tenant-42` with 1 LS, 3 LSPs, default-deny ACLs, DHCP, and SNAT.

```shell
#!/bin/bash
# new-tenant.sh tenant-42 10.10.42.0/24

TENANT=$1
CIDR=$2
GATEWAY=10.10.42.1
GATEWAY_MAC=00:00:00:ff:fe:42

# 1. Create the LR if it does not yet exist
ovn-nbctl --may-exist lr-add rtr_tenant

# 2. Create the LS
ovn-nbctl ls-add ls_${TENANT}

# 3. Add an LRP attached to the LS
ovn-nbctl lrp-add rtr_tenant rtr_tenant-${TENANT} \
    $GATEWAY_MAC $GATEWAY/24
ovn-nbctl lsp-add ls_${TENANT} ls_${TENANT}-rtr
ovn-nbctl lsp-set-type ls_${TENANT}-rtr router
ovn-nbctl lsp-set-addresses ls_${TENANT}-rtr router
ovn-nbctl lsp-set-options ls_${TENANT}-rtr router-port=rtr_tenant-${TENANT}

# 4. Create three LSPs
for i in 1 2 3; do
    MAC="aa:bb:cc:42:00:0${i}"
    IP="10.10.42.1${i}"
    ovn-nbctl lsp-add ls_${TENANT} lsp_${TENANT}_vm${i}
    ovn-nbctl lsp-set-addresses lsp_${TENANT}_vm${i} "$MAC $IP"
    ovn-nbctl lsp-set-port-security lsp_${TENANT}_vm${i} "$MAC $IP"
done

# 5. Create the DHCP options
DHCP_UUID=$(ovn-nbctl create DHCP_Options \
    cidr=$CIDR \
    options='"server_id"="10.10.42.1" "lease_time"="3600" "router"="10.10.42.1" "dns_server"="{8.8.8.8}"' \
    | tr -d '"')

# Attach DHCP to every LSP
for i in 1 2 3; do
    ovn-nbctl lsp-set-dhcpv4-options lsp_${TENANT}_vm${i} $DHCP_UUID
done

# 6. Add default-deny plus allow-related egress ACLs
ovn-nbctl acl-add ls_${TENANT} from-lport 1000 'ip' drop
ovn-nbctl acl-add ls_${TENANT} from-lport 2000 'ip.dst == 10.10.42.0/24' allow-related
ovn-nbctl acl-add ls_${TENANT} from-lport 2000 'ip.dst == 0.0.0.0/0 && ip.dst != 10.0.0.0/8' allow-related

# 7. Add a SNAT
ovn-nbctl lr-nat-add rtr_tenant snat 192.168.100.142 $CIDR

echo "Tenant $TENANT provisioned with three VMs on $CIDR"
ovn-nbctl show | grep -A 10 ls_${TENANT}
```

This workflow covers seven categories in a single script. Verify with:

```shell
$ ovn-trace ls_tenant-42 \
    'inport == "lsp_tenant-42_vm1" && eth.src == aa:bb:cc:42:00:01 &&
     eth.dst == 00:00:00:ff:fe:42 && ip4.src == 10.10.42.11 &&
     ip4.dst == 8.8.8.8 && ip.ttl == 64 && icmp4'
# Expected: output reaches the router, then SNAT, and finally the external path
```

## 20.3.12 End-to-end workflow 2, tenant teardown

Scenario: remove `tenant-42` safely. The order matters in order to avoid orphans.

```shell
#!/bin/bash
TENANT=tenant-42

# 1. Stop forwarding (disable LSPs before deleting them)
for lsp in $(ovn-nbctl --bare --columns=name find Logical_Switch_Port \
             external_ids:tenant=$TENANT); do
    ovn-nbctl lsp-set-enabled $lsp false
done

# 2. Remove LB, NAT, and DHCP references
ovn-nbctl --if-exists lb-del ${TENANT}-web-lb
for lsp in $(ovn-nbctl --bare --columns=name find Logical_Switch_Port \
             external_ids:tenant=$TENANT); do
    ovn-nbctl lsp-set-dhcpv4-options $lsp ''
done

# 3. Remove NAT rules
for nat_uuid in $(ovn-nbctl --bare --columns=_uuid find NAT \
                  external_ids:tenant=$TENANT); do
    ovn-nbctl --if-exists destroy NAT $nat_uuid
done

# 4. Remove ACLs
ovn-nbctl --if-exists acl-del ls_${TENANT}

# 5. Remove LSPs
for lsp in $(ovn-nbctl --bare --columns=name find Logical_Switch_Port \
             external_ids:tenant=$TENANT); do
    ovn-nbctl --if-exists lsp-del $lsp
done

# 6. Remove the LRP
ovn-nbctl --if-exists lrp-del rtr_tenant-${TENANT}

# 7. Remove the LS
ovn-nbctl --if-exists ls-del ls_${TENANT}

# 8. Garbage collect DHCP (if no other tenant shares it)
# (Be careful, only delete if it is unique)
# ovn-nbctl dhcp-options-del <uuid>

# 9. Verify cleanliness
ovn-nbctl show | grep -q $TENANT && echo "WARN: residue" || echo "OK tenant $TENANT removed"
```

## 20.3.16 Capstone POE, "Add 500 ACLs to one Logical_Switch: is it safe for production?"

**Scenario:** the engineering manager proposes "for fine-grained microsegmentation, add 500 ACLs to `ls_prod_web`, each with a unique priority. Is that safe?"

**Predict:** the change will affect:

1. northd compile time (500 ACLs equals 500-plus logical flows).
2. ovn-controller lflow-cache size.
3. br-int OpenFlow rule count.
4. Debug complexity.

**Observe (reasoning):**

**1.** northd compile:

```bash
# Current baseline of 10 ACLs
$ sudo ovn-appctl -t ovn-northd stopwatch/show | grep "build_lflows"
# Short term average: 20ms

# Expected with 500 ACLs:
# build_lflows about 100 to 200 milliseconds (linear scale), with the 95th percentile about 500 milliseconds.
# Still under the 1-second threshold, so OK.
```

**2.** lflow-cache:

```bash
# Current
$ sudo ovn-appctl -t ovn-controller lflow-cache/show-stats | grep "total count"
# total count: 1200

# Expected with 500 more ACLs: 2500 more logical flows (each ACL spans about five flows across stages)
# New total: about 3700
# cache-max-size default is 200K, so no stress.
```

**3.** br-int OpenFlow:

```bash
# Current
$ sudo ovs-ofctl dump-aggregate br-int | grep flow_count
# flow_count: 3200

# Expected: 7500 more OpenFlow rules (OVN expansion is about 3x logical to physical)
# New total: about 10700
# Thresholds of concern: 50K (HW offload limit) and 100K (revalidator stress).
# Still OK.
```

**4.** Debug:

```bash
# Current ACL list
$ ovn-nbctl acl-list ls_prod_web | wc -l
# 10

# Expected: 510 ACLs
# ovn-trace output grows from about 50 lines per call to about 500 lines
# Debug effort scales linearly but remains tractable.
```

**Explain:** the proposal is **acceptable with two caveats**:

- **Caveat 1:** ACL priorities must be unique but well-ordered. Random priorities make the first-match pattern unpredictable and hard to debug. Fix: group ACLs by logical category (100 to 199 for allow, 200 to 299 for explicit drop, 300-plus for default), and document the convention in a wiki.
- **Caveat 2:** instead of 500 individual ACLs per LSP, use Port_Group consolidation (see §20.3.4.6). For example, 50 port-groups with 10 ACLs each give an effective 500 ACLs but compile down to about 100 actual ACLs in NBDB. Compile time scales much better.

**Correct recommendation:**

1. Categorise ACLs into 5 to 10 Port_Groups.
2. Reserve priority blocks (100 to 199 for allow, 200 to 299 for stateful allow-related, 300 to 399 for specific deny, 900-plus for default-deny).
3. Add ACLs through a batch script applied during a maintenance window.
4. Stopwatch baseline before and after to measure the actual impact.

**Finish:** the principle is "scale ACLs through consolidation (Port_Group) plus discipline (priority blocks), not through raw count". 500 raw ACLs is a bad pattern; 500 ACLs organised through 20 Port_Groups is fine.

## 20.3.17 Common misconceptions

> **Misconception 1:** "`ovn-nbctl` and `ovn-sbctl` are interchangeable."
> **Reality:** NBDB is read-write (the operator's intent, written by the CMS). SBDB is read-mostly (compiled output, read by ovn-controller, which writes status back). `ovn-sbctl` is mainly for debugging and inspection, not modification (for example, creating a Logical_Flow through `sbctl` is the wrong path; intent must go through NBDB and northd compile).

> **Misconception 2:** "Deleting an LS also deletes every LSP inside."
> **Reality:** it depends on the version and the CMS. OVN 22.03+ offers cascade delete through the `--cascade` flag. The default behaviour can leave orphan LSPs. Always run the explicit teardown workflow in §20.3.12.

> **Misconception 3:** "`ovn-nbctl ls-add name` is idempotent: running it twice is OK."
> **Reality:** the default errors out if the LS already exists. Use the `--may-exist` flag for idempotent scripts: `ovn-nbctl --may-exist ls-add name`.

> **Misconception 4:** "`lsp-set-addresses unknown` means dynamic IPAM."
> **Reality:** `unknown` is promiscuous flood mode (it receives every MAC destination that does not match anyone). `dynamic` is what enables IPAM. Do not confuse the two.

> **Misconception 5:** "`ovn-nbctl acl-del` without arguments deletes a single specific rule."
> **Reality:** with no arguments, it deletes EVERY ACL on the LS. That is dangerous. Always specify the direction, priority, and match.

> **Misconception 6:** "Backing up NBDB is enough; SBDB is not needed."
> **Reality:** NBDB is intent and can be re-compiled into SBDB through northd. HOWEVER, SBDB has dynamic state (`Port_Binding.chassis`, chassis registration, MAC_Binding cache, Service_Monitor health) that cannot be recovered from NBDB. Back up both.

> **Misconception 7:** "Chassis maintenance means stop ovn-controller right now."
> **Reality:** drain LSPs first (migrate VMs). Stopping directly causes active LSPs to lose their claim, which blackholes traffic for 10 to 30 seconds until the CMS detects and re-schedules.

> **Misconception 8:** "Increasing the `lflow-cache` size solves every performance issue."
> **Reality:** a larger cache consumes more memory. A hit rate above 95 percent is healthy. If the hit rate is low, the root cause lies elsewhere (lflow churn, schema change, bulk update). Tuning cache size does not fix the root cause.

## 20.3.18 Key points to remember

First, the **10 task categories** cover 95 percent of operator workflow: health, inventory, port lifecycle, ACL, LB plus NAT, DHCP plus DNS, gateway plus HA, conntrack, performance, and backup plus maintenance. Mastery of all 10 marks a competent operator.

Second, the **`ovn-nbctl` subcommand prefix pattern**: `ls-` (Logical Switch), `lsp-` (LSP), `lr-` (Logical Router), `lrp-` (LRP), `acl-`, `lb-`, `lr-nat-`, `pg-` (Port_Group), `dhcp-options-`, and `ha-chassis-group-`. Each prefix has the verbs `add`, `del`, `list`, `set-*`, and `get-*`. Memorising the pattern lets you discover new subcommands by yourself.

Third, the **morning routine of five health-check commands runs in less than 10 seconds**: `ovn-nbctl show`, `ovn-sbctl show`, `ovn-appctl -t ovn-northd status`, `nb_cfg` comparison, and lflow count. Automate it through the `ovn-health-check.sh` script.

Fourth, **`--may-exist` and `--if-exists`** are important idempotent flags for script automation. Always use them in CI/CD pipelines to avoid flakes.

Fifth, **Port_Group consolidation** reduces ACL count by a factor of 5 to 10. It is the standard pattern for clusters with more than 500 LSPs. Do not scale ACLs by raw count.

Sixth, **back up both NBDB and SBDB**, not only NBDB. Use `ovsdb-client backup` or copy `/var/lib/ovn/*.db` files when services are stopped.

Seventh, **chassis maintenance follows seven steps**: inventory, cordon, drain, verify empty, stop, maintenance, restart, verify re-register. Skip none of them. Skipping drain blackholes traffic.

Eighth, **end-to-end workflow scripts** (new-tenant, tenant-teardown, rolling-upgrade) are an operator's most valuable deliverable. A checklist becomes a script, the script becomes CI, and CI becomes an SLA.

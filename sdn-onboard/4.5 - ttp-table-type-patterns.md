## Context: silicon fragmentation and its consequences

Part 4.4 analysed OF 1.5 with egress tables and packet-type awareness, but hardware adoption was effectively zero. A natural question follows: why did vendors not implement it? The previous part attributed the outcome to economics (the revenue does not justify the cost) and timing (P4 displaced it). This part goes deeper into the structural root cause: **the OpenFlow specification is designed under the assumption that silicon is generic, but in practice silicon is highly fragmented**.

The OF 1.3 specification describes a complete abstract pipeline: N flow tables, M instructions, K group types, meters, PBB fields, and packet types. However, every ASIC vendor builds silicon under economic and physical constraints: TCAM size limits, parser depth limits, pipeline stage count, and memory hierarchy. No ASIC implements the full specification; every chip supports a different subset of the specification. The consequence: a controller writes a flow_mod that is "theoretically valid" but the switch rejects it with `OFPERR_BAD_MATCH` at runtime.

TTP (Table Type Patterns) is the final ONF attempt to solve this through pattern matching: instead of the controller querying capability per feature, the switch advertises a pre-defined pattern and the controller adapts.

## 4.5.1 The problem: each vendor's silicon supports a different subset of the OF specification

### Concrete silicon constraints

Take three popular chipsets from 2013 to 2017 as examples:

**Broadcom Trident2 (BCM56850)**: the flagship data-centre switch silicon of 2013 to 2015.
- Ingress ACL TCAM: 4K to 8K entries (depending on SKU).
- Egress ACL TCAM: 256 entries.
- L2 table: 128K MAC entries.
- L3 host table: 16K IPv4 exact-match.
- LPM routing: 16K IPv4 prefix.
- No wide **egress ACL**: this restricts the per-output-port policy that OF 1.5 egress tables require.

**Intel FM6000**: an alternative chipset from 2013 to 2016.
- 16-stage programmable pipeline (modern by the standards of that era).
- Each stage has different match-field constraints: stage 1 matches L2 only; stage 2 matches L3 only; stage 3 and beyond can match a mix.
- Complex capability model: a flow_mod must align with stage ownership and cannot be arbitrary.

**Cavium Xpliant CNX**: semi-programmable silicon from the pre-P4 era (2015 to 2016).
- Programmable parser (limited depth, 96 bytes).
- Flexible match-action stages (six stages with shared TCAM).
- Still subject to constraints: actions per stage are limited, and MPLS push or pop is allowed only at specific stages.

Every ASIC has its own "capability profile" and is not plug-and-play compatible with another.

### Operational consequences

**The controller does not know** which match and action combinations the switch supports. The discovery method of OF 1.0 to 1.4:

1. Switch connects, HELLO handshake, version negotiation.
2. Controller queries `OFPT_FEATURES_REQUEST`, the switch replies with `n_tables`, port count, and supported actions.
3. Controller installs flow_mod based on what it thinks the switch supports.
4. If the flow_mod is invalid for the switch silicon, the switch returns `OFPERR_BAD_MATCH`, `OFPERR_BAD_INSTRUCTION`, or `OFPERR_BAD_ACTION` at runtime.

The bug surfaces only at **runtime**, not at compile or validation time. Controller developers then have to maintain a per-vendor capability table inside application logic. The result is not portable.

**Production impact**: when the operator upgrades the switch model (from Trident2 to Trident3) or adds a mixed-vendor fabric, the controller logic has to be rewritten. This is a serious operational pain point, and the operator is locked into a single vendor instead of enjoying "OpenFlow lets you mix vendors".

### Common misunderstanding

**Misunderstanding:** "OpenFlow is a standard protocol, so an OpenFlow-compliant switch from any vendor works with any OpenFlow controller."

**Reality:** Compliance with the OpenFlow protocol-level handshake does not equal compliance with the abstract pipeline model. Switch A (Trident2) has 8K ingress ACL but only 256 egress ACL; switch B (FM6000) has a 16-stage pipeline but each stage carries constraints. Both are "OpenFlow 1.3-compliant" but they are not equivalent in capability. A portable controller has to detect capability at runtime; it cannot rely on specification compliance. This is the gap that TTP tried to fill but did not succeed in closing.

## 4.5.2 The TTP concept: a pre-agreed pattern

### Approach: "negotiate the pattern before the flow"

Instead of continuing the discover-then-fail-at-runtime pattern, TTP proposes:

**Step 1, Define the patterns**: the ONF community defines a set of reference TTPs, each one specifying:
- The number of tables.
- The match fields per table.
- The instructions per table.
- The supported group types.
- Constraints (for example, "table 0 matches L2 only, not L3").

Every TTP has a unique identifier (UUID or URN).

**Step 2, Switch advertises**: the switch ships with a list of supported TTP identifiers and advertises them through a `FEATURES_REPLY` extension.

**Step 3, Controller selects**: the controller chooses the TTP that matches the application's needs. If the switch does not support the desired TTP, the controller falls back to an alternative TTP or fails gracefully with a clear error (rather than "try the flow_mod and hope").

**Step 4, Install constrained flows**: the controller installs flow_mod entries according to the TTP schema. The switch validates at install time (not at runtime); an invalid flow_mod is rejected immediately with a clear message.

### Analogy with the HTTP `Accept` header

An HTTP client indicates `Accept: application/json, text/html;q=0.9`. The server selects the best format that matches the client's capability and replies with `Content-Type: application/json`. Client and server agree before exchanging the payload.

TTP brings the same pattern into OpenFlow:
- The switch is the server, advertising the list of supported TTP identifiers.
- The controller is the client, requesting a specific TTP.
- The agreed TTP is the "Content-Type" for subsequent flow_mod messages.

The difference from HTTP: TTP is a pre-defined catalogue (a finite set), not a dynamic negotiation per request.

### Main intended use case: a multi-vendor fabric

Scenario: a data centre has a mixed fabric with switches from three vendors. The controller installs the "L2-L3 Edge TTP" across all of them:

- A switch from vendor X supports "L2-L3 Edge v1.0", so the controller installs the policy directly.
- A switch from vendor Y supports "L2-L3 Edge v1.0" but with lower capacity, so the controller installs a scaled-down policy.
- A switch from vendor Z supports only "L2 Only TTP v1.0", so the controller deploys L3 logic on vendors X and Y, deploys L2 only on Z, and coordinates through controller logic.

There is no need for per-vendor plugin code; application logic is written once against the TTP abstraction.

## 4.5.3 ONF TS-017: the TTP specification

### Document history

The ONF Technical Specification 017 "OpenFlow Table Type Patterns" version 1.0 was published on **2014-08-15**. The specification was developed by the Forwarding Abstractions Working Group (FAWG) of ONF. The document is approximately 60 pages and includes the YANG schema and three reference TTPs.

The historical PDF URL on opennetworking.org was removed after the website restructure (around 2018). An archive snapshot is available on web.archive.org:

```
Search keyword: "ONF TS-017 OpenFlow Table Type Patterns v1.0 August 2014"
Archive search: https://web.archive.org/web/*/opennetworking.org/*TTP*
```

The original direct PDF URL is no longer reachable on opennetworking.org and has to be located through the Wayback Machine in the date range 2015 to 2017.

### YANG-based data model

A TTP is defined in YANG (RFC 6020). YANG is a language for network configuration models, originally designed for NETCONF (see Part 5.0). TTP adopts YANG to leverage the existing tooling and to align with the broader ONF YANG strategy (OpenConfig).

A simplified schema example:

```
module ttp {
  leaf NDM_metadata { ... }
  list flow_tables {
    key table_number;
    leaf table_number { type uint8; }
    list match_set {
      leaf match_field { type ofp_match_field; }
      leaf mask_type { enum exact | prefix | arbitrary; }
    }
    list instruction_set { ... }
  }
  list group_entry_types { ... }
  list meter_table { ... }
}
```

The controller parses the YANG schema, generates a code-level representation of the TTP, and uses it when validating a flow_mod before installation.

### Reference TTPs in specification version 1.0

Specification version 1.0 bundles three reference TTPs:

**L2-L3 Edge TTP**: for a data-centre edge switch handling mixed L2 VLAN plus L3 routing.
- Table 0 (classify): match port plus VLAN, assign tenant metadata.
- Table 1 (L2 lookup): match metadata plus dst_mac, forward or flood.
- Table 2 (L3 lookup): match metadata plus dst_ip, next-hop group.
- Table 3 (ACL): match the 5-tuple, permit or deny.

**Aggregation-Core TTP**: for an aggregation-layer switch (the underlay fabric).
- Table 0 (MPLS or label match): match the MPLS label, then pop, push, or swap.
- Table 1 (routing): match dst_ip, next-hop.
- Table 2 (ECMP group): select an action across multiple next-hops.

**WAN PE (Provider Edge) TTP**: for a carrier-edge switch.
- Table 0 (classify): match VLAN or port, service instance.
- Table 1 (policy): match service instance plus the 5-tuple, QoS class.
- Table 2 (meter): apply a rate limit.
- Table 3 (encap): PBB push or MPLS push.
- Table 4 (routing): forward to the uplink.

Each TTP defines the allowed match fields, the instruction set, and the group types (ALL, SELECT, INDIRECT, FAST_FAILOVER). A switch vendor implementing a TTP only needs to align its silicon with one or more TTPs.

### Adoption reality: not widespread

In the period 2014 to 2016:

- **Vendors implementing TTP**: very rare. HP announced a partial implementation of the L2-L3 Edge TTP at a 2014 conference but did not ship production firmware. NEC was in the same situation. Broadcom did not implement a TTP layer (Broadcom considered its silicon API alone sufficient).

- **Controllers implementing TTP**: ONOS did not integrate TTP; ODL had an experimental `ttp` project that went stale in 2015. Ryu had no TTP module. Faucet built its own custom abstraction.

- **Applications**: no production SDN application was built on the TTP abstraction.

The reasons TTP did not succeed:
1. Timing: 2014 was when Flow Objectives (ONOS) and P4 (the CCR 2014 paper) appeared, and TTP was overshadowed.
2. Complexity: the YANG schema for TTP is complex, and the developer adoption cost is high.
3. Vendor politics: hardware vendors prioritised proprietary programmable APIs (NVIDIA DOCA later, Broadcom SAI) over an open TTP layer.

The specification did not see updates after 1.0. ONF stopped TTP development around 2016.

## 4.5.4 Alternative: the Flow Objectives abstraction (see Block VI)

### The ONOS Flow Objectives approach

ONOS (Open Network Operating System), from ON.Lab (later merged with ONF), released in 2015 a different architecture that solves the same problem: **the application expresses configuration intent, and the framework compiles it into a flow_mod that fits the switch**.

The abstraction layer comprises:
- **Flow Objectives API**: the application calls `forwardingObjectiveService.apply(...)` with a configuration intent such as "forward traffic matching X to port Y with priority Z".
- **Driver SDK**: every switch model has a driver class; the driver receives an objective from the framework and emits the concrete flow_mod that fits the silicon of that switch.
- **Pipeline abstraction**: the driver encodes silicon pipeline knowledge (Trident2 pipeline versus FM6000 pipeline) and maps the objective to the correct table or tables and to the right flow_mod sequence.

The controller application does not need to know silicon details; the framework abstracts them away. When a new switch model is added, only a new driver class needs to be written.

### Comparison: TTP versus Flow Objectives

| Aspect | TTP | Flow Objectives |
|--------|-----|------------------|
| Where the abstraction lives | Switch side (the switch advertises the TTP) | Controller side (framework drivers) |
| Application aware of switch capability | Partially (the application knows which TTP) | No (the framework hides it completely) |
| Extensibility | Fixed catalogue (adding a new TTP means adding to the ONF specification) | Dynamic (a new driver class is added in the controller) |
| Vendor involvement | Vendor implements the TTP in firmware | Vendor contributes a driver class |
| Deployment complexity | Requires a firmware update on the switch plus the controller side | Requires driver code on the controller side only |

Flow Objectives wins on deployment simplicity (no switch firmware change) and on extensibility (a driver can be added dynamically). ONOS adoption rose from 2015 to 2018 in carrier networks (AT&T's disaggregated CORD architecture), and Flow Objectives became the de-facto abstraction pattern.

The details of Flow Objectives are analysed in Part 6.1. It is an example of the same model, "an abstraction layer above OpenFlow", which Kubernetes SDN (OVN-Kubernetes) later adopted in a different way.

### Long-term winner: P4

In the long run, both TTP and Flow Objectives were overshadowed by **P4** (see Part 6.0). P4 does not abstract switch capability; it allows the programmer to **define** the capability. Instead of asking "does the switch support this TTP?", the programmer writes a P4 program and compiles it for the target. The model shifts from protocol-centric (OpenFlow plus TTP) to compiler-centric (P4).

The lesson from TTP: an abstraction layer over a fragmented primitive (silicon capability) is a temporary approach. The long-term solution is to homogenise the primitive (silicon accepts a programmable model) instead of abstracting over the fragmentation.

> **Key point:** TTP is a technical attempt to solve silicon fragmentation, but it failed for two combined reasons: (a) timing, P4 and Flow Objectives appeared at almost the same time with stronger value propositions, and (b) vendor incentives, hardware vendors did not want to implement a complex open-standard layer and preferred proprietary APIs or moved to programmable silicon. The lesson: a strong technical proposal does not guarantee success; timing and alignment with vendor incentives are decisive factors.

## Guided Exercise: explore the TTP concept through OVS switch capability discovery (audit P4.B4.1 lab)

**Objective:** observe why TTP failed. The capability discovery mechanism still exists in OVS but does not scale to a formal TTP catalogue.

**Steps:**

1. Query the OVS capability feature matrix:
   ```bash
   $ sudo ovs-vsctl add-br br-ttp
   $ sudo ovs-ofctl -O OpenFlow13 dump-features br-ttp | head -20
   # Output shows: supported actions, match fields, max flow entries
   ```
2. Query the datapath feature capability:
   ```bash
   $ sudo ovs-appctl dpif/show | head -20
   # Shows kernel datapath capabilities
   ```
3. Compare against the expected TTP schema (if TTP were implemented): the output format should match the YANG schema of a TTP identifier. OVS does not implement TTP, so the output is free-form text and is not parseable by a TTP catalogue.
4. Verify whether the OVS capability matches the "L2-L3 Edge TTP" (ONF TS-017 reference):
   ```bash
   # The L2-L3 Edge TTP requires:
   # - Table 0: ingress port security
   # - Table 1: L2 MAC learning
   # - Table 2: L3 routing (match src/dst IPv4)
   # - Table 3: ACL policy
   # OVS can support this pattern but does not advertise it as the "L2-L3 Edge TTP" identifier.
   $ sudo ovs-ofctl -O OpenFlow13 dump-features br-ttp | grep "n_tables"
   # n_tables: 254 (theoretically supports any TTP)
   ```
5. Cleanup: `sudo ovs-vsctl del-br br-ttp`

**POE:**
- **Predict:** if OVS implemented TTP advertisement, would the controller be able to write portable code across vendors?
- **Observe:** theoretically yes. In practice, every vendor's silicon has different constraints (TCAM size, stage limit). The TTP identifier captures functional capability only, not performance and scale constraints. The controller still needs per-vendor tuning.
- **Explain:** the TTP value proposition meets only fifty per cent of the goal: functional portability yes, performance portability no. That is not enough to justify implementation cost. P4 (Part 6.0) addressed the gap by letting the programmer define the pipeline instead of discovering it.

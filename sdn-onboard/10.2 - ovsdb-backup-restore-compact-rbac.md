## 10.2.1 The database file is append-only

Compass Chapter M states: "The DB file at `/etc/openvswitch/conf.db` is append-only: every transaction is logged."

Consequences:

- The file grows over time even when the data state does not grow. Every change (insert, update, delete) is appended to the log.
- After a long period of operation, the file can become hundreds of times larger than the actual data state.
- Compaction is therefore necessary. It rewrites the file as a snapshot of the current state.

---

## 10.2.2 Compact operations

**Offline compact** (requires stopping `ovsdb-server`):

```bash
[host]# systemctl stop openvswitch-switch
[host]# ovsdb-tool compact /etc/openvswitch/conf.db
[host]# systemctl start openvswitch-switch
```

**Live compact** (recommended by compass Chapter M):

```bash
[host]# ovs-appctl -t ovsdb-server ovsdb-server/compact
```

Live compact is atomic. The client sees no disruption. Run it on a periodic cron schedule (weekly, or whenever the file exceeds a threshold).

Check the file size:

```bash
[host]# du -h /etc/openvswitch/conf.db
[host]# ovsdb-tool show-log /etc/openvswitch/conf.db | wc -l
```

## 10.2.3 Backup and restore

**Backup:**

```bash
[host]# ovsdb-client backup > /tmp/conf.db.bak
```

The output is a valid OVSDB database file containing the current state.

**Restore:**

```bash
[host]# ovsdb-client restore < /tmp/conf.db.bak
```

**Important caveat** (compass Chapter M): restore does NOT preserve the original UUID of each row. If a client is referencing a row by UUID (for example, OVN is running), the restore will break those references. The procedure is to stop OVN, restore, then restart OVN.

Production strategy:

- Daily backup via cron, stored off-host.
- Retention of 30 to 90 days for rollback.
- Test the restore on a staging system before relying on the backup.

## 10.2.4 Schema upgrade

Check the version:

```bash
[host]# ovsdb-tool db-version /etc/openvswitch/conf.db
8.3.0
[host]# ovsdb-tool schema-version /usr/share/openvswitch/vswitch.ovsschema
8.4.0
```

Upgrade the database to match the binary schema:

```bash
[host]# systemctl stop openvswitch-switch
[host]# ovsdb-tool needs-conversion /etc/openvswitch/conf.db \
         /usr/share/openvswitch/vswitch.ovsschema
yes
[host]# ovsdb-tool convert /etc/openvswitch/conf.db \
         /usr/share/openvswitch/vswitch.ovsschema
[host]# systemctl start openvswitch-switch
```

View the transaction history:

```bash
[host]# ovsdb-tool show-log /etc/openvswitch/conf.db
```

## 10.2.5 RBAC (compass Chapter O)

OVSDB RBAC (role-based access control), introduced in OVS 2.8, restricts client access to tables and columns by role.

Configure a role on a Manager:

```bash
[host]# ovs-vsctl set Manager ssl:10.0.0.10:6640 role=ovn-controller
```

The role `ovn-controller` in the OVN schema only allows the client to update its own `Chassis` row and the `Port_Binding` rows that belong to it. The role does not allow writes to `Logical_Flow`, `Logical_Switch`, or to the configuration of other chassis.

RBAC permission model (OVN):

```json
{
  "name": "ovn-controller",
  "permissions": {
    "Chassis": {"authorization": ["name"], "insert_delete": true},
    "Port_Binding": {"authorization": ["chassis"], "update": true}
  }
}
```

Verification: `ovsdb-client` running as a restricted principal that tries to write outside its role fails with a permission error.

Use case: in a multi-tenant cloud with many chassis, each chassis authenticates with a TLS certificate whose CN matches the chassis name. RBAC then guarantees that one chassis cannot tamper with another chassis's data.

## 10.2.6 Operational checklist

Daily:

```bash
[cron]# ovsdb-client backup > /backup/conf.db.$(date +%F).bak
```

Weekly:

```bash
[cron]# ovs-appctl -t ovsdb-server ovsdb-server/compact
```

Before an upgrade:

```bash
[admin]# ovsdb-client backup > /tmp/pre-upgrade.bak
[admin]# ovsdb-tool needs-conversion /etc/openvswitch/conf.db <new schema>
```

Monitoring alert when the file exceeds 1 GB:

```bash
[monitor]$ du -b /etc/openvswitch/conf.db | awk '$1 > 1073741824 {print "too big"}'
```

### Guided Exercise 6: backup and compact cycle

**Outcomes:** back up, generate transactions, compact, and verify that the file size shrinks.

**Steps:**

**1.** Baseline: `du -h /etc/openvswitch/conf.db`

**2.** Inflate the file:

```bash
[host]# for i in $(seq 1 100); do
          ovs-vsctl add-br br-test-$i
          ovs-vsctl del-br br-test-$i
        done
[host]# du -h /etc/openvswitch/conf.db         # the file has grown
```

**3.** Backup: `ovsdb-client backup > /tmp/conf.db.bak`

**4.** Compact:

```bash
[host]# ovs-appctl -t ovsdb-server ovsdb-server/compact
[host]# du -h /etc/openvswitch/conf.db         # size shrinks significantly
```

**5.** Verify the data: `ovs-vsctl list Bridge`. The current bridge list is still complete.

---

## Key points to remember

First, OVSDB is append-only. The file grows without bound. Compact periodically (weekly cron) to keep it small.

Second, live compact is the production-standard method. It does not stop the service.

Third, RBAC (OVS 2.8 and later) is the security foundation for multi-tenant OVN. Each chassis authenticates with TLS plus a restricted role.

Fourth, backup and restore do not preserve UUIDs. This is an important caveat. Coordinate the control plane restart with the restore.

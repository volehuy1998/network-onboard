## 10.5.1 Baseline performance characteristics

OVSDB is not designed for the high transaction throughput of a SQL database. It is a configuration store for OVS and OVN. The characteristic workload is **low write rate, high read rate, reads through subscription**. Concrete characteristics on commodity hardware (Intel Xeon Silver, 16 GB RAM, NVMe SSD):

**Write transaction throughput:**

- Single-node `ovsdb-server`: 2,000 to 5,000 transactions per second (TPS) with small transactions (1 to 5 operations).
- 3-node Raft cluster: 800 to 1,500 TPS (lower because the log must be replicated to a majority).
- 5-node Raft cluster: 600 to 1,000 TPS (lower again. Adding nodes does not increase write throughput, it only increases fault tolerance).

**Commit latency (time from when the client sends a transaction to when it receives a response):**

- Single-node: 1 to 5 ms (processing time plus disk fsync).
- 3-node Raft: 5 to 15 ms (additional round-trip to the majority plus fsync on every node).
- Cross-AZ Raft (nodes in different data centres): 20 to 100 ms (bound by WAN latency).

**Read throughput (`monitor_cond` subscription):**

- A single node serving 100 concurrent clients: 50 to 100 MB/s combined throughput.
- With well-targeted `monitor_cond` (each client receives only the relevant rows): scales nearly linearly to more than 1,000 clients.
- Without `monitor_cond` (full monitor): the bottleneck appears at 10 to 50 clients because of duplicated bandwidth.

**Memory footprint:**

- A 10,000-row database: about 50 to 100 MB of RAM.
- 100,000 rows: 500 MB to 1 GB.
- 1,000,000 rows (very large OVN SB cluster): 5 to 10 GB.
- Memory holds more than just the data. The Raft log (before compaction) plus per-client `monitor_cond` state can take 2 to 3 times the data size.

**Database file size on disk:**

- Without compaction the file grows linearly with the number of transactions (append-only log). After one million transactions it can reach 1 to 10 GB.
- After compaction it is roughly 2 to 5 times the in-memory data size.
- The Raft snapshot frequency (`cluster/change-election-timer` and the internal snapshot threshold) influences how often compaction is triggered.

## 10.5.2 Factors that affect performance at scale

When an OVN cluster grows to hundreds or thousands of chassis, the OVSDB SB DB becomes the most common bottleneck. Specific factors:

**(1) The number of Logical Flow rows.** Roughly linear in `Port_Binding` count times pipeline complexity. A cluster with 1,000 chassis times 100 pods per chassis times 50 Logical Flows per pod equals 5 million Logical Flows. Each `ovn-northd` compile transaction can insert or update tens of thousands of rows, which drives the latency to several seconds.

**(2) `monitor_cond` fanout.** Each `ovn-controller` subscribes to a subset. 1,000 chassis means 1,000 active subscriptions at the same time. The server must process every transaction times 1,000 comparisons to produce the per-client delta notification. The CPU overhead grows linearly with fanout.

**(3) Raft log growth.** Every transaction is appended to the Raft log. As the log grows, the replay time after a reboot or reconnection grows. `cluster/change-election-timer` and `ovsdb-tool compact` must be run periodically to keep the log small.

**(4) Conntrack-style ACL versus Logical Flow.** Stateful OVN ACL produces many Logical Flows because each rule compiles to 3 to 5 flows (one per state: new, established, related). A cluster with 10,000 ACL rules means 30,000 to 50,000 Logical Flow rows.

**(5) IDL memory per client.** Each `ovn-controller` caches a subset of the SB DB. The subset is small (filtered by `monitor_cond`), but in OVN 22.03 it can be 100 to 500 MB per chassis.

**(6) Disk I/O performance.** The fsync after each commit is a disk-bound bottleneck. NVMe SSD reduces fsync from 10 ms (SATA SSD) to 100 microseconds, which improves throughput by 10 to 100 times.

## 10.5.3 Benchmark methodology

To measure the real throughput of an OVSDB cluster before deploying to production:

**Tool 1: `ovn-scale-test` (Red Hat).** The official OVN scale-test framework. It generates a workload similar to a production OVN: it creates Logical Switch and Port resources and measures the `ovn-northd` compile latency and the `ovn-controller` OpenFlow-install latency. Set up a multi-node cluster, then run:

```
[host]# cd ovn-scale-test
[host]# ./rally.sh ovn-boot-servers --servers-per-node=100 --nodes=10
```

The test reports TPS, p50, p95, p99 latency, and memory usage.

**Tool 2: `ovsdb-client` plus a custom script.** Write a Python or shell script that submits transactions continuously and measures TPS:

```python
import time, json
from ovs.db.idl import Idl, Transaction
idl = Idl("tcp:127.0.0.1:6641", schema)
count = 0
start = time.time()
while time.time() - start < 60:  # 60 seconds
    txn = Transaction(idl)
    # build a simple transaction: insert row, then commit
    status = txn.commit_block()
    count += 1
print(f"TPS: {count/60}")
```

**Tool 3: `perf` profile of `ovsdb-server`.** When latency unexpectedly rises:

```
[host]# perf record -p $(pidof ovsdb-server) -F 99 -g -- sleep 30
[host]# perf report
```

Identify the function that dominates CPU usage (often JSON parsing, Raft replication, or `monitor_cond` diff).

**Tool 4: `iostat` for fsync latency:**

```
[host]# iostat -x 1
```

The `w_await` column (write await time) is the average fsync latency. If it exceeds 10 ms the disk is saturated, or the hardware is slow.

**Tool 5: OVSDB coverage counters:**

```
[host]# ovs-appctl -t ovsdb-server coverage/show
```

Internal counters such as `transact_total`, `monitor_notifications`, and `raft_append_entries`. Track them over time to detect anomalies.

## 10.5.4 Main tuning techniques

**(1) Tune the Raft `election_timer`.** The unixctl command tunes the leader-election timeout, measured in milliseconds, so the cluster tolerates network jitter between nodes. It is NOT a snapshot-frequency knob.

```
[host]# ovs-appctl -t ovsdb-server cluster/change-election-timer 4000
```

The default is 1000 ms (verified at v2.17.9 in `ovsdb/raft.c`, where `ELECTION_BASE_MSEC=1000`, with hard limits `ELECTION_MIN_MSEC=100` and `ELECTION_MAX_MSEC=600000`). Raise it to 4000 ms when the round-trip time between cluster nodes is high (cross-AZ deployments, p99 above 8 ms) so that spurious elections are rare. The heartbeat interval is `election_timer / 3`, so a 4000 ms timer drives heartbeats every 1333 ms. Important: this command does not change Raft snapshot frequency.

**(1b) Raft snapshot frequency.** OVS does not expose a runtime unixctl knob to set snapshot frequency directly. The default behaviour is to snapshot when the in-memory log crosses an internal size threshold (about 10 times the previous snapshot, or memory above roughly 100 MB; see `raft_notify_snapshot_recommended()` in `ovsdb/raft.c`). Operators tune snapshot behaviour through schema-level configuration and through the periodic `ovsdb-server/compact` command shown in subsection (2), not through a `cluster/change-*` unixctl command.

**(2) Compact the file periodically.** Use `ovsdb-client compact`:

```
[host]# ovsdb-client compact tcp:127.0.0.1:6641 OVN_Northbound
```

Run hourly or daily depending on the write rate. Production OVN clusters typically have a cron job for this.

**(3) Read-only server (OVSDB RO replica).** In a configuration of 3-node Raft plus several read-only replicas:

```
[host]# ovsdb-server --remote=ptcp:6641 /path/to/conf.db --read-only
```

A read-only server replays the log from the leader but does not accept writes. `ovn-controller` can subscribe from a RO server, which reduces the load on the leader.

**(4) Minimize the columns in the IDL.** Reduce the columns that each client subscribes to (covered in 10.4). The impact on memory and bandwidth is large.

**(5) Batch transactions.** `ovn-northd` uses parallel compilation and combines many changes into one transaction, which reduces the total number of transactions.

**(6) Hugepages for `ovsdb-server` memory.** Reduces TLB misses:

```
[host]# echo 2048 > /sys/kernel/mm/hugepages/hugepages-2048kB/nr_hugepages
```

Not a major tuning step, but it helps when the OVSDB memory footprint exceeds 1 GB.

> **Architecture note:** Raft consensus by design requires majority commit (a quorum of `n/2 + 1`). There is no official "skip Raft sync" mechanism in upstream OVS or OVN. Any claim about a flag that increases throughput by bypassing Raft must be verified carefully against the upstream code (`ovsdb/raft.c`) and the `ovn-nb.xml` schema. To increase write throughput, the practical techniques are: reduce the cluster size (3 instead of 5), reduce the commit rate (batch transactions through `ovn-northd-backoff-interval-ms`), or split the workload across separate OVSDB instances.

## 10.5.5 Typical bottlenecks and how to detect them

When an OVN cluster is slow, the diagnostic chain is usually as follows.

**Symptom 1: `ovn-controller` reacts slowly to NB changes.**

Check the `nb_cfg` delta between NB and SB:

```
[host]# ovn-nbctl get NB_Global . nb_cfg    # high value
[host]# ovn-sbctl get SB_Global . nb_cfg    # much lower value
```

A large delta means `ovn-northd` is not catching up. Continue the diagnosis:

```
[host]# ovs-appctl -t ovn-northd coverage/show | grep northd_run_time
```

**Symptom 2: high `ovsdb-server` CPU usage.**

```
[host]# top -p $(pidof ovsdb-server)
```

This is usually caused by a large `monitor_cond` fanout. Check the number of subscribers:

```
[host]# ovs-appctl -t ovsdb-server cluster/status OVN_Southbound
```

The output shows the number of connected clients. If it exceeds 500, consider splitting the workload across multiple `ovsdb-server` instances.

**Symptom 3: Raft elections happen continuously.**

The `ovsdb-server.log` file shows:

```
election timer expired, starting new election
```

Caused by a network partition or an overloaded leader. Check the network latency between nodes:

```
[host]# ping node2_ip
[host]# ping node3_ip
```

A latency above 10 ms between Raft nodes is a problem. The nodes should be in the same LAN or the same AZ.

**Symptom 4: disk full or slow.**

```
[host]# df -h /var/lib/ovn
[host]# iostat -x 1 | grep nvme
```

If the disk is slow, the priority is to upgrade to a faster SSD.

### Guided Exercise 1: benchmark transaction throughput with ovn-scale-test

> **Lab verification status:** the output is based on the Red Hat ovn-scale-test 2023 report. Pending verification on the lab host.

**Goal:** measure the TPS of the OVSDB NB DB under a synthetic workload to understand the characteristics before deploying to production.

**Setup:**

- An Ubuntu 22.04 node running OVN 22.03.
- Clone `ovn-scale-test`: `git clone https://github.com/ovn-org/ovn-scale-test`.

**Steps:**

**1.** Create a small workload (100 LS, 10 ports per LS):

```
[host]# cd ovn-scale-test
[host]# cat > workload.json <<EOF
{
  "OVN.create_and_list_lswitches": [
    {"args": {"amount": 100, "batch": 10}, "runner": {"type": "constant", "times": 1}}
  ],
  "OVN.create_and_list_lports": [
    {"args": {"amount": 10}, "runner": {"type": "constant", "times": 100}}
  ]
}
EOF
[host]# ./run.sh workload.json
```

**2.** Observe the output:

```
TPS report:
  create_and_list_lswitches: 145 TPS
  create_and_list_lports: 320 TPS (batch=10, so 32 txn/s)

Latency percentiles (ms):
  p50: 6.8
  p95: 14.2
  p99: 28.5
```

**3.** Scale the workload (1,000 LS, 100 ports per LS):

```
[host]# # Edit workload.json: amount 1000 and 100
[host]# ./run.sh workload.json
```

**4.** Compare TPS. At a larger scale TPS usually decreases because Logical Flow compilation is more expensive.

**Lessons learned:**

First, OVSDB TPS is not a fixed number. It decreases as the database grows. With an OVN cluster that has 100,000 Logical Flows, a single compile transaction can take 100 to 500 ms of CPU on `ovn-northd`.

Second, benchmarking before deployment helps avoid surprises in production. A 10,000-chassis cluster cannot run with the OVSDB configuration sized for 100 chassis.

**Cleanup:**

```
[host]# cd ovn-scale-test && ./cleanup.sh
```

### Guided Exercise 2: profile ovsdb-server with perf and a flamegraph

> **Lab verification status:** pending verification on the lab host.

**Goal:** identify the hot path inside `ovsdb-server` at scale.

**Setup:** `perf` plus the flamegraph scripts.

**Steps:**

**1.** While the benchmark from Exercise 1 is running:

```
[host]# perf record -F 99 -g -p $(pidof ovsdb-server) -- sleep 30
[host]# perf script > out.perf
[host]# stackcollapse-perf.pl out.perf | flamegraph.pl > ovsdb.svg
```

**2.** Open `ovsdb.svg` and look at the dominant functions. Common ones:

- `ovsdb_txn_process`: transaction processing.
- `json_to_value` and `value_to_json`: JSON parsing and serialization.
- `ovsdb_monitor_update`: computing the delta for `monitor_cond`.
- `raft_log_append`: appending to the Raft log.

**3.** If `json_to_value` is above 30 percent the bottleneck is JSON parsing. Consider upgrading to a newer OVS version with a faster JSON path.

**Lessons learned:**

First, a flamegraph is the necessary tool to understand what `ovsdb-server` is doing when CPU is high.

Second, profiling results guide the optimization direction. For example, if `monitor_cond` is dominant, the fix is to reduce fanout (split the server) or upgrade the hardware.

## 13.10.1 Why must DHCP run in the data plane?

In a traditional architecture, a DHCP server is a userspace process (ISC DHCPD, `dnsmasq`, or Kea) running on a host. It listens for DHCPDISCOVER broadcasts on a network interface. This model does not fit OVN well, for three reasons.

**Problem 1, broadcast does not cross chassis cleanly.** In an overlay network, a pod on chassis A broadcasts DHCPDISCOVER. If the DHCP server runs on chassis B, the broadcast must be flooded through Geneve tunnels to every chassis. This flooding consumes bandwidth and is used only for DHCP.

**Problem 2, single point of failure.** A single userspace DHCP server crashes and new pods cannot get an IP address. You must deploy an HA DHCP cluster, which adds complexity.

**Problem 3, scaling.** A Kubernetes cluster with 10K pods and a high pod churn rate (constant scale up and scale down) can generate thousands of DHCPDISCOVER messages per second. A single userspace DHCP server cannot keep up.

**The OVN solution: native DHCP.** Each `ovn-controller` on each chassis answers DHCP requests for its local pods. There is no broadcast across chassis, no single server, and the system scales linearly with the number of chassis. The mechanism:

- The CMS defines `DHCP_Options` in the NB DB (subnet, gateway, DNS servers, `lease_time`, and so on).
- Each LSP is assigned `dhcpv4_options`, which points to a `DHCP_Options` record.
- `ovn-northd` generates `Logical_Flow` rules in the table `ls_in_dhcp_options` (matching DHCPDISCOVER and DHCPREQUEST) and in `ls_in_dhcp_response` (building DHCPOFFER and DHCPACK).
- `ovn-controller` translates the rules to OpenFlow with the action `put_dhcp_opts`. This action builds the DHCPOFFER or DHCPACK packet directly in the kernel datapath, with no userspace involvement.

## 13.10.2 The structure of the `DHCP_Options` table

The `DHCP_Options` table in the NB DB has the following main fields:

- `cidr`, the subnet that DHCP serves (for example, `10.0.0.0/24`).
- `options`, a map from a DHCP option code to a value:
  - `server_id` (option 54), the DHCP server IP, usually the gateway IP.
  - `server_mac` (a special option), the server MAC, used as the source MAC of the DHCP response.
  - `lease_time` (option 51), lease duration in seconds.
  - `router` (option 3), default gateway for the client.
  - `dns_server` (option 6), DNS server list.
  - `netmask` (option 1), subnet mask.
  - `domain_name` (option 15), domain search.
  - `mtu` (option 26), interface MTU.
  - `classless_static_route` (option 121), static routes beyond the default.
- `external_ids`, a free-form map for CMS labels.

Example configuration:

```
[host]# ovn-nbctl dhcp-options-create 10.0.0.0/24
<uuid_dhcp>
[host]# ovn-nbctl dhcp-options-set-options <uuid_dhcp> \
         server_id=10.0.0.1 \
         server_mac="ca:fe:ba:be:00:01" \
         lease_time=3600 \
         router=10.0.0.1 \
         dns_server="8.8.8.8,8.8.4.4" \
         netmask=255.255.255.0 \
         mtu=1500
```

**Assigning DHCP options to an LSP:**

```
[host]# ovn-nbctl lsp-set-dhcpv4-options vm1 <uuid_dhcp>
```

When an LSP is assigned `dhcpv4_options`, `ovn-northd` generates `Logical_Flow` rules that handle DHCP requests from that port.

## 13.10.3 Logical Flow that handles DHCP

`ovn-northd` generates three groups of `Logical_Flow` rules for DHCP.

**Group 1, `ls_in_dhcp_options` (table 16):**
Match DHCPDISCOVER and DHCPREQUEST from the client. Apply the action `put_dhcp_opts` to build the response:

```
table=16 (ls_in_dhcp_options), priority=100,
  match=(inport == "vm1" && eth.src == aa:bb:cc:dd:ee:01 && ip4.src == 0.0.0.0
         && ip4.dst == 255.255.255.255 && udp.src == 68 && udp.dst == 67),
  action=(reg0[3] = put_dhcp_opts(offerip = 10.0.0.5,
                                   server_id = 10.0.0.1,
                                   router = 10.0.0.1,
                                   dns_server = {8.8.8.8, 8.8.4.4},
                                   lease_time = 3600,
                                   netmask = 255.255.255.0);
          next;)
```

The action `put_dhcp_opts` is a custom OVN action. It replaces the DHCP payload of the current packet with a DHCPOFFER or DHCPACK, and at the same time swaps the source and destination MAC and IP so that the packet returns to the client. `reg0[3]` is set to 1 on success, or 0 on failure (for example, if `offerip` is not within the subnet).

**Group 2, `ls_in_dhcp_response` (table 17):**
After the packet has been transformed into DHCPOFFER or DHCPACK, table 17 resubmits it through the pipeline in the reverse direction back to the client:

```
table=17 (ls_in_dhcp_response), priority=100,
  match=(reg0[3] == 1),
  action=(eth.dst = eth.src; eth.src = aa:aa:aa:aa:aa:aa; /* server_mac */
          ip4.src = 10.0.0.1; ip4.dst = 10.0.0.5; udp.src = 67; udp.dst = 68;
          outport = inport; flags.loopback = 1; output;)
```

**Group 3, DHCP drop (table 16 fallback):**
If a DHCP request does not match the `DHCP_Options` of the port, drop it:

```
table=16, priority=0, match=(udp.dst == 67), action=(drop;)
```

This logic ensures that only ports with a DHCP configuration receive a response. A port without `dhcpv4_options` has its requests dropped, which prevents DHCP snooping.

## 13.10.4 DHCPv6 and stateless configuration

OVN supports DHCPv6 through the `DHCPv6_Options` table, similar to `DHCP_Options` but with IPv6 option codes. OVN also supports **SLAAC (Stateless Address Auto-Configuration)** through Router Advertisement (RA), without a DHCPv6 server:

- `ovn-northd` generates `Logical_Flow` rules that send RA messages periodically from any `Logical_Router_Port` (LRP) that has an IPv6 address.
- An IPv6-capable pod receives the RA and self-configures an IPv6 address from the prefix (a global unicast address from the LRP plus an EUI-64 or a random suffix).

SLAAC configuration:

```
[host]# ovn-nbctl set Logical_Router_Port lrp1 ipv6_ra_configs:address_mode=slaac
[host]# ovn-nbctl set Logical_Router_Port lrp1 ipv6_ra_configs:send_periodic=true
[host]# ovn-nbctl set Logical_Router_Port lrp1 ipv6_ra_configs:max_interval=600
```

**Combining stateful DHCPv6 with stateless SLAAC:** some networks need IPv6 self-configuration for most clients (SLAAC) but DHCPv6 for the DNS server and the domain name (DHCPv6 stateless mode, options only, no address). OVN supports this through `ra_configs:address_mode=dhcpv6_stateless`.

## 13.10.5 Native DNS in OVN

OVN also supports a native DNS resolver in the data plane, through the `DNS` table in NB DB. There is no need to deploy a separate DNS server (CoreDNS, `dnsmasq`) for the cluster.

**Structure of the `DNS` table:**

- `records`, a map from `fqdn` to `ipv4_list,ipv6_list` (for example, `"pod1.cluster.local" → "10.0.0.5"`).
- `external_ids`, free-form labels.

**Attaching a `DNS` record to a `Logical_Switch`:**

```
[host]# ovn-nbctl create DNS records={"\"vm1.ovn.local\""="10.0.0.10"}
<uuid_dns>
[host]# ovn-nbctl add Logical_Switch ls1 dns_records <uuid_dns>
```

**Logical Flow that handles DNS queries:**
The table `ls_in_dns_lookup` (table 18) matches DNS query packets (UDP destination port 53) and applies the action `dns_lookup()` to look up the `DNS` table:

```
table=18 (ls_in_dns_lookup), priority=100,
  match=(udp.dst == 53),
  action=(reg0[4] = dns_lookup(); next;)
```

The action `dns_lookup` searches the `fqdn` from the DNS query against the `DNS` table. If found, it builds a DNS response in the current packet.

The table `ls_in_dns_response` (table 19) sends the response back to the client:

```
table=19 (ls_in_dns_response), priority=100,
  match=(reg0[4] == 1),
  action=(eth.dst <-> eth.src; ip4.src <-> ip4.dst; udp.src <-> udp.dst;
          outport = inport; flags.loopback = 1; output;)
```

**Limitations of native DNS:** it supports only A, AAAA, PTR, and SRV records. It does not support zone transfer, DNSSEC, or recursive resolution (if a query for an `fqdn` not in the `DNS` table fails to match, it is dropped, not forwarded to an upstream DNS). Production environments that need recursive DNS usually deploy CoreDNS as the upstream for OVN native DNS.

## 13.10.6 Integration with external DHCP and DNS

In an enterprise environment, DHCP and DNS often have a central IPAM (Infoblox, BIND), which cannot be replaced by OVN native services. OVN supports two integration models.

**Model 1, DHCP Relay.** OVN implements the `localnet` port type, which allows a `Logical_Switch` to bridge with an external VLAN. A DHCP request from a pod is broadcast in the `Logical_Switch`, OVN floods it through the `localnet` port to the external DHCP server, the external server replies, and OVN forwards the reply back to the pod. This model keeps the DHCP server centralized but sacrifices the distributed nature of OVN.

**Model 2, external DNS upstream.** OVN native DNS can forward unresolved queries to an upstream resolver. OVN 22.03 and later support `DNS.options:upstream_dns_servers`. If native DNS does not match, the query is forwarded to the upstream. This combines a local cache for fast lookup with an external upstream for the full DNS namespace.

First, OVN native DHCP is fully handled in the OVS datapath. There is no userspace process processing DHCP. Performance is high (line-rate) and the design is distributed (each chassis answers DHCP for its local ports).

Third, the action `put_dhcp_opts` is a low-level OVS datapath action. It transforms the packet inside the kernel without copying the packet up to userspace. This is why OVN DHCP latency is below one millisecond, compared to about ten milliseconds for `dnsmasq`.

## 13.10.Y `put_dhcp_opts` and `dns_lookup` action source-level details

DHCP and DNS native services in OVN are not implemented as a userspace daemon (such as `dnsmasq`). They are implemented through two OpenFlow actions: `put_dhcp_opts` (DHCP) and `dns_lookup` (DNS).

### `put_dhcp_opts` action

This is an OVS action implemented in `ofproto/ofproto-dpif-xlate.c` plus the kernel datapath path. When a packet matches a DHCP request (UDP 67/68), the action builds a DHCP reply directly in the datapath.

Source anchor (`branch-22.03`):

- [`controller/pinctrl.c`](https://github.com/ovn-org/ovn/blob/branch-22.03/controller/pinctrl.c), function `pinctrl_handle_put_dhcp_opts()`. `ovn-controller` runs a separate thread (`pinctrl`) that handles actions which need userspace involvement (DHCP, ARP responder, ND).
- Function flow: receive a controller-bound packet, parse the DHCP options field, look up the `DHCP_Options` row from NBDB, build the DHCP reply, and inject the reply back to the OVS datapath.

Pseudo-code workflow:

```c
// pinctrl_handle_put_dhcp_opts() simplified
dhcp_request = parse_packet(skb);
dhcp_opts = lookup_dhcp_options_for_lsp(lsp_uuid);
dhcp_reply = build_reply(dhcp_request, dhcp_opts);
inject_packet(dhcp_reply, lsp_inport);
```

Latency: less than 1 ms p50 and less than 5 ms p99 for a healthy cluster. Compared to a `dnsmasq` userspace daemon (10 to 50 ms), OVN native is about ten times faster.

### `dns_lookup` action

The `dns_lookup` action is similar, but the lookup data comes from the NBDB `DNS` table:

```bash
[host]# ovn-nbctl create DNS records='{"web.app=10.0.0.10","db.app=10.0.0.20"}'
[host]# ovn-nbctl set Logical_Switch ls1 dns_records=@dns
```

The action `dns_lookup` parses the DNS query packet, matches the `qname` against the `records` field, and builds the DNS response. The function is `pinctrl_handle_dns_lookup()` in `controller/pinctrl.c`.

Limitation: it only supports A and AAAA record lookups from a static map. No recursive resolver, no CNAME chain. For internal service discovery (pod-to-service), this is enough. It cannot replace public DNS.

### Pipeline NBDB to SBDB compile

`ovn-northd` compiles the NBDB `DHCP_Options` into `Logical_Flow` rules:

```
Stage ls_in_dhcp_options:
  priority=100, match=(inport == "vm1" && eth.src == aa:00:00:00:00:01 &&
                       ip4.src == 0.0.0.0 && ip4.dst == 255.255.255.255 &&
                       udp.src == 68 && udp.dst == 67),
  action=(reg0[3] = put_dhcp_opts(...);
          next;)

Stage ls_in_dhcp_response:
  priority=100, match=(inport == "vm1" && reg0[3] == 1),
  action=(eth.dst = eth.src;
          eth.src = 40:44:00:00:00:01;
          ip4.dst = ip4.src;
          ip4.src = 10.0.0.1;
          udp.src = 67;
          udp.dst = 68;
          outport = inport;
          flags.loopback = 1;
          output;)
```

Two stages: `ls_in_dhcp_options` builds the reply payload, and `ls_in_dhcp_response` swaps the headers (source and destination) and outputs the packet back through `inport` (loopback flag).

## 13.10.Z Anatomy Template A: `ovn-nbctl get Logical_Switch_Port dhcpv4_options`

> **Context:** You are checking the DHCP options binding for a specific LSP, debugging the case where a VM does not receive an IP.

```bash
[host]# ovn-nbctl get Logical_Switch_Port vm1 dhcpv4_options
abc12345-abcd-1234-...

[host]# ovn-nbctl list DHCP_Options abc12345-...
_uuid               : abc12345-abcd-1234-...
cidr                : "10.0.0.0/24"
external_ids        : {neutron:subnet_id="..."}
options             : {dns_server="{8.8.8.8,8.8.4.4}",
                       domain_name="lab.local",
                       lease_time="3600",
                       mtu="1442",
                       netmask="255.255.255.0",
                       router="10.0.0.1",
                       server_id="10.0.0.1",
                       server_mac="40:44:00:00:00:01"}
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| `cidr` | Matches the LS subnet | Mismatch | Empty | LSP-LS subnet conflict |
| `options:server_id` | Gateway IP of the LS | Empty | Mismatches `router` | DHCP reply source IP confusion |
| `options:server_mac` | MAC unique to the gateway | Empty | Conflict | DHCP reply source MAC duplicate |
| `options:lease_time` | 3600 (1 hour) baseline | Less than 60 seconds (frequent renew) | More than 86400 (cache stale) | Tune to VM lifecycle |
| `options:router` | Default gateway IP | Empty | Different from `server_id` | Routing broken |
| `options:dns_server` | One or more DNS servers | Empty | Unreachable IP | DNS resolution fails |
| `options:netmask` | Matches the `cidr` mask | Mismatch | Empty | The VM thinks the subnet is wrong |
| `options:mtu` | Matches underlay minus overhead (1442 for Geneve) | 1500 default | Underlay mismatch | PMTUD black hole, see Part 11.1 |

**Failure scenarios:**

- `dhcpv4_options` UUID is empty: the LSP has no DHCP configuration. The VM must use a static IP. To add one: `ovn-nbctl --id=@dhcp create DHCP_Options ... -- set Logical_Switch_Port vm1 dhcpv4_options=@dhcp`.
- `cidr` mismatch with the LS: the VM receives an IP in the wrong subnet. Recreate `DHCP_Options` with the correct `cidr`.
- `mtu=1500` over a Geneve underlay of 1500 causes fragmentation. Set `mtu=1442` or increase the underlay MTU (Part 11.1).

## 13.10.W Capstone POE: OVN native DHCP versus `dnsmasq`

> **Context:** A cluster needs to serve DHCP for 1,000 VMs or containers starting at the same time (cold cluster boot). An engineer chooses `dnsmasq` because it is easy to debug and has a clear log file.

**Claim to verify:** "`dnsmasq` is simpler, so it is more stable for a mass DHCP burst (1,000 VMs starting at the same time)."

**Predict:** Wrong. `dnsmasq` is a single-threaded userspace daemon, so it is a bottleneck under burst load and has high p99 latency.

**Observe (estimated):**

- **`dnsmasq` setup:** one process on one chassis (the gateway). 1,000 VMs broadcast DHCPDISCOVER at the same time, and `dnsmasq` queues them.
  - Latency p50: 50 ms (process queue plus parse).
  - Latency p99: 2 to 5 seconds (queue full plus retransmit).
  - Drop rate: 5 to 10 percent (UDP socket queue overflow).
  - Affected VMs: 50 to 100 VMs need to retry.

- **OVN native DHCP:** each chassis has an `ovn-controller`, and each chassis handles DHCP for its local LSPs. The work is distributed and parallel.
  - Latency p50: 0.5 ms.
  - Latency p99: less than 5 ms.
  - Drop rate: 0 percent.
  - All 1,000 VMs bootstrap in less than 2 seconds total.

**Explain:**

1. **`dnsmasq` is centralized:** one process queues 1,000 requests, processes serially, and the queue overflows.
2. **OVN native is distributed:** N chassis times M VMs per chassis equals parallel processing. Each chassis only handles a subset, so there is no bottleneck.
3. **Latency:** `dnsmasq` adds userspace context-switch and UDP socket overhead. OVN replies inline in the datapath through the kernel or the userspace `ovn-controller` (a single fast hop).
4. **Debugging:** `dnsmasq` has a clear log file. OVN uses `ovn-trace ls1 'inport=...&&udp.dst=67'` plus `ovs-ofctl dump-flows br-int | grep put_dhcp_opts`. The methods differ but both give visibility.

**Falsification test:**

- Set up a 100-VM mass-boot lab. Measure DHCP latency p50 and p99 with `dnsmasq` versus OVN native.
- If `dnsmasq` p99 is less than OVN p99, that is a counter-example. In practice, OVN always wins for bursts of more than 50 concurrent requests.

**Where the claim is wrong:** "Simpler, so more stable" assumes a small scale. For a mass DHCP burst, distributed beats centralized. OVN native is production-grade for large clusters.

> **Rule:** Use OVN native DHCP for a production cluster (10 or more chassis, 100 or more VMs). Use `dnsmasq` for a development or test environment, or for bare-metal hosts that do not run OVN.

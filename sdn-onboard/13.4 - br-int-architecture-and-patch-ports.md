## 13.4.1 br-int là integration bridge

Compass Ch F canonical pattern:

```
      VMs (tap interfaces)          Physical uplink
          │                              │
   ┌──────▼──────┐    patch    ┌────────▼──────┐
   │   br-int    ├─────────────┤    br-ex      │
   │ (OpenFlow)  │             │ (VLAN+NAT)    │
   └─────┬───────┘             └───────────────┘
         │ patch
   ┌─────▼───────┐
   │   br-tun    │ Geneve tunnels
   └─────────────┘
```

- **br-int**: "integration bridge" — VM tap + OpenFlow rules của OVN.
- **br-ex**: external — physical uplink, VLAN, NAT.
- **br-tun**: tunnel Geneve tới remote chassis.

Tất cả traffic VM đi qua br-int.

---

## 13.4.2 Tại sao integration bridge?

1. **Single policy enforcement**: mọi OpenFlow rules OVN cài trên br-int.
2. **Uniform pipeline**: VM-to-VM local, cross-chassis, external đều qua br-int.
3. **Separation of concerns**: br-ex handle physical specifics, br-tun handle overlay, br-int handle logical.

## 13.4.3 Patch port mechanism

Compass Ch F: "a patch port is a zero-copy tail-call, not a real queue."

```bash
[host]# ovs-vsctl add-port br-int patch-to-ex -- \
         set Interface patch-to-ex type=patch options:peer=patch-to-int
[host]# ovs-vsctl add-port br-ex patch-to-int -- \
         set Interface patch-to-int type=patch options:peer=patch-to-ex
```

Kernel: khi packet hit output action đến patch port, kernel gọi `ovs_vport_receive` với bridge peer — không enqueue, không copy. Cross-bridge ~0 overhead.

So với veth pair: veth có synthetic queue, overhead ~1-2μs. Patch port ~0.

## 13.4.4 OVN chassis setup

ovn-controller cần 4 config:

```bash
[chassis]# ovs-vsctl set Open_vSwitch . external_ids:ovn-remote=tcp:<SB-IP>:6642
[chassis]# ovs-vsctl set Open_vSwitch . external_ids:ovn-encap-type=geneve
[chassis]# ovs-vsctl set Open_vSwitch . external_ids:ovn-encap-ip=<chassis-underlay-IP>
[chassis]# ovs-vsctl set Open_vSwitch . external_ids:ovn-bridge-mappings=physnet1:br-ex
```

Khi ovn-controller start:

1. Create `br-int` nếu chưa có.
2. Register chassis vào SBDB `Chassis`.
3. Monitor Port_Binding cho chassis local.
4. Create tunnel port đến remote chassis.
5. Install OpenFlow rules dựa Logical_Flow.

## 13.4.5 Bridge mapping cho localnet

```bash
[central]# ovn-nbctl lsp-add ls-provider provider-port \
         -- lsp-set-type provider-port localnet \
         -- lsp-set-addresses provider-port unknown \
         -- lsp-set-options provider-port network_name=physnet1
```

ovn-controller match `network_name=physnet1` với `ovn-bridge-mappings=physnet1:br-ex` → install patch port từ br-int tới br-ex.

## 13.4.6 Debug br-int

```bash
[chassis]# ovs-vsctl show
[chassis]# ovs-ofctl -O OpenFlow13 dump-flows br-int | head -20
[chassis]# ovs-appctl ofproto/trace br-int "in_port=<vm-port>,..."
```

OVN logical flows compile thành hàng trăm OpenFlow rules trong br-int. Cookie encode logical flow ID.

`ovn-detrace` parse `ofproto/trace` output + map OpenFlow rule → logical flow trong SBDB.

---

## Điểm cốt lõi cần nhớ

Thứ nhất, `br-int` là integration bridge — mọi VM + tunnel + provider terminate ở đây. Single policy enforcement point.

Thứ hai, patch port zero-copy tail-call — OVS advantage so với Linux bridge cho multi-bridge pattern.

Thứ ba, chassis setup qua 4 `external_ids`: ovn-remote, ovn-encap-type, ovn-encap-ip, ovn-bridge-mappings.

Thứ tư, `ovn-detrace` + `ofproto/trace` combo debug mạnh — map OpenFlow → logical flow ID.

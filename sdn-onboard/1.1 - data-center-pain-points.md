## Context

Part 1.0 sketched the broad picture: the vertically integrated model blocked innovation, vendor lock-in tied down the roadmap, east-west traffic exploded past the original design, STP locked 40 to 50 percent of the capacity, and manual operations could not keep up with the application tier. This part goes deeper into four concrete pain points at the data center technical layer, problems that any network engineer working with cloud infrastructure between 2008 and 2014 ran into directly.

These four pain points are not abstract theory. They are visible on syslog, on the performance counters of the ToR switch, and on the dashboards of the cloud operator. Each one forced a part of the SDN architecture to come into existence as a remedy, and understanding the root cause helps you read the parts that follow (VXLAN in block XI, OVN ACL in block XIII, OVS conntrack in part 9.9) not as a set of disconnected technologies, but as concrete answers to concrete questions. Without this background, OpenFlow looks like a sudden invention and OVN looks like a purely technical product. With this background, both are revealed as the inevitable next step.

## 1.1.1 The bloated L2 broadcast domain: when flooding crosses the switch CPU threshold

### The flooding mechanism: four kinds of broadcast and multicast traffic

In an Ethernet broadcast domain, any frame whose destination MAC is `ff:ff:ff:ff:ff:ff` (broadcast) or a multicast group (`01:00:5e:xx:xx:xx` for IPv4 multicast) is flooded by the switch to every port in the same VLAN, per the IEEE 802.1D forwarding rules. Three common broadcast sources run continuously in every Ethernet network:

- **ARP request** (RFC 826, David Plummer, 1982-11): every time host A wants to send an IP packet to host B in the same subnet and does not have an entry for B in its ARP cache, A broadcasts the ARP request "Who owns IP 10.1.2.3?". Frequency: every cache entry times out after about 240 seconds (Linux default `net.ipv4.neigh.default.gc_stale_time = 60s`, Windows 2 to 10 minutes), but combined with millions of new flows per hour, the average ARP broadcast in a 10K-host cluster can reach thousands of packets per second.
- **DHCP DISCOVER** (RFC 2131, Ralph Droms, 1997-03): when a host boots or renews its lease, it broadcasts a DHCP DISCOVER. With a VM pool that has high churn (start and stop continuously by autoscaling), DHCP broadcast becomes a continuous signal.
- **Unknown unicast flooding**: when the switch does not yet know which port the destination MAC sits on (no FDB entry), it floods the unicast frame to every port in the VLAN. This happens when the MAC aging timeout expires (default 300 seconds on Cisco and on the Linux bridge), or when the MAC table overflows (FDB overflow).

### Why flooding hits a physical ceiling

The ceiling is not bandwidth. A 10 Gbps uplink can carry millions of broadcast packets per second if you only count raw bandwidth. The real ceiling is the CPU at every endpoint that has to process every broadcast frame. The Linux kernel receives a broadcast frame through an interrupt, enqueues it into an `sk_buff`, runs the Ethernet demultiplexer to decide whether the frame is relevant to the host, pushes it up the TCP/IP stack if it is, and drops it if it is not. The full pipeline costs about 1 to 2 microseconds per packet on a 2015-era server-grade x86 CPU, per the experimental measurements of Rizzo in the netmap paper (Luigi Rizzo, "netmap: a novel framework for fast packet I/O", USENIX ATC 2012).

With a 100K packet-per-second broadcast storm, every host spends 100 to 200 ms of CPU per second just processing broadcasts that are not relevant to it, equivalent to 10 to 20 percent of one CPU core. On a farm of a few hundred VMs in the same broadcast domain, the CPU burned just to listen to broadcasts becomes an unaffordable cost.

The ToR switch is also affected. A Broadcom Trident II (2013) has a 128K-entry MAC FDB per chip, enough for a cluster of a few thousand MACs. When the VLAN scales out to tens of thousands of hosts (for example, AWS EC2 classic networking before VPC), the MAC learning table starts to overflow, unknown unicast flooding rises sharply, and the entire VLAN falls into a self-amplifying spiral.

RFC 7348 §1 (the VXLAN problem statement) records this directly: *"In switched networks, STP [...] limits the size of the Layer 2 network that can be created. This limits the scalability of the virtual network in a datacenter."* RFC 7348 §3.3 continues the analysis: when a hypervisor can act as an L2 bridge for VMs, every ToR switch has to learn the MAC of every VM, and that count can grow to 20 to 50 VMs per hypervisor times thousands of hypervisors = hundreds of thousands of MACs from the VM layer alone, well past the FDB capacity of commercial hardware in 2013.

### The "L2 not larger than one rack" rule of the modern data center

The architectural consequence is that large data centers cap the broadcast domain at a few hundred hosts or fewer by default. The rule of thumb from Facebook Fabric (2014) is that a broadcast domain does not span more than one rack (40 to 80 hosts depending on density). Google Jupiter goes further: every ToR is an L3 edge, with no L2 span between ToRs, and all cross-rack traffic goes through IP routing.

This constraint is in direct conflict with the need for VM live migration. When a VM moves between hypervisors in the same VLAN (to reshape the workload, to rebalance CPU), it requires the same broadcast domain at both locations. If L2 is capped at one rack, the VM can only migrate inside the rack, not across racks. This is why overlay protocols such as VXLAN (RFC 7348) and Geneve (RFC 8926) appeared: they let L2 stretch logically over an L3 underlay while still respecting the physical limit of the broadcast domain.

> **Key topic:** L2 broadcast does not scale, not because of bandwidth but because of endpoint CPU. A broadcast frame sent to 10K hosts triggers 10K interrupt events plus 10K runs of the demultiplexer, with CPU cost linear in the host count. The overlay network (VXLAN) does not reduce the host count. It encapsulates broadcast inside a private tunnel for each logical network so that the physical underlay can run as L3 routing (which has no broadcast by nature).

## 1.1.2 The 4094 VLAN cap: a wall in front of the public cloud

### The math of the 12-bit VID

IEEE 802.1Q defines a 4-byte VLAN tag inserted after the MAC header. Its VLAN Identifier (VID) field is 12 bits long, per IEEE 802.1Q-2018 §9.6. VID 0 is reserved for "priority tagged" (used for PCP for QoS but does not partition the network), and VID 4095 is reserved for implementation use. In practice the usable count is 4094.

That number is more than enough for a traditional enterprise: a mid-sized company has 20 to 50 departments at 1 to 2 VLANs each, for a total below 100. A large bank or university might use 500 to 1000 VLANs across branches, faculties, and management networks, still well below the 4094 limit.

### The cloud pulls the threshold past the cliff

The public cloud breaks this math. AWS Virtual Private Cloud (VPC), Azure Virtual Network (VNet), and GCP VPC Network are all per-customer logical broadcast domains. AWS announced at re:Invent 2022 that EC2 runs more than 100 million instances per day across 30 regions, with each instance belonging to a VPC and an average customer running several VPCs (dev, staging, prod, per project). A large region such as us-east-1 may have millions of active VPCs at the same time.

Suppose a single physical AWS pod (a fault isolation unit) serves 10,000 tenants with at least one VPC per tenant and each VPC needing 1 to 3 subnet broadcast domains. The pod needs 10K to 30K VLANs, 2.5 to 7.5 times the 4094 cap. Adding more pods does not solve it: every pod still has to support every tenant it serves.

Before VXLAN became commercially available around 2012 to 2013, cloud operators had to use complex workarounds: Q-in-Q (802.1ad) raises the address space to 24 bits by stacking two VLAN tags, but it requires every switch on the path to support it and doubles the processing overhead. MPLS label-based multi-tenant segmentation in place of VLAN puts the burden of MPLS configuration on every ToR, which is impractical at hyperscale.

RFC 7348 §3.1 records this verbatim: *"A 12-bit VLAN ID is used in the Ethernet data frames to divide the larger Layer 2 network into multiple broadcast domains. This has served well for many networks, but in larger multi-tenant datacenters, [...] in order to provide multi-tenant isolation [...] a means of identifying tenants separately from the VLAN ID is required."*

### Why VXLAN chose 24 bits

VXLAN (RFC 7348, Mahalingam et al., 2014-08) solves this with a 24-bit VXLAN Network Identifier (VNI), which gives 16,777,216 usable segments. The 24-bit choice is not arbitrary. It is the largest size that fits all the criteria below at once:

- It fits into a 32-bit word, easy to handle in silicon.
- It still leaves 8 reserved bits for flag bits.
- It provides about 4000 times the 12-bit VLAN address space, enough for every realistic cloud scenario for the next 20 years.

Geneve (RFC 8926, Gross et al., 2020-11) inherits the 24-bit VNI but adds variable-length TLV (Type-Length-Value) options for extra metadata. This lets OVN embed extra information such as the logical port ID and the datapath ID into every packet, not just the tenant ID. (Part 11.0 on VXLAN and Geneve analyzes the encapsulation mechanism in detail.)

> **Key topic:** VLAN is not "wrong" or "old". It is still fine for an enterprise with 4094 tenants or fewer. VXLAN and Geneve do not replace VLAN at the physical access layer (the bridge between the VM and the hypervisor is still a VLAN). They add a logical identification layer on top of the L3 underlay. Both layers coexist: VXLAN and Geneve for cross-DC tenant isolation, VLAN for port-level access tagging inside each host.

## 1.1.3 ECMP hash imbalance: when an elephant flow takes over an uplink

### ECMP basics and the 5-tuple hash

Equal-Cost Multi-Path routing (ECMP) lets a router distribute traffic across several next hops with the same metric. The distribution is not per-packet (which would cause TCP reordering) but per-flow, based on a hash of the 5-tuple (source IP, destination IP, protocol, source port, destination port). The typical hash function is CRC32 over the concatenation of the 5-tuple, or XOR combined with bit rotation per RFC 2992 (D. Thaler, C. Hopps, 2000-11).

In a modern leaf-spine architecture with 4 spines, each leaf has 4 uplinks. With n flows passing through the leaf, the ideal expectation is that each uplink gets n / 4 flows. This holds when there are many flows of similar size.

### Elephant flows break the distribution

Real data center traffic is not evenly distributed. Hedera (Al-Fares, Radhakrishnan, Raghavan, Huang, Vahdat. "Hedera: Dynamic Flow Scheduling for Data Center Networks", NSDI 2010) measured a production trace and found:

- Most flows are "mice flows" (small, short): less than 100 KB, lasting less than 100 ms.
- A few are "elephant flows" (large, long): more than 10 MB, lasting more than 1 second, only a tiny share by count (1 to 2 percent) but carrying 70 to 80 percent of the total bytes.

When two elephant flows hash to the same ECMP path, they share one uplink and cause an "ECMP collision". That uplink saturates while the other three sit underused. Hedera measured this on the UCSD PortLand testbed and through simulation with production traces: with an elephant-flow-heavy workload, ECMP can drop well below the optimal scheduling, with a recorded lower bound around 50 to 60 percent of the bisection bandwidth capacity. A meaningful share of the physical infrastructure is wasted not because of oversubscription, but because of imbalance across the uplinks.

With a MapReduce shuffle workload, the situation is even worse. Each shuffle stage has hundreds of large flows in parallel between reducers, all subject to the ECMP hash, and collisions occur randomly and accumulate into local hot spots.

### Pre-SDN workarounds: CONGA and flowlet switching

CONGA (Alizadeh, Edsall, Dharmapurikar, Vaidyanathan, Chu, Fingerhut, Lam, Matus, Pan, Yadav, Varghese. "CONGA: Distributed Congestion-Aware Load Balancing for Datacenters", SIGCOMM 2014) proposed flowlet switching: split a long flow into flowlets (packet segments separated by more than 50 microseconds, usually corresponding to TCP idle moments), and let each flowlet take a different path. This way an elephant flow does not stick to one uplink. It rebalances itself when an idle moment arrives. CONGA was measured on a 64-server testbed and reported a 30 to 50 percent throughput improvement over standard ECMP.

CONGA, however, requires special silicon with per-link counters to feed back congestion. Cisco Tetration (later the Cisco Nexus 9000 with native CONGA) was the only commercial implementation before 2018. The open-source world has no CONGA equivalent.

### SDN solves it through centralized visibility

The SDN controller has a view of the entire traffic matrix (through flow counters from the switches) and can detect elephant flows and reassign their paths programmatically. The Hedera paper itself was an early SDN controller prototype for this purpose, even though OpenFlow was not yet mature at the time. Today:

- OVS megaflow plus Open vSwitch with DPDK provides nanosecond-level flow statistics, exported through sFlow and IPFIX (part 9.8).
- OVN does not do flow scheduling directly but exposes flow data to an upstream controller (ONOS, Google B4).
- Google B4 WAN (Jain et al., SIGCOMM 2013) is the first SDN production system to solve the elephant flow problem at WAN scale (not data center scale) through centralized traffic engineering.

> **Key topic:** ECMP collision is not a bug. It is a deliberate trade-off of hash-based distributed routing, which gives up worst-case balance for simplicity (stateless, per-flow). SDN does not fix ECMP. SDN adds a control layer that can override the ECMP decision for elephant flows based on real-time measurements. This is why every hyperscale data center has an SDN layer on top: not to replace ECMP, but to rescue ECMP when it hits a pathological case.

## 1.1.4 Manual middle-box insertion: firewalls and load balancers

### The traditional model: VLAN chaining and bridged appliances

Before SDN, inserting a middle-box (firewall, load balancer, WAN accelerator, DPI) into the traffic path required careful physical and logical work. Three patterns were common:

- **VLAN chaining**: create two VLANs "before" and "after" the middle-box, point the default gateway at the "before" VLAN, give the middle-box two interfaces (one in the "before" VLAN, one in the "after" VLAN), and let traffic from the "before" VLAN go into the middle-box, get processed, and exit on the "after" VLAN. Changing the order of the service chain (for example, inserting a WAF between the firewall and the load balancer) required adding more VLANs, more physical or sub-interface VLAN interfaces on the middle-box, and a routing table update, usually inside a maintenance window.
- **Policy-Based Routing (PBR)**: a normal routing table picks the next hop based on the destination IP, but PBR lets you choose a next hop based on the source IP, the destination port, or another ACL. PBR pushes inspection-worthy traffic into the middle-box. Downside: PBR does not health-check the middle-box automatically; if the middle-box dies, PBR keeps pointing at it and creates a black hole.
- **Transparent bridging**: the middle-box sits inline as an L2 bridge, with traffic flowing through it without the endpoints knowing. Compact, but it requires the middle-box to have Ethernet switching capability and not to change the L2 topology beyond what STP can tolerate.

In all three patterns, the configuration is scattered across switches, routers, and appliances, with no single place that holds the full view of the active service chain. Tracing whether a packet actually went through firewall, then load balancer, then app has to be done manually with `tcpdump` at every hop.

### RFC 7498 and the three core pain points

The IETF identified this pattern as an anti-pattern around 2014. RFC 7498 (Quinn, Nadeau. "Problem Statement for Service Function Chaining", 2015-04) lists the three main pain points:

**Topology dependence**: *"The insertion, removal, and re-ordering of service functions [...] typically requires significant reconfiguration of the network topology itself."* (RFC 7498 §3.1). The order of the service chain is locked into the physical topology and cannot change quickly.

**Service proliferation complexity**: with many service classes (web traffic through firewall, WAF, and load balancer; database traffic through firewall and encryption only; VoIP through QoS shaper and SRTP termination), you need many independent chains, each with its own topology. Managing a chain-by-class matrix by manual configuration becomes unworkable when the number of classes goes past about 10.

**Lack of end-to-end visibility**: there is no standard protocol for one middle-box to tell another middle-box what decision it has applied. The firewall does not tell the load balancer that the packet is "trusted internal", the WAF does not tell the next hop that the HTTP GET method was already inspected. Each middle-box decides independently and sometimes duplicates work or contradicts the next box.

### SDN solves it through flow-based steering

SDN, especially with the OpenFlow multi-table pipeline (part 4.0, OF 1.1) and OVN logical flows (part 13), changes the model:

- The service chain is no longer a physical topology. It is a logical definition in the OVN NBDB or in an OpenFlow group table.
- The steering order is decided by the controller and pushed down as flow entries. It can change at runtime without recabling.
- Metadata travels with the packet (the OVN logical port ID in Geneve options, or the Network Service Header NSH per RFC 8300, Quinn et al., 2018-01) so that middle-boxes can communicate decisions through the packet rather than out of band.

Part 12.2 (micro-segmentation and Service Function Chaining) goes deep into NSH and the OVN ACL as two approaches to SFC. This part only needs to record that manual middle-box insertion is a measurable pain point (number of weeks to implement each new service chain, number of incidents caused by misconfiguration), not a theoretical concern.

> **Key topic:** Service chaining before SDN was trapped in the physical topology because the packet did not carry metadata about "which stage of the chain am I in?". SDN puts metadata into the packet (NSH) or into the flow state (OVN conntrack) so that the chain logic can be separated from the physical topology. This is the prerequisite for large-scale micro-segmentation (thousands of rules per tenant) at cloud scale.

### ▶ Guided Exercise 1: Measure the CPU impact of a broadcast storm with Mininet

**Context:** You need to see, with a real experiment, why a broadcast domain does not scale with the host count, rather than just reading the theory. Mininet lets you build a topology with any number of hosts on a single Linux machine and measure CPU usage with `top` while you inject artificial broadcasts.

**Prerequisites:** Mininet 2.3 and OVS 2.17 installed per part 0.1. The machine has at least 4 CPU cores so that you can clearly tell the baseline from the load.

**Step 1, Predict:**

Before you run anything, write down your prediction: when you raise the host count in the same broadcast domain from 10 to 100 to 1000, the average CPU usage on each host (measured with `mpstat` or `top`) will follow which pattern?

- Linear in the host count?
- Quadratic (O(n^2)) because every host has to process broadcasts from every other host?
- A knee, after which the CPU saturates?

Write your prediction down before continuing.

**Step 2, Observe:**

Build a 10-host topology on one switch:

```
$ sudo mn --topo single,10 --switch ovs --controller none
mininet> h1 arping -c 1 -b 10.0.0.255 &
mininet> dump
```

From host h1, send 100 ARP requests broadcast to a non-existent address:

```
mininet> h1 arping -c 100 -w 1 10.0.0.99
```

In a second terminal (not the Mininet one), measure the CPU usage of the Mininet process:

```
$ mpstat 1 5
```

Record the sum of `%user` plus `%soft` (the soft IRQ portion the kernel spends on broadcast handling).

Repeat with a 100-host topology:

```
mininet> exit
$ sudo mn --topo single,100 --switch ovs --controller none
mininet> h1 arping -c 100 -w 1 10.0.0.99
```

And with 1000 hosts (if your machine has enough RAM, each Mininet host costs about 2 to 5 MB):

```
mininet> exit
$ sudo mn --topo single,1000 --switch ovs --controller none
mininet> h1 arping -c 100 -w 1 10.0.0.199
```

Record `%user` and `%soft` for each case. Sample table:

| Host count | Average %user | Average %soft | Total CPU load |
|------------|---------------|----------------|----------------|
| 10 | about 5% | about 2% | about 7% |
| 100 | about 25% | about 15% | about 40% |
| 1000 | about 70% | about 25% | about 95% |

**Step 3, Explain:**

The exact numbers depend on your hardware, but the trend is consistent: CPU rises faster than linearly. The reason: in Mininet every host is a network namespace on the same kernel, so every broadcast frame triggers `netif_receive_skb` on every namespace, with no hardware isolation. In a real data center, the broadcast also reaches every NIC in the VLAN, and the kernel of each host has to filter it out.

The point is not the exact number. It is that when you see CPU exceed 80 percent only because of 100 ARP requests per second per host across 100 to 1000 hosts, you feel directly why the hyperscale operator has to cap the broadcast domain at the rack or use an overlay to localize broadcast.

A preview of part 11.3 GRE lab: with the same topology but using an OVS bridge plus a GRE tunnel between two sites, broadcast only floods inside the tenant and does not cross the tunnel. Hosts outside the tenant are not affected. This is the essence of the "L2 scalability problem" that VXLAN and Geneve solve.

**Step 4, Cleanup:**

```
mininet> exit
$ sudo mn -c
```

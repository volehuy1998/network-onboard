## §13.5b.1 type="" (default OVN-native)

**Concept.** This is the default type. A VM or container is attached through libvirt or a CNI plugin. `ovn-controller` claims the port by reading `Open_vSwitch.external_ids:iface-id` and matching it against the `Logical_Switch_Port` name. The data structure is one row in the SBDB `Port_Binding` table.

**Importance.** This is the most common `Port_Binding` type in production OVN deployments. Every VM and container workload attached through libvirt or a CNI plugin uses it.

**How it works.** The claim mechanism: `ovn-controller` polls `Open_vSwitch.external_ids:iface-id` from the local OVS, matches it against `Logical_Switch_Port.name`, and updates `Port_Binding.chassis` to the local chassis UUID. Race condition: when multiple chassis attempt to claim the same port at the same time, first-write-wins through an OVSDB lock.

**Version differences.** OVN 2.4 introduced this type. It has been stable since.

**Real-world incident.** In production, a VM live migration triggered a claim race that caused a transient connectivity loss. The fix was to serialize the migration steps.

**Synthetic exercise.** Run `ovs-vsctl set Interface tap-vm1 external_ids:iface-id=lsp-vm1` and observe the `Port_Binding.chassis` update.

**Comparison with other systems.** Direct analog of OpenStack Neutron port binding. Similar to NSX-T VIF binding.

## §13.5b.2 type="patch"

**Concept.** A patch port between a `Logical_Switch` and a `Logical_Router`, or between two `Logical_Switch` instances internally. There is no physical port. It is a logical bridge between two datapaths.

**Importance.** Tier 1. Layer 3 routing inside OVN requires a patch port between `Logical_Switch` and `Logical_Router`.

**How it works.** `ovn-northd` compiles a `Logical_Router_Port` into a pair of patch ports, one on the `Logical_Switch` side and one on the `Logical_Router` side. Both sides have `Port_Binding.type=patch`.

**Version differences.** OVN 2.4. Stable.

**Real-world incident.** This is the standard L3 routing pattern. Every `Logical_Router` connects to a `Logical_Switch` through a patch port pair.

**Synthetic exercise.** Run `ovn-nbctl lr-add lr-main; ovn-nbctl lrp-add lr-main lrp-to-ls 00:00:00:00:00:01 10.0.0.1/24`.

**Comparison with other systems.** Analogous to a Linux bridge veth pair. Similar in concept to a Cisco SVI (Switched Virtual Interface).

## §13.5b.3 type="localnet"

**Concept.** A provider network bridge. There is no chassis claim. The port broadcasts on a physical bridge. The data structure is one row in the SBDB `Port_Binding` table.

**Importance.** Tier 2 medium. Connects an OVN logical network to the physical underlay (a provider network).

**How it works.** There is no claim. The packet egresses directly to the physical bridge with the specified VLAN tag, or untagged. Multiple chassis can bridge at the same time.

**Version differences.** OVN 2.6 introduced this type. OVN 21.06 added multi-network support.

**Real-world incident.** In production, a provider VLAN connects a VM to an external L2 network (for example, management VLAN 100).

**Synthetic exercise.** Run `ovn-nbctl lsp-add ls-tenant lsp-localnet-100; ovn-nbctl lsp-set-type lsp-localnet-100 localnet; ovn-nbctl lsp-set-options lsp-localnet-100 network_name=physnet1`.

**Comparison with other systems.** Direct analog of an OpenStack Neutron flat or VLAN provider network. Similar to an NSX-T VLAN-backed segment.

## §13.5b.4 type="l2gateway"

**Concept.** An L2 bridge to an external network through a specific chassis (gateway-only).

**Importance.** Tier 2 medium. Rare in modern OVN deployments because it has mostly been replaced by `localnet`.

**How it works.** The port is claimed by a specific chassis (a single point). Broadcast traffic is forwarded only through that chassis.

**Version differences.** OVN 2.4 introduced this type. It is mostly deprecated since OVN 22.03 in favor of `localnet` plus `HA_Chassis_Group`.

**Synthetic exercise.** Legacy configuration. Uncommon in new deployments.

**Comparison with other systems.** Direct analog of the OpenStack Neutron L2 gateway service plugin.

## §13.5b.5 type="l3gateway"

**Concept.** A distributed gateway router with redirect-chassis SNAT and DNAT. This is a legacy term replaced by `chassisredirect` since OVN 21.06.

**Importance.** Tier 2 medium (legacy).

**Version differences.** Pre-OVN 21.06 distributed gateway architecture used `l3gateway`. OVN 21.06 and later migrate to `chassisredirect`.

**Comparison with other systems.** Direct analog of the OpenStack Neutron L3 agent (legacy).

## §13.5b.6 type="chassisredirect"

**Concept.** A logical switch port that exists virtually on the redirect chassis for a Distributed Gateway Port (DGP).

**Importance.** This type is central to the distributed gateway architecture. Without it, north-south traffic from a `Logical_Router` cannot egress through the redirect chassis.

**How it works.** The port is synthesized (no real interface). It exists only logically. Set the `redirect-chassis` option on the `Logical_Router`. The pipeline detects this through `REGBIT_NAT_REDIRECT` (cited in Part 13.18).

**Version differences.** OVN 21.06 introduced this type.

**Real-world incident.** This is the distributed gateway pattern (cited in Part 13.18 `REGBIT_NAT_REDIRECT` incident, 2024 Q3).

**Synthetic exercise.** Run `ovn-nbctl set Logical_Router lr-main options:redirect-chassis=chassis-1`.

**Comparison with other systems.** Similar in concept to Cisco HSRP (active gateway designation).

## §13.5b.7 type="vtep"

**Concept.** Hardware VTEP integration through the VTEP schema (`ovs-vtep`). It bridges OVN logical L2 with a hardware switch that supports VXLAN.

**Importance.** Tier 3 peripheral. A niche use case (hardware VTEP gateway).

**How it works.** The `ovs-vtep` daemon decodes hardware VTEP commands. OVN treats the VTEP as a remote chassis. Cross-link Part 9.29 for `vtep-ctl`.

**Version differences.** OVN 2.6 introduced this type.

**Synthetic exercise.** Run `ovn-nbctl lsp-add ls-tenant lsp-vtep; ovn-nbctl lsp-set-type lsp-vtep vtep` and configure `ovs-vtep`.

**Comparison with other systems.** Direct analog of Cisco Nexus 9000 VTEP integration. Similar to Arista L3 leaf VTEP.

## §13.5b.8 type="external"

**Concept.** An external port managed by an external system (for example, direct SR-IOV passthrough).

**Importance.** Tier 3 peripheral. Use case: SR-IOV VF directly attached to a VM.

**Version differences.** OVN 21.06 introduced this type.

**Synthetic exercise.** Niche. Specific to SR-IOV deployments.

**Comparison with other systems.** Analog of OpenStack Neutron direct port binding. Hardware vendor specific.

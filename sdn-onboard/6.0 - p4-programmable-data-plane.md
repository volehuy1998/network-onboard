## Context: passing the limits of OpenFlow

Part 4.6 summarised five structural limits of OpenFlow: flow-table explosion, controller latency, distribution scale, vendor silicon mismatch, and the L4 to L7 abstraction gap. The common root cause is that the OpenFlow specification **fixes** the structure of match fields and actions. Adding a new field requires a specification revision. Hardware vendors cannot implement the entire specification because of silicon constraints. The controller cannot remain portable across heterogeneous fabrics.

P4 solves these limitations through a model shift. Instead of having the protocol **implicitly define** match fields and actions, P4 lets the **programmer define** the data plane behaviour using a domain-specific language. The parser, the match-action pipeline, and the deparser are all programmable. The silicon vendor implements a P4 compiler for the target. The programmer writes P4 once and compiles for multiple targets.

The principle is **target-independent programmability**, similar to how the C programming language lets the programmer write code once and compile for x86, ARM, or RISC-V. Instead of "the OpenFlow specification dictates silicon behaviour", a P4 program dictates behaviour and the silicon adapts.

## 6.0.1 P4 origin: ACM CCR July 2014

### The paper and the authors

The official P4 paper is "P4: Programming Protocol-Independent Packet Processors" by Bosshart et al., ACM SIGCOMM Computer Communication Review (CCR), Volume 44, Issue 3, July 2014. URL: `https://arxiv.org/abs/1312.1719` (arXiv mirror).

Authors (11 people):

- **Pat Bosshart**: ex-Texas Instruments silicon designer.
- **Dan Daly**: Intel.
- **Glen Gibb**: Stanford.
- **Martin Izzard**: Texas Instruments.
- **Nick McKeown**: Stanford (co-author of OpenFlow, Part 3.0).
- **Jennifer Rexford**: Princeton.
- **Cole Schlesinger**: Princeton.
- **Dan Talayco**: Big Switch Networks.
- **Amin Vahdat**: Google.
- **George Varghese**: Microsoft Research.
- **David Walker**: Princeton.

The author list mixes academia (Stanford, Princeton) and industry (Intel, Texas Instruments, Big Switch, Google, Microsoft Research). Unlike the OpenFlow paper (CCR April 2008), which had primarily Stanford authors, the P4 paper reflects a field that had matured and now featured multi-organisation collaboration.

### Motivation

The paper opens with three problem statements about OpenFlow:

**Problem 1**: the OpenFlow specification has a fixed set of header fields. When a new protocol (VXLAN, Geneve, NSH) appears, the community must wait for a specification revision. The 12-month to 24-month cycle does not match the pace of protocol evolution.

**Problem 2**: switch silicon is heterogeneous. Broadcom Trident2, Mellanox Spectrum, and Intel FM6000 have different constraints (see Part 4.5.1). The OpenFlow specification does not expose primitives, and silicon vendors keep their APIs proprietary (the Broadcom SDK is closed source).

**Problem 3**: the parser is fixed. OpenFlow assumes a standard L2, L3, and L4 header sequence. A custom protocol (for example, RDMA over Converged Ethernet, or custom encapsulation for security) cannot be expressed.

P4 addresses all three through a programmable abstraction.

### Language principles

The paper proposes three language principles:

**1. Protocol independence**: P4 does not hard-code header formats. The programmer defines headers. The switch is just a compiler target.

**2. Target independence**: P4 code is portable across silicon targets. Tofino, Broadcom-based devices, and the software BMv2 model all have a P4 compiler.

**3. Field reconfigurability**: reload the P4 program and the switch behaviour changes at runtime (no hardware upgrade needed).

After the paper was published, the consortium formalised into the P4 Language Consortium, which was later absorbed into the Open Networking Foundation and the Linux Foundation.

## 6.0.2 P4_14 versus P4_16: language evolution

### P4_14 (2014 to 2016)

The original language from the 2014 paper. Syntax example (simplified):

```
header ethernet_t {
    fields {
        dstAddr : 48;
        srcAddr : 48;
        etherType : 16;
    }
}

parser parse_ethernet {
    extract(ethernet);
    return select(latest.etherType) {
        0x0800 : parse_ipv4;
        default : ingress;
    }
}

table forwarding {
    reads {
        ethernet.dstAddr : exact;
    }
    actions {
        forward;
        drop;
    }
}
```

Issues with P4_14:

- **Verbose**: a lot of boilerplate, which made the programmer tired of writing and maintaining the code.
- **Tightly coupled to a reference architecture**: P4_14 assumed a specific pipeline structure (ingress pipeline, egress pipeline, deparser). It was hard to target devices with a different pipeline.
- **Limited composition**: it was hard to write reusable library code.

### P4_16 (2016 onward)

The 2016 refactored version had two goals: a lean core language and a modular architecture. Syntax example:

```p4
#include <core.p4>
#include <v1model.p4>

header ethernet_t {
    bit<48> dstAddr;
    bit<48> srcAddr;
    bit<16> etherType;
}

parser MyParser(packet_in packet, out headers hdr, inout metadata meta, inout standard_metadata_t std) {
    state start {
        packet.extract(hdr.ethernet);
        transition select(hdr.ethernet.etherType) {
            0x0800 : parse_ipv4;
            default : accept;
        }
    }
}

control MyIngress(inout headers hdr, inout metadata meta, inout standard_metadata_t std) {
    action forward(bit<9> port) { std.egress_spec = port; }
    action drop() { mark_to_drop(std); }

    table ipv4_lpm {
        key = { hdr.ipv4.dstAddr : lpm; }
        actions = { forward; drop; }
        default_action = drop();
    }

    apply { if (hdr.ipv4.isValid()) ipv4_lpm.apply(); }
}
```

Changes:

- The core language is minimal (parser, control blocks, tables, actions).
- **PSA (Portable Switch Architecture)** was externalised into a separate specification. PSA defines standard blocks (parser, ingress, egress, deparser, packet_in and packet_out). The programmer imports PSA instead of using a hard-coded reference architecture.
- A better type system (fixed-width bit types, explicit extraction).
- Library support (include files, extern declarations).

Since 2019, **P4_16 plus PSA is the standard**. P4_14 is deprecated, and the compile-to-P4_16 path is also deprecated.

## 6.0.3 PISA (Protocol-Independent Switch Architecture)

### Abstract machine model

PISA is the abstract hardware model that a P4 programmer targets. The structure is:

```
Input -> Programmable Parser -> Match-Action Pipeline (N stages) -> Programmable Deparser -> Output
                                     |
                                 Stage 0  (TCAM + SRAM + ALU)
                                 Stage 1
                                 ...
                                 Stage N
```

**Parser**: a programmable state machine. The input is a bit stream and the output is a set of extracted headers. The programmer defines state transitions based on header fields.

**Match-Action Pipeline**: N sequential stages. Each stage has:

- **Match unit**: a lookup table with TCAM (ternary) for wildcarded match, and SRAM (exact) for hash-based match.
- **Action unit**: an ALU (Arithmetic Logic Unit) that modifies headers, metadata, or tables.

The programmer defines N tables, and each table maps to a specific stage or to multiple stages.

**Deparser**: the inverse of the parser. It serialises the modified headers back into a bit stream for output.

### Difference from a fixed-function ASIC

A traditional switch ASIC (Broadcom Trident2, Mellanox Spectrum-1 or Spectrum-2) has a hard-coded pipeline:

```
L2 MAC lookup -> L3 LPM -> ACL -> Queue -> Output
```

Header parsing is fixed (it assumes Ethernet, plus IPv4, ARP, or MPLS). Actions are limited (modify MAC, modify DSCP, and so on). Adding a new protocol means re-spinning the silicon (a new mask ROM).

PISA silicon (Tofino) has a **programmable** pipeline:

- The parser is programmable. It can extract any header format.
- Tables are programmable. The programmer defines width, depth, and action per stage.
- Actions are programmable. The ALU can execute arbitrary computation within the power budget.

The hardware area of PISA versus a fixed-function ASIC is roughly 10 to 20 percent larger (overhead for programmability), and the trade-off is acceptable for the flexibility gained.

### Tofino as the reference PISA silicon

Barefoot Networks was founded in 2013 by Pat Bosshart (the first author of the P4 paper) and a team of former Microsoft Research and Stanford engineers. They launched the Tofino ASIC in 2016:

- **Tofino 1 (2016)**: 3.3 Tbps, 12 stages, programmable.
- **Tofino 2 (2018)**: 12.8 Tbps, scaled up.
- **Tofino 3 (2021)**: announced, but production volume was unclear.

Tofino was the first commercial PISA silicon. It validated the P4 model in production. Customer adoption included Alibaba Cloud, Tencent, and Facebook (limited deployment).

## 6.0.4 Targets: Tofino ASIC, BMv2 simulator, eBPF, DPDK

### P4 target ecosystem

A P4 programmer writes once and compiles for multiple targets through the `p4c` compiler:

**Tofino ASIC (Barefoot, then Intel)**: production silicon. The `p4c-tofino` backend generates microcode for the Tofino pipeline. Real bandwidth is 3 to 12 Tbps.

**BMv2 (Behavioral Model v2)**: a software reference switch for development and testing. Not production speed (it is a kernel-mode C++ implementation reaching about 1 to 5 Gbps on a commodity server), but it implements P4 semantics exactly for verification.

**T4P4S (DPDK target)**: P4 compiles to DPDK userspace C code. The target is a commodity NIC with DPDK acceleration. It reaches production speed for modest workloads (10 to 40 Gbps per core).

**eBPF and XDP target**: P4 compiles to eBPF bytecode and is loaded into the Linux kernel. The target is a commodity server with no special hardware. eBPF and XDP reach about 20 to 50 Gbps on a fast NIC.

**FPGA targets (NetFPGA, Xilinx)**: research-grade FPGA boards.

### The `p4c` compiler infrastructure

`p4c` at `https://github.com/p4lang/p4c` is the compiler framework:

- **Front-end**: parses P4 source and runs semantic analysis.
- **Mid-end**: optimises the intermediate representation.
- **Back-ends**: target-specific code generation (Tofino, BMv2, eBPF, DPDK).

The architecture is modular: adding a new target means writing a new back-end without modifying the front-end.

### Trade-offs when choosing a target

| Target | Real bandwidth | Cost | Use case |
|--------|------------|------|----------|
| Tofino ASIC | 3 to 12 Tbps | Very high (silicon plus ODM switch) | Production data centre switch |
| BMv2 | 1 to 5 Gbps | Free | Development and test |
| T4P4S DPDK | 10 to 40 Gbps per core | Moderate (commodity NIC plus server) | Mid-bandwidth production, NFV |
| eBPF and XDP | 20 to 50 Gbps | Low (commodity NIC plus server) | Edge compute, container networking |

After the Tofino end of life, soft targets (DPDK, eBPF) became more important for P4 sustainability.

## 6.0.5 P4Runtime API

### A gRPC-based southbound interface

A P4 program defines what the switch can do (table schema, action set). The controller still needs to install runtime entries (specific table entries for specific matches). P4Runtime is the southbound API for this purpose.

**Architecture**:

- **Protocol**: gRPC with Protocol Buffers (protobuf).
- **Version**: current v1.3 and later (2023).
- **Operations**: install, delete, and modify a table entry. Read counters and registers. Send packet-in and packet-out. Manage multicast groups. Manage sessions.

**Specification**: `https://github.com/p4lang/p4runtime` with documentation in the `docs/` directory.

### Independence from version

A key feature is a **single API for multiple P4 program versions**. The controller connects to the switch, retrieves the schema of the current P4 program (table names, match types, action parameters) via `GetForwardingPipelineConfig`, and installs entries based on that schema.

When the P4 program is updated (a table definition changes), the controller adapts automatically through a schema reread. Unlike OpenFlow (a fixed schema across versions, with the controller hard-coded to match fields), P4Runtime is flexible for evolving P4 programs.

### Replacing OpenFlow

P4Runtime is designed as an **OpenFlow replacement for P4-capable switches**. The analogy:

| OpenFlow | P4Runtime |
|----------|-----------|
| Fixed pipeline (defined by the specification) | Programmable pipeline (defined by the P4 program) |
| Hard-coded match fields | Schema-based match from the P4 program |
| Fixed action set | Actions from the P4 program |
| Protobuf wire format | gRPC plus protobuf |

Production adoption: ONOS integrated P4Runtime in 2018. Stratum (Part 5.2.4) runs P4Runtime as a thin switch agent.

### Standardisation

P4Runtime was standardised through ONF and ONOS collaboration in 2018. Subsequent releases were v1.1 (2019), v1.2 (2020), and v1.3 (2023). It is maintained as an open specification on GitHub (`p4lang/p4runtime`).

## 6.0.6 Intel acquisition in 2019, Tofino end of life in 2023, current state

### Intel acquisition

Intel acquired Barefoot Networks in June 2019, reportedly for about 500 million United States dollars. The rationale was that Intel's data centre group wanted to pursue programmable networking as a vertical expansion from CPU plus Altera FPGA (acquired in 2015) plus the Mount Evans IPU (a SmartNIC equivalent to ConnectX).

The Barefoot team was retained and continued the Tofino roadmap inside the Intel Connectivity Group.

### Tofino cancellation in January 2023

Intel announced the end of life for the Tofino family in January 2023. Reasons (Intel-disclosed plus industry speculation):

- **Silicon research and development cost**: programmable silicon is expensive to iterate compared to off-the-shelf Broadcom or Marvell silicon.
- **Market share**: Tofino adoption remained niche (primarily Alibaba, Tencent, and a few telcos), and the volume was not enough to justify the Intel overhead.
- **Strategic shift**: Intel focused on AI, HPC, and the Mount Evans IPU. Switch silicon was deprioritised.
- **Post-acquisition uncertainty**: cultural fit between Barefoot and the Intel organisation was a challenge.

Impact: the P4 ecosystem lost its only production silicon target. Customers (Alibaba and operators with Tofino deployments) had to plan migration over a multi-year period.

### Current state in 2026

P4 momentum continues through software targets and the community:

**P4.org community**: `https://p4.org/` maintains specifications, tools, and the annual P4 workshop. The community is active.

**Software targets are mature**:

- **BMv2** is stable for education.
- **T4P4S** is active for NFV.
- **eBPF and XDP** are growing in adoption for edge compute.

**Alternative hardware**:

- **AMD Pensando DPU**: programmable silicon following the Pensando acquisition in 2022. It uses a P4-like programming model via "PSM" (a P4-compatible state machine).
- **NVIDIA BlueField DPU**: a ConnectX-based SmartNIC with DOCA (Part 9.5). DOCA Pipeline is a P4-inspired abstraction. It is not exactly the P4 specification but uses a similar model.
- **Cisco Silicon One**: a custom ASIC with limited programmability. Not full P4.

**Research continues**:

- Academic P4 research is strong (SIGCOMM and NSDI from 2020 to 2025 have frequent P4 papers).
- Use cases include in-network compute (P4 accelerators for a key-value store), telemetry (INT, In-band Network Telemetry), and consensus (Paxos in the data plane).

### Common misconception about P4

> **Common misconception:** "P4 died after the Tofino end of life in 2023."
>
> **Reality:** the P4 ecosystem survived the Tofino cancellation. The language standard is stable, the community is active, and multiple software targets are mature. Alternative hardware (AMD Pensando, NVIDIA BlueField) fills the commercial gap with P4-inspired programming. Academic research thrives. Rumours of new silicon startups pursuing P4 (between 2024 and 2026) suggest a commercial revival is possible. The legacy of the P4 model is that it brought the programmable data plane concept into the mainstream, even if the specific Tofino silicon line ended.

> **Key point:** P4 is a model evolution beyond OpenFlow, not OpenFlow version 2.0. The fixed-protocol model of OpenFlow does not scale with protocol diversity. The programmable model of P4 absorbs flexibility at the cost of complexity. The Tofino end of life in 2023 is a commercial setback but not the death of the ecosystem. Long-term trajectory: protocol standardisation shifts toward programmable, and P4 has the chance to revive when alternative silicon matures (AMD Pensando, NVIDIA BlueField DPU, or future purpose-built startup chipsets).

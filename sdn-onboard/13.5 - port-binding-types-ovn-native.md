## 13.5.1 Port_Binding overview — role + schema

`Port_Binding` row trong SBDB biểu diễn binding giữa **logical port** (NBDB LSP/LRP definition) và **physical chassis** (nơi traffic của port được xử lý). Mỗi LSP/LRP trong NBDB tương ứng với một Port_Binding row trong SBDB (do `ovn-northd` compile sinh ra).

### Schema columns quan trọng

```shell
$ sudo ovn-sbctl list Port_Binding lsp_web_01
_uuid          : 7a3c9f82-...
logical_port   : "lsp_web_01"
datapath       : <datapath_uuid for Logical_Switch>
chassis        : [<chassis_uuid>]        # NULL hoặc chassis claiming port
type           : ""                       # 8 values: "" / localnet / l2gateway / ...
tunnel_key     : 5                        # Per-datapath unique ID
mac            : ["aa:bb:cc:00:00:01 10.0.0.5"]
external_ids   : {tenant="T1"}
options        : {requested-chassis="chassis-A"}  # Optional pin
parent_port    : []                       # Cho nested port (virtual)
tag            : []                       # VLAN tag cho localnet
gateway_chassis: []                       # Dedicated column cho chassisredirect
ha_chassis_group: []                      # HA_Chassis_Group reference
up             : true                     # Binding state up/down
nat_addresses  : []                       # Cho logical router port
virtual_parent : []                       # Cho virtual type
```

### Claim mechanism

ovn-controller trên mỗi chassis subscribe SBDB table `Port_Binding`. Khi thấy row phù hợp điều kiện claim (VD: OVS port có `external_ids:iface-id` khớp với `logical_port`), controller gọi OVSDB transaction:

```sql
UPDATE Port_Binding SET chassis=<this_chassis_uuid> WHERE _uuid=<row_uuid>
```

SBDB Raft replicate update sang tất cả Raft follower → tất cả ovn-controller khác thấy binding mới trong < 500ms.

## 13.5.2 Type `""` (default VIF) — VM tap claim

**Mặc định** — Port_Binding của VM tap (virtual interface). Type là empty string `""`. Type phổ biến nhất (>95% Port_Binding trong DC triển khai).

### Claim condition

ovn-controller trên chassis X claim Port_Binding type `""` khi:
1. OVS bridge `br-int` có port (tap interface) với `external_ids:iface-id=<logical_port>`.
2. Port_Binding chưa claim bởi chassis khác (chassis=NULL hoặc chassis=this).

### Provisioning workflow

```
1. CMS orchestrator (Nova/libvirt/Kubelet) allocate VM.
2. CMS call: ovn-nbctl lsp-add <ls> <lsp_name>
   → NBDB: 1 row Logical_Switch_Port
3. ovn-northd compile:
   → SBDB: 1 row Port_Binding (type="", chassis=NULL)
4. CMS plug tap vào br-int:
   → ovs-vsctl add-port br-int tap-vmX
   → ovs-vsctl set Interface tap-vmX external_ids:iface-id=<lsp_name>
5. ovn-controller trên chassis X detect tap + iface-id match:
   → SBDB: UPDATE Port_Binding SET chassis=<X>
6. ovn-controller install OpenFlow rule match port cho LSP.
7. VM now functional.
```

### Multichassis binding (OVN 22.06+)

Khi VM live migration, OVN 22.06+ hỗ trợ dual-chassis binding trong window migration:

```
source chassis A: Port_Binding.chassis=A + additional_chassis=[]
→ migration started:
   destination chassis B prepare + claim as additional
   Port_Binding.chassis=A + additional_chassis=[B]
   Traffic replicate cả 2 chassis trong window 3-18s (Part 19.0)
→ migration done:
   source release
   Port_Binding.chassis=B + additional_chassis=[]
```

Trước OVN 22.06, live migration gây 15-30s downtime. Multichassis + `activation-strategy=rarp` (OVN 24.03+) giảm xuống < 500ms.

> **Chủ đề trọng tâm:** Claim logic dựa **entirely** vào iface-id match. Nếu hypervisor orchestrator không set `external_ids:iface-id` khi plug tap, ovn-controller không claim → VM không có network. Đây là pain point phổ biến: manual libvirt provisioning thiếu iface-id, hoặc CNI plugin bug. Debug bắt đầu bằng `ovs-vsctl get Interface tap-vmX external_ids:iface-id` — nếu empty, fix ở orchestrator layer, không phải OVN layer.

## 13.5.3 Type `localnet` — provider network extension

LSP type `localnet` connect logical switch đến **physical network** (provider network).

### Đặc điểm

- **Không claim chassis** — available trên tất cả chassis có bridge mapping tương ứng.
- Mỗi chassis map `options:network_name` tới OVS bridge qua `external_ids:ovn-bridge-mappings`.
- ovn-controller cài đặt patch port giữa `br-int` và external bridge (br-ex/br-provider).

### Use case

Provider network (OpenStack flat provider), shared internet gateway, VLAN trunk tới physical switch upstream.

```bash
[admin]# ovn-nbctl ls-add ls-provider-public
[admin]# ovn-nbctl lsp-add ls-provider-public provider-port
[admin]# ovn-nbctl lsp-set-type provider-port localnet
[admin]# ovn-nbctl lsp-set-addresses provider-port unknown
[admin]# ovn-nbctl lsp-set-options provider-port network_name=physnet1
```

Trên chassis với `ovn-bridge-mappings=physnet1:br-ex`:
- ovn-controller tạo patch port pair `patch-provnet-physnet1` ↔ `patch-br-ex-to-br-int`.
- Traffic destined tới LSP type `localnet` output qua patch → `br-ex` → physical NIC.

### VLAN tagging

Khi provider network cần VLAN tag:

```bash
[admin]# ovn-nbctl set Logical_Switch_Port provider-port tag=100
```

Flow compile push VLAN 100 trước khi output patch. Upstream switch thấy frame VLAN 100 → forward đúng.

## 13.5.4 Type `chassisredirect` + `l3gateway` — north-south NAT

**Distributed Logical Router (LR)** xử lý L2/L3 chuyển tiếp distributed trên mọi chassis (mỗi chassis có local OpenFlow rule). Nhưng **north-south NAT** không distributed được — một external IP chỉ thuộc một chassis vật lý tại một thời điểm (ARP announce, routing table).

### `chassisredirect` Port_Binding

Là "shadow" của Logical Router Port (LRP) trên **gateway chassis**. Traffic north-bound redirect qua chassisredirect → gateway chassis xử lý NAT → `br-ex`.

```bash
[admin]# ovn-nbctl lrp-set-gateway-chassis lrp-external chassis-1 100
[admin]# ovn-nbctl lrp-set-gateway-chassis lrp-external chassis-2 50
```

`ovn-northd` sinh Port_Binding type `chassisredirect` với `ha_chassis_group` reference tới HA_Chassis_Group row chứa list chassis sorted by priority.

### Failover via BFD

HA_Chassis_Group dùng BFD để phát hiện failure. Khi chassis-1 (priority 100) fail:
- BFD session timeout (default 5 giây, tunable xuống 1 giây).
- HA_Chassis_Group reelect chassis-2 là active.
- Port_Binding chassisredirect update chassis field.
- ovn-controller trên chassis-2 cài đặt OpenFlow rule cho NAT.

Downtime typical 3-5s với default BFD, < 1s với `bfd_min_rx=100ms`.

Xem Part 13.6 cho BFD deep-dive.

### `l3gateway` Port_Binding

Biến thể khác cho gateway routing không dùng distributed LR. Rarely used post OVN 20.06 (distributed LR đã stable).

## 13.5.5 Type `patch` — inter-datapath forwarding

LSP type `router` (LSP connect tới LRP) auto-generate Port_Binding type `patch`. Do `ovn-northd` sinh, engineer không manual create.

Dùng cho pipeline traversal từ Logical_Switch sang Logical_Router (qua LSP type `router`).

```
LS tenant1 LSP `ls-to-router` (type=router, connected LRP lrp-tenant1)
→ northd compile → Port_Binding `ls-to-router` type=patch
→ OpenFlow rule: output to Logical_Router pipeline
```

---

## 13.5.6 Type `l2gateway` — hardware VTEP extension

LSP gateway từ OVN logical network đến **hardware L2 switch** (via VTEP protocol, RFC 7047 extension).

### Use case

Extend OVN logical switch đến hardware VTEP (Cisco Nexus 9K, Arista 7280 với VXLAN VTEP). Pattern ESR (Extended Service Range) — OVN VM trong cùng broadcast domain với bare-metal server sau hardware switch.

```bash
[admin]# ovn-nbctl lsp-set-type vif-hw-vtep l2gateway
[admin]# ovn-nbctl lsp-set-options vif-hw-vtep network_name=physnet1 \
         l2gateway-chassis=chassis-1
```

`l2gateway-chassis` chỉ định chassis xử lý VTEP protocol với hardware switch. ovn-controller trên chassis đó maintain VTEP OVSDB connection với hardware switch.

Trade-off: thêm operational complexity (VTEP protocol + hardware switch firmware compat) nhưng cho phép hybrid physical-virtual topology.

## 13.5.7 Type `localport` — per-chassis metadata service

**Đặc biệt** — Port_Binding tồn tại trên **mọi chassis** với cùng MAC + IP. Traffic destined tới localport KHÔNG đi qua tunnel — stay local trên chassis nguồn.

### Use case: metadata service

OpenStack Nova + Kubernetes đều cần metadata endpoint (`169.254.169.254` IANA link-local). Mỗi VM truy cập metadata qua HTTP GET. Thay vì route metadata traffic tập trung (single gateway chassis), OVN dùng localport:

```bash
[admin]# ovn-nbctl lsp-add ls-tenant metadata-port
[admin]# ovn-nbctl lsp-set-type metadata-port localport
[admin]# ovn-nbctl lsp-set-addresses metadata-port \
         "fa:16:3e:00:00:01 169.254.169.254"
```

Mỗi chassis:
- ovn-controller cài đặt OpenFlow rule khớp với dst=169.254.169.254 → output port internal (local proxy agent).
- Proxy agent (ví dụ Neutron metadata agent) handle HTTP → forward tới CMS metadata API.
- Reply về đúng VM nguồn, không cross-chassis.

Lợi ích: scale metadata service linear theo chassis count (không bottleneck một gateway).

## 13.5.8 Type `virtual` — VRRP shared address

OVN 21.03+ hỗ trợ `virtual` Port_Binding cho floating IP shared bởi nhiều VM (VRRP pattern).

```bash
[admin]# ovn-nbctl lsp-add ls-tenant vip-floating
[admin]# ovn-nbctl lsp-set-type vip-floating virtual
[admin]# ovn-nbctl lsp-set-options vip-floating \
         virtual-parents=vm-master,vm-backup virtual-ip=10.0.0.100
```

Khi một VM (vm-master) announce Gratuitous ARP cho 10.0.0.100, ovn-controller bind virtual port tới VM đó. Khi vm-master fail, VRRP re-announce từ vm-backup, ovn-controller switch binding. Smooth handover cho HA application.

Use case: HAProxy/Keepalived floating VIP, database cluster master IP.

---

## 13.5.9 Type `remote` (OVN 22.06+) — inter-cluster peering

OVN cluster-to-cluster peering qua OVN Interconnection (IC). Port trên remote cluster visible dưới type `remote` trong local SBDB. Cho phép extend logical switch/router across Kubernetes cluster hoặc DC region.

```bash
[local-cluster]# ovn-ic-nbctl ts-add ts-shared
[local-cluster]# ovn-ic-nbctl ts-set-route ts-shared 10.0.0.0/16 <remote-cluster>
```

Rarely used trong single-cluster deployment. Xem Part 15.1 OVN-IC section cho multi-cluster pattern.

## 13.5.10 Anatomy Template A — `ovn-sbctl list Port_Binding`

**Bối cảnh khi chạy:** Debug "VM không có network" — step 2 của 5-step diagnostic (sau check NBDB LSP exists).

**Command và output** *(reproduced OVN 22.03.8, VM lsp_web_01 trên chassis-hypervisor-A)*:

```shell
$ sudo ovn-sbctl --columns=logical_port,chassis,type,tunnel_key,mac,up,options list Port_Binding lsp_web_01

logical_port    : "lsp_web_01"
chassis         : [<chassis_A_uuid>]
type            : ""
tunnel_key      : 4
mac             : ["aa:bb:cc:00:00:01 10.0.0.5"]
up              : true
options         : {requested-chassis="chassis-A"}
```

**9-attribute anatomy:**

| Field | Giá trị mẫu | Ý nghĩa | Red flag nếu |
|-------|-------------|---------|--------------|
| `logical_port` | `"lsp_web_01"` | Match với `ovs-vsctl get Interface tap external_ids:iface-id` | Mismatch = claim không trigger |
| `chassis` | `[<uuid>]` hoặc `[]` | Chassis đang claim. `[]` = unbound | Empty + VM running = claim fail (no iface-id trên OVS) |
| `type` | `""` / `localnet` / ... | 8 type. `""` = VIF | Mismatch với expected use case = misconfiguration |
| `tunnel_key` | `4` | Per-datapath unique ID | Collision (2 port cùng tunnel_key) = northd bug |
| `mac` | `["aa:bb:... 10.0.0.5"]` | MAC + addresses from NBDB | Thiếu IP = `addresses=unknown`, DHCP/ARP responder không work |
| `up` | `true` / `false` | Binding state | `false` = chassis claim nhưng interface down |
| `options:requested-chassis` | `"chassis-A"` | Optional chassis pin | Conflict với physical location of VM |
| `options:activation-strategy` | `"rarp"` (OVN 24.03+) | Migration activation | Missing = fallback old migration behavior |
| `additional_chassis` | `[]` hoặc `[uuid]` | Multichassis binding (migration window) | Non-empty + no migration in progress = leaked binding |

**Kịch bản bẻ gãy:**

- **`chassis=[]` + VM running + no iface-id trên tap**: CMS không set iface-id. Fix: `ovs-vsctl set Interface tap-vmX external_ids:iface-id=lsp_web_01`. Sau 1-2s ovn-controller claim.
- **`chassis=[B]` nhưng VM đang ở chassis A**: stale claim từ chassis B (crashed hoặc network partition). Fix: restart ovn-controller A, hoặc manual `ovn-sbctl chassis-del B` nếu B chắc chắn dead.
- **`additional_chassis=[C]` stuck sau migration done**: OVN 22.06 migration edge case. Fix: manual restart ovn-controller trên C hoặc `ovn-sbctl set Port_Binding <uuid> additional_chassis=[]`.
- **`up=false` + `chassis=[A]`**: interface operationally down. Check `ovs-vsctl --columns=name,admin_state,link_state list Interface tap-vmX`. If `link_state=down` → physical-layer issue (link flapping).

## 13.5.11 ▶ Guided Exercise — Observe Port_Binding claim workflow

**Mục tiêu:** Quan sát ovn-controller claim Port_Binding khi VIF plug + release khi VIF unplug, time latency.

**Môi trường:** Lab 2-chassis (Part 13.4 GE 1) với ovn-central trên node1 + ovn-host trên node1+node2.

**Bước:**

1. Trên node1 (central), tạo LSP chưa bind:
   ```bash
   node1$ sudo ovn-nbctl ls-add ls-claim-test
   node1$ sudo ovn-nbctl lsp-add ls-claim-test vm-claim-test
   node1$ sudo ovn-nbctl lsp-set-addresses vm-claim-test "aa:bb:cc:de:ad:01 10.0.0.99"
   ```
2. Verify chassis=NULL:
   ```bash
   node1$ sudo ovn-sbctl --columns=logical_port,chassis,up list Port_Binding vm-claim-test
   # Kỳ vọng: chassis=[], up=false
   ```
3. Monitor SBDB change real-time (ở terminal khác):
   ```bash
   node1$ sudo ovsdb-client -v monitor \
          unix:/var/run/ovn/ovnsb_db.sock OVN_Southbound Port_Binding \
          chassis logical_port 2>&1 | grep vm-claim-test
   ```
4. Trên node2, plug tap với iface-id:
   ```bash
   node2$ sudo ovs-vsctl add-port br-int tap-claim -- \
          set Interface tap-claim type=internal \
          external_ids:iface-id=vm-claim-test
   ```
5. Observe SBDB change trong < 1 giây:
   ```
   row "lsp_web_01" (vm-claim-test): chassis [] → [<node2-chassis-uuid>]
   ```
6. Re-check:
   ```bash
   node1$ sudo ovn-sbctl --columns=logical_port,chassis,up list Port_Binding vm-claim-test
   # Kỳ vọng: chassis=[<node2-uuid>], up=true
   ```
7. Unplug → observe release:
   ```bash
   node2$ sudo ovs-vsctl del-port br-int tap-claim
   # Observer thấy: chassis [<uuid>] → []
   node1$ sudo ovn-sbctl --columns=chassis,up list Port_Binding vm-claim-test
   # Kỳ vọng: chassis=[], up=false
   ```

**POE questions:**
- **Predict:** Nếu plug tap vào node1 đồng thời với node2, cả 2 chassis có claim không? SBDB trả lời ra sao?
- **Observe:** OVSDB transaction UPDATE là CAS (compare-and-swap). Chỉ 1 chassis thành công (whichever gửi transaction tới leader đầu tiên). Chassis kia get transaction conflict, retry, và phát hiện chassis đã set → không claim.
- **Explain:** Raft consensus ensure exactly-one claimer at a time. Nếu cần dual-chassis (migration scenario), phải dùng `additional_chassis` (OVN 22.06+) — separate column, không conflict với `chassis`.

**Cleanup:**
```bash
node1$ sudo ovn-nbctl ls-del ls-claim-test
```

## 13.5.12 Debug Port_Binding — 5-step diagnostic

Khi VM "không có network", truy vết Port_Binding qua 5 bước:

**Step 1 — NBDB LSP exists?**
```bash
node1$ sudo ovn-nbctl show ls-tenant1 | grep <vm-mac>
```
Nếu không có → CMS chưa tạo LSP. Fix ở CMS.

**Step 2 — SBDB Port_Binding exists?**
```bash
node1$ sudo ovn-sbctl --columns=logical_port,chassis,up,type list Port_Binding <lsp_name>
```
Nếu không có → ovn-northd chưa compile. Check `ovn-northd` log + status.

**Step 3 — OVS tap có iface-id match?**
```bash
node2$ sudo ovs-vsctl --columns=name,external_ids list Interface | grep <lsp_name>
```
Nếu không có → CMS không set iface-id. Fix ở CMS.

**Step 4 — Port_Binding.chassis claimed?**
```bash
node1$ sudo ovn-sbctl --columns=chassis list Port_Binding <lsp_name>
```
Nếu empty sau 5s từ step 3 → ovn-controller crash hoặc OVSDB connection dead. Check `ovn-appctl connection-status`.

**Step 5 — OpenFlow rule cho VM installed?**
```bash
node2$ sudo ovs-ofctl -O OpenFlow13 dump-flows br-int | grep <vm-mac>
```
Nếu không có → ovn-controller crashed sau claim. Check `/var/log/ovn/ovn-controller.log`.

> **Hiểu sai phổ biến:** *"Port_Binding tồn tại trong NBDB."*
>
> **Thực tế:** Port_Binding chỉ tồn tại trong **SBDB**, không phải NBDB. NBDB chứa `Logical_Switch_Port` (LSP ý đồ cấu hình). `ovn-northd` compile LSP → Port_Binding trong SBDB. Nhầm lẫn này thường gây confusion khi query wrong table: `ovn-nbctl list Port_Binding` error vì NBDB không có table đó. Đúng: `ovn-sbctl list Port_Binding`.

## 13.5.13 Điểm cốt lõi cần nhớ

Thứ nhất, Port_Binding là link giữa logical port (NBDB LSP/LRP) và physical chassis (SBDB runtime state). 8 type phục vụ use case khác nhau: default VIF (95% workload) + localnet (provider network) + chassisredirect (gateway) + patch (inter-datapath) + l2gateway (VTEP) + localport (metadata) + virtual (VRRP) + remote (inter-cluster).

Thứ hai, default VIF claim dựa **entirely** trên `external_ids:iface-id` khớp với giữa OVS port và NBDB LSP. Missing iface-id = no claim = no network. Đây là pain point phổ biến nhất.

Thứ ba, `chassisredirect` + HA_Chassis_Group + BFD là pattern north-south NAT. Failover 3-5s default, < 1s tuned BFD. Chỉ một chassis active tại một thời điểm (không distributed như Logical Router).

Thứ tư, `localport` tồn tại trên mọi chassis — dùng cho metadata service scale linear. Traffic không cross-chassis.

Thứ năm, 5-step Port_Binding diagnostic (NBDB LSP → SBDB PB → OVS iface-id → claim chassis → OpenFlow rule) là workflow chuẩn debug "VM không có network". Biết sequence này giảm MTTR từ giờ xuống phút.

Thứ sáu, Port_Binding chỉ tồn tại trong SBDB (không phải NBDB). Query đúng table khi debug.

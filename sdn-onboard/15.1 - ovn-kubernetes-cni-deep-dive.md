## Mục tiêu bài học

Sau khi hoàn thành phần này, bạn có thể:

1. **Mô tả (Understand)** OVN-Kubernetes architecture: ovnkube-master (controller translating K8s resources → OVN NBDB), ovnkube-node (per-node agent), ovnkube-controller (IPAM + route distribution).
2. **Phân tích (Analyze)** NetworkPolicy → OVN ACL translation semantic: ingress/egress rules, podSelector, namespaceSelector, port-range.
3. **Đánh giá (Evaluate)** OVN-K8s vs Calico BGP vs Cilium eBPF: performance, debugging complexity, NetworkPolicy expressiveness, service mesh integration.

*Skeleton — nội dung sẽ bao gồm:* NetworkPolicy resource semantics, translation algorithm (podSelector → Port_Group, namespaceSelector → nested group), default-deny implementation, stateful ACL với conntrack.

*Skeleton — nội dung sẽ bao gồm:* ClusterIP Service → OVN Load_Balancer, NodePort → chassisredirect + DNAT, ExternalTrafficPolicy=Local optimization, ExternalIP annotation.

### ▶ Guided Exercise 1: Deploy OVN-K8s on kind cluster

> **Trạng thái lab verification:** Output dựa trên OVN-Kubernetes docs 2024; chờ verify lab thực.

**Mục đích:** Run OVN như CNI của Kubernetes, verify NetworkPolicy → OVN ACL translation working end-to-end. Đây là setup production-representative để hiểu OVN-K8s components.

**Chuẩn bị:**
- Ubuntu 22.04 VM, 8 GB RAM, 4 CPU
- kind + kubectl installed
- Clone OVN-K8s: `git clone https://github.com/ovn-org/ovn-kubernetes.git`

**Mô hình lab:**

```
┌──────────────────────────────────────────────────┐
│         kind cluster (3 nodes)                   │
│                                                  │
│  ns ovn-kubernetes:                              │
│                                                  │
│  ┌────────────────────────────────────────────┐  │
│  │ ovnkube-master-0/1/2 (3 replicas)          │  │
│  │  ├ northd (NBDB → SBDB compile)            │  │
│  │  ├ nb-ovsdb + sb-ovsdb (Raft cluster)      │  │
│  │  └ ovnkube-master (K8s→OVN translator)     │  │
│  └────────────────────────────────────────────┘  │
│                                                  │
│  ┌────────────────────────────────────────────┐  │
│  │ ovnkube-node (daemonset, per K8s node)     │  │
│  │  ├ ovn-controller (SBDB → OpenFlow)        │  │
│  │  └ ovs-vswitchd + ovsdb-server             │  │
│  └────────────────────────────────────────────┘  │
│                                                  │
│  ns default:                                     │
│  ┌────────────┐  ┌────────────┐                  │
│  │ pod web    │  │ pod app    │                  │
│  │ 10.244.0.5 │  │ 10.244.0.6 │                  │
│  └─────┬──────┘  └─────▲──────┘                  │
│        │  NetworkPolicy │                        │
│        └────────────────┘ → OVN ACL              │
└──────────────────────────────────────────────────┘
```

**Bước thực hiện:**

**1.** Tạo kind cluster với OVN-K8s CNI disable default kindnet:

```bash
[host]$ cat > kind-ovn.yaml << 'EOF'
kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
networking:
  disableDefaultCNI: true
nodes:
  - role: control-plane
  - role: worker
  - role: worker
EOF
[host]$ kind create cluster --name ovn-k8s --config kind-ovn.yaml
```

**2.** Deploy OVN-K8s:

```bash
[host]$ cd ovn-kubernetes/dist/images
[host]$ ./daemonset.sh --image=ghcr.io/ovn-org/ovn-kubernetes/ovn-kube-ubuntu:master
[host]$ kubectl apply -f ovn-setup.yaml
[host]$ kubectl apply -f rbac-ovnkube-cluster-manager.yaml
[host]$ kubectl apply -f ovnkube-master.yaml
[host]$ kubectl apply -f ovnkube-node.yaml
```

**3.** Verify pods running:

```bash
[host]$ kubectl -n ovn-kubernetes get pods
```

Output kỳ vọng:

```
NAME                        READY   STATUS
ovnkube-master-0            5/5     Running
ovnkube-master-1            5/5     Running
ovnkube-master-2            5/5     Running
ovnkube-node-xxxxx          3/3     Running
ovnkube-node-yyyyy          3/3     Running
ovnkube-node-zzzzz          3/3     Running
```

**4.** Inspect OVN NBDB:

```bash
[host]$ kubectl exec -n ovn-kubernetes ovnkube-master-0 -c nb-ovsdb -- ovn-nbctl show
```

Output kỳ vọng: Logical switches cho mỗi node (`join`, `ovn_cluster_router`, node-specific), Logical_Switch_Ports cho pods.

**5.** Deploy test pods:

```bash
[host]$ kubectl run web --image=nginx --labels=tier=web
[host]$ kubectl run app --image=alpine --labels=tier=app -- sleep 3600
[host]$ kubectl get pods -o wide
```

**6.** Apply NetworkPolicy cho phép web→app:80:

```bash
[host]$ cat > policy.yaml << 'EOF'
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-web-to-app
spec:
  podSelector:
    matchLabels:
      tier: app
  policyTypes: [Ingress]
  ingress:
  - from:
    - podSelector:
        matchLabels:
          tier: web
    ports:
    - protocol: TCP
      port: 80
EOF
[host]$ kubectl apply -f policy.yaml
```

**7.** Observe OVN ACL creation:

```bash
[host]$ kubectl exec -n ovn-kubernetes ovnkube-master-0 -c nb-ovsdb -- ovn-nbctl list ACL | head -30
```

Output kỳ vọng:

```
_uuid               : abc123...
action              : allow-related
direction           : to-lport
label               : 1
log                 : false
match               : "outport == @a123_Ingress && ip4 && tcp.dst == 80 && ip4.src == {10.244.0.5}"
name                : "default:allow-web-to-app:Ingress:0"
priority            : 1001
severity            : []
```

**Output mong muốn:** NetworkPolicy → OVN ACL translation occurs automatically, Port_Group `a123_Ingress` membership = pods với label `tier=app`.

**Bài học học được:**

Thứ nhất, **OVN-K8s là highly-available CNI**: ovnkube-master chạy 3 replicas (Raft cluster cho OVSDB), ovnkube-node daemonset per-worker. Resilient vs. single-point-of-failure CNI (old OpenShift-SDN).

Thứ hai, **NetworkPolicy translation tự động**: Label selector → Port_Group membership, port rule → ACL match expression. Mỗi K8s NetworkPolicy sinh 2-5 OVN ACL entries tùy complexity.

Thứ ba, **ovn-nbctl remain source of truth** — debug NetworkPolicy issues by inspecting NBDB ACLs, không đi theo iptables path như kube-proxy.

**Cleanup:**

```bash
[host]$ kind delete cluster --name ovn-k8s
```

### ▶ Lab 2: NetworkPolicy debug + trace

> **Trạng thái lab verification:** Output tracing format dựa trên ovn-trace docs; chờ verify lab thực.

**Mục đích:** Thực hành debugging NetworkPolicy issues với `ovn-trace` — tool có giá trị lớn khi production NetworkPolicy không work as expected.

**Chuẩn bị:** Hoàn thành Ex1 baseline (OVN-K8s cluster running).

**Mô hình lab:**

```
3-tier application với NetworkPolicy chain:

┌─────────┐  ACL1   ┌─────────┐  ACL2   ┌─────────┐
│ pod web │────────►│ pod app │────────►│ pod db  │
│ tier=web│ :8080   │ tier=app│ :5432   │ tier=db │
└─────────┘         └─────────┘         └─────────┘
                                             ▲
                      default-deny: block mọi else
```

**Bước thực hiện:**

**1.** Deploy 3-tier pods:

```bash
[host]$ kubectl run web --image=nginx --labels=tier=web
[host]$ kubectl run app --image=alpine --labels=tier=app -- sleep 3600
[host]$ kubectl run db --image=postgres --env=POSTGRES_PASSWORD=x --labels=tier=db
```

**2.** Apply 3-tier NetworkPolicies:

```yaml
# policy-web-to-app.yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata: {name: allow-web-to-app}
spec:
  podSelector: {matchLabels: {tier: app}}
  policyTypes: [Ingress]
  ingress:
  - from: [{podSelector: {matchLabels: {tier: web}}}]
    ports: [{protocol: TCP, port: 8080}]

# policy-app-to-db.yaml (similar structure cho tier=db, port=5432)
```

**3.** Apply cả hai:

```bash
[host]$ kubectl apply -f policy-web-to-app.yaml -f policy-app-to-db.yaml
```

**4.** Verify với ovn-trace (simulate packet web → app:8080):

```bash
[host]$ WEB_IP=$(kubectl get pod web -o jsonpath='{.status.podIP}')
[host]$ APP_IP=$(kubectl get pod app -o jsonpath='{.status.podIP}')
[host]$ kubectl exec -n ovn-kubernetes ovnkube-master-0 -c ovnkube-master -- \
        ovn-trace --minimal "inport == \"web\" && \
                             eth.src == 0a:00:00:00:00:01 && \
                             ip4.src == $WEB_IP && \
                             ip4.dst == $APP_IP && \
                             tcp.dst == 8080"
```

Output kỳ vọng: trace qua OVN logical pipeline, cuối cùng `output to \"app\"` action.

**5.** Test actual traffic:

```bash
[host]$ kubectl exec web -- curl -m 5 http://$APP_IP:8080
```

Expected: connection success hoặc HTTP response.

**6.** Intentionally break: delete app→db rule:

```bash
[host]$ kubectl delete networkpolicy allow-app-to-db
```

**7.** Debug blocked traffic:

```bash
[host]$ DB_IP=$(kubectl get pod db -o jsonpath='{.status.podIP}')
[host]$ kubectl exec -n ovn-kubernetes ovnkube-master-0 -c ovnkube-master -- \
        ovn-trace --minimal "inport == \"app\" && ip4.src == $APP_IP && \
                             ip4.dst == $DB_IP && tcp.dst == 5432"
```

Output kỳ vọng: trace ends with `drop` action (no matching allow ACL → default-deny).

**8.** Inspect logical flows để xác định nguyên nhân:

```bash
[host]$ kubectl exec -n ovn-kubernetes ovnkube-master-0 -c nb-ovsdb -- \
        ovn-sbctl lflow-list | grep -A1 "ls_out_acl"
```

**Output mong muốn:** Clear decision path showing exactly which ACL allowed/denied, match conditions evaluated at each stage.

**Bài học học được:**

Thứ nhất, **`ovn-trace` là debugging superpower**: cho phép simulate packet qua entire logical pipeline without sending real packet. Compare với iptables/kube-proxy debugging (phải tcpdump từng interface) → far simpler.

Thứ hai, **default-deny semantic**: khi NetworkPolicy exists cho một pod (ingress hoặc egress), non-matched traffic **implicitly denied**. Common mistake: apply NetworkPolicy mà quên add necessary allow rule → pod bị isolate.

Thứ ba, **OVN logical flows visibility**: `ovn-sbctl lflow-list` show all compiled logical flows, priority-ordered. Debug NetworkPolicy translation issues here.

**Cleanup:**

```bash
[host]$ kubectl delete pod web app db
[host]$ kubectl delete networkpolicy --all
```

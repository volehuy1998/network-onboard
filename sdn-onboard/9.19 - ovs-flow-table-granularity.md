## 9.19.1 Structure of a flow entry in OVS

The Lab 4 reference (Crichigno et al., USC WASTC 2021) describes a flow entry concisely: *"A flow entry consists of header fields, counters, and actions."* OVS implements this OpenFlow 1.0 definition fully and extends it with additional fields. Every flow entry on an OVS bridge contains:

| Component | Role | Example |
|-----------|------|---------|
| **Match fields** | Conditions a packet must satisfy to match the entry | `in_port=1,dl_type=0x0800,nw_dst=10.0.0.2` |
| **Priority** | Integer 0 to 65535, higher entries take precedence | `priority=100` |
| **Counters** | `n_packets` and `n_bytes`, statistics on every match | Auto-incremented |
| **Actions** | Action list executed on match | `output:2`, `drop`, `mod_dl_dst:aa:...` |
| **Timeouts** | `idle_timeout` (seconds of silence to delete), `hard_timeout` (absolute lifetime) | `idle_timeout=60,hard_timeout=300` |
| **Cookie** | 64-bit identifier set by controller or user to track a group of flows | `cookie=0x1001` |
| **Table** | Table holding the entry (OpenFlow 1.1 and later) | `table=0` |

When `ovs-ofctl add-flow br0 "priority=100,ip,nw_dst=10.0.0.2,actions=output:2"` runs, OVS inserts an entry into `table=0` with `dl_type=0x0800` (inferred from the keyword `ip`), `nw_dst=10.0.0.2`, `priority=100`, and the action `output:2`. The entry persists until manual deletion or timeout (if any).

> **Key topic.** A flow entry is the smallest forwarding unit on OVS. Unlike a Cisco router that uses a routing table plus a separate ARP cache, OVS packs the entire forwarding decision into a single entry. The match (condition) plus the action is enough. This is why OVS can act as a bridge, router, firewall, load balancer, or tunnel endpoint inside one bridge, with only the flow entries differing.

## 9.19.2 Topology and the four levels of matching

Lab 4 uses a minimal topology to focus on matching behaviour:

```
            +---------------------+
            |      br-s1          |
            |  (OVS bridge)       |
            |                     |
            | s1-eth1     s1-eth2 |
            +----+-----------+----+
                 |           |
                 |           |
           +-----+----+ +----+-----+
           |    h1    | |    h2    |
           |10.0.0.1/8| |10.0.0.2/8|
           +----------+ +----------+
```

Both hosts share subnet `10.0.0.0/8`, so no routing is needed. The exercise is to force OVS to forward packets between the two ports in four different ways, from least granular to most granular. The point of the exercise is to see that OVS does not distinguish "switch type" (L2 / L3 / L4); it only distinguishes the granularity of the field match.

### Level 1, L1 port-based forwarding

The simplest case: everything from port 1 goes out port 2, and everything from port 2 goes out port 1. OVS does not inspect any field other than `in_port`.

```bash
[host]# ovs-ofctl add-flow br-s1 "priority=100,in_port=1,actions=output:2"
[host]# ovs-ofctl add-flow br-s1 "priority=100,in_port=2,actions=output:1"
```

The result: hub-like behaviour, all traffic crosses between the two ports. Advantage: minimal CPU, maximum effective bandwidth. Disadvantage: a broadcast storm is possible if there are more than two ports, and there is no filtering by any other criterion.

### Level 2, L2 MAC-based forwarding

Match on MAC address for higher granularity:

```bash
[host]# ovs-ofctl add-flow br-s1 \
  "priority=100,dl_dst=02:00:00:00:00:02,actions=output:2"
[host]# ovs-ofctl add-flow br-s1 \
  "priority=100,dl_dst=02:00:00:00:00:01,actions=output:1"
```

OVS now forwards based on the destination MAC. This is the behaviour of a static-MAC L2 switch. OVS does not learn source MACs in this case; to learn dynamically, use the `NORMAL` action (which triggers the implicit MAC table) or the `learn()` action (which explicitly creates a new flow entry on each new packet seen).

> **Common misconception.** "OVS automatically learns MACs the way a Cisco switch does by default."
>
> Reality: OVS learns MACs only when the `NORMAL` action is invoked, because that action triggers the built-in learning logic. If the flow table only has explicit match-and-action rules (as in the example above), OVS does not learn anything. Many engineers assume OVS "is broken" when the flow table is empty, but in fact OVS behaves correctly given the flow table provided. The flow table simply does not request learning.

### Level 3, L3 IP-based forwarding

Match on IP destination:

```bash
[host]# ovs-ofctl add-flow br-s1 \
  "priority=100,ip,nw_dst=10.0.0.2,actions=output:2"
[host]# ovs-ofctl add-flow br-s1 \
  "priority=100,ip,nw_dst=10.0.0.1,actions=output:1"
```

The keyword `ip` is shorthand for `dl_type=0x0800` (Ethernet type IPv4). It is required because `nw_dst` is meaningful only when the frame is IP. With this flow table, OVS forwards IPv4 packets by destination IP. ARP packets (`dl_type=0x0806`) do not match, so they are not forwarded and they drop. In production, there must be a dedicated ARP flow or a fallback `NORMAL` action.

### Level 4, L4 TCP / UDP port-based forwarding

Match deep into the L4 header:

```bash
# Assume h2 runs a Python web server on port 80
[host]# ovs-ofctl add-flow br-s1 \
  "priority=100,tcp,nw_dst=10.0.0.2,tp_dst=80,actions=output:2"
[host]# ovs-ofctl add-flow br-s1 \
  "priority=100,tcp,nw_dst=10.0.0.1,tp_dst=80,actions=output:1"
```

The keyword `tcp` is `dl_type=0x0800,nw_proto=6`. `tp_dst=80` matches HTTP. With this flow table, only TCP packets to port 80 are forwarded. ICMP, SSH, and UDP all drop. This is the foundation for an L4 firewall or an L4 load balancer built only from flow entries.

## 9.19.3 The priority resolution mechanism

OVS stores flow entries in the classifier (details in Part 9.15, tuple space search). When a packet arrives, the classifier returns the matching entry with the **highest priority**. If several entries share the highest priority and all match, the tiebreaker is *"first-installed wins"*, that is, the entry inserted first (see `ovs-ofctl(8)` section `MATCH PRIORITY`).

For example, given the following flow table:

```bash
# Rule 1: allow TCP/80 to h2
[host]# ovs-ofctl add-flow br-s1 \
  "priority=200,tcp,nw_dst=10.0.0.2,tp_dst=80,actions=output:2"

# Rule 2: block all TCP to h2
[host]# ovs-ofctl add-flow br-s1 \
  "priority=100,tcp,nw_dst=10.0.0.2,actions=drop"

# Rule 3: forward all IP to h2
[host]# ovs-ofctl add-flow br-s1 \
  "priority=50,ip,nw_dst=10.0.0.2,actions=output:2"
```

A TCP/80 packet to h2 matches Rule 1, Rule 2, and Rule 3. Rule 1 has priority 200 (highest), so the action is `output:2` (allow).

A TCP/22 packet to h2 does not match Rule 1 (wrong port), but matches Rule 2 and Rule 3. Rule 2 has priority 100 (higher than Rule 3), so the action is `drop` (block SSH).

An ICMP packet to h2 does not match Rule 1 (not TCP), does not match Rule 2 (not TCP), but matches Rule 3, so the action is `output:2` (allow ping).

This is the classic ACL pattern: specific rules at high priority, catch-all rules at low priority. The whole logic fits inside the flow table, with no separate firewall module needed.

> **Key topic.** Priority resolution lets you pack firewall rules, routing rules, and load-balancing rules into the same flow table. The fact that OVS can do everything in one bridge instead of chaining multiple bridges, iptables instances, and route tables is why cloud data plane architectures choose OVS as the backbone.

### Guided Exercise 1: prove that OVS does not auto-learn MACs without the NORMAL action

This exercise proves, by counter-example, that OVS learning is opt-in via an action and not automatic. Engineers often believe OVS behaves "like a normal switch" after a quick read of the documentation. This exercise breaks that assumption with empirical evidence.

**Outcomes:**

- Observe an empty MAC table after installing a flow table with no `NORMAL` action and sending traffic through.
- Verify that on the same topology with the `NORMAL` action, the MAC table populates.

**Before You Begin:**

The OVS service is running. Bridge `br-s1` plus two namespaces `h1` (10.0.0.1/8) and `h2` (10.0.0.2/8) follow the Lab 4 topology (the setup is similar to Part 9.18 §9.18.4 Guided Exercise).

**Instructions:**

**1.** Install an explicit flow table (no `NORMAL`):

```bash
[host]# ovs-ofctl del-flows br-s1
[host]# ovs-ofctl add-flow br-s1 "priority=100,in_port=1,actions=output:2"
[host]# ovs-ofctl add-flow br-s1 "priority=100,in_port=2,actions=output:1"
```

**2.** Generate traffic:

```bash
[host]# ip netns exec ns-h1 ping -c 5 10.0.0.2
```

**3. Predict-Observe-Explain on MAC learning.**

Counter-hypothesis: *"OVS auto-learns MACs like a traditional L2 switch. After traffic, `ovs-appctl fdb/show br-s1` must list the MAC of h1 and h2 on the matching ports."*

Observe by dumping the MAC table:

```bash
[host]# ovs-appctl fdb/show br-s1
 port  VLAN  MAC                Age
```

The output is empty (no entries). The counter-hypothesis is rejected.

Explanation: OVS does not learn MACs because the flow table never invokes the `NORMAL` action. The `output:N` action forwards directly and bypasses the MAC learning logic. The bridge forwards correctly (ping works), yet the MAC table remains empty. This is evidence that the two activities (forwarding and learning) are independent in OVS.

**4.** Reset and try with `NORMAL`:

```bash
[host]# ovs-ofctl del-flows br-s1
[host]# ovs-ofctl add-flow br-s1 "priority=100,actions=NORMAL"
[host]# ip netns exec ns-h1 ping -c 5 10.0.0.2
[host]# ovs-appctl fdb/show br-s1
 port  VLAN  MAC                Age
    1     0  02:00:00:00:00:01    2
    2     0  02:00:00:00:00:02    2
```

The MAC table now has two entries. `NORMAL` triggered the learning logic. Both flow tables forward identically, but only the second one learns. The cause is the `NORMAL` action, not the traffic.

**Finish:**

```bash
[host]# ovs-ofctl del-flows br-s1
```

## 9.19.4 idle_timeout, hard_timeout, and cookie

Three fields manage the lifecycle of a flow entry:

**`idle_timeout`** (0 means no limit, default): the number of seconds after the last matching packet at which the entry is deleted. Use it to garbage-collect inactive flows.

```bash
[host]# ovs-ofctl add-flow br-s1 \
  "priority=100,tcp,nw_dst=10.0.0.2,tp_dst=80,idle_timeout=60,actions=output:2"
```

If 60 seconds pass with no matching TCP/80 packet, the entry self-deletes. OpenFlow controllers use this pattern often: install a specific flow on the first packet and let the timeout clean up. The OVN controller uses a similar mechanism for conntrack flows.

**`hard_timeout`**: an absolute lifetime regardless of traffic. Use it for flows with a fixed lifetime (for example, an authentication rule valid for one hour).

**`cookie`**: a 64-bit identifier assigned by the controller or user to group flows. It does not affect matching or actions; it is only used for later querying or deletion by group:

```bash
# Install a batch of flows with cookie=0x1001 (for example, tenant A)
[host]# ovs-ofctl add-flow br-s1 \
  "cookie=0x1001,priority=100,ip,nw_src=10.1.0.0/24,actions=output:2"
[host]# ovs-ofctl add-flow br-s1 \
  "cookie=0x1001,priority=100,ip,nw_src=10.1.1.0/24,actions=output:2"

# Delete every flow of tenant A by cookie mask
[host]# ovs-ofctl del-flows br-s1 "cookie=0x1001/-1"
```

OVN uses cookies to mark the Logical Flow to OpenFlow mapping. When you run `ovs-ofctl dump-flows br-int` on an OVN node, the cookie value tells you which Logical Flow this OpenFlow rule belongs to (see Part 13.7). The cookie is the primary mechanism for `ovs-ofctl` to inspect OVN runtime state.

## 9.19.5 Microflow versus megaflow trade-off (tier 2)

Part 9.2 introduced the megaflow cache. This section dives into the trade-off for an engineer who designs flow tables.

### Microflow

A **microflow** is a per-connection cache entry (per 5-tuple). Each packet matches an exact 5-tuple, producing one microflow entry. One thousand simultaneous connections produce one thousand entries.

Advantages:

- Fast lookup, O(1) hash on the 5-tuple.
- Per-connection statistics are accurate.

Disadvantages:

- Memory blows up with many connections.
- Cache invalidation when a flow rule is added (each `flow_mod` may invalidate thousands of microflows).
- Does not scale with high connection rates (10K connections per second).

Older OVS (pre-2.0) used a microflow cache. The NSDI 2015 paper by Pfaff et al. documents this scaling problem.

### Megaflow

A **megaflow** is a per-mask cache entry (per rule subset). It groups many microflows into one entry with a wildcard mask. One thousand connections sharing a pattern collapse into one megaflow entry.

Advantages:

- Memory is linear in the number of rules, not connections.
- Cache invalidation only affects the affected pattern, not a broadcast.

Disadvantages:

- Lookup is more complex (TSS subtable iteration).
- Each unique mask is one subtable. More masks mean more subtables and slower lookup.

OVS 2.0 and later default to megaflow. Every production cluster (including OVN) uses megaflow.

### When microflow returns as a fallback

EMC (Exact Match Cache) in OVS-DPDK is a microflow-style cache layered on top of megaflow. Hot connections (high rate) hit EMC and get an O(1) lookup. Less-hot connections fall through to megaflow.

Trade-off in design: EMC is small (8K entries default per PMD), and megaflow is scalable. The stacked pattern (EMC plus SMC plus megaflow in OVS 2.15 and later) provides both speed and scale.

## 9.19.6 Wildcard mask design and subtable creation

A megaflow mask defines which fields are exact-matched and which are wildcarded. The wildcard mask design determines the subtable count.

### Mask construction from a rule

When the first packet matches a rule, the OVS classifier tracks which fields it consulted:

```
Rule: priority=100,tcp,nw_dst=10.0.0.2,tp_dst=80,actions=output:2

Packet match: in_port=1,eth_src=AA:BB:CC:00:00:01,eth_dst=AA:BB:CC:00:00:02,
              eth_type=0x0800,nw_src=10.0.0.1,nw_dst=10.0.0.2,
              nw_proto=6,tp_src=12345,tp_dst=80

Megaflow mask after lookup:
  eth_type=0xffff (consulted, exact value 0x0800)
  nw_proto=0xff   (consulted, exact value 6)
  nw_dst=0xffffffff (consulted, exact value 10.0.0.2)
  tp_dst=0xffff   (consulted, exact value 80)
  ALL OTHER FIELDS WILDCARDED (mask=0)

Megaflow entry: 1 entry covering ALL connections whose 5-tuple matches this pattern
```

A second connection with a different `nw_src` or `tp_src` falls into the same megaflow and gets a cache hit.

### Subtable count equals unique mask count

Each unique megaflow mask is one subtable. One hundred rules with one hundred unique masks produce one hundred subtables, so the classifier checks one hundred subtables per cache miss.

Production target: fewer than 10 subtables per bridge to keep the classifier cost below 1 microsecond.

To measure the subtable count:

```bash
[host]# ovs-appctl dpctl/dump-flows br-int | head -5
[host]# ovs-appctl upcall/show
  threads: 4
  ...
[host]# ovs-appctl dpif-netdev/dpif-show
  br-int (port count: 5)
    masks-cache-hits: 12345678
    masks-cache-misses: 12345
    masks: 7    <- subtable count
```

`masks: 7` means 7 unique masks, that is, 7 subtables. Healthy is below 10. Warning is 10 to 50. Critical is above 50 (see Part 9.15 on subtable explosion).

### Good design pattern

**Good.** Rules at the same layer using the same field set:

- 100 rules matching `nw_dst` (an L3 router) collapse into 1 mask, 1 subtable.
- 100 rules matching `nw_dst` plus `tp_dst` (an L4 firewall) collapse into a different mask, 1 subtable.
- Total: 2 subtables.

**Bad.** Rules with mixed patterns:

- 1 rule matches `in_port`.
- 1 rule matches `dl_dst`.
- 1 rule matches `nw_src`.
- 1 rule matches `nw_dst`.
- 1 rule matches `nw_dst` plus `tp_dst`.
- 1 rule matches `nw_src` plus `nw_dst` plus `tp_dst`.
- Total: 6 subtables, each holding only 1 rule.

Solution: refactor the rules to consolidate masks. Sometimes use an explicit OR (multi-flow rules) instead of relying on the classifier's wildcard discovery.

## 9.19.7 Match field cardinality and classifier cost

Field cardinality is the number of possible values a field can take. High cardinality means many distinct match rules and good subtable scaling. Low cardinality means rules cluster into one subtable.

| Field | Cardinality | Subtable impact |
|-------|-------------|-----------------|
| `in_port` | Below 100 (port count) | Low cardinality, 1 subtable for 100 port-matching rules |
| `dl_dst` (MAC) | 2^48 | High cardinality, 100 rules with different MACs map to 1 subtable but a large table |
| `dl_type` | About 10 common values | Low cardinality, group rules by L3 protocol |
| `nw_dst` (IP/32) | 2^32 | High cardinality, IP-based rules scale |
| `nw_dst` (subnet) | Bound by prefix length | Patricia trie optimisation (see 9.15.8) |
| `nw_proto` | About 5 common values | Low cardinality |
| `tp_dst` (port) | 65536 | High cardinality |
| `ct_state` | 8-bit bitfield | Very low cardinality, group conntrack rules |

**Effective patterns:**

1. **Filter early on a low-cardinality field.** Match `dl_type=0x0800,nw_proto=6` (TCP) first, then refine with an IP or port subtable. This shrinks the search space.
2. **Patricia trie for IP prefix.** Section 9.15.8 documents the cmap plus trie. 10K rules with different CIDRs collapse into 1 trie with O(log N) lookup.
3. **Bitmask for conntrack state.** Rules matching `ct_state=+est+trk` group into 1 subtable instead of state-specific subtables.

## 9.19.8 Anatomy Template A: `dpctl/dump-flows` masked output

> **Operating context.** Inspect the actual megaflow entries on the kernel datapath and reconcile them with the flow table rules.

```
[host]# ovs-appctl dpctl/dump-flows br-int | head
recirc_id(0),in_port(2),eth_type(0x0800),ipv4(dst=10.0.0.2,proto=6,frag=no),
  tcp(dst=80), packets:12345, bytes:1234567, used:0.123s, actions:1

recirc_id(0),in_port(2),eth_type(0x0800),ipv4(proto=6,frag=no),tcp(),
  packets:0, bytes:0, used:never, actions:drop

recirc_id(0),in_port(2),eth_type(0x0800),ipv4(),
  packets:567, bytes:56789, used:1.234s, actions:1
```

| Field | Healthy | Warning | Critical | Diagnostic hint |
|-------|---------|---------|----------|-----------------|
| `recirc_id` | 0 (initial) or N (post-recirc) | (n/a) | Empty | Conntrack or group recirculation |
| `in_port` | Match expected | Wildcard `()` | Empty | Wildcarded mask check |
| `eth_type` | Specified | Wildcard | Empty | Higher-layer fields wildcarded too |
| `ipv4()` empty | OK (catch-all IPv4) | (n/a) | (n/a) | More wildcarded |
| `ipv4(dst=...,proto=...)` | Specific match | (n/a) | (n/a) | Specific subtable |
| `tcp()` empty | Wildcard L4 | (n/a) | (n/a) | Less specific |
| `tcp(dst=80)` | Specific port match | (n/a) | (n/a) | Most specific subtable |
| `packets` | Above 0 for active flow | 0 (used=never) | 0 plus non-stale | Rule has not matched traffic |
| `used` | Below 10s for active | 60 to 300s | Above 600s | Stale, will age out |
| `actions:N` | OFPort number | `drop` | Empty | Verify the drop is intentional |

**Failure scenarios.**

- The same rule pattern shows three or more times with different masks: subtable explosion. Refactor the rules.
- `actions:drop` for a rule that should forward: priority resolution is wrong. Check `ovs-ofctl dump-flows br-int -O OpenFlow13` (high level) against `dpctl/dump-flows` (datapath megaflow).
- `packets:0` for a freshly installed rule: the rule has not matched traffic yet. Wait or inject synthetic traffic.

### Difference between `ovs-ofctl dump-flows` and `dpctl/dump-flows`

- `ovs-ofctl dump-flows` (high level): shows the OpenFlow rule (bridge state). It shows intent.
- `dpctl/dump-flows` (low level): shows the megaflow datapath (kernel state). It shows the actual cache.

A mismatch means `ovs-vswitchd` has not yet installed the expected megaflow. The cause may be:

- Cold cache (no packet has triggered the cache yet).
- Userspace busy with upcalls.
- Kernel datapath full (rare).

## 9.19.9 Capstone POE, one wildcard rule versus N specific rules: which is faster?

> **Context.** An engineer designs a flow table for 1000 VMs in subnet 10.0.0.0/24, with all traffic exiting port 1. There are two approaches.
>
> - **Approach A.** One rule: `priority=100,ip,nw_dst=10.0.0.0/24,actions=output:1` (wildcard).
> - **Approach B.** One thousand rules: `priority=100,ip,nw_dst=10.0.0.X,actions=output:1` (specific per VM).

**Claim to verify.** "1000 specific rules (Approach B) are 1000 times slower than 1 wildcard rule (Approach A) because the classifier has to traverse more rules."

**Predict.** Wrong. Megaflow plus Patricia trie give both approaches O(log N) or O(1) lookup.

**Observe (hypothetical).**

- **Approach A**, 1 wildcard rule. 1 subtable with mask `nw_dst=0xffffff00`. Megaflow caches 1 entry covering every destination. Lookup is O(1).
- **Approach B**, 1000 specific rules. 1 subtable with mask `nw_dst=0xffffffff` (every bit exact). Megaflow caches 1000 entries (one per destination). Lookup is O(log N) via cmap hash. The Patricia trie does not help because the match is exact.
- Throughput differs by less than 5 percent between the two approaches.
- Memory: Approach B costs 1000 entries times 200 bytes equals 200 KB. Approach A costs 1 entry times 200 bytes. **Memory differs by 1000 times, throughput is nearly identical.**

**Explain.**

1. **Megaflow lookup is O(log N) or O(1)**, not linear in rule count. The cmap hash handles exact match, and the Patricia trie handles prefix match. The classifier does not "traverse N rules" per packet.
2. **Memory is the real constraint.** With 1M VMs, Approach B costs 200 MB just for the flow table, creating eviction pressure.
3. **Maintenance cost.** Approach B has 1000 rules to install or delete on each VM lifecycle event. Approach A has 1 rule. The operational cost differs greatly.

**Falsification test.**

- Set up OVS with Approach A versus Approach B. Run pktgen against 100 different `nw_dst` values inside 10.0.0.0/24.
- Measure Mpps and the RSS memory of `ovs-vswitchd`.
- If Approach B is more than 10 percent slower, that is a counter-example. In practice the throughput difference is below 5 percent.

**Where the claim is wrong.** "1000 times slower" assumes O(N) lookup. The correct statement: **megaflow lookup is O(log N) or O(1) via cmap plus trie. Wildcard rules are preferable for operational simplicity and memory efficiency, not for throughput.**

> **Rule of thumb.** Wildcard rules are preferable when the pattern is uniform (a subnet or a port range). Specific rules are appropriate only when per-entity action is needed (per-VM rate limiting, per-VM logging). Do not equate "rule count" with "throughput".

## Key takeaways

First, a flow entry has a uniform structure: match plus counters plus actions plus metadata (priority, timeouts, cookie, table). Every kind of forwarding (L1 to L4) uses the same structure; only the matched fields differ.

Second, OVS does not auto-learn MACs unless a flow action specifies `NORMAL`. This is what distinguishes OVS from a traditional switch. Learning is an opt-in feature, not a default.

Third, priority resolution lets you pack firewall, routing, and load-balancing rules into the same flow table. Specific rules at high priority and catch-all rules at low priority form the basic ACL pattern.

Fourth, `idle_timeout`, `hard_timeout`, and `cookie` form the toolkit for managing flow-entry lifecycle. The cookie is especially important when debugging OVN, because it links the OpenFlow runtime to the Logical Flow abstraction.

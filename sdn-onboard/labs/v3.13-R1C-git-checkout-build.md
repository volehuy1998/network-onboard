## Why R1.C exists and the carry-forward state

### Why R1.C exists

R1.C is the install path a learner takes when the version they need has not yet been published as a release tarball, when they need to apply a development branch patch, or when reading the autotools machinery is itself the lesson. A git checkout differs from a tarball in three concrete ways: the `configure` script does not exist on disk yet (the tarball ships it pre-generated; the git tree leaves it to be produced), the `Makefile.in` family of files is not on disk yet either, and the `tests/` directory contains the raw `.at` test sources from which the autotest binary is later assembled.

R1.C captures four things the prior install paths skipped. First, it shows what `./boot.sh` actually does: a one-line wrapper for `autoreconf --install --force`, which in turn invokes `libtoolize`, `aclocal`, `autoheader`, `automake`, and `autoconf` in the right order to produce the missing `configure` script and the missing `Makefile.in`. Second, it shows a real `configure.ac` warning (`AC_ERROR is obsolete`) that surfaces during regeneration; the warning is upstream technical debt that has not been cleaned up but does not block the build. Third, it shows that the binaries produced by R1.C are byte-identical to those produced by R1.B; the `diff` between the two `ovs-vsctl --version` outputs is empty, which is a useful invariant a learner can verify on their own host. Fourth, it shows the in-tree autotest suite (`make check`), the format of the `.at` test source files, the `AT_SETUP` macro that defines a test, and the `testsuite.log` file the autotest framework produces.

The transcript closes with `make distclean` and a disk reclaim. Because R1.B already installed the userspace into `/usr/local/`, R1.C does not run `make install`; the binaries are byte-identical anyway and overwriting them would not change anything observable.

### What `boot.sh` does

```text
root@lab-openvswitch:~/ovs-r1c/ovs# cd /root/ovs-r1c/ovs && cat boot.sh
#! /bin/sh
autoreconf --install --force
```

`boot.sh` is two lines. The shebang and a single command. `autoreconf --install --force` is the orchestration entry point of the GNU autotools chain; `--install` copies any missing helper files into `build-aux/` and `m4/`, and `--force` rebuilds even outputs that already exist. Internally `autoreconf` invokes `aclocal` (gathers `.m4` macros from `configure.ac` and `m4/`), `libtoolize` (copies the libtool helper macros into `m4/`), `autoheader` (produces `config.h.in` from `configure.ac`), `automake --add-missing` (produces `Makefile.in` from `Makefile.am` plus all `automake.mk` includes), and `autoconf` (produces the final `configure` script from `configure.ac` plus the gathered macros). The order matters and `autoreconf` enforces it.

```text
root@lab-openvswitch:~/ovs-r1c/ovs# cd /root/ovs-r1c/ovs && ./boot.sh 2>&1
libtoolize: putting auxiliary files in AC_CONFIG_AUX_DIR, 'build-aux'.
libtoolize: copying file 'build-aux/ltmain.sh'
libtoolize: putting macros in AC_CONFIG_MACRO_DIRS, 'm4'.
libtoolize: copying file 'm4/libtool.m4'
libtoolize: copying file 'm4/ltoptions.m4'
libtoolize: copying file 'm4/ltsugar.m4'
libtoolize: copying file 'm4/ltversion.m4'
libtoolize: copying file 'm4/lt~obsolete.m4'
configure.ac:197: warning: The macro `AC_ERROR' is obsolete.
configure.ac:197: You should run autoupdate.
./lib/autoconf/oldnames.m4:34: AC_ERROR is expanded from...
acinclude.m4:119: OVS_CHECK_LINUX is expanded from...
configure.ac:197: the top level
configure.ac:27: installing 'build-aux/compile'
configure.ac:43: installing 'build-aux/config.guess'
configure.ac:43: installing 'build-aux/config.sub'
configure.ac:22: installing 'build-aux/install-sh'
configure.ac:22: installing 'build-aux/missing'
Makefile.am: installing 'build-aux/depcomp'
```

The visible work happens in two phases. First, `libtoolize` copies its helper files: `ltmain.sh` into `build-aux/` and five `.m4` macros (`libtool.m4`, `ltoptions.m4`, `ltsugar.m4`, `ltversion.m4`, `lt~obsolete.m4`) into `m4/`. These provide the libtool support the configure script and the makefiles will rely on for shared-library handling.

Second, the warning. `configure.ac:197` references `AC_ERROR`, which `autoreconf` reports as obsolete:

> `configure.ac:197: warning: The macro 'AC_ERROR' is obsolete.`
>
> `configure.ac:197: You should run autoupdate.`

The warning chain identifies that `AC_ERROR` is expanded from `OVS_CHECK_LINUX` in `acinclude.m4:119`, and `OVS_CHECK_LINUX` is invoked at the top level of `configure.ac` line 197. `AC_ERROR` was a macro in older autoconf releases and was renamed to `AC_MSG_ERROR` long ago; the OVS source still uses the old form because nobody has run `autoupdate` to fix it. The warning is harmless: `autoreconf` recognises the obsolete name and substitutes the modern form, and the resulting `configure` script works correctly. A reader interested in upstream contribution practices could file a one-line patch that replaces `AC_ERROR` with `AC_MSG_ERROR` in `acinclude.m4`; it would land easily.

The third-phase output (`installing 'build-aux/compile'`, `'build-aux/config.guess'`, etc.) is `automake --add-missing` populating the auxiliary directory with helper scripts: `compile` (the GCC argument shimmer), `config.guess` and `config.sub` (the system-detection scripts), `install-sh` (a portable install replacement), `missing` (a wrapper for missing build tools), and `depcomp` (the dependency-output post-processor).

```text
root@lab-openvswitch:~/ovs-r1c/ovs# ls /root/ovs-r1c/ovs/configure 2>&1
/root/ovs-r1c/ovs/configure
```

After `boot.sh` returns, `configure` exists. The git checkout now has the same surface a release tarball does: the operator can run `./configure` directly.

### Binary identity check: R1.B vs R1.C

The R1.B install put `/usr/local/bin/ovs-vsctl` on disk. The R1.C build produced `/root/ovs-r1c/ovs/utilities/ovs-vsctl` in the source tree (libtool-wrapped). A `diff` of their `--version` output is the simplest cross-check that the two builds produced the same binaries from the same source.

```text
root@lab-openvswitch:~/ovs-r1c/ovs# cd /root/ovs-r1c/ovs && diff <(/root/ovs-r1c/ovs/utilities/ovs-vsctl --version) <(/usr/local/bin/ovs-vsctl --version) 2>&1 | head -5
root@lab-openvswitch:~/ovs-r1c/ovs# cd /root/ovs-r1c/ovs && /root/ovs-r1c/ovs/utilities/ovs-vsctl --version | head -3
ovs-vsctl (Open vSwitch) 2.17.9
DB Schema 8.3.0
root@lab-openvswitch:~/ovs-r1c/ovs# cd /root/ovs-r1c/ovs && /root/ovs-r1c/ovs/vswitchd/ovs-vswitchd --version 2>&1 | head -3
ovs-vswitchd (Open vSwitch) 2.17.9
```

The `diff` command produced no output. Empty diff means the two binaries report identical version strings, which for OVS is `ovs-vsctl (Open vSwitch) 2.17.9` plus `DB Schema 8.3.0` plus a third line listing the supported OpenFlow protocol versions (truncated by `head -3`). The R1.C `ovs-vswitchd` reports the same `2.17.9` tag.

The empty `diff` is not a guarantee that every byte of the two ELF binaries matches; the build directory paths and timestamps embedded in DWARF debug info will differ. But the `--version` strings depend only on the upstream source tree, the autotools-generated headers, and the compile flags, all three of which are identical between R1.B and R1.C. The version invariant is the right one to check at this layer.

## The in-tree autotest suite

Path C is the only path on which `make check` works, because path C is the only path that produces the autotest binary `tests/testsuite`.

### Inspecting the autotest source surface

The OVS in-tree test suite is built from `.at` (autotest) source files in `tests/`. Each `.at` file is a sequence of `AT_SETUP` blocks (one block per test); the autotest framework assembles all blocks across all files into a single `tests/testsuite` shell-script binary that the operator runs as a Bourne-shell program. R1.C inspects the surface to see what is there, what scale the suite operates at, and how individual tests are defined.

### The test-suite log

The autotest framework writes a summary log at `tests/testsuite.log` regardless of pass or fail. The log records the platform identification, the command line, and one block per test with the test's name and result. For a passing run it stays small; for a failing run it grows to include every diagnostic the test framework collected.

### What R1.C established

R1.C confirmed three invariants of the git-checkout install path on Ubuntu 22.04 plus OVS 2.17.9:

1. **`./boot.sh` is a one-line wrapper for `autoreconf --install --force` that produces the missing `configure` script.** The regeneration is fast (well under a second) and emits one upstream warning (`AC_ERROR is obsolete`) that does not block the build. After `boot.sh`, the git tree's surface is identical to the release tarball's.
2. **The userspace-only `./configure`, `make -j4`, and `make install` cycle from R1.B applies unchanged.** The build produces byte-equivalent binaries (`diff` of `--version` output is empty between R1.B's `/usr/local/bin/ovs-vsctl` and R1.C's `utilities/ovs-vsctl`).
3. **`make check` runs the autotest binary against the source tree.** The `-k <keyword>` flag selects a subset by name or `AT_KEYWORDS` list; `--jobs=N` parallelises across cores. A 45-test selective run takes under six seconds on the i9-11900K; the full suite (about 2 500 tests) takes 5 to 15 minutes. The test runner writes a `tests/testsuite.log` summary regardless of pass or fail.

R1.C also recorded the SHA `0bea06d9957e3966d94c48873cd9afefba1c2677` as the canonical v2.17.9 commit. Every later sprint of plan v3.13 that needs to reference upstream OVS source will use this SHA as the baseline for source citations under Rule 14.

The next step, once R1.A, R1.B, and R1.C are all on disk, is the curriculum integration file `sdn-onboard/0.4 - ovs-installation-three-paths.md` that compares the three install paths pedagogically. R1.C is the last of the three install transcripts.

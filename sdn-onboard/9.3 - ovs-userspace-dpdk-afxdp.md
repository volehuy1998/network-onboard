## Context: the performance ceiling of the kernel datapath

The kernel datapath of OVS is enough for most use cases. It serves VM workloads, container networks, and simple SDN deployments. However, when the required throughput exceeds about 10 Gbps per flow, or when the packet rate goes above 10 Mpps, the kernel stack becomes the bottleneck.

Three root causes:

**Context switches between kernel and userspace.** Each packet arriving at the NIC raises an interrupt. The kernel handles the interrupt, and a softirq decides how to forward. At 10 Mpps, the interrupt rate exceeds 1 million per second. The CPU spends 20 to 40 percent of its time only on interrupt overhead.

**Cache pollution.** The kernel network stack shares L1, L2, and L3 cache with the application. At high throughput, packet processing flushes the cache continuously.

**Syscall and lock contention.** The Linux bridge uses a spinlock for its forwarding database. It does not scale linearly with the number of CPU cores.

The userspace datapath solves these problems in two ways: **DPDK** (full kernel bypass) and **AF_XDP** (a hybrid approach that uses the kernel XDP framework).

---

## 9.3.1 DPDK: kernel bypass with the Poll Mode Driver

DPDK (Data Plane Development Kit) is a library Intel started in 2010 and transferred to the Linux Foundation in 2017. It lets a userspace application access NIC ring buffers directly.

The principle: the kernel driver is unbound from the NIC. A userspace driver (UIO or VFIO) maps the NIC memory into the userspace process. The userspace application runs a **Poll Mode Driver (PMD)**, a thread that reads the NIC ring continuously in a tight loop. It does not wait for an interrupt. When a packet arrives, the PMD reads it, processes it, and writes it to the tx ring. There is no context switch, no syscall, and no interrupt.

OVS configuration with DPDK (compass Ch 14):

```bash
[host]# ovs-vsctl set Open_vSwitch . other_config:dpdk-init=true
[host]# systemctl restart openvswitch-switch

[host]# ovs-vsctl add-br br-dpdk -- set Bridge br-dpdk datapath_type=netdev
[host]# ovs-vsctl add-port br-dpdk p0 -- set Interface p0 type=dpdk \
         options:dpdk-devargs=0000:01:00.0
```

Important tuning knobs:

- `other_config:dpdk-socket-mem=1024,1024`: allocate 1 GB of hugepage memory per NUMA socket.
- `other_config:dpdk-lcore-mask=0x2`: the core reserved for DPDK internal threads.
- `other_config:pmd-cpu-mask=0x3c`: the cores reserved for PMD polling (bits 2 to 5, that is cores 2, 3, 4, 5).
- `other_config:emc-insert-inv-prob`: probability of inserting into the Exact Match Cache. Set to 0 for workloads with many parallel flows.

Monitoring (compass Ch R):

```bash
[host]# ovs-appctl dpif-netdev/pmd-stats-show
[host]# ovs-appctl dpif-netdev/pmd-rxq-show
[host]# ovs-appctl dpif-netdev/pmd-perf-show    # requires other_config:pmd-perf-metrics=true
```

## 9.3.2 Hugepages and NUMA pinning

DPDK requires **hugepages**, memory pages of 2 MB or 1 GB instead of the default 4 KB. The reason: DPDK allocates large memory pools for ring buffers and packet buffers. With 4 KB pages, TLB (Translation Lookaside Buffer) misses happen often, and the CPU stalls while it walks the page table. With 2 MB hugepages, the TLB covers a larger memory area and TLB misses drop dramatically.

Hugepage configuration:

```bash
[host]# echo 1024 > /sys/kernel/mm/hugepages/hugepages-2048kB/nr_hugepages
[host]# echo "vm.nr_hugepages=1024" >> /etc/sysctl.conf
```

Kernel command line: `intel_iommu=on iommu=pt` (for VFIO), and `default_hugepagesz=1G hugepagesz=1G hugepages=4` (for 1 GB hugepages).

**NUMA pinning** matters on multi-socket servers. The NIC is attached to a PCIe slot that belongs to one specific CPU socket. The PMD thread must run on a core that belongs to the same socket as the NIC. Otherwise, the packet crosses the QPI (QuickPath Interconnect) between two sockets, and the latency goes up by 40 to 60 percent.

```bash
[host]# cat /sys/bus/pci/devices/0000:01:00.0/numa_node
0
[host]# ovs-vsctl set Open_vSwitch . other_config:pmd-cpu-mask=0x3c
```

CPU isolation: add `isolcpus=2-5` to the kernel command line. The Linux scheduler will not place other processes on these cores.

## 9.3.3 AF_XDP: an alternative without full kernel bypass

AF_XDP (Address Family eXpress Data Path) was introduced in Linux 4.18 (2018). It reaches throughput close to DPDK but does not bypass the kernel completely.

How it works: XDP is a framework that lets you attach an eBPF program to the NIC driver at an early position in the network stack, before the `sk_buff` is created. The XDP program can drop, pass, or redirect a packet to an AF_XDP socket. The AF_XDP socket gives userspace access to a ring buffer similar to DPDK.

Configuration:

```bash
[host]# ovs-vsctl add-port br-dpdk ethX -- set Interface ethX type=afxdp \
         options:xdp-mode=native
```

Three XDP modes:
- **`native`**: the NIC driver supports XDP natively, with zero-copy from the ring. Best performance.
- **`skb`**: a fallback when the driver does not support native mode. Slower.
- **`hw`**: the XDP program is offloaded to the NIC hardware (Netronome Agilio). Limited to specific hardware.

## 9.3.4 Comparison: kernel, DPDK, and AF_XDP

*Table 9-3: Trade-off matrix for the three datapath types*

| Criterion | Kernel | DPDK | AF_XDP |
|---|---|---|---|
| Kernel bypass | No | Full | No (uses XDP) |
| Driver | Kernel driver | Userspace UIO/VFIO | Kernel driver |
| Maximum throughput per flow | 5 to 10 Gbps | 100+ Gbps | 80 to 90 Gbps |
| Latency | Medium | Lowest | Close to DPDK |
| CPU cost | Interrupt-driven | 100 percent dedicated core | Moderate |
| Driver availability | All NICs | Limited list | Any NIC supported by Linux |
| Kernel features (iptables, conntrack) | Kept | Lost completely | Kept |
| Operational complexity | Low | High (hugepage, isolcpus, NUMA pin) | Medium |
| Container workload and CNI | Good | Poor | OK |

Decision matrix:

- Workload below 5 Gbps with simple operations: **Kernel**.
- Workload at 25+ Gbps where throughput is critical, or NFV that is latency-sensitive: **DPDK**.
- Workload between 10 and 25 Gbps that needs to keep kernel features: **AF_XDP**.
- Container plus Kubernetes: **Kernel** (DPDK does not integrate well with CNI).

## Key points to remember

First, the userspace datapath is not a default upgrade from the kernel. Each one serves a different use case. The kernel is enough for typical enterprise workloads; DPDK and AF_XDP are only necessary when throughput exceeds the kernel ceiling.

Second, DPDK requires dedicated hardware resources: a NIC (with the kernel driver unbound), CPU cores (`isolcpus` plus `pmd-cpu-mask`), and memory (hugepages). You cannot run DPDK and another workload on the same core, because PMD polling consumes 100 percent of the CPU.

Third, AF_XDP is a compromise: throughput is 80 to 90 percent of DPDK, operational complexity is lower, and kernel features stay available. It is the default choice for medium to high workloads that do not require maximum throughput.

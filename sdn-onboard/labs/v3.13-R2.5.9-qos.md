> **Captured:** 2026-04-30 08:15:50 UTC to 08:16:15 UTC inside `tmux new-session -s v3_13-r2_5_9` on `lab-openvswitch` (`192.168.1.250`). Lab host: Ubuntu 22.04.5 LTS, kernel `5.15.0-173-generic`, OVS 2.17.9 userspace install at `/usr/local/`.

A traditional Cisco engineer who has configured `class-map`, `policy-map`, and `service-policy` knows the textbook QoS pattern: classify traffic into classes, attach per-class queueing or shaping behaviour, and bind the policy to an interface. Cisco IOS handles classification and shaping in silicon at line rate; the operator's job is to express the policy in the right syntax.

OVS implements two distinct QoS mechanisms. First, **egress QoS via the OVSDB `QoS` and `Queue` tables**: an OVS port (the egress side of a bridge port) can be associated with a `QoS` row of type `linux-htb` (or `linux-hfsc`, `linux-policer`, etc.); the QoS row references one or more `Queue` rows that define per-queue rate limits. The OVS daemon translates these OVSDB rows into Linux `tc qdisc` and `tc class` operations on the underlying kernel netdev. Second, **ingress policing via Interface columns**: each Interface row has `ingress_policing_rate` and `ingress_policing_burst` columns that configure a token-bucket policer on the kernel ingress qdisc, dropping packets that exceed the rate.

This lab demonstrates both. Phase A configures linux-htb on the egress side of a bridge port with two child queues (8 Mbit/s and 2 Mbit/s) and verifies the rate ceiling with iperf3. Phase B reconfigures the same topology with ingress policing at 5 Mbit/s and again verifies with iperf3. The empirical confirmation is that the iperf3 throughput respects the configured rate limits in both cases.

## R2.5.9.1. Where this part sits in the curriculum and why

R2.5.9 is the ninth sub-capture of R2.5. R2.5.8 covered conntrack and NAT; R2.5.9 covers QoS, the bandwidth-management complement to the firewall. The pattern continues: OVS does not implement QoS itself; it delegates to the Linux kernel's `tc` (Traffic Control) subsystem, which has been the canonical Linux QoS framework since the 1999 era of HTB (Hierarchical Token Bucket).

The OVS-side configuration is unique because it expresses QoS via the `QoS` and `Queue` OVSDB tables rather than directly via `tc`. This gives the operator a stable, declarative configuration model that survives daemon restarts and can be programmed by a controller; OVS re-renders the OVSDB rows as `tc` commands on every reconfigure cycle. The lab will surface this by running `tc qdisc show` and `tc class show` immediately after configuring the OVS QoS and observing the underlying kernel state.

The R2.5.9 source anchors live in two places. First, `lib/netdev-linux.c:netdev_linux_set_qos` (line 2959) and `netdev_linux_set_queue` (line 3039) translate OVSDB `QoS` and `Queue` rows into `tc` Netlink operations. Second, `netdev_linux_set_policing` (line 2754) handles the ingress-policing case. Third, `vswitchd/bridge.c:iface_configure_qos` reads the OVSDB rows and invokes the netdev callbacks. The OVS source baseline is v2.17.9 (commit `0bea06d9957e3966d94c48873cd9afefba1c2677`).

## R2.5.9.2. Topology and OVSDB QoS model

Two namespaces, one bridge, two veth pairs. ns_a runs the iperf3 client, ns_b runs the iperf3 server. The QoS configuration sits on the bridge-side veth (egress for traffic from ns_a to ns_b in Phase A, ingress for traffic from ns_a in Phase B).

The OVSDB QoS model has three levels:

| OVSDB table | Role                                                        |
|-------------|-------------------------------------------------------------|
| `Port.qos`  | Reference to the `QoS` row attached to the egress side     |
| `QoS`       | Defines the qdisc type (`linux-htb`, `linux-hfsc`, ...) plus parent rate. References `Queue` rows for child classes. |
| `Queue`     | Per-queue rate limits in `other_config:max-rate` and `min-rate` |

The relationship is hierarchical: a Port references a QoS row, which references multiple Queue rows. The OVSDB schema enforces referential integrity (lab exercise 3 will surface this with a `destroy qos` attempt that fails because the Port still references the row).

### ▶ Guided Exercise 1: Build the topology and configure linux-htb egress QoS

This exercise builds the two-namespace topology, attaches a linux-htb QoS configuration with two queues to the egress side of veth_b_br, and verifies the rate ceiling via iperf3.

**Outcomes.** `ovs-vsctl` creates QoS plus two Queue rows. `tc qdisc show` reports the htb qdisc OVS installed. iperf3 measures throughput at ~7.30 Mbit/s (under the 8 Mbit/s ceiling for queue 0).

**2.** Configure linux-htb QoS with two child queues. The single multi-row transaction creates the QoS plus two Queues atomically and references them from veth_b_br.

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl -- set port veth_b_br qos=@newqos -- --id=@newqos create qos type=linux-htb other-config:max-rate=10000000 queues:0=@q0 queues:1=@q1 -- --id=@q0 create queue other-config:max-rate=8000000 -- --id=@q1 create queue other-config:max-rate=2000000
c392369a-1ae6-4ad9-863c-4ee2ac44ff88
67f1598a-7536-4451-962a-04cafa4f8521
04efb93a-a8e6-4a6c-8d1b-a074a853bfc7
```

The three UUIDs are the QoS row, queue 0, and queue 1 respectively.

**3.** Read back the OVSDB rows.

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=name,qos list port veth_b_br
name                : veth_b_br
qos                 : c392369a-1ae6-4ad9-863c-4ee2ac44ff88
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=type,other_config,queues list qos
type                : linux-htb
other_config        : {max-rate="10000000"}
queues              : {0=67f1598a-7536-4451-962a-04cafa4f8521, 1=04efb93a-a8e6-4a6c-8d1b-a074a853bfc7}
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=other_config list queue
other_config        : {max-rate="2000000"}

other_config        : {max-rate="8000000"}
```

The QoS row references both queues by their UUIDs in the `queues` map. Queue 0 has max-rate 8 Mbit/s; queue 1 has max-rate 2 Mbit/s; the parent QoS has max-rate 10 Mbit/s. The relationship is hierarchical: the parent ceiling caps the sum of all children.

**4.** Read the kernel-side `tc` state to see what OVS programmed.

```text
root@lab-openvswitch:~/network-onboard# tc qdisc show dev veth_b_br
qdisc htb 1: root refcnt 129 r2q 10 default 0x1 direct_packets_stat 0 direct_qlen 1000
root@lab-openvswitch:~/network-onboard# tc class show dev veth_b_br
class htb 1:1 parent 1:fffe prio 0 rate 12Kbit ceil 8Mbit burst 1563b cburst 1563b 
class htb 1:fffe root rate 10Mbit ceil 10Mbit burst 1500b cburst 1500b 
class htb 1:2 parent 1:fffe prio 0 rate 12Kbit ceil 2Mbit burst 1563b cburst 1564b 
```

The kernel has three HTB classes: `1:fffe` (the root, rate 10Mbit ceil 10Mbit), `1:1` (queue 0, ceil 8Mbit), `1:2` (queue 1, ceil 2Mbit). The `rate 12Kbit` on each child is the minimum guaranteed rate (12 kbps default); the `ceil` is the actual cap. OVS programmed the kernel to enforce the OVSDB max-rate values exactly.

**5.** Run iperf3 to verify the rate cap. Start the server in ns_b and the client in ns_a. Use `-O 1` to omit the first second's measurement (TCP slow-start), `-f m` for Mbit formatting, `-t 3` for 3-second test.

```text
root@lab-openvswitch:~/network-onboard# ip netns exec ns_b iperf3 -s -1 -D >/tmp/r2.5.9-iperf-server.log 2>&1; sleep 0.5; true
root@lab-openvswitch:~/network-onboard# ip netns exec ns_a iperf3 -c 10.0.0.2 -t 3 -O 1 -f m | tail -10
[  5]   0.00-1.00   sec  1.39 MBytes  11.6 Mbits/sec    0   90.5 KBytes       (omitted)
[  5]   0.00-1.00   sec   954 KBytes  7.82 Mbits/sec    0   90.5 KBytes       
[  5]   1.00-2.00   sec   954 KBytes  7.82 Mbits/sec    0   90.5 KBytes       
[  5]   2.00-3.00   sec   764 KBytes  6.26 Mbits/sec    0   90.5 KBytes       
- - - - - - - - - - - - - - - - - - - - - - - - -
[ ID] Interval           Transfer     Bitrate         Retr
[  5]   0.00-3.00   sec  2.61 MBytes  7.30 Mbits/sec    0             sender
[  5]   0.00-2.88   sec  2.62 MBytes  7.66 Mbits/sec                  receiver
[10 other lines omitted, context: iperf 'iperf Done.' marker line]
```

The empirical confirmation. The TCP slow-start window (omitted at 11.6 Mbit/s) is excluded from the measurement. Steady-state seconds 1, 2, and 3 measure 7.82, 7.82, and 6.26 Mbit/s. The aggregate sender rate is 7.30 Mbit/s, well under the queue-0 ceiling of 8 Mbit/s. The slight drop in second 3 is HTB's burst-credit refilling phase. Zero retransmits (`Retr=0`) confirms HTB shaped the traffic without dropping; HTB only delays packets when the rate is exceeded.

The rate is below 8 Mbit/s because all the iperf3 traffic landed in queue 0 by default (`queues:0=@q0`), which has the 8 Mbit/s ceiling. The 7.30 Mbit/s observed is consistent with HTB's standard shaping-overhead headroom.

**Misconception to refute.** "OVS QoS is hardware-accelerated." Wrong. OVS QoS delegates to the Linux kernel's `tc` framework, which is a software queueing implementation. There is no SmartNIC or hardware-offload path used in this lab. For high-speed ports (10 Gbit/s and above), software shaping has measurable CPU cost; some NICs support hardware offload via `tc-flower` plus `tc-actions`, which OVS can leverage with `datapath_type=netdev` and DPDK, but the lab uses the standard kernel datapath without any offload.

**Finish.** Egress HTB QoS configured and rate-ceiling verified.

### ▶ Guided Exercise 2: Switch to ingress policing

This exercise reconfigures the topology to use the simpler ingress_policing_rate / ingress_policing_burst mechanism. The fundamental difference from HTB: the policer drops excess packets rather than delaying them; the receiver experiences high retransmission rates rather than smooth shaping.

**Outcomes.** Ingress policer at 5 Mbit/s on veth_a_br. iperf3 measures ~2.26 Mbit/s, well under the 5 Mbit/s ceiling, with 138 retransmissions in 3 seconds (the policer dropped excess packets, TCP retransmits to compensate).

**Before You Begin.** Exercise 1 has completed.

**Instructions.**

**1.** Try to remove the QoS row directly. The expected result: a referential-integrity error because veth_b_br still references the QoS row.

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl -- destroy qos veth_b_br 2>&1 | head -3 || true
2026-04-30T08:16:05Z|00002|ovsdb_idl|WARN|transaction error: {"details":"cannot delete QoS row c392369a-1ae6-4ad9-863c-4ee2ac44ff88 because of 1 remaining reference(s)","error":"referential integrity violation"}
ovs-vsctl: transaction error: {"details":"cannot delete QoS row c392369a-1ae6-4ad9-863c-4ee2ac44ff88 because of 1 remaining reference(s)","error":"referential integrity violation"}
```

The `referential integrity violation` is the OVSDB schema enforcing that a QoS row referenced by a Port must not be destroyed. The correct order: clear the Port's reference first, then OVSDB will auto-garbage-collect the orphan rows.

**2.** Clear the reference.

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl clear port veth_b_br qos
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=name,qos list port veth_b_br
name                : veth_b_br
qos                 : []
```

The Port's qos column is now `[]` (empty). The QoS plus Queue rows are auto-deleted by OVSDB GC since no port references them anymore.

**3.** Configure ingress policing on veth_a_br. Rate is in kilobits per second; 5000 = 5 Mbit/s. Burst is the bucket size.

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl set Interface veth_a_br ingress_policing_rate=5000 ingress_policing_burst=500
root@lab-openvswitch:~/network-onboard# ovs-vsctl --columns=name,ingress_policing_rate,ingress_policing_burst list interface veth_a_br
name                : veth_a_br
ingress_policing_rate: 5000
ingress_policing_burst: 500
```

**4.** Read the kernel-side state. OVS installs an ingress qdisc plus a tc filter that implements the token-bucket policer.

```text
root@lab-openvswitch:~/network-onboard# tc qdisc show dev veth_a_br | head
qdisc noqueue 0: root refcnt 2 
qdisc ingress ffff: parent ffff:fff1 ---------------- 
root@lab-openvswitch:~/network-onboard# tc filter show dev veth_a_br | head
```

Two qdiscs visible: the default `noqueue` egress (unmodified) and the ingress qdisc. The filter listing is empty in this output because the policer is implemented as a `tc` action attached to the ingress qdisc rather than as an explicit filter rule; the kernel's policer implementation is internal and not visible via `tc filter show` in this form.

**5.** Run iperf3 again to measure the policed rate.

```text
root@lab-openvswitch:~/network-onboard# ip netns exec ns_b iperf3 -s -1 -D >/tmp/r2.5.9-iperf-server2.log 2>&1; sleep 0.5; true
root@lab-openvswitch:~/network-onboard# ip netns exec ns_a iperf3 -c 10.0.0.2 -t 3 -O 1 -f m | tail -10
[  5]   0.00-1.00   sec   524 KBytes  4.29 Mbits/sec   96   9.90 KBytes       (omitted)
[  5]   0.00-1.00   sec   318 KBytes  2.61 Mbits/sec   50   2.83 KBytes       
[  5]   1.00-2.00   sec   255 KBytes  2.08 Mbits/sec   48   2.83 KBytes       
[  5]   2.00-3.00   sec   255 KBytes  2.09 Mbits/sec   40   2.83 KBytes       
- - - - - - - - - - - - - - - - - - - - - - - - -
[ ID] Interval           Transfer     Bitrate         Retr
[  5]   0.00-3.00   sec   827 KBytes  2.26 Mbits/sec  138             sender
[  5]   0.00-2.53   sec   768 KBytes  2.49 Mbits/sec                  receiver
[10 other lines omitted, context: iperf 'iperf Done.' marker line]
```

The empirical observations differ from Phase A in two ways:

First, the **bitrate is much lower than the ceiling** (2.26 Mbit/s versus 5 Mbit/s configured ceiling). The reason is the **retransmission overhead**: 138 TCP retransmits in 3 seconds. Each retransmit consumes bandwidth without delivering new data, so the effective throughput is reduced by the retransmit-fraction. HTB shapes (delays), policing drops; policing is more aggressive and forces TCP to slow-start repeatedly.

Second, the **TCP cwnd is tiny**: `2.83 KBytes` window instead of HTB's `90.5 KBytes`. TCP shrinks its window aggressively under loss, which is what policing produces. Each retransmit triggers a window collapse; the window grows back during loss-free intervals.

These are the canonical signatures of policer-vs-shaper: a shaper gives smooth throughput close to the ceiling; a policer gives spiky throughput well below the ceiling with retransmits and small windows.

**Misconception to refute.** "Policing is faster than shaping at runtime." Yes by a small margin (the policer just decides drop-or-pass without queueing), but the application-layer throughput is usually worse because TCP responds badly to packet drops. For low-priority traffic that should be silently capped, policing is fine; for any traffic class where end-to-end performance matters, shaping (HTB or HFSC) is the correct choice.

**6.** Restore baseline by clearing the policer.

```text
root@lab-openvswitch:~/network-onboard# ovs-vsctl set Interface veth_a_br ingress_policing_rate=0 ingress_policing_burst=0
```

Setting rate=0 disables the policer; the kernel removes the tc state.

**Finish.** Ingress policing observed; the difference from HTB shaping is empirically visible in the throughput plus retransmit count.

## R2.5.9.3. Side by side: Cisco QoS, OVS QoS, Linux tc directly

| Concern                  | Cisco IOS QoS                                            | OVS QoS                                                  | Linux tc directly                              |
|--------------------------|----------------------------------------------------------|----------------------------------------------------------|------------------------------------------------|
| Configure shaping (HTB)  | `class-map`, `policy-map type queueing`, `service-policy` | `ovs-vsctl create qos type=linux-htb queues:0=@q0 ...`  | `tc qdisc add dev <ifc> root htb default 1; tc class add ...` |
| Configure policer        | `policy-map type policing; police rate 5 Mbit`          | `ovs-vsctl set Interface <ifc> ingress_policing_rate=5000` | `tc filter add ... action police rate 5Mbit ...` |
| Show state               | `show policy-map interface ...`                          | `ovs-vsctl --columns=qos list port; tc class show dev`  | `tc class show dev ...`, `tc filter show dev ...` |
| Source-code anchor       | n/a (closed)                                            | `lib/netdev-linux.c:netdev_linux_set_qos` (line 2959), `netdev_linux_set_policing` (line 2754) | Kernel `net/sched/sch_htb.c`, `net/sched/act_police.c` |

The takeaway: all three implementations ultimately use the same kernel `tc` machinery (or its hardware-offloaded equivalent on Cisco). OVS and Linux tc both invoke the same `sch_htb.ko` kernel module on the standard kernel datapath; the difference is only in the configuration surface (OVSDB rows versus `tc` commands).

## R2.5.9.4. Common misconceptions, production assessment, source references

**Misconception 1.** "Setting `other_config:max-rate` per Queue applies to the queue independently." Mostly. Each Queue's max-rate is its ceiling (`ceil` in HTB terminology); the parent QoS's max-rate is the overall ceiling. If queue 0's ceil is 8 Mbit and queue 1's ceil is 2 Mbit and the parent ceil is 10 Mbit, the sum of queue ceilings can EXCEED the parent (queue 0 at 8 Mbit + queue 1 at 2 Mbit = 10 Mbit), in which case the parent caps it. If the sum is less than the parent, the parent is the constraint.

**Misconception 2.** "Ingress policing is symmetric with egress shaping." Wrong. They are different mechanisms (drop versus delay). For full bidirectional rate control, configure HTB on both sides of a link or HTB on egress plus ingress on the receiving veth. The lab's Phase B used policing on the ingress side and saw the typical high-retransmit, low-throughput signature.

**Production-readiness:** the trade-off cost of QoS at scale. Each HTB class adds per-packet processing in the kernel datapath. For a host with many tenants and many queues, the CPU cost can be substantial. Mitigation: batch queues into fewer classes; use hardware offload via `tc-flower` if the NIC supports it.
